## Summary

Keep this PR open. Current main still lacks an EasyRunner guide, but the proposed docs have blocking deployment and security issues: the guide omits the required public Control UI origin, normalizes a break-glass device-auth bypass, has an update flow that does not rebuild the local image, and describes automatic config creation that the current container entrypoint does not provide.

## What This Changes

The PR adds `docs/platforms/easyrunner.md` and registers the new EasyRunner page in `docs/docs.json` and `docs/platforms/index.md`.

## Best Possible Solution

Rebase or replace the guide in the current hosting docs structure, add an explicit config/setup step for bind mode and `gateway.controlUi.allowedOrigins`, use trusted-proxy or normal device-pairing guidance instead of the device-auth bypass, and make update instructions rebuild the local Podman image before redeploying.

## Reproduction Assessment

Yes. The defects are reproducible by inspection against current main and the PR head: current main has no EasyRunner page, the PR has no `allowedOrigins`/bind setup, current gateway startup requires explicit origins for non-loopback Control UI, and the PR update command redeploys without rebuilding the referenced local image.

## Solution Assessment

No. The PR is useful source work, but this version is not the best way to solve the docs gap because the deployment recipe conflicts with current Control UI security/startup requirements and needs a rebase plus security-correct proxy guidance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Configure the public Control UI origin before deploy:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but the compose/config steps never add that origin to `gateway.controlUi.allowedOrigins`. Current gateway startup rejects non-loopback Control UI deployments without explicit origins, so this recipe can fail before users ever reach the UI.
  - confidence: 0.9
- **[P1] Stop requiring the device-auth bypass:** `docs/platforms/easyrunner.md:280`
  - body: The guide says `dangerouslyDisableDeviceAuth` is required for reverse-proxy setups. Current docs and audit checks classify that flag as break-glass only and a severe downgrade, so publishing this as normal setup would weaken remote Control UI authentication.
  - confidence: 0.94
- **[P2] Add the missing persistent config setup step:** `docs/platforms/easyrunner.md:108-112`
  - body: The guide claims the container automatically creates `/home/node/.openclaw/openclaw.json` with gateway and plugin settings, but the current image command only starts the gateway with `--allow-unconfigured`; that bypass does not create a durable config. Users need an explicit config creation or onboarding step before relying on those settings.
  - confidence: 0.84
- **[P2] Rebuild the local image before redeploying updates:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls new source and runs `er app deploy`, but the app runs `localhost/openclaw:latest`. Without a `podman build -t localhost/openclaw:latest .` step before redeploy, EasyRunner can keep launching the previously built OpenClaw version.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The docs-only PR introduces security-sensitive deployment guidance that normalizes disabling Control UI device authentication for reverse-proxy access.

Concerns:

- **[high] Device-auth bypass documented as required:** `docs/platforms/easyrunner.md:280`
  - body: The proposed EasyRunner guide presents `gateway.controlUi.dangerouslyDisableDeviceAuth=true` as required for reverse proxies even though current docs and audit checks treat it as a severe break-glass downgrade. Users following it would disable device identity checks for remote Control UI access.
  - confidence: 0.94
- **[medium] Public origin allowlist omitted:** `docs/platforms/easyrunner.md:102`
  - body: The guide directs users to a public HTTPS domain but does not configure `gateway.controlUi.allowedOrigins`; this conflicts with current non-loopback Control UI hardening and can push operators toward dangerous fallback settings.
  - confidence: 0.88
