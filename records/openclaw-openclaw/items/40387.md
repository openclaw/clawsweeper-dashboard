## Summary

Keep open: the EasyRunner guide is not on current main and no canonical duplicate was found, but the PR still has blocking Control UI security/docs issues and lacks real EasyRunner deployment proof.

## What This Changes

The PR adds a new EasyRunner platform deployment guide and links it from `docs/docs.json` and `docs/platforms/index.md`.

## Best Possible Solution

Publish a revised guide only after it bootstraps durable gateway config, allowlists the public HTTPS origin, preserves normal Control UI device pairing or trusted-proxy auth, rebuilds the local image during updates, and includes real EasyRunner proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head and current main shows the guide omits the public `allowedOrigins` config, documents a break-glass device-auth bypass as required, and claims config auto-creation without a bootstrap step. No real EasyRunner deployment proof is present for the platform-specific recipe.

## Solution Assessment

No. The useful docs direction is plausible, but the current PR is not the best solution because it normalizes unsafe Control UI auth guidance and relies on unsupported automatic config creation instead of the documented Docker and Control UI setup model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but the setup never adds that full origin to `gateway.controlUi.allowedOrigins`. Current Control UI docs require explicit origins for non-loopback deployments, so users can end up with a deployed gateway whose public dashboard fails browser-origin checks or pushes them toward unsafe fallback flags.
  - confidence: 0.91
- **[P1] Do not require the device-auth bypass:** `docs/platforms/easyrunner.md:262-280`
  - body: This section presents `dangerouslyDisableDeviceAuth` as required for reverse-proxy access. Current security docs mark that flag as break-glass only because it disables Control UI device identity checks entirely, so publishing it as normal setup would unnecessarily weaken remote UI authentication.
  - confidence: 0.95
- **[P2] Bootstrap the config before claiming it exists:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide says the container automatically creates a durable config with `gateway.mode`, proxy settings, and plugin slots, but the Docker image only starts the gateway with `--allow-unconfigured`, which does not write or repair config. Add explicit pre-start config creation or a mounted config example before relying on those values.
  - confidence: 0.89
- **[P2] Rebuild the local image during updates:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls new source and redeploys, but this guide runs `localhost/openclaw:latest`. Without rebuilding that image before `er app deploy`, users can keep running the previously built OpenClaw image after an update.
  - confidence: 0.86
