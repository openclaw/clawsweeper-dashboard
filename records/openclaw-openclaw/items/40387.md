## Summary

Keep open: the EasyRunner guide is useful in scope, but the current docs patch would publish incorrect and security-sensitive gateway deployment guidance around Control UI origins, device auth, startup config, updates, and memory-core.

## What This Changes

The PR adds an EasyRunner platform deployment guide and links it from the docs navigation and platforms index.

## Best Possible Solution

Publish a revised EasyRunner guide only after it creates pre-start gateway config with explicit public allowedOrigins, avoids device-auth bypass, preserves memory-core defaults, rebuilds images during updates, and has platform/docs approval with deployment proof.

## Reproduction Assessment

Yes for the review findings: PR-head docs plus current gateway source, tests, and docs show the missing allowedOrigins, unsupported automatic config claim, dangerous device-auth guidance, stale update flow, and memory-core mismatch. I did not run a live EasyRunner deployment.

## Solution Assessment

No. The guide's scope is useful, but this patch is not the best solution until it follows the current gateway security, Docker/container setup, and bundled plugin contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Add the public Control UI origin to setup:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to https://your-domain.com, but it never configures that origin in gateway.controlUi.allowedOrigins. Current non-loopback Control UI startup requires an explicit public browser-origin allowlist, so the deployed UI can fail to connect until the guide creates config with the exact HTTPS origin before startup.
  - confidence: 0.94
- **[P1] Remove the device-auth bypass requirement:** `docs/platforms/easyrunner.md:262-280`
  - body: The troubleshooting and security sections say dangerouslyDisableDeviceAuth is required behind a reverse proxy. Current security docs reserve that setting for break-glass debugging because it disables Control UI device identity checks, so publishing this as production guidance would weaken gateway auth.
  - confidence: 0.96
- **[P2] Create gateway config before first start:** `docs/platforms/easyrunner.md:108-113`
  - body: This section says the container automatically creates config with gateway.mode, trusted proxies, device-auth bypass, and memory settings. Current gateway startup blocks missing or incomplete config instead of repairing it, and the Docker setup path writes config before first start, so readers need an explicit mounted config or onboarding step before deployment.
  - confidence: 0.91
- **[P2] Rebuild the local image before redeploying:** `docs/platforms/easyrunner.md:203-208`
  - body: The update flow pulls new source and redeploys, but the compose file runs localhost/openclaw:latest. Without rebuilding that local image before er app deploy, EasyRunner can keep launching the previously built OpenClaw version.
  - confidence: 0.87
- **[P2] Remove the memory-core workaround:** `docs/platforms/easyrunner.md:236-248`
  - body: The troubleshooting section says the default memory plugin is unavailable in the container and tells users to disable memory. Current docs and slots code make memory-core an included plugin and the default memory slot, so this workaround would disable the default memory plugin instead of fixing a real container issue.
  - confidence: 0.84
