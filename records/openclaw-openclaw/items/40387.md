## Summary

Keep this PR open: current main does not already include EasyRunner docs, and the PR is a useful draft, but the current guide still conflicts with the supported Control UI security/config model and would leave users with a broken or unsafe deployment recipe.

## What This Changes

The PR adds an EasyRunner platform deployment guide plus docs navigation and platforms-index entries.

## Best Possible Solution

Publish a revised EasyRunner guide that creates durable gateway config before startup, allowlists the public HTTPS origin, preserves normal device pairing or trusted-proxy auth, and rebuilds the local image before redeploy.

## Reproduction Assessment

Yes. The defects are reproducible by source inspection of PR head 3b8538925178ff995278620c85b39a199e3b2797 against current main: the guide's public URL, automatic-config claim, and device-auth instructions contradict current startup and Control UI docs.

## Solution Assessment

No. The PR is a useful docs draft, but it should be revised around the supported Docker and Control UI security model before merge instead of documenting a break-glass bypass as normal setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but the compose/config steps never add that origin to `gateway.controlUi.allowedOrigins`. Current startup rejects non-loopback Control UI deployments without explicit origins, so this recipe can fail before the dashboard is usable.
  - confidence: 0.92
- **[P1] Remove the device-auth bypass from normal setup:** `docs/platforms/easyrunner.md:280`
  - body: This guide presents `gateway.controlUi.dangerouslyDisableDeviceAuth` as required for reverse proxies. Current Control UI docs classify that flag as a severe break-glass downgrade, so publishing it as standard setup unnecessarily weakens remote UI authentication.
  - confidence: 0.95
- **[P2] Replace the automatic config claim with bootstrap steps:** `docs/platforms/easyrunner.md:108-113`
  - body: The Docker image default only runs `gateway --allow-unconfigured`, and that flag does not write or repair config. The guide should add an explicit pre-start config creation or mounted config example before claiming `gateway.mode`, trusted proxies, device auth, or plugin slots are created automatically.
  - confidence: 0.89
- **[P2] Rebuild the local image during updates:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls source and redeploys, but the app runs the local `localhost/openclaw:latest` image. Without rebuilding that image before `er app deploy`, users can keep running the previous OpenClaw build after an update.
  - confidence: 0.86
