## Summary

Keep open; the PR is a useful docs contribution, but the proposed EasyRunner guide still conflicts with current gateway startup, Control UI origin, device-auth, update, and memory plugin contracts.

## What This Changes

The PR adds an EasyRunner platform deployment guide and links it from the docs navigation and platform index.

## Best Possible Solution

Publish a revised EasyRunner guide only after it creates and validates pre-start gateway config with explicit allowedOrigins, avoids device-auth bypass, keeps the default memory-core contract, rebuilds images during updates, and gets docs/platform approval.

## Reproduction Assessment

Yes for the review findings: PR-head docs plus current gateway source, tests, and docs show the missing allowedOrigins, unsupported automatic config claim, dangerous device-auth guidance, stale update flow, and memory-core mismatch. I did not run a live EasyRunner deployment.

## Solution Assessment

No. The guide's scope is useful, but the proposed content is not the narrowest maintainable solution until it follows current gateway security, Docker/container, and bundled plugin contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Add the public Control UI origin to setup:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but it never configures that origin in `gateway.controlUi.allowedOrigins`. Current container/non-loopback Control UI startup rejects missing public origins, so the deployed UI can fail to connect until the guide creates config with the exact HTTPS origin before startup.
  - confidence: 0.94
- **[P1] Remove the device-auth bypass requirement:** `docs/platforms/easyrunner.md:262-280`
  - body: The troubleshooting and security sections say `dangerouslyDisableDeviceAuth` is required behind a reverse proxy. Current security docs reserve that setting for break-glass debugging because it disables Control UI device identity checks, so publishing this as production guidance would weaken gateway auth.
  - confidence: 0.96
- **[P2] Create gateway config before first start:** `docs/platforms/easyrunner.md:108-113`
  - body: This section says the container automatically creates config with `gateway.mode`, trusted proxies, device-auth bypass, and memory settings. Current gateway startup blocks missing or incomplete config instead of repairing it, and the Docker CMD does not write EasyRunner config, so readers need an explicit mounted config or onboarding step before deployment.
  - confidence: 0.91
- **[P2] Rebuild the local image before redeploying:** `docs/platforms/easyrunner.md:203-208`
  - body: The update flow pulls source and redeploys, but the compose file runs `localhost/openclaw:latest`. Without rebuilding that image before `er app deploy`, EasyRunner can keep launching the previously built OpenClaw version.
  - confidence: 0.87
- **[P2] Remove the memory-core workaround:** `docs/platforms/easyrunner.md:236-248`
  - body: The troubleshooting section says the default `memory-core` plugin is unavailable in the container and tells users to disable memory. Current docs and slots code make `memory-core` an included plugin and the default memory slot, so this workaround would disable the default memory plugin instead of fixing a real container issue.
  - confidence: 0.84
