## Summary

This external docs PR should stay open for revision rather than cleanup close: the EasyRunner guide is not present on current main, but the proposed instructions conflict with the current Gateway and Control UI security model and still lack real deployment proof.

## What This Changes

The PR adds a new EasyRunner platform deployment guide and links it from the docs navigation and platform index.

## Best Possible Solution

Publish a revised EasyRunner guide only after it bootstraps durable gateway config before startup, allowlists the public HTTPS origin, preserves normal Control UI device pairing or trusted-proxy auth, and includes real deployment proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR's public HTTPS Control UI flow omits the required allowedOrigins config and normalizes a break-glass auth bypass. No real EasyRunner deployment proof is present for the platform-specific pieces.

## Solution Assessment

No. The useful docs direction is plausible, but the current PR is not the best solution because it documents an unsafe bypass and a claimed automatic config write instead of following the supported Docker and Control UI security model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but the compose/config steps never add that origin to `gateway.controlUi.allowedOrigins`. Current startup rejects non-loopback Control UI deployments without explicit origins, so this recipe can fail before the dashboard is usable.
  - confidence: 0.92
- **[P1] Remove the device-auth bypass from normal setup:** `docs/platforms/easyrunner.md:262-280`
  - body: The guide presents `dangerouslyDisableDeviceAuth` as required for reverse proxies. Current Control UI docs classify that flag as a severe break-glass downgrade, so publishing it as standard setup unnecessarily weakens remote UI authentication.
  - confidence: 0.95
- **[P2] Replace the automatic config claim with bootstrap steps:** `docs/platforms/easyrunner.md:108-113`
  - body: The Docker image default only runs `gateway --allow-unconfigured`, and that flag does not write or repair config. The guide should add an explicit pre-start config creation or mounted config example before claiming gateway mode, trusted proxies, device auth, or plugin slots are created automatically.
  - confidence: 0.89
- **[P2] Rebuild the local image during updates:** `docs/platforms/easyrunner.md:200-208`
  - body: The update flow pulls source and redeploys, but the app runs the local `localhost/openclaw:latest` image. Without rebuilding that image before `er app deploy`, users can keep running the previous OpenClaw build after an update.
  - confidence: 0.86
