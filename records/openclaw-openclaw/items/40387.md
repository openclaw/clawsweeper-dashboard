## Summary

Keep open: the EasyRunner guide is not on current main and is not superseded, but the PR still documents an unsafe and currently incompatible remote Control UI setup. The earlier review context was partly addressed, but the remaining Control UI origin/config/device-auth issues are real merge blockers, and the external PR has no after-fix EasyRunner proof.

## What This Changes

The PR adds a new EasyRunner platform deployment guide and wires it into the docs navigation and platform index.

## Best Possible Solution

Publish a revised EasyRunner guide that bootstraps durable gateway config before startup, allowlists the public HTTPS origin, preserves normal Control UI device pairing or trusted-proxy auth, and includes real deployment proof.

## Reproduction Assessment

Yes, source-reproducible. PR head instructs users to open a public HTTPS Control UI and disable device auth, while current main requires explicit allowedOrigins for non-loopback Control UI and documents device-auth disable as break-glass only.

## Solution Assessment

No. The docs draft is useful, but the maintainable solution is to revise it around the supported Docker and Control UI security model rather than merge instructions that rely on a dangerous bypass and a claimed automatic config write.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but the compose/config steps never add that origin to `gateway.controlUi.allowedOrigins`. Current startup rejects non-loopback Control UI deployments without explicit origins, so this recipe can fail before the dashboard is usable.
  - confidence: 0.92
- **[P1] Remove the device-auth bypass from normal setup:** `docs/platforms/easyrunner.md:262-274`
  - body: The guide presents `dangerouslyDisableDeviceAuth` as required for reverse proxies. Current Control UI docs classify that flag as a severe break-glass downgrade, so publishing it as standard setup unnecessarily weakens remote UI authentication.
  - confidence: 0.95
- **[P2] Replace the automatic config claim with bootstrap steps:** `docs/platforms/easyrunner.md:108-113`
  - body: The Docker image default only runs `gateway --allow-unconfigured`, and that flag does not write or repair config. The guide should add an explicit pre-start config creation or mounted config example before claiming gateway mode, trusted proxies, device auth, or plugin slots are created automatically.
  - confidence: 0.89
- **[P2] Rebuild the local image during updates:** `docs/platforms/easyrunner.md:203-208`
  - body: The update flow pulls source and redeploys, but the app runs the local `localhost/openclaw:latest` image. Without rebuilding that image before `er app deploy`, users can keep running the previous OpenClaw build after an update.
  - confidence: 0.86
