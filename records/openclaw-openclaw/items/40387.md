## Summary

Keep open: current head still documents an unsafe and partly non-working deployment path, including public Control UI access without an explicit origin allowlist, production device-auth bypass guidance, config creation claims that the container startup path does not provide, stale image update steps, and incorrect memory-core troubleshooting.

## What This Changes

The PR adds an EasyRunner platform deployment page and links it from the docs navigation and platform index.

## Best Possible Solution

Publish only a revised guide that matches current Docker and gateway security contracts: pre-start config with explicit public allowedOrigins, no device-auth bypass, correct memory-core guidance, image rebuilds during updates, and docs/product approval.

## Reproduction Assessment

Yes for the PR review findings: PR-head docs and current-main source show the origin allowlist, device-auth, config startup, update, and memory-core mismatches. I did not run a live EasyRunner deployment.

## Solution Assessment

No. The proposed guide is useful in scope, but it is not merge-ready until it follows the current gateway security, Docker setup, and bundled plugin contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but never adds that origin to `gateway.controlUi.allowedOrigins`. Current WebSocket handshakes reject non-allowlisted browser origins, and startup seeding only covers local origins, so the public EasyRunner URL will fail until the guide sets the exact public origin before first start.
  - confidence: 0.94
- **[P1] Remove the device-auth bypass guidance:** `docs/platforms/easyrunner.md:280`
  - body: The guide says `dangerouslyDisableDeviceAuth` is required behind a reverse proxy, but current security docs reserve that setting for break-glass debugging because it disables Control UI device identity checks. Production EasyRunner docs should use HTTPS plus proper allowed origins/auth instead of requiring this downgrade.
  - confidence: 0.96
- **[P2] Create gateway config before describing it:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide says first start automatically creates durable gateway config with mode, trusted proxies, device-auth bypass, and memory settings. The image default starts with `gateway --allow-unconfigured`, which bypasses the startup guard and does not repair config, so readers need an explicit pre-start onboard/config step or a mounted config example.
  - confidence: 0.91
- **[P2] Rebuild the image before redeploying updates:** `docs/platforms/easyrunner.md:203-208`
  - body: The update flow pulls new source and redeploys but does not rebuild `localhost/openclaw:latest`. EasyRunner can keep launching the previously built image, so the maintenance steps need a `podman build -t localhost/openclaw:latest .` step before `er app deploy`.
  - confidence: 0.87
- **[P2] Remove the memory-core workaround:** `docs/platforms/easyrunner.md:236-248`
  - body: The troubleshooting section says the default `memory-core` plugin is not available in the container and tells users to disable memory. Current plugin docs list `memory-core` as included in OpenClaw and the default memory slot, so this would disable the default memory plugin instead of documenting a real container fix.
  - confidence: 0.84
