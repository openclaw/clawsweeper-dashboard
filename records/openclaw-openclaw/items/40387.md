## Summary

Keep open: the EasyRunner guide is not implemented on current main, but this PR still has blocking docs correctness/security issues and no after-fix deployment proof.

## What This Changes

The PR adds an EasyRunner platform deployment guide and links it from the docs navigation and platform index.

## Best Possible Solution

Publish an EasyRunner guide only after it uses explicit pre-start gateway config, allowlists the public HTTPS origin, preserves normal Control UI device identity or uses trusted-proxy auth correctly, rebuilds the local image during updates, and includes redacted EasyRunner proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head and current main shows the missing public origin, unsafe device-auth guidance, unsupported config auto-creation claim, and stale update flow. No live EasyRunner deployment reproduction is present.

## Solution Assessment

No. The docs direction may be useful, but the current patch is not the best solution because it normalizes an unsafe Control UI auth bypass and relies on unsupported automatic config creation instead of the documented gateway setup model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com` but never configures `gateway.controlUi.allowedOrigins`. Current non-loopback/container startup requires explicit full origins, so this deployment can fail startup or browser-origin checks.
  - confidence: 0.92
- **[P1] Do not require the device-auth bypass:** `docs/platforms/easyrunner.md:262-280`
  - body: The troubleshooting and security sections say `dangerouslyDisableDeviceAuth` is required behind a reverse proxy. Current security docs treat that flag as a break-glass severe downgrade, so users would unnecessarily disable Control UI device identity checks.
  - confidence: 0.95
- **[P2] Create gateway config before relying on it:** `docs/platforms/easyrunner.md:108-113`
  - body: This says first start automatically creates durable config with `gateway.mode`, proxy settings, and plugin slots, but the Docker image only starts `gateway --allow-unconfigured`; that flag does not write or repair config. Add an explicit pre-start config step or mounted config example.
  - confidence: 0.9
- **[P2] Rebuild the image before redeploying updates:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls source and redeploys but never rebuilds `localhost/openclaw:latest`, so EasyRunner can keep running the previously built image after an update.
  - confidence: 0.87
