## Summary

Keep open: current main still has no EasyRunner platform guide, but this PR is not merge-ready because the guide conflicts with current Control UI security/config contracts and lacks real EasyRunner deployment proof.

## What This Changes

The PR adds an EasyRunner platform deployment guide and links it from the docs navigation and platforms index.

## Best Possible Solution

Publish a revised guide only after it bootstraps durable gateway config, allowlists the public HTTPS origin, preserves normal Control UI device pairing or trusted-proxy auth, rebuilds the local image during updates, and includes real EasyRunner proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head and current main shows the missing origin, unsafe device-auth guidance, unsupported config auto-creation claim, and stale update flow. No live EasyRunner deployment reproduction is present.

## Solution Assessment

No. The docs direction is plausible, but the current patch is not the best solution because it normalizes unsafe Control UI auth guidance and relies on unsupported automatic config creation instead of the documented Docker and Control UI setup model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but never adds that full origin to `gateway.controlUi.allowedOrigins`. Current non-loopback Control UI deployments require explicit full origins, so the deployed dashboard can fail browser-origin checks or push operators toward unsafe fallback flags.
  - confidence: 0.91
- **[P1] Stop requiring the device-auth bypass:** `docs/platforms/easyrunner.md:262-280`
  - body: This section presents `dangerouslyDisableDeviceAuth` as required for reverse-proxy access. Current security docs mark that flag as break-glass only because it disables Control UI device identity checks entirely, so this would unnecessarily weaken remote UI authentication for readers following the guide.
  - confidence: 0.95
- **[P2] Create config before relying on it:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide says the container automatically creates a durable config with `gateway.mode`, proxy settings, and plugin slots, but the Docker image only starts the gateway with `--allow-unconfigured`, which does not write or repair config. Add an explicit pre-start config step or mounted config example before relying on those values.
  - confidence: 0.89
- **[P2] Rebuild the local image before redeploying:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls new source and redeploys, but the guide runs `localhost/openclaw:latest`. Without rebuilding that image before `er app deploy`, users can keep running the previously built OpenClaw image after an update.
  - confidence: 0.86
