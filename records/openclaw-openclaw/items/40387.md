## Summary

Keep open: current main does not contain an EasyRunner guide, but this branch still needs docs and security corrections before it can merge because it conflicts with current gateway reverse-proxy and container setup contracts.

## What This Changes

Adds a new EasyRunner platform deployment guide and links it from docs navigation and the platforms index.

## Best Possible Solution

A publishable guide should match current container gateway setup: pre-start config, explicit public allowedOrigins, no device-auth bypass, update-time image rebuilds, and verified EasyRunner deployment proof.

## Reproduction Assessment

Yes for the PR review findings: source inspection of the PR head and current main shows the public-origin, device-auth, pre-start config, update rebuild, and memory-core mismatches. I did not run a live EasyRunner deployment.

## Solution Assessment

No. The proposed guide is useful in scope, but it is not the best current solution until it is revised to follow the gateway security and Docker setup contracts already documented on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but it never adds that origin to `gateway.controlUi.allowedOrigins`. Current gateway startup only seeds localhost origins, so browser sessions from the public EasyRunner domain will fail the non-loopback Control UI origin check.
  - confidence: 0.93
- **[P1] Remove the device-auth bypass guidance:** `docs/platforms/easyrunner.md:280`
  - body: This recommends `dangerouslyDisableDeviceAuth` for reverse-proxy access, but current security docs reserve that setting for break-glass debugging because it disables Control UI device identity checks. Production EasyRunner docs should use token/password or trusted-proxy setup instead of requiring this downgrade.
  - confidence: 0.96
- **[P2] Create gateway config before describing it:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide says first start automatically creates durable gateway config with mode, trusted proxies, device-auth bypass, and memory settings. The image command uses `gateway --allow-unconfigured`, which bypasses the guard and does not write that config, so readers need an explicit pre-start onboard/config step or a mounted config example.
  - confidence: 0.9
- **[P2] Rebuild the image during updates:** `docs/platforms/easyrunner.md:203-208`
  - body: The update flow pulls source and redeploys but never rebuilds `localhost/openclaw:latest`. EasyRunner can keep launching the previously built image, so the maintenance steps need a `podman build -t localhost/openclaw:latest .` step before `er app deploy`.
  - confidence: 0.87
- **[P2] Remove the memory-core workaround:** `docs/platforms/easyrunner.md:236-248`
  - body: The troubleshooting section says the default `memory-core` plugin is not available in the container and tells users to disable memory. Current plugin docs list memory-core as included in OpenClaw, so this would disable the default memory plugin instead of documenting a real container fix.
  - confidence: 0.83
