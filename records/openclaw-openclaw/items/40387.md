## Summary

Keep open. Current main still lacks an EasyRunner deployment guide, but this PR head is dirty and still conflicts with current remote Control UI security requirements by omitting the public origin allowlist and normalizing `dangerouslyDisableDeviceAuth`.

## What This Changes

The PR adds an EasyRunner deployment guide under `docs/platforms` and registers it in docs navigation and the platforms index.

## Best Possible Solution

Replace or rebase the guide into the current install/hosting docs structure with an explicit bootstrap config flow, exact public `allowedOrigins`, normal device auth or trusted-proxy auth, and image rebuild steps.

## Reproduction Assessment

Yes. The defects are reproducible by source inspection of the PR head against current main: the guide exposes a public HTTPS origin without `allowedOrigins`, documents a device-auth bypass as required, and claims config is created by startup even though `--allow-unconfigured` does not repair config.

## Solution Assessment

No. The PR is useful in scope, but this version is not the best solution because it publishes unsafe remote Control UI guidance and is based on stale docs navigation; the safer path is a revised guide aligned with Docker, Control UI, and security docs.

## Review Findings

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but it never adds that full origin to `gateway.controlUi.allowedOrigins`. Current runtime and docs require explicit non-loopback browser origins, so the public Control UI can fail to connect after deployment or push users toward dangerous fallback settings.
  - confidence: 0.92
- **[P1] Stop requiring the device-auth bypass:** `docs/platforms/easyrunner.md:280`
  - body: This documents `gateway.controlUi.dangerouslyDisableDeviceAuth` as required for reverse proxies. Current security docs treat that flag as break-glass only because it disables Control UI device identity checks, so this would normalize a severe remote-auth downgrade.
  - confidence: 0.95
- **[P2] Add an explicit bootstrap config step:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide says the container automatically creates `gateway.mode`, trusted-proxy, device-auth, and plugin-slot settings, but the image default only runs the gateway with `--allow-unconfigured`, which explicitly does not repair config. Add a real onboarding/config creation step or mounted config example before relying on those settings.
  - confidence: 0.88
- **[P2] Rebuild the local image during updates:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls new source and redeploys EasyRunner, but the app runs `localhost/openclaw:latest`. Without rebuilding that local Podman image in the update commands, redeploy can keep serving the previous OpenClaw build.
  - confidence: 0.87
- **[P2] Register the guide in the current hosting nav:** `docs/docs.json:1143`
  - body: The PR adds `platforms/easyrunner` to the stale Platforms overview block, while current main has moved hosting/deployment pages under Install > Hosting. Rebase and register the guide in the current docs navigation so it appears with the rest of the deployment docs.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The docs-only PR introduces security-sensitive remote Control UI guidance that disables device identity checks and omits the public origin allowlist.

Concerns:

- **[high] Device-auth bypass documented as required:** `docs/platforms/easyrunner.md:280`
  - body: The guide presents `gateway.controlUi.dangerouslyDisableDeviceAuth=true` as normal reverse-proxy setup even though current docs and audit checks classify it as a critical break-glass setting. Users following it would disable Control UI device identity checks for remote access.

  - body: The guide tells users to open a public HTTPS domain without configuring `gateway.controlUi.allowedOrigins` for that origin. This conflicts with current origin hardening and can push operators toward unsafe fallback flags.
