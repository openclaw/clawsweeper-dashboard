## Summary

This PR is useful as a platform-docs draft, but it is not ready to merge because the guide still conflicts with the current Control UI security model and gives incomplete deployment/update steps.

## What This Changes

The PR adds an EasyRunner deployment guide and wires it into the docs sidebar and platforms index.

## Best Possible Solution

Publish a revised guide that creates durable gateway config with the public allowedOrigins value, preserves normal device auth or a documented trusted-proxy model, and rebuilds the local Podman image before redeploying updates.

## Reproduction Assessment

Yes. The defects are reproducible by source inspection of the PR head against current main: the guide points users at a public HTTPS domain without configuring that origin, claims config is auto-created, and documents a break-glass device-auth bypass as required setup.

## Solution Assessment

No. The PR is a useful draft, but the maintainable solution is to align the EasyRunner recipe with the existing Docker/Control UI security contract instead of relying on unsafe bypass guidance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but it never adds that origin to `gateway.controlUi.allowedOrigins`. Current startup/runtime rejects non-loopback Control UI origins unless they are explicitly configured, so the public UI can fail to connect after deploy.
  - confidence: 0.93
- **[P1] Remove required device-auth bypass guidance:** `docs/platforms/easyrunner.md:262-274`
  - body: This section says `dangerouslyDisableDeviceAuth` is required for reverse-proxy access. Current docs treat that flag as break-glass only because it disables Control UI device identity checks, so publishing it as normal setup weakens remote UI authentication.
  - confidence: 0.95
- **[P2] Replace the automatic config claim with bootstrap steps:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide says first start creates `gateway.mode`, trusted proxies, device-auth bypass, and plugin-slot settings, but the container command only starts the gateway with `--allow-unconfigured`. Add an explicit config creation step or mounted config example before relying on those settings.
  - confidence: 0.9
- **[P2] Rebuild the local image before redeploying:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls new source and runs `er app deploy`, but this app runs the local `localhost/openclaw:latest` image. Without a rebuild step, redeploy can keep launching the previous OpenClaw build.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The docs normalize a device-auth bypass and omit the public origin allowlist required by current Control UI hardening.

Concerns:

- **[high] Device-auth bypass documented as required:** `docs/platforms/easyrunner.md:262`
  - body: The guide instructs users to set `dangerouslyDisableDeviceAuth` for reverse-proxy access, but current OpenClaw docs label it as a severe break-glass downgrade. Users following the guide would unnecessarily disable Control UI device identity checks for remote access.
  - confidence: 0.95
- **[medium] Public Control UI origin not allowlisted:** `docs/platforms/easyrunner.md:102`
  - body: The guide directs users to `https://your-domain.com` without configuring that origin in `gateway.controlUi.allowedOrigins`. This conflicts with current origin hardening and can push operators toward unsafe fallback settings.
  - confidence: 0.93
