## Summary

This PR should stay open: the EasyRunner guide is not implemented on current main, but the proposed docs still conflict with current gateway security and Docker setup contracts in ways that would produce broken or weakened public deployments.

## What This Changes

The PR adds a new EasyRunner platform deployment guide and links it from docs navigation and the platforms index.

## Best Possible Solution

Publish the guide only after it matches current Docker and gateway security contracts: explicit public allowedOrigins, pre-start config, no device-auth bypass, update-time image rebuilds, memory-core left enabled, and maintainer docs/product approval.

## Reproduction Assessment

Yes for the PR review findings: source inspection of the PR head and current main shows the public-origin, device-auth, pre-start config, update rebuild, and memory-core mismatches. I did not run a live EasyRunner deployment.

## Solution Assessment

No. The proposed guide is useful in scope, but it is not the best current solution until revised to follow the gateway security, Docker setup, and bundled plugin contracts already documented on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but it never adds that origin to `gateway.controlUi.allowedOrigins`. Current browser-origin checks reject Control UI WebSocket sessions whose Origin is not allowlisted, so the public EasyRunner URL will fail until the guide includes the exact public origin in pre-start config.
  - confidence: 0.94
- **[P1] Remove the device-auth bypass guidance:** `docs/platforms/easyrunner.md:280`
  - body: The guide recommends `dangerouslyDisableDeviceAuth` for reverse-proxy access, but current security docs reserve that setting for break-glass debugging because it disables Control UI device identity checks. Production EasyRunner docs should use HTTPS plus proper allowed origins/auth instead of requiring this downgrade.
  - confidence: 0.96
- **[P2] Create gateway config before describing it:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide says first start automatically creates durable gateway config with mode, trusted proxies, device-auth bypass, and memory settings. The image default starts with `gateway --allow-unconfigured`, which bypasses the startup guard and does not repair config, so readers need an explicit pre-start onboard/config step or a mounted config example.
  - confidence: 0.91
- **[P2] Rebuild the image before redeploying updates:** `docs/platforms/easyrunner.md:203-208`
  - body: The update flow pulls new source and redeploys but does not rebuild `localhost/openclaw:latest`. EasyRunner can keep launching the previously built image, so the maintenance steps need a `podman build -t localhost/openclaw:latest .` step before `er app deploy`.
  - confidence: 0.87
- **[P2] Remove the memory-core workaround:** `docs/platforms/easyrunner.md:236-248`
  - body: The troubleshooting section says the default `memory-core` plugin is not available in the container and tells users to disable memory. Current plugin docs list `memory-core` as included in OpenClaw and the default memory slot, so this would disable the default memory plugin instead of documenting a real container fix.
  - confidence: 0.84
