## Summary

Keep open. Current main still does not include an EasyRunner guide, but this PR head should not merge because its deployment flow conflicts with the current Control UI origin and device-auth security model and overstates what the container bootstrap creates.

## What This Changes

The PR adds an EasyRunner platform deployment guide, a docs sidebar entry, and a platforms index link.

## Best Possible Solution

Publish a revised EasyRunner guide that creates durable gateway config, sets the public allowedOrigins value, preserves normal device auth or trusted-proxy auth, and includes rebuild-before-redeploy steps.

## Reproduction Assessment

Yes. The defects are reproducible by source inspection of the PR head against current main: the guide points users at a public HTTPS domain without configuring that origin and presents a break-glass device-auth bypass as required setup.

## Solution Assessment

No. The PR is a useful docs draft, but this is not the best solution because the deployment recipe should follow the existing Docker and Control UI security model instead of relying on unsafe bypass guidance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but it never configures that origin in `gateway.controlUi.allowedOrigins`. Current Control UI docs and runtime require explicit non-loopback origins, so browser/WebSocket requests from the public domain can be rejected after deploy.
  - confidence: 0.91
- **[P1] Stop requiring the device-auth bypass:** `docs/platforms/easyrunner.md:280`
  - body: This section presents `gateway.controlUi.dangerouslyDisableDeviceAuth` as required for reverse-proxy access. Current docs treat that flag as break-glass only because it disables Control UI device identity checks, so publishing it as normal setup weakens remote UI authentication.
  - confidence: 0.95
- **[P2] Replace the automatic config claim with real bootstrap steps:** `docs/platforms/easyrunner.md:108-113`
  - body: The guide says first start automatically creates `gateway.mode`, proxy, device-auth, and plugin-slot settings, but the image command only starts `gateway --allow-unconfigured`. Add an explicit config creation step or mounted config example before relying on those settings.
  - confidence: 0.88
- **[P2] Rebuild the local image before redeploying:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls new source and runs `er app deploy`, but the app runs the local `localhost/openclaw:latest` image. Without a rebuild step here, redeploy can keep launching the previous OpenClaw build.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The docs-only PR introduces unsafe Control UI reverse-proxy guidance by normalizing a device-auth bypass and omitting the required public origin allowlist.

Concerns:

- **[high] Device-auth bypass documented as required:** `docs/platforms/easyrunner.md:280`
  - body: The guide says `dangerouslyDisableDeviceAuth` is required for reverse-proxy setups, but current OpenClaw docs label it as a severe break-glass downgrade. Users following the guide would unnecessarily disable browser device identity checks for remote Control UI access.
  - confidence: 0.95

  - body: The guide directs users to `https://your-domain.com` without configuring `gateway.controlUi.allowedOrigins` for that origin. This conflicts with current origin hardening and can push operators toward unsafe fallback settings.
