## Summary

Keep open: the EasyRunner guide is potentially useful, but the PR head still conflicts with the current Gateway and Control UI security model, is stale against the current docs navigation, and lacks real EasyRunner deployment proof.

## What This Changes

Adds an EasyRunner deployment guide and links it from the docs navigation and platforms index.

## Best Possible Solution

Publish a revised EasyRunner guide under the current Install > Hosting docs structure, with explicit gateway bootstrap config, public origin allowlisting, normal Control UI device/auth handling, image rebuild steps, and real deployment proof.

## Reproduction Assessment

Yes, source-reproducible. The PR head tells readers to open a public HTTPS Control UI and disable device auth, while current main refuses non-loopback Control UI without explicit allowedOrigins and documents that bypass as break-glass only.

## Solution Assessment

No. The useful docs draft should be revised around the supported Gateway bootstrap and Control UI security model, likely under the current Install > Hosting docs, before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but the compose/config steps never add that origin to `gateway.controlUi.allowedOrigins`. Current main refuses non-loopback Control UI startup without explicit origins, so this deployment can fail before the dashboard is usable.
  - confidence: 0.93
- **[P1] Remove the device-auth bypass from normal setup:** `docs/platforms/easyrunner.md:262-280`
  - body: The guide presents `dangerouslyDisableDeviceAuth` as required for reverse proxies. Current Control UI docs classify that flag as break-glass only and a severe security downgrade, so this would teach operators to weaken remote UI authentication unnecessarily.
  - confidence: 0.95
- **[P2] Replace the automatic config claim with bootstrap steps:** `docs/platforms/easyrunner.md:108-113`
  - body: The Docker image starts `gateway --allow-unconfigured`, and current gateway docs say that flag does not write or repair config. The guide should create or mount durable config explicitly before claiming gateway mode, trusted proxies, device auth, or plugin slots are created automatically.
  - confidence: 0.9
- **[P2] Rebuild the local image during updates:** `docs/platforms/easyrunner.md:203-208`
  - body: The update flow pulls source and redeploys, but the app runs the local `localhost/openclaw:latest` image. Without rebuilding that image before `er app deploy`, users can keep running the previous OpenClaw build after an update.
  - confidence: 0.86
- **[P3] Move the guide into the install hosting section:** `docs/platforms/easyrunner.md:1`
  - body: Current main organizes deployment guides under Install > Hosting as `docs/install/*`, while Platforms is now OS-focused. Rebase this as an install hosting guide instead of adding a new `docs/platforms/*` deployment page and stale Platforms navigation entry.
  - confidence: 0.82
