## Summary

Keep open. Current main still has no EasyRunner deployment page, and this PR is useful source work, but the current head is dirty against main and the guide still conflicts with current Control UI origin/device-auth security requirements plus bootstrap/update docs behavior.

## What This Changes

The PR adds `docs/platforms/easyrunner.md` and registers the EasyRunner page in docs navigation and the platforms index.

## Best Possible Solution

Rebase or replace the guide in the current docs structure with a security-correct EasyRunner flow that creates durable config, sets the public `allowedOrigins`, preserves normal device auth or trusted-proxy auth, and rebuilds the local image before redeploying.

## Reproduction Assessment

Yes. The defects are reproducible by source inspection of the PR head against current main: the PR tells users to open a public HTTPS origin and require `dangerouslyDisableDeviceAuth`, while current docs/runtime require explicit remote origins and treat that bypass as break-glass only.

## Solution Assessment

No. The PR is a useful draft, but this version is not the best solution because it publishes unsafe reverse-proxy guidance and skips durable bootstrap/update steps; the safer path is a revised docs recipe aligned with the existing Docker and Control UI security model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but it never configures that origin in `gateway.controlUi.allowedOrigins`. Current startup may seed localhost origins for container bootstraps, but browser requests from the public HTTPS domain still need to be explicitly allowed, or the Control UI connection can fail after deploy.
  - confidence: 0.9
- **[P1] Stop requiring the device-auth bypass:** `docs/platforms/easyrunner.md:280`
  - body: The guide presents `gateway.controlUi.dangerouslyDisableDeviceAuth` as required for reverse proxies. Current Control UI docs classify that flag as break-glass only because it disables device identity checks, so publishing it as normal setup weakens remote UI authentication.
  - confidence: 0.94
- **[P2] Add a real bootstrap config step:** `docs/platforms/easyrunner.md:108-113`
  - body: This section says the container automatically creates `gateway.mode`, proxy, device-auth, and plugin-slot settings, but the image command only starts `gateway --allow-unconfigured`, which current docs say does not write or repair config. Add an explicit onboarding/config creation step or mounted config example before relying on these settings.
  - confidence: 0.86
- **[P2] Rebuild the local image before redeploying:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls new source and runs `er app deploy`, but the app runs `localhost/openclaw:latest`. Without rebuilding that local Podman image first, redeploy can keep launching the previous OpenClaw build.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The docs-only PR introduces security-sensitive deployment guidance that normalizes disabling Control UI device authentication and omits the public origin allowlist.

Concerns:

- **[high] Device-auth bypass documented as required:** `docs/platforms/easyrunner.md:280`
  - body: The guide presents `gateway.controlUi.dangerouslyDisableDeviceAuth=true` as required for reverse-proxy setups even though current docs treat it as a severe break-glass downgrade. Users following the guide would disable Control UI device identity checks for remote access.
  - confidence: 0.94

  - body: The guide directs users to a public HTTPS domain without configuring `gateway.controlUi.allowedOrigins` for that origin. This conflicts with current Control UI origin hardening and can push operators toward unsafe fallback settings.
