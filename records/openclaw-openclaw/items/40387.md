## Summary

Keep this PR open: current main still has no EasyRunner guide, but the proposed guide has blocking deployment-security and correctness issues before it should publish.

## What This Changes

This PR adds an EasyRunner deployment guide and wires it into the docs sidebar and platform index.

## Best Possible Solution

Publish a revised EasyRunner guide with explicit durable gateway config, the public allowedOrigins value, normal device auth or trusted-proxy auth, and rebuild-before-redeploy steps.

## Reproduction Assessment

Yes. The defects are reproducible by source inspection of the PR head against current main: the guide omits the public Control UI origin, requires a break-glass device-auth bypass, overclaims automatic config creation, and skips image rebuild during update.

## Solution Assessment

No. The PR is a useful draft, but the best solution is a security-correct deployment recipe that follows the existing Docker and Control UI origin/auth model instead of documenting unsafe bypass settings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Configure the public Control UI origin:** `docs/platforms/easyrunner.md:102`
  - body: The guide sends users to `https://your-domain.com`, but the compose/config steps never add that origin to `gateway.controlUi.allowedOrigins`. Current startup only seeds localhost/127.0.0.1 origins, so WebSocket requests from the public HTTPS page can still fail origin checks.
  - confidence: 0.92
- **[P1] Stop requiring the device-auth bypass:** `docs/platforms/easyrunner.md:280`
  - body: This says `dangerouslyDisableDeviceAuth` is required for reverse proxies. Current security docs classify it as break-glass only; publishing it as normal setup disables Control UI device identity checks for remote deployments.
  - confidence: 0.95
- **[P2] Replace the automatic config claim with explicit bootstrap:** `docs/platforms/easyrunner.md:108-113`
  - body: The compose command only starts the gateway with `--allow-unconfigured`; current CLI help says that does not repair config, and startup does not create the listed gateway mode, trusted proxy, device-auth, or plugin-slot settings. Add real config creation or a mounted config example before relying on those fields.
  - confidence: 0.88
- **[P2] Rebuild the local image before redeploying:** `docs/platforms/easyrunner.md:207-208`
  - body: The update flow pulls source and runs `er app deploy`, but the app runs the previously built `localhost/openclaw:latest` image. Add a `podman build -t localhost/openclaw:latest .` step before deploy so updates take effect.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The docs-only PR introduces unsafe remote Control UI guidance by normalizing a device-auth bypass and omitting the required public origin allowlist.

Concerns:

- **[high] Device-auth bypass documented as required:** `docs/platforms/easyrunner.md:280`
  - body: The guide says `dangerouslyDisableDeviceAuth` is required for reverse-proxy setups, but current docs label it as a severe break-glass downgrade. Users following the guide would unnecessarily disable browser device identity checks for remote Control UI access.

  - body: The guide directs users to `https://your-domain.com` without configuring that origin in `gateway.controlUi.allowedOrigins`, conflicting with current origin hardening and likely pushing operators toward unsafe fallback settings.
