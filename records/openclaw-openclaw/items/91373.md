## Summary

Keep this PR open, but the current patch is too broad: it fixes the cron fallback symptom by changing shared agent/subagent string-model fallback semantics that current tests and docs treat as strict.

## What This Changes

The branch changes string agent model configs to return no fallback override, then updates agent-scope and isolated cron fallback tests so default model fallbacks are inherited.

## Best Possible Solution

Preserve shipped strict string-model semantics in the shared agent helpers and implement the linked isolated-cron inheritance only inside the cron fallback policy path, with regression coverage for both strict non-cron behavior and cron fallback inheritance.

## Reproduction Assessment

Yes, by source inspection: current main returns `[]` for a string agent model, `resolveCronFallbacksOverride` passes that through, and `resolveModelCandidateChain` treats `[]` as an explicit strict override. I did not run tests because this review is read-only.

## Solution Assessment

No, the current patch is not the best fix because it changes the shared helper used by normal agent and subagent flows. The maintainable fix is to keep strict shared semantics and handle the cron-specific fallthrough in the cron fallback resolver.
