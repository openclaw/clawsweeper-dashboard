## Summary

Keep open. Current main still serializes sender identity into model-facing inbound prompt text, and both the TUI/webchat and Telegram paths can still populate the reported identity fields; the related #47216 remains open and only covers the direct webchat/TUI subset.

## What This Changes

The issue requests stripping internal sender and routing metadata from model-facing prompt context for TUI/webchat and Telegram messages while preserving that identity for internal routing.

## Best Possible Solution

Land a scoped prompt-context change that keeps sender identity internal, suppresses transport-only webchat/TUI sender blocks, and privacy-minimizes Telegram identifiers with regression coverage for both paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by sending through TUI or Telegram, following the populated `Sender*` fields into `buildInboundUserContextPrefix`, and observing that `get-reply-run` prepends the resulting sender block to the model-facing prompt body.

## Solution Assessment

No. Current main is not the best solution because it still exposes sender metadata to the model; #47216 is a useful partial direct-webchat fix, but the maintainable fix should also settle the shared Telegram/privacy boundary.
