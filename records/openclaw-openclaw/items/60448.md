## Summary

Keep open. The operator UI/TUI transport-identity part has been fixed on main, but current main still carries Telegram sender id, name, and username into the model-facing inbound metadata path; changing that needs a maintainer privacy/attribution decision.

## What This Changes

The issue asks OpenClaw to stop exposing internal sender/routing JSON metadata to AI prompt context for TUI and Telegram turns.

## Best Possible Solution

Define a Telegram/shared inbound-context privacy policy, then minimize or internalize model-visible sender identity while preserving routing, authorization, reply, and tool contracts with regression coverage.

## Reproduction Assessment

Yes for the remaining Telegram side by source trace: Telegram sets sender id/name/username, shared turn context stores them, and prompt assembly renders them into model-facing inbound metadata. The original TUI transport-identity symptom is not reproducible on current main because operator UI clients are filtered before sender fields are set.

## Solution Assessment

No, current main is only a partial solution. The TUI/control UI producer-boundary fix is right, but the Telegram sender-privacy boundary needs an explicit maintainer decision before a narrow suppression or minimization patch is safe.
