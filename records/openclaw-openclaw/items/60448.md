## Summary

Keep open. Current main still builds a model-facing sender metadata prefix from inbound sender fields and prepends it before the user body; both the gateway/TUI path and Telegram path can still populate those fields with transport identity or real Telegram identity. The related open PR #47216 is only a partial direct-webchat/TUI fix and has not landed, so this issue still needs maintainer review for the shared privacy boundary.

## What This Changes

The issue requests filtering internal sender and routing metadata out of model-facing context for TUI and Telegram messages while keeping that identity available internally.

## Best Possible Solution

Land a scoped prompt-context change that separates internal sender/routing identity from model-visible text, suppresses transport-only TUI/webchat identities, and privacy-minimizes Telegram identifiers unless a channel explicitly opts into a safe prompt label. Regression coverage should exercise both direct webchat/TUI and Telegram direct-message paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: send through TUI or Telegram, follow the populated sender fields into `buildInboundUserContextPrefix`, and observe that `get-reply-run` prepends the resulting sender metadata block to the model-facing prompt body.

## Solution Assessment

No. Current main is not the best solution because it still exposes sender metadata to the model; #47216 is a reasonable partial fix for direct webchat/TUI, but the maintainable solution should address the shared prompt-context boundary and Telegram privacy behavior too.
