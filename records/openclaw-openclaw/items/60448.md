## Summary

Keep open. Current main has fixed the TUI/operator-webchat transport-identity leak, but Telegram still supplies sender name/id/username into the shared inbound context path and that context remains model-facing.

## What This Changes

The issue reports sender/routing metadata JSON being included in model-facing prompt context for TUI and Telegram messages.

## Best Possible Solution

Keep the merged operator UI producer-boundary fix, then define and implement a Telegram/shared inbound-context policy that keeps routing identity internal or privacy-minimized with regression coverage.

## Reproduction Assessment

Yes for the remaining Telegram side by source trace: Telegram fills sender fields, `buildInboundUserContextPrefix` serializes them, and prompt-prelude carries inbound context to the model path. No for the original TUI transport identity on current main, because operator UI clients are filtered before sender fields are set.

## Solution Assessment

No, current main is only a partial solution. The TUI/control UI fix is the right producer-boundary direction, but the remaining Telegram sender-privacy boundary needs an explicit maintainer policy before a narrow code change should remove or minimize model-visible identity fields.
