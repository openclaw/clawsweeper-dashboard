## Summary

Keep open. Current main and the latest release still make Telegram sender id/name/username model-visible in untrusted prompt context; the TUI transport-identity half is fixed, but the remaining privacy/security contract needs maintainer policy review.

## What This Changes

The issue asks OpenClaw to stop injecting internal routing and sender identity metadata into AI prompt context for TUI and Telegram turns.

## Best Possible Solution

Define one maintainer-approved inbound sender privacy and attribution contract, then minimize or pseudonymize model-visible Telegram/shared sender metadata while preserving routing, authorization, reply, and speaker-continuity requirements with focused tests.

## Reproduction Assessment

Yes. The remaining Telegram path is source-reproducible: Telegram supplies sender id/name/username, shared inbound context stores them, and the prompt builder renders them into model-visible user-role metadata.

## Solution Assessment

Unclear as a direct stripping patch. The TUI producer-boundary fix is already the right outcome, but the remaining Telegram/shared sender metadata needs a maintainer-approved privacy and attribution policy before implementation.
