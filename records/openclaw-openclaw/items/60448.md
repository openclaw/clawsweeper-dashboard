## Summary

Keep open. The TUI/operator-webchat transport identity leak was fixed by https://github.com/openclaw/openclaw/pull/78790, but current main still carries Telegram sender name/id/username into the model-facing sender metadata block, and no canonical open follow-up for that privacy boundary was found.

## What This Changes

The issue asks OpenClaw to keep internal sender/routing metadata, including TUI and Telegram sender JSON, out of AI-facing prompt context.

## Best Possible Solution

Keep the merged operator UI fix, then make an explicit maintainer privacy decision for Telegram/model-visible sender identity and land a scoped shared prompt-boundary change with regression coverage if minimization is desired.

## Reproduction Assessment

Yes for the remaining Telegram path: source trace shows Telegram populates sender fields, `buildInboundUserContextPrefix` renders them into `Sender (untrusted metadata)`, and `get-reply-run` carries that text into the agent prompt. The TUI/operator-webchat half no longer reproduces on current main because operator UI clients are filtered before sender fields are set.

## Solution Assessment

Unclear. Current main is the right direction for the TUI/operator UI leak, but the remaining Telegram behavior needs a maintainer privacy/product decision before an automated code change suppresses sender attribution.
