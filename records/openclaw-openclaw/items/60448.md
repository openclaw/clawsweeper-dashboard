## Summary

Keep open. Current main and v2026.5.19 fix the TUI/operator transport-identity half, but Telegram still passes sender id/name/username into shared inbound context and the prompt path still makes those values model-visible as untrusted metadata; the remaining change needs maintainer privacy/security policy rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to prevent internal sender/routing metadata JSON, including real names, usernames, and ids, from reaching the AI prompt context in TUI and Telegram workflows.

## Best Possible Solution

Define one maintainer-approved inbound sender privacy and identity contract, then minimize or pseudonymize model-visible Telegram/shared sender metadata while preserving routing, authorization, reply, and stable trusted-identity use cases with focused regression coverage.

## Reproduction Assessment

Yes. The remaining Telegram path is source-reproducible: Telegram supplies sender id/name/username, shared inbound context stores them, and the prompt/runtime context path makes them model-visible; the original TUI transport identity symptom is not reproducible from current source because operator UI clients are filtered before sender fields are set.

## Solution Assessment

Unclear as a direct stripping patch. The TUI producer-boundary fix is already the right outcome, but the remaining Telegram/shared sender metadata needs a maintainer-approved privacy and attribution policy before implementation.
