## Label Justifications

- `P1`: The issue is a concrete privacy/security-sensitive prompt-context leak involving real user identifiers in active TUI and Telegram workflows.
- `impact:security`: The issue concerns human-identifying sender names, usernames, and IDs being exposed to model-facing prompt context.

## Summary

Keep open. Current main fixed the TUI/operator UI transport-identity part, but Telegram still supplies sender id/name/username into the shared inbound context and the prompt builder still renders those values into model-visible untrusted metadata; the remaining fix needs a maintainer privacy/identity policy decision.

## What This Changes

The issue asks OpenClaw to prevent model-facing prompts from receiving internal sender/routing metadata such as TUI client IDs and Telegram user IDs, names, and usernames.

## Best Possible Solution

Define one maintainer-approved inbound sender privacy and identity contract, then minimize or pseudonymize model-visible Telegram/shared sender metadata while preserving routing, authorization, reply, and stable trusted-identity use cases with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible for the remaining Telegram path: Telegram supplies sender id/name/username, shared inbound context stores them, and buildInboundUserContextPrefix renders them into model-visible untrusted metadata. The original TUI transport identity symptom is not reproducible from current source because operator UI clients are filtered before sender fields are set.

## Solution Assessment

Unclear as a direct stripping patch: the TUI producer-boundary fix is already the right outcome, but the remaining Telegram/shared sender metadata needs a maintainer-approved privacy and attribution policy before implementation.
