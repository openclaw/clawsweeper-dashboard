## Summary

Keep open. Current main fixes the operator/TUI sender injection path, but Telegram sender identity still reaches model-facing current-turn context, and the remaining privacy/prompt-policy work is not merged.

## What This Changes

The issue asks OpenClaw to stop exposing sender identity and routing metadata in AI-facing prompt context for TUI/webchat and Telegram messages.

## Best Possible Solution

Settle a shared inbound metadata privacy policy that keeps transport-only identities out of prompts, privacy-minimizes Telegram sender fields, preserves necessary attribution/routing semantics, and updates prompt-builder plus sanitizer tests together.

## Reproduction Assessment

Yes, for the remaining Telegram portion by source inspection: Telegram still supplies SenderName/SenderId/SenderUsername, buildInboundUserContextPrefix still renders those into current-turn metadata, and runCliAgent receives that context. The TUI/operator subset no longer reproduces on current main because the gateway now suppresses operator UI sender fields.

## Solution Assessment

No. Current main is only a partial solution: PR https://github.com/openclaw/openclaw/pull/78790 fixes the TUI/operator producer boundary, while the Telegram privacy boundary remains open and overlaps unmerged compact-metadata work.
