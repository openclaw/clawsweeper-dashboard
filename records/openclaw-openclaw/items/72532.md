## Summary

Keep open: the patch is not safe as written because direct Feishu command paths keep `ctx.content` as the command probe and `CommandBody`, so preserving a leading bot `<at>` tag can hide slash commands. It also lacks inspectable real behavior proof and overlaps with the narrower open fix path at https://github.com/openclaw/openclaw/pull/72661.

## What This Changes

The PR makes Feishu `normalizeMentions` preserve the receiving bot's own `<at>` tag everywhere and updates Feishu tests/comments for that behavior.

## Best Possible Solution

Land one corrected Feishu plugin fix, preferably the narrower leading-addressing approach from https://github.com/openclaw/openclaw/pull/72661 or equivalent, with regression coverage for multi-bot self-mentions and direct/group mention-prefixed slash commands.

## Reproduction Assessment

Yes, source-level. With this PR, a direct Feishu message like `@_bot_1 /model` keeps a leading `<at>` tag in `ctx.content`, direct command probing and `CommandBody` keep that text, and core command normalization no longer sees a leading slash.

## Solution Assessment

No. Making `botStripId` a global no-op is broader than needed; the maintainable fix is to preserve inline self-mentions while keeping leading addressing self-mentions slash-clean, or to separate agent-facing mention context from command-probe text.
