## Summary

Keep open. The Feishu multi-bot bug is valid, but this PR is not mergeable as written because blanket-preserving the receiving bot mention can leave direct slash-command bodies prefixed with an `<at>` tag, the claimed live Feishu testing is not inspectable, and https://github.com/openclaw/openclaw/pull/72661 overlaps the same root cause with a narrower approach.

## What This Changes

The PR makes Feishu `normalizeMentions` ignore `botStripId` so all `<at>` tags, including the receiving bot's, remain in `ctx.content`, and updates related tests/comments to expect preserved self-mentions.

## Best Possible Solution

Land one canonical Feishu plugin fix that preserves semantic inline self-mentions while keeping leading mention-prefixed command bodies clean, with regression coverage and redacted real Feishu proof.

## Reproduction Assessment

Yes, at source level. With this PR, a direct Feishu `@_bot_1 /model` body keeps a leading `<at>` tag in `ctx.content`, direct command probing and `CommandBody` keep that text, and core direct command normalization no longer sees a leading slash.

## Solution Assessment

No. Making `botStripId` a global no-op is broader than needed; the maintainable direction is to preserve inline self-mentions while keeping addressing prefixes command-clean or separating command-probe text from agent-facing content.
