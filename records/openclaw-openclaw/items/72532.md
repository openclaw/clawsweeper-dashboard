## Summary

Keep open for revision: the Feishu multi-bot problem is real, but this PR’s global self-mention preservation regresses direct slash-command handling and still lacks inspectable real Feishu after-fix proof.

## What This Changes

The PR changes Feishu mention normalization to preserve the receiving bot’s own `<at>` tag everywhere, updates the parse comment, and changes strip-mention tests to expect preserved self-mentions.

## Best Possible Solution

Land one corrected Feishu plugin fix, preferably the narrower leading-addressing approach from https://github.com/openclaw/openclaw/pull/72661 or equivalent, with regression coverage for multi-bot self-mentions and direct/group mention-prefixed slash commands plus real Feishu proof.

## Reproduction Assessment

Yes, source-level. With this PR, a direct Feishu message like `@_bot_1 /model` keeps a leading `<at>` tag in `ctx.content`, direct command probing and `CommandBody` keep that text, and core direct command normalization no longer sees a leading slash.

## Solution Assessment

No, not as written. Making `botStripId` a global no-op is broader than needed; the maintainable fix is to preserve inline self-mentions while keeping leading addressing self-mentions slash-clean, or to separate agent-facing mention context from command-probe text.
