## Summary

Keep open: the PR targets a real Feishu NO_REPLY bug, but the current diff is too broad and would regress direct-message slash command handling. The same remaining work is already being tracked by the narrower open replacement path at https://github.com/openclaw/openclaw/pull/72661.

## What This Changes

The PR changes Feishu mention normalization to preserve the receiving bot's own `<at>` tag everywhere and updates related Feishu tests/comments to expect that behavior.

## Best Possible Solution

Land one corrected Feishu plugin fix, preferably the narrower leading-addressing approach from https://github.com/openclaw/openclaw/pull/72661 or equivalent, with regression coverage for multi-bot self-mentions and mention-prefixed slash commands.

## Reproduction Assessment

Yes, source-level. With this PR, a direct Feishu message like `@_bot_1 /model` keeps a leading `<at>` tag in `ctx.content`, direct command probing and `CommandBody` keep that text, and core command normalization no longer sees a leading slash.

## Solution Assessment

No. Making `botStripId` a global no-op is broader than needed; the maintainable fix is to preserve inline self-mentions while keeping leading addressing self-mentions slash-clean, or to separate agent-facing mention context from command probe text.
