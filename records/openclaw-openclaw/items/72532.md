## Summary

Keep open. The PR addresses a real Feishu NO_REPLY report, but the branch is broader than the safe fix: making `botStripId` a global no-op preserves leading self-mentions in direct command bodies and can break direct slash commands, while #72661 already tracks the narrower leading-prefix approach.

## What This Changes

The PR changes Feishu mention normalization to preserve the receiving bot's own `<at>` tag in all contexts and updates Feishu tests/comments to expect that behavior.

## Best Possible Solution

Land #72661 or equivalent: strip only leading Feishu bot self-mentions used as addressing prefixes, preserve inline self-mentions for multi-bot routing, and cover both command and NO_REPLY regressions with tests and changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: a direct Feishu `@Bot /model` message would retain a leading `<at>` tag in `CommandBody`, and direct core command parsing does not strip that tag before requiring a leading slash.

## Solution Assessment

No. Making `botStripId` a global no-op is broader than needed; the narrower maintainable fix is to preserve semantic inline self-mentions without changing leading addressing-prefix behavior used by command dispatch.
