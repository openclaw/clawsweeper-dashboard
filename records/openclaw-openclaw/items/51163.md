## Summary

Keep open: current main does not contain the proposed thread-context store, and this PR remains the canonical tracker for that remaining work, but the patch is not mergeable yet because the new interrupt path and context store still have routing, registry, persistence, and provider-normalization blockers.

## What This Changes

The PR adds a chat `/interrupt <message>` handler with tests plus a JSON-backed thread-context store wired into cron isolated-agent delivery and new-session prompt context.

## Best Possible Solution

Refresh and likely split the work: keep existing `/steer`/`sessions.steer` semantics for current interrupt-and-refine behavior, then land thread-context continuity with full delivery-context preservation, provider-normalized keys, serialized storage, registry/docs/changelog coverage, and focused channel tests.

## Reproduction Assessment

Yes. The blockers are reproducible by static contract checks against current main: explicit delivery does not inherit account context, internal delivered agent runs are rerouted or rejected, native commands are registry-driven, and the proposed store write is an unlocked read-modify-write.

## Solution Assessment

No. The direction is useful, but the patch is not the narrowest maintainable solution yet; interrupts should use or preserve the current session delivery abstractions, and the thread-context store needs routing-safe keys plus serialized writes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the interrupt delivery account:** `src/auto-reply/reply/commands-session-interrupt.ts:86-87`
  - body: This requeue sends an explicit `channel`/`to` target but omits the originating `accountId`. Current Gateway delivery only inherits the session account for implicit targets, so an interrupt from a non-default Slack, Telegram, or similar account can resume on the wrong account or fail delivery.
  - confidence: 0.88
- **[P1] Keep internal interrupts session-local:** `src/auto-reply/reply/commands-session-interrupt.ts:85`
  - body: The handler forces `deliver: true` even when the command originates from the internal/webchat surface. Current `agent` delivery either resolves that to a configured external channel or rejects it, after the active run has already been aborted, so webchat interrupts can misroute or fail instead of staying in-session.
  - confidence: 0.84
- **[P2] Register the command in the shared catalog:** `src/auto-reply/reply/commands-core.ts:203`
  - body: Adding the handler here makes text parsing recognize `/interrupt`, but `/commands` and native Discord, Telegram, and Slack command deployment come from the shared command registry. Without a registry entry, native-command users will not see or invoke the new command consistently.
  - confidence: 0.86
- **[P2] Serialize thread-context store updates:** `src/sessions/thread-context-store.ts:140-144`
  - body: `saveThreadContext` loads the whole JSON file, mutates it, and writes it back without a per-file lock. Overlapping cron or isolated-agent deliveries can both read the same snapshot and the later write can drop the earlier context entry.
  - confidence: 0.9
- **[P2] Normalize Telegram chat ids before saving context:** `src/cron/isolated-agent/run.ts:943`
  - body: For explicit Telegram topic delivery, target parsing stores the bare chat id in `resolvedDelivery.to`, while inbound replies look up context with `OriginatingTo` shaped like `telegram:<chatId>`. Saving the bare value means later replies in the same topic can miss the prior-context note.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Security review needs attention: the diff adds no dependency or supply-chain change, but the interrupt requeue can route redirected output through the wrong account or outside the internal session surface.

Concerns:

- **[medium] Interrupt requeue can use the wrong account:** `src/auto-reply/reply/commands-session-interrupt.ts:86`
  - body: The new interrupt path supplies explicit delivery routing without preserving `accountId`; on multi-account channels, redirected output can be sent from the default or wrong account, creating a delivery-integrity and confidentiality risk.
  - confidence: 0.84

  - body: For internal/webchat origins, `deliver: true` asks the Gateway to resolve an outbound delivery channel instead of keeping the replacement turn session-local, so private webchat instructions can be misrouted to a configured external target or fail after abort.
