Full review comments:

- **[P1] Preserve the source account and thread on requeue:** `src/auto-reply/reply/commands-session-interrupt.ts:82-90`
  - body: The requeued `agent` call sends explicit `channel` and `to` but drops the originating `accountId` and `threadId`. Current delivery planning only gets those values from request fields for this path, so multi-account or threaded interrupts can resume through the default account or parent conversation.
  - confidence: 0.91
- **[P1] Do not force delivery for internal webchat interrupts:** `src/auto-reply/reply/commands-session-interrupt.ts:86-88`
  - body: For webchat/internal chat, this can resolve the channel to the internal message channel while still setting `deliver: true`. The gateway rejects that delivered internal-channel request after the active run has already been aborted, leaving the redirect unfired.
  - confidence: 0.84
- **[P2] Register /interrupt in the command registry:** `src/auto-reply/reply/commands-core.ts:206`
  - body: The handler is added to the text-command pipeline, but the builtin command registry has no `/interrupt` entry. Slack and Discord native commands are generated from that registry, so native-command installations will not expose the new command.
  - confidence: 0.88
- **[P2] Serialize thread-context store writes:** `src/sessions/thread-context-store.ts:137-141`
  - body: `saveThreadContext` performs a read-modify-write without a per-file queue or lock. Overlapping cron deliveries can load the same JSON snapshot and the later write will drop the earlier context entry.
  - confidence: 0.86
- **[P2] Normalize Telegram chat keys before saving context:** `src/cron/isolated-agent/run.ts:942-946`
  - body: Cron saves `resolvedDelivery.to` as the chat key, but Telegram explicit-topic delivery strips the topic from `to` while inbound forum replies can include the `:topic:` target form. Those keys can miss each other, so prior-thread context is skipped for explicit Telegram thread deliveries.
  - confidence: 0.82
