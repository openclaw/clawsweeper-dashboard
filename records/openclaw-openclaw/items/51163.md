## Summary

Keep this PR open, but it is not mergeable as-is. The branch contains useful work, especially for the cross-session thread-context request, but the current diff still has delivery routing/security blockers, a storage race, a native-command registry gap, Telegram key mismatches, and no after-fix real behavior proof.

## What This Changes

Adds a chat `/interrupt <message>` command plus a JSON-backed thread-context store for cron/isolated-agent thread follow-ups, with tests and wiring in reply/cron paths.

## Best Possible Solution

Refresh and likely split the work: keep current `/steer`/`sessions.steer` semantics, then land thread-context continuity with routing-safe keys, serialized writes, registry/docs coverage, focused tests, and redacted real-channel proof.

## Reproduction Assessment

Yes. The blockers are source-reproducible from the PR head and current main contracts: explicit requeue loses account/thread context, internal delivery is not session-local, native commands are registry-driven, and the store update is an unlocked read-modify-write.

## Solution Assessment

No. The direction is useful, but this patch is not the narrowest maintainable solution yet; it should preserve existing delivery abstractions, serialize persistence, normalize provider keys, and decide command UX before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass the originating account when requeueing interrupts:** `src/auto-reply/reply/commands-session-interrupt.ts:86-88`
  - body: This `agent` call supplies explicit `channel`/`to` but omits `accountId`. Current delivery planning only inherits the saved session account for implicit targets, so interrupts from non-default Slack, Telegram, or similar accounts can resume on the wrong account or fail delivery.
  - confidence: 0.88
- **[P1] Keep internal interrupts session-local:** `src/auto-reply/reply/commands-session-interrupt.ts:86`
  - body: For webchat/internal origins this still sets `deliver: true`. Current Gateway handling treats delivered internal requests as external-delivery attempts or rejects them when no channel can be selected, so the replacement turn can misroute or fail after aborting the active run.
  - confidence: 0.84
- **[P1] Preserve the originating thread when requeueing interrupts:** `src/auto-reply/reply/commands-session-interrupt.ts:86-88`
  - body: The requeue forwards `channel` and `to` but not the originating `threadId`. For threaded Slack or Telegram conversations, current delivery resolution uses request thread data for explicit targets, so the redirected reply can leave the original thread/topic.
  - confidence: 0.84
- **[P2] Register `/interrupt` in the shared command catalog:** `src/auto-reply/reply/commands-core.ts:206`
  - body: Adding the handler makes text parsing recognize `/interrupt`, but `/commands` and native Discord, Telegram, and Slack command deployment are driven by the shared registry. Without a registry entry, native-command users will not see or invoke the new command consistently.
  - confidence: 0.86
- **[P2] Serialize thread-context store updates:** `src/sessions/thread-context-store.ts:137-141`
  - body: `saveThreadContext` loads the whole JSON file, mutates it, and writes it back without a per-file lock. Overlapping cron or isolated-agent deliveries can both read the same snapshot and the later write can drop the earlier context entry.
  - confidence: 0.9
- **[P2] Normalize chat ids before saving Telegram thread context:** `src/cron/isolated-agent/run.ts:945`
  - body: For explicit Telegram topic delivery, target parsing stores the bare chat id in `resolvedDelivery.to`, while inbound lookups use `OriginatingTo` values with the Telegram prefix and sometimes topic suffix. The saved key can therefore miss the later reply key, skipping the prior-context note.
  - confidence: 0.84
