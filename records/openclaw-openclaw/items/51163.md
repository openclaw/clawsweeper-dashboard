## Summary

Keep open. Current main does not contain the PR's feature, and the branch remains useful source work, but its latest head still has concrete routing, registry, and persistence blockers; the interrupt path is especially risky for multi-account and internal/webchat surfaces.

## What This Changes

This PR adds a chat `/interrupt <message>` handler that aborts and requeues a running agent turn, plus a JSON-backed thread-context store wired into cron delivery and new-session prompt construction.

## Best Possible Solution

Refresh and likely split the work: land `/interrupt <message>` through the current command registry and `sessions.steer` or an equivalent session-local steering seam, preserving account, thread, internal/webchat, and subagent semantics; then land the thread-context store with provider-normalized keys, serialized writes, TTL/cap pruning, and focused cron/channel tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The blockers reproduce by static contract checks against current main and the latest PR patch: explicit delivery loses account fallback, internal delivery is not session-local, native commands are registry-derived, and the store update path is an unlocked read-modify-write.

## Solution Assessment

Is this the best way to solve the issue? No. The feature direction is useful, but the proposed merge shape hand-rolls delivery behavior that current main already centralizes in session steering and delivery-target contracts; reusing those seams is the narrower maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the interrupt delivery account:** `src/auto-reply/reply/commands-session-interrupt.ts:84-90`
  - body: This requeue passes an explicit `channel`/`to` pair but omits `accountId`. Current delivery resolution only inherits the session account for implicit targets, so `/interrupt` from a non-default Slack, Telegram, or similar account can resume through the wrong account or fail delivery. Include the originating account id or use a session-steering path that preserves it.
  - confidence: 0.9
- **[P1] Keep internal interrupts session-local:** `src/auto-reply/reply/commands-session-interrupt.ts:84`
  - body: The handler always sets `deliver: true`, including when the command comes from webchat/internal surfaces. Current `agent` delivery either tries to select an external channel or rejects internal delivery after the run has already been aborted, so webchat interrupts can fail or route outside the active session. Gate delivery by surface or use `sessions.steer`.
  - confidence: 0.84
- **[P2] Register the command in the shared catalog:** `src/auto-reply/reply/commands-core.ts:203`
  - body: Adding the text handler alone leaves `/interrupt` out of `commands-registry.shared.ts`. Native command deployment and the generated `/commands` catalog are built from that registry, so supported native surfaces will not expose the new command.
  - confidence: 0.87
- **[P2] Serialize thread-context store updates:** `src/sessions/thread-context-store.ts:134-139`
  - body: `saveThreadContext` still loads the JSON store, mutates one entry, prunes, and writes it back without per-file serialization. Overlapping cron or isolated-agent deliveries can both read the same snapshot and the later write drops the earlier context.
  - confidence: 0.89
- **[P2] Normalize Telegram chat ids before saving context:** `src/cron/isolated-agent/run.ts:943`
  - body: For explicit Telegram delivery targets, `resolvedDelivery.to` is the bare chat id, while inbound replies look up `ctx.OriginatingTo` in `telegram:<chatId>` form. Saving the bare value here means explicit Telegram threaded cron replies will not find the prior-context record.
  - confidence: 0.85

## Security Review

Status: needs_attention

Summary: The diff adds no supply-chain risk, but the new interrupt delivery path can misroute redirected replies across account or internal-channel boundaries.

Concerns:

- **[medium] Interrupt requeue can use the wrong account:** `src/auto-reply/reply/commands-session-interrupt.ts:84`
  - body: The new `/interrupt` path requeues an agent run with explicit `channel` and `to` but no `accountId`. In multi-account channel setups this can send the redirected response through a default or wrong account, creating a delivery-integrity and confidentiality risk.
  - confidence: 0.86
- **[medium] Webchat interrupts can escape session-local routing:** `src/auto-reply/reply/commands-session-interrupt.ts:84`
  - body: Because the handler forces `deliver: true`, internal/webchat commands may trigger external delivery selection instead of staying inside the active web session. That is a concrete routing-scope concern for a command that first aborts the current run.
  - confidence: 0.78
