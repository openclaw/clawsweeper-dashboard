## Summary

Keep this PR open, but not mergeable as-is: the linked feature work is still useful, yet the branch is conflicting, lacks contributor real-behavior proof, and has source-reproducible routing, registry, persistence, and Telegram keying defects.

## What This Changes

The PR adds a chat `/interrupt <message>` handler plus a JSON-backed thread-context store for cron/isolated-agent follow-up replies in the same IM thread.

## Best Possible Solution

Refresh and likely split the work: keep current `/steer`/`sessions.steer` semantics, then land any remaining thread-context continuity through routing-safe keys, serialized persistence, registry/docs coverage, focused tests, and redacted real-channel proof.

## Reproduction Assessment

Yes. The merge blockers are source-reproducible from the PR head and current main contracts: explicit interrupt requeue loses account routing, internal delivery is not session-local, native commands are registry-driven, the store update is an unlocked read-modify-write, and Telegram explicit targets use a different chat-id shape than inbound context.

## Solution Assessment

No. The direction is useful, but this patch is not the narrowest maintainable solution yet; it should preserve existing delivery abstractions, serialize persistence, normalize provider keys, register the command surface, and settle the `/interrupt` versus `/steer` UX before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the originating account on interrupt requeues:** `src/auto-reply/reply/commands-session-interrupt.ts:86-90`
  - body: This requeue supplies an explicit `channel`/`to` pair but omits `accountId`. Current delivery planning only inherits the saved session account for implicit targets, so interrupts from a non-default Slack, Telegram, or similar account can resume on the wrong account or fail delivery.
  - confidence: 0.88
- **[P1] Keep webchat interrupts session-local:** `src/auto-reply/reply/commands-session-interrupt.ts:86`
  - body: For webchat/internal origins this still forces `deliver: true`. Current Gateway handling resolves internal delivered agent requests through the configured outbound channel or errors when no target exists, so a private webchat interrupt can misroute externally or fail after aborting the active run.
  - confidence: 0.82
- **[P2] Register /interrupt in the shared command catalog:** `src/auto-reply/reply/commands-core.ts:206`
  - body: Adding the handler makes text parsing recognize `/interrupt`, but `/commands` and native Discord, Telegram, and Slack command deployment are driven by the shared registry. Without a registry entry, native-command users will not see or invoke the new command consistently.
  - confidence: 0.86
- **[P2] Serialize thread-context store writes:** `src/sessions/thread-context-store.ts:137-141`
  - body: `saveThreadContext` loads the whole JSON file, mutates it, and writes it back without a per-file lock. Overlapping cron or isolated-agent deliveries can both read the same snapshot and the later write can drop the earlier context entry.
  - confidence: 0.9
- **[P2] Canonicalize Telegram chat ids before saving context:** `src/cron/isolated-agent/run.ts:945`
  - body: For explicit Telegram delivery, the resolved outbound target can be the bare chat id, while inbound lookups use `OriginatingTo` values with the `telegram:` prefix. Saving that raw value as the thread-context chat id can make later replies miss the stored prior-context note.
  - confidence: 0.84
