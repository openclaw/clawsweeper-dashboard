## Summary

Keeping this PR open: it proposes useful new `/interrupt` and cross-session thread-context behavior, but the current branch still has correctness and security-sensitive routing defects, lacks contributor real behavior proof, and is not old enough for a mostly-implemented stale-branch close.

## What This Changes

The PR adds a text `/interrupt <message>` handler that aborts and requeues a running session, plus a JSON-backed thread-context store for cron or isolated-agent follow-up replies in the same IM thread.

## Best Possible Solution

Refresh and likely split the work so `/interrupt` reuses the existing session steering/control contracts, preserves account/thread/internal routing facts, registers the command surface, serializes context persistence, normalizes provider keys, and includes redacted real-channel proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection against current main shows the routing, registry, persistence, and Telegram-key contracts the branch must satisfy. The underlying request is a feature, so there is no existing broken contract to reproduce as a bug.

## Solution Assessment

No. The direction is useful, but the proposed implementation is not the narrowest maintainable solution until it preserves delivery facts, aligns with command registration and session steering, and proves the behavior in a real channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the originating account on interrupt requeues:** `src/auto-reply/reply/commands-session-interrupt.ts:80-90`
  - body: The replacement turn sends explicit `channel` and `to` values but omits `accountId`. Current delivery planning only reuses the saved session account for implicit targets, so interrupts from a non-default channel account can resume from the wrong account or fail.
  - confidence: 0.88
- **[P1] Keep internal interrupts session-local:** `src/auto-reply/reply/commands-session-interrupt.ts:86`
  - body: The requeue uses `deliver: true` even when the command came from webchat/internal chat. Current gateway handling rejects delivered agent requests that resolve to the internal channel, so webchat interrupts can abort the run and then fail to redirect it.
  - confidence: 0.82
- **[P1] Preserve the thread id when requeueing interrupts:** `src/auto-reply/reply/commands-session-interrupt.ts:80-88`
  - body: The replacement turn forwards `channel` and `to` but not `threadId`. For threaded providers where the target does not encode the thread, explicit delivery stops inheriting the saved session thread and the redirected reply can land in the parent chat.
  - confidence: 0.84
- **[P2] Register or drop the `/interrupt` command surface:** `src/auto-reply/reply/commands-core.ts:203-207`
  - body: The text parser recognizes `/interrupt`, but `/commands` and native Discord, Slack, and Telegram command deployment are registry-driven. Without a registry entry, native-command users will not see or invoke this command consistently.
  - confidence: 0.86
- **[P2] Serialize thread-context store writes:** `src/sessions/thread-context-store.ts:137-141`
  - body: `saveThreadContext` loads the JSON store, mutates it, and writes it back without a per-file lock. Overlapping cron or isolated-agent deliveries can both read the same snapshot and the later write can drop the earlier context entry.
  - confidence: 0.9
- **[P2] Normalize Telegram chat ids before saving thread context:** `src/cron/isolated-agent/run.ts:942-946`
  - body: Explicit Telegram delivery stores the resolved bare chat id, while inbound lookups use `OriginatingTo` values with the `telegram:` prefix. Those keys will not match, so explicit Telegram thread deliveries can skip the prior-context note.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or workflow change, but the interrupt requeue can misroute redirected output across account, thread, or internal-session boundaries.

Concerns:

- **[medium] Interrupt requeue can use the wrong account:** `src/auto-reply/reply/commands-session-interrupt.ts:86`
  - body: The new interrupt path supplies explicit delivery routing without preserving `accountId`; on multi-account channels, redirected output can be sent from the default or wrong account, creating a delivery-integrity and confidentiality risk.
  - confidence: 0.84
- **[medium] Internal interrupts can escape session-local delivery:** `src/auto-reply/reply/commands-session-interrupt.ts:86`
  - body: For internal or webchat origins, `deliver: true` asks the gateway to resolve outbound delivery instead of keeping the replacement turn session-local, so private webchat instructions can be misrouted or fail after abort.
  - confidence: 0.78
