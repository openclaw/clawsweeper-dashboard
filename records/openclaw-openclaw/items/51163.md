## Summary

Keep open: the cross-session thread-context work remains useful and this PR is the canonical linked implementation path, but the current branch is not mergeable as-is because the interrupt requeue path loses routing facts, the thread-context store has concurrency and Telegram key issues, and there is no real behavior proof.

## What This Changes

The PR adds a text `/interrupt <message>` handler that aborts and requeues a session, plus a JSON-backed thread-context store for cron or isolated-agent follow-up replies.

## Best Possible Solution

Refresh and likely split the work: keep hard redirects on the existing `sessions.steer` path, then land thread-context continuity with routing-safe keys, serialized persistence, registry/docs/tests, and redacted real-channel proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR head against current main shows the account, internal-delivery, thread-id, registry, store-write, and Telegram-key problems. The underlying request is a feature request, not a strict bug reproduction.

## Solution Assessment

No. The direction is useful, but the patch is not the narrowest maintainable solution until it aligns with `sessions.steer`, preserves routing facts, serializes persistence, normalizes provider keys, and has real-channel proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the originating account on interrupt requeues:** `src/auto-reply/reply/commands-session-interrupt.ts:80-90`
  - body: This requeue supplies an explicit `channel`/`to` pair but omits `accountId`. Current delivery planning only inherits the saved session account for implicit targets, so interrupts from a non-default account can resume on the wrong account or fail.
  - confidence: 0.88
- **[P1] Keep internal interrupts session-local:** `src/auto-reply/reply/commands-session-interrupt.ts:86`
  - body: `deliver: true` is set even when the command originated from webchat/internal chat. Current agent handling tries to resolve an external delivery channel for delivered internal requests, so a private interrupt can misroute or fail after the run was aborted.
  - confidence: 0.82
- **[P1] Preserve the thread id when requeueing interrupts:** `src/auto-reply/reply/commands-session-interrupt.ts:80-88`
  - body: The replacement turn forwards `channel` and `to` but not `threadId`. For threaded surfaces where the target does not encode the thread, current explicit agent delivery can stop inheriting the saved thread and post the redirected reply in the parent chat.
  - confidence: 0.84
- **[P2] Register or drop the `/interrupt` command surface:** `src/auto-reply/reply/commands-core.ts:206`
  - body: Adding the handler makes text parsing recognize `/interrupt`, but `/commands` and native Discord, Telegram, and Slack command deployment are registry-driven. Without a registry entry, native-command users will not see or invoke this command consistently.
  - confidence: 0.86
- **[P2] Serialize thread-context store writes:** `src/sessions/thread-context-store.ts:137-141`
  - body: `saveThreadContext` loads the whole JSON file, mutates it, and writes it back without a per-file lock. Overlapping cron or isolated-agent deliveries can both read the same snapshot and the later write can drop the earlier context entry.
  - confidence: 0.9
- **[P2] Normalize Telegram chat ids before saving thread context:** `src/cron/isolated-agent/run.ts:942-946`
  - body: For explicit Telegram delivery, the resolved outbound target can be the bare chat id, while inbound lookups use `OriginatingTo` values with the `telegram:` prefix. Saving that raw value as `chatId` can make later replies miss the stored prior-context note.
  - confidence: 0.84
