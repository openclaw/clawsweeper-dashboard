## Label Justifications

- `P2`: The reported reconnect loop is a concrete gateway protocol-compatibility bug with limited blast radius to version-skewed macOS/OpenClawKit connections.
- `impact:crash-loop`: The issue is about repeatedly reconnecting after a terminal protocol mismatch instead of pausing for an update.
- `maturity:stable`: The affected gateway-runtime feature is M4 Stable and includes Protocol Compatibility and WebSocket Connection in the stable shortlist.

## Summary

Current main already fixes the macOS/OpenClawKit protocol-mismatch reconnect behavior: Swift now parses `PROTOCOL_MISMATCH` as a known detail code and treats it as non-recoverable, with merged OpenClawKit coverage. The remaining browser Control UI reconnect-policy gap is separate and still tracked at https://github.com/openclaw/openclaw/issues/98413.

## What This Changes

The issue asks for macOS/OpenClawKit `GatewayChannel` to treat structured `PROTOCOL_MISMATCH` connect failures as terminal so it does not reconnect-loop.

## Best Possible Solution

Keep the current main OpenClawKit protocol-mismatch handling, close this macOS issue as implemented, and let the separate Control UI issue carry the browser reconnect-policy work.

## Reproduction Assessment

Yes, at source level for the pre-fix behavior: v2026.6.11 lacks the Swift `PROTOCOL_MISMATCH` enum/non-recoverable branch, while current main includes it. I did not run Swift tests in this read-only review.

## Solution Assessment

Yes. The merged fix is the narrow owner-boundary solution because it adds the missing structured Swift detail code and reuses the existing `isNonRecoverable` reconnect-pause gate instead of adding a parallel reconnect policy.
