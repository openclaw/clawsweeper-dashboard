The issue reports an ACPX config test mismatch where the test expects source-mode MCP server args while the resolver returns built dist MCP server args when dist exists.

## Best Possible Solution

Land one focused fix PR that makes `extensions/acpx/src/config.test.ts` cover both source fallback and dist-present MCP server resolution, then close or supersede the duplicate open PR.

Yes. The narrow maintainable solution is test-only: preserve the resolver and explicit opt-in bridge behavior, but make the ACPX config test hermetic for both resolver paths.
