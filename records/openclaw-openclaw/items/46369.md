## Summary

Keep open: current main and the latest release still remove the cron job row without directly cleaning the matching cron session rows, and the reaper path still skips base cron session keys. The linked cleanup and retention PRs are implementation candidates, but none has landed and maintainers still need to choose the safe lifecycle boundary for base cron sessions.

## What This Changes

The issue asks for `cron.remove` to clean up deleted cron jobs' session-store records and related cron run artifacts so ghost sessions stop appearing in the Control UI.

## Best Possible Solution

Land one maintainer-approved cron deletion/session lifecycle path that removes deleted-job ghost sessions without pruning live base cron state or creating a second cleanup policy.

## Reproduction Assessment

Yes for a source-level reproduction: current `cron.remove` deletes the job row only, while base cron session keys are skipped by the reaper predicate. I did not run a live Control UI reproduction in this read-only review.

## Solution Assessment

Unclear: direct cleanup is plausible for deleted jobs, but the overlapping retention PR shows base cron rows can carry model/auth/label state for live jobs. Maintainers need to choose the lifecycle boundary before this should be treated as an automation-ready fix.
