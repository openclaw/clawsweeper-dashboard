## Summary

Keep open: current main and the latest release still remove only the cron job record, while session-store cron rows can remain and the reaper only manages `:run:` session keys. The issue has an open linked fork PR plus overlapping retention work, so this needs maintainer review of the cleanup semantics rather than closure or a duplicate automated fix lane.

## What This Changes

Deleting a cron job via `cron.remove` should also clean up the deleted job's persisted session records and relevant run artifacts so the Control UI does not show ghost cron sessions.

## Best Possible Solution

Land or adopt a narrow cron-owned cleanup that removes deleted-job session rows through the session lifecycle API, coordinates with the guarded retention work, and leaves current SQLite run-history policy intact.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron.remove` deletes only the job record while the existing reaper only processes `:run:` cron session keys; I did not run a mutating Control UI repro in this read-only review.

## Solution Assessment

Unclear. The cleanup direction is right, but the best fix needs maintainer agreement on immediate delete-time cleanup versus retention-time cleanup and on preserving live cron base-session state.
