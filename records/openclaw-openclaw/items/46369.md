## Summary

Current main still leaves deleted cron jobs’ session records behind. The linked implementation PR remains open but is conflicting and targets obsolete JSONL run logs, so this issue is still necessary while maintainers decide whether to refresh that work or replace it with a current-storage repair.

## What This Changes

Clean up a deleted cron job’s persisted session metadata and associated run history so it no longer appears as a ghost session.

## Best Possible Solution

Remove the deleted job’s exact base and per-run session entries from its effective agent session store, explicitly define retention or deletion of its SQLite run history, and verify that a Control UI refresh no longer lists the job.

## Reproduction Assessment

Yes, from current-main source: `cron.remove` performs no session cleanup, while the only cron reaper intentionally excludes base cron sessions. This establishes the persistence path even though a live Control UI reproduction was not run during the read-only review.

## Solution Assessment

The proposed delete-time session cleanup is the narrow root fix, but the old JSONL deletion approach is no longer the best complete solution because current run history is SQLite-backed; its retention behavior needs an explicit maintainer decision.
