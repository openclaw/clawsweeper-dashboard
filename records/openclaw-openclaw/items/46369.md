## Summary

Keep open: current main and v2026.6.6 still leave cron-owned base session records behind when a job is removed, and the open related PRs have not landed a complete deletion-cleanup policy.

## What This Changes

Deleting a cron job via `cron.remove` should clean up associated cron session-store records and run-history artifacts so deleted jobs do not leave stale session state.

## Best Possible Solution

Land or replace a cron-owned cleanup that removes deleted-job cron session rows and any intended run-history state, coordinated with https://github.com/openclaw/openclaw/pull/90194's base-session retention semantics.

## Reproduction Assessment

Yes. Source inspection shows `cron.remove` only mutates the cron job store while cron base session keys are persisted separately and skipped by the current reaper; I did not run a mutating local repro in this read-only review.

## Solution Assessment

Unclear. The bug is valid, but the best fix should coordinate delete-time cleanup with current SQLite run logs and the broader base-session retention work in https://github.com/openclaw/openclaw/pull/90194.
