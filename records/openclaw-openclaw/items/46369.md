## Summary

Keep open: current main still removes only the cron job record, while the missing session-store and run-log cleanup is tracked by an open, unmerged fix PR.

## What This Changes

Deleting a cron job via `cron.remove` should also remove that job's session-store entries and run log file so the Control UI no longer lists ghost cron sessions.

## Best Possible Solution

Implement cleanup in the cron service remove path, covering both base and per-run cron session keys plus `cron/runs/<jobId>.jsonl`, then land or replace the linked fix PR with regression coverage.

## Reproduction Assessment

Yes, source-reproducible: `cron.remove` reaches `src/cron/service/ops.ts` and only removes the job from the cron store, while current session cleanup is retention-based and run-key-only. I did not live-run the UI path in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix belongs in `CronService.remove` after successful deletion, using existing session-store and run-log helpers. Filtering ghosts in the Control UI would hide the stale data rather than clean the owning cron artifacts.
