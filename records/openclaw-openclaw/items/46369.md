## Summary

This issue is still necessary: current main and the latest release still remove only the cron job row, while cron session cleanup remains limited to expired per-run sessions. There is an open closing PR for this issue, but it is currently conflicting and cannot be treated as a landed fix.

## What This Changes

The issue asks for `cron.remove` to clean up a deleted cron job's session-store entries and run history so deleted cron jobs stop appearing as ghost sessions in the Control UI.

## Best Possible Solution

Keep this as the canonical bug until a focused cron-owned cleanup lands: `cron.remove` should remove or archive deleted-job session rows, handle the current SQLite-backed run-log model, and add regression coverage for the Control UI ghost-session path.

## Reproduction Assessment

Yes from source inspection, but I did not run a live UI repro. `cron.remove` currently removes only the cron job row, while the session reaper intentionally preserves base cron session rows and only prunes expired `:run:` rows.

## Solution Assessment

Unclear as a final implementation path. The correct boundary is cron-owned cleanup on job removal, but maintainers need to settle delete/archive/run-history semantics and either repair or replace the conflicting linked PR.
