## Summary

Keep open. Current main still removes only the cron job definition, while the base cron session entry and run-log artifact remain outside the deletion path; the open, unmerged PR #46506 is the paired implementation path and should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests that cron job deletion also remove the deleted job's session-store records and cron run-log file so the Control UI does not show ghost cron sessions.

## Best Possible Solution

Implement cron-owned cleanup immediately after a successful job deletion: remove the base cron session entry and any matching run-session entries for the deleted job, unlink the safe `runs/<jobId>.jsonl` path, tolerate missing artifacts, preserve unrelated sessions, and cover the behavior with focused cron service tests.

## Reproduction Assessment

Yes. Seed or create an isolated cron job with a base `agent:<id>:cron:<jobId>` session entry and a run log, call `cron.remove`, then list sessions; current main removes the job record but leaves the visible base session and does not unlink the run log.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the cron removal path after successful deletion, using existing session-store and run-log helpers; #46506 appears aligned but still needs maintainer review, rebase, landing, or replacement.
