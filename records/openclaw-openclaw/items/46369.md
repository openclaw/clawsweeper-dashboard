## What This Changes

Deleting a cron job through `cron.remove` should also clean up the job's persisted cron session records and related run artifacts so deleted jobs do not leave ghost sessions.

## Best Possible Solution

Land or replace the linked cleanup PR with a narrow core fix that removes deleted-job session entries through the session lifecycle path and preserves the intended retention/archive semantics.

## Reproduction Assessment

Yes by source inspection, not by a live run: current main's `cron.remove` removes only the job store entry, while the existing reaper is retention-based and scoped to cron run-session keys.

## Solution Assessment

Unclear until maintainers choose the cleanup semantics; the best implementation should use the storage-neutral session lifecycle direction rather than adding another ad hoc `sessions.json` mutation path.
