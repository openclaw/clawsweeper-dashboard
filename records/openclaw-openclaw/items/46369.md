## Summary

Keep open: current main and v2026.6.6 still remove only the cron job record, while the cron session reaper only sweeps `:run:` session keys and explicitly preserves base cron session rows. The remaining fix path is active but not landed: an open conflicting fork PR targets this issue, and https://github.com/openclaw/openclaw/pull/90194 covers the broader base-cron retention decision.

## What This Changes

The issue asks `cron.remove` and Control UI deletion to clean up deleted cron jobs' session-store rows and run history so ghost cron sessions stop appearing.

## Best Possible Solution

Land or replace the cron-owned cleanup so deleted jobs remove the appropriate base session rows and retained run history, coordinated with https://github.com/openclaw/openclaw/pull/90194's retention semantics.

## Reproduction Assessment

Yes. Source inspection shows `cron.remove` only mutates the cron job store, while base cron session keys are deliberately excluded from the current reaper path; I did not run a mutating local repro in this read-only review.

## Solution Assessment

Unclear. Cleaning up deleted-job sessions is directionally right, but the best permanent fix should coordinate deletion cleanup with broader cron retention behavior so existing persistent cron session history is not pruned unexpectedly.
