## Summary

Keep open. Current main still removes only the cron job definition, while the base cron session entry remains listable and the per-job run log is not unlinked; open PR #46506 is the paired implementation path and should land or be replaced before this issue closes.

## What This Changes

The issue asks for `cron.remove` to clean up the deleted cron job's persisted session-store records and run-log artifact so ghost sessions stop appearing in the Control UI.

## Best Possible Solution

Add cron-owned cleanup after a successful deletion and outside the cron service lock: remove the base and run session entries for the deleted job, unlink the safe `runs/<jobId>.jsonl` path, tolerate missing artifacts, preserve unrelated sessions, and cover it with focused cron service tests.

## Reproduction Assessment

Yes. Seed an isolated cron job with `agent:main:cron:<jobId>` and `agent:main:cron:<jobId>:run:<runId>` entries plus a `runs/<jobId>.jsonl` file, call `cron.remove`, then list sessions; current main removes the job record but leaves the base session visible and does not unlink the run log.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the cron removal path using existing session-store and run-log helpers; the linked PR direction is aligned but needs maintainer review, rebase, landing, or replacement before closure.
