## Summary

This PR remains potentially useful, but both exported send helpers are nonfunctional with the pinned grammY runtime: unsupported Bot API methods are dynamically available through `api.raw`, not directly on `api`, so each wrapper reaches its own “does not expose” error. The supplied proof covers normalization only and does not catch the transport failure.

## What This Changes

The PR adds and publicly re-exports Telegram Bot API 10.1 rich-message normalization, send, and draft compatibility helpers with mock-based unit tests.

## Best Possible Solution

Call the methods through a narrowly augmented `api.raw` contract using the official single payload object, return accurate Telegram result types, and export the seam only after focused tests and redacted live Bot API proof pass.

## Reproduction Assessment

Yes, from source with high confidence: passing a real grammY 1.43 `Bot.api` to either helper leaves the direct method lookup undefined, causing the helper’s own error before any Telegram request. A live failing invocation was not run during this read-only review.

## Solution Assessment

No. A compatibility helper is reasonable, but the correct dependency seam is grammY’s proxied `api.raw` payload API, not an invented convenience method on `api`; public export should follow accurate typing and live proof.
