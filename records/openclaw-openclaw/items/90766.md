- `impact:session-state`: The issue is about stale subagent run/session state being treated as resumable after restart.

## Summary

Keep open: current main still preserves stale restart-aborted subagent runs for recovery, and several open linked fixing PRs target the same bug but none has merged yet.

## What This Changes

Old restart-aborted subagent runs should age out after the existing stale-run window instead of being recovered after long gateway downtime.

## Best Possible Solution

Land one narrow fix that applies the shared stale-unended liveness cutoff to restart-aborted restore and orphan recovery while preserving fresh restart recovery and updating the regression tests/docs.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: current main has a test that seeds a three-hour-old `abortedLastRun` subagent run and expects `agent.wait`. I did not run a live multi-hour gateway restart in this read-only review.

## Solution Assessment

Yes, the best fix is the narrow owner-boundary change already reflected by the linked implementation candidates: reuse the existing stale-unended liveness policy in restore/orphan recovery, not a new config option or broader recovery redesign.
