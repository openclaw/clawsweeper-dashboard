## Label Justifications

- `P2`: This is a normal-priority lifecycle bug affecting subagent deployments after long gateway downtime, with a bounded agents/session-state blast radius.
- `impact:session-state`: The issue is about stale subagent run and session recovery state being treated as resumable after it should have aged out.

## Summary

Keep open: current main and the latest release still preserve stale restart-aborted subagent runs for orphan recovery, and several open mergeable PRs are already targeting the exact stale-age-out bug.

## What This Changes

The issue asks OpenClaw to retire old restart-aborted subagent orphan runs after the existing stale-run window instead of automatically resuming them after long downtime.

## Best Possible Solution

Land one narrow fix that applies the existing stale-unended liveness policy in both restore reconciliation and restart orphan recovery, with tests for stale and fresh aborted runs plus docs aligned to the new boundary.

## Reproduction Assessment

Yes. Current main has source and test coverage showing a three-hour-old `abortedLastRun` subagent run remains restart-recoverable and triggers `agent.wait`; I did not run a live gateway restart in this read-only review.

## Solution Assessment

Yes. The best fix is to reuse the existing stale-unended liveness policy at the restore and orphan-recovery boundaries, rather than adding a new config knob or separate timeout policy.
