## Summary

Keep open: current main still treats restart-aborted stale subagent runs as eligible for orphan recovery, and the docs/tests explicitly preserve that path rather than aging it out after the stale-run window.

## What This Changes

The issue asks for restart-time subagent orphan recovery to age out very old orphaned runs instead of replaying or resuming them after downtime.

## Best Possible Solution

Apply the existing stale-unended subagent liveness cutoff to restart-aborted orphan recovery, retire old entries as stale/lost, and update the recovery tests plus subagents docs to make the boundary explicit.

Yes from source inspection and existing tests: current main has a test that keeps a three-hour-old `abortedLastRun` subagent run recoverable and calls `agent.wait`. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing stale-unended liveness policy in the orphan-recovery/restore boundary, while preserving fresh restart recovery and rapid re-wedge tombstones.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/90766. Current main preserves stale unended restored subagent runs when the child session has `abortedLastRun: true`, so a run older than `STALE_UNENDED_SUBAGENT_RUN_MS` can still be resumed after restart. Add a narrow age-out guard so restart-time orphan recovery retires/marks stale old aborted orphan runs instead of sending a synthetic resume, while preserving fresh restart recovery and the rapid re-wedge tombstone behavior from https://github.com/openclaw/openclaw/issues/74864. Prefer reusing `isStaleUnendedSubagentRun` or the existing liveness cutoff rather than adding config/env. Update or replace the test that currently expects a three-hour-old aborted run to recover, add direct orphan-recovery coverage for stale aborted entries, and update `docs/tools/subagents.md`. Do not change provider routing, plugin APIs, or unrelated task maintenance behavior.
