## Summary

This issue remains necessary: current main and v2026.6.6 retain the reported provenance mismatch, Microsoft Teams depends on the denied keyed-store capability, and no canonical issue or pull request currently owns a fix. The repair is likely narrow, but the authoritative trust rule needs explicit maintainer security and product judgment.

## What This Changes

Fix official Microsoft Teams installs from ClawHub being denied keyed-store access despite installing from ClawHub’s official channel.

## Best Possible Solution

Define one auditable official-provenance contract across catalog metadata, install records, runtime capability enforcement, and doctor/install diagnostics; either support the official ClawHub route while retaining all identity checks or prevent and repair that route before channel startup, with Microsoft Teams and sibling-plugin regression coverage.

## Reproduction Assessment

Yes, at source level with high confidence: an official-channel ClawHub record for Microsoft Teams cannot pass the predicate because its catalog entry lacks `clawhubSpec`, while the plugin opens keyed stores in normal runtime paths. The reporter also provides a concrete v2026.6.5 Kubernetes before/after comparison between ClawHub and npm installation.

## Solution Assessment

The diagnosis is correct, but no suggested repair is automatically best: maintainers must decide whether official-channel ClawHub provenance is authorized for npm-preferred catalog entries or whether those installs should be rejected and migrated before runtime.
