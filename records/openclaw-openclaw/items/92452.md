## Summary

Keep open: current main and latest release still have the reported provenance mismatch, so an official ClawHub install of Microsoft Teams remains unable to satisfy the keyed-store trust gate.

## What This Changes

The issue reports that installing `@openclaw/msteams` from ClawHub creates an official install record that cannot be marked trusted, causing the channel to restart-loop when it opens keyed state.

## Best Possible Solution

Define a generic official-provenance rule that safely trusts official-channel ClawHub npm-pack installs only when package identity, plugin id, install record, and path match the official catalog, and add install or doctor diagnostics for impossible trust states.

## Reproduction Assessment

Yes, at source level: an official-channel ClawHub install record for Microsoft Teams cannot pass the current predicate because the official catalog entry lacks `clawhubSpec`, and Microsoft Teams opens keyed stores during startup. I did not run the reported Kubernetes deployment in this read-only review.

## Solution Assessment

Unclear as a product decision: the fix should not be a broad ClawHub trust bypass, but a provenance-safe official install rule plus actionable install or doctor diagnostics is the narrowest maintainable direction.
