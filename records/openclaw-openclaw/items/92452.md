## Summary

Keep open: current main and latest v2026.6.8 still contain the reported provenance mismatch, and the only narrow candidate fix was closed unmerged. The issue remains a real security-boundary bug for official Microsoft Teams ClawHub installs.

## What This Changes

The issue reports that official ClawHub installs of the Microsoft Teams plugin are not trusted when the official catalog entry is npmSpec-only, causing keyed-store access to fail and the channel to restart repeatedly.

## Best Possible Solution

Implement a narrow official-ClawHub/npm-only trust rule that preserves URL, official-channel, package identity, plugin-id, and installed-path checks, with regression coverage and real install/startup proof; keep broader self-hosted load-path trust work separate.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 require clawhubSpec for official ClawHub trust, while Microsoft Teams is npmSpec-only and uses openKeyedStore. I did not run a live Kubernetes or ClawHub/Microsoft Teams deployment in this read-only review.

## Solution Assessment

Unclear until maintainers accept the trust-boundary rule; the narrow code location is the manifest trust predicate, but the fix must preserve official ClawHub URL, channel, package, plugin-id, and install-record checks. A Microsoft Teams-only catalog exception or raw trust bypass would be the wrong shape.
