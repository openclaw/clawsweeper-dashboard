## Summary

Keep open: current main and v2026.6.6 still have the reported provenance mismatch, and the open related issues cover broader or different trust-policy questions rather than replacing this specific ClawHub Microsoft Teams failure.

## What This Changes

A ClawHub install of `@openclaw/msteams` from the official catalog remains untrusted because the catalog entry is npm-only, causing keyed-store startup access to fail and the channel to restart.

## Best Possible Solution

Choose a provenance-safe official ClawHub trust and diagnostic rule that verifies package identity, official channel, install record, and plugin id, then add coverage for npm-only official catalog entries.

## Reproduction Assessment

Yes, at source level: an official-channel ClawHub install for Microsoft Teams cannot satisfy the current trusted-install predicate because the catalog lacks `clawhubSpec`, and Teams opens keyed state during startup. I did not run the reported Kubernetes deployment.

## Solution Assessment

Unclear until maintainers choose the trust policy: a generic provenance-safe official ClawHub rule plus install or doctor diagnostics is better than a Teams-only exception or a broad ClawHub-source bypass.
