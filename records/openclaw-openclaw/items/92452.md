## Label Justifications

- `P1`: A supported official installation command can leave a major messaging channel unusable and repeatedly restarting for real users.
- `impact:crash-loop`: The Microsoft Teams channel exits immediately and repeatedly auto-restarts when its required keyed-state access is denied.
- `impact:security`: The underlying defect is in the provenance boundary controlling which external plugins receive trusted persistent-state capabilities.

## Summary

Current main and v2026.6.6 still contain the reported provenance mismatch: the official Microsoft Teams catalog entry is npm-only, while official ClawHub trust requires a catalog `clawhubSpec`, causing the plugin’s required keyed-state calls to fail. No canonical duplicate or fix PR was found, so this issue remains necessary.

## What This Changes

Allow an official ClawHub installation of the Microsoft Teams plugin to satisfy the keyed-store trust boundary or fail earlier with an actionable installation diagnostic.

## Best Possible Solution

Preserve package, installed-path, plugin-id, and official-channel verification while making official ClawHub provenance independent of which distribution method is preferred in catalog metadata; reject or clearly diagnose impossible trust states before channel startup and add sibling official-plugin regression coverage.

## Reproduction Assessment

Yes, at source level: an official-channel ClawHub record for Microsoft Teams cannot pass the current predicate because its catalog entry lacks `clawhubSpec`, and the plugin opens keyed stores during startup. This read-only review did not run a live Kubernetes deployment.

## Solution Assessment

No single suggested option is sufficient by itself; the best fix is a generic provenance-safe classification plus an actionable install or doctor diagnostic, rather than a Microsoft Teams-only metadata exception or a later runtime message alone.
