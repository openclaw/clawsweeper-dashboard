## Summary

Keep open: current main and v2026.6.6 still have the reported official ClawHub provenance mismatch, so an official Microsoft Teams ClawHub install can fail the trusted keyed-state gate while the npm install path is trusted.

## What This Changes

The issue asks OpenClaw to make official ClawHub installs of `@openclaw/msteams` pass the keyed-store trust gate or fail with an actionable install/doctor diagnostic.

## Best Possible Solution

Preserve package, installed-path, plugin-id, and official-channel verification while allowing official ClawHub provenance to be trusted when it matches the cataloged official package, and add install or doctor diagnostics plus regression coverage.

## Reproduction Assessment

Yes, at source level: an official-channel ClawHub install record for Microsoft Teams cannot satisfy the current predicate because the catalog entry lacks `clawhubSpec`, and the plugin opens keyed stores during runtime startup. I did not run the live Kubernetes deployment.

## Solution Assessment

No single suggested option is sufficient by itself. The best path is a generic provenance-safe trust rule plus an actionable install or doctor diagnostic, not a Teams-only exception or an unconditional load-path trust knob.
