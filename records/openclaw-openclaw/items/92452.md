## Summary

Keep open: current main and the latest v2026.6.6 release still contain the reported official-provenance mismatch, so a ClawHub official Microsoft Teams install with npm-only catalog metadata cannot become a trusted keyed-state plugin. Related open issues cover broader self-hosted and community-plugin trust questions, but neither replaces this narrower ClawHub-official Microsoft Teams failure.

## What This Changes

The issue reports that installing `@openclaw/msteams` through ClawHub official metadata leaves the plugin untrusted for keyed state because the official catalog entry is npmSpec-only, causing Teams channel restart loops.

## Best Possible Solution

Keep this open for a maintainer-approved, provenance-safe fix that trusts only official ClawHub records whose package identity matches the official catalog, including npmSpec-only official entries, and adds a focused install/doctor diagnostic plus regression coverage.

## Reproduction Assessment

Yes at source level: an official-channel ClawHub Teams install cannot satisfy the current `clawhubSpec` predicate, and Teams opens keyed stores through the gated runtime API. I did not run the live Kubernetes/OCI deployment.

## Solution Assessment

No single suggested option is best by itself. A Microsoft Teams-only catalog patch is too narrow, while a safe core fix should preserve official package identity, install-record/path matching, and the official ClawHub channel before granting trusted state access.
