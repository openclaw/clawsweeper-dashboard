## Summary

Current main has a working Telegram voice-to-STT-to-agent-context path and explicitly marks transcripts untrusted, but it lacks the requested structured end-to-end handoff telemetry. The request remains useful, yet its cross-channel schema, privacy policy, and model-visible boundary require maintainer product judgment before implementation.

## What This Changes

Add structured observability for Telegram voice media download, STT execution and trust, and transcript insertion into the correct agent turn and session.

## Best Possible Solution

Define a generic typed media-processing outcome carried from attachment receipt through transcription and context construction, expose a redacted correlation record to support tooling, and give the agent only compact media-kind and transcript trust or failure state.

## Reproduction Assessment

Not applicable: current main has a source-verifiable transcription and insertion path, while this issue requests new lifecycle telemetry and agent-visible failure semantics rather than reporting a concrete failing workflow.

## Solution Assessment

No, not as a Telegram-specific collection of diagnostic fields. The maintainable solution is a generic media-processing outcome with privacy-safe support telemetry and a much smaller model-visible trust state.
