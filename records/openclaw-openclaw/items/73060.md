## Summary

Keep open. Current main still ignores `threadId` on the direct Mattermost message-tool send path, and the PR is a narrow functional fix, but it still needs the stale fallback comment updated and the required user-facing changelog entry before merge.

## What This Changes

The PR changes the Mattermost plugin send action to resolve `replyToId` from explicit `replyToId`, then `threadId`, then legacy `replyTo`, and adds regression tests for trimmed `threadId` and precedence.

## Best Possible Solution

Land this focused Mattermost plugin fix after adding the changelog entry and correcting the fallback comment, while keeping #67784 open for the remaining `filePath`/`path` and `mediaLocalRoots` work and treating #58439 as the separate media-local-roots PR.

## Reproduction Assessment

Yes. Call `mattermostPlugin.actions.handleAction` with `action: "send"`, a Mattermost channel target, and `threadId` but no `replyToId`; current main forwards no `replyToId` to `sendMessageMattermost`, so no Mattermost `rootId` is set.

## Solution Assessment

Yes, with small cleanup. The `replyToId ?? threadId ?? replyTo` order is the narrow maintainable fix because it preserves explicit `replyToId` precedence and matches existing Mattermost outbound helper behavior.
