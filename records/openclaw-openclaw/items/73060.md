## Summary

Keep open: current main and the latest release still ignore explicit `threadId` in Mattermost direct send actions, and this PR is a focused, source-supported fix with sufficient real Mattermost output supplied.

## What This Changes

The PR adds `params.threadId` as a normalized fallback between `replyToId` and legacy `replyTo` for Mattermost direct send actions and adds two regression tests.

## Best Possible Solution

Land this focused Mattermost handler fix after normal maintainer and required-check review, while leaving broader implicit thread-context inheritance to the canonical Mattermost threading work.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.6` compute Mattermost direct-send `replyToId` from `replyToId` and `replyTo` only, so a send with only `threadId` does not set Mattermost `root_id`. I did not run a local Mattermost send during this read-only review.

## Solution Assessment

Yes. Adding normalized `threadId` after explicit `replyToId` and before legacy `replyTo` in the Mattermost-owned handler is the narrowest maintainable fix for explicit thread targets and avoids broader thread-context policy changes.
