## Summary

Keep open. Current main and the latest release still ignore explicit Mattermost `threadId` in the direct message-tool send path, while this PR applies the narrow fallback, updates the stale comment, adds focused regression coverage, and now includes redacted live Mattermost API output.

## What This Changes

The PR adds `params.threadId` as a normalized fallback between `replyToId` and legacy `replyTo` for Mattermost direct send actions, with regression tests for fallback and precedence.

## Best Possible Solution

Land this focused Mattermost handler fix after normal maintainer review and CI, while keeping broader implicit thread-context inheritance work separate.

## Reproduction Assessment

Yes, from source inspection: a Mattermost `message(action: "send")` call with `threadId` and no `replyToId` reaches `handleAction`, which currently computes `replyToId` without considering `threadId`.

## Solution Assessment

Yes. Adding normalized `threadId` between explicit `replyToId` and legacy `replyTo` is the narrow Mattermost-owned fix for explicit thread targets, and the broader implicit-thread work is not a safe superseder.
