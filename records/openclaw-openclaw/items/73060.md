## Summary

Keep open. The PR is a focused, plausible bug fix for Mattermost direct message-tool sends, but it still needs contributor-provided real Mattermost after-fix proof before merge and has one small stale-comment cleanup.

## What This Changes

The PR adds `params.threadId` as a normalized fallback between `replyToId` and legacy `replyTo` in Mattermost message-tool sends, plus regression tests for trimming and precedence.

## Best Possible Solution

Land this focused Mattermost handler fix after updating the fallback comment and adding redacted real Mattermost proof, while keeping broader media-root and implicit thread-context work coordinated through https://github.com/openclaw/openclaw/issues/67784, https://github.com/openclaw/openclaw/pull/58439, and https://github.com/openclaw/openclaw/pull/52120.

## Reproduction Assessment

Yes, at source level. On current main, a Mattermost `message` tool send with `threadId` and no `replyToId` reaches `handleAction`, which computes `replyToId` only from `replyToId` and legacy `replyTo`, so the sender receives no Mattermost `rootId`.

## Solution Assessment

Yes, with small cleanup. Adding normalized `threadId` between `replyToId` and legacy `replyTo` is the narrow maintainable fix for explicit thread targets and matches existing Mattermost outbound adapter precedence.
