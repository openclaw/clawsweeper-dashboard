## Summary

Keep open. Current main still ignores `threadId` on Mattermost direct message-tool sends, while this PR applies the right narrow fallback and adds focused regression tests; it still needs a changelog entry and stale comment cleanup before merge.

## What This Changes

The PR updates Mattermost send-action reply resolution to fall back from explicit `replyToId` to `threadId` to legacy `replyTo`, with tests for trimmed `threadId` and `replyToId` precedence.

## Best Possible Solution

Land this focused Mattermost plugin fix after adding the changelog entry and correcting the fallback comment, while leaving #67784 open for the remaining `filePath`/`path` and `mediaLocalRoots` work and treating #58439 as the separate media-local-roots PR.

## Reproduction Assessment

Yes. Call `mattermostPlugin.actions.handleAction` with `action: "send"`, a Mattermost channel target, and `threadId` but no `replyToId`; current main forwards no `replyToId` to `sendMessageMattermost`, so Mattermost receives no `rootId`.

## Solution Assessment

Yes, with small cleanup. The `replyToId` to `threadId` to `replyTo` order is the narrow maintainable fix because it preserves explicit reply precedence and matches existing Mattermost outbound helper behavior.
