## Summary

Keep open: current main still drops explicit `threadId` on the Mattermost direct message-tool send path, and this focused PR fixes that source-level bug but still needs contributor-provided real Mattermost proof before merge.

## What This Changes

The PR adds `params.threadId` as a normalized fallback in Mattermost message-tool sends and adds regression tests for trimming and `replyToId` precedence.

## Best Possible Solution

Land this focused Mattermost handler fix after updating the fallback comment and adding redacted real Mattermost proof, while coordinating broader handler cleanup through the related Mattermost issue and companion PRs.

## Reproduction Assessment

Yes, at source level. On current main, a Mattermost `message` tool send with `threadId` and no `replyToId` reaches `handleAction`, which computes `replyToId` only from `replyToId` and legacy `replyTo`, so the sender receives no Mattermost `root_id`.

## Solution Assessment

Yes, with small cleanup. Adding normalized `threadId` between `replyToId` and legacy `replyTo` is the narrow maintainable fix for explicit thread targets and matches existing Mattermost outbound adapter precedence.
