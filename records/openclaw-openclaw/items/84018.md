## Summary

Keep open: current main and v2026.6.5 already attempt xAI OAuth refresh, but the reported Cloudflare failure remains plausible for profiles that persisted the old `/oauth/token` endpoint because current refresh still prefers a cached `tokenEndpoint` over rediscovery.

## What This Changes

The issue asks OpenClaw to make xAI OAuth renewal work reliably when refresh requests are Cloudflare-blocked, and to document or warn when renewal cannot be automated.

## Best Possible Solution

Repair or normalize stale xAI OAuth token-endpoint state to the current discovered `/oauth2/token` endpoint, then add focused refresh coverage and user-facing docs or diagnostics for any remaining xAI-side blocking.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source plus live endpoint checks show a stale cached `/oauth/token` endpoint would still hit a Cloudflare 403, but I could not run a valid expired xAI OAuth profile refresh without real credentials.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the ownership boundary: the likely fix is a targeted auth-state repair or endpoint canonicalization, not a broad duplicate refresh implementation.
