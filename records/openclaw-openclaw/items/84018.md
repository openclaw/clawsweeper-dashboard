## Summary

Keep open: current main still trusts a saved xAI OAuth token endpoint during refresh, and the reported Cloudflare block is plausible for the `/oauth/token` endpoint even though current xAI discovery advertises `/oauth2/token`; this needs live expired-profile verification before closing or changing behavior.

## What This Changes

The issue asks for xAI OAuth credentials to renew automatically instead of requiring browser re-authentication when the access token expires.

## Best Possible Solution

Keep the fix in the xAI plugin: live-verify an expired SuperGrok profile, repair stale saved token endpoints to the current discovered `/oauth2/token` path if that proves to be the cause, and document/manual-reauth guidance only if xAI blocks the supported endpoint too.

## Reproduction Assessment

No high-confidence end-to-end reproduction yet. Source confirms xAI refresh uses a saved token endpoint, and live endpoint probes reproduce Cloudflare blocking on the reported `/oauth/token` path while `/oauth2/token` reaches the OAuth server, but I did not have a valid expired SuperGrok refresh profile.

## Solution Assessment

Unclear. The most promising narrow fix is to stop reusing stale xAI token endpoints and rely on current OIDC discovery, but maintainers should verify with a real expired profile before declaring this an upstream-only limitation.
