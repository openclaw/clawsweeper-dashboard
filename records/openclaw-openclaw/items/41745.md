## Summary

Keep open: the reported session-state bug is source-reproducible on current main and the PR is pointed at the right fix, but the current head still has blocking correctness issues and lacks required real-run proof.

## What This Changes

The PR moves `systemSent` persistence out of pre-run skill snapshot writes and into post-success reply/followup session accounting, with regression tests.

## Best Possible Solution

Land a rebased patch that defers `systemSent` disk persistence until successful reply/followup completion, preserves snapshot hydration, guards optional session entries, and includes redacted provider-failure retry proof.

## Reproduction Assessment

Yes, source-reproducible. Current main prepares the skill snapshot before model execution and persists `systemSent: true` in that pre-run phase, so a first-call provider failure can leave disk state saying the system prompt was delivered.

## Solution Assessment

No, not as submitted. Deferring persistence is the right narrow fix, but this head still has an optional-entry mutation defect, would drop current-main snapshot hydration, and needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Guard the optional session entry before mutating it:** `src/auto-reply/reply/session-run-accounting.ts:95`
  - body: `sessionEntry?.systemSent !== true` is also true when no entry was passed, but the success path then assigns `sessionEntry.systemSent = true`. This both matches the current TS18048 CI failure and can throw after a successful run if callers only have `storePath`/`sessionKey`, so require an entry before mutating or update a safe local/store value.
  - confidence: 0.97
- **[P2] Preserve hydrated snapshots when reusing skills:** `src/auto-reply/reply/session-updates.ts:173`
  - body: This branch reuses `current.skillsSnapshot` directly when no refresh is needed. Current main hydrates that snapshot because persisted entries may have `resolvedSkills` stripped; dropping that behavior can make cold-resumed sessions miss skills/tool definitions. Keep the deferred `systemSent` change while preserving `hydrateResolvedSkills(...)`.
  - confidence: 0.88
