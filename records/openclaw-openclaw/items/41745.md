## Summary

Keep open. This PR targets a real session-state bug that is still present on current main, and the latest revision addresses the earlier followup and best-effort persistence feedback, but it is not merge-ready because the new helper mutates an optional session entry and the user-facing fix still needs the required changelog entry.

## What This Changes

The PR removes the first-turn `systemSent` disk write from `ensureSkillSnapshot()`, adds post-success `systemSent` persistence for normal and followup runs, and adds regression coverage for failed first calls, silent successes, and refreshed followup session state.

## Best Possible Solution

Keep this PR open and revise it so `systemSent` is never persisted before the first model response, is persisted through a shared post-success helper for normal and followup runs, safely handles missing in-memory entries, includes the required changelog note, and passes the targeted session/followup tests plus the changed gate.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce the reported class: first-turn preparation persists `systemSent` before `runReplyAgent()`, so a first provider failure can leave disk state saying the system prompt was sent. The PR-specific blocker is reproducible by typechecking or calling the helper with `storePath` and `sessionKey` but no `sessionEntry`.

## Solution Assessment

No, not yet. The direction is the right narrow fix, and the earlier followup-state findings appear addressed, but the helper needs a safe optional-entry guard and the PR needs a changelog entry before this is the best maintainable solution.

Full review comments:

- **[P1] Guard the optional session entry before mutating it:** `src/auto-reply/reply/session-run-accounting.ts:85-95`
