## Summary

Keep this PR open: current main still has the pre-run `systemSent` persistence path, and the PR's deferred-persistence design is the right direction, but the latest head still has a type-checking blocker plus a required changelog gap before merge.

## What This Changes

The PR removes the first-turn `systemSent` disk write from skill snapshot persistence and adds post-success marker persistence for normal and followup reply runs with regression coverage.

## Best Possible Solution

Land the deferred marker-persistence design after making the in-memory assignment conditional, preserving best-effort disk writes after successful runs, and adding a single Unreleased changelog fix bullet.

## Reproduction Assessment

Yes. Source inspection on current main shows `ensureSkillSnapshot()` persists `systemSent: true` before `runReplyAgent()`, which is enough to reproduce the corrupt marker path after a first-call provider failure without needing a live failing provider run.

## Solution Assessment

No, not as submitted. The deferred-persistence approach is the narrow maintainable fix, but the optional `sessionEntry` assignment must be guarded and the user-facing fix needs the required changelog entry before merge.
