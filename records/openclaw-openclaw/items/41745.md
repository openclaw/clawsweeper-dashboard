## Summary

Keep open. The PR targets a valid source-reproducible session-state bug that is still present on current main and in the latest release, but the current head is not mergeable because it still has a blocking optional-entry type/runtime bug, drops current snapshot hydration behavior, and lacks contributor real behavior proof.

## What This Changes

The PR removes the first-turn `systemSent` disk write from `ensureSkillSnapshot`, adds post-success persistence for reply and followup runs, and adds regression coverage around retry/session persistence.

## Best Possible Solution

Land a rebased patch that defers `systemSent` disk writes until verified successful reply/followup completion, preserves current snapshot hydration, handles optional entries safely, and includes real retry proof.

## Reproduction Assessment

Yes, source-reproducible. Current main persists `systemSent: true` during `ensureSkillSnapshot` before model execution, so a first-call provider failure can leave disk state saying the system prompt was delivered.

## Solution Assessment

No, not as submitted. Deferring persistence is the right narrow direction, but this head must guard the optional session entry, preserve `hydrateResolvedSkills(...)`, and include real retry proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Guard the optional session entry before marking it sent:** `src/auto-reply/reply/session-run-accounting.ts:95`
  - body: `sessionEntry` is optional, but this assignment runs after an optional-chain check that does not narrow it. CI already reports TS18048 here, and at runtime a successful store update with no entry would be caught as a false persistence failure, so guard the mutation or update a known entry.
  - confidence: 0.97
- **[P2] Preserve hydrated skill snapshots on first-turn reuse:** `src/auto-reply/reply/session-updates.ts:173`
  - body: This branch reuses `current.skillsSnapshot` directly when no refresh is needed, but current main hydrates reused snapshots because persisted session entries strip `resolvedSkills`. A cold-resumed first turn can otherwise pass a stripped snapshot to the runner and lose runtime skill definitions.
  - confidence: 0.91
