## Summary

Keep open: current main still has the source-reproducible premature `systemSent` persistence bug, but this PR is not mergeable yet because real behavior proof is missing and the head still has correctness/type issues around optional session entries and current snapshot hydration.

## What This Changes

The PR removes `systemSent` from `ensureSkillSnapshot` disk writes, adds post-success `systemSent` persistence for reply and followup runs, and adds regression tests for retry/session persistence.

## Best Possible Solution

Land a rebased patch that defers `systemSent` disk writes until verified successful reply/followup completion, preserves current snapshot hydration, guards optional entries safely, and includes redacted real retry proof.

## Reproduction Assessment

Yes, source-reproducible. Current main persists `systemSent: true` during `ensureSkillSnapshot` before model execution, so a first-call provider failure can leave disk state saying the system prompt was delivered.

## Solution Assessment

No, not as submitted. Deferring persistence is the right narrow direction, but the branch must preserve current snapshot hydration, guard optional entries, rebase cleanly, and provide real retry proof before merge.

Overall correctness: patch is incorrect

- **[P2] Guard the optional session entry before marking it sent:** `src/auto-reply/reply/session-run-accounting.ts:95`
  - body: `sessionEntry` is optional, but this assignment runs after an optional-chain check that does not narrow it. CI reports TS18048 here, and at runtime a successful store update with no in-memory entry would be logged as a persistence failure, so guard the mutation or update a known entry.

A cold-resumed first turn can otherwise pass a stripped snapshot to the runner and lose runtime skill definitions.
  - confidence: 0.91
