## Summary

Keep this PR open. Current main still has the pre-run `systemSent: true` persistence that the PR targets, so the change is not obsolete, but the latest PR head still has blocking correctness issues in the new post-success persistence path.

## What This Changes

The PR removes the first-turn `systemSent` disk write from `ensureSkillSnapshot()`, adds post-success `systemSent` persistence for normal and queued followup runs, and adds regression coverage for failed first calls and silent followup successes.

## Best Possible Solution

Keep the PR open and revise it so `systemSent` is removed from pre-run disk persistence, then written after successful normal and followup runs through a shared best-effort helper that uses the active session entry and has regression coverage for failed first calls, successful retries, silent/tool-only successes, and reset or compaction followups.

## Reproduction Assessment

Yes. The source path is clear: current main persists `systemSent: true` in `ensureSkillSnapshot()` before `runReplyAgent()`, so a first-call provider failure after that point can leave disk state saying the system prompt was sent. The PR-specific findings are reproducible by making `updateSessionStoreEntry()` reject and by running a followup after reset/compaction where the captured entry still has `systemSent: true`.

## Solution Assessment

No, not yet. The direction is the right narrow fix, but the current patch should make the post-success write non-fatal and pass the active followup session entry before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Make systemSent persistence non-fatal:** `src/auto-reply/reply/session-run-accounting.ts:88-92`
  - body: `persistSystemSentAfterSuccess()` awaits `updateSessionStoreEntry()` directly, and both callers await this helper before reply delivery. If the session store lock times out or the write fails, a successful model run becomes a failed reply; this marker should be best-effort, matching the nearby usage persistence path that catches and logs write failures.
  - confidence: 0.88
- **[P2] Use the active followup session entry:** `src/auto-reply/reply/followup-runner.ts:412`
  - body: The followup call passes the captured `sessionEntry`, but the runner may already have a newer `activeSessionEntry` after preflight reset or compaction. If the captured entry still has `systemSent: true`, the helper skips writing the marker for the reset session, leaving it in first-turn mode and repeatedly resending the full system prompt.
  - confidence: 0.86
