## Summary

Keep open: current main still logs the `active_embedded_run` observe-only recovery skip/outcome path at warn level, and the focused fix PR for this issue was closed without merging. The detector has gained better active-work handling for the normal healthy path, but the recovery double-check branch that motivated the report still has warn-level noise when reached.

## What This Changes

The issue asks to stop warn-level diagnostic log flooding when long-running embedded agent sessions are healthy and recovery is skipped as `active_embedded_run`.

## Best Possible Solution

Demote only the benign `active_embedded_run` observe-only recovery skip/outcome logs to debug, update the focused expectations, and preserve warn-level logging for real stuck, stalled, aborted, released, or failed recovery outcomes.

## Reproduction Assessment

Yes, at source level: current main’s active-embedded-run observe-only recovery branch still calls `diag.warn`, and the focused unit test asserts those warning messages. I did not establish a live cron reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is the already-proposed log-level demotion for only the `active_embedded_run` observe-only skip path. Broader detector-side active-work unification belongs with the adjacent queue/active-run divergence issue, not as a prerequisite here.
