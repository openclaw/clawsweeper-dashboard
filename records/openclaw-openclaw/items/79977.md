## Summary

Current main no longer routes healthy active embedded runs through the stuck-session recovery path that produced the warn-level flood; active work is classified separately and non-queued long-running active-work console logging is debug-only, with regression coverage in place.

## What This Changes

The issue asks OpenClaw diagnostics to stop emitting repeated warn-level stuck-session and recovery-skip logs for healthy long-running embedded agent runs.

## Best Possible Solution

Keep the shipped diagnostics classification/log-level behavior and ask for a fresh report with v2026.5.3 or newer logs if warn-level recovery-skip flooding still appears.

## Reproduction Assessment

No on current main. Source and regression tests show healthy active embedded runs now classify as long-running active work, avoid warn-level console logging for the non-queued case, and do not call stuck-session recovery.

## Solution Assessment

Yes. Solving this at the heartbeat/classification layer is the maintainable fix because it preserves warnings for truly stuck, stalled, queued, failed, or mutating recovery states while removing the healthy active-run log flood.
