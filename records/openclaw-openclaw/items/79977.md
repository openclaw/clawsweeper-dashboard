## Summary

Close: current main and shipped v2026.5.3 already avoid the reported healthy long-running embedded-run warning flood by classifying active progress as long-running, routing queueDepth=0 active-work logs to debug, and skipping recovery for that path.

## What This Changes

Report asks to stop warn-level stuck-session and recovery-skip log floods for healthy long-running embedded agent sessions.

## Best Possible Solution

Keep the shipped diagnostics classifier/log-level behavior and ask for a fresh report only if the same warn-level flood reproduces on v2026.5.3 or newer with current logs.

## Reproduction Assessment

No current-main failing path: source and tests show healthy active embedded work is long_running, debug-level for queueDepth=0, and does not request recovery. The original logs match the reporter's older 2026.4.29 behavior before the shipped diagnostics fixes.

## Solution Assessment

Yes. The current implementation is narrower than changing thresholds because it prevents healthy active-progress sessions from entering stuck recovery while preserving warnings for real stalled, stuck, queued, and recovery-eligible cases.
