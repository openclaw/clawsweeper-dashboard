## Summary

Keep open: the fix is aimed at the right shared dispatcher path, but the current wrapper still allows stale delivery if supersession happens while an async `beforeDeliver` or `message_sending` hook is pending.

## What This Changes

The PR adds a shared foreground auto-reply freshness fence in `src/auto-reply/dispatch.ts`, regression coverage in `src/auto-reply/dispatch.freshness.test.ts`, and a changelog entry for #76905.

## Best Possible Solution

Keep the shared freshness fence, but check supersession again after any configured delivery hook resolves and cover that async race in the new regression test.

## Reproduction Assessment

Yes, source-reproducible: current main routes WhatsApp foreground replies through the shared buffered dispatcher with stable session/target fields and no generation guard. The PR models the core race, and the remaining async-hook gap is visible from the patched source.

## Solution Assessment

No, not yet: the shared dispatcher fence is the right maintainable direction, but it must re-check after async `beforeDeliver` work before allowing the payload to proceed to delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Recheck freshness after beforeDeliver hooks:** `src/auto-reply/dispatch.ts:243`
  - body: The wrapper only checks the fence before awaiting the configured `beforeDeliver` hook. If a `message_sending` plugin hook or channel hook is async, a newer same-key inbound can start while that await is pending; the older payload then returns and is delivered without another generation check, leaving the stale-final race open. Please check `isForegroundReplyFenceSuperseded` again after the hook resolves and before returning the payload.
  - confidence: 0.88
