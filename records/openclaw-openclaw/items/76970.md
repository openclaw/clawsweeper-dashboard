## Summary

Keep open: the PR targets the right shared auto-reply path for #76905, but the new fence advances before inbound dedupe, so duplicate or in-flight replays of the same provider message can suppress the legitimate original reply.

## What This Changes

The PR adds a shared auto-reply foreground freshness fence in `src/auto-reply/dispatch.ts`, regression tests in `src/auto-reply/dispatch.freshness.test.ts`, and a changelog entry for #76905.

## Best Possible Solution

Keep the shared dispatcher freshness fence, but only advance it for accepted non-duplicate inbound turns or make same-message replays reuse the active generation, with a regression test for duplicate replay.

## Reproduction Assessment

Yes, source-reproducible: current main has an async buffered-delivery window without a shared freshness fence, and the PR’s new duplicate-replay regression is visible from the order of `beginForegroundReplyFence` before `claimInboundDedupe`. I did not perform a live WhatsApp/Codex timing repro.

## Solution Assessment

No, not as-is: the shared fence is the narrow maintainable direction, but it must not treat duplicate or in-flight replays of the same provider message as a newer user turn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Do not advance the fence for duplicate turns:** `src/auto-reply/dispatch.ts:233`

Please only advance freshness for an accepted non-duplicate turn, or make same-message replays reuse the active generation, and cover that replay case.

## Repair Work Prompt

On this PR, fix the foreground reply freshness fence so duplicate or in-flight replays of the same provider message do not suppress the legitimate original reply. The current patch calls `beginForegroundReplyFence` in `src/auto-reply/dispatch.ts` before `dispatchReplyFromConfig` runs `claimInboundDedupe`, so a replay with the same `MessageSid` can increment generation and then be skipped, causing the first run’s final to be cancelled. Update the fence so only accepted non-duplicate inbound turns advance freshness, or make same active provider message ids reuse the current generation. Add regression coverage in `src/auto-reply/dispatch.freshness.test.ts` that starts an old dispatch, starts a duplicate/in-flight dispatch with the same `MessageSid`, verifies the duplicate returns no queued final, then releases the old final and confirms it still delivers. Preserve the existing newer-message suppression and different-target isolation behavior; keep the scope to auto-reply dispatch/tests, with changelog wording changed only if needed.
