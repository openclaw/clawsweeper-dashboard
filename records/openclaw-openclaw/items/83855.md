## Summary

This should stay open: current `main` still caches the first non-test config snapshot promise before it settles, and a rejection from the snapshot path would be reused by later config-guard calls in the same process.

## What This Changes

The issue reports that `getConfigSnapshot` permanently caches a rejected config snapshot promise instead of retrying after a transient `readConfigFileSnapshot()` failure.

## Best Possible Solution

Clear the module-level snapshot promise only when the cached read rejects, preserve successful snapshot reuse, and add a focused regression test for retry after rejection.

## Reproduction Assessment

Yes by source inspection: the non-Vitest path caches the first promise before it settles and later returns the same promise. I did not execute a live repro in this read-only review.

## Solution Assessment

Yes: clearing the cache on rejection is the narrowest maintainable fix because it preserves the performance cache for successful reads while allowing transient failures to retry.
