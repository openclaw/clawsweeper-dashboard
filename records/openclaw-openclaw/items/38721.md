## Summary

Keep open. Current main still has the gateway shutdown watchdog, bounded websocket close handling, and restart drains for command-queue tasks, embedded runs, and runtime dependency installs, but I did not find a gateway stop/restart path that enumerates, cancels, or bounded-drains ProcessSupervisor child/PTY runs. The discussion already separated the merged websocket fix in #61565 from this child-process/pipe-handle blocker, and the current code still supports that distinction.

## What This Changes

The issue reports gateway stop/restart timeouts on OpenClaw 2026.3.2 when active child-process and pipe handles remain referenced, and asks whether shutdown should cancel or detach those child processes plus how to debug the responsible command.

## Best Possible Solution

Add a small core ProcessSupervisor shutdown contract that can return privacy-safe active-run metadata, cancel all active child/PTY runs, and wait for bounded idle, then wire it into gateway stop/restart ahead of the force-exit watchdog while preserving the existing command-queue, embedded-run, runtime-deps, and websocket drains.

## Reproduction Assessment

Yes. The issue includes concrete Node diagnostic evidence of active process and pipe handles during shutdown, and current main has a source-level reproduction path: start a long-running supervised exec/PTY run, then stop or restart the gateway and observe that the shutdown path never asks ProcessSupervisor to cancel or drain it.

## Solution Assessment

Yes, a narrow supervisor API plus gateway wiring is the best maintainable direction. Relying only on systemd control-group cleanup would still leave OpenClaw with a dirty shutdown and would not give the gateway the bounded, observable cleanup path this report asks for.
