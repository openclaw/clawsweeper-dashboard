## Summary

Keep open: current main and v2026.5.22 still do not have a gateway-close contract to drain or cancel active process-supervisor child runs. The recently merged process-cancellation work helps timeout/cancel behavior, but it does not wire a supervisor drain into gateway shutdown, and the issue discussion already separates this from the websocket shutdown fix.

## What This Changes

Reports Ubuntu/systemd gateway shutdown timeouts with active child-process and pipe handles in Node diagnostic reports, asking whether exec/tool children should detach or cancel during shutdown and how to identify the stuck child command.

## Best Possible Solution

Confirm the symptom on current main under a managed systemd gateway; if it still reproduces, add a bounded gateway-owned supervisor child-run drain/cancel path that preserves service-managed process-group safety and keeps the broader session operator API in its canonical thread.

## Reproduction Assessment

No high-confidence current-main live reproduction has been run. Source inspection still shows supervised child runs can exist while gateway close lacks a process-supervisor drain hook, but service-managed child behavior changed after the original 2026.3.2 report.

## Solution Assessment

Unclear until current-main systemd proof and maintainer process-boundary policy are settled. The likely maintainable fix is a bounded gateway-owned supervisor drain/cancel hook, not treating the websocket close timeout fix or process-cancellation escalation as complete substitutes.
