## Summary

Keep this open: current main and v2026.6.6 still do not visibly cancel or reap active exec/process children during normal gateway stop, and the merged websocket-close fix is adjacent rather than a fix for this child-process shutdown blocker.

## What This Changes

The issue asks OpenClaw to fix gateway shutdown timeouts where active child-process handles and pipes can leave dirty shutdowns and lane waits during supervised stop/restart cycles.

## Best Possible Solution

Define one gateway stop policy that rejects new work, drains or aborts active runs through their canonical owners, cancels supervisor-owned child process scopes with a bounded grace window, and documents any preserved background-process exception.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source and test inspection show the relevant current behavior: normal shutdown does not abort active runs, while exec children are only cancelled through owner-specific paths.

## Solution Assessment

No, current main is not the complete solution. The best fix needs an owner-approved shutdown policy for active runs and child-process scopes rather than relying on the unrelated websocket close bound or manual process cleanup.
