## Summary

Keep open. Current main still has a gateway shutdown watchdog and bounded websocket shutdown, but there is no ProcessSupervisor-wide drain/cancel path wired into stop or restart, so the reporter's active child-process handle evidence still matches an unfixed source path.

## What This Changes

The issue asks OpenClaw to stop gateway shutdown from timing out when active exec/PTY child-process handles remain referenced during gateway stop or restart.

## Best Possible Solution

Add a small internal ProcessSupervisor shutdown contract and wire it into gateway stop/restart before the force-exit watchdog, with focused regression coverage for a long-running supervised child run.

## Reproduction Assessment

Yes, at source level. The report gives concrete Node diagnostic evidence with active process and pipe handles, and current main still has supervised exec/PTY runs without a gateway-wide drain/cancel path.

## Solution Assessment

Yes. A narrow supervisor active-run drain/cancel API plus gateway lifecycle wiring is the maintainable fix; relying on external supervisor cleanup would still leave OpenClaw with dirty in-process shutdown behavior.

## Repair Work Prompt

Fix the gateway dirty-shutdown bug where active ProcessSupervisor child/PTY runs can keep referenced process/pipe handles alive and trip the shutdown watchdog. Add a small internal supervisor shutdown API that can enumerate/cancel and bounded-wait active runs, then wire it into gateway stop/restart before the force-exit watchdog without adding user-facing CLI flags or changing session-kill feature scope. Preserve existing command-queue, embedded-run, websocket, HTTP, MCP, LSP, and plugin cleanup behavior. Add focused regression tests for a long-running supervised exec/PTY run being canceled/drained during gateway shutdown. Add a one-line CHANGELOG.md fix entry because this changes user-facing gateway shutdown behavior. Do not broaden this into the session-level exec-list/kill feature tracked by #12554.
