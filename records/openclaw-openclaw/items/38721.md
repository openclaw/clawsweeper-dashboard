## Summary

Keep open. Current main still has the gateway shutdown watchdog and supervised exec/PTY child runs, but no gateway lifecycle hook counts, drains, or cancels active ProcessSupervisor children, so the reporter's child-process handle evidence still maps to an unfixed source path; the merged WebSocket close fix addressed a different blocker.

## What This Changes

The issue asks OpenClaw to prevent dirty gateway stop/restart timeouts when active child-process and pipe handles from tool exec or internal subprocesses remain referenced.

## Best Possible Solution

Add a small internal ProcessSupervisor active-run count plus bounded cancel/drain hook and call it from gateway stop/restart before the watchdog, leaving public session-kill APIs and SIGTERM-first policy to separate tracks.

## Reproduction Assessment

Yes, at source level. Run the gateway with an active supervised exec or PTY child, then stop or restart it; current main has no gateway-wide ProcessSupervisor drain/cancel path, matching the reporter's active child-process and pipe-handle diagnostics.

## Solution Assessment

Yes. A narrow internal supervisor drain/cancel integration in the gateway lifecycle is the maintainable fix; the WebSocket close fix and broader session-process or signal-escalation issues cover adjacent but different work.

## Repair Work Prompt

Fix the gateway shutdown timeout caused by active ProcessSupervisor exec/PTY child runs. On current main, supervised exec runs are spawned through `getProcessSupervisor()`, but gateway stop/restart only drains command-queue tasks and embedded runs; it does not count, wait for, or cancel active ProcessSupervisor children before the shutdown watchdog. Add a small internal ProcessSupervisor active-run count and bounded cancel/drain/wait contract, wire it into gateway stop/restart before force exit, and add focused regression coverage. Do not add public `sessions_kill`/exec-list APIs, do not redesign SIGTERM-first cancellation policy from https://github.com/openclaw/openclaw/issues/66399, and do not change service-manager process-group policy beyond what is needed for this gateway drain. Include a changelog entry if the fix is user-visible.
