## Summary

Keep this open: current main has bounded several shutdown phases and fixed a separate websocket close hang, but normal gateway stop still does not visibly cancel or reap active exec/process children before the shutdown watchdog can fire.

## What This Changes

The issue reports gateway shutdown timeouts on Ubuntu/systemd when Node diagnostic reports show active referenced child-process and pipe handles during stop/restart.

## Best Possible Solution

Add an owner-approved gateway stop policy that rejects new command work, aborts or drains active agent/process children through their canonical owners, and gives them a bounded settle window before the watchdog exits dirty.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection shows a plausible current-main path: normal stop skips the restart active-work drain while exec/process children terminate only when their owner calls abort or cancel.

## Solution Assessment

No, current main is not the complete solution for this report. The websocket close timeout fix and restart drain are useful, but the best fix should define one shutdown-owned child-process cancellation policy instead of relying on unrelated close steps.
