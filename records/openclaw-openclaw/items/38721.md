## Label Justifications

- `P1`: The report describes repeated dirty gateway shutdowns and lane waits in a real systemd install, which can break an active agent/channel workflow.
- `impact:session-state`: The unresolved child process can outlive or delay gateway shutdown, leaving session and lane state in a dirty or delayed state.

## Summary

Keep open: current main still has a source-backed shutdown timeout path where supervised exec/CLI child processes can remain active, while the gateway shutdown handler has no process-supervisor cancel or drain step. The websocket-server timeout fixed by https://github.com/openclaw/openclaw/pull/61565 is adjacent but does not cover the child-process handle root cause reported here.

## What This Changes

The issue reports dirty gateway shutdowns on systemd where Node diagnostic reports show active referenced child-process and pipe handles during the shutdown timeout.

## Best Possible Solution

Add a bounded internal gateway shutdown step that cancels or drains active gateway-owned supervised process runs, while leaving session-wide operator controls to https://github.com/openclaw/openclaw/issues/12554 and signal-escalation policy to https://github.com/openclaw/openclaw/issues/66399.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection plus the reporter's Node diagnostic reports provide a credible current-main path: supervised children can remain active while gateway shutdown lacks a process-supervisor cleanup hook.

## Solution Assessment

Yes. The narrow maintainable solution is internal gateway shutdown cleanup for active supervised processes; broader session operator controls and signal escalation are already tracked separately.
