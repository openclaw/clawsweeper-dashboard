## Summary

Keep open: current main and v2026.6.5 still have the shutdown watchdog and lower-level process-supervisor pieces, but I did not find a gateway shutdown path that cancels or reaps OpenClaw-owned child exec handles before the dirty-shutdown timeout. The websocket close fix in https://github.com/openclaw/openclaw/pull/61565 addresses a different blocker, and https://github.com/openclaw/openclaw/issues/12554 tracks the broader operator session-kill API rather than this shutdown lifecycle bug.

## What This Changes

The issue reports gateway stop/restart timeouts where Node reports show referenced child-process and pipe handles, followed by dirty shutdown and lane-wait symptoms.

## Best Possible Solution

Define a shutdown policy that cancels or reaps only OpenClaw-owned active exec/process-supervisor runs during gateway stop/restart, with bounded grace and diagnostics, while leaving the broader operator-facing session-kill API to the linked design item.

## Reproduction Assessment

Yes at source level: current main shows no gateway stop path that cancels supervised exec scopes or child-process handles before the shutdown watchdog. I did not run a live Ubuntu/systemd reproduction with Node diagnostic reports.

## Solution Assessment

Unclear as an implementation shape: the fix belongs in gateway shutdown/process ownership, but maintainers need to choose cancel versus detach versus bounded wait semantics before automation writes code.
