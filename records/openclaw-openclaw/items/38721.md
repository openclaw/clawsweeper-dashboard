## Summary

Keep open. Current main still has the outer gateway shutdown watchdog and bounded WebSocket close path, but supervised exec/PTY child runs are not counted, drained, or cancelled during gateway stop/restart, so the reporter's active child-process handle evidence still matches an unfixed source path.

## What This Changes

The issue asks for gateway shutdown/restart to drain or cancel supervised child processes so active process and pipe handles do not cause dirty shutdown timeouts and lane waits.

## Best Possible Solution

Add a small internal ProcessSupervisor active-run drain/cancel contract and wire it into gateway stop/restart before the force-exit watchdog, keeping public session-wide process controls and SIGTERM-first cancellation policy in their existing separate discussions.

## Reproduction Assessment

Yes, at source level. The report gives concrete Node diagnostic evidence with active process and pipe handles, and current main still has supervised exec/PTY/CLI child runs without a gateway-wide shutdown drain or cancel path.

## Solution Assessment

Yes. A narrow supervisor active-run drain/cancel API plus gateway lifecycle wiring is the maintainable fix; related WebSocket and subsystem-timeout work solves adjacent blockers but not the child-process handle path.
