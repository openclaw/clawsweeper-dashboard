## Summary

Keep open. Current main still lacks the requested Linux Sharp-failure fallback, but this PR is not merge-ready because the proposed `vipsthumbnail` invocation uses an unsupported quality option and the new decoder path needs guard parity review before it handles untrusted images.

## What This Changes

The PR adds a Linux x64 `vipsthumbnail` fallback in `src/media/image-ops.ts` for `resizeToJpeg()` when the Sharp-backed media-understanding image ops path fails.

## Best Possible Solution

Keep the PR open for revision: use a libvips-compatible output spec for JPEG quality, preserve the existing decoded-pixel and backend override contracts, add focused tests plus the required changelog entry, then have a maintainer decide whether a system `vipsthumbnail` backend is acceptable for Linux x64 fallback.

## Reproduction Assessment

Yes. Current main can be exercised by forcing the Sharp import or resize path to fail on Linux/non-sips and then calling `optimizeImageToJpeg()` or uploading an image; the PR-specific fallback failure is source-verified because libvips 8.15.1 does not define a `-Q` `vipsthumbnail` option.

## Solution Assessment

No. A system libvips fallback may be a reasonable direction, but this implementation is not the narrowest safe fix because the CLI arguments are incompatible and the new decoder path does not yet show guard parity with the Sharp backend.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass quality through the vips output spec:** `src/media/image-ops.ts:484`
  - body: `vipsthumbnail` does not accept a standalone `-Q` option in libvips 8.15.1; quality is passed in the output spec, for example `out.jpg[Q=85]`. As written, the fallback exits with an option error and `resizeToJpeg()` rethrows the original Sharp failure, so the target CPU path remains broken.
  - confidence: 0.94
- **[P2] Preserve the decoded-pixel cap for vips:** `src/media/image-ops.ts:484`
  - body: The Sharp backend wraps every decode with `limitInputPixels`, but this fallback invokes `vipsthumbnail` directly after only the header preflight in `assertImagePixelLimit()`. Formats outside that header parser can reach libvips without the existing decoded-pixel cap, so add an equivalent guard or restrict the fallback to formats the preflight can bound.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/media/image-ops.ts:623-624`
  - body: This is a user-facing `fix` for image uploads on affected Linux systems, so repo policy requires a `CHANGELOG.md` entry before merge. Add a single-line entry with appropriate contributor credit.
  - confidence: 0.87
