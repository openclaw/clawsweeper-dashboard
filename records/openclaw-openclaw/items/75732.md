## Summary

Keep open: current main still has no Linux `vipsthumbnail` fallback, and this PR targets a real linked image-upload bug, but the patch is not ready because the libvips command is incompatible, the new decoder path lacks guard parity, and the user-facing fix needs a changelog entry plus maintainer approval.

## What This Changes

The PR modifies `src/media/image-ops.ts` to add a Linux x64 `vipsthumbnail` fallback after Sharp-backed JPEG resizing fails, including an `OPENCLAW_IMAGE_BACKEND=vips` override.

## Best Possible Solution

Land a corrected media image fallback that uses libvips output-spec quality, preserves decoded-pixel guard parity, adds focused tests and changelog coverage, and has maintainer approval for the backend policy.

## Reproduction Assessment

Yes. Source inspection shows current main still delegates Linux non-sips JPEG resizing to Sharp, so a Sharp load/resize failure flows through `optimizeImageToJpeg()` to `Failed to optimize image`; the PR-specific fallback failure is source-verified from the `vipsthumbnail` option contract.

## Solution Assessment

No. A libvips fallback may be the right surface for this bug, but this implementation is not the best fix until the CLI invocation works, guard parity is restored, changelog coverage is added, and the external decoder policy is approved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass quality through the vips output spec:** `src/media/image-ops.ts:486`
  - body: `vipsthumbnail` does not accept `-Q` as a standalone option; JPEG quality has to be part of the output spec, such as `out.jpg[Q=85]`. With the current arguments, the fallback exits with an option error and `resizeToJpeg()` rethrows the original Sharp failure, so the target SSE2-only path remains broken.
  - confidence: 0.94
- **[P2] Preserve the decoded-pixel limit in the vips path:** `src/media/image-ops.ts:484-486`
  - body: The existing Sharp backend decodes with `limitInputPixels`, but this fallback hands the uploaded buffer to `vipsthumbnail` after only the header preflight. For formats the header parser does not understand but libvips can decode, the new path has no equivalent decoded-size budget before external processing.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/media/image-ops.ts:623-624`
  - body: This is a user-facing `fix` for image uploads on affected Linux systems, so repo policy requires a single-line `CHANGELOG.md` entry under the active Fixes section with appropriate contributor credit.
  - confidence: 0.88
