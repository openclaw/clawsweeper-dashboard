## Summary

Keep open. The linked bug is plausible and this PR is the live closing implementation, but the branch is not mergeable: it references undefined helpers, calls `vipsthumbnail` with an unsupported standalone `-Q`, omits guard parity for the new decoder path, and lacks inspectable real behavior proof; current main only has the narrower under-cap original-image fallback from https://github.com/openclaw/openclaw/pull/77117.

## What This Changes

Adds a Linux x64 `vipsthumbnail` fallback in `src/media/image-ops.ts` when Sharp-backed JPEG resizing fails.

## Best Possible Solution

Keep the Linux compatibility direction, but implement it with defined temp helpers, the correct libvips output contract, equivalent image safety guards, focused tests, maintainer approval for the backend policy, and real after-fix proof from the affected environment.

## Reproduction Assessment

Yes for source-level review: current main can still reach Sharp-backed resizing, and the PR-specific failures are visible from the diff and upstream `vipsthumbnail` option source. The exact SSE2-only KVM failure remains reporter/PR-author verified rather than live-reproduced here.

## Solution Assessment

No. A libvips fallback may be the right direction, but this implementation is not the best fix because it does not compile, uses the wrong CLI contract, lacks guard parity, and still needs backend-policy approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Define the temp-file helpers before using them:** `src/media/image-ops.ts:472-475`
  - body: `vipsResizeToJpeg()` calls `withTempDir`, `path.join`, `fs.writeFile`, and `fs.readFile`, but this PR does not import or define those symbols in `src/media/image-ops.ts`. The branch will fail typecheck/build before the fallback can run.
  - confidence: 0.96
- **[P1] Pass JPEG quality through the vips output spec:** `src/media/image-ops.ts:483-485`
  - body: `vipsthumbnail` does not expose `-Q` as a standalone option, so this fallback command will fail and rethrow the original Sharp error. Put JPEG quality in the output spec, for example `out.jpg[Q=85]`, or use the supported libvips API for quality options.
  - confidence: 0.93
- **[P2] Preserve the decoded-pixel guard in the vips path:** `src/media/image-ops.ts:483-486`
  - body: The Sharp backend enforces `limitInputPixels`, but the new external decoder path runs `vipsthumbnail` after only the current header preflight. Add an equivalent decoded-size guard or restrict the fallback to inputs whose dimensions have been positively bounded before invoking libvips.
  - confidence: 0.83
