## Summary

This PR should stay open because it targets a real current-main gap, but the proposed fallback still has merge-blocking correctness and security-policy issues. Current main continues to route Linux image resizing through the Sharp-backed bundled media image ops path, while the PR adds a new `vipsthumbnail` execution path whose CLI arguments and guard parity need correction before maintainer acceptance.

## What This Changes

The PR adds a Linux x64 `vipsthumbnail` fallback in `src/media/image-ops.ts` for JPEG resizing when the Sharp-backed resize path fails and the backend is not explicitly forced to `sharp` or `sips`.

## Best Possible Solution

Keep the Linux compatibility fix in this PR or a replacement branch, but make the libvips invocation match the `vipsthumbnail` contract, preserve the existing image-size safety contract, add focused tests and a changelog entry, then have a maintainer approve whether a system `vipsthumbnail` backend is acceptable for OpenClaw runtime image optimization.

## Reproduction Assessment

Yes. The linked report gives a concrete runtime reproduction on Ubuntu under a Proxmox Common KVM/SSE2-only CPU, and current main still lacks a fallback for Sharp load failures on that path. The PR-specific breakage is reproducible from the command contract: `vipsthumbnail` has no standalone `-Q` option.

## Solution Assessment

No. A libvips fallback may be a viable direction, but this patch is not yet the best implementation because the CLI call is incompatible and the new decoder path does not preserve the Sharp path's decoded-pixel guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass quality through the vips output spec:** `src/media/image-ops.ts:484`
  - body: `vipsthumbnail` does not accept `-Q` as a standalone option; JPEG quality must be part of the output spec, for example `out.jpg[Q=85]`. With the current arguments, the fallback exits with an option error and `resizeToJpeg()` rethrows the original Sharp failure, so the target SSE2-only path remains broken.
  - confidence: 0.94
- **[P2] Preserve the decoded-pixel limit in the vips path:** `src/media/image-ops.ts:484`
  - body: The existing Sharp backend decodes with `limitInputPixels`, but this fallback hands the uploaded buffer to `vipsthumbnail` after only the header preflight. Add an equivalent decoded-size guard or restrict the fallback to formats whose dimensions are bounded by the existing preflight before invoking the external decoder.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/media/image-ops.ts:623-624`
  - body: This is a user-facing `fix` for image uploads on affected Linux systems, so repo policy requires a `CHANGELOG.md` entry before merge. Add a single-line entry under the active Fixes section with appropriate contributor credit.
  - confidence: 0.88
