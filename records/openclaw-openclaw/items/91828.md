## Summary

Keep open: the PR still targets a real unfixed memory-wiki bridge reliability bug on current main, and the branch is clean/mergeable with no concrete line-level correctness finding. The remaining concerns are maintainer-visible merge risk around shared in-flight wiki state, overlap with a narrower retry PR, and the gap between local gateway/load proof and the reporter's larger WSL2 workload.

## What This Changes

The PR adds bounded `path-mismatch` retry handling for imported memory-wiki source-page writes and coalesces concurrent bridge-mode imported-source sync plus index refresh for equivalent vault/config inputs.

## Best Possible Solution

Land one canonical memory-wiki reliability fix that preserves fs-safe safety failures, retries only transient `path-mismatch`, coalesces equivalent bridge imports, and leaves any broader batching/backpressure work to a follow-up if reporter-scale proof demands it.

## Reproduction Assessment

## Solution Assessment

Yes, this is a good broad fix shape: caller-side retry keeps fs-safe policy failures intact, and source-boundary coalescing covers gateway, tool, and CLI callers without adding a bridge-only synchronization layer. The main remaining decision is whether maintainers want this broader PR instead of the narrower overlapping retry PR.
