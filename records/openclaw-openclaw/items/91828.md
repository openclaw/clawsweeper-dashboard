## Best Possible Solution

Land one canonical memory-wiki bridge reliability fix if maintainers accept the coalescing identity and proof scope; otherwise narrow this PR to the accepted retry behavior and leave broader coalescing to the canonical issue.

## Reproduction Assessment

Yes, at source level: current main still routes the affected gateway/tool methods through imported-source sync without coalescing, and the remaining stat/read path is outside the existing guarded write retry. ## Solution Assessment

Yes, this is a maintainable owner-boundary fix if maintainers accept the scope: it keeps fs-safe strict, builds on the shared guarded write helper, retries only the transient imported-source path-mismatch class, and coalesces at the source-sync boundary. A global fs-safe retry or duplicate bridge-only helper would be broader or more divergent.
