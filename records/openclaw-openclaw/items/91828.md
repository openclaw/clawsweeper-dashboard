## Summary

Keep this PR open: current main still has the memory-wiki bridge hard-fail/duplicate-sync behavior, and the patch is a plausible targeted fix with no concrete blocking code finding from this review. The remaining gate is maintainer acceptance of the same-key coalescing freshness tradeoff plus normal broad CI/Testbox proof before merge.

## What This Changes

The PR adds bounded fs-safe `path-mismatch` retries for memory-wiki imported source page writes and coalesces concurrent bridge imported-source sync plus index refresh by vault/config/agent workspace inputs.

## Best Possible Solution

Land after maintainers accept the source-sync coalescing freshness tradeoff and broad CI/Testbox covers the wider type/build surface; otherwise split the retry-only hardening from the coalescing policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source inspection shows the current gateway methods sync imported sources before `wiki.status`, `wiki.importInsights`, and `wiki.palace`, while `writeImportedSourcePage` turns fs-safe `path-mismatch` into a hard error without retry; I did not run a live reproduction in this read-only checkout.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes: caller-side retry in the shared source page writer is the narrow boundary that covers bridge and unsafe-local imports while preserving fs-safe refusal cases. The coalescing layer is a reasonable load mitigation, but its freshness tradeoff needs explicit maintainer acceptance.
