## Summary

This PR fixes the current-main missing-key warning through the correct generic provider metadata, but its new “ready” wording exceeds that contract: Ollama also has `requiresCredential: false` while still requiring a running service and `ollama signin`. The branch should stay open for a narrow wording and regression-test repair.

Likely related people: @vincentkoc is the recent contributor to the current wizard finalize path; @NormallyGaussian introduced the merged Parallel Free default that exposed the bug and has prior merged history in this area.

## What This Changes

The PR updates onboarding finalize messaging so enabled keyless web-search providers are reported as ready, adds matching English and Chinese locale strings, and adds regression coverage.

## Best Possible Solution

Keep the generic `requiresCredential` branch, but make its copy state only the guaranteed fact that no API key is required; do not assert full readiness unless the provider contract gains a separate verified-prerequisite signal.

## Reproduction Assessment

Yes, from source inspection. Current main deterministically sends an enabled keyless provider without a stored credential to the missing-key branch, while the PR’s Ollama wording regression follows directly from the same provider metadata and setup contracts.

## Solution Assessment

No, not in its current wording. Branching on the existing generic metadata is the narrowest maintainable fix, but that metadata guarantees only that an API credential is unnecessary—not that every external prerequisite has been satisfied.
