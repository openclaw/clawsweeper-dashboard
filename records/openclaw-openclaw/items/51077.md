## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the PR's central status-item behavior because left-click opens the dismissing panel path, while the PR branch introduces merge-blocking shared iOS chat regressions, duplicate macOS visibility callbacks, and unrelated direct Gemini image-generation/secret-handling changes that bypass the existing provider runtime.

## What This Changes

The PR changes macOS menu-bar chat to toggle a persistent resizable NSWindow, adds toolbar/settings Talk controls, drag-and-drop and resizable composer behavior, and later adds direct Gemini image-generation UI/service/metadata changes in the shared chat surface.

## Best Possible Solution

Keep the useful source work but ask for a rebased, narrow macOS PR: wire status-item left-click to the existing persistent NSWindow path, add a macOS-only resizable composer, preserve or drop already-shipped drag/drop and Talk crash overlap as appropriate, and move Gemini/image-generation work to the existing image_generate provider/plugin runtime in a separate product and security review.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: current main routes status-item left-click through MenuBar.toggleWebChatPanel() to WebChatManager.togglePanel(), and that panel closes on outside clicks; the PR diff also clearly applies the fixed composer/resize handle to shared OpenClawChatView, which apps/ios/Sources/Chat/ChatSheet.swift uses.

## Solution Assessment

No. The persistent-window direction is plausible, but this branch is not the best solution because it mixes platform-specific UI with shared iOS layout, double-calls visibility callbacks, and adds a direct provider path instead of using the existing image-generation runtime and secrets model.
