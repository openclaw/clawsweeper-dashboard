## Summary

Keep open. Current main does not fully replace the macOS status-item chat panel with the requested persistent window/resizable composer, and this PR still contains potentially useful source work. It is not merge-ready because the branch mixes macOS-only UI changes into shared iOS chat code, double-fires window visibility callbacks, and adds direct Gemini image-generation credential/network handling outside the existing provider runtime.

## What This Changes

The PR changes macOS status-item chat to use a persistent resizable NSWindow, adds toolbar/settings Talk controls, drag-and-drop and resizable composer behavior, and adds direct Gemini image-generation UI, service, and image metadata code in the shared chat surface.

## Best Possible Solution

Keep the useful macOS source work, but ask for a narrowed rebased PR or replacement branch: status-item left-click should use the existing persistent NSWindow path, composer resizing should be macOS-only, already-shipped drag/drop and Talk crash overlap should be reconciled with current main, and image generation should go through the existing image_generate provider/runtime and SecretRef/auth surfaces instead of a direct app client.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: the PR's shared ChatView layout is used by apps/ios/Sources/Chat/ChatSheet.swift, WebChatManager.close() already emits visibility changes through the controller callback, and the direct image-generation branch returns before transport.sendMessage.

## Solution Assessment

No. The persistent-window direction is plausible, but this branch is not the best implementation because it mixes unrelated provider/security work with macOS UI changes and puts macOS-only resize behavior in the shared iOS chat view.

  - body: This branch bypasses transport.sendMessage whenever isImageGenModel matches, then appends generated results only to local message state.   - confidence: 0.84
