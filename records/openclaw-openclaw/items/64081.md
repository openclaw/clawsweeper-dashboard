## Summary

Keep open: current main and v2026.6.6 still default implicit deliverable ACP/subagent and Task Flow notifications to `done_only`, and there is no typed or documented config-level default task notify policy. The request remains valid, but the public config shape needs maintainer product direction.

## What This Changes

Request a config-level default notification policy for background task completion messages, especially ACP and subagent tasks, while preserving current behavior when unset.

## Best Possible Solution

Add a typed core config surface only after maintainers choose global versus per-runtime semantics; explicit task or flow `notifyPolicy` should win, unset config should preserve current defaults, and docs/tests should cover ACP, subagent, Task Flow, cron, CLI, and media siblings.

## Reproduction Assessment

Yes at source level. Current main creates deliverable ACP/subagent tasks without an explicit notify policy, defaults them to `done_only`, and exposes no schema-backed default config key.

## Solution Assessment

Unclear until maintainers choose the public policy shape. A config default is plausible, but the best implementation must avoid overlapping with narrower channel-specific work such as https://github.com/openclaw/openclaw/issues/87665 and preserve existing explicit task policy behavior.
