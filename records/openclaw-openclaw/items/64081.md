## Summary

Keep open: current main and the latest release still lack a config-level default task notification policy. The source, docs, and related discussion show this remains a real product/config request rather than an already-implemented fix or a duplicate; narrower related issues either point back here or cover adjacent delivery bugs.

## What This Changes

The issue requests a config-level default task notification policy so users can suppress raw ACP/subagent background task completion messages without racing per-task CLI overrides.

## Best Possible Solution

Add a backwards-compatible core task default resolver only after maintainers choose the config shape; preserve current behavior when unset, keep explicit task/flow notifyPolicy values authoritative, keep cron/CLI/media silent defaults intact, and update schema/docs/tests together.

## Reproduction Assessment

Yes, at source level: current main still defaults deliverable session tasks without explicit notifyPolicy to done_only, while the strict agent defaults schema and docs expose no config-level task notify default.

## Solution Assessment

Unclear. The request is valid, but the best solution depends on maintainer choice of global versus per-runtime defaults and how to coordinate channel-specific visibility needs without putting owner-specific policy in core.
