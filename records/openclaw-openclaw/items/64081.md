## Summary

Keep open. Current main and v2026.6.6 still hardcode implicit deliverable task notification defaults to `done_only` and expose no schema-backed default task notify config; related items are narrower or adjacent and do not implement this requested config surface.

## What This Changes

The issue requests a config-level default notify policy for ACP, subagent, cron, and CLI background task completion notifications.

## Best Possible Solution

Add a backward-compatible task notify default resolver under a maintainer-approved agent-defaults config shape, preserve current behavior when unset, keep explicit task/flow policies authoritative, and update schema, types, docs, and focused tests together.

## Reproduction Assessment

Yes at source level: current main defaults deliverable task records and task flows to `done_only` when no `notifyPolicy` is passed, while ACP/subagent creation paths omit it and docs only document per-task CLI override. I did not run a live Telegram scenario in this read-only review.

## Solution Assessment

Unclear pending maintainer direction. A config-level default is a plausible core solution, but the exact global versus per-runtime shape becomes a public config contract and should be approved before implementation.
