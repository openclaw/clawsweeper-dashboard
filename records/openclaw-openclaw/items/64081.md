## Summary

Keep this issue open. Current main still lacks a typed, schema-backed, or documented config-level default task notification policy; ACP, subagent, and Task Flow records still fall back to hardcoded `done_only` unless the caller supplies `notifyPolicy`, while #73905 only resolved the narrower cron-spawned subagent notification behavior.

## What This Changes

Add a config-level default task notification policy so users can suppress or tune raw background task completion messages while preserving explicit per-task overrides.

## Best Possible Solution

Add a backwards-compatible core config surface, likely under a maintainer-approved `agents.defaults.tasks` shape, resolve the effective policy before ACP, subagent, and Task Flow records are created, preserve current behavior when unset, keep explicit task/flow `notifyPolicy` authoritative, and document the behavior on the public tasks, subagents, ACP, and config pages.

## Reproduction Assessment

Yes. This is a feature/config gap rather than a live runtime bug: static inspection shows no accepted config key in the strict agent-defaults schema/types/docs, and the ACP/subagent/Task Flow creation paths still omit `notifyPolicy` so deliverable tasks inherit `done_only`.

## Solution Assessment

Unclear because there is no patch and the public API shape is the main unresolved question. The maintainable direction is one coherent config contract across task registry, ACP, subagent, and Task Flow creation rather than runtime-specific partial knobs.
