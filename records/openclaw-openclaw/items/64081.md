## Summary

Keep open: current main and the latest release still lack a config-level default task notify policy, while ACP, subagent, and Task Flow creation paths continue to fall back to hardcoded notify defaults unless a caller passes an explicit policy.

## What This Changes

The issue requests a config-level default notify policy for background task completion notifications so operators can suppress raw subagent and ACP task banners by default.

## Best Possible Solution

Add a maintainer-approved, backward-compatible task notify default resolver that preserves current behavior when unset, lets explicit task or flow policies win, keeps cron/CLI/media silent defaults intact, and updates schema, docs, and focused tests together.

## Reproduction Assessment

Yes, at source level: unset deliverable task policies still resolve to `done_only`, ACP and subagent creation paths omit `notifyPolicy`, and current schema/docs expose no default policy key. I did not run a live Telegram reproduction because the source contract already proves the reported default behavior.

## Solution Assessment

Unclear: a config-level control is a plausible maintainable solution, but maintainers still need to choose the public config path and semantics before implementation. A safe fix should avoid channel-specific policy in core and keep current behavior when unset.
