## Summary

Keep open. Current main still has hardcoded task notification defaults and no typed, schema-backed, documented config-level task notify policy; the related ACP binding issue is narrower, and the cron-specific issue was closed because that cron path is already handled.

## What This Changes

The issue requests a config-level default notification policy for background tasks so users can suppress raw ACP and subagent completion messages without racing per-task overrides.

## Best Possible Solution

Add one backward-compatible `agents.defaults.tasks` notify-default contract, resolve it before ACP, subagent, and Task Flow records are created, preserve current defaults when unset, and document precedence rules.

## Reproduction Assessment

Yes, for the configuration gap: current-main source inspection shows hardcoded notify defaults and strict agent-defaults config with no task notify key. This is source-reproducible feature work rather than a failing bug reproduction.

## Solution Assessment

Unclear on exact shape because there is no patch and the public config contract needs maintainer choice. The maintainable direction is one coherent, backward-compatible default policy shared across task registry, Task Flow, ACP, and subagent creation.
