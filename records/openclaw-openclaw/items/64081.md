## Summary

Keep open. Current main still lacks a typed, schema-backed, or documented config-level default task notification policy, and the related reports/PRs only cover narrower delivery symptoms rather than the requested config contract.

## What This Changes

The issue requests a config-level default task notification policy so ACP, subagent, and Task Flow completion messages can be silenced without per-task races.

## Best Possible Solution

Add a backward-compatible `agents.defaults.tasks`-level config contract that resolves an effective notify policy before task or flow records are created, preserves current behavior when unset, and documents explicit-policy precedence.

## Reproduction Assessment

Yes for the configuration gap: source inspection on current main shows fixed defaults and no accepted config key in the strict agent-defaults schema/types/docs. This is a source-level feature gap rather than a failing bug reproduction.

## Solution Assessment

Unclear because there is no patch and the public API shape is the central unresolved question. The maintainable direction is one coherent config contract across task registry, ACP, subagent, and Task Flow creation rather than a runtime-specific partial knob.
