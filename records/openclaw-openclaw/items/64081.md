## Summary

Keep this issue open. Current main still has no typed, schema-backed, or documented config-level default task notification policy, and the ACP, subagent, and Task Flow creation paths still fall back to hardcoded `done_only` unless a caller supplies `notifyPolicy`.

## What This Changes

Add a config-level default task notification policy so users can suppress or tune raw background task completion messages while preserving explicit per-task overrides.

## Best Possible Solution

Add a backwards-compatible core config surface, likely under a maintainer-approved `agents.defaults.tasks` shape, that resolves the effective policy before ACP, subagent, and Task Flow records are created while preserving existing defaults when unset.

## Reproduction Assessment

Yes for the configuration gap. Source inspection on current main shows no accepted config key in the strict agent-defaults schema/types/docs, and the main ACP, subagent, and Task Flow creation paths still omit `notifyPolicy`, causing deliverable records to inherit `done_only`.

## Solution Assessment

Unclear because there is no patch and the public API shape is the central unresolved question. The maintainable direction is one coherent config contract across task registry, ACP, subagent, and Task Flow creation rather than a runtime-specific partial knob.
