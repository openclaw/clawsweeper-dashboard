- `impact:message-loss`: The issue concerns task completion messages being delivered to requester channels or sessions without a configurable policy to suppress or tune them.

## Summary

Keep open. Current main still lacks a typed, schema-backed, documented config-level task notify default; ACP, subagent, and Task Flow paths still rely on hardcoded defaults, while related items only cover narrower delivery symptoms.

## What This Changes

Requests a config-level default notify policy so ACP and subagent task completion notifications can be silenced or tuned without racing per-task CLI changes.

## Best Possible Solution

Add one backward-compatible `agents.defaults.tasks` notify-default contract, resolve it before ACP, subagent, and Task Flow records are created, preserve current defaults when unset, and document the precedence rules.

## Reproduction Assessment

Yes, for the configuration gap: current-main source inspection shows hardcoded notify defaults and strict agent-defaults config with no task notify key. This is source-reproducible feature work rather than a failing bug reproduction.

## Solution Assessment

Unclear on exact shape because there is no patch and the public config contract needs maintainer choice. The maintainable direction is one coherent, backward-compatible default policy shared across task registry, Task Flow, ACP, and subagent creation.
