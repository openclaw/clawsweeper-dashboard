## Summary

Keep open. Current main and the latest release still have hardcoded task-notification defaults and no schema-backed config key for a default task notify policy; the remaining work is a maintainer product/config decision, not a completed cleanup target.

## What This Changes

The issue requests a config-level default task notify policy so ACP and subagent background task completions can default to `silent`, `done_only`, or `state_changes` without racing per-task CLI updates.

## Best Possible Solution

Add one maintainer-approved, backward-compatible task notify default resolver under agent defaults, preserve current behavior when unset, keep explicit per-task policies authoritative, and align schema, docs, and focused task/flow tests.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` show ACP, subagent, and Task Flow paths falling back to hardcoded `done_only` unless a caller supplies `notifyPolicy`, and the agent-defaults schema has no task notify default. I did not run a live Telegram scenario in this read-only cleanup review.

## Solution Assessment

Unclear pending maintainer direction. A config-level default is a plausible core solution, but maintainers should choose the exact global versus per-runtime shape before implementation because this becomes a public config contract.
