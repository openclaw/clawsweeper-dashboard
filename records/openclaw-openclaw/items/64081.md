## Summary

Keep open. Current main and the latest release still have hardcoded task notification defaults and no schema-backed config key for a default task notify policy; related items are either narrower cron behavior that was closed as already implemented or adjacent ACP delivery work, not a canonical config implementation.

## What This Changes

The issue requests a config-level default notification policy so ACP and subagent task completions can default to `silent`, `done_only`, or `state_changes` without racing per-task CLI updates.

## Best Possible Solution

Add one maintainer-approved, backward-compatible task notify default resolver under agent defaults, keep explicit per-task policies authoritative, preserve existing behavior when unset, and align schema, types, docs, and focused task/flow tests.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` show ACP, subagent, and Task Flow paths falling back to hardcoded `done_only` unless a caller supplies `notifyPolicy`, and the agent-defaults schema has no task notify default. I did not run a live Telegram scenario, so the proof is static rather than end-to-end.

## Solution Assessment

Unclear pending maintainer direction. A config-level default is a plausible core solution, but maintainers should choose the exact global versus per-runtime shape before implementation because this becomes a public config contract.
