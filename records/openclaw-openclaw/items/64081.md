## Summary

Keep open: current main and v2026.6.8 still lack a config-level default task notification policy, while ACP, subagent, and Task Flow defaults remain source-backed `done_only` unless callers pass an explicit policy. Related items are narrower channel/runtime visibility work, not a merged canonical implementation of this requested config surface.

## What This Changes

Requests a config-level default task notification policy so users can change ACP/subagent task completion message behavior without racing per-task `openclaw tasks notify` overrides.

## Best Possible Solution

Add a maintainer-approved, backward-compatible task notify default resolver that preserves current behavior when unset, lets explicit task/flow policies win, keeps cron/CLI/media silent defaults intact, and updates schema, docs, and focused tests together.

## Reproduction Assessment

Yes. At source level, ACP/subagent creation paths omit `notifyPolicy`, task and flow registries default unset deliverable policies to `done_only`, and the current schema/docs expose only per-task `openclaw tasks notify`.

## Solution Assessment

Unclear. The requested config-level control is valid, but maintainers still need to choose global versus per-runtime or inherited defaults and avoid channel-specific policy leaking into core.
