## Summary

Keep this issue open. Current main still has no typed, schema-backed, or documented config-level default task notification policy; the raw delivery behavior is now more nuanced for subagents, but the requested public config surface still does not exist.

## What This Changes

The issue requests a config-level default task notification policy so users can suppress or tune raw background task completion messages while preserving explicit per-task overrides.

## Best Possible Solution

Add a backwards-compatible core config surface, likely under a maintainer-approved `agents.defaults.tasks` shape, that resolves the effective default before ACP, subagent, and Task Flow records are created while preserving today’s behavior when unset.

## Reproduction Assessment

Yes for the configuration gap. Source inspection on current main shows no accepted config key in the strict agent-defaults schema/types/docs, and the ACP, subagent, and Task Flow creation paths still do not read a config-level default notify policy.

## Solution Assessment

Unclear because there is no patch and the public API shape is the central unresolved question. The maintainable direction is one coherent config contract across task registry, ACP, subagent, and Task Flow creation rather than a runtime-specific partial knob.
