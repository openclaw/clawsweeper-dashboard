## Summary

Keep open: current main and v2026.6.10 still lack a config-level default task notify policy, and the ACP, subagent, and Task Flow creation paths continue to fall back to hardcoded notify defaults unless a caller passes `notifyPolicy`. Related open issues cover announce suppression or narrower Discord/Codex visibility problems rather than replacing this broader config-default decision.

## What This Changes

The issue requests a config-level default task notify policy so users can suppress or change background ACP/subagent task completion notifications without racing per-task overrides.

## Best Possible Solution

Add a maintainer-approved, backward-compatible task notify default resolver that preserves current behavior when unset, lets explicit task or flow policies win, keeps cron/CLI/media silent defaults intact, and updates schema, docs, and tests together.

## Reproduction Assessment

Yes. Source inspection shows deliverable tasks without explicit `notifyPolicy` still resolve to `done_only`, and the ACP/subagent callers omit that field; I did not run a live Telegram repro because the source contract directly proves the reported default behavior.

## Solution Assessment

Unclear. A config-level default is plausible, but maintainers still need to choose the public config path and semantics before implementation; a safe design should keep explicit per-task policies winning and preserve cron/CLI/media silent defaults.
