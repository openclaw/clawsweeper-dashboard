- `impact:message-loss`: The issue concerns unwanted task completion messages being delivered to requester channels instead of being suppressible by policy.

## Summary

Keep open. Current main still has hardcoded task notify defaults and no typed, schema-backed, documented config-level default, while related work only covers narrower ACP or cron/subagent symptoms rather than this broader config contract.

## What This Changes

Add a user-configurable default task notification policy for background tasks so users can make ACP/subagent task completions silent by default instead of changing each task after creation.

## Best Possible Solution

Add one backward-compatible task notify default contract, likely under `agents.defaults.tasks`, resolve it before ACP, subagent, and Task Flow records are created, preserve existing behavior when unset, and let explicit task or flow policy win.

## Reproduction Assessment

Yes for the configuration gap: current-main source inspection shows hardcoded notify defaults and no accepted agent-defaults task notify key. This is source-reproducible feature work rather than a failing bug reproduction.

## Solution Assessment

Unclear on exact shape because there is no patch and maintainers still need to choose global versus per-runtime defaults and precedence. The maintainable direction is one coherent config contract across task registry, Task Flow, ACP, and subagent creation.
