## Summary

Keep open: current main and v2026.6.6 still default deliverable ACP/subagent and Task Flow notifications to `done_only` and expose no typed or documented config-level default. The remaining blocker is maintainer product direction on the public config shape.

## What This Changes

The issue requests a config-level default task notification policy so ACP/subagent background task notices can default to `silent` or another policy instead of relying on hardcoded per-task defaults.

## Best Possible Solution

Add a backwards-compatible core config surface only after maintainers choose global versus per-runtime semantics; explicit task or flow `notifyPolicy` should win, unset config should preserve current defaults, and docs/tests should cover ACP, subagent, Task Flow, cron, CLI, and media siblings.

## Reproduction Assessment

Yes, at source level. Current main creates deliverable ACP/subagent and Task Flow records without an explicit notify policy, defaults them to `done_only`, and exposes no schema-backed default config key.

## Solution Assessment

Unclear until maintainers choose the public policy shape. A config default is plausible, but the best implementation must avoid overlapping with narrower channel-specific work and preserve explicit task policy behavior.
