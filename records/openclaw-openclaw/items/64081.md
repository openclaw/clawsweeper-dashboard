## Summary

Keep open. Current main and v2026.6.6 still expose hardcoded/per-task task notification defaults rather than a schema-backed config default, and the remaining work is a public config-contract decision rather than stale or already implemented work.

## What This Changes

Request a config-level default task notify policy so ACP, subagent, and related background task completions can default to `silent`, `done_only`, or `state_changes` without per-task races.

## Best Possible Solution

Add a maintainer-approved `agents.defaults.tasks` notify-policy resolver that preserves current behavior when unset, lets explicit task or flow policies win, and updates schema, docs, and focused tests together.

## Reproduction Assessment

Yes, at source level. Current main shows ACP, subagent, and Task Flow paths falling back to hardcoded `done_only` unless a caller supplies `notifyPolicy`, and no live Telegram run was needed to confirm the missing config surface.

## Solution Assessment

Unclear pending maintainer direction. A config-level default is a plausible core solution, but maintainers should choose global versus per-runtime shape before implementation because it becomes a public config contract.
