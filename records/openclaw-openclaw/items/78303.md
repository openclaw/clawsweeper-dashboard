## Summary

Keep open: current main still passes bundle-MCP tool results directly to the agent, and this PR is the active implementation path, but the patch still needs focused fixes before merge because the transcript sanitizer does not cover every approval command form the parser accepts and the public copy still advertises unsupported approval decisions.

## What This Changes

The PR adds an MCP consent envelope that routes opted-in bundle-MCP tool calls through existing plugin approvals, adds `mcp.approvals.enabled`, changes `/approve` reply UX, and adds docs/tests/changelog coverage.

## Best Possible Solution

Land the materializer-level consent gate only after the sanitizer covers every parser-accepted approval command form, the public docs/copy match the supported `allow-once`/`deny` contract, and maintainer security/product review accepts the envelope approach.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the sanitizer only rewrites `/approve`, while the same PR's parser accepts slashless `approve` and implicit single-decision approval text. I did not execute tests in this read-only review.

## Solution Assessment

No, not yet. Intercepting bundle-MCP tool results in the materializer and reusing plugin approvals is a plausible architecture, but the security sanitizer and public decision copy must be corrected before this branch is the best merge candidate.
