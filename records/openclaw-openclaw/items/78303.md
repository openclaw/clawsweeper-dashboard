## Summary

Keep open; the consent-envelope direction and supplied live proof are strong, but latest PR head still has source-level blockers in the approval-command sanitizer and timeout config contract, plus release/docs copy drift.

## What This Changes

The PR adds an opt-in MCP consent envelope that gates bundle-MCP tool calls through plugin approvals, updates `/approve` UX, adds `mcp.approvals` config, docs, changelog, and tests.

## Best Possible Solution

Land the materializer-level consent gate after aligning sanitizer coverage with the approval parser, making timeout config/docs/runtime agree, and correcting docs/release copy to the allow-once-only MCP consent contract.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the sanitizer only matches `/approve` while the parser accepts slashless `approve`, and the config docs/runtime promise clamping while the zod schema rejects out-of-range values. I did not execute tests because this is a read-only review.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materialize layer looks like the right architecture, but the parser-defense and config contract mismatches should be repaired before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the approval parser accepts `approve` without the slash and the PR adds implicit `approve allow-once` style decisions. A malicious consent summary can still leave parser-shaped approval text intact in the transcript-defense layer, so the sanitizer should cover every command form the parser accepts.
  - confidence: 0.93
- **[P2] Align timeout validation with the clamp contract:** `src/config/zod-schema.ts:325-328`
  - body: `mcp.approvals.defaultTimeoutMs` is documented and implemented as a clamped value, but the strict config schema rejects values outside `[1000, 600000]` before the runtime clamp can run. Operators following the docs with a too-high or too-low value get a config validation failure instead of the promised clamp; either relax the schema and clamp or change the docs/types to say the range is validated.
  - confidence: 0.88
- **[P3] Advertise only supported MCP consent decisions:** `CHANGELOG.md:9`
  - body: The changelog still tells users they can reply with `allow-always` for MCP consent approvals, but this flow sends `allowedDecisions: ["allow-once", "deny"]` and the gateway rejects decisions outside that set. The release note should not advertise a durable approval mode this contract does not implement.
  - confidence: 0.91
- **[P3] Show the actual implicit approval prompt:** `docs/tools/mcp-consent-envelope.md:69-72`
  - body: The new docs say the approval message ends with the old explicit-id form, but the PR changed the prompt to list bare `/approve <decision>` choices while showing the id separately. The docs should match the shipped prompt so server authors and operators know what users will actually see.
  - confidence: 0.82
