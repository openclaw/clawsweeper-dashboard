## Summary

Keep open: the MCP consent-envelope direction is useful and the PR includes live proof, but the current head still has blocking approval-routing/security and config-contract defects.

## What This Changes

The PR adds bundle-MCP consent-envelope detection, plugin-approval re-calls, no-ID approval replies, `mcp.approvals.enabled`, docs, changelog text, and focused approval/MCP tests.

## Best Possible Solution

Land the MCP consent envelope after the implicit approval shortcut is scoped to the inbound approval target or made explicit-ID-only, the MCP opt-out applies on every agent path, and sanitizer/public copy match the parser and supported decisions.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the global implicit approval resolver, the ignored config flag on the main agent path, and the sanitizer/parser mismatch. I did not execute the code because the review is read-only.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but the current patch needs approval-target scoping, config propagation, and defense-in-depth cleanup before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope bare approval replies to the originating target:** `src/auto-reply/reply/commands-approve.ts:212-239`
  - body: The new `/approve <decision>` branch lists all pending plugin and exec approvals and binds to the only global candidate. If the sole pending approval belongs to another chat/session, an authorized sender elsewhere can approve it without seeing its prompt or ID; filter candidates by the current channel/account/thread/session metadata or require an explicit ID.
  - confidence: 0.93
- **[P2] Honor the MCP approval opt-out in agent runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1241-1242`
  - body: The documented `mcp.approvals.enabled: false` flag is only resolved in `createBundleMcpToolRuntime`, but the normal agent attempt path calls `materializeBundleMcpToolsForRun` directly and only passes `agentId`/`sessionKey`. In this path the flag stays undefined, so consent envelopes are still gated despite the deployment opt-out.
  - confidence: 0.9
- **[P2] Sanitize slashless approve commands:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without a slash and this PR makes `approve allow-once` a valid implicit approval form. A malicious envelope summary can still render a parser-shaped approval line in the defense-in-depth path this sanitizer is meant to close.
  - confidence: 0.88
- **[P3] Fix public copy for one-shot MCP approvals:** `CHANGELOG.md:42`
  - body: The consent request now limits MCP approvals to `allow-once` and `deny`, but the changelog still advertises `allow-always`. That tells operators this adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.87
