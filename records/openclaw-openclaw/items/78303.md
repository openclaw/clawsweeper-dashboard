## Summary

Keep open: the PR targets a real bundle-MCP approval gap, but it is not merge-ready because the proposed protocol changes upgrade behavior by default, leaves the first-call token boundary unsettled, omits full origin/call binding for approvals, and lacks current-head real behavior proof.

## What This Changes

Adds a bundle-MCP consent-envelope flow that detects approval-required MCP tool results, routes them through plugin approvals and `/approve`, re-calls with `confirmation_token`, and adds MCP approval config and docs.

## Best Possible Solution

Land this or a replacement only after maintainers settle the opt-in/default protocol, reserve token redemption without breaking non-envelope tools, bind approvals to the full origin and exact call, and attach current-head live proof.

## Reproduction Assessment

## Solution Assessment

The bundle-MCP materializer is a plausible enforcement layer, but this is not yet the best fix until the default, token-redemption, exact-call binding, and official MCP elicitation alignment decisions are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Make MCP approvals opt-in or prove the upgrade path:** `src/agents/agent-bundle-mcp-materialize.ts:87-89`
  - body: This defaults the new consent gate on for every existing `mcp.servers[]` deployment unless users set `mcp.approvals.enabled: false`. Any server that already returns this envelope will stop returning a normal model-visible result and instead wait on plugin approval routing after upgrade, which can prompt unexpectedly or time out when approval delivery is not configured. Please either make the runtime behavior explicitly opt-in or include maintainer-approved upgrade proof/docs for the default-on change.
  - confidence: 0.87
- **[P1] Reserve confirmation_token before the first consent call:** `src/agents/agent-bundle-mcp-materialize.ts:130-138`
  - body: The first MCP call still sends `params.input` unchanged, so a model-supplied `confirmation_token` reaches the MCP server before OpenClaw has detected an envelope or issued a channel approval. That contradicts the PR's security property that the model cannot fabricate or redeem approval tokens; the protocol needs a way to reserve or scrub this field for consent-capable tools without breaking non-envelope tools.
  - confidence: 0.89
- **[P1] Bind MCP approvals to the full origin and exact call:** `src/agents/agent-bundle-mcp-consent.ts:213-216`
  - body: The approval request only carries channel and target, while the existing plugin approval path also carries account and thread origin fields, and this new consent flow has no canonical digest binding the approval to the exact server/tool/arguments being retried. For side-effecting MCP tools, approvals should be scoped at least as tightly as the current plugin approval path so a reply from another account/thread or a changed retry cannot authorize the wrong call.
  - confidence: 0.84
