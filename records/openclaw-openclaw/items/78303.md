## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reserve confirmation_token before first MCP call:** `src/agents/agent-bundle-mcp-materialize.ts:134-138`
  - body: This first call still forwards the model's original input, including any `confirmation_token`, before OpenClaw knows whether the server speaks the consent protocol. A consent-aware server that treats that field as redemption can execute before channel approval, so the token field needs to be reserved for the consent path or the server contract needs stronger proof that model-supplied tokens cannot redeem.
  - confidence: 0.89
- **[P1] Make default-on MCP approvals upgrade-safe:** `src/agents/agent-bundle-mcp-materialize.ts:87-89`
  - body: `resolveMcpApprovalsConfig` enables the new gate unless users explicitly set `mcp.approvals.enabled: false`. Existing installs with envelope-speaking MCP servers would change from pass-through results to approval prompts, unavailable results, or timeouts on upgrade, so this needs opt-in/default-off behavior or explicit maintainer upgrade proof.
  - confidence: 0.88
- **[P1] Carry full approval origin into MCP consent requests:** `src/agents/agent-bundle-mcp-consent.ts:215-216`
  - body: The MCP consent request only forwards `turnSourceChannel` and `turnSourceTo`, while the existing plugin approval hook also forwards account and thread origin. Multi-account or threaded transports cannot prove the exact originating surface without those fields, so plumb account/thread data and bind the pending approval to the exact MCP call before relying on channel approval as the security boundary.
  - confidence: 0.84
