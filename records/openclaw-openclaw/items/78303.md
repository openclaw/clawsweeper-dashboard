## Summary

Keep open: this is an active, useful security-adjacent feature PR with sufficient live behavior proof, but the latest head still has a concrete approval-text sanitizer mismatch and misleading `allow-always` release copy before it should merge.

## What This Changes

This PR adds an MCP consent-envelope gate for bundle-MCP tool calls, routes approvals through the existing plugin approval pipeline, updates `/approve` reply UX, adds `mcp.approvals.enabled`, and includes docs, tests, and changelog text.

## Best Possible Solution

Land the consent envelope only after the approval transcript defense covers every parser-accepted command form, public copy and config metadata match the actual `allow-once`/`deny` contract, and maintainers accept the custom envelope as compatible with future MCP elicitation support.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the sanitizer only rewrites `/approve` while the parser accepts slashless `approve`, and the changelog still advertises `allow-always` while the request only allows `allow-once`/`deny`. I did not execute tests because this was a read-only review.

## Solution Assessment

No, not yet. The materializer-level interception and reuse of plugin approvals look like the right implementation shape, but the approval-text defense, release/config copy, and public contract polish need to be corrected before merge.
