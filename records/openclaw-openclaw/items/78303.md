## Summary

Keep open. The MCP approval direction is useful and the contributor supplied live proof, but the latest head still has a security-sensitive implicit approval scoping bug and an incomplete approval-command sanitizer.

## What This Changes

The branch adds an MCP consent-envelope approval gate for bundle-MCP tools, an opt-out config key, approval prompt/parser changes, docs, changelog, and focused tests.

## Best Possible Solution

Land the MCP consent envelope after implicit approval resolution is target-scoped or explicit-ID-only, tool-emitted approval text is neutralized for every parser form, and docs/changelog match the one-shot prompt.

## Reproduction Assessment

Yes for the blocking review findings: source inspection shows implicit approval resolution uses unfiltered global pending lists, and the sanitizer only rewrites /approve while the parser accepts slashless approve. The broader MCP happy path is supported by contributor live logs, not a local rerun.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look maintainable, but the implicit approval shortcut must be target-scoped or explicit-ID-only and the sanitizer/docs need to match the parser and one-shot decision set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope implicit approvals to the delivered target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path still lists all pending plugin and exec approvals and binds to the sole global candidate. On a shared gateway, an authorized sender in one chat can approve another channel/session's only pending request without seeing that prompt or ID. Filter implicit candidates to the approval delivered to this sender/account/thread/session, or require the explicit ID unless the candidate belongs to the current delivery target.
  - confidence: 0.9
- **[P2] Sanitize slashless approve commands too:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without a slash and this PR makes `approve allow-once` a valid implicit approval shape. A malicious MCP summary containing a newline followed by `approve allow-once` remains command-shaped in the defense-in-depth layer this sanitizer is meant to provide.
  - confidence: 0.86
- **[P3] Align the consent copy with one-shot approvals:** `CHANGELOG.md:32`
  - body: The implementation restricts MCP consent requests to `allow-once` and `deny`, and the prompt now uses bare `/approve <decision>` examples, but the changelog still advertises `allow-always` and the old explicit-id command shape. That tells operators this adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.9
