## Summary

Keep this PR open: the MCP consent-envelope direction is useful and has live proof, but the current branch still has security-sensitive approval-path blockers around implicit approval scoping and command sanitization.

## What This Changes

The PR adds a bundle-MCP consent-envelope approval gate, implicit approval reply UX, docs, config schema support, changelog text, and focused approval tests.

## Best Possible Solution

Land the MCP consent envelope only after implicit approval resolution is target-scoped or explicit-ID-only, tool-emitted approval text is neutralized for every parser form, and public copy matches the one-shot decision set.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows implicit approvals are resolved from global pending lists, and the sanitizer only rewrites slash-prefixed approve commands while the parser accepts slashless approve. I did not run the PR tests because this review is read-only.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look maintainable, but the implicit approval shortcut must be target-scoped or explicit-ID-only and the sanitizer/docs need to match the parser and one-shot decision set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope implicit approval resolution to the current target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path still lists all pending plugin and exec approvals, merges them, and binds to the sole global candidate. An authorized sender in one chat can therefore approve a different channel/session's only pending request without seeing its prompt or ID. Filter the candidates to the current delivery target/account/thread/session, or require an explicit ID unless the candidate belongs to this inbound context.
  - confidence: 0.9
- **[P2] Sanitize slashless approve commands too:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without a slash and this PR makes `approve allow-once` a valid implicit approval shape. A malicious MCP summary containing a newline followed by `approve allow-once` remains command-shaped in the defense-in-depth layer this sanitizer is meant to provide.
  - confidence: 0.86
- **[P3] Align the changelog with one-shot MCP approvals:** `CHANGELOG.md:33`
  - body: The implementation sends MCP consent requests with `allowedDecisions: ["allow-once", "deny"]`, and the docs now describe allow-once-only behavior, but the changelog still advertises `allow-always` and the old explicit-id command shape. That tells operators this adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.9
