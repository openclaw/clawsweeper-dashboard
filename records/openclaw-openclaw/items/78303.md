## Summary

Keep open: current main still lacks bundle-MCP consent-envelope handling, and this PR is the active implementation path, but the current head still has a security-relevant sanitizer/parser mismatch plus stale user-facing copy that should be repaired before merge.

## What This Changes

The PR adds an opt-in MCP consent envelope that routes bundle-MCP tool calls through plugin approvals, adds MCP approval config/docs/tests, and changes the approval reply UX to support implicit single-pending approvals.

## Best Possible Solution

Land the materializer-level consent-envelope design only after the sanitizer matches every approval command form accepted by the parser, user-facing copy matches the supported decisions and prompt format, and maintainers explicitly keep MCP-native elicitation compatibility tracked if this PR does not cover it.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the sanitizer only rewrites `/approve`, while the approval parser accepts slashless `approve` and implicit single-decision approval commands. I did not run tests because this was a read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but the sanitizer/tests and release/docs copy need to match the actual approval parser and allowed-decision contract before merge.
