## Summary

Keep open: the feature is useful and the PR includes live behavior proof, but latest head still leaves a parser/sanitizer mismatch in the approval path that this security hardening is meant to close. The changed changelog copy also advertises an MCP approval decision that the implementation rejects.

## What This Changes

The PR adds an MCP consent envelope that routes opted-in bundle-MCP tool calls through plugin approvals, scrubs model-supplied confirmation tokens, adds `mcp.approvals.enabled`, updates `/approve` UX, docs, tests, and changelog text.

## Best Possible Solution

Land the consent envelope after every parser-accepted approval command form is neutralized in tool-emitted text and public release copy matches the supported `allow-once`/`deny` decisions.

## Reproduction Assessment

Yes for the review finding: PR-head source shows the sanitizer only matches `/approve`, while the approval parser accepts slashless `approve` and the new implicit single-decision form. I did not execute code in this read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but the approval transcript defense and public decision copy need to match the actual parser and gateway contract before merge.

## Review Findings

Full review comments:

- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:95-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the approval parser accepts slashless `approve` and this PR makes `approve allow-once` a valid implicit approval when one request is pending. A malicious envelope summary can still contain a newline followed by a parser-shaped approval line, leaving the defense-in-depth transcript-poisoning path this helper is meant to close.
  - confidence: 0.91

That release note would direct operators to a durable approval path the implementation intentionally does not support.
  - confidence: 0.87
