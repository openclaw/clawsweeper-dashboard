## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Block first-call token redemption:** `src/agents/agent-bundle-mcp-materialize.ts:134-138`
  - body: `callMcpToolWithConsent` sends the original model input to the MCP server before any approval exists, including a model-supplied `confirmation_token`. For an envelope-speaking mutating tool, a leaked or stale valid token can be redeemed on this first call and bypass the channel approval path, so consent-capable calls need a pre-approval strip/reject path or an explicit maintainer-approved contract that makes this safe.
  - confidence: 0.83
- **[P1] Bind MCP approvals to account and thread:** `src/agents/agent-bundle-mcp-consent.ts:213-216`
  - body: The approval request only carries channel and target, while the runner has account and thread fields and `plugin.approval.request` stores them. Multi-account or threaded channels can lose the exact surface binding used by existing approval delivery and implicit resolution, so these fields need to be threaded through with coverage.
  - confidence: 0.86
- **[P1] Make default-on approval gating upgrade-safe:** `src/agents/agent-bundle-mcp-materialize.ts:87-89`
  - body: `consentEnabled` defaults to true, so existing deployments whose MCP servers already return consent-shaped envelopes can stop seeing the legacy model-visible result and instead wait for plugin approval routing, time out, or return unavailable when forwarding is not configured. This needs explicit maintainer acceptance with upgrade proof, or a compatibility-safe default/migration path.
  - confidence: 0.78
