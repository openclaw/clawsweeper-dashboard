## Summary

Keep this PR open: the MCP consent direction is useful and has credible live proof, but current PR head still leaves concrete approval-path blockers around implicit approval scope, the documented config opt-out, and command-text sanitization.

## What This Changes

The PR adds a bundle-MCP consent envelope that routes opted-in MCP tool calls through plugin approvals, updates `/approve` UX, adds `mcp.approvals.enabled`, and documents/tests the new contract.

## Best Possible Solution

Land the consent envelope after the implicit approval shortcut is target-scoped or made explicit, the MCP opt-out reaches every materialization path, and sanitizer/docs/changelog copy match the supported decisions.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the global implicit resolver, the missing config flag on the normal agent path, and the sanitizer/parser mismatch. I did not execute the PR because this review is read-only.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but the current patch needs approval-target scoping, config propagation, and defense-in-depth cleanup before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope implicit approvals to the current approval surface:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path still combines all pending plugin and exec approvals and accepts the sole global candidate. If the only pending approval belongs to another chat/session, an authorized sender elsewhere can approve it without seeing that prompt; filter by the originating channel/account/thread/session metadata or require an explicit ID.
  - confidence: 0.94
- **[P2] Pass the MCP approval opt-out on agent attempts:** `src/agents/pi-embedded-runner/run/attempt.ts:1241-1242`
  - body: The normal embedded attempt path calls `materializeBundleMcpToolsForRun` directly and only passes session identity. Because it never passes `params.config.mcp?.approvals?.enabled`, deployments with `mcp.approvals.enabled: false` still gate consent envelopes on the main agent run path.
  - confidence: 0.92
- **[P2] Neutralize slashless approval commands in MCP text:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the parser accepts slashless `approve` and this PR makes `approve allow-once` a valid implicit approval form. A malicious envelope summary can still render a parser-shaped approval line in the defense-in-depth path this sanitizer is meant to close.
  - confidence: 0.9
- **[P3] Align changelog copy with one-shot MCP approvals:** `CHANGELOG.md:43`
  - body: The consent request now limits MCP approvals to `allow-once` and `deny`, but the changelog still advertises `allow-always`. That tells operators this adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.86
