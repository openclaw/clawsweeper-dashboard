## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Add approvals to the strict MCP schema:** `src/config/types.mcp.ts:43-49`
  - body: The PR documents and reads `mcp.approvals.enabled`, but `McpConfigSchema` still only accepts `servers` and `sessionIdleTtlMs`. Users who set the documented opt-out will fail config validation before this runtime flag can be used, so add the schema/help/generated config metadata for the new config surface.
  - confidence: 0.97
- **[P2] Clamp approval descriptions before requesting:** `src/agents/pi-bundle-mcp-consent.ts:155-165`
  - body: `envelope.summary` is MCP-server-controlled and is concatenated directly into the plugin approval description. The gateway caps plugin approval descriptions at 256 characters, so a long valid summary makes `plugin.approval.request` fail instead of prompting the user.
  - confidence: 0.96
- **[P2] Sanitize the whole approval description:** `src/agents/pi-bundle-mcp-consent.ts:155-165`
  - body: Only the envelope summary is sanitized, but `ctx.toolName` also comes from the MCP server catalog and is concatenated into the rendered approval description. A malicious tool name containing a newline and `/approve ...` can still place a magic-looking approval command in chat, so sanitize or safely render the full server/tool/summary payload.
  - confidence: 0.86
- **[P2] Honor the opt-out before scrubbing tokens:** `src/agents/pi-bundle-mcp-materialize.ts:102-114`
  - body: `confirmation_token` is stripped before checking `consentEnabled === false`. That makes the documented opt-out unable to pass the envelope flow back to the model/server verbatim, because a follow-up model call with `confirmation_token` is still altered when approvals are disabled.
  - confidence: 0.92
- **[P2] Race approval waits against run cancellation:** `src/agents/pi-bundle-mcp-materialize.ts:244-257`
  - body: `AnyAgentTool.execute` receives an `AbortSignal`, and the existing plugin approval path races `waitDecision` against it so stopped runs do not wait for the full timeout. This wrapper drops the signal, so cancelling a run during MCP consent can leave the tool call pending until expiry.
  - confidence: 0.88
- **[P2] Remove allow-always until it has durable semantics:** `src/agents/pi-bundle-mcp-consent.ts:158-172`
  - body: The MCP approval request omits `allowedDecisions`, so plugin approvals default to `allow-always` even though the consent path handles it exactly like a one-shot approval. Users choosing `allow-always` will still be prompted again on the next envelope, so persist a real policy or omit that decision for MCP consent.
  - confidence: 0.88
- **[P2] Authorize implicit approval lookups before listing:** `src/auto-reply/reply/commands-approve.ts:171-217`
  - body: The new bare `/approve <decision>` path lists pending exec/plugin approvals before sender authorization and gateway-scope checks. Unauthorized senders can probe pending approval state, and the ambiguous case replies with an explicit approval id; move authorization before listing and avoid exposing ids to unauthorized callers.
  - confidence: 0.92
- **[P2] Use a reserved sender fixture:** `src/agents/pi-bundle-mcp-consent.test.ts:429-438`
  - body: This new test commits a plausible E.164 WhatsApp sender value inside a session key. Repo policy says not to commit real phone numbers, so use an obviously reserved fake value instead.
  - confidence: 0.9
- **[P2] Keep approval prompt tests aligned:** `src/infra/plugin-approvals.ts:98-101`
  - body: The PR changes plugin approval prompt copy to bare `/approve <decision>` examples, but CI still fails because `plugin-approval-forwarder.test.ts` expects the old `/approve <id> ...` text. Update the affected test expectation, or keep the old explicit-id wording if that is still required.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds a security-sensitive approval path and still has concrete authorization, prompt-injection, availability, decision-semantics, and privacy issues to fix before merge.

Concerns:

- **[medium] Implicit approval lookup runs before sender authorization:** `src/auto-reply/reply/commands-approve.ts:171`
  - body: The bare `/approve <decision>` flow lists pending approvals and can reveal approval counts or an explicit id before checking whether the sender may approve anything on that channel.
  - confidence: 0.92
- **[medium] MCP-controlled approval text is not fully safe:** `src/agents/pi-bundle-mcp-consent.ts:155`
  - body: The approval description includes unsanitized MCP catalog text and is not bounded before gateway validation, leaving prompt-injection-looking transcript text and fail-closed availability behavior for malicious or malformed servers.
  - confidence: 0.88
- **[medium] MCP allow-always is exposed without policy storage:** `src/agents/pi-bundle-mcp-consent.ts:158`
  - body: The request inherits plugin approval `allow-always` by default, but MCP consent treats it like `allow-once`, so users may believe they granted a durable policy that the runtime does not enforce.
  - confidence: 0.86
- **[low] Phone-number-like fixture committed in test:** `src/agents/pi-bundle-mcp-consent.test.ts:430`
  - body: The new test uses a plausible WhatsApp E.164 sender in a session key; even if synthetic, it conflicts with the repository policy to avoid committing real phone numbers.
  - confidence: 0.9
