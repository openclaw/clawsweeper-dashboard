## Summary

Keep open: current main does not have a bundle-MCP approval gate, and this PR is a useful implementation path, but the latest head still has blocking schema, authorization, cancellation, payload-safety, decision-semantics, and fixture issues.

## What This Changes

The PR adds a bundle-MCP consent-envelope flow that routes opted-in MCP tool calls through plugin approvals, plus config typing, docs, tests, changelog, approval routing metadata, and implicit-id `/approve` UX.

## Best Possible Solution

Land a maintainer-approved MCP approval contract after fixing schema/help alignment, authorization order, bounded sanitized payloads, cancellation, and decision semantics.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature/security-hardening PR. Source inspection and linked live proof show the current-main baseline passes bundle-MCP results to the agent without a runtime approval gate.

## Solution Assessment

No: the materializer placement and reuse of plugin approvals are plausible, but this is not the best mergeable solution until the schema, auth order, payload bounds/sanitization, cancellation, allow-always semantics, and privacy fixture issues are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Add approvals to the strict MCP schema:** `src/config/types.mcp.ts:43-49`
  - body: The PR documents and reads `mcp.approvals.enabled`, but `McpConfigSchema` still only accepts `servers` and `sessionIdleTtlMs`. Users who set the documented opt-out will fail config validation before the runtime flag can be used, so add the schema/help/label metadata for the new config surface.
  - confidence: 0.97
- **[P2] Bound MCP approval descriptions before validation:** `src/agents/pi-bundle-mcp-consent.ts:155-165`
  - body: `envelope.summary` is MCP-server-controlled and is concatenated directly into the plugin approval description. The gateway caps plugin approval descriptions at 256 characters, so a long valid summary makes `plugin.approval.request` fail instead of prompting the user.
  - confidence: 0.96
- **[P2] Sanitize the full MCP approval description:** `src/agents/pi-bundle-mcp-consent.ts:155-165`
  - body: Only the envelope summary is sanitized, but `ctx.toolName` also comes from the MCP server catalog and is concatenated into the rendered approval description. A malicious tool name containing a newline and `/approve ...` can still place a magic-looking approval command in chat, so sanitize or render the full server/tool/summary payload safely.
  - confidence: 0.84
- **[P2] Honor the opt-out before scrubbing tokens:** `src/agents/pi-bundle-mcp-materialize.ts:102-114`
  - body: `confirmation_token` is stripped before checking `consentEnabled === false`. That makes the documented opt-out unable to pass the envelope flow back to the model/server verbatim, because a follow-up model call with `confirmation_token` is still altered when approvals are disabled.
  - confidence: 0.92
- **[P2] Race MCP approval waits against cancellation:** `src/agents/pi-bundle-mcp-materialize.ts:244-257`
  - body: `AnyAgentTool.execute` receives an `AbortSignal`, and the existing plugin approval path races `waitDecision` against it so stopped runs do not wait for the full timeout. This wrapper ignores the signal, so cancelling a run during MCP consent can leave the tool call pending until expiry.
  - confidence: 0.88
- **[P2] Implement or hide MCP allow-always semantics:** `src/agents/pi-bundle-mcp-consent.ts:158-172`
  - body: The MCP approval request uses the plugin approval default decisions, which includes `allow-always`, but the consent path handles it exactly like a one-shot approval. Users choosing `allow-always` will still be prompted again on the next envelope, so persist a real policy or omit that decision for MCP consent.
  - confidence: 0.86
- **[P2] Authorize implicit approvals before listing pending IDs:** `src/auto-reply/reply/commands-approve.ts:171-198`
  - body: The new implicit `/approve <decision>` path lists pending exec/plugin approvals before sender authorization and gateway-scope checks. Unauthorized senders can probe pending approval state, and the ambiguous case replies with an explicit approval id; move authorization before listing and avoid exposing ids to unauthorized callers.
  - confidence: 0.91
- **[P2] Replace the phone-number-like fixture:** `src/agents/pi-bundle-mcp-consent.test.ts:430-438`
  - body: This new test commits a plausible E.164 WhatsApp sender value inside a session key. Repo policy says not to commit real phone numbers, so use an obviously reserved fake value instead.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a security-sensitive approval path and still has concrete authorization, prompt-injection, availability, decision-semantics, and privacy issues to fix before merge.

Concerns:

- **[medium] Implicit approval lookup runs before sender authorization:** `src/auto-reply/reply/commands-approve.ts:171`
  - body: The new bare `/approve <decision>` flow lists pending approvals and can reveal approval counts or an explicit id before checking whether the sender may approve anything on that channel.
  - confidence: 0.91
- **[medium] Unbounded server text can suppress approvals:** `src/agents/pi-bundle-mcp-consent.ts:155`
  - body: A malicious or buggy MCP server can return an overlong summary and make `plugin.approval.request` fail validation, turning the approval flow into a denial instead of a user prompt.
  - confidence: 0.94
- **[medium] MCP tool names can still inject approval-looking text:** `src/agents/pi-bundle-mcp-consent.ts:155`
  - body: The sanitizer is applied to `summary`, but the rendered approval description also includes MCP-controlled `toolName`; a newline-bearing tool name can still place a `/approve`-looking command into chat output.
  - confidence: 0.84
- **[medium] MCP allow-always is advertised without durable effect:** `src/agents/pi-bundle-mcp-consent.ts:158`
  - body: The approval UI can offer `allow-always`, but the MCP consent path treats it as one one-shot confirmation. That weakens user expectations for a security decision and should be hidden or implemented deliberately.
  - confidence: 0.86
- **[low] Real-looking phone number in test fixture:** `src/agents/pi-bundle-mcp-consent.test.ts:430`
  - body: The new consent test includes a plausible E.164 WhatsApp sender in a session key; repository policy forbids committing real phone numbers, so the fixture should use a reserved fake value.
  - confidence: 0.9
