## Summary

Keep this PR open: the feature is plausible and the supplied live proof is meaningful, but the latest diff still has security-relevant approval scoping and command-sanitization issues that should block merge.

## What This Changes

The PR adds an opt-in MCP consent-envelope flow that intercepts bundle-MCP tool results, routes approval through plugin approvals, updates approval reply UX, adds `mcp.approvals.enabled`, and documents/tests the contract.

## Best Possible Solution

Land a scoped MCP approval contract only after implicit approval resolution is bound to the delivered approval target, tool-emitted command text is sanitized consistently with the parser, and the user-facing docs/changelog match the one-shot decision set.

## Reproduction Assessment

Yes for the review findings: the latest PR source shows global implicit approval selection and a sanitizer that misses a parser-accepted command form. The broader MCP baseline/fix behavior is supported by the PR's live logs and current-main source, not re-run locally.

## Solution Assessment

No, not yet. Materializer-level interception and reusing plugin approvals are reasonable, but the implicit approval shortcut must be target-scoped or explicit-ID-only, and sanitizer/copy must match the parser and actual decision set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope implicit approvals to the delivered target:** `src/auto-reply/reply/commands-approve.ts:239-256`
  - body: The bare `/approve <decision>` path lists every pending plugin and exec approval, then binds to the sole global candidate. On a shared gateway, an authorized sender in one chat can approve another channel/session's only pending request without receiving that prompt or ID. Filter candidates to the approval routed to this sender/account/thread/session, or require the explicit ID unless the candidate belongs to the current delivery target.
  - confidence: 0.9
- **[P2] Sanitize slashless approve commands too:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser also accepts `approve` without a slash. A malicious MCP summary containing a newline followed by `approve allow-once` remains command-shaped if tool-emitted text is ever echoed into the text-command path, which is the exact defense-in-depth case this sanitizer is meant to cover.
  - confidence: 0.84
- **[P3] Remove allow-always from the MCP changelog entry:** `CHANGELOG.md:13`
  - body: The consent request now sends `allowedDecisions: ["allow-once", "deny"]`, and the docs describe only one-shot approval, but the changelog still advertises `allow-always`. That tells operators this PR adds a durable policy path that the implementation intentionally does not support.
  - confidence: 0.92
