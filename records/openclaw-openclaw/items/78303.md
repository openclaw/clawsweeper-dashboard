## Summary

Keep open: the PR addresses a real bundle-MCP approval gap and has sufficient live proof, but PR head still has approval-command security and release/docs blockers before it is mergeable.

## What This Changes

The PR adds MCP consent-envelope detection, routes matching bundle-MCP tool calls through plugin approvals, re-calls approved tools with a confirmation token, and updates approval UX, config, docs, tests, and changelog.

## Best Possible Solution

Land the materializer-level consent design only after scoping implicit approval resolution, aligning sanitizer/parser behavior, fixing release/docs discoverability, and completing maintainer security/product review.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the implicit approval shortcut resolves a global pending approval, the sanitizer misses the parser-accepted slashless command form, and the release copy disagrees with request-scoped allowed decisions. I did not run tests in this read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals are the right area, but the approval shortcut and transcript defense need to match the actual trust boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope implicit approvals to the prompting route:** `src/auto-reply/reply/commands-approve.ts:244-261`
  - body: The new bare `/approve <decision>` path lists every pending plugin and exec approval through the backend client, merges both lists, and resolves the sole global candidate. In a multi-user gateway, an authorized sender can approve the only pending request even if it was delivered to another channel/session and they never saw that approval id; keep the shortcut scoped to the request's delivery route or require the explicit id.
  - confidence: 0.86
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without the slash. A malicious envelope summary can still include a parser-shaped `approve allow-once ...` line, so the defense-in-depth transcript-poisoning layer still does not cover every command form the parser honors.
  - confidence: 0.92
- **[P3] Advertise only supported MCP consent decisions:** `CHANGELOG.md:9`
  - body: The changelog tells operators that MCP consent accepts `allow-always`, but the implementation sends `allowedDecisions: ["allow-once", "deny"]` and the gateway rejects decisions outside that set. This release copy would point users at a durable approval mode the flow intentionally does not support.
  - confidence: 0.91
- **[P3] Link the consent page in docs navigation:** `docs/tools/mcp-consent-envelope.md:1`
  - body: The PR adds `docs/tools/mcp-consent-envelope.md`, but `docs/docs.json` is not updated with `tools/mcp-consent-envelope`. Without a nav entry or another durable docs link, the new server-author contract is hard to discover from the published docs.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR improves the MCP trust boundary, but two approval-command paths still need security fixes before merge.

Concerns:

- **[high] Bare approvals can resolve another route's request:** `src/auto-reply/reply/commands-approve.ts:244`
  - body: The implicit `/approve <decision>` shortcut selects the only pending approval globally instead of proving it was delivered to the current sender/channel/session, widening approval authority beyond the prompt recipient when only one request is pending.
  - confidence: 0.86
- **[medium] Slashless approval text is not neutralized:** `src/agents/pi-bundle-mcp-consent.ts:100`
  - body: MCP-emitted approval text only rewrites `/approve`, while the parser accepts slashless `approve`; this leaves a command-shaped self-approval string in the compromised-output defense layer.
  - confidence: 0.92
