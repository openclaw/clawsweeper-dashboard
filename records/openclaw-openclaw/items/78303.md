## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Strip tokens before the first consent-capable call:** `src/agents/agent-bundle-mcp-materialize.ts:134-138`
  - body: `callMcpToolWithConsent` sends the original input to the MCP server before any approval is created, including a model-supplied `confirmation_token`. For an envelope-speaking mutating tool, a leaked or stale valid token can be redeemed on this first call and bypass the channel approval path entirely, which contradicts the PR's trust-boundary guarantee; the implementation needs an opt-in signal that lets OpenClaw strip or reject these tokens before the first consent-capable call.
  - confidence: 0.83
- **[P1] Bind MCP approvals to account and thread:** `src/agents/agent-bundle-mcp-consent.ts:213-216`
  - body: The approval request only carries `turnSourceChannel` and `turnSourceTo`, while the runner already has `agentAccountId` and `messageThreadId` and `plugin.approval.request` can store them. In multi-account or threaded channels this loses the same origin binding used by existing approval delivery/resolution paths, so a mutating MCP approval can be delivered or resolved on the wrong surface.
  - confidence: 0.86
- **[P1] Make default-on approval gating upgrade-safe:** `src/agents/agent-bundle-mcp-materialize.ts:87-89`
  - body: `consentEnabled` defaults to true, so existing deployments whose MCP servers already return `requires_confirmation` envelopes will stop seeing the legacy model-visible envelope and instead wait for plugin approval routing, time out, or return unavailable when forwarding is not configured. This default needs explicit maintainer acceptance with upgrade proof, or a compatibility-safe default/migration path.
  - confidence: 0.78
