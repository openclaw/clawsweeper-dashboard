## Summary

Keep open. The feature has useful live proof and a plausible owner boundary, but the latest head still introduces approval-path bugs: implicit approvals resolve against global pending requests, and MCP-emitted text is not sanitized for the slashless approve form the parser accepts.

## What This Changes

The branch adds a bundle-MCP consent-envelope approval gate, `mcp.approvals` config, no-ID `/approve` UX, docs/changelog text, and focused tests.

## Best Possible Solution

Land the MCP consent envelope after the approval shortcut is target-scoped, the sanitizer matches every parser-accepted approve form, and public copy describes the one-shot decision set.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the no-ID approval path resolving a global pending list, and the parser/sanitizer mismatch is visible from the two changed files. I did not run tests because this is a read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look maintainable, but the implicit approval shortcut must be target-scoped or explicit-ID-only and the sanitizer/copy must match the parser and allowed decisions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope implicit approvals to the inbound target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path still lists all pending plugin and exec approvals, merges them, and binds to the sole global candidate. An authorized sender in one chat can therefore approve a different channel/session's only pending request without seeing its prompt or ID. Filter candidates to the current delivery target/account/thread/session, or require an explicit ID unless the candidate is proven to belong to this inbound context.
  - confidence: 0.92
- **[P2] Sanitize slashless approve commands too:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without a slash and this PR makes `approve allow-once` a valid implicit approval shape. A malicious MCP summary containing a newline followed by `approve allow-once` remains command-shaped in the defense-in-depth layer this sanitizer is meant to provide.
  - confidence: 0.89
- **[P3] Align the changelog with one-shot MCP approvals:** `CHANGELOG.md:37`
  - body: The implementation sends MCP consent requests with `allowedDecisions: ["allow-once", "deny"]`, and the docs describe allow-once-only behavior, but the changelog still advertises `allow-always`. That tells operators this adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.9
