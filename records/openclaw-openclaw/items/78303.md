## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Block first-call token redemption:** `src/agents/agent-bundle-mcp-materialize.ts:134-138`
  - body: `callMcpToolWithConsent` calls the MCP server once with the original model input before consent detection. If that input contains a leaked or stale `confirmation_token`, a consent-capable server can redeem it and return success before OpenClaw ever creates the channel approval, so the PR's core guarantee that only the runtime supplies the token is not enforced.
  - confidence: 0.9
- **[P1] Bind MCP approvals to account and thread:** `src/agents/embedded-agent-runner/run/attempt.ts:1457-1460`
  - body: The approval request only carries `channel` and `channelTarget`; it omits the runner's account and thread origin fields even though plugin approvals already support `turnSourceAccountId` and `turnSourceThreadId`. In multi-account or threaded channels this can deliver or resolve the approval against the wrong surface, or make implicit approval matching fail closed despite the originating thread being known.
  - confidence: 0.86
- **[P1] Make default-on approval gating upgrade-safe:** `src/agents/agent-bundle-mcp-materialize.ts:87-91`
  - body: `resolveMcpApprovalsConfig` enables the new consent gate unless users explicitly set `mcp.approvals.enabled: false`. Existing deployments whose MCP servers already emit this envelope can change from the legacy model-visible flow to approval prompts, no-route unavailable results, or timeouts after upgrade, so this needs an explicit compatibility plan or maintainer acceptance before merge.
  - confidence: 0.82
