## Summary

Keep this PR open: the MCP consent-envelope direction is useful and not implemented on current main, but the current branch still has source-proven correctness and security blockers before merge.

## What This Changes

This PR adds an opt-in MCP consent envelope for bundle-MCP tools, routes consent through plugin approvals, updates `/approve` reply UX, and adds config, docs, changelog, and focused tests.

## Best Possible Solution

Land the materializer-level consent gate after centralizing MCP approval config resolution for every run path and aligning sanitizer, parser, schema, docs, and changelog contracts.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the normal embedded run path bypasses MCP approval config, and the sanitizer/parser mismatch is visible by comparing the consent module with the `/approve` parser. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materialization layer is the right shape, but the branch needs narrow repairs before it matches the documented config and security contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass MCP approval config into embedded runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1313-1327`
  - body: `runEmbeddedAttempt` calls `materializeBundleMcpToolsForRun` directly but only passes `agentId` and `sessionKey`. Because `mcp.approvals.enabled` and `mcp.approvals.defaultTimeoutMs` are resolved only in `createBundleMcpToolRuntime`, normal agent turns ignore the documented opt-out and timeout settings.
  - confidence: 0.94
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` rewrites `/approve` only, but the command parser accepts `approve` without the slash. A malicious consent summary can still leave parser-shaped approval text intact in the transcript-defense layer, so either sanitize both spellings or narrow the parser.
  - confidence: 0.91
- **[P2] Align timeout validation with the clamp contract:** `src/config/zod-schema.ts:340`
  - body: The docs and runtime describe `mcp.approvals.defaultTimeoutMs` as clamped to `[1000, 600000]`, but the strict schema rejects out-of-range values before the runtime clamp can apply. Relax the schema and clamp, or document validation instead of clamping.
  - confidence: 0.88
- **[P3] Advertise only supported MCP consent decisions:** `CHANGELOG.md:9`
  - body: The changelog still tells users they can reply with `allow-always`, but MCP consent requests send `allowedDecisions: ["allow-once", "deny"]`. Update the release note so users do not try a durable approval mode this path does not implement.
  - confidence: 0.9
