## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Add the approvals key to the strict MCP schema:** `src/config/types.mcp.ts:43-49`
  - body: The PR documents and reads `mcp.approvals.enabled`, but the strict `McpConfigSchema` still only accepts `servers` and `sessionIdleTtlMs`. A user setting the documented opt-out will fail config validation before the new runtime flag can be used.
  - confidence: 0.97
- **[P2] Bound MCP approval descriptions before validation:** `src/agents/pi-bundle-mcp-consent.ts:131-140`
  - body: `envelope.summary` is MCP-server-controlled and is concatenated directly into the plugin approval description. The gateway caps plugin approval descriptions at 256 characters, so a long valid summary makes the approval request fail instead of prompting the user.
  - confidence: 0.96
- **[P2] Honor the opt-out before scrubbing tokens:** `src/agents/pi-bundle-mcp-materialize.ts:102-114`
  - body: `confirmation_token` is stripped before checking `consentEnabled === false`. That makes the documented opt-out unable to pass an envelope flow back to the model/server, because a follow-up model call with `confirmation_token` is still altered even when approvals are disabled.
  - confidence: 0.91
- **[P2] Race approval waits against run cancellation:** `src/agents/pi-bundle-mcp-materialize.ts:244-257`
  - body: `AnyAgentTool.execute` receives an `AbortSignal`, and the existing plugin approval path uses it so stopped runs do not wait for the full timeout. This wrapper ignores the signal, so cancelling a run during MCP consent can leave the tool call pending until expiry.
  - confidence: 0.88
- **[P2] Implement or hide allow-always semantics:** `src/agents/pi-bundle-mcp-materialize.ts:151-157`
  - body: The PR advertises `allow-always`, but handles it exactly like `allow-once`: one re-call and no durable server/tool policy. Users choosing `allow-always` will still be prompted again on the next envelope, so either persist a real MCP policy or do not offer that decision for this path.
  - confidence: 0.84
- **[P2] Replace the real-looking phone number fixture:** `src/agents/pi-bundle-mcp-consent.test.ts:392`
  - body: This new test commits a real-looking WhatsApp sender number inside a session key. Repo policy says not to commit real phone numbers; use an obviously reserved fake E.164 value instead.
  - confidence: 0.9
