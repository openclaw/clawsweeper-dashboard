## Summary

Keep open: current main still lacks bundle-MCP consent-envelope handling, and the PR is the active implementation for the linked feature issue, but the latest head still has blocking config and approval-command defense gaps before merge.

## What This Changes

The PR adds a bundle-MCP consent envelope, routes opted-in MCP tool calls through plugin approvals, adds `mcp.approvals` config, updates `/approve` UX, tests the flow, and documents the contract.

## Best Possible Solution

Land this consent-envelope direction after the normal run path honors the documented `mcp.approvals` config and the transcript defense covers every approval command form the parser accepts.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the normal attempt path does not pass config-derived consent settings into the materializer, and the sanitizer regex excludes parser-accepted slashless `approve` commands. I did not run tests in this read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but config wiring and sanitizer coverage need to match the documented contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Thread MCP approval config into normal runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1312-1326`
  - body: The primary embedded attempt path calls `materializeBundleMcpToolsForRun` directly and only passes `agentId`/`sessionKey`. Because it bypasses the config resolution in `createBundleMcpToolRuntime`, `mcp.approvals.enabled: false` still gates normal agent runs and `mcp.approvals.defaultTimeoutMs` is ignored there, even though the docs/types/schema promise both settings.
  - confidence: 0.93
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without the slash and this PR makes `approve allow-once` a complete implicit approval when one request is pending. A compromised envelope summary can still contain parser-shaped approval text, leaving the defense-in-depth transcript-poisoning gap this helper is meant to close.
  - confidence: 0.92
- **[P3] Advertise only supported MCP approval decisions:** `CHANGELOG.md:9`
  - body: The changelog tells users to reply with `allow-always` for MCP consent approvals, but this PR sends `allowedDecisions: ["allow-once", "deny"]` for that flow. That release note would point operators at a durable approval decision the implementation intentionally does not support.
  - confidence: 0.88
