## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Pass the session key into approval requests:** `src/agents/pi-bundle-mcp-materialize.ts:239-248`
  - body: `getOrCreateSessionMcpRuntime` stores the run's `sessionKey` on the runtime, and approval forwarding resolves the requester conversation from `request.request.sessionKey`. This wrapper never passes `params.runtime.sessionKey` into `callMcpToolWithConsent`, so session-mode approval delivery can report no route instead of prompting the channel that started the tool call.
  - confidence: 0.92
- **[P2] Add approvals to the strict MCP config schema:** `src/config/types.mcp.ts:43-49`
  - body: The new `mcp.approvals.enabled` key is documented, typed, and read by `createBundleMcpToolRuntime`, but the strict `McpConfigSchema` still accepts only `servers` and `sessionIdleTtlMs`. A user setting the documented opt-out will fail config validation before this code can see it, so add the nested schema/help/label coverage with the type change.
  - confidence: 0.97
- **[P2] Clamp approval descriptions before gateway validation:** `src/agents/pi-bundle-mcp-consent.ts:131-140`
  - body: `envelope.summary` is supplied by the MCP server and is concatenated directly into the plugin approval `description`. The gateway rejects plugin approval descriptions over 256 characters, so a long but otherwise valid summary makes the approval request fail and denies the tool without prompting; truncate or otherwise bound this text before sending it.
  - confidence: 0.95
- **[P2] Bypass token scrubbing when approvals are disabled:** `src/agents/pi-bundle-mcp-materialize.ts:102-114`
  - body: `confirmation_token` is stripped before checking `consentEnabled === false`. That makes the documented opt-out unable to pass the envelope flow back to the model/server, because a follow-up call with `confirmation_token` is still altered even when MCP approvals are disabled.
  - confidence: 0.91
- **[P2] Treat request-time null as approval unavailable:** `src/agents/pi-bundle-mcp-consent.ts:163-178`
  - body: Current gateway code uses `decision: null` on the initial `plugin.approval.request` response for the no-approval-route case, while accepted two-phase requests omit `decision` and return `status: "accepted"`. Falling through to `waitDecision` on null turns an immediate no-route failure into a misleading waitDecision/error-deny path; preserve the existing unavailable handling.
  - confidence: 0.9
- **[P2] Forward abort signals into approval waits:** `src/agents/pi-bundle-mcp-materialize.ts:237-248`
  - body: `AnyAgentTool.execute` receives an AbortSignal, and the existing plugin approval path races `waitDecision` against it so stopped runs do not block until timeout. This wrapper drops the signal before entering the approval flow, so cancelling a run during MCP consent can leave the tool call waiting for the full approval timeout.
  - confidence: 0.87
- **[P2] Make allow-always durable or stop offering it:** `src/agents/pi-bundle-mcp-materialize.ts:151-157`
  - body: The branch accepts `allow-always` but handles it exactly like `allow-once`: it re-calls once with `confirmation_token` and records no server/tool policy for future envelopes. Users who choose `allow-always` will still be prompted again on the next call, so either implement a real durable policy/cache or restrict this MCP path to decisions it can honor.
  - confidence: 0.84
