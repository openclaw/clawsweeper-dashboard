## Summary

Keep open. The MCP consent direction and supplied live proof are strong, but the latest head still has two source-level blockers in security-sensitive approval handling: idless `/approve` can resolve the sole device-visible approval without channel/session binding, and `decision: null` is treated as pending even though current gateway code uses it for no-route/expired approval responses.

## What This Changes

The branch adds an opt-in MCP consent envelope, routes matching bundle-MCP tool calls through plugin approvals, changes `/approve` prompt and implicit-id handling, and adds config, docs, changelog, and tests.

## Best Possible Solution

Land the materializer-level MCP consent gate only after idless approvals are bound to the initiating approval surface and gateway no-route/null/timeout semantics stay distinct from user denial.

## Reproduction Assessment

Yes for the review blockers: current main's gateway source shows `decision: null` is a no-route/expired response, and the PR head source shows implicit approvals accept the sole listed candidate without channel/session filtering. I did not run tests because this is a read-only review.

## Solution Assessment

No, not yet. Reusing plugin approvals from the MCP materializer is a maintainable direction, but the proposed patch must preserve existing approval routing and no-route semantics before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Bind implicit approvals to the initiating surface:** `src/auto-reply/reply/commands-approve.ts:244-261`
  - body: This accepts `/approve allow-once` by choosing the only pending approval returned by the gateway client's plugin/exec lists, but those lists are visibility-scoped to the backend/device client rather than to the chat, account, thread, agent, or session that sent the command. An authorized sender in one chat can resolve the only pending approval for another chat on the same gateway device, so filter implicit candidates to the initiating approval surface before accepting one.
  - confidence: 0.88
- **[P2] Handle null decisions as unavailable before waiting:** `src/agents/pi-bundle-mcp-consent.ts:222-229`
  - body: Current gateway code returns `{ decision: null }` after expiring a request when no approval route exists; accepted two-phase requests omit `decision` and return `status: "accepted"`. Falling through to `plugin.approval.waitDecision` on a null decision waits on an already-expired approval and reports a generic denial instead of preserving the no-route/timeout result, so mirror the existing plugin approval caller and handle null immediately.
  - confidence: 0.9
