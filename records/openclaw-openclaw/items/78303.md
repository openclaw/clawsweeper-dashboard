## Summary

Keep open: current main still lacks MCP consent-envelope gating, and this PR is active and useful, but PR head still has concrete approval-path and public-copy defects that should be fixed before merge.

## What This Changes

Adds an MCP consent-envelope approval gate for bundle-MCP tool calls, threads session identity into approval delivery, updates no-ID approval replies, adds config/docs/changelog text, and adds focused tests.

## Best Possible Solution

Land the MCP consent envelope only after implicit approvals are target-scoped or explicit-ID-only, tool-emitted approval text is sanitized for every parser-accepted command form, and public copy matches the one-shot decision set.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the implicit approval path resolving a global pending list, and the parser/sanitizer mismatch is visible from the changed files. I did not run tests because this is a read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look maintainable, but the implicit approval shortcut must be target-scoped or removed, and sanitizer/public copy must match the parser and allowed decisions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope implicit approvals to the inbound target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path still lists all pending plugin and exec approvals, merges them, and binds to the sole global candidate. An authorized sender in one chat can therefore approve a different channel/session's only pending request without seeing its prompt or ID; filter candidates to this inbound target or require an explicit ID.
  - confidence: 0.92
- **[P2] Sanitize slashless approve commands too:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without a slash and this PR makes `approve allow-once` a valid implicit approval shape. A malicious MCP summary containing a newline followed by `approve allow-once` remains command-shaped in the defense-in-depth layer this sanitizer is meant to provide.
  - confidence: 0.89
- **[P3] Align the changelog with one-shot MCP approvals:** `CHANGELOG.md:42`
  - body: The implementation sends MCP consent requests with `allowedDecisions: ["allow-once", "deny"]`, and the docs describe allow-once-only behavior, but the changelog still advertises `allow-always`. That tells operators this adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.9
