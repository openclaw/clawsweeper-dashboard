## Summary

Keep open: current main does not implement generic bundle-MCP consent-envelope gating, and this PR still carries useful security hardening with real behavior proof. It is not merge-ready as-is because the head is currently dirty against the latest base and the default-on fail-closed approval behavior still needs an explicit maintainer acceptance before landing.

## What This Changes

The PR adds a bundle-MCP consent-envelope flow that routes opted-in MCP tool calls through plugin approvals, adds `mcp.approvals.*` config/docs/tests, and adjusts `/approve` parsing and prompt behavior.

## Best Possible Solution

Land a rebased version that keeps the materializer-level consent gate, documents and accepts the default-on fail-closed behavior, preserves explicit-id approval UX until session-bound implicit approvals are safe, and verifies the real channel approval path.

## Reproduction Assessment

Yes for the baseline behavior: current main returns bundle-MCP tool results directly from `runtime.callTool`, and the linked issue/PR comments include a live vanilla 2026.4.24 WhatsApp trace where the model saw the `action_id`. I did not run tests in this read-only review.

## Solution Assessment

Mostly yes: the bundle-MCP materializer is the right owner boundary for third-party `mcp.servers` tools, and reusing plugin approvals avoids a new gateway approval kind. The safest merge shape still needs a rebase, explicit maintainer acceptance of the default-on fail-closed behavior, and a small timeout-feedback cleanup.
