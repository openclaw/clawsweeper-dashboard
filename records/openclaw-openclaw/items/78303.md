Overall confidence: 0.93

Full review comments:

- **[P2] Add approvals to the strict MCP schema:** `src/config/zod-schema.ts:297-301`
  - body: The PR documents and reads `mcp.approvals.enabled`, but `McpConfigSchema` still only accepts `servers` and `sessionIdleTtlMs`. A user setting the documented opt-out will fail config validation before this runtime flag can be used.
  - confidence: 0.97
- **[P2] Bound MCP approval descriptions before validation:** `src/agents/pi-bundle-mcp-consent.ts:155-165`
  - body: `envelope.summary` is MCP-server-controlled and is concatenated directly into the plugin approval description. The gateway caps plugin approval descriptions at 256 characters, so a long valid summary makes the approval request fail instead of prompting the user.
  - confidence: 0.96
- **[P2] Honor the opt-out before scrubbing tokens:** `src/agents/pi-bundle-mcp-materialize.ts:102-114`
  - body: `confirmation_token` is stripped before checking `consentEnabled === false`. That makes the documented opt-out unable to pass the envelope flow back to the model/server verbatim, because a follow-up model call with `confirmation_token` is still altered when approvals are disabled.
  - confidence: 0.92
- **[P2] Race MCP approval waits against cancellation:** `src/agents/pi-bundle-mcp-materialize.ts:244-257`
  - body: `AnyAgentTool.execute` receives an `AbortSignal`, and the existing plugin approval path races `waitDecision` against it so stopped runs do not wait for the full timeout. This wrapper ignores the signal, so cancelling a run during MCP consent can leave the tool call pending until expiry.
  - confidence: 0.88
- **[P2] Implement or hide MCP allow-always semantics:** `src/agents/pi-bundle-mcp-materialize.ts:151-157`
  - body: The MCP approval request defaults to offering `allow-always`, but the consent path handles it exactly like `allow-once`: one re-call and no durable server/tool policy. Users choosing `allow-always` will still be prompted again on the next envelope, so persist a real policy or omit that decision for MCP consent.
  - confidence: 0.84
- **[P2] Authorize implicit approvals before listing pending IDs:** `src/auto-reply/reply/commands-approve.ts:171-198`
  - body: The new implicit `/approve <decision>` path lists pending exec/plugin approvals before sender authorization and gateway-scope checks. Unauthorized senders can now probe pending approval state, and the ambiguous case replies with an explicit approval id; move the auth checks before listing and avoid exposing ids to unauthorized callers.
  - confidence: 0.91
- **[P2] Preserve allowed-decision prompt behavior:** `src/infra/plugin-approvals.ts:70-74`
  - body: The new plugin approval prompt hard-codes all three bare commands, including `allow-always`. Current plugin approvals can restrict `allowedDecisions`, so this will tell users to send decisions the gateway rejects for approvals that intentionally omit `allow-always`.
  - confidence: 0.86
- **[P2] Replace the real-looking phone number fixture:** `src/agents/pi-bundle-mcp-consent.test.ts:430-438`
  - body: This new test commits a real-looking E.164 WhatsApp sender value inside a session key. Repo policy says not to commit real phone numbers; use an obviously reserved fake value instead.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a security-sensitive approval path and still has concrete authorization, availability, decision-semantics, and privacy concerns to fix before merge.

Concerns:

- **[medium] Implicit approval lookup runs before sender authorization:** `src/auto-reply/reply/commands-approve.ts:171`
  - body: The new bare `/approve <decision>` flow lists pending approvals and can reveal approval counts or an explicit id before checking whether the sender may approve anything on that channel.
  - confidence: 0.91
- **[medium] Unbounded server summary can suppress approvals:** `src/agents/pi-bundle-mcp-consent.ts:155`
  - body: A malicious or buggy MCP server can return an overlong summary and make `plugin.approval.request` fail validation, turning the approval flow into a denial instead of a user prompt.
  - confidence: 0.94
- **[medium] MCP allow-always is advertised without durable effect:** `src/agents/pi-bundle-mcp-materialize.ts:151`
  - body: The approval UI can offer `allow-always`, but the MCP consent path treats it as one one-shot confirmation. That weakens user expectations for a security decision and should be hidden or implemented deliberately.
  - confidence: 0.83
- **[low] Real-looking phone number in test fixture:** `src/agents/pi-bundle-mcp-consent.test.ts:430`
  - body: The new consent test includes a plausible E.164 WhatsApp sender in a session key; repository policy forbids committing real phone numbers, so the fixture should use a reserved fake value.
  - confidence: 0.9
