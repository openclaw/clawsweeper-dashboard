## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Forward session metadata into approval requests:** `src/agents/pi-bundle-mcp-materialize.ts:239-248`
  - body: `SessionMcpRuntime` already carries the run's `sessionKey`, and approval forwarding/filtering uses the request session to resolve the target conversation. This call never passes `params.runtime.sessionKey`, so session-mode plugin approval delivery can miss the channel that started the MCP tool call.
  - confidence: 0.92
- **[P2] Add the approvals key to the strict MCP schema:** `src/config/types.mcp.ts:43-49`
  - body: The new `mcp.approvals.enabled` key is typed, documented, and read, but current config validation still accepts only `servers` and `sessionIdleTtlMs` under `mcp`. A user setting the documented opt-out will fail strict config validation before this code can use it.
  - confidence: 0.97
- **[P2] Bound MCP approval descriptions before RPC validation:** `src/agents/pi-bundle-mcp-consent.ts:131-140`
  - body: `envelope.summary` comes from the MCP server and is concatenated directly into the plugin approval description. The gateway rejects plugin approval descriptions over 256 characters, so a long valid summary makes the approval request fail instead of prompting the user.
  - confidence: 0.95
- **[P2] Honor the opt-out before scrubbing confirmation tokens:** `src/agents/pi-bundle-mcp-materialize.ts:102-113`
  - body: `confirmation_token` is stripped before checking `consentEnabled === false`. That makes the documented opt-out unable to pass an envelope flow back to the model/server, because a follow-up model call with `confirmation_token` is still altered even when approvals are disabled.
  - confidence: 0.91
- **[P2] Keep no-route null decisions on the immediate failure path:** `src/agents/pi-bundle-mcp-consent.ts:159-166`
  - body: Current gateway request handling uses `decision: null` on `plugin.approval.request` for the no-approval-route case, while accepted two-phase requests omit `decision`. Falling through to `waitDecision` on null masks the existing unavailable path and returns a misleading generic denial.
  - confidence: 0.9
- **[P2] Race approval waits against run cancellation:** `src/agents/pi-bundle-mcp-materialize.ts:237-248`
  - body: `AnyAgentTool.execute` receives an `AbortSignal`, and the existing plugin approval path races `waitDecision` against it so stopped runs do not wait for the full timeout. This wrapper drops the signal before entering the approval flow, so cancelling a run during MCP consent can leave the tool call waiting until expiry.
  - confidence: 0.87
- **[P2] Make allow-always durable or do not advertise it:** `src/agents/pi-bundle-mcp-materialize.ts:151-157`
  - body: The branch accepts `allow-always` but handles it exactly like `allow-once`: it re-calls once and records no server/tool policy for future envelopes. Users choosing `allow-always` will still be prompted again on the next call, so either persist a real policy or restrict this MCP path to decisions it can honor.
  - confidence: 0.84
