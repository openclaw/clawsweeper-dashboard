## Summary

Keep open. The consent-envelope direction is useful and the PR includes credible live after-fix proof, but the latest head still leaves a parser-accepted slashless approval command form unsanitized and the changelog advertises an MCP decision path the implementation rejects.

## What This Changes

The PR adds an opt-in MCP consent envelope that gates bundle-MCP tool calls through plugin approvals, updates approval reply UX/config/docs, and adds MCP consent tests.

## Best Possible Solution

Land the consent-envelope approach after aligning the sanitizer/tests with the approval parser, correcting release copy, and recording the appropriate channel-visible proof for approval chat behavior.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the sanitizer only rewrites `/approve`, while the approval parser accepts slashless `approve` and the new implicit single-decision form. I did not run tests in this read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but the transcript defense and release copy need to match the parser and allowed-decision contract before merge.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the approval parser accepts `approve` without the slash and this PR makes `approve allow-once` a valid implicit approval when one request is pending. A malicious envelope summary can still contain a line with that parser-shaped text, so the defense-in-depth transcript-poisoning layer does not cover every command form the parser accepts.

That release note would direct operators to a durable approval path this flow intentionally does not support.
  - confidence: 0.9
