## Summary

Keep open: the PR addresses a real bundle-MCP approval gap and has supplied live proof, but PR head still has source-visible sanitizer and config/docs contract mismatches that should be repaired before maintainer acceptance.

## What This Changes

The PR adds an opt-in MCP consent envelope that gates bundle-MCP tool calls through plugin approvals, updates `/approve` UX, adds `mcp.approvals` config, docs, tests, and changelog copy.

## Best Possible Solution

Land the materializer-level consent gate after covering every parser-accepted approval command form, making config validation match the documented timeout contract, correcting user-facing copy, and getting maintainer sign-off on the MCP approval contract.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the sanitizer only matches `/approve` while the parser accepts slashless `approve`, and the timeout docs/runtime promise clamping while the schema rejects out-of-range values. The current-main baseline is also source-visible because bundle-MCP results return directly to the agent.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materializer layer looks like the right architecture, but the parser-defense and config contract mismatches should be repaired before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` rewrites only `/approve`, but the approval parser accepts `approve` without the slash. A malicious MCP summary can still contain parser-shaped `approve allow-once` text in the transcript-defense layer, so the sanitizer should cover every command form the parser accepts or the parser should stop accepting slashless approval commands.
  - confidence: 0.93
- **[P2] Align timeout validation with the clamp contract:** `src/config/zod-schema.ts:324-328`
  - body: `mcp.approvals.defaultTimeoutMs` is documented and implemented as a clamped value, but the strict config schema rejects values outside `[1000, 600000]` before the runtime clamp can run. Operators following the docs with a too-high or too-low value get config validation failure instead of the promised clamp.
  - confidence: 0.88
- **[P3] Advertise only supported MCP consent decisions:** `CHANGELOG.md:9`
  - body: The changelog still tells users they can reply with `allow-always` for MCP consent approvals, but this flow sends `allowedDecisions: ["allow-once", "deny"]` and the gateway rejects unavailable decisions. The release note should not advertise a durable approval mode this contract does not implement.
  - confidence: 0.91
- **[P3] Show the actual implicit approval prompt:** `docs/tools/mcp-consent-envelope.md:69-71`
  - body: The new docs say the approval message ends with the old explicit-id form, but the PR changed the prompt to list bare `/approve <decision>` choices while showing the id separately. The docs should match the shipped prompt so server authors and operators know what users will actually see.
  - confidence: 0.82
