## Summary

Keep open: the MCP consent-envelope feature is not on current main and the PR remains useful, but PR-head source still has approval-target scoping, opt-out propagation, and sanitizer/copy gaps that should block merge.

## What This Changes

This PR adds a bundle-MCP consent envelope that routes opted-in MCP tool calls through plugin approvals, re-calls approved tools with `confirmation_token`, and updates approval UX, config, docs, tests, and changelog text.

## Best Possible Solution

Land the consent envelope only after implicit approvals are scoped to the delivered approval target, the opt-out reaches every materialization path, and public sanitizer/docs/changelog copy matches the supported parser and decisions.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the global implicit approval resolver, the missing config propagation on the normal agent path, and the sanitizer/parser mismatch. I did not execute the code because this was a read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but the current patch needs approval-target scoping, config propagation, and defense-in-depth cleanup before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope implicit approvals to the delivered target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path resolves against every pending plugin and exec approval and accepts the single global candidate. An authorized sender in another chat/session can approve a prompt they were not shown; filter candidates by the request's channel/account/thread/session metadata or require an explicit ID.
  - confidence: 0.94
- **[P2] Honor the MCP approval opt-out in agent runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1229-1243`
  - body: The normal embedded attempt path calls `materializeBundleMcpToolsForRun` directly and only passes `agentId`/`sessionKey`. Because it never passes `params.config.mcp?.approvals?.enabled`, `mcp.approvals.enabled: false` still gates consent envelopes on the main agent run path despite the documented opt-out.
  - confidence: 0.93
- **[P2] Sanitize slashless approve commands:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` still only rewrites `/approve`, but the parser accepts `approve` without a slash and this PR makes `approve allow-once` a valid implicit approval form. A malicious envelope summary can still render a parser-shaped approval line in the defense-in-depth path this sanitizer is meant to close.
  - confidence: 0.91
- **[P3] Fix public copy for one-shot MCP approvals:** `CHANGELOG.md:43`
  - body: The consent request limits MCP approvals to `allow-once` and `deny`, but the changelog still advertises `allow-always`. That tells operators this adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.88
