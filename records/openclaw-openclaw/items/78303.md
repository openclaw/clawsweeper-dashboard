## Summary

Keep open: current main does not implement MCP consent-envelope gating, and this PR is the active implementation for the linked feature, but the latest head still has approval-boundary blockers that should be fixed before merge.

## What This Changes

The PR adds an opt-in MCP consent envelope that routes mutating bundle-MCP tool calls through existing plugin approvals, plus implicit approval replies, config/docs/tests, and changelog coverage.

## Best Possible Solution

Land the materializer-level MCP consent envelope after the implicit approval shortcut is scoped to the originating approval target, the MCP opt-out is honored in normal agent runs, and command-sanitizer/docs copy match the parser and emitted prompt.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the global implicit resolver, the missing config propagation in the normal attempt path, and the sanitizer/parser mismatch. I did not execute tests because this review is read-only.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materialization layer is the right architecture, but the current patch needs targeted approval scoping, config propagation, and sanitizer cleanup before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope implicit approvals to the originating target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The new bare `/approve <decision>` branch still lists every pending plugin and exec approval and binds to the only global candidate. If the only pending approval belongs to another chat/session, an authorized sender elsewhere can resolve it without seeing its prompt; filter candidates by the current channel/account/thread/session metadata or require an explicit ID.
  - confidence: 0.93
- **[P2] Pass the MCP approval opt-out into agent attempts:** `src/agents/pi-embedded-runner/run/attempt.ts:1229-1243`
  - body: The normal agent attempt path calls `materializeBundleMcpToolsForRun` directly and only passes `agentId`/`sessionKey`. In that path `mcp.approvals.enabled: false` is never applied, so deployments that opt out still gate consent envelopes during regular agent runs.
  - confidence: 0.9
- **[P2] Sanitize bare approve commands too:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without a slash and the new implicit form makes `approve allow-once` command-shaped. A malicious MCP summary can still render parser-shaped approval text in the defense-in-depth path this sanitizer is meant to close.
  - confidence: 0.89
- **[P3] Align the consent docs with the emitted reply format:** `docs/tools/mcp-consent-envelope.md:69-71`
  - body: The new docs say the approval message ends with `/approve <id> allow-once|deny`, but the implementation now emits bare `/approve allow-once | /approve deny` tokens with the ID on its own line for fallback. Update the docs so server authors and operators see the real prompt shape.
  - confidence: 0.86
