## What This Changes

The PR adds an MCP consent-envelope path for bundle-MCP tools, routes opted-in calls through plugin approvals, re-calls approved tools with `confirmation_token`, and updates approval commands, config, docs, changelog, and tests.

## Best Possible Solution

Land the consent-envelope design after fixing the failure-attribution bug and having maintainers explicitly accept the default-on fail-closed policy and security boundary.

## Reproduction Assessment

Yes. Current main source shows bundle-MCP tool results pass straight to agent-visible output, and the linked issue includes live vanilla 2026.4.24 evidence where an MCP `action_id` reached chat; I did not run a live channel deployment in this read-only review.

## Solution Assessment

Mostly yes. The bundle-MCP materialize layer is the right boundary because third-party `mcp.servers[]` do not go through plugin SDK hooks, but infrastructure-failure attribution should be fixed before maintainers accept the default-on policy.
