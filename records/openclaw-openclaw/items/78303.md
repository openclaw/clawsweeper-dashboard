## Summary

Keep open: this PR addresses a real MCP trust-boundary gap with supplied live proof, but the current head introduces an implicit approval path that can bind to the sole pending approval globally and leaves a bare `approve` injection shape unsanitized.

## What This Changes

The PR adds a bundle-MCP consent-envelope flow, routes opted-in MCP tool calls through plugin approvals, updates approval reply UX, adds `mcp.approvals.enabled`, tests, docs, and a changelog entry.

## Best Possible Solution

Land a scoped MCP consent contract after fixing implicit approval scoping, completing command-text sanitization, and aligning user-facing approval copy with the one-shot decision set.

## Reproduction Assessment

Yes for the review findings from source: the PR resolves bare approval decisions against the sole global pending record, and the sanitizer only covers `/approve` while the parser accepts bare `approve`. The broader MCP baseline/fix behavior is supported by the PR's live logs rather than re-run locally.

## Solution Assessment

No, not yet. Materializer-level interception and plugin-approval reuse are the right direction, but implicit approvals need delivery-context scoping or explicit IDs, and the sanitizer/copy must match the parser and actual decision set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Scope implicit approvals to the requesting channel:** `src/auto-reply/reply/commands-approve.ts:239-256`
  - body: The implicit `/approve <decision>` path lists all pending plugin and exec approvals, then binds to the sole global candidate. On a shared gateway, an authorized sender in one channel can approve another session's only pending action without receiving its prompt or ID. Filter candidates to the approval request's channel/account/session target, or require the explicit id unless the pending approval is routed to this sender.
  - confidence: 0.88
- **[P2] Neutralize bare approve commands in MCP summaries:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: The sanitizer only rewrites `/approve`, but the command parser accepts `approve` without a slash and this PR adds one-token implicit approvals. A malicious summary containing a newline followed by `approve allow-once` remains command-shaped if any channel echo or renderer regression feeds tool-emitted text back into the text command path. Sanitize the bare parser form too, or stop accepting slashless approval commands here.
  - confidence: 0.81
- **[P3] Align the changelog with one-shot MCP approvals:** `CHANGELOG.md:13`
  - body: The implementation sends `allowedDecisions: ["allow-once", "deny"]`, but the new changelog entry still advertises `allow-always`. That tells operators MCP approvals have a durable policy that this PR intentionally does not implement.
  - confidence: 0.92
