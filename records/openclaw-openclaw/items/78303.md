## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the approval parser still accepts the slashless form and the test suite explicitly verifies `approve abc allow-once`. That leaves one parser-accepted command shape intact in MCP-emitted text, undermining the defense-in-depth transcript poisoning fix; either sanitize the slashless verb too or remove that command form from the parser contract.
  - confidence: 0.94
- **[P2] Align timeout validation with the clamp contract:** `src/config/zod-schema.ts:327`
  - body: `mcp.approvals.defaultTimeoutMs` is documented and implemented as clamped to `[1000, 600000]`, but this strict schema rejects out-of-range values during config parsing before the clamp can run. Operators following the docs with a too-low or too-high value get a validation failure instead of the promised clamp, so either relax the schema to allow runtime clamping or change the public contract to validation.
  - confidence: 0.89
- **[P3] Advertise only supported MCP consent decisions:** `CHANGELOG.md:9`
  - body: The changelog still tells users to reply with `allow-always`, but this flow now sends `allowedDecisions: ["allow-once", "deny"]` and the rendered prompt/test expectations omit `allow-always`. The release note should not advertise a durable approval mode the MCP consent path does not support.
  - confidence: 0.91
- **[P3] Show the actual implicit approval prompt:** `docs/tools/mcp-consent-envelope.md:69-71`
  - body: The docs say the approval message ends with `Reply with: /approve <id> allow-once|deny`, but the PR changes the renderer to list bare `/approve <decision>` choices while showing the id separately. Server authors and operators should see the actual prompt format documented.
  - confidence: 0.84
