## Summary

Keep open: the PR targets a real current-main gap and includes convincing live proof, but PR head still leaves a parser-accepted slashless approval command form unsanitized and its changelog advertises an unsupported MCP approval decision.

## What This Changes

The PR adds an MCP consent-envelope path that routes opted-in bundle-MCP tool calls through plugin approvals, adds `mcp.approvals` config, updates `/approve` parsing/prompt copy, and documents/tests the contract.

## Best Possible Solution

Land the consent-envelope approach only after the sanitizer and tests cover every approval command form accepted by the parser, and public release copy matches the actual `allow-once`/`deny` MCP consent contract.

## Reproduction Assessment

Yes for the review finding: PR-head source shows the sanitizer only matches `/approve`, while the `/approve` parser accepts slashless `approve` and the new implicit single-decision form. I did not run tests in this read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but the transcript defense and changelog copy need to match the parser and allowed-decision contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the approval parser accepts `approve` without the slash and this PR makes `approve allow-once` a valid implicit approval when one request is pending. A malicious envelope summary can still contain a newline followed by that parser-shaped text, leaving the defense-in-depth transcript-poisoning path this helper is meant to close.
  - confidence: 0.92
- **[P3] Advertise only supported MCP approval decisions:** `CHANGELOG.md:97`
  - body: The changelog still tells users to reply with `allow-always` for MCP consent approvals, but this PR sends `allowedDecisions: ["allow-once", "deny"]` and the gateway rejects decisions outside a request's allowed set. That release note would direct operators to a durable approval path the implementation intentionally does not support.
  - confidence: 0.88
