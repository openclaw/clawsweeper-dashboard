## Summary

Keep open: the PR implements a plausible bundle-MCP consent gate and includes useful live proof, but PR-head source still has blocking config propagation, sanitizer/parser, and config contract mismatches before maintainer security/product review can finish.

## What This Changes

This PR adds an opt-in MCP consent envelope for bundle-MCP tools, routes approval through plugin approvals, changes `/approve` reply UX, and adds config, docs, changelog, and focused tests.

## Best Possible Solution

Land the materializer-level consent gate after centralizing MCP approval config resolution for every run path, aligning sanitizer/parser/schema/docs/changelog contracts, and getting explicit maintainer approval for the new trust boundary.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the normal embedded run path bypasses the new MCP approval config, the sanitizer/parser mismatch is visible in adjacent source, and the schema/docs clamp mismatch is direct. I did not execute tests because this was a read-only review.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materialization layer is a maintainable shape, but the branch needs narrow repairs before it matches its documented config and security contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass MCP approval config into embedded runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1313-1327`
  - body: `runEmbeddedAttempt` materializes the session MCP runtime directly but never passes the new `mcp.approvals.enabled` or `mcp.approvals.defaultTimeoutMs` settings. Normal agent turns therefore ignore the documented opt-out and timeout override, because those config defaults are only resolved by `createBundleMcpToolRuntime`, which this path bypasses.
  - confidence: 0.94
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: The sanitizer only rewrites `/approve`, while `commands-approve.ts` accepts slashless `approve` as the same command. Since this defense is meant to cover parser-shaped tool-emitted text, either sanitize both accepted spellings or narrow the parser so the sanitizer and command grammar match.
  - confidence: 0.9
- **[P2] Align timeout validation with the clamp contract:** `src/config/zod-schema.ts:340`
  - body: The docs and runtime describe `mcp.approvals.defaultTimeoutMs` as clamped to `[1000, 600000]`, but the strict schema rejects out-of-range values before the runtime clamp can apply. Relax the schema and clamp at runtime, or document this as validation rather than clamping.
  - confidence: 0.88
- **[P3] Advertise only supported MCP consent decisions:** `CHANGELOG.md:9`
  - body: This changelog entry still tells users they can reply with `allow-always`, but the MCP consent request sends `allowedDecisions: ["allow-once", "deny"]`. Update the release note so users do not try a durable approval mode this path does not implement.
  - confidence: 0.9
