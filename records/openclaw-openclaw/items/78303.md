## Summary

Keep open: the PR is a plausible core feature with supplied live proof, but the current head still has source-proven blockers in config propagation, approval-text sanitization, and the documented timeout contract.

## What This Changes

Adds an opt-in MCP consent envelope around bundle-MCP tool execution, routes approvals through plugin approvals and `/approve`, adds `mcp.approvals` config/docs/tests/changelog, and updates plugin approval prompt text.

## Best Possible Solution

Land the materializer-level MCP consent gate only after the normal embedded run path honors the config contract and the approval parser, sanitizer, schema, docs, and changelog describe one consistent approval surface.

## Reproduction Assessment

Yes for the review blockers: the PR-head source shows the embedded attempt path bypasses the new MCP approval config, and the sanitizer/parser mismatch is visible by comparing the consent module with the approval command parser. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materialization layer is a good shape, but the branch needs narrow repairs before the implementation matches its documented config and security contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass MCP approval config into embedded attempts:** `src/agents/pi-embedded-runner/run/attempt.ts:1632-1646`
  - body: `runEmbeddedAttempt` uses `materializeBundleMcpToolsForRun` directly but only forwards `agentId` and `sessionKey`. That bypasses the config resolution in `createBundleMcpToolRuntime`, so normal agent turns ignore `mcp.approvals.enabled: false` and `mcp.approvals.defaultTimeoutMs`.
  - confidence: 0.94
- **[P2] Neutralize every parser-accepted approval command:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: The sanitizer only rewrites `/approve`, but `commands-approve.ts` accepts `approve` without the slash. A malicious MCP summary can still leave command-shaped approval text intact in rendered tool output, so either sanitize the slashless form too or stop accepting it.
  - confidence: 0.9
- **[P2] Make timeout validation match the clamp contract:** `src/config/zod-schema.ts:382`
  - body: The docs and runtime describe `mcp.approvals.defaultTimeoutMs` as clamped to `[1000, 600000]`, but the strict schema rejects out-of-range values before the clamp can run. Either relax validation and clamp at runtime, or document validation instead of clamping.
  - confidence: 0.88
- **[P3] Advertise only supported MCP consent decisions:** `CHANGELOG.md:9`
  - body: The changelog says MCP consent accepts `allow-always`, but this path now sends `allowedDecisions: ["allow-once", "deny"]`. Update the user-facing entry so operators do not try a durable approval mode this implementation does not support.
  - confidence: 0.9
