## Summary

Keep open: the PR addresses a real bundle-MCP approval gap, but PR head still has blocking approval-path issues around implicit approval scoping, config opt-out propagation, and sanitizer coverage.

## What This Changes

Adds an MCP consent-envelope path that intercepts opted-in bundle-MCP tool results, routes them through plugin approvals, re-calls approved tools with `confirmation_token`, and updates approval UX, config, docs, tests, and changelog.

## Best Possible Solution

Land the consent envelope only after target-scoped implicit approval, config opt-out propagation, parser-matched sanitization, and user-visible copy are aligned, with maintainer sign-off on the approval contract.

## Reproduction Assessment

Yes for the PR review findings: PR-head source shows the global implicit approval resolver, the missing config opt-out on the normal agent path, and the sanitizer/parser mismatch. I did not execute the code because this review was read-only.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but the current patch needs approval-target scoping, config propagation, and defense-in-depth cleanup before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope implicit approvals to the originating target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path still lists every pending plugin and exec approval and binds to the only global candidate. If the sole pending approval belongs to another chat/session, an authorized sender elsewhere can resolve it without seeing that prompt; filter candidates by the current approval target metadata or require an explicit id.
  - confidence: 0.94
- **[P2] Honor the MCP approval opt-out in agent runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1229-1243`
  - body: The normal embedded attempt path calls `materializeBundleMcpToolsForRun` directly and only passes `agentId`/`sessionKey`. Because it never passes `params.config.mcp.approvals.enabled`, `mcp.approvals.enabled: false` still gates consent envelopes on the main agent run path despite the documented opt-out.
  - confidence: 0.93
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:95-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the parser accepts slashless `approve` and this PR makes `approve allow-once` a valid implicit approval form. A malicious envelope summary can include a newline followed by a parser-shaped approval line that remains intact in the defense-in-depth path this sanitizer is meant to close.
  - confidence: 0.9
- **[P3] Fix public copy for MCP approval replies:** `CHANGELOG.md:44`
  - body: The changelog still advertises `/approve <id> allow-once|allow-always|deny`, but this PR now restricts MCP consent approvals to `allow-once`/`deny` and renders bare `/approve <decision>` hints. That tells operators to use a durable approval path the implementation intentionally does not provide.
  - confidence: 0.86
