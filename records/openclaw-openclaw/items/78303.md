## Summary

Keep open: the MCP consent direction is plausible and the contributor supplied useful live proof, but the latest PR head still has a security-sensitive approval-routing bug plus a gateway contract mismatch that should block merge.

## What This Changes

The PR adds an opt-in MCP consent envelope that routes bundle-MCP tool calls through plugin approvals and idless `/approve` replies, with docs, config, tests, and changelog updates.

## Best Possible Solution

Land a materializer-level MCP consent gate only after idless approval resolution remains bound to the initiating channel/session and gateway no-route responses preserve their current semantics.

## Reproduction Assessment

Yes for the review blockers: the approval parser and gateway no-route behavior are source-reproducible from the PR head and current gateway handlers. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Reusing plugin approvals from the MCP materializer is a good direction, but the idless approval UX must stay scoped to the same channel/session and `decision: null` must continue to mean no approval route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Bind implicit approvals to the current chat:** `src/auto-reply/reply/commands-approve.ts:244-261`
  - body: This accepts `/approve allow-once` by picking the sole pending approval visible to the gateway client, but that visibility is client/device scoped rather than the channel/account that sent the command. A sender in one chat can resolve the only pending approval for another chat/session, so filter implicit candidates with the existing channel/account/session binding helpers before accepting one.
  - confidence: 0.88
- **[P2] Preserve `decision: null` as no approval route:** `src/agents/pi-bundle-mcp-consent.ts:226-229`
  - body: The current gateway returns `{decision: null}` after expiring the approval because no approval route exists; accepted two-phase requests omit `decision`. Falling through to `waitDecision` turns an unavailable route into a generic user denial, so handle null immediately as unavailable/error and update the regression test to exercise the real gateway contract.
  - confidence: 0.9
