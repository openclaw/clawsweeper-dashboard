## Summary

Keep open: this is an active external feature/security PR, not implemented on main, and the latest head still leaves the implicit approval shortcut globally scoped plus sanitizer/copy mismatches.

## What This Changes

The PR adds an opt-in MCP consent-envelope flow that intercepts bundle-MCP tool results, asks through plugin approvals, re-calls with a confirmation token, updates `/approve` UX, adds `mcp.approvals.enabled`, and documents/tests the contract.

## Best Possible Solution

Land a scoped MCP approval contract only after implicit approval resolution is bound to the delivered target or explicit ID, tool-emitted command text is sanitized consistently with the parser, and user-facing docs/changelog match the one-shot decision set.

## Reproduction Assessment

Yes for the review findings: source shows `/approve <decision>` resolving from global pending approval lists, and the sanitizer only rewrites `/approve` while the parser accepts slashless `approve`. The broader feature behavior is supported by contributor live logs, not a local rerun.

## Solution Assessment

No, not yet. Materializer-level interception and reusing plugin approvals are plausible, but the implicit approval shortcut must be target-scoped or explicit-ID-only, and sanitizer/copy must match the parser and actual decision set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope implicit approvals to the delivered target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path lists every pending plugin and exec approval, then binds to the sole global candidate. On a shared gateway, an authorized sender in one chat can approve another channel/session's only pending request without receiving that prompt or ID. Filter candidates to the approval routed to this sender/account/thread/session, or require the explicit ID unless the candidate belongs to the current delivery target.
  - confidence: 0.9
- **[P2] Sanitize slashless approve commands too:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser also accepts `approve` without a slash, and this PR makes `approve allow-once` a valid implicit approval. A malicious MCP summary containing a newline followed by `approve allow-once` remains command-shaped in the defense-in-depth layer this sanitizer is meant to provide.
  - confidence: 0.86
- **[P3] Align the changelog with one-shot MCP approvals:** `CHANGELOG.md:31`
  - body: The consent request now sends `allowedDecisions: ["allow-once", "deny"]`, and the docs describe one-shot approval, but the changelog still advertises `allow-always`. That tells operators this PR adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.92
