## Summary

Keep open: the consent-envelope direction is useful and live-proofed, but the latest PR head still has source-proven blockers around config propagation, approval-command sanitization, and the documented timeout contract.

## What This Changes

The PR adds an opt-in MCP consent envelope that gates bundle-MCP tool calls through plugin approval before re-calling tools with a confirmation token, plus approval UX, config, docs, changelog, and tests.

## Best Possible Solution

Land the materializer-level consent gate after centralizing MCP approval config resolution for every entry path, matching sanitizer coverage to the parser, and correcting docs/changelog to the allow-once/deny contract.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source shows the normal run path bypasses config resolution, and the sanitizer/parser mismatch is visible by comparing the consent module with the approval command parser. I did not run tests because this was a read-only review.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materialize layer is a plausible solution, but the current patch needs narrow repairs before it matches its documented config and security contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass MCP approval config into normal runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1313-1327`
  - body: `createBundleMcpToolRuntime` is the only path that reads `cfg.mcp.approvals`, but normal agent turns call `materializeBundleMcpToolsForRun` directly here. That means `mcp.approvals.enabled: false` and `mcp.approvals.defaultTimeoutMs` are ignored in the primary runtime path; pass the resolved values here or move the resolver into a shared helper used by both paths.
  - confidence: 0.94
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without the slash. A malicious consent summary can still leave parser-shaped approval text intact in the transcript-defense layer, so the sanitizer/tests should cover the same command spellings the parser accepts or the parser should be narrowed.
  - confidence: 0.91
- **[P2] Align timeout validation with the clamp contract:** `src/config/zod-schema.ts:337-341`
  - body: The docs and runtime say `mcp.approvals.defaultTimeoutMs` is clamped to `[1000, 600000]`, but the strict schema rejects out-of-range values before the runtime clamp can run. Operators following the docs get a config validation failure instead of the promised clamp; either relax the schema and clamp or document validation instead.
  - confidence: 0.88
- **[P3] Advertise only supported MCP consent decisions:** `CHANGELOG.md:9`
  - body: The changelog says users can reply with `allow-always`, but the MCP consent requester sends `allowedDecisions: ["allow-once", "deny"]` and the gateway rejects unavailable decisions. Update the release note so users do not try a durable approval mode this contract does not implement.
  - confidence: 0.92
- **[P3] Show the actual implicit approval prompt:** `docs/tools/mcp-consent-envelope.md:69-71`
  - body: The new docs describe the approval message as the old explicit-id form, but the PR changed plugin approval prompts to list bare `/approve <decision>` choices while showing the id separately. Make the docs match the shipped prompt so operators and MCP server authors know what users will see.
  - confidence: 0.82
