## Summary

Keep open: the feature is useful and has credible live proof, but the current branch still introduces approval authorization and defense-in-depth gaps that should block merge.

## What This Changes

The PR adds a bundle-MCP consent-envelope approval gate, `mcp.approvals` config, implicit `/approve` reply UX, docs/changelog text, and focused tests.

## Best Possible Solution

Land the consent envelope only after implicit approvals are target-scoped or made explicit-ID-only, sanitizer coverage matches every parser-accepted command form, public copy reflects one-shot decisions, and maintainers sign off on the MCP elicitation tradeoff.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the implicit-id resolver querying global pending lists without target matching, and the sanitizer only rewrites `/approve` while the parser accepts slashless `approve`. I did not run tests because this is a read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look maintainable, but the implicit approval shortcut and sanitizer/copy mismatches need repair before product/security review can approve the approach.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope implicit approvals to the inbound target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path still lists every pending plugin and exec approval, merges them, and approves the sole global candidate. An authorized sender in one chat can therefore approve a different channel/session's only pending request without seeing its prompt or ID; filter candidates to the current delivery target/account/thread/session, or require an explicit ID unless the candidate is proven to belong to this inbound context.
  - confidence: 0.93
- **[P2] Sanitize slashless approval commands too:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without a slash and this PR makes `approve allow-once` a valid implicit approval shape. A malicious MCP summary that begins with `approve allow-once` remains command-shaped in the defense layer this sanitizer is meant to provide.
  - confidence: 0.9
- **[P3] Align the changelog with one-shot MCP approvals:** `CHANGELOG.md:37`
  - body: The implementation sends MCP consent requests with `allowedDecisions: ["allow-once", "deny"]`, and the docs describe allow-once-only behavior, but the changelog still advertises `allow-always`. That tells operators this adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.9
