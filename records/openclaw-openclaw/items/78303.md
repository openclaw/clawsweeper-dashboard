## Summary

Keep this PR open: current main does not implement MCP consent envelopes, and the branch still has source-proven correctness/security blockers despite useful live proof and green CI signals.

## What This Changes

The PR adds a default-on MCP consent envelope path for bundle-MCP tool calls, routes approvals through plugin approvals, updates approval command UX/config/docs/changelog, and adds focused tests.

## Best Possible Solution

Land the materializer-level consent gate after every run path honors the mcp.approvals contract, non-consent tools remain compatible, and approval parser/sanitizer/docs/schema behavior is aligned.

Yes for the review blockers by source inspection: the normal embedded run path ignores the new config, and the parser/sanitizer plus schema/docs mismatches are visible in PR-head source. I did not execute tests because the checkout must remain read-only.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materialization layer is the right shape, but this branch needs narrow compatibility, config, sanitizer, schema, and docs repairs before maintainer product review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass MCP approval config into embedded runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1632-1645`
  - body: `runEmbeddedAttempt` materializes the session MCP runtime directly and only passes `agentId`/`sessionKey`. Normal agent turns therefore ignore `mcp.approvals.enabled` and `mcp.approvals.defaultTimeoutMs`, so the documented opt-out and timeout setting do not work on the main path.
  - confidence: 0.95
- **[P2] Preserve inputs for non-consent tools:** `src/agents/pi-bundle-mcp-materialize.ts:105-114`
  - body: The wrapper strips `confirmation_token` before it knows whether the tool returned a consent envelope. That changes existing/non-envelope MCP tools that legitimately declare a `confirmation_token` input, contradicting the pass-through contract for servers that do not opt in.
  - confidence: 0.86
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` rewrites only `/approve`, but the command parser accepts `approve` without the slash. A malicious consent summary can still leave parser-shaped approval text intact in the transcript-defense layer, so sanitize both accepted spellings or narrow the parser contract.
  - confidence: 0.91
- **[P2] Align timeout validation with the clamp contract:** `src/config/zod-schema.ts:382`
  - body: The docs and runtime describe `mcp.approvals.defaultTimeoutMs` as clamped to `[1000, 600000]`, but the strict schema rejects out-of-range values before runtime clamping can apply. Either relax schema validation and clamp, or document rejection instead of clamping.
  - confidence: 0.88
- **[P3] Advertise only supported MCP consent behavior:** `CHANGELOG.md:9`
  - body: The changelog still says MCP consent users can reply with `allow-always` and that non-envelope servers are unaffected. The PR sends only `allow-once`/`deny`, and the unconditional `confirmation_token` scrub means non-envelope tools with that input name are affected.
  - confidence: 0.87
