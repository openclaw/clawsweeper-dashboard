## Summary

Keep open: the MCP consent direction is useful and the contributor supplied real live proof, but the current branch weakens the approval boundary by letting a bare `/approve <decision>` select the only backend-visible pending approval without tying it to the chat or session that received the prompt.

## What This Changes

Adds an opt-in MCP consent envelope around bundle-MCP tool calls, routes approval through plugin approvals and `/approve`, adds `mcp.approvals` config/docs/changelog, and updates approval prompt and command handling.

## Best Possible Solution

Land the materializer-level MCP consent gate after implicit approvals are scoped to the delivered chat/session or require explicit IDs, and after maintainers accept the new MCP envelope contract.

## Reproduction Assessment

Yes for the review blocker: source inspection shows the no-id `/approve <decision>` path combines backend-visible pending approvals without route filtering. A concrete repro would create one pending approval from another session, then send bare `/approve allow-once` from an authorized different chat.

## Solution Assessment

No, not yet. Reusing plugin approvals for MCP consent is the right general shape, but the implicit approval shortcut must preserve per-prompt routing instead of treating one backend-visible pending record as safe to approve.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope implicit approvals to the delivered chat:** `src/auto-reply/reply/commands-approve.ts:244-261`
  - body: The new no-id `/approve <decision>` branch combines every pending plugin and exec approval returned by the backend client, then resolves the sole candidate without checking that it belongs to this command's channel, account, sender, or session. Since gateway list visibility is tied to backend/requester metadata rather than the chat sender who typed the command, an authorized sender in a different chat can approve the one pending mutating request. Filter candidates to the current delivered route/session, or keep requiring an explicit ID.
  - confidence: 0.88
