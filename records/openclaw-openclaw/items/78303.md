## Summary

Keep open: the consent-envelope direction is valuable and has strong real-world proof, but the current patch has a merge-blocking compatibility problem because it reserves and strips a generic MCP tool argument before any server has opted into the envelope.

## What This Changes

The PR adds a default-on bundle-MCP consent envelope flow that routes approval-required MCP tool results through plugin approvals and `/approve`, plus config, docs, approval parser changes, and tests.

## Best Possible Solution

Land the consent-envelope feature only after the confirmation-token contract is compatibility-safe, either by using a namespaced/reserved field with maintainer approval or by adding an explicit opt-in path that leaves non-envelope tools' arguments unchanged.
