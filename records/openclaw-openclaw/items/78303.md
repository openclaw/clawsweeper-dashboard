## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Add approvals to the strict MCP config schema:** `src/config/types.mcp.ts:43-49`
  - body: The new `mcp.approvals.enabled` key is documented, typed, and read by `createBundleMcpToolRuntime`, but the runtime config schema is still strict and only accepts `mcp.servers` plus `mcp.sessionIdleTtlMs`. Setting the documented opt-out will be rejected before this code can see it, so please add the nested `approvals.enabled` schema/help/generated-doc coverage with this type change.
  - confidence: 0.97
- **[P2] Clamp approval descriptions before gateway validation:** `src/agents/pi-bundle-mcp-consent.ts:131-140`
  - body: `envelope.summary` comes from the MCP server and is concatenated into the plugin approval `description` without a length bound, but the gateway rejects descriptions longer than its 256-character limit. A long but valid summary will make `plugin.approval.request` fail and deny the tool without prompting the user; truncate or otherwise bound the text before sending it.
  - confidence: 0.92
- **[P2] Honor the consent opt-out before scrubbing tokens:** `src/agents/pi-bundle-mcp-materialize.ts:102-113`
  - body: `confirmation_token` is stripped before the `consentEnabled === false` branch. That makes the documented opt-out unable to pass the envelope flow back to the model/server, because a follow-up call with `confirmation_token` will still be altered even when MCP approvals are disabled. Check the opt-out before scrubbing, or otherwise bypass token rewriting when the gate is disabled.
  - confidence: 0.88
- **[P2] Make allow-always persist or stop offering it:** `src/agents/pi-bundle-mcp-materialize.ts:151-157`
  - body: The branch accepts `allow-always` but treats it exactly like `allow-once`: it re-calls the tool once with `confirmation_token` and stores no server/tool policy for future envelopes. Users who pick `allow-always` will be prompted again on the next call, and the MCP server never receives the distinction, so either add a real allow-always policy/cache or restrict this MCP path to decisions it can honor.
  - confidence: 0.84
