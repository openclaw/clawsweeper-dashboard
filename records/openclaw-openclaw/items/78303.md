## Summary

Keep open: the PR is useful and live-proven, but current head still leaves the new no-ID approval shortcut globally scoped and the MCP-output sanitizer does not cover every approve form accepted by the parser.

## What This Changes

The branch adds a bundle-MCP consent-envelope approval gate, mcp.approvals config support, no-ID /approve UX, docs/changelog text, and focused tests.

## Best Possible Solution

Land the MCP consent envelope only after no-ID approvals are target-scoped or explicit-ID-only, tool-emitted approval text is neutralized for every parser-accepted form, and public copy matches the one-shot decision set.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows no-ID approvals query global pending lists and the sanitizer only matches /approve while the parser accepts approve without a slash. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look maintainable, but the no-ID approval shortcut must be target-scoped or explicit-ID-only and the sanitizer/changelog need to match the parser and allowed decisions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope implicit approvals to the inbound target:** `src/auto-reply/reply/commands-approve.ts:212-256`
  - body: The bare `/approve <decision>` path still lists all pending plugin and exec approvals, merges them, and binds to the sole global candidate. An authorized sender in one chat can therefore approve a different channel/session's only pending request without seeing its prompt or ID. Filter candidates to the current delivery target/account/thread/session, or require an explicit ID unless the candidate is proven to belong to this inbound context.
  - confidence: 0.91
- **[P2] Sanitize slashless approve commands too:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts `approve` without a slash and this PR makes `approve allow-once` a valid implicit approval shape. A malicious MCP summary containing a newline followed by `approve allow-once` remains command-shaped in the defense-in-depth layer this sanitizer is meant to provide.
  - confidence: 0.88
- **[P3] Align the changelog with one-shot MCP approvals:** `CHANGELOG.md:33`
  - body: The implementation sends MCP consent requests with `allowedDecisions: ["allow-once", "deny"]`, and the docs now describe allow-once-only behavior, but the changelog still advertises `allow-always`. That tells operators this adds a durable policy path the implementation intentionally does not support.
  - confidence: 0.9
