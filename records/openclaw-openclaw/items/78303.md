## Label Justifications

- `P1`: The PR touches a security-sensitive MCP approval boundary and can affect live channel-mediated approval workflows.
- `merge-risk: 🚨 compatibility`: Default-on `mcp.approvals.enabled` can change existing envelope-speaking MCP deployments from model-visible results to prompts, timeouts, or unavailable results after upgrade.
- `merge-risk: 🚨 message-delivery`: The approval prompt and `/approve` handling depend on correct channel, target, account, and thread routing, and the PR currently carries only part of that origin.
- `merge-risk: 🚨 security-boundary`: The patch is meant to prevent model-mediated approval, but it still forwards model-supplied `confirmation_token` before user approval.

## Summary

Keep open: current main still lacks bundle-MCP consent-envelope approval handling, and this PR remains the canonical implementation candidate, but it should not merge before the remaining security/compatibility issues and current-head proof gap are resolved.

## What This Changes

Adds a bundle-MCP consent-envelope flow that routes qualifying MCP tool calls through plugin approvals, extends `/approve` handling, and adds `mcp.approvals` config, docs, and regression tests.

## Best Possible Solution

Land a maintainer-approved MCP consent contract that reserves token redemption to the runtime, binds approvals to the exact call and full origin, uses upgrade-safe defaults, and includes current-head live proof.

## Reproduction Assessment

Yes. Current main directly returns bundle-MCP tool results, including `structuredContent`, to the agent without a consent-envelope approval bridge; I verified that path by source inspection and the linked issue includes live HomeBrain/WhatsApp evidence.

## Solution Assessment

No. The materialization boundary is plausible, but this patch is not yet the best fix because it forwards model-supplied confirmation tokens on the first call, defaults the new gate on without upgrade proof, and omits full origin binding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Reserve `confirmation_token` before first MCP execution:** `src/agents/agent-bundle-mcp-materialize.ts:137`
  - body: The first MCP call still forwards the model's input unchanged, including any `confirmation_token`. That means token redemption is not actually reserved for the channel-approved re-call path; a model or prompt-injected tool call can still present a token to an envelope-speaking server before OpenClaw has issued or resolved an approval.
  - confidence: 0.88
- **[P1] Make MCP approvals opt-in or prove upgrade safety:** `src/agents/agent-bundle-mcp-materialize.ts:89`
  - body: `mcp.approvals.enabled` resolves to true unless explicitly disabled, so existing MCP servers that already return `requires_confirmation` envelopes can start blocking on approval delivery after upgrade. If no plugin approval route is configured, those tool calls now fail as unavailable or time out instead of preserving the prior model-visible result path.
  - confidence: 0.86
- **[P1] Carry the full approval origin into MCP consent requests:** `src/agents/agent-bundle-mcp-consent.ts:216`
  - body: The MCP consent request only forwards channel and target, even though the runner has account/thread context and plugin approval payloads support those fields. For multi-account or threaded channels, approval delivery and implicit `/approve` scoping cannot be bound to the exact originating surface.
  - confidence: 0.82
