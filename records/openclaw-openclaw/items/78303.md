## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Block first-call token redemption:** `src/agents/agent-bundle-mcp-materialize.ts:134-138`
  - body: The consent wrapper still sends `params.input` to the MCP server before detecting an envelope. If that input contains `confirmation_token`, an envelope-speaking server can treat the first call as a redemption and mutate state before OpenClaw has issued or resolved any approval. Scrub or reject model-supplied confirmation tokens before the consent-negotiation call, or gate this behind an explicit maintainer-approved compatibility path.
  - confidence: 0.88
- **[P1] Bind MCP approvals to the full origin:** `src/agents/embedded-agent-runner/run/attempt.ts:1457-1460`
  - body: The new approval request carries `agentId`, `sessionKey`, channel, and target, but not the available account/thread identity or a canonical call/arguments digest. That leaves the approval record weaker than the exact call and conversation the user is approving, and explicit `/approve <id>` resolution can still authorize a copied or misdelivered id from another eligible surface. Carry and verify the full origin plus call instance before treating this as a security boundary.
  - confidence: 0.82
- **[P1] Make default-on gating upgrade-safe:** `src/agents/agent-bundle-mcp-materialize.ts:87-89`
  - body: `mcp.approvals.enabled` defaults to true for every existing config. That is not a no-op for deployments whose MCP servers already return a consent envelope: after upgrade their calls can start blocking on channel approval, fail unavailable when no delivery route exists, or time out instead of preserving the prior model-visible envelope behavior. Make this opt-in, migrate with explicit compatibility proof, or get a maintainer decision documenting the breaking default.
  - confidence: 0.86
