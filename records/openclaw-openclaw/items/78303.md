## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Make MCP approvals opt-in or prove the upgrade path:** `src/agents/agent-bundle-mcp-materialize.ts:87-89`
  - body: This resolves consentEnabled to true unless users set `mcp.approvals.enabled: false`. Any existing MCP server that already returns this envelope will stop returning a normal model-visible result and instead wait on plugin approval routing after upgrade, so this needs explicit opt-in or maintainer-approved upgrade proof/docs for the default-on change.
  - confidence: 0.87
- **[P1] Reserve confirmation_token before the first consent call:** `src/agents/agent-bundle-mcp-materialize.ts:130-138`
  - body: The first MCP call still sends `params.input` unchanged, so a model-supplied `confirmation_token` reaches the MCP server before OpenClaw has detected an envelope or issued channel approval. That weakens the claimed anti-self-approval boundary; the protocol needs a way to reserve or scrub this field for consent-capable tools without breaking non-envelope tools.
  - confidence: 0.89
- **[P1] Bind MCP approvals to the full origin and exact call:** `src/agents/agent-bundle-mcp-consent.ts:213-216`
  - body: The approval request forwards channel and target, but not the account/thread origin fields used by the existing plugin approval path, and it has no canonical digest binding the approval to the exact server/tool/arguments being retried. Side-effecting MCP approvals should be scoped at least as tightly as current plugin approvals so a different account/thread or changed retry cannot authorize the wrong call.
  - confidence: 0.84
