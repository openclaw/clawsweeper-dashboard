## Summary

Keep this PR open: the feature is useful and current main does not already implement it, but PR head still has approval-path correctness and security blockers that should be fixed before merge.

## What This Changes

The PR adds an MCP consent-envelope flow that routes opted-in bundle-MCP tool calls through plugin approvals, re-calls approved tools with `confirmation_token`, updates `/approve` UX, config, docs, tests, and changelog text.

## Best Possible Solution

Land the materializer-level MCP consent gate after the implicit approval shortcut is target-scoped or removed, the opt-out is propagated through every materialization path, sanitizer coverage matches the parser, and public copy reflects the supported decisions.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the global implicit resolver, the missing config flag on the normal agent path, and the sanitizer/parser mismatch. I did not execute the code because this was a read-only review.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materialization layer is the right direction, but the current patch needs scoped implicit approvals, config propagation, and sanitizer cleanup before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope implicit approvals to the originating target:** `src/auto-reply/reply/commands-approve.ts:239`
  - body: The bare `/approve <decision>` path lists all pending plugin and exec approvals and binds to the only global candidate. If the sole pending approval was delivered to another channel/account/thread/session, an authorized sender elsewhere can approve it without seeing that prompt; filter candidates by the current approval target metadata or require an explicit id.
  - confidence: 0.94
- **[P2] Honor the MCP approval opt-out in agent runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1229-1243`
  - body: The normal embedded attempt path calls `materializeBundleMcpToolsForRun` directly and only passes `agentId` and `sessionKey`. Because it never passes the resolved `params.config.mcp.approvals.enabled` flag, `mcp.approvals.enabled: false` still gates consent envelopes on the main agent run path despite the documented opt-out.
  - confidence: 0.93
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the command parser accepts slashless `approve`, and this PR makes `approve allow-once` a valid implicit approval form. A malicious envelope summary can include a newline followed by a parser-shaped slashless approval line that remains intact in the defense-in-depth path this sanitizer is meant to close.
  - confidence: 0.9
- **[P3] Fix public copy for MCP approval replies:** `CHANGELOG.md:44`
  - body: The changelog still advertises `/approve <id> allow-once|allow-always|deny`, but this PR now restricts MCP consent approvals to `allow-once`/`deny` and renders bare `/approve <decision>` hints. That tells operators to use a durable approval path the implementation intentionally does not provide.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR adds a useful approval gate, but the implicit approval shortcut and incomplete sanitizer leave concrete approval-path hardening gaps.

Concerns:

- **[high] Implicit approval can cross approval targets:** `src/auto-reply/reply/commands-approve.ts:239`
  - body: The no-id approval path resolves against the single global pending approval after sender/kind authorization, without checking that the approval was delivered to the current channel/account/thread/session.

  - body: MCP-emitted approval text only rewrites `/approve`, while the command parser accepts slashless `approve`; this leaves a command-shaped self-approval string in the compromised-output defense layer.
