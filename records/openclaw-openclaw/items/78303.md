## Summary

Keep open: current PR head still has merge blockers. The normal embedded agent run path bypasses the helper that reads `mcp.approvals` config, and the transcript-defense sanitizer does not cover every approval command spelling accepted by the parser.

## What This Changes

The PR adds an opt-in MCP consent envelope that routes bundle-MCP tool calls through plugin approvals before re-calling the tool with a confirmation token.

## Best Possible Solution

Land the materializer-level consent gate after centralizing MCP approval config resolution for every entry path, matching sanitizer coverage to the parser, and aligning docs/changelog with the allow-once/deny contract.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the normal run path bypasses config resolution, and the sanitizer/parser mismatch is visible by comparing the consent module with the `/approve` parser. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Reusing plugin approvals at the bundle-MCP materialize layer is the right shape, but the current PR needs narrow repairs before it matches its documented config and security contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass MCP approval config into normal runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1313-1327`
  - body: `createBundleMcpToolRuntime` is the only path that reads `cfg.mcp.approvals`, but `runEmbeddedAttempt` calls `materializeBundleMcpToolsForRun` directly. Normal agent turns therefore ignore `mcp.approvals.enabled: false` and `mcp.approvals.defaultTimeoutMs`; pass those values here or use a shared resolver so the documented config works in the primary runtime path.
  - confidence: 0.94
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:100-104`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the parser accepts `approve` without the slash. A malicious consent summary can still leave parser-shaped approval text intact in the transcript-defense layer, so the sanitizer/tests should cover the same command spellings the parser accepts or the parser should be narrowed.
  - confidence: 0.91
- **[P2] Align timeout validation with the clamp contract:** `src/config/zod-schema.ts:337-341`
  - body: The docs and runtime say `mcp.approvals.defaultTimeoutMs` is clamped to `[1000, 600000]`, but the strict schema rejects out-of-range values before the runtime clamp can run. Operators following the docs with a too-low or too-high value get config validation failure instead of the promised clamp; either relax the schema and clamp or document validation instead.
  - confidence: 0.88
- **[P3] Advertise only supported MCP consent decisions:** `CHANGELOG.md:9`
  - body: The changelog says users can reply with `allow-always`, but the MCP consent requester sends `allowedDecisions: ["allow-once", "deny"]` and the gateway rejects unavailable decisions. Update the release note so users do not try a durable approval mode this contract does not implement.
  - confidence: 0.92
