## Summary

Keep this PR open: the MCP consent-envelope direction is useful and has live proof, but current PR head still has approval-path blockers that should be fixed before merge.

## What This Changes

The PR adds an MCP consent envelope that routes opted-in bundle-MCP tool calls through plugin approvals, redacts action tokens from the model, updates approval reply UX, and adds config, docs, tests, and changelog text.

## Best Possible Solution

Land the MCP consent envelope after no-id approvals are scoped to the originating target, the MCP opt-out reaches every materialization path, parser-accepted approval text is neutralized, and public docs/copy match the supported decisions.

## Reproduction Assessment

Yes for the review findings: PR-head source shows the global implicit approval resolver, the missing opt-out propagation on the normal agent path, and the sanitizer/parser mismatch. I did not execute the code because this was a read-only review.

## Solution Assessment

No, not yet. Materializer-level interception and reuse of plugin approvals look like the right architecture, but the patch needs approval-target scoping, config propagation, and defense-in-depth cleanup before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope implicit approvals to the originating target:** `src/auto-reply/reply/commands-approve.ts:239-256`
  - body: The bare `/approve <decision>` path still lists every pending plugin and exec approval and binds to the only global candidate. If the sole pending approval belongs to another channel, account, thread, or session, an authorized sender elsewhere can approve it without seeing that prompt; filter candidates by the current approval target metadata or require an explicit id.
  - confidence: 0.94
- **[P2] Honor the MCP approval opt-out in agent runs:** `src/agents/pi-embedded-runner/run/attempt.ts:1241-1242`
  - body: The normal embedded attempt path calls `materializeBundleMcpToolsForRun` directly and only passes `agentId`/`sessionKey`. Because it never passes `params.config.mcp.approvals.enabled`, `mcp.approvals.enabled: false` still gates consent envelopes on the main agent run path despite the documented opt-out.
  - confidence: 0.93
- **[P2] Neutralize slashless approval commands:** `src/agents/pi-bundle-mcp-consent.ts:92-96`
  - body: `sanitiseToolEmittedApprovalText` only rewrites `/approve`, but the parser accepts slashless `approve`, and this PR makes `approve allow-once` a valid implicit approval form. A malicious envelope summary can still include a parser-shaped approval line in the defense-in-depth path this sanitizer is meant to close.
  - confidence: 0.9
- **[P3] Fix the MCP approval reply copy:** `docs/tools/mcp-consent-envelope.md:69-71`
  - body: The new docs say the prompt ends with `/approve <id> allow-once|deny`, and the changelog still advertises `allow-always`, but this PR renders bare `/approve <decision>` prompts and restricts MCP consent approvals to `allow-once`/`deny`. Please align the public copy with the implementation.
  - confidence: 0.84
