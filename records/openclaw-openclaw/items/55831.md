## Summary

Keep open. Current main still leaves the linked phantom-count bug reachable, but this PR should not merge as-is because its catch-path change clears last-known-good session data on any transient request failure; the linked bug remains open and this needs a targeted revision and rebase.

## What This Changes

The PR changes the Control UI sessions controller to assign an empty successful `sessions.list` response as `null`, and also clears `sessionsResult` at the start of the catch path.

## Best Possible Solution

Revise this PR narrowly on current main: assign `state.sessionsResult = res ?? null` on successful `sessions.list`, run checkpoint invalidation/refetch only when `res` exists, preserve last-known-good data for generic request errors, keep the missing-scope clear path, and add focused controller tests for empty success and transient-error behavior.

## Reproduction Assessment

Yes. Seed `state.sessionsResult` with a non-empty result, mock `client.request("sessions.list")` to resolve `undefined`, and call `loadSessions`; current main leaves the stale result in place. The introduced PR regression is similarly reproducible by seeding a result and making the request reject, which the PR would clear to `null`.

## Solution Assessment

No. The `res ?? null` success-path assignment is the right narrow fix, but clearing `sessionsResult` for every exception is broader than the reported empty-response bug and can regress transient reconnect or RPC failure UX.

## Review Findings
