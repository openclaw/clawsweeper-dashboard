## Summary

Keep open. Current main still has the reported stale-success bug, but this PR is not merge-ready because it also clears the cached sessions result on every generic request error, which regresses the current last-known-good behavior during transient reconnect or RPC failures.

## What This Changes

This PR changes `ui/src/ui/controllers/sessions.ts` so `loadSessions` assigns `sessionsResult = res ?? null` after `sessions.list` and clears `sessionsResult` before handling any request error.

## Best Possible Solution

Revise this PR on current main: assign `state.sessionsResult = res ?? null` after a successful `sessions.list`, keep checkpoint invalidation/refetch work inside `if (res)`, preserve generic-error last-known-good behavior, keep the missing-scope clear path, and add focused controller tests plus a changelog entry.

## Reproduction Assessment

Yes. Seed `state.sessionsResult` with a non-empty result, mock `client.request("sessions.list")` to resolve `undefined`, call `loadSessions`, and current main leaves the stale result because the assignment is guarded by `if (res)`. A second focused path can mock a generic rejection and verify the PR incorrectly clears the cached result.

## Solution Assessment

No. The success-path `res ?? null` assignment is the right narrow fix, but clearing `sessionsResult` for every exception is broader than the reported bug and can regress transient reconnect or RPC failure UX.
