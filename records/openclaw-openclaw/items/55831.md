## Summary

Keep open. Current main still has the reported stale-success cache bug, but this PR is not merge-ready because it also clears cached sessions on generic request errors, which regresses the current last-known-good behavior during transient RPC or reconnect failures.

## What This Changes

This PR changes the Control UI sessions controller to assign `sessionsResult = res ?? null` after `sessions.list` and to clear `sessionsResult` before handling any request error.

## Best Possible Solution

Revise this PR on current main so successful undefined `sessions.list` responses clear `sessionsResult`, checkpoint refresh work remains guarded by `if (res)`, generic errors preserve last-known-good data, and tests plus changelog are added.

## Reproduction Assessment

Yes. A high-confidence controller-level reproduction is to seed `state.sessionsResult` with a non-empty result, mock `client.request("sessions.list")` to resolve `undefined`, and call `loadSessions`; current main leaves the stale result because the assignment is guarded by `if (res)`.

## Solution Assessment

No. The success-path `res ?? null` assignment is the right narrow fix, but clearing `sessionsResult` for every exception is broader than the reported bug and can regress transient reconnect or RPC failure UX.
