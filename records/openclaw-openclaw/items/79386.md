## Summary

Keep this PR open. The reported cascade is source-reproducible and the branch targets the right runtime path, but the patch currently treats all resumed Claude CLI `timeout` failures as poisoned no-output watchdog failures, and the external PR still lacks real after-fix behavior proof.

## What This Changes

The branch adds Claude CLI resumed-session timeout recovery in `cli-runner` and `attempt-execution`, regression tests, and a changelog entry for the resume-watchdog cascade.

## Best Possible Solution

Keep the hook-aware cold retry design, but gate it on a no-output-watchdog-specific signal and mirror that predicate in the outer store cleanup before adding real redacted runtime proof.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced. Current main preserves the persisted binding after resumed `claude-cli` timeout paths because the cleanup gate is `session_expired`-only, while the linked issue describes the same session id hitting repeated no-output watchdog kills.

## Solution Assessment

No. The direction is right, but the implementation should distinguish no-output watchdog timeouts from overall timeout failures before retrying cold or clearing the persisted session binding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Limit cold retry to no-output resume timeouts:** `src/agents/cli-runner.ts:495`
  - body: The new predicate treats every `FailoverError` with `reason === "timeout"` as a poisoned resume. Current runner code uses that same reason for both no-output watchdog kills and overall turn timeouts, so a normal long-running resumed Claude CLI turn that exceeds the overall budget would be retried cold and have its session binding cleared, doubling the timeout window and potentially re-running side effects. Please gate this on the no-output watchdog signal, or a more specific timeout subtype, and use the same predicate in the outer cleanup path.
  - confidence: 0.88
