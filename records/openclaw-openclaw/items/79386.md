## Summary

Keep open: the bug is valid and this PR moves the retry to the right hook-aware layer, but one correctness gap remains and the external contributor still has not supplied real behavior proof.

## What This Changes

The PR adds Claude CLI poisoned-resume timeout recovery in `cli-runner.ts`, clears stale CLI session bindings in `attempt-execution.ts`, adds regression tests, and updates the changelog.

## Best Possible Solution

Clear the active Claude CLI binding once a resumed watchdog timeout is observed, even if the cold retry later fails for a different reason, then land after real after-fix proof is added.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main can raise `FailoverError reason=timeout` from the no-output watchdog while the persisted CLI binding cleanup/retry paths only handle `session_expired` or missing transcripts; I did not run a live poisoned-transcript setup.

## Solution Assessment

No. The hook-aware in-runner retry is the right direction, but the outer cleanup only catches final `timeout`/`session_expired`, so a cold retry that fails with a different FailoverError can leave the original poisoned resume id persisted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clear the poisoned binding after retry failures too:** `src/agents/command/attempt-execution.ts:548`
  - body: When the resumed Claude timeout triggers the in-runner cold retry, a cold retry that fails with `auth`, `billing`, `rate_limit`, or another non-timeout `FailoverError` reaches this outer catch without satisfying `isPoisonedResumeTimeout`. That skips `clearCliSessionInStore`, leaving the original poisoned session id in the store so the next turn can resume the same dead transcript again. Clear the active binding once the original resumed-timeout path is known, even if the cold retry's final error is different.
  - confidence: 0.84
