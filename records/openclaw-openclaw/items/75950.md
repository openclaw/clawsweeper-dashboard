## Summary

Keep open. Current main still hard-blocks Chrome realtime audio to macOS BlackHole on local and chrome-node paths, while the Google Meet docs describe BlackHole/SoX and alternatives but not a Linux PulseAudio/PipeWire setup or explicit Linux unsupported path.

## What This Changes

Support Linux local Chrome realtime Google Meet joins or document a recommended Linux transport/setup path instead of surfacing only a macOS BlackHole blocker.

## Best Possible Solution

Make the current Linux support boundary explicit in setup diagnostics and docs, while tracking native PulseAudio/PipeWire support separately if maintainers want that backend.

## Reproduction Assessment

Yes. The issue gives the exact setup command and current main maps it to `setupStatus`, where non-darwin Chrome realtime mode throws the same macOS-only BlackHole error and records it as `chrome-local-audio-device`.

## Solution Assessment

Unclear because no patch is proposed. The narrow maintainable first step is Linux-specific guidance in setup/docs; implementing a full PulseAudio/PipeWire duplex backend should be scoped separately.
