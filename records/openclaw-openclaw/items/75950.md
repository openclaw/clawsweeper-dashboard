## Summary

Keep open. Current main still hard-blocks local Chrome realtime audio to macOS BlackHole, and the Google Meet docs do not provide a Linux PulseAudio/PipeWire setup path; the existing transcribe, chrome-node, and Twilio alternatives make a narrow docs/diagnostic repair feasible.

## What This Changes

Requests Linux support or clearer Linux setup guidance for Google Meet local Chrome realtime joins blocked by the macOS BlackHole audio backend.

## Best Possible Solution

Add Linux-specific setup diagnostics and docs that state local Chrome realtime is unsupported today and point users to transcribe mode, Twilio, or a macOS chrome-node, while tracking native Linux audio backend support separately if maintainers want it.

## Reproduction Assessment

Yes. The reported `openclaw googlemeet setup --json` path maps directly to `setupStatus`, which calls the non-darwin BlackHole guard and returns the same failing `chrome-local-audio-device` check for local Chrome realtime mode.

## Solution Assessment

Unclear because no patch is proposed. The safest first repair is explicit Linux guidance in setup/docs; implementing PulseAudio/PipeWire duplex audio should be a separate scoped change.
