## Summary

This is a focused bug-fix PR with good live terminal proof now supplied, but the diff still has one blocker: it compares the active runtime snapshot against the post-auto-enable config, so implicit auto-enabled channel configs can still fall back to source-shaped SecretRefs.

## What This Changes

The PR adds a config-only active secrets runtime accessor, uses it for gateway `message.action` dispatch, and adds send-handler regression tests for resolved config selection, stale snapshot fallback, and send-path isolation.

## Best Possible Solution

Keep the narrow gateway-only approach, but select the active runtime snapshot against the pre-auto-enable request config or cover the auto-enabled source shape so SecretRef-backed message actions work for both explicit and implicit channel enablement.

## Reproduction Assessment

Yes, with source/proof caveats: the PR body gives a live before/after Discord `channel-info` path, and source inspection shows current gateway dispatch can pass source-shaped config into action handling. I did not run the live Discord reproduction in this read-only review.

## Solution Assessment

No, not yet: the gateway-only, config-only accessor is the right boundary, but the implementation should handle configs changed by plugin auto-enable before comparing with the active runtime source snapshot.
