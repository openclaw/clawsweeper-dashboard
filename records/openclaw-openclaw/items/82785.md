## Summary

Keep open: current main still drops the checked auth source from the missing-key error, and an open closing PR already targets the fix. The shell check also found duplicated agent-run missing-key messages that should be considered before declaring the user-visible run path fully fixed.

## What This Changes

The issue asks for `requireApiKey` missing-key errors to include `ResolvedProviderAuth.source` so users know which env var, profile, or config source was checked.

## Best Possible Solution

Use one shared missing-auth formatter that includes `auth.source` for `requireApiKey` callers and any direct agent runtime paths that already hold `ResolvedProviderAuth`.

## Reproduction Assessment

Yes. Source inspection shows a `ResolvedProviderAuth` with `source: "env: OPENAI_API_KEY"` reaches `requireApiKey`, and the current throw omits that source; I did not run tests because this review is read-only.

## Solution Assessment

Unclear as a complete fix until the paired PR is reviewed. Adding `auth.source` to `requireApiKey` is the narrow helper fix, but the agent-run path should also avoid duplicating the old source-less message.
