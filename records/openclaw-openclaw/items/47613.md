## Summary

Keep open. Current main still has the no-callback Gateway reasoning-stream gap, but this PR is not merge-ready against current main because it decouples the callback by making `streamReasoning` depend only on `reasoningMode`, which would lose the current thinking-visibility gate on rebase; overlapping open PRs also need one maintainer-chosen canonical path.

## What This Changes

The PR changes `src/agents/pi-embedded-subscribe.ts` so `streamReasoning` no longer depends on `onReasoningStream`, `emitReasoningStream()` can broadcast `stream: "thinking"` events without that callback, and the callback is invoked only when present.

## Best Possible Solution

Land one canonical implementation that decouples the Gateway `emitAgentEvent` broadcast from channel-specific callbacks while preserving `reasoningMode === "stream"`, `thinkingLevel !== "off"`, and silent-turn gates, then add no-callback Gateway/WebSocket regression coverage plus the required docs and changelog updates.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Gateway `chat.send` run for a session with `reasoningLevel: "stream"` and no `onReasoningStream` callback: current main stores and forwards the reasoning level, but `streamReasoning` and `emitReasoningStream()` still require the missing callback before broadcasting.

## Solution Assessment

No. The PR targets the right coupling bug, but the maintainable fix should remove only the callback dependency, preserve current visibility and silent-run gates, and coordinate with the overlapping Gateway reasoning-stream PRs before merge.
