## Summary

Keep open. The no-callback Gateway reasoning-stream gap is real on current main, but this PR is stale against the current reasoning visibility gates and overlaps other open implementations, so it needs maintainer choice and a rebase/revision rather than cleanup closure.

## What This Changes

The PR changes `src/agents/pi-embedded-subscribe.ts` so reasoning stream broadcasts no longer require `onReasoningStream`, while keeping the channel callback optional when present.

## Best Possible Solution

Land one canonical Gateway reasoning-stream implementation that decouples the WebSocket/node broadcast from channel-specific callbacks, preserves explicit stream-mode, thinking-enabled, and silent-turn gates, adds no-callback Gateway regression coverage, and updates `docs/tools/thinking.md` if Gateway live reasoning is intentionally supported.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Gateway `chat.send` run for a session with `reasoningLevel: "stream"`: current main can carry the session reasoning value into the embedded runner, but the Gateway path supplies no `onReasoningStream`, so `emitReasoningStream()` returns before broadcasting `stream: "thinking"`.

## Solution Assessment

No. The direction is useful, but this exact PR is not the best current-main fix because it does not preserve the current thinking visibility gate; the safer repair is to remove only the callback dependency and cover the no-callback Gateway path.
