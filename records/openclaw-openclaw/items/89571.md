## Summary

Keep open: this focused maintainer draft fixes a real provider-tool schema hook robustness gap and I found no blocking code defect, but it changes a public Plugin SDK helper and needs maintainer acceptance before merge.

## What This Changes

The PR centralizes guarded provider tool-field reads in `src/plugin-sdk/provider-tools.ts` and adds regressions for unreadable `parameters` getters and immutable parameter descriptors.

## Best Possible Solution

Land the shared guarded provider-tool helper after maintainer review accepts the public SDK behavior and the draft is ready, keeping provider families on one SDK seam.

## Reproduction Assessment

Yes at source level: current main's provider-family helpers directly read `tool.parameters` and `tool.name`, so a throwing getter can abort direct hook execution or diagnostics. I did not run tests in this read-only review; the PR adds focused regression coverage for the failing shape.

## Solution Assessment

Yes: centralizing guarded field reads in `src/plugin-sdk/provider-tools.ts` is the narrowest maintainable fix for the shared Gemini/OpenAI/DeepSeek SDK seam. Per-provider try/catch would duplicate the same contract and drift over time.
