## What This Changes

The PR refactors `src/plugin-sdk/provider-tools.ts` and its tests so Gemini, OpenAI, and DeepSeek provider tool schema hooks guard unreadable tool fields and preserve descriptor-heavy tools when normalizing parameters.

## Best Possible Solution

Land the shared guarded provider-tool helper after maintainer review accepts the draft and validation, keeping bundled and third-party providers on the existing family-level SDK seam.

## Reproduction Assessment

Yes at source level: current main's public provider-tools helpers directly access `tool.parameters` and `tool.name` in provider hook maps, so a throwing getter can abort direct hook execution. I did not run tests in this read-only review; the PR adds a focused regression for the DeepSeek family path.

## Solution Assessment

Yes: centralizing the guarded reads in `src/plugin-sdk/provider-tools.ts` matches the family-level SDK seam used by Gemini, OpenAI, and DeepSeek and avoids per-provider duplicate try/catch logic. The remaining question is landing readiness for this draft maintainer PR, not a different fix location.
