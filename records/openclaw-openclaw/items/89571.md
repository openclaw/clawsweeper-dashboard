## What This Changes

The branch refactors provider tool-schema normalization and diagnostics to use guarded tool entry/name/parameters reads, with regressions for unreadable parameters and immutable descriptors.

## Best Possible Solution

Keep the centralized guarded provider-tool helper and land it after draft validation finishes, rather than adding per-provider defensive copies.

## Reproduction Assessment

Yes at source level: current main directly reads `tool.parameters` inside provider normalization/inspection maps, so a throwing getter would abort that hook; the PR adds a targeted regression for that path. I did not run tests because this was a read-only review.

## Solution Assessment

Yes: the PR fixes the issue at the shared provider-tool family helper, which is the narrow owner boundary used by OpenAI, DeepSeek, and Gemini provider plugins. Per-provider try/catch blocks would duplicate behavior and drift from the existing runtime projection guard.
