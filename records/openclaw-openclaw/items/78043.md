## Summary

Keep open: current main still has a source-level path that carries the Linux node daemon gateway token into inline systemd unit rendering, and the linked closing PR remains open, conflicting, and proof-gated.

## What This Changes

Prevent Linux node daemon installs from writing `OPENCLAW_GATEWAY_TOKEN` into inline systemd `Environment=` entries.

## Best Possible Solution

Land the linked PR or an equivalent narrow security fix that carries env source metadata through node install, writes the gateway token only to the owner-only systemd env file, and sanitizes stale inline unit backups.

## Reproduction Assessment

Yes for a source-level path: current main copies `OPENCLAW_GATEWAY_TOKEN` into the node service environment, does not carry source metadata through the node install plan, and renders remaining systemd env inline. I did not run a live Linux user-systemd install in this read-only review.

## Solution Assessment

Yes: reusing the existing `environmentValueSources` contract and 0600 systemd service env-file path is the narrow maintainable fix. The linked PR targets that direction, but this issue should stay open until the fix merges with real Linux behavior proof.
