## Summary

Keep open: current main and the latest release still hard-gate cache-ttl pruning and cache-ttl marker writes through eligibility that rejects OpenAI, so the reported OpenAI idle-gap pruning path remains disabled.

## What This Changes

The issue asks OpenClaw to make `contextPruning.mode: "cache-ttl"` prune stale tool results for OpenAI models instead of being skipped by provider eligibility.

## Best Possible Solution

Keep this issue open and land a narrow fix that decouples pruning/marker eligibility from Anthropic-style cache-control eligibility, allows OpenAI automatic-cache models to use the existing `cache-ttl` pruning mode, and proves OpenAI payloads stay free of Anthropic cache markers.

## Reproduction Assessment

Yes for the core bug: source inspection shows OpenAI remains ineligible, the pruning extension is skipped, and marker writes are gated off. The live cost-recovery numbers still need fresh current-main telemetry if maintainers want to quantify impact before landing.

## Solution Assessment

Yes at the direction level: use the existing `contextPruning.mode: "cache-ttl"` contract and separate pruning eligibility from provider cache-control semantics. Adding a new config knob should not be necessary unless maintainers decide OpenAI pruning must remain opt-in beyond the existing mode.
