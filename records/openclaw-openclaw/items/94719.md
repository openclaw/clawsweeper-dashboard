## Label Justifications

- `P1`: The PR targets a current Anthropic OAuth failure that blocks Claude subscription-backed gateway and agent runs.
- `merge-risk: 🚨 compatibility`: The diff can break existing setups where the `claude` CLI is installed externally but its package metadata is not resolvable from OpenClaw's runtime module graph.
- `merge-risk: 🚨 auth-provider`: The changed code constructs Anthropic OAuth bearer identity headers, so resolver failure changes whether those authenticated requests can run.
- `merge-risk: 🚨 availability`: The exported module-load constant can throw during provider or transport import, potentially preventing Anthropic runtime paths from starting.

## Summary

Keep open: this targets a valid Anthropic OAuth bug, but the patch is not merge-ready because it can make Anthropic provider imports fail when Claude Code is installed only as the supported external CLI, includes a resolver test that fails its own validation contract, and still lacks patched OpenClaw runtime proof.

## What This Changes

The PR adds a shared Claude Code version resolver, imports its module-load constant into both Anthropic OAuth header builders, and adds resolver tests.

## Best Possible Solution

Use one version source that respects the supported Claude CLI/backend packaging boundary, avoid import-time provider failure, cover both OAuth builders with focused tests, and add redacted OpenClaw runtime or network proof.

Yes for the source-level stale-header path: current main builds `claude-cli/2.1.75` in both Anthropic OAuth builders. I did not run a live Anthropic OAuth rejection in this read-only review.

## Solution Assessment

No. Sharing one version source across both headers is the right direction, but this implementation uses an unproven import-time module-graph lookup for an external CLI package and includes a failing test fixture.
