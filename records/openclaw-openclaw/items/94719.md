## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Defer Claude package resolution out of module import:** `src/llm/utils/claude-code-version.ts:101`
  - body: `CLAUDE_CODE_VERSION` is evaluated while this helper module is imported, and the resolver now throws if `@anthropic-ai/claude-code/package.json` is not resolvable. The root package does not depend on that CLI package, so importing either Anthropic runtime can fail before even API-key or non-OAuth requests are built. Resolve inside the OAuth header path or add an explicit compatibility/startup contract instead.
  - confidence: 0.94
- **[P2] Use the live user-agent helper in the OAuth headers:** `src/llm/providers/anthropic.ts:1018`
  - body: The new `claudeCodeUserAgent()` helper is the only code path that re-reads the active resolver, but both production OAuth headers still interpolate the frozen `CLAUDE_CODE_VERSION` import. That means an upgraded Claude Code package can still leave a long-running OpenClaw process sending a stale header until restart, and the helper's per-call contract is not exercised by the real request path.
  - confidence: 0.9
- **[P2] Make the digit-leading fixture match the validator:** `src/llm/utils/claude-code-version.test.ts:47-48`
  - body: This test says it accepts a digit-leading string, but the injected value starts with `v`. The implementation rejects anything that does not match `/^\d/`, so this assertion will fail once the module imports successfully; use a value like `2.0.0-beta` or change the validation contract.
  - confidence: 0.96
