## Summary

Keep open: current main still intentionally rejects symlinked approval cwd paths and exposes no allowSymlinkCwd-style option; the requested escape hatch is a new security-sensitive config and UX policy, not an already-implemented cleanup.

## What This Changes

The issue requests a default-off approval setting that would allow trusted symlink cwd paths for approval-bound system.run while preserving canonical-path display and inode revalidation.

## Best Possible Solution

Maintainers should decide the threat-model policy first; if accepted, implement a narrow default-off opt-in that shows both requested and canonical cwd, binds the canonical target identity, keeps drift revalidation fail-closed, and updates schema, docs, CLI/UI surfaces, changelog, and regression tests.

## Reproduction Assessment

Yes for the current behavior: source and regression tests on current main show approval-backed system.run rejects direct symlink cwd paths and mutable symlink path components. This is not a broken current contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security review. The proposed design is plausible, but adding allowSymlinkCwd-style config is a policy change; the safer fallback is improved denial guidance and docs if the opt-in is rejected.
