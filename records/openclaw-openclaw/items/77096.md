## Summary

Keep open: current main and `v2026.6.10` still intentionally deny approval-bound `system.run` when the cwd or a mutable cwd path component is a symlink, and the remaining work is a maintainer security/product decision around a new default-off exec trust knob.

## What This Changes

The issue requests a default-off opt-in setting that lets approval-bound `system.run` use symlinked working directories while preserving canonical-path approval safety.

## Best Possible Solution

Maintainers should either approve a narrow, documented trust-knob contract that shows both requested and canonical cwd and preserves inode revalidation, or explicitly keep/document the canonical-cwd-only policy.

## Reproduction Assessment

Yes from source: current main still rejects approval-bound `system.run` when the cwd itself or a mutable path component is a symlink. I did not run a live filesystem repro in this read-only review.
