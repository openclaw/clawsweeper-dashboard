## Summary

Keep open: current main still intentionally rejects symlinked approval cwd paths, and the requested escape hatch is a new security-sensitive config and UX policy rather than an already-implemented cleanup item.

## What This Changes

Issue requests a default-off opt-in that allows approval-bound `system.run` to proceed from trusted symlink cwd paths while preserving canonical-path approval binding.

## Best Possible Solution

Maintainers should decide the threat-model policy first; if accepted, implement a narrow default-off opt-in that displays both requested and canonical cwd, pins the canonical target identity, keeps revalidation fail-closed, and updates schema, docs, CLI/UI surfaces, changelog, and regression tests.

## Reproduction Assessment

Yes for the current behavior by source and regression-test inspection: current main rejects approval-backed `system.run` from direct cwd symlinks and mutable symlink path components. This is not a failing existing contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security review. The proposed design is plausible, but adding `allowSymlinkCwd`-style config is a policy change; the safer fallback is to improve denial guidance while the threat-model decision is made.
