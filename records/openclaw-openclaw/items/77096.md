## Summary

Keep open: current main and the latest release still intentionally reject approval-backed symlink cwd paths, and the requested escape hatch is a new security-sensitive config/product decision. An open implementation PR now exists, but it is not merged and already has review/security/proof blockers, so this issue should remain as the tracking discussion.

## What This Changes

Request a default-off exec approval option that allows trusted symlink cwd paths while binding approval to the canonical cwd identity.

## Best Possible Solution

If maintainers accept the threat model, land a narrow default-off opt-in that displays requested and canonical cwd, records the symlink context in the approval plan, preserves canonical identity revalidation, and updates schema, docs, and regression tests without unrelated dependency changes.

## Reproduction Assessment

Yes for the current behavior: source and regression tests on current main show approval-backed system.run rejects direct symlink cwd paths and mutable symlink path components. This is not a failing existing contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security/product review. The direction is plausible, but the current open PR is not the best form as submitted because it lacks the requested approval UX binding and includes unrelated dependency-policy changes.
