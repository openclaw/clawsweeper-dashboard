## Summary

Keep open: current main still intentionally rejects symlinked approval cwd paths, and the requested escape hatch is a new security-sensitive config and UX policy rather than an already-implemented cleanup item.

## What This Changes

The issue requests a default-off opt-in that allows approval-bound `system.run` from symlinked cwd paths while showing the canonical target and preserving inode-bound revalidation.

## Best Possible Solution

Maintainers should decide the threat-model policy first; if accepted, implement a narrow default-off opt-in that displays both requested and canonical cwd, pins the canonical target identity, keeps existing revalidation fail-closed, and updates schema, docs, CLI/UI surfaces, and regression tests.

## Reproduction Assessment

Yes for the current behavior: source and regression tests on current main show approval-backed `system.run` rejects symlink cwd paths and mutable symlink path components. This is not a failing existing contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security review. The proposed design is plausible, but adding `allowSymlinkCwd`-style config is a policy change; the safer fallback is to improve denial guidance while the threat-model decision is made.
