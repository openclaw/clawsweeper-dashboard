## Summary

Keep open: current `main` still intentionally rejects approval-bound `system.run` from symlinked cwd paths, and the requested escape hatch is a new security-sensitive config/policy change rather than an already-implemented cleanup item.

## What This Changes

The issue requests a default-off opt-in for approval-bound `system.run` to permit explicitly trusted symlink cwd paths while preserving canonical identity binding.

## Best Possible Solution

Maintainers should first decide the threat model; if accepted, implement a narrow default-off opt-in that shows both requested and canonical cwd, pins the canonical target identity, keeps revalidation fail-closed, and updates schema, docs, and tests.

## Reproduction Assessment

Yes for the current behavior: source and regression tests on current `main` show approval-backed `system.run` rejects direct cwd symlinks and mutable symlink path components. This is not a failing existing contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security review. The proposed design is plausible, but an `allowSymlinkCwd`-style setting changes the approval threat model, and improving denial guidance is the safer fallback if maintainers decline the policy change.
