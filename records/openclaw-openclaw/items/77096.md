## Summary

Keep open: current main still intentionally rejects symlinked approval cwd paths, and the requested escape hatch is a new security-sensitive config and approval UX policy rather than an already-implemented cleanup item.

## What This Changes

Request a default-off exec approvals setting that lets approval-bound `system.run` use symlinked cwd paths while displaying and pinning the canonical target.

## Best Possible Solution

Maintainers should decide the threat-model policy first; if accepted, implement a narrow default-off opt-in with canonical-target display, inode identity pinning, fail-closed revalidation, schema/docs/UI coverage, and regression tests.

## Reproduction Assessment

Yes for the current behavior: source and regression tests on current main show approval-backed `system.run` rejects symlink cwd paths and mutable symlink path components. This is not a failing existing contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security review. The proposed default-off design is plausible, but adding an `allowSymlinkCwd`-style config is a policy change; improving denial guidance is the safer fallback until the threat model is accepted.
