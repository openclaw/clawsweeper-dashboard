## Summary

Keep open: current main and the latest release still deny approval-bound system.run when the cwd or a mutable cwd path component is a symlink, and the requested opt-in would relax a security boundary plus add new config/protocol semantics that need maintainer product/security review.

## What This Changes

Request an opt-in setting that permits approval-bound system.run from trusted symlink cwd paths while preserving canonical-path approval display and revalidation.

## Best Possible Solution

Keep the secure default, then either improve the denial UX/docs or add a maintainer-approved opt-in that displays both requested and canonical cwd, pins the realpath identity, and covers symlink rebinds with focused tests.

## Reproduction Assessment

Yes, at source level. Current main rejects symlink cwd and mutable symlink cwd components in resolveCanonicalApprovalCwdSync, and current tests assert approval-based execution is denied for those paths; I did not run tests because this review is read-only.

## Solution Assessment

Unclear. The proposed opt-in is plausible, but the exact config shape, approval display contract, protocol payload, and security tradeoff need maintainer decision; improved error messaging and docs may be the safer first step.
