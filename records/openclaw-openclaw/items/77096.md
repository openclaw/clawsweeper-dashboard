## Summary

Keep open: current main still intentionally rejects symlinked approval cwd paths, and the requested escape hatch is a new security-sensitive config and UX policy rather than an already-implemented cleanup item.

## What This Changes

Request to add a default-off exec approvals setting that permits explicitly trusted symlink cwd paths while showing and pinning the canonical target, or at least improve denial guidance.

## Best Possible Solution

Maintainers should decide the threat-model policy first; if accepted, implement a narrow default-off opt-in that displays requested and canonical cwd, pins the realpath identity, keeps fail-closed revalidation, and updates schema, docs, UI/CLI surfaces, and regression tests.

## Reproduction Assessment

Yes for the current behavior: source and regression tests on current main show approval-backed `system.run` rejects symlink cwd paths and mutable symlink path components. This is not a failing existing contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security review. The proposed design is plausible, but adding an `allowSymlinkCwd`-style config is a policy change; the safer fallback is improved denial guidance while the threat-model decision is made.
