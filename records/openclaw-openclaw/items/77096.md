## Summary

Keep open: current main still intentionally rejects symlinked approval cwd paths, and there is no existing `allowSymlinkCwd`-style config. The request is a new security-sensitive opt-in policy, not an already-implemented cleanup item.

## What This Changes

Request a default-off exec-approvals option that permits explicitly trusted symlink cwd paths for approval-bound `system.run` while preserving canonical target display and inode revalidation.

## Best Possible Solution

Maintainers should first approve the threat-model policy; if accepted, implement a narrow default-off opt-in with schema/docs/CLI or UI exposure, canonical requested-target prompt text, preserved fail-closed revalidation, and regression tests.

## Reproduction Assessment

Yes for the current behavior: source and regression tests on current main show approval-backed `system.run` rejects direct and parent symlink cwd paths. This demonstrates the reported limitation, but it is not a failing existing contract.

## Solution Assessment

Unclear pending maintainer security review. The proposed default-off opt-in is plausible only if approval UX displays both requested and canonical cwd and revalidation remains bound to canonical file identity; otherwise the safer near-term path is better denial guidance.
