## Summary

Keep open: current main still deliberately rejects approval-bound `system.run` cwd symlinks, and the requested escape hatch is a new security-sensitive config/UX policy rather than an already-implemented cleanup item.

## What This Changes

The issue requests a default-off exec approvals option that allows approval-bound `system.run` to use symlinked cwd paths when the user explicitly trusts the requested and canonical paths.

## Best Possible Solution

Maintainers should first decide the threat-model policy; if accepted, implement a narrow default-off opt-in that shows requested and canonical cwd, pins the canonical target identity, keeps revalidation fail-closed, and updates schema, docs, UI/CLI surfaces, and regression tests.

## Reproduction Assessment

Yes for the current behavior: current source and regression tests show approval-backed `system.run` rejects symlink cwd paths and mutable symlink path components. This is not a failing existing contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security review. The proposed design is plausible, but adding `allowSymlinkCwd`-style configuration changes the approval trust model; the safer interim path is clearer denial guidance and docs.
