## Summary

Keep open: current main still intentionally rejects approval-bound node `system.run` cwd symlinks, and the request is a new security-sensitive opt-in policy rather than an already-implemented cleanup item.

## What This Changes

Allow users to opt into approval-bound `system.run` execution from symlinked cwd paths while preserving canonical-path approval binding.

## Best Possible Solution

Maintainers should first approve the threat-model policy; if accepted, add a default-off opt-in that displays requested and canonical cwd, binds approval to the canonical target identity, keeps revalidation fail-closed, and updates schema, docs, CLI/UI surfaces, and regression tests.

## Reproduction Assessment

Yes for the current behavior: source and regression tests on current main show approval-backed node `system.run` rejects direct symlink cwd paths and mutable symlink path components. This is not a failing existing contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security review. The proposed design is plausible, but adding an `allowSymlinkCwd`-style config is a policy change; the safer fallback is to improve denial guidance while the threat-model decision is made.
