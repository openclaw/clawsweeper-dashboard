## Summary

Keep open: current main still intentionally denies symlinked approval cwd paths, and the requested escape hatch is a new security-sensitive config and approval UX policy rather than conservative cleanup.

## What This Changes

The issue requests a default-off exec approval setting to allow explicitly trusted symlink cwd paths while preserving canonical cwd approval binding.

## Best Possible Solution

Maintainers should decide the threat-model policy first; if accepted, implement a narrow default-off opt-in that displays requested and canonical cwd, binds the canonical target identity, keeps revalidation fail-closed, and updates schema, UI/CLI, docs, and regression tests.

## Reproduction Assessment

Yes for the current behavior: current source rejects direct symlink cwd and mutable symlink path components, and regression tests construct both cases. This is not a failing existing contract; it is a request to change the security policy.

## Solution Assessment

Unclear pending maintainer security review. The proposed realpath display plus identity pinning design is plausible, but adding an `allowSymlinkCwd`-style option is a policy decision; the safer fallback is improved denial guidance unless maintainers accept the opt-in.
