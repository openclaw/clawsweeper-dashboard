## Summary

Keep open: current main and v2026.5.12 still reject approval-backed symlink cwd/path-component execution, and the requested escape hatch is a new security-sensitive config/product decision. The related implementation PR remains open with review and proof blockers, so this issue should stay as the tracker.

## What This Changes

The issue requests a default-off opt-in that lets approval-bound system.run run from trusted symlink cwd/path layouts while preserving canonical cwd approval and inode revalidation.

## Best Possible Solution

If maintainers accept the threat model, land a narrow default-off opt-in that displays requested and canonical cwd, preserves canonical identity revalidation, updates schema/docs/tests, and excludes unrelated dependency-policy changes.

## Reproduction Assessment

Yes for the current behavior: source and regression tests on current main show approval-backed system.run denies direct symlink cwd paths and mutable symlink path components. This is not a failing existing contract; it is a request for a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security/product review. The direction is plausible, but the linked PR is not the best form as submitted because it lacks the requested approval UX binding and has unrelated dependency-policy churn.
