## Summary

Keep open: current main and v2026.6.6 still intentionally deny approval-bound system.run when the cwd or a mutable cwd path component is a symlink, and relaxing that guard needs maintainer product/security approval rather than cleanup closure.

## What This Changes

The issue requests a default-off opt-in that lets trusted users approve system.run from symlinked cwd paths while preserving canonical target display and inode revalidation.

## Best Possible Solution

Keep the secure default, then either improve denial UX/docs or land a maintainer-approved opt-in that displays requested and canonical cwd, pins realpath identity, and covers symlink rebinds with focused tests.

## Reproduction Assessment

Yes, at source level. Current main rejects both symlink cwd and mutable symlink cwd components in resolveCanonicalApprovalCwdSync, and current tests assert those approval-backed runs are denied; I did not run tests because this review is read-only.

## Solution Assessment

Unclear. The requested opt-in is plausible and now has an open implementation candidate at https://github.com/openclaw/openclaw/pull/80532, but the exact config shape, approval prompt contract, and security tradeoff need maintainer approval before merge.
