## Summary

Keep open: current main and v2026.6.6 still reject approval-bound system.run from symlinked cwd paths, and no supported opt-in exists; the related open PR needs separate maintainer security/config review before this issue can close.

## What This Changes

The issue requests an opt-in way to allow approval-bound system.run from symlinked working directories while preserving approval safety.

## Best Possible Solution

Make a maintainer decision on the safe opt-in shape, then land a focused change that preserves the current default, shows canonical and requested cwd in approval UX, carries the symlink policy through revalidation, and updates focused tests/docs.

## Reproduction Assessment

Yes. Source inspection and the existing POSIX regression test show that approved system.run with a symlinked cwd or symlinked parent path is denied on current main; I did not run the test because this review was read-only.

## Solution Assessment

Unclear. The requested opt-in is plausible and aligned with trusted high-power workflows, but the permanent config surface and approval UX need maintainer security/product judgment; the minimum safer fallback would improve the denial message and docs without relaxing the guard.
