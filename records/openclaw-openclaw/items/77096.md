## Summary

Keep open: current main still intentionally rejects approval-backed `system.run` cwd symlinks, and the requested escape hatch is a new security-sensitive config/UX policy rather than an already-implemented cleanup item.

## What This Changes

The issue requests default-off `execApprovals` settings that allow trusted symlink cwd paths while showing and binding the canonical target during approval.

## Best Possible Solution

Maintainers should decide the security policy first; if accepted, implement a narrow default-off opt-in that displays both paths, pins the canonical target identity, keeps fail-closed revalidation, and updates schema, docs, UI/CLI surfaces, and regression tests.

## Reproduction Assessment

Yes for the current intentional behavior: source and regression tests on current main show approval-backed `system.run` rejects symlink cwd paths and mutable symlink path components. This is not a failing existing contract; it is a request to add a guarded opt-in.

## Solution Assessment

Unclear pending maintainer security review. The proposed design is plausible, but adding `allowSymlinkCwd`-style config is a policy change; the safer fallback is to improve denial guidance while the threat-model decision is made.
