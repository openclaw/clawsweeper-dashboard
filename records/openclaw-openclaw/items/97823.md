## Summary

This PR should stay open for maintainer review: the parser fix is focused and appears correct, but it intentionally changes undefined malformed Telegram target parsing and scoped Telegram guidance leaves live topic-delivery proof as an optional maintainer risk reducer.

## What This Changes

The PR adds a shared oversplit-colon guard to Telegram target parsing and regression tests for malformed multi-colon targets while preserving supported numeric, username, t.me, and topic suffix formats.

## Best Possible Solution

Merge the parser-level guard after maintainers accept the malformed-target compatibility tradeoff, optionally adding live Telegram topic proof if they want transport-level confidence.

## Reproduction Assessment

Yes by source inspection. Current main's greedy suffix parsers can split malformed multi-colon targets into chatId residue plus messageThreadId; I did not run tests because this review is read-only.

## Solution Assessment

Yes. parseTelegramTarget is the right owner boundary because send, outbound adapters, action threading, normalization, and writeback paths consume its chatId/messageThreadId result, while later send-path guards would duplicate the invariant and miss other callers.
