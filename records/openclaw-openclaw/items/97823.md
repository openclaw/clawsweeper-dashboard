## Summary

Keep open: the PR fixes the previously flagged sibling parser gap and now has terminal proof, but the new no-colon regex is too broad and can drop current URL-form Telegram lookup targets with thread suffixes.

## What This Changes

The PR changes both Telegram target suffix regexes from greedy chat segments to no-colon chat segments and adds regression tests for malformed multi-colon targets plus valid suffix forms.

## Best Possible Solution

Reject malformed extra-colon targets while preserving supported numeric, username, and t.me lookup targets with thread suffixes in `parseTelegramTarget`, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main greedily splits `https://t.me/mychannel:topic:9` into a lookup target plus thread id, while the PR's `[^:]+` branch would fall through as one unresolvable chat id.

## Solution Assessment

No. Narrowing both parsers is the right layer, but the best fix must reject malformed extra-colon strings without dropping current URL-form Telegram lookup targets with thread suffixes.
