## Summary

No blocking patch findings remain after the v4 update; the PR now tightens both Telegram suffix parsers and covers the malformed t.me-looking edge from the prior review. ## What This Changes

The PR adds an oversplit-colon guard to Telegram target parsing and regression tests for malformed multi-colon targets while preserving documented numeric, username, t.me, and topic suffix formats.

## Best Possible Solution

Keep the parser fix in parseTelegramTarget, preserve documented target formats, reject malformed extra-colon suffixes before they set chatId/messageThreadId, and let maintainers decide whether to require live Telegram proof before merge.

## Reproduction Assessment

Yes. Source inspection of current main shows the greedy suffix parsers can split malformed multi-colon targets into chatId residue plus messageThreadId; I did not run tests because this review is read-only.

## Solution Assessment

Yes. parseTelegramTarget is the right owner boundary, and the v4 helper mirrors the existing t.me lookup shape without changing documented target formats; direct normalization would risk rewriting the raw delivery target instead of only detecting oversplit inputs.
