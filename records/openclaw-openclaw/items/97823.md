## Summary

Keep open for normal maintainer review: current main can mis-split malformed multi-colon Telegram targets, and this PR's shared parser guard appears correct, but the compatibility tradeoff and optional live Telegram topic proof remain merge decisions.

## What This Changes

The PR adds an oversplit-colon guard to Telegram target parsing and regression tests for malformed multi-colon targets while preserving documented numeric, username, t.me, and topic suffix formats.

## Best Possible Solution

Keep the guard in parseTelegramTarget, preserve documented target forms, reject malformed extra-colon suffixes before they set chatId/messageThreadId, and let maintainers decide whether live Telegram proof is required before merge.

## Reproduction Assessment

Yes by source inspection. Current main's greedy suffix parsers can split malformed multi-colon targets into chatId residue plus messageThreadId; I did not run tests because this review is read-only.

## Solution Assessment

Yes. parseTelegramTarget is the right owner boundary because send, poll, sticker, and action threading paths consume its chatId/messageThreadId result; pushing the guard later into send resolution would duplicate the invariant and still leave other callers exposed.
