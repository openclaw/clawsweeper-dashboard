## Summary

Keep open: the patch fixes the legacy `chatId:threadId` regex, but the sibling `chatId:topic:threadId` parser still accepts colon-bearing chat IDs and can silently corrupt the parsed target. The PR also needs stronger contributor proof/context before merge.

## What This Changes

The PR narrows Telegram's legacy `chatId:threadId` parser and adds parser regression tests for malformed multi-colon targets.

## Best Possible Solution

Apply one Telegram target invariant to both suffix parsers: supported `:topic:<id>` and legacy `:<id>` forms should only split valid non-colon chat IDs, with regression tests and updated contributor proof/context.

## Reproduction Assessment

Yes. Source inspection and a read-only regex probe show current main misparses colon-bearing targets in both suffix parsers, and the proposed patch still leaves `a:b:topic:42` parsed as chatId `a:b` with thread `42`.

## Solution Assessment

No. Narrowing the legacy colon regex is a correct part of the fix, but the best fix must update the sibling `:topic:` parser and tests so Telegram target parsing has one consistent no-colon chatId rule.
