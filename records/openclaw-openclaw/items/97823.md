## Summary

The v2 PR fixes the earlier `https://t.me/...:<id>` compatibility problem, but it still breaks the current explicit `t.me/...:topic:<id>` URL-form path by letting the legacy colon parser return a chat id ending in `:topic`. That is a concrete compatibility and message-delivery blocker before merge.

## What This Changes

The PR changes Telegram target parsing and tests so malformed multi-colon targets fall back as full strings while preserving t.me URL-form numeric thread suffixes.

## Best Possible Solution

Keep the parser repair, but preserve both `t.me/...:<id>` and `t.me/...:topic:<id>` lookup-target suffixes while malformed extra-colon strings fall through unchanged, with focused regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main splits `t.me/mychannel:topic:9` in the explicit topic branch, while the PR rejects that branch and can pass `t.me/mychannel:topic` to lookup with thread id `9`.

## Solution Assessment

No. `parseTelegramTarget` is the right layer, but the best fix must reject malformed extra-colon strings without dropping current t.me URL-form explicit topic suffixes.
