## Summary

Keep open: current main still only exposes image-provider inventory through the local CLI and `image_generate` list path; the Gateway method table, handler set, and protocol docs lack a read-only image-provider RPC. This remains a valid additive Gateway feature request that needs maintainer API approval, not a cleanup close candidate.

## What This Changes

Add a read-only Gateway RPC such as `image.providers` that returns secret-free image-generation provider metadata for control UIs.

## Best Possible Solution

Add a typed additive Gateway method under `operator.read` that reuses the image-generation provider registry/list helpers and returns secret-free provider metadata, with protocol schema, docs, SDK/client follow-through, and focused tests kept in sync.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an additive Gateway feature request. Source inspection on current main gives high-confidence absence evidence because the local CLI/tool list paths exist while the Gateway table, handlers, and protocol docs do not expose an image-provider RPC.

## Solution Assessment

Unclear until maintainers approve the public API shape. An additive read-only Gateway RPC appears aligned with `tts.providers`, but the exact method name, response fields, and protocol/client scope should be decided before implementation.
