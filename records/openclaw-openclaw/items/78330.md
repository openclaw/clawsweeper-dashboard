## Summary

Keep open: current main has image provider inventory through the local CLI and `image_generate action=list`, but the Gateway protocol, handler registry, read-scope table, docs, SDK, and UI surfaces do not expose a read-only `image.providers` RPC.

## What This Changes

Request a read-only `operator.read` Gateway RPC that returns image-generation provider metadata without credentials or secret config values.

## Best Possible Solution

Add a typed, additive core Gateway method under `operator.read` that reuses the existing image-generation provider registry and returns a secret-free provider catalog, with protocol schema, docs, tests, and user-facing changelog coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high-confidence absence evidence because current main has CLI/tool image provider listing but no Gateway/protocol/UI/SDK `image.providers` surface.

## Solution Assessment

Unclear pending maintainer API approval. An additive read-only Gateway method appears maintainable and aligned with `tts.providers`, but the exact method name, response fields, and SDK/UI follow-through are public protocol decisions.
