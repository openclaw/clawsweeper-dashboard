## Summary

Keep open: current main still has image provider inventory through the CLI and `image_generate action=list`, but the Gateway protocol, handler registry, advertised method list, read-scope table, SDK helper surface, and docs do not expose a read-only image-provider RPC. This is a new public protocol/API feature, so it needs maintainer shape approval rather than conservative cleanup closure.

## What This Changes

The issue requests a read-only Gateway RPC for image generation provider metadata so external control UIs do not need to spawn `openclaw infer image providers --json`.

## Best Possible Solution

Add a typed, additive core Gateway method under `operator.read` that reuses the existing image-generation provider registry and returns a secret-free provider catalog, with protocol schema, docs, tests, and SDK/UI follow-through once the public shape is approved.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high-confidence absence evidence for the requested Gateway RPC while confirming equivalent data exists in CLI and agent-tool paths.

## Solution Assessment

Unclear pending maintainer API approval. An additive read-only Gateway method appears aligned with `tts.providers`, but the method name, response fields, and client/SDK follow-through are public protocol decisions.
