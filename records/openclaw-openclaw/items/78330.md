## Summary

Keep open: current main still lacks a Gateway RPC for image-generation provider inventory, while the local CLI and `image_generate` list path show the data source already exists. This is a concrete additive Gateway feature request, but it changes a public protocol/method surface and should get maintainer API approval before implementation.

## What This Changes

The issue requests a read-only Gateway RPC, analogous to `tts.providers`, that returns secret-free image-generation provider metadata for external control UIs.

## Best Possible Solution

Add an additive, typed `operator.read` Gateway method that reuses the image-generation provider registry/list helpers and returns only derived, secret-free metadata, with protocol docs and focused Gateway tests updated together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a request for a new Gateway method. Source inspection gives high-confidence absence evidence because current main has local/tool image-provider listing paths but no Gateway descriptor, handler, or protocol docs for an image-provider inventory RPC.

## Solution Assessment

Unclear until maintainers approve the public API shape. An additive read-only Gateway method modeled on `tts.providers` appears maintainable, but method naming, returned fields, and protocol/client scope need product/API review.
