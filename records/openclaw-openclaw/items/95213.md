## Summary

Keep open, but the patch is not safe to merge: it turns plugin-owned `config.summaryModel` into an implicit model-override trust grant, while current code and docs keep that authorization behind explicit `plugins.entries.<id>.llm` policy.

## What This Changes

The PR changes `resolvePluginLlmOverridePolicy` to synthesize an LLM override policy from `plugins.entries.<id>.config.summaryModel` when no explicit `llm` policy exists.

## Best Possible Solution

Land one canonical explicit `plugins.entries.<id>.llm` policy propagation fix for context-engine runtime calls, preserving allowlists and proving the live LCM startup path.

## Reproduction Assessment

Yes at source/report level: current main still routes context-engine model overrides through explicit plugin `llm` policy while the linked issue provides concrete denial logs. I did not run a live gateway/lossless-claw compaction reproduction in this read-only review.

## Solution Assessment

No. Deriving authorization from plugin-owned `config.summaryModel` is not the best fix because current docs, schema, and runtime policy make explicit `llm.allowModelOverride` the trust source; the safer fix is explicit-policy propagation without widening trust.
