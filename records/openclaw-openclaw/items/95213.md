## Summary

Keep this PR open but do not merge it as-is. The branch changes model-override authorization by deriving trusted LLM policy from plugin-owned `config.summaryModel`, while the documented and current runtime contract requires explicit `plugins.entries.<id>.llm.allowModelOverride`; it also only provides test output, not real gateway/LCM proof.

## What This Changes

The PR adds a runtime fallback that creates an LLM override policy from a plugin entry's `config.summaryModel` when no explicit `llm` policy exists.

## Best Possible Solution

Land one canonical fix that preserves explicit `plugins.entries.<id>.llm` as the model-override trust source and repairs the startup/runtime config lifecycle for context-engine `runtime.llm.complete`, with focused tests and redacted live/log proof.

## Reproduction Assessment

Yes at source level for the PR defect: the diff grants override policy from `config.summaryModel`, while current docs and runtime require explicit `llm` policy. I did not run a live current-main LCM compaction repro in this read-only review.

## Solution Assessment

No. This is a plausible workaround, but the best fix is to preserve the explicit `llm` trust gate and repair the startup/runtime policy propagation that the linked issue reports.
