## Summary

Keep open, but this PR is not merge-ready: the patch turns plugin-owned `config.summaryModel` into implicit runtime LLM model-override authority, while current docs and schema keep that trust gate under explicit `plugins.entries.<id>.llm` policy.

## What This Changes

The PR changes runtime LLM policy resolution to infer `allowModelOverride` and `allowedModels` from `plugins.entries.<id>.config.summaryModel` when no explicit `llm` policy exists.

## Best Possible Solution

Land one canonical allowlist-preserving fix that makes context-engine LLM calls honor the owning plugin's explicit `plugins.entries.<id>.llm` policy, with redacted live startup or LCM proof.

## Reproduction Assessment

Yes at source/report level: current main still has context-engine authority with model overrides disabled and fallback plugin-policy lookup, while the linked issue supplies concrete denial logs. I did not run a live gateway/lossless-claw compaction reproduction in this read-only review.

## Solution Assessment

No. Deriving authorization from plugin-owned `config.summaryModel` is not the best fix because current docs, schema, and runtime policy make explicit `llm.allowModelOverride` the trust source.
