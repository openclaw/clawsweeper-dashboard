## Summary

Keep open. Current main still preserves same-session serialization and now has a generic Gateway-timeout embedded fallback, but the queue wait data remains diagnostic-only and no Gateway/CLI contract surfaces a typed session-lane busy or lane-wait timeout with lane, queueAhead, waitedMs, or retry guidance.

## What This Changes

The issue requests a clear CLI-visible busy or timeout error when `openclaw agent --agent <id> --message ...` queues behind an already-running turn for the same session lane.

## Best Possible Solution

Add an additive queue/Gateway/CLI result for blocked same-session dispatch: preserve the existing one-run-per-session behavior, expose a typed lane-busy or lane-wait-timeout terminal response with lane, queueAhead, waitedMs, and retry guidance, print it in text and JSON CLI modes, and document the wait/queue policy.
