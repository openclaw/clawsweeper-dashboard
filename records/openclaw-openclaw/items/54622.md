## Summary

Keep open. Current main still lacks a caller-visible lane-busy or lane-wait-timeout result for blocked same-session `openclaw agent` dispatches; queue diagnostics were improved, but the CLI/Gateway path still turns this into generic accepted-then-wait behavior and timeout fallback without lane, queueAhead, or waitedMs metadata.

## What This Changes

The issue requests a clear CLI error when a second `openclaw agent --agent <id> --message ...` dispatch is blocked behind an already-running same-session lane.

## Best Possible Solution

Add an additive queue/Gateway/CLI lane-wait timeout or busy result that preserves same-session serialization and prints text/JSON with lane, queueAhead, waitedMs, and recovery guidance.

## Reproduction Assessment

Yes, at source level. Current main still serializes same-session `openclaw agent` work through the session lane, records wait facts inside the queue, and exposes only accepted-then-timeout/fallback behavior to the CLI rather than a typed lane-busy result.

## Solution Assessment

Yes. The narrow maintainable repair is to surface the queue's existing blocked-lane facts through an additive Gateway/CLI result contract, not to relax session serialization or add a new default queuing policy.
