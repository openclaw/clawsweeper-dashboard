## Summary

Keep open. Current main has a documented/tested generic Gateway-timeout embedded fallback, but the blocked same-session lane path still does not return a caller-visible lane-busy or lane-wait-timeout result carrying lane, queueAhead, and waitedMs.

## What This Changes

Request that `openclaw agent --agent <id> --message ...` report a clear lane-busy or wait-timeout error when a same-session run is already occupying the lane.

## Best Possible Solution

Add an additive Gateway/CLI lane-wait timeout or busy result that preserves session serialization and prints clear text plus JSON output with lane, queueAhead, waitedMs, and guidance.

## Reproduction Assessment

Yes, at source level. A second `openclaw agent` request for the same session is accepted, queued behind the active session lane, and the queue only logs wait diagnostics while the CLI eventually reports a generic Gateway timeout/fallback rather than the blocked-lane facts.

## Solution Assessment

Yes. The narrow maintainable fix is to surface the already-known queue-wait facts through an additive queue/Gateway/CLI result contract, not to change session-lane serialization or add a new wait-policy flag by default.
