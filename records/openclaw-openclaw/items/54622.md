## Summary

Keep open. Current main still has the source-level failure path: same-session Gateway agent runs serialize through the session lane, lane-wait facts are emitted only as diagnostics, and the CLI waits on a generic final Gateway response or timeout fallback without actionable lane-busy metadata.

## What This Changes

Surface a clear lane-busy or lane-wait timeout error from `openclaw agent` when a same-session Gateway dispatch waits behind active work.

## Best Possible Solution

Add an additive queue/Gateway/CLI lane-wait timeout or busy result that preserves same-session serialization and prints lane, queueAhead or activeAhead, waitedMs, and operator guidance.

## Reproduction Assessment

Yes, at source level. A same-session agent request is serialized behind active session work, while queue wait facts remain diagnostic-only and the CLI waits for a generic Gateway final response without lane-busy metadata.

## Solution Assessment

Yes. The narrow maintainable fix is to surface the existing blocked-lane facts through an additive queue/Gateway/CLI error or result contract, not to relax session serialization or add a new queue policy by default.
