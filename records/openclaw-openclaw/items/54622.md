## Summary

Keep open: current `main` and latest `v2026.6.8` still log lane-wait facts without surfacing a structured session-lane-busy CLI result, and the linked fixing PR is open rather than merged.

## What This Changes

The issue requests an actionable CLI error when a same-session `openclaw agent` dispatch waits behind a blocked session lane instead of exiting silently.

## Best Possible Solution

Keep this issue open until the linked PR is reviewed, refreshed if needed, and either merged with maintainer-approved CLI lane-busy semantics or replaced by a narrower implementation that surfaces the diagnostic without unwanted queue-policy changes.

## Reproduction Assessment

Yes at source level. Current `src/process/command-queue.ts` logs the lane-wait facts, while current `src/commands/agent-via-gateway.ts` and `src/gateway/server-methods/agent.ts` do not expose a structured lane-busy CLI response; I did not run a live two-CLI reproduction in this read-only review.

## Solution Assessment

Unclear as a product contract, but the linked PR is the right implementation candidate to review. The source boundary should be queue/Gateway/CLI, while maintainers decide whether blocked same-session CLI dispatch should fail fast or preserve eventual queueing and only surface diagnostics.
