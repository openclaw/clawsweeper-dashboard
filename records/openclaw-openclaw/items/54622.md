## Summary

Current main still has the reported gap: the queue emits lane-wait diagnostics, but the CLI/Gateway path does not expose a structured lane-busy result to the caller. The issue should stay open because https://github.com/openclaw/openclaw/pull/90933 is an open linked implementation candidate and needs review or replacement before this report can close.

## What This Changes

The issue requests that blocked same-session CLI agent dispatches surface a clear lane-busy error instead of exiting silently.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/90933 with a narrow fix that propagates existing lane-wait facts to the CLI without changing queue ordering or adding unrelated policy.

## Reproduction Assessment

Yes, source-level: current main logs `lane wait exceeded` in `src/process/command-queue.ts` and forwards only a generic wait callback, while `src/commands/agent-via-gateway.ts` has no lane-busy response handling. I did not run the two-terminal live reproduction in this read-only review.

## Solution Assessment

Not yet proven as final; the issue direction is right, but the best implementation is the narrow queue-to-Gateway-to-CLI propagation being reviewed in the linked PR, with a maintainer decision on exit-code behavior.
