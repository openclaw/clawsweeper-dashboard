- `impact:message-loss`: A blocked CLI agent dispatch can leave the caller without a useful reply or failure signal, causing automation to treat an agent message as lost.

## Summary

Keep open. Current main has a documented generic Gateway-timeout embedded fallback, but the session-lane wait facts are still diagnostic-only and are not returned through the Gateway/CLI as actionable lane-busy metadata.

## What This Changes

The issue asks for `openclaw agent` to report an actionable lane-busy or wait-timeout error when a same-session CLI dispatch is blocked behind active work.

## Best Possible Solution

Add an additive queue/Gateway/CLI lane-wait timeout or busy result that preserves same-session serialization and prints lane, queueAhead or activeAhead, waitedMs, and guidance.

## Reproduction Assessment

Yes, at source level. A same-session `openclaw agent` request is serialized behind active session work, while queue wait facts remain logged diagnostics and the CLI only has a generic Gateway-timeout fallback path.

## Solution Assessment

Yes. The narrow maintainable fix is to surface the existing blocked-lane facts through an additive queue/Gateway/CLI error or result contract, not to relax session serialization or add a new queue policy by default.
