## Summary

Keep open. Current main still serializes same-session agent runs, but the queue wait is only logged as a diagnostic warning and the Gateway/CLI path still lacks a typed lane-busy or lane-wait-timeout result with lane, queueAhead, waitedMs, and retry guidance.

## What This Changes

The issue requests that `openclaw agent --agent <id> --message ...` surface a clear lane-busy or wait-timeout error instead of exiting silently when another run is already active for the same session.

## Best Possible Solution

Add an additive Gateway/CLI contract for blocked same-session agent dispatch: preserve session-lane serialization, return or emit a typed lane-busy/lane-wait-timeout terminal result with lane, queueAhead, waitedMs, retry guidance, and make `openclaw agent` print it clearly in text and JSON modes.

## Reproduction Assessment

Yes. The issue gives a concrete two-dispatch reproduction, and current source confirms same-session runs queue behind `session:<key>` while the only queue-wait signal is a diagnostic log/callback rather than a caller-visible terminal result.

## Solution Assessment

Yes, the narrow maintainable fix is to surface the existing queue-wait information through an additive Gateway/CLI result contract rather than changing the safe serialization semantics. Any `--force-queue` or wait-policy flag should be treated as optional product follow-up unless maintainers explicitly want it in the same PR.
