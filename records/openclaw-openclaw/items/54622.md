## Summary

Keep open. Current main still lacks a caller-visible lane-busy or lane-wait-timeout result for blocked same-session `openclaw agent` dispatch; the queue records wait facts internally and the CLI has generic Gateway-timeout fallback, but lane, queueAhead, and waitedMs are not returned to the caller.

## What This Changes

The issue asks the CLI/Gateway path to surface an actionable error when a same-session agent lane is blocked instead of exiting silently or timing out generically.

## Best Possible Solution

Add an additive queue/Gateway/CLI lane-wait timeout result that preserves same-session serialization and prints text/JSON with lane, queueAhead, waitedMs, and guidance.

## Reproduction Assessment

Yes, at source level. A second same-session `openclaw agent` request is accepted and waits behind the active session lane; current source exposes only generic Gateway timeout/fallback to the CLI, not the lane-wait facts already known by the queue.

## Solution Assessment

Yes. The narrow maintainable fix is to surface the existing blocked-lane facts through an additive queue/Gateway/CLI error or result contract, not to relax session serialization or add `--force-queue` by default.
