## Label Justifications

- `P2`: This is a normal-priority CLI/Gateway bug with a clear source path and limited blast radius to blocked same-session agent dispatch.
- `impact:message-loss`: A blocked CLI agent dispatch can leave the caller without a useful reply or failure signal, causing automation to treat an agent message as lost.

## Summary

Keep open. Current `main` still preserves the source-level failure path: same-session agent runs serialize through the session lane, lane-wait facts are emitted as diagnostics, and the Gateway/CLI path has no caller-visible lane-busy or wait-timeout result carrying those facts.

## What This Changes

The issue requests a clear CLI/Gateway error when a blocked same-session agent relay waits behind an active session lane instead of exiting without actionable lane-busy details.

## Best Possible Solution

Add an additive queue/Gateway/CLI lane-wait timeout or busy result that preserves same-session serialization and prints lane, queueAhead or activeAhead, waitedMs, and operator guidance.

## Reproduction Assessment

Yes, at source level. A same-session agent request serializes behind active session work, while queue wait facts remain diagnostic-only and the CLI waits for a generic Gateway final response without lane-busy metadata.

## Solution Assessment

Yes. The narrow maintainable fix is to surface the existing blocked-lane facts through an additive queue/Gateway/CLI error or result contract, not to relax session serialization or add a new queue policy by default.
