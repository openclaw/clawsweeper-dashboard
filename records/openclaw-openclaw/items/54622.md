## Summary

Keep open: current main still preserves the source-level failure shape where a blocked same-session CLI agent dispatch can wait behind the session lane while lane-wait details remain diagnostics instead of a caller-visible busy or wait-timeout result.

## What This Changes

The issue requests a clear CLI/Gateway error when an agent relay request is blocked by an occupied session lane instead of exiting silently with code 1.

## Best Possible Solution

Add an additive lane-wait timeout or busy result through the command queue, Gateway agent protocol, and CLI relay while preserving same-session serialization and printing lane, queueAhead or activeAhead, waitedMs, and operator guidance.

## Reproduction Assessment

Yes, at source level. Current main still serializes same-session agent requests through enqueueSession while queue wait facts remain diagnostic-only and the CLI waits for a generic Gateway final response.

## Solution Assessment

Yes. The maintainable fix is an additive queue/Gateway/CLI error or result contract that surfaces existing blocked-lane facts, not a change to session serialization or a new default queue policy.
