## Summary

Keep open. Current main still logs blocked-lane facts only through diagnostics while same-session Gateway agent runs serialize behind the session lane and the CLI waits for a generic final Gateway response/fallback rather than surfacing actionable lane-busy metadata.

## What This Changes

The issue asks for `openclaw agent` to print a clear lane-busy or wait-timeout error when a same-session request is blocked behind active work.

## Best Possible Solution

Add an additive queue/Gateway/CLI lane-wait timeout or busy result that preserves same-session serialization and prints lane, queueAhead or activeAhead, waitedMs, and operator guidance.

## Reproduction Assessment

Yes, at source level. A same-session agent request is serialized behind active session work, while queue wait facts remain logged diagnostics and the CLI waits for a generic Gateway final response without lane-busy metadata.

## Solution Assessment

Yes. The narrow maintainable fix is to surface the existing blocked-lane facts through an additive queue/Gateway/CLI error or result contract, not to relax session serialization or add a new queue policy by default.
