## Summary

Keep open: current main still has the reported failure shape. The queue emits the lane-wait facts internally and recent related work improved that diagnostic, but there is still no structured Gateway or CLI lane-busy result for the blocked relay path.

## What This Changes

The issue asks for `openclaw agent --agent <id> --message ...` to print a clear busy or wait-timeout error when the target session lane is blocked instead of exiting silently with code 1.

## Best Possible Solution

Add an additive lane-wait or busy result through the command queue, Gateway agent response path, and CLI relay while preserving same-session serialization and rendering lane, waitedMs, queueAhead or activeAhead, and operator guidance.

## Reproduction Assessment

Yes, at source level. Current main serializes same-session agent requests through enqueueSession while lane wait facts remain diagnostics-only and the CLI waits for a generic Gateway final response.

## Solution Assessment

Yes. The maintainable fix is an additive queue/Gateway/CLI error or result contract that surfaces existing blocked-lane facts, not a change to session serialization or a new default queue policy.
