## Summary

This issue is still necessary: current main still has the lane-wait facts only as queue diagnostics, while the CLI/Gateway path has no merged lane-busy result for callers. ## What This Changes

The issue asks the CLI agent relay to show an actionable session-lane-busy error instead of exiting silently when a same-session lane wait exceeds the timeout.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/90933 is reviewed/refreshed and either lands a CLI-visible lane-busy diagnostic or maintainers choose a different queueing contract.

## Reproduction Assessment

Yes, source-level reproduction is credible: current main records lane-wait facts in the command queue, while the CLI/Gateway path has no lane-busy result or renderer. I did not perform the live two-terminal repro in this read-only review, so this is not marked fully reproduced.

## Solution Assessment

Unclear until maintainers settle the CLI queueing contract. Propagating existing queue wait facts through Gateway to the CLI is the right boundary, but maintainers need to decide whether the busy state should reject queued CLI turns or only surface diagnostics while preserving eventual queueing.
