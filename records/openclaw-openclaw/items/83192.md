## Summary

Keep open: current main still has the reported state-cleanup behavior, and the latest shipped release has the same `sessions.patch model:null` path that clears model override fields while leaving or setting `liveModelSwitchPending`. The report is narrow and source-reproducible, but the repair needs to preserve active-run live reset semantics rather than blindly dropping the flag in every reset path.

## What This Changes

The issue reports that clearing a session model pin through `sessions.patch` leaves `liveModelSwitchPending` persisted, which can make `/model` render stale ghost model labels after a provider is removed.

## Best Possible Solution

Teach the explicit model-clear path to drain stale pending state without losing active-run reset-to-default switching, and add regression coverage for both the session-store fields and the affected `/model` display path.

## Reproduction Assessment

Yes, at source level: applying a concrete `sessions.patch` model followed by `model: null` follows the current code path that clears override fields while preserving or setting `liveModelSwitchPending`. I did not run a live gateway/Telegram repro because this review is read-only and must not create artifacts.

## Solution Assessment

No, the suggested unconditional `clearLiveModelSwitchPending` call is probably too blunt because `model:null` may need to live-switch an active run back to the default model. The safer fix is to drain the orphaned flag only when no active switch remains pending, and to harden display/resolution surfaces so absent override fields cannot synthesize stale labels.
