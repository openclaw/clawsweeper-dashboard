## Summary

Current main still has the reported orphan-flag behavior: `sessions.patch` with `model: null` clears the override fields but marks the reset as a pending live switch, and the existing focused test currently expects that persisted flag. This is a valid, narrow session-state bug rather than a stale or duplicate cleanup item.

## What This Changes

The issue reports that clearing a pinned session model with `sessions.patch` leaves `liveModelSwitchPending` persisted and can surface stale model labels after provider removal.

## Best Possible Solution

Clear `liveModelSwitchPending` as part of explicit model reset, add focused regression coverage for `sessions.patch model: null`, and only add a defensive display guard if the status/menu path can still synthesize stale selections after the flag is gone.

## Reproduction Assessment

Yes, at source level: current main has a focused `sessions.patch` test where `{ model: null }` clears override fields while expecting `liveModelSwitchPending` to remain `true`. I did not run the gateway repro because this review was constrained to read-only inspection.

## Solution Assessment

Yes, the narrowest maintainable fix is to drain or avoid setting `liveModelSwitchPending` on explicit model reset, then update the existing regression test. A broader Telegram/status rewrite is not needed unless a focused test proves stale display synthesis remains after the reset fix.
