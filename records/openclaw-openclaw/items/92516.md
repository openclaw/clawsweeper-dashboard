## Label Justifications

- `P1`: A released official external channel can load and start in a common self-hosted deployment model, then repeatedly exit when it needs keyed state.
- `impact:security`: The requested deployment capability crosses the trust/provenance boundary for plugins accessing host-managed persistent state.
- `impact:crash-loop`: The reported denial exits the channel and triggers automatic restart attempts.

## Summary

Keep open: current main still blocks an image-baked official external channel loaded through `plugins.load.paths` from keyed state unless it has trusted install-record provenance, and the related open issues/closed PR do not replace this distinct immutable-container deployment contract gap.

## What This Changes

The issue requests a supported trust/provenance path for self-hosted or image-baked external channel plugins that need `openKeyedStore`, plus clearer headless channel provisioning support.

## Best Possible Solution

Add a provenance-preserving official plugin install/export path for immutable containers, with doctor/docs coverage, while keeping arbitrary load-path plugins untrusted by default.

## Reproduction Assessment

Yes at source level: a load-path official channel without matching trusted install metadata deterministically fails the current `openKeyedStore` gate. I did not run the reporter's ECS/Fargate environment in this read-only review.

## Solution Assessment

No to raw trust allowlists or implicit load-path trust as the best fix, because they weaken provenance. The safer solution is a portable official-install contract that remains verifiable at runtime.
