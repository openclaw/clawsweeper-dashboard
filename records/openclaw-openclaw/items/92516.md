## Label Justifications

- `P1`: A source-confirmed trust/provenance gap makes externalized official channels unusable through repeated startup failures in a real self-hosted deployment model.
- `impact:security`: The requested capability crosses the deliberate provenance boundary guarding privileged plugin state APIs.
- `impact:crash-loop`: MS Teams opens the denied stores during startup, causing the channel to exit and auto-restart repeatedly.

## Summary

Current main and v2026.6.6 still require a non-bundled plugin to match a recorded official install before privileged keyed-state access, while MS Teams opens those stores during startup; immutable image-baked installs therefore remain unsupported, and the narrower container provenance gap is not replaced by the broader community-plugin or ClawHub reports.

## What This Changes

Add a supported immutable-container deployment path for externalized official channel plugins that require trusted keyed state.

## Best Possible Solution

Define one canonical portable official-install provenance format that can be produced during image build, verified against package identity and the official catalog at runtime, and documented for immutable containers without treating arbitrary load paths as trusted.

## Reproduction Assessment

Yes. Current source provides a high-confidence reproduction chain: an unrecorded config-path plugin cannot receive `trustedOfficialInstall`, and MS Teams synchronously calls the gated keyed-store API during startup; this review did not run the reporter's ECS environment.

## Solution Assessment

No, not all proposed options are equally safe. Implicitly trusting configured paths would weaken provenance; a portable, verifiable official-install record is the narrowest approach that preserves the existing boundary, while non-interactive channel setup should be handled separately.
