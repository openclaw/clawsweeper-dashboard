## Label Justifications

- `P1`: A released official channel can load and start but repeatedly terminate in a common immutable self-hosted deployment model.
- `impact:security`: The requested capability crosses the provenance boundary controlling plugin access to host-managed persistent state.
- `impact:crash-loop`: The keyed-state denial exits the channel and triggers repeated automatic restarts.

## Summary

Current main and the latest release still require bundled origin or verified official-install provenance for keyed plugin state, while an image-baked plugin supplied through `plugins.load.paths` has no supported way to obtain that provenance. The issue remains distinct from the broader community-plugin capability request and the ClawHub catalog mismatch, and resolving it requires an explicit security and product decision rather than cleanup closure.

## What This Changes

Add a supported deployment path for image-baked external channel plugins that need trusted keyed state in immutable or containerized environments.

## Best Possible Solution

Add a generic build-stage install/export contract for official plugins that produces host-verifiable provenance usable from a separate immutable runtime root, rejects identity or path mismatches, diagnoses failures before restart loops, and leaves arbitrary load-path plugins untrusted.

## Reproduction Assessment

Yes, at source level with high confidence: a `plugins.load.paths` plugin without matching trusted official-install metadata deterministically throws when it opens keyed state. The report also includes a concrete v2026.6.6 container trace, although this review did not recreate the ECS environment.

## Solution Assessment

No to implicit load-path trust or an unrestricted trust allowlist, because those weaken provenance. A portable, verifiable official-install contract is the narrowest maintainable direction; non-interactive channel provisioning should be considered separately.
