## Summary

Keep open: current main and v2026.6.8 still require bundled or trusted official-install provenance before an external channel can use keyed state, and the reported immutable-container case still has no supported install-record path that survives a mounted state directory.

## What This Changes

The issue asks for a supported self-hosted/container deployment path for externalized channel plugins that preserves trusted keyed-state provenance without relying on raw load-path trust.

## Best Possible Solution

Define a supported immutable-container official plugin install/provenance contract, likely with a separate image-baked plugin root or export/import flow, while keeping arbitrary load-path plugins untrusted and documenting the Docker handoff.

## Reproduction Assessment

Yes at source level: a load-path or otherwise unrecorded external channel lacks trustedOfficialInstall and deterministically hits the openKeyedStore gate. I did not run the reporter's ECS/Fargate environment in this read-only review.

## Solution Assessment

No for raw trust allowlists or implicit load-path trust, because those are not the safest way to solve the issue. The best path is a maintainer-approved portable official-install provenance contract that keeps package identity verifiable.
