## Label Justifications

- `P1`: A released official external channel can load and start in a common self-hosted container deployment, then repeatedly terminate before serving users.
- `impact:security`: The requested fix crosses the trust/provenance boundary that controls plugin access to host-managed persistent state.
- `impact:crash-loop`: The keyed-state denial exits the channel and triggers automatic restart attempts.

## Summary

Keep open: current main and v2026.6.6 still require bundled origin or trusted official-install provenance for keyed state, while image-baked `plugins.load.paths` official channels have no supported way to create that trust record. Related open issues cover broader community keyed-state access, WhatsApp Docker symptoms, and a ClawHub catalog mismatch, but they do not replace this immutable-container deployment gap.

## What This Changes

Support externalized official channel plugins, especially image-baked Microsoft Teams in self-hosted containers, so they can access trusted keyed state without crash-looping.

## Best Possible Solution

Add a maintainer-approved portable official-plugin install/provenance contract for immutable images, backed by verifiable package identity and docs/doctor diagnostics; keep arbitrary load paths and id allowlists untrusted by default.

## Reproduction Assessment

Yes at source level: a load-path plugin without matching trusted official-install metadata deterministically throws when it opens keyed state; the report also includes a v2026.6.6 container trace, but this review did not run ECS/Docker.

## Solution Assessment

Unclear until maintainers choose the trust contract; raw config trust or implicit load-path trust is not the best fix because it weakens provenance, while portable official-install provenance or a documented image-baked distribution path fits the boundary better.
