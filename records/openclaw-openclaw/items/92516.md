## Label Justifications

- `P1`: A released official external channel can load in a common self-hosted container pattern and then repeatedly terminate when it reaches trusted keyed state.
- `impact:security`: The requested fix crosses the provenance boundary that controls which plugins can access host-managed persistent state APIs.
- `impact:crash-loop`: The reported denial exits the channel process path and triggers repeated automatic restarts.

## Summary

Keep open: current main and v2026.6.6 still enforce keyed-state access only for bundled or trusted official installs, while image-baked `plugins.load.paths` channels have no supported install-record trust path. This is a real self-hosted official-channel deployment gap, but the fix needs a maintainer decision because it changes plugin provenance and state-access security boundaries.

## What This Changes

The issue asks for a supported way to run externalized official channel plugins from immutable/self-hosted containers when those plugins need trusted keyed state.

## Best Possible Solution

Add a provenance-safe portable official-plugin install/export path for immutable containers and document that path, while keeping arbitrary load-path plugins untrusted by default.

## Reproduction Assessment

Yes at source level: a `plugins.load.paths` channel without matching trusted official-install metadata deterministically fails the keyed-state gate when it opens `openKeyedStore`. The reporter also provides a concrete OpenClaw 2026.6.6 container trace, but this review did not run ECS/Fargate.

## Solution Assessment

Unclear until maintainers choose the trust contract; implicit load-path trust or a raw config trust bit is not the best default because it weakens provenance. The safer direction is a verifiable official-plugin portable install/export path plus docs and upgrade tests.
