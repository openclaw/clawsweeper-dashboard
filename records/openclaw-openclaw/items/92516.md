## Label Justifications

## Summary

Keep open: current main and v2026.6.6 still enforce the keyed-store trust gate, and there is still no supported portable trust/provenance path for an image-baked `plugins.load.paths` channel. The related issues cover broader community access, WhatsApp Docker installs, and a ClawHub-specific MS Teams provenance mismatch, but none supersedes this immutable-container deployment gap.

## What This Changes

The issue asks for a supported self-hosted/container deployment path for official externalized channel plugins that need `openKeyedStore` without relying on an interactive managed install under the writable user state directory.

## Best Possible Solution

Define an explicit, auditable portable official-install provenance path for immutable/container deployments, keep arbitrary load-path plugins untrusted by default, and document the build-stage/runtime handoff separately from channel provisioning UX.

## Reproduction Assessment

Yes at source level: a non-bundled plugin record without trusted official-install provenance deterministically throws when it opens keyed state. I did not run the AWS ECS/Fargate container scenario live.

## Solution Assessment

No for implicit load-path trust or raw trust allowlists, because those would weaken package provenance. The best fix is a maintainer-approved portable official-install/provenance contract with focused docs and tests.
