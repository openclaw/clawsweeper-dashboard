## Label Justifications

- `P1`: A released official external channel can load and start but then repeatedly terminate in a common immutable self-hosted deployment model.
- `impact:crash-loop`: The keyed-state denial exits the channel and triggers repeated automatic restarts.
- `impact:security`: The requested deployment capability crosses the provenance boundary controlling plugin access to host-managed persistent state.

## Summary

The issue is source-reproducible on current main: an image-baked official channel loaded through `plugins.load.paths` lacks the recorded official-install provenance required by the keyed-state proxy, so MS Teams can start and then enter its restart loop. Related reports cover community-plugin access, WhatsApp Docker installs, and a ClawHub-specific MS Teams catalog mismatch, but none replaces this distinct immutable-image deployment contract gap.

## What This Changes

Support immutable/container image installation of official external channel plugins so their host-trusted state APIs work without installing into the runtime-mounted state directory.

## Best Possible Solution

Add a generic build-stage install/export path for official plugins that records verifiable package identity and supports a separate immutable runtime plugin root, then document the container handoff while leaving arbitrary load-path plugins untrusted.

## Reproduction Assessment

Yes at source level: a `plugins.load.paths` plugin without matching trusted official-install metadata deterministically throws when it opens keyed state. The issue also provides a concrete OpenClaw 2026.6.6 container trace, though this review did not run the ECS environment.

## Solution Assessment

No to the proposed raw trust allowlists or implicit load-path trust, because those weaken provenance. The maintainable solution is an explicit portable official-install contract whose package identity remains host-verifiable.
