- `impact:crash-loop`: The reported keyed-state denial exits the channel and triggers repeated automatic restarts.

## Summary

Keep open: current main and v2026.6.8 still require bundled origin or trusted official-install provenance before a channel can open keyed state, and the reporter's mounted-state immutable-container deployment gap is not solved by the closed ClawHub-only PR.

## What This Changes

The issue asks for a supported self-hosted container path for official externalized channel plugins that preserves trusted install provenance when the runtime OpenClaw state directory is mounted separately.

## Best Possible Solution

Define a portable official-plugin install/export and runtime plugin-root contract that preserves host-verifiable package identity across mounted state directories, while leaving arbitrary load-path plugins fail-closed.

## Reproduction Assessment

Yes, at source level: a configured-path external channel without matching trusted official-install metadata deterministically hits the keyed-state gate when it calls openKeyedStore. I did not run the reporter's ECS/Fargate environment in this read-only review.

## Solution Assessment

No for raw trusted allowlists or implicit load-path trust; those are not the best security shape. The best direction is a maintainer-approved portable official-install/provenance contract that works with mounted runtime state.
