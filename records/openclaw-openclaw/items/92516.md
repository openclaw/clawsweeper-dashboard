## Label Justifications

- `P1`: A released official external channel can load in a common self-hosted container model and then repeatedly terminate because trusted state provenance cannot be preserved across the mounted state directory.
- `impact:security`: The remaining fix crosses the trust boundary that controls which external plugins receive host-managed persistent state APIs.
- `impact:crash-loop`: The reported failure exits the channel after startup and triggers repeated auto-restart attempts.

## Summary

Keep open: current main and v2026.6.8 still have no supported immutable-container path that preserves official external-channel provenance when the runtime state directory is mounted over the build-time install state. The raw load-path trust options should stay rejected, but the narrower portable official-install/root contract remains unresolved.

## What This Changes

The issue requests a supported self-hosted/container deployment path for official external channel plugins that need trusted keyed state without treating arbitrary load-path packages as trusted.

## Best Possible Solution

Define a maintainer-approved portable official-plugin install/provenance contract for immutable container images, with an image-baked plugin root or export/import flow that survives a mounted state directory while keeping arbitrary load paths untrusted.

## Reproduction Assessment

Yes. Source inspection shows a plugins.load.paths or other untrusted external channel cannot receive trusted keyed state, and the issue provides concrete OpenClaw 2026.6.6 container logs for the resulting channel restart loop.

## Solution Assessment

Unclear as an implementation until maintainers choose the public contract. The best path is not raw load-path trust, but a provenance-preserving official-plugin install/export root that works with immutable images and mounted runtime state.
