## Label Justifications

- `P1`: A released official external channel can load in a self-hosted container and then repeatedly terminate, blocking a real deployment workflow.
- `impact:security`: The requested fix changes the trust boundary that controls which external plugins can access host-managed persistent keyed state.
- `impact:crash-loop`: The reported failure exits the channel after startup and triggers automatic restart attempts.

## Summary

Keep open: current main and v2026.6.6 still require bundled origin or trusted official-install provenance before a channel plugin can use keyed state, and this image-baked/container trust path is not covered by the broader community-plugin request or the ClawHub-specific MS Teams report.

## What This Changes

The issue asks for a supported trust/provisioning path so self-hosted external channel plugins loaded from immutable container paths can use `openKeyedStore` without crash-looping.

## Best Possible Solution

Define a generic portable official-plugin install/provenance contract for immutable/container deployments, document the Docker handoff, and keep arbitrary load-path plugins untrusted unless maintainers explicitly approve a narrower capability model.

## Reproduction Assessment

Yes at source level: a non-bundled load-path plugin without matching trusted official-install metadata still deterministically throws when it opens keyed state. This review did not run the reporter's ECS/Fargate environment.

## Solution Assessment

No to implicit load-path trust or a raw trust allowlist as the default solve, because those bypass package provenance. A portable official-install/provenance contract is the safer maintainable direction.
