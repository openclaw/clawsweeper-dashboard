## Label Justifications

- `P1`: A released official external channel can load and start in a common immutable self-hosted deployment model, then repeatedly terminate before serving users.
- `impact:security`: The requested fix changes the provenance boundary controlling plugin access to host-managed persistent keyed state.
- `impact:crash-loop`: The keyed-state denial exits the channel and triggers repeated automatic restart attempts.

## Summary

Keep open: current main and v2026.6.6 still require bundled or trusted official-install provenance for keyed state, and an image-baked plugin loaded through `plugins.load.paths` has no supported trust path. The related open reports cover broader community keyed-state access, WhatsApp Docker symptoms, and a ClawHub catalog mismatch, but none owns this exact immutable self-hosted official-channel deployment contract.

## What This Changes

The issue asks for a supported container/IaC trust and provisioning path for external official channel plugins such as MS Teams when they are image-baked and need keyed state.

## Best Possible Solution

Add a generic portable official-plugin install/export/import provenance path for immutable container images, with verifiable package identity plus Docker/IaC docs and doctor diagnostics, while leaving arbitrary load-path plugins untrusted by default.

## Reproduction Assessment

Yes at source level: a `plugins.load.paths` MS Teams candidate without a matching official install record lacks `trustedOfficialInstall`, so `openKeyedStore` throws on startup. The report also supplies a concrete v2026.6.6 ECS/Fargate trace, but this read-only review did not run the live container environment.

## Solution Assessment

No to raw trust allowlists or implicit load-path trust because those weaken the provenance boundary. The best fix is a maintainer-approved portable official-install contract plus diagnostics and docs; non-interactive channel provisioning can be a follow-up unless required for that contract.
