## Label Justifications

- `P1`: A shipped official external channel can load and then repeatedly fail in a credible immutable self-hosted deployment model.
- `impact:crash-loop`: The trusted keyed-store denial causes the channel to exit and enter automatic restart attempts.
- `impact:security`: The requested capability changes how externally installed code can cross a privileged persistent-state trust boundary.

## Summary

Current main and v2026.6.6 contain the reported trust dead-end: load-path plugins cannot earn official-install trust, while Microsoft Teams relies on the trusted-only keyed-state API. This is a valid security-sensitive deployment capability gap requiring maintainer product judgment, not an implemented, duplicate, or out-of-repo item.

## What This Changes

Provide a supported immutable-container installation and trust path for official external channel plugins that require keyed state.

## Best Possible Solution

Add portable, integrity-checked official-install provenance for image-baked plugins, preserve arbitrary load paths as untrusted by default, and document and test a fully declarative container startup workflow.

## Reproduction Assessment

Yes at high source confidence: shipped and current code make load-path candidates ineligible for official trust, while Microsoft Teams uses the trusted-only keyed-state API. A live ECS run was not performed, but the reported failure follows directly from these contracts.

## Solution Assessment

The requested outcome is appropriate, but automatically trusting configured paths or plugin ids is not the best solution. Portable trust should be tied to official catalog identity and package integrity or equivalent auditable install provenance.
