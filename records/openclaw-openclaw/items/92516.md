## Summary

Keep open: current main and the latest release still require bundled origin or trusted official-install provenance before external channel plugins can use keyed state, and the existing docs/code do not provide the requested immutable-image path where official plugin package/provenance survives a mounted runtime state directory.

## What This Changes

The issue asks for a supported self-hosted container deployment path where official externalized channel plugins can retain host-verifiable trust and use keyed plugin state when the runtime state directory is mounted separately.

## Best Possible Solution

Keep the fail-closed trust gate and design a supported portable official-plugin install/provenance path for immutable images whose plugin package and install record survive a mounted runtime state directory.

## Reproduction Assessment

Yes. Current source shows a non-bundled plugin without matching trusted official-install metadata will throw when it calls openKeyedStore, and MS Teams calls that API; I did not run the ECS/Fargate container reproduction in this read-only pass.

## Solution Assessment

No to raw trust allowlists or implicit load-path trust; the best path is a provenance-preserving portable install/deployment contract that keeps arbitrary copied packages untrusted.
