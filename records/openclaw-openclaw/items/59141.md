## Summary

Keep open: the stale-recall problem is still valid, but this PR is not merge-ready because it targets the legacy recall hook, introduces a capture regression for explicit remember questions, and has no after-fix real behavior proof.

## What This Changes

The PR updates memory-lancedb source and tests to detect freshness-sensitive queries, overfetch and recency-rerank LanceDB results, add timestamp/provenance hints, adjust capture heuristics, and cover newer-versus-older recall ordering.

## Best Possible Solution

Land a refreshed memory-lancedb fix on before_prompt_build that preserves latest-user-message normalization and timeout behavior, keeps explicit remember requests capturable, adds timestamp-aware reranking/provenance context, and includes real setup proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to seed two matching LanceDB memories where the older item has stronger vector similarity, ask a latest/current prompt, and observe current main injecting db.search(vector, 3, 0.3) results without recency reranking or timestamp hints.

## Solution Assessment

No. Recency-aware ranking and timestamp context are the right direction, but this branch must be ported to the current before_prompt_build path and must not drop explicit remember requests phrased as questions.
