## Summary

Keep open; current main still writes device identity key material with direct `writeFileSync`, but this PR should not merge as-is because its bespoke atomic helper is weaker than the existing shared JSON writer and can throw on rename-overwrite failures.

## What This Changes

This PR replaces direct device identity JSON writes with a local write-temp-then-rename helper and simplifies the read/parent-directory flow.

## Best Possible Solution

Have device identity persistence use `saveJsonFile` or an equivalent shared helper with secure permissions, random temp paths, fsync, overwrite fallback, cleanup, and focused tests for create, repair, regenerate, and write-failure preservation.

## Reproduction Assessment

Yes. A focused Vitest can create or corrupt an existing identity, mock `fs.renameSync` to throw EPERM or EEXIST during repair/regeneration, and assert `loadOrCreateDeviceIdentity` completes while preserving the expected stored identity.

## Solution Assessment

No. The owner boundary is right and the temp mode is now secure, but the bespoke helper is not the narrowest maintainable fix when `src/infra/json-file.ts` already provides stronger tested atomic JSON semantics.
