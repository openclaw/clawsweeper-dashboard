## Summary

Keep open. The PR is a useful, narrow data-integrity fix, and current main still writes device identity key material directly, but the proposed helper is not merge-ready because it does not reuse or match the stronger atomic JSON replacement behavior already present in main and it lacks targeted failure-path coverage.

## What This Changes

The PR changes `src/infra/device-identity.ts` to replace direct device identity JSON writes with a local temp-file-and-rename helper, default temp files to `0o600`, simplify reads, and wrap `ensureDir` error handling.

## Best Possible Solution

Land a refreshed implementation that routes device identity JSON persistence through `saveJsonFile` or an equivalent shared helper with random exclusive temp files, secure permissions, overwrite fallback, cleanup, and focused tests for create, repair, failure preservation, and temp cleanup.

## Reproduction Assessment

Yes. A focused Vitest can create an identity, corrupt the stored `deviceId`, mock `fs.renameSync` to throw EPERM or EEXIST while replacing the existing file, and assert that `loadOrCreateDeviceIdentity` preserves or repairs the existing key material instead of regenerating or throwing.

## Solution Assessment

No. The PR chooses the right owner boundary and fixes a real direct-write weakness, but a local minimal helper is not the narrowest maintainable solution now that `src/infra/json-file.ts` already provides stronger atomic JSON write semantics.

## Review Findings
