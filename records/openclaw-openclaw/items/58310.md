## Summary

Keep open. Current main still writes device identity JSON with direct `fs.writeFileSync`, and the PR addresses a valid key-material data-integrity gap, but the proposed helper is weaker than the existing atomic JSON writer and can fail on overwrite fallback paths before merge.

## What This Changes

The PR replaces direct device identity JSON writes with a local write-temp-then-rename helper, simplifies the read path, and wraps parent-directory creation.

## Best Possible Solution

Land a refreshed implementation that routes device identity JSON writes through `saveJsonFile` or an equivalent shared helper with 0o600 permissions, random exclusive temp files, fsync, overwrite fallback, cleanup, and targeted tests.

## Reproduction Assessment

Yes. A focused test can create an identity, corrupt the stored `deviceId`, mock `fs.renameSync` to throw EPERM/EEXIST when replacing an existing target, and assert the identity repair/regeneration path preserves or completes cleanly instead of throwing.

## Solution Assessment

No. The owner boundary is right and the 0o600 temp mode is now fixed, but adding a bespoke minimal helper is not the best solution when `src/infra/json-file.ts` already carries the repo's stronger atomic replacement behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Handle overwrite fallback for identity atomic writes:** `src/infra/device-identity.ts:48`
  - body: The new helper replaces an existing identity with plain `fs.renameSync(tmpPath, filePath)`. The repo's shared JSON writer already handles EPERM/EEXIST by copying over the target, so on Windows-style overwrite failure the repair/regenerate path can throw instead of preserving or repairing the identity. Reuse `saveJsonFile` or mirror the same fallback here.
  - confidence: 0.84
