## Summary

The PR targets a plausible P1 isolated-cron regression, but the latest fallback still collapses every missing `sourceDelivery` case to `cron_none`, which can change legacy announce/webhook delivery behavior for the exact version-skew boundary it is trying to support.

## What This Changes

The PR makes the isolated cron executor tolerate missing `sourceDelivery` by constructing a fallback source-delivery plan and adds unit tests for undefined and valid plans.

## Best Possible Solution

Keep the crash guard, but reconstruct the missing plan from any legacy `sourceReplyDeliveryMode`, `toolPolicy`, and resolved target fields when present, then require redacted real cron runtime proof for the affected upgrade path.

## Reproduction Assessment

Yes from source for the version-skew crash shape: current `main` dereferences `params.sourceDelivery.sourceReplyDeliveryMode`, so a stale caller that omits `sourceDelivery` fails during executor setup. I did not reproduce a real installed stale-artifact cron run in this read-only review.

## Solution Assessment

No. Guarding the executor boundary is the right direction, but this patch should preserve the legacy delivery fields instead of collapsing every missing-plan case to `cron_none`.
