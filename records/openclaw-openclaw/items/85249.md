## Summary

Keep this PR open for maintainer review: the patch is a plausible and mechanically correct fix for the open isolated-cron crash, but merging it intentionally accepts a stale/version-skew executor fallback and visible delivery reconstruction contract that maintainers should approve.

## What This Changes

The PR makes `sourceDelivery` optional at the isolated cron executor boundary, adds a shared cron source-delivery fallback planner, reuses that planner from the normal cron path, and adds focused guard/parity tests.

## Best Possible Solution

Land after maintainers explicitly accept the stale-artifact compatibility behavior, with optional live announce-transport proof if they want visible delivery confirmation before merge.

## Reproduction Assessment

Yes at the source boundary: current main and `v2026.6.10` still dereference `params.sourceDelivery.sourceReplyDeliveryMode`, and the PR proof exercises the omitted-`sourceDelivery` stale-caller shape. I did not reproduce a live packaged stale-artifact cron deployment in this read-only review.

## Solution Assessment

Yes mechanically if maintainers accept fallback compatibility: the patch shares one planner between normal and fallback paths and avoids legacy field sniffing. The safer alternative, if maintainers do not want runtime tolerance for stale artifacts, is rebuild or doctor-style remediation instead of this fallback.
