## Summary

Keep open for maintainer review: the patch appears correct and well-proven, but current main and v2026.6.8 still lack the guard, and merging intentionally supports a version-skew executor fallback across cron message delivery.

## What This Changes

The PR makes isolated cron `sourceDelivery` optional at the executor boundary, reconstructs it from job delivery config when missing, shares the cron source-delivery planner with the normal path, and adds focused guard/parity tests.

## Best Possible Solution

Merge the shared guarded planner only after maintainers accept the runtime-skew fallback contract and required checks are clean; otherwise replace this with a packaging, restart, or doctor remediation path.

## Reproduction Assessment

Yes at source level: current main dereferences `params.sourceDelivery.sourceReplyDeliveryMode`, and the linked issue plus PR proof cover a stale caller omitting `sourceDelivery`. I did not run a live packaged cross-version skew job in this read-only review.

## Solution Assessment

Yes if maintainers accept runtime compatibility for this boundary: sharing the normal cron source-delivery planner with the fallback avoids duplicate policy. The safer alternative, if that contract is rejected, is packaging/restart/doctor remediation rather than a runtime fallback.
