## Label Justifications

- `P1`: The linked user report says isolated cron agent runs fail before the agent starts in recent releases, which is an urgent broken workflow.
- `merge-risk: 🚨 compatibility`: The fallback handles a version-skew boundary and can break upgrade behavior if it ignores legacy fields from stale callers.
- `merge-risk: 🚨 message-delivery`: The fallback can change source-reply suppression, message-tool forcing, and target selection for cron messages.

## Summary

Keep open: the guard is aimed at a real isolated-cron crash, but the latest diff still reconstructs the fallback from the new `resolvedDelivery.channel` shape instead of the legacy lazy-boundary fields, so a version-skewed caller can avoid the TypeError while losing delivery policy and target semantics.

## What This Changes

The PR makes `sourceDelivery` optional in the isolated cron executor, creates a fallback source delivery plan when omitted, logs a warning, and adds source-delivery guard tests.

## Best Possible Solution

Keep `SourceDeliveryPlan` required for normal current callers, but add a narrow version-skew fallback that reconstructs the plan from legacy `messageChannel`, `sourceReplyDeliveryMode`, `toolPolicy`, and resolved target fields when `sourceDelivery` is absent.

## Reproduction Assessment

Yes from source for the version-skew case: pre-refactor code passed separate delivery fields while current `run-executor.ts` requires and dereferences `sourceDelivery`. I did not establish a live stale-build or installed-package reproduction in this read-only review.

## Solution Assessment

No. Guarding the lazy boundary is the right place, but this implementation should preserve the legacy boundary fields instead of hard-coding a degraded `cron_none` source delivery plan.
