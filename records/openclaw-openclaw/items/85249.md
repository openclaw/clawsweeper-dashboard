## Summary

Keep this PR open for explicit cron-owner judgment, but do not merge it as-is: the source-level crash guard is coherent and current main does not contain it, yet the patch converts an internal prepared-plan invariant into a runtime compatibility contract for stale or partially rebuilt artifacts without establishing that mixed-artifact execution is supported. The supplied harness is useful, but it does not prove a real isolated announce cron completes with exactly one channel post.

## What This Changes

The PR makes isolated cron execution reconstruct a missing source-delivery plan from the job and resolved target, centralizes cron source-delivery planning, and adds focused regression tests.

## Best Possible Solution

Keep `sourceDelivery` as a canonical prepared fact and prevent incompatible artifacts through the build, update, cache, or restart lifecycle; if cron owners intentionally support mixed-artifact execution, define that compatibility envelope explicitly and require exact-once real-channel proof for each supported shape.

## Reproduction Assessment

Yes for the direct failure condition: an out-of-contract caller omitting `sourceDelivery` reaches a required-field dereference on current main. The evidence does not yet establish a high-confidence supported production path that legitimately mixes those incompatible artifacts.

## Solution Assessment

No, not without an ownership decision. The reconstruction is internally consistent, but preventing mixed artifacts is the narrower canonical solution unless maintainers explicitly choose to support and version this compatibility behavior.
