## Summary

Keep open for maintainer review: this PR is still the active candidate fix for the open isolated-cron `sourceDelivery` crash, and current `main` plus v2026.6.9 still have the unguarded executor dereference. I found no blocking patch defect, but the runtime fallback and delivery reconstruction are compatibility/message-delivery decisions that need maintainer acceptance before merge.

## What This Changes

The PR makes isolated cron execution tolerate omitted `sourceDelivery` by reconstructing a shared cron source-delivery plan and adds focused regression coverage for announce, none, webhook, and duplicate-suppression behavior.

## Best Possible Solution

Land this guard after maintainers accept the version-skew fallback contract; otherwise replace it with an explicit rebuild or doctor remediation path while keeping one canonical cron source-delivery planner.

## Reproduction Assessment

Yes, at source level: current `main` and v2026.6.9 still directly read `params.sourceDelivery.sourceReplyDeliveryMode`, and the PR proof shows an omitted-`sourceDelivery` caller throws that TypeError before the guard. I did not run a live packaged cron job in this read-only review.

## Solution Assessment

Yes, if maintainers accept runtime compatibility for version skew: sharing one planner avoids policy drift and the tests cover the missing-boundary modes. The safer alternative is a stricter rebuild or doctor remediation path, which is a maintainer product decision rather than a narrow code defect.
