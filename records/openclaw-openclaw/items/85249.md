## Summary

Keep open: this is a plausible fix for the isolated-cron crash, but the PR still duplicates the cron source-delivery policy table and only provides mocked/Docker test proof rather than a real stale-artifact, Gateway cron, or live transport run after the final head.

## What This Changes

The PR makes isolated cron execution tolerate missing `sourceDelivery`, rebuilds a fallback `SourceDeliveryPlan` from the cron job delivery config and resolved target, logs that fallback, and adds guard tests.

## Best Possible Solution

Move the normal cron source-delivery planner into one shared helper used by both preparation and fallback, then land with current-head real cron/Gateway or Telegram proof for the missing-`sourceDelivery` path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main dereferences `params.sourceDelivery` in the executor, so an undefined value from the lazy boundary throws before the cron run starts; I did not recreate the actual stale compiled artifact setup.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the guard is directionally right, but the fallback should call the same planner as normal cron delivery and needs real current-head runtime proof before merge.
