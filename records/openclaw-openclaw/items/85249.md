## Label Justifications

- `P1`: The linked regression reports all isolated cron jobs failing immediately in a shipped version, which is a broken scheduled agent workflow.
- `merge-risk: 🚨 compatibility`: The PR changes an internal executor boundary to tolerate stale callers but does not preserve the older caller's full parameter contract.
- `merge-risk: 🚨 message-delivery`: The fallback disables the message tool and drops source reply delivery mode when `sourceDelivery` is missing, which can suppress or alter cron announce delivery.

## Summary

Keep open: this targets a real isolated-cron crash path, but the compatibility fallback drops the legacy delivery policy and the PR only provides mock-based proof.

## What This Changes

The PR makes `sourceDelivery` optional in the isolated cron executor, adds undefined fallbacks and a warning, and adds a focused unit regression test file.

## Best Possible Solution

Preserve the current `SourceDeliveryPlan` contract for normal calls, and if version-skew compatibility is needed, reconstruct or honor the legacy `sourceReplyDeliveryMode` and `toolPolicy` fields with a real isolated-cron proof path.

## Reproduction Assessment

Yes from source for the version-skew case: an old caller can omit `sourceDelivery`, while current `run-executor.ts` dereferences it directly. I did not establish a live stale-build or installed-package reproduction in this read-only review.

## Solution Assessment

No. The guard avoids the immediate TypeError, but the fallback should preserve the legacy delivery policy instead of changing message-tool and source-reply behavior.
