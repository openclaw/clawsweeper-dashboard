## Summary

Keep open: this PR still addresses a live cron regression, and current main plus v2026.6.1 still dereference `params.sourceDelivery` without a guard. I found no blocking line-level patch defect, but the fallback is compatibility- and message-delivery-sensitive, so it needs maintainer acceptance before merge.

## What This Changes

The PR makes isolated cron execution tolerate missing `sourceDelivery` by rebuilding the cron source-delivery plan from job delivery config through a shared planner, with regression coverage for stale-caller delivery modes.

## Best Possible Solution

Land the shared planner and fallback if maintainers accept stale-artifact compatibility, while keeping cron announce delivery runner-owned with the message tool optional and duplicate suppression preserved.

## Reproduction Assessment

Yes. Current main and v2026.6.1 still dereference `params.sourceDelivery` before a guard, and the linked reports plus supplied terminal proof show omitted `sourceDelivery` produces the reported TypeError; I did not run a live cron locally in this read-only review.

## Solution Assessment

Yes, if maintainers accept the compatibility policy. The shared helper is the best code shape I found because it reuses current cron delivery semantics instead of sniffing legacy fields, while the alternative is an intentional fail-closed upgrade decision.
