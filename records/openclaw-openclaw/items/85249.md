## Summary

Keep open: current main and v2026.6.10 still have the unguarded isolated-cron sourceDelivery dereference, and this PR is the live candidate fix for the linked crash. I found no discrete line-level patch defect, but the compatibility fallback and message-delivery reconstruction still need maintainer acceptance before merge.

## What This Changes

The PR makes isolated cron executor sourceDelivery optional, reconstructs a missing plan from cron delivery config via a shared helper, rewires the normal planner to that helper, and adds guard/parity tests.

## Best Possible Solution

Land the shared-helper fallback only after maintainers accept the upgrade contract and, ideally, capture one live cron announce transport smoke to confirm no crash and no duplicate final delivery.

## Reproduction Assessment

Yes for the stale/missing-argument crash shape: current main and v2026.6.10 directly dereference params.sourceDelivery.sourceReplyDeliveryMode, and the PR body supplies terminal proof for omitted sourceDelivery. A clean normal cron path still passes sourceDelivery, so live end-to-end reproduction depends on version skew or stale artifacts.

## Solution Assessment

Yes, with maintainer acceptance: sharing the normal planner for fallback reconstruction is cleaner than legacy field sniffing and reduces future drift. The unresolved question is policy, not a narrow code defect: whether OpenClaw should tolerate this stale executor boundary at runtime.
