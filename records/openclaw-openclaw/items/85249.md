## Summary

Keep open: current main and v2026.6.6 still have the unguarded isolated-cron `sourceDelivery` dereference, and this PR is the open closing fix for that user-reported crash. ## What This Changes

The PR makes isolated cron executor `sourceDelivery` optional, adds a shared cron source-delivery planner/fallback helper used by `run.ts` and `run-executor.ts`, and adds regression/parity tests for missing `sourceDelivery`.

## Best Possible Solution

Merge the shared-planner fallback after maintainers accept current-job-config recovery semantics; optional live transport proof would reduce the remaining delivery-path uncertainty.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 still dereference `params.sourceDelivery`, and the PR body supplies Docker/host terminal output for a stale caller omitting `sourceDelivery`. I did not run a live packaged stale-module cron job in this read-only review.

## Solution Assessment

Yes, this is the best maintainable shape if maintainers accept runtime compatibility fallback because it shares the normal planner instead of preserving duplicated legacy field sniffing. The safer alternative is to reject runtime fallback and handle affected installs through rebuild or upgrade guidance, but that leaves the reported crash path unresolved.
