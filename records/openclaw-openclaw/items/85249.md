## Label Justifications

- `P1`: The PR targets a reported regression where isolated agentTurn cron jobs can fail before the agent starts.
- `merge-risk: 🚨 compatibility`: The patch makes a missing `sourceDelivery` executor argument a tolerated runtime fallback, which is upgrade-sensitive behavior.

## Summary

Keep open: the PR still fixes a real current-main and latest-release crash shape, and the branch now avoids the earlier drift risk by sharing the cron source-delivery planner between the normal path and missing-sourceDelivery fallback. There is no discrete line-level correctness defect from the patch, but the compatibility fallback and message-delivery semantics still need maintainer acceptance before merge.

## What This Changes

The PR makes isolated cron executor `sourceDelivery` optional, reconstructs a missing plan through a shared cron source-delivery helper, rewires the normal planner to use that helper, and adds guard/parity regression tests.

## Best Possible Solution

Land the shared-helper fallback only after maintainers accept the upgrade contract and, ideally, capture one live cron announce transport smoke to confirm no crash and no duplicate final delivery.

## Reproduction Assessment

Yes for the stale/missing-argument crash shape: current main and v2026.6.10 directly dereference `params.sourceDelivery.sourceReplyDeliveryMode`, and the PR body supplies terminal proof for omitted `sourceDelivery`. A clean normal cron path still passes `sourceDelivery`, so live end-to-end reproduction depends on version skew or stale artifacts.

## Solution Assessment

Yes, with maintainer acceptance: sharing the normal planner for fallback reconstruction is a cleaner fix than legacy field sniffing and reduces future drift. The unresolved question is policy, not a narrower mechanical code defect: whether OpenClaw should tolerate this stale executor boundary at runtime.
