## Summary

Keep open: current main and v2026.6.1 still dereference `sourceDelivery` without a guard, while this PR adds a plausible shared fallback plus current-head terminal proof. I did not find a blocking patch defect; the remaining concern is maintainer approval of the stale lazy-boundary compatibility policy.

## What This Changes

This PR makes `sourceDelivery` optional across the isolated cron executor boundary, rebuilds the cron source-delivery plan from job delivery config when absent, shares that planner with the normal run path, and adds focused guard tests.

## Best Possible Solution

Adopt one canonical cron source-delivery planner if maintainers want stale executor artifacts to keep working; otherwise choose an explicit fail-closed rebuild or restart path instead of implicit reconstruction.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.1 dereference `params.sourceDelivery.sourceReplyDeliveryMode` without a guard, and the PR body supplies current-head terminal proof for the omitted-`sourceDelivery` stale-caller path.

## Solution Assessment

Yes for the chosen compatibility policy: sharing the planner avoids drift and matches current cron delivery semantics. The unresolved question is whether maintainers want runtime reconstruction for stale executor artifacts or a fail-closed rebuild/restart contract.
