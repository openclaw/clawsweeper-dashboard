## Summary

Keep open: current main still has the source-level crash path, and this PR targets the right executor boundary, but it should not merge while the new fallback duplicates the canonical cron source-delivery policy and lacks current-head real behavior proof.

## What This Changes

The PR makes isolated cron source delivery optional at the executor boundary, adds a fallback plan helper, and adds regression tests for stale or missing `sourceDelivery` inputs.

## Best Possible Solution

Move the canonical cron source-delivery plan construction into one shared helper used by both normal preparation and the missing-`sourceDelivery` fallback, keep stale legacy fields ignored, and add current-head real cron/Gateway or Telegram proof before merge.

## Reproduction Assessment

Yes, source-level: current main dereferences `params.sourceDelivery.sourceReplyDeliveryMode`, and the linked regression describes a stale boundary where that value is missing. I did not reproduce the stale compiled artifact or ESM-cache setup in a live Gateway run.

## Solution Assessment

No. The guard belongs at the executor boundary, but the best fix should share the canonical cron source-delivery builder instead of copying the webhook/none/announce policy into a second fallback implementation.
