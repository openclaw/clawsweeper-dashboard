## Label Justifications

- `P1`: The linked user report says all isolated cron agent runs fail before the agent starts in recent releases, which is an urgent broken workflow.
- `merge-risk: 🚨 compatibility`: The PR changes the lazy-boundary compatibility behavior for stale callers and may not preserve the pre-refactor fields those callers still pass.
- `merge-risk: 🚨 message-delivery`: The fallback can alter message-tool forcing, source-reply mode, and target metadata, which can change where or how cron output is delivered.

## Summary

Keep open: the PR addresses a plausible P1 isolated-cron version-skew crash, but the fallback currently changes message-delivery policy for the exact stale-caller shape and the proof remains a Docker unit-test simulation rather than real cron runtime evidence.

## What This Changes

The PR makes `sourceDelivery` optional in the isolated cron executor, creates a fallback source delivery plan when it is missing, logs a warning, and adds focused regression tests.

## Best Possible Solution

Keep `SourceDeliveryPlan` required for normal callers, and add a narrow version-skew fallback that reconstructs the plan from legacy `sourceReplyDeliveryMode`, `toolPolicy`, `messageChannel`, and full resolved target fields when they are present.

## Reproduction Assessment

Yes from source for the version-skew case: a stale caller can omit `sourceDelivery` while current `run-executor.ts` dereferences it. I did not establish a live stale-build or installed-package reproduction in this read-only review.

## Solution Assessment

No. Guarding the crash is the right boundary, but this implementation should preserve the legacy delivery policy and full target instead of constructing a degraded `cron_none` fallback.
