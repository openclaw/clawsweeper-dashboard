## Summary

This is a focused, plausible fix for the isolated-cron crash, but it should remain open because it converts mixed build/module artifacts into a supported message-delivery compatibility path without an explicit cron-owner decision, and the supplied proof still does not demonstrate one real isolated announce run completing with exactly one channel post.

## What This Changes

The PR makes isolated cron execution reconstruct a missing source-delivery plan from the job and resolved target, centralizes cron source-delivery planning, and adds focused regression coverage.

## Best Possible Solution

Retain one required prepared SourceDeliveryPlan across the executor boundary and fix or fail clearly at the artifact/restart lifecycle boundary; if cron owners intentionally support mixed-version recovery, define the exact legacy input contract, reconstruct from its prepared facts, add a removal plan, and prove exact-once delivery on a real channel before merge.

## Solution Assessment

No. The exception is real and the implementation is internally coherent, but reconstructing current delivery policy inside the executor is not the safest owner boundary until cron maintainers explicitly choose mixed-artifact compatibility and exact-once transport behavior is proven.
