## Summary

Keep open: the crash guard is useful, but the latest fallback still drops legacy `sourceReplyDeliveryMode` on the real stale `executeCronRun` entry point, so skewed announce cron runs can change delivery behavior instead of only avoiding the TypeError.

## What This Changes

The PR makes isolated cron executor `sourceDelivery` optional, reconstructs a fallback delivery plan from resolved delivery and legacy tool policy, logs the fallback, and adds source-delivery guard tests.

## Best Possible Solution

Keep the new `SourceDeliveryPlan` contract for current callers, but make the missing-`sourceDelivery` compatibility path in `executeCronRun` preserve legacy `sourceReplyDeliveryMode` and `toolPolicy`, with regression coverage that invokes `executeCronRun` for stale announce and webhook callers.

## Reproduction Assessment

Yes, from source inspection and contributor Docker terminal proof. Current main has the failing dereference, and the pre-refactor lazy caller shape passed legacy fields without `sourceDelivery` across the executor boundary.

## Solution Assessment

No. The crash guard is needed, but the latest patch tests legacy mode at `createCronPromptExecutor` while real stale callers enter through `executeCronRun`, where that legacy mode is still lost.
