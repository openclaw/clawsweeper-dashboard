## Label Justifications

- `P1`: The linked regression reports all isolated cron jobs failing immediately in a shipped version, which is a broken scheduled agent workflow.
- `merge-risk: 🚨 compatibility`: The PR changes an internal executor boundary to tolerate stale callers without preserving the older caller's full parameter contract.
- `merge-risk: 🚨 message-delivery`: The missing-plan fallback disables the message tool and drops source reply delivery mode, which can suppress or alter cron announce delivery.

## Summary

The PR targets a real isolated-cron crash path, but its missing-`sourceDelivery` fallback changes the legacy delivery policy it is trying to tolerate, and the external real-behavior proof gate is still not met.

## What This Changes

The PR makes `sourceDelivery` optional in the isolated cron executor, adds fallback defaults and a warning, and adds a unit test for missing and present `sourceDelivery` cases.

## Best Possible Solution

Keep the `SourceDeliveryPlan` contract for normal calls, but if stale-caller compatibility is needed, reconstruct or honor the legacy `sourceReplyDeliveryMode` and `toolPolicy` fields and prove it with a real isolated-cron runtime path.

## Reproduction Assessment

Yes from source for the version-skew case: a stale caller can omit `sourceDelivery` while current `run-executor.ts` dereferences it directly. I did not establish a live stale-build or installed-package reproduction in this read-only review.

## Solution Assessment

No. The guard avoids the immediate TypeError, but the fallback should preserve the legacy delivery policy instead of changing message-tool and source-reply behavior.
