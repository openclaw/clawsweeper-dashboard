## Summary

Keep open: current main and the latest release still have the unguarded isolated-cron `sourceDelivery` dereference, while this PR is the focused candidate fix with strong source-boundary proof. The remaining blocker is maintainer acceptance of the stale-artifact fallback and its message-delivery semantics, not an obvious code defect.

## What This Changes

The branch makes isolated cron executor `sourceDelivery` optional, reconstructs it from cron delivery config through a shared helper, rewires the normal planner to use that helper, and adds guard/parity tests.

## Best Possible Solution

Keep this PR open as the candidate fix and merge it only after maintainers accept the stale-artifact fallback contract, with optional live transport proof if they want channel-level confidence before landing.

## Reproduction Assessment

Yes for source-level reproduction: current main/latest release still dereference `params.sourceDelivery.sourceReplyDeliveryMode`, and the PR proof shows omitted `sourceDelivery` throws the reported TypeError. I did not run a live packaged cron job in this read-only review.

## Solution Assessment

Yes if maintainers want runtime compatibility for stale executor artifacts: the shared helper is narrower than duplicating delivery policy or sniffing legacy fields. If maintainers reject that fallback contract, the safer alternative is rebuild/reinstall/doctor remediation rather than this runtime guard.
