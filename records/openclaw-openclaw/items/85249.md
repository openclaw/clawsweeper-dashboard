## Summary

Keep open: current main still requires `sourceDelivery` at the isolated cron executor boundary, while this PR adds a guarded shared planner that appears correct. The remaining blocker is maintainer acceptance of a runtime compatibility fallback on a cron message-delivery path, not a narrow contributor-facing defect.

## What This Changes

The PR makes isolated cron `sourceDelivery` optional at the executor boundary, rebuilds a missing plan through a shared cron source-delivery helper, and adds guard/parity regression tests.

## Best Possible Solution

Accept the shared fallback only with cron/runtime owner sign-off, keeping the parity and missing-`ok` duplicate-suppression tests as the guardrail.

## Reproduction Assessment

Yes. Current main dereferences `params.sourceDelivery.sourceReplyDeliveryMode`, and the PR body supplies terminal output showing the same stale-caller executor invocation throws on upstream/main and succeeds on the branch.

## Solution Assessment

Yes, if maintainers accept the compatibility premise. Sharing the normal cron planner is cleaner than duplicate fallback policy; the safer alternative is to fail closed with a rebuild/restart diagnostic instead of supporting mixed artifacts.
