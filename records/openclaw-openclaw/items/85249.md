## Summary

Keep open: the regression and crash proof are real, but current PR head still drops legacy delivery mode in the actual `executeCronRun` version-skew path, so it can change message delivery while preventing the TypeError.

## What This Changes

The PR makes isolated cron executor `sourceDelivery` optional, reconstructs a fallback delivery plan from legacy fields, logs the fallback, and adds source-delivery guard tests.

## Best Possible Solution

Share one compatibility builder, or otherwise build the `executeCronRun` fallback from all legacy delivery fields before merge, then cover stale announce and webhook callers through `executeCronRun`.

Yes, there is a high-confidence source reproduction path: call the current-main executor with `sourceDelivery` omitted and it dereferences `params.sourceDelivery.sourceReplyDeliveryMode`. The PR body also includes before/after Docker terminal proof for that compiled-dist path.

## Solution Assessment

No. The direction is right, but preserving legacy fields only in the direct `createCronPromptExecutor` fallback misses the real `executeCronRun` lazy boundary, so the fix should move or share that fallback and test the execute path.
