## Summary

Keep open: current main and latest release still have the unguarded isolated-cron `sourceDelivery` dereference, while this PR adds a focused current-job-config fallback with regression coverage. I found no concrete line-level correctness finding, but the compatibility and message-delivery semantics still need maintainer acceptance before merge.

## What This Changes

The PR makes isolated cron executor `sourceDelivery` optional at the lazy execution boundary, reconstructs missing plans through a shared cron source-delivery helper, and adds guard/parity tests.

## Best Possible Solution

Merge the shared-planner fallback once maintainers accept current-job-config recovery semantics; a live Telegram or Discord cron smoke would reduce the remaining delivery-path uncertainty.

## Reproduction Assessment

Yes at source level. Current main and `v2026.6.6` still dereference `params.sourceDelivery`, and the PR body supplies Docker/host terminal output for a stale caller omitting `sourceDelivery`; I did not run a live packaged stale-module cron job in this read-only review.

## Solution Assessment

Yes, this is the best maintainable fix if maintainers accept runtime compatibility fallback: it shares the normal cron planner instead of keeping duplicated legacy field sniffing. The safer alternative is to reject runtime fallback and handle affected installs through rebuild or upgrade guidance, but that leaves the reported crash path unresolved.
