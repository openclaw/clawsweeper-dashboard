## Summary

Keep open: the PR targets a real isolated-cron crash path, but the new fallback can force message delivery when legacy policy is absent and the external-contributor proof is still stale/simulated rather than current-head real cron behavior.

## What This Changes

The PR makes isolated cron `sourceDelivery` optional, reconstructs a fallback source-delivery plan from legacy fields, logs the fallback, and adds regression coverage for stale isolated cron callers.

## Best Possible Solution

Keep the guard, but make the no-policy fallback fail closed or mirror current cron source-delivery defaults, preserve only validated legacy fields, refresh current-head real cron proof, and get cron/source-delivery owner acceptance for any intentional legacy delivery semantics.

No high-confidence real version-skew reproduction was established. Current main source clearly crashes if `sourceDelivery` is omitted, and the linked issue plus PR proof support the symptom, but the actual stale compiled/ESM-cache path was not reproduced.

## Solution Assessment

No, not as submitted. The guard is the right direction, but the no-policy fallback should not force message delivery, and preserving pre-refactor announce semantics needs explicit owner acceptance.
