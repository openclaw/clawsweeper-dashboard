## Summary

Keep open: current main and the latest release still have the unguarded isolated-cron `sourceDelivery` dereference, while this PR is the linked fix candidate and I did not find a concrete patch-introduced defect. The remaining decision is whether maintainers want to support this version-skew fallback contract at runtime.

## What This Changes

The PR makes isolated cron source delivery optional at the executor boundary, adds a fallback planner derived from cron job delivery config, moves the normal cron planner into a shared helper, and adds regression tests.

## Best Possible Solution

Merge a single canonical guarded source-delivery planner after maintainer acceptance of the version-skew fallback contract and normal required checks.

## Reproduction Assessment

Partially yes from source and linked proof: current main has an unguarded `params.sourceDelivery.sourceReplyDeliveryMode` dereference, and the PR proof exercises an omitted-`sourceDelivery` stale-caller path. I did not run a live packaged cron job in this read-only review.

## Solution Assessment

Yes, if maintainers accept the fallback contract: the PR reuses the normal cron source-delivery planner for the fallback path instead of duplicating legacy-field sniffing. The alternative is not a code tweak but a maintainer choice to reject runtime skew support and fix packaging/restart behavior instead.
