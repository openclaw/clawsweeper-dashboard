## Review Findings

Overall confidence: 0.86

Full review comments:

- **[P1] Default missing legacy policy to current cron semantics:** `src/cron/isolated-agent/run-executor.ts:205`
  - body: When `sourceDelivery` is absent and no legacy `toolPolicy` is present, this fallback forces the message tool via `?? true`. Current main builds both `cron_none` and `cron_announce` plans with `messageToolForced: false`, so the guard can make an upgraded or version-skew cron run expose and force a message tool that current cron would not. Make both missing-policy fallbacks use current source-delivery defaults unless an actual legacy policy explicitly says otherwise.
  - confidence: 0.88
- **[P2] Keep stale announce fallback runner-owned:** `src/cron/isolated-agent/run-executor.ts:469-483`
  - body: A legacy `message_tool_only` value is converted into `owner: "message_tool"` with `directFallback: false`. Since current main changed announce cron plans to `owner: "direct_fallback"` with an unforced message tool, this can reintroduce model-owned visible delivery in the version-skew path and bypass the runner fallback contract. Preserve target/tool flags as needed, but keep announce fallback aligned with the current source-delivery plan or get an explicit compatibility decision.
  - confidence: 0.8
