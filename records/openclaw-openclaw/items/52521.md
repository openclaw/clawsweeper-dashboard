## Summary

Keep open: the underlying usage-attribution bug is source-reproducible, but this PR is not merge-ready because the patch buckets whole-session totals by a mutable latest-origin snapshot, changes session row metadata semantics, uses unvalidated origin keys, and still lacks contributor real behavior proof.

## What This Changes

The PR changes gateway `sessions.usage` to prefer `origin.provider` and `origin.chatType` over delivery metadata, with a regression test for swapped webchat and Telegram attribution.

## Best Possible Solution

Fix attribution with an explicit origin-versus-delivery contract: aggregate on validated attribution at the right granularity, preserve or expose delivery metadata clearly, and cover swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes, source-level: seed a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main uses the delivery channel for `byChannelMap` and `sessions[].channel`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The inversion handles the simple swapped-session case, but it is not the best final fix because it relabels whole-session summaries and row metadata from a mutable latest-origin snapshot without validating aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()` returns totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, mixed-origin bridged sessions can have historical usage moved to whichever origin was stored last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also returned as `sessions[].channel` and `sessions[].chatType`. Current UI filters, badges, rankings, and CSV export read those row fields directly, so the patch removes delivery metadata even though aggregate attribution is the reported bug.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: Needs attention: the diff promotes unvalidated `origin.provider` strings into session row metadata and aggregate channel keys.

Concerns:

- **[low] Validate origin provider before aggregate bucketing:** `src/gateway/server-methods/usage.ts:621`
  - body: `origin.provider` is derived from message context fields and `normalizeMessageChannel()` preserves unknown normalized strings; after this diff it becomes a row `channel` and `byChannelMap` key, allowing spoofed or high-cardinality buckets unless constrained or grouped.
  - confidence: 0.78
