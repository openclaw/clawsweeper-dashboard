## Summary

Keep open: the swapped usage-attribution bug is still plausible on current main, but this PR’s session-level priority flip has blocking correctness concerns and a concrete low-severity aggregate-key security concern before merge.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery metadata, with a swapped webchat/Telegram regression test.

## Best Possible Solution

Fix `/usage` with explicit origin-versus-delivery semantics: preserve or intentionally expose delivery metadata, compute aggregates at the right origin granularity, constrain aggregate keys, and cover swapped plus mixed-origin sessions in tests.

## Reproduction Assessment

Yes for the reported swapped webchat/Telegram scenario: current main visibly prefers delivery metadata, and the PR adds a focused fixture that would fail before the priority inversion. The mixed-origin regression risk is source-proven rather than covered by an existing executable test.

## Solution Assessment

No. The proposed priority inversion is small, but it relabels row metadata and still cannot accurately attribute whole-session totals when a session contains multiple origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: This line makes `origin.provider` the bucket for `loadSessionCostSummary()` totals, which cover the whole session/date range. Since `mergeOrigin()` overwrites origin metadata on later inbound turns, a mixed bridged session can relabel prior usage as the latest origin instead of splitting usage by turn.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The same origin-first `channel`/`chatType` values are emitted as `sessions[].channel` and `sessions[].chatType`. Current UI badges, filters, and CSV export read those row fields directly and do not have a separate delivery field, so this removes delivery-side metadata while only aggregate attribution is reported as wrong.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/usage.ts:621`
  - body: This is a user-facing `/usage` bug fix, but the PR changes only code and tests. Repository policy requires a `CHANGELOG.md` entry for user-facing fixes before merge.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it expands origin metadata into aggregate keys without validation or cardinality bounds.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:621`
  - body: The patch makes `origin.provider` the preferred `byChannelMap` key while channel normalization preserves unknown strings. If inbound origin metadata can be influenced by a caller or plugin, usage buckets can be spoofed or made high-cardinality unless keys are allowlisted or bounded.
  - confidence: 0.72
