## Summary

Keep this PR open for maintainer review: current main still has the swapped usage-attribution path, but this patch is too coarse because it applies a mutable session-level origin to whole-session totals and row metadata. A related broader PR #61315 is open, but it is not merged and does not make this PR safely closeable yet.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider`/`origin.chatType` over delivery `channel`/`chatType` and adds a swapped webchat/Telegram regression test.

## Best Possible Solution

Fix `/usage` with explicit origin-versus-delivery semantics: aggregate from the correct origin granularity, preserve or expose delivery metadata, validate aggregate keys, add regression coverage, and include a changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main still buckets by `storeEntry.channel` before `origin.provider`, so a webchat-origin session delivered through Telegram will aggregate under Telegram; the PR's fixture describes that exact path.

## Solution Assessment

No. The two-line inversion fixes the simplest swapped case, but it uses a mutable session origin for whole-session totals and row fields; a safer fix separates aggregate attribution from delivery row metadata and constrains channel keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Because stored origin metadata is merged and overwritten by later inbound turns, a mixed bridged session can move all historical usage into whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filters, query suggestions, and CSV export, so this removes delivery-side metadata while only aggregate attribution is reported as wrong.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/usage.ts:621`
  - body: This is a user-facing `/usage` behavior fix, but the diff only changes code and tests. Repository policy requires a `CHANGELOG.md` entry for user-facing fixes before merge.
  - confidence: 0.94

## Security Review

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it makes unvalidated origin provider strings preferred usage aggregation keys.

Concerns:
