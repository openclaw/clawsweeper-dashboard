## Summary

Keep open: the reported attribution bug is still present on current main, but this PR is not ready because it buckets whole-session totals by a mutable latest-origin snapshot, changes row metadata consumed by the UI, lacks aggregate-key validation, and has no contributor real behavior proof.

## What This Changes

The PR changes gateway `sessions.usage` to prefer `origin.provider` and `origin.chatType` over store delivery metadata and adds regression coverage for swapped webchat/Telegram usage attribution.

## Best Possible Solution

Define an explicit origin-versus-delivery usage contract: aggregate by validated origin at the correct granularity, preserve delivery metadata or expose it separately, and cover swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes, source-level: seed or inspect a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main resolves delivery metadata first and uses that value for `byChannelMap`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The two-line inversion handles simple swapped sessions, but it is not the best final fix because it relabels whole-session summaries and row metadata from a mutable latest-origin snapshot without validating aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: The session cost summary loaded above represents totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Because `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, a mixed-origin bridged session can have all historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filter options, `channel:`/`chat:` queries, overview rows, and CSV export, so this removes delivery-side metadata even though only aggregate attribution is the reported bug.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff uses unvalidated origin provider values as session row metadata and aggregate channel keys.

Concerns:
