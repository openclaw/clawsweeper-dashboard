## Summary

Keep open. Current main still has the reported delivery-first `/usage` bucketing, and this PR targets a real bug, but the proposed two-line inversion is not merge-safe because it relabels whole-session totals by a mutable origin snapshot, changes row metadata consumed by the UI/CSV, and uses unconstrained origin-provider keys; related PR #61315 is still open, so this PR is not terminally superseded.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a swapped webchat/Telegram regression test.

## Best Possible Solution

Fix `/usage` with explicit origin-versus-delivery semantics: compute channel aggregates at the correct origin granularity, preserve or explicitly expose delivery metadata for row consumers, constrain aggregate keys, and add regression coverage plus changelog.

## Reproduction Assessment

Yes. The swapped webchat/Telegram path is source-reproducible on current main because `sessions.usage` still prefers delivery metadata, and the PR adds a focused fixture for that scenario.

## Solution Assessment

No. The priority inversion is too coarse because the available summary is session-level and the stored origin is mutable; a safer fix should separate aggregate origin attribution from row delivery metadata and bound accepted channel keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but `entry.origin.provider` is a mutable session-level snapshot updated by later inbound metadata. In a bridged session with mixed origins, this buckets all historical usage under whichever origin was stored last instead of the actual turn origins.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filters, query suggestions, and CSV export, so this removes delivery-side metadata even though only aggregate attribution was reported as wrong.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/usage.ts:621`
  - body: This is a user-facing `/usage` behavior fix, but the diff only changes code and tests. Repository policy requires a `CHANGELOG.md` entry for user-facing fixes before merge.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it promotes unvalidated origin metadata into aggregate keys.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:621`
  - body: The patch makes `origin.provider` the preferred `byChannelMap` key while channel normalization preserves unknown strings. If inbound origin metadata is caller- or plugin-influenced, usage buckets can be spoofed or made high-cardinality unless keys are allowlisted or bounded.
