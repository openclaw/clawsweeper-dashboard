## Summary

Keep this PR open, but do not merge it as-is. The underlying usage-attribution bug is valid-looking, but the patch relabels whole-session totals and row metadata using a mutable origin snapshot, and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR changes gateway `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery metadata, with a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery contract: compute channel aggregates from the correct origin granularity, preserve or expose delivery row metadata, validate aggregate keys, and test swapped webchat/Telegram plus mixed-origin sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed sessions where `storeEntry.channel` and `origin.provider` are crossed; current main resolves the aggregate key from delivery `channel` first, and the PR's regression test models that setup, though I did not run tests in this read-only review.

## Solution Assessment

No. Inverting the priority is too coarse because session usage totals are whole-session summaries while `origin` is a mutable session snapshot; a safer fix should split aggregate attribution from row metadata and constrain channel keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `usage` represents the cost summary for the whole session/date range, but this line buckets the entire summary by the session-level `origin.provider`. Because `mergeOrigin()` overwrites provider/chatType when later inbound metadata arrives, a mixed-origin bridged session can be retroactively relabeled to whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The same origin-first values are also returned as `sessions[].channel` and `sessions[].chatType`. Current UI filtering, detail badges, and CSV export read those row fields directly, so this drops delivery channel/group metadata even though only aggregate attribution is being fixed.
  - confidence: 0.78
- **[P3] Constrain origin providers before aggregating:** `src/gateway/server-methods/usage.ts:621`
  - body: `deriveSessionOrigin()` normalizes unknown channel strings instead of allowlisting them, so preferring `origin.provider` lets arbitrary metadata become `byChannel` keys. Validate against known deliverable channels or bucket unknowns before using this value in the aggregate map.
  - confidence: 0.72

## Security Review
