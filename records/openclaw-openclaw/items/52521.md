- `P2`: This is a real usage-dashboard correctness bug with limited blast radius, not an urgent core runtime outage.

- `merge-risk: 🚨 security-boundary`: The diff promotes unvalidated origin-provider strings into returned metadata and aggregate keys, raising spoofing and high-cardinality concerns.

## Summary

Keep open: this targets a real usage-dashboard attribution bug, but the branch is not merge-ready because the two-line origin-first change also relabels row metadata, can misattribute whole mixed-origin sessions to the latest stored origin, promotes unvalidated origin strings into aggregate keys, and still lacks real behavior proof.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, plus a regression test for swapped webchat and Telegram attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery contract: validate aggregate keys, preserve or split delivery metadata for row consumers, and cover swapped plus mixed-origin sessions with tests and real proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: seed a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main uses delivery metadata for both `byChannelMap` and `sessions[].channel`.

## Solution Assessment

Is this the best way to solve the issue? No. The two-line inversion handles the simple swapped case, but the maintainable fix needs explicit origin versus delivery semantics, validated aggregate keys, and mixed-origin coverage.
