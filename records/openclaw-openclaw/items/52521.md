## Summary

Keep open, but not merge-ready: current main and the latest release still use delivery metadata for usage channel attribution, while this PR conflates aggregate attribution with row metadata and mutable session-level origin state. The branch also lacks after-fix real behavior proof and needs maintainer/API/security judgment before merge.

## What This Changes

The branch flips `sessions.usage` channel/chatType resolution to prefer origin metadata over delivery metadata and adds a swapped webchat/Telegram regression test.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery model: compute channel aggregates from a validated origin key at the right granularity, preserve or deliberately split row metadata, and cover swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes from source: current main and `v2026.5.22` still bucket usage by `storeEntry.channel` before `origin.provider`, and the linked report describes crossed webchat/Telegram usage where those values differ. I did not run tests because this was a read-only review.

## Solution Assessment

No as written: the PR addresses a real bug, but the shared `channel` local drives both aggregate buckets and returned row metadata, and `origin.provider` is a mutable session snapshot. A safer fix should separate validated aggregate attribution from delivery/origin row fields or explicitly document the API change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this changed line buckets all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, a mixed-origin bridged session can have all historical usage relabeled as whichever origin was seen last.
  - confidence: 0.88
- **[P1] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filter options, `channel:`/`chat:` queries, aggregate rebuilding, and CSV export, so this silently removes delivery-side metadata even though only the aggregate bucket needs origin attribution.
  - confidence: 0.84
- **[P2] Validate origin providers before using them as buckets:** `src/gateway/server-methods/usage.ts:621`
  - body: `origin.provider` is normalized but unknown channel strings are preserved, and this line now uses that value directly as an aggregate key. Constrain it to known channel ids or group invalid values before returning arbitrary channel buckets.
  - confidence: 0.78
