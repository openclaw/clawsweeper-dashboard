## Label Justifications

- `P2`: This is a real usage-dashboard correctness bug with limited blast radius, not an urgent core runtime outage.
- `merge-risk: 🚨 compatibility`: The PR changes the observable meaning of existing `sessions[].channel` and `sessions[].chatType` fields for UI/API consumers.
- `merge-risk: 🚨 session-state`: The patch buckets whole-session usage under mutable session origin metadata, which can mis-associate historical usage for mixed-origin sessions.
- `merge-risk: 🚨 security-boundary`: The diff promotes unvalidated origin-provider strings into returned metadata and aggregate keys, raising spoofing and high-cardinality concerns.

## Summary

Keep open: the usage-attribution bug is real and the closed reporter issue now points here, but this PR is not merge-ready because the two-line inversion changes row metadata semantics, buckets whole-session totals by a mutable origin snapshot, uses unvalidated origin keys, and still lacks real behavior proof.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat and Telegram attribution.

## Best Possible Solution

Fix usage attribution with an explicit origin-versus-delivery contract: aggregate by validated origin at the correct granularity, keep or split delivery metadata for row consumers, and cover swapped plus mixed-origin sessions with real proof.

## Reproduction Assessment

Yes, source-reproducible: seed a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main feeds delivery-first channel into both `byChannelMap` and `sessions[].channel`. I did not execute tests because this review is read-only.

## Solution Assessment

No. The two-line inversion handles the simple swapped case, but the maintainable fix needs explicit origin versus delivery semantics, validated aggregate keys, and mixed-origin coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: The usage summary feeding this block is a total for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, mixed-origin bridged sessions can have historical usage moved to whichever origin was stored last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also returned as `sessions[].channel` and `sessions[].chatType`. Current UI filters, badges, rankings, and CSV export read those row fields directly, so the patch removes delivery metadata even though aggregate attribution is the reported bug.
  - confidence: 0.79
