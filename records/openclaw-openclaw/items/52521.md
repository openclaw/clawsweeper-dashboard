## Summary

Keep open: the reported usage-attribution bug is valid, but this PR is not merge-ready because it changes row metadata semantics, can mis-bucket mixed-origin sessions, promotes unvalidated origin provider strings into aggregate keys, and still lacks after-fix real behavior proof.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat and Telegram attribution.

## Best Possible Solution

Fix usage attribution with an explicit origin-versus-delivery contract: preserve or expose delivery metadata for rows, aggregate by validated origin at the right granularity, and cover swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes, at source level: seed a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main uses the delivery channel for both `byChannelMap` and `sessions[].channel`. I did not run a live `/usage` proof because this review is read-only.

## Solution Assessment

No. The two-line inversion is too broad because it couples aggregate attribution to row metadata, trusts mutable session-level origin state, and does not validate aggregate keys; the safer fix separates origin attribution from delivery metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()` returns totals for the whole session/date range, but `entry.origin.provider` is a mutable session-level snapshot. In a mixed-origin bridged session, this line can retroactively move all historical usage into whichever origin was persisted last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The same origin-first value is emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those fields for badges, filters, and CSV export, so the patch removes delivery-side metadata even though the aggregate bucket is the part that needs attribution work.
  - confidence: 0.8
- **[P2] Validate origin providers before bucketing:** `src/gateway/server-methods/usage.ts:621`
  - body: `normalizeMessageChannel()` preserves unknown channel strings, so preferring `origin.provider` here can expose spoofed or high-cardinality bucket keys in `aggregates.byChannel`. Constrain the value to known/allowed channels or group invalid values before using it as an aggregate key.
  - confidence: 0.73
