## Summary

Keep open: current main still has the delivery-first usage aggregation path this PR targets, and this PR remains the canonical linked fix candidate. The proposed patch is not ready because it buckets whole-session summaries by a mutable origin snapshot, changes row metadata consumed by the UI, lacks aggregate-key validation, and has no contributor real behavior proof.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery metadata, with a regression test for swapped webchat/Telegram usage attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery contract: validate aggregate keys, attribute at the granularity supported by cost summaries, preserve delivery metadata or expose explicit fields, and add focused regression plus real dashboard proof.

## Reproduction Assessment

Yes, source-level: seed a `sessions.usage` store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main resolves delivery metadata first and uses that value for `byChannelMap`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The inversion covers the simple swapped-session case, but it is not the best final fix because it relabels whole-session summaries and row metadata from a mutable origin snapshot without validating aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()`/`loadSessionCostSummary()` return totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Because `mergeOrigin()` overwrites provider and chat type when later inbound metadata arrives, a mixed-origin bridged session can have all historical usage relabeled as whichever origin was stored last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The same origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. Current usage UI filters, detail badges, and CSV export consume those row fields directly, so this drops delivery-side metadata even though the aggregate bucket is the part that needs attribution work.
  - confidence: 0.79

## Security Review
