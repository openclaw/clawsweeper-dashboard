## Summary

Current main still has the delivery-first attribution bug and the linked report was closed in favor of this PR, but this branch should not merge as-is because it relabels whole-session totals from a mutable origin snapshot, changes row metadata semantics, lacks real behavior proof, and leaves a low-severity aggregate-key risk.

## What This Changes

The PR changes gateway `sessions.usage` to prefer `origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat and Telegram attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery usage contract: compute channel aggregates at the correct granularity, preserve or split delivery metadata for row consumers, validate aggregate keys, and cover swapped-channel plus mixed-origin sessions.

## Reproduction Assessment

Yes, source-level: seed or inspect a session store entry where `storeEntry.channel` differs from `origin.provider`; current main resolves delivery metadata first at `src/gateway/server-methods/usage.ts:1094` and uses that value for `byChannelMap`. I did not run tests in this read-only review.

## Solution Assessment

No. The PR targets the right symptom, but the current patch is not the best final fix because it buckets whole-session summaries by a mutable origin snapshot and changes row API semantics instead of separating attribution from delivery metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()` and `loadSessionCostSummary()` return totals for the whole session/date range, but `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives. In a mixed-origin bridged session, this line can retroactively move all historical usage into whichever origin was stored last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. Current usage UI filters, badges, details, and CSV export read those row fields directly, so this removes the delivery-side signal even though aggregate attribution is the reported bug.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff prefers unbounded `origin.provider` values as both session row metadata and the aggregate channel key, which needs validation or grouping before merge.
