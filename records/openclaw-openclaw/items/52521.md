## Summary

Keep this PR open, but do not merge it as-is: current main still has the reported delivery-first attribution path, while the PR’s two-line inversion creates whole-session attribution, row-metadata, proof, and low-severity aggregation-key concerns that need maintainer/API judgment.

## What This Changes

The PR changes gateway `sessions.usage` to prefer `origin.provider`/`origin.chatType` over stored delivery `channel`/`chatType`, and adds a swapped webchat/Telegram regression test.

## Best Possible Solution

Rework the fix around an explicit origin-versus-delivery usage contract: preserve or separately expose delivery metadata, validate aggregate keys, and cover swapped-channel plus mixed-origin session behavior in regression tests.

## Reproduction Assessment

Yes, source-level. Seed a session store entry where `storeEntry.channel` differs from `origin.provider`; current main resolves the delivery channel first at `src/gateway/server-methods/usage.ts:1094` and then buckets usage by that value.

## Solution Assessment

No. The two-line inversion is too broad because it applies a mutable session-level origin snapshot to whole-session totals and row metadata; a safer fix should separate aggregate attribution from delivery row fields and constrain aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary`/the cached summary path returns totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Because `mergeOrigin()` overwrites provider/chatType when later inbound metadata arrives, a mixed-origin bridged session can have historical usage relabeled as whichever origin was stored last.
  - confidence: 0.87
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for filters, badges, and CSV export, and the existing `origin` field contains origin metadata rather than the delivery route, so callers lose the delivery-side signal.
  - confidence: 0.8

## Security Review

Status: needs_attention
