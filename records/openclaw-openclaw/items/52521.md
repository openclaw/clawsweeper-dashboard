## Summary

Keep open: the reported usage-attribution bug is valid on current main, but this PR's two-line inversion changes row metadata semantics, can misbucket mixed-origin sessions, and promotes unvalidated origin strings without real behavior proof.

## What This Changes

The PR changes `sessions.usage` to prefer stored origin provider/chatType over delivery channel/chatType and adds a swapped webchat/Telegram regression test.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery contract: validate aggregate keys, preserve or split delivery metadata for row consumers, and cover swapped plus mixed-origin sessions with tests and real proof.

## Reproduction Assessment

Yes, from source: seed `sessions.usage` with a store entry whose delivery `channel` differs from `origin.provider`; current main uses the delivery channel for both `byChannelMap` and `sessions[].channel`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The inversion covers the simple swapped case, but the maintainable fix needs separate origin versus delivery semantics, validated aggregate keys, and mixed-origin coverage.

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this line buckets all totals by mutable `storeEntry.origin.provider`.   - confidence: 0.87

  - confidence: 0.81
