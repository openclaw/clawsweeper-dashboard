## Summary

Keep open: current main still has the source-level usage attribution bug, but this PR is not mergeable as-is because it buckets whole-session summaries by a mutable origin snapshot, changes row metadata semantics, lacks real behavior proof, and needs aggregate-key validation.

## What This Changes

The PR changes gateway `sessions.usage` to prefer `storeEntry.origin.provider`/`origin.chatType` over delivery `channel`/`chatType`, and adds a regression test for swapped webchat and Telegram attribution.

## Best Possible Solution

Fix the bug with an explicit origin-versus-delivery usage contract: compute channel aggregates at the right granularity, preserve or split row metadata intentionally, validate aggregate keys, and cover swapped-channel plus mixed-origin sessions.

## Reproduction Assessment

Yes, source-level: seed or inspect a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main resolves delivery metadata first and uses that value for `byChannelMap`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The two-line inversion addresses the swapped bucket symptom for simple sessions, but it is not the best final fix because it relabels whole-session summaries and row metadata from a mutable origin snapshot without validating aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid bucketing whole sessions by the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()` returns totals for the whole session/date range, but `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives. In a mixed-origin bridged session, this line can retroactively move all historical usage into whichever origin was stored last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. Current usage filters, badges, details, and CSV export read those row fields directly, so this removes the delivery-side signal even though aggregate attribution is the reported bug.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff prefers unbounded `origin.provider` values as session row metadata and aggregate channel keys, so validation or grouping is needed before merge.
