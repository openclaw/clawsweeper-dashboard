## Summary

Keep open. The underlying usage-attribution bug still exists on current main, but this PR’s two-line inversion is not merge-ready because it can relabel whole-session totals from a mutable latest-origin snapshot, changes existing row metadata semantics, lacks contributor real behavior proof, and raises a low-severity aggregate-key validation concern.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery metadata, with a regression test for swapped webchat/Telegram usage attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery contract: compute channel aggregates from validated attribution at the right granularity, keep row metadata semantics clear, and cover swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes, source-level: seed a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main resolves delivery metadata first and uses that value for `byChannelMap`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The inversion handles the simple swapped-session case, but it is not the best final fix because it relabels whole-session summaries and row metadata from a mutable latest-origin snapshot without validating aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole-session totals to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()`/`loadSessionCostSummary()` produce totals for a whole session/date range, but this line buckets the entire summary under `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the provider when later inbound metadata arrives, mixed-origin bridged sessions can have historical usage moved to whichever origin was stored last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The same origin-first `channel`/`chatType` values are returned as `sessions[].channel`/`chatType`. Current UI filters, badges, and CSV export read those row fields directly and do not recover delivery metadata from `origin`, so this removes information callers currently get while only aggregate bucket attribution needed to change.
  - confidence: 0.79

## Security Review
