## Summary

Keep open: this is a plausible usage-attribution bug, but the PR's two-line inversion is too broad because it applies a mutable session-level origin snapshot to whole-session totals and row metadata, lacks aggregate-key constraints, and still needs contributor real behavior proof.

## What This Changes

The branch changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat and Telegram attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery usage contract: compute channel aggregates at the right granularity, preserve or deliberately split delivery metadata for row consumers, constrain aggregate keys, and cover swapped-channel plus mixed-origin sessions.

## Reproduction Assessment

Yes, source-level: seed or inspect a session store entry where `storeEntry.channel` differs from `origin.provider`; current main resolves delivery metadata first at `src/gateway/server-methods/usage.ts:1094` and uses that value for `byChannelMap`. I did not run tests in this read-only review.

## Solution Assessment

No. The PR's direction targets the right symptom, but the current patch is not the best fix because it reuses a mutable session-level origin snapshot for whole-session totals and row metadata instead of defining separate attribution and delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` and the cache-backed path return totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Because `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, a mixed-origin bridged session can have historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. Existing usage UI filters, badges, details, and CSV export read those row fields directly, so this removes the delivery-side signal even though only aggregate attribution is being targeted.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff prefers unbounded `origin.provider` values as the session row and aggregate channel key, creating a low-severity spoofing and high-cardinality bucket risk.

Concerns:

- **[low] Validate origin provider before aggregate bucketing:** `src/gateway/server-methods/usage.ts:621`
  - body: `origin.provider` is derived from message context and `normalizeMessageChannel()` preserves unknown normalized strings; using it directly as `sessions[].channel` and the `byChannelMap` key can create spoofed or high-cardinality usage buckets unless constrained or grouped.
  - confidence: 0.78
