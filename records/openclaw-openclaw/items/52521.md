## Summary

Keep open: the linked usage-attribution bug still appears present on current main, but this PR should not merge as-is because it relabels whole-session totals using mutable session-origin metadata, changes row fields that UI consumers already use, and lacks real behavior proof.

## What This Changes

The branch changes gateway `sessions.usage` to prefer session origin provider/chatType over stored delivery channel/chatType and adds a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Rework the fix around an explicit origin-versus-delivery usage contract, with validated aggregate keys, preserved or separately exposed delivery metadata, and regression coverage for swapped-channel and mixed-origin sessions.

## Reproduction Assessment

Yes. A focused source-level reproduction can seed sessions where `storeEntry.channel` differs from `origin.provider`; current main line 1094 will bucket by delivery channel, though I did not run tests in this read-only review.

## Solution Assessment

No. The two-line inversion is too broad because it applies a mutable session-level origin snapshot to whole-session totals and row metadata; a safer solution should separate aggregate attribution from row delivery fields and validate aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()` returns totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Because `mergeOrigin()` overwrites provider/chatType when later inbound metadata arrives, a mixed-origin bridged session can have historical usage relabeled as whichever origin was stored last.
  - confidence: 0.87
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The same origin-first `channel`/`chatType` values are emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for filters, badges, and CSV export, so this removes the delivery-side metadata even though the aggregate bucket is the part that needs attribution work.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: Security review found a low-severity aggregation-key concern introduced by preferring unbounded `origin.provider` values for channel attribution.

Concerns:

- **[low] Validate origin provider before aggregate bucketing:** `src/gateway/server-methods/usage.ts:621`
  - body: `origin.provider` is derived from message context and `normalizeMessageChannel()` preserves unknown normalized strings; using it directly as `sessions[].channel` and the `byChannelMap` key can create spoofed or high-cardinality buckets unless the value is constrained or grouped.
  - confidence: 0.78
