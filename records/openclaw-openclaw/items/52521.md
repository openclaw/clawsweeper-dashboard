## Summary

Keep open: current main still has the delivery-first attribution path, but this PR should not merge as-is because it buckets whole-session usage by a mutable origin snapshot, relabels row metadata consumed by the UI/CSV, and leaves origin aggregate keys unconstrained.

## What This Changes

The PR makes `sessions.usage` prefer `storeEntry.origin.provider`/`origin.chatType` over delivery metadata and adds a swapped webchat/Telegram regression test.

## Best Possible Solution

Fix `/usage` with explicit origin-versus-delivery semantics: aggregate at the correct origin granularity, preserve or expose delivery metadata for rows, constrain channel keys, and add regression coverage plus changelog.

## Reproduction Assessment

Yes. The swapped webchat/Telegram path is source-reproducible on current main because `sessions.usage` still prefers delivery metadata, and the PR adds a focused fixture for that scenario.

## Solution Assessment

No. The two-line priority inversion is too coarse because the available summary is session-level and the stored origin is mutable, so it can misattribute mixed-origin sessions and change row metadata without an explicit API decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: This line uses the session's `origin.provider` for totals that cover the whole transcript/date range. Because `mergeOrigin()` overwrites origin metadata on later inbound turns, a mixed bridged session can move all historical usage into whichever origin was seen last instead of splitting by actual turn origin.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The same origin-first `channel`/`chatType` values are returned as `sessions[].channel` and `sessions[].chatType`. The UI uses those row fields for badges, filters, query suggestions, and CSV export, and there is no separate delivery field, so this removes delivery-side metadata while only aggregate attribution is reported as wrong.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/usage.ts:621`
  - body: This is a user-facing `/usage` behavior fix, but the diff only changes code and tests. Repository policy requires a `CHANGELOG.md` entry for user-facing fixes before merge.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it expands origin metadata into aggregate keys without validation or cardinality bounds.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:621`
