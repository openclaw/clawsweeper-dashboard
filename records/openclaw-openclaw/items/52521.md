## Label Justifications

- `P2`: This is a normal-priority usage dashboard attribution bug with limited blast radius but visible user impact.
- `merge-risk: 🚨 compatibility`: The diff changes the meaning of returned usage session row fields that existing UI/API consumers use for filters, badges, and CSV export.

- `merge-risk: 🚨 security-boundary`: The diff promotes unvalidated origin provider strings into public row fields and aggregate bucket keys.

## Summary

Keep open. The linked usage-attribution bug is still plausible on current main, but this PR is not merge-ready because the two-line origin-first change also changes returned row semantics, can retroactively bucket a mixed-origin session under the latest stored origin, promotes unvalidated origin strings into aggregate keys, and still lacks real after-fix behavior proof.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for crossed webchat and Telegram attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery contract: aggregate by validated origin at the right granularity, preserve or separately expose delivery row metadata, and cover swapped plus mixed-origin sessions with real proof.

## Reproduction Assessment

Yes from source inspection: seed a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main chooses delivery metadata first and uses that value for `aggregates.byChannel`. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No: the two-line shared-variable change is not the best fix because it couples aggregate attribution to row metadata and session-level origin snapshots. A safer solution should split origin aggregation from delivery row fields and validate the aggregate key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:1197`
  - body: `loadSessionCostSummaryFromCache()` returns totals for the whole session/date range, but this origin-first value is a mutable session-level snapshot. Because inbound metadata can overwrite `entry.origin.provider`, a mixed-origin bridged session can have all historical usage relabeled as whichever origin was stored last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on usage rows:** `src/gateway/server-methods/usage.ts:1197-1198`
  - body: The same origin-first `channel`/`chatType` values are emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI reads those row fields for `channel:`/`chat:` filters, badges, overview rows, and CSV export, so the PR removes the delivery-side metadata even though the aggregate bucket is the part that needs origin attribution.
  - confidence: 0.79
- **[P2] Validate origin providers before bucketing:** `src/gateway/server-methods/usage.ts:1197`
  - body: `normalizeMessageChannel()` preserves unknown normalized strings, but this change would prefer `origin.provider` as both the public row channel and the `byChannelMap` key. Constrain or group invalid origin providers before using them as aggregate keys to avoid spoofed or high-cardinality buckets.
  - confidence: 0.72
