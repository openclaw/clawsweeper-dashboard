## Summary

Keep this PR open, but do not merge it as-is: the reported attribution bug is still visible on current main, while this patch relabels whole-session totals from a mutable origin snapshot, changes row metadata semantics, lacks real behavior proof, and needs maintainer/API judgment.

## What This Changes

The PR changes gateway `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Define an explicit origin-versus-delivery usage contract: compute channel aggregates from the correct validated origin granularity, preserve or explicitly expose delivery metadata for row consumers, and cover swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes, source-level: seed or inspect a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main resolves delivery metadata first and uses that value for `byChannelMap`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The two-line inversion handles simple swapped sessions, but it is not the best final fix because it relabels whole-session summaries and row metadata from a mutable latest-origin snapshot without validating aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()` returns totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider on later inbound metadata, a mixed-origin bridged session can have all historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filter options, `channel:`/`chat:` queries, overview rows, and CSV export, so this removes delivery-side metadata even though the aggregate bucket is the part that needs origin attribution.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff uses unvalidated origin provider values as session row metadata and aggregate channel keys.

Concerns:

- **[low] Validate origin provider before aggregate bucketing:** `src/gateway/server-methods/usage.ts:621`

  - confidence: 0.78
