## Summary

Keep open. Current main still has the delivery-channel-first attribution path, but this PR’s origin-first swap is too broad because it relabels whole-session totals and row metadata while also using unconstrained origin values as aggregate keys.

## What This Changes

The PR changes `sessions.usage` to prefer `origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat and Telegram attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery usage contract: compute channel aggregates at the right attribution granularity, preserve or explicitly expose delivery metadata for row consumers, validate aggregate keys, and cover swapped-channel plus mixed-origin sessions in tests.

## Reproduction Assessment

Yes, source-level. Seed or inspect a session store entry where `storeEntry.channel` differs from `origin.provider`; current main resolves the delivery channel first at `src/gateway/server-methods/usage.ts:1094` and then uses that value for `byChannelMap`.

## Solution Assessment

No. The two-line inversion is too broad because it applies a mutable session-level origin snapshot to whole-session totals and row metadata; a safer fix should separate aggregate attribution from delivery row fields and constrain aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` and the cache-backed path return totals for the whole session/date range, but this line would bucket all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, a mixed-origin bridged session can have historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI reads those row fields for filter options, `channel:`/`chat:` queries, badges, and CSV export, so callers lose the delivery-side metadata even though only aggregate attribution is being targeted.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff introduces a low-severity aggregation-key concern by preferring unbounded `origin.provider` values for channel attribution.
