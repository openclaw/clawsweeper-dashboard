## Summary

Keep this PR open, but it is not merge-ready: it targets a real usage attribution bug, yet the two-line origin-first swap conflates aggregate attribution with row metadata and relies on mutable, unvalidated session-origin state.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider/chatType` over delivery `channel/chatType` and adds a regression test for crossed webchat/Telegram attribution.

## Best Possible Solution

Fix attribution with an explicit origin-versus-delivery model: compute validated channel aggregates at the right granularity, preserve or explicitly expose delivery metadata for row consumers, and cover swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes, at source level: seed a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main uses delivery metadata for both `byChannelMap` and `sessions[].channel`. I did not run tests because this review is read-only.

## Solution Assessment

No. The two-line origin-first swap is too broad because it changes row semantics and relies on mutable session-level origin state; a safer fix should separate validated aggregate attribution from delivery row metadata.

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()` returns totals for the whole session/date range, but this line buckets those totals by `storeEntry.origin.provider`.
