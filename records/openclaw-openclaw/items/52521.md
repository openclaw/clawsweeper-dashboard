## Summary

Keep open: the reported attribution bug is real on current main, but this PR is not ready because the session-level origin priority can relabel whole-session totals with a mutable latest origin and changes row metadata semantics without a clear API decision.

## What This Changes

The PR makes `sessions.usage` prefer `storeEntry.origin.provider`/`origin.chatType` over delivery `channel`/`chatType` and adds a swapped webchat/Telegram regression test.

## Best Possible Solution

Fix `/usage` by explicitly separating origin attribution from delivery metadata, computing channel aggregates at the correct granularity with bounded keys, and covering swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes. Source inspection on current main shows `sessions.usage` still prefers delivery metadata before `origin.provider`, so a swapped webchat/Telegram store entry will bucket under the delivery channel; the PR's regression fixture matches that path.

## Solution Assessment

No. The two-line inversion is too coarse because it buckets whole-session summaries by a mutable origin snapshot and changes row metadata consumers without settling the origin-versus-delivery API semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Because `mergeOrigin()` overwrites origin metadata on later inbound turns, a mixed bridged session can move all historical usage into whichever origin was seen last instead of splitting by actual turn origin.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The same origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. Current UI filters, badges, and CSV export consume those row fields directly and have no separate delivery field, so this removes delivery-side metadata while only aggregate attribution is reported as wrong.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/usage.ts:621`
  - body: This is a user-facing `/usage` behavior fix, but the diff only changes code and tests. Repository policy requires a `CHANGELOG.md` entry for user-facing fixes before merge.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it promotes mutable origin metadata into aggregate keys without validation or bounds.

Concerns:
