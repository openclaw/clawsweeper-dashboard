## Summary

Keep open: the linked usage-attribution bug is valid, but this PR’s session-level origin swap can relabel whole mixed-origin sessions, changes row metadata consumed by the UI, and still needs real behavior proof before merge.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Fix attribution with an explicit origin-versus-delivery contract: aggregate at the correct origin granularity, preserve or add delivery metadata for row consumers, constrain aggregate keys, and cover swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main picks delivery `storeEntry.channel` before origin for the channel bucket, matching the linked webchat/Telegram swap report. I did not execute tests because this was a read-only checkout review.

## Solution Assessment

No. The PR addresses the obvious priority inversion, but the current patch is not the best final solution because it applies a mutable session-level origin to whole-session totals and row metadata instead of modeling origin and delivery separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:1094`
  - body: `loadSessionCostSummaryFromCache()` is merged into one session-level usage summary before a single channel key is selected. Since `mergeOrigin()` overwrites provider/chatType on later inbound metadata, this origin-first line can retroactively move all historical usage in a mixed-origin bridged session into whichever origin was recorded last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:1094-1095`
  - body: The same origin-first `channel` and `chatType` values are emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those fields for badges, filter options, query filters, and CSV export, so callers lose the delivery-side metadata even though only the aggregate bucket needs origin attribution.
  - confidence: 0.78
- **[P3] Validate origin provider before bucketing usage:** `src/gateway/server-methods/usage.ts:1094`
  - body: `origin.provider` is normalized but unknown channel strings are preserved, and this patch promotes that value into `byChannelMap` and row output. Validate against known channel ids or cap unknown buckets before using origin metadata as an aggregate key.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The patch promotes origin-derived provider strings into primary usage buckets without validation or cardinality control.

Concerns:

- **[low] Validate origin-derived channel keys:** `src/gateway/server-methods/usage.ts:1094`
  - body: `normalizeMessageChannel` preserves unknown strings, so preferring `origin.provider` can create spoofed or high-cardinality `byChannel` keys when origin metadata is untrusted or noisy.
  - confidence: 0.72
