## Summary

Keep open: current main still has the reported attribution bug, but this PR is not merge-ready because it changes public row semantics, can mis-bucket mixed-origin session history, promotes unvalidated origin keys, and lacks real behavior proof.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat and Telegram attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery contract: validate aggregate keys, preserve or split delivery metadata for row consumers, and cover swapped plus mixed-origin sessions with tests and real proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main uses delivery metadata for `byChannelMap` and `sessions[].channel`.

## Solution Assessment

No. The two-line inversion addresses the simple swapped-session case, but the maintainable fix needs explicit origin versus delivery semantics, validated aggregate keys, and mixed-origin coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()` returns totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, mixed-origin bridged sessions can have historical usage moved to whichever origin was stored last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also returned as `sessions[].channel` and `sessions[].chatType`. Current UI filters, badges, rankings, and CSV export read those row fields directly, so the patch removes delivery metadata even though aggregate attribution is the reported bug.
  - confidence: 0.79
