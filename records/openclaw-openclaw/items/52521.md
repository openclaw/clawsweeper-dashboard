## Summary

Keep open. Current `main` still has delivery-channel-first attribution, and this PR targets a valid-looking `/usage` bug, but the proposed two-line inversion is not safe to merge because it also changes row metadata and can misattribute mixed-origin sessions while adding unconstrained origin-derived aggregate keys.

## What This Changes

The PR changes `sessions.usage` so `origin.provider` and `origin.chatType` override delivery `channel` and `chatType`, and adds a regression test for swapped webchat/Telegram sessions.

## Best Possible Solution

Fix `/usage` with explicit origin-vs-delivery semantics: preserve or intentionally rename row metadata, compute channel aggregates from the correct origin granularity, constrain aggregate keys, and cover both swapped webchat/Telegram sessions and mixed-origin long-lived sessions in tests.

## Reproduction Assessment

Yes for the reported swapped webchat/Telegram scenario: current main visibly prefers delivery metadata, and the PR adds a focused fixture that would fail before the priority inversion. The mixed-origin regression risk is supported by source inspection rather than an existing executable test.

## Solution Assessment

No. The proposed priority inversion is small, but it is not the best fix because it relabels row metadata and still cannot accurately attribute whole-session totals when a session has multiple origins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this changed line buckets all of those totals by `storeEntry.origin.provider`. Because `mergeOrigin()` overwrites the saved provider on later inbound metadata, a mixed-origin bridged session can move all prior usage into whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. Current usage UI filters, badges, and CSV export consume those row fields directly, so the PR removes delivery-side metadata even though only aggregate attribution is the reported bug.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it expands origin metadata into aggregate keys without validation or cardinality bounds.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:621`
