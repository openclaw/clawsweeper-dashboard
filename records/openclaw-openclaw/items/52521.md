## Summary

Keep open: the underlying usage-attribution bug is still visible from current `main`, but this PR is not merge-ready because it changes both aggregate attribution and row metadata using a mutable, unvalidated session-level origin snapshot and still lacks contributor real behavior proof.

## What This Changes

The PR changes gateway `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with regression coverage for swapped webchat and Telegram usage attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery usage contract: compute channel aggregates from the right granularity, validate aggregate keys, preserve or separately expose delivery metadata, and cover swapped plus mixed-origin sessions.

## Reproduction Assessment

Yes, source-level: seed or inspect a session store entry where `storeEntry.channel` differs from `storeEntry.origin.provider`; current `main` resolves delivery metadata first and uses that value for `byChannelMap`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The two-line inversion handles simple swapped sessions, but it is not the best final fix because it relabels whole-session summaries and row metadata from a mutable latest-origin snapshot without validating aggregate keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummaryFromCache()` returns totals for the whole session/date range, but this line would bucket all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, a mixed-origin bridged session can have all historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel` and `chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. Usage filters, detail badges, overview metadata, and CSV export consume those row fields directly, so this removes the delivery-side signal even though the aggregate bucket is the reported bug.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: Needs attention: the diff uses unvalidated origin provider values as session row metadata and aggregate channel keys.

Concerns:

- **[low] Validate origin provider before aggregate bucketing:** `src/gateway/server-methods/usage.ts:621`
