## Summary

Keep open. The PR adds a useful Plugin SDK hook seam that current main does not have, but the latest patch is not merge-ready because the new extractor can report a different path than the `apply_patch` executor will actually apply, which is a policy-bypass risk for consumers of `derivedPaths`.

## What This Changes

The PR adds host-derived `apply_patch` target-path extraction, exposes `derivedPaths` on trusted-policy and `before_tool_call` events, updates Plugin SDK docs/API baseline/changelog, and adds focused unit, contract, and e2e coverage.

## Best Possible Solution

Keep the additive `derivedPaths` seam, but make the extractor mirror the executor's hunk-header path semantics exactly, add regression coverage for extra spaces after apply_patch markers, and land only after the focused parser/hook tests plus API baseline and lint checks are green.

## Reproduction Assessment

Yes. A patch hunk such as `*** Add File:  src/new.ts` is enough to compare: current executor parsing preserves the extra leading space in the target path after slicing the trimmed header, while the PR extractor trims it away before exposing `derivedPaths`.

## Solution Assessment

No, not yet. The high-level seam is the right narrow direction, but the parser must preserve the same target path the executor will use; a shared marker reader or an executor-mirroring `line.trim().slice(...)` helper would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve marker path bytes when deriving paths:** `src/agents/apply-patch-paths.ts:42`
  - body: `pushPath` trims the extracted path before publishing `derivedPaths`, but `apply_patch` parses the header with `line.trim()` and then slices after the marker, preserving any extra leading spaces after the marker as part of the actual filename. A patch like `*** Add File:  src/new.ts` will be reported to trusted policies as `src/new.ts` but applied to ` src/new.ts`, so an allowlist policy that trusts `derivedPaths` can approve the wrong path. Please mirror the executor's header parsing and add a regression test for extra spaces after the marker.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces a path-policy metadata mismatch that can mislead trusted and before_tool_call policy hooks.

Concerns:

- **[high] Derived paths can differ from applied paths:** `src/agents/apply-patch-paths.ts:42`
  - body: The extractor trims the raw marker path before exposing it to policy hooks, while the executor preserves additional leading spaces after the marker as part of the filename. Policies that allow `src/**` from `derivedPaths` could approve a patch that actually targets ` src/**`.
  - confidence: 0.88
