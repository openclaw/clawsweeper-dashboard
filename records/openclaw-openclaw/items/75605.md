## Summary

Keep open. Current main does not already provide the requested `derivedPaths` hook metadata, and the PR is a plausible narrow SDK seam, but the latest head still has two execution/metadata mismatches in the security-sensitive path facts that trusted policies may consume.

## What This Changes

The PR adds host-derived `apply_patch` target-path extraction and a parser registry that attaches `derivedPaths` to trusted policy and `before_tool_call` hook events, plus docs, changelog, SDK baseline, and tests.

## Best Possible Solution

Keep the additive `derivedPaths` seam, but derive paths from the same payload bytes and sandbox resolver used by `apply_patch` execution, or omit `derivedPaths` where the host cannot map targets with the same semantics. Add regression coverage for extra marker payload whitespace and sandbox container or bind-mount paths.

## Reproduction Assessment

Yes. The mismatch is reproducible by comparing current main `parseOneHunk` on `*** Add File:  src/new.ts` with the PR extractor's `raw.trim()`, and by comparing sandboxed execution through `sandbox.bridge.resolvePath` with PR derivation that only receives `cwd`.

## Solution Assessment

No. The feature direction is narrow and maintainable, but this implementation is not the best solution until the metadata and execution target semantics match or the metadata is intentionally absent for unsupported resolution contexts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve marker payload bytes for derived paths:** `src/agents/apply-patch-paths.ts:56-61`
  - body: `normalizePatchPath` trims the marker payload before resolving it. The executor trims the whole header and then slices after the marker, so `*** Add File:  src/new.ts` applies to a path with a leading space while `derivedPaths` reports `src/new.ts`. Preserve the sliced payload bytes before resolution so policies do not evaluate a different file from the one applied.
  - confidence: 0.93
- **[P2] Resolve sandboxed paths through the apply_patch bridge:** `src/agents/pi-tools.ts:737`
  - body: Passing only `cwd: sandboxRoot ?? workspaceRoot` makes derivation use host cwd normalization, but sandboxed `apply_patch` execution resolves through `sandbox.bridge.resolvePath`. Container or bind-mount inputs such as `/workspace/src/new.ts` can be reported as the container path while execution writes the mapped host path. Use the same bridge/resolver for derivation, or omit `derivedPaths` for sandboxed calls.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: Security review needs attention because trusted-policy path metadata can still disagree with the actual `apply_patch` target set.

Concerns:

- **[high] Marker payload trimming changes policy targets:** `src/agents/apply-patch-paths.ts:56`
  - body: The extractor trims payload bytes after the marker, while the executor preserves leading payload space after slicing the trimmed header. A policy can therefore review one path while `apply_patch` writes another.
  - confidence: 0.92
- **[medium] Sandbox bridge targets are not mirrored:** `src/agents/pi-tools.ts:737`
  - body: Sandboxed execution resolves through the sandbox FS bridge, including container and bind-mount paths, but the new derivation path only receives a cwd. Policies consuming `derivedPaths` can see a different path than the execution target.
  - confidence: 0.86
