## Summary

Keep open. The latest force-push fixes the two earlier ClawSweeper findings for same-path rewrites and marker-like hunk body lines, but the trusted-policy event builder can still carry stale `derivedPaths` when rewritten params no longer derive any paths, which is risky for policy consumers.

## What This Changes

The PR adds host-derived `apply_patch` target paths to plugin `before_tool_call` and trusted-policy events, plus parser and hook tests and generated SDK API baseline hashes.

## Best Possible Solution

Keep the host-owned derivation seam, but build trusted-policy events from a clean event shape so stale `derivedPaths` cannot survive any params rewrite. Then document the new public hook field and add the required plugin SDK changelog entry before the draft becomes merge-ready.

## Reproduction Assessment

Yes. A focused test with one trusted policy rewriting an `apply_patch` event from a real path to `{ input: "not a patch" }` or `{}` and a later policy reading `event.derivedPaths` would show the original path list still present.

## Solution Assessment

No. The direction is sound and narrower than the old consolidated stack, but this head still does not make `derivedPaths` exactly match the params seen by every trusted policy; constructing the event without prior derived metadata is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clear derived paths when rewrites remove targets:** `src/plugins/trusted-tool-policy.ts:22-24`
  - body: `buildEvent` starts from the original event, which can already contain `derivedPaths`, and then only spreads recomputed metadata when the current params derive paths. If an earlier trusted policy rewrites an `apply_patch` call to params with no recognized targets, later policies see rewritten `event.params` with the old path list. Build from an event shape that omits previous derived metadata, or explicitly clear it when derivation is empty.
  - confidence: 0.9
- **[P3] Document the new hook metadata field:** `src/plugins/hook-types.ts:405-412`
  - body: This adds a public `before_tool_call` event field, but the plugin hook docs still list only `toolName`, `params`, `runId`, and `toolCallId`, and there is no changelog entry. Plugin authors need the docs and changelog to learn when `derivedPaths` is available and what absence means.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR improves policy metadata, but one stale-metadata edge can still mislead trusted policy checks after a params rewrite removes all derivable paths.

Concerns:

- **[medium] Stale paths can survive no-target rewrites:** `src/plugins/trusted-tool-policy.ts:22`
  - body: Because the trusted-policy event builder spreads the original event before recomputed metadata, a rewrite from a path-bearing patch to params with no derived targets leaves the old `derivedPaths` visible to later policies. Policy gates treating this field as authoritative could approve, block, or request approval for the wrong files.
  - confidence: 0.88
