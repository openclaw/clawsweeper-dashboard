## Summary

Keep this PR open. Current main does not have the proposed derived-path hook metadata, and the PR is a useful additive Plugin SDK direction, but the current patch still publishes security-sensitive `derivedPaths` that can disagree with the targets actually used by `apply_patch`.

## What This Changes

The PR adds host-derived `apply_patch` target-path metadata to trusted tool policy and `before_tool_call` hook events, with docs, SDK API baseline updates, changelog text, and focused tests.

## Best Possible Solution

Keep the additive `derivedPaths` seam, but make path derivation share or exactly mirror the `apply_patch` executor's marker payload parsing and sandbox path resolution. Add regressions for extra marker payload spaces and sandbox container/bind paths, then rerun the focused parser, hook contract, API baseline, lint, and whitespace checks.

## Reproduction Assessment

Yes. The marker mismatch is reproducible by comparing `line.trim().slice(marker.length)` with the PR extractor's `raw.trim()` for `*** Add File:  src/new.ts`; the sandbox mismatch follows from executor source using `sandbox.bridge.resolvePath` while the PR deriver receives only a cwd.

## Solution Assessment

No. The seam is narrow and maintainable, but this implementation is not the best fix until derived metadata is produced by the same target semantics as execution or omitted when the host cannot derive it authoritatively enough for policy use.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve apply_patch marker path payloads:** `src/agents/apply-patch-paths.ts:55-60`
  - body: `normalizePatchPath` trims the marker payload before resolving it. The executor trims the whole header and then slices after the marker, so `*** Add File:  src/new.ts` applies to `<cwd>/ src/new.ts` while `derivedPaths` reports `<cwd>/src/new.ts`. Preserve the sliced payload bytes before resolving so trusted policies do not evaluate a different file from the one applied.
  - confidence: 0.93
- **[P2] Resolve sandbox derived paths through the bridge:** `src/agents/pi-tools.ts:737`
  - body: Passing only `cwd: sandboxRoot ?? workspaceRoot` makes the deriver use host cwd normalization, but sandboxed `apply_patch` execution resolves paths through `sandbox.bridge.resolvePath`. Container or bind-mount inputs such as `/workspace/src/new.ts` can therefore be reported as the container path while execution touches the mapped host path. Use the same bridge/resolver for derivation, or omit `derivedPaths` for sandboxed calls.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces trusted-policy path metadata that can still disagree with the actual `apply_patch` target set.

Concerns:

- **[high] Marker payload trimming changes policy targets:** `src/agents/apply-patch-paths.ts:55`
  - body: The extractor trims payload bytes after the marker, while the executor preserves leading payload space after slicing the trimmed header. A policy can therefore review one path while `apply_patch` writes another.
  - confidence: 0.92
- **[medium] Sandbox bridge targets are not mirrored:** `src/agents/pi-tools.ts:737`
  - body: Sandboxed execution resolves through the sandbox FS bridge, including container mount paths, but the new derivation path only receives a cwd. Policies consuming `derivedPaths` can see container paths instead of the mapped execution targets.
  - confidence: 0.84
