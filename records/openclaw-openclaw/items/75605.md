## Summary

Keep open. Current main does not implement the new `derivedPaths` hook metadata, and the proposed patch has concrete correctness and policy-safety blockers before it can merge.

## What This Changes

The PR adds a host-owned `apply_patch` target-path extractor, a host tool-param parser registry, `derivedPaths` on trusted-policy and `before_tool_call` events, and focused tests.

## Best Possible Solution

Keep the host-owned derivation seam, but compute path metadata from the exact params each trusted policy and normal hook receives. The extractor should follow the apply-patch hunk grammar closely enough to ignore body/context lines, and the public Plugin SDK change should include contract tests plus a changelog entry.

## Reproduction Assessment

Yes. A targeted test with one trusted policy rewriting an `apply_patch` input and a later policy or normal hook reading `derivedPaths` would show stale paths from the original params. A parser test with an update-hunk context line containing `*** Add File: fake.ts` would show the false target extraction.

## Solution Assessment

No. Host-derived file facts are the right product direction, but this patch is not yet the best implementation because it decouples `derivedPaths` from policy-adjusted params and treats patch body text as hunk headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recompute derived paths after policy rewrites:** `src/agents/pi-tools.before-tool-call.ts:494`
  - body: `derivedPaths` is computed once from the original params and then reused for trusted policies and normal hooks. Since trusted policies can replace `params`, later policies and hooks can receive rewritten params paired with stale paths from the old patch, which can make policy decisions about the wrong files.
  - confidence: 0.9
- **[P2] Ignore patch body lines when extracting targets:** `src/agents/apply-patch-paths.ts:67-68`
  - body: The extractor trims every line before matching patch markers. A valid update hunk can contain a context/body line whose text starts with `*** Add File:` after the leading hunk prefix, and this code would report that fake path as a touched target.
  - confidence: 0.87
- **[P3] Add the required Plugin SDK changelog entry:** `src/plugins/hook-types.ts:405`
  - body: This adds a public `before_tool_call` event field that plugin authors can consume, but the PR does not update `CHANGELOG.md`. User-facing Plugin SDK changes need a changelog entry before merge.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds path metadata intended for policy decisions, but the metadata can diverge from the actual params or include fake targets from patch body text.

Concerns:

- **[medium] Stale path metadata can mislead policy checks:** `src/agents/pi-tools.before-tool-call.ts:494`
  - body: Because `derivedPaths` is computed before trusted policies run, a later policy can inspect adjusted params while still receiving paths from the original patch. Any policy treating `derivedPaths` as authoritative could allow, block, or require approval for the wrong files.
  - confidence: 0.86
- **[medium] Marker-like patch body text can create fake targets:** `src/agents/apply-patch-paths.ts:67`
  - body: The line-trimming extractor can classify update-hunk content as a hunk header. A policy that trusts these derived paths may make a decision based on files the patch does not actually touch.
  - confidence: 0.8
