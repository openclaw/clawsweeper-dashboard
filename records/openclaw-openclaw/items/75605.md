## Summary

Keep open. Current main does not implement the new host-derived `derivedPaths` surface, and this draft PR is the active fresh-main replacement for a closed predecessor slice, but the proposed patch has blocking correctness issues in the path metadata it exposes to tool policy hooks.

## What This Changes

The PR adds an `apply_patch` target-path extractor, a small host tool-param parser registry, `derivedPaths` on `before_tool_call` events, and focused parser/hook tests.

## Best Possible Solution

Keep the host-owned derivation seam, but make `derivedPaths` match the exact params each trusted policy and normal hook receives, and make extraction grammar-aware enough to ignore patch body text. Add focused tests for trusted-policy rewrites, later-policy inspection, normal hook events after rewrites, and marker-like context lines; include the required user-facing plugin SDK changelog entry before the PR leaves draft.

## Reproduction Assessment

Yes. A targeted test with one trusted policy rewriting an `apply_patch` input and a later policy or hook reading `derivedPaths` would show the path list still reflects the original params. A second parser test with an update-hunk context line like ` *** Add File: fake.ts` would show a non-target path being reported.

## Solution Assessment

No. Host-derived path metadata is the right direction, but this implementation is not yet the narrowest maintainable solution because it decouples the metadata from policy-adjusted params and scans patch body lines as headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recompute derived paths after trusted policy rewrites:** `src/agents/pi-tools.before-tool-call.ts:494`
  - body: `derivedPaths` is calculated once from the original params, then reused for trusted-policy and normal `before_tool_call` events. Trusted policies can return replacement `params`, and the current runner intentionally feeds those adjusted params to later policies; after a rewrite, policy/hook consumers can see new `event.params` with stale paths from the old patch. Recompute from the adjusted params for each policy/hook event, or omit the field when params change.
  - confidence: 0.9
- **[P2] Ignore patch body lines when extracting targets:** `src/agents/apply-patch-paths.ts:67-68`
  - body: The extractor trims every line before checking for hunk markers. A valid update hunk can include a context line whose content starts with `*** Add File:` after the leading context space; trimming that space records a path the patch does not touch, so plugins relying on host-derived paths can make policy decisions on fake targets. Track hunk headers instead of scanning trimmed body lines.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds path metadata intended for trusted tool policy decisions, but that metadata can diverge from the actual `apply_patch` params after policy rewrites.

Concerns:

- **[medium] Stale path metadata can mislead policy checks:** `src/agents/pi-tools.before-tool-call.ts:494`
  - body: Because `derivedPaths` is computed before trusted policies run, a later policy can inspect adjusted params while still receiving paths from the original patch. Any bundled policy that treats `derivedPaths` as authoritative could allow, block, or require approval for the wrong files.
  - confidence: 0.86
