## Summary

Keep open: current main does not contain the new `derivedPaths` hook metadata, so the PR is still relevant, but the current head still has blocking `apply_patch` parser mismatches before this metadata can be trusted by policy hooks.

## What This Changes

The PR adds host-derived `apply_patch` target path metadata (`derivedPaths`) for trusted policy and `before_tool_call` hooks, with parser/helper tests, hook docs, API baseline, and changelog updates.

## Best Possible Solution

Keep the additive `derivedPaths` contract, but make the host extractor share or exactly mirror the executor’s `apply_patch` parsing and path-resolution semantics before exposing the values to trusted policy and plugin hooks. Add regressions for extra marker payload spaces and marker-looking update context lines, then land with the focused parser, hook, SDK API, docs, lint, and exact-head CI checks green.

## Reproduction Assessment

Yes. `*** Add File:  src/new.ts` demonstrates the payload-space mismatch, and an update hunk containing `  *** Delete File: src/dead.ts` demonstrates the update-body/header mismatch against current main’s `apply_patch` parser.

## Solution Assessment

No. The proposed seam is the right narrow product direction, but the current implementation is not the best fix until `derivedPaths` uses the same executable target semantics as `apply_patch`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mirror apply_patch path payload parsing:** `src/agents/apply-patch-paths.ts:47`
  - body: `normalizePatchPath` trims the marker payload before emitting `derivedPaths`, but the executor trims the full header and then slices after the marker. For `*** Add File:  src/new.ts`, execution targets ` src/new.ts` while the hook reports `src/new.ts`, so a trusted policy can approve a different path than the one applied. Mirror or share the executor parsing before publishing this metadata.
  - confidence: 0.92
- **[P2] Treat indented update body markers as context:** `src/agents/apply-patch-paths.ts:147-149`
  - body: The update scan stops when a two-space-indented marker-looking line is normalized as a header, but `apply_patch` only ends an update on a raw line that starts with `***`. A line like `  *** Delete File: src/dead.ts` after a hunk is context the executor will keep in the update, while `derivedPaths` reports a delete target that is not touched. Align this scan with `parseOneHunk` and cover the context-line case.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR introduces trusted-policy path metadata that can still disagree with the actual `apply_patch` targets.

Concerns:

- **[high] Derived paths can differ from applied paths:** `src/agents/apply-patch-paths.ts:47`
  - body: The extractor trims leading payload space after an `apply_patch` marker, while the executor preserves that byte after slicing the marker. Policies that rely on `derivedPaths` can therefore evaluate `src/new.ts` while the tool writes ` src/new.ts`.
  - confidence: 0.92
- **[low] Derived paths can include non-applied update body paths:** `src/agents/apply-patch-paths.ts:147`
  - body: The update scanner can promote a two-space-indented marker-looking context line into a separate derived target, so trusted policies may evaluate a target set that is broader than what the executor applies.
  - confidence: 0.86
