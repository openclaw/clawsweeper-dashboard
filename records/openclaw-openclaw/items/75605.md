## Summary

Keep open. Current main does not yet have the requested derived-path hook seam, so the PR is not redundant, but the proposed parser still publishes `derivedPaths` that can differ from the path `apply_patch` will execute against. That mismatch is security-relevant because trusted policy hooks may authorize based on the derived metadata before the patch runs.

## What This Changes

The PR adds host-derived `apply_patch` target paths via a small tool-param parser registry, stamps `derivedPaths` onto trusted-policy and `before_tool_call` events, updates Plugin SDK docs/API/changelog, and adds focused unit, contract, and e2e coverage.

## Best Possible Solution

Keep the additive `derivedPaths` seam, but make the extractor share or exactly mirror the executor's hunk-header and update-body parsing semantics before publishing policy metadata. Add regressions for extra spaces after markers and marker-like update context lines, then land only after the parser/hook tests, API baseline check, lint, and exact-head CI are green.

## Reproduction Assessment

Yes. A patch header like `*** Add File:  src/new.ts` is enough to compare the current executor path (` src/new.ts`) with the PR extractor output (`src/new.ts`).

## Solution Assessment

No, not yet. The seam is the right narrow product direction, but the parser is not the best implementation until it either shares the executor parser or preserves the same executable target semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve apply_patch marker path bytes:** `src/agents/apply-patch-paths.ts:47-51`
  - body: `normalizePatchPath` trims the marker payload before emitting `derivedPaths`, but `apply_patch` trims the whole header and then slices after the marker. For `*** Add File:  src/new.ts`, execution targets ` src/new.ts` while the hook event reports `src/new.ts`, so a trusted policy can approve a different path than the one applied. Mirror the executor's path payload semantics before publishing policy metadata.
  - confidence: 0.9
- **[P2] Mirror update-hunk marker boundaries:** `src/agents/apply-patch-paths.ts:136-137`
  - body: Inside an update hunk, the executor stops only on headers that literally start with `***`; a line like `  *** Delete File: src/dead.ts` is valid context and is not a delete hunk. The new scanner breaks on `normalizeMarkerHeaderLine` and the new test expects `src/dead.ts`, so `derivedPaths` can include files that will not be touched. Align the update scanner with `parseUpdateFileChunk`.
  - confidence: 0.86
