## Summary

Keep open. Current main does not already implement the requested `derivedPaths` hook seam, and the PR still has a blocking correctness/security issue: its new extractor misses valid `apply_patch` hunks with leading header whitespace even though the real parser trims those headers before applying the patch.

## What This Changes

The PR adds host-derived `apply_patch` target path extraction, a narrow host tool-param parser registry, `derivedPaths` metadata on trusted-policy and `before_tool_call` events, plus docs, changelog, API baseline, and focused tests.

## Best Possible Solution

Keep the additive host-owned `derivedPaths` seam, but first make the extractor match the executable `apply_patch` grammar and add regression coverage for whitespace-normalized hunk headers before maintainer validation and merge.

## Reproduction Assessment

Yes. A high-confidence reproduction is an `apply_patch` payload whose hunk header starts with spaces: the current parser trims and accepts that header, while the PR extractor reads the raw line and returns no path metadata.

## Solution Assessment

No, not yet. The overall direction is a narrow maintainable seam, but the implementation must normalize add/update/delete/move header candidates consistently with `src/agents/apply-patch.ts` before plugins can safely rely on `derivedPaths`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Normalize hunk header whitespace before extracting paths:** `src/agents/apply-patch-paths.ts:71-89`
  - body: `parseOneHunk` trims hunk headers before matching them, so an `apply_patch` payload with leading spaces before `*** Add File:` is valid and will still execute. This extractor checks raw `line.startsWith(...)`, leaving `derivedPaths` absent for trusted and `before_tool_call` policies and allowing path-based policies to be bypassed. Trim or trim-start the candidate header, including the move-marker lookahead, before matching and slicing.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete path-policy bypass risk because host-derived path metadata can omit valid `apply_patch` targets with indented hunk headers.

Concerns:

- **[high] Path policy bypass via whitespace mismatch:** `src/agents/apply-patch-paths.ts:71`
  - body: Trusted policies and hooks may rely on `derivedPaths` for write-target decisions, but the extractor misses hunk headers with leading whitespace even though the real parser trims and accepts them. If merged as-is, a plugin policy could see no target paths while the patch still writes the file.
  - confidence: 0.92
