## Summary

Keep open. Current main does not contain the new `derivedPaths` seam, and the PR branch still has blocking mismatches between security-relevant derived path metadata and the actual `apply_patch` execution target semantics.

## What This Changes

The PR adds host-derived `apply_patch` target-path metadata for `before_tool_call` and trusted policy hooks with docs/API/test coverage, and also carries small Matrix DM-owner and noninteractive onboarding fixes.

## Best Possible Solution

Keep the additive `derivedPaths` seam, but derive targets with the same marker payload parsing and sandbox bridge resolution used by `apply_patch` execution, or omit derived metadata where the host cannot map targets authoritatively enough for policy use. After repair, rebase onto current main and rerun the focused parser, hook, SDK API, lint, and whitespace checks.

## Reproduction Assessment

Yes. The payload mismatch reproduces by comparing `line.trim().slice(marker.length)` with the PR extractor's trimmed payload for `*** Add File:  src/new.ts`; the sandbox mismatch follows from current executor source using `sandbox.bridge.resolvePath` while the PR deriver receives only a cwd.

## Solution Assessment

No. The requested seam is narrow and maintainable, but this implementation is not the best fix until derived metadata mirrors execution target semantics or is withheld when it cannot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve apply_patch marker payload bytes:** `src/agents/apply-patch-paths.ts:56-61`
  - body: `normalizePatchPath` trims the marker payload before resolving it. The executor trims the whole header and then slices after the marker, so `*** Add File:  src/new.ts` applies to `<cwd>/ src/new.ts` while `derivedPaths` reports `<cwd>/src/new.ts`. Preserve the sliced payload bytes before resolving so trusted policies do not evaluate a different file from the one applied.
  - confidence: 0.93
- **[P2] Resolve derived paths through the sandbox bridge:** `src/agents/pi-tools.ts:737`
  - body: Passing only `cwd: sandboxRoot ?? workspaceRoot` makes the deriver use host cwd normalization, but sandboxed `apply_patch` execution resolves paths through `sandbox.bridge.resolvePath`. Container or bind-mount inputs such as `/workspace/src/new.ts` can therefore be reported as the container path while execution touches the mapped host path. Use the same bridge/resolver for derivation, or omit `derivedPaths` for sandboxed calls.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention because trusted-policy path metadata can still disagree with the actual `apply_patch` target set.

Concerns:

- **[high] Marker payload trimming changes policy targets:** `src/agents/apply-patch-paths.ts:56`
  - body: The extractor trims payload bytes after the marker, while the executor preserves leading payload space after slicing the trimmed header. A policy can therefore review one path while `apply_patch` writes another.
  - confidence: 0.92
- **[medium] Sandbox bridge targets are not mirrored:** `src/agents/pi-tools.ts:737`
  - body: Sandboxed execution resolves through the sandbox FS bridge, including container mount paths, but the new derivation path only receives a cwd. Policies consuming `derivedPaths` can see container paths instead of the mapped execution targets.
  - confidence: 0.84
