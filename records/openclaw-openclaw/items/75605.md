## Summary

Keep open. Current main does not have the derived tool-path seam, so the PR is still useful, but the current head has a trusted-policy stale-metadata edge where `derivedPaths` can survive after params are rewritten to a shape with no derivable paths. The public hook docs and changelog also need to be aligned before this plugin SDK API change is merge-ready.

## What This Changes

The PR adds host-derived `apply_patch` target paths to plugin `before_tool_call` and trusted-policy events, with a parser helper, tests, and generated SDK API baseline hash updates.

## Best Possible Solution

Keep the host-owned derivation seam, but ensure `derivedPaths` is recomputed or cleared from a clean event shape for every trusted-policy and normal hook event. Add coverage for rewrites that remove all target paths, then document the public hook field and add the required plugin SDK changelog entry.

## Reproduction Assessment

Yes. A focused test with one trusted policy rewriting an `apply_patch` call from a path-bearing patch to `{ input: "not a patch" }` or `{}` and a later policy reading `event.derivedPaths` would reproduce the stale path list at the PR head.

## Solution Assessment

No. The direction is sound and narrower than the old consolidated stack, but this implementation still lets trusted policies see path metadata that no longer matches the rewritten params; building each policy event without prior derived metadata is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clear derived paths when rewrites remove targets:** `src/plugins/trusted-tool-policy.ts:22-24`
  - body: `buildEvent` starts from the original event, which can already include `derivedPaths`, and then only overlays recomputed metadata when derivation finds paths. If an earlier trusted policy rewrites an `apply_patch` call to params with no recognized targets, later policies still see the old path list with the new params. Build from an event shape that omits previous derived metadata, or explicitly clear it when derivation is empty.
  - confidence: 0.9
- **[P3] Document derivedPaths before exposing it:** `src/plugins/hook-types.ts:408-414`
  - body: This adds a public `before_tool_call` event field, but the plugin hook docs still enumerate only `toolName`, `params`, `runId`, and `toolCallId`, and there is no changelog entry. Plugin authors need the docs and changelog to know when `derivedPaths` is available and what absence means.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Security review needs attention because stale host-derived path metadata can mislead trusted tool policy decisions after params rewrites.

Concerns:

- **[medium] Stale paths can survive trusted-policy rewrites:** `src/plugins/trusted-tool-policy.ts:22`
  - body: Because the trusted-policy event builder spreads the original event before recomputed metadata, a rewrite from a path-bearing patch to params with no derived targets leaves the old `derivedPaths` visible to later policies. Policy gates treating this field as authoritative could approve, block, or request approval for the wrong files.
  - confidence: 0.88
