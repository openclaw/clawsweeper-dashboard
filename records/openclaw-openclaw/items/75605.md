## Summary

Keep this PR open. The additive `derivedPaths` hook seam is useful and current `main` does not already implement it, but the current PR head still publishes policy-significant `apply_patch` path metadata that can disagree with the executor in marker payload and update-hunk boundary cases.

## What This Changes

The PR adds host-derived `apply_patch` target paths to trusted tool policy and `before_tool_call` hook events, updates Plugin SDK hook docs/API baseline/changelog, and adds focused parser, hook, and contract tests.

## Best Possible Solution

Keep the additive `derivedPaths` seam, but make the extractor share or exactly mirror the executor’s `apply_patch` path semantics before publishing trusted policy metadata. Add regressions for extra marker payload spaces and marker-like update context lines, then land with the parser, hook contract, API baseline, docs, lint, and exact-head CI proof green.

## Reproduction Assessment

Yes. A patch header like `*** Add File:  src/new.ts` demonstrates the payload mismatch, and an update hunk followed by `  *** Delete File: src/dead.ts` demonstrates the update-boundary mismatch against current `apply-patch.ts`.

## Solution Assessment

No. The product direction is narrow and maintainable, but the current implementation is not the best fix until its derived metadata uses the same executable target semantics as `apply_patch`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve apply_patch marker path payloads:** `src/agents/apply-patch-paths.ts:47-52`
  - body: `normalizePatchPath` trims the marker payload before emitting `derivedPaths`, but the executor trims the whole header and then slices after the marker. For `*** Add File:  src/new.ts`, execution targets ` src/new.ts` while the hook event reports `src/new.ts`, so a trusted policy can approve a different path than the one applied. Mirror or share the executor's path payload semantics before publishing policy metadata.
  - confidence: 0.91
- **[P2] Mirror update-hunk header boundaries:** `src/agents/apply-patch-paths.ts:147-149`
  - body: The update scan treats a two-space-indented `*** Delete File:` line as the next top-level marker, but `apply_patch` only stops update parsing when the raw line starts with `***`; that indented line is valid update context and will not delete a file. This makes `derivedPaths` include files the tool will not touch, so align the scan with `parseOneHunk` and update the new test expectation.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive trusted-policy path metadata that can disagree with the actual `apply_patch` targets.

Concerns:

- **[high] Derived paths can differ from applied paths:** `src/agents/apply-patch-paths.ts:47`
  - body: The extractor trims extra path bytes after an `apply_patch` marker and promotes some indented update-body marker-like lines, while the executor preserves those marker payload bytes and treats those indented lines as context. Policies that rely on `derivedPaths` can therefore evaluate a different target set from the tool that executes.
  - confidence: 0.9
