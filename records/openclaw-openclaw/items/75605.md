## Summary

Keep this draft PR open. Current main does not contain the proposed `derivedPaths` hook seam, so the branch is not obsolete, but the latest head still has a security-relevant stale metadata edge in trusted policy evaluation and the public plugin docs/changelog are not aligned with the new SDK surface.

## What This Changes

The PR adds host-derived `apply_patch` target paths to plugin `before_tool_call` and trusted-policy events, with a parser helper, registry, tests, and SDK API baseline hash updates.

## Best Possible Solution

Keep the host-owned derivation seam, but construct trusted-policy events from a clean event shape or explicitly clear `derivedPaths` when recomputation produces no paths. Then document the new public hook field, add the required plugin SDK changelog entry, refresh the API baseline if needed, and rerun the focused parser/hook/policy validation before marking the draft ready.

## Reproduction Assessment

Yes. A focused trusted-policy test can start with an `apply_patch` event that derives paths, have the first policy rewrite params to `{ input: "not a patch" }` or `{}`, and assert that a later policy sees `event.derivedPaths` absent rather than the original path list.

## Solution Assessment

No. The overall direction is a narrow and maintainable SDK seam, but this implementation is not yet the best version because trusted-policy metadata can disagree with rewritten params; rebuilding the event without stale derived metadata is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clear derived paths when rewrites remove targets:** `src/plugins/trusted-tool-policy.ts:22-24`
  - body: `buildEvent` starts from the original event, which can already contain `derivedPaths`, and only overwrites it when the current params derive paths. If an earlier trusted policy rewrites an `apply_patch` call to params with no recognized targets, later policies see rewritten `event.params` with the old path list. Build from an event shape that omits previous derived metadata, or explicitly clear it when derivation is empty.
  - confidence: 0.9
- **[P3] Document and changelog the hook metadata:** `src/plugins/hook-types.ts:405-412`
  - body: This adds a public `before_tool_call` event field, but the plugin hook docs still list only `toolName`, `params`, `runId`, and `toolCallId`, and there is no changelog entry for the user-facing SDK feature. Plugin authors need the docs and changelog to know when `derivedPaths` is available and what absence means.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch improves policy metadata, but a stale `derivedPaths` edge can mislead trusted policy decisions after params are rewritten to a shape with no derivable paths.

Concerns:

- **[medium] Stale paths can survive no-target rewrites:** `src/plugins/trusted-tool-policy.ts:22`
  - body: Because the trusted-policy event builder spreads the original event before recomputed metadata, a rewrite from a path-bearing patch to params with no derived targets leaves the old `derivedPaths` visible to later policies. Policy gates treating this field as authoritative could approve, block, or request approval for the wrong files.
  - confidence: 0.88
