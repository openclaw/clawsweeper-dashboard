## Summary

Keep open: the feature direction is useful, but the PR is not merge-ready because the new `cron.get` path misses cold lazy cron state and the external-PR real behavior proof gate is still unmet.

## What This Changes

Adds single-job cron inspection through `CronService.getJob`, Gateway `cron.get`, `openclaw cron get <id>`, the agent cron tool, protocol schemas, docs, changelog, and focused tests.

## Best Possible Solution

Keep the additive API shape, but make lazy cron `getJob` load persisted state before returning, update the CLI index, add regression coverage for the cold lazy path, and require real Gateway/CLI proof before merge.

## Reproduction Assessment

Not applicable as a feature PR. The current-main gap is source-confirmed: there is no `cron.get` method, and single-job CLI inspection still goes through `cron.list` and client-side filtering.

## Solution Assessment

No, not yet. The proposed API shape is the right narrow direction, but the implementation must load the lazy cron service for `getJob` and keep the public CLI index in sync before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Load the lazy cron service before get:** `src/gateway/server-methods/cron.ts:244`
  - body: The normal Gateway path uses `createLazyGatewayCronState`, whose `getJob` still returns `undefined` while cron is cold. Since this handler now awaits `context.cron.getJob(jobId)` and treats a missing value as not found, `openclaw cron get <id>` can fail after a restart for persisted jobs until another cron call warms the service; make lazy `getJob` load the real cron service and cover that path.
  - confidence: 0.93
- **[P3] Update the CLI index for cron get:** `docs/cli/cron.md:215`
  - body: The new command is documented in cron-specific docs, but the top-level CLI index still lists the cron command tree without `get`. Add `get` there so the public command index matches the new CLI surface.
  - confidence: 0.9
