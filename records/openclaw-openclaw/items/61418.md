## Summary

Keep open. Current main still has auth-aware model visibility in the Gateway/shared catalog path, but the local embedded TUI `/models` path still returns the raw allowed/full catalog instead of filtering to configured or auth-available providers.

## What This Changes

The issue requests that TUI `/models` filter model choices to configured/auth-available providers, matching `openclaw models list`.

## Best Possible Solution

Reuse `resolveVisibleModelCatalog` from `EmbeddedTuiBackend.listModels()` with the correct config, workspace, and auth context, then add embedded-backend regression coverage for unauthenticated catalog rows.

## Reproduction Assessment

Yes, at source level: `openclaw chat`, `openclaw terminal`, and `openclaw tui --local` use `EmbeddedTuiBackend`, whose `listModels()` still returns the full catalog when no explicit allowlist narrows it. I did not live-run the terminal UI in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to align the embedded backend with the existing shared `resolveVisibleModelCatalog` policy rather than adding a new TUI-only filter or changing Gateway semantics.
