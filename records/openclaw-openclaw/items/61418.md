## Summary

Keep open. Current main has the auth-aware Gateway model-list policy, but the local embedded TUI backend still falls back to the raw full catalog when no explicit allowlist exists, so the reported `/models` noise can still occur in `openclaw chat`, `openclaw terminal`, or `openclaw tui --local`.

## What This Changes

The issue requests that the TUI `/models` picker filter to authenticated/configured models, matching `openclaw models list`.

## Best Possible Solution

Reuse `resolveVisibleModelCatalog` from `EmbeddedTuiBackend.listModels()` with the right config, workspace, and auth context, then add embedded-backend regression coverage for unauthenticated catalog rows.

## Reproduction Assessment

Yes, at source level: `openclaw chat` and `openclaw terminal` route to local embedded mode, and `EmbeddedTuiBackend.listModels()` still returns the full catalog when no allowlist exists. I did not run a live TUI session in this read-only review, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is to align the embedded backend with the already-shared Gateway visible-catalog policy, rather than adding another bespoke filter or changing Gateway semantics.
