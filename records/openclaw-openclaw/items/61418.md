## Summary

Keep open. Current main appears to fix the Gateway-backed TUI model picker by routing `models.list` through a configured/auth-backed visible catalog, but the local embedded TUI still has a separate `listModels()` path that falls back to the full catalog whenever no explicit allowlist exists, so the reported `/models` auth-filtering problem can still remain for `openclaw chat` / `openclaw tui --local`.

## What This Changes

The request is to make the TUI `/models` command and model picker show only configured or authenticated models, matching the default `openclaw models list` behavior instead of showing unauthenticated provider catalogs.

## Best Possible Solution

Unify the local embedded TUI model picker with the Gateway visible-catalog semantics: `agents.defaults.models` should win, configured provider models should remain visible, and catalog rows should be included only when the provider has usable auth, with full-catalog behavior reserved for explicit debug/all views.

## Reproduction Assessment

Unclear for the reporter’s exact 2026.4.2 environment because the issue lacks a runnable config/auth-store fixture. For current main, there is a high-confidence source-level reproduction for local embedded mode: no explicit allowlist plus a mixed catalog causes `EmbeddedTuiBackend.listModels()` to return the full catalog instead of the auth-filtered visible catalog.

## Solution Assessment

No, current main is not the complete best solution across TUI modes. The Gateway implementation is the right reusable direction, but the embedded TUI should call the same visible-catalog helper with the correct config/workspace/auth context instead of falling back to the raw catalog.
