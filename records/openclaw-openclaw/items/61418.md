## Summary

Keep open. Current main fixes the Gateway-backed TUI model picker, but the local embedded TUI path still falls back to the full model catalog when no explicit allowlist exists, which preserves the reported unauthenticated-model list for `openclaw chat` / `openclaw tui --local`.

## What This Changes

The issue asks for terminal TUI `/models` to use the same auth/configured filtering as `openclaw models list` instead of showing unauthenticated catalog rows.

## Best Possible Solution

Make `EmbeddedTuiBackend.listModels()` use the same visible-catalog semantics as Gateway `models.list`: allowlist first, configured models next, auth-backed catalog rows only, and raw full catalog only for explicit debug/all views.

## Reproduction Assessment

Yes for current main at source level: `openclaw chat` and `openclaw tui --local` use `EmbeddedTuiBackend`, whose `listModels()` returns the full catalog whenever no explicit allowlist is present. The exact reporter environment is not needed to prove that remaining local-mode path.

## Solution Assessment

No, current main is only a partial solution because Gateway mode uses the right shared helper while embedded mode still has bespoke fallback logic. The narrow maintainable fix is to reuse or mirror `resolveVisibleModelCatalog` in the embedded backend with the correct config/workspace/auth context.
