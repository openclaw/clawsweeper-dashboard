## Summary

This PR should stay open: the intended owner-only, default-denied HTTP `exec` shape is plausible and the PR body now has live proof, but the current patch drops important exec policy defaults on a new command-execution surface and its added test would not catch the production registration gap.

## What This Changes

The branch adds a lazy owner-only HTTP `exec` tool registration for `/tools/invoke`, documents the explicit opt-in path, and adds a positive allowlisted exec invoke test.

## Best Possible Solution

Expose HTTP `exec` through the existing owner-only, default-denied opt-in while sharing the normal exec default resolver and proving both deny-by-default and constrained allowlist behavior.

## Reproduction Assessment

Yes. Source inspection shows current main can remove `exec` from the default HTTP deny list, but the production HTTP tool pool does not register agent-side `exec`, so an allowlisted direct invoke still resolves as unavailable.

## Solution Assessment

No. Lazy owner-only registration is the right shape, but the PR should reuse the normal exec-default resolution and add a regression test that fails against current main before exposing this RCE-capable HTTP path.
