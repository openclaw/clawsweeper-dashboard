## Summary

This PR targets a real current-main gap in the documented `/tools/invoke` allowlist behavior, but it should stay open because the new HTTP `exec` wrapper does not preserve all existing exec defaults and the PR body does not show actual after-fix live output.

## What This Changes

The PR adds a lazy owner-only `exec` tool to Gateway HTTP `/tools/invoke` when `gateway.tools.allow` re-enables it, updates the HTTP invoke docs, and adds a unit test for opt-in exec invocation.

## Best Possible Solution

Keep the opt-in HTTP exec direction, but construct it from the same exec default semantics as agent exec, preserve default-deny and audit warnings, and require redacted after-fix terminal/log proof before merge.

## Reproduction Assessment

Yes. From source, current main can remove `exec` from the HTTP deny list, but the HTTP tool pool never registers an `exec` tool, so an allowlisted direct invoke still resolves to `404 not_found`.

## Solution Assessment

No. Lazy registration is the right shape, but the proposed implementation should reuse or mirror the full exec-default resolver instead of hand-copying a partial subset of `tools.exec` settings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Forward the full exec defaults:** `src/gateway/tool-resolution.ts:48-56`
  - body: The HTTP exec wrapper copies only part of the existing exec config.
