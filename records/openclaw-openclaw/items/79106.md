## Summary

Keep open for normal PR review: current main still lacks HTTP `exec` registration, the updated branch addresses the earlier full-defaults and changelog concerns, and the remaining issue is a non-blocking but actionable regression-test gap.

## What This Changes

The PR registers owner-only HTTP `exec` for `/tools/invoke` behind the existing `gateway.tools.allow` opt-in, shares exec config resolution with agent exec, and updates docs, tests, and changelog.

## Best Possible Solution

Keep the owner-only, default-denied HTTP exec registration with shared exec config resolution, tighten the regression test to exercise the lazy registration path, then land after normal maintainer/security validation.

## Reproduction Assessment

Yes. Source inspection of current main shows `gateway.tools.allow` can remove `exec` from the default HTTP deny list, but the production HTTP tool pool never registers agent-side `exec`, so allowlisting cannot make the tool invokable.

## Solution Assessment

Yes, with a small test fix. The shared resolver plus owner-only, default-denied lazy registration is the narrow maintainable path; the regression test should be adjusted so it fails against current main.
