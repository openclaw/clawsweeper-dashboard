## Summary

Keep open: the PR addresses a real Gateway HTTP tool-resolution gap and now includes after-fix live curl output, but the patch still drops security-relevant exec defaults on the new opt-in HTTP command-execution path and is missing the required changelog entry.

## What This Changes

The PR adds a lazy owner-only `exec` tool to Gateway HTTP `/tools/invoke` when explicitly allowlisted, documents the opt-in path, and adds an HTTP invoke test.

## Best Possible Solution

Expose HTTP `exec` only through the existing default-denied, owner-only opt-in while reusing the same exec default semantics as agent exec and documenting the user-facing change in the changelog.

## Reproduction Assessment

Yes. Source inspection shows current main can remove `exec` from the HTTP deny list, but the HTTP tool pool is built from `createOpenClawTools`, which does not register the agent-side `exec` tool, so an allowlisted direct invoke still resolves as unavailable.

## Solution Assessment

No. Lazy owner-only registration is the right shape, but the proposed implementation should reuse or mirror the full exec-default resolver before exposing this dangerous HTTP path.

## Review Findings
