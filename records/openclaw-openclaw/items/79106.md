## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enforce owner/admin gating for HTTP exec:** `src/gateway/tool-resolution.ts:179`
  - body: Setting `ownerOnly: true` on the new lazy tool does not enforce anything in this resolver. Once `gateway.tools.allow` removes `exec` from the default deny list, a trusted-proxy caller with `operator.write` but without `operator.admin` can still receive the tool because owner filtering is name-based elsewhere. Add `exec` to the owner/admin deny path and cover the non-admin trusted-proxy case before exposing shell execution.
  - confidence: 0.93
- **[P1] Decide or block the SDK RPC exec surface:** `src/gateway/tool-resolution.ts:168-180`
  - body: This registers `exec` for every `surface === "http"` resolver call, but the shared invoke helper is also used by SDK RPC `tools.invoke`. That makes the new shell capability reachable outside the documented HTTP-only scope, with no RPC docs, tests, or proof. Split the surfaces or intentionally document and test the SDK admin path.
  - confidence: 0.87
- **[P1] Preserve mode-derived exec policy on rebase:** `src/agents/exec-config-resolution.ts:10-13`
  - body: The extracted resolver only forwards legacy `security`/`ask` values. Current main resolves `tools.exec.mode` through the agent exec policy layer, so rebasing this helper without carrying that behavior would make configs such as `mode: "deny"` stop fail-closing for the new HTTP exec path and possibly the agent path too.
  - confidence: 0.84
- **[P3] Remove the direct changelog edit:** `CHANGELOG.md:10`
  - body: `CHANGELOG.md` is release-owned in this repository, so normal feature PRs should leave the release note context in the PR body or squash message instead of editing the unreleased changelog directly.
  - confidence: 0.96
