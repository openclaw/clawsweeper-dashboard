## Summary

Keep open. Current main still does not show response duration in the per-reply usage footer, and this contributor PR is useful but not mergeable as-is: it targets a helper that has since moved, changes the existing cost separator, adds stopwatch data to the documented tokens-only mode, and samples the duration after delivery drains.

## What This Changes

The PR adds an optional `durationMs` usage-footer parameter, formats a stopwatch segment in the response usage line, and passes a `runReplyAgent` duration into diagnostics and the footer.

## Best Possible Solution

Rebase the change onto the current `agent-runner-usage-line.ts` helper, preserve existing footer text when no duration is shown, gate duration to the chosen rich footer or a new explicit mode, measure before delivery drains or label it as end-to-end wall time, and add focused formatter/runner tests plus docs/changelog if the visible footer contract changes.

## Reproduction Assessment

Yes. Source inspection of the PR diff against current main is enough to reproduce the defects: the delimiter changes in the formatter, `durationMs` is passed for every non-off usage mode, and the sampled value is after block/tool delivery drains.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best implementation because it is stale against current main and needs compatibility, mode-gating, timing-semantics, and test updates before merge.

## Review Findings

Full review comments:

- **[P2] Preserve the cost separator when duration is absent:** `src/auto-reply/reply/agent-runner-utils.ts:139`
  - body: The new `parts.join(" / ")` path changes existing cost output from ` · est ...` to `/ est ...` even when `durationMs` is omitted. That breaks the stated compatibility contract and can regress display assertions or parsers that depend on the existing footer shape.
  - confidence: 0.93
- **[P2] Gate duration out of the tokens-only footer:** `src/auto-reply/reply/agent-runner.ts:598`
  - body: `responseUsageMode !== "off"` includes both `tokens` and `full`, so passing `durationMs` here makes `/usage tokens` include stopwatch metadata. The docs define that mode as tokens-only, so duration should be limited to the richer mode or a separate explicit mode.
  - confidence: 0.9
- **[P2] Measure or label the full wall-clock duration accurately:** `src/auto-reply/reply/agent-runner.ts:540`
  - body: This value is sampled after block reply flushing and pending tool-result delivery drains, but the footer presents it as response duration. Delivery delay can inflate the displayed time, so measure before those waits or label the value as end-to-end wall time.
  - confidence: 0.84
