## Summary

Keep open. The feature is still not implemented on current main, but the PR branch has blocking correctness issues, no real behavior proof, and is now conflicting with the current usage-footer helper split.

## What This Changes

The PR adds a stopwatch duration parameter to per-reply usage footer formatting and passes elapsed reply-run time into the footer while reusing it for model-usage diagnostics.

## Best Possible Solution

Rebase onto the current usage-line helper, preserve existing cache/cost separators, add duration only to the product-approved footer mode and timing boundary, then cover it with focused tests plus real behavior proof.

## Reproduction Assessment

Yes. Source inspection of the PR diff against current main reproduces the blockers: omitted duration still changes cost formatting, and passing duration before the `full` branch expands `/usage tokens`.

## Solution Assessment

No. The current patch is not the best mergeable shape because it changes existing footer text, broadens the lean tokens mode, and needs a maintainer decision about duration placement and timing semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the cost separator when duration is absent:** `src/auto-reply/reply/agent-runner-utils.ts:139`
  - body: The formatter now joins every segment with ` / `, so calls without `durationMs` change from `Usage: ... · est $0.001` to `Usage: ... / est $0.001`. That contradicts the compatibility claim and can break consumers or snapshots that match the existing footer text.
  - confidence: 0.91
- **[P2] Keep duration out of the tokens-only footer:** `src/auto-reply/reply/agent-runner.ts:595`
  - body: Passing `durationMs` unconditionally means `/usage tokens` receives `⏱ ...` too, even though current docs define that mode as token-only and reserve richer metadata for `full`. Gate the duration on the approved mode, or introduce a deliberate new mode before changing the minimal footer.
  - confidence: 0.87
- **[P2] Capture response duration before delivery drains:** `src/auto-reply/reply/agent-runner.ts:537`
  - body: This timestamp is sampled after current runner code flushes block replies and drains pending tool delivery tasks, so the visible `response duration` can include post-generation delivery waits. Capture the intended model/tool turn boundary before those drains, or label the field as end-to-end wall time.
  - confidence: 0.78
