## Summary

Keep open: current main and the latest release still show the ACP bridge fallback key colliding with ACPX runtime-session detection, which matches the reported `acp_session_init_failed`/echo/`end_turn` failure. Open PR https://github.com/openclaw/openclaw/pull/62552 appears to be the direct fix path, so this issue should stay open until that or an equivalent narrower fix lands.

## What This Changes

The issue reports ACP bridge prompts failing with `acp_session_init_failed`, echoing the prompt and ending the turn instead of streaming assistant chunks.

## Best Possible Solution

Default ACP bridge sessions should use a non-ACPX bridge key namespace and pending prompt matching should normalize raw versus Gateway-canonical keys, while true ACPX stale-session detection remains fail-closed.

## Reproduction Assessment

Yes, at source level. Current main shows the built-in client creates a metadata-free session, the bridge maps it to an ACP-shaped fallback key, and ACP runtime dispatch treats ACP-shaped keys without `entry.acp` as stale errors; I did not run a live ACP client in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is the central ACP bridge-key separation shown in the open fix PR, with any unrelated task maintenance kept separate unless independently needed.
