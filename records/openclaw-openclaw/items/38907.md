## Summary

Keep open: current main still routes default Gateway-backed ACP bridge sessions through metadata-backed ACPX dispatch when the bridge creates an `acp:<uuid>` session key without ACP runtime metadata, producing the reported `ACP_SESSION_INIT_FAILED` path before normal Gateway generation can run.

## What This Changes

The issue asks for `openclaw acp client` bridge sessions to stream assistant chunks instead of failing on the first prompt with `acp_session_init_failed`, echoed input, and `[end_turn]`.

## Best Possible Solution

Fix the bridge/runtime boundary so Gateway-backed `openclaw acp` bridge sessions are not claimed as metadata-backed ACPX sessions unless persisted ACP metadata or an explicit binding exists, while preserving stale-runtime handling for real ACPX sessions.

## Reproduction Assessment

Yes. The current-main source path is deterministic: debug `newSession` creates no ACP metadata, the bridge chooses an `acp:<uuid>` fallback key, and ACPX reply dispatch treats that metadata-free ACP-shaped key as stale before normal Gateway generation can run.

## Solution Assessment

Yes. The narrow maintainable fix is to separate Gateway-backed bridge session routing from ACPX metadata-backed runtime detection, rather than adding a broad fallback or weakening ACPX stale-session cleanup globally.
