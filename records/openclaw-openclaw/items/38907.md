## Summary

Keep open: current `main` and the latest release still route metadata-free ACP bridge sessions through ACP runtime-shaped keys, while ACP runtime resolution still treats those keys without persisted ACP metadata as stale; the linked repair PR is open but conflicting and not merge-ready.

## What This Changes

ACP bridge sessions should stream assistant chunks on the first prompt instead of failing with `acp_session_init_failed`, echoed input, and `end_turn`.

## Best Possible Solution

Land a focused ACP bridge-key fix or replacement PR that keeps metadata-free bridge sessions out of ACP runtime key space, preserves pending-prompt reconciliation, updates ACP docs/tests, and closes this issue after merge.

## Reproduction Assessment

Yes, at source level: current `main` creates metadata-free ACP client sessions, falls back to `acp:<uuid>`, and classifies ACP-shaped keys without stored ACP metadata as stale. I did not run a live ACP client in this read-only review.

## Solution Assessment

Yes for the repair direction, but not yet on `main`: the narrow maintainable fix is to separate metadata-free bridge keys from ACP runtime keys while preserving raw/canonical pending-prompt matching. The open repair PR is useful evidence of that direction but still needs rebase/proof or replacement before merge.
