## What This Changes

The issue reports that `openclaw acp client` fails on the first prompt with `acp_session_init_failed`, echoing input and ending the turn without assistant chunks.

## Best Possible Solution

Land a focused ACP bridge-key fix or replacement PR that keeps metadata-free bridge sessions out of ACP runtime key space, preserves raw/canonical pending-prompt matching, and updates ACP docs/tests for the new contract.

## Reproduction Assessment

Yes, at source level: the ACP client creates a session without `_meta`, the bridge falls back to `acp:<uuid>`, and the ACP manager classifies ACP-shaped keys without stored `acp` metadata as stale. I did not run the live client in this read-only review.

## Solution Assessment

Yes for the repair direction, but it is not solved on main yet. Separating metadata-free bridge keys from ACP runtime keys is the narrow maintainable fix, and the existing open PR or an equivalent rebased replacement should land before closing this issue.
