## Summary

Keep open: current main still has the ACP bridge/runtime boundary that can turn a metadata-free debug ACP bridge session into an ACPX stale-session error before normal Gateway-backed generation can run.

## What This Changes

`openclaw acp client` should create bridge sessions whose first prompt streams assistant chunks instead of failing with `ACP_SESSION_INIT_FAILED`, echoed input, and `end_turn`.

## Best Possible Solution

Keep the issue open and make a narrow bridge/runtime boundary fix so default Gateway-backed ACP bridge sessions are not claimed as metadata-backed ACPX runtime sessions unless persisted ACP metadata or an explicit binding exists.

## Reproduction Assessment

Yes. The source-level path is deterministic on current main: the debug client creates a metadata-free session, the bridge maps it to `acp:<uuid>`, and ACP dispatch treats that key as stale when no ACP metadata exists; the issue comments also include a confirmed macOS repro.

## Solution Assessment

Yes. The narrow maintainable fix is to separate default Gateway-backed bridge session keys or dispatch fallthrough from ACPX metadata-backed session detection while preserving explicit ACP runtime sessions, bindings, labels, and `loadSession`.
