## Summary

Keep open: current main and v2026.6.6 still have a source-reproducible ACP bridge/control-plane key collision where the bridge creates `acp:` fallback sessions but ACP manager treats agent-scoped ACP-shaped sessions without persisted metadata as stale before normal model dispatch.

## What This Changes

The issue requests fixing `openclaw acp client` so first prompts stream assistant chunks instead of failing with `ACP_SESSION_INIT_FAILED` and ending without generated output.

## Best Possible Solution

Keep ACP bridge sessions out of control-plane ACP runtime classification unless persisted `SessionAcpMeta` exists, while leaving spawned ACP session stale-metadata failures intact.

## Reproduction Assessment

Yes by source inspection, though not by a live run in this read-only review. The bridge creates `acp:` sessions, Gateway canonicalizes them to ACP-shaped agent keys, and current agent dispatch throws stale ACP metadata before normal model output can stream.

## Solution Assessment

Yes, the report is a valid existing-behavior bug, but the best fix is not to add control-plane metadata to bridge sessions. The narrower fix is to separate bridge session routing from ACP runtime metadata detection or carry an explicit bridge-origin signal through dispatch.
