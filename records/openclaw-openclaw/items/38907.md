## Summary

Current main still contains the reported ACP bridge/runtime boundary bug: the debug ACP client creates a metadata-free bridge session, the bridge default still uses an ACP-shaped Gateway session key, and ACP dispatch treats ACP-shaped keys without stored ACP metadata as stale instead of falling through to normal Gateway-backed generation.

## What This Changes

The issue reports that default `openclaw acp client` sessions fail on the first prompt with `acp_session_init_failed`, echoing input plus `end_turn` instead of streaming assistant chunks.

## Best Possible Solution

Keep this open and fix the ACP bridge/runtime boundary so default Gateway-backed `openclaw acp` bridge sessions use a key shape or dispatch path that ACPX runtime dispatch does not claim as metadata-backed ACP, while preserving explicit session keys, labels, `loadSession`, configured ACP bindings, `/acp spawn`, and ACP runtime spawns.

## Reproduction Assessment

Yes. The code-level reproduction path is deterministic: default `openclaw acp client` sends `newSession` without metadata, the bridge assigns `acp:<uuid>`, and the first `chat.send` can be intercepted as stale ACP metadata; the issue comments also report a confirmed macOS arm64 repro.

## Solution Assessment

Yes. The best fix is a narrow bridge/runtime boundary change plus regression coverage for the default first-prompt path; disabling ACP stale-session detection globally would be a broader and riskier workaround.
