## Summary

Keep open: current main still contains the source path that can turn metadata-free ACP bridge sessions into ACPX stale-session errors, matching the reported logs, and no merged or open item clearly owns this exact default bridge failure.

## What This Changes

The issue reports that `openclaw acp client` default bridge sessions fail on the first prompt with `acp_session_init_failed` instead of streaming assistant chunks.

## Best Possible Solution

Keep Gateway-backed ACP bridge routing separate from ACPX runtime-session detection so default bridge sessions generate normally unless persisted ACP metadata or an explicit ACPX binding exists.

## Reproduction Assessment

Yes. Source inspection on current main shows the built-in client creates a metadata-free session, the bridge maps it to an ACP-shaped fallback key, and ACPX reply dispatch treats that key as stale when `entry.acp` is absent.

## Solution Assessment

Yes. The narrow maintainable fix is to require real ACPX metadata or an intentional binding before ACPX runtime dispatch claims a bridge session, while keeping stale-session errors for actual ACPX sessions.
