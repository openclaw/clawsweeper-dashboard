## Summary

Keep open: current main still has the same source path that can classify the built-in ACP bridge’s metadata-free fallback session as a stale ACPX runtime session, which matches the reported `acp_session_init_failed`/echo/`end_turn` behavior. No open PR or canonical issue currently owns this exact default bridge failure.

## What This Changes

The issue reports that `openclaw acp client` bridge sessions fail on the first prompt with missing ACP metadata, echoing input and ending the turn instead of streaming assistant chunks.

## Best Possible Solution

Keep Gateway-backed ACP bridge routing separate from ACPX runtime-session detection so default bridge sessions generate normally unless persisted ACP metadata or an explicit ACPX binding exists.

## Reproduction Assessment

Yes, at source level. Current main shows the built-in client creates a metadata-free session, the bridge maps it to an ACP-shaped fallback key, and ACP runtime dispatch treats ACP-shaped keys without `entry.acp` as stale errors.

## Solution Assessment

Yes. The narrow maintainable fix is to require real ACPX metadata or an intentional binding before ACP runtime dispatch claims a bridge session, while keeping stale-session errors for actual ACPX sessions.
