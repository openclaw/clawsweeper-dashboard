## Summary

Keep open: current `main` and latest release still create metadata-free ACP bridge sessions with `acp:<uuid>` fallback keys, while ACP runtime resolution treats ACP-shaped keys without persisted ACP metadata as stale and returns `ACP_SESSION_INIT_FAILED`. The related repair PR is open but conflicting, so this issue remains the right user-facing tracker.

## What This Changes

The issue reports that `openclaw acp client` fails on the first prompt with `acp_session_init_failed`, echoing input plus `end_turn` instead of streaming assistant chunks.

## Best Possible Solution

Land a focused ACP bridge-key fix or replacement PR that keeps metadata-free bridge sessions out of ACP runtime key space, preserves raw/canonical pending-prompt matching, updates focused ACP coverage, and closes this issue after merge.

## Reproduction Assessment

Yes, at source level: current `main` creates metadata-free ACP client sessions, falls back to `acp:<uuid>`, and classifies ACP-shaped keys without stored ACP metadata as stale. I did not run the live ACP client flow in this read-only review.

## Solution Assessment

Yes for the repair direction, but not yet on `main`: the narrow maintainable fix is to separate metadata-free bridge keys from ACP runtime keys while preserving pending-prompt reconciliation. The open repair PR is useful evidence of that direction but still needs rebase/proof or replacement before merge.
