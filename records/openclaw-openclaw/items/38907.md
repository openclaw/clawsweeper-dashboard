## Summary

Keep open: current main still maps default Gateway-backed ACP bridge sessions to metadata-free `acp:<uuid>` keys, while ACP dispatch still treats any ACP-shaped key without persisted ACP metadata as a stale ACPX runtime session and returns `ACP_SESSION_INIT_FAILED`.

## What This Changes

The issue requests a fix for `openclaw acp client` sessions that fail on first prompt with echoed input and `end_turn` instead of assistant chunks.

## Best Possible Solution

Keep Gateway-backed ACP bridge routing separate from ACPX runtime-session detection so default bridge sessions can generate normally unless explicit ACPX metadata or a binding exists.

## Reproduction Assessment

Yes, source-reproducible on current main. The built-in client creates a metadata-free session, the bridge maps it to `acp:<uuid>`, and ACP dispatch returns the stale-session `ACP_SESSION_INIT_FAILED` path before normal Gateway generation can run.

## Solution Assessment

Yes. The narrow maintainable fix is to stop claiming default bridge `acp:<uuid>` Gateway session keys as ACPX runtime sessions unless persisted ACP metadata or an explicit binding exists, while preserving `/acp spawn`, labels, explicit session keys, and `loadSession`.
