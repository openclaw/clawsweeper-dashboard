## Summary

Keep open: current `main` and the latest release still have the metadata-free ACP bridge path using an `acp:` fallback key, while runtime session resolution treats ACP-shaped keys without ACP metadata as stale `ACP_SESSION_INIT_FAILED`. The open fix path is https://github.com/openclaw/openclaw/pull/62552, but it is not merged and still needs maintainer handling, so this issue should remain open.

## What This Changes

The issue reports that ACP bridge sessions created by `openclaw acp client` fail on the first prompt with `acp_session_init_failed`, echoing input and ending the turn without assistant chunks.

## Best Possible Solution

Default metadata-free ACP bridge sessions should use a non-ACPX bridge key namespace and pending prompt matching should handle raw versus Gateway-canonical keys, while true ACPX stale-session detection remains fail-closed.

## Reproduction Assessment

Yes, at source level. Current `main` creates metadata-free ACP client sessions, maps them to `acp:<uuid>`, and then classifies ACP-shaped keys without `entry.acp` as stale `ACP_SESSION_INIT_FAILED`; I did not run the live client path.

## Solution Assessment

Yes for the core direction: separating bridge fallback keys from ACP runtime keys is the narrow maintainable fix. The current open PR should be reviewed, rebased, and likely narrowed because unrelated task-maintenance changes and missing real behavior proof are still open concerns.
