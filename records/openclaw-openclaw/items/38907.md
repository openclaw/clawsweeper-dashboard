## Summary

Keep open. Current main and the latest release still create default ACP client sessions with an `acp:<uuid>` key, and the agent runtime still rejects ACP-shaped keys when no ACP metadata is persisted; an open focused PR appears to target this, but it is not merged or proof-positive yet.

## What This Changes

The issue requests that `openclaw acp client` sessions initialize correctly and stream assistant `agent_message_chunk` updates instead of failing with `acp_session_init_failed` and returning only `end_turn`.

## Best Possible Solution

Land or replace the focused ACP bridge-session fix so default interactive client sessions no longer use metadata-required ACP runtime keys, with real behavior proof showing the first prompt streams assistant text.

## Reproduction Assessment

Yes. Source inspection shows `openclaw acp client` creates a fallback `acp:<uuid>` session, then `chat.send` routes through agent command, where ACP-shaped sessions without persisted metadata are rejected before a native run starts.

## Solution Assessment

No current merged solution exists. The best fix is a narrow ACP bridge-session key or initialization change, plus regression coverage and real proof that an interactive ACP client prompt emits `agent_message_chunk` before `end_turn`.
