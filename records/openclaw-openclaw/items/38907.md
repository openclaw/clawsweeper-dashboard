## Summary

Keep open: current main still maps metadata-free ACP bridge sessions to `acp:<uuid>`, then the ACPX reply-dispatch path treats that ACP-shaped key as a stale ACP runtime session and returns the reported init-failure path before normal Gateway generation can run.

## What This Changes

The issue reports that `openclaw acp client` creates ACP bridge sessions that fail on the first prompt with `acp_session_init_failed` instead of streaming assistant chunks.

## Best Possible Solution

Keep Gateway-backed ACP bridge routing separate from ACPX runtime-session detection so default bridge sessions generate normally unless explicit ACPX metadata or an intentional ACPX binding exists.

## Reproduction Assessment

Yes. Source inspection on current main shows the built-in ACP client creates a metadata-free session, the bridge maps it to `acp:<uuid>`, and ACPX dispatch resolves that ACP-shaped key as stale when persisted ACP metadata is absent.

## Solution Assessment

Yes. The narrow maintainable fix is to stop claiming default bridge fallback keys as ACPX runtime sessions unless persisted ACP metadata or an explicit binding exists, while preserving real ACPX stale-session handling.
