## Summary

Keep open. Current main and release v2026.6.5 still give metadata-free ACP bridge sessions an `acp:<uuid>` fallback key, while the ACP control plane treats that namespace as metadata-backed runtime state and fails closed when metadata is absent. The focused repair at https://github.com/openclaw/openclaw/pull/62552 remains open, conflicting, and lacks real bridge proof, so it is not yet an implementation close.

## What This Changes

Fix ACP bridge sessions that end without assistant chunks because their generated session key is mistaken for a metadata-backed ACP runtime session.

## Best Possible Solution

Reserve the `acp:` namespace for metadata-backed runtime sessions, assign metadata-free bridge sessions a distinct namespace, reconcile raw and gateway-canonical bridge keys deterministically, and prove a real first-prompt round trip produces assistant chunks before landing.

## Reproduction Assessment

Yes, from source with high confidence: create a default ACP client session without `_meta`; its `acp:<uuid>` fallback is classified as a runtime ACP session and rejected because no ACP metadata was persisted. The reporter and another user supplied matching observed output, although this read-only review did not execute the live bridge.

## Solution Assessment

Yes in direction. A distinct bridge namespace is the narrowest maintainable fix because it removes the identity collision without weakening fail-closed metadata checks for actual ACP runtime sessions; the final patch must also handle canonical gateway keys and demonstrate the real client path.
