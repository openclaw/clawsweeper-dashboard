## Summary

Keep open: current `main` still has the bridge/runtime-boundary misclassification described here. Metadata-free `openclaw acp client` sessions still fall back to ACP-shaped keys, and ACP dispatch still turns ACP-shaped keys without persisted ACP metadata into `ACP_SESSION_INIT_FAILED` before normal Gateway generation can run.

## What This Changes

The issue reports that `openclaw acp client --server-verbose` fails on the first prompt with `acp_session_init_failed`, echoed input, and no streamed assistant chunks.

## Best Possible Solution

Keep Gateway-backed ACP bridge routing separate from ACPX runtime-session detection so default bridge sessions can generate normally unless explicit ACPX metadata or an intentional ACPX binding exists.

## Reproduction Assessment

Yes, source-reproducible on current `main`. The built-in client creates a metadata-free session, the bridge maps it to `acp:<uuid>`, and ACP dispatch returns the stale-session `ACP_SESSION_INIT_FAILED` path before normal Gateway generation can run.

## Solution Assessment

Yes. The narrow maintainable solution is to stop claiming default bridge fallback keys as ACPX runtime sessions unless persisted ACP metadata or an explicit binding exists, while leaving real ACPX stale-session handling intact.
