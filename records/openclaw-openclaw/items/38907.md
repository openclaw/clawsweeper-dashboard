## Summary

Keep open: current main still lets default Gateway-backed ACP bridge sessions use `acp:<uuid>` keys, while ACPX dispatch treats metadata-free ACP-shaped keys as stale ACP runtime sessions and returns `ACP_SESSION_INIT_FAILED`.

## What This Changes

The issue reports that `openclaw acp client` sessions fail on the first prompt with `acp_session_init_failed`, echoing input and ending the turn without assistant chunks.

## Best Possible Solution

Keep the issue open and make a narrow bridge/runtime boundary fix so default Gateway-backed ACP bridge sessions are not claimed as metadata-backed ACPX runtime sessions unless persisted ACP metadata or an explicit binding exists.

## Reproduction Assessment

Yes. The source path is deterministic on current main: debug `newSession` sends no ACP metadata, the bridge maps it to `acp:<uuid>`, and ACP dispatch treats that metadata-free ACP key as a stale ACPX session before normal Gateway-backed generation can run.

## Solution Assessment

Yes. The narrow maintainable fix is to separate Gateway-backed bridge session routing from ACPX metadata-backed runtime detection while preserving `/acp spawn`, explicit session keys, bindings, labels, and `loadSession`.
