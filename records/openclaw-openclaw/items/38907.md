## Summary

Current main still appears to contain the reported ACP bridge/runtime boundary bug: the debug ACP client creates a metadata-free bridge session, the bridge default still uses an `acp:<uuid>` Gateway session key, and ACP reply dispatch treats ACP-shaped keys without stored ACP metadata as stale `ACP_SESSION_INIT_FAILED` instead of falling through to normal Gateway-backed generation.

## What This Changes

The issue asks for `openclaw acp client` bridge sessions to stream assistant chunks instead of failing first prompt handling with `acp_session_init_failed`.

## Best Possible Solution

Default Gateway-backed ACP bridge sessions should use a key shape or dispatch distinction that ACP runtime dispatch does not claim as metadata-backed ACPX state, while preserving explicit `_meta.sessionKey`, `_meta.sessionLabel`, CLI `--session`, `loadSession`, `/acp spawn`, configured ACP bindings, and `sessions_spawn({ runtime: "acp" })` behavior. Add regression coverage for the default first-prompt path proving `agent_message_chunk` output instead of `ACP_SESSION_INIT_FAILED`.
