## Label Justifications

## Summary

Keep this open: current main still has a source-visible collision between Gateway-backed ACP bridge sessions and ACP runtime sessions, so the first bridge prompt can be treated as a stale ACP runtime session instead of producing assistant chunks.

## What This Changes

The issue reports that `openclaw acp client` creates an ACP bridge session that fails on the first prompt with `acp_session_init_failed`, echoing input and ending the turn without assistant chunks.

## Best Possible Solution

Separate Gateway-backed ACP bridge sessions from ACP runtime session detection, or pass an explicit bridge-session signal, so bridge-created sessions can run normal Gateway chat while stale ACP runtime sessions still fail closed.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `newSession` creates an `acp:<uuid>` bridge key, Gateway canonicalizes it to `agent:<agent>:acp:<uuid>`, and the agent command path throws the missing-metadata ACP error before a normal turn can stream. I did not execute the live client in this read-only review.

## Solution Assessment

No, the current behavior is not the best solution because the bridge's documented Gateway-backed sessions share a key shape with ACP runtime sessions. The safer fix is to distinguish bridge-created sessions from ACP runtime sessions without weakening the existing fail-closed guard for real stale ACP runtime metadata.
