## Summary

Keep open: current main still routes default ACP bridge sessions through an ACP-shaped fallback key without persisted ACP metadata, so the first prompt can be claimed as a stale ACP runtime session and return `acp_session_init_failed`. Open PR #62552 targets the same key-collision fix but is still open and previously needed changes.

## What This Changes

The issue requests that `openclaw acp client` default bridge sessions stream assistant chunks instead of failing with `acp_session_init_failed` on the first prompt.

## Best Possible Solution

Land a narrow, rebased ACP bridge fix that keeps metadata-free Gateway-backed bridge sessions out of ACPX metadata-backed runtime detection, preserves explicit `/acp spawn` sessions, updates docs/changelog, and adds focused regressions.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw acp client` creates a metadata-free session, `newSession` maps it to `acp:<uuid>`, and ACP dispatch classifies ACP-shaped keys without `SessionAcpMeta` as stale; the issue comments also include a confirmed macOS repro.

## Solution Assessment

Yes. The narrow maintainable solution is to separate default Gateway-backed ACP bridge session keys or dispatch fallthrough from ACPX metadata-backed session detection while preserving explicit ACP runtime sessions and bindings.
