## Summary

Keep open: current `main` and `v2026.5.12` still create metadata-free ACP bridge sessions with an `acp:` fallback key, while ACP-shaped keys without ACP metadata are treated as stale runtime sessions. The linked fix PR is still open, so the issue should remain open until a narrowed/rebased fix lands.

## What This Changes

The issue reports that `openclaw acp client` sessions fail on the first prompt with `acp_session_init_failed`, echoing the user input and ending the turn without assistant chunks.

## Best Possible Solution

Land a focused ACP bridge-key fix that uses a non-ACPX namespace for metadata-free bridge sessions and preserves pending-prompt matching across raw and Gateway-canonical keys.

## Reproduction Assessment

Yes, at source level. Current `main` creates metadata-free ACP bridge sessions with `acp:<uuid>`, and ACP runtime resolution treats ACP-shaped keys without stored ACP metadata as `ACP_SESSION_INIT_FAILED`.

## Solution Assessment

Yes for the core direction: separating bridge fallback keys from ACP runtime keys is the narrow maintainable fix. The existing PR should be reviewed and likely narrowed rather than duplicating a second repair job.
