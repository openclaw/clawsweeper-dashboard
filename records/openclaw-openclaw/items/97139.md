## Summary

Keep open: this PR remains useful because current main still creates the OpenAI Responses SDK client without the guarded fetch hook. The patch is a narrow match for the OpenAI SDK custom-fetch contract and the existing transport-stream pattern, but maintainers still need to accept the compatibility and availability impact of applying guarded SSRF and stream-size failure modes to this provider.

## What This Changes

The PR injects `buildGuardedModelFetch(model)` into the OpenAI Responses SDK client and adds a guarded-fetch regression test for oversized streaming JSON bodies.

## Best Possible Solution

Land the narrow guarded-fetch wiring once maintainers accept the stricter provider policy; keep broader non-OK body capping in a shared follow-up rather than expanding this branch.

## Reproduction Assessment

Yes. Current main constructs the OpenAI Responses SDK client without a custom fetch, and the pinned SDK uses its configured fetch for requests while its SSE chunk iterator concatenates bytes until a double newline; I did not rerun tests in this read-only review.

## Solution Assessment

Yes for the provider wiring. Injecting the existing guarded fetch into the SDK constructor is the narrowest maintainable fix for OK streaming responses because the OpenAI SDK supports that hook and sibling transport clients already use it; the non-OK body cap should remain a shared follow-up.
