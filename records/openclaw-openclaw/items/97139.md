## Summary

Keep open: the PR uses the right SDK fetch seam, but as written it does not bound the native OpenAI Responses SSE path because the shared sanitizer opts out for `provider: "openai"` at `api.openai.com`. ## What This Changes

The PR injects `buildGuardedModelFetch(model)` into the OpenAI Responses SDK client and adds regression coverage for oversized streaming JSON through that guarded fetch path.

## Best Possible Solution

Keep the narrow SDK fetch-hook approach, but make native OpenAI Responses streams use a bounded sanitizer and add regression plus real-server proof for the default `api.openai.com` path; leave non-OK body policy to a shared follow-up only if that scope is explicit.

## Reproduction Assessment

## Solution Assessment

No. The SDK fetch hook is the right seam, but this implementation only proves the custom/proxy endpoint path; the best fix is to bound the native OpenAI Responses stream path and test that exact default endpoint behavior.
