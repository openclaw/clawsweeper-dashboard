## Summary

Keep open. Current main still has the reported runner/provider gap: Telegram reaches the shared embedded Pi runner, tools are built and allowlisted during session creation, but there is no pre-prompt reapply/diagnostic on main and Anthropic only serializes the `context.tools` it receives. The open, unmerged PR #75496 explicitly targets this issue, so the issue should stay open until that PR or an equivalent fix is reviewed, validated, and merged.

## What This Changes

Reports that Telegram direct-Anthropic runs lose the computed coding tool set between embedded runner setup and Anthropic transport, causing `tools: []` at the provider and plain-text `tool_use` output.

## Best Possible Solution

Review and land PR #75496 or an equivalent narrow runner fix that preserves the active tool allowlist immediately before embedded Pi prompt dispatch, with regression coverage for non-empty Telegram/direct-Anthropic tools and no fallback that masks legitimate raw no-tool probes.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with version, macOS/Telegram/direct-Anthropic setup, config values, and debug probes showing tools built in the runner but empty at provider dispatch; current main still matches the static gap.

## Solution Assessment

Unclear until the linked PR is reviewed and validated live. Reapplying the session tool allowlist immediately before prompt dispatch is the narrow plausible fix; adding an Anthropic transport fallback from empty `context.tools` would risk hiding legitimate no-tool runs.
