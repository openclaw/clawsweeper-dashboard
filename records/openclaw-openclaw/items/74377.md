## Summary

Keep open. Current main still has the reported Telegram -> embedded Pi -> Anthropic boundary: OpenClaw builds and registers run tools, but the Anthropic transport only forwards the `context.tools` array it receives, and there is still no current-main regression proving a Telegram-origin direct-Anthropic run preserves a non-empty provider-facing tool array.

## What This Changes

The issue reports that Telegram-origin direct Anthropic runs compute non-empty OpenClaw tools in the embedded runner but send `tools: []` to Anthropic, causing tool requests to appear as plain text instead of API tool_use blocks.

## Best Possible Solution

Add a focused regression that drives a Telegram-origin embedded run far enough to observe the direct Anthropic stream context, then fix the runner/session handoff so OpenClaw-managed custom tools remain active in Pi state whenever the computed effective tool set is non-empty.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path with version, channel, provider, config, model, and debug probes; current main also lacks an automated test that proves the provider-facing `context.tools` array stays non-empty.

## Solution Assessment

Not applicable as a patch yet. The narrow maintainable direction is to preserve tool state through Pi session creation/active-tool refresh, not to paper over the symptom in the Anthropic transport.
