## What This Changes

Fix the Telegram/direct-Anthropic regression where the embedded runner computes tools but the Anthropic provider receives `tools: []`, causing tool use to appear as plain text instead of API tool-use blocks.

## Best Possible Solution

Review and land PR #75496 or an equivalent narrow fix that preserves the active tool allowlist immediately before embedded Pi prompt dispatch, with regression coverage proving Telegram/direct-Anthropic runs deliver non-empty tools while legitimate raw no-tool probes remain no-tool.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path with OpenClaw version, macOS/Telegram/direct-Anthropic setup, config values, and debug probes showing tools built in the runner but empty at provider dispatch.

## Solution Assessment

Unclear until the linked PR is reviewed and validated live. Reapplying the session tool allowlist immediately before prompt dispatch is a narrow plausible fix, while adding an Anthropic transport fallback from empty `context.tools` would risk masking legitimate no-tool runs.
