## Summary

Keep open. Current main still matches the reported runner/provider gap: Telegram enters the shared inbound reply runner, tools are built for the embedded Pi session, but the active tool allowlist is only applied at session setup and Anthropic serializes only the `context.tools` it receives; open PR #75496 explicitly targets this issue and is not merged.

## What This Changes

Telegram/direct-Anthropic turns should preserve non-empty coding tools through Pi prompt dispatch so Anthropic receives actual tool definitions instead of `tools: []`.

## Best Possible Solution

Review and land PR #75496 or an equivalent narrow runner fix that reapplies the active tool allowlist before prompt dispatch while preserving legitimate raw no-tool probes and adding regression coverage.

## Reproduction Assessment

Yes. The reporter provided version, platform, Telegram/direct-Anthropic config, and debug logs showing tools built in the runner but empty at provider dispatch; current main still has the same static gap.

## Solution Assessment

Unclear until the linked PR is reviewed and validated live. A runner-side pre-prompt allowlist reapply is the narrow plausible fix, while an Anthropic transport fallback from empty `context.tools` would risk masking legitimate no-tool runs.
