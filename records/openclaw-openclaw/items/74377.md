## Summary

Keep open. Current main and v2026.5.7 still lack a prompt-boundary active-tool allowlist reapply, and https://github.com/openclaw/openclaw/pull/75496 explicitly targets this issue but is still open.

## What This Changes

Telegram/direct-Anthropic runs can reach the provider with `tools: []` even after the embedded runner computed and allowlisted tools.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75496 or an equivalent runner-side fix that restores the policy-filtered active tools immediately before prompt dispatch while preserving legitimate raw no-tool probes.

## Reproduction Assessment

Yes, source-reproducible. The reporter provided concrete Telegram/direct-Anthropic steps and logs, and current source still shows tools set before prompt dispatch while Anthropic forwards `context.tools`; I did not rerun the live Telegram setup.

## Solution Assessment

Yes. Reapplying the already policy-filtered Pi allowlist at the prompt boundary is the narrow maintainable fix; adding an Anthropic transport fallback for empty tools would risk masking legitimate no-tool runs.
