## Summary

Keep open. Current main still has the reported runner/provider gap: Telegram enters the shared embedded Pi runner, OpenClaw tools are built and allowlisted once at session creation, prompt dispatch does not reapply that allowlist, and Anthropic serializes only the `context.tools` it receives; open PR #75496 already targets this issue and should be reviewed instead of closing early.

## What This Changes

The issue reports that Telegram direct-Anthropic runs build coding tools in the embedded runner but pass an empty tools array to the Anthropic provider, blocking tool execution.

## Best Possible Solution

Review and land PR #75496 or an equivalent narrow runner fix that restores the session tool allowlist immediately before prompt dispatch without adding an Anthropic transport fallback that masks legitimate no-tool runs.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with version, channel, provider, config, and debug logs; current-main static tracing still matches the reported Telegram-to-runner-to-Anthropic propagation gap.

## Solution Assessment

Unclear until the linked PR is reviewed and live-validated. Reapplying the active tool allowlist before prompt dispatch is the narrow plausible fix, while falling back inside Anthropic transport would hide legitimate empty-tool probes.
