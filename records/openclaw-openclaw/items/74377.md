## Summary

Keep open. Current main still has the reported runner/provider gap: Telegram reaches the shared embedded runner, the runner builds and allowlists tools once during session setup, and the Anthropic transport serializes only the `context.tools` it receives. The linked closing PR #75496 is open and unmerged, so this issue should remain open until that PR is reviewed, validated, and merged or replaced.

## What This Changes

The issue reports that Telegram/direct-Anthropic runs compute tool definitions in the embedded runner but send an empty `tools` array to the Anthropic provider.

## Best Possible Solution

Review and validate PR #75496, or a narrow replacement, that re-applies the OpenClaw-managed tool allowlist immediately before embedded Pi prompt dispatch while preserving raw no-tool model probes and adding regression coverage for the active-tool wipe path.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path with OpenClaw 2026.4.22, Telegram DM, direct Anthropic, tool config, and debug probes showing 17 runner tools becoming an empty provider context; this review did not run the live Telegram setup.

## Solution Assessment

Unclear until the linked PR is reviewed and live-validated. Re-applying the session allowlist at prompt dispatch is a narrow plausible fix, while adding a transport fallback from empty `context.tools` to runner tools would risk masking legitimate no-tool calls.
