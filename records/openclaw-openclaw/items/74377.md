## Summary

Keep open. Current main still shows the reported runner/provider gap: Telegram reaches the shared embedded runner, the runner computes and allowlists tool definitions once, prompt dispatch later has no active-tool reapply hook, and the Anthropic transport serializes only the `context.tools` array it receives. The linked closing PR #75496 is still open and unmerged, so the issue should remain open until that PR is reviewed, validated, and merged or replaced.

## What This Changes

The issue reports a Telegram/direct-Anthropic regression where the embedded Pi runner computes tool definitions but Anthropic receives an empty tools array, preventing real tool calls.

## Best Possible Solution

Review and validate PR #75496, or a replacement with the same narrow boundary: preserve or reapply the embedded runner's active tool allowlist immediately before Pi prompt dispatch, add focused regression coverage for non-empty provider tools, and avoid an Anthropic transport fallback that would mask legitimate raw no-tool runs.

## Reproduction Assessment

Yes. The report gives a concrete OpenClaw version, Telegram/direct-Anthropic setup, config, and debug probes, and current main source tracing confirms that path reaches the embedded runner and then the Anthropic payload builder.

## Solution Assessment

Unclear for the proposed PR until reviewer and live validation are complete. The safest direction is a narrow runner-side active-tool preservation fix; a transport fallback from empty context.tools to effectiveTools would be broader and could break intentional no-tool probes.
