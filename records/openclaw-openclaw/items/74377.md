## Summary

Keep open. Current main still has the reported runner/provider gap: the embedded runner builds and allowlists tool definitions, but the Anthropic transport only serializes the `context.tools` it receives, and the linked fix PR is still open rather than merged. The issue has a concrete Telegram/direct-Anthropic reproduction, fresh discussion, and an open closing PR, so it should stay open until that PR is reviewed, validated, and merged or replaced.

## What This Changes

The issue reports that Telegram-origin Anthropic runs receive an empty provider-facing tools array even though the embedded runner computed 17 effective tools, causing tool calls to be emitted as plain text instead of API `tool_use` blocks.

## Best Possible Solution

Review and validate PR #75496 or a replacement that preserves the active tool allowlist immediately before embedded Pi prompt dispatch, with regression coverage proving Telegram/direct-Anthropic runs deliver non-empty tool definitions to the provider while legitimate raw no-tool probes remain no-tool.

## Reproduction Assessment

Yes. The issue provides a high-confidence live reproduction path with version, Telegram channel, direct Anthropic provider, config, and debug probes; the discussion further narrows the failure to embedded runner active-tool state before the Anthropic call.

## Solution Assessment

Unclear until the linked PR is reviewed and live-validated. Re-applying the session tool allowlist before prompt dispatch is a narrow plausible fix, while a transport fallback from empty `context.tools` to runner tools would risk masking legitimate no-tool requests.
