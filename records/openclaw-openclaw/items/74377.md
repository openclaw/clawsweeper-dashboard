## Summary

Keep open. Current main still matches the reported runner/provider gap: Telegram uses the shared inbound runner, the embedded runner builds and allowlists tools only at session setup, prompt dispatch does not reapply that allowlist, and Anthropic serializes only the `context.tools` it receives. The linked closing PR is still open, so this issue should remain open until that PR or an equivalent fix lands. ([mintlify.com](https://www.mintlify.com/badlogic/pi-mono/api/coding-agent/extension-api))

## What This Changes

Telegram/direct-Anthropic runs should preserve the runner's computed coding tools through Pi prompt dispatch so Anthropic receives real tool definitions instead of `tools: []`.

## Best Possible Solution

Review and land PR #75496, or an equivalent narrow runner fix that reapplies the active tool allowlist before prompt dispatch while preserving legitimate raw no-tool probes.

## Reproduction Assessment

Yes. The reporter provided version, platform, Telegram/direct-Anthropic config, and debug logs showing non-empty runner tools becoming provider-facing empty tools; current main still has the same static gap.

## Solution Assessment

Yes, the linked runner-side direction appears to be the narrow maintainable fix. A provider-side fallback from empty `context.tools` would risk masking legitimate no-tool runs.
