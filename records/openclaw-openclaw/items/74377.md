## Summary

Keep open. Current main still has the reported runner/provider gap: Telegram enters the shared inbound reply runner, the embedded runner builds and allowlists tools only during session setup, prompt submission does not reapply the allowlist, and the Anthropic transport forwards `context.tools` as received. PR #75496 explicitly closes this issue but is still open/unmerged, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Fix the Telegram direct-Anthropic runner path so computed coding tools arrive at the provider as API tool definitions instead of `tools: []`.

## Best Possible Solution

Review and land PR #75496 or an equivalent narrow runner fix that preserves or reapplies the session tool allowlist immediately before prompt dispatch, with regression coverage for Telegram/direct-Anthropic non-empty tools and legitimate raw no-tool runs.

## Reproduction Assessment

Yes. The issue provides a concrete reproduction path with version, platform, Telegram channel, direct Anthropic provider, config, and debug logs showing tools built in the runner but empty at provider dispatch; this review traced that path through current main.

## Solution Assessment

Yes for keeping the issue open. The safest implementation direction is a runner-level active-tool preservation/reapply fix, while an Anthropic transport fallback from empty `context.tools` would risk masking legitimate no-tool probes.
