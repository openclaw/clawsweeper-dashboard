## Summary

Keep this PR open: the release-drift guard is a useful idea, but current head would not detect the cited v2026.6.1 versus v0.6.3 drift because its latest-release lookup and bundled-version extraction do not match the real GitHub release contract, and the contributor proof is still test/snippet-only.

## What This Changes

The branch currently adds only a PowerShell advisory script at `.github/scripts/check-windows-node-version.ps1` to compare the Windows Hub version promoted on an OpenClaw release with the latest `openclaw-windows-node` release.

## Best Possible Solution

Make the promoted Windows Hub source version a durable contract, either by parsing the existing release verification marker deliberately or by having the promotion workflow publish machine-readable version metadata, then cover that shape with tests and a real script run.

## Reproduction Assessment

Yes, from source and live release inspection. The current v2026.6.1 manifest has only checksums, the release body records v0.6.0 in a different marker shape, `openclaw-windows-node` latest is v0.6.3, and GitHub CLI treats latest release lookup as the no-tag path.

## Solution Assessment

No. The idea is useful, but this implementation guesses metadata that the current promotion workflow does not publish and the default latest lookup fails before comparison; the maintainable fix is to make the promoted source version an explicit release contract.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P1] Call the latest-release API instead of tag `latest`:** `.github/scripts/check-windows-node-version.ps1:165`
  - body: The default path passes `-Tag "latest"` into `Get-VersionFromRelease`, which calls `gh release view $Tag`. GitHub CLI fetches the latest release only when no tag argument is supplied, and the companion repo has no release tag named `latest`, so the script can exit before doing any drift comparison. Treat `latest` as a special case or call the `/releases/latest` API.
  - confidence: 0.93
- **[P1] Use the promoted release contract for bundled version:** `.github/scripts/check-windows-node-version.ps1:73-91`
  - body: This parser looks for a `# Version:` line in `OpenClawCompanion-SHA256SUMS.txt` or `openclaw-windows-node@...` in the release body, but the promotion workflow writes only checksum lines and v2026.6.1 records the source as a release URL marker. With the real release shape, `$bundledVersion` stays null, so `-FailOnLag` reports an undetermined version instead of the actual v0.6.0 to v0.6.3 lag.
  - confidence: 0.95
