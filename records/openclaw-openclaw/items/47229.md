## Summary

Keep open: current main still has a source-proven Anthropic streamed tool argument precision-loss path. The original SDK-specific interception suggestion is partly obsolete because normal Anthropic Messages transport now uses OpenClaw's custom SSE client, but that client still parses `input_json_delta` tool JSON with a dependency path that starts with plain `JSON.parse`.

## What This Changes

The issue requests preserving large integer literals in Anthropic streamed `tool_use.input` arguments so IDs such as Discord snowflakes are not rounded before tool execution.

## Best Possible Solution

Parse Anthropic streamed tool input with an unsafe-integer-preserving parser in the current transport path, with focused regression coverage for unsafe and safe integers and no dependency on Ollama plugin internals.

## Reproduction Assessment

Yes, at source level. Current main appends `input_json_delta` into `partialJson` and parses it with `parseStreamingJson`, whose dependency starts with `JSON.parse`; a focused Node probe shows the reported literal is rounded by that primitive parse.

## Solution Assessment

No, the SDK monkey-patch suggested in the report is no longer the best current fix because normal Anthropic transport now uses OpenClaw's custom SSE client. The narrow maintainable solution is to preserve unsafe integer literals while parsing `block.partialJson` in `src/agents/anthropic-transport-stream.ts`.
