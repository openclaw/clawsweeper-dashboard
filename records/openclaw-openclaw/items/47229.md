## Summary

Keep open: current main no longer relies on the Anthropic SDK MessageStream path for normal Anthropic transport, but the same precision-loss bug remains because streamed `input_json_delta` tool arguments are still parsed through a plain JSON parse path before OpenClaw sees the final tool call.

## What This Changes

Preserve unsafe integer literals as strings when parsing Anthropic Messages `tool_use` input JSON during streaming.

## Best Possible Solution

Add a shared unsafe-integer-preserving JSON parser for Anthropic streamed tool arguments, reuse the existing Ollama logic without crossing plugin-private boundaries, and cover both unsafe and safe integer cases in focused Anthropic transport tests.

## Reproduction Assessment

Yes, at source level: current main appends `input_json_delta` fragments and passes complete tool JSON to `parseStreamingJson`, whose dependency implementation uses `JSON.parse`. The primitive parse rounds the reported Discord snowflake literal, so the code path is high-confidence even though I did not run the full Vitest stream test in this read-only review.

## Solution Assessment

No, the SDK monkey-patch is no longer the narrowest fix for current main because the main Anthropic transport now uses a custom SSE client. The maintainable fix is to parse Anthropic tool input with an unsafe-integer-preserving parser before assigning `block.arguments`, plus regression coverage.
