## Label Justifications

- `P2`: This is a normal-priority installer hardening fix with limited surface area but real security and availability implications.
- `merge-risk: 🚨 compatibility`: The PR changes a previously permissive installer path to reject third-party setup scripts unless their first line starts with a shebang.
- `merge-risk: 🚨 availability`: If an upstream setup script format changes benignly, fresh installs can fail before Homebrew or NodeSource setup runs.

## Summary

This PR remains useful and should stay open: current main still downloads and executes Homebrew and NodeSource setup scripts without structural validation, while the PR adds a focused fail-closed guard with adequate terminal proof. I found no blocking correctness finding, but the strict shebang contract changes installer availability behavior and needs installer/security owner acceptance before merge.

## What This Changes

The PR adds `validate_downloaded_script()` to `scripts/install.sh`, applies it before executing Homebrew and NodeSource setup scripts, and adds a standalone shell harness for malformed-download cases.

## Best Possible Solution

Land centralized downloaded-script validation after owner acceptance of the strict shebang policy, or narrowly widen the validator for reviewed harmless prefixes while preserving malformed-download rejection.

## Reproduction Assessment

Yes. Current main is source-reproducible because `scripts/install.sh` downloads temp files and runs `/bin/bash`, `bash "$tmp"`, or `sudo bash "$tmp"` on the Homebrew and NodeSource paths; the PR body adds after-fix terminal proof against malformed HTTP and HTTPS responses.

## Solution Assessment

Yes, with maintainer acceptance of the strict policy. A centralized validation helper at the execution boundary is the narrow maintainable fix; the only open choice is whether first-line shebang validation should tolerate reviewed harmless prefixes.

## Risks / Open Questions

- The validator intentionally rejects any downloaded setup script whose first line does not start with `#!`; if Homebrew or NodeSource later publish a Bash-compatible script with a UTF-8 BOM, leading blank line, or leading comment, OpenClaw installs can fail before package setup.
- This changes installer behavior from permissive execution to fail-closed validation for third-party setup downloads, so maintainers should explicitly accept the availability tradeoff.
- The malformed-download harness is useful proof but is not wired into the maintained installer test or CI lanes, so regression coverage after merge depends on maintainer choice.
- The same author has another open NodeSource installer PR at https://github.com/openclaw/openclaw/pull/82926; maintainers should reconcile the overlapping `scripts/install.sh` area before landing both.
