## Summary

Keep open for maintainer review: the unvalidated remote-script execution path is still present on current main and in the latest release, and this PR is a focused hardening change with sufficient terminal proof. ## What This Changes

The PR adds `validate_downloaded_script` to `scripts/install.sh` before Homebrew and NodeSource setup scripts are executed, plus a standalone shell validation harness.

## Best Possible Solution

Land centralized downloaded-script validation after installer owners accept the strict shebang contract, preferably with CI-backed malformed-download coverage if they want a long-term guardrail.

## Reproduction Assessment

Yes from source inspection: current main and the latest release download Homebrew and NodeSource setup scripts into temp files and invoke bash on them without content validation. I did not run a malformed-download repro in the checkout because this review is read-only.

## Solution Assessment

Likely yes, subject to installer-owner policy. Centralized non-empty and shebang validation at the remote-script execution boundary is a narrow maintainable hardening, but the strict byte-zero `#!` contract should be an explicit maintainer choice.

## Risks / Open Questions

- The new byte-zero shebang rule intentionally fails closed if Homebrew or NodeSource later publish a bash-compatible setup script with a UTF-8 BOM, leading blank/comment line, or no shebang.
- A future third-party formatting change would surface as an OpenClaw install-time failure rather than falling through to bash, so installer owners should explicitly accept that availability tradeoff.
- The added malformed-download coverage is a standalone shell harness rather than the existing `test/scripts/install-sh.test.ts` lane, so CI may not protect the helper unless maintainers wire it in.
