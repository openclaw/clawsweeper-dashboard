## Summary

Keep open: current main and the latest release still execute downloaded Homebrew and NodeSource setup scripts without structural validation, and this PR addresses a real installer hardening gap. The remaining actionable issue is narrow: the new harness uses a GNU-only C1 assertion, while the stricter fail-closed installer behavior should be an explicit maintainer acceptance point before merge.

## What This Changes

The branch adds `validate_downloaded_script()` to `scripts/install.sh`, checks Homebrew and NodeSource downloaded setup scripts before `bash`, sanitizes rejected first-line diagnostics, and adds `scripts/test-install-validation.sh`.

## Best Possible Solution

Keep the installer validation, replace the GNU-only harness assertion with a portable byte check, then have maintainers explicitly accept the fail-closed behavior for Homebrew and NodeSource setup scripts.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.22` shows downloaded Homebrew and NodeSource setup scripts are executed from temp files without a validation step; this review did not run a live malicious download because the task is read-only.

## Solution Assessment

Mostly yes. Structural non-empty and shebang validation is a bounded hardening path and current upstream scripts satisfy it, but the harness portability issue should be fixed and the fail-closed installer behavior needs maintainer acceptance.

Full review comments:

- **[P3] Use a portable C1 diagnostic assertion:** `scripts/test-install-validation.sh:140`
  - body: The harness uses `grep -qP` to detect control bytes. BSD/macOS `grep` does not support `-P`; because the command is negated, that unsupported exit status makes the assertion pass instead of proving bytes were stripped, so the harness can report a false green on macOS. Use a portable check such as Perl or an explicit byte loop.
  - confidence: 0.87
