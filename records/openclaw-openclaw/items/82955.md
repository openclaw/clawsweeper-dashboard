## Label Justifications

- `P2`: This is a normal-priority installer security hardening with limited surface area but real compatibility and availability implications.
- `merge-risk: 🚨 compatibility`: Existing installs that previously attempted to execute any downloaded body will now reject empty or non-shebang setup content before running Homebrew or NodeSource scripts.
- `merge-risk: 🚨 availability`: A future upstream script format change or proxy/CDN response that lacks a leading shebang would make the installer fail closed before dependency setup.

## Summary

Keep open for maintainer review: the patch is a focused installer hardening with sufficient terminal proof and no blocking code findings, but it intentionally changes remote installer/setup script handling from execute-any-download to fail-closed on empty or non-shebang content.

## What This Changes

The PR adds `validate_downloaded_script` to `scripts/install.sh`, applies it before Homebrew and NodeSource downloaded scripts are executed, and adds a standalone shell validation harness.

## Best Possible Solution

Land the centralized fail-closed validation only after maintainers accept the stricter installer contract, keeping the regression proof focused on malformed downloads and current Homebrew/NodeSource script shape.

## Reproduction Assessment

Yes from source inspection and supplied live-output proof: current main downloads remote setup content and runs it with bash, while the PR demonstrates malformed HTTP/HTTPS bodies are rejected after the patch. I did not execute the full installer path locally because this review was read-only.

## Solution Assessment

Yes, with maintainer acceptance of the policy change: central non-empty/raw-shebang validation is a narrow hardening that avoids brittle checksum pins for frequently changing third-party scripts. The main remaining question is whether fail-closed behavior is acceptable for the installer contract.
