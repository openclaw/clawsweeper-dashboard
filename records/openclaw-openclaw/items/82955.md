## Summary

Keep open: the installer hardening is useful and the contributor supplied strong terminal proof, but the proposed validator still checks the shebang after storing bytes in a Bash variable, so NUL-bearing payloads can pass the new validation gate. That needs a small repair before merge, plus maintainer acceptance of the new fail-closed installer behavior.

## What This Changes

The PR adds a `validate_downloaded_script()` non-empty/shebang check before executing Homebrew and NodeSource setup scripts and adds a shell validation harness.

## Best Possible Solution

Keep the centralized validation, but make the header decision raw-byte safe, add NUL/C1/bad-download fixtures, and have maintainers explicitly accept the fail-closed installer policy for these third-party setup scripts.

## Reproduction Assessment

Yes for the blocking PR defect: a focused shell probe using the same command-substitution pattern shows Bash drops NUL bytes and the `#!` prefix check accepts the payload. Current main has no validator, so the check is against the proposed patch path rather than a current-main bug reproduction.

## Solution Assessment

No, not yet. Centralizing validation before remote script execution is the right direction, but the implementation should make the shebang decision from raw bytes and then preserve the existing installer flow once maintainers accept the fail-closed policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Check raw header bytes before accepting shebang:** `scripts/install.sh:98-99`
  - body: `first_line` is populated through command substitution, and Bash drops NUL bytes there. A response beginning with `#!\0...` is normalized into a string that still matches `#!`, so the new validator can pass a binary/truncated payload into `/bin/bash` instead of rejecting it. Compare the first two bytes directly, and add a NUL-prefixed fixture so this stays covered.
  - confidence: 0.9
- **[P3] Use a portable C1 diagnostic assertion:** `scripts/test-install-validation.sh:140`
  - body: `grep -P` is not available on macOS/BSD grep, and because it is under `!`, an unsupported option can make the C1 assertion pass for the wrong reason. Use a portable byte check such as `od`, `perl`, or a `tr` round-trip so the harness actually proves the sanitizer on every supported shell environment.
  - confidence: 0.82
