## Best Possible Solution

Land the hardening if maintainers accept fail-closed installer semantics for malformed third-party setup-script downloads, preserving the current main installer guards while adding this structural validation.

## Reproduction Assessment

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff: a raw-byte non-empty/shebang check is a narrow hardening step and avoids brittle hashes for third-party scripts that change frequently. If maintainers do not want fail-closed installer behavior, the safer product alternative is a softer warning or explicit strict mode.
