## Summary

Keep this PR open: the structural validation is useful and well scoped, but the Homebrew `run_remote_bash` path still validates after an unchecked failed download, so a partial `#!`-prefixed response can still execute.

## What This Changes

The PR adds `validate_downloaded_script()` to `scripts/install.sh`, applies it to Homebrew and NodeSource download-then-execute paths, and adds a shell validation harness.

## Best Possible Solution

Keep the new structural validation, but first propagate `download_file` failure before validation/execution and cover the failed-partial-download case.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a downloader that writes a `#!`-prefixed partial file and exits non-zero would reach validation and execution in `run_remote_bash`. I did not run the harness because this review is read-only.

## Solution Assessment

No, not yet: structural validation is the right direction, but it must return on failed downloads before validating or executing the temp file.
