## Summary

Keep this PR open for maintainer review: it is a focused installer hardening with sufficient terminal proof and no blocking correctness finding, but it intentionally changes installer compatibility by rejecting third-party setup scripts unless they start with a first-line shebang.

## What This Changes

The PR adds a `validate_downloaded_script()` guard before Homebrew and NodeSource setup scripts are executed, plus a standalone shell validation harness.

## Best Possible Solution

Land the validation hardening if installer owners accept the fail-closed shebang policy; otherwise preserve compatibility by making stricter structural validation an explicit installer policy or mode.

## Reproduction Assessment

Yes. Current main is source-reproducible: the Homebrew and NodeSource paths download remote setup scripts into temp files and execute them with bash before any structural validation.

## Solution Assessment

Yes if maintainers accept the fail-closed contract: the helper covers all current download-then-execute paths and current upstream scripts start with shebangs. If that policy is too strict for installer compatibility, the safer solution is an explicit compatibility or strict-mode decision before merge.

## Risks / Open Questions

- The new first-line shebang requirement fails closed if Homebrew or NodeSource later publishes a bash-compatible setup script with a BOM, leading blank line, or no shebang, so existing install flows could stop at runtime until OpenClaw adjusts the validator.
- The standalone shell harness copies the validation logic instead of exercising `scripts/install.sh` through the existing Vitest installer lane, so future drift would not be caught by normal installer tests unless maintainers wire or port the cases.
- I did not run the full live macOS/Linux installer flow in this read-only review; the PR body’s real proof covers local HTTP/HTTPS malformed responses and valid scripts.
