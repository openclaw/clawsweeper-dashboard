## Summary

## What This Changes

The PR adds `validate_downloaded_script` to `scripts/install.sh`, calls it before executing Homebrew and NodeSource downloads, and adds a standalone shell validation harness.

## Best Possible Solution

Land the structural validation if installer owners accept the fail-closed shebang contract, with durable installer-lane coverage added first if maintainers want regression proof in CI.

Yes, at source level: current main downloads Homebrew and NodeSource setup scripts into temp files and executes them with `/bin/bash`, `bash`, or `sudo bash` without content checks. I did not run a full installer repro in this read-only review.

## Solution Assessment

Yes, with a maintainer policy caveat: non-empty plus first-line shebang validation and sanitized diagnostics is a narrow hardening layer, but unconditional fail-closed installer behavior needs owner acceptance.

## Risks / Open Questions

- The new fail-closed shebang requirement can stop installs if Homebrew or NodeSource later serves a bash-compatible setup script with a BOM, blank/comment preamble, or no shebang, even though current upstream scripts start with `#!/bin/bash`.
- The malformed-download cases are proven by PR-body terminal output and a standalone shell harness, but they are not yet durable coverage in the existing installer Vitest or Docker smoke lanes.
- This is structural validation, not cryptographic trust; a trusted upstream response that begins with a shebang still executes inside the existing third-party-script trust boundary.
