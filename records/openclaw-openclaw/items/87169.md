## Summary

Keep this PR open, but it is not merge-ready: the requested split-tenant Teams Graph support is still missing on current main, and this branch only applies the new tenant in one Graph helper while sibling Teams Graph callers continue to use the bot-tenant token provider.

## What This Changes

The PR adds `channels.msteams.graphTenantId` to Teams config schema/types/docs/generated metadata and changes `resolveGraphToken` to acquire Microsoft Graph tokens from that tenant.

## Best Possible Solution

Land one canonical Teams Graph-tenant implementation that routes every Teams Graph caller through a shared tenant-aware token-provider path, keeps bot tokens on `tenantId`, documents the config surface, and includes redacted real split-tenant proof.

## Reproduction Assessment

Not applicable for the feature request itself; source review shows current main has no `graphTenantId` support and the PR's proposed behavior can be assessed from the token-routing code path.

## Solution Assessment

No. The safer implementation is to pass the optional Graph tenant through the shared Teams token-provider path using the upstream `getGraphToken(tenantId?)` contract, rather than cloning credentials in only `resolveGraphToken`.
