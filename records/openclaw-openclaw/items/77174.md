## Summary

Keep open. The PR targets a real BlueBubbles typing cleanup gap, but the proposed implementation can turn an already-successful reaction into a failed tool action when the stop-typing request rejects.

## What This Changes

The PR exports BlueBubbles typing through the action runtime and adds a reaction-action test expecting the react handler to clear typing after sending a reaction.

## Best Possible Solution

Keep the reaction cleanup, but make stop-typing best-effort so reaction success is preserved even if the BlueBubbles DELETE cleanup fails.

## Reproduction Assessment

Yes, at source level: current main's BlueBubbles react handler sends the reaction and returns without a stop-typing call, so a focused unit test can reproduce the missing cleanup. I did not live-test a BlueBubbles server in this read-only review.

## Solution Assessment

No. Clearing typing after the reaction is the right boundary, but awaiting the unreliable cleanup request is not the safest implementation; align it with the existing catch-and-log stop pattern.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep typing cleanup from failing successful reactions:** `extensions/bluebubbles/src/actions.ts:253`
  - body: The reaction has already been sent when this new awaited cleanup runs. `sendBlueBubblesTyping(false)` performs the BlueBubbles DELETE `/typing` request and propagates non-OK responses, while the existing reply finalizer treats stop-typing as best-effort because that DELETE is documented as unreliable. If the cleanup rejects, this action will now report failure even though the tapback was applied. Catch/log or otherwise swallow cleanup failures so reaction success is preserved.
  - confidence: 0.88
