## Summary

Keep open: this is a narrow, source-backed Telegram bug fix, but the external PR still lacks required live Telegram proof and changes upgrade behavior for no-account sends when an implicit default token and configured default account both exist.

## What This Changes

The PR pre-resolves omitted Telegram account IDs through the configured default before shared fallback resolution, and adds resolver tests for omitted/default and explicit account routing.

## Best Possible Solution

Land the narrow resolver fix after redacted live multi-bot Telegram proof covers omitted-account default routing and explicit-account routing, with maintainer acceptance of the upgrade behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source-level reproduction: current main resolves an omitted Telegram account as the implicit default first and returns it when that account has credentials, bypassing the configured default. I did not establish a live Telegram reproduction.

## Solution Assessment

Is this the best way to solve the issue? Yes for the code shape: the token decision belongs in resolveTelegramAccount, and pre-resolving the configured default matches existing Telegram default selection plus the Nextcloud Talk sibling resolver. It is not merge-ready without live Telegram proof or a maintainer proof waiver.
