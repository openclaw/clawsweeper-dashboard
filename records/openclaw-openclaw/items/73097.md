## Summary

Keep open. Current main supports Claude CLI execution through the explicit `agentRuntime.id: "claude-cli"` contract, and Anthropic CLI migration now writes that contract, but a `cliBackends`-only configuration still does not make native subagent or PI execution switch away from the canonical Anthropic provider path. Because the remaining choice changes runtime and billing semantics, this needs maintainer direction rather than cleanup closure.

## What This Changes

The issue asks for PI/native subagent, model-run, boot, and fallback dispatch to honor configured Claude CLI backends so chat and spawned-agent paths use the same transport for Claude OAuth credentials.

## Best Possible Solution

Maintainers should choose and document one supported contract: either keep Claude CLI selection explicit through `agentRuntime.id: "claude-cli"` with doctor/status guidance for `cliBackends`-only configs, or add a deliberate opt-in that routes eligible Anthropic PI, native subagent, raw model-run, boot, and fallback paths through the configured CLI backend with focused regression coverage.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With `anthropic/*`, configured `agents.defaults.cliBackends.claude-cli`, and no `agentRuntime.id`, current source leaves CLI resolution undefined and the child run reaches the PI/provider path; live Anthropic billing is not needed to prove that transport split.

## Solution Assessment

Unclear. Inferring CLI transport from `cliBackends` alone is not clearly the narrowest maintainable fix because current docs make `agentRuntime` the executor-selection contract; an explicit opt-in or doctor repair may be safer than silent inference.
