## Summary

Keep open. Current main has an explicit provider/model `agentRuntime.id` path that can route Anthropic turns through Claude CLI, but the reported `cliBackends`/Anthropic CLI setup gap is not fully closed because official Claude CLI auth migration still writes legacy whole-agent `agentRuntime`, which current docs and resolver ignore for runtime selection.

## What This Changes

The issue requests that PI/native subagent execution honor Claude CLI backend configuration so the same OAuth credential is not routed through direct Anthropic API transport for subagents and other embedded paths.

## Best Possible Solution

Align the Claude CLI setup path with the documented provider/model runtime contract, or deliberately add an explicit opt-in for cliBackends-backed embedded execution with clear docs and diagnostics.

## Reproduction Assessment

Yes, at source level: the Anthropic CLI auth migration writes a whole-agent runtime key, while the current resolver only reads provider/model runtime policy before deciding whether to call the CLI runner. I did not run a live Anthropic billing replay in this read-only review.

## Solution Assessment

Unclear. Inferring from `cliBackends` would match the issue text but conflicts with the documented `agentRuntime` contract; the safer implementation likely fixes Anthropic CLI setup/migration to write provider/model runtime policy unless maintainers choose a broader cliBackends opt-in.
