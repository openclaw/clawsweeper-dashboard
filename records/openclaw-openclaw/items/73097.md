## Summary

Keep open. Current main still uses `agentRuntime` as the supported Claude CLI execution selector; a `cliBackends`-only config is documented as a text-first fallback surface and still does not make PI/native subagent execution switch away from the Anthropic provider path. The remaining question is a maintainer/runtime-policy decision, not conservative cleanup.

## What This Changes

The issue asks OpenClaw to make PI/subagent execution honor Claude CLI backend configuration so Claude OAuth traffic does not split between CLI and direct Anthropic API transports.

## Best Possible Solution

Maintainers should define one supported contract for Claude CLI execution across chat, PI, native subagents, raw model runs, and fallback paths: either keep `agentRuntime.id: "claude-cli"` as the explicit selector with stronger doctor/docs diagnostics for `cliBackends`-only setups, or add a deliberate opt-in that routes eligible Anthropic PI paths through the configured Claude CLI backend with focused regression coverage.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With canonical `anthropic/*`, configured `agents.defaults.cliBackends.claude-cli`, and no `agentRuntime.id`, current source leaves CLI resolution undefined and the agent/subagent path reaches `runEmbeddedPiAgent`; live Anthropic billing is not needed to verify that transport split.

## Solution Assessment

Unclear. The requested `cliBackends` inference is not obviously the best fix because current docs and migration code make `agentRuntime` the executor-selection contract; an explicit opt-in or doctor repair may be safer than silent inference.
