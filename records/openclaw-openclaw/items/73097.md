## Summary

Keep open. Current main still does not infer Claude CLI execution from a `cliBackends`-only configuration; it uses `agentRuntime.id` as the supported execution selector, and changing that would intentionally alter Anthropic transport and billing behavior, so this needs maintainer runtime-policy judgment rather than conservative cleanup.

## What This Changes

Request PI/subagent/model-run paths to route eligible Anthropic OAuth calls through the configured Claude CLI backend instead of direct Anthropic API when `cliBackends.claude-cli` is configured.

## Best Possible Solution

Define one explicit Claude CLI execution contract across chat, PI, native subagents, raw model runs, and fallback paths: either keep `agentRuntime.id: "claude-cli"` as the required selector with stronger doctor/docs diagnostics for `cliBackends`-only setups, or add a deliberate opt-in that routes eligible Anthropic PI paths through Claude CLI with focused tests.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With an Anthropic model, `agents.defaults.cliBackends.claude-cli`, and no `agentRuntime.id`, source inspection shows CLI resolution remains undefined and the agent/subagent path reaches `runEmbeddedPiAgent`; a live Anthropic billing account is not required to reproduce the transport split.

## Solution Assessment

Unclear. The requested `cliBackends` inference is not obviously the best fix because current docs, migration code, and tests make `agentRuntime.id` the execution selector; an explicit opt-in or diagnostic repair may be safer than silent inference.
