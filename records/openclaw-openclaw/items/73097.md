## Summary

Keep open. Current main supports Claude CLI execution through the explicit `agentRuntime.id: "claude-cli"` contract, and current onboarding/migration can set that contract, but the reported `cliBackends`-only PI/subagent routing asymmetry still exists and changes runtime and billing semantics enough to need maintainer direction. https://github.com/openclaw/openclaw/issues/73097

## What This Changes

The issue asks OpenClaw PI, native subagent, and related model execution paths to honor configured Claude CLI `cliBackends` for eligible Anthropic OAuth routing so they use the same transport path as chat.

## Best Possible Solution

Define one supported contract and make it consistent: either keep Claude CLI selection explicit through `agentRuntime.id: "claude-cli"` with doctor/docs checks for `cliBackends`-only setups, or add a deliberate opt-in policy that lets eligible Anthropic PI, native subagent, raw model-run, and fallback paths route through a configured Claude CLI backend with regression coverage.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With canonical `anthropic/*`, configured `agents.defaults.cliBackends.claude-cli`, and no `agentRuntime.id`, current source leaves execution on the canonical Anthropic provider and the native subagent path reaches `runEmbeddedPiAgent`.

## Solution Assessment

Unclear. Inferring CLI transport from `cliBackends` alone is not clearly the safest fix because current docs and migration code use `agentRuntime` as the executor-selection contract; an explicit opt-in or doctor repair may be safer than silent inference.
