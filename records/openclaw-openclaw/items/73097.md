## Summary

Keep open. Current main documents and implements provider/model `agentRuntime.id` as the executor-selection contract, but `cliBackends`-only configuration still does not select Claude CLI for PI/subagent-style execution, and recent discussion shows this remains a runtime and billing policy question rather than a closeable implemented bug.

## What This Changes

The issue requests that PI/subagent execution honor Claude CLI backend configuration so Claude OAuth-backed chat and subagent/model-helper paths use the same transport and billing behavior.

## Best Possible Solution

Define one documented Claude CLI transport contract across chat, subagents, model helpers, and fallback paths: either enforce provider/model `agentRuntime.id` with clear diagnostics for `cliBackends`-only setups, or add an explicit opt-in that safely delegates eligible Anthropic execution through Claude CLI.

## Reproduction Assessment

Yes, at source level. Configure Anthropic with `agents.defaults.cliBackends.claude-cli` but no provider/model `agentRuntime.id`; the resolver has no CLI runtime to return, so non-raw agent execution can continue to the embedded PI path.

## Solution Assessment

Unclear. Inferring transport from `cliBackends` would address the reported asymmetry, but current docs, schema, and tests make provider/model `agentRuntime.id` the explicit executor-selection contract, so maintainers need to choose the policy boundary before implementation.
