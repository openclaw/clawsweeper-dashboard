## Summary

Keep open. Current main still does not use a `cliBackends`-only Anthropic/Claude CLI configuration to select Claude CLI execution for PI/native subagent paths, while current docs and tests define `agentRuntime.id` as the execution selector, so this remains a runtime-policy feature decision rather than a closeable implemented bug.

## What This Changes

The issue asks for PI/native subagent model dispatch to honor configured Claude CLI backends so Claude OAuth usage stays on the same CLI transport as chat.

## Best Possible Solution

Define one documented Claude CLI transport contract across chat, native subagents, raw model runs, and fallback paths: either diagnose the required `agentRuntime.id` explicitly or add a deliberate opt-in for eligible Anthropic paths to delegate through Claude CLI.

## Reproduction Assessment

Yes, source inspection gives a high-confidence routing reproduction: configure an Anthropic model with `agents.defaults.cliBackends.claude-cli` but no `agentRuntime.id`, then native `sessions_spawn` reaches the agent path without a runtime override and the resolver has no CLI runtime to return.

## Solution Assessment

Unclear. Inferring execution transport from `cliBackends` would address the reported asymmetry, but it would change the documented `agentRuntime.id` selector contract and billing semantics, so maintainers should choose diagnostics versus an explicit opt-in behavior.
