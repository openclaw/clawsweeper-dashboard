## Summary

Keep open. Current main still treats `agentRuntime.id: "claude-cli"` as the supported Claude CLI execution selector, while `cliBackends` alone remains a fallback/backend configuration surface and does not make PI or native subagent dispatch switch away from the Anthropic provider path.

## What This Changes

The issue asks OpenClaw PI, subagent, raw model-run, and fallback dispatch to consult configured Claude `cliBackends` so Claude CLI OAuth users do not fall through to direct Anthropic API billing.

## Best Possible Solution

Define one Claude CLI execution contract across chat, PI, native subagents, raw model runs, and fallback paths: either require and diagnose `agentRuntime.id: "claude-cli"`, or add an explicit opt-in that routes eligible Anthropic PI paths through the configured Claude CLI backend with docs and regression coverage.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With an Anthropic model and `agents.defaults.cliBackends.claude-cli` but no `agentRuntime.id`, current source leaves CLI resolution undefined and reaches the PI/provider path; live Anthropic billing is not required to verify that transport split.

## Solution Assessment

Unclear. The requested cliBackends inference is plausible, but current docs, tests, and migration code make `agentRuntime` the executor-selection contract, so a maintainer should choose between explicit diagnostics and a deliberate opt-in before implementation.
