## Summary

Keep open. Current main supports Claude CLI execution through the explicit `agentRuntime.id: "claude-cli"` contract and the Anthropic CLI auth migration, but it still does not infer CLI transport from `agents.defaults.cliBackends` alone for canonical `anthropic/*` PI/subagent dispatch. That remaining behavior is a maintainer-level runtime and billing contract decision, not a conservative cleanup close.

## What This Changes

The issue requests that PI/subagent execution honor configured Claude CLI backends for Anthropic OAuth credentials instead of falling through to direct API transport.

## Best Possible Solution

Define one supported contract and make it consistent: either keep Claude CLI selection explicit through `agentRuntime.id: "claude-cli"` with doctor/docs checks for `cliBackends`-only setups, or add a deliberate opt-in policy that lets eligible Anthropic PI, subagent, and model paths route through a configured Claude CLI backend with regression coverage for auth forwarding, fallback behavior, and subagent spawn.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With canonical `anthropic/*`, configured `agents.defaults.cliBackends.claude-cli`, and no `agentRuntime.id`, current source leaves the execution provider as `anthropic` and falls through to embedded PI for the native subagent path.

## Solution Assessment

Unclear. Inferring CLI transport from `cliBackends` alone is not clearly the safest fix because current docs and setup code use `agentRuntime` as the executor-selection contract; a safer alternative may be an explicit opt-in or doctor repair instead of silent inference.
