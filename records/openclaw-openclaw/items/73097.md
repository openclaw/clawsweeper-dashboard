## Summary

Keep open. Current main supports Claude CLI execution through the explicit `agentRuntime.id: "claude-cli"` contract and current Anthropic CLI migration writes that contract, but a `cliBackends`-only configuration still does not cause PI/native subagent execution to switch away from the canonical Anthropic provider path. Because changing that would affect runtime and billing semantics, this needs maintainer direction rather than conservative cleanup.

## What This Changes

The issue requests consistent Claude CLI transport selection for OAuth-backed Anthropic runs so PI, native subagents, raw model runs, fallback chains, and chat do not split across direct API and CLI backend paths for the same credential.

## Best Possible Solution

Define one supported contract and make all diagnostics align with it: either keep Claude CLI selection explicit through `agentRuntime.id: "claude-cli"` with doctor/docs checks for `cliBackends`-only setups, or add a deliberate opt-in that lets eligible Anthropic PI, native subagent, raw model-run, and fallback paths route through the configured Claude CLI backend with focused regression coverage.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With canonical `anthropic/*`, configured `agents.defaults.cliBackends.claude-cli`, and no `agentRuntime.id`, current source leaves CLI resolution undefined and the agent/subagent path reaches `runEmbeddedPiAgent`; live Anthropic billing impact is not needed to verify that split.

## Solution Assessment

Unclear. Inferring CLI transport from `cliBackends` alone is not clearly the best fix because current docs and migration code make `agentRuntime` the executor-selection contract; an explicit opt-in or doctor repair may be safer than silent inference.
