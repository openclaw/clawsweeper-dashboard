## Summary

Keep open. Current main still uses `agentRuntime.id` or legacy runtime model prefixes as the Claude CLI execution selector; `cliBackends` alone remains backend configuration and does not opt PI, native subagent spawn, or raw model runs out of the Anthropic provider path.

## What This Changes

Request to make PI/native subagent and related model execution consult configured Claude CLI backends so Claude OAuth traffic uses the same CLI transport as chat.

## Best Possible Solution

Define one documented Claude CLI execution contract across chat, PI/native subagents, raw model runs, and fallback paths: either diagnose the explicit `agentRuntime.id` requirement clearly or add a deliberate opt-in that routes eligible Anthropic PI paths through the configured CLI backend.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With an Anthropic model and `agents.defaults.cliBackends.claude-cli` but no `agentRuntime.id`, the resolver has no CLI runtime to return and the attempt reaches `runEmbeddedPiAgent`; live billing is not required to prove that transport split.

## Solution Assessment

Unclear. Inferring execution from `cliBackends` would address the reported asymmetry, but current docs, migration code, and tests make `agentRuntime` the executor-selection contract, so a maintainer should choose between stricter diagnostics and a documented opt-in before code changes.
