## Summary

Keep open. Current main still has prompt guidance and a guarded first-class gateway config tool, but raw `exec` can still run model-issued OpenClaw config/model/gateway self-management commands without a hard confirmation boundary when host exec is in `security=full` plus `ask=off`, matching the reported risk.

## What This Changes

The issue requests an explicit confirmation boundary before an agent executes example OpenClaw config, model, or gateway commands shown during a model-switching conversation.

## Best Possible Solution

Add a hard exec-boundary policy for OpenClaw self-management commands so model-initiated `openclaw config set`, model/default changes, gateway lifecycle, update/uninstall/reset, and similar commands require explicit user-approved intent through raw shell execution. Keep the first-class `gateway` tool as the preferred config/restart path and coordinate the command taxonomy with #65307 and #39528.

## Reproduction Assessment

Yes. The issue logs show the raw `exec` calls, and current source gives a high-confidence static reproduction path: with host exec at `security=full` and `ask=off`, `openclaw config set ...` passes the current hard preflight and does not trigger `requiresExecApproval`.

## Solution Assessment

No for prompt-only mitigation or example-labeling alone; those are useful guidance but not a security boundary. The narrow maintainable fix belongs in raw exec approval/preflight policy, with focused tests and docs updates for any changed YOLO behavior.
