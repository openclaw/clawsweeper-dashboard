## Summary

Keep open. Current main still has prompt guidance and a guarded first-class gateway config tool, but the raw `exec` path still lacks a hard approval boundary for model-issued OpenClaw config/model/gateway self-management commands under host `security=full` plus `ask=off`, which matches the reported unintended config mutation risk.

## What This Changes

The issue asks OpenClaw to prevent agents from automatically executing example `openclaw config set`, model-switch, and `openclaw gateway restart` commands without explicit user confirmation.

## Best Possible Solution

Add a hard raw-exec approval boundary for OpenClaw self-management commands, covering config/model mutations, gateway lifecycle, update/uninstall/reset, and similar commands, while keeping the first-class `gateway` tool as the preferred config/restart path and coordinating the broader destructive-command policy with #65307 and #39528.

## Reproduction Assessment

Yes. The issue logs provide concrete raw `exec` calls, and current source gives a high-confidence static reproduction path: with host exec at `security=full` and `ask=off`, `openclaw config set ...` and `openclaw gateway restart` do not trigger `requiresExecApproval`; I did not run a live mutation because that would change config or restart the gateway.

## Solution Assessment

No for prompt-only mitigation; the prompt guidance and first-class gateway guard are useful but not a security boundary for raw shell execution. The maintainable fix belongs in exec approval/preflight policy with focused tests for the protected OpenClaw command class.
