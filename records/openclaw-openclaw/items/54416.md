## Summary

Keep open. Current main has prompt guidance and a guarded first-class gateway config tool, but the raw `exec` path still does not add a hard confirmation boundary for model-issued OpenClaw config/model/gateway self-management commands under host `security=full` plus `ask=off`, which matches the reported risk.

## What This Changes

The issue requests that agent-shown example OpenClaw config/model/gateway commands require explicit user confirmation instead of being automatically executed.

## Best Possible Solution

Add a hard exec-boundary policy for OpenClaw self-management commands so model-initiated `openclaw config set`, model/default changes, gateway lifecycle, update/uninstall/reset, and similar commands require explicit user-approved intent through raw shell execution. Keep the first-class `gateway` tool as the preferred config/restart path and coordinate the command taxonomy with #65307 and the config/restart cluster in #39528.

## Reproduction Assessment

Yes. The issue logs show the raw `exec` calls, and current source gives a high-confidence static reproduction path: with host exec at `security=full` and `ask=off`, `openclaw config set ...` passes the current hard preflight and does not trigger `requiresExecApproval`.

## Solution Assessment

No for prompt-only mitigation; it is useful guidance but not a sufficient security boundary. The narrow maintainable fix belongs in the raw exec approval/preflight policy with focused tests, plus docs alignment for any changed YOLO behavior.
