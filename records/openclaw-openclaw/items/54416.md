## Summary

Keep open. Current main and v2026.5.22 still document and implement raw host exec in full/off mode without a hard confirmation class for OpenClaw config/model/provider changes or gateway restarts, so the reported consent gap remains real and security-sensitive.

## What This Changes

The issue requests that model-issued example commands such as OpenClaw config writes, provider/model changes, and gateway restarts not run through exec without explicit user confirmation.

## Best Possible Solution

Add a deterministic exec-boundary policy for OpenClaw self-management commands, covering config writes, provider/model/default changes, gateway lifecycle, updates, and self-removal, with documented behavior for intentional full/off host automation.

## Reproduction Assessment

Yes, at source level: current main resolves non-sandbox host exec to full/off and the approval predicate does not prompt for the reported OpenClaw config or restart commands. I did not run the destructive live commands because they would intentionally mutate config and restart the gateway.

## Solution Assessment

Unclear until maintainers define the self-management command taxonomy and the intended exception to full/off host exec semantics. Prompt-only or example-comment-only mitigation is not durable enough; the maintainable fix should be deterministic exec approval or preflight policy with regression coverage.
