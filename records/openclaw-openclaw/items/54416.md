## Summary

Keep open. Current main and the latest release still allow raw host exec to run OpenClaw config/model/provider changes and gateway restarts under the documented full/off policy, so the reported consent gap remains real and security-sensitive.

## What This Changes

The issue asks OpenClaw to prevent agents from executing example OpenClaw config, model, provider, and gateway commands without explicit user confirmation.

## Best Possible Solution

Add a deterministic exec-boundary policy for OpenClaw self-management commands, covering config writes, provider/model/default changes, gateway lifecycle, updates, and self-removal while documenting how the exception composes with intentional full/off host automation.

## Reproduction Assessment

Yes, at source level: current main resolves gateway/node host exec to full/off and the approval predicate does not prompt for the reported OpenClaw config or restart commands. I did not run the destructive live commands because they would intentionally mutate config and restart the gateway.

## Solution Assessment

Unclear until maintainers define the self-management command taxonomy and the intended exception to full/off host exec semantics. Prompt-only or example-comment-only mitigation is not durable enough; the maintainable fix should be deterministic exec approval or preflight policy with regression coverage.
