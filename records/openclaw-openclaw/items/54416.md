## Summary

Keep open. Current main still documents and implements host `exec` `security=full` plus `ask=off` as no-approval mode, and the hard shell preflight only blocks `/approve` and interactive `openclaw channels login`, not `openclaw config set`, model changes, or `openclaw gateway restart`.

## What This Changes

The issue asks for a hard confirmation boundary so model-generated example commands like `openclaw config set` and `openclaw gateway restart` are not executed automatically through raw `exec`.

## Best Possible Solution

Define and implement a shared exec approval/preflight rule for OpenClaw self-management commands, with tests and docs that state the intended exception to full/off behavior.

## Reproduction Assessment

Yes at source level: the issue logs show raw `exec` calls, and current main still has a full/off host exec path that does not prompt for config or gateway lifecycle commands. I did not run the destructive live commands because they would intentionally mutate OpenClaw config and restart the gateway.

## Solution Assessment

Unclear until maintainers define the self-management command taxonomy and YOLO-mode policy. Prompt-only or comment-only mitigation is not a sufficient boundary; the maintainable fix belongs in exec approval/preflight policy with regression coverage.
