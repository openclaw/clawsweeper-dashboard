## Label Justifications

- `P1`: The report describes a real user-facing agent workflow that can mutate config and restart the gateway without a deterministic consent boundary.
- `impact:data-loss`: Unapproved config writes can overwrite or corrupt user configuration and model defaults.
- `impact:security`: The issue is about moving sensitive host exec and config mutation from model self-discipline to a deterministic approval boundary.
- `impact:auth-provider`: The reported commands alter provider credentials, provider routing, base URLs, and default model selection.

## Summary

Keep open: current main and the latest release still allow raw gateway/node host exec in full/off mode without a deterministic confirmation class for OpenClaw config writes, model-default changes, or gateway lifecycle commands.

## What This Changes

Require explicit confirmation before agent-issued raw exec runs OpenClaw config/model/gateway self-management commands from example model-switching guidance.

## Best Possible Solution

Define a deterministic raw-exec approval or preflight boundary for OpenClaw self-management commands, covering config writes, model-default changes, gateway lifecycle, update, uninstall, and similar operations, with docs and regression coverage for full/off semantics.

## Reproduction Assessment

Yes at source level: current main still has full/off raw host exec paths that do not prompt for `openclaw config set` or `openclaw gateway restart`. I did not run the destructive live commands because they would mutate config and restart the gateway.

## Solution Assessment

Unclear until maintainers define the self-management command taxonomy and the intended exception to documented full/off behavior. Prompt-only guidance is not sufficient; the durable fix belongs in deterministic exec approval or preflight policy plus tests.
