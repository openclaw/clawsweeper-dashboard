## Summary

Keep open. Current main and v2026.5.28 still gate live thinking events on `onReasoningStream`, while this PR removes that gate with focused tests and supplied proof; no safe merged superseder exists, but merge remains a maintainer decision because it expands live message delivery and carries broad sidecar churn.

## What This Changes

The PR broadens embedded agent reasoning streaming so `/reasoning stream` no-callback gateway/WebSocket-style runs emit live `agent.stream === "thinking"` events, updates docs, adds regression tests, and includes formatting-only sidecar edits.

## Best Possible Solution

Land a narrowed or maintainer-accepted version that keeps `reasoningMode === "stream" && canShowReasoning`, retains silent suppression and callback compatibility, documents cross-client behavior, and avoids unrelated churn if maintainers want a smaller review surface.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.5.28 require `onReasoningStream` before `emitAgentEvent({ stream: "thinking" })`, while the `chat.send` replyOptions path does not provide that callback; I did not run a local live repro in this read-only pass.

## Solution Assessment

Yes, the runtime shape is the best narrow fix: make the existing callback optional while keeping `canShowReasoning` and `silentExpected` gates. Adding no-op callbacks at every no-callback entry point would duplicate channel behavior and be a weaker owner-boundary fix.
