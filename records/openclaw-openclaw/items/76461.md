## Summary

Keep open: current source makes `updatedAtMs` part of the split cron state sidecar, but the restart path only persists load-time repairs when another scheduler change happens. A stable future `nextRunAtMs` with a missing or invalid `updatedAtMs` can therefore be repaired in memory without writing `jobs-state.json`, which matches the reported monitor noise.

## What This Changes

Fix the cron scheduler so `jobs-state.json` reliably persists repaired or initialized `updatedAtMs` after Gateway restarts.

## Best Possible Solution

Persist corrected cron runtime-state sidecar data when load/startup repairs `updatedAtMs`, without reintroducing runtime fields into `jobs.json` or adding a new config knob.

## Reproduction Assessment

Yes, source-level: create or retain a `jobs-state.json` entry with a valid future `nextRunAtMs` but missing or invalid `updatedAtMs`; on restart, `loadCronStore` repairs the timestamp in memory, but `start` only calls `persist` if another scheduler change is detected.

## Solution Assessment

Yes with a narrow implementation: mark load-time state repairs dirty or otherwise force a sidecar-only persist after startup repair. Avoid broad startup rewrites of `jobs.json` and keep the split-state contract intact.
