## Summary

Keep open: current main still repairs missing or invalid cron `updatedAtMs` in memory, but a restart path with an otherwise valid future `nextRunAtMs` does not call the sidecar persist path, so `jobs-state.json` can remain stale on disk.

## What This Changes

The issue asks the cron scheduler to persist missing or invalid `updatedAtMs` repairs into `jobs-state.json` after Gateway restarts.

## Best Possible Solution

Persist corrected runtime-state sidecar data when cron load/startup repairs `updatedAtMs`, while keeping `jobs.json` definition-only and avoiding a new config knob.

## Reproduction Assessment

Yes, source-level: create or retain a `jobs-state.json` entry with a valid future `nextRunAtMs` but missing or invalid `updatedAtMs`; `loadCronStore` repairs the timestamp in memory, `recomputeNextRunsForMaintenance` returns false, and `start` does not call `persist`. I did not run a disk-writing repro because the review had to keep the checkout read-only.

## Solution Assessment

Yes: the narrow maintainable fix is to mark load-time sidecar repairs dirty or otherwise force a sidecar-only persist after startup repair. Avoid broad startup rewrites of `jobs.json` or changes to cron scheduling semantics.
