## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Normalize per-job hook paths before validation:** `src/cron/hooks.ts:201-212`
  - body: `isValidJobHookPath` checks the raw persisted value, but `loadHookModule` later resolves the same value through `resolveUserPath`, which trims and expands paths. Whitespace-prefixed absolute/traversal paths or `~`-relative values can pass this guard and then load code outside the intended per-job relative scope.
  - confidence: 0.94
- **[P2] Validate per-job hook entry types before path checks:** `src/cron/hooks.ts:67-71`
  - body: `job.hooks` comes from persisted job data, but this loop assumes the selected hook point is a string array. A malformed value such as `[123]` can throw inside the path helper before normal hook isolation and run finalization, leaving the job to depend on stale-run recovery.
  - confidence: 0.89
- **[P2] Reuse stable hook import specifiers:** `src/cron/hooks.ts:192`
  - body: Every file-backed hook import uses `cacheBust: true`, so recurring cron hooks create a fresh ESM module instance on each run. Long-lived gateway processes should reuse stable imports or provide an explicit reload mechanism instead of growing the module cache by default.
  - confidence: 0.82

## Security Review

  - body: Per-job hooks are intended to be relative filesystem paths, but validation checks the raw value while import resolution trims and expands it later; whitespace-prefixed absolute/traversal paths and home-relative paths can evade the restriction.
