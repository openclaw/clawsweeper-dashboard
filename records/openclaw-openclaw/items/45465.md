## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Persist hooks through the current SQLite cron store:** `src/cron/types.ts:146`
  - body: Adding `hooks` to `CronJob` is not enough on current main: cron jobs are rebuilt from SQLite columns in `rowToCronJob`, and unknown `job_json` fields are not projected into the runtime job. Per-job hooks would be silently absent after load unless the current SQLite row codec/schema/normalization path is updated and tested.
  - confidence: 0.88
- **[P1] Normalize per-job hook paths before validation:** `src/cron/hooks.ts:201-212`
  - body: `isValidJobHookPath` checks the raw string, but `loadHookModule` later resolves the same value through `resolveUserPath`, which trims first. Values such as `" /etc/hook.cjs"` or spaced traversal paths can pass validation and then resolve outside the intended per-job relative path boundary.
  - confidence: 0.94
- **[P2] Validate persisted hook entries before path checks:** `src/cron/hooks.ts:69-74`
  - body: `job.hooks` is persisted data, but this loop assumes every entry is a string before passing it to `isValidJobHookPath`. A malformed stored value such as `[123]` can throw in `path.isAbsolute` before normal cron finalization, so the patch should narrow the array and skip or repair invalid entries first.
  - confidence: 0.86
- **[P2] Avoid cache-busted imports for recurring hooks:** `src/cron/hooks.ts:192`
  - body: This imports hook files with `cacheBust: true` on every run, and the shared loader turns that into a unique module specifier. Recurring cron hooks in a long-lived gateway should reuse a stable module entry or have an explicit bounded reload mechanism rather than growing module instances per execution.
  - confidence: 0.78
