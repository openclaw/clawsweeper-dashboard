## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks cron lifecycle hooks, while the proposed patch adds the requested surface but leaves concrete correctness and security hardening issues in the per-job hook path handling and repeated dynamic imports.

## What This Changes

The PR adds global and per-job cron lifecycle hook configuration, a hook runner, Gateway protocol schema fields, and scheduled/startup/manual cron execution wiring for beforeRun, afterComplete, onFailure, and afterRun hook points.

## Best Possible Solution

Keep the PR open as an implementation candidate, but require a maintainer decision on the core-vs-plugin boundary and harden the proposed hook runner before merge. A mergeable version should validate persisted job hook input before path handling, close the whitespace path escape, avoid unbounded module-cache growth, and keep config schema, Gateway protocol, docs, and scheduled/startup/manual cron tests aligned.

## Reproduction Assessment

Yes. The current-main absence is reproducible by source inspection of the cron config/schema/job/execution paths, and the PR bugs are reproducible by following a whitespace-prefixed per-job hook path through isValidJobHookPath and then resolveUserPath before import.

## Solution Assessment

No. The PR is directionally close to the requested feature, but dynamic script execution from cron config and jobs.json is security-sensitive, and the current patch leaves blocking input-validation and long-lived-process memory issues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trim paths before validating per-job hooks:** `src/cron/hooks.ts:201-203`
  - body: `isValidJobHookPath` checks the raw `scriptPath`, but `loadHookModule` later resolves the same value through `resolveUserPath`, which trims whitespace before import. A persisted value like `" /abs/hook.cjs"` or `" ../outside/hook.cjs"` can pass the raw absolute/traversal checks and then be trimmed into an escaped import target, bypassing the intended per-job restriction. Trim once before every validation check, or reject paths with surrounding whitespace.
  - confidence: 0.93
- **[P2] Validate per-job hook entries before path checks:** `src/cron/hooks.ts:70-71`
  - body: `job.hooks?.[hookPoint]` comes from persisted job data, but each element is passed directly to `isValidJobHookPath`. A malformed jobs.json value such as `[123]` will throw inside `path.isAbsolute`, escaping the hook isolation path before normal cron finalization and potentially leaving the run marked active until recovery. Check that the field is an array of strings before validating paths.
  - confidence: 0.88
- **[P2] Reuse stable hook import specifiers:** `src/cron/hooks.ts:192`
  - body: Every file-backed hook import uses `cacheBust: true`, and the shared module loader implements that by appending a timestamp query. In Node ESM those unique specifiers are cached separately, so a recurring cron hook will load a new module instance on every run and grow gateway memory over time. Use a stable import by default, or make reload/cache busting an explicit bounded operation.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces dynamic cron hook script execution and currently leaves a concrete per-job path validation bypass in the less-trusted persisted job hook path.

Concerns:

- **[medium] Per-job hook path restriction can be bypassed:** `src/cron/hooks.ts:201`
  - body: Per-job hooks are described as restricted because jobs.json is less trusted, but validation checks the raw path while import resolution trims it later. Whitespace-prefixed absolute or traversal paths can therefore evade the restriction and load code outside the intended relative hook area.
  - confidence: 0.93
- **[medium] Dynamic cron script execution needs product security review:** `src/cron/hooks.ts:181`
  - body: Global and per-job cron hook configuration can execute local modules during scheduled, catch-up, and manual runs. Even with admin-controlled global config, this is a new code-execution surface and should be explicitly approved with documented trust boundaries before merge.
  - confidence: 0.8
