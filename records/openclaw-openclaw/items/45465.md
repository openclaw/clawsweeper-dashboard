## Summary

Keep open. Current main does not implement cron lifecycle hooks, and this PR is still the active implementation candidate, but it should not merge yet because source review at PR head still shows a per-job path restriction bypass, malformed persisted hook entries can escape normal isolation, stable import behavior is unresolved, and the feature creates a new core cron code-execution surface that needs maintainer security/product approval.

## What This Changes

The PR adds global and per-job cron lifecycle hook configuration, schema/protocol types, a hook runner, and scheduled/startup/manual dispatch for beforeRun, afterComplete, onFailure, and afterRun hook points.

## Best Possible Solution

Keep the PR open as the implementation candidate, but require a hardened revision that validates persisted job hook input before path handling, trims or rejects surrounding whitespace before safety checks, avoids per-run unique file import specifiers, documents the code-execution trust boundary, and aligns docs, changelog, schema, and scheduled/startup/manual cron tests.

## Reproduction Assessment

Yes. Current-main absence is reproducible by source inspection of CronConfig, the strict config schema, CronJob, and the scheduled/startup/manual execution paths; the PR bugs are reproducible by following whitespace-prefixed and malformed per-job hook values through loadHookEntries, isValidJobHookPath, resolveUserPath, and importFileModule.

## Solution Assessment

No. The feature direction may be useful, but this patch is not yet the best mergeable form because it adds a security-sensitive code-execution surface while leaving per-job input validation and long-lived-process import behavior unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trim paths before checking per-job hook safety:** `src/cron/hooks.ts:203`
  - body: `isValidJobHookPath` checks the raw value, but `loadHookModule` later resolves the same string through `resolveUserPath`, which trims first. A persisted value like `" /abs/hook.cjs"` or `" ../outside/hook.cjs"` can pass these checks and then import outside the intended per-job relative scope.
  - confidence: 0.93
- **[P2] Validate per-job hook entries before path checks:** `src/cron/hooks.ts:70-71`
  - body: `job.hooks?.[hookPoint]` comes from persisted job data, but the loop passes each element directly to `isValidJobHookPath`. Malformed jobs.json values such as `[123]` can throw before normal hook isolation and finalization, so guard that this is an array of strings before iterating.
  - confidence: 0.88
- **[P2] Reuse stable hook import specifiers:** `src/cron/hooks.ts:192`
  - body: Every file-backed hook import uses `cacheBust: true`, and the shared loader appends a timestamp query for that mode. Node caches each unique ESM specifier, so recurring hooks create new module instances on every run and can grow Gateway memory over time.
  - confidence: 0.82
- **[P2] Document the new cron hook contract:** `src/config/types.cron.ts:99`
  - body: This adds a user-facing `cron.hooks` config surface and persisted job hooks but the PR does not update cron docs or the changelog. Operators need the trust boundary, path rules, hook points, timeout behavior, and examples documented before this code-execution surface ships.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Security review needs attention: the diff introduces cron-driven dynamic script execution and still has a concrete per-job path restriction bypass.

Concerns:

- **[medium] Per-job hook path restriction can be bypassed:** `src/cron/hooks.ts:201`
  - body: Per-job hooks are described as restricted because jobs.json is less trusted, but validation checks the raw path while import resolution trims it later. Whitespace-prefixed absolute or traversal paths can evade the restriction and load code outside the intended relative hook area.
  - confidence: 0.93
- **[medium] Dynamic cron script execution needs approval:** `src/cron/hooks.ts:181`
  - body: Global and per-job cron hook configuration can execute local modules during scheduled, catch-up, and manual runs. This is a new code-execution surface and should be explicitly approved with documented trust boundaries before merge.
  - confidence: 0.8
