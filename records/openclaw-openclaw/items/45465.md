## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Persist per-job hooks through SQLite rows:** `src/cron/types.ts:146`
  - body: Adding `hooks` to the runtime job type is not enough on current main: cron jobs are loaded from SQLite rows, and `rowToCronJob` reconstructs the runtime job from columns without rehydrating `job_json.hooks`. Per-job hooks configured through the job store would disappear after reload/restart, so the advertised per-job lifecycle behavior will silently stop working unless the row codec, normalization, and tests are updated together.
  - confidence: 0.86
- **[P1] Normalize per-job paths before validation:** `src/cron/hooks.ts:201-213`
  - body: `isValidJobHookPath` validates the raw persisted string, but `loadHookModule` later resolves the same value through `resolveUserPath`, which trims first. A value like `" /tmp/hook.cjs"` passes the absolute/traversal checks here and is then trimmed into an absolute filesystem import, bypassing the per-job relative-path boundary.
  - confidence: 0.9
- **[P2] Narrow per-job hook entries before path checks:** `src/cron/hooks.ts:70-71`
  - body: Persisted cron data can be malformed, but this loop assumes every `job.hooks[hookPoint]` entry is a string before calling `isValidJobHookPath`. A non-string entry can throw in `path.isAbsolute` before normal run finalization, leaving cron state to rely on stale-run recovery instead of recording a skipped/failed result.
  - confidence: 0.84
- **[P2] Avoid cache-busting every hook import:** `src/cron/hooks.ts:192`
  - body: This imports file hooks with `cacheBust: true` on every execution, and the shared loader implements that by appending a timestamp query to the file URL. Node caches each unique ESM specifier for the process lifetime, so frequent cron hooks can grow gateway memory without a bound or reload owner.
  - confidence: 0.8
