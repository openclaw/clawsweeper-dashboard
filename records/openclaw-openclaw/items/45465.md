## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Normalize per-job hook paths before enforcing restrictions:** `src/cron/hooks.ts:201-212`
  - body: `isValidJobHookPath` checks the raw string, but `loadHookModule` later calls `resolveUserPath`, which trims before importing. A persisted per-job value such as `" /etc/passwd"` or `" ../outside/hook.cjs"` passes the raw absolute/traversal checks here and is then trimmed into the escaped filesystem path, bypassing the intended per-job safety boundary.
  - confidence: 0.93
- **[P2] Narrow persisted per-job hook entries before iterating:** `src/cron/hooks.ts:69-74`
  - body: Cron jobs are loaded from persisted state and this new `hooks` field is not normalized before `loadHookEntries` reads it. If `job.hooks[hookPoint]` is malformed, such as `[123]` or a non-array truthy value, this loop can throw before the cron run reaches normal result finalization and cleanup.
  - confidence: 0.88
- **[P2] Resolve relative hooks from a stable config root:** `src/cron/service/timer.ts:640`
  - body: `hookBasePath` is derived from `state.deps.storePath`, but `cron.store` is configurable and current docs describe it as a logical store key/import path. Moving the cron store would silently remap relative hook scripts to a different directory, so hook script resolution needs a stable OpenClaw config/workspace root shared by scheduled, catch-up, and manual runs.
  - confidence: 0.8
