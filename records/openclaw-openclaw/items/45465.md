## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Normalize per-job hook paths before trusting them:** `src/cron/hooks.ts:201-216`
  - body: `isValidJobHookPath` checks the raw `scriptPath`, but `loadHookModule` later passes the same value through `resolveUserPath`, which trims whitespace before resolving. A persisted value like `" /etc/hook.cjs"` or `" ../outside/hook.cjs"` can pass these checks and then import outside the intended relative hook scope; trim and normalize once, reject `~`/absolute/traversal values after normalization, and import that same validated value.
  - confidence: 0.93
- **[P2] Validate persisted hook arrays before iterating:** `src/cron/hooks.ts:67-71`
  - body: `job.hooks?.[hookPoint]` comes from persisted job data, but this loop assumes it is an array of strings. A malformed stored value such as `[123]` reaches `path.isAbsolute` and throws before the normal cron result/finalization path, which can leave the job marked running until stale-run recovery; guard with `Array.isArray` and `typeof script === "string"` before path checks.
  - confidence: 0.89
- **[P2] Reuse stable hook module specifiers:** `src/cron/hooks.ts:192`
  - body: Passing `cacheBust: true` on every hook import creates a fresh ESM specifier each cron run, and Node keeps those module instances cached for the process lifetime. Repeated scheduled hooks can grow memory over time; default to stable imports and add an explicit bounded reload mechanism only if live hook editing is required.
  - confidence: 0.86
