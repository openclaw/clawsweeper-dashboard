## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Trim and type-check per-job hook paths before validation:** `src/cron/hooks.ts:70-71`
  - body: `jobs.json` is persisted data, but this loop assumes every entry is a string and validates the raw value while `loadHookModule` later trims it through `resolveUserPath`. A value like `" /etc/passwd"` or `[123]` can either bypass the per-job relative-path restriction after trimming or throw before the cron run reaches normal finalization. Normalize to a non-empty string before the safety checks and reject non-string entries with coverage.
  - confidence: 0.89
- **[P2] Stop cache-busting every hook import:** `src/cron/hooks.ts:192`
  - body: `cacheBust: true` gives every cron hook execution a fresh file URL, and the module loader appends a timestamp query for cache-busted imports. Node keeps each unique ESM specifier cached for the life of the process, so recurring cron hooks will leak module instances in long-lived gateways; use stable imports by default or add an explicit bounded reload mechanism.
  - confidence: 0.84

## Risks / Open Questions

- Merging this PR would add unattended Gateway-side execution of user-configured modules from global config and persisted cron job data, so maintainers need an explicit trust model for path scope, environment, secrets, logging, timeouts, reload behavior, and audit semantics.
