## Summary

Keep open. Current main does not implement cron lifecycle hooks, so this PR is not obsolete, but the latest PR head still has blocking validation and long-lived runtime issues around a new dynamic script-execution surface.

## What This Changes

The PR adds global and per-job cron lifecycle hook configuration, schema/protocol types, a hook runner, and scheduled/startup/manual dispatch for beforeRun, afterComplete, onFailure, and afterRun hook points.

## Best Possible Solution

Keep the PR open as an implementation candidate, but require maintainer product/security approval and a hardened revision before merge. A mergeable version should validate persisted job hook input before path handling, trim or reject surrounding whitespace before safety checks, avoid per-run unique import specifiers, and keep config schema, Gateway protocol, docs, changelog, and scheduled/startup/manual cron tests aligned.

## Reproduction Assessment

Yes. Current-main absence is reproducible by source inspection of CronConfig, the strict config schema, CronJob, and the scheduled/startup/manual execution paths. The PR bugs are reproducible by following a whitespace-prefixed per-job hook path through loadHookEntries/isValidJobHookPath and then through resolveUserPath before import.

## Solution Assessment

No. The feature direction may be useful, but this patch is not yet the narrowest maintainable solution because it adds a security-sensitive code-execution surface while leaving input-validation and long-lived-process import behavior unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trim paths before enforcing per-job path limits:** `src/cron/hooks.ts:203`
  - body: `isValidJobHookPath` checks the raw value, but `loadHookModule` later resolves the same string through `resolveUserPath`, which trims first. A persisted value like `" /abs/hook.cjs"` or `" ../outside/hook.cjs"` can pass these checks and then import outside the intended per-job relative scope.
  - confidence: 0.93
- **[P2] Validate per-job hook entries before path checks:** `src/cron/hooks.ts:70-71`
  - body: `job.hooks?.[hookPoint]` comes from persisted job data, but the loop passes each element directly to `isValidJobHookPath`. Malformed jobs.json values such as `[123]` can throw before normal hook isolation/finalization, so guard that this is an array of strings before iterating.
  - confidence: 0.88
- **[P2] Reuse stable hook import specifiers:** `src/cron/hooks.ts:192`
  - body: Every file-backed hook import uses `cacheBust: true`, and the shared loader appends a timestamp query for that mode. Node caches each unique ESM specifier, so recurring hooks create new module instances on every run and can grow Gateway memory over time.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a new cron-driven dynamic script-execution path and still has a concrete per-job path restriction bypass.

Concerns:

- **[medium] Per-job hook path restriction can be bypassed:** `src/cron/hooks.ts:201`
  - body: Per-job hooks are described as restricted because jobs.json is less trusted, but validation checks the raw path while import resolution trims it later. Whitespace-prefixed absolute or traversal paths can evade the restriction and load code outside the intended relative hook area.
  - confidence: 0.93
- **[medium] Dynamic cron script execution needs approval:** `src/cron/hooks.ts:181`
  - body: Global and per-job cron hook configuration can execute local modules during scheduled, catch-up, and manual runs. This is a new code-execution surface and should be explicitly approved with documented trust boundaries before merge.
  - confidence: 0.8
