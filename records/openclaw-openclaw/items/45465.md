## Summary

Keep open: this PR is still a plausible implementation candidate for cron lifecycle hooks, but the latest branch is not merge-ready because it has concrete path/input/security blockers and no inspectable real behavior proof.

## What This Changes

The PR adds global and per-job cron lifecycle hook configuration, a hook runner, schemas/tests, and scheduled/startup/manual cron execution integration for user-defined hook scripts.

## Best Possible Solution

Select one hardened additive cron hook or pre-run gate contract, then land or supersede the open implementation with aligned types, Gateway schemas, runtime paths, docs/tests, security policy, and redacted real behavior proof.

## Reproduction Assessment

Yes for the PR-introduced review defects by source inspection: the PR head validates raw per-job hook paths before trimmed resolution, trusts persisted hook entries before iterating, and cache-busts file imports. This is not a current-main bug reproduction because current main does not include cron lifecycle hooks.

## Solution Assessment

No for the patch as written. The feature direction may be useful, but the mergeable path needs path/input hardening, stable import behavior or explicit reload semantics, a settled host-execution trust boundary, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Normalize per-job hook paths before validation:** `src/cron/hooks.ts:201-212`
  - body: `isValidJobHookPath` checks the raw persisted value, but `loadHookModule` later resolves it through `resolveUserPath`, which trims and expands paths. Values such as whitespace-prefixed absolute/traversal paths or `~`-relative paths can pass this guard and then load code outside the intended relative per-job scope.
  - confidence: 0.94
- **[P2] Validate per-job hook entry types before path checks:** `src/cron/hooks.ts:67-71`
  - body: `job.hooks` comes from persisted job data, but this loop assumes the selected hook point is a string array. A malformed value such as `[123]` can throw inside the path helper before normal hook isolation/finalization, leaving the run to depend on stale-run recovery.
  - confidence: 0.89
- **[P2] Reuse stable hook import specifiers:** `src/cron/hooks.ts:192`
  - body: Every file-backed hook import uses `cacheBust: true`, so recurring cron hooks create a fresh ESM module instance on each run. Long-lived gateway processes should reuse stable imports or expose an explicit reload mechanism instead of growing the module cache by default.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces unattended cron-triggered local module execution and still has concrete per-job path hardening gaps.

Concerns:

- **[medium] Per-job hook path restriction can be bypassed:** `src/cron/hooks.ts:201`
  - body: Per-job hooks are intended to be relative filesystem paths, but the safety check uses the raw value while import resolution trims and expands it later. Whitespace-prefixed absolute/traversal paths and home-relative paths can evade the restriction and load code outside the intended scope.
  - confidence: 0.94
- **[medium] Scheduled local module execution needs an explicit trust model:** `src/cron/hooks.ts:181`
  - body: Global and per-job cron hook settings execute local modules during scheduled, startup catch-up, and manual runs. This unattended code-execution surface needs documented operator scope, path/env/secret handling, approval interaction, timeout/cancellation, resource limits, and logging behavior before merge.
  - confidence: 0.86
