## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Trim per-job hook paths before enforcing restrictions:** `src/cron/hooks.ts:201-212`
  - body: `isValidJobHookPath` checks the raw value, but `loadHookModule` later resolves through `resolveUserPath`, which trims first. A persisted value like `" /etc/hook.cjs"` or `" ../outside.cjs"` can pass the per-job guard and then import outside the intended relative hook scope.
  - confidence: 0.92
- **[P2] Validate persisted hook entries before path checks:** `src/cron/hooks.ts:67-74`
  - body: `jobs.json` rows are loaded as records without deep schema validation, so this loop can receive non-array or non-string hook values. Calling path helpers on a malformed entry throws before normal cron finalization and can leave the run in recovery rather than cleanly skipping the bad hook.
  - confidence: 0.86
- **[P1] Rebase hooks around current cron run bookkeeping:** `src/cron/service/timer.ts:628-681`
  - body: The hook wrapper is built on an older timer path and does not carry current-main `markCronJobActive`, task-run IDs, diagnostics, and finished-event state through the new before/after paths. Rebase the hook integration onto the current `runDueJob`, startup catch-up, and manual-run code instead of replacing that bookkeeping.
  - confidence: 0.83
- **[P2] Use stable hook module imports or bounded reloads:** `src/cron/hooks.ts:191-192`
  - body: `cacheBust: true` appends a fresh query string for each import, and Node keeps each ESM specifier cached. A recurring cron hook can therefore grow module instances over the lifetime of the Gateway; use stable imports by default or a bounded explicit reload mechanism.
  - confidence: 0.84
