## Summary

Keep this PR open, but it is not merge-ready: current main still lacks cron lifecycle hooks, while the branch is conflicted, missing inspectable real behavior proof, and still has concrete security/availability blockers in the hook path handling and import strategy.

## What This Changes

The PR adds global `cron.hooks` and per-job cron lifecycle hooks that run user scripts at `beforeRun`, `afterComplete`, `onFailure`, and `afterRun`.

## Best Possible Solution

Land a hardened cron hook contract only after maintainer/security approval, a clean rebase onto current cron runtime, stable import semantics, validated per-job hook input, and redacted real setup proof across all cron execution paths.

## Reproduction Assessment

Yes for the PR-introduced review risks by source inspection: the hook runner checks raw per-job paths, imports with cache-busted specifiers, and lacks type validation before path checks. This is not a current-main bug reproduction because the feature is not on main.

## Solution Assessment

No for the PR as currently proposed: the broad lifecycle hook design needs maintainer/security approval, rebase work, and hardening before it is the best core solution. A narrower pre-run gate or documented admin-only contract may be safer depending on the product decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trim per-job hook paths before enforcing restrictions:** `src/cron/hooks.ts:201-212`
  - body: `isValidJobHookPath()` validates the raw string, but `loadHookModule()` later resolves through `resolveUserPath()`, which trims first. A persisted value like `" /etc/passwd"` or `" ../outside/hook.cjs"` passes the guard here and is then imported as an absolute/traversal path, bypassing the intended per-job boundary.
  - confidence: 0.92
- **[P2] Validate persisted hook entry types before path checks:** `src/cron/hooks.ts:67-71`
  - body: `job.hooks` comes from persisted job data, but this loop assumes every entry is a string before calling `path.isAbsolute()`. A malformed entry such as `[123]` throws before cron can apply normal result finalization, which can leave the job marked running until recovery.
  - confidence: 0.88
- **[P2] Reuse stable hook import specifiers:** `src/cron/hooks.ts:192`
  - body: `cacheBust: true` appends a fresh timestamp query to every file import, and Node caches each ESM specifier indefinitely. A frequently running cron hook will keep adding module instances in the gateway process; use stable imports by default or an explicit bounded reload mode.
  - confidence: 0.86
