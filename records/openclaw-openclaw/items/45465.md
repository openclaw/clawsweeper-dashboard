## Summary

Keep open: current main still does not implement persisted cron lifecycle hooks, and this PR is still a plausible implementation candidate rather than cleanup-ready stale work. It should not merge as-is because PR-head source still shows path/input hardening defects, cache-busted hook imports, missing real behavior proof, and an unsettled unattended script-execution trust boundary; the related feature thread at https://github.com/openclaw/openclaw/issues/9465 and narrower preHook PR at https://github.com/openclaw/openclaw/pull/62195 should be coordinated rather than using this as an auto-close.

## What This Changes

The PR adds global `cron.hooks`, per-job cron hook paths, a cron hook runner, cron config/protocol schema updates, and lifecycle dispatch around scheduled, startup catch-up, and manual cron runs.

## Best Possible Solution

Settle one hardened cron hook/pre-run gate contract, then land or explicitly supersede this through the canonical feature work with schema, protocol, runtime, docs, tests, security policy, and real cron proof aligned.

## Reproduction Assessment

Yes for the PR defects: PR-head source shows raw persisted hook paths flowing into a trim-before-resolve importer, unvalidated persisted hook entry iteration, and cache-busted file imports. For the requested feature itself, current-main source inspection verifies the lifecycle hook contract is absent from config/types, Gateway schemas, and cron execution paths.

## Solution Assessment

No. The direction may be useful, but this patch is not the best mergeable form until the path/input hardening, import reuse, security trust model, current-main rebase, docs/API completeness, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trim per-job hook paths before safety checks:** `src/cron/hooks.ts:201-203`
  - body: `isValidJobHookPath` checks the raw persisted value, but hook loading later resolves the same value through `resolveUserPath`, which trims first. Values like `" /abs/hook.cjs"` or `" ../outside/hook.cjs"` can pass this guard and then resolve outside the intended per-job relative scope.
  - confidence: 0.93
- **[P2] Validate persisted hook entries before iterating:** `src/cron/hooks.ts:67-71`
  - body: `job.hooks` comes from persisted job data, but this loop assumes the selected hook point is a string array before calling path helpers. A malformed value such as `[123]` can throw before normal hook isolation and finalization.
  - confidence: 0.88
- **[P2] Reuse stable hook import specifiers:** `src/cron/hooks.ts:192`
  - body: Every file-backed hook import uses `cacheBust: true`, so recurring cron hooks create fresh ESM module instances on each run. Long-lived Gateway processes should reuse stable imports or use explicit reload semantics.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds cron-driven local script/module execution and still has a concrete per-job path restriction bypass.

Concerns:

- **[medium] Per-job hook path restriction can be bypassed:** `src/cron/hooks.ts:201`
  - body: Per-job hooks are intended to be relative filesystem paths, but the safety check uses the raw string while import resolution trims it later. Whitespace-prefixed absolute or traversal paths can evade the restriction and load code outside the intended scope.
  - confidence: 0.93
- **[medium] Scheduled script execution needs approval:** `src/cron/hooks.ts:181`
  - body: Global and per-job cron hook settings execute local modules during scheduled, startup catch-up, and manual runs. This is a new unattended code-execution surface and needs documented operator trust boundaries before merge.
  - confidence: 0.85
