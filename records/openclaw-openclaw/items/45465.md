## Summary

Keep open: current main still lacks cron lifecycle hooks, and this PR remains a plausible implementation candidate, but the latest head has source-confirmed correctness/security blockers and no inspectable real behavior proof.

## What This Changes

The PR adds global and per-job cron lifecycle hooks, hook config/types/schemas, a hook runner, tests, and scheduled/startup/manual cron execution integration.

## Best Possible Solution

Choose one hardened cron lifecycle or pre-run gate contract, fix the path/input/import blockers, align schemas/docs/tests/security policy, and require real scheduled/startup/manual proof before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: the latest head validates raw per-job hook paths before trimmed resolution, trusts persisted hook entries before iterating, and cache-busts file imports. This is not a current-main bug reproduction because current main does not contain the lifecycle hook feature.

## Solution Assessment

No for the patch as written. The feature direction may be useful, but the mergeable path needs path/input hardening, stable import behavior or explicit reload semantics, a settled host-execution trust boundary, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trim per-job hook paths before safety checks:** `src/cron/hooks.ts:201-212`
  - body: `isValidJobHookPath` checks the raw persisted value, but `loadHookModule` later resolves the same value through `resolveUserPath`, which trims first. Values like `" /abs/hook.cjs"` or `" ../outside/hook.cjs"` can pass the guard and then resolve outside the intended per-job relative scope.
  - confidence: 0.93
- **[P2] Validate persisted hook entries before iterating:** `src/cron/hooks.ts:67-71`
  - body: `job.hooks` comes from persisted job data, but this loop assumes the selected hook point is a string array before calling path helpers. A malformed value such as `[123]` can throw before normal hook isolation and finalization.
  - confidence: 0.88
- **[P2] Reuse stable hook import specifiers:** `src/cron/hooks.ts:192`
  - body: Every file-backed hook import uses `cacheBust: true`, so recurring cron hooks create fresh ESM module instances on each run. Long-lived Gateway processes should reuse stable imports or provide an explicit reload mechanism.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces unattended cron-triggered local module execution and still has a concrete per-job path restriction bypass.

Concerns:

- **[medium] Per-job hook path restriction can be bypassed:** `src/cron/hooks.ts:201`
  - body: Per-job hooks are intended to be relative filesystem paths, but the safety check uses the raw string while import resolution trims it later. Whitespace-prefixed absolute or traversal paths can evade the restriction and load code outside the intended scope.
  - confidence: 0.93
- **[medium] Scheduled local module execution needs an explicit trust model:** `src/cron/hooks.ts:181`
  - body: Global and per-job cron hook settings execute local modules during scheduled, startup catch-up, and manual runs. This is a new unattended code-execution surface and needs documented operator scope, path/env/secret handling, approval interaction, timeout/cancellation, and logging behavior before merge.
  - confidence: 0.85
