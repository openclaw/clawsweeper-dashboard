## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trim per-job hook paths before safety checks:** `src/cron/hooks.ts:201-209`
  - body: `isValidJobHookPath` checks the raw persisted value, but `loadHookModule` later resolves the same value through `resolveUserPath`, which trims first. Values like `" /abs/hook.cjs"` or `" ../outside/hook.cjs"` can pass this guard and then resolve outside the intended per-job relative scope.
  - confidence: 0.93
- **[P2] Validate persisted hook entries before iterating:** `src/cron/hooks.ts:67-71`
  - body: `job.hooks` comes from persisted job data, but this loop assumes the selected hook point is a string array before calling path helpers. A malformed value such as `[123]` can throw before normal hook isolation and finalization.
  - confidence: 0.88
- **[P2] Reuse stable hook import specifiers:** `src/cron/hooks.ts:192`
  - body: Every file-backed hook import uses `cacheBust: true`, so recurring cron hooks create fresh ESM module instances on each run. Long-lived Gateway processes should reuse stable imports or provide an explicit reload mechanism.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds unattended cron-triggered local module execution and still has a concrete per-job path restriction bypass.

Concerns:

- **[medium] Per-job hook path restriction can be bypassed:** `src/cron/hooks.ts:201`
  - body: Per-job hooks are intended to be relative filesystem paths, but the safety check uses the raw string while import resolution trims it later. Whitespace-prefixed absolute or traversal paths can evade the restriction and load code outside the intended scope.
  - confidence: 0.93

  - body: Global and per-job cron hook settings execute local modules during scheduled, startup catch-up, and manual runs. This is a new unattended code-execution surface and needs a documented operator trust boundary before merge.
  - confidence: 0.85
