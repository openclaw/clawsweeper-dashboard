## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Normalize per-job paths before validating them:** `src/cron/hooks.ts:201-212`
  - body: Per-job hooks are supposed to reject absolute paths and parent traversal, but this check runs on the raw string while loadHookModule resolves through resolveUserPath, which trims before import. A stored value such as `" /etc/passwd"` or `" ../outside.cjs"` can pass this guard and then resolve outside the intended relative hook scope.
  - confidence: 0.88
- **[P2] Avoid per-run cache-busted hook imports:** `src/cron/hooks.ts:191-192`
  - body: Every hook run imports the script with `cacheBust: true`, which changes the ESM specifier on each execution and leaves Node retaining a new module instance for every frequent cron invocation. This can grow memory over time in the cron hot path; use a lifecycle-owned reload/cache policy instead of timestamp imports per run.
  - confidence: 0.82
- **[P2] Persist hooks through the current cron store:** `src/cron/types.ts:143-146`
  - body: Adding `hooks` only to the runtime cron job shape is not enough on current main, where cron jobs are reconstructed from canonical SQLite row fields and job_json is not merged back wholesale. Rebasing this branch without updating the row codec/schema and migration path can silently drop configured hooks after reload.
  - confidence: 0.78
- **[P2] Expose hooks on add/update protocol or keep them internal:** `src/gateway/protocol/schema/cron.ts:265-285`
  - body: The PR returns `hooks` in `CronJobSchema`, but the add and patch schemas still omit that field. RPC clients, UI, and CLI cannot create or update the advertised per-job hooks through the protocol, leaving only hand-edited stores or private paths able to use the feature.
  - confidence: 0.86
