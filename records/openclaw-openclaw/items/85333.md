## Summary

Current main does not close the regression: the snapshot scan still validates each collected cached path, doctor health contributions still run sequentially, gateway calls still create and stop a fresh client per request, and related repair PRs remain open rather than merged.

## What This Changes

The issue reports a v2026.5.20 `openclaw doctor --fix` performance regression in multi-agent installs, with suspected stale session snapshot scan fanout, heavier doctor checks, and gateway WebSocket reconnect flooding after update.

## Best Possible Solution

Land a measured, coherent doctor/update performance fix that bounds duplicate stale snapshot/path work, avoids avoidable per-call gateway reconnect churn, and proves both update-time and repeated `doctor --fix` behavior on a large synthetic or real multi-agent setup.

## Reproduction Assessment

No high-confidence live reproduction from this checkout. Source inspection matches the reported unbounded/serial paths and fresh-per-call gateway client behavior, and the reporter supplied concrete timings and logs, but I did not recreate the 90-session VPS benchmark.

## Solution Assessment

Yes, keeping this open and routing through the active repair PRs is the best current path. A new duplicate repair job would add noise until maintainers choose between the gateway-pooling and doctor-performance approaches or consolidate them.
