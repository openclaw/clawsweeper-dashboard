## Summary

Keep open. The reporter added credible log evidence that the long doctor/update stall is a stale pre-auth gateway backend retry flood, while current main still has the unbounded session-snapshot scan from the first report and no server-side throttling for missing shared-token auth.

## What This Changes

The issue reports a v2026.5.20 regression where `openclaw doctor --fix` and update can stall for minutes after upgrade, first attributed to stale session snapshot scanning and later to a gateway WebSocket auth retry flood from an old backend.

## Best Possible Solution

Target the update-time stale backend auth flood first, then measure and bound duplicate session snapshot validation if it still contributes materially to `doctor --fix` runtime.

## Reproduction Assessment

No high-confidence local reproduction. Source inspection matches the per-path snapshot scan and unthrottled missing-token auth path, and the reporter supplied concrete timing and log evidence, but I did not recreate the 90-session 5.19-to-5.20 upgrade scenario.

## Solution Assessment

Unclear as a single fix. A narrow repair should first stop stale pre-token gateway backends from flooding the new gateway, while snapshot-scan batching should be validated separately against the same large-store shape.
