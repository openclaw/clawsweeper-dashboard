## Summary

Keep open: current main still exposes bundled MCP tool schemas from external servers without a generic dereference/union-normalization pass, and the reported Notion server schema really uses a `parent` `$ref` to a `oneOf` union.

## What This Changes

The issue asks OpenClaw to resolve nested `$ref`/`oneOf` MCP tool parameter schemas so Notion page creation `parent` arguments validate and reach the Notion MCP server.

## Best Possible Solution

Add a generic bundled-MCP schema normalization step that resolves local `$defs`/`$ref` and preserves or flattens object `oneOf`/`anyOf` shapes into provider-safe tool parameters, with a Notion-like regression fixture.

## Reproduction Assessment

No live high-confidence reproduction was run; source inspection gives a clear reproduction path. The Notion v2.2.1 schema has the reported `$ref`/`oneOf` `parent` shape, and current main passes that schema through on bundled MCP tools after the normal schema-normalization path.

## Solution Assessment

Yes, the narrow maintainable fix is in generic bundled MCP schema normalization, not a Notion special case. It should keep runtime call arguments unchanged while making the exposed tool schema acceptable to OpenClaw/provider validation.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78737. Current main materializes bundled MCP tools with raw `inputSchema`, so schemas like Notion `notion__API-post-page` where `parent` is a local `$ref` to a `oneOf` object union can fail OpenClaw/provider validation before the MCP server receives the call. Implement a generic fix, not a Notion-specific workaround: normalize bundled MCP tool parameter schemas before they are exposed to the agent/provider, resolving local `#/$defs/...` or `#/definitions/...` references and making object `oneOf`/`anyOf` unions provider-safe where possible. Preserve runtime tool-call argument shape and do not change `mcp.servers` config semantics. Add regression coverage with a synthetic Notion-like MCP catalog schema containing `parent: { $ref: '#/$defs/parentRequest' }` and `$defs.parentRequest.oneOf` variants for `page_id` and `database_id`, asserting the materialized/effective tool schema no longer exposes unresolved local refs in the provider path and still forwards valid inputs to `runtime.callTool`. Add a user-facing changelog fix entry.
