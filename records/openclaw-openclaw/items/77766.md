## Summary

Keep open: current main treats this as documented strict-selection behavior rather than broken fallback plumbing, and the requested fallback-only pin exception needs maintainer product policy.

## What This Changes

The issue requests preserving or warning about the global fallback chain when a session is pinned to a model that appears only in configured fallbacks.

## Best Possible Solution

Decide and document the intended session-pin policy, then align code and regression tests either to keep strict pins with clearer diagnostics or to allow fallback-only pins to recover through a defined next candidate.

## Reproduction Assessment

Yes for the observed current behavior: the source path sends an explicit empty fallback override for user-pinned session models, so the candidate chain can collapse to the requested model only. I did not establish a live failing timeout transcript in this read-only review.

## Solution Assessment

No as an automatic bug fix: adding the configured primary as a fallback would conflict with the documented strict user-override policy. The safer path is a maintainer policy decision followed by code, docs, and tests that encode the chosen behavior.
