## Summary

Keep open: current main still intentionally treats user-pinned session models as strict exact selections, so the reported fallback-chain collapse remains a product-policy question rather than an implemented fix or duplicate cleanup case.

## What This Changes

Requests that a session pinned to a model that only appears in global fallbacks keep a recovery path, or at least report the exhausted fallback clearly instead of collapsing to one candidate.

## Best Possible Solution

Maintainers should decide whether strict user pins remain the contract or whether fallback-only pins get a narrow documented recovery rule, then update code, docs, and regression tests together.

## Reproduction Assessment

Yes, source-reproducible: a user session model override resolves to fallbacksOverride: [], and the fallback builder treats that empty override as exact, so the chain can collapse to the pinned model only. I did not run a live timeout transcript.

## Solution Assessment

No, not as an automatic bug fix: the proposed empty-override expansion conflicts with the current documented strict-selection contract. A maintainer policy decision should come first, then a narrow implementation if the contract changes.
