## Summary

Close as duplicate/superseded: the current strict user-pinned fallback behavior is still present, but the remaining policy question and active fix review are now tracked by https://github.com/openclaw/openclaw/issues/84865 and its linked PRs.

## What This Changes

Requests that a user-pinned session selecting a model from the global fallback list preserve fallback recovery or warn instead of collapsing to one candidate with `fallbackConfigured:false`.

## Best Possible Solution

Review the canonical open issue and PRs, then either keep user-selected models strict with visible failure behavior or add an explicit maintainer-approved resilient fallback mode with docs and tests.

## Reproduction Assessment

Yes, source-reproducible: current main returns `[]` for user-sourced session model overrides, and the fallback runner treats an empty override as an exact one-model chain. I did not run the live timeout/provider-outage path in this read-only review.

## Solution Assessment

No, not as originally proposed: adding the configured primary whenever the chain collapses conflicts with the current strict empty-override contract. The better path is the canonical policy review on user-pin fallback semantics, with docs and regression tests matching the accepted behavior.
