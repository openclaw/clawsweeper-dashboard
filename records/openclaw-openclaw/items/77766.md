## Summary

Keep open: current main and v2026.6.6 still treat user-pinned session models and empty fallback overrides as strict selections, so the reported single-candidate `fallbackConfigured:false` path remains source-reproducible. The direct fix PR closed unmerged, and the broader open user-pin fallback PR is not a safe canonical replacement because it changes documented routing policy and still needs maintainer judgment.

## What This Changes

The issue asks OpenClaw to preserve or warn about the fallback chain when a user pins a session to a model that is also the configured fallback, instead of dedup leaving a one-model chain.

## Best Possible Solution

Make an explicit maintainer decision on strict user pins versus resilience; if resilience should win, implement a session-pin-specific policy with docs and focused tests rather than broadening all empty fallback override semantics.

## Reproduction Assessment

Yes for the source-level path: user session model overrides resolve to an empty fallback override, candidate construction treats that as exact, and `fallbackConfigured` becomes false when only one candidate remains. No live timeout was run in this read-only review.

## Solution Assessment

Unclear. Appending the configured primary would address the surprise, but the direct PR broadened empty-override semantics; the maintainable fix needs an approved session-pin policy or an explicit rejection of fallback preservation for exact pins.
