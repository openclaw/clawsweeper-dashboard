## Summary

Keep open for maintainer decision: current main and v2026.6.6 still make user-pinned session models strict, while the requested fallback-chain preservation would change the documented model-selection contract. The related merged PR improves visible fallback failure handling, but it does not implement fallback-chain preservation for user pins; the direct fix PR closed unmerged because it broadened empty-override semantics.

## What This Changes

Request to preserve or warn on the fallback chain when a session is pinned to a model that only appears in configured model fallbacks.

## Best Possible Solution

Make an explicit maintainer decision on the user-pin fallback contract; if resilience should win, implement a session-pin-specific policy with focused tests and docs, otherwise close as intended behavior with the current failover docs as the supported contract.

## Reproduction Assessment

Yes. Source inspection on current main shows user-sourced session overrides return `[]` from `resolveEffectiveModelFallbacks`, and `resolveFallbackCandidatesUncached` treats an empty override as an exact single-candidate chain; I did not run a live timeout in this read-only review.

## Solution Assessment

Unclear. Appending the configured primary would address the reported surprise, but the broad version conflicts with the current strict user-selection contract; the safer path is a maintainer-approved, session-pin-specific policy or an explicit decision to keep strict behavior.
