## Summary

Keep open: current main and latest shipped release still treat explicit user model pins as strict selections, so the reported `fallbackConfigured:false` path remains source-reproducible. The linked merged PR improves visible failure handling, but it does not implement the requested fallback-chain preservation, and changing that contract needs maintainer product judgment.

## What This Changes

The issue asks OpenClaw to preserve or warn about the global fallback chain when a user pins a session to a model that only appears in configured fallbacks.

## Best Possible Solution

Make an explicit maintainer decision on the user-pin contract; if resilience should win, implement a session-pin-specific policy with focused tests and docs, not a broad change to all empty fallback overrides.

## Reproduction Assessment

Yes. Source inspection shows a user session model override resolves to an empty fallback override, and current candidate construction treats that as an exact single-candidate chain; I did not run a live timeout because this review is read-only.

## Solution Assessment

Unclear. The proposed configured-primary fallback would address the surprise, but it conflicts with the current documented strict user-selection contract and should be narrowed or rejected by maintainers first.
