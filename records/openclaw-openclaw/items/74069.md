## Summary

Keep open. The report is recent, current main still contains the staged npm-global verify/rollback path, and the latest comment identifies a concrete `v`-prefixed version mismatch that current source still compares as a raw string.

## What This Changes

The issue reports that `openclaw update --tag 2026.4.25` fails to downgrade an npm-global install from 2026.4.26 to 2026.4.25 and leaves the installed version unchanged.

## Best Possible Solution

Canonicalize explicit npm version specs for staged global verification, preserve real mismatch protection, and prove both a bare downgrade and a `v`-prefixed release tag update in focused tests plus package-acceptance proof.

## Reproduction Assessment

Yes for source triage, but not as a live current-main run. The latest `v`-prefixed failure is directly reproducible from current source because the verifier compares raw expected and installed version strings, while the original bare downgrade still needs live package-manager isolation.

## Solution Assessment

Unclear as a complete fix until the original bare downgrade is isolated. The narrowest maintainable direction is to canonicalize semver-like expected versions at the verifier boundary and add regression proof for both reported update shapes.
