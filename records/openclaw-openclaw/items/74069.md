## Summary

Keep open. Current main still has the staged npm verification path that can roll back with `global install verify`, and the May 11 v-prefixed tag reproduction is source-reproducible because planning accepts `v...` versions while exact install verification compares the raw requested version against the bare installed package version.

## What This Changes

Fix npm-global `openclaw update --tag <older version>` so explicit downgrades such as 2026.4.26 to 2026.4.25 do not roll back with `global install verify`.

## Best Possible Solution

Normalize exact version handling consistently across target planning, install-spec verification, and staged package checks, then prove explicit downgrade and v-prefixed tag cases without weakening rollback safeguards.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the related v-prefixed `--tag` rollback: current main preserves `openclaw@v...` for exact verification and compares it to the bare package version. The original bare downgrade has real reporter output, but I did not run a live npm-global downgrade here.

## Solution Assessment

Is this the best way to solve the issue? Not yet; current main detects the mismatch and safely rolls back, but it does not normalize exact version comparisons or prove the explicit downgrade path succeeds. The narrow repair is to normalize expected versions and add focused downgrade/v-prefix coverage while preserving verification failures for real mismatches.
