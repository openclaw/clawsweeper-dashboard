## Summary

Keep open. Current main still accepts a confirmed npm downgrade and sends it through the staged global install, exact-version verification, and swap path, but the source and tests do not prove the reported 2026.4.26-to-2026.4.25 Linux npm-global downgrade now succeeds. The earlier ClawSweeper comment correctly identified this as an installer/update bug needing maintainer follow-up rather than cleanup closure.

## What This Changes

Allow `openclaw update --tag 2026.4.25` to downgrade an npm global install from 2026.4.26 without failing `global install verify`.

## Best Possible Solution

Fix the npm global downgrade path so an explicit older target installs into the staged prefix, verifies as that exact version, swaps into the live global root, and preserves post-update doctor, plugin sync, completion refresh, and restart behavior. Add focused regression coverage for the 2026.4.26-to-2026.4.25 case plus Linux npm-global package proof before release.

## Reproduction Assessment

Yes for triage: the report gives an exact command and output, and current main still has the matching staged install plus exact-version verification path. A live downgrade was not run because this review was constrained to read-only repository inspection.

## Solution Assessment

No. Current main is not yet the best solution because it lacks proof that the explicit older version reaches and survives the staged install, verify, and swap flow; the narrow fix belongs in the package update helper and installer tests with real package acceptance proof.
