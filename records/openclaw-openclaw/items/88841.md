## Summary

Keep this PR open for normal maintainer merge review. The updated branch is a focused defensive recovery: it re-arms reconnect only for an abnormal close on the active socket when no intentional disconnect is recorded, then still applies fatal-close handling. The live production evidence is unusually strong, and I found no discrete correctness or security defect. The residual uncertainty is that the transition which left reconnect disabled on an active socket remains unidentified.

## What This Changes

The PR makes Discord reconnect after abnormal non-intentional gateway closes when reconnect was unexpectedly disabled, adds diagnostic logging, and covers recovery and intentional-disconnect behavior.

## Best Possible Solution

Merge the instrumented close-handler recovery after maintainer acceptance, then use its distinct diagnostic event to identify and eliminate the hidden stale-state transition if another occurrence is captured.

## Reproduction Assessment

## Solution Assessment

Yes, with documented root-cause uncertainty. The active-socket close handler is the narrowest reliable boundary for recovering abnormal non-intentional closes, and the branch preserves intentional, normal, stale-socket, and fatal-close behavior.
