## Summary

Keep open: the reconnect mitigation is plausible and backed by strong live logs, but the PR currently misses its own traceability requirement because the new re-arm diagnostic is emitted only as a generic debug message and is not promoted by Discord gateway logging. Maintainer review is still needed for the instrumented close-handler mitigation because the production-observed stale state has no identified public lifecycle transition.

## What This Changes

The PR adds an intentional-disconnect flag, re-arms Discord gateway reconnect for abnormal non-intentional closes when reconnect was disabled, and adds gateway regression tests.

## Best Possible Solution

Promote and test the re-arm diagnostic through Discord gateway logging, then have a maintainer decide whether the bounded close-handler recovery is acceptable while the hidden owner transition is traced from future logs.

## Reproduction Assessment

Partly: the PR body provides redacted live logs showing stock runtime dead-lane behavior after a 1006 close, and current main has the early-return path when shouldReconnect is false. I do not see a high-confidence public API path that constructs the active-socket, reconnect-disabled state on current main.

## Solution Assessment

Not as-is: the close handler is a reasonable choke-point mitigation for the observed state, but the new tracing signal is not promoted to operational gateway logs. The safer merge path is to fix that diagnostic gap, then let the Discord owner decide whether the mitigation is acceptable without a public-path root cause.
