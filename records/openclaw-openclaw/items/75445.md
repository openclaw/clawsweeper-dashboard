## Summary

Keep open: the reported SecretRef crash is a valid plugin-owned bug, but the latest PR still has a P2 Telegram multi-account regression and only provides unit-test output, not required real embedded-run proof.

## What This Changes

The branch adds Telegram unresolved-SecretRef fallbacks for channel-action, inline-button, and reaction discovery, adds Discord/Telegram regression tests, and updates CHANGELOG.

## Best Possible Solution

Keep the plugin-owned discovery hardening, but degrade per Telegram account or facet so valid accounts still advertise actions, then require after-fix embedded reply proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main reaches Telegram prompt capability and reaction helpers that resolve raw non-env SecretRefs before runtime snapshot resolution. I did not run a live embedded reply because this review is read-only.

## Solution Assessment

No. Catching unresolved SecretRefs in plugin discovery is the right layer, but the latest channel-wide Telegram fallback is too broad and should skip only the unresolved account or return conservative per-facet defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Skip unresolved Telegram accounts instead of disabling discovery:** `extensions/telegram/src/channel-actions.ts:74-79`
  - body: When `listEnabledTelegramAccounts(cfg)` throws on one unresolved SecretRef, this catch returns empty discovery for the whole Telegram channel. In a mixed multi-account config, that hides actions for other enabled accounts with usable tokens, so discovery should filter or inspect the failing account instead of aborting channel-wide.
