## Summary

Keep open. The bug is valid and the PR is pointed at the right plugin-owned discovery layer, but it is not merge-ready: the Telegram fallbacks still drop explicit inline-button and reaction configuration, and the Discord part is stale against current main's config-inspection behavior for SecretRef-backed discovery.

## What This Changes

The PR wraps Discord and Telegram channel-action and Telegram prompt-capability discovery paths so unresolved SecretRef credentials do not crash embedded prompt preparation, adds regression tests, and adds a changelog entry for the embedded reply crash.

## Best Possible Solution

Land a rebased plugin-owned fix that keeps SecretRef resolution strict for runtime send/probe paths, uses config-only or inspect-based discovery during prompt prep, preserves explicit Telegram inline-button and reaction settings, and keeps current main's Discord inspect-based action discovery.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call Telegram prompt capability or reaction guidance discovery with a config containing a non-env SecretRef botToken; current main reaches resolveTelegramAccount and strict non-env SecretRef resolution before model output.

## Solution Assessment

No, not as submitted. The layer is right, but the fallback should read merged config without resolving credentials instead of replacing explicit settings with defaults, and the Discord part should align with current main's inspect-based discovery.
