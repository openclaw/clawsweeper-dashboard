## Summary

Keep open: this is a real current-main Telegram embedded prompt-prep bug and the PR is in the right plugin-owned area, but the current head still drops all unscoped Telegram action discovery when one enabled account has an unresolved SecretRef, and the supplied proof is resolver-test output rather than a real embedded/channel run.

## What This Changes

The PR adds Telegram unresolved-SecretRef fallbacks for message-action, inline-button, and reaction prompt discovery, adds Discord/Telegram regression tests, and updates the changelog.

## Best Possible Solution

Keep the plugin-local SecretRef tolerance, but degrade Telegram action discovery per account so unresolved accounts are skipped while healthy accounts still advertise actions, then add redacted embedded/channel proof with SecretRef-backed credentials.

## Reproduction Assessment

Yes by source inspection: current main's embedded prompt-prep calls Telegram prompt/action discovery with raw config, and those helpers can reach strict non-env SecretRef resolution before model output. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. Catching unresolved SecretRefs in plugin discovery is the right layer, but the unscoped Telegram action path still aborts channel-wide discovery instead of skipping only the unresolved account.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Skip only unresolved Telegram accounts during discovery:** `extensions/telegram/src/channel-actions.ts:76-77`
  - body: `listEnabledTelegramAccounts(cfg)` resolves every enabled account before filtering. In a multi-account setup, one exec/file-backed account can throw into this catch and return `null`, hiding actions for healthy sibling accounts during prompt prep. Degrade per account, for example with account inspection or per-account resolution, instead of aborting channel-wide discovery.
  - confidence: 0.92
