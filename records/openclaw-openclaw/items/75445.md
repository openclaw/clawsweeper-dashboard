## Summary

Keep open. The source-level bug is real and the PR covers several Telegram discovery paths, but PR head still drops all unscoped Telegram message-action discovery when any enabled account has an unresolved SecretRef, and the supplied real-behavior proof is still resolver-test output rather than an embedded/channel run.

## What This Changes

The PR adds Telegram fail-soft handling for unresolved SecretRef tokens in message-action, inline-button, and reaction prompt discovery, adds Discord/Telegram regression tests, and updates the changelog.

## Best Possible Solution

Use plugin-local read-only account inspection or per-account degradation for optional discovery so unresolved credentials do not crash prompt prep and healthy accounts still advertise capabilities, while strict runtime send/probe behavior remains unchanged.

## Reproduction Assessment

Yes by source inspection and linked issue logs. Current main still routes embedded prompt prep through raw config into Telegram account/token resolution, where non-env SecretRefs intentionally throw before model output.

## Solution Assessment

No. Fail-soft plugin discovery is the right layer, but the current PR head aborts unscoped Telegram discovery for all accounts when one account has an unresolved SecretRef; filtering or inspecting per account is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Skip unresolved accounts during discovery:** `extensions/telegram/src/channel-actions.ts:76-77`
  - body: `listEnabledTelegramAccounts(cfg)` resolves every enabled account before filtering, so one exec/file-backed account throws into this catch and returns `null` for the entire unscoped Telegram discovery surface. In multi-account installs that hides healthy env/plain accounts from prompt prep; degrade per account, for example by inspecting account config or resolving accounts individually and filtering only the failing account.
  - confidence: 0.92
