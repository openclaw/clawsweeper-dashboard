Keep open. Current main still lacks the Bonjour no-valid-addresses classifier and the PR targets a real linked crash, but it is not merge-ready because the new shared classification can still drive the advertise-failure recovery path, and the branch also bundles a separate WhatsApp QR runtime fix.

## What This Changes

The PR adds a Bonjour/CIAO `no-valid-addresses` classification and warning path with tests/changelog, and also routes WhatsApp login QR output through the injected runtime with a changelog entry.

## Best Possible Solution

Land a narrow Bonjour fix that classifies this ciao assertion and treats it as handled without recovery in every Bonjour handler path, with classifier and advertiser regression coverage; split the WhatsApp QR fix if maintainers want separate review.

## Reproduction Assessment

Yes, by source inspection. Current main misses the exact ciao assertion, the pinned ciao 1.3.7 package contains that assertion, and unclassified rejections fall through to the global fatal handler.

## Solution Assessment

No, not as-is. The direction is right, but the new shared `no-valid-addresses` kind should also bypass `requestCiaoRecovery` in `handleAdvertiseFailure`, and the unrelated WhatsApp change needs an intentional keep-or-split decision.
