## Summary

Keep open. The Bonjour fix addresses a real current-main bug, but the PR is not merge-ready because the new shared `no-valid-addresses` classification still triggers advertiser recovery through `handleAdvertiseFailure`, which conflicts with the intended no-recovery behavior.

## What This Changes

The PR adds a Bonjour/CIAO `no-valid-addresses` classifier with tests and changelog text, and also routes WhatsApp login QR output through the injected runtime.

## Best Possible Solution

Land a narrow Bonjour fix that classifies this ciao assertion and treats it as handled without recovery in every Bonjour handler path, with classifier and advertiser regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main misses the exact ciao assertion, the pinned ciao 1.3.7 package contains that assertion, and unclassified rejections fall through to the fatal unhandled-rejection path.

## Solution Assessment

No, not as-is. The classifier and process-level warning direction is right, but the shared new kind also needs no-recovery handling in `handleAdvertiseFailure`; otherwise one ciao surface can still restart the advertiser for an address-less interface.
