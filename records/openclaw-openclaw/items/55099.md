## Summary

Keep open. Current main still allows Telegram transport fallback to promote to forced IPv4 when a fetch-failed aggregate contains timeout/socket codes, without checking whether the IPv4 leg was an intentional loopback refusal or preserving the configured verbatim IPv6 path in that case. The current tests assert sticky IPv4 persistence but do not cover the reported `/etc/hosts` loopback failure, and the latest comments add a related repeated-fallback symptom rather than making the report obsolete.

## What This Changes

The issue reports that Telegram’s sticky IPv4 fallback can still latch or repeatedly re-arm incorrectly on macOS dual-stack edge cases, especially with `dnsResultOrder: "verbatim"` and loopback IPv4 refusal.

## Best Possible Solution

Keep the fix inside the Telegram plugin: make fallback classification and attempt state aware of loopback IPv4 refusals, avoid promoting or staying on forced IPv4 when that promoted path is known-dead, and preserve the existing IPv4/pinned-IP fallback for real `ETIMEDOUT`, `EHOSTUNREACH`, and media-download reachability failures.

## Reproduction Assessment

Yes. A focused unit reproduction can model `resolveTelegramFetch` with a first `TypeError("fetch failed")` aggregate containing loopback IPv4 `ECONNREFUSED` and IPv6 `ETIMEDOUT`, then verify current main promotes to `family: 4` and remains sticky after the loopback retry fails; live macOS/TUN networking was not run in this read-only pass.

## Solution Assessment

Yes, the best direction is a narrow Telegram-owned transport fix with regression tests. Moving this into core or disabling sticky fallback broadly would be less maintainable because #40740 and #45327 intentionally rely on the existing direct Telegram fallback chain for other network failures.
