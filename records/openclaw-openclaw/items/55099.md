## Summary

Keep open. Current main still has a Telegram transport fallback path that promotes to IPv4-only based on timeout/socket codes without checking whether the IPv4 candidate was a loopback-refused address, so the reported dead-path latch remains plausible and source-reproducible despite newer polling lifecycle mitigations.

## What This Changes

The issue asks OpenClaw to stop Telegram’s sticky IPv4-only fallback from latching onto intentionally loopback-refused IPv4 when IPv6 is the intended working route.

## Best Possible Solution

Keep the repair in the Telegram plugin transport layer: make fallback promotion loopback/address-aware, avoid retaining IPv4-only fallback when IPv4 resolved to a known-dead loopback path, and add focused regression coverage for mixed aggregate errors under `dnsResultOrder: "verbatim"`.

## Reproduction Assessment

Yes at source level. A `fetch failed` aggregate containing a timeout-class IPv6 error and loopback IPv4 `ECONNREFUSED` still qualifies for fallback, then the sticky IPv4 attempt starts before success and can latch onto the dead family.

## Solution Assessment

Yes, a narrow Telegram-owned transport classifier/state fix is the best direction. Changing global DNS defaults or moving this into core would be broader than needed and could regress the resolver-scoped transport model.
