## Summary

Keep open. Current main still promotes Telegram transport to an IPv4-only dispatcher when a fetch-failed error graph contains a known timeout/socket code, without checking that the IPv4 address was loopback-refused or that the configured `verbatim` IPv6 path had previously worked. The newer polling-lifecycle work partly addresses sticky state across restarts, but it does not solve the original loopback-IPv4 fallback classifier problem.

## What This Changes

The issue requests a Telegram transport fix so sticky IPv4 fallback does not latch onto a dead loopback IPv4 path when macOS `/etc/hosts` and `dnsResultOrder: "verbatim"` intentionally route Telegram over IPv6.

## Best Possible Solution

Keep the fix in the Telegram plugin transport layer: make fallback classification/address-state loopback-aware, avoid promoting or retaining IPv4-only fallback when IPv4 resolved to loopback and failed by design, and add focused regression tests for mixed aggregate errors under `dnsResultOrder: "verbatim"` while preserving existing real broken-IPv6 and media fallback behavior.

## Reproduction Assessment

Yes at code level. The issue provides concrete config/logs, and current source gives a deterministic regression-test shape: a `fetch failed` aggregate containing loopback IPv4 `ECONNREFUSED` plus a known IPv6 timeout still promotes to the IPv4-only dispatcher.

## Solution Assessment

Yes, a narrow Telegram-owned classifier/state fix is the best direction. Moving this into core or changing global DNS defaults would be broader than needed and would risk breaking the existing resolver-scoped transport model from #40740.
