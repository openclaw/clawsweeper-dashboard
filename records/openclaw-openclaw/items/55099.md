## Summary

Keep open. Current main still allows a mixed Telegram network error to promote the transport to IPv4-only before that fallback has succeeded, and a following loopback `ECONNREFUSED` path can leave the sticky attempt on the dead IPv4 dispatcher.

## What This Changes

The issue asks OpenClaw to stop Telegram's sticky IPv4-only fallback from latching onto or repeatedly re-entering a dead dispatcher when mixed IPv4/IPv6 failures occur on macOS network setups.

## Best Possible Solution

Keep the fix in the Telegram plugin transport layer: make fallback promotion address-aware, avoid retaining an IPv4-only dispatcher when the IPv4 candidate is loopback-refused, and cover the mixed aggregate error case with focused regression tests.

## Reproduction Assessment

Yes, at source level. A primary fetch failure containing IPv6 timeout-class codes plus IPv4 loopback `ECONNREFUSED` still promotes to IPv4-only, and a subsequent plain `ECONNREFUSED` does not demote or continue safely.

## Solution Assessment

Yes, a narrow Telegram transport classifier/state repair is the best direction. Changing global DNS defaults or moving this into core networking would be broader than the plugin-owned fallback defect.
