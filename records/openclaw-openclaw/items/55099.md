## Summary

Keep open. Current main still has a source-visible Telegram path where a mixed timeout/refused error can promote the transport to IPv4-only before that fallback has succeeded, leaving later requests stuck on a refused IPv4 dispatcher.

## What This Changes

The issue asks OpenClaw to stop Telegram's sticky IPv4-only fallback from latching onto a dead IPv4 dispatcher when IPv6 is the intended working path and IPv4 is intentionally refused through `/etc/hosts`.

## Best Possible Solution

Fix the Telegram plugin transport layer so IPv4-only fallback is promoted only after a successful or clearly useful IPv4 attempt, demotes or avoids stickiness when the IPv4 candidate is loopback-refused, and has focused regression coverage for mixed IPv6 timeout plus IPv4 `ECONNREFUSED` under `dnsResultOrder: "verbatim"`.

## Reproduction Assessment

Yes, at source level. Current main still has a path where a mixed Telegram network error can qualify IPv4 fallback and promote the IPv4-only dispatcher before that dispatcher has proved it can connect.

## Solution Assessment

Yes, a narrow Telegram transport classifier and sticky-attempt state repair is the best direction. Changing global DNS defaults or adding a user-facing config workaround would be broader than the plugin-owned fallback defect.

## Repair Work Prompt

Fix the Telegram sticky IPv4 fallback bug reported in https://github.com/openclaw/openclaw/issues/55099. Keep the change inside the Telegram plugin transport layer; do not add global DNS defaults, new user config, or core networking policy. Make fallback promotion address-aware so a mixed IPv6 timeout plus IPv4 loopback `ECONNREFUSED` under `channels.telegram.network.dnsResultOrder: "verbatim"` does not retain a dead IPv4-only dispatcher before that attempt succeeds. Add focused regression coverage in `extensions/telegram/src/fetch.test.ts` for the reported mixed aggregate case and for non-promotion or demotion when the IPv4 candidate is loopback-refused. Preserve existing proxy and caller-owned dispatcher behavior, sticky IPv4 success behavior for genuinely IPv6-broken hosts, pinned-IP fallback behavior, and recovery/cooldown behavior.
