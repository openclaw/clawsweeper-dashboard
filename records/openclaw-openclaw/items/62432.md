## Label Justifications

- `P2`: This is a real agent runtime bug with process fan-out and token waste, but it is bounded to long-running exec orchestration rather than a global outage.
- `impact:crash-loop`: The reported failure mode is a runaway loop that repeatedly launches background processes and can make sessions appear stuck.
- `impact:session-state`: The bug is a tool-state transition failure where a session starts independent exec sessions instead of polling the in-flight process.

## Summary

Keep open: current main still creates a fresh process session for each repeated long-running exec call and lacks a same-command in-flight guard before the “Command still running” response path.

## What This Changes

The issue asks OpenClaw to stop repeated identical long-running exec relaunches after a non-terminal “Command still running” result and steer the session toward process polling or intervention.

## Best Possible Solution

Add a provider-agnostic in-flight exec identity guard that returns the existing session id with process-poll guidance or rejects duplicate launches while preserving legitimate terminal or intentionally separate reruns.

## Reproduction Assessment

Yes, as a source-level reproduction: repeated exec calls still flow through runExecProcess and register new session ids, with no command/cwd/session identity lookup before registration. I did not run a live Xiaomi/MiMo replay in this read-only review.

## Solution Assessment

Yes. A narrow provider-agnostic runtime guard is the best fix because it prevents process fan-out across models without hard-coding Xiaomi/MiMo policy or relying on opt-in loop detection as the only protection.
