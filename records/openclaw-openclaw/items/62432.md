## Label Justifications

- `P2`: This is a real agent runtime bug with process fan-out and token waste, but it is bounded to long-running exec orchestration.
- `impact:crash-loop`: The reported failure mode is a repeated tool loop that can make sessions appear stuck and keep launching processes.
- `impact:session-state`: The bug is a tool-state transition failure where the agent loses the intended in-flight exec/process follow-up state.

## Summary

Keep open: current main still creates and registers a fresh exec process session before returning the running-result guidance, and the inspected registry/tool-loop paths do not provide a default same-command in-flight guard.

## What This Changes

The issue requests a runtime guard so repeated long-running `exec` calls follow the existing `process` session or stop instead of relaunching the same in-flight command.

## Best Possible Solution

Add a provider-agnostic in-flight exec identity guard that returns the existing session id with process-poll guidance or rejects duplicate launches while preserving legitimate terminal or explicitly separate reruns.

## Reproduction Assessment

Yes, as a source-level reproduction: repeated identical `exec` calls still flow through `runExecProcess`, allocate a new session id, and register a new session before the running result tells the model to use `process`. I did not run a live Xiaomi/MiMo replay.

## Solution Assessment

Yes. A narrow provider-agnostic runtime guard is the maintainable fix because it prevents process fan-out across models without hard-coding Xiaomi/MiMo policy or relying on opt-in loop detection as the only protection.
