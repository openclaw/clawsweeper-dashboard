## Summary

Keep this open. Current main still starts a fresh exec process session before returning the non-terminal running result, while the only repeated-running exec loop protection remains behind opt-in loop detection that is disabled by default.

## What This Changes

The issue requests a provider-agnostic runtime guard so Xiaomi/MiMo sessions do not relaunch identical exec commands after a `Command still running` result.

## Best Possible Solution

Add a default, provider-agnostic in-flight exec coalescing guard in the exec/process layer that returns the existing running session or a clear poll requirement before spawning another process.

## Reproduction Assessment

Yes. With the documented default `tools.loopDetection.enabled: false`, a repeated identical long-running `exec` follows the same `runExecProcess` path and allocates a new session before returning `status: running`.

## Solution Assessment

Yes. A narrow runtime dedupe/coalescing guard is the maintainable fix because it prevents process fan-out for every model/provider; Xiaomi-specific routing should stay out of the core fix unless maintainers choose that policy separately.
