## Summary

Keep this open. Current main still starts a fresh exec process session before returning the non-terminal running result, while repeated-running exec protection remains behind opt-in loop detection that is disabled by default.

## What This Changes

The issue asks for runtime protection against repeated identical long-running `exec` relaunches after a non-terminal running result.

## Best Possible Solution

Add provider-agnostic in-flight exec coalescing in the exec/process runtime so identical still-running work returns the existing session or an explicit poll-required result before spawning another process.

## Reproduction Assessment

Yes. With the documented default `tools.loopDetection.enabled: false`, a repeated identical long-running `exec` follows the same `runExecProcess` path and registers a fresh session before returning `status: running`.

## Solution Assessment

Yes. A narrow runtime coalescing guard is the maintainable fix because it protects all providers and avoids putting Xiaomi/MiMo-specific routing policy into core orchestration.
