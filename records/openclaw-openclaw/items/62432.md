## Summary

Keep this open. Current main has useful opt-in loop detection for repeated running exec outcomes, but the default exec path still creates a fresh process session before returning the non-terminal result, and loop detection remains disabled by default. That does not yet satisfy the requested runtime guard against Xiaomi/MiMo repeatedly relaunching identical in-flight exec commands.

## What This Changes

The issue asks OpenClaw to stop agents, especially Xiaomi/MiMo-backed sessions, from relaunching identical long-running `exec` commands after a non-terminal `Command still running` result and to require `process` polling or intervention instead.

## Best Possible Solution

Add a default runtime-level guard in the exec/process layer that recognizes an identical in-flight exec within the same agent/session scope and returns the existing running session or a clear `process(action="poll")` requirement instead of spawning another process. Keep the fix provider-agnostic, add focused regression coverage, and leave provider-specific Xiaomi routing out of the core solution unless maintainers explicitly want that policy.

## Reproduction Assessment

Yes. With `tools.loopDetection.enabled` at its documented default `false`, repeatedly issuing the same long-running `exec` in one session reaches `runExecProcess` each time, which creates a fresh session before returning the running result.

## Solution Assessment

Yes for a narrow runtime dedupe/coalescing guard; that directly prevents process fan-out regardless of model behavior. Provider-aware routing is not the best primary fix because the failure mode is a generic tool-state transition problem.
