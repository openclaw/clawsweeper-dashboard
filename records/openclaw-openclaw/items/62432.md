## Summary

Current main still accepts a repeated identical long-running exec by starting a new process session before returning the non-terminal running result; the only repeated-call protection I found is opt-in loop detection, which remains disabled by default.

## What This Changes

The issue requests a provider-agnostic guard so repeated long-running exec calls do not relaunch identical processes instead of polling the existing process.

## Best Possible Solution

Add a default runtime guard in the exec/process layer that coalesces or rejects an identical in-flight exec and directs the model to poll the existing session, without making Xiaomi/MiMo-specific policy part of core.

## Reproduction Assessment

Yes. Source inspection on current main shows that a repeated identical long-running exec calls runExecProcess again, which creates and registers a fresh session before returning `status: running`; no default guard intercepts it.

## Solution Assessment

Yes. A narrow provider-agnostic in-flight exec coalescing guard is the best fix because it addresses the runtime fan-out failure for all models while leaving provider routing policy out of this bug fix.

## Repair Work Prompt

Fix the default repeated-running exec fan-out reported in https://github.com/openclaw/openclaw/issues/62432. Add a provider-agnostic guard in the exec/process layer so an identical in-flight long-running exec from the same relevant session/scope/workdir/command does not spawn another process after `Command still running`; instead return the existing running session or a clear `process(action="poll")` requirement. Keep matching narrow so legitimate intentional duplicate commands are not globally denied. Do not make this Xiaomi/MiMo-specific and do not rely on `tools.loopDetection.enabled`, which is disabled by default.
