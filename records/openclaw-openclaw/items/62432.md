## Label Justifications

- `P2`: The issue is a real agent runtime bug with process fan-out and token waste, but it is bounded to long-running exec orchestration rather than a global outage.
- `impact:crash-loop`: The reported failure mode is a runaway loop that repeatedly launches background processes and can make sessions appear stuck.
- `impact:session-state`: The bug is a tool-state transition failure where the session loses track of an in-flight exec and starts independent sessions instead of polling the existing one.

## Summary

Codex review should keep this open: current main still allows the same long-running exec command to create another background session before the running result exists, and the loop-detection improvements only help when the opt-in guard is enabled.

## What This Changes

The issue asks OpenClaw to prevent repeated identical long-running exec calls from spawning new background sessions after a non-terminal running result, steering follow-up to process polling or intervention instead.

## Best Possible Solution

Add a provider-agnostic guard in the exec/process runtime that coalesces or rejects identical in-flight backgrounded execs, returns the existing session id with explicit process-poll guidance, and keeps opt-in loop detection as a secondary safety net.

## Reproduction Assessment

Yes, as a source-level reproduction: current main calls runExecProcess for each repeated exec, runExecProcess registers a fresh session, and the registry has no same-command in-flight coalescing. I did not run a live Xiaomi/MiMo replay in this read-only review.

## Solution Assessment

Yes. A narrow provider-agnostic in-flight exec coalescing or rejection guard is the best fit because it prevents process fan-out for every model without hard-coding Xiaomi/MiMo routing policy or changing the loop-detection default.
