## Summary

Current main still appears to accept a repeated identical long-running `exec` by creating another background session before returning the non-terminal running result. The existing tool-loop guard can catch repeats only when enabled, and docs still say it is disabled by default, so this remains a narrow core runtime bug rather than cleanup-close material.

## What This Changes

Prevent repeated identical long-running `exec` relaunches after a `Command still running` result by directing the agent to the existing process instead of spawning another one.

## Best Possible Solution

Add a provider-agnostic runtime guard in the exec/process layer that coalesces or rejects identical in-flight backgrounded execs and returns the existing session with explicit `process poll` guidance.

## Reproduction Assessment

Yes, as a source-level reproduction: on current main, issuing the same long-running `exec` again takes the same `runExecProcess` path and registers another session before returning `status: running`. I did not run a live Xiaomi/MiMo session in this read-only review.

## Solution Assessment

Yes. A narrow provider-agnostic in-flight exec coalescing or rejection guard is the maintainable fix because it prevents process fan-out for all models without hard-coding Xiaomi/MiMo routing policy into core.
