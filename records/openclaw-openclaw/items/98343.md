## Summary

Keep open: the PR still removes a shipped default Google Gemini CLI runtime provider/backend registration while manifest metadata, runtime contracts, existing tests, and docs still expose that provider. The env-gated opt-in also needs maintainer-approved compatibility direction and stronger real setup proof before merge.

## What This Changes

The PR adds an env-controlled policy helper that disables default Google Gemini CLI runtime provider/backend registration unless `OPENCLAW_ENABLE_GOOGLE_GEMINI_CLI_HARNESS` is set, preserves setup backend registration, and updates Google docs, metadata, and tests.

## Best Possible Solution

Keep Gemini CLI runtime registration compatible by default, or land an explicit maintainer-approved opt-in migration that aligns manifest metadata, docs, contracts, setup behavior, tests, and upgrade proof together.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.11` register `google-gemini-cli` by default, while the PR head hides that runtime registration unless the new env flag is set. I did not run a live upgrade flow in this read-only review.

## Solution Assessment

No, not as-is. The env gate is a plausible mechanism, but the best fix either preserves the shipped runtime default or makes the opt-in migration explicit across manifest metadata, docs, contracts, setup behavior, tests, and real upgrade proof.
