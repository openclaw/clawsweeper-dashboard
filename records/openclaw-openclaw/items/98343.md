## Summary

Keep open: current main and the latest release register `google-gemini-cli` by default, but this PR hides that runtime provider/backend unless a new env var is set while manifest metadata, docs, and provider runtime contracts still advertise it.

## What This Changes

The branch adds an `OPENCLAW_ENABLE_GOOGLE_GEMINI_CLI_HARNESS` env gate around bundled Google runtime Gemini CLI provider/backend registration, preserves setup-registry backend registration, and updates Google provider docs, metadata, and tests.

## Best Possible Solution

Keep shipped Gemini CLI runtime registration compatible by default, or land a maintainer-approved opt-in migration that aligns manifest metadata, docs, contracts, setup behavior, tests, and upgrade proof together.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.11` register `google-gemini-cli` by default, while PR head hides that runtime registration unless the new env flag is set. I did not run a live upgrade flow in this read-only review.

## Solution Assessment

No, not as-is. The env gate is a plausible mechanism, but the best fix either preserves the shipped runtime default or makes the opt-in migration explicit across manifest metadata, docs, contracts, setup behavior, tests, and real upgrade proof.
