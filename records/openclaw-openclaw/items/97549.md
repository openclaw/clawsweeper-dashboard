- `merge-risk: 🚨 security-boundary`: The new overflow path can expose raw provider download URLs that may include signed or tokenized access data.

## Summary

Keep this PR open: the bounded read is the right hardening direction, but the new overflow error exposes the full provider URL, the contributor proof is mock-only, and maintainers should explicitly accept the fail-closed behavior for oversized no-output video downloads.

## What This Changes

The PR replaces the CLI no-output video URL fallback's unbounded `arrayBuffer()` read with `readResponseWithLimit` using the generated-video byte cap and adds focused capability CLI tests.

## Best Possible Solution

Keep the shared bounded reader in the CLI fallback, remove the raw URL from the overflow error and matching test assertion, preserve `--output` as the large-video path, and add redacted real CLI proof before merge.

## Reproduction Assessment

Yes from source: current main fetches the provider URL and uses `Buffer.from(await response.arrayBuffer())` when `--output` is absent. A mock provider returning a large streaming response exercises the unbounded path, although I did not run tests in this read-only review.

## Solution Assessment

Mostly yes: the CLI fallback is the right owner boundary and the shared bounded reader matches sibling media code. As submitted, it is not merge-ready because the overflow error should be URL-free and the external PR still needs real behavior proof.
