## Summary

Keep open for maintainer review: the patch is a focused hardening fix for a real unbounded CLI download path, but it intentionally changes oversized no-output video URL downloads to fail closed and needs compatibility acceptance before merge.

## What This Changes

The PR replaces the no-output CLI video URL fallback's unbounded `arrayBuffer()` read with `readResponseWithLimit` under the generated-video byte cap and adds focused CLI regression tests.

## Best Possible Solution

Land the bounded-reader change once maintainers accept the fail-closed cap, keeping `--output` as the large-video streaming path and keeping overflow errors URL-free.

## Reproduction Assessment

Yes, from source inspection rather than executing a current-main repro. Current main fetches `video.url` and buffers `response.arrayBuffer()` when `--output` is absent; the PR body includes a real loopback HTTP negative control for the bounded behavior.

## Solution Assessment

Yes, with maintainer sign-off. The CLI no-output fallback is the implicated unbounded allocation path, and the shared helper matches sibling generated-media downloaders; if maintainers do not accept fail-closed behavior, they need to choose a different large-video default.
