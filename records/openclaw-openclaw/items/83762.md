## Label Justifications

- `P1`: This is a current official Slack channel workflow regression where thread follow-up messages can be skipped after participation state is not persisted.
- `impact:message-loss`: Unmentioned Slack thread replies can be dropped by mention gating even after the bot has already participated.
- `impact:session-state`: The failing surface is the persisted thread participation state used to remember prior Slack bot participation.

## Summary

Keep open: current main still has the reported contract mismatch between the official npm Slack plugin path and the bundled-only keyed-state guard, so this is not stale, implemented, or a duplicate.

## What This Changes

The issue reports that Slack thread participation is not durably persisted when `@openclaw/slack` is loaded from a global/npm install, causing later unmentioned Slack thread replies to be skipped.

## Best Possible Solution

Align the runtime-state contract for official externalized channel plugins, then add regression coverage for global-origin Slack thread participation persistence and improve the warning so the thrown reason is visible in logs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the runtime-state failure: a global Slack plugin calls `runtime.state.openKeyedStore`, and the registry proxy throws for every non-bundled origin. I did not execute a live Slack workspace scenario in this read-only review.

## Solution Assessment

Unclear. The narrowest code change may be to allow trusted official externalized plugins to use keyed state, but the public SDK currently documents this API as bundled-only, so maintainers should choose the contract before a repair PR changes the boundary.
