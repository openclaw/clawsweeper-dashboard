## Label Justifications

- `P1`: The report describes a broken Slack channel workflow where expected follow-up thread replies are skipped for official global plugin installs.
- `impact:message-loss`: Unmentioned Slack thread replies can be suppressed as no-mention even after the bot already participated in the thread.
- `impact:session-state`: The failure is in persisted Slack thread participation state used to preserve thread activation across cache/process boundaries.

## Summary

Keep open: current main and v2026.5.18 still restrict `runtime.state.openKeyedStore()` to bundled plugins while Slack uses that API for persistent thread participation, and https://github.com/openclaw/openclaw/pull/83775 is the open fixing PR.

## What This Changes

Fix Slack thread participation persistence for official Slack plugins loaded as global/npm installs so follow-up thread replies can remain eligible after prior bot participation.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83775 after proof and contract-doc alignment, then close this issue as implemented.

## Reproduction Assessment

Yes from source inspection: a non-bundled global Slack plugin reaches the bundled-only `openKeyedStore()` gate, the Slack cache catches the error, and the implicit thread mention path then lacks persisted participation. I did not run a live Slack repro in this read-only review.

## Solution Assessment

Yes, the linked PR's direction is the narrowest maintainable fix because it reuses the existing `trustedOfficialInstall` contract while keeping untrusted global/workspace plugins blocked.
