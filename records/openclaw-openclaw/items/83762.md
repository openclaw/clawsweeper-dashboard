- `impact:message-loss`: The reported user-visible failure is skipped Slack thread replies without a fresh explicit mention.
- `impact:session-state`: The root failure is loss of persisted Slack thread participation state after falling back to memory-only tracking.

## Summary

Keep open: current main and the latest release still reject keyed plugin state for non-bundled plugins, while Slack persists thread participation through that API. The linked fix PR is open and should be reviewed or landed before this issue closes.

## What This Changes

The issue reports that globally installed official Slack cannot persist thread participation, causing later unmentioned Slack thread replies to be skipped after the in-memory cache is lost.

## Best Possible Solution

Land a focused trusted-official keyed-state fix, keep untrusted global and workspace plugins blocked, update any public SDK wording that describes the access rule, and let the linked PR close this issue after merge.

## Reproduction Assessment

Yes, from source inspection: a globally loaded Slack plugin calls `runtime.state.openKeyedStore`, and current main rejects non-bundled plugin origins before Slack falls back to memory-only participation state. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Mostly yes: using the existing `trustedOfficialInstall` contract is the narrow maintainable fix for official external channel plugins, provided the PR preserves untrusted-plugin denial and aligns docs/proof before merge.
