## Summary

Keep open. Current main still matches the reported blocking order in both failover paths: non-timeout profile failure marking is awaited before auth-profile rotation, and that marking path writes through the locked auth-profile store. The same-author PR #57283 explicitly closes this issue and remains open/unmerged, so the issue should stay open while that implementation path is reviewed or replaced.

## What This Changes

The issue requests reordering auth-profile failover so rate-limit cooldown marking cannot block rotation to the next profile under auth-store file-lock contention.

## Best Possible Solution

Land PR #57283 or an equivalent focused fix on current main that rotates auth profiles before any awaited cooldown write in both prompt and assistant failover paths, keeps timeout semantics unchanged, records the failed profile's cooldown asynchronously or with a bounded wait, and adds targeted tests that prove a delayed failure-marking promise cannot block rate-limit rotation.

## Reproduction Assessment

Yes, for a source-level reproduction: current main has an awaited non-timeout failure-marking call before advanceAuthProfile, and that call reaches the locked auth-store writer. A live contention reproduction would strengthen timing proof, but it is not required to see the blocking path.

## Solution Assessment

Yes in direction: rotating before cooldown marking is the narrowest maintainable fix because the cooldown applies to the failed profile, not the candidate profile being advanced to. The current PR should be rebased to the extracted assistant-failover helper and should preserve warning guards and eventual cooldown logging.
