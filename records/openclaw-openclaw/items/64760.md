## Summary

Keep open. Current main has compact fallback, path compaction, count limits, and per-agent skill visibility, but it still injects prompt-facing skill metadata on every run and lacks lazy matching, a names-only catalog mode, or a resolver-backed Skill-tool flow that would satisfy the request safely.

## What This Changes

The issue requests reducing skill catalog prompt overhead by lazy-loading or using names-only skill metadata instead of injecting full prompt-facing skill entries every turn.

## Best Possible Solution

Keep the existing compact fallback, then add a documented skill catalog mode or resolver-backed loading path that materially reduces large catalogs while preserving reliable skill selection and SKILL.md loading. The likely end state should update the skill prompt formatter, system prompt guidance, config/docs if a user-facing mode is added, and focused tests for full, compact, names-only or resolver behavior.

## Reproduction Assessment

Yes. The source and tests give a high-confidence reproduction path: buildWorkspaceSkillsPrompt emits descriptions and locations under budget, falls back to name-plus-location compact output only after the full block exceeds the limit, and has no lazy or names-only branch.

## Solution Assessment

Unclear. The requested optimization is valid, but names-only injection by itself is not the safest solution while the prompt still instructs the model to scan descriptions and read SKILL.md at a location; a resolver/Skill-tool flow or explicit documented prompt mode is a more maintainable direction.
