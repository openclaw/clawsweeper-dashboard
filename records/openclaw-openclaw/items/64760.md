## Summary

Keep open. Current main has useful mitigations for large skill catalogs, including compact fallback, path compaction, count limits, and per-agent visibility controls, but it still injects prompt-facing skill metadata every run and does not implement lazy matching, names-only routing, or a configurable names-only prompt mode.

## What This Changes

The issue requests reducing per-turn skill prompt overhead by lazy-loading skill descriptions or otherwise replacing the full installed-skill catalog with a smaller routing surface.

## Best Possible Solution

Keep the existing compact fallback, then add a documented skill catalog mode that reduces large catalogs without breaking skill loading. The safest end state is either a Skill-tool/name resolver that lets the prompt list only names, or a configurable compact/names-only mode with tests and docs that make the routing tradeoff explicit.

## Reproduction Assessment

Yes. The current behavior is reproducible from source by building `buildWorkspaceSkillsPrompt` with visible skill entries: under budget it emits descriptions and locations, and over budget it emits compact name-plus-location entries rather than a names-only or lazy-selected catalog.

## Solution Assessment

Unclear. The requested direction is valid, but names-only prompt injection alone is not the best fix while the system prompt still requires `<location>` for `read`; a resolver/tool-backed lazy path or explicit documented prompt mode would be safer.
