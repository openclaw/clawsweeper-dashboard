## Summary

Keep open. Current main still builds a model-facing skills catalog on every eligible run; compact fallback, path compaction, count limits, and per-agent skill allowlists reduce the cost but do not implement lazy matching, a names-only catalog mode, or a resolver-backed skill loading flow.

## What This Changes

The issue requests reducing skill prompt overhead by lazy-loading skill metadata or injecting a smaller skills catalog instead of full prompt-facing entries every turn.

## Best Possible Solution

Keep the existing compact fallback, then add a documented skill catalog mode or resolver-backed loading path that materially reduces large catalogs while preserving reliable skill selection and SKILL.md loading.

## Reproduction Assessment

Yes. Source and tests give a high-confidence reproduction path: buildWorkspaceSkillsPrompt emits descriptions and locations under budget, falls back to name-plus-location compact output only after the full block exceeds the limit, and has no lazy or names-only branch.

## Solution Assessment

Unclear. The requested optimization is valid, but names-only injection by itself is not the safest fix while the prompt contract still depends on descriptions and locations; a resolver-backed flow or explicit documented prompt mode needs maintainer direction.
