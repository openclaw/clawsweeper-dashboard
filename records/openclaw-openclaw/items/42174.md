## Summary

Keep this PR open. Current main still rejects the reported Windows drive-letter Docker bind during config and runtime validation, but the proposed patch touches sandbox host bind policy and still has correctness/security issues that should block merge until maintainer review.

## What This Changes

The PR updates sandbox host-path normalization, runtime bind validation, allowed-root filtering, and Docker config schema validation to accept Windows drive-letter bind sources.

## Best Possible Solution

Implement Windows bind support through one shared absolute-host-path helper and parser contract: accept only real drive-absolute forms like C:/path or C:\path, use that helper in config validation, runtime validation, and security audit, preserve blocked-path and allowed-root canonicalization, compare Windows paths with Windows case semantics, and add focused config/runtime/audit tests for accepted and rejected Windows bind cases.

## Reproduction Assessment

Yes. The PR body gives a concrete Windows 11 Docker Desktop config using D:/data/openclaw/src:/src:ro, and current main source inspection shows config validation splits that bind at the drive-letter colon while runtime validation rejects any non-/ source.

## Solution Assessment

No. The direction is valid, but this patch spreads ad hoc drive-letter regex checks across schema and runtime policy and still accepts drive-relative strings as absolute; the safer solution is a centralized Windows-aware absolute-path contract with security regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Require a slash after Windows drive letters:** `src/agents/sandbox/validate-sandbox-security.ts:98-99`
  - body: The new drive-letter escape hatch matches `C:` without requiring `C:/` or `C:\`, so a drive-relative source like `C:relative:/data:ro` is treated as absolute. That skips the non-absolute classification in config/runtime validation and can also hide the audit warning for a source path that is not safely canonicalizable.
  - confidence: 0.86
- **[P2] Compare Windows bind roots case-insensitively:** `src/agents/sandbox/host-paths.ts:31-33`
  - body: Only uppercasing the drive letter leaves the later allowed-root prefix check case-sensitive for the rest of the path. On Windows, `C:/Users/Alice/Project` and `c:/users/alice/project/src` can name the same tree, but this patch can still reject the bind as outside the allowed roots, preserving a false Windows error.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes sandbox bind-source policy and currently weakens non-absolute path classification for drive-relative Windows strings.

Concerns:

- **[medium] Drive-relative bind sources bypass non-absolute classification:** `src/agents/sandbox/validate-sandbox-security.ts:98`
  - body: The new `/^[A-Za-z]:/` acceptance treats `C:relative` as if it were an absolute Windows path. Windows drive-relative paths are not absolute, so config validation and `getBlockedBindReason` should continue to reject or audit them instead of allowing them into the bind policy path.
  - confidence: 0.86
