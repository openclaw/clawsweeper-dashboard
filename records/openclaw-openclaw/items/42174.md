## Summary

Keep open. The Windows bind rejection is still real on current main, and the PR is directionally useful, but the patch changes sandbox host-bind policy and still leaves Windows path comparisons on POSIX-style case-sensitive rules. That needs maintainer/security review, especially alongside the open adjacent Windows bind-security work in #63074.

## What This Changes

The PR updates sandbox Docker bind parsing, host-path normalization, runtime bind validation, allowed-root filtering, and Docker config schema validation to accept Windows drive-letter bind sources.

## Best Possible Solution

Implement Windows bind support through one shared host-path contract: parse binds with the existing bind-spec helper, accept only drive-absolute forms such as C:/path or C:\path, preserve the original path for Docker, compare blocked paths and allowed roots with Windows-aware case semantics, keep canonical existing-ancestor checks, and add config/runtime/audit tests plus a changelog entry.

## Reproduction Assessment

Yes. The PR body gives a concrete Windows 11 Docker Desktop config, and current main source inspection shows the config schema splits D:/ at the drive-letter colon while runtime validation rejects every non-'/' source.

## Solution Assessment

No. The requested direction is valid, but this patch spreads ad hoc drive-letter checks across schema/runtime policy and does not make blocked-path or allowed-root comparisons Windows-aware; a centralized helper is the safer maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Compare Windows policy paths case-insensitively:** `src/agents/sandbox/host-paths.ts:31-33`
  - body: The patch only uppercases the drive letter before feeding Windows paths into blocked-path and allowed-root checks that still use case-sensitive string comparisons. On Windows, `C:/Users/Alice/.ssh` and `c:/users/alice/.ssh` can name the same directory, so lower-cased bind sources can evade the blocked home-credential check when external binds are allowed, and equivalent in-workspace binds can be rejected. Keep the original path for Docker, but compare a Windows-aware policy key.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes sandbox bind-source policy and currently leaves Windows blocked-path enforcement vulnerable to path-casing mismatches.

Concerns:

- **[medium] Windows path case can bypass blocked credential checks:** `src/agents/sandbox/host-paths.ts:31`
  - body: Blocked host paths are compared as case-sensitive strings, but Windows paths are normally case-insensitive. Because the patch normalizes only the drive letter, a lower-cased credential bind can miss the blocked home path and proceed when the external-bind override is enabled.
  - confidence: 0.78
