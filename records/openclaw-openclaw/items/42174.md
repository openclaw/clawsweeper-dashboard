## Summary

Keep open: current main still rejects the reported Windows drive-letter Docker bind at config validation and runtime validation, and this PR is not merge-ready because it admits Windows paths without making sandbox policy comparisons Windows-aware. The related #63074 work is useful security overlap, but it is open and does not fully supersede generic Windows bind support.

## What This Changes

The PR updates sandbox Docker bind parsing and validation so Windows drive-letter host sources such as `D:/data:/src:ro` can pass config and runtime checks.

## Best Possible Solution

Centralize a Windows-aware sandbox host-path policy that parses binds once, accepts only drive-absolute sources, compares policy keys with Windows case semantics, preserves Docker-facing paths, and covers config validation, runtime validation, audit, and Docker create-args with tests and a changelog entry.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: the PR body gives Windows 11 Docker Desktop steps, and current main still splits `D:/...` to `D` in config validation while runtime validation rejects non-`/` sources. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No. The requested direction is valid, but this patch is not the best fix because it spreads Windows checks across schema/runtime code and leaves blocked-path and allowed-root comparisons case-sensitive; a shared Windows-aware policy helper is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Compare Windows sandbox policy paths case-insensitively:** `src/agents/sandbox/host-paths.ts:33`
  - body: After this PR admits drive-letter bind sources, `normalizeSandboxHostPath` only uppercases the drive letter while blocked credential roots and allowed source roots are still checked with case-sensitive string prefixes. On Windows, a casing variant such as `c:/users/alice/.ssh` can miss a HOME-derived `C:/Users/Alice/.ssh` deny entry, so the policy key needs Windows path semantics while preserving the original Docker path.
  - confidence: 0.78
