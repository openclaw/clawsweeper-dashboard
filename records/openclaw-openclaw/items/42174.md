## Summary

Keep open: current main still rejects Windows drive-letter sandbox binds, so the PR addresses a real Windows Docker-bind gap, but the patch should not merge until the sandbox path policy uses Windows-aware canonicalization and comparisons and is reconciled with #63074.

## What This Changes

The PR changes sandbox Docker bind validation to accept Windows drive-letter host paths in config validation, runtime sandbox checks, and host-path normalization.

## Best Possible Solution

Centralize a Windows-aware sandbox bind policy that parses bind specs once, admits only drive-absolute sources under the intended policy, canonicalizes and compares with Windows semantics while preserving Docker-facing paths, then covers config validation, runtime validation, audit, Docker args, tests, and changelog.

## Reproduction Assessment

Yes, source-level. Current main reduces D:/... bind specs to D in config validation and rejects non-/ sources at runtime, matching the PR body's Windows startup failure without needing a live Windows gateway run.

## Solution Assessment

No. The direction is useful, but this patch admits drive-letter sources without making blocked-root and allowed-root policy comparisons Windows-aware; a shared policy helper is safer than scattered regex branches.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Compare Windows sandbox paths with Windows semantics:** `src/agents/sandbox/validate-sandbox-security.ts:99`
  - body: After this PR accepts drive-letter bind sources, the later blocked-root and allowed-root checks still use case-sensitive POSIX string prefixes. On Windows, casing variants can refer to the same directory, so a bind such as `c:/Users/Alice/.ssh:/mnt:ro` can miss a `C:/Users/Alice/.ssh` deny entry unless the policy uses a Windows-aware comparison key before admitting drive paths.
  - confidence: 0.82
