## Summary

Keep open: the reported Windows bind-mount problem is still present on current main, but this PR changes sandbox host-path admission without a complete Windows-aware security policy and overlaps the open Windows bind-security work in #63074.

## What This Changes

The PR updates sandbox Docker bind parsing and validation so Windows drive-letter host paths can be accepted for custom bind mounts.

## Best Possible Solution

Define one shared Windows-aware sandbox bind policy that parses bind specs once, admits only approved drive-absolute sources, applies Windows path comparison semantics for deny and allow roots, preserves Docker-facing paths, updates config/runtime/audit coverage, and reconciles with #63074.

## Reproduction Assessment

Yes, source-level. Current main still reduces `D:/...:/src:ro` to source `D` in config validation and rejects non-`/` runtime sources; I did not run the Windows gateway live.

## Solution Assessment

No. The patch points at the right user problem, but the maintainable solution needs a centralized Windows-aware sandbox path policy rather than scattered drive-letter exceptions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Compare Windows bind policy paths case-insensitively:** `src/agents/sandbox/host-paths.ts:34`
  - body: After this PR admits drive-letter bind sources, normalization only uppercases the drive letter while blocked credential roots and allowed roots are still checked with case-sensitive prefix logic. On Windows, casing variants can refer to the same directory, so paths like `c:/users/alice/.ssh` can miss a `C:/Users/Alice/.ssh` deny entry unless policy comparisons use a Windows-aware key while preserving the Docker-facing path.
  - confidence: 0.78
