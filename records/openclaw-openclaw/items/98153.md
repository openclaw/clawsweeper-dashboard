## Label Justifications

- `P1`: Latest-release iOS Talk can ignore configured Gateway speech providers and start an unconfigured realtime path, breaking a real mobile voice workflow for non-ElevenLabs providers.
- `impact:auth-provider`: The failure is provider-routing specific: iOS chooses realtime/OpenAI handling instead of the configured Gateway Talk speech provider and its credentials/config.

## Summary

Keep open: current main and the latest release still match the report. iOS Talk classifies any Gateway speech provider other than ElevenLabs as realtime, then native playback only tries app-local ElevenLabs or iOS system speech instead of the configured Gateway `talk.speak` provider.

## What This Changes

The issue asks iOS Talk Gateway Default to treat configured Gateway speech/TTS providers as native `stt-tts` providers and use Gateway `talk.speak` before local/system fallback.

## Best Possible Solution

Update iOS Talk to derive realtime from the realtime config/catalog mode, keep speech providers on the native STT -> Gateway chat -> `talk.speak` path, and add regression tests for non-ElevenLabs Gateway Default providers.

## Reproduction Assessment

Yes by source inspection and reporter evidence: current iOS still turns non-ElevenLabs `talk.provider` values into realtime mode, then lacks a `talk.speak` playback path. I did not run a live iPhone repro in this read-only review.

## Solution Assessment

Yes: fixing the iOS routing/playback boundary is the narrowest maintainable solution because docs, Gateway RPCs, Android, and macOS already define the `talk.speak` path. The fix should not add a new config option or change Gateway protocol unless the existing RPC proves insufficient.
