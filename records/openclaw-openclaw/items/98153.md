## Label Justifications

- `P1`: The issue breaks a real iOS Talk voice workflow for users with configured Gateway speech providers by routing them into an unconfigured realtime path and system voice fallback.
- `impact:auth-provider`: The failure is provider-routing specific: iOS chooses realtime/OpenAI handling instead of the configured Gateway Talk speech provider and its credentials/config.

## Summary

Keep open: current main and v2026.6.11 still match the reported iOS Talk provider-routing bug, and two open PRs now candidate-fix this issue but have not merged.

## What This Changes

The issue asks iOS Talk Gateway Default to keep speech/TTS providers on the native path and use Gateway `talk.speak` before falling back to local or system voices.

## Best Possible Solution

Land one reviewed iOS fix that derives realtime from parsed realtime mode/catalog, adds Gateway `talk.speak` playback for speech providers, preserves explicit OpenAI Realtime behavior, and includes iOS regression proof.

## Reproduction Assessment

Yes, by source-level reproduction and reporter evidence. Current iOS still turns non-ElevenLabs Gateway speech providers into realtime mode and lacks a Gateway `talk.speak` playback branch, though no live iPhone run was performed here.

## Solution Assessment

Yes, the narrow iOS client fix is the best path: derive realtime from the realtime config/catalog and route speech providers to Gateway `talk.speak`. The issue should stay open until one candidate PR merges or maintainers choose a different repair.
