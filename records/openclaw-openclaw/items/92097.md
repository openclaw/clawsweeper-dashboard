## Summary

This issue is still necessary: the latest stable release and current main both retain the reported pluginless custom-provider authentication gap, and no merged fix or canonical open replacement was identified. The source config correctly retains a non-secret marker, but the shared auth resolver does not recover the already-materialized API key from the runtime snapshot unless a provider plugin first returns synthetic auth.

## What This Changes

Restore authentication for custom `openai-responses` providers whose generated `models.json` contains a `secretref-managed` API-key marker backed by a resolved non-env SecretRef.

## Best Possible Solution

Have the shared provider-auth boundary recover the materialized custom-provider key from the canonical runtime snapshot when source config contains the managed marker, while preserving unresolved-marker rejection, credential precedence, and SecretRef-safe persistence.

## Reproduction Assessment

Yes at source level: a pluginless custom provider with a managed API-key marker cannot reach the materialized runtime snapshot and ultimately has no usable auth. This read-only review did not run the reporter's live upstream request.

## Solution Assessment

Yes. The narrowest maintainable solution is in the shared provider-auth resolver, not Codex routing or a provider-specific plugin, because the secrets runtime has already materialized the canonical credential.
