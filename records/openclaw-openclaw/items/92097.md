## Summary

Current main still contains the v2026.6.5 regression: custom-provider synthetic auth returns before consulting the materialized runtime snapshot when direct config auth is absent, while Codex’s Responses proxy contract expects a real bearer API key. No merged or open canonical fix was found, so this focused release regression remains necessary.

## What This Changes

Restore file-backed SecretRef API-key resolution for custom OpenAI Responses providers after the v2026.6.5 auth-routing regression.

## Best Possible Solution

Update the shared provider-auth resolver to use the already materialized runtime snapshot when source config contains a managed SecretRef marker and higher-priority auth is absent, with regression coverage proving resolved markers work and unresolved markers remain blocked.

## Reproduction Assessment

Yes at source level with high confidence: current main retains the released early-return path that bypasses the runtime snapshot for the reported custom-provider state. A live secret-bearing provider request was not run during this read-only review.

## Solution Assessment

Yes. Repairing the existing shared auth boundary to consume the canonical materialized snapshot is narrower and safer than adding config, provider-specific routing, or a parallel credential mechanism.
