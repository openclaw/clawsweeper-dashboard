## Summary

Keep open: the Feishu problem is plausible, but this patch is not safe to merge as written because it removes event-unique reaction IDs under the current dedupe path and does not include real Feishu after-fix proof.

## What This Changes

The PR changes Feishu reaction synthetic event IDs, reply anchoring for reaction events, reaction verification timeout, missing chat type inference from chat IDs, and related tests.

## Best Possible Solution

Rework the PR to keep event-unique synthetic message IDs, use the existing reply_target_message_id seam for the original Feishu message anchor, preserve thread/root metadata before session resolution, and add focused tests plus real Feishu proof.

## Reproduction Assessment

Yes for the blocking review finding from source inspection: two same-message, same-action, same-emoji, same-sender reaction events produce the same synthetic message_id in the PR, and current main dedupes processed Feishu messages by that key for 24 hours. I did not run a live Feishu reproduction or local tests because this was a read-only review.

## Solution Assessment

No. The maintainable fix should preserve a unique synthetic processing ID and carry the original Feishu message ID through the existing reply-target fields instead of making the synthetic ID deterministic and parsing it late in bot.ts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep reaction synthetic IDs event-unique:** `extensions/feishu/src/monitor.account.ts:126`
  - body: The new synthetic ID is deterministic for messageId/action/emoji/sender. `handleFeishuMessage` dedupes on the resolved message key and the persistent Feishu dedupe TTL is 24 hours, so a legitimate repeat such as created -> deleted -> created for the same user and emoji reuses the same key and the later event is skipped. Keep an event-unique token in `message_id` and use a separate reply target for the original Feishu message.
  - confidence: 0.9
- **[P2] Normalize reaction reply targets before session routing:** `extensions/feishu/src/bot.ts:950-955`
  - body: This computes the original message ID only at the outbound reply calculation, but topic hydration and `resolveFeishuGroupSession()` already ran earlier using the synthetic `ctx.messageId`. For group_topic or group_topic_sender reactions, history can still be read and written under a synthetic topic key while the visible reply targets the original message. Put the original message into the synthetic event's reply/thread metadata or normalize before session resolution.
  - confidence: 0.78
