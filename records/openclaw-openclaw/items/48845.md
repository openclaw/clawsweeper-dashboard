## Summary

Keep open: the PR targets a plausible Feishu bug, but the current branch still has source-provable reaction dedupe and topic-routing regressions, and it lacks required real Feishu behavior proof.

## What This Changes

The branch changes Feishu reaction synthetic IDs, reply anchoring, fetched-message chatType parsing, and reaction tests.

## Best Possible Solution

Rework the branch to infer missing chatType only with validated Feishu contracts, preserve topic_group, keep synthetic reaction IDs event-unique, carry the original message through reply/thread metadata before routing, and add focused tests plus real Feishu proof.

## Reproduction Assessment

Yes for the blocking review findings from source inspection: the PR's deterministic reaction ID collides with current main's 24-hour message-key dedupe, and the topic routing path uses ctx.messageId before the PR unwraps it for outbound replies. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No. The maintainable fix should keep a per-event synthetic processing ID, put the original Feishu message ID into existing reply/thread metadata before session routing, and preserve current chat-type contracts instead of overriding them with late string parsing.

  - body: The new synthetic ID is deterministic for messageId, action, emoji, and sender.   - confidence: 0.9

  - body: This unwraps the original message only when choosing the outbound reply target, but group/topic session routing has already used the synthetic ctx.messageId. In group_topic or group_topic_sender mode, history and bindings can be written under the synthetic reaction key while the visible reply targets the original message; put the real anchor into reply/thread metadata before routing.
  - confidence: 0.84
- **[P2] Preserve topic_group when inferring chat types:** `extensions/feishu/src/send.ts:265-271`
  - body: Current main accepts topic_group from fetched Feishu messages, but this branch's parser only treats group/private/p2p as explicit chat types before falling back to chat_id prefixes. A fetched topic_group item with an oc_ chat_id is downgraded to group, breaking topic-specific routing; keep topic_group as an explicit valid value before prefix inference.
  - confidence: 0.86
