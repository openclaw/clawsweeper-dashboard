## Summary

Keep open. The PR addresses a real Feishu plugin gap because current main still drops reaction lookups when Feishu omits `chat_type`, but the branch should not merge as-is: its reaction IDs still collide for repeated same-user lifecycle actions, topic routing metadata is not preserved before session resolution, and the branch is stale against current main's Feishu topic parsing/test layout.

## What This Changes

The PR changes the Feishu plugin to infer missing chat types from Feishu IDs, alter reaction synthetic event IDs and reply targeting, raise reaction verification timeout, and update reaction tests.

## Best Possible Solution

Refresh the Feishu plugin fix so missing `chat_type` is inferred only through well-tested Feishu-owned rules, reaction synthetic IDs remain event-unique while carrying the original message anchor separately, and topic root/thread metadata is available before session routing.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: repeated same-action reactions for the same original message, emoji, and sender reuse the PR's deterministic `message_id`, and current Feishu dedupe drops repeats by `event.message.message_id`; topic-scoped reactions also lack root/thread metadata before routing is resolved.

## Solution Assessment

No. The chat-type fallback is in the right owner module, but this is not the best mergeable shape because it mixes dedupe identity with reply anchoring and does not preserve current topic-session behavior on a refreshed main branch.

  - confidence: 0.8
