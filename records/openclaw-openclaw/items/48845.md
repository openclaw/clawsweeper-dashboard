## Summary

Keep this PR open. The reported Feishu behavior is still not implemented on the checked-out main, but the current PR patch is stale against main and still has blocking functional issues around reaction dedupe and topic-session routing.

## What This Changes

The PR modifies the Feishu plugin's message-fetch parsing, reaction synthetic event construction, reply-target selection, and reaction tests to infer missing chat types and adjust reaction handling.

## Best Possible Solution

Keep the Feishu fix in the Feishu plugin, but refresh it against current main with an event-unique synthetic reaction ID, a separate original-message anchor for replies, preserved root/thread metadata for topic sessions, and focused regression coverage for chat-type inference and repeated reaction lifecycle events.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence path: with the PR's deterministic ID, two repeated `reaction.created` events for the same message, emoji, and sender reuse the same dedupe key, and current `handleFeishuMessage` drops repeats by `event.message.message_id`.

## Solution Assessment

No. Inferring missing Feishu chat type is the right narrow direction, but the proposed reaction fix is not the best mergeable shape because it conflates dedupe identity, reply anchoring, and topic-session metadata, and the patch is stale against current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep reaction IDs unique across repeated actions:** `extensions/feishu/src/monitor.account.ts:129`
  - body: The new synthetic `message_id` is deterministic for `messageId + action + emoji + sender`. A user who removes and then re-adds the same emoji reuses the same `created` ID within the Feishu dedupe TTL, so `handleFeishuMessage` treats the later event as a duplicate and drops it. Include an event-unique component while carrying the original message ID separately for replies.
  - confidence: 0.88
- **[P2] Carry topic metadata when unwrapping reaction replies:** `extensions/feishu/src/bot.ts:950-954`
  - body: This unwraps the reaction ID only for the outbound reply target, but group routing/history were already resolved earlier from the synthetic `ctx.messageId` and the synthetic event still has no `root_id` or `thread_id`. In topic-scoped Feishu groups, the reply can anchor to the original message while OpenClaw reads and writes the wrong topic session. Preserve the original root/thread metadata before routing.
  - confidence: 0.78
