## Summary

Keep open. Current main still records generated `memory/.dreams/session-corpus/*.txt` entries into the short-term recall store, has no store-level retention prune, and only filters `maxAgeDays` at promotion-ranking time. The report is partly nuanced because session transcript corpus ingestion is documented and tested as intentional, so a blanket exclusion of all session-corpus signals would need maintainer design rather than an automatic close.

## What This Changes

The issue requests a fix for unbounded `memory/.dreams/short-term-recall.json` growth by preventing generated session-corpus artifacts from feeding back as recall candidates and by adding durable retention metadata/pruning.

## Best Possible Solution

Define a memory-core retention contract for `short-term-recall.json`: add stable creation metadata for new and migrated entries, prune expired or low-signal entries after ingestion/repair/status paths, and separate generated session-corpus artifacts from organic recall so transcript-derived signals do not self-amplify. Preserve intentional redacted session transcript dreaming unless maintainers explicitly decide to remove that product behavior.

## Reproduction Assessment

Yes. The existing dreaming phase test path constructs session transcript ingestion and verifies ranked candidates include `memory/.dreams/session-corpus/<date>.txt`; source inspection also shows the store write path has no retention prune.

## Solution Assessment

Unclear. The issue's retention direction is sound, but the least-invasive suggestion to exclude all session-corpus paths is probably not the best standalone fix because session transcript ingestion is documented and tested as part of dreaming.
