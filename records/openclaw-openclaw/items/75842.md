## Summary

Keep open. Current main still records `memory/.dreams/session-corpus/*` paths into the short-term recall store and has no store-level retention or `firstRecordedAt`; newer contamination repair only handles dreaming-prompt echoes, while docs and tests show transcript-corpus ingestion is intentional enough to need maintainer policy.

## What This Changes

The issue reports that memory-core dreaming records generated session-corpus artifacts into `short-term-recall.json` without retention metadata or pruning.

## Best Possible Solution

Define a memory-core retention contract that records stable creation metadata, prunes `short-term-recall.json` independently of promotion age, and separates generated transcript-corpus signals from organic recall without breaking intentional redacted transcript dreaming.

## Reproduction Assessment

Yes. Current main has a code and test path where light dreaming writes `memory/.dreams/session-corpus/<date>.txt`, records it through `recordShortTermRecalls`, and ranks that path as a candidate.

## Solution Assessment

Unclear. Adding retention metadata and pruning is directionally right, but excluding all session-corpus paths alone is not clearly the best fix because current docs and tests intentionally feed redacted transcript corpus into dreaming.
