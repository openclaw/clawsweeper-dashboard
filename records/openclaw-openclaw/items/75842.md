## Summary

Keep open. Current main still persists session-corpus hits into `short-term-recall.json` and lacks store-level retention or `firstRecordedAt`; because session transcript ingestion is documented and tested, the fix needs a maintainer contract decision rather than cleanup closure.

## What This Changes

The issue asks memory-core dreaming to stop letting generated session-corpus entries grow short-term recall unboundedly, preferably by excluding that source, adding retention, and recording creation metadata.

## Best Possible Solution

Define a bounded short-term recall contract for memory-core that preserves intentional transcript ingestion while separating generated/source classes, adding stable creation metadata, and pruning obsolete recall entries.

## Reproduction Assessment

Yes. Source inspection on current main shows session-corpus batches are appended, recorded into the short-term recall store, and still accepted by ranking/tests; no live run was needed to establish the code path.

## Solution Assessment

Unclear. Excluding all session-corpus paths alone conflicts with documented and tested transcript ingestion, so the safer direction is a maintainer-approved retention/source-class contract with stable creation metadata.
