## Summary

Keep open. Current main still writes session-corpus results into short-term recall and treats those paths as rankable, while the store write path has no source/age retention and no stable `firstRecordedAt`; related open issues cover adjacent pollution and scoring symptoms but not this precise retention/source-metadata boundary.

## What This Changes

The issue requests preventing dreaming session-corpus entries from unboundedly filling `memory/.dreams/short-term-recall.json`, with retention and creation-time metadata for pruning.

## Best Possible Solution

Define a memory-core source-class and retention contract that keeps generated session-corpus evidence bounded or internal-only, preserves stable creation metadata, and aligns repair, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main appends session-corpus lines, records them into `short-term-recall.json`, and has no store-level retention path. I did not run a multi-day live cron or replay the reporter's private workspaces.

## Solution Assessment

Unclear. Excluding all session-corpus entries is plausible for bloat prevention, but it changes documented and tested transcript-ingestion behavior; a maintained source-class plus retention contract is safer.
