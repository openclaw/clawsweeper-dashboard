## Summary

Keep open. Current main still records `memory/.dreams/session-corpus/*.txt` entries into the short-term recall store and lacks a store-level retention prune, while docs and tests show transcript corpus ingestion is intentional enough to need maintainer policy before changing behavior.

## What This Changes

The issue asks memory-core to prevent unbounded `short-term-recall.json` growth from generated session-corpus artifacts by adding source separation, retention, and stable creation metadata.

## Best Possible Solution

Define a memory-core retention contract for `short-term-recall.json` that stamps stable creation metadata, prunes expired or low-signal entries after ingestion/repair/status paths, and separates generated transcript-corpus signals from organic recall without removing intended redacted transcript dreaming.

## Reproduction Assessment

Yes. The current test path triggers light dreaming and verifies a `memory/.dreams/session-corpus/<date>.txt` candidate is ranked, and source inspection shows the same path is persisted without a retention prune.

## Solution Assessment

Unclear. The retention and creation-metadata direction is sound, but excluding all session-corpus paths alone is not the best standalone fix because session transcript ingestion is a documented and tested dreaming input.
