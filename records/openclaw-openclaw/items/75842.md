## Summary

Keep open. Current main still treats `memory/.dreams/session-corpus/*` as short-term recall input and persists those entries without store-level retention or creation-time metadata; the fix needs maintainer judgment because session transcript ingestion is documented and covered by tests.

## What This Changes

The issue requests bounding memory-core short-term recall growth by preventing generated session-corpus artifacts from becoming unbounded recall candidates, adding retention metadata, or both.

## Best Possible Solution

Define a memory-core short-term recall retention/source-class contract that preserves intentional transcript-derived signals, prevents generated artifacts from feeding themselves indefinitely, and adds stable creation metadata for pruning.

## Reproduction Assessment

Yes. Source inspection on current main shows session-corpus batches are appended, accepted as short-term memory paths, and written into `short-term-recall.json` without store-level pruning.

## Solution Assessment

Unclear. Retention plus stable creation metadata looks maintainable, but excluding all session-corpus paths alone conflicts with the documented and tested transcript-ingestion feature.
