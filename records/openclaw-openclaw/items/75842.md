## Summary

Keep open. Current main still treats generated session-corpus files as valid short-term recall sources and writes them into the recall store, while existing docs and tests show transcript-derived corpus ingestion is intentional enough that a blanket exclusion needs maintainer policy before automation changes promotion behavior.

## What This Changes

The issue requests a memory-core dreaming fix so `memory/.dreams/short-term-recall.json` does not grow unbounded from generated `memory/.dreams/session-corpus/*` recall candidates.

## Best Possible Solution

Define a memory-core retention contract for the short-term recall store, preserve or replace intentional redacted transcript ingestion deliberately, and separate generated corpus artifacts from durable promotion eligibility so dreaming cannot amplify its own intermediate outputs over time.

## Reproduction Assessment

Yes. A high-confidence reproduction exists in the current test path: session transcript ingestion creates `memory/.dreams/session-corpus/<date>.txt`, and the ranked candidate assertion includes that generated path.

## Solution Assessment

Unclear. The report's retention and source-separation direction is sound, but the simplest proposed fix of excluding all session-corpus paths is not clearly the best standalone solution because current docs and tests intentionally use redacted session transcript corpus as dreaming input.
