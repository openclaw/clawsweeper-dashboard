## Summary

Keep open. Current main still persists generated `memory/.dreams/session-corpus/*.txt` entries into `short-term-recall.json`, and the newer artifact repair path only handles suspicious narrative self-ingestion rather than adding retention, creation metadata, or a source-class boundary for ordinary session corpus entries.

## What This Changes

The issue asks memory-core dreaming to stop unbounded `short-term-recall.json` growth caused by generated session-corpus artifacts being recorded as recall candidates.

## Best Possible Solution

Define a memory-core source-class and retention contract that bounds `short-term-recall.json`, adds stable creation metadata, and decides how redacted session-corpus signals should contribute without self-amplifying.

## Reproduction Assessment

Yes. Source inspection on current main shows session-corpus batches are appended, accepted as short-term memory paths, and recorded into the recall store; tests and docs still exercise that eligibility.

## Solution Assessment

Unclear. Retention plus creation metadata is the maintainable direction, but excluding all session-corpus paths alone would conflict with the documented and tested transcript-ingestion feature.
