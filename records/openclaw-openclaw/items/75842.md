## Summary

Keep open. Current main still accepts `memory/.dreams/session-corpus/*` as short-term recall input and records appended session-corpus batches without store-level retention or `firstRecordedAt`; the newer repair path only archives detected contaminated artifacts and does not define bounded recall-store behavior.

## What This Changes

Prevent memory-core dreaming from growing `memory/.dreams/short-term-recall.json` with session-corpus-derived entries by adding source filtering, retention, and stable creation metadata.

## Best Possible Solution

Define a memory-core short-term recall source and retention contract that keeps intentional redacted transcript signals bounded, excludes generated dreaming artifacts where appropriate, and persists stable creation metadata for pruning.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: session-corpus batches are appended and then recorded into `short-term-recall.json`, while the write path has no retention pass. I did not mutate a live workspace to reproduce the file growth.

## Solution Assessment

Unclear. Retention plus stable source/creation metadata looks like the maintainable direction, but excluding all `session-corpus` paths alone conflicts with documented and tested transcript-ingestion behavior.
