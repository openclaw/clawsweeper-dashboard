## Summary

Keep open. Current main still records `memory/.dreams/session-corpus/*` paths into the short-term recall store and lacks a store-retention pruning path; existing self-ingestion guards reduce one feedback loop, but they do not address the unbounded store-growth concern reported here.

## What This Changes

The issue requests a memory-core dreaming fix so `short-term-recall.json` stops growing without bound from session-corpus recall candidates and has reliable creation/retention metadata.

## Best Possible Solution

Keep the issue open and resolve it in memory-core by preserving legitimate session transcript ingestion while adding an explicit retention contract for the short-term store, stable creation metadata for new and migrated entries, and tests/docs that distinguish transcript-derived signals from generated dreaming artifacts.

## Reproduction Assessment

Yes. A code-level reproduction is available from current main: `recordShortTermRecalls` accepts `memory/.dreams/session-corpus/*.txt` paths, and existing tests already assert those paths are rankable short-term candidates; the long-running growth claim is also consistent with the absence of any age-based pruning path.

## Solution Assessment

Unclear. Excluding every session-corpus entry is not clearly the best fix because session transcript ingestion is documented and tested as a dreaming source; a safer maintainable direction is to add retention and creation metadata first, then narrow any exclusion to generated dreaming artifacts only.
