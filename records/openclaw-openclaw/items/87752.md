## Summary

Keep this open: current main still has a source path where MiniMax fast mode rewrites `MiniMax-M2.7` to `MiniMax-M2.7-highspeed`, and existing tests/docs still expect that behavior rather than guarding it against an explicit provider model list or subscription availability.

## What This Changes

The issue asks OpenClaw to stop failover from calling undeclared `minimax-portal/MiniMax-M2.7-highspeed` and to report a clear no-eligible-fallback failure instead.

## Best Possible Solution

Keep the issue open and repair or clarify MiniMax fast-mode eligibility so explicit provider model configuration and the configured fallback chain are honored before highspeed is called.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the MiniMax fast-mode wrapper can rewrite M2.7 to highspeed and current tests expect that rewrite, which matches the reported unconfigured-model appearance when fast mode is active.

## Solution Assessment

Unclear. A maintainable fix should target the MiniMax fast-mode/config eligibility boundary, but subscription-plan validation and documented `/fast` semantics need maintainer direction before automation changes behavior.
