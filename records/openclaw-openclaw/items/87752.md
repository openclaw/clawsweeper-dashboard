## Summary

Keep open: current main and the latest release still document and implement MiniMax fast-mode rewriting from `MiniMax-M2.7` to `MiniMax-M2.7-highspeed`, and the related commitments PR covers only one hardcoded call site rather than the broader highspeed eligibility problem.

## What This Changes

The issue asks OpenClaw to stop selecting or calling `MiniMax-M2.7-highspeed` when users configured only standard MiniMax models and their subscription lacks highspeed access.

## Best Possible Solution

Define and implement a MiniMax fast-alias eligibility rule so fast mode only uses highspeed when the model is explicitly configured or otherwise known available, while preserving intentional highspeed opt-ins.

## Reproduction Assessment

Yes, from source at medium confidence: current main still rewrites MiniMax M2.7 to highspeed under fast mode and tests/docs expect it. I did not run a live MiniMax Starter-plan account, so the subscription failure is validated from reporter logs and source behavior rather than live proof.

## Solution Assessment

No current implementation solves the whole issue. The open commitments PR is a partial call-site fix; the broader best fix needs maintainer direction on whether MiniMax fast aliases require explicit config or capability eligibility before routing.
