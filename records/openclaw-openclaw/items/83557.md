- `impact:message-loss`: The parent can receive an accepted-looking spawn result while no useful child completion is delivered.

## Summary

Keep open. Current main still lets the model-facing `sessions_spawn` path accept a native child run before the embedded child has actually started, so an OpenAI/Codex thinking rejection after that point can match the reported accepted-but-no-work failure shape; the open related PR only covers omitted/inherited thinking, not this explicit non-`off` OpenAI startup failure.

## What This Changes

The issue reports that ad-hoc `sessions_spawn` subagent calls on OpenAI/OpenAI-Codex models silently fast-fail when `thinking` is non-`off`.

## Best Possible Solution

Make native `sessions_spawn` either start a real child run with the requested supported OpenAI/Codex thinking level or surface a structured startup/unsupported-level error to the parent before the delegation is treated as completed.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the current path can acknowledge `sessions_spawn` before provider/thinking validation and child startup finish, matching the reported silent accepted-with-no-work failure shape.

## Solution Assessment

Unclear until the exact OpenAI rejection branch is reproduced. The best direction is a narrow runtime fix around `sessions_spawn`, `spawnSubagentDirect`, and gateway child startup so accepted spawns cannot hide immediate child-run failures.
