## Summary

Keep open. The narrowed report is not covered by the prior metadata/forwarding checks: current main forwards `sessions_spawn.thinking` into the subagent gateway run, but the gateway acknowledges the child run before the embedded agent actually starts, so a provider/thinking rejection can still look like an accepted spawn with no useful child work.

## What This Changes

Report that ad-hoc subagent spawns on OpenAI GPT models fast-fail silently whenever runtime thinking is non-`off`, while preconfigured subagents still work.

## Best Possible Solution

Make ad-hoc native subagent spawn either start a real child run with the requested supported OpenAI/Codex thinking level or return a visible unsupported-level/startup error before the parent treats the spawn as accepted.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The issue provides concrete live steps, and source inspection shows the reported surface can acknowledge a spawn before the provider/thinking validation and embedded run actually start.

## Solution Assessment

Unclear until the exact OpenAI rejection branch is reproduced. The best direction is a narrow runtime fix around `sessions_spawn`/`spawnSubagentDirect`/gateway agent startup so accepted spawns cannot hide immediate child-run failures.
