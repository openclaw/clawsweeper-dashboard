## Summary

Keep open. Current main has useful mitigations and several related fixes, but source inspection still shows whole-store `sessions.json` parse/stringify paths and a transcript-count path that can build a parsed full-transcript index before serving bounded history; the open session-store bloat tracker and prompt-stripping PR only cover part of this issue.

## What This Changes

The issue asks for OpenClaw session metadata and transcript reads to become lazy, streamed, paginated, or otherwise bounded instead of loading full session files into memory.

## Best Possible Solution

Keep this open until OpenClaw has bounded or indexed session metadata plus transcript count/read paths, while using https://github.com/openclaw/openclaw/issues/55334 and https://github.com/openclaw/openclaw/pull/78259 for the narrower static-payload mitigation work.

## Reproduction Assessment

Yes, from source with high confidence: large `sessions.json` cache misses still parse the whole file, writes still stringify the whole store, and recent-history stats can still build a parsed transcript index on count-cache miss. I did not run a live heap/RSS benchmark in this read-only review.

## Solution Assessment

No, closing this as implemented or fully duplicated is not the best path. Current main and related open work are mitigations; the maintainable fix needs a coordinated persisted-store and bounded transcript-count/read design.
