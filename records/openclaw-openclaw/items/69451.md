## Summary

Keep open. Current main has useful mitigations, but source inspection still shows whole-file session metadata parsing/serialization and transcript-count paths that can materialize parsed transcript indexes before serving bounded history.

## What This Changes

The issue asks OpenClaw to stop loading large `sessions.json` and JSONL session histories wholesale and replace them with lazy, streamed, or bounded session storage and reads.

## Best Possible Solution

Introduce bounded or indexed session metadata and transcript-count/read paths so common gateway history and send APIs do not parse or retain full stores/transcripts, while landing payload-only reductions as narrower mitigations when they are safe.

## Reproduction Assessment

Yes, from source with high confidence: a large store cache miss or writer fallback still parses and serializes the full `sessions.json`, and recent-history stats can still build a parsed transcript index for counts. I did not run a live heap/RSS benchmark in this read-only review.

## Solution Assessment

No, closing as implemented or fully duplicated is not the best path. Current main and the related open PRs are mitigations; the maintainable fix needs a coordinated persisted-store and bounded transcript-count/read design.
