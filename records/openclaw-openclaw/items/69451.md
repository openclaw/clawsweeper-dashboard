## Summary

Keeping this open. Current main has useful mitigations around bounded reads, pruning, and lazy-ish session list behavior, but the core reported failure mode still exists at source level: the gateway session store still loads/parses the entire `sessions.json` on cache miss and writes the whole object back on mutation, while some transcript count/index paths still materialize more history than a bounded page needs.

## What This Changes

The issue requests reducing gateway memory growth from monolithic `sessions.json` parsing and whole-transcript JSONL reads by adding lazy loading, streaming or pagination, sharding, limits, and bounded caching.

## Best Possible Solution

Keep this as the gateway/session memory tracker until `sessions.json` is bounded, sharded, or indexed and transcript history/count APIs can satisfy bounded reads without full parsed transcript indexes, coordinated with the open session-store and SQLite transcript follow-ups.

## Reproduction Assessment

Yes, at source level. Current main still has whole-file `sessions.json` parse/stringify paths and a history stats path that can materialize a full parsed transcript index on cache miss, though I did not run a weeks-long RSS benchmark.

## Solution Assessment

No complete solution is on main yet. The right fix is broader than replacing one `readFileSync`: it should introduce bounded metadata/indexing and transcript read semantics while preserving session resume and upgrade compatibility.
