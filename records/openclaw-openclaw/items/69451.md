## Summary

Keep open. Current main has useful mitigations for bounded transcript tails, session-list lookup, and retention, but source inspection still shows whole-store `sessions.json` parse/stringify paths and a recent-history count path that can build a parsed full-transcript index before returning bounded history; the open store-bloat tracker and prompt-stripping PR only cover part of this issue.

## What This Changes

The issue asks OpenClaw to stop loading large `sessions.json` and JSONL session history files wholesale, using lazy loading, streaming or pagination, metadata sharding, and bounded caching to reduce gateway RSS.

## Best Possible Solution

Keep this open until OpenClaw has bounded or indexed session metadata plus transcript count/read paths, while coordinating the static-payload work through https://github.com/openclaw/openclaw/issues/55334 and https://github.com/openclaw/openclaw/pull/78259.

## Reproduction Assessment

Yes. Source inspection shows large `sessions.json` cache misses still parse the whole file, writes still stringify the whole store, and recent-history stats can still build a parsed transcript index on count-cache miss; no live RSS benchmark was run here.

## Solution Assessment

No, closing this as implemented or fully duplicated is not the best path. Current main and related open work are mitigations; the maintainable fix needs a coordinated persisted-store and bounded transcript-count/read design.
