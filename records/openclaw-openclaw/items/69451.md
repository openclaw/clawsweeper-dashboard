## Summary

Keep open. Current main has meaningful bounded-read and cache mitigations, but the monolithic metadata store and full transcript index/count paths still leave the reported memory-scaling problem unresolved.

## What This Changes

The issue asks OpenClaw to stop loading large session metadata and transcript files fully into memory and to provide bounded or streaming session-history access.

## Best Possible Solution

Keep this as the broad storage-performance tracker until session metadata uses per-entry or indexed storage and transcript history/count APIs have bounded, cursor-aware readers that do not cache full parsed transcripts for ordinary history views.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists with a large `sessions.json` cache miss or writer fallback plus a large transcript accessed through `sessions.get`, which still reads/parses the whole metadata store and can build a full transcript index for the message count.

## Solution Assessment

No. Current main's cache, pruning, list limiting, and bounded tail reads are useful mitigations, but the durable fix needs staged storage/index work coordinated with the directory-backed metadata effort rather than closing this as solved.
