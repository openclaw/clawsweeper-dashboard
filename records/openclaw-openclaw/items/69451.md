## Summary

Keep open. Current main has useful mitigations for session-store size, caching, and bounded tail reads, but `sessions.json` is still a whole-file metadata store and ordinary recent-history/count paths can still build a full parsed transcript index; the open https://github.com/openclaw/openclaw/issues/55334 covers the static-payload/store-bloat subset but does not fully replace this tracker.

## What This Changes

The issue requests bounded or lazy loading for session metadata and JSONL transcript history to prevent gateway memory growth from large session stores.

## Best Possible Solution

Keep this as the storage-memory tracker until session metadata uses bounded per-entry or indexed storage and recent-history/count APIs avoid full parsed transcript indexes, while linking https://github.com/openclaw/openclaw/issues/55334 for static payload dedupe work.

## Reproduction Assessment

Yes, from source with high confidence: a large `sessions.json` cache miss or writer fallback still reads/parses/stringifies the whole store, and first-page history with a large transcript still asks for a full count through the parsed transcript index. I did not run a live heap/RSS benchmark in this read-only review.

## Solution Assessment

No, closing this as implemented or fully duplicated would be premature. Current main is a useful mitigation, but the maintainable end state needs a coordinated storage schema and bounded transcript-count/read design rather than another narrow `readFileSync` replacement.
