## Summary

Keep open. Current main has meaningful mitigations, but the central memory behavior is still source-reproducible: `sessions.json` remains a whole-file metadata store and recent-history stats can still materialize a full transcript index before serving a bounded page.

## What This Changes

The issue reports gateway memory growth from whole-file `sessions.json` parsing plus full JSONL transcript reads, and asks for lazy or streamed storage with bounded caching.

## Best Possible Solution

Keep this as the broader storage-memory tracker until session metadata uses bounded or indexed storage and ordinary recent-history/count APIs avoid full transcript materialization; use https://github.com/openclaw/openclaw/issues/55334 for the static payload subset.

## Reproduction Assessment

Yes, from source with high confidence: a large store cache miss or writer fallback still parses and serializes the whole `sessions.json`, and recent-history stats can still build a full parsed transcript index for counts. I did not run a live heap/RSS benchmark in this read-only review.

## Solution Assessment

No, closing as implemented or fully duplicated is not the best path. Current main has useful mitigations, but the maintainable fix needs a coordinated persisted-store and bounded transcript-count/read design rather than another narrow `readFileSync` replacement.
