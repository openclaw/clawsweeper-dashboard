## Summary

Keep open. Current `main` has useful mitigations for session-store size, caching, and bounded transcript reads, but the metadata store still has whole-file read/write paths and ordinary history/count flows can still build a full parsed transcript index.

## What This Changes

The issue reports gateway RSS growth from wholesale `sessions.json` and JSONL transcript loading as session history accumulates.

## Best Possible Solution

Keep this as the storage-performance tracker until session metadata uses bounded per-entry or indexed storage and history/count APIs avoid caching full parsed transcripts for ordinary recent-history views.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction remains: use a large `sessions.json` cache miss or writer fallback plus a large transcript accessed through `sessions.get`, which still reads/parses the whole metadata store and can build a full parsed transcript index for the count.

## Solution Assessment

No. Current `main` is a useful mitigation, but closing this would leave the core whole-store metadata layout and unbounded transcript-count/index memory behavior unresolved.
