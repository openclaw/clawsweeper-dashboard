## Summary

Keep open. Current main has useful mitigations for session-store growth and session-list enrichment, but the core reported hot paths still materialize the full `sessions.json` store and full transcript snapshots for several session/history reads, so the production memory/RSS concern is not implemented away yet.

## What This Changes

The issue requests lazy, bounded, or streaming session storage and transcript access so large `sessions.json` and JSONL history files do not get fully loaded into memory.

## Best Possible Solution

Keep this issue as part of the session-storage performance cluster until the metadata store no longer requires whole-store hot-path reads/writes and transcript history APIs have bounded cursor-aware readers or precomputed summary/index fields for usage, title, preview, and history windows.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: use a large `sessions.json` plus a large transcript, then hit session update/load or `sessions.get`/HTTP history paths; current main still reads/parses or serializes the whole file before applying response limits.

## Solution Assessment

No. Current main's pruning, cache, bounded preview, and limit-before-enrichment mitigations are useful but partial; the maintainable fix is staged storage/index work rather than closing this as solved by the current code.
