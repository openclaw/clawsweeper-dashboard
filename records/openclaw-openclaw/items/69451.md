## Label Justifications

- `P1`: The report describes gateway RSS growing into multi-gigabyte territory from normal session history, which can break long-running agent/channel workflows on real machines.
- `impact:session-state`: The affected surface is persisted session metadata, session transcripts, transcript counts, and session-history reads.
- `impact:crash-loop`: The failure mode is process-level memory pressure and potential gateway unavailability after session history accumulates.

## Summary

Keep open. Current main has several mitigations for session-list hydration and cleanup, but the reported memory class is not fully fixed: `sessions.json` is still a monolithic full-file parse/stringify path and recent-history stats can still build a parsed full-transcript index on cache miss.

## What This Changes

The issue asks OpenClaw to stop loading large session metadata and transcript files wholesale, replacing the current behavior with lazy, streamed, paginated, or bounded session storage reads.

## Best Possible Solution

Keep this open until OpenClaw has a compatibility-safe bounded session metadata design and transcript count/read path, while coordinating shared static-payload work through https://github.com/openclaw/openclaw/issues/55334 and https://github.com/openclaw/openclaw/pull/78259.

## Reproduction Assessment

Yes, at source level. Current main still full-parses and full-serializes `sessions.json` on cold/cache-miss paths, and recent-history stats can still build a parsed transcript index; no live 831-file RSS benchmark was run here.

## Solution Assessment

No, closing this as implemented or fully duplicated is not the best path. Current main and related work are useful mitigations, but the maintainable fix needs a coordinated persisted-store and bounded transcript-count/read design.
