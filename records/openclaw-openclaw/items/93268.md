## Summary

Keep this PR open. The reported fast-path status bug is real, but the current fallback is too broad and can turn existing FTS-only indexes into a false `Vector store: ready` status.

## What This Changes

The PR changes the built-in memory manager status fast path to infer vector-store readiness from existing indexed chunks when sqlite-vec availability has not been probed in the current process.

## Best Possible Solution

Make the fast-path fallback infer readiness only from semantic or vector-backed persisted state, and preserve unknown or disabled status for FTS-only indexes.

## Reproduction Assessment

Yes from source inspection: current main leaves `storeAvailable` unknown when `this.vector.available` is still `null`, and the PR fallback would return true for any `chunks` row. I did not run the CLI in this read-only review.

## Solution Assessment

No. The fix targets the right field, but `hasIndexedChunks()` is broader than the reported vector-ready condition; a semantic/vector-backed persisted-state check is the safer narrow repair.

- **[P1] Narrow the fallback to semantic vector chunks:** `extensions/memory-core/src/memory/manager.ts:1152`
  - body: This fallback calls `hasIndexedChunks()`, which returns true for any row in `chunks`. Use a check that distinguishes semantic/vector-backed chunks, or keep the state unknown for FTS-only indexes.
