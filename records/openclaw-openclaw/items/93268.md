## Summary

Keep this PR open, but not merge-ready: the current patch fixes the reported fast-path symptom by looking for any indexed chunk, which can falsely report `Vector store: ready` for FTS-only indexes that have chunks but no semantic/vector-backed rows.

## What This Changes

The PR changes the built-in memory manager status fast path so `storeAvailable` falls back to `hasIndexedChunks()` when sqlite-vec availability has not been probed.

## Best Possible Solution

Make the fast-path fallback distinguish semantic or vector-backed persisted state from FTS-only chunks, then cover both the real ready case and the FTS-only non-ready case in regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main leaves `storeAvailable` unknown when `this.vector.available` is still `null`, and the PR fallback would return true for FTS-only `chunks` rows.

## Solution Assessment

No. The PR targets the right status field, but `hasIndexedChunks()` is broader than vector readiness; a semantic/vector-backed persisted-state check is the safer narrow repair.
