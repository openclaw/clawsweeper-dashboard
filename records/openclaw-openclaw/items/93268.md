## Summary

Keep open. The reported fast-path status bug is real, but this patch uses the generic `chunks` table as the readiness signal, which can over-report `Vector store: ready` for degraded indexes where sqlite-vec was unavailable and no `chunks_vec` rows were written.

## What This Changes

This PR changes memory-core status so builtin vector store availability falls back from the lazy sqlite-vec availability flag to `hasIndexedChunks()`.

## Best Possible Solution

Infer fast-path readiness only from durable evidence that vector rows were actually created, then add regression coverage for both ready and degraded/no-vector cases.

## Reproduction Assessment

Yes, from source inspection. Plain `openclaw memory status` skips vector probes, and the current indexing path can insert `chunks` while skipping `chunks_vec` when sqlite-vec is unavailable.

## Solution Assessment

No. The PR fixes the happy-path symptom, but `hasIndexedChunks()` is too broad for a sqlite-vec store readiness signal because generic chunks can exist without vector rows.
