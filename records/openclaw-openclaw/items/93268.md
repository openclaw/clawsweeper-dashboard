The PR changes memory-core status to infer `vector.storeAvailable` from persisted `vectorDims` metadata when current-process vector availability has not been probed, and adds regression coverage for status-only managers.

## Best Possible Solution

Land the `vectorDims` metadata fallback only after real CLI status proof shows a previously indexed vector DB reports ready on the plain fast path, then close or supersede the competing chunk-heuristic PR.

## Reproduction Assessment

Yes by source inspection: plain `memory status` skips vector probing, current main returns undefined `storeAvailable`, and the CLI renders that as `Vector store: unknown`. I did not run the full CLI reproduction in a real indexed setup during this read-only review.

## Solution Assessment

Yes for the code shape: persisted `vectorDims` is a narrower vector-store readiness signal than chunk-row or semantic-row heuristics. The remaining issue is proof and canonical-PR selection, not a known patch defect.
