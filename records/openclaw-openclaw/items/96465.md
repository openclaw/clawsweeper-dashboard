## Summary

Keep open: the PR addresses a real active-memory/QMD lifecycle bug with strong targeted proof, but it is not cleanup-closeable because the remaining question is the memory runtime/API contract. The compatibility blocker is that the existing close facade would silently become a narrower resource-release path when the new hook exists.

## What This Changes

The PR adds an optional memory runtime resource-release hook, makes active-memory cleanup prefer it, wires memory-core to release local index resources without closing shared QMD managers, and updates focused tests plus the Plugin SDK API baseline.

## Best Possible Solution

Preserve the existing full-close facade and add a clearly named active-memory resource-release path unless maintainers explicitly accept narrowing the close facade semantics.

## Reproduction Assessment

Yes. Current main source shows active-memory timeout cleanup calls closeActiveMemorySearchManager, and memory-core's scoped close deletes/closes the cached QMD manager before local index cleanup; I did not run a live gateway timeout reproduction in this read-only pass.

## Solution Assessment

No, not yet. Preserving the shared QMD manager is the right behavior, but the safest API shape is an explicit resource-release facade unless maintainers approve changing the existing close facade semantics.
