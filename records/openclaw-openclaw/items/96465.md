## Summary

Keep open for maintainer/API ownership review. The PR is a viable, well-proven bug-fix candidate for the active-memory plus QMD manager-lifecycle regression, but it intentionally changes plugin-facing memory lifecycle contracts, so the remaining decision is maintainer acceptance rather than cleanup closure or automated repair.

## What This Changes

The PR adds an opt-in `scope: "resources"` active-memory cleanup path and an optional memory runtime release hook so timeout cleanup can preserve memory-core's shared QMD manager while still releasing disposable recall resources.

## Best Possible Solution

Land this PR if maintainers accept the scoped memory lifecycle contract; otherwise request the same QMD-preserving behavior behind a separately named or narrower memory SDK facade.

## Reproduction Assessment

Yes, at source level: current main routes active-memory timeout cleanup into full manager close, and memory-core then closes the cached QMD manager later chat memory_search calls can reuse. I did not run a live gateway timeout reproduction in this read-only review.

## Solution Assessment

Conditionally yes: the latest PR preserves the default full-close contract and narrows only the active-memory timeout cleanup path, which is maintainable if maintainers accept the scoped lifecycle API. If they reject that API shape, the same behavior should move to a separately named release facade.
