## Summary

Keep open: the reported Linux systemd system-unit gap is real and not fully solved on current main, but this PR is not merge-ready because it conflicts with current main and introduces scope-selection and privilege-policy regressions.

## What This Changes

The PR changes Linux gateway systemd status/lifecycle and update restart helper behavior to detect `/etc/systemd/system` units and route commands to system-scoped systemctl.

## Best Possible Solution

Land a coordinated Linux systemd service-scope fix that derives scope from live enabled/active ownership or explicit service facts, keeps read-only probes unprivileged, handles mutating system-unit actions with explicit privilege or operator guidance, and preserves current update restart diagnostics.

## Reproduction Assessment

Yes, at source level: current main still routes daemon lifecycle/status through user-scoped systemctl while the gateway docs support system units under `/etc/systemd/system`. I did not run a live WSL/systemd reproduction in this read-only review.

## Solution Assessment

No, not as currently implemented. The direction is valid, but file-existence scope selection, blanket sudo use, and stale update-helper changes are not the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Choose service scope from live ownership:** `src/daemon/systemd.ts:86-90`
  - body: `resolveSystemdServiceContext` switches to system scope solely because the `/etc` unit file exists. If a user-scoped gateway is active while an old or disabled system unit remains, status/restart/stop/uninstall target the wrong service; base the choice on active/enabled ownership or explicit service facts instead.
  - confidence: 0.9
- **[P2] Reserve sudo for mutating system-unit actions:** `src/daemon/systemd.ts:307-309`
  - body: `execSystemctlSystem` wraps every non-root system-scope call in `sudo -n`, including read-only `is-enabled` and `show` probes used by status. Those normally work unprivileged, so status can fail on healthy system-unit installs that do not grant passwordless sudo.
  - confidence: 0.91
- **[P2] Surface system-unit unlink permission failures:** `src/daemon/systemd.ts:592-598`
  - body: Once `context.unitPath` can point under `/etc/systemd/system`, `fs.unlink` runs without sudo and catches EACCES/EPERM as `not found`. A non-root uninstall can leave the unit installed while printing a misleading message.
  - confidence: 0.9
- **[P2] Preserve current restart handoff diagnostics:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main logs Linux update restart attempts and prints explicit sudo guidance when a system-scoped unit is detected. This branch emits automatic `sudo -n systemctl restart` instead, which can fail in root/no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
