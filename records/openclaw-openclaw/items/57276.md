## Summary

Keep this PR open: the reported bug is still valid for daemon status/control on current main, but the branch is not safe to merge as written because it mixes a useful systemd-scope fix with privileged-control behavior and update-helper changes that conflict with current main’s safer handoff path.

## What This Changes

The PR adds Linux systemd system-unit detection to daemon status/control and update restart-helper generation, with tests for system-scoped units.

## Best Possible Solution

Rework this as a focused daemon systemd-scope fix that detects system units for status/runtime, avoids sudo for read-only probes, uses an explicit maintainer-approved policy for mutating commands, preserves current update-helper guidance, and adds user/system-scope regression tests.

## Reproduction Assessment

Yes, at source level. Current docs allow `/etc/systemd/system` gateway units, while current main still reads and controls the canonical Linux gateway service through user-unit paths and `systemctl --user`.

## Solution Assessment

No, not as currently implemented. The PR targets the right daemon gap, but the safer path is to keep read-only systemd probes unprivileged, preserve current update-helper handoff behavior, and make system-unit mutation policy explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` is used for `is-enabled` and `show` as well as restart/stop, so non-root `gateway status` would require passwordless sudo even though plain system `systemctl show` and `is-enabled` are normally readable. Keep sudo for explicit mutations or try direct reads first.
  - confidence: 0.91
- **[P2] Handle system-unit removal permissions explicitly:** `src/daemon/systemd.ts:586-590`
  - body: Once `context.unitPath` can point under `/etc/systemd/system`, this `fs.unlink` runs as the current user and the broad catch reports EACCES/EPERM as "not found". Non-root system-unit uninstalls can leave the unit file installed while printing misleading output.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main’s Linux restart helper logs restart attempts and emits explicit sudo guidance when it detects a system-scoped unit. This branch replaces that with automatic `sudo -n systemctl restart`, which can fail in root/no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:86`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong systemctl arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
