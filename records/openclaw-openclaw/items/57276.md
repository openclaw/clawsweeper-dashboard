## Summary

Keep open. Current main only partially supersedes the update-helper portion via https://github.com/openclaw/openclaw/pull/74920; daemon status and control paths still use the user-scoped systemd unit, so the branch remains relevant but is not mergeable as-is.

## What This Changes

The PR adds `/etc/systemd/system` detection for Linux gateway systemd units, routes daemon systemd reads/control through a system/user context, changes the update restart helper to use `sudo -n systemctl`, and adds focused tests.

## Best Possible Solution

Rebase or replace the branch on current main with a system-unit context that reads status unprivileged, uses explicit/root-aware privilege only for mutations, preserves current update guidance/logging, and keeps user-unit behavior unchanged.

## Reproduction Assessment

Yes, at source level. Current main resolves managed systemd unit files under the user service directory and routes enabled/runtime/control calls through `systemctl --user`, while docs support `/etc/systemd/system` system units.

## Solution Assessment

No, not as currently implemented. The PR targets the right daemon surface, but blanket sudo routing, misleading system-unit unlink handling, current-main update-helper drift, and stale test mocks need correction first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlForContext` sends system-scope `is-enabled` and `show` through `sudo -n`, so `openclaw gateway status` can fail for normal operators without passwordless sudo even though read-only systemctl queries normally work unprivileged. Route read-only checks directly, or fall back to sudo only for mutating calls or real permission errors.
  - confidence: 0.91
- **[P2] Handle system-unit unlink failures explicitly:** `src/daemon/systemd.ts:596-598`
  - body: When `context.unitPath` points under `/etc/systemd/system`, a non-root `fs.unlink` usually throws `EACCES` or `EPERM`. The broad catch reports the unit as not found, leaving it installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Preserve current system-unit update guidance:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main detects system-scoped units in the detached update helper and exits with explicit `sudo systemctl restart` guidance plus restart logging. This branch replaces that with automatic `sudo -n systemctl restart`, which can fail on root/no-sudo hosts and drops the safer operator handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:86`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful now take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the extra system access for user-scope cases.
  - confidence: 0.84
