## Summary

Keep open: current main still misses the system-only systemd gateway path for daemon status/control and update restart gating, and related reports now treat this PR as the canonical implementation thread. The branch is not merge-ready because it forces sudo for read-only systemctl calls, mishandles system-unit uninstall permissions, and is stale against current restart-helper behavior.

## What This Changes

The PR adds Linux systemd system-unit detection for daemon status/control and update restart script generation, with tests for system-scoped unit handling.

## Best Possible Solution

Rebase or replace this with a focused systemd-scope patch that detects system units in service state, reads status without unnecessary sudo, uses privilege only for explicit mutations, preserves update-helper logging/guidance, and covers both user and system scopes.

## Reproduction Assessment

Yes, at source level. Current docs allow /etc system units, while current main's service state, enabled check, runtime status, and package-update restart preparation still flow through user-unit paths and user-scoped systemctl for the managed service path.

## Solution Assessment

No, not as currently implemented. The PR targets the right remaining gap, but the patch should be revised to avoid sudo for read-only status, handle system-unit uninstall permissions, and preserve current restart-helper guidance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` is used for `is-enabled` and `show` as well as restart/stop, so non-root `gateway status` would require passwordless sudo even though plain system `systemctl show` and `is-enabled` are normally readable. Keep sudo for explicit mutations or only after a direct read fails for permissions.
  - confidence: 0.91
- **[P2] Handle system-unit removal permissions explicitly:** `src/daemon/systemd.ts:586-590`
  - body: Once `context.unitPath` can point under `/etc/systemd/system`, this `fs.unlink` runs as the current user and the broad catch reports EACCES/EPERM as "not found". Non-root system-unit uninstalls can leave the unit file installed while printing misleading output.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's Linux restart helper logs restart attempts and emits explicit sudo guidance when it detects a system-scoped unit. This branch's helper path replaces that with automatic `sudo -n systemctl restart`, which can fail in root/no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:86`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong systemctl arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
