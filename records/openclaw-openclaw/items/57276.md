## Summary

Keep open. Current main still does not manage system-scoped systemd gateway units in the normal Linux lifecycle path, and the PR addresses a real WSL/systemd gap, but the proposed patch has blocking correctness issues around sudo use, system-unit uninstall, update restart scripts, and test mocks. The open maintainer PR #74920 overlaps part of the update-restart/diagnostics surface, but it is not merged and does not fully supersede this daemon lifecycle PR.

## What This Changes

This PR adds Linux systemd service-context detection for `/etc/systemd/system/<unit>.service`, routes daemon status/control and the update restart helper between system and user scopes, and adds tests for those paths.

## Best Possible Solution

Keep the useful system-vs-user scope detection, but revise it into a shared systemd service context with separate privilege policy: use direct `systemctl` for read-only system-unit status/show, use root-aware or sudo-aware paths only for mutating operations, handle system-unit removal explicitly, share the system-unit path helper with the restart helper, and preserve regression coverage for both user and system units.

## Reproduction Assessment

Yes. On current main, a Linux/WSL host with only `/etc/systemd/system/openclaw-gateway.service` will still hit the user-unit path in `isSystemdServiceEnabled`, `readSystemdServiceRuntime`, and restart helper generation; the PR body also reports live WSL verification of the intended behavior.

## Solution Assessment

No, not as-is. The direction is maintainable, but the current patch conflates system scope with sudo, does not safely uninstall system units, and leaves test mocks inconsistent with the new `/etc/systemd/system` probe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use unprivileged systemctl for read-only status:** `src/daemon/systemd.ts:308`
  - body: System-scoped `isSystemdServiceEnabled` and runtime reads now flow through `execSystemctlSystem`, which runs `sudo -n systemctl` for every non-root system-scope command. On normal hosts, `systemctl is-enabled` and `systemctl show` are readable without sudo; requiring passwordless sudo makes `gateway status` fail with a sudo error even when the system unit is healthy. Use direct `systemctl` for read-only operations and reserve sudo for mutating actions.
  - confidence: 0.88
- **[P2] Remove system units with a privileged path:** `src/daemon/systemd.ts:592-594`
  - body: Once `context.unitPath` can point at `/etc/systemd/system/...`, uninstall still calls `fs.unlink(unitPath)` as the current user and reports every failure as not found. A non-root uninstall of a system unit will hit `EACCES`, leave the unit installed, and print the wrong result; use a privileged removal path or surface a permission error.
  - confidence: 0.9
- **[P2] Avoid sudo-only restart scripts for root updates:** `src/cli/update-cli/restart-helper.ts:88`
  - body: The generated Linux restart helper emits `sudo -n systemctl restart` whenever a system unit exists, even when the updater script runs as root or in a minimal environment without sudo. That makes updates fail to restart a system-scoped gateway despite root already having permission; emit plain `systemctl` for root and use sudo only for non-root callers.
  - confidence: 0.86
- **[P2] Keep user-scope tests on the user-systemd path:** `src/daemon/systemd.test.ts:75`
  - body: The new `/etc/systemd/system` probe treats any successful `fs.access` as a system unit, but existing helpers such as `readManagedServiceEnabled` mock `fs.access` as always successful while still expecting `systemctl --user`. Those tests now exercise the sudo path by accident; make the mocks reject the system-unit probe unless the case is explicitly system-scoped.
  - confidence: 0.84

## Risks / Open Questions

- The PR as written can require passwordless sudo for read-only systemd `is-enabled` and `show` calls that usually work unprivileged, making `gateway status` fail on normal system-unit installs.
- System-unit uninstall can leave `/etc/systemd/system/<unit>.service` behind while reporting it as not found because direct `fs.unlink` permission failures are swallowed.
- The generated update restart script can fail on root or minimal system installs because it emits `sudo -n systemctl restart` whenever a system unit exists.
- Existing user-scope tests with unconditional `fs.access` mocks are likely to switch onto the system-scope path unless the mocks become path-specific.
- The open #74920 overlap should be reconciled by maintainers before merge so the update-restart behavior is not solved two different ways.
