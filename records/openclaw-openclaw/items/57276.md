## Summary

Keep open: the daemon-side systemd system-unit bug is still source-reproducible on current main, but this branch is not safe to merge because it routes read-only status through sudo, mishandles system-unit uninstall failures, regresses the newer update-helper policy, and leaves user-scope tests with stale mocks.

## What This Changes

The PR adds Linux systemd system-unit detection for gateway status/control paths, changes the update restart helper to use `sudo -n systemctl` for `/etc/systemd/system` units, and adds focused tests.

## Best Possible Solution

Rebase or replace the branch with a daemon-focused system-unit context that reads status unprivileged, uses explicit/root-aware privilege only for mutations, preserves current update-helper guidance, and keeps user-unit behavior unchanged.

## Reproduction Assessment

Yes, at source level. Current main still resolves managed Linux gateway unit files from the user service directory and routes enabled/runtime/control calls through `systemctl --user`, while docs support `/etc/systemd/system` system units.

## Solution Assessment

No, not as currently implemented. The PR targets the right daemon surface, but blanket sudo routing, misleading system-unit unlink handling, current-main update-helper drift, and stale test mocks need correction first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` sends system-scope `is-enabled` and `show` through `sudo -n`, so `openclaw gateway status` can fail for normal operators without passwordless sudo even though read-only systemctl queries normally work unprivileged. Route read-only checks directly, or fall back to sudo only for mutating calls or real permission errors.
  - confidence: 0.91
- **[P2] Handle system-unit unlink failures explicitly:** `src/daemon/systemd.ts:593-598`
  - body: When `context.unitPath` points under `/etc/systemd/system`, a non-root `fs.unlink` usually throws `EACCES` or `EPERM`. The broad catch reports the unit as not found, leaving it installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Preserve current system-unit update guidance:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main detects system-scoped units in the detached update helper and exits with explicit `sudo systemctl restart` guidance plus restart logging. This branch replaces that with automatic `sudo -n systemctl restart`, which can fail on root/no-sudo hosts and drops the safer operator handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests on the user path:** `src/daemon/systemd.test.ts:75-76`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so tests that mock every `fs.access` call as successful now take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the extra system access for user-scope cases.
  - confidence: 0.84
