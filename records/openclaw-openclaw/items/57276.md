## Summary

Keep this PR open because it targets a real Linux gateway bug, but the branch is not mergeable as-is: current main still misses system-scoped daemon status/control paths, while the PR overuses sudo, mishandles system-unit uninstall, and conflicts with newer update-helper behavior already merged on main.

## What This Changes

The PR adds system-vs-user systemd detection to Linux daemon status/control paths and update restart script generation, with tests for system-scoped units.

## Best Possible Solution

Implement focused daemon-systemd support that detects system units for status/runtime/control, keeps read-only queries unprivileged, uses privilege only for explicit mutations, handles uninstall permissions honestly, and preserves the current update restart guidance.

## Reproduction Assessment

Yes, at source level. Current docs support `/etc/systemd/system` gateway units, while current main still uses the user unit path or `systemctl --user` for normal daemon managed-command, enabled, runtime, stop, and restart paths.

## Solution Assessment

No, not as currently implemented. The PR addresses the real gap, but the branch should be rebased or replaced so read-only systemd queries avoid sudo, system-unit uninstall reports or handles permissions correctly, and the newer update-helper behavior is retained.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` is used for read-only `is-enabled` and `show` paths as well as `stop` and `restart`, so `gateway status` can require passwordless sudo even though plain system `systemctl` reads are normally available. Keep sudo for explicit mutations or try direct reads before falling back.
  - confidence: 0.91
- **[P2] Handle system-unit removal permissions explicitly:** `src/daemon/systemd.ts:594-597`
  - body: When `context.unitPath` points under `/etc/systemd/system`, `fs.unlink` runs as the current user and EACCES/EPERM is caught as `not found`. Non-root system-unit uninstalls can leave the unit installed while printing a misleading message.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's Linux restart helper logs restart attempts and emits explicit sudo guidance when it sees a system-scoped unit. This branch replaces that with automatic `sudo -n restart`, which can fail in root or no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.test.ts:73-76`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong systemctl arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
