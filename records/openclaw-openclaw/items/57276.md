## Summary

Keep open: current main still has the daemon-side systemd scope gap, but this PR is not merge-ready because it routes read-only systemd checks through sudo, mishandles system-unit unlink failures, conflicts with the merged update-helper guidance, and leaves brittle user-scope tests.

## What This Changes

This PR detects `/etc/systemd/system/<unit>.service` in the Linux daemon and update restart helper, routes system-scope operations through `sudo -n systemctl`, and adds focused tests.

## Best Possible Solution

Rebase or replace this with a daemon-focused system-unit context that reads status without sudo, uses explicit/root-aware privilege only for mutations, preserves current update-restart guidance, and covers user/system paths with focused tests.

## Reproduction Assessment

Yes, at source level. Current main documents Linux system units but the normal daemon status, enabled checks, runtime reads, and control actions still resolve the user unit path and use the user systemctl helper.

## Solution Assessment

No, not as currently implemented. The PR touches the right surface, but blanket sudo routing, misleading unlink handling, stale update-helper behavior, and brittle tests should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` sends every system-scope call through `sudo -n` for non-root users, including read-only `is-enabled` and `show` used by status. That makes normal status checks fail on hosts without passwordless sudo even when plain system-scope `systemctl` probes are enough.
  - confidence: 0.91
- **[P2] Handle system-unit unlink failures explicitly:** `src/daemon/systemd.ts:596-598`
  - body: When `context.unitPath` points under `/etc/systemd/system`, a non-root `fs.unlink` usually throws `EACCES` or `EPERM`. The broad catch reports the unit as not found, leaving it installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Preserve current system-unit update guidance:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's restart helper detects system-scoped units and emits explicit `sudo systemctl restart` guidance with restart logging. This branch replaces that path with automatic `sudo -n systemctl restart`, which can fail on root/no-sudo hosts and drops the merged operator handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:646-648`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful now take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the extra system access for user-scope cases.
  - confidence: 0.84
