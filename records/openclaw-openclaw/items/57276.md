## Summary

Keep this PR open: it targets a real docs-supported systemd system-unit gap, but current main only partially covers update guidance and the branch still has correctness blockers around sudo policy, uninstall behavior, root-run restart scripts, and stale tests.

## What This Changes

The PR adds Linux systemd system-unit detection for daemon lifecycle helpers and the update restart-helper, with tests for `/etc/systemd/system/<unit>.service` routing.

## Best Possible Solution

Revise or replace the branch on current main so system-unit status stays unprivileged, privileged mutations are explicit and safe, uninstall/update paths handle system scope accurately, and user-unit behavior remains unchanged.

## Reproduction Assessment

Yes, at source level: current main documents `/etc/systemd/system` gateway units but daemon lifecycle and status helpers still resolve and control only the user-scoped unit path. I did not run a live WSL/systemd host in this read-only pass.

## Solution Assessment

No, not as currently implemented. The PR targets the right files, but blanket sudo routing, misleading system-unit unlink handling, root restart-script behavior, and stale tests need correction after the privilege policy is settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid sudo for read-only systemd checks:** `src/daemon/systemd.ts:305`
  - body: `execSystemctlForContext` routes system-scope `is-enabled` and `show` through `sudo -n`, so `gateway status` can fail for non-root users without passwordless sudo even though read-only systemd queries normally work unprivileged. Reserve sudo for mutating operations or try direct `systemctl` for read-only checks.
  - confidence: 0.91
- **[P2] Handle system-unit removal errors explicitly:** `src/daemon/systemd.ts:595-598`
  - body: When `context.unitPath` points under `/etc/systemd/system`, a non-root `fs.unlink` normally throws `EACCES` or `EPERM`. The broad catch reports the unit as not found, leaving it installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Avoid sudo in root-run restart scripts:** `src/cli/update-cli/restart-helper.ts:88`
  - body: The generated Linux update script always emits `sudo -n systemctl restart` for system units. If the detached script already runs as root or the host lacks `sudo`, the restart fails even though plain `systemctl restart` would work.
  - confidence: 0.86
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:84-86`
  - body: The new `/etc/systemd/system` probe runs before the user-unit check, so existing tests that mock all `fs.access` calls as successful now select the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the extra system access for user-scope cases.
  - confidence: 0.84
