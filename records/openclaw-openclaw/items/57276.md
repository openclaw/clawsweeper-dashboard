## Summary

Keep open; the Linux systemd system-unit gap is real, but this branch is not merge-ready because its scope-selection and privilege behavior can break existing user-service installs and non-passwordless-sudo hosts.

## What This Changes

The PR adds Linux systemd system-unit detection for gateway status/lifecycle commands and the update restart helper, with focused test coverage.

## Best Possible Solution

Land a coordinated systemd-scope fix that derives scope from active/enabled ownership or explicit service facts, keeps read-only probes unprivileged, handles mutating system-unit actions with explicit privilege or operator guidance, and reconciles with current update-helper and status/runtime work.

## Reproduction Assessment

Yes, at source level: current docs support system units under /etc, while current main's daemon status and lifecycle code still resolves the user unit path and calls user-scoped systemctl. I did not run a live WSL/systemd reproduction in this read-only review.

## Solution Assessment

No, not as implemented. The direction is valid, but file-existence scope selection, blanket sudo use, and stale update-helper changes are not the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Choose systemd scope from live ownership:** `src/daemon/systemd.ts:86-90`
  - body: `resolveSystemdServiceContext` switches to system scope solely because the `/etc` unit file exists. If a user-scoped gateway is active while an old or disabled system unit remains, status/restart/stop/uninstall target the wrong service; base the choice on active/enabled ownership or explicit service facts instead.
  - confidence: 0.9
- **[P2] Reserve sudo for mutating system-unit actions:** `src/daemon/systemd.ts:307-309`
  - body: `execSystemctlSystem` wraps every non-root system-scope call in `sudo -n`, including read-only `is-enabled` and `show` probes used by status. Those normally work unprivileged, so status can fail on healthy system-unit installs that do not grant passwordless sudo.
  - confidence: 0.91
- **[P2] Surface system-unit removal permission failures:** `src/daemon/systemd.ts:592-598`
  - body: Once `context.unitPath` can point under `/etc/systemd/system`, `fs.unlink` runs without sudo and catches EACCES/EPERM as `not found`. A non-root uninstall can leave the unit installed while printing a misleading success-adjacent message.
  - confidence: 0.9
- **[P2] Preserve the current restart handoff diagnostics:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main logs Linux update restart attempts and prints explicit sudo guidance when a system-scoped unit is detected. This branch emits automatic `sudo -n systemctl restart` instead, which can fail in root/no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
- **[P2] Make user-scope fs.access mocks path-specific:** `src/daemon/systemd.test.ts:73-76`
  - body: The shared test helper still mocks every `fs.access` call as successful, so the new `/etc/systemd/system` probe is also treated as present and user-scope tests expecting `systemctl --user` switch to the system-scope path.
  - confidence: 0.86
