## Summary

This PR should stay open because it addresses a real Linux gateway/systemd gap, but it is not safe to merge as-is: current main has already superseded the update-helper part, while the daemon-side system-unit support still has correctness and privileged-service policy issues.

## What This Changes

The PR adds system-scoped systemd unit detection to Linux daemon status/control paths and update restart-helper generation, with focused tests for system-unit routing.

## Best Possible Solution

Rework this as a focused daemon-systemd patch that detects system units for supported service discovery/control, keeps read-only probes unprivileged, uses privilege only for explicit mutations, reports system-unit uninstall permission failures honestly, and leaves current main's update-helper handoff behavior intact.

## Reproduction Assessment

Yes, at source level. Current docs support /etc/systemd/system gateway units, while current main still reads the user unit path and routes daemon lifecycle/status calls through systemctl --user.

## Solution Assessment

No, not as currently implemented. The direction is valid, but the branch should be reworked around unprivileged read probes, explicit privileged mutations, honest uninstall errors, and current main's update-helper handoff behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` is used for read-only `is-enabled` and `show` paths as well as `stop` and `restart`, so `gateway status` can require passwordless sudo even though plain system `systemctl` reads are normally available. Keep sudo for explicit mutations or try direct reads before falling back.
  - confidence: 0.91
- **[P2] Handle system-unit removal permissions explicitly:** `src/daemon/systemd.ts:594-597`
  - body: When `context.unitPath` points under `/etc/systemd/system`, `fs.unlink` runs as the current user and `EACCES` or `EPERM` is caught as `not found`. Non-root system-unit uninstalls can leave the unit installed while printing a misleading message.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's Linux restart helper logs restart attempts and emits explicit sudo guidance when it sees a system-scoped unit. This branch replaces that with automatic `sudo -n restart`, which can fail in root or no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests on the user path:** `src/daemon/systemd.ts:646-648`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong systemctl arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
