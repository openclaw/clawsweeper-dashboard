## Summary

Keep open: current main still lacks full system-scoped systemd daemon lifecycle support, so this PR is not obsolete, but the proposed patch has merge-blocking privilege-handling and test-regression issues.

## What This Changes

The PR adds Linux systemd system-unit detection in daemon service helpers and the update restart-helper, switching `/etc/systemd/system/<unit>.service` cases from user-scoped `systemctl --user` to system-level `systemctl`/`sudo` paths with tests.

## Best Possible Solution

Settle the supported system-unit lifecycle boundary, then revise this PR or replace it so read-only diagnostics stay unprivileged, privileged mutations are explicit, update gating detects system units safely, and existing user-unit behavior remains unchanged.

## Reproduction Assessment

Yes, at source level. Current main documents `/etc/systemd/system` gateway units but the daemon lifecycle and package-update gate still resolve the managed Linux service through the user unit path and user-scoped `systemctl`.

## Solution Assessment

No, not as currently implemented. The PR targets the right files, but blanket sudo routing, misleading system-unit unlink handling, root restart-script behavior, and stale test mocks need correction after maintainers confirm the privilege policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid sudo for read-only systemd checks:** `src/daemon/systemd.ts:311`
  - body: `execSystemctlForContext` sends `is-enabled` and `show` through the same system-scope helper used for mutations, so non-root `gateway status` can require passwordless sudo for read-only queries. Run read-only system-bus checks directly, or reserve sudo for mutating commands.
  - confidence: 0.91
- **[P2] Handle system-unit removal errors explicitly:** `src/daemon/systemd.ts:595`
  - body: When `context.unitPath` is under `/etc/systemd/system`, `fs.unlink` will usually throw `EACCES` for a non-root caller. The broad catch reports the unit as not found, leaving the system unit installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Avoid sudo in root-run restart scripts:** `src/cli/update-cli/restart-helper.ts:88`
  - body: The generated Linux update script always emits `sudo -n systemctl restart` for system units. If the detached script is already running as root, or on a minimal system without sudo, restart fails even though plain `systemctl restart` would work.
  - confidence: 0.86
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:84-86`
  - body: The new `/etc/systemd/system` probe runs before the user-unit check, so existing tests that mock every `fs.access` call as successful now take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the redundant system access in user-scope cases.
  - confidence: 0.84
