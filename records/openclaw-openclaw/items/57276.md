## Summary

Keep open. Current main only has partial systemd-scope diagnostics/guidance, while the central daemon/status/restart lifecycle support this PR targets is still not implemented; however the branch is not merge-ready because it changes privileged systemd behavior and has concrete correctness blockers.

## What This Changes

The PR adds Linux `/etc/systemd/system/<unit>.service` detection to daemon systemd helpers and the update restart-helper, routing system-scoped units through system-level `systemctl` with tests.

## Best Possible Solution

Settle the supported system-unit lifecycle boundary, then revise or replace this PR so read-only diagnostics stay unprivileged, privileged mutations are explicit, update-state gating detects system units safely, and user-unit behavior remains unchanged.

## Reproduction Assessment

Yes, at source level. Current main resolves the managed Linux gateway service from the user unit path and user-scoped `systemctl`, while docs and duplicate issue evidence cover valid `/etc/systemd/system` deployments that this path does not manage.

## Solution Assessment

No, not as currently implemented. The PR targets the right files, but blanket sudo routing, misleading system-unit unlink handling, root restart-script behavior, and stale test mocks need correction after maintainers confirm the privilege policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid sudo for read-only systemd checks:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlForContext` sends `is-enabled` and `show` through the same system-scope helper used for mutations, so non-root `gateway status` can require passwordless sudo for read-only queries. Reserve sudo for mutating commands or try direct `systemctl` first for read-only checks.
  - confidence: 0.91
- **[P2] Handle system-unit removal errors explicitly:** `src/daemon/systemd.ts:594-598`
  - body: When `context.unitPath` is under `/etc/systemd/system`, `fs.unlink` will usually throw `EACCES` for a non-root caller. The broad catch reports the unit as not found, leaving the system unit installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Avoid sudo in root-run restart scripts:** `src/cli/update-cli/restart-helper.ts:88`
  - body: The generated Linux update script always emits `sudo -n systemctl restart` for system units. If the detached script is already running as root, or on a minimal system without sudo, restart fails even though plain `systemctl restart` would work.
  - confidence: 0.86
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:86`
  - body: The new `/etc/systemd/system` probe runs before the user-unit check, so existing tests that mock every `fs.access` call as successful now take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the redundant system access in user-scope cases.
  - confidence: 0.84
