## Summary

Keep open, but not merge-ready: the real WSL proof is useful and current main still has a systemd user-only lifecycle/status gap, while this branch can redirect healthy user-scoped installs to a stale system unit and require passwordless sudo for read-only status checks.

## What This Changes

The PR adds Linux systemd system-unit detection to daemon status/lifecycle helpers and the update restart script, with focused tests for system-scope paths.

## Best Possible Solution

Land a coordinated systemd-scope fix that derives service scope from active/enabled ownership or explicit service facts, keeps read-only probes unprivileged, gives explicit privilege guidance for mutating system-unit actions, and reconciles with the current update-helper and status-runtime work.

## Reproduction Assessment

Yes, at source level: current docs support system units under /etc, while current main's daemon status and lifecycle paths still resolve the user unit and call user-scoped systemctl. I did not run a live WSL/systemd reproduction in this read-only review, but the PR body includes after-fix live WSL output.

## Solution Assessment

No. The branch demonstrates the right problem, but file-presence scope selection and sudo for read-only probes are not the safest maintainable solution; live service ownership plus unprivileged reads and explicit mutating-action policy is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Choose systemd scope from live service ownership:** `src/daemon/systemd.ts:86-90`
  - body: Selecting system scope solely because `/etc/systemd/system/<unit>.service` exists can redirect a healthy user-scoped gateway to a stale or disabled system unit. If both unit files exist, status, stop, restart, and uninstall now operate on the system unit even when the active service is the user unit, so scope needs to come from active/enabled ownership or explicit service facts.
  - confidence: 0.92
- **[P2] Keep read-only systemd probes unprivileged:** `src/daemon/systemd.ts:307-309`
  - body: `execSystemctlSystem` wraps every non-root system-scope call in `sudo -n`, including read-only `is-enabled` and `show` calls used by status. On normal systems without passwordless sudo, `gateway status` can fail even though those systemctl reads can be performed directly.
  - confidence: 0.9
- **[P2] Surface system-unit removal permission failures:** `src/daemon/systemd.ts:593-598`
  - body: With system scope selected, `unitPath` can be under `/etc/systemd/system`, but uninstall still removes it with `fs.unlink` and catches every error as "not found". A non-root `EACCES` leaves the unit installed while reporting the wrong outcome.
  - confidence: 0.88
- **[P2] Avoid sudo-only update restarts:** `src/cli/update-cli/restart-helper.ts:82-88`
  - body: The generated Linux restart script always emits `sudo -n systemctl restart` when the system unit file exists. That fails when the updater is already root on a minimal host without sudo, and it also replaces current main's clearer system-scope detection and operator guidance.
  - confidence: 0.85
