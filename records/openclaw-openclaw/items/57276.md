## Summary

Keep open: the system-scoped systemd daemon gap is source-reproducible and current main does not fully solve it, but this PR still has correctness blockers around sudo routing, uninstall errors, stale test mocks, and current-main update guidance.

## What This Changes

The branch adds system/user systemd context resolution for daemon status/control and generates sudo-based Linux update restart scripts when `/etc/systemd/system/<unit>.service` exists.

## Best Possible Solution

Revise or replace the branch on current main so system-unit status/read paths stay unprivileged, privileged mutations are explicit and root-aware, uninstall handles `/etc` units accurately, update behavior preserves current guidance/logging policy, and user-unit behavior remains unchanged.

## Reproduction Assessment

Yes, at source level. Current main documents system-level gateway units but the daemon service helpers still resolve the managed systemd unit under the user service directory and route through `systemctl --user`; I did not run a live systemd host.

## Solution Assessment

No, not as currently implemented. The PR targets the right surface, but blanket sudo routing, misleading system-unit unlink handling, root restart-script behavior, stale tests, and current-main drift need correction first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlForContext` sends system-scope `is-enabled` and `show` through `sudo -n`, so `openclaw gateway status` can fail for a normal operator without passwordless sudo even though read-only systemctl queries normally work unprivileged. Route read-only checks directly, or fall back to sudo only for mutating calls or real permission errors.
  - confidence: 0.91
- **[P2] Handle system-unit unlink failures explicitly:** `src/daemon/systemd.ts:596-598`
  - body: When `context.unitPath` points under `/etc/systemd/system`, a non-root `fs.unlink` usually throws `EACCES` or `EPERM`. The broad catch reports the unit as not found, leaving it installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Avoid sudo when the restart script runs as root:** `src/cli/update-cli/restart-helper.ts:88`
  - body: The generated Linux update script emits `sudo -n systemctl restart` whenever a system unit exists. If the detached script already runs as root or the host lacks `sudo`, the restart fails even though plain `systemctl restart` would work.
  - confidence: 0.86
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:646-648`
  - body: The new context resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful now take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the extra system access for user-scope cases.
  - confidence: 0.84
