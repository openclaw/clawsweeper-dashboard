## Summary

Keep this PR open because it targets a real Linux daemon gap, but the current branch is conflicting and not safe to merge as-is: it overuses sudo for read-only systemd queries, mishandles system-unit uninstall permissions, and regresses newer update-helper behavior already on main.

## What This Changes

The PR adds Linux systemd system-vs-user unit detection for gateway daemon status/control paths and update restart script generation, with focused tests.

## Best Possible Solution

Rebase or replace this with a focused daemon-systemd patch that detects system units for command/status/control, keeps read-only status unprivileged, uses privilege only for explicit mutations, handles system-unit uninstall honestly, and preserves the current update-helper handoff guidance.

## Reproduction Assessment

Yes, at source level. Current docs support /etc/systemd/system gateway units, while current main still reads the user unit path and routes normal lifecycle/status calls through systemctl --user.

## Solution Assessment

No, not as currently implemented. The direction is valid, but the branch should be reworked around unprivileged read probes, explicit privileged mutations, honest uninstall errors, and the current update-helper behavior already on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` is used for read-only `is-enabled` and `show` paths as well as `stop` and `restart`, so `gateway status` can require passwordless sudo even though plain system `systemctl` reads are normally available. Keep sudo for explicit mutations or try direct reads before falling back.
  - confidence: 0.91
- **[P2] Handle system-unit removal permissions explicitly:** `src/daemon/systemd.ts:594-597`
  - body: When `context.unitPath` points under `/etc/systemd/system`, `fs.unlink` runs as the current user and `EACCES` or `EPERM` is caught as `not found`. Non-root system-unit uninstalls can leave the unit installed while printing a misleading message.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's Linux restart helper logs restart attempts and emits explicit sudo guidance when it sees a system-scoped unit. This branch replaces that with automatic `sudo -n restart`, which can fail in root or no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests on the user path:** `src/daemon/systemd.ts:643-646`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong systemctl arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
