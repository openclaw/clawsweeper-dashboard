## Summary

Keep open: current main still leaves the managed Linux gateway service adapter centered on user-scoped systemd units, and this PR is the canonical implementation thread for the remaining system-unit lifecycle gap. The branch is not mergeable as-is because it routes read-only status through sudo, mishandles system-unit unlink failures, conflicts with newer update restart guidance, and needs a semantic rebase.

## What This Changes

The PR adds Linux systemd system-unit detection to daemon status/control and update restart script generation, with focused tests for system-scoped units.

## Best Possible Solution

Revise or replace the branch on current main so system-unit status/read paths stay unprivileged, privileged mutations are explicit and root-aware, uninstall handles `/etc` units accurately, update behavior preserves current guidance/logging policy, and user-unit behavior remains unchanged.

## Reproduction Assessment

Yes, at source level. Current main documents system-level gateway units, but the managed Linux service adapter still resolves the gateway unit under the user service directory and routes managed status/control through `systemctl --user`; I did not run a live systemd host.

## Solution Assessment

No, not as currently implemented. The PR targets the right surface, but blanket sudo routing, misleading system-unit unlink handling, root restart-script behavior, stale tests, and current-main drift need correction first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep read-only systemd checks unprivileged:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlForContext` sends system-scope `is-enabled` and `show` through `sudo -n`, so `openclaw gateway status` can fail for a normal operator without passwordless sudo even though read-only systemctl queries normally work unprivileged. Route read-only checks directly, or fall back to sudo only for mutating calls or real permission errors.
  - confidence: 0.91
- **[P2] Report system-unit unlink permission failures:** `src/daemon/systemd.ts:596-598`
  - body: When `context.unitPath` points under `/etc/systemd/system`, a non-root `fs.unlink` usually throws `EACCES` or `EPERM`. The broad catch reports the unit as not found, leaving it installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Skip sudo in root-owned restart scripts:** `src/cli/update-cli/restart-helper.ts:88`
  - body: The generated Linux update script emits `sudo -n systemctl restart` whenever a system unit exists. If the detached script already runs as root or the host lacks `sudo`, the restart fails even though plain `systemctl restart` would work.
  - confidence: 0.86
- **[P2] Make user-scope fs.access mocks path-specific:** `src/daemon/systemd.ts:86`
  - body: The new context resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful now take the system-scope path and assert the wrong `systemctl` arguments. Reject the system-unit probe in user-scope tests unless the test is explicitly exercising system scope.
  - confidence: 0.84
