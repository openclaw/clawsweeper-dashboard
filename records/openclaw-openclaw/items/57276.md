## Summary

Keep this PR open. Current main still does not implement the PR's central managed system-scoped systemd lifecycle behavior, but the proposed patch is not merge-ready because it overuses `sudo -n` for read-only status paths, mishandles system-unit uninstall, and likely breaks existing user-scope tests.

## What This Changes

The PR adds `/etc/systemd/system` unit detection to the Linux daemon and update restart helper, routes detected system-scoped units through systemctl/sudo, and adds Vitest coverage for system-scoped status, stop, restart, and restart-script generation.

## Best Possible Solution

Decide the intended product boundary first. If OpenClaw should manage system-scoped systemd units, add a shared service context that detects system units, uses direct `systemctl` for read-only status/show, uses root-aware privileged handling only for mutating operations, reports uninstall permission failures explicitly, preserves current update-restart diagnostics, and covers both user and system scopes in focused tests.

## Reproduction Assessment

Yes. Against current main, a host with only `/etc/systemd/system/openclaw-gateway.service` and no user unit follows the user-unit `fs.access` and `execSystemctlUser` paths, so managed status/restart can misreport or target the wrong scope; the PR blockers are reproducible from the added `sudo -n` routing and existing unconditional `fs.access` mocks.

## Solution Assessment

No. The PR targets the right files and user problem, but the current implementation is not the best fix because it treats read-only systemd queries like privileged mutations, always emits sudo in the update restart helper for system units, and does not handle `/etc/systemd/system` removal permissions correctly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use direct systemctl for read-only system scope checks:** `src/daemon/systemd.ts:308`
  - body: `execSystemctlForContext` sends every system-scope command through `execSystemctlSystem`, and `execSystemctlSystem` uses `sudo -n` for non-root callers. That means `gateway status`/`is-enabled` can fail on normal systems without passwordless sudo even though read-only `systemctl show` and `is-enabled` usually work unprivileged. Reserve sudo for mutating operations or try direct systemctl first for read-only commands.
  - confidence: 0.91
- **[P2] Handle system-unit file removal permissions:** `src/daemon/systemd.ts:592`
  - body: After `context.unitPath` can point at `/etc/systemd/system/...`, uninstall still removes it with plain `fs.unlink` and the existing catch reports "not found" for all failures. A non-root uninstall will typically leave the system unit installed with `EACCES` hidden as a missing file, so this path needs privileged removal or an explicit permission error.
  - confidence: 0.9
- **[P2] Avoid requiring sudo when the restart script runs as root:** `src/cli/update-cli/restart-helper.ts:88`
  - body: When a system unit exists, the generated Linux restart script always emits `sudo -n systemctl restart ...`. Root-run or minimal global installs may not have `sudo`, so the update restart can fail even though plain `systemctl restart ...` would work. Generate a root/no-sudo-aware command or preserve explicit guidance for non-root callers.
  - confidence: 0.86
- **[P2] Keep user-scope tests from matching the system probe:** `src/daemon/systemd.ts:86`
  - body: The new `/etc/systemd/system` probe runs before user-scope checks, but existing tests such as `readManagedServiceEnabled` mock `fs.access` as succeeding for every path. Those tests will now accidentally select the system-scope path and assert the wrong `systemctl` arguments; make the mocks path-specific or skip the redundant system-scope access check.
  - confidence: 0.84
