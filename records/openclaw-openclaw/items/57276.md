## Summary

Keep open. Current main still does not fully support documented system-scoped Linux gateway units, and this PR is the canonical thread for that work, but the patch has blocking correctness issues around sudo policy, uninstall behavior, update-helper behavior, and test mocks.

## What This Changes

The branch adds Linux systemd system-unit detection to daemon status/control and update restart script generation, with tests for system-scoped unit handling.

## Best Possible Solution

Rebase or replace this with a daemon/systemd-scope patch that detects system units, reads status without unnecessary sudo, uses privilege only for explicit mutations, preserves update-helper logging/guidance, and covers both user and system scopes.

## Reproduction Assessment

Yes, at source level. Current docs document /etc system units, but current main's managed-service detection, ExecStart parsing, and runtime status path still use user-unit paths and user-scoped systemctl; I did not run a live systemd system-service update in this checkout.

## Solution Assessment

No, not as currently implemented. The PR targets the right surface, but blanket sudo routing, system-unit unlink handling, current update-helper replacement, and brittle test mocks need changes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` sends every system-scope operation through `sudo -n` for non-root users, including read-only `is-enabled` and `show` calls used by gateway status. Hosts without passwordless sudo will still fail status even though plain system `systemctl` can normally read these properties.
  - confidence: 0.91
- **[P2] Handle system-unit removal permissions explicitly:** `src/daemon/systemd.ts:586-590`
  - body: Resolving uninstall through `resolveSystemdServiceContext` lets `unitPath` point under `/etc/systemd/system`, but the later `fs.unlink` still runs as the current user and the broad catch reports every failure as not found. Non-root system-unit uninstalls can leave the unit installed while printing a misleading success-adjacent message.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's Linux restart helper logs restart attempts and emits explicit sudo guidance when it sees a system-scoped unit. This branch replaces that with automatic `sudo -n systemctl restart`, which can fail for root/no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:86`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong systemctl arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
