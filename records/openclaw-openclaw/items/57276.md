## Summary

Keep open: current main still lacks full daemon lifecycle/status support for documented system-scoped Linux gateway units, but this PR is not mergeable as-is because it routes read-only probes through sudo, mishandles system-unit unlink failures, and would replace newer update-helper handoff behavior on main.

## What This Changes

The PR adds Linux systemd system-vs-user unit detection for gateway daemon status/control paths and update restart-script generation, with focused tests.

## Best Possible Solution

Rework or replace this with a focused systemd-scope patch that detects real system units, keeps read-only status probes unprivileged, uses explicit privilege or operator handoff for lifecycle mutations, preserves current update-helper guidance, and adds focused tests plus live Linux/WSL proof.

## Reproduction Assessment

Yes, at source level: current main still resolves daemon status/lifecycle through user-unit paths while docs support system units under /etc/systemd/system. I did not run a live WSL/systemd reproduction in this read-only review.

## Solution Assessment

No, not as currently implemented. The direction is valid, but the patch should avoid sudo for read-only probes, handle system-unit removal permissions honestly, and preserve current main's update-helper handoff behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` wraps every system-scoped operation in `sudo -n`, including read-only `is-enabled` and `show` calls used by gateway status. On typical installs, plain system `systemctl show`/`is-enabled` can work without passwordless sudo, so status can fail where it should remain a read-only probe.
  - confidence: 0.91
- **[P2] Handle system-unit removal permissions explicitly:** `src/daemon/systemd.ts:592-597`
  - body: When `context.unitPath` points under `/etc/systemd/system`, `fs.unlink` runs as the current user and EACCES/EPERM is caught as `not found`. Non-root system-unit uninstalls can leave the unit installed while printing a misleading success-path message.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's Linux restart helper logs restart attempts and emits explicit sudo guidance when it detects a system-scoped unit. This branch replaces that with automatic `sudo -n restart`, which can fail in root or no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests on the user path:** `src/daemon/systemd.ts:86`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
