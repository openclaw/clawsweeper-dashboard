## Summary

Keep this PR open: current main only covers the update-helper half with operator guidance, while the daemon status/control paths still use user-scoped systemd. The proposed branch still has meaningful daemon-side work, but it is not mergeable as-is because the privileged systemd strategy and stale update-helper overlap need maintainer review.

## What This Changes

The PR adds Linux systemd system-unit detection to gateway daemon status/control and the update restart helper, with tests for system-vs-user unit selection.

## Best Possible Solution

Define the supported boundary for system-scoped systemd units, then implement daemon-side detection/status/control in a way that preserves current update diagnostics, avoids unnecessary sudo for read-only checks, and reports privileged mutation failures clearly.

## Reproduction Assessment

Yes, at source level. Current main still resolves and controls only the user systemd unit in the daemon lifecycle, while the PR body provides live WSL evidence for the system-unit behavior it is trying to support.

## Solution Assessment

No. The PR targets the right surface, but it is not the best fix as-is because it forces sudo for read-only checks, can hide system-unit removal permission failures, emits sudo in root-run restart scripts, and overlaps stale update-helper code now handled differently on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Run read-only system checks without sudo:** `src/daemon/systemd.ts:307-309`
  - body: `execSystemctlSystem` routes every system-scope command through `sudo -n` for non-root users, including read-only `is-enabled` and `show` calls used by status. That makes ordinary diagnostics fail on installs without passwordless sudo even though those systemctl queries normally work unprivileged.
  - confidence: 0.91
- **[P2] Report system-unit removal permission failures:** `src/daemon/systemd.ts:594-598`
  - body: After `context.unitPath` can point at `/etc/systemd/system/...`, uninstall still removes it with plain `fs.unlink` and reports every error as not found. A non-root uninstall will typically leave the system unit installed with `EACCES` hidden, so this needs privileged removal or an explicit permission error.
  - confidence: 0.9
- **[P2] Avoid sudo when restart scripts already run as root:** `src/cli/update-cli/restart-helper.ts:88`
  - body: The generated Linux update restart script always emits `sudo -n systemctl restart ...` for system units. If the script is already running as root or on a minimal system without sudo, the restart can fail even though plain `systemctl restart ...` would work.
  - confidence: 0.86
- **[P2] Keep user-scope tests from matching the system probe:** `src/daemon/systemd.ts:86`
  - body: The new `/etc/systemd/system` probe runs before the user-unit check, so existing tests that mock all `fs.access` calls as successful now select the system path and assert the wrong systemctl arguments. Make those mocks path-specific or avoid the redundant system access in user-scope cases.
  - confidence: 0.84
