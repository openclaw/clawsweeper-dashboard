## Summary

Keep this PR open: current main still lacks the daemon-side system-scoped systemd lifecycle support this PR targets, and related reports now route the remaining work here, but the proposed patch has blocking correctness issues around sudo policy, uninstall behavior, and tests.

## What This Changes

The PR adds Linux `/etc/systemd/system` unit detection to daemon systemd helpers and update restart-helper scripts, routing system units through system-scope `systemctl` with focused tests.

## Best Possible Solution

Settle the supported system-unit lifecycle boundary, then revise or replace this PR with a rebased implementation that preserves current diagnostics, uses unprivileged read-only `systemctl`, handles privileged mutations explicitly, and keeps user-unit behavior unchanged.

## Reproduction Assessment

Yes, at source level. Current main still checks the user unit path and calls `systemctl --user`, while the PR body and related reports describe real WSL/system-service cases where system units are active but the daemon/update paths miss them.

## Solution Assessment

No. The PR targets the right surface, but it is not the best fix as-is because it forces sudo for read-only checks, hides system-unit unlink permission failures, emits sudo in root-run restart scripts, and needs to preserve newer current-main diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid sudo for read-only systemd checks:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlForContext` sends `is-enabled` and `show` through this helper, so non-root `gateway status` now requires passwordless sudo even though those systemctl queries normally work unprivileged. Reserve sudo for mutating operations, or try direct `systemctl` first for read-only checks.
  - confidence: 0.91
- **[P2] Handle system-unit removal errors explicitly:** `src/daemon/systemd.ts:594-598`
  - body: When `context.unitPath` points at `/etc/systemd/system/...`, `fs.unlink` will usually throw `EACCES` for a non-root caller. The broad catch reports the unit as not found, leaving the system unit installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Avoid sudo in root-run restart scripts:** `src/cli/update-cli/restart-helper.ts:88`
  - body: The generated Linux update script always emits `sudo -n systemctl restart` for system units. If the detached script is already running as root, or on a minimal system without sudo, restart fails even though plain `systemctl restart` would work.
  - confidence: 0.86
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:86`
  - body: The new `/etc/systemd/system` probe runs before the user-unit check, so existing tests that mock every `fs.access` call as successful now take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the redundant system access in user-scope cases.
  - confidence: 0.84
