## Summary

Keep open. Current main has the update-helper half of system-scoped systemd handling from the related merged recovery work, but the daemon adapter still reads, reports, stops, restarts, and uninstalls only the user-scoped unit; this PR targets a real bug but remains incorrect as proposed because it routes read-only status through sudo, mishandles system-unit removal failures, and conflicts with current update restart guidance.

## What This Changes

The branch adds Linux systemd system-unit detection to the daemon adapter and update restart helper, routes detected system units through sudo-backed systemctl, and adds focused tests.

## Best Possible Solution

Rebase or replace this with a daemon-focused system-unit context that reads status via plain systemctl, uses root-aware privilege only for mutations, preserves current update-helper guidance/logging, and covers both user and system scopes.

## Reproduction Assessment

Yes, at source level. Current docs describe system-level gateway units, while current main's daemon adapter still reads and controls only the user-scoped unit path.

## Solution Assessment

No, not as currently implemented. The PR touches the right daemon surface, but blanket sudo routing, system-unit unlink handling, stale update-helper behavior, and brittle test mocks need changes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` sends every system-scope operation through `sudo -n` for non-root users, including read-only `is-enabled` and `show` calls used by gateway status. Hosts without passwordless sudo will still fail status even though plain system `systemctl` can read these properties.
  - confidence: 0.91
- **[P2] Handle system-unit unlink failures explicitly:** `src/daemon/systemd.ts:594-598`
  - body: When `context.unitPath` points under `/etc/systemd/system`, a non-root `fs.unlink` usually throws `EACCES` or `EPERM`. The broad catch reports the unit as not found, leaving the system unit installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's restart helper detects system-scoped units and emits explicit sudo restart guidance with durable logging. This branch replaces that path with automatic `sudo -n systemctl restart`, which can fail for root or no-sudo environments and drops the operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:86`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong systemctl arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
