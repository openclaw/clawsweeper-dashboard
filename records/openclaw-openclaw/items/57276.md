## Summary

Keep this PR open, but not merge-ready: current main and the latest release still leave daemon lifecycle/status primarily user-systemd scoped, while this branch introduces system-scope support with blocking scope-selection, sudo-policy, uninstall, and update-helper compatibility problems.

## What This Changes

The branch adds Linux `/etc/systemd/system/<unit>.service` detection to daemon systemd helpers and the update restart script, routing matched service operations through system-scoped `systemctl`/`sudo` with tests.

## Best Possible Solution

Land one coordinated systemd-scope design that derives scope from active/enabled service ownership or explicit service facts, keeps read-only probes unprivileged, defines mutating-action privilege policy, and reconciles daemon lifecycle with current update-helper and status-runtime work.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.5.22` still use the user unit path and `systemctl --user` for daemon enabled/runtime checks while docs describe system-level gateway units. I did not run a live WSL/systemd reproduction in this read-only review, but the PR body includes after-fix live WSL output.

## Solution Assessment

No. The branch demonstrates the right user problem, but file-presence scope selection and sudo for read-only probes are not the safest maintainable solution; service ownership facts and a clear privilege policy are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Choose systemd scope from live service ownership:** `src/daemon/systemd.ts:86-90`
  - body: Selecting system scope solely because `/etc/systemd/system/<unit>.service` exists can redirect status, stop, restart, and uninstall away from the active user-scoped gateway when a stale or disabled system unit is present. Scope needs to come from active/enabled ownership or explicit service facts, not file presence alone.
  - confidence: 0.92
- **[P2] Keep read-only systemd probes unprivileged:** `src/daemon/systemd.ts:304-311`
  - body: `execSystemctlSystem` wraps every non-root system-scope call in `sudo -n`, including read-only `is-enabled` and `show` calls used by status. On normal systems without passwordless sudo, `gateway status` can fail even though those systemctl reads can be performed directly.
  - confidence: 0.9
- **[P2] Surface system-unit removal permission failures:** `src/daemon/systemd.ts:593-598`
  - body: With system scope selected, `unitPath` can be under `/etc/systemd/system`, but uninstall still removes it with `fs.unlink` and catches every error as not found. A non-root `EACCES` leaves the unit installed while reporting the wrong outcome.
  - confidence: 0.88
- **[P2] Avoid sudo-only update restarts:** `src/cli/update-cli/restart-helper.ts:82-88`
  - body: The generated Linux restart script always emits `sudo -n systemctl restart` when the system unit file exists. That fails when the updater is already root on a minimal host without sudo, and it replaces current main's clearer system-scope detection and operator guidance.
  - confidence: 0.85
