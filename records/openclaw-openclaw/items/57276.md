## Summary

Keep open. Current main still does not manage system-scoped systemd gateway units for daemon status/restart, and the open PR remains the canonical thread, but the patch is not merge-ready because it needs product-boundary approval and revision against newer restart diagnostics.

## What This Changes

The PR adds `/etc/systemd/system` unit detection to Linux gateway systemd lifecycle paths and the update restart helper, routing detected system-scope operations through `sudo -n systemctl` with focused Vitest coverage.

## Best Possible Solution

Decide the supported lifecycle boundary first; if system-scoped units are supported, implement a shared service context with direct read-only `systemctl`, root-aware privileged mutations, explicit uninstall permission handling, preserved update diagnostics, and focused docs/tests/changelog.

## Reproduction Assessment

Yes. Source inspection on current main shows a system unit under `/etc/systemd/system` is not read by the daemon status/restart paths because those paths still resolve the user unit and call `systemctl --user`; the PR blockers are also visible from the diff and review comments.

## Solution Assessment

No. The PR targets the right surface, but it is not the best fix as-is because it requires sudo for read-only checks, hides system-unit removal permission failures, emits sudo even for root-run restart scripts, and needs to preserve #74920 diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use direct systemctl for read-only system scope checks:** `src/daemon/systemd.ts:311`
  - body: `execSystemctlForContext` routes `is-enabled` and `show` through `sudo -n` for non-root system-scope callers. Those read-only systemctl calls normally work unprivileged, so `gateway status` can fail on installs without passwordless sudo even though the status query itself should be allowed.
  - confidence: 0.91
- **[P2] Handle system-unit file removal permissions:** `src/daemon/systemd.ts:595`
  - body: After `context.unitPath` can point at `/etc/systemd/system/...`, uninstall still removes it with plain `fs.unlink` and reports every failure as not found. A non-root uninstall will typically leave the system unit installed with `EACCES` hidden, so this needs privileged removal or an explicit permission error.
  - confidence: 0.9
- **[P2] Avoid requiring sudo when the restart script runs as root:** `src/cli/update-cli/restart-helper.ts:88`
  - body: The generated Linux update restart script always emits `sudo -n systemctl restart ...` for system units. Root-run or minimal installs may not have `sudo`, so the script can fail even though plain `systemctl restart ...` would work.
  - confidence: 0.86
- **[P2] Keep user-scope tests from matching the system probe:** `src/daemon/systemd.ts:86`
  - body: The new `/etc/systemd/system` probe runs before user-scope checks, so existing tests that mock all `fs.access` calls as successful will accidentally select the system path and assert the wrong `systemctl` arguments. Make those mocks path-specific or skip the redundant system-scope access check.
  - confidence: 0.84
