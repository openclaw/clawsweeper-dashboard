## Summary

Keep open: the PR targets a real Linux systemd gap, but it is not mergeable as-is. Current main still routes daemon status/restart through user-scoped systemd while docs describe system units, and the branch overuses sudo, mishandles system-unit uninstall failures, and conflicts with newer update-helper behavior merged in https://github.com/openclaw/openclaw/pull/74920.

## What This Changes

The PR detects `/etc/systemd/system/<unit>.service` for Linux gateway systemd units, routes daemon and update-restart helper calls through system scope when present, and adds focused tests.

## Best Possible Solution

Rebase or replace this with a daemon-focused system-unit context that reads status without sudo, uses explicit/root-aware privilege only for mutations, preserves the merged update restart guidance, and keeps user-unit behavior covered by focused tests.

## Reproduction Assessment

Yes, at source level. Current main documents systemd system units but daemon status, enabled checks, runtime reads, and control actions still resolve the user unit path and use the user systemctl helper.

## Solution Assessment

No, not as currently implemented. The PR touches the right surface, but blanket sudo routing, misleading unlink handling, stale update-helper behavior, and brittle tests should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` sends system-scope `is-enabled` and `show` through `sudo -n`, so `openclaw gateway status` can fail for normal operators without passwordless sudo even though read-only systemctl queries normally work unprivileged. Route read-only checks directly, or fall back to sudo only for mutating calls or real permission errors.
  - confidence: 0.91
- **[P2] Handle system-unit unlink failures explicitly:** `src/daemon/systemd.ts:596-598`
  - body: When `context.unitPath` points under `/etc/systemd/system`, a non-root `fs.unlink` usually throws EACCES or EPERM. The broad catch reports the unit as not found, leaving it installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Preserve current system-unit update guidance:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main detects system-scoped units in the detached update helper and exits with explicit `sudo systemctl restart` guidance plus restart logging. This branch replaces that with automatic `sudo -n systemctl restart`, which can fail on root/no-sudo hosts and drops the safer operator handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:646-648`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful now take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the extra system access for user-scope cases.
  - confidence: 0.84
