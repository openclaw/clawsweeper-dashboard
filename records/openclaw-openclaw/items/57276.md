## Summary

## What This Changes

The PR adds Linux systemd system-unit detection to daemon status/lifecycle helpers and switches the update restart script to system-scoped systemctl behavior when /etc system units exist, with focused tests.

## Best Possible Solution

Land a coordinated systemd-scope fix that derives scope from live service ownership or explicit service facts, keeps read-only probes unprivileged, defines privilege behavior for mutating actions, and reconciles with current update-helper and system-bus status work.

## Reproduction Assessment

Yes, at source level: current main's daemon lifecycle/status code resolves the user unit and calls user-scoped systemctl, while current docs describe system-level gateway units under /etc/systemd/system. I did not run a live WSL/systemd reproduction in this read-only review, but the PR body includes after-fix live WSL output.

## Solution Assessment

No. The branch demonstrates the right problem, but file-presence scope selection and sudo for read-only probes are not the safest maintainable solution; live service ownership plus unprivileged reads and explicit mutating-action policy is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Choose systemd scope from live service ownership:** `src/daemon/systemd.ts:86-90`
  - body: Selecting system scope solely because `/etc/systemd/system/<unit>.service` exists can redirect a healthy user-scoped gateway to a stale or disabled system unit. If both unit files exist, status, stop, restart, and uninstall now operate on the system unit even when the active service is the user unit, so scope needs to come from active/enabled ownership or explicit service facts.
  - confidence: 0.92
- **[P2] Keep read-only systemd probes unprivileged:** `src/daemon/systemd.ts:307-309`
  - body: `execSystemctlSystem` wraps every non-root system-scope call in `sudo -n`, including read-only `is-enabled` and `show` calls used by status. On normal systems without passwordless sudo, `gateway status` can fail even though those systemctl reads can be performed directly.
  - confidence: 0.9
- **[P2] Surface system-unit removal permission failures:** `src/daemon/systemd.ts:593-598`
  - body: With system scope selected, `unitPath` can be under `/etc/systemd/system`, but uninstall still removes it with `fs.unlink` and catches every error as "not found". A non-root `EACCES` leaves the unit installed while reporting the wrong outcome.
  - confidence: 0.88
- **[P2] Avoid sudo-only update restarts:** `src/cli/update-cli/restart-helper.ts:82-88`
  - body: The generated Linux restart script always emits `sudo -n systemctl restart` when the system unit file exists. That fails when the updater is already root on a minimal host without sudo, and it replaces current main's clearer system-scope detection and operator guidance.
  - confidence: 0.85
