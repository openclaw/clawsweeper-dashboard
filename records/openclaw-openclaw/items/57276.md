## Summary

Keep open. The reported daemon problem is still real on current main, but this PR is conflicting and its current diff has merge-blocking policy/correctness issues around sudo use, system-unit uninstall behavior, and stale update-helper handling.

## What This Changes

The branch adds Linux systemd system-vs-user unit detection to daemon service handling and the update restart helper, with tests for system-scoped units.

## Best Possible Solution

Replace or rebase this with a daemon-focused system-unit context that reads status without unnecessary sudo, uses privilege only for mutations, handles /etc unit removal explicitly, preserves current update-helper guidance/logging, and covers both scopes in tests.

## Reproduction Assessment

Yes, at source level. Current docs describe system-level gateway units, while current main's daemon adapter still reads and controls the normal service through the user-scoped unit path.

## Solution Assessment

No, not as currently implemented. The diff touches the right daemon surface, but blanket sudo routing, system-unit unlink handling, stale update-helper behavior, and brittle mocks need changes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:307-308`
  - body: `execSystemctlSystem` sends every system-scope operation through `sudo -n` for non-root users, including read-only `is-enabled` and `show` calls used by gateway status. Hosts without passwordless sudo will still fail status even though plain system `systemctl` can read these properties.
  - confidence: 0.91
- **[P2] Handle system-unit unlink failures explicitly:** `src/daemon/systemd.ts:594-597`
  - body: When `context.unitPath` points under `/etc/systemd/system`, a non-root `fs.unlink` usually throws `EACCES` or `EPERM`. The broad catch reports the unit as not found, leaving the system unit installed and giving misleading uninstall output.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's restart helper detects system-scoped units and emits explicit sudo restart guidance with durable logging. This branch replaces that path with automatic `sudo -n systemctl restart`, which can fail for root or no-sudo environments and drops the operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests from selecting system scope:** `src/daemon/systemd.ts:86`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong systemctl arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
