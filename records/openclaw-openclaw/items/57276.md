## Summary

Keep open: the reported Linux systemd system-unit bug is real and current main does not fully cover daemon lifecycle/status for system-scoped gateway units, but this branch should not merge as-is because its scope selection and sudo policy can misroute existing installs and regress current update-helper behavior.

## What This Changes

This PR adds system-vs-user systemd unit detection for Linux gateway status, control, uninstall, service command reads, and the update restart helper, with focused tests.

## Best Possible Solution

Land a coordinated systemd-scope fix that preserves user-scope compatibility, uses unprivileged read-only system-bus probes, handles mutating system-unit actions with explicit privilege or operator handoff, preserves current update diagnostics, and coordinates with https://github.com/openclaw/openclaw/pull/68909.

Yes, at source level: current main documents system-level Linux gateway units while the daemon lifecycle/runtime paths still use the user unit path and user-scoped systemctl. I did not run a live WSL/systemd reproduction in this read-only review.

## Solution Assessment

No, not as implemented. The direction is valid, but the branch should be reworked around stable scope ownership, unprivileged read-only probes, explicit system-unit privilege handling, and the current update-helper/status work already on main or in the related open PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Choose systemd scope from live ownership:** `src/daemon/systemd.ts:86-90`
  - body: `resolveSystemdServiceContext` switches to system scope solely because `/etc/systemd/system/<unit>.service` exists. If a user-scoped gateway is active while an old or disabled system unit file remains, status/restart/stop/uninstall will target the wrong scope; base the decision on active/enabled ownership or explicit service facts instead.
  - confidence: 0.88
- **[P2] Do not sudo read-only status probes:** `src/daemon/systemd.ts:307-309`
  - body: `execSystemctlSystem` wraps every non-root system-scope call in `sudo -n`, including read-only `is-enabled` and `show` calls used by status. Those normally work without sudo, so status can fail on otherwise healthy system-unit installs that do not grant passwordless sudo.
  - confidence: 0.91
- **[P2] Report system-unit removal failures honestly:** `src/daemon/systemd.ts:592-597`
  - body: Once `context.unitPath` can point under `/etc/systemd/system`, `fs.unlink` runs without sudo and EACCES/EPERM is caught as `not found`. A non-root uninstall can leave the system unit installed while printing a misleading message, so system-scope removal needs an escalated path or an explicit permission error.
  - confidence: 0.9
- **[P2] Preserve the current restart handoff diagnostics:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main logs Linux update restart attempts and prints explicit sudo guidance when a system-scoped unit is detected. This branch replaces that with automatic `sudo -n systemctl restart`, which can fail in root/no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
