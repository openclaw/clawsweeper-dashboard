## Summary

Keep open: the reported system-scoped systemd gap is real, but this PR is not mergeable as-is because current main already shipped a safer update restart handoff, while the branch still adds risky sudo routing and misleading system-unit uninstall behavior.

## What This Changes

The PR adds Linux gateway systemd service-scope detection in daemon status/control/uninstall/read-command paths and changes the update restart helper/tests to target system units when `/etc/systemd/system/<unit>.service` exists.

## Best Possible Solution

Rework or replace this as a coordinated systemd-scope fix that keeps read-only probes unprivileged, preserves current update-helper guidance, handles privileged lifecycle operations explicitly, and aligns with the status work in https://github.com/openclaw/openclaw/pull/68909.

## Reproduction Assessment

Yes. Source inspection shows current main still uses user-unit paths and `systemctl --user` for daemon enabled/runtime checks while docs support `/etc/systemd/system` gateway units; I did not run a live WSL/systemd host in this read-only review.

## Solution Assessment

No, not as currently implemented. The direction is useful, but the PR should avoid sudo for read-only probes, preserve current main's update-helper handoff, and make system-unit uninstall/privileged lifecycle behavior explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the current update restart handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's Linux restart helper logs restart attempts and emits explicit sudo guidance when it detects a system-scoped unit. This branch replaces that with automatic `sudo -n systemctl restart`, which can fail in root/no-sudo environments and drops the shipped operator-facing handoff.
  - confidence: 0.9
- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:308`
  - body: `execSystemctlSystem` wraps every system-scoped operation in `sudo -n`, including read-only `is-enabled` and `show` calls used by gateway status. Those probes normally work unprivileged, so status can fail on installs without passwordless sudo even though the service is readable.
  - confidence: 0.91
- **[P2] Handle system-unit removal permissions explicitly:** `src/daemon/systemd.ts:594-597`
  - body: When `context.unitPath` points under `/etc/systemd/system`, `fs.unlink` runs as the current user and EACCES/EPERM is caught as not found. Non-root system-unit uninstalls can leave the unit installed while printing a misleading message.
  - confidence: 0.9
- **[P2] Keep user-scope tests on the user path:** `src/daemon/systemd.test.ts:75`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
