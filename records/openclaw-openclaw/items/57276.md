## Summary

Keep open: current main still lacks full daemon system-scope support for documented `/etc/systemd/system` gateway units, and this PR has relevant live WSL proof. The branch is not mergeable as-is because it routes read-only status probes through `sudo`, hides system-unit removal permission failures, and replaces the current update-helper guidance that landed in https://github.com/openclaw/openclaw/pull/74920.

## What This Changes

The PR adds Linux `/etc/systemd/system` unit detection to daemon status/control/uninstall/read-command paths and changes the update restart helper to choose system-scope restarts, with focused tests.

## Best Possible Solution

Rework or replace this as a coordinated systemd-scope fix that uses unprivileged system-bus reads for status, explicit privilege or operator handoff for mutating system units, honest uninstall permission handling, and focused tests plus redacted live Linux/WSL proof.

## Reproduction Assessment

Yes, at source level: current main documents system-level gateway units but daemon status and lifecycle still resolve user unit paths and user-scoped `systemctl` calls. I did not run a live WSL/systemd reproduction in this read-only review.

## Solution Assessment

No, not as currently implemented. The direction is valid, but the patch should avoid sudo for read-only probes, preserve the current update-helper handoff, and handle system-unit removal permissions explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reserve sudo for mutating systemd calls:** `src/daemon/systemd.ts:305`
  - body: `execSystemctlSystem` wraps every system-scoped operation in `sudo -n`, including read-only `is-enabled` and `show` calls used by gateway status. Those probes are normally readable without sudo, so status can fail on installs without passwordless sudo even though the service is visible.
  - confidence: 0.91
- **[P2] Handle system-unit removal permissions explicitly:** `src/daemon/systemd.ts:592-597`
  - body: When `context.unitPath` points under `/etc/systemd/system`, `fs.unlink` runs as the current user and EACCES/EPERM is caught as `not found`. Non-root system-unit uninstalls can leave the unit installed while printing a misleading message.
  - confidence: 0.9
- **[P2] Preserve the current system-unit update handoff:** `src/cli/update-cli/restart-helper.ts:88`
  - body: Current main's Linux restart helper logs restart attempts and emits explicit sudo guidance when it detects a system-scoped unit. This branch replaces that with automatic `sudo -n restart`, which can fail in root or no-sudo environments and drops the durable operator-facing handoff.
  - confidence: 0.88
- **[P2] Keep user-scope tests on the user path:** `src/daemon/systemd.ts:86`
  - body: The new resolver probes `/etc/systemd/system` before the user unit path, so existing tests that mock every `fs.access` call as successful take the system-scope path and assert the wrong `systemctl` arguments. Make those mocks path-specific or avoid the extra access in user-scope cases.
  - confidence: 0.84
