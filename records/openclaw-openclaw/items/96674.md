## Summary

Keep open: the patch addresses the direct Vitest helper probe, but the documented UI E2E command still runs an unchanged preflight Chromium probe first, so the reported Windows hang can still occur before this helper is loaded.

## What This Changes

The PR changes the Control UI E2E Chromium capability helper to run `--headless=new --version` with a 5-second `spawnSync` timeout and treat `ETIMEDOUT` as runnable.

## Best Possible Solution

Use one bounded headless Chromium capability probe for both the Vitest helper and the preflight installer script, with focused tests for timeout fallback.

## Reproduction Assessment

Unclear for live Windows reproduction because I did not run Chrome for Testing 148 on Windows here. The source path is reproducible: `pnpm test:ui:e2e` reaches the unchanged preflight `--version` probe before the patched helper can run.

## Solution Assessment

No. Updating the helper is a plausible mitigation, but the best fix must also update `scripts/ensure-playwright-chromium.mjs` and its injected-spawn tests so the documented entrypoint uses the same bounded headless probe.
