## Summary

Keep open: the parser upper-bound fix is the right central change for the linked bug, but merging this head by itself can make `openclaw node run --port 70000` silently fall back to the configured/default port because that caller treats `parsePort(...) === null` as omission. Pair this with the existing node-run explicit-port validation work before automerge.

## What This Changes

The branch adds a 65535 upper-bound guard to the shared CLI `parsePort` helper and a new colocated Vitest test for nullish, invalid, boundary, and out-of-range inputs.

## Best Possible Solution

Keep the shared parser range guard, but merge it only with explicit `node run --port` validation so fallback is used only when the option is omitted.

## Reproduction Assessment

Yes. Source inspection shows current main accepts out-of-range values in `parsePort`, and the PR would convert them to null; the same source path shows `node run` currently turns null into a configured/default port fallback. I did not execute a live CLI repro in this read-only review.

## Solution Assessment

No, not as-is. The shared parser guard is the narrow right fix for the linked parser bug, but it should be paired with node-run explicit-port validation or sequenced after that fix lands.
