## Summary

Keep open. Current main has already absorbed some adjacent startup work, but it still does not implement the PR's central secrets-runtime auth-store shortcut or async-lock timeout, and the PR branch is not safe to merge as-is because it drops external auth overlays from shared OAuth recovery callers and leaves ungated profiling output in production paths.

## What This Changes

The PR changes gateway startup/auth/secrets paths to skip external auth overlays during secrets-runtime auth-store loads, add an async-lock timeout, adjust Windows mDNS discovery, parallelize startup maintenance, alter Windows UI spawn handling, and add gateway/client/TUI profiling logs.

## Best Possible Solution

Keep the useful startup-latency work, but land it as a narrower current-main change: split a no-external-auth loader specifically for SecretRef startup snapshot resolution, leave shared OAuth refresh/recovery loaders overlay-capable, remove temporary profile logs or route them through the existing gated startup trace, add focused auth-loader and async-lock tests, and validate with the gateway startup benchmark plus the changed gate.

## Reproduction Assessment

Yes for the PR defects: source inspection reproduces that the PR changes a shared secrets-runtime auth loader used by OAuth recovery paths, and the PR patch visibly adds unconditional stderr profiling logs. The startup-latency improvement itself still needs a fresh current-main benchmark because the branch is partly superseded.

## Solution Assessment

No. The PR identifies a plausible bottleneck, but changing the shared loader globally and adding ungated production diagnostics is not the narrowest maintainable fix; a secrets-preflight-specific loader split with tests is safer.
