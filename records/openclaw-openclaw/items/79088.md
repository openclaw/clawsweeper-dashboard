## Summary

Keep open: the report matches current source. `openclaw update` still keys Linux package-update stop/restart decisions off the user-scoped systemd service state, so a system-level Gateway service can be left running while the package tree updates.

## What This Changes

Request that `openclaw update` detect system-scope systemd Gateway services and avoid reporting a completed restart when it cannot restart that scope.

## Best Possible Solution

Teach the package-update restart preflight to detect an active system-scope OpenClaw Gateway service and return a structured warning or failure with sudo restart guidance instead of silently treating it as no managed service.

## Reproduction Assessment

Yes, source-reproducible: a package update with no user unit makes `serviceState.installed` false, so no stop or restart helper is prepared even if a system-level Gateway unit is active. I did not perform a live systemd system-service run in this checkout.

## Solution Assessment

No, the current implementation is not the best fix because it only detects the system-scoped unit inside a helper that system-only installs never reach. The maintainable path is to reuse the existing system service discovery/status seams before package swap and fail or warn explicitly when sudo/system scope is required.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79088. In the Linux package-manager `openclaw update` flow, detect when no user-scoped managed Gateway service is installed but a system-scope OpenClaw Gateway service is present or active, including custom names such as `openclaw.service` when the unit runs `openclaw gateway` or carries the OpenClaw service marker. Do not auto-run sudo and do not add a new config flag unless strictly necessary. Before replacing the package tree, avoid silently skipping service stop/restart for that system scope; after update, return a structured warning or non-success restart result with concrete `sudo systemctl restart <unit>` guidance so `--json` callers cannot treat the restart as completed. Reuse existing seams where possible: `src/daemon/inspect.ts` system service discovery, `src/daemon/systemd.ts` scope-aware activity checks, and `src/cli/update-cli/restart-helper.ts` system-scope guidance. Add regression coverage for a system-only unit path and for JSON-mode behavior. Add a user-facing `CHANGELOG.md` fix entry crediting @islandpreneur007 if policy allows. Preserve existing user-scoped systemd, macOS LaunchAgent, and Windows scheduled-task behavior.
