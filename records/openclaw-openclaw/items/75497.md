## Summary

Keep this PR open. Current main still allows an older same-id ClawHub install record to remain ahead of a newer bundled plugin during `openclaw update`, and the PR addresses that path with a focused version-aware guard, regression tests, and changelog coverage; I did not find a discrete patch-introduced correctness or security issue.

## What This Changes

The PR adds version comparison helpers and a ClawHub-to-bundled reconciliation step in `src/plugins/update.ts`, expands `src/plugins/update.test.ts` coverage for older/newer ClawHub and npm same-id cases, and adds a changelog fix entry.

## Best Possible Solution

Land this PR or an equivalent focused fix after the normal maintainer and CI gates pass, keeping the behavior generic to ClawHub same-id plugin records and preserving explicit npm-installed plugin update semantics.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is present in the PR: seed a same-id ClawHub WhatsApp install at `2026.2.09`, provide a bundled WhatsApp source at `2026.4.25`, and verify `syncPluginsForUpdateChannel` switches the install record and load path back to the bundled plugin.

## Solution Assessment

Yes. Moving the guard into `syncPluginsForUpdateChannel` is the narrowest maintainable point because it reconciles bundled sources before the later external ClawHub refresh, and the PR preserves explicit npm installs on the npm update path.
