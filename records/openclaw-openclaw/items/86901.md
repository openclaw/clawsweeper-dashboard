This PR adds a Feishu table-count guard before static interactive card sends so messages with more than five Markdown tables fall back to post/text delivery.

## Best Possible Solution

Rebase the narrow static-card table-count guard onto current main, add focused tests for over-limit table fallback in outbound and reply-dispatcher paths, and keep streaming CardKit work on the separate canonical issue.

## Solution Assessment

Yes, the narrow static-card preflight is the best current mitigation because it avoids sending a Feishu payload the official card contract rejects and reuses the existing post/table conversion path. It still needs a current-main rebase and tests, and streaming CardKit behavior should remain separate unless maintainers choose a broader fix.
