## Summary

Keep open. Current main still runs the `config.secrets.*_in_config` security-audit checks against resolved config values, so documented `${ENV_VAR}` placeholders can still be reported as stored secrets before the authored form reaches `looksLikeEnvRef()`.

## What This Changes

The issue reports a false positive in `openclaw security audit --deep` when authored `${ENV_VAR}` config references are evaluated after config resolution.

## Best Possible Solution

Add authored-value or field-level source provenance to the security-audit path for `gateway.auth.password` and `hooks.token`, while preserving plaintext warnings when the resolved value originates from config-owned secret data.

## Reproduction Assessment

Yes. Source inspection on current main shows config env substitution happens before the snapshot/runtime view reaches `collectSecretsInConfigFindings`, and that collector only sees resolved strings.

## Solution Assessment

No. Current main is not the best solution, and the suggested sourceConfig-only threading needs refinement because current `sourceConfig` is already post-substitution; the safer path is explicit authored-value or env-source provenance.
