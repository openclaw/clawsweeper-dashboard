## Summary

Keep open. Current main still evaluates the `gateway.auth.password` and `hooks.token` secret-in-config findings against resolved runtime values, while documented `${VAR}` config references are substituted before the audit collector sees them.

## What This Changes

Suppress security audit secret-in-config findings for `gateway.auth.password` and `hooks.token` when those fields are authored as documented `${ENV_VAR}` references.

## Best Possible Solution

Add authored-value or field-level env-ref provenance to the security audit path, then use it only to distinguish exact env references from literal plaintext while keeping the existing runtime hardening checks intact.

## Reproduction Assessment

Yes. The static current-main path is high confidence: `${VAR}` is substituted during config read, the security CLI supplies runtime config, and `collectSecretsInConfigFindings` checks only the resolved `cfg` values.

## Solution Assessment

No. Current main is not the best solution, and the suggested `sourceConfig`-only threading needs refinement because current `sourceConfig` is already post-substitution.
