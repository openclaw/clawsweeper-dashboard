## Summary

Keep open. Current main still runs the security-audit secret-in-config checks on resolved runtime values, so documented `${VAR}` config references can be misclassified before `looksLikeEnvRef()` sees the authored placeholder.

## What This Changes

The issue asks `openclaw security audit` to avoid reporting `gateway.auth.password` and `hooks.token` as stored secrets when those fields are authored as documented `${VAR}` environment references.

## Best Possible Solution

Add authored-value or field-level env-ref provenance to the security audit path, then use it only to distinguish exact env references from literal plaintext while preserving the existing runtime hardening checks.

## Reproduction Assessment

Yes, with source inspection rather than a live command. The path is high-confidence: docs allow `${VAR}` config strings, config read substitutes them before the snapshot/runtime view, and `collectSecretsInConfigFindings` only checks the resolved `cfg` strings.

## Solution Assessment

No. Current main is not the best solution, and the proposed sourceConfig-only threading needs refinement because the current `sourceConfig` is already post-substitution; the safer fix is explicit authored-value or env-ref provenance for the audit decision.
