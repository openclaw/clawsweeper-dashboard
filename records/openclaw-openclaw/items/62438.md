## Summary

Keep open. Current main still runs the `config.secrets.*_in_config` security-audit checks against resolved config values, so documented `${ENV_VAR}` references can still be reported as stored secrets before the authored placeholder is available to `looksLikeEnvRef()`.

## What This Changes

The issue asks `openclaw security audit --deep` to avoid `config.secrets.*_in_config` findings when `gateway.auth.password` or `hooks.token` is authored as a documented `${ENV_VAR}` reference.

## Best Possible Solution

Add authored env-reference or field-level source provenance to the security-audit path for these checks, while preserving plaintext warnings when the resolved value originates from config-owned env data.

## Reproduction Assessment

Yes, with source inspection rather than a live command. Current main documents `${VAR_NAME}` config substitutions, resolves them before snapshot/runtime use, and then `collectSecretsInConfigFindings` checks only resolved strings.

## Solution Assessment

No. Current main is not the best solution, and the suggested sourceConfig-only threading needs refinement because current `sourceConfig` is already post-substitution; the safer path is explicit authored-value or env-source provenance.
