## Summary

Keep open. Current main still runs the security audit secret-in-config collector on the resolved runtime config, so documented `${VAR}` values for `hooks.token` and `gateway.auth.password` can still be misreported after substitution.

## What This Changes

The issue asks `openclaw security audit --deep` to suppress `config.secrets.hooks_token_in_config` and `config.secrets.gateway_password_in_config` when those fields are authored as `${ENV_VAR}` references.

## Best Possible Solution

Implement a source-aware security-audit path that can identify authored exact env references for `gateway.auth.password` and `hooks.token`, while preserving warnings for literal plaintext and for env values that originate from plaintext stored in the config itself.

## Reproduction Assessment

Yes. The static path is high-confidence: author `hooks.token` or `gateway.auth.password` as `${VAR}`, have config loading resolve it to a non-empty value, and `collectSecretsInConfigFindings` will evaluate the resolved secret string rather than the authored sentinel.

## Solution Assessment

No, current main is not the best solution. The right direction is source-aware suppression, but the minimal suggested `sourceConfig` threading needs adjustment because current `sourceConfig` is already post-substitution.
