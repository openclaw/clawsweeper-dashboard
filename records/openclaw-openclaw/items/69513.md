## Summary

Keep open. Current main still validates cron announce channels before persistence but does not validate explicit primary or failure-destination account IDs, and an open implementation PR already targets the same bug but needs maintainer handling before it can land.

## What This Changes

The issue asks OpenClaw to reject or warn on cron announce jobs whose explicit delivery accountId is not configured for the selected channel.

## Best Possible Solution

Land a focused gateway-side guard for cron.add and cron.update that validates explicit announce account IDs for primary and failure destinations through read-only configured-account metadata while preserving implicit/default-account behavior.

## Reproduction Assessment

Yes, from source inspection rather than a live cron create run. Current main forwards --account to delivery.accountId, shape-checks it, validates only announce channels, and then persists after that incomplete guard.

## Solution Assessment

Yes. Gateway validation before cron.add/update persistence is the narrowest maintainable boundary, but the open implementation should avoid full plugin loading and use read-only account metadata or a config-derived helper.
