## Summary

Keep open. Current main still validates cron announce channels before persistence but does not validate explicit primary or failure-destination account IDs, and the open implementation candidates at https://github.com/openclaw/openclaw/pull/69562 and https://github.com/iamlukethedev/openclaw/pull/2 already target the same bug.

## What This Changes

Reject cron announce jobs whose explicit delivery account ID is not configured for the selected channel.

## Best Possible Solution

Land a focused gateway validation change that rejects unknown explicit account IDs for cron.add and cron.update, including failure destinations, using read-only configured-account metadata and targeted gateway regression tests.

## Reproduction Assessment

Yes. The source path is high-confidence: the CLI forwards --account to delivery.accountId, the protocol only shape-checks it, gateway validation remains channel-only, and cron.add persists after that incomplete guard.

## Solution Assessment

Yes. Gateway-side validation is the narrowest maintainable fix because it covers both CLI and RPC callers before jobs are stored; the implementation should avoid full channel plugin loading in this hot path.
