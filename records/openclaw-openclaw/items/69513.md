## Summary

Keep open. Current main still validates cron announce channels before persistence but does not validate explicit primary or failure-destination account IDs, and open implementation candidates already target the same bug.

## What This Changes

The issue requests create/update-time validation that rejects invalid cron announce account IDs against configured channel accounts before jobs are persisted.

## Best Possible Solution

Land focused gateway-side validation that rejects unknown explicit announce account IDs for cron.add and cron.update, including failure destinations, using read-only configured-account metadata and targeted gateway regression tests.

## Reproduction Assessment

Yes. The source path is high-confidence: the CLI forwards --account to delivery.accountId, the protocol only shape-checks it, gateway validation remains channel-only, and cron.add persists after that incomplete guard.

## Solution Assessment

Yes. Gateway-side validation is the narrowest maintainable fix because it covers CLI and RPC callers before persistence; it should avoid full channel plugin loading in this hot path.
