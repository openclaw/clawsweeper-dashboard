## Summary

Keep open. Current main and the latest stable release still only validate cron announce channels before persistence; explicit primary and failure-destination account IDs are still just shaped as strings, and open implementation candidates already target this bug.

## What This Changes

The issue asks OpenClaw to reject or warn on `openclaw cron create --announce --account <id>` when the account ID is not configured for the selected channel.

## Best Possible Solution

Land a focused gateway-side guard for `cron.add` and `cron.update` that validates explicit primary and failure-destination announce account IDs through lightweight configured-account metadata while preserving implicit/default-account behavior.

## Reproduction Assessment

Yes, by source inspection. The CLI forwards `--account` into `delivery.accountId`, protocol validation only shape-checks it, gateway validation only checks channels, and persistence happens after that incomplete guard.

## Solution Assessment

Yes on the boundary: gateway-side validation before persistence is the narrow maintainable fix. The implementation should avoid full channel plugin loading and use read-only configured-account metadata or an equivalent lightweight helper.
