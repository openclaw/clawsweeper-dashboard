## Summary

Keep open. Current main still validates cron announce channels before persistence but does not validate explicit primary or failure-destination account IDs, and an open implementation PR already targets this bug.

## What This Changes

The issue asks cron creation to reject invalid explicit announce delivery account IDs instead of persisting them.

## Best Possible Solution

Land a focused gateway-side guard for cron.add and cron.update that rejects unknown explicit announce account IDs for primary delivery and failure destinations via lightweight configured-account metadata while preserving implicit/default-account behavior.

## Reproduction Assessment

Yes. The source path is high-confidence: the CLI forwards --account to delivery.accountId, protocol validation only shape-checks it, gateway validation only checks channels, and cron.add persists after that incomplete guard.

## Solution Assessment

Yes. Gateway-side validation before persistence is the narrowest maintainable fix because it covers CLI and RPC callers while leaving delivery execution and account auto-resolution behavior unchanged.
