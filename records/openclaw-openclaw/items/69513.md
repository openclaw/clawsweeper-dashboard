## Label Justifications

- `P2`: This is a real gateway cron validation bug with limited blast radius, not an emergency runtime outage.
- `impact:message-loss`: Invalid cron delivery account IDs can cause scheduled announcements to be missed, dropped, or routed through the wrong account.

## Summary

This should stay open: current main still accepts the syntactically valid `delivery.accountId` field without checking configured channel accounts, and the linked fix work has not merged.

## What This Changes

Validate explicit cron announce delivery account IDs against configured channel accounts at cron create/update time instead of persisting unknown values.

## Best Possible Solution

Land or replace the focused gateway-side validation using read-only configured-account metadata for explicit primary and failure-destination account IDs, with regression coverage for cron.add, cron.update, and compatibility modes.

## Reproduction Assessment

Yes. I did not run the write-producing CLI command in this read-only review, but source inspection shows the CLI forwards `--account`, protocol validation only shape-checks it, gateway validation omits account membership, and runtime preserves the explicit value.

## Solution Assessment

Yes in principle. The maintainable fix is create/update-time gateway validation before persistence, but it should use read-only configured-account metadata rather than loading full channel plugin runtimes.
