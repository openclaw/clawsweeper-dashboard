- `impact:message-loss`: A bad cron announce accountId can cause scheduled messages to be missed, fail silently, or route through the wrong channel account.

## Summary

Keep open: current main and v2026.6.1 still validate cron announce channels but not explicit accountId membership, and the open fix PR at https://github.com/openclaw/openclaw/pull/69562 has not landed.

## What This Changes

Validate cron create/update announce delivery accountIds against configured channel accounts instead of storing unknown account IDs.

## Best Possible Solution

Land or replace the open fix PR with a focused gateway-side guard for cron.add and cron.update that validates explicit primary and failure-destination announce account IDs while preserving omitted-account fallback behavior.

## Reproduction Assessment

Yes, from source inspection rather than a live run. Current main forwards --account into delivery.accountId, shape-checks it only as a string, validates only the announce channel, and persists after that guard.

## Solution Assessment

Yes. Gateway validation before persistence is the right boundary, and the fix should cover cron.add, cron.update, primary delivery, and failureDestination without changing omitted-account fallback behavior.
