## Summary

Keep open. Current main still validates cron announce channels but not explicit primary or failure-destination account IDs before persistence, and open PR context already targets the same narrow bug.

## What This Changes

The issue asks for `cron create` and related cron write paths to reject explicit announce `accountId` values that are not configured for the selected channel.

## Best Possible Solution

Land one focused gateway-side validation change that rejects unknown explicit announce account IDs for `cron.add` and `cron.update`, including failure destinations, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. The reported CLI path maps directly to current source: the CLI forwards `--account`, the gateway validates only delivery shape/channel, and persistence follows that incomplete guard.

## Solution Assessment

Yes. Gateway-side validation against the selected channel's configured account IDs is the narrowest maintainable fix because it covers CLI and RPC callers before jobs are stored.
