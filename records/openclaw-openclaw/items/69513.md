## Summary

Keep open. Current main still validates cron announce delivery channels before persistence but does not validate explicit account IDs, and an open implementation PR already targets the same bug.

## What This Changes

The issue asks for `openclaw cron create --announce --account <id>` to reject or warn on account IDs that are not configured for the selected channel.

## Best Possible Solution

Land a focused gateway-side guard for cron.add and cron.update that validates explicit announce account IDs for primary and failure destinations against configured channel accounts while preserving omitted-account fallback behavior.

## Reproduction Assessment

Yes. The source path is high-confidence: the CLI forwards `--account` to `delivery.accountId`, protocol validation only shape-checks it, gateway validation checks only channels, and cron.add persists after that incomplete guard.

## Solution Assessment

Yes. Gateway validation before persistence is the right boundary, but the active PR should be reviewed or ported so the account check uses read-only configured-account metadata rather than full plugin loading.
