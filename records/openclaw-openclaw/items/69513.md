## Summary

Keep open. Current main still has the reported validation gap, and an open implementation PR already targets the same cron accountId bug, so the issue should remain open until that PR or a replacement lands.

## What This Changes

The issue requests create/update-time validation so cron announce jobs cannot persist an explicit delivery accountId that is not configured for the selected channel.

## Best Possible Solution

Land a focused gateway-side guard for cron.add and cron.update that validates explicit announce account IDs for primary and failure destinations through read-only configured-account metadata while preserving omitted-account fallback behavior.

## Reproduction Assessment

Yes. The source path is high-confidence: the CLI forwards --account to delivery.accountId, protocol validation only shape-checks it, gateway validation checks channels but not account membership, and cron.add persists after that guard.

## Solution Assessment

Yes. Gateway validation before persistence is the narrowest maintainable fix, but the active PR should be reviewed or ported so account checks use read-only configured-account metadata rather than full plugin loading.
