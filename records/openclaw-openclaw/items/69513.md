## Label Justifications

- `P2`: This is a real cron delivery validation bug with limited blast radius, not an emergency runtime outage.
- `impact:message-loss`: A bad accountId can leave scheduled announce delivery missed, silently failed, or routed through the wrong channel account.

## Summary

Keep open. Current main and the latest release still validate cron announce channels but not explicit accountId membership, and the open fix PR at https://github.com/openclaw/openclaw/pull/69562 has not landed and was previously flagged for proof and validation-boundary issues.

## What This Changes

The issue asks for cron create/update to reject or warn on explicit announce delivery account IDs that are not configured for the selected channel.

## Best Possible Solution

Land a focused gateway-side guard for cron.add and cron.update that validates explicit announce account IDs for primary and failure destinations through read-only configured-account metadata while preserving omitted-account fallback behavior.

## Reproduction Assessment

Yes, source-level. Current main forwards --account into delivery.accountId, only shape-checks accountId in the protocol schema, validates announce channels in the gateway, and persists the job after that incomplete guard.

## Solution Assessment

Yes. Gateway validation before persistence is the right boundary, but the implementation should use read-only configured-account metadata and cover cron.update plus failureDestination without changing omitted-account fallback behavior.
