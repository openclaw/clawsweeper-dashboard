## Summary

Keep open. Current main still has the reported validation gap: cron announce creation/update validates channels before persistence but not explicit account IDs, and an open implementation PR already targets the same bug.

## What This Changes

The issue requests creation-time validation that `openclaw cron create --announce --channel <id> --account <id>` only persists account IDs configured for the chosen channel.

## Best Possible Solution

Land a focused gateway-side guard for cron.add and cron.update that validates explicit announce account IDs for primary and failure destinations against configured channel accounts while preserving omitted-account fallback behavior.

## Reproduction Assessment

Yes. The source path is high-confidence: the CLI forwards --account to delivery.accountId, the protocol only shape-checks it, gateway validation checks only channels, and cron.add persists after that incomplete guard.

## Solution Assessment

Yes. Gateway validation before cron.add/update persistence is the narrowest maintainable boundary; the implementation should use read-only configured-account metadata and keep implicit/default-account behavior unchanged.
