## Summary

Keep open. Current main still validates cron announce channel configuration before persistence but does not validate explicit primary or failure-destination account IDs, and open implementation candidates already target the same bug.

## What This Changes

The issue requests creation/update-time validation so cron announce jobs reject account IDs that are not configured for the selected channel.

## Best Possible Solution

Land a focused gateway-side guard for `cron.add` and `cron.update` that rejects unknown explicit announce account IDs for both primary delivery and failure destinations, preferably by reviewing, porting, or replacing the linked open validation PR.

## Reproduction Assessment

Yes. The source path is high-confidence: the CLI forwards `--account` to `delivery.accountId`, protocol validation only shape-checks it, gateway validation only checks channels, and `cron.add` persists after that incomplete guard.

## Solution Assessment

Yes. Gateway-side validation is the narrowest maintainable fix because it covers CLI and RPC callers before persistence while preserving delivery execution and account auto-resolution behavior.
