## Summary

Keep open. Current main still does not fully satisfy the reported explicit `sessionKey="current"` path for an unstored channel-plugin requester, and open PRs #74142 and #72306 already explicitly target the same narrow fix, so this issue should stay open until one canonical PR lands or both are closed.

## What This Changes

Request that `session_status` resolve `sessionKey="current"` to the requester session for channel-plugin agents while preserving literal `current` key and sessionId precedence.

## Best Possible Solution

Land one canonical fix that keeps literal `current` key/sessionId precedence, then after those paths and gateway lookup miss, treats explicit `sessionKey="current"` as the requester’s own session with self-scoped visibility and a channel-style unstored requester regression test.

## Reproduction Assessment

Yes. The report includes concrete gateway logs, and current main has a static reproduction path: channel-style requester not present in the store plus explicit `sessionKey="current"` can miss store and gateway lookup, ignore the non-sessionId requester fallback, skip implicit fallback, and throw.

## Solution Assessment

Yes. The proposed final fallback after literal-current resolution is the narrowest maintainable solution because it preserves existing literal-session behavior while making the documented alias work for channel-plugin requesters.
