## Summary

Keep open. Current main still has the reported explicit `sessionKey="current"` gap for channel-style requester sessions, and open PRs #74142 and #72306 already target the same narrow fix path.

## What This Changes

Fix `session_status` so explicit `sessionKey="current"` resolves to the requester session for channel-plugin agents after literal `current` key/sessionId lookups miss.

## Best Possible Solution

Land one canonical fix from #74142 or #72306 that preserves literal `current` key/sessionId precedence, treats explicit `sessionKey="current"` as the requester session only after those lookups miss, keeps visibility scoped as self, and adds an unstored channel-style requester regression test.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: use a channel-style requester key with no stored requester/current session entry and call `session_status` with explicit `sessionKey="current"`; the resolver can fall through to `Unknown sessionKey: current`.

## Solution Assessment

Yes. The proposed final fallback after literal-current store and gateway lookup paths is the narrowest maintainable fix because it preserves documented literal-session precedence while making the advertised alias work for channel-plugin requesters.
