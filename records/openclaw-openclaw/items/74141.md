## Summary

Keep open. Current main still has the explicit sessionKey="current" gap for channel-plugin requester sessions, and two open PRs already target the narrow fix, so the issue should remain open until one canonical fix lands.

## What This Changes

The issue requests that session_status resolve explicit sessionKey="current" to the channel-plugin requester's own session instead of throwing when that requester is not yet persisted in the session store.

## Best Possible Solution

Land one canonical fix that preserves literal current key/sessionId precedence, treats explicit sessionKey="current" as the requester after store and gateway literal lookups miss, keeps visibility scoped as self, and adds an unstored channel-style requester regression test.

## Reproduction Assessment

Yes. The report includes concrete gateway logs, and current main has a static reproduction path: a channel-style requester key with no matching store entry plus explicit sessionKey="current" can fall through to the Unknown sessionKey throw.

## Solution Assessment

Yes. The proposed late fallback after the literal-current store and gateway lookup paths is the narrowest maintainable fix because it preserves existing literal-session precedence while making the documented alias universal.
