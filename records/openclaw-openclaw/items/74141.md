## Summary

Keep open. Current main still matches the reported `session_status` failure path for explicit `sessionKey="current"` from channel-plugin requesters without a stored session entry, and two open closing PRs already target the narrow fix.

## What This Changes

Explicit `session_status` calls with `sessionKey="current"` should resolve to the caller's own session for channel-plugin-driven agents instead of failing after store and gateway lookup misses.

## Best Possible Solution

Land one canonical implementation that preserves literal `current` key/sessionId precedence, adds a requester-owned fallback for explicit `sessionKey="current"` after store and gateway misses, treats that fallback as self for visibility, and covers an unstored channel-style requester in `src/agents/openclaw-tools.session-status.test.ts`.

## Reproduction Assessment

Yes. The reporter provided concrete gateway logs, and current main has a static reproduction path: an explicit `sessionKey="current"` can miss store/gateway lookup, produce a requester-key resolution that `session_status` ignores, skip the implicit fallback, and throw `Unknown sessionKey: current`.

## Solution Assessment

Yes. The proposed fallback after the literal-current paths is the narrowest maintainable direction because it keeps existing literal session precedence while making the documented current-session alias work for channel-plugin requesters.
