## Summary

Keep open. Current main still advertises `sessionKey="current"` as the current-session alias, but the `session_status` resolution path can still ignore the requester-key result for channel-style agents and throw after the explicit-current lookup misses; open PRs #74142 and #72306 already target the same narrow fix.

## What This Changes

Fix `session_status` so explicit `sessionKey="current"` resolves to the requester session for channel-plugin agents after literal `current` key/sessionId lookups miss.

## Best Possible Solution

Land one canonical fix, likely from #74142 or #72306, that preserves literal `current` key/sessionId precedence, treats explicit `sessionKey="current"` as the requester session only after those lookups miss, keeps visibility checks scoped as self, and adds an unstored channel-style requester regression test.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction path: channel-style requester key, no stored requester/current session entry, and explicit `sessionKey="current"`; the code can skip the key-based requester resolution and throw `Unknown sessionKey: current`.

## Solution Assessment

Yes. The proposed final fallback after literal-current store and gateway lookup paths is the narrowest maintainable fix because it preserves documented literal-session precedence while making the advertised alias work for channel-plugin requesters.
