## Summary

Keep open. Current main still has the reported explicit sessionKey="current" gap for channel-plugin requester keys, and two open PRs already explicitly target the same narrow fix path.

## What This Changes

The issue asks session_status to resolve explicit sessionKey="current" to the requester session for channel-plugin agents instead of throwing and causing retry latency.

## Best Possible Solution

Land one canonical fix that preserves literal current key/sessionId precedence, treats explicit sessionKey="current" as the requester’s own session after store and gateway lookups miss, keeps visibility checks scoped as self, and adds an unstored channel-style requester regression test.

## Reproduction Assessment

Yes. The issue includes concrete gateway logs, and current main has a static reproduction path: call session_status with a channel-style requester key, no stored requester/current session entry, and explicit sessionKey="current"; the resolver can still fall through to Unknown sessionKey.

## Solution Assessment

Yes. The proposed final fallback after literal-current lookup paths is the narrowest maintainable fix because it preserves documented literal-session precedence while making the advertised alias work for channel-plugin requesters.
