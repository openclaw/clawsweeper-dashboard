## Summary

Keep open. Current main still has the explicit `sessionKey="current"` gap for unstored channel-plugin requester sessions, and open PRs #74142 and #72306 already target the same narrow fix path.

## What This Changes

The issue requests that `session_status` resolve explicit `sessionKey="current"` to the requester session for channel-plugin agents after literal `current` lookups miss.

## Best Possible Solution

Land one canonical fix from #74142 or #72306 that preserves literal `current` key/sessionId precedence, treats explicit `current` as the requester only after those lookups miss, keeps visibility scoped as self, and adds the unstored channel-style requester regression test.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: call `session_status` with explicit `sessionKey="current"` from a channel-style requester that has no stored requester/current entry, and the resolution chain can reach `Unknown sessionKey: current`.

## Solution Assessment

Yes. The proposed final fallback after literal-current store and gateway lookup paths is the narrowest maintainable fix because it preserves documented literal-session precedence while making the advertised alias work for channel-plugin requesters.
