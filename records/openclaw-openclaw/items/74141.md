## Summary

Keep open. Current main still has the reported `session_status` mismatch for explicit `sessionKey="current"` when the requester is a channel-plugin session without a stored session entry, and two open PRs already target the same fix path, so this issue should remain open until one canonical PR lands or both are closed.

## What This Changes

The issue asks for `session_status` to resolve explicit `sessionKey="current"` to the requester session for channel-plugin agents instead of throwing `Unknown sessionKey: current`.

## Best Possible Solution

Land one canonical fix that preserves literal `current` key/sessionId precedence, adds a final requester-owned fallback for explicit `sessionKey="current"` when store and gateway lookup miss, keeps visibility checks treating that fallback as self, and includes a focused channel-style requester regression test.

## Reproduction Assessment

Yes. The report includes concrete channel-plugin logs, and current main has a static reproduction path: explicit `sessionKey="current"` can miss store/gateway lookup, return a non-sessionId requester resolution that `session_status` ignores, and then throw `Unknown sessionKey: current`.

## Solution Assessment

Yes. The proposed fallback after literal `current` lookup is the narrowest maintainable fix because it preserves existing literal-session precedence while making the documented current-session alias work for channel-plugin requesters.
