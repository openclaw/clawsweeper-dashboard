## Summary

Keep open. Current main still has the explicit `sessionKey="current"` channel-plugin mismatch, and two open closing PRs already target the narrow fix path, so this issue should remain open until one canonical PR lands or both are closed.

## What This Changes

Request `session_status` to resolve explicit `sessionKey="current"` to a channel-plugin requester’s own session after literal lookups fail, avoiding retry latency.

## Best Possible Solution

Land one canonical fix, preferably by reviewing and choosing between #74142 and #72306, that preserves literal `current` key/sessionId precedence, adds a final requester-owned fallback for explicit `sessionKey="current"`, treats that fallback as self for visibility checks, and includes a focused channel-style requester regression test.

## Reproduction Assessment

Yes. The report includes concrete channel-plugin logs, and current main has a static reproduction path: explicit `sessionKey="current"` can miss store/gateway lookup, return a non-sessionId requester resolution that `session_status` ignores, and then throw `Unknown sessionKey: current`.

## Solution Assessment

Yes. The proposed fallback after literal `current` lookup is the narrowest maintainable fix because it preserves existing literal-session precedence while making the documented current-session alias work for channel-plugin requesters.
