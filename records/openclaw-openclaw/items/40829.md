## Summary

Keep open: the reported media behavior still appears real on current main, but this PR cannot merge as-is because it targets legacy files that are no longer tracked and its PNG fast path needs validation before preserving original bytes.

## What This Changes

The PR modifies legacy web media loader/tests to preserve under-cap PNGs and rename JPEG-converted outputs to `.jpg`.

## Best Possible Solution

Port the useful behavior to `src/media/web-media.ts` with decoder-validated under-cap PNG preservation, all-JPEG filename normalization, and active shared/WhatsApp regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main still sends non-alpha PNG optimization through the JPEG path and the active WhatsApp test expects the tiny wrong-extension PNG to return `image/jpeg`. I did not run tests because this review is read-only.

## Solution Assessment

No. The requested behavior is reasonable, but this branch targets obsolete files and the early return should validate PNG bytes before preserving the original buffer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Port the PNG preservation to the active loader:** `src/web/media.ts:213-220`
  - body: Current main no longer tracks `src/web/media.ts`; WhatsApp reaches `loadWebMedia` through the plugin SDK facade into `src/media/web-media.ts`. This hunk needs to be rebased or replaced in the active loader before it can affect current runtime behavior.
  - confidence: 0.93
