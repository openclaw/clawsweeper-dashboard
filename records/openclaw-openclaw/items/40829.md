## Summary

Keep this PR open. Current main still has the reported behavior on the active shared media loader, while the PR branch edits the old `src/web/media.ts` path and its early-return approach still needs PNG payload validation before it is safe to rebase.

## What This Changes

The PR branch changes the web media loader to preserve under-cap PNG images as PNG and to rename any JPEG-optimized output to a `.jpg` filename.

## Best Possible Solution

Rebase or replace the PR on `src/media/web-media.ts`, preserving only validated under-cap PNG bytes, normalizing filenames for every JPEG output, and covering the shared loader plus WhatsApp wrapper with focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows non-alpha PNG input still reaches `optimizeImageToJpeg`, and the active WhatsApp-facing test still expects the tiny PNG fixture to return `image/jpeg`.

## Solution Assessment

No. The intended behavior is correct, but the branch is not the best current fix because it targets the old loader path and returns under-cap PNG bytes before proving the payload is a valid PNG.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate PNG bytes before preserving under-cap images:** `src/web/media.ts:216-222`
  - body: This early return trusts PNG classification from metadata or filename, but MIME detection can fall back to the extension. A mislabeled under-cap `.png` would now be returned as `image/png` without the decode/re-encode failure the old path provided, pushing invalid bytes downstream.
  - confidence: 0.88
