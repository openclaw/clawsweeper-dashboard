## Summary

Keep open: the media behavior still appears unsolved on current main, but this PR is not mergeable as-is because it targets deleted legacy files, lacks required real behavior proof, and adds an unvalidated PNG pass-through.

## What This Changes

The PR preserves already-under-cap PNG output in `loadWebMedia`, normalizes JPEG-converted filenames to `.jpg`, and updates legacy media tests.

## Best Possible Solution

Port the behavior to `src/media/web-media.ts` with decoder-validated under-cap PNG preservation, all-JPEG filename normalization, active shared/WhatsApp regression coverage, and real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows the active shared loader still sends non-alpha PNGs through JPEG optimization, and the active WhatsApp regression still expects the tiny wrong-extension PNG case to return `image/jpeg`.

## Solution Assessment

No. The intended fix direction is reasonable, but the PR must target the active shared loader and validate PNG bytes before preserving original data.
