## Summary

Keep open. The PR captures a valid media-quality bug, but it is not mergeable as-is: current main no longer uses the legacy `src/web/media.ts` path, the active shared loader still converts non-alpha under-cap PNGs to JPEG, and the proposed early return would bypass image decode validation before returning `image/png`.

## What This Changes

The PR branch changes the legacy web media loader to preserve under-cap PNG images and rename JPEG-optimized outputs to `.jpg`, with matching legacy loader tests.

## Best Possible Solution

Port the useful behavior to `src/media/web-media.ts`: preserve under-cap PNG bytes only after decode-backed validation, normalize filenames for every JPEG-optimized output, update shared and WhatsApp-facing tests, and add the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows non-alpha PNG input still reaches JPEG optimization, and the active WhatsApp regression test still expects the tiny PNG fixture to return `image/jpeg`.

## Solution Assessment

No. The requested behavior is worth fixing, but this PR is not the best current solution because it targets obsolete files and returns PNG-classified bytes before validation; the safer fix belongs in the active shared loader.
