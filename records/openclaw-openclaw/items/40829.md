## Summary

Keep open: current main still has the under-cap PNG-to-JPEG behavior, but this PR targets the now-deleted `src/web` paths and its fast path would bypass current image decode validation for malformed PNG-labeled inputs.

## What This Changes

The branch changes web media image optimization so under-cap PNGs are returned as PNG and any JPEG-formatted optimized output gets a `.jpg` filename, with matching media tests.

## Best Possible Solution

Port the fix to the current shared web media loader, preserve under-cap PNGs only after validating the buffer as a real PNG, normalize JPEG output filenames, and add regression coverage for preservation plus malformed PNG rejection.

## Reproduction Assessment

Yes from source inspection: current main still sends non-alpha PNG images through JPEG optimization, and the current WhatsApp test asserts `image/jpeg` for the under-cap PNG case. I did not run the test suite in this read-only review.

## Solution Assessment

No, not as written. The intended behavior is narrow and useful, but the implementation should be ported to `src/media/web-media.ts` and should validate PNG bytes before returning the original under-cap buffer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate PNG bytes before preserving them:** `src/web/media.ts:218-225`
  - body: The new fast path returns any buffer classified as PNG by metadata or filename before decoding it. Because MIME detection can fall back to a `.png` extension when sniffing finds no magic bytes, malformed under-cap files would now be returned as `image/png` instead of failing during image processing; keep the preservation path behind a real PNG validation/decode check.
  - confidence: 0.88
