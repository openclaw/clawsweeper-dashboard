## Summary

Keep open: the reported schema rejection is real from source inspection, but the patch currently accepts structured-output modes that the `/v1/responses` runtime does not honor, and the external after-fix proof only exercises schema parsing rather than the patched HTTP endpoint.

## What This Changes

The PR adds a `text` field to the `/v1/responses` request schema, adds schema unit tests for `text.format` and `verbosity`, and records a changelog fix.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` shape now, reject or implement structured-output formats before advertising support for them, and add focused schema/HTTP coverage plus redacted patched-gateway proof.

## Reproduction Assessment

Yes. Current main is source-reproducible with high confidence: the strict schema lacks `text`, and the HTTP handler returns 400 on schema parse failure; I did not execute the endpoint because this was a read-only review.

## Solution Assessment

No. The narrow maintainable fix is to accept the default `text` request shape needed by SDK clients while continuing to reject `json_schema` and `json_object` until the runtime can enforce those output contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
