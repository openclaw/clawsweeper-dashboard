## Summary

Keep this PR open because the reported `/v1/responses` schema bug is still present on current `main`, but the branch should not merge as-is: it accepts structured `text.format` modes that the runtime does not enforce and still lacks patched-gateway real behavior proof.

## What This Changes

This PR adds a top-level `text` object to the `/v1/responses` request schema, adds schema tests, and records a changelog fix.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` request shape, reject unsupported structured formats until they are enforced, and keep focused schema/HTTP regression coverage.

## Reproduction Assessment

Yes, at source level: current `main` validates `/v1/responses` with a strict schema that lacks `text`, then returns HTTP 400 on parse failure. The PR body also includes a before-fix curl capture showing that same failure.

## Solution Assessment

No. Accepting the SDK-default `text` shape is the right narrow fix, but accepting `json_schema` and `json_object` before runtime enforcement is not the safest implementation.
