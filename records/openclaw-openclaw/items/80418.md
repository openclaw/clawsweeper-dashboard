## Summary

Keep open: the reported `/v1/responses` schema bug is real, but this PR still accepts unsupported structured-output formats without runtime enforcement and lacks after-fix proof from the patched gateway or n8n flow.

## What This Changes

The PR widens `/v1/responses` request validation to accept a top-level `text` object, adds schema tests, and records a changelog fix for OpenAI SDK 6.x parity.

## Best Possible Solution

Accept only the SDK-default `text` shape needed for OpenAI SDK 6.x parity, reject structured output formats until `/v1/responses` can enforce them, and keep focused schema/HTTP regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows a strict request schema without `text`, followed by an HTTP 400 parse-failure path in the `/v1/responses` handler; I did not execute the gateway because this review is read-only.

## Solution Assessment

No. Schema acceptance for the SDK-default text shape is the right narrow fix, but accepting `json_schema` and `json_object` before runtime enforcement is not the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reject structured text formats until runtime enforces them:** `src/gateway/open-responses.schema.ts:228-235`
  - body: The schema now accepts `text.format.type` values `json_schema` and `json_object`, but the handler never reads `payload.text` before dispatching the agent. Callers requesting structured output would now receive HTTP 200 with ordinary text instead of guaranteed JSON/schema output or a clear unsupported-parameter error, so this should reject unsupported formats or implement the runtime contract.
  - confidence: 0.89
