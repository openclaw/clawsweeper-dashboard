## Summary

This PR addresses a real `/v1/responses` compatibility bug, but it should stay open because the diff also accepts structured output formats that the runtime does not honor and the external real behavior proof is still insufficient.

## What This Changes

This PR widens `/v1/responses` request validation to accept a `text` object with `format` and `verbosity`, adds schema tests, and records a changelog fix.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` shape on `/v1/responses`, keep unsupported structured formats rejected until runtime can enforce them, and cover the HTTP/schema path with focused regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: POST `/v1/responses` with a top-level `text` field on current main reaches the strict schema, which lacks `text`, and the handler returns HTTP 400 on parse failure. I did not execute the endpoint because this was a read-only review.

## Solution Assessment

No. Accepting the SDK-default `text` shape is the right narrow fix, but accepting `json_schema` and `json_object` before the runtime enforces those output contracts is not the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject structured text formats until runtime enforces them:** `src/gateway/open-responses.schema.ts:228-240`
  - body: The new schema accepts `text.format.type` values `json_schema` and `json_object`, but the handler never reads `payload.text` before dispatching the agent. Callers requesting structured output would now receive a 200 with ordinary text instead of guaranteed JSON/schema output or a clear unsupported-parameter error, so this fix should accept only the supported `text` format or wire these modes into runtime enforcement.
  - confidence: 0.89
