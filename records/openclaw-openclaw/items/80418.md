## What This Changes

The PR adds a `text` field to the `/v1/responses` request schema, adds schema tests for accepted/rejected `text` payloads, and edits `CHANGELOG.md`.

## Best Possible Solution

Land a narrow Gateway schema/test fix that accepts the SDK-default `text` request while rejecting or implementing non-default text controls, removes the changelog edit, and includes redacted patched-Gateway proof.

## Reproduction Assessment

Yes at source level: current `main` has a strict `/v1/responses` schema without top-level `text`, and the handler returns HTTP 400 before dispatch on schema failure. I did not run a live gateway because this review is read-only.

## Solution Assessment

No, not as submitted. Accepting the SDK-default `text` object is the right narrow fix, but accepting structured-output and verbosity controls without forwarding or rejecting them is not the safest API contract.
