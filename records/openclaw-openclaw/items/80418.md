## Summary

Keep open: this PR targets a real `/v1/responses` compatibility bug, but the patch accepts structured output formats that the handler still ignores and it lacks after-fix proof from the patched gateway path.

## What This Changes

This PR adds a `text` object to the `/v1/responses` request schema, adds focused schema tests, and records a changelog fix.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` shape, and either reject unsupported structured formats with a clear 400 or wire them through the existing `responseFormat` runtime path with HTTP coverage.

## Reproduction Assessment

Yes at source level: POSTing a `/v1/responses` body with top-level `text` reaches a strict schema that lacks that key, and the handler returns HTTP 400 on parse failure. I did not execute the endpoint because this review is read-only.

## Solution Assessment

No. Accepting the SDK-default text shape is the right narrow fix, but accepting `json_schema` and `json_object` before the Responses handler enforces or forwards those output contracts is not the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject structured text formats until runtime enforces them:** `src/gateway/open-responses.schema.ts:228-236`
  - body: The new schema accepts `text.format.type` values `json_schema` and `json_object`, but `handleOpenResponsesHttpRequest` never reads `payload.text` or forwards a `responseFormat` into `streamParams`. Callers requesting structured output would now receive a 200 with ordinary text instead of guaranteed JSON/schema output or a clear unsupported-parameter error, so keep this schema widening to supported text output or wire these modes into runtime enforcement.
  - confidence: 0.89
