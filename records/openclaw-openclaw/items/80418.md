## Summary

Keep open: current main still rejects the reported valid `text` request shape, so the PR targets a real gateway compatibility bug, but the patch is not merge-ready because it accepts unimplemented structured-output modes and still lacks after-fix proof through the patched gateway path.

## What This Changes

The PR adds a top-level `text` object to the `/v1/responses` request schema, schema tests for that object, and a changelog fix entry.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` compatibility shape, reject unsupported structured formats until runtime can enforce them, and cover the gateway/schema path with focused regression proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: POST `/v1/responses` with a top-level `text` field reaches the strict schema that lacks `text`, then the handler returns HTTP 400 before agent dispatch.

## Solution Assessment

No. The narrow schema acceptance for default text is the right fix direction, but accepting `json_schema` and `json_object` before runtime enforcement is not the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject structured text formats until they are enforced:** `src/gateway/open-responses.schema.ts:227-239`
  - body: The schema now accepts `text.format.type` values `json_schema` and `json_object`, but the handler never reads `payload.text` before dispatching the agent. Callers requesting structured output would get a 200 with ordinary text instead of guaranteed JSON/schema output or a clear unsupported-parameter error, so keep these formats rejected until they are enforced at runtime.
  - confidence: 0.89
