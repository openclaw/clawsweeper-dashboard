## Summary

Keep open: current main still rejects the SDK-default `/v1/responses` `text` field, so the central bug is real, but this PR is not merge-ready because it accepts unsupported structured-output and verbosity controls without handler support and still lacks after-fix patched-Gateway proof.

## What This Changes

This PR adds a top-level `text` object to the `/v1/responses` request schema, adds schema-only tests for text formats and verbosity, and edits `CHANGELOG.md`.

## Best Possible Solution

Land a narrow Gateway schema/test fix that accepts the SDK-default `text` request while either rejecting or implementing non-default text controls, removes the changelog edit, and includes redacted patched-Gateway proof.

## Reproduction Assessment

Yes, at source level: current main's strict `/v1/responses` schema has no top-level `text`, and the handler returns HTTP 400 on schema failure before dispatch. I did not run the Gateway because this cleanup review is read-only.

## Solution Assessment

No, not as submitted. Accepting the SDK-default `text` object is the right narrow bug fix, but accepting structured-output and verbosity controls without forwarding or rejecting them is not the safest API contract.
