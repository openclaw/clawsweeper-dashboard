## Summary

Keep open: current main still rejects the SDK-default `text` request shape, but this PR is not merge-ready because it accepts unsupported structured-output modes and lacks after-fix proof from the patched gateway path.

## What This Changes

The PR adds a top-level `text` object to `/v1/responses` request validation, adds schema tests, and records a changelog fix.

## Best Possible Solution

Land a narrow schema fix that accepts the SDK-default `text.format: { type: "text" }` request shape, rejects unsupported structured formats until runtime support exists, and proves the patched endpoint path.

## Reproduction Assessment

Yes, at source level: POST `/v1/responses` with a top-level `text` field reaches the strict schema that lacks `text`, and the handler returns 400 on parse failure. I did not execute the endpoint because this was a read-only review.

## Solution Assessment

No. Accepting the SDK-default `text` shape is the right narrow fix, but accepting `json_schema` and `json_object` before the runtime enforces those output contracts is not the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject structured text formats until runtime enforces them:** `src/gateway/open-responses.schema.ts:228-240`
  - body: The schema accepts `text.format.type` values `json_schema` and `json_object`, but `handleOpenResponsesHttpRequest` never reads `payload.text` before non-streaming or streaming dispatch. A caller requesting structured output would now get a successful ordinary-text response instead of guaranteed JSON/schema output or a clear unsupported-parameter error, so keep those modes rejected until runtime support lands.
  - confidence: 0.89
