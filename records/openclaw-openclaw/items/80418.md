## Summary

Keep open: the reported compatibility bug is real from source and shipped-release inspection, but this patch still accepts structured output modes that the runtime does not enforce and lacks after-fix proof through the patched gateway path.

## What This Changes

This PR widens the `/v1/responses` request schema to accept a `text` object, adds schema tests, and records a changelog fix.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` request shape, keep unsupported structured formats rejected until runtime can enforce them, and require redacted patched-gateway proof before merge.

## Reproduction Assessment

Yes at source level: current main and `v2026.5.7` use a strict `CreateResponseBodySchema` without `text`, and the handler returns HTTP 400 on parse failure. I did not execute the endpoint because this review is read-only.

## Solution Assessment

No. Accepting the SDK-default `text` shape is the right narrow fix, but accepting `json_schema` and `json_object` before the runtime enforces those output contracts is not the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject structured text formats until runtime enforces them:** `src/gateway/open-responses.schema.ts:228-238`
  - body: The new schema accepts `text.format.type` values `json_schema` and `json_object`, but the handler never reads `payload.text` before dispatching the agent. Callers requesting structured output would now receive a 200 with ordinary text instead of guaranteed JSON/schema output or a clear unsupported-parameter error, so this fix should accept only the supported `text` format or wire these modes into runtime enforcement.
  - confidence: 0.89
