## Summary

Keep this PR open, but it should not merge as-is: the core SDK-default `text` compatibility bug is real, while the patch also accepts structured output formats that the gateway does not enforce and still lacks after-fix proof from the patched endpoint.

## What This Changes

This PR widens `/v1/responses` request validation to accept a top-level `text` object, adds schema tests, and records a changelog fix.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` shape, reject unsupported structured formats until runtime enforces them, and cover the HTTP/schema path with focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: POST `/v1/responses` with a top-level `text` field reaches the strict schema on current main and returns HTTP 400 before dispatch.

## Solution Assessment

No. Accepting the SDK-default `text` object is the right narrow fix, but accepting `json_schema` and `json_object` before runtime enforcement is not the best implementation.
