## Summary

Keep open: the reported `/v1/responses` schema bug is valid and the PR targets the right gateway surface, but it should not merge as written because it accepts structured output formats that the runtime does not enforce and still lacks after-fix proof from the patched gateway path.

## What This Changes

The PR widens `/v1/responses` request validation to accept a top-level `text` object, adds schema tests for that contract, and records a changelog fix.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` request shape, keep unsupported structured formats rejected until runtime enforcement exists, and require patched-gateway proof before merge.

## Reproduction Assessment

Yes, at source level: POST `/v1/responses` with a top-level `text` field on current `main` reaches a strict schema that lacks `text`, then returns HTTP 400 on parse failure. I did not execute the endpoint because this review is read-only.

## Solution Assessment

No. Accepting the SDK-default `text` shape is the right narrow fix, but accepting `json_schema` and `json_object` before the runtime enforces those output contracts is not the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Reject structured text formats until runtime enforces them:** `src/gateway/open-responses.schema.ts:228-240`
