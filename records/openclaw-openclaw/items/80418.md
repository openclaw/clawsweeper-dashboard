## Summary

Keep this PR open: the core bug is real and current `main` still rejects the OpenAI Responses `text` field, but the patch also accepts structured output modes that the runtime explicitly ignores, and the external-PR real behavior proof still stops at patched schema parsing rather than a patched gateway or n8n round trip.

## What This Changes

The PR widens `/v1/responses` request validation to accept a top-level `text` object, adds schema tests for that contract, and adds a changelog entry.

## Best Possible Solution

Land a narrow gateway schema fix that accepts the SDK-default `text.format.type: "text"` request shape, preserves strict rejection for unimplemented structured formats, drops the changelog edit, and includes patched-gateway proof plus focused schema or HTTP coverage.

## Reproduction Assessment

Yes. Current `main` has a strict `CreateResponseBodySchema` without a `text` key, and the PR body provides a before-fix curl failure against `/v1/responses`; I did not run a live gateway in this read-only review.

## Solution Assessment

No, not as submitted. Accepting the default text format is the right narrow fix, but accepting `json_schema` and `json_object` before runtime enforcement creates an OpenAI-compatible API mismatch.
