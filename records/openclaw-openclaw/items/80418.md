## Summary

Keep open: the reported `/v1/responses` compatibility bug is valid, but this patch currently accepts structured output modes that the runtime does not enforce and still lacks after-fix proof from a patched gateway or n8n flow.

## What This Changes

The PR widens the `/v1/responses` request schema to accept `text`, adds colocated schema tests, and records the compatibility fix in the changelog.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` shape, keep unsupported structured formats rejected until runtime can enforce them, and add patched-gateway proof for the HTTP path.

## Reproduction Assessment

Yes, at source level: POSTing a top-level `text` field reaches the strict current-main schema without that key, then the handler returns HTTP 400 on parse failure. I did not execute the endpoint because this was a read-only review.

## Solution Assessment

No. Adding support for the SDK-default `text` object is the right narrow fix, but accepting `json_schema` and `json_object` before the runtime enforces those output contracts is not the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reject unsupported structured text formats:** `src/gateway/open-responses.schema.ts:227-239`
  - body: The new schema accepts `json_schema` and `json_object`, but `handleOpenResponsesHttpRequest` never reads `payload.text` before returning normal output text. A caller requesting structured output would now get a 200 with unconstrained text instead of guaranteed JSON/schema output or a clear unsupported-parameter error, so this should reject those modes or enforce them at runtime.
  - confidence: 0.89
