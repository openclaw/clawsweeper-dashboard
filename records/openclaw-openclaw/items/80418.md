## Summary

Keep open: current main still rejects the SDK-default `text` request shape, but this PR should not merge as-is because it also accepts structured output formats that the `/v1/responses` runtime does not enforce and it still lacks after-fix proof through the patched gateway path.

## What This Changes

This branch adds a `text` object to `CreateResponseBodySchema`, adds schema tests for that request shape, and records a changelog fix for OpenAI SDK 6.x / LangChain clients.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` request shape, reject unsupported structured formats until runtime can enforce them, and keep focused schema or HTTP regression coverage.

## Reproduction Assessment

Yes, at source level: current main validates `/v1/responses` bodies with a strict schema that lacks `text`, and the handler returns HTTP 400 on schema parse failure. I did not run the gateway because this was a read-only review.

## Solution Assessment

No. Accepting the SDK-default `text` shape is the right narrow fix, but accepting `json_schema` and `json_object` before the runtime enforces those output contracts is not the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject unsupported structured text formats:** `src/gateway/open-responses.schema.ts:224-239`
  - body: The new schema accepts `text.format.type` values `json_schema` and `json_object`, but the handler never reads `payload.text` before dispatching the agent. Callers requesting structured output would now receive a 200 with ordinary text instead of guaranteed JSON/schema output or a clear unsupported-parameter error, so this should accept only the supported text format or wire these modes into runtime enforcement.
  - confidence: 0.89
