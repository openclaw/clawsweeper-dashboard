## Summary

Keep open: the reported compatibility bug is real and the patch is directionally useful, but this revision still accepts unsupported structured-output formats and lacks after-fix proof from the patched endpoint or n8n path.

## What This Changes

The PR adds a `text` field to the `/v1/responses` request schema, adds colocated schema tests, and records a gateway changelog fix.

## Best Possible Solution

Accept the SDK-default `text.format: { type: "text" }` request shape on `/v1/responses`, reject unsupported structured formats until runtime enforcement exists, and keep focused schema/HTTP regression coverage.

## Reproduction Assessment

Yes, at source level: current main's strict `CreateResponseBodySchema` lacks top-level `text`, and `handleOpenResponsesHttpRequest` returns HTTP 400 on schema parse failure. I did not execute the endpoint in this read-only review, so this is source-reproducible rather than live reproduced.

## Solution Assessment

No. Accepting the SDK-default `text` object is the right narrow fix, but accepting `json_schema` and `json_object` before runtime enforces those output contracts is not the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject structured text formats until runtime enforces them:** `src/gateway/open-responses.schema.ts:228-240`
  - body: The schema now accepts `json_schema` and `json_object`, but `handleOpenResponsesHttpRequest` never reads `payload.text` before dispatching the agent. Clients requesting structured output can get a 200 with ordinary text instead of guaranteed JSON/schema output or a clear unsupported-parameter error, so this should accept only the supported text format or wire these modes into runtime enforcement.
  - confidence: 0.89
