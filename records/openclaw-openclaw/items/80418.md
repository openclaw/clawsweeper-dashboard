## Summary

Keep open: current main and the latest release still reject the reported `text` request field, so the compatibility fix is still needed, but this PR is not merge-ready because after-fix real behavior proof is missing and the schema accepts structured formats that the Gateway runtime does not enforce.

## What This Changes

The PR adds a `text` object to the `/v1/responses` request schema, adds schema tests for that field, and edits `CHANGELOG.md`.

## Best Possible Solution

Land a narrow Gateway schema/test fix that accepts the SDK default `text` shape while either rejecting unsupported structured formats or wiring them into the existing Responses `responseFormat` runtime path, with redacted patched-Gateway proof before merge.

## Reproduction Assessment

Yes. Current main has a strict `/v1/responses` request schema without `text`, and the handler returns HTTP 400 on schema parse failure before dispatch.

## Solution Assessment

No, not as submitted. Accepting the SDK-default `text` object is the right narrow fix, but accepting structured formats without enforcement is not the safest API contract.
