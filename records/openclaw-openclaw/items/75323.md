## Summary

Keep open. Current main and v2026.5.2 still dispose stdio MCP runtimes by closing the transport and signaling the wrapper process tree only; there is no Docker container identity, labeling, cgroup inspection, or container removal path, so this remains a real Docker-wrapper lifecycle gap distinct from the generic process leak closed in #65694.

## What This Changes

The issue asks OpenClaw to clean up Docker containers launched by Docker-wrapped stdio MCP servers when parallel specialist sessions end.

## Best Possible Solution

Add session-owned container tracking or labeling for Docker-wrapped stdio MCP servers, then remove only containers proven to belong to that MCP session while keeping process-tree signaling as the generic fallback.

## Reproduction Assessment

No. The report includes concrete Docker output and source inspection shows only wrapper PID/process-group cleanup, but this review did not live-run a Docker-wrapped UniFi MCP server under parallel sessions_spawn on current main.

## Solution Assessment

No for the suggested immediate proc/image-based docker rm approach. The safer maintainable fix should establish OpenClaw-owned container identity, such as labels or recorded container ids, before deleting containers during MCP runtime disposal.
