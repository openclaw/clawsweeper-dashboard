## Summary

Keep open. Current main has the generic MCP stdio process-tree cleanup, but the inspected source still only disposes MCP clients/transports and kills the wrapper process group; there is no Docker/container ownership tracking or cleanup for containers launched indirectly by npm/npx MCP wrappers, and the existing Docker cleanup smoke covers a plain Node stdio probe rather than this container-spawning case.

## What This Changes

The issue reports that Docker containers launched by Docker-wrapped stdio MCP servers remain orphaned after parallel `sessions_spawn` subagent runs end.

## Best Possible Solution

Design a precise MCP-launched-container ownership contract, such as wrapper-provided container IDs or OpenClaw-applied labels, and clean only those session-owned containers during MCP runtime disposal while keeping process-tree teardown as the generic stdio fallback.

## Reproduction Assessment

Unclear. The report gives a plausible path with parallel `sessions_spawn` runs and a Docker-wrapped UniFi stdio MCP server, and source inspection confirms no Docker-aware cleanup, but this review did not live-reproduce the container leak.

## Solution Assessment

No, the immediate suggested broad Docker sweep is not the safest fix. The maintainable solution should track OpenClaw/session-owned containers explicitly and avoid deleting containers by image name or inferred ancestry alone.
