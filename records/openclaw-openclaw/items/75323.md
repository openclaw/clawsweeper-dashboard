## Summary

Keep open. Current main still disposes one-shot subagent MCP runtimes through stdio transport close and generic process-tree signaling, with no Docker/container ownership tracking for containers launched indirectly by npm/npx MCP wrappers, so the reported Docker-wrapper leak remains plausible and distinct from the earlier generic MCP process leak.

## What This Changes

Report that Docker containers launched indirectly by stdio MCP wrappers can survive cleanup after parallel one-shot subagent sessions.

## Best Possible Solution

Add an ownership-tracked MCP container cleanup contract: record or require session-scoped container identifiers or labels for containers created by stdio MCP wrappers, remove only those session-owned containers during MCP runtime disposal, and keep the generic process-tree cleanup as the fallback for ordinary child processes.

## Reproduction Assessment

Unclear. The issue provides a concrete Docker-wrapper scenario and current source inspection confirms no container-aware cleanup path, but this read-only review did not execute the UniFi MCP wrapper under parallel subagent spawns.

## Solution Assessment

No. The broad suggested immediate cleanup by parsing process state and force-removing containers is not the safest default; the maintainable fix needs explicit OpenClaw ownership of the containers being removed.
