## Summary

Keep open. Current main still only disposes MCP stdio transports through generic process-tree signaling and session-runtime teardown; I found no Docker/container ownership tracking or cleanup path for containers launched indirectly by npm/npx MCP wrappers during parallel subagent runs, so the narrower Docker-wrapped leak is not resolved by the prior generic MCP leak work.

## What This Changes

The issue reports that Docker-backed stdio MCP servers launched through wrapper commands can survive session cleanup during parallel specialist-agent runs.

## Best Possible Solution

Keep the issue open and design an ownership-aware MCP container lifecycle contract: record or require OpenClaw-owned labels/metadata for containers launched for a specific MCP session, then remove only those session-owned containers during runtime disposal while retaining process-tree cleanup as the generic stdio fallback.

## Reproduction Assessment

Unclear. The issue provides concrete observed containers and the source path confirms there is no Docker-aware cleanup, but this review did not run the reported UniFi/npx wrapper scenario live.

## Solution Assessment

No, the current generic process-tree cleanup is not the best complete solution for Docker-wrapped MCP servers. The maintainable direction is explicit OpenClaw-owned container tracking/labeling rather than broad post-hoc Docker sweeps.
