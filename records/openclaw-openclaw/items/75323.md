## Summary

Keep open. Current main still disposes stdio MCP servers through `transport.close()` and `killProcessTree(pid)` only, with no container id, Docker label, cgroup, or ownership tracking for containers launched indirectly by npm/npx wrappers; the prior #65694 cleanup covers generic MCP process cleanup, not this Docker-wrapper case.

## What This Changes

Report that parallel `sessions_spawn` subagent runs can leave Docker containers launched indirectly by stdio MCP wrappers after OpenClaw kills only the wrapper process.

## Best Possible Solution

Introduce an explicit ownership-tracked cleanup contract for containers spawned for one MCP stdio session, then retire only those session-owned containers during MCP runtime disposal while preserving the generic process-tree fallback.

## Reproduction Assessment

Unclear. The report includes concrete Docker output and the source path confirms no Docker-aware cleanup, but this review did not live-run the UniFi npm wrapper under parallel `sessions_spawn`.

## Solution Assessment

No for the broad suggested `docker rm -f` style cleanup. The maintainable fix should first establish container ownership, such as recorded IDs or labels, before deleting containers during session cleanup.
