## Summary

Keep open. Current main still retires MCP stdio runtimes by closing transports and signaling the wrapper process tree, but it does not track or remove Docker containers launched indirectly by npm/npx MCP wrappers during parallel subagent runs. The existing Docker smoke covers host OS process cleanup, not wrapper-launched container cleanup, so the narrower Docker leak reported here remains unresolved.

## What This Changes

The issue asks OpenClaw to clean up Docker containers started indirectly by stdio MCP server wrappers when parallel `sessions_spawn` subagent runs end.

## Best Possible Solution

Keep this open and add a precise ownership model for MCP-launched containers: record or require OpenClaw-owned container labels/IDs for stdio MCP wrappers, remove only those containers when the owning session runtime is retired, and add a Docker E2E that launches a real wrapper-created child container during parallel subagent cleanup.

## Reproduction Assessment

Unclear. The issue provides a concrete live reproduction path with parallel `sessions_spawn` and a Docker-wrapped UniFi MCP server, and source inspection confirms the missing cleanup path, but this review did not execute that Docker scenario.

## Solution Assessment

No, not as the issue's immediate `/proc` parsing plus broad `docker rm -f` suggestion. The maintainable path is a scoped ownership contract that can prove which containers OpenClaw started before removing them.
