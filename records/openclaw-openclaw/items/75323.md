## Summary

Keep open. Current main still cleans up Docker-wrapped stdio MCP servers only through MCP runtime disposal, transport close, and generic process-tree signaling; I found no Docker/container ownership tracking or cleanup path for containers launched indirectly by npm/npx wrappers during one-shot parallel subagent runs.

## What This Changes

The issue requests completing MCP stdio cleanup so Docker containers launched by wrapper-based MCP servers do not survive after parallel specialist-agent sessions end.

## Best Possible Solution

Keep this open and design an ownership-tracked MCP container cleanup contract: record or require OpenClaw-owned container identifiers, labels, or runtime metadata for Docker-backed MCP launches, then remove only those session-owned containers during MCP runtime disposal while preserving generic process-tree cleanup as the fallback.

## Reproduction Assessment

Unclear. The issue provides concrete observed Docker output and source inspection confirms the missing Docker-aware cleanup path, but this review did not run the reported Docker-wrapped UniFi MCP server with parallel subagent spawns.

## Solution Assessment

No, not as the issue's immediate broad force-remove suggestion. The maintainable fix should avoid sweeping unrelated Docker state and instead clean only containers OpenClaw can prove were launched for the retiring MCP session.
