## Summary

Keep open. Current main still cleans MCP stdio runtimes by disposing the session runtime and signaling the spawned wrapper process tree, but it does not track or remove Docker containers launched indirectly by npm/npx MCP wrappers, so this remains distinct from the generic MCP process leak closed in #65694.

## What This Changes

The issue reports that Docker containers spawned indirectly by npm/npx-wrapped stdio MCP servers survive session cleanup during parallel specialist agent runs.

## Best Possible Solution

Add an ownership-tracked cleanup contract for containers spawned for one MCP stdio session, then remove only those session-owned containers during MCP runtime disposal while keeping generic process-tree cleanup as the fallback.

## Reproduction Assessment

Unclear. The issue provides concrete Docker output and a plausible `sessions_spawn` path, and source inspection confirms no Docker-aware cleanup path, but this review did not live-run the reported UniFi wrapper under parallel subagents.

## Solution Assessment

No for the broad immediate `docker rm -f` style suggestion. The safer maintainable fix should first establish container ownership, such as recorded container ids or labels, before deleting containers during session cleanup.
