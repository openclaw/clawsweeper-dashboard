## Summary

Keep open. Current main still retires stdio MCP runtimes by closing the transport and sending generic process-tree signals to the wrapper process; I found no Docker/container ownership tracking or cleanup path for containers launched indirectly by npm/npx-style MCP wrappers. The report is distinct from the earlier generic MCP process leak because the Docker daemon can keep a container alive after the wrapper exits.

## What This Changes

The issue requests cleanup for Docker containers launched indirectly by stdio MCP wrapper commands when parallel `sessions_spawn` subagent MCP sessions end.

## Best Possible Solution

Add an explicit MCP container ownership contract: record container ids or require OpenClaw-owned labels for containers created by a stdio MCP session, remove only those session-owned containers during runtime disposal, and keep generic process-tree cleanup as the fallback for ordinary stdio child processes.

## Reproduction Assessment

Unclear. The issue includes concrete Docker output and source inspection confirms no Docker-aware cleanup path, but this review did not run the reported UniFi MCP wrapper under parallel subagent spawns.

## Solution Assessment

No. The issue's immediate `docker rm -f`/process-ancestry idea is not the safest endpoint; the maintainable fix should first establish explicit OpenClaw ownership of the containers being removed.
