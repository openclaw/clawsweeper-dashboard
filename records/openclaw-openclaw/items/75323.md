## Summary

Keep open. Current main still disposes stdio MCP runtimes through transport close plus wrapper process-tree signaling only; neither the runtime path nor the existing Docker smoke tracks or removes Docker containers started by MCP wrappers, so the Docker-specific parallel-spawn leak remains unresolved and needs an ownership policy before implementation.

## What This Changes

The issue requests Docker-aware cleanup for stdio MCP servers launched during parallel specialist-agent sessions so wrapper-spawned containers do not accumulate.

## Best Possible Solution

Add session-owned Docker container identity tracking for stdio MCP launches, then remove only containers proven to belong to that MCP runtime while keeping process-tree signaling as the generic fallback.

## Reproduction Assessment

No high-confidence live reproduction was established. The report has concrete Docker output and source inspection shows current-main cleanup only reaches wrapper process cleanup, but this review did not run a Docker-wrapped UniFi MCP server under parallel sessions_spawn.

## Solution Assessment

No for the suggested immediate /proc parsing or broad docker rm -f cleanup. The maintainable fix should first establish OpenClaw-owned container identity, such as labels, cidfiles, or recorded container ids, before deleting containers at MCP runtime disposal.
