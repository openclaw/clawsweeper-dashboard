## Summary

Keep open. Current main still retires MCP stdio runtimes through transport disposal and generic process-tree signaling only; there is no Docker/container ownership tracking or cleanup path for containers launched indirectly by npm/npx stdio MCP wrappers, so this remains distinct from the generic process leak tracked in #65694.

## What This Changes

The issue reports Docker containers orphaned by Docker-wrapped stdio MCP servers during parallel specialist-agent spawns.

## Best Possible Solution

Add an ownership-tracked MCP container cleanup contract, such as explicit labels or recorded container ids for containers created for one MCP stdio session, then remove only those session-owned containers during runtime disposal while keeping generic process-tree cleanup as the fallback.

## Reproduction Assessment

Unclear. The issue includes concrete Docker output and source inspection confirms there is no Docker-aware cleanup path, but this review did not run the reported UniFi MCP wrapper under parallel subagent spawns.

## Solution Assessment

No for the broad immediate `docker rm -f` style suggestion. The maintainable fix should first establish OpenClaw ownership of the specific containers being removed, then clean them up through the MCP runtime disposal path.
