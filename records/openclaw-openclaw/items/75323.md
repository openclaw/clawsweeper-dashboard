## Summary

Keep open: current main still disposes stdio MCP runtimes by closing the transport and signaling the wrapper process tree, but it has no Docker-aware container ownership or removal path for wrappers that leave containers running.

## What This Changes

Request Docker-aware cleanup for containers spawned by npm/npx-wrapped stdio MCP servers during parallel `sessions_spawn` cleanup.

## Best Possible Solution

Add a session-owned container tracking or labeling cleanup contract for Docker-wrapped stdio MCP servers, then remove only containers proven to belong to that MCP runtime while keeping process-tree signaling as the generic fallback.

## Reproduction Assessment

No. The issue includes live Docker container output and source review shows only wrapper/process-group teardown, but I did not run a current-main UniFi Docker-wrapped MCP server through parallel `sessions_spawn`.

## Solution Assessment

No for the immediate broad `docker rm -f` suggestion. The maintainable fix should first establish OpenClaw-owned container identity, such as labels or recorded container ids, before deleting containers during MCP runtime disposal.
