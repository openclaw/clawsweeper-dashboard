## Summary

Keep open. Current main and the latest release still retire MCP stdio runtimes by closing the transport and signaling the wrapper process tree only; there is no Docker-aware ownership or container removal path, so this remains a valid Docker-wrapper gap distinct from #65694.

## What This Changes

The issue asks OpenClaw to finish MCP stdio cleanup so Docker containers launched by wrapper-based MCP servers are retired when parallel `sessions_spawn` runs end.

## Best Possible Solution

Add a session-owned container tracking or labeling cleanup contract for Docker-wrapped stdio MCP servers, then remove only containers proven to belong to that MCP session while keeping process-tree signaling as the generic fallback.

## Reproduction Assessment

No. The report includes concrete Docker output and source inspection confirms only wrapper PID cleanup, but this review did not live-run a Docker-wrapped UniFi MCP server under parallel `sessions_spawn` on current main.

## Solution Assessment

No for the broad immediate container-removal suggestion. The safer maintainable fix should establish OpenClaw-owned container identity, such as labels or recorded container ids, before deleting containers during MCP runtime disposal.
