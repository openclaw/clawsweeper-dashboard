## Summary

Keep open. Current main still retires one-shot subagent MCP runtimes through transport disposal and generic process-tree signaling only; I found no Docker/container ownership tracking or cleanup path for containers launched indirectly by npm/npx stdio MCP wrappers. The report is distinct from #65694 because Docker daemon-owned containers can remain alive after the wrapper process exits.

## What This Changes

The issue requests Docker-aware cleanup for stdio MCP servers launched through npm/npx wrappers during parallel `sessions_spawn` subagent runs so session disposal removes wrapper-launched containers.

## Best Possible Solution

Add an ownership-tracked MCP container cleanup contract: identify containers created for a specific MCP stdio session through labels, recorded container ids, wrapper integration, or another explicit runtime contract, then remove only those session-owned containers during MCP runtime disposal while preserving generic process-tree cleanup as the fallback.

## Reproduction Assessment

Unclear. The issue includes concrete Docker output and source inspection confirms no Docker-aware cleanup path, but this review did not run the reported UniFi MCP wrapper under parallel `sessions_spawn` subagents.

## Solution Assessment

No. The immediate process/cgroup-discovery plus `docker rm -f` suggestion is not the safest maintainable fix; the better path is explicit OpenClaw ownership of the containers being removed.
