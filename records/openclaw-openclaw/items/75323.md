## Summary

Keep open. Current main still disposes Docker-wrapped stdio MCP servers through generic runtime disposal and process-tree signaling only, with no container id, label, cgroup, or Docker API ownership path for containers launched indirectly by npm/npx wrappers.

## What This Changes

The issue reports Docker containers left behind when parallel specialist sessions launch Docker-wrapped stdio MCP servers through npm/npx wrappers.

## Best Possible Solution

Add an ownership-tracked cleanup contract for containers spawned for one MCP stdio session, then remove only those session-owned containers during runtime disposal while keeping process-tree signaling as the fallback.

## Reproduction Assessment

Unclear. The report includes concrete Docker output and the current code trace confirms no Docker-aware cleanup path, but this review did not live-run the UniFi wrapper or a wrapper-launched-container scenario.

## Solution Assessment

No for the suggested immediate generic docker rm approach. The safer maintainable fix needs container ownership metadata, labels, recorded ids, or an explicit wrapper/plugin contract before deleting containers during MCP cleanup.
