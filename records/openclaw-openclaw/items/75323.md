## Summary

Keep open. Current main still cleans MCP stdio runtimes through transport disposal and generic process-tree signaling only; it does not track or reap Docker containers started indirectly by npm/npx MCP wrappers, so this remains distinct from the generic leak fixed for #65694.

## What This Changes

The issue requests Docker-aware cleanup for MCP stdio servers launched through wrapper commands during parallel specialist-agent sessions.

## Best Possible Solution

Add an ownership-tracked MCP container cleanup contract, such as explicit labels or recorded container ids for containers created for one MCP stdio session, then remove only those session-owned containers during runtime disposal.

## Reproduction Assessment

Unclear. The issue includes concrete Docker output and current source inspection confirms there is no Docker-aware cleanup path, but this review did not run the reported UniFi wrapper under parallel specialist spawns.

## Solution Assessment

No for the broad immediate docker rm -f style suggestion. The maintainable fix should first establish OpenClaw ownership of the specific containers being removed, then clean them up through the MCP runtime disposal path.
