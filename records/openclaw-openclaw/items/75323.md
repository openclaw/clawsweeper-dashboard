## Summary

Keep open. Current main still cleans up Docker-wrapped stdio MCP servers through session runtime disposal and generic process signaling only; it does not record or remove containers launched indirectly by npm/npx wrappers, so this remains distinct from the generic MCP leak tracked in #65694.

## What This Changes

Request cleanup for Docker containers launched indirectly by stdio MCP wrappers during parallel `sessions_spawn` subagent runs.

## Best Possible Solution

Add an ownership-tracked cleanup contract for containers spawned for one MCP stdio session, then remove only those session-owned containers during MCP runtime disposal while keeping `killProcessTree` as the fallback.

## Reproduction Assessment

Unclear. The report includes concrete Docker output and a plausible `sessions_spawn` path, and current source tracing confirms no Docker-aware cleanup path, but this review did not live-run the reported UniFi wrapper under parallel subagents.

## Solution Assessment

No for the broad immediate `docker rm -f` style suggestion. The safer maintainable fix should establish container ownership, such as recorded container ids or labels, before deleting containers during session cleanup.
