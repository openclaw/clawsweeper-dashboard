## Summary

Keep open. Current main still retires MCP runtimes through stdio transport close and process-tree signaling only; I found no Docker-aware ownership, label, cgroup, container-id, or removal path, so this remains distinct from the generic MCP cleanup fixed for #65694.

## What This Changes

The issue asks OpenClaw to clean up Docker containers launched by wrapper-based stdio MCP servers after parallel run-mode `sessions_spawn` subagents finish.

## Best Possible Solution

Add a session-owned container tracking or labeling cleanup contract for Docker-wrapped stdio MCP servers, then remove only containers proven to belong to that MCP session while keeping process-tree signaling as the generic fallback.

## Reproduction Assessment

No. The report includes concrete Docker output and current source confirms only wrapper PID cleanup, but this review did not live-run a Docker-wrapped UniFi MCP server under parallel `sessions_spawn` on current main.

## Solution Assessment

No for the broad immediate `docker rm -f` style suggestion. The safer maintainable fix should establish OpenClaw-owned container identity, such as labels or recorded container ids, before deleting containers during MCP runtime disposal.
