## Summary

Keep open. Current main still resolves one-shot subagent MCP cleanup to runtime disposal, transport close, and generic process-tree signaling; I found no Docker/container ownership tracking or cleanup for containers launched indirectly by npm/npx MCP wrappers, and the existing Docker smoke covers a Node stdio process rather than a wrapper-launched container.

## What This Changes

Request to complete MCP stdio cleanup so parallel one-shot subagents using npm/npx-wrapped Docker MCP servers do not leave orphaned Docker containers after session cleanup.

## Best Possible Solution

Keep the issue open and design a precise MCP container ownership model: record or require OpenClaw-owned container identifiers, labels, or runtime metadata for Docker-backed MCP launches, then remove only those session-owned containers during MCP runtime disposal while retaining process-tree cleanup as the generic stdio fallback.

## Reproduction Assessment

Unclear. The issue provides concrete steps and source inspection confirms the missing Docker-aware cleanup path, but this review did not live-run the reported Docker-wrapped UniFi MCP server with parallel subagent spawns.

## Solution Assessment

No. The issue’s immediate suggestion to enumerate and force-remove Docker containers is too broad; the maintainable path is an ownership-tracked cleanup contract that cannot sweep unrelated host containers.
