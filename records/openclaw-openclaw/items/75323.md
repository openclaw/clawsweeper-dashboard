## Summary

Keep open. Current main still handles this path through bundle MCP runtime disposal and generic process-tree signaling only; I found no Docker/container ownership tracking or removal path for containers launched indirectly by npm/npx stdio MCP wrappers. The existing Docker smoke verifies stdio probe PIDs exit, but it does not exercise a wrapper-launched Docker child container.

## What This Changes

Request Docker-aware cleanup for MCP stdio servers launched through wrappers during parallel `sessions_spawn` runs so session end removes spawned containers as well as wrapper processes.

## Best Possible Solution

Add an explicit ownership-tracked MCP container cleanup contract: record or label containers created for a specific MCP stdio session, remove only those session-owned containers during runtime disposal, and keep generic process-tree cleanup as the fallback for ordinary subprocesses.

## Reproduction Assessment

Unclear. The issue includes concrete Docker `ps` evidence and source inspection confirms no Docker-aware cleanup path, but this review did not run the reported UniFi wrapper or a synthetic wrapper-launched-container scenario under parallel `sessions_spawn`.

## Solution Assessment

No. The current implementation is insufficient for Docker-daemon-managed children, and the report's immediate `docker rm -f` style suggestion is not the safest fix; the maintainable path needs explicit OpenClaw ownership markers before removal.
