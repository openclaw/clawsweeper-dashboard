## Summary

Keep open. Current main still retires bundled stdio MCP runtimes by closing MCP transports and signaling the spawned process tree, but there is no Docker/container ownership tracking or cleanup path for containers launched indirectly by npm/npx MCP wrappers during parallel subagent runs. The existing Docker cleanup smoke verifies a Node probe process exits, not that a wrapper-launched container is removed.

## What This Changes

Request to complete MCP stdio cleanup so Docker containers launched by npm/npx-wrapped MCP servers during parallel `sessions_spawn` runs are terminated when the owning session ends.

## Best Possible Solution

Add an ownership-tracked Docker-backed MCP cleanup contract: record or require OpenClaw-owned container identifiers, labels, or runtime metadata for Docker-backed MCP launches, then remove only containers proven to belong to the retiring session while keeping process-tree cleanup as the generic fallback.

## Reproduction Assessment

Unclear. The issue provides concrete Docker output and current-main source inspection confirms the missing Docker-aware cleanup path, but this review did not run the reported UniFi Docker MCP wrapper under parallel `sessions_spawn`.

## Solution Assessment

No, not the issue's broad immediate `docker rm -f` style suggestion. The maintainable solution should track OpenClaw-owned containers and delete only those, with tests for the wrapper-launched-container case.
