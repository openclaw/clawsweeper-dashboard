## Summary

Keep open. Current main still only closes MCP stdio transports and sends process-tree signals to the wrapper process; I found no Docker/container ownership tracking or cleanup for containers launched indirectly by npm/npx MCP wrappers during parallel subagent runs, so the narrower Docker-wrapper leak remains unresolved after the prior generic MCP cleanup fix.

## What This Changes

The issue reports that Docker-backed MCP stdio servers launched through npm/npx wrappers can remain running after parallel `sessions_spawn` cleanup kills only the wrapper process.

## Best Possible Solution

Keep the issue open and design a precise MCP-launched container ownership model: either require/propagate OpenClaw session labels or otherwise track containers OpenClaw can prove belong to the retiring MCP session, then remove only those owned containers while preserving process-tree cleanup as the generic stdio fallback.

## Reproduction Assessment

Unclear. The issue gives concrete steps and current source inspection confirms the missing Docker-aware cleanup path, but this review did not live-run the reported Docker-wrapped UniFi MCP server parallel-spawn scenario.

## Solution Assessment

No, not as the broad immediate cleanup suggestion. The maintainable fix should avoid sweeping Docker state by ancestry, image, or cgroup guesses and instead clean only containers with a proven OpenClaw/session ownership contract.
