## Summary

Keep open. Current main still has only generic MCP stdio transport disposal and process-tree signaling; I found no Docker/container ownership tracking or cleanup for containers launched indirectly by npm/npx MCP wrappers during parallel subagent runs, so the narrower report is not resolved by the prior generic leak fix.

## What This Changes

The issue reports that Docker-backed MCP stdio servers launched through npm/npx wrappers can remain running after parallel `sessions_spawn` cleanup kills only the wrapper process.

## Best Possible Solution

Keep the issue open and design a precise ownership model for MCP-launched containers: record or require OpenClaw-owned container labels/metadata at launch, then remove only those session-owned containers during MCP runtime disposal while keeping process-tree cleanup as the generic stdio fallback.

## Reproduction Assessment

Unclear. The issue includes concrete reproduction details and source inspection confirms the missing Docker-aware cleanup path, but this review did not run the reported Docker-wrapped UniFi MCP server scenario live.

## Solution Assessment

No, not as the issue's broad immediate cleanup suggestion. The maintainable fix should avoid parsing or sweeping unrelated Docker state and instead clean only containers OpenClaw can prove were launched for the retiring MCP session.
