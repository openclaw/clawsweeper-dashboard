## Summary

Keep open. Current main still retires MCP stdio runtimes through generic transport disposal and process signaling only; it does not track or remove Docker containers launched indirectly by npm/npx wrapper MCP servers, so this remains a distinct follow-up from the generic process leak closed in #65694.

## What This Changes

Request cleanup for Docker containers launched indirectly by stdio MCP wrappers during parallel subagent sessions.

## Best Possible Solution

Add an ownership-tracked cleanup contract for containers spawned for one MCP stdio session, then remove only those session-owned containers during MCP runtime disposal while keeping generic process-tree cleanup as the fallback.

## Reproduction Assessment

Unclear. The issue provides concrete Docker output and a plausible sessions_spawn path, and source inspection confirms no Docker-aware cleanup path, but this review did not live-run the reported UniFi wrapper under parallel subagents.

## Solution Assessment

No for the broad immediate docker rm -f style suggestion. The safer maintainable fix should establish container ownership, such as recorded container ids or labels, before deleting containers during session cleanup.
