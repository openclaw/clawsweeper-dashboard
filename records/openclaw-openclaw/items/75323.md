## Summary

Keep open. Current main still retires one-shot subagent MCP runtimes through transport disposal and generic process-tree signaling only; I found no Docker/container ownership tracking or cleanup path for containers launched indirectly by npm/npx stdio MCP wrappers. The report is distinct from the earlier generic MCP process leak because the Docker daemon can keep the container alive after the wrapper process exits.

## What This Changes

The issue requests Docker-aware cleanup for MCP stdio servers launched through npm/npx wrappers during parallel specialist sessions, because current process-tree cleanup can leave Docker containers running.

## Best Possible Solution

Add an ownership-tracked MCP container cleanup contract: identify containers created for a specific MCP stdio session through labels, recorded container ids, or another explicit runtime contract, then remove only those session-owned containers during MCP runtime disposal while preserving generic process-tree cleanup as the fallback.

## Reproduction Assessment

Unclear. The issue includes concrete Docker output and source inspection confirms there is no Docker-aware cleanup path, but this review did not run the reported UniFi MCP wrapper under parallel subagent spawns.

## Solution Assessment

No. The broad immediate `docker rm -f` style suggestion is not the safest fix; the maintainable solution should first establish OpenClaw ownership of the specific containers being removed.
