## Summary

Keep open: current main and the latest release still only close stdio MCP runtimes through the wrapper process transport, with no Docker/container ownership or removal path for wrappers that leave containers running.

## What This Changes

The issue asks OpenClaw to fully clean up Docker-wrapped stdio MCP servers spawned during parallel specialist sessions.

## Best Possible Solution

Add session-owned container tracking or labeling for Docker-wrapped stdio MCP servers, then remove only containers proven to belong to that OpenClaw MCP runtime while preserving process-tree signaling as the generic fallback.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection shows the current cleanup path is still process-tree-only and the issue includes live Docker container output, but I did not execute the UniFi Docker-wrapped MCP server through parallel sessions_spawn.

## Solution Assessment

No for the issue's immediate broad docker rm suggestion. The maintainable path should first establish OpenClaw-owned container identity, such as labels or recorded container ids, before deleting containers during MCP runtime disposal.
