## Summary

Keep open: the PR targets a valid Discord multi-bot slash-command bug and the application-id cache namespace is the right direction, but the current patch still leaves shared cache-file writes vulnerable to concurrent account startup dropping another application's scoped entry.

## What This Changes

The PR scopes Discord slash-command deploy-cache keys by Discord application id, adds multi-application regression tests, and records a Discord changelog fix.

## Best Possible Solution

Land the app-id cache namespace with a small merge or serialization guard around shared hash-file writes, plus concurrent regression coverage preserving every application's cache entry.

## Reproduction Assessment

Yes. Source inspection shows current main stores bare command-scope hashes in one shared Discord cache file and returns before REST reconcile on a matching hash; the remaining PR race is source-reproducible from concurrent account startup and background deploy.

## Solution Assessment

No, not yet. Application-id scoping is the narrow fix for the reported cross-application skip, but the shared cache persistence should merge current on-disk hashes or serialize writes before this is safe to merge.
