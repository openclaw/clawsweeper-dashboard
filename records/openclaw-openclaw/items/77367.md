## Summary

Keep open: this is an active focused bug-fix PR for a real Discord multi-bot regression that still exists on current main, but it is not merge-ready because the branch is conflicting with current main, the after-fix proof is still mock/test-only, and one low-priority mutex cleanup remains.

## What This Changes

The PR scopes Discord command-deploy cache keys by Discord application id, serializes shared cache persistence, adds regression tests, and updates the changelog.

## Best Possible Solution

Rebase the branch onto current main while preserving `privateFileStore`, fix the lock-map cleanup, and land after redacted live multi-bot Discord proof or maintainer-equivalent proof confirms secondary application commands register.

## Reproduction Assessment

Yes. Source inspection on current main plus the linked issue’s multi-bot steps gives a high-confidence path: a shared cache file and bare `global:reconcile` key can make a later Discord application skip reconciliation.

## Solution Assessment

Yes for the core direction. Application-id cache namespacing plus serialized read-merge-write persistence is the narrow maintainable fix; the branch still needs proof, rebase work, and a small mutex cleanup before merge.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Delete released cache locks:** `extensions/discord/src/internal/command-deploy.ts:52`
  - body: The cleanup compares the stored map value to `previous.then(() => next)`, which creates a fresh promise each time, so the entry is never deleted after release. Store the chained promise in a local variable and compare/delete that same object to avoid retaining one lock per unique cache path for the process lifetime.
