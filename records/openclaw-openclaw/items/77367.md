## Summary

Keep open: this is a plausible narrow Discord bug fix and the prior P2 cache-race findings appear addressed, but the external PR still has only stub/CI proof and needs real multi-bot Discord behavior proof before merge.

## What This Changes

The PR scopes Discord command-deploy cache keys by Discord application id, serializes shared cache writes, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the application-id cache namespace and serialized persistence, then merge after live multi-bot Discord proof is added; the lock-map cleanup can be fixed or accepted as non-blocking.

## Reproduction Assessment

Yes. Source inspection and the linked reports give a high-confidence path: all Discord accounts share one cache file, current main stores `global:reconcile` without an application id, and a matching hash returns before REST reconciliation.

## Solution Assessment

Yes. Application-id-scoped cache keys preserve same-application rate-limit protection while preventing one Discord application from suppressing another application's reconcile; the direct Discord REST mirroring workaround is not the maintainable fix.
