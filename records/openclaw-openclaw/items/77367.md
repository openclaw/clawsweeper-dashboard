## Summary

Keep open: the PR addresses a source-reproducible Discord multi-bot slash-command cache bug, but the current head is not merge-ready because its proof is mock-only and it needs to be rebased without regressing the current private file-store path.

## What This Changes

The PR scopes Discord slash-command deploy cache keys by application id, serializes shared cache persistence, adds regression tests, and updates the changelog.

## Best Possible Solution

Rebase the fix onto current main, keep application-id scoped cache keys with serialized merge/write behavior through `privateFileStore`, clean up the lock-map release check, then merge after real multi-bot Discord proof is supplied.

## Reproduction Assessment

Yes, by source inspection. Current main stores bare command-deploy hashes in one shared Discord cache file and returns before REST reconcile on a matching hash, so a second application with the same command set can skip its own command registration.

## Solution Assessment

Yes in direction: application-id cache scoping plus serialized shared-cache persistence is the narrow maintainable fix. The current head still needs to be rebased onto the private-store implementation and backed by real behavior proof before merge.
