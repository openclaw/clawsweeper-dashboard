## Summary

Keep open: the PR is aimed at a valid Discord multi-bot command-cache bug, but the latest revision still leaves a shared-cache write race and the external PR lacks required after-fix real behavior proof.

## What This Changes

The PR scopes Discord slash-command deploy-cache keys by application id, adds multi-application cache tests, merges on-disk hashes before persisting, and adds a changelog entry.

## Best Possible Solution

Keep the application-id cache namespace, serialize shared command-deploy cache persistence per store path, add a true concurrent regression test, and require real multi-bot Discord proof before merging.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main shares one Discord cache file, uses bare command-scope keys, and returns before REST reconcile on a matching hash. The PR's remaining write race is also source-reproducible from concurrent account startup and the unsynchronized cache read/merge/rename sequence.

## Solution Assessment

No. Application-id scoping is the narrow maintainable fix for the reported collision, but this revision still needs per-path cache-write serialization or an equivalent lock plus real after-fix Discord/OpenClaw proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Serialize shared cache writes:** `extensions/discord/src/internal/command-deploy.ts:195-226`
  - body: The re-read still leaves the shared cache update as read -> merge -> write -> rename without a per-path lock. If two Discord account deployers call `persistHashes()` together, both can read the same snapshot and the later rename drops the other app's `app:<id>` entry, so the next startup loses the rate-limit protection this cache is meant to preserve; serialize writes per `hashStorePath` and add a true concurrent regression.
  - confidence: 0.91
