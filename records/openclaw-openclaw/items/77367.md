## Summary

Keep open: the root Discord multi-bot bug is valid and the application-id cache scoping direction is right, but this PR is not merge-ready because proof is mock-only and the branch regresses current main's private file-store persistence; a small lock cleanup remains.

## What This Changes

The PR scopes Discord slash-command deploy cache keys by Discord application id, serializes shared cache persistence, adds regression tests, and adds a changelog fix entry.

## Best Possible Solution

Rebase the application-id cache scoping onto current main, keep the serialized read-merge-write path on the private/fs-safe store, clean the lock deletion, and merge after live multi-bot Discord proof is posted.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: two Discord applications share one cache file and bare keys, so the second app can return before REST reconcile when the command hash matches; the linked user reports also confirm the symptom live.

## Solution Assessment

No, not as-is. Application-id cache scoping is the right core fix, but this branch should preserve current main's private file-store persistence and fix the lock cleanup before normal merge review.
