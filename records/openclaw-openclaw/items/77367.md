## Summary

Keep open: the application-id cache namespace addresses the reported multi-bot collision, but the latest head still permits concurrent deployers to overwrite each other's newly scoped cache entries during shared-file persistence.

## What This Changes

The PR scopes Discord slash-command deploy-cache keys by application id, adds multi-application cache regression tests, merges on-disk hashes before persisting, and adds a changelog entry.

## Best Possible Solution

Keep the application-id namespace, then serialize or lock shared command-deploy cache persistence per store path so concurrent account deployers preserve every application's entry.

## Reproduction Assessment

Yes. Source inspection shows main shares one Discord command-deploy cache file with bare command-scope keys and returns before REST reconcile on a matching hash; the PR's remaining race is also source-reproducible from concurrent background deployers.

## Solution Assessment

No. Application-id key scoping is the right narrow fix, but the latest read/merge/rename persistence path still needs shared-file serialization or an equivalent atomic update.

On this PR, keep the application-id cache key scoping for the linked Discord multi-bot slash-command bug. Fix the remaining race in `extensions/discord/src/internal/command-deploy.ts`: `persistHashes` currently re-reads and merges the cache but does not serialize the read -> merge -> rename sequence, so simultaneous deployers can still overwrite each other's `app:<id>:...` entries. Add a per-`hashStorePath` async mutex or equivalent scoped serialization for shared cache persistence while preserving best-effort cache semantics and avoiding core/gateway policy changes. Add or update `extensions/discord/src/internal/command-deploy.test.ts` with a true concurrent multi-deployer regression that would fail when two persist operations read the same old snapshot before either rename. Keep the existing changelog entry; do not change Discord auth, config shape, or command registration behavior beyond cache persistence.
