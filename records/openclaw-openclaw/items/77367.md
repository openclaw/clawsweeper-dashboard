The PR scopes Discord command-deploy cache keys by application ID, serializes and merges shared cache writes, and adds multi-application persistence regression coverage.

## Best Possible Solution

Land the application-scoped serialized cache implementation while explicitly accepting the bounded first-start reconcile; the cache is rebuildable transient state, so a legacy-key migration would add complexity without preserving user data.

## Solution Assessment

Yes. Application identity is the missing cache invariant, and the per-path serialized read-merge-write cycle is the narrowest design that preserves independent application hashes without introducing another persistence mechanism.
