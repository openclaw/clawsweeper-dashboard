## Summary

Keep open: the PR addresses a valid Discord multi-bot slash-command cache bug, but the current head can still lose another application’s cache entry during shared-file writes and needs a narrow persistence fix before merge.

## What This Changes

The PR prefixes Discord command-deploy cache keys with the resolved application id, adds multi-application regression tests, and adds a Discord changelog entry.

## Best Possible Solution

Keep the app-scoped cache-key approach, but merge or serialize shared cache-file updates and add a concurrent deployer regression test before landing.

## Reproduction Assessment

Yes. Source inspection shows current main shares one deploy-cache file, uses bare command-scope keys, and skips REST reconciliation on a matching hash; the PR-introduced cache-loss concern is also source-reproducible from stale in-memory maps plus snapshot renames.

## Solution Assessment

No, not yet. Scoping by application id is the right core fix, but the shared-file writer also needs to merge or serialize updates so one app’s successful deploy does not erase another app’s cache entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Merge shared cache entries before writing app-scoped keys:** `extensions/discord/src/internal/command-deploy.ts:111`
  - body: With the new per-app key prefix, two deployers that loaded `command-deploy-cache.json` before either write can each persist only its own `app:<id>:...` entry, and the later rename drops the other app's cache entry. Gateway account startup can overlap and command deploy runs in the background, so this makes at least one app re-reconcile on later restarts despite no command changes. Re-read and merge the current on-disk hashes before persisting, or serialize writes for this shared file.
  - confidence: 0.86
