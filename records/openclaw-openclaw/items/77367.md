## What This Changes

The PR scopes Discord command-deploy cache keys by application id, serializes shared cache persistence, adds multi-application and concurrency regression tests, and updates the changelog.

## Best Possible Solution

Keep the application-id namespace and per-path cache serialization, then merge only after live multi-bot Discord proof is supplied and the small lock-map cleanup is addressed or accepted as non-blocking.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a shared hash-store path plus bare cache keys lets a second application with the same command hash return before REST reconcile; the PR adds stub REST regression tests for that path, but no live setup proof.

## Solution Assessment

Yes for the core code direction: application-id cache keys plus serialized shared-cache persistence is the narrow maintainable fix. The remaining merge blocker is proof from a real setup, with one low-priority cleanup in the mutex bookkeeping.
