## Summary

Keep open: the Discord fix direction is sound and targets a real current-main bug, but the current PR head still has a large unrelated changelog rewrite, a cache-lock cleanup leak, and only test/stub proof rather than live Discord evidence.

## What This Changes

The PR scopes Discord command-deploy cache keys by application id, serializes cache persistence with an in-process per-path mutex, and adds command-deploy regression tests plus a changelog entry.

## Best Possible Solution

Land the application-id cache namespace plus serialized read-merge-write persistence after restoring current changelog content, fixing lock cleanup, and adding redacted live multi-bot Discord proof.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: multiple Discord accounts share one cache file, bare `global:reconcile` keys can collide, and matching hashes return before REST reconciliation; the linked canonical issue also has live v2026.5.7 confirmation.

## Solution Assessment

Yes for the core direction, but not for the current head as-is. Scoping the cache by Discord application id while preserving same-application cache skips is the narrow maintainable fix, but the branch still needs changelog restoration, lock cleanup, and live proof.
