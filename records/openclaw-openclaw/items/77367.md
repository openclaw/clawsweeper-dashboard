## Summary

Keep open: this PR addresses a real Discord multi-bot command deployment bug with the right core approach, and CI is green, but the external PR still has only stubbed/test proof plus one low-priority lock cleanup issue before merge.

## What This Changes

The PR prefixes Discord command-deploy cache keys with the application id, serializes shared cache persistence, merges on-disk cache state, and adds multi-application regression tests plus a changelog entry.

## Best Possible Solution

Land the application-id cache namespace plus serialized read-merge-write persistence after live Discord proof is supplied and the lock cleanup is tightened.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: shared `command-deploy-cache.json`, unscoped `global:reconcile`, and an early return before Discord REST reconciliation; I did not run a live Discord setup.

## Solution Assessment

Yes for the core direction. Scoping the existing cache by Discord application id preserves rate-limit protection while fixing multi-bot collisions, with a small lock cleanup still needed before landing.

Full review comments:

- **[P3] Delete released cache locks:** `extensions/discord/src/internal/command-deploy.ts:52`
  - body: The value stored in the map is a chained promise, but the finally block calls `previous.then(() => next)` again, which creates a fresh promise that can never equal the stored value. Store the chained promise once, put that exact object in the map, and compare/delete that object so released cache paths do not stay retained for the process lifetime.
