## Summary

Keep this issue open. Current main still models cron results as a single `delivery` route with `none`, `announce`, or `webhook`; it does not provide deterministic `output.targets`, file append/write routing, multi-target fan-out, or main/named-session result injection. The existing ClawSweeper keep-open comment remains accurate against the newer main SHA, and the related cron-output reports are adjacent but do not supersede this broader feature request.

## What This Changes

The issue requests a first-class cron job output contract that can route final job results to files, channels, sessions, or discard them silently without relying on prompt instructions.

## Best Possible Solution

Design an additive, schema-backed core cron output contract, or a versioned extension of delivery, that validates file, channel/webhook, session, and silent targets; records per-target status in run logs; defines mixed-success behavior; and updates CLI, Control UI, protocol schemas, docs, and focused runtime tests. If maintainers want plugin ownership for some target types, first add a narrow core/plugin API that safely exposes final cron responses and target config.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is statically verifiable on current main: cron schemas, runtime planning, UI, and docs expose only the existing single delivery path, so the proposed output.targets shape has no accepted source or protocol surface.

## Solution Assessment

Unclear from the issue alone whether a new output field is better than extending delivery. The maintainable direction is an additive, schema-backed core contract with explicit security rules for file and session targets, not more prompt instructions.
