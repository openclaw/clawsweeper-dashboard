## Summary

Keep open: current main still has a source-level path where `doctor --fix` rewrites an `openai-codex/*` route to `openai/*` with `agentRuntime.id: "pi"` when the Codex runtime is not ready, and the current OpenAI docs define that resulting route as direct API-key billing.

## What This Changes

The issue requests that upgrades not convert OpenAI Codex subscription routes into direct OpenAI API billing when an API key is also present.

## Best Possible Solution

Make doctor fail closed on billing changes: do not rewrite a subscription-backed `openai-codex/*` PI route to `openai/*` with `pi`; only rewrite to `openai/*` when the Codex runtime will actually be selected, otherwise preserve the OAuth route with an actionable warning.

## Reproduction Assessment

Yes, as a source-level reproduction: `maybeRepairCodexRoutes` with `model: "openai-codex/gpt-5.5"`, `shouldRepair: true`, and no ready Codex runtime currently produces `openai/gpt-5.5` plus `agentRuntime.id: "pi"`. I did not execute tests because this review is read-only.

## Solution Assessment

No, current main is not the best solution because it can turn an explicit subscription/OAuth route into a direct API-key route. The safer fix is to preserve `openai-codex/*` unless the repair can prove and set the Codex runtime path.
