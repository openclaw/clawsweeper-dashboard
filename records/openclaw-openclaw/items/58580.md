## Summary

Keep open. Current main has useful partial recovery for long `processing` diagnostics, context-overflow resets, specific transcript corruption, and manual `sessions.reset`, but it does not implement the requested health check that tracks inbound messages with no successful reply over time and auto-refreshes only the affected channel/session.

## What This Changes

The issue requests a gateway health check that detects sessions receiving messages without successful replies for a prolonged interval, logs a warning, and clears or refreshes only the affected session.

## Best Possible Solution

Add a conservative session-health state machine in the gateway reply path: record inbound messages that expect a user-visible response, record successful final delivery per session/channel, count prolonged no-success or repeated pre-reply failures, and after a configurable threshold reset or prune only that session through the existing gateway session-reset authority after proving no active embedded/reply/lane work owns it. It should emit diagnostics/logs and avoid replacing the separate stale-`running` lane recovery tracked by #59878.

## Reproduction Assessment

Unclear. The incident report is credible and source inspection shows the missing health policy, but there is no minimal fixture or live reproduction here for the exact false-alive Telegram failure after restart.

## Solution Assessment

No, current main is not the complete solution. The existing stuck-session diagnostics are useful for long `processing` lanes, but the requested fix needs an explicit no-successful-response health check rather than overloading warning-only diagnostic lane cleanup.
