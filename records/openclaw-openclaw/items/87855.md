The PR wraps Telegram stale spool claim recovery rename with ENOENT/EEXIST handling and adds a regression test for the race.

## Best Possible Solution

Keep the narrow runtime guard, replace the test with one that forces ENOENT/EEXIST from fs.rename during recovery, and add redacted terminal or live-log proof that invokes the patched recovery path before merge.

## Reproduction Assessment

Yes, source-reproducible: current main can throw ENOENT from the final recovery rename after an earlier stat/pathExists check, and polling catches that as a drain failure. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

The runtime shape is the right fix because it mirrors the existing release-claim rename handling, but the PR is not merge-ready until the regression test actually hits the rename race and the proof exercises patched OpenClaw code.
