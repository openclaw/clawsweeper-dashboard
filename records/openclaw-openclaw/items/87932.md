Full review comments:

- **[P2] Resolve strings before accepting threshold config:** `src/config/zod-schema.agent-defaults.ts:161-169`
  - body: This schema now validates percentage strings for the main compaction thresholds, but the current agent-settings reader still uses numeric-only guards for `reserveTokens`, `keepRecentTokens`, and `reserveTokensFloor`, and manual compaction only treats `keepRecentTokens` as explicit when it is a number. A config like `reserveTokens: "4%"` would validate and be advertised, then be ignored or fall back in those paths, so either route these fields through a context-aware resolver everywhere or keep the schema numeric until that is done.
  - confidence: 0.92
- **[P2] Preserve softThresholdTokens strings before planning:** `src/config/zod-schema.agent-defaults.ts:198-200`
  - body: The PR accepts `memoryFlush.softThresholdTokens` as a string, but memory-core still normalizes that field with a number-only helper and replaces strings with the default before `agent-runner-memory` can resolve them. That means `"40%"` validates but the active flush plan can still use the old default, so the plan needs to carry or resolve the raw percentage value.
  - confidence: 0.9
- **[P2] Keep keepRecentTokens validation positive:** `src/config/zod-schema.agent-defaults.ts:164-166`
  - body: Current schema rejects `keepRecentTokens: 0`, and the runtime still treats zero as invalid through `toPositiveInt`. This union changes the numeric branch to nonnegative, so an invalid keep-recent config would now validate and then be silently ignored; keep the positive constraint unless zero has a documented runtime meaning.
  - confidence: 0.87
