## Summary

Keep open. Current main and the latest release still use the default backoff schedule at the recurring error safety-net floor, while sibling runnability/recompute paths and docs use configured `cron.retry.backoffMs`; this PR fixes that divergent call and includes sufficient SQLite-readback proof with no blocking findings.

## What This Changes

The PR makes recurring cron error backoff persistence honor configured `cron.retry.backoffMs` and adds a service-path SQLite readback regression test.

## Best Possible Solution

Land the focused call-site fix after normal exact-head checks, keeping persisted recurring error backoff aligned with the existing `cron.retry` contract.

## Reproduction Assessment

Yes. Current main is source-reproducible: the recurring error floor omits configured `cron.retry.backoffMs` while sibling gates, recompute, and docs use the configured schedule.

## Solution Assessment

Yes. Passing the existing configured schedule into the one divergent `errorBackoffMs` call is the narrowest maintainable fix; a new helper, config option, or broader scheduler refactor would add churn without improving the owner boundary.
