## Summary

Keep open. Current main still intentionally retires the Chrome Extension Relay, rejects `driver: "extension"`, and migrates legacy relay config to Chrome MCP `existing-session`; that does not restore the requested existing-tab relay, auto-reattach, video-control, or chat-automation behavior. The related browser-backend/plugin-interface work is still open, so this needs maintainer product and security direction rather than cleanup closure.

## What This Changes

The issue requests restoring the removed Chrome Extension Relay as an optional browser driver, including port 18792 relay support, bundled extension files, legacy `driver: "extension"` compatibility, and a migration path for users of extension-specific browser automation.

## Best Possible Solution

Maintainers should make the browser strategy explicit: either keep the legacy extension relay retired and document the migration gaps clearly, or design a vetted browser-backend/plugin path and maintain an optional Chrome Extension Relay with security review, tests, docs, packaging, and legacy-config migration behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main rejects `driver: "extension"` in the profile creation route, has a contract test asserting that rejection, and keeps the relay startup helper as a no-op.

## Solution Assessment

Unclear. Directly restoring the old relay in core may not be the best solution because OpenClaw’s current direction favors optional high-power capabilities through plugins, but the browser-backend plugin API needed for that path appears to remain open in #19289.
