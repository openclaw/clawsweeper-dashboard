## Summary

Keep open. Current main still intentionally removes the Chrome Extension Relay, rejects `driver: "extension"`, and documents Chrome MCP `existing-session` as the replacement; the open related work at https://github.com/openclaw/openclaw/issues/53599 and https://github.com/openclaw/openclaw/issues/19289 is relevant but does not by itself resolve this broader restore/migration request. The linked FSB project is useful external reference for scoped extension-style browser control, not a current OpenClaw replacement for the old relay path.

## What This Changes

Request restoring the removed Chrome Extension Relay as an optional browser driver or migration path for existing-tab control, auto-reattach, video/chat automation, and direct Chrome workflows.

## Best Possible Solution

Maintainers should make a browser strategy decision: keep the old relay retired with clearer migration-gap docs, or design a hardened opt-in extension relay/browser-backend path with scoped tab ownership, visible logs, consent/credential guardrails, tests, packaging, and docs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure or create a profile with `driver: "extension"`: current main rejects it in schema and routes, keeps the relay helper as a no-op, and has a contract test asserting the unsupported-driver 400.

## Solution Assessment

Unclear. Directly restoring the old core relay may not be the best maintainable fix; a maintainer-approved optional browser backend/plugin API or explicitly documented retirement path is safer than an automated rollback.
