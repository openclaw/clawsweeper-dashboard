## Summary

Keep open. Current main still intentionally removes the Chrome Extension Relay, rejects `driver: "extension"`, and migrates legacy relay config to Chrome MCP `existing-session`; the FSB reference is useful prior art, but restoring or replacing this workflow still needs maintainer browser/API/security direction.

## What This Changes

Restore Chrome Extension Relay as an optional browser driver, including relay startup on port 18792, extension assets, legacy `driver: "extension"` compatibility, or a migration path for dependent users.

## Best Possible Solution

Maintainers should decide the browser strategy: keep the relay retired with clearer migration-gap docs, or design a hardened optional relay/browser-backend API coordinated with https://github.com/openclaw/openclaw/issues/19289 and informed by FSB/ClawHosters-style scoped bridge references.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but source inspection proves the current limitation: `driver: "extension"` is rejected, relay startup is a no-op, and doctor migrates legacy config to `existing-session`.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Directly restoring the old core relay may not be the best maintainable path; a versioned browser backend/plugin API with scoped tab ownership, visible logging, consent, and security review is likely safer if maintainers want to support this workflow.
