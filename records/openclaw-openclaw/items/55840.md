## Summary

Keep open. Current main still intentionally removes the Chrome Extension Relay, rejects `driver: "extension"`, and migrates legacy relay config to Chrome MCP `existing-session`; restoring the requested existing-tab relay, auto-reattach, video-control, and chat-automation workflows needs maintainer browser-strategy and security direction.

## What This Changes

Restore Chrome Extension Relay as an optional browser driver with port 18792 relay support, packaged extension assets, legacy config compatibility, and migration guidance.

## Best Possible Solution

Maintainers should either keep the legacy relay retired with clear migration-limit docs, or design a vetted optional browser-backend/plugin path and maintain a security-reviewed Chrome Extension Relay with tests, packaging, docs, and migration behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main rejects `driver: "extension"`, tests assert that rejection, and doctor rewrites legacy relay config to `existing-session`.

## Solution Assessment

Unclear. Directly restoring the old relay in core may not be the narrowest maintainable solution because the project direction favors optional plugin surfaces, while a browser-backend driver seam is still a product/API decision.
