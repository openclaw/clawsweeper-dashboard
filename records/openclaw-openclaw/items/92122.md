## Summary

Keep this open. Current main still has a plausible live-only WebChat ordering race: pre-tool assistant stream segments can be timestamped with the client clock while the matching tool card keeps the earlier event timestamp, and refresh/history reload can rebuild the durable transcript in the correct order.

## What This Changes

The issue asks for WebChat live streaming to keep assistant text, tool calls, tool results, and later assistant text in the same temporal order shown after refresh.

## Best Possible Solution

Fix the Control UI live stream/tool projection so assistant segments keep causal order with their matching tool cards even when event and client timestamps differ, then add a regression test for the delayed-tool-event case.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows a plausible current-main path where timestamp sorting can move a tool card ahead of the assistant stream segment it follows.

## Solution Assessment

Yes, the best fix appears to be in the Control UI live projection layer. The durable chat.history path already reloads into the correct order, so changing the gateway transcript contract would be broader than needed.
