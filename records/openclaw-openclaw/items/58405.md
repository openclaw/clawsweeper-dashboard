## Summary

Keep this PR open. Current main and the latest release do not implement per-request OpenResponses skill selection, but the submitted diff is not merge-ready because it can widen an agent's configured skill allowlist and still lacks real gateway behavior proof.

## What This Changes

The PR adds an optional `skills` request-body field to `/v1/responses`, threads it into agent command skill snapshot creation, and documents it as an OpenClaw-specific extension.

## Best Possible Solution

Define request `skills` as an ephemeral narrowing selector over the selected agent's effective configured allowlist, reject or intersect outside names, and add focused tests plus redacted live gateway proof.

## Reproduction Assessment

Yes for the introduced PR defect by source inspection: configure an agent with a narrow `agents.list[].skills`, keep another workspace skill available, and send `/v1/responses` with that other skill in `skills`. The PR path uses the request list as the complete snapshot filter.

## Solution Assessment

No, not as written. The feature direction is useful, but replacing the configured final allowlist is not the safest contract unless maintainers explicitly approve request-time elevation semantics.
