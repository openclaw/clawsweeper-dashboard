## Summary

Keep open; current main does not implement the requested `/v1/responses` field, but this PR should not merge as-is because the request body can replace the effective agent skill allowlist and expose skills outside the configured agent boundary.

## What This Changes

The PR adds an OpenClaw-specific optional `skills` array to `/v1/responses`, threads it into Gateway agent execution as an ephemeral skill snapshot override, and documents the field.

## Best Possible Solution

Implement per-request `skills` as an ephemeral narrowing selector constrained by the effective agent allowlist, or explicitly approve and document an elevation contract before merging; then add focused tests and a changelog entry.

## Reproduction Assessment

Yes. Configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR path uses the request list as the full snapshot filter.

## Solution Assessment

No. The feature shape is useful, but replacing the configured allowlist is not the narrowest maintainable contract for a request-scoped field that affects prompt skill visibility and skill env injection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep request skills within the agent allowlist:** `src/agents/agent-command.ts:505-507`
  - body: The override path uses the request body as the whole skill filter whenever `skills` is present. That lets a caller select skills outside `agents.list[].skills`, even though the docs describe that list as the final set for an agent and snapshot membership also drives skill env injection. Intersect with the effective agent filter, or reject names outside it, unless this API gets an explicit elevation contract.
  - confidence: 0.9

  - confidence: 0.92
