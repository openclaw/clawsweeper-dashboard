## Summary

Keep open: this is a still-relevant feature PR, but it should not merge as-is because the request-level `skills` override replaces the configured agent skill allowlist and can widen the run's available skills and skill-specific env exposure. The external PR also has only test/typecheck proof, not real gateway behavior proof.

## What This Changes

The PR adds an optional OpenClaw-specific `skills` array to `/v1/responses`, threads it into agent command execution as an ephemeral skill snapshot override, and documents the request field.

## Best Possible Solution

Implement `skills` as an ephemeral narrowing selector by default: intersect with or reject names outside the effective configured agent allowlist, preserve non-persistence and current snapshot refresh semantics, document the extension, add targeted tests, and provide real gateway proof.

## Reproduction Assessment

Yes. Configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR path uses the request list as the full snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the final configured allowlist is not the narrowest maintainable or security-preserving contract; the request field should narrow the effective allowlist unless maintainers explicitly approve elevation semantics.
