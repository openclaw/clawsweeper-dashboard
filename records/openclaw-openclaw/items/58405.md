## What This Changes

The PR adds an optional `skills` array to `/v1/responses`, threads it into agent skill snapshot construction, and documents it as an OpenClaw-specific per-request skill allowlist override.

## Best Possible Solution

Define request `skills` as an ephemeral narrowing selector over the selected agent's effective configured allowlist, reject or intersect outside names, then add focused tests and redacted live gateway proof.

## Reproduction Assessment

Yes. For the introduced PR defect, configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR path uses the request list as the complete snapshot filter by source inspection.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; request skills should narrow or reject names outside the effective agent allowlist unless maintainers explicitly approve wider semantics.
