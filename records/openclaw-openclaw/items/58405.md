## Summary

Keep open: current main does not already implement this `/v1/responses` request field, and the PR still needs a security/contract decision because the proposed override can widen access beyond the selected agent’s configured skill allowlist.

## What This Changes

The PR adds an optional OpenClaw-specific `skills` array to `/v1/responses`, threads it into agent command execution as a per-request skill snapshot selector, and documents the new request field.

## Best Possible Solution

Define the request `skills` field as an ephemeral narrowing selector over the selected agent’s effective configured allowlist, unless maintainers explicitly approve wider request-time elevation semantics.

## Reproduction Assessment

Yes for the introduced PR defect by source inspection: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill. The PR path uses the request list as the full snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; the request field should narrow or reject names outside the effective agent allowlist unless maintainers approve wider semantics.
