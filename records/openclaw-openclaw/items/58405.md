## Summary

Keep open, but do not merge as-is. Current main still lacks the requested `/v1/responses` `skills` request field, and the feature direction is plausible, but this patch can replace a configured agent skill allowlist from the request body and expose skills the selected agent was configured not to use; it also omits the required changelog entry for a user-facing API feature.

## What This Changes

The PR adds an optional OpenClaw-specific `skills` array to `POST /v1/responses`, threads it through Gateway agent command execution, rebuilds an ephemeral skill snapshot, and documents the field.

## Best Possible Solution

Implement per-request skills as an ephemeral narrowing selector by default: accept the field, pass it into agent command execution, rebuild but do not persist the snapshot for that run, and intersect it with the selected agent's effective configured allowlist or reject requested names outside that allowlist. If maintainers intentionally want elevation semantics, add an explicit permission model, docs, tests, and release notes for that broader capability.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR's override branch uses the request filter as the whole snapshot filter.

## Solution Assessment

No. The feature is useful, but replacing the configured allowlist is not the narrowest safe implementation; constraining the request field by the effective agent allowlist preserves the existing security and docs contract.
