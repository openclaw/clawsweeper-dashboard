## Summary

Keep open: current main does not implement the new `/v1/responses` field, and the PR still needs real behavior proof plus a security/API decision because it can replace an agent's configured skill allowlist.

## What This Changes

The PR adds an optional OpenClaw-specific `skills` array to `/v1/responses`, threads it into agent command skill snapshot construction, and documents ephemeral per-request behavior.

## Best Possible Solution

Implement `skills` as an ephemeral narrowing selector over the effective configured agent allowlist unless maintainers approve elevation semantics, then add focused regression tests, align the docs, and provide redacted real gateway proof.

## Reproduction Assessment

Yes, source-reproducible for the PR-introduced defect: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR path uses that request list as the full snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; the request field should narrow or reject names outside the effective agent allowlist unless maintainers explicitly approve wider semantics.
