## Summary

Keep open with blocking review findings. The feature is not implemented on current main and remains a plausible OpenResponses enhancement, but the proposed implementation treats the request field as a replacement for configured agent skill allowlists, which can widen a restricted agent's available skills and related skill env handling.

## What This Changes

The PR adds an optional OpenClaw-specific `skills` array to `POST /v1/responses`, threads it through Gateway agent command execution, uses it for an ephemeral skill snapshot, and documents the request field.

## Best Possible Solution

Implement per-request skills as an ephemeral narrowing selector by default: accept the field, pass it into the agent command, rebuild but do not persist the snapshot for that run, and either intersect it with the effective configured agent allowlist or reject requested names outside that allowlist. If maintainers intentionally want elevation semantics, add an explicit permission contract, docs, tests, and release notes for that broader capability.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR's new branch uses the request filter instead of `resolveAgentSkillsFilter`, and snapshot membership drives prompt and skill env handling.

## Solution Assessment

No. The overall feature direction is reasonable, but replacing the configured allowlist is not the safest maintainable fix; the request field should be constrained by the effective agent allowlist unless maintainers explicitly approve a separate elevation model.
