## Summary

Keep open: current main does not implement this `/v1/responses` request field, but the PR’s current implementation replaces the configured agent skill allowlist with caller-supplied skills, which is a security/API contract blocker until maintainers choose narrowing versus elevation semantics.

## What This Changes

The PR adds an optional `skills` field to `/v1/responses`, threads it through gateway handling into agent command execution, and documents it as an OpenClaw-specific per-request skill allowlist override.

## Best Possible Solution

Implement `skills` as an ephemeral narrowing selector by default: intersect with or reject names outside the effective configured agent allowlist, preserve non-persistence and current snapshot refresh semantics, document the extension, add focused tests, and provide real gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR path uses the request list as the full snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the final configured allowlist is not the narrowest maintainable or security-preserving contract; the request field should narrow the effective allowlist unless maintainers explicitly approve elevation semantics.
