## What This Changes

The PR adds an optional OpenClaw-specific `skills` array to `/v1/responses`, threads it into agent command skill snapshot creation, and documents the request field.

## Best Possible Solution

Define request `skills` as an ephemeral narrowing selector over the selected agent's effective configured allowlist, reject or intersect outside names, cover allowed/outside/empty/omitted and non-persistence cases, then add redacted live gateway proof.

## Reproduction Assessment

Yes for the introduced PR defect by source inspection. Configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR path uses that request list as the complete snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; request skills should narrow or reject names outside the effective agent allowlist unless maintainers explicitly approve elevation semantics.
