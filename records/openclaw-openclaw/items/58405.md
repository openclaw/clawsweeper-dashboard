## Summary

Keep open. The feature is not implemented on current main, but this PR should not merge as-is because the request-level skill list can bypass configured agent allowlists and the external PR still lacks real gateway behavior proof.

## What This Changes

The PR adds a `skills` field to `/v1/responses`, forwards it as `skillsOverride` into agent command execution, rebuilds non-persisted skill snapshots when present, and documents the new request field.

## Best Possible Solution

Define `skills` as an ephemeral narrowing selector over the effective configured agent allowlist unless maintainers explicitly approve elevation semantics; add regression coverage, align docs with that contract, refresh the branch, and require redacted real gateway proof.

## Reproduction Assessment

Yes for the introduced PR bug by source inspection: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill. The PR path uses the request list as the full snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; the request field should narrow or reject names outside the effective agent allowlist unless maintainers approve wider semantics.
