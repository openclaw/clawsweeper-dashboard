## Summary

Keep open: current main still does not implement a per-request OpenResponses skills field, but the PR still replaces the selected agent's configured skill allowlist with caller-supplied names and still lacks real gateway proof with real skills.

## What This Changes

The PR adds an OpenClaw-specific `skills` request field to `/v1/responses`, threads it through gateway agent execution as `skillsOverride`, avoids persisting override snapshots, and documents the new field.

## Best Possible Solution

Define `skills` as an ephemeral narrowing selector over the effective configured agent allowlist unless maintainers explicitly approve elevation semantics, then add regression coverage, align docs, and require redacted real gateway proof.

## Reproduction Assessment

Yes, by source inspection for the introduced PR bug: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill. The PR path uses the request list as the full snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; the request field should narrow or reject names outside the effective agent allowlist unless maintainers approve wider semantics.
