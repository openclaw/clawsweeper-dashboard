## Summary

Keep open: the feature is not implemented on current main and may be useful, but this PR currently widens configured agent skill allowlists from the request body and lacks real gateway proof, so it should not merge as-is.

## What This Changes

The PR adds an optional OpenClaw-specific `skills` request field to `/v1/responses`, threads it into agent command skill snapshot selection, and documents it as an ephemeral per-request override.

## Best Possible Solution

Implement `skills` as an ephemeral narrowing selector over the effective configured agent allowlist unless maintainers explicitly approve elevation semantics, add focused regression coverage, align the docs, and provide redacted real gateway proof.

## Reproduction Assessment

Yes for the introduced PR bug by source inspection: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill. The PR path uses the request list as the full snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; the request field should narrow or reject names outside the effective agent allowlist unless maintainers approve wider semantics.
