## Summary

Keep open: the feature is useful and current main does not already implement it, but the PR currently lets request bodies widen a restricted agent's skill snapshot and still lacks real behavior proof from a live `/v1/responses` request.

## What This Changes

Adds an optional OpenClaw-specific `skills` array to `/v1/responses`, threads it into agent command skill snapshot building, and documents it as an ephemeral per-request override.

## Best Possible Solution

Implement `skills` as an ephemeral narrowing selector over the effective configured agent allowlist unless maintainers explicitly approve elevation semantics, add focused gateway/agent regression tests, align the docs, and provide redacted real gateway proof.

## Reproduction Assessment

Yes, source-reproducible for the introduced PR bug: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR path uses the request list as the full snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; the request field should narrow or reject names outside the effective agent allowlist unless maintainers approve wider semantics.
