## Summary

Keep open. Current main still lacks the request field, and the PR is useful directionally, but it is not merge-ready because the override can widen a configured agent skill allowlist and the user-facing feature has no changelog entry.

## What This Changes

The PR adds an OpenClaw-specific `skills` array to `POST /v1/responses`, threads it into Gateway agent command execution as an ephemeral skill snapshot override, and documents the request field.

## Best Possible Solution

Implement per-request skills as an ephemeral narrowing selector by default: accept the field, rebuild but do not persist the run snapshot, and intersect with or reject names outside the effective configured agent allowlist unless maintainers explicitly approve elevation semantics.

## Reproduction Assessment

Yes. Configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR path uses the request filter as the full snapshot filter.

## Solution Assessment

No. The feature shape is reasonable, but replacing the configured allowlist is not the safest maintainable contract; the request field should be constrained by the effective agent allowlist unless an explicit elevation model is approved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep request skills within the configured allowlist:** `src/agents/agent-command.ts:505-507`
  - body: The override path uses the request body as the whole skill filter whenever `skills` is present. That lets a caller select skills outside `agents.list[].skills`, even though the docs describe that list as the final set for an agent; snapshot membership also drives skill env injection. Intersect with the effective agent filter, or reject names outside it, unless this API gets an explicit elevation contract.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff can widen a restricted agent's skill set from a request body, so it needs API/security policy attention before merge.

Concerns:

- **[high] Request skills can bypass agent allowlists:** `src/agents/agent-command.ts:505`
  - body: The proposed `skillsOverride` path replaces the configured agent skill filter rather than constraining it. Since skill snapshots feed prompt construction and per-skill env override application, this can expose skills and skill-specific host env config that the selected agent's allowlist was meant to withhold.
