## Summary

Keep open for maintainer review: current main still does not implement per-request skills on `/v1/responses`, but this PR introduces an allowlist-elevation/security concern and lacks required real behavior proof.

## What This Changes

The PR adds an optional OpenClaw-specific `skills` request field to `/v1/responses`, threads it into agent command execution as `skillsOverride`, avoids persisting override snapshots, and documents the new field.

## Best Possible Solution

Define `skills` as an ephemeral narrowing selector over the effective configured agent allowlist unless maintainers explicitly approve elevation semantics; add regression coverage, align docs with that contract, and require redacted real gateway proof.

## Reproduction Assessment

Yes, by source inspection for the introduced PR bug: configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill. The PR path uses the request list as the full snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; the request field should narrow or reject names outside the effective agent allowlist unless maintainers approve wider semantics.
