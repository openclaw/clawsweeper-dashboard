## Summary

Keep this PR open, but it is not merge-ready: current main does not implement the proposed request field, and the branch would let request-level `skills` replace the selected agent's configured final allowlist while also lacking real gateway proof.

## What This Changes

The branch adds an optional `skills` field to `/v1/responses`, validates and documents it, and threads it into agent command execution as a per-request skill snapshot override.

## Best Possible Solution

Define request `skills` as an ephemeral narrowing selector over the selected agent's effective configured allowlist, or explicitly document and approve wider request-time elevation semantics before merge.

## Reproduction Assessment

Yes for the introduced PR defect by source inspection. Configure an agent with `agents.list[].skills: ["safe"]`, keep another workspace skill available, and send `/v1/responses` with `skills` naming the other skill; the PR path uses the request list as the complete snapshot filter.

## Solution Assessment

No. The feature direction is useful, but replacing the configured final allowlist is not the safest contract; request skills should narrow or reject names outside the effective agent allowlist unless maintainers explicitly approve wider semantics.
