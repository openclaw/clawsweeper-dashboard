## Summary

Keep this open. Current main still has the received-hook metadata escape hatch and Telegram already computes visible reply/forward context, but the shared received-hook mappers do not project the requested `forward`, `viaBot`, or `replyTo` metadata, and no duplicate or fix PR was found.

## What This Changes

Request to populate Telegram received-hook metadata with forwarded-message, inline-bot, and reply-parent provenance from inbound Bot API updates.

## Best Possible Solution

Add a bounded Telegram-owned provenance object to received-hook metadata, reusing visible normalized reply/forward fields where possible, adding `viaBot` from the inbound update, truncating parent text, and covering both plugin and internal received-hook paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Telegram inbound update with `forward_origin`, `via_bot`, or `reply_to_message` reaches received-hook mappers that currently emit only generic metadata.

## Solution Assessment

Yes. Using the existing `metadata` escape hatch is the narrowest maintainable direction; adding top-level SDK fields or new Bot API fetches would be broader than needed.
