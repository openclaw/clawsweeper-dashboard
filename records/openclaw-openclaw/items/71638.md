## Summary

Keep this open. Current main still exposes hook metadata and Telegram already derives visible reply and forwarded-origin context for inbound agent payloads, but the message-received hook mappers do not project Telegram forward, via-bot, or reply provenance into `metadata`.

## What This Changes

Request to add Telegram forward, via-bot, and reply provenance to `MessageReceivedHookEvent.context.metadata` for inbound Telegram messages.

## Best Possible Solution

Add a Telegram-owned provenance projection that feeds the existing hook `metadata` field with structured `forward`, `viaBot`, and bounded `replyTo` details for both plugin and internal received-hook paths, while avoiding Telegram-specific top-level SDK fields.

## Reproduction Assessment

Yes. A focused unit test can construct a Telegram inbound message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message`, then assert current main's received-hook metadata lacks `forward`, `viaBot`, and `replyTo`; I did not run tests because this review was read-only.

## Solution Assessment

Yes. Using the existing hook `metadata` field is the narrowest maintainable fix because plugin docs already reserve it for channel-specific extras, and the sensitive Telegram-specific derivation belongs inside the Telegram plugin boundary.
