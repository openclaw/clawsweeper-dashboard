## Summary

Keep this issue open. Current main still has the metadata hook escape hatch and already derives Telegram reply/forward context for agent-facing behavior, but the received-hook mapping paths do not project `metadata.forward`, `metadata.viaBot`, or `metadata.replyTo` for hook consumers.

## What This Changes

Expose Telegram forward, via-bot, and reply provenance on inbound Telegram `MessageReceivedHookEvent.context.metadata`.

## Best Possible Solution

Add a Telegram-owned, bounded provenance projection into received-hook metadata for both normal dispatch and mention-skipped ingest, keeping core hook fields channel-agnostic.

## Reproduction Assessment

Yes. A focused unit test can feed a Telegram message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message` through the Telegram inbound context and received-hook mapper paths, then assert current main lacks `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo`.

## Solution Assessment

Yes. The existing `metadata` escape hatch is the narrowest maintainable surface for Telegram-specific provenance; the safer implementation is a bounded camelCase projection rather than raw Bot API passthrough or new generic top-level hook fields.
