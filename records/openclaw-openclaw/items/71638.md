## Summary

Keep this issue open. Current main still supports the hook metadata escape hatch and Telegram already computes visible reply/forward context for the agent turn, but the received-hook mappers do not expose forward, via-bot, or reply provenance in `metadata`, and there is no open paired implementation PR in the provided context.

## What This Changes

Expose Telegram forward, via-bot, and reply provenance on `MessageReceivedHookEvent.context.metadata` for received message hooks.

## Best Possible Solution

Add a Telegram-owned provenance projection that feeds both plugin and internal received-hook paths through the existing `metadata` escape hatch, covers normal dispatch and mention-skipped ingest, includes `via_bot`, and keeps core SDK fields channel-agnostic.

## Reproduction Assessment

Yes. A focused unit test can build a Telegram inbound message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message`, then assert current main's received hook metadata lacks `forward`, `viaBot`, and `replyTo`; this review did not run tests because it was required to keep the checkout read-only.

## Solution Assessment

Yes. Using `metadata` is the narrowest maintainable solution because the SDK and docs already reserve it for channel-specific extras, while Telegram-specific provenance belongs in the Telegram plugin boundary rather than new generic top-level hook fields.
