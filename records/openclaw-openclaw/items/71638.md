## Summary

Keep this open. Current main still has the hook metadata escape hatch and Telegram already derives visible reply/forward context for the inbound agent payload, but the received-hook mappers do not project Telegram forward, via-bot, or reply provenance into `metadata`; there is no protected label or open paired fix PR in the provided context.

## What This Changes

Request to add Telegram forward, via-bot, and reply provenance into `MessageReceivedHookEvent.context.metadata` for received Telegram messages.

## Best Possible Solution

Add a Telegram-owned provenance projection that reuses the existing hook `metadata` field for `forward`, `viaBot`, and bounded `replyTo` details, feeds both plugin and internal received-hook paths through the shared mapper, and avoids adding Telegram-specific top-level SDK fields.

## Reproduction Assessment

Yes. A focused unit test can construct a Telegram inbound message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message`, then assert current main's received hook metadata lacks `forward`, `viaBot`, and `replyTo`. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. Using `metadata` is the narrowest maintainable fix because plugin docs already reserve it for channel-specific extras, while Telegram-specific provenance should stay in the Telegram plugin boundary.
