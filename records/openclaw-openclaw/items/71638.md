## Summary

Keep this issue open because current main still has the hook metadata escape hatch, but the Telegram received-hook path does not carry `forward`, `viaBot`, or `replyTo` metadata through either the plugin or internal hook mappers.

## What This Changes

Expose Telegram forward, via-bot, and reply provenance on `MessageReceivedHookEvent.context.metadata` without changing existing fields.

## Best Possible Solution

Add a Telegram-owned bounded provenance projection through the canonical inbound hook context into plugin and internal received-hook mappers, including normal dispatch and mention-skipped ingest.

## Reproduction Assessment

Yes. A focused unit test can build a Telegram inbound message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message`, then assert current main's received-hook metadata lacks `forward`, `viaBot`, and `replyTo`.

## Solution Assessment

Yes. Using existing hook `metadata` is the narrowest maintainable solution because the SDK already reserves it for channel-specific extras and this data is Telegram-owned provenance.
