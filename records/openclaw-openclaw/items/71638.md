Keep this issue open. Current main still has the generic hook metadata surface and Telegram already derives reply/forward provenance for agent context, but the received-hook paths do not expose `metadata.forward`, `metadata.viaBot`, or `metadata.replyTo` to hook consumers.

## What This Changes

Request additive Telegram provenance fields for forwarded, via-bot, and reply context on `MessageReceivedHookEvent.context.metadata`.

## Best Possible Solution

Add a Telegram-owned, additive provenance projection into received-hook metadata for normal dispatch and mention-skipped ingest, with bounded reply text and no generic top-level hook fields.

Yes. Source inspection shows current main maps Telegram received hooks without `forward`, `viaBot`, or `replyTo`; a focused unit test can feed a Telegram message with those Bot API fields and assert the metadata is absent today.

## Solution Assessment

Yes. The existing `metadata` escape hatch is the narrowest maintainable path because the fields are Telegram-specific provenance and should not become generic required hook fields.
