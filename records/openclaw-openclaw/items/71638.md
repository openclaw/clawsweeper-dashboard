## Summary

Keep this issue open. Current main still exposes generic received-hook metadata and already derives Telegram reply/forward context for agent-facing behavior, but the plugin/internal received-hook paths do not project `metadata.forward`, `metadata.viaBot`, or `metadata.replyTo` for hook consumers.

## What This Changes

The issue requests Telegram received-message hooks to include forward, via-bot, and reply-to provenance from inbound Telegram updates.

## Best Possible Solution

Add a Telegram-owned, additive provenance projection into received-hook metadata for both normal dispatch and mention-skipped ingest, keeping core hook fields channel-agnostic and reply details bounded.

## Reproduction Assessment

Yes. A focused test can feed a Telegram message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message` through the Telegram context and received-hook mapper paths, then assert current main lacks `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo`; tests were not run because this review is read-only.

## Solution Assessment

Yes. Using the existing `metadata` escape hatch is the narrowest maintainable path because the data is Telegram-specific provenance and does not need new generic top-level hook fields.
