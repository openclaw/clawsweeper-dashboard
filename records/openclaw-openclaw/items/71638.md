## Summary

Keep this open. Current main still exposes the generic hook metadata escape hatch, but the shared received-hook mapper and Telegram ingest paths do not carry Telegram forward, via-bot, or reply provenance into `metadata`, even though Telegram already derives visible reply and forward context for the inbound session payload.

## What This Changes

Surface Telegram `forward_from`/`forward_origin`, `via_bot`, and `reply_to_message` provenance on received message hook metadata.

## Best Possible Solution

Add an additive, Telegram-owned provenance projection that reuses the existing hook `metadata` field, respects current context-visibility gates, bounds reply text, and feeds both plugin and internal received-hook paths through the shared mapper.

## Reproduction Assessment

Yes. A focused unit test can construct a Telegram inbound message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message`, then assert that current main's received hook payload lacks `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo`.

## Solution Assessment

Yes. Using `metadata` is the narrowest maintainable path because it keeps Telegram-specific provenance out of cross-channel top-level hook fields while matching the existing plugin SDK guidance for channel-specific extras.
