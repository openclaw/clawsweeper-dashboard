## Summary

Keep this issue open. Current main still exposes generic hook metadata and Telegram already normalizes reply/forward context for agent prompts, but `message_received` hook payloads do not carry `metadata.forward`, `metadata.viaBot`, or `metadata.replyTo`.

## What This Changes

Expose Telegram forward, inline-bot, and reply provenance on `MessageReceivedHookEvent.context.metadata` for inbound Telegram messages.

## Best Possible Solution

Add a Telegram-owned, additive metadata projection for received hooks, reusing existing reply/forward normalization where possible and keeping the payload bounded and provider-specific.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: feed Telegram inbound context with `forward_origin`, `via_bot`, or `reply_to_message` and the current shared received-hook mappers will emit only generic metadata, not the requested provenance keys.

## Solution Assessment

Yes. The existing `metadata` escape hatch is the narrowest maintainable surface; adding generic top-level hook fields or new Bot API calls would be broader than needed.
