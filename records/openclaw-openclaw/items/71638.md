## Summary

Keep this open. Current main still has the hook metadata escape hatch and Telegram already derives reply/forward provenance for agent context, but `message_received` hook payloads do not carry Telegram `forward_origin`/legacy forward fields, `via_bot`, or `reply_to_message` provenance into `metadata`; no canonical open PR or duplicate was found in the searched refs.

## What This Changes

Request additive Telegram `MessageReceivedHookEvent.context.metadata` fields for forwarded-message provenance, inline `via_bot` provenance, and bounded reply parent context.

## Best Possible Solution

Add a Telegram-owned provenance projection that feeds the existing `message_received` hook metadata field through the shared mapper, reuses the current context-visibility gates, bounds reply text, and covers both normal inbound dispatch and mention-skipped ingest hooks without adding Telegram-specific top-level core fields.

## Reproduction Assessment

Yes. A focused unit test can construct a Telegram inbound message with `forward_origin`, `via_bot`, and `reply_to_message`, dispatch it through the received-hook path, and assert that current main lacks `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo`.

## Solution Assessment

Yes. Using `metadata` is the narrowest maintainable solution because the requested data is Telegram-specific provenance while the SDK already documents `message_received.metadata` for channel-specific extras.
