## Summary

Keep this issue open. Current main still has the hook metadata escape hatch and Telegram already derives visible reply/forward context for the agent turn, but the plugin/internal received-hook mappers still do not expose `forward`, `viaBot`, or `replyTo` in metadata.

## What This Changes

Expose Telegram forward, via-bot, and reply provenance on received-message hook metadata.

## Best Possible Solution

Add a Telegram-owned provenance projection for both plugin and internal received-hook paths, including normal dispatch and mention-skipped ingest, while keeping core hook fields channel-agnostic.

## Reproduction Assessment

Yes. A focused unit test can build a Telegram inbound message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message`, then assert current main's received hook metadata lacks `forward`, `viaBot`, and `replyTo`; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Using existing hook `metadata` is the narrowest maintainable solution because the SDK already reserves it for channel-specific extras, and this data is Telegram-owned provenance rather than a generic top-level hook contract.
