## Summary

Keep this open. Current main still exposes only generic `message:received` metadata while Telegram already derives reply and forward provenance for the inbound agent context, and there is still no `via_bot` extraction path. This remains a valid, narrow additive Telegram hook repair rather than a close candidate.

## What This Changes

The issue requests that Telegram `message:received` hook events include forward provenance, inline `via_bot`, and bounded reply context in `MessageReceivedHookEvent.context.metadata`.

## Best Possible Solution

Add a Telegram-owned provenance metadata builder for `forward`, `viaBot`, and bounded `replyTo`, thread that data through the canonical inbound hook context, and merge it into both plugin and internal received-hook payloads, including the no-mention ingest path.

## Reproduction Assessment

Yes. A focused test can construct a Telegram inbound message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message`, then assert current main emits received-hook metadata without `forward`, `viaBot`, or `replyTo`.

## Solution Assessment

Yes. Using the existing hook `metadata` escape hatch is the narrowest maintainable solution because it keeps Telegram-specific provenance out of top-level cross-channel SDK fields while preserving backwards compatibility.
