## Summary

Keep this open. Current main still has the reported Telegram received-hook provenance gap: the hook metadata escape hatch exists, and Telegram already derives visible reply and forward context for inbound processing, but the canonical hook context and both received-hook mappers only carry generic routing/sender metadata and there is no via-bot extraction.

## What This Changes

Request to add Telegram forward, via-bot, and reply provenance to received message hook metadata for downstream consumers.

## Best Possible Solution

Add a narrow, additive provenance payload under received-hook metadata, sourced from Telegram-owned extraction and then carried through the shared canonical hook mapper to both plugin and internal received-hook events. Keep Telegram-specific details out of top-level cross-channel fields, bound reply text, and cover both normal dispatch and ingest-only paths.

## Reproduction Assessment

Yes. A focused unit test can construct a Telegram inbound message with forward_origin or legacy forward fields, via_bot, and reply_to_message, then assert current main emits received-hook metadata without metadata.forward, metadata.viaBot, or metadata.replyTo.

## Solution Assessment

Yes. Using the existing metadata field is the narrowest maintainable direction because it is additive and keeps provider-specific Telegram provenance scoped to provider metadata; the safer implementation should reuse the current Telegram visibility checks instead of exposing raw Bot API payloads wholesale.
