## Summary

Keep this open. Current main still has the reported Telegram received-hook metadata gap: reply and forward provenance is already derived for Telegram inbound context, but the shared received-hook mapper and the Telegram ingest-only path still emit only generic routing and sender metadata, and there is no via-bot extraction.

## What This Changes

The issue asks the Telegram plugin to populate received-message hook metadata with forwarded-message, inline-bot, and reply context from raw Telegram updates.

## Best Possible Solution

Add a generic inbound-hook metadata passthrough and populate it from Telegram-owned provenance extraction for `forward`, `viaBot`, and bounded `replyTo` data, then merge it into both plugin and internal received-hook payloads with tests for normal dispatch and mention-skip ingest.

## Reproduction Assessment

Yes. A focused test can construct a Telegram inbound message containing `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message`, then assert that current main's received hook payload lacks `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo`.

## Solution Assessment

Yes. An additive metadata payload is the narrowest maintainable solution because it keeps Telegram-specific provenance out of top-level cross-channel SDK fields while using the existing hook metadata escape hatch.
