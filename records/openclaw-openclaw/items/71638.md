## Summary

Keep this open. Current main still has the hook metadata escape hatch and Telegram already computes reply/forward context for prompt/session handling, but the received-hook mappers and Telegram emit paths still do not project the requested `forward`, `viaBot`, or `replyTo` metadata, and I found no canonical duplicate or fix PR.

## What This Changes

Surface Telegram forward, inline-bot, and reply provenance on received message hook metadata.

## Best Possible Solution

Add a bounded Telegram-owned provenance projection into received-hook metadata, reusing existing reply/forward normalization where appropriate and avoiding new Bot API calls or top-level SDK fields.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: feed a Telegram inbound update with `forward_origin`, `via_bot`, or `reply_to_message`, and the received-hook mappers will emit only generic metadata fields.

## Solution Assessment

Yes. Using the existing `metadata` escape hatch is the narrowest maintainable direction; adding raw top-level hook fields or new Telegram API fetches would be broader than needed.
