## Summary

Keep this open. Current main still has the hook metadata escape hatch and Telegram already normalizes reply/forward context for prompts and cached message context, but received-hook events still omit the requested `forward`, `viaBot`, and `replyTo` metadata and no canonical duplicate or fix PR was found.

## What This Changes

The issue asks to project Telegram forward, inline-bot, and reply provenance into `MessageReceivedHookEvent` metadata.

## Best Possible Solution

Add a Telegram-owned, bounded metadata projection on received hooks using the existing hook metadata escape hatch and the existing Telegram reply/forward normalization where it fits.

## Solution Assessment

Yes. The requested `metadata` projection is the narrowest maintainable direction; adding generic top-level hook fields or extra Bot API fetches would be broader than needed.
