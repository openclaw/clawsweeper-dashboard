The issue requests Telegram MessageReceivedHookEvent.context.metadata fields for forwarded-message, inline-via-bot, and reply provenance from inbound Telegram messages.

## Best Possible Solution

Add bounded, visibility-aware Telegram provenance metadata through the canonical inbound hook path, covering both normal dispatch and silent ingest with focused tests for forward, viaBot, reply, redaction, and truncation.

## Solution Assessment

Not as written. Exposing provenance is the right direction, but the best solution should normalize and bound the fields using existing Telegram visibility/redaction rules rather than passing raw Bot API objects directly.
