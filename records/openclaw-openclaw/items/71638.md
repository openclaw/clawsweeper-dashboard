## Summary

Keep this issue open. Current main still has the hook metadata escape hatch and Telegram already derives reply and forwarded context for the agent turn, but the received-hook mapper paths do not expose `metadata.forward`, `metadata.viaBot`, or `metadata.replyTo` for downstream hook consumers.

## What This Changes

The request is to populate Telegram received-hook metadata with forward, via-bot, and reply-to provenance already present on inbound Telegram updates.

## Best Possible Solution

Add a Telegram-owned, additive provenance projection into received-hook metadata for both normal dispatch and mention-skipped ingest, keeping core hook fields channel-agnostic and reply details bounded.

## Reproduction Assessment

Yes. A focused unit test can feed a Telegram message with `forward_origin` or legacy forward fields, `via_bot`, and `reply_to_message` through the Telegram/context mapper path and assert current main's received-hook metadata lacks `forward`, `viaBot`, and `replyTo`; tests were not run because this review is read-only.

## Solution Assessment

Yes. Using the existing `metadata` escape hatch is the narrowest maintainable solution because the data is Telegram-specific provenance and does not need new generic top-level hook fields.
