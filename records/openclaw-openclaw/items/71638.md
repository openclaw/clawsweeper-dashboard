## Summary

Keep this open. Current main has the generic hook metadata escape hatch and Telegram already normalizes reply/forward context for agent input, but the received hook payload still does not carry the requested `metadata.forward`, `metadata.viaBot`, or `metadata.replyTo` projection.

## What This Changes

Request additive Telegram-specific provenance metadata for forwarded, inline-bot, and replied-to inbound messages on `MessageReceivedHookEvent.context.metadata`.

## Best Possible Solution

Add a Telegram-owned, additive metadata projection for received hooks, reusing existing normalized reply/forward helpers where possible and keeping the payload bounded and provider-specific.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send a Telegram inbound update with `forward_origin`, `via_bot`, or `reply_to_message`; the current received-hook mappers emit only generic metadata and omit the requested provenance keys.

## Solution Assessment

Yes. The existing `metadata` escape hatch is the narrowest maintainable surface; adding generic top-level hook fields or new Bot API calls would be broader than needed.
