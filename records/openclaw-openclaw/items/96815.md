## Summary

Keep open: current main still contains the Codex app-server path that formats Codex `usageLimitExceeded` or usage-limit text into the reported Gateway message, and v2026.6.10 shipped the same mapping. The related merged fallback-classifier PR handles downstream fallback classification but does not prove or fix the same-account `codex exec` versus app-server false-positive claim.

## What This Changes

The issue asks OpenClaw to stop returning a false Codex subscription usage-limit error from Gateway when the same ChatGPT OAuth account, CODEX_HOME, and model work through `codex exec`.

## Best Possible Solution

Run a redacted live Codex app-server versus `codex exec` comparison with the same CODEX_HOME/profile/model, then fix the proven auth/home/runtime or error-classification boundary with focused regression coverage.

## Reproduction Assessment

Partly: source inspection reproduces the exact Gateway message when Codex app-server returns `UsageLimitExceeded` or usage-limit text, but this review did not live-reproduce the claimed false positive using the same ChatGPT OAuth account, CODEX_HOME, and model where `codex exec` succeeds.

## Solution Assessment

Unclear: suppressing or relabeling the usage-limit message is not the best fix until maintainers prove whether OpenClaw is selecting the wrong CODEX_HOME/auth profile/thread/runtime settings, or Codex app-server legitimately reports a different usage-limit state than `codex exec`.
