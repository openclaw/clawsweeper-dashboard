## Summary

Keep open. Current `main` and the latest release still write `openclaw backup create` archives through an adjacent random `.tmp` file and only clean it during normal JavaScript unwinding, so a timeout or termination can still strand the large temporary archive described in the report.

## What This Changes

The issue requests that `openclaw backup create` clean up large temporary archive files when backup creation is interrupted by timeout or process termination.

## Best Possible Solution

Backup creation should own the active temporary archive lifecycle: track the exact temp archive path during the write window, clean it on graceful interruption or explicit abort, preserve no-overwrite final publication, avoid deleting unrelated `*.tmp` files, and document unavoidable hard-kill limits.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the reported timeout scenario plus current-main source inspection: backup writes a large adjacent `.tmp` archive and has no backup-scoped signal or abort cleanup around the tar write.

## Solution Assessment

Yes. A narrow backup-owned cleanup path for graceful termination is the best maintainable fix; broad startup sweeping or deleting arbitrary `*.tmp` files would risk removing unrelated user files and would not address the active archive lifecycle.
