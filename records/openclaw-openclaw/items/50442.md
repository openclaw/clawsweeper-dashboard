- `P2`: The bug can consume substantial disk space during interrupted backups, but it is bounded to the backup command and has a workaround.
- `impact:crash-loop`: The reported failure mode is disk-space exhaustion from orphaned backup temp files, which can affect process and host availability.
- `impact:data-loss`: The affected surface is backup/state archival, where incomplete interrupted runs and disk pressure can compromise recoverability.

## Summary

Keep open: current main and v2026.5.18 still create the active backup tarball as an output-adjacent `.tmp` file and only remove it during normal unwinding, while the linked fixing PR remains open and conflicting.

## What This Changes

The issue asks `openclaw backup create` to avoid leaving large temporary archive files behind when a timeout or signal interrupts backup creation.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/80598 or an equivalent narrow backup fix that moves active temp archives out of the output directory, cleans up on graceful interruption, preserves signal exit semantics, and documents hard-kill limits.

## Reproduction Assessment

Yes, source-backed: run `openclaw backup create` long enough to enter the `tar.c` write to the output-adjacent temp archive, then terminate the process before normal unwinding. I did not run the live timeout in this read-only review, so this is not marked as directly reproduced.

## Solution Assessment

Yes for the direction: a backup-owned temp archive lifecycle and graceful-signal cleanup is the narrow maintainable fix. The currently linked PR or a replacement still needs to land before this issue can close.
