## Summary

Keep open: current main and the latest release still create the active backup archive as an adjacent `<output>.<uuid>.tmp` file and only remove it during normal `finally` unwinding. The open cleanup PR is the active implementation path, but it has not merged yet.

## What This Changes

The issue asks for `openclaw backup create` to stop leaving large adjacent `.tmp` archive files after timeouts or interrupted runs.

## Best Possible Solution

Review and either land the open backup temp cleanup PR or replace it with a stronger lifecycle cleanup that covers graceful signals/abort plus next-run stale cleanup while documenting unavoidable SIGKILL/crash limits.

## Reproduction Assessment

Yes at source level: current main constructs `<output>.<uuid>.tmp` and removes only that exact active temp path during normal unwinding. I did not run a live timeout or kill-path reproduction because this review is read-only.

## Solution Assessment

Unclear as a complete fix until the open PR is merged or replaced. A narrow next-run cleanup is useful, but the best end state should also define graceful signal/abort cleanup and document limits for SIGKILL/crash cases.
