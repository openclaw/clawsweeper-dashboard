## Summary

Keep open: current main and the latest release still leave the active backup archive as an adjacent random `.tmp` file whose cleanup depends on normal JavaScript unwinding, so timeout or graceful termination cleanup remains unfixed.

## What This Changes

`openclaw backup create` should clean up active `.tmp` archive files when interrupted or timeout-killed so repeated failures do not exhaust disk space.

## Best Possible Solution

Backup creation should own the active temporary archive lifecycle: track the exact temp archive path, clean it up on graceful termination or abort during the archive-write window, preserve no-overwrite publication, and document unavoidable hard-kill limits.

## Reproduction Assessment

Yes, source-backed: run `openclaw backup create` long enough to create the adjacent temp archive, then terminate it during `tar.c` before normal unwinding. I did not perform the live timeout in this read-only review, so this is not marked as reproduced.

## Solution Assessment

Yes. A narrow backup-owned graceful-interruption cleanup path is the maintainable fix; broad startup sweeps or deleting arbitrary `*.tmp` files would be riskier and the large-backup exclude PR addresses a separate size/stall problem.
