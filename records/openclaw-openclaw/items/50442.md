## Summary

Keep open. Current main still writes the active backup archive to an adjacent random `.tmp` file and only removes it during normal JavaScript unwinding, while the CLI path has no backup-scoped SIGINT/SIGTERM or abort cleanup for externally timed-out runs.

## What This Changes

The issue reports that `openclaw backup create` can leave large adjacent `.tmp` archives behind when the process is killed by a timeout.

## Best Possible Solution

Backup creation should own the active temp archive lifecycle: track the exact temp archive path, clean it on graceful interruption or abort during the archive-write window, preserve no-overwrite publication, and document hard-kill limits.

## Reproduction Assessment

Yes, by source-backed reproduction: run `openclaw backup create` long enough to create the adjacent temp archive, then terminate it with SIGTERM or a timeout before normal unwinding reaches `finally`. I did not run a live timeout because this review kept the checkout read-only and artifact-free.

## Solution Assessment

Yes. A narrow backup-owned graceful-interruption cleanup path is the best maintainable fix; broad startup sweeps or deleting arbitrary `*.tmp` files would risk unrelated user files and still miss the active archive lifecycle.

## Risks / Open Questions

- A graceful in-process fix can cover SIGINT, SIGTERM, and explicit abort paths, but SIGKILL, host crashes, and power loss cannot reliably run JavaScript cleanup.
- Signal cleanup must preserve normal CLI exit codes and remove scoped handlers after backup completion so unrelated commands do not inherit backup-specific behavior.
- The fix should track and remove only the active backup temp archive path, not sweep arbitrary user `.tmp` files.
