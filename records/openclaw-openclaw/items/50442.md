## Summary

Keep open: current main and the latest release still write the active backup archive as an output-adjacent random `.tmp` file and rely on normal `finally` cleanup, while the linked closing PR is still open and conflicting.

## What This Changes

The issue reports that `openclaw backup create` can leave large output-adjacent `.tmp` archives after timeouts, exhausting disk space.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/80598 or a narrow replacement that stages the active archive under the backup temp directory, handles interrupt cleanup, and preserves failure exit semantics.

## Reproduction Assessment

Yes, source-backed: current main writes the active tarball beside the requested output path and only removes it during normal `finally` cleanup. I did not run an interrupting live backup because this review is read-only, but the source path matches the reported timeout leak.

## Solution Assessment

Yes, the maintainable direction is a narrow backup-path fix: keep the active temp archive in the staging/system temp root and wire interrupt cleanup at the CLI boundary. The issue should remain open until the linked PR or an equivalent replacement merges.
