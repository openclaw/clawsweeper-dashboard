## Summary

Keep open: current main and v2026.5.19 still build the active backup archive as an output-adjacent `.tmp` file and clean it only during normal unwinding, while the specific fixing PR remains open and conflicting.

## What This Changes

The issue reports that interrupted `openclaw backup create` runs can leave large `.tmp` archive files behind and exhaust disk space.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/80598 or an equivalent narrow backup fix that moves active temp archives out of the output directory, cleans up graceful interruption, preserves signal exit semantics, and documents hard-kill limits.

## Reproduction Assessment

Yes, source-backed: current main writes the active tar output beside `outputPath` and removes it only in `finally`; terminating during the `tar.c` write can bypass that cleanup. I did not run a live timeout reproduction in this read-only review.

## Solution Assessment

Yes for the direction: backup-owned temp archive lifecycle and graceful-signal cleanup is the narrow maintainable fix. The linked PR or a replacement still needs to land before this issue can close.
