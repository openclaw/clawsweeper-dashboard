## Summary

Keep open: current main and v2026.5.12 still write the active backup tarball as an adjacent random `.tmp` file and only remove it during normal unwinding; the linked closing PR is still open and unmerged.

## What This Changes

The issue requests that `openclaw backup create` clean up temporary archive files when interrupted by timeout or signal.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/80598 or an equivalent narrow backup fix that moves active temp archives out of the output directory, registers scoped graceful-signal cleanup, preserves exit semantics, and documents hard-kill limits.

## Reproduction Assessment

Yes, source-backed: run `openclaw backup create` long enough to enter the `tar.c` write to the adjacent temp archive, then terminate the process before normal unwinding. I did not run the live timeout in this read-only review, so this is not marked as directly reproduced.

## Solution Assessment

Yes for the direction: a backup-owned temp archive lifecycle and graceful-signal cleanup is the narrow maintainable fix. The currently linked PR still needs review/repair before it can close this issue.
