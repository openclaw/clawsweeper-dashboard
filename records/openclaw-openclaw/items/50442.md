## Summary

Keep open. Current main still writes `openclaw backup create` output through an adjacent random `.tmp` archive and only removes it during normal async unwinding, while the CLI path has no backup-scoped SIGINT/SIGTERM, timeout, or abort cleanup. The issue discussion describes real disk exhaustion and a manual cleanup workaround, but not a shipped OpenClaw fix.

## What This Changes

The issue requests that `openclaw backup create` clean up large temporary archive files left behind when backup creation is interrupted by a timeout or exception.

## Best Possible Solution

Backup creation should own the active temporary archive lifecycle: track the exact temp archive path during the write window, clean it on graceful interruption or abort, preserve no-overwrite final publication, avoid deleting unrelated `*.tmp` files, and document unavoidable hard-kill limits.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the reported timeout scenario plus current-main source inspection: the backup writer creates an adjacent `.tmp` archive and relies on normal `finally` cleanup, while the CLI path has no backup-scoped signal or abort cleanup. I did not run a live timeout because this review is required to keep the checkout read-only and artifact-free.

## Solution Assessment

Yes. A narrow backup-owned cleanup path for graceful termination is the best maintainable fix; broad startup sweeping or deleting arbitrary `*.tmp` files would risk removing unrelated user files and would not address the active archive lifecycle.

## Risks / Open Questions

- A graceful in-process fix can cover SIGINT, SIGTERM, and explicit abort paths, but SIGKILL, host crashes, and power loss cannot reliably run JavaScript cleanup.
- Signal cleanup must preserve normal CLI exit codes and remove scoped handlers after backup completion so unrelated commands do not inherit backup-specific behavior.
- The fix should stay focused on the active temp archive lifecycle and avoid broad deletion of arbitrary `*.tmp` files; related large-backup behavior in #42273 and archive-time skip work in #67472 may need awareness but should not be conflated.
