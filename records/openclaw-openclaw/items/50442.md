## Summary

Keep open. Current main still writes `openclaw backup create` through an adjacent random `.tmp` archive and only removes it during normal async cleanup; the CLI path has no backup-scoped SIGINT/SIGTERM or abort cleanup for externally timed-out runs.

## What This Changes

The issue requests that `openclaw backup create` clean up large temporary backup archives left behind when the process times out or is interrupted.

## Best Possible Solution

Backup creation should own the active temporary archive lifecycle: track the exact temp path, clean it on graceful termination or abort during the write window, preserve no-overwrite final publication, avoid deleting unrelated `*.tmp` files, and document hard-kill limits.

## Reproduction Assessment

Yes. The high-confidence path is the reported timeout-killed `openclaw backup create` run plus current-main source inspection: the writer creates an adjacent `.tmp` archive and relies on normal `finally` cleanup while backup has no scoped signal or abort handler. I did not run a live timeout because this review had to keep the checkout read-only and artifact-free.

## Solution Assessment

Yes. A narrow backup-owned graceful-interruption cleanup path is the best maintainable fix; broad startup sweeping or deleting arbitrary `*.tmp` files would risk removing unrelated user files and would not address the active archive lifecycle.

## Risks / Open Questions

- A graceful in-process fix can cover SIGINT, SIGTERM, explicit aborts, and similar cooperative timeout paths, but SIGKILL, host crashes, and power loss cannot reliably run JavaScript cleanup.
- Signal cleanup must preserve normal CLI exit codes and remove scoped handlers after backup completion so unrelated commands do not inherit backup-specific behavior.
- Related backup issues such as #42273 and PR #67472 touch the same archive writer, so a fix should avoid conflicting with large-backup and ENOENT-race work.
