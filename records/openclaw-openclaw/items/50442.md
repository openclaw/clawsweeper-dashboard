## Summary

Keep open. Current main still writes `openclaw backup create` through a sibling random `.tmp` archive and only cleans it during normal JavaScript unwinding, so timeout or signal termination can still leave the large temp file described in the report.

## What This Changes

The issue asks for `openclaw backup create` to clean up large temporary backup archives left behind after timeout or interruption.

## Best Possible Solution

Backup creation should own the active temp archive lifecycle with scoped interruption cleanup while preserving no-overwrite final publication and documenting unavoidable hard-kill limits.

## Reproduction Assessment

Yes. The reported timeout scenario is backed by current-main source inspection: backup creation writes a sibling `.tmp` archive and has no backup-scoped signal or abort cleanup; I did not run a live timeout because this review is read-only and artifact-free.

## Solution Assessment

Yes. A narrow backup-owned cleanup path for graceful termination is the best maintainable fix; startup sweeping or deleting arbitrary `*.tmp` files would be broader and riskier.

## Repair Work Prompt

Fix the `openclaw backup create` temp archive leak reported in https://github.com/openclaw/openclaw/issues/50442. Current main writes the active archive to `<output>.<uuid>.tmp` in `src/infra/backup-create.ts` and only removes it during normal `finally` cleanup; add scoped cleanup for graceful interruption or explicit abort during the archive-write window. Keep the fix narrow: track and remove only the exact temp archive created by this backup run, preserve existing no-overwrite publication behavior, keep the normal tar failure cleanup tests passing, detach any signal handlers after completion, preserve CLI exit codes for SIGINT/SIGTERM, and do not add broad startup/load-time `.tmp` sweeping. Add focused regression coverage for graceful SIGINT/SIGTERM or AbortSignal-style cancellation around `openclaw backup create`; mention the unavoidable SIGKILL/host-crash limit in docs if user-visible. Related context: #42273 also reports large backup stalls and dangling `.tmp` files, but do not expand this PR into progress indicators or exclude-log feature work unless required for the cleanup seam.
