## Summary

Keep open: current main still does not resolve a bare Feishu username-like target to a user `open_id`, and the only directly related implementation attempt is closed unmerged. This is useful Feishu plugin work, but it is a new target-resolution capability relative to the current ID-focused Feishu docs and prompt hints.

## What This Changes

Resolve Feishu username-like message targets such as `jaydenli` to a unique user `open_id`, or produce a clearer supported-format error.

## Best Possible Solution

Add a Feishu-owned, ambiguity-safe target resolver or peer-directory enhancement that maps one unique username-like input to a user `open_id`, with docs and tests for supported lookup behavior.

## Reproduction Assessment

Yes, at source level: `jaydenli` is not ID-like, is classified as a group target, misses the current Feishu directory lookup, has no Feishu fallback resolver, and reaches `unknownTargetError`. I did not run a live Feishu delivery because no live channel credentials were used in this read-only review.

## Solution Assessment

Yes, the best maintainable direction is Feishu-owned target resolution, not a generic change to shared bare-token heuristics. The implementation should require a unique match and preserve ambiguity errors to avoid unsafe delivery.
