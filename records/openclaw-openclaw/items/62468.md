## Summary

Keep open. Current main still cannot resolve a bare Feishu username like `jaydenli` to a user `open_id`: the Feishu plugin only advertises ID-like target recognition, the generic resolver defaults bare tokens to group lookup, and the peer directory lookup still lacks the username/display-name matching described by the closed unmerged follow-up PR.

## What This Changes

The issue requests Feishu outbound/message-tool targets to accept a username-like value such as `jaydenli`, or provide clearer lookup/error guidance, instead of failing with `Unknown target`.

## Best Possible Solution

Implement this in the Feishu plugin boundary: add Feishu-owned username-like target resolution to a unique user `open_id`, enhance peer matching to handle compact names and stable user fields, return clear ambiguous/missing-target errors, and update focused tests plus Feishu docs/changelog. Core outbound resolution should stay generic unless a reusable seam is strictly needed.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: `target: "jaydenli"` becomes `to`, `resolveChannelTarget` treats it as a non-ID bare group query, Feishu has no plugin fallback resolver, and the path returns `Unknown target`. No live Feishu credentials were found in env/profile for an API run, but the failing pre-send resolver path is statically clear.

## Solution Assessment

Yes. A Feishu-owned resolver is the narrowest maintainable fix because Feishu knows its ID prefixes, directory fields, and permission behavior; changing generic core to treat all bare tokens as users would risk regressions for channels where bare names are groups.
