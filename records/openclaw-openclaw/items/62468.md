## Summary

Keep open: current main and v2026.5.18 still route a bare Feishu username through the generic group lookup path, and Feishu has no fallback resolver for username/display-name misses. A direct fix attempt exists, but it was closed unmerged.

## What This Changes

Feishu message delivery to a bare username like `jaydenli` should resolve to the matching user/open_id instead of failing with `Unknown target`.

## Best Possible Solution

Add a Feishu-owned, ambiguity-safe target resolver or lookup path that maps one unique username/display-name/email/nickname match to `open_id`, preserves explicit ID behavior, and updates the hint/docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `jaydenli` is not ID-like, defaults to group lookup, and reaches `Unknown target` after the directory miss because Feishu has no fallback resolver; I did not run live Feishu delivery in this read-only review.

## Solution Assessment

Yes. The maintainable fix belongs in the Feishu plugin's resolver/directory behavior rather than changing generic core bare-target defaults; a peer-list-only patch is insufficient unless the bare-token miss path can reach it safely.
