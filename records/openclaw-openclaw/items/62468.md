## What This Changes

The issue requests Feishu message delivery to resolve bare username/display-name targets such as `jaydenli` to a valid `open_id`, or provide a clearer lookup/error path.

## Best Possible Solution

Add a Feishu-owned, ambiguity-safe target resolver or lookup path that maps one unique username/display-name match to `open_id`, keeps explicit ID targets unchanged, and documents the supported lookup behavior.

## Solution Assessment

Yes, the maintainable direction is a Feishu-owned resolver or directory lookup enhancement, not a generic core change to bare-token behavior. It should preserve ambiguity errors and explicit `user:open_id`/`chat:chat_id` behavior.
