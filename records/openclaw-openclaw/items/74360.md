## Summary

Keep open. The PR addresses a real current-main SSRF policy bug, but it is not merge-ready because the new empty trusted policy is still bypassed when `HTTP_PROXY`/`HTTPS_PROXY` puts `withTrustedWebToolsEndpoint` in trusted env-proxy mode, and the new public Plugin SDK exports still lack the generated API baseline update.

## What This Changes

The PR splits web-tool SSRF policies into strict trusted and permissive self-hosted helpers, switches SearXNG to the self-hosted helper, exports those helpers through Plugin SDK web entrypoints, and adds SSRF policy tests.

## Best Possible Solution

Land a maintainer-approved SSRF fix that keeps fixed public web-search providers strict even when proxy env vars are configured, preserves a clearly named validated self-hosted path for SearXNG-style endpoints, updates the generated SDK API baseline for any public helper exports, and lets #74357 close only after the PR merges.

## Reproduction Assessment

Yes. Current-main source proves all shared web-search providers inherit a private-network-allowing trusted policy, and the PR's remaining gap can be reproduced with a focused test that sets `HTTP_PROXY`/`HTTPS_PROXY`, redirects from a public URL to `http://127.0.0.1`, and calls `withTrustedWebToolsEndpoint`. I did not run tests because this sweep is read-only.

## Solution Assessment

No as submitted. The policy split and SearXNG carveout are the right direction, but the trusted endpoint still enters proxy mode where the target policy is not enforced locally, and the public SDK baseline is missing.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Enforce strict checks before env proxying:** `src/agents/tools/web-guarded-fetch.ts:10`
  - body: Setting the trusted policy to `{}` only affects direct and NO_PROXY paths. `withTrustedWebToolsEndpoint` still forces `TRUSTED_ENV_PROXY`; when `HTTP_PROXY` or `HTTPS_PROXY` applies, `fetchWithSsrFGuard` creates an `EnvHttpProxyAgent` without calling `resolvePinnedHostnameWithPolicy`, so a cloud-provider redirect to a private literal such as `http://127.0.0.1` is handed to the proxy instead of being blocked. Add a pre-proxy host/IP policy check or keep these fixed public endpoints on the strict guarded path.
  - confidence: 0.83

  - confidence: 0.88
