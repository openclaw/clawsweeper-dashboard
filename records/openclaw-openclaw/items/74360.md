## Summary

Keep open. The underlying SSRF report is real on current main, and the PR is directionally useful, but the branch still needs security review for the SearXNG public-HTTPS path plus a Plugin SDK API baseline update before it is mergeable.

## What This Changes

The PR splits shared web-tool SSRF handling into strict trusted endpoints and explicit self-hosted endpoints, switches SearXNG to the self-hosted helper, exports those helpers through the Plugin SDK, and adds SSRF policy tests.

## Best Possible Solution

Land a maintainer-approved SSRF fix that keeps fixed public web-search providers strict by default, grants private-network access only after SearXNG-style endpoints are proven private or explicitly operator-trusted, updates Plugin SDK baseline/docs for any new public helpers, adds the required changelog entry, and lets #74357 close after merge.

## Reproduction Assessment

Yes. Current-main source proves the broad private-network policy, and the PR-specific concern follows from a documented public https:// SearXNG base URL that redirects to loopback/RFC1918 while routed through the private-network-allowing helper; I did not run tests in this read-only sweep.

## Solution Assessment

No as submitted. Splitting strict and self-hosted helpers is the right direction, but the PR should not route documented public HTTPS SearXNG endpoints through the permissive helper without stronger private/operator-trust validation, and the public SDK export needs its baseline update.
