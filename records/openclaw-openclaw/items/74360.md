## Summary

Keep this PR open. The main SSRF direction is useful and current PR-head checks are green, but the patch is still not merge-ready because the SearXNG carveout applies the private-network-allowing helper to documented public HTTPS SearXNG endpoints, and the user-facing security fix is missing the required changelog entry.

## What This Changes

The PR makes shared trusted web-tool fetches strict by default, adds explicit self-hosted web fetch/search helpers, switches SearXNG to the self-hosted helper, exports the helpers through the Plugin SDK, and adds SSRF policy tests.

## Best Possible Solution

Land a maintainer-approved SSRF fix that keeps fixed public web-search providers strict by default, uses the private-network self-hosted helper only after proving the configured endpoint is operator-trusted/private, preserves SearXNG LAN support, adds focused coverage for public-HTTPS SearXNG redirects, and records the security fix in the changelog.

## Reproduction Assessment

Yes. Current-main source proof shows the broad private-network policy, and the PR-specific concern can be reproduced conceptually with a documented public HTTPS SearXNG base URL that redirects to loopback/RFC1918 while the PR routes it through the private-network-allowing helper; I did not run tests in this read-only sweep.

## Solution Assessment

No as submitted. Splitting strict and self-hosted helpers is the right maintainable direction, but SearXNG needs strict handling for public HTTPS endpoints or a stronger private-target validation before using the self-hosted policy.
