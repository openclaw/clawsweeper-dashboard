## Summary

Keep open: current main still has a credible Telegram multi-account startup gap because the gateway cap limits account task handoff, but Telegram `getMe` probes are launched inside background `startAccount` promises and can still fan out across configured accounts. The broader host-level circuit-breaker work is related to https://github.com/openclaw/openclaw/issues/77900, but this issue has distinct startup-probe evidence.

## What This Changes

The issue reports that 16 Telegram bot accounts start their `getMe` health checks together, causing delayed fetch timeouts and asks for bounded or staggered startup plus breaker behavior.

## Best Possible Solution

Add a narrow Telegram startup-probe fanout guard, or adjust the channel lifecycle seam so the pre-monitor `getMe` phase is actually covered by the existing startup cap, while leaving the reusable host circuit breaker to the related transport issue.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection does show a plausible current-main path: Telegram `startAccount` awaits `getMe`, while the gateway account cap hands that work to background promises without waiting for the preflight to finish.

## Solution Assessment

No, not as the broad worker-thread/configurable-breaker proposal. The narrow maintainable fix is to bound or stagger Telegram startup probes first, then handle host-level breaker behavior in https://github.com/openclaw/openclaw/issues/77900 if needed.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78353 as a narrow Telegram startup reliability bug. Current main has a generic channel startup cap, but `src/gateway/server-channels.ts` hands `startAccount` to a background promise before Telegram's `getMe` preflight finishes, so many Telegram accounts can still run `probeTelegram` concurrently. Add a bounded startup-probe guard for Telegram accounts, or adjust the channel lifecycle seam so the pre-monitor `getMe` phase is actually covered by the existing cap. Prefer fixed safe defaults matching the existing cap; do not add a new user-facing config option unless the existing contracts make that unavoidable. Do not broaden this into the host-level transport circuit breaker from https://github.com/openclaw/openclaw/issues/77900. Add regression coverage proving six or more Telegram accounts do not have more than four concurrent startup probes while long-running monitors still hand off and stop cleanly. Add a user-facing CHANGELOG.md fix entry if code changes land.
