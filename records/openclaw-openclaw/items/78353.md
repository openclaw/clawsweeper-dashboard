## Summary

Keep open: current main still has a credible Telegram multi-account startup gap because the generic gateway cap bounds only the account startup handoff, while Telegram `getMe` probes run inside background `startAccount` tasks that can accumulate across configured accounts. The broader host circuit-breaker issue at https://github.com/openclaw/openclaw/issues/77900 is related, but this report has distinct startup-probe evidence.

## What This Changes

The issue asks OpenClaw to prevent many Telegram bot accounts from launching startup `getMe` probes without backpressure or circuit breaking when they start together.

## Best Possible Solution

Add a narrow Telegram startup-probe fanout guard, or introduce an explicit pre-monitor readiness seam that the existing startup cap can cover, while leaving reusable host-level circuit breaker work to the related transport issue.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection does show a current-main path where Telegram `startAccount` awaits `getMe`, while the gateway account cap can finish after handing that work to background promises.

## Solution Assessment

No, not as the broad worker-thread/configurable-breaker proposal. The narrow maintainable fix is to bound or stagger Telegram startup probes first, then handle host-level breaker behavior in https://github.com/openclaw/openclaw/issues/77900 if needed.

Current main caps account startup handoff in `src/gateway/server-channels.ts`, but Telegram `startAccount` launches the `getMe` probe inside the background account task, so many configured Telegram accounts can still run startup probes together.
