## Summary

Keep open: current `main` still has a credible Telegram multi-account startup-probe fanout gap. The shared gateway cap limits account handoff work, but Telegram `getMe` probes run inside background `startAccount` tasks and can still accumulate across configured accounts.

## What This Changes

The issue reports 16 Telegram bot accounts timing out together during startup `getMe` probes, with nominal 10s fetch timeouts stretching to 41-82s and cascading event-loop starvation.

## Best Possible Solution

Bound or stagger Telegram startup `getMe` probes, or adjust the channel lifecycle seam so the pre-monitor probe phase is actually covered by the existing startup cap, while leaving host-level breaker design separate.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a high-confidence current-main path: Telegram `startAccount` awaits `getMe`, while the shared gateway cap hands that work to background account tasks without awaiting the preflight.

## Solution Assessment

No to the broad worker-thread/configurable circuit-breaker proposal as the first fix. The narrower maintainable repair is to bound or stagger Telegram startup probes, with regression coverage that proves `getMe` fanout is capped.

## Repair Work Prompt

Fix the Telegram multi-account startup probe fanout reported at https://github.com/openclaw/openclaw/issues/78353. Current main caps gateway account handoff at four, but `src/gateway/server-channels.ts` starts plugin `startAccount` as a background task; Telegram then awaits `probeTelegram(.../getMe...)` inside `extensions/telegram/src/channel.ts`, so many configured Telegram accounts can still have startup `getMe` probes in flight. Implement a narrow fix that bounds or staggers the pre-monitor Telegram startup probe phase, or adjusts the channel lifecycle seam so the pre-monitor preflight is covered by the existing startup cap. Do not add operator config, worker threads, or a broad host circuit-breaker unless maintainers explicitly choose that scope. Add regression coverage proving `startAccount`/Telegram startup probe concurrency is capped, preserve plugin boundaries, and keep the older generic channel startup cap behavior intact. Coordinate with https://github.com/openclaw/openclaw/issues/78352 and do not treat the closed config-only request at https://github.com/openclaw/openclaw/issues/78173 as sufficient proof.
