## Summary

Keep open: the PR targets a real Feishu streaming-card content-loss path, but the new broad containment heuristic can preserve stale accumulated text when a shorter final payload should replace the preview, and the external real Feishu proof artifact is still missing.

## What This Changes

The PR changes Feishu streaming-card final delivery to keep accumulated streamed text when the final payload is already contained in it, with a regression test for a last-block final.

## Best Possible Solution

Land a Feishu-local final-retention fix that preserves the accumulated text only for a safe last-block/suffix case, then require redacted Feishu/Lark CardKit proof or a maintainer proof override before merge.

## Reproduction Assessment

Yes, at source level. Current main appends block deliveries into `streamText` but then replaces it with the final payload, so a final carrying only the last block collapses the card; I did not run a live Feishu tenant session.

## Solution Assessment

No as submitted. The Feishu dispatcher is the right owner boundary, but the broad `includes` condition should be narrowed so final payloads that intentionally replace or shorten a preview still work.
