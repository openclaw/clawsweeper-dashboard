## Summary

Keep open. Current main still has a split Feishu path: the shared `message` tool `send`/`thread-reply` action builds a markdown-only card from `presentation`, while the adjacent outbound payload renderer already creates native Feishu button actions with structured callback values.

## What This Changes

The issue asks for Feishu `message` tool sends with `presentation` buttons to render as native interactive cards instead of markdown/plain-text fallback.

## Best Possible Solution

Make the Feishu plugin-owned `send` and `thread-reply` action path share the existing native card/button renderer and `ocf1` callback envelope while keeping conservative fallbacks.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu `message` tool `send` with `presentation.blocks[].type === "buttons"`; current main routes it to the Feishu action handler and the test expects markdown fallback text.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin action path, preferably by sharing the existing outbound card builder instead of adding a core special case.
