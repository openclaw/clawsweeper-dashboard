## Summary

Keep open. The report is recent and describes a documented Linux headless node exec workflow failing on the latest release; current `main` and `v2026.5.20` still build Linux node exec as `/bin/sh -lc ...` and the node host surfaces `spawn` errors directly, so there is no safe close reason yet.

## What This Changes

The issue reports `exec host=node` failing on connected Linux headless nodes with `spawn /bin/sh ENOENT` even though `/bin/sh` exists.

## Best Possible Solution

Reproduce on a Linux headless node host under systemd, then make the node-owned execution path choose a stable cwd or emit precise shell/cwd diagnostics with focused regression coverage and no weakening of exec approvals.

## Reproduction Assessment

No high-confidence live reproduction yet: the source path traces the reported error shape from `/bin/sh -lc` generation to node-host `spawn(argv[0], ...)`, but I did not run the Linux headless systemd-node setup. A local Node probe also shows the same error can occur when cwd is missing, so the root cause still needs live confirmation.

## Solution Assessment

Unclear until live repro: the likely fix belongs in node-host shell/cwd resolution or diagnostics, but changing shell selection or fallback behavior without proving the cause would risk masking a different service-environment problem.
