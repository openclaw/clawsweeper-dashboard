## Summary

Keep open. The report is recent, external, and describes a documented `exec host=node` workflow failing on the latest release; current `main` still builds Linux node exec as `/bin/sh -lc ...` and the node host directly spawns that argv, so the observed failure is source-plausible and not fixed on main.

## What This Changes

The issue reports that Linux headless node exec fails with `spawn /bin/sh ENOENT` on connected OpenClaw 2026.5.20 nodes even though `/bin/sh` exists.

## Best Possible Solution

Reproduce on a Linux headless node host, then make shell selection or diagnostics robust in the node-owned execution path with focused regression coverage and no weakening of exec approvals.

## Reproduction Assessment

No, not as a live reproduction: I did not run a Linux headless node systemd setup. The source path is clear enough to trace the reported error shape from `/bin/sh -lc` generation to node-host `spawn(argv[0], ...)`.

## Solution Assessment

Unclear until live repro: the likely fix belongs in node-host shell resolution or diagnostics, but the ENOENT despite `/bin/sh` existing needs confirmation before choosing a fallback strategy.
