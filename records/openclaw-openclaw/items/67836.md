## Summary

Keep open: current main still lacks this Memory v2 feature, and the PR is a broad bundled-memory implementation with unresolved correctness, privacy, and proof blockers rather than a cleanup-close candidate.

## What This Changes

The PR adds a default-off SQLite-backed Memory v2 foundation to `memory-core`, including sidecar schema/repo code, ingest and rerank pipelines, dreaming shadow-touch wiring, sidecar CLI commands, manifest config, docs, and tests.

## Best Possible Solution

Split or harden the Memory v2 work into reviewable slices, settle the persistence/privacy contract, fix the CLI and rerank semantics, and merge only with real opt-in behavior proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the latest PR head shows the sidecar write fan-out, read-only schema writes, zero-exit rejected inputs, pre-slice shadow touches, and nondeterministic duplicate-location selection. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. A default-off Memory v2 foundation is a plausible direction, but this PR is still too broad and has correctness plus privacy-review blockers; narrower slices or focused fixes are the safer path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope sidecar writes to one default agent:** `extensions/memory-core/src/cli.runtime.ts:2255`
  - body: The mutating sidecar commands resolve omitted `--agent` through `resolveAgentIds`, which expands to every configured agent while the new help says the option defaults to the default agent. A plain pin/status/salience/supersede command can therefore mutate multiple sidecar DBs unintentionally.
  - confidence: 0.9
- **[P2] Keep sidecar stats and list read-only:** `extensions/memory-core/src/cli.runtime.ts:2000`
  - body: `stats` and `list` open existing databases through `openSidecarDatabase`, which runs schema initialization and metadata writes. Commands documented as read-only should inspect without migrations or writes, otherwise they can mutate state and fail on read-only storage.
  - confidence: 0.88
- **[P2] Fail rejected sidecar write inputs:** `extensions/memory-core/src/cli.runtime.ts:2229-2231`
  - body: When a ref prefix is missing or ambiguous, the renderer logs the rejection and continues without setting a non-zero exit code. Scripts cannot distinguish a skipped mutation from a successful write, and the invalid argument paths follow the same success-exit pattern.
  - confidence: 0.87
- **[P2] Touch only surfaced search hits:** `extensions/memory-core/src/memory-v2/rerank/wrapper.ts:90-100`
  - body: `shadowOnRecall` records touches for every candidate passed into the wrapper, but `memory_search` still limits final output after rerank. Rows dropped from the returned result set can get `last_accessed_at` refreshed, skewing later recency scoring toward content the user never saw.
  - confidence: 0.84
- **[P2] Add a stable duplicate-location tie-breaker:** `extensions/memory-core/src/memory-v2/rerank/lookup.ts:31-35`
  - body: The lookup query has no `ORDER BY`, and equal salience plus equal `last_accessed_at` falls back to SQLite row order. Duplicate `location_id` rows can therefore select different pinned/status signals between runs.
  - confidence: 0.82
