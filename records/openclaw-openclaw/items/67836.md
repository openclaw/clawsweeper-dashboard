## Summary

Keep open. Current main does not contain Memory v2, so the PR is not obsolete, but the branch still needs maintainer review and fixes: it is stale against current `memory_search` session-visibility/runtime-config behavior, and the new sidecar CLI/rerank paths have blocking correctness and privacy-review concerns.

## What This Changes

The PR adds a default-off SQLite-backed Memory v2 foundation to `memory-core`, including sidecar schema/store code, ingest and rerank pipelines, dreaming shadow-touch wiring, sidecar CLI commands, manifest config, docs, and tests.

## Best Possible Solution

Keep the Memory v2 foundation under review, rebase it onto current main, preserve all existing memory-search visibility/runtime-config behavior, and fix the sidecar CLI/rerank issues before considering merge. Given the size and privacy surface, maintainers should strongly consider splitting schema/store, ingest, rerank, dreaming touch, and CLI writes into narrower reviewed slices with explicit retention/redaction docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review findings by source inspection: the PR head omits current-main memory-search session hooks, uses all-agent resolution for sidecar commands, initializes schema in read-only paths, returns success on rejected writes, touches pre-slice hits, and has nondeterministic duplicate-location selection. I did not run runtime tests because the task is read-only.

## Solution Assessment

Is this the best way to solve the issue? No, not yet. A default-off foundation is a plausible direction, but this broad PR should first be rebased, split or tightly reviewed, and corrected so it preserves current memory-search safety behavior and has clear sidecar privacy semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve current memory_search safety hooks:** `extensions/memory-core/index.ts:48-52`
  - body: The PR-head wiring is based on an older `createMemorySearchTool` shape and passes only `ctx.config`/`sessionKey` plus rerank. Current main passes execution-time config and `sandboxed`, and `tools.ts` enforces `corpus=sessions` plus session visibility. Rebase this hunk so adding rerank does not drop those current-main protections.
  - confidence: 0.91
- **[P1] Scope sidecar writes to one default agent:** `extensions/memory-core/src/cli.runtime.ts:2255`
  - body: The new sidecar writers reuse `resolveAgentIds`, which expands an omitted `--agent` to every configured agent. The sidecar docs and option text say these commands default to the default agent, so a plain `pin`, `status`, `salience`, or `supersede` can mutate multiple sidecars unintentionally.
  - confidence: 0.9
- **[P2] Keep stats and list read-only:** `extensions/memory-core/src/cli.runtime.ts:2000`
  - body: The `stats`/`list` inspection paths open existing DBs with `openSidecarDatabase`, which always runs schema initialization and writes metadata/DDL. Commands documented as read-only should use a no-migration read opener so they work in read-only contexts and do not mutate the sidecar.
  - confidence: 0.88
- **[P2] Fail rejected ref resolutions:** `extensions/memory-core/src/cli.runtime.ts:2229-2231`
  - body: When a ref prefix is missing or ambiguous, the renderer only logs the rejection and the command exits successfully. Scripts cannot distinguish a skipped mutation from a successful one, so rejected write targets should set a non-zero exit code.
  - confidence: 0.89
- **[P2] Fail invalid sidecar arguments:** `extensions/memory-core/src/cli.runtime.ts:2283-2289`
  - body: Invalid sidecar arguments such as an unknown status or salience value only print a warning and return without setting a failing exit code. Automation will treat the command as successful even though no mutation happened.
  - confidence: 0.86
- **[P2] Touch only surfaced search hits:** `extensions/memory-core/src/memory-v2/rerank/wrapper.ts:90-100`
  - body: `shadowOnRecall` records touches for every candidate passed into the wrapper, but `memory_search` sorts and slices after this call. Rows that are dropped by `maxResults` can still get `last_accessed_at` refreshed, which skews later recency scoring toward content the user never saw.
  - confidence: 0.84
- **[P2] Add a deterministic duplicate-location tie-breaker:** `extensions/memory-core/src/memory-v2/rerank/lookup.ts:31-35`
  - body: Duplicate `location_id` rows with equal salience and `last_accessed_at` are selected according to SQLite's row order because the query has no `ORDER BY` and `preferCandidate` has no final tie-break. Add stable SQL ordering or compare a stable field so rerank output cannot drift between runs.
  - confidence: 0.82
