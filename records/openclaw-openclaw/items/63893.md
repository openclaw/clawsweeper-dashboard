Keep open: current main and the latest release still have the TaskFlow SQLite migration gap, and this PR is a focused, proof-sufficient fix. The remaining maintainer action is choosing the canonical implementation because an overlapping open PR targets the same schema repair.

## What This Changes

The PR adds a TaskFlow SQLite `flow_runs` rebuild migration, legacy/hybrid schema regression tests, and a changelog entry for upgraded installs that still have `owner_session_key`.

## Best Possible Solution

Land one canonical TaskFlow SQLite migration that preserves rows, normalizes `owner_key`, removes the legacy `owner_session_key` constraint, recreates indexes, and keeps regression coverage.

## Solution Assessment

Yes. Rebuilding the TaskFlow SQLite table in the migration layer is the narrow maintainable repair; maintainers should consolidate this PR with https://github.com/openclaw/openclaw/pull/65767 rather than changing the webhook owner contract.
