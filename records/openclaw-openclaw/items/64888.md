## Summary

Keep open. Current main still lacks a `cancelled` embedded-run liveness state, still maps abort-without-timeout to `blocked`, and still has attempt-dispatch/error paths that can produce terminal lifecycle events without replay/liveness metadata.

## What This Changes

Request to add an embedded runner error boundary for terminal lifecycle metadata and distinguish user cancellation with a `cancelled` liveness state.

## Best Possible Solution

Extend the embedded liveness contract with `cancelled`, return it only for user aborts that are not timeouts, and add a runner-level error backstop that emits explicit replay/liveness metadata before rethrowing uncaught attempt failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `resolveRunLivenessState({ payloadCount: 0, aborted: true, timedOut: false, ... })` reaches the current `blocked` branch, and a rejected `runEmbeddedAttemptWithBackend` crosses an unguarded await before fallback lifecycle metadata can be set.

## Solution Assessment

Yes. The narrow maintainable fix is to update the central liveness type/resolver and lifecycle backstop behavior with focused regression coverage, without redesigning the harness SPI or adding config.

## Repair Work Prompt

Fix issue #64888 by updating the embedded runner lifecycle contract. Add `cancelled` to `EmbeddedRunLivenessState`, return `cancelled` from `resolveRunLivenessState` only when the run was user-aborted without timing out and produced no payloads, and update downstream lifecycle consumers/tests that currently assume only `working | paused | blocked | abandoned`. Add a runner-level error boundary around attempt dispatch so uncaught `runEmbeddedAttemptWithBackend` failures emit fallback terminal lifecycle metadata with `replayInvalid: true` and a blocked/error liveness state before rethrowing. Keep the change within the embedded runner/lifecycle surfaces; do not redesign `AgentHarness`, add config, or change unrelated fallback policy. Add focused regression coverage for user abort and forced attempt throw, and add a user-facing `CHANGELOG.md` fix entry if the implementation changes shipped behavior.
