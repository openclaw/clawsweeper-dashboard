## Summary

Keep open. Current main and the latest release still have the reported lifecycle gaps: user aborts without timeout collapse into `blocked`, and a rejected embedded attempt exits the runner without a fallback terminal liveness/replay metadata path.

## What This Changes

The issue requests an uncaught-exception terminal lifecycle boundary around embedded attempts and a distinct `cancelled` liveness state for user-initiated aborts.

## Best Possible Solution

Add `cancelled` at the shared embedded liveness contract/resolver boundary, propagate it through lifecycle consumers, and add a runner-level terminal metadata/error backstop for uncaught attempt failures with focused regression coverage.

## Reproduction Assessment

Yes, by source inspection on current main. An aborted, non-timeout, zero-payload run reaches the `blocked` branch, and a rejected `runEmbeddedAttemptWithBackend` exits before a normalized attempt can provide terminal lifecycle metadata.

## Solution Assessment

Yes. The narrow maintainable fix is to update the central liveness contract/resolver and runner error backstop, then cover abort and thrown-attempt paths in the existing embedded-runner and lifecycle tests.
