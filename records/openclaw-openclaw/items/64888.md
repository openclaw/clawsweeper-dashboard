## Summary

Keep open. Current main still lacks the requested `cancelled` embedded-run liveness state, still maps abort-without-timeout to `blocked`, and still has an unguarded attempt dispatch path where a backend rejection can bypass terminal lifecycle metadata before cleanup.

## What This Changes

Request to add an embedded-run terminal error boundary and distinguish user cancellation from blocked liveness with a new `cancelled` state.

## Best Possible Solution

Extend the central embedded liveness contract with `cancelled`, return it only for user aborts that are not timeouts, and add a runner-level error backstop that emits explicit terminal replay/liveness metadata before rethrowing uncaught attempt failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `resolveRunLivenessState({ payloadCount: 0, aborted: true, timedOut: false, ... })` reaches the current `blocked` branch, and a rejected `runEmbeddedAttemptWithBackend` follows an unguarded await before any local fallback terminal meta is set.

## Solution Assessment

Yes. The narrow maintainable fix is to update the typed liveness resolver and downstream consumers, then add focused lifecycle regression coverage and an attempt-dispatch error boundary rather than redesigning the runner or harness SPI.

## Repair Work Prompt

Fix the embedded-runner lifecycle gaps from https://github.com/openclaw/openclaw/issues/64888. Add `cancelled` to `EmbeddedRunLivenessState`, update `resolveRunLivenessState` so `aborted: true` with `timedOut: false` and no payloads returns `cancelled`, and update downstream lifecycle consumers/tests that currently assume only `working | paused | blocked | abandoned`. Add a local error boundary around the attempt dispatch path so uncaught `runEmbeddedAttemptWithBackend` failures emit explicit terminal lifecycle metadata, at minimum `replayInvalid: true` and `livenessState: "blocked"`, before rethrowing. Keep the fix inside the embedded runner/auto-reply lifecycle boundary; do not add config, redesign the harness SPI, or change unrelated fallback policy. Add/update regression tests for a simulated user abort and a forced attempt-dispatch throw. Add a single-line `CHANGELOG.md` fix entry unless maintainers classify the change as telemetry-only.
