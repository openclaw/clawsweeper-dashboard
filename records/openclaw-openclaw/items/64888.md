## Summary

Keep open. Current main still lacks a `cancelled` embedded-run liveness state, still maps abort-without-timeout to `blocked`, and still has an unguarded embedded-attempt dispatch path where a rejected backend attempt can bypass terminal replay/liveness metadata.

## What This Changes

Add an embedded-runner error boundary for uncaught attempt failures and distinguish user-cancelled embedded runs from blocked or timed-out liveness states.

## Best Possible Solution

Extend the embedded liveness contract with `cancelled`, return it for user aborts that are not timeouts, and add a runner-level rejection backstop that emits explicit replay/liveness metadata before rethrowing uncaught attempt failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `resolveRunLivenessState({ payloadCount: 0, aborted: true, timedOut: false, ... })` reaches the `blocked` branch, and a rejected `runEmbeddedAttemptWithBackend` crosses an unguarded await before fallback lifecycle metadata is set.

## Solution Assessment

Yes. The narrow maintainable fix is to update the central liveness type/resolver and add a focused dispatch error backstop with regression coverage, without adding config or redesigning the harness SPI.
