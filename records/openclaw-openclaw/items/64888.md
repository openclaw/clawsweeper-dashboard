## Summary

Keep open. Current main still lacks the requested `cancelled` embedded-run liveness state, still maps abort-without-timeout to `blocked`, and still awaits attempt dispatch without a local terminal lifecycle error backstop before the cleanup-only `finally`.

## What This Changes

The issue requests explicit terminal lifecycle metadata on uncaught embedded-runner errors and a distinct `cancelled` liveness state for user aborts.

## Best Possible Solution

Extend the embedded liveness contract with `cancelled`, return it only for user aborts that are not timeouts, and add a runner-level error backstop that emits explicit terminal replay/liveness metadata before rethrowing uncaught attempt failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `resolveRunLivenessState({ payloadCount: 0, aborted: true, timedOut: false, ... })` reaches the current `blocked` branch, and a rejected `runEmbeddedAttemptWithBackend` follows an unguarded await before terminal lifecycle metadata can be set.

## Solution Assessment

Yes. The narrow maintainable fix is to update the central typed liveness resolver, add focused lifecycle regression coverage, and add an attempt-dispatch error boundary rather than redesigning the embedded runner or harness SPI.
