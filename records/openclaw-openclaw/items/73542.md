## Summary

Keep open. Current main still routes outbound pre-send `message_sending` through the typed plugin hook runner only, while hook-directory handlers are registered in the internal hook registry; the open PR #73570 references this issue with closing syntax, so this issue should remain open until that PR lands or maintainers replace it.

## What This Changes

The issue requests that user-defined hook-directory handlers registered for `message_sending` fire before outbound delivery, matching the plugin hook path.

## Best Possible Solution

Review and land or replace PR #73570 with a canonical pre-delivery bridge that preserves plugin modification/cancel ordering, supports the event spelling users can register from `HOOK.md`, and adds focused tests/docs for the internal pre-send event semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has a pre-send plugin-only call path, literal registration for hook-directory metadata events, and no internal pre-send bridge before delivery.

## Solution Assessment

Unclear. Bridging the pre-send path is the right direction, but the final implementation should explicitly handle `message_sending` versus `message:sending` compatibility and ordering before maintainers treat the issue as fixed.
