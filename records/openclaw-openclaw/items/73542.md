## Summary

Keep open. Current main still routes pre-send outbound delivery through the typed plugin `message_sending` hook runner only, while hook-directory handlers register through the internal hook registry. Open PR #73570 already references this issue with closing syntax, so maintainers should review, land, or replace that implementation rather than closing the issue now.

## What This Changes

The issue asks for user-defined `HOOK.md` handlers declaring `message_sending` to fire before outbound message delivery, matching the existing plugin pre-send hook behavior.

## Best Possible Solution

Land or replace PR #73570 with a canonical pre-send internal hook bridge in the shared outbound delivery path, before channel send, with explicit event-name compatibility, ordering semantics relative to plugin modification/cancel hooks, and focused tests/docs for the supported internal hook contract.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: hook-directory metadata registers literal event keys, internal dispatch only reaches `message` and `message:<action>` listeners, and current outbound pre-send delivery only invokes plugin `runMessageSending`.

## Solution Assessment

Yes, the narrow maintainable direction is to bridge pre-send internal dispatch in `deliverOutboundPayloads` or a helper next to `applyMessageSendingHook`. The implementation should not stop at a new undocumented notification event if existing `message_sending` hook-directory users are the compatibility target.
