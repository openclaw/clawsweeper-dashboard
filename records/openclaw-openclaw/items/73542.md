## Summary

Keep open. Current main still routes pre-send outbound delivery through the typed plugin `message_sending` runner only, while user hook-directory handlers are registered through the internal hook registry. Open PR #73570 already references this issue with closing syntax, so this issue should remain open until that PR is landed, adapted, or replaced.

## What This Changes

User-defined hook-directory handlers that register for `message_sending` should fire before outbound channel delivery, matching the pre-send lifecycle point used by typed plugin hooks.

## Best Possible Solution

Land or replace PR #73570 with a canonical pre-send internal hook bridge in the shared outbound delivery path before channel send, with explicit `message_sending` and `message:sending` compatibility, ordering relative to plugin rewrite/cancel behavior, focused regression tests, docs, and the required changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: hook-directory metadata registers literal event keys, internal dispatch reaches only `message` and `message:<action>` listeners, and current pre-send outbound delivery only invokes the typed plugin `runMessageSending` path.

## Solution Assessment

Unclear. Adding an internal bridge in `deliverOutboundPayloads` is the right narrow direction, but the final solution must define event-key compatibility and pre-delivery ordering after plugin cancellation or rewrite rather than adding only an undocumented fire-and-forget notification.
