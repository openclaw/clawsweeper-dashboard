## Summary

Keep open. Current main still invokes the outbound pre-send `message_sending` path only through the typed plugin hook runner, while hook-directory handlers are registered in the internal hook registry. Open PR #73570 already references this issue with closing syntax, so the issue should remain open until that PR is reviewed, landed, or replaced.

## What This Changes

User-defined hook-directory handlers registered for `message_sending` should fire before outbound channel delivery, matching the pre-send plugin hook surface.

## Best Possible Solution

Land or replace PR #73570 with a canonical pre-send internal hook bridge in the shared outbound delivery path, before channel send, with explicit event-name compatibility, ordering relative to plugin modification/cancel hooks, and focused tests/docs for the supported internal hook contract.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: hook-directory metadata registers literal event keys, `triggerInternalHook` dispatches only `type` and `type:action`, and current pre-send outbound delivery invokes only plugin `runMessageSending` before channel delivery.

## Solution Assessment

Yes, the narrowest maintainable solution is an outbound pre-send internal hook bridge next to `applyMessageSendingHook`. The PR or replacement should explicitly settle `message_sending` versus `message:sending` compatibility and run before delivery; modification/cancel support for internal hooks needs a deliberate contract decision.
