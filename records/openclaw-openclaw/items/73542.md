## Summary

Keep open: current main still runs the pre-delivery `message_sending` path only through the plugin hook runner, while user-defined internal hooks are registered separately and only get the existing post-delivery `message:sent` bridge. Open PR #73570 references this issue with closing syntax and should be reviewed or replaced before this issue is closed.

## What This Changes

The issue asks for user-defined hook-directory hooks for outbound pre-send messages to fire before delivery, matching the existing plugin `message_sending` lifecycle point.

## Best Possible Solution

Review and land #73570 or a narrow replacement that emits a documented pre-delivery internal hook from the shared outbound delivery path with clear ordering, naming compatibility, and observe-versus-modify semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: directory hook metadata registers event names verbatim, internal dispatch only reaches `message` and `message:<action>` listeners, and current main never creates a pre-delivery `message/sending` internal event.

## Solution Assessment

Unclear. Bridging from the shared outbound delivery path is the right boundary, but the merge-ready solution should settle whether hook-directory handlers listen as `message_sending` or `message:sending` and whether they are observe-only or can intercept delivery.
