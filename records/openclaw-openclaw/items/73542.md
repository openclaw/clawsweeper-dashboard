## Label Justifications

- `P2`: This is a meaningful hook contract and outbound guardrail gap, but it is bounded to extension behavior rather than a core outage.
- `impact:security`: The requested pre-send hook is intended for output filtering, validation, audit, and guardrail logic before user-visible delivery.

## Summary

Keep open: current main still does not define or dispatch a standalone HOOK.md/internal `message_sending` pre-send event, while the supported rewrite/cancel surface is the typed plugin `message_sending` hook. The direct fixing PR is closed unmerged, and the currently open related PRs target a different plugin-hook bypass rather than this internal-hook contract.

## What This Changes

Request to make user-defined HOOK.md hooks registered for `message_sending` fire before outbound message delivery.

## Best Possible Solution

Define the standalone pre-send hook contract explicitly, then either implement and document `message:sending` with ordering/tests or reject typed-name HOOK.md registrations with diagnostics and direct guardrails to `api.on("message_sending", ...)`.

## Reproduction Assessment

Yes, at source level: HOOK.md event names register verbatim, internal dispatch only reaches `type` or `type:action` listeners, and current outbound delivery runs typed plugin `message_sending` plus internal `message:sent` only. I did not run a live hook-directory setup in this read-only review.

## Solution Assessment

Unclear. The requested behavior is plausible, but the best fix depends on a maintainer decision about whether standalone internal hooks should get a pre-send policy event or whether unsupported typed hook names should be rejected or documented as plugin-only.
