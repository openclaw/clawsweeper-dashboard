## Summary

Keep open. Current main still routes pre-send outbound delivery through the typed plugin `message_sending` runner only, while hook-directory handlers are registered through the internal hook registry. Open PR #73570 already references this issue with closing syntax, so the issue should remain open until that PR is landed, adapted, or replaced.

## What This Changes

The issue requests that user-defined hook-directory handlers registered for `message_sending` fire before outbound message delivery, matching typed plugin `message_sending` behavior.

## Best Possible Solution

Land or replace PR #73570 with a canonical pre-send internal hook bridge in the shared outbound delivery path before channel send, with explicit `message_sending`/`message:sending` compatibility, ordering semantics relative to plugin modification and cancel hooks, and focused tests plus docs for the supported internal hook contract.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: hook-directory metadata registers literal event keys, internal dispatch reaches only `message` and `message:<action>` listeners, and current pre-send outbound delivery only invokes the typed plugin `runMessageSending` path.

## Solution Assessment

Partly. Bridging pre-send dispatch in `deliverOutboundPayloads` is the right narrow direction, but the final patch must explicitly cover event-name compatibility and pre-delivery ordering rather than only adding an undocumented fire-and-forget `message:sending` notification.
