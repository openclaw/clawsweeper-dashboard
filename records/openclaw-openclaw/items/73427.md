## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP omits the OpenAI origin while the browser WebRTC transport posts SDP to OpenAI, the OpenAI browser-session payload lacks the requested input audio/VAD/transcription settings, and the WebRTC data-channel handler still drops OpenAI error and lifecycle events. The closed Google Live report #73601 is adjacent Talk UI work, not a duplicate of this OpenAI WebRTC path.

## What This Changes

The issue requests fixing Control UI OpenAI Realtime Talk so CSP permits the required OpenAI WebRTC call endpoint, the browser session uses explicit realtime audio input settings, and the UI surfaces OpenAI realtime errors and lifecycle status instead of showing a silent live state.

## Best Possible Solution

Land a focused OpenAI browser Talk fix in the existing bundled OpenAI/Control UI path: add a tight `connect-src` allowance for the OpenAI Realtime origin, include the needed browser-session audio input/VAD/transcription settings, surface OpenAI `error` plus basic speech/response lifecycle events through Talk status, and cover those behaviors with targeted tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.26 steps, and current main statically reproduces the blockers by combining a direct browser SDP POST to OpenAI with a CSP that omits that origin; the silent-error path is visible because the WebRTC handler returns on unrecognized OpenAI events while the UI can still render `Talk live`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the existing OpenAI browser Talk path and Control UI CSP builder, not in a broad configurable CSP feature; future CSP changes should stay limited to the required OpenAI Realtime origin.
