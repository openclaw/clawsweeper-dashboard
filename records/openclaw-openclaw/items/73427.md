## Summary

Keep open. Current main still matches the report: Control UI CSP does not allow the OpenAI HTTPS origin used by the browser WebRTC SDP POST, and the OpenAI WebRTC data-channel handler still ignores realtime error and lifecycle events that would make the UI failure actionable.

## What This Changes

The issue requests fixes for Control UI Realtime Talk so OpenAI WebRTC SDP requests pass CSP, browser sessions include explicit realtime audio input/VAD settings, and realtime errors/lifecycle states surface in the UI.

## Best Possible Solution

Land a focused OpenAI browser Talk fix in the CSP builder, OpenAI browser-session payload, and WebRTC transport diagnostics, with targeted tests and a changelog entry.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main combines a direct browser fetch to https://api.openai.com/v1/realtime/calls with a connect-src directive that omits that origin, and the data-channel switch visibly drops diagnostic events.

## Solution Assessment

Yes. The narrow maintainable fix is to update the existing Control UI CSP, OpenAI provider browser-session creation, and WebRTC transport callbacks; the broader configurable CSP request and the closed Google Live issue do not replace this provider-specific bug.
