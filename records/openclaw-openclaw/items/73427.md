## Summary

Keep open. Current main still matches the report: the Control UI CSP omits the OpenAI API origin used by the browser WebRTC SDP POST, and the OpenAI WebRTC data-channel handler still drops error and lifecycle events that would give the UI an actionable failure state.

## What This Changes

The issue reports that Control UI Realtime Talk's OpenAI browser WebRTC path is blocked by CSP and hides realtime audio diagnostics after connection.

## Best Possible Solution

Land a focused OpenAI browser Talk fix: allow the required OpenAI Realtime endpoint through Control UI CSP, add explicit browser-session audio input/VAD configuration, surface data-channel error/lifecycle status, and cover it with targeted tests plus a changelog entry.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main combines a direct browser fetch to https://api.openai.com/v1/realtime/calls with a connect-src directive that omits that origin, and the event switch visibly ignores OpenAI diagnostic events.

## Solution Assessment

Yes. The narrow maintainable fix is in the existing Control UI CSP, OpenAI browser-session provider, and WebRTC transport; the broader configurable CSP issue and the closed Google Live issue do not replace this bundled OpenAI provider bug.
