## Summary

Keep open. Current main still contains the reported OpenAI browser Talk mismatch: the Control UI CSP omits the OpenAI API origin, the browser WebRTC transport posts SDP directly to OpenAI, and the WebRTC data-channel handler still ignores OpenAI error/lifecycle events that would make silent failures actionable. The related #73601 work was Google Live-specific and does not supersede this OpenAI WebRTC path.

## What This Changes

Fix Control UI OpenAI Realtime Talk so CSP permits the OpenAI WebRTC SDP exchange and the UI/session path surfaces realtime audio errors and lifecycle status.

## Best Possible Solution

Land a focused OpenAI browser Talk fix in the existing provider and Control UI path: add a tight OpenAI Realtime CSP allowance, surface OpenAI error and basic speech/response lifecycle events through Talk status, validate the browser session audio/VAD configuration against the current OpenAI API, and cover the behavior with targeted tests and a user-facing changelog entry.

Yes. The issue gives concrete 2026.4.26 steps, and current main statically reproduces the two blockers by combining a direct browser SDP POST to OpenAI with a CSP that omits that origin, plus a data-channel handler that drops diagnostic OpenAI events.

## Solution Assessment

Yes. The best repair is a narrow OpenAI browser Talk change in the existing CSP, WebRTC transport, and OpenAI browser-session code; #73601 fixed adjacent Google Live behavior but does not address this provider path.
