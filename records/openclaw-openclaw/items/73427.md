## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP omits the OpenAI origin, the browser WebRTC transport posts SDP directly to that origin, the OpenAI browser-session payload only configures output voice, and the data-channel handler drops OpenAI error/lifecycle events so the UI can show a non-actionable live state.

## What This Changes

The issue asks OpenClaw to fix OpenAI Control UI Realtime Talk by allowing the required OpenAI WebRTC endpoint in CSP, sending appropriate browser-session audio/VAD settings, and surfacing realtime audio errors and lifecycle status in the UI.

## Best Possible Solution

Land a focused OpenAI browser Talk fix in the existing bundled provider path: add a tight CSP allowance for the OpenAI Realtime origin, include the needed browser session input audio/VAD/transcription settings, surface OpenAI error and basic speech/response lifecycle events through Talk status, and cover the CSP, provider payload, transport event handling, and UI status behavior with targeted tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.26 steps, and current main statically reproduces the blockers by combining a direct browser SDP POST to OpenAI with a CSP that omits that origin; the silent-error path is visible because unknown OpenAI data-channel events are ignored while the UI can still render Talk live.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the existing OpenAI browser Talk implementation rather than add a broad configurable CSP feature; the related Google Live report is adjacent browser Talk status work, not a duplicate of this OpenAI WebRTC path.
