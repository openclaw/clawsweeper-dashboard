## Summary

Keep open. Current main still has the reported OpenAI browser Realtime Talk blockers: the Control UI CSP does not allow the OpenAI Realtime API origin used by the browser SDP POST, the WebRTC data-channel handler still drops OpenAI error and lifecycle events, and the browser session minting payload still lacks the requested input audio/VAD configuration.

## What This Changes

The issue requests fixing OpenAI Realtime Talk in the Control UI by allowing the OpenAI Realtime SDP origin in CSP, configuring browser audio input/VAD, and surfacing realtime error/lifecycle events.

## Best Possible Solution

Land a focused OpenAI browser Talk fix: add a tight Control UI CSP allowance for `https://api.openai.com`, surface OpenAI WebRTC `error` and basic speech/response lifecycle events through the existing Talk status callbacks, align the browser-session audio input/VAD payload with the current OpenAI Realtime schema, and cover the path with targeted tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main combines a direct browser SDP POST to the OpenAI Realtime API with a CSP `connect-src` that omits that origin, and the current data-channel handler visibly ignores non-transcript/tool diagnostic events.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the existing Control UI CSP, OpenAI WebRTC transport, and OpenAI realtime provider path; the broader configurable CSP request in #47565 and the closed Google Live status issue #73601 do not replace this OpenAI-specific repair.
