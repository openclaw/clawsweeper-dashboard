## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP omits the OpenAI API origin needed for the WebRTC SDP POST, the OpenAI browser session payload lacks the explicit input audio/VAD settings described in the report, and the WebRTC data-channel handler still drops OpenAI error and lifecycle events before the UI can display them.

## What This Changes

The issue requests a focused Control UI Realtime Talk fix for the bundled OpenAI browser provider: allow the OpenAI WebRTC endpoint through CSP, configure browser-session audio/VAD behavior, and surface realtime error/lifecycle status in the UI.

## Best Possible Solution

Land a focused OpenAI browser Talk fix in the existing Control UI and OpenAI plugin paths: allow the required OpenAI Realtime browser endpoint through CSP, add provider-appropriate browser-session audio input/VAD configuration after checking the current API contract, map error/speech/response events to visible Talk status, and cover the behavior with targeted tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main combines a direct browser fetch to https://api.openai.com/v1/realtime/calls with a CSP connect-src that omits that origin, and the WebRTC data-channel handler visibly ignores the diagnostic OpenAI event types described by the report.

## Solution Assessment

Yes. The best path is a narrow repair in the existing CSP builder, OpenAI browser session provider, and WebRTC transport; the broader configurable CSP issue and the closed Google Live issue do not replace this specific bundled OpenAI provider bug.
