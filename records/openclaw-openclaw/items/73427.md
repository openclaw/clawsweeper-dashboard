## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP omits the OpenAI API origin while the WebRTC transport posts SDP there, and the WebRTC data-channel handler still drops OpenAI error and lifecycle events so the UI can remain in a non-actionable live state.

## What This Changes

The issue asks OpenClaw to fix Control UI Realtime Talk for the bundled OpenAI WebRTC provider by allowing the OpenAI realtime endpoint in CSP, configuring browser audio/VAD behavior, and surfacing realtime audio errors and lifecycle status in the UI.

## Best Possible Solution

Land a focused OpenAI browser Talk fix in the existing provider and Control UI path: add a tight CSP allowance for the OpenAI Realtime origin, surface OpenAI error and basic speech/response lifecycle events through Talk status, validate the browser session audio/VAD configuration against the current OpenAI API, and cover those behaviors with targeted tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.26 steps, and current main statically reproduces the blockers by combining a direct browser SDP POST to OpenAI with a CSP that omits that origin; the silent-error path is visible because unknown OpenAI data-channel events are ignored while the UI can still render Talk live.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the existing OpenAI browser Talk implementation rather than add a broad configurable CSP surface; #73601 is adjacent browser Talk status work for Google Live, not a duplicate of this OpenAI WebRTC path.
