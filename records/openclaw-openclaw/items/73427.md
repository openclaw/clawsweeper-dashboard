## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP omits the OpenAI API origin used by the WebRTC SDP request, and the browser data-channel handler still drops OpenAI error and lifecycle events instead of surfacing an actionable UI state.

## What This Changes

The issue requests fixing Control UI Realtime Talk so the bundled OpenAI WebRTC path is allowed by CSP, configures browser speech-to-speech audio input explicitly, and exposes realtime error/speech/response lifecycle status in the UI.

## Best Possible Solution

Land a focused OpenAI browser Talk fix: allow the required OpenAI Realtime browser endpoint through Control UI CSP, mirror existing realtime lifecycle/error status handling in the WebRTC transport, add explicit browser-session audio input/VAD configuration after checking the provider contract, and cover the behavior with targeted tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main combines a direct browser fetch to https://api.openai.com/v1/realtime/calls with a CSP connect-src that omits that origin, and the OpenAI WebRTC event switch visibly ignores error/speech/response lifecycle events.

## Solution Assessment

Yes. The narrow maintainable direction is to fix the existing Control UI CSP, OpenAI WebRTC transport, and OpenAI browser-session provider path; the broader configurable CSP request in #47565 and the closed Google Live issue #73601 do not replace this bundled OpenAI provider bug.
