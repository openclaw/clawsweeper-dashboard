## Summary

Keep open. Current main still has the reported OpenAI browser Realtime Talk blockers: Control UI CSP does not allow the OpenAI API origin used by the browser SDP POST, and the WebRTC data-channel handler still ignores OpenAI error and lifecycle events after the session reaches the live state.

## What This Changes

The issue requests a focused Control UI/OpenAI Realtime Talk fix for the CSP WebRTC endpoint, surfaced realtime audio errors, lifecycle status, and browser-session audio/VAD configuration.

## Best Possible Solution

Land a focused OpenAI browser Talk fix: add a tight CSP connect-src allowance for the OpenAI Realtime origin, surface OpenAI error and basic speech/response lifecycle events through the existing Talk status callbacks, validate and add the required browser-session input audio/VAD configuration, and cover the path with targeted tests and a user-facing changelog entry.

Yes. The static reproduction is high confidence: current main combines a direct browser fetch to https://api.openai.com/v1/realtime/calls with a CSP connect-src that omits that origin, and the current WebRTC event switch visibly drops provider errors and lifecycle events.

Yes. The narrow maintainable fix belongs in the existing Control UI CSP, OpenAI WebRTC transport, and OpenAI realtime provider path; #73601 is adjacent Google Live status work, and the broader configurable CSP request in #47565 is not required to solve this bundled provider bug.
