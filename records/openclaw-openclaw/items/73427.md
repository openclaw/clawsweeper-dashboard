## Summary

Keep open. Current main still matches the report: Control UI CSP omits the OpenAI API origin used by the browser SDP POST, the OpenAI browser-session payload lacks explicit input/VAD settings, and the WebRTC data-channel handler drops OpenAI error/lifecycle events instead of surfacing an actionable UI state.

## What This Changes

The issue requests fixing Control UI OpenAI Realtime Talk by allowing the WebRTC SDP endpoint through CSP, minting browser sessions with explicit audio input/VAD config, and surfacing realtime audio errors/status.

## Best Possible Solution

Land a focused OpenAI browser Talk fix: allow only the required OpenAI API origin in Control UI CSP, add explicit OpenAI browser-session audio input/VAD/transcription config, surface data-channel error/lifecycle status, and cover the behavior with targeted tests plus a changelog entry.

Yes. Static reproduction is high confidence: current main combines a direct browser fetch to https://api.openai.com/v1/realtime/calls with a connect-src directive that omits that origin, and the event switch visibly ignores OpenAI diagnostic events.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the existing Control UI CSP, OpenAI browser-session provider, and WebRTC transport rather than in a broad configurable CSP feature or a core migration.
