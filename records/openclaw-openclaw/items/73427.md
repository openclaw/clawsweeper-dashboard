## Summary

Keep open. Current main still has the reported OpenAI browser Realtime Talk blockers: the Control UI CSP omits the OpenAI API origin used by the browser SDP POST, the WebRTC data-channel handler drops OpenAI error and lifecycle events, and the browser session minting payload lacks the requested input audio/VAD configuration.

## What This Changes

Request to unblock Control UI OpenAI Realtime Talk by allowing the OpenAI WebRTC SDP origin, configuring browser input audio/VAD, and surfacing realtime error and lifecycle status in the UI.

## Best Possible Solution

Land a focused OpenAI browser Talk fix in the existing Control UI CSP, OpenAI WebRTC transport, and OpenAI realtime provider path, with targeted tests and a user-facing changelog entry; leave broader configurable CSP policy and Google Live behavior to their separate tracks.

## Reproduction Assessment

Yes. The static reproduction is high confidence: current main combines a direct browser fetch to https://api.openai.com/v1/realtime/calls with a CSP connect-src that omits that origin, and the current WebRTC event switch visibly drops provider error and lifecycle events.

## Solution Assessment

Yes. The narrow maintainable path is to fix the bundled OpenAI browser Talk flow in place; #73601 is Google-specific and #47565 is broader than needed for this product bug.
