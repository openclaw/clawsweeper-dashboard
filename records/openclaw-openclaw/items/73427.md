## Summary

Keep open. Current main still contains both reported OpenAI browser Talk blockers: the Control UI CSP does not allow the OpenAI API origin used by the WebRTC SDP POST, and the OpenAI WebRTC data-channel handler still drops error and lifecycle events instead of surfacing an actionable UI state.

## What This Changes

Fix Control UI Realtime Talk so OpenAI WebRTC SDP setup passes CSP and OpenAI realtime audio errors, VAD, and response lifecycle status are visible in the UI.

## Best Possible Solution

Land a focused fix in the existing OpenAI browser Talk path: add a narrow CSP allowance for the OpenAI Realtime origin, surface OpenAI error and basic speech/response lifecycle events through the existing Talk status callbacks, validate the browser session audio/VAD defaults, and add targeted tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete reproduction steps, and current main statically reproduces the two core blockers by combining a direct browser POST to https://api.openai.com/v1/realtime/calls with a CSP that omits that origin, plus a data-channel switch that ignores OpenAI error and lifecycle events.

## Solution Assessment

Yes. The narrow maintainable fix is in the existing Control UI CSP, OpenAI browser session minting, and WebRTC Talk event handler; a broad configurable CSP surface or a Google Live status fix would not directly solve this OpenAI-specific failure.
