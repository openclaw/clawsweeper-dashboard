## Summary

Keep this PR open because it addresses a real session-lock availability bug that current main still has, but the submitted patch is not merge-ready: the bounded abort path can still leave the drain state stuck when another release path already owns the drain, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes the embedded attempt session-lock controller so abort cleanup races retained-write drain against the abort settle timeout, force-releases the held lock on timeout, poisons later writes as takeover, and adds focused controller tests.

## Best Possible Solution

Fix the drain-owner takeover interleaving, add a regression for that ordering, and attach redacted real/component proof that a hung abort releases within the bound while late writes are poisoned.

## Reproduction Assessment

Yes, at source level: current main waits indefinitely in releaseHeldLockForAbort when a retained write never settles, and the PR adds a focused never-settling retained-write test. I did not run a live reproduction because this review is read-only.

## Solution Assessment

No, not yet. Bounded force-release is the right layer for the reported in-process abort window, but this branch is not the best mergeable fix until the stuck drain-owner interleaving is fixed and after-fix real behavior proof is supplied.
