## Summary

Keep open: current main and the latest release still reject empty `aws-sdk` auth before Bedrock media execution, while this PR is a focused candidate fix with no blocking code findings. The remaining decision is maintainer acceptance of keyless SDK-chain auth for media paths, not an automated repair task.

## What This Changes

The PR lets image and shared audio/video media-understanding auth resolution use Bedrock `aws-sdk` keyless execution instead of requiring a static API key.

## Best Possible Solution

Land this focused fix, or a same-shape maintainer replacement, once maintainers accept SDK-chain auth as the intended Bedrock media behavior and decide whether the current proof is enough.

## Reproduction Assessment

Yes. Current main and `v2026.6.10` still call `requireApiKey` on empty `aws-sdk` auth in the image and media runner paths, and the Bedrock docs plus linked live reports define the expected SDK-chain behavior.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I found: it reuses the existing keyless runner path, mirrors the chat-path `aws-sdk` allowance, avoids persisting empty runtime keys, and leaves non-`aws-sdk` missing-key errors fail-closed.
