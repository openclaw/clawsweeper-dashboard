## Summary

Keep open. Current main still lacks the requested host-owned `maxTurnsPerRun` / `retryBudget` contract and structured termination event; the closest existing protections are wall-clock timeouts and opt-in pre-tool loop detection, while the related open items are narrower.

## What This Changes

Request host-level per-dispatch turn and retry budgets with default/per-agent config, enforced termination, trajectory recording, and non-zero dispatch exit when caps fire.

## Best Possible Solution

Define a host-owned run-budget policy in the runtime/harness contract, enforce it before model and tool continuation, emit a structured trajectory termination, and document inheritance across supported runtimes.

## Reproduction Assessment

Yes. Static inspection of current main is enough: the requested symbols are absent, the schema is strict, and the nearest in-tree guard is opt-in loop detection rather than a required per-dispatch budget.

## Solution Assessment

Unclear. Harness-level enforcement is the right class of solution, but the exact public config shape and cross-runtime API coverage need maintainer approval before a safe implementation.

## Risks / Open Questions

- Public config placement still needs maintainer design because the proposal uses `agents.defaults` / per-agent overrides while current loop settings live under `tools.loopDetection` and `agents.list[].tools.loopDetection`.
- Runtime coverage needs an explicit boundary across PI, registered/plugin harnesses, ACP/native harnesses, and CLI-backed runs so the guarantee is not accidentally PI-only.
- Existing `agents.defaults.timeoutSeconds` and LLM idle watchdog behavior are time-based caps, so their interaction with deterministic turn/error budgets needs to be specified.
- Landing #75924 would help a narrower cross-tool error-cascade case but would not close this broader host-owned turn/retry-budget request.
