## Summary

Keep open. Current main still lacks the requested host-owned turn cap and retry budget; the nearest existing protection is opt-in, pattern-based tool-loop detection and wall-clock timeout handling, so the reported safety gap remains valid but needs maintainer API/runtime scope approval before implementation.

## What This Changes

The issue requests configurable harness-level `maxTurnsPerRun` and `retryBudget` limits that terminate runaway tool-calling dispatches and record a structured trajectory event.

## Best Possible Solution

Define a documented host-owned run-budget contract in the agent runtime/harness layer, resolve global and per-agent values before agent execution, enforce limits before subsequent model/tool cycles, emit a structured trajectory termination event, and document which native/plugin harnesses are covered.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the requested config keys and trajectory event are absent, while the nearest guard is opt-in loop detection plus a wall-clock timeout rather than a hard per-dispatch budget.

## Solution Assessment

Unclear. Harness-level enforcement is the right class of solution, but the exact config placement and cross-runtime API boundary should be approved before a fix PR so the safety contract is not partial or misleading.
