## Summary

Keep open: current main and the latest release still have the documented `browser tabs --browser-profile` failure path because the profile option is parent-only. The gateway/NVM part is partly covered by current audit behavior and the open policy thread, but that does not supersede the browser regression.

## What This Changes

The issue asks to fix two regressions: post-subcommand browser profile CLI syntax no longer works for WSL2 browser docs, and doctor restarts a gateway service that still points at an unstable NVM Node path.

## Best Possible Solution

Restore backward-compatible browser profile parsing for the documented WSL2 `browser tabs --browser-profile <name>` flow, keep the current parent-option syntax working, and leave durable NVM gateway runtime policy to the canonical thread.

## Reproduction Assessment

Yes for the browser regression by source inspection and latest-release check: `--browser-profile` is only registered on `browser`, while the WSL2 guide still places it after `tabs`. No live WSL2/systemd reproduction was established for the gateway loop.

## Solution Assessment

Yes for a narrow browser compatibility repair: restore or document support for the previously documented post-subcommand profile option without changing current parent-option syntax. The gateway service runtime policy should stay with https://github.com/openclaw/openclaw/issues/60612 rather than expanding this repair.
