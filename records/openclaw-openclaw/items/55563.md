## Summary

Keep open: current main and v2026.5.18 still document `openclaw browser tabs --browser-profile remote`, but the browser `tabs` subcommand only reads the parent option and does not accept `--browser-profile` after the subcommand. The gateway cycling half overlaps the still-open version-manager runtime policy thread, so the focused repair path is the browser CLI compatibility regression first.

## What This Changes

The issue asks to restore the documented post-subcommand browser-profile CLI syntax and address gateway restart cycling after `doctor --fix` on WSL2/systemd services using NVM-managed Node.

## Best Possible Solution

Restore local backward-compatible browser-profile parsing for documented browser subcommands with focused regression coverage, while keeping durable version-manager Gateway runtime policy in https://github.com/openclaw/openclaw/issues/60612.

## Reproduction Assessment

Yes for the browser regression by source and dependency inspection: root positional options make post-subcommand parent flags unknown unless the subcommand registers them, and current `tabs` does not. No for the WSL2 gateway cycling loop in this environment; that part is supported by issue context and adjacent doctor/service-audit source only.

## Solution Assessment

Yes for the narrow browser repair: add compatibility parsing and tests without disabling root positional options or changing existing parent-option syntax. The gateway service behavior should remain tied to the existing runtime-policy issue unless maintainers want a separate WSL2/systemd repair.
