## Summary

Keep open: current main still documents a `browser tabs --browser-profile` flow while the `tabs` subcommand only reads browser parent options and does not register that option locally. The gateway/NVM service half overlaps newer service-audit behavior and the open runtime-policy thread, but that does not resolve the browser CLI compatibility regression.

## What This Changes

The issue reports two regressions: post-subcommand browser profile CLI syntax fails for WSL2 browser docs, and `doctor --fix` can leave a gateway service cycling when its systemd unit points at an unstable version-manager Node path.

## Best Possible Solution

Restore backward-compatible browser profile parsing for documented browser subcommand forms, add focused CLI coverage, and leave the durable version-manager Gateway runtime policy to https://github.com/openclaw/openclaw/issues/60612 unless a distinct WSL2 service-loop repro remains after that policy is resolved.

## Reproduction Assessment

## Solution Assessment

Yes for the narrow browser fix: restore compatibility and tests without changing the existing parent-option syntax. The gateway service behavior should remain scoped to the existing runtime-policy issue unless maintainers want a separate WSL2/systemd repair.
