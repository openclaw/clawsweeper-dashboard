## Summary

Keep open. Current main still has a user-facing browser CLI/docs mismatch: `--browser-profile` is registered on the parent `browser` command while the WSL2 troubleshooting guide still shows it after `tabs` and `open`. The gateway half has later service-audit mitigation for version-managed Node paths, but there is not enough source/test proof that the exact WSL2 systemd+nvm failed-unit cycling case is fully fixed.

## What This Changes

The issue reports two v2026.3.24 regressions: the documented `openclaw browser tabs --browser-profile remote` form fails, and `openclaw doctor --fix` can restart but not repair a WSL2 systemd gateway service pinned to an unstable nvm Node path.

## Best Possible Solution

Keep this issue open and resolve the two regressions deliberately: restore `--browser-profile` compatibility for the documented browser leaf commands or update the WSL2 guide and migration notes if the new ordering is intentional, then add focused proof that `doctor --fix` repairs or clearly diagnoses WSL2 systemd services whose ExecStart points at a removed nvm Node path.

## Reproduction Assessment

Yes for the browser regression: source inspection shows the option is parent-scoped, positional options are enabled, and the WSL2 guide still publishes the after-subcommand form. Unclear for the gateway regression: current source can detect version-managed Node paths, but there is no high-confidence current-main reproduction or end-to-end test for the reported WSL2 systemd+nvm failed-unit cycling case.

## Solution Assessment

Unclear. The narrowest browser fix is either a compatibility option on affected leaf commands with tests or a deliberate docs/changelog migration update; the gateway half needs a focused doctor/service-audit test or live WSL2 proof before treating the existing mitigation as sufficient.
