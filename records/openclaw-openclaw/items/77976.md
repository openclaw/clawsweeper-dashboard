## Summary

Keep open: current main still has the reported unbounded `agentRunCache` shape, and the same-author tracking PR is open, so the issue should remain paired with that implementation candidate until it merges or is replaced.

## What This Changes

The issue reports that the gateway `agent.wait` lifecycle snapshot cache is TTL-bound but not size-bound, so sustained unique terminal run events can grow memory with run rate over the 10-minute TTL window.

## Best Possible Solution

Land #77973 or an equivalent narrow gateway patch that enforces a hard entry cap after cache inserts, preserves the existing TTL/wait semantics, and adds regression coverage plus a changelog entry.

## Reproduction Assessment

Yes, from source inspection: emitting many unique terminal lifecycle events within 10 minutes reaches `recordAgentRunSnapshot`, where TTL pruning removes none of the fresh entries and the Map grows one entry per unique run. I did not run a live stress reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to add a hard max-entry enforcement point in `agent-job.ts` next to the existing TTL pruning and cover it through the real lifecycle event path. The current issue should stay open while the linked PR is reviewed rather than spawning a second competing fix.
