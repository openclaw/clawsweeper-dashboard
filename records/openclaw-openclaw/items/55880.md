## Summary

Keep open. Current main has partial fixes for macOS update recovery, including installed-but-unloaded LaunchAgent re-bootstrap and stricter unhealthy-restart failure handling, but it still does not repair or deterministically fail the missing-service path during non-interactive update recovery. Cron failure notifications also remain fire-and-forget and separate from the persisted run delivery status, so the visible-notification mismatch is not fully resolved.

## What This Changes

The issue asks OpenClaw to make scheduled self-update failures visibly reportable and to recover macOS LaunchAgent gateway states that update/restart currently leaves down.

## Best Possible Solution

Keep the issue open and resolve it with a maintainer-scoped macOS self-update recovery design: package update recovery should share a safe gateway repair path for installed-but-unloaded and missing LaunchAgent states, wait for readiness without the startup race, and record failure-notification attempts separately from primary cron delivery status. The provider credential question should be reproduced narrowly before changing update preflight semantics.

## Reproduction Assessment

Yes for the central recovery gap: a package update with serviceState.installed false leaves restart prep disabled, and non-interactive update doctor mode will not install or restart the gateway service. The provider-preflight subcase is unclear because current memory-search doctor code appears to warn rather than throw.

## Solution Assessment

No, current main is not the complete fix. The recent LaunchAgent and package-restart hardening is useful, but the missing-service state and notification-accounting path still need explicit handling rather than relying on `gateway restart` and fire-and-forget failure announcements.
