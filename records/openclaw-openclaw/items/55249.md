## Summary

Keep open. Current main has the Gateway/session-store label contract and programmatic label paths, but the requested user-facing CLI workflow is still missing: no `openclaw sessions label` subcommand is registered, local text output still centers the opaque key, and the local display-row projection drops label/displayName fields before JSON output.

## What This Changes

Add persistent human-readable session labels/nicknames that can be set from the CLI and shown in `openclaw sessions` text and JSON listings, with optional programmatic and auto-labeling support.

## Best Possible Solution

Add a focused CLI/docs/test layer over the existing `sessions.patch` label contract: set and clear stored labels, preserve label/displayName in local rows, render them in text and JSON listings, and leave automatic label generation as a separate follow-up.

## Reproduction Assessment

Yes. Source inspection on current main provides a high-confidence reproduction path: inspect or invoke the registered sessions commands and local display projection, which show no label subcommand and no label/displayName field in local list output.

## Solution Assessment

Yes. Reusing `parseSessionLabel`, `sessions.patch`, and the Gateway list/displayName semantics is the narrowest maintainable solution; creating a separate subject/displayName mutation path would duplicate existing contract behavior.
