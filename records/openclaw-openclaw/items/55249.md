Keep open: current main has backend label primitives and Control UI label editing, but the requested first-class CLI/chat naming surfaces and `openclaw sessions` label display still remain and need maintainer product scoping.

## What This Changes

The issue requests persistent human-readable session labels/nicknames with CLI listing and mutation support, programmatic visibility, chat/TUI commands, and optional auto-labeling.

## Best Possible Solution

Choose one canonical label contract, reuse `sessions.patch` and existing session metadata, add set/show/clear plus list/search display surfaces, and track auto-labeling as a separate opt-in follow-up.

## Reproduction Assessment

Yes. Source inspection on current main shows existing label storage and Control UI editing, but no CLI label/rename subcommand, no `/label` command, and no label column or primary friendly-name path in `openclaw sessions` text output.

## Solution Assessment

Unclear until maintainer scoping is decided. The existing Gateway `sessions.patch` label contract is the right foundation, but the issue should not bundle manual naming, CLI display, chat commands, and automatic title generation without a product decision.
