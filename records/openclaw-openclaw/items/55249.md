## Summary

Keep open. Current main still has the backend/session-store primitives for labels, but the user-facing CLI workflow requested here remains incomplete: no `openclaw sessions label` command is registered, local `openclaw sessions` table output does not show labels, and local JSON rows cannot include labels because the display-row projection drops them.

## What This Changes

Request to add human-readable session labels or nicknames, including a CLI label command and visible label fields in session listings and JSON output.

## Best Possible Solution

Add a focused CLI/docs/test layer on top of the existing label store contract: support setting and clearing a stored session label, project label/displayName through local session rows, render it in text and JSON listings without dropping existing fields, and leave automatic label generation as a separate follow-up unless maintainers explicitly scope it in.

## Reproduction Assessment

Yes for the observable absence. Source inspection on current main shows there is no sessions label subcommand, no label/displayName field in local display rows, and no label column or docs entry for the requested CLI workflow.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing `parseSessionLabel` and Gateway/store semantics rather than adding a second subject/displayName mutation path or redesigning the session protocol.
