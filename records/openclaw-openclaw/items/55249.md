## Summary

Keep open. Current main and v2026.5.12 still have backend session-label primitives, but the requested user-facing workflow is not implemented: there is no `openclaw sessions label` command, no `/label` chat/TUI command, and local `openclaw sessions` text/JSON output still drops stored labels. Related items either point back to this issue as canonical or cover narrower auto-label/status display work.

## What This Changes

Add human-readable session labels/nicknames with CLI/chat/programmatic setting and label display in session listings.

## Best Possible Solution

Implement a focused manual label path over the existing `sessions.patch` label contract, show labels in local CLI text/JSON without hiding raw keys, add the `/label` chat/TUI workflow, document it, and defer auto-generated labels to the narrower follow-up.

## Reproduction Assessment

Not applicable as a bug reproduction, but the feature gap is source-verifiable on current main: the CLI registration, local row projection, text renderer, docs, and slash-command search still lack a label-setting/listing workflow.

## Solution Assessment

Yes. Reusing the existing `label` field, `sessions.patch`, `parseSessionLabel`, and Gateway list semantics is the narrow maintainable solution; adding a parallel mutable `displayName`/`subject` contract or bundling auto-title generation would be broader than needed.
