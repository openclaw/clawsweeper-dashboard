## Summary

Keep open. Current main has backend label storage, Gateway listing, and reset preservation, but the requested CLI workflow remains incomplete: no `sessions label` command is registered, and local `openclaw sessions` table/JSON rows still do not project or render stored labels.

## What This Changes

Request adding persistent human-readable session labels/nicknames, including a CLI command and label visibility in `openclaw sessions` text and JSON output.

## Best Possible Solution

Add a focused CLI/docs/test layer on the existing `sessions.patch` label contract, projecting stored labels into local rows and rendering them in text and JSON listings while leaving auto-generated labels for a separate follow-up.

## Reproduction Assessment

Yes. Source inspection on current main shows no registered label-management subcommand, no local label/displayName projection, and no label column in the table path, while Gateway tests already prove backend label storage/listing exists.

## Solution Assessment

Yes. Reusing `sessions.patch` and `parseSessionLabel` is the narrow maintainable path; adding a separate subject/displayName mutation path would duplicate existing session metadata semantics.
