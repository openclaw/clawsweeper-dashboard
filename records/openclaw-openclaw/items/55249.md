## Summary

Keep open. Current main has backend, Gateway, reset, and Control UI primitives for session labels, but the requested CLI workflow remains incomplete: there is no `openclaw sessions label` command, local session table output still centers opaque keys, and local JSON output drops stored label/displayName fields before rendering.

## What This Changes

Add user-settable session labels/nicknames that persist and appear in `openclaw sessions` table and JSON output, with optional programmatic/default label support.

## Best Possible Solution

Add a focused CLI/docs/test layer on top of the existing label store contract: set and clear a stored session label, project label/displayName through local session rows, render it in text and JSON listings without dropping existing fields, and leave automatic label generation as a separate product decision.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: create a session-store entry with `label`, then `openclaw sessions --json` cannot include it because `toSessionDisplayRows` drops the field, and the text renderer has no label column or fallback; the command tree also has no label-setting subcommand.

## Solution Assessment

Yes for the narrow repair. The maintainable solution is to reuse or mirror the existing `parseSessionLabel` and `sessions.patch` semantics instead of introducing a second subject/displayName mutation path; auto-labeling should not block the CLI label fix.
