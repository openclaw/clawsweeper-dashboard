## Summary

Keep open. Current main has the session-store and Gateway label primitives, plus label preservation across reset, but the requested user-facing workflow is still missing: no `openclaw sessions label` command, no `/label` chat/TUI command, and local `openclaw sessions` text/JSON output still projects opaque keys instead of stored labels.

## What This Changes

Request to let users assign persistent human-readable session labels or nicknames and show them in session listings, JSON output, and chat/TUI workflows.

## Best Possible Solution

Implement a focused manual label path over the existing `sessions.patch` contract: set/clear stored labels, show sanitized labels in local text and JSON listings without hiding resolvable keys, add the chat `/label` workflow, document it, and defer auto-label generation to a separate follow-up.

## Reproduction Assessment

Not applicable as a bug reproduction, but the feature gap is source-verifiable on current main: the CLI registration, local display row projection, and slash-command search show no label-setting workflow or local label output.

## Solution Assessment

Yes. Reusing `sessions.patch`, `parseSessionLabel`, and Gateway `label`/`displayName` semantics is the narrowest maintainable path; adding a parallel subject/displayName mutation path or auto-labeling in the first PR would be unnecessary scope.
