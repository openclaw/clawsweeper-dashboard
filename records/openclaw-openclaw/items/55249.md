## Summary

Keep open. Current main still has the backend and Control UI pieces for session labels, but the requested user-facing workflow is not complete: there is no `openclaw sessions label` command, no `/label` chat command, and local `openclaw sessions` text/JSON output still omits stored labels.

## What This Changes

Request to add persistent human-readable labels or nicknames for sessions, show them in `openclaw sessions` text/JSON output, and optionally support chat/TUI labeling and auto-labeling.

## Best Possible Solution

Implement the focused manual label path over the existing `sessions.patch` label contract, show stored labels in local CLI text/JSON while preserving raw keys, add the `/label` chat/TUI workflow, and defer auto-generated labels or richer typed metadata to narrower follow-ups.

## Reproduction Assessment

Not applicable as a bug reproduction, but the feature gap is source-verifiable on current main: the CLI registration, local row projection, text renderer, docs, and slash-command registry still lack the requested label-setting/listing workflow.

## Solution Assessment

Yes. Reusing the existing `label` field, `sessions.patch`, Gateway list semantics, and Control UI label path is the narrow maintainable solution; adding a parallel mutable `displayName`/`subject` contract or bundling auto-title generation would be broader than needed.
