## Summary

Keep open: the PR addresses a real doctor repair gap, but the new symlink repair trusts persisted npm dependency keys when deriving a filesystem write path, so it needs a small path-boundary fix before merge.

## What This Changes

The PR adds doctor warnings and `doctor --fix` repair for missing or stale managed npm plugin `node_modules/openclaw` peer links, plus docs, changelog, and regression tests.

## Best Possible Solution

Keep the doctor peer-link repair, but validate managed npm package names or assert resolved paths stay inside the managed npm root before any remove or symlink operation.

## Reproduction Assessment

Source-reproducible, not live-reproduced: current main links the host peer during install but has no doctor repair for missing or stale links after managed npm drift. I did not execute a filesystem reproduction because this review is read-only.

## Solution Assessment

No: doctor is the right repair boundary, but this implementation should guard the derived package/link paths before mutating the filesystem.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Keep peer-link repair inside the managed npm root:** `src/commands/doctor-plugin-registry.ts:294`
  - body: This repair trusts dependency keys from the managed npm root `package.json` to build `npmRoot/node_modules/<packageName>`. If that persisted manifest is corrupted with a key containing `../`, `doctor --fix` can remove and recreate an escaped `<dir>/node_modules/openclaw` path outside the managed npm root. Validate registry package names or resolve-and-check the derived package/link path before calling `rmSync`/`symlinkSync`.
  - confidence: 0.82
