## Summary

Current main provides mature opt-in tool sandbox backends and a public backend SDK, but it does not solve the central request: sandboxing remains off by default, native plugins still share the Gateway process trust boundary, and there are no tiered presets or per-plugin environment, filesystem, network, and resource policies. This remains useful as the canonical architecture tracker and requires explicit product, security, compatibility, and milestone decisions rather than one automated implementation.

## What This Changes

Request a unified secure-by-default sandbox architecture with cross-platform isolation, tiered presets, plugin capability manifests, credential scoping, and resource controls.

## Best Possible Solution

Keep this as the canonical owner-reviewed architecture tracker, split delivery into independently reviewable milestones, and converge on secure defaults, isolated execution boundaries, declarative capability policy, credential scoping, resource controls, and tested migrations while reusing the existing sandbox backend SDK.

## Risks / Open Questions

- The proposal spans plugin execution architecture, security boundaries, public SDK contracts, configuration defaults, credentials, migrations, and several operating-system isolation mechanisms; treating it as one implementation would be difficult to review, validate, and roll back safely.
- Changing the default from off to an enforced preset can break existing tools, native plugins, network access, host integrations, and deployment workflows unless fresh-install and upgrade behavior are designed and proven together.
- A capability manifest is not itself an isolation boundary; enforcing declarations only in an in-process proxy would not prevent compromised native plugin code from directly accessing process memory, environment variables, filesystem APIs, or network APIs.
- Platform-native implementations have materially different availability and trust properties, so a nominally unified preset could provide inconsistent protection unless capability detection and fail-open versus fail-closed behavior are explicit.
