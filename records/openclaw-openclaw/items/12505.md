## Summary

Keep open. Current main has useful opt-in tool sandbox backends and a plugin sandbox-backend SDK seam, but the requested secure-by-default tiered presets, native plugin/process isolation, IPC-proxied plugin execution, manifest-enforced env/fs/net/resource policy, and default credential scoping are still not implemented; the live discussion also treats this as the canonical architecture tracker.

## What This Changes

The issue requests a unified built-in sandbox architecture with multi-platform isolation, secure-by-default tiered presets, plugin manifest policies, and credential/resource controls.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker, or split it into reviewed milestones for default presets, plugin isolation, IPC/proxy SDK enforcement, manifest policies, credential scoping, resource limits, and migration docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence feature-gap verification path from current source and docs: sandboxing is optional tool isolation, native plugins remain in-process, and config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is maintainer security/product design followed by sequenced milestones across config/schema, plugin loader, SDK IPC/proxying, sandbox backends, manifest policy, credentials, resource limits, and migration tests.
