## Summary

Current main provides useful opt-in Docker, SSH, OpenShell, and plugin-provided tool-sandbox backends, but it still lacks the central requested plugin process boundary, secure-by-default presets, IPC-proxied SDK, per-plugin capability policy, and credential scoping; this should remain the canonical owner-reviewed architecture tracker.

## What This Changes

Requests a unified, secure-by-default, multi-platform sandbox architecture with tiered presets, plugin isolation, capability manifests, credential scoping, and resource controls.

## Best Possible Solution

Keep this as the canonical architecture tracker, approve a threat model and compatibility plan, then split plugin process isolation, IPC SDK enforcement, capability manifests, credential scoping, platform backends, resource controls, and preset migration into focused owner-reviewed milestones.

## Reproduction Assessment

Not applicable; this issue proposes a new security architecture and default posture rather than reporting one narrowly defined current-main failure. Source inspection does confirm that the requested plugin-isolation and preset capabilities remain absent.

## Solution Assessment

No, the complete umbrella proposal should not be implemented as one change. The best approach is a staged, threat-modeled design using the existing generic sandbox backend groundwork while separately reviewing plugin APIs, defaults, credentials, platform enforcement, and migration safety.
