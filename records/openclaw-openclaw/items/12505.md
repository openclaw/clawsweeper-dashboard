## Summary

Keep open. Current main still has useful opt-in tool sandbox backends and a plugin sandbox-backend seam, but it does not implement the central request for secure-by-default tiered presets, plugin/process isolation, IPC-proxied plugin execution, manifest-enforced env/fs/net/resource policies, or default credential scoping; the existing discussion also treats this as the canonical sandbox/isolation architecture tracker rather than resolved by narrower Docker or local-user sandbox work.

## What This Changes

Request a unified, multi-platform sandbox architecture with secure-by-default tiered presets, plugin/process isolation, IPC-proxied plugin execution, manifest policy enforcement, credential scoping, and resource limits.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker, or split it into reviewed milestones for default presets, plugin isolation, IPC/proxy SDK enforcement, manifest policies, credential scoping, resource limits, and migration docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence feature-gap verification path from current source and docs: sandboxing is optional tool isolation, native plugins remain in-process, and config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is maintainer security/product design followed by sequenced milestones across config/schema, plugin loader, SDK IPC/proxying, sandbox backends, manifest policy, credentials, resource limits, and migration tests.
