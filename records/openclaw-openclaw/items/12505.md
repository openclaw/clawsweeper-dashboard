## Summary

Keep open. Current main has opt-in tool sandbox backends and a sandbox-backend SDK seam, but it still lacks the central secure-by-default plugin/process isolation architecture, and recent discussion continues to treat this as the canonical sandbox/isolation tracker.

## What This Changes

The issue requests a unified secure-by-default sandbox architecture with multi-platform isolation, tiered presets, manifest-based policy, plugin/process isolation, credential scoping, and resource limits.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker, or split it into reviewed milestones for default presets, plugin isolation, IPC/proxy SDK enforcement, manifest policy, credential scoping, resource limits, and migration docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction. Source and docs provide high-confidence feature-gap verification: sandboxing is optional tool isolation, native plugins remain in-process, and config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is security/product design followed by sequenced milestones across config/schema, plugin loader, SDK IPC/proxying, sandbox backends, manifest policy, credentials, resource limits, and migration tests.
