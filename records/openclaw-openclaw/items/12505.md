## Summary

Keep open. Current main has opt-in tool sandbox backends, policy checks, and sandbox helper APIs, but it still does not implement the central request for secure-by-default tiered presets, plugin/process isolation, IPC-proxied plugin execution, per-plugin env/fs/net/resource manifest policy, or default credential scoping.

## What This Changes

Request a unified built-in sandbox architecture with multi-platform backends, tiered presets, plugin capability manifests, credential scoping, resource controls, and a migration path toward secure defaults.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox architecture tracker until maintainers split it into staged milestones for default posture, plugin/process isolation, IPC/proxy SDK enforcement, manifest policy, credential scoping, resource controls, and migration compatibility.

## Reproduction Assessment

Not applicable as a feature request, but the current gap is source-verifiable: main still documents optional tool sandboxing, defaults sandbox mode to off, and documents native plugins as in-process and not sandboxed.

## Solution Assessment

No, not as one broad automated change. The best path is staged maintainer design and implementation across config defaults, plugin/runtime boundaries, SDK IPC contracts, security policy, and migration proof.
