## Summary

Keep this open as the canonical owner-reviewed sandbox and plugin-isolation architecture tracker. Current main has mature opt-in tool sandbox backends and a public backend SDK, but native plugins still execute in the Gateway process, sandboxing remains off by default, and there is no tiered preset or per-plugin credential, filesystem, network, and resource capability contract; the recent maintainer discussion and active narrower issues also treat this work as unresolved.

## What This Changes

Add a unified, secure-by-default, multi-platform sandbox architecture with tiered presets, isolated plugin execution, scoped credentials, capability manifests, and resource controls.

## Best Possible Solution

Keep this issue as the canonical architecture and product-decision tracker, define a threat model and compatibility plan, then split implementation into reviewed milestones for plugin process/IPC boundaries, manifest capabilities and credential scoping, platform backends, resource enforcement, tiered defaults, and doctor-backed migration while preserving the existing Docker/SSH/OpenShell backend seam.

## Reproduction Assessment

Not applicable: this is a new architecture and policy request rather than a regression with a failing reproduction. Source inspection does confirm the central gap that native plugins remain host-process code and sandboxing remains optional tool isolation.

## Solution Assessment

No, not as one monolithic implementation. The desired security outcome is aligned, but the proposed preset names, default transition, IPC protocol, capability schema, fallback guarantees, and platform mechanisms require owner-reviewed design and should be delivered as narrow milestones that reuse the existing sandbox backend SDK.

## Risks / Open Questions

- Changing from opt-in sandboxing to a secure-by-default preset can break existing plugins, tools, setup flows, and stored configuration unless maintainers define an explicit staged transition and doctor migration.
- Plugin process isolation, credential scoping, network and filesystem enforcement, resource controls, and three operating-system implementations are distinct security boundaries; combining them in one implementation would make threat review and upgrade proof difficult.
- The proposed platform mechanisms and fallback behavior require a maintained threat model: a child process with filtered environment is not equivalent to filesystem, network, or syscall isolation.
