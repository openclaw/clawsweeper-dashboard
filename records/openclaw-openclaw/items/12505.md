## Summary

Keep open. Current `origin/main` still provides optional tool sandbox backends and a plugin sandbox-backend seam, but it does not implement the requested secure-by-default plugin/process isolation architecture, tiered presets, IPC-proxied plugin execution, per-plugin manifest env/fs/net/resource policy enforcement, or default credential scoping; prior maintainer review and the narrower local-user sandbox issue at https://github.com/openclaw/openclaw/issues/78965 both treat this issue as the canonical sandbox/isolation tracker rather than resolved by the partial Docker work at https://github.com/openclaw/openclaw/pull/29974.

## What This Changes

Request a unified, secure-by-default sandbox architecture with multi-platform plugin/process isolation, tiered presets, manifest-declared policies, IPC proxying, credential scoping, and resource controls.

## Best Possible Solution

Retain this as the canonical sandbox/isolation tracker, or split it into reviewed milestones for default presets, plugin isolation, IPC/proxy SDK enforcement, manifest policies, credential scoping, resource limits, and migration docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence feature-gap verification path from current source and docs: sandboxing is optional tool isolation, native plugins remain in-process, and config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is maintainer security/product design followed by sequenced milestones across config/schema, plugin loader, SDK IPC/proxying, sandbox backends, manifest policy, credentials, resource limits, and migration tests.
