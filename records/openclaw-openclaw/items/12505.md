## Summary

Keep open. Current main has opt-in tool sandbox backends and a plugin SDK backend seam, but the central request remains unresolved: secure-by-default tiered presets, native plugin/process isolation, IPC-proxied plugin execution, and per-plugin env/fs/net/resource policy enforcement are still missing, and the existing maintainer discussion already treats this as canonical unfinished sandbox architecture work.

## What This Changes

The issue requests a secure-by-default, multi-platform sandbox architecture with tiered presets, plugin/process isolation, IPC/proxy SDK enforcement, and manifest-declared env/fs/net/resource policies.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones for default presets, native plugin execution isolation, manifest policy, credential scoping, backend enforcement, and migration compatibility.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence verification path for the feature gap by inspecting current config, schema, docs, plugin manifest, and plugin loader behavior to confirm sandboxing remains optional tool isolation while native plugins still run in-process.

## Solution Assessment

No for an automated one-shot implementation. The requested direction is broad and security-sensitive; the maintainable path is maintainer design review followed by separate milestones with targeted config, plugin-loader, SDK, backend, and migration tests.
