## Summary

Keep open. Current main still has optional tool sandbox backends and a plugin backend seam, but it does not implement the requested secure-by-default plugin/process isolation architecture; recent maintainer and ClawSweeper comments also treat this as unresolved canonical sandbox work.

## What This Changes

Request a secure-by-default, multi-platform sandbox architecture with tiered presets, plugin/process isolation, IPC proxying, and per-plugin manifest policies.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones for defaults, plugin execution isolation, manifest policy, credential scoping, backend enforcement, and migration compatibility.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence verification path for the feature gap: inspect current config, schema, docs, plugin manifest, and plugin loader behavior to confirm sandboxing remains optional tool isolation while native plugins still run in-process.

## Solution Assessment

No for an automated one-shot implementation; the requested direction is security-relevant and broad. The maintainable path is maintainer design review followed by separate milestones with targeted config, plugin-loader, SDK, backend, and migration tests.
