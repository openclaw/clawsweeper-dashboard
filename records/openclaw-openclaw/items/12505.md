## Summary

Keep open. Current main still has optional tool/backend sandboxing and a plugin sandbox-backend seam, but it does not implement the central secure-by-default plugin isolation architecture, tiered presets, manifest policy enforcement, IPC proxying, resource limits, or credential scoping requested here.

## What This Changes

The issue requests a unified, secure-by-default, multi-platform sandbox architecture with tiered presets, plugin/process isolation, manifest-enforced env/fs/net/resource policies, and credential scoping.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones for defaults, native plugin isolation, IPC/proxy SDK enforcement, manifest policy, credentials, resources, and migration compatibility.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence feature-gap verification path from current source and docs: sandboxing is optional tool isolation, native plugins remain in-process, and config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is maintainer design review followed by sequenced milestones across config/schema, plugin loader, SDK proxying, sandbox backends, manifest policy, credentials, resource limits, and migration tests.
