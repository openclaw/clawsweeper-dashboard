## Summary

Keep open. Current main has useful opt-in tool sandbox backends and a sandbox-backend SDK seam, but it still lacks the requested secure-by-default tiered presets, plugin/process isolation, IPC-proxied plugin execution, per-plugin manifest policy, and default credential scoping.

## What This Changes

Request to add a unified, multi-platform sandbox architecture with secure-by-default presets and per-plugin env/fs/net/resource policy enforcement.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker, or split it into reviewed milestones for default presets, plugin isolation, IPC/proxy SDK enforcement, manifest policy, credential scoping, resource limits, and migration docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction. Source and docs give high-confidence feature-gap verification: sandboxing is optional tool isolation, native plugins remain in-process, and config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is security/product design followed by sequenced milestones across config/schema, plugin loader, SDK IPC/proxying, sandbox backends, manifest policy, credentials, resource limits, and migration tests.
