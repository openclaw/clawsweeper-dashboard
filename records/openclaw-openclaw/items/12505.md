## Summary

Current main has useful opt-in tool sandbox backends and a plugin-extensible backend registry, but it does not implement the issue’s central secure-by-default tiered presets, isolated plugin execution, IPC-proxied SDK, or per-plugin credential/filesystem/network/resource policies. The issue remains the broad canonical architecture tracker and requires staged product and security decisions rather than a single automated fix.

## What This Changes

Add a secure-by-default, multi-platform sandbox architecture with tiered presets, isolated plugin execution, capability manifests, credential scoping, network/filesystem controls, and resource limits.

## Best Possible Solution

Keep this as the canonical owner-reviewed architecture tracker, then split it into reviewed milestones for threat model and preset semantics, plugin process/IPC contracts, manifest-declared capabilities and credential scoping, platform backend enforcement, compatibility migrations, and secure-default rollout while retaining the current Docker, SSH, and OpenShell groundwork.

## Solution Assessment

Unclear as a complete implementation design. Secure defaults and stronger isolation are aligned goals, but the proposed all-at-once manager, preset, IPC, manifest, and platform design should be decomposed and validated against existing plugin and sandbox contracts before implementation.

## Risks / Open Questions

- Changing the default from off to a restrictive preset could break existing tools, plugins, provider authentication, networking, and deployment assumptions without a staged compatibility and doctor-migration plan.
- The proposed platform mechanisms have materially different security guarantees and prerequisite models, so a single preset name could imply protections that some backends cannot enforce.
- Plugin process isolation and an IPC-proxied SDK would alter a large external plugin contract; capability semantics, failure handling, performance, and migration behavior need explicit owner review.
- The broad tracker overlaps narrower open security and sandbox items, including https://github.com/openclaw/openclaw/issues/78965 and https://github.com/openclaw/openclaw/issues/58730, so milestone ownership must avoid competing implementations.
