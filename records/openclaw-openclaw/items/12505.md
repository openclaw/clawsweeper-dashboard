## Summary

Keep open: current main and v2026.6.6 provide useful opt-in tool sandbox backends and a plugin SDK backend seam, but native plugins still run in-process and the requested secure-by-default tiered presets, plugin/process isolation, IPC proxying, per-plugin env/fs/net/resource policies, credential scoping, and resource limits are not implemented.

## What This Changes

The issue requests a unified built-in sandbox architecture with multi-platform backends, secure-by-default tiered presets, plugin/process isolation, manifest-scoped policy, credential scoping, and resource controls.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones for preset defaults, plugin/process isolation, IPC/proxy SDK enforcement, manifest policy, credential scoping, resource controls, and migration guidance.

## Reproduction Assessment

Not applicable as a bug reproduction: this issue requests new sandbox architecture and policy surfaces rather than broken behavior under an existing contract; source and docs confirm the capability is absent from current main and v2026.6.6.

## Solution Assessment

No as one monolithic implementation; the security goal fits the project, but maintainers should split it into reviewed milestones with explicit threat model, config migration, plugin SDK, and platform-backend decisions.
