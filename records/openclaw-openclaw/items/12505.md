## Summary

Keep open. Current main still provides opt-in tool sandbox backends and a plugin sandbox-backend seam, but the central request for secure-by-default tiered presets, native plugin/process isolation, IPC-proxied plugin execution, per-plugin env/fs/net/resource manifest policy, and default credential scoping is not implemented; live comments also treat this as the canonical sandbox/isolation architecture tracker.

## What This Changes

Request secure-by-default multi-platform sandbox architecture with tiered presets, plugin/process isolation, IPC proxying, manifest policy, and credential/resource scoping.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker, or split it into reviewed milestones for default presets, plugin isolation, IPC/proxy SDK enforcement, manifest policies, credential scoping, resource limits, and migration docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction. Source and docs provide high-confidence feature-gap verification: sandboxing is optional tool isolation, native plugins remain in-process, and config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is maintainer security/product design followed by sequenced milestones across config/schema, plugin loader, SDK IPC/proxying, sandbox backends, manifest policy, credentials, resource limits, and migration tests.
