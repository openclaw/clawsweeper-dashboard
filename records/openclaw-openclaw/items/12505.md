## Summary

Keep open. Current main has opt-in agent tool sandbox backends and a public sandbox-backend SDK seam, but the central secure-by-default plugin/process isolation architecture remains missing, and the recent issue discussion still treats this as the canonical tracker rather than resolved work.

## What This Changes

The issue requests a unified secure-by-default sandbox architecture with multi-platform isolation, tiered presets, plugin capability manifests, IPC proxying, credential scoping, and resource controls.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker, or split it into reviewed milestones for default presets, plugin isolation, IPC/proxy SDK enforcement, manifest policy, credential scoping, resource limits, and migration docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction. Source and docs provide high-confidence feature-gap verification: sandboxing is optional tool isolation, native plugins remain in-process, and config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is maintainer security/product design followed by sequenced milestones across config/schema, plugin loader, SDK IPC/proxying, sandbox backends, manifest policy, credentials, resource limits, and migration tests.
