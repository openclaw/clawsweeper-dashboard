## Summary

Current main still only provides optional agent tool sandbox backends and SDK/backend seams, while the requested secure-by-default plugin/process isolation architecture remains missing; maintainer discussion already treats this as the canonical security-sensitive sandbox tracker rather than resolved work.

## What This Changes

Request a unified, multi-platform sandbox architecture with tiered presets, plugin/process isolation, IPC-proxied SDK enforcement, per-plugin manifest policies, credential scoping, and resource limits.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones. The target end state should be a staged design and implementation for secure-by-default presets, native plugin or skill execution isolation, IPC/proxy SDK enforcement, manifest-declared env/fs/net/resource policies, credential scoping, resource controls, and migration guidance while preserving existing optional backend work.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence verification path for the feature gap: inspect current config, schema, docs, plugin manifest, and plugin architecture to confirm sandboxing remains optional tool isolation while native plugins still run in-process.

## Solution Assessment

No for an automated one-shot fix; the issue describes the right security direction but not a narrow implementation unit. The maintainable path is maintainer-owned design review followed by separate milestones for defaults, plugin process boundaries, manifest policy, credential scoping, backend enforcement, and migration compatibility.
