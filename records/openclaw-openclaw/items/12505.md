## Summary

Keep open. Current main still implements optional tool/backend sandboxing while native plugins run in-process, with no secure-by-default tiered presets, IPC-proxied plugin execution, credential scoping, or manifest-enforced env/fs/net/resource policies; the existing maintainer discussion also treats this as unresolved canonical architecture work.

## What This Changes

Request a unified secure-by-default sandbox architecture with multi-platform backends, tiered presets, plugin/process isolation, per-plugin manifest policy, resource limits, and credential scoping.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones for default presets, native plugin execution isolation, IPC/proxy SDK enforcement, manifest policy, credential scoping, resource limits, and migration compatibility.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence feature-gap verification path from current source and docs: sandboxing is optional tool isolation, native plugins are still in-process, and the config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is maintainer design review followed by sequenced milestones across config/schema, plugin loader, SDK proxying, sandbox backends, manifest policy, credentials, and migration tests.
