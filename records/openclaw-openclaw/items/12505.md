## Summary

Keep open. Current main still has opt-in tool sandbox backends and a plugin-facing sandbox backend seam, but it does not implement the central request: secure-by-default tiered presets, native plugin/process isolation, IPC-proxied plugin execution, manifest-enforced env/fs/net/resource policy, credential scoping, or resource limits.

## What This Changes

The issue requests a built-in secure-by-default sandbox architecture with multi-platform isolation, tiered presets, per-plugin manifest policy, credential scoping, IPC proxying, and resource limits.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker, or split it into reviewed milestones for presets, plugin/process isolation, IPC proxying, manifest policy, credential scoping, resource controls, and migration docs.

## Reproduction Assessment

Not applicable as a normal bug reproduction. Source and docs give high-confidence feature-gap verification: sandboxing is optional tool isolation, native plugins remain in-process, and config/manifest contracts lack the requested preset and policy surfaces.

## Solution Assessment

No for a one-shot automated fix. The maintainable path is security/product design followed by staged changes across config/schema, plugin loader, SDK IPC/proxying, sandbox backends, manifest policy, credentials, resource limits, and migration tests.
