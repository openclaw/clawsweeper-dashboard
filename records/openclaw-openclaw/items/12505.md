## Summary

Current main provides opt-in tool sandbox backends and a plugin-extensible backend registry, but it still loads native plugins in process, defaults agent sandboxing to off, and has no tiered secure-by-default presets or per-plugin credential, filesystem, network, and resource policies. The explicit reviewer discussion also treats this as the canonical architecture tracker, while related items cover narrower pieces rather than superseding it.

## What This Changes

Request a secure-by-default, cross-platform sandbox architecture with tiered presets, isolated plugin execution, scoped credentials, capability manifests, and resource controls.

## Best Possible Solution

Keep this as the canonical owner-reviewed architecture tracker, then split delivery into separately reviewable milestones for secure defaults, plugin execution isolation, IPC and SDK contracts, credential and capability scoping, platform backends, resource controls, and upgrade migrations while preserving existing sandbox backends.

## Reproduction Assessment

Not applicable: this is a broad security architecture and product-posture request rather than a bounded defect with a current-main failure path. Source inspection does confirm the described architectural gaps around in-process plugins and opt-in sandboxing.

## Solution Assessment

No, not as one unified implementation. The direction is valuable, but the narrowest maintainable solution is a staged architecture with independently reviewed trust boundaries, public contracts, defaults, compatibility behavior, and platform validation.
