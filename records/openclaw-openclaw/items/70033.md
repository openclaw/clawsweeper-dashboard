## Summary

Keep this open. Current main still has the Bedrock-specific gap described in the prior ClawSweeper review: streamed malformed/empty tool-call argument repair is installed only for Kimi/Anthropic, OpenAI completions, and selected Responses transports, while the reported `amazon-bedrock` / `bedrock-converse-stream` path still bypasses it.

## What This Changes

The issue requests reliable handling for large Bedrock-generated `write` and `exec` tool calls that arrive with empty `{}` arguments instead of required fields.

## Best Possible Solution

Add Bedrock-aware malformed or empty required-argument handling in the provider stream or embedded-runner repair layer, producing recoverable arguments when safe and a clear retryable provider/tool-call error when not.

## Reproduction Assessment

Yes for a focused code-level reproduction: simulate `amazon-bedrock` with `bedrock-converse-stream` and a streamed required-argument tool call whose arguments remain `{}` or malformed, then verify current main does not install the repair wrapper. ## Solution Assessment

No, current main is not the best solution because the adjacent repair is scoped away from Bedrock. The narrow maintainable path is Bedrock stream repair/error handling, not changing `write` or `exec` schemas or globally chunking all large content first.
