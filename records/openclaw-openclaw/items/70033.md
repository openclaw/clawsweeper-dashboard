## Summary

Keep open. Current main still routes Amazon Bedrock Converse streams around the malformed tool-call argument repair path, while `write` and `exec` still require structured arguments, so the reported repeated `{}` validation failures remain actionable.

## What This Changes

The issue asks OpenClaw to stop repeated empty-argument `write`/`exec` tool failures for large content on Amazon Bedrock, preferably with retry or clearer failure handling.

## Best Possible Solution

Add focused Bedrock Converse regression coverage and either enable the existing malformed-argument repair for that stream shape or fail closed with clear retry feedback before `write`/`exec` validation loops repeat.

## Reproduction Assessment

Yes, at source level: simulate `amazon-bedrock` with API `bedrock-converse-stream` producing a required-argument `write` or `exec` call whose arguments remain `{}` or malformed. Current main does not install the repair wrapper for that provider/API pair, so the call proceeds to normal validation.

## Solution Assessment

Yes for the narrow bug: Bedrock-specific repair coverage or a clear fail-closed retry signal is the maintainable boundary. Broader large-argument chunking or file-backed argument APIs should stay with the separate feature trackers.
