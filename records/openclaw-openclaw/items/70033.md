## Summary

Keep this open. Current main still has a Bedrock-specific gap: malformed streamed tool-call argument repair is gated to Kimi/Anthropic and `openai-completions`, while the reported provider uses `bedrock-converse-stream` and the Amazon Bedrock plugin wrapper does not add equivalent repair or retry handling.

## What This Changes

The issue asks OpenClaw to handle large Bedrock `write`/`exec` tool calls that arrive with empty `{}` arguments by retrying or surfacing a clear error instead of repeated validation failures.

## Best Possible Solution

Add Bedrock-aware malformed/empty required-argument handling in the provider stream or embedded-runner repair layer, reusing the existing wrapper where safe and producing a retryable, clear provider/tool-call error when arguments cannot be recovered. Cover `write` and `exec` required-parameter cases with focused unit tests.

## Reproduction Assessment

Yes for a focused code-level reproduction: simulate `amazon-bedrock` with `bedrock-converse-stream` and a required-argument tool call whose streamed arguments stay empty or malformed; current main bypasses the repair wrapper and can still pass `{}` to validation. The live 5-7KB Bedrock threshold remains unverified.

## Solution Assessment

No, current main is not the best solution because its adjacent malformed-argument repair is scoped away from Bedrock. The narrower maintainable fix is Bedrock stream repair/error handling, not changing `write`/`exec` schemas or broadly chunking all large content first.
