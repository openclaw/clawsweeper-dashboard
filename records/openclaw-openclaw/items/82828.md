## Summary

Keep open. The PR addresses a real source-level preservation gap and the supplied real-host proof is sufficient for the service-regeneration path, but the diff also makes `OPENCLAW_CLI_CONTAINER_BYPASS` durable even though current main treats it as an internal container-routing sentinel. Maintainer approval is still needed for the final root/container env allowlist.

## What This Changes

The PR exports the service-env preservation helper, allowlists three `OPENCLAW_*` env keys across gateway service regeneration, and adds helper regression tests.

## Best Possible Solution

Land a maintainer-approved narrow allowlist that preserves only env keys intended as durable service configuration, with regression coverage for preserved and still-dropped `OPENCLAW_*` keys.

## Reproduction Assessment

Yes at source level: current main drops every `OPENCLAW_*` key from preserved existing service env, and install plus doctor repair both pass existing service env through that helper. I did not reproduce the full root restart outage because the global root guard was reverted on main.

## Solution Assessment

No as written: preserving `OPENCLAW_ALLOW_ROOT` may be the right narrow fix, but preserving `OPENCLAW_CLI_CONTAINER_BYPASS` makes an internal recursion sentinel durable. The safer path is a maintainer-confirmed allowlist and tests that keep internal sentinels out unless policy explicitly changes.
