## Summary

Keep open. The PR addresses a real helper-level behavior, but the allowlist includes an internal container bypass sentinel and the current root-guard rationale is stale because the global root refusal was reverted on main.

## What This Changes

The PR exports the daemon service-env preservation helper, adds a three-key `OPENCLAW_*` preservation allowlist, and adds focused tests for those keys.

## Best Possible Solution

Confirm the root/container persistence policy, remove the internal bypass sentinel from any durable allowlist, and land only after live Linux service-regeneration proof shows the intended env survives without preserving arbitrary `OPENCLAW_*` keys.

## Reproduction Assessment

Yes for the helper-level behavior: current main drops all `OPENCLAW_*` keys from preserved existing service env. No high-confidence live reproduction of the root restart outage exists against current main because the global root guard has been reverted and the PR proof does not exercise service regeneration.

## Solution Assessment

No in the current form: a narrow allowlist may be reasonable, but it should not preserve `OPENCLAW_CLI_CONTAINER_BYPASS` while current main treats that key as an internal recursion sentinel. Maintainers should confirm the remaining keys and require real regeneration proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not persist the container bypass sentinel:** `src/commands/daemon-install-helpers.ts:350`
  - body: `OPENCLAW_CLI_CONTAINER_BYPASS` is not an operator opt-in on current main; `maybeRunCliInContainer` treats it as an unconditional recursion sentinel. Preserving it into a gateway unit can make later OpenClaw CLI subprocesses launched from that service environment skip `--container` forwarding, so keep this internal bypass out of the durable service-env allowlist.
  - confidence: 0.83
