## Summary

Keep open: current main still intentionally excludes context overflow from model fallback and documents overflow recovery as compaction/retry, so the requested larger-context fallback routing remains a product decision rather than implemented cleanup.

## What This Changes

Request automatic fallback to a configured larger-context model when the primary model exceeds its context window.

## Best Possible Solution

Make the overflow policy explicit: either keep and document compaction-only recovery, or implement bounded post-recovery fallback to larger candidates with headroom checks, replay-safety guards, logging, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows context overflow errors are excluded from `runWithModelFallback`, handled by active-model compaction/truncation, and returned as blocked `context_overflow` when exhausted; I did not run a live 200k-token provider scenario.

## Solution Assessment

Unclear until maintainers decide the policy. If accepted, the narrow maintainable fix is post-compaction fallback only to candidates with sufficient context headroom, not treating every overflow as generic failover.
