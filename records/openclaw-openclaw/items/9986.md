## Summary

Keep open: current main and v2026.6.8 still intentionally keep context overflow inside active-model compaction/retry and blocked-overflow handling, while the direct larger-fallback implementation PR closed unmerged with the underlying request left valid.

## What This Changes

The issue requests using configured model fallbacks when the active model exhausts its context window, selecting a fallback with enough context headroom and logging the event.

## Best Possible Solution

Maintain the current compaction-only behavior only if maintainers explicitly choose and document it; otherwise land a guarded implementation that tries only larger-context fallbacks after compaction and oversized-tool-result recovery are exhausted, with replay-safety, logging, docs, and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main still rethrows escaped context-overflow errors and returns blocked embedded context_overflow results instead of trying a larger configured fallback model.

## Solution Assessment

Unclear until maintainers decide the policy. A guarded post-recovery larger-window fallback is plausible, but unconditional fallback would be risky without context-window filtering and replay-safety gates.
