## Summary

Keep open. Current main still handles context overflow as same-model compaction or tool-result truncation followed by a blocked `context_overflow` result, while the fallback loop explicitly rethrows overflow instead of trying a larger configured fallback model.

## What This Changes

Request automatic fallback to a configured larger-context model when the primary model exceeds its context window.

## Best Possible Solution

Define and document a guarded overflow fallback policy that runs only after existing recovery is exhausted, filters candidates by sufficient context headroom, preserves replay safety, logs the route, and adds regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows a high-confidence path: overflow is detected, same-model recovery is attempted, exhausted recovery returns blocked `context_overflow`, and the outer fallback runner rethrows overflow before rotating candidates.

## Solution Assessment

No. Current main deliberately uses compaction-only overflow handling and does not satisfy the requested larger-context fallback behavior; the safer solution needs maintainer-approved replay and headroom rules.
