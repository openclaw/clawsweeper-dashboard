## Summary

Keep open: current main still intentionally excludes context overflow from model fallback and documents overflow recovery as compaction/retry, so larger-context fallback routing remains an unresolved product decision.

## What This Changes

Request automatic routing to a configured fallback model with enough context headroom when the active model exceeds its context limit.

## Best Possible Solution

Make the overflow policy explicit: either document compaction-only recovery as intentional, or implement guarded post-recovery fallback to larger candidates with headroom checks, replay-safety guards, logging, docs, and regression coverage.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: likely context-overflow errors are excluded from runWithModelFallback, handled by active-model compaction/truncation, and returned as blocked context_overflow when recovery is exhausted. I did not run a live 200k-token provider scenario.

## Solution Assessment

Unclear until maintainers decide the policy. If accepted, the narrow maintainable solution is post-compaction fallback only to candidates with sufficient context headroom, not treating every overflow as generic failover.
