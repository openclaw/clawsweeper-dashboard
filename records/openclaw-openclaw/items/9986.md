## Summary

Keep open: current main and the latest release still treat context overflow as active-model compaction/retry or a blocked overflow result, not as a trigger to select a configured larger-context fallback model. The direct implementation PR was closed unmerged as stale, with its author explicitly saying the underlying request remains valid.

## What This Changes

The issue requests automatic routing to a configured larger-context fallback model when the active model exceeds its context window.

## Best Possible Solution

Keep this issue open until maintainers either document that context overflow remains compaction-only, or land a fresh guarded implementation that runs only after compaction/truncation recovery is exhausted, filters for larger effective context windows, preserves replay safety, logs the event, and adds regression coverage.

## Reproduction Assessment

Yes, source-level. Current main rethrows escaped context-overflow errors and returns blocked embedded context-overflow results instead of trying a larger configured fallback model; the earlier PR also recorded deterministic Testbox before/after proof for the same path.

## Solution Assessment

No, current main is not the best solution for the requested behavior because it intentionally excludes context overflow from fallback. The best path is either an explicit maintainer decision to keep compaction-only semantics, or a fresh guarded fallback implementation with replay-safety and context-window filtering.
