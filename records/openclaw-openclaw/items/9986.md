## Summary

Keep open: current main still treats context overflow as compaction/retry-only and explicitly excludes it from model fallback, so larger-context fallback routing remains unresolved.

## What This Changes

Request automatic routing to a configured fallback model with enough context headroom when the active model exceeds its context window.

## Best Possible Solution

Make the overflow policy explicit: either keep and document compaction-only recovery, or implement guarded post-recovery fallback to candidates with sufficient context headroom, replay-safety checks, logs, docs, and regression coverage.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: likely context-overflow errors are excluded from runWithModelFallback, handled by active-model compaction/truncation, and returned as blocked context_overflow when recovery is exhausted. I did not run a live 200k-token provider scenario.

## Solution Assessment

Unclear until maintainers decide the policy. If accepted, the narrow maintainable solution is post-compaction fallback only to candidates with sufficient context headroom, not treating every overflow as generic failover.
