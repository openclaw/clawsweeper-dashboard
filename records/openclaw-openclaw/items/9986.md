## Summary

Keep open: current main still treats context overflow as a compaction/truncation recovery path on the active model and explicitly keeps it out of model fallback, so the requested smaller-primary to larger-fallback routing remains unresolved and needs maintainer policy judgment.

## What This Changes

The issue requests routing context-length overflow to a configured fallback model with enough context headroom instead of failing or blocking on the primary model.

## Best Possible Solution

Make the overflow policy explicit: either keep and document compaction-only recovery, or implement guarded post-recovery fallback only to candidates with sufficient context headroom, replay-safety guards, logs, docs, and regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main rethrows context-overflow errors out of runWithModelFallback, handles them with active-model compaction/truncation, and returns blocked context_overflow after recovery is exhausted.

## Solution Assessment

Unclear until maintainers decide the policy. If accepted, the narrow maintainable solution is post-compaction fallback only to larger candidates with verified headroom, not treating every overflow as generic failover.
