## Summary

Keep open. Current main still explicitly routes context overflow through active-model compaction, oversized-tool-result truncation, and then a blocked context-overflow result; the model fallback loop still excludes likely context-overflow errors rather than selecting a larger configured fallback model.

## What This Changes

Request automatic model fallback when the primary model exceeds its context window, choosing a configured fallback model with enough context headroom.

## Best Possible Solution

Implement or document a maintainer-approved policy: after active-model compaction and oversized-tool-result truncation are exhausted, allow fallback only to replay-safe candidates whose resolved context window has enough headroom, log the fallback event, and add regression coverage for smaller-primary to larger-fallback routing.

## Reproduction Assessment

Yes. The source-level path is high-confidence: context-overflow errors are detected in the embedded runner, retried through compaction/truncation on the same model, and finally returned as blocked context_overflow while runWithModelFallback rethrows overflow before candidate rotation.

## Solution Assessment

No. The current implementation is intentionally compaction-only for overflow and does not satisfy the requested configured larger-context fallback behavior; a guarded fallback path after recovery exhaustion would be the narrower maintainable fix if maintainers approve the policy.
