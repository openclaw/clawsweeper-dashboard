## Summary

Keep open: current main still intentionally handles context overflow through compaction/retry and a final blocked overflow result, while recent maintainer and ClawSweeper comments kept this enhancement open for policy follow-up.

## What This Changes

Request that context-limit failures try a configured larger-context fallback model instead of ending in compaction-only recovery or an overflow error.

## Best Possible Solution

Make the overflow policy explicit: either document compaction-only recovery as intended, or add a guarded post-recovery fallback path with context-window headroom checks, replay-safety guards, fallback logging, and regression coverage.

## Reproduction Assessment

Yes, source-reproducible from current main: overflow errors are rethrown before model fallback, then handled by active-model compaction/retry and finally returned as blocked context_overflow results. I did not run a live 200k-token provider scenario in this read-only review.

## Solution Assessment

Unclear until maintainers decide the product policy. If accepted, the narrow maintainable solution is post-compaction fallback only to sufficiently large candidates, not treating every overflow as generic failover.
