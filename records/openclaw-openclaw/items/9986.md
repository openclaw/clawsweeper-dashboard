## Summary

Keep open: current main and v2026.6.8 still handle context overflow with active-model compaction/retry or a blocked `context_overflow` result, not by selecting a larger configured fallback model. The direct implementation PR was closed unmerged while explicitly leaving the underlying request valid.

## What This Changes

The issue requests that context-length overflow trigger configured model fallback to a candidate with enough context headroom.

## Best Possible Solution

Either document that context overflow remains compaction-only, or land a fresh guarded implementation that tries only larger-context fallbacks after compaction and oversized-tool-result recovery are exhausted, reusing current context-window resolution and replay-safety gates.

## Reproduction Assessment

Yes, source-level. Current main still rethrows escaped context-overflow errors and returns blocked embedded `context_overflow` results instead of trying a larger configured fallback model.

## Solution Assessment

No, current main is not the best end state for the requested behavior. A maintainable fix needs a fresh post-recovery fallback design or an explicit maintainer decision to keep overflow compaction-only.
