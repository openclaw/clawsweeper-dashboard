## Label Justifications

- `P2`: The missing recovery path can block long-running sessions, but it is a bounded enhancement with an available manual recovery and an open implementation PR.
- `impact:session-state`: The issue concerns conversations becoming blocked when accumulated context exceeds the active model window.
- `impact:auth-provider`: The requested behavior changes provider/model fallback routing based on candidate context capacity.

## Summary

Current main still treats exhausted context overflow as active-model recovery failure rather than a model-fallback trigger, and the open maintainer PR https://github.com/openclaw/openclaw/pull/82754 specifically owns the proposed larger-context fallback behavior; this issue should remain open until that PR is resolved.

## What This Changes

Route exhausted context-overflow runs to a configured fallback model with a sufficiently larger context window.

## Best Possible Solution

Land a guarded design that keeps compaction and oversized-result recovery first, then advances only to a known larger fallback that can fit the observed prompt, preserves side-effect safety, and emits a clear fallback decision; otherwise document overflow as intentionally compaction-only.

## Reproduction Assessment

Yes from source and the linked Testbox evidence: an overflow escaping active-model recovery is rethrown or returned blocked before the configured larger fallback is attempted; this review did not rerun the scenario locally.

## Solution Assessment

The guarded direction in https://github.com/openclaw/openclaw/pull/82754 is plausible, but whether overflow should change models after compaction is exhausted remains a product and replay-safety decision rather than a routine bug fix.
