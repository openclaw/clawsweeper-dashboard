## Summary

Keep this open. Current main still documents and implements context overflow as compaction/retry plus a blocked overflow result, while the model fallback path deliberately excludes context-overflow errors; the related closed items cover adjacent failover or graceful-overflow cases, not larger-window fallback routing.

## What This Changes

The issue requests automatically routing context-length overflow from a smaller primary model to a configured fallback model with a larger context window.

## Best Possible Solution

Maintain the documented compaction-only policy if that is intentional, or implement a guarded fallback path that runs only after overflow recovery is exhausted, filters candidates by sufficient context headroom, logs the transition, and adds regression coverage for smaller-primary to larger-fallback routing.

## Reproduction Assessment

Yes, from source inspection and existing focused tests. Current main has a clear source path where context overflow is compacted/retried on the active model and then returned as context_overflow, while thrown overflow is rethrown before model fallback advances.

## Solution Assessment

No, the requested behavior is not currently implemented. The narrowest maintainable solution would need an explicit product decision and then a guarded post-recovery fallback path rather than treating every overflow as a generic failover error.
