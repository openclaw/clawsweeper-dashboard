## Summary

Keep open. Current main has partly superseded the PR by passing a current-attempt assistant into embedded runner payload construction, but this branch still has a concrete blocked-hook selector defect, is conflicting, and lacks required real behavior proof.

## What This Changes

The PR changes pi embedded runner expect-final payload construction so hook-active runs clear raw assistantTexts and select the final assistant from the post-hook message snapshot.

## Best Possible Solution

Use the existing current-attempt/prePromptMessageCount boundary for hook-aware payloads, return no final payload for blocked current-turn assistant writes, preserve the current main final-payload improvements, and add regression plus redacted runtime proof for rewritten and blocked hook outcomes.

## Reproduction Assessment

Yes for the PR defect by source inspection: with prior session history and a before_message_write hook blocking the current user and assistant writes, the proposed no-user fallback can select an older assistant after run.ts clears assistantTexts. I did not run a live CLI or embedded-runner reproduction in this read-only review.

## Solution Assessment

No. The runner payload surface is the right place, but this helper is not the best fix because current main already has a current-attempt assistant boundary and the PR's no-user fallback can select prior-session content.
