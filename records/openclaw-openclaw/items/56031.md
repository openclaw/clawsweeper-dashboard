## Summary

Keep open. The requested post-hook payload behavior is not on current main, and this PR still has a correctness/security blocker: blocked current-turn writes can make the new selector reuse an older assistant reply, with no real behavior proof attached yet.

## What This Changes

The PR changes pi embedded runner payload construction to ignore raw `assistantTexts` when `before_message_write` hooks are registered, select a final assistant from the message snapshot, and add a focused helper test.

## Best Possible Solution

Use a current-attempt-bounded post-hook assistant for hook-active payloads, return no final payload for fully blocked turns, and keep payloads aligned with persisted transcript semantics plus regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior session history and a `before_message_write` hook blocking both current user and assistant writes, the PR selector falls back to the full snapshot and can choose an older assistant after `run.ts` clears raw `assistantTexts`.

## Solution Assessment

No. The runner payload surface is the right place, but the selector should be bounded by the current attempt using existing `prePromptMessageCount` or `currentAttemptAssistant` semantics instead of the latest persisted user/no-user fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep blocked turns from reusing prior replies:** `src/agents/pi-embedded-runner/run/helpers.ts:84-85`
  - body: When `before_message_write` blocks the current user and assistant writes, there may be no current-turn user boundary in `messagesSnapshot`. The new fallback treats the full transcript as the active turn after `run.ts` clears `assistantTexts`, so `--expect-final` can emit an older assistant reply for the new request; bound selection to the current attempt instead.
