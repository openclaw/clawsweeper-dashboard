## Summary

Keep open. The PR targets a real embedded-runner payload mismatch, but the branch conflicts with current main, lacks real behavior proof, and its selector can reuse a prior assistant when hooks block current writes.

## What This Changes

The PR changes pi embedded runner expect-final payload construction to prefer a post-hook assistant snapshot when `before_message_write` hooks are active and adds a focused helper test.

## Best Possible Solution

Use current main's `currentAttemptAssistant` and `prePromptMessageCount` boundary, return no final payload for blocked current assistant writes, preserve the newer payload improvements, and add regression plus redacted runtime proof.

## Reproduction Assessment

Yes, for the PR defect by source inspection: with prior session history and a `before_message_write` hook that blocks current writes, the no-user fallback can select an older assistant after `run.ts` clears `assistantTexts`. I did not run a live CLI or embedded-runner reproduction because this review is read-only.

## Solution Assessment

No. The runner payload surface is the right place, but this helper should be bounded to the current attempt instead of using a no-user full-snapshot fallback that can select prior-session content.
