## Summary

Keep open. The PR targets a valid hook-aware payload mismatch, but the proposed selector can reuse prior transcript assistant content on blocked-hook turns, current main now has a safer current-attempt boundary, and contributor real behavior proof is still missing.

## What This Changes

The PR changes pi embedded runner expect-final payload construction to derive final assistant payloads from a post-hook message snapshot when before_message_write hooks are registered.

## Best Possible Solution

Use current main's currentAttemptAssistant/prePromptMessageCount boundary, return no final payload for blocked current assistant writes, preserve the newer payload improvements, and add regression plus redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection: with prior session history and a before_message_write hook that blocks current user and assistant writes, the proposed no-user fallback can select an older assistant after run.ts clears assistantTexts. I did not run a live CLI or embedded-runner reproduction in this read-only review.

## Solution Assessment

No. The runner payload surface is the right place, but this helper should be bounded to the current attempt instead of using a no-user full-snapshot fallback that can select prior-session content.
