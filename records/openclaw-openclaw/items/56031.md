## Summary

Keep open: the PR targets a real pi embedded runner result-surface bug, but the proposed selector can still return a prior assistant when hooks block the current turn, and the external PR lacks after-fix real behavior proof.

## What This Changes

The branch clears raw assistant texts for hook-active pi embedded runner payloads, selects a final assistant from the post-hook message snapshot, and adds focused selector unit tests.

## Best Possible Solution

Use the existing current-attempt boundary, such as attempt.currentAttemptAssistant or prePromptMessageCount, for hook-active payloads and cover blocked-current-turn history cases before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior transcript history, if before_message_write blocks the current user and assistant writes, the proposed helper can select a prior assistant from messagesSnapshot.

## Solution Assessment

No. The PR targets the right runner layer, but selecting after the latest persisted user is weaker than the existing current-attempt boundary and can reuse prior replies on blocked turns.
