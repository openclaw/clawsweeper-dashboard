## Summary

Keep open: the PR targets a real embedded runner payload mismatch, but the branch is not merge-ready because its selector can reuse prior transcript assistant content on blocked hook paths, it conflicts with newer current-main payload assembly, and it still lacks after-fix real behavior proof.

## What This Changes

The PR changes pi embedded runner expect-final payload construction to derive final assistant payloads from a post-hook message snapshot when `before_message_write` hooks are registered, with a focused helper test.

## Best Possible Solution

Use the current-main `currentAttemptAssistant` and `prePromptMessageCount` boundary, suppress raw pre-hook assistant text only for hook-finalized result surfaces, preserve the newer payload hardening, and add regression plus redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection: with prior session history and a `before_message_write` hook that blocks current user and assistant writes, the PR helper can select an older assistant after `run.ts` clears raw assistant text. I did not run a live embedded-runner reproduction because this review is read-only.

## Solution Assessment

No. The runner payload surface is the right place, but the selector should be bounded to the current attempt using the existing current-assistant/pre-prompt boundary instead of a no-user full-snapshot fallback.
