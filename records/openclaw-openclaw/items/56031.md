## Summary

Keep open. The PR targets a real hook/result-surface mismatch, but the branch is now conflicting, lacks after-fix real behavior proof, and the proposed selector can fall back to an older transcript assistant on blocked-hook paths.

## What This Changes

This PR changes pi embedded runner expect-final payload construction to ignore raw assistantTexts when before_message_write hooks exist and select a final assistant from the post-hook message snapshot, with a focused helper test.

## Best Possible Solution

Use the current-main currentAttemptAssistant and prePromptMessageCount boundary, suppress raw pre-hook assistant text only for hook-finalized result surfaces, preserve the newer payload hardening, and add regression plus redacted runtime proof.

## Reproduction Assessment

Yes. By source inspection, a session with prior history and a before_message_write hook that blocks both current user and assistant writes can make the helper select an older assistant after run.ts clears raw assistant text.

## Solution Assessment

No. The runner payload surface is the right place, but assistant selection should be bounded to the current attempt using the existing current-assistant/pre-prompt boundary instead of a no-user full-snapshot fallback.
