## Summary

Keep open. Current main still builds embedded runner final payloads from raw pre-hook assistant text, so the PR addresses a real remaining bug, but the proposed selector can still return an older assistant when `before_message_write` blocks the current turn and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR changes pi embedded runner payload construction so `--expect-final` uses a post-hook assistant snapshot when `before_message_write` hooks are registered, and adds a helper plus focused tests for that selection.

## Best Possible Solution

Revise the PR to derive final payloads from the post-hook current-attempt assistant boundary, preferably by reusing `attempt.currentAttemptAssistant` or a selector that takes `prePromptMessageCount`, while clearing stale raw `assistantTexts` only on the hook-active path. Add regression coverage for rewritten assistant, blocked assistant, and blocked current user plus blocked assistant with prior history, then add a single-line changelog entry for the user-facing JSON/final payload fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a session with prior user/assistant history where `before_message_write` blocks both the current user message and current assistant message; the proposed helper can then select the previous assistant from the post-hook snapshot. Current main's original stale-output bug is also visible from the raw `assistantTexts` precedence in payload construction.

## Solution Assessment

No, not yet. The PR is aimed at the right owner module and data source, but selecting by the latest persisted user is weaker than using the runner's existing current-attempt boundary, so it needs a small revision before it is the best maintainable fix.
