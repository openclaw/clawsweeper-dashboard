## Summary

Keep open: the branch targets a real hook/result mismatch, but the proposed selector can mis-associate prior transcript assistant content with the current payload, overlaps newer current-assistant payload assembly, and still lacks after-fix real behavior proof.

## What This Changes

The PR changes pi embedded runner expect-final payload construction to prefer a post-hook assistant snapshot when `before_message_write` hooks are registered and adds a focused helper test.

## Best Possible Solution

Use the current-main `currentAttemptAssistant` and `prePromptMessageCount` boundary, suppress raw pre-hook assistant text only for hook-finalized result surfaces, preserve newer payload hardening, and add regressions plus redacted runtime proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: with prior session history and a `before_message_write` hook that blocks current user and assistant writes, the PR helper can select an older assistant after `run.ts` clears raw assistant text. I did not run live embedded-runner proof because this review is read-only.

## Solution Assessment

No. The runner payload surface is right, but the selector should be bounded to the current attempt using the existing current-assistant/pre-prompt boundary instead of a no-user full-snapshot fallback.
