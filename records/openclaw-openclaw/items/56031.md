## Summary

Keep open. The PR targets a real payload mismatch, but the proposed selector can reuse prior assistant content when hooks block the current turn, and the external PR still lacks after-fix real behavior proof.

## What This Changes

This PR changes pi embedded runner payload construction so hook-active `--expect-final` results use a post-hook assistant selected from the message snapshot, and adds a focused helper test.

## Best Possible Solution

Use a current-attempt-bounded post-hook assistant for hook-active payloads, return no final payload for blocked turns, and keep payloads aligned with persisted transcript semantics plus regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior session history and hooks blocking both current user and assistant writes, the proposed selector has no current-turn boundary and can select an older assistant after raw `assistantTexts` are cleared.

## Solution Assessment

No. The runner layer is the right surface, but selecting after the latest persisted user or the full snapshot is weaker than using the existing current-attempt boundary such as `prePromptMessageCount` or `attempt.currentAttemptAssistant`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep blocked turns from reusing prior replies:** `src/agents/pi-embedded-runner/run/helpers.ts:84-85`
  - body: When `before_message_write` blocks both the current user and assistant writes, the latest persisted user can be from a prior turn or absent. After `run.ts` clears `assistantTexts`, this helper can feed an old assistant into `--expect-final`; bound selection to the current attempt instead.
