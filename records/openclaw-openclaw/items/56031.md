## Summary

Keep open. Current main still has the pi embedded runner result-surface mismatch, but this PR still has a blocking selector bug for blocked current turns and lacks contributor real behavior proof.

## What This Changes

The PR changes pi embedded runner payload construction to clear raw assistant texts when `before_message_write` hooks exist, choose a final assistant from the post-hook message snapshot, and add selector unit tests.

## Best Possible Solution

Use the existing current-attempt boundary, such as `attempt.currentAttemptAssistant` or `prePromptMessageCount`, for hook-active payloads and add coverage for blocked-current-turn history before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior transcript history, if `before_message_write` blocks the current user and assistant writes, the PR helper can select a prior assistant from `messagesSnapshot`.

## Solution Assessment

No. The PR targets the right runner layer, but selecting after the latest persisted user is weaker than the existing current-attempt boundary and can reuse prior replies on blocked turns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep blocked turns from reusing prior replies:** `src/agents/pi-embedded-runner/run/helpers.ts:80-85`
  - body: When `before_message_write` blocks the current user and assistant writes, `messagesSnapshot` can contain only prior history. This helper then anchors on the latest persisted user, or the whole snapshot when no user exists, and can return an old assistant as the new `--expect-final` payload after raw `assistantTexts` have been cleared; use the current-attempt boundary instead.
  - confidence: 0.9
