## Summary

Keep this PR open because it targets a real `--expect-final`/`before_message_write` payload mismatch, but the branch is conflicting, uses the retired `pi-embedded-runner` path, lacks real behavior proof, and introduces a stale-assistant selection bug in blocked-turn cases.

## What This Changes

The PR changes embedded runner payload construction so `--expect-final` responses use a post-hook assistant snapshot when `before_message_write` hooks are active.

## Best Possible Solution

Port the fix to current `src/agents/embedded-agent-runner` by using the existing current-attempt assistant boundary and suppressing raw pre-hook `assistantTexts` whenever `before_message_write` can rewrite or block the final assistant payload.

## Reproduction Assessment

Yes from source for the PR defect: if the current user and assistant writes are blocked by `before_message_write`, the helper can select a previous assistant because it keys off persisted user messages instead of the pre-prompt boundary. I did not run the live CLI scenario in this read-only review.

## Solution Assessment

No. The requested direction is valid, but this PR is no longer the best implementation because current main has a better current-attempt assistant boundary and the branch still relies on a stale latest-user scan in a retired path.
