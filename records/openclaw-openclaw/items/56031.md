## Summary

Keep open. The PR targets a real pi embedded runner result-surface bug, but the proposed selector can still reuse a prior-turn assistant when `before_message_write` blocks the current persisted messages, and the contributor has not provided real after-fix behavior proof.

## What This Changes

The PR changes pi embedded runner `--expect-final` payload construction to skip raw `assistantTexts` when `before_message_write` hooks exist and select a final assistant from the post-hook message snapshot, with a focused helper test.

## Best Possible Solution

Use the existing current-attempt boundary, such as `attempt.currentAttemptAssistant` or `prePromptMessageCount`, for hook-active payloads and add coverage for replaced plus blocked-history snapshots before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence PR-diff path: with prior session history, if `before_message_write` blocks the current user and assistant writes, the helper can select an assistant from a prior turn after `run.ts` has cleared raw `assistantTexts`.

## Solution Assessment

No. The PR targets the right runner layer, but selecting after the latest persisted user message is weaker than the existing current-attempt boundary and can reuse prior replies on blocked turns.
