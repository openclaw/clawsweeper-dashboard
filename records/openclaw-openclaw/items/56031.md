## Summary

Keep open. The reported payload mismatch is still source-backed on current main, but this PR’s selector can reuse a prior assistant message when write hooks block the current turn, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes pi embedded runner payload construction to ignore raw `assistantTexts` when `before_message_write` hooks are registered, select a final assistant from the post-hook snapshot, and add focused helper tests.

## Best Possible Solution

Use a current-attempt-bounded post-hook assistant for hook-active payloads, return no final payload for fully blocked turns, and keep JSON payloads aligned with persisted transcript semantics plus regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior session history and a `before_message_write` hook blocking both current user and assistant writes, the PR selector falls back to the full snapshot and can choose an older assistant after `run.ts` clears raw `assistantTexts`.

## Solution Assessment

No. The runner payload surface is the right place, but the selector should be bounded by the current attempt using existing `prePromptMessageCount` or `currentAttemptAssistant` semantics instead of the latest persisted user/no-user fallback.

## Review Findings
