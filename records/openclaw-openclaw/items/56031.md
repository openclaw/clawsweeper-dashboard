## Summary

Keep open: the PR targets a real current-main payload mismatch, but the proposed selector can still emit a prior assistant for blocked current turns, and the PR needs changelog plus rebase/replacement work before it can land.

## What This Changes

The PR clears raw assistant stream text for `before_message_write` hook runs, derives the final payload assistant from `messagesSnapshot`, and adds focused helper coverage.

## Best Possible Solution

Use the existing current-attempt post-hook boundary for hook-active payloads, return no final payload when hooks block the current assistant, add regression coverage for prior-history blocked turns, and include the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: when `before_message_write` blocks both current user and assistant writes, the PR clears raw assistant text and can then select a prior assistant from `messagesSnapshot`.

## Solution Assessment

No. The PR is in the right runner layer, but the latest-persisted-user selector is weaker than the existing current-attempt boundary and the user-facing fix still needs changelog coverage.
