## Summary

Keep open. The PR targets a real result-surface mismatch, but the proposed selector can reuse an older assistant reply when the current turn is blocked, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes pi embedded runner expect-final payload construction to prefer a post-hook assistant snapshot when before_message_write hooks are present and adds a focused helper test.

## Best Possible Solution

Use the existing current-attempt boundary, such as attempt.currentAttemptAssistant or prePromptMessageCount, for hook-active payloads and add blocked-history plus replacement coverage before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence PR-diff path: with prior session history, if before_message_write blocks the current user and assistant writes, the helper can select an assistant from a prior turn after run.ts clears raw assistantTexts.

## Solution Assessment

No. The PR targets the right runner layer, but selecting after the latest persisted user message is weaker than the existing current-attempt boundary and can reuse prior replies on blocked turns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep blocked turns from reusing prior replies:** `src/agents/pi-embedded-runner/run/helpers.ts:84-85`

This selector then scans that old turn while run.ts has cleared assistantTexts, so --expect-final can emit a prior reply for the new request; use the current-attempt boundary instead.
