## Summary

Keep open. This is a real result-surface bug, but the proposed branch still has a blocking selector bug, lacks after-fix real behavior proof, and current main has not implemented the central fix.

## What This Changes

The PR wires pi embedded runner expect-final payloads to clear raw assistant text and derive the final assistant from a post-hook message snapshot when before_message_write hooks are registered.

## Best Possible Solution

Use a current-attempt-bounded post-hook assistant for hook-active payloads, return no final payload for fully blocked turns, and keep JSON payloads aligned with persisted transcript semantics plus regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior session history and a before_message_write hook blocking both current-turn user and assistant writes, the PR selector can fall back to the full snapshot and choose an older assistant after run.ts clears assistantTexts.

## Solution Assessment

No. The runner payload surface is the right place, but the selector should be bounded to the current attempt using prePromptMessageCount or currentAttemptAssistant semantics instead of the latest persisted user/no-user fallback.

## Review Findings

Overall correctness: patch is incorrect
