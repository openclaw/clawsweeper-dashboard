## Summary

Keep open. Current main still has the stale raw-assistant payload path this PR is trying to fix, but the proposed selector can reuse a prior assistant when write hooks block the current user and assistant messages, so the patch needs a small correctness repair before merge.

## What This Changes

The PR changes pi embedded runner payload construction so hook-active `--expect-final` results use a post-hook assistant snapshot, and adds a selector helper with focused tests.

## Best Possible Solution

Use the existing current-attempt assistant boundary for hook-active payloads, clear stale raw `assistantTexts` only on that path, add blocked-current-user/prior-history regression coverage, and add the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior transcript history, if `before_message_write` blocks the current user write and current assistant write, the PR helper can select a prior assistant from `messagesSnapshot`.

## Solution Assessment

No. The PR targets the right runner layer and data source, but selecting after the latest persisted user is weaker than using the runner's existing current-attempt boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep blocked turns from reusing prior replies:** `src/agents/pi-embedded-runner/run/helpers.ts:80-85`

This helper then falls back to the whole snapshot and can return an old assistant, so `--expect-final` reports a previous reply for the new request; use the current-attempt boundary instead.
