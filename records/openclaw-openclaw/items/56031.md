## Summary

Keep open. The PR targets a real pi embedded runner result-surface bug and current main does not already implement the hook-aware payload path, but the proposed selector can reuse prior-turn assistant content on blocked hook paths and the contributor has not provided real behavior proof.

## What This Changes

The PR changes pi embedded runner `--expect-final` payload construction to use a post-hook assistant snapshot when `before_message_write` hooks are registered.

## Best Possible Solution

Use the existing current-attempt boundary, such as `currentAttemptAssistant` or `prePromptMessageCount`, for hook-active payloads and add coverage for replaced plus blocked-history snapshots before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence PR-diff path: with prior session history, if `before_message_write` blocks the current user and assistant writes, the helper can scan prior messages and select an old assistant.

## Solution Assessment

No. The PR targets the right runner layer, but selecting after the latest persisted user is weaker than the existing current-attempt boundary and can reuse prior replies on blocked turns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep blocked turns from reusing prior replies:** `src/agents/pi-embedded-runner/run/helpers.ts:84-85`
  - body: When `before_message_write` blocks the current user and assistant writes, `messagesSnapshot` can contain only prior history. This fallback scans that prior turn, and `run.ts` has already cleared `assistantTexts`, so `--expect-final` can emit an old reply for the new request; use the current-attempt boundary instead.
  - confidence: 0.9

## Security Review
