## Summary

Keep open. The PR targets a real post-hook payload mismatch, but the proposed selector can reuse a prior assistant when hooks block the current turn, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes pi embedded runner payload construction to ignore raw assistant texts when `before_message_write` hooks exist, select an assistant from the post-hook message snapshot, and add selector tests.

## Best Possible Solution

Use the existing current-attempt boundary, such as `currentAttemptAssistant` or `prePromptMessageCount`, for hook-active payloads and add coverage for replaced plus blocked-history snapshots before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence PR-diff path: with prior transcript history, if `before_message_write` blocks current user and assistant writes, the helper can scan only prior messages and select an old assistant.

## Solution Assessment

No. The PR targets the right runner layer, but selecting after the latest persisted user is weaker than the existing current-attempt boundary and can reuse prior replies on blocked turns.
