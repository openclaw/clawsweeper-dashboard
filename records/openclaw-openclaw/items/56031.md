## Summary

Keep open. The PR targets a real post-hook payload mismatch, but the proposed selector can reuse a prior assistant reply when hooks block the current turn, and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR changes pi embedded runner payload construction to skip raw assistant texts when `before_message_write` hooks exist, select a final assistant from the post-hook snapshot, and add a focused helper test.

## Best Possible Solution

Use a current-attempt-bounded post-hook assistant for hook-active payloads, preserve blocked turns as no final payload, and keep JSON payloads aligned with persisted transcript semantics plus regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior session history and a `before_message_write` hook blocking both current user and assistant writes, the helper has no current-turn user boundary and can select a prior assistant after `run.ts` clears `assistantTexts`.

## Solution Assessment

No. The runner layer is the right surface, but selecting after the latest persisted user or the full snapshot is weaker than using the existing current-attempt boundary such as `attempt.currentAttemptAssistant` or `prePromptMessageCount`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep blocked turns from reusing prior replies:** `src/agents/pi-embedded-runner/run/helpers.ts:84-85`
