## Summary

Keep open: the PR targets a real hook-result mismatch, but the branch is now conflicting against the renamed runner and the added helper can still return stale prior assistant content when hooks block the current turn; real after-fix proof is also missing.

## What This Changes

The PR changes the old Pi embedded-runner payload path to use a post-hook assistant snapshot for `--expect-final` payloads when `before_message_write` hooks are active, with a helper and focused tests.

## Best Possible Solution

Port the fix to the current embedded-runner payload boundary, make hook-mutated and hook-blocked final payloads use a current-attempt post-hook source instead of raw `assistantTexts`, and require real `agent --json` proof with a minimal `before_message_write` hook.

## Reproduction Assessment

Yes at source level, but not as a live run: the added helper's no-user fallback can select an older assistant when hook-blocked messages leave no current user/assistant in the snapshot. The PR does not include real `agent --json` proof with a `before_message_write` hook.

## Solution Assessment

No. The direction is plausible, but the best fix should be ported to the current embedded-runner path and use the current-attempt boundary already present on main, while also suppressing raw `assistantTexts` when hooks block or rewrite final output.
