## Summary

Keep open. Current main still does not implement the requested pi-runner post-hook payload behavior, and the PR branch still has a blocking selector-boundary bug plus missing real behavior proof.

## What This Changes

The PR changes pi embedded runner payload construction so hook-active `--expect-final` results use a post-hook assistant selected from the message snapshot, and adds a focused selector test.

## Best Possible Solution

Use a current-attempt-bounded post-hook assistant for hook-active payloads, return no final payload for blocked turns, and keep payloads aligned with persisted transcript semantics plus regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior session history and hooks blocking both current user and assistant writes, the PR selector has no current-turn boundary and can select an older assistant after raw `assistantTexts` are cleared.

## Solution Assessment

No. The runner layer is the right surface, but selecting after the latest persisted user or from the full snapshot is weaker than using the existing current-attempt boundary such as `prePromptMessageCount` or `attempt.currentAttemptAssistant`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:
