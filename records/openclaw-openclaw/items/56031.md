## Summary

Keep open: current main still has the raw-payload mismatch, and this PR targets the right layer, but the proposed selector can return a prior assistant when hooks block the current turn, so it needs a focused repair before merge.

## What This Changes

The PR changes pi embedded runner payload construction to clear raw assistant text when `before_message_write` hooks are registered, select a final assistant from the post-hook snapshot, and add a selector unit test.

## Best Possible Solution

Use the existing current-attempt assistant boundary for hook-active payloads, clear stale raw `assistantTexts` only on that path, cover blocked-current-user/prior-history regression cases, and add the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior transcript history, if `before_message_write` blocks both the current user write and current assistant write, the PR helper can select a prior assistant from `messagesSnapshot`.

## Solution Assessment

No. The PR targets the right runner layer, but selecting by the latest persisted user is weaker than using `prePromptMessageCount` or `attempt.currentAttemptAssistant`, and the user-facing fix still needs a changelog entry.
