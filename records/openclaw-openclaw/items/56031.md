## Summary

Keep open. The target stale-payload bug is valid and current main still has the raw assistant-text result surface, but this PR has a blocking selector-boundary regression and still lacks after-fix real behavior proof.

## What This Changes

The PR changes pi embedded runner payload construction to skip raw assistant texts when `before_message_write` hooks exist, select the final assistant from the post-hook message snapshot, and add a focused helper test.

## Best Possible Solution

Use a current-attempt-bounded post-hook assistant for hook-active payloads, preserve blocked turns as no final payload, and keep JSON payloads aligned with persisted transcript semantics plus regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with prior session history and hooks blocking both current user and assistant writes, the PR helper has no current-turn boundary and can select an old assistant after `run.ts` clears `assistantTexts`.

## Solution Assessment

No. The runner layer is the right surface, but selecting after the latest persisted user or the full snapshot is weaker than using the existing current-attempt boundary such as `prePromptMessageCount` or `attempt.currentAttemptAssistant`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep blocked turns from reusing prior replies:** `src/agents/pi-embedded-runner/run/helpers.ts:84-85`
  - body: When `before_message_write` blocks both current user and assistant writes, the latest persisted user can be from a prior turn or absent. After `run.ts` clears `assistantTexts`, this helper can feed an old assistant into `--expect-final`; bound selection to the current attempt instead.
  - confidence: 0.9
