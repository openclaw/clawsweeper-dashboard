## Summary

Keep this PR open. Current main still builds pi embedded runner final payloads from raw attempt assistant text before falling back to the post-hook transcript message, so the reported `before_message_write` mismatch remains real; the proposed patch is not ready because its helper can select an older assistant when the current turn is fully blocked by write hooks.

## What This Changes

The PR changes pi embedded runner payload construction so `--expect-final` uses a post-hook assistant snapshot when `before_message_write` hooks are registered, adds a helper to select the final assistant from a message snapshot, and adds focused helper tests.

## Best Possible Solution

Revise the PR so hook-aware `--expect-final` payloads use a post-hook current-turn assistant boundary, preferably the existing pre-prompt message count/current-attempt signal or an equivalent marker, and return no final assistant when the latest turn is fully blocked instead of searching prior transcript history.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to run an embedded agent turn with a `before_message_write` hook that rewrites or blocks the final assistant, then compare the persisted post-hook transcript with the `--expect-final` or `agent --json` payload; the PR’s remaining edge case is reproduced by adding prior session history and blocking both the current user and assistant writes.

## Solution Assessment

No, not yet. The patch is the right narrow area to fix, but the helper needs a reliable current-turn boundary so hook-blocked turns cannot emit older assistant content.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Prevent fallback to prior assistant after a fully blocked turn:** `src/agents/pi-embedded-runner/run/helpers.ts:84-88`
  - body: When `before_message_write` blocks both the current user message and the current assistant message, the post-hook snapshot may contain only prior session history. This fallback searches the whole snapshot when no user message is found, so `run.ts` can report an older assistant reply as the `--expect-final` result for the new turn. Please treat this as no current final assistant, or pass a reliable current-turn boundary into the selector.
  - confidence: 0.91
