## Summary

Keep open. Current main partially supersedes the thinking-label side of this PR through session-row defaults, but it still does not carry an effective reasoning default to the Web UI, and the PR branch cannot merge as-is because its proposed `chat.history` reasoning/default-cache wiring can diverge from the runtime resolver and fresher session rows.

## What This Changes

The PR extends `chat.history` with effective thinking/reasoning values, stores those values in Control UI chat state, uses them for chat rendering/default labels, and adds gateway/UI regression tests.

## Best Possible Solution

Keep the merged session-row thinking-default path, then add a backward-compatible authoritative reasoning-default contract that matches runtime resolution exactly. The UI should render defaults from fresh session/default data where possible, avoid stale history-derived cache precedence after model/session changes, and include focused Gateway and Control UI tests for per-agent, global, and session override precedence.

## Reproduction Assessment

Yes. A fresh Web UI chat session with `agents.defaults.reasoningDefault` or `agents.list[].reasoningDefault` but no persisted session override can be traced through current main: runtime resolves the configured default, while `chat.history` and `renderChat` do not expose/use that effective value for reasoning visibility.

## Solution Assessment

No. The PR is aimed at the right bug, but the proposed solution is not the best current path because it introduces history-derived UI caches that can go stale and its reasoning resolver omits the merged global default contract; the safer fix is to share or mirror the runtime/default resolver through the session data boundary used by the UI.
