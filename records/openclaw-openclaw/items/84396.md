## Label Justifications

- `P1`: The PR addresses prompt-visible provider credential exposure and currently has a high-impact replace-mode secret persistence defect.
- `merge-risk: 🚨 compatibility`: Changing generated `models.json` credential persistence can alter upgrade and replace-mode behavior for existing provider configs.
- `merge-risk: 🚨 auth-provider`: The `apiKey` field is part of provider validation and runtime auth resolution, so marker replacement and preservation directly affect model authentication.
- `merge-risk: 🚨 security-boundary`: The diff changes how resolved secrets are kept out of prompt-visible agent config, and the current mode bug can leave plaintext credentials serialized.

## Summary

Keep this PR open, but it still needs a targeted fix: the latest revision fixes the earlier merge-mode credential clobber, yet it now uses existing plaintext keys as preservation evidence even when `models.mode` is `replace`, which can leave a resolved secret in prompt-visible `models.json`.

## What This Changes

The PR changes models.json planning to replace newly resolved plaintext provider `apiKey` values with `secretref-managed`, preserve existing user-owned merge-mode apiKeys, and add regression tests.

## Best Possible Solution

Land one canonical planner-level Layer 1 fix that writes non-secret markers for generated plaintext keys, preserves existing user-owned auth only in merge mode, and proves fresh, merge, and replace behavior.

## Reproduction Assessment

Yes, source-reproducible. In PR head, build a `models.mode: "replace"` plan where `existingParsed.providers.custom.apiKey` equals the newly resolved `custom.apiKey`; the unconditional preservation map lets the sanitizer keep that plaintext instead of writing the marker.

## Solution Assessment

No, not quite. Marker substitution with exact existing-value preservation is the right shape, but existing plaintext preservation must be scoped to merge mode so replace mode remains a true rewrite and does not leak resolved secrets.
