## Summary

Keep open: the PR targets a real secret-exposure problem, but the patch is not merge-ready because it can make custom provider entries in generated `models.json` invalid or unavailable by deleting the `apiKey` field instead of replacing plaintext with a non-secret marker. It also lacks after-fix real behavior proof for this external security-sensitive runtime path.

## What This Changes

The PR adds a models config planning sanitizer that omits non-marker provider `apiKey` strings from generated `models.json` while preserving known env markers, with focused regression coverage.

## Best Possible Solution

Keep plaintext secrets out of `models.json`, but persist a pi-compatible non-secret auth marker for resolved credentials and prove both fresh generation and upgrade from existing custom provider entries still load and authenticate.

## Reproduction Assessment

Yes. Source inspection shows the PR can generate a non-built-in custom provider with models but no `apiKey`, and pi-coding-agent 0.75.1 rejects that provider shape; I did not run a runtime write test because this review is read-only.

## Solution Assessment

No. Deleting the field is too broad; the maintainable fix should replace resolved plaintext with a non-secret marker or other runtime-only auth contract that keeps custom provider validation and upgrades working.
