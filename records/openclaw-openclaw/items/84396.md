## Summary

Keep open. The supplied proof covers the new sanitizer output and pi-coding-agent does require an `apiKey` field for non-built-in custom providers, but the patch still rewrites existing `models.json`-owned plaintext auth to `secretref-managed`, which current OpenClaw auth code treats as not usable.

## What This Changes

The PR changes models.json planning so non-marker provider `apiKey` strings are serialized as `secretref-managed` and adds regression tests for fresh and merge planning paths.

## Best Possible Solution

Keep generated resolved secrets out of `models.json` while preserving existing user-owned `models.json` auth unless a tested migration moves those credentials to auth profiles or SecretRef storage.

## Reproduction Assessment

Yes. Source inspection shows the PR sanitizer runs after `mergeWithExistingProviderSecrets`, so an existing plaintext `models.json` key can be preserved and then replaced with `secretref-managed`; current model-auth source and tests treat that marker as non-usable auth.

## Solution Assessment

No. Marker substitution is a good shape for OpenClaw-resolved/generated secrets, but applying it indiscriminately after merge is not the safest fix because it removes existing user-owned credentials without migration.
