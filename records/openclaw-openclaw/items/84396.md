## Summary

Keep this PR open: it addresses a real secret-leak path, but the current patch can break existing custom-provider auth by rewriting user-owned `models.json` API keys, and its real-behavior proof is stale against the current head.

## What This Changes

This PR adds a `models.json` planning sanitizer that replaces non-marker provider `apiKey` values with `secretref-managed` and adds regression tests for fresh and merge planning.

## Best Possible Solution

Sanitize only OpenClaw-resolved/generated secrets while preserving or safely migrating user-authored `models.json` credentials, then prove current-head fresh and upgrade behavior.

## Reproduction Assessment

Yes. Source inspection shows current `main` serializes merged provider configs without a final plaintext-key scrub, and the PR's own focused test/probe demonstrates that a planned provider can carry a plaintext `apiKey`; I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is right for OpenClaw-resolved secrets, but unconditional replacement after merge is not the safest fix because it can remove the only supported auth source for existing `models.json` custom providers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve existing models.json auth keys:** `src/agents/models-config.plan.ts:115`
  - body: This sanitizer runs after `mergeWithExistingProviderSecrets`, so merge mode has already copied providers that exist only in the user's agent `models.json`. For those custom providers, replacing a literal `apiKey` with `secretref-managed` removes the only credential source; `resolveUsableCustomProviderApiKey` does not treat that marker as usable for non-local providers, so the next run can fail with missing auth. Restrict this rewrite to keys known to be generated from OpenClaw-managed config/SecretRefs, or add a tested migration that moves existing literals to a runtime auth store before replacing them.
  - confidence: 0.9
