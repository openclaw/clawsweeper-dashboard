## Summary

Keep open: the PR targets a real plaintext-key serialization problem, but the current sanitizer is too broad because it runs after merge and can replace existing `models.json` credentials that current main treats as usable runtime auth.

## What This Changes

The PR adds a planner-time sanitizer that converts non-marker provider `apiKey` strings in generated `models.json` to `secretref-managed` and adds fresh/merge planning tests.

## Best Possible Solution

Keep generated resolved secrets out of `models.json` while preserving existing user-owned `models.json` auth unless a tested migration moves those credentials to auth profiles or SecretRef storage.

## Reproduction Assessment

Yes. Source inspection shows current main can serialize merged plaintext provider keys, and the PR’s own current-head terminal probe demonstrates the new marker output; the auth-regression path is also source-reproducible from the added merge test and current `model-auth` tests.

## Solution Assessment

No. The direction is right for generated resolved secrets, but replacing every non-marker key after merge is not the narrowest maintainable fix because it also removes existing user-owned `models.json` credentials.
