## Label Justifications

- `P1`: The PR targets an active provider-secret leak and still has a merge-blocking auth/security edge case affecting real upgrades.
- `merge-risk: 🚨 compatibility`: Changing apiKey serialization can break custom providers whose only credential lives in an existing models.json.

- `merge-risk: 🚨 security-boundary`: The remaining bypass can leave plaintext provider credentials in a file that the security roadmap treats as prompt-facing.

## Summary

Keep open: the latest revision is a real improvement and preserves models.json-only custom-provider credentials, but its provenance check still trusts an old matching plaintext value outside the merge path, so a generated secret can remain in the prompt-facing models.json after this PR.

## What This Changes

The branch adds a models.json apiKey sanitizer that writes `secretref-managed` for newly resolved plaintext provider keys, preserves matching existing non-marker keys, and adds planner regression coverage.

## Best Possible Solution

Keep the sanitizer, but tie plaintext preservation to actual merge-mode user-owned credentials and scrub replace-mode or runtime-resolved secrets to markers, with upgrade coverage for both cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `models.mode: "replace"`, an old `existingParsed.providers.<id>.apiKey` matching the current resolved plaintext is collected at the PR call site and then exempted from marker replacement.

## Solution Assessment

No. Marker substitution is the right write-boundary direction, but the preservation rule should be tied to actual merge-mode compatibility provenance rather than any matching old plaintext value.
