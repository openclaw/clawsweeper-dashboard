## Label Justifications

- `P1`: The PR targets provider credential leakage into generated `models.json`, affecting auth and security-sensitive agent context.
- `merge-risk: 🚨 compatibility`: Changing which plaintext `models.json` keys are preserved can break custom-provider setups whose only credential lives in that file.
- `merge-risk: 🚨 auth-provider`: The patch changes persisted provider `apiKey` values from runtime plaintext to marker values, which can affect credential resolution paths.
- `merge-risk: 🚨 security-boundary`: Generated `models.json` can be agent-readable prompt context, so preserving or stripping plaintext keys is a credential-boundary decision.

## Summary

Keep open. Current main still serializes provider data directly into `models.json`, and the latest patch now appears to cover the prior provenance hole, but merge needs maintainer policy review because it intentionally preserves some plaintext credentials for compatibility and overlaps another open sanitizer PR.

## What This Changes

The branch adds a `models.json` serialization sanitizer for provider `apiKey` values plus regression coverage for resolved-key scrubbing, source/plaintext preservation, and merge/replace mode behavior.

## Best Possible Solution

Land one canonical `models.json` sanitizer with an explicit policy for existing plaintext keys, then close or supersede competing PRs while keeping the broader secrets roadmap issue open for remaining isolation work.

## Reproduction Assessment

Yes. Current main serializes `finalProviders` directly into `models.json`, and the PR body includes an after-fix terminal probe showing the changed serialization behavior for the key scenarios.

## Solution Assessment

Yes, with maintainer policy review. A serialization-boundary sanitizer is the narrow repair for resolved provider keys, but preserving existing/source plaintext versus stricter scrubbing needs an explicit compatibility decision.
