## Label Justifications

- `merge-risk: 🚨 compatibility`: The sanitizer can change existing plaintext credential behavior for supported custom-provider configurations.

## Summary

Keep open: the latest revision fixes the earlier deletion/preservation problem, but the sanitizer still replaces credentials without proving a separate runtime auth source and can also preserve prior generated leaks as if they were user-owned.

## What This Changes

The PR adds a models.json write sanitizer that replaces non-marker provider apiKey strings with secretref-managed, preserves matching existing keys, and adds planner regression tests.

## Best Possible Solution

Keep generated resolved secrets out of models.json only when runtime auth can still supply the credential, and add upgrade coverage that separates existing user-owned models.json keys from previously generated leaks.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a custom provider with only a plaintext models.providers.*.apiKey reaches the PR sanitizer, gets secretref-managed, and pi-coding-agent resolves that marker as a literal unless AuthStorage has another credential.

## Solution Assessment

No. Marker substitution is a good direction for generated secrets, but the current implementation needs stronger provenance or migration so it does not break config-owned auth or preserve previously generated leaked keys.
