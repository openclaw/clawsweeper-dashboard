## Summary

Keep open: current main and the latest release still contain the source-level path that can send a targetless `message_tool_only` plugin-channel reply to the internal source sink, and the open candidate fix at https://github.com/openclaw/openclaw/pull/94952 has not merged.

## What This Changes

The issue asks for targetless `message(action=send)` in `message_tool_only` mode to resolve the current external plugin-channel conversation instead of being captured by the internal source-reply sink.

## Best Possible Solution

Review and land a narrow fix through https://github.com/openclaw/openclaw/pull/94952 or a replacement that preserves `message_tool_only` privacy while allowing one targetless current-source send from a send-capable external plugin channel to reach that same conversation.

## Reproduction Assessment

Yes at source level: current main checks the internal-source predicate before channel selection and send execution, and action-only external plugins are not accepted by the activated outbound gate used by that predicate. I did not run a fresh live Lansenger group-chat repro in this read-only pass.

## Solution Assessment

Unclear as currently proposed: the active PR is in the right area, but prior review found its capability check too broad and its proof incomplete. The safer solution is to use an activated send-surface or explicit send-action contract, cover the full runner path, and add redacted real external-channel proof.
