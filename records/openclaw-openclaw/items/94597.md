## Summary

Keep open: current main still has a source-backed gap where targetless `message_tool_only` sends can be intercepted by the internal source-reply sink before an external/action-only plugin channel gets a chance to handle the send, and no merged canonical fix now owns this exact behavior.

## What This Changes

The issue asks OpenClaw to deliver targetless `message(action="send")` calls back to the current external plugin group conversation under `messages.groupChat.visibleReplies: "message_tool"`.

## Best Possible Solution

Preserve `message_tool_only` privacy while adding a maintainer-approved plugin-safe path so a send-capable external group/channel plugin can deliver one targetless current-source send to the same conversation, with regression coverage for action-only and outbound plugin surfaces.

## Reproduction Assessment

Yes at source level, but not live: current main checks targetless `message_tool_only` sends for the internal sink before normal plugin routing, and action-only plugins are not considered activated outbound surfaces by that escape. I did not run the Lansenger plugin transport flow.

No for the exact proposed reroute as an automatic fix: the source evidence supports the bug, but the best repair needs a maintainer decision on the internal-source sink versus plugin-send boundary before code changes.
