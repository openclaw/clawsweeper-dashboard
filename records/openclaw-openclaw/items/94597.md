## Label Justifications

- `P1`: The issue reports a current release channel workflow where tool-only group replies from an external plugin channel are not delivered at all.
- `impact:message-loss`: The affected behavior silently drops the visible reply that the agent attempted to send with the shared message tool.

## Summary

Keep open: the reported release/current-main behavior is not implemented safely for external plugin group-channel source replies, and current source shows targetless `message_tool_only` sends can enter an internal-source path that is not a normal plugin send while room-event delivery suppression can still prevent the mirror payload from reaching the channel.

## What This Changes

The issue requests that targetless `message(action="send")` in `message_tool_only` mode deliver to the active external plugin channel conversation instead of disappearing through the internal source-reply sink.

## Best Possible Solution

Preserve the privacy and duplicate-send guarantees of `message_tool_only`, but add a core/plugin-safe delivery path so one targetless current-source `message(action="send")` from an external plugin group/channel produces one visible reply to that source conversation.

## Reproduction Assessment

Yes at source level: targetless `message_tool_only` sends enter the internal source-reply path, and room-event channel dispatch suppresses marked mirror payloads unless the turn is an explicit command. I did not run the external Lansenger plugin live flow in this read-only review.

## Solution Assessment

No for the exact proposed rerouting as written. The best fix should first decide the owner boundary between the intentional internal-source sink and send-capable plugin delivery, then add focused coverage for external plugin group/channel `message_tool_only` text sends without making ordinary final text public.
