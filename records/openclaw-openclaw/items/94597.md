## Summary

Keep open: current main still has a source-level path where a targetless `message(action=send)` in `message_tool_only` mode can be captured by the internal source-reply gate or fail target validation before an external plugin send path runs. The related merged fixes cover stale webchat context and Telegram usage decoration, but not this external plugin current-channel routing case.

## What This Changes

The issue asks OpenClaw core to let targetless `message(action=send)` resolve the current external plugin channel conversation under `message_tool_only` instead of diverting or failing before plugin delivery.

## Best Possible Solution

Teach the shared message-tool routing path to carry or infer a routable current external plugin target before the internal sink decision, and only fall back to the private source-reply sink when no plugin-capable current-source delivery route exists.

## Reproduction Assessment

Partly: source inspection gives a clear current-main path where targetless sends can be intercepted before plugin delivery or fail target validation when the current plugin target is not carried into `toolContext`. I did not run a live external plugin group-chat reproduction.

## Solution Assessment

Yes, directionally: fixing the shared current-source routing boundary is narrower and more maintainable than requiring every plugin user to pass explicit `to` or switching rooms back to automatic replies. The fix should not add a config option or plugin-specific special case.
