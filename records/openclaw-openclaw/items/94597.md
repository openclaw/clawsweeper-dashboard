- `impact:message-loss`: The affected behavior can silently drop the visible channel reply that the agent attempted to send with the shared message tool.

## Summary

Keep open: current main and v2026.6.8 still have the targetless `message_tool_only` internal-source routing/suppression behavior that can prevent external plugin group-channel sends from becoming visible channel replies.

## What This Changes

The issue requests that targetless `message(action="send")` calls under `message_tool_only` resolve the current external plugin conversation target instead of being swallowed by the internal source-reply sink.

## Best Possible Solution

Preserve `message_tool_only` privacy while adding a plugin-safe path so one targetless current-source send from a send-capable external group/channel plugin produces one visible reply to that same source conversation.

## Reproduction Assessment

Yes at source level: targetless `message_tool_only` sends can enter the internal-source sink before plugin routing, and room-event suppression can keep marked mirror payloads out of channel delivery. I did not run the external Lansenger plugin live flow in this read-only review.

## Solution Assessment

No for the exact proposed rerouting as a ready-to-automate fix. The durable solution should first choose the owner boundary between the internal-source sink and send-capable plugin delivery, then add focused coverage for external plugin group/channel text sends without making ordinary final text public.
