## Summary

Keep open: this PR targets a real memory_search liveness bug, but it patches retired legacy files while current main executes memory_search from extensions/memory-core, so it cannot fix active sessions as-is. It also still needs real after-fix behavior proof before merge.

## What This Changes

The PR adds a 30-second Promise.race timeout and timeout-specific unavailable payload handling to the legacy src/agents/tools memory_search tool, plus a fake-timer unit test.

## Best Possible Solution

Port or replace the PR with a memory-core tool-boundary deadline that covers lazy runtime load, manager acquisition, active search, and timeout-specific unavailable metadata, with focused regression tests.

## Reproduction Assessment

Yes, for source-level reproduction: current main can stub getMemorySearchManager or memory.manager.search to return a never-settling promise, and extensions/memory-core/src/tools.ts awaits those paths without a deadline. I did not run a live Telegram/iCloud setup.

## Solution Assessment

No. A hard fail-open timeout is the right direction, but this PR applies it to retired src/agents/tools memory-tool files instead of the active extensions/memory-core implementation.
