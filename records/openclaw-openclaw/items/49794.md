## Summary

Keep open and request changes: the liveness bug is valid, but this branch updates the retired core memory tool while current main dispatches `memory_search` from `extensions/memory-core`, so it would not protect the active path after rebase.

## What This Changes

The PR wraps the legacy `src/agents/tools/memory-tool.ts` `memory_search` flow in a 30-second timeout and adds a fake-timer timeout test.

## Best Possible Solution

Port the fail-open deadline to `extensions/memory-core` around runtime loading, manager acquisition, and active search, return timeout-specific unavailable metadata, preserve corpus behavior, and add focused regression coverage plus maintainer-owned changelog handling.

## Reproduction Assessment

Yes for source-level reproduction: stub `getMemorySearchManager` or `memory.manager.search` to return a never-settling promise, and current `extensions/memory-core/src/tools.ts` awaits it without a deadline. I did not run a live Telegram/iCloud setup in this read-only review.

## Solution Assessment

No. The hard fail-open timeout is the right direction, but this PR applies it to a retired file instead of the active memory-core tool path.
