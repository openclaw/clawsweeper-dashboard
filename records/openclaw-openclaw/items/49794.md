## Summary

Keep open: the liveness bug is still source-reproducible on current main, but this PR is not merge-ready because it targets retired legacy memory tool files instead of the active memory-core plugin path and it lacks real after-fix behavior proof.

## What This Changes

The PR wraps legacy `src/agents/tools/memory-tool.ts` `memory_search` execution in a 30-second `Promise.race` timeout and adds fake-timer unit coverage.

## Best Possible Solution

Port or replace this PR with a memory-core implementation that wraps runtime loading, manager acquisition, and search in a fail-open deadline, preserves timeout-specific unavailable metadata, and adds regression coverage in the active memory-core tests.

## Reproduction Assessment

Yes, for a source-level reproduction: a stubbed `getMemoryManagerContext` or `memory.manager.search` that returns a never-settling promise leaves the active `extensions/memory-core/src/tools.ts` execute path awaiting forever. I did not run live Telegram/iCloud proof in this read-only review.

## Solution Assessment

No, not as currently written. The useful timeout idea should be applied to the active memory-core tool boundary, and the hard deadline should be reconciled with the existing provider-specific embedding query budgets.
