## Summary

Keep open: the liveness bug is real and the fix direction is useful, but this branch changes removed legacy files while current main runs `memory_search` through `memory-core` and still lacks an outer fail-open deadline.

## What This Changes

The PR adds a 30-second `Promise.race` deadline and timeout-specific unavailable-result test coverage to the legacy `src/agents/tools/memory-tool` implementation.

## Best Possible Solution

Port or replace this with a `memory-core` fix that wraps the active `memory_search` boundary in a hard fail-open deadline and adds stalled-promise regression coverage.

## Reproduction Assessment

Yes, source-level: a stubbed `loadMemoryToolRuntime`, `getMemorySearchManager`, or `memory.manager.search` promise that never settles will keep the active `memory-core` execute path pending. I did not reproduce the original live Telegram/iCloud mount stall.

## Solution Assessment

No. A hard fail-open timeout is the right repair direction, but this PR applies it to retired legacy files instead of the active `extensions/memory-core` implementation.
