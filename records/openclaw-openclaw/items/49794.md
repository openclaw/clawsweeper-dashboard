## Summary

Keep open: the liveness bug still exists on current main because the active `memory_search` path in `memory-core` awaits manager setup and search without a wall-clock guard, while this PR targets the removed legacy core tool.

## What This Changes

This PR wraps the legacy core `memory_search` execution in a 30-second hard timeout and adds timeout-specific unavailable-result assertions.

## Best Possible Solution

Port the fail-open timeout to `extensions/memory-core`, covering manager acquisition and active search, return timeout-specific unavailable metadata, preserve corpus behavior, and add a changelog entry with focused regression coverage.

## Reproduction Assessment

Yes. A memory-core unit test can mock `memory.manager.search` or manager acquisition to never settle, and source inspection shows the active execute path awaits those promises without a timeout.

## Solution Assessment

No. The tool-boundary fail-open approach is appropriate, but this PR is not the best current implementation because it patches a retired file instead of the active `memory-core` tool path.
