## Summary

Keep open: the liveness bug is still plausible on current main, but this PR targets the retired legacy memory tool files, so it would not protect the active `memory-core` `memory_search` path.

## What This Changes

The PR adds a 30-second hard timeout and timeout-specific unavailable-result assertions to the legacy `src/agents/tools` `memory_search` implementation.

## Best Possible Solution

Port or replace this PR with a `memory-core` fix that wraps the full active `memory_search` boundary in a hard fail-open deadline and adds stalled-promise regression coverage.

## Reproduction Assessment

Yes, source-level: a stubbed `loadMemoryToolRuntime`, `getMemorySearchManager`, or `memory.manager.search` promise that never settles will keep the active `memory-core` tool execution pending. The live Telegram/iCloud mount scenario was not reproduced here.

## Solution Assessment

No. A hard fail-open timeout is the right fix direction, but this PR applies it to retired legacy files instead of the active `extensions/memory-core` implementation.
