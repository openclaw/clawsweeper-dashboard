## Summary

Keep open. Current main has strong transcript/runtime-context mitigations for system-event leakage, but it still does not provide the requested per-event SystemEvent audience/classification primitive, and the linked awareness-event PR discussion explicitly scopes that work away from delivery routing.

## What This Changes

Request a SystemEvent-level audience/classification contract for internal versus user-facing exec, cron, and heartbeat events across producers and consumers.

## Best Possible Solution

Keep this as the canonical design tracker until maintainers accept an event audience model or split the remaining exec, cron, and heartbeat delivery cases into narrower canonical follow-ups.

## Reproduction Assessment

Yes for the source-level classification gap: enqueue exec- or cron-shaped SystemEvent entries and trace current main through maybeNotifyOnExit, heartbeat prompt selection, and drainFormattedSystemEvents; there is still no event audience for those consumers to honor.

## Solution Assessment

Unclear pending maintainer design. The current runtime-context split is a good mitigation for visible transcript leakage, but the proposed event-level audience contract changes API and delivery semantics across several owners.
