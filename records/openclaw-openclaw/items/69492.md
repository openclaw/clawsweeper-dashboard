## Summary

Keep open. Current main still lacks the requested per-event SystemEvent audience/classification primitive; the relevant event producers, heartbeat relay, prompt construction, and history projection still rely on delivery flags plus text predicates. The related #72201 work is described in the issue discussion as an open, deliberately partial slice, not a merged replacement for the broader design tracker.

## What This Changes

Request to add per-event SystemEvent audience classification so internal exec, cron, and heartbeat events do not leak into user-facing transcripts or channel delivery paths.

## Best Possible Solution

Keep this as the canonical design tracker until maintainers either land an accepted SystemEvent audience model across producers, heartbeat relay, and history/display consumers, or explicitly split the remaining exec/cron/heartbeat delivery cases into narrower canonical follow-ups.

## Reproduction Assessment

Yes for the implementation gap. A source-level reproduction is to enqueue exec- or cron-shaped SystemEvent entries and trace current main through drainFormattedSystemEvents, heartbeat prompt selection, and chat history projection: there is no audience field for those consumers to honor.

## Solution Assessment

Unclear pending maintainer design. The proposed audience primitive is plausible, but current main and the open partial PRs show maintainers are still balancing hidden runtime context against explicit user-delivery paths.
