## Summary

Keep open. Current main still lacks the requested SystemEvent audience/classification primitive, and the code still separates internal versus user-facing behavior with delivery flags plus text-shaped exec/cron predicates rather than an event-level contract.

## What This Changes

Request a per-event SystemEvent audience classification to prevent internal exec, cron, and heartbeat system events from leaking into user-facing transcripts or channels.

## Best Possible Solution

Keep this as the canonical design tracker until maintainers either accept a SystemEvent audience model across producers and consumers or split the remaining exec/cron/heartbeat delivery cases into narrower canonical follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is to enqueue exec- or cron-shaped SystemEvent entries and trace current main through maybeNotifyOnExit, heartbeat prompt selection, and drainFormattedSystemEvents; there is no audience field for those consumers to honor.

## Solution Assessment

Unclear pending maintainer design. The proposed audience primitive is plausible, but current main and the partial related PRs show the boundary between hidden runtime context and explicit user delivery still needs owner direction.
