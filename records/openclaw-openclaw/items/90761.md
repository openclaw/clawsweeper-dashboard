Full review comments:

- **[P1] Reject changed SSH host keys before forwarding:** `apps/android/app/src/main/java/ai/openclaw/app/gateway/SshTunnelManager.kt:157`
  - body: `check()` returns `CHANGED` for a mismatched saved host key, but `promptYesNo()` always returns `true`. In JSch's `StrictHostKeyChecking=ask` path that means a changed SSH host key is accepted instead of aborting, so gateway traffic can be forwarded through an impersonated SSH server.
  - confidence: 0.96
- **[P1] Keep stored operator tokens on the device-token path:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:2770`
  - body: When only a stored operator token exists, current main intentionally returns no explicit token so `GatewaySession` classifies it as `DEVICE_TOKEN` and uses stored scopes/retry handling. Passing that stored token as `GatewayConnectAuth.token` makes it an explicit shared-token path for existing operator sessions.
  - confidence: 0.93
- **[P2] Preserve reasoning-only recovery:** `src/shared/text/reasoning-tags.ts:133`
  - body: Current tests recover visible text from a fully wrapped but unclosed reasoning reply like `<think>Visible answer...`. Returning an empty string here drops those malformed local-model replies instead of preserving the existing recovery behavior.
  - confidence: 0.91
- **[P2] Mask SSH secrets in Settings:** `apps/android/app/src/main/java/ai/openclaw/app/ui/SettingsScreens.kt:1007-1014`
  - body: The Settings SSH password field uses the generic `ClawTextField`, which has no password visual transformation, so users type stored SSH secrets in clear text. The same new screen also uses that unmasked component for private-key and passphrase entry, unlike the onboarding password field.
  - confidence: 0.86
