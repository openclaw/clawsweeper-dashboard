Full review comments:

- **[P1] Verify SSH host keys before forwarding:** `apps/android/app/src/main/java/ai/openclaw/app/gateway/SshTunnelManager.kt:119`
  - body: This sets `StrictHostKeyChecking` to `no` for the SSH session that forwards gateway traffic. A spoofed or changed SSH host would be accepted before the Android app sends gateway traffic through it, so the SSH feature needs pinned or TOFU-backed host-key verification before merge.
  - confidence: 0.96
- **[P1] Keep stored operator tokens on the device-token path:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:2723`
  - body: When only a stored operator token exists, current main returns a connect auth with no explicit token so `GatewaySession` classifies it as `DEVICE_TOKEN`. Passing the stored token as `GatewayConnectAuth.token` makes it look like a shared token and can lose stored-scope/device-token retry behavior for existing operator sessions.
  - confidence: 0.93
- **[P2] Pass the settings navigation callback:** `apps/android/app/src/main/java/ai/openclaw/app/ui/ShellScreen.kt:875`
  - body: The new Gateway settings button calls `onNavigate(SettingsRoute.SshTunnel)`, but the settings shell opens `SettingsDetailScreen` without passing `onRouteChange`, so that callback is the default no-op. As a result, tapping Configure SSH Tunnel from Settings does not open the new SSH settings screen.
  - confidence: 0.9
- **[P2] Preserve unclosed reasoning-only recovery:** `src/shared/text/reasoning-tags.ts:133`
  - body: Current tests expect a fully wrapped but unclosed reasoning-only reply like `<think>Answer...` to recover the visible answer. Returning an empty string here drops those malformed provider replies instead of preserving the existing recovery behavior.
  - confidence: 0.91
