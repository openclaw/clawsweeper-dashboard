## Summary

Keep open: current main still has the Memory Wiki contract mismatch. The implementation treats top-level `sources/*.md` as structured source pages that compile and lint with required metadata, while the public docs still describe `sources/` as imported raw material in a way that can reasonably read as preserve-as-is Markdown.

## What This Changes

The issue requests either relaxing Memory Wiki lint for raw files under `sources/` or clarifying that `sources/*.md` are structured source pages requiring metadata.

## Best Possible Solution

Document `sources/` as structured source-page storage and raw original preservation via ingest, bridge, or unsafe-local wrappers; leave lint behavior unchanged unless maintainers explicitly want a separate raw-drop feature.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a plain Markdown file under top-level `sources/` without frontmatter is read by compile and then lint emits missing `id`, missing `pageType`, and unknown freshness diagnostics. I did not run the CLI because `wiki lint` writes a report and this review is read-only.

## Solution Assessment

Yes for a focused repair. Clarifying the docs is the narrowest maintainable fix because compile, lint, search, import, and tests all depend on structured source pages; excluding all `sources/` files from lint would undercut that contract.
