## Summary

Keep open: current main still has the Memory Wiki contract mismatch. The code treats top-level `sources/*.md` files as structured wiki source pages that compile and lint with required metadata, while the public docs still describe `sources/` as imported raw material in a way that can reasonably read as preserve-as-is raw Markdown.

## What This Changes

The issue asks to resolve whether raw Markdown dropped under Memory Wiki `sources/` should be exempt from frontmatter linting or whether the docs should clarify that source pages are structured wiki pages.

## Best Possible Solution

Clarify the Memory Wiki and CLI docs that `sources/*.md` are structured wiki source pages requiring metadata such as `id`, `pageType: source`, and freshness/provenance fields; raw originals should be brought in through supported import paths and preserved inside the rendered source-page content/provenance. A separate raw-drop mode should be tracked separately if maintainers want that product behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: put a Markdown file without frontmatter at top-level `sources/`, then `openclaw wiki lint` compiles it as a source page and emits missing `id`, missing `pageType`, and missing freshness diagnostics.

## Solution Assessment

Yes. The narrowest maintainable fix is the issue's docs-clarification option, because weakening lint for all `sources/` files would conflict with compile, query, status, ingest, and bridge expectations for structured source pages.
