## Summary

Keep open: current main still allows direct dashboard/webchat-style turns to resolve to `message_tool_only` from global or harness defaults without considering whether there is an external source target for the message tool to infer.

## What This Changes

The issue asks OpenClaw to keep automatic final delivery for internal dashboard/webchat-only turns that lack an external send target, while preserving message-tool-only delivery for external-origin routes.

## Best Possible Solution

Add a narrow source-reply delivery guard for internal-only webchat with no external origin or explicit deliver route, and cover direct dashboard fallback plus external-route preservation in regression tests.

## Reproduction Assessment

Yes from source inspection: a direct turn with global or harness `message_tool` visible-reply defaults resolves to `message_tool_only`, while the message tool can only infer an omitted target from `toolContext.currentChannelId`. I did not run Vitest or a live dashboard session because this review must keep the checkout read-only.

## Solution Assessment

Yes: the proposed policy-level coercion is the narrowest maintainable fix because the low-level message target validator is correct and external-origin webchat routes still need message-tool-only delivery.
