## Summary

Keep open: current main still allows direct dashboard/webchat-style turns to resolve to `message_tool_only` from global or harness defaults without checking whether there is an external source target for the message tool to infer.

## What This Changes

The issue requests that internal-only dashboard/webchat turns stay on automatic final delivery instead of message-tool-only delivery when no external source conversation target exists.

## Best Possible Solution

Add a narrow source-reply delivery policy guard for internal-only webchat with no external origin or explicit deliver route, plus regression tests for direct dashboard fallback and external-origin preservation.

## Reproduction Assessment

Yes from source inspection: a direct turn with global or harness `message_tool` visible-reply defaults resolves to `message_tool_only`, while the message tool can only infer a missing send target from `toolContext.currentChannelId`. I did not run Vitest or a live dashboard session because this was a read-only cleanup review.

## Solution Assessment

Yes: the proposed policy-level coercion is the narrowest maintainable fix because the low-level message target validator is correct and external-origin webchat routes still need message-tool-only delivery.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81092 with a focused auto-reply policy change. Internal-only dashboard/webchat turns with no external origin or explicit deliver route must resolve source reply delivery to `automatic` even when global config or the active harness default prefers `message_tool`; webchat turns that are operating on behalf of an external origin or explicit deliver route must continue to allow `message_tool_only`. Do not weaken `message.send` target validation and do not try to solve the broader fail-closed behavior tracked at https://github.com/openclaw/openclaw/issues/49876 in this PR. Likely implementation should extend the source-reply delivery context or dispatch-side policy using existing internal-webchat concepts, then add regression coverage for direct dashboard/webchat with global `messages.visibleReplies: "message_tool"`, direct dashboard/webchat with harness `sourceVisibleReplies: "message_tool"`, external-origin webchat remaining message-tool-only, and `dispatchReplyFromConfig` queueing a normal final reply for the dashboard fallback. Add a single-line `CHANGELOG.md` fix entry because this affects visible dashboard/webchat reply behavior.
