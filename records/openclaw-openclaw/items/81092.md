## Summary

Keep open. Current main still lets internal dashboard/webchat turns reach `message_tool_only` via global or harness defaults even though the WebChat route has no external `OriginatingTo`, so the report is a valid narrow delivery-policy bug rather than a duplicate of the broader fail-closed issue.

## What This Changes

Prevent dashboard/webchat-only turns from using message-tool-only visible reply delivery when no external source target exists.

## Best Possible Solution

Teach the source-reply delivery policy about internal-only WebChat context and coerce only that impossible tool-only case back to automatic final delivery, while preserving target validation and external routed WebChat behavior.

## Reproduction Assessment

Yes, this is source-reproducible: current `chat.send` can create a direct WebChat context with no `OriginatingTo`, while the reply policy can still choose `message_tool_only` and the message tool then has no default target. I did not run a live dashboard turn because this review was read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to coerce internal-only WebChat turns without an external origin or explicit deliver route to `automatic`, rather than weakening `message.send` target validation or changing broader fail-closed behavior.
