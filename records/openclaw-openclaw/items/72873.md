## Summary

Keep open. Current main still asks Telegram for edited-message updates and can key them, but it does not register a runtime `edited_message` handler that dispatches edited content into the inbound pipeline. Open PR #73104 is the active unmerged closing implementation, so the issue should remain open until that PR lands or is replaced.

## What This Changes

Process Telegram `edited_message` updates, with debounce/deduplication, so streaming bot edits are not dropped after the first partial message.

## Best Possible Solution

Keep the fix owned by the Telegram plugin: register an `edited_message` handler, debounce by chat/message id, reuse the existing inbound path, skip bot-authored echo loops, and cover registration, coalescing, dedupe, and dispatch with focused tests. The practical path is to review and land #73104 or replace it with an equivalent narrow fix.

## Reproduction Assessment

Yes. A synthetic grammY `edited_message` update is enough to reproduce the source-level gap because current main requests and keys the update type but has no registered handler that passes it into the inbound pipeline.

## Solution Assessment

Yes. A narrow Telegram plugin fix with debounce and focused tests is the maintainable solution; because #73104 already uses closing syntax for this issue, the issue should stay open until that PR lands or is replaced.
