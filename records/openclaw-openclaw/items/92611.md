## Summary

Current `main` and the v2026.6.6 release already normalize the reported `openrouter/anthropic/...` ref before probe/runtime execution, so the OpenRouter transport receives provider `openrouter` with model `anthropic/...` rather than the full prefixed config key.

## What This Changes

The issue reports that OpenRouter Anthropic model refs are sent to the OpenRouter API with the internal `openrouter/` routing prefix still attached.

## Best Possible Solution

Keep the shipped model-selection normalization and close this report; reopen or file a new issue only with current-version request payload proof showing a prefixed OpenRouter model still reaches the API.

## Reproduction Assessment

No. The current and v2026.6.6 source path resolves `openrouter/anthropic/...` into provider `openrouter` and model `anthropic/...` before the probe/runtime call, so the reported prefixed payload path does not reproduce from source inspection.

## Solution Assessment

Yes. The existing model-selection normalization is the safer solution because it preserves valid OpenRouter slugs such as `openrouter/auto` while treating `openrouter/anthropic/...` as a user-facing routing ref.
