## Summary

Keep open: the exact `models status --probe` candidate path appears to normalize `openrouter/anthropic/...` before execution, but current main still has a reachable provider-runtime path where an OpenRouter-prefixed model id is left unchanged and then copied into the OpenAI-compatible request payload. That is enough to keep this as a narrow OpenRouter-owned repair rather than close it from the partially safe CLI path.

## What This Changes

OpenRouter Anthropic model refs should keep the user-facing `openrouter/...` config/routing label while sending the provider-local `anthropic/...` slug to OpenRouter.

## Best Possible Solution

Normalize the OpenRouter wire-model slug at the provider-owned resolved-model or outgoing-payload boundary, preserving `openrouter/...` as config/display routing while sending provider-local slugs to verified OpenRouter API routes.

## Reproduction Assessment

Yes, but only at source level: a direct OpenRouter provider/model path can preserve `openrouter/anthropic/...`, and the OpenAI-compatible transport sends `model.id` unchanged. The specific `models status --probe` path appears to normalize candidates first, so live credentialed proof is still useful before calling the user’s exact command reproduced.

## Solution Assessment

Yes. The best fix is a provider-owned normalization at the OpenRouter resolved-model or payload boundary, not a generic shared transport slash-strip, because `openrouter/auto` and custom proxy routes must remain valid.
