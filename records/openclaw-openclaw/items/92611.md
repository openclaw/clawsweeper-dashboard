## Summary

Keep open: current main still appears to send the selected OpenRouter model id through the OpenAI-compatible payload unchanged, while the OpenRouter plugin does not rewrite `openrouter/anthropic/...` to the OpenRouter catalog slug before dispatch.

## What This Changes

The issue reports that OpenRouter Anthropic model requests send the internal `openrouter/`-prefixed model id to the OpenRouter API instead of the provider catalog id.

## Best Possible Solution

Normalize the OpenRouter API payload model id in the OpenRouter-owned provider path while preserving `openrouter/...` as the user-facing config and routing ref, then add regression coverage for prefixed and unprefixed OpenRouter Anthropic refs.

## Reproduction Assessment

Yes at source level. Current main builds OpenAI-compatible request params with `model: model.id`, and no inspected OpenRouter runtime hook rewrites `payload.model` from `openrouter/anthropic/...` to `anthropic/...`.

## Solution Assessment

Yes. The best fix is provider-owned normalization in the OpenRouter request path rather than a shared OpenAI-compatible special case, because the `openrouter/` prefix is an OpenClaw routing ref and not an OpenRouter catalog id.
