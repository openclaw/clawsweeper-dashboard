## Summary

Keep open: current main and v2026.6.6 still leave OpenRouter-prefixed model ids unchanged before the OpenAI-compatible request payload, and two open PRs now claim the fix; this issue should stay open until one lands.

## What This Changes

The issue requests fixing OpenRouter Anthropic model routing so internal refs like `openrouter/anthropic/claude-sonnet-4.6` are sent to OpenRouter as `anthropic/claude-sonnet-4.6`.

## Best Possible Solution

Land the focused OpenRouter plugin fix that reuses `normalizeOpenRouterModelId` inside `normalizeResolvedModel`, with regression coverage for prefixed and unprefixed OpenRouter model ids.

## Reproduction Assessment

Yes, source-reproducible: current main resolves OpenRouter dynamic models with the requested id, does not strip `openrouter/` in the OpenRouter resolved-model hook, and sends `model.id` as the request `model` field. I did not run a live OpenRouter call in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is in the OpenRouter plugin's resolved-model normalization hook, not in the shared OpenAI-compatible transport. That keeps provider-owned API-facing model-id rules inside the provider plugin boundary.
