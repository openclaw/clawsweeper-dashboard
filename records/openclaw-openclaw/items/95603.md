## What This Changes

Adds an opt-in `${VAR:json}` config env-substitution modifier that parses whole-value environment JSON arrays/objects and preserves those references during config write-back.

## Best Possible Solution

Land only if maintainers accept `${VAR:json}` as public config syntax, with docs covering arrays/objects, escape behavior, malformed values, missing vars, and write-back preservation.

## Reproduction Assessment

Yes for current behavior by source inspection: current main always returns a string for string env substitutions, so `${VAR}` cannot materialize an array/object before schema validation. I did not run a live repro because this was a read-only review.

## Solution Assessment

No, not merge-ready as submitted. The opt-in implementation is a plausible narrow solution and the earlier parser defects are fixed, but the public config syntax needs maintainer sponsorship and docs.
