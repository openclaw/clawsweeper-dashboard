## Summary

Keep open: the proposed capability addresses a real config gap and includes useful terminal proof, but the patch is not merge-ready because it changes every whole-value config env reference from string substitution to content-based type inference before schema validation, including credential surfaces.

## What This Changes

The PR changes `resolveConfigEnvVars` to parse a whole-value `${VAR}` as a JSON array/object when the environment value looks like one, and adds focused substitution tests.

## Best Possible Solution

Preserve current string substitution by default and add a maintainer-approved structured-env path that only parses arrays/objects when the target config contract explicitly asks for it.

## Reproduction Assessment

Yes for the current behavior as source-reproducible: current main returns `substituteString` for every string, so `${ALLOW_FROM}` resolves to a string before schema validation. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR is a plausible implementation of the requested capability, but not the best fix because global pre-validation content sniffing changes the string-substitution contract; an explicit or schema-aware structured-env path is safer.
