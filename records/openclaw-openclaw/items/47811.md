## Summary

The concrete profile-qualified state and LaunchAgent behavior is shipped, but the umbrella issue still has a meaningful unresolved remainder: exact-case profile identities and explicit path/label overrides can create confusing or inconsistent operator truth, and the reported cross-profile service selection has not been isolated on current macOS behavior.

## What This Changes

Clarify and correct profile, config, service, and session targeting for macOS installations using multiple named profile homes.

## Best Possible Solution

Define exact-case named profiles as one canonical identity, show the resolved profile plus effective state/config/service values on operator commands, and fail or prominently warn before lifecycle actions when those values disagree; split unrelated plugin attachment and agent-tool forcing requests into narrower items.

## Reproduction Assessment

No, we do not yet have a high-confidence failing reproduction on current main. Source establishes the exact case-sensitive and override-sensitive resolution path, but a current macOS trace of the selected profile, effective environment, plist label, and launchctl target is still missing.

## Solution Assessment

No, automatic casing normalization or support for two competing layout models is not the safest fix. The maintainable path is to retain one canonical per-profile layout and add explicit identity/mismatch diagnostics after maintainers decide the compatibility contract for case variants and overrides.
