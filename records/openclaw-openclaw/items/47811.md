## Summary

Keep open. Current main has better multi-gateway docs and gateway-status drift diagnostics, but it does not fully resolve the reported ambiguity: profile names are still case-preserved into state paths and service labels, and the supported contract for mixed `~/.openclaw` plus `~/.openclaw-*` layouts still needs maintainer judgment.

## What This Changes

The issue asks OpenClaw to make multi-profile home resolution, profile casing, LaunchAgent targeting, and plugin/config/load truth authoritative for mixed `~/.openclaw` and `~/.openclaw-*` macOS layouts.

## Best Possible Solution

Maintainers should first define the supported profile/home/casing contract, then split follow-up work into explicit resolved-profile/service/config/status output and any separate `OPENCLAW_HOME` path repairs.

## Reproduction Assessment

Partly. Source inspection gives a high-confidence path for the casing footgun because `--profile Eagle` and `--profile eagle` derive different state/config paths and LaunchAgent labels, but the full Tern/eagle wrong-boundary report still lacks a current-main live macOS reproduction.

## Solution Assessment

Unclear. The requested direction is reasonable, but choosing case-insensitive normalization versus explicit fail-closed messaging and defining mixed-home support are product/contract decisions rather than a narrow mechanical bug fix.
