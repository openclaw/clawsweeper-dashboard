## Summary

Current `main` already gives named profiles separate state/config roots, targets profile-qualified macOS LaunchAgents, and shows both CLI/service config paths plus the service label in `gateway status`. A meaningful remainder still exists: profile names remain case-preserving, the mixed-layout policy is not stated authoritatively, and the reported `eagle` command reaching Tern has not been isolated on current macOS behavior. Keep this open for a focused live reproduction and explicit compatibility/product decision.

Likely related people:
- @steipete — high confidence; introduced named CLI profiles and profile-aware gateway service names.
- @vincentkoc — high confidence; recently repaired adjacent macOS LaunchAgent supervision behavior.
- @sebslight — medium confidence; centralized home and state path resolution used by profiles.

## What This Changes

Clarify and harden named-profile resolution, per-profile state/config paths, macOS service targeting, and operator diagnostics for multi-profile installations.

## Best Possible Solution

Adopt one documented canonical profile identity model; keep state directory, config path, service label, and stored service environment derived from it; expose those resolved values consistently in status and doctor; and only change case behavior through a doctor-backed migration after a minimal macOS reproduction identifies any remaining wrong-target defect.

## Reproduction Assessment

Partly yes: source inspection gives a high-confidence path for the case-sensitive identity behavior and config-path mismatch diagnostics. No high-confidence current-main reproduction yet demonstrates `--profile eagle` selecting the Tern LaunchAgent on macOS.

## Solution Assessment

No, not as one broad change; current status behavior already covers part of the requested truth surface, while case normalization and mixed-layout semantics require an upgrade-sensitive product decision and the alleged wrong-target path needs an isolated reproduction.
