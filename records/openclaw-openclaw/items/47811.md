## Summary

Keep open. Current main has narrower fixes and better diagnostics for profile-aware gateway status/restart, but the remaining asks around case policy, mixed `OPENCLAW_HOME`/profile-home semantics, and authoritative operator truth still need maintainer contract decisions.

## What This Changes

The issue asks OpenClaw to clarify and harden multi-profile home/config/service targeting for mixed `~/.openclaw` and `~/.openclaw-*` macOS layouts.

## Best Possible Solution

Maintainers should define the supported profile/home/casing contract, then split implementation into explicit resolved-profile/service/config/status output, any fail-closed or normalization behavior, and separate `OPENCLAW_HOME` path repairs.

## Reproduction Assessment

Partly. Source inspection gives a high-confidence path for the casing footgun because `--profile Eagle` and `--profile eagle` still derive different state/config paths and LaunchAgent labels, but the full wrong-boundary macOS report was not live-reproduced on current main.

## Solution Assessment

Unclear. The requested direction is reasonable, but choosing case-insensitive normalization versus explicit fail-closed messaging and defining mixed-home support are product/contract decisions rather than one narrow mechanical fix.
