## Summary

Keep this issue open. Current main supports named profile homes and documents isolated multi-gateway setups, and status has partial config-mismatch diagnostics, but it still does not define or loudly reject custom-profile casing and does not provide the requested authoritative target report tying profile, config path, state directory, service label/process boundary, probe URL, and session store together.

## What This Changes

The issue requests clearer and safer multi-profile targeting/config truth for macOS setups using multiple `~/.openclaw-*` homes, including profile resolution output, casing behavior, service targeting, and config/load diagnostics.

## Best Possible Solution

Define the supported custom-profile semantics first, then update core CLI/gateway behavior to fail loudly or migrate safely on ambiguous casing and to print one authoritative target context for status/restart/agent flows: resolved profile, config path, state directory, LaunchAgent/service label, process/probe target, and session store. Add macOS LaunchAgent tests for multiple named homes and case-mismatched profile names.

## Reproduction Assessment

Unclear. The report gives credible real-host observations and current source confirms case-sensitive custom profile/state/service behavior, but there is not yet a high-confidence one-command reproduction for the specific wrong-profile service-boundary resolution on current main.

## Solution Assessment

Unclear. There is no patch to assess, and the narrowest maintainable path depends on a maintainer decision about whether custom profile names should be canonicalized, rejected on case mismatch, or kept case-sensitive with clearer target output.
