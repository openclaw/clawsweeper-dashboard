## Summary

Keep open. Current main supports named profile homes and isolated multi-gateway setups, but it still preserves custom profile casing and the status path does not provide the requested authoritative target report tying profile, config, state, service label, probe target, and session store together.

## What This Changes

Clarify and harden multi-profile CLI/gateway targeting so named profile homes, config paths, service boundaries, and session state are visibly resolved and not confused.

## Best Possible Solution

Define supported custom-profile semantics, then update CLI/gateway diagnostics to fail loudly or migrate safely on ambiguous casing and print one authoritative target context for status, restart, and agent flows.

## Reproduction Assessment

No high-confidence one-command reproduction is available for the reported wrong-profile status behavior. Source inspection does reproduce the remaining ambiguity: custom profiles are case-preserving and status lacks a complete target-context block.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. The narrow maintainable direction is to decide whether custom profiles are case-sensitive, normalized, or rejected on mismatch, then make status/restart output expose the resolved target explicitly.
