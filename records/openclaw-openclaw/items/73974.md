## Summary

Keep open: current main still lacks a Mac-app LaunchAgent ownership guard for doctor repairs, but this PR is not mergeable as-is because it can misclassify the CLI `mac` profile as OpenClaw.app ownership and suppress legitimate repair.

## What This Changes

The PR adds macOS OpenClaw.app LaunchAgent ownership detection to doctor daemon and service-config repair paths, updates doctor docs and the changelog, and adds launchd scanner regression tests.

## Best Possible Solution

Keep the app ownership guard, but require app-specific launchd evidence before suppressing CLI repair and preserve repair behavior for profile-named CLI services.

## Reproduction Assessment

Yes. The original duplicate-runtime problem has concrete macOS app plus `doctor --fix` steps, and the PR regression is source-reproducible with `OPENCLAW_PROFILE=mac` plus a CLI-managed `ai.openclaw.mac` plist or loaded job.

## Solution Assessment

No. The product direction is narrow and appropriate, but label-only ownership detection is not the safest implementation because OpenClaw’s existing profile-label contract can produce the same label.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Disambiguate the mac profile from the app LaunchAgent:** `src/commands/doctor-service-repair-policy.ts:38`
  - body: `OPENCLAW_PROFILE=mac` resolves to `ai.openclaw.mac`, so this condition enables app-owner detection for a legitimate CLI profile. Since the scanner accepts any plist with that label without checking app-specific `ProgramArguments`, a CLI-owned `mac` profile LaunchAgent can make `doctor --fix` skip install/start/restart/config repair even though OpenClaw.app is not managing the runtime. Require app-specific ownership evidence before suppressing repair.
