## Summary

Keep open. The PR targets a valid unresolved macOS doctor/LaunchAgent ownership bug, and current main still lacks an app-owner guard, but the proposed implementation can falsely suppress CLI repair for a legitimate `OPENCLAW_PROFILE=mac` standalone LaunchAgent because it treats `ai.openclaw.mac` as app ownership by label alone.

## What This Changes

The PR adds macOS OpenClaw.app LaunchAgent ownership detection to doctor daemon and service-config repair paths, plus launchd scanner helpers, docs, changelog, and regression tests so CLI repair skips standalone gateway mutations when the app owns launchd.

## Best Possible Solution

Revise this PR so app ownership is detected from app-specific plist or loaded-job evidence, such as `ProgramArguments` pointing at `OpenClaw.app/Contents/MacOS/OpenClaw`, while preserving normal profile-specific CLI LaunchAgent repair and the existing app-owned skip behavior for default local mode.

## Reproduction Assessment

Yes. The linked #67631 report gives concrete macOS reproduction steps for the user bug, and the PR regression can be reproduced with `OPENCLAW_PROFILE=mac` plus a CLI-owned `ai.openclaw.mac` LaunchAgent plist that would be mistaken for OpenClaw.app ownership by the PR's label-only scanner.

## Solution Assessment

No. The product direction is right, but this patch is not the best implementation because label-only ownership detection suppresses legitimate CLI repair; the safer fix is to require app-specific evidence before skipping install/start/restart/config repair.
