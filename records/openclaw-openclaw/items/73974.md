## Summary

Keep open. The PR addresses a real current-main gap in macOS doctor repair, but the proposed implementation still has a blocking label-collision bug: a CLI profile named `mac` resolves to the same `ai.openclaw.mac` label and can be mistaken for OpenClaw.app ownership.

## What This Changes

The PR adds macOS OpenClaw.app LaunchAgent ownership detection to doctor daemon and service-config repair paths, plus docs, changelog, constants, scanner helpers, and regression tests.

## Best Possible Solution

Revise this PR in place so app ownership is detected from app-specific plist or loaded-job evidence, such as `ProgramArguments` pointing at `OpenClaw.app/Contents/MacOS/OpenClaw`, while preserving default-profile behavior and adding explicit regression coverage for `OPENCLAW_PROFILE=mac` and unrelated profile labels.

## Reproduction Assessment

Yes. The linked #67631 report gives concrete macOS reproduction steps, and the PR-specific regression is reproducible by setting `OPENCLAW_PROFILE=mac` with a CLI LaunchAgent whose label is `ai.openclaw.mac`; the PR's label-only detection would then skip CLI repair as if OpenClaw.app owned launchd.

## Solution Assessment

No. The product direction is right, but the implementation should not treat the label alone as ownership; it needs an app-specific marker before suppressing CLI install/start/restart/config repair.
