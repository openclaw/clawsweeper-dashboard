## Summary

Keep open. Current main still allows doctor repair to install, bootstrap, start, restart, or rewrite the standalone gateway LaunchAgent without checking whether OpenClaw.app owns the local macOS launchd path, so the PR is not obsolete; however, the proposed patch is not safe to merge because it can misclassify a CLI `OPENCLAW_PROFILE=mac` LaunchAgent as app-owned and suppress legitimate CLI repair.

## What This Changes

The PR adds Darwin OpenClaw.app LaunchAgent ownership detection to doctor daemon repair and service-config repair, updates doctor docs and changelog text, and adds launchd scanner/regression tests.

## Best Possible Solution

Revise this PR in place, or replace it with an equivalent narrow PR, so app ownership is detected from app-specific plist or loaded-job evidence such as `ProgramArguments` pointing at `OpenClaw.app/Contents/MacOS/OpenClaw`; preserve the doctor skip behavior for real app ownership and add explicit regression coverage for `OPENCLAW_PROFILE=mac`.

## Reproduction Assessment

Yes. #67631 gives a high-confidence macOS app plus `openclaw doctor --fix` duplicate-runtime reproduction, and the PR-specific regression is reproducible by combining `OPENCLAW_PROFILE=mac` with a CLI-installed `ai.openclaw.mac` LaunchAgent plist whose `ProgramArguments` run `openclaw gateway`.

## Solution Assessment

No. The product direction is right, but the current patch is not the best implementation because it suppresses repair based on a colliding label rather than a Mac-app-specific ownership marker.
