## Summary

Keep open. The PR targets a valid macOS app/CLI launchd ownership bug, but the current diff can falsely treat a CLI `OPENCLAW_PROFILE=mac` LaunchAgent as OpenClaw.app ownership because the new guard and scanner rely on the `ai.openclaw.mac` label without app-specific evidence.

## What This Changes

The PR adds macOS OpenClaw.app LaunchAgent ownership detection to doctor daemon and service-config repair paths, updates doctor docs and the changelog, and adds launchd scanner regression tests.

## Best Possible Solution

Keep the app ownership guard, but require app-specific launchd evidence before suppressing CLI repair and preserve normal repair behavior for profile-named CLI services.

## Reproduction Assessment

Yes. The original duplicate-runtime bug has concrete macOS app plus `openclaw doctor --fix` steps in #67631 and #42573, and the PR regression is source-reproducible with `OPENCLAW_PROFILE=mac` plus a CLI-managed `ai.openclaw.mac` LaunchAgent.

## Solution Assessment

No. The requested guard is the right product direction, but label-only app ownership detection is not the narrowest maintainable implementation because OpenClaw profile labels can collide with the app label.

## Review Findings

- **[P1] Disambiguate the mac profile from the app LaunchAgent:** `src/commands/doctor-service-repair-policy.ts:38`
