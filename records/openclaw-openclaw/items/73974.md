## Summary

Keep open. Current main still lacks the macOS app-owned LaunchAgent guard, and this PR addresses a real duplicate-runtime report, but the proposed patch has a blocking false-positive path for the existing CLI `mac` profile.

## What This Changes

The PR adds macOS OpenClaw.app LaunchAgent ownership detection to doctor daemon and service-config repair paths, updates doctor docs and the changelog, and adds launchd scanner regression tests.

## Best Possible Solution

Keep the product direction, but require app-specific launchd evidence before suppressing CLI repair: validate that the plist or loaded job points at `OpenClaw.app/Contents/MacOS/OpenClaw`, preserve default-profile behavior, and add regression coverage for `OPENCLAW_PROFILE=mac` plus nearby profile labels.

## Reproduction Assessment

Yes. The linked issue gives concrete macOS duplicate-runtime steps, and the PR regression is reproducible statically by combining `OPENCLAW_PROFILE=mac`, which resolves to `ai.openclaw.mac`, with the PR's label-only plist ownership scan.

## Solution Assessment

No. Skipping CLI repair when OpenClaw.app owns launchd is the right narrow direction, but label-only ownership detection is not the best implementation because it collides with an existing CLI profile label.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Disambiguate the mac profile from the app LaunchAgent:** `src/commands/doctor-service-repair-policy.ts:38`
  - body: `OPENCLAW_PROFILE=mac` resolves to `ai.openclaw.mac`, so this condition enables app-owner detection for a legitimate CLI profile. Require app-specific evidence before suppressing repair.
