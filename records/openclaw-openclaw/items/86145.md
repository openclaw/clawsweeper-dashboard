## Summary

Keep open: current main has pre-compaction safeguards, but I did not find a final OpenAI-compatible payload invariant that rejects or repairs a system/tool-only `messages[]` before it reaches Qwen, so the reported production failure is still actionable.

## What This Changes

The issue asks OpenClaw to prevent compaction from sending OpenAI-compatible Qwen requests whose `messages[]` lack any user or assistant turn, returning a clear recovery path instead of an upstream 400.

## Best Possible Solution

Add a generic OpenAI-compatible payload/transcript invariant: after compaction and transport transforms, preserve or recover at least one real user/assistant turn, otherwise abort locally with an actionable history-exhausted error before sending upstream.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review because no sanitized JSONL or outbound payload capture is attached. Source inspection does show a credible failing path: current main lacks a post-conversion user/assistant role invariant for OpenAI-compatible payloads.

## Solution Assessment

Yes for the minimum fix direction: a narrow generic guard with regression coverage is the maintainable first step. If captured sessions prove the root cause is earlier compaction pruning, the same PR should preserve the latest real turn rather than adding Qwen-specific policy.
