## Summary

Keep open: current main has several compaction safeguards, but the OpenAI-compatible payload path still does not enforce that the final `messages[]` sent to Qwen contains a non-empty `user` or `assistant` entry after conversion and provider shaping.

## What This Changes

The issue requests a fix for long-session compaction producing Qwen/OpenAI-compatible outbound payloads with only `system` or `tool` messages and no real conversation turn.

## Best Possible Solution

Add a generic OpenAI-compatible outbound payload invariant after compaction and transport shaping: preserve/recover a real recent user or assistant turn when available, otherwise fail locally with a clear history-exhausted error before Qwen sees an invalid request.

## Reproduction Assessment

No high-confidence live reproduction was run because no sanitized session JSONL or outbound request capture is attached. Source inspection still shows a credible current-main gap: the final OpenAI-compatible payload path does not reject or repair system/tool-only `messages[]`.

## Solution Assessment

Yes, a narrow generic guard at the OpenAI-compatible payload or final transcript boundary is the maintainable direction. A Qwen-specific workaround or fake user message would be weaker than preserving real context or failing locally before the upstream 400.
