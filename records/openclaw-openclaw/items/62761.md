## Label Justifications

- `P1`: The issue describes a real iMessage channel workflow that can reprocess old messages and trigger outgoing reply loops after restart.
- `impact:message-loss`: The affected behavior is duplicated/replayed channel messages being treated as new inbound events.
- `impact:session-state`: Old iMessage rows can be injected into fresh agent turns, causing conversation and transcript state drift.

## Summary

Keep open. Current main includes the normal `is_from_me` drop path and merged opt-in iMessage catchup, but the default live `watch.subscribe` path still starts without a persisted rowid or time fence, so the canonical restart/iCloud stale-replay bug is not fully resolved.

## What This Changes

The issue asks OpenClaw to persist or fence the iMessage watch cursor so restarts do not reprocess old Messages rows and trigger outbound loops.

## Best Possible Solution

Add an iMessage-owned default stale-replay guard, likely a per-account high-water rowid or startup time fence for live watch, while keeping broad historical catchup explicitly bounded and opt-in.

## Reproduction Assessment

Yes at source level, but not with a live macOS run in this review. Current main still subscribes live with no persisted cursor or `start` fence by default, and catchup remains disabled unless configured.

## Solution Assessment

Unclear until maintainer review. The merged opt-in catchup is a good partial solution, but the remaining default startup guard needs an explicit policy so OpenClaw avoids stale replay without unexpectedly enabling broad history replay.
