## Label Justifications

- `P1`: The issue describes a real iMessage channel workflow that can reprocess old rows and trigger outgoing reply loops after restart.
- `impact:message-loss`: Old or duplicated iMessage rows can be treated as new inbound channel messages and produce duplicate agent replies.
- `impact:session-state`: Stale iMessage rows can enter fresh agent turns and drift the transcript/session state away from the live conversation.

## Summary

Keep open. Current main and the latest release include the normal outbound `is_from_me` drop path and opt-in iMessage catchup, but the default live `watch.subscribe` startup path still has no persisted rowid or startup time fence, so the restart stale-replay problem remains unresolved.

## What This Changes

The issue requests persistent iMessage cursor or startup-fence behavior so gateway restarts and update restarts do not replay old Messages rows as fresh inbound messages.

## Best Possible Solution

Add an iMessage-owned default stale-replay guard, likely a per-account high-water rowid or startup time fence for live watch, while keeping broader historical catchup explicit, bounded, and documented.

## Reproduction Assessment

Yes at source level, but not with a live macOS run in this review. Current main and v2026.5.19 still reach the default live `watch.subscribe` path without a persisted rowid or startup time fence when catchup is disabled.

## Solution Assessment

Unclear until maintainer policy review. The merged opt-in catchup is a good partial solution, but the best default guard needs to avoid both stale replay and surprise broad history replay for existing users.
