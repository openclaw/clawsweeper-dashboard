## Summary

Keep open. Current main fixed the normal `is_from_me` echo-loop subcase and added opt-in iMessage catchup, but the default live restart path still does not pass a `since_rowid`/`start` cursor or startup fence, and the related open PR is not merged.

## What This Changes

The issue asks OpenClaw to persist an iMessage rowid cursor and resume or fence `imsg watch` after update/restart so old rows are not processed again.

## Best Possible Solution

Land a maintainer-approved iMessage cursor/resume path that persists the per-account last processed rowid, passes `since_rowid`/`start` or applies an equivalent startup fence by default, and preserves bounded opt-in catchup semantics where appropriate.

## Reproduction Assessment

Yes at source level, but not with a live macOS run. Current main still subscribes to `watch.subscribe` without a cursor/fence and catchup is disabled by default, so a mocked imsg restart/notification path can exercise the remaining replay gap while current `is_from_me` drops cover only the echo-loop subcase.

## Solution Assessment

Unclear until maintainer review. The open cursor-resume PR is on the right plugin boundary, but current main's opt-in catchup does not fully solve this default restart replay issue, and the PR still needs proof, concurrency fixes, and default-semantics approval.
