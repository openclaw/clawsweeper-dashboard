## Summary

Keep open. Current main has addressed the later `is_from_me` echo-loop subcase for normal DMs, but it still does not persist an iMessage replay cursor or pass a resume point into `watch.subscribe`; related replay/echo reports were already closed into this issue as the canonical tracker.

## What This Changes

The issue requests durable iMessage rowid cursor persistence so restarts or update failures do not cause already-processed Messages rows to be treated as fresh inbound messages.

## Best Possible Solution

Add an iMessage-owned per-account high-water cursor with an explicit startup fence: load the stored rowid before subscribing, pass the dependency-supported `since_rowid` value to `watch.subscribe`, persist the latest safe rowid after messages are handled or intentionally skipped, and keep the existing `is_from_me`/self-chat guards intact with restart replay regression coverage.

## Reproduction Assessment

Yes, for the remaining missing-cursor behavior. The source-level reproduction is to start the legacy iMessage monitor and inspect the mocked `watch.subscribe` request: current main sends only `attachments`, while the issue and related reports provide live restart/replay logs for the user-visible failure.

## Solution Assessment

Yes, but only if the implementation includes a first-run/startup high-water fence and per-account persistence. The already-landed `is_from_me` filtering is necessary for the echo subcase but is not the narrowest complete solution for restart replay cursor loss.
