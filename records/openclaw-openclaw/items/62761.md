## Summary

Keep open. Current main fixed the normal-DM `is_from_me` echo loop and added opt-in iMessage downtime catchup, but the default live `watch.subscribe` path still sends no `since_rowid` or startup high-water fence, so the original restart-replay protection is not fully implemented.

## What This Changes

The issue requests durable iMessage rowid cursor persistence so restarts do not replay already-processed chat history as new inbound messages.

## Best Possible Solution

Add a default iMessage startup replay fence or explicit resume behavior that prevents pre-start rows from dispatching while preserving the opt-in catchup feature for messages received while the gateway was offline.

## Reproduction Assessment

Yes at source level, but not with a live macOS run. Current main still subscribes to iMessage watch without a cursor/fence and catchup is opt-in, so a mocked watch notification stream can exercise the remaining replay path while current from-me drops cover only the echo subcase.

## Solution Assessment

Unclear until maintainer review. The merged catchup implementation is useful for the inverse missed-message problem, but the narrow fix for this issue should avoid changing default catchup semantics unless maintainers explicitly want that behavior.
