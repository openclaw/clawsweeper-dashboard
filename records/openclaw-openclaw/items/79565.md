## Summary

Keep open: the PR is aimed at a real Android reconnect bug, but it is not merge-ready because the new connection publication point can race with the queue flush and the PR has no after-fix real behavior proof.

## What This Changes

The PR moves Android `GatewaySession.currentConnection` publication until after the connect RPC, adds a bounded in-memory queue for notification-forwarded node events, and adds a Robolectric regression test.

## Best Possible Solution

Separate post-handshake readiness from callback delivery so node events cannot precede connect, then flush or reconcile forwarded notifications only after the node session is definitely send-ready, with a changelog entry and real Android/Gateway reconnect proof.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: `currentConnection` is set before `conn.connect()` completes, and the notification sink sends through that connection during the reconnect window; the PR also adds a focused withheld-connect test, though it was not run here.

## Solution Assessment

No, not as submitted. Moving `currentConnection` after `conn.connect()` blocks pre-handshake sends, but because `onConnected` still fires inside `conn.connect()`, the new queue flush can run before the connection is visible to senders.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Publish ready connections before flushing the queue:** `apps/android/app/src/main/java/ai/openclaw/app/gateway/GatewaySession.kt:901`
  - body: This line now runs only after `conn.connect()` returns, but `conn.connect()` calls `handleConnectSuccess()` and `onConnected` before it completes. `NodeRuntime` starts `flushPendingForwardedNodeEvents()` from that callback, so a queued notification can be drained while `currentConnection` is still null, fail `sendNodeEvent`, and remain stuck until a later reconnect.
