## Summary

Keep open. Current main still lets an omitted message-tool `accountId` become the inherited/default account before the calling agent binding is considered, and the focused fix PR at https://github.com/openclaw/openclaw/pull/74052 was closed unmerged after review found an additional SecretRef scoping concern.

## What This Changes

Default omitted Slack `message` tool account selection to the calling agent's configured channel account while preserving explicit `accountId` overrides.

## Best Possible Solution

Implement a narrow shared account-selection path where explicit `accountId` wins, the calling agent's route binding is consulted next, inherited/default account is last, and message-tool SecretRef resolution uses the same selected account.

## Reproduction Assessment

Yes, source-level. A `runMessageAction` send with omitted `params.accountId`, `defaultAccountId: "parent-account"`, `agentId: "agent-b"`, and an `agent-b` binding to `account-b` will choose the default at `message-action-runner.ts:1048` and skip the binding lookup.

## Solution Assessment

Yes, with one implementation caveat. The requested precedence is the narrow maintainable fix, but the account must be selected early enough that SecretRef scoping and dispatch use the same identity.
