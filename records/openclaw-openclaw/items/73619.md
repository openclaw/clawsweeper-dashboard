## Summary

Keep open: current main still resolves omitted message-tool account IDs from inherited/default context before the calling-agent binding, and the focused fix PR was closed unmerged.

## What This Changes

Default an omitted Slack/message-tool `accountId` to the calling agent's configured channel account instead of inherited delivery context.

## Best Possible Solution

Keep explicit `accountId` authoritative, then resolve the calling agent's channel/target binding before inherited/default fallback, with message-tool SecretRef scoping using the same selected account.

## Reproduction Assessment

Yes. By source inspection, `runMessageAction` with omitted `params.accountId`, `defaultAccountId: "ops"`, `agentId: "agent-b"`, and an `agent-b` binding to `account-b` chooses `ops` before the binding lookup can run.

## Solution Assessment

Yes. The narrow shared message-tool/outbound-runner fix is the right boundary, provided SecretRef scoping follows the same selected bound account.
