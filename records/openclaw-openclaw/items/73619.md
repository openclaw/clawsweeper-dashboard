## Summary

Keep open. Current main still lets an omitted message-tool `accountId` become the inherited/default account before the calling agent binding can run, and the previously linked fix PR was closed unmerged.

## What This Changes

Default omitted `message` tool `accountId` to the calling agent's channel binding before falling back to inherited/default account context.

## Best Possible Solution

Land a focused shared outbound/message-tool fix: explicit `accountId` first, calling-agent channel/peer binding second, inherited/default account last, with SecretRef scoping aligned to the selected account.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: omitted `params.accountId` plus `defaultAccountId: "ops"`, `agentId: "agent-b"`, and an `agent-b` binding to `account-b` chooses `ops` before the binding lookup can run.

## Solution Assessment

Yes. The narrow maintainable solution is to fix precedence in the shared message-tool/outbound runner path and keep Slack plugins/config untouched; a Slack-specific workaround would leave other channel accounts inconsistent.
