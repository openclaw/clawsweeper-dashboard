## Summary

Keep open. Current main still lets omitted message-tool `accountId` resolve from inherited/default account context before the calling agent binding can win, and the focused fix PR at https://github.com/openclaw/openclaw/pull/74052 was closed unmerged with no open replacement found.

## What This Changes

Default omitted message-tool Slack `accountId` values to the calling agent's configured channel account while preserving explicit overrides.

## Best Possible Solution

Implement a narrow shared message-tool/outbound fix: explicit `accountId` first, calling-agent binding next, inherited/default account last, with SecretRef scoping aligned to the selected account.

## Reproduction Assessment

Yes, source-reproducible. A focused `runMessageAction` case with omitted `params.accountId`, `defaultAccountId: "ops"`, `agentId: "agent-b"`, and an `agent-b` binding to `account-b` will choose `ops` on current main before the binding lookup can run.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the shared message-tool/outbound runner path rather than Slack-specific delivery. The repair should also select or defer account-scoped SecretRefs consistently so the dispatched account and resolved secrets match.
