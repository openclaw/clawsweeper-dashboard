## Summary

Keep open: current main and the latest release still resolve omitted message-tool account IDs from the inherited/default account before the calling-agent binding, and the focused fix PR was closed unmerged.

## What This Changes

Default omitted Slack `message` tool `accountId` values to the calling agent's configured channel account while preserving explicit overrides.

## Best Possible Solution

Keep explicit `accountId` authoritative, then resolve the calling agent's channel/target binding before inherited/default fallback, with message-tool SecretRef scoping using the same selected account.

## Reproduction Assessment

Yes. By source inspection, a `runMessageAction` call with omitted `params.accountId`, `defaultAccountId: "ops"`, `agentId: "agent-b"`, and an `agent-b` binding to `account-b` will choose `ops` before the binding lookup can run.

## Solution Assessment

Yes, with one caveat. The narrow shared message-tool/outbound-runner fix is the right owner boundary, but it must also make SecretRef scoping follow the selected bound account.
