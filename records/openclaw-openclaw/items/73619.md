## Summary

Keep open. Current main still lets an omitted message-tool `accountId` resolve through the inherited/default account before the calling agent's channel binding, and the focused closing PR was closed unmerged.

## What This Changes

Default omitted `message` tool account selection to the calling agent's own Slack/channel account binding instead of inherited delivery context, while preserving explicit overrides.

## Best Possible Solution

Fix the shared message-tool/outbound account resolver so explicit `accountId` wins, the calling agent's bound channel account is next, and inherited/default account context is only the final fallback.

## Reproduction Assessment

Yes, source-reproducible. A `runMessageAction` call with omitted `params.accountId`, `defaultAccountId: "ops"`, `agentId: "agent-b"`, and an `agent-b` binding to `account-b` chooses `ops` before the binding lookup can run.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared message-tool/outbound account-selection path, with SecretRef scoping aligned to the selected bound account; Slack-specific delivery code should not need special casing.
