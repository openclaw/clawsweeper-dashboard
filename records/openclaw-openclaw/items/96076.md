## Summary

Keep this PR open, but it is not merge-ready: the patch still only appends discovered source-root plugins to the manifest registry, while Gateway startup planning consumes the unchanged installed index. ## What This Changes

The PR changes `src/plugins/plugin-metadata-snapshot.ts` so a non-empty installed-plugin index also scans source-root plugins and appends unindexed manifest records to the manifest registry.

## Best Possible Solution

Land one canonical registry-level fix that re-derives stale persisted source-root plugin indexes, keeps validation and startup on the same plugin set, adds focused regression coverage, and includes redacted Gateway or runtime-registry proof.

## Reproduction Assessment

Yes, source-reproducible. Current main can reuse a non-empty persisted plugin index for validation/startup paths that do not include a newly added source-root memory plugin; I did not run the live Windows Gateway flow in this read-only review.

## Solution Assessment

No. Appending manifest records is a plausible mitigation but not the best fix because startup planning still consumes `metadataSnapshot.index`; the safer fix is to re-derive stale registry state or update the index and manifest view atomically.
