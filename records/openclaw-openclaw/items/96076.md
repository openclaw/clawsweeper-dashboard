## Summary

Keep open with a blocking finding: the patch makes validation see source-root manifests but leaves the plugin registry snapshot index stale, so Gateway startup can still skip loading the configured memory plugin.

## What This Changes

The branch appends non-indexed source-root plugin manifests to the plugin metadata snapshot when a persisted installed-plugin index exists.

## Best Possible Solution

Fix the stale persisted registry at the registry snapshot boundary so the derived index and manifest registry agree, then add regression coverage and real Gateway or registry proof.

## Reproduction Assessment

Yes, source-reproducible: current main validates `plugins.slots.memory` against manifest registry ids derived from the persisted index path, and startup loading later uses ids derived from `index.plugins`. I did not run the live Windows Gateway scenario in this read-only review.

## Solution Assessment

No, this is not the best fix as written because it updates only the manifest view used by validation while leaving the startup index stale. The safer owner-boundary fix is to invalidate or rederive the stale persisted registry so all downstream metadata consumers see one consistent snapshot.
