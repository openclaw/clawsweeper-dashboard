## Summary

Keep open: the patch is not merge-ready because it appends unindexed source-root plugins only to the manifest registry while Gateway startup still plans runtime plugin loading from the unchanged metadata snapshot index. The updated terminal banner is also insufficient real behavior proof, and a sibling open PR targets the same regression at the registry boundary with regression coverage.

## What This Changes

The PR imports `discoverOpenClawPlugins` and merges unindexed source-root plugin manifests into `plugin-metadata-snapshot` when the installed plugin index is non-empty.

## Best Possible Solution

Fix stale persisted registry handling so the metadata snapshot index and manifest registry stay aligned, add focused stale-index coverage, and require redacted Gateway or runtime-registry proof before landing one canonical PR.

## Reproduction Assessment

Yes. Source inspection shows current main can reuse a non-empty persisted plugin index and then derive validation IDs and startup IDs from paths that do not include a newly added source-root memory plugin; I did not run the live Windows Gateway flow in this read-only review.

## Solution Assessment

No. Appending manifest records is too narrow because the startup planner still consumes `metadataSnapshot.index`; the safer fix is to re-derive stale registry state or update the index and manifest view atomically.
