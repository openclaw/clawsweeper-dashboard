## Summary

Keep open. The PR targets the right Gateway chat projection bug, but the patch broadcasts assistant media as raw image URLs instead of using the existing managed outgoing image path, so local/data media safeguards are bypassed, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds a per-run Gateway chat media buffer, threads assistant event `mediaUrls` into live delta/final chat payloads, and adds unit coverage for text+media and media-only assistant events.

## Best Possible Solution

Project assistant-event media through the same managed outgoing image path used by WebChat replies, while preserving text buffering/throttling and adding live/final/history coverage for remote, data URL, and local media cases.

## Reproduction Assessment

Yes, from source. A chat-linked assistant event with `data.text` and `data.mediaUrls` on current main reaches the Gateway chat projection as text/delta only and produces text-only delta/final content.

## Solution Assessment

No. Carrying media is the right direction, but emitting raw image URLs is not the safest maintainable fix because the existing managed media contract handles limits, local-root checks, MIME validation, retention, and authenticated image serving.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Route media through managed image handling:** `src/gateway/server-chat.ts:214`
  - body: The new helper emits assistant `mediaUrls` directly as `{ type: "image", url }`. Assistant media can be data URLs or local paths, while the existing WebChat path routes reply media through `createManagedOutgoingImageBlocks` to enforce local-root access, image type/size limits, and `/api/chat/media/outgoing/...` serving; bypassing that here can leak raw local/data media in live/final chat payloads and misses the established retention path.
  - confidence: 0.87
