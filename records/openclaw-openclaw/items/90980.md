## What This Changes

The issue reports that a wedged Docker engine can leave Docker sandbox initialization unbounded, causing sandboxed agent runs to hang and making agents appear offline.

## Best Possible Solution

Land a bounded Docker sandbox control-plane timeout and explicit degradation behavior through the linked PR or a narrow replacement, with regression coverage and real wedged-Docker proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Docker child that never reaches `error` or `close` leaves `execDockerRaw` pending, and sandbox container init calls that helper without a deadline.

## Solution Assessment

Unclear as a complete fix. A Docker control-plane timeout is the right first layer, but the fail-soft behavior and any user-visible timeout config/default need maintainer review while handling the linked PR.
