## What This Changes

The issue reports that a present but unresponsive Docker engine can leave sandbox Docker calls pending indefinitely, causing sandboxed runs and perceived gateway availability to stall instead of failing with a bounded error.

## Best Possible Solution

Land one canonical sandbox Docker timeout/fail-soft fix with regression coverage and real wedged-engine proof, then close this issue from the merged PR once current main contains the bounded behavior.

## Reproduction Assessment

Yes by source inspection: current main awaits Docker sandbox initialization through helpers that have no default deadline, so a Docker child that never emits error or close can keep the promise pending. I did not run the live OrbStack/macOS wedged-engine scenario.

## Solution Assessment

Yes at the problem level: bounding Docker sandbox calls and failing soft is the maintainable direction. The current open PR still needs maintainer review on the exact timeout boundary, config surface, and proof before this issue can close.
