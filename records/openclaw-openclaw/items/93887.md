## Summary

Keep open: the PR targets a real SSH target parsing bug, but the proposed guard is placed after the explicit-port branch, so malformed inputs such as `host::22` or `:host:22` can still return a leading/trailing-colon host.

## What This Changes

The branch adds a `parseSshTarget` guard to reject host parts that start or end with `:` and adds regression tests for four malformed SSH targets.

## Best Possible Solution

Keep the parser-level fix, but validate the sliced host in the explicit-port branch too and cover those malformed cases in `src/infra/ssh-tunnel.test.ts`.

## Reproduction Assessment

Yes. Source inspection of the PR merge ref shows `host::22` enters the explicit-port branch, slices the host to `host:`, and returns before the new guard runs.

## Solution Assessment

No. The parser is the right layer, but this implementation is too narrow until both parse branches apply the same host-boundary validation.
