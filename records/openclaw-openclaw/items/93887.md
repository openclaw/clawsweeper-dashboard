## Summary

Keep open: current main and latest release still allow stray-colon SSH host strings through `parseSshTarget`, while this PR fixes the shared parser boundary with targeted regression coverage and sufficient live runtime proof. I found no blocking patch defects.

## What This Changes

The PR adds a shared malformed-host guard in `parseSshTarget` and tests SSH target host parts that start or end with stray colons.

## Best Possible Solution

Land the shared parser-level validation after normal required checks, keeping sandbox SSH and gateway status consumers on one target contract.

## Reproduction Assessment

Yes. Current main source shows `host:` and explicit-port sliced cases can produce parsed hosts with stray colons before those values flow into SSH config or argv; I did not run a fresh local runtime test in this read-only review.

## Solution Assessment

Yes. The shared parser is the best fix boundary because sandbox SSH, gateway status resolution, and auto-discovery all consume `parseSshTarget`; caller-specific filtering would duplicate the invariant.
