## Summary

Keep open: current main still allows stray leading or trailing colon text to survive as the parsed SSH host, while this PR fixes the shared parser boundary and adds focused regression coverage without introducing a concrete merge blocker.

## What This Changes

The branch adds shared SSH host validation in `parseSshTarget` for leading or trailing colon host parts and covers both parser branches with regression tests.

## Best Possible Solution

Land the shared parser-boundary validation with its regression coverage so malformed SSH targets are rejected before SSH config or argv construction.

## Reproduction Assessment

Yes, from source inspection. Current main leaves `host:` and `:22` style values inside the parsed host, and callers then use that parsed host for SSH config or tunnel startup.

## Solution Assessment

Yes. The parser is the shared validation boundary for sandbox SSH config generation, gateway status tunnel startup, and discovery filtering, so one guard there is cleaner than caller-specific rejection.
