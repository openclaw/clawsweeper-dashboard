## Summary

Keep open: the direction is useful, but the patch only honors `browser.enabled=false`; current docs and setup code also treat `plugins.entries.browser.enabled=false` as disabling browser support, and the PR has no after-fix real behavior proof yet.

## What This Changes

The PR guards browser tab cleanup in auto-reply session rollover and gateway `sessions.reset` when `browser.enabled` is false, with regression tests for both paths.

## Best Possible Solution

Use one disabled-browser predicate that covers both documented disable forms before reset cleanup calls, add focused coverage for both configs, and include redacted real-run proof for the disabled-browser reset path.

## Reproduction Assessment

Yes from source inspection: current main calls the browser maintenance facade from reset and rollover paths, and the facade loads the bundled browser surface for non-empty session keys. I did not run a live repro because this review is read-only.

## Solution Assessment

No: the proposed guard is the right shape for `browser.enabled=false`, but it does not cover the documented plugin-disable form and does not include real runtime proof. A shared disabled-browser check with focused tests is the narrower maintainable fix.
