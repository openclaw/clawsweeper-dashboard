## Summary

Keep this PR open: the bug and direction are valid, and the supplied real-runtime proof is strong, but the latest patch no longer honors `plugins.entries.browser.enabled=false` from the per-call reset config before falling through to the browser maintenance facade.

## What This Changes

The PR routes auto-reply rollover and gateway `sessions.reset` browser cleanup through a lifecycle helper, makes the browser maintenance SDK facade activation-aware, and adds focused disabled-browser regression coverage.

## Best Possible Solution

Keep the activation-aware facade, but add a cfg-based effective browser plugin gate in lifecycle cleanup so both `browser.enabled=false` and `plugins.entries.browser.enabled=false` skip reset cleanup, with reset-path tests and a build/import-topology gate.

## Reproduction Assessment

Yes, at source level: current main unconditionally invokes browser cleanup from auto-reply rollover and gateway reset, and the PR body includes real before/after runtime evidence for `browser.enabled=false`. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: the implementation direction is good, but the latest patch should use the passed config for both root and plugin-entry disablement instead of relying on the process-global facade activation check for `plugins.entries.browser.enabled=false`.
