## Summary

Keep this open: current main and v2026.6.6 still expose phase and timing facts for the relevant hook events, but not structured step or percent progress. The Feishu live-card validation is useful plugin-side evidence, but the missing upstream contract still needs maintainer API semantics before implementation.

## What This Changes

Request to add optional structured progress metrics to `before_tool_call`, `after_tool_call`, and `agent_end` hook payloads for plugin-rendered live progress cards.

## Best Possible Solution

Define a narrow optional producer-supplied progress contract with explicit semantics, wire it only from runtimes that have authoritative progress facts, and document that unknown totals, percentages, and ETA remain absent.

## Reproduction Assessment

Yes, at the source-contract level: current main and v2026.6.6 define the named hook payloads without the requested structured progress metrics. This is a missing capability rather than a broken existing documented behavior.

## Solution Assessment

No, not exactly as proposed: enriching hook payloads is the right area, but synthesizing totals or percentages from ordinary tool-call lifecycle events is not a reliable source of truth. The best path is an explicit producer-owned progress contract with absent values when progress is unknown.
