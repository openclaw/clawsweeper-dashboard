## Summary

Keep open. Current main still exposes typed plugin hooks for `before_tool_call`, `after_tool_call`, and `agent_end`, but the source contracts, runtime emitters, tests, and public docs do not provide the requested structured progress `metrics` object. The report is recent, concrete, and describes a real plugin API gap rather than stale support noise or ClawHub-only work.

## What This Changes

The issue requests optional structured progress metrics on plugin hook events so plugin-rendered live progress cards can show `x/y` or percentages instead of only textual phase updates.

## Best Possible Solution

Keep the issue open for an additive plugin API design. The best end state is either an optional `metrics` object on the existing hook events or a dedicated progress hook, backed by a monotonic step model that never reports successful `100%` on failed or cancelled turns, with typed contracts, Pi and harness runtime emitters, regression tests, and docs updated together.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to inspect current main hook types, emitters, and docs: none expose or emit the requested metrics, and targeted search only finds unrelated update-runner step fields.

## Solution Assessment

Unclear. Adding optional metrics to existing hooks is backward-compatible and close to the requested minimal change, but a maintainer should decide whether existing hook events or a dedicated progress hook is the right contract before encoding step-count semantics.
