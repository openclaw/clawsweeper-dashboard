## Summary

Keep open: current main still does not expose structured progress metrics on the requested hook events, and adding them would create a new public plugin SDK contract that needs maintainer/API design before implementation.

## What This Changes

Expose optional `metrics` fields such as `stepIndex`, `stepTotal`, `percent`, and `etaSec` on `before_tool_call`, `after_tool_call`, and `agent_end` hook payloads for plugin-rendered live progress cards.

## Best Possible Solution

Define and document a stable optional plugin hook progress metrics contract, then emit it consistently from all shared tool and lifecycle hook paths with explicit monotonic and terminal-state semantics.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing contract. Source inspection confirms current main lacks the requested metrics fields on the relevant hook types and emitters.

## Solution Assessment

Unclear. The proposed optional shape is plausibly backward-compatible, but the best implementation needs maintainer agreement on public SDK naming and progress semantics before changing hook payload contracts.
