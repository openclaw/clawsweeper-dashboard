## Summary

Keep open: current main and the latest shipped tag still lack the requested structured numeric progress metrics on plugin hook payloads, and the remaining work is a public plugin API/progress-semantics decision rather than cleanup closure.

## What This Changes

The issue requests optional structured numeric progress metrics on `before_tool_call`, `after_tool_call`, and `agent_end` hook payloads so live progress-card plugins can render percent or step counts.

## Best Possible Solution

Define one documented optional numeric progress metrics contract for plugin hooks, then emit it consistently through the tool and agent-end paths with focused contract, emitter, docs, and integration coverage.

## Reproduction Assessment

No high-confidence runtime bug reproduction is needed because this is a feature/API request. Source inspection gives high confidence that current main and `v2026.5.22` lack the requested metric fields on the relevant hook types and emitters.

## Solution Assessment

Unclear: enriching existing hook payloads is plausible and backward-compatible, but maintainers should first settle numeric progress semantics across planning, retries, parallel tools, failures, cancellation, and terminal state.
