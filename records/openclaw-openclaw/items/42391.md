## Summary

Keep open: current main and the latest shipped release still do not expose structured numeric progress metrics on the relevant plugin hook payloads, and the remaining work needs a public plugin API/progress-semantics decision rather than cleanup closure.

## What This Changes

The issue requests optional `metrics` fields such as step index, step total, percent, and ETA on `before_tool_call`, `after_tool_call`, and `agent_end` hook events for plugin-rendered live progress cards.

## Best Possible Solution

Define one documented optional numeric progress metrics contract for plugin hooks, then emit it consistently through the tool and agent-end paths with focused contract, emitter, docs, and integration coverage.

## Reproduction Assessment

Yes at source level: current main's hook event types and emitters lack the requested metric fields, and `git grep` finds no requested metric names in the relevant hook files. It is not a runtime bug reproduction because this is a new API/feature request.

## Solution Assessment

Unclear: enriching existing hook payloads is plausible and backward-compatible, but maintainers should first settle numeric progress semantics across planning, retries, parallel tools, failures, cancellation, and terminal state.
