## Summary

Keep open: current main and v2026.6.8 still expose only identity, result, error, and duration fields on the requested plugin hook events. The remaining work is a public Plugin SDK progress-semantics decision, not an already implemented or clearly duplicated bug fix.

## What This Changes

Request to add optional structured progress metrics to plugin hook payloads so plugins can render numeric live progress cards.

## Best Possible Solution

Define a narrow optional plugin-facing progress metric contract sourced from authoritative producer events, update the hook types/producers/docs/tests together, and omit unknown values instead of guessing from generic tool-call counts.

## Reproduction Assessment

Yes, at source-contract level: current main defines the requested hook event types and producers without structured progress metric fields. A live Feishu card run is not required to verify that the public API field is missing.

## Solution Assessment

Unclear as proposed: enriching hook payloads is plausible, but the exact step/percent semantics need maintainer agreement because generic tool lifecycle hooks are not authoritative for totals. The safer solution is producer-owned metrics with absent values when progress is unknown.
