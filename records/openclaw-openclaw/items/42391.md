## Summary

Keep open: current main and the latest release still lack structured `metrics` on the requested plugin hook events. The remaining work is a public Plugin SDK progress-semantics decision, not stale cleanup or an already-implemented fix.

## What This Changes

Request to add optional structured progress metrics to `before_tool_call`, `after_tool_call`, and `agent_end` hook events so plugin live progress cards can render numeric progress.

## Best Possible Solution

Define a narrow optional plugin-facing progress metric contract sourced from authoritative run/progress producers, update hook types, producers, docs, and tests together, and omit unknown values instead of guessing.

## Reproduction Assessment

Yes, at source-contract level: current main defines and produces the requested hook events without structured progress metric fields. A live Feishu card run is not required to verify that the public API field is missing.

## Solution Assessment

Unclear as proposed: enriching hook payloads is plausible, but exact step/percent semantics need maintainer agreement because generic tool lifecycle hooks are not authoritative for totals. The safer path is producer-owned metrics with absent values when progress is unknown.
