## Summary

Keep open: current main still lacks the requested structured numeric progress metrics on plugin hook payloads, and the remaining work is a public plugin API and progress-semantics design decision rather than cleanup.

## What This Changes

The issue requests optional numeric progress metrics on `before_tool_call`, `after_tool_call`, and `agent_end` hook payloads so plugins can render live progress cards.

## Best Possible Solution

Define one documented optional numeric progress metrics contract for plugin hooks, then emit it consistently through tool and agent-end paths with focused contract and integration coverage.

## Reproduction Assessment

No high-confidence runtime bug reproduction is needed because this is a feature/API request. The absence is source-reproducible: current main and `v2026.5.22` lack the requested metrics fields on the hook types and emitters.

## Solution Assessment

Unclear until maintainers settle the public progress semantics. Enriching existing hook payloads is plausible and backward-compatible, but the contract needs decisions for planning, retries, parallel tools, failures, cancellation, and terminal state.
