## Summary

Keep open: current main still lacks structured numeric progress metrics on the requested plugin hook payloads, and this is a public plugin API/product-semantics decision rather than conservative cleanup.

## What This Changes

Request to add optional `metrics` fields such as `stepIndex`, `stepTotal`, `percent`, and `etaSec` to `before_tool_call`, `after_tool_call`, and `agent_end` hook events for plugin live progress cards.

## Best Possible Solution

Define one documented optional progress metrics contract for plugin hooks, then emit it consistently through the relevant tool and agent-end paths after maintainers settle the semantics.

## Reproduction Assessment

Yes at the source-contract level, but not as a broken-behavior bug. Current main and `v2026.5.20` both omit the requested metric fields from the relevant hook event types and emitters.

## Solution Assessment

Unclear: enriching existing hook payloads is plausible and backward-compatible for ignoring consumers, but maintainers need to decide numeric progress semantics before implementation.
