## Summary

Keep open: current main and v2026.6.10 still define and emit the requested hook events without numeric progress metrics. The remaining work is a public Plugin SDK progress-semantics decision, so stale cleanup would be premature.

## What This Changes

The issue requests optional structured progress metrics on existing plugin hook events so live progress-card plugins can render step counts, percentages, or ETA.

## Best Possible Solution

Define a narrow optional plugin-facing progress metric contract from authoritative run/progress producers, update hook types, producers, docs, and tests together, and omit unknown values instead of guessing percentages.

## Reproduction Assessment

Yes, at source-contract level: current main defines and emits the requested hook events without structured progress metric fields. A live Feishu card run is not needed to verify the missing public API field.

## Solution Assessment

Unclear as proposed: enriching hook payloads is plausible, but exact step, percent, and ETA semantics need maintainer agreement because generic tool lifecycle events are not authoritative for totals. The safer implementation is producer-owned metrics with absent values when progress is unknown.

## Risks / Open Questions

- Generic tool-call counts are not authoritative progress totals for dynamic, retried, cancelled, failed, or branched runs, so naive percentages could mislead plugin UIs.
- Adding metrics to Plugin SDK hook payloads creates a public compatibility contract across embedded, CLI, provider, and channel-backed run paths; maintainers need to choose producer ownership and omission semantics.
