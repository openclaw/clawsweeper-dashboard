## Summary

Keep this open: current main and v2026.6.6 still do not expose structured step or percent metrics on the requested plugin hook events, and no canonical duplicate or merged fix now owns the work. The request fits the plugin API direction, but it needs maintainer agreement on authoritative progress semantics before implementation.

## What This Changes

The issue requests optional structured progress metrics on existing plugin hook events so plugin-based live progress cards can render x/y or percent completion.

## Best Possible Solution

Define a narrow optional plugin-facing progress metric contract sourced from authoritative producer events, with unknown values omitted rather than guessed.

## Reproduction Assessment

Yes, at source-contract level: current main defines the requested hook event types and producers without structured progress metric fields. A live Feishu card is not required to verify that the public API field is missing.

## Solution Assessment

No, not exactly as proposed: enriching hook payloads is plausible, but generic tool-call lifecycle hooks are not authoritative enough for totals or percentages. The safer path is producer-owned metrics with omitted values when progress is unknown.

- Structured progress semantics are not just a field addition: generic tool-call counts can mislead plugins when runs retry, branch, cancel, fail, or discover work dynamically.
- Adding a public Plugin SDK metrics field before defining the authoritative producer boundary could leave plugins with inconsistent progress behavior across embedded, Codex, CLI, and channel-backed run paths.
