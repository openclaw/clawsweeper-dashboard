## Summary

Keep this open: current main and v2026.6.6 still lack structured step or percent metrics in the public hook payloads. The Feishu live-card context is useful evidence of plugin demand, but this needs a maintainer API/product decision for producer semantics before implementation.

## What This Changes

Add an optional structured `metrics` object to `agent_end`, `before_tool_call`, and `after_tool_call` hook payloads so plugins can render numeric live progress cards.

## Best Possible Solution

Define a narrow optional progress metrics contract in the public Plugin SDK, backed only by runtimes with authoritative progress facts, with docs and tests specifying monotonicity, success/failure terminal behavior, and absent values when progress is unknown.

## Reproduction Assessment

Yes, at source-contract level: current main defines the relevant public hook event types and producers without structured metrics, and v2026.6.6 shows the same release surface. I did not run a live Feishu card because this is a missing API capability rather than broken existing documented behavior.

## Solution Assessment

No, not exactly as proposed: enriching hook payloads is plausible, but generic tool-call lifecycle events are not authoritative enough for totals or percentages. The safer path is producer-owned metrics with absent values when progress is unknown.
