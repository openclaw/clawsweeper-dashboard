## Summary

Keep open: current main and the latest release still expose Microsoft Teams as one singleton bot/app credential set, while the requested multi-bot Teams identity model would add config, auth, routing, setup, and migration surface that needs maintainer design.

## What This Changes

Request Microsoft Teams support for multiple Azure/Bot Framework bot identities on one OpenClaw Gateway, routable to different agents through account bindings.

## Best Possible Solution

Design a named channels.msteams.accounts model that preserves the current singleton as the default account, routes through bindings[].match.accountId, and isolates each bot identity's credentials, webhook, and token state with a doctor/setup migration.

## Reproduction Assessment

Yes for the source-level limitation: current main has a strict singleton Teams config schema, a top-level single-account adapter, singleton setup writes, and one monitor credential path. I did not run a live Teams bot because the missing account schema and adapter path are directly visible in source.

## Solution Assessment

No for the issue's exact proposed array plus agentBinding shape; the better OpenClaw-shaped solution is named channels.msteams.accounts wired through existing accountId bindings. The current requested direction is useful, but it needs maintainer product/API design before implementation.
