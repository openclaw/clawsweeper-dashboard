## Summary

Keep open. Current main still models Microsoft Teams as one top-level bot credential/default account, while the issue discussion and the closed duplicate at https://github.com/openclaw/openclaw/issues/72876 show a valid remaining multi-bot identity gap; the open tracker at https://github.com/openclaw/openclaw/pull/74163 still lists this as unresolved.

## What This Changes

Request support for multiple Azure/Microsoft Teams bot identities on one OpenClaw gateway, with routing to different agents.

## Best Possible Solution

Implement this in the bundled Microsoft Teams plugin as channels.msteams.accounts plus defaultAccount, preserving the current top-level config as the implicit default and keeping agent selection in bindings[].match.accountId.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a runtime bug, but yes from source inspection: current config/schema/setup/runtime code only exposes one Teams account while multi-agent routing depends on accountId for per-agent identity.

## Solution Assessment

Is this the best way to solve the issue? No for the exact proposed array plus agentBinding shape; the maintainable path is the existing account-map/default-account contract with routing left in the bindings system.

## Security Review

Status: needs_attention

Summary: The feature is security-sensitive because it changes Teams credential selection, webhook routing, JWT validation, and token storage isolation.

Concerns:

- **[medium] Tie account routing to verified bot identity:** `extensions/msteams/src/monitor.ts:302`
  - body: The current runtime builds one Bot Framework JWT validator from one credential set. Multi-account support must validate the incoming token against the selected bot app identity before dispatching under an accountId.
  - confidence: 0.78
- **[medium] Scope Teams delegated and SSO tokens by account:** `extensions/msteams/src/token.ts:145`
  - body: Delegated tokens use one msteams-delegated.json store, and SSO tokens are keyed by connectionName and userId without an account or tenant dimension. Multi-account support should isolate these stores per account/tenant.
  - confidence: 0.72
