## Summary

Keep open. Current main still exposes Microsoft Teams as one top-level bot credential/default account, and the related duplicate plus open Microsoft tracker show the multi-bot Teams identity request remains unresolved. https://github.com/openclaw/openclaw/issues/71058

## What This Changes

Support multiple Azure/Teams bot configurations on one OpenClaw Gateway so distinct Teams bot identities can bind to different agents.

## Best Possible Solution

Implement this inside the bundled Microsoft Teams plugin with channels.msteams.accounts plus defaultAccount, preserve top-level config as the backward-compatible implicit default, scope credentials/webhooks/policies/token stores per account, carry accountId through inbound/outbound/status/setup/docs, and leave agent routing in bindings[].match.accountId.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for source-level verification: current main exposes one msteams account and setup/runtime paths collapse to DEFAULT_ACCOUNT_ID.

## Solution Assessment

Is this the best way to solve the issue? No for the exact proposed array plus agentBinding shape; the maintainable direction is the existing channel accounts/defaultAccount contract with agent selection left in bindings.

## Security Review

Status: needs_attention

Summary: No PR diff is under review, but the requested feature is security-sensitive because it changes Teams credential selection, webhook authentication, routing, and token isolation.

Concerns:

- **[medium] Tie account routing to verified bot identity:** `extensions/msteams/src/monitor.ts:286`
  - body: The current webhook builds one Bot Framework JWT validator from one credential set. Multi-account support must select and validate the account against the incoming bot identity before dispatching under an accountId.
  - confidence: 0.78
- **[medium] Scope Teams delegated tokens by account:** `extensions/msteams/src/token.ts:144`
  - body: Delegated OAuth currently uses one msteams-delegated.json store, and SSO tokens are keyed by connection name and user ID rather than account or tenant. Multi-bot support needs account/tenant partitioning.
  - confidence: 0.72
