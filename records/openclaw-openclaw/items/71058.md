## Summary

Keep open. Current main still models Microsoft Teams as a single top-level bot credential/default account, while the discussion and duplicate #72876 describe a valid remaining multi-bot identity gap; the open #74163 tracker lists this as unresolved rather than implementing it. https://github.com/openclaw/openclaw/issues/71058

## What This Changes

Request support for multiple Microsoft Teams/Azure bot identities on one OpenClaw Gateway, with each identity routeable to a specific agent.

## Best Possible Solution

Implement this in the bundled Microsoft Teams plugin using `channels.msteams.accounts` plus `defaultAccount`, preserve today’s top-level config as the backward-compatible implicit default account, account-scope credentials/webhook/policies/Graph and token stores, carry `accountId` through inbound routing/outbound replies/status/setup/docs, and keep agent selection in `bindings[].match.accountId`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for source-level verification: current config/schema/setup/runtime code only exposes one Teams account while multi-agent routing depends on account IDs for per-agent identity.

## Solution Assessment

Is this the best way to solve the issue? No for the exact proposed array plus `agentBinding` shape; the safer maintainable path is the existing account-map/default-account contract with agent routing left in the bindings system.

## Security Review

Status: needs_attention

Summary: No PR diff is under review, but this feature is security-sensitive because it changes Teams credential selection, webhook routing, JWT validation, and token storage isolation.

Concerns:

- **[medium] Tie account routing to verified bot identity:** `extensions/msteams/src/monitor.ts:286`
  - body: The current runtime builds one Bot Framework JWT validator from one credential set. Multi-account support must validate the incoming token against the selected bot app identity before dispatching under an `accountId`, or one configured Teams bot could be routed as another.
  - confidence: 0.78
- **[medium] Scope Teams delegated tokens by account:** `extensions/msteams/src/token.ts:144`
  - body: Delegated OAuth tokens currently use one `msteams-delegated.json` store, and SSO tokens are keyed by connection name and user ID without an account or tenant dimension. Multi-account Teams support should scope these stores by account and tenant to avoid credential reuse across bot identities.
  - confidence: 0.72
