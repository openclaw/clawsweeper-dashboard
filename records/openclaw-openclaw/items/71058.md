## Summary

Keep open. Current main still models Microsoft Teams as one top-level bot credential/default account, so the requested multi-bot identity support is not implemented; the related closed duplicate and open Microsoft tracker do not replace this feature-level report.

## What This Changes

Add support for multiple Microsoft Teams/Azure Bot Framework bot identities on one OpenClaw Gateway, with each bot account routable to a specific agent.

## Best Possible Solution

Implement this in the bundled Microsoft Teams plugin using the existing `channels.<channel>.accounts` plus `defaultAccount` pattern, preserve today’s top-level config as the backward-compatible implicit default, account-scope credentials/webhook/allowlists/Graph and token stores, carry `accountId` through inbound routing/outbound replies/status/setup/docs, and keep agent selection in the existing `bindings` system.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from current-main source and docs: Teams exposes one configured/default account while the multi-agent account-map pattern is implemented and documented for other channels.

## Solution Assessment

No for the exact proposed array plus `agentBinding` shape. The safer maintainable direction is the existing account-map/default-account contract plus a Teams-specific account-selection design reviewed for Bot Framework auth isolation.

## Security Review

Status: needs_attention

Summary: No PR diff is under review, but the requested feature changes Teams credential selection and webhook/account isolation, so it needs security design review before implementation.

Concerns:

- **[medium] Tie account routing to verified bot identity:** `extensions/msteams/src/monitor.ts:286`
  - body: The current runtime builds one JWT validator from one credential set. Multi-account Teams support must validate the Bot Framework token against the selected bot app identity before dispatching under an `accountId`, or one configured bot could be routed as another.
  - confidence: 0.77
- **[medium] Account-scope delegated token storage:** `extensions/msteams/src/token.ts:144`
  - body: Teams delegated tokens currently use one store filename. A multi-account implementation should scope delegated/SSO token storage by account and tenant so credentials from one bot identity cannot be reused by another.
  - confidence: 0.68
