## Summary

Keep open. Current main still implements Microsoft Teams as a singleton channel account, while this issue is the canonical open request for multiple Teams bot identities on one gateway; the duplicate at https://github.com/openclaw/openclaw/issues/72876 was closed in favor of this issue and the Microsoft tracker at https://github.com/openclaw/openclaw/pull/74163 still lists it as remaining work.

## What This Changes

Request multi-account Microsoft Teams/Azure bot configuration on one gateway so different Teams bot identities can route to different agents.

## Best Possible Solution

Add first-class channels.msteams.accounts/defaultAccount support that promotes legacy top-level config to the default account, routes through bindings[].match.accountId, and isolates credentials, validators, webhook paths, Graph tokens, SSO, and delegated token state per account.

## Reproduction Assessment

Yes. Source inspection shows current main exposes only one strict top-level channels.msteams credential object and the Teams setup/config adapters resolve only the default account.

## Solution Assessment

No for the exact proposed array plus agentBinding shape. The maintainable path is the existing account-map/defaultAccount contract with agent routing left in bindings[].match.accountId.
