## Summary

Keep open. Current main still exposes Microsoft Teams as a singleton channel account while the issue and related discussion ask for multiple Teams bot identities on one gateway, and the open Microsoft tracker treats this as remaining Teams-channel work rather than resolved cleanup.

## What This Changes

The issue requests multiple Microsoft Teams bot configurations on one OpenClaw gateway so different agents can present as separate Teams bot identities.

## Best Possible Solution

Add first-class msteams account-map support that preserves the existing singleton config as the default account and routes agents through the established bindings[].match.accountId contract.

## Reproduction Assessment

Yes, from source inspection rather than a failing runtime bug: current main accepts one top-level Teams credential set and the msteams adapter/setup paths resolve only the default account.

## Solution Assessment

No for the exact proposed array and agentBinding shape; the narrower maintainable direction is the existing account-map/defaultAccount contract with agent selection left in bindings.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because multi-bot Teams support would change credential selection, Bot Framework JWT validation, webhook routing, and delegated token isolation.

- **[medium] Tie account routing to verified bot identity:** `extensions/msteams/src/monitor.ts:304`
  - body: Current startup builds one Bot Framework JWT validator from one credential set; multi-account support must validate inbound tokens against the selected bot app before dispatching under an accountId.
  - confidence: 0.84

  - body: Delegated tokens currently use one shared filename, and SSO token storage is keyed by connectionName and userId rather than account, so multi-account support needs account/tenant isolation.
