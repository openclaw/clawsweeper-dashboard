## Summary

Keep open. Current `main` still treats Microsoft Teams as one top-level Bot Framework credential/default account, while the issue discussion, closed duplicate, and Microsoft tracker show this remains the canonical multi-bot Teams identity request rather than cleanup-closeable noise.

## What This Changes

Request support for multiple Microsoft Teams bot identities on one OpenClaw gateway, routed to different agents instead of sharing one Teams bot identity.

## Best Possible Solution

Add bundled `msteams` multi-account support using `channels.msteams.accounts` and `defaultAccount`, preserve the existing top-level config as the implicit default account, and keep agent selection in `bindings[].match.accountId`.

## Reproduction Assessment

Yes, from source inspection rather than a runtime bug: current `main` only exposes Teams as a single top-level credential/default account while multi-agent routing depends on `accountId` for per-agent identity.

## Solution Assessment

No for the exact proposed array plus `agentBinding` shape; the narrower maintainable path is to extend the existing account-map/default-account contract and leave agent routing in bindings.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it changes Teams credential selection, Bot Framework JWT validation, webhook routing, and delegated/SSO token isolation.

Concerns:

- **[medium] Tie account routing to verified bot identity:** `extensions/msteams/src/monitor.ts:304`
  - body: The current runtime builds one Bot Framework JWT validator from one credential set. Multi-account support must validate the incoming token against the selected bot app identity before dispatching under an accountId.
  - confidence: 0.84
- **[medium] Scope Teams delegated and SSO tokens by account:** `extensions/msteams/src/token.ts:145`
  - body: Delegated tokens currently use one shared Teams token filename, and SSO setup is driven from the singleton msteams config. Multi-account support should isolate these stores per account and tenant.
  - confidence: 0.78
- **[medium] Avoid default-account pairing leakage:** `extensions/msteams/src/monitor-handler/access.ts:55`
  - body: The inbound access controller currently creates pairing state with `DEFAULT_ACCOUNT_ID`; multi-account Teams must pass the verified account id into pairing and allowlist resolution.
  - confidence: 0.74
