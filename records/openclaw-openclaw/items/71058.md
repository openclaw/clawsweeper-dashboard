## Summary

Keep open: current main and v2026.6.8 still expose Microsoft Teams as one singleton bot/app credential set, while the requested multi-bot Teams identity model would add a new config/auth/routing surface that needs maintainer design before implementation.

## What This Changes

The issue requests support for multiple Microsoft Teams/Azure bot identities on one OpenClaw Gateway, routed to different agents by account identity.

## Best Possible Solution

Add a maintainer-approved Teams account model using named channels.msteams.accounts entries, explicit default-account behavior, bindings[].match.accountId routing, and doctor/setup/token migrations that preserve the current singleton config as the default account.

## Reproduction Assessment

Yes. Source inspection on current main shows Teams config, setup, monitor, sender access, and credential resolution all use one top-level channels.msteams credential set and DEFAULT_ACCOUNT_ID, so multiple Teams bot identities cannot be configured today.

## Solution Assessment

No for the exact proposed array plus agentBinding shape. The best OpenClaw-shaped solution is a named channels.msteams.accounts model wired through existing accountId bindings, with explicit migration and token/credential isolation.
