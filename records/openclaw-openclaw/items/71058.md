## Summary

Keep open. Current main still models Microsoft Teams as one configured/default bot account, while the issue asks for multiple Teams bot identities on one gateway. The older duplicate is closed and the open Microsoft tracker is broad planning, not an implementation PR, so this issue remains a valid feature/design item.

## What This Changes

Add support for multiple Microsoft Teams bot/app identities on one OpenClaw gateway, with routing to different agents while preserving the existing single-object configuration.

## Best Possible Solution

Implement this in the bundled Microsoft Teams plugin using the existing channels.<channel>.accounts plus defaultAccount pattern, preserve today’s top-level channels.msteams config as the backward-compatible implicit default account, account-scope credentials/webhooks/allowlists/Graph and delegated token stores, carry accountId through inbound routing/outbound replies/status/setup/docs, and keep agent selection in the existing bindings system.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from current-main source and docs: Teams exposes one configured/default account while the shared multi-agent docs reserve accountId routing for channels with accounts/defaultAccount support.

## Solution Assessment

No for the exact proposed array plus agentBinding shape. The maintainable direction is the existing account-map/default-account contract plus a Teams-specific security design for Bot Framework credential selection and account isolation.

## Security Review

Status: needs_attention

Summary: No PR diff is under review, but the requested feature changes Teams credential selection and account isolation, so it needs security design review before implementation.

Concerns:

- **[medium] Tie account routing to verified bot identity:** `extensions/msteams/src/monitor.ts:286`
  - body: The current runtime builds one JWT validator from one credential set. Multi-account Teams support must validate the Bot Framework token against the selected bot app identity before dispatching under an accountId, or one configured bot could be routed as another.
  - confidence: 0.78
- **[medium] Scope delegated and SSO token stores by account:** `extensions/msteams/src/token.ts:144`
  - body: Teams delegated and SSO token storage is currently channel-wide or keyed by connection/user, not by bot account and tenant. A multi-account implementation should isolate these stores so tokens from one bot identity cannot be reused by another.
  - confidence: 0.72
