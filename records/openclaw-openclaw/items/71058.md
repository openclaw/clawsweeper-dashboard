## Summary

Keep open. Current main still exposes Microsoft Teams as one top-level bot credential/default account, while this issue, duplicate #72876, and tracker #74163 describe an unresolved multi-bot identity gap for single-gateway deployments. https://github.com/openclaw/openclaw/issues/71058

## What This Changes

Request to let one OpenClaw Gateway host multiple Microsoft Teams/Azure bot identities and bind them to separate agents.

## Best Possible Solution

Implement this in the bundled Microsoft Teams plugin with channels.msteams.accounts plus defaultAccount, keep today’s top-level config as the backward-compatible implicit default, and route agents through bindings[].match.accountId rather than a new agentBinding field.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source-level verification: current config, schema, setup, docs, and runtime code expose only one Teams account while multi-agent routing already depends on accountId for per-agent identity.

## Solution Assessment

Is this the best way to solve the issue? No for the exact proposed array plus agentBinding shape; the maintainable path is the existing account-map/default-account contract with agent selection left in the bindings system.

## Security Review

Status: needs_attention

Summary: No patch is under review, but the requested multi-bot Teams feature is security-sensitive because it changes credential selection, webhook identity routing, and token isolation.

Concerns:

- **[medium] Tie account routing to verified bot identity:** `extensions/msteams/src/monitor.ts:286`
  - body: Multi-account support must validate the incoming Bot Framework token against the selected bot app identity before dispatching under an accountId, or one configured Teams bot could be routed as another.
  - confidence: 0.78
- **[medium] Scope Teams token stores by account and tenant:** `extensions/msteams/src/token.ts:144`
  - body: Delegated and SSO token stores are currently single-channel stores keyed without an account dimension, so multi-account support should isolate tokens by account and tenant to avoid credential reuse across bot identities.
  - confidence: 0.72
