## Summary

Keep open. Current main and the latest release still model Microsoft Teams as one default-account credential set, the related duplicate https://github.com/openclaw/openclaw/issues/72876 was closed back to this issue, and no linked PR closes it; the remaining work is a valid Teams multi-account feature that needs maintainer design.

## What This Changes

Request Teams support for multiple Azure/Bot Framework app identities on one OpenClaw gateway, routed to different agents instead of one shared `channels.msteams` bot identity.

## Best Possible Solution

Add first-class `channels.msteams.accounts` support that preserves the existing singleton config as the implicit default account and routes agents through `bindings[].match.accountId` with account-scoped credential and token isolation.

## Reproduction Assessment

Yes, source-level. Current main exposes one top-level Teams credential set, the strict schema lacks an account map, and the Teams setup/channel adapter resolves only the default account; this is a feature gap rather than a failing runtime bug.

## Solution Assessment

No for the exact proposed array and `agentBinding` shape. The maintainable direction is to extend the existing account-map/default-account model and keep agent selection in `bindings[].match.accountId`.
