## Summary

Keep open. Current main still treats Microsoft Teams as one Bot Framework credential/default account, and the closed duplicate plus the open Microsoft tracker confirm this remains the canonical multi-bot Teams identity request rather than cleanup-closeable noise.

## What This Changes

Add support for multiple Microsoft Teams/Azure bot identities on one OpenClaw gateway so different agents can route through distinct Teams apps.

## Best Possible Solution

Add bundled msteams multi-account support using channels.msteams.accounts and defaultAccount while preserving the current top-level config as the implicit default and keeping agent routing in bindings[].match.accountId.

## Reproduction Assessment

Yes from source inspection, not as a runtime bug: current main only exposes Teams as a single top-level credential/default account while multi-agent routing depends on accountId for per-agent identity.

## Solution Assessment

No for the exact proposed array plus agentBinding shape; the narrower maintainable direction is the existing account-map/defaultAccount contract with agent selection left in bindings.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would change Teams credential selection, Bot Framework JWT validation, webhook routing, and OAuth/SSO token isolation.

Concerns:

- **[medium] Tie account routing to verified bot identity:** `extensions/msteams/src/monitor.ts:304`
  - body: The current runtime builds one Bot Framework JWT validator from one credential set. Multi-account support must validate the incoming token against the selected bot app identity before dispatching under an accountId.
  - confidence: 0.82
- **[medium] Scope Teams delegated and SSO tokens by account:** `extensions/msteams/src/token.ts:145`
  - body: Delegated tokens currently use one shared filename, and SSO setup is driven from the singleton msteams config. Multi-account support should isolate these stores per account and tenant.
  - confidence: 0.76
