## Summary

Keep open: current main and the latest release still expose Microsoft Teams as one singleton bot credential/webhook path with the default account only, and the related search found only a closed duplicate plus an adjacent tracker, not an implementation owner.

## What This Changes

The issue requests multiple Microsoft Teams/Azure bot identities on one OpenClaw Gateway so different Teams apps can route to different agents.

## Best Possible Solution

Design a Teams-owned channels.msteams.accounts.<id> model with default-account migration, account-scoped credentials/webhooks/auth state, inbound accountId routing, setup/docs, and focused upgrade tests.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main and v2026.6.10 expose only singleton channels.msteams credentials/webhook config and default-account inbound routing.

## Solution Assessment

No for the exact array plus agentBinding sketch. The maintainable OpenClaw-shaped solution is a named channels.msteams.accounts.<id> design wired through existing accountId bindings, with maintainer approval for webhook and migration details.
