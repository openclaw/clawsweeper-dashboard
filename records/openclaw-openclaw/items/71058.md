## Summary

Keep open. Current main still keeps Microsoft Teams on the top-level/default account adapter, while the request asks for multiple Teams bot identities on one gateway; the related closed duplicate points back to the same remaining feature rather than a shipped implementation.

## What This Changes

Request support for multiple Microsoft Teams/Azure bot configurations on one OpenClaw gateway, with routing to distinct agents instead of one shared Teams bot identity.

## Best Possible Solution

Add first-class msteams account-map support that preserves the existing singleton config as the implicit default account and routes agents through the established bindings[].match.accountId contract.

## Reproduction Assessment

Yes, from source inspection rather than a failing runtime bug: current main accepts one top-level Teams credential set and resolves the msteams channel as the default account only.

## Solution Assessment

No for the exact proposed array and agentBinding shape; the maintainable direction is to use the existing channel account-map/default-account model and leave agent selection in bindings[].match.accountId.
