## Summary

Keep open. Current main and the latest shipped release still expose Microsoft Teams as a singleton bot/app account, while the issue discussion and the closed duplicate show this is the canonical request for first-class multi-account Teams support.

## What This Changes

Request to let one OpenClaw gateway run multiple Microsoft Teams/Azure bot identities and bind those accounts to different agents.

## Best Possible Solution

Add first-class `channels.msteams.accounts` and `defaultAccount` support, preserve the current singleton config as the default account, route agents through `bindings[].match.accountId`, and account-scope Teams credential validation plus delegated/SSO token storage.

## Reproduction Assessment

Yes. Source inspection shows current main still accepts one strict top-level `channels.msteams` credential object and the adapter/setup paths resolve only the default account, so the requested multi-bot shape is not available.

## Solution Assessment

No for the exact proposed array and `agentBinding` shape. The maintainable direction is the existing account-map/default-account contract with agent routing left in `bindings[].match.accountId`.
