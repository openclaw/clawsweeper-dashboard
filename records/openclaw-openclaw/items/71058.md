## Label Justifications

- `impact:auth-provider`: The requested feature changes how Teams Azure bot app credentials, tenant settings, and delegated tokens are selected per bot identity.

## Summary

Keep open: current main and v2026.6.8 still expose Microsoft Teams as one singleton bot/app credential set, while the requested multi-bot Teams identity model would add a new config/auth/routing surface that needs maintainer design before implementation.

## What This Changes

Request support for multiple Microsoft Teams/Azure bot identities on one OpenClaw gateway with per-agent routing.

## Best Possible Solution

Add a maintainer-approved named Teams account model using channels.msteams.accounts, explicit default-account behavior, bindings[].match.accountId routing, and doctor/setup/token migrations that preserve the current singleton config as the default account.

## Reproduction Assessment

Yes for the source-level limitation. Current main shows Teams config, setup, credential resolution, and channel account status all use one top-level channels.msteams credential set and DEFAULT_ACCOUNT_ID.

## Solution Assessment

No for the exact proposed array plus agentBinding shape. The better OpenClaw-shaped solution is a named channels.msteams.accounts model wired through existing accountId bindings, with explicit singleton migration and token/credential isolation.
