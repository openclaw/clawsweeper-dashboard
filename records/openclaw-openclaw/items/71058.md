## Summary

Keep open: current main and latest release v2026.6.8 still expose Microsoft Teams as one singleton bot/app credential set, while the requested multi-bot Teams identity model requires config, auth, routing, setup, and migration design.

## What This Changes

Support multiple Microsoft Teams/Azure bot identities on one OpenClaw gateway and bind each bot account to the appropriate agent.

## Best Possible Solution

Design and implement a Teams-owned `channels.msteams.accounts.<id>` model with default-account migration, per-account credentials/webhooks/serviceUrl/auth state, bindings, setup/docs, and focused tests.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.8 have a strict singleton Teams config schema, a default-account-only adapter/setup path, and one monitor credential path; no live Teams bot was run in this read-only review.

## Solution Assessment

No for the issue's exact array plus `agentBinding` sketch; the better OpenClaw-shaped path is named `channels.msteams.accounts` wired through existing `accountId` bindings, but maintainers still need to choose the API and migration details.
