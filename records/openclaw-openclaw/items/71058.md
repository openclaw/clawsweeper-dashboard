## Summary

Keep open: current main and the latest release still expose Microsoft Teams as a single bot credential/webhook configuration, and the active multi-account implementation is still an open conflicting PR that needs maintainer review.

## What This Changes

Request Microsoft Teams support for multiple Azure bot identities/accounts on one OpenClaw Gateway, routed to different agents.

## Best Possible Solution

Land a reviewed Teams accounts/defaultAccount implementation in the msteams plugin that preserves legacy root config and proves account-scoped credentials, routing, auth, and state behavior.

## Reproduction Assessment

Yes. Source inspection of current main shows singleton channels.msteams credentials/webhook config, strict schema without accounts/defaultAccount, and inbound access fixed to DEFAULT_ACCOUNT_ID.

## Solution Assessment

No, not as originally sketched. The issue’s array plus agentBinding proposal is less aligned with existing OpenClaw routing than a Teams-owned accounts/defaultAccount model wired through bindings[].match.accountId, which the open candidate PR is attempting but has not merged.
