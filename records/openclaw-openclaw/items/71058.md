## Summary

Keep open. Current main still treats Microsoft Teams as a singleton Bot Framework credential/default account, and the related duplicate was closed into this issue while the Microsoft tracker still lists this work as unresolved.

## What This Changes

The issue requests multiple Microsoft Teams bot/app accounts on one OpenClaw gateway so different agents can appear as distinct Teams identities.

## Best Possible Solution

Add bundled msteams multi-account support using channels.msteams.accounts and defaultAccount, preserve current top-level config as the implicit default, and keep agent selection in bindings[].match.accountId.

## Reproduction Assessment

Yes. Source inspection on current main shows Teams still has a strict singleton config, a top-level default-account adapter, and setup behavior that ignores non-default account IDs.

## Solution Assessment

No for the exact proposed array plus agentBinding shape. The maintainable path is OpenClaw's existing account-map/defaultAccount contract with agent routing left in bindings, plus explicit auth and token-isolation design.

Status: needs_attention

Summary: The request is security-sensitive because it would change Teams credential selection, Bot Framework JWT validation, webhook routing, and OAuth/SSO token isolation.
