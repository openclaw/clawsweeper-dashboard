## Summary

Keep open: current main still exposes Microsoft Teams as one bot credential/webhook configuration, while the active multi-account implementation is an open PR that has not merged.

## What This Changes

Request support for multiple Microsoft Teams/Azure bot identities on one OpenClaw gateway, routed to different agents.

## Best Possible Solution

Land a reviewed Teams accounts/defaultAccount implementation in the msteams plugin, preserving legacy root config and proving account-scoped credentials, routing, auth, and state behavior.

## Reproduction Assessment

Yes. Source inspection of current main shows singleton channels.msteams credentials/webhook config, a strict schema without accounts/defaultAccount, and inbound pairing fixed to DEFAULT_ACCOUNT_ID.

## Solution Assessment

Not yet. The issue's array plus agentBinding sketch is less aligned with existing OpenClaw routing than a Teams-owned accounts/defaultAccount model wired through bindings[].match.accountId, and the open PR still needs maintainer acceptance before it becomes the solution.
