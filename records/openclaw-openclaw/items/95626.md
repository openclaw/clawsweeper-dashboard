## Summary

Keep open: the source patch appears to cover the opt-in rich send/edit fallback paths, but the proof is still mock-only and a recent maintainer comment questions whether fallback belongs behind an explicit `richMessages` opt-in.

## What This Changes

The PR adds Telegram rich-message unsupported-error detection and retries rich send/edit failures through standard HTML `sendMessage`/`editMessageText` paths with focused tests.

## Best Possible Solution

Land the fallback only if maintainers want automatic recovery for opt-in richMessages, with redacted live Telegram/Web proof and an exercised integration test; otherwise close or supersede it with clear guidance to disable richMessages on unsupported clients.

## Reproduction Assessment

Source-level yes, live no: current main has rich send/edit paths that propagate unsupported-rich errors when `richMessages` is enabled, and linked reports show the client symptom. This read-only review did not establish a live Telegram/Web reproduction on current main or the PR head.

## Solution Assessment

Unclear as a product decision: the Telegram plugin is the right implementation boundary if fallback is wanted, and the source patch is mostly narrow. It is not merge-ready until maintainers accept fallback semantics for an opt-in setting and real Telegram proof is supplied.
