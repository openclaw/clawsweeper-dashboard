## Summary

Keep open: the report is still actionable. Current main and the latest shipped release still have the forced-runtime harness selector treating `claude-cli` as a missing plugin harness, while the current auth-profile sync and doctor surfaces diagnose or scoped-sync Claude CLI credentials but do not prove the 4.x-to-5.x missing-profile/`lastGood` upgrade repair requested here.

## What This Changes

The issue reports that upgrading from OpenClaw 2026.4.15 to 2026.5.18 can leave Claude CLI-backed agents without the `anthropic:claude-cli` auth profile/runtime mapping and returning `MissingAgentHarnessError`.

## Best Possible Solution

Land or adapt the selector-path fix from https://github.com/openclaw/openclaw/pull/84966, then add a tested doctor/upgrade repair that detects legacy Claude CLI usage, imports safe headless Claude CLI credentials into `anthropic:claude-cli`, restores any required runtime/profile mapping, and emits a migration receipt for unresolved cases.

## Reproduction Assessment

No, not for the full upgrade path with high confidence: reproducing that needs a 2026.4.15 macOS Keychain-backed install upgraded to 2026.5.x. The selector-path trigger is source-reproducible on current main because `claude-cli` still falls through forced harness selection to `MissingAgentHarnessError`.

## Solution Assessment

Unclear for the whole issue. The linked selector PR is a plausible narrow fix for the second trigger, but the original upgrade/auth-profile repair should be a maintainer-approved doctor or upgrade migration with Keychain-proofed behavior and a receipt rather than an ad hoc startup rewrite.
