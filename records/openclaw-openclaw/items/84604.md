## Summary

The report describes a concrete upgrade-time auth-provider regression that is not covered by the already-closed followup queue issue; current `main` still has a source-visible path where `claude-cli` is only registered when an Anthropic auth profile resolves, while the 4.x config migration only normalizes model/runtime config and does not create `anthropic:claude-cli` profile or `lastGood.claude-cli` state. It should stay open as a narrow repair candidate.

## What This Changes

The issue requests a 4.x to 5.x upgrade migration or repair path that preserves existing `claude-cli` authentication by creating the required auth-profile and last-good mapping.

## Best Possible Solution

Add a targeted upgrade repair for legacy `claude-cli` runtimes that creates the missing Anthropic auth profile and last-good mapping only when the existing CLI login is detectable, with regression coverage for fresh installs, upgraded 4.x config, and existing explicit profile choices.

## Reproduction Assessment

Yes for source reproduction: current `main` shows `claude-cli` registration exits when no Anthropic auth profile or last-good mapping exists, and the 4.x migration code does not create that mapping. I did not live-replay a macOS Keychain upgrade from `2026.4.15`, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes, a bounded migration or `doctor --fix` repair is the right shape if it preserves existing explicit profile choices. The fix should not broaden core plugin policy; it should live in the Anthropic/CLI auth repair path with config migration tests.
