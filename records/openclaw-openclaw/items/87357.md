## Label Justifications

- `P2`: The report describes a bounded but real Telegram runtime failure after v2026.5.26 where plugin-state writes can block message handling.
- `impact:session-state`: The failure is in persisted Telegram plugin-state cache namespaces used to carry message context across runtime operations.
- `impact:message-loss`: The issue reports inbound Telegram messages receiving a stock failure reply once cache writes are locked out.

## Summary

This is a valid source-reproducible Telegram plugin-state cap bug, but the same remaining work is already tracked by the open canonical issue at https://github.com/openclaw/openclaw/issues/87332, including the need to decouple Telegram message-cache eviction from the 1000-row per-plugin hard cap.

## What This Changes

Fix Telegram plugin-state limit handling so message-cache writes can evict old entries instead of being blocked by the per-plugin 1000-row cap when another Telegram namespace has live rows.

## Best Possible Solution

Track the fix on https://github.com/openclaw/openclaw/issues/87332 and include this equality-plus-co-tenant regression case in the canonical fix and tests.

## Reproduction Assessment

Yes, by source inspection: seed 999 live rows in `telegram.message-cache` and one live row in `telegram.bot-info-cache`, then register a new message-cache key. Current main reaches `namespaceCount == maxEntries`, skips namespace eviction, then throws on the 1001-row plugin total.

## Solution Assessment

Yes for routing: closing this as a duplicate is the best cleanup because the canonical open issue already owns the same Telegram plugin-state cap failure and should absorb this narrower regression case.
