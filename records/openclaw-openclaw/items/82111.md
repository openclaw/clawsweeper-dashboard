## Summary

Keep open: the reported cron/external-channel failure is plausible and the patch is aimed at the right setup window, but it preloads plugins with different runtime-subagent options than the embedded cron run and the supplied proof is too weak for an external PR merge gate.

## What This Changes

The PR adds a lazy runtime-plugin facade, calls `ensureRuntimePluginsLoaded` during cron isolated context preparation, and adds harness coverage for that load path.

## Best Possible Solution

Align cron preloading with the embedded runner's gateway-bindable runtime options, add ordering coverage, and coordinate with https://github.com/openclaw/openclaw/pull/78978 so the final external-channel cron fix is coherent.

## Reproduction Assessment

Yes at source level, not from a live run here. Current main resolves cron delivery before runtime plugins are loaded for the embedded cron run, and the related external-channel issue documents the same missing-plugin symptom.

## Solution Assessment

No. Early runtime plugin loading is a reasonable direction, but this implementation must use the same gateway-bindable load options as the later embedded run and should lock the ordering requirement in the regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the gateway-bindable plugin runtime mode:** `src/cron/isolated-agent/run.ts:553`
  - body: Cron later calls `runEmbeddedPiAgent` with `allowGatewaySubagentBinding: true`, so preloading here without that option warms the default runtime-subagent cache key. The same cron run can then activate runtime plugins again in gateway-bindable mode, duplicating plugin initialization and hook side effects; pass the same option here and assert it in the new test.
  - confidence: 0.9
