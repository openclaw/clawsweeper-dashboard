## Summary

Keep this PR open: the reported cron external-channel failure is plausible and the lazy-load placement is useful, but the current patch can activate runtime plugins twice because the new preload does not use the same gateway-bindable runtime mode as the later embedded cron run.

## What This Changes

The branch adds a lazy runtime-plugin facade for cron isolated runs and a test asserting runtime plugins are loaded during cron setup.

## Best Possible Solution

Keep the lazy cron preload, but pass the gateway-bindable option through that preload and add coverage for both ordering and matching runtime-plugin load options.

## Reproduction Assessment

No independent live reproduction was run in this read-only review. Source inspection gives a clear path: current main resolves cron model/delivery before the embedded runner loads runtime plugins, and the PR body supplies after-fix Windows cron console output.

## Solution Assessment

No; the direction is right, but the current implementation is not the best fix because it preloads default-mode plugins before a gateway-bindable embedded run. Aligning the preload options is the narrower maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass gateway-bindable mode into the preload:** `src/cron/isolated-agent/run.ts:553`
  - body: This preload runs before cron later calls `runEmbeddedPiAgent` with `allowGatewaySubagentBinding: true`, but it does not pass that option here. The loader can reuse gateway-bindable registries for later default loads, not default registries for later gateway-bindable loads, so this can activate external plugins twice in one cron run. Pass the same gateway-bindable option into the early load and cover it in the new test.
  - confidence: 0.86
