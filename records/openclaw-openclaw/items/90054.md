## Summary

Keep open: the PR is aimed at the right gateway descriptor boundary, but it does not yet isolate descriptor array row access, so an unreadable row can still abort discovery before later healthy descriptors are processed.

## What This Changes

The PR adds shared gateway descriptor name/scope readers and routes gateway method listing, scope lookup, web-login discovery, startup discovery, and plugin registry construction through guarded descriptor reads with focused regression tests.

## Best Possible Solution

Use one centralized descriptor projection that catches both descriptor row access and descriptor property access, then route the gateway descriptor consumers through it.

## Reproduction Assessment

Yes. Current main directly reads descriptor properties, and the PR head still leaves a source-reproducible row-access throw path because `for...of` and `flatMap` read the array element before the guarded descriptor helper runs.

## Solution Assessment

No. Centralizing the guarded descriptor reader is the right layer, but the current implementation needs to guard row access too or it remains narrower than the stated row-isolation fix.
