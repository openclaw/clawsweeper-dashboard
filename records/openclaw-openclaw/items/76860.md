## Summary

Keep open: current main still has a source-reproducible destructive rewrite path for `gateway.systemd.env`, and the related closed reports cover unit-file directives or env expectations rather than preserving this env-file's existing contents.

## What This Changes

The issue asks OpenClaw upgrade/doctor systemd repair to preserve operator-customized `gateway.systemd.env` keys, back up the file before rewriting it, and notify operators when it changes.

## Best Possible Solution

Make systemd env-file staging non-destructive: parse the existing file, preserve operator-owned keys, back up before content-changing writes, emit a clear repair/update note, and add regression coverage for doctor/update staging.

## Reproduction Assessment

Yes, source-reproducible: create a state-dir `.env` plus an existing `gateway.systemd.env` with extra keys, then exercise `stageSystemdService` or doctor service repair; the current writer replaces the file with only `.env`-derived entries. I did not run a live systemd upgrade because this review is read-only.

## Solution Assessment

Yes, the narrowest maintainable fix is in the systemd service env-file writer and its doctor/update tests. It should avoid changing provider auth precedence or broader service-install policy.
