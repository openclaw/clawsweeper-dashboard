## Summary

Keep open. Current main still has the reported Gateway no-auth backend handshake gap, and the issue is paired with open PR #75781 using closing syntax, so it should stay open until that PR or a corrected replacement lands.

## What This Changes

The issue reports that `gateway.auth.mode: "none"` rejects internal backend WebSocket RPCs without device identity, breaking subagent spawning and Gateway tool calls.

## Best Possible Solution

Land a narrow Gateway auth fix, through #75781 or a corrected replacement, that treats local backend no-auth coordination as satisfied without widening browser, remote, proxied, or non-backend pairing bypasses.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: no-auth mode produces `authMethod: "none"`, current shared-auth and backend self-pairing checks do not accept it, and the missing-device path closes with `1008: device identity required`.

## Solution Assessment

No for the one-line helper-only proposal as written; the safer fix should cover both `resolveConnectAuthState` and `shouldSkipLocalBackendSelfPairing`, with tests that exercise the actual no-auth WebSocket handshake state.
