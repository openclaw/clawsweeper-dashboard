## Summary

Keep open. Current main still has the Gateway auth gap, and the issue is paired with open PR #75781 using closing syntax, so it should stay open until that PR or a corrected replacement lands.

## What This Changes

Allow local internal backend Gateway RPC connections to work under `gateway.auth.mode: "none"` instead of failing with `device identity required`.

## Best Possible Solution

Land a narrow Gateway auth fix through #75781 or a corrected replacement that treats no-auth as satisfied only for trusted local backend `gateway-client` coordination and adds helper plus handshake/auth-context coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `auth.mode: "none"` resolves to auth method `none`, current shared-auth and backend self-pairing checks exclude it, and a device-less backend path reaches the 1008 `device identity required` close.

## Solution Assessment

No for the body's helper-only proposed fix. The safer fix should cover both the backend self-pairing helper and the upstream `sharedAuthOk` state so the actual no-auth handshake path is exercised without widening the bypass.
