## Summary

Keep open. Current main still has the reported gateway auth gap, and the provided context shows an open paired PR (#75781) with closing syntax, so this issue should stay open until that PR or a corrected replacement lands.

## What This Changes

The issue requests that trusted loopback backend RPC and subagent connections continue to work when `gateway.auth.mode` is set to `"none"`.

## Best Possible Solution

Land a narrow gateway WebSocket auth fix, through #75781 or a corrected replacement, that allows only direct-loopback backend `gateway-client` coordination under `auth.mode: "none"` while preserving normal device-identity enforcement for other clients. The fix should cover both the auth-context state and backend self-pairing helper behavior with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: mode `none` resolves to auth method `none`, current shared-auth and backend self-pairing checks do not accept that method, and a device-less backend path can fall through to the `1008: device identity required` close.

## Solution Assessment

Unclear for the proposed one-line fix. Adding `"none"` only to the helper’s shared-auth method check is likely insufficient while `resolveConnectAuthState` still leaves `sharedAuthOk` false for mode `none`; the safer fix covers the full handshake state path and local/backend-only guardrails.
