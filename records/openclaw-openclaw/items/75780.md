## Summary

Keep open. Current main still has the reported Gateway no-auth backend handshake gap: no-auth resolves as method `none`, but the shared-auth state and local backend self-pairing helper still only treat token/password as shared auth, so device-less local backend clients can still reach `1008: device identity required`. Open PR #75781 references this issue with closing syntax, so this should stay open until that PR or a corrected replacement lands.

## What This Changes

Request that local backend RPC and subagent connections work under documented loopback `gateway.auth.mode: "none"` instead of failing with `device identity required`.

## Best Possible Solution

Review and land #75781 or a corrected replacement that makes no-auth local backend self-connections satisfy the narrow backend pairing bypass while preserving remote/browser/proxy rejection and adding auth-context plus handshake regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `authorizeHttpGatewayConnect` returns method `none`, no token/password makes `sharedAuthOk` false, `shouldSkipLocalBackendSelfPairing` rejects method `none`, and the missing-device branch closes with `1008`.

## Solution Assessment

Unclear for the proposed one-line helper change because current `sharedAuthOk` is also false for no-auth. The narrow maintainable fix should cover the auth-context state and the full backend handshake path, not only the helper predicate.
