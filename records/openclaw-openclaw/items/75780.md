## Summary

Keep open. Current main still has the reported Gateway no-auth backend handshake gap, and open PR #75781 references this issue with closing syntax, so the issue should remain open until that PR or a corrected replacement lands.

## What This Changes

The issue requests that local backend Gateway clients keep working without device identity when `gateway.auth.mode` is `"none"` for trusted loopback setups.

## Best Possible Solution

Review and land #75781 or a corrected replacement that makes no-auth local backend self-connections satisfy the narrow pairing bypass while preserving remote/browser/proxy rejection and adding regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: no-auth returns method `none`, `sharedAuthOk` remains false for that method, the backend self-pairing helper rejects it, and the missing-device branch closes with `1008`.

## Solution Assessment

Unclear for the proposed one-line helper fix. The requested direction is right, but the maintainable fix should cover both auth-state and helper behavior in the full backend handshake path.
