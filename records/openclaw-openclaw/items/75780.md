## Summary

Keep open. Current main still excludes `gateway.auth.mode: "none"` from the auth-satisfied local backend self-pairing path, and open PR #75781 is the referenced implementation candidate, so this issue should remain open until a reviewed fix lands.

## What This Changes

Allow internal Gateway backend RPCs, including subagent spawning, to work under `gateway.auth.mode: "none"` without requiring device identity on trusted loopback connections.

## Best Possible Solution

Implement a narrow Gateway handshake fix that recognizes no-auth as auth-satisfied only for trusted local backend coordination, with coverage for auth context, locality, helper behavior, and the 1008 rejection path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: no-auth returns method `none`, current shared-auth and self-pairing checks do not accept it, and a device-less backend connection can reach the `1008: device identity required` branch.

## Solution Assessment

Unclear. The helper-only change proposed in the issue is not enough by itself because `resolveConnectAuthState` does not mark method `none` as auth-satisfied; the maintainable fix should cover both auth context and the local backend pairing helper.
