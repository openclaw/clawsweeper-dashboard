## Summary

Keep open. Current main still has the reported Gateway auth gap, and the issue is paired with open PR #75781 using closing syntax, so it should stay open until that PR or a corrected replacement lands.

## What This Changes

Allow internal loopback backend Gateway RPC and subagent connections to work under documented `gateway.auth.mode="none"` instead of closing for missing device identity.

## Best Possible Solution

Land a narrow Gateway auth fix, through #75781 or a corrected replacement, that allows only trusted local backend `gateway-client` coordination under `auth.mode: "none"` and adds helper plus handshake/auth-context coverage for that flow.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `auth.mode: "none"` produces auth method `none`, current shared-auth and backend self-pairing checks do not accept it, and the device-less path closes with `1008: device identity required`.

## Solution Assessment

Unclear for the proposed helper-only direction. The safer fix should cover both the backend self-pairing helper and the upstream shared-auth state so the actual `none` handshake path is exercised without widening the bypass.
