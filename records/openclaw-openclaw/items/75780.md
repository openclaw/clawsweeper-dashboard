## Summary

Keep open. Current main and latest release still have the Gateway auth gap: no-auth resolves to auth method `none`, but shared-auth and backend self-pairing only admit token/password/trusted-proxy/device-token paths, and #75781 is still an open closing PR for this issue.

## What This Changes

The issue asks for trusted local backend RPC and subagent connections to work when `gateway.auth.mode` is `none` without requiring device identity pairing.

## Best Possible Solution

Land a narrow Gateway auth fix through #75781 or a corrected replacement that covers the no-auth backend coordination path end to end and adds focused auth-context/helper/handshake coverage without widening remote or browser bypasses.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `gateway.auth.mode: "none"`, auth succeeds as method `none`, `sharedAuthOk` remains false, the backend self-pairing skip stays false, and the missing-device branch closes with `1008: device identity required`.

## Solution Assessment

Unclear for the proposed helper-only direction. Adding `none` only to `usesSharedSecretAuth` does not address the current `sharedAuthOk` calculation, so the safer solution should cover both the upstream auth state and the backend self-pairing decision with tests for the actual no-auth handshake.
