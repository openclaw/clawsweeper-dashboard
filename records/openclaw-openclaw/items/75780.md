## Summary

Keep open. Current main still has the reported Gateway no-auth backend handshake gap: `auth.mode: "none"` is documented and authenticates as method `none`, but the shared-auth and backend self-pairing paths still do not treat that method as eligible, so device-less local backend connections can reach the `1008: device identity required` rejection. Open PR #75781 references this issue with closing syntax, so this issue should stay open until that PR or a corrected replacement lands.

## What This Changes

Allow internal local backend Gateway RPC and subagent connections to work under `gateway.auth.mode: "none"` instead of failing device identity checks.

## Best Possible Solution

Review and land #75781 or a corrected replacement that permits only trusted local `gateway-client` backend coordination under `auth.mode: "none"`, with auth-context, helper, and handshake tests preserving remote/browser/proxy rejection.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: no-auth returns method `none`, current shared-auth and backend skip checks do not accept it, and missing device identity reaches the `1008: device identity required` branch.

## Solution Assessment

Unclear for the proposed one-line helper direction. The narrow maintainable fix should exercise the full no-auth backend handshake path because `sharedAuthOk` is also false for no-auth on current main.
