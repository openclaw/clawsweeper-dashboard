## Summary

Keep open. Current main still has the reported backend-auth gap, and the issue is paired with open PR #75781 using closing syntax, so it should stay open until that PR or a replacement lands.

## What This Changes

The issue reports that `gateway.auth.mode: "none"` causes internal backend gateway-client RPC connections, including subagent spawning and gateway tool calls, to fail with `device identity required`.

## Best Possible Solution

Land a narrow gateway auth fix, through #75781 or a corrected replacement, that allows only direct-loopback backend `gateway-client` coordination under `auth.mode: "none"` while keeping remote and browser/node clients on the normal device-identity path. The fix should include helper-level and handshake/auth-context coverage for the actual `none` flow.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: mode `none` returns auth method `none`, current shared-auth state and backend self-pairing checks do not accept it, and the device-less WebSocket path closes with `1008: device identity required`.

## Solution Assessment

Unclear for the proposed PR as described. Adding `"none"` only to the helper’s `usesSharedSecretAuth` check may not be sufficient unless the `sharedAuthOk` calculation and an end-to-end handshake case are also covered.
