## Summary

Keep this PR open, but it should not merge as written. Current main still has the hyphenated-token splitting gap the PR targets, while the proposed guard is too broad and disables the existing long-token overflow protection for dash-only and generic hyphenated runs.

## What This Changes

The PR changes `src/tui/tui-formatters.ts` so `isCopySensitiveToken` treats file-like tokens containing either underscores or hyphens as copy-sensitive.

## Best Possible Solution

Preserve genuinely copy-sensitive hyphenated identifiers in TUI output without disabling generic long-token chunking for dash-only separators or prose-like hyphen runs, and add focused sanitizer regression tests for both preserved and still-chunked cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: with the PR condition, a token such as `-`.repeat(60) matches `FILE_LIKE_RE`, contains `-`, returns before `chunkToken`, and is no longer constrained to 32-character display segments.

## Solution Assessment

No as written. The maintainable fix should narrow hyphen preservation to copy-sensitive identifiers or context-aware code/table cases while keeping generic long-token wrapping intact, preferably with tests aligned with the overlapping formatter coverage in #39565.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep generic hyphen runs chunkable:** `src/tui/tui-formatters.ts:82`
  - body: This guard treats every long token containing `-` and only `[A-Za-z0-9._-]` as copy-sensitive. That includes strings like `-`.repeat(60), which currently get split by `normalizeLongTokenForDisplay` to protect narrow terminals; after this change they bypass `chunkToken` and can overflow the TUI. Please narrow the hyphen heuristic and add regression tests for dash-only or prose-like tokens that must still wrap.
  - confidence: 0.88
