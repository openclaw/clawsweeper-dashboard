## Summary

Keep this PR open, but it should not merge as written. Current main still has the hyphenated-token splitting gap, while the proposed guard is too broad and can disable the existing long-token overflow protection for dash-only or generic hyphenated runs.

## What This Changes

The PR changes `src/tui/tui-formatters.ts` so `isCopySensitiveToken` treats file-like tokens containing either underscores or hyphens as copy-sensitive.

## Best Possible Solution

Preserve genuinely copy-sensitive hyphenated identifiers in TUI output with a narrower classifier, keep generic dash-only and prose-like long tokens chunkable, and add regression tests for both preserved and still-wrapped cases.

## Reproduction Assessment

Yes. With the PR condition, a token such as `-`.repeat(60) matches `FILE_LIKE_RE`, contains `-`, returns before `chunkToken`, and is no longer constrained to 32-character display segments.

## Solution Assessment

No as written. The safer fix should identify copy-sensitive hyphenated identifiers or code-like contexts without turning every long `[A-Za-z0-9._-]` token containing a hyphen into an unwrapped token.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep generic hyphen runs chunkable:** `src/tui/tui-formatters.ts:82`
  - body: This guard treats every long token containing `-` and only `[A-Za-z0-9._-]` as copy-sensitive. That includes strings like `-`.repeat(60) or long prose-like hyphen runs, which currently get split by `normalizeLongTokenForDisplay` to protect narrow terminals; after this change they bypass `chunkToken` and can overflow the TUI. Please narrow the hyphen heuristic and add regression tests for dash-only/prose tokens that must still wrap.
  - confidence: 0.88
