## Summary

Keep this PR open. The SQLite-backed QMD export cache is useful and the real-runtime proof is strong, but current head still trusts a persisted cached markdown target before deriving or validating the expected target inside the current export directory; the schema/fingerprint tradeoffs and dirty merge state also need maintainer handling before merge.

## What This Changes

The PR adds a per-agent SQLite `qmd_session_export_cache` table, a QMD session-export stat/fingerprint fast path, QMD-specific plugin SDK cache helpers, generated API/schema updates, and regression tests.

## Best Possible Solution

Keep the SQLite-backed cache behind QMD-specific helpers, derive or validate cached markdown targets inside the current export directory before fast-pathing, add the stale-target regression test, then rebase and let maintainers explicitly accept the schema and fingerprint tradeoffs.

## Solution Assessment

No, not yet. The SQLite cache and QMD-specific helper seam are an acceptable direction, but the fast path should validate or derive persisted targets before trusting them.
