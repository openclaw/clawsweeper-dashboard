- `P2`: This is a normal-priority QMD performance/storage improvement with a concrete fast-path performance blocker and limited blast radius.

- `merge-risk: 🚨 session-state`: The export cache controls generated session markdown used by QMD, so cache identity or freshness mistakes can leave session exports stale or missing.

## Summary

Keep open: the latest target-validation fix addresses the previous stale-target blocker, but the cache read path still refreshes the shared agent DB registry once per session file, so large QMD exports can do thousands of synchronous SQLite writes before the fast path skips rebuild work.

## What This Changes

The PR adds a SQLite-backed QMD session export cache, a stat/fingerprint fast path in `exportSessions()`, QMD cache helpers on the memory-core host SDK facade, schema/API baseline updates, and regression tests.

## Best Possible Solution

Land the SQLite-backed cache only after batching cache access for the export pass, while keeping raw DB access internal and explicitly accepting the schema-version, fingerprint, and SDK-contract tradeoffs.

## Reproduction Assessment

Yes for the review finding: source inspection shows `exportSessions()` calls the cache read helper inside the session-file loop, and that helper reaches an agent-DB open path that refreshes shared registry state even for cached handles. The PR's main cache behavior has terminal real-runtime proof, but I did not rerun the full 4,052-session benchmark.

## Solution Assessment

Not yet. The SQLite cache and narrow helper seam are a plausible owner-boundary solution, but cache reads should be batched or scoped so the fast path does not add one global registry write per transcript.
