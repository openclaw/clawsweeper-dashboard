## Summary

Keep open: current main still rebuilds every QMD session export through `buildSessionEntry()`, and this PR is a useful performance path, but the persistent-cache fast path still trusts a stored `target` before deriving or validating the expected export target.

## What This Changes

The PR adds a per-agent SQLite `qmd_session_export_cache`, a QMD stat/fingerprint export fast path, QMD cache helper exports through the memory-core host foundation facade, generated schema/API baseline updates, and regression tests.

## Best Possible Solution

Land the SQLite-backed cache after deriving or validating cached export targets, while keeping raw DB access internal and making the schema-version and fingerprint tradeoffs explicit maintainer choices.

## Reproduction Assessment

Yes for the review finding: source inspection shows a current-scope cache row can match stat/fingerprint and skip rebuild while using `cached.target` before the expected target is derived. The PR's main performance claim has real runtime proof for the cache path, but not a rerun of the full production-sized benchmark.

## Solution Assessment

Not yet. The SQLite cache and narrow helper seam are a plausible owner-boundary solution, but the fast path should derive or validate the target before skipping and maintainers still need to accept the schema-version and fingerprint tradeoffs.
