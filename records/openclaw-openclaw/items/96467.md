## Summary

Keep open: current main and v2026.6.10 still run `models status --probe` as a live embedded model turn with `maxTokens`, and the timeout is only passed into that run path rather than applied as a whole-command wall-clock cap. The docs now warn that probes are real requests, but the timeout behavior, refresh-only command, and startup cost still need maintainer scoping.

## What This Changes

The issue requests faster `models status`/`models auth list`, a wall-clock honoring `--probe-timeout`, and a refresh-only auth check that avoids a completion call.

## Best Possible Solution

Keep the existing warning that probes are real requests, add a narrow wall-clock cap/regression test for `--probe-timeout`, then separately decide whether a refresh-only auth command and startup-performance project belong in this CLI surface.

## Reproduction Assessment

Yes at source level: current main still performs a live embedded model run with `maxTokens` for each probe target and lacks an outer wall-clock cap around the whole probe. I did not run the live Anthropic/Windows timing repro in this read-only review.

## Solution Assessment

Unclear for the full issue: honoring a wall-clock cap is a narrow maintainable fix, but adding `models auth refresh` and deeper startup-performance changes need maintainer product scoping.
