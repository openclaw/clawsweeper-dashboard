## Summary

Keep open: current main and v2026.6.10 still run `models status --probe` through a live embedded model call with `maxTokens`, and `--probe-timeout` is passed into each embedded run rather than enforced as a whole-command wall-clock cap. Current docs already warn that probes are real requests, but the timeout behavior, refresh-only auth command, and startup-overhead work remain unresolved.

## What This Changes

The issue requests a cheaper and bounded `models status --probe` flow: honor `--probe-timeout` as a wall-clock cap, add refresh-only auth, reduce auth-command startup overhead, and clarify probe cost.

## Best Possible Solution

Keep the current real-request documentation, add a narrow wall-clock cap or clearly renamed timeout behavior for the existing probe path, and separately scope refresh-only auth plus startup-performance work.

## Reproduction Assessment

Yes, at source level: current main still performs a live embedded model run with `maxTokens` for each probe target and has no outer wall-clock cap around the full probe. I did not run the reporter's live Anthropic/Windows timing repro in this read-only review.

Unclear for the full issue: a wall-clock cap for `--probe-timeout` is a narrow maintainable fix, but a refresh-only auth command and startup-performance work need maintainer scoping before implementation.
