## Summary

Keep open. The iMessage cleanup belongs on the right code path, but this PR is not merge-ready because it lacks after-fix real behavior proof and the new prefix heuristic can strip legitimate leading replacement characters.

## What This Changes

The PR adds iMessage monitor text cleanup for leading attributedBody replacement/control prefixes and regression tests around the existing length-prefixed UTF-8 normalizer.

## Best Possible Solution

Keep the fix in the iMessage monitor parser, but narrow stripping to observed U+FFFD/control corruption shapes and merge only with regression coverage plus redacted real imsg/OpenClaw proof.

## Reproduction Assessment

Yes for the source-level bug: current main routes iMessage text through a helper that only strips exact length-prefixed wrappers, so the attributedBody U+FFFD/control examples in the PR would pass through unchanged. I did not run a live macOS/imsg reproduction.

## Solution Assessment

No. The parser is the right surface, but the current heuristic should require control-marker evidence before stripping replacement-character prefixes and should be backed by real after-fix runtime proof.
