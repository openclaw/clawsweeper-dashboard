## Summary

Keep open. Current main still lacks a stable `sessions_spawn` recovery identity: native and ACP spawn paths create fresh child sessions and gateway idempotency keys on each invocation, so a lost acknowledgement followed by a retry can still start duplicate child runs. The linked ClawHub skill is useful operational guidance, but it does not close the core retry gap.

## What This Changes

Add recovery/idempotency handling so repeated `sessions_spawn` attempts after a lost acknowledgement converge on the already accepted child run instead of creating duplicate work.

## Best Possible Solution

Add an additive requester-scoped recovery mapping for `sessions_spawn` so native and ACP retries return the original accepted child run before another child session is created.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: invoke `sessions_spawn`, lose the parent acknowledgement, then invoke the same spawn request again; current native and ACP paths generate new child session keys plus new gateway idempotency keys.

## Solution Assessment

Yes, a `sessions_spawn` recovery identity is the narrowest maintainable fix. Gateway-level idempotency and the linked ClawHub deployment skill are useful but insufficient because the spawn layer gives each retry a different key before the gateway can dedupe it.
