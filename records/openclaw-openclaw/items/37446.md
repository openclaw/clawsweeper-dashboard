## Summary

Current main fixes the narrower `message(action=send)` lost-ack retry when the same agent run repeats equivalent delivery parameters, but it does not define a logical-operation identity shared by replacement sub-agent runs or arbitrary external side-effecting tools. The issue should remain open for a maintainer product decision rather than being closed as implemented or delegated entirely to the linked ClawHub workaround.

## What This Changes

Add a durable logical-action idempotency mechanism so sub-agent timeout recovery cannot repeat already-committed external side effects.

## Best Possible Solution

Define a narrow generic logical-operation identity in agent/tool execution context, preserve it only across explicitly classified recovery attempts, and let each side-effecting tool or plugin map it to a target-supported idempotency or reconciliation contract.

## Reproduction Assessment

No, not at high confidence: the report provides credible observations, but no current-main steps, tool trace, or logs prove that OpenClaw itself respawns and reissues the same arbitrary POST. Source inspection does confirm that replacement agent invocations receive fresh request identities while only selected tools implement stable deduplication.

## Solution Assessment

Unclear: server-side idempotency is the appropriate primitive when a target supports it, but none of the three proposals alone safely covers arbitrary tools and replacement runs. The maintainable direction is an explicit logical-operation contract with owner-tool reconciliation, not a universal transport-layer promise.
