## Summary

Keep open: current main has a useful partial fix for `message(action=send)` retries, but it does not yet provide the broader logical sub-agent action idempotency contract requested for side-effecting POST retries after lost acknowledgements.

## What This Changes

The issue requests an idempotency/recovery mechanism so repeated sub-agent side-effecting POST attempts do not create duplicate comments, actions, or other external effects after a timeout or dropped response.

## Best Possible Solution

Define a narrow OpenClaw-owned logical-action idempotency contract for sub-agent spawn/session/message side effects, keep the current message-tool mitigation, and leave service-specific external API idempotency patterns to plugins, skills, or caller code.

## Reproduction Assessment

No high-confidence live repro was established for the exact dropped-ack API POST scenario. Source inspection does show the remaining gap: current `sessions_spawn` creates fresh child run idempotency per spawn attempt and does not expose a stable logical retry key.

## Solution Assessment

Unclear as proposed. Current main's message-tool fix is a good partial mitigation, but the broader issue needs a maintainer-approved contract for logical action identity rather than a blanket transport dedupe rule.
