## Summary

Keep open. Current main has scoped Gateway, hook, and announce dedupe, but sub-agent spawn and recovery still create fresh operation keys and do not expose a stable retry identity; the linked ClawHub skill is useful production guidance, not a replacement for a core reliability contract.

## What This Changes

The issue requests a stable sub-agent idempotency/recovery mechanism so lost-ack retries do not duplicate side-effecting POST actions.

## Best Possible Solution

Define and document a stable sub-agent operation identity that survives timeout/recovery retries, propagate it through OpenClaw-owned side-effecting Gateway/tool calls, and document where external services must still supply their own idempotency keys.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the issue context alone. Source inspection does show the current gap: current dedupe works for reused keys, while spawn and orphan resume paths generate fresh random keys and do not expose a stable operation identity.

## Solution Assessment

Unclear. The proposed directions are plausible, but the best core fix needs maintainer choice between stable run/action identity propagation, endpoint-level dedupe, and plugin/tool guidance for external services.
