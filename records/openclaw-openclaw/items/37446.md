## Summary

Keep open. The issue remains a real reliability/design gap: current source appears to provide exact idempotency only when callers reuse the same key, while subagent spawn/control/recovery paths still mint new keys for retry-like operations; however the report needs maintainer product/API decisions and live lost-ack proof before a concrete fix should be queued.

## What This Changes

Request stable idempotency or recovery identity for sub-agent timeout retry paths so successful-but-unacknowledged POSTs do not create duplicate external side effects.

## Best Possible Solution

Define a durable subagent operation identity that survives spawn retry, steer/restart, and orphan resume for OpenClaw-owned side effects, while documenting that external APIs still need their own idempotency contracts.

## Reproduction Assessment

No high-confidence live reproduction is attached. Source inspection gives a medium-confidence path because Gateway dedupe is exact-key scoped while subagent spawn, control, and orphan recovery can generate fresh keys.

## Solution Assessment

Unclear. The proposed key and last-N approaches are plausible, but the maintainable direction should be a shared recovery identity contract rather than content matching or one-off per-endpoint guards.
