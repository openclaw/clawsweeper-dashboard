## Summary

Keep open. Current main has idempotency for some internal Gateway and announce paths, but the reported broader subagent recovery side-effect boundary is not solved; choosing run identity propagation, outbound idempotency headers, or session-level dedupe needs maintainer API/product judgment.

## What This Changes

The issue requests an idempotency mechanism so subagent timeout or restart recovery cannot repeat side-effecting POST-like actions after a lost acknowledgement.

## Best Possible Solution

Define a stable subagent operation identity that can flow through recovery and side-effecting tool/plugin calls, then document which OpenClaw-owned endpoints or helpers dedupe on that identity and where external services must still supply their own idempotency.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the issue context alone. Source inspection does show the current design gap: internal Gateway calls dedupe only when a stable idempotency key is reused, while recovery and downstream external side effects do not have a general propagated operation identity.

## Solution Assessment

Unclear. The proposed directions are plausible, but the best core fix needs maintainer choice between stable run/action identity propagation, OpenClaw endpoint dedupe, and plugin/tool guidance for external-service idempotency.
