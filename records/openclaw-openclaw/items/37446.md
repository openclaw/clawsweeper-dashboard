## Summary

Keep open. Current main still has exact-key Gateway dedupe, while subagent spawn, steer/restart, and orphan-recovery paths can mint fresh keys for a new logical recovery attempt, so the lost-ack duplicate side-effect problem is not clearly solved and needs a maintainer contract decision.

## What This Changes

The issue requests durable idempotency for subagent timeout/retry recovery so successful-but-unacknowledged POST-like actions are not repeated as duplicate comments, messages, or external API side effects.

## Best Possible Solution

Define one durable subagent operation identity for OpenClaw-owned retry/recovery side effects, reuse it across spawn retry, steer/restart, and orphan resume, and document where third-party APIs still require their own idempotency keys.

## Reproduction Assessment

No high-confidence live reproduction is attached. Source inspection gives a medium-confidence current-main path: Gateway dedupe is exact-key scoped while subagent spawn, control, and orphan recovery can still generate fresh keys for the same logical recovery flow.

## Solution Assessment

Unclear. Idempotency keys are the right direction, but maintainers should choose the durable operation-identity scope instead of landing a one-off last-N or content-matching workaround.
