## Summary

Keep open. Current main has scoped Gateway idempotency for retries that reuse the same key and newer orphan-recovery duplicate guards, but native sub-agent spawn/recovery still generates fresh identities and there is no chosen core contract for replay-safe side-effecting work; the linked ClawHub skill is useful operational guidance, not a replacement for that core decision.

## What This Changes

Add idempotency or recovery semantics so sub-agent timeout retries do not duplicate already-successful POST/actions.

## Best Possible Solution

Define and document a stable sub-agent operation or action identity that survives timeout/recovery retries, propagate it through OpenClaw-owned side-effecting Gateway/tool paths, and document where external services still need their own idempotency keys.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the issue context alone. Source inspection does show the current-main gap: Gateway dedupes reused keys, while native spawn and orphan recovery still generate fresh keys for retry/recovery paths.

## Solution Assessment

Unclear. The proposed idempotency-key, last-N check, and session-scoped dedupe approaches are plausible, but maintainers need to choose the shared API/reliability contract before implementation.
