## Summary

Keep open. Current main still only dedupes Gateway side effects for the exact idempotency key it receives, while sub-agent spawn, orphan recovery, and outbound helper paths can still mint fresh keys across timeout/recovery attempts; the linked ClawHub skill is a useful workaround pattern, not a replacement for a core retry contract.

## What This Changes

The issue requests stable idempotency for sub-agent timeout/recovery side effects so retries do not create duplicate API posts or actions.

## Best Possible Solution

Define a stable sub-agent operation/action identity that survives timeout and recovery retries, propagate it through OpenClaw-owned side-effecting Gateway and outbound paths, and document where external services must still supply their own idempotency keys.

## Reproduction Assessment

No, not as a high-confidence live lost-ack reproduction. Source inspection does show the current-main gap: Gateway dedupe works for the same key, while sub-agent spawn, orphan recovery, and outbound action paths can generate fresh keys across retries.

## Solution Assessment

Unclear. The proposed idempotency-key, last-N check, and session-scoped dedupe approaches are plausible, but the maintainable fix should be a shared retry/recovery contract rather than content matching in one caller.
