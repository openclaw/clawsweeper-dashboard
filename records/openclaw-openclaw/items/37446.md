## Summary

Keep open. Current main still has same-key Gateway dedupe, but sub-agent spawn, orphan recovery, and message-action fallback paths can create fresh idempotency identities, so the lost-ack duplicate side-effect problem is not fully fixed; the linked ClawHub skill is a useful operational workaround, not a core reliability contract.

## What This Changes

The issue asks OpenClaw to prevent duplicate side-effecting API posts when sub-agent timeout recovery retries after a successful POST whose response acknowledgement was lost.

## Best Possible Solution

Define a stable sub-agent operation or action identity that survives timeout and recovery retries, propagate it through OpenClaw-owned side-effecting Gateway/tool paths, and document where external APIs still require their own idempotency keys.

## Reproduction Assessment

No, not as a high-confidence live lost-ack reproduction. Source inspection shows same-key dedupe exists while several sub-agent and message-action recovery paths still generate fresh keys when the caller did not provide a stable one.

## Solution Assessment

Unclear. The proposed idempotency-key, last-N check, and session-scoped dedupe approaches are plausible, but the maintainable fix should be a shared retry/recovery contract rather than ad hoc content matching in one path.
