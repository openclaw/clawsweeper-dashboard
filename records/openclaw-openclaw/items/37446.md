## Summary

Keep open. Current main has exact-key Gateway dedupe and adjacent duplicate-delivery fixes, but it still does not provide a stable sub-agent retry/recovery identity that would survive a lost acknowledgement or timeout-recovery replay; the linked ClawHub skill is a useful workaround pattern, not a core fix.

## What This Changes

The issue requests idempotency and recovery handling so sub-agent timeout retries do not duplicate side-effecting POST/API actions.

## Best Possible Solution

Define a stable sub-agent operation/action identity that survives timeout and recovery retries for OpenClaw-owned side effects, then document where external APIs still require their own idempotency keys.

## Reproduction Assessment

No, not with a high-confidence live lost-ack reproduction. Source inspection does show the current-main path: Gateway dedupe is exact-key scoped while native spawn and orphan recovery can still generate fresh keys.

## Solution Assessment

Unclear. The proposed idempotency-key approaches are plausible, but the maintainable fix should be a shared retry/recovery identity contract rather than a one-off content match or caller shim.
