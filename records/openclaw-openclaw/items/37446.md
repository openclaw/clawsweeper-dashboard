## Summary

Current main and the latest release solve ambiguous retries for OpenClaw-owned `message(action=send)` calls, but they do not establish idempotency for the broader arbitrary external API posts described here. The issue should remain open, narrowed to concrete OpenClaw-owned side-effect surfaces and a maintainer decision about the supported recovery identity contract.

## What This Changes

Prevent duplicate external side effects when a sub-agent retries after a request may have committed but its acknowledgement was lost.

## Best Possible Solution

Keep the existing message-tool behavior, then define stable logical-action identity only for concrete OpenClaw-owned side-effecting tools; pass explicit provider idempotency keys where supported and document existence-check recovery for APIs that lack them.

## Reproduction Assessment

Partly yes: the formerly broken message-tool retry path is source-reproducible and covered by a regression test. The broader arbitrary API-post behavior lacks a high-confidence current-main reproduction because the report does not identify the OpenClaw tool, command, or retry path that reissued those writes.

## Solution Assessment

No, none of the broad proposals is yet the best repository-wide fix. Per-tool stable logical-action keys are safer than session-wide content blocking or last-N inference, while third-party APIs must provide idempotency or a resource-specific existence check.
