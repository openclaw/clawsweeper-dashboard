## Summary

Keep open. Current main has useful same-key Gateway and hook idempotency, and the linked ClawHub skill is a reasonable operational workaround, but OpenClaw still lacks a stable sub-agent timeout/lost-ack identity that survives recovery retries across OpenClaw-owned side-effecting paths.

## What This Changes

The issue asks OpenClaw to prevent duplicate side-effecting API posts when sub-agent timeout recovery retries after a successful request whose acknowledgement was lost.

## Best Possible Solution

Define a stable sub-agent operation/action identity that survives timeout and recovery retries, propagate it through OpenClaw-owned side-effecting Gateway/tool paths, and document where third-party APIs must still receive their own idempotency keys.

## Reproduction Assessment

No, not as a high-confidence live lost-ack reproduction. Source inspection does show the gap: same-key dedupe exists, while sub-agent spawn/recovery and outbound fallback paths still mint fresh keys when no stable caller key is carried forward.

## Solution Assessment

Unclear. The proposed idempotency-key, last-N check, and session-scoped dedupe approaches are plausible, but the best fix should be a shared retry/recovery contract rather than ad hoc content matching in one path.
