## Summary

Keep open. Current main still has same-key Gateway/send dedupe only; sub-agent spawn and recovery paths still create fresh identities, so timeout/recovery replay safety still needs a maintainer-owned reliability contract.

## What This Changes

The issue requests a sub-agent idempotency/recovery mechanism so lost acknowledgements or timeout recovery do not repeat the same side-effecting POST/action.

## Best Possible Solution

Define and document a stable sub-agent operation/action identity that survives parent timeout and recovery retries, propagate it through OpenClaw-owned side-effecting Gateway/tool paths, and document where external services still require their own idempotency keys.

## Reproduction Assessment

No, not as a high-confidence end-to-end lost-ack reproduction. Source inspection does show the gap: current Gateway dedupe works for reused keys, while sub-agent spawn, sessions_send, control, and orphan recovery paths still create fresh idempotency keys for new side-effecting attempts.

## Solution Assessment

Unclear. The proposed idempotency-key, last-N check, and session-scoped dedupe approaches are plausible, but the best fix needs a maintainer decision on the shared sub-agent retry/recovery contract before implementation.
