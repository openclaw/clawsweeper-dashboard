## Summary

Keep open. Current main has Gateway request dedupe and stable sub-agent announcement dedupe, but the `sessions_spawn` and orphan-recovery paths still create fresh random identities for new child runs/resume attempts, so the reported lost-ack retry gap is not closed; the linked ClawHub skill is useful operational guidance, not a replacement for a core reliability contract.

## What This Changes

The issue requests an idempotency/recovery contract for sub-agent timeout retries so successful side-effecting POSTs are not repeated after a lost acknowledgement.

## Best Possible Solution

Define and document a stable sub-agent operation or action identity that survives timeout/recovery retries, propagate it through OpenClaw-owned side-effecting paths, and document where external services must still provide their own idempotency keys.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the issue context alone. Source inspection does show the current gap: Gateway dedupes reused keys, while `sessions_spawn` and orphan resume create fresh random identities.

## Solution Assessment

Unclear. The proposed directions are plausible, but maintainers need to choose the core contract between stable run/action identity propagation, endpoint-level dedupe, and plugin/tool guidance for external services.
