## Summary

Keep open. Current main still dedupes only when the same Gateway idempotency key is reused, while native sub-agent spawn and orphan-recovery paths still create fresh identities for new/recovery runs, so the reported replay-safety gap remains a maintainer-owned reliability contract.

## What This Changes

Add an idempotency or recovery mechanism so sub-agent timeout recovery does not replay successful side-effecting POSTs.

## Best Possible Solution

Define a stable sub-agent operation/action identity that survives timeout and recovery retries, propagate it through OpenClaw-owned side-effecting Gateway/tool paths, and document where external APIs still need their own idempotency keys.

## Reproduction Assessment

No, not as a high-confidence live lost-ack reproduction. Source inspection does show the gap: same-key Gateway dedupe exists, while native spawn and orphan recovery still generate fresh keys.

## Solution Assessment

Unclear. The proposed idempotency-key, last-N check, and session-scoped dedupe approaches are plausible, but maintainers need to choose the shared retry/recovery contract before implementation.
