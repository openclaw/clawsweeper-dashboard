## Summary

Keep open. Current main still has the reported `sessions_spawn` idempotency gap: both native and ACP spawn paths mint fresh child session keys and fresh gateway `agent` idempotency keys on each invocation, while gateway dedupe only protects retries that reuse the same idempotency key. The ClawHub skill linked in the discussion is useful operational guidance, but it cannot prevent OpenClaw core from creating a second child run after a lost parent acknowledgement.

## What This Changes

The issue requests idempotent recovery for sub-agent POST/side-effect retries after lost acknowledgements.

## Best Possible Solution

Add an additive `sessions_spawn` recovery contract that accepts or derives a requester-scoped recovery key, persists the key-to-child-run mapping, and makes native and ACP retry paths return the original accepted child run after timeout or lost acknowledgement. Keep service-specific idempotency guidance in ClawHub/docs, but make OpenClaw core responsible for not starting duplicate child sessions for the same recovered spawn request.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to invoke the same `sessions_spawn` tool call again after losing the parent acknowledgement; current main has no stable recovery parameter and both native and ACP implementations generate new child session and gateway idempotency IDs.

## Solution Assessment

Yes, an additive `sessions_spawn` recovery identity is the narrowest maintainable direction. Gateway-level idempotency and the linked ClawHub skill are useful, but they do not make repeated parent spawn calls converge before a second child run is created.
