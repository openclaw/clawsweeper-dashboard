## Summary

Keep open; current main still only dedupes Gateway agent calls when the same idempotency key is reused, while the sub-agent spawn and orphan-recovery paths create fresh identities, so the reported recovery/idempotency gap remains a core reliability contract question.

## What This Changes

Request a stable idempotency/recovery mechanism so sub-agent timeout retries do not repeat side-effecting POST actions.

## Best Possible Solution

Define and document a stable sub-agent operation or action identity that survives timeout/recovery retries, propagate it through OpenClaw-owned side-effecting paths, and clarify where external services must still provide their own idempotency keys.

## Reproduction Assessment

No high-confidence end-to-end reproduction path is established from the checkout alone. Source inspection shows the gap: Gateway dedupes only reused agent idempotency keys, while `sessions_spawn` and orphan resume create fresh random keys.

## Solution Assessment

Unclear. The proposed directions are plausible, but maintainers need to choose the core contract before implementation; an ad hoc content dedupe would be less maintainable than a documented operation identity.
