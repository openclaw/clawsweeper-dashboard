## Summary

Keep open. Current main has scoped Gateway and completion-handoff dedupe, plus newer `taskName`/`sessions_yield` orchestration guidance, but the native sub-agent spawn and orphan-recovery paths still create fresh random identities, so timeout recovery still lacks a stable operation identity for side-effecting work.

## What This Changes

Add an idempotency/recovery contract for sub-agent side-effecting operations so lost acknowledgements do not create duplicate API posts.

## Best Possible Solution

Define and document a stable sub-agent operation or action identity that survives timeout/recovery retries, propagate it through OpenClaw-owned side-effecting Gateway/tool paths, and document where external services must still provide their own idempotency keys.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the issue context alone. Source inspection does show the current-main gap: Gateway dedupes reused keys, while `sessions_spawn` and orphan recovery still create fresh random identities.

## Solution Assessment

Unclear. The proposed directions are plausible, but maintainers need to choose the core contract between stable run/action identity propagation, endpoint-level dedupe windows, and plugin/tool guidance for external services.
