## Summary

Current main fixes one concrete lost-ack duplicate path for `message(action=send)`, but this issue asks for a broader identity and deduplication contract across replacement sub-agent runs and arbitrary external POST tools. That broader behavior is neither implemented nor reproducible from the report, and the target service must cooperate for server-side exactly-once semantics, so the issue remains useful but needs a maintainer product/ownership decision before code work.

## What This Changes

Prevent duplicate external side effects when sub-agent timeout recovery retries a logical POST whose original response may have been lost after commit.

## Best Possible Solution

Define a narrow logical-operation identity for explicitly supported recovery/retry paths, persist and propagate it through side-effecting tool contracts, and let each API owner implement durable claim/result semantics or documented preflight checks without adding a universal content-based suppression layer.

## Reproduction Assessment

No. The report gives production observations but not the OpenClaw tool, endpoint, recovery command, configuration, or controlled current-main sequence needed to reproduce a duplicate arbitrary POST; only the narrower message-tool retry class is source-backed and tested.

## Solution Assessment

No. A generic transport dedupe header or last-N heuristic is not sufficient for arbitrary APIs; the best direction is a scoped logical-operation contract with durable server cooperation at each side-effect owner.
