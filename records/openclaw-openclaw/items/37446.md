## Summary

Keep open. Current main still has Gateway-level dedupe when a caller reuses an idempotency key, but the sub-agent spawn/recovery/tool paths still create fresh random identities, so the core reliability gap described in the issue remains and needs maintainer contract design.

## What This Changes

The issue requests idempotency or retry-recovery protection so sub-agent timeout recovery cannot repeat side-effecting POST actions.

## Best Possible Solution

Define a stable sub-agent operation or action identity that survives timeout/recovery retries, propagate it through OpenClaw-owned side-effecting paths, and document where external services must still provide their own idempotency keys.

## Reproduction Assessment

No, there is not a high-confidence live reproduction from current main in this review. Source inspection does show the current gap: Gateway dedupes reused keys, while sessions_spawn, sessions_send, and orphan resume still generate fresh random identities for new attempts.

## Solution Assessment

Unclear. The proposed directions are plausible, but maintainers need to choose the core contract between stable run/action identity propagation, endpoint-level dedupe, and plugin/tool guidance for external services before implementation is safe.
