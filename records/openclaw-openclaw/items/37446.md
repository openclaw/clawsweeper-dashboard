## Summary

Keep open. Current main has Gateway idempotency-key requirements and exact-key dedupe, but sub-agent spawn and orphan recovery still mint fresh identities, so a lost acknowledgement or timeout-recovery replay can still become a distinct side-effecting operation; the linked ClawHub skill is a useful workaround, not a core recovery contract.

## What This Changes

Request a stable idempotency and recovery mechanism so sub-agent timeout retries do not repeat side-effecting POSTs or duplicate comments/actions.

## Best Possible Solution

Define a stable sub-agent operation or action identity that survives timeout and recovery retries for OpenClaw-owned side effects, and document where external APIs still require their own idempotency keys.

## Reproduction Assessment

No, not as a high-confidence live lost-ack reproduction. Source inspection does show the current-main path: Gateway dedupe is exact-key scoped while native spawn and orphan recovery can still generate fresh keys.

## Solution Assessment

Unclear. The proposed idempotency-key approaches are plausible, but the maintainable solution should be a shared retry/recovery identity contract rather than a one-off content match or caller shim.
