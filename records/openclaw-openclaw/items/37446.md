## Summary

Keep open. Current main dedupes repeated Gateway `agent` calls only when the same idempotency key is reused, while `sessions_spawn` and orphan recovery still mint fresh child/session keys, so the stable operation identity gap remains.

## What This Changes

Request a stable idempotency/recovery identity for sub-agent side-effecting POST retries so lost acknowledgements do not create duplicate actions.

## Best Possible Solution

Define and document a stable sub-agent operation or action identity that survives timeout/recovery retries, propagate it through OpenClaw-owned side-effecting Gateway/tool paths, and document where external services must still supply their own idempotency keys.

## Reproduction Assessment

No high-confidence end-to-end lost-ack reproduction is available from the issue context alone. Source inspection does show the current-main gap: Gateway dedupes reused keys, while `sessions_spawn` and orphan recovery still create fresh random identities.

## Solution Assessment

Unclear. The proposed directions are plausible, but maintainers need to choose the core contract between stable run/action identity propagation, endpoint-level dedupe windows, and plugin/tool guidance for external services.
