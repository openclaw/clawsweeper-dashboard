Full review comments:

- **[P1] Keep target parsing behind plugin contracts:** `src/channels/plugins/target-parsing.ts:5-7`
  - body: These top-level imports make a core command/binding path load private Discord, Telegram, and Teams extension modules directly. That bypasses the registered channel plugin contract and can drag disabled or optional channel runtime dependencies into core startup paths; use a lightweight public plugin artifact or registry-provided parser instead.
  - confidence: 0.87
- **[P1] Remove Teams host ops from the SDK facade:** `src/plugin-sdk/msteams.ts:111-118`
  - body: Exporting Teams send, edit, typing, and channel Graph operations through `openclaw/plugin-sdk/msteams` makes host credential-backed Teams actions available as public plugin API. This needs an explicit approved contract or a generic lane/channel capability before third-party plugins get this channel-specific authority surface.
  - confidence: 0.9
- **[P1] Migrate channel metadata before requiring it:** `extensions/msteams/src/graph-messages.ts:140-143`
  - body: Existing stored Teams channel conversations predate `graphChannelId`, but this branch hard-fails Graph channel operations when that field is absent. Add a doctor/backfill path or compatibility recovery before making the new metadata required, otherwise upgraded channel bindings can stop working until a fresh inbound activity happens to repopulate the record.
  - confidence: 0.88
- **[P2] Preserve graphChannelId on sparse activities:** `extensions/msteams/src/monitor-handler/message-handler.ts:288`
  - body: This upsert always includes `graphChannelId` from optional `activity.channelData.channel.id`, while the store merge path can overlay incoming fields onto existing ones. A Teams activity without channel metadata can therefore erase a previously known `graphChannelId` and cause later Graph channel operations to hit the new missing-metadata error.
  - confidence: 0.82
