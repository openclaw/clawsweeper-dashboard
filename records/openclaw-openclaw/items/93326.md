## Summary

Keep open: current `main` still has no guard for `.docodex`-style Office-extension corruption, and the related fix PRs are all closed unmerged. The issue is narrow enough for a focused repair, but I would route it as validation/retry guidance rather than silently rewriting shell commands.

## What This Changes

The issue reports that DOCX-related tasks can corrupt `.docx` paths into `.docodex` before file-tool or shell execution.

## Best Possible Solution

Add a narrow OpenClaw file-tool path guard that rejects or revalidates known corrupted Office extensions with actionable retry guidance, while leaving broad shell-command rewriting out unless maintainers explicitly approve it.

## Reproduction Assessment

Yes from source inspection, but not from a live run: current file tools accept any non-empty `path` string and would pass `report.docodex` to filesystem resolution rather than blocking or correcting it.

## Solution Assessment

No final implementation is present yet. The best repair appears to be narrow validation in OpenClaw’s file-tool parameter or workspace-guard path, not broad shell-command rewriting or silent autocorrection without maintainer approval.
