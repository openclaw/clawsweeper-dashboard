## Summary

Current main and the latest release still pass `.docodex` through the file-tool path pipeline, and the only related fix PRs are closed unmerged, so this issue remains actionable rather than cleanup-closeable.

## What This Changes

The issue reports that DOCX filenames can be corrupted from `.docx` to `.docodex` before OpenClaw executes file-tool or shell operations.

## Best Possible Solution

Add a narrow, tested file-tool path guard or correction in shared parameter handling, and leave shell-command mitigation to a separate parser-backed design unless the fix can prove it is safe.

## Reproduction Assessment

No, not as a high-confidence live model reproduction: the report has no transcript or logs and the model rewrite is intermittent. Source inspection does show the guard gap once `.docodex` is supplied as a file-tool path.

## Solution Assessment

Yes for a bounded first repair: shared file-tool parameter validation is the narrowest maintainable place to catch this before filesystem execution. Shell command handling should not be silently rewritten without a stronger command parser boundary.
