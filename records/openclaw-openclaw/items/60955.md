## Label Justifications

- `P1`: The issue affects a real Telegram/direct agent workflow by letting the user-visible transcript imply execution even when the turn ledger has no tool activity.
- `impact:session-state`: The assistant transcript can claim command execution while session/tool state records no corresponding tool call or result.

## Summary

Keep open. Current main and the latest stable release still treat tool-looking assistant text as warning-only diagnostics, and the visible reply paths can still deliver first-person shell-fence execution text without same-turn tool lifecycle evidence; the focused fix PR is closed unmerged.

## What This Changes

The issue requests that Telegram/direct chat stop delivering exec-looking assistant text as an operational reply when the same turn has no real tool calls or tool results.

## Best Possible Solution

Add a channel-agnostic embedded reply guard that replaces or blocks strong execution-intent shell-fence replies when the current assistant turn lacks real tool lifecycle evidence, with coverage for text_end, message_end, and tool-backed bypass cases.

## Reproduction Assessment

Yes, at source level. A no-tool assistant message with first-person install/check/run text plus a fenced shell command can still flow through warning-only diagnostics and visible reply emission on current main and v2026.5.18.

## Solution Assessment

Yes. A narrow embedded visible-output guard is the maintainable fix because it addresses the Telegram/direct truthfulness bug without changing exec policy, provider defaults, or the broader strict-agentic contract.
