## Summary

Keep this open. Current main still has warning-only diagnostics and embedded reply paths that can deliver execution-intent fenced shell text without same-turn tool lifecycle evidence, while the focused fix PR was closed unmerged.

## What This Changes

Prevent Telegram direct/main sessions from delivering execution-intent fenced shell replies unless the same turn has real tool-call or tool-result evidence.

## Best Possible Solution

Add a channel-agnostic guard at the embedded assistant reply delivery boundary that rewrites or blocks strong execution-intent shell-fence replies unless the same turn has real tool lifecycle evidence.

## Reproduction Assessment

Yes. Reproduce at source level by passing first-person install/check/run text plus a fenced bash command through the embedded message_end or text_end delivery path with no tool_execution_start/end, then observing the reply remains deliverable.

## Solution Assessment

Yes. A narrow embedded delivery-seam guard is the best direction because it fixes Telegram/direct delivery truthfulness without changing exec policy, provider defaults, or legitimate tool-backed replies.

Fix openclaw/openclaw#60955. Current main can deliver Telegram/direct embedded assistant replies that say they will install/check/run something and include a runnable fenced shell/bash block even when the turn has no real tool activity. Implement a narrow channel-agnostic guard at the embedded assistant reply delivery boundary, likely around src/agents/pi-embedded-subscribe.ts and src/agents/pi-embedded-subscribe.handlers.messages.ts, with any detector helper in src/agents/pi-embedded-utils.ts or a small local helper.
