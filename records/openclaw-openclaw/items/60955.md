## Summary

Keep this open. Current main still has the embedded assistant reply paths that can deliver visible exec-looking shell-fence text without a tool-activity gate, and the linked focused fix PR was closed unmerged rather than landing the guard.

## What This Changes

The issue requests preventing Telegram/direct assistant replies from presenting exec-looking shell output as operational work when the turn has no real tool calls or tool results.

## Best Possible Solution

Add a channel-agnostic guard at the embedded assistant reply delivery boundary that rewrites strong execution-intent shell-fence replies to a truthful blocked/error response unless the same turn has real tool lifecycle evidence.

## Reproduction Assessment

Yes. Reproduce at source level by sending an assistant message with first-person install/check/run text plus a fenced bash command through `subscribeEmbeddedPiSession`/`handleMessageEnd` with no `tool_execution_start` or `tool_execution_end`, then observing the block reply remains deliverable.

## Solution Assessment

Yes. A narrow #61132-style guard at the embedded delivery seam is the best direction because it fixes Telegram/direct without changing exec tool policy, provider defaults, or legitimate tool-backed replies.
