## Summary

Keep this open. Current main still treats tool-looking assistant text as warning-only diagnostics and the embedded message delivery paths can still emit execution-intent fenced shell text without same-turn tool lifecycle evidence; the focused fix PR was closed unmerged, and related broader issues now point back to this as the canonical remaining bug.

## What This Changes

The issue asks OpenClaw to prevent Telegram/direct replies from looking like command execution unless the same turn has real tool activity or an explicit blocked/error state.

## Best Possible Solution

Add a channel-agnostic guard at the embedded assistant reply delivery boundary that rewrites or blocks strong execution-intent shell-fence replies unless the same turn has real tool lifecycle evidence, while preserving legitimate tool-backed replies and benign command examples.

## Reproduction Assessment

Yes, at source level. Pass first-person install/check/run text plus a fenced shell command through the embedded message_end or text_end delivery path with no tool_execution_start/end events, and current main still reaches visible reply emission after only warning-only diagnostics.

## Solution Assessment

Yes. A narrow embedded delivery-seam guard is the best fix because it addresses the Telegram/direct truthfulness bug without changing exec policy, provider defaults, broad strict-agentic semantics, or legitimate tool-backed replies.
