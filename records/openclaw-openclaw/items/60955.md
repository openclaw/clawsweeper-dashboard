## Summary

Keep this open. The reported Telegram/direct failure mode still appears valid on current main: the embedded assistant reply path emits visible assistant text and block replies without tying exec-looking fenced shell text to a real tool lifecycle, while the linked focused fix PR is closed unmerged.

## What This Changes

The issue requests preventing Telegram direct replies that look like command execution from being delivered unless the same turn has real tool activity or an explicit blocked/error state.

## Best Possible Solution

Add a channel-agnostic guard at the embedded assistant reply delivery boundary that recognizes strong execution-intent prose plus runnable shell fences when there has been no real tool lifecycle, replaces it with a truthful blocked/error reply, and preserves tool-backed and benign example output.

## Reproduction Assessment

Yes. The report gives a concrete real Telegram session timeline, and current main has a focused source-level reproduction path: send an assistant message containing first-person install/check text plus a fenced bash command through the embedded message_end/text_end block-reply path with no tool events and observe that delivery is still eligible.

## Solution Assessment

Yes. The narrow #61132-style direction is the maintainable fix: guard at the embedded delivery boundary, make it provider/channel agnostic, bypass it for actual tool activity, and cover both blocked and tool-backed cases in tests.
