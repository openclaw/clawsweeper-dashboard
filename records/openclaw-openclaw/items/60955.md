## Summary

Keep open. Current main still only warns about tool-looking assistant text and both embedded visible-reply paths can still emit first-person shell-fence execution text without same-turn tool lifecycle evidence; the focused fix PR is closed unmerged.

## What This Changes

The issue reports Telegram/direct assistant replies that look like shell execution even though the turn has no real tool calls or tool results.

## Best Possible Solution

Add a channel-agnostic embedded reply guard that rewrites or blocks strong execution-intent shell-fence replies unless the same assistant turn has real tool lifecycle evidence, while preserving legitimate tool-backed replies and command examples.

## Reproduction Assessment

Yes, at source level. A no-tool assistant message with first-person install/check/run text plus a fenced shell command still passes through message_end or text_end visible reply emission after warning-only diagnostics on current main.

## Solution Assessment

Yes. A narrow embedded delivery-seam guard is the maintainable fix because it addresses the Telegram/direct truthfulness bug without changing exec policy, provider defaults, or the broader strict-agentic contract.
