- `impact:session-state`: Assistant-visible transcript state can claim execution while session/tool state records no matching tool call or result.

## Summary

Keep open: current `main` and the latest release still only warn on some tool-call-shaped assistant text and then continue normal visible reply delivery, while the reported Telegram shape is first-person execution intent plus a fenced shell command with no same-turn tool activity. The prior focused fix PR was closed unmerged, and the broader simulated-tool policy remains tracked separately.

## What This Changes

The issue requests a guard that prevents Telegram/direct replies from showing exec-looking shell text as operational output unless the same assistant turn has real tool-call or tool-result evidence.

## Best Possible Solution

Add a channel-agnostic embedded visible-reply guard that replaces or blocks strong first-person execution-intent shell-fence replies when the current assistant turn lacks real tool lifecycle evidence, with explicit bypass coverage for real tool-backed turns.

## Reproduction Assessment

Yes, at source level. A no-tool assistant message with first-person install/check/run text plus a fenced shell command can still pass through current visible reply paths without a same-turn tool-activity guard.

## Solution Assessment

Yes. A narrow visible-output guard is the maintainable fix because it addresses the Telegram/direct truthfulness bug without changing provider defaults, exec policy, or the broader strict-agentic contract.
