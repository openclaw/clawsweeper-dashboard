## Summary

Keep this open: current main still has warning-only diagnostics and embedded reply paths that can deliver exec-looking fenced shell text without same-turn tool lifecycle evidence, while the focused fix PR was closed unmerged.

## What This Changes

The issue requests preventing Telegram direct/main sessions from delivering first-person exec-looking fenced shell replies unless a real tool call or blocked/error state exists.

## Best Possible Solution

Add a channel-agnostic guard at the embedded assistant reply delivery boundary that rewrites or blocks strong execution-intent shell-fence replies unless the same turn has real tool lifecycle evidence.

## Reproduction Assessment

Yes. Reproduce at source level by sending first-person install/check/run text plus a fenced bash command through the embedded message_end or text_end delivery path with no tool_execution_start/end, then observing that current main still delivers the reply.

## Solution Assessment

Yes. A narrow embedded delivery-seam guard is the maintainable direction because it fixes Telegram/direct truthfulness without changing exec policy, provider defaults, or legitimate tool-backed replies.
