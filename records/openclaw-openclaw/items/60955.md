## Summary

Keep this open. Current main still has only warning-only diagnostics for tool-looking assistant text, and both embedded block-reply delivery paths can still emit first-person execution-intent shell fences without same-turn tool lifecycle evidence; the focused fix PR remains closed unmerged.

## What This Changes

The issue requests preventing Telegram/direct embedded replies from presenting exec-looking shell commands as operational output when the turn produced no real tool calls or tool results.

## Best Possible Solution

Add a channel-agnostic guard at the embedded assistant reply delivery boundary that rewrites or blocks strong execution-intent shell-fence replies unless the same assistant turn has real tool lifecycle evidence, while preserving legitimate tool-backed replies and command examples.

## Reproduction Assessment

Yes, at source level. Passing first-person install/check/run text plus a fenced shell command through message_end or text_end with no tool_execution_start/end evidence still reaches visible reply emission after only warning-only diagnostics.

## Solution Assessment

Yes. A narrow embedded delivery-seam guard is the maintainable fix because it addresses the Telegram/direct truthfulness bug without changing exec policy, provider defaults, or the broader strict-agentic contract.
