## Summary

Keep open: the PR targets a real Telegram fallback bug, but its generic numeric errno collection can suppress existing retry fallback for numeric-only retryable network errors, and the supplied proof does not exercise the production transport/classifier path.

## What This Changes

The PR changes Telegram fetch fallback classification so EADDRNOTAVAIL string or Linux numeric errno 99 skips alternative Telegram API IP retry, with two regression tests.

## Best Possible Solution

Use the Telegram fetch classifier as the owner boundary, but make local-socket failure detection narrow enough that remote-reachability fallback remains unchanged for API and media paths.

## Reproduction Assessment

Yes for source-level reproduction: current main retries code-less `fetch failed` envelopes, while this PR changes numeric-only errno classification in the shared Telegram fallback predicate. I did not establish a live Telegram outage reproduction in this read-only pass.

## Solution Assessment

No: the owner module is the right layer, but collecting all numeric errnos as raw strings is too broad for the symbolic fallback predicate. The safer fix is narrow local-socket failure detection or explicit errno-to-symbol normalization while preserving remote-reachability fallback.
