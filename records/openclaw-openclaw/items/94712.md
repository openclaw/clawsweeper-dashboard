## Summary

Keep open: this PR targets a real, still-open Telegram transport fallback bug, but the branch is not merge-ready because the generic numeric errno collection can disable existing retry fallback for numeric-only retryable network errors, and the contributor has only supplied mocked/unit-test proof.

## What This Changes

The PR adds numeric errno collection to Telegram fetch fallback classification, fails fast for EADDRNOTAVAIL/99, and adds two mocked fetch tests.

## Best Possible Solution

Preserve retry behavior for numeric-only remote-reachability errnos while suppressing local socket allocation failures, then land one reviewed candidate with real transport proof or maintainer override.

## Reproduction Assessment

Yes for source-level review: current main retries code-less fetch failed envelopes with no collected codes, while this PR makes numeric-only retryable errnos non-empty and non-matching. I did not establish a live Telegram outage reproduction in this read-only review.

## Solution Assessment

No: the owner module is the right layer, but collecting all numeric errnos as raw strings is too broad for the existing fallback predicate. The safer fix is to map numeric retryable errnos to symbols or narrowly detect local socket allocation failures while preserving remote-reachability fallback.
