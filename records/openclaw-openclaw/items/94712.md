## Summary

Keep open: the PR targets a real Telegram fallback bug, but it is not merge-ready because the raw numeric errno collection can suppress existing retry fallback for numeric-only retryable network errors, and the external PR has only unit-test evidence rather than real Telegram/socket transport proof.

## What This Changes

The PR adds numeric errno collection to Telegram fetch fallback classification, suppresses fallback for EADDRNOTAVAIL/99, and adds two mocked fetch tests.

## Best Possible Solution

Preserve retry behavior for numeric-only retryable errnos while suppressing local socket allocation failures, add focused regression coverage for both paths, and require real Telegram/socket transport proof or maintainer override before merge.

## Reproduction Assessment

Yes for source-level review: a code-less fetch failed envelope with a numeric-only retryable errno would retry on current main but not after this PR because ctx.codes becomes non-empty without a symbolic fallback code. I did not run a live Telegram outage reproduction in this read-only review.

## Solution Assessment

No: the owner module is the right place, but adding all numeric errnos as raw strings is too broad and changes existing fallback classification. The safer fix is to normalize numeric errno values to symbolic codes or narrowly special-case local socket allocation failures while preserving retryable remote-reachability cases.
