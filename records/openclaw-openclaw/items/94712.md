## Summary

Keep this PR open: it targets a real Telegram fallback bug, but the current patch is not merge-ready because it broadly adds raw numeric errno values to a symbolic retry classifier, lacks real transport-path proof, and overlaps an open sibling candidate for the same report.

## What This Changes

The PR adds numeric errno collection to Telegram fetch fallback, suppresses fallback for EADDRNOTAVAIL/99, and adds two regression tests.

## Best Possible Solution

Keep the fix in the Telegram fetch classifier, but normalize or narrowly map EADDRNOTAVAIL detection while preserving retryable network fallback behavior, then require real transport-path proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes for source-level reproduction: current main retries symbolic retry codes and code-less `fetch failed` envelopes through the Telegram fallback predicate, while this PR changes that shared predicate. I did not run a live socket-exhaustion or Telegram transport reproduction in this read-only review.

## Solution Assessment

No: the Telegram fetch module is the right owner boundary, but collecting every numeric errno as a raw code is too broad for a symbolic retry allowlist. The safer fix is explicit errno-to-symbol normalization or a narrow local-socket guard that preserves existing retryable fallbacks.
