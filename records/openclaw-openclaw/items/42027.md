## Summary

Keep open: the browser diagnostic/error work is still useful, but the submitted patch targets obsolete core browser paths while current main has moved the active implementation to `extensions/browser`; the exec and cron parts are already superseded by different mainline fixes.

## What This Changes

The PR changes exec PATH recovery, browser status diagnostics and HTTP error summarization, and cron manual force-run lane scheduling, with regression tests across those areas.

## Best Possible Solution

Replace or split the PR: keep current main exec auto-routing and CronNested behavior, and port only the missing browser layered status diagnostics plus bounded HTML upstream-error summarization into `extensions/browser`.

## Reproduction Assessment

Yes, by source inspection. The active `extensions/browser` status route/type/CLI lack layered diagnostics and `client-fetch` still throws raw non-rate-limit response text; the exec and cron failures are not reproduced because current main solves them differently.

## Solution Assessment

No, not as submitted. The browser direction is right, but the patch edits obsolete core browser paths and introduces timeout/unbounded-output issues, so a narrow `extensions/browser` replacement is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reuse the status timeout for config snapshots:** `src/cli/browser-cli-shared.ts:87`
  - body: The new `browser status` path fetches `config.get` without the same bounded timeout used for the status request. If the gateway config path is slow or degraded, the command can hang for the default RPC timeout before falling back, which is a user-visible responsiveness regression.
  - confidence: 0.88
- **[P3] Cap HTML fallback summaries:** `src/browser/client-fetch.ts:156-163`
  - body: When an HTML error page has no title or h1, the fallback strips tags and returns the full remaining body. Large proxy pages can still produce multi-kilobyte browser errors, so the fallback should sanitize and cap the text before throwing it.
  - confidence: 0.77
