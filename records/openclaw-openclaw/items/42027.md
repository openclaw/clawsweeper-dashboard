## Summary

Keep open: the direct PR is not correct against current main. The exec and cron pieces are already superseded by current auto-host and CronNested behavior, while the useful browser diagnostics and HTML-error summarization work still needs a plugin-layout port under `extensions/browser`.

## What This Changes

The PR changes exec PATH recovery, browser status diagnostics and HTML browser-service error summarization, and cron force-run lane scheduling, with regression tests across those areas.

## Best Possible Solution

Replace or split the PR: leave current main exec auto-routing and CronNested behavior unchanged, and port only the missing browser layered status diagnostics plus bounded HTML upstream-error summarization into `extensions/browser`.

## Reproduction Assessment

Yes. Source inspection on current main shows the remaining browser gap: the active status route/type/CLI lack layered diagnostics and `client-fetch` still throws raw non-rate-limit response text; the exec and cron failures are not reproduced because current main solves them differently.

## Solution Assessment

No, not as-is. The browser direction is useful, but the submitted patch edits obsolete core paths and has timeout/unbounded-output defects; a narrow `extensions/browser` replacement is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reuse the status timeout for config snapshots:** `src/cli/browser-cli-shared.ts:87`
  - body: The new `browser status` path fetches `config.get` without passing the same bounded timeout used for the status request. If the gateway config path is slow or degraded, the diagnostic command can hang for the default RPC timeout before falling back, which is a user-visible responsiveness regression.
  - confidence: 0.88
- **[P3] Cap the HTML fallback summary:** `src/browser/client-fetch.ts:156-163`
  - body: When an HTML error page has no title or h1, the fallback strips tags and returns the full remaining body. Large or unusual proxy pages can still produce multi-kilobyte browser errors, so the fallback should sanitize and cap the text before throwing it.
  - confidence: 0.77
