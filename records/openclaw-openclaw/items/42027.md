## Summary

Keep open: this PR still has useful intent, but it is not mergeable as submitted because exec and cron are already handled on current main through different paths, while the remaining browser diagnostics work targets removed core browser/CLI paths and lacks contributor real-behavior proof.

## What This Changes

The PR modifies legacy core exec, browser, CLI, and cron lane code to add login-shell PATH recovery, layered browser status diagnostics with HTML error summarization, and a separate manual cron lane.

## Best Possible Solution

Replace or split this branch: keep current main's exec auto-routing and CronNested lane behavior, and port only the remaining browser diagnostics plus bounded HTML upstream-error summaries to extensions/browser or the narrower follow-up at https://github.com/openclaw/openclaw/issues/59209.

## Reproduction Assessment

Yes, by source inspection for the review findings: current main removed the core browser paths this PR edits, the active browser plugin lacks the proposed diagnostics payload, and the PR body lacks proof artifacts. I did not live-reproduce WSL2 + Windows CDP behavior in this review.

## Solution Assessment

No, not as submitted. The narrow maintainable path is to leave the already-landed exec and cron fixes alone and port the remaining browser work to the active browser plugin with bounded, redacted diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Port diagnostics to the active browser plugin:** `src/browser/routes/basic.ts:74-79`
  - body: Current main serves browser status from extensions/browser, while this PR adds the diagnostics payload under the removed src/browser route. After a rebase, the active `openclaw browser status` path would still have no layered diagnostics, so the remaining browser fix needs to move to the plugin route, type, and CLI.
  - confidence: 0.92
- **[P2] Bound the config snapshot with the status timeout:** `src/cli/browser-cli-shared.ts:87`
  - body: The new status diagnostics path fetches `config.get` without the short timeout used by the browser status request. If the gateway config path is slow or partially unavailable, `openclaw browser status` can wait on the default RPC timeout before falling back, which is a responsiveness regression.
  - confidence: 0.86
- **[P3] Cap HTML fallback summaries:** `src/browser/client-fetch.ts:157-163`
  - body: When an HTML error page has no title or h1, the fallback strips tags and returns the full remaining body. Large proxy pages can still produce multi-kilobyte browser errors, so the fallback should sanitize entities and cap the text before throwing it.
  - confidence: 0.78
