## Summary

Keep open for maintainer handling: current main covers the exec and cron parts by different implementations, but the browser diagnostics are still not implemented in the active browser plugin and this stale branch has blocking review findings plus missing real behavior proof.

## What This Changes

The PR changes exec target PATH recovery, browser status/HTTP diagnostics, and cron force-run lane scheduling, with related regression tests.

## Best Possible Solution

Keep the shipped exec auto-target and cron nested-lane behavior, then replace or split the stale browser diagnostic work into the active browser plugin with bounded HTTP summaries and real-world proof.

## Reproduction Assessment

Yes, for the PR blockers by source inspection: the branch wires browser diagnostics into legacy src/browser paths absent from current main, adds an unbounded config.get status call, and leaves HTML fallback summaries uncapped. I did not live-run the Windows/WSL2 CDP or cron setup.

## Solution Assessment

No, not as submitted. The maintainable path is to keep main's exec and cron fixes, then port any remaining browser diagnostics and bounded HTML error handling to extensions/browser or track a narrower follow-up.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Port diagnostics to the active browser plugin:** `src/browser/routes/basic.ts:74-79`
  - body: Current main serves browser status from extensions/browser, while this PR adds the diagnostics payload under the legacy src/browser route. After a rebase, openclaw browser status would still miss the proposed diagnostics unless the route, type, and CLI changes move to the plugin.
  - confidence: 0.92
- **[P2] Reuse the status timeout for config snapshots:** `src/cli/browser-cli-shared.ts:61-64`
  - body: The new browser status diagnostics path fetches config.get without the bounded timeout used by the browser status request. If gateway config is slow or partially unavailable, openclaw browser status can wait on the longer default RPC timeout before falling back, which is a responsiveness regression.
  - confidence: 0.86
- **[P3] Cap HTML fallback summaries:** `src/browser/client-fetch.ts:157-163`
  - body: When an HTML error page has no title or h1, the fallback strips tags and returns the full remaining body. Large proxy pages can still produce multi-kilobyte browser errors, so the fallback should sanitize entities and cap the text before throwing it.
  - confidence: 0.78
