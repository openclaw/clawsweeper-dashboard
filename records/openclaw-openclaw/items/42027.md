## Summary

Keep open for maintainer handling: current main covers the exec fallback and cron deadlock classes, but the PR still contains unique browser diagnostics work, targets obsolete `src/browser` paths, lacks real behavior proof, and has blocking review findings.

## What This Changes

The PR changes exec PATH fallback, browser status and HTML error diagnostics, CLI browser status config reads, and cron force-run lane scheduling with regression tests.

## Best Possible Solution

Keep the shipped exec auto-target and cron nested-lane fixes, then rework or split the remaining browser diagnostics into `extensions/browser` with bounded error summaries, status-timeout preservation, and real-world proof.

## Reproduction Assessment

Yes, for the PR blockers by source inspection: current main no longer has `src/browser`, the branch adds an unbounded browser `config.get` status call, and its HTML fallback summary is uncapped. I did not live-run the Windows/WSL2 CDP or cron setup.

## Solution Assessment

No, not as submitted. The maintainable path is to keep main's exec and cron fixes, then port any remaining browser diagnostics and bounded HTML error handling to the active browser plugin or track them as narrower follow-up work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Port diagnostics to the active browser plugin:** `src/browser/routes/basic.ts:74-79`
  - body: Current main serves browser status from `extensions/browser`, while this PR adds the diagnostics payload under the legacy `src/browser` route. After a rebase, `openclaw browser status` would still miss the proposed diagnostics unless the route, type, and CLI changes move to the active browser plugin.
  - confidence: 0.92
- **[P2] Reuse the status timeout for config snapshots:** `src/cli/browser-cli-shared.ts:61-64`
  - body: The new browser status diagnostics path fetches `config.get` without the bounded timeout used by the browser status request. If gateway config is slow or partially unavailable, `openclaw browser status` can wait on the longer default RPC timeout before falling back, which is a responsiveness regression.
  - confidence: 0.86
- **[P3] Cap HTML fallback summaries:** `src/browser/client-fetch.ts:157-163`
  - body: When an HTML error page has no title or h1, the fallback strips tags and returns the full remaining body. Large proxy pages can still produce multi-kilobyte browser errors, so the fallback should sanitize entities and cap the text before throwing it.
  - confidence: 0.78
