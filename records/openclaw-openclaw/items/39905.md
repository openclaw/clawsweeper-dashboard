## Summary

Keep open. The per-conversation Telegram reply/streaming config request is not implemented on current main, but this branch is not mergeable as-is because it targets obsolete runtime paths, uses the old scalar streaming shape for new scoped config, lacks real Telegram proof, and mixes in unrelated secrets changes.

## What This Changes

The PR adds Telegram direct/group/topic `replyToMode` and `streaming` overrides, routes them into normal/native dispatch, documents the config surface, adjusts quoted-DM streaming, and includes unrelated secrets/test churn.

## Best Possible Solution

Ship a focused Telegram plugin change in `extensions/telegram/src` that layers scoped `replyToMode` and nested `streaming` overrides over account defaults for normal/native replies, omits unrelated secrets changes, updates docs/tests, and includes redacted real Telegram proof.

## Reproduction Assessment

Yes at source level for the current gap: current main's Telegram scoped schemas/types do not accept `replyToMode` or `streaming`, and normal/native dispatch still receives account-level values only. Live Telegram behavior proof is still missing.

## Solution Assessment

No. The proposed branch targets obsolete `src/telegram` files and legacy scalar streaming config; the maintainable solution is a focused plugin implementation using the nested streaming contract after maintainer config-surface review.
