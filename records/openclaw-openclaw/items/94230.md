## Summary

Keep open. The branch is a focused MCP catalog latency mitigation and current main still serializes this work, but merge should wait for slow-MCP after-fix proof and maintainer acceptance of the higher startup fan-out.

## What This Changes

This PR restructures `SessionMcpRuntime.getCatalog()` to precompute server metadata and run configured MCP server connection plus tools/list work concurrently before merging catalog results.

## Best Possible Solution

Land a bounded parallel MCP catalog fix after deterministic or real slow-MCP proof shows max-of-servers timing and maintainers accept the fan-out tradeoff; track lazy loading and warm pooling separately.

## Reproduction Assessment

Yes. Current main awaits each configured MCP server's connect and tools/list inside one `getCatalog()` loop on the `bundle-tools` path; I did not run a live latency reproduction.

## Solution Assessment

Yes as a bounded mitigation, but not as the complete issue solution. Parallelizing independent MCP discovery addresses the immediate sum-of-servers path; lazy loading, warm pooling, or a simple-chat fast path should remain maintainer-scoped follow-up work.
