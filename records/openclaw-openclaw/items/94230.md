## Label Justifications

- `P2`: The PR targets noticeable per-request agent preparation latency in MCP-heavy setups with a bounded agent-runtime blast radius.
- `merge-risk: 🚨 availability`: The diff changes configured MCP server startup and tools/list work from serialized execution to concurrent fan-out, which can affect resource pressure and stalls in real environments.

## Summary

Keep open: current main still serializes bundled MCP catalog loading and this PR is a useful bounded candidate, but it should not merge until it has after-fix multi-server timing proof and maintainer acceptance of the startup fan-out tradeoff.

## What This Changes

This PR refactors `createSessionMcpRuntime().getCatalog()` to precompute MCP server metadata and connect/list tools for configured MCP servers concurrently via `Promise.allSettled` before merging catalog entries.

## Best Possible Solution

Land a bounded parallel MCP catalog fix after deterministic or real slow-MCP proof shows max-of-servers timing and maintainers accept the fan-out tradeoff; track lazy loading and warm pooling separately.

## Reproduction Assessment

Yes. Current main awaits each configured MCP server's connect and tools/list inside one `getCatalog()` loop on the `bundle-tools` path; I did not run a live latency reproduction in this read-only review.

## Solution Assessment

Yes as a bounded mitigation, but not as the complete issue solution. Parallelizing independent MCP discovery addresses the immediate sum-of-servers path; lazy loading, warm pooling, or a simple-chat fast path should remain maintainer-scoped follow-up work.
