## What This Changes

The PR parallelizes configured MCP server connection and tool listing inside `SessionMcpRuntime.getCatalog()` while preserving per-server catalog diagnostics.

## Best Possible Solution

Land a narrow parallel catalog-loading fix only after redacted real or deterministic slow-MCP proof shows max-of-servers timing while preserving diagnostics, disposal, and catalog invalidation; keep lazy loading and warm pooling as separate follow-up work.

## Reproduction Assessment

## Solution Assessment

Yes as an acceptable bounded mitigation, but not as the complete best end state for the linked issue. Parallelizing independent server discovery addresses the immediate sum-of-timeouts path; lazy loading, warm pooling, or a simple-chat fast path should remain separate maintainer-approved work.
