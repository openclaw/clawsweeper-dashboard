## Summary

Keep this PR open: current main and the latest release still serialize MCP catalog loading, and no merged or proof-positive replacement supersedes this branch. The patch is a plausible bounded mitigation, but it still needs real or deterministic slow-MCP timing proof and maintainer acceptance of the concurrent startup fan-out tradeoff before merge.

## What This Changes

The PR changes `SessionMcpRuntime.getCatalog()` to precompute MCP server names, then connect and list configured MCP servers concurrently with `Promise.allSettled`.

## Best Possible Solution

Land a narrow parallel catalog-loading fix only after redacted real or deterministic slow-MCP proof shows max-of-servers timing while preserving diagnostics, disposal, and catalog invalidation; keep lazy loading and warm pooling as separate follow-up work.

## Reproduction Assessment

Yes from source for the central behavior: current main and `v2026.6.8` await MCP connect and tools/list for each configured server inside one loop. I did not run a live before/after latency reproduction in this read-only review.

## Solution Assessment

Yes as an acceptable bounded mitigation, but not as the complete best end state for the linked issue. Parallelizing independent server discovery addresses the immediate sum-of-timeouts path; lazy loading, warm pooling, or a simple-chat fast path should remain separate maintainer-approved work.
