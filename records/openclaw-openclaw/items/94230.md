## Summary

Keep this PR open: it targets a real current-main and latest-release bottleneck in bundled MCP catalog loading, but it is still a live merge candidate rather than stale cleanup. The remaining blockers are stronger after-fix multi-server latency proof and maintainer acceptance of the concurrent startup fan-out risk.

## What This Changes

The branch rewrites `SessionMcpRuntime.getCatalog()` to precompute MCP server names first, then connect and list configured MCP servers concurrently with `Promise.allSettled` while returning per-server diagnostics.

## Best Possible Solution

Land a narrow parallel catalog-loading fix only after redacted real or deterministic slow-MCP proof shows max-of-servers timing while preserving diagnostics, disposal, and catalog invalidation; keep lazy loading and warm pooling as separate follow-up work.

## Reproduction Assessment

Yes from source for the central behavior: current main and `v2026.6.8` await MCP connect and `tools/list` for each configured server inside one loop. I did not establish a live before/after latency reproduction in this read-only review.

## Solution Assessment

Yes as an acceptable bounded mitigation, but not as the complete best end state for the linked issue. Parallelizing independent server discovery addresses the immediate sum-of-timeouts path; lazy loading, warm pooling, or a simple-chat fast path should remain separate maintainer-approved work.
