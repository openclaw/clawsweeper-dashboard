## Summary

Keep this PR open: the MCP catalog parallelization is a plausible bounded performance fix, but the contributor proof still does not show an after-fix slow multi-server MCP run where bundle-tools wall time drops to max-of-server timing.

## What This Changes

The PR rewrites `SessionMcpRuntime.getCatalog()` to precompute MCP server metadata, connect/list configured servers concurrently with `Promise.allSettled`, and merge per-server catalog results afterward.

## Best Possible Solution

Land the narrow parallel catalog-loading fix after redacted slow-MCP proof confirms max-of-servers timing and lifecycle behavior; keep lazy loading and warm pooling as separate maintainer-sponsored work.

## Reproduction Assessment

Yes from source for the central behavior: current main awaits each configured MCP server's connect and tools/list work inside one `getCatalog()` loop. I did not establish a live before/after latency run in this read-only review.

## Solution Assessment

Yes as a bounded mitigation for the sequential per-server I/O path, but not as the complete lazy-loading or warm-pool solution requested by the linked issue. The maintainable path is to land this only after slow-MCP behavior proof and keep broader loading strategy separate.
