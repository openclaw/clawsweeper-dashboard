## Summary

## What This Changes

This PR refactors `SessionMcpRuntime.getCatalog()` to precompute MCP server metadata, connect/list configured servers concurrently, and adds a timing regression test plus standalone timing script.

## Best Possible Solution

Land a bounded parallel MCP catalog fix after the timing test asserts max-of-server behavior, redacted output proves the changed path, and maintainers accept the fan-out tradeoff; track lazy loading and warm pooling separately.

## Reproduction Assessment

Yes from source for the central behavior: current main awaits each configured MCP server connection and tools/list call inside one loop on the `bundle-tools` path. The new test defect is also source-reproducible because its threshold is above the serialized delay sum.

## Solution Assessment

No as submitted. Parallelizing independent MCP discovery is an acceptable bounded mitigation, but the regression test and proof need correction before this is the best merge candidate; broader lazy loading remains separate maintainer-scoped work.
