## Summary

Keep open: current main still performs per-server MCP catalog connection and tools/list sequentially, and this PR is a plausible candidate fix for the linked latency issue, but the supplied proof does not show an after-fix multi-server latency run and the runtime fan-out change still needs maintainer proof before merge.

## What This Changes

The branch rewrites `SessionMcpRuntime.getCatalog()` to precompute server names, connect and list configured MCP servers through `Promise.allSettled`, and merge the per-server catalog results afterward.

## Best Possible Solution

Land a narrow parallel catalog-loading fix after real or deterministic slow-MCP proof shows max-of-servers behavior while preserving diagnostics, disposal, and catalog invalidation; keep lazy loading and warm pooling as separate maintainer-sponsored work.

## Reproduction Assessment

Yes from source for the central behavior: current main awaits MCP connect and tools/list for each configured server inside one `getCatalog()` loop. I did not establish a live before/after latency reproduction in this read-only review.

## Solution Assessment

Yes as an acceptable bounded mitigation, but not the complete best end state for the linked issue. Parallelizing independent server discovery addresses the immediate sum-of-timeouts path; lazy loading or a simple-chat fast path should remain separate maintainer-approved work.
