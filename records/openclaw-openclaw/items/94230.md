## Summary

Keep open: current main and the latest release still serialize the MCP catalog loop, and this PR is a plausible narrow fix for the linked latency issue. It should not be closed or superseded, but it still needs real or deterministic slow-MCP timing proof plus maintainer acceptance of the concurrent fan-out availability tradeoff before merge.

## What This Changes

The PR restructures `SessionMcpRuntime.getCatalog()` to precompute MCP server names synchronously, then connect to configured MCP servers and list tools concurrently with `Promise.allSettled`.

## Best Possible Solution

Land a bounded parallel MCP catalog-loading fix only after redacted real or deterministic slow-MCP proof shows max-of-servers timing while preserving diagnostics, disposal, and catalog invalidation; track lazy loading and warm pooling separately.

## Reproduction Assessment

Yes from source for the central behavior: current main awaits each configured MCP server connection and tools/list call inside one loop on the `bundle-tools` path. I did not run a live multi-server latency reproduction in this read-only review.

## Solution Assessment

Yes as an acceptable bounded mitigation, but not as the complete best end state for the linked issue. Parallelizing independent server discovery addresses the immediate sum-of-timeouts path; lazy loading, warm pooling, or a simple-chat fast path should remain separate maintainer-approved work.
