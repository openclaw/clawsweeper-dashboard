## Summary

Keep open: current main still builds the bundle MCP catalog by awaiting each configured server in sequence, and this PR is a plausible bounded mitigation for the linked latency issue, but the external PR has not yet shown a real after-fix MCP latency run.

## What This Changes

The PR rewrites `src/agents/agent-bundle-mcp-runtime.ts` so bundle MCP server connection and tools/list work runs concurrently after deterministic server-name preparation.

## Best Possible Solution

Land a narrow parallel catalog-loading fix only after a redacted real MCP latency run or deterministic slow-server test proves max-of-servers behavior while preserving disposal, diagnostics, and catalog invalidation.

## Reproduction Assessment

Yes from source for the central behavior: current main awaits connect and tools/list for each configured MCP server inside one `getCatalog()` loop. I did not run a live multi-server latency reproduction in this read-only review.

## Solution Assessment

Yes as a bounded mitigation, but not the complete best end state for the linked issue. Parallelizing independent server discovery is the right layer for the immediate sum-of-timeouts problem; lazy loading or fast-path skipping should remain separate owner-approved work.
