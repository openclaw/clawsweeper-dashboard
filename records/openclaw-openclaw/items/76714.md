## Summary

Keep open: the PR addresses a real Codex binding auth-routing bug, but the latest head still decides native Codex routing from the auth profile id prefix, so custom profile ids backed by `provider: "openai-codex"` can still take the public OpenAI route.

## What This Changes

The PR updates Codex `/codex bind` and app-server binding/auth code to carry Codex auth profile ids, omit public OpenAI provider metadata for native-looking Codex profiles, add clean failure replies, add regression tests, and add a changelog entry.

## Best Possible Solution

Keep the fix plugin-local and resolve native Codex routing from the selected auth credential provider, with custom-profile-id regression coverage, before merging.

## Reproduction Assessment

Yes. Source inspection of current main shows bound Codex starts/resumes can carry `authProfileId` alongside `modelProvider: "openai"`, and the auth profile contract shows provider semantics live on the stored credential rather than the profile id string.

## Solution Assessment

No. The PR direction is right, but prefix-based native-auth detection is not the narrowest correct fix; it should use the selected profile credential provider or carry an explicit resolved native-auth flag.
