## Summary

Keep this PR open: it is the right plugin-local area for the Codex binding OAuth routing bug, but the current patch still misses valid Codex OAuth profiles whose profile id does not start with `openai-codex`.

## What This Changes

The PR updates Codex `/codex bind` and app-server binding code to preserve Codex OAuth auth profiles, avoid conflicting public OpenAI provider metadata, recover stale bound threads, add regression tests, and add a changelog entry.

## Best Possible Solution

Keep the fix plugin-local, but base native Codex routing on the selected auth profile credential provider or an explicit resolved native-auth flag rather than the profile id string.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the original routing bug, and PR diff inspection shows the remaining blocker with a valid `provider: "openai-codex"` credential stored under a custom profile id such as `work`.

## Solution Assessment

No. The direction is right, but the implementation should resolve native Codex auth from credential/provider data instead of assuming the profile id begins with `openai-codex`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve native Codex auth from the credential provider:** `extensions/codex/src/app-server/session-binding.ts:120-127`
  - body: `isCodexAppServerNativeAuthProfileId` infers native Codex auth from the profile id prefix, but profile ids are user-chosen while the auth store carries the real `credential.provider`. A valid OpenAI Codex OAuth profile named `work` will make this return false, so thread start/resume can still send `modelProvider: "openai"` and take the same public-OpenAI 401 route this PR is fixing. Please look up the selected profile in the auth store, or carry an explicit native-auth flag, before suppressing provider metadata.
  - confidence: 0.9
