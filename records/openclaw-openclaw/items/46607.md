## Summary

Keep open: current main still lacks the targeted Google/Gemini headless audio @file inlining, and this PR is the canonical follow-up for the narrower closed #46601, but the patch is not ready because it hooks a stale fallback seam and uses the process cwd as the local-file trust root.

## What This Changes

The branch adds a Google/Gemini payload rewrite that converts resolvable local audio @file references in user text parts into inlineData parts, plus regression tests for successful, missing, absolute, and oversized refs.

## Best Possible Solution

Land a narrow reworked implementation in the active Google provider wrapper or shared Google transport seam, resolving audio refs against the active workspace or sandbox root with realpath checks, shared MIME detection, size limits, unresolved-ref fallback, and targeted Google/Gemini headless payload tests.

## Reproduction Assessment

Yes. Source inspection of current main shows Google string prompts remain text parts and there is no inline audio @file rewrite, while #46601’s close comment records this PR as the remaining canonical implementation path.

## Solution Assessment

No. The requested behavior is useful, but this PR is not the best implementation because it patches a stale/fallback seam and uses process.cwd() for file resolution; the safer path is the active Google provider or transport seam with an explicit workspace/sandbox trust boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Resolve audio refs against the active workspace:** `src/agents/pi-embedded-runner/extra-params.ts:251-256`
  - body: `resolveGoogleInlineAudioPath` anchors file reads to `process.cwd()`, but embedded/headless runs already carry `workspaceDir` through the provider wrapper context. When the gateway cwd differs from the session workspace, this can miss valid refs or inline files from the wrong root; pass the active workspace or sandbox root into this rewrite and realpath-check against that root.
  - confidence: 0.88
- **[P2] Install the rewrite in the active Google stream hook:** `src/agents/pi-embedded-runner/extra-params.ts:368`
  - body: The rewrite is called from `createGoogleThinkingPayloadWrapper`, but current main routes Google/Gemini CLI through provider-owned stream hooks and skips the fallback wrapper after provider wrapping. After rebase this path will not run for the provider this PR targets; move it to the active Google provider wrapper or shared Google transport seam.
  - confidence: 0.84
- **[P2] Accept common MP4 brands for .m4a refs:** `src/agents/pi-embedded-runner/extra-params.ts:242-244`
  - body: The `.m4a` sniff accepts only audio-specific major brands. Valid m4a files often use generic MP4 brands such as `isom` or `mp42`, so those refs can remain plain text despite the supported extension; reuse shared MIME detection or inspect compatible brands before rejecting them.
  - confidence: 0.78
