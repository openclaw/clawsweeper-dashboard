## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bind audio refs to the effective workspace:** `src/agents/pi-embedded-runner/extra-params.ts:251-253`
  - body: resolveGoogleInlineAudioPath fixes its trust boundary at process.cwd(), while the runner already passes an effective workspace into applyExtraParamsToAgent. When cwd differs from the session workspace or sandbox, an @audio ref can read or skip the wrong local file before sending bytes to Google; thread the runtime workspace/sandbox root into this resolver.
  - confidence: 0.88
- **[P2] Budget for the encoded inline request:** `src/agents/pi-embedded-runner/extra-params.ts:315-327`
  - body: The guard allows any raw file up to 20 MiB, then base64-encodes it into inlineData. Google's audio docs cap the total inline request, including text prompts, system instructions, and inline files, at 20 MB, so a near-limit raw file can become an oversized request; leave headroom, check encoded size, or use the Files API. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/audio?utm_source=openai))
  - confidence: 0.82
- **[P2] Accept valid audio MP4 brands before rejecting .m4a:** `src/agents/pi-embedded-runner/extra-params.ts:242-244`
  - body: The MP4 sniff only accepts a small allowlist of ftyp major brands, so an otherwise valid .m4a whose major brand is outside that set is left as plain text and never becomes inlineData. Use the shared MIME detector or check compatible brands so advertised .m4a support does not silently fail.
  - confidence: 0.74
