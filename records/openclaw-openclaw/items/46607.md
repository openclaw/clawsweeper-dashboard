## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve inline refs from the session workspace:** `src/agents/pi-embedded-runner/extra-params.ts:251-257`
  - body: `resolveGoogleInlineAudioPath` binds file reads to `process.cwd()`, but embedded sessions already carry a `workspaceDir` that can differ from the OpenClaw process cwd. This can inline audio bytes from the wrong local boundary; thread the runtime workspace into the wrapper and validate refs there.
  - confidence: 0.86
- **[P2] Emit only Gemini-supported audio MIME types:** `src/agents/pi-embedded-runner/extra-params.ts:327`
  - body: The inliner can send sniffed `audio/opus` and `audio/mp4`, but the Gemini audio docs list supported audio MIME types such as `audio/ogg`, `audio/mp3`, `audio/aac`, and `audio/flac`, not those values. Unsupported refs should be mapped to an accepted MIME only when valid or left as text.
  - confidence: 0.83
- **[P2] Cap the encoded inline request size:** `src/agents/pi-embedded-runner/extra-params.ts:315`
  - body: The guard compares the raw file size to 20 MiB, but this path embeds base64 in the JSON request and Google's limit applies to the total request size. Audio files well under 20 MiB can still produce oversized requests after base64 expansion and prompt overhead.
  - confidence: 0.81
- **[P2] Accept common m4a MP4 brands before preserving refs:** `src/agents/pi-embedded-runner/extra-params.ts:242-244`
  - body: The PR advertises `.m4a` support, but the MP4 branch only accepts a narrow set of major brands and rejects common `.m4a` files marked `isom` or `mp42`. Those valid refs will remain plain text instead of becoming audio input.
  - confidence: 0.74
