## Summary

Keep open. Current main still lacks the requested Google/Gemini inline audio `@file` conversion, and this PR is the narrow canonical implementation thread for https://github.com/openclaw/openclaw/issues/46601, but the branch needs real behavior proof, a rebase onto the current Google wrapper surface, and correctness fixes before merge.

## What This Changes

The branch adds Google/Gemini payload rewriting that converts local `@*.ogg|*.opus|*.mp3|*.wav|*.m4a` text refs into `inlineData` audio parts, with regression tests for resolvable, missing, absolute, and oversized refs.

## Best Possible Solution

Port the useful inlineData conversion onto current main's shared Google payload wrapper or provider-owned seam, resolve refs against the session workspace, reuse or align shared MIME/size policy, and keep broader read-tool audio parity in https://github.com/openclaw/openclaw/issues/35835.

## Reproduction Assessment

Yes, by source inspection: current main's Google wrapper only sanitizes thinking config, and the documented Gemini CLI audio fallback still uses `read_many_files` rather than binary `inlineData`. I did not run a live Gemini headless request.

## Solution Assessment

No, not as-is. The implementation direction is useful, but the patch should be rebased onto the current wrapper surface, use the session workspace boundary, handle common `.m4a` containers, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve audio refs against the session workspace:** `src/agents/pi-embedded-runner/extra-params.ts:250-256`
  - body: The new inline path resolves refs under `process.cwd()`, but current runtime calls already carry an effective workspace into `applyExtraParamsToAgent`. In sessions where the node process cwd differs from the session workspace, this will either fail to inline the requested `@audio` file or read/send bytes from the wrong directory boundary; use the passed workspace/sandbox root for this file policy.
  - confidence: 0.86
- **[P2] Accept common m4a MP4 brands:** `src/agents/pi-embedded-runner/extra-params.ts:242-245`
  - body: The `.m4a` sniff only accepts a small set of MP4 major brands. Valid `.m4a` files commonly use generic MP4/ISO major brands such as `isom` or `mp42` with audio-compatible brands elsewhere, so those files are left as plain text refs and never become `inlineData`. Broaden this check or reuse shared MIME detection before rejecting the file.
  - confidence: 0.82
