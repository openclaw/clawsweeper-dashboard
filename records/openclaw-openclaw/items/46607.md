## What This Changes

The PR adds Google/Gemini payload rewriting so resolvable local audio `@file` refs in user text parts become `inlineData` audio parts, with regression tests for inlined, missing, absolute, and oversized refs.

## Best Possible Solution

Rebase onto current main and implement the audio inlining at the current Google provider/wrapper boundary with runtime workspace or sandbox-bound reads, shared MIME detection, encoded or total request budgeting, explicit compatibility behavior for unresolved/plain refs, focused tests, and redacted real Gemini proof.

## Solution Assessment

Full review comments:

- **[P2] Resolve audio refs from the runtime workspace:** `src/agents/pi-embedded-runner/extra-params.ts:250-256`
  - body: `resolveGoogleInlineAudioPath` anchors trust at `process.cwd()`, while current runs pass the effective workspace into `applyExtraParamsToAgent`. Sandboxed or headless sessions can read or skip files relative to the wrong root before bytes are sent to Google, so thread the runtime workspace or sandbox root into this resolver.
  - confidence: 0.88
- **[P2] Cap the encoded request, not the raw file:** `src/agents/pi-embedded-runner/extra-params.ts:314-327`
  - body: The guard admits any raw file up to `GOOGLE_INLINE_AUDIO_MAX_BYTES` and then base64-encodes it into `inlineData`. Near-limit files can still exceed the intended inline request budget once encoded and combined with prompt/system text, so leave headroom, check the encoded payload, or use a Files API fallback.
  - confidence: 0.84
- **[P2] Reuse shared media MIME detection:** `src/agents/pi-embedded-runner/extra-params.ts:242-244`
  - body: The local MP4 sniffer only accepts a small major-brand allowlist, so valid `.m4a` files that the shared detector accepts as `audio/mp4` can remain plain text refs. Use or extend the shared media helpers instead of maintaining a Google-only sniffer.
  - confidence: 0.76
