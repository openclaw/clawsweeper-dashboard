## Summary

Keep open: current main still does not implement Google/Gemini inline audio `@file` refs, but this PR is not merge-ready because it lacks real Gemini proof and still has workspace-boundary, request-size, MIME, and stale-branch blockers.

## What This Changes

The PR rewrites Google/Gemini payload text parts so resolvable local audio `@file` refs become Gemini `inlineData` parts and adds regression tests for inlined, missing, absolute, and oversized refs.

## Best Possible Solution

Rebase onto current main and implement inline Google audio at the current provider wrapper boundary with runtime workspace or sandbox resolution, shared MIME handling, total request-size or Files API behavior, focused tests, and redacted real Gemini proof.

## Reproduction Assessment

Yes, at source level: current main's Google wrapper only sanitizes thinking payloads and has no audio `@file` inlining. I did not run a live Gemini audio session in this read-only review.

## Solution Assessment

No as proposed. The direction is plausible, but the branch needs a current-main rebase, runtime workspace or sandbox-bound file reads, shared MIME handling, total request-size or Files API behavior, and real provider proof before it is the maintainable fix.
