## Summary

## What This Changes

The PR rewrites Google/Gemini headless payload text parts so local audio @file references become Gemini inlineData parts, with embedded-runner regression coverage.

## Best Possible Solution

Rebase the implementation onto the current shared Google wrapper/runtime path, resolve audio refs against the effective workspace or sandbox, enforce the full inline request budget and provider MIME contract, preserve unresolved refs as text, and add redacted real Gemini proof.

## Solution Assessment

No, not as-is. The direction is useful, but the branch needs a current-main rebase, workspace/sandbox-bound file reads, total inline request-size handling, provider MIME-contract proof, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve audio refs from the runtime workspace:** `src/agents/pi-embedded-runner/extra-params.ts:251-253`
  - body: resolveGoogleInlineAudioPath fixes its trust boundary at process.cwd(), while the current runner passes an effective workspace into applyExtraParamsToAgent. When cwd differs from the session workspace or sandbox, an @audio ref can read or skip the wrong local file before sending bytes to Google; thread the runtime workspace/sandbox root into this resolver.
  - confidence: 0.88
- **[P2] Budget for the encoded inline request size:** `src/agents/pi-embedded-runner/extra-params.ts:315`
  - body: The guard allows any raw file up to 20 MiB, then base64-encodes it into inlineData. Google's audio docs cap the total inline request, including text and inline files, at 20 MB, so a near-limit raw file can become an oversized JSON request; leave headroom, check encoded size, or use the Files API. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/audio?utm_source=openai))
  - confidence: 0.82
