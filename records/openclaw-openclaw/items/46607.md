## Summary

Keep open. Current main still lacks the requested inline Gemini audio attachment path, but this PR is not mergeable as-is because it introduces prompt-selected local file egress with the wrong workspace boundary, can emit unsupported Gemini audio MIME values, and lacks required real behavior proof.

## What This Changes

The branch adds Google/Gemini payload rewriting and tests to convert local user-text `@*.ogg|*.opus|*.mp3|*.wav|*.m4a` refs into `inlineData` audio parts before dispatch.

## Best Possible Solution

Integrate the change into the current shared Google wrapper with `workspaceDir`-bound path resolution, Gemini-supported MIME handling, focused regression coverage, and redacted real Gemini headless proof.

## Reproduction Assessment

Yes, at source level: current main's Google wrapper does not inline audio refs, and the current docs/runner still route Gemini CLI audio through `read_many_files`. I did not run a live Gemini session in this read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but the PR must use the session workspace boundary, honor the Gemini MIME contract, and provide real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve inline refs from the session workspace:** `src/agents/pi-embedded-runner/extra-params.ts:251-256`
  - body: `resolveGoogleInlineAudioPath` binds local file reads to `process.cwd()`, but embedded sessions already carry `workspaceDir` and that can differ from the OpenClaw process cwd. Prompt-selected audio refs should resolve and realpath-check against the session workspace before bytes are sent to the provider.
  - confidence: 0.88
- **[P2] Emit only Gemini-supported audio MIME types:** `src/agents/pi-embedded-runner/extra-params.ts:327`
  - body: The sniffer can return `audio/opus` and `audio/mp4`, and this line sends that value directly as Gemini `inlineData`. Unsupported values should stay as text or be converted/mapped through a proven Gemini-supported representation before dispatch. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/audio?utm_source=openai))
  - confidence: 0.84
