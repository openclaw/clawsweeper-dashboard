## What This Changes

The branch adds Google/Gemini payload rewriting and regression tests to convert supported local audio `@file` references into Gemini `inlineData` parts.

## Best Possible Solution

Rebase into the current shared Google wrapper, resolve refs against the runtime workspace, emit only Gemini-supported audio MIME values or preserve unsupported refs, keep size/path/MIME guards, and require redacted real Gemini headless proof before merge.

## Reproduction Assessment

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Emit only Gemini-supported audio MIME types:** `src/agents/pi-embedded-runner/extra-params.ts:319-327`
  - body: The inliner can send `audio/opus` for Ogg Opus and `audio/mp4` for `.m4a`, but the Gemini audio contract currently documents inline/request audio formats as `audio/wav`, `audio/mp3`, `audio/aiff`, `audio/aac`, `audio/ogg`, and `audio/flac`. These refs can become rejected Google requests instead of falling back to text, so gate or map MIME values to formats Gemini accepts. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/audio))
  - confidence: 0.82
- **[P2] Resolve inline audio refs from the runtime workspace:** `src/agents/pi-embedded-runner/extra-params.ts:250-257`
  - body: `resolveGoogleInlineAudioPath` uses `process.cwd()` as the workspace root. In sessions where the runtime workspace differs from the OpenClaw process cwd, relative audio refs can be read and inlined from the wrong boundary; thread the current `workspaceDir` into the wrapper and resolve against that root.
  - confidence: 0.84
