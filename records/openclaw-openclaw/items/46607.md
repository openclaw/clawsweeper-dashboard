## Summary

Keep open. Current main still lacks the central Google/Gemini audio @file inlining behavior, but this PR is not merge-ready because it resolves prompt-selected file reads from process cwd, forwards MIME values outside the documented Gemini audio set, is conflicting against current main, and has no contributor real-behavior proof.

## What This Changes

The PR adds Google/Gemini headless payload rewriting that converts resolvable local audio @file text references into inlineData audio parts and adds regression tests.

## Best Possible Solution

Rebase the implementation onto the current shared Google wrapper, resolve refs against the runtime workspace or sandbox boundary, gate size and MIME values to the proven Gemini contract, preserve unresolved refs as text, and add redacted real Gemini headless proof.

## Reproduction Assessment

Yes at source level: current main has no Google/Gemini audio @file inlining and still routes Gemini CLI media through read_many_files with {{MediaPath}}. I did not run a live Gemini audio session in this read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but the PR needs current-main integration, workspace-bound file reads, a proven Gemini MIME contract, and real behavior proof before merge.

- **[P2] Resolve audio refs from the runtime workspace:** `src/agents/pi-embedded-runner/extra-params.ts:251-256`
  - body: resolveGoogleInlineAudioPath anchors prompt-selected file reads to process.cwd(), but the current runner passes the effective workspace into applyExtraParamsToAgent. When cwd differs from the session workspace or sandbox, an @audio ref can read or reject the wrong local path before sending bytes to Google.
  - confidence: 0.88

The Google audio API documents a narrower supported list for this path, so unsupported values should stay text or be mapped through a proven supported representation.
  - confidence: 0.84
