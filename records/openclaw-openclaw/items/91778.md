## Summary

Keep this issue open as a canonical shipped memory regression: v2026.6.6 and current main retain the affected identity gates, multiple users report successful indexing followed by disabled semantic search, and the open related PRs address narrower SQLite-handle and unregistered-provider failures rather than the full configured-provider identity problem.

## What This Changes

Fix `memory_search` becoming unavailable with `index metadata is missing` after successful indexing on configured OpenAI-compatible embedding providers.

## Best Possible Solution

Keep this issue as the broader canonical outage tracker, fix memory-core so fully initialized configured-provider identity is persisted and validated by explicit indexing, separately repair any loss of configured provider during sync, and retain fail-closed explicit rebuild behavior for genuine provider, model, scope, or settings changes.

## Reproduction Assessment

Yes at source level: shipped/current code contains the relevant identity gates and several users supplied repeatable CLI/runtime observations. A controlled current-main live run is still required to split the provider-identity, auth-routing, metadata-write, and stale-handle variants with high confidence.

## Solution Assessment

No, the suggested force-sync-on-search stopgap is not the best general solution because it may silently trigger costly re-embedding and conflate genuine mismatches with lifecycle bugs. The narrow maintainable fix is to establish canonical identity after provider initialization and explicit indexing, plus a separate provider-routing repair if the configured provider is later discarded.
