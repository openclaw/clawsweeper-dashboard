## Summary

This is a current, actionable memory-core regression: the persistent runtime manager can retain the SQLite file that an external atomic rebuild renames and deletes, while the replacement at the canonical path contains valid metadata. Current main has no database-file identity refresh before search, and the related repair work is still open rather than merged.

## What This Changes

Fix semantic memory search remaining paused with “index metadata is missing” after a successful CLI rebuild replaces the SQLite index beneath a persistent gateway manager.

## Best Possible Solution

Have the persistent memory manager detect when its open SQLite database no longer represents the canonical index path and reopen that path before identity validation and search, with a two-manager regression test proving recovery after atomic rebuild.

## Reproduction Assessment

Yes. Source inspection shows a cached gateway manager can retain the database that atomic rebuild renames and deletes, and the issue plus https://github.com/openclaw/openclaw/issues/91910 provide matching post-rebuild runtime diagnostics.

## Solution Assessment

Yes, the narrowest maintainable direction is lifecycle-owned SQLite handle refresh before search/status; changing the status command or adding provider-specific fallback would not repair the stale backing file.
