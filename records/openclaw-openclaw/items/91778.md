## Label Justifications

- `P1`: Multiple users report that the supported memory-search workflow is unusable in released builds, with vector retrieval remaining paused after explicit rebuilds.
- `impact:session-state`: The failure prevents agents from semantically retrieving persisted memory and prior-session context.
- `impact:auth-provider`: The failure is coupled to embedding-provider initialization, identity, and in one reported path incorrect OpenAI credential routing.

## Summary

The issue remains necessary: v2026.6.5 contains the fail-closed index-identity behavior, current main still lacks the focused stale provider-key repair proposed in an open PR, and the related open PRs cover distinct partial causes rather than fully resolving this report.

## What This Changes

Investigate and fix persistent vector memory search failures that report missing or mismatched index identity after successful rebuilds with OpenAI-compatible embedding providers.

## Best Possible Solution

Land a narrowly identity-gated repair that replaces only the known pre-initialization FTS-only providerKey with the initialized provider identity, retain strict rejection for genuine setting changes, and verify CLI rebuild plus runtime `memory_search` against an OpenAI-compatible endpoint.

## Reproduction Assessment

Yes, at source level with high confidence: released identity validation can pause search after an inconsistent provider identity, and multiple users supplied consistent rebuild/runtime evidence. This read-only pass did not execute the private OpenRouter or LM Studio environments.

## Solution Assessment

Yes, the focused stale-sentinel repair is the narrowest maintainable direction, provided it is combined with the necessary provider-initialization behavior and live-tested on the reported OpenAI-compatible path; blanket identity bypass or automatic full reindex is not the best fix.
