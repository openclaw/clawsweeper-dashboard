## Summary

Keep this issue open pending a current-release Windows retest. Release v2026.6.6 added relevant piped-stdin teardown after the reported v2026.6.1, but source inspection cannot prove that it fixes the exact `.cmd` plus `shell: true` process tree, and the reported `--version` hang points toward possible shell-wrapper ownership because OpenClaw’s version fast path explicitly exits.

## What This Changes

Ensure short-lived OpenClaw CLI commands exit normally when spawned with piped standard streams on Windows.

## Best Possible Solution

Run the three proposed Windows spawn variants against v2026.6.6 and capture child exit events plus the process tree; only change OpenClaw if the direct Node entry remains alive, otherwise document or repair the `.cmd` wrapper invocation contract.

## Reproduction Assessment

No, not against current main or v2026.6.6. The report provides a concrete v2026.6.1 spawn program, but current code contains newer piped-stdin cleanup and no evidence identifies whether Node or cmd.exe remains alive.

## Solution Assessment

Unclear until the Windows process owner is established. The current non-TTY stdin teardown is the narrow OpenClaw-owned solution; a wrapper-only hang should be solved through invocation guidance or wrapper handling rather than a general forced exit.
