## Summary

Keep open. The requested restart fan-out is still not implemented on current main, but this PR is not correct enough to merge as-is because it is built on an older restart-sentinel implementation and its new fan-out only queues system events without requesting heartbeat wakes or applying the current non-resumable session filters.

## What This Changes

The PR adds a `wakeRecentlyActiveSessions()` helper to `src/gateway/server-restart-sentinel.ts` that enumerates agent session stores and enqueues GatewayRestart system events for the triggering session plus sessions updated within 60 seconds before the restart timestamp.

## Best Possible Solution

Implement the fan-out on current main’s restart startup path using the same event-plus-heartbeat semantics as `enqueueRestartSentinelWake`. Selection should be deterministic, skip pseudo and non-resumable session kinds, use current session timestamp/status contracts, and include focused tests for cross-session selection, dedupe, no-route behavior, and skipped session kinds.

## Reproduction Assessment

Yes. On current main, a restart sentinel only wakes the sentinel session key; adding another recently active session row has no code path that enqueues GatewayRestart or requests a heartbeat for that second session.

## Solution Assessment

No, this PR is not the best current solution as written. The direction is useful, but it should be reworked on current main and pair every selected system event with a targeted heartbeat while filtering non-resumable sessions through the same contracts used by restart recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Request heartbeats for broadcast wakes:** `src/gateway/server-restart-sentinel.ts:179`
  - body: `wakeRecentlyActiveSessions` only calls `enqueueSystemEvent` for additional sessions. System events are consumed on a later heartbeat or turn, so these sessions can still stay silent after the restart; request `requestHeartbeatNow({ reason: "wake", sessionKey: key })` for each selected session, matching the current restart wake path.
  - confidence: 0.9
- **[P2] Filter non-resumable sessions before fan-out:** `src/gateway/server-restart-sentinel.ts:166-170`
  - body: The loop skips only cron run keys plus `global` and `unknown`, but ACP, subagent, and other cron sessions have separate recovery lifecycles. Broadcasting a trusted GatewayRestart into those rows can wake the wrong runtime path or duplicate recovery; reuse the current ACP/subagent/cron and entry-status filters before enqueueing.
  - confidence: 0.82
