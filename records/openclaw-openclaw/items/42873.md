## Summary

Keep open: the user problem is still source-reproducible on current main, but this PR is not merge-ready because it only changes one normalization path, is currently conflicting, lacks contributor real behavior proof, and overlaps newer unresolved silent-token policy alternatives.

## What This Changes

The PR adds an own-line `NO_REPLY` detector and updates reply normalization tests so text followed by a standalone trailing silent token is treated as silent while inline token stripping remains.

## Best Possible Solution

Choose one shared, case-insensitive silent-token classifier and apply it consistently across normalization, directive parsing, cron/subagent delivery, gateway projection/history, and tests, with documented false-positive boundaries and real Telegram/runtime proof.

## Reproduction Assessment

Yes, source-reproducible. Current main's normalizer and test coverage show that `text\n\nNO_REPLY` is stripped to visible preamble text rather than being fully suppressed; no runtime test was run because this review is read-only.

## Solution Assessment

No, not as-is. The suppression direction may be useful, but this patch adds a case-sensitive helper only to the normalizer and does not settle the shared pipeline policy or proof requirements.
