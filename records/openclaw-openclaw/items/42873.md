## Summary

Keep open: current main still preserves preamble text before a trailing own-line silent token, but this PR is not merge-ready because it applies the policy in only one normalization path, lacks real behavior proof, and overlaps with open competing fixes that need maintainer selection.

## What This Changes

The PR adds an own-line `NO_REPLY` detector and changes reply normalization/tests so `text\nNO_REPLY` is treated as a silent reply while inline token stripping is preserved.

## Best Possible Solution

Choose one shared case-insensitive silent-token classifier and apply it consistently across normalization, directive parsing, outbound planning, cron/subagent delivery, gateway display/history, and tests while preserving media, channelData, and legitimate inline mentions.

## Reproduction Assessment

Yes. Source inspection and the current normalizer test show `summary\n\nNO_REPLY` strips the token but keeps the preamble as a deliverable payload; I did not execute tests because this review is read-only.

## Solution Assessment

No, not as-is. Own-line suppression may be useful, but the patch only changes normalization and needs a shared, case-insensitive policy across final delivery and display paths plus real behavior proof.
