## Summary

Keep open: current main still has the reported own-line `NO_REPLY` leak, but this PR is not ready to merge because it only applies the new rule in one normalization path and misses established case-insensitive silent-token behavior while overlapping with #46626 and #69394.

## What This Changes

The PR adds an own-line `NO_REPLY` detector, applies it during reply payload normalization, and updates token/normalization tests so trailing own-line `NO_REPLY` suppresses the whole message while inline tokens are stripped.

## Best Possible Solution

Land one canonical current-main fix that defines case-insensitive own-line/trailing silent-token semantics and applies a shared detector through normalize, directive, cron, gateway/live delivery paths while preserving intended inline-token stripping.

## Reproduction Assessment

Yes, by source inspection. Current main strips trailing `NO_REPLY` and keeps the preceding text, and the existing normalization test still asserts that behavior for a `File's there...\n\nNO_REPLY` payload.

## Solution Assessment

No. The PR is a useful narrow direction, but it is not the best complete fix because it applies the own-line rule only in normalization and the new detector is case-sensitive despite existing case-insensitive token handling.
