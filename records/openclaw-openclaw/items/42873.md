## Summary

Keep open. Current main still has the reported `NO_REPLY` preamble leak in `normalizeReplyPayload`, but this PR is dirty against main and the proposed own-line matcher is case-sensitive despite the existing silent-token contract being case-insensitive; overlapping open fixes mean maintainers should choose one canonical direction before merge.

## What This Changes

The PR adds an `isSilentTokenOnOwnLine` helper and uses it in auto-reply normalization/tests so a trailing standalone `NO_REPLY` suppresses the whole payload while inline tokens are still stripped.

## Best Possible Solution

Land one rebased canonical fix that uses shared cached case-insensitive silent-token matching, covers the final delivery paths that can leak user-visible text, preserves media/channelData and inline substantive mentions, and includes a changeloged regression test.

## Reproduction Assessment

Yes. Source inspection on current main shows `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` reaches the mixed-content strip path and current tests assert that the preamble remains deliverable.

## Solution Assessment

No, not as-is. The direction is useful, but the PR misses mixed-case own-line tokens and is dirty against current main, so a rebased canonical fix should reconcile this branch with the newer overlapping PRs.
