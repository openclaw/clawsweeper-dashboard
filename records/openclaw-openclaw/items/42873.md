## Summary

Keep open: the PR targets a real source-reproducible silent-reply leak, but it is not ready to merge because it lacks after-fix real behavior proof, only handles one delivery path, and has a mixed-case matching gap against the existing silent-token contract.

## What This Changes

The PR adds an own-line `NO_REPLY` detector, uses it in `normalizeReplyPayload`, and updates token and reply-normalization tests so `text\nNO_REPLY` is fully suppressed while inline tokens are still stripped.

## Best Possible Solution

Settle one canonical silent-token policy, then implement a shared cached case-insensitive classifier across normalization and final delivery guards with false-positive coverage and real behavior proof.

## Reproduction Assessment

Yes, source-reproducible. Current main code and tests show `normalizeReplyPayload({ text: "summary\n\nNO_REPLY" })` strips the token but preserves a non-empty preamble; I did not run tests because this review is read-only.

## Solution Assessment

No. Own-line suppression is a plausible direction, but this branch handles only the normalizer path, misses mixed-case own-line tokens, lacks real behavior proof, and needs maintainer agreement on the final silent-token semantics.
