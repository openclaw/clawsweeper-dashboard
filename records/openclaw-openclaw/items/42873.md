## Summary

Keep open. Current main still has the normalizer-side trailing `NO_REPLY` preamble leak, so the PR is not obsolete, but the branch is stale and currently regresses case-insensitive silent-token handling. Related open work (#46626 and #69394) covers overlapping behavior, so this needs maintainer coordination rather than cleanup closure.

## What This Changes

The PR adds an own-line `NO_REPLY` detector, applies it in auto-reply normalization, and updates token and normalizer tests to suppress whole text payloads when a trailing standalone silent token is present.

## Best Possible Solution

Land one coordinated, rebased fix that uses shared case-insensitive silent-token detection, suppresses turn-level text when the model clearly ends with a standalone or otherwise agreed trailing silent token, preserves inline user-facing mentions and media/channelData behavior, and covers the final delivery paths that can still leak the token or preamble.

## Reproduction Assessment

Yes. Static current-main inspection is enough: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` reaches `stripSilentToken`, leaves a non-empty preamble, and the current test suite asserts that non-null result.

## Solution Assessment

No, not as-is. The direction is useful, but the branch needs a current-main rebase and a case-insensitive shared detector before it is the safest maintainable fix; maintainers should also decide whether to land it alone or supersede it with #46626 or #69394.
