## Summary

Keep this PR open for maintainer decision: current main still preserves the reported `text\n\nNO_REPLY` preamble in `normalizeReplyPayload`, but this PR is now stale and only covers one delivery path while related open PRs propose different suppression semantics.

## What This Changes

The PR adds a silent-token helper and updates auto-reply normalization/tests so `NO_REPLY` on its own trailing line suppresses the whole payload while inline token stripping still delivers remaining text.

## Best Possible Solution

Land one canonical current-main fix that centralizes trailing or own-line silent-token detection across normalize, reply-directive, cron, and gateway delivery gates while preserving the intended inline-token behavior.

## Reproduction Assessment

Yes. Source inspection of current main shows `normalizeReplyPayload` strips the trailing token and returns the preamble, and the existing test suite explicitly asserts that behavior for `File's there...\n\nNO_REPLY`.

## Solution Assessment

No. This PR is a useful narrow direction, but it is not the best complete fix because it only updates one normalization path and misses current case-insensitive silent-token expectations; a shared detector applied at every delivery gate is safer.

A mixed-case token on its own trailing line would bypass the new whole-payload suppression and remain visible.
