## What This Changes

The PR adds an own-line `NO_REPLY` detector, applies it in reply normalization, and updates token and normalizer tests so newline-prefixed silent-token endings suppress the whole reply while inline tokens remain stripped.

## Best Possible Solution

Land or replace this with a rebased, documented, case-insensitive silent-reply classifier applied consistently across normalization, directive/outbound planning, cron/gateway paths, and real delivery proof while preserving legitimate `NO_REPLY` mentions and media/channelData behavior.

Yes, at source level. Current main strips a trailing mixed `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the adjacent test asserts that strip-and-deliver behavior; I did not run a live channel flow in this read-only review.

## Solution Assessment

No, not as-is. The normalizer-only change is a useful slice, but the maintainable solution needs a shared case-insensitive policy applied across final delivery/display paths plus real behavior proof.
