## Summary

Keep this PR open. Current main still strips a trailing own-line `NO_REPLY` and returns the preamble as deliverable auto-reply text, but the PR is not merge-ready because its proposed own-line path is case-sensitive and overlaps active alternatives that need maintainer consolidation.

## What This Changes

The PR adds an `isSilentTokenOnOwnLine` helper, uses it in auto-reply normalization to suppress a whole payload when `NO_REPLY` appears on its own trailing line, and updates token/normalizer tests.

## Best Possible Solution

Consolidate this cluster into one maintained patch: add a cached, case-insensitive silent-token predicate for the intended own-line or trailing-token policy, preserve inline stripping, glued-leading-token handling, JSON envelope suppression, media/channelData behavior, and cron's mixed-case coverage, then close duplicate PRs after the chosen fix lands.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` follows the strip-then-deliver path, and the current test suite asserts that non-null preamble result.

## Solution Assessment

No, not as-is. The PR's direction is useful, but the patch should be rebased and made case-insensitive, then coordinated with #46626 and #69394 so the silent-reply contract is fixed consistently across delivery paths.
