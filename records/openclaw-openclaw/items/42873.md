## Summary

Keep open: the proposed direction addresses a real source-level leak shape, but this branch is still incomplete, conflicting, lacks contributor real-behavior proof, and changes a silent-token policy boundary that maintainers still need to choose.

## What This Changes

The branch adds an own-line `NO_REPLY` detector, applies it in `normalizeReplyPayload`, and updates token/normalizer tests so such payloads are suppressed instead of stripped.

## Best Possible Solution

Define one shared, documented, case-insensitive silent-reply classifier and land a rebased implementation across normalization, directive/outbound planning, cron/subagent handling, gateway projection/history, with focused tests and real delivery proof.

## Reproduction Assessment

Yes, at source level. Current main strips a trailing own-line `NO_REPLY` and returns the non-empty preamble from `normalizeReplyPayload`, and the current normalizer test asserts that strip-and-deliver behavior.

## Solution Assessment

No, not as-is. The branch covers only one normalizer path and is case-sensitive; the safer path is a shared policy/classifier applied consistently across delivery and projection surfaces.
