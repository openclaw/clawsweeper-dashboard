## Summary

Keep open: the PR addresses a real source-reproducible leak shape, but it is incomplete as a merge candidate because it only wires own-line suppression into normalization, the same-author follow-up reports remaining delivery leaks, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds an own-line `NO_REPLY` detector, uses it in `normalizeReplyPayload`, and updates token/normalization tests so text followed by a standalone trailing `NO_REPLY` is fully suppressed while inline token stripping is preserved.

## Best Possible Solution

Choose one shared, cached, case-insensitive silent-reply classifier, apply it consistently across normalization, directive parsing, cron/subagent delivery, and live chat projection, then update tests/docs and require real Telegram-visible proof for the chat behavior.

## Reproduction Assessment

Yes, source-reproducible: current main strips a trailing `NO_REPLY` token and returns the non-empty preamble from `normalizeReplyPayload`, and the current test suite asserts that behavior. The requested own-line suppression is not yet the documented contract, so the expected behavior still needs maintainer policy agreement.

## Solution Assessment

No. The own-line direction may be reasonable, but this patch is incomplete without shared final delivery coverage and case-insensitive matching; it also needs real behavior proof before merge.
