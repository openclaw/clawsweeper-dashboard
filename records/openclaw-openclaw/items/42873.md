## Summary

Keep open: current main still has the reported strip-and-deliver behavior, but this PR is not merge-ready because it only changes the shared normalizer path, conflicts with newer silent-token work, and lacks inspectable real behavior proof for the claimed channel-visible leak.

## What This Changes

The PR adds an own-line `NO_REPLY` detector and changes reply normalization/tests so text ending with a standalone `NO_REPLY` is suppressed instead of delivered with the token stripped.

## Best Possible Solution

Define one shared, documented, case-insensitive silent-reply classifier and land a rebased canonical implementation across normalization, directive/outbound planning, cron/subagent handling, gateway projection/history, with focused tests and real delivery proof.

## Reproduction Assessment

Yes, at source level. Current main strips a trailing own-line `NO_REPLY` and returns the non-empty preamble in `normalizeReplyPayload`, and the colocated test still asserts that strip-and-deliver behavior.

## Solution Assessment

No, not as-is. The direction may be useful, but this branch covers only one normalizer path, is case-sensitive, conflicts with newer code, and needs maintainer policy selection plus real behavior proof.
