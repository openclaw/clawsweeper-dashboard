## Summary

Keep open: the PR targets a real silent-reply leak class, but this branch is incomplete, lacks external after-fix proof, and changes current tested/docs-backed `NO_REPLY` semantics that still need maintainer selection across competing open PRs.

## What This Changes

Adds an own-line `NO_REPLY` detector, wires it into `normalizeReplyPayload`, and updates token and normalizer tests so text followed by a standalone trailing `NO_REPLY` is fully suppressed while inline token stripping is preserved.

## Best Possible Solution

Choose one shared, cached, case-insensitive silent-reply classifier, apply it consistently across normalization, directive parsing, cron/subagent delivery, live chat projection, docs, and tests, then require redacted real Telegram/runtime proof for the visible behavior.

## Reproduction Assessment

Yes, source-reproducible: current main strips a trailing `NO_REPLY` token and returns the non-empty preamble from `normalizeReplyPayload`, and the current tests assert that behavior. The broader own-line suppression expectation is not yet the documented contract.

## Solution Assessment

No. The own-line direction may be reasonable, but this patch is incomplete without shared final delivery coverage, case-insensitive matching, maintainer policy selection, and real behavior proof.
