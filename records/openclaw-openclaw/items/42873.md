## Summary

Keep open: current main still has the reported strip-and-deliver behavior, but this PR is not ready to merge because it lacks real behavior proof and only changes one normalization path. The overlapping follow-up at https://github.com/openclaw/openclaw/pull/46626 and canonical issue at https://github.com/openclaw/openclaw/issues/42472 show the remaining decision is a shared silent-token policy, not this narrow branch alone.

## What This Changes

The PR adds an own-line `NO_REPLY` detector, uses it in `normalizeReplyPayload`, and updates token/normalizer tests so standalone trailing `NO_REPLY` suppresses the full normalized reply while inline `NO_REPLY` is stripped.

## Best Possible Solution

Choose one documented, case-insensitive silent-token classifier and apply it consistently across normalization, directive parsing, cron/subagent handling, gateway projection/history, with regression tests and redacted real Telegram or runtime proof.

## Reproduction Assessment

Yes, source-reproducible. Current main turns `summary\n\nNO_REPLY` into deliverable preamble text in `normalizeReplyPayload`, and the current test suite asserts that behavior.

## Solution Assessment

No, not as-is. The normalizer-only patch leaves other delivery/display paths on exact or leading-token checks, and the new own-line helper is case-sensitive despite current docs and helpers accepting `no_reply`.
