## Summary

Keep open: the PR is useful source work, but it is not merge-ready because it only applies own-line `NO_REPLY` suppression in the normalizer, misses current case-insensitive silent-token expectations, and overlaps with broader open alternatives that require a maintainer policy choice.

## What This Changes

The PR adds `isSilentTokenOnOwnLine()` and changes reply normalization/tests so a trailing own-line `NO_REPLY` suppresses the whole reply while inline tokens are still stripped.

## Best Possible Solution

Define one shared, cached, case-insensitive silent-token classifier, apply it consistently through normalization, directive parsing, outbound planning, cron/subagent handling, and gateway live/history projection, then update tests and docs while preserving legitimate inline mentions.

## Reproduction Assessment

Yes, source-reproducible. Current main strips the trailing token and returns the preamble, and the existing normalizer test explicitly asserts that behavior for `File's there. Not urgent.\n\nNO_REPLY`.

## Solution Assessment

No. This PR is a useful narrow step, but the best fix needs a shared case-insensitive classifier at the final delivery gates after maintainers choose the intended trailing-token policy.
