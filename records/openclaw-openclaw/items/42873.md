## Summary

Keep open: the PR targets a real source-reproducible leak shape, but this branch is incomplete because it only changes reply normalization, misses case-insensitive own-line tokens, and still lacks required real behavior proof for an external Telegram-visible change.

## What This Changes

The branch adds an own-line silent-token detector, uses it in reply payload normalization, and updates token and normalizer tests to suppress whole messages ending with own-line NO_REPLY while preserving inline stripping.

## Best Possible Solution

Choose one shared, case-insensitive silent-token policy and apply it consistently across normalization, final delivery, gateway display/history, cron paths, and tests while preserving media, channelData, and legitimate inline mentions.

## Reproduction Assessment

Yes, source-reproducible: current main strips a trailing own-line NO_REPLY and preserves the non-empty preamble in normalizeReplyPayload, and the current regression test asserts that behavior. I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The own-line suppression direction may be useful, but this patch needs a shared case-insensitive classifier wired through final delivery/display gates plus maintainer policy selection and real behavior proof.
