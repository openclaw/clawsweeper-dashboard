## Summary

Keep open. Current main still has the normalizer-side trailing `NO_REPLY` preamble leak, but this PR is not merge-ready because it predates current case-insensitive silent-token handling and overlaps open follow-up/alternative fixes that need maintainer choice.

## What This Changes

The PR adds an own-line `NO_REPLY` detector to auto-reply normalization and updates token/normalizer tests so trailing standalone silent tokens suppress the whole payload while inline tokens continue to be stripped.

## Best Possible Solution

Land one coordinated, rebased fix with a shared cached case-insensitive silent-token predicate, consistent final-delivery coverage, and regression tests that preserve inline user-facing mentions plus media/channelData behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` reaches `stripSilentToken`, keeps the non-empty preamble, and the current test suite asserts that behavior.

## Solution Assessment

No, not as-is. The direction is useful, but the PR should be rebased around current shared case-insensitive token handling and reconciled with #46626 and #69394 before merge.
