## Summary

Keep open: the requested behavior is not on current main, but this branch is not ready to merge because it is conflicting, lacks real behavior proof, and implements only one contested slice of the silent-reply contract.

## What This Changes

The PR adds an `isSilentTokenOnOwnLine` helper and changes `normalizeReplyPayload` tests/logic to suppress a whole reply when trailing `NO_REPLY` appears on its own line while preserving inline token stripping.

## Best Possible Solution

Choose one canonical, case-insensitive silent-reply classifier and apply it consistently through normalization and final delivery boundaries, with tests that preserve legitimate inline mentions.

## Reproduction Assessment

Yes, source-reproducible. Current main shows `normalizeReplyPayload({ text: "summary\n\nNO_REPLY" })` strips the token but keeps the preamble, and the current regression test asserts that behavior.

## Solution Assessment

No. The own-line direction is useful, but this patch is not the best final solution as-is because it is normalize-only, case-sensitive, conflicting, and competing with broader open proposals.
