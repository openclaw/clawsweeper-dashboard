## Summary

Keep open: current main still has the preamble-preserving normalize path, but this branch should not merge as-is because it lacks after-fix real behavior proof, only covers one delivery layer, and introduces a case-insensitive silent-token gap while related open PRs propose competing semantics.

## What This Changes

The PR adds an own-line silent-token helper and changes auto-reply normalization so text followed by a standalone trailing NO_REPLY is suppressed while inline NO_REPLY text is still stripped and delivered.

## Best Possible Solution

Define one shared, cached, case-insensitive silent-reply policy for exact, own-line, trailing, and inline token cases, apply it consistently through normalization and final delivery guards, then update focused tests and docs.

## Reproduction Assessment

Yes, source-reproducible: current main strips a trailing NO_REPLY token and returns the non-empty preamble from `normalizeReplyPayload`, and the current test suite asserts that behavior. The requested own-line suppression is not yet the documented contract, so the expected behavior still needs maintainer policy agreement.

## Solution Assessment

No. The own-line direction is plausible and narrower than a broad contains-based rule, but this patch is incomplete without final delivery coverage and case-insensitive matching, and it needs real behavior proof before merge.
