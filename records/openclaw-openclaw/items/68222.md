## Summary

Keep open. Current main still has the Kimi tagged-tool parser path that can turn tagged `sessions_yield` text into an executable tool call, and the runner still treats executed `sessions_yield` as a clean paused/end-turn exit; I did not find source or regression-test evidence that the premature pause/manual-resume symptom has been fixed.

## What This Changes

Prevent Kimi Code long-running tasks from being interrupted by unexpected `sessions_yield` tool calls that pause the run until manual input.

## Best Possible Solution

Add a narrow, transcript-backed guard or recovery path for unexpected Kimi-origin `sessions_yield` when no pending child follow-up is actually being awaited, while preserving the documented subagent yield flow.

## Reproduction Assessment

Unclear whether we have a high-confidence live reproduction path. Source inspection shows the failure mechanics if Kimi emits tagged `sessions_yield`, but the issue lacks a raw transcript and I did not establish a live current-main Kimi run that triggers the first unwanted yield.

## Solution Assessment

Unclear because there is no proposed patch. A blanket Kimi or runner block is not the best fix; the maintainable direction is a state-aware guard or recovery path that preserves legitimate `sessions_yield` semantics.
