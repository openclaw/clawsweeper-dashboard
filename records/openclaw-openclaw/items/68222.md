## Summary

Keep open. Current main removed one broad accepted-note recommendation, but Kimi tagged tool markup can still become a structured `sessions_yield` call, and the runner still turns that call into a paused end turn.

## What This Changes

The issue reports that Kimi Code can emit `sessions_yield` during long-running tasks, causing OpenClaw to pause until the user sends another message.

## Best Possible Solution

Add a transcript-backed, state-aware guard or recovery path for unexpected Kimi-origin `sessions_yield` calls when no spawned child completion is actually being awaited, while preserving legitimate subagent orchestration semantics.

## Reproduction Assessment

No, not with high confidence. The screenshot and current source show the pause path if Kimi emits tagged `sessions_yield`, but there is no raw current-main Kimi transcript or live provider run proving the trigger on today's main.

## Solution Assessment

Unclear because there is no patch. The best direction is a narrow state-aware Kimi or runner fix backed by a redacted transcript, not a blanket disablement of `sessions_yield`.
