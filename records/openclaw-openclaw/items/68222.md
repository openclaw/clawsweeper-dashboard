## Summary

Keep open. Current main still routes Kimi Code through the bundled Kimi provider, still rewrites Kimi tagged tool-call text into executable `toolCall` blocks, and still treats an executed `sessions_yield` as an intentional paused/end-turn exit. I found no current source, docs, tests, release notes, or related-item evidence that the unwanted Kimi-generated `sessions_yield` pause/manual-resume symptom has been fixed or superseded.

## What This Changes

The issue reports that Kimi Code can emit `sessions_yield` during long-running tasks, pausing OpenClaw until the user sends another message.

## Best Possible Solution

Add a narrow, state-aware guard or recovery path for `sessions_yield` execution when there is no pending spawned-child completion or expected follow-up event, and cover the Kimi tagged-tool stream path with a regression test proving that an unwanted Kimi `sessions_yield` does not leave the parent run paused for manual input.

## Reproduction Assessment

Unclear. The issue gives plausible manual steps and a screenshot, and current-main code inspection confirms the interruption mechanics, but there is no captured Kimi transcript or live reproduction proving the first unwanted yield emission.

## Solution Assessment

Unclear. Suppressing all `sessions_yield` calls is not the best solution because it is a real subagent orchestration tool; the maintainable path is a narrow runtime guard or recovery rule keyed to expected follow-up/subagent state plus Kimi-specific regression coverage.
