## Summary

Keep open. Current main still has a plausible Kimi-specific pause path: Kimi tagged tool markup is converted into executable tool calls without a `sessions_yield` guard, and the runner still treats `sessions_yield` as a paused end turn; the merged accepted-note cleanup did not prove this report fixed.

## What This Changes

The issue reports that Kimi Code can emit `sessions_yield` during long-running tasks, pausing the conversation until the user manually sends another message.

## Best Possible Solution

Add a transcript-backed, state-aware guard or recovery path for unexpected Kimi-origin `sessions_yield` calls when no spawned child completion is actually being awaited, while preserving legitimate subagent orchestration.

## Reproduction Assessment

No, not with high-confidence live proof. The screenshot and current source show the pause path if Kimi emits tagged `sessions_yield`, but there is no raw current-main Kimi transcript or live provider run proving the trigger today.

## Solution Assessment

Unclear because there is no patch. The safest direction is a narrow Kimi or runner guard backed by a redacted transcript, not a blanket disablement of `sessions_yield`.
