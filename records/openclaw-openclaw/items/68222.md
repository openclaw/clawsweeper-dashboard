## Summary

Keep open. Current main still has a plausible Kimi-specific path where tagged tool markup can become an executable `sessions_yield` call, and the runner still treats executed `sessions_yield` as an intentional paused end-turn; a recent guidance cleanup removed one prompt source but does not prove the reported unwanted Kimi yield is fixed.

## What This Changes

The issue reports that Kimi Code can emit `sessions_yield` during long-running tasks, causing OpenClaw to pause until the user sends another message.

## Best Possible Solution

Add a transcript-backed, state-aware guard or recovery path for unexpected Kimi-origin `sessions_yield` calls when no spawned child completion is actually being awaited, while preserving legitimate subagent orchestration semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live: the screenshot and current source show the pause path if Kimi emits tagged `sessions_yield`, but there is no raw current-main Kimi transcript or provider run proving the trigger on today’s main.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no patch; the best direction is a narrow state-aware Kimi or runner fix, not a broad disablement of `sessions_yield`.
