## Summary

Keep open. Current main still exposes Kimi Code, still rewrites Kimi tagged tool-call text into executable tool calls without a `sessions_yield` guard, and still maps any executed `sessions_yield` to a clean paused/end-turn run exit; I did not find source, test, or history evidence that the premature-yield/manual-resume symptom has been fixed.

## What This Changes

The issue reports that Kimi Code can emit `sessions_yield` during long-running conversations, pausing execution until manual user input resumes the run.

## Best Possible Solution

Add a narrow guard or recovery path that prevents an unexpected Kimi-origin `sessions_yield` from leaving the parent run paused when no spawned-child follow-up is actually pending, and cover that path with a captured Kimi transcript regression while preserving legitimate sub-agent yield semantics.

## Reproduction Assessment

Unclear. The issue gives plausible manual steps and a screenshot for Kimi Code on OpenClaw 2026.4.15, and source inspection confirms the pause mechanics, but there is no raw transcript or live current-main reproduction that proves the first unwanted `sessions_yield` emission.

## Solution Assessment

Unclear. There is no proposed patch; the best direction is not to block all `sessions_yield` calls, but to add a transcript-backed, state-aware guard or recovery path around unexpected Kimi yield calls.
