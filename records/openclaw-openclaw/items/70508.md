## Summary

Keep this PR open. Current main still has the Mattermost draft-preview overwrite path this PR is trying to fix, and the linked #43020 remains the canonical open issue, but the PR head is not safely mergeable because it adds an awaited draft-boundary flush in callback paths that are not serialized with partial updates and it changes the tool-status output without updating the existing test contract.

## What This Changes

The PR changes Mattermost draft streaming to track streamed content, rotate or flush draft preview posts at assistant/reasoning/tool boundaries, add item-event-aware tool-status updates, and render Mattermost tool status with emoji/title text.

## Best Possible Solution

Land one Mattermost-local fix that preserves streamed assistant text before assistant, reasoning, and tool boundaries, serializes boundary rotation with partial updates, keeps tool-status rendering within an intentional tested contract, and adds a focused regression test for partial text, tool boundary, next assistant text, and final delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: in current main, a Mattermost turn calls onPartialReply into one draftStream, then assistant/reasoning boundaries only reset lastPartialText and onToolStart writes tool status into that same draft post, so multi-step replies can overwrite earlier streamed text.

## Solution Assessment

No. The overall Mattermost-local direction is right, but this PR is not the best mergeable solution yet because the awaited boundary flush can race with fire-and-forget partial callbacks, and the status-text contract changed without matching tests or an explicit product choice.
