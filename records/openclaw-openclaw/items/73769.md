## Summary

Keep open: current main does not contain this PR's explicit memory CLI bootstrap, and the submitted guard can skip the bootstrap in the cold-registry case it is meant to fix because it checks the capability-aware provider list rather than registered adapters. The PR is small and repairable, but it needs a code adjustment and changelog entry before merge.

## What This Changes

The PR re-exports memory embedding provider registry helpers through the memory CLI host runtime, calls a new built-in provider registration guard before memory manager creation, and adds CLI regression tests for the registration path.

## Best Possible Solution

Use one canonical cold-registry strategy for memory embeddings: seed memory-core built-in adapters before the memory CLI manager path without consulting the capability-aware provider list, and keep the regression focused on a cold registered registry with capability fallback visible. Coordinate the final branch against #72937 and include the required active changelog entry.

## Reproduction Assessment

Yes. A focused unit reproduction is a cold registered-provider set with the memory-core manifest capability available: `listMemoryEmbeddingProviders()` can return `local`, the PR guard exits, and `registerMemoryEmbeddingProvider` is never called.

## Solution Assessment

No as submitted. The safer narrow fix is to call `registerBuiltInMemoryEmbeddingProviders` unconditionally, or check whether the specific built-in adapters are already registered, because the helper already owns registered-only idempotency.
