## Summary

Keep open: current main still does not add the PR’s explicit memory CLI provider bootstrap, and the proposed guard has a concrete correctness bug because it checks the capability-aware provider list instead of the registered adapter set. The overlapping open #72937 also means maintainers should choose the canonical bootstrap boundary before landing either version.

## What This Changes

The PR re-exports memory embedding provider registry helpers through the memory CLI host runtime, calls a built-in provider registration guard before memory manager creation, and adds memory CLI tests for registration and skip-if-present paths.

## Best Possible Solution

Adopt one canonical cold-registry strategy for memory embeddings: either add an explicit bootstrap at the accepted CLI or manager boundary that relies on registered-only adapter state, or keep capability fallback as the contract and add focused regression coverage for `memory status`, `memory index`, and `memory search` in the cold-registry shape.

## Reproduction Assessment

Yes. At source/unit level, mock `listMemoryEmbeddingProviders()` to return a capability adapter while no adapter is registered; the PR’s guard returns before `registerBuiltInMemoryEmbeddingProviders(...)`, so the built-in `local` adapter is never seeded.

## Solution Assessment

No as submitted. The direction is plausible, but the guard should check registered adapters only or call the existing built-in registration helper and let it perform idempotent filtering; the bootstrap location should also be reconciled with #72937.
