## Summary

Keep this PR open, but it needs changes before merge. The intended memory CLI provider bootstrap is relevant, but the submitted guard checks the capability-aware provider list, which can be non-empty while the process-global registered adapter set is still cold, so the new bootstrap can skip the registration it is meant to perform. The PR also lacks the required user-facing changelog entry.

## What This Changes

The PR re-exports memory embedding registry helpers through the memory CLI host runtime, adds a memory CLI bootstrap before manager creation, and adds CLI regression tests for provider registration.

## Best Possible Solution

Use one canonical cold-registry strategy for memory embeddings: either seed built-in adapters before the CLI manager path using registered-only state, or move the bootstrap to the manager/provider boundary if that is the shared fix. The final patch should include a regression where capability fallback exposes `local` while no registered adapter exists, plus the required changelog entry.

## Reproduction Assessment

Yes. A focused unit reproduction is a cold registered-provider set with the memory-core manifest capability available: `listMemoryEmbeddingProviders()` can return `local`, the PR guard exits, and `registerMemoryEmbeddingProvider` is never called.

## Solution Assessment

No as submitted. The safer narrow fix is to call `registerBuiltInMemoryEmbeddingProviders` unconditionally, or check `listRegisteredMemoryEmbeddingProviderAdapters()` before skipping, because the built-in helper already performs registered-only idempotency.
