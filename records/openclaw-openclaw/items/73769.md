## Summary

Keep open: the PR is aimed at a real memory CLI/provider bootstrap gap that is not implemented on current main, but the submitted guard can skip the very registration it is trying to add because it checks the capability-aware provider list instead of registered-only state.

## What This Changes

The PR re-exports memory embedding provider registry helpers through the memory CLI host runtime, calls a new built-in provider registration guard before memory manager creation, and adds CLI tests for cold and already-populated registration paths.

## Best Possible Solution

Use one canonical cold-registry strategy for memory embeddings: seed built-in memory-core adapters before memory manager/provider initialization using registered-only state, or call the built-in registration helper unconditionally and let its existing filter handle idempotency. The final patch should include the missing capability-fallback regression and an active changelog entry, while coordinating with #72937 so duplicate initialization logic does not land twice.

## Reproduction Assessment

Yes. A focused unit reproduction is a cold registered-provider set with the memory-core manifest capability available: `listMemoryEmbeddingProviders()` can return `local`, the PR guard exits, and `registerMemoryEmbeddingProvider` is never called.

## Solution Assessment

No as submitted. The safer narrow fix is to rely on `registerBuiltInMemoryEmbeddingProviders()` unconditionally, or skip only after checking `listRegisteredMemoryEmbeddingProviderAdapters()`, because capability fallback providers are not the same as registered adapters.
