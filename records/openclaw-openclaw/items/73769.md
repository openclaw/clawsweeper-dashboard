## Summary

Keep this PR open, but it is not ready as submitted. The memory CLI bootstrap direction is plausible and current main still does not explicitly seed built-in memory embedding providers before `getMemorySearchManager`, but the new guard checks the capability-aware provider list, so it can skip registration in the cold-registry shape the PR is trying to fix.

## What This Changes

The PR re-exports memory embedding provider registry helpers through the memory CLI host runtime, calls a new built-in provider registration guard before memory manager creation, and adds CLI tests for cold and already-populated registration paths.

## Best Possible Solution

Use one canonical cold-registry strategy for memory embeddings: seed built-in memory-core adapters before the CLI manager path using registered-only state, or move the bootstrap to the manager/provider boundary if that is the chosen shared fix. The final patch should include a regression for capability-fallback providers with an empty registered adapter set and coordinate with the overlapping provider-registration PR.

## Reproduction Assessment

Yes. A focused unit reproduction is a cold registered-provider set with the memory-core manifest capability available: `listMemoryEmbeddingProviders()` can return `local`, the PR guard exits, and `registerMemoryEmbeddingProvider` is never called.

## Solution Assessment

No as submitted. The narrow safer fix is to call `registerBuiltInMemoryEmbeddingProviders` unconditionally, or check `listRegisteredMemoryEmbeddingProviderAdapters()` before skipping, because the built-in helper already performs registered-only idempotency.
