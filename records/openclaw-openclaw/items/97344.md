## Summary

Keep this PR open: the Z.AI owner-boundary repair moved in the right direction, but the new static API-key doctor loop now calls an OAuth-oriented provider hook for every static profile, which can produce false `static [invalid_api_key]` warnings for valid sibling-provider API-key profiles.

## What This Changes

The PR rejects pasted Z.AI onboarding command strings during Z.AI API-key setup and surfaces provider-owned static API-key doctor hints with regression coverage.

## Best Possible Solution

Keep the Z.AI input rejection, but make static doctor surfacing safe for existing provider hooks by gating OAuth-only hints or adding profile-type-aware hook coverage before merge.

## Reproduction Assessment

Yes, at source level: current main accepts non-empty Z.AI API-key text into `zai:default`, and current doctor skips static API-key profiles. The introduced review finding is also source-reproducible from the new static loop plus the existing Anthropic OAuth repair hook.

## Solution Assessment

No as currently patched: the Z.AI rejection and provider-owned hint are the right direction, but the generic static doctor loop needs to avoid invoking OAuth-only provider hints as static API-key invalidation.
