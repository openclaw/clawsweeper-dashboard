## Label Justifications

- `P1`: The issue affects guarded outbound-send denial and auditability for real channel sends, which can break a security-sensitive agent/channel workflow.
- `impact:security`: The report is about guard plugins failing to block or audit outbound sends when grants are missing or revoked.
- `impact:session-state`: The requested fix depends on preserving `agentId`, session, and request identity for downstream correlation.
- `impact:message-loss`: The gateway path currently collapses hook-cancelled delivery into `No delivery result`, losing the structured sent-versus-blocked message outcome.

## Summary

Keep open. Current main still leaves the gateway `send` path unable to surface hook-cancelled sends as a structured guarded denial, while the outbound `message_sending` hook context still lacks the requested agent/session/request identity, and open related PRs need maintainer coordination before this issue can close.

## What This Changes

Define one outbound `message.send` path that runs through `message_sending`, returns either a real delivery result or explicit guarded denial, and carries identity for audit correlation.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/63007 in coordination with the open hook-identity work so one documented outbound path returns sent-versus-blocked outcomes and carries the agreed correlation fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: hook cancellation records a suppressed durable outcome, but gateway `send` ignores that status and throws `No delivery result`, while the hook context omits requested identity fields.

## Solution Assessment

Unclear. Durable suppression is the right base, but maintainers still need to choose whether the canonical contract is gateway-only or must also align direct CLI and plugin-action sends.
