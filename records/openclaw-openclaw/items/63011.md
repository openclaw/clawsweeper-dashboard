## Summary

Keep open. Current main still does not provide one canonical outbound `message.send` path that is simultaneously hook-bearing, deliverable, and auditable: hook cancellation still produces no structured blocked result, gateway `send` still collapses that empty delivery into `No delivery result`, and the hook context does not receive the requested outbound identity fields. The issue is also paired with open draft PR #63007, so it should remain open for maintainer review rather than be swept closed or sent to a competing repair lane.

## What This Changes

The issue requests a canonical outbound send contract that runs through `message_sending`, returns real delivery results on allow, returns explicit guarded-denial results on block, and preserves agent/session/request identity for audit correlation.

## Best Possible Solution

Ship or replace #63007 with a maintainer-approved outbound-send contract that threads agent/session/request identity into `message_sending`, returns explicit guarded-denial outcomes on cancellation, preserves provider delivery results on success, and clearly aligns or documents the supported canonical path across gateway, direct CLI, and plugin-action sends.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to register a `message_sending` hook that returns `{ cancel: true }`; `deliverOutboundPayloads` then returns no delivery result, and gateway `send` maps that to `No delivery result`. The issue also provides a concrete Telegram/Tessera live sequence for the same split.

## Solution Assessment

Unclear. The paired draft PR #63007 is a plausible narrow solution if maintainers choose gateway `send` as the canonical path, but the issue text also asks about direct CLI and plugin-action semantics, which may need a broader contract decision.
