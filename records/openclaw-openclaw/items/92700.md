## Summary

Keep open: the branch addresses a real source-reproducible read-tool bug, but the current fixed-order legacy fallback is not the best or safest repair because it can silently decode invalid UTF-8 as GBK on every platform before the correct legacy decoder is reached.

## What This Changes

The PR changes the read tool to decode invalid UTF-8 buffers through fixed legacy CJK TextDecoder fallbacks and adds a standalone repro script.

## Best Possible Solution

Use the existing Windows active-codepage decoder, or an equivalent single selected platform encoding, in the read text path with focused GBK and valid-UTF-8 regression coverage.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 decode read-tool text buffers as UTF-8, and the reported GBK sample is invalid UTF-8. I did not run a live Chinese Windows read-tool session in this read-only review.

## Solution Assessment

No. The broad fixed-order heuristic is not the best fix; the existing Windows codepage-aware decoder is narrower and avoids changing non-Windows or non-GBK legacy file behavior.
