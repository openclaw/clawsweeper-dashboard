## Summary

This PR targets a valid read-tool encoding bug, but the current fallback is broader than the reported Windows codepage problem and can silently decode other legacy files as GBK before their actual encoding is tried. Keep it open for a narrower owner-boundary fix rather than passing or closing it as-is.

## What This Changes

The branch adds a read-tool buffer decoder that tries strict UTF-8 first, then falls back through several CJK legacy encodings, plus a standalone GBK and Shift_JIS repro script.

## Best Possible Solution

Reuse or extend the existing Windows active-codepage decoder for the read-tool text path, with focused GBK regression coverage and no cross-platform ordered guessing unless maintainers explicitly choose a broader encoding-detection policy.

## Reproduction Assessment

Yes at source level: current main decodes read-tool text buffers as UTF-8, and the linked issue's GBK byte sample would mojibake through that path. I did not run a live Chinese Windows read-tool session in this read-only review.

## Solution Assessment

No as submitted: automatic fallback without a new API is the right direction, but the fixed CJK order is too broad and can misdecode other legacy files before consulting the platform's actual code page. The narrower maintainable fix is to reuse or extend the existing Windows active-codepage decoder for read-tool text.

Full review comments:

- **[P1] Narrow fallback to the active Windows encoding:** `src/agents/sessions/tools/read.ts:141-142`
  - body: This changes every non-image read call to try `gbk`/`gb18030` first whenever strict UTF-8 fails, regardless of platform or the user's actual locale. Because `TextDecoder('gbk')` can return a non-empty string for many byte sequences, Shift_JIS, Big5, EUC-KR, Windows-1252, and other legacy files can be silently returned as plausible but wrong GBK text before their correct decoder is tried; the PR body already shows the Shift_JIS sample comes back garbled. Please reuse or extend the existing Windows codepage-aware decoder so the fallback follows the active Windows encoding instead of a global hard-coded order.
  - confidence: 0.93
- **[P2] Move the repro into focused tests:** `scripts/repro-92664.mjs:86-94`
  - body: The added script is a one-off repo script that writes temp files and treats a garbled Shift_JIS decode as a passing proof path, so it can keep passing while the broad fallback is wrong. This should be focused regression coverage for the read tool and the Windows codepage helper, not a permanent script that encodes weak assertions as proof.
  - confidence: 0.82
