## Summary

Keep open for revision: the PR adds a plausible explicit-encoding mitigation, but the implementation routes advertised `gbk` and `shift_jis` values through `Buffer.toString()`, which rejects those labels at runtime, and the supplied proof only demonstrates latin1 plumbing rather than the reported GBK Windows behavior.

## What This Changes

Adds an optional `encoding` parameter to the built-in `read` session tool and a standalone repro script for non-UTF-8 file reads.

## Best Possible Solution

Prefer the existing Windows-codepage-aware decoder or another supported `TextDecoder`-based contract, then add focused GBK read-tool regression coverage and real after-fix proof before choosing this branch over sibling approaches.

## Reproduction Assessment

Yes at source level: current main decodes read-tool text buffers as UTF-8, and the linked issue demonstrates GBK bytes becoming mojibake under that path. I did not run a live Chinese Windows read-tool session in this read-only review.

## Solution Assessment

No as written: the explicit parameter is a possible product direction, but this implementation uses the wrong decoding API for the advertised encodings and leaves typed read-tool contracts stale. The narrower in-tree Windows-codepage fallback remains the safer default direction unless maintainers explicitly choose a new API.
