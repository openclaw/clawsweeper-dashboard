## Summary

Keep open: the linked Chinese Windows GBK read bug is valid, but this branch is not merge-ready because the fixed-order legacy fallback silently treats other invalid-UTF-8 files as GBK and the added proof script counts mojibake as success. The narrower sibling at https://github.com/openclaw/openclaw/pull/92682 is still open and proof-blocked, so it is not a safe superseding close target yet.

## What This Changes

The PR changes the read tool to try strict UTF-8 and then fixed CJK legacy TextDecoder fallbacks, and adds a standalone GBK/UTF-8/Shift_JIS repro script.

## Best Possible Solution

Use the existing Windows active-codepage decoder in the read text path, add focused read-tool GBK and valid-UTF-8 coverage, and require refreshed after-fix proof before closing the linked user bug.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 decode read-tool text buffers as UTF-8, and the linked GBK byte sample is invalid UTF-8. I did not run a live Chinese Windows read-tool session in this read-only review.

## Solution Assessment

No. The broad fixed-order heuristic is not the best fix; the existing Windows codepage-aware decoder is narrower and avoids changing non-Windows or non-GBK legacy file behavior.
