## Summary

Keep open: the PR targets a real read-tool bug, but the implementation wires the new `encoding` parameter to `Buffer.toString`, which does not support the advertised GBK or Shift_JIS values, so the linked Chinese Windows case would still fail. The branch also introduces lint errors in the proof script, and the closer sibling PR at https://github.com/openclaw/openclaw/pull/92682 is not proof-positive enough to make this PR safely superseded.

## What This Changes

The PR adds an optional read-tool `encoding` argument, routes text decoding through that argument, updates the tool description, and adds a repo-root reproduction script.

## Best Possible Solution

Use the existing dependency-free strict-UTF-8-then-Windows-codepage/TextDecoder path for read-tool text content, add focused GBK regression coverage, and choose one canonical sibling PR before merging any new read-tool API surface.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` decode read-tool text buffers as UTF-8, and Node probes show GBK bytes need `TextDecoder` or another supported decoder rather than `Buffer.toString`. I did not run a live Chinese Windows read-tool session in this read-only review.

## Solution Assessment

No. Adding an explicit encoding parameter is a plausible product direction, but this implementation uses the wrong Node API for the advertised encodings and should be replaced by the existing decoder path or a validated, contract-aligned API.
