## Summary

Keep open. Current main still has a source-level AltGr failure path: OpenClaw delegates printable editor input to pi-tui, and the pinned pi-tui decoder still rejects printable CSI-u events that carry Alt/Ctrl modifier bits, which is the reported German AltGr `@`/`€` path.

## What This Changes

The issue reports German-layout TUI input duplication and broken AltGr printable characters in the gateway-client/TUI input path.

## Best Possible Solution

Land a focused AltGr-aware TUI input fix, preferably upstream in pi-tui or through a narrow OpenClaw adapter, with regressions for German `@`/`€` and duplicate printable handling.

## Reproduction Assessment

Yes for the remaining AltGr/Kitty path by source inspection: printable CSI-u data with Alt/Ctrl modifier bits is rejected before editor insertion. No live German keyboard run was performed, so duplicate non-backspace printable behavior is not fully reproduced.

## Solution Assessment

No current complete solution is on main. The existing StdinBuffer behavior may partially mitigate duplicated unmodified printables, but the narrow maintainable fix still needs AltGr-aware printable decoding without broadening normal Alt/Ctrl shortcut insertion.
