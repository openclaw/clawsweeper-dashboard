## Summary

Keep open. Current main still delegates TUI editor printable input to pi-tui, and the pinned `@earendil-works/pi-tui@0.74.0` decoder still rejects printable Kitty CSI-u events when Alt/Ctrl modifier bits are present, which keeps the reported German AltGr `@`/`€` path unresolved.

## What This Changes

The issue asks OpenClaw to fix German-layout TUI input so printable characters are inserted once and AltGr combinations such as `@` and `€` work in the gateway-client composer.

## Best Possible Solution

Land a focused AltGr-aware printable input fix in the TUI/pi-tui boundary, preferably upstream in pi-tui or through a narrow OpenClaw adapter, with regression coverage for German AltGr `@`/`€` and duplicate printable handling.

## Reproduction Assessment

Yes for the AltGr/Kitty path by source inspection: current main still routes editor input to pi-tui, and the pinned pi-tui decoder rejects printable CSI-u input with Alt/Ctrl modifier bits. No live German keyboard run was performed, so the duplicate plain printable half remains only partially reproduced.

## Solution Assessment

No complete solution is on current main. The maintainable fix should be narrow AltGr-aware printable decoding or a dependency upgrade that preserves normal Alt/Ctrl shortcut matching instead of broadly inserting all modified keys.
