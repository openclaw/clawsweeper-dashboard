## Summary

Keep open. Current main still delegates TUI editor input to the pinned pi-tui decoder, and pi-tui 0.70.6 still rejects printable Kitty CSI-u events when Alt/Ctrl modifier bits are present, leaving the reported German AltGr @/€ path unresolved. The recent pi-tui bump and OpenClaw backspace dedupe work look like partial progress for duplicate key events, but there is no current-main coverage or implementation for German-layout AltGr printable insertion.

## What This Changes

The issue requests a fix for German keyboard layout input in the TUI, where printable characters can duplicate and AltGr combinations such as @ and € fail.

## Best Possible Solution

Fix the TUI input boundary narrowly so AltGr-produced printable text is inserted as text while real Alt/Ctrl shortcuts remain shortcuts. The preferred end state is an upstream pi-tui fix consumed by OpenClaw, or a small OpenClaw adapter with focused regression coverage for German-layout @/€ and duplicate printable handling until upstream carries it.

## Reproduction Assessment

Yes for the remaining AltGr path: a synthetic Kitty printable CSI-u event with Alt/Ctrl modifier bits is rejected by pi-tui 0.70.6 before CustomEditor can insert it. The broader duplicate-character half still needs a live German-layout or terminal-sequence regression check on top of the current upstream unmodified-printable dedupe.

## Solution Assessment

No current complete solution is on main. The pi-tui 0.70.6 bump and OpenClaw backspace dedupe are partial, but the narrow maintainable fix still needs AltGr-aware printable decoding without broadening normal Alt/Ctrl shortcut behavior.
