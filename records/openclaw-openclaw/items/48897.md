## Summary

Keep open. Current main and the latest release still delegate TUI printable editor input to pi-tui, and the pinned pi-tui decoder still rejects Alt/Ctrl-modified Kitty CSI-u printable sequences, so the German AltGr path remains unresolved.

## What This Changes

The issue reports German-layout TUI input corrupting printable text and failing AltGr characters such as `@` and `€` in the gateway-client/TUI editor.

## Best Possible Solution

Land a focused AltGr-aware printable input fix at the TUI/pi-tui boundary, with regression coverage for German AltGr `@`/`€` and live proof that plain German printable characters insert once.

## Reproduction Assessment

Partly yes. The AltGr/Kitty CSI-u path is source-reproducible by OpenClaw call-path inspection and a direct pi-tui decoder probe; the duplicated plain printable examples still need live German-layout terminal proof.

## Solution Assessment

No complete solution is on current main. The maintainable fix is a narrow adapter or upstream pi-tui change that allows AltGr-produced printable text without turning real Alt/Ctrl shortcuts into inserted characters.
