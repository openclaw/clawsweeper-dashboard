## Summary

Keep open. Current main and the latest release still route TUI editor printable input through the pinned pi-tui decoder, and that decoder still rejects Alt/Ctrl-modified printable Kitty CSI-u events, so the reported German AltGr path remains unresolved.

## What This Changes

The issue asks for German keyboard layout input in the TUI to insert printable characters once and handle AltGr combinations such as `@` and `€` correctly.

## Best Possible Solution

Land a focused AltGr-aware printable input fix at the TUI/pi-tui boundary, with regression coverage for German AltGr `@`/`€` and proof that plain German printable characters insert once.

## Reproduction Assessment

Yes for the AltGr/Kitty path by source inspection and a direct pi-tui decoder probe: current main routes editor input through pi-tui, and the pinned decoder rejects Alt/Ctrl-modified printable CSI-u input. No live German keyboard run was performed, so the duplicate plain printable half is not fully reproduced here.

## Solution Assessment

No complete solution is on current main. The best fix should narrowly preserve normal Alt/Ctrl shortcuts while allowing AltGr-produced printable characters, preferably upstream in pi-tui or through a small OpenClaw-side adapter with tests.
