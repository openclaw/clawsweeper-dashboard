## Summary

Keep open: current `main` still lowercases Signal group IDs in central routing/session metadata paths, and the issue provides credible v2026.5.12 runtime evidence that this breaks Signal group auto-replies. Prior related reports were closed as narrower fixes or already-implemented paths, but no open canonical fix PR now owns this new regression surface.

## What This Changes

The issue reports a v2026.5.12 regression where Signal group auto-replies are generated but not delivered because mixed-case Signal group IDs are lowercased in routing/session state.

## Best Possible Solution

Preserve Signal group IDs as opaque case-sensitive strings across route/session/delivery keys and add regression coverage; if corrupted persisted state needs repair, put that repair in `openclaw doctor --fix`.

## Reproduction Assessment

Yes, at source level: a mixed-case Signal group ID flows from `event-handler.ts` into `resolveAgentRoute`, then through helpers that lowercase the group peer/session key and stored `groupId`. I did not run a live Signal daemon reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is provider-aware opaque-ID preservation at routing/session normalization boundaries, with Signal regression tests across inbound, route resolution, explicit session normalization, persistence metadata, and delivery.
