## Summary

Keep open: current main still has multiple central routing/session-key paths that lowercase Signal group peer IDs before or after Signal-specific target normalization, so the reported v2026.5.12 regression is still plausible and source-reproducible.

## What This Changes

The issue reports a regression where Signal group auto-replies in v2026.5.12 fail because mixed-case Signal group IDs are lowercased in session/routing/delivery paths.

## Best Possible Solution

Preserve provider-owned opaque IDs through routing, explicit session normalization, inbound session recording, and group history keys, with Signal group regression coverage around mixed-case Base64 group IDs.

## Reproduction Assessment

Yes, at source level: a mixed-case Signal group ID flows from Signal inbound handling into resolveAgentRoute and then through generic lowercase session-key paths. I did not run a live Signal daemon reproduction in this read-only review.

## Solution Assessment

Yes, the maintainable fix is a central provider-aware opaque-ID rule at routing/session-key boundaries, not another Signal-only call-site patch; existing Signal target normalization already preserves group target casing but is bypassed by generic lowercase normalization.
