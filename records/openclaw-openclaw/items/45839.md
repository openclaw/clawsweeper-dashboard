## Summary

Keep open. Current main and the latest release still have generic Telegram inline-button and exec-approval plumbing, but pairing approval is still text-command based and the one-tap approve/deny path needs a security/product contract before implementation.

## What This Changes

The issue requests Telegram inline Approve/Deny buttons for pairing requests so operators do not need to paste CLI or slash commands.

## Best Possible Solution

Define a typed, replay-safe pairing approval action contract that covers DM pairing and device-pair requests, then render it as Telegram inline buttons while existing pairing state and authorization checks remain the source of truth.

## Reproduction Assessment

Yes, as a source-level missing capability. Current main and v2026.6.8 still emit pairing approval instructions as text commands, and the Telegram callback handler has no pairing approve/deny branch.

## Solution Assessment

Unclear. Inline buttons are a plausible UX, but the best fix is a typed approval/callback contract with explicit auth, expiry, replay, deny, and scope semantics rather than raw command-string callbacks.
