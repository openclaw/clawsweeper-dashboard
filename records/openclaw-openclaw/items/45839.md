## Summary

Keep open. Current main has generic Telegram inline-button and callback infrastructure, but the DM pairing challenge and device-pair notification paths still send text-only approval instructions, and the requested one-tap approve/deny flow crosses a pairing authorization boundary.

## What This Changes

Request adding Telegram inline approve/deny buttons to pairing notifications so operators can resolve pairing requests without copying CLI or slash commands.

## Best Possible Solution

Add a pairing-specific Telegram callback workflow that reuses existing inline-button primitives, binds callbacks to pairing request IDs, preserves approval scopes, handles expiry/replay and duplicate taps, supports reject semantics, and updates or clears buttons after resolution.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction because this is a feature request; source inspection gives high confidence that current main still requires text commands for pairing approval and has no pairing-specific Telegram callback branch.

## Solution Assessment

Is this the best way to solve the issue? No current implementation fully solves this; the maintainable path is a scoped pairing callback workflow, not generic callback text injection or shelling out to copied CLI commands.
