## Summary

Keep this open. Current main has generic Telegram inline-button callbacks, and the discussion correctly identifies them as reusable building blocks, but pairing approve/deny buttons are still not implemented for DM pairing replies or device-pair notifications.

## What This Changes

Request to add Telegram inline Approve/Deny buttons to pairing notifications so operators can approve or deny pairing requests without copying CLI commands.

## Best Possible Solution

Design a pairing action surface that reuses Telegram inline callbacks, adds buttons to the relevant pairing notifications, preserves existing scope checks, and defines exactly which pairing types support deny/reject.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs give a high-confidence current-behavior check: pairing flows remain text-command based while generic Telegram button support exists separately.

## Solution Assessment

No current implementation fully solves this. The requested one-tap flow is useful, but the maintainable path needs explicit pairing authorization and notification-target policy rather than a blind callback shortcut.
