## Summary

Keep open. Current main has generic Telegram inline-button and callback infrastructure, but pairing challenge and device-pair notification paths still send text-only approval instructions, and the approve/deny pairing authorization policy remains security-sensitive.

## What This Changes

The issue requests Telegram inline Approve/Deny buttons for pairing requests instead of requiring operators to paste CLI or slash commands.

## Best Possible Solution

Add a pairing-specific Telegram callback workflow that reuses existing inline-button primitives, binds callbacks to pairing request IDs, preserves approval scopes, handles expiry/replay and duplicate taps, supports reject semantics, and updates or clears buttons after resolution.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main still requires text commands for pairing approval and has no pairing-specific Telegram callback branch.

## Solution Assessment

No current implementation fully solves this. The maintainable path is a scoped pairing callback workflow, not generic callback text injection or shelling out to copied CLI commands.
