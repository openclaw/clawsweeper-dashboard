## Summary

Keep open: current main and v2026.6.10 still render Telegram/device-pair pairing approval as command text, while existing Telegram inline-button infrastructure only handles exec/plugin approvals and generic callbacks; the remaining work needs a security/product decision before implementation.

## What This Changes

The issue requests Telegram inline approve and deny buttons for pairing requests so operators do not need to paste CLI approval commands.

## Best Possible Solution

Define a typed, replay-safe pairing approval action contract and render it as Telegram inline buttons while the existing pairing stores and approval-time scope checks remain authoritative.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.10 emit pairing approval instructions as text commands, and the Telegram callback handler has no pairing approve/deny branch; no live Telegram scenario was run in this read-only review.

## Solution Assessment

Unclear: inline buttons are plausible, but the best fix needs typed action/callback semantics with explicit owner auth, scope handling, expiry, replay/idempotency, denial, and first-owner bootstrap behavior.
