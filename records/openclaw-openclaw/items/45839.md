## Summary

Keep this open. Current main and the latest shipped release still keep Telegram pairing flows text-command based, while the requested one-tap approve/deny path adds unresolved authorization and reject-semantics decisions.

## What This Changes

Request Telegram inline Approve/Deny buttons on pairing notifications so operators can act without copying CLI commands.

## Best Possible Solution

Design a pairing-specific Telegram callback workflow that reuses existing inline-button primitives, binds callbacks to inspected request IDs, preserves operator scope checks, handles expiry/replay, and updates or clears buttons after resolution.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and tests give a high-confidence current-behavior check: pairing replies and device-pair notifications still use text approval commands.

## Solution Assessment

No current implementation fully solves this. The maintainable direction is pairing-specific callback handling with explicit authorization and post-action state handling, not a generic callback fallthrough to approval commands.
