## Summary

Keep this open. Current main has generic Telegram inline callbacks and plugin/exec approval handling, but DM pairing and device-pair notifications still present text-only approve commands, there is no pairing-specific approve/deny callback branch, and the May 8 scope report remains a real product/security design question.

## What This Changes

Request Telegram inline buttons that approve or deny pairing requests instead of requiring operators to paste pairing commands.

## Best Possible Solution

Design a first-class pairing action surface that reuses Telegram inline callbacks, binds buttons to specific pending requests, edits/removes buttons after resolution, defines authorized notification targets, and preserves operator.admin requirements for admin-scoped pairing requests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Current-main source and tests give a high-confidence behavior check: pairing replies and device-pair notifications are text-only, and Telegram command-owner approvals only carry operator.pairing today.

## Solution Assessment

No current implementation fully solves it. The one-tap workflow is a useful direction, but the maintainable fix needs explicit pairing authorization and notification policy rather than a blind Telegram callback or broad scope change.
