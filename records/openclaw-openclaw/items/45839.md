## Summary

Keep open. Current main still sends pairing notifications and replies as text commands; generic Telegram inline-button infrastructure exists, but pairing-specific approve/reject callbacks and approval-scope policy are not implemented yet.

## What This Changes

The issue requests Telegram inline Approve/Deny buttons for pairing requests instead of requiring operators to paste text approval commands.

## Best Possible Solution

Add a pairing-specific Telegram callback workflow that uses existing inline-button primitives, binds callbacks to inspected request IDs, preserves caller-scope checks, handles expiry/replay, supports reject semantics, and clears or updates buttons after resolution.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs give a high-confidence current-behavior check: pairing flows still require text approval commands.

## Solution Assessment

No current implementation fully solves this. The requested direction is useful, but the maintainable fix should call scoped pairing APIs from structured Telegram callbacks rather than shelling out to CLI commands or falling through as generic callback text.
