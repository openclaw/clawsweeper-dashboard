## Summary

Keep this open. Current main has generic Telegram inline-button callbacks and native exec/plugin approval buttons, but channel pairing and device-pair notifications still use text commands; the recent Control UI note also matches source where Telegram command owners get only pairing scope, not admin scope.

## What This Changes

The issue requests Telegram inline Approve/Deny buttons for pairing requests so operators do not need to paste pairing CLI commands.

## Best Possible Solution

Design an explicit pairing action surface that reuses Telegram inline callbacks, defines notification targets and authorized approvers, supports reject only where the backing store allows it, and preserves operator.admin checks.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: channel pairing replies and device-pair notifications are text-only, and the Telegram owner-scope limitation is covered by current tests.

## Solution Assessment

No current implementation fully solves this. The one-tap flow is a useful direction, but the maintainable fix needs an explicit pairing authorization and notification policy rather than a blind Telegram callback or scope broadening.
