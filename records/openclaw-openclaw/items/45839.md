## Summary

Keep this open. Current main has generic Telegram inline-button callbacks and native exec/plugin approval buttons, but DM channel pairing still uses text-only pairing replies and has no pairing-specific Telegram approve/deny callback path.

## What This Changes

The issue requests Telegram inline approve/deny buttons for WhatsApp pairing notifications so operators do not need to paste pairing CLI commands.

## Best Possible Solution

Design an explicit channel-pairing action surface that reuses Telegram inline-button infrastructure, defines who may approve or deny, adds a real pending-request rejection path if denial is required, and documents the notification target policy.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection does give a high-confidence current-behavior check: channel pairing replies are text-only and the Telegram callback handler has no pairing approve/deny branch.

## Solution Assessment

No current implementation or related closed issue fully solves it. The maintainable direction is to extend the pairing approval surface deliberately rather than piggybacking on generic agent callback text.
