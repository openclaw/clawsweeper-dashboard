## Summary

Keep this open. Current main has generic Telegram inline-button and callback plumbing, but pairing notifications and DM pairing replies still use text-only approval commands, and the latest discussion adds unresolved authorization and deny/reject semantics.

## What This Changes

Request adding Telegram inline Approve/Deny buttons to pairing notifications so operators can approve or reject requests without copying CLI commands.

## Best Possible Solution

Design a pairing action surface that reuses existing Telegram callback primitives, adds buttons to the relevant pairing notifications, preserves operator scope checks, and defines reject semantics for both pairing-code and device-pair requests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, and tests give a high-confidence current-behavior check: pairing flows remain text-command based while generic Telegram button support exists separately.

## Solution Assessment

No current implementation fully solves this. The maintainable path is a pairing-specific callback workflow with explicit authorization and post-action state handling, not a blind shortcut from arbitrary callback data to approval commands.
