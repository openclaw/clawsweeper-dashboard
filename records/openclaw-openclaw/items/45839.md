## Summary

Keep this open. Current main has generic Telegram inline-button callbacks and native exec/plugin approval buttons, but DM pairing still uses text-only pairing replies and has no pairing-specific Telegram approve/deny callback path.

## What This Changes

Request to add Telegram inline Approve/Deny buttons for DM pairing notifications so operators can resolve pairing without copying a CLI command.

## Best Possible Solution

Add a first-class DM pairing action surface that can render Telegram buttons, authorize only configured operators/owners, implement explicit approve and reject semantics, and update the pairing and Telegram docs.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Current-main source inspection gives a high-confidence behavior check: pairing issues text replies and Telegram callback handling has no pairing action branch.

## Solution Assessment

No current implementation fully solves it. Reusing the generic Telegram button transport is sensible, but the maintainable fix should wire a dedicated pairing action API instead of routing security decisions through generic agent callback text.
