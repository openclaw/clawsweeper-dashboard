## Summary

Keep open. Current main has generic Telegram inline buttons, plugin callback dispatch, and native exec approval buttons, but DM pairing still uses text-only pairing challenges plus CLI approval, and there is no current pairing approve/deny callback path.

## What This Changes

Request Telegram inline approve/deny buttons for pairing requests so operators do not need to paste `openclaw pairing approve/deny` commands manually.

## Best Possible Solution

Add a first-class pairing action UX that preserves the existing DM pairing security model: define where operator notifications are delivered, add audited approve and reject semantics for pending DM pairing requests, authorize Telegram callbacks only for configured operators, update the button state after action, and cover Telegram plus affected pairing-store behavior in tests and docs.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with `channels.telegram.dmPolicy = "pairing"`, an unauthorized DM enters `enforceTelegramDmAccess`, receives a text-only pairing code reply, and current Telegram callback handling has no pairing approve/deny branch.

## Solution Assessment

No. Generic Telegram inline buttons are useful infrastructure, but they route callbacks to plugins or agent text by default; the requested pairing workflow needs a first-class pairing action path that mutates pairing state safely instead of relying on an agent turn or pasted CLI command.
