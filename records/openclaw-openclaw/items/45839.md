## Summary

Keep open. Current main still keeps Telegram pairing approval text-command based, while the requested one-tap approval path crosses pairing authorization, replay/expiry, and reject/deny UX decisions that should be settled by maintainers first.

## What This Changes

Request to add Telegram inline Approve/Deny buttons for pairing notifications instead of requiring copied CLI commands.

## Best Possible Solution

Design a pairing-specific Telegram callback workflow that reuses existing inline-button primitives, binds callbacks to inspected request IDs, preserves caller scope checks, handles expiry/replay, and updates or clears buttons after resolution.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, tests, docs, and v2026.5.12 inspection give a high-confidence current-behavior check that pairing replies and device-pair notifications still use text approval commands.

## Solution Assessment

No current implementation fully solves this. The maintainable direction is a pairing-specific callback path with explicit authorization and post-action state handling, not a generic callback fallthrough to approval commands.
