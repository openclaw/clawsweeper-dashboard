## Summary

Keep open. Current main and v2026.6.6 have generic Telegram inline-button/callback plumbing, but pairing approval still resolves through text or CLI commands, and one-tap approve/deny needs a security/product contract before implementation.

## What This Changes

The issue requests Telegram inline approve/deny buttons for pairing requests so operators do not have to copy and run pairing approval commands manually.

## Best Possible Solution

Design a generic, replay-safe pairing approval action contract that covers DM channel pairing and device-pair requests, then let Telegram render that contract as inline buttons while existing pairing state and authorization checks remain the source of truth.

## Reproduction Assessment

Yes. Current main and v2026.6.6 still emit pairing approval instructions as text commands, and the Telegram callback handler has no pairing approve/deny branch.

## Solution Assessment

Unclear. Inline buttons are a plausible UX, but the best fix is a typed approval/callback contract with explicit auth, expiry, replay, and deny semantics rather than raw command-string callbacks.
