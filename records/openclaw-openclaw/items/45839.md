## Label Justifications

- `P2`: This is a useful workflow and availability improvement with bounded surface area, but it changes pairing authorization behavior and needs maintainer review.
- `impact:security`: The requested buttons would approve or deny device/browser pairing, which is a trust and token issuance boundary.

## Summary

Keep open. Current main has generic Telegram inline-button and callback infrastructure, but the pairing flows still send text-only approval instructions and lack a pairing-specific approve/deny callback path, reject UX, replay/expiry semantics, and a settled approval-scope policy.

## What This Changes

Request Telegram pairing notifications with inline Approve/Deny buttons that perform pairing approval or denial instead of requiring copied CLI commands.

## Best Possible Solution

Add a pairing-specific Telegram callback workflow that reuses existing inline-button primitives, binds callbacks to pairing request IDs, preserves approval scopes, handles expiry/replay and duplicate taps, supports reject semantics, and updates or clears buttons after resolution.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main still requires text commands for DM/device pairing approval and has no pairing-specific Telegram callback branch.

## Solution Assessment

No current implementation fully solves this. The maintainable path is a scoped pairing callback workflow, not generic callback text injection or shelling out to copied CLI commands.
