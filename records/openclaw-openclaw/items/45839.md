## Summary

Keep open. Current main and the latest release have generic Telegram inline-button and callback plumbing, but pairing approval still resolves through CLI/text commands, and the requested one-tap approve/deny path needs a security/product contract before implementation.

## What This Changes

Request to add Telegram inline Approve/Deny buttons for pairing requests so operators do not copy pairing approval commands by hand.

## Best Possible Solution

Design a generic, replay-safe pairing approval action contract, then let Telegram render it as inline buttons while keeping pairing state and authorization checks in the existing pairing owners.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection does give a high-confidence current-state check: DM pairing and device-pair approval still require text/CLI commands rather than Telegram approve/deny buttons.

## Solution Assessment

Unclear. Inline buttons are a plausible UX, but the best fix should be a typed approval/callback contract with explicit auth, expiry, and deny semantics, not raw command-string callbacks.
