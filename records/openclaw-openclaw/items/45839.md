## Summary

Keep this open. Current main already has generic Telegram inline-button callbacks and native exec/plugin approval buttons, and the discussion correctly points at those as reusable building blocks, but pairing-specific approve/deny buttons are still not implemented: DM pairing replies and device-pair notifications still tell operators to run text commands, and the added Control UI scope concern matches current source/tests.

## What This Changes

The issue requests Telegram pairing notifications with inline Approve/Deny buttons so operators can resolve pairing requests without copying CLI commands.

## Best Possible Solution

Design a pairing action surface that reuses Telegram inline callbacks, adds buttons to the appropriate pairing notifications, preserves exact scope checks, and only exposes reject/deny where the backing pairing store supports it.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and docs give a high-confidence current-behavior check: pairing flows are command/text based while generic Telegram button support exists separately.

## Solution Assessment

No current implementation fully solves this. The one-tap flow is a useful direction, but the maintainable fix needs an explicit pairing authorization and notification policy rather than a blind Telegram callback or broad scope grant.
