## Summary

Keep open. Current main and the latest release have generic Telegram inline-button support and device-pair Telegram notifications, but pairing approval paths still rely on text commands and there is no pairing-specific approve/deny callback workflow.

## What This Changes

The issue requests Telegram inline buttons that approve or deny pairing requests instead of requiring operators to copy and run pairing commands.

## Best Possible Solution

Add a pairing-specific Telegram callback workflow that reuses existing inline-button primitives, binds actions to pairing request IDs or codes, preserves approval scopes, defines deny/reject semantics, handles expiry/replay/duplicate taps, and updates or clears buttons after resolution.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main and `v2026.5.22` gives high confidence that pairing approval still requires text commands and no pairing-specific Telegram callback branch exists.

## Solution Assessment

No current implementation fully solves this. The maintainable path is a scoped pairing callback workflow with explicit authorization and replay semantics, not generic callback text injection or shelling out to copied CLI commands.
