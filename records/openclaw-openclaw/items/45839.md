## Summary

Keep this open. Current main has generic Telegram inline-button and callback plumbing, but pairing still sends text-only approve commands and has no pairing-specific approve/deny callback path or DM pairing deny surface.

## What This Changes

The issue requests Telegram inline approve/deny buttons for pairing notifications so operators can respond without pasting pairing commands manually.

## Best Possible Solution

Design a pairing action surface that reuses Telegram inline callbacks, adds buttons to the relevant pairing notifications, preserves existing scope checks, and defines deny/reject semantics per pairing type.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, and tests give a high-confidence current-behavior check: pairing flows remain text-command based while generic Telegram button support exists separately.

## Solution Assessment

No current implementation fully solves this. The requested one-tap flow is useful, but the maintainable path needs explicit pairing authorization and notification-target policy rather than a blind callback shortcut.
