## Summary

Keep open. Current main still sends pairing replies and device-pair notifications as text commands; Telegram inline-button infrastructure exists for other flows, but pairing-specific approve/reject callbacks and approval-scope policy remain unimplemented.

## What This Changes

The issue requests Telegram inline buttons that let an operator approve or deny pairing requests instead of copying and running text commands.

## Best Possible Solution

Add a pairing-specific Telegram callback workflow that reuses existing inline-button primitives, binds callbacks to request IDs, preserves approval scopes, handles expiry/replay, supports reject semantics, and clears or updates buttons after resolution.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: pairing flows still require text commands and no pairing-specific Telegram callback exists.

## Solution Assessment

No current implementation fully solves this. The maintainable path is a scoped pairing callback workflow, not shelling out to CLI text commands or relying on generic callback text injection.
