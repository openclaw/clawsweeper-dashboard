## Summary

Keep this open. Current main has generic Telegram callback and inline-button infrastructure, plus device-pair `/pair approve`, but DM pairing and device-pair notifications still resolve through text commands rather than pairing-specific approve/deny buttons; the latest Control UI pain point also matches source where Telegram command owners receive only `operator.pairing`, which cannot approve `operator.admin` browser pairing requests.

## What This Changes

The issue requests Telegram inline approve/deny buttons for pairing requests so operators can resolve pairing from Telegram instead of copying CLI commands.

## Best Possible Solution

Design an explicit pairing action surface that reuses Telegram inline callbacks, defines the notification target, supports approve and reject where the backing pairing store allows it, and preserves `operator.admin` requirements for admin/browser pairing.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: pairing replies and device-pair notifications are text-only, and the `/pair approve` owner-scope limitation is covered by current tests.

## Solution Assessment

No current implementation fully solves this. The requested one-tap flow is directionally useful, but the maintainable fix needs explicit pairing authorization and notification policy rather than a blind Telegram callback or scope broadening.
