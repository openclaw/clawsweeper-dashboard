## Summary

Keep open. Current main has generic Telegram inline-button and callback infrastructure, and the agent message-button request at https://github.com/openclaw/openclaw/issues/59573 is closed as implemented, but the actual pairing paths still send text commands rather than pairing-specific approve/deny buttons; the latest browser-pairing comment also matches the current `operator.pairing` versus `operator.admin` authorization boundary.

## What This Changes

The issue requests Telegram inline approve and deny buttons for pairing requests instead of requiring operators to paste pairing CLI commands.

## Best Possible Solution

Design an explicit pairing action surface that reuses Telegram structured callbacks, defines the notification target, supports approve and reject where the backing pairing store allows it, and preserves `operator.admin` requirements for admin/browser pairing.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence current-behavior check: pairing replies and device-pair notifications are text-only, and the `/pair approve` owner-scope limitation is covered by current tests.

## Solution Assessment

No current implementation fully solves this. The requested one-tap flow is directionally useful, but the maintainable fix needs explicit pairing authorization and notification policy rather than a blind Telegram callback or scope broadening.
