## Best Possible Solution

Revise the branch so diagnostics share runtime session resolution, timeout setup always clears its connect watchdog on setup failure, and the focused heartbeat/timeout/doctor tests cover those paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear the connect timer when stream setup fails:** `src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:275`
  - body: If `maybeStream` rejects before the timeout, this race rejects with the provider/setup error but leaves `connectTimeoutPromise` scheduled. When the timer later fires it calls `onIdleTimeout` and reports a false initial-stream timeout for a request that already failed; clear the connect timer on rejection or in a `finally` before rethrowing.
  - confidence: 0.88
- **[P2] Canonicalize the configured session before doctor lookup:** `src/commands/doctor-heartbeat-session-target.ts:55`
  - body: The runtime resolves `heartbeat.session` through `toAgentStoreSessionKey` and `canonicalizeMainSessionAlias` before loading `store[canonical]`, but this doctor check looks up the raw configured string. Valid alias or unprefixed session values can therefore warn as missing even though the heartbeat runtime finds the session; reuse the runtime resolution/canonicalization before emitting this warning.
  - confidence: 0.9
