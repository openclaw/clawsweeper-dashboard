## Summary

Keep this PR open. The core heartbeat fixes are well motivated and the supplied live gateway logs are useful, but the new doctor warning can produce false warnings for valid configurations before merge.

## What This Changes

The PR parallelizes multi-agent heartbeat broadcast, scopes busy-lane checks per agent, appends HEARTBEAT.md prose to more prompt paths, adds an initial LLM stream connect timeout, and adds doctor coverage/docs/tests for missing heartbeat session targets.

## Best Possible Solution

Keep the heartbeat runner and timeout fixes, but make the doctor check mirror runtime enablement and delivery resolution before it emits a missing-session warning.

## Reproduction Assessment

Yes. The original heartbeat issues have source-level support in current main and supplied live logs; the doctor false-warning regression is reproducible from the PR source against the current delivery and disabled-cadence contracts.

## Solution Assessment

No. The main fix direction is maintainable, but the doctor warning should be narrowed so it only warns when a missing session entry actually causes dropped heartbeat delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip explicitly routed heartbeat recipients:** `src/commands/doctor-heartbeat-session-target.ts:85-87`
  - body: Please gate this warning on whether delivery actually depends on the missing session entry. With `heartbeat.target: "telegram"` plus `heartbeat.to`, `resolveHeartbeatDeliveryTarget` uses the explicit recipient even when `entry` is undefined, and existing coverage expects that path to deliver. This helper currently warns solely because the store entry is missing and tells the operator replies will be dropped as `reason="no-target"`, which is false for explicitly routed heartbeat configs.
  - confidence: 0.88
- **[P3] Skip disabled heartbeat cadences:** `src/commands/doctor-heartbeat-session-target.ts:64-68`
  - body: Please mirror the runtime interval check before emitting this warning. `every: "0m"` is the documented disable path and `resolveHeartbeatIntervalMs` returns null for it, so the runner will not run that heartbeat. As written, a disabled heartbeat that still has a stale `session` and non-none `target` gets a warning saying heartbeats will run and drop replies.
  - confidence: 0.79
