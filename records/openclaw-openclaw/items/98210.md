## Label Justifications

## Summary

Keep open for maintainer review: the patch fixes a real scoped Talk SecretRef readback bug and looks technically correct, but it intentionally changes a credential-bearing Gateway response and needs explicit boundary acceptance before merge.

## What This Changes

The PR updates Gateway `talk.config` to materialize only the active Talk provider `apiKey` for scoped `includeSecrets` callers while preserving source-shaped provider maps and redacted read-scope responses, with focused Gateway tests.

## Best Possible Solution

Land this narrow Gateway fix after maintainers accept `operator.talk.secrets` as the boundary for resolved Talk provider credentials; otherwise replace it with a narrower native credential handoff contract.

## Reproduction Assessment

Yes: source inspection shows current main returns early for `includeSecrets:true` before runtime SecretRef resolution, while the iOS parser only accepts a string `apiKey`. I did not run a physical iOS session in this read-only review.

## Solution Assessment

Yes for the code layer: Gateway owns runtime SecretRef resolution and iOS cannot resolve SecretRef objects. The remaining question is maintainer acceptance of `operator.talk.secrets` as the credential boundary for native clients.
