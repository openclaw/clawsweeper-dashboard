## Summary

Keep open: this is a plausible and well-proven fix for the linked iOS Talk SecretRef bug, but the PR still has a concrete lint failure and it intentionally changes a credential-bearing Gateway response boundary that needs maintainer acceptance before merge.

## What This Changes

The PR changes Gateway `talk.config` so authorized `includeSecrets` callers can receive the active Talk provider API key as a string in `talk.resolved.config.apiKey`, while source provider maps stay redacted/source-shaped and new regressions cover scoped and read-scope behavior.

## Best Possible Solution

Fix the lint-only CI failure, then land this narrow Gateway response change only after maintainers accept `operator.talk.secrets` as the native credential-read boundary; otherwise replace it with a narrower native handoff contract.

## Reproduction Assessment

Yes, source inspection on current main shows `includeSecrets` returns before runtime SecretRef resolution, while iOS only accepts a string `apiKey`. I did not run a physical iOS session in this read-only review.

## Solution Assessment

Yes for the code layer: Gateway owns runtime SecretRef materialization and iOS cannot resolve SecretRef objects. The best merge path is to fix the lint-only CI failure and have maintainers explicitly accept the `operator.talk.secrets` credential boundary.
