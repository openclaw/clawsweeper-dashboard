## Label Justifications

- `P1`: The PR targets a broken native iOS Talk workflow where configured SecretRef-backed ElevenLabs credentials are treated as missing and may push users toward plaintext credential workarounds.
- `merge-risk: 🚨 compatibility`: The scoped `talk.config` response shape changes for `talk.resolved.config.apiKey` from a SecretRef object to a resolved string.

## Summary

Keep open: the patch is a narrow code-correct fix for the linked iOS Talk SecretRef bug with sufficient live Gateway proof, but it changes credential-bearing response semantics and needs maintainer confirmation before merge.

## What This Changes

This PR changes Gateway `talk.config` so scoped `includeSecrets` callers receive a resolved Talk provider `apiKey` string under `talk.resolved.config.apiKey`, with Gateway tests for scoped resolution and read-scope redaction.

## Best Possible Solution

Land the narrow Gateway response fix after a maintainer confirms `operator.talk.secrets` is the intended boundary for resolved Talk provider credentials; otherwise design a narrower native Talk credential handoff.

## Reproduction Assessment

Yes: source inspection shows current main and `v2026.6.11` return early for `includeSecrets:true`, while the iOS parser only accepts a string API key. I did not run a failing physical iOS session in this read-only review.

## Solution Assessment

Yes for the code layer: Gateway owns the resolved config payload and iOS cannot resolve SecretRef objects. The remaining merge question is maintainer acceptance of `operator.talk.secrets` as the credential boundary.
