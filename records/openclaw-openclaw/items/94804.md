## Summary

Keep open: current main has targeted wake, origin delivery-context carry, hooks, plugin event subscriptions, and dedupe building blocks, but it still lacks the requested first-class async route registry with lifecycle, matching, ambiguity handling, privacy policy, and safe action policy.

## What This Changes

The issue requests a first-class async result routing layer that maps later external events back to the originating topic session, posts an optional visible summary, and wakes that session safely.

## Best Possible Solution

Define a narrow plugin/API-friendly async route contract that composes existing hooks, targeted wake, delivery context, dedupe, and approval policy before deciding whether any core storage or Gateway-owned registry is warranted.

## Reproduction Assessment

Not applicable: this is a feature request, not a reported broken current contract. Source inspection confirms the lower-level wake, hook, delivery-context, and plugin-event pieces exist while the requested async route registry and policy layer are not present.

## Solution Assessment

Unclear as an implementation direction. A core route registry may be useful, but VISION.md and AGENTS.md both push toward a lean core and plugin/API seams, so maintainers should choose the boundary before implementation.
