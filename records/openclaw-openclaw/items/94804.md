## Summary

Keep open: current main and the latest release have targeted wake, hook mappings, delivery context, commitments, and dedupe building blocks, but not the requested first-class async route registry with external-event matching, route lifecycle, ambiguity handling, privacy policy, and safe action policy. This is a product/API boundary decision, not a narrow repair.

## What This Changes

Request a first-class async result routing layer that maps later external events to originating topic sessions, posts safe summaries, dedupes events, and wakes the saved session.

## Best Possible Solution

Define the minimal product/API contract first: which pieces belong in core, which event-source watchers live in plugins, what storage and privacy policy apply, and how the route runner composes existing hooks, wake, delivery, and dedupe primitives.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior with a current-main reproduction path. Source inspection confirms the lower-level wake, hook, delivery-context, and dedupe pieces exist while the requested route registry and matcher layer are not implemented.

## Solution Assessment

Unclear until maintainers choose the boundary. A standard callback-routing contract is plausible, but the maintainable path should reuse existing hook, wake, plugin-state, dedupe, and delivery primitives instead of adding a parallel orchestration stack.
