## Summary

Keep open: current main and v2026.6.8 have the lower-level targeted wake, delivery-context, hook, and dedupe pieces, but they still do not provide the requested first-class async route registry with lifecycle, matching, ambiguity handling, privacy policy, and safe action policy.

## What This Changes

The issue requests a framework-level async result routing layer that maps later external events back to the originating topic session, posts a summary, and wakes that session safely.

## Best Possible Solution

Define a narrow plugin/API-friendly async route contract that composes existing hooks, targeted wake, delivery context, dedupe, and approval policy before deciding whether any core-owned storage or Gateway registry is warranted.

## Reproduction Assessment

Not applicable: this is a feature request, not a reported broken current contract. Source inspection confirms the lower-level wake, hook, delivery-context, and dedupe pieces exist while the requested async route registry and policy layer are not present.

## Solution Assessment

Unclear as an implementation direction. A core route registry may be useful, but VISION.md and AGENTS.md both push toward a lean core and plugin/API seams, so maintainers should choose the boundary before implementation.
