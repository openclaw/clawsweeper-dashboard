## Summary

Keep open: current main and v2026.6.8 provide targeted wake, hooks, task delivery, commitments, and dedupe primitives, but not the requested first-class async route registry with lifecycle, matching, ambiguity handling, privacy policy, approval-safe action policy, and durable or semi-durable dedupe.

## What This Changes

The issue requests a first-class async route registry that maps later external events back to the originating topic session for visible summary, targeted wake, dedupe, and approval-safe follow-up.

## Best Possible Solution

Define a narrow plugin/API-backed async route contract first, reusing existing targeted wake and delivery primitives and adding SQLite-backed state only where route lifecycle, dedupe, and approval policy are explicit.

## Solution Assessment

Unclear as an implementation: maintainers should choose the product/API boundary first because a full core registry could duplicate existing hook, cron, task, commitment, and plugin paths.
