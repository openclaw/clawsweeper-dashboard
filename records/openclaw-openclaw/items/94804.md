## Summary

Keep open: current main and v2026.6.8 already have targeted wake/session-key primitives, hook mappings, delivery-context carry, and dedupe helpers, but they do not provide the requested first-class async route registry with route lifecycle, matching, privacy, ambiguity, and action policy.

## What This Changes

The issue requests a first-class async result routing layer that maps later external events back to the originating topic session, posts a safe visible summary, wakes that session, and handles matching, dedupe, privacy, and action policy.

## Best Possible Solution

Keep this open for maintainer design: define a narrow plugin/API-friendly async route contract that composes existing hooks, targeted wake, delivery context, and dedupe primitives without growing a heavyweight core event bus by default.

## Reproduction Assessment

Not applicable: this is a feature request, not a reported broken current contract. Source inspection confirms the low-level targeted wake pieces exist, while the requested route registry and policy layer are not present as a current-main feature.

## Solution Assessment

Unclear as an implementation direction. A core route registry may be useful, but VISION.md pushes optional capability toward plugins and warns against heavy orchestration layers, so maintainers should choose the API/product boundary before implementation.
