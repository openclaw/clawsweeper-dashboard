## Summary

Keep open: current main has the lower-level targeted wake, hook, cron, delivery-context, dedupe, and plugin workflow pieces, but it still lacks the requested first-class async route registry with lifecycle, matching, ambiguity handling, privacy policy, and safe action policy.

## What This Changes

The issue requests a first-class async route registry that can match later external events to originating topic sessions, deliver summaries, dedupe, and wake the saved session with safety policy.

## Best Possible Solution

Define a narrow plugin/API-friendly async route contract that composes existing hooks, targeted wake, delivery context, dedupe, and approval policy before deciding whether any Gateway-owned registry or storage belongs in core.

## Reproduction Assessment

Not applicable: this is a feature request, not a reproducible bug report. Source inspection confirms lower-level wake, hook, delivery-context, and plugin-event pieces exist while the requested async route registry and policy layer are not present.

## Solution Assessment

Unclear as an implementation direction. A core registry might be useful, but the safer next step is a maintainer product/API decision that defines whether this belongs in core storage, a plugin contract, or a ClawHub/plugin pattern.
