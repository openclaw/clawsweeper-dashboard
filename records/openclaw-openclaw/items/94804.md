## Summary

Keep open: current main and v2026.6.8 have targeted wake, hook mappings, commitments, dedupe, and delivery-context building blocks, but not the requested first-class async route registry with lifecycle, external-event matching, ambiguity handling, privacy policy, and safe action policy.

## What This Changes

The issue requests a first-class async route registry that matches later external events back to originating topic sessions, posts safe summaries, dedupes events, and wakes the saved session.

## Best Possible Solution

Define a narrow plugin/API-friendly async route contract that composes existing hooks, targeted wake, delivery context, dedupe, commitments, and approval policy before deciding whether any core-owned registry is warranted.

## Reproduction Assessment

Not applicable: this is a feature request, not a reported broken current contract. Source inspection confirms the lower-level wake, hook, delivery-context, dedupe, and commitment pieces exist while the requested async route registry and policy layer are not present.

## Solution Assessment

Unclear as an implementation direction. A core route registry may be useful, but VISION.md and AGENTS.md both push toward a lean core and plugin/API seams, so maintainers should choose the boundary before implementation.
