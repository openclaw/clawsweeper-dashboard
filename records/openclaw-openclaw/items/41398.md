## Summary

Keep open. Current main still has source-level Google prefix paths, but this PR is not mergeable as-is because it reintroduces Gemini API keys in request URLs, regresses current Gemini 3 Pro alias normalization, lacks real behavior proof, and overlaps the open consolidation work at https://github.com/openclaw/openclaw/pull/41517 and https://github.com/openclaw/openclaw/issues/71932.

## What This Changes

The branch adds Google model prefix stripping across Google helpers, manifest normalization, native Gemini transport/PDF URL construction, and regression tests, while also changing unrelated browser, Signal, channel catalog, and bundled-plugin guard code.

## Best Possible Solution

Consolidate this PR with https://github.com/openclaw/openclaw/pull/41517 and https://github.com/openclaw/openclaw/issues/71932 into one narrow Google-owned fix that strips only redundant Google prefixes before alias lookup and native Gemini URL/PDF construction, preserves header-based credentials and current Gemini aliases, and drops unrelated changes.

## Reproduction Assessment

Yes. Source inspection on current main shows prefixed Google IDs can remain unstripped through the Google helper and native/PDF URL builders, producing `models/google/...` paths for the reported class.

## Solution Assessment

No. The prefix-stripping direction is correct, but this PR is not the best mergeable solution because it regresses credential placement and alias behavior; the safer path is a focused provider-scoped normalization repair with the overlapping PRs reconciled.
