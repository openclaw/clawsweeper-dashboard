## Summary

Keep open. The Google prefix bug is valid and still source-reproducible on current main, but this PR is not mergeable as-is because it leaks Gemini PDF API keys into URLs, regresses current Gemini 3 Pro alias canonicalization, lacks after-fix real behavior proof, and mixes unrelated browser, Signal, channel catalog, and bundled-plugin changes.

## What This Changes

The PR strips Google-owned model prefixes in Google model normalization, manifest policies, native Gemini transport/PDF URL construction, adds regression tests, and also changes unrelated browser trash, Signal installer, channel catalog, and bundled-plugin override code.

## Best Possible Solution

Consolidate the Google prefix work into one narrow provider-scoped fix that strips redundant Google prefixes before alias lookup and native Gemini URL/PDF construction, preserves header-based credentials and current aliases, and drops unrelated files.

## Reproduction Assessment

Yes. Current main has a source-level path where `google/gemini-*` can pass through Google normalization and native/PDF URL builders, producing a `models/google/...` request path; I did not run live Google API traffic in this read-only review.

## Solution Assessment

No. The prefix-stripping direction is right, but this PR is not the best mergeable solution because it regresses credential placement and Gemini aliases while carrying unrelated changes.
