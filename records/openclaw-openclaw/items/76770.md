## Summary

Keep open: current main still supports static tool reduction, but it does not expose a user-facing defer-loading flag, lazy discovery index, or first-party OpenClaw MCP schema materialization flow.

## What This Changes

The issue requests a config-controlled way to defer or selectively lazy-load first-party OpenClaw MCP tool schemas to reduce per-turn prompt cost.

## Best Possible Solution

Add a documented lazy schema contract that composes with existing profile and allow/deny policy, uses provider-specific defer support only where safe, and preserves eager fallback behavior elsewhere.

## Reproduction Assessment

Not applicable; this is a feature request for new config/runtime behavior, not a broken existing behavior. Source inspection confirms current main eagerly builds and returns selected tool schemas and has no lazy first-party schema discovery path.

## Solution Assessment

Unclear; the requested directions are plausible, but the best implementation needs maintainer design across tool policy, MCP loopback, Codex dynamic tools, plugin tools, and eager-provider fallback behavior.
