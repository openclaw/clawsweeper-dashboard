## Summary

Keep open: current main still has static tool profile/allow/deny controls and Codex-side dynamic-tool exclusion, but it does not expose a user-facing defer-loading flag, lazy tool index, or schema materialization flow for first-party OpenClaw MCP tools. The existing ClawSweeper comment reached the same conclusion, and this pass rechecked the relevant source/docs on the newer main SHA.

## What This Changes

Expose user-editable first-party OpenClaw MCP tool deferral or per-tool gating so long-running sessions can avoid eagerly sending every built-in tool schema on every turn.

## Best Possible Solution

Add a documented lazy schema contract that composes with existing tool policy, uses provider-specific defer support where safe, and preserves eager fallback behavior for runtimes that cannot lazy-load schemas.

## Reproduction Assessment

Not applicable; this is a feature request for new config/runtime behavior, not broken existing behavior. Source inspection confirms current main has static filtering controls but no first-party lazy schema discovery path.

## Solution Assessment

Unclear; the requested options are plausible, but the best implementation needs maintainer design across OpenClaw tool policy, Codex deferLoading support, bundled/plugin tools, and eager-provider fallback behavior.
