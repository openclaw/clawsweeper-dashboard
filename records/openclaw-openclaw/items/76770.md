## Summary

Keep open: current main already has static tool profile and allow/deny controls that can reduce the exposed tool set, but it does not implement a `defer_loading`/tool-index flow or lazy schema discovery for first-party tools.

## What This Changes

Expose user-configurable deferred or per-tool loading for first-party OpenClaw MCP/tool schemas to reduce repeated prompt token overhead in long-running agents.

## Best Possible Solution

Design a first-class lazy schema/index contract across the agent runtime, MCP surfaces, provider adapters, config schema, docs, and tests while preserving existing static allow/deny policy as the conservative baseline.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current behavior. Source inspection confirms current main supports static filtering but not a high-confidence deferred schema discovery path.

## Solution Assessment

Unclear: the requested direction is plausible, but the best design needs maintainer product and adapter-level architecture decisions. Static `tools.allow`/`tools.deny` is a useful workaround, not the same as lazy-loading disabled schemas later.
