## Summary

Keep open: current main supports static tool reduction through profiles and allow/deny policy, but it does not implement a user-facing defer-loading flag, per-tool lazy discovery index, or first-party OpenClaw tool schema materialization flow.

## What This Changes

Request to expose a user-editable lazy or deferred loading mechanism for first-party OpenClaw MCP/OpenClaw tool schemas.

## Best Possible Solution

Add a documented runtime-owned lazy schema contract that composes with existing profile and allow/deny policy, uses provider-specific defer support only where safe, and preserves eager fallback behavior elsewhere.

## Reproduction Assessment

Not applicable; this is a feature request for new config/runtime behavior, not a broken existing contract. Source inspection confirms current main has static filtering but no lazy first-party schema discovery path.

## Solution Assessment

Unclear; the requested direction is plausible, but the best implementation needs maintainer design across OpenClaw tool policy, bundled MCP/OpenClaw tools, Codex `deferLoading`, and provider fallback behavior.
