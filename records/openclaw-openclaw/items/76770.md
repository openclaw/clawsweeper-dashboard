## Summary

Keep open: current main has static profile/allow/deny controls, runtime-scoped tool allowlists, and Codex dynamic-tool excludes, but it does not expose a user-facing defer-loading flag, lazy tools index, or first-party OpenClaw schema materialization flow. The existing ClawSweeper comment remains accurate after rechecking current main.

## What This Changes

Requests a user-configurable way to defer or disable first-party OpenClaw MCP tool schemas so only a small core is sent eagerly and remaining tools can be materialized later.

## Best Possible Solution

Design and document a lazy schema/materialization contract that composes with existing tool policy, uses runtime-specific defer support where safe, and preserves eager fallback behavior elsewhere.

Not applicable; this is a feature request for new config/runtime behavior, not a broken existing contract. Source inspection confirms current main has static filtering and Codex static excludes, but no user-editable defer-loading or tools-index materialization path.
