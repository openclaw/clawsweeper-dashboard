## Summary

Keep open: current main has static tool reduction through profiles and allow/deny policy, plus a Codex dynamic-tool transport, but it does not implement a user-facing defer-loading flag, first-party OpenClaw tools index, or lazy schema materialization flow.

## What This Changes

Request to add user-editable defer-loading or per-tool gating for first-party OpenClaw MCP tool schemas so only a small core is eagerly exposed and the rest can be discovered later.

## Best Possible Solution

Design a documented lazy schema contract that composes with existing profiles and allow/deny policy, uses provider defer support where safe, and preserves eager fallback behavior elsewhere.

## Reproduction Assessment

Not applicable; this is a feature request for new config/runtime behavior, not a broken existing contract. Source inspection confirms current main only provides static filtering and eager registration for selected OpenClaw tools.

## Solution Assessment

Unclear; the requested direction is plausible, but the maintainable fix needs maintainer design across OpenClaw tool policy, bundle MCP, Codex deferLoading, and provider fallback behavior.
