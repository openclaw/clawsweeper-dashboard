## Summary

Keep open: current main supports static tool policy and the Codex adapter has dynamic-tool filtering controls, but there is still no user-facing defer-loading flag, lazy MCP tools index, or first-party OpenClaw MCP schema materialization flow.

## What This Changes

The issue requests a user-editable defer-loading or per-tool gating configuration for first-party OpenClaw MCP tool schemas.

## Best Possible Solution

Add a documented lazy schema contract that composes with existing profile and allow/deny policy, uses provider-specific defer support only where safe, and preserves eager fallback behavior elsewhere.

## Reproduction Assessment

Not applicable; this is a feature request for new config/runtime behavior, not broken existing behavior. Source inspection confirms current main has static filtering and eager schema listing, but no lazy first-party MCP schema discovery path.

## Solution Assessment

Unclear; the requested directions are plausible, but the best implementation needs maintainer design across OpenClaw MCP loopback, PI/Codex dynamic tools, provider defer support, and eager-provider fallback behavior.
