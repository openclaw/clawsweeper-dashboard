## Summary

Keep open: current main already has static `tools.profile` and `tools.allow`/`tools.deny` filtering that can reduce the declared tool set, but it does not expose the requested first-party MCP/OpenClaw tool schema defer-loading or discovery flow.

## What This Changes

Expose a user-editable first-party OpenClaw MCP tool defer/gating mechanism so long-running agents can keep a small core schema set in prompt and load the rest lazily.

## Best Possible Solution

Design a core/runtime-owned lazy tool schema mechanism that preserves existing `tools.profile` and allow/deny semantics, documents the new config, and uses provider-specific dynamic-tool support only where the selected runtime can honor it safely.

## Reproduction Assessment

Not applicable; this is a feature request for new config/runtime behavior, not a broken existing contract. Source inspection confirms current main has static filtering but no lazy first-party tool schema discovery path.

## Solution Assessment

Unclear; the requested direction is plausible, but the best shape needs maintainer design across Pi custom tools, Codex dynamic tools, MCP naming, and fallback behavior for providers that cannot defer schemas.
