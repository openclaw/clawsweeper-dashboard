## Summary

Keep open: current main supports static tool policy and has a generated Codex protocol field for deferred dynamic tools, but it does not expose a user-facing defer-loading flag, lazy discovery index, or first-party MCP schema materialization flow.

## What This Changes

Request a user-editable config flag or per-tool policy that defers or lazily discovers first-party OpenClaw MCP tool schemas.

## Best Possible Solution

Design a documented lazy schema contract that composes with existing profile/allow/deny/byProvider policy, uses provider-supported defer semantics where available, and falls back to eager declarations elsewhere.

## Reproduction Assessment

Not applicable; this is a feature request for new config/runtime behavior, not broken existing behavior. Source inspection confirms current main has static filtering and Codex protocol type support, but no lazy first-party MCP schema discovery path.

## Solution Assessment

Unclear; the requested directions are plausible, but Option A overlaps existing static policy while Option B requires a new cross-runtime schema materialization contract. The maintainable path should be designed before implementation.
