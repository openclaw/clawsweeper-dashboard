## Summary

Keep open: current main still offers static tool profiles, allow/deny policy, and Codex-specific static excludes, but it does not expose `defer_loading`, per-first-party OpenClaw tool enablement with later rediscovery, or a `tools_index` lazy materialization flow. This is a new cross-runtime config and tool-schema contract, so it needs maintainer design rather than cleanup closure.

## What This Changes

Request a user-editable config path to defer or gate first-party OpenClaw MCP/dynamic tool schemas so long-running agents can keep a small eager tool set and lazy-load the rest.

## Best Possible Solution

Design and document a lazy schema/exposure contract that composes with existing tool policy and falls back to eager declarations where provider support is not safe or available.

## Reproduction Assessment

Not applicable; this is a feature request for new config/runtime behavior, not a broken existing contract. Source inspection confirms current main has static profile/allow/deny and Codex exclude controls but no lazy OpenClaw tool discovery or schema materialization path.

## Solution Assessment

Unclear; the requested directions are plausible, but Option A versus Option B and provider fallback behavior need maintainer design. The maintainable path should start with a documented contract around `tools.*` policy, runtime support, and eager fallback before adding config.
