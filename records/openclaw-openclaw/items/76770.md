## Summary

Keep open: current main has static tool profiles, allow/deny/byProvider filtering, MCP loopback schema caching, and Codex dynamic-tool exclusion controls, but no user-facing defer-loading flag or lazy first-party OpenClaw MCP schema discovery/materialization flow.

## What This Changes

Expose a user-editable lazy-loading or per-tool gating control so first-party OpenClaw MCP tool schemas do not all appear in every turn.

## Best Possible Solution

Add a documented lazy schema contract that composes with existing profile and allow/deny policy, uses provider-specific defer support only where safe, and preserves eager fallback behavior elsewhere.

## Reproduction Assessment

Not applicable; this is a feature request for new config/runtime behavior, not a broken existing contract. Source inspection confirms static filtering exists today, but no lazy first-party schema discovery path is present.

## Solution Assessment

Unclear; the requested shapes are plausible, but the best implementation needs maintainer design across OpenClaw tool policy, MCP loopback schema exposure, Codex deferLoading support, bundled/plugin tools, and eager-provider fallback behavior.
