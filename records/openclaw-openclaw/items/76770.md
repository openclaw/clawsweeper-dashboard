## Summary

Current main already provides the requested small-core tool-schema control through `tools.allow`/`tools.deny` and profiles, and a main-only follow-up now avoids optional media/PDF factory work when those tools are denied; the exact `defer_loading` spelling is not present, but the central prompt-cost trimming path is implemented.

## What This Changes

Request to let users trim first-party OpenClaw MCP/core tool schemas, either through per-tool gating or a lazy `defer_loading` discovery mode.

## Best Possible Solution

Keep the existing `tools.allow`/`tools.deny` policy path as the supported way to trim first-party tool schemas, and track any future dynamic discovery loader as a separate product design item if maintainers want it.

## Reproduction Assessment

Not applicable: this is a feature/config request, not a broken-behavior report. Source, docs, and tests show the current-main configuration path for trimming callable tool schemas.

## Solution Assessment

Yes for the central trimming need: `tools.allow`/`tools.deny` is narrower and already integrated into the existing tool policy pipeline before schemas are sent. A full dynamic lazy-discovery tool would be a separate enhancement rather than required to solve this request.
