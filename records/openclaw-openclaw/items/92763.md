## Summary

Keep open: this targets a real Gateway attachment-gate bug, but the broad `gemini-*` fallback can override explicit text-only metadata and the external PR only provides unit-test proof, not real post-patch Gateway/Gemini CLI behavior.

## What This Changes

The PR adds two `google-gemini-cli` Gemini-name fallback branches in `resolveGatewayModelSupportsImages` so stale or missing catalog rows are treated as image-capable.

## Best Possible Solution

Restrict the fallback to contract-backed Gemini generative image-capable families, add stale/missing/negative resolver coverage, and require redacted real Gateway/Gemini CLI proof before merge.

## Reproduction Assessment

Yes. The linked issue provides a concrete failing Gateway command, and current source shows the resolver can return false before attachment parsing throws the reported `UnsupportedAttachmentError`; I did not run the live Gemini CLI flow in this read-only review.

## Solution Assessment

No, not as written. A provider-specific fallback is a plausible mitigation, but the predicate should follow OpenClaw’s narrower Google model-family contract and include negative coverage before merge.
