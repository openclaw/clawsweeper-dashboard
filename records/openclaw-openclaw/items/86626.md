The PR adds final `message_end` unwrapping for standalone `message(action=send)` JSON in gated OpenAI completions and message-tool-only cases, plus a focused handler regression test.

## Best Possible Solution

Align standalone envelope route detection with the message-tool resolver so both `to` and `target` are treated as routed, add a regression for the `to` automatic path, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, from source inspection: an assistant `message_end` with `api: "openai-completions"` and a standalone `message` JSON envelope using `arguments.to` is classified as unrouted by the PR guard even though current message-tool routing supports `to`. ## Solution Assessment

No, not yet. The gated unwrap approach is reasonable, but it needs to match the existing message-tool target contract and include real behavior proof before this is the best mergeable fix.
