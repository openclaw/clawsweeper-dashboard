## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject collapsed empty segments in named-DM parsing:** `src/sessions/session-key-utils.ts:184-185`
  - body: `parseNamedDmSessionKey` delegates to `parseAgentSessionKey`, which drops empty `:` segments before this validation runs. A malformed key like `agent:main:dm-named:123::work` can therefore be accepted as `{ peerId: "123", name: "work" }`; validate the raw suffix or reject empty segments before returning a named-DM key.
  - confidence: 0.92
- **[P2] Fail closed for malformed stored active names:** `src/gateway/session-utils.ts:866-870`
  - body: `activeNamedSession` is persisted session-store data, but `getActiveNamedSessionKey` passes it directly into `buildNamedDmSessionKey`. If a stored row contains `:`, active-session resolution throws instead of returning `null`; validate the stored name and fail closed before building the key.
  - confidence: 0.9
- **[P2] Prioritize dm-named before token chat-type checks:** `src/sessions/session-key-utils.ts:50`
  - body: Adding `dm-named` to the direct-token branch after `group` and `channel` still misclassifies valid named DM keys whose user-chosen name is `group` or `channel`. Since send policy can derive chat type from the key, these named DMs can receive group/channel policy rules unless `dm-named` is recognized before arbitrary name tokens.
  - confidence: 0.91
- **[P3] Correct the helper nullability docs:** `docs/concepts/session.md:79`
  - body: The docs say all three helpers accept `null` or `undefined`, but `buildNamedDmSessionKey` requires a params object and immediately reads its fields. Narrow that sentence to the parser/predicate helpers or change the implementation and types to match.
  - confidence: 0.78
