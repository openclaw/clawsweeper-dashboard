## Summary

Keep this PR open: current main still lacks the named-DM infrastructure, but the proposed branch has blocking parser, persisted-state, and chat-type classification defects, plus missing real behavior proof and an unresolved product question about landing infrastructure before routing is wired.

## What This Changes

This PR adds named-DM session key helpers, an optional `activeNamedSession` session-store field, gateway store helpers, tests, session docs, and changelog edits without routing integration.

## Best Possible Solution

If named-DM session switching is still desired, land a rebased, narrower implementation with strict raw key parsing, fail-closed persisted-state handling, explicit `dm-named` direct classification before token policy checks, accurate docs, focused regression tests, and real after-fix proof.

## Reproduction Assessment

Yes, for the PR review defects by source inspection: the parser delegates through an empty-token-filtering parser, stored active names can throw during lookup, and chat-type classification checks group/channel tokens before `dm-named`. I did not execute tests or a live flow because this pass is read-only.

## Solution Assessment

No. The infrastructure direction may be useful, but this branch is not the best merge shape until parser strictness, persisted-state behavior, policy classification, docs accuracy, proof, and product timing are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject collapsed empty segments in named-DM parsing:** `src/sessions/session-key-utils.ts:180-185`
  - body: `parseNamedDmSessionKey` relies on `parseAgentSessionKey`, which drops empty `:` segments before this code validates the named-DM shape. A malformed key like `agent:main:dm-named:123::work` can therefore be accepted as `{ peerId: "123", name: "work" }`; parse the raw scoped suffix or explicitly reject empty segments before returning a valid named-DM key.
  - confidence: 0.92
- **[P2] Fail closed for malformed stored active names:** `src/gateway/session-utils.ts:866-870`
  - body: `activeNamedSession` is persisted session-store data, but `getActiveNamedSessionKey` passes it straight into `buildNamedDmSessionKey`. If an existing or manually edited store row contains `:`, callers get an exception during active-session resolution instead of the advertised `string | null` behavior; validate the stored name and return `null` or repair it before building the key.
  - confidence: 0.9
- **[P2] Prioritize dm-named before token chat-type checks:** `src/sessions/session-key-utils.ts:50`
  - body: Adding `dm-named` to the direct-token branch after the `group` and `channel` checks still misclassifies valid named-DM keys whose user-chosen name is `group` or `channel`. Since send policy can fall back to key-derived chat type, those sessions can pick up group/channel policy rules unless `dm-named` is recognized before scanning arbitrary name tokens.
  - confidence: 0.91
- **[P3] Correct the helper nullability docs:** `docs/concepts/session.md:79`
  - body: The docs say all three helpers accept `null` or `undefined`, but `buildNamedDmSessionKey` requires a params object and immediately reads its fields. Narrow that statement to the parser/predicate helpers or change the implementation and types to match the documented contract.
  - confidence: 0.78
