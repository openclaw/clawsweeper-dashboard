## Summary

Keep this PR open. The macOS fallback/auth-preservation direction matches the linked bug reports, but the current diff still adds a rejected-config artifact path that can contain preserved gateway auth without explicit private permissions, and it still lacks the required user-facing changelog entry.

## What This Changes

The PR changes macOS config saving to block stale/auth-like Gateway fallback writes, preserve `gateway.auth`, reject suspicious mode/size clobbers with rejected artifacts and audit logs, surface save rejections, and add focused Swift regression tests.

## Best Possible Solution

Land the macOS guard after rejected config artifacts either use owner-only permissions matching the core writer or omit secret-bearing fields from the artifact, then add the required Unreleased Fixes changelog entry. The longer-term maintainable direction is still to move macOS UI writes toward conflict-aware patch flows rather than full-config replacement.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: current main catches a failed local-mode Gateway `config.set` and falls back to a direct full-file save, while the PR’s new test models a stale Gateway rejection and asserts the config file remains unchanged.

## Solution Assessment

No. The fallback guard and auth/mode preservation are the right narrow direction, but the current implementation is not the best safe fix until the new rejected payload artifact is private or redacted and the user-facing changelog requirement is met.
