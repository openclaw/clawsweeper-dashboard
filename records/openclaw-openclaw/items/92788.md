## Summary

Keep open: current main and the latest release still miss channel derivation for account-scoped DM session keys, while this PR makes the existing sendPolicy channel rule apply to that documented key shape. I found no blocking code finding; maintainers should be aware that merging intentionally makes existing channel-scoped deny rules stricter for these DMs.

## What This Changes

The branch broadens `deriveChannelFromKey` and adds a regression test using `buildAgentPeerSessionKey` for `per-account-channel-peer` direct-message sessions.

## Best Possible Solution

Land the focused helper and regression-test change with merge or release-note context that channel-scoped deny rules now apply to account-scoped DM session keys.

## Reproduction Assessment

Yes. Source inspection shows `buildAgentPeerSessionKey` emits `agent:<agentId>:<channel>:<accountId>:direct:<peerId>` while current `deriveChannelFromKey` only recognizes peer kinds at `parts[1]`, so the account-scoped path cannot derive a channel without another fallback.

## Solution Assessment

Yes. Updating the single shared derivation helper is the narrowest maintainable fix; changing individual send-policy call sites would duplicate key parsing and risk another one-sided policy path.
