## Summary

Current main still lacks all three requested native contracts: file and web-search tools accept one path/query per call, context engines manage lifecycle/compaction rather than model-addressable named blocks, and `update_plan` tracks progress without runner-enforced step gating. No newer canonical issue or viable PR owns the work, and the recent discussion explicitly keeps it open; however, this should remain a maintainer-reviewed product umbrella rather than an implementation-ready ticket because the three tracks have separate owners, persistence concerns, and API decisions.

## What This Changes

Add native batched file/search operations, AI-controlled named context block loading and unloading, and opt-in plan-step enforcement with configurable warning or blocking behavior.

## Best Possible Solution

Keep this as the canonical product-design umbrella, then split accepted work into separately owned designs for batch-capable tool schemas, context-engine-managed selective state, and runner-owned plan gating, each with explicit lifecycle, compatibility, and validation contracts.

## Reproduction Assessment

Not applicable as a strict bug reproduction: the requested APIs and enforcement guarantees are new capabilities. Current-source inspection reliably reproduces their absence, but the claimed token savings and plan-skipping percentages are workload-specific observations rather than a defined failing contract.

## Solution Assessment

No, not as one combined implementation. The narrow maintainable approach is to decide and design the batching, context-state, and plan-gating contracts separately against their current owners and the Codex runtime contract.
