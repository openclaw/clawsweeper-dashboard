## Summary

Keep open. Current main still lacks the three requested native surfaces: batched file/search tool inputs, AI-callable named context block management, and submitted-plan step enforcement. The existing update_plan tool, strict-agentic retry guard, context-engine lifecycle, /compact, and web provider tooling are adjacent groundwork, not the requested API/product behavior.

## What This Changes

The issue requests native batched tool calls, AI-callable context block save/load/unload/list tools, and opt-in plan submission/step enforcement for multi-step workflows.

## Best Possible Solution

Keep the issue open and split it into maintainer-reviewed tracks: native or gateway-level batching for file and search tools, an explicit context-block API if AI-controlled save/load/unload/list semantics are desired, and runner-owned opt-in plan state if strict submitted-plan gating is a product goal.

## Reproduction Assessment

Not applicable as a bug reproduction, but current-main verification is high confidence: direct source/docs checks show the requested tool names and batch params are absent, and adjacent mechanisms are documented as tracking, lifecycle, or compaction features.

## Solution Assessment

No current implementation solves the request yet. The maintainable path is to design the three surfaces separately and decide which belong in core APIs versus plugin/context-engine extension points.
