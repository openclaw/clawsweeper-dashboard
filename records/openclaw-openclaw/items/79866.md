## Summary

Keep open: this is a plausible Active Memory bug-fix area, but the PR does not implement the advertised `status=no-results` path for normal `NONE`/empty-summary recall and it lacks after-fix real behavior proof.

## What This Changes

The PR changes Active Memory cache eligibility, terminal `memory_search` early-exit classification, and related tests so empty results are not cached and unavailable searches surface as `unavailable`.

## Best Possible Solution

Keep the no-injection behavior unchanged, add a distinct successful no-result status for normal `NONE`/empty-summary recall, reserve `empty`/`unavailable` for genuinely empty or unavailable memory states, and update tests and docs to pin that contract.

## Reproduction Assessment

Yes, source-reproducible: current main normalizes `NONE` to no summary and then builds an Active Memory result with `status: "empty"`. Adjacent tests already assert `status=empty` for that `NONE` path, though I did not run tests in this read-only review.

## Solution Assessment

No, this PR is not yet the best way to solve the issue. The cache and terminal watcher changes may be useful, but the normal no-summary branch still returns `empty`; the narrow fix is to add and test a distinct no-result status there.
