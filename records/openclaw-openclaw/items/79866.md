## Summary

Keep open: the PR targets a valid Active Memory bug, but the branch still leaves normal `NONE`/empty-summary recall results mapped to `status: "empty"`, so the linked user-visible problem remains.

## What This Changes

The PR changes Active Memory cache eligibility, terminal `memory_search` early-exit classification, and related tests so empty results are not cached and unavailable searches surface as `unavailable`.

## Best Possible Solution

Add a distinct successful no-result status to `ActiveRecallResult` for normalized `NONE`/empty summaries, keep no-injection behavior unchanged, reserve `empty`/`unavailable` for real empty or unavailable states, and update Active Memory tests and docs to pin the contract.

## Reproduction Assessment

Yes, source-reproducible: current main normalizes `NONE` to an empty summary and then constructs an Active Memory result with `status: "empty"`. Adjacent tests already assert `status=empty` for that path.

## Solution Assessment

No, this PR is not yet the best way to solve the issue. It changes cache and terminal-search behavior, but the normal no-summary branch still returns `empty`, which is the central linked bug.
