## Summary

Keep this PR open, but do not merge it as-is: the patch stops caching `empty` results and splits `unavailable`, but PR head still returns `status: "empty"` for the normal `NONE`/no-summary recall path that the linked bug is about, and the supplied proof does not match the branch.

## What This Changes

The PR changes Active Memory caching to avoid cached empty recalls, treats unavailable terminal memory-search results as `unavailable`, and updates focused tests for those paths.

## Best Possible Solution

Land one covered Active Memory fix that emits a chosen no-match status for successful no-summary recall, reserves unavailable/timeout for real failure paths, keeps no-match cache behavior non-sticky, and updates tests plus any user-facing docs.

## Reproduction Assessment

Yes, source-reproducible: current main turns a `NONE` recall reply into an empty summary and persists `status=empty`, and the adjacent tests assert that visible status. I did not run tests because this is a read-only review.

## Solution Assessment

No. This patch handles cache stickiness and unavailable search states, but it does not change the normal no-summary recall classification away from `status=empty`, so it is not the narrow complete fix for the linked behavior yet.
