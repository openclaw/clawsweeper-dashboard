## Summary

Keep open: this PR is protected by the `maintainer` label and adds a plausible Codex node-session feature with real-environment proof, but the diff introduces a Windows resume-path bug that should block merge until fixed.

## What This Changes

The PR adds Codex plugin node-host commands and slash commands to list Codex CLI sessions on paired nodes, bind a chat to one, and route follow-up messages through `codex exec resume`.

## Best Possible Solution

Fix the Codex CLI resume spawn path to reuse the existing cross-platform Windows spawn helper, then have maintainers decide whether the dangerous node command policy and slash-command UX are acceptable for merge.

## Reproduction Assessment

Yes, at source level: a Windows node with Codex installed through npm would hit the new `spawn("codex", ...)` path, while Node's Windows contract requires shell/cmd handling for `.cmd` shims. I did not run a live Windows node in this read-only review.

## Solution Assessment

No, not as submitted: the feature direction is plausible, but the process launch should follow the existing Codex `windows-spawn` helper before the PR is mergeable. Maintainers still need to approve the node-command policy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the Windows spawn helper for Codex CLI resume:** `extensions/codex/src/node-cli-sessions.ts:246-250`
  - body: The new node-host resume path launches `spawn("codex", ...)` directly. On Windows, npm-installed CLIs are normally exposed through `.cmd` shims, and Node requires shell/cmd handling for those; the existing Codex app-server spawn path already uses `resolveWindowsSpawnProgram`/`materializeWindowsSpawnProgram`. Since this command can be explicitly allowlisted on a Windows node, `/codex resume ... --host <node> --bind here` will fail before Codex starts there.
  - confidence: 0.86
