## Summary

Keep open. Current main still has a source-level path where the reported unclassified provider text can miss the existing classifier, fall through as raw `errorMessage`, and be pushed as a channel-visible error reply; https://github.com/openclaw/openclaw/pull/41803 targets this area but is still open and not merged.

## What This Changes

The issue asks OpenClaw to stop sending raw provider `errorMessage` text to user-facing channels when an assistant message ends with `stopReason=error`.

## Best Possible Solution

Land a focused current-main formatter and payload regression that maps truly unclassified provider errors to safe user-facing copy while preserving redacted operator diagnostics and existing classified recovery messages.

## Reproduction Assessment

Yes, source-level. The reported wording misses the current generic-provider classifier, the formatter can still return raw unclassified text, and the payload builder sends that text as an `isError` reply.

## Solution Assessment

Yes for the narrowed direction, but no complete fix is merged. The maintainable path is to review, refresh, or replace https://github.com/openclaw/openclaw/pull/41803 with current-main tests for the reported OpenAI help-center/request-ID wording.
