## Summary

Keep open. Current main still has a user-facing path where unclassified assistant `errorMessage` text can be formatted and delivered as an error reply, and the focused hardening work is still an open PR rather than merged main behavior.

## What This Changes

The issue asks OpenClaw to stop posting raw upstream assistant `errorMessage` text to Slack/Telegram-style delivery surfaces when `stopReason` is `error`.

## Best Possible Solution

Land #41803 or an equivalent narrow fix that makes unclassified assistant errors resolve to a generic user-facing message or suppression, while preserving existing classified friendly messages and keeping raw provider diagnostics only in redacted/admin observability paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to create an assistant message with `stopReason: "error"` and an unclassified `errorMessage`; current main returns that raw string from `formatAssistantErrorText` and then pushes it as an `isError` delivery payload.

## Solution Assessment

Yes. The narrow maintainable fix is to remove the raw catch-all from the user-facing formatter/delivery path or replace it with a generic fallback, then update the lifecycle regression test that currently expects redacted raw text.
