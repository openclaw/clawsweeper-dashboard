## Summary

Keep open. Current main still has a source-reproducible path where unclassified assistant `errorMessage` text can be formatted unchanged and sent as a user-facing error reply, while the focused hardening PR remains open.

## What This Changes

Prevent chat delivery from posting raw upstream `errorMessage` text when an assistant message ends with `stopReason: "error"`.

## Best Possible Solution

Land #41803 or an equivalent narrow formatter/payload hardening that uses generic or suppressed user-facing copy for unrecognized assistant errors while keeping classified errors and redacted operator observability intact.

## Reproduction Assessment

Yes. By source inspection, `formatAssistantErrorText` can return an unclassified raw string, and `buildEmbeddedRunPayloads` sends that result as an `isError` reply when the assistant ended with `stopReason: "error"`.

## Solution Assessment

Yes. The narrow maintainable fix is to replace the raw catch-all with generic or suppressed user-facing copy and update the formatter/lifecycle regression tests; broader channel-level error policy belongs in #54825 and the multi-text split remains in #74674.
