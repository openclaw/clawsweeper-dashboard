## Summary

Keep open. Current main still has a source-reproducible user-facing path where unclassified assistant error text is returned by the formatter and emitted as an `isError` reply, while the focused hardening PR remains open and unmerged.

## What This Changes

Request suppressing or replacing raw unclassified assistant `errorMessage` text before Slack/Telegram-style delivery when `stopReason` is `error`.

## Best Possible Solution

Land #41803 or an equivalent narrow fix that makes unrecognized assistant errors resolve to generic user-facing copy or suppression while keeping raw diagnostics in redacted operator observability paths.

## Reproduction Assessment

Yes. By source inspection, an assistant message with `stopReason: "error"` and an unclassified `errorMessage` can flow through `formatAssistantErrorText` unchanged and then be pushed as an `isError` delivery payload.

## Solution Assessment

Yes. The narrow maintainable solution is to remove or replace the raw catch-all in the user-facing formatter/delivery path; broader configurable channel policy belongs in #54825 and the multi-text selection policy remains separate in #74674.
