## Summary

Keep open. Current main still has a source-reproducible path where an unclassified assistant `errorMessage` is returned by the formatter and delivered as a channel-visible error reply, and the focused open fix PR is not merged yet.

## What This Changes

Prevent `stopReason=error` assistant `errorMessage` text from being delivered verbatim to Slack/Telegram-style channels when the provider error is unclassified.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/41803 with narrow fallback hardening that genericizes only unclassified user-facing assistant errors while keeping recognized diagnostics and redacted operator observations intact.

## Reproduction Assessment

Yes. By source inspection, an assistant message with `stopReason: "error"` and an unclassified `errorMessage` can flow through `formatAssistantErrorText` unchanged and then be pushed as a channel-visible `isError` reply; no live channel run was performed.

## Solution Assessment

Yes. The best fix is the scoped fallback-chain hardening described here: remove or replace the raw catch-all on the user-facing path while preserving recognized error classifications and keeping the broader policy work in separate issues.
