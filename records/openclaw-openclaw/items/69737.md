## Summary

Keep open. Current main still has a user-facing path where unclassified assistant `errorMessage` text can pass through `formatAssistantErrorText` and be emitted as an `isError` reply, and the focused hardening work remains an open PR rather than merged behavior.

## What This Changes

The issue requests that OpenClaw stop posting raw upstream assistant `errorMessage` text to Slack/Telegram-style delivery surfaces when `stopReason` is `error`.

## Best Possible Solution

Land #41803 or an equivalent narrow fix that makes truly unrecognized assistant errors resolve to generic user-facing copy or suppression, while preserving existing classified friendly messages and keeping raw diagnostics only in redacted operator observability paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to create an assistant message with `stopReason: "error"` and an unclassified `errorMessage`; current main can return that raw string from `formatAssistantErrorText` and then push it as an `isError` delivery payload.

## Solution Assessment

Yes, the narrow maintainable direction is to remove the raw catch-all from the user-facing formatter path or replace it with a generic fallback. The implementation should preserve existing actionable diagnostics for classified HTTP/API, billing, auth, schema, transport, and intentionally preserved request-validation cases.
