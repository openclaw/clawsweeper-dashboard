## Summary

Keep open. Current main still lets unclassified assistant error text fall through to the user-facing error payload path, and the focused hardening work is still an open PR rather than merged behavior.

## What This Changes

The issue requests that channel delivery stop exposing raw provider `errorMessage` text for assistant messages ending with `stopReason=error`.

## Best Possible Solution

Land #41803 or an equivalent narrow fix that maps truly unrecognized assistant errors to generic user-facing copy or suppression while keeping raw diagnostics in redacted observability paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an assistant message with `stopReason: "error"` and an unclassified `errorMessage`: current main can return that raw string from `formatAssistantErrorText` and then emit it as an `isError` delivery payload.

## Solution Assessment

Yes. The narrow maintainable direction is to remove or replace the final raw catch-all on the user-facing path, while preserving the existing classified messages and focused diagnostics that tests already cover.
