## Summary

Keep open. Current main and the latest release still have a source-reproducible path where an unclassified assistant `errorMessage` can be returned by the formatter and pushed as a user-facing `isError` reply, while the focused fix PR is still open and blocked by proof/review concerns.

## What This Changes

The issue asks OpenClaw to suppress or genericize unclassified assistant `errorMessage` text instead of posting raw upstream provider diagnostics to user-facing channel delivery.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/41803 with narrow fallback hardening that genericizes only unclassified user-facing assistant errors, preserves actionable diagnostics and redacted operator logs, and leaves the broader policy work to https://github.com/openclaw/openclaw/issues/54825 and https://github.com/openclaw/openclaw/issues/74674.

## Reproduction Assessment

Yes. By source inspection, an assistant message with `stopReason: "error"` and an unclassified `errorMessage` can flow through `formatAssistantErrorText` unchanged and then be pushed as a channel-visible `isError` reply; no live channel run was performed in this read-only review.

## Solution Assessment

Yes, the scoped fix direction is the right maintainable path: remove or replace the raw catch-all on the user-facing path while preserving recognized diagnostics. The existing open PR is useful but should be refreshed or replaced to address proof and log-safety concerns.
