## Summary

Keep open. Current main still has a source-reproducible path where unclassified assistant error text is returned by the formatter and emitted as a user-facing error payload, while the narrow suppression PR remains open and unmerged.

## What This Changes

Prevent user-facing delivery channels from posting raw provider `errorMessage` text when the last assistant message ends with `stopReason: "error"`.

## Best Possible Solution

Land #41803 after aligning lifecycle and formatter regression coverage, or merge an equivalent narrow fix that replaces unclassified assistant errors with generic user-facing copy while preserving redacted operator diagnostics.

## Reproduction Assessment

Yes. Source inspection on current main shows an assistant message with `stopReason: "error"` and an unclassified `errorMessage` can flow through `formatAssistantErrorText` unchanged and then be pushed as an `isError` delivery payload.

## Solution Assessment

Yes. The narrow maintainable fix is to remove or replace the raw catch-all at the formatter/delivery boundary while leaving recognized friendly classifications and redacted logs intact; broader channel-specific policy belongs in #54825.
