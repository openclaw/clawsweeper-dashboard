## Summary

Keep open. Current main and the latest shipped tag still have a source-reproducible path where unclassified assistant `errorMessage` text can become a channel-visible `isError` reply, while the focused hardening PR remains open and blocked on review/proof concerns.

## What This Changes

Suppress or replace raw unclassified provider `errorMessage` text before Slack/Telegram-style delivery when an assistant message ends with `stopReason: "error"`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/41803 with a narrow fallback hardening that genericizes only unclassified user-facing assistant errors, keeps redacted operator diagnostics, and preserves the separately tracked policy work.

## Reproduction Assessment

Yes. By source inspection, an assistant message with `stopReason: "error"` and an unclassified `errorMessage` can flow through `formatAssistantErrorText` unchanged and then be pushed as an `isError` delivery payload.

## Solution Assessment

Yes, the scoped direction is the narrow maintainable fix: remove or replace the raw catch-all on the user-facing path while preserving recognized/actionable diagnostics. Broader configurable channel policy belongs in https://github.com/openclaw/openclaw/issues/54825 and the multi-text selection policy remains in https://github.com/openclaw/openclaw/issues/74674.
