## Summary

Keep this issue open. Current main has SecretRef runtime, env-template, redaction, RPC, CLI, and docs foundations, but the requested source-aware Control UI/config UX is still missing: structured SecretRefs render as read-only generic placeholders and SecretInput unions still use the plain string editor path.

## What This Changes

Request first-class `.env` and environment-backed secret configuration UX with source metadata, env-reference preservation, a secrets helper, and migration guidance in CLI/Control UI.

## Best Possible Solution

Define a redacted SecretRef source/status metadata contract first, then implement a source-aware SecretInput renderer with aligned CLI/docs/tests and no resolved secret values sent to the browser.

## Reproduction Assessment

Yes. The current Control UI form rendering path can be inspected through the existing env SecretRef browser test for `channels.discord.token`, which asserts an empty read-only input with a generic Structured SecretRef placeholder.

## Solution Assessment

Unclear until maintainers set the disclosure boundary. The narrow maintainable path is a redacted metadata contract plus UI/docs/tests, not reusing resolved-secret RPC payloads or adding placeholder-only labels.
