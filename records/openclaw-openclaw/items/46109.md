## Summary

Keep open: current main and v2026.6.6 still render structured SecretRefs in Control UI as empty read-only generic placeholders, while the requested source-aware env/config/unset secret UX remains a product/security-sensitive feature.

## What This Changes

Requests first-class env-backed SecretRef UX in CLI and Control UI so secret fields show masked source and resolution metadata instead of appearing blank or requiring plaintext entry.

## Best Possible Solution

Extend the existing SecretRef contracts with a source-aware SecretInput renderer and metadata flow: masked display, safe source labels, resolved/unresolved status, explicit config/env/unset editing, and aligned docs/tests.

## Reproduction Assessment

Yes. Source inspection and current browser tests show a SecretRef object such as `channels.discord.token` renders as an empty read-only generic placeholder rather than a source-aware env-backed field.

## Solution Assessment

Yes at the product direction level, but it should reuse the current SecretRef registry, redaction, and secrets CLI contracts instead of adding another config/env surface; the disclosure rules need maintainer security review first.
