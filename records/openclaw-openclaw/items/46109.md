## Summary

Keep this open. Current main has the SecretRef runtime, CLI, redaction, and docs foundation, but the requested source-aware Control UI/config UX is still missing: structured env SecretRefs render as read-only empty generic placeholders, and SecretInput unions still normalize to the string editor path.

## What This Changes

The issue requests first-class environment-backed secret configuration UX in the CLI and Control UI, including source-aware masked fields, env/config/unset state, and migration guidance.

## Best Possible Solution

Add a source-aware SecretInput renderer and metadata flow on top of existing SecretRef contracts, with masked display, safe source labels, approved env identity display, resolved/unresolved state, explicit config/env/unset editing, and aligned docs/tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction because this is a feature request; source and test inspection give high-confidence evidence of the current gap in the Control UI form path.

## Solution Assessment

Is this the best way to solve the issue? Yes as a product direction, but not as an automated bug fix; the maintainable path is to extend the existing SecretRef registry/config UI schema analysis and secrets workflow after maintainers decide what metadata is safe to expose.
