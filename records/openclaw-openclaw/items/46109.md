## Summary

Keep this open. Current main has a solid SecretRef runtime, CLI, redaction/restore, and docs foundation, but the requested source-aware Control UI/config UX is still missing: structured SecretRefs render as generic read-only empty inputs, and SecretInput unions are still normalized to the string editor path. The remaining work is security-sensitive product design, not a stale cleanup close.

## What This Changes

Request first-class env-backed secret configuration UX with masked source-aware Control UI/CLI fields, env-reference editing, and migration guidance.

## Best Possible Solution

Add a source-aware SecretInput renderer and metadata flow on top of existing SecretRef contracts, with masked display, approved source labels, env identity when safe, resolved/unresolved state, explicit config/env/unset editing, and aligned docs/tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report with an existing expected behavior. Source and tests do show the current gap clearly: env SecretRefs render as read-only empty generic placeholders rather than source-aware fields.

## Solution Assessment

Yes as a product direction, but not as an automated bug fix. The maintainable path is to extend the existing SecretRef registry/config UI schema analysis and secrets workflow after maintainers decide what metadata is safe to expose.
