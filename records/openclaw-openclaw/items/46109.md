## Summary

Keep this open. Current main has a strong SecretRef runtime, CLI, redaction/restore, and docs foundation, but the requested source-aware Control UI/config UX is still missing: env-backed structured SecretRefs still render as generic read-only empty inputs rather than masked config/env/unset state.

## What This Changes

The issue requests first-class environment-backed secret configuration UX in CLI and Control UI, including source-aware masked display and env/config/unset editing.

## Best Possible Solution

Add a source-aware SecretInput renderer and metadata flow on top of the existing SecretRef contracts, with masked display, safe source labels, approved env identity, resolved/unresolved state, explicit config/env/unset editing, and aligned docs/tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing contract. Source and test inspection show current main still renders env SecretRefs as generic read-only empty inputs rather than source-aware fields.

## Solution Assessment

Yes as a product direction, but not as an automated bug fix. The maintainable path is to extend the existing SecretRef registry/config UI schema analysis rather than adding field-specific badges or ad hoc env handling.
