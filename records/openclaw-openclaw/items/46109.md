## Summary

Keep this open. Current main and the latest release still have the SecretRef runtime, CLI, redaction, and docs foundation, but the requested source-aware Control UI/config UX is not implemented: structured env SecretRefs still render as read-only empty generic placeholders, and SecretInput unions still collapse to the string editor path.

## What This Changes

The issue requests first-class env-backed secret UX in CLI and Control UI so secret fields show masked source-aware config/env/unset state instead of ambiguous blank fields.

## Best Possible Solution

Add a source-aware SecretInput renderer and metadata flow on top of existing SecretRef contracts, with masked display, safe source labels, env identity only where approved, resolved/unresolved state, explicit config/env/unset editing, and aligned docs/tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, because this is a feature request; source and test inspection do give high-confidence evidence of the current gap: env SecretRefs render as read-only empty generic placeholders rather than source-aware fields.

## Solution Assessment

Is this the best way to solve the issue? Yes as a product direction, but not as an automated bug fix; the maintainable path is to extend the existing SecretRef registry/config UI schema analysis and secrets workflow after maintainers decide what metadata is safe to expose.
