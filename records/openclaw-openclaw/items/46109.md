## Summary

Keep this open. Current main has strong SecretRef runtime, CLI, redaction/restore, and docs foundations, but the requested source-aware Control UI/config UX is still missing: structured SecretRefs still render as empty read-only inputs with generic placeholders, and SecretInput unions still collapse to the plain string editor path. Because the remaining work controls browser-visible secret metadata, it needs maintainer product/security judgment before implementation.

## What This Changes

Request first-class .env and SecretRef-backed secret UX that shows config/env/unset source state, preserves env references, and guides migration without exposing secret values.

## Best Possible Solution

Define a redacted SecretRef metadata contract first, then implement a source-aware SecretInput renderer and aligned CLI/docs/tests that show masked source labels, safe env identity when allowed, resolved/unresolved state, and explicit config/env/unset editing without sending resolved secrets to the browser.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the current Control UI form rendering path for an env SecretRef such as channels.discord.token; current code and tests show a read-only empty input with a generic Structured SecretRef placeholder instead of source-aware state.

## Solution Assessment

Unclear until the disclosure boundary is set. The maintainable solution is a bounded metadata contract plus UI/docs/tests, not a placeholder-only change that guesses which secret-source details are safe to expose.
