## Summary

Keep this issue open. Current main has SecretRef runtime, env-template parsing, redaction/restore, CLI, RPC, and docs foundations, but the requested source-aware Control UI/config UX is still missing: structured SecretRefs render as read-only generic placeholders and SecretInput unions still collapse to the plain string editor path.

## What This Changes

Request first-class .env and environment-backed secret UX that shows secret source/status in CLI and Control UI without exposing secret values.

## Best Possible Solution

Define a redacted SecretRef source/status metadata contract, then implement a source-aware SecretInput renderer with aligned CLI/docs/tests and no resolved secret values sent to the browser.

## Reproduction Assessment

Yes. The current form rendering path and existing browser test reproduce the observable gap for channels.discord.token by asserting a read-only empty input with a generic Structured SecretRef placeholder instead of source-aware env/config/unset metadata.

## Solution Assessment

Unclear until maintainers set the disclosure boundary. The narrow maintainable solution is a redacted metadata flow plus SecretInput UI/docs/tests, not reusing resolved-secret RPC payloads or adding placeholder-only labels.
