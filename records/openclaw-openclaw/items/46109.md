## Summary

Keep this issue open. Current main has strong SecretRef runtime, CLI, redaction, and round-trip foundations, including related hardening after #55034, but the requested source-aware Control UI/config UX is still missing and needs a maintainer decision about safe browser-visible secret metadata.

## What This Changes

Request first-class .env/environment-backed secret config UX so CLI and Control UI can show config/env/unset source, preserve env SecretRefs, and avoid blank-looking env-backed secret fields.

## Best Possible Solution

Define a redacted SecretRef source/status metadata contract, then implement a source-aware SecretInput renderer and Secrets helper with aligned CLI/docs/tests and no resolved secret values in browser payloads.

## Reproduction Assessment

Yes. Source inspection on current main shows an env SecretRef at channels.discord.token renders as a read-only empty input with a generic Structured SecretRef placeholder, not a source-aware env-backed secret field.

## Solution Assessment

Unclear until maintainers define the disclosure boundary. The narrow maintainable path is a redacted metadata contract plus Control UI renderer/docs/tests, not reusing resolved-secret RPC payloads or adding labels only in the frontend.
