## Summary

Keep this issue open. Current main has strong SecretRef runtime, redaction, CLI, and docs foundations, including the narrower round-trip fix related by #55034, but the requested source-aware Control UI/config UX is still missing: structured SecretRefs render as generic read-only placeholders and SecretInput schemas still use the plain string editor path.

## What This Changes

The issue requests first-class environment-backed secret configuration UX in the CLI and Control UI, including source badges, env-reference preservation, masked visibility, and a Secrets/Environment helper.

## Best Possible Solution

Define a redacted SecretRef source/status metadata contract, then implement a source-aware SecretInput renderer and Secrets helper with aligned CLI/docs/tests and no resolved secret values in browser payloads.

## Reproduction Assessment

Yes. The current-main source and browser fixture show the UX gap: an env SecretRef at channels.discord.token renders as a read-only empty input with a generic Structured SecretRef placeholder, not a source-aware env-backed secret field.

## Solution Assessment

Unclear until maintainers set the disclosure boundary. The narrow maintainable path is a redacted metadata contract plus Control UI renderer/docs/tests, not reusing resolved-secret RPC payloads or adding placeholder-only labels.

## Security Review

Status: needs_attention

Summary: This feature is security-sensitive because it decides what secret-source metadata becomes visible in the browser UI.

Concerns:

- **[medium] Define safe SecretRef metadata disclosure:**
  - body: Showing env var names, provider aliases, config paths, or resolution status can reveal deployment structure, so maintainers should explicitly define the browser-visible metadata boundary before implementation.
  - confidence: 0.88
- **[medium] Avoid resolved-secret payloads for UI metadata:** `src/gateway/server-methods/secrets.ts:93`
  - body: The existing secrets.resolve handler returns assignment values from the runtime resolver, so source/status UI metadata should use a redacted metadata flow instead of a path that can carry secret material.
  - confidence: 0.84
