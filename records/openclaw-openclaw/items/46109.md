## Summary

Keep open: current main and v2026.6.10 still have SecretRef runtime, CLI, and redaction safety, but the Control UI still renders env-backed SecretRefs as generic read-only structured placeholders instead of source-aware config/env/unset UX.

## What This Changes

Request first-class env-backed secret configuration UX in CLI and Control UI, including masked source-aware display, env-reference preservation, and clearer migration guidance.

## Best Possible Solution

Keep this as the canonical product/security design item and implement a source-aware SecretInput renderer plus metadata flow that stays aligned with the existing SecretRef CLI, docs, and gateway contracts.

## Reproduction Assessment

Yes. Source inspection and the current browser test show an env SecretRef such as channels.discord.token renders as a read-only empty input with a generic structured placeholder, not a source-aware env-backed field.

## Solution Assessment

Unclear. A source-aware SecretInput renderer and metadata flow look like the right maintainable direction, but maintainers need to decide safe disclosure and editing semantics before implementation.
