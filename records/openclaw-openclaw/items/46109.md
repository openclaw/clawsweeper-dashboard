## Summary

Keep this open. Current main has SecretRef runtime, CLI, docs, and Control UI write-path hardening, but the requested source-aware config UX is still missing: structured SecretRefs render as a generic read-only empty input, and SecretInput schemas are still normalized to the string editor path.

## What This Changes

Request first-class .env and environment-backed secret UX in CLI and Control UI, including masked source-aware display, env-reference preservation, and Secrets/Environment helpers.

## Best Possible Solution

Extend the existing SecretRef and secret-target model into a source-aware SecretInput renderer with masked display, safe source labels, approved env identity, resolved/unresolved status, explicit config/env/unset editing, and aligned docs/tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing contract. Source and test inspection show current main still renders env SecretRefs as generic read-only empty inputs rather than source-aware secret fields.

## Solution Assessment

Yes as a product direction, but not as an automated bug fix. The maintainable path is to build on the existing SecretRef and secret-target contracts rather than adding field-specific badges.
