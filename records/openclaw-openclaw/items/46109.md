## Summary

Keep this issue open. Current main has the SecretRef runtime, CLI, redaction, and docs foundation, but the requested source-aware Control UI/config UX is still missing: env-backed SecretRefs render as generic read-only empty inputs rather than masked fields with config/env/unset source metadata.

## What This Changes

Request first-class `.env` and environment-backed secret configuration UX that shows secret source metadata, preserves env references, and improves Control UI/CLI guidance.

## Best Possible Solution

Define a redacted SecretRef source/status metadata contract, then implement a source-aware SecretInput renderer with aligned CLI/docs/tests and no resolved secret values sent to the browser.

## Reproduction Assessment

Yes. The current form path can be reproduced by rendering a config containing `channels.discord.token` as an env SecretRef; the existing browser test asserts the field is read-only, empty, and uses a generic Structured SecretRef placeholder.

## Solution Assessment

Unclear until maintainers set the disclosure boundary. The narrow maintainable path is a redacted metadata flow plus UI/docs/tests, not reusing resolved-secret RPC payloads or adding placeholder-only labels.
