## Summary

Keep open: current main and v2026.6.10 still render env-backed SecretRefs as generic read-only structured placeholders, while related SecretRef hardening work fixed round-trip safety rather than the requested source-aware config/env/unset UX.

## What This Changes

The issue requests first-class Control UI and CLI UX for environment-backed secret configuration, including source-aware masked display, env-reference editing, and migration guidance.

## Best Possible Solution

Add a maintainer-approved source-aware SecretInput renderer and metadata flow that provides masked display, safe source labels, env identity when appropriate, resolved/unresolved status, config/env/unset editing, and aligned docs/tests across the existing SecretRef contracts.

## Reproduction Assessment

Yes. Source inspection and the current browser test show an env SecretRef such as `channels.discord.token` renders as a read-only empty input with a generic structured placeholder, not a source-aware env-backed field.

## Solution Assessment

Unclear. A source-aware SecretInput renderer and metadata flow look like the maintainable direction, but maintainers need to decide safe disclosure and editing semantics before implementation.
