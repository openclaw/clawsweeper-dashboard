## Summary

Keep open. Current main still resolves the Discord route before applying cross-bot target signals, checks explicit mentions and reply metadata only against the running handler's single bot user ID, and has no shared suppression result that would prevent the default owner from responding when another configured bot is targeted. The issue is valid, but related open PRs #47393 and #67460 overlap the same Discord owner and mention-suppression behavior, so this should remain a maintainer-coordinated fix rather than conservative cleanup.

## What This Changes

The issue requests deterministic Discord multi-bot routing precedence: explicit bot mentions first, reply-target bot second, and channel default owner only when no targeted signal applies.

## Best Possible Solution

Implement Discord-owned target selection before shared default routing: build a configured Discord bot user ID to OpenClaw account/agent map, select explicitly mentioned bot accounts first, select the referenced-message bot author second, and fall back to the per-channel default owner only when no targeted signal exists. Add focused Discord preflight and shared mention-gating tests proving targeted cases suppress the default owner, with only a small generic core seam if the plugin needs to express suppression downstream.

## Reproduction Assessment

Yes. GTZhou provided concrete live guild-channel reproduction steps, and current-main inspection shows the same failure path: routing is selected before target signals and mention/reply checks compare only against the running botUserId.

## Solution Assessment

Yes. The requested precedence is the narrow maintainable fix, but it should be implemented inside the Discord plugin boundary with a generic suppression seam only if needed; hardcoding Discord bot identities into shared routing would be the wrong layer.
