## Label Justifications

- `P1`: The issue affects live multi-bot Discord routing and can cause missing, duplicate, or wrong-agent responses in channel workflows.
- `impact:message-loss`: Targeted Discord messages can be consumed by the default owner or fail to reach the intended bot response path.
- `impact:session-state`: Routing a targeted Discord message to the default owner can attach the turn to the wrong agent session and conversation state.

## Summary

Keep open: current main and v2026.6.8 still resolve the Discord route before cross-bot mention or reply-target selection, compare mention/reply metadata only to the running botUserId, and lack the requested default-owner suppression state.

## What This Changes

The issue requests deterministic Discord multi-bot routing where explicit bot mentions win, reply-target bot authors win next, and the channel default owner responds only when no targeted signal exists.

## Best Possible Solution

Confirm the Discord active-target contract, then implement Discord-owned target selection before default routing with a configured bot-user-id to account/agent map, explicit mention first, reply author second, and focused suppression tests.

## Reproduction Assessment

Yes, at source level: current main resolves Discord routing before mention/reply target selection and compares mention/reply metadata only to the current handler botUserId. I did not run a live Discord multi-bot replay in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the product contract; the requested precedence is plausible, but the maintainable shape should live in the Discord plugin target-selection path rather than generic mention suppression alone.
