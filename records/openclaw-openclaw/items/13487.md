## Label Justifications

- `P1`: Current multi-bot Discord routing can suppress the intended responder, select the default agent instead, or allow duplicate replies in an active channel workflow.
- `impact:message-loss`: Mentioned or replied-to bot messages can fail to produce the intended visible response or can be handled by the wrong responder.
- `impact:session-state`: Falling through to the default owner can associate the turn and subsequent context with the wrong agent session.

## Summary

Current main still cannot enforce cross-bot mention-over-reply-over-default routing: Discord chooses the effective route before inspecting target metadata, and reply recognition is limited to the bot running that handler. The open suppression PR is related but does not provide Discord identity mapping or reply-target route selection, so this issue remains necessary and needs a coordinated product decision rather than another overlapping automated fix.

## What This Changes

Requests deterministic Discord multi-bot routing where explicit bot mentions win over reply targets, reply targets win over the channel default owner, and non-target owners are suppressed.

## Best Possible Solution

Implement a Discord-owned target-selection stage that resolves configured bot user IDs to account or agent routes, selects explicit bot mentions first, otherwise selects the referenced bot author, and only then falls back to existing binding and default-owner routing; cover suppression, multi-mention policy, bound threads, and unmatched external mentions with focused tests.

## Reproduction Assessment

Yes, at source level with high confidence: current main selects the route before Discord target metadata and only recognizes mentions or replies for the running bot ID. A full live multi-bot Discord setup was not executed in this read-only review.

## Solution Assessment

Yes in direction, but the best maintainable solution is a Discord-owned target-selection stage before normal route resolution, not a generic shared suppression flag alone. Maintainers still need to define the intentional behavior for multiple explicit bot mentions.
