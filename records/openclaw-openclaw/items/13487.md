## Label Justifications

- `P1`: The issue affects real multi-bot Discord channel workflows where messages can route to the wrong responder, produce double responses, or fail to produce the expected visible reply.
- `impact:message-loss`: Discord messages can be dropped, misrouted, or duplicated when the default owner is not suppressed for explicit mention or reply-target cases.
- `impact:session-state`: Routing a targeted Discord message to the default owner can associate the turn with the wrong agent/session context.

## Summary

Keep open: current main still resolves the Discord route before it derives explicit bot-mention or reply-target state, and the remaining related PRs are unmerged or partial rather than a safe canonical fix.

## What This Changes

The issue asks Discord inbound routing to target explicit bot mentions first, reply-targeted bots second, and the per-channel default owner only when neither targeted signal exists.

## Best Possible Solution

Implement the precedence in the Discord plugin’s preflight/routing path by mapping configured Discord bot user IDs to OpenClaw accounts or agents, selecting explicit mentions before reply targets, and falling back to the default owner only when no targeted signal exists.

## Reproduction Assessment

Yes at source level: current main resolves the Discord route before targeted mention/reply metadata and only compares those metadata fields to the running botUserId. I did not run a live Discord repro in this read-only review, but the code path and user reproduction comment both support the failure mode.

## Solution Assessment

Yes on direction, but not as an unsupervised fix yet: the maintainable solution is a Discord-owned target-selection pass before default routing, with tests for mention, reply-target, and untagged default-owner cases. Maintainers still need to settle the exact precedence/compatibility behavior and how to handle the overlapping partial PR.
