## Label Justifications

- `P1`: The missing precedence breaks real multi-bot Discord workflows by causing no response, duplicate responses, or the wrong responder.
- `impact:message-loss`: Targeted Discord messages can be suppressed, unanswered, duplicated, or delivered to the wrong bot.
- `impact:session-state`: Default fallback can associate a targeted conversation with the wrong agent and session.

## Summary

Current main does not implement the requested cross-bot precedence: Discord resolves the running account’s route before evaluating native mention and reply metadata, and the only open related PR covers generic mention suppression rather than replied-to-bot selection. The issue should remain open for an explicit maintainer product decision and a Discord-owned implementation.

## What This Changes

Route Discord multi-bot messages to explicitly mentioned bots first, the replied-to bot second, and the channel default owner only when neither targeted signal exists.

## Best Possible Solution

Implement a Discord-owned target selector that maps configured bot user IDs to Discord accounts, chooses explicit native mentions before referenced-message authors, and only then resolves that account’s agent route; use the existing channel owner/default route solely when no targeted identity is present.

## Reproduction Assessment

Yes, at source level with high confidence: current main cannot select among configured Discord bot identities from native mention or reply-author metadata before route resolution. The discussion also supplies a live scenario where plain mention works but reply-shaped messages do not produce a visible reply.

## Solution Assessment

Yes for the requested precedence, but not via the partial generic suppression approach alone. The narrow maintainable solution is Discord-specific account target selection before shared routing, with focused mention, reply, fallback, and suppression tests.
