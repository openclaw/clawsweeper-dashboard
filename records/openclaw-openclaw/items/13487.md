## Label Justifications

- `P1`: The issue affects real multi-bot Discord channel routing and can produce missing or duplicate responses in active channel workflows.
- `impact:message-loss`: The reported behavior can suppress the intended bot response or allow the default owner to consume targeted traffic.
- `impact:session-state`: Routing a Discord message to the wrong bot/agent can attach the turn to the wrong agent session and conversation state.

## Summary

Keep open: current main and the latest release still route Discord preflight through the existing binding/default path before any cross-bot target selection, and reply/mention detection is scoped only to the running bot account. Related PRs were closed unmerged, while the open related issues cover adjacent or broader slices rather than replacing this exact precedence request.

## What This Changes

Request strict Discord multi-bot routing precedence so explicit bot mentions win first, reply-target bot authors win second, and the per-channel default owner is used only when no target signal exists.

## Best Possible Solution

Confirm the product semantics, then implement Discord target selection before default routing: map configured bot user IDs to OpenClaw accounts/agents, choose explicit mentions first, choose referenced bot authors second, and use the default owner only when no target signal exists.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main resolves the route before mention/reply target selection and only compares Discord mention/reply metadata with the current handler's botUserId. I did not run a live Discord multi-bot scenario in this read-only review.

## Solution Assessment

Unclear until maintainer product direction is confirmed; the requested precedence is a plausible end state, but the durable fix needs an owner decision on whether the target map and suppression state live in Discord preflight, shared routing, or a new channel contract.
