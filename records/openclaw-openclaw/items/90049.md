## Summary

Keep open: current main has generic heartbeat failure copy and configured model fallback support, but it does not yet distinguish model/provider initialization failures from assistant-turn failures in the heartbeat path. The diagnostic gap is actionable; the heartbeat-specific fallback/retry part needs maintainer policy judgment because documented model fallback already exists.

## What This Changes

The issue requests clearer heartbeat diagnostics and optional fallback handling when a heartbeat model/provider fails before assistant code runs.

## Best Possible Solution

Preserve the safe transcript sentinel, add heartbeat/provider failure classification from existing error metadata, and use the existing configured model fallback chain unless maintainers explicitly choose new heartbeat-only fallback semantics.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does confirm that heartbeat turns use the normal reply runner and the canonical stream-error sentinel can appear for failures before model content.

## Solution Assessment

Partly: distinguishing provider/model initialization failures from assistant failures is the right maintainable direction. The suggested heartbeat-specific fallback retry needs maintainer judgment because OpenClaw already has documented configured model fallback semantics.
