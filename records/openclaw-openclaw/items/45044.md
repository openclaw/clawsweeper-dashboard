## Summary

Keep this PR open. The data-loss bug still exists on current main and the PR's predicate change is the right direction, but the latest diff still has a transcript privacy regression plus portable-test, wrapper, and changelog blockers before it should merge.

## What This Changes

The PR narrows embedded session initialization resets to header-only transcripts, archives before clearing that reset case, adds unit coverage for restart-mid-turn transcripts, and adds a shell verification wrapper.

## Best Possible Solution

Amend this PR so the `hasAnyMessage` guard remains, the header-only archive/reset path recreates transcripts at `0o600` with coverage, support files follow repo wrappers and portable paths, and the active changelog records the fix before targeted and changed-gate validation.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: an existing transcript with a session header plus a user message and no assistant reply satisfies the `!hasAssistant` reset branch after the gateway reopens the session manager.

## Solution Assessment

No as submitted. The predicate change is the narrow maintainable fix for the data-loss bug, but the added archive/reset sequence must preserve transcript privacy and the test/script/changelog surface needs cleanup first.

  - body: After `archiveFileOnDisk` renames the existing transcript, this write recreates the original path with the process default mode instead of the transcript's `0o600` mode. ## Security Review

Status: needs_attention

Summary: The diff introduces a transcript privacy regression in the header-only archive/reset path.

Concerns:

- **[medium] Transcript reset recreates file without private mode:** `src/agents/pi-embedded-runner/session-manager-init.ts:56`
  - body: The PR archives the existing transcript with `renameSync` and then recreates the original path using the string-encoding `writeFile` overload. Since normal transcript creation uses `0o600`, later transcript content could be appended to a less-private file after a header-only reset.
