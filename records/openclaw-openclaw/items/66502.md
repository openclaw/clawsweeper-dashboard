## Summary

Keep this PR open: it is a focused, still-useful daemon service ProgramArguments hardening patch, and current main/latest release still do not cover the exact non-dist pnpm wrapper path case. It is not merge-ready because the branch conflicts with newer service-entrypoint code and has only unit-test validation, not real pnpm global upgrade or service-restart proof.

## What This Changes

The PR extends daemon service CLI entrypoint candidate discovery to map versioned pnpm package paths back to stable `node_modules/openclaw/dist` paths and adds a Vitest regression.

## Best Possible Solution

Rebase onto current main by folding the pnpm package-root candidate into the existing service-entrypoint resolver, preserve current symlink/dist behavior, and require redacted pnpm global upgrade service proof before merge.

## Reproduction Assessment

Yes at source level: set `process.argv[1]` to a versioned `.pnpm/.../node_modules/openclaw/openclaw.mjs` path and make only the stable `node_modules/openclaw/dist/index.js` accessible; current main does not add that stable package-root candidate. I did not run a live pnpm global upgrade reproduction in this read-only review.

## Solution Assessment

Mostly yes, but not merge-ready: the service-install resolver is the right layer for persisted daemon ProgramArguments, while update self-respawn remains separately tracked by https://github.com/openclaw/openclaw/issues/52313. The branch should now be rebased into the current `gateway-entrypoint` helper instead of landing the older parallel parsing shape as-is.
