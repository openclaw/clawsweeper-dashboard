## Summary

Keep this PR open, but it still needs changes before merge. Current main does not implement the versioned pnpm `openclaw.mjs` fallback, and the PR’s added candidate is tried after the version-pinned package `dist` candidate, so a normal pnpm global install where both paths exist would still persist the old `.pnpm/openclaw@...` service path.

## What This Changes

The PR adds pnpm package-root candidate discovery and a regression test so gateway service ProgramArguments can resolve from a versioned `.pnpm/.../openclaw.mjs` entrypoint to a stable `node_modules/openclaw/dist` entrypoint.

## Best Possible Solution

Land a focused daemon resolver change that prioritizes stable `node_modules/openclaw/dist` candidates for versioned pnpm `openclaw.mjs` invocations while preserving existing dist canonicalization, `.bin` fallback, runtime, and wrapper behavior, with a regression where both stable and versioned entries are accessible.

## Reproduction Assessment

Yes. A deterministic unit fixture can set `process.argv[1]` to a versioned pnpm `openclaw.mjs` path, make both `.pnpm/openclaw@.../dist/index.js` and stable `node_modules/openclaw/dist/index.js` accessible, and observe that the PR’s candidate order returns the versioned path first.

## Solution Assessment

No. The direction is right, but appending stable pnpm candidates after normal package-root dist candidates is not the narrowest correct fix; the stable pnpm package-root candidate needs priority before the versioned dist candidate.
