## Summary

Keep this PR open. The pnpm global upgrade regression remains plausible on current main, and the PR targets the right resolver path, but the patch still tries the stable pnpm package-root candidate after versioned package dist candidates, so it can leave service ProgramArguments pinned to `.pnpm/openclaw@...` when both paths exist.

## What This Changes

The PR adds pnpm package-root candidate discovery and a regression test so gateway service ProgramArguments can resolve from a versioned `.pnpm/.../openclaw.mjs` entrypoint to a stable `node_modules/openclaw/dist` entrypoint.

## Best Possible Solution

Revise the resolver on current main so a versioned `.pnpm/<package>@<version>/node_modules/<package>/openclaw.mjs` input derives the stable `node_modules/<package>/dist/*` candidates and tries them before any version-pinned package dist entries, while preserving wrapper and dist-symlink behavior.

## Reproduction Assessment

Yes. A deterministic unit fixture can set `process.argv[1]` to a versioned pnpm `openclaw.mjs` path, make both the versioned dist and stable package dist accessible, and observe that current main and this PR's ordering select the versioned path first.

## Solution Assessment

No. The direction is right, but appending stable pnpm candidates after normal package-root dist candidates is not the narrow complete fix; the stable candidate needs priority plus regression coverage where both paths exist.
