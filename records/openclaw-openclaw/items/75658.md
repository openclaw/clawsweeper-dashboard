## Summary

Keep open: current main partially cleans stale deleted/reset archives through store-save maintenance, but explicit `openclaw sessions cleanup --dry-run` still does not plan or report those archive files, and the unreferenced-artifact sweep still excludes `.deleted.*` and `.reset.*` archive names. The prior implementation PRs for this issue are closed unmerged, so no live linked PR currently owns the remaining cleanup/reporting gap.

## What This Changes

The issue asks `openclaw sessions cleanup` to remove and report physical `.deleted.*` and `.reset.*` session archive files, plus orphan checkpoint files, instead of only pruning `sessions.json` entries.

## Best Possible Solution

Reuse the canonical archived-transcript cleanup contract from the explicit `sessions cleanup` preview/apply flow, surface archive-specific counts in text and JSON dry-run output, and keep the existing deleted/reset retention semantics unchanged.

## Reproduction Assessment

Yes, at source level: the dry-run cleanup path only reports `pruneUnreferencedSessionArtifacts`, and that helper excludes `.deleted.*` and `.reset.*` archive names. I did not run a live filesystem CLI repro because this review is read-only.

## Solution Assessment

Yes for keeping it open and queueing a narrow fix. The maintainable solution is to share the existing archive cleanup contract with explicit cleanup preview/apply instead of duplicating archive expiration rules.

Update CLI text/JSON summary only as needed for the new archive cleanup result; do not edit CHANGELOG.md.
