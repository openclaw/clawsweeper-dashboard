## Summary

Keep open: current main still lacks a first-class `sessions cleanup` phase that plans/reports/applies ordinary `.deleted`, `.reset`, and orphan checkpoint file cleanup, and open PR #75672 remains the active closing implementation candidate.

## What This Changes

Have `openclaw sessions cleanup` report and remove stale physical `.deleted`, `.reset`, and orphan checkpoint transcript files instead of only pruning `sessions.json` entries.

## Best Possible Solution

Review and land or reject #75672; the desired end state is a cleanup-command archive maintenance phase that reports counts/bytes in dry-run and JSON output, applies under `--enforce`, and reuses the existing archive/checkpoint classifiers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with no disk budget configured, `enforceSessionDiskBudget` returns `null`, while `previewStoreCleanup` only plans index pruning/capping/missing-entry work, so a store containing only `.deleted`, `.reset`, and orphan checkpoint files can report zero removals.

## Solution Assessment

Yes on direction, pending PR review. The narrow maintainable solution is to wire physical artifact planning into `openclaw sessions cleanup` itself, with retention/checkpoint safety, rather than relying on write-time maintenance or optional disk-budget pressure.
