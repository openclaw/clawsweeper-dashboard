## Summary

Keep open: current main and v2026.6.10 only partially solve the report. The apply path can reach archive retention through store maintenance, but `sessions cleanup --dry-run` still previews only the unreferenced-artifact sweep that excludes `.deleted.*` and `.reset.*`, and the CLI summary still has no deleted/reset archive cleanup count.

## What This Changes

The issue requests that `openclaw sessions cleanup` remove and report physical `.deleted.*`, `.reset.*`, and orphan checkpoint session artifacts instead of only pruning session index entries.

## Best Possible Solution

Reuse the existing archive-retention contract for both cleanup preview and apply reporting, so stale deleted/reset archives are counted consistently without changing archive naming or adding new config.

## Reproduction Assessment

Yes, at source level: current dry-run cleanup delegates to an unreferenced-artifact sweep that excludes deleted/reset archive names. I did not run a live filesystem repro because this review is read-only.

## Solution Assessment

Yes for a narrow repair, if it shares the existing archive cleanup owner. A fix that adds a parallel deleted/reset expiration predicate would be the wrong shape because it could drift from store maintenance.
