## Summary

No blocking findings. The PR targets a real compatibility gap: current main treats bundled channel configs as pass-through during base parsing, then validates Discord with the generated strict channel schema, while the existing Discord doctor contract already treats per-channel `allow` as a legacy alias for `enabled`.

## What This Changes

The PR migrates legacy Discord per-channel `allow` aliases to canonical `enabled` during config validation and adds regression coverage for top-level and account-scoped Discord configs.

## Best Possible Solution

Land this narrow validation-time compatibility migration if normal branch checks remain green, then let the linked Discord regression issue close from the merged fix.

## Reproduction Assessment

Yes. Source inspection shows current main validates Discord channel configs with a strict generated schema that lacks `allow`, and the linked issue reports that exact startup-blocking path; I did not run local commands because this review is read-only.

## Solution Assessment

Yes. Canonicalizing the legacy alias before schema/plugin validation is the narrowest maintainable fix because it reuses the same compatibility meaning already implemented by Discord doctor without weakening the generated schema.
