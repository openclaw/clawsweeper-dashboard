## Summary

This PR should stay open because it addresses a real plugin hook limitation and supplies usable live proof, but it is not merge-ready: the new agent-facing fields can be stale after inline rewrites, and the public hook event shape needs compatibility/docs review before merge.

## What This Changes

The branch adds optional inbound body and attachment/media fields to the `before_agent_reply` plugin hook event and populates them from `getReplyFromConfig` with regression tests.

## Best Possible Solution

Land an additive, documented hook contract that derives bodies and attachments from the same finalized context the agent will use, with regression coverage for inline rewrites and path/url media shapes.

## Reproduction Assessment

Yes, source inspection proves current main calls `runBeforeAgentReply` with only `cleanedBody`, so attachment-aware hooks cannot observe current-turn media there. The PR body also includes redacted before/after live logs from a Telegram gateway.

## Solution Assessment

No, not yet. The hook is the right layer for a synthetic-reply gate, but the proposed event must use post-inline session text and a documented media contract before it is the best fix.

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use post-inline session text for hook content:** `src/auto-reply/reply/get-reply.ts:952-954`
  - body: `handleInlineActions` can rewrite only `sessionCtx.Body`, `sessionCtx.BodyForAgent`, and `sessionCtx.BodyStripped`, and the agent run later uses `sessionCtx.BodyStripped ?? sessionCtx.Body` as its prompt base. Since the new `content`/`bodyForAgent` fields are read from `ctx.BodyForAgent`, plugins can gate on stale pre-rewrite text while the agent sees a different body. Please derive these fields from the post-inline session context or `cleanedBody`, and add a regression for an inline rewrite.
  - confidence: 0.87
- **[P2] Cover URL-only media entries:** `src/auto-reply/reply/get-reply.ts:933-941`
  - body: Current media detection and finalization count `MediaUrl`/`MediaUrls` as inbound media, and the new attachment type makes `path` optional. Building rows only from `MediaPaths` means URL-only media contexts still produce no `attachments`, so a channel-agnostic media gate can miss valid current-turn media. Build rows from the path/url count, or explicitly document and test materialized-path-only semantics.
  - confidence: 0.78
