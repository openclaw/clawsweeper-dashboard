## Summary

Keep open: current main and v2026.6.1 cover part of the report with listener-backed status and a detached self-restart helper fallback, but the Windows service restart path still throws on a nonzero `schtasks /Run` result instead of falling back, so the central failure mode is not fully implemented.

## What This Changes

The issue requests better Windows gateway restart diagnostics, a graceful fallback when `schtasks /Run` cannot relaunch the Scheduled Task, and consistent status truth when a verified gateway listener is alive.

## Best Possible Solution

Make the Windows Scheduled Task restart path use one canonical fallback policy: preserve stderr/stdout diagnostics, launch the existing task script fallback when `/Run` fails or no-ops, and keep listener-backed status as the health truth.

## Reproduction Assessment

Unclear for live reproduction: the report has concrete Windows steps and logs, and source inspection shows the remaining nonzero `schtasks /Run` branch still throws without fallback, but I did not reproduce it on a Windows Scheduled Task host in this read-only review.

## Solution Assessment

No: current main only partially solves the issue. The best fix is to extend the existing Windows restart fallback and diagnostics to the direct `/Run` failure branch while preserving the safe restart deferral and listener-backed status behavior.
