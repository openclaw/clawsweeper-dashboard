## Summary

Keep open: current main still has the Discord skills-check false negative, but this PR should not merge as-is because it can report Discord ready for empty or tokenless account maps and it lacks contributor-provided real behavior proof.

## What This Changes

The PR adds `anyConfig` skill requirements across parsing, evaluation, formatting, and tests, then changes the bundled Discord skill to accept either `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Fix Discord skill readiness against token-bearing credentials, either with a precise account-token requirement or a narrow readiness predicate that mirrors `resolveDiscordToken`, with regression coverage for empty/tokenless account maps and redacted real `openclaw skills check` proof.

## Reproduction Assessment

Yes. Source inspection shows current main requires only `channels.discord.token` while Discord runtime/docs support account-scoped tokens, and the PR branch is source-reproducible as a false-positive because non-null account objects are truthy.

## Solution Assessment

No. A generic OR-config requirement may be useful, but checking `channels.discord.accounts` is not the best fix because the account container does not prove that any usable Discord token exists.
