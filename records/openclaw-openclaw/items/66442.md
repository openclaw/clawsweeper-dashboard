## Summary

This PR should stay open, but it is not merge-ready: it fixes a real current-main Discord skill false-negative, while the proposed `channels.discord.accounts` predicate can create a false-positive when no token-bearing account exists.

## What This Changes

The PR adds an `anyConfig` skill requirement through frontmatter/runtime/status formatting/tests and changes the bundled Discord skill to accept `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Keep the OR-requirement direction only if the Discord gate proves the root token or at least one token-bearing Discord account, with empty/tokenless-account regression coverage and redacted real CLI proof.

## Reproduction Assessment

Yes, source-level. Current main requires only `channels.discord.token` for the Discord skill while runtime and docs support `channels.discord.accounts.<id>.token`, so account-token-only configs still hit the false missing-requirement path.

## Solution Assessment

No. Generic OR config support is a reasonable shape, but this patch points the alternative at the broad `channels.discord.accounts` object instead of a concrete usable token.
