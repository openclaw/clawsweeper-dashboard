## Summary

Keep open. The PR addresses a valid current-main skills check gap, but the proposed Discord gate can report the skill as ready when `channels.discord.accounts` exists without any usable token because config path truthiness treats objects as satisfied while Discord token resolution still requires a concrete token.

## What This Changes

The PR adds an `anyConfig` OR requirement across skill metadata parsing, runtime/status evaluation, CLI formatting, and tests, then changes the bundled Discord skill to accept either top-level Discord token config or multi-account Discord config.

## Best Possible Solution

Keep the generic OR-requirement direction, but make the Discord gate prove a usable credential rather than the presence of the `accounts` container. A maintainable fix would support a documented wildcard or structured config requirement such as `channels.discord.accounts.*.token`, cover empty and tokenless account records in tests, and document the new public skill metadata field.

## Reproduction Assessment

Yes. With the PR semantics, a config containing `channels.discord.accounts: {}` or `{ work: { enabled: true } }` and no token satisfies the proposed `channels.discord.accounts` branch, while `resolveDiscordToken` returns no usable token.

## Solution Assessment

No. `anyConfig` is a reasonable generic seam, but using the accounts object itself is not the narrowest correct fix because it conflates container presence with Discord readiness; the gate should target an actual token-bearing path or predicate.
