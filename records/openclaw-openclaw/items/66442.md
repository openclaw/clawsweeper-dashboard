## Summary

Keep this PR open because it is the canonical fix path for the closed linked Discord skills-check bug, but it is not merge-ready: the new Discord requirement can be satisfied by an empty accounts object, the branch is currently conflicting, and the external PR still has only test-plan proof.

## What This Changes

The PR adds a new `metadata.openclaw.requires.anyConfig` OR requirement, wires it through shared skill/frontmatter/runtime requirement evaluation and CLI formatting, and changes the bundled Discord skill to accept either `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Land a refreshed patch that keeps the OR-config contract narrow but verifies a usable Discord token source, documents the new requirement field, and includes redacted real `openclaw skills check` output for a multi-account config.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: set `channels.discord.accounts` to an empty object with no root token, and the PR's `anyConfig` check can pass because config truthiness treats any object as satisfied.

## Solution Assessment

No. The `anyConfig` direction is plausible, but checking `channels.discord.accounts` is too broad; a token-aware account path or dedicated predicate is the safer fix.
