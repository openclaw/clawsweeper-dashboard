## Summary

Keep open. The PR addresses a real remaining gap from #66088 and the generic `anyConfig` direction is useful, but the current Discord gate is not mergeable because it treats the `channels.discord.accounts` container as proof of a usable credential; empty or tokenless account records would make `openclaw skills check` report Discord ready when the runtime still has no token.

## What This Changes

The PR adds `anyConfig` OR-style config requirements across skill metadata parsing, runtime/status evaluation, CLI formatting, and tests, then changes the bundled Discord skill to accept either `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Land a corrected version that keeps OR-style config requirements but makes Discord readiness depend on an actual top-level token, account token, or documented equivalent token-bearing predicate, with regression tests for empty and tokenless account configs plus docs and changelog coverage.

## Reproduction Assessment

Yes. On the PR branch, a config with `channels.discord.accounts: {}` or `{ work: { enabled: true } }` and no top-level/env token would satisfy the proposed `channels.discord.accounts` requirement, while `resolveDiscordToken` still returns no usable token.

## Solution Assessment

No. `anyConfig` is a reasonable generic seam, but using the accounts object itself is not the narrowest correct fix; the gate should target a credential-bearing account path or equivalent predicate.
