## Summary

Keep open: this PR targets a real Discord skills-check mismatch, but the current diff can mark Discord ready for empty or tokenless account containers, and the external contributor proof gate is still unmet.

## What This Changes

The PR adds an `anyConfig` skill requirement type, wires it through parsing/evaluation/formatting/tests, and changes the bundled Discord skill to accept either `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Keep the generic OR-config seam only if it can express token-bearing account readiness; otherwise add a narrow Discord readiness predicate or wildcard check with regression coverage for valid account tokens and empty or tokenless accounts.

## Reproduction Assessment

Yes. Source inspection shows current main still reports only `channels.discord.token`, and the PR branch would satisfy `channels.discord.accounts` for any non-null object even when no usable Discord token exists.

## Solution Assessment

No. `anyConfig` is a reasonable direction, but this PR should target credential-bearing account paths or an equivalent readiness predicate rather than the accounts container itself.
