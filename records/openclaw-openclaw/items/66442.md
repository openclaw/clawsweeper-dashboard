## Summary

Keep this PR open, but do not merge it as-is: it targets a real Discord multi-account skills-check bug, yet the proposed `channels.discord.accounts` gate can mark Discord ready when no usable token exists, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds `anyConfig` OR-style skill requirements across parsing, evaluation, formatting, and tests, then changes the bundled Discord skill to accept either `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Fix the skills check against token-bearing Discord readiness, either by supporting a precise wildcard/account-token config requirement or by adding a narrow readiness predicate that mirrors `resolveDiscordToken`, with regression coverage for valid account tokens and empty or tokenless account maps.

## Reproduction Assessment

Yes. Source inspection shows current main still gates the bundled Discord skill only on `channels.discord.token`, while docs and runtime support `channels.discord.accounts.<id>.token`; the PR branch is also source-reproducible as a false-positive for empty or tokenless account maps.

## Solution Assessment

No. A generic OR-config requirement may be reasonable, but checking `channels.discord.accounts` is not the best fix because account containers do not prove an available Discord token.
