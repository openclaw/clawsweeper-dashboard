## Summary

Keep this PR open: it addresses a real current-main Discord skills-check bug, but the proposed `channels.discord.accounts` gate introduces a false-positive readiness path and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR adds an `anyConfig` OR requirement through skill metadata parsing, requirement evaluation, CLI formatting, and tests, then changes the bundled Discord skill gate to accept `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Keep the OR-requirement direction only if it proves a usable Discord account token, add regression coverage for tokenless account maps, and require redacted real `openclaw skills check` proof before merge.

## Reproduction Assessment

Yes, source-level. Current main requires `channels.discord.token` for the Discord skill while Discord runtime and docs support `channels.discord.accounts.<id>.token`, so account-token-only configs still hit the reported false missing requirement path.

## Solution Assessment

No. Generic OR config support is a reasonable shape, but `channels.discord.accounts` is too broad because an object can be truthy without any usable token; the safer fix should prove at least one token-bearing Discord account or use an equivalent credential predicate.
