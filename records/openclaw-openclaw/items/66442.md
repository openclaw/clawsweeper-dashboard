## What This Changes

The PR adds an `anyConfig` OR requirement through skill metadata parsing, requirement evaluation, CLI formatting, and tests, then changes the bundled Discord skill to accept either `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Keep the OR-requirement direction only if the Discord gate proves the root token or at least one token-bearing Discord account, with regression coverage for empty/tokenless accounts and redacted real CLI proof.

## Reproduction Assessment

Yes, source-level. Current main requires only `channels.discord.token` for the Discord skill while docs and runtime token resolution support `channels.discord.accounts.<id>.token`, so account-token-only configs still hit the false missing-requirement path.

## Solution Assessment

No. Generic OR config support is a reasonable shape, but this PR's `channels.discord.accounts` predicate is too broad because it does not prove a token-bearing account.
