## Summary

Keep this PR open because current main still has the Discord multi-account skills-check bug and the linked issue was closed in favor of this PR, but the branch is not merge-ready: the proposed `channels.discord.accounts` readiness path can pass without any usable token, the branch currently conflicts with main, and real behavior proof is still only test-plan proof.

## What This Changes

The PR adds an `anyConfig` OR requirement shape for skills and changes the bundled Discord skill gate to accept either `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Land a rebased fix that uses a token-aware Discord readiness signal for root token, default env token, or `channels.discord.accounts.<id>.token`, with regression coverage for both valid account tokens and empty account maps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main reports only the root token requirement, and the PR can falsely pass when `channels.discord.accounts` is a truthy object without a usable token.

## Solution Assessment

No. The generic OR requirement is a plausible direction, but this implementation is too broad for a credential readiness check; a token-aware path or predicate is the safer maintainable fix.
