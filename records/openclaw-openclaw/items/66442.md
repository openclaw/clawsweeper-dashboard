## Summary

Keep open. The PR targets a real current-main skills-check mismatch for valid Discord multi-account configs, but the proposed `channels.discord.accounts` gate can produce a ready status without any usable account token, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds `anyConfig` requirement support, wires it through skill metadata parsing/evaluation/CLI formatting/tests, and changes the bundled Discord skill to accept either `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Keep the OR-requirement direction only if it can prove a token-bearing Discord account, then update the Discord skill/tests/formatting and add real before/after `openclaw skills check` proof.

## Reproduction Assessment

Yes, source-level. Current main requires `channels.discord.token` for the Discord skill while Discord docs and token resolution support `channels.discord.accounts.<id>.token`, so an account-token-only config still matches the reported false missing requirement path.

## Solution Assessment

No. Generic `anyConfig` support is a reasonable shape, but this PR’s `channels.discord.accounts` requirement is too broad because an accounts object can be truthy without any usable token; the safer fix should require a concrete token-bearing account path or equivalent predicate.
