## Summary

Keep this PR open, but it is not merge-ready: the new detector currently misses the production external-content wrapper format, so the central provenance signal would be absent on real web_fetch/browser/file-fetch paths.

## What This Changes

The PR adds an `externalContent` provenance signal to before_tool_call hook events and contexts, seeds it from wrapped external content in prompts and tool results, documents it, and adds regression coverage.

## Best Possible Solution

Parse the current `wrapExternalContent` marker format or shared structured wrapper metadata, add regression coverage that uses the production wrapper helper, then land only after maintainer approval of the public hook API semantics and real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows production `wrapExternalContent` emits `id="..."` markers while the PR detector requires ` BEGIN>>>`, so a real web_fetch result would be skipped.

## Solution Assessment

No. The propagation idea is useful, but the best fix must parse the production wrapper format or shared wrapper metadata and lock that with tests before exposing the public hook contract.
