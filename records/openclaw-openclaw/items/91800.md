## Summary

Keep this PR open, but the current patch is not merge-ready: the public security signal is useful, yet current head can still miss external-content provenance for production split-context tool runs and for external content already present in session history.

## What This Changes

The PR adds externalContent provenance types and fields to before_tool_call/trusted policy hook payloads, derives sources from wrapped external content, refreshes the signal after tool results, and updates hook docs/tests plus minor docs CI fixes.

## Best Possible Solution

Keep the additive hook idea, but make provenance run-owned, seed it from prompt/session/prompt-hook context, refresh it from wrapped tool results for all tool wrappers, and document the finalized source taxonomy with production-path coverage.

## Reproduction Assessment

Yes by source inspection. A production run that wraps web_fetch and exec with separate core tool contexts, or one with wrapped external content already in active session history, can reach before_tool_call without the promised externalContent signal; I did not run tests in this read-only review.

## Solution Assessment

No. The additive hook field is a good direction, but wrapper-local mutation plus prompt-only seeding is too narrow for a run-monotonic security signal; run-owned provenance state is the maintainable fix.

Full review comments:

- **[P1] Share external provenance across run-scoped tool contexts:** `src/agents/agent-tools.before-tool-call.ts:1376`
  - body: This only mutates the HookContext captured by the tool that just finished. Core tool assembly wraps each tool with a fresh object, and cataloging returns already wrapped tools unchanged, so a web_fetch result does not update the later exec hook context in production. Move the state to a run-owned object shared by all wrappers and cover that split-context path.
  - confidence: 0.92
- **[P1] Seed provenance from session messages, not only the prompt:** `src/agents/embedded-agent-runner/run/attempt.ts:1701`
  - body: The embedded runner sends activeSession.messages through prompt assembly, but initialToolHookExternalContent only scans params.prompt. External content already present in session history can therefore influence a tool call while event.externalContent remains unset. Include active/pruned session messages in the run-owned provenance seed.
  - confidence: 0.86
