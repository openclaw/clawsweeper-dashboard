## What This Changes

## Best Possible Solution

Use one run-owned provenance state across embedded prompt detection, core tool wrappers, tool-search/code-mode adapter hooks, client tools, and post-tool-result refresh, then document and prove that production path.

## Reproduction Assessment

Yes, source-reproducible: production core tools are wrapped with separate HookContext objects, while the PR refreshes only the context captured by the executing wrapper. I did not run tests because this review is read-only.

## Solution Assessment

No: adding the provenance field is a reasonable direction, but the implementation needs a run-shared provenance state and a regression through production tool construction rather than manually shared wrapper contexts.
