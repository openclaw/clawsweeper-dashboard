## Summary

Keep open: this is useful security-hook plumbing, but final head still lets untrusted wrapped content influence the reported source list, and the supplied proof is Vitest output rather than a real OpenClaw agent/plugin run.

## What This Changes

The PR adds plugin-facing external-content provenance fields, detects wrapped external content in prompts and tool results, propagates it through tool policy hook contexts, and documents the hook contract.

## Best Possible Solution

Parse provenance only from trusted wrapper metadata or structured tool-result metadata, share one detector across prompt-history and tool-result paths, add spoofed Source-line regression coverage, then land after maintainer approval of the public hook taxonomy and real behavior proof.

## Reproduction Assessment

Yes. By source inspection, a wrapped external string whose untrusted body contains an extra `Source: Email` line will be scanned by the new detector and can add a forged source to `externalContent.sources`.

## Solution Assessment

No. The propagation shape is useful, but the best fix must make provenance extraction host-authored, remove duplicate detector drift, and publish the plugin API only after maintainer approval.
