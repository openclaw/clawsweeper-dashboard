## Summary

Keep this PR open, but it is not merge-ready: the new public provenance signal is useful, yet current head still refreshes only wrapper-local hook context, so production core-tool paths can under-report same-run external content to later policy hooks.

## What This Changes

The PR adds optional `externalContent` provenance to `before_tool_call` hook event/context contracts, detects wrapped untrusted content in embedded runs and tool results, documents it, and adds focused tests.

## Best Possible Solution

Use one run-owned provenance state across embedded prompt detection, core tool wrappers, tool-search/code-mode adapter hooks, client tools, and post-tool-result refresh, then align docs, regression tests, and real proof to that production path.

## Reproduction Assessment

Yes, source-reproducible: production core tools are wrapped with separate context objects and already-wrapped tools skip the later adapter hook path, so a `web_fetch` result can still leave a later `exec` policy hook without `externalContent`. I did not run tests because this review is read-only.

## Solution Assessment

No: adding the public provenance field is directionally reasonable, but the current implementation is not the best fix because it relies on manually shared wrapper contexts. The safer shape is shared run-owned provenance consumed by every hook path before documenting monotonic same-run semantics.
