## Summary

Keep open: this is a plausible security-boundary improvement, but the current patch still under-reports production same-run external content because it mutates wrapper-local HookContext objects and the supplied proof exercises a manually shared harness instead of the embedded-run tool construction path.

## What This Changes

The PR adds optional externalContent provenance to before_tool_call hook event/context types, detects wrapped external content in embedded runs, refreshes it after tool results, adds tests, and documents the hook field.

## Best Possible Solution

Use one run-owned provenance state across embedded prompt detection, core tool wrappers, tool-search/code-mode adapter hooks, client tools, and post-tool-result refresh, then document and prove that production path.

Yes, source-reproducible: the PR refreshes only a wrapper-local HookContext while production core tools are wrapped with separate contexts and skip the later adapter hook path. I did not run tests because this review is read-only.

## Solution Assessment

No: adding the provenance field is directionally reasonable, but the implementation needs run-owned shared provenance state and a production-path regression instead of manually shared wrapper contexts.
