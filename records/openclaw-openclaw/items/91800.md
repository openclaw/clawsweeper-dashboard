## Summary

Keep open: the latest head addresses the prior spoofed-Source finding, but the detector still does not recognize the production external-content marker emitted by `wrapExternalContent`, so the central web_fetch-to-later-tool provenance claim fails on the real wrapper path.

## What This Changes

The PR adds plugin-facing external-content provenance to before_tool_call events and contexts, detects wrapped untrusted content from prompt/tool-result context, shares same-run provenance by runId, and updates related tests, docs, and type exports.

## Best Possible Solution

Parse the production wrapExternalContent marker format or structured externalContent metadata from tool results, update regressions to use the real wrapper, keep the detector shared, and land only after maintainer approval of the plugin hook semantics.

## Reproduction Assessment

Yes. Source inspection shows production web_fetch output is wrapped by wrapExternalContent with id-based markers, while the PR detector only recognizes marker lines containing BEGIN.

## Solution Assessment

No. The propagation shape is reasonable, but the implementation must parse production-format wrappers or structured tool-result metadata before this is the best fix.
