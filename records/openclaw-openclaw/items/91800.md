## Summary

Keep this PR open, but the current patch is not merge-ready: it adds a useful external-content provenance signal, yet the refresh path only mutates the just-run tool's HookContext, so production core tools wrapped with separate contexts can still miss the later policy signal.

## What This Changes

The PR adds external-content provenance types and hook payload fields, detects wrapped untrusted content, forwards the signal to before_tool_call/trusted policy hooks, adds tests, and documents the hook contract.

## Best Possible Solution

Keep the additive hook idea, but move provenance into run-owned/shared hook state consumed by core and cataloged tools, add production split-context coverage for external-result-to-exec, and then document the finalized source taxonomy.

## Reproduction Assessment

Yes for the patch defect by source inspection: the PR refreshes only one wrapper context, while current core tool assembly wraps each tool with a fresh context object. I did not run tests because this review was read-only.

## Solution Assessment

No. Mutating per-tool HookContext is too narrow for the documented run-monotonic security signal; a run-owned/shared provenance state is the maintainable fix.
