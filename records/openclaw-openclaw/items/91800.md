## Summary

Keep open: the security signal is useful, but the branch still snapshots provenance too early, so later tool policy hooks can miss external content introduced by a same-run web/browser result; the contributor proof is also test-only.

## What This Changes

The branch adds optional `externalContent` provenance to `before_tool_call` event/context and trusted policy hook context, deriving it from wrapped external-content markers in the embedded runner.

## Best Possible Solution

Ship a public hook signal whose provenance is updated whenever the active run gains external content, documented for plugin authors, with regression and redacted real-run proof.

## Reproduction Assessment

Yes, source-reproducible: the PR seeds `catalogToolHookContext.externalContent` before tool execution and only merges prompt-build content, while web/browser tools can add wrapped external content later in the same run. I did not execute tests because this review is read-only.

## Solution Assessment

No: adding the hook field is a plausible direction, but the implementation is too narrow until provenance is refreshed after same-run external tool results and the public SDK field is documented.
