## Summary

Keep open: the hook provenance idea is useful, but the branch is not merge-ready because the production core-tool path can still lose the security signal and the patch adds a broad ambient type declaration.

## What This Changes

The PR adds external-content provenance to before_tool_call/plugin hook payloads, embedded-run detection and refresh logic, tests, SDK types, and hook docs, plus a docs nav/glossary CI repair commit.

## Best Possible Solution

Keep the additive hook idea, move provenance into shared run-owned state used by all wrapped tools in one run, remove the ambient declaration, and merge only after maintainers confirm the public source taxonomy.

## Reproduction Assessment

Yes for the patch defect by source inspection: current core tool assembly uses separate hook context objects per tool, while the PR refreshes only the just-completed wrapper context.

## Solution Assessment

No. The additive hook idea is useful, but mutating per-tool HookContext is too narrow for the documented run-monotonic security signal; shared run-owned provenance state is the maintainable fix.

Full review comments:

- **[P1] Share provenance across core tool wrappers:** `src/agents/agent-tools.before-tool-call.ts:1376`
  - body: This refresh only mutates the HookContext captured by the tool that just completed. `createOpenClawCodingTools()` wraps each core tool with a fresh context object, and catalog wrapping skips tools that already have the before_tool_call marker, so a `web_fetch` result will not update the later `exec` hook context in the production path. The documented run-monotonic policy signal can be false-negative unless the provenance lives in shared run-owned state.
  - confidence: 0.92

  - body: This `.d.ts` file declares `activeSession` as a global for all source files included by `tsconfig.core.json`. That can mask future undeclared `activeSession` mistakes anywhere in the core build instead of keeping the type fix local to the runner code.
  - confidence: 0.86
