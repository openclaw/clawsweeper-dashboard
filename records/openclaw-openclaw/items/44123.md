## Summary

Keep open: current main still uses the `/new` or `/reset` hook dispatch time for session-memory artifact dates, but this PR needs a refresh before merge because it would regress the shipped local-time behavior and can stamp active content with a reset transcript's timestamp.

## What This Changes

The PR adds a session-header timestamp reader and regression tests so session-memory filenames and headers use the previous transcript creation time while the fallback HHMM slug stays based on the hook event time.

## Best Possible Solution

Refresh the patch so session-memory selects one transcript source, reads that transcript's header timestamp once, formats the selected instant with the existing local-time helper, keeps event time only for the conflict slug, and records the user-facing fix in the changelog/docs.

## Reproduction Assessment

Yes by source inspection. Create a previous session JSONL with a Day 1 session header, trigger `/new` on Day 2, and current main will derive the memory filename/header from `event.timestamp`, which is created at hook dispatch time.

## Solution Assessment

No. The direction is right, but the current PR implementation is not the best fix until it preserves host-local formatting and ties reset-header fallback to the transcript actually used for content.
