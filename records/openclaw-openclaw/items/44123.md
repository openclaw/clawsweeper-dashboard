## Summary

Keep this PR open, but it is not merge-ready. The requested session-creation-date behavior is still missing on current main, while the PR head is dirty against main and would reintroduce UTC timestamp formatting plus an over-broad reset-file timestamp fallback.

## What This Changes

The PR changes the bundled session-memory hook to read a previous session JSONL header timestamp for memory filenames and headers, keeps event time for the HHMM fallback slug, and adds regression tests for header, fallback, and reset-file cases.

## Best Possible Solution

Refresh this PR or open a narrow replacement that reads the previous session creation instant from the transcript layer, uses the same active-vs-reset fallback decision as message recovery, formats the selected instant with the existing host-local timestamp helper, keeps event time only for the HHMM conflict slug, and documents/changelogs the user-visible behavior.

## Reproduction Assessment

Yes. A focused reproduction can create a previous session JSONL whose header timestamp is Day 1, invoke the session-memory hook with a Day 2 /new event timestamp, and observe current main naming the memory file from Day 2.

## Solution Assessment

No. The direction is right, but this patch is not the best implementation because it is stale against the host-local timestamp fix and scans reset timestamp fallbacks in cases where the active transcript content is still used.
