## Summary

Keep open: the requested session-creation date behavior is still meaningful and not implemented on current main, but this branch is not mergeable as-is because it regresses current host-local timestamp formatting and can choose a reset-file timestamp for active transcript content. The merged local-time PR explicitly kept this session-creation date work separate, so it is not obsolete or superseded.

## What This Changes

The PR adds session-header timestamp parsing so bundled session-memory filenames and headers use the archived session creation time, plus regression tests for fallback cases.

## Best Possible Solution

Refresh on current main, select the session-created timestamp from the transcript source actually being archived, format it through the current local-time helper, preserve collision-safe filename suffixing, and add redacted real-run proof.

## Reproduction Assessment

Yes, source-level reproduction is clear: current main stamps session-memory artifacts from the /new or /reset event timestamp, so a transcript created on one day and archived on another is named for the command day. I did not execute it because this review is read-only.

## Solution Assessment

No, not as-is. The direction is reasonable, but the patch should preserve current local-time formatting and align reset timestamp fallback with the same active-versus-reset transcript decision used for content recovery.
