## What This Changes

The branch adds session transcript creation-timestamp parsing and regression tests so bundled session-memory filenames and headers use the session creation date instead of the `/new` or `/reset` event date.

## Best Possible Solution

Refresh this as a narrow timestamp-source fix against current main, selecting the timestamp from the same transcript source used for captured content, formatting through the current local-time helper, and preserving opt-in slugging, collision suffixes, and background writes.

## Reproduction Assessment

Yes, by source inspection. Current main derives the session-memory artifact date and header from the command event timestamp, so a session created before a day boundary and archived after it can be stamped with the command day.

## Solution Assessment

No, not as-is. The requested direction is plausible, but the branch needs a current-main refresh and maintainer alignment with the broader human-day timestamp semantics before it is the maintainable fix.

Full review comments:

- **[P2] Preserve local-time formatting for selected timestamps:** `src/hooks/bundled/session-memory/handler.ts:327-333`
  - body: Current main and the docs use host local date/time for session-memory artifacts. Formatting the selected session timestamp and fallback timestamp with `toISOString()` can put files back on the wrong local day for non-UTC users, so route the chosen timestamp through the existing local-time formatter.
  - confidence: 0.9
- **[P2] Gate reset timestamp fallback on the selected content source:** `src/hooks/bundled/session-memory/handler.ts:306-316`
  - body: The reset timestamp scan runs whenever the active file lacks a valid session header, even when that active transcript still has content. That can stamp active content with an older reset sibling's date, so the timestamp must come from the same transcript source selected for content.
  - confidence: 0.87
- **[P2] Keep LLM slug generation opt-in:** `src/hooks/bundled/session-memory/handler.ts:360`
  - body: Current main and the docs only call the model when `llmSlug: true`. Changing the condition to allow every non-false value makes ordinary enabled session-memory hooks call the provider during `/new` or `/reset`, adding unexpected latency and cost.
  - confidence: 0.9
- **[P2] Preserve collision-safe memory filenames:** `src/hooks/bundled/session-memory/handler.ts:378`
  - body: Current main adds a numeric suffix when the target session-memory filename already exists. Building `${dateStr}-${slug}.md` directly can overwrite or lose one artifact when two captures share the same selected date and slug.
  - confidence: 0.86
- **[P2] Keep memory capture off the command reply path:** `src/hooks/bundled/session-memory/handler.ts:224`
  - body: Current main starts session-memory writes in a background promise so command acknowledgements are not delayed. Keeping this hook as an awaited async handler can make `/new` and `/reset` wait on transcript reads and optional slug generation again.
  - confidence: 0.84
