## Summary

Keep open: the requested behavior is source-reproducible on current main and the PR is useful, but the branch is dirty against current main, lacks external real behavior proof, and needs repair to preserve the shipped local-time filename behavior while tightening reset-file fallback logic.

## What This Changes

The PR adds session-header timestamp parsing for session-memory filenames and headers, plus regression tests for normal, fallback, and rotated transcript cases.

## Best Possible Solution

Rework this on current main so the session creation timestamp flows through the existing local-time formatter, reset timestamp fallback only follows the rotated-transcript content fallback path, and regression coverage includes intact, rotated, invalid-header, and timezone-crossing cases.

## Reproduction Assessment

Yes, at source level. Current main formats the memory file date solely from `event.timestamp`, so a session transcript whose first `type: "session"` timestamp is Day 1 but whose `/new` event is Day 2 will still be filed under Day 2.

## Solution Assessment

No, not as currently written. The direction is reasonable, but the branch must be rebased onto current main, preserve the local-time formatter, and restrict reset timestamp fallback to the rotated-transcript case before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve local-time formatting for session dates:** `src/hooks/bundled/session-memory/handler.ts:329-333`
  - body: Current main intentionally formats session-memory filenames and headers in the host local timezone. This branch derives both the session-created date and fallback date with `toISOString()`, so after rebase a session created near a local/UTC date boundary would regress back to UTC-dated memory files and a `UTC` header. Feed the selected timestamp through the existing local formatter instead.
  - confidence: 0.88
- **[P2] Gate reset timestamp fallback to rotated transcripts:** `src/hooks/bundled/session-memory/handler.ts:306-316`
  - body: The reset sibling scan runs for every missing or invalid session header. For a non-empty active transcript with a bad or legacy first line, this can choose an older `*.reset.*` file's timestamp even though the current transcript content is the one being saved; use the same empty/rotated predicate as content fallback before reading reset timestamps.
  - confidence: 0.86
- **[P3] Avoid reading full transcripts twice:** `src/hooks/bundled/session-memory/handler.ts:148-149`
  - body: `getSessionCreatedAt()` reads the entire transcript, and the handler then reads it again for conversation extraction. Large transcripts will pay double I/O and memory cost on `/new` and `/reset`; read only the first line or share the transcript read with content extraction.
  - confidence: 0.75
