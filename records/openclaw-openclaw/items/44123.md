## Summary

Keep open: the PR addresses a plausible session-memory chronology bug, but current main has not implemented it and this head is not merge-ready because it can regress local-time formatting, can borrow reset timestamps for non-rotated transcripts, and lacks inspectable real-behavior proof.

## What This Changes

The PR changes the bundled session-memory hook to derive memory filename and header dates from a session JSONL header timestamp, with fallback behavior and regression tests.

## Best Possible Solution

Refresh the branch on current main, choose the session timestamp once, format it through the existing local-time formatter, align reset timestamp fallback with transcript recovery, and provide redacted real-run proof of a cross-day `/new` capture.

## Reproduction Assessment

Yes, source-level reproduction is clear: current main names and headers the memory file from the `/new` event timestamp, so a Day 1 transcript archived on Day 2 will be stamped as Day 2. I did not run it because this review is read-only.

## Solution Assessment

No, not as-is. The direction is reasonable, but the implementation should reuse the local-time formatter for the selected timestamp and only consult reset timestamps when the active transcript is actually rotated/empty.
