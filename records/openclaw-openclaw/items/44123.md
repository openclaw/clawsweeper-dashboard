## Summary

Keep this PR open: current main still does not implement session-creation-date memory filenames, but the PR head is not merge-ready because it would regress the current host-local timestamp contract, can mismatch reset fallback dates with active transcript content, and lacks inspectable real behavior proof.

## What This Changes

The PR changes the bundled session-memory hook to derive memory filename and header dates from the session transcript creation timestamp and adds regression coverage.

## Best Possible Solution

Refresh the branch on current main, select the session-created timestamp only from the transcript source actually being archived, format that selected timestamp through the local-time formatter, keep event time only for collision-safe fallback slugs when intended, and add redacted real-run proof.

## Reproduction Assessment

Yes. Source-level reproduction is clear: current main names and headers the memory artifact from the `/new` or `/reset` event timestamp, so a transcript created on one local day but archived on another is stamped with the command day; I did not execute it because this review is read-only.

## Solution Assessment

No, not as-is. The direction is reasonable, but the patch should reuse current local-time formatting and gate reset timestamp fallback on the same content-source decision used by transcript recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve local-time formatting for session timestamps:** `src/hooks/bundled/session-memory/handler.ts:327-333`
  - body: Current main and the docs use the host local date/time for session-memory filenames and headers. Formatting the selected timestamp with `toISOString()` would switch both session-created and fallback paths back to UTC, regressing the current local-time behavior.
  - confidence: 0.86
- **[P2] Gate reset-date fallback on rotated transcript content:** `src/hooks/bundled/session-memory/handler.ts:306`
  - body: The reset timestamp fallback runs for every missing or invalid active session header. If the active transcript still has usable messages, content comes from that active file but the date can come from an older reset sibling, stamping the current conversation with the wrong session date.
  - confidence: 0.82
