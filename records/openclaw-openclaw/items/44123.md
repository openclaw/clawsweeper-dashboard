## Summary

Keep open: the proposed behavior is still useful and current main still dates session-memory artifacts from the command event, but this PR is not mergeable as-is because it regresses host-local timestamp formatting, can choose reset timestamps for active content, and lacks inspectable real behavior proof.

## What This Changes

The PR adds session JSONL header timestamp parsing and tests so session-memory filenames and headers use session creation time while keeping the fallback HHMM slug tied to the command event time.

## Best Possible Solution

Refresh the PR against current main, derive the archive timestamp from the same transcript source that provides the saved content, preserve `formatLocalSessionTimestamp` and current filename collision behavior, and attach redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the filename date and header time derive from `event.timestamp`, so a session created before midnight and archived by `/new` after midnight is dated by the command time.

## Solution Assessment

No, not as-is. The direction is reasonable, but the patch should preserve host-local formatting and choose reset timestamps only when the reset transcript is also the content source.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve local-time formatting for session timestamps:** `src/hooks/bundled/session-memory/handler.ts:327-333`
  - body: Current main formats session-memory filenames and headers in host local time. This code converts both session-created and fallback event timestamps with `toISOString()`, so sessions near local midnight can be saved under the wrong local date/header.
  - confidence: 0.86
- **[P2] Gate reset-date fallback on rotated content:** `src/hooks/bundled/session-memory/handler.ts:306`
  - body: The reset scan runs whenever the active header is missing or invalid. If the active transcript still has usable messages and an older reset sibling exists, this can stamp the active conversation with the reset file's creation date; timestamp selection should follow the same primary-first predicate as content recovery.
  - confidence: 0.82
