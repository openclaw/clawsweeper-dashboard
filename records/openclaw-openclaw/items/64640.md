## Summary

Current main already solves the two observable problems this PR targets: memory-search config checks no longer cold-load memory provider capability plugins during doctor/status resolution, and webhooks registration is synchronous with SecretRef secrets resolved at request time. The PR branch is stale against that design and would reintroduce an async webhooks registration path that the current loader rejects.

## What This Changes

The PR branch forwards active config into memory-search embedding provider lookups, adds memory-search regression coverage, changes webhooks inline-secret registration versus async SecretRef fallback handling, and adds webhooks tests.

## Best Possible Solution

Close this PR as implemented on main and keep the shipped design: memory-search config resolution stays registry-only/config-safe, while webhooks keep synchronous route registration with request-time SecretRef auth from #67941.

## Reproduction Assessment

Yes. The stale PR behavior is reproducible from the diff plus current loader semantics: any route registration inside the promise callback runs after `register()` returns and the guarded API has closed. Current-main tests cover the intended synchronous SecretRef-backed webhooks path and inactive-memory doctor behavior.

## Solution Assessment

No, the PR branch is not the best way to solve the issue now. Current main is narrower and safer because it avoids runtime capability-provider loading in memory config checks and preserves request-time webhook secret resolution instead of splitting inline and async registration paths.
