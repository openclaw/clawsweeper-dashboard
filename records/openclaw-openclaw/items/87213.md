## Summary

Keep open: current `main` and the latest release still perform the raw `boundNode !== requestedNode` check before selector resolution, and the issue is already paired with an open closing PR that directly targets this bug.

## What This Changes

The issue reports that node-bound exec rejects a valid `host=node` request when the configured bound node and requested node are different selector strings for the same physical node.

## Best Possible Solution

Review and land or close the linked PR; the desired end state is to resolve both selectors through the shared node resolver, compare canonical node IDs, and keep regression coverage for same-node and different-node cases.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: pass a configured `boundNode` full id and a `requestedNode` display name for the same listed node, and current code throws before the resolver can canonicalize either selector. I did not run a live paired-node setup.

## Solution Assessment

Yes, the narrow fix is to resolve selectors first and compare canonical node identity before enforcing the bound-node guard. The linked PR appears to target that path directly.
