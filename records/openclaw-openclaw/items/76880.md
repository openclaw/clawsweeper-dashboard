## Summary

Keep open: the reported `res?.headers.get is not a function` path is still source-reproducible on current main, and the response-header fix is only present in open PR #76791 rather than merged main.

## What This Changes

Google Chat inbound webhook auth should recover from the May 1 regression where google-auth-library receives non-`Headers` response headers during certificate retrieval.

## Best Possible Solution

Port or land the response-header normalization from #76791 in the Google Chat auth transport with focused regression coverage and changelog coverage, then track the later `unsupported_grant_type` report separately if it remains reproducible.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main routes Google Chat `app-url` verification through `OAuth2Client.verifyIdToken()`, google-auth-library 10.6.2 calls `res?.headers.get('cache-control')`, and current main lacks response-header normalization.

## Solution Assessment

Is this the best way to solve the issue? Yes for the response-header failure: keeping the fix inside `extensions/googlechat/src/google-auth.runtime.ts` is the narrow plugin-owned repair; the open PR still needs rebase/port and review before it can close this issue.
