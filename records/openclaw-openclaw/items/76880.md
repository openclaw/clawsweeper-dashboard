## Summary

Keep open: the reporter’s follow-up is still valid on current main. The prior request-header fix is present, but the Google Chat auth transport still does not normalize response headers before google-auth-library reads `res?.headers.get('cache-control')`.

## What This Changes

Google Chat inbound webhook authentication regressed after the earlier certificate-header fix, now failing during verification certificate response handling.

## Best Possible Solution

Patch the Google Chat plugin’s Google auth transport so Gaxios response headers also satisfy the `Headers.get()` contract, add focused regression coverage, and land that as the canonical fix for this issue.

Do we have a high-confidence way to reproduce the issue? Yes at source level: inbound Google Chat `app-url` verification still reaches google-auth-library’s certificate response path with a transport that only normalizes request headers; a live Google Chat app was not run in this review.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix belongs in `extensions/googlechat/src/google-auth.runtime.ts` by normalizing response headers for google-auth-library, rather than changing core channel or gateway behavior.
