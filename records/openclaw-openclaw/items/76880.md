## Summary

Keep open: current main does not implement the response-header normalization needed for the reporter's second Google Chat auth failure. The existing 01e2755 fix only normalizes request headers, while google-auth-library 10.6.2 reads certificate response cache control through `res.headers.get(...)`, matching the current rejection.

## What This Changes

Restore Google Chat inbound webhook auth after a regression where certificate retrieval fails because the custom Google auth transport can expose response headers without a Fetch `Headers.get` method.

## Best Possible Solution

Patch the Google Chat plugin-owned Google auth transport to normalize Gaxios response headers to native `Headers`, keep the request-header fix, and add regression coverage plus a changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible but not live-reproduced: configure Google Chat and send a webhook for the `app-url` auth path; current source calls `verifyIdToken`, and the pinned dependency accesses `res.headers.get('cache-control')` after the custom transport path.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix belongs in `extensions/googlechat` around the custom Google auth transport response compatibility, without a core or plugin API change.
