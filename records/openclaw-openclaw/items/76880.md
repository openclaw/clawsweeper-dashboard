## Summary

Keep open: current `main` still has the reporter’s follow-up failure mode. The Google Chat auth transport normalizes request headers, but it does not normalize Gaxios response headers before `google-auth-library` reads `res?.headers.get('cache-control')`; the related response-header PR was closed unmerged.

## What This Changes

The issue reports a Google Chat inbound webhook regression where Google auth certificate verification rejects webhook auth because response headers no longer satisfy the `Headers.get()` contract.

## Best Possible Solution

Patch the Google Chat plugin’s Google auth transport to normalize response headers through the existing Gaxios response-interceptor seam, add focused regression coverage, and keep the fix plugin-owned.

## Reproduction Assessment

Yes, at source level: current `main` routes Google Chat `app-url` verification through `google-auth-library` with a transport that only normalizes request headers, while the pinned upstream reads response headers through `.get()`. A live Google Chat webhook was not run in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to normalize Gaxios response headers inside `extensions/googlechat/src/google-auth.runtime.ts` and cover that helper in the existing runtime test, not to change core channel or gateway behavior.
