## Summary

Keep open: current main still routes Google Chat `app-url` webhook verification through the custom google-auth/Gaxios transport, and that transport only normalizes request headers; the reported response-header failure is still not fixed on main.

## What This Changes

Restore Google Chat inbound webhook auth after the earlier request-header fix left google-auth-library response headers incompatible with the Fetch `Headers.get()` contract.

## Best Possible Solution

Add response-header normalization to the Google Chat plugin's shared google-auth transport, keep it plugin-owned, add focused regression coverage, and include a user-facing changelog fix entry.

## Reproduction Assessment

Yes at source level: inbound `app-url` verification still reaches google-auth-library's certificate path, which reads `res?.headers.get('cache-control')`, while current main only normalizes request headers. A live Google Chat webhook was not run in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to normalize Gaxios response headers inside `extensions/googlechat/src/google-auth.runtime.ts`, not to change core gateway routing, add config, or update dependency versions first.
