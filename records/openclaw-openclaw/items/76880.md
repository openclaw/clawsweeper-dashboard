## Summary

Keep open: current main still matches the reporter's follow-up failure path. The previous request-header compatibility fix is present, but the Google Chat auth transport does not normalize Gaxios response headers before google-auth-library reads certificate cache-control via `headers.get`.

## What This Changes

The issue asks to restore inbound Google Chat webhook authentication after the prior request-header fix changed the failure from `config.headers.has` to `res?.headers.get`.

## Best Possible Solution

Patch the Google Chat plugin's shared Google auth transport so plain-object Gaxios response headers are restored to Fetch `Headers`, preserve the existing request-header and SSRF-guard behavior, and add focused regression coverage.

## Reproduction Assessment

Yes at source level: inbound `app-url` verification still uses the custom Gaxios transport and google-auth-library 10.6.2 reads certificate response headers through `.get()`. I did not run a live Google Chat app, so this is `source_reproducible` rather than a live local reproduction.

## Solution Assessment

Yes: the narrow maintainable fix is response-header normalization in `extensions/googlechat/src/google-auth.runtime.ts`, not a core channel or gateway change. PR #76906 is useful source context, but it closed unmerged and current main still lacks the behavior.
