## Summary

Keep open: current main still matches the reporter’s follow-up. The prior request-header fix is present, but the Google Chat auth transport still has no response-header normalization before google-auth-library reads certificate response headers with `headers.get()`.

## What This Changes

The issue asks to fix the Google Chat inbound webhook auth regression left after #76742, where certificate retrieval now fails with `res?.headers.get is not a function`.

## Best Possible Solution

Normalize Gaxios response headers in the Google Chat auth transport before google-auth-library consumes them, add focused regression coverage for plain-object and native `Headers` response headers, and keep the fix plugin-owned.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main routes `app-url` webhook auth through `OAuth2Client.verifyIdToken`, the transport only normalizes request headers, and the pinned google-auth-library reads certificate response headers via `res?.headers.get()`.

## Solution Assessment

Is this the best way to solve the issue? Yes: a narrow response-header normalization in `extensions/googlechat/src/google-auth.runtime.ts` is the maintainable fix; changing core channel or gateway behavior would be broader than the failing dependency contract requires.
