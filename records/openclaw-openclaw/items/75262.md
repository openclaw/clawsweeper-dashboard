## Summary

Keep open. Current main does not implement the requested `cron run --as-scheduled` behavior, and this PR is not ready to merge because it applies scheduled cwd by mutating process-global cwd during an awaited Gateway cron run and omits the required docs/changelog/test coverage for a new user-facing CLI/protocol option.

## What This Changes

The branch adds an `--as-scheduled` option to `openclaw cron run`, threads `asScheduled` through CLI, Gateway protocol/handlers, cron service APIs, timer execution, and isolated cron execution, then changes flagged manual isolated runs to `process.chdir` into the agent workspace.

## Best Possible Solution

Implement scheduled-environment parity as explicit run-scoped cron execution policy: carry a typed option through the CLI/Gateway/service contracts, apply cwd/env/context through runner and workspace seams rather than process-global mutation, document the flag, add an active changelog entry, and cover manual-versus-scheduled parity with focused cron tests.

## Reproduction Assessment

Yes for the PR regression. The diff shows `--as-scheduled` reaches `runCronIsolatedAgentTurn`, where the branch calls `process.chdir` before awaiting cron execution; current main already passes `workspaceDir` into both agent runners as a run-scoped parameter.

## Solution Assessment

No. The flag direction may be useful, but mutating process-global cwd is not the best way to solve it; the safer solution is a run-scoped execution policy with explicit workspace, env, and context handling.
