## Summary

Keep open. The PR targets a real cron debugging gap tracked by #71777, but the latest patch is not mergeable as-is because it implements the new flag by changing the Gateway process cwd during an awaited cron run and does not include the user-facing docs or changelog required for the new CLI/protocol option.

## What This Changes

The PR adds an `--as-scheduled` option to `openclaw cron run` and threads an `asScheduled` boolean through the CLI, Gateway cron schema/handler, cron service APIs, timer path, and isolated-agent execution path.

## Best Possible Solution

Implement scheduled-environment parity as an explicit run-scoped cron execution policy: carry the option through typed cron contracts, apply cwd/env/context through existing runner seams, document the CLI behavior, add a changelog entry, and cover manual-versus-scheduled parity with focused cron tests.

## Reproduction Assessment

Yes. The PR-introduced issue is reproducible by source inspection: `--as-scheduled` reaches `runCronIsolatedAgentTurn`, which calls `process.chdir` before awaiting isolated execution, while current main already passes `workspaceDir` to the actual runner calls.

## Solution Assessment

No. The flag direction is plausible, but mutating process-global cwd is not the best fix; scheduled semantics should stay scoped to the cron run through the existing `workspaceDir`/runner contract and any explicit env/context hooks.
