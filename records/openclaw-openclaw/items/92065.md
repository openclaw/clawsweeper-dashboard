## Summary

This PR should close as superseded by the still-open canonical QMD timeout issue: the reported bug is real, but this branch repeats the single `memory.qmd.limits.timeoutMs` plus fixed-overhead wrapper design that a later member review closed unmerged as unsafe for QMD's multi-step search lifecycle.

## What This Changes

The PR changes `memory_search` so QMD-backed searches use the resolved `memory.qmd.limits.timeoutMs` plus 5 seconds as the outer tool deadline, with fake-timer regression tests.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/91947 as the canonical tracker and land a replacement design where QMD owns or exposes a defensible whole-search budget while runtime loading and wiki supplements keep separate bounded deadlines.

## Reproduction Assessment

Yes, source-level with high confidence: current main wraps the entire `memory_search` call in a fixed 15s deadline while QMD can use longer configured per-command timeouts. I did not run a live QMD reranker in this read-only review.

## Solution Assessment

No. This PR is not the best fix because it maps one QMD per-command timeout plus 5s onto a broader tool lifecycle that can include runtime loading, sync, repair, fallback, and multiple collection searches.
