## Summary

Close this PR as superseded: the QMD memory_search timeout bug is real and remains tracked by the canonical issue, but this branch repeats a single-QMD-timeout-plus-fixed-overhead design that later maintainer review rejected as unsafe.

## What This Changes

The PR changes memory-core so QMD-backed memory_search derives its outer deadline from the configured QMD timeout plus 5 seconds, and adds fake-timer regression tests.

## Best Possible Solution

Keep the open canonical issue as the tracker and replace this branch with a QMD-owned whole-search budget that preserves separate bounded deadlines for runtime loading, wiki supplements, cooldown behavior, and non-QMD/default paths.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main and v2026.6.8 use a fixed 15 second memory_search wrapper while the linked report shows QMD query/rerank taking far longer with memory.qmd.limits.timeoutMs configured above 15 seconds.

## Solution Assessment

No. This PR is not the best fix because it maps one QMD per-command timeout plus fixed overhead onto a broader lifecycle that includes runtime loading, sync/wait, fallback, repair, retries, multiple collection groups, and supplements.

Full review comments:

- **[P1] Do not treat one QMD timeout as the whole tool budget:** `extensions/memory-core/src/tools.ts:402-405`
  - body: The new wrapper deadline is `qmdTimeout + 5s` for the entire tool call, but QMD search can spend time in warm/sync/wait steps, multiple collection-group searches, fallback, missing-collection repair, and empty-result retry. Valid searches can still be killed even when each QMD command respects its own timeout, so this needs a whole-search budget or separate bounded phases.
  - confidence: 0.91
- **[P2] Keep runtime resolution inside the fail-open guard:** `extensions/memory-core/src/tools.ts:364-365`
  - body: This resolves/imports the memory runtime before `runMemorySearchToolWithDeadline` starts. Current main keeps that load inside the guarded callback, so a slow or stalled runtime resolution still returns the 15s fail-open result; this patch removes that protection.
  - confidence: 0.79
