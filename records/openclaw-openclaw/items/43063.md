## Summary

Keep open: the requested provider billing JSON is not present on current main, but this PR still has blocking correctness, security, ownership-boundary, CI, and real behavior proof gaps before it can merge.

## What This Changes

The PR adds `auth.providerUsage` JSON output for windowed and metered provider usage, a new OpenRouter billing fetcher, and focused models status tests.

## Best Possible Solution

Implement metered billing behind provider-owned usage hooks, extend the generic JSON schema deliberately, gate OpenRouter billing to JSON output, and verify with focused tests plus redacted live command proof.

## Reproduction Assessment

Yes for the PR blockers: CI reports a failed `check` job, and source inspection shows the Promise-typed variable, non-JSON OpenRouter fetch, and error-status gaps. The requested billing JSON itself is a new feature rather than a current-main bug reproduction.

## Solution Assessment

No, not as written. The safer maintainable path is to keep provider-specific OpenRouter billing behind provider usage hooks, fix error/status handling, and require real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited result from the `Promise.all` branch. This breaks the JSON payload typing and should use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This fetch runs for every non-plain invocation, including default rich `openclaw models status`, but `openrouterUsage` is only emitted in the JSON payload. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays this data.
  - confidence: 0.94
- **[P2] Call the documented OpenRouter key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The fetcher calls `/api/v1/auth/key` for per-key usage. The provider contract needs to use the current documented key endpoint and have tests that would fail if the endpoint path drifts.
  - confidence: 0.86
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is converted to `null`, so invalid credentials, missing management-key permission, or billing failures can produce `status: "ok"` with missing account data instead of an error.
  - confidence: 0.88
- **[P2] Mark partial key failures as provider errors:** `src/infra/provider-usage.openrouter.ts:272-279`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
- **[P2] Keep billing behind provider usage hooks:** `src/commands/models/list.status-command.ts:39`
  - body: This adds an OpenRouter-specific core import and command special case even though current main exposes provider-owned usage hooks for provider HTTP fetch and response normalization. Put OpenRouter parsing behind the OpenRouter plugin and extend only the generic schema in core.
  - confidence: 0.82
