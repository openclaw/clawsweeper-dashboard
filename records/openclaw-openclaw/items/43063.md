## Summary

Keep this PR open, but it is not merge-ready: the feature is useful, current main does not already expose OpenRouter metered billing in status JSON, and the branch still has blocking type, provider-boundary, credentialed-request, and provider-error handling issues plus missing real behavior proof.

## What This Changes

This PR adds OpenRouter metered billing data and a unified `auth.providerUsage` object to `openclaw models status --json`, with tests for the new OpenRouter usage fetcher.

## Best Possible Solution

Expose OpenRouter metered data through the provider-owned usage contract or a deliberate extension of it, keep fetches scoped to modes that render the data, surface provider errors accurately, and validate with redacted live command output.

## Reproduction Assessment

Yes for the PR blockers: PR-head source and CI logs show the type error, non-JSON OpenRouter fetch, and silent non-2xx paths. The requested billing JSON is a new feature rather than a current-main bug reproduction.

## Solution Assessment

No, not as written. The maintainable path is to use the provider-owned usage hook contract or explicitly extend it, then prove the real OpenRouter output before merge.

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited result from the `Promise.all` branch. The PR check fails on this assignment, so use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P1] Route OpenRouter billing through provider usage hooks:** `src/commands/models/list.status-command.ts:377-380`
  - body: Current main routes provider-specific usage behavior through `resolveUsageAuth` and `fetchUsageSnapshot`; adding a separate core OpenRouter fetcher creates competing auth, endpoint, timeout, and error behavior. Move the metered fetch into the OpenRouter provider hook path or extend that contract deliberately.
  - confidence: 0.9
- **[P1] Keep OpenRouter billing out of default status:** `src/commands/models/list.status-command.ts:347-380`
  - body: This fetch runs for every non-plain invocation, including default rich `openclaw models status`, but the OpenRouter metered payload is only emitted in JSON. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays this data.
  - confidence: 0.94
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is converted to `null`, so invalid credentials, missing permission, or billing failures can produce `status: "ok"` with missing account data instead of an error.
  - confidence: 0.88
- **[P2] Mark partial key failures as provider errors:** `src/infra/provider-usage.openrouter.ts:272-280`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out or unauthorized profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
