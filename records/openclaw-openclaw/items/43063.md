## Summary

Keep open. Current main does not implement this `auth.providerUsage` JSON surface, and the PR still has merge blockers: failing TypeScript check, OpenRouter-specific core wiring that bypasses provider usage hooks, incorrect/weak OpenRouter API error handling, a credentialed non-JSON fetch path, and no real behavior proof.

## What This Changes

The PR adds `auth.providerUsage` to `openclaw models status --json`, a new OpenRouter metered billing fetcher, and tests for the OpenRouter usage output.

## Best Possible Solution

Implement OpenRouter billing through provider-owned usage hooks, extend any generic JSON schema deliberately, gate metered fetches to JSON output, use documented endpoints and error semantics, and require focused tests plus redacted live command proof.

## Reproduction Assessment

Yes for the PR blockers: the live `check` job reproduces the TypeScript failure, and source/diff inspection shows the non-JSON OpenRouter fetch plus endpoint and error-status gaps. The requested billing JSON itself is a new feature rather than a current-main bug reproduction.

## Solution Assessment

No, not as written. The maintainable path should keep provider-specific billing fetch/parsing behind the OpenRouter provider plugin, fix endpoint/error handling, and require redacted real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited result from the `Promise.all` branch. The live `check` job fails with TS2322 here and then cascades into `never` errors while building the JSON payload; use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This fetch runs for every non-plain invocation, including default rich `openclaw models status`, but `openrouterUsage` is only emitted in the JSON payload. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays this data.
  - confidence: 0.94
- **[P2] Call the documented key-info endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: OpenRouter documents current-key usage and limit metadata at `/api/v1/key`, but this code calls `/api/v1/auth/key`. Valid keys can therefore be treated as missing data instead of producing the requested JSON usage. ([openrouter.ai](https://openrouter.ai/docs/api/api-reference/api-keys/get-current-key))
  - confidence: 0.9
- **[P2] Return errors for non-2xx key responses:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is converted to `null`, so invalid credentials, missing management-key permission, or billing failures can produce `status: "ok"` with missing account data instead of an error. ([openrouter.ai](https://openrouter.ai/docs/api-reference/get-credits))
  - confidence: 0.88
- **[P2] Mark partial key failures as errors:** `src/infra/provider-usage.openrouter.ts:279`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out or unauthorized profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
- **[P2] Keep billing behind provider usage hooks:** `src/commands/models/list.status-command.ts:39`
  - body: This adds an OpenRouter-specific core import and command special case even though current main exposes provider-owned `resolveUsageAuth` and `fetchUsageSnapshot` hooks for usage endpoints. Put OpenRouter parsing behind the OpenRouter plugin and extend only the generic schema in core if needed.
  - confidence: 0.82
