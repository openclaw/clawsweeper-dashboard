## Summary

Keep open: the requested capability is not implemented on current main, but this PR is not merge-ready because the head fails typecheck, sends OpenRouter credentials from default status output, has endpoint/error-handling problems, and lacks real behavior proof.

## What This Changes

The PR adds an `auth.providerUsage` JSON section for windowed provider usage and OpenRouter metered billing, plus a new OpenRouter billing fetcher and tests.

## Best Possible Solution

Define a reviewed provider-usage JSON contract through provider-owned usage hooks, put OpenRouter billing in the OpenRouter plugin, gate metered fetches to JSON output, fix endpoint/error semantics, add focused tests, and include redacted live command proof.

## Reproduction Assessment

Yes for the PR blockers: the live CI log shows the typecheck failure, and source inspection shows the non-JSON OpenRouter fetch plus endpoint and error-handling gaps. This is a feature PR rather than a current-main bug reproduction.

## Solution Assessment

No. The direction is useful, but this patch is not the narrowest maintainable solution until the schema is reviewed, the OpenRouter logic moves behind provider usage hooks, endpoint/error handling is corrected, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited value; the live `check` job fails with TS2322 here and then cascades into `never` errors while building the JSON payload. Use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This runs for default rich `openclaw models status`, but `openrouterUsage` is only emitted in the JSON payload. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays the data.
  - confidence: 0.94
- **[P2] Call the documented key-info endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: OpenRouter documents current-key usage and limit metadata at `GET /api/v1/key`, but this calls `/api/v1/auth/key`, so valid keys can be dropped as missing data instead of producing the requested JSON usage. ([openrouter.ai](https://openrouter.ai/docs/api/api-reference/api-keys/get-current-key?utm_source=openai))
  - confidence: 0.9
- **[P2] Return errors for non-2xx key responses:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is also converted to `null`, so invalid credentials or missing management permission can produce `status: "ok"` with missing account data instead of an error.
  - confidence: 0.88
- **[P2] Mark partial key failures as errors:** `src/infra/provider-usage.openrouter.ts:279`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out or unauthorized profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
- **[P2] Keep billing behind provider usage hooks:** `src/commands/models/list.status-command.ts:39`
  - body: This adds an OpenRouter-specific core import and command special case even though current main exposes provider-owned `resolveUsageAuth` and `fetchUsageSnapshot` hooks for usage endpoints. Put the OpenRouter parser behind the OpenRouter plugin and extend the generic schema there.
  - confidence: 0.82
