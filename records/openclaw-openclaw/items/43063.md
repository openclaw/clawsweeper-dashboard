No. The maintainable path is to extend the provider usage hook/schema boundary and keep OpenRouter billing in the OpenRouter provider, not add a core command-level fetcher with separate credential handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move billing into the provider usage hook:** `src/infra/provider-usage.openrouter.ts:1-5`
  - body: Current main already has `resolveUsageAuth`/`fetchUsageSnapshot` so provider plugins own provider-specific billing endpoints. Adding a new core OpenRouter fetcher duplicates credential resolution and HTTP parsing outside that boundary, which will drift from plugin auth behavior; put this in the OpenRouter provider and let `loadProviderUsageSummary` fan out.
  - confidence: 0.9
- **[P1] Type the awaited OpenRouter usage value:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a `Promise<OpenRouterMeteredUsage | null>`, but the code assigns the awaited resolved value to it. This will fail TypeScript; use `Awaited<ReturnType<...>>` or the concrete resolved type.
  - confidence: 0.94
- **[P1] Gate OpenRouter billing calls to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This runs `loadOpenRouterMeteredUsage` for every non-plain status call, but `openrouterUsage` is only rendered in the JSON payload. Default rich `openclaw models status` would therefore send a bearer token and wait on OpenRouter even though it shows none of the metered data.
  - confidence: 0.92
- **[P2] Propagate OpenRouter HTTP failures:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A non-2xx `/auth/key` response returns `null`, and the caller treats `null` as missing data rather than an error. With revoked or unauthorized keys, JSON status can omit the failed key while still reporting `status: "ok"`; return a tracked provider error for non-2xx responses.
  - confidence: 0.88
