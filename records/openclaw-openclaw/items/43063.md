## Summary

Keep open: current main does not implement the requested JSON billing data, but this PR is not ready because it conflicts with current main and needs rework around provider-owned usage hooks plus fixes for unnecessary OpenRouter fetches and error handling.

## What This Changes

The PR adds a `providerUsage` JSON schema to `openclaw models status --json`, adds OpenRouter billing/account/key usage fetching, and updates status tests.

## Best Possible Solution

Extend the provider-usage/plugin contract to support metered snapshots, implement OpenRouter billing in `extensions/openrouter`, and serialize it in `models status --json` only while preserving rich OAuth usage.

## Reproduction Assessment

Not applicable. This is a feature PR; source inspection confirms current main has no `auth.providerUsage` JSON field, and patch inspection plus `git apply --check` show the proposed branch still needs review and rebase.

## Solution Assessment

No. The PR targets a useful surface, but the current implementation puts OpenRouter billing in core, fetches it for non-JSON rich status, and misses non-2xx error reporting; the maintainable path is a generic metered provider-usage contract with OpenRouter owned by its plugin.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move OpenRouter billing behind provider hooks:** `src/commands/models/list.status-command.ts:36-37`
  - body: Current main already exposes provider-owned `resolveUsageAuth`/`fetchUsageSnapshot` hooks and keeps OpenRouter behavior in `extensions/openrouter`. This direct core import and new `src/infra/provider-usage.openrouter.ts` hardcode a bundled provider into the models command, so third-party and future metered providers cannot use the same contract. Please move the OpenRouter fetch/parsing into the OpenRouter plugin and keep the command on a generic usage surface.
  - confidence: 0.9
- **[P2] Avoid fetching OpenRouter data for rich status:** `src/commands/models/list.status-command.ts:375-380`
  - body: This runs `loadOpenRouterMeteredUsage` for every non-plain status call, but `openrouterUsage` is only serialized inside the JSON payload. A normal `openclaw models status` with an OpenRouter key will now make credentialed network requests and can block on OpenRouter timeouts without showing that data. Gate this fetch to `opts.json` unless rich output actually renders it.
  - confidence: 0.92
- **[P2] Report non-2xx OpenRouter responses as errors:** `src/infra/provider-usage.openrouter.ts:173-175`
  - body: `fetchOpenRouterKeyUsage` returns `null` for 401/403/5xx responses, and the caller only records thrown errors, so invalid or revoked keys can be silently omitted while the provider status remains `ok`. Treat non-2xx key and credits responses as tracked provider errors so JSON health does not report incomplete billing data as healthy.
  - confidence: 0.88
