## Label Justifications

- `P2`: This is a useful but non-urgent provider status feature with limited blast radius and several merge blockers.
- `merge-risk: 🚨 auth-provider`: The diff changes how status resolves and sends OpenRouter credentials outside the existing provider usage hook path.

## Summary

Keep open: current main still does not expose `auth.providerUsage` in `openclaw models status --json`, but this PR has blocking correctness, provider-boundary, credentialed-request, and proof gaps before it is mergeable.

## What This Changes

The PR adds OpenRouter metered billing data plus windowed provider usage data to `openclaw models status --json`, with a new OpenRouter usage fetcher and focused status-command tests.

## Best Possible Solution

Add metered usage through provider-owned usage hooks, expose a deliberate additive JSON schema for `--json` only, and require redacted live command output before merge.

## Reproduction Assessment

Yes for the PR blockers: CI logs show the type-check failure, and source inspection shows the non-JSON OpenRouter fetch and silent non-2xx paths. The requested billing JSON itself is a new feature, not a current-main bug reproduction.

## Solution Assessment

No, not as written. The maintainable path is provider-owned usage hooks, JSON-only billing fetches, accurate provider error propagation, and redacted live proof.

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited result from the `Promise.all` branch. The PR check fails on this assignment, so use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P1] Route OpenRouter billing through the provider usage hook:** `src/commands/models/list.status-command.ts:377-380`
  - body: Current main routes provider-specific usage behavior through `resolveUsageAuth` and `fetchUsageSnapshot`, but this adds a separate core OpenRouter fetcher in the status command. That creates competing auth, endpoint, timeout, and error behavior; move the metered fetch into the OpenRouter provider hook instead.
  - confidence: 0.9
- **[P2] Keep OpenRouter billing out of default status:** `src/commands/models/list.status-command.ts:377-380`
  - body: This fetch runs for every non-plain invocation, including default rich `openclaw models status`, but the OpenRouter metered payload is only emitted in JSON. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays this data.
  - confidence: 0.94
- **[P2] Call the documented OpenRouter key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The fetcher calls `/api/v1/auth/key`, while OpenRouter documents the current-key endpoint as `GET /api/v1/key`; use the provider contract endpoint and test that path. ([openrouter.ai](https://openrouter.ai/docs/api/api-reference/api-keys/get-current-key?utm_source=openai))
  - confidence: 0.9
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is converted to `null`, so invalid credentials, missing management-key permission, or billing failures can produce `status: "ok"` with missing account data instead of an error.
  - confidence: 0.88
- **[P2] Mark partial key failures as provider errors:** `src/infra/provider-usage.openrouter.ts:272-279`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out or unauthorized profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
