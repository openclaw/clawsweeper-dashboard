## Summary

Keep this PR open, but it is not merge-ready: the feature is useful and not already on current main, but the patch still has a type-breaking CI failure, an unnecessary credentialed OpenRouter request on non-JSON status output, provider-boundary/API handling issues, and no real behavior proof.

## What This Changes

The PR adds `auth.providerUsage` JSON output for windowed provider usage plus a new OpenRouter metered billing fetcher and focused models-status tests.

## Best Possible Solution

Implement metered billing through the OpenRouter provider usage hooks, extend the generic status JSON schema deliberately, and require focused tests plus redacted real command output before merge.

## Reproduction Assessment

Yes for the PR blockers: live CI reports the Promise-typed assignment failure, and source inspection shows the non-JSON OpenRouter fetch plus error-status gaps. The requested billing JSON is a new feature rather than a current-main bug reproduction.

## Solution Assessment

No, not as written. The safer maintainable solution is to keep OpenRouter billing parsing behind the OpenRouter provider usage hooks, fix status/error handling, and require redacted real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited result from the `Promise.all` branch. The current CI failure shows this breaks type checking; use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This fetch runs for every non-plain invocation, including default rich `openclaw models status`, but `openrouterUsage` is only emitted in the JSON payload. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays this data.
  - confidence: 0.94
- **[P2] Call the documented OpenRouter key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The fetcher calls `/api/v1/auth/key`, but OpenRouter documents the current-key endpoint as `/api/v1/key` and single-key lookup as `/api/v1/keys/:hash`. This needs to use the provider contract endpoint and have tests that fail if the path drifts. ([openrouter.ai](https://openrouter.ai/docs/api/api-reference/api-keys/get-current-key?utm_source=openai))
  - confidence: 0.9
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is converted to `null`, so invalid credentials, missing management-key permission, or billing failures can produce `status: "ok"` with missing account data instead of an error.
  - confidence: 0.88
- **[P2] Mark partial key failures as provider errors:** `src/infra/provider-usage.openrouter.ts:272-279`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out or unauthorized profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
- **[P2] Keep billing behind provider usage hooks:** `src/commands/models/list.status-command.ts:39`
  - body: This adds an OpenRouter-specific core import and command special case even though current main exposes provider-owned usage hooks for provider HTTP fetch and response normalization. Put OpenRouter parsing behind the OpenRouter plugin and extend only the generic schema in core.
  - confidence: 0.82
