## Summary

Keep open: current main does not implement the new `auth.providerUsage` JSON output, and the PR still has blocking issues in the proposed patch, including a TypeScript type mismatch, unnecessary OpenRouter billing calls outside JSON output, and provider-specific billing logic placed in core instead of the existing plugin usage hooks.

## What This Changes

The PR adds OpenRouter metered billing fetches, adds `auth.providerUsage` JSON for windowed and metered providers, and rewires models-status usage fetching for JSON/rich output.

## Best Possible Solution

Rework this as a generic provider-usage JSON schema extension with metered snapshots owned by the OpenRouter plugin, preserving rich/plain status latency and explicit error reporting.

## Reproduction Assessment

Yes for the PR findings by source inspection: the PR head has a Promise `ReturnType` mismatch, runs OpenRouter billing fetches for non-plain status where only JSON consumes the result, and drops non-2xx provider responses as null.

## Solution Assessment

No. The requested feature is useful, but the safer design is to extend the generic provider-usage/plugin contract and implement OpenRouter billing in `extensions/openrouter`, with JSON-only metered fetches and tracked provider errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable is assigned the awaited result and later dereferenced as an object. This should fail typechecking; use `Awaited<ReturnType<...>>` or an explicit resolved usage type.
  - confidence: 0.95
- **[P2] Do not fetch OpenRouter billing for rich status:** `src/commands/models/list.status-command.ts:347-381`
  - body: This block runs for every non-plain invocation, including default `openclaw models status`, but `openrouterUsage` is only emitted in the JSON payload. Users with an OpenRouter key now pay a credentialed remote request/timeout for output that never uses it; gate this fetch to `opts.json`.
  - confidence: 0.94
- **[P2] Mark non-2xx OpenRouter responses as errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: `/auth/key` non-2xx responses return `null`, and the caller only records thrown exceptions, so revoked or failing keys can disappear while provider `status` stays `ok`. Convert non-2xx responses into tracked provider errors or partial-error state.
  - confidence: 0.9
- **[P2] Keep OpenRouter billing behind provider hooks:** `src/commands/models/list.status-command.ts:36-37`
  - body: This adds OpenRouter-specific core infra and a command special case even though provider usage hooks already exist for provider-owned usage endpoint behavior. Add metered support to the generic usage contract and register the OpenRouter implementation from `extensions/openrouter` instead.
  - confidence: 0.83

## Security Review

Status: needs_attention
