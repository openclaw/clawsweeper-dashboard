## Summary

Keep this PR open, but it is not merge-ready: the feature is not on current main, and the branch still has CI, correctness, security, proof, and provider-boundary blockers.

## What This Changes

The branch adds an `auth.providerUsage` JSON payload to `openclaw models status --json`, introduces an OpenRouter metered billing fetcher and tests, and rewires status usage fetching.

## Best Possible Solution

Implement the billing JSON through provider-owned usage hooks, keep OpenRouter-specific parsing in the OpenRouter plugin, gate OpenRouter billing fetches to JSON output, fix endpoint and error semantics, and add focused tests plus redacted live command proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR blockers: CI reproduces the TypeScript failure, and source/diff inspection shows the non-JSON OpenRouter fetch plus endpoint/error-status gaps. The requested billing JSON is a new feature rather than a current-main bug reproduction.

## Solution Assessment

Is this the best way to solve the issue? No, not as written. The maintainable path should use the provider-owned usage hooks, correct OpenRouter API/error handling, reviewed JSON schema semantics, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the resolved OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited result. The PR head fails `pnpm check` with TS2322 here and then cascades into `never` errors while building the JSON payload; use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This call runs for the default rich `openclaw models status` path, but `openrouterUsage` is only emitted in the JSON payload. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays billing data.
  - confidence: 0.94
- **[P2] Call the documented key-info endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: OpenRouter's OpenAPI contract exposes current-key usage and limit metadata at `/key` under the API base path, but this calls `/api/v1/auth/key`. Valid keys can be treated as missing data instead of producing the requested JSON usage.
  - confidence: 0.9
- **[P2] Return errors for non-2xx key responses:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is converted to `null`, so invalid credentials or missing billing permission can produce `status: "ok"` with missing account data instead of an error.
  - confidence: 0.88
- **[P2] Mark partial key failures as errors:** `src/infra/provider-usage.openrouter.ts:279`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
- **[P2] Keep billing behind provider usage hooks:** `src/commands/models/list.status-command.ts:39`
  - body: This adds an OpenRouter-specific core import and command special case even though current main exposes provider-owned `resolveUsageAuth` and `fetchUsageSnapshot` hooks for usage endpoints. Put the OpenRouter parser behind the OpenRouter plugin and extend the generic schema there.
  - confidence: 0.82
