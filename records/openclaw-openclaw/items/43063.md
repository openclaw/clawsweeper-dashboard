## Summary

Keep open: current main does not implement the requested JSON billing data, and this PR still has blocking typecheck/runtime/API issues, missing real behavior proof, and a provider-boundary decision before merge.

## What This Changes

This PR adds an `auth.providerUsage` JSON schema to `openclaw models status --json`, wires windowed OAuth usage into it, and adds a new OpenRouter metered billing fetcher and tests.

## Best Possible Solution

Move OpenRouter metered billing into the provider usage hook contract, expose a deliberate additive JSON schema only for `--json`, surface provider errors accurately, and land with focused tests plus redacted live command output.

## Reproduction Assessment

Yes for the PR blockers: live CI reports the TS2322 failure, and source inspection shows the non-JSON billing fetch plus silent non-2xx failure paths. The requested billing JSON is a new feature rather than a current-main bug reproduction.

## Solution Assessment

No, not as written. The maintainable path is provider-owned usage hooks plus JSON-only fetching and accurate error reporting, not an OpenRouter-specific command-level billing fetcher.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited result from the `Promise.all` branch. The current `check` failure shows this breaks type checking; use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P2] Keep OpenRouter billing off non-JSON status:** `src/commands/models/list.status-command.ts:377-380`
  - body: This fetch runs for every non-plain invocation, including default rich `openclaw models status`, but the OpenRouter metered payload is only emitted in JSON. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays this data.
  - confidence: 0.94
- **[P2] Call the documented OpenRouter key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The fetcher calls `/api/v1/auth/key`, but OpenRouter documents the current-key endpoint as `/api/v1/key`. This needs to use the provider contract endpoint and have tests that fail if the path drifts. ([openrouter.ai](https://openrouter.ai/docs/api/api-reference/api-keys/get-current-key))
  - confidence: 0.9
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is converted to `null`, so invalid credentials, missing management-key permission, or billing failures can produce `status: "ok"` with missing account data instead of an error.
  - confidence: 0.88
- **[P2] Mark partial key failures as provider errors:** `src/infra/provider-usage.openrouter.ts:272-279`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out or unauthorized profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
