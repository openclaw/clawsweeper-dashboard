## Summary

Keep open. The feature is still absent from current main and the PR is useful, but the patch is not merge-ready because it has a typecheck blocker, makes unrendered authenticated OpenRouter billing calls, puts provider-specific billing logic in core instead of the provider plugin seam, and exposes billing metadata without a documented public contract.

## What This Changes

The PR adds an `auth.providerUsage` JSON schema to `openclaw models status --json`, maps existing windowed usage snapshots into it, and adds an OpenRouter metered billing fetcher plus tests.

## Best Possible Solution

Rework this as an intentional provider-usage contract change with OpenRouter billing implemented through the OpenRouter plugin usage hooks, documented JSON schema, changelog coverage, and accurate degraded/error states for provider failures.

## Reproduction Assessment

Yes. Static inspection is enough: current main does not emit `auth.providerUsage`, and the PR diff shows the compile-time type mismatch, non-JSON OpenRouter billing fetch, core-owned OpenRouter billing parser, endpoint mismatch, and silent failure paths.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best maintainable implementation because it bypasses the provider-owned usage seam and exposes billing metadata before the public JSON contract is documented and approved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the Promise type, but this variable stores the awaited value and is later read as an object. This should fail typechecking; use `Awaited<ReturnType<...>>` or an explicit snapshot type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-381`
  - body: `openrouterUsage` is only emitted in the JSON payload, but the fetch runs for every non-plain status invocation. Default rich status can now make authenticated OpenRouter billing requests and wait on timeouts even though it never renders the data.
  - confidence: 0.94
- **[P2] Keep OpenRouter billing inside the provider plugin:** `src/infra/provider-usage.openrouter.ts:1-10`
  - body: This adds an OpenRouter-specific billing fetcher under core infra even though the provider-usage hook already lets plugins own provider-specific usage HTTP parsing. Move this behavior into the OpenRouter provider plugin or extend the generic usage seam.
  - confidence: 0.9
- **[P2] Call OpenRouter's documented key endpoint:** `src/infra/provider-usage.openrouter.ts:162`
  - body: The key-usage fetch requests `/api/v1/auth/key`, while OpenRouter documents the current-key endpoint as `/api/v1/key`. Valid keys can therefore produce failed or empty key usage in the new JSON payload.
  - confidence: 0.95
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:173-175`
  - body: A 401, 403, or 500 key response becomes `null`, and the caller only records thrown errors. Invalid or revoked keys can be silently omitted while the provider still reports an apparently healthy status with incomplete data.
  - confidence: 0.91
- **[P2] Mark partial key fetch failures as degraded:** `src/infra/provider-usage.openrouter.ts:240-248`
  - body: When one key throws but another succeeds, `keyErrors` is ignored unless there are zero successful keys. Multi-key setups can therefore return `status: "ok"` while omitting failed profiles from the JSON payload.
  - confidence: 0.89
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:138-140`
  - body: OpenRouter documents `/credits` as management-key-only, but a non-2xx response is converted to `null` and can still return an `ok` provider when key usage succeeds. That hides real auth or permission failures for account totals.
  - confidence: 0.87
