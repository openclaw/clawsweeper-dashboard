## Summary

Keep open: current main does not expose the requested `auth.providerUsage` JSON, and this PR still needs correctness, boundary, proof, and merge-conflict work before it is merge-ready.

## What This Changes

The PR adds an OpenRouter metered billing fetcher and extends `openclaw models status --json` with windowed and metered provider usage data.

## Best Possible Solution

Move OpenRouter metered billing into provider-owned usage hooks or a deliberate metered extension of that contract, keep billing fetches JSON-only unless rich output renders them, propagate provider errors, and require redacted live command output.

## Reproduction Assessment

Yes for the PR blockers: CI logs show the type-check failure, and source inspection shows the non-JSON OpenRouter fetch plus silent non-2xx paths. The requested billing JSON itself is a new feature, not a current-main bug reproduction.

## Solution Assessment

No, not as written. The safer solution is to use the provider-owned usage hook contract, make metered fetch scope explicit, and validate the real OpenRouter output before merge.

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited result from the `Promise.all` branch. The PR check fails on this assignment, so use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P1] Route OpenRouter billing through provider usage hooks:** `src/commands/models/list.status-command.ts:377-380`
  - body: Current main routes provider-specific usage behavior through `resolveUsageAuth` and `fetchUsageSnapshot`; adding a separate core OpenRouter fetcher creates competing auth, endpoint, timeout, and error behavior. Move the metered fetch into the OpenRouter provider hook path or extend that contract deliberately.
  - confidence: 0.9
- **[P1] Keep OpenRouter billing out of default status:** `src/commands/models/list.status-command.ts:377-380`
  - body: This fetch runs for every non-plain invocation, including default rich `openclaw models status`, but the OpenRouter metered payload is only emitted in JSON. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays this data.
  - confidence: 0.94
- **[P2] Call the documented OpenRouter key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The fetcher calls `/api/v1/auth/key`, while OpenRouter documents the current-key endpoint as `GET /api/v1/key`; use the provider contract endpoint and test that path. ([openrouter.ai](https://openrouter.ai/docs/api/api-reference/api-keys/get-current-key))
  - confidence: 0.9
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is converted to `null`, so invalid credentials, missing management-key permission, or billing failures can produce `status: "ok"` with missing account data instead of an error. ([openrouter.ai](https://openrouter.ai/docs/api-reference/get-credits))
  - confidence: 0.88
- **[P2] Mark partial key failures as provider errors:** `src/infra/provider-usage.openrouter.ts:272-279`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out or unauthorized profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
