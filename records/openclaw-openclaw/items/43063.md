## Summary

Keep open: current main does not implement the requested `auth.providerUsage` JSON surface, and the PR still has blocking issues in typechecking, credentialed default-status fetching, OpenRouter endpoint/error handling, provider-boundary ownership, and real behavior proof.

## What This Changes

The PR adds an `auth.providerUsage` JSON section to `openclaw models status --json`, a core OpenRouter metered billing fetcher, and tests for the new output.

## Best Possible Solution

Add a reviewed provider-usage JSON contract through the existing provider-owned usage hook path, with OpenRouter billing in the OpenRouter plugin, JSON-only metered fetches, correct endpoint/error semantics, focused tests, and redacted real command proof.

## Reproduction Assessment

Yes for the PR blockers: the live CI log reproduces the typecheck failure, and source inspection reproduces the non-plain OpenRouter fetch, endpoint mismatch, and error-handling gaps. This is a feature PR rather than a current-main bug reproduction.

## Solution Assessment

No. The requested capability is useful, but this patch is not the best maintainable shape until the public schema is reviewed, OpenRouter logic is moved behind the provider usage contract, endpoint/error handling is corrected, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited value; the live `check` job fails with TS2322 here and then cascades into `never` errors while building the JSON payload. Use `Awaited<ReturnType<...>>` or an explicit resolved type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This runs for default rich `openclaw models status`, but `openrouterUsage` is only emitted in the JSON payload. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays the data.
  - confidence: 0.94
- **[P2] Call the documented key-info endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: OpenRouter documents current-key usage and limit metadata at `GET /api/v1/key`, but this calls `/api/v1/auth/key`, so valid keys can be dropped as missing data instead of producing the requested JSON usage. ([openrouter.ai](https://openrouter.ai/docs/api/api-reference/api-keys/get-current-key?utm_source=openai))
  - confidence: 0.9
- **[P2] Return errors for non-2xx billing responses:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: The billing helpers convert non-2xx responses into `null`, and the caller treats that as absent data. Revoked keys, a failing key-info request, or a credits request rejected for lacking a management key can therefore produce `status: "ok"` with missing account/key data instead of a provider error.
  - confidence: 0.91
- **[P2] Mark partial key failures as errors:** `src/infra/provider-usage.openrouter.ts:279`
  - body: When at least one key succeeds, key-level failures collected in `keyErrors` leave the provider status as `ok` unless the credits call also failed. A timed-out or unauthorized profile can be silently omitted from JSON health in multi-key setups.
  - confidence: 0.88
- **[P2] Keep billing behind provider usage hooks:** `src/commands/models/list.status-command.ts:39`
  - body: This adds an OpenRouter-specific core import and command special case even though current main exposes provider-owned `resolveUsageAuth` and `fetchUsageSnapshot` hooks for usage endpoints. Put the OpenRouter parser behind the OpenRouter plugin and extend the generic schema there.
  - confidence: 0.82
