## Summary

Keep open. The feature is useful, but the current PR is not merge-ready because it lacks real OpenRouter behavior proof, is merge-conflicting, and still has blocking type, provider-boundary, credentialed-request, and error-reporting problems.

## What This Changes

Adds a new OpenRouter metered billing fetcher and an `auth.providerUsage` JSON schema in `openclaw models status --json`, plus focused tests for OpenRouter usage parsing.

## Best Possible Solution

Expose OpenRouter metered billing through the provider-owned usage contract or an intentional extension of it, fetch only for output modes that display the data, surface endpoint and partial failures accurately, then prove redacted live JSON output.

## Reproduction Assessment

Yes for the PR blockers: PR-head source shows the awaited-type error, default non-plain OpenRouter fetch, and silent non-2xx or partial-failure paths. The requested billing JSON is a new feature rather than a current-main bug reproduction.

## Solution Assessment

No, not as written. The safer solution is to wire OpenRouter billing through provider-owned usage hooks, emit it only in JSON or explicitly rendered usage surfaces, and surface endpoint or partial failures as provider errors.

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the Promise type, but this variable stores the awaited value assigned inside the `Promise.all` branch. This should use `Awaited<ReturnType<...>>` or an explicit resolved type so the check job can pass.
  - confidence: 0.96
- **[P1] Route OpenRouter billing through provider usage hooks:** `src/commands/models/list.status-command.ts:377-380`
  - body: Current main routes provider-specific usage through `resolveUsageAuth` and `fetchUsageSnapshot`; adding a separate core OpenRouter fetcher creates competing auth, endpoint, timeout, and error behavior. Move this logic into the OpenRouter provider usage hook path or get explicit maintainer approval for a new contract.
  - confidence: 0.9
- **[P1] Keep OpenRouter billing out of default status:** `src/commands/models/list.status-command.ts:377-380`
  - body: This fetch runs for every non-plain invocation, including default rich `openclaw models status`, but the OpenRouter metered payload is only rendered in JSON. Users with an OpenRouter key get an unnecessary credentialed request and timeout risk for output that never displays this data.
  - confidence: 0.94
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A non-2xx `/credits` response is converted to `null`, so invalid credentials, missing permission, or billing failures can produce `status: "ok"` with missing account data instead of an error.
  - confidence: 0.88
- **[P2] Treat non-2xx key usage responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A revoked or unauthorized key response is converted to `null`, and the caller treats that as absent data. In mixed-key setups this can silently omit failed keys while the provider still looks healthy.
  - confidence: 0.9
- **[P2] Mark partial key failures as provider errors:** `src/infra/provider-usage.openrouter.ts:272-280`
  - body: When at least one key succeeds, key-level failures leave `status` as `ok` unless the credits call also failed. A timed-out or unauthorized profile can therefore be silently omitted from JSON health.
  - confidence: 0.88
