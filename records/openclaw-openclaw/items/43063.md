## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Move OpenRouter billing behind the provider usage hook:** `src/infra/provider-usage.openrouter.ts:99-127`
  - body: This new core helper resolves OpenRouter credentials and owns provider-specific billing HTTP parsing directly. Current main already has provider usage hooks where provider plugins own usage endpoint fetching, so this creates a second credential and billing path instead of extending `extensions/openrouter`.
  - confidence: 0.9
- **[P1] Gate OpenRouter billing fetch to JSON output:** `src/commands/models/list.status-command.ts:347-380`
  - body: `openrouterUsage` is only emitted in the JSON payload, but this block runs for every non-plain status call. Users with an OpenRouter key would send a bearer token and wait on external billing endpoints during default `models status`, where the data is not rendered.
  - confidence: 0.92
- **[P2] Report partial key failures as errors:** `src/infra/provider-usage.openrouter.ts:272-283`
  - body: When one key succeeds and another key request throws, `keyErrors` is nonempty but `creditsError` can be null, so this partial-success branch leaves `status` as `ok` and silently omits the failed key.
  - confidence: 0.88
- **[P2] Surface non-2xx OpenRouter responses:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: Non-2xx key responses are converted to `null`, and callers treat that as missing data rather than a provider failure. Revoked credentials can therefore produce an apparently healthy provider with missing key rows.
  - confidence: 0.86
