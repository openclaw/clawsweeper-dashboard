## Summary

Keep open: current main still does not expose provider billing data in `models status --json`, and this PR still has blocking correctness, security, and plugin-boundary issues plus no after-fix real behavior proof.

## What This Changes

The PR adds an `auth.providerUsage` JSON section to `models status`, a core OpenRouter billing fetcher, and tests for metered usage data.

## Best Possible Solution

Extend the generic provider-usage JSON/schema contract deliberately, implement OpenRouter billing through `extensions/openrouter`, fetch metered billing only for JSON output, and surface HTTP failures as provider errors.

## Reproduction Assessment

Yes for the PR findings by source inspection: the PR head has a Promise/resolved-type mismatch, fetches OpenRouter billing for default rich output where only JSON consumes it, and collapses non-2xx billing responses to null.

## Solution Assessment

No. The feature is useful, but the current patch is not the narrowest maintainable solution because metered billing should go through the provider usage/plugin contract with explicit schema review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited value and later reads object fields from it. Use `Awaited<ReturnType<...>>` or an explicit resolved usage type so the branch typechecks.
  - confidence: 0.95
- **[P2] Gate OpenRouter billing to JSON output:** `src/commands/models/list.status-command.ts:347-381`
  - body: This block runs for default rich `openclaw models status`, but `openrouterUsage` is only emitted in the JSON payload. Users with an OpenRouter key get an unnecessary credentialed network request and timeout risk for output that never displays the data.
  - confidence: 0.94
- **[P2] Mark non-2xx OpenRouter responses as errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: `/auth/key` non-2xx responses return `null`, and the caller treats that as missing data instead of a provider failure. A revoked key can be omitted while provider status remains healthy, so convert HTTP failures into tracked error or partial-error state.
  - confidence: 0.9
- **[P2] Keep OpenRouter billing behind provider hooks:** `src/commands/models/list.status-command.ts:36-37`
  - body: This adds an OpenRouter-specific core import and command special case even though current provider contracts expose `resolveUsageAuth` and `fetchUsageSnapshot` for provider-owned usage endpoints. Put the OpenRouter billing implementation behind the OpenRouter plugin and extend the generic schema there.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The patch introduces an unnecessary credential-bearing OpenRouter billing request on the default rich status path.

Concerns:

- **[medium] Default status sends OpenRouter credentials:** `src/commands/models/list.status-command.ts:377`
  - body: Because the PR runs `loadOpenRouterMeteredUsage` under `!opts.plain`, default `openclaw models status` can send the configured OpenRouter bearer token to OpenRouter even though only `--json` displays billing data. Gate this request to the JSON path.
