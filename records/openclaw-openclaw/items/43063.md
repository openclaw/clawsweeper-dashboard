## Summary

Keep open: current main does not yet expose the requested OpenRouter/provider billing JSON, and this PR is still useful source work, but the PR head has blocking correctness issues and needs maintainer review before merge.

## What This Changes

The PR adds an OpenRouter metered usage fetcher and wires windowed plus metered provider usage into `openclaw models status --json`.

## Best Possible Solution

Rebase the feature onto current `models status` and provider-usage architecture, finalize the public JSON contract, gate OpenRouter metered fetches to JSON output, and add regression coverage for invalid keys and default status latency.

## Reproduction Assessment

Not applicable as a bug reproduction for the feature request. The blocking PR issues are source-reproducible from the head diff: the awaited OpenRouter usage value is typed as the promise return type, and non-2xx OpenRouter responses collapse to `null` instead of provider errors.

## Solution Assessment

No. The feature direction may be useful, but this PR is not the best mergeable form until it is rebased onto current main, keeps OpenRouter billing fetches out of non-JSON status, and reports failed OpenRouter billing/key calls accurately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Store the awaited OpenRouter usage value:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the Promise return type, but this variable stores the awaited result and the JSON branch later reads `.kind`, `.displayName`, and `.keys` from it. This should be `Awaited<ReturnType<typeof loadOpenRouterMeteredUsage>>` or the exported usage type so the touched file type-checks.
  - confidence: 0.94
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This closure runs for every non-plain status command, including the default rich output, but the OpenRouter metered result is only rendered in the JSON payload. Users with OpenRouter credentials will get an extra credential-bearing network request and up to the timeout cost for output that never displays the data; keep OAuth usage for rich status, but only load OpenRouter billing for `opts.json`.
  - confidence: 0.9
- **[P2] Report non-2xx OpenRouter responses as errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: `fetchOpenRouterKeyUsage` turns 401/403/5xx responses into `null`, and the caller only marks errors for thrown exceptions. A revoked key can therefore produce `status: "ok"` with empty or partial `keys`, masking the auth/billing failure in `models status --json`.
  - confidence: 0.9
