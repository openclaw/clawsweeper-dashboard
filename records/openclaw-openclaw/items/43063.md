## Summary

Keep open. Current main still does not expose the requested `auth.providerUsage` JSON billing surface, but this PR is not merge-ready: the latest head is dirty against main, has a type error, performs OpenRouter billing calls outside the JSON-only surface that consumes them, and adds provider-specific billing HTTP/auth logic under core infra instead of the existing provider-owned usage hook.

## What This Changes

The PR adds an `auth.providerUsage` schema to `openclaw models status --json`, maps existing windowed provider usage into it, and adds an OpenRouter metered billing fetcher with tests.

## Best Possible Solution

Rework this as an intentional provider-usage contract change: extend the generic usage snapshot shape if metered usage belongs in core JSON, implement OpenRouter credential resolution and billing parsing inside the OpenRouter plugin via `resolveUsageAuth`/`fetchUsageSnapshot` or a documented additive seam, fetch OpenRouter billing only for `--json`, report partial/non-2xx provider failures accurately, and document the public billing-data schema.

## Reproduction Assessment

Yes. Static inspection is enough to reproduce the blockers: current main lacks `auth.providerUsage`, while the PR diff shows the wrong awaited type, non-JSON billing calls, core-owned OpenRouter billing logic, endpoint mismatch, and silent non-2xx/partial-failure paths.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best implementation until it rebases onto current `models status`, follows the provider-owned usage boundary, fixes the type and endpoint/error handling, and treats billing metadata as a documented public JSON contract.

  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the Promise type, but this variable stores the awaited value and is later read as an object. This should fail typechecking; use `Awaited<ReturnType<...>>` or an explicit snapshot type.

  - body: `openrouterUsage` is only emitted in the JSON payload, but the fetch runs for every non-plain status invocation. Default rich status can now make authenticated OpenRouter billing requests and wait on timeouts even though it never renders the data.
  - confidence: 0.94
- **[P2] Keep OpenRouter billing inside the provider plugin:** `src/infra/provider-usage.openrouter.ts:1-10`
  - body: This adds an OpenRouter-specific billing fetcher under core infra even though the provider-usage hook already lets plugins own provider-specific usage HTTP parsing. Move this behavior into the OpenRouter provider plugin or extend the generic usage seam.
  - confidence: 0.9
- **[P2] Call OpenRouter's documented key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The key-usage fetch requests `/api/v1/auth/key`, while OpenRouter's current documentation lists the current-key endpoint as `/key` under the `/api/v1` server. Valid keys can therefore produce failed or empty key usage in the new JSON payload.
  - confidence: 0.95
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-195`
  - body: A 401, 403, or 500 key response becomes `null`, and the caller only records thrown errors. Invalid or revoked keys can be silently omitted while the provider still reports an apparently healthy status with incomplete data.
  - confidence: 0.91
- **[P2] Mark partial key fetch failures as degraded:** `src/infra/provider-usage.openrouter.ts:272-280`
  - body: When one key throws but another succeeds, `keyErrors` is ignored unless there are zero successful keys. Multi-key setups can therefore return `status: "ok"` while omitting failed profiles from the JSON payload.
  - confidence: 0.89
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-155`
  - body: OpenRouter documents `/credits` as management-key-only, but a non-2xx response is converted to `null` and can still return an `ok` provider when key usage succeeds. That hides real auth or permission failures for account totals.
  - confidence: 0.87
