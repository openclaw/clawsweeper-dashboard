Full review comments:

- **[P1] Store the awaited OpenRouter usage value:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a `Promise`, but this variable is assigned the awaited result and then read as an object. The PR head should fail type checking and the later `kind/status/account/keys` reads are against the wrong type; use an awaited return type or the exported usage type.
  - confidence: 0.95
- **[P1] Gate OpenRouter billing calls to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This runs `loadOpenRouterMeteredUsage` for every non-plain status invocation, but the OpenRouter payload is only consumed in the JSON branch. Default `openclaw models status` can now send a bearer token and wait on the network for data it never displays.
  - confidence: 0.92
- **[P1] Move OpenRouter billing into provider usage hooks:** `src/infra/provider-usage.openrouter.ts:1-5`
  - body: The new core fetcher resolves OpenRouter profiles and `OPENROUTER_API_KEY` directly instead of using the existing provider-owned `resolveUsageAuth`/`fetchUsageSnapshot` path. That duplicates credential handling, ignores the provider ownership boundary, and will drift from OpenRouter plugin config and aliases.
  - confidence: 0.88
- **[P2] Treat non-2xx billing responses as errors:** `src/infra/provider-usage.openrouter.ts:153-154`
  - body: A 401/403 or other non-2xx response from `/credits` or `/auth/key` is collapsed to `null`, and the caller only records thrown errors. Revoked or failing keys can therefore produce `status: "ok"` with missing account/key data instead of an accurate provider error.
  - confidence: 0.9
