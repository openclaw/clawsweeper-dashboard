## Summary

Keep open: current main still does not expose the requested JSON billing data, but this PR is not merge-ready because the latest head fails typecheck, performs an unnecessary OpenRouter credentialed request on default status output, handles OpenRouter billing failures as missing data, and lacks real behavior proof.

## What This Changes

The PR adds an `auth.providerUsage` JSON section to `openclaw models status --json`, plus a core OpenRouter metered billing fetcher and tests.

## Best Possible Solution

Define an additive provider-usage JSON contract, implement OpenRouter metered billing through the provider usage/plugin boundary, gate metered billing fetches to JSON output, surface HTTP and partial failures accurately, and require redacted real command proof.

## Reproduction Assessment

Yes for the PR findings: source inspection and the PR CI log show the typecheck failure, non-JSON OpenRouter fetch, endpoint/error handling problems, and missing proof. This is a feature PR rather than a current-main bug reproduction.

## Solution Assessment

No. The requested capability is useful, but this patch is not the best shape until the schema is reviewed, OpenRouter logic is moved behind the provider usage contract, endpoint/error handling is corrected, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is a Promise type, but this variable stores the awaited result; CI fails `pnpm check` with TS2322 and cascading `never` errors in the provider-usage JSON block. Use `Awaited<ReturnType<...>>` or an explicit resolved type and keep the snapshot variables typed outside the async closures.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This fetch runs for default rich `openclaw models status`, but `openrouterUsage` is only emitted in the JSON payload. Users with an OpenRouter key get an unnecessary credentialed network request and timeout risk for output that never displays the data.
  - confidence: 0.94
- **[P2] Call the documented key-info endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: OpenRouter's current docs say key usage and credit-limit info are fetched with GET `/api/v1/key`; this code calls `/api/v1/auth/key`, so valid keys can be dropped as missing data instead of producing the requested JSON usage. ([openrouter.ai](https://openrouter.ai/docs/api-reference/limits/))
  - confidence: 0.86
- **[P2] Return errors for non-2xx billing responses:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: Both OpenRouter billing helpers turn non-2xx responses into `null`, and the caller treats that as absent data. Revoked keys or a non-management key hitting `/credits` can therefore produce `status: "ok"` with missing account/key data instead of a provider error.
  - confidence: 0.91
- **[P2] Mark partial key failures as errors:** `src/infra/provider-usage.openrouter.ts:279`
  - body: When at least one key succeeds, key-level failures collected in `keyErrors` leave the provider status as `ok` unless the credits call also failed. In a multi-profile setup, a timeout or rejected key can be silently omitted from JSON health.
  - confidence: 0.88
- **[P2] Keep billing behind provider usage hooks:** `src/commands/models/list.status-command.ts:39`
  - body: This adds an OpenRouter-specific core import and command special case even though the current provider contract exposes `resolveUsageAuth` and `fetchUsageSnapshot` for provider-owned usage endpoints. Put the OpenRouter billing parser behind the OpenRouter plugin and extend the generic usage schema there.
  - confidence: 0.82
