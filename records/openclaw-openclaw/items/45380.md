## Summary

Keep open. Current main still has Unix-style inline environment assignments in package scripts and no main-branch helper or npm script-shell setting that would make those entries work in native Windows shells; the PR discussion also shows the earlier spawn-error and signal-exit review concerns were addressed, so this remains a useful implementation candidate rather than obsolete cleanup.

## What This Changes

The PR adds a `scripts/with-env.mjs` helper and rewrites selected env-prefixed package scripts to launch through it so Windows shells can set the intended environment before running dev and live-test commands.

## Best Possible Solution

Land a narrow cross-platform env-wrapper or equivalent Node-owned entrypoint for the remaining env-prefixed package scripts, with focused coverage for env parsing, child spawn failures, signal exit codes, and Windows command/argument behavior. Rebase this PR or replace it with the same small fix against current main; coordinate with #53664 only if maintainers want one broader Windows script cleanup.

## Reproduction Assessment

Yes. The high-confidence reproduction path is native Windows running `pnpm gateway:dev`: current main still starts that script with `OPENCLAW_SKIP_CHANNELS=1`, and the PR body reports the exact Windows shell error before the helper change.

## Solution Assessment

Yes, with maintainer review. A small repo-local Node wrapper is a narrow maintainable fix that avoids adding a dependency, but the branch should be rebased and validated for Windows shell argument behavior because it touches command execution.
