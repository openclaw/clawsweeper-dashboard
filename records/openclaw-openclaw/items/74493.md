## Summary

Keep open. Current main still has the token-inclusive auth-epoch behavior, and this unmerged PR is a narrow active fix for the reported Claude CLI reset path. I found no discrete blocking correctness or security finding, but the token identity contract still needs maintainer acceptance before merge.

## What This Changes

The PR changes CLI auth-epoch hashing for Claude CLI token-shaped credentials and token auth profiles, adds a Claude null-read fallback, bumps the epoch version, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land this PR or an equivalent narrow patch after maintainer/CODEOWNER acceptance, keeping parser hardening or richer Claude account identity extraction as separate follow-up work unless maintainers choose to expand scope.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to return a Claude CLI credential with type token, change only credential.token, and compare resolveCliAuthEpoch on current main; the changed epoch then trips resolveCliSessionReuse with reason auth-epoch for same-version bindings.

## Solution Assessment

Yes, with a maintainer contract decision. Identity-only auth-epoch hashing plus a version bump and focused regression tests is the narrowest maintainable fix for token-rotation resets; parser hardening is a reasonable defense-in-depth follow-up, not required for the reported symptom.
