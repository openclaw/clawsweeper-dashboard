## Summary

Keep open. The PR targets the right auth-epoch bug, but the current diff also stops static-token auth-profile changes from invalidating stored CLI session reuse, which conflicts with the documented auth boundary.

## What This Changes

The PR changes CLI auth-epoch hashing to use identity fields for Claude token-shaped credentials and token auth profiles, bumps the auth-epoch version to 5, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a narrowed auth-epoch fix that keeps Claude OAuth partial reads and token refreshes stable while preserving session invalidation for true static-token changes or token-only account switches.

## Reproduction Assessment

Yes, source-level. Current main can parse partial Claude OAuth credentials as token-shaped credentials, hash rotating token material into the auth epoch, and reset same-version reusable CLI sessions when the hash changes.

## Solution Assessment

No. Identity-only hashing is the right direction for OAuth partial reads, but applying it to all static-token auth profiles is broader than the current contract; preserve token invalidation unless a stable account identity is available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve static-token invalidation:** `src/agents/cli-auth-epoch.ts:112-120`
  - body: At this point token auth-profile epochs no longer include the token value, so a same-profile static token swap with unchanged `provider`, `tokenRef`, `email`, and `displayName` produces the same auth epoch. Current CLI backend docs say changed static tokens must drop stored session reuse, and without a stable account id this can keep a live CLI session alive after a token-only account change. Keep token material in the hash for generic static tokens, or limit identity-only behavior to credentials with a proven stable identity.
  - confidence: 0.82
