## Summary

Keep open: the reported auth-epoch reset bug is valid and this PR is the focused fix path, but the branch still includes a security-relevant policy change that collapses true missing Claude credentials into the same stable epoch as transient null reads without visible maintainer approval.

## What This Changes

The PR changes Claude CLI and token auth-profile auth-epoch hashing to ignore rotating token material, bumps the epoch version to 5, adds a Claude null-read fallback, and updates regression tests plus changelog.

## Best Possible Solution

Keep the identity-only token hashing and version bump, but settle the null-credential policy by distinguishing transient parse failure from true absence or explicitly approving the fallback with targeted coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows partial Claude credentials can parse as token, token encoders include rotating token material, and same-version auth-epoch drift resets reusable CLI sessions; I did not rerun the live macOS keychain race.

## Solution Assessment

No, not fully. Identity-only hashing and the version bump are the narrow maintainable fix, but the synthetic null fallback changes auth-state absence semantics and needs maintainer approval or a safer distinction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Preserve invalidation when Claude credentials are truly absent:** `src/agents/cli-auth-epoch.ts:167-173`
  - body: The new fallback maps every null Claude credential read to the same identity-less OAuth fingerprint. A null read can also mean the local Claude credential was removed or logout left no readable credential, so same-version bindings can keep reusing the previous Claude CLI session instead of invalidating auth-state loss. Please keep true absence as `undefined`, or add an explicit maintainer-approved distinction between transient read failure and real absence.
  - confidence: 0.78
