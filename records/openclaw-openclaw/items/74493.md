## Summary

Keep this PR open. The identity-only token hashing and regression coverage match the reported Claude CLI auth-epoch reset path, but the synthetic null-credential fallback still needs maintainer auth-state policy review because it can preserve a reusable session after true credential absence or logout.

## What This Changes

The PR changes CLI auth-epoch hashing for Claude token credentials and token auth profiles, bumps the auth-epoch version, adds null-read stability coverage, and updates the changelog.

## Best Possible Solution

Keep the identity-only token hashing and version bump, but remove or narrow the null-credential fallback unless maintainers explicitly approve and test that absence policy.

## Reproduction Assessment

Yes. Source inspection on current main shows partial Claude credentials can parse as `type: "token"`, token encoders include rotating token material, and same-version auth-epoch drift resets reusable CLI sessions; the issue also includes real reset logs, but I did not rerun the live macOS keychain race.

## Solution Assessment

No, not quite. Identity-only hashing at the auth-epoch layer is the narrow fix, but the synthetic null fallback changes true credential-absence behavior and should be separated or explicitly approved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve invalidation for true Claude credential absence:** `src/agents/cli-auth-epoch.ts:167-173`
  - body: This fallback maps every null Claude credential read to the same provider-only fingerprint as an identity-less OAuth credential. A null read can also mean the local Claude credential was removed or logout left no readable file, so a same-version binding can keep reusing the old CLI session instead of resetting on auth-state loss. Please keep true absence invalidating, or get explicit maintainer approval and coverage for the new policy.
  - confidence: 0.78
