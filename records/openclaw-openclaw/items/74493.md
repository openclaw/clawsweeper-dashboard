## Summary

Keep this PR open: the identity-only token hashing is the right direction for the reported rotation resets, but the added Claude null-credential fallback still changes the auth-state invalidation policy and should not merge without maintainer approval or a narrower implementation.

## What This Changes

The PR bumps the CLI auth-epoch version, changes Claude CLI and token auth-profile epoch hashing to ignore rotating token material, adds a Claude null-read fallback, expands auth-epoch regression tests, and updates the changelog.

## Best Possible Solution

Keep the identity-only token hashing and epoch-version bump, but preserve true credential absence invalidation or explicitly approve and test a new absent-credential policy.

## Reproduction Assessment

Yes. Source inspection shows partial Claude credentials can parse as `type: "token"`, token encoders on current main include rotating token material, and auth-epoch drift invalidates reusable CLI sessions; I did not rerun the live macOS keychain race.

## Solution Assessment

No, not quite. Identity-only hashing is the narrow fix for token rotation, but the null fallback broadens the behavior to true credential absence without a settled auth-state policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve invalidation when Claude credentials are absent:** `src/agents/cli-auth-epoch.ts:167-173`
  - body: This fallback hashes a synthetic Claude identity whenever the credential read returns null. Null can also mean the user logged out, removed credentials, or the entry is unreadable, so a same-version binding can keep reusing the old Claude session instead of resetting on auth-state loss; please distinguish transient read failure from true absence, or keep absence as `undefined` unless maintainers approve the new policy with coverage.
  - confidence: 0.78
