## Summary

Keep open: the main identity-only token hashing fix is well supported by source evidence, proof, and green checks, but the PR still makes true null Claude credential reads indistinguishable from a valid identity-less credential, which can mask credential removal/logout without an explicit maintainer-approved policy.

## What This Changes

The PR changes CLI auth-epoch hashing for Claude token credentials and auth-profile token credentials so token rotation and partial keychain reads do not reset reusable CLI sessions.

## Best Possible Solution

Keep the identity-only token hashing and epoch-version bump, but either preserve invalidation for true missing Claude credentials or explicitly approve and test the new absence policy.

## Reproduction Assessment

Yes. Source inspection on current main shows partial Claude credentials can parse as `type: "token"`, token encoders include rotating token material, and same-version auth-epoch drift resets reusable CLI sessions; I did not rerun the live macOS keychain race.

## Solution Assessment

No, not quite. Identity-only hashing and the version bump are the narrow maintainable fix, but the synthetic null fallback should not collapse true credential absence into the same hash without a deliberate policy and coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve invalidation when Claude credentials are absent:** `src/agents/cli-auth-epoch.ts:167-173`
  - body: This fallback maps every null Claude credential read to the same fingerprint as an identity-less OAuth credential. A null read can also mean the local Claude credential was removed or logout left no readable file, so a same-version binding can keep reusing the old Claude session instead of resetting on auth-state loss; please distinguish transient read failure from true absence, or keep absence as `undefined` unless maintainers approve the new policy with coverage.
  - confidence: 0.78
