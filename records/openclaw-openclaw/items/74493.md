## Summary

Keep open. The PR targets a real Claude CLI auth-epoch bug and supplies credible terminal proof, but the latest head still weakens the generic token auth-profile boundary by making same-profile static token swaps invisible to session reuse, which conflicts with the current documented contract.

## What This Changes

The branch bumps the CLI auth epoch version, changes Claude token-shaped credential and token auth-profile hashing to ignore token material, adds auth-epoch regression tests, and updates the changelog.

## Best Possible Solution

Land a narrowed fix that keeps Claude local token-shaped credential epochs stable across partial OAuth reads, preserves static-token auth-profile invalidation, and adds focused regression coverage for both boundaries.

## Reproduction Assessment

Yes, source-level. Current main can parse a partial Claude OAuth credential as token-shaped, hash rotating token material into the auth epoch, and then invalidate same-version reusable sessions when the epoch changes.

## Solution Assessment

No. The Claude-specific encoder change is appropriate, but applying identity-only hashing to every token auth profile is broader than the documented contract and should be narrowed or moved to the Claude parser/sync boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve static-token profile invalidation:** `src/agents/cli-auth-epoch.ts:117-123`
  - body: Dropping `credential.token` from every token auth-profile epoch lets a same-profile static token swap keep reusing the existing CLI session when provider, tokenRef, email, and displayName are unchanged. Current CLI backend docs explicitly include changed static tokens as an auth identity change that drops stored session reuse, and the Claude external CLI sync path already ignores token-shaped Claude reads, so keep token material in this generic branch or handle the Claude partial-read case at a Claude-specific boundary.
  - confidence: 0.86
