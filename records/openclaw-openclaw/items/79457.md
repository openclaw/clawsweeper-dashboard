## Summary

Keep open: current main still has a source-level Feishu group context mismatch that can produce the reported groupId warning, and the user-facing group reply behavior is documented as supported.

## What This Changes

The issue requests fixing Feishu group-chat @mention handling so a bundled Feishu group session replies instead of dropping a caller-provided groupId and sending no final reply.

## Best Possible Solution

Make Feishu group inbound contexts expose the chat/session group identity to group resolution and tool policy while preserving sender identity in `SenderId`, `SenderName`, and agent-facing text, then cover the @mention reply path with a regression test.

## Reproduction Assessment

Yes at source level: a Feishu group message uses a group route session key but sets `From` to the sender id, so the shared group resolver can pass a sender-derived groupId that the trusted policy drops. I did not run a live Feishu group, so the no-reply symptom still needs end-to-end proof after a fix.

## Solution Assessment

Yes, the narrow maintainable fix is in the Feishu/group-context boundary: derive group policy from trusted session or chat id metadata, not model/caller-supplied groupId, and keep duplicate plugin cleanup as a separate setup action.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79457. On current main, Feishu group inbound dispatch builds `From` from the sender (`feishu:<open_id>`) while `ChatType` is `group`; `resolveGroupSessionKey` then derives a sender-shaped groupId, and effective tool policy drops it because the trusted session key is `agent:<agent>:feishu:group:<chat_id>`. Keep the fix inside the Feishu/group-context boundary: preserve speaker identity through `SenderId`, `SenderName`, and agent-facing body text, but make group resolution/tool policy receive the trusted Feishu chat/session group id. Do not trust model/caller-provided groupId, do not weaken `resolveTrustedGroupId`, and do not make old `openclaw-lark` coexistence a requirement. Add regression coverage for a Feishu group @mention/open group dispatch that does not emit/drop a sender-derived groupId and still records/routes the group session. Add a user-facing changelog fix entry unless the final change is tests-only.
