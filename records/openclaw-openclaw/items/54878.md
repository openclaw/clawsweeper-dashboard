## Summary

Keep open. Current main still checks the target `agent:` id before honoring spawned-session lineage, so a parent-owned ACP child with a different agent id is denied in default `tree` visibility; the original `visibility=all` allowlist repro also still requires the child harness id to match `tools.agentToAgent.allow`. The prior fix PRs were closed unmerged, and the remaining work is a real session-visibility access-control fix rather than cleanup.

## What This Changes

The issue requests that a parent agent be able to read its own spawned ACP child session history even when the child session uses a different `agent:` id, without adding the ACP harness id to the generic cross-agent allowlist.

## Best Possible Solution

Implement a narrow lineage-aware session visibility rule in `src/plugin-sdk/session-visibility.ts`: proven spawned children of the requester should be allowed before generic cross-agent rejection, including ACP child keys with different agent ids and the `visibility=all` partial-allowlist case. Add guard-level and `sessions_history` regression tests, then audit `session_status` so its extra cross-agent gate follows the same parent-owned child rule without broadening unrelated A2A access.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: with requester `agent:main:main`, target `agent:codex:acp:<id>`, `tree` visibility, and a spawned list containing that target, the guard returns the cross-agent visibility error before checking `spawnedKeys`; with `visibility=all` and `allow: ["main"]`, the A2A policy still rejects `codex`.

## Solution Assessment

Yes, the best fix direction is the narrow lineage rule described by the issue and prior PRs: allow only verified spawned parent/child access before generic cross-agent checks. A broad `tools.agentToAgent.allow` workaround is not the right solution because it grants peer cross-agent access beyond the parent-owned child case.
