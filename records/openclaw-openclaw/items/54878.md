## Summary

Keep open. Current main still rejects cross-agent ACP child session keys before honoring spawned-session lineage, and the prior fix PRs linked from the discussion were closed unmerged.

## What This Changes

The issue asks for parent agents to read their own spawned ACP child session history without adding the child harness agent id to the broad agent-to-agent allowlist.

## Best Possible Solution

Implement a narrow lineage-aware session visibility rule in the shared guard so verified spawned children are readable by their parent before generic cross-agent denial, with regression coverage for `sessions_history` and an audit of `session_status`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: requester `agent:main:main`, target `agent:codex:acp:<uuid>`, `tree` visibility, and a spawned list containing that target still hit the cross-agent denial before the spawned-key check.

## Solution Assessment

Yes. The narrow maintainable fix is to allow only verified parent-owned spawned children before generic cross-agent checks; adding the harness id to `tools.agentToAgent.allow` is broader than the documented tree visibility model requires.
