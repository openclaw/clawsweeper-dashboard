## Summary

Keep this PR open: current main still lacks the proposed thread-aware resolver and memory guidance, but the branch is not ready to merge because it has source-reproducible behavior/safety findings and no real behavior proof.

## What This Changes

The PR adds auto-reply deictic-context and chat-shaping prompt helpers, wires them into reply prompt construction, updates memory-core recall guidance and tests, and adjusts a Discord wildcard test import.

## Best Possible Solution

Land a revised thread-aware follow-up implementation that keeps nearby context primary, narrows triggers, labels context as untrusted, handles channels/thread starters/Slack metadata, and validates the touched lanes with real proof.

## Reproduction Assessment

Yes for the PR review findings: they are reproducible by source inspection of pure predicates and prompt assembly on the PR head, plus current Slack thread-context formatting. I did not run tests because this review is read-only and test commands can write caches/artifacts.

## Solution Assessment

No. The direction is useful, but this patch is not yet the best maintainable fix because it over-applies shaping, misses shared-chat and thread-starter edge cases, can select transport metadata as the referent, and needs a clearer untrusted prompt boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate punctuation shaping on short follow-ups:** `src/auto-reply/reply/chat-shaping.ts:25`
  - body: `shouldApplyChatShaping` returns true for any message ending in `?` or `!` after only broad deny-lists and a 1200-character cap. Detailed non-follow-up questions can therefore receive persona chat-shaping guidance, which is broader than this PR's stated scope.
  - confidence: 0.86
- **[P2] Pass a shared-chat flag for channels:** `src/auto-reply/reply/get-reply-run.ts:547`
  - body: The new call passes `isGroupChat`, but that flag is only true for `ChatType === "group"`. Channel sessions bypass the no-nearby-context guard, so shared channel messages can get shaping where group chats would not.
  - confidence: 0.78
- **[P2] Honor thread starters in the group shaping guard:** `src/auto-reply/reply/chat-shaping.ts:66`
  - body: For group chats, this early return ignores `threadStarterBody` even though the later nearby-context check includes it. First replies in a thread with only starter context will skip shaping despite having nearby context.
  - confidence: 0.84
- **[P2] Select semantic thread text before metadata:** `src/auto-reply/reply/deictic-context.ts:109`
  - body: `pushCandidate` keeps the last non-empty line, but Slack thread history appends a `[slack message id: ...]` metadata line after each message. The resolver can present that ID line as the referent instead of the message content.
  - confidence: 0.9
- **[P2] Require nearby context for bare pronoun deictics:** `src/auto-reply/reply/deictic-context.ts:26`
  - body: The direct-reference pattern matches ordinary short sentences starting with `it`, `this`, or `that` even when no reply or thread context exists. That can inject a resolver note for plain updates and bias the reply toward unnecessary ambiguity handling.
  - confidence: 0.84
- **[P3] Remove the unbounded json matcher:** `src/auto-reply/reply/chat-shaping.ts:13`
  - body: The extra `/json/u` matcher duplicates the word-boundary JSON check but also matches substrings like `jsonl` or `jsonify`, suppressing chat shaping for messages that are not structured JSON-output requests.
  - confidence: 0.8
