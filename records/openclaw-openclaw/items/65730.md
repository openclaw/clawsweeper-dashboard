## What This Changes

The PR adds auto-reply deictic-context and chat-shaping prompt helpers, wires them into reply prompt construction, updates memory-core recall guidance, adds targeted tests, and adjusts a Discord component test import.

## Best Possible Solution

Land a revised, rebased thread-aware follow-up implementation that keeps nearby reply/thread context primary, narrows triggers to true short follow-ups, preserves current group/channel behavior, labels all channel-derived candidates as untrusted, and covers Slack metadata, thread-starter, and false-positive cases with focused tests.

## Reproduction Assessment

Yes. ## Solution Assessment

No. The direction is plausible, but this patch is not the best form yet because it over-applies shaping, misses or regresses shared-chat edge cases, can select transport metadata as the referent, and needs a clearer untrusted-context boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Gate punctuation-only shaping on short follow-ups:** `src/auto-reply/reply/chat-shaping.ts:25`
  - body: The punctuation signal makes any message ending in `?` or `!` apply chat shaping after only the 1200-character cap. Detailed questions that are not short conversational follow-ups can therefore receive persona/style guidance outside the PR's stated scope.
  - confidence: 0.86
- **[P2] Treat channels as group context for shaping:** `src/auto-reply/reply/get-reply-run.ts:547`
  - body: This branch passes `isGroupChat` into the new helper, but the branch still computes that flag as only `ChatType === "group"`. Channel sessions can therefore bypass the shared-chat no-nearby-context guard, while current main already treats group and channel as shared-chat contexts.
  - confidence: 0.78
- **[P2] Honor thread starters in the group shaping guard:** `src/auto-reply/reply/chat-shaping.ts:66`
  - body: For group sessions, this early return ignores `threadStarterBody`, even though the later nearby-context check includes it. First replies in a thread that only have starter context will skip chat shaping despite having the nearby context the helper is meant to use.
  - confidence: 0.82
- **[P2] Select semantic thread text before metadata:** `src/auto-reply/reply/deictic-context.ts:109`
  - body: `pushCandidate` always keeps the last non-empty line, but Slack thread history appends a `[slack message id: ...]` metadata line after each message. The resolver can present that ID line as the referent instead of the message content.
  - confidence: 0.9
- **[P2] Require nearby context for bare pronoun deictics:** `src/auto-reply/reply/deictic-context.ts:26`
  - body: The first English direct-reference pattern matches ordinary short sentences starting with `it`, `this`, or `that` even when no reply or thread context exists. Messages like `It still fails after reboot.` will get a resolver note with no nearby candidates, biasing the reply toward unnecessary ambiguity handling.
  - confidence: 0.84
- **[P3] Remove the unbounded JSON matcher:** `src/auto-reply/reply/chat-shaping.ts:13`
  - body: The extra `/json/u` matcher duplicates the word-boundary JSON check but also matches substrings like `jsonl` and `jsonify`, which can suppress chat shaping for messages that are not actually structured-output requests.
  - confidence: 0.8
