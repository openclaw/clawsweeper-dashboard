## Summary

Keep open. Current main and v2026.6.10 still have useful session lineage and run-control APIs, but not the requested Control UI orchestration surface: no mission/orchestration route, Sessions remains a flat paginated table, and selected-row bulk actions only delete/archive session records.

## What This Changes

Request to add a scalable Control UI orchestration surface with agent/subagent hierarchy, active-first filters, bulk operations, per-row status metadata, and large-session-count rendering.

## Best Possible Solution

Keep this as the umbrella Control UI subagent-orchestration request and split implementation into maintainer-reviewed slices: read-only scalable hierarchy first, then privileged bulk actions after semantics are chosen.

## Reproduction Assessment

Yes for the source-level feature gap: current main exposes session lineage and run metadata but the browser UI still renders flat paginated session rows with delete-only bulk selection. No runtime bug reproduction is applicable because no existing UI contract promises the requested orchestration board.

## Solution Assessment

Unclear: the requested direction is coherent, but the best implementation depends on maintainer product choice. A read-only hierarchy over existing metadata is the safest first slice; bulk run controls should wait for explicit safety and scope semantics.
