## What This Changes

The PR passes `model.id` into the DeepSeek DSML text filter and tool-call recoverer gate, enabling both when the model id contains `deepseek`.

## Best Possible Solution

Use a normalized model-id predicate that honors explicit non-DeepSeek compat overrides, add focused proxy and override coverage, and require redacted Feishu/proxy proof before merge.

## Reproduction Assessment

Yes, at source level: current main only creates the DSML filter/recoverer when `compat.thinkingFormat` resolves to `deepseek`, so custom/proxy DeepSeek routes with default non-DeepSeek compat do not strip DSML. I did not reproduce the Feishu feedback loop live.

## Solution Assessment

No, not yet: the response-side DSML gate is a plausible owner path, but this patch should honor explicit non-DeepSeek compat, normalize model ids, add coverage, and include real Feishu/proxy proof.
