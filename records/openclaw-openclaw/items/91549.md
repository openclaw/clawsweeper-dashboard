## Summary

Keep open: the PR targets a plausible DSML leak path, but it is not merge-ready because after-fix real behavior proof is missing and the new model-id fallback is case-sensitive, so known uppercase DeepSeek proxy ids can still miss the filter.

## What This Changes

The PR passes `model.id` into `shouldFilterDeepSeekDsmlText` and enables DeepSeek DSML filtering/tool-call recovery when the model id contains lowercase `deepseek`.

## Best Possible Solution

Keep the response-side DSML filter, but gate it through a normalized, compat-respecting DeepSeek predicate with focused regression coverage and redacted real Feishu/proxy proof before merge.

## Reproduction Assessment

Yes, source inspection shows the current-main gate only enables DSML filtering when `compat.thinkingFormat` is `deepseek`; a custom/proxy model without that compat value would not create the filter. I did not reproduce the Feishu feedback loop in a live setup.

## Solution Assessment

No, the current patch is a plausible mitigation but not the best fix yet because it uses a case-sensitive substring check and does not clearly preserve compat ownership. A normalized, bounded predicate with regression tests is safer.
