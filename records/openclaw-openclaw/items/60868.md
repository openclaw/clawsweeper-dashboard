## Summary

Keep this PR open: the chain-aware overflow fix is useful and not implemented on current main, but the latest head still has a blocking raw-error redaction regression in embedded overflow diagnostics and lacks external contributor real-behavior proof.

## What This Changes

The branch adds a shared `.error`/`.cause` message collector for agent context-overflow classification, applies it in model fallback, embedded overflow compaction, and auto-reply failure handling, and includes focused tests plus small Discord, Gateway, MiniMax, and changed-lane edits.

## Best Possible Solution

Keep the shared chain-candidate approach, but keep raw nested messages classifier-only, surface only redacted display/log text, preserve billing/rate-limit precedence, and add a regression covering custom `.error` overflow messages that include secrets.

## Reproduction Assessment

Yes, at source level. On PR head, a `promptError` whose custom `.error` branch contains an `Authorization: Bearer ...` token plus an overflow phrase is collected with `{ redact: false }`, selected as `errorText`, and logged in overflow diagnostics.

## Solution Assessment

No, not yet. Chain-wide classification is the right direction, but the implementation must separate raw classifier candidates from redacted diagnostic or user-visible text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact raw chain messages before logging overflow text:** `src/agents/pi-embedded-runner/run.ts:1296`
  - body: `collectErrorChainMessages(promptError, { redact: false })` can provide the selected `errorText`, and that value is later logged in overflow diagnostics. Custom `.error` branches are not covered by the first formatted candidate, so a nested bearer token plus an overflow phrase can leak; keep raw candidates for predicate checks and log only redacted text.
  - confidence: 0.92
