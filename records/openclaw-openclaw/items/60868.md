## Summary

Keep open: current main still does not implement chain-wide context-overflow detection for these runtime paths, but the latest PR head still has a concrete redaction/security blocker and lacks required real behavior proof.

## What This Changes

The branch adds a shared `.error`/`.cause` chain message collector and applies chain-aware context-overflow classification to model fallback, embedded overflow compaction, and auto-reply failure handling, with focused tests and small adjacent CI/plugin/channel edits.

## Best Possible Solution

Revise the branch so raw chain candidates are used only for predicates, redacted counterparts are used for logs/user-visible text, and focused regressions cover nested `.error` overflow messages containing secrets.

## Reproduction Assessment

Yes, at source level on the latest PR head. A `promptError` with outer `request failed` and nested `.error.message` containing `Authorization: Bearer ... context length exceeded` is selected from the raw candidate list and logged as overflow diagnostic text.

## Solution Assessment

No, not yet. Chain-wide classification is the right direction, but the implementation must keep raw nested messages classifier-only and log or surface only redacted text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact chain messages before logging overflow diagnostics:** `src/agents/pi-embedded-runner/run.ts:1296`
  - body: `collectErrorChainMessages(promptError, { redact: false })` feeds `errorTexts`, and the matching value becomes `contextOverflowError.text` before being logged as `error=...`. A transport `.error.message` like `Authorization: Bearer ... context length exceeded` is not covered by the first `formatErrorMessage(promptError)` candidate, so the raw nested secret can still leak; use raw candidates only for matching and store/log a redacted display value.
  - confidence: 0.92
