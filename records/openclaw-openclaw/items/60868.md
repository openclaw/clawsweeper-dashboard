## Summary

Keep this PR open. The missing chain-aware context-overflow detection is still not implemented on current main, but the latest PR head still has a blocking security regression because raw nested error-chain text can become the embedded-runner overflow diagnostic text and be logged.

## What This Changes

The PR adds error-chain message collection for failover, model fallback, embedded overflow compaction, and auto-reply classification, with related tests plus small Discord/Gateway system-event trust, MiniMax test ordering, channel status typing, and changed-lanes coverage updates.

## Best Possible Solution

Revise the branch so raw chain candidates are used only for internal classification, while diagnostics, logs, stored failure text, Control UI output, and user-facing replies use formatErrorMessage(), default redacted collectErrorChainMessages(), or an equivalent redacted display path. Keep the chain-wide overflow detection and billing/rate-limit precedence tests, and split unrelated churn if maintainers prefer a narrower PR.

## Reproduction Assessment

Yes. A wrapped prompt error such as Error("request failed", { cause: Error("Authorization: Bearer ... context length exceeded") }) reaches collectErrorChainMessages(promptError, { redact: false }) in the PR and the matching raw segment can be logged as overflow error text.

## Solution Assessment

No. The product direction is right, but this is not the best implementation until raw messages stay classifier-only and all surfaced text is redacted; the safer fix is to match against raw or redacted candidates while returning/logging only redacted display text.
