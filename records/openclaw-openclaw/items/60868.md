## Summary

Keep open: current main still lacks the PR's chain-wide context-overflow detection, but the latest PR head has a blocking security regression in the embedded-runner diagnostic path and no contributor-supplied real behavior proof.

## What This Changes

The PR adds a shared error-chain message collector, applies chain-aware context-overflow classification in agent fallback/runtime reply paths, and includes related tests plus small Discord/Gateway trusted-event and changed-lane updates.

## Best Possible Solution

Revise the PR so raw chain candidates are classifier-only while every surfaced diagnostic, log, reply, and metadata string uses redacted display text; retain focused chain-wide tests and split unrelated churn if maintainers prefer a narrower branch.

## Reproduction Assessment

Yes, at source level. A promptError shaped with outer "request failed" and nested .error.message containing an Authorization bearer token plus "context length exceeded" reaches the PR's raw candidate path and can become the surfaced overflow errorText; no tests were executed in this read-only review.

## Solution Assessment

No, not yet. The direction is right, but the implementation still lets raw nested chain messages feed surfaced overflow diagnostics; use paired raw/redacted candidates or an equivalent helper that keeps raw strings internal to classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep raw chain messages out of overflow diagnostics:** `src/agents/pi-embedded-runner/run.ts:1296`
  - body: `collectErrorChainMessages(promptError, { redact: false })` feeds the selected overflow `errorText`, which is then logged and stored in run metadata. A nested provider or transport error containing an Authorization header, API key, or request metadata plus an overflow phrase can bypass `formatErrorMessage()` redaction, so raw candidates should stay classifier-only and surfaced text should be redacted.
