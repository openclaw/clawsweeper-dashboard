## Summary

Keep this PR open. The chain-aware overflow direction is useful and not implemented on current main, but the latest head still has a blocking secret-redaction issue in embedded-runner overflow diagnostics and no external real behavior proof.

## What This Changes

The PR adds a shared error-chain message collector and applies chain-aware context-overflow classification in model fallback, embedded overflow compaction, and auto-reply failure handling, with focused tests plus small Discord/Gateway/changed-lane edits.

## Best Possible Solution

Keep the shared chain-candidate approach, but keep raw candidates classifier-only and surface only redacted display text while preserving billing/rate-limit precedence and requiring contributor after-fix proof.

## Reproduction Assessment

Yes, at source level. On PR head, a promptError with a nested .error or .cause message containing both a bearer token and a context-overflow phrase is collected with { redact: false }, selected as errorText, and logged in overflow diagnostics.

## Solution Assessment

No, not yet. The direction is maintainable, but raw chain messages need to stay internal to classification and the surfaced overflow text should come from redacted candidates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact chain messages before surfacing overflow text:** `src/agents/pi-embedded-runner/run.ts:1296-1302`
  - body: `collectErrorChainMessages(promptError, { redact: false })` can supply the selected `errorText`, and that same value is later logged in overflow diagnostics. A nested provider error containing `Authorization: Bearer ...` plus an overflow phrase bypasses `formatErrorMessage()` redaction here, so keep raw candidates classifier-only and return redacted text for diagnostics.
  - confidence: 0.92
