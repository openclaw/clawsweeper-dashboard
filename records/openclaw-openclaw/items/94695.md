## Summary

Keep open for maintainer architecture, product, and security review: current main and v2026.6.9 still use Gateway HTTP/WebSocket with Tailscale/SSH/Bonjour-style access paths, while this proposal would add a new P2P transport, native packaging, relay/key policy, config surface, and cross-device state sync.

## What This Changes

Request to adopt Iroh as a core P2P/QUIC transport plus CRDT-backed cross-device workspace, session, skill, and memory sync for the Gateway and companion nodes.

## Best Possible Solution

Keep this as a maintainer-owned RFC, then split any accepted direction into a narrow Gateway transport prototype and a separate security-reviewed state-sync design aligned with existing pairing/auth, config, packaging, and SQLite migration rules.

## Reproduction Assessment

Not applicable: this is an architectural feature proposal, not a bug report with a current-main failure path. Source and docs inspection shows the requested Iroh/P2P transport and CRDT sync layer are absent from current main and v2026.6.9.

## Solution Assessment

Unclear: Iroh is a plausible RFC direction, but the issue does not establish it as the best OpenClaw solution. Maintainers first need to decide transport ownership, relay/key trust, native packaging, config defaults, compatibility, and whether state sync belongs in core or a separate plugin/API design.
