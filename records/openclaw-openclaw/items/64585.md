## Summary

Keep open. Current main has shipped partial recovery for stale locks and live diagnostic stalls, but it still does not implement the requested startup scanner that selects ordinary persisted chat sessions by transcript tail and replays them through `sessions.send`; the remaining channel-backed replay policy needs maintainer judgment.

## What This Changes

The issue requests bounded gateway startup recovery that scans persisted chat sessions after restart and replays unanswered sessions through the gateway send path.

## Best Possible Solution

Settle the channel-backed replay policy, then add a bounded startup recovery module that selects eligible chat sessions newest-first, skips internal routes, replays through `sessions.send`, and covers selection plus replay with focused tests.

## Reproduction Assessment

Yes, at source level: an ordinary persisted `running` chat session without `abortedLastRun` is skipped before transcript inspection, and assistant-tail sessions remain blocked by the current resumable-tail classifier. I did not run a live gateway restart reproduction in this read-only pass.

## Solution Assessment

Unclear. A bounded scanner through `sessions.send` is plausible, but automatic replay for channel-backed sessions needs maintainer-approved safety policy before this becomes a narrow implementation task.
