## Summary

Keep open. Current main has partial recovery paths, but it does not implement the requested startup scan that detects unanswered transcript tails and replays them through chat delivery; that is still a product-sensitive recovery policy decision.

## What This Changes

Add gateway startup recovery that scans persisted chat sessions after restart and replays sessions whose transcript tail appears unanswered or invisible.

## Best Possible Solution

Define an owner-approved bounded startup replay policy, then implement it as a narrow gateway recovery module that reuses existing session readers and chat delivery seams without changing the current conservative lock-marked restart recovery semantics.

## Reproduction Assessment

Yes, at source level. A running session with an unanswered transcript tail but no `abortedLastRun` marker will not be selected by current restart recovery, and the test suite explicitly locks that boundary.

## Solution Assessment

Unclear as a policy. The requested transcript-tail replay is a plausible direction, but maintainers should first decide idempotency, channel eligibility, caps, and stale-message semantics before implementation.
