## Summary

Keep open. Current main still has only pending-delivery recovery plus lock-marker-based restart recovery; it does not scan ordinary persisted chat sessions by transcript tail or replay them through `sessions.send`, and the related channel-replay PR shows the remaining policy is not settled enough for automatic cleanup.

## What This Changes

The issue requests bounded gateway startup recovery that scans persisted chat sessions after restart and replays unanswered or invisible-tail conversations through the normal `sessions.send` path.

## Best Possible Solution

Settle the channel-backed replay policy, then add a bounded startup recovery module that selects eligible chat sessions newest-first, skips internal routes, replays through `sessions.send`, and covers selection and replay with focused tests.

## Reproduction Assessment

Yes, at source level: an ordinary persisted `running` chat session without `abortedLastRun` is skipped before transcript inspection, and a marked empty-assistant-tail session is failed rather than replayed. I did not run a live gateway restart reproduction in this read-only review.

## Solution Assessment

Unclear. A bounded scanner through `sessions.send` is plausible, but automatic replay for channel-backed sessions needs maintainer-approved safety policy first because related work is trying to prevent hidden channel replay.
