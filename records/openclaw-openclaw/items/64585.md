## Summary

Keep open. Current main has adjacent delivery recovery, diagnostic stale-lane release, and lock-marked main-session restart recovery, but it still does not implement the requested startup scan of persisted chat session stores by transcript tail or replay eligible conversations through `sessions.send`.

## What This Changes

Request gateway startup recovery that scans persisted chat sessions after restart and replays unanswered transcript-tail candidates through the existing `sessions.send` pipeline with caps and non-chat/internal skips.

## Best Possible Solution

Add a bounded gateway startup recovery module that runs after request context and channel sidecars are ready, scans persisted chat session stores newest-first, selects safe unanswered transcript-tail candidates, skips internal/non-chat sessions, and replays through `sessions.send` with idempotency and summary logging while leaving diagnostic stale-lane release and lock-marked restart recovery as separate safeguards.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a persisted ordinary `running` session without `abortedLastRun` is skipped, and the existing unit test at `src/agents/main-session-restart-recovery.test.ts:160` asserts that behavior without needing a live gateway.

## Solution Assessment

Yes, with the issue's safety constraints. A focused startup scanner using the existing `sessions.send` pipeline is the narrow maintainable direction; broad diagnostic abort behavior or hidden synthetic `agent` replay would be riskier.

Add a bounded module under `src/gateway/` that runs after request context and channel sidecars are ready, scans persisted gateway chat session stores newest-first, detects unanswered transcript tails where the last meaningful message is a user message or an assistant message with no visible text and no tool call, skips non-chat/internal routes (`global`, `unknown`, `cron`, internal, subagent/acp), applies a per-startup cap around 10, and logs scanned/candidates/replayed/failed.
