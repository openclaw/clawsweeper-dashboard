## Summary

Keep open. Current main still lacks the requested startup scanner: restart recovery only handles sessions marked from stale transcript locks, does not select ordinary persisted chat sessions by transcript tail, and does not replay through `sessions.send`; the related open channel-replay safety PR also shows this needs maintainer policy before implementation.

## What This Changes

The issue requests bounded gateway startup recovery that detects persisted stuck chat sessions by transcript tail and replays eligible conversations through the existing session send path.

## Best Possible Solution

Settle the channel-backed replay policy, then add a bounded startup recovery module that selects eligible chat sessions newest-first, skips internal/non-chat routes, replays through `sessions.send`, and covers selection/replay with focused tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: an ordinary persisted `running` chat session without `abortedLastRun` is skipped before transcript inspection, and a marked empty-assistant-tail session is failed rather than replayed. I did not run a live gateway restart reproduction in this read-only review.

## Solution Assessment

Unclear. The bounded scanner through `sessions.send` is plausible, but channel-backed automatic replay needs maintainer policy first because current related work is trying to prevent hidden replay for externally delivered sessions.
