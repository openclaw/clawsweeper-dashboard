## Summary

Keep open: the CJK table half still reaches code-unit width/padding on current main, while the paragraph-break half is a separate Telegram formatting policy question rather than an implemented or duplicated fix. The useful related discussion is already captured by https://github.com/openclaw/openclaw/issues/55512 and the open draft implementation candidate at https://github.com/openclaw/openclaw/pull/55596, but those do not cover the paragraph policy request.

## What This Changes

The issue requests better Telegram agent-message formatting by adding paragraph breaks for long prose and aligning CJK Markdown tables correctly.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/55596 for display-width-aware code tables, then decide the Telegram prose softbreak policy separately in the formatter or prompt surface.

## Reproduction Assessment

Partially yes: the CJK table half is source-reproducible because Telegram code-mode tables reach `renderTableAsCode`, where current main measures and pads with `.length`. No high-confidence current-main repro contract exists for the paragraph half because tests and docs do not define single-newline promotion as expected Telegram behavior.

## Solution Assessment

Unclear for the whole issue. Shared display-width measurement is the narrow fix for CJK tables, but gateway- or formatter-level single-newline promotion needs a maintainer policy choice before implementation.
