## Summary

Keep open: the PR targets a real restart-loss gap, but the latest patch still persists broad sensitive runtime state, does not fully resume cold restored queues without a later callback registration, and the posted proof does not show end-to-end Telegram delivery after a mid-turn restart.

## What This Changes

The PR adds disk-backed JSON persistence for followup queues, restores them at queue module initialization, sweeps restored keys when drain callbacks are registered, and adds persistence tests.

## Best Possible Solution

Persist a minimal private recovery descriptor, restore it through an explicit cold-start delivery path, and prove the real Telegram restart flow with redacted logs or a short recording.

## Reproduction Assessment

Yes. Source inspection shows current main stores both followup queues and drain callbacks in process-global maps, so a full gateway restart clears the in-memory recovery path; I did not run a live Telegram restart repro in this read-only review.

## Solution Assessment

No. The current PR is not the best solution yet because it persists full runtime state and still depends on a later callback registration for cold restored queues; a narrower recovery record plus explicit startup/cold-session drain path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Persist only the recovery record:** `src/auto-reply/reply/queue/persist.ts:99`
  - body: `toPersistedRun()` stores `item.run` directly, and the queue entry also stores `lastRun`, which includes `OpenClawConfig` plus sender and execution context. That config can contain API keys, channel tokens, plugin config, and local-service env, so this new JSON state file should persist only the minimal fields needed to reconstruct a followup through the normal runtime.
  - confidence: 0.9
- **[P2] Resume restored queues without another turn:** `src/auto-reply/reply/queue/drain.ts:41-45`
  - body: The pending-drain sweep only runs when `rememberFollowupDrainCallback()` is called, but after a cold restart callbacks are not registered during channel startup. If the gateway restores a queued Telegram followup and no later same-route message schedules a turn, the item survives on disk but still never dispatches.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch adds an on-disk state file that can contain full runtime config and queued message data, so the persistence shape and file privacy need tightening before merge.
