## Summary

Keep this PR open for maintainer review. The patch now appears coherent at the queue layer and I found no discrete blocking code finding, but it intentionally adds at-rest prompt/routing state and still lacks real transport proof for the user-visible restart/redelivery path.

## What This Changes

The PR adds a JSON persistence/restore layer for followup queues, wires persistence into enqueue/drain/clear/session-refresh mutations, and adds queue restart and on-disk round-trip tests.

## Best Possible Solution

Land a single maintainer-approved recovery contract that preserves queued followups exactly once across restart, documents the local state-file privacy boundary, and leaves proactive cold-session replay to the approved followup API if that remains separate.

## Reproduction Assessment

Yes, at source level: current main stores FOLLOWUP_QUEUES only in a process-global map and has no state-file restore path, so a process restart loses pending followups. I did not run a destructive gateway restart in this read-only review.

## Solution Assessment

Unclear as a final product solution: the queue-local persistence and acknowledgement shape is plausible, but callback-gated restart delivery plus the new at-rest data boundary need maintainer approval against the broader followup API work.
