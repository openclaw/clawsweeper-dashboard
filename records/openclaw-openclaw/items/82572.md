## Summary

Keep open. The persistence direction is useful and current main still lacks durable follow-up queue restore, but the PR head is currently conflicting and still routes restored queues through a targeted heartbeat path that can skip valid agents outside the routine heartbeat map, leaving some persisted follow-ups stranded.

## What This Changes

The PR adds SQLite-backed follow-up queue persistence, legacy JSON sidecar doctor migration, restore-time run rehydration, and gateway-start heartbeat wakes, plus focused queue/gateway/heartbeat tests and a small unrelated Anthropic CI-contract adjustment.

## Best Possible Solution

Rebase onto current main, preserve the current queue routing/context fields, keep the shared SQLite plus doctor migration design, and add a recovery-only heartbeat target or cold-session drain path for restored follow-ups before merge.

## Reproduction Assessment

Yes. Source inspection shows a restored queue requests a targeted followup-queue-restore heartbeat, while the heartbeat runner returns disabled when that session's agent is not in the routine heartbeat map.

## Solution Assessment

No, not yet. Shared SQLite with a doctor migration is the right storage boundary, but the current startup wake path needs a recovery-only target or cold-session drain path before it is the best fix.
