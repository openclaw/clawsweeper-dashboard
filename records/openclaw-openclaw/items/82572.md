## Summary

Keep open: the PR targets a real source-reproducible restart-loss bug, but the current patch still serializes live runtime/config state, writes the recovery file without private permissions, and does not prove live Telegram mid-turn restart delivery.

## What This Changes

The PR adds a JSON persistence layer for followup queues, restore/drain hooks, and focused queue persistence tests.

## Best Possible Solution

Land one maintainer-approved durable restart recovery path that stores a minimal private recovery descriptor, rebuilds through typed untrusted followup/dispatch contracts, and proves live Telegram restart delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted followups live in process-local maps and callbacks, so a full process restart loses or strands pending work without durable recovery.

## Solution Assessment

No. Persisting pending work is the right direction, but serializing the live run object and waiting for a later callback registration is not the narrow maintainable recovery contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Persist only a recovery descriptor:** `src/auto-reply/reply/queue/persist.ts:101`
  - body: `toPersistedRun()` writes `run: item.run`, and the queue entry also writes `lastRun`; current main fills that object with `OpenClawConfig`, sender identity, owner numbers, and execution settings. Persist only the minimal IDs/routing needed to rebuild through normal session/config loading, otherwise this new state file can store secrets and stale runtime policy.
  - confidence: 0.91
- **[P2] Write the queue state file privately:** `src/auto-reply/reply/queue/persist.ts:141-144`
  - body: The new state file is written with plain `fs.writeFileSync`, so prompts and routing metadata inherit the user's umask. Use the existing private state-file pattern with an explicit `0o600` mode, and preferably an atomic temp-file replace.
  - confidence: 0.86
- **[P2] Recover restored queues without another inbound message:** `src/auto-reply/reply/queue/drain.ts:36-45`
  - body: The restored-key sweep only runs when `rememberFollowupDrainCallback()` is called, but a cold gateway restart starts with no callbacks and no startup registration. That means a restored Telegram followup can remain idle until the user sends another same-route message, so the original restart-loss behavior is not fully fixed.
  - confidence: 0.84

## Security Review
