- `P1`: The PR targets real message loss across gateway restarts but currently has security-sensitive persistence and delivery-completeness blockers.
- `merge-risk: 🚨 security-boundary`: Merging as-is would persist full runtime config and identity/auth fields into a durable state file.

## Summary

Keep this PR open, but not merge-ready: the branch addresses a real followup-queue durability gap, while still persisting live runtime/config state and relying on later callback registration for restored delivery. The supplied proof is useful but does not yet show the real Telegram mid-turn restart and post-restart delivery path.

## What This Changes

This PR adds JSON persistence and restoration for followup queues, hooks persistence into queue mutations, moves queue state typing, and adds persistence plus restart-drain tests.

## Best Possible Solution

Land a maintainer-approved recovery path that persists only minimal queue descriptors, rebuilds through session/config/runtime loading, schedules restored work without relying on another user message, and includes live Telegram restart-delivery proof.

## Reproduction Assessment

Yes, from source inspection: current main keeps followup queues and callbacks in process-local maps, so a full gateway restart drops or strands pending queue state. I did not run live Telegram because this review is read-only, but the code path is clear.

## Solution Assessment

No. Persisting queued followups is the right direction, but this patch persists too much live runtime state and does not provide a complete cold-restart delivery contract; the safer path is a minimal descriptor rebuilt through session/config loading.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Persist only a minimal recovery descriptor:** `src/auto-reply/reply/queue/persist.ts:102`
  - body: `toPersistedRun()` stores `run: item.run`, and the queue entry also stores `lastRun`; that run object includes the active config snapshot, sender identity, owner numbers, auth/profile selection, and exec/runtime policy. Persist only minimal route/session/prompt/media metadata and rebuild through normal session/config loading, otherwise the durable queue file can leak secrets and replay stale authorization or provider policy after restart.
  - confidence: 0.93
- **[P2] Do not strand restored queues until another inbound message:** `src/auto-reply/reply/queue/drain.ts:41-45`
  - body: The restored-key sweep only runs when `rememberFollowupDrainCallback()` is called, but after a cold gateway restart there is no per-route callback until another same-route inbound message registers one. That means the queued Telegram followup can survive on disk but remain idle, so restart recovery is still incomplete for the original lost-message case.
  - confidence: 0.88
