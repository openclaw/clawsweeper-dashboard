## Summary

## What This Changes

The PR adds JSON persistence, restore-on-startup, and mutation hooks for auto-reply followup queues so queued runs can survive gateway restarts.

## Best Possible Solution

Land this only as a maintainer-approved phase-one persistence layer after live Telegram proof or an explicit proof override, while leaving proactive cold-session dispatch to the broader followup API work.

## Reproduction Assessment

Yes at source level: current main stores followup queues only in memory, so process exit loses pending entries. I did not establish a full live Telegram mid-turn restart reproduction in this read-only review.

## Solution Assessment

Mostly, for a phase-one durability layer: the patch is scoped to the existing queue path and avoids a broader cold-session dispatcher. If maintainers require immediate startup replay without a new inbound callback, the better solution is the followup API/gateway-restart design in https://github.com/openclaw/openclaw/pull/63330.
