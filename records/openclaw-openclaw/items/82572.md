## Summary

Keep open: this is a useful durability direction, but the current patch is not merge-ready because restored runs can lose auth-profile selection, delivered items can be replayed after a crash, and the supplied proof still stops short of a full Telegram mid-turn restart with exactly-once redelivery.

## What This Changes

The PR adds JSON persistence for followup queues, restores queued runs on startup, schedules restored drains when callbacks register, persists queue mutations, and adds queue persistence/restart tests.

## Best Possible Solution

Preserve or reconstruct auth-profile selection, persist delivery acknowledgements as items drain, and either implement cold-start followup dispatch or explicitly stage that work behind the linked followup API before merging with live Telegram exact-once proof.

## Reproduction Assessment

Yes at source level: current main stores followup queues in process-local global maps, so process exit drops pending entries. I did not run a live Telegram restart reproduction in this read-only review.

## Solution Assessment

No, not yet: JSON persistence is a plausible staged durability layer, but this implementation still loses auth-profile state, can replay stale delivered items, and depends on callback registration rather than a cold-start dispatcher.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve restored auth-profile selection:** `src/auto-reply/reply/queue/persist.ts:53-54`
  - body: Restored runs drop `authProfileId` and `authProfileIdSource`, but the runner still selects credentials from those run fields. A queued turn that originally selected a non-default profile can therefore resume after restart under the default or wrong profile; persist the safe ID/source or recompute them from session state before dispatch, with regression coverage.
  - confidence: 0.91
- **[P1] Persist acknowledgements as items drain:** `src/auto-reply/reply/queue/drain.ts:485-488`
  - body: The state file is updated only from the drain `finally`, so after `drainNextQueueItem` delivers and shifts an item, the on-disk queue can remain stale until the whole drain exits. If the gateway crashes during that window, restart restores and re-delivers prompts that already ran; persist after each successful dequeue/batch removal or add an explicit ack boundary before moving to the next item.
  - confidence: 0.87

## Risks / Open Questions

- Restored queued turns can run under the wrong or default auth profile because the persisted descriptor drops authProfileId/authProfileIdSource and the runner still resolves credentials from those run fields.
- A crash or forced restart after a queued prompt is successfully delivered but before drain finally persists the shortened queue can replay an already-delivered prompt after restart.
- Cold-start delivery still depends on a route callback being registered later; without the broader followup API work, a restored queue may wait for another inbound message on that route.
- The supplied proof exercises persistence and seeded restart mechanics, but not a real Telegram mid-turn queue, gateway restart, and exactly-once redelivery path.
- The state file now durably stores prompts plus routing/session descriptors; 0o600 and the minimal descriptor mitigate this, but maintainers still need to accept that local security boundary.
