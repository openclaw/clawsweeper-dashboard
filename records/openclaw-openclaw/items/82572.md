## Summary

Keep open: the PR addresses a real source-reproducible restart message-loss path, but restored media followups can lose the one-turn model/auth intent and the supplied proof still does not show a live Telegram mid-turn restart redelivery.

## What This Changes

The PR adds a durable JSON persistence layer for followup queues, restores queue state at module startup, reschedules restored drains when callbacks register, and adds queue persistence tests.

## Best Possible Solution

Land a canonical durable followup queue design that persists only minimal private recovery descriptors, preserves or safely rederives every field needed for routing, media model, auth, and fallback intent, and proves live Telegram restart redelivery.

## Reproduction Assessment

Yes, source inspection shows current main stores followup queues and drain callbacks in process-local maps, so queued followups do not survive a process restart; I did not run a live Telegram restart repro in this read-only review.

## Solution Assessment

Not yet: the persistence direction is plausible, but the descriptor drops media model/auth intent and the proof does not yet demonstrate the real Telegram restart-delivery path.
