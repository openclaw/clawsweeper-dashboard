- **[P1] Share restored pending keys across split chunks:** `src/auto-reply/reply/queue/persist.ts:30`
  - body: The restored pending-drain keys are kept in a module-local Set, but this queue path already uses process-global storage for FOLLOWUP_QUEUES and FOLLOWUP_RUN_CALLBACKS because bundled chunks can evaluate separate module copies. If restore adds a key in one persist.ts copy and drain.ts sweeps another copy, rememberFollowupDrainCallback sees an empty set and the restored queue never drains after restart. Store this set on direct globalThis Symbol.for state and cover a split-module restore/drain case.
  - confidence: 0.88
- **[P1] Reset restore-once state before exercising restore:** `src/auto-reply/reply/queue.drain-restart.test.ts:313`
  - body: state.ts calls restoreFollowupQueues at module evaluation, so by the time this test sets OPENCLAW_STATE_DIR the process-wide restored flag is already true. The explicit restore call returns without reading the seeded file, the callback never drains the item, and the PR's checks-node-auto-reply-reply-state-routing job times out here. Clear the restore-once flag or isolate the simulated restart before calling restoreFollowupQueues.
  - confidence: 0.94
