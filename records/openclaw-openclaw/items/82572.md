## Summary

This PR is addressing a real message-loss/session-state problem, but it is not merge-ready: the patch still persists the live followup run object, including secret-capable config and policy state, and the cold-restart delivery path still depends on a later route callback rather than proving actual Telegram redelivery after restart.

## What This Changes

The branch adds JSON persistence and restore scheduling for followup queues, wires persistence into queue mutation paths, and adds queue persistence/restart tests.

## Best Possible Solution

Persist a minimal private recovery descriptor, rebuild run state from the current session/config at drain time, schedule restored work without waiting for a new user message, and prove live Telegram restart redelivery.

## Reproduction Assessment

Yes. Source inspection on current main shows followup queue contents and drain callbacks are process-local maps, so a full gateway restart loses pending followup state.

## Solution Assessment

No. Durable queue recovery is the right direction, but persisting live runtime objects and relying on later callback registration is not the safest recovery contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Persist a recovery descriptor instead of the live run:** `src/auto-reply/reply/queue/persist.ts:102`
  - body: `toPersistedRun()` writes `run: item.run`, and the queue entry also persists `lastRun`. That run contains `config`, owner/contact fields, provider/model/auth choices, policy overrides, and other runtime state; persisting it can leak secrets and replay stale policy after restart. Store only minimal queue/session/routing IDs and rebuild through the normal session/config/auth loaders.
  - confidence: 0.92
- **[P2] Do not require another inbound message to drain restored queues:** `src/auto-reply/reply/queue/drain.ts:36-45`
  - body: The restored-key sweep only runs when `rememberFollowupDrainCallback()` is called. After a cold gateway restart the callback map is empty, and current call sites register callbacks during later enqueue/active message handling, so the preserved followup can remain idle until the user sends another same-route message. Rebuild/register the runner from persisted session state or otherwise schedule restored work on startup.
  - confidence: 0.86
