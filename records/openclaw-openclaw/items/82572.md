## Summary

Keep open: the persistence direction is reasonable, but the current startup recovery wake is still not classified as a heartbeat wake payload, so restored queues can remain stuck for agents with empty or comment-only HEARTBEAT.md files.

## What This Changes

The PR persists followup queues in shared SQLite, migrates the legacy JSON sidecar, restores queued followups on startup, and schedules gateway recovery wakes so restored routes can drain after restart.

## Best Possible Solution

Keep the SQLite persistence and migration approach, but classify followup-queue-restore as a wake payload or use the existing wake reason, add a regression through heartbeat preflight, and then capture redacted real startup-recovery proof.

## Reproduction Assessment

Yes. Current main stores followup queues only in the process-global FOLLOWUP_QUEUES map and enqueue/drain have no persisted restore path, so a process restart loses queued items from source inspection.

No. SQLite persistence is the right general direction, but the current startup wake does not enter the heartbeat wake-payload path and can be skipped before restored queues drain.
