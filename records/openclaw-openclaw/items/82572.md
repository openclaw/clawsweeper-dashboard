## Summary

Keep open: the SQLite persistence direction is useful, but the startup recovery path still depends on heartbeat eligibility, so restored followups can remain undelivered for non-heartbeat-enabled agent routes.

## What This Changes

The PR persists followup queues in shared SQLite, restores them after gateway restart, migrates the legacy JSON sidecar via doctor, and wakes restored routes through the heartbeat runner.

## Best Possible Solution

Keep the SQLite persistence work, but make restored startup delivery independent of heartbeat eligibility or explicitly prove and document the supported heartbeat-gated contract before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: restore a queued followup for an agent that is not in the heartbeat runner's active agent set, then the PR's requestHeartbeat recovery returns disabled and no drain callback is registered.

## Solution Assessment

No; SQLite persistence is the right direction, but using heartbeat as the only startup wake mechanism is not the best fix unless maintainers intentionally limit recovery to heartbeat-enabled agents.
