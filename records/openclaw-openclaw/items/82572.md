## Summary

The branch has a concrete P1 recovery defect: it requests a targeted wake for every restored session, but the reviewed head rejects valid agents omitted from routine heartbeat scheduling, so their queued messages can remain undelivered. The SQLite and doctor-migration direction is sound, but the final-head recording also does not demonstrate the central restart-survival flow.

## What This Changes

The PR persists follow-up queues in shared SQLite, migrates legacy JSON sidecars through doctor, rehydrates queued execution context, and requests gateway-start recovery wakes for restored sessions.

## Best Possible Solution

Keep the shared-SQLite and doctor-migration design, but make `followup-queue-restore` resolve a safe transient target for valid session agents outside routine heartbeat scheduling without enrolling them in interval heartbeats, then demonstrate exactly-once Telegram redelivery on the repaired final head.

## Reproduction Assessment

Yes. A valid restored session whose agent is omitted from routine heartbeat entries follows a deterministic source path to `state.agents.get(...) === undefined` and returns `disabled` before the recovery turn can run; current main's original restart loss is likewise clear because its queue is process-local.

## Solution Assessment

No, not yet. Shared SQLite with doctor-owned migration is the narrowest repository-aligned storage solution, but using a routine-heartbeat membership map as the gate for universal restored-session recovery leaves a supported session class unhandled.
