## Summary

Keep open: the PR addresses a real followup queue message-loss problem, but the current patch still has blocking security and restart-delivery issues, and the supplied proof does not show a real queued Telegram followup delivered after a gateway restart.

## What This Changes

The PR adds JSON persistence for followup queues, restores queued work on module load, schedules restored drains when callbacks register, and adds persistence/restart tests.

## Best Possible Solution

Persist a minimal private recovery descriptor, rebuild queued work through typed session/config loading, write the state atomically with owner-only permissions, schedule restored work without requiring another inbound message, and prove live Telegram restart delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows followup queues and callbacks are process-local, so a full gateway restart loses pending in-memory followup work before this PR's durable layer.

## Solution Assessment

No. Durable queue recovery is the right direction, but this implementation persists secret-capable runtime objects and still relies on later callback registration instead of a minimal startup recovery contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Persist only a minimal recovery descriptor:** `src/auto-reply/reply/queue/persist.ts:101`
  - body: `toPersistedRun()` stores `run: item.run`, and the queue entry also stores `lastRun`; that run object includes `config`, sender identity, owner numbers, auth/model/runtime settings, and other policy-bearing state. Persist a minimal descriptor and rebuild through normal session/config loading, otherwise this durable file can leak secrets and replay stale runtime policy.
  - confidence: 0.92
- **[P2] Schedule restored queues without another inbound message:** `src/auto-reply/reply/queue/drain.ts:41-45`
  - body: The restored-key sweep only runs inside `rememberFollowupDrainCallback()`, but after a cold restart no callback exists until another same-route message registers one. A queued followup can survive on disk but remain idle indefinitely, so restart recovery still fails without user activity.
  - confidence: 0.87
- **[P2] Write queue state privately and atomically:** `src/auto-reply/reply/queue/persist.ts:141-144`
  - body: The new state write uses plain `fs.writeFileSync`, so queued prompts and routing data inherit the process umask and a crash can leave partial JSON. Use the existing private atomic state-file pattern with `0o600` and temp-file replacement before persisting this recovery file.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces durable storage for secret-capable runtime state and writes it without explicit private atomic file handling.

Concerns:

- **[high] Avoid persisting live run config:** `src/auto-reply/reply/queue/persist.ts:101`
  - body: `run: item.run` and `lastRun` can serialize `FollowupRun.run.config`, which is an OpenClawConfig snapshot containing inline env vars and token/password/api-key capable provider or channel fields.
  - confidence: 0.92
- **[medium] Use private permissions for queue state:** `src/auto-reply/reply/queue/persist.ts:141`
  - body: The recovery file is written without an explicit private mode or atomic replacement, leaving queued prompts and route metadata readable according to the process umask and vulnerable to partial writes.
  - confidence: 0.86
