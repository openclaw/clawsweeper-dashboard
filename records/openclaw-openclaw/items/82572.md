## Summary

Keep open; the PR targets a useful durability gap, but the patch is not mergeable because restored queues do not actually resume after a restart, queued media is stripped, the added test fixture is type-invalid, real behavior proof is missing, and the persistence format is too broad from a privacy/security standpoint.

## What This Changes

The PR adds JSON persistence for `FOLLOWUP_QUEUES` under the OpenClaw state directory, with save/restore hooks in enqueue, drain, clear, refresh, and new persistence tests.

## Best Possible Solution

Persist a minimal serializable queue record with prompts, routing, attachments, and run identity, reload current config on startup, then schedule recovery through the existing followup runner with focused regression and live restart proof.

## Reproduction Assessment

Yes, source-reproducible: current main keeps followup queues and drain callbacks in process memory, so a gateway restart loses the active recovery path. I did not run a live restart proof in this read-only review.

## Solution Assessment

No, the proposed patch is not the best current solution because it restores queue data without a runnable startup drain path, drops attachments, and persists overly broad run/config state. A narrower recovery record plus explicit startup scheduling would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Schedule restored queues after startup:** `src/auto-reply/reply/queue/persist.ts:157`
  - body: `restoreFollowupQueues()` repopulates `FOLLOWUP_QUEUES` with `draining: false`, but after a real process restart `FOLLOWUP_RUN_CALLBACKS` is empty and this code never calls `scheduleFollowupDrain()`. A message queued before a crash can therefore sit restored but idle until some later turn happens to create a new callback, which still fails the restart recovery this PR is meant to add.
  - confidence: 0.87
- **[P2] Persist queued media fields:** `src/auto-reply/reply/queue/persist.ts:22-37`
  - body: The persisted run shape omits `images` and `imageOrder`, but current queued runs populate those fields from inbound media and the drain/followup runner passes them to the agent. After a restart, image-bearing followups would be restored as text-only prompts, silently dropping the attachments.
  - confidence: 0.9
- **[P2] Add required run fields to the test fixture:** `src/auto-reply/reply/queue/persist.test.ts:24-32`
  - body: `makeRun()` is declared as `FollowupRun["run"]`, but the object returned here omits required `timeoutMs` and `blockReplyBreak`. The new test file will fail the test-type lane before it can validate persistence behavior.
  - confidence: 0.93
