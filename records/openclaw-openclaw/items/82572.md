## Summary

Keep open. The PR addresses a real restart/session-state gap, but the patch is not merge-ready because it persists secret-capable live runtime state, writes the recovery file without private atomic handling, can still leave restored work idle until another inbound message, and lacks full live Telegram restart-delivery proof.

## What This Changes

The PR adds JSON persistence and restore scheduling for followup queues, plus focused queue persistence and restart-drain tests.

## Best Possible Solution

Land one maintainer-approved recovery path that persists a minimal private descriptor, rebuilds followups through typed session/config loading, schedules restored work without another inbound message, and proves live Telegram restart delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows followup queues and their callbacks are process-local, so a full gateway restart loses or strands pending followup work before this PR's durable layer.

## Solution Assessment

No. Persisting queued work is the right direction, but serializing live runtime/config objects and waiting for a later callback registration is not the narrow durable recovery contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Persist only a minimal recovery descriptor:** `src/auto-reply/reply/queue/persist.ts:101`
  - body: `toPersistedRun()` stores `run: item.run`, and the queue entry also stores `lastRun`; that run object carries `config`, sender identity, owner numbers, auth/model/runtime settings, and other policy-bearing state. Persist only minimal IDs/routing needed to rebuild through normal session/config loading, otherwise the new durable queue file can leak secrets and replay stale runtime policy.
  - confidence: 0.91
- **[P2] Do not strand restored queues until another message:** `src/auto-reply/reply/queue/drain.ts:36-45`
  - body: The restored-key sweep only runs from `rememberFollowupDrainCallback()`, and after a cold gateway restart there is no callback until another same-route inbound message registers one. That means the original queued Telegram followup can survive on disk but remain idle indefinitely, so restart recovery is still incomplete.
  - confidence: 0.86
- **[P2] Write the queue state file privately and atomically:** `src/auto-reply/reply/queue/persist.ts:141-144`
  - body: The new state write uses plain `fs.writeFileSync`, so queued prompts/routing data inherit the process umask and a crash can leave partial JSON. Use the existing private atomic state-file pattern, with `0o600` and a temp-file replace, before persisting this recovery file.
  - confidence: 0.86

## Security Review

Status: needs_attention
