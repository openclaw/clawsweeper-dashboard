## Summary

Keep open. ## What This Changes

The PR persists followup queues in shared SQLite, migrates the legacy JSON sidecar through doctor, restores queued followups on startup, and schedules heartbeat recovery for restored sessions.

## Best Possible Solution

Keep the SQLite persistence and doctor migration direction, but classify restored followup recovery as a heartbeat wake payload, add focused heartbeat-runner regression coverage, then require current-head Telegram or Mantis proof before merge.

## Reproduction Assessment

## Solution Assessment

No. Persisting followup queues in shared SQLite with doctor migration is the right ownership direction, but the startup recovery wake must be integrated with heartbeat wake-payload classification before this is the best mergeable fix.
