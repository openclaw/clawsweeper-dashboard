## Summary

Keep open: the PR targets a real source-reproducible message-loss gap, but the restored-drain trigger can bypass the existing active-run idle guard, and the supplied proof still stops short of live Telegram exactly-once restart delivery.

## What This Changes

The PR adds JSON persistence, restore-on-startup, drain rescheduling, and regression tests for followup queue state across gateway restarts.

## Best Possible Solution

Keep the minimal private state descriptor, but move restored-queue draining onto a path that knows the active run is idle, then land only after live Telegram restart proof or an explicit maintainer decision to defer cold-session startup dispatch to https://github.com/openclaw/openclaw/pull/63330.

## Reproduction Assessment

Yes at source level: current main keeps followup queues only in a process-local global map, so a gateway process restart clears queued items. I did not run a live Telegram mid-turn restart reproduction in this read-only review.

## Solution Assessment

No: the persistence layer is a reasonable base, but scheduling restored drains from callback registration is not the safest integration point because enqueue-time callbacks run before the active-run idle decision. The better path is to trigger restored drains from the existing post-run scheduler or a dedicated followup API that can reconstruct a cold-session runner safely.
