## Summary

Keep open: this is a plausible durability feature, but the current PR has blocking correctness, security, import-cycle, test, and real-behavior-proof gaps before it can be considered for merge.

## What This Changes

The PR adds JSON persistence for followup queues, restores them at queue module load, hooks persistence into enqueue/drain/state mutations, and adds persistence tests.

## Best Possible Solution

Design a minimal private durable queue with a startup drain/reconstruction handoff, preserve supported serializable payloads, and pass import-cycle, type, lint, and focused queue tests before merge.

## Reproduction Assessment

Yes: source inspection gives a high-confidence path, because current main keeps followup queues and drain callbacks only in process-global maps. A queued followup that exists only in those maps will disappear when the gateway process exits before draining.

## Solution Assessment

No: the proposed direction is useful, but this patch restores data without a startup drain path, omits existing queued media fields, writes overly broad state, and currently fails CI gates. A narrower durable queue contract with private storage and explicit recovery is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid persisting full runtime config:** `src/auto-reply/reply/queue/persist.ts:68`
  - body: `toPersistedRun` stores `run: item.run`, which includes `run.config`. That runtime config can contain gateway tokens, channel credentials, provider keys, env values, and other sensitive operator data, and the new JSON write does not use the repo's private-state/fs-safe helpers. Persist only the minimal non-secret execution locator/state needed for recovery, or use the existing secret/private state path with an explicit security contract.
  - confidence: 0.88
- **[P1] Restart restored queues after gateway startup:** `src/auto-reply/reply/queue/state.ts:180`
  - body: `restoreFollowupQueues()` only repopulates `FOLLOWUP_QUEUES`; after a process restart the drain callback map is empty and nothing schedules `scheduleFollowupDrain` until later activity for the same key. In a quiet chat, the recovered message can sit indefinitely, so the advertised restart recovery still does not deliver the queued followup.
  - confidence: 0.86
- **[P2] Break the queue persistence import cycle:** `src/auto-reply/reply/queue/state.ts:4`
  - body: Importing `persist.ts` from `state.ts` while `persist.ts` imports state values creates a runtime value cycle. The PR's policy guard already reports `persist.ts -> state.ts -> persist.ts`; move the restore hook or persistence access behind an acyclic module boundary.
  - confidence: 0.96
- **[P2] Persist queued media fields:** `src/auto-reply/reply/queue/persist.ts:25-38`
  - body: `PersistedFollowupRun` leaves out `images` and `imageOrder`, but current queue draining and followup execution preserve those fields for queued multimodal turns. After a restart, a queued image prompt would be delivered without its images, changing existing queue behavior.
  - confidence: 0.84
- **[P2] Complete the test run fixture:** `src/auto-reply/reply/queue/persist.test.ts:26-34`
  - body: The new `makeRun()` fixture returns a `FollowupRun["run"]` without required `timeoutMs` and `blockReplyBreak`, which fails `check:test-types` on the PR head. Add the required fields or reuse the existing queue test helper fixture.
  - confidence: 0.98
- **[P2] Isolate the empty-queue deletion test:** `src/auto-reply/reply/queue/persist.test.ts:93-94`
  - body: The deletion test assumes the global followup queue map is empty, but CI shows `persistFollowupQueues()` leaves the state file present. Clear or isolate all global queue state before this assertion so the test does not depend on other tests in the shard.
  - confidence: 0.9
