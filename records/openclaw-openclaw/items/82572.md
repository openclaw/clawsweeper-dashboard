## Summary

Keep open: the PR targets a real restart-loss gap, but the current patch is not mergeable because restored queues have no startup drain path, queued media is dropped, the new test fixture does not satisfy current run types, broad run/config state is persisted, and contributor real behavior proof is missing.

## What This Changes

The PR adds JSON persistence for `FOLLOWUP_QUEUES`, restore-on-module-load behavior, save hooks in enqueue/drain/clear/refresh, and persistence tests.

## Best Possible Solution

Define one durable recovery contract that persists a minimal serializable queue record, preserves routing and attachments, reloads live config, replays through typed untrusted dispatch/followup paths, and only consumes recovery data after processing is durably scheduled.

## Reproduction Assessment

Yes, source-reproducible: current main keeps followup queues and drain callbacks in process-local maps, and gateway shutdown clears run state without a durable pending-followup replay path. I did not run a live restart proof in this read-only review.

## Solution Assessment

No, this is not the best current solution because restoring `FOLLOWUP_QUEUES` alone does not recreate startup drain machinery, drops media fields, and persists overly broad stale run/config state. A narrower recovery record plus explicit untrusted replay or drain scheduling would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Schedule restored queues after startup:** `src/auto-reply/reply/queue/persist.ts:157`
  - body: `restoreFollowupQueues()` repopulates `FOLLOWUP_QUEUES` with `draining: false`, but after a real process restart `FOLLOWUP_RUN_CALLBACKS` is empty and this code never calls `scheduleFollowupDrain()`. A message queued before a crash can therefore sit restored but idle until unrelated later traffic creates a new callback, so the restart recovery still fails.
  - confidence: 0.87
- **[P2] Persist queued media fields:** `src/auto-reply/reply/queue/persist.ts:22-37`
  - body: The persisted run shape omits `images` and `imageOrder`, but current queued runs populate those fields from inbound media and the drain/followup runner passes them to the agent. After a restart, image-bearing followups would be restored as text-only prompts, silently dropping attachments.
  - confidence: 0.9
- **[P2] Add required run fields to the test fixture:** `src/auto-reply/reply/queue/persist.test.ts:24-32`
  - body: `makeRun()` is declared as `FollowupRun['run']`, but the object returned here omits required `timeoutMs` and `blockReplyBreak`. The new test file will fail the test-type lane before it can validate persistence behavior.
  - confidence: 0.93
