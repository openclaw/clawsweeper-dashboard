## Summary

Keep open: the PR targets a real source-reproducible queue-loss gap, but the current branch still has blocking recovery and persistence-privacy issues, and the latest proof does not show a real queued Telegram followup delivered after restart.

## What This Changes

The PR adds JSON persistence and restore hooks for followup queue state, plus tests for queue serialization, restore, cleanup, and pending-drain tracking.

## Best Possible Solution

Persist a minimal recovery descriptor with an explicit privacy and file-permission contract, then restore through a startup/cold-session drain path that is reconciled with the followup runtime API work.

## Reproduction Assessment

Yes, source-reproducible: current main keeps followup queues and drain callbacks in process-global maps, so a full gateway process restart clears both. I did not run live Telegram restart proof in this read-only review.

## Solution Assessment

No, the current PR is not the best solution yet because it persists too much sensitive runtime state and restored queues still rely on later callback registration. A narrower persisted descriptor plus an explicit cold-start drain/recovery seam is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Persist only minimal queue recovery data:** `src/auto-reply/reply/queue/persist.ts:99`
  - body: `toPersistedRun()` stores `item.run` verbatim, and the queue entry also stores `lastRun`. That run contains `OpenClawConfig`, including env vars, provider keys/headers, channel tokens, and local-service env, so the new JSON state file can leak sensitive runtime configuration; persist only the identifiers and prompt/routing fields needed to rehydrate safely.
  - confidence: 0.9
- **[P2] Drain restored queues without waiting for later traffic:** `src/auto-reply/reply/queue/drain.ts:36-45`
  - body: The only new cold-restart scheduling hook runs inside `rememberFollowupDrainCallback()`. After a full gateway restart no callback exists for a restored queue until a later inbound turn reaches that route, so a queued Telegram message can remain pending indefinitely if the user does not send another message.
  - confidence: 0.84
- **[P2] Remove the new queue import cycle:** `src/auto-reply/reply/queue/state.ts:4`
  - body: `state.ts` imports `persist.ts`, while `persist.ts` imports live queue state/defaults from `state.ts`. This creates a static cycle in the queue modules and risks failing the repository import-cycle gate; move restore wiring to a non-cyclic bootstrap path or invert the dependency.
  - confidence: 0.78
