## Summary

Keep open: the PR targets a real source-reproducible restart gap, but the current patch still leaves restored queues idle after a cold restart and introduces a sensitive persistence surface by writing full run/config state.

## What This Changes

The PR adds a disk-backed JSON persistence module for followup queues and hooks enqueue, drain, clear, and session-refresh mutations to save and restore queue state.

## Best Possible Solution

Persist a minimal serializable recovery descriptor, restore it through an explicit cold-start delivery path, define the privacy contract for prompts/media/routing state, and prove it with focused tests plus redacted live Telegram restart output.

## Reproduction Assessment

Yes, source-reproducible: current main keeps followup queues and drain callbacks in process-global maps, so a full gateway process restart clears both. I did not run live Telegram restart proof in this read-only review.

## Solution Assessment

No, the current implementation is not the best fix yet because restored entries remain idle until later traffic and the persisted shape is too broad. A narrower recovery record plus a startup-registered drain/cold-session runner would be more maintainable.
