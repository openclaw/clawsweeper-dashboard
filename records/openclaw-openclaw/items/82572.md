## Summary

Keep this PR open, but it is not merge-ready: the durable followup queue direction is useful, yet the new startup recovery wake is still not classified as a heartbeat wake payload, so common empty/comment-only HEARTBEAT.md setups can skip the run before restored queues drain.

The PR persists followup queue entries in shared SQLite, adds a legacy sidecar doctor migration, restores queues on startup, and asks gateway heartbeat recovery to wake restored routes.

## Best Possible Solution

Classify followup-queue-restore as a heartbeat wake payload, add the empty/comment-only HEARTBEAT.md regression plus focused queue recovery coverage, then merge only with live or Mantis proof of restart recovery on the fixed head.

## Solution Assessment

No. Persisting followups in SQLite is the right general direction, but the startup drain mechanism is incomplete until the new wake source is classified and proven against the heartbeat gate.
