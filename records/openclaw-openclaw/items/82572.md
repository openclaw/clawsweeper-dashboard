## Summary

Keep open: the queue-loss problem is real from source inspection, but this patch still persists live runtime state, writes the new state file without private permissions, and does not yet prove or fully implement cold-restart Telegram delivery without another inbound message.

## What This Changes

The PR adds JSON persistence, restore hooks, and tests for followup queue entries across gateway restarts.

## Best Possible Solution

Persist a minimal private recovery descriptor, rebuild runtime state through the normal session/config path, and prove live Telegram mid-turn restart recovery with redacted logs or a short recording.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main keeps both followup queues and drain callbacks in process-global maps, so a full process restart clears them. I did not run a live current-main Telegram restart in this read-only review.

## Solution Assessment

No. The PR is aimed at the right queue surface, but the maintainable fix should avoid serializing the live run/config object, use private durable writes, and add a real cold-start delivery path instead of waiting for a later inbound route callback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Persist only a recovery descriptor:** `src/auto-reply/reply/queue/persist.ts:101`
  - body: `toPersistedRun()` stores `item.run` directly, and the queue entry also stores `lastRun`; current main fills that object with `OpenClawConfig`, sender identity, owner numbers, and exec settings. This can persist API keys, channel tokens, env vars, and private context into a new JSON state file, so the file should contain only the minimal IDs needed to rebuild the followup through normal session/config loading.
  - confidence: 0.9
- **[P2] Create the queue state file privately:** `src/auto-reply/reply/queue/persist.ts:141-144`
  - body: The new state file is written with plain `fs.writeFileSync`, so it inherits the process umask; the posted proof even shows a group/world-readable mode. Queue prompts and routing metadata should be written atomically with private permissions, ideally through the existing private-file pattern.
  - confidence: 0.86
- **[P2] Drain restored queues without another inbound message:** `src/auto-reply/reply/queue/drain.ts:41-45`
  - body: The pending-restored sweep only runs when `rememberFollowupDrainCallback()` is called. After a cold gateway restart those callbacks are empty and current channel startup does not register them, so a restored Telegram followup can sit idle until another same-route inbound message arrives instead of recovering the message that was already queued before restart.
  - confidence: 0.82
