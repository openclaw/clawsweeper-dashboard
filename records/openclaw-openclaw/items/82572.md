## Summary

Keep open: the restart-loss problem is real, but this PR still stores live secret-capable runtime state, writes it without private mode, and does not rebuild cold-start drain execution; overlapping recovery designs also need maintainer selection.

## What This Changes

The PR adds a JSON state file plus enqueue, drain, clear, and restore hooks to persist followup queue entries across gateway restarts.

## Best Possible Solution

Land one maintainer-approved durable restart recovery path that stores a minimal private descriptor, rebuilds through session/config loading and typed untrusted followup or dispatch contracts, schedules recovery without later user traffic, and proves live Telegram restart delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted followups and drain callbacks live only in process-local maps, so a full gateway restart drops or strands pending work; I did not run a live gateway restart in this read-only pass.

## Solution Assessment

No. Persisting pending work is the right direction, but this PR persists live runtime/config objects, lacks private file semantics, and still depends on later callback registration instead of rebuilding a cold-start recovery runner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Persist a minimal recovery descriptor:** `src/auto-reply/reply/queue/persist.ts:101`
  - body: `toPersistedRun()` writes `item.run`, and the queue entry also writes `lastRun`. That object includes `OpenClawConfig`, sender identity, owner numbers, and execution policy, so the new state file can store secrets and stale runtime authority; persist only the IDs/routing needed to rebuild through normal session/config loading.
  - confidence: 0.92
- **[P2] Write queue state with private permissions:** `src/auto-reply/reply/queue/persist.ts:141-144`
  - body: The state file is written with plain `fs.writeFileSync`, so queued prompts and route metadata inherit the user's umask. Use an explicit private mode such as `0o600`, preferably with an atomic temp-file replace, before writing this durable runtime state.
  - confidence: 0.87
- **[P1] Schedule restored queues without later inbound traffic:** `src/auto-reply/reply/queue/drain.ts:41-45`
  - body: The restored-key sweep only runs inside `rememberFollowupDrainCallback()`. After a cold gateway restart there are no callbacks until a later request path registers one, so a restored Telegram followup can remain idle instead of being delivered after restart; rebuild/register the runner from persisted session metadata or during startup.
  - confidence: 0.86
