## Summary

Keep this PR open: the durable queue direction is useful, but the current startup recovery wake can still be skipped by heartbeat preflight before a followup callback is registered, so the central restart-drain path is not merge-ready.

## What This Changes

The PR persists followup queues in shared SQLite, adds legacy JSON sidecar migration, restores queued runs on startup, and requests heartbeat wakes so restored routes can drain after gateway restart.

## Best Possible Solution

Classify restored-followup recovery as a heartbeat wake payload, add the empty/comment-only HEARTBEAT.md regression, then refresh live Telegram proof showing a mid-turn queued followup survives restart and drains without another user message.

## Reproduction Assessment

Yes from source inspection: a restored queue wake uses `source: "followup-queue-restore"`, while heartbeat preflight does not classify that source as a wake payload, so empty/comment-only HEARTBEAT.md can skip the run before drain callback registration.

## Solution Assessment

Not yet: persisting to shared SQLite is the right owner-boundary direction, but startup recovery must use the existing heartbeat wake semantics correctly before this is the best fix.
