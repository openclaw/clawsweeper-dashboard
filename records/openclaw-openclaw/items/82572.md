## Summary

Keep open: the PR addresses a real message-loss gap, but the current patch still serializes secret-capable runtime state, writes the recovery file without private permissions, and does not prove full Telegram restart delivery.

## What This Changes

The PR adds JSON persistence and restore/drain hooks for followup queues, plus focused queue persistence and restart-drain tests.

## Best Possible Solution

Persist a minimal private recovery descriptor, rebuild queued work through normal session/config loading, schedule recovery without needing another inbound message, and prove live Telegram restart delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted followups live in process-local maps and callbacks, so a full gateway restart loses or strands pending queued work without durable recovery.

## Solution Assessment

No. Persisting followup recovery is the right direction, but this patch stores live runtime objects and still depends on later callback registration instead of a narrow restart recovery contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Persist a minimal recovery descriptor:** `src/auto-reply/reply/queue/persist.ts:101`
  - body: `toPersistedRun()` stores `run: item.run`, and the queue entry also writes `lastRun`; the current run object carries the full OpenClaw config, including secret-capable channel/provider/auth fields. Persist only the IDs and routing needed to rebuild through normal session/config loading, otherwise the new queue file can store secrets and stale runtime policy.
  - confidence: 0.92
- **[P2] Write queue state with private permissions:** `src/auto-reply/reply/queue/persist.ts:141-144`
  - body: `fs.writeFileSync` creates `live-chat-followup-queues.json` using the process umask. The file contains user prompts and route metadata, so it needs an explicit private mode such as `0o600`, preferably with an atomic replace.
  - confidence: 0.87
- **[P2] Recover restored queues before another inbound message:** `src/auto-reply/reply/queue/drain.ts:41-45`
  - body: The restored-key sweep only runs when `rememberFollowupDrainCallback()` is called, and current code calls that from enqueue/drain paths rather than channel startup. After a cold restart, a queued Telegram prompt can remain idle until another message for the same route arrives, so restart recovery is not complete.
  - confidence: 0.84
