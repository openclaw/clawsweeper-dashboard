## Summary

Keep open: the queue-loss path is real, but this PR still has blocking privacy and recovery-design issues, and the posted proof does not show end-to-end Telegram delivery after a mid-turn restart.

## What This Changes

The PR adds JSON persistence and restore hooks for followup queue state, pending-drain scheduling on callback registration, and focused queue persistence tests.

## Best Possible Solution

Persist a minimal private recovery descriptor, rebuild runtime state through the normal config/session loading path, and add an explicit cold-start delivery path with redacted real Telegram proof.

## Reproduction Assessment

Yes. Source inspection of current main shows both followup queues and drain callbacks live in process-global maps, so a full gateway restart clears them; I did not run a live Telegram restart in this read-only review.

## Solution Assessment

No. The PR is on the right surface, but the maintainable fix should avoid serializing full runtime state and should deliver restored items through an explicit cold-start runner rather than waiting for later callback registration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Persist only the recovery record:** `src/auto-reply/reply/queue/persist.ts:102`
  - body: `toPersistedRun()` stores `item.run` directly, and the queue entry also stores `lastRun`; that object includes `OpenClawConfig`, sender metadata, exec settings, and secret-capable provider/channel/plugin config. Persisting the full runtime object can leak API keys, tokens, env, and private message context, so the state file should contain only the minimal fields needed to rebuild a followup through normal config/session loading.
  - confidence: 0.9
- **[P2] Write queue state with private permissions:** `src/auto-reply/reply/queue/persist.ts:142-145`
  - body: The new state file is written with plain `fs.writeFileSync`, so it inherits the process umask; the posted proof shows `-rw-rw-r--`. Queue prompts, routing metadata, and any recovery descriptor should be created atomically with private permissions or an existing secure-state helper.
  - confidence: 0.84
- **[P2] Resume restored queues without a new inbound message:** `src/auto-reply/reply/queue/drain.ts:41-45`
  - body: The pending-restored sweep only runs from `rememberFollowupDrainCallback()`. After a cold gateway restart those callbacks are empty and are not registered at channel startup, so a restored Telegram followup can sit on disk until another same-route inbound message arrives instead of recovering the queued message that was already lost by the restart.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a new on-disk queue state file that currently stores too much runtime data without private file permissions.

Concerns:

- **[medium] Avoid persisting full run config:** `src/auto-reply/reply/queue/persist.ts:102`
  - body: `run: item.run` and `lastRun` serialize `FollowupRun.run`, including `OpenClawConfig` fields that can contain provider API keys, channel/plugin credentials, env vars, MCP headers, and local-service configuration.
  - confidence: 0.9

  - body: The new state file is written without an explicit mode; even after narrowing the payload, queued prompts and routing metadata should not depend on the user's umask for privacy.
  - confidence: 0.84
