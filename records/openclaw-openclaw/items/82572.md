## Summary

Keep this PR open: it targets a real followup-queue loss problem, but the current branch is not merge-ready because restored runs are rehydrated with an empty config and restored queues can still wait for a later same-route inbound message before delivery.

## What This Changes

The PR adds JSON persistence, restore state, drain scheduling, and tests for auto-reply followup queues across gateway restarts.

## Best Possible Solution

Persist a minimal recovery descriptor that is rebuilt through current session, config, auth, and channel loaders, schedule restored queues when gateway/channel runtime is ready, and require redacted live Telegram restart-delivery proof before merge.

## Reproduction Assessment

Yes by source inspection: current main keeps `FOLLOWUP_QUEUES` and drain callbacks in process-local global maps, so a gateway process restart drops queued items. I did not run a live Telegram restart reproduction in this read-only review.

## Solution Assessment

No. Durable queue persistence is the right direction, but this implementation should rebuild restored runs from current session/config/auth contracts and kick restored drains from startup-ready runtime instead of relying on an empty config stub and later callback registration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rehydrate restored runs from the current config:** `src/auto-reply/reply/queue/persist.ts:198`
  - body: `rehydrateRun()` stores `{}` as the config for restored items, but the queued execution resolver only swaps to the runtime snapshot when the input is absent or matches the runtime source snapshot. With `{}` it keeps the empty config, so restored followups can lose channel/provider settings, scoped secret resolution, auth profile routing, and reply policy; rebuild from the current config/session store instead of using this stub.
  - confidence: 0.93
- **[P1] Drain restored queues without waiting for another message:** `src/auto-reply/reply/queue/drain.ts:36-45`
  - body: This only schedules restored work when a followup callback is registered, but after a cold gateway restart those callbacks are empty and current channel startup does not register one for the restored key. A queued prompt can therefore remain on disk until another same-route inbound message arrives; add a startup or cold-session runner path that drains restored queues once the gateway/channel runtime is ready.
  - confidence: 0.86
