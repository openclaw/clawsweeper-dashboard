## Summary

Keep this PR open: the underlying queue-loss problem is plausible, but the proposed persistence layer still has correctness and security blockers, and the latest proof does not show real post-restart Telegram delivery.

## What This Changes

The PR adds a JSON persistence module and queue mutation hooks so followup queue state can be restored after gateway restarts.

## Best Possible Solution

Persist a minimal recovery descriptor with private file permissions, restore through an explicit startup or cold-session drain path, and prove the full Telegram restart-and-delivery flow with redacted live evidence.

## Reproduction Assessment

Yes, source-reproducible: current main keeps followup queues and drain callbacks in process-global maps, so a full gateway restart clears both. I did not run live Telegram restart proof in this read-only review.

## Solution Assessment

No, the current PR is not the best solution yet because it persists too much runtime state and restored queues still rely on later callback registration. A narrower persisted schema plus an explicit cold-start recovery seam is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Persist only minimal queue recovery data:** `src/auto-reply/reply/queue/persist.ts:99`
  - body: `toPersistedRun()` stores `item.run` verbatim, and the queue entry also stores `lastRun`. That run contains `OpenClawConfig`, including env vars, provider keys/headers, channel config, and local-service env, so the new JSON state file can leak sensitive runtime configuration; persist only the identifiers and prompt/routing fields needed to rehydrate safely.
  - confidence: 0.9
- **[P2] Drain restored queues without waiting for later traffic:** `src/auto-reply/reply/queue/drain.ts:36-45`
  - body: The only new cold-restart scheduling hook runs inside `rememberFollowupDrainCallback()`. After a full gateway restart no callback exists for a restored queue until a later inbound turn reaches that route, so a queued Telegram message can remain pending indefinitely if the user does not send another message.
  - confidence: 0.84
- **[P2] Remove the new queue import cycle:** `src/auto-reply/reply/queue/state.ts:4`
  - body: `state.ts` imports `persist.ts`, while `persist.ts` imports live queue state/defaults from `state.ts`. This creates a static cycle in the queue modules and risks failing the repository import-cycle gate; move restore wiring to a non-cyclic bootstrap path or invert the dependency.
  - confidence: 0.78
- **[P2] Write the queue state file with private permissions:** `src/auto-reply/reply/queue/persist.ts:137-140`
  - body: The new `writeFileSync` call relies on the process umask for a file that contains queued prompts, media, routing metadata, and currently runtime config. Use an explicit private mode and harden the containing directory, matching existing state stores that use 0700 directories and 0600 files.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a new on-disk queue state file containing sensitive runtime and user-message data without a minimized schema or explicit private permissions.

Concerns:

- **[medium] Avoid persisting full run objects:** `src/auto-reply/reply/queue/persist.ts:99`
  - body: `persistFollowupQueues()` serializes `item.run` and `queue.lastRun` directly. Those objects include OpenClawConfig, which can contain model API keys, channel tokens/config, plugin config, inline env vars, and local-service env values; use a minimal recovery record and define the privacy contract.
  - confidence: 0.9
- **[medium] Use private permissions for queue state:** `src/auto-reply/reply/queue/persist.ts:137`
  - body: The state file is written without an explicit mode, so default umask settings can leave queued prompts, media, and routing data readable by other local users. Match existing sensitive stores with 0700 directories and 0600 files.
  - confidence: 0.82
