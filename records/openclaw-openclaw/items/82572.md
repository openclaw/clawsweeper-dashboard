## Summary

Keep open: this is a useful fix for a real queued-message loss path, but the current patch has blocking auth-provider and duplicate-delivery issues and still lacks full live Telegram restart-redelivery proof.

## What This Changes

This PR adds JSON persistence/restoration for followup queues, writes queue state on mutations, schedules restored drains when callbacks register, and adds queue persistence/drain tests.

## Best Possible Solution

Land durable queue persistence only after restored runs preserve or rebuild current auth/model/session intent, persistent state is updated at delivery acknowledgement points, and redacted live Telegram restart-redelivery proof confirms the user flow; keep broader cold-session startup dispatch design aligned with the followup API work.

## Reproduction Assessment

Yes, source-reproducible: current main stores followup queues and drain callbacks only in process-local global maps, so a process restart drops pending queue entries. I did not run a live Telegram restart reproduction in this read-only review.

## Solution Assessment

No. Durable persistence is the right direction, but this implementation needs the auth-profile restore contract and delivery-ack persistence fixed before it is the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve auth profile selection on restore:** `src/auto-reply/reply/queue/persist.ts:140-181`
  - body: The persisted run allowlist drops `authProfileId` and `authProfileIdSource`, but followup execution resolves credentials from `queued.run` via `resolveRunAuthProfile()`. After restart, a queued turn from a non-default profile can run with the default or missing credentials, so persist the non-secret profile id/source or rebuild them from the session entry before drain and cover that path in tests.
  - confidence: 0.88
- **[P1] Persist after each delivered item leaves the queue:** `src/auto-reply/reply/queue/drain.ts:485-488`
  - body: The state file is refreshed only in the drain `finally` block. Once `drainNextQueueItem()` successfully sends a prompt and shifts it from memory, a crash before `finally` rewrites the file leaves that delivered prompt on disk and restore will send it again; update persistent state at each successful dequeue/summary mutation instead of waiting for finalization.
  - confidence: 0.86

## Risks / Open Questions

- Restored queued turns can run with the wrong or default auth profile because authProfileId/authProfileIdSource are stripped from the persisted descriptor and not rebuilt before resolveRunAuthProfile runs.
