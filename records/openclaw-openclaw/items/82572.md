## Summary

Keep this PR open: the message-loss problem is real and the persistence direction is useful, but the current patch still has merge-blocking auth-profile and replay-safety defects plus incomplete live Telegram proof.

## What This Changes

The PR adds JSON persistence and restoration for followup queue state, writes queue state on mutations, schedules drains for restored queues when callbacks register, and adds queue persistence tests.

## Best Possible Solution

Keep the persistence layer only if it preserves or safely rebuilds queued auth selection, treats dequeue as an acknowledged durable state transition, and is backed by redacted live Telegram restart-redelivery proof.

## Reproduction Assessment

Yes, source-level: current main stores followup queues and drain callbacks only in process-local global maps, so process exit loses pending queue items. I did not run a live Telegram restart reproduction in this read-only review.

## Solution Assessment

No, not merge-ready yet: persistence is the right direction, but the patch needs auth-profile restoration, ack-safe persistence after dequeue, and live Telegram proof before it is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve the queued auth profile:** `src/auto-reply/reply/queue/persist.ts:140-181`
  - body: The restore descriptor drops `authProfileId`/`authProfileIdSource`, and the added test even asserts they stay undefined. Followup execution later calls `resolveRunAuthProfile(candidateRun, provider)`, so a queued turn that was originally bound to a user/session auth profile can restart under the default or wrong credentials. Persist the non-secret selector or rebuild it from the current session entry before dispatch, then cover that restore path.
  - confidence: 0.88
- **[P1] Persist dequeue acknowledgements before continuing:** `src/auto-reply/reply/queue/drain.ts:485-487`
  - body: The PR rewrites the state file only in the drain `finally` block. Since `drainNextQueueItem` shifts the item after `run` returns, a gateway crash after a prompt is delivered but before this final persist leaves the old item on disk and restores it for a duplicate delivery. Persist immediately after each successful item or batch removal, or route all removals through an ack-safe helper with coverage.
  - confidence: 0.9
