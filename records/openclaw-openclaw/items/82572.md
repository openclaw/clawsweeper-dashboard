## Summary

Keep open: the PR is a useful durability repair, but the new startup recovery wake is not wired into heartbeat's existing forced-wake/file-gate contract, so restored followups can still remain stuck after restart.

## What This Changes

The PR persists in-memory followup queues to shared SQLite, migrates legacy JSON sidecars, restores them on startup, and wakes affected sessions so queued followups can drain after gateway restart.

## Best Possible Solution

Land the SQLite persistence only after restore wakes use the same forced-wake and pending-event inspection contract as hook/restart events, with regression coverage for empty HEARTBEAT.md plus startup drain.

## Reproduction Assessment

## Solution Assessment

No; the persistence direction is sound, but the startup wake source is not connected to the heartbeat forced-wake contract that the recovery path depends on.
