## Summary

Keep open; this is a useful persistence slice, but it is not merge-ready because restored queues are parked without a production cold-start drain path, and the proof plus data-at-rest boundary still need maintainer signoff.

## What This Changes

The PR adds JSON persistence and restore for followup queues, persists queue mutations and drain acknowledgements, and adds restart/restore regression tests.

## Best Possible Solution

Land durable queue recovery only after restored items can re-enter a production cold-session drain path with original routing and exactly-once acknowledgement, or route the work through the broader followup API design.

## Reproduction Assessment

Yes, at source level: current main keeps FOLLOWUP_QUEUES in process memory, so a restart drops queued followups. The PR's remaining failure is also source-reproducible because restored queues are only scheduled through callback paths that normal cold-start inbound traffic does not register.

## Solution Assessment

No, not as currently wired. Persisting the queue is a good slice, but the maintainable solution also needs a real cold-session drain or followup API path plus an accepted state-file security boundary.
