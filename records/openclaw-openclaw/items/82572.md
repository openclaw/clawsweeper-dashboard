## Summary

Keep open: the durability direction is useful, but the latest patch can still schedule restored drains for routes whose active turn has not gone idle, and the supplied proof still stops short of a live Telegram mid-turn restart redelivery.

## What This Changes

The PR adds JSON disk persistence/restoration for auto-reply followup queues, persists queue mutations, restores queue state at module initialization, and adds queue persistence/drain regression tests.

## Best Possible Solution

Land a bounded queue persistence fix after restored drains are scheduled only for routes proven idle, the local state remains private/minimal with immediate dequeue acknowledgements, and redacted live Telegram restart proof covers exactly-once in-order delivery; keep proactive cold-session startup dispatch in https://github.com/openclaw/openclaw/pull/63330.

## Reproduction Assessment

Yes at source level: current main keeps followup queues only in a process-global map and has no disk persistence path, so a gateway process restart clears pending queue entries. I did not run a live Telegram restart reproduction in this read-only review.

## Solution Assessment

No, not yet: queue-local persistence is the right area, but the restored-drain sweep must preserve per-route idle ownership and the PR still needs live Telegram restart proof. Startup-time cold-session dispatch is better handled by the separate followup API design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Only drain restored keys that are idle:** `src/auto-reply/reply/queue/drain.ts:57-63`
  - body: `kickFollowupDrainIfIdle(key)` only has an idle guarantee for the `key` passed by the caller, but this loop schedules every restored key with a registered callback. If route B registered its callback during an active turn with `restartIfIdle=false`, then route A later going idle will also drain B, reintroducing the concurrent/out-of-order delivery race this commit is trying to fix. Limit the sweep to the current key or carry per-key idle readiness before scheduling other restored queues.
  - confidence: 0.88
