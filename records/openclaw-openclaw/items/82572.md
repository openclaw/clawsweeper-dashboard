## Summary

Keep open: the restart-loss problem is valid, but this branch still persists live runtime/config state, writes the queue file without private permissions, and lacks live Telegram delivery proof for the changed user flow.

## What This Changes

The PR adds JSON persistence and restore hooks for followup queues plus tests for persisted queue round trips.

## Best Possible Solution

Define one durable replay contract that stores a minimal private descriptor, rebuilds FollowupRun from current session/config state, schedules restored work without relying on another inbound message, and proves live Telegram restart delivery.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted followups live in process-global queue and callback maps, so a full process restart loses pending work without a durable recovery layer.

## Solution Assessment

No. Persistence is the right direction, but serializing FollowupRun/lastRun and relying on later callback registration is not the narrow maintainable recovery contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Persist only a recovery descriptor:** `src/auto-reply/reply/queue/persist.ts:101`
  - body: `toPersistedRun()` writes `run: item.run`, and the queue entry also writes `lastRun`. That object includes `OpenClawConfig` plus sender, owner, and execution policy fields, so this durable state file can leak secrets and replay stale runtime policy; persist only minimal IDs/routing and rebuild through current session/config loading.
  - confidence: 0.91
- **[P2] Write the queue state file privately:** `src/auto-reply/reply/queue/persist.ts:141-144`
  - body: The new queue state file is written with plain `fs.writeFileSync`, so queued prompts, routing metadata, and any persisted run data inherit the process umask. Use the repo's private state-file pattern with explicit `0o600` permissions, preferably via atomic replace plus chmod.
  - confidence: 0.86
- **[P2] Schedule restored work without another inbound message:** `src/auto-reply/reply/queue/drain.ts:36-45`
  - body: Restored queues are only swept when `rememberFollowupDrainCallback()` is called, but a cold gateway restart starts with an empty callback map. In practice the restored Telegram followup can sit idle until another same-route message arrives, so the original restart-loss workflow is not fully recovered.
  - confidence: 0.84
