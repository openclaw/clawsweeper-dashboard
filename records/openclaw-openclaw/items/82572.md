## Summary

Keep open: the restart-loss concern is source-reproducible, but this patch has blocking correctness issues, writes overly broad run/config state to disk, and still lacks contributor real behavior proof. The related followup API work at https://github.com/openclaw/openclaw/pull/63330 overlaps the recovery theme but is open and not a merged superseding implementation.

## What This Changes

The branch adds a JSON persistence module for followup queues and hooks enqueue, drain, clear, and refresh mutations to save and restore that state.

## Best Possible Solution

Persist a minimal serializable recovery descriptor, preserve attachments and routing identity, recreate or schedule a valid followup runner on startup, and validate with focused tests plus redacted live restart and Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: current main keeps followup queues and drain callbacks in process-global maps, so a process restart loses queued followup state and callback machinery. I did not run a live restart or Telegram proof in this read-only review.

## Solution Assessment

No, this is not the best implementation as written: it restores queue data without a startup drain path, drops queued media, and persists broad run/config state. A narrower recovery record plus explicit startup scheduling and privacy review would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Schedule restored queues after startup:** `src/auto-reply/reply/queue/persist.ts:157`
  - body: `restoreFollowupQueues()` only repopulates `FOLLOWUP_QUEUES` with `draining: false`. After a real restart `FOLLOWUP_RUN_CALLBACKS` is empty, so no restored item is scheduled for delivery until unrelated later traffic happens to install a callback, which does not satisfy the restart recovery goal.
  - confidence: 0.87
- **[P2] Persist queued media fields:** `src/auto-reply/reply/queue/persist.ts:22-37`
  - body: The persisted run shape omits `images` and `imageOrder`, but current queued runs can carry inbound media and the followup runner forwards those fields into the agent run. Image-bearing followups would restore as text-only messages after restart.
  - confidence: 0.9
- **[P2] Add required run fields to the test fixture:** `src/auto-reply/reply/queue/persist.test.ts:24-32`
  - body: `makeRun()` is declared as `FollowupRun["run"]`, but the returned object omits required `timeoutMs` and `blockReplyBreak`. The new test file will fail the test-type lane before it can validate persistence behavior.
  - confidence: 0.93
