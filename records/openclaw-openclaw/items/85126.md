## Summary

Keep open. The live issue update now says the wrong auto-selected auth profile affects both TUI/WebChat and Telegram sessions, and current main still has an order-based auto auth-profile resolver that persists an `auto` override during normal runs; the related open PR is nearby but not a complete landed fix for this new per-turn/all-channel variant.

## What This Changes

The issue reports that auto auth-profile resolution pins new sessions to `deepseek:default` instead of the configured `minimax:global` profile/model and can reassert that auto profile across turns.

## Best Possible Solution

Centralize auth-profile auto-selection around the resolved provider/model for the current session, preserve explicit user overrides, clear incompatible auto overrides, and add regression coverage for multi-profile sessions across WebChat/TUI and channel-style paths.

## Reproduction Assessment

No high-confidence local live reproduction was run. The report includes detailed current session-store evidence, and source inspection shows the resolver can persist and re-enter auto auth-profile selection on every run, but a focused multi-profile fixture is still needed for the exact MiniMax-versus-DeepSeek ordering.

## Solution Assessment

Unclear for the existing related PR. The right direction is to fix the resolver/session binding path, but https://github.com/openclaw/openclaw/pull/62710 is open, conflicting, and focused on reset/new-session inheritance rather than proving this all-channel per-turn variant.
