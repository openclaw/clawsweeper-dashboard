## Summary

Keep open. Current main and the latest release still have provider-scoped auth-profile resolution and child-session inheritance paths that can preserve an auto-selected fallback auth profile, and the issue already has an open closing PR plus a broader maintainer follow-up.

## What This Changes

Reported that auto-selected session auth profiles can pin Control UI/WebChat and other sessions to a fallback provider such as DeepSeek instead of the agent's configured primary MiniMax auth profile.

## Best Possible Solution

A single source-backed fix should make auto fallback auth/runtime state temporary and keep session creation, run selection, and display paths consistent while preserving explicit user overrides.

## Reproduction Assessment

Yes, at source level: a session with a MiniMax primary but stale auto `deepseek:default` auth profile still flows through provider-scoped auth order resolution. I did not run the live provider fallback scenario in this read-only pass.

## Solution Assessment

No, current main is not the complete solution yet; it has partial stale model fallback cleanup, but the auth-only/default-provider repair remains represented by the open closing PR and broader resolver tracker.
