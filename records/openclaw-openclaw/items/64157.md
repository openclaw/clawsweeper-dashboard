## What This Changes

The issue asks for Feishu-started conversations to continue delivering assistant replies to Feishu after follow-up messages are sent from the Web Control UI.

## Best Possible Solution

Track the shared operator-reply or WebChat mirroring policy in https://github.com/openclaw/openclaw/issues/45028 and split or replace https://github.com/openclaw/openclaw/pull/46356 with a focused, proof-backed implementation that includes Feishu coverage.

## Reproduction Assessment

Yes, at source level. Control UI sends `deliver:false` as a WebChat client, gateway origin resolution keeps that turn internal, and reply routing suppresses external Feishu delivery unless an explicit deliver route is set.

## Solution Assessment

Yes for cleanup: closing this as a duplicate keeps the remaining work on the older canonical cross-surface routing thread instead of splitting the same policy across channel-specific satellites. The actual code solution still needs maintainer-approved mirroring semantics and live channel proof.
