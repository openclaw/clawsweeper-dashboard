## Summary

Keep open. Current main still routes Web Control UI follow-ups as internal WebChat turns and intentionally prevents inherited external delivery to Feishu, so closing this as cleanup would hide a real cross-surface delivery policy gap.

## What This Changes

The issue asks for Feishu-originated conversations to keep delivering final assistant replies to Feishu after the user interrupts or follows up from Web Control UI.

## Best Possible Solution

Define an explicit cross-surface delivery contract for Feishu-started Web Control UI follow-ups, then update gateway/reply routing and Feishu regression coverage around that approved policy.

## Reproduction Assessment

Yes, at source level. Control UI sends `deliver: false` as a WebChat client, the gateway maps that turn to the internal route, and reply routing suppresses external Feishu delivery without an explicit deliver route; I did not establish a live Feishu WebSocket reproduction.

## Solution Assessment

Unclear because no patch is proposed. A simple fallback to the stored Feishu route would bypass intentional WebChat delivery guards, so the safer fix needs maintainer-approved cross-surface delivery policy first.
