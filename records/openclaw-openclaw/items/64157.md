## Summary

Keep open. Current main still treats Control UI follow-ups as internal WebChat turns and suppresses external Feishu delivery unless an explicit delivery route is present, so this needs a maintainer product/privacy decision rather than stale cleanup.

## What This Changes

Request that Feishu-started conversations continue delivering assistant replies to Feishu after the user interrupts or follows up from the Web Control UI.

## Best Possible Solution

Define an explicit permissioned cross-surface delivery contract, then implement opt-in Feishu/WebChat route inheritance or mirroring with regression coverage for the existing WebChat safety guard.

## Reproduction Assessment

Yes, at source level. Control UI sends deliver:false WebChat chat.send turns, the gateway resolves those as internal, and reply routing suppresses external delivery without ExplicitDeliverRoute; I did not run a live Feishu reproduction.

## Solution Assessment

Unclear because there is no proposed patch. A naive persisted-route fallback is not the best fix until maintainers decide the permission and privacy contract for browser-originated turns pushing replies back into Feishu.
