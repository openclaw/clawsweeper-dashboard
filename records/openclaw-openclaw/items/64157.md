## Summary

Keep open. Current main still treats Control UI follow-ups as internal WebChat turns and suppresses external Feishu delivery without an explicit delivery route, so this is a real cross-surface routing/product-policy decision rather than cleanup closure.

## What This Changes

The issue requests that assistant replies continue to deliver to Feishu when a Feishu-started conversation is continued from the Web Control UI.

## Best Possible Solution

Define an explicit opt-in cross-surface delivery or mirroring contract for Feishu-started Control UI follow-ups, then add gateway/reply-routing and Feishu regression coverage around that contract.

## Reproduction Assessment

Yes, at source level. Control UI sends `deliver: false` as a WebChat client, the gateway converts that turn to an internal route, and reply routing suppresses external Feishu delivery without `ExplicitDeliverRoute`; I did not run a live Feishu WebSocket reproduction.

## Solution Assessment

Unclear because there is no proposed patch. A simple persisted-route fallback would bypass intentional WebChat delivery guards, so the maintainable fix needs a maintainer-approved cross-surface delivery policy first.
