## Summary

Keep open. Current main still suppresses Feishu delivery for Control UI follow-ups because the UI sends internal WebChat `deliver: false` turns and reply routing intentionally blocks external delivery from internal WebChat unless an explicit delivery route is set; changing that needs maintainer product/privacy direction.

## What This Changes

The issue requests preserving Feishu delivery for a conversation after a follow-up is sent from the Web Control UI.

## Best Possible Solution

Define an explicit permissioned cross-surface delivery contract, then implement route inheritance or opt-in mirroring for Feishu-started Control UI follow-ups with regression coverage around the WebChat guard.

## Reproduction Assessment

Yes, at source level. The Control UI sends `chat.send` as WebChat with `deliver: false`, the gateway converts that to an internal route, and reply routing suppresses external Feishu delivery without `ExplicitDeliverRoute`.

## Solution Assessment

Unclear because there is no proposed patch. A simple persisted-route fallback is not the best fix until maintainers define when Control UI-originated turns may deliver back to Feishu.
