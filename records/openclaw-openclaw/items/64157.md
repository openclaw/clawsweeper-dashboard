## Summary

Keep open. Current main still sends Control UI chat turns as internal WebChat with `deliver: false`, and the gateway/reply-routing path blocks those turns from inheriting an established external Feishu route. Feishu inbound does record the original route, so this remains a real cross-surface routing-contract gap rather than cleanup-closeable completed work.

## What This Changes

The issue requests that assistant replies continue to be delivered to the original Feishu thread when a Feishu-started session receives follow-up or interrupt messages from the Web Control UI.

## Best Possible Solution

Define an explicit, permissioned cross-surface delivery contract for externally started sessions, then implement route inheritance or opt-in mirroring for Feishu-started Control UI follow-ups while preserving the existing WebChat safety guard and adding regression coverage.

## Reproduction Assessment

Yes, at code/test level. A Feishu inbound turn records `OriginatingChannel: "feishu"`, but a later Control UI turn sends `deliver: false`, becomes `OriginatingChannel: "webchat"`, and the existing tests assert that internal WebChat main-session turns do not inherit external routes.

## Solution Assessment

Unclear as a code change because there is no proposed patch. A simple stored-route fallback is not the best fix unless maintainers first define when a browser/UI turn is allowed to push back into Feishu; a permissioned route-inheritance or explicit delivery option is safer.
