## Summary

Keep open. Current main still treats Control UI follow-ups as internal WebChat turns and suppresses external Feishu delivery; changing that behavior needs an explicit maintainer product/privacy decision.

## What This Changes

The issue requests that replies continue to deliver to the Feishu thread when a Feishu-started conversation is continued from the Web Control UI.

## Best Possible Solution

Define a permissioned cross-surface delivery contract, then implement opt-in route inheritance or mirroring for Feishu-started Control UI follow-ups with regression coverage around the WebChat safety guard.

## Reproduction Assessment

Yes, at source level. Control UI sends `deliver: false` WebChat turns, the Gateway resolves those as internal, and reply routing suppresses external delivery without `ExplicitDeliverRoute`.

## Solution Assessment

Unclear because there is no proposed patch. A simple persisted-route fallback is not the best fix until maintainers define when browser/UI turns may push replies back into Feishu.
