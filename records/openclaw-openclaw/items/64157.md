## Summary

Keep open. Current main still turns Control UI follow-ups into internal WebChat `deliver: false` turns and then suppresses external reply routing, even when the session has a persisted Feishu route; changing that crosses an intentional WebChat delivery guard and needs maintainer product/privacy direction.

## What This Changes

The issue asks for Feishu-started conversations to keep delivering final assistant replies to Feishu after interrupt or follow-up messages are sent from the Web Control UI.

## Best Possible Solution

Define an explicit permissioned cross-surface delivery contract, then implement route inheritance or opt-in mirroring for Feishu-started Control UI follow-ups with regression coverage around the WebChat safety guard.

## Reproduction Assessment

Yes, at code level. Feishu inbound records the external route, but the Control UI follow-up is sent as `deliver: false` WebChat, and current reply routing suppresses external delivery without `ExplicitDeliverRoute`.

## Solution Assessment

Unclear because there is no proposed patch. A simple persisted-route fallback is not the best fix until maintainers define when browser/UI turns are allowed to push replies back into Feishu.
