## Summary

Keep open. Current main still treats Control UI follow-ups as internal WebChat turns and suppresses external reply routing even when the session has a persisted Feishu route, so this needs maintainer product/privacy direction rather than cleanup closure.

## What This Changes

Request that replies continue to deliver to the Feishu conversation when a Feishu-started session receives follow-up messages from Web Control UI.

## Best Possible Solution

Define an explicit permissioned cross-surface delivery contract, then implement route inheritance or opt-in mirroring for Feishu-started Control UI follow-ups with regression coverage around the WebChat safety guard.

## Reproduction Assessment

Yes, at source level. Control UI sends deliver:false as a WebChat client, the gateway converts that turn to an internal route, and reply routing suppresses external delivery without ExplicitDeliverRoute.

## Solution Assessment

Unclear because there is no proposed patch. A simple persisted-route fallback would bypass intentional WebChat delivery guards, so the maintainable fix needs an explicit cross-surface delivery policy first.
