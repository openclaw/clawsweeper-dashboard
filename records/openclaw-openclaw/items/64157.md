## Summary

Keep open. Current main still treats Control UI follow-ups as internal WebChat turns with external delivery disabled, while Feishu inbound turns do record the original route, so this remains a real cross-surface routing-contract gap that needs maintainer direction.

## What This Changes

The request is to keep assistant replies delivering to the original Feishu conversation after a Feishu-started session is continued or interrupted from the Web Control UI.

## Best Possible Solution

Define an explicit permissioned cross-surface delivery contract, then implement route inheritance or opt-in mirroring for Feishu-started Control UI follow-ups while preserving the WebChat safety guard and adding regression coverage.

## Reproduction Assessment

Yes, at code level. Feishu inbound records the external route, but the Control UI follow-up sends `deliver: false`, becomes an internal WebChat turn, and current reply routing suppresses external delivery without `ExplicitDeliverRoute`.

## Solution Assessment

Unclear because there is no proposed patch. A simple persisted-route fallback is not the best fix until maintainers define when browser/UI turns are allowed to push replies back into Feishu.
