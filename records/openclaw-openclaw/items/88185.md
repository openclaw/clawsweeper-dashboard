## Summary

Keep this PR open: current main still drops milliseconds while resolving offset-less `--at` datetimes in a timezone, and the proposed one-line parser fix plus regression rows match the root cause without adding a new surface.

## What This Changes

The PR passes `parts.millisecond` into the timezone offset `Date.UTC(...)` call and adds fractional-second parser regression rows for UTC and Europe/Oslo.

## Best Possible Solution

Land this narrow parser fix with the added fractional-second regression coverage once the normal maintainer CI and mergeability gates pass.

## Reproduction Assessment

Yes. Source inspection and a read-only Node probe show current main accepts fractional offset-less datetime syntax but rejects those values after millisecond-truncating the timezone offset calculation.

## Solution Assessment

Yes. Passing the already-populated numeric `millisecond` field into `Date.UTC` is the smallest maintainable fix, and the surrounding source shows Copilot's undefined-value concern does not apply because `getZonedDateTimeParts` always returns `utcDate.getUTCMilliseconds()`.
