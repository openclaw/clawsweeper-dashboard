## Summary

Keep open. The PR is not ready because it introduces a scoped CLI-loading regression, misses the required changelog entry, and is currently dirty against main; the linked RFC also asks for manifest-granted access while this PR deliberately keeps host approval, so maintainers need to confirm that trust model before merge.

## What This Changes

The PR adds `permissions.conversationAccess` manifest parsing, registry preservation, docs, generated API baseline hashes, manifest/loader/CLI tests, config-hook policy filtering, and a helper that appends the configured memory slot during scoped CLI command loading.

## Best Possible Solution

Keep the manifest contract additive and explicit, preserve operator approval as the effective grant unless maintainers intentionally approve a different trust model, split or narrow the memory-slot CLI behavior to wiki/memory command paths, add the active-version changelog entry, and rebase before merge.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction for the PR regression is a primary plugin command such as `googlemeet`: command activation resolves `google-meet`, then the PR helper appends the default `memory-core` slot because it checks only that `pluginIds` is non-empty.

## Solution Assessment

No for the memory-slot portion, and unclear for the conversation-access semantics until maintainer confirmation. The safer narrow path is manifest request/disclosure metadata plus explicit host approval, with any memory-slot loading fix kept scoped to wiki or memory-specific activation.
