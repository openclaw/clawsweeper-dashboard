## Summary

Keep open. The manifest metadata portion is a plausible additive contract, but this PR is not merge-ready because it broadens primary-command plugin loading by appending the configured memory slot for every matched command and it omits the required changelog entry for a user-facing plugin manifest/API-doc change. It also intentionally implements request-only metadata rather than the linked RFC's manifest-granted access model, so maintainers should confirm that product/security boundary before closing the RFC.

## What This Changes

The PR adds declarative `permissions.conversationAccess` plugin manifest metadata, preserves it in manifest registry records and SDK/API docs/tests, keeps effective conversation hook access gated by host config, and appends the configured memory-slot plugin during primary CLI command loading.

## Best Possible Solution

Keep `permissions.conversationAccess` as declarative request metadata only if maintainers approve that trust model, keep raw conversation-content hooks gated by explicit host config, scope the memory-slot append to the wiki/memory command path or an explicit memory dependency, add the active-version changelog entry, and leave #71428 open unless maintainers deliberately accept request-only metadata as the final resolution.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is `openclaw googlemeet ...`: current manifests resolve `googlemeet` to `google-meet`, and the PR helper would append the default `memory-core` slot because it only checks that some plugin id was planned.

## Solution Assessment

No for the PR as written. The manifest metadata change is narrow and additive, but the memory-slot loading change should be scoped to memory/wiki activation, and the linked conversation-access trust model needs explicit maintainer confirmation because the PR does not implement manifest-granted access.
