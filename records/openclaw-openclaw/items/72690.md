## Summary

Keep open. The manifest metadata path is additive, but the PR still introduces a concrete lazy-CLI regression by appending the configured memory slot to every primary command with any plugin activation match, and it is also dirty with failed checks. The linked RFC asks for stronger manifest-granted access semantics than this PR implements, so maintainers should explicitly confirm the request-only trust model before treating the related issue as solved.

## What This Changes

The PR adds `permissions.conversationAccess` parsing and registry preservation for plugin manifests, documents it as request metadata gated by host config, adds manifest/loader/CLI tests, refreshes the plugin SDK API baseline hash, and appends the configured memory slot during primary CLI command loading.

## Best Possible Solution

Keep `permissions.conversationAccess` as additive request metadata only if maintainers approve that trust model, keep raw conversation hook access gated by explicit host config, narrow memory-slot loading to the `wiki`/memory command path or another memory-specific activation signal, add the required changelog entry, and leave #71428 open or explicitly narrow it if manifest-granted access remains out of scope.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is `openclaw googlemeet ...`: current manifests resolve `googlemeet` to `google-meet`, and the PR helper would append the default `memory-core` slot because it only checks that some plugin id was planned.

## Solution Assessment

No for the PR as written. The manifest metadata portion is a narrow additive contract, but the memory-slot loading belongs on a memory/wiki-specific path, and the linked conversation-access trust model needs explicit maintainer confirmation because the PR does not implement the RFC’s requested manifest-granted access.
