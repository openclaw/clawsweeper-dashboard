## Summary

Keep open: the PR is a plausible implementation candidate for manifest conversation-access metadata, but the patch introduces blocking regressions in hook timeout policy and CLI plugin load scope, lacks real behavior proof, and still needs maintainer confirmation of the request-only security model.

## What This Changes

The branch adds request-only `permissions.conversationAccess` manifest parsing/preservation/docs/tests/API baseline updates, narrows loader hook policy normalization, and appends the configured memory-slot plugin to lazy primary CLI load scopes.

## Best Possible Solution

Land a narrow manifest permission metadata contract that preserves explicit host approval and existing hook timeout policy, without widening CLI plugin load scopes.

## Reproduction Assessment

Yes. Source inspection against PR head shows configured hook timeout fields are discarded before `createApi`, and any non-empty primary-command activation plan gets the configured/default memory slot appended.

## Solution Assessment

No. Request-only manifest metadata may be a safer direction than self-authorization, but this patch should preserve existing hook policy fields and remove or tightly scope the unrelated memory-slot load change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve configured hook timeouts:** `src/plugins/loader.ts:189-196`
  - body: `resolvePluginLoaderHookPolicy` copies only the two allow flags before passing policy to `createApi`. Current hook policy also carries `timeoutMs` and `timeouts`, so configs such as `plugins.entries.<id>.hooks.timeouts.agent_end` will silently stop overriding hook budgets after this patch.
  - confidence: 0.91
- **[P2] Limit the memory-slot append to memory commands:** `src/plugins/cli-registry-loader.ts:81`
  - body: This appends the configured/default memory slot after any primary command resolves to a plugin. Because `plugins.slots.memory` defaults to `memory-core`, unrelated commands such as `googlemeet` will load memory plugin code too, widening the scoped CLI loading that activation planning currently provides.
  - confidence: 0.88
