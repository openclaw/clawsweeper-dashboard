## Summary

Keep open: this PR is still a plausible implementation candidate for the linked manifest-permission RFC, but it has a blocking overbroad CLI load-scope change and lacks contributor real behavior proof.

## What This Changes

The PR adds request-only `permissions.conversationAccess` manifest parsing/preservation/docs/tests/API baseline updates, sanitizes loader hook-policy input, and appends the configured memory-slot plugin to primary CLI command load scope.

## Best Possible Solution

Land or replace this PR with a narrow manifest metadata contract that preserves explicit host approval, removes or scopes the memory-slot load change, rebases onto current main, and includes redacted runtime proof.

## Reproduction Assessment

Yes for the review finding. Source inspection of the PR head shows any non-empty primary-command activation plan gets the configured memory slot appended, and `googlemeet` is a concrete non-memory command activation path on current main.

## Solution Assessment

No. Request-only manifest metadata is a reasonable direction, but the PR mixes in an overbroad CLI load-scope change and still needs maintainer confirmation of the security model before merge.
