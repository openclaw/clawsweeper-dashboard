## Summary

Keep open. Current main does not implement manifest-level conversation-access metadata, and this PR is a plausible implementation path, but it is not merge-ready because it broadens primary-command plugin loading by appending the configured memory slot to unrelated commands and it omits the required changelog entry for a public plugin manifest contract.

## What This Changes

The PR adds request-only `permissions.conversationAccess` manifest metadata, preserves it in plugin manifest registry records, documents the host-config approval gate, adds focused tests, refreshes the SDK API baseline hash, and changes primary CLI plugin loading to append the configured memory-slot plugin.

## Best Possible Solution

Land a narrow manifest contract only after the CLI loading regression is fixed: preserve request-only `permissions.conversationAccess` metadata, keep raw conversation hooks gated by explicit host config, add release-note coverage, and leave the broader manifest-granted-access RFC open unless maintainers explicitly choose the request-only trust model as final.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an unrelated primary command such as `openclaw googlemeet ...`: the Google Meet manifest maps `googlemeet` to `google-meet`, and the PR helper would append the default `memory-core` slot because it only checks that some plugin id was planned.

## Solution Assessment

No for the PR as written. The manifest metadata portion is narrow and additive, but the memory-slot loading change should be gated to a memory/wiki-specific activation path, and the RFC trust-model mismatch needs maintainer confirmation before treating this as complete.
