## Summary

Keep open. The manifest metadata part is additive and keeps raw conversation hooks gated by host config, but this PR still has a concrete CLI loading regression and is missing required changelog coverage; it also intentionally implements request-only metadata while #71428 proposed manifest-granted access, so maintainers should confirm the trust model before treating the linked RFC as resolved.

## What This Changes

The PR adds request-only `permissions.conversationAccess` manifest parsing and registry preservation, updates plugin/config docs and tests, refreshes the SDK API baseline hash, sanitizes loader hook-policy passthrough, and appends the configured memory-slot plugin during primary CLI command loading.

## Best Possible Solution

Repair the branch so memory-slot loading is scoped to the wiki/memory path, add the active-version changelog entry for the new manifest contract, and then have maintainers decide whether request-only permission metadata is the desired resolution for #71428 or whether that RFC should remain open for a separate consent/grant design.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a non-memory primary command such as `googlemeet`: command activation resolves `google-meet`, the PR helper sees a non-empty plan, `normalizePluginsConfig` supplies default `memory-core`, and `onlyPluginIds` becomes broader than the requested command needs.

## Solution Assessment

No for the PR as written. The manifest parsing/registry preservation is narrow and compatible, but the memory-slot append should be tied to the wiki/memory dependency and the request-only trust model should not silently close an RFC that proposed manifest-granted access.
