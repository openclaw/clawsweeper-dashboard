## Summary

Keep open. The manifest metadata path is conservative, but the PR still broadens primary CLI plugin loading by adding the configured/default memory plugin to every non-empty activation plan, and it lacks the required changelog entry for a documented plugin contract change.

## What This Changes

The PR adds request-only `permissions.conversationAccess` manifest metadata, docs, tests, API baseline updates, and memory-slot inclusion in primary CLI plugin load scope.

## Best Possible Solution

Land the manifest request metadata after scoping memory-slot inclusion to the wiki/memory CLI path, adding release-note coverage, and confirming the request-only trust model for #71428.

## Reproduction Assessment

Yes. A code-level reproduction is a manifest-owned non-memory primary such as `googlemeet`: current main scopes loading to the activation result, while the PR would also append the normalized default `memory-core` slot.

## Solution Assessment

No for the PR as written. The manifest metadata is a narrow maintainable addition, but the memory-slot loading change is broader than the stated scope and should be gated to the commands that actually need memory/wiki cooperation.
