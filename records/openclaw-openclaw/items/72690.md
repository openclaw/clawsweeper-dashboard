## Summary

Keep open. The manifest metadata portion is useful and request-only, but the PR still broadens primary-command CLI loading by appending the memory slot to every non-empty activation plan, and it lacks the required changelog entry for a public plugin manifest contract.

## What This Changes

The PR adds `permissions.conversationAccess` manifest parsing, registry preservation, docs, tests, and API baseline updates while keeping raw conversation hook access gated by host config, and also appends the configured memory-slot plugin during scoped CLI primary-command loading.

## Best Possible Solution

Land this only after scoping memory-slot loading to the intended memory/wiki CLI path, adding an Unreleased changelog entry, and getting maintainer confirmation that request-only manifest metadata is the desired trust model.

## Reproduction Assessment

Yes. A code-level reproduction is a primary command owned by another plugin, such as `googlemeet`: current main scopes loading to the activation-planned plugin ids, while the PR appends the default `memory-core` slot after any non-empty plan.

## Solution Assessment

No for the PR as written. The manifest metadata is narrow and conservative, but the memory-slot loading change is broader than the intended surface and the public contract needs a changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Limit memory-slot append to memory commands:** `src/plugins/cli-registry-loader.ts:80`
  - body: The final patch calls `withPrimaryMemorySlotPluginId` for every non-empty primary-command activation plan. Since normalized config supplies `memory-core` when the slot is omitted, unrelated plugin commands such as `googlemeet` will load memory-core too, undoing the scoped CLI loading added by the activation planner. Gate this append on the memory/wiki command path instead of every planned primary.
  - confidence: 0.9

  - confidence: 0.82
