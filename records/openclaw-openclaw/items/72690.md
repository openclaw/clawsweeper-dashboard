## Summary

Keep open: this PR is still the unmerged implementation candidate for the linked manifest conversation-access RFC, but the patch is not ready because it broadens CLI plugin loading and lacks contributor real-behavior proof.

## What This Changes

The PR adds request-only `permissions.conversationAccess` manifest metadata, preserves it in registry records, updates docs/tests/API baselines, and appends the configured memory-slot plugin to primary CLI load scope.

## Best Possible Solution

Land a narrow manifest metadata contract only after preserving explicit host approval, removing or scoping the memory-slot load change to memory/wiki commands, and adding redacted real-behavior proof.

## Reproduction Assessment

Yes. From source inspection, a primary command owned by another plugin such as `googlemeet` produces a non-empty activation plan, and the PR appends the configured memory slot to that unrelated load scope.

## Solution Assessment

No. The request-only manifest metadata is a plausible direction, but the PR mixes in an overbroad CLI load-scope change and needs maintainer confirmation that request-only metadata is the intended resolution for the RFC.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Limit the memory-slot append to memory commands:** `src/plugins/cli-registry-loader.ts:80`
