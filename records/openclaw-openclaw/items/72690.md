## Summary

Keep open. Current main still lacks manifest-level conversation-access metadata, and the PR is useful, but it introduces an overbroad CLI memory-slot load and is missing required changelog coverage; maintainers also need to confirm whether request-only metadata is an acceptable resolution for #71428.

## What This Changes

The PR adds request-only `permissions.conversationAccess` manifest metadata, preserves it in registry records, documents the host-config approval gate, adds focused tests/API baseline updates, and appends the configured memory-slot plugin to primary CLI load scope.

## Best Possible Solution

Land the manifest metadata only after scoping memory-slot loading to memory/wiki commands, adding release-note coverage, and confirming the request-only trust model for #71428.

## Reproduction Assessment

Yes. A code-level reproduction is a primary command owned by another plugin, such as `googlemeet`: the PR appends the normalized memory slot after any non-empty activation plan, while current main returns only the activation-planned plugin ids.

## Solution Assessment

No for the PR as written. The manifest metadata is narrow and conservative, but the memory-slot loading change is broader than the intended wiki/memory scope and the RFC trust model needs maintainer confirmation.
