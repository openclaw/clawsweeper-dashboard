## Summary

## What This Changes

The issue reports that opening Dreaming diary/imported content in WebChat fails because `wiki.importInsights` returns `GatewayRequestError: unable to resolve opened file path` for bridge-mode dreaming files.

## Best Possible Solution

Reproduce the WebChat/Gateway path on current main with a bridge vault and dreaming artifact, then fix the Memory Wiki bridge/read flow so valid dreaming reports surface without broadening the fs-safe sandbox; if one imported source page is unsafe, isolate and report that artifact instead of failing the entire Dreaming UI.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; source inspection gives a plausible path where `wiki.importInsights` aborts on a bridge `path-mismatch`, but I did not reproduce the reporter's exact vault state against current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear; whitelisting the dreaming directory is not obviously the right fix because current bridge code already supports external dream-report sources, so maintainers should first verify the failing path and preserve fs-safe boundaries.
