## Summary

Keep this PR open: current main still lacks a Bonjour plugin config key for per-instance service names, and the PR is a focused implementation of the right plugin-owned design. It is not merge-ready because the added test imports `createTestPluginApi` from a helper path that no longer exists on current main, so the Bonjour test lane would fail before exercising the new cases.

## What This Changes

The PR adds `plugins.entries.bonjour.config.instanceName` to the Bonjour plugin manifest, wires it through the plugin entry into the existing advertiser `instanceName` option, and updates Bonjour tests, docs, and changelog.

## Best Possible Solution

Land this PR or an equivalent Bonjour-plugin-owned implementation after updating it for current main: keep the manifest schema key, read the value through `api.pluginConfig`, preserve the existing suffix behavior, and avoid adding a core `gateway.bonjour.*` config path.

Yes. A high-confidence source-level reproduction is two gateway instances with the same `ctx.machineDisplayName`: current main formats the same Bonjour service instance name for both, and the discussion includes a live macOS observation of ciao adding the `(2)` suffix.
