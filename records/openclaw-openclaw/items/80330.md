## Summary

Keep open. Current main has bounded restart draining and requester-origin safeguards, but active `model_call` stalls are still observe-only and the report adds concrete, recent mixed Weixin/Feishu delivery-queue evidence that is not fully covered by an existing fix.

## What This Changes

The issue reports stalled gateway restart draining during a `model_call` and subagent completion deliveries being queued for Feishu despite a Weixin requester origin.

## Best Possible Solution

Route the active `model_call` recovery policy through the canonical stuck-session work, then add a focused mixed Weixin/Feishu subagent completion regression that proves queued delivery keeps `channel`, `to`, and `accountId` from the same requester origin.

## Reproduction Assessment

Yes for part of the issue at source level: current main still classifies stalled `model_call` active work as non-recovery-eligible, so it can log `recovery=none`. No high-confidence live reproduction was established for the mixed Weixin/Feishu wrong-channel delivery queue path.

## Solution Assessment

Unclear. A hard 60s drain timeout alone is not the best fix because current main already has configurable drain budgets; the maintainable path is a model-call recovery policy decision plus a targeted mixed-channel routing regression.
