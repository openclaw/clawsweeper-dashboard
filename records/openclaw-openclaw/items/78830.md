## Label Justifications

- `P1`: The issue describes a default active-run channel path where security plugins cannot currently block steering input at the requested boundary.
- `impact:security`: The requested hook is for prompt-injection screening and security-policy enforcement before steering input reaches the active runtime queue.

## Summary

Keep open: current main still lacks the requested steering-specific plugin hook, while normal active-run steering and explicit `/steer` still queue user text before any steering-specific block/rewrite decision. The shipped `before_agent_run` gate is a meaningful later mitigation, but the protected security label and public plugin API boundary require maintainer/security review rather than cleanup closure.

## What This Changes

Request to add a public `before_steering` plugin hook that can inspect, rewrite, or block active-run steering input before normal queue and `/steer` injection.

## Best Possible Solution

Design a documented steering input-gate contract that covers normal inbound steering and explicit `/steer`, or explicitly document why `before_agent_run` is the supported security boundary for steering inputs.

## Reproduction Assessment

Yes, by source inspection. Current main defaults inbound active-run behavior to `steer`, and both normal steering and explicit `/steer` queue text without a steering-specific block/rewrite hook; I did not run a live prompt-injection exploit.

## Solution Assessment

Unclear. A dedicated `before_steering` hook is plausible, but maintainers should first decide whether the durable security boundary is pre-queue, pre-transcript, `before_agent_run`, or a shared runtime steering gate.
