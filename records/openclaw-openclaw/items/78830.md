## Summary

Keep open: current main still has no `before_steering` hook, and inbound plus explicit steering paths queue text into active runtime steering before a steering-specific block/rewrite gate. This is a security-sensitive public plugin API addition, so it needs maintainer/API design rather than cleanup closure or an automatic fix lane.

## What This Changes

Request to add a `before_steering` plugin hook that can inspect, rewrite, or block active-run steering messages before runtime injection.

## Best Possible Solution

Design a documented plugin input-gate seam for active-run steering across normal inbound steering and explicit `/steer`, with clear failure policy, metadata contract, Pi/Codex behavior, and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main defaults inbound queueing to `steer`, active inbound and explicit `/steer` paths call the runtime steering queue directly, and no `before_steering` hook exists in the registry. I did not run a live exploit because this is a security-sensitive public API gap rather than a narrow failing test case.

## Solution Assessment

Unclear. A dedicated `before_steering` hook is plausible, but maintainers should first decide the public API shape, failure policy, metadata exposure, and whether the gate belongs before enqueue or at a shared runtime steering boundary.
