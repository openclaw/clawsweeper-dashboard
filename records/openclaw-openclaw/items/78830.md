## Summary

Keep open: current main still has no `before_steering` hook, and both active inbound steering and explicit `/steer` enqueue steering text without a dedicated per-steering blocking or rewrite hook. Because this is a security-sensitive public plugin API change, it needs maintainer design rather than cleanup closure or an automated fix lane.

## What This Changes

The issue requests a new blocking `before_steering` plugin hook so security plugins can inspect, rewrite, or reject active-run steering messages before runtime injection.

## Best Possible Solution

Design a documented plugin input-gate seam for active-run steering that covers normal inbound steering and explicit `/steer`, with regression coverage for blocking and rewrite behavior before runtime injection.

## Reproduction Assessment

Yes, by source inspection: current main defaults unconfigured queues to `steer`, active inbound and explicit `/steer` paths enqueue text into `queueEmbeddedPiMessageWithOutcome`, and the hook registry has no `before_steering` entry. I did not run a live exploit because the remaining question is a security-sensitive public plugin API gap rather than a narrow runtime crash.

## Solution Assessment

Unclear. A dedicated `before_steering` hook is plausible, but maintainers should first decide the public API, failure policy, metadata contract, audit behavior, and whether the gate belongs before queue admission or at a generic runtime boundary.
