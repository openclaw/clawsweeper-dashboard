## Summary

Close as implemented on current main: the native Codex harness path now resolves `openai/gpt-*` auth through `openai-codex` before forwarding, with regression coverage that starts from `authProfileProvider: "openai"` and still forwards `openai-codex:work`. The latest release `v2026.5.2` does not contain the proof commit, so this is fixed on `main` only for now.

## What This Changes

Prevent native Codex runtime runs for `openai/gpt-*` from forwarding generic `openai:*` auth profiles when a compatible `openai-codex:*` OAuth profile should be used.

## Best Possible Solution

Keep the current main Codex-aware auth selection and let the next release ship it; any stale-session override cleanup should be tracked separately if it still reproduces.

## Reproduction Assessment

No against current main: the source path now auto-selects `openai-codex` auth for `openai/gpt-*` runs forced through `agentRuntime.id: "codex"`, and checked-in tests cover that exact shape. The failing release behavior is source-understood from the app-server bridge rejecting non-`openai-codex` profiles.

## Solution Assessment

Yes. The current main solution is narrow and maintainable: it centralizes Codex harness auth-provider awareness in the runtime auth plan while preserving direct `openai` API-key behavior for non-Codex runs.
