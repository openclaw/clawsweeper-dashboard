## Summary

Keep open. The PR addresses real reply-boundary symptoms, but the current branch is still incorrect: it can corrupt durable external route state, adds overbroad sanitizer heuristics with a hardcoded personal-name bypass, and lacks required after-fix real behavior proof for visible chat delivery.

## What This Changes

The PR suppresses unphased tool-use assistant output, adds self-instruction prelude stripping, removes sessions_send A2A announce delivery, and changes inter-session route resolution/tests.

## Best Possible Solution

Split the work: land a narrow structural tool-use suppression fix with real channel proof, preserve durable external route state while suppressing only current outward sessions_send delivery, and resolve broader A2A dispatch semantics through the canonical tracker.

## Reproduction Assessment

Yes. Source inspection shows current main only suppresses commentary-phase assistant output, and current route resolvers/tests show exactly what durable external-route behavior the PR changes; I did not run live Telegram or Slack proof in this read-only review.

## Solution Assessment

No. The PR addresses real symptoms, but this combined patch is not the best solution because it mixes tool-use suppression, broad sanitizer heuristics, A2A announce semantics, and durable session route mutation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve stored external routes for sessions_send:** `src/auto-reply/reply/session.ts:561-563`
  - body: This makes sessions_send inter-session turns opt out of external-route preservation, so initSessionState can persist webchat and session:* over a Telegram/Discord route.   - confidence: 0.88
- **[P2] Gate monologue stripping on stronger leak signals:** `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts:419-425`
  - body: The marker-count branch keeps only the final block when earlier text has broad phrases like the prompt and I should, which can occur in legitimate multi-paragraph answers. This can silently delete valid user-facing content without a structural leaked-prelude signal.
  - confidence: 0.82
- **[P2] Remove the hardcoded personal-name bypass:** `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts:383`
  - body: The production exclusion regex contains the literal name Bernhard, so any block mentioning that name skips the internal-monologue filter even when it otherwise matches the leak pattern. This looks like a debugging artifact and makes sanitizer behavior depend on one personal name.
  - confidence: 0.9
