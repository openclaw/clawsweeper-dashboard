## Summary

Keep open. The PR targets real user-visible boundary leaks, but current `main` does not yet implement the full requested behavior, and this branch introduces blocking route-persistence and sanitizer-regression issues that should be resolved before merge.

## What This Changes

The PR suppresses unphased tool-use assistant output, adds heuristic self-instruction prelude stripping, removes `sessions_send` A2A announce delivery, and changes session route resolution for `sessions_send` inter-session turns.

## Best Possible Solution

Split the concerns into narrower changes: land a minimal tool-use suppression fix that still preserves explicit `final_answer` output, keep durable external route state intact while suppressing only the current outward delivery if needed, and redesign any self-instruction stripping around stronger structural leak markers plus negative tests for valid replies.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists through the embedded subscription harness for an unphased `stopReason: "toolUse"` assistant message, and through `session-delivery`/`initSessionState` route-resolution cases with an established Telegram or Discord route. I did not run tests because this review was required to be read-only.

## Solution Assessment

No. The patch fixes real symptoms, but it is not the best implementation because it mixes separate policy decisions, can persist internal `webchat`/`session:*` routing over an established external route, and adds broad sanitizer heuristics with a hardcoded personal-name bypass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve stored external routes for sessions_send:** `src/auto-reply/reply/session.ts:561-563`
  - body: This makes every `sessions_send` inter-session turn opt out of the existing external-route preservation path. `initSessionState` then persists the resolver result, so a Telegram/Discord-backed target can have `lastChannel`/`lastTo` replaced with `webchat` and `session:*`, breaking later announce, cron, or system follow-ups until another external message repairs the route. Suppress only the current turn's outward delivery while keeping the stored route intact.
  - confidence: 0.88
- **[P2] Gate monologue stripping on stronger leak signals:** `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts:419-425`
  - body: The marker-count branch returns only the final block whenever the prelude has broad phrases such as `the prompt` and `I should` and the last block has none. Those phrases can occur in legitimate multi-paragraph answers, so this can silently delete valid user-facing content. Require stronger structural evidence of a leaked internal prelude before replacing the full reply.
  - confidence: 0.82
- **[P2] Remove the hardcoded personal-name bypass:** `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts:383`
  - body: The production exclusion regex includes the literal name `Bernhard`, so any block containing that name skips the internal-monologue filter even if it otherwise matches the leak pattern. This looks like a test/debug artifact and makes sanitizer behavior depend on one personal name.
  - confidence: 0.9
