## Summary

Keep open: this targets real reply-boundary bugs, but the branch is not safe to merge as-is because it changes durable route persistence and A2A semantics, adds broad sanitizer heuristics with a hardcoded bypass, and lacks real behavior proof for visible channel delivery.

## What This Changes

The branch suppresses unphased tool-use assistant output, adds heuristic self-instruction prelude stripping, removes `sessions_send` A2A announce delivery, and changes inter-session route resolution for `sessions_send`.

## Best Possible Solution

Split this into narrower work: land a structural tool-use suppression fix that preserves explicit final answers, keep durable external route state intact, and handle dispatch-only A2A semantics through the canonical product/API tracker.

## Reproduction Assessment

Yes, by source inspection and targeted harness paths. Unphased `stopReason: "toolUse"` assistant messages can still pass current-main suppression, while current route resolvers show why overwriting persisted external routes is risky.

## Solution Assessment

No. The patch addresses real symptoms, but it mixes separate fixes, changes default A2A behavior, can persist internal routes over established channel routes, and uses broad sanitizer heuristics instead of stronger structural leak signals.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve stored external routes for sessions_send:** `src/auto-reply/reply/session.ts:561-563`
  - body: This makes every `sessions_send` inter-session turn opt out of the existing external-route preservation path. `initSessionState` persists the resolver output, so a Telegram/Discord-backed target can have `lastChannel`/`lastTo` replaced with `webchat` and `session:*`, breaking later announce, cron, or system follow-ups until another external message repairs the route. Suppress only the current outward delivery while keeping stored route state intact.
  - confidence: 0.88
- **[P2] Gate monologue stripping on stronger leak signals:** `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts:419-425`
  - body: The marker-count branch returns only the final block when the prelude has broad phrases such as `the prompt` and `I should` and the last block has none. Those phrases can appear in legitimate multi-paragraph answers, so this can silently delete valid user-facing content without a leaked prelude.
  - confidence: 0.82
- **[P2] Remove the hardcoded personal-name bypass:** `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts:383`
  - body: The production exclusion regex includes the literal name `Bernhard`, so any block containing that name skips the internal-monologue filter even if it otherwise matches the leak pattern. This looks like a debug artifact and makes sanitizer behavior depend on one personal name.
  - confidence: 0.9
