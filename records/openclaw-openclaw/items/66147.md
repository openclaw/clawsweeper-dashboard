## Summary

Keep open: this is still a plausible feature PR, but the branch is not merge-ready because it changes core agent model-routing and provider/auth boundaries with source-visible P1 defects and no real cross-provider proof.

## What This Changes

The PR adds a public `model.tool` config field plus resolver/schema changes and a Pi embedded runner stream wrapper for routing tool-continuation turns to a configured alternate model.

## Best Possible Solution

Keep the PR open only as an implementation candidate until maintainers choose the split-routing contract; if it stays in core, the effective model must drive tool availability, transport, auth profile/API key, wrappers, fallback, diagnostics, and context-budget behavior.

## Reproduction Assessment

Yes for the PR blockers by source inspection: current main prepares tool support, transport, auth, wrappers, and diagnostics from `params.model`/`params.provider`, while the branch only adds a late stream wrapper. The advertised real Ollama plus Anthropic path is not reproduced because the PR body leaves that integration test unchecked.

## Solution Assessment

No. The proposed wrapper swaps only the `Model` argument on continuation turns; the maintainable solution must define the product/security contract and make the effective model own the full call pipeline, or defer to the plugin/runtime API design thread.
