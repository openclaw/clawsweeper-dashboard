## Summary

Keep this PR open: current main still lacks `model.tool`, but the proposed implementation is not merge-ready because it swaps only the call `Model` after primary-provider transport, auth, transcript policy, diagnostics, and tool enablement are already selected.

## What This Changes

The PR adds a `model.tool` config field, resolver/schema support, and a Pi embedded runner stream wrapper/tests intended to route tool-continuation calls to an alternate model.

## Best Possible Solution

If maintainers keep this in core, the effective routed model should drive the full model-call pipeline; otherwise continue the design in #13225 as an explicit plugin/API routing seam.

## Reproduction Assessment

Yes. Static inspection reproduces the blockers: current main builds tools, auth, provider stream functions, transcript policy, and diagnostics from `params.model`, while the PR diff only swaps the model argument on tool-result continuations.

## Solution Assessment

No. The proposed wrapper is too narrow for the stated cross-provider use case; the maintainable fix is a first-class per-call effective-model pipeline or explicit routing hook that selects all provider-bound behavior together.
