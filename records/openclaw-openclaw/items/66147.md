## Summary

Keep open for maintainer review: current main does not implement split tool-turn model routing, and the related routing-design issue remains open, but this branch is not merge-ready because it adds a public config surface while the runtime routing stays primary-model-bound for tool enablement, transport, auth, transcript policy, and fallback behavior.

## What This Changes

The branch adds a `model.tool` agent config field, resolves a secondary model, and wraps the old Pi embedded stream function so tool-continuation turns swap to that model, with focused unit tests.

## Best Possible Solution

Design the routing as an effective-model provider phase that selects transport, auth profile, fallback chain, transcript policy, tool capability, and context budget for the active continuation model before adding a documented config surface.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR installs routing after primary-model tool enablement and stream/auth setup, so the advertised cross-provider path remains primary-bound. No live integration proof was run or provided.

## Solution Assessment

No. This is a plausible prototype, but the best fix is a current-runner routing phase that resolves the effective model's provider, auth, fallback, transcript, tool, and context contracts before exposing `model.tool`.
