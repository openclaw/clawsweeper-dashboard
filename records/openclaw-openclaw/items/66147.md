## Summary

Keep open: this is a plausible feature PR for split model routing, but current main still lacks the feature and the proposed implementation has correctness, security, and proof blockers that need maintainer architecture review and author revision before merge.

## What This Changes

The PR adds a public `model.tool` config field, resolver/schema tests, and a Pi embedded runner stream wrapper intended to route tool-continuation turns to a configured alternate model.

## Best Possible Solution

Decide the core-vs-plugin API shape first; if this stays in core, the effective model should drive the whole model-call pipeline rather than only the `Model` argument in a late stream wrapper.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the diff swaps only the `Model` argument after the primary-bound model-call pipeline is prepared, and the PR body still leaves the real Ollama plus Anthropic integration test unchecked.

## Solution Assessment

No. The proposed late `StreamFn` swap is too narrow for the advertised cross-provider workflow; a maintainable solution needs an effective-model pipeline or explicit routing contract that owns transport, auth, policy, and diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route tool turns through the effective transport:** `src/agents/pi-embedded-runner/run/attempt.ts:1287-1293`
  - body: The split wrapper swaps only the `model` argument after the session stream function has already been resolved for the primary provider/model. In a split-provider setup such as Ollama primary plus Anthropic tool model, tool-continuation calls can still use primary-bound transport and credential injection instead of the configured tool provider.
  - confidence: 0.92
- **[P1] Select the tool model before the first tool call:** `src/agents/pi-embedded-runner/split-model-stream.ts:44-50`
  - body: Routing activates only when the last message is a tool or toolResult, but the first assistant turn must emit the tool call before any tool result exists. If the local primary is the model that lacks reliable tool support, the configured tool model is never reached for the main advertised workflow.
  - confidence: 0.9
- **[P2] Use current Pi package imports:** `src/agents/pi-embedded-runner/split-model-stream.ts:1-3`
  - body: The new wrapper and tests import `@mariozechner/pi-ai` and `@mariozechner/pi-agent-core`, but current main depends on the renamed `@earendil-works/pi-*` packages. These imports will not resolve after rebasing onto current main.
  - confidence: 0.88
- **[P2] Refresh the generated config contract:** `src/config/zod-schema.agent-model.ts:8`
  - body: Adding `tool` to the strict agent model schema changes the public config surface, but the generated config baseline/docs artifacts are not updated in this PR. The config docs check and user-facing contract would drift.
  - confidence: 0.78
