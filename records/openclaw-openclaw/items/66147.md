## Summary

Keep this PR open. Current main still has no `model.tool` support, so the contribution is not obsolete, but the proposed patch is not merge-ready: it adds a public config key while routing only swaps the `Model` argument after primary-provider transport, auth, provider-specific wrappers, transcript policy, and tool enablement have already been selected. Copilot, Greptile, and earlier ClawSweeper review comments independently flagged the same primary-model-bound behavior.

## What This Changes

The PR adds a `model.tool` config field, resolver/schema support, and a Pi embedded runner stream wrapper/tests intended to route tool-continuation turns to a configured alternate model.

## Best Possible Solution

Keep the PR open while maintainers decide whether split model routing belongs in core `agents.defaults.model` or behind a new plugin/API seam such as #13225. If it stays in core, the effective per-call model should drive the whole model-call pipeline rather than only the `Model` argument passed to one stream wrapper.

## Reproduction Assessment

Yes for the blocking defects. Static current-main proof shows tool enablement, provider stream registration, auth/wrapper setup, transcript policy, and diagnostics are derived from `params.model`; the PR's supplied diff then adds a wrapper that swaps only the call model on tool-result continuation turns.

## Solution Assessment

No. The proposed implementation is too narrow for the stated cross-provider use case; the safer direction is a per-call effective-model pipeline or explicit routing hook that selects transport, auth, wrappers, tool capability, and diagnostics together.

If the primary local model lacks reliable tool support, no tool result exists and the configured tool model is never exercised for the main advertised workflow.
  - confidence: 0.9
