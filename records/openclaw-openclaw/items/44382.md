## What This Changes

The PR adds named TUI color themes with CLI/env/runtime selection, docs/tests, Swift protocol model updates, and a root dependency override/lockfile change.

## Best Possible Solution

Keep the TUI theme work scoped to TUI files, preserve current light-background detection semantics, make runtime theme changes update existing components and chrome through stable theme objects or explicit refresh hooks, remove unrelated protocol/dependency changes, then require real terminal proof before merge.

Full review comments:

- **[P2] Keep existing theme instances live after switching:** `src/tui/theme/theme.ts:223-235`
  - body: `applyPalette()` replaces the exported theme objects, but existing TUI components keep the objects they were constructed with. `SearchableSelectList` stores the theme argument in a field and message/tool Markdown instances receive `markdownTheme` during construction, so after `/theme` mounted widgets can keep the old colors while new widgets use the new palette.
  - confidence: 0.9
- **[P2] Refresh already-rendered chrome after theme changes:** `src/tui/tui-command-handlers.ts:470-473`
  - body: The `/theme` path applies the palette and only requests a render, but header/footer/status strings were already converted to ANSI-colored text by `theme.header()` and `theme.dim()`. Repainting those same strings leaves the chrome partially themed until another code path recomputes it.
  - confidence: 0.86
- **[P2] Preserve 256-color COLORFGBG detection:** `src/tui/theme/detect.ts:29-31`
  - body: The new detector only treats background indexes 7 and 15 as light. Current main handles 16-255 by grayscale/color-cube luminance, so light 256-color terminals such as `COLORFGBG=0;231` now fall back to `dark`, recreating the low-contrast startup problem this PR is trying to fix.
  - confidence: 0.91
