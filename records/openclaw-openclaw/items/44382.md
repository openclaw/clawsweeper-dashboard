## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Remove the root openclaw override from this PR:** `package.json:447`
  - body: This TUI theme branch adds a root `openclaw` override and the lockfile resolves workspace package importers to a published package instead of local workspace links. That changes supply-chain inputs and validation scope in an unrelated feature PR; keep dependency/GHSA work separate.
  - confidence: 0.93
- **[P2] Retheme the editor border explicitly:** `src/tui/tui.ts:532-540`
  - body: `Editor` copies `theme.borderColor` into `this.borderColor` during construction, so the getter proxy still leaves the input border on the old color after `/theme` switches. Use an explicit editor retheme path or a stable border function that delegates at call time.
  - confidence: 0.9
- **[P2] Keep mounted theme objects stable across switches:** `src/tui/theme/theme.ts:223-235`
  - body: `applyPalette()` replaces exported markdown/list/editor theme objects, but mounted Markdown and select-list components keep the object they received at construction. After `/theme`, existing chat content and overlays can render with stale colors; mutate stable delegating objects or rebuild/retheme those components.
  - confidence: 0.9
- **[P2] Preserve 256-color COLORFGBG detection:** `src/tui/theme/detect.ts:28-31`
  - body: The new detector only returns `light` for background indices 7 and 15, so 256-color light terminals such as `COLORFGBG=0;231` or `0;255` fall back to dark. Current main handles the 0-255 range with luminance logic; port that instead of narrowing the heuristic.
  - confidence: 0.88
- **[P2] Refresh cached TUI chrome after theme changes:** `src/tui/tui-command-handlers.ts:469-473`
  - body: The `/theme` handler calls `setTheme()` and requests a render, but header/footer/status strings have already been formatted with ANSI colors. Refresh those cached strings when applying, selecting, or canceling themes so the shell does not stay partially themed.
  - confidence: 0.84
- **[P2] Sync legacy palette exports with active theme:** `src/tui/theme/theme.ts:90`
  - body: `setTheme()` changes `theme`, but `lightMode` and `palette` remain fixed from startup detection, so consumers can see contradictory state such as active Dracula with the legacy light palette. Update the compatibility exports or remove reliance on them before exposing runtime switching.
  - confidence: 0.8
- **[P3] Normalize explicit theme names before lookup:** `src/tui/theme/detect.ts:11-13`
  - body: `OPENCLAW_THEME` is returned verbatim and palette lookup is exact, so mixed-case values like `LiGhT` or `Dracula` fall back to dark even though current main treats the env override case-insensitively. Normalize user input before resolving the palette.
  - confidence: 0.78
