Keep open. Current main only ships the narrower light/dark terminal readability fix; the five-theme TUI surface is still absent, and this PR is not merge-ready because it is stale, mixes unrelated protocol/package-resolution edits, and still has blocking theme-regression and supply-chain findings.

## What This Changes

The PR adds a five-palette TUI theme system with startup/runtime selection, CLI and docs updates, theme tests, plus unrelated Swift protocol regeneration and package-resolution changes.

## Best Possible Solution

Land a narrow TUI-only theme change, if maintainers want it, rebased onto current main's light/dark detection and docs. It should preserve existing detection semantics, update all live/cached TUI surfaces consistently on runtime switches, and leave protocol regeneration plus package-resolution/security fixes to separate owner-reviewed PRs.

## Reproduction Assessment

Yes. Without running tests, the regression path is clear from code: import the PR's detectThemeName with OPENCLAW_THEME=LiGhT or COLORFGBG=0;231/0;255 and it resolves to an unknown/dark path, while current main has tests expecting those cases to remain light.

## Solution Assessment

No. The product idea is useful, but this PR is not the best merge shape as-is because it combines a broad UI feature with unrelated protocol and supply-chain edits and regresses current light-terminal detection behavior; a smaller TUI-only branch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Remove the global openclaw override:** `package.json:447`
  - body: Adding a repo-wide pnpm override for `openclaw` makes workspace packages resolve the published npm package instead of the local workspace copy, as shown by the lockfile entries for packages/clawdbot and packages/moltbot. That breaks local source testing and introduces unrelated supply-chain risk in a TUI theme PR.
  - confidence: 0.9
- **[P2] Preserve 256-color light background detection:** `src/tui/theme/detect.ts:28-31`
  - body: `detectThemeName()` only treats COLORFGBG backgrounds 7 and 15 as light. Current main handles 256-color light backgrounds like `0;231` and `0;255`, so users on those terminals would be auto-switched back to the dark palette.
  - confidence: 0.91
- **[P2] Normalize explicit theme overrides:** `src/tui/theme/detect.ts:11-13`
  - body: The env value is returned verbatim, but palette lookup is exact. Values such as `OPENCLAW_THEME=LiGhT` or `Dracula` therefore fall back to dark even though current main treats OPENCLAW_THEME case-insensitively for light/dark.
  - confidence: 0.86
- **[P2] Refresh the UI after direct theme changes:** `src/tui/tui-command-handlers.ts:480-487`
  - body: The `/theme <name>` branch calls `setTheme(args)` and writes a system message but never requests a render. Direct runtime theme switches can therefore leave the screen unchanged until another event happens.
  - confidence: 0.84
- **[P2] Keep theme objects stable for mounted components:** `src/tui/theme/theme.ts:223-235`
  - body: `applyPalette()` replaces `markdownTheme`, `selectListTheme`, `editorTheme`, and related objects. Components such as markdown messages and searchable select lists store the object passed at construction, so runtime theme changes leave already-mounted UI in the old palette.
  - confidence: 0.82
