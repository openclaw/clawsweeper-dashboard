## Summary

Keep open for maintainer handling. Current main has shipped light/dark TUI contrast support, but it does not implement this PR's named theme registry, CLI flag, or `/theme` command; the PR also still has definite merge blockers, including a supply-chain-sensitive package override and regressions in shipped terminal-background detection behavior.

## What This Changes

The branch adds a five-palette TUI theme system with CLI/env/runtime selection and docs/tests, regenerates Swift protocol models, and changes pnpm package-resolution metadata.

## Best Possible Solution

If maintainers want this feature, land it as a narrow TUI-only change rebased onto current main: preserve the existing light/dark and 256-color detection contract, add named palettes and documented CLI/env/runtime selection, update all mounted TUI chrome and components consistently during runtime switches, and split protocol regeneration plus dependency/security changes into separate owner-reviewed work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Remove the global openclaw override:** `package.json:447`
  - body: Adding a repo-wide `pnpm.overrides.openclaw` makes workspace packages resolve the published npm package instead of the local workspace copy, as shown by the lockfile changes for `packages/clawdbot` and `packages/moltbot`. That breaks local source validation and introduces unrelated supply-chain risk in a TUI theme PR.
  - confidence: 0.9
- **[P2] Preserve 256-color light background detection:** `src/tui/theme/detect.ts:28-31`
  - body: `detectThemeName()` only treats `COLORFGBG` backgrounds 7 and 15 as light. Current main handles 256-color light backgrounds like `0;231` and `0;255`, so users on those terminals would be auto-switched back to the dark palette.
  - confidence: 0.91
- **[P2] Normalize explicit theme overrides:** `src/tui/theme/detect.ts:11-13`
  - body: The env value is returned verbatim, but palette lookup is exact. Values such as `OPENCLAW_THEME=LiGhT` or `Dracula` therefore fall back to dark even though current main treats `OPENCLAW_THEME` case-insensitively for light/dark.
  - confidence: 0.86
- **[P2] Refresh chrome after runtime theme changes:** `src/tui/tui-command-handlers.ts:480-487`
  - body: The `/theme` handler changes the active theme but does not rerun the header, footer, and status text builders that bake ANSI color functions into existing `Text` nodes. A render alone can leave those chrome sections in the previous palette until another state update recomputes them.
  - confidence: 0.82
- **[P2] Keep mounted theme objects stable:** `src/tui/theme/theme.ts:223-235`
  - body: `applyPalette()` replaces exported theme objects such as markdown and select-list themes. Existing markdown messages and selector instances store the object passed at construction, so runtime theme changes can leave already-mounted UI in the old palette.
  - confidence: 0.82
