  - body: This TUI theme branch adds a root `openclaw` override and lockfile rewrites that change local workspace links to registry `openclaw@2026.3.11` for workspace packages.   - body: The new detector only returns `light` for background indices 7 and 15, but current main handles 0-255 `COLORFGBG` values and has tests for light 256-color backgrounds such as 231 and 255. Users on those terminals would regress to the dark palette by default.
  - confidence: 0.9
- **[P2] Update the cached editor border on theme switches:** `src/tui/tui.ts:526-540`
  - body: The proxy passed to `CustomEditor` does not update pi-tui’s cached `borderColor`, because the Editor constructor copies `theme.borderColor` into a field and render uses that field. Runtime `/theme` can still leave the input border in the old palette.
  - confidence: 0.88
- **[P2] Keep mounted theme objects stable across switches:** `src/tui/theme/theme.ts:223-235`
  - body: `applyPalette()` replaces exported markdown/list/editor theme objects, while existing Markdown and selector instances store the object passed at construction. Runtime switching can leave mounted messages and overlays with stale colors; use stable delegating objects or retheme/rebuild those components.
  - confidence: 0.86
- **[P2] Refresh static chrome after applying a theme:** `src/tui/tui-command-handlers.ts:469-473`
  - body: The `/theme` handler calls `setTheme()` and requests a render, but header/footer/status `Text` nodes can already contain ANSI-formatted strings. Recompute those strings on direct apply and picker select so users do not see mixed old and new colors.
  - confidence: 0.78
- **[P3] Normalize explicit theme names before lookup:** `src/tui/theme/detect.ts:11-13`
  - body: `OPENCLAW_THEME` and `/theme` arguments are looked up exactly, so mixed-case values like `LiGhT` or `Dracula` can fall back to dark even though current main accepts mixed-case `OPENCLAW_THEME=LiGhT` for the light theme. Normalize user-provided names before resolving palettes.
  - confidence: 0.76
