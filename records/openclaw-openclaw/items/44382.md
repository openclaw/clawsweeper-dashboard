Full review comments:

- **[P1] Remove the root openclaw override from this PR:** `package.json:447`
  - body: This TUI feature branch adds a root `openclaw` override and rewrites lockfile workspace importers to a published package version, changing supply-chain inputs outside the stated theming work. Split that dependency/security change out of this PR.
  - confidence: 0.93
- **[P2] Retheme the editor border explicitly:** `src/tui/tui.ts:532-540`
  - body: The proxy does not update state copied during `Editor` construction: pi-tui stores `theme.borderColor` in `this.borderColor`, and render uses that cached function. After `/theme`, the input editor border can stay on the old palette unless the editor is rebuilt or its cached border function is updated.
  - confidence: 0.9
- **[P2] Keep mounted theme objects stable across switches:** `src/tui/theme/theme.ts:223-235`
  - body: `applyPalette()` replaces exported markdown/list/editor theme objects, but existing Markdown and select-list components keep the object they received at construction. After `/theme`, already-mounted messages and overlays can render with stale colors; use stable delegating objects or retheme/rebuild those components.
  - confidence: 0.9
- **[P2] Preserve 256-color COLORFGBG detection:** `src/tui/theme/detect.ts:28-31`
  - body: The new detector only returns light for background indices 7 and 15, while current main handles 0-255 backgrounds with luminance logic. Light 256-color terminals such as `COLORFGBG=0;231` or `0;255` would fall back to dark, regressing shipped light-background behavior.
  - confidence: 0.88
- **[P2] Refresh cached TUI chrome after theme changes:** `src/tui/tui-command-handlers.ts:469-473`
  - body: The `/theme` handler calls `setTheme()` and requests a render, but header/footer/status `Text` nodes already contain ANSI-formatted strings. Recompute those strings on direct apply, picker select, and cancel so users do not see mixed old and new colors.
  - confidence: 0.84
- **[P2] Sync legacy palette exports with the active theme:** `src/tui/theme/theme.ts:90`
  - body: `setTheme()` changes the active theme, but `lightMode` and `palette` remain fixed from startup detection. Consumers can see contradictory state such as active Dracula with the legacy light palette; update compatibility exports or avoid exposing runtime switching through stale values.
  - confidence: 0.8
- **[P3] Normalize explicit theme names before lookup:** `src/tui/theme/detect.ts:11-13`
  - body: `OPENCLAW_THEME` is returned verbatim and palette lookup is exact, so mixed-case values like `LiGhT` or `Dracula` fall back to dark even though current main treats the env override case-insensitively.
  - confidence: 0.78
