## Summary

Keep open for maintainer review: current main already shipped the core light/dark readability fix, but this PR still proposes broader runtime theme selection and has blocking runtime-theming regressions, unrelated package-resolution churn, and missing real behavior proof.

## What This Changes

The PR adds a multi-palette TUI theme system with startup/env/CLI/runtime selection, docs/tests, regenerated Swift protocol models, and package/lockfile dependency overrides.

## Best Possible Solution

Land one canonical TUI theming design that preserves current light/dark detection, keeps mounted theme objects and cached chrome in sync, documents the UX, and splits unrelated protocol or dependency changes.

## Reproduction Assessment

Yes, for the blocking review findings via source inspection. Current main stores preformatted chrome strings and tested 256-color detection, while the PR only calls setTheme/requestRender and narrows detection behavior; I did not run a live TUI session in this read-only review.

## Solution Assessment

No. The theme UX direction may be useful, but this branch should be rebased onto current light/dark detection, keep theme references and chrome refresh coherent, and remove unrelated package/protocol churn before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve 256-color background detection:** `src/tui/theme/detect.ts:30-31`
  - body: Current main treats 256-color light backgrounds such as `COLORFGBG=0;255` and `0;231` as light, with tests covering those cases. This new detector only returns `light` for ANSI bg 7 or 15, so users on common 256-color light terminals will start in the dark palette again.
  - confidence: 0.9
- **[P2] Keep mounted theme objects live after setTheme:** `src/tui/theme/theme.ts:223-235`
  - body: `applyPalette()` replaces `markdownTheme`, `selectListTheme`, `settingsListTheme`, and `searchableSelectListTheme` with new objects. Components that were already constructed with the old objects will keep rendering old colors after `/theme`, so the UI can become partially themed even though the editor proxy was added.
  - confidence: 0.88
- **[P2] Refresh cached chrome after theme changes:** `src/tui/tui-command-handlers.ts:469-473`
  - body: The `/theme` handler calls `setTheme()` and `tui.requestRender()`, but the header, footer, and idle status are already stored as ANSI-colored `Text` strings. Without calling the TUI refresh hooks or making those sections lazily themed, runtime switches leave old colors in the chrome until another state update happens.
  - confidence: 0.87
- **[P2] Remove the root openclaw package override:** `package.json:447`
  - body: Adding a root override for `openclaw` forces workspace consumers in the lockfile to resolve a published `openclaw` package instead of the local checkout. That makes local validation exercise stale published code and introduces unrelated supply-chain churn in a TUI theme PR; split or drop this change.
  - confidence: 0.91
