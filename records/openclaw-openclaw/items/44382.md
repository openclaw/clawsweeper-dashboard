## Summary

Keep open: current main still lacks the PR's five named TUI themes, `--theme` flag, and runtime `/theme` command, so it is not obsolete. The branch is useful directionally, but it is not merge-ready because it mixes TUI theming with unrelated generated Swift and package-resolution changes, introduces a security-sensitive `openclaw` override, and regresses current light-terminal detection behavior.

## What This Changes

The branch adds a five-palette TUI theme system with CLI/env/runtime selection, picker docs/tests, generated Swift protocol updates, and pnpm package-resolution changes.

## Best Possible Solution

If maintainers want this feature, land a narrow TUI-only theme system rebased onto current main: keep current light/dark auto-detection semantics, add named palettes and documented CLI/env/runtime selection, update all cached TUI chrome and mounted components consistently on runtime switches, and split protocol regeneration plus package/security changes into separate owner-reviewed PRs.

Full review comments:

- **[P1] Remove the global openclaw override:** `package.json:447`
  - body: Adding a repo-wide `pnpm.overrides.openclaw` makes workspace packages resolve the published npm package instead of the local workspace copy, as shown by the lockfile changes for `packages/clawdbot` and `packages/moltbot`. That breaks local source validation and introduces unrelated supply-chain risk in a TUI theme PR.
  - confidence: 0.9
- **[P2] Preserve 256-color light background detection:** `src/tui/theme/detect.ts:28-31`
  - body: `detectThemeName()` only treats `COLORFGBG` backgrounds 7 and 15 as light. Current main handles 256-color light backgrounds like `0;231` and `0;255`, so users on those terminals would be auto-switched back to the dark palette.
  - confidence: 0.91
- **[P2] Normalize explicit theme overrides:** `src/tui/theme/detect.ts:11-13`
  - body: The env value is returned verbatim, but palette lookup is exact. Values such as `OPENCLAW_THEME=LiGhT` or `Dracula` therefore fall back to dark even though current main treats `OPENCLAW_THEME` case-insensitively for light/dark.
  - confidence: 0.86
- **[P2] Refresh chrome after runtime theme changes:** `src/tui/tui-command-handlers.ts:480-487`
  - body: The `/theme` handler changes the active theme but does not rerun the header, footer, and status text builders that bake ANSI color functions into existing `Text` nodes. A render alone can leave those chrome sections in the previous palette until another unrelated state update recomputes them.
  - confidence: 0.82
- **[P2] Keep theme objects stable for mounted components:** `src/tui/theme/theme.ts:223-235`
  - body: `applyPalette()` replaces exported theme objects such as markdown and select-list themes. Existing markdown messages and selector instances store the object passed at construction, so runtime theme changes can leave already-mounted UI in the old palette.
  - confidence: 0.82
