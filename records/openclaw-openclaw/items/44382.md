## Summary

Keep open. The named TUI theme system is not implemented on current main and may be useful, but this branch is not merge-ready because it regresses existing terminal-background detection, only partially updates runtime-themed UI, lacks real behavior proof, and includes unrelated package-resolution/protocol changes.

## What This Changes

The branch adds named TUI palettes, CLI/env/runtime theme selection, and TUI docs/tests, while also regenerating Swift protocol models and adding a pnpm `openclaw` override.

## Best Possible Solution

Land a narrow, rebased TUI-only theme change if maintainers want this feature: preserve current detection contracts, keep mounted theme objects stable, repaint chrome on switches, remove unrelated package/protocol changes, and coordinate with https://github.com/openclaw/openclaw/pull/38982.

## Reproduction Assessment

Yes. Source inspection of the PR branch against current main shows reproducible regressions for mixed-case `OPENCLAW_THEME`, 256-color light `COLORFGBG` values, mounted runtime theme objects, and workspace package resolution.

## Solution Assessment

No. The theme direction may be useful, but the current PR is too broad and should be split or replaced with a TUI-only implementation that preserves existing contracts and removes unrelated dependency/protocol changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Remove the global openclaw override:** `package.json:447`
  - body: Adding `pnpm.overrides.openclaw` rewrites workspace consumers from local links to a published `openclaw` package in the lockfile. That is unrelated to TUI theming and weakens local validation in a supply-chain-sensitive path.
  - confidence: 0.9
- **[P2] Preserve 256-color light background detection:** `src/tui/theme/detect.ts:28-31`
  - body: `detectThemeName()` only returns `light` for backgrounds 7 and 15, so light 256-color values such as `COLORFGBG=0;231` or `0;255` now fall back to dark even though current main handles them as light.
  - confidence: 0.92
- **[P2] Normalize explicit theme overrides:** `src/tui/theme/detect.ts:10-13`
  - body: The env value is returned verbatim, but palette lookup is exact. Mixed-case values such as `OPENCLAW_THEME=LiGhT` or `Dracula` therefore fall back to dark, regressing current main's case-insensitive light/dark override behavior.
  - confidence: 0.88
- **[P2] Repaint static chrome after theme changes:** `src/tui/tui-command-handlers.ts:481-487`
  - body: The `/theme` handler changes the active palette and requests a render, but header, footer, and status `Text` nodes already contain ANSI-colored strings. They can remain in the previous palette until another state update rebuilds them.
  - confidence: 0.82
- **[P2] Keep mounted theme objects stable:** `src/tui/theme/theme.ts:223-235`
  - body: `applyPalette()` replaces exported theme objects, while components such as markdown messages and searchable selectors store the theme object they received at construction. Already-mounted UI can remain in the old palette after a runtime switch.
  - confidence: 0.84
- **[P2] Keep legacy palette exports in sync:** `src/tui/theme/theme.ts:83-90`
  - body: `lightMode` and `palette` stay fixed from import-time light/dark detection even after `setTheme()` applies a named theme. Consumers of those legacy exports can render contradictory colors after startup or runtime switches.
  - confidence: 0.78
