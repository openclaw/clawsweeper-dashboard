## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Drop the OpenClaw package override from this TUI PR:** `package.json:447`
  - body: The added root `openclaw` override is unrelated to TUI theming, and the lockfile rewrites workspace packages from local workspace links to a published OpenClaw package. That can make builds/tests validate an older registry copy instead of the checked-out source tree, so this dependency/security remediation needs to be removed or handled separately.
  - confidence: 0.92
- **[P2] Preserve current 256-color background detection:** `src/tui/theme/detect.ts:29-31`
  - body: The new detector only returns `light` for ANSI backgrounds 7 or 15. Current main already handles 256-color light backgrounds such as `COLORFGBG=0;255` and `0;231`; replacing that logic would put those users back on the dark palette by default.
  - confidence: 0.9
- **[P2] Keep mounted components on the active theme:** `src/tui/tui-command-handlers.ts:469-474`
  - body: The `/theme` handler calls `setTheme()` and requests a render, but `setTheme()` rebuilds and reassigns exported objects like `markdownTheme` and `searchableSelectListTheme`. Existing markdown, tool, and selector components store those objects, and header/footer/status text is already ANSI-formatted, so runtime switching can leave visible UI sections in the old palette.
  - confidence: 0.87
- **[P2] Normalize explicit theme names before palette lookup:** `src/tui/theme/detect.ts:10-12`
  - body: `detectThemeName()` returns `OPENCLAW_THEME` verbatim, while the palette lookup is exact. Current main treats `OPENCLAW_THEME=LiGhT` case-insensitively; this PR would fall back to dark and lose the explicit light-theme override.
  - confidence: 0.84
