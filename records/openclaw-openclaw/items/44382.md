## Summary

Keep this PR open: the TUI theme direction is still useful and only partly covered by current main, but the branch is not merge-ready because it mixes unrelated package/protocol churn with the feature and still has source-provable runtime theming regressions.

## What This Changes

The PR adds built-in TUI palettes, theme auto-detection, `OPENCLAW_THEME`, `openclaw tui --theme`, a runtime `/theme` picker, docs, Swift protocol regeneration, and package/lockfile override changes.

## Best Possible Solution

Rebase into a focused TUI theming PR that preserves current main's light/dark detection, removes unrelated protocol and package changes, keeps runtime theme objects stable for mounted components, and includes redacted real terminal proof.

Yes for the review findings: source inspection shows captured theme-object references while the PR reassigns theme exports, and the diff shows workspace package resolution being changed through an unrelated override. I did not run a live TUI session because this is a read-only review.

## Solution Assessment

No. The product direction may be worthwhile, but this branch is not the best merge shape because it combines TUI UX work with unrelated dependency/protocol churn and does not fully solve runtime theme propagation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Drop the OpenClaw package override from this TUI PR:** `package.json:447`
  - body: The added root `openclaw` override is unrelated to TUI theming, and the lockfile rewrites workspace packages such as `packages/clawdbot` and `packages/moltbot` from `link:../..` to a published `openclaw@2026.3.11`. That can make local packages build or test against an older registry copy instead of the checked-out workspace, so this dependency/security remediation needs to be removed or handled separately.
  - confidence: 0.92
- **[P2] Keep mounted components on the active theme:** `src/tui/tui-command-handlers.ts:469-474`
  - body: `/theme` calls `setTheme()` and requests a render, but `setTheme()` rebuilds and reassigns objects like `markdownTheme` and `searchableSelectListTheme`. Existing components store those objects or pass them into `HyperlinkMarkdown` at construction time, so already-mounted chat content and selectors can remain in the old palette after a runtime switch.
  - confidence: 0.87
- **[P2] Preserve current 256-color background detection:** `src/tui/theme/detect.ts:29-31`
  - body: The new detector only treats ANSI backgrounds 7 and 15 as light. Current main already tests 256-color light backgrounds such as `COLORFGBG=0;255` and `0;231`; replacing that behavior would put those users back on the dark palette by default.
  - confidence: 0.9
