## Summary

Keep this open. Current main still lacks Fireworks-owned compatibility metadata for the default Kimi router, and the OpenAI-compatible tool conversion path still normalizes schemas without model compat, so the reported `not` schema can still reach Fireworks unchanged. The related open PR #75476 is relevant, but it is not merged into current main and should be reviewed or expanded instead of closing this issue now.

## What This Changes

Request that Fireworks Kimi tool schemas strip unsupported JSON Schema keywords, especially `not`, by declaring model compat metadata and using the existing schema sanitizer before provider dispatch.

## Best Possible Solution

Land one coherent fix that keeps Fireworks limits in the Fireworks plugin manifest, threads selected model compat through OpenAI-compatible tool-schema normalization, and adds focused regression coverage for Fireworks/Kimi and generic OpenAI-completions schema conversion.

## Reproduction Assessment

Yes for a source-level reproduction: a tool schema containing `{"not": {}}` passed through the current OpenAI-compatible conversion has no Fireworks compat metadata and no compat argument to the normalizer. The live provider request was not rerun, but the reported 400 matches the preserved schema path.

## Solution Assessment

Yes, the requested direction is the maintainable fix when paired with the missing OpenAI-completions plumbing: provider metadata should describe Fireworks schema limits, and the generic schema normalizer should consume that metadata.
