## Summary

Keep this open. Current main still lacks the Fireworks catalog compatibility metadata, and the OpenAI-completions tool conversion path still does not forward model compat into schema normalization, so schemas containing `not` can still reach Fireworks unchanged. The related open PR #75476 is relevant to the same symptom, but the current main checkout is not fixed yet and the Fireworks manifest row still needs to be covered.

## What This Changes

The issue requests provider-owned Fireworks/Kimi K2.5 Turbo compatibility metadata so unsupported JSON Schema keywords such as `not` are stripped before tool schemas are sent to Fireworks.

## Best Possible Solution

Land one coherent fix that declares Fireworks-owned `compat.unsupportedToolSchemaKeywords` for the affected Kimi catalog rows, forwards selected model compat through OpenAI-completions tool-schema normalization, and adds focused Fireworks/catalog plus OpenAI schema regression coverage without hard-coding Fireworks behavior in core.

## Reproduction Assessment

Yes for the source-level path: current main has a Fireworks Kimi model row with no unsupported-keyword compat, and the OpenAI-completions conversion calls the normalizer without model compat. The live Fireworks 400 was not re-run in this read-only review, but the reported error matches the preserved `not` schema path.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix when combined with the related OpenAI-completions plumbing: provider metadata should describe Fireworks limits, and generic schema normalization should consume that metadata. A provider-specific core branch would be the less desirable alternative.
