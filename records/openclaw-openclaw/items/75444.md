## Summary

Keep open. Current main has the generic compatibility plumbing to strip provider-unsupported schema keywords, but the bundled Fireworks Kimi catalog rows do not declare any `compat.unsupportedToolSchemaKeywords`, so `not` and the other reported keywords are not removed before a Fireworks request. The requested fix is narrow and provider-owned.

## What This Changes

Request to add Fireworks Kimi tool-schema compatibility metadata so unsupported JSON Schema keywords such as `not` are stripped before sending tools to Fireworks.

## Best Possible Solution

Add a provider-owned Fireworks compatibility patch, preferably in `extensions/fireworks`, so the bundled Kimi rows and dynamic Fireworks models carry `unsupportedToolSchemaKeywords` before tool schemas are built. Keep the fix out of core provider special-cases and cover it with focused catalog/tool-schema tests.

## Reproduction Assessment

Yes. With a Fireworks key, configure `fireworks/accounts/fireworks/routers/kimi-k2p5-turbo` and any tool schema containing `not`; current main will not strip that keyword because the resolved model compat set is empty for the bundled Fireworks row.

## Solution Assessment

Yes for the narrow direction: model/provider compat metadata is the maintainable seam already used by the schema normalizer. The broader keyword list should be added only to the extent it is backed by Fireworks docs or live confirmation.
