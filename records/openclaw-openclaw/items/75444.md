## Summary

Keep open. Current main still lacks Fireworks catalog compatibility metadata for the affected Kimi model, and the OpenAI-completions tool-schema conversion still calls the normalizer without model compat, so the reported `not` schema can still reach Fireworks.

## What This Changes

The issue requests Fireworks model compatibility metadata so Kimi K2.5 Turbo tool schemas strip unsupported JSON Schema keywords such as `not` before provider requests.

## Best Possible Solution

Land one coherent fix that declares Fireworks-owned `compat.unsupportedToolSchemaKeywords` for the affected Kimi catalog rows, forwards selected model compat through OpenAI-completions tool-schema normalization, and adds focused Fireworks catalog plus OpenAI schema regression coverage without hard-coding Fireworks behavior in core.

## Reproduction Assessment

Yes for the source-level path: current main has the Fireworks Kimi row without unsupported-keyword compat, and the OpenAI-completions serializer still does not pass model compat into schema normalization. The live Fireworks 400 was not rerun in this read-only review, but the reported error matches the preserved `not` schema path.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix when paired with the OpenAI-completions compat plumbing. Provider metadata should describe Fireworks limits, and generic schema normalization should consume that metadata rather than adding a Fireworks-specific core branch.
