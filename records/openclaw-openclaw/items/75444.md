## Summary

Keep this open. Current main still lacks Fireworks catalog compatibility metadata for `accounts/fireworks/routers/kimi-k2p5-turbo`, and the OpenAI-family tool-schema conversion path still calls the normalizer without model compat, so a schema containing `not` can still reach Fireworks unchanged. The related PR #75476 is relevant but remains an open implementation candidate, not an implemented fix on main.

## What This Changes

Request to mark the Fireworks Kimi model catalog row with unsupported tool-schema keyword compatibility metadata so keywords like `not` are stripped before Fireworks requests.

## Best Possible Solution

Land one coherent fix that declares Fireworks-owned `compat.unsupportedToolSchemaKeywords` for the affected Kimi catalog row, forwards selected model compat through OpenAI-family tool-schema normalization, and adds focused regression coverage without hard-coding Fireworks behavior in core.

## Reproduction Assessment

Yes for a source-level reproduction path: configure the Fireworks Kimi model with a tool schema containing `{"not": {}}`; current main has no Fireworks compat metadata and does not pass model compat into the OpenAI tool-schema normalizer. The live provider request was not re-run during this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution when paired with the OpenAI-completions plumbing from #75476. Provider-owned catalog metadata should describe Fireworks limits, and generic schema normalization should consume that metadata.
