## Summary

Keep open. Current main still lacks Fireworks `compat.unsupportedToolSchemaKeywords` metadata for the affected Kimi router and still drops model compat in the OpenAI-family tool-schema conversion path, so a schema containing `not` can still reach Fireworks unchanged. The related PR #75476 is an open implementation candidate, not an implemented fix on main.

## What This Changes

Report requests Fireworks Kimi K2.5 Turbo tool-schema compatibility so unsupported JSON Schema keywords such as `not` are stripped before Fireworks requests.

## Best Possible Solution

Land one coherent fix that declares Fireworks-owned `compat.unsupportedToolSchemaKeywords` for the affected Kimi catalog row, threads selected model compat through OpenAI Responses and Chat Completions tool-schema normalization, and adds focused regression coverage plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure the Fireworks Kimi model with a tool schema containing `{ "not": {} }`; current main has no Fireworks compat metadata and does not pass model compat into the OpenAI tool normalizer, so the keyword remains in the request path.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix when paired with the OpenAI-completions plumbing from #75476: provider-specific limits belong in the Fireworks manifest, while core should consume the existing generic compat seam without hard-coding Fireworks behavior.
