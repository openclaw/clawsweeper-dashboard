## Summary

Keep open. Current main still has the reported Fireworks/OpenAI-completions tool-schema gap, and the related open PR is an implementation candidate rather than an implemented fix. Because the same author also opened the narrower canonical root-cause issue, this should not be closed independently before that path resolves.

## What This Changes

The issue requests Fireworks `kimi-k2p5-turbo` tool-schema requests strip unsupported JSON Schema keywords, especially `not`, by declaring model compat and ensuring the OpenAI-completions schema path honors it.

## Best Possible Solution

Land one coherent fix that declares Fireworks-owned `compat.unsupportedToolSchemaKeywords` for the affected Kimi catalog row, threads selected model compat through OpenAI Responses and Chat Completions tool-schema normalization, and adds focused regression coverage plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure the Fireworks Kimi model with a tool schema containing `{ "not": {} }`; current main has no Fireworks compat metadata and does not pass model compat into the OpenAI tool normalizer, so the keyword can still reach the provider.

## Solution Assessment

Yes. The narrow maintainable solution is to use the existing generic compat seam: provider-specific limits belong in the Fireworks manifest, while core should pass model compat through the OpenAI-completions tool-schema normalization path.
