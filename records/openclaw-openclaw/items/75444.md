## Summary

Keep open. Current main still lacks the Fireworks Kimi K2.5 Turbo unsupported-keyword compat metadata, and the OpenAI-compatible tool conversion path still calls the schema normalizer without selected-model compat. Related PR #75476 is relevant but remains open, so the issue is not implemented on main yet.

## What This Changes

The issue requests Fireworks/Kimi K2.5 Turbo tool-schema compatibility so unsupported JSON Schema keywords such as `not` are stripped before Fireworks OpenAI-compatible requests.

## Best Possible Solution

Keep this open until #75476 or an equivalent fix lands with both selected-model compat flowing through the OpenAI-compatible schema path and Fireworks catalog metadata declaring the unsupported keywords, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: select the bundled Fireworks Kimi K2.5 Turbo model and provide a tool schema containing `not`; current main has no Fireworks keyword compat, so the generic stripper has no keyword set to remove.

## Solution Assessment

Yes, the requested direction is the narrow maintainable path: use provider-owned model `compat.unsupportedToolSchemaKeywords` and the existing generic schema stripper rather than adding Fireworks-specific branching in core.
