## Summary

Keep open. Current main still lacks Fireworks `compat.unsupportedToolSchemaKeywords` metadata for the affected Kimi catalog row, and the OpenAI-completions tool conversion path still calls schema normalization without model compat, so a `not` schema can still reach Fireworks unchanged. Related #75467 and open PR #75476 track the same root symptom, but they are not merged into current main and should be treated as the implementation path rather than a close reason.

## What This Changes

The issue requests adding Fireworks unsupported tool-schema keyword compatibility metadata and ensuring tool schemas containing keywords such as `not` are stripped before requests to `fireworks/accounts/fireworks/routers/kimi-k2p5-turbo`.

## Best Possible Solution

Land one coherent fix that declares Fireworks-owned `compat.unsupportedToolSchemaKeywords` for the affected Kimi catalog rows, threads selected model compat through OpenAI-completions tool-schema normalization, and adds focused regression tests without hard-coding Fireworks-specific branches in core.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main's Fireworks Kimi row has no unsupported-keyword compat, and the OpenAI-completions conversion calls the normalizer without `modelCompat`, so a schema like `{"not": {}}` is preserved before dispatch.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution when combined with the OpenAI-completions plumbing. Provider metadata should describe Fireworks limits, and the generic schema normalizer should consume that metadata; a Fireworks-specific core branch would be less maintainable.
