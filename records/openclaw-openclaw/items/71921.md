## Summary

Keep open. Current main still preserves explicit Bedrock model entries that have no matching implicit discovery row and no `input` field, and the image capability checks still reject missing `input`; open PR #72404 is the active implementation candidate that specifically references this issue.

## What This Changes

The issue requests marking Bedrock Claude Opus/Sonnet 4.x model entries as image-capable so image attachments are not rejected as unsupported.

## Best Possible Solution

Land PR #72404 or a provider-owned replacement that refreshes or inherits missing Bedrock image capability metadata for stale explicit/static rows after discovery succeeds, while preserving intentional explicit text-only opt-outs and covering foundation, regional, and global Claude ids.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an explicit `models.providers.amazon-bedrock.models` row for a Claude 4.x id with no `input` and no matching implicit id; current main preserves that row unchanged, and the image gate treats missing `input` as unsupported.

## Solution Assessment

No for the original registry-only framing. Updating built-in model entries would not repair already-persisted stale catalogs; the maintainable fix should refresh or inherit capability metadata during provider merge/discovery while preserving explicit opt-outs.
