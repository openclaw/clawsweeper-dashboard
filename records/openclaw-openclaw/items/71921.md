## Summary

Keep open. Current main still preserves stale explicit Bedrock model rows without `input`, the image capability gate still treats missing `input` as no image support, and open PR #72404 specifically references this issue as the implementation candidate.

## What This Changes

The issue requests that Bedrock-hosted Claude 4.x model metadata preserve or populate `input: ["text", "image"]` so image attachments are not rejected as unsupported.

## Best Possible Solution

Review and land PR #72404 or a provider-owned replacement that refreshes or inherits missing Bedrock `input` capability metadata for stale explicit/static rows after discovery succeeds, while preserving intentional explicit text-only opt-outs and covering foundation, regional, and global Claude ids.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an explicit `models.providers.amazon-bedrock.models` entry for a Claude 4.x id with no `input` and no matching implicit id: current main preserves that row unchanged, then `modelSupportsImages()` returns false for missing `input`.

## Solution Assessment

No for the original registry-only framing. Updating built-in entries would not fix already-persisted stale catalogs; the maintainable solution should repair the merge/refresh behavior or otherwise inherit provider discovery metadata without overriding intentional explicit `input` settings.
