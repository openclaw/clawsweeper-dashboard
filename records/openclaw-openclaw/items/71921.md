## Summary

Keep open. Current main still preserves stale explicit/static Bedrock model rows that omit `input`, and image handling still treats missing `input` as no image support. The provided GitHub context also shows open PR #72404 has closing syntax for this issue, so the issue should stay open until that PR or a replacement lands.

## What This Changes

Mark Bedrock-hosted Claude Opus 4.7, Opus 4.6, and Sonnet 4.6 models as vision-capable so image attachments are not blocked by stale or incomplete model capability metadata.

## Best Possible Solution

Review and land PR #72404 or a provider-owned replacement that restores missing vision capability metadata for stale explicit Bedrock model rows after discovery succeeds, while preserving intentional explicit `input: ["text"]` opt-outs and splitting the startup discovery race if it needs a separate fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an explicit `models.providers.amazon-bedrock.models` entry for a Claude 4.x id with no `input`; current main preserves that row unchanged and then `modelSupportsImages()` returns false for missing `input`.

## Solution Assessment

No for the original registry-only request. Updating built-in or discovered registry entries alone would not repair already-persisted stale explicit catalogs, so the narrower maintainable solution is merge or refresh behavior that restores missing capability metadata without overriding explicit user opt-outs.
