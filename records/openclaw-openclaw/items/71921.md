## Summary

Keep open. Current main still preserves stale explicit Bedrock catalog rows that omit `input`, missing `input` still makes image support false, and PR #72404 is an open same-author implementation candidate that explicitly targets this issue.

## What This Changes

The issue asks OpenClaw to mark Bedrock Claude Opus/Sonnet 4.x models as image-capable so image attachments are not rejected as unsupported.

## Best Possible Solution

Review and land PR #72404, or replace it with a provider-owned fix that repairs missing capability metadata for stale explicit Bedrock catalog entries after discovery succeeds while preserving deliberate `input: ["text"]` opt-outs. Track the runtime-deps staging race separately if maintainers want to eliminate the source of stale catalogs.

## Reproduction Assessment

Yes. A source-level reproduction is an explicit `models.providers.amazon-bedrock.models` entry for a Claude 4.x id with no `input`; current main preserves that row and `modelSupportsImages()` then returns false for missing `input`.

## Solution Assessment

Unclear. PR #72404 is a plausible focused implementation candidate, but maintainers should decide whether generic model-id defaults in core are preferable to a Bedrock-provider stale catalog repair; updating only static registry entries would not fix already-persisted stale catalogs.
