## Summary

Keep open. Current main still has a source-level path where persisted explicit Bedrock model rows without `input` are preserved and then treated as no-vision, while open PR #72404 remains the unmerged implementation candidate with closing syntax.

## What This Changes

Mark Bedrock Claude Opus/Sonnet 4.x model metadata as image-capable so OpenClaw does not block image attachments for those models.

## Best Possible Solution

Land PR #72404 or an equivalent provider-owned merge/catalog refresh fix that restores missing image capability metadata for persisted Bedrock Claude rows without overwriting explicit text-only opt-outs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an explicit `models.providers.amazon-bedrock.models` Claude 4.x row with no `input`; current main preserves that row and then `modelSupportsImages()` returns false for undefined `input`.

## Solution Assessment

No for the original registry-only request. Updating built-in/discovered registry entries would not repair already-persisted static catalogs, so the best fix is merge or refresh behavior for missing capability metadata while preserving explicit user overrides.
