## Summary

Keep open. Current main still preserves stale explicit Bedrock model rows that omit `input`, image handling still treats missing `input` as no vision support, and open PR #72404 remains the active unmerged implementation candidate with closing syntax.

## What This Changes

Issue requests that Bedrock-hosted Claude Opus/Sonnet 4.x model entries carry image input capability so OpenClaw does not reject image attachments as unsupported.

## Best Possible Solution

Review and land PR #72404 or a provider-owned replacement that repairs missing capability metadata for stale explicit Bedrock model rows after discovery succeeds, while preserving intentional explicit `input: ["text"]` opt-outs and tracking the discovery startup race separately if needed.

## Reproduction Assessment

Yes. A source-level reproduction is an explicit `models.providers.amazon-bedrock.models` entry for a Claude 4.x id with no `input`; current main preserves that row unchanged and then `modelSupportsImages()` returns false for missing `input`.

## Solution Assessment

No for the original registry-only requested fix. Updating source/generated registry entries would not repair already-persisted stale explicit catalogs, so the narrower maintainable solution is merge or refresh behavior that restores missing capability metadata without overriding explicit user opt-outs.
