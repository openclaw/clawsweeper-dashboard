## Summary

Keep open: current main and v2026.6.10 still preserve explicit-only Bedrock model rows without `input`, so stale catalogs can still make vision-capable Claude Bedrock models reject image attachments before the provider call. The linked fix PR remains open and unmerged, so this issue is still the canonical tracker.

## What This Changes

The issue asks OpenClaw to mark Bedrock Claude Opus/Sonnet 4.x catalog entries as image-capable so image attachments are not blocked as unsupported.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/72404 is repaired and landed, or replace it with a focused fix that restores Bedrock vision metadata without unsafe inference for custom endpoints or user override loss.

## Reproduction Assessment

Yes, from source: create an explicit-only Bedrock model row without `input` and no matching implicit row, and `mergeProviderModels` returns it unchanged while `modelSupportsImages` treats the missing field as no image support. I did not run a live Bedrock request in this read-only review.

## Solution Assessment

Unclear whether the linked PR's broad family-defaulting approach is the best permanent repair. The narrow maintainable end state is to restore provider-owned Bedrock capability metadata for stale explicit catalogs while preserving explicit user overrides.
