## Summary

Keep open. Current main still preserves explicit-only Bedrock model rows that omit `input`, the image capability gates still treat missing `input` as no image support, and PR #72404 is open and unmerged as the active closing implementation candidate.

## What This Changes

The issue requests that Bedrock Claude Opus/Sonnet 4.x model metadata be marked or refreshed as image-capable so image attachments are not rejected.

## Best Possible Solution

Review and land PR #72404 or a provider-owned replacement that refreshes or inherits missing Bedrock `input` capability metadata for stale explicit/static rows after discovery succeeds, while preserving intentional explicit text-only opt-outs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an explicit `models.providers.amazon-bedrock.models` entry for a Claude 4.x id with no `input` and no matching implicit id: current main preserves that row unchanged, then `modelSupportsImages()` returns false for missing `input`.

## Solution Assessment

No for the original registry-only framing. Updating built-in entries would not fix already-persisted stale catalogs; the maintainable path should repair merge/refresh behavior or otherwise inherit provider discovery metadata without overriding intentional explicit `input` settings.
