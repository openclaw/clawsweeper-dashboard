## Summary

Keep open. Current main still preserves explicit Bedrock model rows that omit `input`, image handling still treats missing `input` as no vision support, and open PR #72404 remains the active unmerged implementation candidate for this issue.

## What This Changes

The issue asks OpenClaw to mark Bedrock-hosted Claude Opus/Sonnet 4.x models as image-capable so image attachments are not rejected as unsupported.

## Best Possible Solution

Review and land PR #72404 or a provider-owned replacement that repairs stale explicit Bedrock model capability metadata after discovery succeeds, while preserving intentional explicit `input: ["text"]` opt-outs and tracking the runtime-deps staging race separately if maintainers want that broader fix.

## Reproduction Assessment

Yes. A source-level reproduction is an explicit `models.providers.amazon-bedrock.models` entry for a Claude 4.x id with no `input` and no matching implicit id; current main preserves that row unchanged, then `modelSupportsImages()` returns false for missing `input`.

## Solution Assessment

No for the original registry-only framing. Updating static model entries would not repair already-persisted stale catalogs, so the narrower maintainable fix is merge/refresh behavior that restores missing capability metadata without overriding explicit user opt-outs.
