## Summary

Keep open: the ordering-based suppression is directionally useful, but the current head still has a blocking matcher bug that can hide real mutating-tool failures and the available Telegram proof is stale and failed against an older head.

## What This Changes

The branch threads embedded block-reply ordering into final payload construction and adds marker-style failure acknowledgement plus regression tests for duplicate tool-warning replies.

## Best Possible Solution

Constrain marker suppression to the explicit OpenClaw code-mode sentinel family, keep the ordering-based read-only suppression, add false-positive coverage, and provide current-head Telegram proof that duplicate warnings are gone without hiding later failures.

## Reproduction Assessment

Yes for the review finding by source inspection: the new line 82 matcher accepts any uppercase `_FAIL` token before action/negation checks, and `docs/reference/test.md` has an ordinary `OPENCLAW_DOCKER_ALL_FAIL_FAST=0` token. Current-head Telegram behavior is not yet reproduced live.

## Solution Assessment

No. The block-reply ordering evidence is the right direction, but the marker acknowledgement path is too broad; the safer fix is exact code-mode sentinel matching plus a false-positive regression and current-head live proof.
