## Summary

Keep open. The duplicate-warning goal is useful, but the patch suppresses terminal read-only tool failures whenever any earlier visible block reply exists, which can hide a later unrecovered failure; the PR also needs real Telegram behavior proof before merge.

## What This Changes

The PR threads visible block-reply counts from embedded attempts into final payload building and uses them to suppress read-only tool warning payloads while keeping mutating failures visible.

## Best Possible Solution

Track ordered recovery or acknowledgement evidence for the last read-only tool failure, suppress only the duplicate-warning case, and prove the behavior in Telegram.

## Reproduction Assessment

Yes, from source inspection: emit a visible block reply, then run a later read-only or probe tool that fails; under this PR, final payload building treats the earlier block reply as a user-facing reply and can suppress the terminal warning.

## Solution Assessment

No. Using a total visible block count is too broad; the maintainable fix should use event ordering or explicit recovery and acknowledgement evidence for the last tool failure.
