## Summary

Keep this PR open, but it is not merge-ready: the new suppression signal is overbroad and can hide the only warning for a read-only/probe tool that fails after an earlier visible block reply, and the PR still lacks real Telegram behavior proof.

## What This Changes

The PR threads `visibleBlockReplyCount` from embedded attempts into final payload building and uses it to suppress recovered read-only/probe tool warning payloads while preserving mutating failures.

## Best Possible Solution

Rework the PR to use explicit recovery or ordering evidence instead of total visible block-reply count, add a regression for post-reply unrecovered read-only failures, and then provide redacted real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: after a visible block reply, a later read-only tool start can record a terminal `lastToolError`, and the PR's positive `visibleBlockReplyCount` then suppresses that warning. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Suppressing by total block-reply count is too broad; the maintainable fix is to pass explicit recovered-warning or ordering evidence and preserve warnings for later unrecovered failures.
