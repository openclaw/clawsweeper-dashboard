## Summary

Keep open: current main still lacks the requested bundle-MCP consent-envelope interception, and an open same-author implementation PR with closing syntax carries the design and live proof. This changes a security-sensitive approval boundary and overlaps MCP-native elicitation work, so it needs maintainer security/product review rather than cleanup closure.

## What This Changes

Request to add an opt-in MCP consent envelope that routes mutating bundle-MCP tool calls through channel-verified plugin approvals with an `mcp.approvals.enabled` config switch.

## Best Possible Solution

Review and land, or replace, https://github.com/openclaw/openclaw/pull/78303 with a security-approved bundle-MCP approval design that documents the server contract and its relationship to MCP-native elicitations.

## Reproduction Assessment

Yes. Source inspection on current main shows bundle-MCP `callTool` results are returned directly to the agent-visible materializer path and the requested `mcp.approvals` config surface is absent; this is a feature/security-hardening gap rather than a broken documented behavior bug.

## Solution Assessment

Unclear. A materializer-level consent-envelope gate that reuses plugin approvals is plausible and the paired PR includes live proof, but final acceptance needs maintainer security/product judgment against MCP-native elicitations and channel-provenance invariants.
