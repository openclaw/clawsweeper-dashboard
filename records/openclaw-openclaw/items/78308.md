## Summary

Keep open: current main still passes bundle-MCP tool results directly back to the agent without consent-envelope handling, and this issue is paired with the open implementation PR at https://github.com/openclaw/openclaw/pull/78303 via closing syntax.

## What This Changes

Request to add an MCP consent envelope so third-party bundle-MCP tools can require channel-mediated user approval before a mutating action is completed.

## Best Possible Solution

Review and land a security-approved bundle-MCP approval design through https://github.com/openclaw/openclaw/pull/78303 or a replacement, while keeping MCP-native elicitation compatibility explicitly tracked if the custom envelope does not cover it.

## Reproduction Assessment

Yes for the current-main baseline: source inspection shows bundle-MCP calls return `CallToolResult` directly to the agent with no consent-envelope interception. This is a feature/security-hardening request rather than a broken documented behavior repro.

## Solution Assessment

Unclear: materializer-level interception and reuse of plugin approvals look like the right area, but maintainers still need to decide the custom envelope versus MCP-native elicitation support and review the linked PR's security details.
