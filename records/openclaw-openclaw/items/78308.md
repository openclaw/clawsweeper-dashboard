## Summary

Keep open: current main still passes bundle-MCP tool results directly back to the agent without consent-envelope interception, and the same-author implementation PR that explicitly closes this issue remains open for maintainer/security review.

## What This Changes

The issue requests an additive MCP consent-envelope contract so third-party MCP tool calls can be paused for channel-mediated user approval before mutating external state.

## Best Possible Solution

Review and land a security-approved bundle-MCP approval design through https://github.com/openclaw/openclaw/pull/78303 or a replacement, while explicitly reconciling it with MCP-native elicitation support.

## Reproduction Assessment

Yes for the current-main baseline by source inspection: bundle-MCP execution returns `CallToolResult` directly to the agent without consent-envelope interception. This is a feature/security-hardening request rather than a broken documented behavior bug.

## Solution Assessment

Unclear: materializer-level interception and reuse of plugin approvals are plausible, but the final design needs maintainer security/product review against MCP-native elicitations and approval-channel provenance.
