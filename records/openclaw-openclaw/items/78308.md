## Summary

Keep open: current main still lacks MCP consent-envelope handling, and the issue is paired with open implementation PR https://github.com/openclaw/openclaw/pull/78303, which explicitly closes it and has supplied live proof but still needs maintainer product/security review.

## What This Changes

The issue requests an additive MCP consent envelope that routes opted-in bundle-MCP tool calls through channel approval, hides the action token from the model, re-calls with a confirmation token after approval, and adds an `mcp.approvals.enabled` config switch.

## Best Possible Solution

Keep the issue open until https://github.com/openclaw/openclaw/pull/78303 or a maintainer-approved replacement lands with source, schema, docs, tests, and MCP-native elicitation compatibility resolved.

## Reproduction Assessment

Not applicable: this is a feature/security-hardening request, not a broken current contract. Source inspection confirms the current baseline: bundle-MCP tools call `runtime.callTool(...)` and return the result without envelope handling.

## Solution Assessment

Unclear: materializer-level interception with plugin approvals is a plausible architecture and the paired PR has strong live proof, but maintainers still need to choose the product/security contract and how it coexists with MCP-native elicitation.
