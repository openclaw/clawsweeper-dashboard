## Summary

Keep open: current main still passes bundle-MCP tool results directly back to the agent without consent-envelope handling, and the same author has an open closing PR that is the active implementation path.

## What This Changes

The issue requests an opt-in MCP consent envelope that routes bundle-MCP tool approval through existing channel/plugin approvals before re-calling the tool with a confirmation token.

## Best Possible Solution

Keep this issue open while maintainers review https://github.com/openclaw/openclaw/pull/78303 or a replacement, then close it only after the chosen MCP approval contract lands on main.

## Reproduction Assessment

Not applicable as a feature/security-hardening request rather than a broken documented behavior. The current-main baseline is still high-confidence from source inspection: bundle-MCP tools call `runtime.callTool(...)` and return the result without consent-envelope handling.

## Solution Assessment

Unclear: the proposed materializer-level envelope targets the right execution surface and can reuse existing plugin approvals, but it adds a new OpenClaw contract/config key and needs maintainer agreement alongside MCP-native elicitation and out-of-band approval alternatives.
