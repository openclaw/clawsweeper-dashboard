## Summary

Keep open: current main still lacks an infrastructure-enforced consent gate for bundle-MCP tool results, and the same-author implementation PR is open with closing syntax, so the issue should remain attached to that review path.

## What This Changes

Request an additive MCP consent envelope that routes mutating bundle-MCP tool calls through the existing channel `/approve` approval pipeline, with a per-deployment opt-out.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/78303 or a maintainer-approved replacement that wires the bundle-MCP approval contract, schema/docs/tests, routing, timeout, and native elicitation tradeoffs cleanly.

Not applicable: this is a feature/security-hardening request, not a broken existing contract. Source inspection gives a high-confidence current-main baseline: bundle-MCP tools directly return `callTool` results without consent-envelope approval handling.

## Solution Assessment

Unclear: reusing plugin approvals at the materializer boundary is plausible, but maintainers still need to decide the custom envelope versus MCP-native elicitation and the exact routing, timeout, opt-out, and allow-always semantics.
