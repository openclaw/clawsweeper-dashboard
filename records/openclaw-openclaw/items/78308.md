## Summary

Keep open: current main and v2026.6.10 still pass bundle-MCP tool results back to the agent without consent-envelope approval handling, and the live-open linked PR remains the canonical implementation target for maintainer security/product review.

## What This Changes

The issue requests channel-mediated approval for side-effecting MCP tool calls by recognizing a consent envelope, hiding action tokens from the model, and re-calling only after verified user approval.

## Best Possible Solution

Keep this canonical issue open until the linked PR or a maintainer-approved replacement lands the MCP consent contract with token redaction, exact-call/origin binding, tests, docs, and current-head real behavior proof.

## Reproduction Assessment

Yes. Current main and v2026.6.10 both call bundle-MCP tools directly and expose structuredContent to the agent without a consent-envelope approval bridge; the issue thread also supplies live HomeBrain/WhatsApp proof, though no live scenario was rerun here.

## Solution Assessment

Unclear. Reusing plugin approvals at the bundle-MCP materialization boundary is plausible, but maintainers still need to choose the exact envelope, call-binding, channel-provenance, and config-default semantics before it is the best permanent solution.
