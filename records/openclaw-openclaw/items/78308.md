## What This Changes

The issue requests an MCP consent envelope so third-party MCP tools can route mutating calls through channel-mediated OpenClaw approvals before the model receives or redeems a server token.

## Best Possible Solution

Review, repair, and either land https://github.com/openclaw/openclaw/pull/78303 or replace it with a maintainer-approved MCP approval design that aligns with the existing plugin approval flow and MCP-native elicitation direction.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature/security-hardening request. Current-main source inspection and the reporter's live baseline show the existing behavior clearly: bundle-MCP tool results are returned to the model without a runtime consent gate.

## Solution Assessment

Unclear: the materializer-level consent envelope is a plausible narrow surface and the linked PR now has live happy-path proof, but maintainers still need to resolve the custom envelope versus MCP-native elicitation direction and the open PR findings before this is the best mergeable solution.
