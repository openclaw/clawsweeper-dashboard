## Summary

Keep open: current main still passes bundle-MCP tool results directly back to the agent and lacks the requested MCP approval config, while the same-author implementation PR is still open and explicitly closes this issue with real-behavior proof and active security-design discussion.

## What This Changes

The issue requests a runtime-enforced MCP consent envelope so bundle-MCP tools can require channel approval before the model sees or redeems confirmation tokens.

## Best Possible Solution

Review and land, or replace, https://github.com/openclaw/openclaw/pull/78303 with a security-approved bundle-MCP approval design that documents the server contract and its relationship to MCP-native elicitations.

## Reproduction Assessment

Yes for the current-main baseline by source inspection: bundle-MCP tool results are returned directly to agent-visible output and the requested mcp.approvals config switch is absent. This is a feature/security-hardening request rather than a broken documented behavior bug.

## Solution Assessment

Unclear: materializer-level consent-envelope interception and reuse of plugin approvals are plausible, but the final shape needs maintainer security/product review against MCP-native elicitations and channel-provenance requirements.
