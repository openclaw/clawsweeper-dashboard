## Summary

Keep this open: current main still has the public SDK zod subpath as a bare zod re-export, and the build config does not force zod to bundle, matching the reported pnpm global install failure mode.

## What This Changes

Fix the published `openclaw/plugin-sdk/zod` subpath so bundled plugins can load when pnpm global installs do not make `zod` resolvable from the OpenClaw package directory.

## Best Possible Solution

Make the published SDK zod artifact self-contained or otherwise resolvable in pnpm global installs, with a focused build/package regression test and no startup mutation of the global package tree.

## Reproduction Assessment

Yes, from source inspection: current main still maps the public SDK zod subpath to a built artifact produced from a bare `export * from "zod"`, with no zod-specific forced bundling rule. I did not run a live pnpm global install in this read-only review.

## Solution Assessment

Yes: a narrow SDK build/package fix is the best direction because `plugin-sdk/zod` is already documented public behavior and global installs are documented as read-only at runtime. A doctor or postinstall symlink workaround would be less maintainable than preventing the bad published artifact.
