The report appears valid and still actionable: the documented SDK subpath is a public runtime export, current source still re-exports bare `zod`, and the latest shipped tarballs contain the same bare runtime export that the reporter identified.

## What This Changes

The issue asks OpenClaw to make `openclaw/plugin-sdk/zod` resolvable in pnpm global installs instead of emitting a bare `zod` runtime import in the published SDK subpath.

## Best Possible Solution

Make `openclaw/plugin-sdk/zod` self-contained in the published package, preferably by bundling or shimming the zod namespace for that SDK entrypoint, then lock it with a package-artifact regression check for pnpm global installs.

## Solution Assessment
