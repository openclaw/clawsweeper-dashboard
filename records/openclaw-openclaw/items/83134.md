## Summary

Keep open: current main and v2026.5.12 still default Web Push VAPID to `mailto:openclaw@localhost`, and the send path passes that subject to `web-push`. The upstream `web-push` README documents `https:`/`mailto:` VAPID subjects and specifically notes Safari `BadJwtToken` failures for localhost subjects, so the reported iOS PWA failure is source-backed and narrow.

## What This Changes

The issue asks OpenClaw to stop auto-generated Web Push VAPID keys from using a localhost subject that Apple Web Push rejects.

## Best Possible Solution

Use a stable valid OpenClaw `https:` or `mailto:` fallback for `DEFAULT_VAPID_SUBJECT`, keep `OPENCLAW_VAPID_SUBJECT` as the operator override, and update focused tests/docs without adding a new config surface unless maintainers want a broader push-config design.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection shows no-env `resolveVapidKeys()` resolves to `mailto:openclaw@localhost`, and the send path passes that value to `web-push`. I did not independently hit Apple’s live push endpoint in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to replace the built-in fallback with a valid stable contact URI and refresh tests/docs; the existing env override already covers operator-specific subjects.
