## Summary

Keep open: current main and the latest shipped release still default Web Push VAPID to `mailto:openclaw@localhost`, the send path passes that subject to `web-push`, and no duplicate fix PR or canonical issue currently owns the repair.

## What This Changes

The issue reports that auto-generated Web Push VAPID keys default to `mailto:openclaw@localhost`, which breaks Apple Web Push delivery for iOS PWAs.

## Best Possible Solution

Change the built-in fallback to a stable valid OpenClaw `https:` or `mailto:` VAPID subject, keep the existing env override semantics, and update tests/docs so generated and existing keypairs use the corrected subject without key deletion.

## Reproduction Assessment

Yes, source-backed for the OpenClaw side: with no `OPENCLAW_VAPID_SUBJECT`, `resolveVapidKeys()` resolves to `mailto:openclaw@localhost`, and `broadcastWebPush()` passes that subject to `webPush.setVapidDetails()`. I did not independently run the Apple endpoint, but the reporter supplied concrete live Apple results.

## Solution Assessment

Yes. The narrow maintainable fix is to replace the invalid built-in fallback with a valid stable contact URI and update coverage/docs; adding an `openclaw.json` setting is optional because an env override already exists.
