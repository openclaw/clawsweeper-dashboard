## Summary

Keep open: current main and v2026.5.12 still default Web Push VAPID to `mailto:openclaw@localhost`, the send path passes that subject to `web-push`, and Apple/web-push docs make the reported `BadJwtToken` failure plausible. ([developer.apple.com](https://developer.apple.com/documentation/usernotifications/sending-web-push-notifications-in-web-apps-and-browsers?language=objc&utm_source=openai))

## What This Changes

The issue asks OpenClaw to stop auto-generating Web Push VAPID identity with a localhost subject that breaks Apple/iOS PWA push delivery.

## Best Possible Solution

Use a stable valid OpenClaw `https:` or `mailto:` VAPID fallback, keep `OPENCLAW_VAPID_SUBJECT` as the operator override, and update the focused tests/docs so existing generated keypairs inherit the corrected subject without deleting keys.

## Reproduction Assessment

Yes, from source for the OpenClaw side: no-env `resolveVapidKeys()` resolves to `mailto:openclaw@localhost`, and the Web Push send path uses that subject. I did not independently run the Apple endpoint, but the reporter provided a concrete Apple push-service result.

## Solution Assessment

Yes, the narrow maintainable fix is to change the built-in fallback subject to a valid stable contact URI and update coverage/docs. Adding an `openclaw.json` setting is optional because an env override already exists.
