## Label Justifications

- `P1`: The bad default can break iOS PWA push delivery for first-install users without an operator override.
- `impact:message-loss`: The affected Web Push path can silently fail to deliver browser notifications to subscribed iOS PWA users.

## Summary

Keep open: current main and the latest shipped release still default Web Push VAPID to `mailto:openclaw@localhost`, and an open linked fix PR already closes this issue but has not merged yet.

## What This Changes

Issue asks OpenClaw to stop auto-generating Web Push VAPID keys with a localhost subject because Apple Web Push rejects that subject for iOS PWA notifications.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83317, or an equivalent narrow change that uses a stable valid `https:`/`mailto:` default while preserving `OPENCLAW_VAPID_SUBJECT` as the override.

## Reproduction Assessment

Yes, at source level: no-env `resolveVapidKeys()` resolves to `mailto:openclaw@localhost`, and the send path passes that subject to `webPush.setVapidDetails()`. I did not run a live Apple endpoint send in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to replace the module-private fallback with a stable valid contact URI, keep the existing env override, and update focused tests/docs rather than adding a new config surface for this bug.
