## Summary

Keep open: current main has watchdogs and limited Claude CLI failover retry, and #83700 fixed stale subagent completion delivery, but the reported stream/API-error classification gap still appears plausible in the live Claude CLI path. The item needs runtime-owner review with a sanitized failing JSONL/log fixture before deciding fail-fast versus retry behavior.

## What This Changes

The issue asks OpenClaw to treat Claude CLI subagent SSE/socket stream errors as terminal retryable failures instead of normal transcript text that can leave the parent Agent call pending.

## Best Possible Solution

Classify Claude CLI stream-json socket/API error frames as terminal recoverable failures at the OpenClaw CLI boundary, route them through existing failover/subagent completion handling, and add retry only where idempotency is explicitly safe.

## Reproduction Assessment

No, not with high confidence from a live run. The current source path plausibly reproduces the report's non-terminal `API Error:` frame behavior, but a redacted failing JSONL/log sample or live repro is still needed to prove the exact Claude Code socket-close sequence.

## Solution Assessment

Unclear. Failing fast on explicit Claude CLI stream API errors looks like the narrowest safe OpenClaw-side repair, while automatic retry needs a maintainer product decision because subagent turns can include tool side effects.
