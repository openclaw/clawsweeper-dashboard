## Summary

Keep open: current main and the latest release still leave the reported Claude CLI stream/API-error path plausible, and the safe boundary between fail-fast classification and automatic retry needs runtime-owner review with a sanitized failing fixture.

## What This Changes

The issue asks for Claude Code SDK or Claude CLI subagent SSE/socket-close failures to retry or fail terminally instead of being treated as ordinary assistant text that leaves the parent waiting.

## Best Possible Solution

Classify explicit Claude CLI stream API-error frames as terminal recoverable failures at the OpenClaw CLI boundary, route them through existing failover/subagent completion handling, and add retry only where idempotency is proven.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a plausible current-main reproduction path: a Claude stream-json assistant frame containing `API Error:` can remain non-terminal unless a later `result.is_error` or process failure is observed.

## Solution Assessment

Unclear as stated: retrying the SDK stream may be unsafe for side-effectful subagent turns. The safer first fix is fail-fast terminal classification at the OpenClaw Claude CLI boundary, with retry added only after a maintainer-owned idempotency decision.
