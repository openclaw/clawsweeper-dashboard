## Summary

Keep open: the report is recent and concrete, and current main still appears to leave the precise assistant-frame `API Error:`/missing-result live Claude stream shape to timeout or downstream recovery rather than immediately classifying it as a retryable or terminal subagent failure.

## What This Changes

The issue asks OpenClaw/Claude CLI subagent handling to treat Claude SDK SSE/socket-close API-error text records as retryable or terminal failures instead of leaving parent Agent calls pending.

## Best Possible Solution

Classify Claude CLI assistant-frame API/socket error records as terminal recoverable failures at the OpenClaw runtime boundary, then route them through existing failover, subagent completion, and notification paths; add direct retries only with idempotence proof and a redacted JSONL/live regression.

## Reproduction Assessment

No high-confidence standalone live reproduction is available yet. Source inspection does show a plausible current-main gap: non-`result` Claude live JSONL records, including assistant `API Error:` text, are streamed and stored but not immediately failed or retried at that boundary.

## Solution Assessment

Unclear as written. Upstream Claude SDK retry may be ideal, but the safer OpenClaw direction is first to fail fast and notify through existing runtime/failover paths, then add retries only where duplicate tool side effects are ruled out.
