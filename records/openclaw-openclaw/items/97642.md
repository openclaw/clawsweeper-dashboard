## Summary

Keep open: the persisted-shape comparison is the right direction, but this patch still leaves runtime-only `resolvedSkills` in the reply-init revision path, and the new linked proof corpus does not show an after-fix Discord two-message flow succeeding.

## What This Changes

The PR changes reply-session initialization revisions to compare a promptRef-projected persisted entry shape and adds a session accessor regression test for hydrated `skillsSnapshot.prompt` versus persisted `promptRef`.

## Best Possible Solution

Use one durable-entry revision helper that strips runtime-only `resolvedSkills` and projects `promptRef`, keep runtime snapshots hydrated, and require direct redacted Discord or Mantis proof on the exact head before merge.

## Reproduction Assessment

Yes at source level: current main JSON-stringifies whole `SessionEntry` revisions while storage can represent the same `skillsSnapshot` as persisted `promptRef`, hydrated `prompt`, or runtime-only `resolvedSkills`. I did not run a live Discord two-message flow in this read-only review.

## Solution Assessment

No. Persisted-shape revision comparison is the right layer, but this implementation should normalize the full durable entry instead of only projecting prompt blobs.
