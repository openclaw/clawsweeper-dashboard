## Summary

Keep open: current main still contains the strict task and task-flow registry `chmodSync` restore paths, and the macOS packaged CLI/probe path implicated in the discussion is not resolved by a merged fix. The newer related report at https://github.com/openclaw/openclaw/issues/83619 is useful evidence for the same filesystem-hardening pattern, but it tracks a narrower exec-approvals site rather than fully superseding this registry/probe issue.

## What This Changes

The issue reports macOS gateway probe/status failures with WebSocket `1006` plus task and task-flow registry restore failures from `EPERM` during `chmod` of the tasks and flows directories.

## Best Possible Solution

Split the remaining work into bounded fixes: macOS packaged launcher/provenance handling or diagnostics, tolerant-but-secure registry directory permission handling for tasks and flows, and a separate heartbeat/main-session investigation only if that later symptom still reproduces.

## Reproduction Assessment

No high-confidence live reproduction is available here because the macOS 26 npm/provenance environment was not exercised. Source inspection does show the current task and task-flow restore paths still throw on non-Windows `chmodSync` failures, matching the reported `EPERM` registry warnings.

## Solution Assessment

Unclear as one patch: the registry permission behavior has a narrow source-level shape, but the packaged macOS probe failure and cross-subsystem chmod policy need maintainer design review. A centralized permission-hardening helper plus separate macOS launcher diagnostic would be safer than three unrelated point fixes.
