## Label Justifications

- `P1`: The issue concerns a cross-session cron metadata disclosure in gateway run history.
- `impact:security`: The affected behavior can reveal another agent's private cron job names to an all-scope history caller.

## Summary

Keep open: current main and `v2026.5.20` still build all-scope `cron.runs` job-name enrichment from an unfiltered cron job list, and the direct closing PR remains open and conflicting.

## What This Changes

The issue reports that all-scope `cron.runs` can expose other agents' cron job names through run-history enrichment.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80146 or an equivalent narrow fix that uses scoped `listPage()` for all-scope job-name enrichment with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron.runs(scope="all")` builds `jobNameById` from unfiltered `context.cron.list()`, and the all-run reader exposes that map in search text and returned `jobName` fields.

## Solution Assessment

Yes. Reusing the existing scoped `listPage()` contract for the enrichment map is the narrow maintainable fix for the reported job-name leak while preserving the separate all-scope run-log behavior.
