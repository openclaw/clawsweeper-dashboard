## Label Justifications

- `P1`: The issue concerns a cross-session cron metadata disclosure in gateway run history.
- `impact:security`: The affected behavior can reveal another agent's private cron job names to an all-scope history caller.
- `impact:session-state`: The leak crosses agent/session boundaries through cron run-history enrichment metadata.

## Summary

Keep open: current main and latest release still build all-scope `cron.runs` job-name enrichment from the unfiltered cron list, and the open closing PR is unmerged and currently conflicting.

## What This Changes

The issue requests that `cron.runs` with all scope stop exposing cron job names from other agents or sessions in run-history responses.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80146, or an equivalent narrow fix that builds all-scope job-name enrichment from scoped `listPage()` results with regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main still builds the all-scope `cron.runs` job-name map from unfiltered `context.cron.list()`, and the all-run reader exposes that map in search text and returned `jobName` fields.

## Solution Assessment

Yes. Reusing the existing scoped `listPage()` contract for the enrichment map is the narrow maintainable fix for the reported job-name leak while preserving the separate all-scope run-log behavior.
