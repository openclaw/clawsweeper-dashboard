## Summary

Keep open: current main and the latest release still use the unfiltered all-jobs list for `cron.runs(scope="all")` job-name enrichment, while the open linked fix PR has not merged yet.

## What This Changes

The issue requests scoping `cron.runs(scope="all")` job-name lookup so run-history responses do not expose cron job names from other agents or sessions.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80146 or an equivalent narrow fix that builds the all-scope job-name map from the scoped `listPage` path with regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: call `cron.runs` with `scope: "all"` after jobs from different agents exist, and current main builds the returned name map from an unfiltered cron list before passing it into the run-log reader.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix for the named leak, because it reuses the existing agent-scoped list contract for enrichment while leaving run-log scanning behavior unchanged.
