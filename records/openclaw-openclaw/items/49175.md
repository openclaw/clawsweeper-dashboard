## Summary

Keep open: the prompt-provided current main SHA and the latest release still build the `cron.runs(scope: "all")` job-name map from the unfiltered cron list, and an open closing PR already targets this issue but has not merged.

## What This Changes

Scope all-scope `cron.runs` job-name enrichment so returned run history cannot attach cron job names from other agents or sessions.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80146 or an equivalent narrow fix that builds the all-scope job-name map from the scoped `listPage` path with regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: call `cron.runs` with `scope: "all"` after jobs from different agents exist, and current main builds the returned name map from an unfiltered cron list before passing it into the run-log reader.

## Solution Assessment

Yes. Reusing the existing `listPage()` agent filter for name enrichment is the narrow maintainable fix for the named leak while leaving the broader all-scope run-log behavior unchanged.
