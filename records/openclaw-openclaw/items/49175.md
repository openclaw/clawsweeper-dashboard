## Label Justifications

- `P1`: The issue covered a cross-session cron job-name disclosure in a gateway path used by agent-tool callers.
- `impact:security`: The reported leak could disclose another agent or user's cron job names across a security boundary.
- `impact:session-state`: The affected behavior was tied to per-agent/per-session cron visibility and caller identity handling.

## Summary

Current main now blocks the reported cross-session job-name disclosure: agent-originated cron calls carry caller identity, and caller-scoped all-scope run history is rejected before the unfiltered job-name map can be built. The fix is current-main-only; no release tag contains the fixing merge commit yet.

## What This Changes

The issue asks for `cron.runs(scope="all")` to stop enriching run-history rows with cron job names from other agents or sessions.

## Best Possible Solution

Keep the current-main server-side caller-scope guard and treat the older open fix PR as obsolete unless maintainers intentionally want a separate filtered all-scope history API.

## Reproduction Assessment

No on current main. The latest release source still has the old unfiltered all-scope map, but current main rejects caller-scoped all-scope history before that call and has a regression test asserting the list is not called.

## Solution Assessment

Yes for the central leak. The current-main fix is a conservative server-side security boundary: agent-originated callers cannot request all-scope run history, while operator/unscoped behavior remains global; a filtered all-scope UX would be a separate product/API choice.
