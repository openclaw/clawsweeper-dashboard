## Label Justifications

- `P1`: The issue concerns a cross-session cron metadata disclosure in gateway run history.
- `impact:security`: The affected behavior can reveal another agent's private cron job names to an all-scope history caller.
- `impact:session-state`: The leak crosses agent/session boundaries through cron run-history enrichment metadata.

## Summary

Keep open: current main and the latest release still build all-scope `cron.runs` job-name enrichment from an unfiltered cron job list, while the direct closing PR at https://github.com/openclaw/openclaw/pull/80146 remains open and conflicting.

## What This Changes

Scope all-scope `cron.runs` job-name enrichment so callers cannot see cron job names from other agents or sessions.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80146 or an equivalent narrow fix that builds `jobNameById` from scoped paginated `listPage()` results with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `cron.runs(scope="all")` builds `jobNameById` from unfiltered `context.cron.list()`, and the all-run reader exposes that map in searchable text and returned `jobName` fields.

## Solution Assessment

Yes. Reusing the existing scoped `listPage()` contract for the enrichment map is the narrow maintainable fix while preserving the separate all-scope run-log behavior.
