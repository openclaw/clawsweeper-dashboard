## Summary

Keep open: current main still matches the reported `cron.runs` all-scope job-name leak, and the related fix PRs are closed unmerged. The broader #35447 tracker is relevant context, but this issue captures a distinct run-history response surface that should remain tracked until a fix lands.

## What This Changes

The issue requests caller-scoped filtering for `cron.runs` with `scope: "all"` so run-history responses do not include cron job names from other agents or sessions.

## Best Possible Solution

Add a caller-aware cron history contract: derive the authenticated caller/session/admin override in the gateway, filter the job-name source by visible `agentId` or `sessionKey` before it reaches `readCronRunLogEntriesPageAll()`, and add a regression test with two owned jobs proving all-scope run history does not attach or query another session's job name.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create jobs with different `agentId` or `sessionKey`, call `cron.runs` with `scope: "all"` as one caller, and the current handler will build `jobNameById` from the unfiltered cron list before returning run-log entries.

## Solution Assessment

Yes, the requested direction is the right narrow fix, but the implementation needs a real caller-aware service/gateway contract rather than only passing existing `list()` options. The safer shape is a shared owner-filter helper plus explicit admin override and regression coverage.
