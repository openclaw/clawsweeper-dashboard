## Summary

Keep open: current main and latest release still build all-scope cron run-history job names from an unfiltered cron job list, and the exact-fix PR/commit referenced from the issue is closed/unmerged while the broader isolation tracker remains open.

## What This Changes

The issue asks to prevent `cron.runs` with `scope: "all"` from exposing cron job names outside the calling session's visible jobs.

## Best Possible Solution

Add a caller-aware cron run-history contract that filters the all-scope job-name source by the caller's visible `agentId` or `sessionKey`, preserves an explicit admin/legacy compatibility path, and adds a two-session regression test for `cron.runs` all-scope names.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create jobs with different `agentId` or `sessionKey`, call `cron.runs` with `scope: "all"` from one read-scoped caller, and current main builds returned/queryable `jobName` values from an unfiltered cron list.

## Solution Assessment

Yes, the narrow fix direction is to filter the job-name map before it reaches `readCronRunLogEntriesPageAll()`. The maintainable version should share caller ownership rules with cron list/mutation handling rather than adding a one-off partial filter.
