## Summary

Keep open: current main and the latest release still strip the receiving Feishu bot's own mention from inbound content, and all linked fix PRs are closed unmerged. The source path is clear enough for a focused fix that preserves multi-bot mention context without regressing mention-prefixed slash commands.

## What This Changes

The issue requests fixing Feishu inbound mention normalization so multi-bot group mentions keep each receiving bot's own mention visible to the agent instead of causing silent NO_REPLY behavior.

## Best Possible Solution

Land one focused Feishu plugin fix that preserves semantic inline self-mentions for multi-bot routing while keeping leading addressing prefixes clean for command parsing, with regression coverage and redacted real Feishu proof when practical.

## Reproduction Assessment

Yes, at source level: a Feishu event containing the receiving bot and another bot mention flows through parseFeishuMessageEvent into normalizeMentions with botOpenId, and normalizeMentions removes the receiving bot's own tag from ctx.content. I did not live-reproduce this in a Feishu tenant.

## Solution Assessment

No for the exact remove-all botStripId suggestion; it risks regressing the established mention-prefixed slash-command behavior. The best fix is narrower: preserve inline/self mention context for the model while stripping or hiding only the leading addressing prefix needed for command parsing.
