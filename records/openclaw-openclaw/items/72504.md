## Summary

Keep open. Current main still strips the receiving Feishu bot's own mention before building the agent-facing body, and there is an open PR with closing syntax plus an alternate open PR, so the issue should remain open until one focused fix lands.

## What This Changes

Fix Feishu mention normalization so multi-bot group messages preserve the receiving bot's own mention for model routing instead of causing silent NO_REPLY behavior.

## Best Possible Solution

Land one focused Feishu fix, preferably through #72661 or a replacement, that preserves self-mention context in agent-facing multi-bot group prompts while keeping mention-prefixed slash commands working and adding regression tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group text event containing Bot A and Bot B mention placeholders while parsing with Bot A's `open_id`; current `normalizeMentions` removes Bot A's placeholder before `BodyForAgent` is constructed.

## Solution Assessment

Unclear for the issue's suggested removal of all `botStripId` behavior. The safer solution is to decouple command probing from model-facing content, or strip only the leading addressing prefix, so #35994 remains protected.
