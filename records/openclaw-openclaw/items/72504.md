## Summary

Keep open: current main and v2026.6.8 still strip the receiving Feishu bot's own mention before building agent context, and the closed prior fix PRs plus the open union_id PR do not own this exact self-mention-strip bug.

## What This Changes

The issue requests preserving the receiving Feishu bot's own <at> mention in agent-visible Feishu content so multi-bot group mentions do not produce silent replies.

## Best Possible Solution

Land a focused Feishu plugin fix that preserves semantic self-mention context for the agent body while keeping leading @Bot command inputs slash-clean for command parsing.

## Reproduction Assessment

Yes, at source level: a Feishu event that mentions the receiving bot and another bot flows through parseFeishuMessageEvent into normalizeMentions with botOpenId, and normalizeMentions removes the receiving bot's own tag from ctx.content. I did not run a live Feishu tenant reproduction.

## Solution Assessment

No for the exact remove-all botStripId suggestion; it is too broad because existing tests and history protect mention-prefixed slash commands. The better fix is narrower: keep command inputs slash-clean while preserving self-mention context where the model needs it.
