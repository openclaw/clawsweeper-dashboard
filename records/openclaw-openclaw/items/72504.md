## Summary

Keep open. Current main still removes the receiving Feishu bot's own mention from agent-facing content, and open PR #72661 explicitly closes this issue with #72532 as an alternate active fix, so this should wait for one focused implementation to land.

## What This Changes

The issue asks Feishu group mention normalization to preserve the receiving bot's own `<at>` context when multiple bots are mentioned together, avoiding `NO_REPLY` routing failures.

## Best Possible Solution

Land #72661 or an equivalent focused Feishu fix that strips only leading addressing prefixes while preserving semantic self-mentions in agent-facing content, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group text event containing Bot A and Bot B mention placeholders while parsing with Bot A's `open_id`; current `normalizeMentions` removes Bot A's placeholder before `BodyForAgent` is built.

## Solution Assessment

No for the issue body's blanket removal of all `botStripId` behavior. The safer fix is to preserve self-mentions that carry semantic routing context while keeping mention-prefixed slash commands protected through leading-prefix stripping or the existing command-probe normalization path.
