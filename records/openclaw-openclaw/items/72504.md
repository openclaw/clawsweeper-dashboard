## Summary

Keep open. Current main still strips the receiving Feishu bot's own mention from normalized content before the agent-facing prompt, and an open PR with closing syntax already tracks a narrower fix, so the issue should remain open until that PR or its alternate is resolved.

## What This Changes

Feishu should preserve enough self-mention context in multi-bot group messages so explicitly mentioned bots do not silently return NO_REPLY.

## Best Possible Solution

Land one focused Feishu change that strips only the leading addressing prefix needed for command probing while preserving inline or repeated self-mentions in the agent prompt, with regression tests for multi-bot group messages; then close this issue after merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group text event containing Bot A and Bot B mentions while processing with Bot A's open_id; current normalizeMentions removes Bot A's placeholder before buildFeishuAgentBody constructs the model-facing prompt.

## Solution Assessment

No for the issue's broad suggested removal of all botStripId behavior, because current code and changelog history show that self-strip protects #35994 slash-command routing. A narrower leading-prefix-only fix like the linked PR's approach is the more maintainable path.
