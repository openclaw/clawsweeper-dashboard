## Summary

Keep open. Current main still strips the receiving Feishu bot's own mention from the agent-facing prompt, so the reported multi-bot prompt-loss path remains live; the issue also has an open closing PR and an open alternate PR, so it should not be closed before one of those lands or is rejected.

## What This Changes

The issue asks Feishu inbound mention normalization to preserve the receiving bot's own mention in multi-bot group messages so explicitly mentioned bots do not return NO_REPLY.

## Best Possible Solution

Land one focused Feishu fix that strips only the leading addressing prefix needed for slash-command routing while preserving inline or repeated self-mentions in ctx.content, with regression coverage for multi-bot group messages; close this issue after the PR merges.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group text event with mentions for the receiving bot and another bot: parseFeishuMessageEvent passes botOpenId into normalizeMentions, which removes the receiving bot's own placeholder before buildFeishuAgentBody constructs the prompt.

## Solution Assessment

No for the issue's suggested full removal of botStripId behavior, because #35994 depends on stripping the leading bot mention for slash-command probing. The narrower leading-prefix-only direction in open PR #72661 is the safer maintainable fix path.
