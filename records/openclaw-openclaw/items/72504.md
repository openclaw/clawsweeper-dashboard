## Summary

Keep open. Current `main` still strips the receiving Feishu bot's own mention before agent-facing content is built, the latest stable release has the same behavior, and live open PR https://github.com/openclaw/openclaw/pull/72661 explicitly closes this issue while https://github.com/openclaw/openclaw/pull/72532 remains an overlapping alternate fix path.

## What This Changes

The issue reports that Feishu group messages mentioning multiple bots lose the receiving bot's own inline `<at>` mention before agent dispatch, causing the bot to return `NO_REPLY`.

## Best Possible Solution

Land one focused Feishu plugin fix that preserves semantic inline self-mentions while keeping leading mention-prefixed slash commands command-clean, with regression coverage and redacted real Feishu proof.

## Reproduction Assessment

Yes, at source level. A Feishu group text event containing Bot A and Bot B mention placeholders, parsed with Bot A's `open_id`, will have Bot A's placeholder removed from `ctx.content` before the agent body is built.

## Solution Assessment

No for the issue's blanket suggestion to remove `botStripId` entirely. The narrower maintainable fix is to preserve inline self-mention context while keeping leading addressing prefixes and slash commands clean, or to separate command-probe text from agent-facing content.
