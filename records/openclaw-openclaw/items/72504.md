## Summary

Keep open. Current main still removes the receiving Feishu bot's own mention from the agent-facing prompt, and an open PR with closing syntax already tracks the focused fix path, so this should stay open until that PR or an equivalent replacement merges.

## What This Changes

The issue requests Feishu mention normalization that preserves the receiving bot's own mention so multi-bot group messages do not route to silent NO_REPLY behavior.

## Best Possible Solution

Land one focused Feishu change that strips only the leading addressing prefix needed for command probing while preserving inline or repeated self-mentions in the agent prompt, with regression coverage for multi-bot group messages; then let the linked issue close after merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group text event containing Bot A and Bot B mentions while processing with Bot A's open_id; current parseFeishuMessageEvent passes Bot A's id into normalizeMentions, which removes Bot A from ctx.content before the agent prompt is built.

## Solution Assessment

No for the issue's broad suggested removal of all botStripId behavior. Existing code, tests, and changelog history show the self-strip also protects #35994 slash-command probing, so a narrower leading-prefix-only fix like #72661 is the safer maintainable solution.
