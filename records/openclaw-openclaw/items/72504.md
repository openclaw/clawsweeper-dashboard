## Summary

Keep open. Current main still strips the receiving Feishu bot's own mention before constructing the agent-facing body, and open PR #72661 explicitly closes this issue with #72532 as an alternate active implementation, so this should wait for one focused fix to land.

## What This Changes

The issue asks Feishu group handling to preserve a receiving bot's own `<at>` mention in multi-bot mentions so each bot can recognize it was addressed instead of returning `NO_REPLY`.

## Best Possible Solution

Land one focused Feishu fix, preferably through #72661 or a replacement, that preserves model-facing self-mention context where it matters while keeping mention-prefixed slash commands covered by regression tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Feishu group text event with Bot A and Bot B mention placeholders while parsing with Bot A's `open_id`; current `normalizeMentions` removes Bot A's placeholder before `BodyForAgent` is constructed.

## Solution Assessment

Unclear for the issue's suggested removal of all `botStripId` behavior. The safer direction is to preserve semantic self-mentions in agent-facing content while keeping command probing independent, or strip only the leading addressing prefix so #35994 remains protected.
