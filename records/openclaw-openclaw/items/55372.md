## Summary

Keep open: current main and the latest stable release still send a restart-sentinel notice to any resolved channel route, and there is no config or request flag to suppress only that outbound notification while preserving wake behavior.

## What This Changes

Request a config option to suppress direct outbound restart-sentinel notifications while keeping session wake and heartbeat behavior.

## Best Possible Solution

Choose a narrow core-owned contract that suppresses only the direct outbound restart notice while preserving system-event wake and explicit continuation behavior, then cover config.patch/apply/update restart paths with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current main still sends the restart-sentinel notice whenever a route resolves, and there is no suppression field in the config protocol. I did not run a live Telegram/webchat restart.

## Solution Assessment

Unclear until maintainers choose the API shape. The problem is real, but placing this under `hooks.internal.entries.restart-sentinel` is probably not the best contract because restart sentinel recovery is gateway core behavior, not an internal hook.
