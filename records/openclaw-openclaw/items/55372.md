## Summary

Keep open: current main and v2026.6.8 still deliver a restart-sentinel outbound notice whenever a channel route resolves, and the requested suppression control remains a new config/API product decision rather than implemented cleanup.

## What This Changes

The issue requests a configuration option to suppress restart-sentinel outbound notifications while preserving session wake, heartbeat, and continuation behavior.

## Best Possible Solution

Maintainers should choose one narrow core-owned suppression contract that gates only the direct outbound restart notice while preserving queued system events, heartbeat wake, continuation routing, and route/thread behavior across config.patch, config.apply, update, and restart paths.

## Reproduction Assessment

Yes, at source level: current main still sends the restart-sentinel notice whenever a route resolves, and the config/update protocol has no suppression field. I did not run a live Telegram or dashboard restart in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape. The problem is real, but placing it under hooks.internal.entries.restart-sentinel is likely not the cleanest contract because restart sentinel recovery is gateway core behavior, not an internal hook.
