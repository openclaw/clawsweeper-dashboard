## Summary

Keep open: current main and v2026.6.10 still send the routed restart-sentinel notice with no supported suppression field, so the requested behavior remains missing. The remaining work is product/API direction for a new public config or request surface, not a strict bug-fix lane.

## What This Changes

The issue requests a public configuration option to suppress restart-sentinel outbound notifications while preserving the restart wake/session behavior.

## Best Possible Solution

Maintainers should choose one narrow core-owned suppression contract that gates only the direct outbound restart notice while preserving queued system events, heartbeat wake, continuation routing, and route/thread behavior.

## Reproduction Assessment

Yes, at source level: current main still delivers the restart-sentinel notice whenever the route resolves, and the protocol/config schema exposes no suppression field. I did not run a live Telegram/dashboard restart in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape. The problem is real, but the best fix must add a narrow public suppression contract without disabling wake or continuation recovery.
