## Summary

Keep open: current main and the latest release still send a routed restart-sentinel notice whenever a channel target resolves, and there is still no config/request field to suppress only that outbound notification while preserving restart wake behavior.

## What This Changes

Request to add a configuration or API option that suppresses visible restart-sentinel notifications while preserving wake, heartbeat, and continuation behavior.

## Best Possible Solution

Define one narrow core-owned suppression contract that gates only the direct outbound restart notice while preserving queued system events, heartbeat wake, and continuation routing, then cover config.patch, config.apply, update, and restart paths with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current main still sends the restart-sentinel notice whenever a route resolves, and the config/update protocol has no suppression field. I did not run a live Telegram or dashboard restart.

## Solution Assessment

Unclear until maintainers choose the API shape. The problem is real, but placing it under hooks.internal.entries.restart-sentinel is probably not the best contract because restart sentinel recovery is gateway core behavior, not an internal hook.
