## Summary

Keep this PR open for review, but the current patch is not ready: the direction is useful, current main does not already provide structured runtime incidents, and the diff has a concrete event-routing bug plus insufficient real fallback-path proof.

## What This Changes

The PR adds a typed `incident` agent event and emits classified runtime-fallback metadata from reply-runner failures while preserving existing fallback reply text.

## Best Possible Solution

Land this as an intentional runtime-observability feature only after preserving incident session metadata for hidden channel runs, adding a regression that observes the real event bus, and aligning the new stream with the intended SDK/docs contract.

## Reproduction Assessment

Yes, by source inspection: register a run context with `isControlUiVisible: false`, emit the new incident with a `sessionKey`, and `emitAgentEvent()` strips it because `incident` is not lifecycle. I did not run a writing test command in this read-only review.

## Solution Assessment

No. The proposed event shape is reasonable, but the implementation should preserve incident routing metadata for hidden channel runs and prove the actual fallback emitter path rather than only the helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve session keys on incident events:** `src/infra/agent-events.ts:325`
  - body: For channel-routed runs, `runAgentTurnWithFallback()` registers the run as not control-UI-visible, and `emitAgentEvent()` only preserves `sessionKey` for visible or lifecycle streams. This new `incident` stream is non-lifecycle, so external Discord/Telegram fallback incidents reach direct `onAgentEvent` observers without the session key the watchdogs need for routing. Include `incident` in the preservation path and add an event-bus regression that does not mock the emitter.
  - confidence: 0.86
