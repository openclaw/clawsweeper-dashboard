## Summary

Keep open: this is a recent focused feature PR, and current main does not already provide structured runtime-fallback incidents. The diff still has a concrete routing bug for hidden channel runs, and the supplied proof exercises the new helper directly rather than the reply-runner fallback path it changes.

## What This Changes

The PR adds a typed `incident` agent-event stream and emits classified reply-runner fallback metadata while keeping the existing user-visible fallback text unchanged.

## Best Possible Solution

Land the feature only after preserving incident routing metadata for hidden channel runs, adding a real event-bus regression that does not mock the emitter, and clarifying the intended observer contract.

## Reproduction Assessment

Yes, by source inspection: register a run context with `isControlUiVisible: false`, emit the PR's new `incident` event with a `sessionKey`, and current `emitAgentEvent()` strips it because only lifecycle events preserve hidden session keys. I did not run a writing test command in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but the implementation should preserve the routing metadata the incident is meant to carry and prove the actual fallback emitter path, not only the helper.
