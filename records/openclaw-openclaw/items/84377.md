## Label Justifications

- `P1`: The PR targets an urgent agent runtime placement/security regression affecting sandboxed Codex/OpenAI workflows.
- `merge-risk: 🚨 security-boundary`: Merging as-is could leave the default sandbox containment gap in place while appearing to fix managed exec placement.
- `merge-risk: 🚨 compatibility`: The PR intentionally changes implicit OpenAI/OpenAI-Codex runtime selection from Codex to PI for managed exec placements, which can change existing runtime behavior.

## Summary

Keep open: the PR is aimed at the right runtime boundary, but it misses the documented default sandbox path because `host=auto` becomes sandbox when a sandbox runtime is active while the new guard only checks literal `node` and `sandbox`.

## What This Changes

The PR threads exec placement into agent harness policy selection so implicit OpenAI/OpenAI-Codex routes stay on PI for managed node or sandbox exec placement, while preserving explicit Codex runtime pins.

## Best Possible Solution

Resolve and pass the effective exec placement into harness policy selection, treating `host=auto` with an active sandbox as managed sandbox placement, while preserving explicit Codex pins and adding regression plus redacted real-session proof.

## Reproduction Assessment

Yes, at source level: current exec resolution defines `auto` as sandbox when sandboxAvailable is true, and the PR's new guard only checks configured host strings. I did not run a full Docker/Gateway canary in this read-only review.

## Solution Assessment

No. The solution is on the right seam, but it should use effective exec placement or sandbox availability rather than only the literal configured host value.
