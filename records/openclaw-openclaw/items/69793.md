## Summary

Keep open. Current main still builds the Telegram no-text photo body from a shape-based `<media:image>` placeholder before saved media `contentType` can correct it, and the only related broad PR that claimed this fix is closed unmerged.

## What This Changes

Telegram inbound media body text should classify downloaded media by actual content type instead of always labeling photo-shaped or non-audio media as images.

## Best Possible Solution

Add a Telegram-local media placeholder classifier that uses saved `allMedia[].contentType` when downloads exist, preserves existing missing-download placeholders and audio transcript behavior, and locks the behavior with focused unit tests.

## Reproduction Assessment

Yes. Source inspection gives a focused reproduction: call `resolveTelegramInboundBody()` with no text, `msg.photo`, and `allMedia: [{ path, contentType: "application/octet-stream" }]`; current main initializes `bodyText` from `<media:image>`. I did not execute it because this review was constrained to read-only inspection.

## Solution Assessment

Yes. Deriving the no-user-text placeholder from saved media MIME in the Telegram body path is the narrow maintainable fix; the hardcoded `image/jpeg` fallback discussed in the issue is not the right solution.
