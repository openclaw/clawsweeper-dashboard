## Summary

Keep open. Current main still lets no-caption Telegram photo-shaped messages seed `bodyText` from the message-shape `<media:image>` placeholder before saved media MIME is considered; the issue discussion's corrected direction matches the source, while the referenced broad PR is closed unmerged and the earlier merged Telegram PR only fixed failed-download placeholder surfacing.

## What This Changes

The issue asks OpenClaw to make Telegram inbound media placeholder text reflect the downloaded media content type instead of always labeling photo-shaped non-audio inbound media as `<media:image>`.

## Best Possible Solution

Add a Telegram-local media placeholder classifier that derives no-user-text placeholders from `allMedia[].contentType` when saved media exists, keeps message-shape placeholders for missing or failed downloads, and covers octet-stream/document, video, mixed attachments, audio transcript, and multiple-image cases in focused body-builder tests.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to call `resolveTelegramInboundBody()` with no user text, a `msg.photo`, and `allMedia: [{ path, contentType: "application/octet-stream" }]`; current code returns `<media:image>` because `rawBody` is set from `resolveTelegramPrimaryMedia()` before saved media MIME can correct it.

## Solution Assessment

Yes. Deriving the placeholder from saved `allMedia[].contentType` in the Telegram body path is the narrowest maintainable fix; the hard-coded photo MIME fallback discussed in the issue is not the right direction, and a broad cross-channel refactor can be separate follow-up.
