## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` params, so this PR targets a real bug linked to #74635 and #62102, but the branch is not merge-ready because it adds a Paperclip-named public protocol key without maintainer/API-owner approval and lacks the required changelog entry.

## What This Changes

This PR adds an optional opaque root-level `paperclip` field to Gateway `AgentParamsSchema`, regenerates Swift protocol models, and adds validator tests for Paperclip payloads.

## Best Possible Solution

Land one additive, documented Gateway `agent` params compatibility contract for opaque adapter metadata, keep strict rejection of unrelated root typos, regenerate clients, and consolidate the Paperclip reports after merge.

## Reproduction Assessment

Yes. A current-main `agent` request with required `message` and `idempotencyKey` plus root `paperclip` follows the reported payload and fails before dispatch because `AgentParamsSchema` is closed and AJV preserves extra fields.

## Solution Assessment

No. The patch likely unblocks the exact Paperclip payload, but the maintainable fix is either a generic opaque adapter metadata contract or explicit protocol-owner approval for a Paperclip-named compatibility field, plus changelog/docs/client alignment.
