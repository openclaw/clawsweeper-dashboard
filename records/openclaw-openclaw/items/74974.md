## Summary

Keep open. Current main still rejects root-level Paperclip metadata on Gateway `agent` requests, so this PR addresses a real compatibility bug, but the branch should not merge as-is until maintainers choose the public protocol shape and resolve the overlap with #69139.

## What This Changes

The branch adds optional opaque `paperclip` metadata to Gateway `AgentParamsSchema`, mirrors it into generated Swift `AgentParams` models, and adds validator tests for accepting Paperclip metadata while preserving rejection of unrelated unknown fields.

## Best Possible Solution

Land one additive, documented Gateway protocol compatibility contract that preserves strict rejection of unrelated root fields, regenerates affected client models, keeps Paperclip regression coverage, includes the required changelog entry, and explicitly resolves whether adapter metadata belongs in a generic namespace or in a maintainer-approved compatibility exception.

## Reproduction Assessment

Yes. A Gateway `agent` request containing `message`, `idempotencyKey`, and a root-level `paperclip` value follows the reported payload and fails current-main `validateAgentParams` because the schema is closed and AJV does not remove additional fields.

## Solution Assessment

No. The proposed branch likely unblocks the immediate Paperclip payload, but a vendor-named root field is not the narrowest maintainable core protocol shape unless a maintainer explicitly approves that exception; a generic opaque adapter metadata field is the safer default.
