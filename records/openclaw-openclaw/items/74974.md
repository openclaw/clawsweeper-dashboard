## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, so the PR addresses a real regression, but the branch is not ready as-is because it makes a Paperclip-named field part of the public Gateway protocol and overlaps open PR #69139 without a maintainer protocol-shape decision.

## What This Changes

This branch adds optional opaque `paperclip` metadata to `AgentParamsSchema`, regenerates Swift `AgentParams` models, and adds validator tests for accepting Paperclip metadata while still rejecting unrelated unknown fields.

## Best Possible Solution

Land one additive, documented Gateway protocol compatibility contract that preserves strict rejection of unrelated root fields, regenerates affected client models, keeps Paperclip regression coverage, and explicitly resolves whether adapter metadata belongs in a generic namespace or in a maintainer-approved compatibility exception.

## Reproduction Assessment

Yes. A Gateway `agent` request containing `message`, `idempotencyKey`, and a root-level `paperclip` value follows the reported payload and fails current-main `validateAgentParams` because the schema is closed and AJV does not remove additional fields.

## Solution Assessment

No. The proposed branch likely unblocks the immediate Paperclip payload, but a vendor-named root field is not the narrowest maintainable core protocol shape unless a maintainer explicitly approves that exception; a generic opaque adapter metadata field is the safer default.
