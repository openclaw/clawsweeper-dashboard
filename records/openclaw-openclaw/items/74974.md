## Summary

Keep this PR open. Current main still rejects the reported Paperclip `agent` payload, so the contribution remains relevant, but the patch is not merge-ready because it adds a vendor-named root Gateway protocol field without a maintainer API decision and still needs the required user-facing changelog entry; it also overlaps with open PR #69139.

## What This Changes

The PR adds optional opaque root-level `paperclip` metadata to Gateway `AgentParamsSchema`, regenerates Swift Gateway models, and adds validator tests for accepting Paperclip metadata while preserving rejection of unrelated unknown fields.

## Best Possible Solution

Land one additive, documented Gateway protocol compatibility contract that preserves strict rejection of unrelated root fields, regenerates affected client artifacts, keeps Paperclip regression coverage, includes a changelog entry, and explicitly resolves whether adapter metadata belongs in a generic namespace or in a maintainer-approved Paperclip compatibility exception.

## Reproduction Assessment

Yes. A Gateway `agent` request containing `message`, `idempotencyKey`, and a root-level `paperclip` value follows the reported payload and fails current-main validation because `AgentParamsSchema` is closed and AJV does not remove additional fields.

## Solution Assessment

No. The PR likely unblocks the immediate Paperclip payload, but a vendor-named root field is not the best maintainable core protocol shape unless a maintainer explicitly approves that compatibility exception; a generic opaque adapter metadata contract is the safer default.
