## Summary

Keep this PR open. The underlying Paperclip compatibility bug is valid on current main, and the branch contains useful schema, generated Swift, and validator-test work, but it is not merge-ready because it adds a vendor-named root key to the public Gateway protocol without a maintainer/API-owner decision and still lacks the required changelog entry.

## What This Changes

The PR adds an optional opaque `paperclip` field to Gateway `AgentParamsSchema`, mirrors it into generated Swift `AgentParams` models, and adds validator tests for accepting Paperclip metadata while still rejecting unrelated unknown fields.

## Best Possible Solution

Land one additive Gateway `agent` params compatibility contract for adapter metadata, preferably a generic opaque namespace unless maintainers explicitly approve a Paperclip compatibility exception, while keeping strict rejection of unrelated root typos, updating generated client artifacts, adding focused regression coverage, adding a changelog entry, and consolidating #62102, #74635, #69139, and #72565 after merge.

## Reproduction Assessment

Yes. A current-main Gateway `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` value follows the reported Paperclip payload and fails `validateAgentParams` before dispatch because the schema is closed and AJV does not strip additional properties.

## Solution Assessment

No. The branch likely unblocks the immediate Paperclip payload, but adding a vendor-named root field to the public Gateway protocol is not clearly the narrowest maintainable contract without maintainer approval; a generic opaque adapter metadata field or an explicit compatibility exception would be safer.
