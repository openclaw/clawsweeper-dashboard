## Summary

Keep open. Current main and the latest stable release still reject root-level Paperclip metadata on Gateway `agent` params, while this PR is the focused open implementation candidate with supplied terminal proof and green checks; the remaining blocker is maintainer protocol/API approval plus conflict resolution.

## What This Changes

This PR adds optional `adapterMeta` and `paperclip` fields to Gateway `AgentParamsSchema`, mirrors them in the shared Swift model, adds validator coverage, and updates the changelog.

## Best Possible Solution

Land a maintainer-approved additive `AgentParams` metadata contract after conflict resolution and exact-head checks, then close the linked Paperclip reports.

## Reproduction Assessment

Yes, from source inspection: a Gateway `agent` request with valid `message` and `idempotencyKey` plus root `paperclip` reaches the closed `AgentParamsSchema` validator and is rejected before dispatch. I did not rerun a live Paperclip Docker dispatch in this read-only review.

## Solution Assessment

Unclear until protocol owners approve the public contract. The generic `adapterMeta` namespace plus a temporary `paperclip` alias is more maintainable than repeated vendor-only root fields, but the namespace shape and alias removal policy are API decisions.
