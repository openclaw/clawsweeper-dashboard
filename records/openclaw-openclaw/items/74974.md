## Summary

Keep open. The bug is real on current main and the PR is a focused implementation candidate, but the branch still needs maintainer/API-owner approval for making `paperclip` a public root Gateway protocol key, plus a changelog entry and CI cleanup before merge.

## What This Changes

The PR adds an optional opaque `paperclip` field to Gateway `AgentParamsSchema`, regenerates Swift protocol models, and adds validator tests for Paperclip payloads and unknown-field rejection.

## Best Possible Solution

Land one maintainer-approved additive Gateway metadata contract, keep strict rejection of unrelated root typos, update generated clients/tests/docs/changelog, and then close the related Paperclip reports against that merged fix.

## Reproduction Assessment

Yes. A current-main `agent` request with valid `message` and `idempotencyKey` plus root-level `paperclip` follows the reported payload and fails `validateAgentParams` before dispatch because the schema is closed and AJV preserves extra fields.

## Solution Assessment

No. The branch likely unblocks the exact Paperclip payload, but making `paperclip` a root public protocol key is not the safest maintainable shape without explicit maintainer approval; a generic opaque adapter metadata namespace or an adapter-side migration is the safer contract if feasible.
