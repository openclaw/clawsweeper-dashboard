## Summary

Keep open. Current main still rejects the reported Paperclip `agent` payload, but this PR is not merge-ready until maintainers choose the public protocol shape and canonical branch among the overlapping Paperclip fixes; it also needs the required changelog entry.

## What This Changes

The PR adds optional opaque `paperclip` metadata to Gateway `AgentParamsSchema`, mirrors it into generated Swift `AgentParams` models, and adds validator coverage for accepting Paperclip metadata while still rejecting unrelated unknown fields.

## Best Possible Solution

Land one additive Gateway protocol compatibility contract for adapter metadata in `agent` params, keep strict rejection of unrelated root typos, update generated client artifacts, add focused regression coverage and a changelog entry, then consolidate #69139, #74635, #62102, and the closed duplicate #72565 against the merged fix.

## Reproduction Assessment

Yes. A current-main Gateway `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` value follows the reported payload and fails `validateAgentParams` before dispatch because the schema is closed and AJV preserves additional fields.

## Solution Assessment

No. The branch likely unblocks the immediate Paperclip payload, but as posted it adds a vendor-named root key to the public Gateway protocol without explicit maintainer approval; a generic opaque adapter metadata namespace, or an approved compatibility exception, is the safer path.
