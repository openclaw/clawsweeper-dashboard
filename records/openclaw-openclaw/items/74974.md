## Summary

Keep open. The Paperclip rejection is valid on current main and this PR contains useful compatibility work, but it is not merge-ready because it adds a vendor-named root Gateway protocol field without explicit protocol-owner approval and still lacks the required changelog entry.

## What This Changes

The PR adds optional opaque root-level `paperclip` metadata to Gateway `AgentParamsSchema`, mirrors it into generated Swift `AgentParams` models, and adds validator tests for accepting Paperclip payloads while rejecting unrelated unknown fields.

## Best Possible Solution

Land one additive Gateway `agent` params compatibility contract for opaque adapter metadata, keep strict rejection of unrelated root typos, update generated clients/tests/docs/changelog, then consolidate the Paperclip issue cluster against the merged fix.

## Reproduction Assessment

Yes. A current-main Gateway `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` value follows the reported payload and fails `validateAgentParams` before dispatch because the schema is closed and AJV preserves extra fields.

## Solution Assessment

No. The branch likely unblocks the exact Paperclip payload, but making `paperclip` a public root protocol key is not the safest maintainable shape unless a maintainer explicitly approves that compatibility exception; a generic opaque adapter metadata namespace is the safer contract.
