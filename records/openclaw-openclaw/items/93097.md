## Summary

Keep open: the patch addresses the right mcporter config-discovery bug, but it drops QMD's per-agent cache env from mcporter-launched `qmd mcp`, so local searches can read the wrong QMD index. The supplied proof is static/unit-test proof only, not a live after-fix `memory_search` through mcporter 0.10+.

## What This Changes

The PR adds a `mcporterEnv` without `XDG_CONFIG_HOME` or `XDG_CACHE_HOME` for qmd-manager mcporter spawns and updates the spawn-env unit test.

## Best Possible Solution

Prevent mcporter's own config lookup from using QMD-scoped `XDG_CONFIG_HOME` while preserving or explicitly injecting QMD's per-agent `QMD_CONFIG_DIR` and `XDG_CACHE_HOME` for any local qmd MCP server, then prove the live mcporter-backed search path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main passes QMD-scoped XDG env into mcporter, mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery, and the patch removes `XDG_CACHE_HOME` from the process env inherited by local qmd MCP servers. I did not run the live daemon scenario in this read-only review.

## Solution Assessment

No, not as submitted. The narrow env split is plausible, but the best fix must separate mcporter config discovery from qmd server runtime env instead of dropping all QMD XDG state from mcporter-launched qmd.
