## Summary

Keep open: current main still has the reported workspace-isolation bug, but the submitted PR is stale/merge-conflicted and still has blocking implementation issues around hot-path config loading and inconsistent agent-id use.

## What This Changes

The branch changes mirrored session transcript header creation to derive `cwd` from a resolved agent workspace and adds focused regression coverage.

## Best Possible Solution

Land one current-main-compatible fix that removes `process.cwd()` from transcript header creation, threads existing config/effective agent context instead of reloading config in delivery bookkeeping, and covers mirrored delivery plus gateway-created transcript paths with regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows new mirrored and gateway transcript headers are still written from `process.cwd()`, and the delivery mirror path is awaited after successful sends.

## Solution Assessment

No. The direction is right, but this patch reloads config inside the mirror append helper, applies the resolved agent id only to header cwd, and is dirty against current main; the safer fix should thread already-available config and one effective agent id through all relevant header writers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Thread existing config into transcript mirroring:** `src/config/sessions/transcript.ts:162`
  - body: `deliverOutboundPayloads()` awaits this mirror append after channel sends have succeeded. Adding `loadConfig()` here means an invalid or unreadable on-disk config can turn bookkeeping into a failed or retryable delivery even though the user already received the message. Thread the already-resolved runtime config/workspace context into this path instead.
  - confidence: 0.86
- **[P2] Use one effective agent id consistently:** `src/config/sessions/transcript.ts:163-167`
  - body: The patch derives a trimmed/default agent id for the header cwd, but store and session-file resolution still use `params.agentId`. Calls with an omitted or whitespace id can therefore derive the header from one agent while resolving the transcript store/path with another. Compute the effective id before store/path resolution and pass it consistently.
  - confidence: 0.78
