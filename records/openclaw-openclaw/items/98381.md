## Summary

Keep this PR open: the underlying mcporter parse bug is real and the guard is in the right neighborhood, but the current patch exposes raw mcporter stdout through the memory_search error path and the provided proof still does not exercise the actual patched QmdMemoryManager/mcporter subprocess path.

## What This Changes

The PR wraps `runQmdSearchViaMcporter` stdout parsing in a try/catch, preserves the caught parse error as the cause, and adds a mocked regression test for non-JSON mcporter stdout.

## Best Possible Solution

Keep the mcporter parse guard and caught-error cause, but remove or redact stdout content from the thrown error and add after-fix proof that exercises the actual QmdMemoryManager/mcporter path.

## Reproduction Assessment

Yes from source, not from an executed repro. Current main parses successful mcporter stdout with a bare `JSON.parse`, so a zero-exit non-JSON stdout would throw a raw SyntaxError from `manager.search`.

## Solution Assessment

No. Wrapping the parse at the mcporter boundary is the right narrow fix, but the best implementation should preserve the cause without embedding raw stdout before visibility filtering and should be backed by actual runtime proof.
