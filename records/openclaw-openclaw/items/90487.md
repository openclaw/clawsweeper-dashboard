## What This Changes

This PR adds stream-only `Accept: text/event-stream` request options for native ChatGPT/Codex Responses SDK calls and narrows missing-`Content-Type` stream normalization to native ChatGPT/Codex SSE bodies.

## Best Possible Solution

Land the focused hardening if maintainers accept fail-closed missing-header behavior for non-native and non-SSE stream responses; otherwise preserve the compatibility path separately and make strict behavior explicit.

## Reproduction Assessment

Yes, from source inspection: current main's Responses stream path calls `buildOpenAISdkRequestOptions(model, signal)` without stream context, so the transport-aware ChatGPT/Codex alias cannot add the SSE `Accept` header. I did not run the failing live endpoint in this read-only review.

## Solution Assessment

Yes, this is a focused and maintainable fix if maintainers accept the fail-closed policy: it scopes request and response tolerance to the native ChatGPT/Codex backend and tests the alias path. A broader missing-header fallback would be riskier because it can mask wrong-base or non-native provider responses.
