## Summary

Keep open: current main and the latest release still only support the existing context-window-relative memory-flush threshold, and the requested absolute token trigger would be a new public compaction config contract that needs maintainer API/product direction.

## What This Changes

The issue asks OpenClaw to add `memoryFlush.absoluteThresholdTokens` so pre-compaction memory flush can trigger at a fixed total token count independent of the active model context window.

## Best Possible Solution

Keep this open as the canonical absolute-token memory-flush threshold request, then implement the accepted contract across schema/types, memory-core planning, runtime threshold evaluation, docs, and focused tests if maintainers approve it.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.6` lack `absoluteThresholdTokens`, and the memory-flush gate still computes a context-window-relative threshold. I did not run a live long-session agent scenario because the missing config surface is already clear from strict schema and runtime source.

## Solution Assessment

Unclear as proposed: an additive absolute field is plausible, but maintainers should decide whether OpenClaw wants an absolute ceiling, percentage/share semantics, or both before changing the public compaction config and memory plugin contract.
