## Summary

Keep this PR open. The new live Discord proof is sufficient for the fallback path, but the patch still has a blocking compatibility problem: it changes documented/current `image_generate` behavior so a distinct image prompt is blocked by any active image task, and the docs still state generated media is not auto-posted as a fallback.

## What This Changes

The PR keeps generated-media tasks active through completion delivery, adds image/music/video duplicate guards, adds direct generated-media fallback delivery, and wraps prompt-stream transcript appends in the session write-lock context.

## Best Possible Solution

Preserve the current distinct-image prompt contract unless maintainers explicitly choose a one-active-image-task policy, and align the generated-media fallback docs/tests with the intended delivery behavior before merge.

## Reproduction Assessment

Yes for the review finding: start one session-backed image task, then call `image_generate` with a different prompt. The PR's own source and test assert the second call returns the active task, while current main and docs allow distinct image prompts to start.

## Solution Assessment

No. The direct fallback is a plausible fix for missed generated-media delivery, but the PR should not silently change image concurrency or leave docs contradicting the new fallback behavior.
