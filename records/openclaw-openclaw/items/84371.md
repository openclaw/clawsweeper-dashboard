## Summary

This PR should stay open for human review: the live proof is strong, but the patch still introduces a documented compatibility break for distinct image prompts and changes generated-media fallback semantics without aligning docs or product policy.

## What This Changes

The PR keeps generated-media tasks active until delivery, adds request-key duplicate guards for image/music/video generation, adds direct generated-media fallback delivery, and wraps prompt-stream transcript appends in the session write-lock context.

## Best Possible Solution

Preserve prompt/request-scoped image concurrency unless maintainers explicitly choose a one-active-image-task policy, and align generated-media fallback docs/tests with the intended direct-delivery behavior before merge.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: start one session-backed image task, then call `image_generate` with a different prompt. The PR source and tests assert the second call returns the active task, while current main code and docs allow distinct image prompts to start.

## Solution Assessment

No. The direct fallback is a plausible fix for missed generated-media delivery, but the PR should not silently change image concurrency or leave docs contradicting the new fallback behavior.
