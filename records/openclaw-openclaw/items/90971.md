## Label Justifications

- `P1`: Affected relocated-state installs can fail every exec tool call when the home OpenClaw directory is symlinked, and the path is part of the command-approval workflow.
- `impact:security`: Exec approvals are a command-execution authorization boundary, and ignoring the configured state directory can bypass or split the intended approval state.
- `impact:other`: The report also describes split runtime state between `$OPENCLAW_STATE_DIR` and the home OpenClaw directory.

## Summary

Keep open: current main and the latest release still resolve exec approvals from a home-relative path, and an open linked PR now owns the fix path. The issue should close only after that PR or an equivalent change merges.

## What This Changes

The issue asks for exec approvals file and socket paths to honor `OPENCLAW_STATE_DIR` instead of always using the home OpenClaw directory.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90972 or an equivalent fix that resolves exec approvals from the canonical state directory while preserving symlink safety.

## Reproduction Assessment

Yes from source inspection: the path helpers use hard-coded `~/.openclaw` defaults while the state-dir resolver contract honors `OPENCLAW_STATE_DIR`. I did not run a filesystem repro because this was a read-only review, but the current source and tests make the failure path clear.

## Solution Assessment

Yes, the best fix is to route default exec approvals paths through `resolveStateDir()` and keep the existing symlink/file destination protections. The linked PR appears to target that narrow boundary, but it still needs normal PR review before this issue closes.
