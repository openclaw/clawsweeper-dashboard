## Summary

Close: the reported `finish_reason: "tool_calls"` response is the documented OpenAI-compatible client-tool contract, not a current OpenClaw bug. The actionable fix for the OpenManager adapter is to execute the returned tool call and send a `role: "tool"` follow-up, or call OpenClaw's `/tools/invoke` endpoint when it wants OpenClaw to invoke an allowed tool.

## What This Changes

The issue requests server-side tool execution or memory prefetch for a caller-supplied `memory_search` function in `/v1/chat/completions`.

## Best Possible Solution

Keep the documented client-tool contract; external adapters should implement the OpenAI tool loop and use `/tools/invoke` only when they need OpenClaw to run an allowed tool.

## Reproduction Assessment

Yes for the reported response shape: current code, tests, docs, and v2026.6.10 release source all show pending client tool calls are returned with `finish_reason: "tool_calls"`, and empty content is allowed. That reproduces the observation but confirms the documented contract rather than a current-main bug.

## Solution Assessment

No. Automatic memory prefetch or server-side execution of caller-declared functions would add new Gateway behavior and policy/config surface; the supported solution is a client-side tool loop plus `/tools/invoke` for allowed OpenClaw-owned tool execution.
