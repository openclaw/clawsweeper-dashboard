## Summary

Keep open. Current main still preserves the config-selected-over-installed precedence and emits only a generic warning, while the install success path, plugin doctor, and inspect output still do not make the just-installed npm/global source visibly inactive or provide the requested repair guidance.

## What This Changes

Request clearer plugin CLI diagnostics when a successful npm/global install is shadowed by an explicit config-selected plugin source with the same id.

## Best Possible Solution

Add targeted diagnostics and repair hints around the existing registry/install-record model while preserving duplicate precedence and requiring explicit user config edits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: config load paths become config-origin candidates, install records become global candidates, config wins precedence, and current install/doctor/inspect output does not surface the shadowed installed source as actionable.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse existing duplicate diagnostics, install records, and registry inspection to warn and suggest repair; it should not change precedence or auto-edit user config.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76605 as a focused plugin CLI diagnostics PR. Preserve existing duplicate precedence: explicit config-selected/local sources must continue to win over installed npm/global sources, and do not auto-delete or auto-edit user config. Add a warning after successful `openclaw plugins install <npm-spec> --force` or equivalent npm/global install when the installed record for that plugin id is not the active source because a config-origin source wins. Add `openclaw plugins doctor` output that turns this condition into an actionable repair hint when the active config-selected source fails to load and an installed global/npm alternate exists. Improve `openclaw plugins inspect <id>` so active source vs installed alternate is visibly distinct or points to doctor when the installed source is shadowed. Keep diagnostics free of secrets/auth state. Update focused tests for install, doctor, inspect, and manifest/registry behavior without changing duplicate-precedence semantics. Add a `CHANGELOG.md` fix entry because this changes user-facing CLI diagnostics.
