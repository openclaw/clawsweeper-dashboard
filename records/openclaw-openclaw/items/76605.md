## Summary

Keep open. Current main preserves config-selected plugin precedence and already emits a generic duplicate warning, but the install success path, plugin doctor, and inspect output do not yet make a just-installed npm/global source visibly inactive or provide the requested repair guidance.

## What This Changes

Add plugin CLI diagnostics that warn when an npm/global plugin install is shadowed by a config-selected source and surface repair guidance in doctor/inspect.

## Best Possible Solution

Add targeted diagnostics around the existing registry/install-record model so users can see that the npm/global install succeeded but is not active, while preserving duplicate precedence and leaving config repair explicit.

## Reproduction Assessment

Yes. The source path is high-confidence: config load paths become config-origin candidates, installed npm records become global candidates, config wins precedence, and current install/doctor/inspect output does not surface the shadowed npm source as actionable.

## Solution Assessment

Yes. The best fix is a narrow diagnostic and repair-hint addition using existing duplicate diagnostics, install records, and registry inspection; it should not change load precedence or auto-edit user config.
