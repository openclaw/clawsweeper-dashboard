## Summary

Keep open. Current main still preserves config-selected plugin precedence and records the generic duplicate warning, but the install success path, plugin doctor output, and inspect output do not yet make the newly installed npm/global source visibly inactive or provide the requested repair guidance.

## What This Changes

The issue requests clearer plugin install, inspect, and doctor diagnostics when a newly installed npm/global plugin is shadowed by an explicit config-selected source with the same plugin id.

## Best Possible Solution

Add targeted diagnostics around the existing registry and install-record model so users can see that the npm/global install succeeded but is not active, while preserving duplicate precedence and leaving config repair explicit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a local/config source and an installed npm/global source with the same plugin id, then config wins precedence while install and doctor omit the requested inactive-source warning.

## Solution Assessment

Yes. The narrow maintainable fix is to add diagnostics and repair hints using existing duplicate diagnostics, install records, and inspect/doctor output paths without changing load precedence or auto-editing user config.

## Repair Work Prompt

Fix the plugin shadowing diagnostics requested in https://github.com/openclaw/openclaw/issues/76605. Preserve existing duplicate precedence semantics and do not auto-delete or mutate user plugin config beyond normal install record writes. Use existing plugin registry diagnostics, install records, and active registry entries to detect when an npm/global installed plugin id is still shadowed by an explicit config-selected source. Add a warning after `openclaw plugins install <npm-spec> --force` when the installed npm/global source is not active because a config source wins. Teach `openclaw plugins doctor` to report an actionable shadowed-source hint when the active config source fails and an installed npm/global alternate exists, with source-specific repair guidance for `plugins.load.paths` versus targeted entry sources. Make `openclaw plugins inspect <id>` visibly distinguish the active source from a shadowed installed alternate or point to `plugins doctor`. Add focused regression tests, keep duplicate-precedence tests intact, and add a user-facing `CHANGELOG.md` fix entry. Do not change plugin loading precedence, do not auto-edit config, and do not expose secrets or auth state in diagnostics.
