## Summary

Keep open: the route-ignore and diagnostics pieces are plausible, but the proposed recovery path is not merge-ready because it treats a broad ACPX code-1 failure as proof that Discord conversation bindings are stale and the PR has no live Discord proof.

## What This Changes

The PR changes Discord route preflight to ignore ACP bindings when ACP/acpx is disabled, unbinds ACP conversation bindings on acpx code-1 turn failures with updated error text, and adds ACPX launch/exit diagnostics plus tests.

## Best Possible Solution

Keep the config-disabled route-ignore and diagnostics, but narrow conversation unbinding to proven stale ACP metadata/resume-state failures or a structured manager recovery signal, then verify with a redacted live Discord turn.

## Reproduction Assessment

No live high-confidence reproduction was provided. Source inspection supports the central failure shape because current main treats ACP session bindings as explicit route intent and lacks a cfg-aware disabled-ACP check, while docs define disabled dispatch/plugin policy as blocking automatic ACP routing.

## Solution Assessment

No. The route-ignore fix is in the right area, but unbinding on every ACPX code-1 failure is broader than the current manager and dependency contracts justify; a structured stale-binding signal or narrower stale-session predicate is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep code-1 exits from unbinding valid ACP routes:** `src/auto-reply/reply/dispatch-acp.ts:228`
  - body: `acpx exited with code 1` is already handled as a broad early-runtime failure: the manager clears the runtime handle and retries once for any such exit, and the acpx dependency does not mark that string as stale-binding proof. By using the same string to call `unbind()`, a valid Discord ACP binding can be removed after repeated ACPX crash/auth/permission failures, so the next message may fall back to the native/default route instead of the bound ACP session. Please restrict unbinding to a stale-session signal or a structured recovery result that proves the binding is invalid.
  - confidence: 0.86
- **[P2] Do not claim cleanup from the generic ACP formatter:** `packages/acp-core/src/runtime/error-text.ts:20`
  - body: `formatAcpRuntimeErrorText` is shared by ACP surfaces that do not call `maybeUnbindStaleBoundConversations`, and it cannot know whether unbind removed anything or failed. This will tell users "I cleared" for matching code-1 failures even when no cleanup happened; keep this formatter neutral or emit the cleanup-specific text only from the dispatch path after successful removal.
  - confidence: 0.82
