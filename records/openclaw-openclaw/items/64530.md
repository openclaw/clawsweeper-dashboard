## Summary

Keep open. Current main still has useful replay-time and live-write safeguards, but it has not added the requested durable reopen/startup repair that appends idempotent recovery rows for unresolved tail tool calls or an explicit assistant marker for orphan user tails. The related open PR #74687 remains adjacent channel restart-recovery work rather than a replacement for this transcript-file repair.

## What This Changes

The issue requests first-party runtime transcript recovery on reopen that preserves existing rows and appends idempotent synthetic recovery rows for unresolved tail tool calls and orphan user tails.

## Best Possible Solution

Add a narrow runtime/session recovery pass before prompt build that inspects only the active persisted tail, appends parent-valid synthetic error toolResult rows for eligible unresolved non-aborted tool calls, appends one explicit assistant recovery marker for an orphan user tail, remains idempotent on repeated reopen, and never re-executes tools or rewrites older history.

## Reproduction Assessment

Yes. A high-confidence static reproducer is a session JSONL tail ending with an assistant toolUse/toolCall message and no matching toolResult, then reopening that session; current tests and source show the file repair pass preserves that tail unchanged. A second reproducer is a persisted user leaf before the next assistant turn, where current main merges or branches instead of appending a recovery marker.

## Solution Assessment

Yes. The requested tail-only, append-only first-party repair seam is the narrowest maintainable solution; the existing replay sanitizer and live guard are useful but do not make the persisted transcript source of truth healthy after restart.
