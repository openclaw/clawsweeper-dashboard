## Summary

Keep open. Current main has live-write guarding, in-memory replay sanitation, and later disk repair for non-tool assistant tails, but it still does not durably append recovery rows for unresolved persisted tail tool calls or explicit orphan-user recovery markers.

## What This Changes

The issue requests first-party reopen/startup transcript repair that appends idempotent recovery rows for unresolved tail tool calls and orphan user tails.

## Best Possible Solution

Add a narrow pre-prompt durable repair pass that inspects only the persisted tail, appends missing synthetic error `toolResult` rows and one explicit orphan-user recovery marker when eligible, and leaves healthy or already-repaired transcripts unchanged.

## Reproduction Assessment

Yes. A high-confidence source reproducer is a session JSONL ending in an assistant tool-call message with no matching tool result; current session-file repair preserves that tail unchanged while replay sanitation only fixes the in-memory provider history.

## Solution Assessment

Yes. The requested tail-only, append-only first-party repair seam is the narrowest maintainable fix because it makes the persisted transcript healthy without re-executing tools or rewriting older history.
