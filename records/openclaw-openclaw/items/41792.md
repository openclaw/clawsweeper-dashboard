## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested `openclaw/plugin-sdk/backup` subpath, so the PR is not obsolete; the blocker is that it promotes backup plan path inventory into the public plugin SDK without the required docs/API/changelog alignment or explicit data-boundary documentation.

## What This Changes

The PR adds a public `openclaw/plugin-sdk/backup` subpath through `package.json`, the plugin SDK entrypoint list, and a new facade that re-exports backup archive helpers, backup plan resolver/types, and backup create option/result types.

## Best Possible Solution

If maintainers want backup planning to be a supported plugin API, land a rebased, narrow facade that documents the trusted-plugin local path boundary, updates the SDK subpath docs and API metadata/baseline hash, adds the required changelog entry, and verifies the generated export/package artifacts.

## Reproduction Assessment

Not applicable as a runtime bug. The static review path is high-confidence: current main lacks the subpath, the PR adds it, and the missing docs/API/changelog plus path-exposure boundary are visible from source and SDK rules.

## Solution Assessment

No. The facade is a plausible shape, but it is incomplete for a public SDK contract until maintainers approve the local-data boundary and the SDK docs, API baseline metadata/hash, package export sync, and changelog are aligned.
