## Summary

Current `main` still drops the already-available provider/model context at the transient overload/rate-limit formatter, so the requested user-facing ambiguity remains. The issue should stay open while open PR #55851, which explicitly closes this issue, and the related narrower PR #58886 are reviewed or consolidated.

## What This Changes

Request to include the upstream API provider name in user-facing overload, rate-limit, and similar provider error messages so users can distinguish provider-side failures from gateway or local failures.

## Best Possible Solution

Land one focused formatter-level implementation that appends sanitized active provider/model context to generic 529/overload and 429/rate-limit fallback messages, preserves existing provider-specific raw rate-limit detail handling, and updates formatter, payload, lifecycle, and channel-facing tests.

## Reproduction Assessment

Yes. Source inspection on current `main` shows the active provider/model reaches `formatAssistantErrorText`, while the transient overload/rate-limit helper still receives only `raw` and returns generic provider-agnostic copy.

## Solution Assessment

Yes. The narrowest maintainable solution is to use the already-threaded active provider/model context at the final friendly error formatter boundary rather than changing provider selection, failover, or channel-specific reply code.
