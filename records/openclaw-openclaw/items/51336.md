## Summary

Keep this open: current main and v2026.5.18 still format overload and generic rate-limit errors without the active provider name, and the active closing implementation at https://github.com/openclaw/openclaw/pull/55851 is still open and conflicting.

## What This Changes

Request to include the active API provider name in user-facing overload and provider-error messages so users can distinguish upstream provider failures from local or gateway failures.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/55851 with a narrow shared-formatter change that uses sanitized active provider/model context for transient overload and rate-limit messages while preserving failover behavior and request-id privacy.

## Reproduction Assessment

Yes for source-level verification, not as a runtime bug reproduction. Current main still returns provider-agnostic overload and generic rate-limit copy while focused tests assert those generic strings.

## Solution Assessment

Yes, the shared formatter is the right maintainable place because active provider/model context already reaches `formatAssistantErrorText()`. The linked implementation still needs rebase plus privacy and failover review before it should close this issue.
