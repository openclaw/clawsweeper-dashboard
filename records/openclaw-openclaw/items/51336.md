## Summary

Keep this open: current main and the latest release still emit generic overload/rate-limit text without active provider context, and the remaining implementation is already owned by the open linked pull request https://github.com/openclaw/openclaw/pull/55851.

## What This Changes

The issue asks for user-facing provider overload/rate-limit errors to name the API provider so users can distinguish upstream provider failures from local gateway/server problems.

## Best Possible Solution

Refresh and review https://github.com/openclaw/openclaw/pull/55851, or replace it with a narrower current-main patch that adds sanitized active provider/model context to the shared transient formatter while preserving provider-specific retry details and avoiding raw identifiers.

## Reproduction Assessment

Yes, at source level. Current main still formats overloaded and generic rate-limit errors through a raw-only helper, and focused tests assert provider-free text even when provider/model context exists nearby.

## Solution Assessment

Yes, the shared formatter path is the right ownership boundary; Telegram should receive the formatted message rather than special-case provider wording. The open implementation should be reviewed or narrowed because its context surface is broader than the original provider-name ask.
