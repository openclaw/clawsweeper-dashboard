This issue should stay open because current main and the latest release still emit generic overload/rate-limit copy without the active provider name, and an open linked PR already owns the direct implementation path.

## What This Changes

Request to include the active API provider name in overload and rate-limit error messages so users can distinguish upstream provider failures from local gateway or channel problems.

## Best Possible Solution

Land the open linked formatter-based implementation, or a maintainer-approved replacement, that adds safe active provider/model context to overload and rate-limit copy while keeping raw identifiers out of user-facing messages.

## Reproduction Assessment

## Solution Assessment
