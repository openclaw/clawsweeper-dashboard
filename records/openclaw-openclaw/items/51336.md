## What This Changes

Request to include the active API provider name in user-facing overload, rate-limit, and similar provider error messages.

## Best Possible Solution

Land a focused formatter-level change that appends sanitized active provider/model context to generic overload and rate-limit messages while preserving failover classification and updating focused tests plus changelog.

## Solution Assessment

Yes on the code boundary, but the exact context scope needs maintainer choice. The narrowest maintainable path is to use the already-threaded active provider/model at the friendly formatter boundary, not channel-specific reply logic or provider selection changes.
