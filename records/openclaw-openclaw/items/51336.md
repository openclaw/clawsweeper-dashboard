## Summary

Keep this open: current main and latest release still use provider-agnostic overload/rate-limit copy, and the linked closing implementation PR remains open and conflicting with prior correctness/security blockers.

## What This Changes

Request to include the active API provider name in user-facing overload and rate-limit errors so provider-side failures are distinguishable from gateway or local failures.

## Best Possible Solution

Keep this as the canonical tracker and land or replace the linked PR with sanitized active provider/model context in the shared transient formatter, preserving provider retry details and request-id privacy.
