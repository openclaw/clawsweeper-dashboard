## What This Changes

The issue requests provider-identifying context in user-facing overload/rate-limit/provider error messages so users can distinguish upstream provider failures from OpenClaw gateway or local failures.

## Best Possible Solution

Refresh and review https://github.com/openclaw/openclaw/pull/55851, or replace it with a narrower current-main patch that adds sanitized active provider/model context to the shared transient formatter while avoiding raw identifiers.

## Reproduction Assessment

Yes, at source level. Current main still formats overloaded and generic rate-limit errors through a raw-only helper, and focused tests assert provider-free text in the affected cases.

## Solution Assessment

Yes, the shared formatter path is the right ownership boundary; Telegram and other channels should receive the formatted message rather than special-case provider wording. The open implementation still needs maintainer review because its context surface is broader than the original provider-name ask.
