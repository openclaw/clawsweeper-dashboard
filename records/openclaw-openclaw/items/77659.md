## Summary

Keep this draft maintainer PR open: the reported bare `openclaw channels` slow-help path is valid and the patch targets the right startup surfaces, but the lazy channel-option registration appears to regress the positional help form for `channels add`.

## What This Changes

The PR skips config/proxy/plugin startup for bare `openclaw channels` parent help, suppresses the help banner on that path, lazy-loads bundled `channels add` option metadata, and adds focused CLI tests plus a changelog entry.

## Best Possible Solution

Keep the fast bare-parent-help path, but make the channel option metadata lazy-load condition cover positional `channels help add` as well as `channels add --help` and normal `channels add` execution.

## Reproduction Assessment

Yes. Source inspection shows current main eagerly registers channel add metadata and runs preAction startup for bare `channels`, and the PR body includes before/after timing for that same path.

## Solution Assessment

No. The direction is narrow and maintainable, but the current guard is too narrow because it misses Commander positional help for `channels add`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve provider flags for positional add help:** `src/cli/channels-cli.ts:50-52`
  - body: OpenClaw configures a positional help command, and Commander dispatches `openclaw channels help add` to the `add` command's help. With this guard keyed only to `commandPath[1] === "add"`, that invocation registers `channels add` without bundled provider options, regressing the existing help surface. Include the positional help form in this lazy-load check so both help paths show provider flags.
  - confidence: 0.84
