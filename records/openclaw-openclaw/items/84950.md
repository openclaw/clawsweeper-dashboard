- **[P1] Generate shrinkwrap from a stable source:** `scripts/generate-npm-shrinkwrap.mjs:86-87`
  - body: `generateShrinkwrap()` runs `npm install --package-lock-only` in a fresh temp directory for every generate/check, so the checked-in shrinkwrap can become stale when upstream transitive ranges publish new versions or npm resolution changes, even if this repo did not change. For a release gate, derive or verify the graph from a stable reviewed lock source, or make the live-resolution policy explicit and isolated from byte-for-byte CI checks.
  - confidence: 0.86
- **[P1] Include plugin shrinkwraps in the git diff pathspec:** `scripts/dependency-changes-report.mjs:191`
  - body: `isDependencyFile()` includes `extensions/*/npm-shrinkwrap.json`, but `gitDiffDependencyFiles()` never asks `git diff` for those paths. A PR that changes only plugin shrinkwraps can therefore report zero dependency file changes, losing the security-review signal this PR is meant to add.
  - confidence: 0.93

  - body: `LEGACY_SHRINKWRAP_COMPAT_MAX` is set to 2026.5.20, while the latest published package is 2026.5.19. That lets the first shrinkwrap-hardened 2026.5.20 tarball pass this standalone validator without `npm-shrinkwrap.json`, contrary to the new fail-closed package contract.
