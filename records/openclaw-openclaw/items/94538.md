## Summary

Keep open for now: this PR targets a real heartbeat privacy bug, but the patch is not mergeable because the new guard is broader than the explicit message-tool privacy contract and it still copies suppressed private text into heartbeat status previews. The related issue remains canonical, and https://github.com/openclaw/openclaw/pull/94088 is the stronger open candidate but still has its own review blocker, so this PR should not be closed solely as safely superseded yet.

## What This Changes

The PR adds a no-tool heartbeat early return in `src/infra/heartbeat-runner.ts` that treats any forced heartbeat response-tool run without `heartbeat_respond` as a silent ack.

## Best Possible Solution

Land a scoped heartbeat-runner guard that only suppresses explicit message-tool privacy fallbacks, preserves marked notices and relayable exec completions, redacts heartbeat previews, and includes regression plus real behavior proof.

Yes. Source inspection gives a high-confidence path: configure message-tool visible replies, return non-empty heartbeat final text without `heartbeat_respond`, and current main falls through to normal delivery; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The heartbeat runner is the right boundary, but this implementation is too broad and still exposes private text through heartbeat status; the safer solution is the scoped guard shape used by the stronger candidate after its preview leak is fixed.
