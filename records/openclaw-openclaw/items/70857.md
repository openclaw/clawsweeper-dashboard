## Summary

Keep open. Current main still contains the reported session-store lock contention path: both main session-store update helpers acquire the write lock and then force cache-bypassing synchronous `sessions.json` loads while the lock is held, and the narrow related fix in #68554 remains open rather than merged or shipped.

## What This Changes

Investigate and fix slow startup, slow replies, and long-held `sessions.json.lock` warnings during OpenClaw gateway/session-store activity, especially on Windows.

## Best Possible Solution

Land #68554 or an equivalent narrow fix that removes redundant cache-bypassing disk rereads from locked session-store update paths, while preserving write ordering, cache coherency, stale-cache protection, and concurrent update correctness. Coordinate with adjacent session-write-lock cleanup PRs only where they touch the same lock-hold behavior.

## Reproduction Assessment

Yes. The issue discussion gives concrete timing logs across Windows and macOS, and current main has a source-level reproduction path: use a large or frequently updated `sessions.json`, trigger queued `updateSessionStore` or `updateSessionStoreEntry` calls, and observe synchronous cache-bypassing reads/parses inside the session-store lock.

## Solution Assessment

Yes for the linked direction, but it is not on main yet. The narrow maintainable solution is to avoid redundant `skipCache` disk rereads inside the serialized write lock without weakening the existing cache write-through and concurrent update guarantees.
