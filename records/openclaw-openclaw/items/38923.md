## Summary

Keep open: this PR contains useful avatar parser/test work, but it is not merge-ready because it does not address the Windows path-containment root cause, drops resolver-side avatar size validation, is stale against current main, and lacks real behavior proof.

## What This Changes

The branch unwraps backtick-wrapped `IDENTITY.md` avatar values, normalizes workspace-relative avatar paths, adds avatar parser/resolver tests and a changelog entry, and changes several `onPayload` wrapper call sites.

## Best Possible Solution

Refresh or replace this with a current-main avatar fix that normalizes Windows namespace/drive-letter comparisons in the shared avatar policy, unwraps single-backtick `IDENTITY.md` avatar values, preserves source/UI override/size checks, and adds focused parser, resolver, and route tests.

## Reproduction Assessment

Yes, at source level. Current main still preserves backticks in parsed avatar values, and the linked Windows report has a credible containment path involving mixed namespace and drive-letter forms; I did not run a live Windows UI reproduction.

## Solution Assessment

No. Backtick unwrapping is useful, but the PR normalizes only the relative avatar string, leaves the Windows containment root cause uncovered, and regresses the current resolver's max-size policy.

  - body: The reported Windows failure can come from comparing a namespaced workspace path with a normal drive-letter avatar path, but this change only normalizes the raw relative avatar string before calling `path.relative`.
