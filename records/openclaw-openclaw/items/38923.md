## Summary

Keep open: this is a plausible focused bug fix, but the current PR head is stale/dirty, lacks real after-fix behavior proof, and regresses current avatar resolver guarantees around UI overrides, source metadata, shared containment, and size limits.

## What This Changes

The PR changes workspace avatar parsing/resolution so `/avatar/{agentId}` can serve avatars defined in `IDENTITY.md`, and also changes several agent stream wrapper `onPayload` call sites.

## Best Possible Solution

Refresh or replace this with a narrow current-main patch that unwraps valid single-backtick avatar values and handles the Windows relative-path case while preserving shared avatar policy, source metadata, `AVATAR_MAX_BYTES`, UI override behavior, and focused route/parser/resolver coverage.

## Reproduction Assessment

Yes at source level: current main preserves backtick-wrapped `IDENTITY.md` avatar values as literal paths, and unresolved local avatar paths produce null metadata or 404 image responses. The exact Windows packaged gateway setup from the linked issue was not live-run.

## Solution Assessment

No. The parser/normalization direction is useful, but this PR head is not the best merge shape because it forks shared avatar containment, drops resolver-side size/source/UI override behavior, is dirty, and lacks real after-fix proof.
