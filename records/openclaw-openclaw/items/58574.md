## Summary

Keep open. Current main still emits the `cron: timer armed` payload with only numeric `nextAt` and has no readable companion field in the cron timer logs or `CronEvent` type, while open PR #58908 directly tracks an additive compatible fix. This issue should remain open until that paired PR is reviewed, rebased or replaced, and merged or explicitly rejected.

## What This Changes

Add a human-readable companion timestamp for cron scheduler `nextAt`/`nextRunAtMs` log and event output while preserving the numeric fields.

## Best Possible Solution

Review and land a narrow additive cron service change that keeps `nextAt`/`nextRunAtMs` numeric, adds ISO-8601 companion fields to the timer/backoff log payloads, and only extends `CronEvent` if maintainers want that as a public event contract; avoid changing persisted cron store shape.

## Reproduction Assessment

Yes. Current main can be checked by reading `armTimer`: the debug payload for `cron: timer armed` has only numeric `nextAt`, so a focused seam test can instantiate a future scheduled job and assert the logger payload lacks a readable companion field.

## Solution Assessment

Yes. The additive companion-field approach is the narrowest maintainable path because it solves operator readability without breaking numeric state, CLI/UI sorting, docs, or third-party consumers of `nextAt`/`nextRunAtMs`.
