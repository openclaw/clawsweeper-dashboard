The issue requests a human-readable timestamp companion for cron scheduling values such as `nextAt` and `nextRunAtMs`.

## Best Possible Solution

Review and land a rebased, range-safe additive implementation through https://github.com/openclaw/openclaw/pull/58908 or a narrow replacement while preserving the existing numeric fields.

## Solution Assessment

Yes as a direction. An additive readable ISO companion field is the maintainable fix because replacing `nextAt` or `nextRunAtMs` in place would break existing log/event/schema consumers.
