## What This Changes

The issue requests a human-readable companion timestamp for cron `nextAt` and `nextRunAtMs` values while preserving numeric compatibility.

## Best Possible Solution

Land or replace the linked additive implementation with a range-safe readable ISO timestamp companion while keeping existing numeric fields stable.

## Reproduction Assessment

Yes. Source inspection on current main shows the relevant cron timer logs, event type, and gateway schema still expose numeric `nextAt`/`nextRunAtMs` fields without a readable companion.

## Solution Assessment

Yes as a direction. An additive readable ISO companion field is the maintainable path because replacing `nextAt` or `nextRunAtMs` in place would break existing log, event, and schema consumers.
