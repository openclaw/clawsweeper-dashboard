## Summary

Keep open. Current main and v2026.5.22 still expose the cron timer, retry/backoff, event, and protocol timestamps as numeric-only fields, while the linked closing PR is still open and conflicting rather than a merged fix.

## What This Changes

The issue asks for cron `nextAt` or `nextRunAtMs` values to gain a human-readable companion timestamp while preserving the numeric value.

## Best Possible Solution

Review and land a narrow additive implementation through https://github.com/openclaw/openclaw/pull/58908 or a replacement, preserving numeric fields while adding range-safe ISO companion fields, focused tests, and redacted runtime log proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the timer log, retry/backoff logs, `CronEvent`, and gateway schema still expose only numeric `nextAt` or `nextRunAtMs` fields.

## Solution Assessment

Yes as a direction. An additive readable ISO companion field is the maintainable fix because replacing `nextAt` or `nextRunAtMs` directly would break existing log, event, and schema consumers.
