## Summary

Keep open. Current main still logs cron timer `nextAt` and emits cron `nextRunAtMs` as epoch milliseconds without a readable companion field, while open PR #58908 explicitly tracks the additive fix for this issue.

## What This Changes

Request a human-readable companion timestamp for cron scheduler `nextAt`/`nextRunAtMs` output while preserving the numeric value.

## Best Possible Solution

Land or replace the narrow additive fix that adds ISO/readable companion fields to cron logs/events while keeping epoch-ms fields stable.

## Reproduction Assessment

Yes. A source inspection path reproduces the gap: current `armTimer` logs only numeric `nextAt`, and targeted search finds no readable companion helper or field on the cron/gateway surfaces.

## Solution Assessment

Yes. The additive-field approach discussed in comments and in PR #58908 is the safest maintainable solution because it avoids changing the existing numeric contract.
