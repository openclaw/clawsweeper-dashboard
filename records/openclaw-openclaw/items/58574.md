## Summary

Keep open. Current main still emits cron scheduling timestamps as epoch milliseconds only, and an open closing PR already tracks the additive readable-field fix, so closing this issue before that PR lands would lose the active implementation thread.

## What This Changes

The issue requests a human-readable companion for cron `nextAt`/`nextRunAtMs` values while preserving the existing numeric epoch-ms contract.

## Best Possible Solution

Land or replace the narrow additive implementation that adds range-safe ISO/readable companion fields to cron logs/events while keeping `nextAt` and `nextRunAtMs` numeric.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: the timer log and event/schema surfaces expose only numeric epoch-ms fields, and targeted search finds no readable companion field.

## Solution Assessment

Yes. The additive companion-field approach discussed in the issue and implemented in the linked PR is the narrowest maintainable solution because it avoids changing the existing numeric contract.
