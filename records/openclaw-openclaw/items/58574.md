## Summary

Keep open. Current main still exposes the relevant cron scheduling log/event fields as epoch milliseconds only, and the linked closing implementation is still an open, conflicting PR rather than a merged fix.

## What This Changes

The issue asks for cron `nextAt`/`nextRunAtMs` output to include a human-readable timestamp while preserving backward compatibility.

## Best Possible Solution

Land or replace a narrow additive implementation that keeps `nextAt`/`nextRunAtMs` numeric and adds range-safe ISO companion fields to cron logs/events.

## Reproduction Assessment

Yes. Source inspection on current main shows the reported cron timer/backoff logs and event/schema surfaces still expose only numeric epoch-ms fields; I did not need a live cron run to confirm the missing readable companion field.

## Solution Assessment

Yes. An additive readable companion field is the narrow maintainable direction because changing the existing numeric fields would break current log/event/schema consumers.
