## Summary

Keep open. Current main still exposes cron scheduling timestamps only as epoch milliseconds, and an open PR already closes this issue, so cleanup should not close it before that implementation lands or is replaced.

## What This Changes

Request adding a human-readable companion value for cron `nextAt`/`nextRunAtMs` timestamps while preserving the numeric timestamp contract.

## Best Possible Solution

Land or replace the narrow additive implementation that adds range-safe readable timestamp companions while preserving `nextAt` and `nextRunAtMs` as numeric fields.

## Reproduction Assessment

Yes. Source inspection on current main shows the timer log, event type, and gateway schema expose only numeric epoch-ms fields, and targeted search found no readable companion field.

## Solution Assessment

Yes. The additive companion-field direction is the narrow maintainable approach because changing `nextAt` or `nextRunAtMs` directly would break existing numeric consumers.
