## Summary

Keep open. Current `main` and the latest release still expose the relevant cron scheduling timestamps as numeric-only fields, and the open closing PR is still unmerged and conflicting rather than a shipped fix.

## What This Changes

The issue asks for a human-readable companion timestamp for cron scheduling fields such as `nextAt` and `nextRunAtMs` while preserving the numeric value.

## Best Possible Solution

Review and land a rebased, range-safe additive implementation through https://github.com/openclaw/openclaw/pull/58908 or a narrow replacement while preserving the existing numeric fields.

## Reproduction Assessment

Yes. Source inspection on current `main` shows the timer log, retry/backoff logs, `CronEvent`, and gateway schema still expose only numeric `nextAt`/`nextRunAtMs` fields.

## Solution Assessment

Yes as a direction. An additive readable ISO companion field is the maintainable solution because replacing `nextAt` or `nextRunAtMs` in place would break existing log, event, and schema consumers.
