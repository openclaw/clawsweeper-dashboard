## Summary

Keep this issue open: current main and the latest release still log the cron wake timestamp as numeric epoch milliseconds only, and an open linked closing PR already owns the implementation path even though it currently needs maintainer refresh/review.

## What This Changes

This issue requests a human-readable companion for cron `nextAt` timestamps so operators do not have to manually convert epoch milliseconds.

## Best Possible Solution

Preserve numeric `nextAt`/`nextRunAtMs` and add an additive ISO-readable companion in the cron timer logging path, using the linked PR as the first review path or replacing it with an equivalent narrow fix if needed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: arm a cron timer with debug logging and the current payload contains numeric `nextAt` without a readable companion.

## Solution Assessment

Yes. An additive readable companion field is the safest shape because replacing `nextAt` would break numeric consumers; PR review should keep any event/API expansion explicit.
