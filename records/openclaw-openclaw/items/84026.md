## Summary

Keep open: current main and the latest checked release still route Feishu `.xlsx` attachments to `im.file.create` as `file_type: "xls"`; the remaining blocker is confirming the live Feishu/Lark upload contract because the pinned SDK types still document `xls`, while the reporter says `xlsx` succeeds.

## What This Changes

The issue reports that Feishu message-tool `.xlsx` attachments fail with a 400 because OpenClaw uploads them with the wrong Feishu `file_type`.

## Best Possible Solution

Confirm Feishu's live upload contract for spreadsheets, then add a focused Feishu media regression that routes or retries `.xlsx` correctly without changing existing `.xls`, document, audio, video, or image behavior.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection does show the reported `.xlsx` attachment path currently uploads with `file_type: "xls"`, matching the suspected failure point.

## Solution Assessment

Unclear as a direct `xls` to `xlsx` switch because the reporter's successful API test conflicts with the pinned SDK's documented type union. The safer solution is a contract-confirmed mapping or fallback with focused regression coverage.
