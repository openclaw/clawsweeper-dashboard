## Summary

Keep open: current main still maps Feishu `.xlsx` attachments to `file_type: "xls"` and passes that value directly into the upload request, and the shipped v2026.5.18 tag has the same behavior. The remaining uncertainty is vendor-contract proof, because the pinned Lark SDK types still advertise `xls` rather than `xlsx` for this API.

## What This Changes

The issue reports that Feishu message-tool `.xlsx` attachments fail with a 400 invalid-request response and asks for correct upload file-type routing.

## Best Possible Solution

Implement a narrow Feishu media routing/regression fix for `.xlsx` attachments after confirming the vendor-accepted `file_type`, leaving other file-type routing unchanged.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the reported path routes `.xlsx` as `xls` and the reporter provided a matching 400 plus direct API workaround, but the SDK type contract still conflicts with the proposed `xlsx` value.

## Solution Assessment

Unclear until vendor behavior is confirmed. The maintainable fix should be a narrow routing or retry change in `extensions/feishu/src/media.ts` with focused regression coverage for `.xlsx`, not a broad media-path rewrite.
