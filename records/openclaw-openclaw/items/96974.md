## Summary

Keep open: current main and v2026.6.10 still hardcode `gemini-2.5-flash` for Gemini web_search, so there is no implemented-on-main close path; however the 404 claim needs maintainer/live API confirmation because an unauthenticated probe cannot validate the model and Google’s current model page still lists `gemini-2.5-flash` as stable with Search grounding support. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/models/gemini-2.5-flash?utm_source=openai))

## What This Changes

The issue reports that Gemini web_search defaults to a hardcoded `gemini-2.5-flash` model that now 404s unless users configure another Gemini web-search model.

## Best Possible Solution

Keep the issue open for a provider-owned maintainer decision: either prove the default really fails with a valid Gemini key and update the Google plugin default/tests/docs, or close later with live/API-contract evidence that the reported 404 is environment-specific.

## Reproduction Assessment

Unclear: current main definitely defaults Gemini web_search to `gemini-2.5-flash` and would POST that model id, but this review did not have a valid API key to reproduce the reported 404. The no-key probe returned 403 and official Google docs still list the stable model as Search-grounding-capable.

## Solution Assessment

Unclear: the fix belongs in the Google plugin boundary, but maintainers should first choose whether the durable behavior is a rolling alias, provider-model inheritance, or a plain default bump. A narrow fix should update source, tests, docs, and live API proof together without adding a runtime fallback stack.
