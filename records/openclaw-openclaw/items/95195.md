## Summary

Keep open: the code is a focused and likely correct fix for the runtime prompt OS metadata bug, but this external PR still lacks real macOS behavior proof for the changed path and overlaps several open candidate PRs for the same linked issue.

## What This Changes

The PR adds a cached runtime OS label to `src/infra/os-summary.ts`, switches CLI, embedded attempt, and compaction runtime prompt metadata to use it, and expands OS-summary tests for Darwin and non-Darwin labels.

## Best Possible Solution

Land one canonical cached macOS product-version fix for all runtime prompt OS metadata paths, explicitly accept the additive OS-summary field or keep it private, then reconcile the overlapping candidate PRs for the linked issue.

## Reproduction Assessment

Yes from source inspection: current main sends `os.type()` plus `os.release()` into the three runtime prompt paths, and the linked issue provides macOS 26.5.1 versus Darwin 25.5.0 host evidence. I did not run a live Tahoe host in this read-only review.

## Solution Assessment

Mostly yes: reusing the cached OS summary helper for runtime prompt labels is the right owner boundary and avoids the uncached `sw_vers` issue seen in sibling PRs. Maintainers should still decide whether the added `osLabel` field is acceptable on serialized status and trajectory outputs.
