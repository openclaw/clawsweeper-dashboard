## Summary

Keep open: current main still has unhandled async SIGTERM/SIGUSR1 signal callbacks, and a rejected lifecycle-runtime import would prevent the intended stop/restart request from being issued cleanly. This is a narrow gateway lifecycle bug with a clear source-level repair path.

## What This Changes

Handle rejected lifecycle-runtime imports in SIGTERM/SIGUSR1 gateway signal callbacks instead of leaving the shutdown or restart request unissued.

## Best Possible Solution

Add explicit error handling around both signal-handler async preludes, log the failure, drive a deterministic safe fallback through the run-loop request path, and lock it with focused rejection-path tests.

## Reproduction Assessment

Yes, source-reproducible. Force `loadGatewayLifecycleRuntimeModule()` to reject in the run-loop signal harness and then invoke the captured SIGTERM/SIGUSR1 listener; current main has no `.catch()` before `request(...)` is reached.

## Solution Assessment

Yes. The narrow maintainable fix is to catch and log rejected signal-handler preludes and choose a safe fallback that does not depend on the failed lifecycle import, plus regression coverage in `run-loop.test.ts`.
