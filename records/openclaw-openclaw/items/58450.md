## Summary

Keep open. Current main has useful GPT-5 plan-only and reminder-specific mitigations, and the reporter later saw improvement on 2026.4.23, but the general invariant from this issue is still not enforced for final replies that promise to “come back” or send a later follow-up without starting a real follow-up mechanism.

## What This Changes

The issue asks OpenClaw to prevent final assistant replies from promising later follow-up unless the turn actually started a cron, subagent, queued follow-up, or equivalent mechanism.

## Best Possible Solution

Add a conservative final-reply consistency guard in the reply pipeline that annotates or rewrites unbacked future-follow-up promises only when no cron, spawned session, queued task, messaging delivery, or other real follow-up mechanism was created in the same turn.

## Reproduction Assessment

Unclear for a high-confidence live repro on current main because model behavior is stochastic and the reporter saw improvement after 2026.4.23. Static source inspection does show the exact guard gap for broader promise-style final text such as “come back with one short message.”

## Solution Assessment

No, the current mitigations are not the complete best solution for this issue. The maintainable path is to extend the existing reply guard using runner metadata for real follow-up evidence, rather than relying only on prompts, commitments, or provider-specific plan-only retries.
