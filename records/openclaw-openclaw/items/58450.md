## Summary

Keep open: current main has prompt guidance, incomplete-turn recovery, a narrow cron-backed reminder disclaimer, and opt-in inferred commitments, but it still does not enforce the broader invariant that a final reply cannot promise a later follow-up unless the same turn actually started a follow-up mechanism.

## What This Changes

The issue requests a reply-pipeline guard that rewrites, rejects, or annotates user-facing future-follow-up promises unless the same turn starts a real follow-up mechanism.

## Best Possible Solution

Define one canonical reply-pipeline invariant for unbacked future-follow-up promises, then implement it across channel delivery with focused tests for cron, sessions_spawn, existing commitment extraction, and false-positive cases.

## Reproduction Assessment

Yes at source level: current main only annotates narrow reminder-like text and optionally extracts commitments after the reply, so a broader unbacked follow-up promise is not enforced by the runtime. No live Telegram/TUI reproduction was run in this read-only pass.

## Solution Assessment

No, current main is only a partial mitigation for the issue. The best solution needs maintainer direction on detection scope and whether the runtime should rewrite, annotate, or gate the outgoing reply.
