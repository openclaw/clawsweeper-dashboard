## Summary

Keep this open: current main has reminder-specific transparency, opt-in inferred commitments, and some planning-only retry guards, but those do not fully enforce the reported same-turn invariant for broader follow-up promises; the issue also has a fresh 2026.6.1 user report and related narrower issues point back here as canonical.

## What This Changes

The issue requests runtime enforcement so final replies cannot promise later follow-up work unless the same turn actually started a real follow-up mechanism.

## Best Possible Solution

Define a maintainer-approved reply-pipeline invariant for unbacked future-work language, then implement it with focused coverage across reminder notes, cron creation, sessions_spawn, message-tool delivery, and opt-in commitments.

## Reproduction Assessment

Partially: source inspection gives a high-confidence path for the remaining guard gap, because current main only enforces reminder-like promises and opt-in delayed commitments. I did not establish a live Telegram/TUI reproduction for the newest 2026.6.1 report.

## Solution Assessment

Unclear as a final design: the reply pipeline is the right enforcement area, but maintainers need to choose whether broad future-work language should be rewritten, annotated, gated, or routed through commitments.
