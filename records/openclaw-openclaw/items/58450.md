## Summary

Keep open. Current main has reminder notes, opt-in commitments, strict-agentic planning retries, and durable final-delivery retries, but none of those fully enforce the requested invariant that a visible future-follow-up promise must be backed by same-turn follow-up evidence or rewritten before delivery.

## What This Changes

The issue asks OpenClaw to prevent final assistant replies from promising later follow-up work unless the turn actually started a background, scheduled, spawned, queued, or similar follow-up mechanism.

## Best Possible Solution

Define a conservative reply-pipeline consistency contract that recognizes concrete future-follow-up promises and validates same-turn evidence such as cron creation, accepted session spawn, queued follow-up work, durable background task creation, or explicit visible send before annotating, rewriting, blocking, or allowing the reply.

## Reproduction Assessment

No, we do not have a high-confidence live current-main reproduction. Source inspection shows that final replies with broader “come back later” language can bypass the existing reminder-only guard and are not checked against generic same-turn follow-up evidence.

## Solution Assessment

Mostly yes: a runtime consistency guard is the right layer, but maintainers need to decide the phrase scope, evidence contract, and whether the system should annotate, rewrite, block, or expose stricter behavior through configuration.
