## Summary

Keep this open: current main has a reminder/cron transparency guard and opt-in inferred commitments, but those do not fully enforce the reported same-turn invariant for broader follow-up promises, and the latest discussion includes a fresh 2026.6.1 reproduction report. This remains the canonical tracker because the related narrower issues were closed as implemented or duplicate while pointing back here for the product decision.

## What This Changes

The issue asks OpenClaw to prevent or annotate user-facing follow-up promises when no real follow-up, background action, or delayed delivery mechanism was started in the same turn.

## Best Possible Solution

Define a maintainer-approved reply-pipeline invariant for unbacked future-work language, then implement it with focused coverage across reminder, cron, sessions_spawn, message tool delivery, and opt-in commitments.

## Reproduction Assessment

Yes, at source level for the remaining gap: current main only guards reminder-like promises backed by cron evidence, and opt-in commitments are not default same-turn enforcement. I did not run a live Telegram/TUI reproduction for the newest report.

## Solution Assessment

Unclear as a final design: the reply pipeline is the right enforcement area, but maintainers need to choose whether broad future-work language should be rewritten, annotated, gated, or routed through commitments.
