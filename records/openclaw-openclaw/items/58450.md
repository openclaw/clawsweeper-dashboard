## Summary

Keep open: current main and the latest release still only provide an append-only reminder/cron note plus opt-in delayed commitments, while the broader invariant that promised future work must be backed by an actual same-turn mechanism remains unresolved and has recent user discussion.

## What This Changes

The issue asks for runtime enforcement that prevents or annotates assistant replies that promise later follow-up work when no concrete follow-up mechanism was started in the same turn.

## Best Possible Solution

Define a maintainer-approved reply-pipeline invariant for unbacked future-work language, then implement it with focused coverage across reminder, cron, sessions_spawn, queued follow-up, message delivery, and opt-in commitments.

## Reproduction Assessment

Yes, at source level: a final reply such as “I'll read memory directly and come back” is not covered by the current reminder regex or default commitments. I did not run a live Telegram/TUI reproduction for the newest report in this read-only review.

## Solution Assessment

Unclear as a final design: the reply pipeline is the right enforcement area, but maintainers need to choose rewrite, annotation, gating, or commitment routing policy before a durable fix.
