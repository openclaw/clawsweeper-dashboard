## Summary

Keep open: current main still only annotates reminder-like unbacked cron promises, while opt-in commitments are post-reply and heartbeat-delivered, so the broader same-turn follow-up invariant remains unresolved and recent discussion reports similar behavior on OpenClaw 2026.6.1.

## What This Changes

Request a reply-pipeline guard so user-visible follow-up promises are backed by an actual background action or are rewritten or annotated before delivery.

## Best Possible Solution

Define a maintainer-approved reply-pipeline invariant for unbacked future-work language, then implement it with focused coverage across reminder, cron, sessions_spawn, message delivery, and opt-in commitments.

## Reproduction Assessment

Yes, at source level: a final reply such as “I'll read memory directly and come back” is not covered by the current reminder regex or by default commitments. I did not run a live Telegram/TUI reproduction for the newest report.

## Solution Assessment

Unclear as a final design: the reply pipeline is the right enforcement area, but maintainers need to choose rewrite, annotation, gating, or commitment routing policy before a durable fix.
