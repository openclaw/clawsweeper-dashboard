## Summary

Keep open. Current main has several useful mitigations for the broader plan-only/reply-loop regression, and the reporter later saw improvement on 2026.4.23, but the source still does not fully enforce the requested invariant for general “I’ll come back / send a follow-up” final replies when no follow-up mechanism was started.

## What This Changes

The issue requests a runtime reply-pipeline guard that prevents final replies from promising later follow-up unless the same turn created a real follow-up mechanism.

## Best Possible Solution

Add a conservative final-reply consistency guard that annotates or rewrites unbacked future-follow-up promises only when no cron, subagent, task, commitment, or other real follow-up mechanism was created.

## Reproduction Assessment

Unclear for a deterministic live repro on current main because model behavior is stochastic and 2026.4.23 improved the related regression. Static inspection does show a high-confidence guard gap for the exact class of broader promise-style final text described here.

## Solution Assessment

No, the current partial mitigation is not the best complete solution for this issue. The maintainable fix should build on the existing reply guard and runner metadata rather than relying on prompts, opt-in commitments, or only the plan-only retry path.

## Repair Work Prompt

Implement a conservative runtime guard for unbacked future follow-up promises in final assistant replies. Current main only has the reminder-specific guard in `src/auto-reply/reply/agent-runner-reminder-guard.ts`; broaden or add a sibling helper so final text like “I'll come back with one short message” or “I'll look into it and send a short follow-up” is annotated or rewritten unless the same turn actually created a real follow-up mechanism. Preserve existing behavior for successful `cron.add` and existing session cron coverage, and do not annotate when a real background mechanism such as `sessions_spawn`, an ACP/subagent task, a queued completion wake, or another explicit follow-up path was started. Thread minimal runner metadata if needed; do not add core owner-specific IDs or broad prompt-only fixes. Add focused Vitest coverage for the issue’s memory-search/one-tool-then-promise shape, the existing reminder cases, and false positives such as answer summaries or restart/status copy. Add a CHANGELOG entry because this changes user-visible reply behavior.
