## Summary

Keep open: current main still has only a narrow append-only reminder guard plus opt-in, post-reply inferred commitments; it does not enforce the broader invariant that a final reply cannot promise later follow-up unless the turn actually started cron, sessions_spawn, queued work, or another backed mechanism.

## What This Changes

The issue requests runtime enforcement for assistant follow-up promises so unbacked “I’ll come back later” style replies are rewritten, rejected, or backed by a real follow-up action.

## Best Possible Solution

Define one canonical reply-pipeline invariant for unbacked future-follow-up promises, then implement it with focused tests across cron, sessions_spawn/subagent, commitment extraction, message-tool progress replies, and false-positive cases.

## Reproduction Assessment

Yes at source level, but not as a live current-main Telegram/TUI repro. Current main only annotates narrow reminder-like text and optionally extracts commitments after the reply, so the broader unbacked follow-up promise remains unenforced.

## Solution Assessment

No, current main is only a partial mitigation. The best fix needs a maintainer decision on detection scope and whether the runtime should rewrite, annotate, block, or force scheduling before delivery.
