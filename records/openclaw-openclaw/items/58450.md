## Summary

Keep open. Current main still has a narrow reminder/cron disclaimer and strict-agentic planning-only recovery, but not the broader reply-pipeline invariant that a final reply promising later follow-up must be backed by same-turn follow-up evidence.

## What This Changes

The issue requests a runtime guard that prevents or annotates final replies promising later follow-up unless the turn actually created a follow-up mechanism.

## Best Possible Solution

Define a conservative reply-pipeline consistency guard that uses existing same-turn evidence for cron creation, spawned sessions, queued work, messaging side effects, and other durable follow-up mechanisms before annotating or rewriting concrete future-follow-up promises.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. Source inspection and a focused regex smoke show the reported 'I'll come back' style final replies still bypass the existing reminder-only guard.

## Solution Assessment

Mostly yes: a narrow runtime consistency guard is the right direction, but maintainers should decide the exact phrase scope and whether to annotate, rewrite, gate, or rely on an opt-in strict mode before implementation.
