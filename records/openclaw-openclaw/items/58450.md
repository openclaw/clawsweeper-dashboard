## Summary

Keep this open: current main and the latest release still only cover reminder-style transparency notes plus opt-in delayed commitments, not a general same-turn invariant that backs or rewrites broader follow-up promises; related narrower issues point back here as the canonical product/runtime decision.

## What This Changes

The issue asks OpenClaw to prevent or rewrite assistant replies that promise a later follow-up when no real follow-up mechanism was started.

## Best Possible Solution

Define a maintainer-approved reply-pipeline invariant for unbacked future-work language, then implement it with focused coverage across reminder, cron, sessions_spawn/sessions_yield, message delivery, goals, and opt-in commitments.

## Reproduction Assessment

Yes, at source level: a non-reminder promise such as checking files and coming back later does not match the narrow reminder guard, and commitments are disabled/post-reply rather than a same-turn enforcement path. I did not run a live Telegram or TUI reproduction for the newest 2026.6.1 report.

## Solution Assessment

Unclear as a final design: the reply pipeline is the right enforcement layer, but maintainers still need to choose whether broad future-work language is rewritten, annotated, gated, converted into a commitment, or left to explicit cron/subagent mechanisms.
