## Summary

Keep open. Current main still has only narrow reminder/cron and planning-only mitigations; it does not enforce the broader invariant that a final reply promising later follow-up must be backed by a real same-turn follow-up mechanism.

## What This Changes

The issue requests a runtime guard that prevents final replies from promising later follow-up unless a real same-turn follow-up mechanism was started.

## Best Possible Solution

Add a conservative final-reply consistency guard in the reply pipeline that covers concrete future-follow-up promises beyond reminders and only annotates or rewrites them when the run has no cron, spawned session, queued task, message delivery, or other real same-turn follow-up evidence.

## Reproduction Assessment

No high-confidence live reproduction was established on current main. Source inspection does show a concrete gap: broader promise-style final replies can bypass the current reminder-only guard and the partial planning-only retry path.

## Solution Assessment

Yes, a narrow runtime consistency guard is the right direction. It should reuse existing run evidence and keep matching conservative, rather than adding a broad config gate or relying on prompts alone.
