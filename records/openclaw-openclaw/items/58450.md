## Summary

Keep open. Current main has overlapping work for reminder-like promises and opt-in inferred commitments, but it does not yet enforce the general “I’ll do more work and come back” promise shape described here, especially with commitments disabled by default.

## What This Changes

The issue asks for a runtime reply-pipeline guard that prevents user-visible promises of later follow-up unless the same turn actually started a background, scheduled, or queued follow-up mechanism.

## Best Possible Solution

Add a default auto-reply post-processing guard for unbacked follow-up promises, sharing the existing reminder-guard style but keyed to recorded same-turn mechanisms such as cron scheduling, subagent/background task creation, or a successfully queued commitment. The guard should annotate or rewrite only the unsafe promise text and preserve normal progress replies, actual tool work, and explicit user-requested scheduling flows.

## Reproduction Assessment

Yes. A focused unit reproduction can use the existing runReplyAgent harness with a mocked final payload like “I need to read the memory files directly and then I’ll come back,” successfulCronAdds set to 0, and commitments disabled; current main will deliver that text without a guard note or follow-up mechanism.

## Solution Assessment

No. The current opt-in commitment extractor and reminder-only note are useful partial defenses, but they are not the best fix for this report because they do not provide default same-turn enforcement for general follow-up promises.
