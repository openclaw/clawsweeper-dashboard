## Summary

Keep open: current main and the latest release still only enforce a narrow cron-backed reminder note plus opt-in post-reply commitments, while the broader unbacked future-follow-up invariant remains unresolved and has a fresh 2026.6.1 user report.

## What This Changes

The issue requests a core reply-pipeline guard so agents cannot make user-visible future-follow-up promises unless the same turn starts a real follow-up mechanism or the response is rewritten/annotated.

## Best Possible Solution

Define a maintainer-approved core reply-pipeline invariant for future-work language, then enforce it with focused coverage for cron, sessions_spawn/subagents, queued follow-ups, message delivery, and opt-in commitments.

## Reproduction Assessment

Yes at source level: a final reply like “I'll read memory directly and come back” is not covered by the current reminder regex, cron evidence check, or default commitment configuration. I did not run a live Telegram/TUI reproduction in this read-only review.

## Solution Assessment

Unclear as a final design: the reply pipeline is the right enforcement area, but maintainers need to choose whether unresolved future-work language should be rewritten, annotated, blocked, or routed into an explicit follow-up mechanism.
