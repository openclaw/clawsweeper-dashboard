## What This Changes

The PR changes device pairing so a subset reconnect refreshes one already-covering pending request in place while broader-scope escalations still mint a replacement request id.

## Best Possible Solution

## Solution Assessment

Yes, this appears to be the narrow owner-boundary fix: it changes device pending-request reconciliation, preserves broader existing scopes, and leaves true escalations on the replacement path. A broader CLI fallback would be a separate product/security decision tracked by the canonical issue.
