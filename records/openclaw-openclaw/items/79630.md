## Summary

Keep open: current main still has the reported lifecycle mismatch between the loop-hook checkpoint and the attempt finalizer, and the fix is narrow enough to route as a focused bug repair.

## What This Changes

The issue requests that `ownsCompaction: true` context engines not receive finalizer `afterTurn` deltas that were already emitted by the per-iteration loop hook.

## Best Possible Solution

Thread the loop hook's latest delivered message checkpoint into finalization and add regression coverage proving the final `afterTurn` delta excludes messages already delivered mid-loop.

## Reproduction Assessment

Yes, from source inspection: an `ownsCompaction` engine can receive loop-level `afterTurn` calls fenced by `lastSeenLength`, then a finalizer call fenced by the original attempt `prePromptMessageCount`. I did not execute a live model/tool loop during this read-only review.

## Solution Assessment

Yes: synchronizing the finalizer checkpoint with the loop hook is the narrow maintainable fix. Adding public lifecycle metadata or stable message IDs would be broader contract work and is not needed for this bug.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79630. Current `main` lets `installContextEngineLoopHook` deliver loop-level `afterTurn` deltas using its private `lastSeenLength`, then `finalizeHarnessContextEngineTurn` reuses the original attempt `prePromptMessageCount`, so persistent `ownsCompaction: true` context engines can see duplicate deltas. Keep the fix generic and private to the runtime: thread the hook's latest delivered checkpoint back to the attempt finalizer, or otherwise pass a finalizer fence that is at least the latest loop-delivered message count. Do not add a public `ContextEngine` API, config option, or content-deduplication policy unless the existing private runtime path cannot support the invariant. Add focused regression coverage that simulates loop-hook delivery followed by finalization and asserts the final `afterTurn` delta excludes already-delivered loop messages. Add a user-facing changelog fix entry. Validate the touched tests and formatting; use Testbox if the changed gate fans out broadly.
