## Summary

Keep open for maintainer review: the patch appears to be the right narrow verifier fix with sufficient real CLI proof, but it intentionally widens a backup archive integrity boundary and needs explicit owner sign-off before merge.

## What This Changes

The branch changes `backup verify` to resolve legacy cwd-relative tar hardlink targets by walking ancestor paths and adds verifier regression tests for valid and corrupt hardlink shapes.

## Best Possible Solution

Land the narrowed verifier compatibility fix after backup/security owner approval, while keeping the create-time truncated archive failure tracked separately.

## Reproduction Assessment

Yes. Source inspection shows current main only checks root-prefixed exact targets, node-tar v7.5.15 writes cwd-relative hardlink linkpaths, and the PR body plus linked issue comments include before/after CLI output for the affected archive shape.

## Solution Assessment

Yes, this appears to be the best narrow verifier fix: resolve the legacy target from the link entry's ancestor chain and require a regular File entry. Changing backup creation would not repair older archives, and a loose suffix scan would weaken integrity checks.
