## Summary

Keep open for maintainer review: current main still only handles the merged root-relative hardlink case, while the PR addresses the node-tar cwd-relative hardlink shape with focused regular-file target checks and strong regression coverage. The remaining blocker is not a code defect I can point to, but explicit owner sign-off on widening a backup archive integrity boundary; the separate truncated temp archive failure in https://github.com/openclaw/openclaw/issues/89257 remains out of scope.

## What This Changes

The branch updates backup verification to resolve legacy cwd-relative tar hardlink targets by walking the link entry ancestor paths and adds regression tests for real tar-created hardlinks, missing/tail-collision, self/link-chain, and non-file targets.

## Best Possible Solution

Land this narrow verifier fix after maintainer sign-off on the hardlink compatibility boundary, while leaving the create-time truncated archive failure tracked separately.

## Reproduction Assessment

Yes, source-reproducible: current main only checks `<root>/<linkpath>` for a cwd-relative `.openclaw/...` target, while tar 7.5.15 computes hardlink linkpaths relative to `cwd`. I did not run tests because this review is read-only.

## Solution Assessment

Yes, this looks like the best narrow fix: ancestor walking matches the dependency contract and the regular-File target set avoids the earlier self/link/non-file over-acceptance cases. The remaining question is maintainer approval of the verifier boundary, not an alternative implementation layer.
