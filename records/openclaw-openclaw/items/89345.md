## Summary

Keep open: the PR addresses a real backup verifier compatibility gap for legacy node-tar hardlink archives, but the current suffix-match implementation relaxes hardlink target validation too far and can let a corrupt link-only target verify as OK.

## What This Changes

The PR changes `backup verify` to accept legacy hardlink linkpaths by exact or suffix entry match and adds regression tests for cwd-relative node-tar hardlink archives.

## Best Possible Solution

Land a narrow verifier compatibility fix that accepts real legacy cwd-relative node-tar hardlink targets only when they resolve to an actual non-Link payload entry, while keeping the linked backup-create temp-archive failure tracked separately.

## Reproduction Assessment

Yes, source-reproducible: node-tar v7.5.15 writes hardlink linkpaths as cwd-relative paths, and current main rejects those when they are not under the manifest archive root. I did not run a local repro because this review kept the checkout read-only.

## Solution Assessment

No: the compatibility direction is right, but accepting suffix matches over all entries is too broad for a verifier. The better fix is a constrained resolver that maps legacy linkpaths to actual non-Link archive payload entries and tests corrupt/self-link cases.
