## Summary

Keep this PR open: the central verifier hardlink bug is still present on current main, and the current patch looks like a bounded fix with no remaining line-level correctness finding after the final hardening commit. The remaining blocker is maintainer approval of the backup archive integrity boundary because the verifier intentionally accepts an additional legacy hardlink target shape.

## What This Changes

The PR changes `backup verify` to resolve legacy tar hardlink targets by walking the link entry's archive ancestor paths and adds regression coverage for cwd-relative, root-relative, missing, self-link, link-to-link, and non-file target cases.

## Best Possible Solution

Land the bounded verifier compatibility fix after maintainer approval of the hardlink integrity boundary, while keeping the separate backup-create failure tracked in the open related issue.

## Reproduction Assessment

Yes, from source inspection and the PR's real-archive test shape: current main prepends only the archive root to `.openclaw/state/...`, while node-tar records hardlink targets relative to `cwd`. I did not run the tests because this review is read-only, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, this appears to be the best narrow fix for the verifier half: resolving from the link entry's ancestors matches node-tar's cwd-relative linkpath behavior and avoids a loose suffix scan. The safer alternative, keeping only root-relative resolution, leaves the reported legacy archive shape broken.
