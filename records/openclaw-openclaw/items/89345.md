## Summary

Keep open: this PR still addresses a real verifier gap on current main and the latest release, and the patch appears correct, but it needs maintainer sign-off because it intentionally broadens a backup archive integrity boundary.

## What This Changes

The branch changes `backup verify` to resolve legacy cwd-relative tar hardlink targets by walking the link entry ancestors against regular `File` entries and adds regression coverage for valid and corrupt hardlink shapes.

## Best Possible Solution

Land the narrowed verifier compatibility fix after backup/security owner approval, while keeping the create-time truncated archive failure tracked separately.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` only check root-prefixed exact targets, node-tar writes cwd-relative hardlink linkpaths, and the PR body plus linked issue comments include before/after CLI output for the affected archive shape.

## Solution Assessment

Yes, this appears to be the best narrow verifier fix: resolve the legacy target from the link entry's ancestor chain and require a regular `File` entry. Changing backup creation would not repair older archives, and a loose suffix scan would weaken integrity checks.
