The PR changes `backup verify` to resolve legacy tar hardlink targets by walking the link entry's ancestor directories and adds verifier regressions for valid and corrupt hardlink shapes.

## Best Possible Solution

Land the narrowed verifier compatibility fix after backup/security owner approval, while keeping the create-time truncated archive failure tracked separately in https://github.com/openclaw/openclaw/issues/89257.

## Reproduction Assessment

Yes for the verifier half: current main's source only checks exact root-prefixed/root-relative targets, node-tar writes cwd-relative hardlink `linkpath` values, and the PR body plus linked issue comments provide before/after CLI output. ## Solution Assessment

Yes, this appears to be the best narrow verifier fix: resolve the legacy target from the link entry's ancestor chain and require a regular File entry. Changing backup creation would not repair older archives, and a loose suffix scan would weaken integrity checks.
