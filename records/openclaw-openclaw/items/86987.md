## Summary

Keep open: current main and the latest release still leave the Web UI nodes view showing only effective caps/commands, while the backend and CLI now expose pending approval/reapproval diagnostics that explain the empty effective surface.

## What This Changes

The issue reports a Windows/Docker gateway regression where the Web UI shows paired, connected nodes with empty Caps/Commands and node commands cannot be executed.

## Best Possible Solution

Keep the shipped approval boundary, then add a narrow Control UI nodes-view diagnostic for approvalState, pendingRequestId, pending declared caps/commands, and the approve/reapprove remediation.

## Reproduction Assessment

Yes for the remaining Web UI gap from source: node.list can expose pending approval fields while effective caps/commands stay empty, and the Control UI render path ignores those fields. No live Windows/Docker repro was run for post-approval command failures.

## Solution Assessment

No, closing this as implemented would be premature because the backend/CLI fix is only partial for the reported Web UI symptom. The narrow maintainable path is to render existing gateway diagnostics in the Control UI without weakening node approval policy.
