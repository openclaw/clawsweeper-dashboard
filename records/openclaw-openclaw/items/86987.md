## Summary

Keep open: PR 92547 fixed and shipped the Gateway/CLI pending approval diagnostics, but the Control UI nodes view still ignores those fields and can still present a connected node as empty Caps/Commands without the needed approval context.

## What This Changes

The issue reports a Windows node plus Docker/Synology gateway regression where the Gateway Web UI shows connected nodes with empty Caps and no runnable commands after 2026.5.18+ upgrades.

## Best Possible Solution

Keep the backend approval boundary from PR 92547, and add a narrow Control UI nodes-view display for approvalState, pendingRequestId, pending declared caps/commands, and the appropriate approve command when the gateway exposes them.

## Reproduction Assessment

Yes for the remaining Web UI gap from source: current main exposes pending approval fields in node.list, while ui/src/ui/views/nodes.ts still renders only effective caps/commands and status chips. No live Windows/Docker repro was run, so post-approval command failures remain unproven here.

## Solution Assessment

No, the current shipped backend/CLI fix is only a partial solution for this issue's Web UI wording. The best next step is a narrow Control UI diagnostic display that uses the existing gated fields rather than changing node approval policy.
