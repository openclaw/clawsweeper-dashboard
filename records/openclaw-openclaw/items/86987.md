## Summary

Keep open: current main and the latest release still leave the Control UI nodes view showing only effective caps/commands, while the gateway and CLI expose pending approval/reapproval diagnostics that explain why the effective surface is empty.

## What This Changes

The issue reports a Windows/Docker gateway regression where paired, connected nodes show empty Caps in the Web UI and cannot run node commands after gateway 5.18+.

## Best Possible Solution

Render the existing node approval state, pending request ID, pending declared caps/commands, and approve/reapprove remediation in the Control UI without making pending commands effective; track any post-reapproval Windows execution failure separately.

## Reproduction Assessment

Yes for the remaining Web UI gap from source: node.list can expose pending approval fields while effective caps/commands stay empty, and the Control UI render path ignores those fields. No live Windows/Docker repro was run for post-reapproval command failures.

## Solution Assessment

No, closing this as implemented would be premature because the backend/CLI diagnostics do not resolve the reported Web UI symptom. The narrow maintainable fix is to render existing gateway diagnostics in the Control UI while preserving the approval boundary.
