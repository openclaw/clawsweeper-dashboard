## Summary

Keep open. Current `main` still fires `message:received` and `message:sent` internal hooks without draining messages pushed into `event.messages`, while open closing PRs remain the active fix path for this bug.

## What This Changes

The issue reports that internal hook replies pushed from `message:received` or `message:sent` should be delivered to the user instead of being dropped.

## Best Possible Solution

Land one reconciled closing PR that drains hook messages only on safe reply-capable surfaces, preserves send policy and loop suppression, and updates docs plus regression tests before closing this tracker.

## Reproduction Assessment

Yes, at source level. Registering a `message:received` or `message:sent` internal hook that pushes text into `event.messages` exercises current-main paths that emit the hook but never retain or drain the event afterward.

## Solution Assessment

Yes, keeping this open is the right current outcome because open closing PRs already target the fix. The narrow maintainable solution is to drain messages only where OpenClaw has a safe reply target and to make the documented hook contract explicit.
