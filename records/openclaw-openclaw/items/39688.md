## Summary

Current main and the latest release still fire the internal `message:received` hook without draining `event.messages`, and an open linked implementation PR is the active path. Keep this issue open until https://github.com/openclaw/openclaw/pull/41375 is refreshed, landed, or replaced.

## What This Changes

The issue reports that internal hook handlers for `message:received` and `message:sent` can push replies into `event.messages`, but those pushed replies are not sent back to the user.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent policy-aware fix that drains `message:received` hook replies through existing reply routing, while keeping `message:sent` non-replying unless maintainers approve a loop-safe contract.

## Reproduction Assessment

Yes, at source level. Current main creates and triggers a mutable `message:received` internal hook event, but the normal inbound dispatch path never drains that event's `messages` array into delivery.

## Solution Assessment

No for the broad `message:sent` auto-reply request. The best narrow fix is `message:received` delivery through the existing policy-aware reply route, with `message:sent` left observation-only unless maintainers approve a separate contract.
