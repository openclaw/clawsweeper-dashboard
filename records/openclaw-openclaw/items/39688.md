## Summary

Keep open. Current main and v2026.5.20 still fire internal message hooks without retaining or draining hook-added `event.messages`, and the open closing PR remains unmerged, conflicting, and under blocker review.

## What This Changes

The issue asks for content pushed by `message:received` and `message:sent` internal hooks to be delivered back to the user.

## Best Possible Solution

Land a current-main fix that drains `message:received` hook replies through normal reply-target and send-policy checks, while documenting lifecycle-only hooks like `message:sent` as non-replying unless maintainers explicitly expand that contract.

## Reproduction Assessment

Yes. Source inspection on current main shows `message:received` and `message:sent` hooks fire with inline events, but only command hooks retain and drain `hookEvent.messages`.

## Solution Assessment

Not yet. The linked PR is the right direction, but it still needs sender-target routing, send-policy enforcement, malformed-message hardening, and a final replyable-surface contract decision.
