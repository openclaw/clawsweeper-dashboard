## Summary

Keep this issue open: current main and the latest release still fire the `message:received` internal hook without draining `event.messages`, and an open maintainer PR is already the canonical fix candidate. The `message:sent` part should remain a product/API boundary question because current docs explicitly treat it as lifecycle-only, not replyable.

## What This Changes

The issue reports that internal hook replies stored in `event.messages` for `message:received` and `message:sent` are triggered but not sent back to the user, observed on OpenClaw 2026.2.26 with Feishu.

## Best Possible Solution

Land or replace the open maintainer PR with a current-main fix that drains `message:received` hook replies on replyable surfaces while preserving the documented `message:sent` no-reply boundary.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main triggers the `message:received` internal hook asynchronously and does not retain the event for draining pushed `event.messages`. I did not run a live Feishu or channel harness reproduction in this read-only review.

## Solution Assessment

No, the broad requested behavior is not the best fix as written because current docs make `message:sent` lifecycle-only. The best fix is the narrower replyable-surface path for `message:received`, with routing and send-policy safeguards.
