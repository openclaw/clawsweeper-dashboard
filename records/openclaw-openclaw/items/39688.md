## Summary

Keep open: current main still triggers internal `message:received`/`message:sent` hooks without draining `event.messages`, while the docs and hook type contract describe those messages as user-deliverable. Open PRs #40314 and #41375 explicitly close this issue, so the issue should stay open until one of those fixes or a successor lands.

## What This Changes

Internal `message:received` and `message:sent` hooks should deliver messages pushed to `event.messages` back to the user.

## Best Possible Solution

Land #41375 or a successor that drains hook replies on reply-capable inbound surfaces, preserves loop protection, and clarifies lifecycle-only hook semantics before closing this issue.

## Reproduction Assessment

Yes, source-reproducible. A `message:received` hook can push to `event.messages`, but current main triggers that hook fire-and-forget and never drains the event, unlike the command hook path that routes pushed messages.

## Solution Assessment

Yes for the linked conservative direction: route accumulated hook messages only when there is a reply-capable inbound target, and avoid automatic `message:sent` reply delivery unless maintainers explicitly define loop-safe semantics.
