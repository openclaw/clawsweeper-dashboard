## Summary

Keep open: current main and the latest release still fire `message:received` internal hooks without retaining or draining `event.messages`, while current docs describe `message:received` as replyable. The open maintainer PR https://github.com/openclaw/openclaw/pull/41375 already uses closing syntax for this issue and should remain the canonical fix path rather than opening another repair lane.

## What This Changes

The issue reports that internal hook `event.messages` entries from `message:received` and `message:sent` are not delivered to users, with Feishu as the reported channel.

## Best Possible Solution

Rebase and repair the canonical PR, then land `message:received` hook reply draining only on replyable surfaces with sender targeting, send-policy gating, hook-message validation, docs, and regression coverage aligned; keep `message:sent` lifecycle-only unless maintainers explicitly expand that contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: register a `message:received` internal hook that pushes text into `event.messages`; current main fires the hook without retaining or draining that array, so nothing is routed back.

## Solution Assessment

Yes for the maintainer-owned direction, but not yet for the current open PR state. The best solution is to repair and land the existing canonical PR because it matches the documented replyable-surface contract while excluding lifecycle-only `message:sent` auto-replies.
