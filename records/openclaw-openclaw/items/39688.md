## Summary

Keep open: current `main` and the latest release still fire `message:received` internal hooks without draining `event.messages`, and the same-repo closing fix candidate is still open with maintainer-review and safety blockers.

## What This Changes

The issue reports that internal hook replies accumulated in `event.messages` for `message:received` and `message:sent` are triggered but not sent back to the user, observed on Feishu in OpenClaw 2026.2.26.

## Best Possible Solution

Review, repair, and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent maintainer-approved patch that drains `message:received` hook replies on reply-capable surfaces while keeping `message:sent` lifecycle-only unless maintainers explicitly change that contract.

## Reproduction Assessment

Yes at source level: register an internal `message:received` hook that pushes to `event.messages`; current `main` fires the hook without retaining the event object or draining the messages afterward. I did not run a live Feishu/WebChat/Telegram reproduction because this review is read-only.

## Solution Assessment

Yes for keeping this issue open behind the active fix path. The best implementation should be the linked maintainer PR or an equivalent patch for `message:received`; extending `message:sent` auto-replies is a separate product/API decision.
