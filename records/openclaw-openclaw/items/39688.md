## Summary

Keep this issue open: current `main` still fires `message:received` internal hooks without draining `event.messages`, and the active fix is the open linked PR https://github.com/openclaw/openclaw/pull/41375, which still needs rebase and review fixes before merge.

## What This Changes

The issue requests that content pushed into internal hook `event.messages` for `message:received` and `message:sent` be delivered back to the user.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/41375 with a current-main patch that drains `message:received` hook replies through existing safe routing and send-policy decisions, while keeping `message:sent` lifecycle-only unless maintainers explicitly change that contract.

## Reproduction Assessment

Yes from source. Current `main` fires `message:received` internal hooks via `fireAndForgetHook` and never reads the resulting `event.messages`, while the command hook path does drain and route that array; I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No for closing now. The narrow `message:received` drain is the right direction, but the active PR still needs rebase plus routing and send-policy fixes; `message:sent` auto-replies should not be added without a maintainer-approved contract change.
