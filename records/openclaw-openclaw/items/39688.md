## Summary

Keep open. Current `main` still emits the internal `message:received` and `message:sent` hooks without delivering messages pushed into `event.messages`, and an open maintainer-labeled PR remains the active closing path.

## What This Changes

The issue reports that internal hook replies pushed during `message:received` or `message:sent` are not sent back to the user.

## Best Possible Solution

Land `https://github.com/openclaw/openclaw/pull/41375` or an equivalent patch that drains `message:received` hook replies only when OpenClaw has a safe reply target, and updates the docs to define which hook surfaces auto-deliver `event.messages`.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to register a `message:received` internal hook that pushes into `event.messages` and send an inbound routed message; current `main` fire-and-forgets the event and never drains the messages array.

## Solution Assessment

Yes. The best current path is the active PR or an equivalent narrow fix, because it routes hook replies only where a safe reply target exists instead of making every lifecycle hook auto-reply.
