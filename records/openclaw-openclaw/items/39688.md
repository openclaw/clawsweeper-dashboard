- `impact:message-loss`: Hook handler output placed in `event.messages` can be silently dropped instead of delivered to users.

## Summary

Keep open. Current `main` and the latest release still fire the reported internal hooks without draining hook-added `event.messages`, and the open closing PR at https://github.com/openclaw/openclaw/pull/41375 remains the active implementation path.

## What This Changes

The issue requests that content pushed into `event.messages` by internal `message:received` and `message:sent` hooks be sent back to users instead of silently dropped.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/41375 or an equivalent maintainer-approved patch that drains hook replies only on safe reply-capable paths and aligns docs and regression tests with that contract.

## Reproduction Assessment

Yes, at source level. A `message:received` or `message:sent` internal hook that pushes text into `event.messages` reaches current-main hook handlers, but the call sites never retain or drain the mutated event object.

## Solution Assessment

Yes. Keeping this issue open is the right current path because an open closing PR already owns the implementation, and the final fix needs the maintained reply-surface boundary rather than unconditional lifecycle-hook auto-replies.
