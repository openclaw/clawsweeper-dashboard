## Summary

Keep open. Current main still drops messages added by internal message hooks, and open closing PRs remain active implementation candidates, so this issue should stay as the canonical tracker until one lands.

## What This Changes

The issue reports that internal `message:received` and `message:sent` hooks can populate `event.messages`, but those messages are not sent back to the user.

## Best Possible Solution

Land one maintained closing PR, such as https://github.com/openclaw/openclaw/pull/41375 or the repaired contributor path at https://github.com/openclaw/openclaw/pull/40314, with docs defining exactly which hook surfaces auto-deliver `event.messages`.

## Reproduction Assessment

Yes, at source level. Registering a `message:received` or `message:sent` internal hook that pushes into `event.messages` exercises current-main paths that emit the hook but never drain those messages into delivery.

## Solution Assessment

Yes. The narrow maintainable solution is to await and drain hook messages only where OpenClaw has a safe reply target, then align the docs and regression tests with that contract.
