## Summary

Keep open. Current main still emits `message:received` and `message:sent` internal hooks without draining `event.messages`, and an open maintainer-labeled PR already closes this issue as the active implementation path.

## What This Changes

The issue reports that internal hook replies pushed into `event.messages` from `message:received` and `message:sent` are not delivered back to users.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent maintainer-approved patch that drains hook replies only on safe reply-capable surfaces and updates docs/tests to match that contract.

## Reproduction Assessment

Yes, at source level. Registering a `message:received` or `message:sent` internal hook that pushes text into `event.messages` exercises current-main paths that emit the hook but never drain those messages into delivery.

## Solution Assessment

Yes for keeping this issue open. The narrow maintainable path is to review and land the active maintainer PR or equivalent fix rather than closing the canonical tracker before the implementation merges.
