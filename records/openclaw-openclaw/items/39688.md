## Summary

Keep open: current main and the latest release still fire `message:received` and `message:sent` internal hooks without draining `event.messages`, and the active same-repo fix path is the open maintainer PR that explicitly closes this issue.

## What This Changes

The issue reports that internal hook replies pushed into `event.messages` for `message:received` and `message:sent` are not sent back to the user.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/41375 with a current-main fix that drains `message:received` hook replies through existing reply routing and send-policy checks, while keeping `message:sent` lifecycle-only unless maintainers explicitly expand that contract.

## Reproduction Assessment

Yes, at source level. Current main fires `message:received` internal hooks without retaining the event, so handler-pushed `event.messages` are dropped; `message:sent` also fires without a drain but is currently documented as lifecycle-only.

## Solution Assessment

Yes for the `message:received` portion if the linked PR is repaired and merged. The `message:sent` portion should not be folded in without maintainer confirmation because current docs explicitly exclude lifecycle-only events from automatic reply delivery.
