## Summary

Keep this open: current main and the latest release still fire `message:received` internal hooks without draining `event.messages`, and the linked open implementation PR is the active repair path but is conflicting and has review-raised policy/routing blockers.

## What This Changes

The issue reports that internal hook replies pushed into `event.messages` for `message:received` and `message:sent` are not sent back to users, observed on Feishu in OpenClaw 2026.2.26.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent current-main fix that drains `message:received` hook replies through the policy-aware reply route while leaving `message:sent` observation-only unless maintainers approve a separate contract.

## Reproduction Assessment

Yes, at source level. Current main creates and triggers a mutable `message:received` internal hook event, but the normal inbound dispatch path never drains that event's `messages` array into delivery.

## Solution Assessment

No for the broad `message:sent` auto-reply request. The best narrow fix is `message:received` delivery through the existing policy-aware reply route, with `message:sent` left observation-only unless maintainers approve a separate contract.
