- `impact:message-loss`: Hook-generated reply text can be accepted into `event.messages` and then silently not delivered to the user.

## Summary

Keep open: current main and the latest release still fire `message:received` and `message:sent` internal hooks without draining `event.messages`, while https://github.com/openclaw/openclaw/pull/41375 is the open closing implementation path and is not merged yet.

## What This Changes

The issue asks for internal hook replies pushed into `event.messages` from `message:received` and `message:sent` hooks to be delivered back to users.

## Best Possible Solution

Repair and land or replace https://github.com/openclaw/openclaw/pull/41375 so `message:received` hook replies are delivered only through replyable, policy-respecting routes, while keeping `message:sent` lifecycle-only unless maintainers explicitly change that contract.

## Reproduction Assessment

Yes, at source level. A `message:received` internal hook can push into `event.messages`, but current main fires the hook without retaining or draining that array; `message:sent` likewise emits as lifecycle-only with no reply drain.

## Solution Assessment

No, the issue is not solved on current main or the latest release. The best path is to finish the linked maintainer PR or a replacement that honors sender routing and sendPolicy, not to close this issue or queue a parallel repair.
