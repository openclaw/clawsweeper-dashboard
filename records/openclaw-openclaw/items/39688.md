## Summary

Keep open. Current main still fires internal message hooks without retaining or draining hook-added `event.messages`; the open closing PR at https://github.com/openclaw/openclaw/pull/41375 owns the fix path but is unmerged and still has routing, send-policy, and contract review blockers.

## What This Changes

The issue requests that messages pushed into `event.messages` by internal `message:received` and `message:sent` hooks be delivered to the user instead of silently ignored.

## Best Possible Solution

Land a current-main fix that drains `message:received` hook replies through the normal reply target, send-policy, and source-delivery checks, while keeping `message:sent` lifecycle-only unless maintainers explicitly expand that contract.

## Reproduction Assessment

Yes. Source inspection on current main shows `message:received` and `message:sent` hooks fire with inline events, but only command hooks retain and drain `hookEvent.messages`.

## Solution Assessment

Not yet. The narrow maintainable fix is to deliver `message:received` replies after normal send-policy and routing checks, and to treat `message:sent` as a product contract decision rather than silently adding a new reply surface.
