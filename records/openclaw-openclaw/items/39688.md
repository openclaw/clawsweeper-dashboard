Keep open. Current main still drops hook-added messages for `message:received` and `message:sent`, and the linked open maintainer PR remains the active fix path with reply-routing boundaries to settle before closure.

## What This Changes

The issue asks for internal message hook output pushed into `event.messages` by `message:received` and `message:sent` handlers to be delivered back to the user, observed on Feishu in OpenClaw 2026.2.26.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/41375 or an equivalent maintainer-approved patch that drains hook replies only on reply-capable surfaces, updates docs and regression tests, and closes this issue after merge.

## Reproduction Assessment

Yes. Source inspection is enough: `message:received` and `message:sent` hook handlers can mutate `event.messages`, but the current call sites fire the hook without retaining or draining the mutated event object.

## Solution Assessment

Yes. Keeping this issue open is the right current path because an open closing PR owns the implementation, and the fix needs a maintained reply-surface boundary rather than unconditional lifecycle-hook auto-replies.
