## Summary

Keep open. Current main and the latest release still fire `message:received` and `message:sent` internal hooks without draining hook-added `event.messages`, while the active closing PR at https://github.com/openclaw/openclaw/pull/41375 is still open, conflicting, and carrying routing/policy review concerns.

## What This Changes

The issue asks for internal hook handler output pushed into `event.messages` during `message:received` and `message:sent` hooks to be delivered to the user.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent maintainer-approved patch that drains hook replies only on safe reply-capable surfaces, honors send policy, updates docs/tests, and then closes this issue after merge.

## Reproduction Assessment

Yes. Source inspection on current main shows hook events can collect messages, but the `message:received` and `message:sent` call sites pass inline events into `triggerInternalHook` via `fireAndForgetHook` and never retain or drain `event.messages`.

## Solution Assessment

Yes for the triage outcome: keeping this issue open is the right path while the linked implementation is reviewed. The code fix should preserve a safe reply-surface boundary rather than unconditionally auto-replying from every lifecycle hook.
