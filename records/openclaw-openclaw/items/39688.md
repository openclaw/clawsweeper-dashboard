## Summary

Keep open. Current main still triggers `message:received` and `message:sent` internal hooks without consuming hook-produced `event.messages`, and open maintainer-labeled PR #41375 remains the active closing implementation path.

## What This Changes

The issue reports that `message:received` and `message:sent` internal hooks can populate `event.messages`, but those messages are not sent back to the user.

## Best Possible Solution

Keep this issue open until PR #41375 or an equivalent maintainer-approved patch lands. The end state should drain hook-produced replies only when OpenClaw has a safe reply target, preserve reset/new behavior, add regression coverage for that delivery contract, and update the hooks docs so lifecycle-only surfaces do not promise automatic delivery.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to register a `message:received` internal hook that pushes into `event.messages` and send a Feishu or other routed inbound message; current main fires the hook in a fire-and-forget path and never drains the messages array.

## Solution Assessment

Yes for the active linked approach, not for current main. Reviewing, updating, and landing PR #41375 or an equivalent maintainer fix is narrower and safer than opening another parallel repair PR, especially because it limits automatic delivery to reply-capable surfaces.
