- `impact:message-loss`: Messages pushed by internal `message:received` hooks are silently dropped instead of being delivered to the user.

## Summary

Current main and the latest release still fire the internal `message:received` hook without draining `event.messages`, so the reported message-loss bug remains open; the active maintainer PR is the canonical fix candidate and the issue should stay open until that path lands or is replaced.

## What This Changes

The issue reports that internal hook handlers for `message:received` and `message:sent` can add replies to `event.messages`, but those replies are not sent back to users.

## Best Possible Solution

Review, repair, and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent maintainer-approved current-main patch that drains `message:received` hook replies through existing reply-target and send-policy rules, while keeping `message:sent` lifecycle-only unless maintainers explicitly expand that contract.

## Reproduction Assessment

Yes, at source level. Current main creates and triggers the `message:received` internal hook event without retaining or draining `event.messages`, while command hooks already drain that array through reply routing.

## Solution Assessment

Yes, the best current path is the open maintainer PR after repair, not a new duplicate fix PR. The `message:sent` part should remain a documented lifecycle boundary unless maintainers choose to expand it separately.
