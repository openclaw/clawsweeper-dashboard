## Summary

Keep open: current `main` and the latest release still fire `message:received` internal hooks without retaining or draining `event.messages`, and the same-repository linked implementation candidate remains open rather than merged.

## What This Changes

The issue reports that internal hook replies pushed to `event.messages` by `message:received` and `message:sent` hooks are triggered but not delivered back to the user.

## Best Possible Solution

Repair and land the linked PR for `message:received` on safe reply-capable routing, while keeping `message:sent` lifecycle-only unless maintainers approve a new reply contract.

## Reproduction Assessment

Yes, at source level: current main fires `message:received` internal hooks without retaining or draining the mutable `event.messages` array. I did not run a live Feishu provider reproduction in this read-only review.

## Solution Assessment

Partly: delivering `message:received` replies through the existing route-reply boundary is the right repair shape, but the active PR still needs rebase and review fixes. Expanding `message:sent` auto-delivery is not the narrow bug fix under the current docs contract.
