## Summary

Keep open: current main and the latest release still fire `message:received` internal hooks without draining `event.messages`, and the open linked repair PR remains the active but not-yet-merged path.

## What This Changes

The issue reports that internal hook replies pushed into `event.messages` from `message:received` and `message:sent` are not sent back to users.

## Best Possible Solution

Refresh or replace the linked PR with a current-main fix that drains `message:received` hook replies through the policy-aware reply route, while keeping `message:sent` observational unless maintainers approve a new delivery contract.

## Reproduction Assessment

Yes, at source level: current main creates and triggers a mutable `message:received` internal hook event but never drains that event's `messages` array into delivery. I did not run live Feishu proof because this review is read-only.

## Solution Assessment

No for the full requested shape: `message:received` delivery is a docs-backed bug, but making `message:sent` auto-reply would add a new lifecycle hook contract. The narrow maintainable solution is to fix `message:received` via the policy-aware reply route and leave `message:sent` lifecycle-only unless maintainers decide otherwise.
