## Summary

Keep open. Current main still emits `message:received` and `message:sent` internal hook events without draining hook-produced `event.messages`, and open maintainer-labeled PR #41375 is the active closing path.

## What This Changes

Internal `message:received` and `message:sent` hooks are reported to drop `event.messages` replies instead of sending them to the user, originally observed on Feishu in 2026.2.26.

## Best Possible Solution

Land PR #41375 or an equivalent maintainer-approved patch that drains `message:received` replies when OpenClaw has a safe routing target and updates the hooks docs to define which surfaces auto-deliver `event.messages`.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to register a `message:received` internal hook that pushes to `event.messages` and send an inbound routed message; current main fire-and-forgets the event and never drains the messages array.

## Solution Assessment

Yes. The best current path is PR #41375 or an equivalent narrow fix, because it routes hook replies only where a safe reply target exists instead of making every lifecycle hook auto-reply.
