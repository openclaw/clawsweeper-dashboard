## Label Justifications

- `P2`: This is a source-confirmed behavior bug with silently dropped hook output, but it is limited to internal hook reply delivery and already has an open fix path.
- `impact:message-loss`: Hook handler output placed in `event.messages` can be silently dropped instead of being delivered to the user.

## Summary

Keep open. Current `main` still documents `message:received` as a replyable internal-hook surface, but the inbound hook event is fired inline and never drained; the open closing PR at https://github.com/openclaw/openclaw/pull/41375 is the active fix path and is still unmerged/conflicting.

## What This Changes

Internal hook handlers can push text into `event.messages`, but this issue reports that `message:received` and `message:sent` hook output is not sent back to users.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent patch that drains `message:received` hook messages only on reply-capable, policy-compliant surfaces and documents lifecycle hooks that intentionally remain non-replying.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `event.messages` can be populated by internal hook handlers, but the `message:received` call site passes an inline event into `triggerInternalHook` and never retains or drains the resulting messages.

## Solution Assessment

Yes for triage: keeping this issue open is the right outcome while the linked implementation is reviewed. The eventual fix should preserve reply-target and send-policy boundaries and should not silently change lifecycle-only `message:sent` semantics without maintainer approval.
