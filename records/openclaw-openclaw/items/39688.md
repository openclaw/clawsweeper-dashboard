## Summary

Current main still exposes `InternalHookEvent.messages` as user-facing output but fires `message:received` and `message:sent` hooks without draining that output. The open canonical PR remains necessary, but it is conflicting and its discussion identifies an unresolved send-policy boundary, so this issue should stay open until that PR is safely revised, reviewed, and either landed or closed.

## What This Changes

Deliver messages accumulated by internal `message:received` and `message:sent` hooks instead of silently discarding them after hook execution.

## Best Possible Solution

Revise the canonical PR so `message:received` hook replies use the same send-policy and safe-target checks as ordinary replies, while keeping `message:sent` observational unless maintainers approve a separate loop-safe delivery contract.

## Reproduction Assessment

Yes. Current source explicitly declares hook messages as user-facing output, while both message-event callers trigger their hooks without consuming that output.

## Solution Assessment

The linked reply-capable inbound approach is directionally correct, but it is not yet the best safe implementation until normal send-policy enforcement is preserved. Unconditional `message:sent` replies should not be added without a distinct recursion-safe product contract.
