## Summary

Keep open: current main and v2026.6.10 still fire the internal `message:received` hook without retaining or draining `event.messages`, while https://github.com/openclaw/openclaw/pull/41375 remains the open linked fix candidate.

## What This Changes

The issue requests that messages pushed from internal `message:received` and `message:sent` hooks be delivered back to the user instead of silently staying in `event.messages`.

## Best Possible Solution

Repair and land the linked PR so `message:received` hook replies are drained only through safe reply-capable routing, while keeping `message:sent` lifecycle-only unless maintainers approve a new contract.

## Reproduction Assessment

Yes at source level: current main fires `message:received` internal hooks without retaining or draining the mutable `event.messages` array. I did not run a live Feishu provider reproduction in this read-only review.

## Solution Assessment

Partly: delivering `message:received` replies through the existing route-reply path is the right boundary, but the active PR still needs repair before it is the best fix. Expanding `message:sent` auto-delivery is not part of the safe bug fix without maintainer product direction.
