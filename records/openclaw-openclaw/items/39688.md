## Summary

Keep open. Current `main` and `v2026.5.12` still document `event.messages` as a user-reply channel while the `message:received` and `message:sent` internal-hook paths fire hooks without draining those messages; the active path is the open closing PR at https://github.com/openclaw/openclaw/pull/41375.

## What This Changes

The issue reports that internal hook replies pushed into `event.messages` by `message:received` and `message:sent` hooks are not sent back to users, observed on Feishu in OpenClaw 2026.2.26.

## Best Possible Solution

Review, fix as needed, and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent maintainer-approved patch that drains hook replies only on safe reply-capable paths, preserves send policy, and aligns docs/tests with the final lifecycle-hook contract.

## Reproduction Assessment

Yes, at source level. A `message:received` or `message:sent` internal hook that pushes into `event.messages` exercises current-main paths that emit the hook but never retain or drain that mutable event afterward.

## Solution Assessment

Yes, keeping this issue open is the right current path because an open closing PR already owns the implementation. The final fix should preserve routing and send-policy constraints rather than making every lifecycle hook auto-reply by default.
