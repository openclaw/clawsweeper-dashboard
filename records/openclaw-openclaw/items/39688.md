## Label Justifications

- `P2`: This is a source-confirmed behavior bug with silently dropped hook output, but it is limited to internal hook reply delivery and already has an open fix path.
- `impact:message-loss`: Hook handler output placed in `event.messages` can be silently dropped instead of being delivered to the user.

## Summary

Keep open. Current main and the latest release still fire `message:received` and `message:sent` internal hooks without draining hook-added `event.messages`, and the open closing PR at https://github.com/openclaw/openclaw/pull/41375 still needs maintainer review before this issue can close.

## What This Changes

Internal hook handlers that push to `event.messages` on `message:received` or `message:sent` should have their output delivered to the user instead of silently dropped.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent patch that drains hook replies only on reply-capable, policy-compliant surfaces and updates docs/tests for lifecycle hooks that intentionally remain non-replying.

## Reproduction Assessment

Yes. Source inspection on current main shows `event.messages` can be populated by internal hook handlers, but the `message:received` and `message:sent` call sites pass inline events into `triggerInternalHook` and never retain or drain the resulting messages.

## Solution Assessment

Yes for the triage outcome: keeping this issue open is the right path while the linked implementation is reviewed. The code fix should preserve safe reply-surface and send-policy boundaries instead of auto-replying from every lifecycle hook.
