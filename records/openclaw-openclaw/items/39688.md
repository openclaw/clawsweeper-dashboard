## Summary

Keep open. Current main still fires the reported internal message hooks without draining hook-added `event.messages`, and the open closing PR at https://github.com/openclaw/openclaw/pull/41375 remains the active fix path.

## What This Changes

The issue reports that internal hook replies pushed into `event.messages` for `message:received` and `message:sent` are not delivered to the user.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/41375 or an equivalent maintainer-approved patch that drains hook replies only on reply-capable paths and updates docs and regression tests to match that contract.

## Reproduction Assessment

Yes. Source inspection is enough: a `message:received` or `message:sent` hook can mutate `event.messages`, but the current call sites fire the hook without retaining or draining the mutated event object.

## Solution Assessment

Yes. Keeping this issue open is the right current path because an open closing PR owns the implementation, and the fix needs a maintained reply-surface boundary rather than unconditional lifecycle-hook auto-replies.
