## Summary

Keep open: current main and v2026.6.6 still fire `message:received` internal hooks without draining `event.messages`, and the linked open repair PR is still unmerged with policy/routing blockers.

## What This Changes

The issue requests that internal hook replies pushed to `event.messages` from `message:received` and `message:sent` be delivered back to the user, reported on Feishu in OpenClaw 2026.2.26.

## Best Possible Solution

Refresh or replace https://github.com/openclaw/openclaw/pull/41375 with a current-main fix that drains `message:received` hook replies through the policy-aware reply route, targets the originating sender, and leaves `message:sent` observational unless maintainers approve a new contract.

## Reproduction Assessment

Yes, at source level. Current main creates and triggers a mutable `message:received` internal hook event, but the normal inbound dispatch path never drains that event's `messages` array into delivery.

## Solution Assessment

No for the full requested shape. The best narrow fix is `message:received` delivery through the existing policy-aware reply route, with `message:sent` left lifecycle-only unless maintainers deliberately approve that new behavior.
