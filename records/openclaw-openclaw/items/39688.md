## Summary

Keep open. Current `main` and the latest shipped release still fire `message:received` and `message:sent` internal hooks without draining `event.messages`, and the active path is the open maintainer-labeled closing PR at https://github.com/openclaw/openclaw/pull/41375 rather than closing this issue before that fix lands.

## What This Changes

The issue requests that messages pushed into `event.messages` by `message:received` and `message:sent` internal hooks be delivered back to the user.

## Best Possible Solution

Review, fix as needed, and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent maintainer-approved patch that drains hook messages only on safe reply targets, preserves send policy, and clarifies docs/tests for lifecycle-only hooks.

## Reproduction Assessment

Yes, at source level. A `message:received` or `message:sent` hook that pushes into `event.messages` exercises current-main paths that emit the hook but never retain or drain that mutable event afterward.

## Solution Assessment

Yes, keeping this issue open is the best current outcome because an open maintainer-labeled PR already closes it. The maintainable fix should deliver only on safe reply-capable surfaces and document whether lifecycle-only `message:sent` remains non-replying.
