## Summary

Keep open. Current main still drops `event.messages` from `message:received` and `message:sent` internal hooks, and the active fix path is the open maintainer-labeled closing PR rather than a separate cleanup close.

## What This Changes

The issue requests that messages pushed by `message:received` and `message:sent` internal hook handlers be sent back to the user.

## Best Possible Solution

Land one maintainer-approved fix that drains hook messages only where OpenClaw has a safe reply target, preserves send policy and loop suppression, and updates docs plus regression tests.

## Reproduction Assessment

Yes, at source level. Registering a `message:received` or `message:sent` internal hook that pushes into `event.messages` exercises current-main paths that emit the hook but never retain or drain the event afterward.

## Solution Assessment

Yes, keeping this issue open is the right current outcome because an open closing PR already targets the fix. The narrow maintainable solution is to drain messages only on safe reply-capable surfaces and clarify the documented hook contract.
