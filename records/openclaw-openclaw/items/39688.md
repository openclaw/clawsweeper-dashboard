## Summary

Current main still has the reported source-level gap for `message:received`: the hook event is fired, but pushed `event.messages` are not drained or routed. This should stay open because the open closing PR is the canonical implementation path, but it is still dirty and has policy/routing concerns to resolve before it can land.

## What This Changes

The issue reports that internal hook handlers for `message:received` and `message:sent` append to `event.messages`, but those messages are not sent back to the Feishu user.

## Best Possible Solution

Revise or replace the open canonical PR so `message:received` hook messages drain through the current policy-aware originating-reply path, while keeping `message:sent` observational unless maintainers approve a separate loop-safe delivery contract.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: register a `message:received` hook that pushes to `event.messages`, then trigger an inbound message and observe that current main fires the hook without any delivery drain. I did not run a live Feishu repro because this review is read-only.

## Solution Assessment

No, sending output for both named message hooks is not automatically the best fix. The maintainable path is to drain `message:received` through the existing policy-aware reply boundary and treat `message:sent` delivery as a separate maintainer decision.
