- `impact:message-loss`: Hook handler output placed in `event.messages` can be silently dropped instead of being delivered to the user.

## Summary

Keep open. Current `main` and the latest release still fire `message:received` and `message:sent` internal hooks without delivering hook-added `event.messages`, and the open closing PR at https://github.com/openclaw/openclaw/pull/41375 still needs maintainer review before this can close.

## What This Changes

The issue asks for internal `message:received` and `message:sent` hooks to send messages pushed into `event.messages` back to the user.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/41375 or an equivalent patch that drains hook replies only on reply-capable, policy-compliant surfaces and updates docs/tests for lifecycle hooks that intentionally remain non-replying.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `event.messages` can be populated by internal hook handlers, but the `message:received` and `message:sent` call sites pass inline events into `triggerInternalHook` and never retain or drain the resulting messages.

## Solution Assessment

Yes for triage: keeping this issue open is the right outcome while the linked implementation is reviewed. The eventual fix should preserve reply-target and send-policy boundaries instead of blindly auto-replying from every lifecycle hook.
