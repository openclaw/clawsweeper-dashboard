## Summary

Keep open. Current main still fires the internal `message:received` and `message:sent` hooks without consuming hook-produced `event.messages`, and open maintainer-labeled PR #41375 is the active closing fix path.

## What This Changes

The issue requests that `event.messages` pushed by `message:received` and `message:sent` internal hooks be delivered to the user instead of being silently ignored.

## Best Possible Solution

Land or close PR #41375 as the canonical path; the desired end state is to drain internal hook replies only where OpenClaw has a safe reply target, with regression coverage and docs that clarify lifecycle-only hooks.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to register a `message:received` hook that pushes to `event.messages` and send a routed inbound message; current main fire-and-forgets the hook event and never drains the messages array.

## Solution Assessment

Yes. The narrow maintainable path is PR #41375 or an equivalent patch that routes `message:received` replies only when a safe target exists and updates docs instead of making every lifecycle hook auto-reply.
