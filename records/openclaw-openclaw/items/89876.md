## Summary

Keep open: current main and v2026.6.9 still only have Telegram callback, inline-button, managed-select, poll, and plugin-interactive primitives; they do not provide the requested multi-question decision card, draft/refinement state, or one structured decision packet delivered to the agent.

## What This Changes

The issue requests a Telegram-native multi-question decision packet card flow for bounded human input before an agent starts work.

## Best Possible Solution

Define the packet contract and state owner first, then implement the Telegram renderer, callback tokens, draft/refinement handling, structured submit/cancel behavior, and targeted Telegram tests at the right plugin or SDK boundary.

## Reproduction Assessment

Not applicable: this is a new Telegram UX capability rather than broken existing behavior. Source inspection confirms the lower-level callback primitives exist but the requested packet flow does not.

## Solution Assessment

Unclear until maintainers choose the permanent contract. A Telegram-local first slice is plausible, but the best implementation depends on how agents request and receive structured packets and whether a shared SDK seam is needed.
