## Summary

Keep open: current main and the latest release have Telegram inline buttons, callback routing, editable callback messages, and managed select primitives, but they still lack the requested multi-question decision-card runtime, draft/refinement state, and structured submitted packet delivered to the agent.

## What This Changes

Add a Telegram-native multi-question decision packet card flow with selectable options, revisions, refinements, submit/cancel semantics, and one structured agent input.

## Best Possible Solution

Define the packet contract and state owner first, then implement the renderer, deterministic callback tokens, draft/refinement handling, structured submit/cancel behavior, and targeted Telegram tests inside the Telegram plugin unless maintainers choose a shared SDK seam.

## Reproduction Assessment

Not applicable: this is a new Telegram UX capability rather than broken existing behavior. Source inspection gives high-confidence absence on current main while confirming the lower-level Telegram primitives exist.

## Solution Assessment

Unclear until maintainers choose the permanent contract. A Telegram-local first slice is plausible, but the best implementation depends on how agents request and receive the structured packet and whether a shared SDK seam is needed.
