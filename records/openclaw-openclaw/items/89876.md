## Summary

Keep open: current main and v2026.6.8 have Telegram inline buttons, callback dispatch, managed select primitives, and plugin interactive handlers, but they do not implement the requested multi-question decision-card runtime, refinement/draft state, or one structured packet delivered to the agent.

## What This Changes

The issue requests a Telegram-native multi-question decision packet card flow that lets a user revise bounded answers and submit one structured packet to an agent.

## Best Possible Solution

Define the packet contract and state ownership first, then implement the renderer, callback tokens, draft/refinement handling, structured submit/cancel behavior, and targeted Telegram tests inside the Telegram plugin unless a shared SDK seam is explicitly needed.

## Reproduction Assessment

Not applicable: this is a new Telegram UX feature rather than broken existing behavior. Source inspection gives high-confidence absence on current main while confirming the lower-level Telegram primitives exist.

## Solution Assessment

Unclear until maintainers choose the permanent contract. A Telegram-local first slice is plausible, but the best implementation depends on how agents request and receive the structured packet and whether a shared SDK seam is needed.
