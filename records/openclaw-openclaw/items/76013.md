## Summary

Keep open. Current main still has short Telegram outbound method guards and only forces timeout-triggered transport fallback for startup/control-plane methods, so the reported Windows/Node 24 `sendChatAction`/`sendMessage` delivery failure is not implemented as fixed.

## What This Changes

The issue asks OpenClaw to stop Telegram Bot API sends from failing under the current hard request timeouts on Windows/Node 24, likely by raising or honoring configurable timeouts and improving transport fallback.

## Best Possible Solution

Keep the existing HTTP/1.1 and IPv4 fallback hardening, then make Telegram outbound guards honor the configured timeout where safe and add fallback/retry only for non-duplicating operations or pre-connect failures.

## Solution Assessment
