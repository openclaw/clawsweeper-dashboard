## Summary

Keep open. The report is fresh and source inspection shows current main still hard-limits Telegram inbound Bot API `sendChatAction` to 10s and `sendMessage` to 20s, while timeout-triggered transport fallback is limited to startup/control methods; recent main hardening helps, but it is not enough evidence to close this Windows/Node 24 outbound timeout as implemented.

## What This Changes

The issue reports Telegram outbound Bot API requests timing out under undici on Windows/Node 24 and asks for longer or alternative Telegram transport handling.

## Best Possible Solution

Keep the HTTP/1.1 and IPv4 fallback hardening, then make Telegram outbound method guards honor the configured timeout and promote/retry only safe methods such as `sendChatAction` without making `sendMessage` duplicate-prone.

## Reproduction Assessment

Partial yes: the issue provides concrete logs and current-main source confirms the short outbound guards and limited timeout fallback path. A live Windows/Node 24 Telegram reproduction was not attempted.

## Solution Assessment

Unclear for the suggested native-https replacement; a narrower fix in the Telegram plugin should first honor `channels.telegram.timeoutSeconds` and improve safe fallback behavior around outbound methods.
