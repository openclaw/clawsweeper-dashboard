## Summary

Keep open: the bug is source-reproducible and the PR targets the right helper, but the patch currently bypasses media-directive preservation and lacks inspectable after-fix channel proof; the branch is also conflicting with current main.

## What This Changes

The PR changes `buildEmbeddedRunPayloads` to detect phase metadata on the last assistant message and prefer phase-filtered visible text for channel-delivered replies.

## Best Possible Solution

Rebase on current main, add regression coverage for a single raw streamed phase-tagged assistant text, and make phase filtering use the existing media-aware final-answer source without changing unphased providers or aborted-run suppression.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence helper-level reproduction: call `buildEmbeddedRunPayloads` with one raw `assistantTexts` entry containing commentary plus final text and a phase-tagged `lastAssistant`; current main returns the raw streamed entry.

## Solution Assessment

No, not as written. The phase-aware direction is narrow, but the new branch runs before media-aware raw final-answer preservation; the safer fix is to preserve `fallbackAnswerSourceText`/`shouldPreferRawAnswerText` while suppressing commentary.
