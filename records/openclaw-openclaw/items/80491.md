## Summary

Keep open: the reported channel-delivery leak is valid and the PR targets the right helper, but the current diff still allows commentary-only phase output to fall back to raw streamed text and it regresses existing media-directive preservation.

## What This Changes

The PR changes `buildEmbeddedRunPayloads` to detect `textSignature.phase` on `lastAssistant` and use `extractAssistantVisibleText` for channel payload text when phase metadata exists.

## Best Possible Solution

Keep the phase-aware payload fix, but implement it around the media-aware final-answer source, add single-stream and commentary-only regression coverage, and require redacted real channel output before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence helper-level reproduction: call `buildEmbeddedRunPayloads` with one raw streamed `assistantTexts` entry containing commentary plus final text and a phase-tagged `lastAssistant`; current main returns the raw streamed entry in the one-entry case. The linked report confirms the same behavior on v2026.5.7 with Telegram/iMessage channel delivery.

## Solution Assessment

No, the current PR ordering is not the best fix yet: it should suppress raw phase-tagged text even when `final_answer` is empty and preserve the existing media-aware raw final-answer path. A corrected helper-level fix with focused regression tests is the narrower maintainable path.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Suppress raw phase output when final text is empty:** `src/agents/pi-embedded-runner/run/payloads.ts:393-394`

That drops the `mediaUrl`/`mediaUrls` expected by existing payload tests; choose the media-aware final-answer source when filtering phases.
  - confidence: 0.91
