## Summary

Keep open: the reported channel-delivery leak is valid and the phase-filter direction matches the linked issue and maintainer review comment, but this diff currently short-circuits the existing media-preserving raw-final-answer path and can drop attachments for phase-tagged final answers. The after-fix proof is also second-hand text only, not inspectable live output, logs, screenshots, or a recording.

## What This Changes

This PR changes `buildEmbeddedRunPayloads` to detect phase metadata on the last assistant message and prefer `extractAssistantVisibleText` for channel delivery when phase tags exist.

## Best Possible Solution

Preserve the phase-aware channel fix, but route through the existing raw/media-aware final-answer source when needed and add focused regression coverage for the single streamed-text phase-tagged case before merging.

## Reproduction Assessment

Yes, source inspection gives a high-confidence helper-level reproduction: call `buildEmbeddedRunPayloads` with one raw streamed assistant text containing commentary plus final text and a phase-tagged `lastAssistant`; current main returns the raw streamed entry. The linked issue reports the same behavior on v2026.5.7 with Telegram/iMessage channel delivery.

## Solution Assessment

No, not as currently ordered: the phase branch runs before the existing media-directive preservation path. The safer fix is to use the phase-filtered final answer for phase-tagged content while preserving `shouldPreferRawAnswerText`/`fallbackAnswerSourceText` when raw final text carries channel directives.
