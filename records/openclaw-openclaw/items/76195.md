## Summary

Current main and the v2026.4.29 release source both serialize macOS LaunchAgent `ProgramArguments` from raw strings; the generated env-wrapper path is built internally and never passes through Markdown or terminal-link formatting, so the reported `[path](url)` plist value does not match the repository implementation.

## What This Changes

The issue reports that macOS gateway install writes Markdown link syntax into the LaunchAgent `ProgramArguments[0]` path, causing launchd to execute a non-existent wrapper path.

## Best Possible Solution

Leave the LaunchAgent serializer unchanged unless a fresh report includes a raw plist or diagnostics export proving an in-repo path can still produce Markdown syntax.

## Reproduction Assessment

No; the source-level reproduction path does not reach a Markdown formatter. Both current main and v2026.4.29 build and serialize the generated env-wrapper path as a plain XML string, with unit coverage for that exact shape.

## Solution Assessment

Yes; closing without a code change is the narrowest maintainable outcome because adding a sanitizer would mask an unproven external/copying artifact rather than fixing an observable in-repo defect.
