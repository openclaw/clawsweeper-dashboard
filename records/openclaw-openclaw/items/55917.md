## Summary

Keep open. Current main and v2026.5.22 still let Telegram document `getFile()` file-too-big or exhausted-retry failures return `null`, produce an empty attachment list, and render a document placeholder without a saved media path; the focused document fix remains open at https://github.com/openclaw/openclaw/pull/55970.

## What This Changes

The issue requests reliable Telegram inbound document attachment handling so documents do not arrive as placeholder-only `<media:document>` messages without a usable saved file path.

## Best Possible Solution

Land or replace a narrow Telegram plugin fix that carries document-unavailable reasons through media resolution and body rendering while preserving attachment metadata and not promising public Bot API downloads above Telegram's 20MB limit.

## Reproduction Assessment

Yes, source-level: mock Telegram `ctx.getFile()` to reject with a file-too-big error or exhaust retries, and current main returns `null`, builds `allMedia = []`, and can render `<media:document>` without a saved path. I did not run a live Telegram document replay in this read-only review.

## Solution Assessment

Yes for keeping the issue open behind the linked repair path, but the fix should surface explicit unavailable-document state rather than implying public Bot API downloads above Telegram's documented limit are always possible.
