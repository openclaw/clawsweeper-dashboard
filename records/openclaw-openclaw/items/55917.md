## Summary

Keep open. Current main still allows the reported Telegram document path to degrade into placeholder-only agent input: a Telegram `getFile` “file is too big” failure returns no media, the body renderer can use only `<media:document>`, and the finalized inbound context has no media path fields. The linked PR #55970 remains the active implementation candidate rather than a reason to close this issue before it lands.

## What This Changes

The issue requests that failed Telegram inbound document downloads produce a usable attachment/path, or at least not silently arrive at the agent as only `<media:document>` with no inbound media file.

## Best Possible Solution

Keep this issue open until PR #55970 lands or an equivalent Telegram-owned fix is merged. The end state should preserve the `<media:document>` placeholder for failed downloads, add an agent-visible unavailable reason that distinguishes public Bot API size-limit failures from generic download failures, and document the difference between `channels.telegram.mediaMaxMb` and Telegram's public Bot API download cap plus the self-hosted `apiRoot`/`trustedLocalFileRoots` path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make `ctx.getFile()` throw Telegram's `file is too big` error for a document: current tests show `resolveMedia` returns `null`, and the body/session code then has only `<media:document>` and no `MediaPath`/`MediaPaths`.

## Solution Assessment

Unclear for the open PR because this review did not inspect its full branch diff in the checkout, but the direction described by PR #55970 is the narrowest maintainable fix: surface an explicit unavailable reason in Telegram-owned inbound handling without pretending the public Bot API can download files beyond its limit.
