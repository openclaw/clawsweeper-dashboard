## Summary

Keep open. Current main still has a source-confirmed Telegram document fallback path where a `getFile` failure returns no media, the agent can receive only `<media:document>`, and the open PR #55970 has not landed.

## What This Changes

The issue asks Telegram inbound documents that degrade to `<media:document>` with no saved file/path to become usable or at least explicit attachment failures, especially for 20+ MB files despite `mediaMaxMb=100`.

## Best Possible Solution

Keep the Telegram-owned fix narrow: preserve placeholder delivery, add an agent-visible unavailable reason for file-too-large and generic download failures, and document the public Bot API limit versus self-hosted `apiRoot` plus `trustedLocalFileRoots`.

## Reproduction Assessment

Yes, from source inspection: mock a Telegram document whose `ctx.getFile()` rejects with `file is too big`; `resolveMedia` returns null, the handler enqueues `allMedia=[]`, and body/context construction can produce `<media:document>` without `MediaPaths`. I did not run a live Telegram bot transfer in this read-only pass.

## Solution Assessment

Yes for the maintainer path: PR #55970-style reason plumbing is the narrowest maintainable fix for the ambiguous fallback. Trying to make the public Bot API download files it rejects is not the right core fix; larger-file support should use the existing self-hosted/custom API root path.
