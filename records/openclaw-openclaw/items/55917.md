## Summary

Keep open: current main and the latest release still have a Telegram path where `getFile()` failure returns no media, the handler continues with an empty attachment list, and body rendering can produce a document placeholder without a saved file. The related document-fallback PR is open, conflicting, and proof-blocked, while the newer local Bot API path PR only covers one concrete materialization cause.

## What This Changes

The issue reports intermittent Telegram document ingest where incoming files degrade to `<media:document>` without a stored inbound attachment path.

## Best Possible Solution

Keep this issue open as the canonical document report until Telegram inbound media either materializes the document bytes or exposes a clear unavailable state without misleading placeholder-only delivery.

## Reproduction Assessment

Yes, at source level: make Telegram `ctx.getFile()` fail for a document, and current main returns `null`, passes `allMedia = []`, then renders a media placeholder without an attachment path. I did not run a live Telegram bot reproduction in this read-only review.
