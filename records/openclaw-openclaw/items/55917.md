## Summary

Keep open. Current main and the latest release still let Telegram `getFile` file-too-big or exhausted-retry failures become `null` media, leaving document-only messages as `<media:document>` with no saved attachment path; the focused fix candidate remains open and unmerged at https://github.com/openclaw/openclaw/pull/55970.

## What This Changes

The issue reports intermittent Telegram document messages reaching the agent as `<media:document>` without a saved inbound file or usable attachment path.

## Best Possible Solution

Land or replace a narrow Telegram plugin fix that carries document-unavailable reasons through resolver, debounce, media-group, reply-context, and body-rendering paths while respecting the public Bot API 20MB limit and existing self-hosted API configuration.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: mock Telegram `ctx.getFile()` to reject with a file-too-big error or exhaust retries, then `resolveMedia` returns `null`, `allMedia` becomes empty, and body rendering falls back to `<media:document>`.

## Solution Assessment

Yes for the current repair direction, but the issue is not solved yet. The maintainable fix is explicit unavailable-reason plumbing and proof, not implying public Bot API downloads can exceed Telegram's documented limit.
