## Summary

Keep open. Current main still has a source-confirmed Telegram document fallback where `getFile` failures return no media, the agent can receive only `<media:document>` without an attachment or reason, and https://github.com/openclaw/openclaw/pull/55970 remains the open implementation candidate.

## What This Changes

Telegram inbound documents should be downloaded into inbound media storage and exposed to agents as attachments instead of degrading to `<media:document>`.

## Best Possible Solution

Land a narrow Telegram plugin fix that preserves placeholder delivery but adds agent-visible unavailable reasons for file-too-large and generic download failures, with focused regression coverage and docs for public Bot API limits versus self-hosted `apiRoot`.

## Reproduction Assessment

Yes, source-reproducible: mock a Telegram document whose `ctx.getFile()` rejects with `file is too big` or exhausts retries; current main returns `null`, enqueues `allMedia=[]`, and builds a placeholder-only body/context.

## Solution Assessment

Yes for the direction: https://github.com/openclaw/openclaw/pull/55970-style reason plumbing is narrower and safer than trying to bypass Telegram's public Bot API download cap, though that PR still needs review/repair before landing.
