## Label Justifications

- `P2`: This is a normal-priority Telegram inbound media bug with clear user impact but limited blast radius to Telegram document handling.
- `impact:data-loss`: The reported document bytes are not saved under inbound media storage, so user-provided attachment data is effectively dropped.
- `impact:message-loss`: The channel message reaches the agent without the actual media payload, leaving only a placeholder in the transcript.

## Summary

Keep open: current main and v2026.6.6 still let Telegram getFile/file-too-big failures return null, leave `allMedia` empty, and build a placeholder-only document body; the related fix PR at https://github.com/openclaw/openclaw/pull/55970 is open but not merged and was already flagged as partial.

## What This Changes

Telegram inbound document failures should not degrade to `<media:document>` without a real attachment or clear unavailable state.

## Best Possible Solution

Consolidate the Telegram getFile-null failure path so documents either materialize as attachments or reach the user and agent with an explicit unavailable reason, then close or rework the stale partial PRs.

## Reproduction Assessment

Yes from source, but not from a live Telegram run here. Force `ctx.getFile()` to return a file-too-big error or exhaust retries for a document, and current main returns null, builds `allMedia = []`, and can emit a placeholder-only body.

## Solution Assessment

No, current main is not the best solution because it preserves delivery by hiding the attachment failure. The best fix should make successful downloads real attachments and make failed downloads explicit instead of silently rendering a bare placeholder.
