## Label Justifications

- `P2`: Telegram document delivery can lose the usable attachment, but the blast radius is bounded to Telegram inbound media handling.
- `impact:data-loss`: The reported document bytes are not saved under inbound media storage when this failure path occurs.
- `impact:message-loss`: The Telegram message reaches the agent without the actual document payload, leaving only placeholder text.

## Summary

Keep open: current main and v2026.6.8 still have a source-proven Telegram document path where `getFile()` failures produce no saved media, the handler only warns on thrown errors, and body rendering can still leave placeholder-only context.

## What This Changes

The issue asks for Telegram document messages to arrive as saved inbound attachments with agent-visible paths instead of sometimes degrading to `<media:document>` only.

## Best Possible Solution

Keep this open until the Telegram plugin either materializes document bytes through the configured public/local Bot API path or exposes a clear unavailable-media state without misleading placeholder-only delivery.

## Reproduction Assessment

Yes, at source level: make Telegram `ctx.getFile()` fail for a document and current main returns `null`, continues with `allMedia = []`, and can render a media placeholder without an attachment path. I did not run a live Telegram bot reproduction in this read-only review.

## Solution Assessment

Unclear as a single branch. The maintainable solution should consolidate generic `getFile()` unavailable-media behavior with the local Bot API materialization path, instead of treating the closed document-fallback PR or the open local-path PR as a complete fix by itself.
