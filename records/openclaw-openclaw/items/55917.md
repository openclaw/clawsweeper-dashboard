## Label Justifications

- `P2`: This is a normal-priority Telegram plugin bug with clear user-visible attachment loss but limited channel-scoped blast radius.
- `impact:message-loss`: The issue is about Telegram message content arriving without the document attachment that was sent.
- `impact:data-loss`: The reported failure silently omits the inbound user file from saved media storage and the agent-visible context.

## Summary

Keep open. Current main still has a source-confirmed Telegram document path where `getFile` failures return no media, the handler turns that into `allMedia=[]`, and the agent context exposes no saved media path; the focused implementation candidate remains open at https://github.com/openclaw/openclaw/pull/55970.

## What This Changes

Telegram inbound documents should be saved as real attachments instead of reaching the agent only as `<media:document>` when download or attachment binding fails.

## Best Possible Solution

Land or replace the narrow Telegram plugin fix so failed document downloads preserve the placeholder while carrying an agent-visible unavailable reason through resolver, debounce, media-group, and body-rendering paths, without pretending public Bot API downloads can exceed Telegram's limit.

## Reproduction Assessment

Yes. Mock a Telegram document whose `ctx.getFile()` rejects with `file is too big` or exhausts retries; current main returns `null`, builds `allMedia=[]`, and exposes no saved media path to the agent.

## Solution Assessment

Yes for the repair direction, but it is not solved yet. Explicit unavailable-reason plumbing is narrower than bypassing Telegram's public Bot API limit, and the open PR still needs to be landed or replaced before this issue can close.
