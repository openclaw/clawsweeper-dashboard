## Summary

Keep open. Current main still has a source-confirmed Telegram document path where `getFile` failure returns no media, the inbound turn can reach the agent as `<media:document>` without `MediaPaths`, and PR #55970 remains the linked implementation candidate rather than a merged fix.

## What This Changes

Telegram inbound document messages can intermittently arrive as `<media:document>` without a usable attachment path or saved inbound file.

## Best Possible Solution

Keep the fix in the Telegram plugin: preserve message delivery, add an agent-visible unavailable reason for file-too-large and generic download failures, and keep larger-file support tied to the existing custom/self-hosted Bot API configuration path.

## Reproduction Assessment

Yes. From source inspection, mock a Telegram document whose `ctx.getFile()` rejects with `file is too big`: `resolveMedia` returns `null`, the handler enqueues `allMedia=[]`, the body can be `<media:document>`, and no `MediaPaths` are attached.

## Solution Assessment

Yes. The narrow maintainable solution is PR #55970-style reason plumbing in the Telegram fallback path; trying to make the public Bot API download files it rejects would be outside this bug fix.
