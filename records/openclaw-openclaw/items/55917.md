- `impact:data-loss`: The reported failure silently omits the inbound user file from saved media storage and the agent-visible context.
- `impact:message-loss`: The issue is about Telegram message content arriving without the document attachment that was sent.

## Summary

Keep open. Current main and the latest release still let Telegram document `getFile` failures become `null` media, then an empty media list and no agent-visible saved path; the focused implementation candidate at https://github.com/openclaw/openclaw/pull/55970 remains open and unmerged.

## What This Changes

The issue reports intermittent Telegram document delivery where the agent receives only `<media:document>` and no saved inbound attachment path.

## Best Possible Solution

Land or replace the narrow Telegram plugin fix so failed document downloads preserve `<media:document>` plus an agent-visible unavailable reason through resolver, debounce, media-group, and body-rendering paths, without implying public Bot API downloads can exceed Telegram's limit.

## Reproduction Assessment

Yes. Mock Telegram `ctx.getFile()` to reject with a file-too-big error or exhaust retries; current main returns `null`, builds `allMedia` as empty, and exposes no saved media path to the agent.

## Solution Assessment

Yes for the current repair direction, but the issue is not solved yet. The maintainable path is explicit unavailable-reason plumbing plus focused coverage and live Telegram proof, not a broad attempt to bypass Telegram's public Bot API download limit.
