## Summary

Keep open: current main and v2026.6.8 still have a source-proven Telegram document path where `getFile()` failures resolve to `null`, no attachment is saved, and body construction can still produce placeholder-only context. The related local Bot API fix at https://github.com/openclaw/openclaw/pull/91984 is open but partial and not merged.

## What This Changes

The issue reports intermittent Telegram inbound document downloads where the agent receives only `<media:document>` and no saved inbound attachment path despite the configured media size cap.

## Best Possible Solution

Keep this open until the Telegram plugin either materializes document bytes through the configured public or local Bot API path, or exposes a clear unavailable-media state without misleading placeholder-only delivery.

## Reproduction Assessment

Yes, at source level: make Telegram `ctx.getFile()` fail for a document on current main; `resolveMedia()` returns `null`, the handler proceeds with `allMedia = []`, and body construction can emit `<media:document>` or `file_id` metadata without an attachment path. I did not run a live Telegram bot reproduction in this read-only review.

## Solution Assessment

Unclear as a single branch. The maintainable fix should consolidate generic `getFile()` unavailable-media handling with local Bot API materialization, rather than treating the closed document-fallback PR or the open local-path PR as complete by itself.
