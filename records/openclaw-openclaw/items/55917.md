## Summary

Keep open: current main and latest release still have a source-proven Telegram document path where `ctx.getFile()` failures return `null`, no attachment is saved, and message body construction can still expose placeholder-only media context.

## What This Changes

The issue reports Telegram document messages intermittently reaching the agent as only `<media:document>` instead of a saved inbound attachment path.

## Best Possible Solution

Consolidate Telegram unavailable-media handling so document bytes materialize through supported public or local Bot API paths when possible, and otherwise expose a clear unavailable-attachment state instead of placeholder-only context.

## Reproduction Assessment

Yes. At source level, make Telegram `ctx.getFile()` fail for a document: current `resolveMedia()` returns `null`, the handler continues with no saved media, and body construction can still emit the document placeholder without an attachment path.

## Solution Assessment

Unclear as a single branch. The maintainable fix should settle the generic unavailable-media contract and coordinate with the local Bot API materialization work instead of treating either the closed fallback PR or the open local-path PR as complete by itself.
