## Label Justifications

- `P2`: This is a real Telegram attachment-ingest bug with limited channel-specific blast radius.
- `impact:data-loss`: The reported failure leaves no inbound file path or saved media artifact for a user-sent document.
- `impact:message-loss`: The agent receives only a placeholder instead of the document attachment content.

## Summary

Keep open. Current main and the latest release still let failed Telegram `getFile()` paths return no media, so a document can still reach the agent as placeholder-only text; the related fix PR is open but unmerged and dirty.

## What This Changes

Telegram inbound documents should be saved and exposed as real attachments instead of degrading to only `<media:document>` when media ingest fails.

## Best Possible Solution

Land a narrow Telegram fix that preserves the supported download paths, surfaces an explicit unavailable reason when Telegram cannot provide a file, and documents the difference between OpenClaw `mediaMaxMb` and Telegram Bot API download limits.

## Reproduction Assessment

Yes at source level, but not as a live Telegram run. Current code and tests show `getFile()` file-too-big or retry-exhaustion returns `null`, which then reaches the body builder with an empty media list and can render only the media placeholder.

## Solution Assessment

No, current main is not the best end state because it still leaves the agent without a real attachment or reason. The best path is to review and either land or replace https://github.com/openclaw/openclaw/pull/55970 with focused fallback messaging and proof.
