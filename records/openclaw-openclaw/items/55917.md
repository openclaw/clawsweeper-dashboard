## Summary

Keep open. Current main still has a source-confirmed Telegram document path where `getFile` failures return no media, the handler continues with an empty attachment list, and the focused implementation candidate remains open at https://github.com/openclaw/openclaw/pull/55970.

## What This Changes

The issue reports intermittent Telegram document messages reaching the agent as only `<media:document>` with no saved inbound file or usable attachment path.

## Best Possible Solution

Land or replace the narrow Telegram plugin fix so failed document downloads preserve the placeholder while carrying an agent-visible unavailable reason through resolver, debounce, media-group, and body rendering paths, without pretending public Bot API downloads can exceed Telegram's limit.

## Reproduction Assessment

Yes. Mock a Telegram document whose `ctx.getFile()` rejects with `file is too big` or exhausts retries; current main returns `null`, builds `allMedia=[]`, and exposes no saved media path to the agent.

## Solution Assessment

Yes for the repair direction, but not solved yet. Explicit unavailable-reason plumbing is narrower than bypassing Telegram's public Bot API limit, and the open PR still needs rebase/proof/propagation work before it can resolve this issue.
