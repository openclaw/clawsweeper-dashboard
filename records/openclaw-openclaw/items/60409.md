## Summary

Keep open: the reset-archive fallback is useful, but this PR is not merge-ready because the current production history surfaces use async transcript readers that still bypass the sync fallback, and the contributor has not supplied real behavior proof. The related discussion points to complementary work in https://github.com/openclaw/openclaw/pull/71537 and async fallback work in https://github.com/openclaw/openclaw/pull/76134, so this needs coordination rather than cleanup closure.

## What This Changes

The PR adds synchronous reset-archive discovery for session transcripts and tests that `readSessionMessages` plus sync title derivation can use the newest `.reset` archive when the active transcript is absent.

## Best Possible Solution

Route active-or-latest-reset archive resolution through a shared resolver used by sync and async history, count, visit, and title paths, or coordinate with https://github.com/openclaw/openclaw/pull/76134 before merge, with focused regression tests and redacted real Gateway proof.

## Reproduction Assessment

Yes, at source level. With only a `<sessionId>.jsonl.reset.<timestamp>` archive and no active `<sessionId>.jsonl`, current main's `chat.history` path calls `readRecentSessionMessagesAsync`, which returns `[]` when `findExistingTranscriptPath` finds no active candidate.

## Solution Assessment

No. The helper is reasonable, but limiting it to synchronous readers is not the narrow maintainable fix for the advertised `chat.history` and `sessions_history` behavior because those production paths use async readers.
