## Summary

Keep this PR open, but it is not merge-ready: the helper is useful, yet the advertised `chat.history` and `sessions_history` production paths still use async readers that this branch does not route through the reset-archive fallback. The branch also lacks contributor real-behavior proof and now conflicts with current main, while related open PRs cover overlapping archive-recovery choices.

## What This Changes

The PR adds sync reset-archive lookup for session messages and title fields, with tests for newest archive selection, legacy paths, non-canonical transcript names, and stale-candidate exclusion.

## Best Possible Solution

Use one shared active-or-latest-reset archive resolver across sync and async message/title/count paths, or converge this branch with https://github.com/openclaw/openclaw/pull/76134 and https://github.com/openclaw/openclaw/pull/73883, then require redacted real Gateway proof before merge.

## Reproduction Assessment

Yes, at source level: with only a `<sessionId>.jsonl.reset.<timestamp>` archive and no active `<sessionId>.jsonl`, current main's `chat.history` calls `readRecentSessionMessagesAsync`, which returns `[]` when `findExistingTranscriptPath` finds no active candidate. I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

No. The helper is directionally useful, but wiring it only into sync readers does not fix the advertised `chat.history` and `sessions_history` surfaces; the maintainable fix needs the async readers/counts too or explicit coordination with the related archive-history PRs.
