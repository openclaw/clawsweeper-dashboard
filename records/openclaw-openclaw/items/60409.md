## Summary

Keep open: the branch contains useful sync-reader work, but it still does not fix the advertised production `chat.history` and `sessions_history` behavior because those surfaces use async readers that remain active-transcript-only on current main. It also still needs real behavior proof beyond unit/CI evidence before merge.

## What This Changes

Adds reset-archive discovery and wires it into synchronous session message and title-field reads, with focused reset-archive fallback tests.

## Best Possible Solution

Route reset-archive fallback through a shared active-or-latest-reset resolver used by sync and async history, count, and title paths, coordinated with https://github.com/openclaw/openclaw/pull/76134 and backed by redacted real Gateway/session-history proof.

## Reproduction Assessment

Yes, at source level. With only a `<sessionId>.jsonl.reset.<timestamp>` archive and no active `<sessionId>.jsonl`, current main's async `chat.history` path calls `readRecentSessionMessagesAsync`, which returns `[]` when `findExistingTranscriptPath` finds no active candidate.

## Solution Assessment

No. The helper is reasonable for sync callers, but wiring it only into synchronous readers is not the narrow maintainable fix for the advertised `chat.history` and `sessions_history` behavior because those production surfaces use async readers.
