## Summary

Keep open: the branch adds a focused sync fallback, but current main routes the advertised `chat.history` and sessions-history HTTP surfaces through async readers, so archive-only sessions can still return empty there. The nearby open work in https://github.com/openclaw/openclaw/pull/76134 and https://github.com/openclaw/openclaw/pull/73883 covers the two maintainer choices: automatic async fallback versus explicit archived-history opt-in.

## What This Changes

The PR adds reset-archive lookup to synchronous session transcript readers and tests archive-only, active-wins, latest-archive, legacy-dir, non-canonical-name, and stale-candidate cases.

## Best Possible Solution

Converge on one archive-read policy: either move this into a shared active-or-latest-reset implementation used by the async production readers and prove the Gateway path, or defer default recovery to an explicit archived-history API.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main archives reset transcripts, while `chat.history` and sessions-history HTTP call async readers that return empty when `findExistingTranscriptPath` finds no active transcript. I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

No, not as written. The sync fallback is useful, but the narrow maintainable fix must cover the async production readers or intentionally choose the explicit `includeArchived` API path before this branch merges.
