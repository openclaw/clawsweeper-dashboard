## Summary

Keep open, but this PR is not ready to merge: the diff is directionally useful, yet current `chat.history` and `sessions_history` run through async readers that the branch does not change. The related work in https://github.com/openclaw/openclaw/pull/76134 and https://github.com/openclaw/openclaw/pull/73883 is useful context for converging the reset-archive behavior.

## What This Changes

The PR adds reset-archive lookup helpers, wires sync session message/title reads to use the latest `.reset.*` archive when the primary transcript is missing, and adds session-utils tests.

## Best Possible Solution

Converge the sync helper with the async fallback work in https://github.com/openclaw/openclaw/pull/76134 and the explicit archive API direction in https://github.com/openclaw/openclaw/pull/73883 so one archive resolver covers the production Gateway paths with clear reset semantics.

## Reproduction Assessment

Yes, at source level: with only a `<sessionId>.jsonl.reset.<timestamp>` archive and no active `<sessionId>.jsonl`, current main's `chat.history` calls `readRecentSessionMessagesAsync`, which returns `[]` when `findExistingTranscriptPath` finds no active transcript. I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

No. The helper is useful, but wiring it only into sync readers does not fix the advertised `chat.history` and `sessions_history` surfaces; the maintainable fix needs the async readers and title/count paths or coordination with the related archive-history PRs.
