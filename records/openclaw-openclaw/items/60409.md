## Summary

Keep this PR open, but it is not merge-ready: it adds a useful sync reset-archive fallback, yet the advertised `chat.history` and sessions-history production paths on current `main` use async readers that this branch does not change. The related async-path replacement at https://github.com/openclaw/openclaw/pull/76134 now covers the production readers with supplied proof, while https://github.com/openclaw/openclaw/pull/73883 still carries the explicit archived-history product question.

## What This Changes

The PR adds a synchronous latest reset-archive lookup for session transcript reads and title fields, plus focused tests for archive-only, active-wins, latest-archive, legacy-dir, non-canonical-name, and stale-candidate cases.

## Best Possible Solution

Converge on one archive-read policy: either land a shared active-or-latest-reset implementation for the async production readers, or choose the explicit `includeArchived` API path and keep automatic fallback out of default history reads.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current `main` creates `.jsonl.reset.<timestamp>` archives, while `chat.history` and sessions-history HTTP call async readers that return empty when no active transcript exists. I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

No, not as written. The sync fallback is useful, but the narrow maintainable fix must cover the async production readers or intentionally defer to an explicit archived-history API before this branch merges.
