## Summary

Keep open. Current main does not implement the requested Telegram unknown-group action, and this PR still has source-reproducible authorization/topic/cooldown defects plus missing real Telegram proof.

## What This Changes

The PR adds optional Telegram config fields and runtime handling to ignore, warn, or leave unapproved groups, with focused tests.

## Best Possible Solution

Land a revised Telegram preflight implementation that precisely classifies truly unknown groups, shares the action across normal messages and native commands, uses the Telegram thread helper, bounds cooldown state, updates docs/config surfaces, and includes redacted real Telegram proof.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a new feature. The PR defects are source-reproducible from the diff and current-main paths: wildcard-covered groups can be misclassified, native commands bypass the action, General-topic warnings bypass the helper, and cooldown entries never expire.

## Solution Assessment

No. Telegram inbound preflight is the right owner boundary, but this implementation needs shared policy handling, safe topic sends, cooldown lifecycle cleanup, docs/config alignment, and real proof before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Classify only truly unknown groups:** `extensions/telegram/src/unknown-group.ts:11-12`
  - body: `group-policy-allowlist-empty` can describe a wildcard-covered configured group with no sender allowlist entries. With `unknownGroupAction: "leave"`, this can eject the bot from a configured wildcard group instead of preserving the existing sender-auth block.
  - confidence: 0.9
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1647`
  - body: The warn/leave branch only runs in the normal message handler. Native slash commands are registered first and still return the static group-denied response, so `/help` or plugin commands in an unknown group bypass the configured action.
  - confidence: 0.9
- **[P2] Use the thread params helper for warnings:** `extensions/telegram/src/bot-handlers.runtime.ts:1660`
  - body: This send passes `resolvedThreadId` directly. For a forum General topic that can be `1`, which the existing helper deliberately omits because Telegram rejects it, so warn can fail and leave can happen without the configured warning.
  - confidence: 0.92
- **[P3] Evict expired cooldown entries:** `extensions/telegram/src/unknown-group.ts:25`
  - body: `lastActionAt` stores every unknown chat id and never removes entries after the cooldown expires or after a successful leave. A long-running bot exposed to many unknown groups will retain those ids indefinitely.
  - confidence: 0.86
