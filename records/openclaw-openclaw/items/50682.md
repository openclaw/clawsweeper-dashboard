## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Apply unknown-group actions before command handlers:** `extensions/telegram/src/bot-handlers.runtime.ts:1646`
  - body: The new warn/leave logic only runs inside the normal message handler. `createTelegramBotCore` registers native commands first, and `resolveTelegramCommandAuth()` handles `group-chat-not-allowed` by replying and returning, so a first `/help` or plugin command from an unknown group bypasses the configured warn/leave action and leaves the bot present.
  - confidence: 0.9
- **[P1] Do not treat wildcard-covered groups as unknown:** `extensions/telegram/src/unknown-group.ts:10-11`
  - body: `group-policy-allowlist-empty` can occur for groups covered by `groups["*"]` when sender allowlists are empty; current group policy code intentionally does not count wildcard matches as `chatExplicitlyAllowed`. Classifying that reason as an unknown chat means `unknownGroupAction: "leave"` can eject the bot from groups that are covered by configured wildcard policy rather than truly unregistered groups.
  - confidence: 0.82
- **[P2] Use the Telegram thread helper for warning sends:** `extensions/telegram/src/bot-handlers.runtime.ts:1662`
  - body: This path passes `resolvedThreadId` directly to `sendMessage`. For forum General topics that can be `1`, while the existing `buildTelegramThreadParams()` helper intentionally omits `message_thread_id=1` because Telegram rejects message sends with that id, so warn/leave can lose the configured warning in General-topic forums.
  - confidence: 0.88
- **[P2] Document the new Telegram config contract:** `src/config/zod-schema.providers-core.ts:283-285`
  - body: The PR adds three strict public Telegram config fields, but there is no matching docs or config-help update. Operators would have a new security-sensitive `openclaw.json` surface without the normal discoverability and contract text.
  - confidence: 0.76
