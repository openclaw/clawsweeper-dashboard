## Summary

Keep open. Current main does not implement Telegram unknown-group warn/leave handling, and this PR is a useful plugin feature but still has source-reproducible access-control, forum-threading, and release-documentation blockers.

## What This Changes

The PR adds optional Telegram `unknownGroupAction`, `unknownGroupMessage`, and `unknownGroupCooldownMs` config plus regular-message warn/leave handling, cooldown state, and tests.

## Best Possible Solution

Revise the Telegram plugin feature so unknown-chat handling is shared across regular messages and native commands, classifies only truly unlisted chats as unknown, uses existing thread helpers, bounds cooldown state, diagnoses missing warning messages, and adds docs, changelog, and regression tests.

## Reproduction Assessment

Yes. The blocking issues are source-reproducible from the PR diff plus current main: wildcard-only groups can produce `group-policy-allowlist-empty`, slash commands authorize through `resolveTelegramCommandAuth`, and General forum sends must go through `buildTelegramThreadParams()`.

## Solution Assessment

No. The feature belongs in the Telegram plugin, but this implementation is not the narrowest maintainable solution until it reuses existing policy and thread helpers across all inbound command paths and ships the user-facing config documentation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop treating wildcard allowlists as unknown groups:** `extensions/telegram/src/unknown-group.ts:10-11`
  - body: `group-policy-allowlist-empty` can describe a wildcard-configured group where the chat is covered by `groups["*"]` but no sender allowlist is configured. With `unknownGroupAction: "leave"`, this would eject the bot from wildcard-enabled groups rather than only truly unlisted chats; classify only `group-chat-not-allowed` or carry explicit unlisted-chat state.
  - confidence: 0.93
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1646-1650`
  - body: The warn/leave handling is wired into the regular message handler only. Telegram native slash commands authorize through `resolveTelegramCommandAuth`, so `/help` or plugin commands in an unknown group can bypass the configured action and leave the bot present in a group it should leave.
  - confidence: 0.88
- **[P2] Use the thread helper for forum warnings:** `extensions/telegram/src/bot-handlers.runtime.ts:1659-1662`
  - body: The warning send uses the resolved thread id directly. Current `buildTelegramThreadParams()` omits `message_thread_id=1` for General forum topics because Telegram rejects it, so warn/leave can fail to post the configured warning in General topics.
  - confidence: 0.9
- **[P2] Prune expired cooldown entries:** `extensions/telegram/src/unknown-group.ts:16-27`
  - body: The cooldown map records every chat that triggers warn/leave and never removes entries. Long-running bots added to many unknown groups retain one entry per chat after cooldown expiry or after leaving, so this feature can grow memory without bound.
  - confidence: 0.82
- **[P3] Surface warn mode without a message:** `extensions/telegram/src/unknown-group.ts:36-43`
  - body: When `unknownGroupAction` is `"warn"` and no `unknownGroupMessage` is configured, the code records cooldown and sends nothing. Reject or diagnose that config so operators know the configured warning is not visible.
  - confidence: 0.75
- **[P3] Add the required user-facing docs and changelog:** `src/config/types.telegram.ts:203-218`
  - body: This introduces new Telegram config keys and externally visible behavior, but the PR does not update Telegram docs or `CHANGELOG.md`. OpenClaw policy requires user-facing feature/config changes to ship with matching docs and a changelog entry.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or workflow risk, but it changes Telegram access-control behavior and has concrete policy-enforcement gaps.

Concerns:

- **[medium] Native commands bypass unknown-group policy:** `extensions/telegram/src/bot-handlers.runtime.ts:1646`
  - body: Configured warn/leave behavior is added only to the regular message path, while Telegram native commands use a separate auth path that sends fixed rejection responses. Deployments relying on auto-leave from arbitrary groups would still keep the bot in those groups after command activity.
  - confidence: 0.86
- **[medium] Wildcard groups can be auto-left:** `extensions/telegram/src/unknown-group.ts:10`
  - body: Classifying `group-policy-allowlist-empty` as unknown can make the bot warn or leave groups intentionally covered by wildcard group config. That is an availability and config-safety risk in an access-control feature.
  - confidence: 0.9
