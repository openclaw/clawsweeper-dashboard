## Summary

Keep open. The feature addresses the paired request in https://github.com/openclaw/openclaw/issues/50681, but this PR is not safe to merge as-is because the current diff can misapply auto-leave to wildcard-configured groups, misses native command traffic, and lacks inspectable real Telegram proof.

## What This Changes

The PR adds optional Telegram `unknownGroupAction`, `unknownGroupMessage`, and `unknownGroupCooldownMs` config fields plus runtime/test handling for ignore, warn, and leave behavior in unknown groups.

## Best Possible Solution

Keep the paired feature open, but revise it into one shared Telegram unknown-group preflight that distinguishes truly unregistered groups from sender-policy blocks, is reused by native commands and normal messages, uses existing thread-parameter helpers, updates the public config/docs surface, and includes redacted real Telegram proof.

## Reproduction Assessment

Yes, from source inspection. A wildcard `channels.telegram.groups["*"]` config with `groupPolicy: "allowlist"` and no sender allowlist reaches `group-policy-allowlist-empty`, and the PR classifies that as an unknown group; slash commands also stay on the native command auth path before the new normal-message action path runs.

## Solution Assessment

No. The requested feature is useful, but this implementation needs a shared preflight and stricter unknown-group classification before auto-leave can be a maintainable Telegram policy surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Stop treating wildcard-covered groups as unknown:** `extensions/telegram/src/unknown-group.ts:12`
  - body: `group-policy-allowlist-empty` is also returned for wildcard-covered groups when no sender allowlist is configured, so `unknownGroupAction: "leave"` can warn and eject the bot from groups intentionally covered by `groups: {"*": ...}`. Restrict the unknown-group action to a truly unregistered chat such as `group-chat-not-allowed`, or carry an explicit classification from group policy.
  - confidence: 0.9
- **[P2] Use the Telegram thread helper for warning sends:** `extensions/telegram/src/bot-handlers.runtime.ts:1660`
  - body: The new warning send builds `{ message_thread_id: resolvedThreadId }` directly. For General forum topics `resolvedThreadId` can be `1`, and the existing helper intentionally omits that value because Telegram rejects it, so warn can fail before a leave action sends the configured message.
  - confidence: 0.88
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1647`
  - body: The new action is only in the normal message handler, but Telegram native commands are registered ahead of that path and resolve auth inside `resolveTelegramCommandAuth()`. A first `/help` or plugin command in an unknown group still gets the static command denial and never runs the configured ignore/warn/leave behavior.
  - confidence: 0.86
- **[P3] Prune expired unknown-group cooldown entries:** `extensions/telegram/src/unknown-group.ts:32`
  - body: `lastActionAt` only grows as new unknown chat IDs are recorded. Long-running bots added to many groups retain expired entries even after cooldown windows pass or after `leaveChat`, so evict expired entries when checking cooldown or otherwise bound the tracker.
  - confidence: 0.78
