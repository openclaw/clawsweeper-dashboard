Full review comments:

- **[P1] Keep sender-policy blocks out of unknown-group actions:** `extensions/telegram/src/unknown-group.ts:11`
  - body: `group-policy-allowlist-empty` also covers configured wildcard groups that have not authorized the sender. Treating that as an unknown chat means `unknownGroupAction: "leave"` can eject the bot from a group the operator deliberately covered with `groups["*"]`; classify only `group-chat-not-allowed` or carry a stronger chat-unknown signal from `evaluateTelegramGroupPolicyAccess`.
  - confidence: 0.88
- **[P1] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1647`
  - body: The new warn/leave branch only runs in the normal message handler. Native `/help` and plugin commands are handled by `bot.command` before `registerTelegramHandlers`, and `resolveTelegramCommandAuth()` currently sends a fixed denial for `group-chat-not-allowed`, so command updates in unknown groups never warn or leave as configured.
  - confidence: 0.9
- **[P2] Use the Telegram thread helper for warnings:** `extensions/telegram/src/bot-handlers.runtime.ts:1660`
  - body: Sending `{ message_thread_id: resolvedThreadId }` directly bypasses `buildTelegramThreadParams`, which intentionally drops General topic id `1` because Telegram rejects it. In a General-topic forum, `warn` will fail and `leave` will leave without sending the configured warning.
  - confidence: 0.86
- **[P2] Complete the new config contract:** `src/config/zod-schema.providers-core.ts:280-282`
  - body: These new operator-facing Telegram config keys are added to the strict schema, but the PR does not update the surrounding public config contract: docs/help/generated metadata or baseline coverage. Add the matching docs/metadata/test updates so users can discover and validate the new behavior consistently.
  - confidence: 0.78
