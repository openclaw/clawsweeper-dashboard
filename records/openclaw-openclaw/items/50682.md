## Summary

Keep open. Current main still lacks the requested Telegram unknown-group action feature, and this PR remains the paired implementation path, but the patch has blocking authorization/topic/contract issues and no attached after-fix real Telegram proof.

## What This Changes

The PR adds optional Telegram `unknownGroupAction`, `unknownGroupMessage`, and `unknownGroupCooldownMs` config plus ignore, warn, and leave handling in the Telegram message path with focused tests.

## Best Possible Solution

Land a revised Telegram preflight implementation that precisely classifies truly unknown chats, shares the action across normal messages and native commands, uses the Telegram thread helper, bounds cooldown state, updates docs/config surfaces, and includes redacted real Telegram proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a new feature. The blocking PR defects are source-reproducible from the diff and current main: wildcard-covered groups are misclassified, native commands bypass the action, General-topic warnings bypass the helper, and cooldown entries never expire.

## Solution Assessment

No. Telegram inbound preflight is the right owner boundary, but this patch needs shared policy handling, safer topic sends, config-boundary cleanup, cooldown lifecycle cleanup, docs/config alignment, and real proof before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use a narrow config SDK import:** `extensions/telegram/src/unknown-group.ts:1`
  - body: Production extension code is guarded against importing from `openclaw/plugin-sdk/config-runtime`, even for type-only imports. This new file should import `TelegramAccountConfig` from a narrow config contract subpath or the config-boundary guard will flag the PR.
  - confidence: 0.91
- **[P1] Classify only truly unknown groups:** `extensions/telegram/src/unknown-group.ts:12`
  - body: `group-policy-allowlist-empty` can describe a wildcard-covered configured group with no sender allowlist entries, not an unregistered chat. With `unknownGroupAction: "leave"`, this can remove the bot from a configured wildcard group instead of preserving the existing sender-auth block.
  - confidence: 0.9
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1641-1642`
  - body: The warn/leave branch only runs in the normal message handler. Native slash commands are registered separately and still return the static group-denied response, so `/help` or plugin commands in an unknown group bypass the configured action.
  - confidence: 0.9
- **[P2] Use the thread params helper for warnings:** `extensions/telegram/src/bot-handlers.runtime.ts:1657`
  - body: This send passes `resolvedThreadId` directly. For a forum General topic that can be `1`, which the existing helper deliberately omits because Telegram rejects it, so warn can fail and leave can happen without the configured warning.
  - confidence: 0.92
- **[P3] Evict expired cooldown entries:** `extensions/telegram/src/unknown-group.ts:27`
  - body: `lastActionAt` stores every unknown chat id and never removes entries after the cooldown expires or after a successful leave. A long-running bot exposed to many unknown groups will retain those ids indefinitely.
  - confidence: 0.86
