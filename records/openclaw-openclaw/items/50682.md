## Summary

Keep this PR open. Current main still lacks the requested Telegram unknown-group action surface, and the PR is a plausible Telegram-plugin feature, but the proposed patch has blocking access-control and routing defects before it can merge; it is also paired with open issue #50681, so this PR should not be cleaned up independently.

## What This Changes

The PR adds optional Telegram `unknownGroupAction`, `unknownGroupMessage`, and `unknownGroupCooldownMs` config plus runtime ignore/warn/leave handling, cooldown tracking, `leaveChat` test harness wiring, and focused Telegram tests.

## Best Possible Solution

Keep the PR and paired issue open, then land the Telegram feature only after unknown-group enforcement is shared between normal messages and native commands, unknown-chat classification distinguishes unlisted chats from wildcard-configured groups, forum warning sends use the existing thread helper, cooldown storage is bounded, no-message warn mode is diagnosed or rejected, and docs/changelog coverage is added for the new public config.

## Reproduction Assessment

Yes. A code-level reproduction is clear: configure Telegram with `groupPolicy: "allowlist"`, wildcard `groups: {"*": {...}}`, no sender allowlist, and `unknownGroupAction: "leave"`; current group policy can return `group-policy-allowlist-empty`, which the PR classifies as unknown. A slash command in an unlisted group also exercises `resolveTelegramCommandAuth`, which the PR does not change.

## Solution Assessment

No. The feature direction belongs in the Telegram plugin, but this patch is not the best implementation until it uses one enforcement path for regular messages and native commands and reuses the existing group-policy and thread helpers correctly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop treating wildcard allowlists as unknown groups:** `extensions/telegram/src/unknown-group.ts:10`
  - body: `group-policy-allowlist-empty` also covers wildcard `groups: {"*": ...}` where the chat is configured but no sender allowlist exists. With `unknownGroupAction: "leave"`, this can eject the bot from wildcard-enabled groups rather than only unknown groups; classify only truly unlisted chats or carry explicit chat-allowlist state.
  - confidence: 0.93
- **[P2] Use the thread helper for forum warnings:** `extensions/telegram/src/bot-handlers.runtime.ts:1659-1662`
  - body: The new warning send passes `resolvedThreadId` directly. Current `buildTelegramThreadParams()` omits `message_thread_id=1` for General forum topics because Telegram rejects it, so warn/leave can fail to post the configured warning in General topics.
  - confidence: 0.9
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1646-1650`
  - body: Unknown-group handling only runs in the regular message handler. Native slash commands are registered before that handler and authorize inside `resolveTelegramCommandAuth`, so `/help` or plugin commands in an unknown group bypass the configured ignore/warn/leave action.
  - confidence: 0.88
- **[P2] Prune expired cooldown entries:** `extensions/telegram/src/unknown-group.ts:16-27`
  - body: `lastActionAt` is append-only. A bot added to many unique unknown groups retains one map entry per chat after cooldown expiry or after leaving, so the new feature can grow memory without bound in long-running deployments.
  - confidence: 0.82
- **[P3] Surface warn mode without a message:** `extensions/telegram/src/unknown-group.ts:39-43`
  - body: When `unknownGroupAction` is `"warn"` but no message is set, the code records cooldown and sends nothing, making the configured action indistinguishable from ignore. Reject or diagnose that config so operators know the warning is not visible.
  - confidence: 0.75

## Security Review

Status: needs_attention

Summary: The diff does not add dependency, workflow, lockfile, publishing, or third-party code execution risk, but it changes Telegram access-control behavior and has concrete policy-enforcement gaps.

Concerns:

- **[medium] Unknown-group policy bypasses native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1646`
  - body: Configured warn/leave behavior is added only to the regular message handler, while Telegram slash/native commands use a separate auth path that sends fixed rejection responses. Deployments relying on auto-leave from arbitrary groups would still keep the bot in those groups after command activity.
  - confidence: 0.86
- **[medium] Wildcard groups can be auto-left:** `extensions/telegram/src/unknown-group.ts:10`
  - body: Classifying `group-policy-allowlist-empty` as unknown can make the bot warn or leave groups intentionally covered by wildcard group config. This is an availability and config-safety risk in an access-control feature.
  - confidence: 0.9
