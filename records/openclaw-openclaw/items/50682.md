## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Do not treat sender allowlist failures as unknown groups:** `extensions/telegram/src/unknown-group.ts:12`
  - body: Including `group-policy-allowlist-empty` makes this path fire for authorization setup states, not just unknown chats. For example, a `groups: {"*": ...}` config intentionally covers the group while sender filtering still applies, so `unknownGroupAction: "leave"` can eject the bot from a configured group instead of only blocking the sender.
  - confidence: 0.9
- **[P2] Use the Telegram thread helper for warnings:** `extensions/telegram/src/bot-handlers.runtime.ts:1660`
  - body: This sends `message_thread_id` directly from `resolvedThreadId`. Current Telegram helpers intentionally omit id `1` for General forum topics because Telegram rejects it, so `warn` can silently fail there and `leave` can happen without the configured warning.
  - confidence: 0.88
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1644-1645`
  - body: The new action path only runs from the normal message handler, but native slash commands are registered first and authorize groups independently. An unknown group whose first interaction is `/help` or another native command still gets the old static denial and never warns or leaves as configured.
  - confidence: 0.86
- **[P2] Publish the new config surface everywhere:** `src/config/zod-schema.providers-core.ts:283-285`
  - body: The strict Telegram schema accepts three new public fields, but the PR does not update the generated channel config metadata, UI hints, or public Telegram docs. That leaves setup/control surfaces and operator guidance out of sync with the accepted config shape.
  - confidence: 0.78
