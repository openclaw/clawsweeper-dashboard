## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Classify only real unlisted chats as unknown:** `extensions/telegram/src/unknown-group.ts:12`
  - body: `group-policy-allowlist-empty` is also returned when a wildcard group entry exists but no sender allowlist is configured, so `unknownGroupAction: "leave"` can eject the bot from groups covered by `groups: {"*": ...}` rather than truly unknown chats. Tie warn/leave to an explicit chat-allowlist rejection, or carry an explicit unknown-chat result from the policy helper.
  - confidence: 0.91
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1645-1646`
  - body: The new warn/leave branch only runs in the message-like handler, while native slash commands are registered earlier and authorize through `resolveTelegramCommandAuth()`. If the first update in an unknown group is `/status` or another command, the bot sends the old static denial and never performs the configured warn/leave action.
  - confidence: 0.88
- **[P2] Use Telegram thread params for warning sends:** `extensions/telegram/src/bot-handlers.runtime.ts:1660`
  - body: This direct `message_thread_id` construction bypasses `buildTelegramThreadParams()`, which intentionally omits `message_thread_id=1` for General forum-topic sends. In General topics, `warn` can fail and `leave` can leave without delivering the configured warning.
  - confidence: 0.9
- **[P3] Document the new Telegram config keys:** `src/config/types.telegram.ts:213-220`
  - body: These new strict-schema keys are public operator-facing config, but the Telegram docs still only describe group blocking through `groups`, `groupPolicy`, and sender allowlists. Add docs for the warn/leave semantics, cooldown, and security caveats so the config contract is discoverable.
  - confidence: 0.82
