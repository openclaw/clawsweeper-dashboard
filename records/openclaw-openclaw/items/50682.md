## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep sender allowlist misses out of unknown-group actions:** `extensions/telegram/src/unknown-group.ts:12`
  - body: `group-policy-allowlist-empty` also occurs for wildcard-configured groups such as `groups: {"*": ...}` when no sender allowlist is present. With this classification, opting into `unknownGroupAction: "leave"` can make the bot leave a group that is intentionally covered by the group-ID policy instead of treating it as a sender-auth misconfiguration.
  - confidence: 0.89
- **[P1] Share unknown-group actions with native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1645-1648`
  - body: The warn/leave action only runs in the regular message handler. Native slash and plugin commands are registered before that handler and return from `resolveTelegramCommandAuth()` on group policy failures, so `/help` or a plugin command in an unknown group leaves the bot in the chat instead of honoring the configured action.
  - confidence: 0.87
- **[P2] Use the Telegram thread helper for warning sends:** `extensions/telegram/src/bot-handlers.runtime.ts:1657-1661`
  - body: This sends raw `resolvedThreadId` as `message_thread_id`; for a forum's General topic the resolved id is `1`, and the existing Telegram helper explicitly omits that id because Telegram rejects message sends with it. Use `buildTelegramThreadParams` with a forum thread spec before sending the warning.
  - confidence: 0.9
- **[P3] Document the new Telegram config fields:** `src/config/types.telegram.ts:213-220`
  - body: The PR adds three user-facing Telegram config fields but does not update the Telegram channel docs or config reference. Operators need the supported values, defaults, and interaction with `groups`, `groupPolicy`, and `groupAllowFrom` documented before this ships.
  - confidence: 0.78
