## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P1] Do not classify wildcard allowlists as unknown groups:** `extensions/telegram/src/unknown-group.ts:12`
  - body: `group-policy-allowlist-empty` is also returned for wildcard-only `groups: {"*": ...}` when no sender allowlist is configured. With `unknownGroupAction: "leave"`, this can eject the bot from groups intentionally covered by wildcard group config instead of only groups absent from the allowlist.
  - confidence: 0.92
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1647-1668`
  - body: This action path only runs in the normal message handler. Native Telegram slash/plugin commands are registered before that handler and resolve `group-chat-not-allowed` inside command auth with a static denial, so `/help` or plugin commands from an unknown group bypass warn/leave entirely.
  - confidence: 0.9
- **[P2] Reuse the thread-param helper for warning sends:** `extensions/telegram/src/bot-handlers.runtime.ts:1660`
  - body: Passing raw `resolvedThreadId` sends `message_thread_id: 1` for General forum topics, while the existing helper omits id 1 because Telegram rejects it. In that topic, `warn` fails and `leave` can leave without the configured warning.
  - confidence: 0.88
- **[P2] Align the generated config and docs surfaces:** `src/config/zod-schema.providers-core.ts:280-282`
  - body: Adding Telegram config fields only to the type/schema leaves generated channel metadata and config docs baselines out of sync with the user-visible surface. Update the generated metadata/docs baseline and public Telegram docs with the new options and validation expectations.
  - confidence: 0.8
