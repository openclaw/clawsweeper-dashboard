## Summary

Keep open. This is a plausible paired implementation for the open Telegram unknown-group request, but the patch has blocking correctness and security gaps around wildcard group classification, General-topic warning sends, native slash commands, and it lacks required real Telegram proof.

## What This Changes

The PR adds optional Telegram unknown-group action/message/cooldown config, a cooldown helper, normal-message warn/leave handling, and Telegram tests.

## Best Possible Solution

Revise the Telegram preflight design so a precise unknown-group action is shared by normal messages and native commands, uses the existing thread helper for topic sends, updates config/docs/metadata, and ships with redacted live Telegram proof.

## Reproduction Assessment

Yes for the review findings by source inspection: current tests/source establish the wildcard allowlist and General-topic cases, and current startup/auth code shows native commands bypass the new handler-only action path. I did not run live Telegram during this read-only review.

## Solution Assessment

No. The handler-only implementation is not the best maintainable fix because it conflates access-denial reasons, bypasses native commands, and hand-builds topic params instead of reusing the established Telegram helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Only leave chats that are truly unknown:** `extensions/telegram/src/unknown-group.ts:12`
  - body: Please do not classify `group-policy-allowlist-empty` as an unknown chat. Current group access uses that reason when a wildcard `groups["*"]` entry matched but no sender allowlist is configured, so `unknownGroupAction: "leave"` would eject the bot from a group that is intentionally covered by config. Carry an explicit chat-not-allowed reason, or otherwise distinguish absent group entries from wildcard/sender-allowlist failures before warning or leaving.
  - confidence: 0.94
- **[P2] Build warning topic params with the Telegram helper:** `extensions/telegram/src/bot-handlers.runtime.ts:1660`
  - body: Route the warning send through `buildTelegramThreadParams` or equivalent. `resolvedThreadId` is `1` for the General forum topic, and existing Telegram helpers intentionally omit `message_thread_id=1` because Telegram rejects sends with that value; this direct params object will 400 before the configured warning is posted.
  - confidence: 0.9
- **[P2] Apply unknown-group actions to native commands:** `src/config/types.telegram.ts:203-211`
  - body: The new config promises an action for unregistered-group messages, but slash-command messages are consumed by native command middleware before `registerTelegramHandlers` and still return the hard-coded group denial. A first `/help` or plugin command in an unknown group will bypass warn/leave, leaving the bot in the chat despite `unknownGroupAction: "leave"`.
  - confidence: 0.89
- **[P3] Prune expired unknown-group cooldown entries:** `extensions/telegram/src/unknown-group.ts:17-27`
  - body: The cooldown map records every unknown chat that triggers warn/leave and never deletes entries. A long-running bot added to many unrelated groups retains those chat IDs forever after the cooldown expires or after leaving; delete expired entries during cooldown checks or add a bounded eviction policy.
  - confidence: 0.82
