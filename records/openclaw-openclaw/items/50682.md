## Summary

Keep open. The PR implements a useful Telegram feature that current main does not have, but the patch still has blocking Telegram policy gaps and lacks required after-fix real Telegram proof.

## What This Changes

The PR adds Telegram config, runtime, and tests for unknown-group ignore, warn, and leave actions before assistant execution.

## Best Possible Solution

Land a revised Telegram preflight implementation that classifies only genuinely unregistered groups, shares the action across normal messages and native commands, uses the existing thread helper, bounds cooldown state, updates public docs, and includes redacted real Telegram proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a new Telegram feature. The PR defects are source-reproducible from the diff and current-main paths: sender-allowlist misses are misclassified, slash commands bypass the action, General-topic warnings bypass the helper, and cooldown entries are never evicted.

## Solution Assessment

No. Telegram inbound preflight is the right owner boundary, but this implementation is not yet the narrow maintainable solution until the shared policy path, topic sends, cooldown lifecycle, docs, and real proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Stop treating sender-auth misses as unknown groups:** `extensions/telegram/src/unknown-group.ts:11`
  - body: `group-policy-allowlist-empty` is used for wildcard-matched groups that are configured but lack sender allowlist entries. Treating that as unknown means `warn` or `leave` can run in a configured wildcard group instead of preserving the existing sender-auth block.
  - confidence: 0.9
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1644-1646`
  - body: The new action branch only runs in the normal message handler. Native Telegram commands are registered before that handler and still return the static group-denied response, so `/help` or plugin commands in an unknown group never warn or leave as configured.
  - confidence: 0.9
- **[P2] Use the Telegram thread helper for warning sends:** `extensions/telegram/src/bot-handlers.runtime.ts:1657-1661`
  - body: This path sends `message_thread_id` directly from `resolvedThreadId`. For a forum General topic that can be `1`, which the existing helper deliberately omits because Telegram rejects it, so warn can fail and leave can happen without the configured warning.
  - confidence: 0.92
- **[P3] Evict expired cooldown entries:** `extensions/telegram/src/unknown-group.ts:24-27`
  - body: `lastActionAt` grows for every unknown chat id and entries are never removed after the cooldown expires or after a successful leave. A long-running bot exposed to many unknown groups will retain those ids indefinitely.
  - confidence: 0.86
