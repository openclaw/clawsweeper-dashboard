## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the narrow config contract import:** `extensions/telegram/src/unknown-group.ts:1`
  - body: `unknown-group.ts` imports a type from the broad `openclaw/plugin-sdk/config-runtime` barrel. Current boundary tests flag that exact pattern for extension production code, so this should use the narrow config contract/type subpath used by the sibling Telegram modules instead.
  - confidence: 0.96
- **[P1] Do not auto-leave wildcard-matched groups:** `extensions/telegram/src/unknown-group.ts:11`
  - body: `group-policy-allowlist-empty` is not the same as an unknown chat: current tests cover the case where `groups["*"]` matches but no sender allowlist is configured. With this classification, `unknownGroupAction: "leave"` can eject the bot from a group intentionally covered by wildcard config, so the unknown action needs more precise context or should only treat `group-chat-not-allowed` as unknown.
  - confidence: 0.9
- **[P2] Route warning sends through the thread helper:** `extensions/telegram/src/bot-handlers.runtime.ts:1662`
  - body: This sends `{ message_thread_id: resolvedThreadId }` directly. For forum General topics, `resolvedThreadId` can be `1`, and current `buildTelegramThreadParams()` intentionally omits that value because Telegram rejects message sends with `message_thread_id=1`; use the helper so `warn` does not fail before a `leave`.
  - confidence: 0.88
- **[P2] Apply unknown-group actions to native commands:** `src/config/zod-schema.providers-core.ts:280`
  - body: Adding `unknownGroupAction` creates a config contract for unknown group handling, but native slash commands are registered before the normal message handlers and still send the fixed `This group is not allowed.` denial from native command auth. If the first update in an unknown group is `/help` or another native/plugin command, the configured warn/leave action is bypassed.
  - confidence: 0.86
