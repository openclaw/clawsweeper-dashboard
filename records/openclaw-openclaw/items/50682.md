## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not treat sender-policy misses as unknown chats:** `extensions/telegram/src/unknown-group.ts:12`
  - body: `group-policy-allowlist-empty` is also returned for wildcard-configured groups with no sender allowlist, which current tests preserve as a sender-authorization block. Classifying it as an unknown chat means `unknownGroupAction: "leave"` can eject the bot from a group that was intentionally covered by `groups: { "*": ... }`, so this helper needs a narrower chat-unknown signal before it drives warn/leave behavior.
  - confidence: 0.88
- **[P2] Reuse the Telegram thread-param helper for warnings:** `extensions/telegram/src/bot-handlers.runtime.ts:1660`
  - body: This send path passes `resolvedThreadId` directly as `message_thread_id`, but existing Telegram helpers deliberately omit General forum topic id `1` because Telegram rejects message sends with that value. In a General topic, the configured warning can fail and the PR will leave the group without the promised warning.
  - confidence: 0.86
- **[P2] Apply unknown-group actions before native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1644-1646`
  - body: The new warn/leave logic only runs in the normal message handler, while Telegram native command handlers are registered first and own slash-command auth failures. If the first unknown-group activity is `/help` or another native/plugin command, the bot can reply with the old auth message and stay in the group instead of following `unknownGroupAction`.
  - confidence: 0.84
- **[P3] Prune expired unknown-group cooldown entries:** `extensions/telegram/src/unknown-group.ts:27`
  - body: `lastActionAt` never removes entries, so a long-running bot exposed to many unknown group IDs can retain one map entry per chat forever even after cooldown expiry or successful leave. Evict expired entries when checking cooldown, or otherwise bound the tracker.
  - confidence: 0.76
