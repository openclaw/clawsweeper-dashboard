## Summary

Keep open. Current main still lacks the requested Telegram unknown-group warn/leave feature, and this PR is the paired implementation path, but the patch has blocking authorization/topic handling defects and no attached real Telegram proof.

## What This Changes

The PR adds Telegram config fields plus runtime and tests for ignore, warn, and leave actions when messages arrive from unknown groups.

## Best Possible Solution

Land a revised Telegram preflight implementation that classifies only genuinely unknown groups, shares the action across normal messages and native commands, uses the Telegram thread helper, bounds cooldown state, updates docs/config surfaces, and includes redacted real Telegram proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a new feature. The PR defects are source-reproducible from the diff and current-main paths: sender-allowlist misses are misclassified, native commands bypass the action, General-topic warnings bypass the helper, and cooldown entries never expire.

## Solution Assessment

No. Telegram inbound preflight is the right owner boundary, but this implementation is not yet the narrow maintainable solution until shared policy handling, topic sends, cooldown lifecycle, docs, and real proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Stop treating sender-auth misses as unknown groups:** `extensions/telegram/src/unknown-group.ts:11`
  - body: `group-policy-allowlist-empty` is used for wildcard-covered groups that lack sender allowlist entries, not only for unknown chats. With `unknownGroupAction: "leave"`, this can eject the bot from a configured wildcard group instead of preserving the existing sender-auth block.
  - confidence: 0.9
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1644-1646`
  - body: The new warn/leave branch only runs in the normal message handler. Native slash commands are registered before that handler and still return the static group-denied response, so `/help` or plugin commands in an unknown group bypass the configured action.
  - confidence: 0.9
- **[P2] Use the Telegram thread helper for warning sends:** `extensions/telegram/src/bot-handlers.runtime.ts:1657-1661`
  - body: This path passes `resolvedThreadId` directly. For a forum General topic, that can be `1`, which the existing helper deliberately omits because Telegram rejects it, so warn can fail and leave can happen without the configured warning.
  - confidence: 0.92
- **[P3] Evict expired cooldown entries:** `extensions/telegram/src/unknown-group.ts:24-27`
  - body: `lastActionAt` stores every unknown chat id and never removes entries after the cooldown expires or after a successful leave. A long-running bot exposed to many unknown groups will retain those ids indefinitely.
  - confidence: 0.86
