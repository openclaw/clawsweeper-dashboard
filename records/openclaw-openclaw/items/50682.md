## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Constrain unknown-group classification before leaving:** `extensions/telegram/src/unknown-group.ts:11`
  - body: `group-policy-allowlist-empty` is not always an unknown group: current `group-access.ts` treats wildcard `groups["*"]` matches as configured-but-not-explicit so sender filtering still applies. Classifying that reason as unknown means `unknownGroupAction: "leave"` can remove the bot from a group intentionally covered by wildcard config instead of only from truly unregistered chats.
  - confidence: 0.91
- **[P2] Apply unknown-group actions to native commands:** `extensions/telegram/src/bot-handlers.runtime.ts:1644`
  - body: The new warn/leave handling only runs inside the normal message handler, but Telegram native commands are registered first and authorize through `resolveTelegramCommandAuth()`, which currently sends a static denial for `group-chat-not-allowed`. A `/help` or plugin command in an unknown group will bypass the configured action and keep the bot in the group.
  - confidence: 0.88
- **[P2] Use the Telegram thread helper for warning sends:** `extensions/telegram/src/bot-handlers.runtime.ts:1660`
  - body: This sends raw `message_thread_id` when `resolvedThreadId` is set, but the existing helper intentionally omits forum General-topic id `1` because Telegram rejects it for sends. In a General-topic forum, warn will 400 and leave mode can leave without posting the configured warning.
  - confidence: 0.87
- **[P3] Evict expired unknown-group cooldown entries:** `extensions/telegram/src/unknown-group.ts:16-32`
  - body: `lastActionAt` only grows and never deletes expired chat ids. A long-running bot that is added to many unknown groups can accumulate stale cooldown entries even after the bot leaves those chats, so the tracker should prune expired entries or otherwise bound the map.
  - confidence: 0.82
