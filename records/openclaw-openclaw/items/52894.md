## Summary

Keep open. Current main and v2026.5.18 still keep skill-driven Homebrew setup manual, while automatic Homebrew bootstrapping from onboarding or gateway-backed skill installs needs maintainer product/security direction before a fix is safe.

## What This Changes

Automatically install Homebrew when a brew-backed skill install is selected and `brew` is missing, instead of only showing the install command.

## Best Possible Solution

The best end state is a maintainer-approved, explicit-consent Homebrew bootstrap path shared by interactive onboarding and gateway-backed skill installs; otherwise keep the manual requirement and improve the guidance.

## Reproduction Assessment

Yes for the current feature gap: source inspection shows that selecting a brew-backed missing skill with no `brew` only displays the Homebrew install command and then calls the existing skill installer. This is not a bug against current docs, which explicitly say skill installs do not auto-install Homebrew.

## Solution Assessment

Unclear. Automatic Homebrew installation would reduce onboarding friction, but it is only the best solution if maintainers approve the remote-installer trust, consent, and platform policy; otherwise clearer manual guidance is the safer direction.

## Risks / Open Questions

- Automatically installing Homebrew from skill onboarding or gateway-backed installs would execute a remote installer and may require elevated privileges, so consent, provenance, platform scope, logging, timeout, and failure behavior are unresolved.
- The standalone `install.sh` macOS path already bootstraps Homebrew, but skill onboarding and gateway-backed installs explicitly do not; changing only one skill path could leave inconsistent operator expectations.
- Adjacent Linux/Docker skill-install feedback in https://github.com/openclaw/openclaw/issues/57555 may constrain the final design, especially for container environments where docs currently require custom images or manual dependencies.
