## Summary

Keep open: current main and v2026.6.8 still document and resolve Browserbase as a static `cdpUrl`, while the remaining session-API work is covered by an open implementation candidate with product, credential, and security-review blockers.

## What This Changes

The issue asks OpenClaw to replace the documented static Browserbase CDP URL flow with Browserbase session API-backed `connectUrl` creation, refresh, and release handling.

## Best Possible Solution

Land a maintainer-approved Browserbase session lifecycle design in the browser plugin with redacted credential handling, clear session release semantics, docs, regression tests, and live Browserbase proof; use https://github.com/openclaw/openclaw/pull/83306 and https://github.com/openclaw/openclaw/issues/78337 as review context.

## Reproduction Assessment

Yes for source-level reproduction: current docs still prescribe a static Browserbase `cdpUrl`, and current runtime passes that resolved URL unchanged into CDP readiness and attach paths. A full live OpenClaw failure still needs a real Browserbase key and live browser-tool run.

## Solution Assessment

No, the current static `cdpUrl` path is not the best remaining solution. A Browserbase session-aware driver or equivalent owner-approved plugin/runtime seam is the maintainable direction, but the open PR still needs route, redaction, credential-surface, and proof work before it can close this issue.
