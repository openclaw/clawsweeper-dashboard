## Summary

Keep open: current main and the latest release still document Browserbase's static WebSocket CDP URL and still use the generic CDP attach path with no Browserbase Sessions API lifecycle. The exact 404 was not live-reproduced, and related user context reports the static URL can still work, so this needs browser/product owner judgment rather than cleanup close.

## What This Changes

Request Browserbase browser profiles to use Browserbase session creation and returned connect URLs instead of relying only on the documented static cdpUrl.

## Best Possible Solution

Define the Browserbase browser-profile contract, then either implement explicit session creation/cleanup with redaction-safe credentials or update docs to require a user-supplied Browserbase-created connectUrl.

## Reproduction Assessment

No high-confidence live reproduction of the reported 404 exists because this review did not have a Browserbase API key. Source inspection does show current main lacks any Sessions API flow and still preserves the static cdpUrl path.

## Solution Assessment

Unclear. The proposed `browserbase.com` URL detection is brittle; an explicit Browserbase profile/session contract or a docs-only connectUrl requirement would be more maintainable.
