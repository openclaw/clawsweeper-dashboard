## Summary

Keep open: current main and v2026.6.6 still document and preserve the static Browserbase `cdpUrl`, while Browserbase’s current API docs show session creation via `POST /v1/sessions` returning `connectUrl`; the linked Browserbase driver PR remains open, conflicting, and review-blocked rather than a safe close target. ([docs.browserbase.com](https://docs.browserbase.com/reference/api/create-a-session?utm_source=openai))

## What This Changes

Browserbase browser profiles should stop relying on the documented static `cdpUrl` and resolve Browserbase session `connectUrl` values before CDP attach.

## Best Possible Solution

Land a maintainer-approved Browserbase session lifecycle path in the browser plugin that resolves fresh `connectUrl` values through an approved credential surface, updates docs/schema/tests, and closes this issue after the fix merges.

## Reproduction Assessment

Yes by source inspection for the missing OpenClaw behavior: current main still resolves Browserbase as a static `cdpUrl` and lacks a session API `connectUrl` path. I did not perform a credentialed Browserbase live repro with a real API key in this read-only review.

## Solution Assessment

No, not as a narrow automatic fix: the reported session API direction is valid, but the best implementation needs maintainer approval for the Browserbase credential/session contract and should be coordinated with the active Browserbase driver PR or its successor.
