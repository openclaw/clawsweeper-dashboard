## Summary

Keep open. Current main still advertises and routes Browserbase as a static generic WebSocket CDP profile, while Browserbase’s current docs describe creating a session through the Sessions API and using the returned connection URL. ([docs.browserbase.com](https://docs.browserbase.com/reference/api/create-a-session?utm_source=openai))

## What This Changes

Request Browserbase-aware browser profile handling that creates a Browserbase session, uses the returned CDP connect URL, and releases the session instead of relying on the documented static WebSocket URL.

## Best Possible Solution

Restore the documented Browserbase integration in the browser plugin: create Browserbase sessions through the official API, connect to the returned URL, release only OpenClaw-created sessions, and update docs/tests/changelog while preserving generic remote CDP behavior.

## Reproduction Assessment

Yes, at source level. Current main still documents and resolves Browserbase as a static generic `wss://connect.browserbase.com` CDP URL, and the browser runtime has no Browserbase Sessions API path; I did not run a live Browserbase-key command.

## Solution Assessment

Yes. The narrow maintainable fix is Browserbase session lifecycle support inside the browser plugin, not a substring-only fallback in generic CDP helpers or core-wide provider policy.
