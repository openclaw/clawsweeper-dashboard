## Summary

Keep open: current main and v2026.6.6 still document and use the static Browserbase `cdpUrl` path with no Browserbase session API handling, and the related fix PR is open, conflicting, and still needs maintainer direction.

## What This Changes

The issue asks OpenClaw's browser profile integration to stop relying on static Browserbase `cdpUrl` values and create or resolve Browserbase session `connectUrl` values through the Sessions API.

## Best Possible Solution

Keep this issue as the canonical Browserbase session-lifecycle bug, then land a revised Browserbase driver or update the docs to a supported workflow after maintainers settle the session, credential, and upgrade contract.

## Reproduction Assessment

Yes, by source inspection and the current Browserbase API contract: OpenClaw still documents and routes a static `cdpUrl`, while Browserbase now documents `POST /v1/sessions` returning `connectUrl`. I did not run a credentialed Browserbase session in this read-only review.

## Solution Assessment

No, not yet: Browserbase session lifecycle handling is the right problem, but the exact driver/config/credential shape needs maintainer approval. The active PR is a plausible candidate, but it is not merged or merge-ready.
