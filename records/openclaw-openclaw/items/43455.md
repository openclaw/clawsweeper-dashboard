## Summary

Keep open: current main still documents and preserves the static Browserbase WebSocket CDP URL and still uses the generic CDP attach path, with no Browserbase Sessions API lifecycle. The exact reported 404 was not live-reproduced and Browserbase docs remain mixed, so this needs browser/product owner judgment rather than cleanup close.

## What This Changes

The issue asks OpenClaw to stop relying on a static Browserbase `cdpUrl` and create/manage Browserbase browser sessions through the Browserbase Sessions API.

## Best Possible Solution

Define the Browserbase browser-profile contract, then either implement explicit session creation/cleanup with redaction-safe credentials or change the docs to require users to supply a pre-created `connectUrl`.

## Reproduction Assessment

No, not for the exact 404: this review did not have a Browserbase key for live validation. Source inspection does show current main lacks the Sessions API flow and still follows the static CDP URL path.

## Solution Assessment

Unclear. Detecting `browserbase.com` inside arbitrary `cdpUrl` values is brittle; an explicit provider/profile session contract or a docs-only pre-created `connectUrl` requirement would be more maintainable.
