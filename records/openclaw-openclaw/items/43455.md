## Summary

Keep open: current main still documents the static Browserbase `wss://connect.browserbase.com?apiKey=...` profile and still only runs the generic CDP attach path. Browserbase's current Playwright/API docs emphasize creating a session and connecting to the returned `connectUrl`, but the reported 404 was not live-reproduced and related user context says the static URL can still work, so this needs browser/product owner judgment rather than cleanup close.

## What This Changes

The issue asks OpenClaw to replace the documented static Browserbase CDP URL flow with Browserbase session creation, `connectUrl` attachment, and session cleanup.

## Best Possible Solution

Define the Browserbase browser-profile contract, then either implement explicit session creation/cleanup with redaction-safe credentials or change OpenClaw docs to require users to provide a Browserbase-created `connectUrl`.

## Reproduction Assessment

No high-confidence live reproduction of the reported 404 exists because this review did not have a Browserbase API key. Source inspection does show current main lacks the Sessions API flow, but related user context says the documented static URL may still work when formatted exactly.

## Solution Assessment

Unclear. The proposed `browserbase.com` URL detection is brittle; an explicit browser profile/provider session contract or a docs-only `connectUrl` requirement would be more maintainable.
