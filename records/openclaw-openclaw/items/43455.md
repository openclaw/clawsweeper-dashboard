## Summary

Keep open. Current main still documents a static Browserbase CDP URL and routes it through generic CDP discovery/direct attach, while Browserbase now documents creating a session first and connecting with the returned session URL.

## What This Changes

The issue asks OpenClaw to replace the documented static Browserbase `cdpUrl` flow with Browserbase Sessions API lifecycle handling.

## Best Possible Solution

Add explicit Browserbase session handling in the browser plugin: create sessions through Browserbase's API before CDP attach, connect to `connectUrl`, clean up only OpenClaw-created sessions appropriately, and update docs/tests/changelog while preserving generic remote CDP behavior.

## Reproduction Assessment

Yes. Current main still resolves the documented Browserbase profile to the static connect URL, that URL returns HTTP 404 in a safe probe, and the runtime has no Sessions API path before `connectOverCDP()`; I did not run a live Browserbase-key OpenClaw command.

## Solution Assessment

Yes. The narrow maintainable fix is explicit Browserbase session support in the browser plugin, not a generic substring-only fallback inside arbitrary CDP URLs.
