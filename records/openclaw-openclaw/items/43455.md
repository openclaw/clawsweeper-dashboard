## Summary

Keep open. Current main still documents the static Browserbase `cdpUrl` and routes it through generic CDP discovery/direct attach, while Browserbase’s current docs describe creating a session with `POST /v1/sessions`, using the returned `connectUrl`, and releasing sessions through the session update API. 

## What This Changes

The issue requests Browserbase-aware browser session lifecycle support because the documented static Browserbase CDP URL now returns 404.

## Best Possible Solution

Add explicit Browserbase provider/session handling inside the browser plugin: accept safe credential and session options, create a Browserbase session before CDP attach, connect Playwright to the returned `connectUrl`, release only OpenClaw-created sessions when appropriate, and update the docs/schema/help while preserving generic remote CDP behavior for other providers.

## Reproduction Assessment

Yes. The documented Browserbase profile still resolves to the static `wss://connect.browserbase.com` URL on current main, that URL returns HTTP 404 in a credential-free probe, and the browser runtime has no REST create-session path before `connectOverCDP()`.

## Solution Assessment

Yes, the Sessions API direction is the narrow maintainable fix, but it should be explicit Browserbase profile/session support rather than only substring-detecting `browserbase.com` inside arbitrary CDP URLs.
