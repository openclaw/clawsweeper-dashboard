## Summary

Keep open. Current main still documents Browserbase as a static WebSocket CDP profile and still routes it through the generic CDP attach path, while Browserbase's current docs center session creation through the Sessions API and a returned connection URL.

## What This Changes

The issue asks OpenClaw to replace the documented static Browserbase CDP URL flow with automatic Browserbase session creation and attachment to the returned connectUrl.

## Best Possible Solution

Add Browserbase-owned session handling in the browser plugin: create a session before CDP attach, connect to the returned `connectUrl`, release only OpenClaw-created sessions, and update docs/tests while preserving generic remote CDP behavior.

## Reproduction Assessment

Yes, at source level for the OpenClaw gap: the documented Browserbase profile resolves to a static WebSocket URL and reaches generic CDP attach with no Sessions API path. The exact Browserbase 404 was not live-reproduced because no Browserbase credential was present, and one related report says the static URL worked on 2026.5.4.

## Solution Assessment

Yes. The narrow maintainable fix is explicit Browserbase session lifecycle support in the browser plugin, not a generic core fallback or broad CDP helper rewrite.
