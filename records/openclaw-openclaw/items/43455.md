## Summary

Current main and the latest release still expose only static remote-CDP configuration, while Browserbase’s supported connection flow requires a session-derived connection URL. The linked implementation PR remains open and conflicting, and it proposes persistent keep-alive semantics rather than the issue’s ephemeral auto-create/auto-close lifecycle, so the issue remains necessary pending a maintainer lifecycle and credential-contract decision.

## What This Changes

Add a Browserbase-aware browser profile path that creates or resolves session connection URLs and manages the intended session lifecycle.

## Best Possible Solution

Define one canonical Browserbase profile contract in the browser plugin with an explicit ephemeral-or-persistent mode, supported secret resolution, deterministic release for OpenClaw-owned sessions, clear stale-session behavior, and live proof covering creation, connection, repeated attachment when persistent, and cleanup.

## Reproduction Assessment

Yes at the contract and source level: current main only consumes a configured CDP URL, while Browserbase’s supported session flow returns connection data dynamically. A credentialed live Browserbase run is still required before accepting an implementation.

## Solution Assessment

No, the issue’s proposed hostname detection and URL-secret extraction should not be adopted as-is; Sessions API integration is needed, but maintainers must first choose explicit ephemeral versus persistent ownership and a canonical credential/config surface.
