## Summary

## What This Changes

The issue reports that the documented Browserbase static CDP URL no longer works and asks OpenClaw to create or resolve Browserbase sessions before attaching.

## Best Possible Solution

Land a maintainer-approved Browserbase profile/session lifecycle path in the browser plugin, with explicit credential and session config, SecretRef/redaction treatment, docs, regression tests, and redacted end-to-end Browserbase proof; use https://github.com/openclaw/openclaw/pull/83306 or a successor only after its blockers are resolved.

## Reproduction Assessment

Yes, source-level: current main documents and preserves a static Browserbase `cdpUrl`, while Browserbase’s current Sessions API documents creating or retrieving sessions with a returned `connectUrl`. I did not run a credentialed Browserbase/OpenClaw live flow in this read-only review. ([docs.browserbase.com](https://docs.browserbase.com/reference/api/create-a-session))

## Solution Assessment

No, not exactly as proposed: detecting `browserbase.com` inside `cdpUrl` would be brittle for a credentialed lifecycle feature. The better shape is an explicit Browserbase profile/session contract, with maintainers deciding auto-create versus keep-alive semantics and the credential surface before implementation lands.
