## Summary

Keep open: the patch targets the Telegram reasoning-lane buffer, but it does not solve the linked stale-turn path because the generation it compares only advances for authorized abort commands, while ordinary newer Telegram messages and background completion delivery still lack a turn guard.

## What This Changes

The PR adds a buffered final-answer generation field and passes the Telegram abort-fence generation when buffering and flushing reasoning-lane final answers.

## Best Possible Solution

Add turn-bound delivery semantics for Telegram and background completion paths so late work is either suppressed, rebound to its original turn, or delivered as an explicit proactive update, with regression coverage for the reported interleaving.

## Reproduction Assessment

Yes for source-level reproduction: current main routes sub-agent completion events to an active Telegram requester or direct fallback without a per-inbound turn guard, and Telegram's fence only advances for abort commands. I did not run a live Telegram E2E.

## Solution Assessment

No. The PR's generation check is too narrow because it uses `abortFenceGeneration`, which does not change for ordinary newer user messages or background completion delivery.
