## What This Changes

The issue reports a core inbound-dispatch race where same-thread follow-up messages are queued after run completion but never drained until stuck-session recovery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92284, or a narrow equivalent replacement with real race proof, then close this issue as implemented.

## Reproduction Assessment

No live current-main reproduction was run. The current source still exposes the reported race shape, and the discussion includes Slack and WhatsApp logs on shipped releases showing `queued_work_without_active_run` after terminal run progress.

## Solution Assessment

Yes, the best fix belongs in the queue enqueue/drain boundary: enqueue should kick an idle drain when it adds work, while the queued-followup admission path still prevents concurrent reply runs.
