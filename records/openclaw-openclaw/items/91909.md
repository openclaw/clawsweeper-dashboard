## Summary

Keep open. Current main and the latest release still have the reported enqueue-followup shape: the active-run path enqueues with idle restart disabled and only starts a drain when one immediate liveness recheck says the previous run is no longer active, so the terminal-window race remains a plausible source-backed bug with no canonical merged fix.

## What This Changes

The issue requests that queued inbound messages arriving just after a run completes reliably start a new follow-up run instead of waiting for stuck-session recovery.

## Best Possible Solution

Make the generic follow-up queue and active-run lifecycle guarantee that queued work cannot remain idle after active-run teardown, with a regression test for the terminal-window race rather than a Slack-specific workaround.
