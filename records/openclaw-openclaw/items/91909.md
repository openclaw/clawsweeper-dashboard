## Summary

Current main still contains the reported vulnerable decision: the enqueue-followup path disables enqueue-side restart, then relies on a one-time active-run snapshot to decide whether to schedule a drain. Existing idle-restart tests do not cover this caller because it passes `restartIfIdle=false`, so this issue remains necessary and is a narrow repair candidate.

## What This Changes

Fix the inbound follow-up dispatch race that can leave queued work without an active run after the previous run completes.

## Best Possible Solution

Make successful enqueue-followup dispatch guarantee that queued work has either an active drain owner or a newly scheduled drain, without relying on a non-atomic `isRunActive` snapshot, and add deterministic caller-level coverage for enqueue during terminal run cleanup.

## Reproduction Assessment

Yes, at source level with high confidence: enqueue while the prior run still reports active, after its final drain opportunity, causes the current caller to disable enqueue restart and skip explicit scheduling. A live timing reproduction on current main remains useful but is not required to identify the missing drain guarantee.

## Solution Assessment

No, the current snapshot-and-defer approach is not the best solution because it splits drain ownership between enqueue and run completion without atomic coordination. The queue/enqueue boundary should own the invariant, with caller-level race coverage preventing duplicate drains.
