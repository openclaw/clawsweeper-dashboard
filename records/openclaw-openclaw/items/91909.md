## What This Changes

The issue reports a race where inbound follow-up work is queued after a run completes but no new run starts, leaving the agent silent until stuck-session recovery intervenes.

## Best Possible Solution

Land or replace the linked closing PR with a fix that guarantees queued follow-up work starts draining when the queue is idle, with coverage or live proof for the post-completion inbound race.

## Reproduction Assessment

Yes at source level: enqueue a follow-up during the terminal active-run window so `isRunActive()` still returns true while the prior completion drain has already passed, leaving queued work without an idle restart. I did not run a live Slack or WhatsApp timing reproduction in this read-only review.

## Solution Assessment

Yes, the best fix direction is the narrow queue-invariant repair already proposed by the linked PR or an equivalent change: when enqueue leaves work in an idle follow-up queue, kick the drain from the enqueue path instead of relying on a stale active-run snapshot.
