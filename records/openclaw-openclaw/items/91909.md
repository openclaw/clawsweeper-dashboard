## Summary

Current main still contains the exact non-atomic decision described: enqueue disables the queue’s idle restart, then a single `isRunActive()` snapshot decides whether to schedule a drain. Existing restart tests cover queue deletion/recreation, not the terminal-run clearing interleaving, so this remains a credible narrow runtime bug rather than an implemented or stale report.

## What This Changes

Fix a race where a same-session inbound follow-up can remain queued without an active run until watchdog recovery.

## Best Possible Solution

Make the runner and queue share one atomic ownership rule: whenever follow-up work is nonempty after the active run clears, exactly one drain is guaranteed, with focused coverage for enqueue during terminal clearing and for no double-drain.

## Reproduction Assessment

No live current-main reproduction was run, but yes at the source level: the exact branch can enqueue with idle restart disabled and decline to schedule based on a non-atomic active-run snapshot, producing the diagnostic state named in the report.

## Solution Assessment

Yes in principle, but the best fix is an ownership invariant rather than merely flipping a boolean: guarantee exactly one drain after terminal clearing and retain protection against concurrent double-drain.
