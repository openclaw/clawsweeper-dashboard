## Summary

The bug remains present in current main and was independently reproduced on the latest v2026.6.6 release across Slack and WhatsApp. The linked fix PR remains open, so this issue should stay open until that implementation is proven and merged or replaced.

## What This Changes

Fix a race where inbound follow-up messages remain queued without an active drain after the previous agent run completes.

## Best Possible Solution

Land or replace the linked PR with a focused fix that guarantees an idle drain whenever follow-up work exists, adds deterministic terminal-window race coverage, and verifies stale active-work ownership clears after run completion.

## Reproduction Assessment

Yes. The issue provides concrete timing-sensitive Slack steps, an independent sanitized WhatsApp reproduction on v2026.6.6, and current source retains the matching enqueue/drain race.

## Solution Assessment

The proposed enqueue-time self-kick is the narrowest likely fix, but the best resolution also needs a deterministic interleaving test and confirmation that the observed stale recovery owner is consequential rather than a separate defect.
