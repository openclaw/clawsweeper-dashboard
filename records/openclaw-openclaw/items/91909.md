## Summary

Keep open. Current main and the latest release still have the source shape described by the report: active-run followups can be enqueued with idle restart disabled and no drain scheduled when the single liveness check still reports an active run.

## What This Changes

The issue reports a timing-sensitive inbound dispatch race where a same-thread follow-up message can remain queued after run completion without starting a new run.

## Best Possible Solution

Make the generic follow-up queue and active-run lifecycle guarantee that queued work cannot remain idle after active-run teardown, with regression coverage for the terminal-window race rather than a Slack-specific workaround.

## Reproduction Assessment

No live reproduction was run, but current source shows the reported `restartIfIdle=false` plus single liveness-snapshot path, and adjacent tests leave the terminal-window race uncovered. That is enough to keep it open as source-reproducible, not enough to call it fully reproduced.
