## Summary

Keep this PR open: the requested same-profile overload retry is not present on current main, but the proposed patch still has a blocking cancellation bug and is stale against the current embedded-runner/failover split.

## What This Changes

The PR adds a bounded same-auth-profile retry loop for overloaded LLM failures, a `retry_same_profile` failover decision, and a harness test for assistant overloaded retries.

## Best Possible Solution

Re-apply the same-profile overloaded retry against current main's runner/failover structure, preserve cancellation as `AbortError`, and cover prompt, assistant, and abort paths with focused tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the patch defect: abort during the new retry sleep, which currently rejects as a plain error and can bypass model-fallback cancellation handling. I did not run a live Anthropic 529 reproduction because that condition is not deterministic.

## Solution Assessment

No. The retry direction is reasonable, but the proposed implementation should first normalize retry-sleep aborts and be adapted to the current `run.ts` plus `run/assistant-failover.ts` structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Normalize retry-sleep aborts:** `src/agents/pi-embedded-runner/run.ts:935`
  - body: The new retry helper awaits `sleepWithAbort` directly. That helper rejects aborts with a plain `Error("aborted")`, while model fallback only rethrows errors named `AbortError`; canceling during this delay can be recorded as a candidate failure and continue into fallback attempts instead of stopping. Wrap this await like the existing overload backoff path before merging.
  - confidence: 0.92
