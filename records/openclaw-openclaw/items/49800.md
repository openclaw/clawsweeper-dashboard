## Summary

Keep open. Current main still lacks same-auth-profile retry for overloaded LLM failures, and this PR remains useful, but it needs changes before merge because the new retry sleep can turn user cancellation into an ordinary provider failure and the branch is behind the current assistant-failover helper split.

## What This Changes

The PR adds a bounded same-auth-profile overloaded retry loop in the embedded runner, extends failover decision logging with `retry_same_profile`, and adds retry-focused runner coverage plus harness mocks.

## Best Possible Solution

Land a rebased or replacement implementation on current main that adds bounded same-auth-profile retry for overloaded prompt and assistant failures, preserves cancellation as `AbortError`, updates the failover decision type/logging and tests, adds the required changelog entry, and leaves the broader configurable transient retry policy tracked separately by #49376 unless maintainers explicitly combine them.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with a single auth profile, overloaded prompt or assistant failures attempt profile rotation, cannot rotate, and then fall through to fallback or surface-error behavior without retrying the same profile.

## Solution Assessment

No, not as currently proposed. The narrow same-profile retry direction is maintainable, but the patch should normalize retry-sleep aborts like the existing overload backoff helper and be rebased into the current assistant-failover helper structure before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve cancellation during overload retry sleeps:** `src/agents/pi-embedded-runner/run.ts:935`
  - body: The new retry helper awaits `sleepWithAbort` directly. That helper rejects with a plain `Error("aborted")`, while the outer fallback runner only treats `name === "AbortError"` as user cancellation, so stopping a run during this delay can be handled as a normal provider failure and continue into extra fallback attempts/API calls. Wrap the rejected sleep as an `AbortError`, matching `maybeBackoffBeforeOverloadFailover`.
  - confidence: 0.91
