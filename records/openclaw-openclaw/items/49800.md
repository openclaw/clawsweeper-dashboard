## Summary

Keep open. Current main still lacks same-auth-profile retry for overloaded LLM failures, and this PR targets the right product gap, but the proposed patch has a blocking cancellation regression because the new retry sleep can throw a plain aborted error that the model-fallback runner will not treat as user cancellation.

## What This Changes

The PR adds a bounded same-auth-profile retry loop for overloaded prompt and assistant failures, extends failover decision logging with `retry_same_profile`, and adds overloaded retry test coverage plus harness mocks.

## Best Possible Solution

Land a rebased or replacement implementation that adds bounded same-auth-profile retry for overloaded prompt and assistant failures, preserves cancellation as `AbortError`, fits the current `auth.cooldowns` profile-rotation model, and leaves the broader configurable transient LLM retry policy to #49376 unless maintainers explicitly combine the work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with a single auth profile, overloaded prompt or assistant failures try `advanceAuthProfile()`, cannot rotate, and then fall through to fallback or surface-error behavior without a same-profile retry; the patch regression is reproducible by aborting during the new retry sleep.

## Solution Assessment

No, not as currently proposed. The narrow same-profile retry direction is maintainable, but the patch must wrap retry-sleep aborts like the existing overload backoff helper and be rebased onto the current `run/assistant-failover.ts` split before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve cancellation during overload retry sleeps:** `src/agents/pi-embedded-runner/run.ts:935`
  - body: The new retry helper awaits `sleepWithAbort` directly. That helper rejects aborted sleeps with a plain `Error("aborted")`, while the outer fallback runner only treats `name === "AbortError"` as user cancellation, so stopping a run during this delay can be misclassified as a provider failure and continue into fallback attempts/API calls. Wrap the rejected sleep as an `AbortError`, matching `maybeBackoffBeforeOverloadFailover`.
  - confidence: 0.91
