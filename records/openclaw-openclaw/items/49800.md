## Summary

Keep open: current main still does not implement same-auth-profile retry for single-profile overloaded errors, but this PR needs changes before merge because the new retry sleep can misclassify cancellation and the branch is stale against the current failover helper structure.

## What This Changes

The PR adds a bounded same-auth-profile overloaded retry helper, retry logging/observation value, harness mocks, and a behavioral overloaded-retry test.

## Best Possible Solution

A rebased or replacement implementation should keep the retry bounded, preserve existing profile-rotation and model-fallback semantics, normalize retry-sleep cancellation to `AbortError`, and leave the broader configurable LLM retry policy to #49376.

## Reproduction Assessment

Yes. Source inspection shows current main lacks a same-profile retry branch, and the PR cancellation bug is reproducible from the direct `sleepWithAbort` call plus the fallback runner's strict `AbortError` contract.

## Solution Assessment

No, not as-is. The retry direction is narrow and useful, but the implementation should be ported to current main's failover helpers and wrap aborted retry sleeps the same way the existing overload backoff path does.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wrap retry sleep aborts as AbortError:** `src/agents/pi-embedded-runner/run.ts:933`
  - body: The new same-profile retry helper awaits `sleepWithAbort(...)` directly. If the run is canceled during this backoff, `sleepWithAbort` throws a plain `Error("aborted")`, while the outer fallback runner only treats `name === "AbortError"` as user cancellation, so canceled runs can fall through as normal failures. Wrap this sleep like `maybeBackoffBeforeOverloadFailover` does.
  - confidence: 0.91

  - body: This changes user-visible LLM recovery behavior by silently retrying overloaded requests before surfacing an error, but the PR does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing fix/feature/perf changes before merge.
