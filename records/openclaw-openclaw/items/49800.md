## Summary

Keep open. Current main has related overload profile-rotation and backoff controls, but it does not implement this PR's same-profile retry path; the PR remains useful source work, but the latest diff still has a blocking cancellation bug and is stale against the helperized failover code.

## What This Changes

The PR adds a bounded same-auth-profile retry loop for overloaded LLM errors, a `retry_same_profile` failover observation decision, and runner test harness coverage.

## Best Possible Solution

Reapply the bounded same-profile overloaded retry to current main's failover structure, preserve existing profile rotation and model fallback semantics, normalize cancellation to `AbortError`, and leave broader configurable LLM retry policy to #49376.

## Reproduction Assessment

Yes, from source inspection rather than live provider behavior. Current main lacks a same-profile retry branch, and the PR diff's cancellation bug is source-reproducible from the direct `sleepWithAbort` call plus the fallback runner's strict `AbortError` check.

## Solution Assessment

No, not as-is. The retry direction is narrow and useful, but the implementation should be rebased or replaced on current main and must wrap retry-sleep aborts the same way the existing overload backoff path does.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Wrap retry sleep aborts as AbortError:** `src/agents/pi-embedded-runner/run.ts:933`
  - body: The new same-profile retry helper awaits `sleepWithAbort` directly. If the run is canceled during this backoff, that can throw a plain `Error("aborted")`, while the outer fallback runner only treats `name === "AbortError"` as user cancellation. This can misclassify a canceled run as a normal failure and continue into fallback/API work; wrap this sleep like `maybeBackoffBeforeOverloadFailover` does.
