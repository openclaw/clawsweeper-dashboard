## Summary

Keep open. The retry direction addresses a real overloaded-provider gap, but this PR is not mergeable as-is because the new retry sleep does not preserve abort semantics, contributor real behavior proof is missing, and current main has since refactored the runner/failover path.

## What This Changes

The PR adds a bounded same-profile retry loop for overloaded LLM errors, a new failover decision value, harness mocks, and an overloaded-retry runner test.

## Best Possible Solution

Rebase the same-profile overload retry onto current failover helpers, preserve abort semantics exactly like the existing overload backoff path, settle its relationship to configurable provider retry, and attach redacted real runtime proof before merge.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection shows current main lacks the same-profile retry path, and the PR's test mocks two overloaded assistant attempts before success, but no real Anthropic 529 after-fix run is provided.

## Solution Assessment

No. The retry direction is plausible, but this patch should not merge until the retry sleep normalizes aborts, the branch is adapted to current main, and the contributor supplies real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Normalize retry sleep aborts:** `src/agents/pi-embedded-runner/run.ts:935`
  - body: This new retry helper awaits `sleepWithAbort` directly. That helper rejects cancellation as a plain `Error("aborted")`, while the fallback runner only treats `AbortError` as user cancellation, so stopping a run during this backoff can be recorded as a normal candidate failure and continue into extra fallback/API attempts. Wrap this sleep the same way `maybeBackoffBeforeOverloadFailover` does.
  - confidence: 0.9
