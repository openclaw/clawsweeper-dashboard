Overall confidence: 0.88

Full review comments:

- **[P1] Route Speakeasy synthesis through canonical TTS:** `extensions/telegram/src/speakeasy-voice.ts:313`
  - body: This shells out to a workspace script for Telegram callback text even though OpenClaw already has `messages.tts`/`synthesizeSpeech` with provider config, SecretRef coverage, fallback/status behavior, and a `voice-note` target. That makes an enabled Telegram chat a remote trigger for a local script and creates a second TTS contract that will drift from the supported provider path.
  - confidence: 0.9
- **[P2] Validate voice-note output before calling sendVoice:** `extensions/telegram/src/bot-handlers.runtime.ts:2225`
  - body: The callback path sends whatever file the script prints directly through `sendVoice`; the new test even stubs `/tmp/speakeasy.mp3` as the generated output. Current Telegram delivery only uses `sendVoice` after checking voice-compatible audio, so this can make real taps fail for non-Opus/non-voice-note output instead of safely sending audio or falling back.
  - confidence: 0.86
- **[P2] Avoid caching buttons under unresolved handles:** `extensions/telegram/src/outbound-adapter.ts:149`
  - body: `parseTelegramTarget` supports `@username` and `t.me` targets, but this caches the Speakeasy entry under that unresolved handle before delivery. Telegram callbacks carry the numeric chat id and `resolveSpeakeasyCachedText` requires an exact match, so buttons sent to handle targets will always look expired when tapped.
  - confidence: 0.83
- **[P2] Serialize Speakeasy cache updates:** `extensions/telegram/src/speakeasy-voice.ts:218-221`
  - body: This second read/merge/write is still not a write guard: two concurrent replies can both reload the old cache before either write, and the later write can drop the earlier callback entry. That leaves a visible button whose cache id no longer resolves.
  - confidence: 0.8
