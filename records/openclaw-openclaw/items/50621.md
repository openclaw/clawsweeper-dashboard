## Summary

Keep open: current main still matches the reported cron status bug. Main-session `systemEvent` jobs still use the generic cron timeout, `wakeMode="now"` still awaits heartbeat completion, and the related duplicate plus closed unmerged fix PRs leave this as the canonical tracker.

## What This Changes

The issue asks OpenClaw to stop marking long-running main-session `systemEvent` cron jobs as timed out when the event has been delivered and the main-session agent continues or completes the work.

## Best Possible Solution

Make the main-session `systemEvent` cron contract explicit in code and docs: after event delivery and wake scheduling, the scheduler should not report failure solely because the triggered main-session heartbeat turn outlives the generic cron timeout, while isolated `agentTurn` timeout behavior stays intact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a main-session `systemEvent` job with `wakeMode="now"` and a `runHeartbeatOnce` promise that stays unresolved past `DEFAULT_JOB_TIMEOUT_MS` will be raced out by `executeJobCoreWithTimeout` and recorded with the reported timeout error.

## Solution Assessment

Yes, the maintainable fix should address the false cron-status failure at the main-session `systemEvent` boundary rather than only raising the timeout to 60 minutes. A new timeout config can be avoided unless maintainers deliberately choose blocking completion semantics for main-session events.

## Repair Work Prompt

Fix openclaw/openclaw#50621. Current main still lets main-session `systemEvent` cron jobs with `wakeMode: "now"` wait on `runHeartbeatOnce` under the generic 10-minute cron timeout, so delivered events can be recorded as `cron: job execution timed out` while the main agent continues or completes the work. Treat #70325 as a duplicate with useful confirmation, and treat closed PRs #50670/#50698 as prior context only; do not merely raise the timeout if the false-failure semantics remain. Keep the fix scoped to cron main-session `systemEvent` behavior, preserve isolated `agentTurn` timeout and cleanup behavior, add/update regression tests for a wake-now main `systemEvent` whose heartbeat promise outlives `DEFAULT_JOB_TIMEOUT_MS`, update cron docs if the main-session status contract changes, and add a user-facing changelog fix entry. Stop and ask if the implementation would require a broad new config option or a product-policy rewrite outside cron runtime semantics.
