## Label Justifications

- `P2`: This is a normal-priority cron runtime bug with limited surface area but real false-failure reports from long-running main-session jobs.
- `impact:session-state`: Cron run state can diverge from actual main-session work state when the session completes work but the scheduler records an error.
- `impact:message-loss`: A false timeout can encourage duplicate delivery or retries for a system event that was already processed.

## Summary

Keep open: current main and the latest release still apply the generic cron watchdog to main-session `systemEvent` jobs while `wakeMode: "now"` waits on heartbeat work, so the false `cron: job execution timed out` state remains source-reproducible. The duplicate report and closed unmerged fix PRs confirm the same failure mode, but no merged fix currently resolves it.

## What This Changes

The issue reports main-session cron `systemEvent` jobs being marked timed out after about 10-16 minutes even though the main-session agent work is still running or has completed.

## Best Possible Solution

Apply an agent-turn-scale safety timeout to the blocking main-session `systemEvent` plus `wakeMode: "now"` path while preserving the 10-minute default for non-blocking `systemEvent` jobs and existing isolated `agentTurn` cleanup behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a main-session `systemEvent` job with `wakeMode: "now"` awaits `runHeartbeatOnce` while the cron wrapper races that work against the generic 10-minute timeout.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust timeout policy for the blocking main-session `systemEvent` wake-now path; a broad new config option or global unbounded `systemEvent` timeout is not required for this bug.
