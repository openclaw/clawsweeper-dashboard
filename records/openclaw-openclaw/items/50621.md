## Summary

Current main and the latest release already make main-session wake-now systemEvent cron jobs fire-and-forget: cron queues a heartbeat and records the wrapper run as ok instead of waiting for the long agent turn to finish under the generic job watchdog.

## What This Changes

The issue reports that main-session systemEvent cron jobs can be marked timed out while the underlying agent work continues or completes.

## Best Possible Solution

Keep the shipped fire-and-forget main-session cron behavior and use a new issue only if a current v2026.6.8-or-newer setup still records false timeout failures.

## Reproduction Assessment

Yes for the original bug from source and linked reports: older code awaited runHeartbeatOnce under the generic systemEvent timeout. I did not reproduce a failing current-main path because current main now queues the heartbeat and records the wrapper as ok.

## Solution Assessment

Yes. The shipped fix is narrower than removing the timeout or adding config because it makes the blocking main-session wake-now path fire-and-forget while preserving bounded cron watchdog behavior elsewhere.
