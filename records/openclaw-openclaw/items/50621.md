## Label Justifications

- `P2`: This is a real cron runtime bug with duplicate reports, but the blast radius is limited to long-running main-session `systemEvent` jobs.
- `impact:session-state`: Cron run state can diverge from main-session agent state by recording a timeout even when the session processed the event.
- `impact:message-loss`: A false timeout on a delivered system event can trigger retries or failure handling that duplicates or misreports scheduled message delivery.

## Summary

Keep open: current main and the latest release still apply the generic cron timeout to main-session `systemEvent` jobs while `wakeMode: "now"` awaits `runHeartbeatOnce`, so the false timeout status remains source-reproducible. The duplicate reports and closed fix attempts preserve useful context, but no open focused PR currently owns this exact cron timeout boundary.

## What This Changes

Main-session cron `systemEvent` jobs should not be marked as timed out by the scheduler while the main agent turn is still running or has completed successfully.

## Best Possible Solution

Make the main-session `systemEvent` wake-now status boundary explicit: cron should record success after delivery plus verified wake scheduling or start, while isolated `agentTurn` timeout cleanup remains unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a main-session `systemEvent` job with `wakeMode: "now"` whose `runHeartbeatOnce` promise remains pending past `DEFAULT_JOB_TIMEOUT_MS` is raced out by `executeJobCoreWithTimeout` and recorded as a cron timeout.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust the main-session `systemEvent` wake-now status or timeout boundary, not to fold this into broader liveness-timeout work or add a new user-facing config first.
