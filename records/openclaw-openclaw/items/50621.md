## Label Justifications

- `P2`: This is a normal-priority cron runtime bug with limited surface area but real false-failure reports from long-running main-session jobs.
- `impact:session-state`: Cron run state can diverge from actual main-session work state when the scheduler records an error for work that may have completed.
- `impact:message-loss`: A false timeout can encourage duplicate delivery or retries for a system event that was already processed.

## Summary

Keep open: current main and v2026.5.22 still apply the generic cron watchdog to main-session `systemEvent` jobs while `wakeMode: "now"` waits for heartbeat work, so the false `cron: job execution timed out` state remains source-reproducible. The duplicate report and closed unmerged fix attempts add useful confirmation, but no merged or open canonical fix resolves this exact main-session timeout path.

## What This Changes

The issue requests that main-session cron `systemEvent` jobs stop being marked as timed out by the cron scheduler while long-running main-session agent work is still active.

## Best Possible Solution

Apply an agent-turn-scale safety timeout to the blocking main-session `systemEvent` plus `wakeMode: "now"` path while preserving the 10-minute default for non-blocking `systemEvent` jobs and existing isolated `agentTurn` timeout behavior.

## Reproduction Assessment

Yes at source level. A main-session `systemEvent` job with `wakeMode: "now"` awaits `runHeartbeatOnce` while the cron wrapper races that work against the generic 10-minute timeout.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust the timeout policy for the blocking main-session `systemEvent` wake-now path; a broad new config option or global `systemEvent` timeout change is not required for this bug.
