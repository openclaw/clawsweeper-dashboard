## Summary

Keep open: current main and the latest release still let a wake-now main-session `systemEvent` cron job sit inside the generic cron timeout while awaiting `runHeartbeatOnce`, so the false `cron: job execution timed out` status remains source-reproducible. The duplicate report and prior closed PRs all point to the same focused cron timeout boundary, and no open focused fix PR currently owns it.

## What This Changes

The issue asks cron not to mark a delivered main-session `systemEvent` job as timed out while the main-session agent turn continues running.

## Best Possible Solution

Make the main-session `systemEvent` wake-now status boundary explicit: cron should record success after delivery plus verified wake scheduling or start, while isolated `agentTurn` timeout cleanup remains unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a main-session `systemEvent` job with `wakeMode: "now"` whose `runHeartbeatOnce` promise remains pending past `DEFAULT_JOB_TIMEOUT_MS` is raced out by `executeJobCoreWithTimeout` and recorded as a cron timeout.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust the main-session `systemEvent` wake-now status or timeout boundary, not to fold this into the broader liveness-timeout proposal or add a new user-facing config first.
