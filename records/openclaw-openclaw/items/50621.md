## Label Justifications

- `P2`: This is a normal-priority cron runtime bug with limited surface area but real false-failure reports from long-running main-session jobs.
- `impact:session-state`: Cron run state can diverge from actual main-session work state when the session completes work but the scheduler records an error.
- `impact:message-loss`: A false timeout can encourage duplicate delivery or retries for a system event that was already processed.

## Summary

Keep open: current main and the latest release still apply the generic cron watchdog to main-session `systemEvent` jobs while `wakeMode: "now"` waits on heartbeat work, so the false `cron: job execution timed out` state remains source-reproducible. The duplicate reports and closed unmerged fix PRs add useful confirmation, but no merged fix or open canonical PR currently resolves this issue.

## What This Changes

The issue requests that main-session cron `systemEvent` jobs avoid false scheduler timeout failures when the agent continues running in the main session.

## Best Possible Solution

Apply an agent-turn-scale safety timeout to the blocking main-session `systemEvent` + `wakeMode: "now"` path while preserving the 10-minute default for non-blocking `systemEvent` jobs and existing isolated `agentTurn` cleanup behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a main-session `systemEvent` job with `wakeMode: "now"` awaits `runHeartbeatOnce` while the cron wrapper races that work against the generic 10-minute timeout.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust the timeout policy for the blocking main-session `systemEvent` wake-now path; a broad new config option or global `systemEvent` timeout change is not required for this bug.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/50621. Current main gives `sessionTarget: "main"` + `payload.kind: "systemEvent"` + `wakeMode: "now"` the generic 10-minute cron timeout even though that path awaits `runHeartbeatOnce` and can block on real main-session agent work. Implement the narrow timeout-policy repair so this blocking path uses `AGENT_TURN_SAFETY_TIMEOUT_MS`, while preserving the generic `DEFAULT_JOB_TIMEOUT_MS` for non-blocking `systemEvent` jobs such as `wakeMode: "next-heartbeat"` and preserving existing isolated `agentTurn` explicit timeout/cleanup behavior. Add focused regression coverage for the new timeout-policy branch and its inverse, keep normalization/type behavior unchanged unless required by the narrow fix, update `CHANGELOG.md` if user-visible, and stop if the solution would require a new user config option or broader cron product decision.
