## Label Justifications

- `P2`: This is a normal-priority cron runtime bug with limited surface area but real false-failure reports from long-running main-session jobs.
- `impact:message-loss`: The duplicate context warns that retrying after a false timeout can duplicate cron events that were already delivered or processed.
- `impact:session-state`: Cron run state can diverge from the actual main-session work state when the session completes work but the scheduler records an error.

## Summary

Keep open: current main and the latest release still route main-session `systemEvent` cron jobs through the generic cron watchdog while `wakeMode: "now"` waits on heartbeat work, so the false `cron: job execution timed out` state remains source-reproducible. The linked duplicate adds useful evidence that the timeout can diverge from real completed work, and the related timeout PRs were closed unmerged.

## What This Changes

The issue asks OpenClaw to stop marking long-running main-session `systemEvent` cron jobs as timed out when the main session is still processing or has processed the event.

## Best Possible Solution

Make the main-session `systemEvent` wake-now status boundary explicit: record success after delivery plus a confirmed wake/start signal, or intentionally use an agent-turn-scale timeout for that blocking path, while preserving isolated `agentTurn` timeout cleanup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a main-session `systemEvent` job with `wakeMode: "now"` awaits `runHeartbeatOnce` while the cron wrapper races it against the generic timeout; I did not run a live Gateway cron scenario.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust the main-session `systemEvent` wake-now timeout/status boundary, not to fold this into broad liveness-timeout work or require a new config option first.
