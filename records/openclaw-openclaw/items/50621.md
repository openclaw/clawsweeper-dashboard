## Summary

Keep open: current main still matches the reported cron status bug. Main-session systemEvent jobs still use the generic cron timeout, wakeMode="now" still awaits heartbeat completion, and the duplicate report plus closed unmerged fix PRs leave this as the canonical tracker; Ryce's KeepMyClaw sidecar idea is useful workaround context but not an OpenClaw scheduler fix.

## What This Changes

Main-session systemEvent cron jobs should stop being falsely marked as timed out during long heartbeat agent turns, either by fire-and-forget delivery or by an explicit configurable/documented timeout contract.

## Best Possible Solution

Make the main-session systemEvent cron contract explicit in code and docs. The durable fix should prevent wakeMode="now" from reporting failure solely because the triggered main-session heartbeat turn runs longer than the generic cron lane timeout, while preserving isolated agentTurn timeout behavior and adding regression coverage for the false-failure case.

## Reproduction Assessment

Yes. A focused current-main reproduction is a main/sessionTarget systemEvent cron job with wakeMode="now" and a runHeartbeatOnce promise that stays unresolved past DEFAULT_JOB_TIMEOUT_MS; executeJobCoreWithTimeout will race out and record the reported timeout error.

## Solution Assessment

Unclear for the closed PR approach: raising this path to the 60-minute agent-turn ceiling would reduce failures but would not make systemEvent main-session delivery fire-and-forget or eliminate false failures for longer work. The safer end state is to record success after enqueueing and waking, or deliberately expose/document a systemEvent timeout if maintainers want blocking status semantics.

## Repair Work Prompt

Fix the main-session systemEvent cron timeout false-failure tracked by https://github.com/openclaw/openclaw/issues/50621 and duplicate https://github.com/openclaw/openclaw/issues/70325. Current main enqueues the system event but then waits for runHeartbeatOnce under the generic 10-minute cron timeout, so long-running main-session work can complete while cron records "cron: job execution timed out". Implement a focused cron contract change so sessionTarget="main" + payload.kind="systemEvent" + wakeMode="now" records success after enqueueing and scheduling/immediately requesting the heartbeat, rather than waiting for the full heartbeat agent turn to finish. Preserve isolated agentTurn timeout behavior and do not broaden bundled-plugin/core boundaries. Update or replace the existing wakeMode-now test that currently asserts blocking behavior, add a regression test that a never-resolving/long heartbeat does not mark a main systemEvent run timed out, and update docs/automation/cron-jobs.md if any user-visible timeout semantics remain. Add a changelog fix entry crediting @LeandroLE and @TradingData684 if behavior changes. Related source PRs #50670 and #50698 used a 60-minute timeout increase; treat them as context, but do not use a larger timeout as the sole fix unless a maintainer explicitly chooses blocking semantics.
