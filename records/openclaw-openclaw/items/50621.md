## Label Justifications

- `P2`: This is a normal-priority cron runtime correctness bug with a limited but real main-session automation impact.
- `impact:session-state`: Cron run state can report an error even when the main-session agent work is still running or has completed.
- `impact:message-loss`: A false timeout can make operators retry a system event that may already have been delivered and processed.

## Summary

Keep open: current main and the latest release still apply the generic cron watchdog to main-session `systemEvent` jobs while `wakeMode: "now"` awaits `runHeartbeatOnce`, so the false `cron: job execution timed out` state remains source-reproducible. Duplicate user evidence and closed unmerged fix attempts confirm the root cause, but no merged canonical fix owns this exact path yet.

## What This Changes

Main-session `systemEvent` cron jobs that run with `wakeMode: "now"` should not be marked failed by the generic cron timeout while long-running main-session agent work is still active.

## Best Possible Solution

Adjust the timeout policy only for the blocking main-session `systemEvent` plus `wakeMode: "now"` path, using the existing agent-turn-scale safety ceiling while preserving the generic default for fire-and-forget system events.

## Reproduction Assessment

Yes at source level. A main-session `systemEvent` job with `wakeMode: "now"` awaits `runHeartbeatOnce` while the cron wrapper races that work against the generic 10-minute timeout.

## Solution Assessment

Yes. The narrow maintainable fix is to special-case the blocking main-session wake-now path in timeout policy; adding a broad new config option is not needed to fix the reported bug.
