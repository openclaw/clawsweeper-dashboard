## Summary

Keep open: current main still has the reported false cron-status path for main-session `systemEvent` jobs. The scheduler applies the generic cron timeout to non-`agentTurn` jobs while `wakeMode="now"` awaits the main heartbeat, so long-running main work can be recorded as `cron: job execution timed out` even after the event was delivered.

## What This Changes

The issue requests that main-session cron `systemEvent` jobs avoid false timeout failures while the main agent turn continues processing delivered work.

## Best Possible Solution

Make the main-session `systemEvent` wake-now contract explicit: after event delivery and verified wake scheduling/start, cron should not report failure solely because the triggered main heartbeat outlives the generic cron timeout, while isolated `agentTurn` timeout cleanup remains intact.

## Reproduction Assessment

Yes, from source inspection. A main-session `systemEvent` job with `wakeMode="now"` and a `runHeartbeatOnce` promise that stays unresolved past `DEFAULT_JOB_TIMEOUT_MS` will be raced out by `executeJobCoreWithTimeout` and recorded with the reported timeout error.

## Solution Assessment

Yes, the narrow maintainable path is to fix the main-session `systemEvent` timeout/status boundary rather than add a broad new cron mode. The repair should preserve the existing wake-now heartbeat-start guarantee from #652 while preventing false timeout status for long main-session work.
