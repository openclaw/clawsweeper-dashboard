## Summary

Keep open: current main and the latest release still apply the generic cron job timeout to main-session `systemEvent` jobs while `wakeMode: "now"` waits for `runHeartbeatOnce`, so the false `cron: job execution timed out` state remains source-reproducible. Related PRs that tried to adjust this exact timeout were closed unmerged, and current open cron PRs I checked do not own this focused fix.

## What This Changes

The issue reports that main-session cron `systemEvent` jobs can be recorded as timed out while the agent continues or completes work in the main session.

## Best Possible Solution

Make the main-session `systemEvent` wake-now status boundary explicit: cron should record success after delivery plus verified wake scheduling or start, while isolated `agentTurn` timeout cleanup remains unchanged.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a main-session `systemEvent` job with `wakeMode: "now"` whose `runHeartbeatOnce` promise remains pending past `DEFAULT_JOB_TIMEOUT_MS` is raced out by `executeJobCoreWithTimeout` and recorded as a cron timeout. I did not run an end-to-end macOS Gateway cron scenario in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust the main-session `systemEvent` wake-now status or timeout boundary, not to fold this into broad liveness-timeout work or add a new user-facing config first.

Current main still runs main-session `payload.kind="systemEvent"` / `wakeMode="now"` jobs under the generic cron timeout while awaiting `runHeartbeatOnce`, so long main-session work can be delivered or completed but the cron run records `cron: job execution timed out`. Make the main-session `systemEvent` wake-now status boundary explicit: record success after event delivery plus verified wake scheduling/start, or otherwise avoid racing the long main-session agent turn against `DEFAULT_JOB_TIMEOUT_MS`.
