## Summary

Keep open. Current main and the latest release provide explicit `canvas snapshot`, `browser snapshot`, and `browser screenshot` paths, but `canvas present`/`canvas navigate` and `browser navigate` still return state/JSON rather than a chat-visible preview, so the remaining behavior needs a product and security decision instead of cleanup closure.

## What This Changes

The issue asks for canvas/browser navigation to produce reliable chat-visible page visualization or confirmation instead of node/validation errors.

## Best Possible Solution

Define the supported chat-visible contract: keep explicit snapshot/screenshot plus clearer guidance/errors, or add an opt-in preview path coordinated with browser-preview and screenshot-privacy decisions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection shows state-changing canvas/browser actions still do not emit a chat-visible preview, and the `node required` path remains possible when no default node can be selected. The exact LM Studio/Gemma failure was not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? Unclear: automatic previews are only one possible fix, and explicit snapshot/screenshot already exists. The safer solution needs a maintainer decision on UX, privacy, and whether errors/docs or an opt-in preview should own the behavior.
