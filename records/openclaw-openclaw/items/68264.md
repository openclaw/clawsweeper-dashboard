## Summary

Keep open. Current main still has an explicit-preview contract: canvas/browser state-changing actions return JSON success or navigation results, while visible output requires separate snapshot/screenshot or WebChat embed paths, so the reported chat-visible confirmation gap is not fully resolved and needs a product decision.

## What This Changes

The issue asks for canvas/browser state-changing actions to provide reliable chat-visible rendering or confirmation instead of returning node-selection or validation errors.

## Best Possible Solution

Define the intended canvas/browser chat-visible contract, then either make the chosen state-changing actions produce preview output or tighten tool/docs/error guidance around explicit snapshot/screenshot usage with focused regression coverage.

## Reproduction Assessment

Partially yes: source inspection on current main shows the non-previewing `present`/`navigate` paths and the remaining `node required` branch when no default node exists. A high-confidence live reproduction of the full chat-visible UI gap still needs a WebChat plus node/browser setup.
