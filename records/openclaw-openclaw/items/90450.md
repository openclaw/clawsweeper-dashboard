## Summary

Keep open: current main and the latest release still discard streamed Claude CLI text when the final result event is empty, and I found no blocking defect in the focused parser patch. The remaining blockers are maintainer acceptance of the provider/message-delivery behavior change and completion of current-head checks.

## What This Changes

The PR changes Claude CLI JSONL parsing to preserve accumulated streamed assistant text when the final result event has empty text, records an internal tool-use flag, and adds parser plus runner regression coverage.

## Best Possible Solution

Land the focused parser fix after current-head checks complete and maintainers accept streamed Claude text winning over an empty final result; keep synthetic-placeholder handling in the separate open issue/PR path.

## Reproduction Assessment

Yes by source path and production logs, though I did not run a local live Claude repro. Current main and v2026.6.9 return the empty final Claude result before preserving streamed deltas, and the runner then treats empty parsed text as empty_response unless silent replies are explicitly allowed.

## Solution Assessment

Yes, this is the narrowest maintainable fix shape I found. It repairs the JSONL parser boundary used by ordinary CLI execution and Claude live sessions while preserving the runner's empty-response failover contract; broader synthetic-placeholder handling remains separate.
