## Summary

Keep this PR open: current main still has the central Claude CLI empty-result parsing problem, and the branch now fixes the earlier syntax/import break, but the runner change still needs a focused repair because it changes the empty-response fallback contract for tool-only turns.

## What This Changes

The PR changes Claude CLI JSONL parsing to preserve accumulated streamed text when the final result is empty, records tool-use presence, and skips empty-response failover for tool-only CLI outputs, with parser regression tests.

## Best Possible Solution

Keep the streamed-text parser fallback, but preserve the existing fallback default by allowing tool-only empty silence only through the existing allowEmptyAssistantReplyAsSilent policy or another explicit maintainer-approved policy.

## Reproduction Assessment

Yes, from source inspection: current main parses an empty Claude result event into empty text and the runner throws empty_response unless the silent-empty policy is true. I did not run tests because this review is read-only, but the PR's parser fixtures show a direct current-main failure shape.

## Solution Assessment

No, not as written: preserving streamed text in the parser is the right layer, but bypassing empty_response for all tool-only turns is broader than the existing policy path. The safer maintainable fix is parser preservation plus policy-gated tool-only silence.
