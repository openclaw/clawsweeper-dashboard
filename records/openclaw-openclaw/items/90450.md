## Summary

Keep open: current main and v2026.6.8 still let an empty Claude result record override already-streamed assistant text, while this PR fixes both JSONL parser paths with focused regression tests and sufficient live-log proof. The remaining decision is maintainer acceptance of the fallback/provider-routing behavior, not a concrete patch defect.

## What This Changes

The branch updates `src/agents/cli-output.ts` and parser/runner tests to preserve accumulated Claude stream-json text when a final result record is empty and to record Claude tool-use presence.

## Best Possible Solution

Land the parser-layer fallback after maintainers accept the provider-routing semantics and required checks stay green; avoid moving this into a channel-specific workaround.

## Reproduction Assessment

Yes, source-reproducible: a Claude JSONL stream with `content_block_delta` text followed by `result: ""` exercises both current parser paths that return the empty final result. I did not run tests in this read-only pass, but the current code and added fixtures make the failing path clear.

## Solution Assessment

Yes, this is the right layer: normalizing Claude CLI output in `cli-output.ts` fixes both live-session and process-run consumers while preserving the runner's default empty-response fallback for genuinely empty turns. A channel-specific or fallback-only fix would miss one of those parser consumers.
