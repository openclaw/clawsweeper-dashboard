## Summary

Keep open: current main still has the reported Claude CLI JSONL path that can discard streamed assistant text when the final result is empty, and this PR is a focused fix with credible production log proof. The only maintainer decision before merge is accepting the tool-call-only empty-result behavior, which changes fallback/message-delivery semantics.

## What This Changes

The PR preserves accumulated Claude CLI JSONL streamed assistant text when a final result event has empty text, adds a CliOutput.hadToolCalls flag, and skips empty-response failover for tool-call-only CLI turns.

## Best Possible Solution

Land the streamed-text preservation path once maintainers explicitly accept the tool-only no-payload fallback contract, or split that policy into a separately tested follow-up.

## Reproduction Assessment

Yes for the current-source path: current main returns an empty Claude result before consulting accumulated streamed deltas, and the PR body supplies before/after production logs for the intermittent live symptom. I did not run a live Claude CLI reproduction because the reporter describes the empty-result event as intermittent.

## Solution Assessment

Mostly yes: preserving streamed deltas in the parser is the narrow owner-boundary fix for dropped assistant text. The tool-only no-failover branch is plausible but policy-sensitive, so maintainers should explicitly accept or test that contract before merge.
