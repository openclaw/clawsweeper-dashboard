## Summary

Keep open: the streamed-text parser fix is well supported, but the runner change still widens Claude tool-only empty replies into successful no-payload turns, bypassing existing fallback behavior unless maintainers explicitly accept that product contract.

## What This Changes

The PR updates Claude CLI JSONL parsing to preserve accumulated streamed text when a final result is empty, records Claude tool-use presence, and uses that flag to skip empty-response failover for tool-only turns.

## Best Possible Solution

Land the streamed-text preservation and metadata, but keep tool-only empty success behind `allowEmptyAssistantReplyAsSilent` or another explicit maintainer-approved policy with runner coverage for both states.

## Reproduction Assessment

Yes, by source inspection and the PR's live logs: current main returns an empty Claude result before considering streamed deltas, then `runPreparedCliAgent` raises `empty_response` unless the silent-empty flag is set. I did not execute the intermittent live Claude condition locally.

## Solution Assessment

No, not as currently written. Preserving accumulated streamed text is the right layer, but making any tool-use empty result bypass fallback by default is broader than the existing silent-reply policy allows.
