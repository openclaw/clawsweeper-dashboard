## Summary

Keep open: this PR targets a valid gateway restart/channel reload race and the prior syntax blocker is fixed, but the current patch is still a partial mitigation because its restart-pending helper can go false before shutdown/restart finishes and before stale reload work is drained.

## What This Changes

The PR exports a restart-pending helper, checks it during gateway channel reload deferral and per-channel restart, and adds a mocked regression test for the deferred reload path.

## Best Possible Solution

Use gateway-owned config-reloader cancellation/drain that stays active through shutdown/restart, stop or drain the reloader before channel teardown, recheck immediately before channel start, and then add redacted real runtime proof.

## Solution Assessment

No. The PR targets the right bug, but the maintainable fix is lifecycle-owned reload cancellation/drain with a final pre-start cancellation check instead of transient pending-signal polling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Track restart lifecycle past signal consumption:** `src/infra/restart.ts:113-115`
  - body: `isGatewayRestartPending()` only covers restart preparation or an unconsumed SIGUSR1 token. The run loop marks that token handled before it calls the restart request/close path, so a deferred reload waking during shutdown can see `false` and continue into channel stop/start; keep a lifecycle cancellation signal alive until old reload work is drained.
  - confidence: 0.91
- **[P2] Recheck cancellation after stopping the channel:** `src/gateway/server-reload-handlers.ts:513-523`
  - body: The new gate runs before `stopChannel`, but if the restart becomes pending while stop is in flight, the stale reload still proceeds to `startChannel(name)`. Recheck the restart/reload cancellation state immediately before starting the channel and cover that interleaving.
  - confidence: 0.86
