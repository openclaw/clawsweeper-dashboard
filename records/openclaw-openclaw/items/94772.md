## Summary

## What This Changes

The PR adds gateway restart-pending and module-level abort checks so deferred channel reloads skip channel stop/start during SIGUSR1 in-process restart.

## Best Possible Solution

Move cancellation into the config-reloader/shutdown lifecycle so restart aborts and drains active hot reload work, and only abort channel reloads for SIGUSR1 paths that actually proceed to restart.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Abort only when SIGUSR1 will actually restart:** `src/cli/gateway-cli/run-loop.ts:746`
  - body: This branch calls `abortPendingChannelReloads()` before it knows whether an unauthorised external SIGUSR1 will be ignored. When external SIGUSR1 is disallowed, the handler returns without restarting or recreating reload handlers, leaving the module-level abort flag set so later channel config reloads skip stop/start even though no restart is in progress.
  - confidence: 0.92
- **[P1] Do not clear old reload cancellation from the new lifecycle:** `src/gateway/server-reload-handlers.ts:229`
  - body: Resetting the process-global abort flag when new reload handlers are created can clear cancellation while an old `onHotReload` is still alive, because current `configReloader.stop()` does not await active reload work. The old deferred reload can then wake with cancellation false and still stop/start channels during the restart window this PR is meant to close.
  - confidence: 0.88
- **[P2] Propagate cancellation before plugin pre-stop:** `src/gateway/server-reload-handlers.ts:332-336`
  - body: `waitForActiveWorkBeforeChannelReload()` still returns `void`, so the plugin-reload `beforeReplace` path treats a cancelled wait the same as a successful drain and continues into `stopChannel`. Return a cancelled result or recheck before the pre-stop loop so plugin reloads cannot stop channels during restart after the new skip path fires.
  - confidence: 0.85
