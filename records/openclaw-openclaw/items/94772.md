## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate reload aborts on accepted restarts:** `src/cli/gateway-cli/run-loop.ts:746`
  - body: `abortPendingChannelReloads()` now runs before this branch verifies that an unauthenticated external SIGUSR1 is allowed. When `commands.restart=false`, the handler logs that the signal was ignored and returns without restarting, but the abort flag remains set until a new reload handler is created, so the next channel config reload can be skipped even though no restart will apply the new config. Move the abort call into paths that actually accept or schedule a restart, and add an ignored-SIGUSR1 regression test.
  - confidence: 0.9
- **[P1] Keep abort state scoped to the old reload lifecycle:** `src/gateway/server-reload-handlers.ts:229`
  - body: Resetting the module-level abort flag at every `createGatewayReloadHandlers()` call lets the new in-process restart lifecycle clear the cancellation that the old deferred reload is waiting to observe. If the old poll wakes after the new handler is created, it can still continue to stop/start the channel and collide with the restart. Use an old-lifecycle-owned signal/generation or have the config reloader stop/drain active reload work instead of resetting shared state globally.
  - confidence: 0.88
- **[P2] Propagate cancellation before plugin pre-stop:** `src/gateway/server-reload-handlers.ts:332-336`
  - body: When `waitForActiveWorkBeforeChannelReload()` sees cancellation it only logs and returns `void`, so the plugin reload `beforeReplace` caller continues after the await and can still stop channel runtimes during restart before the later restart-channel guard runs. Return a cancellation result or pass an abort signal through this path and skip the plugin pre-stop work when restart has superseded the reload.
  - confidence: 0.84
