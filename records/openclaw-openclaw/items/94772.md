## Summary

Keep open: this is a useful candidate for the linked gateway crash-loop bug, but the patch is still too narrow because it relies on transient restart-pending state and does not prove the live gateway/channel/SIGUSR1 path.

## What This Changes

The PR exports restart-pending state, gates gateway channel hot-reload deferral and per-channel restart on it, and adds a regression test for a deferred reload becoming restart-pending.

## Best Possible Solution

Use a gateway-owned reload cancellation/drain contract that remains active from restart acceptance through old-lifecycle shutdown, return a cancelled result to reload callers, and recheck cancellation immediately before channel start.

## Reproduction Assessment

Yes, source-reproducible: the linked issue gives a concrete EADDRINUSE timing/log recipe, and current source shows deferred reload work can proceed independently of the SIGUSR1 restart lifecycle. I did not run a live Telegram/SIGUSR1 repro in this read-only review.

## Solution Assessment

No. The PR targets the right race, but the best fix is lifecycle-owned cancellation/drain plus caller-visible cancellation, not a helper based on an already-consumable restart token.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep restart cancellation alive through shutdown:** `src/infra/restart.ts:113-115`
  - body: `isGatewayRestartPending()` only tracks preparation or an unconsumed SIGUSR1 token. The run loop marks that token handled before shutdown/restart begins, so a deferred reload that wakes in the reported window can see `false` and still stop/start channels; use lifecycle cancellation that stays active until old reload work is stopped or drained.
  - confidence: 0.91
- **[P2] Signal cancelled waits to plugin reload callers:** `src/gateway/server-reload-handlers.ts:314-318`
  - body: The new early return from `waitForActiveWorkBeforeChannelReload()` is indistinguishable from a successful drain. The plugin reload pre-replace path awaits this helper and then stops channels, so a restart-pending skip can still run stale channel stop work during restart; return a cancelled result or gate the pre-stop path too.
  - confidence: 0.86
- **[P2] Recheck restart cancellation before channel start:** `src/gateway/server-reload-handlers.ts:513-518`
  - body: The new guard runs before `await params.stopChannel(...)`. If SIGUSR1 is accepted while stop is in flight, this continuation still calls `startChannel(name)` and can race the restarted gateway; recheck the durable cancellation signal after stop and before start.
  - confidence: 0.83
