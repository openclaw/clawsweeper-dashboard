## Summary

Keep open: this PR targets a real gateway/channel restart race, but the current flag-based patch is still merge-blocked by lifecycle cancellation bugs and insufficient live proof for the final durable-flag version.

## What This Changes

The branch adds gateway restart-pending and module-level abort checks so deferred channel reloads skip channel stop/start during SIGUSR1 in-process restart.

## Best Possible Solution

Move cancellation into the config-reloader and shutdown/restart lifecycle so restart aborts and drains active hot reload work before channel teardown/startup, while only aborting channel reloads for SIGUSR1 paths that actually proceed to restart.

## Reproduction Assessment

Yes, at source level: the linked issue has a concrete EADDRINUSE timeline, current main does not drain active hot reload work, and the PR changes the implicated SIGUSR1/channel reload paths. I did not run a live Telegram/SIGUSR1 reproduction in this read-only review.

## Solution Assessment

No. The PR targets the right race, but a module-level flag set from SIGUSR1 handling is weaker than lifecycle-owned config-reloader cancellation/drain and currently mishandles ignored SIGUSR1 signals.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Abort channel reloads only when SIGUSR1 will restart:** `src/cli/gateway-cli/run-loop.ts:746`
  - body: This call runs before the handler knows whether an unauthorized external SIGUSR1 will be ignored. If external SIGUSR1 is disallowed, the handler returns without restarting or recreating reload handlers, leaving the module-level abort flag set so later channel config reloads skip stop/start even though no restart is in progress.
  - confidence: 0.92
- **[P1] Keep old reload cancellation alive until old work exits:** `src/gateway/server-reload-handlers.ts:229`
  - body: Resetting the process-global abort flag when new reload handlers are created can clear cancellation while an old onHotReload is still alive, because current configReloader.stop() does not await active reload work. The old deferred reload can then wake with cancellation false and still stop/start channels during the restart window this PR is meant to close.
  - confidence: 0.88
- **[P2] Return cancellation before plugin pre-stop continues:** `src/gateway/server-reload-handlers.ts:332-336`
  - body: waitForActiveWorkBeforeChannelReload() still returns void, so the plugin-reload beforeReplace path treats a cancelled wait the same as a successful drain and continues into stopChannel. Return a cancelled result or recheck before the pre-stop loop so plugin reloads cannot stop channels during restart after the new skip path fires.
  - confidence: 0.85
