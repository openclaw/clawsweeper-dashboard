## Summary

Keep open: the patch is not mergeable because the new restart-pending gate can go false as soon as SIGUSR1 is marked handled, before gateway shutdown and the replacement channel start finish, so it can miss the central stale deferred-reload race. The PR also has only unit-test proof, not real after-fix gateway behavior proof.

## What This Changes

The PR exports a gateway-restart-pending predicate, checks it around deferred channel reload/restart, and adds a mocked Vitest regression for the race.

## Best Possible Solution

Use a durable gateway reload shutdown/restart cancellation contract: abort or drain active config hot reload work and stop the config reloader before channel teardown, then skip channel stop/start when that cancellation is observed.

## Reproduction Assessment

Yes, at source level. The linked issue gives a concrete deferred reload plus gateway restart timeline, and current main still lets a running hot reload outlive shutdown without a cancellation context.

## Solution Assessment

No. The proposed gate checks transient restart emission/preparation state, but the run loop clears that state before the shutdown/restart lifecycle that stale reload work must avoid.
