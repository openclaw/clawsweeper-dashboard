## Summary

## Best Possible Solution

Use a lifecycle-owned cancellation/drain fix for the config reloader and gateway close ordering, or rework this branch so cancellation is scoped to accepted restarts and old reload lifetimes with live gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate reload aborts on accepted restarts:** `src/cli/gateway-cli/run-loop.ts:746`
  - body: This abort runs before the branch knows whether an external SIGUSR1 is allowed. When external restart is disabled, the handler returns without restarting but leaves `pendingChannelReloadAborted` set, so a later channel config reload in the same lifecycle can skip stop/start even though no restart will apply the new config.
  - confidence: 0.91
- **[P1] Keep cancellation scoped to the old reload lifecycle:** `src/gateway/server-reload-handlers.ts:229`
  - body: Resetting the process-global abort flag whenever a new reload handler is created lets the new in-process restart lifecycle clear the cancellation before old deferred reload work wakes up. That old work can then still reach channel stop/start after restart begins, preserving the duplicate-start race this PR is meant to fix.
  - confidence: 0.88
- **[P2] Return cancellation before plugin pre-stop continues:** `src/gateway/server-reload-handlers.ts:332-336`
  - body: The cancellation check logs and returns from `waitForActiveWorkBeforeChannelReload`, but callers cannot tell that the wait was cancelled. The plugin reload `beforeReplace` caller continues after the await and can still stop channels during restart instead of exiting the stale reload path.
  - confidence: 0.86
