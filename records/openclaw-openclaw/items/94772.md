Keep open: the bug is valid, but this PR is still too narrow because its restart-pending helper stops reporting true once the SIGUSR1 handler marks the signal handled, while stale deferred reload work can still run during shutdown.

## Best Possible Solution

Thread a restart/shutdown cancellation signal through active config reloads, abort or drain the config reloader before channel teardown, and make channel stop/start honor that cancellation before adding real gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Track restart lifecycle past SIGUSR1 consumption:** `src/infra/restart.ts:114`
  - body: `isGatewayRestartPending()` only checks `pendingRestartPreparing || hasUnconsumedRestartSignal()`, but the SIGUSR1 handler marks the token handled before it starts restart shutdown. A deferred reload that wakes during `server.close()` will see false and still run the stale channel stop/start path, which is the reported race window.
  - confidence: 0.92
- **[P2] Recheck cancellation after stopping the channel:** `src/gateway/server-reload-handlers.ts:513-518`
  - body: The new guard runs before `stopChannel`, but `stopChannel` awaits. If a restart becomes pending while that await is in flight, this reload still falls through to `startChannel`, so the duplicate-start race remains possible unless the cancellation is rechecked or shared across the whole stop/start sequence.
  - confidence: 0.84
