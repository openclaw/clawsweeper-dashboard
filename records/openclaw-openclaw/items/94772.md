## Summary

Keep open: the PR targets the valid gateway/channel restart race, but it is not merge-ready because the new pending-restart gate is too short-lived and is not checked at the final async boundary before channel start; contributor proof is also test-only.

## What This Changes

The PR exports a gateway restart-pending helper, uses it to skip deferred channel reload work during SIGUSR1 restart windows, and adds a regression test for mid-deferral cancellation.

## Best Possible Solution

Use a durable gateway reload cancellation/drain contract, or an equivalent shutdown/restart lifecycle signal, and gate the final channel start on that state before merging.

## Reproduction Assessment

Yes, at source level. The canonical report at https://github.com/openclaw/openclaw/issues/79487 gives a concrete log timeline, and current main shows deferred channel reload work can continue separately from SIGUSR1 restart shutdown; I did not run a live Telegram webhook reproduction.

## Solution Assessment

No. A short-lived restart-pending helper is only a partial mitigation; the maintainable fix is to cancel/drain active config reload work across the restart lifecycle and recheck before the final channel start.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use a durable restart lifecycle signal:** `src/infra/restart.ts:110-112`
  - body: `isGatewayRestartPending()` only covers pending preparation or an unconsumed SIGUSR1 marker. The run loop consumes that marker with `markGatewaySigusr1RestartHandled()` before it calls `request("restart", "SIGUSR1", ...)`, so a deferred channel reload that wakes during the actual shutdown window can see this helper return false and still start the channel, reproducing the EADDRINUSE race from https://github.com/openclaw/openclaw/issues/79487. Gate this on a cancellation/restart state that lasts until old reload work is aborted or drained.
  - confidence: 0.93
- **[P2] Recheck after stopping before startChannel:** `src/gateway/server-reload-handlers.ts:510-515`
  - body: The new restart gate in `restartChannel` runs before `await params.stopChannel(...)`. If SIGUSR1 restart becomes pending while that await is in flight, the continuation still calls `params.startChannel(name)` with no second check, leaving the duplicate-start race open. Add the cancellation check immediately before `startChannel` and cover that interleaving.
  - confidence: 0.86
