## Summary

Keep open: the PR targets the right gateway reload/restart race, but the current patch is still a partial mitigation because its restart-pending helper can go false before shutdown/restart finishes and some channel reload callers treat a skipped wait as success.

## What This Changes

The PR adds restart-pending checks to gateway channel hot-reload deferral and per-channel restart logic, exports a restart-pending helper, and adds a mocked regression test.

## Best Possible Solution

Use a gateway-owned reload cancellation/drain contract that remains active from restart acceptance through old-lifecycle shutdown, have reload callers distinguish cancelled waits from drained waits, and recheck cancellation immediately before channel start.

## Reproduction Assessment

Yes, at source level: the linked report gives a concrete EADDRINUSE timeline, and current source shows active hot reload callbacks are not aborted or drained before channel shutdown. I did not run a live Telegram/SIGUSR1 reproduction in this read-only review.

## Solution Assessment

No. The PR targets the right race, but the best fix is lifecycle-owned config-reloader cancellation/drain plus a final pre-start guard, not a helper based on a consumed SIGUSR1 token.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep restart cancellation alive through shutdown:** `src/infra/restart.ts:113-115`
  - body: `isGatewayRestartPending()` only tracks preparation or an unconsumed SIGUSR1 token. The run loop marks that token handled before shutdown/restart begins, so a deferred reload that wakes in the reported window can see `false` and still stop/start channels; use lifecycle cancellation that stays active until old reload work is stopped or drained.
  - confidence: 0.91
- **[P2] Signal skipped waits to plugin-reload callers:** `src/gateway/server-reload-handlers.ts:314-318`
  - body: The new early return from `waitForActiveWorkBeforeChannelReload()` is indistinguishable from a successful drain. The plugin reload pre-replace path awaits this helper and then stops channels, so a restart-pending skip can still run stale channel stop work during restart; return a cancelled result or gate the pre-stop path too.
  - confidence: 0.86
- **[P2] Recheck cancellation immediately before channel start:** `src/gateway/server-reload-handlers.ts:513-518`
  - body: The new guard runs before `await params.stopChannel(...)`. If SIGUSR1 is accepted while stop is in flight, this continuation still calls `startChannel(name)` and can race the restarted gateway; recheck the durable cancellation signal after stop and before start.
  - confidence: 0.83
