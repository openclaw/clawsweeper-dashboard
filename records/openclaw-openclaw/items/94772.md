## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Tie cancellation to the reload lifecycle:** `src/gateway/server-reload-handlers.ts:291-293`
  - body: `isGatewayRestartPending()` is only a transient signal-state check. The SIGUSR1 path consumes the restart token before the in-process restart finishes, and the config reloader does not abort or drain an active `runReload`, so this can return false while stale reload work still overlaps restart teardown/startup. The fix needs a durable reload cancellation/drain owned by the gateway lifecycle, not just polling this helper.
  - confidence: 0.9
- **[P2] Recheck restart cancellation before starting:** `src/gateway/server-reload-handlers.ts:520-523`
  - body: This check happens before `await params.stopChannel(name)`, but if a restart becomes pending while the stop is awaiting, the stale reload path still continues into `startChannel(name)`. Add a second cancellation check immediately after stop and before start, with coverage for that interleaving.
  - confidence: 0.87
