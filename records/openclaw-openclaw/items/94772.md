## Summary

## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep restart cancellation alive through shutdown:** `src/infra/restart.ts:112-114`
  - body: `isGatewayRestartPending()` only observes `pendingRestartPreparing` or an unconsumed restart signal. The SIGUSR1 handler marks that signal handled before it calls into the restart request, so a deferred reload that wakes a few hundred milliseconds later during shutdown can see `false` and still run `stopChannel`/`startChannel`, preserving the EADDRINUSE race this PR is meant to fix.
  - confidence: 0.9
- **[P1] Return a cancelled wait to plugin reload callers:** `src/gateway/server-reload-handlers.ts:290-296`
  - body: The new skip path returns `void` from `waitForActiveWorkBeforeChannelReload`, but `stopChannelsBeforePluginReplace` treats any return as a successful wait and immediately proceeds to stop channels before plugin replacement. If a restart is pending here, the PR can still stop channels during the restart path instead of cancelling the reload work.
  - confidence: 0.86
- **[P2] Recheck cancellation immediately before channel start:** `src/gateway/server-reload-handlers.ts:510-514`
  - body: The added restart-pending guard runs before `stopChannel`, but the restart signal can arrive after that await and before `startChannel`. In that interleaving this stale hot-reload path still starts the old channel while the in-process restart starts the new lifecycle, so add a cancellation check after the stop and before the start.
  - confidence: 0.78
