## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Track restart state through shutdown:** `src/infra/restart.ts:113-114`
  - body: `isGatewayRestartPending()` returns false once the SIGUSR1 token is handled, but the run loop marks it handled before it calls `request("restart")` and closes the server. A deferred reload waking during that shutdown window will still run stale channel stop/start, leaving the reported EADDRINUSE race open; keep a restart/shutdown lifecycle flag true until the next start/reset.
  - confidence: 0.92
- **[P2] Recheck cancellation after stopping channels:** `src/gateway/server-reload-handlers.ts:513-518`
  - body: The new guard runs before `stopChannel`, but that call is awaited. If SIGUSR1 becomes pending during the await, the reload still falls through to `startChannel`, so duplicate starts can still happen; recheck cancellation immediately before `startChannel` or carry an abort signal through the whole stop/start sequence.
  - confidence: 0.84
- **[P2] Return cancellation from the deferral wait:** `src/gateway/server-reload-handlers.ts:291-295`
  - body: When `waitForActiveWorkBeforeChannelReload` returns because a restart is pending, callers cannot distinguish that from a successfully drained wait. The plugin-reload `beforeReplace` path can continue into channel stops after this “skip”; return a typed cancellation result and stop the caller.
  - confidence: 0.78
