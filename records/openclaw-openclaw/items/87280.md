## Summary

Keep open. The change targets a real gateway auth-rotation gap, but the current patch can reject newly tagged browser device tokens on the Control UI assistant-media HTTP path and still lacks real browser/gateway proof.

## What This Changes

The PR adds issuer-generation metadata to browser device tokens minted from shared gateway auth, rejects stale generations during verification, and adds pairing-layer regression tests.

## Best Possible Solution

Keep the issuer-generation model, but update every browser-token validation path to receive the required gateway generation or preserve tagged tokens outside rotation checks, then prove the live Control UI/WebChat rotation flow.

## Reproduction Assessment

Yes for the review finding by source inspection: PR-head verifyDeviceToken rejects any tagged shared-gateway token when requiredSharedGatewaySessionGeneration is omitted, and current Control UI assistant-media auth omits it. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No. The issuer-generation design is directionally right, but the implementation needs sibling HTTP validation coverage before this is the narrow maintainable fix.
