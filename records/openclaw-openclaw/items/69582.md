## Summary

Keep open. Current main still has a source-backed path where malformed `process(action={})` is routed through generic process handling instead of a schema-aware action error, and the broader GLM `{}` contamination claim still needs a raw provider transcript before adding provider-specific cleanup.

## What This Changes

The issue requests preventing malformed `{}` tool parameters from contaminating tool calls and causing repeated process/edit invocation failures with local GLM models.

## Best Possible Solution

Add narrow schema-aware validation for invalid `process.action` type/value at the tool invocation boundary, then add GLM/Z.AI-specific argument repair only if a raw streamed-argument fixture proves provider-side malformed payloads.

## Reproduction Assessment

Unclear for the full GLM contamination path because the report does not include raw provider/tool-call payloads. The reported `process(action={})` symptom is source-reproducible: current main casts raw args and can return the generic missing-`sessionId` result before validating the action type.

## Solution Assessment

No, blanket parameter cleaning is not the best way to solve this. The safer fix is targeted runtime validation with non-retry guidance for invalid `process.action`, plus fixture-backed provider repair only for malformed whole tool-call payloads.
