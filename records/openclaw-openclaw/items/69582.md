## Summary

Keep open. Current main still does not provide the default runtime validation or safe repair needed for the reported GLM `{}` argument contamination: `process(action={})` can still reach the misleading missing-sessionId path, and the generic loop detectors remain opt-in. The prior ClawSweeper comment is still directionally correct on this newer main SHA; later unknown-tool loop fixes cover a different failure class.

## What This Changes

The issue reports local GLM-5 tool argument contamination where tool parameters gain trailing `{}` or become `{}`, causing repeated failed tool invocations.

## Best Possible Solution

Add narrow schema-aware validation at the OpenClaw-owned tool invocation boundary, starting with `process.action` wrong-type handling and clear non-retry guidance. Only add GLM/Z.AI-specific argument repair if a streamed-argument fixture proves the `{}` suffix is a provider parse artifact, and preserve literal `{}` in valid file content.

## Reproduction Assessment

Unclear for the full GLM injection because there is no raw provider transcript. There is a high-confidence focused source reproduction for the `process(action={})` symptom: current main accepts the malformed object far enough to return the generic missing-sessionId failure instead of a type-specific validation error.

## Solution Assessment

No, the proposed blanket parameter cleaning is not the best way to solve this because `{}` can be valid user text. The maintainable fix is targeted runtime validation/diagnostics for invalid tool-argument types, plus provider-specific repair only if backed by a fixture.
