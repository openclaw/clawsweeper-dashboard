## Summary

Keep this PR open. Current main still lacks the PR's intended structured Slack file ID propagation from inbound Slack media into agent message context; the existing implementation only exposes file IDs in placeholder text, while the message tool schema can now accept a caller-supplied `fileId`. The PR is useful but needs maintainer review because it extends the shared agent-facing context contract and is stale against the current Slack media type file layout.

## What This Changes

The PR adds optional Slack file IDs to `SlackMediaResult`, copies `file.id` from `resolveSlackMedia()`, and exposes `MediaFileId`/`MediaFileIds` on the finalized message context.

## Best Possible Solution

Rebase the PR onto current main, add `fileId?: string` to the current Slack media type, propagate it through `resolveSlackMedia()` and `prepareSlackMessage()` using the maintainer-approved context field shape, then add focused Slack media/context tests and a changelog entry before merge.

Yes. A source-level reproduction is high confidence: on current main, a Slack file with `id` is rendered into placeholder text, but `SlackMediaResult` and `prepareSlackMessage()` do not expose `MediaFileId` or `MediaFileIds` in structured context.

## Solution Assessment

Unclear. The PR's direction is plausible and additive, but it is not yet the best merge-ready solution until maintainers approve the shared context API shape and the branch is rebased with tests and changelog coverage.
