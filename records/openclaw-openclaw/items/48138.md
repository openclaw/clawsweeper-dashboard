## Summary

Keep this PR open, but do not merge it as-is. The PR fills a real Feishu document-management gap and the earlier `old_owner_perm` default concern has been repaired, but it exposes document ownership transfer through the default-enabled `feishu_doc` tool with caller-selected target principals, which conflicts with the current Feishu permission boundary and adjacent trusted-requester hardening.

## What This Changes

The PR adds a `transfer` action to `feishu_doc`, wires it to Lark `drive.permissionMember.transferOwner`, defaults omitted `old_owner_perm` to `full_access`, adds unit coverage for that payload, and adds a changelog entry.

## Best Possible Solution

Support Feishu ownership transfer only after a maintainer chooses the security boundary, preferably by moving the operation behind the default-disabled permission-management surface or binding the transfer target to trusted Feishu requester or owner-approved context, then document the required `docs:permission.member:transfer` scope and add tests for both the SDK payload and denied/unauthorized target behavior.

## Reproduction Assessment

Yes. On current main, `transfer` is absent from the schema and dispatch table, so the feature gap is reproducible by source inspection; the PR security concern is reproducible from the diff because it accepts caller-controlled `member_type` and `member_id` under the default-enabled document tool.

## Solution Assessment

No. The SDK method and `old_owner_perm` default are technically aligned, but exposing arbitrary ownership transfer through `feishu_doc` is not the narrowest maintainable solution; a default-disabled permission boundary or trusted-requester-bound transfer target is safer.
