## Summary

Keep this PR open for maintainer review: it adds a useful Feishu ownership-transfer capability, but the current diff exposes caller-selected ownership transfer through the default-enabled document tool instead of an explicitly sensitive permission boundary.

## What This Changes

The PR adds a `transfer` action to `feishu_doc`, wires it to Lark `permissionMember.transferOwner`, adds a defaulted `old_owner_perm`, unit coverage, and a changelog entry.

## Best Possible Solution

Expose Feishu ownership transfer only through a sensitive permission-management surface or a trusted-requester/owner-approved flow, with docs and tests for allowed and denied behavior.

## Reproduction Assessment

Yes for the review finding. Source inspection shows current main has no transfer schema or dispatch path, and the PR diff shows the new caller-controlled `member_type` and `member_id` flow into `transferOwner`.

## Solution Assessment

No. The SDK wiring and default handling are focused, but placing arbitrary ownership transfer in default-enabled `feishu_doc` is not the safest maintainable boundary.
