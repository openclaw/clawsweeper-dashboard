## Summary

Keep this PR open for maintainer review. Current main still lacks Feishu document ownership transfer, and the PR is a focused contribution, but it adds a caller-selected ownership transfer path to the default-enabled document tool despite existing Feishu permission-management hardening.

## What This Changes

This PR adds a `transfer` action to `feishu_doc`, dispatches it to `client.drive.permissionMember.transferOwner`, adds a shared `old_owner_perm` default, covers that default in `docx.test.ts`, and adds a changelog entry.

## Best Possible Solution

Support Feishu ownership transfer only after a maintainer decides the security boundary, then land it with the transfer action gated consistently with sensitive permission management, documented `docs:permission.member:transfer` scope guidance, and tests for both the SDK payload and unauthorized/cross-context caller behavior.

## Reproduction Assessment

Yes. On current main, the schema and dispatch have no `transfer` action, so a transfer request would be rejected as an unknown action; the PR security concern is reproducible by reading the added schema and dispatch that accept caller-supplied `member_id`.

## Solution Assessment

No. The SDK method choice and `old_owner_perm` default are technically aligned, but exposing arbitrary ownership transfer through default-enabled `feishu_doc` is not the narrowest maintainable security boundary; a default-disabled permission surface or trusted-requester-bound target is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate ownership transfer behind trusted permission policy:** `extensions/feishu/src/doc-schema.ts:192`
  - body: `feishu_doc` is enabled by default, but this new schema lets any tool caller choose `member_type` and `member_id` for `transferOwner`. That reopens caller-selected principal changes in the document tool, while current main puts permission management behind default-disabled `feishu_perm` and binds doc-create grants to trusted Feishu requester context. Please move this behind the sensitive permission boundary or restrict the target to verified requester context before exposing it.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive document ownership transfer path in a default-enabled tool with caller-controlled target principals.

Concerns:

- **[high] Default-enabled ownership transfer to arbitrary principal:** `extensions/feishu/src/doc-schema.ts:192`
  - body: The new `transfer` action accepts caller-controlled `member_type` and `member_id` under `feishu_doc`, while current configuration enables `doc` by default and disables `perm` because it is sensitive. For bot-owned documents, this could let an allowed prompt transfer ownership to an attacker-chosen Feishu account unless the action is moved behind the permission tool boundary or bound to trusted requester context.
