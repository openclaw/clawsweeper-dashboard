## Summary

Current main and release v2026.6.5 still use full accumulated text for Feishu CardKit streaming updates, matching the reported payload-growth mechanism. The issue also has a clean open fixing PR, so it remains necessary while maintainers decide whether a tenant-dependent append mode is an acceptable config surface and require live Feishu proof of both latency improvement and complete final content.

## What This Changes

Add a safe opt-in Feishu streaming-card mode that sends appended content for latency-sensitive deployments while preserving full-content updates as the default.

## Best Possible Solution

Keep full-content updates as the documented safe default, then accept a narrowly scoped mitigation only after a credentialed Feishu comparison proves materially lower long-reply latency and complete update/final content; first consider whether bounded batching or adaptive throttling can solve the performance problem without a tenant-specific permanent config knob.

## Reproduction Assessment

Partly yes: current source and tests reproduce the cumulative full-content payload mechanism with high confidence, but the reported latency magnitude and append semantics require a credentialed live Feishu tenant to reproduce.

## Solution Assessment

Unclear: the proposed opt-in append mode is plausible and preserves the default, but it is not yet proven better than batching or adaptive throttling and lacks live proof that it avoids truncation through final close.
