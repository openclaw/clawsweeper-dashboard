## Summary

Current main and release v2026.6.5 still send full accumulated text for each Feishu streaming-card update, matching the reported payload-growth mechanism. The same author’s open closing PR owns the proposed mitigation, so this issue remains necessary while maintainers decide whether a tenant-dependent append mode is an acceptable permanent config surface and require live Feishu proof of both latency improvement and complete content.

## What This Changes

Request an opt-in Feishu streaming-card mode that sends suffix-only updates to reduce severe long-reply latency while preserving full-content updates as the default.

## Best Possible Solution

Keep full-content updates as the safe default and land a Feishu-owned mitigation only after a credentialed long-reply comparison proves lower latency and complete update/final content; maintainers should also compare bounded batching or adaptive throttling before accepting a permanent tenant-specific mode.

## Reproduction Assessment

No, not at high confidence in this checkout: source and tests reproduce the cumulative full-payload mechanism, but the reported latency magnitude and tenant-specific append behavior require a credentialed live Feishu environment.

## Solution Assessment

Unclear: preserving full mode by default is safer than reverting it, but an opt-in append setting is only the best solution if live proof shows it materially improves latency without reviving truncation; batching or adaptive throttling may avoid a permanent config fork.
