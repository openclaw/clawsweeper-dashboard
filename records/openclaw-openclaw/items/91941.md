## Summary

The issue remains necessary and is correctly paired with an open implementation PR. Current main sends growing full-content payloads as required by the documented CardKit contract, but the reported live latency regression has not been independently reproduced and the proposed append mode may trade latency for the truncation behavior that full-content updates fixed.

## What This Changes

Address severe long-reply Feishu streaming-card latency after full accumulated text replaced suffix-only CardKit updates, potentially through an opt-in append mode.

## Best Possible Solution

Reproduce the slowdown on exact current main in a live Feishu tenant, capture request count, cumulative payload sizes, and per-update duration, then optimize supported full-content streaming through batching, lower update frequency, or CardKit streaming parameters; expose append mode only if maintainers explicitly accept and document its tenant-specific truncation risk with positive live proof.

## Reproduction Assessment

No, not at high confidence yet. Current source and the shipped release confirm cumulative full-content updates, but only the reporter's deployment establishes the claimed latency magnitude, and the linked PR did not perform an exact-head live Feishu test.

## Solution Assessment

Unclear. The proposed opt-in append mode is bounded and preserves the default, but it conflicts with the documented CardKit payload contract and may reintroduce truncation; tuning the cadence of supported full-content updates is the safer first solution to validate.
