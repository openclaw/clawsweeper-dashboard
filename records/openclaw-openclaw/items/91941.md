## Summary

Current main still has the cumulative-payload behavior, and the report supplies credible live regression evidence. The paired open PR owns the proposed mitigation but has not yet demonstrated the latency improvement and complete final content in a live Feishu tenant, so this issue remains necessary.

## What This Changes

Add an opt-in append-only Feishu streaming-card mode to reduce long-reply latency while retaining full-content updates as the default.

## Best Possible Solution

Keep full-content updates as the default; accept an opt-in append escape hatch only with clear compatibility warnings and live Feishu proof showing both materially improved long-reply latency and complete final card content.

## Reproduction Assessment

Yes for the mechanism: current main repeatedly transmits the accumulated response, and the reporter provides concrete live timings plus a rollback comparison. The exact tenant-side 20–58 second latency has not been independently reproduced.

## Solution Assessment

Unclear. The proposed opt-in mode is narrowly scoped, but it restores behavior with known cross-deployment content risks and therefore requires live correctness proof and a maintainer product decision.
