## Summary

Keep open: current main still discards Control UI composer drafts on session switch and this PR is the canonical remaining draft-preservation work, but the branch is conflicting, lacks real behavior proof, and still has correctness/privacy gaps.

## What This Changes

The PR adds a module-level in-memory per-session chat draft cache and wires save/restore calls into several Control UI session and agent switch paths.

## Best Possible Solution

Land one focused Control UI implementation that preserves drafts through the shared session-switch lifecycle, scopes or clears cached draft text on deletion/context changes, and covers session, agent, and sidebar switching in tests.

## Reproduction Assessment

Yes, at source level. On current main, switchChatSession() reaches resetChatStateForSessionSwitch(), which assigns state.chatMessage = "", so a typed draft is discarded when switching sessions.

## Solution Assessment

No, not as submitted. The save/restore direction is reasonable, but the final fix should be rebased onto the centralized session-switch helper, include cache cleanup/scoping, add regression coverage, and supply real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve sidebar session drafts:** `ui/src/ui/app-render.ts:1438`
  - body: As submitted, the PR-head chat sidebar onSessionSelect path still switches sessions directly instead of using the new save/restore flow. Selecting a sidebar session can carry the outgoing draft into the destination session and skip restoring that destination session's draft, so the visible feature remains incomplete.
  - confidence: 0.86
- **[P3] Clear deleted sessions from the draft cache:** `ui/src/ui/app-render.helpers.ts:37`
  - body: The new module-level draft cache never prunes entries when sessions are deleted. Add a clear helper and call it after successful session deletion so long-lived tabs do not retain removed-session text until reload.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No supply-chain or code-execution risk was found, but the proposed draft cache has a low-severity local draft-isolation concern.

Concerns:

- **[low] Scope cached drafts to the active UI context:** `ui/src/ui/app-render.helpers.ts:37`
  - body: The PR stores unsent text in a module-level Map keyed only by sessionKey. In a long-lived browser tab, reused keys such as main after reconnecting to another gateway or workspace could restore prior draft text in the wrong context unless the cache is scoped or cleared.
  - confidence: 0.7
