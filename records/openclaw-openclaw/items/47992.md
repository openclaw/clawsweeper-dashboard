## Summary

Keep open. Current main still clears chatMessage on session switches, so the user problem remains, but the PR head is conflicting/stale, lacks real behavior proof, and has concrete draft coverage and cache-scoping issues before merge.

## What This Changes

The PR adds an in-memory Map-backed chat draft cache and wires save/restore calls into Control UI session and agent switching callbacks.

## Best Possible Solution

Ship one rebased Control UI change that preserves composer drafts in the centralized session-switch helper, clears removed or empty session drafts, keeps queue behavior intact, and adds focused regression coverage for session, agent, and sidebar switches.

## Reproduction Assessment

Yes, at source level. Setting state.chatMessage and switching sessions on current main reaches resetChatStateForSessionSwitch(), which clears the draft with state.chatMessage = "".

## Solution Assessment

No, not as written. The save/restore direction is reasonable, but the PR head misses a visible sidebar selection path, needs cache scoping/cleanup, is stale against current main, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve drafts from sidebar session selection:** `ui/src/ui/app-render.ts:1438`
  - body: The PR head still switches chat-sidebar selections with state.setSessionKey(key) directly. Selecting a session from the sidebar can carry the outgoing draft into the destination session and does not restore the destination session's saved draft, so the proposed feature is incomplete for a visible switch path.
  - confidence: 0.86
- **[P3] Clear removed sessions from the draft cache:** `ui/src/ui/app-render.helpers.ts:37`
  - body: chatDraftCache is module-level state with no eviction hook for deleted sessions. Drafts for removed sessions would stay in memory until page reload; expose a clear helper and call it from the session deletion flow to keep the cache bounded.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No supply-chain or code-execution risk was found, but the draft cache introduces a low local draft-isolation concern.

Concerns:

- **[low] Scope cached drafts to the active UI context:** `ui/src/ui/app-render.helpers.ts:37`
  - body: The PR stores unsent text in a module-level Map keyed only by sessionKey. In a long-lived browser tab, reused keys such as main after reconnecting or changing gateway/workspace could restore a prior draft in the wrong context unless the cache is scoped or cleared.
  - confidence: 0.7
