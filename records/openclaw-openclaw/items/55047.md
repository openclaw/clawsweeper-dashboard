## Summary

Keep open. Current main still has a credible Control UI blank-screen crash path matching the report and the prior discussion: malformed persisted cron jobs can reach shared render state, and the shared Control UI render plus Cron edit/list consumers still dereference `job.payload.kind` without guarding the payload shape. The linked PR #54552 was useful context but is closed unmerged, so the observable bug is not resolved on main.

## What This Changes

The issue reports that Control UI left-menu sections such as channels, instances, sessions, agents, and skills now render blank pages instead of editable configuration views after updating OpenClaw 3.24 on Ubuntu.

## Best Possible Solution

Make cron data exposure and Control UI consumption resilient to malformed persisted cron rows: normalize, filter, or annotate invalid rows before `cron.list`/`cron.listPage` exposes them; guard shared render and Cron edit/list paths before reading payload fields; and add regression coverage proving malformed cron rows cannot blank unrelated Control UI sections.

## Reproduction Assessment

Yes. A persisted cron job with a missing or non-inferrable payload can flow through cron store/list into `state.cronJobs`, and `renderApp` then reads `job.payload.kind` before rendering the selected tab, which is a high-confidence code-level reproduction path. The exact reporter data is missing, but it is not needed to show this remaining failure class.

## Solution Assessment

Yes, the best fix is a narrow repair across the cron data boundary and the UI consumption sites, not a broad Control UI rewrite. The closed #54552 direction was useful but too narrow and unmerged; current main needs a complete guard/filter plus tests.
