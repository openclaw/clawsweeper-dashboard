## Summary

Keep open: current main and v2026.6.8 still have finite subagent announce retry/give-up behavior and a pending direct-handoff branch that can still report an undelivered completion. The issue is also paired with open, unmerged implementation PRs, so the safe maintainer path is to review or replace those fixes rather than close the canonical report.

## What This Changes

The issue asks OpenClaw to stop silently losing subagent completion results by retrying or surfacing announce failures, notifying or restarting on timeout, and avoiding unverified completion claims.

## Best Possible Solution

Keep this as the broad canonical reliability issue until maintainers land or replace the open subagent completion delivery fixes with one durable, exactly-once requester-visible completion/failure model and split any remaining timeout restart or file-verification policy into narrower follow-ups.

## Reproduction Assessment

Yes, at source level: current main still has finite retry-limit/expiry give-up handling and a completion_handoff_pending path that returns delivered:false. I did not run a live current-main Telegram/Slack/Discord reproduction in this read-only review.

## Solution Assessment

Unclear as a complete solution. The narrow best fix is a canonical exactly-once requester-visible completion or failure event on terminal give-up; auto-restart and file-claim verification likely need separate maintainer decisions or narrower issues.
