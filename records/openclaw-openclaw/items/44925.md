## Summary

Keep this open: current main has partial recovery work, but it still retains finite announce retry/give-up behavior and current docs still describe sub-agent announce as best-effort, while multiple related implementation PRs remain open and unmerged.

## What This Changes

The issue asks OpenClaw to stop silently losing subagent completions by retrying or surfacing announce failures, notifying or restarting on timeout, and verifying file-creation claims before completion is announced.

## Best Possible Solution

Keep this issue as the umbrella until maintainers choose and land a proof-positive single-delivery/durable-recovery path, then close it from the merged PR that covers announce failure, timeout notification, and requester wake semantics.

## Reproduction Assessment

Yes, by source inspection and issue discussion: a sessions_spawn/sessions_yield flow can still reach retry-limit or expiry paths, and current docs still treat announce delivery as best-effort. I did not run a live Telegram/Slack/Discord repro in this read-only review.

## Solution Assessment

No, current main is only a partial solution: it preserves and steers some pending completions, but the durable retry/notification/timeout-recovery semantics still need a maintainer-chosen implementation. The best fix is a single canonical recovery path with duplicate-delivery protection and live channel proof.
