## Summary

Keep open. Current main has meaningful partial hardening, but source, docs, tests, and fresh OpenClaw 2026.5.20 Telegram evidence still show unresolved best-effort subagent completion delivery with finite retry exhaustion, suspended/discarded final delivery, timeout/recovery gaps, and message-tool delivery failures.

## What This Changes

The issue asks for reliable subagent completion handling so completed, timed-out, pruned, or file-producing child work reaches the requester visibly or fails with an explicit terminal notification.

## Best Possible Solution

Keep this as the broad reliability tracking issue until OpenClaw has one durable, idempotent async completion contract covering child completion, requester resumption, external delivery confirmation, timeout/prune notification, duplicate prevention, and result/file verification.

## Reproduction Assessment

Yes, at source level. Current main still has finite retry exhaustion, suspended-delivery discard, message-tool-only failure states, timeout terminalization, and a Claude stream-json parser path that does not finalize on `message_stop`/`end_turn`; recent issue comments add release-level production traces.

## Solution Assessment

No for a narrow retry-only, notify-only, or parser-only patch. The maintainable fix is a single durable and idempotent completion/delivery state model with explicit success, failure, timeout, restart, and duplicate-prevention semantics.
