## Summary

Keep open. Current main has meaningful subagent completion-delivery hardening, but the central failure family is still live: retry exhaustion can still become suspended or failed delivery, suspended final delivery can later be discarded, timeout handling finalizes without an auto-restart policy, and the latest comments add current production evidence for delayed requester resumption and a CLI stream-json parser drop path.

## What This Changes

The issue asks for reliable subagent completion handling: retry and notify on announce failure, notify or restart on timeout, and verify claimed file/result outputs before announcing completion.

## Best Possible Solution

Keep this as the canonical tracker for a durable, deduped subagent completion state model that separates child completion, requester-session resumption, external delivery confirmation, timeout/restart policy, parser turn finalization, and result/file verification.

## Reproduction Assessment

Yes, source-level. Current main still has finite retry give-up, suspended delivery that can expire or be discarded, timeout finalization without restart, and a stream-json parser path that can miss completed long-lived CLI turns without a final result line; recent comments add current Telegram production evidence.

## Solution Assessment

No for a narrow retry-only or notify-only patch. The maintainable fix needs an explicit durable delivery and resumption state model with dedupe, timeout policy, parser turn finalization, and verification semantics, while narrow PRs can land as components of that design.
