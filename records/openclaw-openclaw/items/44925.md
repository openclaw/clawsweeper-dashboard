## Summary

Keep open. Current main has meaningful partial hardening, but it still implements and documents best-effort, finite subagent completion delivery with retry-limit give-up or suspended delivery, timeout terminalization instead of restart, and fresh OpenClaw 2026.5.20 reports show the same Telegram/Discord/macOS failure family.

## What This Changes

Requests reliable subagent completion handling so successful, timed-out, pruned, or artifact-producing child work resumes/notifies the requester instead of disappearing silently.

## Best Possible Solution

Keep this as the canonical broad reliability issue until OpenClaw has one durable, idempotent async completion contract covering child completion, requester resumption, external delivery confirmation, timeout/prune notification, duplicate prevention, and result/artifact verification.

## Reproduction Assessment

Yes, at source level. Current main still has finite retry exhaustion, suspended-delivery discard/blocked states, timeout terminalization without restart, and a Claude stream-json parser path that does not finalize on message_stop/end_turn alone; recent comments add OpenClaw 2026.5.20 production traces.

## Solution Assessment

No for a narrow retry-only, notify-only, or parser-only patch. The maintainable fix is a single durable and idempotent completion/delivery state model with explicit success, failure, timeout, restart, artifact verification, and duplicate-prevention semantics.
