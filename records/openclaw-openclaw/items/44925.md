## Summary

Keep open. Current main contains partial subagent delivery hardening, but the reported failure family is still present: announce delivery has a finite retry-limit give-up path, timeout handling finalizes a timed-out task instead of restarting it, docs still describe sub-agent announce as best-effort, and the latest discussion adds current 2026.4.26 production evidence for the same retry-limit failure mode.

## What This Changes

The issue requests reliable subagent completion handling so announce failures, timeouts, and unverifiable result/file claims are retried, surfaced, restarted, or verified instead of being silently lost.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is a durable completion and delivery model that separates child completion, parent-session recording, external channel delivery, retry persistence, timeout/restart policy, deduplication, retained error categories, and result/artifact verification, with regression coverage for fast responders and Telegram-style async delivery.

## Reproduction Assessment

Yes. The April 29 comment gives a concrete two-agent `sessions_spawn` fast-responder path for the retry-limit failure, and current main still contains the finite retry/give-up path that matches it; live Telegram proof was not run in this read-only review.

## Solution Assessment

No. Current main is partial hardening, and open PR #45200 covers only one symptom; the maintainable fix needs a clear durable completion/delivery policy before a narrow repair PR can safely cover the whole issue.
