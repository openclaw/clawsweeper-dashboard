## Summary

Keep open. Current main has partial subagent delivery hardening, including a pending-final-delivery marker, but the reported failure family is not fully resolved: completion delivery can still hit bounded give-up, failed task delivery status, orphan pruning, terminal timeout handling, and no general verification of claimed file outputs.

## What This Changes

The issue requests reliable subagent completion handling so announce failures retry or surface, timeouts notify or recover, and claimed file outputs are verified.

## Best Possible Solution

Keep this as the broad reliability tracker until maintainers define an explicit async completion state model that persists a self-contained result before cleanup, retries or expires delivery deliberately, prevents duplicate user-visible sends, surfaces terminal failures/timeouts, and defines artifact/file-claim verification semantics.

## Reproduction Assessment

Yes for the main delivery-loss paths at source level: current main still has direct-failure plus queue-none dispatch, retry-limit give-up, failed delivery status, orphan pruning, and timeout finalization paths that match the reported logs. I did not run a live Telegram reproduction, and the file-claim verification pattern remains less directly reproducible from source alone.

## Solution Assessment

No. Current main is a partial hardening, but a notify-only or retry-count-only fix is not enough; the maintainable solution needs a durable delivery state model, explicit timeout policy, deduplication, and result/artifact verification rules.
