## Summary

Keep open. Current main still has source-backed completion-delivery loss windows despite several partial hardening patches, and the newest discussion reports the same class on OpenClaw 2026.5.20 with timeout, prune, and missing notification symptoms.

## What This Changes

The issue asks OpenClaw to stop losing asynchronous subagent completions by retrying or surfacing announce failures, notifying or recovering on timeouts, and verifying claimed file outputs.

## Best Possible Solution

Keep this as the broad tracking issue until OpenClaw has one durable async completion contract covering child completion, requester resumption, visible delivery confirmation, timeout/prune terminal events, idempotent replay, deduplication, and result/file verification.

## Reproduction Assessment

Yes, source-level. Current main still has finite retry-limit give-up, failed/suspended delivery terminal paths, suspended-delivery discard, orphan prune, and a Claude stream-json parser path that does not finalize on message_stop/end_turn; recent comments add production traces on the latest release.

## Solution Assessment

No for a narrow retry-only, notify-only, or parser-only patch. The maintainable fix is a single delivery/resumption state model with durable replay, explicit success/failure semantics, timeout/prune policy, result verification, and duplicate prevention.
