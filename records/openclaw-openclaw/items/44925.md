## Summary

Keep open. Current main has meaningful hardening, but it still has finite announce retry exhaustion, suspended/failed completion delivery states, timeout terminalization without a clear restart policy, and fresh 2026.5.20 reports of the same Telegram/macOS-visible failure class.

## What This Changes

The issue asks OpenClaw to stop silently losing or delaying subagent completions by retrying, notifying, restarting or failing visibly, and verifying claimed file/result delivery.

## Best Possible Solution

Keep this as the broad reliability tracker until OpenClaw has one durable, idempotent async completion contract covering child completion, requester resumption, external delivery confirmation, timeout/prune notification, duplicate prevention, and result/file verification.

## Reproduction Assessment

Yes, at source level. Current main still has finite retry exhaustion, suspended/failed delivery bookkeeping, terminal timeout cleanup without auto-restart, and tests that encode the retry-limit behavior; recent comments add live 2026.5.20 production traces.

## Solution Assessment

No for a narrow retry-only, notify-only, or parser-only patch. The maintainable fix is a single durable and idempotent completion/delivery state model with explicit success, failure, timeout, restart, and duplicate-prevention semantics.
