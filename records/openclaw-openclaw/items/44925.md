## Summary

Keep open. Current main still has a finite subagent announce retry budget and terminal give-up path, while recent issue comments report the same retry-limit failure and delivery_failed task records on recent releases; existing hardening is partial rather than a complete reliable async completion model.

## What This Changes

The issue asks OpenClaw to stop silently losing async subagent completions by retrying or surfacing failed announces, handling timeouts visibly, and verifying claimed file results.

## Best Possible Solution

Keep this as the canonical reliability issue for an explicit async completion state model covering child completion, parent recording, exactly-once user-visible delivery, timeout policy, deduplication, and result/artifact verification.

## Reproduction Assessment

Yes. The April 29 comment gives a concrete fast-responder two-agent sessions_spawn path, and current main still contains the finite retry-limit give-up path that matches the reported logs; I did not run live Telegram proof in this read-only review.

## Solution Assessment

No. Current main has useful fallback hardening, but it still terminates in delivery_failed/give-up states and does not define the durable delivery, restart/timeout, and artifact-verification semantics needed to solve the whole issue safely.
