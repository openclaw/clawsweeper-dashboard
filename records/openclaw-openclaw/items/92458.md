## Summary

The regression affects v2026.6.5, but the already-published v2026.6.6 release and current main accept the migrated `provider: "openai"` OAuth credential, so Codex-backed agents no longer hit the reported validator rejection.

## What This Changes

Fix the provider-unification upgrade path so migrated OpenAI OAuth profiles remain usable by Codex-backed agents.

## Best Possible Solution

Keep the canonical unified `openai` OAuth profile and direct affected v2026.6.5 users to upgrade to v2026.6.6 or later rather than restoring the retired `openai-codex` runtime shape.

## Reproduction Assessment

Yes for v2026.6.5: the reported migrated-store shape and validator path are specific and internally consistent. The same failure is no longer expected on v2026.6.6 or current main because `provider: "openai"` is accepted.

## Solution Assessment

No to the proposed migration re-tagging and parallel runtime compatibility shapes. The shipped solution correctly keeps `openai` canonical and aligns the Codex validator with that migrated state.
