## Summary

Current main and v2026.6.6 already solve the central bug: migrated unified `openai` OAuth profiles are accepted by the Codex app-server auth validator, so `openai:chatgpt-default` no longer hits the reported rejection path.

## What This Changes

The issue reports that the OpenAI provider-unification migration rewrites legacy Codex OAuth profiles into `openai:chatgpt-default`, which v2026.6.5 rejected during Codex-backed agent launch.

## Best Possible Solution

Keep the v2026.6.6/current-main unified OpenAI auth behavior; users still seeing this on v2026.6.5 should upgrade and reopen with redacted store and launch logs only if v2026.6.6 still fails.

## Reproduction Assessment

Yes for the reported release path by source inspection: the migration writes unified `openai` OAuth profiles and the launch validator must accept them. No current-main failure remains because v2026.6.6 accepts the migrated provider shape.

## Solution Assessment

Yes. Accepting unified `openai` as Codex app-server auth is the narrow maintainable fix because it preserves the provider-unification migration instead of restoring legacy `openai-codex` runtime ownership.
