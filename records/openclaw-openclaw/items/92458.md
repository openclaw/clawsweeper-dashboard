## Label Justifications

- `P1`: The v2026.6.5 upgrade path could make previously working Codex-backed agents fail immediately at launch.
- `impact:auth-provider`: The issue concerns an OAuth profile provider-identity mismatch introduced during provider migration.

## Summary

Current main and v2026.6.6 resolve the reported upgrade failure: the Codex app-server bridge accepts the canonical OpenAI OAuth identity produced by provider unification, so migrated agents no longer fail the profile validator.

## What This Changes

Fix an upgrade regression where OpenAI provider unification rewrites legacy Codex OAuth profiles into a shape rejected when launching Codex-backed agents.

## Best Possible Solution

Keep the canonical `openai` migration and the shipped Codex bridge behavior that accepts canonical OpenAI OAuth credentials; users affected on v2026.6.5 should upgrade to v2026.6.6 or newer.

## Reproduction Assessment

Yes, from source and release comparison: canonical `provider: "openai"` OAuth credentials trigger the reported failure only when evaluated by the incompatible validator, while the v2026.6.6 validator accepts that identity.

## Solution Assessment

Yes. Accepting canonical OpenAI OAuth in the Codex bridge matches provider unification and upstream Codex's token-based contract; reverting credentials to the retired provider or maintaining competing runtime shapes would be less maintainable.
