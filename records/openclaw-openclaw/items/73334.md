## Summary

Keep open. The refactor is useful, but the PR is not ready because the shared Feishu parser applies the comment-link 22-28 character token guard to Bitable URL parsing, narrowing inputs that current main accepts; the SQLite helper extraction looks behavior-preserving and the security pass is clear.

## What This Changes

This PR extracts Feishu document-link parsing into a shared helper and moves duplicated task SQLite number/JSON helpers into a shared task-store module with focused tests.

## Best Possible Solution

Keep the shared helpers, but make Bitable parsing preserve its current broad token acceptance or use a Bitable-specific parser mode while leaving the stricter comment-link token guard intact.

## Reproduction Assessment

Yes, by source inspection: current main's `parseBitableUrl()` matches any alphanumeric `/base/` or `/wiki/` token, while the PR sends `feishu_bitable_get_meta` through a 22-28 character token guard. A focused regression case is `https://example.test/base/app_token?table=tbl_main`, which currently parses and would be rejected by the PR.

## Solution Assessment

No. Extracting shared parsing is the right direction, but the proposed implementation should preserve the existing Bitable token contract and keep strict token-length validation scoped to comment-link resolution.
