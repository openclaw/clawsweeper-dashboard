## Summary

Keep open. Current `main` still has a fixed TUI footer, no footer template or footer-field config, and no shipped plugin/status-provider contribution path; the related open PRs cover pieces of the design but do not supersede this broader cost, balance, and rate-limit footer request.

## What This Changes

The issue requests configurable TUI footer fields for model, tokens, estimated session cost, provider balance, and rate-limit cooldown status.

## Best Possible Solution

Keep this as the canonical product/API tracker until maintainers choose a generic, typed, cached, secret-safe footer/status contribution model with matching config, docs, and TUI tests.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives high-confidence verification that current `main` lacks the requested configurable footer fields and only exposes adjacent usage data through `/status`, `/usage`, CLI, and menu surfaces.

## Solution Assessment

Unclear until maintainers choose the UX and extension API. The narrowest maintainable direction appears to be a generic footer/template or status-provider contract, not hard-coding cost, balance, and rate-limit logic directly into the TUI footer renderer.
