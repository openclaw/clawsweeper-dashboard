## Summary

Keep open: current main and the latest release still have a fixed TUI footer, while the implemented usage surfaces cover related chat/CLI usage reporting rather than configurable persistent TUI footer fields for cost, balance, and cooldown status. The related plugin-status-provider work is useful context, but it is narrower and still open, so this issue remains the broader product/API tracker.

## What This Changes

The issue requests configurable TUI footer fields for model, tokens, estimated cost, provider balance, and rate-limit or cooldown status.

## Best Possible Solution

Keep this as the broad TUI footer observability tracker, use existing /status and /usage surfaces meanwhile, and let maintainers decide whether the durable shape is a PluginStatusProvider seam, a TUI config surface, or a narrower provider-usage footer follow-up.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source and release inspection confirm that the requested configurable live TUI footer is absent on current main and v2026.6.10.

## Solution Assessment

No; the requested direct core footer template and live provider-balance poller are not clearly the best fix without maintainer direction. A plugin/API seam may be cleaner for optional status fields, but current related work does not yet cover TUI rendering or the cost/balance/cooldown field semantics.
