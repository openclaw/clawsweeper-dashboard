## Summary

Keep open. ## What This Changes

Request configurable TUI footer/status fields that can show model, token usage, estimated cost, provider balance, and rate-limit cooldown state.

## Best Possible Solution

Keep this as the broader product/API tracker until maintainers choose a generic, cached, secret-safe footer/status contribution model that reuses existing usage/quota data and coordinates with the narrower plugin status-provider work.

## Reproduction Assessment

## Solution Assessment

Unclear until maintainers choose the UX and extension contract. The narrowest maintainable direction appears to be a generic footer/status provider or template contract built on existing usage data, not hard-coding cost, balance, and rate-limit logic directly into `updateFooter()`.
