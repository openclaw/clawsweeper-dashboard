## Summary

Keep open. Current main and the latest release still have a fixed TUI footer plus adjacent `/status`, `/usage`, and provider-usage surfaces, but they do not provide configurable footer fields, provider balance/cooldown display, or a landed plugin status-provider render path.

## What This Changes

Request configurable TUI footer/status fields for model, token, estimated cost, provider balance, and rate-limit cooldown information.

## Best Possible Solution

Keep this as the broader product/API tracker until maintainers choose a generic, typed, cached, secret-safe footer/status contribution model that reuses existing usage/quota data and coordinates with the narrower plugin status-provider work.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, latest-release, and live related-item inspection give high-confidence verification that the requested configurable footer/status fields are absent from current main.

## Solution Assessment

Unclear until maintainers choose the UX and extension contract. The narrowest maintainable direction appears to be a generic footer/status provider or template contract built on existing usage data, not hard-coding cost, balance, and rate-limit logic directly into `updateFooter()`.
