## Summary

Keep open: current main and v2026.6.10 still have a fixed persistent TUI footer, while existing `/usage`, `/status`, and provider-usage work only partially cover cost/quota visibility. Related PluginStatusProvider and provider-quota footer work are useful but narrower, so this remains the broad product/API tracker.

## What This Changes

The issue requests configurable persistent TUI footer fields for model, token count, estimated cost, provider balance, and rate-limit cooldown status.

## Best Possible Solution

Keep this as the broad TUI footer observability tracker, use existing `/status` and `/usage` surfaces meanwhile, and have maintainers choose whether the durable shape is a PluginStatusProvider seam, a TUI config surface, or narrower provider-usage footer follow-ups.

## Reproduction Assessment

Not applicable as a bug repro; this is a feature request. Source inspection verifies the requested configurable persistent TUI footer is absent on current main and v2026.6.10, while related `/usage` and `/status` surfaces exist.

## Solution Assessment

No; the proposed direct core footer template and balance poller are not clearly the best solution without maintainer direction. A Plugin SDK status-provider seam or narrower reuse of provider-usage surfaces may be cleaner, but those paths do not yet cover the full requested TUI behavior.
