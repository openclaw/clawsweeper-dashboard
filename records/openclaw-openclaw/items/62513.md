## Summary

Keep open: current main and the latest release still call the SQLite-backed `hasIndexedContent()` probe from memory search, and the same-repo fix PR was closed unmerged. The open fork PR is useful context, but it is conflicting and not maintainer-modifiable, so maintainers still need to adopt, replace, or close that candidate.

## What This Changes

The issue requests caching the builtin memory manager's indexed-content check so repeated searches do not query SQLite just to decide whether content exists.

## Best Possible Solution

Add a manager-owned indexed-content cache with complete invalidation across all memory index mutation paths, plus focused tests for warm searches, writes, deletes, reset, and reindex/publish flows.

## Reproduction Assessment

Yes, source inspection is high-confidence: current main calls `hasIndexedContent()` from `search()`, and that helper directly queries SQLite. I did not establish a measured runtime benchmark for the performance impact.

## Solution Assessment

Yes, the requested in-memory cache is the narrow maintainable direction if invalidation is complete. The current open fork PR is only a candidate because it is conflicting and not maintainer-modifiable upstream.
