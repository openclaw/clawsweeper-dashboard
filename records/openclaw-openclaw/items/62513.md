## Summary

Keep open: current main and v2026.6.8 still run the SQLite-backed `hasIndexedContent()` probe on the builtin memory search path, and the only same-repo fix PR was closed unmerged. The open fork PR is relevant context, but it is conflicting and not maintainer-modifiable, so it is not proof that the issue is fixed.

## What This Changes

Request caching the memory-core `hasIndexedContent()` result so repeated memory searches avoid redundant SQLite existence probes.

## Best Possible Solution

The end state should be a manager-owned cached indexed-content state invalidated by all index mutation paths, with focused tests for warm searches, writes, deletes, reset, and safe-reindex DB swaps.

## Reproduction Assessment

Yes at source level: current main invokes `hasIndexedContent()` from `search()`, and that helper directly queries SQLite. I did not establish a measured runtime benchmark for the performance impact.

## Solution Assessment

Yes, a small manager-owned cache is the narrow maintainable fix if invalidation is complete. The current open fork PR is only a candidate because it is conflicting and not maintainer-modifiable upstream.
