## Summary

Keep open. Current main still has the reported hot path, and #62597 is an open, unmerged implementation candidate that explicitly closes this issue, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests caching `MemoryIndexManager.hasIndexedContent()` after indexing so repeated memory searches avoid redundant SQLite probes.

## Best Possible Solution

The desired end state is a memory-core-local indexed-content cache with exhaustive invalidation around chunk/FTS mutations and focused regression coverage that repeated search preflight no longer performs redundant SQLite probes.

## Reproduction Assessment

Yes. On current main, a nonblank `MemoryIndexManager.search()` call reaches `hasIndexedContent()`, and that method directly performs SQLite `SELECT 1` probes; a focused unit test can spy on `db.prepare` or count calls across repeated searches.

## Solution Assessment

Yes. Caching inside `memory-core` is the narrowest maintainable direction, provided the implementation updates or invalidates the cache in the indexing, deletion, stale cleanup, and reset paths rather than adding a core-level workaround.
