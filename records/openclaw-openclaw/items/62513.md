## Summary

Keep open. Current main still performs repeated SQLite preflight probes in memory-core search, and #62597 is an open, unmerged PR with closing syntax for this issue, so cleanup should wait for that PR to land or be replaced.

## What This Changes

Cache memory-core's indexed-content preflight so repeated memory searches do not query SQLite through `hasIndexedContent()` every time.

## Best Possible Solution

Land or replace #62597 with a memory-core-local indexed-content cache that is updated after successful chunk writes and invalidated on deletes, stale cleanup, resets, and database replacement paths.

## Reproduction Assessment

Yes. On current main, any nonblank `MemoryIndexManager.search()` reaches `hasIndexedContent()`, and that helper prepares SQLite `SELECT 1` probes; a focused regression test can spy on `db.prepare` across repeated searches.

## Solution Assessment

Yes. Caching inside memory-core with explicit invalidation around index mutations is the narrowest maintainable direction; the existing open PR is the right implementation lane if reviewed and refreshed.
