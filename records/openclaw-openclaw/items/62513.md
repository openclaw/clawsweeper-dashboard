## What This Changes

The issue requests caching `hasIndexedContent()` in memory search so repeated searches avoid redundant SQLite existence probes.

## Best Possible Solution

Land a narrow memory-core-local cache that preserves the existing bootstrap behavior, avoids repeated indexed-content probes after the result is known, and explicitly invalidates on chunk insert/delete/reset plus DB handle replacement paths.

Add a manager-local process cache for the indexed-content boolean while preserving the current first-search bootstrap sync behavior; set or invalidate it around all chunk writes, file/session stale-row deletes, `resetIndex()`, safe reindex DB swaps/reopens, and readonly-recovery DB reopen paths. Treat https://github.com/openclaw/openclaw/pull/62597 and https://github.com/BingqingLyu/openclaw/pull/2108 as useful source work, but create a current-main replacement rather than reviving the stale conflicting branch; credit the source work in the PR body if reused. Do not add config, schema changes, public API, docs, or CHANGELOG.md edits. Add focused regression coverage that repeated searches after indexed content avoid repeated `SELECT 1` probes and that deletes/resets/DB-handle replacement invalidate the cache.
