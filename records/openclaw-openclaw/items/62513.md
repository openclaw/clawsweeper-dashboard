## Summary

Keep open. Current main and the latest release still run the uncached SQLite existence probes from memory search, and the open implementation PR at https://github.com/openclaw/openclaw/pull/62597 still uses closing syntax for this issue, so this should wait for that PR to land or be replaced.

## What This Changes

The issue requests caching `MemoryIndexManager.hasIndexedContent()` so repeated memory searches do not hit SQLite for the same indexed-content existence check.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/62597 with a manager-local cache scoped to the active DB handle and invalidated on every chunk/table mutation and DB swap.

## Reproduction Assessment

Yes. Source inspection on current main shows each `search()` reaches `hasIndexedContent()`, which prepares direct SQLite existence probes; a focused regression can count prepare/query calls across repeated searches.

## Solution Assessment

Yes. A memory-core-local indexed-content cache is the narrow maintainable direction, provided it is invalidated across writes, deletes, resets, safe reindex, and readonly-recovery DB replacement.
