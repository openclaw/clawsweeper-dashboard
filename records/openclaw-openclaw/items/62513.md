## Summary

Keep open. Current main and v2026.5.18 still run uncached SQLite existence probes from memory search, and the open implementation PR at https://github.com/openclaw/openclaw/pull/62597 still uses closing syntax for this issue, so this should wait for that PR to land or be replaced.

## What This Changes

Request caching memory-core `hasIndexedContent()` so repeated `search()` calls avoid redundant SQLite existence probes.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/62597 with a manager-local cache scoped to the active DB handle, invalidated on every indexed-table mutation and DB swap, and covered by focused regression proof.

## Reproduction Assessment

Yes. Source inspection on current main shows each `search()` reaches `hasIndexedContent()`, which prepares direct SQLite existence probes; a focused regression can count prepare/get calls across repeated searches.

## Solution Assessment

Yes, with the linked PR refreshed or replaced. A memory-core-local indexed-content cache is the narrow maintainable direction if it is invalidated across writes, deletes, resets, safe reindex, and readonly-recovery DB replacement.
