## Summary

Keep open. Current main still runs the reported uncached SQLite existence probes from memory search, and an open PR still uses closing syntax for this issue, so the issue should wait for that PR to land or be replaced.

## What This Changes

Request caching `hasIndexedContent()` in memory-core so repeated memory searches do not prepare direct SQLite existence probes every time.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/62597 with a manager-local indexed-content cache invalidated on every indexed-table mutation and active DB handle replacement, with focused regression proof.

## Reproduction Assessment

Yes. Source inspection on current main shows repeated `search()` calls reach `hasIndexedContent()`, which prepares direct SQLite existence probes; a focused regression can count prepare/get calls across repeated searches.

## Solution Assessment

Yes, with the linked PR refreshed or replaced. A memory-core-local cache is the narrow maintainable approach if it is invalidated across writes, deletes, resets, safe reindex, and readonly-recovery DB replacement.
