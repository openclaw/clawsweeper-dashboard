## Summary

Keep open. Current main and the latest release still run SQLite existence probes from `search()` without an indexed-content cache, and the open implementation PR at https://github.com/openclaw/openclaw/pull/62597 uses closing syntax for this issue, so the issue should wait for that PR to land or be replaced.

## What This Changes

Request caching memory-core indexed-content presence so repeated searches avoid redundant SQLite probes.

## Best Possible Solution

Refresh, review, and land or replace https://github.com/openclaw/openclaw/pull/62597 with a memory-core-local cache scoped to the active DB handle and invalidated on every index mutation.

## Reproduction Assessment

Yes. Source inspection on current main shows `MemoryIndexManager.search()` calls `hasIndexedContent()` for each search and that method prepares SQLite existence probes; a focused regression test can count `db.prepare` calls across repeated searches.

## Solution Assessment

Yes. A manager-lifetime indexed-content cache with complete invalidation around writes, deletes, resets, reindex DB swaps, and readonly-recovery DB replacement is the narrow maintainable solution.
