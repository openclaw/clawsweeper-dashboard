## Summary

Keep this PR open for changes: the underlying scheduled live/E2E workflow permission gap is plausible, but this patch puts the new scope at the workflow level while the reusable-workflow caller already has its own job-level permissions map, so the called workflow would still not receive `actions: read`. The PR also still lacks contributor-provided real workflow proof.

## What This Changes

The PR adds `actions: read` to the top-level permissions block in `.github/workflows/openclaw-scheduled-live-checks.yml`.

## Best Possible Solution

Move `actions: read` into `jobs.live_and_openwebui_checks.permissions`, remove the trailing whitespace, and attach redacted workflow-run proof showing the scheduled live/E2E reusable workflow path succeeds.

## Reproduction Assessment

Yes, by source inspection: current main's scheduled caller job has a job-level permissions map without `actions: read`, while the called reusable workflow requests that scope for `prepare_docker_e2e_image`. I did not dispatch the live workflow.

## Solution Assessment

No. The narrower maintainable fix is to grant `actions: read` inside the caller job's permissions block, matching the release caller pattern, instead of changing only the top-level default.
