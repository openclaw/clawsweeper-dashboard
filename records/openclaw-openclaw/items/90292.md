## Label Justifications

- `impact:data-loss`: The reported failure discards generated dream diary prose and replaces the expected user-facing diary entry.

- `impact:security`: The fallback symptom can expose raw memory staging fragments such as conversation summaries or session identifiers in a human-readable diary file.

## Summary

Current main and the latest release already solve the central report: the supported plugin `getSessionMessages` path unwraps transcript envelopes before memory-core sees them, `extractNarrativeText` handles the resulting flat assistant messages, and fallback diary writes now use a generic placeholder instead of raw snippets.

## What This Changes

The issue reports that memory-core dreaming writes fallback snippets to DREAMS.md because narrative text extraction allegedly misses cron-wrapped subagent messages.

## Best Possible Solution

Keep the v2026.6.1-and-newer path where sessions.get normalizes transcript envelopes and memory-core fallback entries never persist raw staging snippets.

## Reproduction Assessment

No high-confidence current-main reproduction path remains. I did not run a live cron job, but current source and release-tag tests show `sessions.get` returns flat messages and fallback diary writes no longer include raw snippets.

## Solution Assessment

Yes. The best current solution is the existing boundary: normalize transcript envelopes in the gateway/session reader and keep memory-core fallback generic, instead of teaching the diary extractor to depend on raw transcript JSONL envelopes.
