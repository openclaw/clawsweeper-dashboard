## Label Justifications

- `P2`: Repeated live reports show degraded bundled memory output, but the background dreaming failure does not block core agent or channel operation.
- `impact:data-loss`: Valid generated narrative text is silently discarded and replaced by raw fallback snippets.
- `impact:session-state`: Completed subagent session output is not reliably readable at the host plugin’s extraction point.

## Summary

This remains necessary: a live run on release v2026.6.5 shows `agent.wait` returning before `model.completed`, while current main still reads `sessions.get` immediately after a successful wait and falls back if text is absent. The original envelope diagnosis is not supported by current source, and overlapping reports describe related but not identical lifecycle failures, so this issue provides distinct evidence for an unresolved plugin-subagent completion contract.

## What This Changes

Ensure memory-core dreaming preserves valid narrative subagent output instead of writing fallback snippets when the final assistant message is not yet readable.

## Best Possible Solution

Make successful plugin-subagent completion expose durably readable final assistant output before `agent.wait` resolves, coordinate cleanup after consumers can read it, and retain memory-core fallback only for genuinely failed or empty runs with focused lifecycle regression tests.

## Reproduction Assessment

Yes. The June 10 v2026.6.5 trace gives a high-confidence live reproduction with ordered `agent.wait`, fallback append, `model.completed`, and final-text timestamps, and current-main source contains the matching immediate wait-then-read path.

## Solution Assessment

No, the proposed envelope-unwrapping change is not the best solution because the gateway session reader already normalizes message envelopes. The narrow maintainable fix is a gateway-owned completion, persistence, and cleanup-ordering contract, with memory-core tests proving that successful output is readable before fallback selection.
