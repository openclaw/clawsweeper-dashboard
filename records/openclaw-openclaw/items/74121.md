## Summary

Keep open. The latest diff still has one blocking behavior issue: it marks every embedded fallback chain as tool-required even though current main supports raw/no-tool model runs, while the auth-profile format cooldown slice is already on main and the PR body still describes blank-message changes that the second commit removed.

## What This Changes

The PR adds a tool-capability filter for fallback candidates, passes `requiresTools` from the embedded agent command path, exempts `format` failover reasons from auth-profile cooldown, and adds focused tests.

## Best Possible Solution

Rebase and narrow the PR to the remaining fallback policy: skip explicitly tool-incompatible fallbacks only when the actual attempt will construct tools, while preserving raw/no-tool fallback behavior and leaving already-fixed auth/OpenRouter metadata work on main.

## Reproduction Assessment

Yes for the blocking regression by source inspection: current main has raw `modelRun` and `promptMode: "none"` paths that disable tool construction, while the PR would mark the whole embedded fallback chain as tool-required.

## Solution Assessment

No. Reusing `supportsModelTools()` is the right contract, but the PR should derive `requiresTools` from the actual run mode or tool-construction plan instead of setting it unconditionally.
