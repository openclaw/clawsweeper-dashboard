The PR adds tool-incompatible fallback filtering, passes an unconditional `requiresTools: true` from the embedded agent path, and adds a `format` auth-profile cooldown exemption with tests.

## Best Possible Solution

Revise the PR so fallback filtering is gated on the actual tool-construction mode, rebase away already-shipped auth/blank-message pieces, and add redacted real behavior proof for the remaining runtime change.

## Reproduction Assessment

Yes, by source inspection: current main has raw `modelRun` and `promptMode: "none"` paths that disable tool construction, while the PR would mark the whole embedded fallback chain as requiring tools.

## Solution Assessment

No. Reusing `supportsModelTools()` is the right contract, but the PR should derive tool-required fallback filtering from the actual run mode or tool-construction plan instead of setting it unconditionally.
