## Summary

Keep open: the PR still contains useful fallback-policy work, but the current patch is not merge-ready because it marks all embedded fallbacks as tool-required, is stale against current-main auth cooldown behavior, and lacks required real behavior proof.

## What This Changes

The branch adds a `requiresTools` fallback filter, passes `requiresTools: true` from the embedded agent command path, and exempts `format` failover reasons from auth-profile cooldown with tests.

## Best Possible Solution

Rebase and narrow the PR so fallback filtering is gated by the actual tool-construction plan, current-main auth exemptions are preserved, stale blank-message scope text is removed, and real runtime proof is added.

## Reproduction Assessment

Yes by source inspection. Current main has raw `modelRun` and `promptMode: "none"` paths that intentionally disable tool construction, while the PR marks the whole embedded fallback chain as tool-required; current main also exempts `server_error` cooldowns that the PR head lacks.

## Solution Assessment

No. The `supportsModelTools()` contract is the right building block, but the PR should derive tool requirement from the actual run mode/tool-construction plan and preserve current-main auth policy during rebase.
