## Summary

Keep open. Current main still records `format` failover reasons as auth-profile failures and still resolves configured model fallbacks without a tool-capability filter; this PR addresses both gaps narrowly. The latest PR head no longer includes the blank-message replay changes described in the original body, so the earlier compact-path warning concern is obsolete, but the remaining behavior change still needs docs and changelog cleanup before merge.

## What This Changes

The PR adds a `requiresTools` fallback-resolution option, passes it from the embedded agent command path, skips configured fallback models whose config definition explicitly has `compat.supportsTools === false`, and exempts `format` failover reasons from auth-profile cooldown with focused tests.

## Best Possible Solution

Land the focused fallback/cooldown behavior after adding operator-facing docs and a release-note entry, while leaving current main's existing blank-user replay defenses in place and not reintroducing the reverted blank-message helper.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `resolveAuthProfileFailureReason({ failoverReason: "format" })` returns `format`, and configured fallback refs are added without consulting `supportsModelTools`. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, with small documentation and changelog follow-up. The PR uses the existing `supportsModelTools` helper and keeps unknown tool support permissive, which is a narrow maintainable fix for configured fallback entries; the broader primary-model compatibility behavior remains separately tracked by #64175.
