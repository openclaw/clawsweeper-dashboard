## Summary

Keep open: the latest head fixed the earlier stale-session batching problem, but it still has two merge-blocking correctness gaps around pricing-cache invalidation and running rebuild work on the Gateway event loop; the user-facing fix also needs changelog coverage.

## What This Changes

The PR replaces request-path usage transcript scans with a durable per-agent usage-cost cache, cache-status RPC/UI reporting, append-only refresh logic, and focused backend/UI tests.

## Best Possible Solution

Land the durable cache after cache freshness tracks every pricing source, rebuild work cannot starve Gateway handling, and the user-facing performance fix is documented in the changelog.

## Reproduction Assessment

Yes: source inspection on current main shows both `usage.cost` and `sessions.usage` reaching transcript-scanning helpers on request paths. I did not run the live 90-244s timing reproduction because this review is read-only.

## Solution Assessment

No: the durable cache direction is maintainable, but the current PR head is not the best mergeable fix until pricing invalidation includes all sources, refresh work is isolated from the Gateway event loop, and changelog coverage is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Include every pricing source in the cache fingerprint:** `src/infra/session-cost-usage.ts:170-198`
  - body: The fingerprint only serializes `config.models.providers`, but `resolveModelCostConfig` also uses packaged `models.json` and the gateway model-pricing cache. If remote pricing populates after the first rebuild, or packaged pricing changes without transcript size/mtime changing, entries remain fresh and keep serving stale or zero costs. Include those pricing sources or their metadata in invalidation.
  - confidence: 0.91
- **[P2] Move refresh scans off the Gateway event loop:** `src/infra/session-cost-usage.ts:1245-1249`
  - body: `requestCostUsageCacheRefresh` queues `refreshCostUsageCache` with `setTimeout`, but the JSONL parsing, cost resolution, and session-summary recomputation still run in the Gateway Node process. Large rebuilds can still starve RPC/channel handling, which is one of the failures this PR is meant to fix. Use an actual worker/process/thread or a scanner that yields in bounded chunks.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/usage.ts:350-354`
  - body: This is a user-visible gateway and Control UI performance fix, but the PR head does not update `CHANGELOG.md`. Add a single-line entry under the active Unreleased Fixes section before merge.
