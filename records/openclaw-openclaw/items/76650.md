## Summary

Keep open: current main still has the slow usage scan path and this PR is relevant, but the latest head reintroduces request blocking for an empty durable cache by rebuilding all stale transcripts synchronously from `usage.cost`.

## What This Changes

The PR adds a durable per-agent usage/cost aggregate cache, cache status API/UI plumbing, append-only refresh support, pricing invalidation, queued refreshes, and focused regression coverage.

## Best Possible Solution

Keep the durable aggregate cache, but make cold and stale cache reads return immediately with cache status while a bounded/off-request refresh fills the cache, with regression coverage proving the gateway handler does not wait for a full rebuild.

## Reproduction Assessment

Yes. Source inspection of current main shows `usage.cost` awaits transcript enumeration and JSONL scanning on cache miss, so a profile with many or large usage-counted transcripts is a high-confidence reproduction path for the reported stalls.

## Solution Assessment

No. The durable cache is the right direction, but the current PR head synchronously rebuilds an empty cache from the gateway request path, which preserves the cold-cache version of the reported stall.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid blocking usage.cost on a cold cache:** `src/infra/session-cost-usage.ts:1093-1094`
  - body: The gateway calls this helper with `refreshMode: "sync-when-empty"`, so when the durable cache is missing or invalidated and `cachedFiles === 0`, the RPC awaits `refreshCostUsageCache()`. That refresh enumerates and scans every stale transcript before responding, which preserves the reported 90-244s dashboard/gateway stall on first refresh. Return a stale/refreshing cache status and queue the rebuild, or cap synchronous work to a small bounded slice.
  - confidence: 0.86
