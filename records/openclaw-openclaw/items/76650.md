## Summary

Keep open: the PR addresses a real usage aggregation stall, but the current head still makes the gateway await an uncapped cold-cache rebuild when no durable cache entries exist, preserving the cold-start version of the reported dashboard/gateway stall.

## What This Changes

The PR adds a durable per-agent usage/cost aggregate cache with cache status API/UI plumbing, append-only refresh support, pricing invalidation, and focused regression tests.

## Best Possible Solution

Keep the durable aggregate cache direction, but make gateway usage reads return immediately with stale/partial/refreshing cache status while a bounded off-request refresh fills the cache, then add the required changelog entry.

## Reproduction Assessment

Yes. Source inspection of current main shows `usage.cost` awaits transcript enumeration and JSONL scanning on cache miss, so a profile with many or large usage-counted transcripts is a high-confidence reproduction path for the reported stalls.

## Solution Assessment

No. The durable cache is the right direction, but the current PR head still performs an uncapped synchronous rebuild when the cache is empty; the safer fix is a nonblocking or tightly bounded cold-cache path.
