## Summary

Keep open. The PR addresses a real current-main usage performance problem, but the latest head still has merge-blocking correctness issues in worker startup, date-range handling, pricing freshness, plus a security-sensitive full-config worker handoff.

## What This Changes

The PR adds a durable per-agent usage-cost transcript aggregate cache, a detached rebuild worker, cache-status fields on gateway usage responses, and Control UI cache-state messaging.

## Best Possible Solution

Keep the durable-cache direction, but fix worker entry resolution, preserve per-range session summaries, add pricing/config-aware invalidation, and pass only minimal secret-free worker inputs before merge.

## Reproduction Assessment

Yes. Current main's `usage.cost` and `sessions.usage` paths are source-reproducible as request-path transcript scans; I did not live-reproduce the reported 90-244s timing in this read-only review.

## Solution Assessment

No. A durable aggregate cache is a maintainable direction, but the current PR head is not the best mergeable solution until the worker startup, range correctness, pricing freshness, and config-handoff issues are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Fix the worker URL so refreshes can start:** `src/infra/session-cost-usage.ts:1077-1082`
  - body: `requestCostUsageCacheRefresh` runs from `src/infra/session-cost-usage.ts`, so `./infra/session-cost-usage-cache-worker.js` resolves to an extra nested `infra/` directory in source and dist. The detached child starts with a missing file, leaving stale caches unable to rebuild through the request path this PR depends on.
  - confidence: 0.95
- **[P2] Preserve the selected date range when using session cache:** `src/gateway/server-methods/usage.ts:669-683`
  - body: `sessions.usage` used to pass `startMs` and `endMs` into `loadSessionCostSummary`, but the cached path now loads a full-session summary and only checks session overlap. A long-running session with any activity in range will contribute lifetime tokens and cost to that selected range.
  - confidence: 0.93
- **[P2] Invalidate cached costs when pricing inputs change:** `src/infra/session-cost-usage.ts:297-309`
  - body: Cache freshness only compares transcript size/mtime and session-summary presence, but cached entries are produced through config-dependent cost resolution and tiered-pricing recomputation. Changing model pricing without touching transcripts will keep serving old costs indefinitely.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Needs attention: the PR adds a detached worker and serializes full runtime config into the child environment.

Concerns:

- **[medium] Avoid serializing full config into the worker environment:** `src/infra/session-cost-usage.ts:1089`
  - body: `OPENCLAW_USAGE_COST_CACHE_CONFIG` receives `JSON.stringify(params?.config ?? {})` for a detached child process. OpenClaw config can include sensitive API keys, tokens, and headers, so the worker should receive only the minimal secret-free pricing data it needs or use an existing safer config-loading path.
  - confidence: 0.76
