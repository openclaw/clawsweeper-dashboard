## Summary

Keep this PR open: the durable cache direction is useful and the latest head fixes part of the cold aggregate path, but `sessions.usage` can still synchronously refresh across every stale transcript and the required changelog entry is still missing.

## What This Changes

The PR adds a durable per-agent usage/cost transcript aggregate cache, cache status API/UI plumbing, append-only refresh support, pricing invalidation, and focused regression tests.

## Best Possible Solution

Keep the durable cache approach, but make gateway usage reads return promptly by bounding synchronous work to the requested session/range and queueing full cache rebuilds off-request, then record the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection of current main shows both `usage.cost` and `sessions.usage` await JSONL transcript scans in gateway request handlers, so a profile with many or large usage-counted transcripts is a high-confidence reproduction path; I did not run a live large-profile test.

## Solution Assessment

No. The cache design is the right narrow direction, but the current PR head still has an unbounded synchronous session-summary refresh path; restricting that path and queueing broader rebuilds is the safer fix.

## Review Findings

- **[P2] Bound synchronous session summary refreshes:** `src/infra/session-cost-usage.ts:1043-1047`
