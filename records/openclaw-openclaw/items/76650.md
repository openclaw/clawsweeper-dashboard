## Summary

Keep open. Current main still has the scan-on-request usage path, and this PR is aimed at a real performance problem, but the proposed patch is not mergeable because the detached worker cannot resolve its own entry file and the cached session summaries break existing date-range accounting.

## What This Changes

The PR adds a durable per-agent usage-cost cache, a detached rebuild worker, cache-status fields on usage RPC responses, and Control UI indicators for stale or rebuilding usage data.

## Best Possible Solution

Keep the durable-cache direction, but make the worker entry resolution reliable, preserve date-range semantics for session summaries, invalidate cached cost data when pricing inputs change, and avoid passing secret-bearing config through process environment.

## Reproduction Assessment

Yes for the underlying issue from source inspection: current main serves `usage.cost` and `sessions.usage` through transcript scans on the gateway request path. I did not reproduce the 90-244s live profile timing in this read-only review.

## Solution Assessment

No. A durable aggregate cache is a reasonable direction, but this patch currently breaks worker startup, date-range correctness, and pricing/config freshness, so it is not the best mergeable implementation yet.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Fix the worker URL so refreshes can start:** `src/infra/session-cost-usage.ts:965-970`
  - body: `requestCostUsageCacheRefresh` lives in `src/infra/session-cost-usage.ts`, so `new URL("./infra/session-cost-usage-cache-worker.js", import.meta.url)` resolves to an extra nested `infra/` directory in both source and dist. The cache rebuild child therefore starts Node with a missing file and stale caches never get rebuilt on the request path this PR depends on.
  - confidence: 0.95
- **[P2] Preserve the selected date range when using session cache:** `src/gateway/server-methods/usage.ts:669-683`
  - body: `sessions.usage` used to pass `startMs`/`endMs` into `loadSessionCostSummary`, but the cached path now loads a full-session summary and only checks whether the session overlaps the range. A long-running session with any activity in the selected range will contribute all lifetime tokens/cost to that range's totals and aggregates.
  - confidence: 0.93
- **[P2] Invalidate cached costs when pricing inputs change:** `src/infra/session-cost-usage.ts:285-289`
  - body: Cache freshness only compares transcript size/mtime and whether a session summary exists, but the cached entries are produced through config-dependent cost resolution and tiered-pricing recomputation. Changing model pricing or refreshed gateway pricing without touching transcripts leaves `usage.cost` and `sessions.usage` serving old costs indefinitely.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds an internal worker process, and the current config handoff can expose secret-bearing runtime config through the child environment.

Concerns:

- **[medium] Avoid serializing full config into the worker environment:** `src/infra/session-cost-usage.ts:977`
  - body: `OPENCLAW_USAGE_COST_CACHE_CONFIG` receives `JSON.stringify(params?.config ?? {})` for a detached child. OpenClaw configs can contain credential surfaces or sensitive provider headers, so the worker should receive only the minimal secret-free pricing/config data it needs or use an existing safer runtime config mechanism.
  - confidence: 0.76
