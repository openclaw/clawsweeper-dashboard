## Summary

Keep open: the durable-cache direction addresses the source-reproducible request-path transcript scans, but the latest PR head still has merge-blocking cache correctness gaps around cold `sessions.usage` refreshes and pricing invalidation.

## What This Changes

The PR adds a durable per-agent usage-cost aggregate cache, cached gateway usage handlers, cache-status fields for the Usage UI, and related regression tests.

## Best Possible Solution

Keep the durable cache, but coalesce all requested session-summary files per agent, make cache freshness cover every pricing source used by cost resolution, add the required changelog entry, and validate the targeted usage/gateway/UI lanes.

## Reproduction Assessment

Yes, at source level. Current main's `usage.cost` and `sessions.usage` handlers call transcript-scanning helpers directly on the gateway request path, though I did not live-reproduce the reported 90-244s timings in this read-only pass.

## Solution Assessment

No. The durable aggregate cache is a maintainable direction, but the current PR head is not the best mergeable solution until session-summary refresh coalescing and full pricing-source invalidation are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Coalesce all session summary refresh requests:** `src/infra/session-cost-usage.ts:1168-1170`
  - body: `sessions.usage` calls `loadSessionCostSummaryFromCache` once per displayed session. On a cold or stale cache, the first call schedules a refresh with one `sessionFiles` entry, then later calls for the same agent hit this early return and drop their file paths. The refresh only materializes one session summary, so most rows can keep returning `usage: null` until repeated dashboard refreshes fill them one by one.
  - confidence: 0.9
- **[P2] Include all pricing sources in cache freshness:** `src/infra/session-cost-usage.ts:157-185`
  - body: The durable cache fingerprint only serializes `params.config.models.providers`, but cost resolution still reads `models.json` first and then falls back to cached gateway model pricing. If either of those pricing sources changes without transcript mtime/size changes, cached entries remain fresh and `usage.cost` keeps serving old costs.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `ui/src/ui/views/usage.ts:712-718`
  - body: This is a user-facing `fix(usage)` change that adds gateway response fields and Usage UI messaging, but the PR files do not include `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing fixes.
  - confidence: 0.82
