## Summary

Keep open. Current main at aa748 still routes Bedrock `aws-sdk` media-understanding auth through unconditional API-key requirements in the image, MiniMax VLM fallback, and audio/video provider execution paths, while closing PR #72092 remains open and unmerged.

## What This Changes

Report requests that Bedrock media-understanding image and audio/video paths accept `aws-sdk` auth without a static API key, matching chat/completion behavior.

## Best Possible Solution

Land or supersede #72092 with a narrow media-understanding fix: allow empty API-key material only for `aws-sdk` auth in the image and audio/video media auth resolvers, avoid storing empty runtime keys, preserve missing-key failures for other modes, and keep regression coverage for the Bedrock image path plus provider execution auth.

## Reproduction Assessment

Yes. Configure a Bedrock image model with `auth.mode === "aws-sdk"` and no `apiKey`, then call the image tool; current main reaches `requireApiKey` in media auth resolution and throws before the AWS SDK credential chain can run.

## Solution Assessment

Yes. Mirroring the chat path's missing-key allowance in the media-specific auth resolvers is the narrow maintainable fix; weakening `requireApiKey` globally would broaden behavior beyond the reported Bedrock `aws-sdk` case.
