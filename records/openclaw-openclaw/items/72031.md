## Summary

Keep open. Current main still routes Bedrock `aws-sdk` media auth through unconditional API-key requirements, and the linked closing PR #72092 is open rather than merged, so the issue should remain open until that PR lands or is superseded.

## What This Changes

The issue requests that Bedrock media-understanding image and audio/video paths allow `auth.mode === "aws-sdk"` without a static API key, matching the chat/completion auth path.

## Best Possible Solution

Land or supersede #72092 with a narrow media-understanding fix: allow empty API-key material only for `aws-sdk` auth in the image and audio/video media auth resolvers, avoid storing empty runtime keys, preserve missing-key failures for other modes, and keep regression coverage for the Bedrock image path plus provider execution auth.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a Bedrock image model so auth resolves to `mode: "aws-sdk"` with no `apiKey`, then call the image tool; current main reaches `requireApiKey` and throws before the AWS SDK credential chain can run.

## Solution Assessment

Yes. Mirroring the chat path's missing-key allowance in the media-specific auth resolvers is the narrow maintainable fix; weakening `requireApiKey` globally would broaden behavior beyond the reported Bedrock `aws-sdk` case.
