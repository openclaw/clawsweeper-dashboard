## Label Justifications

- `P1`: The issue blocks real Bedrock image/PDF/media workflows for AWS SDK credential users while the same provider can work for chat.
- `impact:auth-provider`: The failure is in provider auth handling: `aws-sdk` mode is routed through API-key-only guards before credential-chain auth can run.

## Summary

Current main and the latest release still contain the reported Bedrock `aws-sdk` media-auth failure, and the linked fix PR is open but unmerged/conflicting, so this issue remains necessary as the canonical tracker.

## What This Changes

The issue requests that media-understanding image/PDF/audio/video paths allow Bedrock `aws-sdk` auth without a static API key, matching the chat path.

## Best Possible Solution

Land or replace the linked fix with a current-main patch that lets Bedrock `aws-sdk` media-understanding calls proceed without static API keys while preserving non-`aws-sdk` missing-key errors and retry behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure Bedrock with `auth.mode: "aws-sdk"` and no literal `apiKey`, then invoke image or media-understanding so `requireApiKey` throws before SDK credential resolution.

## Solution Assessment

Yes, the linked fix direction is the right owner-boundary shape: extend the existing chat-path `aws-sdk` missing-key allowance to media-understanding without adding a new config option or credential workaround. The active PR still needs rebase/review before it can close the issue.
