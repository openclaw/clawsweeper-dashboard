- `P1`: This is a real provider-auth regression that blocks documented Bedrock `aws-sdk` users from media understanding workflows.
- `impact:auth-provider`: The failure is caused by media provider auth requiring a static API key when the configured mode is `aws-sdk`.

## Summary

Keep open: current main still rejects keyless Bedrock `aws-sdk` auth in the media-understanding image, fallback, and audio/video paths, and the focused open fix PR has not landed.

## What This Changes

The issue requests that media-understanding image and related audio/video paths allow Bedrock `aws-sdk` auth without requiring a literal API key, matching the chat path.

## Best Possible Solution

Review, refresh, and land https://github.com/openclaw/openclaw/pull/72092 or an equivalent narrow fix that permits empty `aws-sdk` media auth while preserving strict missing-key errors for other auth modes.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: a Bedrock media model with `auth: "aws-sdk"` and no `apiKey` reaches strict `requireApiKey` before the SDK credential chain can run.

## Solution Assessment

Yes. Mirroring the existing chat missing-key allowance across image, fallback, and audio/video media execution is the narrow maintainable fix; the linked PR or an equivalent refreshed patch is the right path.
