## Summary

Keep this issue open: current main and the latest release still route Bedrock media auth through strict API-key guards, and the linked closing PR https://github.com/openclaw/openclaw/pull/72092 remains open rather than merged.

## What This Changes

The issue requests keyless `aws-sdk` auth handling for Bedrock media-understanding tools when AWS SDK credentials are available but no static API key exists.

## Best Possible Solution

Review and land, replace, or extend https://github.com/openclaw/openclaw/pull/72092 so Bedrock media paths use keyless `aws-sdk` auth while non-`aws-sdk` modes keep strict missing-key errors.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure Bedrock with `auth: "aws-sdk"` and no literal `apiKey`, then invoke an affected media path so `requireApiKey` throws before SDK credentials can resolve.

## Solution Assessment

Yes for the core bug boundary. The maintainable solution is to reuse the existing keyless `aws-sdk` behavior already used by chat/runner paths, with maintainers deciding whether the later PDF reports belong in the same PR or a follow-up.
