## Summary

Keep open. Current main and the latest release still have the reported media-understanding auth gap, while an open PR explicitly closes this issue and remains the canonical implementation path.

## What This Changes

The issue reports that Bedrock image/media-understanding requests using `auth: "aws-sdk"` fail because media paths require a literal API key even though chat supports keyless AWS SDK credentials.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/72092, or an equivalent narrow fix, so resolved `aws-sdk` auth can proceed without a literal key only in the affected media paths while strict errors remain for other auth modes.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure Bedrock media auth as `aws-sdk` with no literal `apiKey`, then the image and media runner paths call strict `requireApiKey` before provider execution. I did not run a live Bedrock media request in this read-only sweep.

## Solution Assessment

Yes. The maintainable fix is to align media-understanding with the existing chat `aws-sdk` missing-key allowance and keep `requireApiKey` strict globally, with regression coverage for image and runner media paths.
