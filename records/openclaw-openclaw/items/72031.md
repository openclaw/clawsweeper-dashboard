## Summary

Keep open: current main still rejects keyless Bedrock `aws-sdk` auth in the media-understanding image/fallback and audio/video paths, and a focused open PR explicitly closes this issue.

## What This Changes

The issue asks the media-understanding image path to allow Amazon Bedrock `aws-sdk` auth without a literal API key, matching existing chat completion behavior.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/72092, or an equivalent narrow fix, so only `aws-sdk` media auth can proceed without a literal key while other auth modes keep strict missing-key errors.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure an image-capable `amazon-bedrock` model with `auth: "aws-sdk"` and no `apiKey`; current main reaches strict `requireApiKey` in the image/fallback and media runner paths before provider execution.

## Solution Assessment

Yes. The maintainable fix is to align media-understanding with the existing text-runtime `aws-sdk` missing-key allowance, add focused regression coverage, and preserve strict errors for non-`aws-sdk` auth modes.
