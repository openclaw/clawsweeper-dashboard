## Summary

Current main still matches the reported LINE multi-account startup bug, so this should stay open. The default LINE account is listed from `channels.line.accounts.default` but the resolver skips that account entry when reading credentials, named LINE accounts without explicit `enabled: true` resolve disabled, and gateway startup returns before the LINE monitor can register the webhook route; the linked fix PR #47528 is closed and unmerged.

## What This Changes

The issue reports that LINE multi-account configuration using `channels.line.accounts` prevents default and named webhook routes from registering, returning 404.

## Best Possible Solution

Fix the LINE plugin's account resolution in the extension: resolve `accounts.default` as the default account source, treat configured named accounts as enabled unless explicitly disabled while still honoring top-level `channels.line.enabled: false`, and add focused regression coverage that proves both the default and custom webhook-path accounts reach startup/route registration.

## Reproduction Assessment

Yes. The supplied config maps directly to current source: `accounts.default` is ignored for credential resolution, `twgreen` resolves disabled because it lacks `enabled: true`, and gateway startup exits before `startAccount` can register LINE routes.

## Solution Assessment

Yes, the narrowest maintainable fix is in the LINE plugin account resolver and its tests, following the closed #47528 direction. A core HTTP-route or registry change is not the best first move because route registration is already downstream of the failing LINE account-resolution checks.
