## Label Justifications

- `P2`: This is a normal-priority LINE channel regression with limited blast radius to multi-account webhook configurations.
- `impact:message-loss`: The reported 404 webhook routes prevent inbound LINE messages from reaching OpenClaw for the affected accounts.
- `impact:auth-provider`: The root cause is LINE account credential resolution: configured tokens and secrets under `accounts.default` are ignored on current main.

## Summary

Keep open: current main and the latest release still have the LINE resolver behavior that skips `channels.line.accounts.default` and treats configured named accounts without explicit `enabled: true` as disabled. The open linked fix PR https://github.com/openclaw/openclaw/pull/81471 targets this exact account-resolution bug with focused tests and real LINE webhook proof, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that LINE multi-account configs under `channels.line.accounts` make both default and named webhook routes return 404 because the routes are not registered.

## Best Possible Solution

Land or replace the open linked PR with a narrow LINE resolver fix that reads `accounts.default`, default-enables configured named accounts, preserves explicit `enabled: false`, and keeps regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main reproduction path: `accounts.default` is skipped for default credentials, named accounts without explicit `enabled` resolve disabled, and gateway startup does not register webhooks for disabled or unconfigured accounts.

## Solution Assessment

Yes. The best path is a LINE-owned resolver fix like the open linked PR, not a new core API or a separate repair branch while that PR remains viable.
