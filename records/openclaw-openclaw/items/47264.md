## Label Justifications

- `P2`: This is a normal-priority LINE channel bug with limited blast radius to multi-account LINE webhook setups.
- `impact:message-loss`: Skipped webhook registration means inbound LINE messages never reach the channel handler.
- `impact:auth-provider`: The failure is rooted in LINE account credential and enablement resolution for default and named accounts.

## Summary

Keep open: current main and the latest release still have the LINE resolver behavior that ignores `channels.line.accounts.default` and defaults configured named LINE accounts to disabled. The open, mergeable closing PR https://github.com/openclaw/openclaw/pull/81471 directly targets this bug with focused tests and real LINE webhook proof, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue reports that LINE multi-account configs using `channels.line.accounts` fail to register webhook routes, causing both default and named LINE webhook POSTs to return 404.

## Best Possible Solution

Land the focused LINE resolver fix from the open closing PR, preserving explicit `enabled: false` as the opt-out and keeping the regression tests for `accounts.default` and named-account startup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `accounts.default` is skipped, named accounts without explicit `enabled` resolve disabled, and gateway startup returns before the LINE monitor registers webhook routes.

## Solution Assessment

Yes. The best fix is the narrow LINE-owned resolver change already proposed in the linked PR, because it matches the sibling Telegram account semantics without adding core API or config surface.
