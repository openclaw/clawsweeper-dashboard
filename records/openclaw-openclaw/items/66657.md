Keep open. Current `main` and the latest release still have source-level Feishu missing-content drop/crash paths, and https://github.com/openclaw/openclaw/pull/77109 is the open implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that Feishu group or direct messages with missing or empty content can crash dispatch when the channel calls string methods such as `.trim()` on undefined content.

## Best Possible Solution

Review, refresh, and land https://github.com/openclaw/openclaw/pull/77109 or an equivalent narrow replacement that accepts or normalizes missing content at the Feishu receive boundary, normalizes parser content before mention handling, guards downstream string calls, and adds focused regression proof.

## Reproduction Assessment

Yes, source-reproducible: current `main` rejects absent `message.content` at the Feishu receive parser and still has parser, sequential-key, and debounce paths where unnormalized content can reach string methods. I did not run a live Feishu tenant replay in this read-only review.

## Solution Assessment

No, current `main` is not the best way to solve the issue because it still contains the missing-content paths. The best fix is a narrow Feishu receive/parser normalization with downstream guards and regression proof, delivered through the linked PR or an equivalent replacement.
