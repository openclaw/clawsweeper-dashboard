## Summary

Keep open. Current main still has Feishu empty-content paths that assume parsed content is a string, and the normal receive parser rejects missing or empty content instead of safely normalizing it through the Feishu message pipeline. The related OpenClaw PRs were closed unmerged, while the only still-open implementation reference is in an external fork.

## What This Changes

Handle Feishu/Lark group messages with empty or undefined content, especially mention-only group messages, without throwing `TypeError: Cannot read properties of undefined (reading 'trim')`.

## Best Possible Solution

Normalize Feishu inbound message content to a string at the parser or receive-handler boundary, keep the fix inside the Feishu plugin, preserve group mention gating and `/stop`/`/btw` lane behavior, avoid creating blank LLM turns, and add regression coverage for missing-content and mention-only events.

## Reproduction Assessment

Yes. A focused source-level reproduction is to construct a Feishu message event whose runtime `message.content` is absent or undefined and pass it through `getFeishuSequentialKey` or the injected debounce resolver path, which still reaches unsafe string methods; normal current ingress may instead reject the payload as malformed before dispatch.

## Solution Assessment

Yes, a narrow Feishu plugin repair is the best path. Boundary normalization plus regression tests is safer than only adding optional chaining at the two reported call sites, and no core or cross-channel API change appears necessary.
