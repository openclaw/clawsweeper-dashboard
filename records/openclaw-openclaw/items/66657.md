## Summary

Keep open. Current main still has source-level Feishu paths that can propagate runtime-missing message content into `.trim()` or `.replace()`, and the linked open implementation PR explicitly owns the remaining fix work.

## What This Changes

The issue reports a Feishu dispatch crash when empty or mention-only group messages leave message content undefined and downstream code calls string methods on it.

## Best Possible Solution

Normalize missing Feishu message content to a string at the parser boundary, keep downstream string-method paths safe, add focused mention-metadata regression coverage, and close this only after the linked PR or a narrow replacement merges.

## Reproduction Assessment

Yes. Source inspection shows runtime-missing `message.content` can reach `parseFeishuMessageEvent()`, propagate through `parseMessageContent()`, and hit `.replace()` or `.trim()` before the empty-message skip; I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment

Yes. The narrow best fix is parser-boundary normalization plus downstream guards and regression coverage; the linked open PR is the current implementation candidate, but it needs refresh/review before this issue can close.
