## Summary

Keep open. Current main and the latest release still have source-level Feishu missing-content paths that can crash or drop inbound messages, and https://github.com/openclaw/openclaw/pull/77109 is the open implementation candidate but is not merged.

## What This Changes

The issue reports Feishu mention-only or empty-content group messages leaving `message.content` undefined and breaking dispatch when parser-derived text is trimmed or normalized without a string guard.

## Best Possible Solution

Land or replace the linked Feishu fix with parser-boundary normalization, receive-payload handling for missing content, downstream string guards, and focused missing-content regression coverage.

## Reproduction Assessment

Yes. Source inspection shows runtime-missing `message.content` can remain unnormalized, propagate through Feishu parser/mention handling, and reach `.trim()` or be rejected before dispatch; I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment

Yes in direction, not complete on main. The maintainable fix is to normalize missing content at the parser/receive boundary and keep downstream string-method paths safe; the linked open PR is the current candidate but needs refresh/review before this issue can close.
