## Summary

Keep open. Current main still has unsafe Feishu missing-content paths, and an open implementation PR explicitly closes this issue, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks OpenClaw to avoid crashing Feishu group-message handling when a mention-only or otherwise empty event has missing `message.content`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77109 with a narrow Feishu fix that normalizes missing inbound content at the parser or receive boundary and adds regression coverage for missing-content mention events.

## Reproduction Assessment

Yes, at source level. Construct a Feishu text event with missing `message.content` and mention metadata, then exercise `parseFeishuMessageEvent()` through `getFeishuSequentialKey()` or the receive/debounce path on current main.

## Solution Assessment

No, the issue's two-site optional-chaining suggestion is too narrow by itself. The maintainable solution should enforce a safe string-content boundary for Feishu parsing/receive handling and keep the empty-message skip behavior covered by tests.
