## Summary

Keep open. Current main still has unsafe Feishu empty-content handling and would either trim an undefined parsed content value in internal receive helpers or reject a missing-content mention event before the existing empty-message path can handle it; the target-repo fix PRs for this issue were closed unmerged, and the remaining implementation reference is external.

## What This Changes

Handle Feishu group messages that mention the bot without a text body so missing or empty `message.content` does not crash or prevent safe handling.

## Best Possible Solution

Normalize Feishu inbound message content to a string at the parser or receive-handler boundary, preserve existing group mention gating, debounce, sequential-lane, media, and empty-message behavior, and add regression coverage for mention-only and missing-content events.

## Reproduction Assessment

Yes, at source level: construct a Feishu text event with a bot mention and missing or undefined `message.content`, then exercise `getFeishuSequentialKey` or the receive/debounce path. I did not live-replay Feishu traffic, and current main also shows a missing-content payload would be rejected before dispatch.

## Solution Assessment

Yes, a narrow Feishu plugin repair is the best path. Optional chaining only at the two reported trim sites is likely incomplete because the parser, receive validator, debounce resolver, and final empty-message handling need a consistent string-content boundary.
