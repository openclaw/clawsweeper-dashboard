## Summary

Keep open. Current main still has source-level unsafe Feishu missing-content paths, and live metadata shows an open implementation PR with closing syntax for this issue, so the issue should wait for that PR or a replacement to merge.

## What This Changes

The issue reports a Feishu crash when mention-only or otherwise missing-content messages reach parsing, sequential-key, or debounce paths that assume text content is a string.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77109 with parser-boundary content normalization, defensive trim guards, and regression coverage for missing `message.content` with mention metadata.

## Reproduction Assessment

Yes, at source level. Construct a Feishu text event with `message.content` removed and mention metadata present, then exercise `parseFeishuMessageEvent()` through `getFeishuSequentialKey()` or the receive/debounce resolver on current main.

## Solution Assessment

No, the issue's two-site optional-chaining suggestion is too narrow by itself. The better fix is to normalize missing content at the Feishu parser or receive boundary and keep downstream trim/preview paths defensive with regression coverage.
