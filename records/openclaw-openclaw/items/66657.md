## Summary

Keep open. The Feishu parser/sequential-key path on current main still lacks safe normalization for runtime-missing message content, and the linked open implementation PR explicitly targets this issue; close this only after that PR or a narrow replacement lands.

## What This Changes

Feishu should handle group or direct message events with empty or missing content without throwing string-method TypeErrors or dropping dispatch unexpectedly.

## Best Possible Solution

Normalize missing Feishu message content to a string at the parser boundary, keep downstream string-method paths safe, add focused mention-metadata regression coverage, and close this after the linked PR or a narrow replacement merges.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main can propagate runtime-missing Feishu content into parsed.content and then call trim or replace on it. I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment

Yes, the best fix is the narrow parser-boundary normalization plus focused downstream guards and regression coverage, not a broader Feishu routing change. The linked open PR is the current implementation candidate, but it needs review or replacement before this issue can close.
