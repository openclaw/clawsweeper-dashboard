## Summary

Keep open. Current main still has unsafe Feishu missing-content paths, and the open linked implementation PR explicitly owns the same Feishu empty-content crash, so this should close only after that PR or a narrow replacement merges.

## What This Changes

The issue reports Feishu group mention-only or empty-content messages crashing dispatch with `Cannot read properties of undefined (reading 'trim')` and dropping the inbound message.

## Best Possible Solution

Normalize missing Feishu `message.content` to a string at the parser or receive boundary, keep downstream string-method paths safe, cover mention-only missing-content events, then close this when the linked fix PR or a narrow replacement merges.

## Reproduction Assessment

Yes. Source inspection shows runtime-missing `message.content` can reach `parseFeishuMessageEvent()` and then `.replace()` or `.trim()` paths before the empty-message skip; I did not run a live Feishu tenant replay in this read-only review.

## Solution Assessment

Partly. Optional chaining at the trim sites is a plausible mitigation, but the best fix is parser/receive-boundary normalization plus downstream guards and focused regression coverage.
