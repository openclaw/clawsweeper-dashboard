## Summary

Keep open. Current main and v2026.6.10 still contain Feishu missing-content source paths, and https://github.com/openclaw/openclaw/pull/77109 is an open implementation candidate rather than a merged fix.

## What This Changes

Feishu group messages with only an @mention or missing/empty content can fail dispatch because Feishu parsing and routing paths still assume message content is present and string-like.

## Best Possible Solution

Review, refresh, and land https://github.com/openclaw/openclaw/pull/77109 or an equivalent narrow Feishu fix that accepts or normalizes missing content at the receive boundary, normalizes parser content before mention handling, guards downstream string calls, and adds focused regression proof.

## Reproduction Assessment

Yes. Source inspection shows current main still rejects absent Feishu `message.content` at the receive boundary and still has parser, sequential-key, and debounce paths where unnormalized content reaches string methods; I did not run a live Feishu tenant replay in this read-only review.

## Solution Assessment

No. Current main is not the best end state because the missing-content paths remain; the open linked PR is the right narrow direction, but the issue should close only after that PR or an equivalent fix lands.
