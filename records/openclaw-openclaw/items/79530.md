## Summary

Keep open: current main partially exposes assistant model metadata in the Control UI's collapsed Context details, but the requested visible per-message badge and TUI attribution are not implemented and need a UI/product decision.

## What This Changes

The issue requests a per-assistant-message model name badge/tag in Control UI WebChat and TUI for multi-model sessions and spawned tasks.

## Best Possible Solution

Define a small cross-surface message metadata design, then make Control UI and TUI render the recorded assistant model in that design when the message carries model data, falling back cleanly when it does not.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; source inspection is enough to show the current Control UI model is hidden in collapsed Context details and TUI assistant messages render without per-message model attribution.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the intended UX; a visible badge next to the assistant name is plausible, but the existing collapsed Context metadata means the narrowest change might be a discoverability/display adjustment rather than new data plumbing.
