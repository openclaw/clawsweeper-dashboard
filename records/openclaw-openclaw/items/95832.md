## Summary

Keep open: the PR is still necessary because current main and the latest release route Twilio voice-call REST traffic only through US1, and this branch appears to be a correct implementation candidate with sufficient safe regional proof; the remaining gate is maintainer acceptance of the new public config/auth-routing surface.

## What This Changes

Adds optional `twilio.region` voice-call config, derives exact Twilio REST hostnames for `us1`, `ie1`, and `au1`, applies those hostnames to Twilio POST/status requests and SSRF allowlists, and updates docs, manifest metadata, and tests.

## Best Possible Solution

Land the region-only `twilio.region` contract if maintainers accept the new plugin config surface, keeping exact enum-derived Twilio hostnames coupled to SSRF allowlists and the linked feature issue as the product record.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.9` hardcode Twilio REST requests and SSRF allowlists to `api.twilio.com`, with no `twilio.region` config. I did not run a credentialed current-main regional failure path.

## Solution Assessment

Yes, this appears to be the best narrow implementation shape: it avoids arbitrary base URLs, derives exact Twilio hostnames from a closed region enum, and keeps SSRF policy coupled to the request host. ([twilio.com](https://www.twilio.com/docs/global-infrastructure/using-the-twilio-rest-api-in-a-non-us-region))
