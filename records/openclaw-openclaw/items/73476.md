## Summary

Keep open: current main does not already implement `directReply`, and the proposed feature direction is understandable, but the PR still lets a tool terminate the run after a block-reply callback that may be suppressed or not visibly delivered.

## What This Changes

This PR adds an optional `directReply` tool-result flag and helpers, wires it through embedded agent subscriptions/runs to emit block replies and abort follow-up inference, and also changes QA/CI tests plus a parity-gate workflow setting.

## Best Possible Solution

Add the direct-reply contract only through a typed, send-policy-aware delivery path that reports visible delivery success before aborting, and split unrelated CI/QA stabilization into separate changes.

## Reproduction Assessment

Yes. A code-level reproduction is a successful tool result with `{ directReply: true, content: [{ type: "text", text: "done" }] }` in a non-block-streaming or `message_tool_only` source turn; the PR can abort before any visible final reply is delivered.

## Solution Assessment

No. The feature may be useful, but a fire-and-forget block callback followed by terminal abort is not the narrowest safe solution because current delivery policy can buffer or suppress that callback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Confirm direct replies before aborting:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1200-1205`
  - body: This calls `abortRun("direct_reply")` immediately after `emitBlockReply`, but current block delivery buffers text-only or captioned-media blocks when block streaming is off and drops block replies entirely under `message_tool_only`/suppressed delivery. A successful directReply can therefore finish with no visible user response; route through a send-policy-aware delivery path and abort only after delivery is allowed and confirmed.
  - confidence: 0.9
