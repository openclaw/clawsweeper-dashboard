## Summary

Keep open: current main does not implement the proposed directReply contract, and the latest PR head still aborts the agent run after a block-reply callback that may not actually deliver visible text under existing delivery policy paths.

## What This Changes

The PR adds optional directReply tool-result helpers and embedded-agent wiring to emit tool output as block replies and abort the follow-up LLM pass, plus tests, a changelog entry, QA mock stabilization, and a parity-gate workflow runtime setting.

## Best Possible Solution

Add a typed direct-reply contract only if it routes through a send-policy-aware delivery path that can report success, aborts the run after visible delivery is allowed and confirmed, and splits unrelated CI/QA stabilization into separate changes.

## Reproduction Assessment

Yes. A code-level reproduction is a successful tool result with `{ directReply: true, content: [{ type: "text", text: "done" }] }` while block streaming is disabled, or any turn using `message_tool_only`; the PR emits a block reply and then aborts before current delivery code can send a final reply.

## Solution Assessment

No. The feature direction may be useful, but aborting after a fire-and-forget block callback is not the narrowest safe solution; direct replies should use or thread a confirmed final/direct delivery result before terminal abort.
