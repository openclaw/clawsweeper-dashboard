## Summary

Keep this PR open, but do not merge it as-is: the contributor clarified a real use case for external single-capability skills, yet the current patch still terminates the run after scheduling a block reply without proving that source-chat delivery actually happened, and the submitted real-behavior proof is narrative-only.

## What This Changes

The PR adds a `directReply` tool-result flag and helper APIs, wires it through embedded agent tool handling and runner abort handling, adds directReply tests and a changelog entry, and includes small QA/test stability changes.

## Best Possible Solution

Route direct replies through a typed, send-policy-aware delivery path that confirms visible delivery or records a final-payload fallback before aborting, with maintainer-approved trust policy and focused coverage for suppressed-source modes.

## Reproduction Assessment

Yes, at source level for the blocking review finding: a successful text directReply with block streaming disabled, or any `message_tool_only`/suppressed source turn, reaches the new abort path before current delivery code has guaranteed visible output.

## Solution Assessment

No. The PR treats the block-reply callback as confirmed delivery; the safer solution should respect source visibility policy and terminate only after a send result or final fallback is secured.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Confirm direct-reply delivery before aborting:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1220-1225`
  - body: This emits the direct reply through the block-reply callback and immediately aborts the run. Text-only block replies are normally reconstructed into final text when block streaming is off, and source-suppressed modes return before sending, so this can turn a successful tool result into an invisible completed turn. Route directReply through a delivery path that reports visible delivery, or queue a final fallback before aborting.
