## Summary

Keep open: the latest PR head still aborts after a block-reply callback without confirming visible delivery, so text-only or source-suppressed direct replies can finish as intentional success with no user-visible output. The author’s external-agent scenario is useful context and the earlier Greptile nits were addressed, but this still needs stronger real behavior proof and maintainer agreement on the direct-delivery trust model.

## What This Changes

The PR adds a `directReply` tool-result flag and helpers, wires embedded-run abort handling, emits tool text/media through block replies, treats `direct_reply` aborts as successful, adds tests/changelog, and includes QA/test-stability fixes.

## Best Possible Solution

Route direct replies through a typed, send-policy-aware delivery path that confirms visible delivery or queues a final-reply fallback before aborting the model loop.

## Reproduction Assessment

Yes, at source level for the blocking review finding: a successful text directReply with block streaming disabled, or any `message_tool_only`/suppressed source turn, reaches the new abort path before current delivery code has guaranteed visible output.

## Solution Assessment

No. The PR uses the block-reply callback as if it were confirmed delivery; the safer path should respect source visibility policy and terminate only after a send result or fallback is secured.
