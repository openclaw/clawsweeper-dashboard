## Summary

Keep open: the PR adds a plausible new agent-tool direct-delivery feature, but the current branch still has a blocking delivery-policy bug, needs maintainer approval for the tool-controlled final-inference bypass, and lacks real after-fix runtime proof.

## What This Changes

The PR adds directReply tool-result helpers, wires embedded subscriber/runner abort handling, updates tests and changelog, and includes unrelated QA/test stability edits.

## Best Possible Solution

Add a typed, send-policy-aware direct-reply contract that only aborts after confirmed source-visible delivery or a preserved final fallback, with real gateway/channel proof for automatic and suppressed source-delivery modes.

## Reproduction Assessment

Yes, at source level: a directReply text result calls emitBlockReply and then aborts, while current main can suppress or buffer the block instead of visibly sending it under message_tool_only or disabled block streaming. I did not run live proof because this was a read-only review.

## Solution Assessment

No. Callback invocation is not the best delivery proof; the safer solution is a typed delivery path that honors source-reply policy and keeps a final fallback unless visible delivery is confirmed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Confirm delivery before aborting direct replies:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1225`
