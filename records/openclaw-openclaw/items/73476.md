## Summary

Keep this PR open, but it is not ready to merge: it adds a new tool-result delivery feature and still aborts the run after invoking a block-reply callback without proving that a visible send actually happened under existing delivery suppression and block-streaming rules.

## What This Changes

The PR adds an opt-in `directReply` tool-result flag that emits tool output as a block reply, aborts the embedded run as `direct_reply`, adds helper APIs/tests/changelog, and includes several QA/test stability changes.

## Best Possible Solution

Route directReply through a typed, send-policy-aware delivery path that confirms a visible send or preserves a final fallback before aborting, with tests for block streaming disabled and message-tool-only suppression.

## Reproduction Assessment

Yes, at source level. The PR directReply path can invoke `emitBlockReply` and abort, while current main can legally suppress that block reply or buffer text-only blocks into final text instead of sending immediately.

## Solution Assessment

No. Callback invocation is not the best delivery proof; the maintainable solution should make the delivery layer report success or retain a final-payload fallback before ending the run.
