## Summary

Keep open: current main still has no directReply tool-result contract, and the PR targets a real external-agent result use case, but the latest diff can still abort a turn before the reply is actually deliverable and also mixes unrelated CI/QA changes.

## What This Changes

The PR adds an optional directReply tool-result contract with embedded-agent abort wiring, helper constructors, tests, a changelog entry, QA mock/test stabilization, and a parity-gate Node 24 workflow env override.

## Best Possible Solution

Implement direct replies as a typed, policy-aware delivery path that confirms visible delivery or deliberately falls back before aborting, and split unrelated CI/QA stabilization from the feature PR.

## Reproduction Assessment

Yes. A source-level reproduction is a successful tool result with directReply true and text content while block streaming is disabled, or while source delivery is message_tool_only; current delivery code either buffers/drops the block path while the PR aborts before final reconstruction.

## Solution Assessment

No. The feature direction is plausible, but aborting after a fire-and-forget block callback is not the narrowest maintainable solution because current delivery policy can suppress or defer that callback path.

## Review Findings

- **[P1] Confirm delivery before aborting direct replies:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1231`
