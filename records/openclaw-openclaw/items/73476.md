## Summary

Keep open; the latest head fixes some earlier edge cases, but it still treats block-reply callback invocation as visible delivery before aborting the run, which conflicts with existing private/buffered delivery paths.

## What This Changes

The PR adds directReply helper APIs, forwards abortRun into embedded tool subscription, treats direct_reply as a successful runner abort, adds tests/changelog, and includes several QA/test stability changes.

## Best Possible Solution

Route directReply through a typed, send-policy-aware delivery path that confirms visible delivery or records a final-payload fallback before aborting, with coverage for block streaming disabled and message_tool_only source modes.

## Reproduction Assessment

Yes, at source level. A text directReply can reach ctx.emitBlockReply followed by abortRun while current main either buffers text-only blocks when streaming is off or suppresses block replies entirely for message_tool_only/source-suppressed turns.

## Solution Assessment

No. The patch uses callback invocation as delivery proof; the maintainable fix should terminate only after the existing delivery layer reports a visible send or a final fallback is preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Confirm visible delivery before direct-reply abort:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1220-1225`
  - body: DirectReply calls the block-reply callback and then aborts the run. That callback can legally no-op for text-only blocks when block streaming is disabled, and source-suppressed/message_tool_only turns return before sending, so a successful tool result can complete with no visible reply. Route this through a delivery path that confirms a send or preserves a final fallback before aborting.
