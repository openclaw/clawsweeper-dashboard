## Summary

Keep open: the feature direction is plausible, but the patch still aborts after a block-reply callback that can be buffered or suppressed, and the PR still lacks real after-fix behavior proof.

## What This Changes

The PR adds a tool-result `directReply` flag and helpers, wires embedded-agent abort support, emits tool text/media through block replies, treats `direct_reply` aborts as successful, adds tests/changelog, and includes QA/test-stability fixes.

## Best Possible Solution

Route direct replies through a typed, send-policy-aware delivery path that confirms visible delivery or queues a final-reply fallback before aborting the model loop.

## Reproduction Assessment

Yes, at source level for the blocking review finding. A successful text directReply with block streaming disabled, or any `sourceReplyDeliveryMode: "message_tool_only"` turn, reaches the new abort path before current delivery code has guaranteed visible output.

## Solution Assessment

No. The current PR path is not the best fix because it uses the block-reply callback as if it were confirmed delivery; the safer design should respect source visibility policy and only terminate after a send result or fallback is secured.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Confirm direct-reply delivery before aborting:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1220-1225`
  - body: This emits the direct reply through the block-reply callback and immediately aborts the run. Text-only block replies are normally reconstructed into the final reply when block streaming is off, and source-suppressed modes return before sending, so this can turn a successful tool result into an invisible completed turn. Route directReply through a delivery path that reports visible delivery, or queue a final fallback before aborting.
  - confidence: 0.91
