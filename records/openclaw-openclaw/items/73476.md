## Summary

Keep open. Current main does not implement a direct-reply tool result contract, so the PR is still a live feature proposal, but the latest head still aborts the embedded run after requesting a block reply without knowing whether source-delivery policy allowed or queued it. The mixed CI/QA stabilization changes also need maintainer/CI-owner review rather than cleanup closure.

## What This Changes

The PR adds a `directReply` flag and helpers for agent tool results, wires embedded-agent abort handling, adds tests and changelog coverage, and also adjusts parity-gate, QA mock, gateway, command, and test-script assertions.

## Best Possible Solution

Define a typed direct-reply contract that participates in existing source delivery policy: preserve current media filtering, make abort contingent on visible delivery being allowed and actually queued, and fall back to normal inference or explicit `message` tool guidance when automatic delivery is suppressed. Split unrelated CI/QA stabilization into separate reviewable changes.

## Reproduction Assessment

Yes. A code-level reproduction is a default group/channel or explicit `message_tool_only` turn where a successful tool returns `{ directReply: true, content: [{ type: "text", text: "done" }] }`: the PR branch reaches `abortRun("direct_reply")`, while current main's dispatch wrapper drops block replies under `suppressDelivery`.

## Solution Assessment

No. The direct-reply seam may be the right product direction for external-agent tools, but this implementation is not the narrowest maintainable fix until the abort decision is tied to delivery policy or a confirmed queued reply, and the unrelated CI/QA changes are separated.
