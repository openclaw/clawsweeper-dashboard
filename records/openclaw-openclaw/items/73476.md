## Summary

Keep open. Current main does not implement a directReply tool-result contract, and the PR still has a blocking delivery bug: it emits through the block-reply path and then aborts, but current block-reply delivery can intentionally drop text-only blocks when block streaming is off or source delivery is suppressed, producing a successful invisible turn.

## What This Changes

The PR adds a `directReply` tool-result flag and helpers, wires embedded-agent abort callbacks through the Pi subscription path, adds direct text/media reply handling and tests, and also changes parity-gate/QA/test assertions.

## Best Possible Solution

Keep the feature direction narrow: add a typed direct-reply contract only if it uses a delivery path that can report whether text/media was actually queued or sent, honors `message_tool_only` and send-policy suppression, and aborts the run only after visible delivery is allowed and successful. Split the parity-gate and QA stabilization work into separate reviewable changes.

## Reproduction Assessment

Yes. A code-level reproduction is a successful tool result with `{ directReply: true, content: [{ type: "text", text: "done" }] }` while block streaming is disabled, or any turn using `message_tool_only`; current delivery code drops that block reply and the PR then aborts the run as successful.

## Solution Assessment

No. The direct-reply seam may be useful for external-agent tools, but this implementation uses block-reply emission as if it were confirmed delivery; the safer solution is to thread delivery policy/success through the directReply branch or use an explicit final/message delivery path before aborting.
