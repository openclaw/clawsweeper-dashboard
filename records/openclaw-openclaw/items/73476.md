## Summary

Keep open. Current main does not have the requested direct-reply tool-result contract, and the PR targets a real external-agent use case, but the patch should not merge as-is because it can mark a turn successful after current visible-reply policy suppresses the emitted block reply.

## What This Changes

The PR adds `directReply` tool-result helpers and embedded-agent abort wiring to deliver tool text/media directly, plus tests, a changelog entry, parity-gate Node 24 env wiring, QA mock stabilization, and several test assertion updates.

## Best Possible Solution

Land a narrow, typed direct-reply contract only after it participates in source delivery policy: abort only after visible delivery is allowed or actually queued, fall back to normal inference or message-tool guidance when suppressed, and split unrelated CI/QA stabilization into separate reviewable changes.

## Reproduction Assessment

Yes. A code-level reproduction is a successful tool result with `{ directReply: true, content: [{ type: "text", text: "done" }] }` during a default group/channel turn or any turn configured with `messages.visibleReplies: "message_tool"`; current wrappers drop the block reply while the PR aborts the model run as `direct_reply`.

## Solution Assessment

No. The direct-reply seam may be the right product direction for external-agent tools, but this implementation is not the narrowest maintainable solution until delivery policy or delivery success is threaded into the abort decision.

## Review Findings

Overall correctness: patch is incorrect
