## Summary

Keep open: the latest head still aborts after a block-reply callback that can legally drop or buffer text under default non-block-streaming and source-suppressed delivery modes, so directReply can produce an invisible successful turn. The external PR also still lacks real after-fix behavior proof.

## What This Changes

The PR adds a directReply tool-result contract, helper functions, embedded-agent abort wiring, tests, a changelog entry, and several QA/test-stability adjustments.

## Best Possible Solution

Route direct replies through a typed delivery path that respects source visibility policy and only aborts after visible delivery is confirmed or a final-reply fallback is queued, with unrelated QA/test stabilizers split if maintainers want a smaller review surface.

## Reproduction Assessment

Yes, at source level. A successful tool result like directReplyResult("done") with block streaming disabled, or any sourceReplyDeliveryMode=message_tool_only turn, reaches the new abort path before the current delivery code can send a visible final reply.

## Solution Assessment

No. The feature direction may be useful, but aborting after a fire-and-forget block callback is not the narrowest maintainable solution; the direct path should use a send-policy-aware delivery result or final-reply fallback before terminating the run.
