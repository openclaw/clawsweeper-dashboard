## Summary

Keep open: current main still has no directReply tool-result contract, and the PR addresses a real external-agent use case, but the latest diff still aborts after a block-reply callback that can be suppressed or buffered without visible delivery.

## What This Changes

The PR adds a directReply tool-result flag and helpers, wires embedded-agent block-reply plus abort handling, adds tests and changelog coverage, and includes QA mock/test plus parity workflow changes.

## Best Possible Solution

Add a typed direct-reply contract only after it routes through send-policy-aware delivery that confirms or records visible delivery before aborting, and split unrelated CI/QA stabilization into separate changes.

## Reproduction Assessment

Yes. A code-level reproduction is a successful tool result with top-level directReply true and text while block streaming is disabled, or the same path under message_tool_only delivery; the PR aborts before current final/block delivery paths can guarantee a visible reply.

## Solution Assessment

No. The feature direction is useful, but aborting after a fire-and-forget block callback is not the narrowest maintainable solution; direct delivery should be confirmed through the existing send-policy path first.
