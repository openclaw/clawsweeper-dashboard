## Summary

Keep open: this is a recent feature PR, but the current diff still treats callback invocation as delivery proof before aborting the run, and the supplied proof is placeholder-style rather than real after-fix output.

## What This Changes

The PR adds an opt-in `directReply` flag/helpers for agent tool results, wires embedded-run abort handling, adds directReply tests/changelog, and includes QA/test stability adjustments.

## Best Possible Solution

Route directReply through a typed, send-policy-aware delivery path that confirms a visible send or preserves a final fallback before aborting, with coverage for `message_tool_only` and disabled block streaming.

## Reproduction Assessment

Yes, at source level. The PR directReply path can invoke `emitBlockReply` and abort, while current delivery can suppress or buffer the block instead of sending it visibly.

## Solution Assessment

No. Callback invocation is not the best delivery proof; the maintainable fix should make delivery policy explicit and preserve a fallback before ending the run.
