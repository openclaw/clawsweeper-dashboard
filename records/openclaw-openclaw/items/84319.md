## Summary

Keep open: current main still suppresses Slack `isReasoning` payloads only in native streaming, while normal delivery, shared reply delivery, and slash-command replies can send them unfiltered. The linked fix PR at https://github.com/openclaw/openclaw/pull/84322 is still open with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Slack non-streaming, streaming-fallback, and slash-command delivery paths can expose Claude reasoning payloads that should remain hidden.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84322 after real Slack proof, or apply an equivalent Slack-owned suppression boundary for normal, fallback, helper, and slash reply delivery with focused regression tests.

## Reproduction Assessment

Yes, by source inspection: send a Slack reply payload with `isReasoning: true` while native streaming is disabled, in fallback delivery, or through slash replies, and current main has no suppression check before the Slack send/respond helpers. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is a Slack delivery-boundary filter plus regression tests, matching the existing streaming and generic-dispatch suppression contract. The issue should stay open while the linked PR is reviewed and proved, not be closed before merge.
