## Summary

Keep open: current main still has Slack reasoning suppression only in the native streaming branch, while normal delivery, stream fallback through `deliverReplies`, and slash-command replies can send `isReasoning` payloads. The source path is narrow enough for a focused Slack plugin fix with regression tests.

## What This Changes

Suppress Claude reasoning/thinking reply payloads across Slack non-streaming, fallback, and slash-command delivery paths.

## Best Possible Solution

Add a single Slack-owned suppression boundary shared by normal replies, fallback replies, and slash replies, reusing the existing reply-payload reasoning contract where appropriate and preserving deliberate heartbeat reasoning delivery that clears the flag before routing.

## Reproduction Assessment

Yes, by source inspection: send a Slack reply payload with `isReasoning: true` while native streaming is disabled or while using slash replies, and the current helper path has no suppression check before sending. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow Slack delivery-boundary filter plus regression tests for normal delivery, fallback delivery, and slash replies. Adding a config option or changing model reasoning generation would be broader than the bug requires.
