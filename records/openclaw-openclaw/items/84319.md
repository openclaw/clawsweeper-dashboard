## Label Justifications

- `P1`: The issue describes model reasoning text being posted to real Slack users through channel delivery paths that should suppress it.
- `impact:message-loss`: The affected surface is Slack message delivery: payloads that should be suppressed can become user-visible channel or slash-command messages.

## Summary

Keep open: current main still has a source-proven Slack delivery gap, and two open pull requests explicitly reference this issue with closing syntax. The issue should stay open until one focused fix lands or the linked PRs are closed.

## What This Changes

The issue asks Slack to suppress Claude reasoning/thinking payloads in non-streaming, fallback, and slash-command delivery paths.

## Best Possible Solution

Land one focused Slack plugin fix that filters `isReasoning` before normal, fallback, helper, and slash delivery, with regression tests, then close the issue from the merged PR.

## Reproduction Assessment

Yes, at source level: send a Slack `ReplyPayload` with `isReasoning: true` while native streaming is disabled, or pass one through `deliverSlackSlashReplies`; current main routes it to Slack send/respond helpers without a suppression check. I did not run a live Slack workspace reproduction.

## Solution Assessment

Yes: the narrow maintainable fix is a Slack-owned delivery-boundary filter plus focused regression tests, matching the existing generic and streaming reasoning contract. Because open PRs already reference this issue with closing syntax, the next step is PR review/landing rather than a new repair branch.
