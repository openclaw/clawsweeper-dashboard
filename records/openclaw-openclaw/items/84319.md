## Label Justifications

- `P1`: The issue describes internal reasoning text being delivered into real Slack channels for Claude/Bedrock users, which is an urgent broken channel workflow.
- `impact:message-loss`: The affected problem class is Slack message delivery: payloads that should be suppressed can be sent into the user-visible transcript.

## Summary

Keep open: current main still suppresses Slack reasoning payloads only on the native streaming branch, while normal delivery, fallback delivery, and slash-command replies can still send `isReasoning` payloads. The open linked fix PR should be reviewed or replaced before this issue closes.

## What This Changes

The issue asks Slack to suppress Claude/Bedrock reasoning payloads in non-streaming, fallback, and slash-command delivery paths.

## Best Possible Solution

Land the focused Slack-owned suppression fix in https://github.com/openclaw/openclaw/pull/84322, or an equivalent replacement, with regression tests and redacted live Slack proof that reasoning text no longer appears.

## Reproduction Assessment

Yes by source inspection: send a Slack reply payload with `isReasoning: true` while native streaming is disabled or through slash replies, and current main has no guard before sending. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to add a Slack delivery-boundary suppression check shared by normal replies, fallback replies, and slash replies. A config option or provider-level reasoning change would be broader than this bug requires.
