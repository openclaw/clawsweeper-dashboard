## Summary

Keep open: current main still suppresses Slack reasoning payloads only on the native streaming path, while normal delivery and slash-command helpers can still send `isReasoning` payloads. The issue is already paired with the open fix PR https://github.com/openclaw/openclaw/pull/84322, so it should stay open until that PR lands or is replaced.

## What This Changes

The issue requests Slack to suppress Claude reasoning/thinking payloads across non-streaming, fallback, and slash-command delivery paths.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84322, or an equivalent narrow Slack-owned suppression boundary with regression tests, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, by source inspection: send a Slack reply payload with `isReasoning: true` while native streaming is disabled or through slash replies, and current main has no suppression check before sending. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: a narrow Slack delivery-boundary filter with regression coverage for normal, fallback, and slash replies is the maintainable fix. Changing provider reasoning generation or adding a config option would be broader than the existing payload contract requires.
