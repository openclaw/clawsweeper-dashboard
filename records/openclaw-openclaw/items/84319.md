- `impact:message-loss`: The failure is an incorrect channel delivery path that sends payloads users should never receive.

## Summary

Keep open: current main still has source-reproducible Slack non-streaming and slash-command paths that can send `isReasoning` payloads, and https://github.com/openclaw/openclaw/pull/84322 is the active closing PR to review rather than closing this issue early.

## What This Changes

The issue asks OpenClaw to suppress Claude `isReasoning` thinking payloads across Slack non-streaming, stream-fallback, normal, and slash-command delivery paths.

## Best Possible Solution

Review and land the focused Slack-local suppression in https://github.com/openclaw/openclaw/pull/84322 with regression coverage for normal, fallback, block/media, and slash replies; leave broader inter-tool narration policy to https://github.com/openclaw/openclaw/issues/25592.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main emits `isReasoning` reply payloads and Slack normal, helper, and slash delivery paths do not suppress them. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to guard `isReasoning` before every Slack send boundary that can bypass native streaming. The linked PR follows that direction and should be reviewed rather than closing the issue before it merges.
