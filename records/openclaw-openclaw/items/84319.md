## What This Changes

Suppress Claude reasoning/thinking payloads from Slack non-streaming, fallback, helper, and slash-command delivery paths.

## Best Possible Solution

Review and land or reject the linked Slack plugin PR; the desired end state is one Slack-owned reasoning suppression boundary shared by normal replies, stream fallback, helper delivery, and slash replies.

Yes. By source inspection, send a Slack `ReplyPayload` with `isReasoning: true` while native streaming is disabled or through slash replies, and current main has no suppression guard before the send/respond helpers.

## Solution Assessment

Yes. A narrow Slack delivery-boundary filter with regression coverage is the best fit; changing provider reasoning generation or adding a config option would be broader than this bug requires.
