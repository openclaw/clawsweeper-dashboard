- `P1`: The issue describes internal reasoning text being visibly delivered into Slack channels through normal user workflows.

## Summary

Keep open: current main still only suppresses Slack reasoning payloads in the native streaming branch, while normal delivery, stream fallback, shared reply helpers, and slash replies can still send `isReasoning` payloads. The issue is also paired with an open closing PR, so it should remain open until that fix lands or is replaced.

## What This Changes

Prevent Claude reasoning/thinking payloads from reaching Slack non-streaming, fallback, shared reply, and slash-command delivery paths.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84322 or an equivalent Slack-owned suppression boundary with focused regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes, by source inspection: send a Slack reply payload with `isReasoning: true` through non-streaming normal delivery or slash-command replies, and current main has no guard before building/sending Slack messages. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: a narrow Slack delivery-boundary filter plus regression tests is the best fix shape, matching the existing streaming and generic dispatch contracts without adding config or changing model reasoning generation.
