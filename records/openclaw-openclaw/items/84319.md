## Summary

Keep open: current main and the latest release still suppress Slack reasoning payloads only in the native streaming branch, while normal delivery, fallback delivery through the shared helper, and slash-command replies remain unguarded. The linked closing PR is open, so the issue should track that review rather than be closed early.

## What This Changes

The issue asks Slack to suppress Claude reasoning/thinking payloads in non-streaming, stream-fallback, and slash-command delivery paths.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84322, or an equivalent Slack-owned guard that filters reasoning payloads before every Slack send boundary with focused regression coverage.

## Reproduction Assessment

Yes. By source inspection, send an `isReasoning: true` Slack reply payload through non-streaming delivery or slash-command replies and the current helper path has no suppression check before sending.

## Solution Assessment

Yes. A narrow Slack delivery-boundary filter with tests for normal delivery, fallback delivery, and slash replies is the maintainable fix; changing model reasoning generation or adding a config option would be broader than needed.
