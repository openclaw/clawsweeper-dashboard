## Summary

Keep open. Current main and `v2026.6.8` still generate macOS LaunchAgent plists with `StandardErrorPath` set to `/dev/null`; https://github.com/openclaw/openclaw/pull/90828 is the active open candidate fix but has not merged.

## What This Changes

Issue reports that macOS LaunchAgent generation sends gateway stderr to `/dev/null`, hiding launchd-level diagnostics instead of writing a real stderr log.

## Best Possible Solution

Review and either land or revise https://github.com/openclaw/openclaw/pull/90828 so the plist writer, diagnostics, status output, docs, tests, and macOS launchd proof agree on the desired stderr contract.

## Reproduction Assessment

Yes, source-level: current main still passes `/dev/null` as the LaunchAgent `StandardErrorPath` in both install and restart rewrite paths, and an adjacent test asserts that value. The reporter also posted live macOS reproduction comments, but I did not run a live LaunchAgent install in this read-only review.

## Solution Assessment

Partly. Routing stderr to the resolved launchd log path is the narrow daemon-layer fix, but the best merge shape also needs aligned diagnostics/docs, retention judgment, and real macOS proof.
