## Summary

Keep open. Current main and v2026.6.1 still generate macOS Gateway LaunchAgent plists with stderr suppressed to `/dev/null`, and the new comments add credible live diagnostic-impact evidence; because the changelog and runtime hints also describe suppression as intentional, this needs maintainer logging-contract review rather than cleanup closure.

## What This Changes

The issue requests restoring macOS Gateway LaunchAgent stderr from `/dev/null` to a real diagnostic log path such as `gateway.err.log`.

## Best Possible Solution

Maintainers should choose the macOS service logging contract explicitly, then either keep stderr suppressed with a durable alternate diagnostic path or restore LaunchAgent stderr to the resolved `gateway.err.log` path with install, restart-rewrite, status, and upgrade-repair coverage.

## Reproduction Assessment

Yes for the reported plist value: current source writes `/dev/null` as `StandardErrorPath`, and current tests assert that generated and rewritten plists contain it. The live outage and memory-pressure scenarios are credible issue evidence but were not reproduced in this read-only Linux worker.

## Solution Assessment

Unclear. Using the existing `resolveGatewaySupervisorLogPaths(env).stderrPath` is a narrow technical fix, but current changelog and status hints show stderr suppression is intentional shipped behavior that needs maintainer approval before reversal.
