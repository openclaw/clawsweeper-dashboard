## Summary

Keep open. Current main and v2026.6.1 do write the macOS Gateway LaunchAgent stderr path to `/dev/null`, but the changelog says that suppression was intentional for launchd stability, and the cited subsystem diagnostics are also routed through OpenClaw file logging; changing this needs maintainer judgment rather than cleanup closure.

## What This Changes

The issue requests restoring a real macOS LaunchAgent `StandardErrorPath` for the Gateway instead of `/dev/null`.

## Best Possible Solution

Make a maintainer decision on the macOS launchd logging contract: either preserve stderr suppression and document the supported file-log/`openclaw logs` path clearly, or restore a safe stderr file path with regression coverage for the prior launchd stability failures.

## Reproduction Assessment

No high-confidence live reproduction of the full diagnostic-loss claim is available from this read-only review. Source inspection does reproduce the plist value: current main and v2026.6.1 set `StandardErrorPath` to `/dev/null`, while current logging code still routes the cited subsystem warnings to file logs.

## Solution Assessment

Unclear. Directly switching `StandardErrorPath` to `gateway.err.log` is plausible, but not proven best because current release notes say stderr suppression fixed launchd startup/log failures; maintainers should choose the logging contract first.
