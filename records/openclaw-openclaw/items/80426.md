## Summary

Keep open: the feature is well scoped, but the PR should not merge yet because the new ack cleanup flag can be set after a suppressed reasoning-only Mattermost payload, causing `removeAckAfterReply` to remove the ack even though no visible reply was delivered.

## What This Changes

The PR adds Mattermost automatic ack reactions with channel/account ackReaction overrides, emoji-name normalization, docs, generated config metadata, and unit coverage.

## Best Possible Solution

Keep the Mattermost-owned ack feature, but make ack cleanup depend on an actual visible Mattermost reply delivery or preview finalization and add a regression for suppressed reasoning or silent replies.

## Reproduction Assessment

Yes for the review finding: source inspection shows a reasoning-prefixed Mattermost final is intentionally suppressed, while the PR sets the delivered flag after that helper returns. That gives a high-confidence path where `removeAckAfterReply` can remove an ack without a visible reply.

## Solution Assessment

No; the current implementation should not use every successful helper return as a delivered-reply signal. The safer fix is to return or track delivery status from the Mattermost preview/final delivery path and only cleanup after `normal-delivered` or `preview-finalized` behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Track actual delivery before removing the ack:** `extensions/mattermost/src/mattermost/monitor.ts:1729`
  - body: Set `anyReplyDelivered` only when a visible reply was actually sent or a preview was finalized. `deliverMattermostReplyWithDraftPreview` returns early for reasoning payloads, and current tests assert those payloads do not call `deliverFinal`; with this line after the await, a reasoning-only or otherwise suppressed payload still marks the turn as delivered and `removeAckAfterReply` can remove the ack even though Mattermost never got a visible reply.
  - confidence: 0.9
