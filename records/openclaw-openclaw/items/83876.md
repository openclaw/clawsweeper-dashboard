## Summary

This should stay open: current main and the latest release still let `browser click-coords abc 200` build a `clickCoords` body with `NaN`, and the open linked PR is the correct implementation path but has not merged yet.

## What This Changes

Reject non-finite `browser click-coords` coordinates before sending the browser action request, with resize validation kept in the shared resize helper.

## Best Possible Solution

Review and land the linked fix PR, then close this issue as implemented once the guard is on main and covered by focused regression proof.

## Reproduction Assessment

Yes, at source level: `browser click-coords abc 200` flows through bare `Number()` parsing and builds a `/act` body before any finite-number guard. A read-only Node probe confirms that the resulting `NaN` coordinate serializes to JSON `null`.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite `click-coords` coordinates at the CLI boundary before constructing the action body, while leaving resize validation in the existing shared helper.
