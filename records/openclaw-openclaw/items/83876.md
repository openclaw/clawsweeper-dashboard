## Summary

This looks like a real current-main browser CLI validation bug: `click-coords` still sends unguarded `Number()` results into the browser action body, while the resize path is already guarded in its shared helper.

## What This Changes

The issue asks OpenClaw to reject invalid numeric browser CLI coordinates locally instead of allowing NaN values to become null in the browser action request.

## Best Possible Solution

Add a finite-number guard for `click-coords` before building the action body, keep the existing resize helper validation, and cover the local rejection path with a focused CLI regression test.

## Reproduction Assessment

Yes from source inspection: `browser click-coords abc 200` flows through `Number('abc')` with no guard before the `/act` body is built. I did not execute the CLI itself because this review must remain read-only.

## Solution Assessment

Yes; the best fix is a narrow local validation check plus a regression test, not a new command mode or broader browser-runtime change.
