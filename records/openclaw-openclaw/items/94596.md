## Summary

Keep open: current main still has the SSH tunnel caller probing the preferred local port without a host even though the rest of that tunnel path is IPv4-loopback scoped, so the reported address-family mismatch remains on this surface.

## What This Changes

The issue requests making `startSshPortForward` call the scoped port-availability helper for `127.0.0.1` so IPv4-loopback occupants trigger the existing ephemeral-port fallback.

## Best Possible Solution

Update the SSH tunnel caller to probe `127.0.0.1` for the preferred local port and add focused coverage showing the existing fallback chooses an ephemeral port when that IPv4-loopback port is occupied.

## Reproduction Assessment

Yes, source-level for the affected platform: current main probes the preferred SSH tunnel port without a host and then forwards and checks on `127.0.0.1`. I did not run a live dual-stack affected-host reproduction in this read-only review.

## Solution Assessment

Yes. Passing `127.0.0.1` at the SSH caller is the narrowest maintainable fix because it matches `pickEphemeralPort`, `canConnectLocal`, and the `ssh -L` binding without changing global helper semantics.
