## Summary

Keep open: current main still calls the now host-scoped port helper without a host in the SSH tunnel path, while the tunnel allocation and connection path are IPv4-loopback scoped. The merged related fix only covered the Browser Chrome caller and did not update `startSshPortForward`.

## What This Changes

The issue requests scoping `startSshPortForward`'s preferred-port preflight to `127.0.0.1` so it matches the SSH local-forward interface.

## Best Possible Solution

Update the SSH tunnel caller to probe `127.0.0.1` for the preferred local port and add focused coverage showing an IPv4-loopback occupant triggers the existing ephemeral-port fallback.

## Reproduction Assessment

Yes, source-level for the affected platform: current main probes the preferred SSH tunnel port without a host and then forwards on `127.0.0.1`. I did not reproduce the false-negative on this Linux runner because its hostless bind conflicts with the IPv4 occupant.

## Solution Assessment

Yes. Passing `127.0.0.1` at the SSH caller is the narrowest maintainable fix because it matches the interface used by `pickEphemeralPort`, `canConnectLocal`, and the `ssh -L` binding without changing global helper semantics.
