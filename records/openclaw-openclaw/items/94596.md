## Summary

Keep open: current `main` and the latest release still have the hostless SSH tunnel preflight, and several open PRs now reference this issue as candidate fixes rather than a merged implementation.

## What This Changes

The issue asks for `startSshPortForward` to pass `127.0.0.1` to `ensurePortAvailable` so the SSH tunnel port preflight checks the same IPv4 loopback interface used by the tunnel.

## Best Possible Solution

Land one canonical SSH tunnel fix that scopes the preflight to `127.0.0.1`, handles the resulting `PortInUseError` fallback path, and includes focused regression coverage; then close this issue and the duplicate PRs.

## Reproduction Assessment

Yes. Source inspection shows current `main` probes the preferred SSH tunnel port without a host while the same path binds and checks `127.0.0.1`; I did not run a live dual-stack IPv4-only occupant repro in this read-only review.

## Solution Assessment

Yes. The narrow fix is to scope this caller to `127.0.0.1` and cover the `PortInUseError` fallback behavior; the remaining task is choosing and landing one existing candidate PR.
