## Summary

Keep open: current main and v2026.6.8 still have the hostless SSH tunnel preferred-port preflight, and several open PRs are candidate fixes, so this should close only after a canonical PR lands.

## What This Changes

The issue asks to make SSH tunnel preferred-port preflight probe the IPv4 loopback interface instead of using a hostless port probe.

## Best Possible Solution

Land one canonical SSH tunnel fix that scopes the preflight to `127.0.0.1`, routes `PortInUseError` to the ephemeral-port fallback, and adds focused regression coverage.

## Reproduction Assessment

Yes, source-level: current main probes the preferred SSH tunnel port without a host while the same tunnel path binds and checks `127.0.0.1`. I did not run a live affected dual-stack host reproduction in this read-only pass.

## Solution Assessment

Yes, with one caveat: the best fix is the caller-scoped `127.0.0.1` probe plus `PortInUseError` fallback handling, not a one-line host argument alone.
