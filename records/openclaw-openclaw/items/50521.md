## Summary

Keep open. The hardening remains valid, but this PR is now stale against current main: OpenClaw delegates archive path validation and extraction to `@openclaw/fs-safe@0.2.1`, whose archive-entry validator still lacks null-byte rejection, while this PR edits the old local implementation and is currently conflicting.

## What This Changes

The PR adds an early null-byte rejection to archive entry path validation and adds two tests for null bytes in filename and directory components.

## Best Possible Solution

Land an equivalent null-byte rejection in the current fs-safe archive validator, update OpenClaw to the fixed dependency, keep focused regression coverage, and gate the security hardening with maintainer review plus real behavior proof.

## Reproduction Assessment

Yes. Source inspection of `@openclaw/fs-safe@0.2.1` shows no null-byte guard in `validateArchiveEntryPath()`, and a read-only Node probe confirms normalization preserves the byte while filesystem calls reject it with `ERR_INVALID_ARG_VALUE`.

## Solution Assessment

No, not as currently based. The guard belongs in the current `@openclaw/fs-safe` archive-entry implementation or a released dependency update consumed by OpenClaw, not in the old local helper that current main has replaced with a re-export.
