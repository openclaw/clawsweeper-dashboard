## Summary

Keep open: the security hardening is still valid, but this PR is not correct against current main because archive validation and extraction now come from `@openclaw/fs-safe`, while the PR changes the old local helper and still lacks real behavior proof.

## What This Changes

The PR adds a null-byte guard to archive entry path validation and two regression tests for null bytes in archive entry names.

## Best Possible Solution

Land the null-byte rejection in `@openclaw/fs-safe`'s archive-entry validator, publish/update OpenClaw to that fixed dependency, and keep focused OpenClaw regression coverage for archive entry null-byte rejection.

## Reproduction Assessment

Yes. Source inspection shows current main delegates archive path validation to `@openclaw/fs-safe@0.2.1`, whose published validator lacks a null-byte guard, and a read-only Node probe confirms normalization preserves the byte before lower-level fs APIs reject it.

## Solution Assessment

No. The validation check is appropriate, but this PR applies it to the pre-fs-safe local helper; the maintainable fix belongs in `@openclaw/fs-safe` followed by an OpenClaw dependency update and regression tests.
