## Summary

Keep open: this is useful security hardening, but the branch now edits the old local archive-path helper while current main delegates runtime archive validation to `@openclaw/fs-safe`; the external PR proof gate is also still missing.

## What This Changes

The PR adds a null-byte rejection guard to archive entry path validation and two validator tests for null bytes in file and directory names.

## Best Possible Solution

Land the null-byte rejection in `@openclaw/fs-safe` archive-entry validation, update OpenClaw to the fixed package, and keep focused validator plus ZIP/TAR preflight regression coverage.

## Reproduction Assessment

Yes, source-level: current main re-exports archive validation and extraction from pinned `@openclaw/fs-safe@0.2.4`, whose published validator lacks a `\0` guard while ZIP/TAR preflight routes entry paths through it. I did not run a runtime extraction probe in this read-only review.

## Solution Assessment

No. The proposed guard is conceptually correct, but the maintainable fix belongs in the `@openclaw/fs-safe` archive validator plus an OpenClaw dependency update, not in the obsolete local helper body.
