## Summary

Keep open: the security hardening is valid, but the branch now targets an obsolete local validator while current main delegates archive validation and extraction to `@openclaw/fs-safe@0.2.4`, which still lacks the null-byte guard.

## What This Changes

The branch adds a `\0` guard to the former local `validateArchiveEntryPath()` implementation and adds two archive path tests for null-byte entry names.

## Best Possible Solution

Land null-byte rejection in `@openclaw/fs-safe`'s archive-entry validator, publish or consume the fixed package in OpenClaw, and keep focused OpenClaw regression coverage for archive entries containing null bytes.

## Reproduction Assessment

Yes. Source inspection of current main plus the pinned `@openclaw/fs-safe@0.2.4` package shows the live validator lacks a null-byte check and both ZIP and TAR extraction call that validator.

## Solution Assessment

No. The proposed guard is conceptually right, but after current-main refactoring it belongs in `@openclaw/fs-safe` and then in an OpenClaw dependency update rather than in the obsolete local implementation.
