## Summary

Keep open: the security hardening goal is valid, but this PR is stale against current main and patches the old local archive-path helper. Current main delegates archive validation and extraction to `@openclaw/fs-safe@0.2.2`, whose published validator still accepts null-byte entry paths, so the fix needs to move to that package boundary before OpenClaw can consume it.

## What This Changes

The PR adds explicit null-byte rejection to archive entry path validation and two regression tests for null-byte entry names.

## Best Possible Solution

Land null-byte rejection in `@openclaw/fs-safe`'s archive-entry validator, publish/update OpenClaw to that fixed dependency, and keep focused OpenClaw regression coverage for null-byte archive entries.

## Reproduction Assessment

Yes. Current main re-exports archive validation from `@openclaw/fs-safe@0.2.2`, and a read-only Node probe against that package accepted both null-byte path examples.

## Solution Assessment

No. The guard is conceptually right, but this PR applies it to the pre-fs-safe local helper; the maintainable fix belongs in the fs-safe archive-entry implementation followed by an OpenClaw dependency update.
