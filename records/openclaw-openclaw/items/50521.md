## Summary

Keep open: the hardening is still useful, but the branch is no longer the right patch because current main delegates archive validation and extraction to `@openclaw/fs-safe`, and the pinned package still lacks null-byte rejection. Contributor real behavior proof is also absent.

## What This Changes

The PR adds a null-byte guard to the former local archive entry path validator and adds two regression tests for null-byte archive entry names.

## Best Possible Solution

Land null-byte rejection in `@openclaw/fs-safe`'s archive-entry validator, publish or consume the fixed package in OpenClaw, and keep focused OpenClaw regression coverage for archive entries containing null bytes.

## Reproduction Assessment

Yes. Source inspection shows current main re-exports archive validation/extraction from `@openclaw/fs-safe@0.2.3`, and that package validates traversal/absolute/drive paths without a null-byte guard.

## Solution Assessment

No. The proposed guard is conceptually right, but it belongs in the dependency-owned `@openclaw/fs-safe` archive validator now used by the runtime path, followed by an OpenClaw dependency update.
