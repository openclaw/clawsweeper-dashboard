## Summary

Keep open. The null-byte rejection is a useful security hardening direction, but current main moved the live archive validator and extractor into `@openclaw/fs-safe@0.2.4`, so this PR edits an obsolete local helper shape and would not fix runtime archive behavior after rebase; contributor real behavior proof is also still missing.

## What This Changes

The branch adds a null-byte rejection guard to the old local `src/infra/archive-path.ts` validator and adds two archive-path regression tests.

## Best Possible Solution

Land null-byte rejection in `@openclaw/fs-safe` archive-entry validation, publish or consume the fixed package in OpenClaw, and keep focused regression coverage for validator plus ZIP/TAR preflight behavior.

## Reproduction Assessment

Yes, source-level: current main re-exports archive validation and extraction from pinned `@openclaw/fs-safe@0.2.4`, whose published validator lacks a `\0` guard while ZIP/TAR preflight routes entry paths through it. I did not run a runtime extraction probe in this read-only review.

## Solution Assessment

No. The proposed guard is conceptually right, but the maintainable fix belongs in the `@openclaw/fs-safe` archive validator plus an OpenClaw dependency update, not in the obsolete local helper body.
