## Summary

Keep open: the null-byte rejection is valid security hardening, but this branch edits the pre-fs-safe local validator while current main delegates archive validation and extraction to `@openclaw/fs-safe@0.2.2`, whose published archive-entry validator still lacks the guard. The PR also still lacks after-fix real behavior proof.

## What This Changes

The PR adds explicit null-byte rejection to archive entry path validation plus two regression test cases for null-byte archive entry names.

## Best Possible Solution

Land null-byte rejection in `@openclaw/fs-safe`'s archive-entry validator, publish/update OpenClaw to that fixed dependency, and keep focused OpenClaw regression coverage for null-byte archive entries.

## Reproduction Assessment

Yes, from source inspection. Current main consumes `@openclaw/fs-safe@0.2.2`; its archive validator has no null-byte guard, and both zip and tar extraction paths call that validator before resolving archive output paths.

## Solution Assessment

No. The guard is conceptually right, but placing it in the old local helper is no longer the maintainable fix because current main delegates the runtime path to `@openclaw/fs-safe/archive`.
