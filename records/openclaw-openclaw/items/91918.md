## Label Justifications

- `P1`: The stable-release regression breaks active safety and observability workflows during normal agent tool execution.
- `impact:security`: A configured before_tool_call approval gate can silently stop enforcing its tool-execution boundary.

## Summary

Current main still preserves a gateway-bound global hook runner when a later default-mode plugin registry activates, and the focused loader test explicitly confirms hooks unique to that later registry remain invisible. This matches the reported silent tool-hook failure, so the security-sensitive regression remains necessary to fix.

## What This Changes

Restore before_tool_call and after_tool_call execution for enabled local extensions after upgrading to v2026.6.5.

## Best Possible Solution

Use one canonical runtime-visible hook registry, or explicitly compose the gateway and run-scoped registries, so tool dispatch sees local extension hooks without discarding gateway lifecycle hooks; add regression coverage for blocking before_tool_call and observational after_tool_call handlers.

## Reproduction Assessment

Yes, at source level with high confidence: current tests explicitly demonstrate that hooks unique to a later default-mode registry are absent from the preserved global runner used by tool dispatch. A packaged live reproduction was not run in this read-only review.

## Solution Assessment

Yes, repairing registry and runner lifecycle ownership is the narrowest maintainable solution. Additional activation flags, permissions, configuration, or tool-specific fallback dispatch would preserve the split source of truth rather than fix it.
