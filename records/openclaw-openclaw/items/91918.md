## Label Justifications

- `P1`: This is a released regression that silently breaks safety and observability hooks during normal agent tool execution.
- `impact:security`: A configured local `before_tool_call` approval gate can stop enforcing the tool-execution boundary while appearing registered.

## Summary

This issue is still necessary: current main and the latest release still preserve a gateway-bound hook runner across later default-mode plugin activation, while tool dispatch consults that single runner. The open fix PR is linked with closing syntax and appears to own the implementation path, so this issue should stay open until that PR or an equivalent fix merges.

## What This Changes

Restore `before_tool_call` and `after_tool_call` firing for enabled local global-dir plugins after the 2026.5.27 to 2026.6.5 regression.

## Best Possible Solution

Main should expose one lifecycle-owned runtime-visible hook view, or an equivalent composed registry view, so enabled local plugin tool hooks fire after scoped plugin activations; close this issue only after the linked fix PR or equivalent lands.

## Reproduction Assessment

Yes. Source inspection shows the gateway runner can remain bound to the earlier registry while tool dispatch checks only that runner, and the reporter supplied a concrete released-version rollback comparison.

## Solution Assessment

Yes. Repairing the plugin registry and global hook-runner lifecycle is the narrow maintainable direction; adding more activation, timeout, or docs-only requirements would not fix the split source of truth.
