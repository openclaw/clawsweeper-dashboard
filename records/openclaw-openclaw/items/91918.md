## Label Justifications

- `P1`: A released regression silently disables safety and observability hooks in normal agent and channel-originated tool workflows.
- `impact:security`: A registered `before_tool_call` approval or blocking plugin can be bypassed while the dangerous tool call still executes.

## Summary

The regression remains present on current main: tool-call dispatch consults the singleton global hook runner, while the loader can activate a different default registry and intentionally preserve the earlier gateway runner. That lifecycle split matches the report, is present in v2026.6.5, and has no linked fix PR, so this issue is still necessary.

## What This Changes

Restore `before_tool_call` and `after_tool_call` execution for enabled local extensions after the 2026.6.5 plugin-runtime lifecycle regression.

## Best Possible Solution

Unify active-registry and global-hook-runner lifecycle ownership so every activated canonical registry updates the runner without discarding gateway-bindable capabilities, and add regression coverage for both before/after tool hooks after gateway-to-default runtime activation.

## Reproduction Assessment

Yes at source level with high confidence: gateway-runner preservation plus active-registry replacement leaves tool dispatch consulting a runner bound to a different registry. The reporter also provides a concrete 2026.5.27-versus-2026.6.5 runtime procedure, though this review did not execute it.

## Solution Assessment

Yes. A focused lifecycle repair is the best path; adding activation flags, timeout configuration, or runtime compatibility shims would not fix the split source of truth and would weaken the existing plugin hook contract.
