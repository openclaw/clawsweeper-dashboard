## Label Justifications

- `P1`: The stable-release regression breaks active safety and observability workflows during normal agent tool execution.
- `impact:security`: A configured `before_tool_call` approval gate can silently stop enforcing its tool-execution boundary.

## Summary

Current main still has the registry/runner split described by the report: a later default-mode plugin registry can become active while tool dispatch continues consulting the preserved gateway-bound global hook runner, leaving hooks unique to the later registry invisible. No merged or viable open PR currently owns this security-sensitive regression, so the issue remains necessary.

## What This Changes

Restore `before_tool_call` and `after_tool_call` execution for enabled local extensions after the OpenClaw 2026.6.5 upgrade regression.

## Best Possible Solution

Use one canonical runtime-visible hook registry, or explicitly compose compatible gateway and run-scoped registries, so tool dispatch sees local extension hooks without discarding gateway lifecycle hooks; cover blocking `before_tool_call` and observational `after_tool_call` behavior.

## Reproduction Assessment

Yes, at source level with high confidence: activate a gateway-bindable registry and then a distinct default registry with a tool hook; current main preserves the first runner while tool execution consults only that runner. The reporter also provides a concrete 2026.6.5 versus 2026.5.27 rollback comparison.

## Solution Assessment

No, requiring more activation, permission, or timeout configuration would not solve the split source of truth. Repairing registry and runner lifecycle ownership is the narrowest maintainable solution.
