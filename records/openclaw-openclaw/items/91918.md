## Label Justifications

- `P1`: This is a shipped regression that silently breaks active agent safety and observability workflows for affected local extensions.
- `impact:security`: A registered `before_tool_call` approval or blocking hook can fail open, allowing a tool call that the plugin intended to deny.

## Summary

Current main still contains a source-level lifecycle gap consistent with the report: a later default plugin runtime can preserve the gateway-initialized hook runner instead of rebuilding it from the registry where local extension hooks registered, while agent tools consult only that preserved runner. No merged or open replacement PR currently owns the regression, so this issue remains necessary.

## What This Changes

Restore `before_tool_call` and `after_tool_call` execution for enabled local global-directory extensions after upgrading to OpenClaw 2026.6.5.

## Best Possible Solution

Keep one canonical lifecycle-owned hook runner whose registry includes hooks activated at gateway startup and during later runtime plugin materialization, with integration coverage proving local extensions receive both before- and after-tool-call events without new configuration.

## Reproduction Assessment

Yes, at high-confidence source level. Load a gateway-bindable plugin registry, then activate the default runtime registry containing a local extension hook: the loader preserves the first global runner while tool execution consults only that runner, matching the reported non-firing behavior.

## Solution Assessment

Yes, the narrowest maintainable direction is to repair canonical hook-runner/registry synchronization. Requiring activation metadata, permissions, or new configuration would mask the lifecycle regression and break the already-supported local plugin contract.
