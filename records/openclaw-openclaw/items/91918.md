## Summary

Keep open: this is a credible latest-release regression in the documented plugin tool-hook surface, and current main still has dispatch paths that skip `before_tool_call` and `after_tool_call` when the runner visible at tool-dispatch time does not contain those hooks.

## What This Changes

The issue reports that enabled local plugins register `before_tool_call` and `after_tool_call` handlers after upgrading to 2026.6.5, but those handlers no longer fire during agent tool execution.

## Best Possible Solution

Repair hook-runner ownership or activation so enabled local plugins' tool-hook registrations are visible to the same runner used by agent tool dispatch, then add regression coverage for both `before_tool_call` and `after_tool_call` from a local plugin.

## Reproduction Assessment

No live high-confidence current-main reproduction was run in this read-only sweep. The reporter supplied a concrete release rollback repro, and source inspection shows the dispatch gates that would skip both hooks when the active runner lacks the registered handlers.

## Solution Assessment

Yes, the expected direction is to restore the documented hook runtime contract with focused regression coverage. A new activation or permission requirement would be a product/docs change and is not the best default fix unless maintainers intentionally change the contract.
