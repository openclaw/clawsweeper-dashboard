## Summary

Keep open: this is a credible latest-release regression in a documented plugin safety hook surface, and current main does not provide enough source/history evidence to close it as fixed or duplicate.

## What This Changes

Local global-directory plugins that register before_tool_call and after_tool_call are reported to load successfully but not receive tool-call events after upgrading to 2026.6.5.

## Best Possible Solution

Fix the runtime registry/global hook runner path so enabled local plugins loaded for gateway startup are the same hooks visible to before_tool_call and after_tool_call dispatch, then add regression coverage for both hooks from a local plugin.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. Source inspection matches the reported failure mode: tool calls skip hook execution whenever the global runner visible at dispatch time does not report the registered hook.

## Solution Assessment

Yes, the best fix is a narrow runtime/plugin-loader repair with regression tests, not a new config requirement. If maintainers intentionally require extra activation policy, that should come with doctor/docs warnings, but the current docs support the reporter's expectation.
