## Summary

Keep this open: current main and the latest release still contain the single/global hook-runner visibility split, while the proof-positive fix remains an open unmerged PR. The issue is security-relevant because a registered local `before_tool_call` gate can be skipped before native tool execution.

## What This Changes

The issue reports a v2026.6.5 regression where enabled local extension `before_tool_call` and `after_tool_call` hooks register but do not fire during agent runs after scoped plugin registry activation.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/92016 or an equivalent core plugin-runtime fix that makes tool-call dispatch see the canonical live hook registry without requiring new activation or timeout configuration.

## Reproduction Assessment

Yes, at source level with high confidence. Current main preserves a gateway-bound global runner across a later default activation, and tool execution asks only that runner whether `before_tool_call` hooks exist.

## Solution Assessment

Yes, the best path is to fix hook-registry lifecycle ownership rather than add activation, timeout, or permission configuration. The open linked PR targets that owner boundary, but this issue should remain open until that work is merged or replaced.
