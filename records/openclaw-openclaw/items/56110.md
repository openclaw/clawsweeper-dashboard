## Summary

Keep open. Current main still does not recognize `STATE.md` in the bootstrap filename contract, runtime loader, setup/API surfaces, docs, or the bundled extra-file hook; related #39120 is broader open design work rather than a shipped implementation. The KinthAI comment adds concrete deployed usage, but adding a new always-loaded workspace file changes prompt budget and workspace contract, so this needs maintainer product judgment.

## What This Changes

Request to add `STATE.md` as a first-class workspace bootstrap file for ephemeral agent working state after context compaction.

## Best Possible Solution

Decide the core workspace-bootstrap direction first. If maintainers want `STATE.md` as a product primitive, add it consistently to the workspace filename contract, runtime loading, optional setup/template or file-management surfaces, docs, and focused regression tests; if the desired direction is configurable context files, resolve it through #39120 and document the `STATE.md` configuration path.

## Reproduction Assessment

Yes for the missing requested behavior. A source-level reproduction is clear: a workspace `STATE.md` is absent from the fixed loader entries and rejected by the extra-file basename allowlist, so it will not be injected on current main.

## Solution Assessment

Unclear pending maintainer product judgment. A one-line hardcoded filename addition is not necessarily the best fix until maintainers choose between first-class `STATE.md`, a broader configurable context-file mechanism, or no new default bootstrap surface.
