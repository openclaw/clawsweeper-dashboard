## Summary

This PR should stay open: current main does not implement autoscrub, but the proposed success path still leaves dependency-change label/comment state behind unless a recursive workflow run cleans it, and the diff widens a privileged `pull_request_target` workflow boundary.

## What This Changes

The PR widens the dependency-change awareness workflow to `contents: write` and adds same-repository lockfile autoscrub logic, comments, and script/workflow tests.

## Best Possible Solution

Land this only after autoscrub cleans dependency label/comment state within the same successful run, tests cover that no-recursive-run path, and secops accepts the widened write-token design with the base guard PR.

Yes, source inspection gives a high-confidence reproduction path: a same-repository lockfile-only PR reaches autoscrub, gets label/comment state first, pushes the cleanup commit, and returns before clearing that state. GitHub's `GITHUB_TOKEN` trigger contract means the same run must do the cleanup rather than relying on a recursive workflow run. ([docs.github.com](https://docs.github.com/en/actions/writing-workflows/choosing-when-your-workflow-runs/triggering-a-workflow))

No. Autoscrub is a reasonable direction, but the current implementation needs same-run state cleanup plus secops acceptance of the privileged workflow write path before it is the best maintainable solution.
