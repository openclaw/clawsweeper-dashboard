## Summary

Keep open for maintainer/secops review: the autoscrub direction is useful, but the current guard can auto-remove lockfiles when package.json graph-affecting fields outside dependencies/devDependencies changed, weakening the dependency review boundary.

## What This Changes

The PR adds a pull_request_target autoscrub path that grants the dependency-change workflow contents:write and restores changed package lockfiles from the target branch on same-repository PRs when dependency/devDependency fields are unchanged.

## Best Possible Solution

Keep the autoscrub idea, but enable the write-back path only after autoscrub eligibility covers every graph-affecting manifest field or refuses any changed package.json outside a proven safe set, with regression coverage and secops acceptance of contents:write.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a same-repository PR changing a graph-affecting package.json field such as optionalDependencies, overrides, packageManager, or peerDependencies plus a lockfile leaves dependencyManifestChanges empty and makes autoscrubCandidate true.

## Solution Assessment

No, not as written; automatic lockfile restoration is maintainable only if the manifest gate accounts for all dependency graph fields or refuses changed package.json files outside a proven safe set.

- **[P1] Block autoscrub for all graph-changing manifests:** `scripts/github/dependency-change-awareness.mjs:50`
