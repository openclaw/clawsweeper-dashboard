## Summary

Keep open. Current main and the latest shipped release still hardcode the lane-wait warning fallback at 2,000 ms, diagnostics config/schema/docs do not expose a lane-wait threshold, the related implementation PR closed unmerged, and the broader concurrency tracker does not replace this narrow logging-control request.

## What This Changes

The issue requests a configurable lane-wait diagnostic warning threshold so long-running cron lanes do not flood logs with hardcoded 2,000 ms warnings.

## Best Possible Solution

Add a documented positive-integer `diagnostics.laneWaitWarnMs` default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane/adaptive/session-concurrency work to https://github.com/openclaw/openclaw/issues/53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without `opts.warnAfterMs`; `enqueueCommandInLane` stores the 2,000 ms fallback and no diagnostics config key can override it.

## Solution Assessment

Yes for the narrow logging-control slice. A single global `diagnostics.laneWaitWarnMs` key is the smallest maintainable path here; per-lane thresholds, adaptive behavior, and session concurrency should remain separate.
