## Summary

Keep open. Current main and the latest release still lack a configurable lane wait warning threshold, and the open linked PR is the active implementation candidate rather than merged evidence for closing this issue.

## What This Changes

Request to add a configurable `diagnostics.laneWaitWarnMs` threshold so long-running queued lane work does not flood logs with fixed 2-second warnings.

## Best Possible Solution

Land or refresh the focused linked implementation so `diagnostics.laneWaitWarnMs` defaults to 2,000 ms when unset, preserves explicit per-call overrides, and updates schema, types, help, docs, tests, and reload behavior together.

## Reproduction Assessment

Yes, source-reproducible. Current main still stores a hardcoded 2,000 ms default for queue entries without an explicit override, and the config schema/type/docs expose no matching lane wait threshold key.

## Solution Assessment

Yes in direction, but not resolved yet. A narrow `diagnostics.laneWaitWarnMs` setting is the maintainable fix if maintainers accept the new public config surface and the linked PR resolves its reload and compatibility details.
