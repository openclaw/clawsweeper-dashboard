## Summary

Keep open. Current main still hardcodes the lane wait warning fallback at 2,000 ms, has no `diagnostics.laneWaitWarnMs` schema/type/help/docs surface, and the related implementation attempts are closed unmerged while the broader concurrency issue does not replace this focused logging-control request.

## What This Changes

Request a configurable lane wait diagnostic threshold so long-running cron or lane work does not flood logs with fixed 2-second warnings.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit `warnAfterMs` overrides, cover config reload behavior, and leave per-lane/adaptive timeout and session concurrency changes to separate work.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane entry without `opts.warnAfterMs`; current main stores the 2,000 ms fallback and the proposed config key is absent from schema, types, help, and docs.

## Solution Assessment

Yes, the narrow `diagnostics.laneWaitWarnMs` setting is the best scoped solution for this log-noise issue. Per-lane thresholds, adaptive timeouts, session concurrency, and auto-yield should stay separate from this focused diagnostics control.
