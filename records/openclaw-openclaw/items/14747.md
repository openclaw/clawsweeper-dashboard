## Summary

Keep open. Current main and v2026.6.10 still hardcode the lane wait diagnostic threshold and do not expose `diagnostics.laneWaitWarnMs`; https://github.com/openclaw/openclaw/pull/86676 is the open closing implementation path but is still unmerged and conflicting.

## What This Changes

Request to add a configurable lane wait diagnostic warning threshold so long-running cron or queued work does not flood logs with non-actionable 2-second warnings.

## Best Possible Solution

Land the linked PR or an equivalent narrow implementation that adds documented `diagnostics.laneWaitWarnMs`, defaults to the current 2,000 ms behavior, preserves explicit per-entry `warnAfterMs` overrides, and proves runtime config update behavior.

## Reproduction Assessment

Yes, as a source-level reproduction rather than a live run: current main still stores `opts?.warnAfterMs ?? 2_000` for queue entries and the diagnostics schema/docs have no `laneWaitWarnMs` field.

## Solution Assessment

Yes, the narrow `diagnostics.laneWaitWarnMs` path is the best scoped solution for this issue; per-lane/dynamic timeout work belongs to the broader concurrency discussion rather than this logging-control request.
