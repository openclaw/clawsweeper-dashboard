## Summary

Keep open. Current main still hardcodes the omitted lane wait warning threshold at 2,000 ms and exposes no `diagnostics.laneWaitWarnMs` config/schema/docs surface; the related implementation PR is closed unmerged, while the broader concurrency issue does not supersede this narrower logging-control request.

## What This Changes

Request to add a configurable lane wait diagnostic warning threshold so long-running cron jobs can avoid repeated hardcoded 2-second warnings without disabling diagnostics.

## Best Possible Solution

Add a documented positive millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit `warnAfterMs` overrides, update config schema/types/help/generated docs/tests/changelog, and leave per-lane/adaptive behavior to separate work.

## Reproduction Assessment

Yes, at source level. Enqueue a second task behind an active lane entry without `opts.warnAfterMs`; current main stores a 2,000 ms threshold and later emits `lane wait exceeded`, while the proposed config key is absent from schema, types, metadata, and docs.

## Solution Assessment

Yes. A narrow diagnostics threshold setting is the maintainable solution for this log-noise request; broader concurrency, auto-yield, per-lane, or adaptive timeout behavior belongs in separate work such as #53023.
