## Summary

Keep open. Current main and the latest release still hard-code the lane wait diagnostic default and do not expose `diagnostics.laneWaitWarnMs`; https://github.com/openclaw/openclaw/pull/86676 is the active closing implementation path but remains open and conflicting.

## What This Changes

Request a configurable lane-wait diagnostic warning threshold so long-running cron or background lanes can reduce non-actionable `lane wait exceeded` log noise without disabling diagnostics.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/86676 or an equivalent focused fix that adds documented `diagnostics.laneWaitWarnMs`, preserves the 2,000 ms default and explicit per-entry overrides, and aligns schema, types, help/reference docs, tests, and reload semantics.

## Reproduction Assessment

Yes, at source level: current main still stores `opts?.warnAfterMs ?? 2_000` for queue entries and the diagnostics schema/docs have no `laneWaitWarnMs` field. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes, the narrow `diagnostics.laneWaitWarnMs` field is the best scoped direction for this logging-control request. Per-lane scheduling, adaptive timeouts, and session concurrency belong to broader concurrency work rather than this issue.
