Keep open. Current main and the latest release still lack a `diagnostics.laneWaitWarnMs` config key, and the open linked PR is the active implementation candidate rather than merged evidence for closing this issue.

## What This Changes

Request to make the lane wait diagnostic warning threshold configurable so long-running cron or background work does not flood operator logs with expected waits.

## Best Possible Solution

Land a focused implementation through the linked PR or its refreshed replacement, preserving the 2,000 ms default and explicit per-call overrides while aligning schema, types, help, docs, and reload behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows queue entries without an explicit override still use the hardcoded 2,000 ms threshold, and the diagnostics config schema/type/docs expose no matching key.

## Solution Assessment

Yes in direction, but not resolved yet. A narrow `diagnostics.laneWaitWarnMs` setting is the maintainable fix for this logging-control request if maintainers accept the new config surface and the linked PR resolves its contract/reload gaps.
