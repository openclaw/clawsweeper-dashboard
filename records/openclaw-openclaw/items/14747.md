## Summary

Keep open. Current main still hardcodes omitted lane wait warning thresholds to 2,000 ms, and the diagnostics config/schema/docs surfaces do not expose a lane wait threshold setting.

## What This Changes

Requests a diagnostics config knob to raise the lane wait warning threshold so legitimate long-running cron jobs do not flood gateway logs.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit `warnAfterMs` overrides, update generated config surfaces/docs/tests/changelog, and leave concurrency or auto-yield behavior to #53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second task behind an active lane entry without `opts.warnAfterMs`; current main stores a 2,000 ms threshold and later emits `lane wait exceeded`, while `diagnostics.laneWaitWarnMs` is unsupported by schema, types, generated metadata, and docs.

## Solution Assessment

Yes. A narrow global diagnostics threshold is the maintainable fix for this log-noise issue; per-lane thresholds, adaptive timeout logic, session concurrency, and auto-yield should stay separate from this repair.

Fix openclaw/openclaw#14747 with a narrow core change. Add a positive millisecond `diagnostics.laneWaitWarnMs` config field that defaults to the current 2,000 ms behavior when unset and is used only when `enqueueCommandInLane` callers omit `opts.warnAfterMs`. Preserve explicit overrides such as cron manual runs with `warnAfterMs: 5_000`. Handle config reload semantics so changed values apply to future queue entries without requiring a full restart, and document that already queued entries keep their stored threshold if that remains the chosen behavior. Update zod schema, exported config types, labels/help, generated base config schema, configuration reference, queue docs, focused tests, and CHANGELOG.md. Use #16366 only as context; do not copy unrelated bundled changes. Do not implement per-lane thresholds, session concurrency, auto-yield, or broader #53023 behavior in this PR.
