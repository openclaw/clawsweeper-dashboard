## Summary

Keep open. Current main and the latest release still hardcode the default lane-wait warning threshold, and https://github.com/openclaw/openclaw/pull/86676 is an open, unmerged fix candidate rather than evidence that the requested behavior has landed.

## What This Changes

This issue requests a configurable lane-wait diagnostic warning threshold, such as `diagnostics.laneWaitWarnMs`, instead of the hardcoded default.

## Best Possible Solution

Land a refreshed focused PR that preserves the 2,000 ms default and explicit per-call `warnAfterMs` overrides while aligning schema, types, help/reference docs, generated docs, and reload semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows queue entries without an explicit override still use the hardcoded 2,000 ms threshold, and the diagnostics schema/types/docs expose no matching lane-wait threshold key.

## Solution Assessment

Yes in direction, but it is not resolved yet. A narrow `diagnostics.laneWaitWarnMs` setting is the maintainable path if maintainers accept the public config surface and the linked PR aligns the full config contract.
