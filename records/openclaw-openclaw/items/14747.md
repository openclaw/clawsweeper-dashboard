## Summary

Keep open. Current main and the latest release still default lane-wait diagnostics to a hardcoded 2,000 ms when callers do not pass `warnAfterMs`, the diagnostics schema/docs do not expose `diagnostics.laneWaitWarnMs`, and the live linked implementation PR is open but conflicting rather than merged.

## What This Changes

The issue requests a configurable lane-wait diagnostic warning threshold so long-running lanes can avoid repeated non-actionable `lane wait exceeded` log noise.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/86676 with a narrow `diagnostics.laneWaitWarnMs` schema/docs/tests implementation, while leaving broader session concurrency and auto-yield work to https://github.com/openclaw/openclaw/issues/53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without `opts.warnAfterMs`; current main stores the `2_000` fallback and the diagnostics schema/docs expose no override key.

## Solution Assessment

Yes for the narrow logging-control slice. A documented positive-integer `diagnostics.laneWaitWarnMs` default is the smallest maintainable core change; per-lane thresholds, adaptive behavior, and session concurrency should remain separate.
