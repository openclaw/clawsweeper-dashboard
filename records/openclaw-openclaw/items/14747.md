## Summary

Keep open. Current main still stores the lane wait warning threshold per queue entry and defaults omitted thresholds to a hardcoded 2,000 ms, while diagnostics config, generated schema metadata, and docs only expose stuck-session warning tuning. Related PR #16366 is closed unmerged, and #53023 is broader concurrency/auto-yield work rather than a replacement for this focused logging-control request.

## What This Changes

Request a configurable diagnostics threshold for lane wait warnings so long-running cron or background work can avoid repeated non-actionable `lane wait exceeded` log entries.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` field that defaults to the existing 2,000 ms behavior, supplies the default for future queue entries when `opts.warnAfterMs` is omitted, preserves explicit per-call overrides, handles config reload without a gateway restart, updates generated config surfaces and docs, and leaves per-lane/adaptive timeout or session concurrency work to separate follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enqueue work behind an active lane entry without `opts.warnAfterMs`, wait at least 2,000 ms, and current main emits `lane wait exceeded`; adding `diagnostics.laneWaitWarnMs` is also unsupported by schema, types, generated metadata, and docs.

## Solution Assessment

Yes. A narrow global diagnostics threshold is the best maintainable fix for this issue; per-lane thresholds, adaptive timeout logic, session concurrency, and auto-yield are broader product work and are not required to stop the reported log flood.
