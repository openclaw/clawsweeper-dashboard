## Summary

Keep open. Current main still hardcodes the default lane wait warning threshold at 2,000 ms, exposes no `diagnostics.laneWaitWarnMs` config/schema/docs surface, and the related implementation PRs are closed unmerged while the broader concurrency tracker does not supersede this focused logging-control request.

## What This Changes

Request a configurable lane wait warning threshold so long-running cron jobs can avoid repetitive diagnostic warnings without disabling diagnostics.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit per-call overrides, and keep per-lane, adaptive, concurrency, and auto-yield behavior separate.

## Reproduction Assessment

Yes, at source level. Enqueue a second task behind an active lane entry without `opts.warnAfterMs`; current main stores a 2,000 ms threshold and later emits `lane wait exceeded`, while the proposed config key is absent from schema, types, metadata, and docs.

## Solution Assessment

Yes. A narrow diagnostics threshold setting is the maintainable solution for this log-noise request; broader per-lane thresholds, session concurrency, and auto-yield should remain separate work.
