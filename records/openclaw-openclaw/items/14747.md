## Summary

Keep open. Current `main` still stores a fixed lane-wait warning threshold on each queued entry, the diagnostics config/schema/docs expose only stuck-session tuning, and the focused implementation attempt is closed unmerged while the broader session-concurrency tracker does not supersede this narrower logging-control request.

## What This Changes

The issue requests a configurable lane-wait diagnostic threshold so long-running cron or lane work does not repeatedly emit fixed-threshold warning logs.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane/adaptive/session-concurrency work to https://github.com/openclaw/openclaw/issues/53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without `opts.warnAfterMs`; `enqueueCommandInLane` stores the 2,000 ms fallback and no current diagnostics config key can override it.

## Solution Assessment

Yes for the narrow logging-control slice. A single global diagnostics key is the smallest maintainable fix here; per-lane thresholds, adaptive behavior, and session concurrency should remain separate.
