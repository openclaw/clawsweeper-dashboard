## Summary

Keep open. Current main and the latest tagged release still use a hardcoded queue-entry lane-wait warning default, the diagnostics config/schema/docs do not expose a lane wait threshold, and the only focused implementation PR is closed unmerged while the broader concurrency issue does not supersede this narrower request.

## What This Changes

The issue requests a configurable lane wait diagnostic threshold so long-running cron or queued work can avoid repeated non-actionable warnings.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, keep the existing 2,000 ms behavior when unset, preserve explicit per-call overrides, cover config reload semantics, and leave per-lane/adaptive timeout and concurrency design to separate trackers.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane entry without `opts.warnAfterMs`; current main stores the 2,000 ms fallback and no config key can change that default.

## Solution Assessment

Yes. A narrow `diagnostics.laneWaitWarnMs` setting is the best scoped solution for this log-noise request; per-lane thresholds, adaptive behavior, session concurrency, and auto-yield should remain separate.
