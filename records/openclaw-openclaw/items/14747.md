## Summary

Keep open. Current main still stores a hardcoded 2,000 ms lane-wait warning fallback, diagnostics config/docs do not expose a lane-wait threshold, the direct implementation attempts are closed unmerged, and the broader session-concurrency issue does not supersede this narrower logging-control request.

## What This Changes

Request to make the lane wait diagnostic warning threshold configurable so long-running queued cron work does not flood gateway logs.

## Best Possible Solution

Add a documented positive-millisecond diagnostics.laneWaitWarnMs default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane/adaptive/session-concurrency work to the broader tracker.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without opts.warnAfterMs; enqueueCommandInLane stores the 2,000 ms fallback and no current diagnostics config key can override it.

## Solution Assessment

Yes for the narrow logging-control slice. A single global diagnostics.laneWaitWarnMs key is the smallest maintainable fix here; per-lane thresholds, adaptive behavior, and session concurrency should stay separate.
