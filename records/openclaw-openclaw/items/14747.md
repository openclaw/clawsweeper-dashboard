## Summary

Keep open. Current main and the latest release still store a 2,000 ms default for queued lane wait diagnostics, diagnostics config/schema/docs expose no lane-wait threshold key, and the related implementation PR closed unmerged while the broader concurrency tracker remains separate.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold so long-running cron or agent work can avoid noisy non-actionable gateway warnings.

## Best Possible Solution

Add a documented positive-integer diagnostics.laneWaitWarnMs default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane/adaptive/session-concurrency work to the broader tracker.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without opts.warnAfterMs; enqueueCommandInLane stores the 2,000 ms fallback and the diagnostics config/docs expose no override key.

## Solution Assessment

Yes for the narrow logging-control slice. A single global diagnostics.laneWaitWarnMs key is the smallest maintainable path here; per-lane thresholds, adaptive behavior, and session concurrency should remain separate.
