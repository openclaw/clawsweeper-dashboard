## Summary

Keep open. Current main still hardcodes the lane wait warning default at 2,000 ms, exposes no `diagnostics.laneWaitWarnMs` config/schema/help/docs surface, and the directly related implementation PR is closed unmerged while the broader concurrency umbrella does not replace this focused logging-control request.

## What This Changes

Request to add a configurable diagnostics lane wait warning threshold so long-running cron jobs do not flood logs.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit per-call overrides, cover config reload behavior, and leave per-lane/adaptive/concurrency work to separate trackers.

## Reproduction Assessment

Yes, at source level. Enqueue a second task behind an active lane entry without `opts.warnAfterMs`; current main stores the 2,000 ms fallback and later emits `lane wait exceeded`, while the proposed config key is absent from schema, types, help, and docs.

## Solution Assessment

Yes. A narrow diagnostics threshold setting is the maintainable fix for this log-noise request; per-lane thresholds, adaptive timeouts, session concurrency, and auto-yield should stay separate.
