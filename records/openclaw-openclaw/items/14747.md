## Summary

Keep open. Current main still hardcodes the queued-entry lane wait warning default, the diagnostics config and docs do not expose a lane-wait threshold, and the closed implementation attempts plus broader concurrency tracker do not supersede this narrower logging-control request.

## What This Changes

The issue requests a configurable diagnostics threshold for lane wait warnings so long-running cron-style work does not flood logs.

## Best Possible Solution

Add a documented positive-millisecond diagnostics.laneWaitWarnMs default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane, adaptive, session-concurrency, and auto-yield work to https://github.com/openclaw/openclaw/issues/53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without opts.warnAfterMs; enqueueCommandInLane stores the 2,000 ms fallback and no current diagnostics config key can override it.

## Solution Assessment

Yes for the narrow logging-control slice. A single global diagnostics key is the smallest maintainable fix here; per-lane thresholds, adaptive behavior, and session concurrency should remain separate.
