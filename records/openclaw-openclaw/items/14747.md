## Summary

Keep open. Current main still defaults omitted lane wait warning thresholds to a hardcoded 2,000 ms, and the diagnostics schema, generated metadata, and public docs do not expose a configurable lane wait warning setting. Related PR #16366 is closed unmerged, while #53023 is broader concurrency and auto-yield work rather than a replacement for this focused diagnostics-control request.

## What This Changes

The issue requests a configurable threshold for lane wait diagnostic warnings so long-running cron jobs do not flood gateway logs.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` field that defaults to the existing 2,000 ms behavior, supplies the default for future queue entries when `opts.warnAfterMs` is omitted, preserves explicit per-call overrides, handles config reload without a gateway restart, and updates generated config surfaces, public docs, tests, and changelog.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enqueue work behind an active lane entry without `opts.warnAfterMs`, wait at least 2,000 ms, and current main emits `lane wait exceeded`; adding `diagnostics.laneWaitWarnMs` is also unsupported by schema, types, generated metadata, and docs.

## Solution Assessment

Yes. A narrow global diagnostics threshold is the best maintainable fix for this issue; per-lane thresholds, adaptive timeout logic, session concurrency, and auto-yield belong in separate follow-ups.
