## Summary

Keep open. Current `main` and `v2026.5.19` still use the hardcoded 2,000 ms queue-entry fallback, the diagnostics schema/types/docs still expose no lane-wait threshold, the focused implementation attempt at https://github.com/openclaw/openclaw/pull/16366 closed unmerged, and the broader concurrency tracker at https://github.com/openclaw/openclaw/issues/53023 does not supersede this narrower logging-control request.

## What This Changes

The issue requests a configurable lane-wait diagnostic warning threshold so legitimate long-running queued work can avoid noisy `lane wait exceeded` log spam.

## Best Possible Solution

Add a documented positive-integer `diagnostics.laneWaitWarnMs` default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane, adaptive, and session-concurrency work to https://github.com/openclaw/openclaw/issues/53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without `opts.warnAfterMs`; `enqueueCommandInLane` stores the 2,000 ms fallback and the diagnostics schema/docs expose no override key.

## Solution Assessment

Yes for the narrow logging-control slice. A single global `diagnostics.laneWaitWarnMs` key is the smallest maintainable path here; per-lane thresholds, adaptive behavior, and session concurrency should remain separate.
