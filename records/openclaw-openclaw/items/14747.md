## Summary

The issue remains necessary: current main and v2026.6.5 still use a fixed lane-wait warning default, while the focused closing PR remains open and conflicting. The canonical next step is to refresh and review that PR, not close the issue or start a duplicate implementation.

## What This Changes

Add an optional configuration setting for the lane-wait diagnostic warning threshold so operators can reduce expected log noise while retaining the existing default.

## Best Possible Solution

Land a refreshed focused implementation of `diagnostics.laneWaitWarnMs` that defaults to 2,000 ms, applies only when no per-entry override is supplied, reads the current runtime configuration for new entries, and includes aligned schema/help/docs and focused reload/override tests.

## Reproduction Assessment

Yes at source level: current main demonstrably assigns the fixed 2,000 ms default and emits the warning from that stored value, while the report includes concrete recurring cron log examples. This read-only review did not execute a live current-main scenario.

## Solution Assessment

Yes, the focused global diagnostics setting is the narrowest maintainable direction; per-lane and adaptive timeout proposals would add unnecessary policy, while explicit caller overrides should remain authoritative.
