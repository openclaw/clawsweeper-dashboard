## Summary

Keep open. Current main still stores a hardcoded 2,000 ms lane wait warning threshold when callers omit `warnAfterMs`, and the diagnostics config/schema/docs surfaces still lack a lane wait threshold setting; the related implementation PR is closed unmerged and the broader concurrency issue does not supersede this narrower logging-control request.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold for long-running queued lanes, especially cron jobs, to reduce non-actionable log noise.

## Best Possible Solution

Add a documented positive millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit `warnAfterMs` overrides, update generated config surfaces/docs/tests/changelog, and leave concurrency, auto-yield, per-lane, or adaptive timeout policy to separate work.

## Reproduction Assessment

Yes, at source level. Enqueue a second task behind an active lane entry without `opts.warnAfterMs`; current main stores a 2,000 ms threshold and later emits `lane wait exceeded`, while `diagnostics.laneWaitWarnMs` is unsupported by schema, types, generated metadata, and docs.

## Solution Assessment

Yes. A narrow diagnostics threshold is the maintainable fix for this log-noise request; broader session concurrency, auto-yield, per-lane thresholds, and adaptive timeout behavior should stay separate from this repair.
