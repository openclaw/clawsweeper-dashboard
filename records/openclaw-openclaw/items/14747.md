## Summary

Keep open. Current main and the latest release still hardcode the default lane wait warning threshold, while https://github.com/openclaw/openclaw/pull/86676 is open, unmerged, and explicitly owns the implementation path.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold so legitimate long-running cron or lane work can avoid repeated non-actionable `lane wait exceeded` warnings without disabling diagnostics.

## Best Possible Solution

Review, refresh, and land https://github.com/openclaw/openclaw/pull/86676 or a successor that preserves the 2,000 ms default, keeps explicit per-call overrides, and aligns schema, types, help, docs, tests, and reload behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows queue entries without an explicit override still use the hardcoded 2,000 ms threshold, and the diagnostics config schema/type/docs expose no matching lane wait threshold key.

## Solution Assessment

Yes in direction, but it is not resolved yet. A narrow `diagnostics.laneWaitWarnMs` setting is the maintainable path if maintainers accept the new public config surface and the linked PR aligns the config contract and reload behavior.
