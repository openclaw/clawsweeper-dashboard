## Summary

Keep open. Current main still has a hardcoded lane-wait warning fallback, no diagnostics config/type/help/docs surface for a lane wait threshold, and the related implementation attempts are closed unmerged while the broader concurrency issue does not own this narrow logging-control request.

## What This Changes

The issue requests a configurable lane wait diagnostic threshold so long-running cron jobs can avoid repeated non-actionable wait warnings.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit `warnAfterMs` overrides, cover reload behavior, and leave broader concurrency work to the separate umbrella issue.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane entry without `opts.warnAfterMs`; current main stores the 2,000 ms fallback and no config key can change that default.

## Solution Assessment

Yes. The narrow `diagnostics.laneWaitWarnMs` setting is the best scoped solution for this log-noise request; per-lane thresholds, adaptive behavior, session concurrency, and auto-yield should stay separate.
