## Summary

Keep open. Current main and the latest release still hardcode the default lane wait warning threshold, while https://github.com/openclaw/openclaw/pull/86676 is an open, conflicting implementation candidate with review blockers rather than a merged fix.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold, such as `diagnostics.laneWaitWarnMs`, so long-running cron or queued work can reduce non-actionable log noise.

## Best Possible Solution

Land or replace the focused `diagnostics.laneWaitWarnMs` implementation with aligned schema, typed config, help/reference docs, reload or restart semantics, tests, and the existing 2,000 ms default preserved.

## Reproduction Assessment

Yes, at source level: current main still stores `opts?.warnAfterMs ?? 2_000` for queue entries, and the diagnostics config/schema/docs do not expose `laneWaitWarnMs`. I did not run a live gateway workload because the missing configurability is directly visible in source and release-tag inspection.

## Solution Assessment

Yes on direction, but not by closing this issue now. The narrow queue/config/docs PR shape is the maintainable path, while https://github.com/openclaw/openclaw/pull/86676 still needs maintainer review, conflict resolution, and config-contract alignment before it can resolve this issue.
