## What This Changes

The issue requests a configurable `diagnostics.laneWaitWarnMs` threshold so long-running lane work does not flood diagnostics with hardcoded 2,000 ms wait warnings.

## Best Possible Solution

Keep this issue open until the focused linked PR is refreshed and merged, or maintainers explicitly reject the new diagnostics config surface and choose another logging-control design.

## Solution Assessment

Yes in direction, with conditions. A narrow global `diagnostics.laneWaitWarnMs` defaulting to 2,000 ms is the maintainable fix for this logging-control request, but the active PR still needs the config contract and reload semantics resolved before the issue can close.
