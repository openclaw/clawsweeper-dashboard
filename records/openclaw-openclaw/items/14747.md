## Summary

## What This Changes

Request to add a configurable diagnostics lane-wait warning threshold so long-running cron jobs can avoid repeated non-actionable `lane wait exceeded` logs.

## Best Possible Solution

Review, refresh, and land https://github.com/openclaw/openclaw/pull/86676 or a successor that preserves the 2,000 ms default, keeps explicit per-call overrides, and aligns schema, types, help, docs, tests, and reload semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows queue entries without an explicit override still use the hardcoded 2,000 ms threshold, and the diagnostics schema/type/docs expose no matching lane-wait threshold key.

## Solution Assessment

Yes in direction, but it is not resolved yet. A narrow `diagnostics.laneWaitWarnMs` setting is the maintainable path if maintainers accept the new public config surface and the linked PR aligns the config contract and reload behavior.
