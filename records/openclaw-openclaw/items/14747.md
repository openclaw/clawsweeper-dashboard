## Summary

Keep open. Current main still hardcodes the omitted lane wait warning threshold at 2,000 ms, while the diagnostics config/schema/generated metadata/docs expose only stuck-session tuning; PR #16366 is closed unmerged and #53023 tracks broader concurrency/auto-yield work rather than this focused logging-control request.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold so legitimate long-running queued cron work does not flood gateway logs.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` field that defaults to 2,000 ms, feeds future queue entries when `opts.warnAfterMs` is omitted, preserves explicit overrides, handles config reload without restart, and leaves per-lane/adaptive timeout work to separate issues.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enqueue work behind an active lane entry without `opts.warnAfterMs`, wait at least 2,000 ms, and current main emits `lane wait exceeded`; adding `diagnostics.laneWaitWarnMs` is unsupported by schema, types, generated metadata, and docs.

## Solution Assessment

Yes. A narrow global diagnostics threshold is the best maintainable fix for this issue; per-lane thresholds, adaptive timeout logic, session concurrency, and auto-yield are broader follow-ups.
