## Summary

Keep open. Current main still hardcodes the lane-wait warning fallback, the diagnostics schema/docs expose stuck-session tuning but no lane-wait threshold, the earlier implementation attempts are closed unmerged, and the broader concurrency tracker does not replace this narrower logging-control request.

## What This Changes

Request to add a configurable lane-wait diagnostic warning threshold for long-running queued work.

## Best Possible Solution

Add a documented positive-millisecond diagnostics.laneWaitWarnMs default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane/adaptive/session-concurrency work to https://github.com/openclaw/openclaw/issues/53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without opts.warnAfterMs; enqueueCommandInLane stores the 2,000 ms fallback and no current diagnostics config key can override it.

## Solution Assessment

Yes for the narrow logging-control slice. A single global diagnostics.laneWaitWarnMs key is the smallest maintainable fix here; per-lane thresholds, adaptive behavior, and session concurrency should stay separate.
