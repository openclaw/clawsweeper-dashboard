## Summary

Keep open. Current main and the latest release still hardcode the default lane wait warning threshold at 2,000 ms, the diagnostics config/docs expose no lane-wait threshold key, and the related implementation/workaround PRs are closed unmerged; the broader session-concurrency issue does not supersede this narrower logging-control request.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold, likely as `diagnostics.laneWaitWarnMs`, to reduce noisy queue wait warnings for legitimate long-running cron jobs.

## Best Possible Solution

Add a documented positive-integer `diagnostics.laneWaitWarnMs` default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane/adaptive/session-concurrency work to https://github.com/openclaw/openclaw/issues/53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without `opts.warnAfterMs`; `enqueueCommandInLane` stores the 2,000 ms fallback and the diagnostics schema/docs expose no override key.

## Solution Assessment

Yes for the narrow logging-control slice. A single global `diagnostics.laneWaitWarnMs` key is the smallest maintainable path here; per-lane thresholds, adaptive behavior, and session concurrency should remain separate.
