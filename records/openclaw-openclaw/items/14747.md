## Label Justifications

- `impact:other`: The fixed threshold can generate hundreds of non-actionable warnings and obscure useful gateway diagnostics.

## Summary

Current main and the latest release still use the hardcoded 2,000 ms lane-wait threshold, while the open linked implementation PR remains the active closing path; this issue should stay open until that PR is landed or closed.

## What This Changes

Add a configurable lane-wait diagnostic warning threshold so operators can reduce non-actionable gateway log noise while retaining the existing default.

## Best Possible Solution

Land a focused, current-main-compatible implementation that adds a positive `diagnostics.laneWaitWarnMs` setting, preserves the 2,000 ms fallback and explicit per-entry overrides, and proves both startup and live config-update behavior.

## Reproduction Assessment

Yes, from source inspection: enqueue two commands in one lane so the second waits more than 2,000 ms without an explicit override; current main will emit the warning and provides no config field to raise that default. The supplied issue logs and linked PR runtime harness independently match this path.

## Solution Assessment

Yes in direction: a diagnostics-scoped positive millisecond setting with the existing default is narrower than per-lane policy or adaptive timeouts. Maintainers still need to approve the new config surface and refresh the linked PR against current main.
