## Summary

Keep open. Current main still stores the lane wait warning threshold per queue entry with a hardcoded 2,000 ms default, the diagnostics config/type/help/docs surfaces still lack a lane wait setting, and the focused implementation PR is closed unmerged while the broader concurrency issue does not supersede this logging-control request.

## What This Changes

Request a configurable lane wait warning threshold so long-running queued cron work does not flood diagnostics with the fixed short-delay warning.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, keep the existing 2,000 ms default when unset, preserve explicit `warnAfterMs` overrides, and leave broader concurrency work to the linked issue.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane entry without `opts.warnAfterMs`; current main stores the 2,000 ms fallback and there is no config key to change that default.

## Solution Assessment

Yes. The narrow `diagnostics.laneWaitWarnMs` setting is the best scoped solution for this log-noise request; per-lane thresholds, adaptive behavior, session concurrency, and auto-yield should stay separate.
