## Summary

Keep open. Current main still hardcodes the default lane wait warning threshold at 2,000 ms and exposes no `diagnostics.laneWaitWarnMs` config/schema/docs surface; the related implementation attempts are closed unmerged, and the broader concurrency issue does not supersede this narrower logging-control request.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold so long-running cron jobs do not flood logs with fixed 2-second wait warnings.

## Best Possible Solution

Add a documented positive millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit `warnAfterMs` overrides, update config schema/types/help/docs/tests/changelog, and leave per-lane, adaptive, concurrency, and auto-yield behavior to separate work.

## Reproduction Assessment

Yes, at source level. Enqueue a second task behind an active lane entry without `opts.warnAfterMs`; current main stores a 2,000 ms threshold and later emits `lane wait exceeded`, while the proposed config key is absent from schema, types, metadata, and docs.

## Solution Assessment

Yes. A narrow diagnostics threshold setting is the maintainable solution for this log-noise request; broader per-lane thresholds, session concurrency, and auto-yield should stay separate from this repair.
