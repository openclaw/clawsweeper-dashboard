## Summary

Keep open. Current main and the latest release still hard-code the queue-entry lane wait warning default and do not expose `diagnostics.laneWaitWarnMs`; the open closing PR is the active implementation path, but it remains unmerged and conflicting.

## What This Changes

The issue requests a configurable threshold for `lane wait exceeded` diagnostics so long-running queued work can avoid noisy hardcoded two-second warnings.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/86676 or a narrow replacement so `diagnostics.laneWaitWarnMs` is documented, schema-backed, default-preserving, and covered by command-queue/config tests before closing this issue.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the missing capability: current main defaults lane wait warnings to `2_000` when callers omit `warnAfterMs`, and the diagnostics schema/docs have no `laneWaitWarnMs` field.

## Solution Assessment

Yes in direction, but not complete here: a narrow `diagnostics.laneWaitWarnMs` setting with the current default preserved is the maintainable core fix, while final acceptance belongs with the linked PR's config-surface review.
