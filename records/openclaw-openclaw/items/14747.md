## Summary

Keep open. Current main still stores a fixed 2,000 ms lane-wait warning fallback for queued commands, exposes no `diagnostics.laneWaitWarnMs` config/schema/type/help/docs surface, and the related implementation PRs are closed unmerged while the broader concurrency issue does not own this narrow log-control request.

## What This Changes

The issue requests a configurable lane-wait diagnostic warning threshold so long-running cron or agent work can avoid repeated non-actionable `lane wait exceeded` logs.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, keep the current 2,000 ms fallback when unset, preserve explicit per-call overrides, and leave per-lane/adaptive timeout behavior to separate design work.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane entry without `opts.warnAfterMs`; current main stores the 2,000 ms fallback and no config key can change that default.

## Solution Assessment

Yes. A narrow `diagnostics.laneWaitWarnMs` setting is the best scoped solution for this log-noise request; per-lane thresholds, adaptive behavior, session concurrency, and auto-yield should stay separate.
