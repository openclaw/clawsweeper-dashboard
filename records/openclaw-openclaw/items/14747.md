## Summary

Keep open. Current main and the latest release still store a fixed 2,000 ms lane-wait warning default on queue entries, diagnostics config/docs do not expose a lane-wait threshold, the earlier implementation PR is closed unmerged, and the broader session-concurrency tracker does not replace this narrower logging-control request.

## What This Changes

The issue requests a configurable diagnostics threshold for lane wait warnings so long-running scheduled work does not flood logs.

## Best Possible Solution

Add a documented positive-millisecond diagnostics.laneWaitWarnMs default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane/adaptive/session-concurrency work to https://github.com/openclaw/openclaw/issues/53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without opts.warnAfterMs; enqueueCommandInLane stores the 2,000 ms fallback and no current diagnostics config key can override it.

## Solution Assessment

Yes for the narrow logging-control slice. A single global diagnostics key is the smallest maintainable fix here; per-lane thresholds, adaptive behavior, and session concurrency should stay separate.
