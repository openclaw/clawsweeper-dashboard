## Summary

Keep open. Current main still stores a fixed lane-wait warning default on queued entries, the diagnostics config/docs surfaces do not expose a lane-wait threshold, and the related implementation attempts are closed unmerged while the recent diagnostic clarification only changed warning text.

## What This Changes

Requests a configurable diagnostics lane-wait warning threshold so long-running cron jobs can avoid fixed short-threshold log noise without disabling diagnostics.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, keep 2,000 ms when unset, preserve explicit overrides, and leave per-lane, adaptive timeout, session concurrency, and auto-yield design to separate trackers.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without `opts.warnAfterMs`; `enqueueCommandInLane` stores the 2,000 ms fallback and no diagnostics config key can override it.

## Solution Assessment

Yes for the narrow logging-control slice. A single global diagnostics key is the smallest maintainable solution here; per-lane thresholds, adaptive behavior, and session concurrency should remain separate.
