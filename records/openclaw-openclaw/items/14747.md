## Summary

Keep open. Current main still hardcodes omitted lane wait warning thresholds to 2,000 ms, and the diagnostics schema, generated metadata, and public docs do not expose a lane wait threshold setting.

## What This Changes

Request adding a configurable lane wait diagnostic warning threshold so legitimate long-running cron or background work does not flood logs with hardcoded short-wait warnings.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit `warnAfterMs` overrides, update generated config surfaces, docs, tests, and changelog, and leave broader concurrency work to #53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second task behind an active lane entry without `opts.warnAfterMs`; after at least 2,000 ms, current main emits `lane wait exceeded`, and `diagnostics.laneWaitWarnMs` is unsupported by schema, types, generated metadata, and docs.

## Solution Assessment

Yes. A narrow global diagnostics threshold is the best maintainable fix for this issue; per-lane thresholds, adaptive timeout logic, session concurrency, and auto-yield should remain separate follow-ups.
