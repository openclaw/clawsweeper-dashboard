## What This Changes

Add a configurable threshold for lane-wait diagnostic warnings while preserving the current 2,000 ms default.

## Best Possible Solution

Refresh and land the focused linked PR, or replace it only if its conflicts are impractical, using a positive `diagnostics.laneWaitWarnMs` value, the unchanged 2,000 ms fallback, preserved explicit per-call overrides, generated config documentation, and runtime-update coverage.

## Reproduction Assessment

Yes at source level: current main stores a 2,000 ms default on each queue entry and emits the warning from that value, while no supported diagnostics setting can change it. The linked PR also supplies built-runtime output demonstrating the configurable behavior, but this review did not execute a current-main live workload.

## Solution Assessment

Yes in direction: one global diagnostics threshold with an unchanged default is narrower and more maintainable than the proposed per-lane or adaptive timeout schemes. The existing linked PR is the right implementation vehicle once rebased and reviewed against current main.
