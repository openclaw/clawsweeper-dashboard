## Summary

Keep open. Current main still stores the lane wait warning threshold per queue entry with a hardcoded 2,000 ms default, the diagnostics config/type/help/docs surfaces still lack a lane wait setting, and the exact implementation PRs are closed unmerged while the broader concurrency issue does not supersede this focused logging-control request.

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold so long-running cron jobs do not flood logs with non-actionable queue wait warnings.

## Best Possible Solution

Add a documented positive-millisecond `diagnostics.laneWaitWarnMs` default for future queue entries, preserve explicit `warnAfterMs` overrides, cover config reload behavior, and leave per-lane/adaptive timeout plus session concurrency or auto-yield work to separate issues.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane entry without `opts.warnAfterMs`; current main stores the 2,000 ms fallback and the proposed config key is absent from schema, types, help, and docs.

## Solution Assessment

Yes. The narrow `diagnostics.laneWaitWarnMs` setting is the best scoped solution for this log-noise request; per-lane thresholds, adaptive behavior, session concurrency, and auto-yield should stay separate.
