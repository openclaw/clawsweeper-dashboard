## Summary

Keep open: current main and the latest release still use the hardcoded lane-wait warning default and expose no `diagnostics.laneWaitWarnMs` config field. The remaining work is already owned by the open linked implementation PR at https://github.com/openclaw/openclaw/pull/86676, so this issue should stay open until that PR lands or maintainers decide to reject/replace the config surface.

## What This Changes

The issue requests a configurable lane-wait diagnostic warning threshold so long-running queued cron work can avoid noisy hardcoded 2-second warnings.

## Best Possible Solution

Refresh and land the focused implementation in https://github.com/openclaw/openclaw/pull/86676, preserving the 2,000 ms default and explicit per-call overrides while aligning schema, types, help, docs, and reload behavior; otherwise explicitly reject or replace that config direction.

## Reproduction Assessment

Yes. Source inspection on current main shows `enqueueCommandInLane` still defaults unset `warnAfterMs` to 2,000 ms, and the strict diagnostics config contract has no `laneWaitWarnMs` field; I did not run a live cron workload.

## Solution Assessment

Yes, if maintainers accept a new config key. A focused global `diagnostics.laneWaitWarnMs` field is narrower than per-lane or adaptive timeout work, and broader concurrency/auto-yield behavior remains separately tracked at https://github.com/openclaw/openclaw/issues/53023.
