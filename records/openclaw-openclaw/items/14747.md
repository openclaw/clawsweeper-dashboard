## Summary

Keep open. Current main and the latest release still store the default lane-wait warning threshold as a hardcoded 2,000 ms queue-entry fallback, diagnostics config/docs expose no lane-wait threshold key, and the linked implementation PR closed unmerged while the broader concurrency issue remains a separate product thread.

## What This Changes

Request to add a configurable diagnostics lane-wait warning threshold so legitimate long-running cron/background jobs can avoid repeated hardcoded two-second warnings without disabling diagnostics.

## Best Possible Solution

Add a documented positive-integer `diagnostics.laneWaitWarnMs` default for future queue entries, keep the 2,000 ms fallback when unset, preserve explicit overrides, and leave per-lane/adaptive/session-concurrency work to https://github.com/openclaw/openclaw/issues/53023.

## Reproduction Assessment

Yes, at source level. Enqueue a second command behind an active lane without `opts.warnAfterMs`; `enqueueCommandInLane` stores the 2,000 ms fallback and the diagnostics config/docs expose no override key.

## Solution Assessment

Yes for the narrow logging-control slice. A single global `diagnostics.laneWaitWarnMs` key is the smallest maintainable path here; per-lane thresholds, adaptive behavior, and session concurrency should remain separate.

## Repair Work Prompt

Implement a narrow fix for https://github.com/openclaw/openclaw/issues/14747. Add a documented positive-integer `diagnostics.laneWaitWarnMs` millisecond config key for the default lane wait diagnostic threshold. Future queued entries without an explicit `opts.warnAfterMs` should use this configured value and fall back to the current 2,000 ms behavior when unset. Preserve explicit per-call overrides, especially the existing cron manual-run `warnAfterMs: 5_000` path. Make reload behavior deliberate: config reloads should affect future queue entries without implying already queued entries are retuned. Update schema/types/help/labels/generated configuration docs and queue docs, add focused command-queue/config-reload/cron coverage, and add a user-facing changelog entry. Do not implement per-lane thresholds, adaptive timeouts, session concurrency, or auto-yield; leave that broader product work to https://github.com/openclaw/openclaw/issues/53023.
