## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold so legitimate long-running lane work can avoid flooding gateway logs.

## Best Possible Solution

Land or refresh the focused linked PR so `diagnostics.laneWaitWarnMs` keeps the 2,000 ms default when unset, preserves explicit per-call overrides, and updates schema, types, help, docs, tests, and reload behavior together.

## Reproduction Assessment

Yes. Source inspection on current main shows queue entries without an explicit override still use a hardcoded 2,000 ms threshold, and the config schema/type/docs expose no matching lane wait threshold key.

## Solution Assessment

Yes in direction, but it is not resolved yet. A narrow `diagnostics.laneWaitWarnMs` setting is the maintainable path if maintainers accept the new public config surface and the linked PR aligns the config contract and reload behavior.
