## Summary

## What This Changes

The issue requests a configurable lane wait diagnostic warning threshold so long-running cron or lane work does not flood operator logs with non-actionable warnings.

## Best Possible Solution

Land or refresh https://github.com/openclaw/openclaw/pull/86676 as the focused implementation, preserving the 2,000 ms default and explicit per-call overrides while aligning schema, types, help, docs, and reload behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows queue entries without an explicit override still use the hardcoded 2,000 ms threshold, and diagnostics config/schema/docs expose no matching key.

## Solution Assessment

Yes in direction, but not resolved yet. A narrow `diagnostics.laneWaitWarnMs` setting is the maintainable fix for this logging-control request if maintainers accept the new config surface and the linked PR resolves its contract/reload gaps.
