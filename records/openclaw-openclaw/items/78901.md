## Summary

Keep open: the targeted billing-error surface is plausible, but this patch puts the billing hint in replay-visible assistant content, while current main deliberately keeps stream-error content on the canonical sentinel and trims only that sentinel before replay; the external PR also lacks required after-fix real behavior proof.

## What This Changes

The PR changes stream-error assistant message construction to emit a billing/credits hint for matching provider errors and adds an OpenRouter insufficient-credits regression test.

## Best Possible Solution

Surface billing failures through errorMessage, formatAssistantErrorText, and final payload/lifecycle rendering while keeping stream-error content on the canonical sentinel; add coverage that trailing billing stream-error turns are not replayed.

## Reproduction Assessment

Yes, source-reproducible for the review finding: the PR emits non-sentinel billing content, while current replay cleanup drops only exact STREAM_ERROR_FALLBACK_TEXT content. A focused regression can build that trailing stream-error assistant turn and verify normalizeAssistantReplayContent retains it.

## Solution Assessment

No. The safer fix is to keep replay-visible content on the sentinel and improve the existing errorMessage-based UI/payload path instead of placing billing diagnostics in assistant content.
