## Summary

Keep this PR open: the goal is plausible, but the patch moves billing diagnostics into replay-visible assistant content, which conflicts with the current sentinel-only replay cleanup path and needs real behavior proof before merge.

## What This Changes

The branch changes stream-error assistant message construction to emit a billing/credits hint for matching provider errors and adds an OpenRouter insufficient-credits regression test.

## Best Possible Solution

Surface billing failures through `errorMessage`, `formatAssistantErrorText`, and final payload/lifecycle rendering while keeping stream-error `content` on the canonical sentinel; add coverage that trailing billing stream-error turns are not replayed.

## Reproduction Assessment

Yes for the review finding: the proposed helper emits non-sentinel billing content, while current replay cleanup only drops exact `STREAM_ERROR_FALLBACK_TEXT` content. A focused regression test can construct that billing stream-error turn as the trailing assistant message and verify `normalizeAssistantReplayContent` does not keep it.

## Solution Assessment

No. The safer fix is to keep replay-visible content as the sentinel and improve the existing `errorMessage`-based UI/payload path, rather than placing billing diagnostics in assistant content that is later replayed.
