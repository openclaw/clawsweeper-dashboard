## Summary

Keep this open. Current main still has a reachable stale CLI task path after the active run context is gone, and the operator-facing task table still displays truncated identifiers that the exact-only lookup path does not accept.

## What This Changes

The issue requests automatic terminal reconciliation for dead `runtime: "cli"` task records and clearer `openclaw tasks cancel` behavior for displayed task identifiers.

## Best Possible Solution

Extend task-registry maintenance so stale CLI tasks are considered live only when an active run context or a valid detached-runtime recovery hook proves ownership, and align task list/show/cancel lookup so operators can act on the identifiers the CLI displays or get an explicit ambiguity error.

## Reproduction Assessment

Yes. A code-level reproduction is a stale active `runtime: "cli"` task older than the 5-minute grace window with no `getAgentRunContext()` result and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by the session store; current maintenance leaves it active.

## Solution Assessment

Yes. The narrowest maintainable solution is in task-registry maintenance plus task CLI lookup/list semantics, with regression tests around blank/main-session CLI rows and preserved async media behavior rather than a broad OS process monitor.
