## Summary

Keep open. Current main has scheduled maintenance and normal gateway-run finalization, plus partial stale CLI handling for channel-backed tasks, but stale CLI rows backed by the durable main session or a blank child session can still remain active after the run context is gone.

## What This Changes

Auto-transition terminated CLI runtime task records out of `running` state and improve `openclaw tasks cancel` lookup feedback.

## Best Possible Solution

Task maintenance should treat stale CLI rows as alive only when a live run context or explicit runtime recovery hook proves ownership, and the task CLI should either display usable full IDs or support unique prefixes with clear ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no live `getAgentRunContext()` result, `lastEventAt` older than the five-minute grace window, and `childSessionKey` either blank or `agent:main:main` backed by a persisted session entry.

## Solution Assessment

Yes. The narrowest maintainable solution is in task-registry maintenance and task CLI lookup semantics with focused regression tests, rather than a broad OS PID monitor.
