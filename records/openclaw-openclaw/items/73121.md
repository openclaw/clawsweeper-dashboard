## Summary

Keep open. Current main has scheduled maintenance, normal gateway-run finalization, and partial chat-backed CLI reconciliation, but stale CLI rows with no active run context can still be kept running when `childSessionKey` is blank or points at the durable main session; the task CLI also still displays shortened IDs while lookup/cancel require exact tokens.

## What This Changes

The issue requests automatic cleanup for dead CLI runtime task records and clearer task-cancel lookup behavior.

## Best Possible Solution

Task maintenance should treat stale CLI rows as live only when there is an active run context or explicit recoverable runtime hook, and task lookup should either show full usable IDs or support unique prefixes with clear ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale active CLI `TaskRecord` beyond the five-minute grace window, no `getAgentRunContext()` result, and either a blank `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted main session row.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in task-registry maintenance and task CLI lookup/display semantics with focused regression tests, not a broad OS PID monitor.
