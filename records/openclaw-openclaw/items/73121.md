## Summary

Keep open. Current main still lets stale CLI task records stay active when they lack a live run context but have a blank or durable `agent:main:main` child session, and the CLI still displays shortened task IDs that exact lookup/cancel does not accept. Open PR #76199 covers the main CLI liveness root cause via #76162, but it is unmerged and does not address this issue's lookup/cancel UX gap.

## What This Changes

The issue asks for dead CLI runtime tasks to auto-terminalize after process or run loss and for task cancellation to work with identifiers shown by the CLI.

## Best Possible Solution

Land #76199 or an equivalent all-CLI run-context liveness fix, then either extend task lookup/list to support usable task identifiers or split that cancel UX into a clear follow-up before closing this broader report.

## Reproduction Assessment

Yes. A focused code-level reproduction is a grace-expired `runtime: "cli"` task with inactive `sourceId`/`runId`, `childSessionKey: "agent:main:main"`, and a matching session-store row; current maintenance keeps it running because the persistent session still counts as backing.

## Solution Assessment

Yes for the liveness direction: making CLI backing depend on active run context is the narrow maintainable fix. Unclear for the full issue until the shortened-ID lookup/cancel behavior is either fixed or explicitly split from the liveness work.
