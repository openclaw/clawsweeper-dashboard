## Summary

Keep open: current `main` has scheduled stale-task maintenance and gateway agent-run finalizers, but stale CLI rows can still remain active when their live run context is gone and the row has either no `childSessionKey` or the durable `agent:main:main` session key. The operator UX gap also remains because the human task list shows truncated IDs while lookup stays exact by task ID, run ID, or session key.

## What This Changes

The issue asks OpenClaw to auto-transition dead CLI runtime task records out of `running` and make task cancel/list identifiers usable with clear failure feedback.

## Best Possible Solution

Make task-registry maintenance treat stale CLI tasks with no live run context as unbacked unless a runtime-specific detached recovery hook proves they are still active, while preserving cron durable recovery and async media task liveness. Align `openclaw tasks list/show/cancel` so displayed identifiers are actionable, either by showing usable full IDs or resolving unambiguous prefixes with explicit ambiguity errors.

## Reproduction Assessment

Yes. A code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no `getAgentRunContext()` result, `lastEventAt` older than the five-minute grace window, and either no `childSessionKey` or `childSessionKey: "agent:main:main"` with a persisted session row; current maintenance keeps it active. The cancel/list UX reproduces by comparing the truncated table ID with exact-only `resolveTaskForLookupToken()`.

## Solution Assessment

Yes. The narrowest maintainable fix is in task-registry maintenance plus task CLI lookup/list semantics, using existing run-context and detached-runtime recovery seams rather than adding a broad OS process monitor or changing session-store retention.
