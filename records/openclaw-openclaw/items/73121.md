## Summary

Keep open. Current main has scheduled task maintenance and partial CLI liveness reconciliation, but stale CLI rows can still remain active when they have no child session or are tied to a durable non-chat session, and the human task table still displays truncated IDs that exact lookup/cancel does not accept.

## What This Changes

The issue requests that dead CLI runtime tasks leave `running` automatically and that task cancellation handle displayed task IDs with clear feedback.

## Best Possible Solution

Task-registry maintenance should terminalize stale CLI rows based on live run context or a recoverable runtime hook, and the task CLI should either display full usable IDs or support unique prefixes with ambiguity errors.

## Reproduction Assessment

Yes, at source level. A stale `runtime: "cli"` active task with no live `getAgentRunContext()` and either no `childSessionKey` or `childSessionKey: "agent:main:main"` still passes backing checks on current main.

## Solution Assessment

Yes. The narrowest maintainable fix is in task-registry maintenance plus task CLI lookup/display semantics, using the existing `lost` terminal path rather than adding a broad OS PID monitor or new config knob.
