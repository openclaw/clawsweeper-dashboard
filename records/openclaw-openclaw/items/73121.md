## Summary

Keep open. Current main has scheduled maintenance and a partial chat-backed CLI liveness fix, but stale CLI task rows can still stay active when they have no child session or are backed by a durable parent session, and the task table still shows shortened IDs that exact lookup/cancel does not resolve as prefixes.

## What This Changes

The issue asks OpenClaw to auto-terminalize dead CLI runtime tasks and make `openclaw tasks cancel` reliable for displayed task identifiers.

## Best Possible Solution

Task-registry maintenance should terminalize stale CLI rows based on live run context or an explicit recoverable runtime hook, and task CLI lookup should either show usable full IDs or support unique prefixes with clear ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale `runtime: "cli"` active `TaskRecord` with no live `getAgentRunContext()` result and either blank `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted session row; current maintenance will not mark it lost.

## Solution Assessment

Yes. The narrowest maintainable repair is in task-registry maintenance plus task CLI lookup/display semantics, not a broad OS PID monitor.
