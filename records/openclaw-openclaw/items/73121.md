## Summary

Keep open. Current main has scheduled maintenance and partial CLI run-context reconciliation, but stale CLI rows with no child session or with a durable non-chat session can still remain active, and the human task table still shows shortened IDs that exact lookup does not accept.

## What This Changes

The issue asks for terminated CLI task records to auto-transition out of `running` and for task cancellation lookup to work reliably from displayed task IDs.

## Best Possible Solution

Task maintenance should terminalize stale CLI rows through the existing run-context/lost-state model, and the task CLI should expose or accept unambiguous identifiers that operators can actually use.

## Reproduction Assessment

Yes, at source level. A grace-expired active `runtime: "cli"` task with no live run context and either no `childSessionKey` or a durable non-chat key such as `agent:main:main` still avoids the `lost` path on current main.

## Solution Assessment

Yes. The narrowest maintainable fix is in task-registry maintenance plus task CLI lookup/display semantics, using existing `lost` and run-result terminalization behavior rather than adding a broad OS PID monitor or new config knob.
