## Summary

Keep open: current main has automatic task maintenance, but stale CLI task rows can still remain active when they are backed only by a blank child session key or the durable main session row, and the human task table still displays truncated IDs that exact lookup cannot resolve.

## What This Changes

The issue requests automatic reconciliation of dead CLI runtime tasks from `running` to a terminal state, plus clearer task cancellation lookup behavior.

## Best Possible Solution

Task-registry maintenance should treat CLI tasks with no live run context as unbacked unless a runtime-specific recovery hook proves they are still active, and task CLI lookup/list output should either display usable identifiers or accept short prefixes with explicit ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no `getAgentRunContext()` result, `lastEventAt` older than the 5-minute grace window, and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted session row; current maintenance leaves those active.

## Solution Assessment

Yes. The narrowest maintainable path is to adjust task-registry maintenance and task CLI lookup semantics, with focused tests, rather than adding a broad OS process monitor or rewriting session retention.
