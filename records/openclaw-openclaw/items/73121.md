## Summary

Keep open. Current main has scheduled task maintenance and partial CLI reconciliation, but stale CLI rows with no child session or a durable `agent:main:main` session can still remain active after the live run context is gone, and human task output still prints shortened IDs that exact lookup cannot resolve.

## What This Changes

The issue requests automatic reconciliation for dead CLI runtime tasks plus clearer task cancellation lookup/error behavior.

## Best Possible Solution

Task-registry maintenance should treat stale CLI rows as backed only when a live run context or explicit runtime recovery hook proves they are active, and task CLI lookup should expose usable full identifiers or unique-prefix resolution with ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale active `runtime: "cli"` task older than the five-minute grace window with no live `getAgentRunContext()` and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted session row; current main leaves those rows active.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in task-registry maintenance and task CLI lookup semantics with focused regression tests, not a broad OS PID monitor.
