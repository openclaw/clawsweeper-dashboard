## Summary

Keep open. Current main has scheduled task maintenance and the earlier stale cron/chat-backed task reconciliation work, but the specific CLI zombie cases in this report are still credible: stale CLI rows with no child session are treated as backed, persisted `agent:main:main` rows can keep dead CLI tasks active after their run context is gone, and the human task table still prints shortened task IDs that exact lookup cannot cancel.

## What This Changes

The issue requests automatic reconciliation of dead CLI runtime task records out of `running` plus clearer task cancellation lookup/feedback.

## Best Possible Solution

Task-registry maintenance should classify stale CLI task rows as backed only when a live run context or a runtime-specific recovery hook proves they are still active, and the task CLI should either display usable full identifiers or resolve unique short prefixes with explicit ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no `getAgentRunContext()` result, `lastEventAt` older than the grace window, and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted session row; current maintenance leaves those active.

## Solution Assessment

Yes. The narrowest maintainable solution is in task-registry maintenance and task CLI lookup/list semantics, with focused regression tests; adding a broad OS PID scanner is riskier and does not match the existing run-context/session-backed design.
