## Summary

Keep open. Current main has scheduled maintenance and a partial chat-backed CLI fix, but stale CLI task rows can still remain active when there is no child session key or when `agent:main:main` is treated as a durable backing session; the task table also still prints shortened IDs while lookup/cancel use exact tokens.

## What This Changes

Request to reconcile terminated CLI runtime tasks out of `running` state and make task cancellation lookup usable for the IDs shown by `openclaw tasks list`.

## Best Possible Solution

Task-registry maintenance should terminalize stale CLI rows unless a live run context or explicit recovery hook proves ownership, and task CLI should either show full usable IDs or resolve unique prefixes with clear ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale active `runtime: "cli"` task older than the five-minute grace window, with no live `getAgentRunContext()` result, and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted session row.

## Solution Assessment

Yes, if scoped to the existing maintenance and lookup seams. The narrow maintainable fix is not an OS PID monitor; it is runtime-aware CLI liveness in `task-registry.maintenance.ts` plus task lookup/cancel behavior that matches the displayed tokens.
