## Summary

Keep this open. Current main has useful partial reconciliation for cron, chat-backed CLI tasks, and normal Gateway promise completion, but the reported zombie path is still reachable for CLI tasks whose active run context is gone while a durable main-session row or blank child-session key is still treated as backing state. The task cancel/list UX gap is also still present because human output truncates IDs while lookup remains exact-only.

## What This Changes

The issue requests automatic reconciliation of terminated CLI runtime tasks and clearer task cancellation behavior for operators.

## Best Possible Solution

Extend task-registry maintenance so CLI tasks with no active run context are not considered alive solely because the durable main session exists or the child session key is blank, while preserving explicit runtime recovery hooks for long-lived media/tool work. Align the task list/show/cancel operator path so the identifier shown to humans is copyable or unambiguous prefixes produce clear errors.

## Reproduction Assessment

Yes. A code-level reproduction is a stale active TaskRecord with runtime "cli", no active getAgentRunContext result, lastEventAt older than the reconcile grace, and childSessionKey "agent:main:main" backed by a durable session-store row; current hasBackingSession returns true and maintenance leaves it running.

## Solution Assessment

Yes for a focused repair. The narrowest maintainable fix is in task-registry maintenance and task CLI lookup/list behavior, not a broad OS process monitor or unrelated session-retention rewrite.
