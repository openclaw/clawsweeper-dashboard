## Summary

Keep open. Current main still has a concrete CLI zombie path: scheduled task maintenance exists, but stale `cli` tasks without a live run context can remain backed by a blank child session or by the durable `agent:main:main` session row, and the human task table still prints shortened IDs while lookup remains exact.

## What This Changes

The issue requests automatic reconciliation of terminated CLI runtime tasks and clearer task cancellation lookup/error behavior.

## Best Possible Solution

Task maintenance should treat stale CLI rows as live only when a live run context or an explicit runtime recovery hook proves activity, and task CLI lookup should expose usable identifiers or unique-prefix resolution with clear ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale `runtime: "cli"` active task older than the five-minute grace window with no live `getAgentRunContext()` result and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted main session row.

## Solution Assessment

Yes. The narrowest maintainable fix is in task-registry maintenance and task CLI lookup semantics, using the existing `lost`/terminal reconciliation model rather than broad OS PID polling.
