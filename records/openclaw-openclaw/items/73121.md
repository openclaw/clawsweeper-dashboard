## Summary

Keep this open. Current main still has a reachable zombie path for stale `runtime: "cli"` task rows: once the active run context is gone, blank child-session keys and durable main-session rows can still be treated as valid backing state, so maintenance does not mark them terminal. The operator UX gap also remains because the human task table truncates IDs while lookup is exact-only.

## What This Changes

The issue requests automatic reconciliation of terminated CLI runtime tasks and clearer task cancellation behavior for operators.

## Best Possible Solution

Extend task-registry maintenance so CLI tasks with no active run context are not considered alive solely because the durable main session exists or because `childSessionKey` is absent, while preserving detached runtime recovery hooks for media/tool work. Align human task list/show/cancel behavior so operators can use the identifiers the CLI displays or receive clear ambiguity errors.

## Reproduction Assessment

Yes. A code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no `getAgentRunContext()` result, `lastEventAt` older than the grace window, and either blank `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a session-store row; current maintenance leaves it active.

## Solution Assessment

Yes for a focused repair. The narrowest maintainable fix is in task-registry maintenance plus task CLI lookup/list semantics, not a broad OS process monitor or unrelated session-retention rewrite.
