## Summary

Keep open. Current main has scheduled task maintenance and prior chat-backed CLI reconciliation, but stale CLI rows with blank or durable main-session backing can still remain active after their live run context is gone, and the human task list still exposes shortened IDs that exact lookup cannot resolve.

## What This Changes

The issue requests automatic reconciliation of dead CLI runtime task records out of `running` plus clearer task cancellation lookup and feedback.

## Best Possible Solution

Task-registry maintenance should treat stale CLI rows as backed only when a live run context or runtime recovery hook proves they are active, and task CLI lookup/list output should expose usable identifiers or unique-prefix resolution with explicit ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no `getAgentRunContext()` result, `lastEventAt` older than the five-minute grace window, and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted session row; current main leaves those active.

## Solution Assessment

Yes. The narrowest maintainable solution is in task-registry maintenance and task CLI lookup semantics with focused regression tests, not a broad OS PID monitor.
