## Summary

Keep this open. Current main has partial CLI task reconciliation, but stale `runtime: "cli"` rows can still remain active when their live run context is gone and the row has either no child session key or a durable main-session key. The task-cancel UX also still displays shortened task IDs while lookup remains exact task ID, run ID, or session key only.

## What This Changes

The issue requests automatic terminal reconciliation for dead CLI runtime tasks and clearer task cancellation behavior for task IDs shown by the CLI.

## Best Possible Solution

Add runtime-aware CLI stale reconciliation before the generic session-store fallback: after the grace window, CLI rows with no live run context should become terminal unless a clearly active CLI-owned tool/runtime keepalive still exists. Align task list/show/cancel lookup so the identifiers displayed to humans are either copyable full IDs or accepted prefixes with explicit ambiguity errors.

## Reproduction Assessment

Yes. A code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no `getAgentRunContext()` match for `sourceId` or `runId`, `lastEventAt` older than the grace window, and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by the session store; current maintenance leaves it active.

## Solution Assessment

Yes for a focused repair. The narrowest maintainable fix is in `src/tasks/task-registry.maintenance.ts` plus task lookup/list/cancel semantics, with regression tests for main-session and blank-key CLI rows; a broad OS process monitor is not required for this reported path.
