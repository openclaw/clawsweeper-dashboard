## Summary

Keep open. Current main has a scheduled task sweeper and gateway-agent completion finalization, but stale CLI task records can still remain active when there is no child session key or when the child key is a durable non-chat session such as `agent:main:main`; the task CLI also still displays shortened IDs that exact lookup/cancel does not accept as prefixes.

## What This Changes

The issue requests automatic terminalization of dead CLI runtime tasks and clearer task cancel/list lookup behavior for stale running task records.

## Best Possible Solution

Task-registry maintenance should terminalize stale CLI rows from authoritative live-run/runtime state, and task CLI lookup should either display full usable IDs or support unique prefixes with clear ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an old active `runtime: "cli"` task with no live `getAgentRunContext()` and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted session row; current maintenance will not mark those rows lost.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in task-registry maintenance plus task CLI lookup/display semantics, not a broad OS PID monitor.
