## Summary

Keep open. Current main has scheduled task maintenance and normal gateway-run finalization, but stale CLI task rows can still remain active when they have no child session or are backed by the durable main session, and the task CLI still displays shortened IDs that lookup/cancel do not resolve as prefixes.

## What This Changes

The issue asks OpenClaw to reconcile dead CLI runtime tasks out of `running` state and improve task cancellation lookup when only shortened task IDs are shown.

## Best Possible Solution

Task-registry maintenance should treat stale CLI rows as backed only by a live run context or an explicit recoverable runtime hook, and task CLI lookup should either show usable full IDs or support unique prefixes with clear ambiguity errors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no live `getAgentRunContext()` result, `lastEventAt` older than the five-minute grace window, and either no `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a persisted session row.

## Solution Assessment

Yes. The narrowest maintainable fix is in task-registry maintenance and task CLI lookup semantics with focused regression tests, not a broad OS PID monitor.
