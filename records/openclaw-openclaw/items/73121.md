## Summary

Keep open: current `main` has partial stale-task reconciliation, but stale CLI runtime rows can still remain active when the live run context is gone and the task has either no child session key or the durable main-session key. The operator-facing cancel/list gap also remains because the human table displays truncated IDs while lookup stays exact-only.

## What This Changes

The issue requests automatic terminal-state reconciliation for dead CLI runtime tasks plus clearer task-cancel/list behavior for displayed task IDs.

## Best Possible Solution

Make task-registry maintenance treat CLI tasks with no live run context as unbacked unless a runtime-specific detached recovery hook proves they are still active, while preserving media/tool recovery semantics. Align `openclaw tasks list/show/cancel` so the identifiers shown in the table are usable or produce explicit ambiguity errors.

## Reproduction Assessment

Yes. A code-level reproduction is a stale active `TaskRecord` with `runtime: "cli"`, no `getAgentRunContext()` result, `lastEventAt` older than the 5-minute grace window, and either blank `childSessionKey` or `childSessionKey: "agent:main:main"` backed by a session-store row; current maintenance leaves it active.

## Solution Assessment

Yes. The narrowest maintainable fix is in task-registry maintenance plus task CLI lookup/list semantics, not a broad OS process monitor or a session-store retention rewrite.
