## Summary

Keep open. Current main has fixed the shared external delivery and Telegram reply slices, but WebChat/Gateway final broadcasts, embedded TUI final events, and plugin-handled message actions still bypass the outbound sent-message hook surfaces, so the umbrella coverage bug remains valid.

## What This Changes

The issue requests consistent `message:sent` and plugin `message_sent` hook emission across outbound dispatch paths, including Telegram replies, plugin sends, and gateway/RPC reply paths.

## Best Possible Solution

Define and document one outbound lifecycle observer seam, or split the remaining work into canonical narrower fixes, so external channel replies, WebChat, Gateway-backed TUI, embedded TUI, and plugin-handled sends invoke the applicable pre-send and post-send hooks exactly once with stable session/run context and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable a sent-message hook and produce a WebChat/Gateway final reply, embedded TUI final reply, or plugin-handled message action; current main uses direct broadcast/emit/plugin-dispatch paths with no sent-message hook call on those routes.

## Solution Assessment

No for closing or for treating the original Telegram-only fix as complete. The narrow maintainable path is either a shared lifecycle observer with explicit exactly-once semantics or maintainer-approved split fixes for WebChat/TUI, plugin actions, and canonical context propagation.
