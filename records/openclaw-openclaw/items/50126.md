## Summary

Keep open. Current main wires the shared outbound and Telegram reply paths, but WebChat/Gateway final broadcasts, embedded TUI final events, plugin-handled message actions, and the shared plugin `message_sent` session context still show source-level coverage gaps; related narrower issues #65792, #70928, and #77284 are open rather than merged fixes.

## What This Changes

The issue asks for consistent internal `message:sent` and plugin `message_sent` hook coverage across all outbound dispatch paths, including Telegram replies, WebChat/TUI delivery, RPC responses, and plugin-handled sends.

## Best Possible Solution

Define one documented outbound lifecycle observer, or split the remaining surfaces into canonical narrower fixes, so every outbound delivery path emits the applicable pre-send and post-send hooks exactly once with stable session/run context.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable a sent-message hook and produce a WebChat/Gateway final reply, embedded TUI final reply, or plugin-handled message action; current main uses direct broadcast/emit/plugin-dispatch paths without sent-message hook emission on those routes.

## Solution Assessment

No for closing or for treating the Telegram/shared-path fixes as complete. The maintainable path needs either a shared lifecycle observer with exactly-once semantics or maintainer-approved split fixes for WebChat/TUI, plugin actions, and canonical context propagation.
