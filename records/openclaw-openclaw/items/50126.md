## Summary

Keep open. Current main has hook-aware shared outbound delivery and Telegram direct replies, but the broader inconsistency still exists for WebChat/Gateway final broadcasts, embedded TUI final events, and plugin-handled message actions that bypass the shared sent-message hook surfaces.

## What This Changes

The issue asks OpenClaw to emit internal `message:sent` and plugin `message_sent` hooks consistently across all outbound dispatch paths, including Telegram replies, Gateway/WebChat, RPC responses, and plugin message sends.

## Best Possible Solution

Define and document one outbound lifecycle observer seam, or explicitly split the remaining gaps into narrower owner-approved fixes, so external channel replies, WebChat, Gateway-backed TUI, embedded TUI, and plugin-handled sends invoke the applicable pre-send and post-send hooks exactly once with stable session/run context and focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable a sent-message hook, produce a WebChat/Gateway or embedded TUI assistant final reply, or send through a plugin-handled message action, and observe that current main uses raw broadcast/emit/plugin-dispatch paths with no sent-message hook runner call; I did not run live delivery because this review is read-only.

## Solution Assessment

No for the original Telegram-only minimal fix as the complete solution: current main covers the direct Telegram slice, but the umbrella inconsistency remains. The maintainable direction is a shared lifecycle observer or maintainer-approved narrower fixes per surface with explicit exactly-once and rewrite/cancel semantics.
