## Summary

Keep open. Current main has hook-aware shared outbound delivery and Telegram reply delivery, but WebChat/Gateway final broadcasts, embedded TUI final events, and plugin-handled message actions still bypass the sent-message hook surfaces, so the umbrella inconsistency remains valid.

## What This Changes

Request to make internal `message:sent` and plugin `message_sent`/`message_sending` hook coverage consistent across outbound dispatch paths, especially Telegram, WebChat/TUI, and plugin-handled sends.

## Best Possible Solution

Define and document one outbound lifecycle observer seam, or split the work into narrower owner-approved fixes, so all outbound final deliveries and plugin-handled sends invoke the applicable hooks exactly once with stable session/run context.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable a sent-message hook, produce a WebChat/Gateway or embedded TUI final assistant reply, or send through a plugin-handled message action, and observe that current main uses raw broadcast/emit/plugin-dispatch paths with no sent-message hook runner call.

## Solution Assessment

No for the original Telegram-only minimal fix as the complete solution: current main covers the Telegram slice, but the umbrella inconsistency remains. The maintainable direction is a shared lifecycle observer or split fixes with explicit rewrite/cancel, transcript, streaming, and exactly-once semantics.
