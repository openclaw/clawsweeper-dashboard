## Summary

Keep open: current main has fixed the shared outbound path and Telegram direct-reply portions, but Gateway/WebChat final broadcasts, embedded TUI final events, and plugin-handled send actions still do not share the same outbound lifecycle hook coverage. Open related PRs cover narrower pre-send or internal-hook slices, not the full remaining umbrella.

## What This Changes

The issue requests consistent outbound lifecycle hook coverage so internal `message:sent` and plugin `message_sent` observers do not miss sent replies across dispatch paths.

## Best Possible Solution

Define and document one exactly-once outbound lifecycle observer seam, or split the remaining Gateway/WebChat, embedded TUI, plugin-handled action, and pre-send bypasses into narrower canonical fixes with focused regression tests.

## Reproduction Assessment

Yes at source level: current `main` still has Gateway/WebChat final broadcasts, embedded TUI final events, and plugin-handled message action returns that bypass the shared outbound sent-hook emitter. I did not run a live channel/plugin scenario in this read-only review.

## Solution Assessment

No, not as a single Telegram-only or one-path patch. The maintainable solution needs either a maintainer-approved exactly-once lifecycle seam or narrower canonical issues for each remaining delivery surface.
