## Summary

Keep open. Current main has fixed the originally confirmed Telegram reply slice and the shared outbound delivery helper is hook-aware, but the broader inconsistency still exists: WebChat/Gateway finals, embedded TUI finals, and plugin-handled message actions still bypass the documented sent-message hook surfaces.

## What This Changes

The issue asks for consistent `message:sent` and `message_sent` hook coverage across outbound dispatch paths, including Telegram replies, WebChat/TUI finals, RPC responses, and plugin-handled sends.

## Best Possible Solution

Define one documented outbound lifecycle observer seam, or split this umbrella into canonical narrower fixes, so external channel replies, WebChat/Gateway finals, embedded TUI finals, and plugin-handled sends invoke the applicable pre-send and post-send hooks exactly once with stable session/run context and regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable a sent-message hook, produce a WebChat/Gateway or embedded TUI assistant final, or send through a plugin-handled message action, then observe that those paths directly broadcast/emit/dispatch without a sent-message hook call; no live run was performed because this review is read-only.

## Solution Assessment

No for the original minimal Telegram-only fix: that slice is now covered on current main, but it does not solve the umbrella. The safer maintainable direction is a maintainer-approved shared observer seam or split follow-up issues per surface with explicit hook timing and exactly-once contracts.
