## Label Justifications

- `P1`: Cron denial classification affects whether scheduled agent executions report blocked commands as fatal failures or ordinary output.
- `merge-risk: 🚨 compatibility`: The PR changes existing cron failure semantics and can break users who depend on fatal denial reporting for blocked scheduled executions.

## Summary

Keep open: the PR is aimed at a real cron classification bug, but the current head still misses the production host=node denial shape because OpenClaw wraps node denials as `UNAVAILABLE` with the `SYSTEM_RUN_DENIED` marker in the message, while the PR tests and proof use `nodeError.code: SYSTEM_RUN_DENIED`. Removing the prose fallback without preserving that actual structured path can make real cron denials look like ordinary output.

## What This Changes

The PR removes prose-token cron denial classification, adds structured tool error-code propagation through embedded tool lifecycle handling, and updates focused cron failure-signal tests.

## Best Possible Solution

Keep the prose-token removal, but normalize canonical denial codes from the actual node-host/gateway error contract and prove both prose-only false positives and real structured denials.

## Reproduction Assessment

Yes. Source inspection of current main shows prose-token denial text is promoted to fatal cron output, and source inspection of the PR shows the production node-host denial wrapper can still be missed after the fallback is removed.

## Solution Assessment

No. The direction is right, but the implementation should preserve canonical denial signals for the actual node-host/gateway error shape rather than only for a synthetic nested `SYSTEM_RUN_DENIED` code.
