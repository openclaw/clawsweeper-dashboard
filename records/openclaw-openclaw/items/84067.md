## Summary

Keep this PR open: the intended move away from prose-token cron denial detection is sound, but the patch currently depends on structured error codes that are not preserved by the existing thrown-tool-error path, and the PR has test-only proof rather than a real cron run.

## What This Changes

The PR removes prose-token cron denial classification, adds `errorCode` to tool error summaries, extracts structured tool error codes from failed tool results, and updates regression coverage.

## Best Possible Solution

Keep the prose-token removal, but propagate typed denial metadata at the tool lifecycle boundary for both returned structured errors and thrown gateway/tool errors, then prove the behavior with regression coverage and a real cron run.

## Reproduction Assessment

Yes. Current main source and tests show `resolveCronPayloadOutcome` promotes denial-looking prose to fatal cron output; I did not run tests because this review is read-only.

## Solution Assessment

No. Removing broad prose scanning is the right direction, but relying only on `lastToolError.errorCode` is not safe until the real thrown-tool-error path preserves that code or has a narrow covered fallback.
