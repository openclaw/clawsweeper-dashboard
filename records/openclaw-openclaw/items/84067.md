## Summary

Keep this PR open: the structured-signal direction is good, but the latest head still misses the current production host=node denial shape, and it removes a documented cron fallback without contract follow-through.

## What This Changes

The PR removes cron prose-token denial classification, adds structured tool error-code propagation through embedded tool lifecycle errors, and updates focused agent/cron regression tests.

## Best Possible Solution

Normalize the actual gateway/node-host denial contract into a canonical fatal cron failure signal while keeping prose-only output non-fatal, then update the scheduled-tasks docs if the fallback contract is intentionally tightened.

## Solution Assessment

No, not yet: the structured-signal approach is the right direction, but the implementation needs to normalize the actual production wrapper or change the source to emit a canonical denial code. The docs also need to match the stricter contract if the fallback is intentionally removed.
