## Summary

Keep open: this is a plausible fix for slow `/models`, but the patch still has merge-blocking auth-cache and watcher availability issues, and the current proof does not cover the new invalidation paths.

## What This Changes

The PR removes the prepared provider-auth TTL, keys warmed auth state by `agentId`, updates model-listing auth checks to use `agentId`, and adds auth-failure plus `auth-profiles.json` invalidation with tests.

## Best Possible Solution

Keep the `agentId` per-agent prewarm direction, but make invalidation a managed gateway lifecycle component that handles watcher errors, closes cleanly, repopulates prepared auth after auth changes, and has redacted real gateway proof for timing plus invalidation before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main shows the 10-second TTL, singleton default-agent warmed state, and `agentDir` guard that can make prepared auth expire or miss; the PR body also includes real before/after timing logs, but I did not run a live gateway in this read-only review.

## Solution Assessment

No, not yet. The `agentId` prewarm direction is maintainable, but the implementation needs managed watcher lifecycle, watcher error handling, and a rewarm path after invalidation before the auth-cache semantic change is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Handle auth-profile watcher errors:** `src/agents/auth-profiles-watcher.ts:22-26`
  - body: The new `chokidar` watcher never registers an `error` listener. If watching an agent auth profile fails, Node treats an unhandled `error` event as thrown, so a filesystem permission or watcher failure can take down the gateway; follow the local `config-reload.ts` pattern by warning and closing or disabling the watcher cleanly.
  - confidence: 0.9
- **[P2] Close the auth-profile watcher on shutdown:** `src/gateway/server-startup-post-attach.ts:1029-1033`
  - body: The watcher handle returned here is discarded, and the gateway close path has no way to call `stop()`. That can leave file watchers and callbacks alive after shutdown or restart; store it in gateway runtime state or expose it as a post-ready sidecar so it is closed with the rest of the gateway.
  - confidence: 0.88
- **[P2] Rewarm after invalidating prepared auth:** `src/gateway/server-startup-post-attach.ts:1028-1032`
  - body: Both new invalidation paths only call `clearCurrentProviderAuthState()`. After an auth failure or external `auth-profiles.json` write, no path repopulates the prepared map until another reload, logout, or restart, so later `/models` calls can remain on the slow per-provider sweep and reintroduce the same Discord timeout this PR is meant to avoid.
  - confidence: 0.84

## Risks / Open Questions

- The PR changes cache freshness from TTL-backed refresh to event-driven invalidation, so stale or missing auth answers now depend on every invalidation path being correct.
