The PR changes `DiffArtifactStore.getArtifact` to compare artifact tokens with `safeEqualSecret` instead of `!==`.

## Best Possible Solution

Land the narrow comparison hardening on current main using `openclaw/plugin-sdk/security-runtime`, preserve the current surrounding imports, and require targeted proof for valid and invalid diffs viewer token requests before merge.

## Reproduction Assessment

Yes, by source inspection: current main validates a 48-hex diffs viewer token and routes it into `DiffArtifactStore.getArtifact`, which still uses `meta.token !== token`. I did not run a live timing side-channel repro, but the affected path and helper contract are clear.

## Solution Assessment

No, not as written: using `safeEqualSecret` is the right narrow fix, but this branch imports it from a non-exported SDK subpath and is stale against the current file. The maintainable solution is the same comparison change using `openclaw/plugin-sdk/security-runtime` on current main.
