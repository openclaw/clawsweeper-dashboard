## Label Justifications

- `P2`: The issue affects a real but bounded single-owner `dmScope=main` workflow without making the core runtime unusable.
- `impact:session-state`: Exec, cron, and completion events can be recorded in orphan per-channel sessions instead of the configured main session.
- `impact:message-loss`: Internal completion and cron events can be routed away from the session that should observe and react to them, with reported leak and misroute symptoms.

## Summary

Keep open. Current main still preserves same-agent direct-DM keys in the exec, cron, system-event, and forced heartbeat paths, and https://github.com/openclaw/openclaw/pull/83743 is an open replacement PR that explicitly closes this issue.

## What This Changes

The issue asks for `session.dmScope=main` to route exec-event and cron-event heartbeat wakes and queued system events to the agent main session instead of per-channel direct-DM sessions.

## Best Possible Solution

Land the narrow shared routing-helper fix from https://github.com/openclaw/openclaw/pull/83743, or close that PR if maintainers reject the approach, while keeping enqueue and wake target normalization aligned for single-owner `dmScope=main` event keys.

## Reproduction Assessment

Yes, at source level. Current main preserves same-agent direct-DM keys through exec enqueue, cron target resolution, and forced heartbeat selection instead of resolving them to `agent:<id>:main` under `session.dmScope=main`.

## Solution Assessment

Yes. Applying the same single-owner `dmScope=main` policy to both event enqueue keys and forced heartbeat wake targets is the narrow maintainable fix; changing only one side would strand events under the other key.
