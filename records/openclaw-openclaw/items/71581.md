## Label Justifications

- `P2`: The issue affects a real but bounded single-owner `dmScope=main` workflow without making the core runtime unusable.
- `impact:session-state`: The reported behavior splits exec and cron completion history into orphan per-channel sessions instead of the configured main session.
- `impact:message-loss`: Internal completion and cron events can be routed away from the session that should observe and react to them.

## Summary

Keep open. Current main still preserves same-agent direct-DM event keys through exec enqueue, cron target resolution, and forced heartbeat wake handling, and the active replacement PR that explicitly fixes this issue is still open.

## What This Changes

Honor `session.dmScope=main` for exec-event and cron-event heartbeat wakes so single-owner direct-DM event keys route to the agent main session.

## Best Possible Solution

Review and land or close the narrower replacement PR at https://github.com/openclaw/openclaw/pull/83743, keeping the normalization shared across enqueue and wake paths and gated to single-owner `dmScope=main` direct-DM event keys.

## Reproduction Assessment

Yes. Source inspection on current main shows a same-agent per-channel direct-message key with `session.dmScope=main` is preserved through exec enqueue, cron target resolution, and forced heartbeat wake selection instead of resolving to `agent:<id>:main`.

## Solution Assessment

Yes. The maintainable fix is to apply the existing single-owner `dmScope=main` policy consistently to both system-event enqueue and forced heartbeat wake resolution; the open replacement PR is the current concrete path for that fix.
