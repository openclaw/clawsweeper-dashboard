## Label Justifications

- `P2`: The issue affects a real but bounded single-owner `dmScope=main` workflow without making the core runtime unusable.
- `impact:session-state`: The reported behavior splits exec, cron, and subagent completion history into orphan per-channel sessions instead of the configured main session.
- `impact:message-loss`: Internal completion and cron events can be routed away from the session that should observe and react to them, with reported user-visible leak/misroute symptoms.

## Summary

Keep open. Current main still preserves same-agent direct-DM event keys through exec enqueue, cron target resolution, and forced heartbeat wake handling, and an open replacement PR explicitly references this issue as the fix path.

## What This Changes

The issue requests that `session.dmScope=main` route exec-event and cron-event heartbeat wakes to the agent main session instead of per-channel direct-message sessions.

## Best Possible Solution

Land or close the narrower linked replacement PR after maintainer review, keeping enqueue-key and wake-target normalization constrained to single-owner `dmScope=main` direct-DM event keys.

## Reproduction Assessment

Yes. Source inspection on current main shows a same-agent per-channel direct-message key with `session.dmScope=main` is preserved through exec enqueue, cron target resolution, and forced heartbeat wake selection instead of resolving to `agent:<id>:main`.

## Solution Assessment

Yes. The best fix is to apply the existing single-owner `dmScope=main` policy consistently to both system-event enqueue and forced heartbeat wake resolution; changing only one side would leave events stranded under the other key.
