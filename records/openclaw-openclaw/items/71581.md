## Summary

Keep open. Current main still preserves same-agent per-channel forced heartbeat keys outside the cron-run helper path, and the issue is paired with an open unmerged implementation PR using closing syntax.

## What This Changes

Request that `session.dmScope=main` also route exec-event and cron-event heartbeat wakes and queued system events to the agent main session for single-owner DM setups.

## Best Possible Solution

Review, repair, land, or replace https://github.com/openclaw/openclaw/pull/72648 with a narrow implementation that normalizes enqueue keys and forced heartbeat targets together under the existing `dmScope=main` single-owner condition.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enqueue an exec or cron system event under a same-agent per-channel DM key and request an `exec-event` or `cron-event` wake; current main resolves and peeks that per-channel key instead of `agent:<id>:main`.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if enqueue and wake routing are normalized together under the same single-owner `dmScope=main` policy; changing only heartbeat resolution would strand queued events under the old key.
