## Summary

Keep open. Current main still treats same-agent forced exec/cron heartbeat session keys literally, while `session.dmScope=main` single-owner handling is wired into inbound channel routing only; open PR #72648 also references this issue with closing syntax, so the issue should remain open until that PR is reviewed, replaced, closed, or merged.

## What This Changes

Honor `session.dmScope=main` for exec-event and cron-event heartbeat wakes so single-owner direct-message activity resolves to the agent main session instead of per-channel orphan sessions.

## Best Possible Solution

Review, land, or replace #72648 with a narrow core change that normalizes both system-event enqueue keys and forced heartbeat wake targets for same-agent direct-message sessions when `dmScope=main` and the existing single-owner allowlist rule is satisfied, while preserving explicit group/channel/custom cron routing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `dmScope=main` with a single-owner Telegram allowlist, enqueue an exec or cron system event under `agent:main:telegram:default:direct:<userId>`, and request an exec-event or cron wake with that forced key; current main resolves and peeks the per-channel queue instead of `agent:main:main`.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if enqueue and wake routing are normalized together; a heartbeat-only rewrite would be incomplete because system events are keyed before preflight peeks and drains them.
