## Label Justifications

- `P2`: The issue affects a real but bounded single-owner `dmScope=main` workflow without making the core runtime unusable.
- `impact:session-state`: Internal exec and cron event history can accumulate in orphan direct-DM sessions instead of the configured main session.
- `impact:message-loss`: Completion and cron events can be routed away from the session that should observe and react to them.

## Summary

Keep open. Current main still preserves same-agent direct-DM keys through exec-event enqueue, cron target resolution, and forced heartbeat wake handling, and the linked open PR is the current implementation path with closing syntax.

## What This Changes

The issue requests that `session.dmScope=main` also route internal exec-event and cron-event heartbeat wakes into the agent main session instead of preserving per-channel direct-DM session keys.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/83743, keeping the fix scoped to single-owner `dmScope=main` event enqueue and wake normalization without changing multi-owner, wildcard, route-binding, or global-scope behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows same-agent direct-DM keys under `session.dmScope=main` are preserved through exec enqueue, cron target resolution, and forced heartbeat selection instead of resolving to `agent:<id>:main`.

## Solution Assessment

Yes. The maintainable path is the linked narrow PR: centralize event-session routing so enqueue keys and heartbeat wake targets apply the same single-owner `dmScope=main` policy.
