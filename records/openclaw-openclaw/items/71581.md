## Label Justifications

- `P2`: The issue affects a real but bounded single-owner `dmScope=main` workflow without making the core runtime unusable.
- `impact:session-state`: Exec and cron event history can accumulate in orphan per-channel sessions instead of the configured main session.
- `impact:message-loss`: Internal completion and cron events can be routed away from the session that should observe and react to them, with reported user-visible leak and misroute symptoms.

## Summary

Keep open: current main and shipped v2026.5.18 still preserve same-agent direct-DM event keys through exec enqueue, cron target resolution, and forced heartbeat selection, while the narrower open PR https://github.com/openclaw/openclaw/pull/83743 is the active fix path.

## What This Changes

Route exec-event and cron-event heartbeat wakes for single-owner `session.dmScope=main` direct-DM sessions to the agent main session instead of orphan per-channel sessions.

## Best Possible Solution

Normalize system-event enqueue keys and forced heartbeat wake targets through one shared `dmScope=main`-aware policy, then close this issue after the narrower replacement PR lands and retire the older conflicting PR.

## Reproduction Assessment

Yes. Source inspection on current main shows a same-agent per-channel direct-message key with `session.dmScope=main` is preserved through exec enqueue, cron target resolution, and forced heartbeat selection instead of resolving to `agent:<id>:main`.

## Solution Assessment

Yes. The maintainable fix is to apply the existing single-owner `dmScope=main` policy consistently to both event enqueue keys and forced heartbeat wake targets; changing only one side would still strand events under the other key.
