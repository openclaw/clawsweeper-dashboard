## Summary

Keep open. Current main still routes forced exec-event and cron-event heartbeats by the literal same-agent session key instead of applying the single-owner `session.dmScope=main` rule, and the issue is paired with open, unmerged PR #72648 using closing syntax.

## What This Changes

Honor `session.dmScope=main` for exec-event and cron-event heartbeat wakes so single-owner direct-message system events are processed in the agent main session.

## Best Possible Solution

Review, land, or replace #72648 with a narrow shared routing helper that maps same-agent direct-message exec/cron system events to `agent:<id>:main` only when `dmScope=main` and the existing single-owner allowlist rule is satisfied, while preserving group, channel, cross-agent, subagent, and explicit custom-session behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `session.dmScope=main` with a single-owner Telegram allowlist, enqueue an exec or cron system event under `agent:main:telegram:default:direct:<userId>`, and trigger an exec-event or cron wake; current main resolves and peeks that per-channel queue instead of `agent:main:main`.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented as shared normalization for both enqueue and wake resolution; a heartbeat-only rewrite would be incomplete because pending system events are keyed before preflight peeks and drains them.
