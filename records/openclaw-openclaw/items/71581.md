## Summary

Keep open. Current main still preserves same-agent non-cron per-channel session keys for exec-event and cron heartbeat wakes, and the issue is already paired with an open unmerged implementation PR that uses closing syntax.

## What This Changes

Honor `session.dmScope=main` for exec-event and cron-event heartbeat wakes so single-owner DM agents process those system events in `agent:<id>:main` instead of per-channel orphan sessions.

## Best Possible Solution

Normalize system-event enqueue keys and forced heartbeat targets through one dmScope-aware single-owner policy seam while preserving current exec trust, delivery-context, subagent, cron-run, and same-agent isolation contracts.

## Reproduction Assessment

Yes, at source level. Enqueue an exec or cron system event under a same-agent per-channel direct-message key and request an `exec-event` or `cron` wake; current main resolves and peeks that per-channel key rather than `agent:<id>:main`.

## Solution Assessment

Yes directionally, but the implementation must normalize enqueue and wake keys together. Changing only heartbeat resolution would leave queued system events under the original per-channel key.
