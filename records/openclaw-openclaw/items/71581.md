## Label Justifications

- `P2`: The issue affects a real but bounded single-owner `dmScope=main` workflow without making the core runtime unusable.
- `impact:session-state`: Internal exec and cron event history can accumulate in orphan direct-DM sessions instead of the configured main session.
- `impact:message-loss`: Completion and cron events can be routed away from the session that should observe and react to them, making the main conversation miss internal work results.

## Summary

Keep open. Current main still preserves same-agent direct-DM event keys through exec enqueue, cron target resolution, and forced heartbeat wake handling, while https://github.com/openclaw/openclaw/pull/83743 is an open closing PR that appears to be the narrower fix path.

## What This Changes

The issue requests that `exec-event` and `cron-event` heartbeat wakes honor `session.dmScope=main` by routing same-agent direct-DM event queues and wakes to the agent main session.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/83743, keeping the fix scoped to single-owner `dmScope=main` event enqueue and wake normalization without changing multi-owner, wildcard, or explicit route-binding behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a same-agent direct-DM key under `session.dmScope=main` is preserved through exec enqueue, cron target resolution, and forced heartbeat wake selection instead of resolving to `agent:<id>:main`.

## Solution Assessment

Yes. The narrow maintainable fix is to apply the existing single-owner `dmScope=main` policy consistently to both event enqueue keys and forced heartbeat wake targets; the open replacement PR is the current review path for that direction.
