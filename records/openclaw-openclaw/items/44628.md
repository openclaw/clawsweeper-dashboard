## Summary

Keep open. The reported systemd lifecycle bug is real, but this PR is not ready because it changes a restart helper that is not the main `openclaw gateway restart` path, weakens Docker mount validation, mis-prioritizes enabled-only units over active units, and lacks real after-fix systemd proof.

## What This Changes

The PR adds user/system systemd scope probing to `triggerOpenClawRestart()` and removes the Docker setup whitespace guard for config paths.

## Best Possible Solution

Move user/system scope handling into the shared Linux gateway service layer, keep Docker mount validation out of this PR unless compose quoting is fixed end to end, add focused daemon lifecycle tests, and require redacted real systemd proof.

## Reproduction Assessment

Yes. The current source path is clear: `openclaw gateway restart` calls the Linux service abstraction, which still reaches `execSystemctlUser()` only; a live systemd host is still needed for after-fix runtime proof.

## Solution Assessment

No. The PR improves a related helper, but the maintainable fix belongs in `src/daemon/systemd.ts` and should not include the unrelated Docker whitespace validation change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move scope detection into the gateway service restart path:** `src/infra/restart.ts:307-309`
  - body: The reported `openclaw gateway restart` flow calls the daemon service abstraction and ends at `restartSystemdService`, which still uses user-scope `systemctl`. Adding scope detection only to `triggerOpenClawRestart()` leaves the linked CLI lifecycle bug unfixed.
  - confidence: 0.9
- **[P2] Prefer active systemd units over enabled-only units:** `src/infra/restart.ts:345-351`
  - body: After the enabled+active cases, this decision tree chooses enabled-only units before active-only units. In a mixed state it can restart or start the inactive scope while leaving the currently running gateway untouched.
  - confidence: 0.86
- **[P2] Keep mount whitespace validation until compose quoting is fixed:** `docker-setup.sh:147`
  - body: Removing this guard lets whitespace-containing host paths pass early validation even though mount specs still reject spaces and generated compose entries are not structurally quoted. Revert this unrelated change or update the whole mount path consistently.
  - confidence: 0.88
- **[P3] Avoid duplicating the detected scope in failure details:** `src/infra/restart.ts:389-392`
  - body: The failure detail wraps `detectionDetail` even though it already contains the scope, producing messages like `user (user (enabled+active))`. Store only the reason or drop the outer scope label.
  - confidence: 0.84
