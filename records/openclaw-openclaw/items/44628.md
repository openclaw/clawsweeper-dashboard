## Summary

Keep open. Current main still routes managed Linux gateway restart/status through the systemd adapter's user-scope calls, while this PR only changes the lower restart helper and also carries an unrelated Docker mount-validation regression. The branch is useful source context, but it should not merge as-is.

## What This Changes

This PR removes a Docker setup whitespace guard and adds synchronous systemctl enabled/active probes inside `triggerOpenClawRestart()` so that helper can prefer a user or system service scope with fallback.

## Best Possible Solution

Land a focused Linux systemd lifecycle fix in the daemon adapter: restart, stop, and runtime/status should try system scope only when the user unit is clearly missing/not loaded, preserve permission and user-bus errors, report the chosen scope where user-visible, and leave Docker mount validation unchanged.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to mock `execFile` so `systemctl --user restart/show openclaw-gateway.service` returns a missing-unit error while system-scope `systemctl restart/show openclaw-gateway.service` succeeds; current main still follows the user-scope adapter path for managed restart and runtime status.

## Solution Assessment

No. This PR is not the best way to solve the issue because it updates `triggerOpenClawRestart()` rather than the managed systemd lifecycle adapter, has a scope-priority bug, and bundles an unrelated Docker validation change; the safer fix belongs in `src/daemon/systemd.ts` with focused regression tests.
