## Summary

Keep open for revision: the linked systemd lifecycle bug is still source-reproducible on current main, but this PR puts scope detection in `triggerOpenClawRestart()` instead of the shared daemon lifecycle path and also weakens Docker mount validation.

## What This Changes

The PR adds Linux systemd user/system probing to `triggerOpenClawRestart()` and removes a Docker setup whitespace guard for config mount paths.

## Best Possible Solution

Move user/system scope handling into `src/daemon/systemd.ts` shared lifecycle operations, keep Docker mount validation unchanged unless compose quoting is fixed end to end, add regression tests, and require redacted real systemd proof.

## Reproduction Assessment

Yes, source-reproducible: current `openclaw gateway restart` reaches `params.service.restart`, the Linux registry maps that to `restartSystemdService`, and that path still calls `execSystemctlUser` only. I did not run a live systemd host in this read-only review.

## Solution Assessment

No. The PR improves a related direct helper, but the maintainable fix belongs in the shared daemon systemd lifecycle layer and should not include the unrelated Docker validation weakening.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move scope detection into the lifecycle restart path:** `src/infra/restart.ts:307-309`
  - body: The reported operator flow uses the gateway service abstraction and ends in `restartSystemdService`, which still calls user-scope `systemctl`. Adding detection only to `triggerOpenClawRestart()` leaves `openclaw gateway restart` unfixed for system-level units.
  - confidence: 0.9
- **[P2] Prefer active units before enabled-only units:** `src/infra/restart.ts:345-351`
  - body: After the enabled+active cases, this decision tree selects enabled-only units before active-only units. In a mixed state it can restart an inactive scope while the currently running gateway in the other scope is left untouched.
  - confidence: 0.86
- **[P2] Keep mount whitespace validation until compose quoting is fixed:** `docker-setup.sh:147`
  - body: Removing this guard lets whitespace-containing host paths pass early validation even though mount specs still reject spaces and generated compose volume entries are not structurally quoted. Revert this unrelated change or update the whole mount path consistently.
  - confidence: 0.88
- **[P3] Avoid duplicating the detected scope in failure details:** `src/infra/restart.ts:389-392`
  - body: The fallback failure detail wraps `detectionDetail` even though it already contains the scope, producing messages like `user (user (enabled+active))`. Store only the reason or drop the outer scope label in both fallback failure paths.
  - confidence: 0.84
