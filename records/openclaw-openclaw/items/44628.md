## Summary

Keep open: current main still shows the systemd gateway lifecycle path as user-scope only, but this PR is not mergeable as-is because it patches a restart helper that is not the main `openclaw gateway restart` service path, weakens unrelated Docker mount validation, and lacks real behavior proof.

## What This Changes

The PR adds systemd user/system scope probing to `triggerOpenClawRestart()` and removes Docker setup whitespace rejection for mount paths.

## Best Possible Solution

Implement system-scope detection or fallback in the shared Linux gateway service layer (`src/daemon/systemd.ts`) with focused lifecycle tests, keep Docker mount validation changes out of this PR, and require redacted real systemd proof before merge.

## Reproduction Assessment

Yes, source-reproducible without a live systemd host: `openclaw gateway restart` routes through `restartSystemdService`, which still calls user-scope systemctl only. A real system-level unit host would be needed for final runtime proof.

## Solution Assessment

No. The PR improves a related helper, but the narrow maintainable fix belongs in the shared gateway systemd lifecycle path and should avoid unrelated Docker validation changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move scope detection into the gateway service path:** `src/infra/restart.ts:307-309`
  - body: The reported `openclaw gateway restart` command does not call this helper; it goes through `runDaemonRestart` -> `resolveGatewayService().restart` -> `restartSystemdService`, which still uses `execSystemctlUser`. With scope detection only in `triggerOpenClawRestart`, the linked CLI bug remains for system-level service installs.
  - confidence: 0.9
- **[P2] Prefer active systemd units before enabled-only units:** `src/infra/restart.ts:345-351`
  - body: In a mixed state where one scope is enabled but inactive and the other is active but not enabled, this decision tree chooses the enabled-only scope first. That can restart or start the wrong unit while leaving the currently running gateway unchanged.
  - confidence: 0.86
- **[P2] Keep mount whitespace validation until compose quoting is fixed:** `docker-setup.sh:147`
  - body: Removing this guard lets whitespace-containing config/workspace paths pass early validation, but generated mounts are still emitted as plain list entries and mount specs still reject spaces. Revert the unrelated change or update the whole Docker mount path to quote/validate consistently.
  - confidence: 0.88
- **[P3] Avoid duplicating the detected scope in failure details:** `src/infra/restart.ts:390-426`
  - body: When `detectionDetail` already contains `user (enabled+active)` or `system (enabled)`, these templates produce nested labels such as `user (user (enabled+active))`. Store only the reason or drop the outer label so operator errors stay readable.
  - confidence: 0.84
