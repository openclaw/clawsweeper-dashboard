## Summary

Keep open: current main and the latest release still allow terminal TaskFlow writes where `endedAt` can be earlier than `updatedAt`, and the linked fix PR is open rather than merged.

## What This Changes

The issue reports `openclaw tasks audit` warnings for succeeded TaskFlows whose `endedAt` is earlier than `updatedAt`.

## Best Possible Solution

Land a focused write-path fix for terminal TaskFlow timestamps with regression coverage and after-fix audit proof, then close this issue from the merged PR.

## Reproduction Assessment

Yes. Source inspection shows `finishFlow`/`failFlow` can persist `endedAt < updatedAt`, and TaskFlow audit reports that exact state as `inconsistent_timestamps`; I did not run a live CLI repro in this read-only review.

## Solution Assessment

Yes in direction: normalizing terminal timestamps in the TaskFlow finish/fail write path is the narrow maintainable fix. The open linked PR still needs scope/proof review before it should resolve this issue.
