## Summary

Keep open: current main still allows terminal TaskFlow writes where `endedAt` can be earlier than `updatedAt`, the audit code reports that exact state, and the open closing PR should be reviewed, landed, or closed before this issue is resolved.

## What This Changes

The issue requests normalizing completed TaskFlow timestamps so `endedAt` is never earlier than `updatedAt` and `openclaw tasks audit` stops warning on successful flows.

## Best Possible Solution

Land a focused TaskFlow terminal timestamp guard with regression coverage and after-fix audit proof, then let the merged PR close this issue.

## Reproduction Assessment

Yes. Source inspection shows `finishFlow` and `failFlow` can persist `endedAt < updatedAt`, and TaskFlow audit reports that exact state as `inconsistent_timestamps`; I did not run a live CLI repro in this read-only review.

## Solution Assessment

Yes in direction. Normalizing terminal timestamps in the TaskFlow finish/fail write path is the narrow maintainable fix, but the linked PR should be reviewed for scope and proof before it resolves this issue.
