## Summary

Keep this open: current main still has plausible idle paths that can rebuild plugin metadata and bundled manifest registry state, and this report is newer and more specific than the older closed gateway-CPU reports.

## What This Changes

The issue reports idle gateway CPU saturation when legacy `plugins.bundledDiscovery: "compat"` is combined with a restrictive `plugins.allow` and many disabled bundled plugins.

## Best Possible Solution

Preserve legacy `compat` semantics while ensuring idle gateway paths reuse prepared plugin metadata or throttle refresh work so disabled bundled manifests are not re-walked continuously.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: the reporter provided concrete profiling steps and source inspection shows current-main paths that can rebuild manifest metadata repeatedly, but I did not live-run the gateway/profile in this review.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the live trigger is isolated; the maintainable direction is to reuse or invalidate a prepared plugin metadata snapshot in idle health/read-only/provider paths, not to make users switch away from `compat` as the only fix.
