## Summary

This should stay open: the report is a concrete gateway performance bug, and current main still has compat provider-discovery paths that can rebuild manifest-registry state instead of consistently using prepared plugin metadata.

## What This Changes

Fix a reported gateway idle CPU hot loop where `plugins.bundledDiscovery: "compat"` with a restrictive `plugins.allow` repeatedly scans bundled plugin manifests.

## Best Possible Solution

Keep compat semantics, but remove broad manifest-registry rediscovery from gateway hot paths by threading the existing startup `PluginMetadataSnapshot` or prepared manifest registry through provider/catalog/status resolution.

## Reproduction Assessment

Yes for source-level triage: the reporter supplied exact config/profile steps, and current source shows compat provider catalog resolution can reload the installed manifest registry when no prepared manifest registry is supplied. I did not live-profile current main during this read-only review.

## Solution Assessment

No, changing the default again or adding a persistent discovery cache is not the best first fix. The narrow maintainable path is to reuse existing gateway/plugin metadata snapshots or prepared runtime facts while preserving the documented `compat` behavior.

## Repair Work Prompt

Fix the gateway idle CPU hot loop reported in https://github.com/openclaw/openclaw/issues/79806 for `plugins.bundledDiscovery: "compat"` plus restrictive `plugins.allow`. Preserve the documented compat semantics and the new-config default of `allowlist`; do not add a persistent discovery/manifest-registry cache and do not perform startup/load-time config migration. Instead, reuse existing prepared plugin metadata such as `PluginMetadataSnapshot`, `manifestRegistry`, `PluginLookUpTable`, or prepared runtime facts so provider catalog/status/session/model paths do not repeatedly call broad manifest-registry discovery when the gateway already has a valid startup snapshot. Add focused regression coverage that repeated compat provider/catalog resolution under restrictive `plugins.allow` reuses a supplied/current manifest registry and does not repeatedly call `loadPluginManifestRegistryForInstalledIndex` or broad plugin discovery. Include a user-facing changelog fix if behavior changes outside tests/internal plumbing.
