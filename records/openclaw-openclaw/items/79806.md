## Summary

Current `main` still has compat-mode paths that can force broad bundled provider discovery outside the configured `plugins.allow`; the report is recent, detailed, and not covered by a merged fix or duplicate I could verify.

## What This Changes

Report high idle gateway CPU/event-loop stalls when legacy `plugins.bundledDiscovery: "compat"` repeatedly scans bundled plugin manifests despite a restrictive plugin allowlist.

## Best Possible Solution

Keep the issue open and fix the compat discovery path so the gateway prepares or reuses one manifest/metadata view per startup or config change instead of re-walking disabled bundled manifests during idle or request-time health/provider paths.

## Reproduction Assessment

Yes for a maintainer with the reported setup: install v2026.5.7/current main with a restrictive `plugins.allow`, set `plugins.bundledDiscovery: "compat"`, start the gateway idle, and compare a CPU profile against `allowlist`. I did not run that live here because the review is read-only, `pnpm` is unavailable, and the report depends on a macOS gateway profile, but the named source frames match current code paths.

## Solution Assessment

Yes: the narrow maintainable direction is to move compat provider discovery onto the gateway's prepared metadata snapshot or another explicit lifecycle-owned registry, with tests proving disabled bundled manifests are not repeatedly scanned. Changing the default alone would not fix migrated legacy configs that doctor intentionally stamps as `compat`.
