Keep open: the report is detailed, recent, and maps to current source paths where idle health/read-only flows can still fall back to plugin metadata and installed-registry scans; related snapshot-reuse PRs are open but not merged or explicitly paired with this issue.

## What This Changes

The issue asks OpenClaw to stop pegging idle Gateway CPU when legacy `plugins.bundledDiscovery: "compat"` repeatedly scans bundled/installed plugin manifests despite a restrictive `plugins.allow`.

## Best Possible Solution

Keep the issue open and fix the hot idle/read-only/provider discovery path to reuse an already-compatible Gateway `PluginMetadataSnapshot` or `PluginLookUpTable`, with explicit invalidation, without adding a hidden wall-clock metadata cache or changing plugin policy as a side effect.

## Reproduction Assessment

Yes, source-level: the reporter gives a concrete v2026.5.7 profile and workaround, and current source still has idle health/read-only paths that can fall back to synchronous plugin metadata and installed-registry scans. I did not live-profile the Gateway in this read-only review.

## Solution Assessment

Unclear whether the already-open snapshot-reuse PRs fully solve this compat-mode idle case. The safest direction is a narrow reuse/invalidation fix through the existing Gateway metadata snapshot seam, not a new TTL cache or a default-policy change bundled into the bug fix.
