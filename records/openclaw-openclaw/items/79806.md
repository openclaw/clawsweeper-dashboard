## Summary

Keep open. The report is recent, detailed, and not covered by an obvious merged duplicate; current main still contains plugin metadata paths that can scan every installed bundled manifest under compat-style discovery, matching the reported CPU profile enough that closing would be premature.

## What This Changes

The issue reports a Gateway idle CPU hot loop when legacy `plugins.bundledDiscovery: "compat"` repeatedly walks bundled plugin manifest/registry metadata despite a restrictive plugin allowlist.

## Best Possible Solution

Instrument the Gateway idle/health/runtime metadata path under compat, then remove the repeated disabled-bundled manifest scan by reusing the prepared plugin metadata snapshot or narrowing the caller to needed plugin IDs, with regression coverage for compat plus a restrictive allowlist.

## Reproduction Assessment

No, we do not have a high-confidence live current-main reproduction from this review. The reporter provided a strong v2026.5.7 profile and source inspection shows matching manifest-registry and disabled-plugin scan paths, but the exact repeated idle caller still needs a current-main profile or trace.

## Solution Assessment

No patch is under review, so the final solution is not yet proven. The narrowest maintainable direction is to keep compat semantics while avoiding repeated broad manifest discovery on idle/hot paths, rather than changing the public config default as the first move.
