## Summary

Current main already addresses the reported idle manifest-registry cost by building a plugin metadata snapshot once during gateway startup and reusing it on the read-only channel and model-catalog paths that previously reloaded manifest discovery work.

## What This Changes

The issue reports sustained gateway CPU saturation in `plugins.bundledDiscovery: "compat"` when a restrictive plugin allowlist leaves many bundled plugins disabled.

## Best Possible Solution

Keep the current main implementation and let the next release carry it; operators on v2026.5.7 can use the documented `plugins.bundledDiscovery: "allowlist"` workaround until upgrading.

## Reproduction Assessment

No high-confidence current-main repro remains from source inspection. The report targets v2026.5.7, and current main now reuses a startup plugin metadata snapshot on the gateway/read-only/model-catalog paths instead of reloading manifest discovery work.

## Solution Assessment

Yes. Moving the manifest registry fact into startup metadata and passing that prepared fact through existing consumers is the maintainable fix direction for this hot path, and it matches the repo guardrails against broad request-time discovery.
