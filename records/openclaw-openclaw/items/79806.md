## Summary

Keep open: the report is fresh against v2026.5.7, current main has no relevant source-side plugin/gateway fix after that release, and source still shows repeated metadata compatibility paths that can synchronously restat every installed plugin manifest/package file, including disabled bundled plugins.

## What This Changes

Investigate and fix idle gateway CPU pegging when legacy `plugins.bundledDiscovery: "compat"` repeatedly walks bundled plugin manifest metadata despite a restrictive plugin allowlist.

## Best Possible Solution

Preserve `compat` semantics but move hot compatibility checks onto an owned snapshot/lookup-table or precomputed inventory fingerprint so idle runtime paths do not repeatedly rescan disabled bundled plugin manifests.

## Reproduction Assessment

Yes, source/profile-assisted: run v2026.5.7 or current main with `plugins.bundledDiscovery: "compat"`, a restrictive `plugins.allow`, and many disabled bundled plugins, then profile an idle gateway. The reporter supplied that profile, and current source still contains the matching sync metadata/fingerprint path.

## Solution Assessment

Yes for a narrow performance fix: eliminate repeated metadata scans through explicit snapshot reuse or precomputed fingerprint data while keeping legacy `compat` behavior intact. Changing the default or adding a wall-clock cache is less clearly maintainable under the current plugin cache contract.
