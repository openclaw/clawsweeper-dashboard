## Summary

Current main and the latest release still have the stale Anthropic OAuth `claude-cli/2.1.75` headers, so this PR is still aimed at a real open bug. It should stay open, but it is not merge-ready because the new resolvers use bare `require` in ESM modules, retain the rejected stale fallback, lack patched OpenClaw runtime-header proof, and GitHub reports the branch as conflicting.

## What This Changes

The PR replaces hardcoded Anthropic Claude Code OAuth user-agent version constants with per-file runtime package-version resolvers and tightens one transport-stream user-agent assertion.

## Best Possible Solution

Land one canonical shared, ESM-safe Claude Code user-agent source for both Anthropic OAuth builders, avoid silently emitting `2.1.75`, add exact header assertions, and require redacted patched OpenClaw runtime proof.

## Reproduction Assessment

Yes for the source-level path: current main and `v2026.6.9` still construct both Anthropic OAuth user-agent headers from `2.1.75`, matching the linked bug's stale-header report. I did not run a credentialed Anthropic OAuth request in this read-only review, so the upstream rejection itself remains unverified here.

## Solution Assessment

No. Resolving the installed package version is a plausible direction, but this implementation duplicates the resolver, uses bare `require` in ESM modules, keeps the rejected stale fallback, and adds a semver-only assertion that still accepts the bad version.
