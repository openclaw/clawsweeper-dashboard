## Summary

Keep open: the underlying Anthropic OAuth bug is real, but this branch is not merge-ready because both added resolvers use bare `require` in ESM modules and can fall back to the stale rejected `2.1.75` header, while the posted proof does not show patched OpenClaw emitting the corrected header.

## What This Changes

The PR replaces both hardcoded Anthropic OAuth Claude CLI user-agent constants with duplicated runtime package-version resolvers and tightens one managed-transport user-agent test.

## Best Possible Solution

Land one canonical shared ESM-safe Claude Code user-agent source for both Anthropic OAuth builders, avoid silently emitting `2.1.75`, add exact non-stale header assertions for both paths, and require redacted patched OpenClaw runtime proof.

## Reproduction Assessment

Yes for the source-level bug and PR defect: current main and `v2026.6.9` still construct both Anthropic OAuth user-agent headers from `2.1.75`, and a Node ESM probe shows bare `require` is undefined. I did not run a credentialed Anthropic request.

## Solution Assessment

No. Resolving a current Claude Code user-agent is the right direction, but this implementation duplicates an ESM-broken resolver, keeps the rejected stale fallback, and adds a regex assertion that still accepts the bad version.
