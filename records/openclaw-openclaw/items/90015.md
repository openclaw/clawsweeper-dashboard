## Summary

Keep open: current main and v2026.6.8 still lack a cap for an opened-but-unclosed DeepSeek DSML recovery block, and the linked fix PR is still open and unmerged.

## What This Changes

The issue reports that `createDeepSeekDsmlToolCallRecoverer` buffers unclosed DeepSeek DSML text without the 256 KB cap used by sibling stream buffers.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/86637 with a byte-counted DSML recoverer cap aligned with sibling stream caps and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main appends to the DSML recovery buffer and only clears an opened-but-unclosed DSML block at final flush, with no cap before then.

## Solution Assessment

Yes in direction. The existing recoverer is the right owner path, but the final fix should count UTF-8 bytes to match sibling caps and remain open until the linked PR or a replacement lands.
