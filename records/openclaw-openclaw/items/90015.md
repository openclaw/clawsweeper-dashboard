## Label Justifications

- `P1`: The issue describes an uncapped provider-controlled buffer in core agent streaming that can exhaust gateway memory and affect live agent availability.
- `impact:crash-loop`: The reported failure mode is unbounded gateway memory growth, which is a process availability and possible OOM/crash class problem.

## Summary

Keep open: current main and the latest release still have the uncapped DeepSeek DSML recovery buffer, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks for a bounded DSML recovery buffer in `createDeepSeekDsmlToolCallRecoverer()` so unclosed provider-emitted DSML blocks cannot grow gateway memory indefinitely.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/86637 so the DSML recoverer enforces a byte-counted 256 KB cap while preserving normal DSML tool-call recovery.

## Reproduction Assessment

Yes, source-reproducible: after a DeepSeek DSML open token, current main appends to `buffer` across chunks and returns without clearing it when no close token is present. I did not run a synthetic stream because this review is read-only.

## Solution Assessment

Yes in direction: the narrow maintainable fix is a byte-counted cap in the existing DSML recoverer, aligned with sibling stream caps. The issue should stay open until the linked PR lands or a replacement provides that fix.
