## Label Justifications

- `P1`: The issue describes an uncapped provider-controlled buffer in the core agent stream that can exhaust gateway memory and affect live agent availability if triggered.
- `impact:crash-loop`: The reported failure mode is process memory growth in the gateway, which is an availability and possible crash/OOM class problem.

## Summary

Keep open: current main still has an uncapped DeepSeek DSML recovery buffer, and the linked implementation PR is open rather than merged, so the issue should continue tracking review and landing.

## What This Changes

The issue asks to cap DeepSeek DSML tool-call recovery buffering for unclosed DSML blocks in `src/agents/openai-transport-stream.ts`.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/86637 so the DSML recoverer has a bounded byte cap while preserving normal DSML tool-call recovery.

## Reproduction Assessment

Yes, source-reproducible: after a DSML open token, `consume(false)` returns when no close token is found, while later `push()` calls keep appending to the same buffer. I did not execute a synthetic stream because this review is read-only.

## Solution Assessment

Yes: adding a byte cap in the existing DSML recoverer is the narrowest maintainable fix and matches the surrounding stream buffer limits. The issue should stay open until the linked PR is merged or replaced.
