## Summary

Keep this issue open: current main and the latest release still reject `messages.groupChat.historyLimit: 0`, while the docs and sibling history-limit schemas define `0` as the disable value. The same reporter has an open focused fix PR, so the right cleanup path is to review and land that PR rather than close the issue early.

## What This Changes

The issue requests changing `GroupChatSchema.historyLimit` from positive-only validation to nonnegative validation so `0` can disable group chat history consistently.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/65359 or an equivalent narrow schema-and-regression-test fix, then close this issue once the fix reaches main.

## Reproduction Assessment

Yes, at source level. Current main validates `messages.groupChat.historyLimit` through `GroupChatSchema`, which uses Zod `.positive()` even though docs and sibling schemas define `0` as valid; I did not run a live Gateway reload because this review was read-only.

## Solution Assessment

Yes. Changing the single `GroupChatSchema.historyLimit` constraint to `.min(0)` and adding Zod plus Gateway `config.patch` regression coverage is the narrowest maintainable fix, and the open linked PR follows that shape.
