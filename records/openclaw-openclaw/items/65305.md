## Summary

Keep open: current main still rejects `messages.groupChat.historyLimit: 0` through `GroupChatSchema`, the docs define `0` as disabling group history, and an open linked PR already owns the intended fix path.

## What This Changes

Allow group-chat history-limit config to accept `0`, matching the documented disabled-history behavior and sibling history-limit schemas.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/65359 or an equivalent current-main patch that changes `GroupChatSchema.historyLimit` to `.min(0)`, adds focused boundary coverage, and verifies the runtime JSON Schema/config patch path accepts zero.

Yes. A high-confidence source-level path exists: both global and per-agent group-chat config validate through `GroupChatSchema`, whose `.positive()` validator rejects `0`; `config.patch` also validates the merged config through the same validator.

## Solution Assessment

Yes. The narrow maintainable fix is to change the single group-chat validator to `.min(0)` and add boundary coverage; the broader silent reload/error-surfacing behavior is a separate UX issue.
