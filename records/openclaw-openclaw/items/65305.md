## Summary

Keep open: current main still validates group-chat history with a strictly positive Zod number even though docs and sibling history-limit schemas allow `0`; the same-author open PR that closes this issue should be reviewed, updated, or replaced before the issue is closed. The latest issue comment also points to the dashboard/config.patch JSON Schema path as part of the same remaining alignment work.

## What This Changes

Allow `messages.groupChat.historyLimit` and per-agent `groupChat.historyLimit` to accept `0` so group history can be disabled consistently.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/65359 or an equivalent current-main patch that changes the validator to `.min(0)`, aligns runtime/base JSON Schema output, and adds focused boundary coverage.

## Reproduction Assessment

Yes. Current main source still routes both global and per-agent group-chat config through `GroupChatSchema`, whose `historyLimit` validator rejects `0`; this read-only review did not run a live config reload.

## Solution Assessment

Yes. The narrow maintainable fix is to accept `0` in the group-chat validator and ensure the generated/runtime schema consumers expose the same `minimum: 0` contract; silent reload reporting is a separate UX concern.
