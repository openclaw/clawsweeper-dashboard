## Summary

Keep open. Current main still relies on inherited process PATH for the Crestodian Claude Code detection path, while the Windows Scheduled Task gateway path deliberately avoids writing PATH into `gateway.cmd`; the existing `agents.defaults.cliBackends.claude-cli.command` escape hatch is documented, but the Windows npm-global auto-detection gap remains actionable.

## What This Changes

The issue asks OpenClaw to detect Windows npm-global Claude Code installs and avoid gateway restart instability after PATH workarounds.

## Best Possible Solution

Add a narrow Windows command-discovery path for Claude Code that checks known npm global locations or persists the resolved binary into `agents.defaults.cliBackends.claude-cli.command`, then keep any remaining restart-loop behavior tracked with the Windows gateway restart work.

## Reproduction Assessment

Yes, at source level. On Windows when the OpenClaw or Gateway process PATH lacks the npm global directory, current main probes only `claude` through the inherited process PATH and has no fallback to `%APPDATA%\npm\claude.cmd`.

## Solution Assessment

No for the proposed new `models.claude-code.path` key; the existing `agents.defaults.cliBackends.claude-cli.command` contract is the maintainable escape hatch. The better fix is a narrow Windows resolver plus regression coverage, with restart-loop changes coordinated separately if needed.
