## Summary

Keep open. Current main still defaults the bundled Claude CLI backend to a bare `claude` command while Windows scheduled-task gateway scripts intentionally avoid carrying `PATH`, so the reported `%APPDATA%\npm` discovery gap remains plausible and actionable.

## What This Changes

The issue asks OpenClaw to detect globally npm-installed Claude Code on Windows and avoid gateway instability when the scheduled-task environment lacks the user's npm PATH.

## Best Possible Solution

Resolve Windows Claude CLI discovery through the existing `claude-cli` backend contract, keep the gateway service path handling stable, and update doctor/docs so users get the exact supported override when automatic discovery cannot work.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear current-main path for the PATH mismatch, but this review did not run a live Windows scheduled-task reproduction or reproduce the manual `gateway.cmd` restart loop.

## Solution Assessment

Is this the best way to solve the issue? The supported direction should reuse `agents.defaults.cliBackends.claude-cli.command` and/or add Windows npm shim discovery; adding a new `models.claude-code.path` key would duplicate the existing config contract.
