## Summary

Keep open. Current main still probes `claude` through the running process PATH for the Crestodian/setup model surface, while the Windows service code intentionally avoids persisting PATH into `gateway.cmd`; the docs and doctor already expose the correct `agents.defaults.cliBackends.claude-cli.command` escape hatch, but the reported Windows npm-global auto-detection gap remains actionable.

## What This Changes

The issue asks OpenClaw to detect Windows npm-global Claude Code installs for setup/Gateway use, or provide a stable binary-path workaround instead of relying on Scheduled Task PATH inheritance.

## Best Possible Solution

Teach the Windows command-detection path to resolve Claude Code from known npm global locations or to persist the resolved executable into the existing `agents.defaults.cliBackends.claude-cli.command` setting, while keeping `models.*` config untouched and documenting the Windows Scheduled Task behavior.

## Reproduction Assessment

Yes, at source level: on Windows when the OpenClaw/Gateway process PATH lacks the npm global directory, `probeLocalCommand("claude")` has no fallback to `%APPDATA%\npm\claude.cmd`, so the reported `Claude Code: not found` state follows. This was not live-run under Windows Scheduled Tasks in this review.

## Solution Assessment

Unclear for the restart-loop half, but the detection fix should reuse the existing CLI backend command contract rather than adding a new `models.claude-code.path` key. A narrow Windows resolver plus regression coverage is the safest maintainable path.
