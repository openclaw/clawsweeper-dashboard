## Summary

Keep open. Current main still relies on PATH-only `claude` probing for Crestodian setup/model status, while Windows service rendering deliberately omits PATH from `gateway.cmd`; the documented `agents.defaults.cliBackends.claude-cli.command` escape hatch exists, but the npm-global Windows auto-detection gap and the reported restart loop are not fixed on main.

## What This Changes

Detect globally npm-installed Claude Code on Windows and keep the Gateway stable when Scheduled Task PATH lacks the npm global prefix.

## Best Possible Solution

Add Windows-aware Claude CLI discovery using the existing CLI backend command contract, and keep the restart-loop cleanup in the narrower Windows restart path or the related PR rather than mutating `gateway.cmd` PATH blindly.

## Reproduction Assessment

Yes, at source level: when a Windows OpenClaw/Gateway process lacks `%APPDATA%\npm` on PATH, `probeLocalCommand("claude")` has no fallback to `claude.cmd`, so Claude Code is reported as not found. I did not live-run a Windows Scheduled Task in this read-only review.

## Solution Assessment

No for adding a new `models.claude-code.path` key; the maintainable fix should reuse `agents.defaults.cliBackends.claude-cli.command` and/or resolve known Windows npm-global locations. The restart-loop half should be handled by focused Windows restart logic, not by freezing PATH into `gateway.cmd`.
