## Summary

Keep open. Current `main` still relies on PATH-only Claude command probing for the setup/model surface, while the Windows Scheduled Task service path intentionally avoids writing PATH into `gateway.cmd`; the existing `agents.defaults.cliBackends.claude-cli.command` escape hatch is documented, but the reported npm-global Windows auto-detection gap and fresh restart-loop report remain unresolved.

## What This Changes

The issue asks OpenClaw to detect a Windows npm-global Claude Code install and avoid or document the Gateway restart loop after PATH workarounds.

## Best Possible Solution

Add a Windows-aware Claude CLI discovery path that resolves known npm-global shims or persists an explicit `agents.defaults.cliBackends.claude-cli.command`, and handle the restart-loop behavior through the existing Windows restart-health path or the related restart PR after concrete logs confirm the cause.

## Reproduction Assessment

Partially yes: source inspection shows `claude` detection fails whenever the OpenClaw/Gateway process PATH lacks the npm-global directory, because the probe has no Windows npm fallback. The Gateway restart loop itself is still unclear without Windows Scheduled Task logs or a live Windows reproduction.

## Solution Assessment

No for adding a new `models.claude-code.path` key; the maintainable path is to reuse the existing `agents.defaults.cliBackends.claude-cli.command` contract or resolve the Windows npm shim before probing. The restart-loop fix should stay in the Windows daemon restart-health/service code rather than mutating `gateway.cmd` by hand.
