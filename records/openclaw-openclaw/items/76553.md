## Best Possible Solution

Add a narrow Windows Claude CLI resolver that uses the existing CLI backend command contract, and handle the restart-loop path through the Windows restart-health/service code or the open restart PR.

## Repair Work Prompt

Fix the Windows Claude Code discovery bug from https://github.com/openclaw/openclaw/issues/76553. Current main probes `claude` via `src/crestodian/overview.ts` -> `src/crestodian/probes.ts`, and `probeLocalCommand()` only uses the process PATH; Windows Scheduled Tasks may not include the npm global directory even when `%APPDATA%\npm\claude.cmd` exists. Add a narrow Windows resolver for Claude CLI discovery, preferably shared with `src/commands/doctor-claude-cli.ts` and `src/infra/executable-path.ts`, that checks safe known npm-global locations derived from `APPDATA`, `LOCALAPPDATA`, and/or `USERPROFILE` or reuses/persists the existing `agents.defaults.cliBackends.claude-cli.command` contract. Do not add `models.claude-code.path`, do not persist ambient PATH into `gateway.cmd`, do not enable shell fallback for arbitrary command resolution, and do not mutate unrelated restart behavior. Add regression coverage for PATH missing the npm global dir while `%APPDATA%\npm\claude.cmd` is present. If user-facing behavior or docs change, add a single-line `CHANGELOG.md` fix entry and update the Claude CLI docs. Treat restart-loop stabilization as related context tracked by #50136 unless the same narrow command-resolution change directly affects it.
