## Summary

Keep open. Current main still detects Claude Code through the running process PATH, while Windows service startup intentionally avoids persisting PATH into `gateway.cmd`; the restart-loop half also overlaps open PR #50136 rather than being resolved on main.

## What This Changes

Request Windows detection of npm-global Claude Code and safer Gateway behavior when Scheduled Tasks do not see the user npm PATH.

## Best Possible Solution

Add Windows-aware Claude CLI discovery or setup-time persistence into `agents.defaults.cliBackends.claude-cli.command`, and resolve the restart cleanup risk through the existing Windows restart path rather than adding `models.*` keys.

## Reproduction Assessment

Yes, at source level for the detection half: `loadCrestodianOverview()` calls `probeLocalCommand("claude")`, which directly spawns the bare command, so a Windows task environment without `%APPDATA%\npm` reports not found. The restart loop was not live-reproduced here, but current restart code still has Windows stale-listener cleanup paths matching the report.

## Solution Assessment

Unclear for the user's manual PATH edit, which is not the best fix. The maintainable path is to resolve or persist the CLI binary through the existing `cliBackends` contract and tighten restart cleanup, not add a new `models.claude-code.path` key.
