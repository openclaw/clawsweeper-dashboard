## Summary

Keep open. Current main still has a source-level Windows Claude Code detection gap: the user-facing model probe spawns bare `claude` from the process PATH, while the Windows Gateway task deliberately avoids persisting PATH into `gateway.cmd`; the documented escape hatch exists, but npm-global auto-detection is not implemented.

## What This Changes

The issue asks OpenClaw to detect npm-installed Claude Code from Windows user-global locations or provide a stable registration path, and to avoid Gateway restart loops after PATH workarounds.

## Best Possible Solution

Resolve Claude CLI command discovery through the existing CLI backend command contract: add Windows-aware npm-global resolution or persist the resolved executable into `agents.defaults.cliBackends.claude-cli.command`, document Scheduled Task PATH behavior, and leave broad restart cleanup to the dedicated gateway restart work.

## Reproduction Assessment

Yes at source level: when the Windows OpenClaw/Gateway process PATH lacks the npm global directory, the current `probeLocalCommand("claude")` path has no `%APPDATA%\npm\claude.cmd` fallback and reports `not found`. I did not live-run Windows Scheduled Tasks in this review.

## Solution Assessment

Yes for a narrow resolver or setup-time persistence using `agents.defaults.cliBackends.claude-cli.command`. Adding a new `models.*` key or editing `gateway.cmd` PATH directly is not the best path because current docs and tests already establish a different contract.
