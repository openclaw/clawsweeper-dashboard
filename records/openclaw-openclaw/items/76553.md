## Summary

Keep open. Current main still uses a direct `claude` process probe for the visible model-status path, while Windows service code intentionally avoids persisting PATH into `gateway.cmd`; the existing documented escape hatch is `agents.defaults.cliBackends.claude-cli.command`, so this remains a narrow Windows command-discovery bug plus a still-unverified restart-loop report.

## What This Changes

Request that OpenClaw detect Windows npm-global Claude Code installs and avoid or diagnose gateway restart loops caused by PATH workarounds.

## Best Possible Solution

Add a Windows-safe Claude CLI resolver or setup repair that records the resolved path under the existing CLI backend command setting, while leaving broader restart stabilization to #50136 or a dedicated Windows reproduction.

## Reproduction Assessment

Yes, at source level for the detection half: with a Windows process PATH that lacks the npm global directory, `loadCrestodianOverview()` calls `probeLocalCommand("claude")` and reports `not found`. The Scheduled Task restart loop is not yet live-reproduced from the provided data.

## Solution Assessment

Unclear for the restart-loop half, but the detection fix should reuse `agents.defaults.cliBackends.claude-cli.command` and Windows command resolution rather than adding `models.*` config. The narrow maintainable path is resolver/doctor/setup coverage plus tests, not a durable PATH mutation in `gateway.cmd`.

Focus on the existing Claude CLI backend path: when resolving/probing `claude` on Windows, account for npm global locations such as `%APPDATA%\npm` and `.cmd` shims, or persist a resolved executable into `agents.defaults.cliBackends.claude-cli.command` through the existing setup/doctor contract.
