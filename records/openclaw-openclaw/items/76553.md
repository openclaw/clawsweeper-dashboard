## Summary

Keep open. Current main still checks Claude Code through a bare `claude` process lookup, while the Windows Scheduled Task gateway path intentionally avoids persisting PATH into `gateway.cmd`; the existing `agents.defaults.cliBackends.claude-cli.command` escape hatch is documented, but the Windows npm-global detection gap and the reporter's follow-up restart loop remain unresolved.

## What This Changes

The issue asks OpenClaw to detect globally npm-installed Claude Code on Windows and avoid or document Gateway restart loops when the Scheduled Task environment lacks the user npm PATH.

## Best Possible Solution

Resolve the Windows Claude Code discovery path through the existing CLI backend command contract, then separately preserve or route the Gateway restart-loop evidence to the Windows restart-health work.

## Reproduction Assessment

Yes for the detection half at source level: when the Gateway process PATH lacks the npm global directory, the current `claude` probe has no Windows npm-global fallback and reports not found. The restart-loop half is not yet high-confidence reproduced from the available logs.

## Solution Assessment

Unclear for the combined issue. The likely maintainable fix for detection is a narrow Windows resolver or persisted `agents.defaults.cliBackends.claude-cli.command` path, while the restart-loop behavior should be validated against #50136 rather than hidden behind PATH mutation.
