## Summary

Keep open: the patch is a narrow improvement for paths that already populate `output.errorMessage`, but current `main` shows the Google/Gemini finishReason path named in the PR still reaches the generic fallback because it only sets `stopReason`.

## What This Changes

The PR changes transport finalization throws in the shared helper and OpenAI Responses transports to prefer `output.errorMessage` and adds shared-helper propagation/fallback tests.

## Best Possible Solution

Preserve provider finishReason details before shared finalization, especially in the Google transport, while keeping the existing fallback for truly missing messages.

## Reproduction Assessment

Yes, from source inspection: a Google stream with `finishReason: "MALFORMED_FUNCTION_CALL"` maps to `stopReason: "error"`, calls `finalizeTransportStream`, and currently has no `output.errorMessage` to propagate. I did not run the test suite in this read-only review.

## Solution Assessment

No, not as-is for the stated Gemini repro. Using `output.errorMessage` at the throw sites is the right narrow pattern, but the Google transport also needs to populate that field from non-success finish reasons before finalization.
