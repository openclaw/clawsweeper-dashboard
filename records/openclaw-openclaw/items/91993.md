## Summary

Keep this PR open for revision: it addresses a source-confirmed ghost-entry leak, but unconditional non-idle TTL eviction can reset legitimate processing state before its next progress event, and the supplied evidence is test-only rather than real Gateway behavior proof.

## What This Changes

The PR changes diagnostic session pruning to delete any entry older than the TTL and adds regression coverage for stale non-idle entries.

## Best Possible Solution

Keep the leak fix, but evict a stale non-idle entry only when existing activity ownership proves that no embedded run, model call, or tool work remains; cover both orphan cleanup and progress-after-TTL state preservation.

## Reproduction Assessment

Yes, from source inspection. Failed recovery can leave a non-idle entry that current pruning never removes; the patch regression is also reproducible by aging a processing entry past the TTL and invoking a progress path that performs prune-before-lookup.

## Solution Assessment

No. Applying the idle-state TTL to every entry is broader than the failed-recovery ghost case; an orphan-aware check is the narrower maintainable fix.

- **[P1] Do not evict live non-idle state before progress lookup:** `src/logging/diagnostic-session-state.ts:61-62`
  - body: `getDiagnosticSessionState()` calls this prune before looking up the entry. Gate non-idle eviction on absence of active work or another orphan signal, and add a regression test that sends progress after the TTL.
