## Label Justifications

- `P2`: The issue is a bounded process-local diagnostic retention bug with limited operational blast radius.
- `merge-risk: 🚨 session-state`: The proposed queue-depth-only TTL gate can delete live non-idle diagnostic state and lose associated generation and history metadata.

## Summary

The reported retention path is real, but the revised condition still conflates queue backlog with active runtime work. Diagnostics intentionally track embedded, model, and tool activity separately, so a live non-idle session can have zero queue depth; deleting it after the TTL can reset its generation and diagnostic metadata while work remains active.

## What This Changes

The PR changes TTL pruning to remove any stale diagnostic session state with non-positive queue depth and adds unit tests for ghost eviction and queued-work preservation.

## Best Possible Solution

Clean up the entry at the recovery lifecycle boundary when the outcome proves the tracked state is orphaned, or add an explicit orphan or terminal signal consumed by pruning; do not use queue depth alone as the active-work authority.

## Reproduction Assessment

Yes at source level: failed recovery can leave non-idle state unchanged, but the contributor did not demonstrate the full current-main Gateway path or establish that zero queue depth means no active work.

## Solution Assessment

No. The narrow maintainable fix should retire state where recovery owns terminal or orphan knowledge, because the diagnostic model deliberately tracks active work separately from queue backlog.
