The PR optimizes task-registry listing and maintenance sweeps, then adds `tasks.retentionMs` and `tasks.sweepIntervalMs` config support.

## Best Possible Solution

Keep the optimization, but route task config through the shared maintenance/config path, update the public task docs and changelog, then run focused task/config validation before marking the PR ready.

Do we have a high-confidence way to reproduce the issue? Source inspection confirms the clone-heavy listing path and hardcoded defaults on current main, but I did not run a live gateway benchmark in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the core optimization is directionally sound, but the config application needs to cover manual maintenance and the public docs/changelog need to match the new behavior.
