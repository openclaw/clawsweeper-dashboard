Full review comments:

- **[P2] Bound sweep intervals before scheduling:** `src/tasks/task-registry.maintenance.ts:1103`
  - body: `tasks.sweepIntervalMs` now accepts any positive integer and this line passes the resolved value straight to Node's timer. Node coerces delays above `2147483647` ms to `1` ms, so a valid config such as a 30-day interval would turn the maintenance sweep into a 1 ms loop instead of reducing cadence; please cap/reject values above the timer maximum or schedule long waits safely.
  - confidence: 0.94
- **[P2] Correct invalid config fallback wording:** `docs/automation/tasks.md:352`
  - body: The docs say invalid `tasks.*` values are rejected at config-load time while leaving defaults in effect, and the changelog repeats that contract. Config loading throws `INVALID_CONFIG` on validation failure, so user-written invalid `openclaw.json` values must be fixed before load succeeds; defaults apply when the keys are omitted or a rejected config write never reaches disk.
  - confidence: 0.88

  - body: This step still says cleanup stamping is `endedAt + 7 days`, but the PR makes `cleanupAfter` use the configured `tasks.retentionMs`. Please describe the configured retention window with the 7-day default.
