## Summary

Keep open: this is useful draft work, but it is not merge-ready because the new task retention config is only applied through scheduled gateway startup while manual task maintenance can still use defaults, and the public docs/changelog do not match the new config surface.

## What This Changes

The PR optimizes task registry listing and maintenance sweeps, adds `tasks.retentionMs`/`tasks.sweepIntervalMs` config schema/types/runtime state, and passes gateway startup config into scheduled task maintenance.

## Best Possible Solution

Land the optimization after task config is applied consistently in scheduled and manual maintenance, the task docs and changelog describe the new knobs, and focused task/config validation passes.

## Reproduction Assessment

Yes for the source-level issue: current main shows clone-heavy listing, hardcoded retention/sweep defaults, and repeated maintenance listing paths. I did not live-reproduce the reported CPU profile or run benchmarks in this read-only review.

## Solution Assessment

No, not yet. The optimization direction is reasonable, but the config application should be shared with the manual maintenance path and the public docs/changelog must be updated before merge.
