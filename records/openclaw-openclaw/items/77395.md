## Summary

Keep open; the PR direction is useful, but the branch still has blocking config-contract issues: an unbounded sweep interval can become a 1ms loop under Node, and docs/changelog still misstate invalid config behavior.

## What This Changes

The PR optimizes task registry listing/maintenance, adds `tasks.retentionMs` and `tasks.sweepIntervalMs` config with schema/docs/types/tests, and wires the config through gateway startup and CLI maintenance.

## Best Possible Solution

Keep the PR open and make `tasks.sweepIntervalMs` timer-safe while aligning docs and changelog with the actual schema rejection contract before merge.

## Reproduction Assessment

Yes for the PR findings. Source shows the configured sweep interval flows into `setInterval`, a Node probe reproduced overflow-to-1ms behavior, and the current config loader throws on schema failure.

## Solution Assessment

No, not yet. The optimization and config seam are maintainable, but the config boundary needs a timer-safe upper bound and the invalid-value wording needs to match startup validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cap task sweep intervals before scheduling:** `src/tasks/task-registry.maintenance.ts:1100`
  - body: `tasks.sweepIntervalMs` is accepted as any positive integer and then passed directly to `setInterval`. Node coerces delays above `2147483647` to `1` ms, so a user trying to raise the sweep cadence past about 24.8 days can accidentally create a near-continuous maintenance loop. Please reject or safely handle timer-overflow values in the schema/runtime config and update the generated docs/tests accordingly.
  - confidence: 0.92
- **[P2] Correct invalid config fallback wording:** `docs/automation/tasks.md:352`
  - body: The docs say invalid `tasks.retentionMs` / `tasks.sweepIntervalMs` values are rejected at load time while leaving defaults in effect, and the changelog repeats that contract. The config loader throws on schema validation failure, so users do not get a successful load with defaults; please say defaults apply when the keys are omitted, and invalid values must be fixed before config validation/startup succeeds.
  - confidence: 0.9
