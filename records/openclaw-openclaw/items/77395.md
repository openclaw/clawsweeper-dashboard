## Summary

Keep open: the PR is a focused task-registry performance/config change with sufficient terminal proof and no security concern, but it should not merge as-is because the new task retention config is not applied on the standalone CLI cancel path before that path stamps `cleanupAfter`.

## What This Changes

The branch optimizes task registry listing and maintenance sweeps, adds `tasks.retentionMs` and `tasks.sweepIntervalMs` config/docs/types/baselines, and wires gateway plus CLI maintenance to the runtime task config.

## Best Possible Solution

Centralize or consistently apply `cfg.tasks` before every task command/runtime entry point that can terminalize or stamp task records, then land the focused performance/config change after maintainer approval of the new task config surface.

## Reproduction Assessment

Yes for the review finding by source inspection: a standalone `openclaw tasks cancel` process reaches `cancelTaskById()` without applying `cfg.tasks`, then `updateTask()` stamps `cleanupAfter` from `getTaskRetentionMs()`. No live busy-gateway before/after reproduction was run for the original performance claim, but current source and the supplied terminal proof cover the main optimized paths.

## Solution Assessment

No, not as currently proposed: the performance and config approach is narrow, but the config application is incomplete for a task command that terminalizes records. The safer fix is to apply task runtime config on the cancel path or centralize task-command config application before terminal mutations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Apply task config before cancelling tasks:** `src/commands/tasks.ts:85`
  - body: The new retention lookup is process-global, but the CLI cancel path loads config without calling `applyTasksConfig` before `cancelTaskById()` terminalizes the record. In a standalone `openclaw tasks cancel` process with `tasks.retentionMs` set, `updateTask()` stamps `cleanupAfter` from the default 7 days, and maintenance later will not restamp because `cleanupAfter` already exists. Please apply `cfg.tasks` on this path, or centralize task command config application before terminal mutations.
