## Summary

Keep this draft PR open: the implementation is useful, but the current docs and changelog still describe invalid `tasks.*` config values as falling back to defaults even though config validation rejects them and startup load throws.

## What This Changes

The PR optimizes task registry sweep/listing work, adds `tasks.retentionMs` and `tasks.sweepIntervalMs`, and wires the new task config through gateway startup, CLI maintenance, schema, docs, generated metadata, public types, and tests.

## Best Possible Solution

Update the public docs and changelog to say defaults apply when the keys are omitted, invalid values must be fixed after schema rejection, and cleanup stamping uses configured retention with a 7-day default.

## Reproduction Assessment

Yes for this PR review. Source inspection shows the schema rejects invalid `tasks.*` values and config load throws on validation failure, while the PR docs and changelog still say defaults remain in effect.

## Solution Assessment

No, not quite yet. The code direction is maintainable, but the public wording should match the validation contract before this draft is ready to merge.

- **[P2] Correct invalid-value fallback wording:** `docs/automation/tasks.md:352`

  - confidence: 0.86
