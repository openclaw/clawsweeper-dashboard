## What This Changes

The PR adds task maintenance retention and sweep-interval config, optimizes task listing/sweep work, and wires gateway startup to pass config into task registry maintenance.

## Best Possible Solution

Keep the optimization, but apply task config through the shared maintenance/config path, update user-facing task docs and changelog, then run the focused task/config validation before the PR leaves draft.

## Reproduction Assessment

## Solution Assessment

Is this the best way to solve the issue? Not yet; the listing/sweep optimizations are directionally sound, but the config application should be shared with manual maintenance and the public docs/changelog need to match the new behavior.

- **[P2] Apply task config before manual maintenance:** `src/tasks/task-registry.maintenance.ts:921-922`
  - body: The new `tasks.retentionMs` value is only installed by `startTaskRegistryMaintenance()`. The CLI path for `openclaw tasks maintenance` calls `configureTaskMaintenanceFromConfig()` and then `previewTaskRegistryMaintenance()`/`runTaskRegistryMaintenance()` without starting the scheduler, so manual cleanup in a separate CLI process still falls back to the default retention and can disagree with the configured gateway. Please apply `cfg.tasks` through the shared maintenance configuration path before preview/apply.
  - confidence: 0.84

That would publish contradictory user guidance for the new config, so please update the task docs alongside the schema changes.
  - confidence: 0.9

  - body: The diff introduces user-facing performance/config behavior but leaves `CHANGELOG.md` untouched. Repo policy requires a changelog entry for user-facing perf/config changes before merge.
