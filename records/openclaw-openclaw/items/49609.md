## Summary

Keep this open. Current main and the latest release still build the sandbox browser with x11vnc/Xvfb behind noVNC, while the linked TigerVNC PR was closed unmerged after review found security-sensitive VNC auth/listening regressions; the report remains a source-reproducible sandbox-browser bug.

## What This Changes

The issue requests fixing sandbox-browser noVNC clipboard paste so non-Latin-1 text such as CJK and Cyrillic reaches Chromium without byte truncation.

## Best Possible Solution

Move the sandbox-browser image to an Extended Clipboard-capable VNC path while preserving password-protected noVNC, loopback-only raw VNC, tokenized observer URLs, and the sandbox-browser security epoch/recreation contract.

## Reproduction Assessment

Yes. The current source still installs and starts x11vnc/Xvfb behind noVNC, and upstream noVNC/x11vnc behavior explains the exact non-Latin-1 truncation path without needing a live container run.

## Solution Assessment

Unclear. Replacing x11vnc with TigerVNC is a plausible dependency-level fix, but the prior PR showed that the best implementation must first preserve the existing sandbox-browser security contract.
