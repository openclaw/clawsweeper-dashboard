## Summary

Keep open; current main and latest release still use x11vnc/Xvfb behind noVNC, upstream noVNC falls back to Latin-1 when the server lacks Extended Clipboard, and the only linked TigerVNC PR was closed unmerged after VNC auth/listening regressions were identified.

## What This Changes

The issue requests UTF-8-safe clipboard transfer for non-Latin-1 text pasted through the sandbox-browser noVNC clipboard panel.

## Best Possible Solution

Move the sandbox-browser image to an Extended Clipboard-capable VNC path while preserving VNC auth, loopback-only raw VNC, tokenized noVNC observer URLs, container recreation semantics, and focused Unicode clipboard proof.

## Reproduction Assessment

Yes, by source and dependency contract. Current main still starts x11vnc/Xvfb behind noVNC, and upstream noVNC/x11vnc behavior explains the reported non-Latin-1 truncation path.

## Solution Assessment

Unclear. Replacing x11vnc with TigerVNC is plausible, but the prior PR showed the implementation must preserve the existing sandbox-browser VNC auth/listening contract before it is the best fix.
