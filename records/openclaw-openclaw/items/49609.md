## Summary

Keep this open. Current main and v2026.5.18 still use x11vnc/Xvfb behind noVNC, the upstream noVNC fallback still explains the reported UTF-8 truncation, and the only linked TigerVNC fix PR was closed unmerged after review found VNC auth/listening regressions.

## What This Changes

The issue asks OpenClaw to stop corrupting non-Latin-1 clipboard text pasted through the sandbox-browser noVNC clipboard panel.

## Best Possible Solution

Move the sandbox-browser image to an Extended Clipboard-capable VNC path while preserving VNC auth, loopback-only raw VNC, tokenized noVNC observer URLs, container recreation semantics, and focused regression proof.

## Reproduction Assessment

Yes, by source and dependency contract. Current main still starts x11vnc/Xvfb behind noVNC, and upstream noVNC/x11vnc behavior explains the reported non-Latin-1 truncation path.

## Solution Assessment

Unclear. Replacing x11vnc with TigerVNC is plausible, but the prior PR showed the implementation must preserve the existing sandbox-browser VNC auth/listening contract before it is the best fix.
