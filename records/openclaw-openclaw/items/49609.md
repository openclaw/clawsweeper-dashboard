## Summary

Keep open: current main and the latest release still run noVNC through x11vnc/Xvfb, and the dependency path still explains Latin-1 truncation for non-Latin-1 clipboard paste. The earlier TigerVNC PR is closed unmerged, so no canonical merged fix owns the remaining work.

## What This Changes

The issue asks OpenClaw to stop corrupting non-Latin-1 text pasted through the sandbox-browser noVNC clipboard path.

## Best Possible Solution

Move sandbox-browser to an Extended Clipboard-capable VNC path while preserving VNC auth, loopback-only raw VNC, tokenized noVNC observer URLs, container recreation behavior, and live Unicode clipboard proof.

## Reproduction Assessment

Yes, by source and dependency contract: current main starts x11vnc behind noVNC, and noVNC v1.3.0 falls back to Latin-1 ClientCutText when Extended Clipboard is unavailable. I did not run a live Docker clipboard round-trip in this read-only review.

## Solution Assessment

Unclear: replacing x11vnc with TigerVNC is plausible, but the exact VNC replacement needs maintainer/security review because it must preserve the sandbox observer auth and loopback constraints.
