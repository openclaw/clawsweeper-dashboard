- `impact:data-loss`: The issue is about user-entered clipboard text being corrupted when pasted through the noVNC sandbox-browser path.

## Summary

Keep open; current main and v2026.5.22 still use x11vnc/Xvfb behind noVNC, and upstream noVNC falls back to Latin-1 ClientCutText when the VNC server does not advertise Extended Clipboard.

## What This Changes

The issue asks for the sandbox-browser noVNC clipboard path to preserve non-Latin-1 Unicode text instead of truncating it through the current VNC fallback path.

## Best Possible Solution

Move sandbox-browser to an Extended Clipboard-capable VNC path while preserving VNC auth, loopback-only raw VNC, tokenized noVNC observer URLs, container recreation semantics, and live Unicode clipboard proof.

## Reproduction Assessment

Yes, by source and dependency contract. Current main starts Xvfb/x11vnc/noVNC, noVNC v1.3.0 falls back to ClientCutText with charCodeAt when Extended Clipboard caps are absent, and RFB ClientCutText is Latin-1; no live Docker clipboard round-trip was run in this read-only review.

## Solution Assessment

Unclear. Replacing x11vnc with TigerVNC is plausible, but the closed attempt showed the fix must preserve VNC auth, loopback raw VNC, observer tokens, and upgrade/recreate behavior before it is the best maintainable path.
