- `impact:data-loss`: The issue is about user-entered clipboard text being corrupted when pasted through the noVNC sandbox-browser path.

## Summary

Keep open; current main and the latest release still run noVNC through x11vnc/Xvfb, and the upstream noVNC fallback path still truncates non-Latin-1 clipboard text when the server does not advertise Extended Clipboard support.

## What This Changes

The issue asks for non-Latin-1 clipboard paste through the sandbox-browser noVNC panel to preserve Unicode text instead of being truncated by the current VNC stack.

## Best Possible Solution

Move the sandbox-browser image to an Extended Clipboard-capable VNC path while preserving VNC auth, loopback-only raw VNC, tokenized noVNC observer URLs, container recreation semantics, and live Unicode clipboard proof.

## Reproduction Assessment

Yes, by source and dependency contract. Current main still starts x11vnc/Xvfb behind noVNC, and upstream noVNC/x11vnc behavior explains the non-Latin-1 truncation path; no live Docker clipboard round-trip was run in this read-only review.

## Solution Assessment

Unclear. Replacing x11vnc with TigerVNC is plausible, but the prior attempt showed the implementation must preserve the existing noVNC auth, loopback, and observer-token contracts before it is the best fix.
