## Summary

Keep this open. Current main and v2026.5.18 still use x11vnc/Xvfb behind noVNC, upstream noVNC/x11vnc source supports the reported UTF-8 truncation path, and the linked TigerVNC PR was closed unmerged with the VNC auth/listening contract still needing maintainer resolution.

## What This Changes

The issue asks OpenClaw to fix sandbox-browser noVNC clipboard paste so non-Latin-1 text reaches Chromium intact instead of being truncated through the legacy VNC clipboard path.

## Best Possible Solution

Move the sandbox-browser image to an Extended Clipboard-capable VNC path while preserving VNC auth, loopback-only raw VNC, tokenized noVNC observer URLs, container recreation semantics, and focused regression proof for CJK/Cyrillic paste.

## Reproduction Assessment

Yes, by source and dependency contract. Current main still uses x11vnc behind noVNC, noVNC falls back to charCodeAt/Uint8Array when Extended Clipboard is absent, and x11vnc 0.9.16 uses legacy CutText in the inspected clipboard path.

## Solution Assessment

Unclear. Replacing x11vnc with TigerVNC is plausible, but the prior PR showed the best fix must preserve OpenClaw's noVNC auth, loopback, and observer-token contract before changing the VNC server stack.
