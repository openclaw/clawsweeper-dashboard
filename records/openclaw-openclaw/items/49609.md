## Summary

Keep this open. Current main and v2026.5.3 still route sandbox-browser noVNC through x11vnc/Xvfb, and the related TigerVNC PR was closed unmerged, so this remains a source-reproducible sandbox-browser infrastructure bug.

## What This Changes

The issue asks OpenClaw to make sandbox-browser noVNC clipboard paste preserve non-Latin-1 text such as CJK and Cyrillic, likely by moving to a VNC server with Extended Clipboard support.

## Best Possible Solution

Keep the issue open and route it to sandbox/secops maintainers for a focused browser image and entrypoint change that uses an Extended Clipboard-capable VNC path, updates the sandbox-browser security hash if existing containers must be recreated, adds targeted coverage or docs as needed, and proves CJK/Cyrillic clipboard round-trip through noVNC.

## Reproduction Assessment

Yes. The issue gives deterministic noVNC clipboard steps, and current main still uses the x11vnc path whose noVNC fallback truncates non-Latin-1 text when Extended Clipboard is not negotiated.

## Solution Assessment

Unclear. Replacing or augmenting x11vnc with an Extended Clipboard-capable VNC server is likely the right direction, but the specific TigerVNC/Xtigervnc swap needs maintainer/secops review for auth, loopback exposure, lifecycle, and container-recreation behavior.
