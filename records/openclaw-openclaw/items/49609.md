## Summary

Keep this open. Current main and v2026.5.7 still use x11vnc/Xvfb behind noVNC, and the linked TigerVNC PR was closed unmerged after review flagged security-sensitive auth/listening regressions, so this remains a source-reproducible sandbox-browser bug.

## What This Changes

The issue asks for sandbox-browser noVNC clipboard support to preserve non-Latin-1 text such as CJK and Cyrillic instead of truncating it through the x11vnc fallback path.

## Best Possible Solution

Use an Extended Clipboard-capable VNC path in the sandbox-browser image while preserving loopback-only raw VNC, default password protection, tokenized noVNC observer URLs, and container recreation via the sandbox-browser security epoch.

## Reproduction Assessment

Yes, from source inspection rather than a live Docker run. Current main builds the browser image with x11vnc/Xvfb and starts x11vnc behind noVNC, while noVNC v1.3 falls back to charCodeAt/Uint8 when Extended Clipboard is not negotiated.

## Solution Assessment

Unclear. Replacing x11vnc with TigerVNC is plausible, but the prior PR showed security regressions around localhost binding and default auth, so the best fix must preserve the existing sandbox-browser security contract.
