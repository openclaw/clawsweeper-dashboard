## Summary

Keep this open. Current main and the latest release still use x11vnc/Xvfb behind noVNC, and the only linked TigerVNC fix PR was closed unmerged after review found security-sensitive VNC auth/listening regressions.

## What This Changes

The issue reports garbled non-Latin-1 clipboard input through the sandbox-browser noVNC panel because the current x11vnc path lacks Extended Clipboard support.

## Best Possible Solution

Move the sandbox-browser image to an Extended Clipboard-capable VNC path while preserving password-protected noVNC, loopback-only raw VNC, tokenized observer URLs, and the sandbox-browser security epoch/recreation contract.

## Reproduction Assessment

Yes, by source and dependency contract. Current main still starts x11vnc/Xvfb behind noVNC, and upstream noVNC/x11vnc behavior explains the reported non-Latin-1 truncation path.

## Solution Assessment

Unclear. Replacing x11vnc with TigerVNC is plausible, but the prior PR showed the implementation must preserve the existing sandbox-browser security contract before it is the best fix.
