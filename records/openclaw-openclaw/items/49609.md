## Label Justifications

- `P2`: The issue is a reproducible sandbox-browser behavior bug with limited blast radius but real impact for users entering non-Latin text through noVNC.
- `impact:data-loss`: User-provided clipboard text is silently transformed before it reaches the browser input, which is a concrete data corruption/loss mode.

## Summary

Keep open: current main and v2026.6.6 still use the sandbox-browser noVNC path through Xvfb + x11vnc, upstream noVNC falls back to Latin-1 truncation when Extended Clipboard is not advertised, and the only linked TigerVNC fix PR closed unmerged after security concerns.

## What This Changes

Fix sandbox-browser noVNC clipboard paste so non-Latin-1 text reaches the sandboxed browser without VNC clipboard truncation.

## Best Possible Solution

Replace or repair the sandbox-browser VNC stack with UTF-8 Extended Clipboard support while preserving existing noVNC password, token, and loopback access semantics and proving the Docker user flow.

## Reproduction Assessment

Yes, for a high-confidence source-level reproduction path: current main and v2026.6.6 still run noVNC through x11vnc, while noVNC v1.3.0 truncates non-Latin-1 text when Extended Clipboard is not advertised. A live Docker clipboard round-trip was not run in this read-only review.

## Solution Assessment

Unclear. Switching to TigerVNC is a plausible dependency-backed fix, but the earlier attempt at https://github.com/openclaw/openclaw/pull/49618 regressed VNC auth/binding behavior, so the best fix needs maintainer-reviewed security and startup proof before implementation.
