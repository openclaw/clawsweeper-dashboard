## Summary

Keep open. Current main and the latest release still send the non-resumable restart-recovery notice asking for a resend, then fail the session; the requested no-retype recover-and-confirm behavior remains a real but product-sensitive core recovery change.

## What This Changes

The issue asks OpenClaw restart recovery to avoid asking users to retype an already-transcribed request when the interrupted transcript tail is non-resumable, using a recover-and-continue or recover-and-confirm flow instead.

## Best Possible Solution

Design a bounded recover-and-confirm state for non-resumable main-session tails that reuses the recovered user request, preserves approval gates, and leaves the existing automatic user/tool/toolResult-tail resume path unchanged.

## Reproduction Assessment

Yes at source level: a marked running main session whose recent transcript tail ends in an assistant message or stale approval-pending tool result takes the resume-block branch, sends the unresumable resend notice, and marks the session failed. I did not run a live gateway-restart reproduction in this read-only review.

## Solution Assessment

Unclear as an exact implementation path: the no-retype direction is sound, but maintainers need to choose the safe product contract for confirm-to-continue, approval reconstruction, or idempotency before an automated fix is appropriate.
