## Summary

Keep open. Current main and the latest release still send the non-resumable restart-recovery notice asking for a resend, then mark the session failed; the closest open PR only addresses WebChat visibility for that same notice and does not implement the requested no-retype recover-and-confirm flow.

## What This Changes

The issue requests a restart-recovery flow that reuses the existing transcript request instead of asking users to retype when the transcript tail is not auto-resumable.

## Best Possible Solution

Design a bounded recover-and-confirm path for non-resumable main-session tails that reuses the recovered user request, preserves approval gates, and leaves existing automatic resume behavior unchanged.

## Reproduction Assessment

Yes at source level: a restart-aborted running main session whose recent transcript tail is assistant output or a stale approval-pending tool result takes the resume-block branch, sends the unresumable notice when a delivery target exists, and marks the session failed. I did not run a live gateway-restart scenario in this read-only review.

## Solution Assessment

Unclear as an exact implementation path: the no-retype direction fits the user problem, but maintainers need to choose the safe contract for confirm-to-continue, approval reconstruction, or idempotency before a fix PR should be queued.
