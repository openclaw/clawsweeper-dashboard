## Summary

Keep open. Current main and v2026.6.10 still ask the user to resend on non-resumable restart-recovery tails, then fail the session; the remaining no-retype recovery behavior is real but product-sensitive because it must preserve side-effect and approval safety.

## What This Changes

The issue requests a restart-recovery flow that reuses the transcript’s last user request instead of forcing a retype when the tail is non-resumable.

## Best Possible Solution

Define a bounded recover-and-confirm state in main-session restart recovery that reuses the recovered user request, preserves approval gates, and leaves the existing automatic user/tool/toolResult-tail resume path unchanged.

## Reproduction Assessment

Yes, at source level: a marked running main session whose recent transcript tail ends in an assistant message or stale approval-pending tool result takes the resume-block branch, sends the resend notice when deliverable, and marks the session failed. I did not run a live gateway-restart reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation path: the no-retype direction is sound, but maintainers still need to choose the safe product contract for confirm-to-continue, approval reconstruction, or idempotency before this should be automated.
