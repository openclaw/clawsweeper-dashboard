## Summary

Keep open. Current main and v2026.6.10 still send the non-resumable restart-recovery resend notice and then fail the session; the open candidate PR is partial and unmerged, so the central no-retype recovery behavior remains unresolved.

## What This Changes

Replace non-resumable restart recovery's resend-and-fail path with a no-retype recover-and-continue or recover-and-confirm flow while preserving side-effect and approval safety.

## Best Possible Solution

Design a bounded core restart-recovery state that reuses the recovered user request, preserves approval gates, and keeps automatic recovery only where side effects are safely idempotent or explicitly confirmed.

## Reproduction Assessment

Yes at source level: a marked running main session whose recent transcript tail ends in an assistant message or stale approval-pending tool result takes the blocked branch, sends the resend notice when deliverable, and marks the session failed. I did not run a live gateway-restart reproduction in this read-only review.

## Solution Assessment

Unclear as an exact implementation path: the no-retype direction is sound, but maintainers need to choose the confirm-to-continue, approval reconstruction, or idempotency contract before this is a narrow repair.
