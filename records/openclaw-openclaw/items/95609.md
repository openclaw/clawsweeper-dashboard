## Summary

Keep open. Current main and v2026.6.9 still take the non-resumable restart-recovery branch by sending the resend notice and failing the session, while the open related PR only covers part of the requested behavior and is still blocked on session-state safety/proof.

## What This Changes

The issue requests a no-retype recover-and-continue or recover-and-confirm flow for non-resumable restart-recovery transcript tails.

## Best Possible Solution

Design a bounded no-retype recovery state that reuses the saved user request, gates side-effect replay with confirmation or idempotency, preserves approval gates, and then land a reviewed canonical PR.

## Reproduction Assessment

Yes, at source level: a marked running main session whose recent transcript tail ends in an assistant message or stale approval-pending tool result takes the resend-notice-and-fail branch. I did not run a live gateway restart reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation path. The no-retype direction is sound, but the best fix needs a maintainer decision on confirm-to-continue, approval reconstruction, or idempotency before the runtime starts a new recovery turn.
