## Summary

Keep open. Current main and the latest release still implement the dead-end resend notice for non-resumable restart-recovery tails, and adjacent WebChat notice work only makes that same notice visible rather than replacing it with recover-and-confirm behavior.

## What This Changes

Issue requests a restart-recovery flow that avoids asking users to retype requests when the interrupted transcript tail is not safely auto-resumable.

## Best Possible Solution

Design a bounded recover-and-confirm state for non-resumable main-session tails that reuses the recovered user request, preserves approval gates, and keeps the existing automatic user/tool/toolResult-tail resume path unchanged.

## Reproduction Assessment

Yes from source inspection: a marked running main session whose recent transcript tail ends in an assistant message or stale approval-pending tool result takes the resume-block branch, sends the unresumable notice, and marks the session failed. I did not run a live gateway-restart reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation path because the issue intentionally offers policy choices: continue, confirm-to-continue, or approval reconstruction. The best solution should be maintainer-chosen and safety-preserving, not an automatic repair-worker guess.
