## Summary

Keep open. Current main and the latest release still send the non-resumable restart-recovery notice asking for a resend and then fail the session; the open related PR is a partial candidate, not a merged implementation of the broader recover-and-confirm request.

## What This Changes

Request to replace restart-recovery resend/fail behavior for non-resumable main-session tails with a no-retype recover-and-continue or recover-and-confirm flow.

## Best Possible Solution

Define a maintainer-approved recover-and-confirm state for non-resumable main-session tails that reuses the recovered user request, preserves approval gates, avoids double-running side effects, and leaves the existing user/tool/toolResult auto-resume path unchanged.

## Reproduction Assessment

Yes, at source level. A restart-aborted running main session whose recent transcript tail ends in an assistant message or stale approval-pending tool result takes the resume-block branch, sends the resend notice when it can, and marks the session failed; I did not run a live gateway-restart reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation plan. The no-retype goal is maintainable, but the best solution needs an explicit product contract for confirm-to-continue, stale approval reconstruction, or idempotency before a fix should land.
