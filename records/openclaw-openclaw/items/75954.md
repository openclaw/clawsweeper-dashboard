## Summary

Keep open. Current main has relevant post-release hardening for session-list responsiveness and stale running lifecycle state, but the restart/session-count symptom is not proven fixed: `openclaw status` still labels all persisted session rows as active, cleanup only removes missing transcripts with `--fix-missing`, and restart recovery can still resume rows that are both `running` and `abortedLastRun`.

## What This Changes

The issue reports that repeated Gateway restarts leave about 80 sessions shown as active, with slow session/node list operations and liveness warnings on OpenClaw 2026.4.29.

## Best Possible Solution

Collect a diagnostics export plus `openclaw sessions cleanup --all-agents --dry-run --fix-missing --json`, then fix the narrow cause: lifecycle/recovery if rows are genuinely stuck running, or status/cleanup wording if the count is persisted history.

## Reproduction Assessment

Unclear. The report has enough commands and metrics to guide investigation, but a high-confidence reproduction needs redacted session-store rows or a diagnostics bundle showing whether the 80 rows are terminal history, missing-transcript entries, or `running`/`abortedLastRun` recovery candidates.

## Solution Assessment

Unclear. The safest solution depends on the store state; blindly resetting sessions on restart would risk losing durable conversation pointers, while the current main fixes only part of the possible lifecycle/performance path.
