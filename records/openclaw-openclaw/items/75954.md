## Summary

Keep open. Current main has session-list responsiveness hardening, but `openclaw status` still labels persisted session-store rows as active and there is not enough diagnostic data to tie the reported 80-session restart/high-CPU loop to a specific fixed or unfixed runtime path.

## What This Changes

The issue reports that after repeated macOS Gateway restarts, `openclaw status` continues showing 80 active sessions with slow `sessions.list` and `node.list` calls plus liveness warnings.

## Best Possible Solution

Collect a diagnostics export and `openclaw sessions cleanup --all-agents --dry-run --fix-missing --json` from current main or the next release, then fix the proven stale-row, recovery, cleanup, or status-wording path without changing durable session semantics unnecessarily.
