## Label Justifications

- `P2`: This was a real session-store performance and state-reporting bug, but the affected 2026.4.29 path is now fixed and shipped.
- `impact:session-state`: The report centers on persisted session rows being retained or recreated and session cleanup not reducing the reported count.
- `impact:crash-loop`: The reported event-loop saturation, high CPU, liveness warnings, and restart pressure are process-availability symptoms.

## Summary

Current main and the latest release already address the reported session-store starvation path: session-store writes no longer fsync inside the writer lock, sessions.list has bounded/yielding row work, and docs clarify that session rows are stored conversation state rather than live socket liveness.

## What This Changes

The issue reports OpenClaw 2026.4.29 keeping about 80 sessions after gateway restarts, causing high CPU, slow sessions.list/node.list calls, and liveness warnings.

## Best Possible Solution

Keep the shipped session-store fsync and sessions.list hot-path fixes, and treat any remaining post-v2026.6.6 session recreation report as a new issue with fresh diagnostics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a current-main failure; the historical 2026.4.29 source path and linked field reports identify the session-store fsync starvation contributor, and current main has the targeted fix.

## Solution Assessment

Is this the best way to solve the issue? Yes for the repo-owned root cause: skipping durable fsync only for the session-store index preserves atomic replacement while avoiding the writer-lock starvation, instead of resetting stored conversation history on every restart.
