## Summary

Keep open. Current main has session-list responsiveness hardening and docs that define session listings as persisted rows, but it still renders the persisted status count as “active” and there is not enough diagnostic data to tie the reported 80-session restart/high-CPU loop to a fixed path.

## What This Changes

The issue reports that after repeated macOS Gateway restarts, OpenClaw 2026.4.29 still shows 80 active sessions with slow `sessions.list`/`node.list` calls and event-loop liveness warnings.

## Best Possible Solution

Collect diagnostics from current main or the next release, then fix the narrow proven path: status wording, stale-row cleanup, restart recovery, session-memory indexing, or another loop shown by the data.

## Reproduction Assessment

Unclear. Source inspection explains the “active” count as persisted stored rows and shows cleanup/recovery boundaries, but the full 80-session restart/high-CPU loop needs sanitized store rows or a diagnostics export to reproduce with high confidence.

## Solution Assessment

Unclear. Current main’s lightweight `sessions.list` path is the right performance direction, but resetting durable session rows on restart would conflict with documented persisted-session behavior unless diagnostics prove those rows are stale or being recreated incorrectly.
