## Summary

Keep open. Current main has meaningful `sessions.list` responsiveness and restart-recovery hardening, but it still labels persisted session-store rows as active and the report does not include enough diagnostics to prove the 80-session restart recreation/high-CPU loop is fixed or to choose a safe repair.

## What This Changes

The issue asks OpenClaw to stop showing or recreating 80 active sessions after gateway restarts and cleanup, because session/node listing becomes slow and the gateway reports event-loop liveness warnings.

## Best Possible Solution

Collect a diagnostics bundle and cleanup dry-run, then either fix the narrow stale-row/recovery path or clarify `openclaw status` wording if the count is persisted history rather than live work.

## Reproduction Assessment

Unclear. Source inspection reproduces the misleading `openclaw status` active wording for persisted rows, but the actual 80-session recreation/high-CPU loop needs sanitized store rows or diagnostics to reproduce with high confidence.

## Solution Assessment

Unclear. Resetting sessions on restart is not the best default because sessions are durable conversation rows; the maintainable fix depends on whether diagnostics show stale live recovery, cleanup semantics, or status wording is the root cause.
