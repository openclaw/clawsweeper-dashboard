## Summary

Keep open. Current main still has hot-path code that can rebuild plugin manifest registries and reopen plugin manifests during embedded-run preparation, matching the reporter's strace evidence and the related VPS/Docker latency reports.

## What This Changes

The issue asks OpenClaw to reuse plugin extension manifest metadata so each embedded agent request does not reopen every bundled plugin manifest many times before first token.

## Best Possible Solution

Thread the existing PluginMetadataSnapshot or PluginLookUpTable through embedded-run request preparation for plugin skills, provider ownership/activation, bundle settings, and contribution lookup, while preserving snapshot invalidation on config, workspace, and installed-index changes.

## Reproduction Assessment

Yes. The issue gives concrete OPENCLAW_DEBUG_INGRESS_TIMING plus strace reproduction steps, and current main contains multiple request-time paths that rebuild manifest registries instead of consistently reusing the gateway metadata snapshot.

## Solution Assessment

Yes. Reusing the existing metadata snapshot/lookup-table seam is the narrow maintainable fix; adding a broad persistent manifest cache or time-based cache would conflict with the documented plugin cache boundary.
