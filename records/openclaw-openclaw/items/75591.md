## Summary

Keep open. Current main has the intended startup metadata snapshot/lookup-table seam, but request-time plugin tool resolution still goes through plugin auto-enable without passing that manifest registry, which can reconstruct the manifest registry on hot paths matching the reporter's syscall evidence.

## What This Changes

The issue reports repeated bundled plugin manifest reads during each embedded-run request, causing long `core-plugin-tools`, `system-prompt`, and `stream-setup` prep stages on a minimal Linux install.

## Best Possible Solution

Reuse the existing Gateway `PluginMetadataSnapshot`/`PluginLookUpTable` or an explicit `manifestRegistry` through request-time plugin tool and auto-enable flows, then add a focused regression test that proves embedded-run tool setup does not cold-reload plugin manifests per request. Keep the existing explicit snapshot boundary rather than adding a broad global manifest cache.

## Reproduction Assessment

Yes. The issue gives a concrete Linux/npm reproduction with `OPENCLAW_DEBUG_INGRESS_TIMING` and strace, and current-main source shows a request-time path that can call manifest registry loading without the startup manifest snapshot.

## Solution Assessment

Yes, the requested direction is the right maintainable fix if implemented through the existing snapshot/lookup-table seam. A global persistent manifest cache would be less aligned with the documented plugin cache boundary.
