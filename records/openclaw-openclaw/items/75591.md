## Summary

Keep open. The report is fresh, detailed, unprotected, and current main still has request-time paths that can rebuild plugin manifest registries instead of consistently reusing the gateway-owned metadata snapshot; that matches the reported repeated `openclaw.plugin.json` opens during `skills`, `core-plugin-tools`, `system-prompt`, and `stream-setup` preparation.

## What This Changes

The issue requests eliminating repeated bundled plugin manifest reads during each embedded agent request so manifests are parsed or looked up from gateway startup metadata instead of rescanned in request prep.

## Best Possible Solution

Thread the existing compatible `PluginMetadataSnapshot` or `PluginLookUpTable` into request-time plugin contribution, provider ownership, plugin skill, and bundle settings resolution so gateway-backed embedded runs reuse startup metadata while config/workspace changes still invalidate correctly.

## Reproduction Assessment

Yes. The issue includes a concrete Linux `OPENCLAW_DEBUG_INGRESS_TIMING=1` plus `strace -e openat` reproduction, and current source shows multiple request-time helper paths that can rebuild manifest registries instead of reusing startup metadata.

## Solution Assessment

Yes, the narrow maintainable direction is to reuse the existing startup metadata snapshot/lookup-table seam across request prep. Adding a broad persistent manifest cache would conflict with the plugin loader guidance; the safer fix is explicit or current-snapshot reuse with invalidation by config/workspace policy hash.

## Repair Work Prompt

Fix the per-request plugin manifest reread regression reported in https://github.com/openclaw/openclaw/issues/75591. Current main can rebuild plugin manifest registries during embedded-run prep even though gateway startup already builds a PluginMetadataSnapshot/PluginLookUpTable. Keep the fix scoped to reusing existing compatible metadata; do not add a new persistent discovery/manifest cache, do not special-case bundled plugin ids, and preserve config/workspace invalidation and third-party plugin behavior.

Expected implementation boundary:
- Reuse `getCurrentPluginMetadataSnapshot` or explicitly threaded `PluginLookUpTable`/`manifestRegistry` in request-time provider ownership/plugin contribution resolution, provider runtime activation planning, plugin skill directory resolution, and bundle Pi settings resolution where applicable.
