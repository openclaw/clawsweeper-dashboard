## Summary

Keep open. The report is fresh, detailed, and still matches current main: gateway startup owns a compatible plugin metadata snapshot, but request-time embedded-run paths still rebuild manifest registries directly, which can explain repeated `openclaw.plugin.json` opens during prep.

## What This Changes

Fix the per-request plugin manifest registry rebuilds that make embedded-run prep stages spend tens of seconds repeatedly reading bundled plugin manifests.

## Best Possible Solution

Reuse the current compatible `PluginMetadataSnapshot`, `PluginLookUpTable`, or explicit manifest registry through embedded-run skills, bundle settings, provider ownership, and contribution resolution, with existing config/workspace/plugin-inventory invalidation preserved and regression tests proving request prep does not rebuild manifest registries.
