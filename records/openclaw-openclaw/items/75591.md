## Summary

Keep open. The issue is fresh, detailed, independently confirmed in later comments, and current main still has request-time paths that can rebuild plugin manifest registries instead of consistently reusing the gateway-owned metadata snapshot.

## What This Changes

The issue reports per-request plugin manifest registry rebuilds causing repeated `openclaw.plugin.json` reads and 60-180s embedded-run prep latency.

## Best Possible Solution

Reuse the gateway-owned `PluginMetadataSnapshot`, `PluginLookUpTable`, or explicit manifest registry through embedded-run skill resolution, provider/contribution lookup, bundle settings, and stream setup, with invalidation tied to config/workspace/plugin inventory changes and regression tests that prove request prep does not rebuild the manifest registry repeatedly.

## Reproduction Assessment

Yes. The issue gives a high-confidence Linux reproduction using `OPENCLAW_DEBUG_INGRESS_TIMING=1` and `strace -e openat`, and current main still shows request-prep call paths that can rebuild plugin manifest registries.

## Solution Assessment

Yes. The narrow maintainable fix is explicit snapshot/lookup-table reuse through the hot path; a hidden persistent manifest cache would conflict with the documented plugin cache boundary.
