## Summary

Keep open. The report is fresh, detailed, corroborated by several related latency reports, and current main still has request-time paths that can rebuild plugin manifest registries during skill snapshot/watch setup and bundle settings resolution instead of consistently reusing the Gateway-owned metadata snapshot.

## What This Changes

The issue requests fixing per-request plugin manifest registry rescans that repeatedly open bundled `openclaw.plugin.json` files and add long embedded-run prep latency.

## Best Possible Solution

Thread the existing Gateway-compatible `PluginMetadataSnapshot`, `PluginLookUpTable`, or explicit manifest registry through request-time skill watching/snapshot building, bundle settings, and provider contribution owner resolution so normal embedded runs reuse startup metadata while config, plugin inventory, workspace, and installed-index changes still rebuild the snapshot explicitly.

Yes. The issue gives concrete Linux timing and `strace` steps, and current main still exposes source-level request paths that can rescan plugin manifests during embedded-run preparation; live syscall counts should be captured by the fix PR for before/after proof.

## Solution Assessment

Yes. The narrow maintainable fix is explicit snapshot/lookup-table reuse through the hot paths; adding a hidden persistent manifest cache would conflict with the documented plugin cache boundary and could mask plugin/config changes.
