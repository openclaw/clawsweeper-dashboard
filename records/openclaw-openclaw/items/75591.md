## Summary

Keep open. The report is fresh, detailed, corroborated by multiple users, and current main still contains request-prep paths that rebuild plugin manifest/registry metadata instead of consistently reusing the Gateway-owned snapshot.

## What This Changes

The issue reports request-time plugin extension manifest reloads causing 60-180 second embedded-run preparation stages on OpenClaw 2026.4.29.

## Best Possible Solution

Thread the Gateway-owned `PluginMetadataSnapshot`, `PluginLookUpTable`, or explicit `PluginManifestRegistry` through embedded-run skill resolution, plugin tool discovery, provider/contribution lookup, bundle settings, and stream setup, with invalidation tied to config, workspace, plugin inventory, and installed-index changes.

## Reproduction Assessment

Yes. The issue gives a high-confidence Linux reproduction using timing traces and `strace`, and multiple comments independently confirm the same prep-stage regression on macOS and across channels; the current source still contains matching cold manifest-registry paths.

## Solution Assessment

Yes. The narrow maintainable fix is explicit snapshot or lookup-table reuse through the hot path; adding hidden persistent metadata caches would conflict with the documented plugin cache boundary and make invalidation harder.
