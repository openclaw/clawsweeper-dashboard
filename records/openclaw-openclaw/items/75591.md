## Summary

Keep open. The report is fresh, detailed, confirmed by multiple users, and current main still has request/runtime fallback paths that can reconstruct plugin manifest registries instead of consistently consuming the gateway-owned metadata snapshot, matching the reported repeated `openclaw.plugin.json` reads.

## What This Changes

The issue requests eliminating repeated per-request plugin manifest reads that inflate embedded-run prep stages to roughly 60-180 seconds before first token.

## Best Possible Solution

Thread the gateway-owned metadata snapshot, lookup table, or explicit manifest registry through embedded-run skill resolution, provider/contribution lookup, bundle settings, and stream setup, with invalidation tied to config/workspace/plugin inventory changes and regression tests that assert request prep does not rebuild manifest registries repeatedly.

## Reproduction Assessment

Yes. The issue provides a high-confidence Linux reproduction using `OPENCLAW_DEBUG_INGRESS_TIMING=1` plus `strace -e openat`, and multiple comments confirm the same prep-stage regression across Linux and macOS.

## Solution Assessment

Yes. The narrow maintainable fix is explicit snapshot/lookup-table reuse through the hot path; adding a hidden persistent manifest cache would conflict with the documented plugin cache boundary.
