## Summary

Keep open. Current main still has request-time paths that can rebuild plugin manifest registries instead of consistently reusing the gateway-owned metadata snapshot, matching the fresh Linux strace evidence and the related prep-latency reports.

## What This Changes

The issue reports that OpenClaw 2026.4.29 rereads bundled plugin manifests over 100 times per request, causing 60s embedded-run preparation stages before first token.

## Best Possible Solution

Thread the existing compatible `PluginMetadataSnapshot`, `PluginLookUpTable`, or explicit manifest registry through embedded-run plugin skills, activation planning, provider ownership/runtime hooks, capability-provider resolution, and plugin tool construction so gateway-backed request prep reuses startup metadata while config, workspace, and plugin-inventory changes still invalidate correctly.

## Repair Work Prompt

Fix the per-request plugin manifest registry rebuilds reported in https://github.com/openclaw/openclaw/issues/75591. The observable bug is that embedded runs on 2026.4.29 repeatedly reopen bundled `dist/extensions/*/openclaw.plugin.json` during `core-plugin-tools`, `system-prompt`, `stream-setup`, and related prep stages. Keep the fix within the existing plugin metadata snapshot / lookup-table design: do not add a hidden persistent manifest cache or wall-clock TTL. Thread a compatible `PluginMetadataSnapshot`, `PluginLookUpTable`, explicit manifest registry, or current gateway snapshot lookup through request-time plugin skill loading, activation planning, provider ownership/runtime hook resolution, capability-provider resolution, and plugin tool construction where those paths currently call `loadPluginManifestRegistryForPluginRegistry` or `loadPluginManifestRegistryForInstalledIndex`. Preserve config/workspace/plugin-inventory invalidation semantics, plugin disablement, SecretRef/runtime config behavior, and third-party plugin compatibility. Add focused regression tests that prove request-time callers can reuse supplied/current metadata without rebuilding manifest registries, especially `resolveEmbeddedRunSkillEntries`/`loadWorkspaceSkillEntries`, provider activation/ownership, and plugin tools/provider hooks. Add a user-facing `CHANGELOG.md` fix entry with reporter credit such as `Thanks @renatmannanov` and any directly used related reporter credit; do not credit bots.
