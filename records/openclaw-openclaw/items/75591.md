## Summary

Keep open. The report is fresh, detailed, independently confirmed, and current main still contains request-time plugin/embedded-run paths that can rebuild manifest registries instead of consistently reusing the gateway-owned plugin metadata snapshot.

## What This Changes

The issue requests eliminating repeated per-request plugin manifest reads so embedded-run prep reuses startup plugin metadata instead of opening each plugin manifest roughly 100 times per request.

## Best Possible Solution

Thread the current compatible PluginMetadataSnapshot or PluginLookUpTable through embedded-run prep for plugin skills, bundle settings, activation/provider ownership, and provider runtime hook resolution, with fallbacks only when no compatible gateway snapshot exists.

## Reproduction Assessment

Yes. The issue gives a concrete Linux OPENCLAW_DEBUG_INGRESS_TIMING plus strace reproduction, and current source still shows request-time registry rebuild paths that plausibly produce the repeated manifest opens.

## Solution Assessment

Yes. The narrow maintainable solution is explicit reuse of the existing startup metadata snapshot/lookup-table seam; adding a broad hidden persistent manifest cache would conflict with the documented plugin cache boundary.
