## Summary

Keep open. The report is fresh, detailed, confirmed by multiple later comments, and current main still has request-time embedded-run skill and contribution/provider paths that can reconstruct plugin manifest registries instead of consistently reusing the gateway-owned metadata snapshot.

## What This Changes

The issue requests eliminating repeated per-request plugin manifest reads that cause 60-180s embedded-run preparation latency.

## Best Possible Solution

Thread the gateway-owned PluginMetadataSnapshot, PluginLookUpTable, or explicit manifest registry through embedded-run skill resolution, provider/contribution lookup, bundle settings, and stream setup, with invalidation tied to config, workspace, and plugin inventory changes plus regression coverage proving request prep does not repeatedly rebuild manifest metadata.

Yes. The report provides a high-confidence Linux reproduction with OPENCLAW_DEBUG_INGRESS_TIMING and strace, and current main still shows request-prep call paths that can rebuild plugin manifest registries.

## Solution Assessment

Yes. The narrow maintainable direction is explicit snapshot or lookup-table reuse through the hot path; a hidden persistent manifest cache would conflict with the documented plugin cache boundary.
