## Summary

Keep open. Current main still has unguarded assistant-content `block.type` reads in the shared transport transform and Anthropic payload conversion paths, and the linked fix PR was closed unmerged, so the reported crash class remains actionable in this repository.

## What This Changes

The issue requests defensive handling in the MiniMax Anthropic-compatible path so empty, thinking-only, or malformed assistant content does not crash cron sessions with a `block.type` TypeError.

## Best Possible Solution

Normalize assistant content at the replay or transport boundary before any `block.type` access, skip malformed/nullish entries, preserve valid text/tool-call/signed-thinking/redacted-thinking semantics, and add focused regression tests for empty, thinking-only, null, undefined, and sparse assistant content.

## Reproduction Assessment

Yes for the source-level crash class: a focused unit test can pass an assistant message with `content: [undefined]`, `content: [null]`, or a sparse slot into the Anthropic transport transform path and observe the current direct `block.type` dereference. The live MiniMax intermittency remains provider-dependent.

## Solution Assessment

Yes, a narrow defensive normalization in `transport-message-transform.ts` and `anthropic-transport-stream.ts`, with tests, is the best maintainable fix. Surfacing all thinking content as visible text by default would be broader and riskier because MiniMax docs already call out reasoning leakage on this path.
