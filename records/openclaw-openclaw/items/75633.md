## Summary

Keep open. Current main still has the unsafe assistant-content block iteration described in the report, and the related fix PR was closed unmerged, so this remains a current, actionable OpenClaw transport hardening bug.

## What This Changes

The issue asks OpenClaw to harden the Anthropic Messages adapter so MiniMax Anthropic-compatible responses with empty, thinking-only, or malformed assistant content do not crash cron sessions with a `block.type` TypeError.

## Best Possible Solution

Add defensive normalization at the shared transport and Anthropic request-conversion boundaries so malformed assistant content blocks are skipped or converted according to existing reasoning/privacy semantics, then add focused regression tests for empty, thinking-only, null, undefined, and sparse content cases.

## Reproduction Assessment

Yes for the source-level crash class: a retained assistant message with a nullish or sparse content entry reaches `block.type` in both the shared transform and Anthropic conversion paths on current main. The live MiniMax percentage remains provider-dependent.

## Solution Assessment

Yes, the narrowest maintainable solution is a guarded transport-boundary normalization plus regression tests. The closed PR is useful context, but the final fix should preserve current thinking/signature handling and avoid exposing hidden reasoning by default.
