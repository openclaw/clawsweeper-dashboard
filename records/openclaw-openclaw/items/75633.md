## Summary

Current main already blocks the reported default MiniMax Anthropic-compatible failure path by injecting `thinking: { type: "disabled" }` for `minimax` and `minimax-portal` Anthropic Messages requests unless the user explicitly overrides thinking. That prevents the thinking-only/no-text MiniMax response shape described in the report from reaching the Anthropic adapter on the normal path, and the behavior is covered by source, tests, docs, and current-main provenance.

## What This Changes

The issue asks OpenClaw to handle MiniMax Anthropic-compatible responses that contain thinking/no text without crashing the Anthropic Messages adapter.

## Best Possible Solution

Keep the current MiniMax-specific default thinking-disable wrapper and ship it in the next release; if a separate sparse or malformed transcript-content crash is reproduced outside this MiniMax no-text response path, track that as a narrower generic adapter hardening issue.

## Reproduction Assessment

Yes for the originally reported release path: a MiniMax Anthropic-compatible request with low output budget and no explicit thinking override can produce thinking-only output. On current main, the normal OpenClaw path blocks that shape by injecting disabled thinking before the provider request.

## Solution Assessment

Yes. For the reported default MiniMax setup, disabling MiniMax thinking at the payload boundary is narrower and safer than surfacing thinking as visible text, because the provider emits non-native reasoning content on this Anthropic-compatible stream path.
