## Summary

Keep open. Current main still has unguarded `block.type` reads in the shared transport transform and Anthropic request conversion path, while the only linked fix PR was closed unmerged, so the reported crash class remains actionable in this repository.

## What This Changes

The issue requests hardening the Anthropic messages adapter so MiniMax Anthropic-compatible responses with empty, thinking-only, null, or sparse assistant content do not crash cron-triggered sessions.

## Best Possible Solution

Add narrow defensive normalization at the transport boundary for assistant content arrays before any `block.type` reads, preserving valid text, toolCall, signed thinking, and redacted-thinking semantics. Cover empty, thinking-only, null, undefined, and sparse content with focused tests, and avoid exposing hidden reasoning as visible assistant text unless existing provider replay policy already makes that conversion intentionally.

## Reproduction Assessment

Yes for the source-level crash class: a replayed assistant message with `content: [undefined]`, `content: [null]`, or a sparse slot reaches `transformTransportMessages` and can throw before failed turns are filtered. The exact MiniMax live intermittency remains provider-dependent and was not reproduced live in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to guard and normalize malformed assistant content blocks in `transport-message-transform.ts` and the Anthropic conversion path, then add regression tests. A broader product change that surfaces thinking as user-visible text by default would be riskier because MiniMax docs already treat reasoning leakage as a concern.
