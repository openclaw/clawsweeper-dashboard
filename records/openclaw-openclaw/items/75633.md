## Summary

Keep open. Current main still has unguarded `block.type` dereferences on assistant content blocks in the Anthropic transport/replay path, and the related fix PR was closed unmerged, so the reported crash class is still actionable in this repository.

## What This Changes

The issue asks OpenClaw to handle MiniMax Anthropic-compatible responses with thinking-only, empty, or malformed content without throwing `Cannot read properties of undefined (reading 'type')`.

## Best Possible Solution

Add narrow defensive normalization around assistant content-block iteration in the shared transport transform and Anthropic conversion path, preserving current thinking/signature semantics, then cover empty, thinking-only, null, and sparse content cases with focused tests. If the provider returns no visible text, keep hidden reasoning private and use the existing continuation/fallback machinery rather than surfacing reasoning as normal assistant output by default.

## Reproduction Assessment

Yes for the source-level crash class: a sparse or nullish assistant content entry reaching `transformTransportMessages` or `convertAnthropicMessages` will hit `block.type` without a guard. The exact MiniMax live failure rate remains provider-dependent and was not reproduced live here.

## Solution Assessment

Unclear for the previously proposed PR because it was closed unmerged and only the compact context is available, but the direction is sound: guard malformed blocks at the transport boundary and add regression tests. A safer complete fix should also verify the thinking-only/no-visible-text path does not silently end the cron run.
