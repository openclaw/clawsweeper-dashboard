## Summary

Keep open. Current main still has unguarded `block.type` reads in the shared transport transform and Anthropic request conversion path, and the linked fix PR #75655 was closed without merging, so the reported crash class remains actionable in this repository.

## What This Changes

The issue requests hardening the Anthropic Messages adapter so MiniMax Anthropic-compatible empty, malformed, or thinking-heavy responses do not crash cron sessions with an unhandled `block.type` TypeError.

## Best Possible Solution

Add narrow defensive normalization before any assistant content-block `type` reads in the transport transform and Anthropic conversion path, preserving valid text, tool call, signed thinking, and redacted-thinking behavior while skipping malformed nullish/sparse blocks. Add focused regression coverage and avoid broad product changes that expose hidden reasoning unless an existing replay policy already intentionally converts it.

## Reproduction Assessment

Yes for the source-level crash class: an assistant replay turn with `content: [undefined]`, `content: [null]`, or a sparse slot can reach `transformTransportMessages` and throw while reading `block.type`. The provider-side MiniMax intermittency remains live-provider dependent and was not reproduced here.

## Solution Assessment

Yes, the best fix is a narrow transport-boundary guard/normalization in `transport-message-transform.ts` and `anthropic-transport-stream.ts` with regression tests. A broader fallback that always surfaces MiniMax thinking as visible text would be less maintainable and could leak reasoning content.
