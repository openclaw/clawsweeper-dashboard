## Summary

Keep open. Current main still has unguarded content-block dereferences in the Anthropic transport replay/conversion path, and the related fix PR was closed unmerged, so the reported crash class remains actionable in this repository.

## What This Changes

Request to make the `anthropic-messages` adapter handle MiniMax Anthropic-compatible thinking-only, empty, or malformed content responses without throwing a TypeError or ending cron sessions with no useful output.

## Best Possible Solution

Add narrow defensive normalization around assistant content-block iteration in the shared transport transform and Anthropic conversion path, preserving current thinking/signature semantics. Cover empty, thinking-only, null, and sparse content cases with focused tests, and let existing empty/reasoning-only retry machinery handle no-visible-text turns instead of surfacing hidden reasoning by default.

## Reproduction Assessment

Yes for the source-level crash class: an assistant message whose `content` array contains a null, undefined, or sparse entry will reach `block.type` in `transformTransportMessages` or Anthropic request reconstruction. The issue's MiniMax live frequency is nondeterministic, but the unsafe dereference is directly visible on current main.

## Solution Assessment

Yes: the narrowest maintainable fix is to guard and normalize malformed content at the transport boundary, then add regression tests around replay and Anthropic payload building. A broader provider-specific workaround alone would leave the shared replay path unsafe.
