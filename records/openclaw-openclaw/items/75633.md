## Summary

Keep open. Current main still has unguarded `block.type` dereferences in the shared transport replay and Anthropic Messages conversion paths, so a null, undefined, or sparse assistant content block can still trigger the reported TypeError before replay filtering finishes. The related fix PR is closed unmerged, leaving the issue actionable.

## What This Changes

Harden the Anthropic Messages adapter so MiniMax thinking-only, empty, or malformed assistant content does not crash cron-triggered sessions.

## Best Possible Solution

Add narrow defensive normalization around assistant content-block iteration in the shared transport transform and Anthropic conversion path, preserving signed/redacted thinking semantics and keeping hidden reasoning private. Cover empty, thinking-only, null, undefined, and sparse assistant content cases with focused regression tests, then let existing continuation or fallback behavior handle no-visible-text responses.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass an assistant message with a nullish or sparse `content` entry through `transformTransportMessages` or Anthropic conversion and current main reads `block.type` before any guard; the live MiniMax rate is provider-dependent.

## Solution Assessment

Yes for the recommended repair direction. The narrow maintainable fix is to guard and normalize malformed assistant content at the transport boundary, not to surface hidden thinking as ordinary assistant text by default.
