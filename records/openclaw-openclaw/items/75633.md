## Summary

Keep open. Current main has a generic empty-error retry that may reduce the reported `stopReason: "error"`, `usage.output: 0`, `content: []` symptom, but the Anthropic Messages replay/conversion code still dereferences `block.type` and `item.type` without guarding malformed or sparse content blocks, and the linked fix PR was closed unmerged. This remains a narrow, actionable adapter hardening bug rather than a cleanup close.

## What This Changes

The issue requests hardening the anthropic-messages adapter so MiniMax Anthropic-compatible thinking-only, empty, or malformed content responses do not crash cron sessions.

## Best Possible Solution

Land a focused adapter hardening PR that treats malformed/sparse content blocks as invalid input to skip or normalize, keeps thinking-only MiniMax/Anthropic-compatible turns from crashing, and adds regression coverage around both shared transport replay and Anthropic Messages conversion.

## Reproduction Assessment

Yes for the source-level failure path: an assistant message whose content array contains `undefined` or `null` reaches current `block.type` dereferences in the shared transform and Anthropic conversion. The exact live MiniMax rate is stochastic, but the issue includes a concrete curl shape and persisted session evidence.

## Solution Assessment

No current merged solution fully addresses the report. The best repair is a narrow guard/normalization change in the Anthropic transport and shared transform, with tests proving empty, thinking-only, null, and undefined content blocks do not throw or silently lose valid text/tool content.
