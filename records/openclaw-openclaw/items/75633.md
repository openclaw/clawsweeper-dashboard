## Summary

Keep open. Current main still has unguarded `block.type` reads in the shared transport transform and Anthropic Messages conversion path, and the only linked fix PR was closed unmerged, so the reported crash class remains actionable in this repository.

## What This Changes

Fix the Anthropic Messages adapter so MiniMax Anthropic-compatible responses with no visible text, thinking-only content, or malformed empty content do not crash cron-triggered sessions with a `TypeError`.

## Best Possible Solution

Add narrow defensive normalization at the transport boundary before any assistant content block dereference. Preserve valid text, toolCall, thinking, signed-thinking, and redacted-thinking semantics; skip or repair nullish malformed blocks; and avoid making hidden reasoning visible unless an existing replay policy intentionally converts it.

## Reproduction Assessment

Yes for the source-level crash class: an assistant message with `content: [undefined]`, `content: [null]`, or a sparse slot can reach `transformTransportMessages` and throw on `block.type`. The exact MiniMax live intermittency remains provider-dependent and was not reproduced live in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to guard and normalize malformed assistant content blocks in `transport-message-transform.ts` and the Anthropic conversion path, with focused regression tests; broadly surfacing thinking as text would be riskier because MiniMax reasoning leakage is already documented as a concern.
