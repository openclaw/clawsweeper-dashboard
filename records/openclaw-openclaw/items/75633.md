## Summary

Current main implements a targeted MiniMax Anthropic Messages guard for the reported failure mode: MiniMax and MiniMax Portal payloads now get thinking disabled unless explicitly overridden, the wrapper is applied on the normal embedded-run stream path, and regression tests cover both provider ids. The proof commit is present on main but is not contained in any local release tag, so this is main-only rather than shipped in the latest release.

## What This Changes

Make Anthropic Messages handling for MiniMax-compatible endpoints avoid crashing when the provider returns reasoning-only or otherwise no visible text content.

## Best Possible Solution

Keep the MiniMax-specific payload guard in main and ship it in the next release; reopen only if MiniMax still returns the malformed reasoning-only shape despite `thinking` being disabled by OpenClaw.

## Reproduction Assessment

Yes. The issue gives a concrete MiniMax curl shape and current-main source/tests provide a high-confidence local proof that the reported provider ids now avoid requesting that thinking-only shape by default.

## Solution Assessment

Yes. Disabling MiniMax thinking at the Anthropic-compatible payload boundary is the narrowest maintainable fix for this report because it leaves core provider contracts generic and avoids changing unrelated Anthropic-compatible providers.
