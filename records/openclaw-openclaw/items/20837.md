## Summary

Keep open. Current main and latest release still expose channel metadata in the normal inbound path but omit channel/capability/action/shell runtime metadata when `/context` and `/export-session` reconstruct the command system prompt; the linked repair PR closed unmerged, and selective reply routing remains a separate product/API decision.

## What This Changes

The issue requests that agents receive incoming communication-channel context and optionally support channel-selective replies.

## Best Possible Solution

Populate `/context` and `/export-session` prompt reconstruction from the existing embedded-runner runtime metadata pattern, then handle channel-selective reply routing as a separate product/API design.

## Reproduction Assessment

Yes, at source level: `/context` and `/export-session` rebuild prompts through `resolveCommandsSystemPromptBundle()`, which still omits channel-related runtime fields on current main. I did not run a live Telegram/dashboard scenario in this read-only review.

## Solution Assessment

Yes for the first part: reusing or extracting the embedded-runner runtime metadata pattern is the narrowest maintainable repair. No for selective reply routing as part of the same change; that needs separate product/API direction.
