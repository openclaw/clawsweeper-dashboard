## Summary

Keep open. Current main exposes channel metadata in the normal inbound run path, but slash-command prompt reconstruction still rebuilds runtime metadata without channel, capabilities, shell, or channel actions; the related fix PR #21271 was closed unmerged.

## What This Changes

The issue asks for agents to know which communication channel produced an inbound message, with optional future control over reply channel selection.

## Best Possible Solution

Populate runtime metadata in resolveCommandsSystemPromptBundle() using the existing embedded-runner pattern, and leave selective reply routing to a separate product/API discussion.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call resolveCommandsSystemPromptBundle() for a Telegram command turn and inspect the buildSystemPromptParams() runtime argument; current main omits channel metadata there while normal inbound metadata includes it.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or extract the existing runtime prompt assembly pattern for slash-command reconstruction only, without changing deterministic reply routing.
