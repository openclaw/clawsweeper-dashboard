- `impact:session-state`: The issue affects runtime context metadata that reconstructed session prompts expose to the agent.

## Summary

Keep open: current main still has the channel fact before command prompt assembly, but `/context` and `/export-session` prompt reconstruction still drops channel, shell, capabilities, and channel action metadata; the related repair PR is closed unmerged.

## What This Changes

The issue requests that agents know which communication channel produced a message and, optionally, use that context for channel-specific replies.

## Best Possible Solution

Thread the already-derived channel and account-scoped runtime facts through `resolveCommandsSystemPromptBundle()` so command-rebuilt prompts match normal agent-run prompts; leave selective reply routing as a separate product/API follow-up.

## Reproduction Assessment

Yes, at source level. Build a command context with `OriginatingChannel`, then follow `/context` or `/export-session` into `resolveCommandsSystemPromptBundle()`, where that channel is dropped before runtime prompt assembly.

## Solution Assessment

Yes for the awareness portion. The narrow maintainable fix is to reuse or extract the normal embedded-runner runtime metadata assembly for command reconstruction, not to add new reply-routing behavior here.
