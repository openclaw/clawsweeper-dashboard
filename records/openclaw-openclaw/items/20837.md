## Summary

Keep open. Current main exposes channel metadata in the normal inbound/run path, but the command prompt reconstruction used by `/context` and `/export-session` still drops the command channel and related runtime metadata; the linked implementation PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to expose the incoming communication channel to the agent and optionally let replies be limited to selected channels.

## Best Possible Solution

Thread existing channel/account runtime facts through `resolveCommandsSystemPromptBundle()` so `/context` and `/export-session` match normal run prompt metadata while keeping reply delivery policy host-controlled.

## Reproduction Assessment

Yes for the scoped remaining gap: source inspection on current main shows `buildCommandContext()` derives a channel, while `resolveCommandsSystemPromptBundle()` drops it before prompt reconstruction. I did not run a live Telegram/webchat session in this read-only review.

## Solution Assessment

Yes for the awareness portion. The narrow maintainable path is to reuse or extract the embedded-runner runtime prompt assembly for command reconstruction, while leaving prompt-driven delivery routing as a separate product/API decision.
