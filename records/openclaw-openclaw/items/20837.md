## Summary

Keep open. Current main already exposes channel metadata to the normal inbound path, but the slash-command prompt reconstruction still omits runtime channel, shell, capability, action, and reaction/message guidance metadata while `/context` and `/export-session` rely on that bundle. The linked fix PR #21271 was closed unmerged, so the narrow command-path gap remains actionable.

## What This Changes

The issue requests that agents receive trusted context about the communication channel that produced a message, with optional future support for channel-selective replies.

## Best Possible Solution

Implement the missing slash-command runtime metadata in `resolveCommandsSystemPromptBundle()` only, reusing the normal embedded-runner prompt pattern so reconstructed `/context` and `/export-session` prompts match normal agent runs without changing deterministic reply routing.

## Repair Work Prompt

Fix this issue by updating `resolveCommandsSystemPromptBundle()` only. In `src/auto-reply/reply/commands-system-prompt.ts`, mirror or extract the runtime prompt metadata pattern from `src/agents/pi-embedded-runner/run/attempt.ts`: resolve machine name, `os.type()/release()/arch()`, `detectRuntimeShell()`, normalized runtime channel from the command/message context, account-aware channel capabilities using the command context account id, message-tool capabilities, channel actions, message-tool hints, and reaction guidance where available. Pass the resulting `runtimeInfo`, `messageToolHints`, and `reactionGuidance` into `buildAgentSystemPrompt()` so `/context` and `/export-session` reconstructed prompts match normal agent runs. Preserve deterministic host-controlled reply routing; do not implement model-selected channel routing or change delivery behavior. Adapt useful ideas from closed PR #21271, but fix the account-id issue noted there. Add focused tests in `commands-system-prompt.test.ts` for channel, account-aware capabilities, shell/host/os fields, message-tool hints/actions, and safe fallback when channel is missing. Add or adjust changelog only if the final change is treated as user-facing under repo policy, with proper thanks attribution.
