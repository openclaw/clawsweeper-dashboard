## Summary

Keep open. Current main already gives the agent source-channel metadata in the normal inbound run path, but the slash-command prompt reconstruction path still rebuilds runtime info with placeholder host fields and no channel/capability/shell/action metadata; the linked fix PR was closed unmerged, so the remaining gap is still actionable.

## What This Changes

The issue asks for agents to know which communication channel a message came from, with selective channel reply routing left as an optional follow-up.

## Best Possible Solution

Implement the missing command prompt runtime metadata in `resolveCommandsSystemPromptBundle()` by reusing or extracting the embedded-runner runtime assembly pattern, with regression tests for `/context` and `/export-session`; leave selective channel reply routing to a separate product/API follow-up.

## Reproduction Assessment

Yes. A focused unit test can call `resolveCommandsSystemPromptBundle()` with `OriginatingChannel: "telegram"` and assert that `buildSystemPromptParams()` receives runtime channel/capability/shell data; current main instead sends placeholder host fields and no channel metadata.

## Solution Assessment

Yes for the agent-awareness portion: the narrow maintainable fix is to populate the command prompt reconstruction path from existing channel/runtime helpers. Prompt-controlled selective routing is not the best fix here because docs define deterministic host-controlled routing.
