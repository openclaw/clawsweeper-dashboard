## Summary

Keep open. Current main already exposes channel metadata in the normal inbound/runtime path, but slash-command prompt reconstruction for `/context` and `/export-session` still drops channel, capabilities, shell, reaction/message guidance, and channel-action metadata; the linked implementation PR is closed unmerged and no replacement PR was found.

## What This Changes

The issue requests that agents know which communication channel a message came from, with optional future control over which channel receives replies.

## Best Possible Solution

Thread existing channel/account runtime facts through `resolveCommandsSystemPromptBundle()` so `/context` and `/export-session` match normal run prompt metadata while keeping delivery routing host-controlled.

## Reproduction Assessment

Yes for the scoped remaining gap. Source inspection on current main shows `buildCommandContext()` derives a channel, while `resolveCommandsSystemPromptBundle()` drops that channel and related runtime metadata before prompt reconstruction.

## Solution Assessment

Yes for the awareness portion. The narrow maintainable solution is to reuse or extract the embedded-runner runtime prompt assembly for command reconstruction, without adding prompt-driven delivery-channel policy.
