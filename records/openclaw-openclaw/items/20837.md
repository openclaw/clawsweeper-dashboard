## Summary

Keep open: current main still exposes channel metadata in the normal inbound and live runner paths, but `/context` and `/export-session` prompt reconstruction still omits channel, capability, shell, and channel-action runtime metadata. The linked repair PR is closed unmerged, and selective reply routing remains a separate product/API decision.

## What This Changes

Request that agents receive communication-channel context and optionally support channel-selective replies.

## Best Possible Solution

Split the work: repair `resolveCommandsSystemPromptBundle()` to reuse or extract the embedded-runner runtime metadata assembly, and leave channel-selective reply behavior for a separate maintainer-designed routing/API change.

## Reproduction Assessment

Yes, at source level: `/context` and `/export-session` rebuild prompts through `resolveCommandsSystemPromptBundle()`, which still omits channel-related runtime fields on current main and v2026.6.6. I did not run a live Telegram/dashboard scenario in this read-only review.

## Solution Assessment

Yes for the first part: reusing or extracting the embedded-runner runtime metadata pattern is the narrowest maintainable repair. No for selective reply routing in the same change; that needs separate maintainer product/API direction.
