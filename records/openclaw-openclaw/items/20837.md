## Summary

Keep open. Current main still drops already-derived channel/account runtime metadata when reconstructing command prompts for `/context` and `/export-session`, while normal agent runs and inbound metadata already carry channel; the linked implementation at https://github.com/openclaw/openclaw/pull/21271 is closed unmerged.

## What This Changes

The issue asks OpenClaw to expose the originating communication channel to the agent and eventually support channel-selective replies.

## Best Possible Solution

Thread existing channel/account runtime facts through `resolveCommandsSystemPromptBundle()` so reconstructed command prompts match normal agent runs, while leaving delivery-channel policy to existing routing and separate product follow-up.

## Reproduction Assessment

Yes for the scoped remaining gap. Source inspection on current main shows `buildCommandContext()` derives a channel, while `resolveCommandsSystemPromptBundle()` drops that channel and related runtime metadata before `/context` and `/export-session` prompt reconstruction.

## Solution Assessment

Yes for the awareness portion. The narrow maintainable fix is to reuse or extract the normal embedded-runner runtime prompt assembly for command reconstruction, without adding prompt-driven delivery-channel policy.
