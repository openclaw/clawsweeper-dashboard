## Summary

Keep open. Current main exposes channel metadata in the normal inbound/runtime prompt path, but the slash-command reconstruction bundle used by `/context` and `/export-session` still rebuilds runtime info without channel, capabilities, shell, or channel-specific prompt guidance; the linked implementation PR is closed unmerged.

## What This Changes

The issue asks for agents to know which communication channel produced an incoming message and, optionally, to support channel-selective replies.

## Best Possible Solution

Thread the existing prepared channel/runtime facts through `resolveCommandsSystemPromptBundle()` so `/context` and `/export-session` match normal run prompt metadata, while leaving selective reply routing to a separate product/API discussion.

## Reproduction Assessment

Yes for the scoped remaining gap. Source inspection on current main shows the command context derives a channel, while `resolveCommandsSystemPromptBundle()` still drops that channel and related runtime metadata before prompt reconstruction.

## Solution Assessment

Yes for the scoped remainder. The narrow maintainable fix is to reuse or extract the embedded-runner runtime prompt assembly for command reconstruction, without adding prompt-driven delivery-channel policy.
