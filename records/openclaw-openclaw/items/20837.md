## Label Justifications

- `P2`: The remaining gap is a normal-priority command prompt reconstruction defect with limited blast radius and a clear source-level fix boundary.
- `impact:session-state`: The issue affects runtime context metadata that reconstructed session prompts expose to the agent.
- `impact:message-loss`: Missing current-channel capabilities and action guidance can make command-reconstructed prompts ambiguous about channel message targeting.

## Summary

Keep open. Current main still carries channel metadata in the normal inbound and agent-run paths, but the slash-command prompt reconstruction path used by `/context` and `/export-session` still drops the already-derived channel/runtime metadata; the related fix attempt is closed unmerged at https://github.com/openclaw/openclaw/pull/21271.

## What This Changes

The issue asks for agents to know which communication channel produced an incoming message, with channel-selective reply routing left as optional follow-up behavior.

## Best Possible Solution

Thread the already-derived channel and account-scoped runtime facts through `resolveCommandsSystemPromptBundle()` so `/context` and `/export-session` match normal agent-run prompts, while leaving selective reply routing for a separate product/API follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a command context with `OriginatingChannel`, then follow `/context` or `/export-session` into `resolveCommandsSystemPromptBundle()`, where that channel is dropped before runtime prompt assembly.

## Solution Assessment

Yes for the awareness portion. The narrow maintainable fix is to reuse or extract the normal embedded-runner runtime metadata assembly for command reconstruction, not to add new reply-routing behavior in this issue.
