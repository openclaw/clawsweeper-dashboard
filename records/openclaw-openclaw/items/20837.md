## Summary

Keep open. Current main already exposes channel metadata in the normal inbound/runtime prompt path, but the slash-command reconstruction bundle used by `/context` and `/export-session` still drops channel, capabilities, shell, and channel-action prompt metadata; the linked implementation PR is closed unmerged and no replacement PR was found.

## What This Changes

Request to expose the originating communication channel to the agent, with optional future control over which channel receives replies.

## Best Possible Solution

Thread the existing channel/account runtime facts through `resolveCommandsSystemPromptBundle()` so `/context` and `/export-session` match normal run prompt metadata while keeping reply routing host-controlled.

## Reproduction Assessment

Yes for the scoped remaining gap. Source inspection on current main shows the command context derives a channel, while `resolveCommandsSystemPromptBundle()` drops that channel and related runtime metadata before prompt reconstruction.

## Solution Assessment

Yes for the scoped awareness work. The narrow maintainable solution is to reuse or extract the embedded-runner runtime prompt assembly for command reconstruction, without adding prompt-driven delivery-channel policy.
