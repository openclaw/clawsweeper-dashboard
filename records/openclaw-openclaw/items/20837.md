## Summary

Keep open. Current main still has the scoped channel-awareness gap: normal inbound and agent-run prompts carry channel metadata, but `/context` and `/export-session` prompt reconstruction still drops channel, account capabilities, shell, and channel action guidance; the linked implementation PR is closed unmerged.

## What This Changes

The issue requests that agents know which communication channel a message came from and optionally support channel-selective replies.

## Best Possible Solution

Thread the already-derived channel/account runtime facts through `resolveCommandsSystemPromptBundle()` so command prompt reconstruction matches normal agent runs, while leaving delivery-channel policy to existing routing and a separate product follow-up.

## Reproduction Assessment

Yes for the scoped remaining gap. Source inspection on current main shows `buildCommandContext()` derives a channel, while `resolveCommandsSystemPromptBundle()` drops that channel and related runtime metadata before `/context` and `/export-session` prompt reconstruction.

## Solution Assessment

Yes for the awareness portion. The narrow maintainable fix is to reuse or extract the normal embedded-runner runtime prompt assembly for command reconstruction, without adding prompt-driven delivery-channel policy.
