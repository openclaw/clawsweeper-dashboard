## Label Justifications

- `P2`: The remaining gap is a normal-priority prompt reconstruction defect with limited blast radius and a clear source-level fix boundary.
- `impact:session-state`: The issue affects runtime context metadata that reconstructed session prompts expose to the agent.
- `impact:message-loss`: Missing current-channel capabilities and message action guidance can make command-reconstructed prompts ambiguous about source-channel delivery behavior.

## Summary

Keep open: current main and the latest release still expose channel metadata in the normal inbound/agent-run path, but reconstructed prompts for `/context` and `/export-session` still pass `unknown` runtime fields and omit channel, capability, shell, and message-action metadata; the linked repair pull request at https://github.com/openclaw/openclaw/pull/21271 is closed unmerged.

## What This Changes

The issue requests communication-channel context in agent prompts, with optional follow-up behavior for channel-selective replies.

## Best Possible Solution

Thread the already-derived channel and account-scoped runtime facts through `resolveCommandsSystemPromptBundle()` so `/context` and `/export-session` rebuilt prompts match normal agent-run prompts, while leaving selective reply routing to a separate product/API follow-up.

## Reproduction Assessment

Yes, at source level. Build a command context with `OriginatingChannel`, then follow `/context` or `/export-session` into `resolveCommandsSystemPromptBundle()`, where that channel is dropped before runtime prompt assembly.

## Solution Assessment

Yes for the channel-awareness portion. The narrow maintainable fix is to reuse or extract the normal embedded-runner runtime metadata assembly for command prompt reconstruction, not to add prompt-directed selective reply routing here.
