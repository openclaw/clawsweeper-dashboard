## Label Justifications

- `P2`: The remaining gap is a normal-priority command prompt reconstruction defect with limited blast radius and a clear source-level fix boundary.
- `impact:session-state`: The issue affects runtime context metadata that reconstructed session prompts expose to the agent.
- `impact:message-loss`: Missing current-channel capabilities and action guidance can make reconstructed prompts ambiguous about channel message targeting.

## Summary

Keep open. Current main still carries channel metadata in the normal inbound and agent-run paths, but slash-command prompt reconstruction for `/context` and `/export-session` drops the already-derived runtime channel metadata before building the system prompt; the related fix attempt at https://github.com/openclaw/openclaw/pull/21271 was closed unmerged.

## What This Changes

The issue asks for agents to receive source-channel context for inbound messages and, optionally, support channel-selective replies.

## Best Possible Solution

Thread the already-derived channel plus account-scoped capabilities, shell, host/OS/arch, and supported channel actions through `resolveCommandsSystemPromptBundle()` so `/context` and `/export-session` match normal agent-run prompts, while leaving selective reply routing to the separate forced-origin routing discussion.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a command context with `OriginatingChannel`, then follow `/context` or `/export-session` into `resolveCommandsSystemPromptBundle()`, where that channel is not forwarded into runtime prompt assembly.

## Solution Assessment

Yes for the awareness portion. The narrow maintainable fix is to reuse or extract the normal embedded-runner runtime metadata assembly for command reconstruction, not to add selective reply routing in this issue.
