## Label Justifications

- `P2`: The remaining gap is a normal-priority command prompt reconstruction defect with limited blast radius and a clear source-level fix boundary.
- `impact:session-state`: The issue affects runtime context metadata that reconstructed session prompts expose to the agent.
- `impact:message-loss`: Missing channel capabilities and message guidance can make command-reconstructed prompts ambiguous about channel message targeting.

## Summary

Keep open. Current main still carries the already-derived channel through command context, but `/context` and `/export-session` prompt reconstruction still rebuild the runtime prompt with `host/os/arch` as `unknown` and without channel, shell, capabilities, reaction guidance, or message-tool hints.

## What This Changes

Request to make agents aware of the inbound communication channel and eventually support channel-selective replies.

## Best Possible Solution

Thread the already-derived source channel and account-scoped runtime facts through `resolveCommandsSystemPromptBundle()` so command-reconstructed prompts match normal agent-run prompts, while leaving selective routing as a separate follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a command context with `OriginatingChannel`, then follow `/context` or `/export-session` into `resolveCommandsSystemPromptBundle()`, where that channel is dropped before runtime prompt assembly.

## Solution Assessment

Yes for the awareness portion. The maintainable fix is to reuse or extract the normal embedded-runner runtime metadata assembly for command reconstruction, not to add new reply-routing behavior in this issue.
