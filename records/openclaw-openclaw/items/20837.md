## Summary

Keep open: current main and v2026.6.5 expose channel metadata in normal inbound agent runs, and current main now forwards channel identity to CLI hooks, but `/context` and `/export-session` still rebuild prompts with unknown runtime fields and without the canonical channel/capability/action/shell metadata. The linked repair PR is closed unmerged, and selective reply routing remains a separate product/API decision.

## What This Changes

The issue requests exposing the inbound communication channel to the agent, with optional selective reply routing by channel.

## Best Possible Solution

Use one canonical runtime-metadata assembly for normal runs and command prompt reconstruction, preserving account-aware channel facts while leaving selective cross-channel delivery for a separately scoped product/API proposal.

## Reproduction Assessment

Yes for the narrow gap: current source shows `/context` and `/export-session` reconstruct prompts without the channel runtime metadata available to normal runs. The optional selective-routing behavior is a feature/product request rather than a reproduced existing-behavior failure.

## Solution Assessment

Yes for the narrow repair: sharing the canonical runtime metadata assembly is cleaner and safer than adding another partial copy. No for implementing selective routing in the same change, because delivery policy requires separate product and API judgment.
