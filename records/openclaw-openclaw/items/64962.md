## Summary

Keep open: current main still has the reported timeout-compaction non-reduction path, and the adjacent open truncation-first PR only targets ordinary tool-heavy overflow recovery rather than this timeout-compaction escalation case.

## What This Changes

The issue requests an escalation ladder when timeout-triggered compaction runs but does not reduce context enough to recover from overflow.

## Best Possible Solution

Define a bounded timeout-overflow recovery ladder in the embedded runner that treats non-reducing timeout compaction as recoverable while making the allowed lossy step explicit and covered by regression tests.

## Reproduction Assessment

Yes at source level. Current main has focused tests where high-token timeout compaction returns `compacted: false` and the runner still returns a blocked timed-out result; this read-only review did not run the test suite.

## Solution Assessment

Unclear as a full implementation plan. The recovery direction is valid, but maintainers should decide the permitted order and lossiness of fallback model use, `compact_then_truncate`, and session rotation before code changes.
