## Summary

Keep open: current main still has the reported timeout-compaction non-reduction path, and the adjacent open truncation-first PR only targets ordinary tool-heavy overflow recovery rather than this timeout-compaction escalation case.

## What This Changes

The issue requests a timeout-compaction recovery ladder when compaction runs but returns `compacted: false` or otherwise does not reduce context enough.

## Best Possible Solution

Define a bounded timeout-recovery ladder in `src/agents/pi-embedded-runner/run.ts` that treats non-reducing timeout compaction as recoverable and uses existing fallback or truncation seams only within an approved lossiness policy.

## Reproduction Assessment

Yes, at source level. Current main has a focused high-prompt-token timeout test where compaction returns `compacted: false`, and the runner still returns a blocked timed-out result.

## Solution Assessment

Unclear as an implementation plan. The requested escalation direction is plausible, but maintainers should decide the allowed order and lossiness of fallback model use, `compact_then_truncate`, and any session rotation before code changes.
