## Summary

Keep open: current main still preserves the reported timeout-compaction non-reduction path. Timeout recovery compaction only retries the prompt when `compacted` is true; `compacted: false` still logs a warning and falls through to the normal timeout error payload.

## What This Changes

The issue requests a recovery ladder when timeout-triggered compaction runs but does not reduce context enough to avoid timeout or overflow failure.

## Best Possible Solution

Add a deliberate timeout-compaction recovery ladder in `src/agents/pi-embedded-runner/run.ts`: after timeout recovery returns `compacted: false` or throws, use the approved compaction fallback policy when available, then reuse existing truncation or compact-then-truncate recovery where applicable, and only then surface a clear blocked/reset outcome. Update timeout-triggered compaction tests so non-reduction asserts escalation instead of timeout fall-through.

## Reproduction Assessment

Yes. The focused `falls through to normal handling when timeout compaction fails` test reproduces the core path with high prompt usage and a `compacted: false` result, and current source shows the same fall-through branch.

## Solution Assessment

No. Current main is not the best solution because it treats non-reducing timeout compaction as ordinary timeout handling; the safer direction is a bounded recovery ladder coordinated with the existing compaction fallback work.
