## Summary

Keep open: current main still preserves the reported timeout-compaction non-reduction path. The recently merged compaction model fallback work covers fallback-eligible compaction errors, but a `compacted: false` timeout recovery still falls through to timeout handling instead of a truncation/reset recovery ladder.

## What This Changes

Request timeout-triggered compaction to escalate when compaction completes but does not reduce context enough, rather than returning a timeout or context-overflow error.

## Best Possible Solution

Define a maintainer-approved timeout recovery ladder in `src/agents/pi-embedded-runner/run.ts` that treats `compacted: false` as recoverable, reuses existing fallback/truncation seams where appropriate, and surfaces a deliberate reset or blocked outcome only after those steps are exhausted.

## Reproduction Assessment

Yes, at source level. Current main has a focused test path for high prompt-token timeout plus `compacted: false`, and the current branch still falls through to a blocked timeout payload.

## Solution Assessment

No. Current main is not the best solution because it handles non-reducing timeout compaction as ordinary timeout handling; the safer direction is a bounded recovery ladder coordinated with existing compaction fallback and truncation behavior.
