## Summary

Keep open: current main still preserves the reported timeout-compaction non-reduction path. The runner only retries when timeout compaction returns `compacted: true`; `compacted: false` still falls through to a timeout error payload, and the focused tests continue to assert that behavior.

## What This Changes

The issue asks for timeout-triggered compaction to escalate when a compaction attempt returns without reducing context instead of surfacing a timeout/context overflow failure.

## Best Possible Solution

Add a deliberate timeout-compaction recovery ladder: after timeout recovery returns `compacted: false` or throws, use the approved compaction fallback model policy when available, then reuse existing truncation/compact-then-truncate recovery where applicable, and only then surface a clear blocked/reset outcome. Keep the model fallback contract coordinated with #64960/#74470 and update the timeout-triggered compaction tests so non-reduction asserts escalation instead of timeout fall-through.

## Reproduction Assessment

Yes. The focused `falls through to normal handling when timeout compaction fails` test reproduces the core path with high prompt usage and a `compacted: false` result, and current source shows the same fall-through branch.

## Solution Assessment

No. Current main is not the best solution because it treats a non-reducing timeout compaction as ordinary timeout handling; the safer direction is an explicit, bounded recovery ladder coordinated with the existing compaction fallback work.
