## Summary

Keep open: current main still preserves the reported timeout-compaction non-reduction path. The closed umbrella issue points this exact recovery ladder back here, and the merged compaction fallback work only covers fallback-eligible compaction errors, not a `compacted: false` result.

## What This Changes

The issue requests an escalation path when timeout-triggered compaction runs but fails to reduce context.

## Best Possible Solution

Define a bounded timeout-recovery ladder in `src/agents/pi-embedded-runner/run.ts` that treats `compacted: false` as recoverable, reuses existing fallback or truncation seams where policy allows, and only surfaces a blocked timeout after those steps are exhausted.

## Reproduction Assessment

Yes, at source level. Current main has a focused high-prompt-token timeout test where compaction returns `compacted: false`, and the runner still falls through to a blocked timeout result.

## Solution Assessment

No, current main is not the best solution because it treats non-reducing timeout compaction as ordinary timeout handling. The requested escalation direction is plausible, but the exact truncation or session-rotation ladder needs maintainer policy approval.
