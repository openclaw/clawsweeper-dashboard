## Summary

Keep open: current main still has a source-reproducible timeout-compaction non-reduction path. The compaction model fallback work covers fallback-eligible provider errors, but it does not route `compacted: false` timeout recovery into `compact_then_truncate` or an approved reset/rotation ladder.

## What This Changes

Request timeout-triggered compaction to escalate from non-reducing compaction into model fallback, compact-then-truncate, and fresh-session recovery instead of returning a timeout/context-overflow outcome.

## Best Possible Solution

Define a maintainer-approved bounded timeout recovery ladder in `src/agents/pi-embedded-runner/run.ts` that treats `compacted: false` as recoverable, reuses existing fallback and truncation seams where appropriate, and only surfaces a blocked/reset outcome after those steps are exhausted.

## Reproduction Assessment

Yes, at source level. Current main has focused timeout-triggered compaction tests for high prompt-token timeout plus `compacted: false`, and those tests still assert a blocked timeout payload rather than an escalation path.

## Solution Assessment

No. Current main is not the best solution because it handles non-reducing timeout compaction as ordinary timeout handling; the safer direction is a bounded recovery ladder coordinated with existing compaction fallback and truncation behavior.
