## Summary

Keep open. Current main still lacks the core phase-aware `agent.wait` timeout attribution contract: gateway wait snapshots, lifecycle snapshots, caller-facing wait types, and protocol docs remain flat, while the broad implementation PR was closed unmerged and adjacent tracks remain split.

## What This Changes

The issue asks to distinguish live gateway timeout phases, prevent false provider/auth cooldown attribution, reduce post-turn lane pressure, and clean up QA/Telegram incident noise.

## Best Possible Solution

Keep this as the canonical timeout-attribution tracker, implement the remaining Gateway/agent contract additively with tests and docs, and leave scheduler fairness to #54488 and QA cleanup/readiness to #65506 unless maintainers intentionally merge those tracks.

## Reproduction Assessment

Yes for the remaining source-level contract gap: current `agent.wait` returns only a flat timeout when no lifecycle or dedupe snapshot resolves, and aborted lifecycle ends collapse to `status: "timeout"`. The live provider incident itself was not rerun.

## Solution Assessment

No as one broad umbrella implementation. The maintainable path is a narrow additive Gateway/agent timeout-attribution contract, with scheduler and QA harness repairs kept in their linked tracks.
