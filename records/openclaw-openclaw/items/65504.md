## Summary

Keep open. Current main still lacks the core phase-aware live-gateway timeout attribution: `agent.wait`, lifecycle snapshots, gateway dedupe snapshots, caller types, and docs still expose flat timeout states, while the related implementation PR is closed unmerged. The narrower auth-profile split-out was closed as a duplicate of this issue, and the QA/readiness and followup-lane starvation tracks remain open separately.

## What This Changes

The issue asks for live gateway failures to distinguish queue, preflight, provider, post-turn, and gateway-draining timeout phases, avoid false Anthropic/auth-profile cooldowns, reduce session-lane latency, and prevent QA/Telegram incident noise.

## Best Possible Solution

Keep this as the canonical OpenClaw timeout-attribution tracking issue: add additive phase/timing metadata through embedded-run telemetry, lifecycle snapshots, dedupe snapshots, `agent.wait`, caller types, protocol docs, and regression tests, while leaving followup-lane fairness to #54488 and qa-lab cleanup/readiness to #65506.

## Reproduction Assessment

Yes for the remaining core contract gap: current `agent.wait` can be exercised with a runId that has no lifecycle or dedupe snapshot and it returns only a flat timeout status. The original live Anthropic incident was not rerun in this read-only review, but source, tests, and docs give a high-confidence reproduction of the missing attribution behavior.

## Solution Assessment

No as one broad umbrella PR; the maintainable path is to split the work. The timeout-attribution contract should be a focused additive gateway/agent change, while #54488 and #65506 continue to track the scheduler and QA harness repairs.
