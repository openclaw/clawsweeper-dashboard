## Summary

Keep open. Current main still lacks the core phase-aware `agent.wait` timeout attribution contract: gateway wait snapshots, lifecycle snapshots, caller types, and protocol docs remain flat, while the broad implementation PR was closed unmerged. Some adjacent work appears partially addressed on main, such as timeout auth-profile cooldown suppression and deferred context-engine maintenance, but the remaining gateway contract work is still valid and overlaps open #54488 and #65506.

## What This Changes

The issue requests phase-aware live gateway timeout attribution and responsiveness fixes across `agent.wait`, embedded-run auth-profile cooldowns, LCM post-turn scheduling, QA gateway cleanup, and Telegram restart noise.

## Best Possible Solution

Keep this as the canonical timeout-attribution tracker, but implement the remaining work as a focused additive Gateway/agent contract change: carry phase and timing metadata through embedded-run telemetry, lifecycle snapshots, dedupe snapshots, `agent.wait`, caller types, protocol docs, and regression tests. Leave followup-lane fairness to #54488 and QA cleanup/readiness to #65506 unless maintainers intentionally merge those tracks back together.

## Reproduction Assessment

Yes for the remaining source-level contract gap: a current-main `agent.wait` with no lifecycle or dedupe terminal snapshot returns only a flat timeout, and aborted lifecycle ends also collapse to `status: "timeout"`. The live provider incident itself was not rerun, but the missing attribution is directly reproducible from the handler/types/tests surface.

## Solution Assessment

No as one broad umbrella implementation; the maintainable path is to split the work. The timeout-attribution contract should be a narrow additive gateway/agent change, while the scheduler and QA harness repairs continue in their existing related tracks.
