- `P0`: The PR targets an access-control boundary bypass for persistent allowlist writes that affect future sender authorization.
- `merge-risk: 🚨 security-boundary`: The current patch can leave a disabled command-origin channel able to mutate another channel's pairing allowlist store.

## Summary

Keep open: the PR is a focused security fix, but the new store-only guard still uses the selected allowlist channel/account as both the command origin and the target, so a disabled command-origin channel can remain unblocked when targeting another pairing store. External real behavior proof is also still missing.

## What This Changes

The PR adds a `configWrites` denial check and focused tests for store-targeted `/allowlist add|remove --store` edits.

## Best Possible Solution

Use the actual command origin for the origin-side `configWrites` check, keep the selected allowlist channel/account as the target, and land with regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows the current-main store-only path skips the `configWrites` guard, and the PR's added guard still permits a source-reproducible cross-channel case because it uses the selected target as the origin.

## Solution Assessment

No. Reusing the existing `configWrites` gate is the right direction, but the patch should preserve true command origin separately from the selected allowlist target before merge.
