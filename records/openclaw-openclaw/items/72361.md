## Summary

Keep this PR open. The reported access-control bug is real on current main and the branch is directionally useful, but the proposed store guard still authorizes against the selected target channel rather than the actual source channel, leaving a cross-channel `configWrites=false` bypass; the user-facing security fix also needs a changelog entry before merge.

## What This Changes

The branch adds a `configWrites` guard before `/allowlist --store` pairing-store mutations and adds allowlist command tests for blocked and allowed same-channel store edits.

## Best Possible Solution

Model `/allowlist` authorization with distinct scopes: derive the origin from the actual command source channel/account, derive the target from the selected `--channel`/`--account` allowlist scope, apply that consistently to config and store edits, and cover same-channel blocked, target-account blocked, allowed, and cross-channel source-denied cases with tests plus an active changelog fix entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main `/allowlist add dm --store 789` from a write-disabled pairing-capable channel skips the config branch guard and reaches the pairing-store mutation; the PR gap can be reproduced with a source-disabled Telegram command targeting an enabled Discord channel via `--channel discord`.

## Solution Assessment

No. Adding the store guard is the right direction, but using the selected target `channelId` as the guard origin is not the narrowest maintainable fix; the safer fix separates origin and target scopes before both config and store authorization.
