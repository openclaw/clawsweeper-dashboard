## Summary

Keep open. The underlying bug is real on current main, but the proposed patch is incomplete: it authorizes store writes using the selected target channel/account instead of the actual command source, leaving a cross-channel `configWrites=false` bypass. The PR also lacks the required user-facing changelog entry.

## What This Changes

The PR adds a `configWrites` guard before store-targeted `/allowlist add|remove` pairing-store mutations and adds focused tests for channel-blocked, account-blocked, and allowed store edits.

## Best Possible Solution

Carry the source and target scopes separately through `/allowlist`: derive the origin from `params.command.channelId`/`params.command.channel` and the current context account, derive the target from the selected allowlist channel/account, apply the same policy to config and store edits, add same-channel and cross-channel regression tests, and add the active-version changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: on current main, send `/allowlist add dm --store --channel discord 789` from an owner on a Telegram source with `channels.telegram.configWrites=false` while Discord writes are enabled; the store-only path reaches the pairing-store update without checking the Telegram origin.

## Solution Assessment

No. The PR adds the right kind of guard for same-channel store edits, but it wires the guard with the selected target scope as the origin; the safer solution is a distinct origin/target authorization path plus a cross-channel regression test.
