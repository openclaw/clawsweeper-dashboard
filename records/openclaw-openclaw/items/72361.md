## Summary

Keep open: current main still has the store-write gap this PR targets, but the PR guard authorizes the selected allowlist target as though it were the command origin, leaving a cross-channel/configWrites bypass. The prior ClawSweeper review raised the same source-vs-target issue, and source inspection confirms it.

## What This Changes

The branch adds a configWrites guard before store-targeted `/allowlist add|remove` pairing-store mutations and adds three focused allowlist command tests.

## Best Possible Solution

Carry the command origin and selected allowlist target as separate scopes, apply configWrites to both store and config edits, and add same-channel plus cross-channel regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send a store-targeted `/allowlist` command from a write-disabled source channel while selecting a different write-enabled target with `--channel`; the PR guard checks the target as the origin.

## Solution Assessment

No. The PR adds the right class of guard for same-channel store edits, but the maintainable fix must authorize the true command origin separately from the selected store target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Authorize store writes against the command origin:** `src/auto-reply/reply/commands-allowlist.ts:562-564`
  - body: This call passes `channelId`/`accountId`, which can come from `--channel`/`--account`, into a helper that treats them as the command origin. A write-disabled source can choose a write-enabled target and still mutate that target's pairing store, so the store guard needs the actual command source scope plus the selected target scope.
  - confidence: 0.88
