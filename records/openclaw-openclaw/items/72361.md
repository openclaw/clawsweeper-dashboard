## Summary

Keep this PR open: it targets a real current-main access-control bug, but the proposed guard still uses the selected allowlist target as the command origin, leaving a cross-channel `configWrites=false` bypass.

## What This Changes

The PR adds a `configWrites` guard before store-targeted `/allowlist add|remove` pairing-store mutations and adds focused tests for blocked and allowed store edits.

## Best Possible Solution

Carry the command origin and selected allowlist target as separate scopes, apply `configWrites` to store and config edits, add same-channel and cross-channel regression tests, and add the active-version changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send a store-targeted `/allowlist` command from a write-disabled source channel while selecting a different write-enabled target with `--channel`; the PR guard checks the target as the origin.

## Solution Assessment

No. The PR adds the right class of guard for same-channel store edits, but the maintainable fix is to authorize the true command origin separately from the selected store target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:
