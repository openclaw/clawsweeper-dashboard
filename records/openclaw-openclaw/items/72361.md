## Summary

Keep this PR open: it is useful source work for #72360, but it is not mergeable yet because the new store guard authorizes the selected target channel/account as the command origin, leaving a cross-channel write-disabled-origin bypass, and the required changelog entry is missing.

## What This Changes

The PR adds a `configWrites` guard before store-targeted `/allowlist add|remove` pairing-store mutations and adds focused tests for blocked and allowed store edits.

## Best Possible Solution

Model command origin and selected allowlist target as separate scopes, enforce `configWrites` for store and config writes against both scopes, add same-channel and cross-channel regression coverage, and add the active-version changelog fix entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send `/allowlist add dm --store --channel <write-enabled-pairing-channel> <id>` from a different write-disabled source channel; the PR guard checks the selected target as the origin.

## Solution Assessment

No. The PR adds the right class of guard for same-channel store edits, but the maintainable fix is to authorize the true command origin separately from the edited store target.

## Review Findings
