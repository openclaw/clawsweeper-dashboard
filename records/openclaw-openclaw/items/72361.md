## Summary

Keep open: the PR is directionally correct for the same-channel `/allowlist --store` bypass, but the new guard still uses the selected allowlist target as the command origin, so cross-channel store edits can bypass an origin channel with `configWrites=false`; the external PR also still needs real behavior proof before merge.

## What This Changes

The PR adds a `configWrites` denial check before store-targeted `/allowlist` pairing-store mutations and adds focused tests for blocked channel/account writes plus the allowed path.

## Best Possible Solution

Update the store-write guard so it authorizes the actual command origin against the selected store target, add regression coverage for cross-channel `--channel ... --store` edits, then provide redacted real behavior proof of the denied write and unchanged store.

## Reproduction Assessment

Yes. Source inspection on current main shows `/allowlist ... --store` skips the only `configWrites` guard and reaches the pairing-store mutation; the PR diff is also source-reproducible for the remaining cross-channel origin/target gap.

## Solution Assessment

No. Reusing the existing `configWrites` gate is the right direction, but this implementation should pass the real command origin separately from the selected store target before it is the best fix.
