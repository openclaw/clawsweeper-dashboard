## Summary

Keep open: the retry direction is useful, but this PR is not merge-ready because the legacy `launchctl load -w` fallback can report success in the wrong/root/headless domain without verifying the intended LaunchAgent actually loaded.

## What This Changes

This PR adds retry-after-bootout and legacy `launchctl load -w` fallback behavior to macOS LaunchAgent bootstrap, with unit tests for duplicate-registration and GUI-domain failures.

## Best Possible Solution

Split the safe duplicate-registration retry from headless/sudo support: land a verified retry for explicit stale registration, and keep missing-GUI/wrong-user paths on clear guidance unless maintainers design a Background or LaunchDaemon path.

## Reproduction Assessment

Yes for source-level behavior: current main performs one `bootstrap` and immediately throws for GUI-domain errors, while the PR tests simulate duplicate-registration and missing-GUI failures. No live macOS proof establishes that the fallback starts the intended user agent.

## Solution Assessment

No. The retry portion is plausible, but unconditional `launchctl load -w` is not the narrowest maintainable fix without root rejection, output or registration verification, and live macOS proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate legacy load before returning success:** `src/daemon/launchd.ts:229-231`
  - body: The fallback returns as soon as `launchctl load -w` exits 0, but the legacy `load` contract only reserves non-zero for improper usage. This can report a successful install while stdout/stderr contains the real load failure or the service never registered, so verify output or the final registration before returning.
  - confidence: 0.9
- **[P1] Reject root-scoped legacy loads:** `src/daemon/launchd.ts:229-231`
  - body: This branch also runs after sudo-triggered `gui/0` bootstrap failures. Legacy launchctl subcommands executed as root target the system domain, so `sudo openclaw gateway install` can now report success while the intended user's LaunchAgent remains uninstalled.
  - confidence: 0.86
- **[P2] Prove headless load starts the agent:** `src/daemon/launchd.ts:229-231`
  - body: For SSH/headless installs, `load -w` defaults per-user jobs to the Aqua session and the generated plist has no Background session policy. Returning success here can still leave the gateway waiting for a GUI login, contradicting the claimed headless install fix.
  - confidence: 0.8
