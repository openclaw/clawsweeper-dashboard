## Summary

Keep open: the duplicate-bootstrap retry is useful, but the PR should not merge as-is because the new legacy `launchctl load -w` fallback can report false success and can target the wrong launchd domain under sudo/root. Current main still preserves the safer GUI-session error path for unsupported macOS contexts.

## What This Changes

The PR updates macOS launchd gateway activation to retry explicit duplicate bootstrap errors, add a legacy `launchctl load -w` fallback for GUI-domain failures, and expand launchd tests for those paths.

## Best Possible Solution

Keep the duplicate-bootstrap retry, but replace the legacy `load -w` fallback with a supported and verified launchd activation path, or preserve the existing GUI-session guidance for SSH/sudo contexts until a deliberate Background/LaunchDaemon design is chosen.

## Reproduction Assessment

Yes. From source inspection and the launchctl man page, the PR can return success on `load` exit code 0 without checking output or registration, and sudo/root runs can target the system domain instead of the intended user LaunchAgent.

## Solution Assessment

No. Retrying explicit duplicate bootstrap errors is a reasonable direction, but an unqualified legacy `load -w` fallback is not the narrowest maintainable fix for SSH/headless/sudo failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate legacy load output before returning success:** `src/daemon/launchd.ts:230-231`
  - body: `launchctl load` is a legacy subcommand whose documented exit status is not a reliable load-success signal: it only returns non-zero for improper usage and otherwise can still print actionable failure output. Returning here on `code === 0` can make `openclaw gateway install` report success even when the agent was not registered, so the fallback needs to inspect output and/or verify the service before succeeding.
  - confidence: 0.87
- **[P1] Reject root-scoped load fallback for LaunchAgents:** `src/daemon/launchd.ts:229`
  - body: This fallback also runs when the GUI-domain failure came from sudo/root (`gui/0`), but legacy `launchctl load` executed as root targets the system domain. That can turn the existing wrong-user error into a successful install in the wrong launchd context, leaving the intended user's LaunchAgent missing and potentially running the gateway with the wrong privilege scope.
  - confidence: 0.84
- **[P2] Do not claim headless success with Aqua-only load:** `src/daemon/launchd.ts:229-231`
  - body: The fallback uses `launchctl load -w` without `-S Background`, while the generated plist has no Background session policy. The launchctl docs make Aqua the default and Aqua agents only load after GUI login, so this can still fail the SSH/headless use case while returning as if installation succeeded.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete launchd privilege/scope concern in the sudo/root fallback path.

Concerns:

- **[medium] Sudo fallback can target the system launchd domain:** `src/daemon/launchd.ts:229`
  - body: Because legacy `launchctl load` targets the system domain when run as root, the new fallback can convert a wrong-user `gui/0` failure into a success path outside the intended per-user LaunchAgent domain. That is a privilege/scope regression for a service install command.
