## Summary

Keep open. Current main does not implement this PR's retry or legacy load fallback, but the PR should not merge as-is because the fallback can report success without proving the LaunchAgent registered or started in the intended per-user domain, especially under sudo/root and headless macOS sessions.

## What This Changes

This PR changes macOS LaunchAgent bootstrap to retry after already-bootstrapped errors, fall back to legacy `launchctl load -w` on GUI-domain failures, and adds launchd unit tests for those paths.

## Best Possible Solution

Keep the duplicate-registration retry direction only if it is tied to a verified duplicate-load signal, and split or redesign the legacy fallback so it rejects root/wrong-user contexts, validates load output, verifies final registration and runtime state, defines Aqua versus Background session behavior, and has live macOS proof before shipping.

## Reproduction Assessment

Yes for source-level behavior: current main has only one bootstrap attempt, and the PR tests simulate duplicate-registration plus missing-GUI-domain failures. The blocking fallback risks are reproducible by inspection against the launchctl legacy contract and should be confirmed with live macOS checks before merge.

## Solution Assessment

No. The retry portion is plausible, but the unconditional legacy `load -w` fallback is not the narrowest maintainable fix without root guards, output parsing, post-load verification, and a documented session model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate legacy load output before returning success:** `src/daemon/launchd.ts:230-231`
  - body: The fallback returns as soon as `launchctl load -w` exits 0, but legacy `load` can return zero while reporting real plist or registration failures in stdout/stderr. Parse the output or verify the service is registered before reporting install success.
  - confidence: 0.9
- **[P1] Reject root legacy load fallback:** `src/daemon/launchd.ts:229-231`
  - body: This fallback also runs after sudo-triggered `gui/0` bootstrap failures. Legacy launchctl targets the system domain when run as root, so this can report success while leaving the intended per-user LaunchAgent uninstalled.
  - confidence: 0.86
- **[P2] Prove the fallback starts a headless-capable agent:** `src/daemon/launchd.ts:229-231`
  - body: The PR uses `launchctl load -w` without `-S Background`, while launchctl defaults per-user legacy loads to Aqua sessions. In SSH/headless contexts this can accept the plist but still defer the agent until GUI login, contradicting the claimed headless success path.
  - confidence: 0.8
