## Summary

Keep this PR open, but it should not merge as-is: the duplicate-bootstrap retry may be useful, while the legacy `launchctl load -w` fallback changes supported macOS LaunchAgent behavior and still has blocking correctness, proof, and root-scope concerns.

## What This Changes

The PR changes macOS launchd gateway installation to retry duplicate bootstrap failures and fall back to legacy `launchctl load -w` when `gui/<uid>` bootstrap is unavailable.

## Best Possible Solution

Keep any narrow duplicate-bootstrap retry only with real macOS proof, and remove or redesign the legacy load fallback around the documented logged-in-session requirement or an explicit LaunchDaemon model.

## Reproduction Assessment

Yes for source-level PR review: PR head can return success from `launchctl load -w` solely on exit code 0 without proving the intended LaunchAgent is registered. I did not establish a live macOS launchctl repro.

## Solution Assessment

No. The retry direction may be useful, but an unqualified legacy `load -w` fallback is not the narrowest maintainable solution for SSH, headless, or sudo installs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate legacy load before returning success:** `src/daemon/launchd.ts:230-231`
  - body: This branch returns as soon as `launchctl load -w` exits 0, without checking output or verifying that the intended LaunchAgent is registered. Legacy `load`/`unload` can return zero outside improper-usage failures, so this can make install report success while the service is still missing. ([man.freebsd.org](https://man.freebsd.org/cgi/man.cgi?apropos=0&manpath=macOS+26.3&query=launchctl&sektion=1&utm_source=openai))
  - confidence: 0.87
- **[P1] Avoid root-scoped load fallback for LaunchAgents:** `src/daemon/launchd.ts:229`
  - body: The fallback is reachable for sudo/wrong-user runs, but legacy `launchctl load` run as root targets the system domain rather than the target desktop user's `gui/<uid>` LaunchAgent domain. Returning success there can leave the intended user service uninstalled. ([man.freebsd.org](https://man.freebsd.org/cgi/man.cgi?apropos=0&manpath=macOS+26.3&query=launchctl&sektion=1&utm_source=openai))
  - confidence: 0.83
- **[P2] Do not claim headless success with Aqua-only load:** `src/daemon/launchd.ts:229-231`
  - body: The fallback runs `launchctl load -w` without selecting a Background session while OpenClaw writes an Interactive per-user LaunchAgent and documents macOS install as requiring a logged-in user session. In SSH/headless contexts this can report success for a job that still depends on GUI-login behavior.
  - confidence: 0.78
