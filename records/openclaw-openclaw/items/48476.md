## Summary

Keep open: the retry idea is still relevant, but this PR is not merge-ready because the legacy `launchctl load -w` fallback can report success in the wrong launchd domain or before the agent is actually registered/running.

## What This Changes

This PR changes macOS LaunchAgent bootstrap to retry explicit already-bootstrapped failures, fall back to `launchctl load -w` for GUI-domain failures, and adds launchd unit coverage for those paths.

## Best Possible Solution

Split the duplicate-registration retry from headless/sudo support: ship a bounded retry with regression coverage, and keep headless or wrong-user installs on explicit guidance unless maintainers design and validate a Background or LaunchDaemon path.

## Reproduction Assessment

Yes for source-level behavior: current main has a single bootstrap attempt, and the PR tests model duplicate-registration and missing-GUI-domain failures. I did not establish live macOS proof for the legacy fallback semantics.

## Solution Assessment

No. The retry-only portion is plausible, but the unconditional legacy `load -w` fallback is not the narrowest maintainable fix without root rejection, output/registration verification, explicit session semantics, and live macOS validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate legacy load before returning success:** `src/daemon/launchd.ts:230-231`
  - body: This returns success solely from `load.code === 0`, but legacy `launchctl load` can return zero while still reporting real load failures in stdout/stderr. Please parse the output or verify the service is registered and running before reporting install success.
  - confidence: 0.9
- **[P1] Reject root-scoped legacy load fallback:** `src/daemon/launchd.ts:229-231`
  - body: This fallback also runs after sudo-triggered `gui/0` failures. Legacy `launchctl load` targets the system domain when run as root, so `sudo openclaw gateway install` can report success while leaving the intended user's LaunchAgent uninstalled.
  - confidence: 0.86
- **[P2] Prove headless fallback starts the agent:** `src/daemon/launchd.ts:229-231`
  - body: For SSH/headless installs, `load -w` defaults per-user jobs to the Aqua session and the generated plist has no Background session policy. An accepted load can still leave the gateway waiting for GUI login, contradicting the claimed successful headless install.
  - confidence: 0.8
