## Summary

Keep open: the duplicate-bootstrap retry is plausible, but the legacy `launchctl load -w` fallback introduces blocking correctness and launchd scope risks, and the external PR still lacks real macOS behavior proof.

## What This Changes

The PR adds duplicate-bootstrap retry handling, a legacy `launchctl load -w` fallback for GUI-domain failures, and launchd unit coverage.

## Best Possible Solution

Keep a narrow, live-verified duplicate-bootstrap retry if maintainers want it, but remove or replace the unqualified legacy fallback until OpenClaw has a supported Background-session or LaunchDaemon model for headless macOS installs.

## Reproduction Assessment

Yes for source-level review: PR head can return success from `launchctl load -w` on exit code 0 without output or registration validation, and the launchctl contract makes that unsafe. I did not run a live macOS launchctl repro.

## Solution Assessment

No. The duplicate-bootstrap retry may be a narrow bug-fix direction, but an unqualified legacy `load -w` fallback is not the narrowest maintainable solution for SSH, headless, or sudo installs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate legacy load output before returning success:** `src/daemon/launchd.ts:230-231`
  - body: `launchctl load` is a legacy subcommand whose exit status is not a reliable load-success signal; this branch returns on `code === 0` before inspecting stderr/stdout or verifying registration, so install can report success even when the agent was not actually loaded. ([keith.github.io](https://keith.github.io/xcode-man-pages/launchctl.1.html))
  - confidence: 0.87
- **[P1] Reject root-scoped load fallback for LaunchAgents:** `src/daemon/launchd.ts:229`
  - body: This fallback can be reached by root-allowed execution, but legacy `launchctl load` run as root targets the system domain. Returning success there can put a user LaunchAgent install in the wrong launchd scope instead of failing with the existing wrong-user guidance. ([keith.github.io](https://keith.github.io/xcode-man-pages/launchctl.1.html))
  - confidence: 0.82
- **[P2] Do not claim headless success with Aqua-only load:** `src/daemon/launchd.ts:229-231`
  - body: The fallback uses `launchctl load -w` without selecting a Background session, while the generated plist has no Background session limit. That can still leave the job tied to GUI-login behavior while the install path reports success for the headless case it is meant to unblock. ([keith.github.io](https://keith.github.io/xcode-man-pages/launchctl.1.html))
  - confidence: 0.78
