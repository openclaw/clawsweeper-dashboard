## Summary

Keep open for maintainer handling: the PR targets a real launchd install failure, but the legacy `launchctl load -w` fallback introduces correctness and privilege-scope regressions, while the safer retry/system-service directions are already split across active related work.

## What This Changes

The branch changes macOS gateway LaunchAgent bootstrap to retry already-bootstrapped failures, fall back to `launchctl load -w` for GUI-domain errors, and adds launchd unit coverage for those paths.

## Best Possible Solution

Keep a narrow, verified retry for transient duplicate/EIO bootstrap failures, and route headless or boot-without-login macOS support through an explicit LaunchDaemon design with matching install, status, lifecycle, docs, and live proof.

## Reproduction Assessment

Yes. Source inspection of PR head plus the launchctl contract gives a high-confidence path: GUI-domain bootstrap failure enters `load -w`, and code 0 returns success without checking output, registration, or domain scope.

## Solution Assessment

No. Retrying duplicate bootstrap failures is a reasonable direction, but an unqualified legacy `load -w` fallback is not the narrowest maintainable fix for SSH/headless/sudo behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate legacy load output before returning success:** `src/daemon/launchd.ts:230-231`
  - body: `launchctl load` only uses non-zero status for improper usage, so this early return can report a successful install even when `load` printed an actionable failure and did not register the agent. Inspect output and/or verify the service before returning success.
  - confidence: 0.87
- **[P1] Reject root-scoped load fallback for LaunchAgents:** `src/daemon/launchd.ts:229`
  - body: This fallback also runs after sudo/root GUI-domain failures, but legacy `launchctl load` executed as root targets the system domain. That can turn a wrong-user failure into a success path outside the intended per-user LaunchAgent scope.
  - confidence: 0.84
- **[P2] Do not claim headless success with an Aqua-only load:** `src/daemon/launchd.ts:229-231`
  - body: The fallback calls `launchctl load -w` without `-S Background`, while OpenClaw's generated plist has no Background session policy. Default Aqua agents only load after GUI login, so this can still miss the SSH/headless goal while returning success.
  - confidence: 0.78
