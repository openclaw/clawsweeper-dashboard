## Summary

Keep open. Current main still does not implement this PR's install-time retry/fallback behavior, but the PR should not merge as written because the legacy `launchctl load -w` fallback can report success without proving registration, can run in the wrong root/system domain under sudo, and has not shown that the generated LaunchAgent is actually headless-capable.

## What This Changes

This PR modifies macOS launchd LaunchAgent bootstrap to retry after explicit already-loaded errors, fall back to legacy `launchctl load -w` when GUI-domain bootstrap fails, and adds launchd unit tests for those paths.

## Best Possible Solution

Split the useful duplicate-registration retry from the headless/sudo fallback. A maintainable fix would add a bounded retry only for a verified duplicate-registration signal, keep wrong-user/headless installs on clear GUI-session guidance, and pursue any true headless macOS support as an explicit LaunchDaemon or Background-session design with live validation.

## Reproduction Assessment

Yes for source-level behavior: current main has a single install bootstrap attempt, and the PR tests simulate duplicate-registration and missing-GUI-domain failures. The blocking fallback risks are reproducible by inspection against OpenClaw's uid/domain code and the launchctl legacy contract, but they still need live macOS confirmation before shipping.

## Solution Assessment

No. The retry portion is plausible, but the unconditional legacy load fallback is not the narrowest maintainable solution without root rejection, output parsing or post-load verification, explicit session semantics, and macOS live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Verify the legacy load actually registers the agent:** `src/daemon/launchd.ts:230-231`
  - body: The fallback returns as soon as `launchctl load -w` exits 0, but legacy `load` can report real plist or registration failures in stdout/stderr while still returning zero. Parse the output or verify the service is registered before reporting install success.
  - confidence: 0.9
- **[P1] Block root-scoped legacy loads:** `src/daemon/launchd.ts:229-231`
  - body: This fallback also runs after sudo-triggered `gui/0` bootstrap failures. Legacy `launchctl load` targets the system domain when run as root, so this can report success while leaving the intended per-user LaunchAgent uninstalled.
  - confidence: 0.86
- **[P2] Prove headless loads start this LaunchAgent:** `src/daemon/launchd.ts:229-231`
  - body: The fallback uses `launchctl load -w` without `-S Background`, and the generated plist has no Background session policy. In SSH/headless contexts it can accept the plist while the agent still waits for an Aqua GUI session, contradicting the claimed install success.
  - confidence: 0.8
