## Summary

Keep open: current main still lacks the intended launchd reinstall recovery, but this PR is not merge-ready because the legacy `launchctl load -w` fallback can report false success and changes sudo/headless service-domain semantics.

## What This Changes

This PR modifies macOS LaunchAgent bootstrap to retry explicit already-loaded failures, fall back to legacy `launchctl load -w` for missing GUI domains, and add launchd unit tests for those paths.

## Best Possible Solution

Split the duplicate-registration retry from headless/sudo support: ship a verified, bounded retry for explicit stale registration, and keep wrong-user/headless installs on clear guidance unless maintainers design and validate explicit Background or LaunchDaemon support.

## Reproduction Assessment

Yes for source-level behavior: current main has a single bootstrap attempt and the PR tests simulate duplicate-registration and missing-GUI-domain failures. The fallback risks are also clear by source inspection and launchctl's documented legacy behavior, but they still need live macOS validation before shipping.

## Solution Assessment

No. The retry portion is plausible, but the unconditional `load -w` fallback is not the narrowest maintainable fix without root rejection, output or registration verification, explicit session semantics, and live macOS proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Verify legacy load before reporting success:** `src/daemon/launchd.ts:230-231`
  - body: The fallback returns as soon as `launchctl load -w` exits 0, but launchctl's legacy `load`/`unload` contract says non-zero is reserved for improper usage and otherwise zero is returned. Parse stdout/stderr or verify the service is registered and running before reporting install success.
  - confidence: 0.9
- **[P1] Block root-scoped legacy loads:** `src/daemon/launchd.ts:229-231`
  - body: This fallback also runs after sudo-triggered `gui/0` failures. Legacy launchctl subcommands target the system domain when executed as root, so this can report success while leaving the intended per-user LaunchAgent uninstalled.
  - confidence: 0.86
- **[P2] Prove headless fallback starts the agent:** `src/daemon/launchd.ts:229-231`
  - body: For SSH/headless installs, `load -w` defaults to the Aqua session and the generated plist has no Background session policy. An accepted load can still leave the gateway waiting for GUI login, contradicting the claimed successful headless install.
  - confidence: 0.8
