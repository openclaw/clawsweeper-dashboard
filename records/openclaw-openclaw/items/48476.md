## Summary

Keep this PR open, but it should not merge as-is: the duplicate-bootstrap retry direction is plausible, while the unqualified legacy `launchctl load -w` fallback still creates false-success and wrong-domain paths for SSH/headless/sudo installs and needs real macOS proof plus maintainer policy review.

## What This Changes

The PR changes macOS launchd gateway installation to retry duplicate bootstrap failures and fall back to legacy `launchctl load -w` when the GUI domain is unavailable.

## Best Possible Solution

Keep any narrow duplicate-bootstrap retry only with live macOS proof, and remove or redesign the legacy load fallback around a supported logged-in-session or LaunchDaemon model.

## Reproduction Assessment

Yes for source-level PR review: PR head can return success from `launchctl load -w` solely on exit code 0 without proving the intended LaunchAgent is registered. I did not establish a live macOS launchctl repro.

## Solution Assessment

No. The retry direction may be useful, but an unqualified legacy `load -w` fallback is not the narrowest maintainable solution for SSH, headless, or sudo installs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate legacy load before returning success:** `src/daemon/launchd.ts:230-231`
  - body: This branch returns as soon as `launchctl load -w` exits 0, without checking output or verifying that the intended LaunchAgent is registered. That can make `openclaw gateway install` report success while the service is still missing.
  - confidence: 0.87
- **[P1] Avoid root-scoped load fallback for LaunchAgents:** `src/daemon/launchd.ts:229`
  - body: The fallback is reachable for the sudo/wrong-user case, but a bare legacy `launchctl load` run in that context does not prove registration in the target desktop user's `gui/<uid>` domain. Returning success there can leave the intended user service uninstalled.
  - confidence: 0.82
- **[P2] Do not claim headless success with Aqua-only load:** `src/daemon/launchd.ts:229-231`
  - body: The fallback runs `launchctl load -w` without selecting a Background session, while OpenClaw writes an Interactive per-user LaunchAgent. In SSH/headless contexts this can report success for a job that still depends on GUI-login behavior.
  - confidence: 0.78
