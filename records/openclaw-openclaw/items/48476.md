## Summary

Keep open for maintainer handling: current main does not contain this PR's retry/load fallback, but the proposed legacy `launchctl load -w` fallback remains unsafe without real macOS proof and launchd policy review.

## What This Changes

The PR changes macOS launchd bootstrap to retry already-registered LaunchAgents and fall back to `launchctl load -w` for GUI-domain failures, with unit coverage.

## Best Possible Solution

Keep any narrow duplicate-bootstrap retry only if it is live-verified, and remove or redesign the legacy load fallback around a supported Background-session or LaunchDaemon model.

## Reproduction Assessment

Yes for source-level review findings: PR head can return success from `launchctl load -w` on exit code 0 without checking output or registration. I did not establish a live macOS launchctl repro.

## Solution Assessment

No. The retry direction may be useful, but an unqualified legacy load fallback is not the narrowest maintainable solution for SSH, headless, or sudo installs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate legacy load before returning success:** `src/daemon/launchd.ts:235-238`
  - body: `launchctl load` can return zero even when the job was not actually loaded, but this branch returns immediately on `load.code === 0` without checking output or verifying registration. That can make `openclaw gateway install` report success while the LaunchAgent is still absent.
  - confidence: 0.87
- **[P1] Avoid root-scoped load fallback for LaunchAgents:** `src/daemon/launchd.ts:235`
  - body: The fallback is reachable for the sudo/wrong-user case this PR claims to handle, but legacy `launchctl load` run as root targets the root/system context. Returning success there can leave the intended desktop user's LaunchAgent unregistered.
  - confidence: 0.82
- **[P2] Do not claim headless success with an Aqua-only load:** `src/daemon/launchd.ts:235-238`
  - body: The fallback runs `launchctl load -w` without selecting a Background session, while the generated plist is an Interactive LaunchAgent. A headless install can therefore be reported as successful even though the job remains tied to GUI-login behavior.
  - confidence: 0.78
