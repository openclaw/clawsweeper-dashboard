## Summary

Keep open: current main has not implemented this PR, and the PR is not merge-ready because the new legacy `launchctl load -w` fallback can report false install success and can change sudo/headless installs into the wrong launchd scope.

## What This Changes

The PR changes macOS launchd bootstrap handling to retry explicit duplicate-registration failures and fall back to legacy `launchctl load -w` for unavailable GUI domains, with launchd unit coverage.

## Best Possible Solution

Keep the useful duplicate-registration retry only when the service is verified as already registered, but do not accept a bare legacy `load -w` fallback until it validates registration and has an explicit sudo/headless product decision.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main throws from the GUI-domain bootstrap path, and the PR diff shows the new fallback accepting `load.code === 0`. I did not run live macOS launchctl in this read-only review.

## Solution Assessment

No. The retry direction is useful, but the current fallback is not the narrowest maintainable fix because it relies on legacy `load` exit status and changes sudo/headless behavior outside the existing LaunchAgent contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate legacy load before returning success:** `src/daemon/launchd.ts:234-235`
  - body: The fallback returns as soon as `launchctl load -w` exits 0. `launchctl(1)` documents that legacy `load`/`unload` generally return zero except improper usage, so plist, permission, or session failures can still be printed while this branch reports install success without verifying the agent registered.
  - confidence: 0.9
- **[P1] Avoid root-scoped load fallback for sudo installs:** `src/daemon/launchd.ts:233`
  - body: GUI-domain failures include the `gui/0` path hit under sudo, but bare legacy `launchctl load` targets the system domain when executed as root. Since install callers pass `process.env` and plist paths come from `HOME`, this can turn the existing wrong-user error into a successful-looking root/system registration instead of the intended user's LaunchAgent.
  - confidence: 0.86
- **[P2] Do not claim headless success with Aqua-only agents:** `src/daemon/launchd.ts:233`
  - body: The fallback uses `load -w` without `-S Background`, while the generated plist has no `LimitLoadToSessionType=Background`. Legacy per-user loading defaults to Aqua, so SSH/headless installs can return success while the gateway still waits for a GUI login.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff broadens a launchd code-execution path by allowing sudo-triggered GUI-domain failures to fall back to root/system-scoped legacy loading.

Concerns:

- **[medium] Sudo fallback may register the gateway in root/system scope:** `src/daemon/launchd.ts:233`
  - body: When the command is run under sudo, `resolveGuiDomain()` targets `gui/0`; the PR then falls back to bare `launchctl load -w`, whose legacy contract targets the system domain as root. That can run or register the gateway outside the intended user's LaunchAgent context.
  - confidence: 0.84
