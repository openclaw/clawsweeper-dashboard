## Summary

Keep open. The duplicate-bootstrap retry direction is useful, but the legacy `launchctl load -w` fallback still adds unverified launchd behavior with a privilege/scope concern, and the external PR has no real macOS behavior proof.

## What This Changes

The PR changes macOS LaunchAgent activation to retry explicit duplicate bootstrap failures, fall back to legacy `launchctl load -w` for GUI-domain failures, and adds launchd unit tests.

## Best Possible Solution

Keep a narrow duplicate-bootstrap retry if verified, but replace or remove the legacy fallback until maintainers choose a supported Background/LaunchDaemon/session model; otherwise preserve the current GUI-session guidance for SSH/sudo contexts.

## Reproduction Assessment

Yes for source-level review: PR head can return success from `launchctl load -w` on exit code 0 without output/registration validation, and dependency docs make that unsafe. I did not run a live macOS launchctl repro.

## Solution Assessment

No. The retry for explicit duplicate bootstrap errors is a reasonable bug-fix direction, but an unqualified legacy `load -w` fallback is not the narrowest maintainable solution for SSH/headless/sudo installs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate legacy load output before returning success:** `src/daemon/launchd.ts:230-231`
  - body: `launchctl load` is a legacy subcommand whose exit status is not a reliable load-success signal; this branch returns on `code === 0` before inspecting stderr/stdout or verifying registration, so install can report success even when the agent was not actually loaded.
  - confidence: 0.87
- **[P1] Reject root-scoped load fallback for LaunchAgents:** `src/daemon/launchd.ts:229`
  - body: This fallback can be reached by root-allowed execution, but legacy `launchctl load` run as root targets the system domain. Returning success there can put a user LaunchAgent install in the wrong launchd scope instead of failing with the existing wrong-user guidance.
  - confidence: 0.82
- **[P2] Do not claim headless success with Aqua-only load:** `src/daemon/launchd.ts:229-231`
  - body: The fallback uses `launchctl load -w` without selecting a Background session, while the generated plist has no `LimitLoadToSessionType=Background`. That can still leave the job tied to Aqua/GUI login behavior while the install path reports success for the headless case it is meant to unblock.
  - confidence: 0.78
