## Summary

Keep open: current main has not adopted the legacy `load -w` fallback, and the PR still has blocking correctness/security concerns plus no real macOS after-fix proof.

## What This Changes

The PR changes macOS launchd bootstrap to retry duplicate registrations and fall back to legacy `launchctl load -w` when the GUI launchd domain is unavailable.

## Best Possible Solution

Keep any narrow duplicate-bootstrap retry only with live macOS proof, and remove or redesign the legacy load fallback around a supported logged-in-session or LaunchDaemon model.

## Reproduction Assessment

Yes for source-level PR review: PR head can return success from `launchctl load -w` solely on exit code 0 without verifying output or registration. I did not establish a live macOS launchctl repro.

## Solution Assessment

No. The retry direction may be useful, but an unqualified legacy `load -w` fallback is not the narrowest maintainable solution for SSH, headless, or sudo installs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate legacy load before returning success:** `src/daemon/launchd.ts:230-231`
  - body: This branch returns as soon as `launchctl load -w` exits 0, but legacy load can still print actionable diagnostics without proving the job was registered. Verify output or registration before reporting install success.
  - confidence: 0.87
- **[P1] Avoid root-scoped load fallback for LaunchAgents:** `src/daemon/launchd.ts:229`
  - body: The fallback is reachable for the sudo/wrong-user case, but legacy `launchctl load` run as root targets the root/system context. Returning success there can leave the intended desktop user's LaunchAgent unregistered.
  - confidence: 0.82
- **[P2] Do not claim headless success with Aqua-only load:** `src/daemon/launchd.ts:229-231`
  - body: The fallback runs `launchctl load -w` without selecting a Background session, while the managed service is a per-user LaunchAgent. A headless install can be reported successful even though the job remains tied to GUI-login behavior.
  - confidence: 0.78
