## Label Justifications

- `P2`: The reported hang can block one agent session and queued messages, but current release/source evidence shows it is already fixed.
- `impact:message-loss`: The issue reports queued inbound messages going unprocessed and the final reply never being delivered while the embedded run holds the session lock.
- `impact:crash-loop`: The issue is about a long-running stuck/hung agent run and watchdog recovery, which is an availability failure even though the process does not crash.

## Summary

Current main and the latest release already bound embedded model generation calls through the shared stream idle watchdog and provider request timeout path, so the 2026.6.1 report is fixed by upgrading to v2026.6.9.

## What This Changes

The issue requests a bounded, configurable timeout for embedded agent model generation calls so hung provider calls fail before the coarse stuck-session watchdog.

## Best Possible Solution

Keep the shipped v2026.6.9 shared stream/provider timeout behavior and ask affected 2026.6.1 users to upgrade rather than adding another embedded-run timeout knob.

## Reproduction Assessment

Yes for the original source-level class: the issue includes concrete stalled-session logs from 2026.6.1, and current source/tests show the fix path that bounds silent model streams and hung stream setup. I did not run a live Google/Gemini hang replay in this read-only review.

## Solution Assessment

Yes. The best shipped solution is the shared embedded stream/provider timeout boundary using `models.providers.<id>.timeoutSeconds` plus the default cloud idle watchdog, not a new one-off embedded-run setting.
