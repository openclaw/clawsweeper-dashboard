## Summary

Keep open: current main still has the Telegram probe fallback gap, and this PR is a narrow useful fix, but it still needs small pre-merge cleanup before landing.

## What This Changes

The PR changes Telegram health probes to cache a full TelegramTransport, force its IPv4 fallback after getMe fetch errors, and adds regression coverage for timeout-then-fallback success.

## Best Possible Solution

Land the Telegram probe fallback fix after adding the Unreleased/Fixes changelog entry, closing evicted cached probe transports, and getting the PR checks green.

## Reproduction Assessment

Yes, source-reproducible: current main’s periodic Telegram probe uses fetchWithTimeout through a fetch-only transport wrapper and does not force fallback after a timeout. The linked report supplies concrete 2026.5.2 logs from an IPv4-working, IPv6-broken Debian host showing recurring getMe timeouts.

## Solution Assessment

Yes, with minor cleanup: reusing the full TelegramTransport and forcing fallback after getMe fetch failures is the narrow maintainable runtime fix and matches the existing bot-client timeout recovery path. The PR should also carry the required changelog entry and close cached transports when they are evicted.
