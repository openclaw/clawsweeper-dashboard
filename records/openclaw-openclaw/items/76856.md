## Summary

Keep open: current main still has the Telegram probe path using only a fetch closure, and this PR is a plausible narrow fix, but the PR body/comments only provide mock/unit-test proof while Telegram maintainer guidance requires real Telegram proof for transport changes.

## What This Changes

The PR changes Telegram getMe probing to cache the full Telegram transport, call `forceFallback()` after probe fetch errors, add a regression test, and add a changelog entry.

## Best Possible Solution

Land a narrow transport-aware probe fix after the contributor adds redacted live Telegram/network proof showing the repeated getMe timeouts stop on an affected host.

## Reproduction Assessment

Yes from source inspection: current main can be exercised by making the first getMe `fetchWithTimeout` abort with a `TimeoutError`, which the probe retry loop does not promote to the transport fallback. I did not run a live IPv4-only Telegram host reproduction in this read-only review.

## Solution Assessment

Yes for the code direction: keeping the full `TelegramTransport` and forcing fallback before retry mirrors the existing timed-out control-call pattern in the Telegram bot client. The merge path still needs real Telegram proof before this should land.
