## Summary

Keep open. Current main still keeps Active Memory in the foreground prompt-build path, local memory search still runs the local embedding and hybrid rerank pipeline in-process, and the docs describe the feature as a blocking memory sub-agent; the related #65309 tracker covers broad latency, but this report adds distinct local-backend event-loop evidence.

## What This Changes

Issue reports long `system-prompt` prep-stage rebuilds and idle event-loop saturation when Active Memory/context injection searches a large local memory database.

## Best Possible Solution

Define a focused perf plan for Active Memory plus memory-core local search: measure the trace split, move or bound CPU-heavy local embedding/rerank work off the main event loop, and add cache/readiness behavior without changing plugin boundaries.

## Reproduction Assessment

Yes for source-level and trace-based reproduction: an eligible Active Memory direct-chat turn awaits `before_prompt_build`, which can run an embedded recall agent that invokes local memory search over the reported large DB. I did not live-run the exact 16K-chunk Linux setup in this read-only review.

## Solution Assessment

No current implementation is the complete fix. Existing hard deadlines and cache TTLs bound some failures, but the maintainable solution needs an explicit non-blocking/off-main or stricter fail-open design for local memory recall.
