## Summary

Keep open. Current main still keeps Active Memory recall on the foreground prompt-build path, and the local memory backend still performs query embedding plus hybrid rerank in-process; the existing cache and circuit breaker are partial mitigations, not a fix for the reported event-loop saturation.

## What This Changes

Investigate and mitigate system-prompt stalls from Active Memory/local memory search over large memory databases.

## Best Possible Solution

Define a measured Active Memory/local-memory perf plan that offloads or bounds local embedding and rerank work, caches stable recall or prompt results against memory revisions, and fails open before interactive turns exceed their latency budget.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an eligible Active Memory chat turn with local `memorySearch.provider = "local"` and a large indexed DB; I did not live-run the reporter's exact 16K-chunk Linux setup in this read-only review.

## Solution Assessment

No current implementation fully solves it. Worker/off-main execution, revision-aware caching, and cooperative yielding are plausible, but the maintainable fix should be chosen from profiling data and preserve plugin/runtime boundaries.
