## What This Changes

The PR adds Docker setup to Crabbox hydration, adds a ClawHub artifact resolver fixture response, and updates kitchen-sink E2E expected diagnostics.

## Best Possible Solution

Keep the PR open, tighten Docker provisioning to a pinned or image-baked Crabbox path with restricted socket access, then land the remaining diagnostic and E2E coverage updates after the external metadata dependency is resolved.

Yes. The source-level path is clear: the kitchen-sink runner invokes Docker, the current Crabbox hydrate workflow does not install or start Docker, and current registry diagnostics include messages missing from the E2E assertion allowlist.

## Solution Assessment

No. The fixture and diagnostic updates are aligned with current code, but the Docker provisioning approach is not the safest maintainable fix because it adds unpinned root code execution and broad socket permissions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid executing Docker's mutable installer in the workflow:** `.github/workflows/crabbox-hydrate.yml:71`
  - body: The hydrate job would run `curl https://get.docker.com | sudo sh` before the same job hydrates provider-secret environment. That gives a mutable third-party install script root execution on a self-hosted runner; use a pre-baked Crabbox image or a pinned package/repository installation path instead.
  - confidence: 0.84
- **[P1] Avoid making the Docker socket world-writable:** `.github/workflows/crabbox-hydrate.yml:82`
  - body: Changing `/var/run/docker.sock` to mode `666` gives every local process root-equivalent Docker control for the lifetime of the hydrated box. Keep access scoped to the runner user/group or use a narrower wrapper so later commands do not inherit a host-control socket opened to all users.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive Docker provisioning in a self-hosted workflow.

Concerns:

  - body: The PR patch pipes `https://get.docker.com` directly into `sudo sh`, allowing a mutable remote script to control the runner before the hydrate job handles provider-secret environment.
