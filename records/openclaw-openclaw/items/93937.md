## What This Changes

The PR adds `total` token support to `formatResponseUsageLine` and tests total-only, empty, and undefined usage formatting.

## Best Possible Solution

Keep the formatter as the shared fallback, render aggregate-only totals without cost unless split buckets are present, add focused formatter coverage, and review alongside the followup-runner fix for the Telegram delivery path.

## Reproduction Assessment

Yes at source level: current main accepts aggregate `usage.total` as nonzero usage, but `formatResponseUsageLine` returns `null` when input and output are absent. I did not run live Telegram in this read-only review.

## Solution Assessment

No. Adding total support in the formatter is the right boundary for the legacy fallback, but aggregate-only totals should not synthesize costs and this PR is not the whole Telegram delivery fix.
