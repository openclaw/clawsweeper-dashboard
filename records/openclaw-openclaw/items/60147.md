## Summary

Current main already resolves the security problem by removing the vendored A2UI lockfile surface and moving A2UI dependency ownership to the Canvas plugin plus the root workspace lockfile; the fix shipped in v2026.5.9-beta.1, and merging this branch now would target obsolete files.

## What This Changes

The branch bumps dependency versions and regenerates npm/pnpm lockfiles under `vendor/a2ui` to address Trivy-reported CVEs.

## Best Possible Solution

Keep A2UI dependency security updates in `extensions/canvas/package.json` and the root `pnpm-lock.yaml`; do not revive `vendor/a2ui`.

## Reproduction Assessment

Not applicable as a current-main bug reproduction. The old check was `trivy fs vendor/a2ui --scanners vuln`, but current main no longer tracks a `vendor/a2ui` tree to scan.

## Solution Assessment

Yes for current main, no for this branch. Removing the vendored dependency tree and making Canvas/A2UI plugin-owned is the more maintainable solution than refreshing deleted lockfiles.
