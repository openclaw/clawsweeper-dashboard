## Summary

Current main has removed the vendored `vendor/a2ui` tree and moved A2UI ownership into the Canvas plugin/root workspace dependency flow, so the security problem this PR targeted no longer exists on the maintained code path.

## What This Changes

The PR updates vendored A2UI package manifests and lockfiles under `vendor/a2ui` to pin dependency versions reported by Trivy as vulnerable.

## Best Possible Solution

Keep A2UI dependency ownership in the Canvas plugin/root workspace path; future A2UI vulnerability fixes should update that package and root lockfile rather than restoring `vendor/a2ui`.

## Reproduction Assessment

Not applicable as a current-main bug reproduction. The old reproduction was `trivy fs vendor/a2ui --scanners vuln`, but current main has no `vendor/a2ui` tree to scan.

## Solution Assessment

Yes. Removing the vendored dependency tree and using plugin/workspace-managed A2UI dependencies is the narrower maintainable solution than updating lockfiles in a deleted tree.
