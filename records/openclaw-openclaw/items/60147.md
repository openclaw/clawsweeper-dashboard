## Summary

Current main already resolves the security problem by removing the vendored `vendor/a2ui` dependency tree and moving A2UI ownership to the Canvas plugin plus the root workspace lockfile. This PR now targets deleted files and would reintroduce an obsolete vendored supply-chain surface rather than update the maintained dependency path.

## What This Changes

The PR updates vendored A2UI package manifests and lockfiles under `vendor/a2ui` to pin CVE-fixed dependency versions.

## Best Possible Solution

Keep A2UI dependency ownership in `extensions/canvas` and the root `pnpm-lock.yaml`; future A2UI CVE work should update that maintained plugin surface.

## Reproduction Assessment

Not applicable as a current-main bug reproduction. The old check was `trivy fs vendor/a2ui --scanners vuln`, but current main no longer tracks a `vendor/a2ui` tree to scan.

## Solution Assessment

Yes for current main, no for this branch. Removing the vendored dependency tree and making Canvas/A2UI plugin-owned is the more maintainable fix than refreshing deleted lockfiles.
