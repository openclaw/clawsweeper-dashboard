## Summary

Close as implemented: current main no longer maintains the vendored `vendor/a2ui` lockfiles this PR updates, and A2UI dependency ownership now lives in the Canvas plugin plus the root workspace lockfile; that ownership move shipped in `v2026.5.9-beta.1`.

## What This Changes

The PR updates dependency pins, overrides, and regenerated lockfiles under the old `vendor/a2ui` tree to clear Trivy-reported CVEs.

## Best Possible Solution

Keep A2UI dependency ownership in `extensions/canvas` and the root lockfile; future A2UI vulnerability fixes should update that maintained plugin surface instead of restoring `vendor/a2ui`.

## Reproduction Assessment

Not applicable as a current-main bug reproduction. The old reproduction was `trivy fs vendor/a2ui --scanners vuln`, but current main no longer has a `vendor/a2ui` tree to scan.

## Solution Assessment

Yes. Removing the vendored dependency tree and making Canvas/A2UI plugin-owned is more maintainable than refreshing lockfiles in a deleted vendored tree.
