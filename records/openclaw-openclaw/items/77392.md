## Summary

Keep open: the report has concrete current-release latency measurements, and current main still lacks Discord-specific per-stage timing to explain the unaccounted delay. The related #53678 is useful context for inbound receive timestamps, but it does not fully cover Discord debounce, queue wait, turn dispatch, first visible model output, and outbound send/edit attribution.

## What This Changes

Request to diagnose or instrument Discord reply latency so trivial replies can be attributed across receive, memory, dispatch, model, and send stages.

## Best Possible Solution

Add privacy-safe Discord lifecycle timing through the existing diagnostics/timeline surface to attribute receive, debounce, preflight, queue wait, turn dispatch, model response, and outbound delivery before changing latency policy.

No high-confidence live reproduction of the 10-11.5 second latency was established in this read-only review. Source inspection does reproduce the observability gap: current main cannot attribute the Discord reply lifecycle across the stages named in the report.

## Solution Assessment

Yes for a narrow diagnostics PR: instrumentation is the safest next step because the current evidence does not identify which stage owns the extra delay. Optimization should wait until those measurements exist.
