## Summary

Keep open: current main has generic diagnostics and some total/delivery timing, but the Discord path still cannot explain the reported receive-to-visible-reply gap stage by stage. The linked #53678 is relevant but narrower because it tracks gateway received timestamps, not Discord debounce, queue wait, model first-token, and outbound send/edit attribution.

## What This Changes

The issue asks for investigation and per-stage Discord lifecycle timing to explain roughly 10-11.5s visible latency for trivial replies.

## Best Possible Solution

Add opt-in, privacy-safe Discord lifecycle timing that correlates receive, debounce, preflight, queue wait, turn dispatch, model first-token, and outbound send/edit before changing latency policy.

## Reproduction Assessment

No high-confidence live reproduction of the 10-11.5s latency was established. Source inspection does show current main cannot attribute that latency across the requested Discord lifecycle stages.

## Solution Assessment

Yes for a diagnostics-first repair: instrumentation is the narrowest maintainable next step because the current evidence does not identify which stage owns the extra delay. Optimization or SLA changes should wait until those measurements exist.
