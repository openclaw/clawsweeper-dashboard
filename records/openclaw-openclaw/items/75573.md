## Summary

Keep open. The PR addresses the Feishu schema validation gap, but it does not yet make the setting work end to end: Feishu reply dispatch still forces block streaming off, the committed bundled channel config metadata remains stale, and the user-facing fix needs a changelog entry.

## What This Changes

The PR adds optional `blockStreaming` to the shared Feishu config schema and adds schema tests for top-level/account boolean acceptance plus non-boolean rejection.

## Best Possible Solution

Ship an end-to-end Feishu fix that validates `channels.feishu.blockStreaming` at top-level and account scope, refreshes generated channel metadata, and makes Feishu honor explicit `blockStreaming: true` without reintroducing dropped block chunks, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. Current `main` has a strict Feishu schema without `blockStreaming`, so the documented `channels.feishu.blockStreaming` path is rejected; the runtime path also shows the PR’s accepted key would still be disabled by Feishu reply options. No live Feishu credentials are needed to reproduce these source-level paths.

## Solution Assessment

No. Adding the schema key is necessary, but it is not the best complete fix because generated metadata and Feishu reply runtime remain unchanged; the safer path is schema plus metadata regeneration plus explicit runtime honoring and tests.
