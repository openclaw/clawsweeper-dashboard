## Summary

Keep open: current main still drops the user-facing `filename` send parameter before Feishu delivery, but this PR is not mergeable as-is because it bypasses guarded media reads, breaks remote non-image uploads, removes upload filename hardening, changes CODEOWNERS/release safety surfaces, and lacks after-fix real Feishu proof.

## What This Changes

The PR threads a filename override through outbound send payloads into Feishu media uploads while also adding Feishu local-file fast paths and unrelated CI, release, CODEOWNERS, docs, scripts, skills, lockfile, and OpenAI WebSocket changes.

## Best Possible Solution

Rework this into a focused Feishu/outbound patch that carries `filename` through the existing guarded send path, preserves upload sanitization and `mediaLocalRoots`, drops unrelated repo changes, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, from source inspection: current main accepts `filename` as a send parameter but `buildSendPayloadParts` omits it from the outbound payload and `executeSendAction` call, so Feishu `sendMediaFeishu` cannot receive the override. I did not run a credentialed live Feishu send.

## Solution Assessment

No. Filename threading is the right direction, but this branch should not bypass the guarded media loader, remove upload hardening, or include unrelated CODEOWNERS/release/script/docs/skills changes.
