## Summary

Keep open. The Feishu filename problem is still not fully solved on current main, but this PR should not merge as-is because it regresses Feishu media safeguards and includes unrelated security-sensitive workflow, ownership, local-doc, script, skill, and lockfile changes.

## What This Changes

This PR threads optional filename values through message-tool/outbound/Feishu media paths and adds Feishu local non-image upload coverage, while also changing CODEOWNERS, CI workflows/actions, local BuildDocument/BuildTools files, skills, OpenAI WebSocket headers, and uv.lock.

## Best Possible Solution

Land a narrow Feishu-focused patch that carries explicit filename values through the message-tool send params, reply payload normalization, outbound context, and Feishu adapter while preserving `loadWebMedia` local-root enforcement, remote URL hydration, upload filename sanitization, CODEOWNERS, workflows, lockfiles, and unrelated local helper files unchanged.

Yes. Static inspection of the PR diff gives a high-confidence reproduction path: send a remote non-image Feishu URL or a disallowed absolute local non-image path through the PR branch and it reaches `uploadFileFeishu` without the current guarded loader behavior.

## Solution Assessment

No. The filename plumbing direction is reasonable, but this branch is not the best way to solve it because it mixes the intended fix with unrelated workflow, ownership, docs, script, skill, and supply-chain changes and weakens existing Feishu media safeguards.
