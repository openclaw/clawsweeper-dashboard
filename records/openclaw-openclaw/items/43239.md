## Summary

Keep open for maintainer handling. The Feishu filename plumbing is still a real gap on current main, but this PR is not mergeable as-is because it mixes that useful change with unrelated workflow/docs/scripts/skills changes and introduces blocking functional and security regressions.

## What This Changes

The PR threads optional filename values through outbound/Feishu media paths and adds a direct local non-image Feishu upload path, while also changing CODEOWNERS, CI workflows/actions, local docs/scripts/skills, OpenAI WebSocket headers, and adding a uv lockfile.

## Best Possible Solution

Land or request a narrow Feishu-only patch that carries an explicit filename through message-tool send params, reply payload normalization, outbound send overrides, ChannelOutboundContext, and the Feishu action/outbound adapters, while preserving loadWebMedia local-root enforcement, upload filename sanitization, CODEOWNERS, workflows, local docs/scripts/skills, and lockfiles unchanged.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path for the PR regressions: send a remote non-image Feishu media URL or a disallowed absolute local non-image path through the PR branch; both paths bypass the current guarded loadWebMedia behavior.

## Solution Assessment

No. The filename plumbing direction is useful, but this branch is not the best solution because it couples the change to unrelated repo/security changes and weakens existing Feishu media safety checks.
