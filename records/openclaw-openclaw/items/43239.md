## Summary

The useful filename-preservation bug is not fully implemented on current main, so this PR should stay open, but the branch is not mergeable as-is. It mixes the Feishu filename plumbing with unrelated repository changes and introduces security-sensitive regressions that current main already avoids.

## What This Changes

The PR threads filename overrides through outbound message sending for Feishu attachments, adds Feishu local-file send handling, and also changes unrelated workflows, CODEOWNERS, local docs/scripts, skills, lockfile, and OpenAI websocket code.

## Best Possible Solution

Land a focused Feishu/outbound patch that only carries a sanitized filename override through the existing guarded media-loader path, with targeted regression tests and redacted live Feishu proof.

## Reproduction Assessment

Yes by source inspection: the message tool exposes `filename`, but current main does not carry it into `ReplyPayload`, `MessageSendParams`, or the Feishu send call. I did not run a live Feishu send.

## Solution Assessment

No. Filename threading is the right direction, but this branch is not the best solution because it bypasses existing media-root/download guards, weakens upload filename hardening, and bundles unrelated workflow, docs, skill, lockfile, and local helper-script changes.
