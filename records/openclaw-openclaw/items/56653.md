## Summary

Keep open: current main and the latest release have Slack reaction handlers and mocked routing coverage, but the report describes zero live Socket Mode reaction envelopes on a version that already had those handlers, so there is not enough evidence to close as implemented.

## What This Changes

The issue reports that Slack reaction_added and reaction_removed Socket Mode events never reach gateway logs or agent sessions in multi-account setups despite configured event subscriptions, scopes, and reaction notification settings.

## Best Possible Solution

Keep this as the reaction-specific Slack Socket Mode report, capture raw current-main multi-account reaction envelopes and gateway trace logs, then patch only the proven Slack plugin or Socket Mode boundary without adding new config surface.

## Reproduction Assessment

No: the issue has concrete live steps and corroboration, but I did not run a real multi-account Slack Socket Mode workspace against current main. Source inspection proves the intended handler and plausible drop points, not a confirmed current-main failure.

## Solution Assessment

Unclear: the best fix depends on whether Slack/Bolt never invokes the listener, OpenClaw drops the envelope by account/team filtering, or later authorization suppresses it. The narrow maintainable path is to capture raw envelopes first, then patch the proven Slack plugin boundary.
