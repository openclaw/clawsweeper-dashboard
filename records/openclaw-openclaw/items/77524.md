## Summary

Keep this PR open: the runtime guard is a useful direction, but the patch currently validates raw tool arguments instead of the prepared arguments that the agent loop actually executes, which can turn existing valid calls into repeated-failure aborts.

## What This Changes

The PR adds a stream guard that fingerprints repeated tool schema validation failures, aborts after three identical failures with diagnostics, wires it into embedded attempts, and adds focused tests.

## Best Possible Solution

Land a guard that mirrors the agent loop’s prepare-then-validate path, keeps the existing stream repair/decoding composition, covers prepared null-argument calls, and records the runtime fix in the changelog.

## Reproduction Assessment

Yes, at source level: a normalized parameterless tool with prepareArguments(null) => {} can be called repeatedly with null args, and current agent execution prepares before validating while this guard validates the raw null first. That makes the PR able to abort calls that current main would execute successfully.

## Solution Assessment

No. The guard should validate the same prepared arguments used by execution; otherwise the abort policy observes a different contract from the runtime it is protecting.

- **[P2] Prepare args before counting schema failures:** `src/agents/pi-embedded-runner/run/attempt.tool-schema-validation-guard.ts:252-256`
