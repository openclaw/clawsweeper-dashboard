## Summary

Keep open: the PR addresses a real local exec workdir bug, but the current patch is not merge-ready because it expands `~` through OpenClaw's effective home rather than the OS user home. Current main and the latest release still have the original raw-tilde fallback behavior, so this should stay open as a candidate fix for the canonical issue.

## What This Changes

This PR changes `resolveWorkdir()` to expand a leading `~` before statting and returning the local exec workdir, while preserving fallback behavior for unavailable paths.

## Best Possible Solution

Use one canonical local exec fix that expands host `workdir` tilde paths against the OS account home, preserves the current fallback policy unless maintainers choose fail-fast separately, and adds divergent `HOME`/`OPENCLAW_HOME` regression coverage.

## Reproduction Assessment

Yes. Source inspection shows local gateway exec reaches `resolveWorkdir()` and current main plus `v2026.6.8` stat the literal `~/...` path before falling back; I did not run a live command in this read-only review.

## Solution Assessment

No. Expanding leading `~` is a good narrow direction, but this implementation uses the effective OpenClaw home instead of the OS account home used by sibling host file tools.

- **[P1] Expand workdir tilde against the OS home:** `src/agents/bash-tools.shared.ts:184`
  - body: `expandHomePrefix(workdir)` uses the effective OpenClaw home when `OPENCLAW_HOME` is set.
