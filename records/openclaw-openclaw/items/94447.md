## Summary

Keep open: this PR targets a valid current-main and latest-release exec bug, but the patch expands `~` through OpenClaw's effective home and can still run from the wrong directory when `OPENCLAW_HOME` diverges from the OS home.

## What This Changes

The PR changes `resolveWorkdir()` in `src/agents/bash-tools.shared.ts` to expand a leading `~` before statting and returning the host exec workdir.

## Best Possible Solution

Use one canonical exec fix that expands host workdir `~` against the OS account home, preserves the existing fallback policy unless maintainers choose fail-fast elsewhere, and covers `OPENCLAW_HOME` divergence in tests.

## Reproduction Assessment

Yes. Source inspection shows local gateway exec still calls `resolveWorkdir()`, and current main plus `v2026.6.8` stat the literal `~/...` path before falling back; I did not run a live exec because this review is read-only.

## Solution Assessment

No. Expanding leading `~` is a narrow fix direction, but this implementation uses the effective OpenClaw home instead of the OS account home, so it needs a small repair and regression coverage before it is the best fix.

  - body: `expandHomePrefix(workdir)` uses the effective OpenClaw home when `OPENCLAW_HOME` is set.
