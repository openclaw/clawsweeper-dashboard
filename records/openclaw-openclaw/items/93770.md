## Label Justifications

- `P1`: The report describes a recent Telegram channel regression that can make outgoing messages unreadable for affected users.
- `impact:message-loss`: Unreadable Telegram replies replace the delivered content with unsupported-message placeholders for affected clients.

## Summary

Current main already provides the requested Telegram rich-message opt-out behavior: standard Telegram HTML delivery is now the default and Bot API 10.1 rich messages require explicit `richMessages: true`. The fix is on main via the merged readable-default PR, but it is not proven shipped in `v2026.6.8`.

## What This Changes

The issue requests a Telegram configuration option that prevents `sendRichMessage` and keeps outgoing Telegram messages readable as plain or standard HTML text.

## Best Possible Solution

Keep the merged default-off `richMessages` implementation, ship it in the next release, and handle any fallback for users who explicitly enable rich messages as a separate narrow Telegram follow-up.

## Reproduction Assessment

Yes for the shipped report by source path: rich-message delivery produces the unsupported-client symptom when used by default, and current main avoids that path unless `richMessages` is explicitly true. I did not run a live Telegram Desktop/Web reproduction in this read-only review.

## Solution Assessment

Yes. The current-main solution is the narrow owner-boundary fix: make broadly readable standard Telegram HTML delivery the default and keep Bot API 10.1 rich messages as explicit opt-in instead of adding a second disable flag.
