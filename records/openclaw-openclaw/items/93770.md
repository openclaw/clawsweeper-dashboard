## Summary

Close: current main already restores readable Telegram delivery by default and exposes Bot API 10.1 rich messages only as an explicit `richMessages` opt-in; the fix is main-only after `v2026.6.8`, so release users need the next patch/current-main build.

## What This Changes

The issue requests a Telegram configuration option to disable `sendRichMessage` and fall back to readable plain or standard HTML Telegram messages on clients that render rich messages as unsupported.

## Best Possible Solution

Keep the merged PR 93279 behavior: standard Telegram HTML/plain delivery by default, with Bot API 10.1 rich messages available only through explicit `channels.telegram.richMessages: true`, and ship that in the next patch release.

## Reproduction Assessment

Yes from source and related report evidence: `v2026.6.8` routes default Telegram text through `sendRichMessage`, while current main gates rich transport behind `richMessages === true` and falls back to standard `sendMessage`. I did not run a live Telegram Desktop/Web repro in this read-only review.

## Solution Assessment

Yes. The current main implementation is the narrow owner-boundary solution: keep broadly compatible Telegram `sendMessage`/HTML as the default and use the existing `richMessages` opt-in instead of adding a second disable flag.
