## Summary

Keep open: current main and the latest release still reject BOM-prefixed markdown frontmatter before skill metadata is parsed, and the open linked implementation PR should remain the fix path until it merges.

## What This Changes

The issue asks for BOM-prefixed `SKILL.md` frontmatter to be parsed so Windows-edited skills still appear in `openclaw skills list`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81748, or an equivalent shared parser fix with parser and workspace skill discovery regression coverage, then close this issue after merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: Node preserves a leading U+FEFF, current main checks for `---` before stripping it, and the skill loader drops entries when parsed descriptions are missing.

## Solution Assessment

Is this the best way to solve the issue? Yes: fixing `src/markdown/frontmatter.ts` is the narrowest maintainable solution because skills delegate to the shared parser; fixing only the skill loader would leave other frontmatter consumers inconsistent.
