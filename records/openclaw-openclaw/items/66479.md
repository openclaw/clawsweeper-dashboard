## Summary

Keep open. Current main and the latest release still normalize line endings and then require the content to start with `---`, so a leading UTF-8 BOM still prevents skill frontmatter from being parsed and can make the skill disappear from `openclaw skills list`; the related Microsoft tracker is a checklist, not an implementation fix.

## What This Changes

The issue asks OpenClaw to discover skills whose `SKILL.md` frontmatter begins with a UTF-8 BOM written by Windows/PowerShell tooling.

## Best Possible Solution

Update the shared markdown frontmatter parser to tolerate one leading UTF-8 BOM before the opening `---`, then add focused coverage proving both direct parser use and workspace skill loading preserve a BOM-prefixed `SKILL.md`.

## Reproduction Assessment

Yes. The source path gives a high-confidence reproduction: U+FEFF before `---` makes `startsWith("---")` false, `parseFrontmatterBlock` returns `{}`, and the skill loader drops the entry because `description` is missing.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/markdown/frontmatter.ts` rather than a skill-only workaround because skills, hooks, and plugin bundle command parsing all consume the shared parser.
