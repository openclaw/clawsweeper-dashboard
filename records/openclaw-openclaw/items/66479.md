## Summary

Keep open: current main and the latest stable release still reject BOM-prefixed frontmatter, and that flows into skill discovery by dropping the skill when `description` is not parsed.

## What This Changes

Accept `SKILL.md` files whose YAML frontmatter starts after a UTF-8 BOM so Windows-edited skills remain discoverable.

## Best Possible Solution

Teach the shared markdown frontmatter parser to ignore exactly one leading UTF-8 BOM before the opening `---`, preserving existing YAML behavior, with direct parser and workspace skill discovery regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. A BOM-prefixed decoded string fails the current `startsWith('---')` gate, and the skill loader drops the resulting entry when `description` is missing.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/markdown/frontmatter.ts`, because skills use the shared parser and the same tolerance should apply consistently to markdown frontmatter consumers.
