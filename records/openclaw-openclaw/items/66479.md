## Summary

Keep open: current main and v2026.5.7 still reject BOM-prefixed markdown frontmatter before skill metadata is parsed, which can drop otherwise valid skills from discovery on Windows.

## What This Changes

The issue requests that `SKILL.md` discovery tolerate a leading UTF-8 BOM before markdown frontmatter.

## Best Possible Solution

Teach the shared markdown frontmatter parser to ignore exactly one leading UTF-8 BOM before the opening `---`, then add parser and workspace skill discovery regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: Node preserves the leading U+FEFF from the reported bytes, current main checks for `---` before stripping it, and the skill loader drops entries without a parsed `description`. I did not run the full Windows CLI flow.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/markdown/frontmatter.ts`; fixing only the skill loader would leave other shared frontmatter consumers inconsistent.
