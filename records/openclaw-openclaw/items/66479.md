## Summary

Keep open: current main and the latest stable release still reject BOM-prefixed markdown frontmatter, and the skill loader then drops the skill because `description` is not parsed. The source path is narrow and no focused open implementation PR was found.

## What This Changes

The issue requests that BOM-prefixed UTF-8 `SKILL.md` files still parse frontmatter so `openclaw skills list` discovers the skill on Windows.

## Best Possible Solution

Teach the shared markdown frontmatter parser to ignore exactly one leading UTF-8 BOM before the opening `---`, with parser and workspace skill discovery regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: Node decodes the reported bytes to leading U+FEFF, current main checks `startsWith('---')` before stripping it, and the skill loader drops entries without parsed descriptions. I did not run a full Windows CLI flow during this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/markdown/frontmatter.ts`; fixing only the skill loader would leave the shared parser inconsistent for other markdown frontmatter consumers.
