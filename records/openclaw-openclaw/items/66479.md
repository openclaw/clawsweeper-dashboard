## Summary

Keep this open. Current main still rejects BOM-prefixed markdown frontmatter before skill parsing, so a Windows-edited `SKILL.md` can still disappear from `openclaw skills list`; the related write-tool BOM report and Microsoft tracker do not implement this focused parser fix.

## What This Changes

The issue requests that BOM-prefixed `SKILL.md` frontmatter remain discoverable on Windows after PowerShell/.NET writes a UTF-8 BOM.

## Best Possible Solution

Update the shared markdown frontmatter parser to tolerate one leading UTF-8 BOM before the opening `---`, then add focused parser and skill-loading coverage proving a BOM-prefixed `SKILL.md` remains discoverable.

## Reproduction Assessment

Yes. The source path and read-only string check give a high-confidence reproduction: U+FEFF before `---` fails the current delimiter check, and the skill loader drops the entry when `description` is not parsed.

## Solution Assessment

Yes. The narrowest maintainable fix is in the shared parser rather than a skill-only workaround, because skills, hooks, and plugin bundle command parsing all consume `parseFrontmatterBlock`.

Fix https://github.com/openclaw/openclaw/issues/66479. Current `src/markdown/frontmatter.ts` rejects BOM-prefixed markdown frontmatter because `extractFrontmatterBlock` checks `normalized.startsWith("---")` before handling a leading U+FEFF.
