## Summary

Keep open: the bug is still present on current `main`, and the linked implementation PR is open and explicitly closes this issue, so the issue should remain open until that PR lands or is rejected.

## What This Changes

The issue reports that BOM-prefixed `SKILL.md` frontmatter is not parsed, causing affected skills to disappear from `openclaw skills list` on Windows.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81748 if it remains correct, then close this issue from the merged implementation.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a leading UTF-8 BOM decodes to U+FEFF, current `main` checks for `---` before removing it, and the skill loader drops entries whose `description` is not parsed. I did not run the native Windows CLI flow in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to handle exactly one leading BOM in the shared markdown frontmatter parser, with parser and workspace skill discovery regression coverage as proposed in the linked PR.
