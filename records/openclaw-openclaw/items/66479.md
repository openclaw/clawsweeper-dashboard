## Summary

Keep open: current main still has the reported BOM failure path in the shared frontmatter parser, and that flows directly into workspace skill discovery.

## What This Changes

The issue asks OpenClaw to keep discovering skills when `SKILL.md` begins with a UTF-8 BOM before the markdown frontmatter delimiter.

## Best Possible Solution

Update the shared markdown frontmatter parser to tolerate exactly one leading UTF-8 BOM before the opening delimiter, with regression coverage for both direct parsing and workspace skill discovery.

## Reproduction Assessment

Yes, from source inspection and a local Node decoding check. A `SKILL.md` beginning with `\uFEFF---` makes the shared parser return empty frontmatter, and the skill loader then drops the entry because the description is missing.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/markdown/frontmatter.ts`, because skills use the shared parser and the same tolerance is appropriate for other markdown-frontmatter consumers.
