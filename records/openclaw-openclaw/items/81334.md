## Summary

Keep open. Current main still has the source-level path that drops a configured external channel when its manifest declares `channels[]` but lacks matching `channelConfigs`, and an open linked PR is already using closing syntax to address this issue.

## What This Changes

The issue reports that third-party channel plugins declaring `channels[]` without `channelConfigs` disappear from `openclaw channels list` and the Control UI after upgrading to v2026.5.7.

## Best Possible Solution

Land a narrow fix that keeps configured manifest-declared third-party channels visible even without `channelConfigs`, while preserving the manifest warning and adding focused regression coverage for the read-only listing path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a configured external channel can be discovered from `channels[]`, but `addManifestChannelPlugins` skips it when `canUseManifestChannelPlugin` finds no `channelConfigs` or `channelCatalogMeta`.

## Solution Assessment

Yes, with one caveat. The maintainable fix is in the generic read-only channel discovery path plus a clearer manifest diagnostic; the open linked PR follows that direction but should be reviewed for unrelated diff cleanup before merge.
