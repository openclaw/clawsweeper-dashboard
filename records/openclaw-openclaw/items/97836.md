## What This Changes

The PR adds a status-only memory session catch-up refresh path and a regression test for marking missing indexed session transcripts dirty without syncing.

## Best Possible Solution

Adopt one canonical fix that makes status-mode dirty detection complete before `manager.status()` is returned, without running a full sync from plain status.

## Reproduction Assessment

Yes, from source inspection: plain status can read `manager.status()` before the detached catch-up scan updates `sessionsDirty`. I did not run a live CLI repro because this review is read-only.

## Solution Assessment

No. Reusing the existing session file/index comparison is the right helper shape, but doing it as an unawaited constructor task is not the best fix for a synchronous status command.
