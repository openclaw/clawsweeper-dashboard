## Summary

Keep this open. Current main still does an exact configured-model id match in the context-window guard, so a bare configured model entry can be missed when the runner passes the provider-qualified runtime id, causing the effective context limit to fall back to runtime metadata.

## What This Changes

The issue asks OpenClaw to treat bare and provider-qualified model ids as equivalent when resolving configured contextTokens for a selected provider.

## Best Possible Solution

Normalize model-id comparison inside resolveContextWindowInfo for the selected provider, then add focused tests covering bare configured ids with provider-qualified runtime ids and the inverse provider-qualified configured id case.

## Reproduction Assessment

Yes, from source inspection: calling resolveContextWindowInfo with provider github-copilot, modelId github-copilot/<id>, and a configured bare <id> cannot hit the current exact m.id === modelId match. I did not execute Vitest because pnpm is unavailable in this environment.

## Solution Assessment

Yes, the narrow maintainable fix is to normalize only the selected provider prefix in resolveContextWindowInfo and cover it with focused guard tests. Broader GitHub Copilot 1M promotion policy should stay separate from this matching bug.
