## Summary

Keep open as a valid narrow bug: current main and the latest release still exact-match configured model IDs in the context-window guard, so a provider-scoped runtime model ref can miss a bare configured model entry and persist the lower runtime cap. A linked implementation PR exists but is open and unmerged, so this issue should stay open until that PR is corrected and landed or replaced.

## What This Changes

The issue requests provider-prefix-aware matching in `resolveContextWindowInfo` so configured `contextTokens` and `contextWindow` apply whether the runtime model ID is bare or provider-scoped.

## Best Possible Solution

Land provider-prefix-only bare/scoped matching inside `resolveContextWindowInfo`, preserve nested slash-containing model IDs, and add focused regression coverage for both runtime-scoped and config-scoped directions.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: call `resolveContextWindowInfo` with provider `github-copilot`, runtime model ID `github-copilot/claude-opus-4.7-1m-internal`, a bare configured model ID, and runtime `modelContextWindow: 128000`; the strict matcher misses the configured 936k cap.

## Solution Assessment

Yes. Fixing the resolver is the narrow maintainable path, but the implementation should strip or prepend only the active provider prefix rather than comparing final slash segments that can collide across nested model IDs.
