## Summary

Keep open as a valid narrow bug. Current `main` still matches configured model entries with strict `m.id === modelId`, so a bare configured model id can miss a provider-scoped runtime id and fall back to the Copilot manifest's smaller context window.

## What This Changes

Honor configured model `contextTokens` when the runtime model id is provider-scoped but the matching provider config entry uses a bare model id.

## Best Possible Solution

Add provider-aware bare/scoped id equivalence inside `resolveContextWindowInfo`, keep matching constrained to the selected provider, and cover the reported Copilot shape with a focused regression test.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: call `resolveContextWindowInfo` with provider `github-copilot`, model id `github-copilot/claude-opus-4.7-1m-internal`, a bare configured model id, and runtime `modelContextWindow: 128000`; the strict matcher misses the configured `contextTokens`.

## Solution Assessment

Yes. The narrow fix is to make the resolver treat bare and provider-scoped ids as equivalent only under the matching provider, matching existing model-resolution semantics without changing Copilot defaults or adding configuration.
