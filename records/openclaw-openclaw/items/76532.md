## Summary

Keep open: current main and the latest release still strict-match configured model IDs in the context-window guard, and the linked fix PR is open and unmerged.

## What This Changes

Honor user-configured contextTokens/contextWindow when runtime model IDs use provider/model form but the provider config stores the provider-local model ID.

## Best Possible Solution

Land a provider-prefix-only matcher in `resolveContextWindowInfo`, with regression coverage for both bare/scoped directions and nested slash-containing model IDs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `resolveContextWindowInfo` with provider `github-copilot`, runtime model ID `github-copilot/claude-opus-4.7-1m-internal`, a bare configured model ID, and runtime `modelContextWindow: 128000`; the strict matcher misses the configured cap.

## Solution Assessment

Yes, fixing the resolver is the narrow maintainable path. The implementation should strip or prepend only the active provider prefix, not compare final slash segments that can collide across nested provider model IDs.
