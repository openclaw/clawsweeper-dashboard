## Label Justifications

- `P2`: This is a narrow but real context-window bug affecting configured large-context model sessions, with an open linked fix path.
- `impact:session-state`: The issue reports wrong `contextTokens` being persisted into session state and shown in session/status surfaces.
- `impact:auth-provider`: The bug is in provider/model reference matching and can cause the selected provider model configuration to be ignored.

## Summary

Keep open: current main and the latest release still exact-match configured model IDs in `resolveContextWindowInfo`, so the reported provider-scoped runtime ref can still miss a bare configured model entry and persist the lower runtime context cap. The open linked fix PR already references this issue, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that provider-scoped runtime model IDs can miss bare `models.providers.<provider>.models[]` entries, causing configured large `contextTokens` to fall back to a smaller runtime/default context window.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77764, or an equivalent narrow fix, with regression coverage for both bare and provider-scoped model ID forms before closing this issue.

## Reproduction Assessment

Yes. A focused source repro is to call `resolveContextWindowInfo` with provider `github-copilot`, model ID `github-copilot/<model>`, and a configured model entry whose `id` is only `<model>`; current main falls through because the guard still uses strict equality.

## Solution Assessment

Yes. A provider-prefix-aware match inside `resolveContextWindowInfo`, with regression tests for both scoped-runtime/bare-config and bare-runtime/scoped-config cases, is the narrow maintainable fix.
