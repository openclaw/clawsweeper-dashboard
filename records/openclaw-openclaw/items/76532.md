## Summary

Keep open. Current main still uses strict model-id equality in `resolveContextWindowInfo`, so a bare configured model entry can miss a provider-scoped runtime id and the resulting runtime context value can be persisted and displayed as the session limit.

## What This Changes

Fix context-window resolution so configured bare model ids and matching `provider/model` runtime ids use the same configured `contextTokens` or `contextWindow` value.

## Best Possible Solution

Teach `resolveContextWindowInfo` to match bare and provider-scoped ids symmetrically within the normalized provider, add focused regression coverage, and leave provider defaults/config schema unchanged.

## Reproduction Assessment

Yes. A focused unit repro can call `resolveContextWindowInfo` with `provider: "github-copilot"`, a bare configured model id, and `modelId: "github-copilot/<id>"`; current source shows it misses the config row and falls back to the runtime model window.

## Solution Assessment

Yes. The narrow maintainable fix is to make the guard's model-id comparison provider-aware, matching the adjacent runner model-resolution helper, with coverage for both bare-to-scoped and scoped-to-bare forms.
