## Summary

Keep open as a valid narrow bug. Current main still exact-matches configured model IDs in the context-window guard, and an open linked fix PR exists but has not merged, so the issue should remain open until that PR is corrected and landed or replaced.

## What This Changes

The issue asks for `resolveContextWindowInfo` to treat bare and provider-scoped model IDs as equivalent when resolving configured `contextTokens` and `contextWindow` values.

## Best Possible Solution

Implement provider-prefix-only bare/scoped matching inside `resolveContextWindowInfo`, preserve slash-containing nested model IDs, and add focused regression coverage for both runtime-scoped and config-scoped directions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `resolveContextWindowInfo` with provider `github-copilot`, runtime model id `github-copilot/claude-opus-4.7-1m-internal`, a bare configured model id, and runtime `modelContextWindow: 128000`; the strict matcher misses the configured cap.

## Solution Assessment

Yes, fixing the resolver is the narrow maintainable path. The implementation should strip or prepend only the active provider prefix, not compare final slash segments that can collide across nested provider model IDs.
