## Summary

Keep open: current main still has a source-level path where WhatsApp voice-note preflight can transcribe audio but cannot deliver the configured transcript echo because the plugin passes no channel/target metadata into the SDK echo path.

## What This Changes

The issue asks for `tools.media.audio.echoTranscript` to fire again for externalized `@openclaw/whatsapp` voice-note transcription.

## Best Possible Solution

Keep the fix in the WhatsApp plugin by passing the existing prepared channel/account/target context into `transcribeFirstAudio`, then add focused regression coverage for direct and broadcast/group preflight echo routing.

## Reproduction Assessment

Yes from source inspection, though not from a live WhatsApp run. The WhatsApp preflight calls can produce a transcript but pass no delivery metadata to the SDK echo sender, which requires channel and target fields to send the echo.

## Solution Assessment

Yes. Reusing the existing SDK `transcribeFirstAudio`/`sendTranscriptEcho` path with complete WhatsApp context is narrower and more maintainable than adding plugin-local echo config or moving WhatsApp-specific behavior back into core.
