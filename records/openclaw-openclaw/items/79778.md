## Summary

Keep open: current main still has the WhatsApp source path that can transcribe a voice note but cannot deliver the configured transcript echo, and an open PR already uses GitHub closing syntax for this issue.

## What This Changes

The issue reports that WhatsApp voice-note transcription reaches the agent, but `tools.media.audio.echoTranscript` no longer sends the configured transcript echo after the WhatsApp plugin was externalized.

## Best Possible Solution

Review and land a focused WhatsApp plugin fix that passes the existing route/account/target context into `transcribeFirstAudio`, with regression coverage for both early and fallback preflight paths.

## Reproduction Assessment

Yes, from source inspection rather than a live WhatsApp run. Current main can transcribe the voice note, then calls the echo sender without the channel/target fields required to deliver the echo.

## Solution Assessment

Yes. Passing existing WhatsApp routing metadata into the existing SDK `transcribeFirstAudio` path is narrower than adding plugin-local echo configuration or moving WhatsApp-specific echo behavior back into core.
