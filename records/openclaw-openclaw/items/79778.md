## Summary

Keep open: the report matches current source. WhatsApp preflight STT calls the shared echo-capable helper with only media fields, so `sendTranscriptEcho` has no provider or destination and cannot deliver the configured echo.

## What This Changes

The issue reports that WhatsApp voice-note preflight transcribes audio but does not send the configured `tools.media.audio.echoTranscript` echo in `@openclaw/whatsapp@2026.5.7`.

## Best Possible Solution

Fix the WhatsApp plugin preflight context so successful voice-note STT includes the same provider, destination, account, and thread metadata that `sendTranscriptEcho` needs, with regression tests for DM and group/broadcast paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: send a WhatsApp audio message with `echoTranscript` enabled, and the current preflight call can transcribe the file but lacks the routing fields needed for echo delivery. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is plugin-local routing-context plumbing into the existing shared preflight helper, mirroring the Telegram fix, rather than adding a new config option or broad core migration.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79778. WhatsApp voice-note preflight currently calls `transcribeFirstAudio` with only `MediaPaths`/`MediaTypes`, so `tools.media.audio.echoTranscript` cannot route an echo even though STT succeeds and the agent body is replaced. Keep the fix inside the WhatsApp plugin/shared media-runtime boundary: pass a proper `MsgContext` into both early `on-message.ts` preflight and `process-message.ts` fallback with `Provider`/`Surface`/`OriginatingChannel: "whatsapp"`, the correct `OriginatingTo` for DM and group conversations, `AccountId`, and any thread/message metadata already available. Mirror the Telegram fix pattern from commit `c5bc4b6892644d3f4edc23114c1804881e3e7cec`; do not add a new config option, do not rerun STT, and avoid double echo when early preflight feeds multiple broadcast agents. Add or update WhatsApp regression coverage proving `transcribeFirstAudio` receives routable context when `echoTranscript` is enabled and that agent body replacement still works. Add a user-facing changelog fix entry if the branch changes behavior.
