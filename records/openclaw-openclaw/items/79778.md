## Summary

Keep open: current main still routes WhatsApp voice-note preflight through `transcribeFirstAudio` with only media path/type context, so the documented `echoTranscript` sender has no channel/target and returns before delivery; https://github.com/openclaw/openclaw/pull/79788 is the right unmerged fix path.

## What This Changes

Report that WhatsApp voice-note transcripts reach the agent context but `tools.media.audio.echoTranscript` no longer delivers the configured echo after the WhatsApp plugin was externalized.

## Best Possible Solution

Land a focused WhatsApp plugin fix that passes existing route/account/target metadata into `transcribeFirstAudio` and keeps regression coverage around both preflight paths.

## Reproduction Assessment

Yes from source inspection, though not from a live WhatsApp run. Both WhatsApp audio preflight call sites pass only media path/type context while `sendTranscriptEcho` requires channel and target fields to deliver the echo.

## Solution Assessment

Yes. Passing the existing WhatsApp routing metadata into the existing SDK preflight path is narrower than adding plugin-local echo config or moving WhatsApp-specific behavior back into core.
