The PR changes Control UI Quick Settings avatar preview rendering so local assistant avatars use the existing fetched blob URL or fallback instead of a protected `/avatar/<agentId>` image URL.

## Best Possible Solution

Refresh the branch against current main, keep the UI-side bearer/blob/fallback approach, and land only after redacted Quick Settings proof shows no direct `/avatar/main` image render.

## Reproduction Assessment

Yes, at source level: current main constructs a protected `/avatar/<agentId>` URL for local Quick Settings avatars, and the gateway route rejects unauthenticated avatar image requests when auth is enabled. I did not run a live browser/Tailscale reproduction in this read-only review.

## Solution Assessment

Yes for the code shape, but not merge-ready yet: reusing the existing bearer-authenticated blob/fallback path is narrower and safer than query-token image URLs or weakening avatar auth. The remaining blockers are branch refresh and direct after-fix Quick Settings proof.
