## Summary

The PR targets a real Control UI auth bug: current main can pass a protected same-origin `/avatar/:agentId` URL into Quick Settings before the authenticated blob URL is available. The patch is narrow, has useful live proof and focused regression tests, and I did not find a blocking correctness or security issue in the diff.

## What This Changes

The PR changes Control UI Quick Settings avatar selection so local assistant avatars use the authenticated blob URL or fallback instead of rendering protected `/avatar/:agentId` routes directly, with focused UI regression coverage.

## Best Possible Solution

Land the narrow Control UI fix after normal PR checks so local avatar preview images consistently use the authenticated blob-fetch path under token/hash-based auth.

## Reproduction Assessment

Yes, from source inspection and the PR's live proof: current main can pass `/avatar/main` into a Quick Settings image path, while the protected avatar route returns 401 without bearer auth in the contributor's live setup. I did not rerun the browser flow locally because this review is read-only.

## Solution Assessment

Yes. Reusing the already authenticated `chatAvatarUrl` blob and falling back while it is unavailable is the narrowest maintainable fix; it avoids adding another auth or image-loading path.
