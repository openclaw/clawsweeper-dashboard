## Summary

Keep open: current main still does not implement the requested force-enable behavior, and this PR contains useful narrow code/test work. It should not merge as-is because the status/CLI contract becomes contradictory, the public docs still omit the new semantics, and the external-PR real behavior proof gate is not satisfied.

## What This Changes

The branch makes `skills.entries.<key>.enabled: true` bypass runtime skill eligibility in runtime inclusion and status reporting, with focused tests for force-enable, force-disable, default eligibility, and bundled allowlist precedence.

## Best Possible Solution

Land a narrow, documented force-enable contract only if maintainers want `enabled: true` to be a blanket runtime-gate override, and make status output distinguish forced eligibility from requirements-passed eligibility.

## Reproduction Assessment

Yes, by source inspection. Current main falls through from `enabled: true` into `evaluateRuntimeEligibility`, and an unmet `requires.bins` check returns false; I did not run tests because this was a read-only review.

## Solution Assessment

No, not as submitted. The core code direction is small, but the PR changes public config semantics and status meaning, so it needs docs/status copy, maintainer acceptance of the broadened override, real behavior proof, and a branch refresh before merge.
