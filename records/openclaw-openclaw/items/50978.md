## Summary

Keep open. Current main still has the reported force-enable gap, and the PR is a focused implementation candidate, but it should not merge as-is because the public skills config docs would remain stale for the new `enabled: true` contract.

## What This Changes

The branch adds an `enabled === true` short-circuit after the bundled-skill allowlist check, updates skill-status eligibility for force-enabled skills, and adds Vitest coverage for force-disable, force-enable, default runtime fallthrough, and allowBundled precedence.

## Best Possible Solution

Land this PR or an equivalent narrow patch only after updating the skills docs for the new force-enable semantics, preserving `skills.allowBundled` precedence, and refreshing CI on current main.

## Reproduction Assessment

Yes. On current main, a skill with an unmet `requires.bins` requirement and `skills.entries.<key>.enabled: true` still reaches `evaluateRuntimeEligibility`, which returns false when `hasBinary` fails; the PR’s new unit test encodes that path.

## Solution Assessment

No, not quite as submitted. The code direction is narrow and matches the requested behavior, but the mergeable solution should also update the public skills config/docs so the changed `enabled: true` contract is documented.
