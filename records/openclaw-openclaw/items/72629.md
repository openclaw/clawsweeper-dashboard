## Summary

Keep open. This is a current architecture/design discussion with concrete cost observations and useful community input; current main has lower-level isolation, bounded history recall, context-engine hooks, fan-out guards, usage reporting, and agent-to-agent loop caps, but it does not yet define tiered conversational context delivery, shared decision summaries, saliency routing, or pessimistic multi-agent budget reservation.

## What This Changes

The issue asks for maintainer/community design input on how OpenClaw should handle token-cost scaling for conversational multi-agent coordination without losing emergent cross-agent information.

## Best Possible Solution

Keep the thread open until maintainers choose the boundary: either a generic coordination/cost-governance primitive, plugin/context-engine hooks for tiered delivery and budget reservation, or explicit docs that these policies belong in scenario-specific orchestrators.

## Reproduction Assessment

No, not as a high-confidence repo-local reproduction. The issue gives a plausible manual setup and comments add concrete observations, but current main does not contain a replayable runner, token traces, or deterministic fixture for this scenario.

## Solution Assessment

Not applicable as a patch because no implementation is proposed. The narrow maintainable next step is maintainer architecture judgment; current main's isolation, recall, usage, and fan-out guards are useful primitives but not a complete scaling policy.
