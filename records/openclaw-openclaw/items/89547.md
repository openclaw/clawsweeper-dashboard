## Summary

Keep open. The implementation is coherent and has solid Telegram proof, but it adds a compatibility-sensitive Telegram config/default surface and silently changes existing group-history behavior for installs that omit the new setting; maintainers need to explicitly accept that privacy-first upgrade policy or require a compatibility-preserving default before merge.

## What This Changes

Adds a Telegram `includeGroupHistoryContext` config mode (`mention-only`, `recent`, `none`) across schema, docs, UI metadata, prompt context construction, recovered forum dispatch, and regression tests.

## Best Possible Solution

Land after an explicit maintainer choice: either keep shipped `recent` behavior as the default and make `mention-only`/`none` opt-in, or approve the privacy-first default with clear upgrade/release-note treatment.

## Reproduction Assessment

Yes. Current main and v2026.6.6 include recent Telegram group history when `historyLimit` is positive, while the PR default resolves omitted config to `mention-only` and the supplied proof shows ambient history is excluded by default.

## Solution Assessment

No, not as-is. The implementation is a plausible and well-covered mechanism, but the default inversion is not the safest upgrade shape unless maintainers explicitly choose the privacy-first break.
