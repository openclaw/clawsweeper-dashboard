## Summary

Keep open: this is a useful fix attempt for a real gateway/systemd bug, but the patch is not merge-ready because it only ignores stale `SUDO_USER` when the guessed sudo-user unit path is missing, so a root shell can still target the wrong user manager when both root and the stale sudo user have an OpenClaw unit. The submitted proof is also mock-only for a Linux systemd behavior change.

## What This Changes

The PR makes the systemd user-scope resolver async, checks for the `SUDO_USER` service unit before using machine scope, and updates the adjacent systemd tests.

## Best Possible Solution

Resolve the systemd target from the actual installed/current service owner so root-owned units under the current root environment win over stale `SUDO_USER`, while preserving intentional sudo-to-user flows and proving both cases with focused tests plus Linux systemd smoke proof.

## Reproduction Assessment

Yes for source reproduction: current main routes effective-root plus non-root `SUDO_USER` through machine scope, and the linked issue provides live Linux before/after evidence. I did not run a fresh live root systemd reproduction in this read-only review.

## Solution Assessment

No: the PR is a plausible partial fix, but using the sudo user's unit existence as the stale-env signal is not the best boundary. The safer fix is to prefer the owner implied by the current/root-installed unit and prove valid sudo-user cases separately.
