## What This Changes

The PR adds optional `engineCandidates` and `corpus` fields to `MemoryCorpusSupplement`, forwards memory search results into corpus supplements, re-exports `MemorySearchResult`, updates the SDK baseline hash, and adds memory-core tests.

## Best Possible Solution

Rebase over current main, preserve the latest `memory_search` behavior, and merge only if maintainers accept the additive `engineCandidates`, `corpus`, and `MemorySearchResult` SDK semantics.

## Reproduction Assessment

Yes for the source-level API gap: current main has `rawResults` in scope before calling corpus supplements, but the supplement signature and helper params do not expose `engineCandidates`. This is a feature/API limitation rather than a broken documented behavior bug.

## Solution Assessment

Mostly yes, pending maintainer SDK approval: passing the already available filtered search results through an optional field is the narrowest way to remove the redundant supplement-side search for post-pipeline consumers. The broader rerank-provider work is complementary rather than a substitute.
