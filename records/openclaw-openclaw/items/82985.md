## Summary

Keep open: the PR is a useful, still-unmerged candidate for the linked memory supplement API gap, but it changes public plugin SDK semantics and is currently dirty/conflicting against main, so it needs maintainer API approval plus a careful rebase before merge.

## What This Changes

The PR adds optional `engineCandidates` and `corpus` fields to `MemoryCorpusSupplement` params, forwards memory search candidates through memory-core, re-exports `MemorySearchResult`, updates the SDK API baseline hash, and adds focused tests.

## Best Possible Solution

Rebase over current main and merge only if maintainers approve the additive supplement params as stable plugin API, keeping this passthrough complementary to the broader pre-MMR rerank-provider work.

## Reproduction Assessment

Yes for the source-level API gap: current main computes `rawResults` before calling corpus supplements, but the supplement type and helper params do not expose `engineCandidates`. This is a feature/API limitation rather than a broken documented behavior bug.

## Solution Assessment

Mostly yes, pending maintainer SDK approval: passing the already available filtered search results through an optional field is the narrowest way to remove a redundant supplement-side search for post-pipeline consumers. The broader pre-MMR rerank-provider work is complementary, not a replacement for this passthrough.
