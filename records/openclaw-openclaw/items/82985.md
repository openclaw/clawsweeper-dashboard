## Summary

Keep open: current main and the latest release still lack the candidate passthrough, and this PR remains the direct implementation candidate for the linked supplement API issue rather than being superseded by the broader rerank-provider work.

## What This Changes

The PR adds optional engine candidates and corpus scope to memory corpus supplement params, forwards raw memory search results to supplements, re-exports the memory search result type, updates the SDK baseline hash, and adds focused memory-core tests.

## Best Possible Solution

Rebase over current main, preserve the current memory_search error-handling path, and merge only if maintainers approve this additive supplement contract as complementary to the broader rerank-provider design.

## Reproduction Assessment

Yes for the source-level API gap: current main computes rawResults before calling corpus supplements, but the supplement type and helper params do not expose engineCandidates. This is a feature/API limitation rather than a broken documented behavior bug.

## Solution Assessment

Mostly yes, pending maintainer SDK approval: passing the already available filtered search results through an optional field is the narrowest way to remove a redundant supplement-side search for post-pipeline consumers. The broader pre-MMR rerank-provider work is complementary, not a replacement for this passthrough.
