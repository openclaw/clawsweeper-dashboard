## Summary

Keep open. Current main still does not implement native LINE sticker send support, a `STICKER:packageId:stickerId` directive path, or structured inbound `packageId`/`stickerId` exposure; the active path remains review of the related focused send-only PR rather than closing this issue early.

## What This Changes

The issue requests LINE sticker send/receive support, sticker directive parsing, sticker-only payload delivery fixes, and a `line-sticker` skill/catalog/editor for AI-driven sticker selection.

## Best Possible Solution

Review and land or revise the focused LINE-native send path in #73378, keep the core change limited to a generic channelData-only coalescer fix, then track inbound sticker metadata and multi-channel sticker abstraction separately. The curated catalog/editor should stay ClawHub-bound unless maintainers explicitly promote it.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: a LINE reply containing `STICKER:packageId:stickerId` is not parsed, outbound LINE delivery has no sticker branch, and inbound stickers only become bracketed text without IDs.

## Solution Assessment

No for the original all-in-one request; yes for the split-scope direction discussed in the issue. A plugin-local LINE send implementation plus a generic channelData-only coalescer fix is narrower than adding a core `ReplyPayload.sticker` and bundled catalog in one change.
