## Summary

Keep open. Current main still lacks LINE native sticker sending and structured inbound sticker IDs, while the active replacement PR only covers the send path and leaves receive-side enrichment plus broader sticker abstraction for maintainer follow-up.

## What This Changes

The issue requests LINE sticker send/receive support and an AI-usable sticker catalog path for natural LINE conversations.

## Best Possible Solution

Land or revise the focused LINE send PR, then split inbound sticker metadata and any cross-channel sticker abstraction into explicit follow-up work while keeping catalog/editor content ClawHub-bound.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence path: a LINE reply containing STICKER:packageId:stickerId is not parsed or dispatched as a native sticker, and incoming stickers do not expose stickerId as structured metadata.

## Solution Assessment

No for the original all-in-one core plus catalog proposal; yes for the current split direction. A LINE-plugin send path via channelData, separate inbound metadata follow-up, and ClawHub catalog/editor keeps the owner boundary narrower.
