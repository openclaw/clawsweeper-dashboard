## Summary

Keep open: current main and v2026.6.10 still lack native LINE sticker sending and structured inbound packageId/stickerId exposure, although current main already fixed the generic channelData-only payload preservation part of the earlier proposal.

## What This Changes

Request to add LINE sticker send and receive support, plus optional skill/catalog-driven sticker selection for LINE users.

## Best Possible Solution

Land a maintainer-scoped LINE plugin send path if approved, decide inbound sticker metadata separately, and keep catalog/editor content ClawHub-first unless maintainers explicitly promote it into core.

## Reproduction Assessment

Yes, at source level: current main has no LINE sticker carrier, no STICKER directive parser, and no LINE sticker send helper, while inbound stickers are only converted into text context. I did not run a live LINE account in this read-only review.

## Solution Assessment

No, not as originally proposed. A core `ReplyPayload.sticker` plus bundled catalog/editor is broader than the current plugin boundary; the safer path is LINE-owned `channelData` send support, separate inbound metadata, and ClawHub ownership for catalog/editor assets.
