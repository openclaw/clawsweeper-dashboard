## Summary

Keep open. Current main still lacks native LINE sticker send support, a LINE `STICKER:packageId:stickerId` directive path, and structured inbound sticker IDs; the best active path remains reviewing the narrower send-only PR before deciding follow-up scope.

## What This Changes

Request LINE bots to send and receive native stickers, parse a `STICKER:` directive, preserve sticker-only payloads, and support a Skill-based sticker catalog/editor.

## Best Possible Solution

Review and land or revise #73378 for LINE-native send support, then track inbound sticker metadata and any multi-channel abstraction separately; keep the sticker catalog/editor ClawHub-bound unless maintainers explicitly promote it.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence path: a LINE reply containing `STICKER:packageId:stickerId` is not parsed or dispatched as a sticker, and inbound stickers are converted to descriptive text without structured IDs.

## Solution Assessment

No for the original all-in-one core/catalog request; yes for the split direction now in discussion. A LINE-plugin send path plus a generic channelData-only coalescer fix is narrower than adding a core `ReplyPayload.sticker` and bundled catalog in one change.
