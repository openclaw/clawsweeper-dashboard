## Summary

Keep open. Current main still lacks LINE native sticker sending, a `STICKER:` directive path, and structured inbound sticker IDs; the active replacement PR is still open and only covers the send-side slice, while catalog/editor work remains a ClawHub/product-boundary decision.

## What This Changes

Request LINE bots to send and receive native LINE stickers, parse `STICKER:packageId:stickerId`, preserve sticker-only payloads, and support skill/catalog-driven sticker selection.

## Best Possible Solution

Review and land or close the focused send-side PR, then track inbound sticker metadata and cross-channel abstraction separately; publish catalog/editor work through ClawHub unless maintainers explicitly approve bundling it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `STICKER:packageId:stickerId` remains ordinary text, LINE has no sticker carrier/send branch, and incoming stickers lose structured IDs.

## Solution Assessment

No for the original all-in-one core plus bundled catalog proposal. The current split direction is better: LINE-owned send support via `channelData`, separate inbound metadata follow-up, and ClawHub for catalog/editor content.
