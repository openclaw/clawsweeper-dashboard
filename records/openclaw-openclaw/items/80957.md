## Summary

Keep open: the fix targets a real stale-context status path, but the current patch breaks existing CLI runtime alias behavior and still lacks after-fix real behavior proof.

## What This Changes

The PR changes `/status` and `session_status(current)` to ignore cached `contextTokens` when persisted runtime model identity appears stale after a selected/default model change, with regression tests.

## Best Possible Solution

Keep the selected/default-model refresh behavior, but make the stale-runtime check alias-aware so semantically equivalent runtime routes keep valid persisted context windows, then add real behavior proof and rerun focused status checks.

## Reproduction Assessment

Yes for the introduced regression: PR CI reproduces the alias status failure. The original stale-window issue is also source-reproducible from current `buildStatusMessage` precedence, but I did not run local tests because this review is read-only.

## Solution Assessment

No. The direction is maintainable, but the implementation treats raw label mismatch as stale even when `areRuntimeModelRefsEquivalent` says the selected and runtime models are the same route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve equivalent runtime aliases when dropping stale context:** `src/status/status-message.ts:672`
  - body: The new non-fallback path treats any raw active/selected label mismatch as stale. Known runtime aliases such as `claude-cli/claude-opus-4-7` versus `anthropic/claude-opus-4-7` are semantically equivalent, so this drops the valid persisted `contextTokens`; CI now fails the existing alias test because the rendered percentage changes from the persisted 1,000,000-token window to the looked-up 1,048,576-token window. Gate the stale-window override on `areRuntimeModelRefsEquivalent(...)` or equivalent alias-aware logic.
  - confidence: 0.92
