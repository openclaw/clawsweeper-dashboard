## Summary

Keep open: current main still lacks the central `ctx.msg` native-command change for Telegram channel posts, but this PR is not merge-ready because it expands a command execution surface without reusing the existing `channel_post` sender/auth/callback semantics and still lacks real Telegram proof.

## What This Changes

The PR changes Telegram native command handlers to read `ctx.msg`, accept `channel_post` updates, authorize channel slash commands through chat allow policy, and add focused auth/sequential-key tests.

## Best Possible Solution

Rebase onto current main and implement channel-post native commands through the same `sender_chat`, group policy, ingress authorization, and callback/session routing semantics as the normal `channel_post` path, then add focused tests and live Telegram proof.

## Reproduction Assessment

No live repro was run, but the review blockers are source-reproducible: the PR enables grammY `ctx.msg` channel posts for native commands while the auth path does not reuse current `sender_chat` normalization or channel-aware callback routing.

## Solution Assessment

No. Using `ctx.msg` is the right dependency-backed primitive, but the best fix must also reuse current channel_post sender identity, group-policy, ingress authorization, and callback/session routing before merge.

Full review comments:

- **[P1] Require explicit channel authorization when access groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `useAccessGroups` off and `commands.allowFrom` unset, this branch passes an empty authorizer list to the shared `configured` gate, which allows the command. That makes native/plugin slash commands run from readable Telegram channels rather than only explicitly allowed chats.
  - confidence: 0.86
- **[P1] Populate sender_chat before command allowlist checks:** `extensions/telegram/src/bot-native-commands.ts:351-352`
  - body: The PR makes channel posts reach command auth, but the auth context still derives sender identity from `msg.from` only. Telegram channel posts normally identify the posting channel via `sender_chat`, so `commands.allowFrom.telegram` and owner allowlists can reject intended channel commands because the allowed channel identity is never passed in.
  - confidence: 0.88
- **[P1] Route channel command callbacks through the channel session:** `extensions/telegram/src/bot-native-commands.ts:341-342`
  - body: Authorizing channel-post commands also enables inline-button native commands such as model or argument menus, but callback handling still treats channel chats as non-group callbacks. Follow-up clicks can be dropped by direct auth or mutate the clicking admin's direct session instead of the channel session.
  - confidence: 0.84
