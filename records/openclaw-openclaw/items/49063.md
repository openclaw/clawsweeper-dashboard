## Summary

Keep open: the central Telegram channel_post command fix is still useful and not implemented on current main, but this PR is not safe to merge because it changes a command-execution auth surface without live Telegram proof and still has channel authorization defects.

## What This Changes

The PR changes Telegram native/plugin slash command handling to read grammY ctx.msg for channel_post updates, treats channel posts as group-like for command auth/routing, and adds channel-post auth and ordering-key tests.

## Best Possible Solution

Refresh the branch on current main, choose a maintainer-approved channel_post auth rule, preserve sender_chat/per-channel policy/callback routing, then prove it with focused tests and live Telegram evidence.

## Reproduction Assessment

Yes, source-level: current main reads ctx.message in native command handlers while grammY documents ctx.msg as covering channel_post. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Using ctx.msg is the right dependency direction, but this branch still needs channel-specific auth policy, sender_chat allowlist handling, per-channel policy, callback routing, and live proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with commands.useAccessGroups=false and no commands.allowFrom, this branch passes an empty authorizer list. The shared configured-mode gate treats that as allowed, so any channel the bot can read can run native/plugin commands even though the PR is scoped to explicitly allowed chats.
  - confidence: 0.91
- **[P2] Populate channel-post sender before commands.allowFrom:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: When commands.allowFrom is configured, channel-post commands depend on sender identity, but this path still derives it only from msg.from. Telegram channel posts normally identify the posting channel via sender_chat, so explicitly allowed channel identities can be rejected once a command allowlist is configured.
  - confidence: 0.87
- **[P2] Honor per-channel policy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298-299`
  - body: The channel command path evaluates group policy without topic/group overrides and then treats any explicit groupConfig as enough to authorize the command. A channel entry with groupPolicy disabled or a narrowed policy can therefore still execute native/plugin commands.
  - confidence: 0.84
- **[P2] Route channel command callbacks through channel sessions:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Switching command handlers to ctx.msg makes interactive native commands such as /models reachable from channels, but callback handling still classifies channel chats as non-group. Button follow-ups can be dropped or applied to the wrong session after the initial channel command succeeds.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch expands Telegram channel command execution and has concrete authorization-policy defects; no dependency or workflow supply-chain change was found.

Concerns:

- **[medium] Channel commands can bypass explicit channel configuration:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: When access groups are off and commands.allowFrom is unset, the channel-post branch passes no authorizer and the shared configured-mode gate allows the command, expanding command execution beyond explicitly configured Telegram channels.
  - confidence: 0.91
- **[medium] Disabled per-channel policy can be ignored:** `extensions/telegram/src/bot-native-commands.ts:298`
  - body: The branch authorizes explicit channel entries from groupConfig while policy evaluation ignores group/topic overrides, so a per-channel disabled or narrowed policy can be bypassed for newly enabled native/plugin commands.
  - confidence: 0.84
- **[medium] Channel sender allowlists can be misapplied:** `extensions/telegram/src/bot-native-commands.ts:227`
  - body: The command path does not synthesize sender_chat into the sender identity before evaluating commands.allowFrom, so channel allowlists can reject intended channels or encourage unsafe fallback policy changes.
  - confidence: 0.87
