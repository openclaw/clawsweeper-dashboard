## Summary

Keep open: using grammY `ctx.msg` is the right entry point for channel-post commands, but the PR is still too narrow for a command-execution surface change because it does not preserve explicit channel authorization, sender_chat allowlist identity, or channel callback/session routing.

## What This Changes

The PR changes Telegram native and plugin command handlers to consume grammY `ctx.msg`, treats `channel_post` updates as group-like command contexts, and adds unit coverage for allowed and denied channel-post command authorization.

## Best Possible Solution

Land a narrow Telegram fix that uses `ctx.msg` but shares the normal channel_post sender normalization and explicit-channel authorization contract, resolves the channel_post `allowFrom` policy, updates callback/session handling, and includes focused tests plus redacted live Telegram proof.

## Reproduction Assessment

Yes from source: current main and v2026.6.8 use `ctx.message` in native/plugin command handlers while grammY command handlers can receive channel posts where `ctx.message` is undefined and `ctx.msg` is the supported shortcut. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is: `ctx.msg` is the correct dependency-backed entry point, but this patch does not carry the normal channel_post sender normalization, explicit-channel authorization, or callback/session routing through the newly enabled command path. The safer fix is to share or mirror the existing channel_post authorization contract and settle the linked allowFrom policy before merge.
