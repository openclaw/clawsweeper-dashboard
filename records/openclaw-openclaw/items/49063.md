## Summary

Keep open: this is a useful Telegram bug-fix direction, but the branch is not merge-ready because it expands channel_post command execution while leaving explicit-channel authorization, sender identity, callback session routing, real Telegram proof, and a dirty merge state unresolved.

## What This Changes

The PR changes Telegram native and plugin command handlers to use grammY `ctx.msg` so `channel_post` updates can invoke slash commands, adds channel-post auth tests, and makes Telegram update/sequential keys prefer `ctx.msg`.

## Best Possible Solution

Refactor channel_post native commands through the existing normalized channel_post authorization/session/callback boundary, define the sender_chat and allowFrom policy, and require redacted live allowed, denied, and callback proof before merge.

## Solution Assessment

No: using raw `ctx.msg` is a plausible entry point, but the best fix needs to share the existing normalized channel_post authorization and callback/session boundaries instead of adding a one-sided native-command path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require explicit channel authorization when access groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For `channel_post`, this branch sets `commandAuthorizers` to `[]` when `useAccessGroups` is false. The shared command gate treats an empty configured-mode authorizer list as allowed, so `/...` in any Telegram channel the bot can read can bypass the PR's stated explicitly-allowed-chat boundary unless the operator also configured `commands.allowFrom`.
  - confidence: 0.88
- **[P1] Normalize channel_post sender before command auth:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Reading raw `ctx.msg` sends channel posts into `resolveTelegramCommandAuth` without the `sender_chat` to `from` normalization used by the regular `channel_post` pipeline. Deployments that authorize commands by `commands.allowFrom` or owner identity therefore have no sender candidate for the posting channel and will reject or evaluate channel commands differently from normal channel_post messages.
  - confidence: 0.86
- **[P1] Route channel command callbacks through channel state:** `extensions/telegram/src/bot-native-commands.ts:341`
  - body: Once this authorizer allows channel_post native commands, commands that render inline buttons, such as `/models`, become reachable from channels. The callback_query handler still treats `chat.type === "channel"` as non-group/direct, so button clicks can be dropped by DM authorization or update the clicker's direct session instead of the channel session.
  - confidence: 0.83
