## Summary

Keep open: the PR addresses a real source-level Telegram channel_post command gap, but the branch is currently conflicting and the patch still has authorization, callback/session, and live-proof blockers on a command-execution surface.

## What This Changes

The PR changes Telegram native and plugin slash-command handlers to read grammY `ctx.msg`, treat `channel_post` messages as group-like for command authorization, and adds channel-post auth/key tests.

## Best Possible Solution

Rework this on current main so channel_post commands use `ctx.msg`, preserve explicit channel and sender_chat authorization semantics, handle or block interactive callbacks deliberately, and include focused tests plus redacted live Telegram proof.

## Reproduction Assessment

Yes at source level: grammY command handlers match channel posts and expose the message through `ctx.msg`, while current main's handlers read `ctx.message` and return early. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No: using `ctx.msg` is the right dependency seam, but this PR is not yet the best fix because it leaves explicit-channel authorization, sender_chat allowlists, callback/session routing, branch conflicts, and real Telegram proof unresolved.

  - confidence: 0.88

The existing callback handler still treats `chat.type === "channel"` as non-group/direct, so follow-up clicks can be dropped by DM authorization or mutate the wrong session; either route channel callbacks through channel state or block interactive channel commands.
