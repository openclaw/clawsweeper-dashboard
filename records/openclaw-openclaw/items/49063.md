Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use sender_chat for channel-post command identity:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: Switching command handlers to `ctx.msg` lets raw channel_post updates enter this auth path, but `resolveTelegramCommandAuth` still derives the sender only from `msg.from`. Channel posts commonly identify the posting channel via `sender_chat`, and `commands.allowFrom`/owner checks use numeric sender candidates, so configured channel commands can be rejected or authorized under the wrong identity unless this path synthesizes sender_chat like the normal channel_post pipeline.
  - confidence: 0.9
- **[P1] Keep channel callbacks on the channel session path:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: Enabling channel_post native commands also enables commands such as `/models` that send inline buttons, but the callback handler still classifies `chat.type === "channel"` as non-group and resolves authorization/session state from the clicking user. Those follow-ups can be dropped or mutate the user's direct session instead of the channel command session.
  - confidence: 0.86
- **[P2] Honor per-channel groupPolicy before allowing commands:** `extensions/telegram/src/bot-native-commands.ts:341`
  - body: The new channel_post authorizer treats any explicit `groupConfig` as allowed after `policyAccess`, but this auth call still uses `useTopicAndGroupOverrides: false`, so a channel entry with its own `groupPolicy: disabled` or narrower allowlist can pass solely because the chat is configured. Channel command authorization should respect the same per-chat policy decision as the normal inbound path.
  - confidence: 0.82
