## Summary

Keep open: current main still lacks native Telegram channel-post command handling, but this branch is not safe to merge as-is because it broadens command authorization in some channel cases, leaves callback/session policy gaps, and still lacks required real Telegram proof.

## What This Changes

The PR changes Telegram native/plugin slash command handlers to read `ctx.msg`, treat `channel_post` as group-like for authorization/keying, and adds channel-post auth and sequential-key tests.

## Best Possible Solution

Define the Telegram `channel_post` authorization rule with https://github.com/openclaw/openclaw/issues/49062, then refresh the PR to reuse channel-post normalization, honor policy and callback routing, add focused tests, and include live Telegram proof.

## Reproduction Assessment

Yes, source-level: current main native/plugin command handlers read `ctx.message` and return when it is absent, while channel-post ingress is handled separately. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Using `ctx.msg` is the right dependency direction, but this branch needs to be rebuilt around current channel-post normalization plus a maintainer-approved auth and callback policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list. The shared configured-mode gate treats that as allowed, so any channel the bot can read can run native/plugin commands even though the change is scoped to explicitly allowed chats.
  - confidence: 0.91
- **[P2] Use sender_chat when checking channel command allowlists:** `extensions/telegram/src/bot-native-commands.ts:351-352`
  - body: When `commands.allowFrom` is configured, the channel-post path relies on `commandsAllowFromAccess`, but the auth context still derives `SenderId` only from `msg.from`. Telegram channel posts normally identify the posting channel via `sender_chat`, so explicitly allowed channel commands can be rejected once a command allowlist is configured.
  - confidence: 0.87
- **[P2] Honor per-channel policy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298-299`
  - body: The channel command path evaluates group policy with `useTopicAndGroupOverrides: false` and then treats any explicit `groupConfig` as enough to authorize the command. A channel entry with a disabled or narrower policy can therefore still execute native/plugin commands.
  - confidence: 0.84
- **[P2] Route channel command callbacks through channel policy:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Switching command handlers to `ctx.msg` makes interactive native commands such as `/models` reachable from channels, but callback handling still classifies `chat.type === "channel"` as non-group and uses DM-style authorization/session routing. Button follow-ups can be dropped or applied to the wrong session after the initial channel command succeeds.
  - confidence: 0.82
