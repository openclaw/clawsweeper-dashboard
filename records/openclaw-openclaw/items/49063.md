## Summary

Keep open: the PR targets a real Telegram channel_post command gap, but it is not merge-ready because the branch is conflicting, lacks inspectable live Telegram proof, and still widens channel command execution before sender and callback/session policy are settled.

## What This Changes

The PR switches Telegram native/plugin command handlers to use grammY ctx.msg for channel_post updates, treats channel posts as group-like during command authorization, adds channel_post auth tests, and prefers ctx.msg in update/sequential keys.

## Best Possible Solution

Rebase, then implement one Telegram channel_post command policy that normalizes sender_chat identity, preserves explicit chat and per-sender authorization semantics, routes callbacks through the channel session, and includes focused tests plus redacted live Telegram proof.

## Reproduction Assessment

Yes, at source level: grammY bot.command detects channel posts and current main still reads ctx.message, so channel_post commands do not reach the handlers. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Using ctx.msg is the right dependency primitive, but this PR is too narrow because channel_post sender identity and callback/session routing remain unresolved before the command surface is enabled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Require explicit channel auth when access groups are off:** `extensions/telegram/src/bot-native-commands.ts:329-341`
  - body: For channel posts with useAccessGroups disabled and commands.allowFrom unset, this branch passes an empty authorizer list into the configured fallback. That helper treats no configured authorizers as allow, so any channel the bot can read can run native/plugin commands even though the PR is scoped to explicitly allowed chats.
  - confidence: 0.87
- **[P1] Populate channel sender identity for command allowlists:** `extensions/telegram/src/bot-native-commands.ts:238-243`
  - body: The PR lets channel_post updates reach command auth, but the auth context still has no SenderId/SenderUsername when Telegram omits msg.from for channel posts. Because command sender authorization only falls back to From for direct chats, commands.allowFrom and owner-style command allowlists cannot match the intended sender_chat identity.
  - confidence: 0.86
- **[P1] Keep channel callbacks on the channel session:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: Enabling channel_post commands also enables commands such as /models that emit inline keyboards, but the existing callback_query handler still treats chat.type === "channel" as non-group and authorizes/routes via the clicking user's direct context. Those follow-ups can be dropped or update the wrong session after the initial channel command succeeds.
  - confidence: 0.82
