## Summary

Keep open. Current main still does not run Telegram native command handlers from `channel_post` updates, so the central bug remains, but this PR is not merge-ready because it creates a separate channel-command authorization path that does not follow current main's normalized `channel_post` sender/auth/session flow.

## What This Changes

The PR changes Telegram native and plugin slash-command handlers to read `ctx.msg` and authorizes Telegram `channel_post` commands from explicitly allowed channel chats.

## Best Possible Solution

Rework this on current main so native `channel_post` slash commands reuse the existing normalized channel-post sender, group allowlist, ingress authorization, and callback/session routing behavior, then prove the allowed and denied Telegram flows in a real Telegram setup.

## Reproduction Assessment

Yes. Source inspection shows current main's native and plugin command handlers read `ctx.message`, while Telegram `channel_post` command payloads are exposed via `ctx.channelPost`/`ctx.msg`; the linked reports include the same observed failure mode.

## Solution Assessment

No. The PR fixes the obvious `ctx.message` gap, but it is not the best current-main shape because it bypasses the existing normalized `channel_post` ingress authorization/session path and leaves callback routing inconsistent.
