## Summary

Keep open, but this PR is not merge-ready: the direction to use grammY `ctx.msg` is valid, while the branch still broadens Telegram channel command authorization, leaves callback/session behavior inconsistent, and lacks required real Telegram proof.

## What This Changes

The PR changes Telegram native/plugin command handlers to read `ctx.msg`, treats `channel_post` as group-like for command authorization and ordering keys, and adds channel-post auth/update-key/sequential-key tests.

## Best Possible Solution

Rebuild this on the current Telegram ingress/auth helpers so channel commands require explicit channel policy, preserve `sender_chat` and callback/session routing, document the chosen policy, and include live allowed/denied Telegram proof.

## Reproduction Assessment

Yes, source-level: current main's native/plugin command handlers read `ctx.message`, while grammY 1.42.0 exposes `ctx.msg` as the cross-surface accessor including `channelPost`. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Using `ctx.msg` is the right dependency direction, but this branch should be reworked to reuse the current channel_post ingress/auth model and settle the auth policy tracked in https://github.com/openclaw/openclaw/issues/49062.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list. The configured-mode gate treats that as allowed, so any channel the bot can read can run native/plugin commands even though the stated boundary is explicitly allowed chats.
  - confidence: 0.9
- **[P2] Populate channel sender identity before allowFrom checks:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: Channel posts normally identify the actor through `sender_chat`, but this auth context still derives `SenderId` and `SenderUsername` only from `msg.from`. Deployments using `commands.allowFrom.telegram` can therefore reject the intended allowed channel identity once this path starts accepting channel posts.
  - confidence: 0.87
- **[P2] Honor per-channel policy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298`
  - body: The channel command path evaluates group policy with `useTopicAndGroupOverrides: false`, then treats any explicit `groupConfig` as enough to authorize the channel. A channel entry with `groupPolicy: "disabled"` or a narrower override can therefore execute commands even though regular channel_post ingress honors those overrides.
  - confidence: 0.82
- **[P2] Route channel button callbacks through channel policy:** `extensions/telegram/src/bot-native-commands.ts:341`
  - body: Allowing initial channel commands makes interactive native commands reachable from channels, but callback handling still classifies `chat.type === "channel"` as non-group and uses DM authorization/session routing. Button follow-ups can be dropped or applied to the wrong session after the initial channel command succeeds.
  - confidence: 0.82
