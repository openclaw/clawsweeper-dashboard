## Summary

Keep open: current main still has the reported Telegram channel-post native-command gap, but this PR is not mergeable as-is because it expands Telegram command authorization in ways that can bypass or misapply existing channel policy and it lacks real Telegram proof.

## What This Changes

The PR changes Telegram native and plugin slash-command handlers to read grammY `ctx.msg` for `channel_post` updates, adds channel-post command authorization logic, and extends Telegram update/sequential-key tests.

## Best Possible Solution

Refresh the Telegram fix around the current channel-post normalization and authorization model, define the remaining policy with https://github.com/openclaw/openclaw/issues/49062, cover allowed/denied/callback cases, and include redacted live Telegram proof.

## Reproduction Assessment

Yes, source-level: current main's native and plugin Telegram command handlers read `ctx.message`, while grammY documents `ctx.msg` as covering `message`, `channelPost`, `editedMessage`, and related message surfaces. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Using `ctx.msg` is the right dependency direction, but this branch should preserve the existing explicit channel authorization, sender normalization, policy override, and callback routing boundaries before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list. The shared configured-mode gate treats that as allowed, so any channel the bot can read can run native/plugin slash commands even though the PR is scoped to explicitly allowed chats.
  - confidence: 0.91
- **[P2] Populate channel sender before command allowlists:** `extensions/telegram/src/bot-native-commands.ts:351-352`
  - body: This now routes channel posts through `commands.allowFrom`, but the auth context still derives `SenderId` only from `msg.from`. Telegram channel posts normally identify the posting channel in `sender_chat`, and group contexts do not fall back from `From`, so an explicitly allowlisted channel can be rejected once `commands.allowFrom.telegram` is configured.
  - confidence: 0.87
- **[P2] Honor per-channel policy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298-299`
  - body: The channel-post path evaluates group policy with `useTopicAndGroupOverrides: false` and then authorizes on `groupConfig` presence. A channel entry whose own `groupPolicy` is `disabled` or narrowed can still execute newly enabled slash commands.
  - confidence: 0.84
- **[P2] Route channel command callbacks through channel policy:** `extensions/telegram/src/bot-native-commands.ts:341`
  - body: The initial channel command can now expose interactive native commands such as `/models`, but callback handling still classifies `chat.type === "channel"` as non-group and uses DM-style authorization/session routing. Button follow-ups can be dropped or applied to the wrong session after the channel command succeeds.
  - confidence: 0.82

## Security Review

Status: needs_attention

Concerns:

- **[medium] Channel commands can bypass explicit channel configuration:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: When access groups are off and `commands.allowFrom` is unset, the channel-post branch passes no authorizer and the shared configured-mode gate allows the command, expanding command execution beyond explicitly configured Telegram channels.
  - confidence: 0.91
- **[medium] Disabled per-channel policy can be ignored:** `extensions/telegram/src/bot-native-commands.ts:298`
  - body: The branch authorizes explicit channel entries from `groupConfig` while policy evaluation ignores group/topic overrides, so a per-channel disabled or narrowed policy can be bypassed for newly enabled native/plugin commands.
  - confidence: 0.84
- **[medium] Channel command allowlists can use the wrong sender identity:** `extensions/telegram/src/bot-native-commands.ts:351`
  - body: The command allowlist path reads `SenderId` from `msg.from` instead of the channel `sender_chat`, so operators configuring `commands.allowFrom.telegram` can get denied channel commands or misleading authorization semantics.
  - confidence: 0.87
