## Summary

Keep open: the PR targets a real Telegram channel_post native-command gap on current main, but the proposed branch still has authorization and callback-routing defects plus missing real Telegram proof before merge.

## What This Changes

The PR switches Telegram native command handling to grammY ctx.msg for channel_post contexts, treats channel posts as group-like for auth/routing/order keys, and adds channel-post auth/key tests.

## Best Possible Solution

Refresh the fix around current channel_post normalization and a maintainer-approved auth policy, with allowed/denied/callback coverage and live Telegram proof before merge.

## Reproduction Assessment

Yes, source-level: current main reads ctx.message in native command handlers while grammY command contexts can carry channelPost through ctx.msg. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Using ctx.msg is the right dependency direction, but this branch needs channel-specific authorization policy and callback routing fixed before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list. The shared `modeWhenAccessGroupsOff: "configured"` gate treats that as allowed, so any channel the bot can read can run native/plugin commands even though the PR is scoped to explicitly allowed chats.
  - confidence: 0.91
- **[P2] Populate channel-post sender before commands.allowFrom:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: When `commands.allowFrom` is configured, channel-post commands depend on `SenderId`, but this path still derives it only from `msg.from`. Telegram channel posts normally identify the posting channel via `sender_chat`, so explicitly allowed channel identities can be rejected once a command allowlist is configured.
  - confidence: 0.87
- **[P2] Honor per-channel policy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298-299`
  - body: The channel command path evaluates group policy with `useTopicAndGroupOverrides: false` and then treats any explicit `groupConfig` as enough to authorize the command. A channel entry with `groupPolicy: "disabled"` or a narrowed policy can therefore still execute native/plugin commands.
  - confidence: 0.84
- **[P2] Route channel command callbacks through channel sessions:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Switching command handlers to `ctx.msg` makes interactive native commands such as `/models` reachable from channels, but callback handling still classifies `chat.type === "channel"` as non-group. Button follow-ups can be dropped or applied to the wrong session after the initial channel command succeeds.
  - confidence: 0.82

## Security Review

Status: needs_attention

Concerns:

- **[medium] Channel commands can bypass explicit channel configuration:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: When access groups are off and `commands.allowFrom` is unset, the channel-post branch passes no authorizer and the shared configured-mode gate allows the command, expanding command execution beyond explicitly configured Telegram channels.
  - confidence: 0.91

  - body: The branch authorizes explicit channel entries from `groupConfig` while policy evaluation ignores group/topic overrides, so a per-channel disabled or narrowed policy can be bypassed for newly enabled native/plugin commands.
  - confidence: 0.84
- **[medium] Channel sender allowlists can be misapplied:** `extensions/telegram/src/bot-native-commands.ts:227`
  - body: The command path does not synthesize `sender_chat` into the sender identity before evaluating `commands.allowFrom`, so channel allowlists can reject intended channels or encourage unsafe fallback policy changes.
  - confidence: 0.87
