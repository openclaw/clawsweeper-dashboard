## Summary

Keep open: the PR addresses a real Telegram channel-command gap, but the branch still has authorization regressions and lacks required after-fix real Telegram proof for this visible auth change.

## What This Changes

The PR changes Telegram native and plugin command handlers to read `ctx.msg`, treats `channel_post` updates as group-like command contexts, and adds channel-post auth/update-key/sequential-key tests.

## Best Possible Solution

Rebuild the change on current Telegram ingress/auth helpers, preserve explicit channel policy plus `sender_chat` and allowlist semantics, add focused auth coverage, and attach live Telegram proof.

## Reproduction Assessment

Yes, source-level: current main command handlers read `ctx.message`, while grammY’s pinned `ctx.msg` includes `channel_post`; I did not run a live Telegram channel session.

## Solution Assessment

No. Using `ctx.msg` is the right dependency direction, but this branch bypasses or fails important authorization paths and needs to be rebuilt around the current channel ingress/auth model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts, `useAccessGroups=false` leaves `commandAuthorizers` empty, and the configured-mode resolver treats that as allowed. That lets `/...` commands run from channels that were not explicitly allowed, which contradicts the PR’s stated boundary.
  - confidence: 0.9
- **[P2] Populate channel senders before commands.allowFrom checks:** `extensions/telegram/src/bot-native-commands.ts:351-352`
  - body: When `commands.allowFrom` is configured, auth depends on `commandsAllowFromAccess`, but the context still derives sender identity only from `msg.from`. Telegram channel posts normally expose the channel identity through `sender_chat`, so allowed channel posts are rejected instead of matching the configured sender.
  - confidence: 0.87
- **[P2] Route channel command callbacks through channel auth:** `extensions/telegram/src/bot-native-commands.ts:341`
  - body: This enables channel commands that can emit inline keyboards, but callback handling still treats `chat.type === "channel"` as non-group and uses DM authorization/session routing. Interactive commands such as model selection can therefore drop button clicks or mutate the wrong session after the initial channel command succeeds.
  - confidence: 0.8
