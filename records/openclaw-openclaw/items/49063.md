## Summary

Keep open: current main and v2026.6.8 still do not allow Telegram native/plugin commands through `channel_post`, but this PR is not merge-ready because the new channel command path misses `sender_chat` authorization, leaves interactive callback/session behavior unproven, and has no real Telegram proof.

## What This Changes

The PR changes Telegram native and plugin command handling to read `ctx.msg`, treat `channel_post` messages as group-like for command auth, and add tests for allowed and denied channel-post command cases.

## Best Possible Solution

Make one Telegram-owned channel_post command path that reuses the existing channel_post sender normalization and policy checks, routes interactive callbacks on the channel session, documents the chosen sender-scoped allowFrom rule, and includes real Telegram proof.

## Reproduction Assessment

Partly: source inspection shows a channel_post without `msg.from` cannot satisfy configured Telegram command allowlists through `sender_chat`, and the current callback path still has a channel/session gap. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No; this is not yet the best fix because it enables channel_post commands without carrying the existing channel_post sender and callback/session invariants through the whole path. The safer direction is to reuse or mirror the current channel_post ingress authorization model and prove the complete flow live.
