## Summary

Keep open: the underlying Telegram channel_post command bug is real, but this stale conflicting branch still has authorization and callback-routing defects, and the external PR has no after-fix live Telegram proof.

## What This Changes

The PR changes Telegram native command handling to read grammY `ctx.msg` for channel posts, treats channel posts as group-like for command authorization/keying, and adds channel-post auth and sequential-key tests.

## Best Possible Solution

Refresh the fix onto current main using the existing channel_post normalization and shared Telegram ingress authorization, settle the sender-scoped channel_post policy tracked at https://github.com/openclaw/openclaw/issues/49062, add allowed/denied/callback coverage, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, source-level: current main uses `ctx.message` in native command handlers while grammY `ctx.msg` includes `channelPost`, so a channel_post command can be consumed before the channel_post handler processes it. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Reading `ctx.msg` is the right dependency direction, but this branch needs a current-main refresh, a maintainer-approved channel_post authorization policy, sender_chat normalization, callback routing, and live proof before it is the maintainable fix.
