## Summary

Keep open: the goal is useful, but this PR is not merge-ready because it is conflicting with current main, lacks real Telegram proof, and introduces concrete Telegram authorization and callback/session risks.

## What This Changes

The PR changes Telegram native and plugin command handlers to read grammY `ctx.msg`, adds channel-post command authorization logic, and adds focused Telegram channel-post auth/update-key tests.

## Best Possible Solution

Land a refreshed implementation that normalizes `channel_post` native commands through the same Telegram ingress authorization/session path as regular channel posts, then prove allowed, denied, `commands.allowFrom`, and interactive callback behavior in real Telegram.

## Reproduction Assessment

Yes, source-level. Current main's Telegram native command handlers read `ctx.message`, while grammY command middleware can match channel posts where `ctx.message` is undefined and `ctx.msg` carries the message; I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. The direction is useful, but the current branch should be rebased onto current main and fixed to reuse shared Telegram channel-post authorization, sender normalization, callback/session routing, and live proof before merge.

Overall confidence: 0.91

Full review comments:

- **[P1] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:333-356`
  - body: For `channel_post` with `useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list into configured-mode command gating. That returns allow when nothing is configured, so any Telegram channel where the bot sees `/...` can execute commands, which is broader than the PR's explicit-chat boundary.
  - confidence: 0.92
- **[P2] Populate sender_chat before command allowlists:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Switching command handlers to `ctx.msg` lets raw `channelPost` messages reach auth, but `resolveTelegramCommandAuth` still takes the sender only from `msg.from`. Telegram channel posts usually identify the posting channel through `sender_chat`, so `commands.allowFrom.telegram` cannot authorize intended channel identities.
  - confidence: 0.88
- **[P2] Honor per-channel policy for channel commands:** `extensions/telegram/src/bot-native-commands.ts:341`
  - body: The new channel-post authorizer treats any explicit `groupConfig` as enough to authorize the command and the policy check does not apply topic/group overrides. A configured channel with `groupPolicy: disabled` or a narrowed per-channel policy can still execute native commands.
  - confidence: 0.84
- **[P1] Route channel callbacks through the channel session:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: After this line enables native channel commands, commands that emit inline keyboards can be used from channels, but the callback handler still treats `chat.type === "channel"` as a direct chat. `/models`-style callbacks can then be rejected by DM rules or update the clicker's direct session instead of the channel session.
  - confidence: 0.86
