## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require explicit channel authorization for channel posts:** `extensions/telegram/src/bot-native-commands.ts:333-344`
  - body: When `isChannelPost` is true and `useAccessGroups` is false, this branch passes an empty authorizer list into `resolveCommandAuthorizedFromAuthorizers(..., "configured")`, which returns allow when nothing is configured. That makes `/...` commands executable from any Telegram channel the bot can read unless `commands.allowFrom` happens to be set, contradicting the explicit-allowed-chat boundary.
  - confidence: 0.9
- **[P1] Normalize sender_chat before command allowlist checks:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Switching command handlers to raw `ctx.msg` makes channel posts enter auth with no synthesized `from`, while the auth code still derives sender identity from `msg.from`. Channel posts usually carry the posting channel in `sender_chat`, so `commands.allowFrom.telegram` and owner-style command allowlists can reject the intended channel identity instead of checking it.
  - confidence: 0.87
- **[P1] Route channel command callbacks through channel state:** `extensions/telegram/src/bot-native-commands.ts:860`
  - body: Enabling channel_post commands also enables native/plugin commands that render inline buttons, but the callback path still treats `chat.type === "channel"` as non-group and resolves authorization/session state through direct-message logic. A channel `/models` follow-up can be dropped or mutate the clicking admin's direct session instead of the channel session.
  - confidence: 0.84
