## Summary

Keep open: the PR targets a real Telegram channel-command gap, but the current patch still widens a command execution surface without explicit-channel parity, leaves policy questions unresolved, and lacks required real Telegram proof.

## What This Changes

The PR changes Telegram native and plugin command handlers to read grammY `ctx.msg`, treats `channel_post` as group-like for command authorization and ordering, and adds channel-post command auth/key tests.

## Best Possible Solution

Rebuild the change on current Telegram ingress/auth helpers so channel commands require explicit channel policy, preserve `sender_chat` and callback/session routing, document the chosen policy, add focused regression coverage, and attach live allowed/denied Telegram proof.

## Reproduction Assessment

Yes, source-level: current main's native and plugin command handlers read `ctx.message` and return for `channel_post`, while grammY 1.42.0 exposes `ctx.msg` for `channelPost`. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Using `ctx.msg` is the right dependency direction, but this branch should be rebuilt around the current `channel_post` ingress/auth model and the policy decision tracked in https://github.com/openclaw/openclaw/issues/49062.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list. The configured-mode gate treats that as allowed, so any channel the bot can read can run native/plugin commands even though current `channel_post` ingress requires a configured channel.
  - confidence: 0.9
- **[P2] Do not drop explicit channel auth under commands.allowFrom:** `extensions/telegram/src/bot-native-commands.ts:351-352`
  - body: When `commands.allowFrom` is configured, the channel-post path ignores the explicit channel authorization result and checks only `SenderId` from `msg.from`. Telegram channel posts normally expose the sender through `sender_chat`, so explicitly configured channels can still be rejected once an operator has any command allowlist configured.
  - confidence: 0.87
- **[P2] Honor per-channel policy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298-299`
  - body: The channel command path evaluates group policy with `useTopicAndGroupOverrides: false`, then treats any explicit `groupConfig` as enough to authorize the channel. A channel entry with `groupPolicy: "disabled"` or a narrower override can therefore execute commands even though regular `channel_post` ingress honors those overrides.
  - confidence: 0.82
- **[P2] Route channel button callbacks through channel policy:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Switching command handlers to `ctx.msg` makes interactive native commands such as `/models` reachable from channels, but callback handling still classifies `chat.type === "channel"` as non-group and uses DM authorization/session routing. Button follow-ups can be dropped or applied to the wrong session after the initial channel command succeeds.
  - confidence: 0.82
