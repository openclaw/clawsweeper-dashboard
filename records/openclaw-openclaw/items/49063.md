## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor per-channel policy before allowing channel posts:** `extensions/telegram/src/bot-native-commands.ts:341`
  - body: This channel-post authorizer treats any explicit `groupConfig` as enough to run commands, but the preceding policy check is evaluated without per-group/topic overrides. A channel configured with `groups[chatId].groupPolicy: "disabled"` or a narrowed policy can still pass here, unlike the normal channel_post ingress path that applies those overrides before dispatch.
  - confidence: 0.88
- **[P1] Use sender_chat for channel command allowlists:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: After switching the handler to `ctx.msg`, channel posts usually have no `msg.from`, so `commands.allowFrom` and owner-style command allowlists see an empty sender even when Telegram provides the channel identity in `sender_chat`. Current channel_post ingress already synthesizes `from` from `sender_chat`; this path should do the same or it will reject intended channels and make allowlist behavior inconsistent.
  - confidence: 0.86
- **[P1] Route channel command callbacks through channel sessions:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Enabling `ctx.msg` lets channel `/models` and other interactive native commands send inline keyboards, but the callback handler still classifies `chat.type === "channel"` as non-group and resolves direct-style authorization/session state. Those button follow-ups can be dropped or mutate a user's DM session instead of the channel session, so channel callbacks need the same channel context before enabling interactive commands.
  - confidence: 0.84
