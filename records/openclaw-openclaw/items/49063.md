## Summary

Keep open: current main still has the reported Telegram native-command gap, but this PR is not safe to merge as-is because it broadens channel command execution without preserving the current channel_post auth/callback model and still lacks real Telegram proof.

## What This Changes

The PR updates Telegram native and plugin command handlers to use grammY `ctx.msg` for `channel_post` commands, treats channel posts as group-like for authorization/order keys, and adds channel-post auth/key tests.

## Best Possible Solution

Rework the PR around the current Telegram ingress/auth helpers so channel commands require explicit channel policy, preserve `sender_chat` and callback/session routing, document the chosen policy, add regression coverage, and attach live allowed/denied Telegram proof.

## Reproduction Assessment

Yes, source-level: current main's native/plugin command handlers read `ctx.message` and return when absent, while grammY 1.42.0 exposes `ctx.msg` for `channelPost`. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Switching to `ctx.msg` is the right dependency direction, but this branch should be rebuilt on the current `channel_post` ingress/auth model and the remaining policy decision in https://github.com/openclaw/openclaw/issues/49062.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list. The configured-mode gate treats that as allowed, so any channel the bot can read can run native/plugin commands even though the stated boundary is explicitly allowed chats.
  - confidence: 0.9
- **[P2] Populate channel sender identity before allowFrom checks:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: Channel posts normally identify the actor through `sender_chat`, but this auth context still derives `SenderId` and `SenderUsername` only from `msg.from`. Deployments using `commands.allowFrom.telegram` can therefore reject the intended allowed channel identity once this path starts accepting channel posts.
  - confidence: 0.87
- **[P2] Honor per-channel policy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298`
  - body: The channel command path evaluates group policy with `useTopicAndGroupOverrides: false`, then treats any explicit `groupConfig` as enough to authorize the channel. A channel entry with `groupPolicy: "disabled"` or a narrower override can therefore execute commands even though regular `channel_post` ingress honors those overrides.
  - confidence: 0.82
- **[P2] Route channel button callbacks through channel policy:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Changing native handlers to accept `ctx.msg` makes interactive commands such as argument menus reachable from channels, but callback handling on current main still classifies `chat.type === "channel"` as non-group and uses DM authorization/session routing. Button follow-ups can be dropped or applied to the wrong session after the initial channel command succeeds.
  - confidence: 0.82
