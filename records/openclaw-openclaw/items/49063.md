## Summary

Keep open. The PR targets a real Telegram channel_post native-command gap, but the current diff still has blocking authorization problems on a command execution surface and lacks required real Telegram proof.

## What This Changes

The PR changes Telegram native/plugin command handlers to read channel_post via ctx.msg, treats channel posts as group-like for auth/routing, adds ctx.msg update/sequential keys, and adds regression tests.

## Best Possible Solution

Route channel_post native commands through the same explicit configured-channel gate and sender_chat normalization used by the inbound channel_post path, define the commands.allowFrom/group allowFrom rule, cover allow/deny/callback cases, and attach real Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main reads ctx.message in Telegram native command handlers and returns before auth when a channel_post has no message, while the existing channel_post inbound path shows the expected configured-channel sender normalization pattern.

## Solution Assessment

No. Reading ctx.msg is a plausible direction, but this diff is not the best final solution because it opens channel commands when access groups are off and does not normalize sender_chat before commands.allowFrom checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Require explicit auth when access groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `useAccessGroups=false`, this branch passes an empty authorizer list into `resolveCommandAuthorizedFromAuthorizers(..., modeWhenAccessGroupsOff: "configured")`, which treats no configured authorizers as allow. That lets any channel the bot can read execute native/plugin commands without the explicit configured-channel gate used by the normal `channel_post` path.
  - confidence: 0.91
- **[P2] Populate sender_chat before commands.allowFrom checks:** `extensions/telegram/src/bot-native-commands.ts:351-352`
  - body: When `commands.allowFrom` is configured, `commandAuthorized` now depends solely on `commandsAllowFromAccess`, but channel posts are passed as raw `ctx.msg` and `senderId` is still derived from `msg.from`. Telegram channel posts usually identify the posting channel in `sender_chat`, so explicit channel IDs in `commands.allowFrom.telegram` are not considered and otherwise allowed channel commands are rejected.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff changes Telegram command authorization and currently widens channel_post command execution in the access-groups-off path.

Concerns:

- **[high] Channel commands can bypass explicit channel authorization:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: The PR's channel_post branch supplies no authorizers when `useAccessGroups` is false, and the command gate treats that as allow; this is broader than the explicit configured-channel boundary used by the existing channel_post message pipeline.
  - confidence: 0.9
- **[medium] commands.allowFrom cannot match sender_chat channel identity:** `extensions/telegram/src/bot-native-commands.ts:351`
  - body: Channel posts commonly carry their identity in `sender_chat`, but the PR keeps command authorization tied to `msg.from`, so configured command allowlists can fail closed or behave inconsistently for channel posts.
  - confidence: 0.85
