## Summary

Keep open: this PR addresses a real source-level Telegram channel_post/native-command gap, but the branch still has authorization and callback correctness blockers and lacks the required real Telegram proof.

## What This Changes

The PR changes Telegram native and plugin command handling to read grammY `ctx.msg` for `channel_post`, treats channel posts as group-like for command auth/routing, and adds channel-post auth/key tests.

## Best Possible Solution

Implement `ctx.msg`-based channel command support only with a maintainer-approved `channel_post` auth policy, shared sender normalization, callback/session routing coverage, focused regression tests, and live Telegram proof.

## Reproduction Assessment

Yes, source-level: current main's grammY command handlers read `ctx.message` and return when it is absent, while grammY 1.42.0 detects commands in channel posts and documents `ctx.msg` for that surface. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. `ctx.msg` is the right dependency direction, but this branch needs channel-specific authorization policy, sender normalization, callback routing fixes, and real Telegram proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization with access groups off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list into configured-mode gating. That path allows when nothing is configured, so any channel the bot can read can execute commands even though the PR is scoped to explicitly allowed chats.
  - confidence: 0.91
- **[P2] Populate channel-post sender identities before allowlists:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: When `commands.allowFrom` is configured, channel-post authorization depends on `SenderId`, but this path still reads it only from `msg.from`. Telegram channel posts normally identify the posting channel via `sender_chat`, so explicitly allowed channel identities can be rejected or pushed toward broader policy.
  - confidence: 0.87
- **[P2] Honor per-channel groupPolicy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298-299`
  - body: The channel command path evaluates group policy with topic/group overrides disabled and then treats any `groupConfig` as enough to authorize the command. A channel entry with `groupPolicy: "disabled"` or a narrowed policy can therefore still execute native/plugin commands.
  - confidence: 0.84
- **[P2] Route channel callback follow-ups through channel sessions:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Using `ctx.msg` makes interactive commands such as `/models` reachable from channel posts, but callback handling still classifies `chat.type === "channel"` as non-group and uses the clicking user/DM path. Button follow-ups can be dropped or mutate the wrong session after the initial command succeeds.
  - confidence: 0.82
