## Summary

Keep open: the PR targets a real source-reproducible Telegram channel_post command gap, but the branch is conflicting with current main, lacks required real Telegram proof, and still mishandles configured command allowlists for channel posts.

## What This Changes

The PR switches Telegram native/plugin command handlers to `ctx.msg`, treats channel posts as group-like command contexts, and adds channel-post auth/update-key tests.

## Best Possible Solution

Rebase onto current main's centralized Telegram ingress/auth helpers, preserve explicit channel allow while honoring configured command allowlists, and add focused tests plus real Telegram channel proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: grammY command handlers can receive `channel_post`, but current main reads `ctx.message`, so a channel-post command has no message and exits before authorization. I did not run a live Telegram session in this read-only review.

## Solution Assessment

No. The direction is plausible, but the current branch is outdated/conflicting and should be rebuilt on current Telegram ingress auth while fixing the `commands.allowFrom` channel identity path.
