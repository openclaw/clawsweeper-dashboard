## Summary

Keep open: current main still has the Telegram native command gap for channel posts, but this PR is not merge-ready because the branch broadens channel command authorization and callback behavior without the required live Telegram proof.

## What This Changes

The PR switches Telegram native and plugin slash command handlers to `ctx.msg`, treats `channel_post` messages as group-like command contexts, and adds channel-post auth/key tests.

## Best Possible Solution

Adopt `ctx.msg` channel-post command handling only after a maintainer-approved Telegram authorization policy, callback/session routing coverage, focused tests, and live Telegram proof are in place.

## Reproduction Assessment

Yes, source-level: current main reads `ctx.message` in native command handlers while grammY command contexts include channel posts through `ctx.msg`. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Moving to `ctx.msg` is the right dependency direction, but this branch needs explicit channel authorization, sender identity, per-channel policy, and callback routing fixes before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list. The shared configured-mode gate treats that as allowed, so any channel the bot can read can run native/plugin commands even though the PR is scoped to explicitly allowed chats.
  - confidence: 0.91
- **[P2] Use sender_chat for channel-post command auth:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: When `commands.allowFrom` is configured, channel-post commands depend on sender candidates, but this path still derives them only from `msg.from`. Telegram channel posts normally identify the posting channel via `sender_chat`, so intended channel identities can be rejected or policy can be loosened to compensate.
  - confidence: 0.87
- **[P2] Honor per-channel policy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298-299`
  - body: The channel command path evaluates group policy with topic/group overrides disabled and then treats any explicit `groupConfig` as enough to authorize the command. A channel entry with `groupPolicy: "disabled"` or a narrowed policy can therefore still execute native/plugin commands.
  - confidence: 0.84
- **[P2] Route channel command callbacks through channel sessions:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Switching command handlers to `ctx.msg` makes interactive native commands such as `/models` reachable from channels, but callback handling still classifies `chat.type === "channel"` as non-group. Button follow-ups can be dropped or applied to the clicking admin's direct session after the initial channel command succeeds.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff expands Telegram command execution from channel posts and still has concrete authorization-policy defects.

Concerns:

- **[medium] Channel commands can bypass explicit channel configuration:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: When access groups are off and `commands.allowFrom` is unset, the channel-post branch passes no authorizer and the configured-mode gate allows the command, expanding execution beyond explicitly configured Telegram channels.
  - confidence: 0.91
- **[medium] Disabled per-channel policy can be ignored:** `extensions/telegram/src/bot-native-commands.ts:298`
  - body: The branch authorizes explicit channel entries from `groupConfig` while policy evaluation ignores group/topic overrides, so a per-channel disabled or narrowed policy can be bypassed for newly enabled native/plugin commands.
  - confidence: 0.84
- **[medium] Channel sender allowlists can be misapplied:** `extensions/telegram/src/bot-native-commands.ts:227`
  - body: The command path does not synthesize `sender_chat` into the sender identity before evaluating command allowlists, so channel allowlists can reject intended channels or encourage unsafe fallback policy changes.
  - confidence: 0.87
