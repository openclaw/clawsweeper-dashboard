## Summary

Keep open. The PR is aimed at a real Telegram channel-post gap, but the current patch changes an authorization surface without preserving the existing sender and callback semantics for channel posts, and it still lacks required real Telegram proof.

## What This Changes

The PR changes Telegram native command handlers to read `ctx.msg`, treats `channel_post` as group-like for auth and routing, adds channel-post auth tests, and includes `ctx.msg` in Telegram update and sequential keys.

## Best Possible Solution

Rework the change on current main so channel native commands reuse the existing `channel_post` sender normalization, preserve group/topic allowlists and callback session routing, then attach live Telegram proof.

## Reproduction Assessment

Yes, source-reproducible. The PR head routes `channel_post` commands through `ctx.msg` and explicit-chat authorizers, while current main's channel-post path synthesizes `sender_chat` and enforces sender allowlists before normal processing.

## Solution Assessment

No. The direction is useful, but the current patch should not bypass or diverge from the existing channel-post sender and group-policy authorization path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve sender allowlists for channel posts:** `extensions/telegram/src/bot-native-commands.ts:333-341`
  - body: This channel-post branch authorizes native/plugin commands from an explicitly configured chat without checking the configured per-group/per-topic sender allowlist. Current channel_post handling synthesizes `sender_chat` and runs the normal group sender checks, so this widens command execution for relay channels that intentionally restrict which posting identity may trigger OpenClaw.
  - confidence: 0.86
- **[P2] Populate sender_chat before commands.allowFrom checks:** `extensions/telegram/src/bot-native-commands.ts:241-245`
  - body: When `commands.allowFrom` is configured, this path checks `resolveCommandAuthorization` but still derives `SenderId` only from `msg.from`. Real channel posts commonly identify the actor through `sender_chat`, and group chat contexts do not fall back to `From`, so allowed channel sender IDs cannot match and the command is rejected even in explicitly allowed channels.
  - confidence: 0.82
- **[P2] Route channel callbacks through channel sessions:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Switching native commands to `ctx.msg` makes channel `/models`-style menu commands reachable, but callback queries from channel messages are still classified as non-group in the callback handler. Those button follow-ups will use DM authorization/session routing instead of the channel conversation, so channel command support needs matching callback routing or a narrower non-interactive scope.
  - confidence: 0.78
