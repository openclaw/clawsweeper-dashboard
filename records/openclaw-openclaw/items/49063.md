## Summary

Keep open: the reported Telegram channel-post command gap is real on current main, but this external PR is not mergeable as-is because it lacks real Telegram proof and introduces authorization/session regressions in the newly enabled command path.

## What This Changes

The PR updates Telegram native/plugin slash command handlers to use grammY `ctx.msg`, treats channel posts as group-like for command authorization/order keys, and adds channel-post auth/key tests.

## Best Possible Solution

A refreshed Telegram fix should reuse current channel-post normalization and authorization, define the remaining policy with https://github.com/openclaw/openclaw/issues/49062, cover allowed/denied/callback cases, and include live Telegram proof.

## Reproduction Assessment

Yes, source-level: current main's native/plugin command handlers still read `ctx.message`, while grammY documents `ctx.msg` as the message shortcut and current channel_post ingress is separate. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Using `ctx.msg` is the right dependency direction, but this branch should be rebuilt around the current channel-post auth/callback model and the policy decision tracked in https://github.com/openclaw/openclaw/issues/49062.

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list. The shared configured-mode gate treats that as allowed, so any channel the bot can read can run native/plugin commands even though the change is scoped to explicitly allowed chats.
  - confidence: 0.91
- **[P2] Use sender_chat when checking channel command allowlists:** `extensions/telegram/src/bot-native-commands.ts:351-352`
  - body: When `commands.allowFrom` is configured, the channel-post path relies on `commandsAllowFromAccess`, but the auth context still derives `SenderId` only from `msg.from`. Telegram channel posts normally identify the posting channel via `sender_chat`, so explicitly allowed channel commands can be rejected once any command allowlist is configured.
  - confidence: 0.87
- **[P2] Honor per-channel policy overrides for channel commands:** `extensions/telegram/src/bot-native-commands.ts:298-299`
  - body: The channel command path evaluates group policy with `useTopicAndGroupOverrides: false` and then treats any explicit `groupConfig` as enough to authorize the command. A channel entry with `groupPolicy: "disabled"` can therefore still execute native/plugin commands.
  - confidence: 0.84
- **[P2] Route channel command callbacks through channel policy:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Switching command handlers to `ctx.msg` makes interactive native commands such as `/models` reachable from channels, but callback handling still classifies `chat.type === "channel"` as non-group and uses DM-style authorization/session routing. Button follow-ups can be dropped or applied to the wrong session after the initial channel command succeeds.
  - confidence: 0.82
