## What This Changes

The PR changes Telegram native/plugin command handlers to read grammY `ctx.msg`, treat `channel_post` updates as command inputs, add channel-post authorization logic, and add focused auth/sequential-key tests.

## Best Possible Solution

Rebase onto current main and implement channel-post native commands through the same `sender_chat`, group policy, ingress authorization, and callback/session routing rules as the normal `channel_post` path, then add focused tests and live Telegram proof.

## Reproduction Assessment

No live repro was run, but the review blockers are source-reproducible: grammY command middleware reaches channel posts, the PR enables that path, and current sender/callback code does not yet treat channel posts consistently. The original Telegram user flow still needs a redacted live artifact.

## Solution Assessment

No. Using `ctx.msg` is necessary, but the best fix must also reuse current channel_post sender identity, group policy, ingress authorization, and callback/session routing before this can safely merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Require explicit channel authorization when access groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `useAccessGroups` off and `commands.allowFrom` unset, this branch gives `resolveCommandAuthorizedFromAuthorizers` an empty authorizer list in `configured` mode, which that helper treats as allow. That makes native/plugin slash commands run from any readable Telegram channel, not only explicitly allowed chats as the PR claims.
  - confidence: 0.86
- **[P1] Use sender_chat before enforcing command allowlists:** `extensions/telegram/src/bot-native-commands.ts:352`
  - body: The channel-post path still builds `senderId` only from `msg.from`, but Telegram channel posts usually identify the posting channel via `sender_chat`. With `commands.allowFrom.telegram` or owner-style command allowlists configured, intended channel posts can be rejected because the authorized channel identity is never passed into command auth.
  - confidence: 0.88
- **[P1] Route channel command callbacks through the channel session:** `extensions/telegram/src/bot-native-commands.ts:341-342`
  - body: Authorizing channel-post commands also enables interactive commands like model/argument menus, but callback handling still treats `chat.type === "channel"` as non-group and resolves authorization/session state as a direct callback. Those follow-ups can be dropped or applied to the clicking admin's direct session instead of the channel session.
  - confidence: 0.84
