## Summary

This PR should stay open because it targets a real Telegram channel_post native-command gap, but the current patch changes Telegram command authorization in ways that can broaden command execution and it still lacks required live Telegram proof.

## What This Changes

The PR changes Telegram native command handlers and tests to use grammY `ctx.msg` for channel_post updates and authorize native/plugin slash commands from Telegram channel posts.

## Best Possible Solution

Rework native channel commands to reuse the existing channel_post normalization and authorization boundary, including explicit configured channel checks, sender_chat-derived identity, groupPolicy/allowFrom behavior, and callback session routing, then attach live Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows current main's native command handlers read `ctx.message`, while channel_post updates are handled by a separate normalized pipeline and grammY exposes `ctx.msg` as the common message shortcut.

## Solution Assessment

No. The patch reaches the right surface via `ctx.msg`, but it diverges from current channel_post normalization, authorization, and callback routing instead of sharing the existing channel boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require explicit channel config for channel posts:** `extensions/telegram/src/bot-native-commands.ts:333-344`
  - body: When `useAccessGroups` is false and `commands.allowFrom` is unset, channel_post commands build an empty authorizer list. `resolveCommandAuthorizedFromAuthorizers(..., "configured")` treats that as allow, which is broader than the existing channel_post message path that requires a configured channel before processing.
  - confidence: 0.88
- **[P2] Populate sender_chat before commands.allowFrom checks:** `extensions/telegram/src/bot-native-commands.ts:238-245`
  - body: Channel posts usually carry the posting identity in `sender_chat`, but this auth path derives sender identity only from `msg.from` while classifying the context as group-like. A configured `commands.allowFrom.telegram` entry for the channel sender therefore cannot match reliably.
  - confidence: 0.84
- **[P2] Route channel command callbacks through channel auth:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: Using `ctx.msg` makes interactive commands such as `/models` reachable from channel posts, but callback handling still treats channel chats as non-group callbacks. Button clicks can be dropped under DM auth or apply to the wrong session instead of the channel session.
  - confidence: 0.78
