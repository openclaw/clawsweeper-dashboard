## Summary

Keep open. Current main still does not implement Telegram native command handling for `channel_post` updates, and this PR is a relevant implementation candidate, but the current diff is not merge-ready because the new channel-post command path has unresolved authorization and callback/session behavior.

## What This Changes

The branch changes Telegram native command handlers to use grammY `ctx.msg`, treats channel posts as group-like for command authorization, adds `ctx.msg` update/sequential-key support, and adds channel_post authorization tests.

## Best Possible Solution

Keep this PR open, refresh it against current main, and make the Telegram channel-post native-command path share the same explicit chat policy result as the inbound channel_post handler while preserving `commands.allowFrom`, per-chat `groupPolicy`, and callback/session behavior. Add regression tests for allowed, denied, disabled, `commands.allowFrom`, and interactive callback channel-post cases.

## Reproduction Assessment

Yes. The original bug is source-reproducible on current main because native command callbacks read only `ctx.message`, while the PR regressions are source-reproducible from the diff by using a channel_post with non-wildcard `commands.allowFrom.telegram` or a configured channel whose per-chat `groupPolicy` is disabled.

## Solution Assessment

No. Switching to `ctx.msg` is the right first step, but the current PR is not the best complete fix because it enables channel-post commands before carrying through `sender_chat`, per-chat policy overrides, and callback/session semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Populate channel sender before commands.allowFrom:** `extensions/telegram/src/bot-native-commands.ts:238-241`
  - body: After switching channel posts into this auth path, the context marks them as group chats but still derives `SenderId` only from `msg.from`. Channel updates may not have `from`, and command auth does not fall back to `From` for group chats, so non-wildcard `commands.allowFrom.telegram` lists reject intended channel commands. Synthesize the channel `sender_chat` identity or keep explicit chat authorization separate before enforcing `commands.allowFrom`.
  - confidence: 0.86
- **[P1] Honor per-channel policy before allowing commands:** `extensions/telegram/src/bot-native-commands.ts:331-340`
  - body: The channel-post authorizer allows any truthy `channelPolicy.groupConfig`, but the policy check above does not use topic/group overrides. A channel explicitly configured with `groupPolicy: "disabled"` still has a `groupConfig` and can pass native command auth. Thread the resolved per-channel policy result through this branch or reject disabled/narrowed channel configs before treating the chat as allowed.
  - confidence: 0.84
- **[P2] Route channel callbacks through the channel scope:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: Enabling native commands from `ctx.msg` also enables commands that send inline keyboards, but the existing callback handler treats `chat.type === "channel"` as non-group and resolves auth/session as a DM. Commands like `/models` can succeed from a channel while their button clicks are dropped or routed to the wrong session. Update the callback path or gate interactive channel commands until channel callbacks use channel policy/session state.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR expands Telegram native command execution to channel posts and has concrete authorization-scope issues that need maintainer attention before merge.

Concerns:

- **[high] Per-channel policy override can be bypassed:** `extensions/telegram/src/bot-native-commands.ts:331`
  - body: The new channel authorizer treats the presence of `groupConfig` as enough to authorize commands, while the policy check ignores per-chat `groupPolicy` overrides. That can allow native command execution in a configured channel whose own policy disables or narrows access.
  - confidence: 0.84
- **[medium] Configured command allowlists cannot match channel sender_chat:** `extensions/telegram/src/bot-native-commands.ts:238`
  - body: When `commands.allowFrom.telegram` is configured, command auth matches sender candidates built from `SenderId`; the PR does not populate that from `sender_chat` for channel posts, so intended non-wildcard allowlists fail closed and leave operators with confusing or overly broad alternatives.
  - confidence: 0.82
