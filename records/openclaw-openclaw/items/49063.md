## Summary

## What This Changes

The PR changes Telegram native and plugin slash-command handlers to read `ctx.msg`, adds channel-post command tests, and authorizes Telegram `channel_post` slash commands from explicitly configured channel chats.

## Best Possible Solution

Update the branch to reuse the current channel_post sender, authorization, session, and callback path for native/plugin slash commands, add focused tests for allowed/denied channel commands and callbacks, and keep the remaining sender-scoped policy decision tracked in https://github.com/openclaw/openclaw/issues/49062.

## Reproduction Assessment

Yes at source level: current main reads ctx.message in native/plugin command handlers, while grammY documents ctx.msg as the shortcut that covers channelPost-style message objects. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No: switching to ctx.msg is a plausible first step, but the PR is too narrow because it does not carry over the existing channel_post sender/auth/session/callback behavior. The better fix is to share or mirror the normalized channel_post ingress path before enabling native commands there.

- **[P1] Preserve channel policy for channel commands:** `extensions/telegram/src/bot-native-commands.ts:287`
  - body: By feeding channel posts into native command auth as `isGroupOrChannel` while the policy call still ignores per-channel/topic overrides, an explicitly configured channel can run slash commands even when its own group policy is disabled or narrowed. Reuse the existing channel_post ingress policy path or honor the same overrides before authorizing the command.
  - confidence: 0.86
- **[P1] Route channel callbacks through the channel session:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Enabling `ctx.msg` makes interactive native commands such as model menus reachable from channels, but the callback handler still treats `chat.type === "channel"` as non-group and resolves state through the clicking user's direct session. Channel command buttons can therefore be dropped or update the wrong session unless the callback path is updated too.
  - confidence: 0.84
- **[P2] Use sender_chat for channel command allowlists:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: For channel_post updates the usable sender identity is normally `sender_chat`, but this auth context only reads `msg.from`. With `commands.allowFrom.telegram` configured, an allowed posting channel cannot match the allowlist, so explicitly allowed channel commands still fail in that common locked-down setup.
  - confidence: 0.82
