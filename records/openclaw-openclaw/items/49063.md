## Summary

Keep this PR open. The change targets a plausible Telegram bug, but the current branch introduces command-authorization regressions on the new `channel_post` native-command path and still lacks real Telegram proof for an auth/transport behavior change.

## What This Changes

The PR changes Telegram native and plugin command handlers to read `ctx.msg` so `channel_post` slash commands can be authorized as group-like contexts, with added unit coverage for allowed and denied channel posts.

## Best Possible Solution

Share the existing channel_post normalization and explicit configured-channel gate with native command auth, define the sender-scoped allowFrom policy in the linked issue, then verify the allowed and denied paths with real Telegram proof plus focused regression tests.

## Reproduction Assessment

Yes, for the review findings by source inspection: the PR routes raw `channel_post` updates into native auth, while current main shows sender_chat normalization, configured-channel gating, and empty-authorizer allow behavior. I did not run tests or live Telegram proof because this was a read-only review.

## Solution Assessment

No. The branch enables the intended surface, but it does not preserve the existing channel_post auth invariants; the safer fix is to reuse the current normalized channel_post identity/gate and make the remaining sender-scoped policy explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Populate channel-post sender identity before auth:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: Changing the handlers to pass `ctx.msg` lets raw `channelPost` objects enter native command auth, but that path still reads `senderId` from `msg.from`. Telegram channel posts commonly carry the posting identity in `sender_chat`, and the existing channel_post message path synthesizes `from` from it before authorization. With `commands.allowFrom` or owner allowlists configured, explicitly allowed channel commands are rejected because the sender candidates are empty. Normalize channel posts, or read `sender_chat`, before both native and plugin command authorization.
  - confidence: 0.91
- **[P1] Require an explicit channel gate when access groups are off:** `extensions/telegram/src/bot-native-commands.ts:343`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this empty authorizer list reaches `resolveCommandAuthorizedFromAuthorizers(..., modeWhenAccessGroupsOff: "configured")`, which returns true when nothing is configured. That bypasses the existing channel_post pipeline's `requireConfiguredGroup` check and allows native/plugin commands from any channel the bot can read, contradicting the explicit-chat boundary in the PR.
  - confidence: 0.9
