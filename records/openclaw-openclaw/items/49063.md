## Summary

Keep open: this PR addresses a real Telegram channel-command gap, but it is not merge-ready because the head still has failing Telegram tests, an authorization gap for channel_post allowlists, and no real Telegram proof for an auth-sensitive behavior change. The paired authorization semantics issue at https://github.com/openclaw/openclaw/issues/49062 is still open.

## What This Changes

The branch changes Telegram native command handling to read `ctx.msg`, treats channel posts as group-like command authorization contexts, adds `ctx.msg` update/sequential keys, and adds channel-post command auth tests.

## Best Possible Solution

Rebase onto current main's Telegram auth helpers, define the channel_post auth policy with the paired issue, carry sender_chat/callback/session behavior through the native command path, and prove the allowed and denied cases with focused tests plus live Telegram evidence.

## Reproduction Assessment

Yes, source-level. Current main's native Telegram command handlers read `ctx.message` and return before auth when it is absent, while this PR is trying to route channel_post command updates through `ctx.msg`; I did not run live Telegram.

## Solution Assessment

No, this PR is not the best mergeable solution as-is. It should be rebased onto the current auth implementation and cover sender_chat allowlist identity, interactive callbacks, focused tests, and live Telegram proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve existing command test contexts after the ctx.msg switch:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Changing the handler to read only `ctx.msg` leaves existing native-command test contexts that still provide `message` returning before any command work runs; the latest head CI shows 21 Telegram failures with zero calls from this path. Either keep a `ctx.msg ?? ctx.message` fallback or update the shared harnesses so existing command coverage still exercises the handler.
  - confidence: 0.89
- **[P1] Use sender_chat for channel-post command allowlists:** `extensions/telegram/src/bot-native-commands.ts:245-246`
  - body: When `commands.allowFrom` is configured, this path authorizes using `SenderId`, but channel posts usually provide the posting identity as `sender_chat` rather than `from`. Because command auth does not use the group `From` value as a sender fallback, an allowlist containing the intended channel identity will still reject the channel command.
  - confidence: 0.84
- **[P2] Route channel command callbacks through channel authorization:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: After `ctx.msg` lets channel_post commands enter this handler, commands that send inline buttons become reachable from channels, but the callback handler still classifies `chat.type === "channel"` as non-group and uses DM-style auth/session routing. Interactive native commands such as model or argument menus can be dropped or mutate the wrong direct session unless the callback path is updated with the channel command behavior.
  - confidence: 0.78
