## Summary

Keep this PR open: current main still has the Telegram channel_post native-command gap, but the proposed patch expands a command authorization surface and still has blocking auth/session issues before merge.

## What This Changes

The PR changes Telegram native command handlers to read grammY `ctx.msg`, treats channel_post messages as group-like for command authorization and threading, and adds channel-post authorization/update-key tests.

## Best Possible Solution

Update this PR into a complete Telegram channel native-command implementation that normalizes channel_post identity consistently, enforces explicit chat and per-channel policy, defines or rejects sender-scoped channel allowlists, preserves channel session scope for callbacks, and adds release-note coverage.

## Reproduction Assessment

Yes. Current main is source-reproducible because Telegram native command handlers return on missing `ctx.message`, while channel_post contexts are exposed through grammY’s channel-post/message shortcut path; the PR regressions are also source-reproducible from the diff and current auth helpers.

## Solution Assessment

No. Reading `ctx.msg` is the right starting point, but this patch is not the best complete fix because it does not carry channel sender identity, per-channel policy, and callback/session scope through the newly enabled command path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Populate channel-post sender identity before command auth:** `extensions/telegram/src/bot-native-commands.ts:238-241`
  - body: After routing channel posts into native command auth, the PR still derives `SenderId` only from `msg.from`. Telegram channel posts normally expose the posting identity through `sender_chat`, and command auth does not fall back to `From` for group-like chats, so non-wildcard `commands.allowFrom.telegram` or owner allowlists reject intended channel commands. Mirror the inbound channel_post sender_chat normalization before enforcing command auth.
  - confidence: 0.88
- **[P1] Honor per-channel policy before allowing native commands:** `extensions/telegram/src/bot-native-commands.ts:331-340`
  - body: The channel-post authorizer treats any truthy `channelPolicy.groupConfig` as enough to allow commands, but the policy check above still runs with topic/group overrides disabled. A configured channel whose own `groupPolicy` disables or narrows access can pass this branch because `groupConfig` exists. Thread the resolved override-aware policy result through this branch before treating the channel as authorized.
  - confidence: 0.84
- **[P2] Route channel callbacks through channel session state:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: Switching native handlers to `ctx.msg` makes channel commands with inline keyboards reachable, but the existing callback handler still classifies `chat.type === "channel"` as non-group and uses DM authorization/session context. Commands like `/models` can succeed from a channel while follow-up button clicks are dropped or applied to the wrong session. Update the callback path or gate interactive channel commands until channel scope is supported.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: This is a user-visible Telegram bug fix, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires a single-line entry under the active fixes section so the release notes capture the behavior change.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR expands Telegram native command execution to channel posts and has concrete authorization-scope issues that should block merge until repaired.

Concerns:

- **[high] Per-channel policy override can be bypassed:** `extensions/telegram/src/bot-native-commands.ts:331`
  - body: The new channel authorizer treats `groupConfig` presence as sufficient while the policy check does not honor per-channel `groupPolicy` overrides, so configured channels with disabled or narrower policy can execute native commands.
  - confidence: 0.84
- **[medium] Channel sender allowlists cannot match sender_chat:** `extensions/telegram/src/bot-native-commands.ts:238`
  - body: The PR does not synthesize `sender_chat` into `SenderId` before command authorization, so configured `commands.allowFrom.telegram` and owner-style allowlists fail closed for intended channel identities and may push operators toward broader command access.
  - confidence: 0.82
- **[medium] Interactive channel commands use non-channel callback scope:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: Newly reachable channel commands can emit inline buttons, but callback handling still treats Telegram channels as non-group chats and resolves authorization/session state through the wrong scope.
