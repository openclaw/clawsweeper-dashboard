## Summary

Keep this PR open. Current main still drops native Telegram `channel_post` slash commands, so the contribution targets a real gap, but the patch expands a command execution surface while leaving concrete channel authorization and callback/session bugs in the proposed path.

## What This Changes

The PR changes Telegram native command handlers to read grammY `ctx.msg`, treats `channel_post` messages as group-like for native-command authorization and threading, adds `ctx.msg` update/sequential-key support, and adds channel-post authorization tests.

## Best Possible Solution

Keep the PR open and update it into a complete Telegram channel native-command implementation: normalize channel_post identity consistently with the inbound handler, require the intended explicit chat policy, honor or explicitly reject sender-scoped per-channel allowlists, route interactive callbacks through channel session state or gate interactive native commands, and add the required changelog entry.

## Reproduction Assessment

Yes. Current main source inspection reproduces the original gap because native Telegram commands return on missing `ctx.message`; the PR regressions are source-reproducible from the diff with a channel_post carrying `sender_chat` but no synthesized `msg.from`, plus configured `commands.allowFrom.telegram` or a per-channel `groupPolicy` override.

## Solution Assessment

No. Using `ctx.msg` is the right starting primitive, and the PR discussion correctly points to grammY's shortcut behavior, but the patch is not the best complete fix because it does not carry channel sender identity, per-channel policy, and callback/session scope through the newly enabled command path.

Full review comments:

- **[P1] Populate channel-post sender identity before command auth:** `extensions/telegram/src/bot-native-commands.ts:238-241`
  - body: After routing channel posts into native command auth, the PR still derives `SenderId` only from `msg.from`. Telegram channel posts usually expose the posting identity through `sender_chat`, and command auth does not fall back to `From` for group-like chats, so non-wildcard `commands.allowFrom.telegram` or owner allowlists reject intended channel commands. Use the same `sender_chat` synthesis as the inbound channel_post path before enforcing command auth.
  - confidence: 0.88
- **[P1] Honor per-channel policy before allowing native commands:** `extensions/telegram/src/bot-native-commands.ts:331-340`
  - body: The channel-post authorizer treats any truthy `channelPolicy.groupConfig` as enough to allow commands, but the policy check above still runs with topic/group overrides disabled. A configured channel whose own `groupPolicy` disables or narrows access can pass this branch because `groupConfig` exists. Thread the resolved override-aware policy result through this branch before treating the channel as authorized.
  - confidence: 0.84
- **[P2] Route channel callbacks through channel session state:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: Switching native handlers to `ctx.msg` makes channel commands with inline keyboards reachable, but the existing callback handler still classifies `chat.type === "channel"` as non-group and uses DM authorization/session context. Commands like `/models` can succeed from a channel while follow-up button clicks are dropped or applied to the wrong session. Update the callback path or gate interactive channel commands until channel scope is supported.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `extensions/telegram/src/bot-native-commands.ts:599`
  - body: This is a user-visible Telegram bug fix, but the PR does not add a `CHANGELOG.md` entry. Repo policy requires a single-line entry under the active fixes section with appropriate source-author credit, so release notes would miss the behavior change.
