## Summary

Keep this PR open. Current main still has the reported Telegram native-command gap, but the PR is not mergeable as-is because it expands channel_post native command execution without carrying channel sender identity, per-channel policy overrides, and interactive callback/session scope through the new path.

## What This Changes

The PR changes Telegram native command handlers to read grammY `ctx.msg`, treats `channel_post` updates as group-like for authorization and threading, adds `ctx.msg` update/sequential-key support, and adds channel-post native-command authorization tests.

## Best Possible Solution

Keep the PR open and update it into a complete Telegram channel native-command implementation: normalize `channel_post` identity consistently with the inbound handler, require the intended explicit chat policy while honoring per-channel `groupPolicy` and `allowFrom` semantics, and either route interactive callbacks through channel session state or gate interactive native commands until that support exists.

## Reproduction Assessment

Yes. Current main is source-reproducible because native Telegram command handlers return on missing `ctx.message`; the PR regressions are source-reproducible from a `channel_post` carrying `sender_chat`, configured command/channel policy, and an interactive command such as `/models`.

## Solution Assessment

No. Switching to `ctx.msg` is the right primitive for grammY channel posts, but this patch is not the best complete fix because the newly enabled path does not preserve channel sender identity, per-channel policy overrides, or callback/session scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Populate channel sender before command auth:** `extensions/telegram/src/bot-native-commands.ts:241-246`
  - body: After this PR routes channel posts into native command auth, `senderId` still comes only from `msg.from`. Telegram channel posts usually expose the posting identity through `sender_chat`, so `commands.allowFrom.telegram` and owner-style command allowlists reject intended channel commands even when the channel identity is configured. Synthesize the same sender identity used by the normal `channel_post` pipeline before calling command auth.
  - confidence: 0.88
- **[P1] Honor per-channel policy before allowing commands:** `extensions/telegram/src/bot-native-commands.ts:332-341`
  - body: The new channel authorizer treats any exact `groupConfig` as sufficient, but the policy check above still ignores topic/group `groupPolicy` overrides. A channel explicitly listed with `groups[chatId].groupPolicy: "disabled"` or a narrower policy can still pass this branch because `groupConfig` exists. Thread the override-aware policy result through this branch before treating the channel as authorized.
  - confidence: 0.84
- **[P2] Route channel callbacks through channel session state:** `extensions/telegram/src/bot-native-commands.ts:602`
  - body: Using `ctx.msg` makes channel commands with inline keyboards reachable, but the callback handler still classifies `chat.type === "channel"` as non-group and resolves auth/session state through DM-style scope. Commands like `/models` can succeed from a channel while button clicks are dropped or applied to the wrong session. Update the callback path, or gate interactive channel commands until channel scope is supported.
  - confidence: 0.78
