## Summary

Keep open: the source problem is real, but this PR still widens Telegram channel command execution without preserving existing channel auth, callback, and per-chat policy boundaries, and it lacks required live Telegram proof.

## What This Changes

The PR changes Telegram native/plugin command handlers to use `ctx.msg`, treats `channel_post` as group-like for native command auth, and adds channel-post auth/update-key/sequential-key tests.

## Best Possible Solution

Rebuild this on the current Telegram ingress/auth helpers so channel commands require explicit channel policy, preserve `sender_chat`/allowFrom and callback/session routing, cover allowed and denied cases, and attach live Telegram proof.

## Reproduction Assessment

Yes, source-level: current main's handlers read `ctx.message`, while grammY's pinned `ctx.msg` includes `channel_post`. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No. Using `ctx.msg` is the right dependency direction, but this branch bypasses explicit-channel and sender/callback policy; the safer path is to reuse the current channel_post ingress/auth model and settle the policy tracked in https://github.com/openclaw/openclaw/issues/49062.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require explicit channel authorization when groups are off:** `extensions/telegram/src/bot-native-commands.ts:344`
  - body: For channel posts with `commands.useAccessGroups=false` and no `commands.allowFrom`, this branch passes an empty authorizer list. The configured-mode gate treats that as allowed, so any channel the bot can read can run native/plugin commands even though the PR's boundary is explicitly allowed chats.
  - confidence: 0.9
- **[P2] Populate channel sender identity before allowFrom checks:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: Channel posts normally identify the actor through `sender_chat`, but this auth context still derives `SenderId` and `SenderUsername` only from `msg.from`. Since command auth does not use the group-like `From` fallback, deployments using `commands.allowFrom.telegram` can reject the intended allowed channel identity.
  - confidence: 0.87
- **[P2] Route channel callback follow-ups through channel policy:** `extensions/telegram/src/bot-native-commands.ts:341`
  - body: Allowing the initial channel command here makes interactive native commands reachable from channels, but callback handling still classifies `chat.type === "channel"` as non-group and uses DM authorization/session routing. Button follow-ups can be dropped or mutate the wrong session after the channel command succeeds.
  - confidence: 0.82
- **[P2] Honor per-channel groupPolicy overrides:** `extensions/telegram/src/bot-native-commands.ts:298`
  - body: The channel command path still calls group policy evaluation with `useTopicAndGroupOverrides: false`, then treats any explicit `groupConfig` as enough to authorize the channel. A channel entry with `groups[chatId].groupPolicy: "disabled"` can therefore pass native command auth even though regular channel_post ingress honors group-level policy overrides.
  - confidence: 0.8
