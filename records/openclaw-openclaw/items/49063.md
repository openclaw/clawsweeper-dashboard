## Summary

Keep this PR open: the requested Telegram channel-post native command behavior is plausible, but the patch still has authorization/session correctness blockers and lacks required real Telegram proof for an external auth-surface change.

## What This Changes

The PR switches Telegram native/plugin command handlers to `ctx.msg`, treats `channel_post` updates as group-like command contexts, and adds channel-post authorization tests.

## Best Possible Solution

Revise the PR to share the existing channel_post normalization/authorization semantics, cover sender_chat and callback/session behavior, then add redacted live Telegram proof for the visible channel command flow.

## Reproduction Assessment

Yes by source inspection: current main's native command handlers read `ctx.message`, while Telegram `channel_post` updates are exposed through `ctx.channelPost`/`ctx.msg`, so `/status` from a channel post exits before dispatch. I did not run a live Telegram repro in this read-only pass.

## Solution Assessment

No. The direction is plausible, but the current patch creates a parallel channel-post command auth path that misses sender_chat, explicit-channel safety, and callback/session consistency; reusing or matching the existing channel_post pipeline is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use sender_chat for channel-post command auth:** `extensions/telegram/src/bot-native-commands.ts:227-228`
  - body: For `channel_post` updates, this path still builds `SenderId` from `msg.from` only. The existing channel-post pipeline synthesizes the sender from `sender_chat`, and command authorization does not fall back to `From` for group/channel chats, so `commands.allowFrom.telegram` or owner allowlists reject an otherwise allowed channel sender.
  - confidence: 0.9
- **[P1] Require configured channels when access groups are off:** `extensions/telegram/src/bot-native-commands.ts:333-344`
  - body: When `useAccessGroups` is false and `commands.allowFrom` is unset, channel posts get `commandAuthorizers = []`; `modeWhenAccessGroupsOff: "configured"` treats that as allowed. That lets slash commands run from any channel where the bot receives `channel_post`, unlike the existing channel-post handler which requires a configured channel entry.
  - confidence: 0.84
- **[P2] Route channel command callbacks through the channel context:** `extensions/telegram/src/bot-native-commands.ts:341`
  - body: This branch enables native/plugin commands from channels, including commands that emit inline keyboards. The callback handler still classifies `chat.type === "channel"` as non-group, so follow-up button clicks use DM-style authorization and session routing, making channel `/models`-style flows drop or mutate the wrong session.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff broadens Telegram command execution and has a concrete authorization-scope gap for channel posts when access groups are disabled.
