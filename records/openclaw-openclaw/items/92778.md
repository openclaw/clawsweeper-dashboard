Keep open: this targets a valid P1 macOS VoiceWake overlay render-loop path, but the async guard can leave overflow state stale and the supplied proof is build/test output rather than live overlay behavior.

## What This Changes

The PR changes the macOS VoiceWake overlay height measurement to defer `model.isOverflowing` updates through `DispatchQueue.main.async` with an equality guard.

## Best Possible Solution

Coalesce or enqueue the latest overflow state so deferred writes cannot go stale, then add redacted live macOS overlay proof showing repeated triggers without pinwheel or SwiftUI warnings.

## Reproduction Assessment

Yes for source reproduction: current main still has the attributed-text `onChange` -> `updateWindowFrame` -> `targetFrame` -> `measuredHeight` -> observable `isOverflowing` write chain. I did not run a live macOS pinwheel reproduction in this read-only review.

## Solution Assessment

No. Deferring the state mutation is the right boundary, but guarding before the async handoff can drop the latest overflow value; the safer fix queues or coalesces the latest value and proves the live overlay path.

- **[P2] Queue the latest overflow value:** `apps/macos/Sources/OpenClaw/VoiceWakeOverlayController+Window.swift:98-100`
  - body: The guard runs before the async handoff, so a later measurement can be skipped while an older queued value is still pending. Queue every measured value with the equality check inside the closure, or coalesce pending overflow state so the last measurement wins.
