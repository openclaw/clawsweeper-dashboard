## Summary

Keep open: current main and the reported v2026.5.28 release still warn on plaintext `models.providers.*.request.headers.*`, and nearby tests intentionally lock that behavior, so this needs a maintainer policy decision rather than cleanup closure.

## What This Changes

The issue asks for `openclaw doctor` and `openclaw secrets audit` to ignore static non-credential model-provider request headers such as `Wafer-ZDR: required`.

## Best Possible Solution

Maintainers should decide the credential policy for model-provider request headers, then either align audit/doctor with the existing sensitive-header-name heuristic for this surface or document that all request-header values are intentionally treated as secret-bearing.

## Reproduction Assessment

Yes from source inspection: current main and v2026.5.28 register `models.providers.*.request.headers.*` as audited and only skip the non-sensitive-name heuristic for top-level provider headers. I did not run the CLI because this review is read-only, but the checked-in tests assert the reported warning path.

## Solution Assessment

Unclear until maintainers choose the security policy. A small code change could apply the existing header-name predicate to request headers, but current tests intentionally require warnings for static request headers, so the best fix may also include docs/test policy updates.
