## Summary

Keep open: current main and the latest release still treat model-provider request headers as audited secret-bearing fields, and tests intentionally expect doctor/audit warnings for static request headers, so this needs a maintainer security policy decision rather than cleanup closure.

## What This Changes

The issue asks secrets audit and doctor to stop warning on static non-credential model-provider request headers such as `Wafer-ZDR: required`.

## Best Possible Solution

Maintainers should define whether request headers use the same sensitive-name heuristic as provider headers, then update audit, doctor, docs/schema hints, and tests consistently or document the all-request-headers warning policy.

## Reproduction Assessment

Yes from source inspection: current main registers `models.providers.*.request.headers.*` as audited and only applies the non-sensitive header-name skip to top-level provider headers. I did not run the CLI because this review is read-only, but adjacent tests assert the reported warning path.

## Solution Assessment

Unclear until maintainers choose the security policy. Applying the existing header-name predicate to request headers is plausible, but the best fix must also update the current docs/test contract or explicitly preserve the conservative warning behavior.
