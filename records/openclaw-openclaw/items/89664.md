## What This Changes

The issue asks for `openclaw doctor` and `openclaw secrets audit` to stop treating static non-credential provider request headers such as `Wafer-ZDR: required` as plaintext secrets.

## Best Possible Solution

Capture a maintainer/secops-approved request-header audit rule, then keep doctor and secrets audit aligned on that rule while preserving Authorization-style findings.

## Reproduction Assessment

Yes from source inspection: current main registers request headers as audited and only skips the non-sensitive header-name predicate for top-level provider headers. I did not run the CLI because this review is read-only, but adjacent tests assert the reported warning path.

## Solution Assessment

Unclear until maintainers choose the security policy. The linked PR’s shared-predicate approach is a narrow plausible fix, but the best solution must explicitly accept the request-header false-negative tradeoff or preserve the conservative warning contract.
