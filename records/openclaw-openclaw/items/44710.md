## Summary

Keep open: current main does not implement this hook, and the PR is a new public tool-access plugin API with blocking correctness and security issues around fail-open behavior, client-name reservation, and stale hook-contract placement.

## What This Changes

The PR adds a public `before_tools_resolve` plugin hook, hook-runner merge logic, agent and compaction tool-list integration, and allow/deny filtering tests.

## Best Possible Solution

If maintainers want this API, land it as a rebased plugin contract change in `src/plugins/hook-types.ts`, fail closed for access-control errors, evaluate once over the final model-visible tool set, prevent denied names from influencing bundled tool materialization, and add docs, changelog, and regression coverage.

## Reproduction Assessment

Yes for the review findings: the PR diff plus current-main source trace show the new hook would use the generic fail-open path, reserve pre-filter client names, and add hook contracts in the old `types.ts` location. Runtime reproduction was not attempted because this was a read-only review.

## Solution Assessment

No. The proposed API is plausible, but this patch is not the best merge shape until it is rebased to current hook contracts and its access-control semantics are fail-closed and applied to the final tool set.
