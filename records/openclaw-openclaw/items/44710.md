## Summary

Keep open. Current main still does not expose a list-level `before_tools_resolve` hook, so the PR is not obsolete, but the current patch is not safe to merge because it adds a model-visible tool access-control surface that can fail open, reserves denied client tool names before filtering, and updates the old hook contract location instead of the current `hook-types.ts` surface.

## What This Changes

The PR adds a public `before_tools_resolve` plugin hook that filters the assembled tool list for normal agent runs and compaction using allow and deny results, with hook runner support and tests.

## Best Possible Solution

If maintainers want this API, land it as a rebased plugin SDK contract change in `src/plugins/hook-types.ts`, fail closed for hook errors/timeouts, apply once over the final post-static tool list without letting denied names affect bundled registration, and document the hook in the public plugin hook catalog with focused regression tests for normal runs, compaction, client tools, bundled tools, allow intersection, and failure behavior.

## Reproduction Assessment

Yes for the review findings: the PR diff and current main source show the fail-open policy gap, pre-filter client-name reservation, and stale contract-file placement. Runtime reproduction was not attempted because the review was read-only.

## Solution Assessment

No. The requested hook is plausible, but this PR is not the best merge shape until it is rebased onto the current hook contract surface and its access-control semantics fail closed without pre-filter client names influencing trusted bundled tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Fail closed when tool filtering hooks fail:** `src/plugins/hooks.ts:765`
  - body: `runBeforeToolsResolve` uses the generic modifying-hook path, but the global runner only marks `before_tool_call` as fail-closed. Since unspecified hooks fail open, a thrown or timed-out tool-list policy hook would log and continue with the unfiltered tool list, exposing tools the plugin intended to hide.
  - confidence: 0.86
- **[P2] Reserve bundled names after client filtering:** `src/agents/pi-embedded-runner/run/attempt.ts:519-522`
  - body: `reservedCoreAndClientNames` includes `clientToolDefsPre` before `before_tools_resolve` filters denied client tools. A denied client-supplied name can still be passed as reserved to bundled MCP/LSP materialization, so it can suppress or rename a trusted bundled tool even though that client tool is not allowed in the final set.
  - confidence: 0.82
- **[P2] Move the hook contract to hook-types:** `src/plugins/types.ts:1817`
  - body: Current main moved `PluginHookName`, `PLUGIN_HOOK_NAMES`, and `PluginHookHandlerMap` into `src/plugins/hook-types.ts`, while this diff adds `before_tools_resolve` to the old `types.ts` contract location. After rebase, registry validation and `api.on` typing will not reliably know about the new hook unless the additions move to the current contract file.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a model-visible tool access-control hook, and the current patch has concrete fail-open and denied-name reservation risks.

Concerns:

- **[high] Tool filter hook fails open:** `src/plugins/hooks.ts:765`
  - body: The hook runner defaults unspecified modifying hooks to fail-open and the global runner only configures `before_tool_call` as fail-closed. A failing `before_tools_resolve` policy hook would therefore expose the unfiltered tool list to the model.
  - confidence: 0.86
- **[medium] Denied client names affect trusted bundled tools:** `src/agents/pi-embedded-runner/run/attempt.ts:519`
  - body: The PR reserves client tool names before the new hook filters them. A denied client tool name can still collide with bundled MCP/LSP names and affect which trusted bundled tools are materialized.
  - confidence: 0.82
