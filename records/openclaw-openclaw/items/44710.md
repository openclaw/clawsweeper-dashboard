## Summary

Keep open: current main still lacks the requested list-level tool filtering hook, but this PR is not safe to merge because the proposed access-control hook can fail open, is stale against the current hook contract split, and still lets denied client tool names affect bundled tool registration.

## What This Changes

The PR adds a public `before_tools_resolve` plugin hook, tool-list allow/deny filtering in agent attempt and compaction paths, and focused hook/filtering tests.

## Best Possible Solution

If maintainers want this API, rebase it onto `src/plugins/hook-types.ts`, make filtering failures fail closed, ensure denied names cannot influence final reservations or replay, and land it with docs, tests, and redacted real-run proof.

## Reproduction Assessment

Yes for the PR review findings: source inspection of PR head shows the fail-open hook path and pre-filter reservations, while current-main inspection shows the hook contract moved to `src/plugins/hook-types.ts`. Runtime reproduction was not attempted because this is a read-only review.

## Solution Assessment

No. The feature direction may be useful, but this branch is not the best merge shape until the hook contract is rebased, access-control failures fail closed, denied names stop affecting bundled tools, and real behavior proof is added.
