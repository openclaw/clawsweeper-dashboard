## Summary

The reported isolated-cron failure is correctly diagnosed, narrowly repaired, and backed by convincing real-process proof, but the patch applies the no-environment fallback only to the primary run builder while the separate Codex side-question builder still removes OpenClaw shell tools in the same zero-environment state.

## What This Changes

The PR restores OpenClaw `exec` and `process` tools for direct Codex app-server runs without a registered Codex environment and adds regression coverage through the real dynamic-tool bridge.

## Best Possible Solution

Make the no-environment shell-ownership rule shared by every Codex app-server turn builder where shell access is permitted, while retaining memory-flush restrictions, plugin exclusions, allowlists, and the side-conversation mutation policy.

## Reproduction Assessment

Yes at high confidence from source: OpenClaw filters its shell tools while bundled Codex registers none when the turn has zero environments. The submitted builder-to-bridge before/after output directly exercises that mechanism, although this review did not run the hosted OAuth cron scenario.

## Solution Assessment

No, not yet: the primary cron path is fixed correctly, but the best maintainable solution must also apply or explicitly reject the same invariant in the independent side-question tool builder.
