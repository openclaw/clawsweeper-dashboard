## Summary

Keep open: this is a focused and useful CLI bug fix, but the current patch still treats a supplied whitespace-only `--timeout` as omitted instead of invalid. The PR also has a protected `maintainer` label, so conservative cleanup should not close it.

## What This Changes

This PR changes the update CLI timeout parser to require full positive integer second strings and adds focused parser tests.

## Best Possible Solution

Keep the full-string integer validation approach, but only treat an absent option as omitted and reject whitespace-only supplied values with parser coverage.

## Reproduction Assessment

Yes, at source level: current main uses `Number.parseInt` in `parseTimeoutMsOrExit`, so `1.5` and `10abc` are accepted as numeric prefixes. The remaining PR defect is also source-reproducible by passing a whitespace-only timeout string to the new parser.

## Solution Assessment

No, not as written: full-string integer validation is the right narrow fix, but the branch treats whitespace-only supplied values as omitted. Return `undefined` only for `timeout === undefined`, then validate the trimmed string.
