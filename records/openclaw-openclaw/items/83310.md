## Summary

Keep this PR open: the underlying timeout parsing bug is valid, but the patch still lets whitespace-only timeout arguments bypass validation and the PR only shows helper-level test/check output rather than real CLI behavior proof.

## What This Changes

The PR changes update CLI timeout parsing to require full-string positive integer seconds and adds helper-level parser tests.

## Best Possible Solution

Keep the full-string integer validation in `parseTimeoutMsOrExit`, return `undefined` only for an omitted argument, reject whitespace-only values, and add real CLI proof before normal maintainer review.

## Reproduction Assessment

Yes, at source level: current main uses `Number.parseInt` in `parseTimeoutMsOrExit`, so partial numeric strings such as `1.5` and `10abc` parse to positive timeout values instead of the invalid-timeout path. I did not run the update command because this review is read-only and update flows can affect install state.

## Solution Assessment

No, not as written: full-string integer validation is the right narrow fix, but returning `undefined` for any empty trimmed string lets whitespace-only supplied values bypass validation. The safer implementation should treat only `undefined` as omitted, then validate the trimmed string.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject whitespace-only timeout values:** `src/cli/update-cli/shared.ts:58-59`
  - body: The new early return treats any value that trims to empty as if `--timeout` was not supplied. A user can pass `--timeout "   "` and bypass the positive-integer validation; current main rejected that path as invalid, and the option/docs contract requires a positive integer in seconds. Return `undefined` only when `timeout` is actually `undefined`, then validate the trimmed string.
