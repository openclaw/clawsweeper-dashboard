## Summary

Keep open: this PR targets a valid source-reproducible update CLI bug and is paired with the open linked issue at https://github.com/openclaw/openclaw/issues/83281, but the current diff still lets whitespace-only timeout values bypass validation and the external PR only provides test/check validation rather than real behavior proof.

## What This Changes

The PR updates `parseTimeoutMsOrExit` to require full-string positive integer seconds and adds focused helper-level tests for valid and invalid timeout strings.

## Best Possible Solution

Keep the fix in `parseTimeoutMsOrExit`, distinguish truly omitted input from whitespace-only input, add a regression case for whitespace-only values, and then review the paired issue-closing PR normally.

## Reproduction Assessment

Yes, at source level: current main parses `1.5` and `10abc` with `Number.parseInt`, producing positive millisecond values instead of the positive-integer error path. I did not run the update command because this review is read-only and update flows can affect install state.

## Solution Assessment

No, not as written: full-string integer validation is the right narrow fix, but the proposed early return for `!trimmed` permits whitespace-only non-integers. The safer fix is to return `undefined` only for an actually omitted argument, then validate the trimmed string.

## Review Findings

Overall correctness: patch is incorrect

Return `undefined` only when `timeout` is actually `undefined`, then validate the trimmed string.
