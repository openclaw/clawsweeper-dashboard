## Summary

Keep this PR open: current main and the latest release still use unbounded `response.json()` for both self-hosted discovery success bodies, and the PR’s bounded-reader approach is a plausible fix. The remaining blocker is maintainer acceptance of the new fail-closed 16 MiB discovery cap, plus refreshing the PR body/proof that still describes the earlier 4 MiB cap.

## What This Changes

The PR routes self-hosted `/models` and llama.cpp `/props` discovery JSON bodies through a shared 16 MiB byte-limited reader and adds unbounded-stream regression tests.

## Best Possible Solution

Land the bounded-reader implementation after the PR body/proof matches the 16 MiB head and maintainers accept that discovery payloads above that cap should fail closed, or tune the cap again before merge.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.6.10 still call `response.json()` on both affected success bodies, so an endless successful JSON stream can be buffered without a byte cap. I did not run a live repro in this read-only review.

## Solution Assessment

Yes, pending cap acceptance. The PR applies the existing bounded response helper at the exact untrusted success-body reads; a content-length-only guard would not protect chunked streams, and the main open question is the chosen byte limit.
