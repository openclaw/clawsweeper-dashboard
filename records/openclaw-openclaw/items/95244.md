## Summary

Keep open: this PR is still necessary because current main and the latest release still use unbounded `response.json()` for self-hosted `/models` and llama.cpp `/props` discovery, while the branch applies the existing bounded-reader pattern with convincing live-output proof. The main remaining decision is maintainer acceptance of the new 4 MiB fail-closed discovery boundary.

## What This Changes

The PR replaces two self-hosted provider discovery `response.json()` reads with a 4 MiB `readResponseWithLimit` helper and adds unbounded-stream regression tests.

## Best Possible Solution

Land the bounded-reader implementation once maintainers accept 4 MiB as the supported discovery response limit, or adjust the cap/docs before merge if large self-hosted catalogs are expected.

## Reproduction Assessment

Yes. Current main and v2026.6.9 call `response.json()` on the two untrusted discovery responses, and the PR body includes live loopback output showing an unbounded stream is capped and cancelled after the patch.

## Solution Assessment

Yes, with a maintainer cap caveat. The PR uses the existing media-core bounded reader at the two implicated self-hosted discovery reads; broader provider JSON hardening would be separate SDK/API compatibility work.
