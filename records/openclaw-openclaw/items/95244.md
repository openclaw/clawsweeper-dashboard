## What This Changes

The PR adds a 4 MiB byte-bounded JSON reader for self-hosted provider discovery and routes both `/models` and llama.cpp `/props` parsing through it with regression coverage.

## Best Possible Solution

Keep the shared bounded-reader fix and merge after exact-head checks if maintainers accept 4 MiB as the self-hosted discovery response cap.

## Reproduction Assessment

Yes from source and contributor proof, though I did not rerun it locally in this read-only review. Current main uses unbounded `response.json()` for `/models` and `/props`, and the PR body shows a loopback unbounded-stream server being capped and aborted after the fix.

## Solution Assessment

Yes, this is the best fix shape I found. It changes the two affected success-body reads at the shared self-hosted provider setup helper, uses the existing media-core bounded reader contract, and preserves the existing fallback paths instead of adding a parallel discovery mechanism.
