## Summary

Keep open: current main and the latest release still use unbounded `response.json()` for self-hosted `/models` and llama.cpp `/props`, while this PR applies the shared bounded reader with targeted overflow/cancel tests and real loopback proof. The remaining maintainer decision is whether 4 MiB is the right fail-closed discovery boundary.

## What This Changes

The PR replaces self-hosted provider discovery JSON parsing with a 4 MiB byte-limited reader and adds regression tests for endless `/models` and `/props` response streams.

## Best Possible Solution

Land the bounded-reader implementation after maintainers accept 4 MiB as the supported discovery response limit, or adjust the cap before merge if known self-hosted catalogs can exceed it.

## Reproduction Assessment

Yes. Current main and v2026.6.9 visibly call `response.json()` on both affected success bodies, and the PR body provides real loopback proof that the patched path caps and aborts an endless response stream.

## Solution Assessment

Yes, with a maintainer-owned limit choice. The PR applies the existing bounded response helper to the exact untrusted success-body reads; the safer alternative is only to tune the 4 MiB cap if supported catalogs require it.
