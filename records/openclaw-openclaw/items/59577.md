## Summary

Keep open, but this PR is not merge-ready. The performance problem is real on current main, but this branch patches the retired `src/memory` layout and has blocking correctness issues; the newer same-author #75179 appears to be the current replacement attempt, so this older paired PR should not be auto-closed until maintainers decide that relationship.

## What This Changes

The PR branch adds hash-based incremental session indexing, offset-aware chunk metadata, schema columns, and benchmark/integration tests under the old `src/memory` tree to avoid re-embedding unchanged session chunks.

## Best Possible Solution

Land or replace this work through a current-main implementation in `extensions/memory-core` and `packages/memory-host-sdk` that preserves unchanged session chunks without breaking FTS-only search, retry/idempotence, FTS citation metadata, and Unicode chunking semantics.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a dirty session transcript reaches `indexFile`, and `writeChunks` clears indexed rows before rewriting all chunks for that changed session file.

## Solution Assessment

No. Hash-based reuse is the right direction, but this PR is not the best implementation because it patches the old memory tree and introduces DB/FTS/failure-ordering regressions; a clean current `memory-core` replacement is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Port the session hook to memory-core:** `src/memory/manager-sync-ops.ts:912-914`
  - body: This replaces the session indexing call in `src/memory`, but current main's active memory engine is `extensions/memory-core`. As submitted, the optimization will not affect the active session sync path.
  - confidence: 0.9
- **[P2] Keep FTS-only session indexing:** `src/memory/manager-embedding-ops.ts:828-833`
  - body: The new incremental method returns when no embedding provider is configured. Current `indexFile` still writes `fts-only` chunks in that mode, so users without embeddings would lose keyword search over session transcripts.
  - confidence: 0.91
- **[P1] Embed replacements before deleting old chunks:** `src/memory/manager-embedding-ops.ts:890-916`
  - body: The incremental path deletes old session rows before requesting embeddings. A provider timeout or rate limit after the delete can leave the session index pruned, unlike the current full path that embeds before mutating indexed rows.
  - confidence: 0.87
- **[P2] Use an upsert for new incremental chunks:** `src/memory/manager-embedding-ops.ts:929-947`
  - body: New chunks are inserted without `ON CONFLICT`, so retrying after a partial sync can hit the same chunk id and fail instead of repairing the stale file record. Match the full-index upsert behavior.
  - confidence: 0.86
- **[P2] Refresh FTS metadata when reusing chunks:** `src/memory/manager-embedding-ops.ts:930-933`
  - body: The hash-match branch updates chunk line/offset metadata without replacing the FTS row. Keyword search reads citation ranges from `chunks_fts`, so reused chunks after line shifts can return stale ranges.
  - confidence: 0.9
- **[P2] Reuse existing chunk sizing semantics:** `src/memory/internal.ts:506`
  - body: The new offset chunker counts raw string length instead of the current weighted, surrogate-safe chunking logic. Switching session sync to it can change CJK/Unicode boundaries and reduce reuse for non-Latin transcripts.
  - confidence: 0.83
