## Summary

Keep open. Current main still has no Feishu recall handling, so the PR addresses a real gap, but the branch is not merge-ready: it does not suppress fallback/direct replies after a recalled source message, does not register media or streaming-card reply outputs for later deletion, and is missing the required changelog entry.

## What This Changes

The PR adds Feishu recalled-message cleanup by registering a recalled-message event handler, keeping an in-memory user-message to bot-reply map with TTL/retry retention, and registering text/card reply-path sends for recall deletion.

## Best Possible Solution

Revise the Feishu plugin implementation so recalled user messages are treated as a reply-boundary state: suppress replies when the source has already been recalled, delete every bot output that was actually sent as a reply to the recalled message, keep retryable deletion failures, and cover the behavior with focused Feishu tests and a changelog entry.

## Reproduction Assessment

Yes. A focused unit path can simulate Feishu returning withdrawn/not-found for the reply target and show the PR still calls the existing fallback direct send; source inspection also shows media and streaming reply outputs never enter the recall map.

## Solution Assessment

No. The direction is appropriate for the Feishu plugin, but the current PR is incomplete; the safer solution is to centralize reply-path result registration and recall suppression across text, card, media, and streaming outputs rather than only wiring text/card helpers.
