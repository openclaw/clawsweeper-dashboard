## Summary

Keep open. Current main still lacks Feishu recalled-message support, but this PR is not mergeable as-is because it changes the observable Feishu send result shape and still does not satisfy the recalled-before-response acceptance criterion.

## What This Changes

The PR adds Feishu recalled-message event handling with an in-memory user-message to bot-reply map, event registration, outbound reply registration, and a send-result flag for reply-path tracking.

## Best Possible Solution

Implement Feishu-owned recall handling with private reply-path tracking, event and send-path tests, deleted-message preflight handling, an explicit decision on fallback semantics, and a user-facing changelog entry.

## Reproduction Assessment

Yes. Current main has focused unit coverage for exact Feishu send-result shape and withdrawn-target direct fallback, and a mocked `im.message.reply` returning withdrawn/not-found would show the PR still sends before any recall preflight.

## Solution Assessment

No. Event wiring and reply tracking are in the right plugin, but the current diff leaks internal state through the public result object and does not actually suppress replies before the fallback path fires.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep reply-path state out of send results:** `extensions/feishu/src/send-result.ts:21-30`
  - body: Adding `viaReplyPath` to `toFeishuSendResult` makes every send return an internal tracking flag. Current main has exact result-shape coverage for `sendMessageFeishu` returning only `{ messageId, chatId }`, so keep this state local to the private reply helper instead of leaking it through `FeishuSendResult`.
  - confidence: 0.9
- **[P2] Check recall before falling back to direct sends:** `extensions/feishu/src/send.ts:467-481`
  - body: The PR exports `isMessageRecalled`, but `sendMessageFeishu` and `sendCardFeishu` still call `sendReplyOrFallbackDirect` first. When Feishu reports the parent as withdrawn/not found because the user recalled it, this path can still create a top-level message, so the recalled-before-response acceptance criterion is not implemented.
  - confidence: 0.86
- **[P2] Treat deleted message-get items as recalled:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: The Feishu SDK type for `im.message.get` includes `data.items[].deleted`, but this helper returns `false` for every successful response. A recalled/deleted message item with `deleted: true` would pass the preflight once the caller is wired.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `CHANGELOG.md:1`
  - body: This is a user-facing Feishu feature, but the PR head has no `CHANGELOG.md` entry for recalled-message handling. Repo policy requires `feat` changes to be recorded before merge.
  - confidence: 0.9
