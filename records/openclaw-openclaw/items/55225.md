## Summary

Keep open: current main still lacks Feishu recall handling, and this PR is useful but not mergeable because the current patch leaks an internal send flag, misses recalled-before-response suppression, and does not handle `deleted: true` message-get responses.

## What This Changes

This PR adds Feishu recalled-message event handling, in-memory user-to-bot reply tracking, outbound reply registration, and reply-path marking on Feishu send results.

## Best Possible Solution

Implement recall handling inside the Feishu plugin with private reply-path tracking, pre-response recall suppression at the right dispatch boundary, `deleted: true` preflight handling, regression tests for before/after recall cases, and a changelog entry.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR calls the existing withdrawn/not-found fallback before any recall preflight, and current exact-result tests would catch the added `viaReplyPath` property. I did not run tests because this was a read-only review.

## Solution Assessment

No. The event wiring belongs in the Feishu plugin, but the patch should keep reply tracking out of public send results and decide recalled-before-response behavior before the direct-send fallback runs.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

  - body: Adding `viaReplyPath` to `toFeishuSendResult` makes send calls return an extra enumerable property. Current `sendMessageFeishu` coverage expects exactly `{ messageId, chatId }`, so this leaks an internal tracking bit into callers/tests; keep the flag private to the reply helper or a registration callback.
  - confidence: 0.9
- **[P2] Check recall before falling back to direct sends:** `extensions/feishu/src/send.ts:467-481`
  - body: The PR registers the bot reply only after `sendReplyOrFallbackDirect` returns. If the user recalls before the bot responds, the existing withdrawn/not-found branch can still create a top-level fallback message, so the first acceptance criterion is not implemented.
  - confidence: 0.86
- **[P2] Treat deleted message-get responses as recalled:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: `isMessageRecalled` returns `false` for every successful `message.get` response. Feishu's message-get response includes `deleted` for recalled or deleted messages, so a `code: 0` response with `deleted: true` would still be treated as present.
  - confidence: 0.84

  - body: This is a user-facing Feishu feature, and the branch changes only Feishu source files. Repo policy requires a `CHANGELOG.md` entry for `feat` changes before merge.
