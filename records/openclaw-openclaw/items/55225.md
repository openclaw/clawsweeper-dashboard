## Summary

Keep open: current main still lacks Feishu recalled-message cleanup, and this PR targets the right Feishu plugin surface, but the branch still has correctness blockers and no real Feishu after-fix proof.

## What This Changes

The PR adds Feishu `im.message.recalled_v1` event handling, in-memory user-to-bot reply tracking, outbound reply registration, and a `viaReplyPath` send-result flag.

## Best Possible Solution

Implement recall handling inside the Feishu plugin with private reply tracking, `im.message.recalled_v1` handling, pre-send recall suppression before direct fallback, `deleted: true` handling, idempotent deletion, focused tests, and real Feishu proof.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR sends or falls back before any recall preflight, exposes internal reply tracking through the send-result shape, and ignores Feishu's `deleted` message state. No live Feishu reproduction was provided.

## Solution Assessment

No. The event wiring belongs in the Feishu plugin, but the patch should keep reply tracking private, preserve current send receipts, check recall before fallback delivery, handle `deleted: true`, and provide real Feishu proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep reply tracking out of send results:** `extensions/feishu/src/send-result.ts:21-30`
  - body: Adding `viaReplyPath` to `toFeishuSendResult` makes an internal tracking bit part of the public Feishu send result and conflicts with the current `{ messageId, chatId, receipt }` contract. Keep this state private in the reply helper instead of adding an enumerable result field or replacing the receipt-kind argument.
  - confidence: 0.9
- **[P2] Check recall before falling back to direct sends:** `extensions/feishu/src/send.ts:467-480`
  - body: `sendMessageFeishu` waits for `sendReplyOrFallbackDirect` before any recall preflight, so a recalled parent can still take the withdrawn/not-found branch and create a top-level fallback message. That misses the stated requirement to skip a reply when the user recalls before the bot responds.
  - confidence: 0.87
- **[P2] Treat deleted message-get responses as recalled:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: `isMessageRecalled` returns `false` for every successful `message.get` response. Feishu message-get items include `deleted`, so a recalled message can still be treated as present when the API returns `code: 0` with a deleted item.
  - confidence: 0.85
