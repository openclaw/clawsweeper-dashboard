## Best Possible Solution

Implement inbound Feishu recall support in the Feishu plugin as an additive change that preserves send receipts, suppresses recalled or deleted parent replies before fallback, handles Feishu deleted flags, and coordinates with https://github.com/openclaw/openclaw/issues/55691 and https://github.com/openclaw/openclaw/pull/83517.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR drops the current receipt return, sends before any recall preflight, and ignores the SDK deleted field. No live Feishu after-fix behavior proof is attached.

## Solution Assessment

No. The feature belongs in the Feishu plugin, but this patch should preserve current send receipts, check recalled or deleted state before fallback delivery, handle the SDK deleted field, and include live Feishu proof.

  - body: Current main returns a durable `receipt` from `toFeishuSendResult`, and Feishu tests assert that shape. This branch replaces that contract with `viaReplyPath`, so keep recall bookkeeping additive or private while still returning the receipt callers expect.

  - body: The pinned Feishu SDK's `message.get` response can return `data.items[].deleted`, but this helper returns `false` for every zero-code response. A recalled message returned as deleted would still be treated as live, so pre-send suppression can miss it.
  - confidence: 0.86
