## Summary

Keep open: current main still does not implement Feishu recall cleanup, but this PR is not ready because the branch still has source-visible correctness blockers, lacks real Feishu proof, and conflicts with current main's Feishu send-result receipt contract.

## What This Changes

The PR adds a Feishu `im.message.recalled_v1` handler, in-memory user-message to bot-reply tracking, outbound reply registration, and a `viaReplyPath` flag on Feishu send results.

## Best Possible Solution

Implement Feishu recall support inside the plugin with private reply tracking, `im.message.recalled_v1` handling, pre-send suppression before fallback delivery, deleted-flag handling, focused tests, and real Feishu proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: the branch sends before recall preflight, exposes internal reply tracking in the send-result shape, and ignores `data.items[].deleted` on successful message-get responses. No live Feishu reproduction or after-fix proof is attached.

## Solution Assessment

No. The feature belongs in the Feishu plugin, but this patch should preserve the current receipt contract, keep recall bookkeeping private, check recall before fallback delivery, handle deleted message-get results, and include real Feishu proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep recall bookkeeping out of send results:** `extensions/feishu/src/send-result.ts:21-30`
  - body: Adding `viaReplyPath` makes internal recall state an enumerable part of the Feishu send result, and this branch is stale against current main's receipt-bearing `FeishuSendResult` contract. Reply tracking should stay private to the send helper instead of changing the outbound result shape.
  - confidence: 0.9
- **[P2] Check recall before falling back to a direct send:** `extensions/feishu/src/send.ts:467-479`
  - body: `sendMessageFeishu` sends through `sendReplyOrFallbackDirect` before any recall preflight. If the parent was recalled before the bot responds, that helper can still create a top-level fallback message, violating the stated requirement to skip replies to recalled messages.
  - confidence: 0.88
- **[P2] Honor deleted message-get responses:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: The SDK's `message.get` response can include `data.items[].deleted`, but this code returns `false` for every successful response. A recalled message returned as deleted would still be treated as live, so pre-send suppression can miss it.
  - confidence: 0.86
