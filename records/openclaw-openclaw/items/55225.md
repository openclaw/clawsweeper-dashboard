## Summary

Keep open and do not merge as-is. Current main still lacks Feishu recall handling, and the PR covers the recalled event hook but leaves recalled-before-reply suppression and non-text/card reply outputs incomplete.

## What This Changes

The PR adds a Feishu `im.message.recalled_v1` handler, an in-memory user-message to bot-reply map, send-result metadata, and text/card reply registration for deleting bot replies after recalls.

## Best Possible Solution

Revise the Feishu plugin so recalled user messages become a shared reply-boundary state across text, card, media, and streaming outputs, with tests, setup docs, and an active-version changelog entry.

## Reproduction Assessment

Yes. A focused unit path can mock Feishu returning withdrawn/not-found for a reply target and assert no fallback direct send after recall; source inspection also shows media and streaming replies use `im.message.reply` without recall registration.

## Solution Assessment

No. The feature belongs in the Feishu plugin, but this PR solves only part of the reply surface; centralizing suppression and registration around the shared Feishu send/reply paths is the narrower maintainable fix.
