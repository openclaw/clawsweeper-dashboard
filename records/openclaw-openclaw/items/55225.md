## Summary

Keep open: the Feishu recall feature is plausible and belongs in the plugin, but this PR is not merge-ready because it is stale against current main's send-result contract, misses the pre-send recall suppression path, and lacks real Feishu behavior proof.

## What This Changes

The PR adds a Feishu `im.message.recalled_v1` handler, in-memory user-message to bot-reply tracking, outbound reply registration, and a `viaReplyPath` flag on Feishu send results.

## Best Possible Solution

Implement Feishu recall support inside the plugin with private reply tracking, `im.message.recalled_v1` handling, pre-send suppression before fallback delivery, deleted-flag handling, focused tests, and real Feishu proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR sends before recall preflight, exposes internal reply tracking in the send-result shape, and ignores `data.items[].deleted` on successful message-get responses. No live Feishu reproduction or after-fix proof is attached.

## Solution Assessment

No. The feature belongs in the Feishu plugin, but this patch should preserve the current receipt contract, keep recall bookkeeping private, check recall before fallback delivery, handle deleted message-get results, and include real Feishu proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep recall bookkeeping out of send results:** `extensions/feishu/src/send-result.ts:21-30`
  - body: The branch changes `toFeishuSendResult` to return `viaReplyPath` and omits current main's required `receipt` field. Reply-path tracking should stay private to the send helper so the Feishu outbound result contract remains compatible.
  - confidence: 0.9
- **[P2] Check recall before sending fallback replies:** `extensions/feishu/src/send.ts:467-479`
  - body: `sendMessageFeishu` sends through `sendReplyOrFallbackDirect` before any recall check, and that helper can create a top-level fallback when the parent was already withdrawn. That violates the stated requirement to skip a response when the user recalls before the bot responds.
  - confidence: 0.88
- **[P2] Honor deleted message-get responses:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: The Feishu SDK's `message.get` response can include `data.items[].deleted`, but this path returns `false` for every successful response. A recalled message returned as deleted would still be treated as live, so pre-send suppression can miss it.
  - confidence: 0.86
