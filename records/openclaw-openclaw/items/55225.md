## Summary

Keep open. Current main still lacks Feishu recalled-message cleanup, but this PR should not merge as-is because the proposed implementation can still send fallback replies after a recall, misses non-text reply outputs, and lacks the required changelog entry.

## What This Changes

The PR adds Feishu recalled-message handling with an in-memory user-message to bot-reply map, a recalled event handler, reply-path send metadata, and text/card reply registration.

## Best Possible Solution

Revise the Feishu plugin so recalled user messages are treated as a reply-boundary state across text, card, media, and streaming outputs, with focused tests and an active-version changelog entry.

## Reproduction Assessment

Yes. A focused unit path can mock Feishu returning withdrawn/not-found for a reply target and show the PR still reaches the existing fallback direct-send behavior; source inspection also shows media and streaming replies never enter the recall map.

## Solution Assessment

No. The feature belongs in the Feishu plugin, but the current PR is incomplete; the safer fix is to centralize recall suppression and reply-result registration across every Feishu reply output path.
