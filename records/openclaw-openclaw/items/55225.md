## Summary

Keep open. Current main does not implement Feishu recall handling, and this PR is useful source work, but it is not mergeable yet because it still allows replies after pre-response recalls, misses media reply cleanup, and omits the operator docs/changelog work needed for the new event subscription.

## What This Changes

The PR adds a Feishu recall-handler module, registers `im.message.recalled_v1`, and records text/card reply IDs so bot replies can be deleted when a source message is recalled.

## Best Possible Solution

Finish the Feishu-owned implementation by wiring recall awareness into reply dispatch/send paths, tracking all visible bot reply IDs, and documenting the new event subscription before merge.

## Reproduction Assessment

Yes. The source-level reproduction is the existing withdrawn-parent send path: a 230011 reply failure still falls back to a top-level send, while the PR only defines `isMessageRecalled` and never uses it to suppress that reply.

## Solution Assessment

No. The direction is right, but the maintainable fix should make recall state part of Feishu reply dispatch for pre-send cancellation and should share reply-ID registration across text, card, streaming, and media delivery.
