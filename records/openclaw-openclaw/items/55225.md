## Summary

Keep this PR open, but it is not merge-ready: the branch addresses a real Feishu recall gap, yet it still leaks internal reply tracking into public send results, can still send fallback replies after a recalled input, misses `deleted: true` in message-get preflight logic, and has no after-fix real Feishu proof.

## What This Changes

The PR adds Feishu recalled-message event handling, in-memory user-to-bot reply mapping, outbound reply registration, and reply-path metadata on Feishu send results.

## Best Possible Solution

Implement recall handling inside the Feishu plugin with private reply-path tracking, `im.message.recalled_v1` handling, pre-send recall suppression before direct fallback, `deleted: true` handling, idempotent bot-reply deletion, focused tests, and real Feishu proof.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR sends or falls back before any recall preflight, exposes an extra send-result property that current exact-result tests would reject, and ignores Feishu's documented `deleted: true` state. I did not run live Feishu because this is a read-only review and no credentials/proof were provided.

## Solution Assessment

No. The event wiring belongs in the Feishu plugin, but the patch should keep reply tracking private, make the recalled-before-response policy explicit before fallback delivery, handle successful deleted message-get responses, and prove the real Feishu flow before merge.

- **[P2] Keep reply tracking out of send results:** `extensions/feishu/src/send-result.ts:21-30`
  - body: Adding `viaReplyPath` to `toFeishuSendResult` makes an internal tracking bit part of the public send result.   - confidence: 0.87

  - body: `isMessageRecalled` returns `false` for every successful `message.get` response. Feishu message-get items include `deleted`, and the docs define `deleted: true` as recalled or deleted, so a recalled message can still be treated as present when the API returns `code: 0`.
  - confidence: 0.85
