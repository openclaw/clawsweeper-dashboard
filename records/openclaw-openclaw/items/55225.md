## Summary

Keep open: current main still lacks Feishu recall support, but this branch remains incorrect because recall-before-response can still fall back to a direct send, deleted message-get responses are treated as live, internal reply tracking is exposed in send results, and no real Feishu proof is attached.

## What This Changes

The PR adds Feishu `im.message.recalled_v1` handling, in-memory user-to-bot reply tracking, outbound reply registration, and a `viaReplyPath` send-result flag.

## Best Possible Solution

Implement Feishu recall support inside the plugin with private reply tracking, `im.message.recalled_v1` handling, pre-send suppression before fallback, deleted-flag handling, idempotent bot-reply deletion, focused tests, and real Feishu proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR sends or falls back before recall preflight, exposes `viaReplyPath`, and returns false on code-0 message-get responses without checking `data.items.deleted`. No live Feishu reproduction is attached.

## Solution Assessment

No. The event wiring belongs in the Feishu plugin, but the patch should keep reply tracking private, preserve the current receipt contract, check recall before top-level fallback delivery, handle deleted get results, and provide real Feishu proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep reply tracking out of send results:** `extensions/feishu/src/send-result.ts:21-30`
  - body: Adding `viaReplyPath` makes internal recall bookkeeping an enumerable part of the Feishu send result. Current main builds receipt-bearing send results and tests assert exact result shapes, so this state should stay private to the send helper instead of changing the result contract.
  - confidence: 0.9
- **[P2] Check recall before falling back to direct sends:** `extensions/feishu/src/send.ts:467-480`
  - body: `sendMessageFeishu` calls `sendReplyOrFallbackDirect` before any recall preflight. If the parent was recalled or withdrawn before the bot responds, that helper can create a top-level fallback message, violating the stated requirement to skip the reply.
  - confidence: 0.87
- **[P2] Treat deleted message-get responses as recalled:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: Feishu `message.get` returns `data.items.deleted`, but this code returns false for every successful response. A recalled message returned as deleted will still be treated as live, so pre-send suppression can miss it.
  - confidence: 0.85
