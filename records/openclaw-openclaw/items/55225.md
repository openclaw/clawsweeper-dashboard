## Summary

Keep open: current main still lacks Feishu recall cleanup, but this PR has blocking implementation issues against the current Feishu send contract and still lacks real Feishu proof.

## What This Changes

The PR adds Feishu message-recall event handling, in-memory user-message to bot-reply tracking, bot-reply deletion on recall, and a `viaReplyPath` send-result flag.

## Best Possible Solution

Implement Feishu recall support inside the plugin with private reply tracking, receipt-preserving sends, pre-send suppression, deleted-state handling, focused tests, live Feishu proof, and coordination with related Feishu recall/delete work.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR drops the current receipt contract, sends before recall preflight, and ignores `data.items[].deleted`. No live Feishu after-fix reproduction proof is attached.

## Solution Assessment

No. The feature belongs in the Feishu plugin, but this patch should preserve the current send-result contract, keep recall bookkeeping private, check recall before fallback delivery, handle deleted message-get results, and include live Feishu proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the Feishu send receipt contract:** `extensions/feishu/src/send-result.ts:21-25`
  - body: Current main requires `FeishuSendResult.receipt`, but this branch changes `toFeishuSendResult` to return `viaReplyPath` and omits the receipt. Keep recall-path bookkeeping private so callers still receive the durable send receipt they expect.
  - confidence: 0.9
- **[P2] Check recall before fallback delivery:** `extensions/feishu/src/send.ts:467-479`
  - body: `sendMessageFeishu` sends through `sendReplyOrFallbackDirect` before consulting `isMessageRecalled`, and that helper can fall back to a top-level send when the parent was already withdrawn. That violates the stated requirement to skip replies when the user recalls before the bot responds.
  - confidence: 0.88
- **[P2] Honor deleted message-get responses:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: The pinned Feishu SDK's `message.get` response can include `data.items[].deleted`, but this path returns `false` for every successful response. A recalled message returned as deleted would still be treated as live, so pre-send suppression can miss it.
  - confidence: 0.86
