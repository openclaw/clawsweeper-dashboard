## Summary

Keep open. Current main still lacks Feishu recall handling, and this PR targets the right plugin surface but remains incomplete: recalled reply targets can still fall back to direct sends, media/streaming replies are not mapped for deletion, and the user-facing feature lacks a changelog entry.

## What This Changes

The PR adds a Feishu recalled-message handler, an in-memory user-to-bot reply map, reply-path send metadata, and text/card reply registration for deleting bot replies after `im.message.recalled_v1`.

## Best Possible Solution

Land a Feishu-local implementation that treats recall as a reply-boundary state for text, cards, media, and streaming outputs, with focused regression tests and an active-version changelog entry.

## Reproduction Assessment

Yes. A focused unit path can mock Feishu `im.message.reply` returning withdrawn/not-found and show the PR still reaches direct-send fallback; source inspection also shows media and streaming reply outputs are not entered into the recall map.

## Solution Assessment

No. The feature belongs in the Feishu plugin, but the current patch should centralize recall suppression and bot-reply registration across every Feishu reply-path output before merge.
