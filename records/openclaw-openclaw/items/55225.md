## Summary

Keep open and do not merge as-is. Current main still lacks Feishu recalled-message support, so the feature is useful, but this PR leaves the recalled-before-response path broken and leaks an internal reply-path flag into the public send result shape.

## What This Changes

This PR adds a Feishu recalled-message event handler, in-memory user-message to bot-reply tracking, and a reply-path marker on Feishu send results for deleting bot replies after user recalls.

## Best Possible Solution

Implement recalled-message handling as Feishu-owned behavior with tests for both after-response cleanup and before-response suppression, while preserving the existing generic withdrawn-target fallback where maintainers still want it.

## Reproduction Assessment

Yes. A focused unit path can mock Feishu reply returning withdrawn/not-found and show the PR still reaches direct-send fallback; current-main tests also give a high-confidence shape regression for the added `viaReplyPath` property.

## Solution Assessment

No. The event wiring is in the right area, but the proposed implementation does not actually use a correct pre-reply recall check and changes the public send result contract for an internal tracking detail.

## Review Findings

Full review comments:

- **[P2] Keep the reply-path flag out of send results:** `extensions/feishu/src/send-result.ts:25-30`
  - body: Adding `viaReplyPath` to every `toFeishuSendResult` return changes the observable `FeishuSendResult` shape. Current main has exact equality coverage for `sendMessageFeishu` returning only `{ messageId, chatId }`, so this will fail or leak an internal tracking flag; keep that flag local to the private reply helper instead.
  - confidence: 0.9
- **[P2] Wire the pre-reply recall check before fallback:** `extensions/feishu/src/send.ts:467-481`
  - body: The PR exports `isMessageRecalled`, but `sendMessageFeishu` and `sendCardFeishu` still call `sendReplyOrFallbackDirect` first. When the user recalls before the bot responds, the withdrawn/not-found branch can still create a top-level fallback reply, so the first acceptance criterion is not implemented.
  - confidence: 0.86
- **[P2] Handle deleted messages in recall preflight:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: Feishu's message-get API can return `code: 0` with a message item marked `deleted: true` for recalled or deleted messages. This helper returns `false` for every successful response, so even if the preflight is wired later it will still treat recalled messages as present.
  - confidence: 0.82

  - body: This is a user-facing Feishu feature, but the branch does not add a `CHANGELOG.md` entry. Repo policy requires `feat` changes to be recorded before merge.
