## What This Changes

The branch adds Feishu `im.message.recalled_v1` handling, an in-memory user-message to bot-reply map, recall-time bot-reply deletion, and a `viaReplyPath` flag on Feishu send results.

## Best Possible Solution

Implement inbound Feishu recall as additive Feishu plugin behavior that preserves send receipts, suppresses recalled or deleted parent replies before fallback delivery, handles Feishu `deleted` fields, and coordinates with the open recall-cancellation and message-management work.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR drops the current receipt return, sends before any recall preflight, and ignores the SDK `deleted` field. No live Feishu after-fix behavior proof is attached.

## Solution Assessment

No. The Feishu plugin is the right boundary, but this patch should preserve current send receipts, check recalled or deleted state before fallback delivery, handle the SDK `deleted` field, and include live Feishu proof.
