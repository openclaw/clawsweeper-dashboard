## Summary

Keep this PR open: current main still lacks Feishu recalled-message handling, and the branch is useful, but the proposed patch still has blocking correctness issues before merge.

## What This Changes

This PR adds Feishu recalled-message event handling, in-memory user-to-bot reply tracking, outbound reply registration, and a reply-path marker on Feishu send results.

## Best Possible Solution

Implement recall handling fully inside the Feishu plugin with private reply tracking, pre-response recall suppression at the right dispatch boundary, deleted-message response handling, streaming-card coverage, regression tests, and a changelog entry.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR changes the public send-result shape, checks recall only after the existing fallback path can send, and ignores Feishu's deleted message-get response state. I did not run tests because this is a read-only review and pnpm is unavailable in the environment.

## Solution Assessment

No. The Feishu-owned recall direction is appropriate, but this patch should keep tracking state private, check recalled-before-response behavior before direct-send fallback, handle deleted message-get responses, and cover streaming-card replies before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep reply tracking out of send results:** `extensions/feishu/src/send-result.ts:21-30`
  - body: Adding viaReplyPath to toFeishuSendResult makes send calls return an extra enumerable property. Current sendMessageFeishu coverage expects exactly { messageId, chatId }, so this leaks an internal tracking bit into callers/tests; keep the flag private to the reply helper or a registration callback.
  - confidence: 0.9
- **[P2] Check recall before direct-send fallback:** `extensions/feishu/src/send.ts:467-481`
  - body: The branch registers the bot reply only after sendReplyOrFallbackDirect returns. If the user recalls before the bot responds, the existing withdrawn/not-found branch can still create a top-level fallback message, so the skip-before-response acceptance criterion is not implemented.
  - confidence: 0.87
- **[P2] Honor deleted message-get responses:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: isMessageRecalled returns false for every successful message.get response. Feishu's message-get response includes deleted for recalled or deleted messages, so a code 0 response with deleted: true would still be treated as present.
  - confidence: 0.86
- **[P2] Track streaming-card replies:** `extensions/feishu/src/send.ts:473-479`
  - body: Registering only in sendMessageFeishu/sendCardFeishu misses FeishuStreamingSession.start, which sends interactive cards through message.reply in the default streaming/card path and keeps the messageId private. A recalled source message can therefore leave a streaming-card bot reply orphaned.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `CHANGELOG.md:1`
  - body: This is a user-facing Feishu feature, but the branch changes only Feishu source files and does not update CHANGELOG.md. Repo policy requires a changelog entry for feat changes before merge.
  - confidence: 0.9
