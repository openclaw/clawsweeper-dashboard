## Summary

Keep this PR open, but it is not merge-ready: current main still lacks Feishu recall handling, and the branch addresses the right plugin-owned surface while leaving several correctness blockers in the proposed implementation.

## What This Changes

This PR adds Feishu recalled-message event handling with an in-memory user-message to bot-reply map and registers reply-path send results for recall cleanup.

## Best Possible Solution

Implement Feishu-owned recall support with private reply tracking at the dispatch/send boundaries, cover recalled-before and recalled-after response cases including media and streaming replies, and add the required changelog entry.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the PR leaks viaReplyPath into exact send results, checks recall too late to prevent fallback sends, misses deleted=true message-get payloads, and leaves media/streaming reply messages untracked. I did not run tests because this was a read-only review and source evidence was sufficient.

## Solution Assessment

No. The Feishu plugin is the right place for the feature, but the proposed implementation should keep tracking state private, suppress recalled-parent replies before fallback sends, and register every reply-path bot message that recall cleanup must delete.

  - confidence: 0.86

  - confidence: 0.84

  - body: The new mapping is only written from sendMessageFeishu and sendCardFeishu. Feishu replies can also be sent through sendMediaFeishu and FeishuStreamingSession.start, so recall cleanup would leave those bot reply messages orphaned.
