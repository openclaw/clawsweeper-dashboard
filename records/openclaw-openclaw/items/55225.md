## Summary

Keep open: the Feishu recall feature is useful and is not on current main, but this PR is not mergeable as-is because it conflicts with the current send-result receipt contract, sends before recall preflight, ignores deleted message-get responses, and lacks real Feishu proof.

## What This Changes

The PR adds Feishu `im.message.recalled_v1` handling, in-memory user-message to bot-reply tracking, outbound reply registration, and a `viaReplyPath` flag on Feishu send results.

## Best Possible Solution

Implement Feishu recall support inside the plugin with private reply tracking, current receipt-preserving send results, pre-send suppression before fallback delivery, deleted-flag handling, focused tests, and live Feishu proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR sends before recall preflight, exposes recall tracking through the send-result shape, and ignores `data.items[].deleted`. No live Feishu reproduction or after-fix proof is attached.

## Solution Assessment

No. The feature belongs in the Feishu plugin, but this patch should preserve the current receipt contract, keep recall bookkeeping private, check recall before fallback delivery, handle deleted message-get results, and include real Feishu proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the Feishu send receipt contract:** `extensions/feishu/src/send-result.ts:21-25`
  - body: Current main requires `FeishuSendResult.receipt`, but this branch changes `toFeishuSendResult` to return `viaReplyPath` and omits the receipt. Keep recall-path bookkeeping private so callers still receive the durable send receipt they expect.
  - confidence: 0.9
- **[P2] Check recall before sending fallback replies:** `extensions/feishu/src/send.ts:467-479`
  - body: `sendMessageFeishu` sends through `sendReplyOrFallbackDirect` before consulting `isMessageRecalled`, and that helper can fall back to a top-level send when the parent was already withdrawn. That violates the stated requirement to skip replies when the user recalls before the bot responds.
  - confidence: 0.88
- **[P2] Honor deleted message-get responses:** `extensions/feishu/src/recall-handler.ts:186-192`
  - body: The Feishu SDK's `message.get` response can include `data.items[].deleted`, but this path returns `false` for every successful response. A recalled message returned as deleted would still be treated as live, so pre-send suppression can miss it.
  - confidence: 0.86
