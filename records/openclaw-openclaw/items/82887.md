## Label Justifications

- `P1`: The PR fixes a broken scheduled-agent workflow where configured model failover can be bypassed and cron runs can be skipped before the model turn starts.
- `merge-risk: 🚨 auth-provider`: Merging changes cron provider selection before execution, so configured fallbacks may switch the run to a different provider/auth path when local preflight fails.

## Summary

Keep open for merge review: current main still skips an isolated cron run as soon as local model preflight fails, while this PR moves cron preflight onto the shared fallback candidate chain, preserves strict `fallbacks: []`, and includes focused regression coverage plus real log proof. I found no blocking correctness or security findings, but the provider/auth fallback behavior is merge-risk-worthy and should be explicitly accepted before landing.

## What This Changes

The PR exports shared model fallback candidate resolution and updates isolated cron runtime, executor handoff, docs, and tests so local-provider preflight walks configured fallbacks before recording a skipped cron run.

## Best Possible Solution

Land the targeted fallback-chain implementation after maintainer acceptance of the provider-routing behavior, keeping explicit `fallbacks: []` strict and preserving the narrowed post-preflight fallback list.

## Reproduction Assessment

Yes. Current main has a source-clear reproduction path: configure an isolated cron job with an unreachable local primary and a configured fallback, then `prepareCronRunContext` returns `skipped` immediately after the primary preflight failure.

## Solution Assessment

Yes. Reusing the shared fallback candidate chain and passing a narrowed fallback override is the narrow maintainable fix because it aligns preflight ordering with runtime fallback while preserving strict empty fallback behavior.
