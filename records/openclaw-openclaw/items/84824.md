## Summary

Keep open: the PR addresses a real gateway checkpoint-retention bug with good filesystem-backed proof, but the current patch adds an uncached synchronous session-store read before every checkpoint persist in the same pressure-sensitive path.

## What This Changes

This PR adds a fixed 128 MiB per-session retained compaction-checkpoint byte budget, trims older checkpoint metadata/files when count or byte limits are exceeded, and adds regression coverage.

## Best Possible Solution

Keep the byte-budget retention, but compute snapshot sizes from the current writer-owned session entry without an extra synchronous store load, then land after maintainers accept the 128 MiB recovery-retention tradeoff.

No, not as-is. The retention direction is sound, but the implementation should avoid rereading and cloning the session store before the writer-owned update path.
