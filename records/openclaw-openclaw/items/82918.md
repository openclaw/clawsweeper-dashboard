## What This Changes

Adds a top-of-file pipe guard to `scripts/install.sh` that buffers piped installer stdin to a temp file and re-execs it, plus a changelog entry.

## Best Possible Solution

Keep the installer-side pipe guard, but only delete the buffered temp file when the marker matches the script path currently being executed and cover both piped-stdin protection and inherited-env safety in installer tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the full WSL2 installer hang was not reproduced here, but current docs still advertise the raw piped command and source inspection shows stdin-sensitive installer paths; the PR-specific env-var deletion defect was directly reproduced with the PR head script.

## Solution Assessment

Is this the best way to solve the issue? Not yet: an installer-side pipe guard is the right direction, but the cleanup must validate that the marker belongs to the current buffered re-exec before removing any file.
