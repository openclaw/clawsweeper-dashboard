## Summary

Keep open for maintainer review, not cleanup close: current main still documents and serves the raw `curl | bash` installer path without a pipe guard, and this PR targets a real stdin-stealing failure with a small installer-side fix plus terminal proof. The remaining merge question is compatibility coverage for the public installer startup path, especially because the new invocation harness is ad hoc rather than part of the existing test/workflow lanes.

## What This Changes

The PR adds a top-of-file pipe guard to `scripts/install.sh` and a shell invocation-matrix helper script for piped, direct, command-string, process-substitution, and `/dev/null` stdin cases.

## Best Possible Solution

Land the installer-side buffering approach after maintainers are satisfied that the invocation matrix is either wired into the existing installer tests/workflow or covered by explicit WSL2/macOS proof.

## Reproduction Assessment

No for a fresh live reproduction of the exact WSL2 hang in this review. The reporter supplied concrete 3/3 logs, current main still documents the vulnerable piped path, and source inspection shows child processes can inherit the script stream.

## Solution Assessment

Mostly yes: an installer-side guard is the right boundary because docs-only `bash -s --` does not protect child stdin. The safer landing path is to keep this guard narrow and make its invocation matrix part of maintained validation or explicit platform proof.
