## Summary

Keep this PR open: it addresses a real documented installer stdin-corruption path, but the current guard still treats `bash -c` command-string execution as piped script input, which can hang or no-op before the installer starts.

## What This Changes

Adds a top-of-file `scripts/install.sh` guard that buffers piped stdin into a temporary file and re-execs Bash before normal installer startup.

## Best Possible Solution

Land the pipe-buffer approach only after it distinguishes real stdin-script execution from command-string execution and locks the supported installer invocation matrix with focused tests.

## Reproduction Assessment

No high-confidence live WSL2 reproduction was run for the original report; current main still has the unbuffered documented pipe path and the linked logs match stdin-stream corruption, so the bug is source-reproducible with medium confidence. The PR-introduced `bash -c` compatibility defect is locally source-proven by Bash showing an empty `BASH_SOURCE[0]`.

## Solution Assessment

No, not as-is. The pipe-buffer/re-exec shape is a narrow maintainable fix, but the trigger needs to distinguish actual stdin-script execution from Bash command-string execution before merge.
