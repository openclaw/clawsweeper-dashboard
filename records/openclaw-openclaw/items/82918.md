## Summary

Keep this PR open: the installer bug is real and the buffering direction is useful, but the current guard still matches non-TTY Bash command-string execution and can skip the installer by re-execing an empty buffered file.

## What This Changes

The PR adds a top-of-file `scripts/install.sh` guard that buffers stdin-piped installer contents into a temporary file and re-execs Bash before normal installer startup.

## Best Possible Solution

Land the pipe-buffer approach after the guard excludes Bash command-string execution in non-TTY contexts and focused tests cover the installer invocation matrix.

## Reproduction Assessment

Yes for the PR-introduced compatibility bug: a local Bash probe shows non-TTY `bash -c` has empty `BASH_SOURCE[0]` and would satisfy the guard while `BASH_EXECUTION_STRING` distinguishes it. The original WSL2 stdin-stealing hang was not live-reproduced here, but current main’s documented unbuffered pipe path and the linked logs make it source-reproducible.

## Solution Assessment

No, not yet; the buffering strategy is plausible, but the current predicate needs to exclude command-string execution and should be backed by invocation-mode tests before merge.
