Keep open: the fix is narrow, but it currently repairs a helper that is not called by the live config read path, so the claimed runtime retry behavior remains unproven and likely unchanged.

## What This Changes

The PR gates async and sync suspicious-config-read dedup marker writes on `restoredFromBackup` and adds retry tests for a failed backup copy followed by a successful retry.

## Best Possible Solution

Align code and tests with the supported config recovery boundary: either wire retry-safe recovery into the live read path with production coverage, or remove/narrow the stale helper if that runtime recovery mode is no longer intended.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Only at the internal-helper level: current main source shows failed helper restores can record the suspicious signature, and the PR tests model that path. I do not see a high-confidence runtime reproduction because the helper is not called by current-main config reads.

## Solution Assessment

Is this the best way to solve the issue? No. Gating the marker on `restoredFromBackup` is correct inside the helper, but the best fix must address the actual runtime config recovery path or explicitly retire this stale helper/test surface.
