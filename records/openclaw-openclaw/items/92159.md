## Summary

Keep open: the linked media-tool bug is plausible, but this PR is not merge-ready because it can override an explicit user-configured text-only model input with bundled catalog image metadata.

## What This Changes

The PR adds a media-understanding fallback that copies bundled static-catalog image input metadata onto a resolved model before the image runtime rejects text-only models.

## Best Possible Solution

Fix capability merging in the model resolver or config-normalization path where omitted input can be distinguished from explicit text-only input, preserve explicit `['text']`, and add focused regression coverage plus real image/pdf tool proof.

## Reproduction Assessment

Yes. Source inspection shows inline provider entries can resolve before catalog metadata, missing input defaults to `['text']`, and current media runtime rejects text-only models; I did not run tests because this review is read-only.

## Solution Assessment

No. The image-runtime fallback is too broad because it cannot distinguish omitted input from an explicit text-only override; the maintainable fix should preserve configured input semantics while filling missing catalog metadata.

- **[P1] Preserve explicit text-only model input:** `src/media-understanding/image.ts:215`
  - body: When an inline provider entry explicitly sets `input: ['text']`, `resolveProviderModelInput` keeps that value and the docs describe it as how to mark a model text-only.
