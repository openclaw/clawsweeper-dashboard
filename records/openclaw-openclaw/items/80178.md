## Summary

This should stay open: current main still hashes a local Claude CLI credential part when the no-prompt local reader starts returning a file credential, then also hashes the auth-profile credential, so a storage-presence flip can change the epoch and invalidate live sessions even when the user identity is unchanged.

## What This Changes

The issue asks OpenClaw to keep Claude CLI session auth epochs stable when the local credential source appears or disappears but the auth-profile identity is unchanged.

## Best Possible Solution

Stabilize Claude CLI auth epochs across local credential storage-presence flips while preserving invalidation for real identity changes, preferably through the existing auth-epoch policy seam or a narrowly scoped resolver change with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: keep the same auth-profile credential, change the Claude local credential reader from `null` to an OAuth credential, and current main hashes a different part list that downstream maps to `invalidatedReason: "auth-epoch"`.

## Solution Assessment

Unclear for the exact proposed fix. Treating the auth profile as canonical whenever it exists matches the incident, but current code and tests intentionally distinguish `combined` from `profile-only`, so the safest solution should make that ownership decision explicit and covered by tests.
