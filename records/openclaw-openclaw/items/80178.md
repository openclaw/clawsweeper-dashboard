## Summary

This should stay open: current main still has the reported source-level path where a Claude local credential read changing from null to present changes the CLI auth epoch and causes live session invalidation. A related open PR appears to cover the null-read/file-presence side, but it is not merged and still needs auth-boundary review.

## What This Changes

Stabilize Claude CLI auth epochs when the local credential storage source appears or disappears but the selected auth profile identity is unchanged.

## Best Possible Solution

Land a narrow auth-epoch fix, likely by resolving the auth-state concern in https://github.com/openclaw/openclaw/pull/74493 or an equivalent patch, with regression coverage for local missing/present storage flips and real identity changes.

## Reproduction Assessment

Yes. Source inspection reproduces the path with high confidence: mock the Claude local credential reader from null to an identity-less OAuth credential while keeping the auth-profile credential unchanged, and the current resolver hashes different part lists that `resolveCliSessionReuse` treats as `auth-epoch` invalidation.

## Solution Assessment

Unclear for the issue's broad proposed resolver change. The safer direction is a narrowly scoped Claude auth-epoch fix or backend policy change that keeps storage-presence flips stable without weakening true account-change or credential-loss invalidation.
