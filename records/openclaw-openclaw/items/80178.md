## Summary

This should stay open: current main and the latest release still allow the Claude CLI local credential file's presence to change the auth epoch while the selected auth profile is unchanged, and an open linked PR already owns the proposed fix path.

## What This Changes

The issue asks OpenClaw to keep reusable Claude CLI sessions stable when local credential storage flips between missing and present without an actual auth identity change.

## Best Possible Solution

Land or replace the linked policy-seam fix so identity-less Claude CLI OAuth profiles do not include local credential file presence in the auth epoch, with regression tests and redacted live session-reuse proof.

## Reproduction Assessment

Yes. Mocking the Claude local credential reader from `null` to an OAuth credential while the auth-profile credential stays unchanged is enough to show different current-main epoch part lists, and `resolveCliSessionReuse` maps the mismatch to `invalidatedReason: "auth-epoch"`.

## Solution Assessment

Yes, the active linked PR is the right narrow path to resolve this issue. It should be reviewed and landed after conflict handling and redacted live Claude CLI proof, rather than closing this issue before the fix merges.
