## Summary

This should stay open: current main and the latest shipped tag still combine the local CLI credential fingerprint with the auth-profile epoch, so the reported storage-presence flip remains source-reproducible. The linked open fix PR at https://github.com/openclaw/openclaw/pull/80265 uses closing syntax for this issue, so the issue should track that PR until it is rebased, proven, and merged.

## What This Changes

The issue reports that `resolveCliAuthEpoch` invalidates live CLI sessions when local Claude CLI credential file presence changes even though the selected auth-profile identity is unchanged.

## Best Possible Solution

Review, rebase, and land the linked policy-seam fix or an equivalent patch that stabilizes identity-less Claude CLI auth epochs across local file-presence flips while preserving invalidation for real identity-bearing changes.

## Reproduction Assessment

Yes. Source inspection shows a deterministic reproduction by mocking the Claude local credential reader from `null` to an identity-less OAuth credential while the auth-profile credential stays unchanged; the reporter also provided live 2026.5.7 `reason=auth-epoch` reset logs.

## Solution Assessment

Yes at the source level: the linked PR's prepare/auth-epoch seam approach is the right narrow direction, but the issue should remain open until that PR or an equivalent fix is proven and merged.
