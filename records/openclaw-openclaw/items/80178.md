## Summary

Keep open: current main still lets the local CLI credential fingerprint and auth-profile credential both contribute to the epoch, so a local credential file-presence flip can still produce an `auth-epoch` invalidation. The open closing PR at https://github.com/openclaw/openclaw/pull/80265 is the active implementation path, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue reports spurious live CLI session invalidation when `resolveCliAuthEpoch` changes only because the local Claude credential file appears or disappears while the auth-profile identity is unchanged.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80265, or replace it with an equivalent maintainer-approved policy-seam fix that adds regression coverage and preserves real account-change invalidation.

## Reproduction Assessment

Yes. Source inspection shows the current resolver hashes different part lists when the local Claude credential reader flips between null and an OAuth credential while the auth-profile credential remains present, and `cli-session` maps that epoch mismatch to `invalidatedReason: "auth-epoch"`.

## Solution Assessment

Yes for keeping this issue open. The narrow implementation path is the open closing PR at https://github.com/openclaw/openclaw/pull/80265, but this issue should not close before that PR merges or a replacement fix lands.
