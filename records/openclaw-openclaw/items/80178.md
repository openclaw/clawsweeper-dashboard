## Label Justifications

- `P1`: The bug can invalidate every live CLI session for affected users and break an active agent workflow.
- `impact:session-state`: The reported failure abandons the live CLI session handle and loses continuity of the agent conversation state.
- `impact:auth-provider`: The root cause is auth epoch calculation across local CLI credentials and auth-profile credentials.

## Summary

Keep open: current main and the latest release still combine the local CLI credential fingerprint with the auth-profile credential, so a local credential source-presence flip can still produce an `auth-epoch` reset. The linked fix PR is closed and unmerged, so no current implementation owns the remaining bug.

## What This Changes

Stabilize CLI auth epoch calculation so identity-less OAuth profile sessions are not invalidated when the local CLI credential file appears or disappears.

## Best Possible Solution

Add a focused auth-epoch policy fix that skips the local fingerprint only for selected identity-less OAuth auth profiles, while preserving bootstrap and identity-bearing invalidation with regression coverage.

## Reproduction Assessment

Yes, by source inspection: mock the local credential reader from null to an identity-less OAuth credential while the same auth profile is selected, and `resolveCliAuthEpoch` will change because current main includes both local and profile parts. That mismatch then reaches `resolveCliSessionReuse` as `invalidatedReason: "auth-epoch"`.

## Solution Assessment

Unclear without maintainer policy review; the narrowest likely fix is the linked-PR direction of moving the exception into `shouldSkipLocalCliCredentialEpoch` for identity-less OAuth profiles. A broader profile-always-owns-identity change would be riskier because combined-mode identity-bearing behavior is intentional today.
