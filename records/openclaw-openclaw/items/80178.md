## Summary

This should stay open: current `main` and shipped `v2026.5.7` still allow a local Claude CLI credential file-presence flip to change the combined auth epoch, and two open PRs explicitly close this issue but have not landed.

## What This Changes

The issue asks OpenClaw to keep CLI auth epochs stable when the local Claude CLI credential storage source appears or disappears while the selected auth-profile identity is unchanged.

## Best Possible Solution

Review and land the narrower policy-seam fix that stabilizes identity-less Claude CLI auth epochs while preserving invalidation for real account/profile changes, then close this issue after merge.

## Reproduction Assessment

Yes. Mocking the Claude local credential reader from `null` to an OAuth credential while the auth-profile credential stays unchanged will change the current resolver's part list, and the session reuse path maps that mismatch to `invalidatedReason: "auth-epoch"`.

## Solution Assessment

No for closing the issue now, and unclear for the broad resolver-level proposal. The maintainable fix should use the existing auth-epoch ownership seam or an explicit maintainer-approved contract change, with focused regression coverage and live proof.
