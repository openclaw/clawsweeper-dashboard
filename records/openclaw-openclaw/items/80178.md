## Summary

Keep open: current main and the latest release still let Claude CLI local credential presence add or remove the `local:` auth-epoch part while the selected profile credential remains, so the reported same-identity session reset path is still present. The direct fix PRs are closed unmerged, and the open related PR is an adjacent auth-profile rotation design thread rather than a completed fix for this source-of-storage flip.

## What This Changes

The issue asks OpenClaw to keep CLI auth epochs stable when Claude CLI local credential storage presence changes but the selected OAuth identity is unchanged.

## Best Possible Solution

Land a focused auth/session policy fix that keeps unchanged Claude CLI identity from resetting live sessions across local source-presence flips while preserving resets for logout, real account changes, and unprovable identity changes.

## Reproduction Assessment

Yes. Source inspection shows current main still changes the auth-epoch part set when the Claude CLI local credential read flips between present and absent, and `resolveCliSessionReuse` invalidates same-version epoch mismatches; I did not run live Claude CLI credentials in this read-only review.

## Solution Assessment

Unclear. The requested direction is plausible and narrow, but maintainers still need to choose the Claude CLI identity boundary so unchanged storage presence does not reset sessions while real or unprovable auth changes still do.
