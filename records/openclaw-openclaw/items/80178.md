## Summary

This should stay open: current main still combines the local CLI credential fingerprint with the stored auth-profile credential, so a local file-presence flip can change the auth epoch and invalidate reusable CLI sessions even when the profile identity has not changed.

## What This Changes

The issue reports that CLI auth epoch calculation invalidates live sessions when a local CLI credential file appears or disappears even though the auth profile identity is unchanged.

## Best Possible Solution

Stabilize the Claude CLI auth epoch across local credential storage-presence flips when the auth profile identity is unchanged, while preserving invalidation for real account/profile changes and adding focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible with high confidence: mock the Claude local credential reader from `null` to an OAuth credential while the auth-profile credential stays unchanged, and the current resolver will hash different part lists. The downstream session-reuse path then maps that epoch mismatch to `invalidatedReason: "auth-epoch"`.

## Solution Assessment

Unclear for the exact broad proposed fix because the current contract intentionally distinguishes combined host+profile epochs from profile-owned epochs. The safest solution should use that seam or a narrowly scoped resolver change, with tests proving storage-presence flips stay stable and real identity changes still invalidate.
