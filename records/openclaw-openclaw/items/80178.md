## Label Justifications

- `P1`: The reported auth-epoch reset can break live CLI session reuse for users and make an active agent lose usable conversation continuity.
- `impact:session-state`: The failure mode invalidates stored CLI session bindings and replaces the live session handle.
- `impact:auth-provider`: The trigger is auth epoch calculation across local CLI credentials and OpenClaw auth profiles.

## Summary

Keep open: current main and the latest release still let Claude CLI local credential presence add or remove the `local:` auth-epoch part while the selected auth-profile part remains, so the reported same-identity source-of-storage flip can still invalidate live CLI sessions. The related implementation attempts are either closed unmerged, merged for adjacent token-rotation behavior only, or still open with maintainer policy questions.

## What This Changes

The issue asks OpenClaw to keep CLI session auth epochs stable when local credential storage appears or disappears but the authenticated identity is unchanged.

## Best Possible Solution

Land a focused auth/session policy fix that keeps unchanged Claude CLI identity from resetting live sessions across local source-presence flips, while preserving resets for logout, real account changes, and unprovable identity changes.

## Reproduction Assessment

Yes. Source inspection shows local credential presence still changes the auth-epoch parts while `resolveCliSessionReuse` invalidates same-version epoch mismatches; I did not run live Claude CLI credentials in this read-only review.

## Solution Assessment

Unclear. The requested direction is plausible and narrow, but maintainers still need to choose the Claude CLI identity boundary so unchanged storage presence does not reset sessions while real or unprovable auth changes still do.
