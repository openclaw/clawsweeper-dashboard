## Summary

Keep this open: current main still lets a Claude CLI local credential file-presence change alter the combined auth epoch while the selected auth profile is unchanged, and that mismatch invalidates reusable CLI sessions.

## What This Changes

The issue asks OpenClaw to keep Claude CLI session auth epochs stable when the local credential storage source appears or disappears but the selected auth-profile identity is unchanged.

## Best Possible Solution

Stabilize Claude CLI auth epochs across local credential storage-presence flips when the selected auth-profile identity is unchanged, while preserving invalidation for real account/profile changes and adding focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible with high confidence: mock the Claude local credential reader from `null` to an OAuth credential while the auth-profile credential stays constant, and current `resolveCliAuthEpoch` hashes different part lists. The downstream `resolveCliSessionReuse` path then returns `invalidatedReason: "auth-epoch"`.

## Solution Assessment

Unclear for the exact proposed fix: making every profile credential canonical would collide with existing combined-epoch behavior for other backends. The safer solution is a narrow Claude/auth-profile policy change or seam adjustment with tests for storage-presence stability and real identity invalidation.
