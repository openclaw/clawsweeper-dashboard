## Best Possible Solution

Land the opt-in copy-token path only after maintainer acceptance of clipboard bearer recovery; otherwise keep the canonical issue pointed at a session or exchange-token dashboard auth design.

## Reproduction Assessment

Yes. Source inspection shows that with a SecretRef-managed `gateway.auth.token`, current dashboard opens/copies a clean URL and `config get` returns the redacted config object, while the Control UI still needs a token after session state is lost.

## Solution Assessment

Unclear until maintainers accept the security boundary. Technically this is a narrow interim fix that avoids stdout and URL leakage, but session or exchange-token auth remains the cleaner long-term solution if plaintext clipboard recovery is rejected.
