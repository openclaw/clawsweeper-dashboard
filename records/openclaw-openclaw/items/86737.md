## Summary

Keep this PR open: the subscription and assistant-transcript direction matches the reported iOS group-chat bug, but the current patch still has a message-delivery blocker in user-echo adoption and the supplied proof does not show the live Gateway-backed iOS group-chat flow.

## What This Changes

The PR updates iOS chat transport to subscribe to `sessions.messages.subscribe`, maps `session.message` frames, and changes shared chat UI handling/tests so active-session transcript messages can update the chat while a run is pending.

## Best Possible Solution

Keep the iOS subscription and assistant transcript path, but constrain user-echo adoption to the actual pending local optimistic send and add a same-content remote or repeated user-message regression before merge.

## Reproduction Assessment

Yes for the code path: current main has a no-op iOS active-session subscription, lacks iOS `session.message` mapping, and filters assistant or pending session messages in the shared view model. I did not run Swift tests in this read-only review, and the live iOS Gateway group-chat reproduction remains unproven here.

## Solution Assessment

No, not yet: subscribing to per-session transcript events and accepting assistant updates is the right owner-boundary fix, but the current echo-adoption helper overmatches same-content user messages. The safer path is to limit adoption to the active optimistic local send and cover the same-content remote/repeated case.
