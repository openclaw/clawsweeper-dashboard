## What This Changes

The PR makes iOS subscribe to `sessions.messages.subscribe`, maps `session.message` frames into the shared chat transport, and changes `ChatViewModel` tests/logic so assistant transcripts append during pending group-chat runs while local user echoes are deduped.

## Best Possible Solution

Land the iOS subscribe/map fix with shared echo adoption gated to a known local optimistic send, then attach redacted live iOS Gateway group-chat proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, by source inspection: current main's iOS transport does not subscribe/map `session.message`, and the shared handler drops assistant messages and pending-run transcript events. The PR body also includes a focused failing/passing Swift transcript, but I did not execute tests in this read-only review.

## Solution Assessment

No. The iOS subscription and assistant-message path are the right direction, but content-only user echo adoption is not the safest fix because it can collapse distinct transcript messages; it should be tied to a tracked local optimistic send.
