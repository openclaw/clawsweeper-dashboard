Keep open: the PR targets a real current-main iOS message-delivery bug, but the new echo-adoption path can hide a legitimate same-content user transcript and the supplied proof still does not demonstrate a live Gateway-backed iOS group chat.

## What This Changes

The PR wires iOS `setActiveSessionKey` to `sessions.messages.subscribe`, maps `session.message` frames, and changes the shared chat view model/tests so assistant transcript events can appear while a run is pending.

## Best Possible Solution

Merge a version that subscribes/maps iOS `session.message` events while only deduping confirmed local echoes, then back it with focused Swift regression coverage plus live iOS/Gateway group-chat proof or a maintainer proof override.

## Reproduction Assessment

Yes, source-reproducible: current main leaves iOS `setActiveSessionKey` as a no-op, drops `session.message` frames, and the shared handler rejects assistant events while pending. I did not run a live iOS group-chat repro here, but the PR body includes a baseline Swift timeout for the assistant transcript case.

## Solution Assessment

No as proposed: the subscription and mapping direction matches the gateway and macOS contract, but echo adoption needs a local-pending discriminator before it is the safest fix.
