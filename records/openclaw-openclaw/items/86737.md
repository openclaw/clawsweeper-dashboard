## Summary

Keep this PR open: the iOS subscription and `session.message` mapping are the right direction for the linked group-chat delivery bug, but the new user-echo adoption can still suppress legitimate repeated or remote same-content user transcripts, and the live iOS Gateway group-chat proof is still missing.

## What This Changes

The PR adds iOS `sessions.messages.subscribe` and `session.message` mapping, updates the shared chat view model to handle assistant transcripts during pending runs and dedupe local user echoes, and adds Swift regression tests.

## Best Possible Solution

Keep the iOS subscription/mapping direction, but gate user-echo adoption to a tracked pending local optimistic send, add repeated same-content `session.message` coverage, and verify with live or maintainer-overridden iOS group-chat proof.

## Reproduction Assessment

Yes, source inspection and the supplied baseline test output give a high-confidence reproduction shape: current main does not subscribe/map iOS `session.message` frames and the shared handler drops assistant messages while a run is pending. I did not run a live iOS Gateway group-chat scenario in this read-only review.

## Solution Assessment

No, not yet: the subscription and event-mapping path is the right fix direction, but the echo-dedupe repair is too broad because it treats any same-content user transcript as the local optimistic echo. The safer solution is to constrain adoption to pending local sends and prove the live iOS group-chat path.
