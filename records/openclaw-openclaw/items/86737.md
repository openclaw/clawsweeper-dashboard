## Summary

Keep open: the fix direction is well targeted, but latest head still has a message-loss regression in user-echo adoption and does not yet show live iOS Gateway-backed group-chat behavior after the patch.

## What This Changes

The PR makes iOS subscribe to per-session transcript events, maps `session.message` frames into the shared chat transport, and changes shared chat reconciliation so assistant transcript messages can appear while a run is pending.

## Best Possible Solution

Keep the iOS subscription and `session.message` mapping, but scope user-echo adoption to a verifiable pending optimistic local send and require live iOS Gateway proof or maintainer override before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main does not subscribe/decode iOS `session.message` frames and the shared view model drops assistant messages while a run is pending. I did not run tests because this review is read-only, but the PR body also includes a baseline Swift test timeout for the same path.

## Solution Assessment

No as submitted: the subscription and assistant-message path are the right repair direction, but the user-echo dedupe must be narrowed so distinct same-content user messages are not lost. The safer fix is to adopt only a known pending local optimistic row and let other user transcripts append normally.
