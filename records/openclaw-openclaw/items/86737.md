Keep this PR open: it addresses a real iOS group-chat message-delivery bug, but the current helper can still drop distinct repeated same-content user transcript events, and the supplied proof does not show the live Gateway-backed iOS group-chat flow.

## What This Changes

The PR updates the iOS gateway chat transport to subscribe to per-session transcript events, maps `session.message` frames, and changes the shared chat view model/tests to show real-time group-chat transcripts while deduping local user echoes.

## Best Possible Solution

Land this after echo adoption is constrained to a confirmed current local optimistic send, a repeated same-content external user transcript regression is added, and maintainers either get redacted live iOS Gateway group-chat proof or explicitly override that proof gate.

## Reproduction Assessment

Yes for the code-level failure path: current main leaves the iOS subscription hook empty and drops assistant or pending-run `session.message` events in the shared view model. I did not reproduce the full live iOS device/simulator group-chat flow locally.

## Solution Assessment

No, not as submitted. The subscription and event mapping direction is the right narrow fix, but echo adoption needs to prove it is consuming only the current local optimistic send before this is safe to merge.
