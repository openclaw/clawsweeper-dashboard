## Summary

This PR targets a real iOS group-chat message-delivery bug, but it is not merge-ready: the new local user echo adoption can suppress legitimate same-content user transcripts, and the supplied proof still does not show the live Gateway-backed iOS group-chat behavior.

## What This Changes

The branch adds iOS `sessions.messages.subscribe` and `session.message` mapping, lets the shared chat view model append assistant transcript events while a run is pending, and adds Swift regression tests for assistant updates plus local user echo dedupe.

## Best Possible Solution

Keep the iOS subscription and assistant transcript direction, but constrain local echo adoption to a real pending local send or server echo identifier, add same-content external-user coverage, and require live iOS Gateway group-chat proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, at source level: current main does not subscribe iOS to per-session transcripts, does not map `session.message`, and filters assistant messages while a run is pending. I did not run a live iOS Gateway group-chat reproduction in this read-only review.

## Solution Assessment

No, not yet. The subscription and assistant transcript path matches the existing macOS/gateway contract, but the user-echo dedupe needs a narrower identity before this is the safest fix.
