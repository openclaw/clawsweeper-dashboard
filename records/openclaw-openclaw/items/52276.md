## Summary

Keep open: current main still uses text-only embedded block-reply dedupe, so the route-scoped fix remains relevant, but this branch is not mergeable because PR head calls an undefined resolver, omits Telegram origin thread context, imports a helper name not exported on current main, and lacks external real behavior proof.

## What This Changes

This PR makes embedded messaging-tool duplicate suppression route-aware by threading provider, target, and account context through runner/subscriber paths and widening message-send target extraction.

## Best Possible Solution

Land a corrected, rebased route-scoped dedupe implementation that uses current helper exports, carries origin thread/account/target context through embedded subscriptions, mirrors outbound message-send normalization, and includes redacted Telegram or equivalent runtime proof.

## Reproduction Assessment

Yes, source-reproducible. Current main's streamed and message_end embedded block-reply paths still suppress by normalized text only, and PR head can be statically checked for the undefined resolver and export mismatch; I did not run tests because this review is read-only.

## Solution Assessment

No. The route-aware direction is appropriate, but this branch needs a rebase, a real resolver/current helper export alignment, origin thread handling, and real behavior proof before merge.
