## Summary

Keep open. Current main still lacks route-aware embedded block-reply suppression, so the underlying fix is still relevant, but this PR head is not mergeable as-is: it calls an undefined resolver and still loses Telegram topic origin/alias context needed for correct suppression.

## What This Changes

The PR makes embedded messaging-tool duplicate suppression route-aware by carrying provider, target, account, and channel context through runner/subscriber paths, widening message-send target extraction, and adding focused tests.

## Best Possible Solution

Land a corrected route-scoped dedupe implementation that wires the existing suppression helper into streamed and message_end block-reply paths, defines or imports the routed-provider resolver, carries origin provider/target/account/thread context through the embedded subscription boundary, and mirrors outbound message-send normalization before recording suppression targets.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR head references an undefined production symbol, and current main can be inspected to show both embedded block-reply paths still suppress by text only. The Telegram topic failure follows from the PR head passing a chat-only `originatingTo` while the suppression helper compares target thread metadata.

## Solution Assessment

No. The PR direction is the right narrow fix, but this branch is not the best mergeable solution until the resolver is defined, origin thread context is propagated, and thread alias precedence matches Telegram execution semantics.
