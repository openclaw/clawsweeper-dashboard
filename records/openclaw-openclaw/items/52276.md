## Summary

Keep open. Current main still has text-only embedded block-reply suppression, while this PR is useful but not mergeable because the latest head still references an undefined resolver and drops Telegram topic context needed for route-scoped suppression.

## What This Changes

The PR makes embedded messaging-tool duplicate suppression route-aware by threading provider, target, account, and channel/thread context through runner/subscriber paths and widening message-send target extraction.

## Best Possible Solution

Land a corrected route-scoped dedupe implementation that wires the existing suppression helper into streamed and message_end block-reply paths, defines the routed-provider resolver, carries origin thread context, and mirrors outbound message-send normalization before recording suppression targets.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR head references an undefined production symbol, and current main can be inspected to show both embedded block-reply paths still suppress by text only; the Telegram topic failure follows from current origin/thread routing code.

## Solution Assessment

No. The PR direction is narrow and appropriate, but this branch is not the best mergeable solution until the resolver is defined, origin thread context is propagated, and thread alias precedence matches Telegram execution semantics.

  - confidence: 0.91

  - confidence: 0.88
