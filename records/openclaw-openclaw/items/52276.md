## Summary

Keep open. Current main still suppresses embedded block replies by normalized text only, while this PR’s route-aware direction remains useful but is not mergeable as-is because the branch has concrete runtime/compile and Telegram topic-routing defects.

## What This Changes

The PR makes embedded messaging-tool duplicate suppression route-aware by threading provider, target, account, and thread context through runner/subscriber paths and widening message-send target extraction.

## Best Possible Solution

Land a corrected route-scoped dedupe implementation that defines the provider resolver, carries origin provider/target/account/thread context through the embedded subscription boundary, and mirrors outbound message-send normalization before recording suppression targets.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main shows text-only suppression in both streamed and message_end paths, and the PR head’s unresolved symbol plus Telegram thread-context mismatch are visible from the provided diff and current runtime code.

## Solution Assessment

No. The PR direction is narrow and appropriate, but this branch is not the best mergeable solution until the resolver is defined, origin thread context is propagated, and thread alias precedence matches Telegram execution semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P0] Define the routed provider resolver before using it:** `src/agents/pi-embedded-runner/run/attempt.ts:383-386`
  - body: runEmbeddedAttempt now calls `resolveRouteMessageProvider(...)`, but the PR head does not declare or import that production symbol. Embedded runs would fail before tool construction and reply handling can proceed.
  - confidence: 0.99
- **[P1] Carry the origin thread into suppression matching:** `src/agents/pi-embedded-runner/run/attempt.ts:1205-1207`
  - body: The subscription receives provider, target, and account context, but not the origin thread id. In Telegram topic sessions the sent target can record a thread, and suppression requires thread parity, so same-topic sends can still emit a duplicate final block reply.
  - confidence: 0.91
- **[P2] Prefer messageThreadId when deriving the dedupe thread:** `src/agents/pi-embedded-subscribe.tools.ts:378-386`
  - body: The extractor records `threadId` before `messageThreadId`, but Telegram action execution resolves those aliases in the opposite order. A call containing both aliases can be delivered to one topic and tracked against another, causing incorrect suppression decisions.
  - confidence: 0.88
