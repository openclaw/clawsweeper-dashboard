## Summary

Keep open: current main still lacks route-scoped embedded block-reply dedupe, and this PR is directionally useful but not mergeable because the branch has concrete compile/runtime and Telegram thread-context defects.

## What This Changes

The PR makes embedded messaging-tool duplicate suppression route-aware by threading provider, target, account, and current channel/thread context through runner and subscriber paths.

## Best Possible Solution

Land a corrected route-scoped dedupe implementation that reuses the current route helper, carries origin provider/target/account/thread context, and mirrors outbound message-send normalization before suppressing final block replies.

## Reproduction Assessment

Yes. Source inspection shows current main still suppresses embedded streamed and message_end block replies by text only, while route-aware matching exists but is not wired into those paths.

## Solution Assessment

No. The PR direction is narrow and appropriate, but this branch is not the best mergeable solution until it defines the routed-provider resolver, uses the current helper export, propagates origin thread context, and aligns Telegram alias precedence.
