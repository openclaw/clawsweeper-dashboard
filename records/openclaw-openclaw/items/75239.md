## Summary

Keep open. Current main still has the reported final-then-transcript ordering risk: a renderable assistant final clears `chatRunId`, a later same-session `session.message` reloads `chat.history` as an idle update, and the focused tests still do not cover a post-final transcript event.

## What This Changes

The issue reports duplicate WebChat assistant bubbles after the earlier #66875 final/history reload mitigation shipped in 2026.4.27.

## Best Possible Solution

Add a focused Control UI regression for renderable assistant final followed by same-session transcript/index events, then make final/history reconciliation idempotent for that ordering while preserving legitimate idle transcript refreshes and tool-output reloads.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a plausible current-main path: renderable final clears `chatRunId`, then a later same-session `session.message` triggers `chat.history`, but the duplicate rendering still needs browser or regression-test proof.

## Solution Assessment

Unclear because no patch is proposed. The maintainable fix is a narrow UI-side idempotency change around final/history reconciliation, not a broad ban on `chat.history` reloads.
