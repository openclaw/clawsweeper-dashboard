## Summary

Keep open. Current main still clears the active WebChat run on a renderable final assistant event and then treats a later same-session `session.message` as an idle transcript update that reloads `chat.history`; the existing regression coverage only proves the earlier deferred-before-final mitigation, not the reporter's final-then-session-message ordering. No relevant Control UI code changed after the latest release tag.

## What This Changes

The issue reports that WebChat/Control UI still renders duplicate assistant reply bubbles on OpenClaw 2026.4.27 after the earlier duplicate-message fixes.

## Best Possible Solution

Add a focused Control UI regression test for a renderable assistant final followed by same-session `session.message` and, if relevant, `sessions.changed`, then make the reload/reconciliation path idempotent for that ordering while preserving legitimate history refreshes.

## Reproduction Assessment

Unclear. The issue gives concrete browser steps and source inspection shows a plausible event-order path, but this review did not live-run Control UI and current tests do not reproduce the final-then-`session.message` ordering.

## Solution Assessment

Unclear until the event-order test is added. The narrowest maintainable direction is a UI reconciliation/idempotence fix around redundant post-final history reloads, not a broad suppression of all `chat.history` refreshes.
