## Summary

Keep open. Current main still has the post-final Control UI path that can reload chat history after a renderable assistant final has already cleared the active run, and existing regression coverage stops at the before-final deferral case rather than the reported final-then-transcript-event ordering.

## What This Changes

The issue reports that WebChat/Control UI still duplicates assistant reply bubbles after the earlier final-event and duplicate-submit fixes shipped in 2026.4.27.

## Best Possible Solution

Add regression-backed Control UI idempotency so a completed renderable assistant final followed by same-session transcript events does not reintroduce duplicate assistant bubbles, while preserving necessary history refreshes for silent finals, tool output hydration, and genuine out-of-band transcript updates.

## Reproduction Assessment

Yes for a focused unit-level reproduction path: emit a renderable same-session `chat` final that clears `chatRunId`, then emit a same-session `session.message` and observe the idle `loadChatHistory` call. End-to-end browser reproduction remains unproven in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable direction is in the existing Control UI final/history reconciliation path. A broad Gateway event change or disabling all idle `session.message` reloads would be riskier because other transcript updates still need to refresh history.
