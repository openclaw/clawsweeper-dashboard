## Summary

Keep open. The issue is a fresh WebChat/Control UI regression report, and current main still has a same-session idle `session.message` path that reloads chat history immediately after a normal renderable final has cleared `chatRunId`; existing tests cover the earlier deferred `session.message`-before-final case but not the reported final-then-`session.message`/`sessions.changed` ordering.

## What This Changes

The issue asks for the remaining WebChat duplicate assistant-bubble regression after the 2026.4.27 fixes for final-event reloads, in-flight submits, and duplicate WebSocket handshakes.

## Best Possible Solution

Add a focused WebChat/Control UI regression test for a renderable assistant final followed by same-session `session.message` and `sessions.changed`, then make the history reload/reconciliation path idempotent for that ordering while preserving legitimate reloads for silent finals, tool-output finals, aborts/errors, and out-of-band transcript changes.
