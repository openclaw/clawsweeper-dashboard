## Summary

Keep open. Current main still has a plausible WebChat post-final reload path: a renderable `chat` final appends the assistant message and clears `chatRunId`, while a later same-session `session.message` with no active run still calls `loadChatHistory`; the existing focused regression covers the before-final deferral, not this reported final-then-transcript-event ordering.

## What This Changes

The issue reports that WebChat Control UI still renders duplicate assistant replies after the earlier 2026.4.27 fixes for final/history reload races.

## Best Possible Solution

Add a focused Control UI regression for renderable assistant final followed by same-session `session.message` and `sessions.changed`, then make the final/history reconciliation idempotent for that ordering while preserving required reloads for silent, tool-only, aborted, error, other-run, and stale-history cases.

## Reproduction Assessment

Unclear. The issue gives concrete browser steps and the current source shows a plausible final-then-transcript-event path, but this review did not run Control UI and the current tests do not cover the reported post-final ordering.

## Solution Assessment

Unclear until a patch exists. The likely maintainable fix is a narrow regression-backed Control UI idempotency change, not a broad suppression of all same-session `session.message` history reloads.
