## Summary

Keep open. Current main still has an immediate same-session `session.message` to `loadChatHistory` path once a normal final has cleared `chatRunId`, and the existing regression coverage proves the earlier `session.message`-before-final mitigation but not the reporter's final/session.message/sessions.changed ordering.

## What This Changes

The issue reports that Control UI WebChat still renders duplicate assistant reply bubbles after the earlier final-event, session.message, and history-reload mitigations in OpenClaw 2026.4.27.

## Best Possible Solution

Add a focused Control UI regression test for a renderable assistant final followed by same-session `session.message` and `sessions.changed`, then make the reconciliation/history-reload path idempotent for that ordering while preserving legitimate history refreshes for silent finals, tool output, errors, aborts, and out-of-band transcript changes.

## Reproduction Assessment

Unclear. The issue gives a concrete browser reproduction path, but this review did not live-run Control UI; the current unit tests cover `session.message` before final, not the reported post-final event ordering.

## Solution Assessment

Unclear. The previous #66875 direction is still the right area, but the best fix should be a narrow regression-backed Control UI reconciliation change rather than broadly disabling `chat.history` reloads.
