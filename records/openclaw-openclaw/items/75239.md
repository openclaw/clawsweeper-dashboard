## Summary

Keep open. The report is recent, concrete, and still plausible on current main: the earlier WebChat fixes cover same-session `session.message` while a run is active and several history-dedup paths, but current code still permits an idle same-session `session.message` reload immediately after a renderable final clears `chatRunId`, and the tests do not cover the reporter's final-then-session-message ordering.

## What This Changes

The issue reports that WebChat Control UI on OpenClaw 2026.4.27 still renders each assistant reply twice after the earlier #66875 and related duplicate-message fixes.

## Best Possible Solution

Add a focused Control UI regression test for a renderable assistant final followed by same-session `session.message` and `sessions.changed`, then make the WebChat reconciliation/history-reload path idempotent for that ordering while preserving legitimate history refreshes for silent finals, tool results, errors, aborts, and external transcript updates.
