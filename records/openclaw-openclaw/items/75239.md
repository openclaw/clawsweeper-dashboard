## Summary

Keep open. Current main still has a plausible post-final WebChat reload path: a renderable assistant `chat` final clears the active run, and a later same-session `session.message` with no active run still triggers `chat.history`; the existing regression coverage covers the before-final deferral but not the final-then-transcript-event ordering reported here.

The issue reports that WebChat/Control UI still renders duplicate assistant replies on 2026.4.27 after the earlier final/history reload mitigations.
