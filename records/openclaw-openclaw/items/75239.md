## Summary

Keep open. Current main still has a plausible post-final same-session reload path: a renderable final clears `chatRunId`, then a later `session.message` for the same session calls `loadChatHistory`; existing regression coverage covers the earlier pre-final deferral but not the reporter's final/session.message/sessions.changed ordering.

## What This Changes

The issue asks OpenClaw to fix remaining WebChat/Control UI duplicate assistant reply rendering after the prior #66875 and related 2026.4.27 mitigations.

## Best Possible Solution

Add a focused Control UI regression test for a renderable assistant final followed by same-session `session.message` and `sessions.changed`, then make the history reload/reconciliation path idempotent for that ordering while preserving the existing legitimate refresh cases.
