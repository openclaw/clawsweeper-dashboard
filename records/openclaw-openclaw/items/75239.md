## Summary

Keep open. Current main has several WebChat duplicate-message mitigations, but there is still no high-confidence close evidence for the reported final-then-transcript-event ordering: a renderable assistant final clears `chatRunId`, a later same-session `session.message` still reloads `chat.history`, and focused coverage does not prove that ordering cannot duplicate or flicker.

## What This Changes

The issue reports that WebChat Control UI still duplicates assistant replies on 2026.4.27 after the #66875 and related WebChat duplicate-message fixes.

## Best Possible Solution

Add a focused Control UI regression for renderable assistant final followed by same-session transcript/index events, then make final/history reconciliation idempotent for that ordering while preserving legitimate idle transcript refreshes and tool-output reloads.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the relevant current-main event path: final clears `chatRunId`, then a later same-session `session.message` calls `loadChatHistory`, but the duplicate bubble still needs a browser or regression-test proof.

## Solution Assessment

Unclear because no patch is proposed. The maintainable solution is a narrow UI-side idempotency fix around final/history reconciliation, not a broad ban on `session.message` or `chat.history` reloads.
