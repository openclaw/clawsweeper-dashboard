## Summary

Keep open: current main has message-id dedupe for WhatsApp redelivery and queued follow-ups, but there is not enough proof that the reported busy-session two-path delivery is fixed; the shared turn path still records inbound context before dispatch and can then enqueue the same turn as a queued follow-up.

## What This Changes

The issue reports that a WhatsApp message received while the agent is busy can be delivered once as queued metadata and again as bare text, causing duplicate agent responses.

## Best Possible Solution

Add a focused regression test around a WhatsApp/shared inbound turn arriving during an active session, then fix the shared busy-queue path so the agent-visible delivery happens exactly once without losing transcript or routing metadata.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was run, but source inspection shows a plausible current-main path where inbound session recording happens before dispatch and the same message can then be represented as a queued follow-up wrapper.

## Solution Assessment

Is this the best way to solve the issue? Unclear until a regression test pins the duplicate surface; the narrowest likely fix belongs in the shared busy follow-up/session-turn path, with WhatsApp covered as the triggering channel.
