## Summary

Current main already routes this path through message-id dedupe at WhatsApp ingress, auto-reply dispatch, and the followup queue, so the reported double delivery should not occur on supported current builds.

## What This Changes

Ensure WhatsApp messages received while an agent run is active are delivered to the agent exactly once instead of once as queued metadata and again as bare text.

## Best Possible Solution

Keep the shipped idempotency path and ask for a fresh report only if a current release still double-delivers with message IDs and gateway logs.

## Reproduction Assessment

No current-main failing reproduction was established. Source inspection plus existing regression tests show the current path skips duplicate WhatsApp MessageSid deliveries and does not start a second bare-text agent run after enqueueing behind an active run.

## Solution Assessment

Yes. The narrow maintainable solution is the existing layered message-id dedupe at WhatsApp ingress, auto-reply dispatch, and followup queue boundaries rather than channel-specific prompt filtering later in the agent turn.
