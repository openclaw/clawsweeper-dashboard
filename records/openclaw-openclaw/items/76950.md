The PR adds a context-engine fallback session-size guard, a new `session.maintenance.contextFallbackGuard` config surface/schema/help/changelog entry, unit tests, and embedded-runner agent-id plumbing.

## Best Possible Solution

Land a reviewed version of the fallback guard that uses an existing recovery path or adds a real archive command, surfaces the offending transcripts clearly, and leaves broader always-on transcript caps to the canonical oversized-session follow-ups.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR emits `openclaw sessions archive`, and current CLI/docs do not define that subcommand. The original gateway overflow scenario is supported by the linked reports, but I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, not as currently written. The guard direction is plausible, but warn/block recovery needs to point to a real operator action and identify the affected transcript before this is the narrowest maintainable fix.
