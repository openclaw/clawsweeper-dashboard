## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route boot-guard errors through an error-capable logger:** `src/gateway/server-startup-post-attach.ts:720`
  - body: `params.log` is typed with only `info` and `warn` in this function, but the new boot-guard logger calls `params.log.error(...)`. This should fail type-checking and may be undefined at runtime for callers that satisfy the current type, so either widen/provide the logger or route errors through an existing error-capable sink such as `logHooks.error`.
  - confidence: 0.95
- **[P2] Replace the nonexistent sessions archive command:** `src/context-engine/fallback-guard.ts:516`
  - body: The warn-mode recovery block tells operators to run `openclaw sessions archive ...`, but current CLI/docs only define `openclaw sessions cleanup` and `openclaw sessions export-trajectory`. In the path where the guard only warns, that sends users to a command that fails, so point to a real recovery path or add and document the command.
  - confidence: 0.93
- **[P2] Reuse the session transcript artifact classifier:** `src/context-engine/fallback-guard.ts:157-176`
  - body: The custom `isLiveSessionTranscript()` filter treats `*.trajectory.jsonl` and compaction checkpoint files as live transcripts, while skipping valid primary names containing substrings like `.deleted`. Reuse `isPrimarySessionTranscriptFileName()` and layer only the new no-context-engine archive exclusion on top.
  - confidence: 0.9
- **[P2] Check the agent state directory for engine history:** `src/context-engine/fallback-guard.ts:147-150`
  - body: From `<state>/agents/<id>/sessions`, this walks up three directories to the global state root, but the heuristic says it is checking the agent state directory for engine DB markers. That makes `action: "auto"` miss per-agent context-engine stores and choose `warn` instead of the intended archive path.
  - confidence: 0.86
- **[P2] Align the documented fallback-guard default:** `src/config/schema.help.ts:1490`
  - body: The runtime constant now defaults to 2 MiB, while the config help/generated schema/type docs still say the default is `1mb`. Users will configure and diagnose from the wrong threshold unless all generated and source help text is updated with the intended default.
  - confidence: 0.92
- **[P2] Branch the operator wording for legacy boot guards:** `src/context-engine/fallback-guard.ts:502-504`
  - body: The boot guard intentionally fires when no context engine is configured, but both warning/archive renderers still say a configured context-engine plugin failed to load. For the legacy/unset path this misdiagnoses the trigger, so branch the wording and recovery prompt around `failedEngineId === "(legacy/none)"` or an explicit reason kind.
  - confidence: 0.88
- **[P3] Include blocked transcript identities in the resolver error:** `src/context-engine/registry.ts:546-555`
  - body: The block path computes the blocked paths but the thrown resolver error only reports a count and threshold. Since `block` is the operator-facing stop condition, include sanitized basenames or shortened paths so the operator knows which transcript to rotate.
  - confidence: 0.82
