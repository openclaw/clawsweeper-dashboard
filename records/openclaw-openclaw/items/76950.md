## Summary

Keep open: current main does not implement this guard, and the PR is relevant, but the proposed default `auto` behavior can archive transcripts that the context engine never stored, so it should not merge as-is.

## What This Changes

The PR adds a context-engine fallback guard, config schema/help/changelog entries, tests, and embedded-runner agent-id plumbing to warn/archive/block oversized session JSONL files when a configured context engine falls back to `legacy`.

## Best Possible Solution

Keep the fallback guard, but make `auto` conservative unless the guard can prove the specific transcript is backed by the missing context engine; otherwise warn and document the new operator-facing setting.

## Reproduction Assessment

Yes, from source inspection: current main falls back to `legacy` for missing/failing non-default context engines without a session-size preflight, matching the #76940 failure mode. I did not run a live gateway reproduction.

## Solution Assessment

No, not as currently proposed: centralizing the guard in the resolver is the right boundary, but default `auto` should not archive all oversized transcripts based only on a global engine database marker. A per-session proof or warn-first fallback is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Make auto archive require per-session engine history:** `src/context-engine/fallback-guard.ts:238-239`
  - body: `auto` currently resolves to `archive` for every oversized live transcript whenever a context-engine DB marker exists in the state root. Lossless Claw supports ignored/stateless sessions that do not store messages in the SQLite DB, so those JSONL files can be the only record; archiving them by default would silently move active history aside on fallback. Resolve `auto` per transcript, or warn unless the guard can prove that specific session is engine-backed.
  - confidence: 0.9
- **[P3] Document the new fallback guard:** `src/config/zod-schema.session.ts:98`
  - body: This adds a user-facing `session.maintenance.contextFallbackGuard` API, but the public session maintenance docs still list only the existing maintenance keys. Add concise docs for `sizeBytes`, `action`, and the default behavior in the session/config docs so operators can discover and tune the guard outside the changelog/schema metadata.
  - confidence: 0.84
