## Summary

Keep open. Current main only exposes coarse session `status` / `hasActiveRun` through `sessions.list` and `sessions.changed`; the requested gateway-owned `sessions.activity.get/list`, versioned statechange event, explicit attention states, bounded child rollup, and stale-run handling are still absent. The merged Sessions UI status work was intentionally UI-only, and earlier activity API attempts were closed unmerged or superseded without landing this contract.

## What This Changes

Add a gateway-owned per-session activity API and WebSocket statechange event with busy/idle, explicit attention states, child rollups, stale-run handling, and docs/tests.

## Best Possible Solution

Design and ship an additive authenticated Gateway activity contract with typed schemas, `sessions.activity.get/list`, `sessions.activity.statechange`, explicit attention transitions, bounded child rollups, stale-run cleanup, reconnect resync docs, and focused gateway/UI tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives a high-confidence absence check: current schemas, method lists, handlers, docs, and tests only expose coarse `sessions.list` status and `sessions.changed` events.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable direction is a typed gateway-owned activity model and versioned WS events, not channel-level status, message-text heuristics, or another UI-only projection.
