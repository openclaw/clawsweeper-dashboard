## Summary

Keep open: current main still lacks the requested public Gateway `sessions.activity.get/list` contract, explicit attention states, stale-run accounting, bounded child rollup, and activity statechange event. Existing `sessions.changed`, session-list `hasActiveRun`, and diagnostic activity tracking are useful adjacent pieces, but they do not solve the multi-client session-state problem described by the reporter and the ClawX follow-up.

## What This Changes

Request a Gateway-owned per-session activity API and WebSocket statechange model for busy/idle state, explicit attention blockers, stale-run recovery, and child-session rollups.

## Best Possible Solution

Design and land an additive Gateway-owned `sessions.activity` contract with explicit runtime and attention state, bounded child rollups, monotonic versions, stale-run recovery, authorization tests, docs, and reconnect guidance before downstream UI or watchdog features consume it.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than broken documented behavior. Source inspection gives a high-confidence current-main gap because Gateway exposes coarse session list/status fields and `sessions.changed`, not the requested activity RPC and statechange model.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. Gateway ownership is the right boundary, while the exact public payload shape, persistence behavior, stale-run handling, authorization, and reconnect semantics need maintainer product/API review.
