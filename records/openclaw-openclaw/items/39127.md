## Summary

Keep open: current main and v2026.6.10 still lack the requested `sessions.activity.get/list`, versioned activity statechange event, explicit attention states, stale-run accounting, and global per-session activity model. The open related PR only covers narrower `blocked`/`stale` status visibility, so this issue remains the canonical product/API design item.

## What This Changes

Request a gateway-owned per-session activity API and WebSocket statechange contract with busy/idle, attention states, stale-run handling, and child-session rollups.

## Best Possible Solution

Define one canonical gateway session activity contract that extends the existing session lifecycle/event model, with explicit producer/clear semantics, versioned WS resync, auth rules, stale-run cleanup, docs, and focused tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this requests a new public gateway/session activity contract, not a broken documented behavior. Source inspection gives high-confidence evidence that current main lacks the requested RPC and WS event surfaces.

## Solution Assessment

Unclear as an implementation shape: the gateway boundary is appropriate, but maintainers still need to settle the canonical session lifecycle, attention-state producer/clear rules, stale-run handling, and persistence model before a safe implementation.
