## Summary

Keep open. Current main has adjacent session list rows, active-run booleans, session change broadcasts, child session metadata, and internal diagnostics, but it still lacks the requested dedicated `sessions.activity.get/list` RPCs, explicit attention state, stale-run accounting, bounded child rollup, and `sessions.activity.statechange` contract.

## What This Changes

The issue requests a gateway-owned per-session activity API with busy/idle runtime state, explicit attention states, child rollups, and WebSocket statechange events.

## Best Possible Solution

Design and land an additive Gateway session-activity contract across protocol schema, server state, RPC handlers, WebSocket events, SDK/docs, and focused tests.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Current-main source inspection gives a high-confidence gap: only coarse `sessions.list`/`sessions.changed` and internal diagnostics exist, not the requested dedicated activity RPC/event model.

## Solution Assessment

Yes in direction, but not as an automated cleanup fix. A gateway-owned activity source is the maintainable boundary, while the public payload shape, persistence, stale handling, child rollup, authorization, and reconnect semantics need maintainer design.
