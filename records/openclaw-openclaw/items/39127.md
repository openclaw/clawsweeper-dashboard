## Summary

Keep open. Current main still does not provide the requested `sessions.activity.get/list` RPCs, `sessions.activity.statechange` WebSocket event, or public runtime/attention/children activity schema; existing surfaces are a lightweight `hasActiveRun` boolean, account-scoped channel health, and diagnostics-only liveness tracking.

## What This Changes

Add a gateway-owned per-session activity API with busy/idle runtime state, explicit attention blockers, child rollups, and versioned WebSocket state-change events.

## Best Possible Solution

Keep this as the canonical Gateway activity API feature and implement an additive, documented protocol contract with explicit runtime/attention transitions, stale cleanup, bounded child rollups, authorization parity with session reads, and focused Gateway tests.

## Reproduction Assessment

Not applicable as a feature/API gap. Source and docs inspection on current main gives a high-confidence verification path: the requested RPCs, event, schemas, and handler model are absent.

## Solution Assessment

Unclear until a Gateway API owner settles the contract. A dedicated additive `sessions.activity.*` surface is plausible, but the current `hasActiveRun` boolean, channel health fields, and diagnostics liveness tracking are not the requested maintainable solution.
