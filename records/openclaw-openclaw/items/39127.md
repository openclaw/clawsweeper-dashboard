## Summary

Keep open: current main and v2026.6.10 still provide coarse session list/status fields and `sessions.changed`, not the requested `sessions.activity.get/list`, explicit attention states, monotonic activity statechange event, stale-run accounting, or bounded child rollup. The open related status PR is only a partial `sessions.list` status improvement, so this issue remains the canonical product/API item.

## What This Changes

Request a gateway-owned per-session activity API and WebSocket statechange model with busy/idle runtime activity, explicit attention states, stale-run handling, and bounded child-session rollups.

## Best Possible Solution

Define one gateway-owned activity/attention contract, then implement the RPC, WebSocket statechange/resync path, stale-run handling, explicit producer transitions, docs, and focused gateway/client tests against that contract.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request. Source and release inspection give high-confidence evidence that the requested gateway activity API and attention-state model are absent today.

## Solution Assessment

Unclear until maintainers settle the protocol and product shape. A narrow UI or `sessions.list` heuristic is not the best solution; the maintainable path is one central gateway activity contract consumed by clients and follow-up watchdog/UI work.
