## Summary

Keep open. Current main still lacks the requested `sessions.activity.get/list` RPCs, `sessions.activity.statechange` event, exported protocol schemas, and runtime activity/attention/children model; the related implementation PRs are closed and unmerged, while the open Web UI PR is a UI/progress slice rather than this core Gateway API.

## What This Changes

The issue requests a gateway-owned per-session activity API and WebSocket statechange contract for busy/idle state, explicit attention blockers, stale-run handling, and child rollups.

## Best Possible Solution

Keep this as the canonical Gateway activity API feature and implement an additive, documented protocol contract with explicit runtime and attention transitions, stale cleanup, bounded child rollups, and focused Gateway tests.

## Reproduction Assessment

Not applicable as a feature/API gap. Source and docs inspection on current main gives a high-confidence verification path: the requested RPCs, event, schemas, and handler model are absent.

## Solution Assessment

Unclear until a Gateway API owner settles the contract. A dedicated additive activity API is plausible, but the current `hasActiveRun` boolean and channel health fields are not the requested maintainable solution.
