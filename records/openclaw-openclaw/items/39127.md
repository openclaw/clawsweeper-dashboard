## Summary

Current main has partial session liveness plumbing, but the requested gateway-owned `sessions.activity.get/list` API, explicit attention state model, stale-run counters, child rollup contract, and `sessions.activity.statechange` WebSocket event are not implemented; this should stay open for maintainer API design.

## What This Changes

Add a gateway-owned per-session activity contract with busy/idle runtime state, explicit waiting states, child rollups, stale-run handling, RPC methods, and WebSocket statechange events.

## Best Possible Solution

Design an additive Gateway session-activity contract that reuses the existing lifecycle, active-run, and session-event plumbing, documents persistence/access rules, and lands with protocol, Gateway, docs, and regression coverage while keeping UI-only status fixes separate.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection gives a high-confidence current-main gap: only coarse `sessions.list` status/live fields exist, not the requested dedicated RPC/event model.

## Solution Assessment

No, closing this as implemented would not be the best solution because current main only covers partial liveness display. The maintainable path is a deliberate additive Gateway protocol design with tests for activity snapshots, explicit attention transitions, stale handling, and reconnect resync.
