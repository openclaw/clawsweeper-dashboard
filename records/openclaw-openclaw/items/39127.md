## Summary

Keep open. Current main still only exposes a session-list `hasActiveRun` boolean plus channel/account health fields, while the requested Gateway-owned `sessions.activity` RPC/event contract, explicit attention states, stale-run handling, and bounded child rollup remain absent; the related implementation PRs are closed and unmerged.

## What This Changes

Request adds a Gateway-owned per-session activity API and WebSocket statechange event for busy/idle runtime state, explicit attention blockers, stale-run handling, and bounded child rollups.

## Best Possible Solution

Keep this as the canonical Gateway API feature and implement an additive schema-first `sessions.activity.get/list` plus versioned statechange contract with explicit attention transitions, stale cleanup, bounded child rollups, docs, and focused Gateway tests.

## Reproduction Assessment

Not applicable as a feature/API gap. Current-main source and docs inspection gives a high-confidence verification path: the requested RPCs, event, exported schemas, and handler coverage are absent.

## Solution Assessment

Unclear until a Gateway API owner settles the contract shape. The requested direction is plausible and additive, but the current `hasActiveRun` boolean is not the requested durable, versioned activity model.
