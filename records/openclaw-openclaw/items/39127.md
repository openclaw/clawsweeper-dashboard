## Summary

Keep open. Current main still has only coarse session-row status and session-change events, while the requested dedicated `sessions.activity.*` RPCs, explicit attention state, stale-run accounting, and activity statechange event are not present; this is a Gateway protocol/product feature, not conservative cleanup.

## What This Changes

The issue requests a Gateway-owned per-session activity API and WebSocket statechange event model for busy, attention, stale-run, and child-rollup state.

## Best Possible Solution

Land an additive Gateway-owned `sessions.activity` protocol and runtime tracker with documented versioning/resync, explicit attention states, bounded child rollups, stale-run handling, and focused tests before UI or watchdog consumers depend on it.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection on current main gives a high-confidence gap: the existing surfaces are coarse `sessions.list` row fields and `sessions.changed`, not the requested activity RPC/event model.

## Solution Assessment

Yes in direction, but not as an automated cleanup fix. Gateway-owned activity state is the right boundary, while the public payload shape, durability, stale-run behavior, authorization, and reconnect semantics need maintainer design.
