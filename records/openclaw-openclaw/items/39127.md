## Summary

Current main still has only coarse session liveness fields and UI projection; the requested dedicated `sessions.activity.get/list` RPCs, explicit attention model, stale-run handling, child rollup, and `sessions.activity.statechange` event remain unimplemented. The recent multi-agent use-case comment also confirms the remaining gap is not covered by `sessions.list` status/live fields.

## What This Changes

Add a gateway-owned per-session activity API and WebSocket statechange contract for busy/idle, explicit attention states, stale-run handling, and child-session rollups.

## Best Possible Solution

Design an additive Gateway session-activity contract that reuses existing lifecycle, active-run, and subagent state, documents persistence/access rules, emits versioned statechange events, and lands with protocol, docs, and regression coverage while keeping UI-only status fixes separate.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken existing behavior. Source inspection gives a high-confidence current-main gap: only coarse `sessions.list` status/live fields exist, not the requested dedicated RPC/event model.

## Solution Assessment

Yes, keeping this open for owner API design is the best current path. Closing as implemented would be incorrect because current main only covers partial liveness display, and a narrow automated fix would be risky without settled activity, attention, stale-run, and child-rollup semantics.
