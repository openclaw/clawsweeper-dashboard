## Summary

Keep open: current main and v2026.6.10 still expose only coarse session rows, status values, session-change/message/tool events, and a Control UI-visible `hasActiveRun` boolean, not the requested gateway-owned `sessions.activity.get/list`, monotonic activity statechange event, explicit attention states, stale-run accounting, or child rollup. The live related PRs are partial and not merged, so this remains the canonical product/API design issue.

## What This Changes

Request a gateway-owned per-session activity API and WebSocket statechange event that reports busy/idle runtime state, explicit attention blockers, stale-run handling, and child-session rollups.

## Best Possible Solution

Keep this as the canonical gateway activity contract issue, then align any implementation PR around a maintainer-approved protocol for activity snapshots, explicit attention states, stale-run cleanup, and bounded child-session rollups.

## Reproduction Assessment

Not applicable as a feature request: source inspection is enough to show current main lacks the requested activity RPC/event and only exposes coarse session status plus `hasActiveRun`. No failing current-main bug reproduction is required to keep the product/API request open.

## Solution Assessment

Unclear: a gateway-owned API is the likely right boundary because clients cannot derive this from channel status or local streams, but the durable schema, attention semantics, stale-run rules, and plugin hook exposure need maintainer API/product confirmation before implementation.
