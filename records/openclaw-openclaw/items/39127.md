## Summary

This issue should stay open: current `main` and the latest release provide partial session visibility through `sessions.list`, `sessions.changed`, `status`, `hasActiveRun`, timing fields, and child-session metadata, but they still do not implement the requested dedicated `sessions.activity.get/list` API, explicit `awaiting_user`/`awaiting_subagent` attention states, monotonic activity versions, stale-run counters, or `sessions.activity.statechange` reconnect contract.

## What This Changes

The issue requests a gateway-owned per-session activity API and WebSocket statechange contract for busy/idle state, explicit attention blockers, stale-run handling, and child-session rollups.

## Best Possible Solution

Extend the canonical Gateway session lifecycle/session-row owner with an additive, authorization-tested activity snapshot and push event that composes existing run and subagent facts, persists explicit attention transitions, and documents reconnect resynchronization.

## Reproduction Assessment

Not applicable: this requests a new public Gateway capability rather than reporting broken documented behavior. Source inspection gives high confidence that the requested activity RPC, explicit attention states, and versioned statechange event are still absent on current `main`.

## Solution Assessment

Yes directionally, but not as a parallel tracker without design. The best shape is to extend the current canonical session lifecycle and subagent registry surfaces with explicit activity/attention semantics rather than adding another source of truth.
