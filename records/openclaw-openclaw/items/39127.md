## Summary

Keep this open: current main has useful coarse session lifecycle/status plumbing, including `sessions.list` `status`/`hasActiveRun` and `sessions.changed`, but it does not expose the requested dedicated `sessions.activity.get/list` API, explicit attention states, stale-run counters, child rollups, or versioned `sessions.activity.statechange` events.

## What This Changes

The issue requests a gateway-owned per-session activity API and WebSocket statechange events with busy/idle, explicit attention states, stale-run handling, and child-session rollups.

## Best Possible Solution

Add an additive typed Gateway activity contract with focused tests and docs, reusing existing lifecycle/run facts where possible while explicitly modeling attention and child rollups instead of extending `sessions.list` ad hoc.

## Reproduction Assessment

Not applicable; this is a feature/API request, not a broken existing behavior report. Source inspection confirms the requested RPC/event contract is absent from current main while related coarse status surfaces exist.

## Solution Assessment

Yes directionally: a dedicated additive Gateway activity API is the maintainable solution for multi-client session state. The current coarse `sessions.list` status is useful partial groundwork, but not a complete substitute for the requested explicit activity model.
