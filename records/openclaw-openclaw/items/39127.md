## Summary

Keep open: current main has adjacent session-list status, child-session metadata, and `sessions.changed`, but it still lacks the requested public `sessions.activity.get/list` contract, explicit attention state, stale-run accounting, and activity statechange event. The prior implementation PRs are closed unmerged, and the merged session-management work only supersedes the smaller status slice, not this broader Gateway API request.

## What This Changes

The issue requests a Gateway-owned per-session activity API with runtime busy/idle state, explicit attention blockers, child rollups, and WebSocket statechange events.

## Best Possible Solution

Design and land an additive Gateway-owned `sessions.activity` contract with explicit runtime and attention state, bounded child rollups, monotonic versions, stale-run recovery, authorization tests, docs, and reconnect guidance before downstream UI/watchdog features consume it.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than broken documented behavior. Source inspection gives a high-confidence current-main gap because the Gateway exposes coarse session list/status fields and `sessions.changed`, not the requested activity RPC and statechange model.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. Gateway ownership is the right boundary, while the exact public payload shape, persistence behavior, stale-run handling, authorization, and reconnect semantics need maintainer product/API review.
