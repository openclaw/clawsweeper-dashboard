## Summary

Keep open. Current main still has only coarse session liveness fields and internal diagnostics; the dedicated `sessions.activity.get/list` RPCs, explicit attention model, stale-run accounting, bounded children rollup, and `sessions.activity.statechange` contract remain missing, and the latest discussion adds a concrete external-client use case.

## What This Changes

Request a gateway-owned per-session activity API with busy/idle state, explicit attention states, child-session rollups, stale-run handling, and WebSocket statechange events.

## Best Possible Solution

Design an additive Gateway session-activity contract that reuses existing lifecycle, active-run, subagent, and diagnostic plumbing, then land protocol schemas, server state, docs, and focused coverage together.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken existing documented behavior. Source inspection gives a high-confidence current-main gap: only coarse `sessions.list`/diagnostic/UI liveness signals exist, not the requested dedicated RPC/event model.

## Solution Assessment

Yes in direction, but not as an automated cleanup fix. A gateway-owned activity source is the maintainable boundary, while the exact snapshot shape, persistence, stale handling, child rollup, authorization, and reconnect semantics need maintainer design before implementation.
