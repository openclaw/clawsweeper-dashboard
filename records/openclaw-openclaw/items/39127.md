## Summary

Keep open: current main still lacks the requested public `sessions.activity.get/list` Gateway contract, explicit attention states, stale-run accounting, and activity statechange WebSocket event. The adjacent session-list status and `sessions.changed` work is useful but remains too coarse for the ClawX multi-session use case described in the discussion.

## What This Changes

The issue requests a gateway-owned per-session activity API and WebSocket statechange events for busy/idle, explicit attention states, stale-run handling, and child-session rollups.

## Best Possible Solution

Design and land an additive Gateway-owned `sessions.activity` contract with explicit runtime and attention state, bounded child rollups, monotonic versions, stale-run recovery, authorization tests, docs, and reconnect guidance before downstream UI or watchdog features consume it.

## Reproduction Assessment

Not applicable; this is a feature/API request rather than broken documented behavior. Source inspection gives a high-confidence current-main gap because the Gateway exposes coarse session list/status fields and `sessions.changed`, not the requested activity RPC and statechange model.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. Gateway ownership is the right boundary, while the exact public payload shape, persistence behavior, stale-run handling, authorization, and reconnect semantics need maintainer product/API review.
