## Summary

Keep open: current main still lacks the requested Gateway-owned `sessions.activity.get/list` RPCs, explicit attention state, stale-run accounting, bounded children rollup, and activity statechange event. Existing session-list/status fields and `sessions.changed` are useful adjacent signals, and the linked implementation attempts provide helpful context, but they do not satisfy the requested public activity contract on main.

## What This Changes

The issue requests an additive Gateway per-session activity API and WebSocket statechange event model for busy/idle, explicit attention states, stale-run handling, and child-session rollups.

## Best Possible Solution

Design and land an additive Gateway-owned `sessions.activity` contract with explicit runtime and attention state, bounded child rollups, monotonic versions, stale-run recovery, authorization tests, docs, and reconnect guidance before downstream UI/watchdog features consume it.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken documented behavior. Source inspection gives a high-confidence current-main gap because the Gateway exposes coarse session list/status fields and `sessions.changed`, not the requested activity RPC and statechange model.

## Solution Assessment

Yes directionally, but not as an automated cleanup fix. Gateway ownership is the right boundary, while the exact public payload shape, persistence behavior, stale-run handling, authorization, and reconnect semantics need maintainer product/API review.
