## Summary

Keep open. Current main still has only coarse session liveness plumbing through `sessions.list`, session lifecycle events, UI badges, and internal diagnostics; the requested dedicated `sessions.activity.get/list` RPCs, explicit attention model, stale-run counters, bounded child rollup, and `sessions.activity.statechange` WebSocket contract are not implemented. The latest issue comment adds a concrete external-client use case, so this should remain a maintainer Gateway API design item.

## What This Changes

The issue requests a gateway-owned per-session activity API and WebSocket statechange event covering busy/idle state, explicit attention blockers, stale-run handling, and child-session rollups.

## Best Possible Solution

Design an additive Gateway session-activity contract that reuses existing lifecycle, active-run, subagent, and diagnostic plumbing; document persistence/access/resync rules; and land protocol, server, docs, and focused regression coverage together while keeping UI-only status improvements separate.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken existing behavior. Source and live discussion give a high-confidence current-main gap: only coarse `sessions.list`/diagnostic/UI liveness signals exist, not the requested dedicated RPC/event model.

## Solution Assessment

No, closing this as implemented or sending it to an automated repair lane would not be the best path. The maintainable solution is a deliberate Gateway protocol design with explicit activity snapshots, attention transitions, stale handling, child rollups, authorization, docs, and tests.
