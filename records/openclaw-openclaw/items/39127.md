## Summary

Keep open. Current main has adjacent session status fields, child-session metadata, `sessions.changed` broadcasts, and internal stall diagnostics, but it still does not provide the requested dedicated `sessions.activity.get/list` RPCs, explicit attention state model, stale-run accounting, bounded child rollup, or versioned `sessions.activity.statechange` contract.

## What This Changes

Request a gateway-owned session-activity API with per-session busy/idle state, explicit waiting states, child rollups, stale-run handling, and WebSocket statechange events.

## Best Possible Solution

Land an additive Gateway protocol and runtime tracker for versioned session activity, then let UI and watchdog work consume that shared contract.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection gives a high-confidence gap: current main exposes coarse session list/event state, not the requested dedicated activity RPC/event model.

## Solution Assessment

Yes in direction, but not as an automated cleanup fix. A gateway-owned activity source is the maintainable boundary, while the public payload shape, persistence, stale handling, child rollup, authorization, and reconnect semantics need maintainer design.
