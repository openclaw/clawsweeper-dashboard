## Summary

Keep open. Current main still has adjacent session status, lifecycle events, UI badges, and internal diagnostics, but not the requested dedicated `sessions.activity.get/list` RPCs, explicit attention model, stale-run accounting, bounded children rollup, or `sessions.activity.statechange` contract; the latest issue discussion adds a concrete external-client use case that the current coarse fields do not cover.

## What This Changes

Request a gateway-owned per-session activity API and WebSocket state-change event model for busy/idle state, explicit attention blockers, stale runs, and child-session rollups.

## Best Possible Solution

Design and land an additive Gateway session-activity contract that reuses existing lifecycle, active-run, subagent, and diagnostic plumbing, with schema, server state, WebSocket events, docs, and focused coverage updated together.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection gives a high-confidence current-main gap: only coarse `sessions.list`/event/diagnostic signals exist, not the requested dedicated RPC/event model.

## Solution Assessment

Yes in direction, but not as an automated cleanup fix. A gateway-owned activity source is the maintainable boundary, while the snapshot shape, persistence, stale handling, child rollup, authorization, and reconnect semantics need maintainer design before implementation.
