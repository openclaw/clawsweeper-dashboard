## Summary

Keep open: current main and v2026.6.6 still provide only coarse session lifecycle and visible active-run flags, not the requested dedicated session activity RPC, monotonic statechange event, explicit attention states, stale-run count, or global per-session active-run model. Recent related issues were closed by routing their remaining attention/status work back to this issue, so this remains the canonical product/API design item.

## What This Changes

The issue requests a gateway-owned per-session activity API and websocket state-change model for busy/idle, explicit attention blockers, stale runs, and child-session rollups.

## Best Possible Solution

Define a small canonical gateway session activity/attention contract integrated with the existing lifecycle and `sessions.changed` path, then add focused docs and tests for resync, authorization, stale-run cleanup, and explicit producer/clear semantics.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this requests a new public gateway/session activity contract. Source inspection does give high-confidence evidence that current main only has basic lifecycle and active-run projection, not the requested attention/activity API.

## Solution Assessment

No, not as the full parallel tracker described in the issue. The gateway boundary is right, but the best maintainable solution should extend the canonical session lifecycle/event model with explicit attention signals and resync semantics rather than create duplicate state ownership.
