## Summary

Keep open. Current main still exposes coarse session-row status and `sessions.changed`, but it does not provide the requested dedicated `sessions.activity.get/list` RPCs, explicit attention state, stale-run accounting, bounded child rollup, or activity statechange event; this remains Gateway protocol/product work.

## What This Changes

The issue requests a Gateway-owned per-session activity API and WebSocket statechange model for busy/idle, explicit attention states, stale-run handling, and child session rollups.

## Best Possible Solution

Land an additive Gateway-owned `sessions.activity` protocol and runtime tracker with documented versioning/resync, explicit attention states, bounded child rollups, stale-run handling, authorization coverage, docs, and focused tests before UI or watchdog consumers depend on it.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection gives a high-confidence current-main gap: existing surfaces are coarse `sessions.list` row fields and `sessions.changed`, not the requested activity RPC/event model.

## Solution Assessment

Yes in direction, but not as an automated cleanup fix. Gateway-owned activity state is the right boundary, while the public payload shape, durability, stale-run behavior, authorization, and reconnect semantics need maintainer design.
