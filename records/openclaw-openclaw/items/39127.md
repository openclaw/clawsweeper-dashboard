## Summary

Keep open. Current main has adjacent session list status, child-session metadata, WebSocket session-change broadcasts, and internal diagnostics, but it still lacks the requested dedicated `sessions.activity.get/list` RPCs, explicit attention state, stale-run accounting, bounded child rollup, and `sessions.activity.statechange` contract.

## What This Changes

The issue requests a gateway-owned per-session activity API and WebSocket statechange events with busy/idle state, explicit attention states, stale-run handling, and child rollups.

## Best Possible Solution

Design and land an additive Gateway session-activity contract across protocol schema, server state, RPC handlers, WebSocket events, SDK/docs, and focused tests.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection gives a high-confidence gap: current main has coarse `sessions.list`/`sessions.changed` state, not the requested dedicated activity RPC/event model.

## Solution Assessment

Yes in direction, but not as an automated cleanup fix. A gateway-owned activity source is the maintainable boundary, while the public payload shape, persistence, stale handling, child rollup, authorization, and reconnect semantics need maintainer design.
