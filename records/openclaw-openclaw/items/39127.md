## Summary

Keep open. Current main has partial session visibility through `sessions.list`, `sessions.changed`, status/timing, and `hasActiveRun`, but it still lacks the requested `sessions.activity.get/list` RPCs, `sessions.activity.statechange` event, explicit attention states, stale-run accounting, and public activity schema.

## What This Changes

Request a gateway-owned per-session activity API with busy/idle runtime state, explicit waiting states, child rollups, stale-run handling, and WebSocket state-change events.

## Best Possible Solution

Keep this as the canonical Gateway activity API feature and implement an additive documented contract with read-scope authorization, explicit runtime and attention transitions, monotonic WS resync behavior, stale cleanup, bounded child rollups, and focused Gateway tests.

## Reproduction Assessment

Not applicable as a feature/API gap. Source inspection on current main verifies the requested RPCs, event, and public activity schema are absent while only partial `sessions.list` and `sessions.changed` status surfaces exist.

## Solution Assessment

Unclear until a Gateway API owner settles the contract. A dedicated additive `sessions.activity.*` surface is plausible, but the current `sessions.list` status fields and lifecycle events are not the full maintainable solution requested here.
