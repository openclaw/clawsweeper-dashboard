## Summary

Keep open. Current main still lacks the requested Gateway-owned per-session activity RPCs, versioned WebSocket statechange event, exported activity schemas, explicit attention state model, stale-run counter, and bounded child rollup. The related implementation PRs #39438 and #41399 are closed and unmerged, and the newer Web UI/status references do not replace the core Gateway API request.

## What This Changes

Request an additive Gateway-owned per-session activity API and WebSocket event model for busy/idle runtime state, explicit attention states, stale-run handling, and bounded child-session rollups.

## Best Possible Solution

Keep this open as the canonical core Gateway API feature. The end state should be an additive schema-first sessions.activity.get/list and sessions.activity.statechange contract using operator.read authorization, run-id-set based activity tracking, stale cleanup, explicit attention transitions, bounded cross-agent child rollups, reconnect resync semantics, public protocol docs, and focused Gateway tests.

## Reproduction Assessment

Not applicable as a feature/API gap. The high-confidence verification path is current-main source and docs inspection, which shows no sessions.activity RPC/event, exported schemas, or handler coverage.

## Solution Assessment

Unclear until a Gateway API owner confirms the contract shape. The requested direction is maintainable if it stays additive and schema-first, but attention durability and cross-agent child rollups should be settled before coding.
