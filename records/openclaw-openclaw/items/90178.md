## Label Justifications

- `P1`: The issue describes real parent sessions hanging with queued user messages after subagent announce retry-limit or expiry give-up.
- `impact:session-state`: The affected path can leave the requester session parked in yielded or processing state without the completion event it is waiting for.
- `impact:message-loss`: The child completion or failure signal that should wake the parent is not delivered on terminal announce give-up, and later user messages can queue behind the stuck turn.

## Summary

Keep open: current main and v2026.6.8 still have retry-limit and expiry give-up paths that record failed or suspended subagent delivery, but do not inject the requester-visible completion or failure event that a yielded parent is waiting for. The task-layer blocked status is only a partial mitigation because non-cancelled subagent terminal task updates are not auto-delivered to the requester session.

## What This Changes

The issue asks for subagent announce retry-limit or expiry give-up to wake the requester session waiting in `sessions_yield`, plus optional configurable retry and expiry controls.

## Best Possible Solution

Keep this issue open for one maintainer-owned core contract: terminal announce give-up should emit exactly one requester-visible failure/completion event that wakes `sessions_yield`; retry and expiry configurability should be a separate product/config decision.

## Reproduction Assessment

Yes, at source level: current main still reaches retry-limit or expiry give-up and records failed/suspended delivery without producing the requester-visible event that `sessions_yield` waits for. I did not run the live Signal/K8s timing-sensitive scenario.

## Solution Assessment

Yes for the narrow parent wake-up: a canonical synthetic requester completion/failure event is the maintainable fix for the deadlock. No for bundling retry/expiry config knobs automatically, because that expands public config surface and needs maintainer product judgment.
