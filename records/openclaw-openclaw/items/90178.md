## Summary

Keep open: current main and v2026.6.6 still have terminal retry-limit/expiry give-up handling that records failed or suspended delivery without injecting the parent-visible completion/failure event that a yielded requester is waiting for. This issue remains the canonical narrow tracker; a later duplicate was closed in favor of it, and broader related PR work is still open and unmerged.

## What This Changes

Issue requests that subagent announce give-up on retry-limit or expiry wake a parent waiting in `sessions_yield`, with optional configurable retry and expiry controls.

## Best Possible Solution

Keep this issue open for a core fix that emits exactly one canonical requester-visible failure/completion event when announce give-up is terminal, and decide retry/expiry configurability separately from that bug fix.

## Reproduction Assessment

Yes, at source level: current main still has retry-limit and expiry give-up handling that records failed or suspended delivery without creating the next model-visible parent completion/failure event that `sessions_yield` waits for. I did not run the live timing-sensitive Signal/K8s scenario.

## Solution Assessment

Yes for the narrow synthetic parent wake-up, which is the maintainable core bug fix for the deadlock. No for automatically adding retry/expiry config knobs in the same repair; that part needs maintainer product judgment because it expands public config surface.
