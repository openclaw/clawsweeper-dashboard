## Summary

Keep open: current main and the latest release still have retry-limit/expiry give-up paths that mark subagent delivery failed or blocked without making the requester-visible wake path eligible. The open candidate PR is the right review path, but it is not merged yet.

## What This Changes

Wake the parent session with a bounded failure/completion event when subagent announce retry-limit or expiry give-up happens, and consider separate retry/expiry configurability.

## Best Possible Solution

Land the constrained blocked-delivery parent wake recovery fix, then track retry/expiry configurability separately as a config/product decision if maintainers want that surface.

## Reproduction Assessment

Yes, at source level: current main reaches retry-limit or expiry give-up and records failed/blocked subagent delivery while the non-cancelled subagent terminal fallback is suppressed. I did not run the timing-sensitive live Signal/K8s scenario.

## Solution Assessment

Yes for the narrow wake-up bug: terminal announce give-up should surface a bounded requester-visible failure/completion event. No for bundling retry/expiry config knobs automatically, because that adds public config surface and needs maintainer direction.
