## Summary

Keep open: current main still analyzes unknown allowlist entries against the currently materialized tool catalog, while the reported optional plugin and configured MCP cases can be valid before those tools are present. The direct fix PR is closed unmerged, and the other related PR handled a separate media-tool `alsoAllow` bug rather than this warning path.

## What This Changes

The issue asks OpenClaw to stop warning that enabled plugin or configured MCP tool allowlist entries are unknown when those tools are valid but not registered yet.

## Best Possible Solution

Add manifest/config-aware known allowlist facts to warning analysis, thread them through the real gateway/agent/embedded callers, and cover optional plugin plus configured MCP glob cases without changing enforcement semantics.

## Reproduction Assessment

Yes, source-reproducible: with `llm-task` or `paperless__*` absent from the current `pluginGroups.all`, the current analyzer classifies the entry as unknown even though the manifest/docs/config make those entries valid. I did not run a live gateway log capture in this read-only review.

## Solution Assessment

Yes: the best fix is narrow warning-analysis alignment with enabled plugin manifests and configured MCP server patterns, not blanket suppression of unknown allowlist diagnostics.
