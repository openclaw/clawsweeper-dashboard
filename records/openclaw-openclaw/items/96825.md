## Summary

Keep open: current main and v2026.6.10 still have Telegram preview-stream paths that trust successful Bot API send/edit results and preview message IDs without proving the user-visible Telegram client received the preview content. An open PR appears to target the text-to-tool-to-text preamble loss subcase, but it is not merged and does not close the broader report yet.

## What This Changes

Reports Telegram preview-streamed assistant chunks being logged as successful sendMessage results while disappearing from the Telegram client, leaving only final or post-tool text visible.

## Best Possible Solution

Land or replace the Telegram fix for the tool-boundary preamble loss, then keep the HTML fallback and preview-finalization observability covered by the existing canonical trackers with redacted live Telegram proof.

## Reproduction Assessment

Yes, but not as a live repro in this pass. Source inspection shows Telegram preview paths accept message IDs from send/edit as delivery evidence, and the open candidate PR supplies live Bot API before/after proof for the text-to-tool-to-text overwrite subcase.

## Solution Assessment

Unclear as a complete fix. The Telegram dispatch/draft boundary is the right layer for the pre-tool preview loss, but the full issue should be resolved without adding a new default-breaking config mode and without losing the related HTML and delivery-mode evidence work.
