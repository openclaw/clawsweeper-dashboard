## Summary

Keep open. Current main and the latest release still refresh bootstrap files from disk on each session turn and replace the cached array when content changes, so the specific session-freeze/cacheWrite claim is not implemented; the open dynamic-file issue covers one possible fix path but does not fully own this report's session-freeze or documentation-position request.

## What This Changes

The issue asks OpenClaw to address prompt-cache cost spikes from mid-session writes to bootstrap-injected files, either by adding real session-freeze behavior, configurable dynamic bootstrap files, or clearer operator documentation.

## Best Possible Solution

Keep this open until maintainers choose and document the cache-stability contract, then implement that choice across bootstrap cache behavior, prompt-boundary config if needed, tests, and prompt-caching docs.

## Reproduction Assessment

Yes at source level: current `getOrLoadBootstrapFiles` refreshes disk contents and the focused test expects changed contents to replace the cached result. I did not live-reproduce the Anthropic billing ratio or run a local test because this was a read-only review.

## Solution Assessment

Unclear as proposed: real session-freeze and dynamic-file configuration are both plausible, but the best maintainable fix requires a maintainer decision on prompt-cache semantics and compatibility. A documentation-only update would be lower risk but would not address the reported cost spiral.
