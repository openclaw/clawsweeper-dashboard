## Summary

Keep open. The report is recent and includes concrete production liveness evidence tied to `memory_search`; current `main` still has plausible memory-core paths that can hold the tool call for a long time, and no canonical related issue fully owns this Discord-specific memory-core failure.

## What This Changes

The issue reports that the bundled memory-core `memory_search` tool can block the Node.js event loop during Discord DMs, causing gateway disconnects and hung agents.

## Best Possible Solution

Bound memory recall so Discord and other gateway I/O remain responsive: first identify the exact blocking path, then add focused regression coverage and either yield/offload/timeout the implicated builtin search, sync, or embedding work without changing unrelated memory behavior.

## Reproduction Assessment

No high-confidence live current-main reproduction is available from this review. Source inspection does show a credible current-main path where `memory_search` awaits in-process search/bootstrap/embedding work long enough to match the reported liveness warning.

## Solution Assessment

Unclear. The requested worker/process isolation and configurable timeout are plausible, but the safer first fix is to isolate which memory backend or embedding path blocked and then bound that path with focused tests.
