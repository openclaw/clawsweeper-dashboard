## Summary

Keep open: the report is current, has two production confirmations on v2026.5.7, and current main still runs builtin memory search work in the gateway request path without an end-to-end `memory_search` timeout or isolation boundary.

## What This Changes

The issue asks for memory-core `memory_search` to stop blocking the Node.js event loop during Discord DMs by adding bounded, non-blocking, fail-soft search behavior.

## Best Possible Solution

Define a bounded, non-blocking `memory_search` execution contract in memory-core, isolate synchronous builtin/provider work from the gateway event loop, and add regression proof that a slow search cannot starve channel I/O.

## Reproduction Assessment

No, not as a live current-main Discord repro. The production logs plus current source make the failure path source-reproducible, but the exact reporter backend/provider and index size were not established.

## Solution Assessment

Unclear. The requested fail-soft timeout direction is reasonable, but the durable fix likely needs backend-aware worker/process isolation or cooperative yielding, not only an awaited-call timeout wrapper.
