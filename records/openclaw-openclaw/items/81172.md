## Summary

Keep open: this is a current, specific production liveness bug, and current main still routes `memory_search` through inline memory manager work without an end-to-end search timeout or event-loop isolation. A second reporter confirmed the same v2026.5.7 behavior, so this should remain available for maintainer prioritization rather than be swept as stale, duplicate, or implemented.

## What This Changes

The issue requests that memory-core `memory_search` stop blocking the Node event loop during Discord DMs by adding bounded, non-blocking search execution and graceful timeout/fallback behavior.

## Best Possible Solution

Define a bounded, non-blocking `memory_search` contract in memory-core, isolate or avoid synchronous search/indexing work on the gateway event loop, fail soft on slow backends, and add regression proof that slow memory search cannot starve Discord or channel heartbeats.

## Reproduction Assessment

No, not as a high-confidence live reproduction. The production logs and current source make the failure path source-reproducible, but I did not reproduce Discord DMs on current main or identify the reporter's exact memory backend/provider.

## Solution Assessment

Unclear. The timeout and fail-soft direction is right, but the safest fix likely needs backend-aware isolation or worker/process execution because a wrapper timeout cannot preempt synchronous SQLite/vector fallback work.
