## Summary

Keep open: current main has only a partial embedding-abort fix and still runs active memory-core search work inline, while the remaining Discord/event-loop starvation report is not covered by the open legacy hard-timeout PR.

## What This Changes

The issue reports that bundled memory-core `memory_search` can block the Node.js event loop long enough to disconnect Discord and hang agents.

## Best Possible Solution

Define a bounded fail-open `memory_search` contract in memory-core, isolate or yield CPU/indexing work that can monopolize the Node event loop, and prove a slow search cannot starve Discord gateway I/O.

## Reproduction Assessment

No live Discord reproduction is available from this review. The production logs plus current source give a high-plausibility source path, but the exact backend/provider and whether the blocking is embedding I/O, manager creation, bootstrap sync, or SQLite/vector CPU work remains unconfirmed.

## Solution Assessment

Unclear. The recent embedding abort fix helps one failure mode, but the best maintainable fix likely needs an active memory-core tool deadline plus backend-aware isolation/yielding rather than only wrapping the current awaited search call.
