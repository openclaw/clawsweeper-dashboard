## Summary

Keep open: this is a current, specific liveness report, and current main still has memory-core search paths that can run long work inside the gateway tool-call path. It overlaps with the open broader memory_search liveness issue, but the Discord event-loop-delay evidence and synchronous builtin search concerns are specific enough to preserve here for maintainer follow-up.

## What This Changes

The issue reports that bundled memory-core `memory_search` can block the Node.js event loop long enough to close Discord gateway connections and hang agents.

## Best Possible Solution

Define a bounded, non-blocking memory-core search contract, isolate or avoid synchronous builtin search/indexing on the gateway event loop, and add regression proof that slow memory search cannot starve channel I/O.

## Reproduction Assessment

No, not for the full live Discord failure. The production log and current source make the path source-reproducible, but I did not reproduce a gateway stall or identify the reporter's active backend/provider and index size.

## Solution Assessment

Unclear. The requested timeout/fail-open direction is right, but the maintainable fix likely needs backend-aware isolation or worker/process execution in addition to any outer deadline.
