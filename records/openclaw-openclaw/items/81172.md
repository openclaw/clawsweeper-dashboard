## Summary

Keep open: this is a current, specific memory-core liveness report, and current main still runs the active `memory_search` path inline on the gateway event loop with several plausible long-running or synchronous paths. The older fail-open issue and PR overlap, but they do not cover the synchronous SQLite/vector/native-work risk called out here.

## What This Changes

The issue reports bundled memory-core `memory_search` blocking the Node.js event loop for 60+ seconds during Discord DMs, causing gateway disconnects and hung agents.

## Best Possible Solution

Define a bounded, non-blocking `memory_search` execution contract in memory-core, isolate or yield synchronous builtin search/indexing/embedding work off the gateway event loop, and add regression proof that slow memory search cannot starve Discord or other channel I/O.

## Reproduction Assessment

No high-confidence live reproduction yet. The production log and current source make the failure path plausible, but this review did not reproduce a Discord gateway stall or identify the reporter's active memory backend/provider.

## Solution Assessment

Unclear. The fail-soft timeout direction is right, but the maintainable fix likely needs backend-aware isolation, worker/process execution, or explicit yielding rather than only wrapping the current awaited search call.
