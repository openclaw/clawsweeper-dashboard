## Summary

Keep open: current `main` has a related partial fix in https://github.com/openclaw/openclaw/pull/82770 for aborting timed-out embedding provider calls, but it does not prove that the reported Discord gateway stall is fixed. The memory search tool still awaits search inline, current defaults still allow 60s remote and 5-minute local query embedding waits, and current source still has synchronous search/indexing paths that can run on the gateway event loop.

## What This Changes

The issue reports that bundled memory-core `memory_search` can block the Node.js event loop for 30-60+ seconds during Discord DMs, causing gateway disconnects and hung agents.

## Best Possible Solution

Define an end-to-end bounded `memory_search` contract in memory-core, keep provider abort propagation, add or adjust timeout configuration/defaults as needed, and prove a slow search cannot starve Discord or other channel I/O.

## Reproduction Assessment

No high-confidence live reproduction was established. The production logs plus current inline search, forced bootstrap sync, long query timeouts, and in-process SQLite fallback make the failure path source-reproducible enough to keep this open, but not enough to declare current main fixed or to auto-repair safely.

## Solution Assessment

Unclear. The merged AbortSignal plumbing is a good partial fix for timed-out embedding providers, but the best final solution likely needs a bounded memory_search contract and isolation or preemption for synchronous search/indexing work, not only provider aborts.
