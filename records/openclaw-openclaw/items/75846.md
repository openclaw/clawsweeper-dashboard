## Summary

Keep open. Current main still wires Claude CLI stream-json parsing only to assistant text deltas, while the existing OpenClaw tool-event contract and downstream consumers already depend on `stream: "tool"` events with `toolCallId`.

## What This Changes

Emit Claude CLI JSONL `tool_use` and `input_json_delta` records as OpenClaw tool stream agent events so iMessage and webchat can display tool activity.

## Best Possible Solution

Normalize Claude CLI stream-json tool start/input deltas into the existing generic `stream: "tool"` contract at the parser and runner boundary, without iMessage- or webchat-specific branches.

## Reproduction Assessment

Yes. Feed `createCliJsonlStreamingParser` a Claude `stream_event` `content_block_start` with `content_block.type: "tool_use"` plus an `input_json_delta`; current main has no tool callback, so no `stream: "tool"` event can be produced.

## Solution Assessment

Yes. A parser-level bridge reused by both one-shot and live Claude CLI runner paths is the narrowest maintainable fix because downstream surfaces already consume generic tool stream events.
