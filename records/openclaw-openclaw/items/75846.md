## Summary

Keep open. Current main still parses Claude CLI stream-json text deltas only and does not turn Claude `tool_use` streaming events into OpenClaw `stream: "tool"` events, while the embedded Pi path already emits those tool events and downstream webchat code requires them to render tool activity.

## What This Changes

The issue requests that the Claude CLI JSONL/stream-json backend surface Claude tool-use activity as OpenClaw `stream: "tool"` agent events so iMessage/webchat consumers can observe tool execution.

## Best Possible Solution

Add a generic Claude stream-json tool-event normalization path in the CLI parser, then wire both one-shot and live Claude CLI runner paths to emit the existing OpenClaw tool-event contract used by embedded Pi, without adding iMessage- or webchat-specific special cases.

## Reproduction Assessment

Yes. A focused unit reproduction is to feed `createCliJsonlStreamingParser` a Claude `stream_event` `content_block_start` with `content_block.type: "tool_use"` plus `input_json_delta`; current main has no tool callback and only records assistant text deltas, so no `stream: "tool"` event can be produced.

## Solution Assessment

Yes. The suggested parser-level bridge is the narrow maintainable fix because the missing signal originates in the Claude CLI JSONL parser, and downstream surfaces already understand `stream: "tool"` once it exists.
