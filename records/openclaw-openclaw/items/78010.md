## Summary

Keep open: this docs PR addresses a real current gap and is close, but the added restore guide incorrectly says the internal archive root matches the archive basename even though `openclaw backup create --output` can use an arbitrary outer filename. That would make the manifest and restore examples fail for custom-named or renamed archives.

## What This Changes

This PR adds a manual restore-from-backup section to `docs/cli/backup.md`, covering archive inspection, same-machine extraction, staged new-machine restore, and migration links.

## Best Possible Solution

Correct the archive-root derivation in the restore examples, then merge the docs-only manual restore guide while leaving a first-class restore subcommand to separate feature work.

## Reproduction Assessment

Yes. Current `main` source shows the backup docs and CLI only expose create/verify, and the PR-specific docs bug is source-proven by the archive-root and output-path contracts.

## Solution Assessment

No, not yet. The docs-only direction is appropriate, but the current PR should derive `<archive-root>` from `openclaw backup verify` output or archive listing instead of assuming it matches the outer filename.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Derive the archive root instead of using the basename:** `docs/cli/backup.md:122`
  - body: The restore guide says `<archive-root>` matches the archive basename, but `openclaw backup create --output my-backup.tar.gz` can write an archive whose internal `archiveRoot` is still the timestamped value. Readers with custom-named or renamed archives will set the wrong `ROOT` and fail to extract the manifest or payload; have the docs take `ROOT` from `openclaw backup verify` output or an archive listing instead.
  - confidence: 0.91
