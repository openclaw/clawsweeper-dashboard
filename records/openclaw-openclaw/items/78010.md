## Summary

Keep open: the docs-only direction is useful and paired with the linked docs gap, but the current PR still gives an incorrect archive-root assumption that can break restores for custom-named or renamed backup archives.

## What This Changes

The PR adds a manual restore-from-backup section and related links to `docs/cli/backup.md`.

## Best Possible Solution

Correct the archive-root derivation in the restore examples, then merge the manual restore guide while leaving the first-class restore subcommand to https://github.com/openclaw/openclaw/pull/59280 or its canonical tracker.

## Reproduction Assessment

Yes. Current main source shows the docs and CLI expose backup create/verify only, and the PR-specific bug is source-proven by the archive-root and output-path contracts.

## Solution Assessment

No, not yet. Manual restore docs are the right narrow fix for the docs gap, but the examples should derive `<archive-root>` from `openclaw backup verify`, an archive listing, or the manifest instead of assuming the outer filename.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Derive the archive root instead of using the basename:** `docs/cli/backup.md:122`
  - body: This says `<archive-root>` matches the archive basename, but `openclaw backup create --output my-backup.tar.gz` can still write a timestamped internal `archiveRoot`. The later `ROOT=...` example will fail for custom-named or renamed archives, so derive the root from `openclaw backup verify`, a tar listing, or the manifest.
