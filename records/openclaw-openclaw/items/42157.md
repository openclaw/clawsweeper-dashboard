## Summary

Keep open. Current main still acknowledges Telegram webhook updates before processing completes and has no persisted inbound webhook queue or startup replay path, so the reported loss window remains source-reproducible.

## What This Changes

Persist or replay Telegram webhook payloads that were acknowledged before processing completes so gateway restarts do not permanently drop them.

## Best Possible Solution

Design a Telegram-owned durable inbound webhook queue with secure local storage, bounded replay/dequeue semantics, and focused tests for ACK-before-processing restart loss, then keep the broader cross-provider catch-up work tracked separately.

## Reproduction Assessment

Yes, by source inspection: current main returns 200 before `bot.handleUpdate` finishes and has no persisted queue or startup replay, so a restart in that interval can drop an already-acknowledged update. I did not run a credentialed live Telegram restart E2E in this read-only review.

## Solution Assessment

Unclear as proposed. Disk-backed queueing is the right direction, but the prior PR design needs maintainer review for privacy, file hardening, bounded deferred work, duplicate/replay semantics, and whether this should be configurable.
