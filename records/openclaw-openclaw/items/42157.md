## Summary

Keep open. Current main and the latest release still ACK validated Telegram webhook payloads before durable handoff or processing completion, and the related persistence PRs are closed unmerged while #55792 is a broader catch-up request.

## What This Changes

The issue requests durable Telegram webhook handling so payloads acknowledged before a gateway restart can be replayed instead of lost.

## Best Possible Solution

Add a Telegram-owned durable webhook queue that persists validated raw updates before ACK, replays pending entries during webhook startup before accepting new deliveries, and removes entries only after full update processing settles.

## Reproduction Assessment

Yes. Source and tests show a high-confidence path: send a valid Telegram webhook update with slow `bot.handleUpdate`, observe the 200 ACK before work finishes, then restart during that window with no durable queue to replay.

## Solution Assessment

Yes. The narrowest maintainable fix is in `extensions/telegram`: auth-before-persist, durable enqueue before ACK, startup replay, and post-processing dequeue; broad cross-provider history catch-up is complementary but does not cover already-ACKed webhook payloads.
