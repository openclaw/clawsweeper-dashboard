## Summary

Keep open. Current main still acknowledges Telegram webhook delivery before update processing completes and has no persistent webhook inbox/replay path; the related broader catch-up issue does not supersede this narrower acknowledged-in-flight loss case.

## What This Changes

The issue requests durable handling for Telegram webhook payloads so acknowledged inbound updates survive gateway restarts before processing completes.

## Best Possible Solution

Add a Telegram-owned durable webhook inbox that persists authenticated payloads atomically before ACK, replays pending entries on startup before accepting new webhooks, deletes only after update processing settles, and documents at-least-once duplicate/retention semantics.

## Reproduction Assessment

Yes, by source inspection and existing tests: current main returns 200 before `bot.handleUpdate` finishes, and there is no replayable queue for that acknowledged payload. I did not run a live credentialed Telegram restart E2E in this read-only review.

## Solution Assessment

Unclear for the prior PR approach. A disk-backed queue is the plausible direction, but it should be designed inside the Telegram plugin with explicit storage, replay, duplicate, and security bounds before a replacement patch is attempted.
