## Label Justifications

- `P1`: Silently losing inbound Telegram webhook messages during routine gateway restarts breaks an active channel workflow for real users.
- `impact:message-loss`: The reported failure mode is Telegram inbound updates being acknowledged and then lost before agent delivery.

## Summary

Keep open. Current main and v2026.6.6 still acknowledge Telegram webhook deliveries before asynchronous update handling finishes, while the durable Telegram ingress spool is wired to isolated polling rather than webhook receipt; the related closed PRs were not merged, and the broader restart catch-up issue does not replace this webhook-specific loss case.

## What This Changes

The issue requests durable handling of inbound Telegram webhook payloads so acknowledged updates can survive gateway restarts.

## Best Possible Solution

Use the existing SQLite-backed Telegram ingress queue for webhook receipt after auth/body validation and before acknowledgment, then complete or release the queued update only after handleUpdate and deferred work settle.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 acknowledge webhook requests before asynchronous Telegram update handling finishes, and the durable Telegram spool is not called from the webhook receipt path. I did not run a live credentialed Telegram restart test in this read-only review.

## Solution Assessment

Yes for durable pre-ACK webhook enqueue using the existing SQLite ingress queue; no for the old closed JSON-file queue proposals. Maintainers still need to decide exact replay/dequeue semantics before a repair lane is safe.
