- `impact:message-loss`: The reported failure is inbound Telegram messages being acknowledged and then lost before reaching the agent.

## Summary

Keep open. Current main and the latest release still acknowledge Telegram webhook deliveries before durable processing, and the existing durable Telegram ingress spool is wired through isolated polling rather than webhook receipt.

## What This Changes

The issue asks OpenClaw to avoid losing already-ACKed inbound Telegram webhook updates when the gateway restarts before processing completes.

## Best Possible Solution

Reuse the existing SQLite-backed Telegram ingress queue for authenticated webhook receipt before returning 2xx, with maintainer-approved replay, dequeue, failure, and retention semantics.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` return 200 from `startTelegramWebhook` before `bot.handleUpdate` finishes, while the durable spool writer is not called by webhook receipt.

## Solution Assessment

No final fix is present yet. The best current direction is not the older standalone JSON queue, but a Telegram-owned integration with the existing SQLite-backed ingress queue and explicit ACK/replay/dequeue policy.
