## Summary

Keep open. Current main and v2026.5.3 still acknowledge validated Telegram webhook updates before processing finishes, with no durable Telegram inbound queue or replay path; the prior persistence PRs are closed unmerged and #55792 tracks broader offline catch-up rather than already-ACKed webhook payloads.

## What This Changes

Persist validated Telegram webhook updates before acknowledging them and replay unfinished updates after gateway restart.

## Best Possible Solution

Add a Telegram-owned durable webhook queue that stores validated raw updates before ACK, replays pending entries during webhook startup before new deliveries, and deletes entries only after full update handling settles.

## Reproduction Assessment

Yes. Source and existing tests provide a high-confidence path: send a valid Telegram webhook update with a slow bot.handleUpdate, observe the 200 ACK before work finishes, then restart in that window with no persisted payload to replay.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Telegram plugin: validate before persisting, durably enqueue before ACK, replay on startup, and dequeue only after complete update handling; #55792 is complementary broader catch-up work, not a substitute.

## Repair Work Prompt

Fix the Telegram webhook restart-loss bug from https://github.com/openclaw/openclaw/issues/42157. Current main ACKs valid webhook payloads before `bot.handleUpdate` finishes, so a restart after ACK loses the payload permanently. Implement the fix inside the Telegram plugin boundary unless a generic SDK seam is clearly required: validate request guards, secret, and JSON body before persistence; durably enqueue the raw validated update before sending HTTP 200; replay pending entries during Telegram webhook startup before accepting new deliveries; remove entries only after full update handling settles, including media group buffering, text-fragment assembly, and inbound debounce work. Do not add a new user config option unless maintainers explicitly ask. Preserve existing webhook auth/rate-limit behavior and avoid persisting unauthenticated payloads. Add focused tests for enqueue-before-ACK, replay-on-startup, dequeue-after-processing, failure/restart retention, and private/safe queue file handling. Update Telegram docs and CHANGELOG.md because this is user-visible reliability behavior.
