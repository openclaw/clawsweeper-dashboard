## Summary

Keep open. Current main still acknowledges Telegram webhook requests before `bot.handleUpdate` completes and has no persisted inbound webhook queue or startup replay path; the broader catch-up issue tracks a related but different provider-history design.

## What This Changes

The issue requests durable handling for inbound Telegram webhook payloads so gateway restarts do not permanently drop already-acknowledged updates.

## Best Possible Solution

Add a Telegram-owned durable inbound webhook queue with secure local storage, bounded replay/dequeue semantics, deferred-work limits, and focused restart-loss tests while keeping the broader cross-provider catch-up design separate.

## Reproduction Assessment

Yes, source-reproducible: current main returns `200` before `bot.handleUpdate` finishes and has no persisted queue or startup replay, so a restart in that interval can drop an already-acknowledged update. I did not run a credentialed live Telegram restart E2E in this read-only review.

## Solution Assessment

Unclear as proposed. Disk-backed queueing is the right direction, but the exact storage, replay, duplicate, privacy, and deferred-work semantics should be chosen before implementation.
