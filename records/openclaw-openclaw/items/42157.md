## Label Justifications

- `P1`: Accepted Telegram webhook messages can be permanently lost during routine gateway restarts, breaking a real channel workflow.
- `impact:message-loss`: The issue is specifically about inbound Telegram messages being dropped after OpenClaw returns a successful webhook response.

## Summary

Keep open. Current main and v2026.5.12 still ACK Telegram webhooks before update handling, and the only durable Telegram ingress spool is wired to isolated polling; the linked queue PRs were closed unmerged after design and security review, so the webhook restart-loss path remains unresolved.

## What This Changes

The issue reports accepted Telegram webhook updates being lost when the gateway restarts before processing finishes.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes validated updates before ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after handler and deferred work complete.

## Reproduction Assessment

Yes. Source inspection shows a valid webhook request can receive 200 while `bot.handleUpdate` is still pending, and webhook startup has no replay path for that accepted payload.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but prior PR review and the current polling-only spool show that ACK, replay, duplicate, storage, deferred-work, and security semantics need maintainer design first.
