## Summary

Keep open. Current main still acknowledges Telegram webhook deliveries before update processing completes, and there is no durable queue or startup replay path for accepted webhook payloads, so the reported restart-loss window remains real by source inspection.

## What This Changes

The issue asks for Telegram webhook-mode inbound payloads to survive gateway restarts after OpenClaw has accepted delivery from Telegram.

## Best Possible Solution

Add a Telegram-owned durable inbound webhook queue that persists validated updates before ACK, replays pending entries before advertising the webhook on startup, deletes only after full handler/deferred completion, and includes bounded replay, dedupe, permissions, tests, and docs.

## Reproduction Assessment

Yes, by source inspection. Current main returns 200 before `bot.handleUpdate` completes, has tests asserting that behavior, and contains no webhook queue/replay path to survive a restart in that window.

## Solution Assessment

Unclear as an automatic fix. A durable plugin-owned queue is the likely direction, but the storage and replay semantics need maintainer/security review before implementation.
