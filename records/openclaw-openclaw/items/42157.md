## Label Justifications

- `P1`: Accepted Telegram webhook messages can be permanently lost during routine gateway restarts, breaking a real channel workflow.
- `impact:message-loss`: The issue is specifically about inbound Telegram messages being dropped after OpenClaw returns a successful webhook response.

## Summary

Keep open. Current main and v2026.5.19 still acknowledge Telegram webhooks before update handling and only persist inbound updates on the isolated polling path, so the accepted-webhook restart-loss path remains unresolved.

## What This Changes

The issue requests durable Telegram webhook ingress so accepted updates can be replayed after gateway restarts instead of being lost after the 200 ACK.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes validated updates before the 2XX ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after handler and deferred work complete.

## Reproduction Assessment

Yes. Source inspection shows a valid webhook request can receive 200 while `bot.handleUpdate` is still pending, and webhook startup has no replay path for that accepted payload.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but ACK, replay, duplicate, storage, deferred-work, and security semantics need maintainer design first.
