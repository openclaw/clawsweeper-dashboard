## Summary

Keep open. Current main and v2026.5.3 still ACK validated Telegram webhook updates before processing finishes, with no durable webhook queue or replay path; the prior specific persistence PRs are closed unmerged, and #55792 tracks broader offline catch-up rather than this already-ACKed webhook window.

## What This Changes

Persist validated inbound Telegram webhook payloads before ACK and replay/delete them across gateway restarts so already-acknowledged updates are not lost.

## Best Possible Solution

Add a Telegram-owned durable webhook queue that stores validated raw updates before ACK, replays pending entries during webhook startup before new deliveries, and deletes entries only after full update handling settles.

## Reproduction Assessment

Yes. Source and existing tests provide a high-confidence path: send a valid Telegram webhook update with a slow bot.handleUpdate, observe the 200 ACK before work finishes, then restart in that window with no persisted payload to replay.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Telegram plugin: validate before persisting, durably enqueue before ACK, replay on startup, and dequeue only after complete update handling; #55792 is related but not a substitute.

## Repair Work Prompt

Fix the Telegram webhook restart-loss bug from https://github.com/openclaw/openclaw/issues/42157. In the Telegram plugin only, add durable inbound webhook persistence: after request guards, secret validation, and JSON body validation, atomically store the raw Telegram update before `respondText(200)`; replay pending entries during webhook startup before accepting new deliveries; delete entries only after full `bot.handleUpdate` processing and Telegram deferred work settle. Preserve the async ACK behavior after durable enqueue, avoid core owner-specific policy, do not add a config knob unless the maintainer explicitly requests one, and stop if the fix would require broader product/API decisions. Add focused tests for ACK-before-processing plus restart replay/dequeue, malformed or orphaned queue files, path traversal/permissions, and no persistence of unauthenticated payloads. Update Telegram docs and CHANGELOG.md because this is user-facing reliability behavior.
