## Summary

Keep open. Current main still acknowledges Telegram webhook deliveries before update processing finishes, and the durable Telegram ingress spool is wired to isolated polling rather than webhook receipt, so the in-flight webhook restart loss remains unresolved and distinct from the broader catch-up request.

## What This Changes

The issue requests durable persistence and replay for accepted inbound Telegram webhook payloads so gateway restarts do not silently drop them.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes validated updates before ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after handler and deferred work complete.

## Reproduction Assessment

Yes, at source level. A valid Telegram webhook request can receive 200 while bot.handleUpdate is still running, and webhook startup has no replay path for that accepted payload.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but the closed PR attempts and current spool design show that replay bounds, deferred work, duplicates, and storage security need maintainer design first.
