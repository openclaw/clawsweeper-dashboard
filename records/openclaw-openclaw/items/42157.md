## Label Justifications

- `P1`: The issue describes inbound Telegram messages being silently lost during ordinary gateway restart paths.
- `impact:message-loss`: The affected behavior is lost or unreplayed inbound Telegram webhook payloads after OpenClaw has already acknowledged delivery.

## Summary

Keep open. Current main and v2026.6.10 still acknowledge Telegram webhook payloads before durable processing, the durable Telegram ingress queue is wired to isolated polling rather than webhook mode, and the direct fix PRs are closed unmerged.

## What This Changes

Persist authenticated Telegram webhook updates before acknowledging them and replay unfinished updates after gateway restarts.

## Best Possible Solution

Wire webhook mode into the existing shared SQLite channel ingress queue: persist authenticated updates before 200 OK, replay pending claims on startup, and complete only after the maintainer-approved processing/deferred-work boundary.

## Reproduction Assessment

Yes. Source inspection shows the webhook handler sends 200 before bot.handleUpdate, and the current test suite asserts processing can remain unfinished after ACK.

## Solution Assessment

Unclear for implementation shape. A durable queue is the right direction, but it should use the existing SQLite channel ingress queue and needs maintainer decisions on replay, duplicate handling, and completion semantics.
