## Summary

Keep open. Current main still acknowledges validated Telegram webhook payloads before durable handoff or processing completion, the tests and docs still describe that async post-ACK behavior, and the prior queue PRs are closed unmerged. The broader open catch-up issue does not directly replace this Telegram-specific webhook persistence gap.

## What This Changes

Persist Telegram webhook payloads across gateway restarts so ACKed inbound updates can be replayed instead of permanently lost.

## Best Possible Solution

Keep this issue open and fix the Telegram-owned webhook path in `extensions/telegram`: durably enqueue the validated raw update before returning 200, replay pending entries during webhook startup before accepting new deliveries, and remove queue entries only after the update and associated deferred Telegram work settle. Coordinate with #55792, but do not fold this narrower persistence bug into the broader cross-channel catch-up request.

## Reproduction Assessment

Yes. The current code and tests provide a high-confidence reproduction path: send a valid Telegram webhook update with slow bot.handleUpdate work, observe the 200 ACK before processing finishes, then kill or restart the gateway during that window with no durable queue to replay on startup.

## Solution Assessment

Yes, the narrowest maintainable fix is a Telegram-owned durable queue before ACK plus startup replay and post-processing dequeue. A broad provider history catch-up feature is complementary, but it would not by itself cover payloads already ACKed to Telegram and then lost in-process.
