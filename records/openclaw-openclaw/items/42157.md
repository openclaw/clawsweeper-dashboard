## Label Justifications

- `P1`: Accepted Telegram webhook messages can be permanently lost during routine gateway restarts, breaking a real channel workflow.
- `impact:message-loss`: The issue is specifically about inbound Telegram messages being dropped after OpenClaw returns a successful webhook response.

## Summary

Keep open. Current main still acknowledges valid Telegram webhook updates before durable processing or replay, the durable ingress spool is wired through polling rather than webhook startup, and the linked webhook-queue attempts are closed unmerged while https://github.com/openclaw/openclaw/issues/55792 tracks a broader missed-message problem rather than this accepted-payload window.

## What This Changes

The issue reports that Telegram webhook updates can be acknowledged before durable processing, so a gateway restart can permanently drop accepted inbound messages.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes validated updates before the 2xx ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes entries only after handler and deferred work complete.

## Reproduction Assessment

Yes, source-level. Current code and tests show a valid webhook request can receive 200 while `bot.handleUpdate` is still pending, and webhook startup has no replay path for that accepted payload.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but ACK, replay, duplicate, storage, deferred-work, and payload-security semantics need maintainer design first.

## Risks / Open Questions

- Persisting webhook payloads can put chat content on disk, so storage location, file permissions, retention, and redaction expectations need an owner decision before implementation.
- At-least-once replay can duplicate Telegram updates if the gateway dies after dispatch but before cleanup, so dedupe and deferred-work completion semantics must be specified rather than patched ad hoc.
- The broader open catch-up issue at https://github.com/openclaw/openclaw/issues/55792 overlaps on user impact but does not by itself replace pre-ACK persistence for already-accepted webhook payloads.
