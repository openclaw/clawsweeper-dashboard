## Summary

Keep open. Current main and the latest release still ACK Telegram webhook deliveries before update processing completes, with no durable webhook receipt queue or startup replay path; the closed queue PR chain is useful design context but was not merged.

## What This Changes

The issue asks for Telegram webhook updates to be persisted and replayed so gateway restarts after a 200 ACK do not permanently lose accepted payloads.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes validated updates before ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after handler and deferred work complete.

## Reproduction Assessment

Yes. Source inspection shows the webhook handler returns 200 before bot.handleUpdate completes, the test suite asserts that timing, and webhook startup has no replay path for accepted payloads.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue or carefully reused Telegram ingress spool is the likely direction, but storage, replay, duplicate, retention, and security semantics need maintainer design first.
