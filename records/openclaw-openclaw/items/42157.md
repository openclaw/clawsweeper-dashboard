## Label Justifications

- `P1`: The issue describes silent loss of inbound Telegram messages during routine gateway restart windows, which can break real channel workflows.
- `impact:message-loss`: The central impact is Telegram webhook updates being acknowledged externally but not delivered to the agent after a restart.

## Summary

Keep open. Current main still returns Telegram webhook 200 before processing and does not durably spool webhook-mode updates, while the existing SQLite ingress spool is wired to isolated polling only.

## What This Changes

The issue asks for inbound Telegram webhook updates to survive gateway restarts instead of being lost after Telegram receives the webhook ACK.

## Best Possible Solution

Integrate authenticated Telegram webhook receipt with the existing SQLite-backed channel ingress queue before ACK, then replay and complete/dead-letter through Telegram-owned processing with explicit dedupe, retention, and backpressure semantics.

## Reproduction Assessment

Yes, at source level. Current main returns 200 before `bot.handleUpdate(...)` and has no durable webhook-mode write before ACK; I did not run a live Telegram restart E2E.

## Solution Assessment

Unclear as previously proposed. The problem is real, but the older file-backed PR shape is not the best current fit; the safer direction is a Telegram-owned integration with the existing SQLite ingress queue and explicit replay semantics.
