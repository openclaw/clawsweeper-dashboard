## Label Justifications

- `P1`: Accepted Telegram webhook messages can be permanently lost during routine gateway restarts, breaking a real channel workflow.
- `impact:message-loss`: The issue is specifically about inbound Telegram messages being dropped after OpenClaw returns a successful webhook response.

## Summary

Keep open. Current main and the latest shipped tag still acknowledge valid Telegram webhook updates before processing completion, while durable Telegram ingress spooling is wired through polling only; the webhook-queue PR attempts linked from the issue are closed unmerged.

## What This Changes

The issue requests durable handling for Telegram webhook updates so payloads accepted with 200 OK are not lost if the gateway restarts before processing completes.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes authenticated, validated updates before the 2xx ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after handler and deferred work complete.

## Reproduction Assessment

Yes, source-level. Current code and tests show a valid webhook request can receive 200 while bot.handleUpdate is still pending, and webhook startup has no replay path for that accepted payload.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but ACK, replay, duplicate, storage, deferred-work, and payload-security semantics need maintainer design first.
