## Label Justifications

- `P1`: Silently losing inbound Telegram webhook messages during normal gateway restarts breaks an active channel workflow for real users.
- `impact:message-loss`: The reported failure mode is Telegram inbound updates being acknowledged and then permanently lost before agent delivery.

## Summary

Keep open. Current main and the latest release still acknowledge Telegram webhook updates before asynchronous handling, while the durable Telegram ingress spool is wired to isolated polling rather than webhook receipt; the closed contributor PRs were not merged and their old JSON-sidecar queue shape is no longer the best fit.

## What This Changes

The issue requests durable Telegram webhook receipt so updates acknowledged before processing are not lost when the gateway restarts.

## Best Possible Solution

Wire Telegram webhook receipt into the existing SQLite-backed ingress queue after auth/body validation and before the 200 response, then complete or delete the queued update only after update handling and deferred work settle.

## Reproduction Assessment

Yes, at source level: the webhook handler returns 200 before asynchronous update processing, and the durable Telegram ingress spool is not called from the webhook receipt path. I did not run a live Telegram restart test in this read-only review.

## Solution Assessment

No for the old closed JSON-file queue proposals; current main already has a SQLite-backed channel ingress queue, and AGENTS.md says OpenClaw-owned runtime queues belong in SQLite. The best fix needs maintainer-chosen webhook enqueue, replay, and dequeue semantics on that existing queue.
