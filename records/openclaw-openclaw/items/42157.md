## Label Justifications

- `P1`: Accepted Telegram webhook messages can be permanently lost during routine gateway restarts, breaking a real channel workflow.
- `impact:message-loss`: The issue is specifically about inbound Telegram messages being dropped after OpenClaw returns a successful webhook response.

## Summary

Keep open. Current main and v2026.5.20 still acknowledge Telegram webhook updates before handling them, and the durable Telegram update spool is only wired through isolated polling; the linked webhook-queue PRs are closed unmerged, so the reported message-loss window remains.

## What This Changes

The issue reports that Telegram webhook payloads are acknowledged with 200 before processing or persistence, so a gateway restart after the ACK can permanently drop inbound updates.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes validated updates before the 2XX ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after handler and deferred work complete.

## Reproduction Assessment

Yes, source-level. Current code and tests show a valid webhook request can receive 200 while `bot.handleUpdate` is still pending, and webhook startup has no replay path for that accepted payload.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but ACK, replay, duplicate, storage, deferred-work, and payload-security semantics need maintainer design first.
