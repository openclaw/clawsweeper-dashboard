## Summary

Keep open. Current main still acknowledges Telegram webhook requests before update processing completes, and the durable ingress spool now present in Telegram is wired to isolated polling rather than webhook receipt, so this in-flight webhook restart loss remains valid and distinct from the broader missed-downtime catch-up issue.

## What This Changes

The issue requests durable handling for accepted Telegram webhook payloads so gateway restarts cannot lose updates after a 200 ACK but before processing finishes.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes validated updates before ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after handler and deferred work complete.

## Reproduction Assessment

Yes. Source inspection shows the webhook handler returns 200 before bot.handleUpdate completes, the test suite asserts that timing, and webhook startup has no replay path for accepted payloads.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue or carefully reused spool is the likely direction, but the storage, replay, duplicate, and security semantics need maintainer design first.
