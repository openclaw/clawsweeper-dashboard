## Summary

Keep open. Current main still returns 200 for Telegram webhook deliveries before `bot.handleUpdate` completes, and there is no durable webhook queue or startup replay path; the prior persistence PRs were closed unmerged and the broader missed-message catch-up issue does not replace this in-flight webhook durability gap.

## What This Changes

Persist and replay accepted Telegram webhook updates so gateway restarts cannot lose payloads after Telegram receives a successful webhook response.

## Best Possible Solution

Add a Telegram-owned durable inbound webhook queue that writes validated updates before ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after the handler and deferred work are complete.

## Reproduction Assessment

Yes, by source inspection. Current main returns 200 before `bot.handleUpdate` completes, has a test asserting that timing, and has no durable queue or startup replay path for accepted webhook payloads.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but storage, replay bounds, duplicate handling, and failure semantics need maintainer/security review before implementation.
