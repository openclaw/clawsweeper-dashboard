## Summary

Keep open. Current main and the latest release still ACK Telegram webhook deliveries before update processing completes, and there is no durable webhook queue or startup replay path; the closed persistence PRs remain useful context but were not merged, and https://github.com/openclaw/openclaw/issues/55792 covers broader downtime catch-up rather than this accepted-in-flight webhook durability gap.

## What This Changes

Request Telegram webhook mode to persist accepted inbound updates and replay them after gateway restart so ACKed payloads are not lost.

## Best Possible Solution

Add a Telegram-plugin-owned durable inbound webhook queue that writes validated updates before ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after handler and deferred work complete.

## Reproduction Assessment

Yes, by source inspection. Current main returns 200 before bot.handleUpdate completes, has a test asserting that timing, and has no durable queue or startup replay path for accepted webhook payloads.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but storage, replay bounds, duplicate handling, and failure semantics need maintainer/security review before implementation.
