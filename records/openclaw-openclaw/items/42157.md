## Label Justifications

- `P1`: Accepted Telegram webhook messages can be silently lost across gateway restarts, which is a broken channel workflow affecting real users.
- `impact:message-loss`: The report is specifically about inbound Telegram messages being dropped after OpenClaw has acknowledged receipt to Telegram.

## Summary

Keep open. Current main and the latest checked release still acknowledge Telegram webhooks before durable receipt or completed processing, while the existing SQLite-backed Telegram ingress spool is wired to isolated polling rather than webhook mode.

## What This Changes

Persist authenticated Telegram webhook payloads before acknowledging or processing them, then replay pending updates after gateway restart so accepted inbound messages are not lost.

## Best Possible Solution

Add a Telegram-owned webhook durable ingress path, preferably reusing the existing SQLite channel ingress queue/spool, that stores authenticated updates before ACK, replays pending entries on startup, and completes entries only after safe processing including deferred work.

## Reproduction Assessment

Yes. Source and tests show Telegram webhook mode returns 200 before asynchronous `bot.handleUpdate`, so a restart after ACK but before durable handling can drop an accepted update.

## Solution Assessment

Unclear. The problem is real, but the best fix should use the existing SQLite-backed channel ingress/spool ownership rather than adding a new file sidecar or second persistence path.
