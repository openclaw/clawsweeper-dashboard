## Summary

Keep open. Current main and the latest shipped release still acknowledge Telegram webhook deliveries before update processing completes, and the durable Telegram ingress spool is only wired to isolated polling, so the in-flight webhook restart-loss path remains unresolved.

## What This Changes

The issue requests durable Telegram webhook ingress so payloads accepted before a gateway restart can be replayed instead of permanently lost.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes validated updates before ACK, replays pending entries on startup with bounded at-least-once semantics, and deletes only after handler and deferred work complete.

## Reproduction Assessment

Yes. Source inspection shows a valid webhook request can receive 200 while `bot.handleUpdate` is still pending, and webhook startup has no replay path for that accepted payload.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but prior PR review and the current polling-only spool show the ACK, replay, duplicate, storage, and deferred-work semantics need maintainer design first.
