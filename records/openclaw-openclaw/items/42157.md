## Summary

Keep open. Current main and the latest release still acknowledge validated Telegram webhook payloads before processing finishes and have no durable Telegram inbound replay path; the prior persistence PRs are closed unmerged, while #55792 is broader catch-up work that does not cover already-ACKed webhook payloads.

## What This Changes

Telegram webhook mode should persist validated inbound updates before acknowledging them and replay pending updates after a gateway restart.

## Best Possible Solution

Add a Telegram-owned durable webhook queue that stores validated raw updates before ACK, replays pending entries during webhook startup before new deliveries, and deletes entries only after the full update handling path settles.

## Reproduction Assessment

Yes. Source and existing tests provide a high-confidence path: send a valid Telegram webhook update with a slow bot.handleUpdate, observe the 200 ACK before work finishes, then restart in that window with no persisted payload to replay.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Telegram plugin: validate before persist, durably enqueue before ACK, replay on startup, and dequeue after processing; broad provider catch-up in #55792 is complementary but does not recover already-ACKed webhook payloads.
