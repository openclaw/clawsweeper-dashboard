## Label Justifications

- `P1`: Silent Telegram webhook loss during routine gateway restarts can break active channel workflows for real users.
- `impact:message-loss`: The reported failure is inbound Telegram messages being acknowledged and then lost before reaching the agent.

## Summary

Keep open. Current main and v2026.6.6 still acknowledge Telegram webhook deliveries before durable processing, while the existing SQLite-backed Telegram ingress spool is wired to isolated polling rather than webhook receipt.

## What This Changes

The issue asks OpenClaw to stop losing Telegram webhook updates that are acknowledged before processing survives a gateway restart.

## Best Possible Solution

Use the existing SQLite-backed channel ingress queue for authenticated webhook updates before returning 2xx, then replay and complete them with bounded Telegram-owned dequeue semantics.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 return 200 from `startTelegramWebhook` before `bot.handleUpdate` finishes, and the durable Telegram ingress spool writer is only wired through isolated polling, not webhook receipt.

## Solution Assessment

No final fix is present yet. The old standalone JSON file queue from the closed PRs is not the best fit under current policy; the safer direction is reusing the existing SQLite-backed ingress queue with maintainer-approved ACK, replay, and dequeue rules.
