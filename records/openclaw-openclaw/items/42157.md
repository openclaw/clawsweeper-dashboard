## Summary

Keep open. Current main still acknowledges Telegram webhook deliveries before processing and has no durable inbound webhook queue or startup replay path, while the related fix PRs were closed unmerged after review concerns.

## What This Changes

Persist inbound Telegram webhook payloads across gateway restarts so acknowledged updates can be replayed instead of lost.

## Best Possible Solution

Add a Telegram-owned durable webhook receipt/replay design with bounded backlog, dedupe/at-least-once semantics, safe local storage, docs, focused tests, and real Telegram transport proof; keep the broader cross-provider catch-up design tracked separately.

## Reproduction Assessment

Yes, by source inspection: current webhook mode returns 200 before `bot.handleUpdate` and has no durable queue or replay path. I did not run a live credentialed Telegram restart E2E.

## Solution Assessment

Unclear as previously proposed. A Telegram-local persistence layer is the likely right boundary, but the replacement implementation needs maintainer decisions on replay bounds, dedupe, file safety, and deferred-work completion before it is the best fix.
