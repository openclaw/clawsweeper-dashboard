## Label Justifications

- `P1`: The issue concerns inbound channel messages that can disappear during normal gateway restarts.
- `impact:message-loss`: The reported failure mode is silent loss of already-ACKed Telegram webhook payloads.

## Summary

Keep open. Current main and `v2026.6.10` still ACK Telegram webhook updates before durable processing, while the direct fix PRs are closed unmerged and the broader missed-message catch-up issue does not replace this narrower webhook durability gap.

## What This Changes

Issue requests durable handling for Telegram webhook payloads so gateway restarts do not silently drop already-ACKed inbound updates.

## Best Possible Solution

Wire webhook mode into a durable Telegram ingress queue that records authenticated updates before sending 200, replays them on startup, and deletes them only after clearly defined at-least-once processing completion.

## Reproduction Assessment

Yes by source inspection: `extensions/telegram/src/webhook.ts` responds 200 before `bot.handleUpdate`, and `extensions/telegram/src/webhook.test.ts` asserts work can still be unfinished after ACK. I did not run a live Telegram restart E2E in this read-only review.

## Solution Assessment

Unclear as an autonomous fix lane. A durable queue is the likely direction, but the exact webhook replay, dedupe, stale-claim, and storage semantics need maintainer confirmation before implementation.
