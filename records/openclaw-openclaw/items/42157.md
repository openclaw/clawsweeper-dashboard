## Label Justifications

- `P1`: Accepted Telegram webhook messages can be permanently lost during routine gateway restarts, breaking a real channel workflow.
- `impact:message-loss`: The issue is specifically about inbound Telegram messages being dropped after OpenClaw returns a successful webhook response.

## Summary

Keep open. Current main and the latest release still acknowledge valid Telegram webhook updates before processing completes, the only durable Telegram ingress spool is wired through long polling, and all issue-linked webhook persistence PRs are closed unmerged; the broader catch-up issue at https://github.com/openclaw/openclaw/issues/55792 is related but does not replace this Telegram webhook-specific accepted-payload durability gap.

## What This Changes

The issue asks OpenClaw to avoid losing accepted inbound Telegram webhook updates when the gateway restarts before processing completes.

## Best Possible Solution

Add a Telegram-plugin-owned durable webhook ingress queue that writes validated updates before the 2xx ACK, replays pending entries on startup with bounded at-least-once semantics, and removes entries only after the handler and any deferred work complete.

## Reproduction Assessment

Yes, source-level. Current code and tests show a valid Telegram webhook request can receive 200 while `bot.handleUpdate` is still pending, and webhook startup has no replay path for that accepted payload.

## Solution Assessment

Unclear as an automatic fix. A plugin-owned durable queue is the likely direction, but ACK, replay, duplicate, storage, deferred-work, and payload-security semantics need maintainer design first.
