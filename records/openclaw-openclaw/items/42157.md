## Label Justifications

- `P1`: The report describes a real channel workflow where inbound Telegram messages can be silently lost during routine gateway restarts after Telegram treats the webhook delivery as successful.
- `impact:message-loss`: The affected problem class is dropped inbound Telegram webhook messages between ACK and completed OpenClaw processing.

## Summary

Keep open. Current main and the latest release still ACK Telegram webhook updates before durable processing, while the only exact fix attempts are closed unmerged and the broader restart catch-up issues are adjacent rather than replacements.

## What This Changes

The issue asks OpenClaw to prevent inbound Telegram webhook updates from being permanently lost when the gateway restarts after Telegram has received a successful webhook response.

## Best Possible Solution

Keep this issue open until Telegram webhook mode records authenticated updates durably before the 2xx ACK and replays or completes them through a maintainer-approved SQLite-backed queue with explicit dedupe and failure semantics.

## Reproduction Assessment

Yes, at source level: current main returns 200 from `startTelegramWebhook` before `bot.handleUpdate(...)` finishes, and no webhook-mode durable enqueue or startup replay path exists. I did not run a live Telegram restart E2E in this read-only review.

## Solution Assessment

Unclear as previously proposed: durable receipt is the right direction, but the older JSON queue PR shape is not the best current fit. The safer solution should integrate Telegram webhook receipt with the existing SQLite-backed channel ingress queue and define replay/dedupe semantics before implementation.
