## Label Justifications

- `P1`: Routine gateway restarts can silently and permanently drop accepted Telegram messages in a real user workflow.
- `impact:message-loss`: The issue directly concerns inbound messages acknowledged by Telegram but never delivered to the agent.

## Summary

Current main still acknowledges valid Telegram webhook updates before processing them in a detached task, and shutdown neither persists nor drains that work, so the reported restart-loss window remains. The open channel catch-up proposal covers messages sent during downtime rather than already-accepted webhook updates, while all three linked persistence PRs closed unmerged. Likely related people include @steipete, Harold Hunt, and Vincent Koc based on merged history around Telegram webhook restart behavior, request handling, and the current module.

## What This Changes

Durably hand off accepted Telegram webhook updates so gateway restarts cannot lose them before processing completes.

## Best Possible Solution

Add a Telegram-plugin-owned SQLite ingress queue that records each validated update before acknowledgment, deletes it only after the complete logical update settles, replays pending entries before accepting new traffic, and explicitly tests at-least-once delivery and restart-safe deduplication.

## Reproduction Assessment

Yes, at source level: submit a valid update, let the handler return HTTP 200, and terminate the gateway before the detached `bot.handleUpdate` task completes. Current main has no durable record to replay afterward.

## Solution Assessment

The durable-delivery objective is the right solution, but the closed file-per-update proposals are not the best current implementation; repository policy and existing SDK seams favor plugin-owned SQLite state with explicit failure and deduplication contracts.

## Risks / Open Questions

- A durable inbox changes the effective guarantee toward at-least-once delivery, so restart-persistent deduplication and completion ownership must prevent duplicate agent replies.
- Failing open when SQLite persistence fails retains a message-loss mode, while failing closed makes webhook availability depend on local state health; maintainers must choose that behavior explicitly.
- The broader downtime catch-up work may later require compatible cursor and deduplication semantics, although it does not replace this fix.
