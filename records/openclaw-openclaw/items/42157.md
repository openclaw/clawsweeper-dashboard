## Summary

Keep open. Current main still returns `200` before `bot.handleUpdate` completes and has no Telegram webhook persistence or startup replay path; the broader catch-up issue is related but does not cover already-ACKed, in-flight webhook payloads.

## What This Changes

The issue requests durable handling for Telegram webhook updates that have already been acknowledged but are still queued or processing when the gateway restarts.

## Best Possible Solution

Add a Telegram-owned durable inbound webhook queue with secure local storage, bounded replay/dequeue semantics, deferred-work limits, duplicate handling, and focused restart-loss tests while keeping broader provider-history catch-up separate.

## Reproduction Assessment

Yes, source-reproducible: current main returns `200` before `bot.handleUpdate` finishes and has no persisted queue or startup replay, so a restart in that interval can drop an already-acknowledged update. I did not run a credentialed live Telegram restart E2E in this read-only review.

## Solution Assessment

Unclear as proposed. Disk-backed queueing is the right direction, but storage hardening, replay/dequeue timing, duplicate behavior, privacy, and deferred-work bounds need maintainer design before implementation.
