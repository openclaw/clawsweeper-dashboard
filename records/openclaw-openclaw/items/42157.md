## Label Justifications

- `P1`: Webhook ACK-before-durable-processing can silently lose inbound user messages during normal gateway restarts.
- `impact:message-loss`: The issue is specifically about lost Telegram inbound webhook payloads.

## Summary

Keep open. Current main and latest release still show Telegram webhook mode returning 200 before asynchronous update handling, while the durable Telegram ingress spool is wired to isolated polling rather than webhook receipt.

## What This Changes

The issue requests durable Telegram webhook payload handling so already-acknowledged inbound updates can be replayed after a gateway restart.

## Best Possible Solution

Extend the existing SQLite-backed channel ingress queue to Telegram webhook mode: enqueue after auth/body validation and before 200 OK, then claim, drain, and delete only after update handling and deferred work settle.

## Reproduction Assessment

Yes, at source level: current webhook source sends 200 before async handling and I did not find a webhook enqueue into the durable Telegram ingress spool. I did not run a live Telegram restart repro in this read-only review.

## Solution Assessment

No. The old closed file-backed JSON queue PRs are not the best fit now that current main has a SQLite-backed channel ingress queue and AGENTS.md says runtime queues should live there.
