## Label Justifications

- `P1`: The issue describes silent loss of inbound Telegram messages during routine gateway restarts, which can break active real-user channel workflows.
- `impact:message-loss`: The reported impact is inbound Telegram webhook updates being acknowledged and then lost before reaching the agent.

## Summary

Keep open. Current main and v2026.6.8 still acknowledge authenticated Telegram webhooks before durable processing, while the existing durable Telegram ingress queue is wired through isolated polling rather than webhook receipt; the linked fix PRs are closed unmerged.

## What This Changes

The issue asks OpenClaw to preserve inbound Telegram webhook updates across gateway restarts instead of losing updates after Telegram has received a 2xx webhook response.

## Best Possible Solution

Keep this issue open and design a Telegram-owned durable webhook receive path, likely reusing the SQLite-backed channel ingress queue with explicit ACK, replay, dequeue, retention, and failure semantics.

## Reproduction Assessment

Yes, at source level: current main returns 200 from `startTelegramWebhook` before `bot.handleUpdate` finishes, and the test suite explicitly preserves that behavior. I did not run a live Telegram restart E2E in this read-only review.

## Solution Assessment

No current fix is present. The older sidecar JSON queue PR direction is not the best current shape because main now has a SQLite-backed channel ingress queue and repository policy prefers canonical state storage over new sidecar files.
