## Summary

Keep this PR open. Current main still lacks a plugin-facing resolved-tools hook/API, and the related feature issue was closed in favor of this PR, but the current branch still has a concrete P2 correctness issue in the new hook context.

## What This Changes

The PR adds an `after_tools_resolved` plugin hook, resolved tool/schema snapshots in the embedded runner, SDK hook types, docs, tests, baseline hashes, and a changelog entry.

## Best Possible Solution

Land an additive resolved-tools API after the hook uses the shared agent context contract, schema snapshots remain cloned and documented, tests cover the contract, and maintainers approve the push-hook design.

## Reproduction Assessment

Not applicable for a feature PR. Source inspection confirms current main lacks the resolved-tools hook/API, and the PR body provides live WhatsApp gateway logs for the proposed after-fix behavior.

## Solution Assessment

No, not as-is. The hook and snapshot approach is plausible, but the patch should reuse `buildAgentHookContextChannelFields(params)` so `after_tools_resolved` correlates with the existing agent hook context contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the shared hook channel context fields:** `src/agents/pi-embedded-runner/run/attempt.ts:1693`
  - body: The new hook builds `ctx.channelId` from `params.messageChannel ?? params.messageProvider`, while the other agent hooks use `buildAgentHookContextChannelFields(params)` to derive the real conversation id from `sessionKey`, channel metadata, and message targets. In channel sessions this can make `after_tools_resolved` disagree with `before_prompt_build`, `llm_input`, and `agent_end`, so plugins cannot reliably correlate the resolved tool snapshot with the rest of the run.
  - confidence: 0.86
