## Summary

Keep this PR open. Current main still lacks an `after_tools_resolved` hook or equivalent plugin-facing resolved-tools API, and the PR remains the canonical implementation path, but the current head still has a concrete hook-context consistency bug before merge.

## What This Changes

The branch adds an `after_tools_resolved` plugin hook, emits resolved tool metadata during embedded agent attempts, documents it, refreshes SDK hash/changelog metadata, and adds focused tests.

## Best Possible Solution

Land the additive resolved-tools hook after it uses the standard hook context contract and the API boundary is accepted, or supersede it with a pull-based resolved-tools API that provides equivalent schema snapshots.

## Reproduction Assessment

Not applicable for the feature request itself. For the PR-head defect, yes: source inspection shows `agent:main:discord:channel:c1` would report `ctx.channelId` as `discord` in the new hook while existing shared hook context reports `c1`.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the emission should reuse `buildAgentHookContextChannelFields(params)` so plugins receive consistent routing metadata across hook phases.
