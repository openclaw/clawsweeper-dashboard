## Summary

Keep this PR open: current main still lacks a normalized resolved-tools hook or pull API, and the related feature issue was closed in favor of this PR. The branch is not ready to merge because the new hook builds `ctx.channelId` differently from the shared agent hook context helper, which would give plugins inconsistent routing metadata across hook phases.

## What This Changes

The PR adds an `after_tools_resolved` plugin hook for embedded agent attempts, documents it, refreshes SDK baseline/changelog data, and adds focused tests for resolved tool metadata and hook dispatch.

## Best Possible Solution

Land an additive resolved-tools hook only after it reuses the shared agent hook context helper and the final API contract is maintainer-approved, or supersede it with a pull-based resolved-tools API that exposes stable schema snapshots.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking PR-head defect is source-reproducible: the new hook sets `channelId` manually while adjacent agent hooks use `buildAgentHookContextChannelFields(params)`.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the emission should reuse the shared channel context helper so plugins receive consistent routing metadata across hook phases.
