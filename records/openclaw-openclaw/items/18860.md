## Summary

Keep this PR open, but it still needs a narrow correctness fix before merge: the new hook is useful and current main still lacks an equivalent resolved-tools API, while PR head emits an inconsistent `channelId` compared with the standard hook context contract.

## What This Changes

The PR adds an `after_tools_resolved` plugin hook, emits resolved tool metadata during embedded agent attempts, updates plugin hook docs/API baseline/changelog, and adds focused tests.

## Best Possible Solution

Land the additive resolved-tools hook after it reuses the standard hook context contract and the plugin API boundary is accepted, or explicitly supersede it with an equivalent pull-based resolved-tools API.

## Reproduction Assessment

Not applicable for the feature request itself. For the PR-head defect, yes: source inspection shows a session key like `agent:main:discord:channel:c1` would report `ctx.channelId` as `discord` in the new hook while existing shared hook context reports `c1`.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the emission should reuse `buildAgentHookContextChannelFields(params)` so plugins receive consistent routing metadata across hook phases.
