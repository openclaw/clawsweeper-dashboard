## Summary

Keep this PR open, but not mergeable as-is. Current main only has partial related visibility through `llm_input.tools`; it still lacks the proposed dedicated `after_tools_resolved` hook, and the PR head still reports inconsistent channel context for the new hook.

## What This Changes

This PR adds an `after_tools_resolved` plugin hook, emits deep-cloned resolved tool schema metadata during embedded attempts, documents it, refreshes the SDK baseline, and adds focused tests.

## Best Possible Solution

Land the additive resolved-tools hook after reusing the shared agent hook context helper, reconciling it with `llm_input.tools`, and preserving the schema snapshot docs/tests.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking review finding is source-reproducible from the PR head: the new hook hand-builds `channelId` while existing agent hooks use the shared helper.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the emitted context should use `buildAgentHookContextChannelFields(params)` so the new hook matches established agent hook routing semantics.
