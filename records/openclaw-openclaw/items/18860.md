## Summary

Keep this PR open. Current main still lacks the resolved-tools plugin hook/API, and the PR is useful, but the current head still builds the new hook context with a manual `channelId` value that can disagree with existing agent hook contexts.

The branch adds an `after_tools_resolved` plugin hook, emits cloned resolved-tool metadata from embedded attempts, and updates plugin docs, tests, API baseline, and changelog.

## Best Possible Solution

Land the additive resolved-tools plugin API after the hook reuses the shared agent context helper, preserves cloned schema snapshots, and receives maintainer approval for the public contract.

## Reproduction Assessment

## Solution Assessment

Is this the best way to solve the issue? No, not as-is; the additive hook is plausible, but the new hook context should use `buildAgentHookContextChannelFields(params)` so it matches existing agent hooks.
