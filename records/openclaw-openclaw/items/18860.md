## Summary

Keep open. Current main still lacks `after_tools_resolved` or a pull-based resolved-tools API, and the PR remains useful but still has a blocking mutable-schema exposure plus a missing changelog entry before it is merge-ready.

## What This Changes

The PR adds a plugin-facing `after_tools_resolved` hook, embedded agent-run emission of resolved tool metadata, SDK API baseline/docs updates, and focused hook tests.

## Best Possible Solution

Land an additive resolved-tools API only after it exposes immutable schema snapshots, documents retry/snapshot semantics, keeps SDK baseline/docs/changelog aligned, and gets maintainer approval for the public hook shape.

## Reproduction Assessment

Yes for the review concern. Current main lacks the hook/API, and the PR defect is source-reproducible from the diff because it publishes `tool.parameters` by reference before the provider/session path consumes the same schema object.

## Solution Assessment

No, not yet. The hook is a narrow maintainable direction, but an observation-only API should publish cloned or immutable schema snapshots and include the required changelog entry before merge.
