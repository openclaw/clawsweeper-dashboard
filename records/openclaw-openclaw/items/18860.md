## Summary

Keep open: current main still lacks a resolved-tools plugin API, and this PR remains the active implementation path with sufficient live log proof, but the patch still has a channel-context mismatch that should be fixed before merge.

## What This Changes

The PR adds an `after_tools_resolved` plugin hook, embedded-runner resolved-tool metadata snapshots, SDK hook types, docs, tests, baseline hash, and a changelog entry.

## Best Possible Solution

Land an additive resolved-tools API only after the hook context matches existing agent hooks, schema snapshots stay cloned and documented, tests cover the contract, and maintainers approve the push-hook design.

## Reproduction Assessment

Not applicable for a feature PR. Source inspection confirms current main lacks the hook/API, and the PR body provides live gateway logs for the proposed after-fix behavior.

## Solution Assessment

No, not as-is. The hook and snapshot approach is plausible, but the patch should reuse the shared agent hook channel-context helper and still needs maintainer approval on the public SDK shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the shared hook channel context fields:** `src/agents/pi-embedded-runner/run/attempt.ts:1693`
  - body: The new hook builds `ctx.channelId` from `params.messageChannel ?? params.messageProvider`, while the other agent hooks use `buildAgentHookContextChannelFields(params)` to derive the real conversation id from `sessionKey`, channel metadata, and message targets. In channel sessions this can make `after_tools_resolved` disagree with `before_prompt_build`, `llm_input`, and `agent_end`, so plugins cannot reliably correlate the resolved tool snapshot with the rest of the run.
