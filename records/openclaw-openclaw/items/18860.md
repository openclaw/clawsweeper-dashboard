## Summary

Keep this PR open. Current main still lacks a plugin-facing resolved tool/schema API, and the PR is the canonical implementation path for the closed linked feature request, but the latest head still has one channel-context correctness issue before merge.

## What This Changes

The PR adds an `after_tools_resolved` plugin hook, emits cloned resolved tool/schema metadata from embedded agent attempts, and updates SDK hook types, docs, tests, generated baselines, and changelog.

## Best Possible Solution

Land an additive resolved-tools plugin API after the new hook reuses the shared agent context contract, preserves cloned schema snapshots, and receives maintainer approval for the final API shape.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable for a feature PR; source inspection on current main confirms the hook/API is absent, and the PR body includes live WhatsApp gateway logs for the proposed after-fix behavior.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is; the additive hook is plausible, but it should reuse `buildAgentHookContextChannelFields(params)` so its context matches existing agent hooks before maintainers settle the push-hook API decision.
