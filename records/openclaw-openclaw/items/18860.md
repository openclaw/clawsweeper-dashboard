## Summary

Keep open: current main still lacks a dedicated final per-attempt resolved-tools plugin hook, and this PR remains useful but has a concrete context bug that should be fixed before merge.

## What This Changes

The PR adds an `after_tools_resolved` plugin hook, emits resolved tool metadata from embedded attempts, documents the hook, updates the SDK API baseline/changelog, and adds focused tests.

## Best Possible Solution

Land the additive resolved-tools hook after reusing the shared agent hook context helper, reconciling the contract with `llm_input.tools`, preserving schema snapshots/docs/tests, and getting maintainer approval for the plugin API boundary.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking review finding is source-reproducible from the PR head: the new hook builds `channelId` manually while existing agent hooks use the shared helper.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the emitted context should use `buildAgentHookContextChannelFields(params)` so `channelId` and `messageProvider` match existing agent hook semantics.
