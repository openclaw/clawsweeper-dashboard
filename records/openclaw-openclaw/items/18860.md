## Summary

Keep this PR open: current main still lacks a dedicated resolved-tools hook or pull API, but the PR head still has a source-reproducible context bug that should block merge until fixed.

## What This Changes

The PR adds an observation-only `after_tools_resolved` plugin hook, emits resolved tool metadata from embedded agent attempts, documents the hook, refreshes the SDK baseline, and adds focused tests.

## Best Possible Solution

Land the additive resolved-tools hook after reusing the shared agent hook context helper, reconciling it with `llm_input.tools`, and preserving the schema snapshot docs/tests.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking finding is source-reproducible from the PR head: `after_tools_resolved` hand-builds `ctx.channelId` while existing agent hooks use `buildAgentHookContextChannelFields(params)`.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the emitted context should use `buildAgentHookContextChannelFields(params)` so the new hook matches established agent hook routing semantics.
