## Summary

Keep open: current main now has a partial overlapping `llm_input.tools` surface, but it is a raw-conversation hook rather than the proposed tool-only resolved-tools hook, and the PR still has a concrete channel-context blocker before it can merge.

## What This Changes

The PR adds an additive `after_tools_resolved` plugin hook for observing per-attempt resolved agent tool metadata and schemas, plus docs, tests, SDK checksum metadata, and a changelog entry.

## Best Possible Solution

Land a maintainer-approved resolved-tools plugin surface that reuses the standard agent hook context helper, clearly explains its relationship to `llm_input.tools`, and keeps docs, SDK baselines, and focused tests aligned with cloned schema snapshots.

## Reproduction Assessment

Yes for the PR-head defect: source inspection shows the new hook reports `channelId` from provider/channel text while current shared hook context resolves channel session keys to conversation ids. Reproduction is not applicable to the underlying request because it is a new plugin API surface.

## Solution Assessment

No, not as-is. The additive hook is a plausible solution, but it should reuse `buildAgentHookContextChannelFields(params)`, add focused regression coverage for that context behavior, and document how it differs from current `llm_input.tools`.
