## Summary

Keep this PR open: current main still lacks the requested resolved-tools plugin API, and the PR remains the active implementation path, but PR head still has a source-reproducible channel-context bug that should block merge.

## What This Changes

The PR adds an observation-only `after_tools_resolved` plugin hook, emits resolved tool metadata from embedded agent attempts, documents it, refreshes the SDK API baseline, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land the additive resolved-tools hook after reusing the shared agent hook context helper and preserving the schema snapshot docs/tests, or explicitly supersede it with a chosen pull-based resolved-tools API.

## Reproduction Assessment

Not applicable for the feature request itself. The blocking finding is source-reproducible from PR head: `after_tools_resolved` sets `ctx.channelId` from `messageChannel` or provider while existing agent hooks use `buildAgentHookContextChannelFields(params)`.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the new emission should reuse `buildAgentHookContextChannelFields(params)` so plugins receive the same channel and provider semantics across hook phases.
