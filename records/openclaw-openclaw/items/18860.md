The PR adds an `after_tools_resolved` plugin hook, embedded-runner emission, SDK hook types/API baseline updates, docs, and focused tests for resolved tool metadata.

## Best Possible Solution

Land the additive resolved-tools hook only after publishing cloned or immutable schema snapshots, documenting snapshot and retry semantics, adding the changelog entry, and refreshing the branch cleanly against current main.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the hook/API, and the PR defect is reproducible from the diff by registering an `after_tools_resolved` handler that mutates `event.tools[0].parameters` before the provider-facing schema object is consumed.

## Solution Assessment

No, not as proposed yet. The hook direction is narrow and maintainable, but an observation-only plugin API should expose a cloned or immutable schema snapshot rather than live provider-facing references.
