## Summary

Keep open: current main still lacks a resolved-tools discovery hook or pull API, and this PR is the active implementation path, but the patch still has a blocking context bug because the new hook hand-builds `channelId` instead of using the shared agent hook context helper.

## What This Changes

The branch adds an `after_tools_resolved` plugin hook, embedded-runner emission of post-policy resolved tool schema snapshots, docs, API baseline hash updates, a changelog entry, and focused tests.

## Best Possible Solution

Land the additive resolved-tools hook only after reusing the shared agent hook context helper, preserving cloned schema snapshots/docs/tests, and getting maintainer approval for the public plugin API contract.

## Reproduction Assessment

Not applicable for a feature PR. Source inspection confirms current main lacks `after_tools_resolved` or an equivalent pull API, and the PR body supplies live gateway logs for the proposed after-fix behavior.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the emitted context should reuse `buildAgentHookContextChannelFields(params)` so `channelId` and `messageProvider` match existing agent hook semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the shared agent hook channel context:** `src/agents/pi-embedded-runner/run/attempt.ts:1693`
  - body: The new hook manually sets `channelId` from `messageChannel ?? messageProvider`, while existing agent hooks use `buildAgentHookContextChannelFields(params)` to derive conversation IDs from session keys or target metadata and keep provider separate.
