## Summary

Keep open. The linked Feishu bug is valid and the Feishu dispatch boundary is a plausible owner, but this patch is not merge-ready because the new bypass treats queued-or-running work as active work, so a later same-chat message can overtake the first ordinary task before it starts; the supplied proof is still source-harness only.

## What This Changes

The PR exposes Feishu sequential queue key presence, bypasses ordinary same-key Feishu dispatch while that key is present, and adds Feishu queue/handler regression tests for ordinary followups plus control and /btw lanes.

## Best Possible Solution

Keep the Feishu dispatch-boundary direction, but expose a running-key signal instead of queue presence, add a same-tick ordering regression, and require redacted real Feishu/runtime proof for ordinary collect/steer plus control and /btw serialization.

## Reproduction Assessment

Yes, source-reproducible. Current main awaits the full same-key Feishu handler before the shared active-run queue sees later messages, and the PR-introduced ordering hole follows directly from queues.has being set before the queued task starts.

## Solution Assessment

No. The Feishu dispatch boundary is the right place to repair the linked bug, but using queued-key presence as active state is not the best fix because it can let a later ordinary message overtake the first.
