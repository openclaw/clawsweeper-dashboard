## Summary

Keep open: current main still derives Skills UI readiness from the gateway process PATH, and the current macOS PATH bootstrap does not add `~/.npm-global/bin` or npm prefix bins for these npm-installed skill CLIs. The later WSL/systemd comment usefully corroborates that the displayed status follows the gateway PATH context, and I did not find an exact merged fix or canonical duplicate.

## What This Changes

The issue asks OpenClaw to show `clawhub` and `mcporter` skills as ready when their required CLI binaries are installed and runnable from the user's terminal.

## Best Possible Solution

Teach the existing gateway PATH bootstrap or daemon service PATH diagnostics to recognize safe common npm-global locations for managed installs, while keeping user-writable paths lower priority and covering the behavior with focused tests.

## Reproduction Assessment

No live reproduction was run, but the issue is source-reproducible with high confidence: start the gateway with a minimal PATH and put `clawhub`/`mcporter` only in `~/.npm-global/bin`, and `hasBinary` will not find them. The WSL/systemd comment corroborates the same gateway-PATH-vs-shell-PATH failure mode.

## Solution Assessment

Yes. The narrow maintainable fix belongs in gateway PATH bootstrap or service PATH handling, not in the UI, because the UI is accurately rendering backend `skills.status` data.
