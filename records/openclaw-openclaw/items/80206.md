## Summary

Keep open: current main still has a plausible source-level path where the Control UI reports missing skill requirements because gateway-side binary detection only sees the gateway process PATH, while common npm-global macOS locations are not added by the gateway bootstrap. I did not find a merged fix or exact canonical duplicate for the `clawhub` and `mcporter` missing-requirements report.

## What This Changes

The issue asks OpenClaw to stop showing bundled `clawhub` and `mcporter` skills as missing required binaries when those npm-installed CLIs work from the user's terminal.

## Best Possible Solution

Teach the existing gateway PATH bootstrap or status diagnostics to recognize safe npm-global and OpenClaw-managed node installer bin locations after trusted system paths, with focused regression coverage.

## Reproduction Assessment

No, not with a high-confidence live reproduction. Source inspection shows a plausible current-main path when the gateway starts with a minimal macOS PATH while the installed `clawhub` and `mcporter` shims live in npm-global or npm-prefix directories only available to the user's terminal shell.

## Solution Assessment

Yes, a narrow backend fix in PATH bootstrap or precise diagnostics is the maintainable direction. The UI should not independently probe the shell because it correctly renders the gateway `skills.status` result.
