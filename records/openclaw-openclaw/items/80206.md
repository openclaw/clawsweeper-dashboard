## Summary

Keep open: current main still computes Skills readiness from the gateway process PATH, and the reported npm-global macOS setup has a plausible source-level path where `clawhub` and `mcporter` work in an interactive terminal but are absent from the gateway PATH used by the Control UI. I did not find a merged fix or exact canonical duplicate for these two npm-installed skill binaries.

## What This Changes

The issue reports that the Control UI Skills page marks `clawhub` and `mcporter` binary requirements as missing even though both commands work in the reporter's terminal.

## Best Possible Solution

Teach the gateway's existing PATH bootstrap or diagnostics to recognize common npm-global binary locations for macOS npm installs while preserving the existing trust ordering; update focused tests and only add docs if the runtime behavior remains intentionally constrained.

## Reproduction Assessment

No, not with high confidence from a live setup. Source inspection shows a plausible current-main reproduction when the gateway starts with a minimal macOS PATH and the npm-installed `clawhub`/`mcporter` shims live only in an npm-global directory that `ensureOpenClawCliOnPath` does not append.

## Solution Assessment

Yes, a narrow fix in PATH bootstrap or a precise diagnostic is the maintainable direction. The fix should not be in the UI rendering layer because the UI is correctly displaying the backend `skills.status` result.
