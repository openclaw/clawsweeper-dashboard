## Best Possible Solution

Keep the one-path bounded error reader if maintainers accept the narrow hardening, but require redacted real behavior proof for a production-like no-body Mattermost error path before merge.

## Reproduction Assessment

Yes for the narrow source condition: current main calls native `json()`/`text()` when `readMattermostError()` sees no readable body. I did not establish a real unbounded server-body reproduction for a true null-body response.

## Solution Assessment

Yes, this is a maintainable small fix shape because deleting the special branch makes all Mattermost error reads use the existing bounded helper without adding API or config surface. The missing piece is real behavior proof, not a code repair I can point to.
