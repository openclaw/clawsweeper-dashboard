## Summary

Keep open: current main contains a narrower inbound-audio message-tool fix, but it still implements tagged TTS as directive-driven and does not provide the requested runtime-owned, delivery-id-bound Telegram voice pipeline or broad delivery-error behavior.

## What This Changes

The issue requests making Telegram voice delivery runtime-owned after model output instead of dependent on model-generated TTS/media tags, with fresh per-turn audio binding and clearer delivery failures.

## Best Possible Solution

Define the intended runtime-owned Telegram voice policy, then bind generated audio to the current turn or delivery id and prove sendVoice/fallback behavior across tagged, always, inbound, and affected runtimes.

## Reproduction Assessment

No for a high-confidence live reproduction: I did not run the Telegram/TTS/MLX setup. Source inspection does confirm the central precondition remains: tagged mode still depends on model directives and generated filenames are not bound to inbound message or delivery ids.

## Solution Assessment

Unclear: a runtime-owned post-model voice transform is a plausible end state, but adding new `post_model` or `force_voice` semantics needs maintainer product direction and upgrade-safe config, docs, tests, and live Telegram proof.
