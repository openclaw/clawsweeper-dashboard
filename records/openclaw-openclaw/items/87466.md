## Summary

Keep open: current main has runtime-owned TTS pieces, but the reported Telegram voice-only failure is not clearly fixed or superseded. Source still makes `tagged` mode depend on model directives, and the observed stale-media/no-response behavior needs a focused Telegram/TTS/runtime investigation rather than cleanup closure.

## What This Changes

The issue reports unstable Telegram voice-only delivery across model runtimes, including stale local voice media reuse and delivery failures surfacing as a generic no-response fallback.

## Best Possible Solution

Keep the issue open and either repair the existing TTS/Telegram delivery path so generated media is turn-bound and delivery failures preserve the real reply, or explicitly design a new voice-only config mode if maintainers want tagged mode to stop relying on model directives.

## Reproduction Assessment

No high-confidence current-main reproduction path was established. The issue provides detailed observed logs, and source inspection confirms the tagged/directive dependency and lack of explicit message-id-bound TTS filename, but that is not enough to reproduce stale reuse end to end.

## Solution Assessment

Unclear as proposed: a new `voice.mode` may be the right product direction, but the narrower maintainable fix may be to keep existing TTS auto modes and repair media binding plus delivery-error reporting. Maintainers should decide whether tagged semantics change or a new explicit mode is added.
