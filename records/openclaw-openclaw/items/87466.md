## Summary

Keep open: current main and the latest release still expose directive-driven `tagged` TTS rather than a runtime-owned Telegram voice policy, and the active related PR only covers a narrower trusted-media suppression path.

## What This Changes

The issue asks OpenClaw to make Telegram voice delivery stable across model runtimes by moving voice generation to an OpenClaw-owned post-model TTS path with per-turn media binding and clearer delivery errors.

## Best Possible Solution

Define one OpenClaw-owned Telegram voice policy using the existing TTS/channel seams, bind generated media to the current turn or delivery id, preserve upgrade-safe defaults, and prove tagged, inbound, always, and delivery-error paths with live Telegram.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. Source inspection confirms current main still has directive-gated `tagged` TTS and related media-suppression preconditions, but stale voice-file reuse needs a real Telegram/local-TTS scenario to prove end to end.

## Solution Assessment

Unclear: the requested post-model voice direction is plausible, but it changes TTS/channel policy and likely config semantics. Maintainers should choose the upgrade-safe voice policy before a repair PR implements it.
