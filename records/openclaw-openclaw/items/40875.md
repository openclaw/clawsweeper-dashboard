## Summary

Keep open. The docs correction is useful because current main still gives no fallback guidance for `heartbeat.model`, but the PR should not merge as written: the current resolver and regression tests show a different-provider override skips unrelated configured fallbacks yet still retries the configured primary when no explicit fallback override is supplied, while the PR says the heartbeat errors out without trying other models. The changelog entry also lacks the required contributor attribution.

## What This Changes

The PR adds a changelog entry and two heartbeat documentation bullets describing when `heartbeat.model` uses or skips the standard model fallback chain.

## Best Possible Solution

Update the current `docs/gateway/heartbeat.md` `model` ParamField to align with `docs/concepts/model-failover.md`: describe normal no-override and same-provider fallback behavior, different-provider configured-fallback skipping, and the configured-primary final retry. Keep runtime code unchanged and add a single-line changelog entry with valid contributor attribution.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a docs-only PR, but the verification path is high confidence: `runHeartbeatOnce` forwards the override into the shared reply path, `resolveFallbackCandidates` defines the candidate chain, and `src/agents/model-fallback.test.ts` covers the configured-primary retry.

## Solution Assessment

Is this the best way to solve the issue? No. The intended docs correction is the right direction, but the current PR text is not the best fix because it omits the configured-primary retry and targets an older docs layout; the safer path is a narrow docs/changelog repair against current main.
