Full review comments:

- **[P2] Rebase cleanup changes onto the current cleanup service:** `src/commands/sessions-cleanup.ts:442`
  - body: The PR head reintroduces an older `sessionsCleanupCommand` implementation and no longer calls current main's gateway cleanup / `runSessionsCleanup` path. Merging it as-is would drop newer cleanup behavior and summary fields; move the remediation hooks onto the current cleanup-service path instead.
  - confidence: 0.88
- **[P2] Honor resetArchiveRetention when purging reset archives:** `src/infra/session-health-remediation-executor.ts:526-527`
  - body: Reset archive deletion uses `snapshot.maintenance.pruneAfterMs`, but current config has a separate `resetArchiveRetention` setting, including `false` to disable reset cleanup. Carry that resolved value into the snapshot/executor so `.reset` archives are not deleted earlier than configured.
  - confidence: 0.93
- **[P2] Base stale archive eligibility on archive timestamps:** `src/infra/session-health-file-discovery.ts:141`
  - body: The stale archive helpers compare retention against file mtime. Because orphan archival is a rename that preserves the old transcript mtime, `--execute-tier 1` can archive an old orphan and then delete that newly archived file in the same pass; use `parseSessionArchiveTimestamp` for `.deleted` and `.reset` archives instead.
  - confidence: 0.9
- **[P2] Avoid escalating orphan temp files to critical drift:** `src/infra/session-health-collector.ts:337-338`
  - body: The collector sets `reconciliationRecommended` for any orphaned temp file, and `deriveIndexHealth` turns that flag into a critical index-health state. A single crashed-write `.tmp` should follow the intended orphan-temp warning path, not make the whole surface critical.
  - confidence: 0.86
