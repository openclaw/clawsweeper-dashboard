## Summary

Keep this PR open. The session-health direction is useful, but the branch is not on current main, still bypasses existing session-file and archive-retention contracts in ways that can misclassify or delete live/recovery data, and it does not include after-fix real behavior proof.

## What This Changes

The PR adds a session-health collector, derived Mission Control surface, remediation planning and execution flags for `openclaw sessions cleanup`, related docs/tests, and pre-commit hook gate changes.

## Best Possible Solution

Align the feature with the existing `sessionFile`, archive timestamp, and reset-retention contracts, add focused regressions for those contracts, then provide redacted real CLI/UI proof before maintainer review.

I did not run them because this review is read-only, but the current-main contracts and PR source make the failures high-confidence.

## Solution Assessment

No. The feature may be valuable, but this branch is not the safest implementation until it reuses the existing session transcript resolution and archive-retention helpers instead of duplicating weaker filesystem rules.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Honor persisted sessionFile paths when finding orphans:** `src/infra/session-health-file-discovery.ts:116-117`
  - body: `discoverOrphanTranscripts` treats the filename stem as the session ID and only checks `indexedSessionIds`. Current session entries can persist a different `sessionFile`; those live transcripts will be classified as orphans and archived, making active sessions look missing.
  - confidence: 0.94
- **[P1] Base archive staleness on archive timestamps:** `src/infra/session-health-file-discovery.ts:141`
  - body: The stale archive check uses file `mtime`, but the executor archives orphan transcripts with `fs.rename`. Old orphan transcripts can therefore be renamed to `.deleted.<now>` and then immediately match the purge action in the same `--execute-tier 1` run, bypassing the archive-first retention window.
  - confidence: 0.91
- **[P1] Honor reset archive retention when purging resets:** `src/infra/session-health-remediation-executor.ts:526`
  - body: Reset archive deletion uses `snapshot.maintenance.pruneAfterMs` instead of the dedicated reset archive retention setting. Configurations with a longer `session.maintenance.resetArchiveRetention`, or `false` to disable reset cleanup, can still have `.reset` recovery archives removed on the generic prune window.
  - confidence: 0.93
- **[P2] Keep orphaned temp files out of critical reconciliation:** `src/infra/session-health-collector.ts:337-338`
  - body: The collector sets `reconciliationRecommended` for any orphaned `.tmp` file, and `deriveIndexHealth` turns that flag into a critical alert. A single crashed atomic-write temp file should stay on the warning/Tier 0 cleanup path, not escalate the whole session health surface to critical.
  - confidence: 0.88
