## Summary

Keep open. Current main still lacks the session-health visibility/remediation surface, and the PR branch still has blocking cleanup-contract bugs in paths that can rename or delete session transcript archives.

## What This Changes

The PR adds session-health collection and derivation, gateway and Control UI health visibility, cleanup remediation dry-run/execution paths, docs/tests, and pre-commit gate behavior changes.

## Best Possible Solution

Split the read-only collector/cache/health/UI visibility from the remediation executor, fix the archive identity and retention contracts before any deletion path lands, and move the pre-commit policy change to a separate infra PR if maintainers want it.

## Reproduction Assessment

Yes. The blockers are reproducible by static contract inspection: the PR compares filenames to sessionId while current main supports entry.sessionFile, uses mtime instead of archive filename timestamps, and ignores resetArchiveRetention for reset archives. No destructive cleanup command was run.

## Solution Assessment

No. The visibility direction is plausible, but the combined implementation is not the safest fix as written; a narrower read-only visibility slice plus a separately reviewed executor would be more maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve orphans from stored sessionFile paths:** `src/infra/session-health-file-discovery.ts:115-117`
  - body: Session entries can persist a `sessionFile` that is not `${sessionId}.jsonl`, but orphan discovery only compares disk basenames to `sessionId`. A live transcript stored under a custom, topic, migrated, or absolute sessionFile can be classified as orphaned and renamed by the archive action.
  - confidence: 0.9
- **[P1] Use archive timestamps for stale deleted files:** `src/infra/session-health-file-discovery.ts:141`
  - body: Stale `.deleted` discovery uses file mtime, but the executor archives orphans with `fs.rename`, which preserves the old transcript mtime. Running tier 1 can archive an old orphan and then immediately delete that fresh archive, bypassing the archive-first safety model.
  - confidence: 0.88
- **[P1] Honor resetArchiveRetention for reset archives:** `src/infra/session-health-remediation-executor.ts:526-527`
  - body: Reset archive purging uses the general `pruneAfterMs` window instead of the dedicated `resetArchiveRetention` policy. Configurations that retain reset archives longer, or disable reset cleanup, can still lose `.reset` recovery artifacts.
  - confidence: 0.9
- **[P2] Execute remediation in the planned agent scope:** `src/infra/session-health-remediation-executor.ts:218-221`
  - body: The plan is derived from all configured agents, but execution always mutates only the default agent store. An action caused by another agent can validate and then report as skipped/resolved while leaving the real target untouched.
  - confidence: 0.84
- **[P2] Do not make temp artifacts reconciliation-critical:** `src/infra/session-health-collector.ts:337-338`
  - body: The collector sets `reconciliationRecommended` when any orphaned temp file exists, and derivation treats that as critical index health. A single crashed atomic-write `.tmp` file should follow the warning cleanup path instead of escalating the whole index indicator to critical.
  - confidence: 0.82
- **[P2] Re-evaluate stale health when serving cached surfaces:** `src/gateway/server/health-state.ts:85-87`
  - body: The gateway attaches the cached derived surface as-is, so the `stale_data` transition only happens when the collector runs. If the collector stalls after a healthy derivation, operators can keep seeing a non-stale status indefinitely.
  - confidence: 0.78
