## Summary

Keep open. Current main does not implement session health lifecycle visibility, and the PR still has source-visible P1 correctness issues around session transcript detection, archive retention, and health severity before it is safe to merge.

## What This Changes

The PR adds session-health collection, derived Mission Control visibility, remediation planning/execution for `openclaw sessions cleanup`, and pre-commit gate behavior changes.

## Best Possible Solution

Address the P1 correctness issues, rebase onto current main, and have a maintainer decide whether this broad session-health/remediation feature should land as one PR or be split into safer slices.

## Reproduction Assessment

Yes. The blocking behavior is reproducible by source inspection on PR head: persisted `sessionFile` paths are ignored, archive staleness uses `mtimeMs`, reset purges use `pruneAfterMs`, and orphan temp files set the reconciliation-critical flag.

## Solution Assessment

No. The direction is useful, but the current patch is not the best safe implementation until the executor follows current session-file and archive-retention contracts and the health surface separates temp cleanup from index reconciliation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Match orphan detection to persisted sessionFile paths:** `src/infra/session-health-file-discovery.ts:115-117`
  - body: Current session entries can persist `sessionFile`, and existing cleanup resolves transcript paths through that field. This PR only extracts a session id from `${name}.jsonl`, so a live transcript with a custom persisted path can be classified as orphaned and renamed to `.deleted.*`.
  - confidence: 0.93
- **[P1] Keep temp cleanup out of reconciliation-critical state:** `src/infra/session-health-collector.ts:337-338`
  - body: The collector sets `reconciliationRecommended` whenever any orphaned `.tmp` file exists, and `deriveIndexHealth` treats that as critical. A single crashed-write temp file therefore escalates the whole surface to critical instead of using the warning cleanup path intended for temp artifacts.
  - confidence: 0.92
- **[P1] Base stale archive checks on archive timestamps:** `src/infra/session-health-file-discovery.ts:141`
  - body: Stale `.deleted` archives are filtered by file `mtime`, but the executor creates archives with `fs.rename`, which preserves the old transcript mtime. Running tier 1 can archive an old orphan transcript and then immediately purge the new archive in the same pass, defeating the archive-first safety model.
  - confidence: 0.94
- **[P1] Respect reset archive retention policy:** `src/infra/session-health-remediation-executor.ts:526-527`
  - body: Reset archive purging uses `snapshot.maintenance.pruneAfterMs` rather than the resolved reset archive retention. Configurations with a longer `resetArchiveRetention` or `false` can still lose `.reset` recovery artifacts on the shorter prune window.
  - confidence: 0.95
