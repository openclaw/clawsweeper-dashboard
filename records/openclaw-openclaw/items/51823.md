## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use sessionFile when detecting orphan transcripts:** `src/infra/session-health-file-discovery.ts:102-104`
  - body: `discoverOrphanTranscripts` treats an active transcript as referenced only when its basename equals an indexed `sessionId`. Current stores can persist a different `entry.sessionFile`, and existing cleanup resolves that field before falling back to `${sessionId}.jsonl`; this would archive a live custom-named transcript as orphaned and make the session appear missing.
  - confidence: 0.93
- **[P1] Base archive retention on archive timestamps:** `src/infra/session-health-file-discovery.ts:126`
  - body: Stale `.deleted` and `.reset` discovery compares retention against file `mtimeMs`. Renaming an old orphan transcript to `.deleted.<timestamp>` preserves the old file mtime, so a tier execution can archive it and then immediately purge it instead of honoring the new archive timestamp.
  - confidence: 0.91
- **[P1] Honor resetArchiveRetention for reset archives:** `src/infra/session-health-remediation-executor.ts:472-473`
  - body: The reset archive executor uses `snapshot.maintenance.pruneAfterMs` as the retention window. Current config has a separate `session.maintenance.resetArchiveRetention` value, including `false` to disable reset cleanup, so this can delete reset recovery archives earlier than configured.
  - confidence: 0.94
- **[P2] Keep temp-file drift out of critical reconciliation:** `src/infra/session-health-collector.ts:304-305`
  - body: The collector marks `reconciliationRecommended` true whenever any orphaned temp file exists, and `deriveIndexHealth` promotes that flag to critical. A single crashed `.tmp` artifact should follow the warning temp-file path instead of raising a critical index-reconciliation alert.
  - confidence: 0.88
