## Summary

Keep this PR open: current main does not contain the session-health work, and the proposed deletion/remediation path still conflicts with existing session-file and archive-retention contracts.

## What This Changes

The PR adds session health collection and derivation, gateway and Mission Control visibility, remediation planning/execution flags for `openclaw sessions cleanup`, docs, and pre-commit hook gate changes.

## Best Possible Solution

Keep the PR open and rework remediation discovery/execution to reuse the existing session path and archive-retention helpers before any safety or product merge decision.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows exact code paths that violate current session-file and archive-retention contracts. This is not a current-main bug reproduction because the session-health implementation is new in the PR.

## Solution Assessment

No. The proposed approach duplicates cleanup semantics and should instead route through `resolveSessionFilePath`, archive timestamp parsing/cleanup helpers, reset-specific retention, and read-time stale-surface validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Respect sessionFile when finding orphan transcripts:** `src/infra/session-health-file-discovery.ts:116-117`
  - body: `discoverOrphanTranscripts` turns each active filename into a session id and only checks `indexedSessionIds`. Current stores can persist a valid `entry.sessionFile` that does not match `${sessionId}.jsonl`, and existing cleanup resolves that field before falling back, so this can archive a live transcript as orphaned.
  - confidence: 0.93
- **[P1] Use archive suffix timestamps for stale archive discovery:** `src/infra/session-health-file-discovery.ts:141`
  - body: `discoverStaleDeletedTranscripts` and the reset variant use `mtimeMs` to enforce retention. Orphan archiving is a rename and preserves old mtimes, while current archive cleanup parses `.deleted.<timestamp>`/`.reset.<timestamp>`, so a newly archived old transcript can be purged immediately in the same tier.
  - confidence: 0.91
- **[P1] Apply resetArchiveRetention to reset purges:** `src/infra/session-health-remediation-executor.ts:526-527`
  - body: Reset purge execution uses `snapshot.maintenance.pruneAfterMs`. Current maintenance has dedicated `resetArchiveRetentionMs`, including disabled cleanup, so `--execute` can delete reset recovery archives earlier than configured.
  - confidence: 0.94
- **[P2] Keep orphaned temp files out of reconciliation critical:** `src/infra/session-health-collector.ts:337-338`
  - body: The collector sets `reconciliationRecommended` when `byDiskState.orphanedTemp > 0`, and `deriveIndexHealth` maps that flag to critical. A single crashed temp file should stay on the warning/remediation path, otherwise Mission Control reports critical index drift for definitional garbage.
  - confidence: 0.88
- **[P2] Recompute stale health state before serving cached surfaces:** `src/gateway/server/health-state.ts:85-87`
  - body: The stale-data check runs only while deriving the cached surface, but the health RPC later returns `readCachedDerivedSurface()` without re-deriving. If the collector stops after writing a healthy surface, operators keep seeing the old healthy level instead of `stale_data`.
  - confidence: 0.86
