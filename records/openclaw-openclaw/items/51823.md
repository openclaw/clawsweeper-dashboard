## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor sessionFile when detecting orphan transcripts:** `src/infra/session-health-file-discovery.ts:115-117`
  - body: `discoverOrphanTranscripts` treats the indexed set as session IDs and compares it to `file.name.replace(/\.jsonl$/, "")`. Current sessions can point at a different `entry.sessionFile`, so a live transcript with a custom/generated path can be marked orphaned and renamed to `.deleted.*`, making the session look missing until repaired. Build the referenced path set with the same resolver used by current session maintenance instead of assuming `${sessionId}.jsonl`.
  - confidence: 0.9
- **[P1] Use archive timestamps for stale archive checks:** `src/infra/session-health-file-discovery.ts:141`
  - body: The stale `.deleted` and `.reset` scans compare retention against filesystem `mtimeMs`. The new orphan-archive executor renames old transcripts into fresh `.deleted.<stamp>` archives, and `fs.rename` can preserve the old mtime, so `--execute-tier 1` can archive an orphan and then immediately purge it in the same pass. Match the existing archive cleanup contract by parsing the archive timestamp suffix instead of using mtime.
  - confidence: 0.91
- **[P1] Respect resetArchiveRetention for reset purges:** `src/infra/session-health-remediation-executor.ts:526-527`
  - body: Reset archive execution still uses `snapshot.maintenance.pruneAfterMs`, but OpenClaw has a separate `session.maintenance.resetArchiveRetention` setting, including `false` to disable reset cleanup. With this path, a user who keeps reset archives longer than deleted archives can have `.reset` recovery data removed on the shorter stale-session window.
  - confidence: 0.88
- **[P2] Avoid critical health for ordinary temp-file leftovers:** `src/infra/session-health-collector.ts:337-338`
  - body: The collector sets `reconciliationRecommended` when any orphaned temp file exists, and `deriveIndexHealth` upgrades that flag to a critical index-health state. A single crashed atomic-write `.tmp` file is already represented by the stale-artifacts indicator and Tier 0 cleanup, so this produces false critical Mission Control alerts for low-risk garbage.
  - confidence: 0.82
