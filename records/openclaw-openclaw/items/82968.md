## Summary

Keep open: current `main` documents and prompts `session_status` as the agent-facing current-time source, but the status builder only passes uptime into the status card and does not pass the available wall-clock `timeLine`, so the reporter's `date` workaround remains relevant.

## What This Changes

The issue requests a reliable agent-facing wall-clock time source and policy so uptime is not used for calendar, scheduling, heartbeat, memory, and deadline reasoning.

## Best Possible Solution

Wire the existing shared wall-clock formatter into `session_status` so it returns local current time plus reference UTC, keep uptime as a separate line, and leave broader scheduling or memory timestamp primitives for separate scoped follow-ups.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: docs and the system prompt direct agents to `session_status` for current time, while the `session_status` path only passes uptime and omits the status renderer's `timeLine`. A live transcript would still be useful proof for the fix PR.

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow repository bug: repair the existing documented `session_status` clock path rather than adding a second native time tool. Broader memory or scheduling contracts should be split if maintainers want them after the clock source is reliable.
