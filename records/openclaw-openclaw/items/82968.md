## Summary

Keep open: current main has shipped timestamp injection and prompt guidance that reduce the original problem, but the documented agent-facing clock path still looks incomplete because `session_status` is advertised as the current-time source while the status builder only wires uptime into the status card.

## What This Changes

Requests an explicit, reliable wall-clock time source for agents so current date/time, timezone, deadlines, scheduling, heartbeat, and chronology logic do not rely on uptime or shell workarounds.

## Best Possible Solution

Make the existing `session_status` clock contract real and keep the shipped timestamp-injection behavior; consider a dedicated `now` tool only as a separate API decision if maintainers want a smaller surface than status.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection shows the documented clock path is not fully wired: the prompt directs agents to `session_status`, but `buildStatusText` supplies uptime and does not supply the `timeLine` that the status renderer can display.

## Solution Assessment

No, adding a brand-new `Date` or `now` tool first is not the narrowest fix. The maintainable path is to repair the already documented `session_status` wall-clock output, then decide separately whether a smaller dedicated clock tool is worth adding.
