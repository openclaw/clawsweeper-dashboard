## Summary

Keep open. Current main and v2026.5.18 still do not expose the documented agent-facing current timestamp through `session_status`, and the broader wall-clock authority request is only partially covered by an open, not-merge-ready PR plus related follow-up issues.

## What This Changes

Request an explicit agent-facing wall-clock time authority so scheduling, heartbeat, memory chronology, deadlines, logs, and relative-date reasoning do not rely on uptime or guesses.

## Best Possible Solution

Land or replace the narrow `session_status` timestamp fix through the shared status renderer, then route heartbeat ISO, freshness metadata, and any dedicated Date/now tool decision to the existing focused follow-ups.

## Reproduction Assessment

Yes, source-level: current main and v2026.5.18 docs direct agents to `session_status` for current time, but `buildStatusText()` does not forward a `timeLine` into the shared status renderer. I did not live-run an agent prompt in this read-only review.

## Solution Assessment

Unclear as a complete solution: extending `session_status` through the shared renderer is the narrowest current path, but a dedicated Date/now tool, heartbeat trigger semantics, memory/log timestamp contracts, and timezone scheduling primitives still need maintainer scope decisions.
