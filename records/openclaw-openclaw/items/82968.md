## Summary

Keep open. Current main has useful wall-clock coverage through message/event timestamp injection and documented timezone handling, but it still does not expose a clear native agent-facing clock authority; the reporter's `exec date` workaround and request for a `Date`/`now` tool remain product/API work rather than a closed bug.

## What This Changes

The issue requests an explicit agent-facing wall-clock time authority and policy so agents do not rely on uptime for temporal reasoning.

## Best Possible Solution

Define one explicit agent-facing wall-clock contract, likely by extending `session_status` with a stable timestamp payload or adding a small `clock.now` tool, then align docs, prompt policy, timezone behavior, and focused tests around that contract.

## Reproduction Assessment

Yes, at source level: current main has timestamp injection and timezone docs, but `session_status` still returns rendered status text without an explicit structured wall-clock value, and there is no native `Date`/`now` tool. I did not live-run a prompt transcript because the remaining gap is an API/product contract rather than a deterministic crash or failing command.

## Solution Assessment

Unclear. Current timestamp injection is a useful mitigation, but the best final solution needs maintainer choice between extending `session_status`, adding a dedicated clock tool, or formalizing runtime context as the supported clock surface.
