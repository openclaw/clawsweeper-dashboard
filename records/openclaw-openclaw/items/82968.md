## Summary

Keep open. Current main has important time-context work already in place, including timestamp injection for agent/chat/heartbeat paths and prompt guidance to use `session_status`, but the documented `session_status` clock contract appears incomplete because the status card path only passes uptime and never a wall-clock `timeLine`.

## What This Changes

The issue requests a reliable agent-facing wall-clock time authority so agents do not rely on uptime or shell `date` workarounds.

## Best Possible Solution

Thread the existing wall-clock helper into `session_status` so the documented status card includes current local and UTC time, with tests; consider a separate `clock.now` tool only as a follow-up product decision.

## Reproduction Assessment

Yes, from source inspection: current docs and prompt guidance define `session_status` as the current-time source, but the status-card builder receives uptime only and no wall-clock `timeLine`. I did not execute a live agent turn in this read-only review.

## Solution Assessment

Yes for the narrow fix: use the existing `session_status` and `resolveCronStyleNow` direction instead of adding a new tool first. A separate Date/now tool can remain a maintainer product follow-up if the existing contract is judged insufficient.
