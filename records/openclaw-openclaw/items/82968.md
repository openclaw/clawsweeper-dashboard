## Summary

This should stay open: current main has partial time awareness through timestamped message envelopes/injection and a system-prompt hint, but it still does not clearly provide the requested dedicated agent-facing wall-clock authority or policy across heartbeat, memory, logs, and scheduling surfaces.

## What This Changes

The issue requests a reliable agent-facing wall-clock time source instead of relying on uptime or `exec date`.

## Best Possible Solution

Define one maintained wall-clock authority, either by restoring and documenting `session_status` as the clock source or by adding a small native `now` tool, then thread that policy through heartbeat, cron, memory, and logging without replacing uptime/monotonic timing for durations.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was run, but source inspection shows current main has timestamp context and uptime while the dedicated agent-facing clock contract remains incomplete or at least inconsistent with the docs.

## Solution Assessment

Is this the best way to solve the issue? Unclear; a native `now` tool is plausible, but maintainers should first decide whether `session_status` is the intended clock authority and whether heartbeat/memory/log timestamping belongs in the same change.
