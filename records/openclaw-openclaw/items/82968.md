## Summary

Keep open: current main has timezone guidance and some timestamped automation paths, but the documented agent-facing `session_status` clock path is still incomplete and the broader native `Date`/`now` authority needs maintainer API/product scope.

## What This Changes

The issue requests a reliable agent-facing wall-clock time source for time-sensitive reasoning, scheduling, heartbeat, memory chronology, and deadline behavior.

## Best Possible Solution

Land or replace the focused `session_status` timestamp fix, then have maintainers decide whether a separate native clock tool/runtime contract is needed for the broader heartbeat, memory, and scheduling policy.

## Reproduction Assessment

Yes for the documented `session_status` gap: source inspection shows docs and prompt guidance point agents to `session_status`, while current main does not thread a wall-clock time line into that status output. The broader native clock-tool and heartbeat/memory acceptance criteria still need product scope and live validation.

## Solution Assessment

Unclear as a whole: adding wall-clock output to `session_status` is the narrowest fix for the documented contract, but a standalone native `Date`/`now` tool and broader timestamp policy require maintainer API direction.
