## Summary

Keep open: current main has several timestamp surfaces, but it still does not provide a clearly complete agent-facing wall-clock authority. The prompt/docs point agents to `session_status` for current time, while the status path currently passes uptime but no populated wall-clock line, so this needs maintainer product/API scope rather than cleanup closure.

## What This Changes

The issue requests an explicit reliable agent-facing wall-clock time source and policy instead of relying on uptime or ad hoc `date`/exec workarounds.

## Best Possible Solution

Define one canonical late-bound wall-clock contract, likely by making `session_status` a real time authority or adding a dedicated `clock.now` tool, then update prompt/docs/tests while keeping uptime reserved for elapsed-duration and health reporting.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows the documented `session_status` time source is not fully wired, but the broader scheduling/memory failures were not live-reproduced against current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the direction is valid, but maintainers should decide whether the canonical surface is `session_status`, a dedicated clock tool, or runtime context before implementation.
