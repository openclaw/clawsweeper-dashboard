## Summary

This issue should stay open because current OpenClaw has wall-clock timestamps in some runtime telemetry paths, but there is no clear agent-facing time authority or prompt/tool policy that satisfies the requested behavior.

## What This Changes

Request an explicit reliable wall-clock time source and policy for agent time-sensitive reasoning, scheduling, heartbeat, memory chronology, and logs.

## Best Possible Solution

Define a small, explicit agent-facing clock contract in the runtime or SDK, then thread it into prompts/tools and timestamped memory or heartbeat surfaces without replacing monotonic timing used for durations.

## Solution Assessment
