Keep open: this is a focused candidate fix for the linked unsafe-local data-loss bug and I did not find a blocking code defect, but it still needs contributor real-behavior proof and maintainer acceptance of the directory-root retention tradeoff before merge.

## What This Changes

The PR changes memory-wiki unsafe-local sync to treat missing or unreadable configured directory roots as prune-protected scopes, adds protected keys before pruning, and adds a regression test for notes surviving an outage.

## Best Possible Solution

Merge a scoped unsafe-local fix that preserves entries for unavailable configured roots, keeps explicit file deletion pruning, and accepts or documents the root-deletion retention tradeoff with real after-fix proof.

## Reproduction Assessment

Yes. The linked issue includes a current-main production-function repro, and source inspection confirms the missing configured directory currently collapses to an empty active set followed by destructive prune; I did not rerun tests in this read-only checkout.

## Solution Assessment

Yes, with one maintainer-visible tradeoff. Protecting prior entries under unavailable configured roots is the maintainable data-preserving fix shape, while explicit configured file deletion still exercises the existing prune path.
