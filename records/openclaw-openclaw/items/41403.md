## Summary

Keep open. Current `main` still does not implement the draft's core docs clarification: the user-facing hook docs continue to blur internal HOOK-style hooks and typed plugin lifecycle hooks, even though the runtime API and registry keep those surfaces distinct.

## Evidence

- **Hooks doc still mixes internal hooks and plugin hooks:** `docs/automation/hooks.md` still introduces only internal hooks vs webhooks, then adds a brief note that plugins can also bundle hooks, without the proposed decision guidance for when to use HOOK-style internal hooks versus typed plugin hooks.
  - file: [docs/automation/hooks.md:11](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/docs/automation/hooks.md#L11)
  - command: `nl -ba docs/automation/hooks.md | sed -n '11,16p;206,210p'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Plugin doc still lumps `registerHook` and `on(...)` together:** `docs/tools/plugin.md` currently lists `registerHook / on(...)` as one shared 'Lifecycle hooks' registration method and only documents typed-hook guard semantics afterward. It does not give the draft's explicit 'choose the right hook API' guidance or mark `api.on(...)` as the preferred middleware/policy surface.
  - file: [docs/tools/plugin.md:327](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/docs/tools/plugin.md#L327)
  - command: `nl -ba docs/tools/plugin.md | sed -n '327,358p'`
  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Runtime code keeps the two hook surfaces separate:** The registry wires `registerHook(...)` through `registerInternalHook(...)` for string event names, while the public API also exposes a separate typed `on(...)` hook registration surface. That matches the PR's premise that these are different contracts.
  - file: [src/plugins/registry.ts:360](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/plugins/registry.ts#L360)

- **Some deeper SDK docs exist, but they do not close this PR:** `docs/plugins/sdk-overview.md` does document `api.on(...)` as a typed lifecycle hook and gives guard semantics, but that clarification has not been carried into the two higher-level pages this PR updates. `docs/plugins/building-plugins.md` still calls `api.registerHook(...)` the generic event-hook registration surface.
  - file: [docs/plugins/sdk-overview.md:198](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/docs/plugins/sdk-overview.md#L198)
