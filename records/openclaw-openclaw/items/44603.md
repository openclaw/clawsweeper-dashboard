## Summary

Keep open: current main still exposes `pty` only as a per-call exec parameter, while `tools.exec` config, default propagation, and docs do not support a global PTY default. The related implementation PR was closed unmerged, so there is no merged replacement to close against.

## What This Changes

The issue requests a global `tools.exec.pty` config default so long-running exec commands can stream intermediate output without relying on each per-call tool invocation to pass `pty: true`.

## Best Possible Solution

Ship a small schema/docs/runtime change that adds `tools.exec.pty` as an optional default, preserves current per-call precedence, and threads the value through all exec host paths without changing approval or sandbox policy.

## Reproduction Assessment

Yes. Source inspection on current main shows `tools.exec.pty` is absent from the strict config schema and runtime defaults, while exec still derives PTY mode only from the per-call `params.pty` value.

## Solution Assessment

Yes. The narrow maintainable fix is to add the typed config/default field and use `(params.pty ?? defaults?.pty) === true` wherever exec chooses PTY mode, while updating focused tests and docs.
