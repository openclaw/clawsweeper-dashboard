## Summary

Current main still has only per-call PTY selection for exec; the requested global `tools.exec.pty` default is not in the config schema, runtime defaults, or docs. The related implementation PR was closed unmerged to manage active PR count, so it remains useful context but not a completed fix.

## What This Changes

The issue requests a global `tools.exec.pty` configuration default so long-running exec commands can stream intermediate output without relying on each tool call to pass `pty: true`.

## Best Possible Solution

If maintainers want this default, add `tools.exec.pty` to the config schema, config types, schema labels/help, exec defaults, and docs, then use it only when the tool call omits `pty` while preserving sandbox behavior and gateway approval handling.

## Reproduction Assessment

Yes for the source-level feature gap: current main has a per-call `pty` parameter and PTY runtime support, but no `tools.exec.pty` config key or defaults plumbing. I did not run an interactive long-running command because the missing config surface is directly visible in source and docs.

## Solution Assessment

Unclear until maintainers approve the global default. If approved, the prior PR’s direction is close, but it must also cover the gateway approval path and add focused docs/tests for the config contract.
