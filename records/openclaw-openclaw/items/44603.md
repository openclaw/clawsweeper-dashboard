## Summary

Keep open: current `main` and `v2026.6.8` still expose `pty` only as a per-call exec argument, not as a `tools.exec` config default. The remaining work is concrete, but it adds a command-execution default surface and should go through maintainer product/security review before implementation.

## What This Changes

The issue requests a global `tools.exec.pty` configuration default so exec calls can use PTY mode without the model passing `pty: true` on every call.

## Best Possible Solution

If maintainers accept the config surface, add a documented opt-in `tools.exec.pty` and matching per-agent default that preserves per-call override, keeps sandbox behavior explicit, and includes focused config/runtime/docs tests.

## Reproduction Assessment

Yes. Source inspection shows `pty` is accepted as a per-call exec argument, while config schema/types/default plumbing and docs do not provide `tools.exec.pty`; I did not run a live streaming-output repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the product/security direction. The requested fallback is plausible, but the maintainable solution must add schema, types, docs, and tests while threading the default through gateway approval and direct execution paths.
