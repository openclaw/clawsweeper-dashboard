## Summary

Keep this open. Current main still supports `pty` only as a per-call exec parameter, while the global and per-agent `tools.exec` config schema, types, docs, labels, and config propagation do not expose a `pty` default. The related implementation PR was closed unmerged, so the requested behavior is still valid and narrow enough for a focused repair PR.

## What This Changes

The issue requests a global `tools.exec.pty` configuration default so long-running exec commands can stream intermediate output without relying on each per-call tool invocation to set `pty: true`.

## Best Possible Solution

Add an optional `pty` exec default through the config schema, config types, labels/help/docs, generated config schema baseline, `resolveExecConfig`, `ExecToolDefaults`, and `createExecTool`; compute PTY from `params.pty ?? defaults?.pty` while preserving the existing sandbox guard and explicit per-call override behavior.

## Reproduction Assessment

Yes. A focused source-level repro is clear: configure `tools.exec.pty: true`, call exec without a per-call `pty`, and observe that current main has no schema/type propagation and still computes `usePty` from only `params.pty`.

## Solution Assessment

Yes. The narrow maintainable fix is a real config default wired through the existing exec-defaults path, not a prompt/skill instruction workaround, with tests proving global default, agent override, per-call override, and sandbox guard behavior.
