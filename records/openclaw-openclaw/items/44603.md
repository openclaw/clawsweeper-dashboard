## Summary

Keep open: current main and the latest release still support `pty` only as a per-call exec argument, while the requested `tools.exec.pty` config default remains absent. The request is source-reproducible, but it needs maintainer product/security review because it adds a global command-execution default and the linked implementation PR was closed unmerged.

## What This Changes

The issue requests a global `tools.exec.pty` configuration default so long-running exec commands can stream intermediate output without the model passing `pty: true` on each call.

## Best Possible Solution

Keep this open for maintainer product/security review, then implement `tools.exec.pty` only if maintainers accept it as an explicit operator-controlled exec knob with schema, docs, and focused direct/gateway approval tests.

## Reproduction Assessment

Yes. Source inspection shows `pty` is currently only read from the per-call tool argument, while `timeoutSec` has a config/default path and the config schema has no `tools.exec.pty` key.

## Solution Assessment

Unclear until maintainers make the product/security decision. The narrow implementation is plausible, but the best fix must cover schema/types/docs plus both direct and gateway approval execution paths.
