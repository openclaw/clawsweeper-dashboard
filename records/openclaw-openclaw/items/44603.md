## Summary

Keep open: current main and the latest release still support `pty` only as a per-call exec parameter, not as a `tools.exec` config default. The same-repo implementation attempts are closed unmerged, and the remaining work needs maintainer product/security review because it adds a global command-execution default surface.

## What This Changes

The issue requests an opt-in `tools.exec.pty` config default so exec calls can use PTY mode without relying on each model tool call to pass `pty: true`.

## Best Possible Solution

If maintainers accept the config surface, add a documented opt-in `tools.exec.pty` and per-agent equivalent that preserves per-call override, keeps sandbox behavior explicit, and covers both gateway approval and direct execution paths with focused tests.

## Reproduction Assessment

Yes. Current source and docs show `pty` exists only as a per-call exec parameter, while `tools.exec` config/default plumbing has no PTY field; I did not run a live streaming-output repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product/security direction. The proposed one-line fallback is only part of the fix; a maintainable implementation must update schema, types, docs, tests, and both gateway approval and direct execution paths.
