## Summary

Keep open: current `main` and the latest release still support `pty` only as a per-call exec parameter, not as a `tools.exec` config default. The remaining work is real, but it is a new security-sensitive config/default surface, and the related implementation attempts are closed unmerged rather than canonical fixes.

## What This Changes

The issue requests a global `tools.exec.pty` configuration default so long-running exec commands can stream intermediate output without relying on each tool call to pass `pty: true`.

## Best Possible Solution

Maintainers should decide whether core should expose `tools.exec.pty` or `tools.exec.defaultPty`; if yes, implement it across schema/types, config resolution, gateway approval and local exec paths, docs, and focused override tests.

## Reproduction Assessment

Yes, at source level. Current `main` exposes per-call `pty` but has no `tools.exec.pty` config schema/type/default forwarding, and the runtime still derives PTY only from `params.pty`.

## Solution Assessment

Unclear as proposed. The issue identifies the right central gap, but the suggested one-line runtime fallback is not the whole fix because config schema, config resolution, docs, tests, and the gateway approval path also need a deliberate treatment.
