## Summary

Keep open. Current main and v2026.5.18 still expose `pty` only as a per-call exec parameter; `tools.exec.pty` is absent from the strict config schema, config types, default resolution, tests, and docs, while the related implementation PRs are closed unmerged. This remains a feature/config request needing maintainer product and security review.

## What This Changes

The issue requests a global `tools.exec.pty` config default so exec commands can use PTY streaming without the model passing `pty: true` on each call.

## Best Possible Solution

After maintainer product/security approval, implement the full `tools.exec.pty` config contract across schema, types, help/docs, and direct/gateway/node default wiring while preserving per-call override and sandbox no-PTY behavior.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that per-call PTY exists and no global `tools.exec.pty` config/default path exists on current main or v2026.5.18.

## Solution Assessment

Unclear until maintainers approve the product/security policy. The maintainable path is the full config contract plus direct and approval-backed runtime wiring, not only a single fallback expression.
