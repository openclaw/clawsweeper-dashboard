## Summary

Current main does not implement a global PTY default, and the prior exact PR closed unmerged for contributor workflow reasons rather than because the request was resolved. The issue should remain open for an explicit maintainer product/security decision because current long-running progress uses background/process notifications, while PTY changes terminal semantics beyond output flushing.

## What This Changes

Allow `tools.exec.pty` to set the default PTY mode for unsandboxed exec calls while retaining per-call override behavior.

## Best Possible Solution

Keep pipe-based execution as the safe default, make long-running progress reliably visible through the existing background/process path, and add a documented global PTY opt-in only if maintainers intentionally accept its broader terminal semantics.

## Reproduction Assessment

Yes for the narrow request: current source definitively lacks a global PTY default and requires per-call `pty: true`. No high-confidence reproduction establishes that the existing background/process progress path cannot solve the reported user-visible wait.

## Solution Assessment

No definitive best fix is established: the fallback is mechanically feasible, but using PTY as a progress control couples progress visibility to interactive-terminal behavior and requires explicit product/security approval.
