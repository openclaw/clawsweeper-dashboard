## Summary

Keep open. Current main and the latest release still expose `pty` only as a per-call exec parameter, while a global `tools.exec.pty` default would add a new command-execution config surface that needs maintainer product and security review before implementation.

## What This Changes

Add a global `tools.exec.pty` configuration option so exec calls can default to PTY mode without relying on the model to pass `pty: true` each time.

## Best Possible Solution

If maintainers approve the config surface, implement `tools.exec.pty` end to end across config types/schema, default resolution, gateway and local exec runtime paths, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that per-call `pty` exists, but no global `tools.exec.pty` config/default path exists on current main.

## Solution Assessment

Unclear until maintainer product review. The requested direction is plausible, but the maintainable fix must cover the full config contract plus gateway and direct exec paths rather than only adding one fallback expression.
