## Summary

Keep open. Current main and the latest release still expose PTY only as a per-call exec parameter; the requested `tools.exec.pty` default is absent from the strict config schema, config types, default resolution, runtime fallback, and docs. The linked implementation PR is closed unmerged and had a review note about a missed gateway-approval path, so this remains a product/security decision rather than cleanup-closeable work.

## What This Changes

Request adds a global `tools.exec.pty` default so long-running exec commands can use PTY streaming when the model omits the per-call `pty` argument.

## Best Possible Solution

If maintainers approve the policy, add an optional PTY default across schema, types, config resolution, direct/gateway runtime wiring, docs, and focused tests while preserving per-call override and sandbox no-PTY behavior.

## Reproduction Assessment

Not applicable as a feature/config request. Source inspection gives high confidence that per-call PTY exists but no global `tools.exec.pty` config/default path exists on current main or v2026.5.19.

## Solution Assessment

Unclear until maintainer product/security review. If accepted, the maintainable solution is the full config contract plus direct and approval-backed runtime wiring, not only a single fallback expression.
