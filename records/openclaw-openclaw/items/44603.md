## Summary

Keep open. Current main still has per-call `pty` support only; the public `tools.exec` config schema/types/default resolver do not expose `tools.exec.pty`, and the linked implementation PRs were closed unmerged. This is a plausible narrow feature, but it adds a new exec config default and needs maintainer product approval before a repair PR.

## What This Changes

Allow `tools.exec.pty: true` in OpenClaw config to make PTY mode the default for exec calls that omit a per-call `pty` value.

## Best Possible Solution

If maintainers accept the feature, add a documented `tools.exec.pty` boolean through config schema/types/help/docs and pass it into both gateway approval and direct exec paths while preserving per-call override and sandbox-disabled behavior.

## Reproduction Assessment

Yes for the configuration gap: current main exposes `pty` on the exec tool-call schema but omits `tools.exec.pty` from config schema/types/default resolution and computes PTY from `params.pty` only. I did not run a live long-running command because this was a read-only review.

## Solution Assessment

Unclear until maintainer approval because the requested path adds a new public exec config option. If approved, the prior PR direction is close, but the implementation must cover both the gateway approval path and the normal direct `usePty` path with focused tests.
