## Summary

Keep open: current main still exposes `pty` only as a per-call exec parameter, while the strict `tools.exec` config schema, defaults type, propagation path, and docs do not support a global PTY default. The two related implementation PRs were closed unmerged, so there is no shipped or current-main replacement to close against.

## What This Changes

Add a global `tools.exec.pty` configuration default so exec commands can use PTY mode without relying on every tool call to pass `pty: true`.

## Best Possible Solution

Ship a small schema/types/runtime/docs change that adds optional `tools.exec.pty`, keeps per-call `pty` precedence, and preserves the existing sandbox guard that disables PTY when sandbox execution is active.

## Reproduction Assessment

Yes. Source inspection on current main shows `tools.exec.pty` is absent from the strict config schema and runtime defaults, while exec still derives PTY mode only from the per-call `params.pty` value.

## Solution Assessment

Yes. The narrowest maintainable fix is to add the typed config/default field and use `(params.pty ?? defaults?.pty) === true` wherever exec chooses PTY mode, including gateway approval, while updating focused tests and docs.
