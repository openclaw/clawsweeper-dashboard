## Summary

Keep open: current main and the latest release still expose `pty` only as a per-call exec parameter, while the config schema, exec defaults, and default-resolution path do not support `tools.exec.pty`. A prior same-repo fix PR was closed unmerged, so the requested config surface still needs maintainer product/security review.

## What This Changes

The issue requests a global `tools.exec.pty` config option so exec commands can default to PTY mode for real-time output without relying on each model tool call to pass `pty: true`.

## Best Possible Solution

If maintainers approve the product/security direction, add an opt-in `tools.exec.pty` default across schema, types, config help/docs, `resolveExecConfig`, direct exec, and gateway approval execution, with tests for per-call override precedence and sandbox behavior.

## Reproduction Assessment

Yes, source-level: current main accepts only per-call `pty`, omits `tools.exec.pty` from config/default plumbing, and computes PTY from `params.pty` only. I did not run a live long-running command in this read-only review.

## Solution Assessment

Unclear until maintainers decide the product/security direction. The requested opt-in config knob is plausible, but the best implementation must cover all default surfaces and preserve explicit per-call and sandbox behavior.
