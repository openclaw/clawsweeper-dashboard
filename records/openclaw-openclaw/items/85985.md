## What This Changes

The PR changes `buildTimeoutAbortSignal` to clamp timeout delays with `resolveSafeTimeoutDelayMs` and adds regression tests for oversized delays in the shared signal builder and `fetchWithTimeout` wrapper.

## Best Possible Solution

Land the narrow clamp through the existing shared timer-delay helper after the normal required checks pass, keeping the regression tests on both the helper and public fetch wrapper.

## Reproduction Assessment

Yes. Source plus Node's timer contract gives a high-confidence reproduction path: pass `MAX_SAFE_TIMEOUT_DELAY_MS + 1` through current `buildTimeoutAbortSignal` and Node will treat the oversized `setTimeout` delay as `1`.

## Solution Assessment

Yes. Reusing `resolveSafeTimeoutDelayMs` is the narrowest maintainable fix because the safe timer ceiling already exists and the PR avoids adding a second clamp implementation.
