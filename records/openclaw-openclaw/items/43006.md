## Summary

Keep open. Current main still exposes and executes one scalar memory-flush plan with one flush marker per compaction cycle, while related RFC PR #43102 was closed unmerged, so progressive staged flush behavior remains unimplemented and needs maintainer design.

## What This Changes

Request staged pre-compaction memory flushes at increasing context pressure, with gentle, urgent, and forced-summary stages instead of one soft-threshold flush.

## Best Possible Solution

Design a backwards-compatible staged memory-flush contract covering staged config/schema, plugin plan shape, per-stage prompts/results, persisted dedupe state, forced-summary handoff, docs, and focused runner plus memory-core tests.

## Reproduction Assessment

Not applicable as a feature request. Source, schema, tests, and docs give a high-confidence check that current main has only a single scalar pre-compaction memory-flush path.

## Solution Assessment

Unclear until maintainer design. The staged direction is plausible, but the maintainable path should be an additive staged memory-flush contract rather than hard-coded 80/90/95 behavior inside the runner.
