## Summary

Keep open. Current main still implements memory flush as a single scalar gate with one per-compaction-cycle dedupe marker; the RFC PR is closed/unmerged, and the related durability issue only partially overlaps this staged progressive-flush request.

## What This Changes

Request progressive, multi-stage memory flush before compaction with staged prompts/thresholds and a forced-summary fallback instead of a single soft threshold.

## Best Possible Solution

Design an additive staged memory-flush contract covering thresholds, prompts, forced-summary behavior, persisted per-stage dedupe, docs, and compatibility for existing scalar config and plugin resolvers.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, schema, and tests provide a high-confidence current-main check that only one scalar memory-flush stage exists.

## Solution Assessment

Unclear until maintainer design. The progressive direction is plausible, but it should be an additive config/plugin stage contract with explicit state rather than hard-coded 80/90/95 behavior in the runner.
