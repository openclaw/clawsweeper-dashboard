## Summary

Keep open. Current main still has a single scalar memory-flush threshold and one flush turn per compaction cycle, while the related RFC PR #43102 was closed unmerged, so the progressive 80%/90%/95% staged behavior remains unimplemented and needs maintainer design.

## What This Changes

Request a multi-stage pre-compaction memory flush that escalates from reminders to urgent memory writes and forced summary behavior as context usage approaches capacity.

## Best Possible Solution

Design this as a backwards-compatible staged memory-flush contract: staged threshold config/schema, stage-specific prompts/results, persisted per-stage dedupe state in `sessions.json`, a defined forced-summary or compaction handoff, docs/help updates, and focused tests across the Gateway runner and memory-core plugin.

## Reproduction Assessment

Not applicable as a feature request. The current-main source, config schema, tests, and docs provide a high-confidence check that only one scalar pre-compaction flush path exists today.

## Solution Assessment

Unclear until maintainer design. The staged direction is plausible, but the narrow maintainable path is an additive memory-flush stage contract rather than hard-coded 80/90/95 behavior inside the runner.
