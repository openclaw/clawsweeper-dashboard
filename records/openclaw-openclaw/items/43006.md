## Summary

Keep open. Current main still exposes a single scalar pre-compaction memory-flush plan and one per-compaction-cycle flush marker, while the linked RFC PR was closed unmerged and the related durability issue does not fully cover staged flush behavior.

## What This Changes

Request progressive, multi-stage memory flush before compaction instead of the current single soft-threshold flush.

## Best Possible Solution

Design a backwards-compatible staged memory-flush contract covering config/schema, plugin plan shape, per-stage prompts/results, persisted dedupe state, forced-summary handoff, docs, and focused runner plus memory-core tests.

## Reproduction Assessment

Not applicable as a feature request. Source, schema, docs, and tests give a high-confidence check that current main only supports a single scalar pre-compaction memory-flush path.

## Solution Assessment

Unclear until maintainer design. The staged direction is plausible, but hard-coding 80/90/95 behavior in the runner would be less maintainable than an additive config/plugin contract with explicit per-stage state.
