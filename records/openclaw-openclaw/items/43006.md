## Summary

Keep open. Current main still implements memory flush as a scalar, one-per-compaction-cycle pre-threshold hook, while the requested staged flush lifecycle would add new config/API/runtime policy that is not fully owned by the adjacent open warning and durability issues.

## What This Changes

Request a multi-stage memory flush lifecycle before compaction with escalating thresholds, prompts, and forced-summary behavior.

## Best Possible Solution

Design one additive staged memory-flush lifecycle with explicit stage state, configurable thresholds/prompts, forced-summary handoff semantics, docs, and focused runner/plugin/config coverage.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, schema, release-tag inspection, and tests give a high-confidence check that current main supports only a scalar memory-flush path and one cycle-level flush marker.

## Solution Assessment

Unclear until maintainer design. The staged direction is plausible, but hard-coding 80/90/95 behavior in the runner would be less maintainable than an additive config or plugin-provided lifecycle contract with explicit persisted stage state.
