## Summary

Keep open. Current main still uses one scalar memory-flush plan, one threshold gate, and one flush marker per compaction cycle; related RFC PR #43102 was closed unmerged, so progressive staged flushing remains unimplemented and needs maintainer design.

## What This Changes

Request staged pre-compaction memory flushes at escalating thresholds, including reminder, urgent durable-memory save, and forced-summary behavior.

## Best Possible Solution

Design a backwards-compatible staged memory-flush contract covering config/schema, plugin plan shape, per-stage prompts/results, persisted stage dedupe, forced-summary handoff, docs, and focused runner plus memory-core tests.

## Reproduction Assessment

Not applicable as a feature request. Source, tests, and docs give a high-confidence check that current main has only a single scalar pre-compaction memory-flush path.

## Solution Assessment

Unclear until maintainer design. The staged direction is plausible, but the maintainable path should be an additive staged contract rather than hard-coded 80/90/95 behavior inside the runner.
