## Summary

Keep open. Current main still exposes a single scalar memory-flush gate and one per-compaction-cycle flush marker; the closed RFC PR did not land, and the related durability issue does not fully cover staged progressive flush behavior.

## What This Changes

Request staged pre-compaction memory flushes with escalating reminders and a forced summary stage instead of one soft-threshold flush.

## Best Possible Solution

Design an additive staged memory-flush contract covering thresholds, prompts, persisted per-stage dedupe, forced-summary handoff, docs, and focused runner/plugin/config tests.

## Reproduction Assessment

Not applicable as a feature request. Source, schema, docs, and tests give a high-confidence check that current main only supports a single scalar memory-flush path.

## Solution Assessment

Unclear until maintainer design. The staged direction is plausible, but hard-coding 80/90/95 behavior in the runner would be less maintainable than an additive config or plugin-provided stage contract with explicit state.
