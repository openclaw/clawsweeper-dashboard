## Summary

Keep open. Current main still has a scalar, one-per-compaction-cycle memory flush; adjacent open issues cover warning/handoff or hard-threshold durability, but they do not fully replace a staged progressive memory-flush design.

## What This Changes

Request progressive multi-stage memory flushing before compaction with gentle and urgent prompts plus a final forced-summary stage instead of the current single soft-threshold flush.

## Best Possible Solution

Design an additive staged memory-flush lifecycle covering configurable thresholds, prompts, persisted per-stage state, forced-summary handoff, docs, and focused runner/plugin/config tests.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, schema, and tests give a high-confidence check that current main supports only a single scalar memory-flush path and one cycle-level flush marker.

## Solution Assessment

Unclear until maintainer design. The staged direction is plausible, but hard-coding 80/90/95 behavior in the runner would be less maintainable than an additive config or plugin-provided lifecycle contract with explicit stage state.
