## Summary

Keep open. Current main still implements one scalar pre-compaction memory-flush plan with one flush marker per compaction cycle; the linked RFC PR was closed unmerged, and the broader durability issue does not replace this staged-flush request.

## What This Changes

Request progressive pre-compaction memory flush stages, such as gentle reminder, urgent flush, and forced summary as context pressure rises.

## Best Possible Solution

Design a backwards-compatible staged memory-flush contract covering staged config/schema, plugin plan shape, per-stage prompts/results, persisted dedupe state, forced-summary handoff, docs, and focused runner plus memory-core tests.

## Reproduction Assessment

Not applicable as a feature request. Source, schema, docs, and tests give a high-confidence check that current main only supports a single scalar pre-compaction memory-flush path.

## Solution Assessment

Unclear until maintainer design. The staged direction is plausible, but hard-coding 80/90/95 behavior in the runner would be less maintainable than an additive config/plugin contract with explicit per-stage state.
