## Summary

Keep open. This PR targets a real Chat Completions parity gap with the existing Responses client-tool path, but the current patch is not ready because streaming can still finalize before pending tool-call metadata is available, client-tool name conflicts fall through as internal errors, and the user-facing API change lacks the required changelog entry.

## What This Changes

The PR extends `src/gateway/openai-http.ts` and its gateway tests to parse Chat Completions `tools`/`tool_choice`, forward client tools to the agent command, emit non-stream and streaming `tool_calls`, and accept `role: "tool"` follow-up messages.

## Best Possible Solution

Revise this branch or create a narrow replacement that mirrors the existing Responses client-tool contract in Chat Completions: parse and forward client tools, wait for the command result or a proven terminal condition before closing tool-call streams, map client-tool conflicts to OpenAI-compatible invalid request errors, and document/changelog the new public API behavior.

## Reproduction Assessment

Yes. The stream bug can be reproduced by emitting `lifecycle:end` and delaying `agentCommandFromIngress()` beyond the one-second timer before returning `meta.stopReason: "tool_calls"`; the conflict path can be reproduced by sending duplicate or built-in-conflicting client tool names and observing the generic Chat Completions error path.

## Solution Assessment

No. The product direction is right, but the timeout-based stream fallback is not the narrowest maintainable fix; Chat Completions should wait for the agent command result, abort, disconnect, or explicit error before finalizing tool-call streams, and should reuse or mirror Responses conflict handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wait for the command result before closing streams:** `src/gateway/openai-http.ts:971-974`
  - body: When `lifecycle:end` arrives before `agentCommandFromIngress(...)` resolves, this timer allows finalization after one second while `resultResolved` is still false. The stream can close with `finish_reason: "stop"`; if the command later resolves with `pendingToolCalls`, the handler returns because `closed` is already true, so clients never receive the `delta.tool_calls` or `tool_calls` finish reason needed for the follow-up loop.
  - confidence: 0.91
- **[P2] Map client tool name conflicts to 400 errors:** `src/gateway/openai-http.ts:839`
  - body: Forwarding `resolvedClientTools` lets duplicate or built-in-conflicting names reach the runner, where `createClientToolNameConflictError()` is thrown. Unlike `/v1/responses`, this adapter does not catch that error, so a malformed client tool schema becomes a generic 500 or stream error instead of an OpenAI-compatible `invalid_request_error`.
  - confidence: 0.79
- **[P3] Add the required user-facing changelog entry:** `src/gateway/openai-http.ts:839`
  - body: This change adds user-visible Chat Completions tool support, but the PR only changes gateway code and tests. Repo policy requires user-facing `fix`/`feat` changes to include a `CHANGELOG.md` entry before merge.
