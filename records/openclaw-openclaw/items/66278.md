## Summary

Keep open: the branch implements the requested Chat Completions tool protocol direction and adds useful coverage, but PR head still reports streaming client-tool conflicts as successful assistant text, the public Chat API docs are not updated, and the supplied live proof only shows request acceptance rather than the full tool-call/follow-up loop.

## What This Changes

The PR updates the OpenAI-compatible Chat Completions gateway adapter, tests, and changelog to parse client function tools/tool_choice, forward client tools, emit tool_calls in sync and SSE responses, accept tool-result messages, and adjust stream finalization.

## Best Possible Solution

Finish the Chat Completions tool contract so stream and non-stream behavior match the Responses client-tool failure model, document the public API subset, and require redacted real gateway proof of tool_calls plus the tool-result follow-up turn before merge.

## Reproduction Assessment

Yes. Source inspection shows current main's Chat adapter lacks the tools/tool_choice/tool_call_id/clientTools path while the Responses adapter has that contract; PR head also contains a source-level reproduction for the remaining streaming conflict behavior.

## Solution Assessment

No. The implementation direction is maintainable, but streaming tool-name conflicts should fail as protocol errors rather than successful assistant text, and the public API docs plus real behavior proof need to cover the new contract.

Clients will treat an invalid tool schema as a successful assistant answer, unlike the non-stream Chat path and the Responses stream path, so prevalidate before opening SSE or emit an error-shaped stream failure instead.

  - confidence: 0.82
