## Summary

Keep open: the PR fixes a real Chat Completions tool-protocol gap and the previous forced-tool-choice blocker was addressed, but the new docs still list forced `tool_choice` modes as supported even though the code and tests reject them.

## What This Changes

The PR updates the OpenAI-compatible Chat Completions gateway, tests, docs, and changelog to parse function tools/tool_choice, forward client tools, return non-stream and stream tool_calls, accept tool-result follow-up turns, and reject unsupported forced tool_choice modes.

## Best Possible Solution

Keep the adapter behavior that accepts auto/none and rejects forced modes until runtime enforcement exists, and update the docs to describe required/named function choices as unsupported for now.

## Reproduction Assessment

Yes. Current main is source-reproducible because the Chat Completions adapter has no `tools`/`tool_choice` request handling and always maps normal non-stream responses to `finish_reason: "stop"`; I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. The code path is now the right narrow solution, but the docs line listing forced `tool_choice` modes as supported contradicts the handler and regression tests.
