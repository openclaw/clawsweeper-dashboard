## Summary

Keep open: the PR addresses a real Chat Completions tool-protocol gap, but the latest head still advertises forced `tool_choice` support without a hard constraint and the supplied real-run proof does not show the advertised tool-call loop.

## What This Changes

The PR adds Chat Completions client function-tool parsing, `tool_choice` handling, `tool_calls` response mapping for streaming and non-streaming responses, tool-role follow-up prompt rendering, docs, tests, and a changelog entry.

## Best Possible Solution

Make the Chat Completions tool contract honest and testable: enforce forced `tool_choice` through the agent/provider path or reject forced modes until that seam exists, then add real gateway proof that shows `tool_calls` and the follow-up loop.

## Reproduction Assessment

Yes. Current main is source-reproducible because `/v1/chat/completions` has no `tools` or `tool_choice` request handling and always maps non-stream results to `finish_reason: "stop"`; I did not run tests because this was a read-only review.

## Solution Assessment

No, not yet. The adapter-level direction is the right place to solve the linked issues, but the PR should not claim support for forced `tool_choice` modes unless they are enforced or explicitly rejected.
