## Summary

Keep this PR open because current main still lacks Chat Completions client-tool support and the PR targets the right gateway gap, but the branch is not merge-ready. The latest head fixed the Greptile error-message polish, while the remaining blockers are the stream finalization race, client-tool conflict error mapping, and required public docs/changelog coverage for the new API behavior.

## What This Changes

The branch updates `src/gateway/openai-http.ts` and gateway tests so `/v1/chat/completions` parses `tools`/`tool_choice`, forwards client tools to agent runs, emits non-stream and streaming `tool_calls`, and accepts `role: "tool"` follow-up messages.

## Best Possible Solution

Revise this branch or create a narrow replacement that mirrors the existing `/v1/responses` client-tool contract in Chat Completions, preserves the gateway auth/model-routing boundary, waits for a real command result or terminal abort/error before finalizing tool-call streams, maps client-tool conflicts as invalid requests, and documents/changelogs the public API behavior.

## Reproduction Assessment

Yes. Current main can be verified by inspecting `src/gateway/openai-http.ts`: the Chat adapter never parses or forwards client tools and always emits normal stop responses. The PR-specific stream bug is reproducible by emitting `lifecycle:end` before delaying `agentCommandFromIngress()` beyond the one-second fallback, and the conflict path is reproducible with duplicate or built-in-conflicting client tool names.

## Solution Assessment

No. The proposed product direction is right, but the timeout-based stream finalization is not the narrowest maintainable fix and the error handling does not yet match the existing Responses client-tool contract. A safer solution should wait for command resolution or explicit abort/error and reuse or mirror the Responses conflict mapping.
