## Summary

Keep open: this PR addresses a real Chat Completions client-tool gap on current main, the supplied real behavior proof is sufficient, and I found no blocking correctness or security issue in the diff.

## What This Changes

This PR extends the OpenAI-compatible Chat Completions gateway endpoint, docs, and tests to parse function tools/tool_choice, return non-stream and streaming tool_calls, accept role: tool follow-up messages, and reject unsupported forced tool_choice modes.

## Best Possible Solution

Land this adapter-level implementation, or a maintainer-approved equivalent, so Chat Completions client-tool loops use the existing clientTools/pendingToolCalls seam without adding Home Assistant-specific behavior.

## Reproduction Assessment

Yes. Current origin/main is source-reproducible: the Chat Completions adapter has no tools/tool_choice request path, no clientTools handoff, and no non-stream or streaming tool_calls response mapping.

## Solution Assessment

Yes. The PR is the narrow maintainable direction because it reuses the existing Responses client-tool seam, confines behavior to the Chat Completions adapter, documents the subset, and rejects forced tool_choice modes until runtime enforcement exists.
