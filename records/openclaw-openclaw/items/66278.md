## Summary

Keep this active PR open: the branch addresses a real gateway Chat Completions tool-protocol gap, includes focused tests and live after-fix proof, and I did not find a concrete correctness or security blocker in the reviewed diff.

## What This Changes

The PR updates the OpenAI-compatible Chat Completions gateway adapter, tests, docs, and changelog to support function tools, `tool_choice`, `tool_calls`, and `role: "tool"` follow-up turns while rejecting unsupported forced modes.

## Best Possible Solution

Land the focused gateway contract once normal CI and merge gates are green, preserving forced `tool_choice` rejection until the agent runtime can enforce it correctly.

## Reproduction Assessment

Yes, source-level. Current main's `/v1/chat/completions` adapter does not parse `tools`/`tool_choice` or forward `clientTools`, and it emits stop-shaped Chat responses instead of OpenAI-compatible tool-call responses.

## Solution Assessment

Yes. Reusing the existing `clientTools` and `pendingToolCalls` gateway seam is the narrow maintainable path, and explicitly rejecting forced `tool_choice` modes is safer than pretending enforcement exists in the agent runtime.
