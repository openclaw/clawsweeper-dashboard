## Summary

Keep this PR open: the latest branch implements the right Gateway API direction and addressed earlier stream-race/changelog/non-stream conflict concerns, but streaming client-tool name conflicts still surface as normal assistant text, and the supplied real proof does not demonstrate the new tool-call loop.

## What This Changes

The branch updates the OpenAI-compatible Chat Completions gateway adapter, tests, and changelog to parse `tools`/`tool_choice`, forward client tools, emit `tool_calls` in sync and SSE responses, accept `role: "tool"` follow-ups, and handle stream finalization and conflict cases.

## Best Possible Solution

Update this branch so Chat Completions mirrors the Responses client-tool contract across non-stream and stream modes, documents the public API behavior, and includes redacted real gateway proof of the tool-call/follow-up loop.

## Reproduction Assessment

Yes. Source inspection of current main shows the Chat adapter ignores the client-tool protocol while the adjacent Responses adapter implements it; the remaining streaming-conflict bug is visible at PR head without running the gateway.

## Solution Assessment

No. The product direction is maintainable, but the streaming conflict path should not turn an invalid client tool schema into assistant prose, and the PR still needs API docs plus stronger real behavior proof.
