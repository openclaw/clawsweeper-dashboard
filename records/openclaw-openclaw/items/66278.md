## Summary

Keep open: this PR addresses a real Chat Completions tool-protocol gap and now includes sufficient live-output proof, but it still advertises forced `tool_choice` modes that are only prompt hints instead of enforced runtime constraints.

## What This Changes

This PR updates Chat Completions gateway code, docs, tests, and changelog to accept function tools/tool_choice, emit tool_calls in streaming and non-stream responses, and render tool-role follow-up messages.

## Best Possible Solution

Keep the Chat Completions client-tool and tool-call mapping direction, but make the forced tool_choice contract honest by either enforcing it through the runtime/provider path or rejecting forced variants until that seam exists.

## Reproduction Assessment

Yes. Current main is source-reproducible because the Chat Completions adapter does not parse `tools`/`tool_choice` and its response paths emit text/stop rather than OpenAI-style tool calls.

## Solution Assessment

No, not yet. The adapter and test direction is the right place to solve the linked problems, but forced `tool_choice` modes need a real runtime constraint or an explicit rejection before the PR should merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject required tool_choice until it is enforceable:** `src/gateway/openai-http.ts:206-213`
  - body: This accepts `tool_choice: "required"` as supported but only appends an extra system prompt, so a model that ignores the hint can still return a normal assistant message with `finish_reason: "stop"` and no `tool_calls`. The Chat contract treats `required` as forcing one or more tool calls, so reject this mode until the runtime can enforce it or thread a real forced-tool constraint through the agent run.
  - confidence: 0.86
