## Summary

Keep open: this PR addresses a real Chat Completions tool-protocol gap, but it still advertises forced `tool_choice` modes without enforcing them, and the supplied terminal proof shows a normal `finish_reason: "stop"` response rather than the advertised tool-call loop.

## What This Changes

The branch adds Chat Completions client-tool parsing, `tool_choice` handling, `tool_calls` response mapping, `role: "tool"` follow-up rendering, docs, a changelog entry, and gateway tests.

## Best Possible Solution

Keep the adapter-level tool-call mapping, but either enforce forced Chat `tool_choice` through the agent/provider path or reject those modes until supported, then add real gateway proof for non-stream tool calls, streaming tool calls, and the follow-up turn.

## Reproduction Assessment

Yes. Current main is source-reproducible: `/v1/chat/completions` has no `tools`/`tool_choice` request handling and its non-stream response always maps to `finish_reason: "stop"`.

## Solution Assessment

Not yet. The Chat adapter is the right surface for the linked issues, but the PR should not mark forced `tool_choice` modes supported unless they are truly enforced or deliberately rejected.

## Review Findings
