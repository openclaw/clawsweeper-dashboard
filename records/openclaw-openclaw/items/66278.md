## Summary

Keep open: this PR is the active implementation for the linked Chat Completions tool-protocol gap, but it needs changes before merge because the forced `tool_choice` modes are accepted as supported without being enforced, and the supplied real behavior proof does not show actual `tool_calls` or the follow-up loop.

## What This Changes

The PR adds Chat Completions client function-tool parsing, `tool_choice` handling, `tool_calls` response mapping, tool-result follow-up prompt rendering, docs, a changelog entry, and gateway tests.

## Best Possible Solution

Make the Chat Completions tool contract honest and testable: either enforce forced `tool_choice` through the agent/provider path, or reject those modes until that seam exists, then add real gateway proof showing `tool_calls` and the follow-up loop.

## Reproduction Assessment

Yes. Current main is source-reproducible: `/v1/chat/completions` has no `tools` or `tool_choice` request handling and always maps non-stream results to `finish_reason: "stop"`; I did not run local tests because this was a read-only review.

## Solution Assessment

Not yet. The adapter-level direction is the right place to solve the linked issues, but the PR should not claim supported forced `tool_choice` modes unless they are truly enforced or deliberately rejected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Reject forced tool_choice until it is enforced:** `src/gateway/openai-http.ts:210-212`
  - body: This accepts `tool_choice: "required"` as supported but only adds an `extraSystemPrompt`, so the model can still return a normal assistant message with `finish_reason: "stop"` and no `tool_calls`. OpenAI-compatible clients use `required` or a named function to force the tool-call loop; either wire a real forced-tool constraint through the run or fail fast for these modes instead of advertising support.
