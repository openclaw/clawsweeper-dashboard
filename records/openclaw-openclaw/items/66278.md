## Summary

Keep this PR open: it targets a real Chat Completions tool-protocol gap that current main still has, but the proposed patch is not merge-ready because it can still drop delayed streamed tool-call metadata, does not map client-tool name conflicts like the adjacent Responses adapter, and is missing required public docs/changelog coverage.

## What This Changes

The PR updates `src/gateway/openai-http.ts` and gateway tests so `/v1/chat/completions` parses `tools`/`tool_choice`, forwards client tools to agent runs, emits non-streaming and streaming `tool_calls`, and accepts `role: "tool"` follow-up messages.

## Best Possible Solution

Land a corrected Chat Completions adapter that mirrors the existing Responses client-tool contract, preserves the Gateway auth/model-routing boundary, waits for a real command result or terminal abort/error before final stream closure, and documents the supported function-tool subset.

## Reproduction Assessment

Yes. Current main can be verified by code inspection: the Chat adapter does not parse/forward client tools or emit `tool_calls`; the PR-specific failures are reproducible with `lifecycle:end` before a command result delayed beyond one second and with duplicate or built-in-conflicting client tool names.

## Solution Assessment

No. The product direction is right, but the timeout-based stream finalization and missing conflict mapping are not the narrowest maintainable fix; the safer path is to wait on command resolution or explicit abort/error and reuse or mirror the Responses invalid-tool-configuration handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wait for command results before closing tool-call streams:** `src/gateway/openai-http.ts:971-974`
  - body: When `lifecycle:end` arrives before `agentCommandFromIngress(...)` resolves, this one-second fallback can let `maybeFinalize()` close the SSE response while `resultResolved` is still false. If the command result later contains `pendingToolCalls`, the closed guard drops the `delta.tool_calls` chunks and terminal `finish_reason: "tool_calls"`, breaking the client follow-up loop.
  - confidence: 0.91
- **[P2] Map client-tool conflicts to Chat invalid requests:** `src/gateway/openai-http.ts:839`
  - body: Once `clientTools` are forwarded, duplicate names or collisions with built-in/reserved tools can throw `createClientToolNameConflictError()` in the runner. Unlike `/v1/responses`, this adapter still falls through to the generic catch, so invalid client tool schemas become 500/internal-error responses instead of OpenAI-compatible `invalid_request_error` results.
  - confidence: 0.84
- **[P3] Document the Chat tool contract:** `src/gateway/openai-http.ts:787-790`
  - body: This introduces public `tools`, `tool_choice`, `tool_calls`, and `role: "tool"` behavior, but the Chat Completions docs still only describe basic messages and streaming. Please document the supported function-tool subset, unsupported `tool_choice` variants, streaming/non-stream shapes, and follow-up loop.
  - confidence: 0.82
