## Summary

Keep this PR open: the updated branch resolves the earlier forced-tool concern by rejecting unsupported forced `tool_choice` modes, and the supplied live gateway output is enough real behavior proof. One docs line still contradicts the implemented contract by listing `required` and named-function `tool_choice` as supported while the code and tests return `400 invalid_request_error` for those shapes.

## What This Changes

The PR updates the OpenAI-compatible Chat Completions gateway, docs, tests, and changelog to parse client function tools/tool_choice, emit tool_calls in non-stream and streaming responses, accept tool-role follow-up turns, and reject unsupported forced modes.

## Best Possible Solution

Keep the gateway implementation direction, but align the Chat tool contract docs so only `auto` and `none` are documented as accepted and forced choices are clearly documented as temporary 400 responses until runtime enforcement exists.

## Reproduction Assessment

Yes. Current main is source-reproducible: the Chat Completions adapter lacks tools/tool_choice request handling and always maps non-stream results to finish_reason "stop" without tool_calls.

## Solution Assessment

No, not quite. The implementation path is now the right narrow fix, but the PR should not merge until the public docs match the explicit rejection behavior for forced tool_choice modes.
