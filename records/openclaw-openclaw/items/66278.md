## Summary

Keep open: this PR targets a real Gateway compatibility gap and adds the right general contract, but it should not merge yet because follow-up prompts still drop assistant `tool_calls`, and the supplied real behavior proof only shows request acceptance with a normal stop response.

## What This Changes

The PR updates the Gateway Chat Completions adapter, docs, tests, changelog, and one Discord voice test mock to support function tools/tool_choice, emit tool_calls, handle role:tool follow-ups, and align stream finalization.

## Best Possible Solution

Keep the PR open, preserve assistant `tool_calls` in follow-up prompt translation, then require redacted real Gateway proof that shows both `tool_calls` output and the `role: "tool"` follow-up before normal maintainer review.

## Reproduction Assessment

Yes. Source inspection of current main shows `/v1/chat/completions` ignores `tools`/`tool_choice`/`tool_call_id` and never forwards `clientTools`, while the adjacent Responses endpoint already has that path; I did not run live requests because this review is read-only.

## Solution Assessment

No, not as-is. The PR uses the right Gateway seam, but it should serialize assistant `tool_calls` from follow-up messages and needs stronger real behavior proof before merge.
