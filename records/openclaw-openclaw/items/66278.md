## Summary

Keep open: the PR is useful, but latest head still treats a streaming client-tool name conflict as a successful assistant stop message, lacks docs for the new public Chat tool contract, and the supplied terminal proof only demonstrates request acceptance rather than the tool_calls/follow-up loop.

## What This Changes

The PR updates the Gateway Chat Completions adapter, tests, and changelog to parse tools/tool_choice, forward client tools, emit tool_calls in sync and SSE responses, accept role: tool follow-up turns, and adjust stream finalization, with small Discord and update-fixture test adjustments.

## Best Possible Solution

Finish the Chat Completions tool contract by aligning streaming conflict failures with the Responses/non-stream error model, documenting the supported API subset, and requiring redacted real gateway proof of tool_calls plus the tool-result follow-up turn before merge.

## Reproduction Assessment

Yes. Current main's Chat adapter lacks the tools/tool_choice/tool_call_id/clientTools path while the Responses adapter has that contract, and PR-head tests show the remaining streaming conflict behavior.

## Solution Assessment

No. The implementation direction is maintainable, but streaming tool-name conflicts should fail as protocol errors rather than successful assistant text, and the public docs plus real behavior proof need to cover the new contract.
