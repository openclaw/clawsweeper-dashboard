## Summary

Keep open. Current main still ships the reported broad mandatory memory-recall wording in the default memory plugin prompt and the `memory_search` tool description, and the generic live-check guidance in the main system prompt does not explicitly carve out current configuration, service, time, or system-state questions.

## What This Changes

The issue asks OpenClaw to narrow memory recall guidance so historical questions use memory search while current system state, configuration, time, and service-status questions go directly to live verification tools.

## Best Possible Solution

The durable end state is a memory-core prompt/tool contract that treats memory search as historical recall for prior work, decisions, durable people/preferences/todos, and remembered dates, while directing mutable facts such as current config, service health, running state, clock/time, versions, and filesystem/package state to live verification first.

## Reproduction Assessment

Yes for the source-level defect: building the memory prompt with `memory_search`/`memory_get` still emits mandatory recall guidance, and the tool metadata still labels `memory_search` as mandatory for broad categories. The exact model trajectory is not live-reproduced here, but the shipped instructions still make the reported behavior plausible.

## Solution Assessment

Yes, the narrow maintainable fix is to update the memory-core prompt section and tool description, then add regression tests that preserve historical recall while excluding current-state/config/time/service queries from mandatory memory search. A broader core prompt rewrite is not needed unless the focused tests show composition still overrides the intended behavior.
