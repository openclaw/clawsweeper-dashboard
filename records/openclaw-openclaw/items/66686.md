## Summary

Keep open. Current main still contains the reported mandatory memory-recall wording in both the default memory prompt section and the `memory_search` tool metadata, while the generic live-check guidance does not clearly override it for current configuration, service, clock, or system-state questions.

## What This Changes

The issue requests narrowing memory-search prompt/tool guidance so historical recall still uses memory while current system-state verification uses live checks first.

## Best Possible Solution

Update the memory-core prompt and `memory_search` description to distinguish historical recall from mutable live state, then add focused regression tests that preserve recall for prior work/preferences/todos while steering current config, service health, clock/time, versions, and filesystem/package state questions to live verification first.

## Reproduction Assessment

Yes for the source-level defect: building the default memory prompt still emits mandatory `memory_search` guidance, and the tool metadata still labels the same broad category as mandatory recall. The exact model decision path is not needed to confirm the shipped instruction conflict.

## Solution Assessment

Yes, a narrow prompt/tool wording change plus regression coverage is the best first fix. A broader system-prompt rewrite or new plugin API is not needed unless focused tests show another prompt layer still overrides the intended current-state exception.
