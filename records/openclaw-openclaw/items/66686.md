## Summary

Keep open. Current main still contains the reported mandatory memory-recall wording in the memory-core prompt and tool metadata, while the live-state guidance is generic and does not give current configuration/service/time questions an explicit exception.

## What This Changes

The issue asks OpenClaw to distinguish historical memory recall questions from current system-state verification questions so live state is checked directly first.

## Best Possible Solution

Refine the memory-core prompt and memory_search descriptions to make historical recall the target and explicitly route current config, service health, clock/time, versions, filesystem, and package-state questions to live verification first.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main still emits mandatory memory_search guidance from memory-core and exposes the same wording in tool metadata, so a model can reasonably follow memory first for state-verification questions.

## Solution Assessment

Yes. A narrow wording change plus regression tests is the best first fix; broader prompt rewrites or new plugin APIs are unnecessary unless focused tests expose another overriding layer.
