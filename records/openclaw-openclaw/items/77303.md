## Summary

This should stay open: current main still carries the selected thinking level into the CLI run metadata, but the Anthropic `claude-cli` argv path never maps it to Claude Code's session-scoped `--effort` flag, so the status surface can advertise a control that the backend does not enforce.

## What This Changes

The issue requests that `claude-cli` sessions either map OpenClaw thinking levels to Claude Code `--effort` or clearly mark the status controls unsupported for that runtime.

## Best Possible Solution

Map supported non-off OpenClaw thinking levels to Claude CLI `--effort` for both fresh and resumed/live sessions, restart live sessions when effort changes via argv fingerprinting, and document that `/reasoning` controls visibility while `/think` controls budget.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: choose `claude-cli`, set a non-off `/think` level, and the run still builds argv from backend args that contain no `--effort`. I did not perform a live Claude CLI run because the `claude` binary is absent here.

## Solution Assessment

Yes, the narrow maintainable fix is to consume the existing `thinkLevel` in the Claude CLI backend path rather than adding a new user config knob. Reasoning visibility should remain a separate display setting unless Claude stream-json reasoning events need parser support.
