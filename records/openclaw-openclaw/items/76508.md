## Summary

Keep open: current main does not implement community-plugin keyed-store access, and this PR is not merge-ready because the documented manifest opt-in would be dropped by real manifest parsing and the branch includes an A2A sessions_send behavior change without matching changelog coverage.

## What This Changes

The PR adds a `contracts.usesKeyedStore` manifest opt-in and docs/tests for community-plugin keyed-store access, and also changes A2A `sessions_send` announce delivery to reply through the requester session.

## Best Possible Solution

If maintainers accept the SDK direction, land a manifest-normalized and fully documented keyed-store capability with parser/registry coverage, and either document or split the unrelated A2A sessions_send behavior change.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a real `openclaw.plugin.json` with `contracts.usesKeyedStore: true` would pass through `normalizeManifestContracts`, which currently preserves only known string-list fields, while the PR tests bypass that parser by mutating `record.contracts` directly.

## Solution Assessment

No. The manifest opt-in is a plausible shape if maintainers approve it, but this patch must preserve the new field through manifest loading, keep existing list-contract helpers type-safe, and document or split the A2A sessions_send change before merge.
