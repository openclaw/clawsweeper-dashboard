## Summary

Keep open. Current main still rejects a `modelAliases`-only CLI backend override before resolver inheritance, but this PR needs refreshed handling for current prepared/live execution paths, the config-doc baseline update, and contributor-supplied real behavior proof before merge.

## What This Changes

The PR makes `agents.defaults.cliBackends.<backendId>.command` optional in config schema/types, adds a resolved backend type, narrows the normal spawn command, and adds a partial-override regression test.

## Best Possible Solution

Accept partial CLI backend override input while preserving `command` as a resolved non-empty string through prepared and live execution paths, refresh the config docs baseline, and require redacted real-run proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main requires `command` in the schema/type while the resolver already merges registered backends with overrides and validates the resolved command after merge, so a `modelAliases`-only override is blocked before inheritance.

## Solution Assessment

No. Making input `command` optional is the right direction, but the branch should also carry the resolved command contract through prepared/live execution paths and update the config docs baseline.
