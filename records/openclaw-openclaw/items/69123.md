## Summary

Keep open: current main still rejects the reported partial CLI backend override, and this PR is a plausible fix path, but the branch needs rebase-aware code fixes, config-baseline refresh, and contributor-supplied real behavior proof before merge.

## What This Changes

The PR makes `agents.defaults.cliBackends.<backendId>.command` optional in config schema/types, adds a resolved backend type plus a spawn guard, and adds a modelAliases-only regression test.

## Best Possible Solution

Accept partial CLI backend override input while preserving `command` as a resolved non-empty string through prepared and live execution paths, refresh the config baseline hash, and require redacted real-run proof.

## Reproduction Assessment

Yes, source-reproducible. Current main requires `command` in both the schema and config type while docs and resolver code define user CLI backend config as merging over plugin defaults, so a modelAliases-only override is blocked before inheritance.

## Solution Assessment

No. Making input `command` optional is the right direction, but the branch should also carry the resolved command contract through prepared/live execution paths and update the config docs baseline before merge.
