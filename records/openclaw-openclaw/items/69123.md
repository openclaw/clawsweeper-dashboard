## Summary

Keep open. The PR targets a real config-validation bug, but the branch is incomplete on current main because the resolved command guarantee is not carried through prepared/live CLI execution and the config docs baseline would drift.

## What This Changes

The PR makes CLI backend `command` optional for partial `agents.defaults.cliBackends` overrides, adds a resolved backend type, narrows normal CLI spawning, and adds a regression test.

## Best Possible Solution

Finish the branch by treating CLI backend config input as partial while preserving `command: string` after resolution through bundle-MCP preparation, normal spawn, and Claude live sessions, then refresh the current config docs baseline and add focused proof.

## Reproduction Assessment

Yes. Current-main source shows `CliBackendSchema.command` and `CliBackendConfig.command` are required while `mergeBackendConfig` and `resolveCliBackendConfig` already support inheriting and validating a command after merge, so a `modelAliases`-only override is blocked at validation.

## Solution Assessment

No. Making the input field optional is the right direction, but this PR should also carry the resolved command type through prepared/live execution and refresh the current config docs baseline.
