## Summary

Keep open. Current main still rejects the partial `cliBackends` override this PR targets, and the branch is directionally correct, but the latest patch is not merge-ready because it leaves generated schema consumers stale and does not carry the resolved `command: string` invariant through the live-session prepared-backend path.

## What This Changes

The PR makes `CliBackendConfig.command` and `CliBackendSchema.command` optional, adds a resolved backend type, narrows the normal CLI spawn command, and adds a regression test for a `modelAliases`-only Claude CLI override.

## Best Possible Solution

Finish the branch by treating CLI backend config input as partial while preserving a resolved `command: string` type across resolver, bundle-MCP preparation, normal spawn, and Claude live-session execution; regenerate config schema/docs baselines and add the required changelog entry.

## Reproduction Assessment

Yes. A config object with `agents.defaults.cliBackends.claude-cli.modelAliases` and no `command` reproduces the current-main validation rejection, and the PR's remaining defects are reproducible by static inspection of the unchanged generated schema and live-session command path.

## Solution Assessment

No. The source schema/type relaxation is the right direction, but this patch is incomplete because schema-driven consumers remain strict and one prepared-backend execution path loses the resolved command guarantee.
