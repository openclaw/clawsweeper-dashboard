## Summary

Keep open: current main still rejects modelAliases-only `cliBackends` overrides and this PR targets that bug, but the branch is not mergeable as-is because the optional `command` type is not carried through prepared/live backend execution, the config docs baseline is stale, and the external PR lacks real behavior proof.

## What This Changes

The PR makes `agents.defaults.cliBackends.<backendId>.command` optional in config schema/types, adds a resolved backend type plus a normal-spawn guard, and adds a regression test for a `modelAliases`-only override.

## Best Possible Solution

Treat CLI backend config input as partial while preserving `command` as a resolved non-empty string everywhere execution can spawn, refresh the config docs baseline, and add focused real-run proof.

## Reproduction Assessment

Yes, from source inspection. Current main requires `command` in schema and type validation while the resolver already merges registered backends with overrides and guards the resolved command after merge, so a `modelAliases`-only override is blocked before inheritance can happen.

## Solution Assessment

No. Making input `command` optional is the right direction, but the branch should also carry the resolved command type through prepared, bundle-MCP, and Claude live execution and update the config docs baseline.
