## Summary

Keep open: current main still has a real source-level degradation path for builtin memory search when the running `node:sqlite` lacks FTS5, and the open closing PR only addresses doctor visibility rather than restoring keyword/BM25 recall.

## What This Changes

The issue asks OpenClaw to handle or visibly diagnose missing FTS5 support in `node:sqlite` so builtin memory hybrid search does not silently lose keyword/BM25 recall.

## Best Possible Solution

Keep this open until a maintainer lands the diagnostic/docs slice and decides whether builtin memory should require a Node SQLite build with FTS5 or provide another supported keyword-search backend.

## Reproduction Assessment

Yes, with an affected Node build. When `CREATE VIRTUAL TABLE ... USING fts5` throws, current source marks FTS unavailable and the hybrid path skips keyword results; my local Node v24.14.1 probe has FTS5, so I did not reproduce the dependency failure locally.

## Solution Assessment

Unclear. The open doctor warning is a good narrow visibility improvement, but restoring exact-token/BM25 recall on affected runtimes needs a deliberate runtime, dependency, or fallback strategy.
