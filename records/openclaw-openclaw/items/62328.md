## Summary

Keep open: current main and v2026.6.8 still treat missing FTS5 as `fts.available=false`, so memory search either drops keyword/BM25 recall or returns no keyword-only results. The open diagnostic PR improves visibility but does not restore keyword fallback, and the broader backend replacement path still needs maintainer approval and proof.

## What This Changes

The issue asks OpenClaw to restore or visibly diagnose memory-core keyword/BM25 search when the active `node:sqlite` build lacks FTS5.

## Best Possible Solution

Adopt a maintainer-approved memory-core strategy that restores keyword recall when FTS5 is missing, or explicitly documents and diagnoses the degraded mode while a backend replacement is reviewed.

## Reproduction Assessment

Yes at source level: current main sets `fts.available=false` when FTS5 table creation fails, then skips keyword/BM25 search or returns no keyword-only results. I did not live-run an affected Node 22/23 binary in this read-only review.

## Solution Assessment

No complete fix is on main. The current diagnostic PR is a useful partial step, but restoring keyword recall or changing the SQLite backend needs maintainer direction and upgrade proof.
