## Summary

Keep open: current main still marks FTS unavailable when `CREATE VIRTUAL TABLE ... USING fts5` fails, then either returns no keyword-only results or skips BM25 keyword recall in hybrid search. The open doctor PR improves visibility but does not restore keyword fallback, and the broader SQLite-backend direction still needs maintainer approval and proof.

## What This Changes

The issue requests restoring or clearly diagnosing memory-core keyword/BM25 recall when the active `node:sqlite` runtime lacks FTS5 support.

## Best Possible Solution

Adopt a maintainer-approved memory-core strategy that restores lexical recall when FTS5 is missing, or explicitly documents and diagnoses the degraded mode while a backend replacement is reviewed.

## Reproduction Assessment

Yes at source level: current main records `fts.available=false` when FTS5 table creation fails, then skips keyword/BM25 search or returns no keyword-only results. I did not live-run an affected Node 22/23 binary in this checkout because the local Node v24.16.0 build has FTS5 enabled.

## Solution Assessment

No complete fix is on main. The current diagnostic PR is a useful partial mitigation, but restoring keyword recall or changing the SQLite backend needs maintainer direction, dependency review, and upgrade proof.
