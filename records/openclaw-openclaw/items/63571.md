## Summary

Keep open. Current main still lacks explicit web-search provider fallbacks, but this PR is not merge-ready because it misses current runtime/provider contracts, validation/docs requirements, and external real behavior proof.

## What This Changes

The PR adds a `tools.web.search.fallbacks` config field, generated config metadata, explicit-provider retry logic in `runWebSearch`, and fallback-order runtime tests.

## Best Possible Solution

Define a backward-compatible fallback contract that preserves strict explicit-provider behavior by default, validates provider ids through plugin metadata, documents query fan-out, and proves the real fallback flow.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config PR. Current-main source inspection shows the fallback config field is absent and explicit provider failures remain fail-fast today.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation needs to be rebased onto current provider-load, runtime config, abort-signal, validation, and documentation contracts before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Load fallback provider owners before resolving candidates:** `src/web-search/runtime.ts:352-353`
  - body: Current main scopes explicit web-search provider loading to the configured provider's owner plugin before resolving candidates. This branch searches fallback ids only inside the already-loaded `candidates`, so a fallback owned by another plugin can never be loaded after rebase and the configured chain may fail fast.
  - confidence: 0.88
- **[P2] Preserve resolved config and abort signals:** `src/web-search/runtime.ts:375-387`
  - body: The fallback path still passes `params.config` into `createTool` and calls `execute(params.args)` without the abort signal. Current main resolves the applicable runtime config snapshot and forwards `params.signal`, so this can bypass resolved SecretRefs and ignore cancellation.
  - confidence: 0.9
- **[P2] Validate fallback provider ids with plugin metadata:** `src/config/zod-schema.agent-runtime.ts:303`
  - body: Current main validates `tools.web.search.provider` against manifest-owned web-search provider ids with stale-plugin warning behavior. Adding `fallbacks` as a plain string array accepts typos and unavailable plugin providers without the same validation.
  - confidence: 0.88
- **[P2] Document fallback query fan-out:** `src/config/types.tools.ts:490-491`
  - body: This config can send the same search query to additional providers after the primary fails, but the public web-search docs are not updated. Users need ordering, explicit-mode behavior, validation, and query fan-out expectations documented before this config ships.
  - confidence: 0.84
