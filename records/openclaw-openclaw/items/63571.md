## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve runtime config and cancellation context:** `src/web-search/runtime.ts:375-387`
  - body: This fallback loop still calls provider tools with `params.config`, no `agentDir`, and no abort signal. Current main resolves the active runtime config snapshot, passes `agentDir` for auth-profile lookup, and forwards `{ signal }` to provider execution; rebasing this as-is would break resolved SecretRefs, agent-scoped credentials, and cancellation.
  - confidence: 0.88
- **[P2] Classify fallback failures before retrying:** `src/web-search/runtime.ts:405-410`
  - body: In explicit fallback mode this catch path retries every thrown provider failure. That can retry auth, permission, or deterministic parameter errors on a different provider and send the same query across another trust/cost boundary, while the canonical request limits fallback to recoverable quota, timeout, network, or 5xx failures.
  - confidence: 0.86
- **[P2] Reject unresolved fallback provider ids:** `src/web-search/runtime.ts:352-357`
  - body: Fallback ids that are not present in `candidates` are silently dropped by this map/filter chain. A typo or unloaded provider can make failover look configured while no fallback will run, so the runtime should surface a clear config/diagnostic error instead of omitting it.
  - confidence: 0.84
- **[P2] Load fallback provider owners before matching:** `src/web-search/runtime.ts:313-318`
  - body: The fallback list is matched only against the initial candidate set. Current provider loading can be scoped by explicit provider ownership, so a fallback owned by another installed plugin may never be loaded and will then be silently skipped.
  - confidence: 0.8
