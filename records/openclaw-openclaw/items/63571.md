Full review comments:

- **[P2] Load fallback provider owners before building candidates:** `src/web-search/runtime.ts:353-356`
  - body: Current main scopes explicit web-search provider loading to the configured provider's owner plugin. This fallback implementation searches only `candidates`, so after rebase a fallback owned by another plugin may never be loaded and the configured fallback chain will fail fast; include fallback owner plugin ids in the load scope before resolving candidates.
  - confidence: 0.86
- **[P2] Preserve runtime config and cancellation:** `src/web-search/runtime.ts:376-387`
  - body: Current main resolves the applicable runtime config snapshot before provider execution and passes `params.signal` into `execute`. This branch's fallback path still passes `params.config` to `createTool` and calls `execute(params.args)`, so fallback searches can bypass resolved SecretRefs and ignore cancellation after rebase.
  - confidence: 0.9
- **[P2] Validate fallback provider ids:** `src/config/zod-schema.agent-runtime.ts:303`
  - body: Current main validates `tools.web.search.provider` against manifest-owned web-search provider ids, but this adds `fallbacks` as an unconstrained string array. Add plugin-aware validation for each fallback id, preserving the stale-plugin warning policy used by the primary provider.
  - confidence: 0.88
- **[P2] Document fallback query fan-out:** `src/config/types.tools.ts:490-491`
  - body: This config can send the same search query to later providers after the primary fails, but the public web-search docs are not updated. Document ordering, explicit-provider-only behavior, fallback id validation, and that fallback providers may receive the query after a failure.
  - confidence: 0.84
