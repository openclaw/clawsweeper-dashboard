- `impact:security`: The requested behavior would manage privileged macOS LaunchDaemon lifecycle operations, root-owned service files, sudo guidance, and service-user ownership.

## Summary

Keep open. Current main still treats the managed macOS Gateway service as a per-user LaunchAgent, while this issue asks for privileged system LaunchDaemon detection and lifecycle handling; the remaining work is real but needs maintainer product/security decisions before an automated fix should proceed.

## What This Changes

Add macOS Gateway lifecycle support for system-level LaunchDaemons so stop, uninstall, and status do not only target the user LaunchAgent or falsely report success while a matching system daemon remains active.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd scope model, keep the current LaunchAgent default, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level: current main routes Darwin Gateway stop/uninstall through the user LaunchAgent service and exposes no system-scope lifecycle option. I did not run a live macOS LaunchDaemon scenario in this read-only review.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and providing elevated guidance is directionally right, but maintainers should choose the CLI scope, privilege model, service user, environment, coexistence, and lifecycle semantics before implementation.
