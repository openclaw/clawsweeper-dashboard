## Summary

Keep open. Current main still lacks a managed system LaunchDaemon lifecycle path for gateway stop/uninstall, and the remaining work overlaps the open install/status design in https://github.com/openclaw/openclaw/issues/51860, so this needs maintainer product/privilege UX direction before implementation.

## What This Changes

Request gateway stop/uninstall to detect or manage system-level macOS LaunchDaemons instead of only the per-user LaunchAgent.

## Best Possible Solution

Define one documented macOS launchd user/system scope model, then make gateway install/status/stop/uninstall consistently detect default LaunchDaemons, avoid false user-scope success, and provide safe elevated-command guidance or explicit system lifecycle flags.

## Reproduction Assessment

Yes for source reproduction: current main still resolves Darwin lifecycle to $HOME/Library/LaunchAgents and gui/<uid>, and stop/uninstall register no system scope flag. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as an implementation plan: detecting the system daemon and printing actionable guidance is the right boundary, but --system versus --daemon naming, sudo behavior, UserName ownership, and status/install coordination need maintainer decision with https://github.com/openclaw/openclaw/issues/51860.
