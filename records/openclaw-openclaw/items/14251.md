## Label Justifications

- `P2`: This is a concrete macOS Gateway lifecycle improvement with real user impact, but it is platform-scoped and blocked on product/security design rather than an emergency regression.
- `impact:security`: The requested behavior would manage root-owned LaunchDaemon plist files, service-user ownership, sudo guidance, and system-domain launchctl operations.
- `impact:crash-loop`: A KeepAlive system daemon can keep restarting or holding the Gateway port after user-scope lifecycle commands report no managed LaunchAgent.

## Summary

Keep open. Current main still manages macOS Gateway lifecycle through the user LaunchAgent adapter only; the related install/status tracker remains open, and this issue is the still-needed lifecycle half for stop/uninstall behavior around system LaunchDaemons.

## What This Changes

Requests managed macOS Gateway stop/uninstall handling for system-scope LaunchDaemons so CLI results do not report success while a matching daemon remains active.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd scope model, keep the current LaunchAgent default, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level: current main routes Darwin Gateway stop/uninstall through the user LaunchAgent service and exposes no system-scope lifecycle option. I did not run a live macOS LaunchDaemon scenario in this read-only review.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and providing elevated guidance is directionally right, but maintainers should choose the CLI scope, privilege model, service user, environment, coexistence, and lifecycle semantics before implementation.
