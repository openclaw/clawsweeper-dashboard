## Summary

Keep open. Current main still manages the macOS gateway lifecycle as a per-user LaunchAgent only; system LaunchDaemons are best-effort diagnostics, not stop/uninstall targets, and the remaining work overlaps the open install/status design in https://github.com/openclaw/openclaw/issues/51860.

## What This Changes

Request gateway stop/uninstall/status lifecycle support for macOS system-level LaunchDaemons under /Library/LaunchDaemons.

## Best Possible Solution

Define one documented macOS launchd user/system scope model, then update gateway install/status/stop/uninstall to detect /Library/LaunchDaemons, avoid false user-scope success, and provide safe elevated-command guidance or explicit system lifecycle flags.

## Reproduction Assessment

Yes for source reproduction: current main resolves Darwin lifecycle to $HOME/Library/LaunchAgents and gui/<uid>, and stop/uninstall register no system scope flag. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan: detecting the system daemon and printing actionable guidance is the right boundary, but --system versus --daemon naming, sudo behavior, UserName ownership, and status/install coordination need maintainer decision with https://github.com/openclaw/openclaw/issues/51860.
