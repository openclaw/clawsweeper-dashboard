## Label Justifications

- `P2`: This is a normal-priority macOS gateway lifecycle improvement with real operational impact but limited platform scope and pending product/security decisions.
- `impact:security`: Managed LaunchDaemon support crosses elevated privilege, root-owned service metadata, execution-user, and environment-boundary decisions.
- `impact:crash-loop`: A KeepAlive system daemon outside CLI lifecycle control can keep restarting and occupying the gateway port after user-scope commands report success.

## Summary

Current main and the latest release still do not provide a macOS system LaunchDaemon target for gateway stop/uninstall; `gateway status --deep` is diagnostic only, so the reported orphan/port-conflict lifecycle gap remains. This should stay open for maintainer product and security review because the fix must share one privileged launchd scope model with https://github.com/openclaw/openclaw/issues/51860.

## What This Changes

Request that macOS gateway stop, uninstall, and status detect or manage `/Library/LaunchDaemons/ai.openclaw.gateway.plist` instead of only the user LaunchAgent path.

## Best Possible Solution

Define one explicit macOS launchd scope contract across install, status, stop, uninstall, doctor, docs, and tests, keeping user LaunchAgent as the safe default and adding only an explicit audited system LaunchDaemon path or guidance flow.

## Reproduction Assessment

Yes, at source level with high confidence: current main resolves Darwin gateway lifecycle through the LaunchAgent registry, user LaunchAgents plist path, and GUI-domain launchctl calls, while system LaunchDaemon discovery is diagnostic only. I did not run a live macOS LaunchDaemon scenario in this read-only review.

## Solution Assessment

No, the issue's isolated helper/flag sketch is only a plausible direction. The maintainable fix is one shared launchd scope model covering privilege, execution user, coexistence, lifecycle, status, doctor, docs, and regression tests.
