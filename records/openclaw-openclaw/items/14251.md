## Label Justifications

- `P2`: This is a concrete macOS Gateway lifecycle improvement with limited platform scope but real impact for users with system-level installs.
- `impact:security`: The requested lifecycle path would manage root-owned LaunchDaemon files and elevated launchctl operations.
- `impact:crash-loop`: A KeepAlive system daemon can continue restarting or holding the Gateway port after the CLI reports the user LaunchAgent as stopped.

## Summary

Keep open. Current main and the latest release still manage macOS Gateway lifecycle as a per-user LaunchAgent; system LaunchDaemon stop/uninstall remains unresolved and this issue is the canonical lifecycle-side tracker paired with the open install/status tracker at https://github.com/openclaw/openclaw/issues/51860.

## What This Changes

Request `openclaw gateway stop` and `openclaw gateway uninstall` to detect or manage macOS system LaunchDaemons under `/Library/LaunchDaemons` instead of reporting success from the user LaunchAgent path.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd model, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level: current main routes Darwin Gateway lifecycle operations to `$HOME/Library/LaunchAgents` and `gui/<uid>` with no system-scope stop/uninstall option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and giving actionable elevated guidance is directionally right, but the scope flag, privilege model, ownership, environment, and lifecycle semantics need maintainer design first.
