## Summary

Keep open. Current main still manages macOS gateway stop/uninstall through the per-user LaunchAgent path and `gui/<uid>` domain; system-level LaunchDaemon lifecycle support remains a real but product-sensitive feature tied to the broader LaunchDaemon design work.

## What This Changes

The issue requests that `openclaw gateway stop`, `uninstall`, and related status behavior detect and safely handle macOS system-level LaunchDaemons under `/Library/LaunchDaemons`.

## Best Possible Solution

Define one documented opt-in macOS LaunchDaemon service model, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes for source reproduction: current main still resolves managed macOS lifecycle to `$HOME/Library/LaunchAgents` and `gui/<uid>`, with no stop/uninstall system-scope option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and giving actionable elevated guidance is the right direction, but the exact flag, privilege, ownership, environment, and lifecycle/status semantics should be decided with the broader LaunchDaemon design thread.
