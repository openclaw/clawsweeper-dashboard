## Label Justifications

- `P2`: This is a real but macOS-bounded lifecycle gap that requires coordinated product and security design rather than emergency action.
- `impact:security`: System LaunchDaemon management crosses elevated-privilege, file-ownership, and service-identity boundaries.
- `impact:crash-loop`: A KeepAlive system daemon can remain active or restart after the user-scope CLI reports that the gateway was stopped.

## Summary

Keep open: current main and v2026.6.5 still manage the macOS gateway lifecycle as a user LaunchAgent, while system LaunchDaemon stop/uninstall remains a valid security-sensitive product decision that is not fully covered by the related install/status issue.

## What This Changes

Add macOS gateway lifecycle detection and management for system-level LaunchDaemons alongside user LaunchAgents.

## Best Possible Solution

Define one operator-controlled user/system launchd scope model, then apply it consistently to install, status, stop, and uninstall with accurate multi-scope reporting, safe privilege guidance, and real macOS upgrade and fresh-install proof.

## Reproduction Assessment

Yes at source level: current lifecycle registration and launchd helpers exclusively target the user LaunchAgent path and GUI domain, so a system-domain daemon is outside their management path. This read-only review did not execute a live macOS LaunchDaemon scenario.

## Solution Assessment

No, the issue’s detailed implementation plan should not be adopted verbatim before maintainers choose a shared scope, privilege, ownership, and multi-installation contract with https://github.com/openclaw/openclaw/issues/51860.
