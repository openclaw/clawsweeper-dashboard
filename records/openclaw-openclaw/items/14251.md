## Summary

Keep open. Current main still manages macOS gateway lifecycle through the per-user LaunchAgent path and `gui/<uid>` domain; LaunchDaemons are only discovered by deep status scans, and this issue remains the specific stop/uninstall lifecycle companion to the open install/status design in https://github.com/openclaw/openclaw/issues/51860.

## What This Changes

Request support for detecting and managing system-level macOS LaunchDaemons in `openclaw gateway stop` and `openclaw gateway uninstall`.

## Best Possible Solution

Define one documented opt-in macOS LaunchDaemon service mode that coordinates install/status/stop/uninstall, privilege guidance, `UserName`, durable environment handling, doctor behavior, docs, and regression tests with https://github.com/openclaw/openclaw/issues/51860.

## Reproduction Assessment

Yes for source reproduction: current main still resolves managed macOS lifecycle to `$HOME/Library/LaunchAgents` and `gui/<uid>`, with no stop/uninstall system-scope option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and giving actionable elevated guidance is the right direction, but the exact flag, privilege, ownership, environment, and lifecycle/status semantics should be decided with the broader LaunchDaemon design thread.
