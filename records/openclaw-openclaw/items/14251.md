## Label Justifications

- `P2`: This is a valid macOS gateway lifecycle improvement with real operational impact, but it is platform-bounded and blocked on a maintainer product/security decision rather than an emergency fix.
- `impact:security`: The requested behavior would detect or manage a privileged system LaunchDaemon and must define sudo, ownership, and service-boundary handling.
- `impact:crash-loop`: The report describes a KeepAlive-managed daemon that can keep running or respawning after CLI stop/uninstall reports success and can block the gateway port.

## Summary

Keep open: current main and the latest release still manage macOS gateway lifecycle as a user LaunchAgent only, while system LaunchDaemon stop/uninstall remains a real product and security decision tied to the open install/status work.

## What This Changes

Request support for detecting and managing system-level macOS LaunchDaemons in `openclaw gateway stop` and `openclaw gateway uninstall` so the CLI does not report success while a privileged gateway service keeps running.

## Best Possible Solution

Ship one coherent macOS launchd user/system scope model covering this lifecycle gap and https://github.com/openclaw/openclaw/issues/51860, with explicit privilege UX, docs, doctor behavior, and regression tests.

## Reproduction Assessment

Yes, at source level: current main registers no system lifecycle flag, the Darwin service adapter is LaunchAgent-only, and stop/uninstall target the user `gui/<uid>` LaunchAgent path. I did not run a live macOS LaunchDaemon repro in this read-only review.

## Solution Assessment

Unclear: detecting both scopes and avoiding false lifecycle success is the right end state, but the exact flag, sudo, ownership, TMPDIR, and status semantics need maintainer direction before implementation.
