- `impact:security`: System LaunchDaemon support crosses elevated privilege, root-owned service metadata, execution-user, and durable environment-boundary choices.

- `maturity:stable`: The issue maps to macos-gateway-host.gateway-service-lifecycle, and macOS Gateway host is currently scored M4 Stable.

## Summary

Keep open. Current main and v2026.6.10 still manage macOS gateway stop/uninstall through the user LaunchAgent path and gui/<uid> launchd domain, while system LaunchDaemon lifecycle support remains a security/product decision shared with the open install/status companion.

## What This Changes

Request support for detecting and safely stopping or uninstalling macOS system-level OpenClaw LaunchDaemons from the gateway lifecycle CLI.

## Best Possible Solution

Define one explicit macOS launchd user/system scope contract across install, status, stop, uninstall, doctor, docs, and tests, keeping the user LaunchAgent path as the safe default and adding only an audited explicit system path or guidance flow.

## Reproduction Assessment

Yes. Source inspection shows the current Darwin gateway lifecycle is wired to user LaunchAgent paths and gui/<uid> launchctl targets with no lifecycle system-scope option; I did not run a live macOS LaunchDaemon scenario in this read-only review.

## Solution Assessment

No. The issue's isolated --system helper sketch is a plausible direction, but the best fix is a shared launchd scope model covering privilege, execution user, coexistence, lifecycle, status, doctor, docs, and tests.
