## Label Justifications

- `P2`: This is a concrete macOS Gateway lifecycle improvement with real user impact but limited platform scope and product/security design still pending.
- `impact:security`: The requested lifecycle path would manage root-owned LaunchDaemon plists, service-user ownership, and elevated launchctl operations.
- `impact:crash-loop`: A KeepAlive system daemon can continue restarting or holding the Gateway port after the CLI reports the user LaunchAgent as stopped or not loaded.

## Summary

Keep open. Current main and v2026.5.19 still expose macOS Gateway lifecycle as a per-user LaunchAgent path, while this issue remains the canonical stop/uninstall side of the broader LaunchDaemon scope decision.

## What This Changes

The issue requests macOS Gateway lifecycle support for detecting and managing system-level LaunchDaemons during stop, uninstall, and status flows.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd scope model, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level: current main routes Darwin Gateway lifecycle operations to `$HOME/Library/LaunchAgents` and `gui/<uid>` with no system-scope stop/uninstall option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and giving actionable elevated guidance is directionally right, but the CLI scope, privilege model, ownership, environment, and lifecycle semantics need maintainer design first.
