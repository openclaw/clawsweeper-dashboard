## Label Justifications

- `impact:security`: System LaunchDaemon management crosses elevated privilege, plist ownership, and process-execution identity boundaries.

## Summary

Keep this issue open. Current main can discover system LaunchDaemons through deep status inspection, but macOS gateway stop and uninstall still operate only on the user LaunchAgent; the open companion https://github.com/openclaw/openclaw/issues/51860 tracks the shared install/status scope decision, while the closed https://github.com/openclaw/openclaw/issues/55208 explicitly points lifecycle handling back here.

## What This Changes

Add safe macOS system-level LaunchDaemon detection and lifecycle handling for gateway stop, uninstall, and status.

## Best Possible Solution

Define one explicit macOS launchd scope model shared by install, status, stop, uninstall, doctor, and diagnostics; retain user LaunchAgent scope as the safe default, require clear operator intent for system scope, and report both installations accurately without automatic sudo surprises.

## Reproduction Assessment

Yes at source level with high confidence. Current main manages only the user LaunchAgent plist and GUI domain, while the system LaunchDaemon scan is diagnostic and no system-scope lifecycle flag exists.

## Solution Assessment

No, the issue's isolated helper and flag sketch should not be implemented independently. The best fix is a shared scope abstraction covering the complete lifecycle and the open install/status companion, with an explicit privilege and coexistence contract.
