## Summary

Keep open. Current main still treats the managed macOS gateway service as a per-user LaunchAgent; system-level LaunchDaemon stop/uninstall support remains a real lifecycle gap and is tied to the broader privileged launchd scope design rather than being superseded by the install/status tracker.

## What This Changes

Improve `openclaw gateway stop` and `openclaw gateway uninstall` so they detect and safely handle macOS system-level LaunchDaemons in `/Library/LaunchDaemons/`.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd model, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes for source reproduction: current main still resolves managed Darwin gateway lifecycle to `$HOME/Library/LaunchAgents` and `gui/<uid>`, with no stop/uninstall system-scope option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and giving actionable elevated guidance is the right direction, but the exact scope flag, privilege model, ownership, environment, and lifecycle semantics should be decided with the broader LaunchDaemon design thread.
