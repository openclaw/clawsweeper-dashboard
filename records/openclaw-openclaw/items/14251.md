## Label Justifications

- `P2`: This is a concrete macOS Gateway lifecycle improvement with real user impact, but it is platform-scoped and blocked on product/security design rather than an emergency regression.
- `impact:security`: The requested lifecycle path would manage root-owned LaunchDaemon plists, service-user ownership, sudo guidance, and system-domain launchctl operations.
- `impact:crash-loop`: A KeepAlive system daemon can continue restarting or holding the Gateway port after user-scope lifecycle commands report the LaunchAgent as stopped or not loaded.

## Summary

Keep open. Current main still lacks a managed macOS system-scope launchd lifecycle path for Gateway stop/uninstall; `status --deep` can discover LaunchDaemons, but the active work still needs maintainer decisions around privileged service ownership and CLI scope semantics.

## What This Changes

The issue requests macOS Gateway stop/uninstall/status lifecycle support for system-level LaunchDaemons in addition to user LaunchAgents.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd scope model, keep the existing LaunchAgent default, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level: current main routes Darwin Gateway lifecycle operations to `$HOME/Library/LaunchAgents` and `gui/<uid>` with no system-scope stop/uninstall option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and giving actionable elevated guidance is directionally right, but maintainers should choose the CLI scope, privilege model, ownership, environment, coexistence, and lifecycle semantics before implementation.
