## Label Justifications

- `P2`: This is a normal-priority macOS gateway lifecycle improvement with real operational impact but limited platform scope and pending product/security decisions.
- `impact:security`: Managed LaunchDaemon support crosses elevated privilege, root-owned service metadata, execution-user, and environment-boundary decisions.
- `impact:crash-loop`: A KeepAlive system daemon outside CLI lifecycle control can keep restarting and occupying the gateway port after user-scope commands report success.

## Summary

Keep open. Current main and the latest release still manage macOS gateway lifecycle through the user LaunchAgent path and GUI launchd domain; the system LaunchDaemon stop/uninstall gap remains and should be coordinated with the open install/status tracker because the fix crosses privilege and service-ownership boundaries.

## What This Changes

Request support for detecting and safely stopping or uninstalling macOS gateway services installed as system LaunchDaemons.

## Best Possible Solution

Define one explicit macOS launchd scope contract across install, status, stop, uninstall, doctor, docs, and tests, keeping user LaunchAgent as the safe default and adding only an explicit audited system LaunchDaemon path or guidance flow.

## Reproduction Assessment

Yes, at source level with high confidence: current main resolves Darwin gateway lifecycle through the LaunchAgent registry, user LaunchAgents plist path, and GUI-domain launchctl calls. I did not run a live macOS LaunchDaemon scenario in this read-only review.

## Solution Assessment

No, the issue's isolated helper/flag sketch is only a plausible direction. The maintainable fix is one shared launchd scope model covering privilege, execution user, coexistence, lifecycle, status, doctor, docs, and regression tests.
