## Summary

Keep open. Current main still manages the macOS gateway lifecycle as a per-user LaunchAgent, while this request needs a supported system LaunchDaemon lifecycle contract for stop/uninstall/status and has unresolved privileged-operation design and security details.

## What This Changes

Request system-level macOS LaunchDaemon detection and lifecycle support so gateway stop/uninstall/status do not report user-scope success while a matching system daemon is still running.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd model, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, for source-level reproduction: current main still resolves the managed Darwin gateway lifecycle to $HOME/Library/LaunchAgents and gui/<uid>, with no stop/uninstall system-scope option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and giving actionable elevated guidance is the right direction, but the scope flag, privilege model, ownership, environment, and lifecycle semantics need maintainer design first.
