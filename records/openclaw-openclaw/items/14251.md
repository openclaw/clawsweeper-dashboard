## Summary

Keep open. Current main still manages the macOS gateway lifecycle through the user LaunchAgent path and GUI launchd domain; the newer `status --deep` path is diagnostic only and does not give stop/uninstall a system scope for `/Library/LaunchDaemons`. Prior maintainer/bot review also kept the item active alongside the broader LaunchDaemon design in #51860.

## What This Changes

Request macOS gateway lifecycle commands to detect system-level LaunchDaemon installs, avoid misleading success when they remain running, and provide a safe system-scope stop/uninstall path or guidance.

## Best Possible Solution

Define a single macOS launchd scope model, then update lifecycle, status, docs, and regression tests so user LaunchAgents and system LaunchDaemons are detected accurately, stop/uninstall never claim user-scope success while a matching system daemon remains active, and privileged actions require explicit operator intent.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `ai.openclaw.gateway` loaded in the `system` launchd domain and no matching user LaunchAgent, current main checks `gui/<uid>/ai.openclaw.gateway` and `~/Library/LaunchAgents`, while the CLI has no lifecycle scope flag.

## Solution Assessment

Unclear. The requested direction is valid, but the narrowest maintainable fix should be chosen after maintainers settle the shared LaunchDaemon install/status/lifecycle contract with #51860 instead of adding ad hoc stop/uninstall behavior alone.
