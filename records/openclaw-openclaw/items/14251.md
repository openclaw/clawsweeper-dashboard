## Summary

Keep open. Current main and the latest release still manage the macOS gateway lifecycle through the user LaunchAgent path and GUI launchd domain; the system LaunchDaemon stop/uninstall gap remains and should be coordinated with https://github.com/openclaw/openclaw/issues/51860 because the fix crosses privilege and service-ownership boundaries.

## What This Changes

The issue requests macOS gateway lifecycle support for detecting and managing system-level LaunchDaemons during stop, uninstall, and status operations.

## Best Possible Solution

Define one explicit macOS launchd user/system scope contract across install, status, stop, uninstall, doctor, docs, and tests, keeping user LaunchAgent as the safe default and adding only an explicit audited system LaunchDaemon path or guidance flow.

## Reproduction Assessment

Yes, at source level with high confidence: current main resolves Darwin gateway lifecycle through the LaunchAgent registry, user LaunchAgents plist path, and GUI-domain launchctl calls, with no lifecycle system-scope option. I did not run a live macOS LaunchDaemon scenario in this read-only review.

## Solution Assessment

No, the issue's isolated helper/flag sketch is only a plausible direction. The maintainable fix is one shared launchd scope model covering privilege, execution user, coexistence, lifecycle, status, doctor, docs, and regression tests.
