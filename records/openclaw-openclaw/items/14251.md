## Summary

Keep open. Current main still manages the macOS Gateway as a per-user LaunchAgent, while the requested system LaunchDaemon stop/uninstall lifecycle remains missing and crosses privileged launchctl/root-owned plist behavior.

## What This Changes

The issue requests macOS Gateway stop, uninstall, and status handling for system-level LaunchDaemons in addition to user LaunchAgents.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd model, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, for source-level reproduction: current main still resolves managed Darwin Gateway lifecycle operations to $HOME/Library/LaunchAgents and gui/<uid>, with no stop/uninstall system-scope option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and giving actionable elevated guidance is directionally right, but the scope flag, privilege model, ownership, environment, and lifecycle semantics need maintainer design first.
