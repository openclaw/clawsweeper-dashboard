## Summary

Keep open. Current main still manages macOS Gateway lifecycle as a per-user LaunchAgent; system LaunchDaemon stop/uninstall support remains unresolved and should stay coordinated with the open install/status tracker at https://github.com/openclaw/openclaw/issues/51860 and the unmerged LaunchDaemon PR history at https://github.com/openclaw/openclaw/pull/33489.

## What This Changes

Requests macOS Gateway lifecycle commands detect and handle `/Library/LaunchDaemons/ai.openclaw.gateway.plist` with safe privileged guidance or an explicit system-scope option.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd model, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level: current main still routes managed Darwin Gateway lifecycle operations to `$HOME/Library/LaunchAgents` and `gui/<uid>`, with no stop/uninstall system-scope option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and giving actionable elevated guidance is directionally right, but the scope flag, privilege model, ownership, environment, and lifecycle semantics need maintainer design first.
