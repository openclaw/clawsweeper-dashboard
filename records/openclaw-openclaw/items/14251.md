## Summary

Keep open. Current main still manages macOS gateway lifecycle operations through the per-user LaunchAgent path and `gui/<uid>` domain; related discussion keeps install/status design in https://github.com/openclaw/openclaw/issues/51860 and treats this issue as the lifecycle tracker for stop/uninstall behavior.

## What This Changes

Request support for detecting and managing system-level macOS LaunchDaemons in gateway stop/uninstall/status lifecycle commands.

## Best Possible Solution

Define one documented opt-in macOS LaunchDaemon scope covering install/status/stop/uninstall, privilege guidance, `UserName`, durable environment handling, doctor behavior, docs, and regression tests coordinated with https://github.com/openclaw/openclaw/issues/51860.

## Reproduction Assessment

Yes for source reproduction: current main still resolves managed macOS lifecycle to `$HOME/Library/LaunchAgents` and `gui/<uid>`, with no stop/uninstall system-scope option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and providing actionable elevated guidance is the right direction, but the exact flag, privilege, ownership, and status/lifecycle semantics should be decided with the broader LaunchDaemon design thread.
