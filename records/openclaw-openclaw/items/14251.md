## Summary

Keep open. Current main still manages the macOS gateway lifecycle as a per-user LaunchAgent; system LaunchDaemons are only surfaced by deep diagnostics, not stop/uninstall targets, and the remaining lifecycle work needs the same privileged scope design as https://github.com/openclaw/openclaw/issues/51860.

## What This Changes

The issue requests macOS gateway stop, uninstall, and status behavior that detects or manages system-level LaunchDaemons alongside user LaunchAgents.

## Best Possible Solution

Define one documented macOS launchd user/system scope model, then update install/status/stop/uninstall to detect `/Library/LaunchDaemons`, avoid false user-scope success, and provide safe elevated-command guidance or explicit system lifecycle flags.

## Reproduction Assessment

Yes for source reproduction: current main resolves managed macOS lifecycle to `$HOME/Library/LaunchAgents` and `gui/<uid>`, with no system-scope stop/uninstall option. I did not run a live macOS LaunchDaemon scenario.

## Solution Assessment

Unclear as a final implementation plan. Detecting a system daemon and printing actionable guidance is the right boundary, but `--system` versus `--daemon`, sudo behavior, and install/status coordination should be decided with https://github.com/openclaw/openclaw/issues/51860.
