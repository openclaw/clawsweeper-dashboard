## Label Justifications

- `P2`: This is a concrete macOS Gateway lifecycle gap with real user impact, but it is platform-scoped and blocked on product/security design rather than an emergency regression.
- `impact:security`: The requested behavior would manage privileged macOS LaunchDaemon lifecycle operations, root-owned service files, sudo guidance, and service-user ownership.
- `impact:crash-loop`: A KeepAlive system daemon can keep restarting or holding the Gateway port after user-scope lifecycle commands report no managed LaunchAgent.

## Summary

Keep open: current main and the latest release still target the macOS Gateway lifecycle through the user LaunchAgent path, while system LaunchDaemon stop/uninstall support remains a product and security design decision.

## What This Changes

Request lifecycle support for detecting and managing macOS system LaunchDaemon Gateway installs so stop, uninstall, and status do not misreport user-scope success while a system daemon remains active.

## Best Possible Solution

Define one documented opt-in macOS user/system launchd scope model that keeps LaunchAgent as the default, then align install, status, stop, uninstall, doctor guidance, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-level: current main routes Darwin Gateway stop/uninstall through the user LaunchAgent service and exposes no system-scope lifecycle option. I did not run a live macOS LaunchDaemon scenario in this read-only review.

## Solution Assessment

Unclear as a final implementation plan. Detecting system daemons and providing elevated guidance is directionally right, but maintainers should choose the CLI scope, privilege model, service user, environment, coexistence, and lifecycle semantics before implementation.
