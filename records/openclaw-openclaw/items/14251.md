## Label Justifications

- `P2`: This is a normal-priority macOS gateway lifecycle improvement with real operational impact but limited platform scope and a product/security decision pending.
- `impact:security`: Managed LaunchDaemon support crosses elevated privilege, root-owned service metadata, execution-user, and environment-boundary decisions.
- `impact:crash-loop`: A KeepAlive system daemon outside CLI lifecycle control can keep restarting and occupying the gateway port after user-scope commands report success.

## Summary

Keep open. Current main still manages macOS gateway lifecycle through the user LaunchAgent path only; `gateway status --deep` can discover system LaunchDaemons diagnostically, but stop/uninstall cannot target them and the remaining fix needs a shared privileged launchd-scope decision with https://github.com/openclaw/openclaw/issues/51860.

## What This Changes

Requests macOS gateway stop/uninstall detection and handling for system-level LaunchDaemons, including accurate reporting and optional system-scope lifecycle flags.

## Best Possible Solution

Define one explicit macOS launchd scope contract shared by install, status, stop, uninstall, doctor, docs, and tests, with user LaunchAgent as the safe default and an explicit audited system LaunchDaemon path or guidance flow.

## Reproduction Assessment

Yes, at source level with high confidence: current main resolves Darwin gateway lifecycle through the LaunchAgent registry, user LaunchAgents plist path, and GUI-domain launchctl calls, while system LaunchDaemon discovery is diagnostic only.

## Solution Assessment

No, the issue's isolated helper/flag sketch is only a plausible direction. The maintainable fix is one shared launchd scope model covering privilege, execution user, coexistence, lifecycle, status, doctor, docs, and regression tests.
