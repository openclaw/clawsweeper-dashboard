## Summary

Keep open: current main and the latest release still enqueue accepted Telegram reactions as passive system-event context, while reaction-only wake behavior, custom-emoji reaction keys, and product-level reaction semantics remain unmerged. The issue is also paired with open implementation PRs that use closing syntax, so cleanup closure would be premature.

## What This Changes

The issue requests first-class Telegram reaction input support: reaction-only agent wake-up, standard and custom emoji normalization, configurable reaction semantics, and observability.

## Best Possible Solution

Adopt or supersede the linked PRs with one default-safe reaction contract that normalizes standard and custom emoji, wakes only authorized opt-in reaction inputs after successful enqueue, documents the config, and includes focused tests plus live Telegram proof.

## Reproduction Assessment

Yes for the wake-trigger subset: source inspection shows an authorized Telegram message_reaction with an added standard emoji reaches enqueueSystemEvent and then returns without a wake call. The broader custom-emoji and per-emoji semantics are a feature/API gap rather than a fully defined existing bug contract.

## Solution Assessment

Unclear until maintainers choose the API. The safest path is a default-safe opt-in wake and stable reaction-key contract, not an unconditional wake for every accepted reaction.
