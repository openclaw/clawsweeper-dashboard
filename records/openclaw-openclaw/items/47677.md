## Summary

Keep this open. Current main still handles Telegram reactions as notification/context events rather than first-class wake or execution triggers, and the issue is paired with open PR #47687 plus the narrower open bug #64752.

## What This Changes

The issue requests first-class Telegram reaction triggers that can wake the agent, normalize standard and custom emoji, support configurable operator semantics, and provide useful reaction-handling observability.

## Best Possible Solution

Review #47687 against #64752 and either land it or replace it with a narrower Telegram-plugin implementation that normalizes standard and custom emoji, maps configured reaction actions, wakes only authorized mapped reactions, logs ignored/rejected/executed decisions, and updates focused tests plus public Telegram docs.

## Reproduction Assessment

Yes. The high-confidence reproduction path is an authorized Telegram DM or group with reactionNotifications enabled: adding a reaction reaches the handler and gets enqueued as a system event, but current main has no wake call or semantic dispatch for that event.

## Solution Assessment

Yes in direction, but it needs review. An opt-in mapped semantics implementation is safer than a broad always-wake switch because it preserves existing notification behavior while adding explicit authorization, action mapping, and observability.
