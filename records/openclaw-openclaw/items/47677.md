## Summary

Keep open: current main and v2026.6.10 still treat Telegram reactions as passive queued standard-emoji system events, while wake behavior, custom-emoji normalization, and product-level semantics remain unmerged product/API work.

## What This Changes

Request first-class Telegram reaction handling for reaction-only wake, standard/custom emoji normalization, configurable semantics, observability, and upgrade-safe behavior.

## Best Possible Solution

Land or close a maintainer-approved opt-in notification-wake design that keeps default queue behavior, normalizes standard/custom emoji reaction keys, and documents the supported semantics boundary.

## Reproduction Assessment

Yes. Source inspection shows accepted Telegram reactions are enqueued as system events without a wake path, and the issue comments include versioned Telegram DM observations of the same behavior.

## Solution Assessment

Unclear until maintainers choose the public API. The safest shape appears to be an opt-in shared notification-wake policy with default queue behavior, plus reaction-key normalization and focused Telegram proof.
