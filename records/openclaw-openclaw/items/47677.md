## Label Justifications

- `P2`: This is a real Telegram workflow gap with a source-reproducible wake subset, but it has limited blast radius and needs product/API review rather than emergency handling.
- `impact:message-loss`: Reaction-only operator inputs can be received and queued without being processed by the agent until another wake occurs, which is effectively lost or delayed channel input.

## Summary

Keep open: current main still only records Telegram reactions as passive notification/context events, and the issue is paired with an open closing PR plus a broader open reaction-trigger contract discussion. The wake-trigger subset is source-reproducible, but first-class per-emoji/custom-emoji semantics require maintainer product/API selection before this can close.

## What This Changes

Request first-class Telegram reaction inputs that can wake agents, normalize standard and custom emoji, support operator semantics, and provide useful reaction observability.

## Best Possible Solution

Adopt or supersede the linked Telegram reaction work with an explicit opt-in contract that normalizes standard/custom emoji, wakes only mapped or otherwise authorized reactions, records ignore/reject/execute outcomes, and updates schema, docs, tests, and live Telegram proof together.

## Reproduction Assessment

Yes for the wake-trigger subset: an accepted Telegram message_reaction reaches enqueueSystemEvent and then returns without a processMessage or heartbeat wake. The broader per-emoji/custom-emoji semantics are new feature behavior, so no existing full contract reproduces that portion as a bug.

## Solution Assessment

Unclear until maintainers choose the public API. The requested direction is coherent, but the maintainable solution needs to decide whether the open narrow Telegram wake PR, the broader Telegram reactionSemantics PR, or the cross-channel opt-in reaction-trigger contract is the right end state.
