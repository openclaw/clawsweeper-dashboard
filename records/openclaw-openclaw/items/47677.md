## Summary

Keep open: current main and the latest release still only enqueue accepted Telegram reaction events as passive system context, while wake policy, stable custom-emoji keys, and first-class reaction semantics remain unmerged product work. The open shared notification-wake PR and the open fork PR are relevant implementation paths, but neither is merged into the target repository.

## What This Changes

Request first-class Telegram reaction triggers that can wake the agent, normalize standard and custom emoji, expose operator semantics, and provide observable upgrade-safe behavior.

## Best Possible Solution

Keep this issue open until maintainers land or reject an owner-approved reaction wake/semantics design, preferably through the shared notification-wake path with explicit safe defaults and stable reaction keys.

## Reproduction Assessment

Yes from source, but not from a live run: current main's Telegram `message_reaction` path enqueues standard emoji reaction events and does not request a heartbeat wake or normalize custom emoji. The issue comments provide matching release-level observations on Telegram DMs.

## Solution Assessment

Unclear as a final design: opt-in shared notification wake looks like the right narrow base, but per-emoji execution semantics need maintainer policy choices before a new autonomous fix PR is safe.
