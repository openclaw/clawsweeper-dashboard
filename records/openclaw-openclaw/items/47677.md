## Summary

Keep open: current main and v2026.6.8 still treat Telegram reactions as passive queued system events, ignore Telegram custom emoji in the reaction handler, and do not expose a wake or per-reaction semantics policy. The linked implementation PR is open and conflicting, and the broader shared reaction-wake API remains an open product decision.

## What This Changes

The issue requests first-class Telegram reaction inputs that can wake the agent, normalize standard and custom emoji reactions, support configurable reaction semantics, and expose debug visibility.

## Best Possible Solution

Keep this issue open and land one maintainer-approved, default-preserving Telegram reaction wake/semantics design, aligned where practical with the broader shared reaction-wake API, with stable standard/custom reaction keys, docs, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes, at source level: current main accepts Telegram message_reaction updates, queues standard emoji reaction system events, and exits without a heartbeat or turn-start call. I did not run a live Telegram account in this read-only review, but the issue discussion includes repeated live reproductions on released versions.

## Solution Assessment

Unclear until maintainers choose the API shape. A default-preserving opt-in Telegram wake/semantics policy is safer than unconditional reaction-triggered execution, and the Telegram-specific design should be reconciled with the broader cross-channel request.
