## Label Justifications

- `P2`: This blocks useful Telegram reaction-only workflows with clear source evidence, but it is scoped to reaction input behavior and requires product/API review rather than emergency handling.
- `impact:message-loss`: Accepted reaction-only operator inputs can be queued without waking an agent turn, effectively suppressing actionable channel input until another wake occurs.

## Summary

Keep open: current main and the latest release still treat accepted Telegram reactions as queued notification context, while reaction-only wake behavior, stable custom-emoji keys, and product-level reaction semantics remain unmerged. The issue is also paired with open implementation PRs that use closing syntax, so the safe path is maintainer review of the linked contract rather than cleanup closure.

## What This Changes

The issue requests first-class Telegram reaction input semantics, including reliable reaction-only wake behavior, standard and custom emoji normalization, operator-level reaction meanings, and observability.

## Best Possible Solution

Adopt or supersede the linked PRs with one default-safe reaction contract that normalizes standard and custom emoji, wakes only authorized opt-in reaction inputs after successful enqueue, documents the config, and includes focused tests plus live Telegram proof.

## Reproduction Assessment

Yes for the wake-trigger subset: source inspection shows an authorized Telegram message_reaction with an added standard emoji reaches enqueueSystemEvent and then returns without any wake call. The broader custom-emoji and per-emoji semantics are a feature/API gap rather than a fully defined existing bug contract.

## Solution Assessment

Unclear until maintainers choose the API. The requested direction is coherent, but the safest implementation should preserve queue-only default behavior while adding an explicit opt-in wake and stable reaction-key semantics contract.
