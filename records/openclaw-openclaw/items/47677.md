## Summary

Keep open: current main and latest release still treat Telegram reactions as passive queued system events, while wake behavior, custom-emoji normalization, and product-level reaction semantics remain unmerged product/API work. The issue overlaps the broader cross-channel tracker and open shared notification-wake PR, but it has Telegram-specific scope that should not be closed before maintainers choose and land the canonical design.

## What This Changes

The issue requests first-class Telegram reaction handling so configured reactions can wake the agent, support standard and custom emoji semantics, and provide observable operator workflows.

## Best Possible Solution

Land one maintainer-approved, default-safe Telegram or shared notification-wake design with stable standard/custom reaction keys, focused tests, docs, upgrade-safe config semantics, and live Telegram proof.

## Reproduction Assessment

Yes, at source level: current main accepts Telegram message_reaction updates, queues added standard emoji as system events, and exits without a heartbeat or turn-start call. I did not run a live Telegram account in this read-only review, but issue comments report repeated live reproductions on released versions.

## Solution Assessment

Unclear until maintainers choose the public API shape. A default-off or queue-by-default wake policy after existing Telegram filters looks safer than unconditional reaction execution or a hardcoded universal emoji map, and it should be reconciled with the broader cross-channel tracker.
