## Summary

Keep open: current main and v2026.5.22 still treat accepted Telegram reactions as queued system events, while reaction-only wake behavior, custom-emoji reaction keys, and a configurable reaction semantics contract remain unresolved. Open implementation PRs already cover this area, so cleanup closure would race the maintainer product/API decision.

## What This Changes

Request first-class Telegram reaction handling for reaction-only agent wake-up, standard and custom emoji normalization, configurable operator semantics, observability, and upgrade-safe behavior.

## Best Possible Solution

Adopt or supersede the linked PRs with one default-safe reaction contract that normalizes standard/custom emoji, wakes only authorized opt-in reactions after successful enqueue, documents the config, and includes focused tests plus live Telegram proof.

## Reproduction Assessment

Yes for the wake-trigger subset: source inspection shows an authorized Telegram `message_reaction` with an added emoji reaches `enqueueSystemEvent` and then returns without any wake call. Custom-emoji normalization and per-emoji semantics are feature/API gaps rather than a fully defined existing bug contract.

## Solution Assessment

Unclear until maintainers choose the API. The safest path is a default-safe opt-in wake and stable reaction-key contract rather than unconditional reaction-triggered execution.
