## Summary

Keep open: current main and the latest shipped release still expose Telegram reactions as notification/context events, not first-class wake or execution triggers. The narrow wake gap is already tracked by an open bug and open PR, while the broader standard/custom emoji semantics requested here still needs maintainer product/API review.

## What This Changes

The issue requests first-class Telegram reaction input, including reaction-only wake, standard/custom emoji normalization, configurable per-reaction semantics, and observability.

## Best Possible Solution

Adopt or supersede the linked Telegram reaction work with a Telegram-owned contract that normalizes standard/custom emoji, gates mapped wake or execution actions behind explicit policy, records ignore/reject/execute outcomes, and updates schema, docs, tests, and live Telegram proof together.

## Reproduction Assessment

Yes for the wake-trigger subset: current main receives authorized Telegram message_reaction updates and enqueues system events, then returns without a wake path. The broader per-emoji execution semantics are new feature behavior, so there is no existing full failing contract to reproduce.

## Solution Assessment

Unclear: the requested direction is coherent, but maintainers need to choose between the narrow wake behavior in https://github.com/openclaw/openclaw/pull/75899 and a broader reactionSemantics API before this should be implemented or closed.
