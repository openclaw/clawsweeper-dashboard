## Summary

Keep open. Current main still only records authorized Telegram reactions as queued context events, while the requested wake triggers, custom-emoji normalization, per-reaction semantics, and operator-action behavior remain unmerged; the issue is also paired with active implementation PRs that should not be pre-closed.

## What This Changes

The issue requests native Telegram reaction handling that can wake agents, normalize standard and custom emoji reactions, support per-reaction semantics, and expose useful observability.

## Best Possible Solution

A Telegram-owned implementation should expose explicitly authorized reaction semantics, normalize standard and custom emoji, wake only configured actions, log queue/ignore/reject/execute decisions, and keep docs, schema, generated metadata, and focused tests aligned.

## Reproduction Assessment

Yes for the wake-trigger subset: current main receives message_reaction updates and enqueues system events, but source inspection shows no wake or dispatch after enqueue. The broader per-emoji execution semantics are a new feature, so there is no failing existing contract to reproduce for that part.

## Solution Assessment

Yes, keeping this issue open is the right cleanup outcome because an open same-author PR and an open narrower wake-fix PR are still implementation candidates. The maintainable fix belongs in the Telegram plugin/config surface with authorization, observability, and tests rather than closing the feature before those PRs are resolved.
