## Summary

Keep open: current main still treats Telegram reactions as notification/context events, not first-class wake or execution triggers. The narrow wake gap is already tracked by open Telegram bug/fix work, while the broader standard/custom emoji semantics still needs maintainer product/API review.

## What This Changes

Request first-class Telegram reaction handling that can wake agents, normalize standard and custom emoji, support per-reaction operator semantics, and expose useful debugging signals.

## Best Possible Solution

Adopt or supersede the linked Telegram reaction work with an explicit, default-off contract that normalizes standard/custom emoji, wakes only mapped/authorized reactions, records ignore/reject/execute outcomes, and updates schema, docs, tests, and live Telegram proof together.

## Reproduction Assessment

Yes for the wake-trigger subset: an accepted Telegram message_reaction reaches enqueueSystemEvent and then the handler returns without processMessage or heartbeat wake. The broader per-emoji/custom-emoji semantics are new feature behavior, so no existing full contract fails for that portion.

## Solution Assessment

Unclear until maintainers choose the public API. The requested direction is coherent, but the maintainable fix needs to decide whether https://github.com/openclaw/openclaw/pull/75899, https://github.com/BingqingLyu/openclaw/pull/803, or the cross-channel contract in https://github.com/openclaw/openclaw/issues/17840 is the right path.
