## Summary

Keep open: current main and the latest release still treat accepted Telegram reactions as passive standard-emoji system events, with no reaction wake/turn policy, no Telegram custom-emoji reaction normalization in the handler, and no first-class reaction semantics config. The issue is also paired with open implementation candidates and a broader product/API tracker, so cleanup should not close it before maintainers choose and land the canonical design.

## What This Changes

Request first-class Telegram reaction input covering reaction-only agent wake, standard/custom emoji normalization, per-reaction semantics, and observability.

## Best Possible Solution

Keep this open until a maintainer-approved, default-safe Telegram or shared notification-wake policy lands with stable standard/custom reaction keys, docs, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes at source level: current main accepts Telegram message_reaction updates, queues standard emoji reaction system events, and exits without a heartbeat or turn-start call. I did not run a live Telegram account in this read-only review, but the issue discussion includes repeated live repros on released versions.

## Solution Assessment

Unclear until maintainers choose the public API shape. A default-off wake policy after existing Telegram filters looks safer than unconditional reaction execution or a hardcoded universal emoji map, and it should be reconciled with the broader cross-channel tracker.
