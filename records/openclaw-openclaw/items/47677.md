## Label Justifications

- `P2`: This is a normal-priority Telegram workflow capability with repeated user friction, but not a core outage, crash loop, or security emergency.
- `impact:message-loss`: Accepted reaction-only input can be queued as context without promptly reaching the routed agent turn, so the operator signal is effectively missed until another wake occurs.

## Summary

Keep open: current main and v2026.6.8 still treat Telegram reactions as passive notification/context events, with no wake policy, custom-emoji normalization, or product-level reaction semantics. The issue also has open, unmerged implementation candidates, so cleanup should not close it before maintainers choose and land the canonical design.

## What This Changes

Request first-class Telegram reaction triggers so standard and custom emoji reactions can wake or control the agent with explicit semantics and observability.

## Best Possible Solution

Keep this issue open until maintainers choose and land a default-safe Telegram or shared notification-wake design with stable standard/custom reaction keys, focused tests, docs, upgrade-safe config semantics, and live Telegram proof.

## Reproduction Assessment

Yes at source level: current main accepts Telegram `message_reaction` updates, queues added standard emoji as system events, and exits without a heartbeat or turn-start call. I did not run a live Telegram account in this read-only review, but issue comments report repeated live reproductions on released versions.

## Solution Assessment

Unclear until maintainers choose the public API shape. A default-off or queue-by-default wake policy after existing Telegram filters looks safer than unconditional reaction execution or a hardcoded universal emoji map, and it should be reconciled with the broader cross-channel tracker.
