## Summary

Keep open: current main and latest release still only enqueue accepted Telegram emoji reaction system events, ignore Telegram custom emoji reactions in the handler, and have no opt-in wake or reaction-semantics config. https://github.com/openclaw/openclaw/issues/64752 was closed as a duplicate pointing here, https://github.com/openclaw/openclaw/issues/17840 only covers the broader cross-channel contract, and https://github.com/BingqingLyu/openclaw/pull/803 is an open implementation candidate that needs maintainer review rather than issue cleanup.

## What This Changes

Request first-class Telegram reaction input semantics so accepted standard and custom emoji reactions can wake agent turns, carry configured meanings, and be observable for debugging.

## Best Possible Solution

Keep this open and review a default-preserving Telegram reaction wake/semantics design, ideally aligned with https://github.com/openclaw/openclaw/issues/17840, with stable standard/custom reaction keys, wake only after existing filters, docs, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes at source level: current main accepts Telegram `message_reaction` updates, queues standard emoji reaction system events, and exits without a heartbeat or turn-start call. I did not run a live Telegram account in this read-only review, but the issue discussion includes repeated live repros on released versions.

## Solution Assessment

Unclear until maintainers choose the API shape. A default-preserving opt-in Telegram wake/semantics policy is safer than unconditional reaction-triggered execution, and any Telegram-specific design should be reconciled with the broader cross-channel request.
