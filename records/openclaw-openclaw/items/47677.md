## Summary

Keep open: current main still treats Telegram reactions as notification-only queued context, filters out Telegram custom emoji reactions, and has no first-class reaction semantics or wake-trigger config; related narrow and broad implementation paths still need maintainer resolution.

## What This Changes

Request first-class Telegram reaction inputs that can wake the agent, normalize standard and custom emoji, map per-reaction operator semantics, and provide observable routing and policy decisions.

## Best Possible Solution

Adopt or supersede the open Telegram reaction work with a Telegram-owned contract that normalizes standard/custom emoji, gates mapped wake or execution actions behind explicit policy, records ignore/reject/execute decisions, and updates schema, docs, and focused tests together.

## Reproduction Assessment

Yes for the wake-trigger subset: current main receives authorized message_reaction updates and enqueues system events, then returns without a wake path. The broader per-emoji execution semantics are new feature behavior, so there is no existing full failing contract to reproduce.

## Solution Assessment

Unclear: the requested direction is valid, but maintainers need to choose between the narrow wake fix and the broader reactionSemantics API before this should be closed or implemented in core.
