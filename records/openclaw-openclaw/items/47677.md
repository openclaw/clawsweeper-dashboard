## Summary

Current main receives, authorizes, and queues Telegram reactions but does not provide reaction-triggered agent wake-up, custom-emoji normalization, or product-level semantics; this issue should remain open because an open pull request explicitly claims the work and still needs maintainer product review.

## What This Changes

Add first-class Telegram reaction inputs that can wake agents, normalize standard and custom emoji, support operator-defined meanings, and expose routing and policy decisions for debugging.

## Best Possible Solution

Keep Telegram responsible for trusted reaction ingestion and stable standard/custom emoji keys, use an explicit queue-by-default generic wake policy, and let agent-level preferences or typed product actions define meaning rather than hardcoding execution semantics in the transport.

## Reproduction Assessment

Yes, at source level with high confidence: current main accepts and queues standard emoji reaction events but has no reaction-specific wake path, and multiple release reports describe the same reaction-without-turn behavior. A fresh live current-main Telegram run was not performed in this read-only review.

## Solution Assessment

No, the broad per-emoji transport configuration is not yet proven to be the best permanent shape. The safer maintainable direction is stable reaction identity plus an opt-in generic wake policy, with semantic meaning owned by agent policy or typed product actions.
