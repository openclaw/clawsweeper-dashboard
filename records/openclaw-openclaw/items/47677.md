## Summary

Keep open. Current main still treats Telegram reactions as authorization-gated notification/context events, not first-class wake or execution triggers, and the supplied GitHub context shows open PR #47687 as the active implementation plus open bug #64752 for the narrower wake gap.

## What This Changes

The issue requests first-class Telegram reaction triggers for wake-up, standard and custom emoji normalization, configurable reaction semantics, and debugging visibility.

## Best Possible Solution

Review #47687 against #64752 and either land it or replace it with a narrow Telegram-plugin implementation that normalizes standard/custom emoji, wakes only authorized mapped reactions, logs ignored/rejected/executed decisions, and updates focused tests plus the public Telegram docs.

## Reproduction Assessment

Yes for the wake-trigger subset by source inspection: current main receives `message_reaction` updates and enqueues system events, but the handler has no heartbeat wake or semantic dispatch path. The broader per-emoji semantics request is a new feature rather than a failing existing contract.

## Solution Assessment

Yes, keeping this issue open behind the active PR and narrower bug is the best current path. The maintainable implementation belongs in the Telegram plugin/config surface with explicit authorization and observability, not as a broad always-wake switch.
