## Summary

Keep open: current main and the latest release still handle accepted Telegram reactions as passive queued system events, while reaction wake policy, custom-emoji normalization, and first-class operator semantics remain unmerged work. This issue is also paired with open implementation PRs that reference it, so the cleanup-safe path is maintainer review of the linked API/product direction rather than closing it now.

## What This Changes

The issue requests first-class Telegram reaction controls that can wake agent sessions, normalize standard and custom emoji, attach operator semantics, and expose debuggable upgrade-safe behavior.

## Best Possible Solution

Adopt or supersede the linked PRs with a default-safe reaction contract that normalizes standard and custom emoji, wakes only authorized opt-in reaction inputs after successful enqueue, and ships matching schema, docs, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes for the wake-trigger subset: source inspection shows an authorized Telegram message_reaction with an added standard emoji reaches enqueueSystemEvent and then returns without a heartbeat or agent-turn path. The broader per-emoji/custom-emoji semantics are a feature/API gap rather than a fully defined existing bug contract.

## Solution Assessment

Unclear until maintainers choose the API. The requested direction is coherent, but the safest implementation should preserve queue-only default behavior while adding an explicit opt-in wake and reaction-key semantics contract.
