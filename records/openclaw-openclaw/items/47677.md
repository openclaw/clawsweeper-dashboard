## Summary

Keep open: current main and v2026.6.6 still treat accepted Telegram reactions as passive system events, while reaction-triggered wake, stable custom-emoji keys, and first-class per-reaction semantics remain unmerged. The relevant implementation candidates are still open/conflicting or closed unmerged, so this issue is still necessary.

## What This Changes

The issue requests first-class Telegram reaction-trigger support so configured reactions can wake agents, normalize standard and custom emoji, express operator semantics, and provide debugging visibility.

## Best Possible Solution

Keep this open until maintainers land or intentionally reject a default-safe reaction wake policy with stable Telegram reaction keys and a clear boundary for per-reaction operator semantics.

## Reproduction Assessment

Yes at source level: current main registers Telegram reaction updates and enqueues system events, but the inspected handler has no heartbeat wake or semantic action dispatch. Issue comments also report the same behavior on released live Telegram DM setups.

## Solution Assessment

Unclear as a final product design. A shared, default-safe notification wake policy looks like the right base, but custom-emoji keys and per-emoji execution semantics need maintainer policy before a new autonomous fix is safe.
