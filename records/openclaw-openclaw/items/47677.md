## Summary

Keep open. Current main still treats Telegram reactions as authorized notification/context events rather than configured wake, custom-emoji, or semantic execution triggers, and open PR #47687 explicitly targets this issue while open bug #64752 tracks the narrower wake-only gap.

## What This Changes

The issue requests first-class Telegram reaction semantics so reaction-only inputs can wake agents, support standard and custom emoji, map per-emoji operator meanings, and expose debugging signals.

## Best Possible Solution

Land or replace the Telegram-plugin implementation with explicit reaction semantics config, standard/custom emoji normalization, authorized mapped wake behavior, observability for ignored/rejected/executed reactions, and focused docs/tests.

## Reproduction Assessment

Yes for the narrower wake-trigger subset by source inspection: current main receives message_reaction updates and enqueues system events, but the Telegram handler has no wake or semantic dispatch path. The broader per-emoji operator semantics request is a new feature/config surface rather than a failing existing contract.

## Solution Assessment

Yes, keeping this issue open behind the active PR and narrower bug is the best current path. The maintainable fix belongs in the Telegram plugin/config surface with explicit authorization, normalization, and observability rather than an unconditional wake switch.
