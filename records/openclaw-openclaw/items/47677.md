## Label Justifications

- `P2`: This is a real Telegram workflow gap with a source-reproducible wake subset, but it has limited blast radius and requires product/API review rather than emergency handling.
- `impact:message-loss`: Reaction-only operator inputs can be received and queued without waking the agent, effectively delaying or losing actionable channel input until another wake occurs.

## Summary

Keep open: current main still treats accepted Telegram reactions as queued notification/context events, while wake behavior, custom-emoji normalization, and reaction semantics remain unmerged product/API work. The issue is also paired with open implementation PRs, including a closing Telegram PR and a maintainer-authored shared wake-policy PR, so it should stay open until maintainers choose and land the intended contract.

## What This Changes

The issue requests first-class Telegram reaction triggers that wake agents, normalize standard and custom emoji, and support operator reaction semantics such as acknowledgments or approvals.

## Best Possible Solution

Adopt or supersede the linked Telegram work with a default-safe, explicit reaction wake/semantics contract that normalizes standard and custom emoji, wakes only authorized mapped reactions, and updates schemas, docs, tests, and live Telegram proof together.

## Reproduction Assessment

Yes for the wake-trigger subset: an authorized Telegram message_reaction with an added standard emoji reaches enqueueSystemEvent and then returns without a heartbeat or processMessage call. The broader per-emoji/custom-emoji semantics are new feature/API behavior rather than a fully defined existing bug contract.

## Solution Assessment

Unclear until maintainers choose the API. The requested direction is coherent, but the maintainable solution should decide between the shared notification wake policy, Telegram-specific reaction semantics, or a narrow wake fix before merging one implementation path.
