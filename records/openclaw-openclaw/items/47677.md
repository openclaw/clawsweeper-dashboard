## Summary

Keep open: current main and v2026.5.22 still enqueue accepted Telegram reactions as passive system events, while reaction-only wake behavior, custom-emoji normalization, and reaction semantics remain unresolved. The issue also has open linked implementation PRs, so cleanup closure would race the maintainer contract decision.

## What This Changes

Request first-class Telegram reaction handling so standard and custom emoji reactions can wake agents, carry configurable semantics, and be observable for operator workflows.

## Best Possible Solution

Adopt or supersede the linked PRs with one default-safe reaction contract that normalizes standard/custom emoji, wakes only authorized opt-in reactions after successful enqueue, documents the config, and includes focused tests plus live Telegram proof.

## Reproduction Assessment

Yes for the wake-trigger subset: source inspection shows an authorized Telegram message_reaction with an added emoji reaches enqueueSystemEvent and then returns without any wake call. The broader custom-emoji and per-emoji semantics are a feature/API gap rather than a fully defined existing bug contract.

## Solution Assessment

Unclear until maintainers choose the API. The safest path is a default-safe opt-in wake and stable reaction-key contract rather than unconditional reaction-triggered execution.
