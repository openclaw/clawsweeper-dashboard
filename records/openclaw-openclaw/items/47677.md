## Summary

Keep open: current main still treats Telegram reactions as passive queued context events, while the requested wake behavior, custom-emoji normalization, per-reaction semantics, and operator action policy remain unmerged and are paired with open implementation PRs.

## What This Changes

The issue requests first-class Telegram reaction input, including reaction-only wakeups, standard and custom emoji normalization, per-emoji semantics, observability, and upgrade-safe behavior.

## Best Possible Solution

Review and either land or supersede the open Telegram reaction PRs with a Telegram-owned design that authorizes configured reaction semantics, normalizes standard and custom emoji, wakes only mapped actions, records clear ignore/reject/execute decisions, and updates schema, docs, and focused tests together.

## Reproduction Assessment

Yes for the wake-trigger subset: current main receives authorized message_reaction updates and enqueues system events, then returns without an agent-turn wake. The broader per-emoji execution semantics are a new feature, so there is no existing failing contract to reproduce for that part.

## Solution Assessment

Unclear for the exact implementation shape: the issue is valid, but maintainers need to choose between the broader reactionSemantics design and the narrower wake fix. Closing this issue before those open PRs are resolved would be premature.
