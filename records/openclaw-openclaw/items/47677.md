## Summary

Keep open: current main and the latest release still treat accepted Telegram reactions as passive system events, while the requested wake policy, stable custom-emoji keys, and first-class per-reaction semantics are not merged. The shared implementation path at https://github.com/openclaw/openclaw/pull/83611 is relevant but still open/conflicting and not proof-cleared, so it cannot close this issue yet.

## What This Changes

The issue requests first-class Telegram reaction handling so configured reactions can wake agents, normalize standard and custom emoji, express operator semantics, and expose debugging signals.

## Best Possible Solution

Keep this open until maintainers land or explicitly scope a shared notification-wake policy, stable Telegram reaction keys, and the product boundary for per-emoji semantics.

## Reproduction Assessment

Yes. Current source shows the narrow wake gap: Telegram reactions are enqueued as system events but do not request a heartbeat wake; issue comments also report that behavior on released Telegram DM setups.

## Solution Assessment

Unclear as a final product design. A shared, default-safe notification wake policy looks like the right base, but custom-emoji keys and per-emoji execution semantics need maintainer policy before a new autonomous fix is safe.
