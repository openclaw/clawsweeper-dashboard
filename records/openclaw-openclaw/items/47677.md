## Summary

Keep this open. Current main still treats Telegram reactions as received notification/context events rather than first-class wake or execution triggers: standard emoji reactions are enqueued as system events, Telegram custom emoji reactions are filtered out, and there is no shipped reaction semantics config or heartbeat wake path. The issue is also paired with open PR #47687 and the narrower open wake-trigger bug #64752, so it should not be closed before that implementation work lands or is superseded.

## What This Changes

The issue requests Telegram reaction-only inputs, including standard and custom emoji, to become configurable first-class signals that can wake the agent, drive operator workflow semantics, and expose useful debugging details.

## Best Possible Solution

Review #47687 against #64752 and either land it or replace it with a narrow Telegram-plugin implementation that normalizes standard and custom emoji, maps configured reactions to no-op/wake/execution semantics, wakes only authorized mapped reactions, logs ignored/rejected/executed decisions, and updates focused tests plus the public Telegram docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists in current main: invoking the Telegram message_reaction handler with reactionNotifications enabled reaches enqueueSystemEvent and returns without any heartbeat wake path; the issue discussion also includes live v2026.4.29 reports that reaction context events arrive but do not trigger a turn.

## Solution Assessment

No for current main; it only implements notification/context delivery. The likely best direction is the open PR’s Telegram-owned semantic mapping and mapped wake behavior, but maintainers still need to review it against the authorization, topic-routing, and mergeability risks before treating it as the final fix.
