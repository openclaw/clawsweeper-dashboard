## Label Justifications

- `P3`: This is a focused native companion-app scrolling and readability enhancement, not a confirmed data-loss, security, crash-loop, or broken core runtime issue.
- `impact:session-state`: The issue is about preserving the user's viewport and reading position as the session transcript updates during streaming and new messages.

## Summary

Keep open: the native iOS and Android transcript behavior is not fully reader-managed on current main or the latest release. SwiftUI has a partial pinned-to-bottom guard but no jump-to-latest affordance or session/send placement policy, while both Android chat lists still animate to a timeline target whenever the timeline changes.

## What This Changes

The issue requests reader-managed scrolling for native iOS and Android chat transcripts, including paused follow mode, a jump-to-latest affordance, and better initial/send anchoring.

## Best Possible Solution

Keep this open for a focused native app UX design and implementation that mirrors the WebChat reader-managed follow pattern where appropriate, adds iOS and Android parity, and proves the behavior on real or simulated mobile surfaces without protocol or persistence changes.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection supports the report: Android always animates to a timeline target, native tests lack reader-managed coverage, and iOS lacks the requested jump-to-latest/session anchoring pieces despite partial pinned-state handling.

## Solution Assessment

Unclear as a final implementation, but the requested direction is plausible and matches the existing WebChat pattern. Maintainers should choose the native UX details for paused follow mode, jump affordance, session restoration, and newly sent prompt placement before an autonomous repair PR.
