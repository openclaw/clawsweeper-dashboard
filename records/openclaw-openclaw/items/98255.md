## Summary

Keep open: current main and v2026.6.11 still do not provide the requested reader-managed native transcript behavior. SwiftUI has partial pinned-to-bottom guards, while Android still animates to timeline targets without reader intent or a jump-to-latest affordance.

## What This Changes

Request reader-managed scrolling for native iOS and Android chat transcripts, including paused follow mode, a Jump to latest affordance, and better initial/send anchoring.

## Best Possible Solution

Implement one maintainer-approved reader-managed follow model across SwiftUI and Android Compose chat: follow only at the live edge, show a native Jump to latest/resume affordance when paused, preserve useful context after send/session restore, and prove it with focused native tests plus device or simulator evidence.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection supports the report: Android auto-animates to timeline targets, iOS lacks the requested jump/session anchoring pieces, and native tests do not cover reader-managed pause/resume behavior.

## Solution Assessment

Unclear as a final implementation, but the requested direction is plausible and matches the existing browser WebChat near-bottom/New messages pattern. Maintainers should choose the native UX details before implementation because the iOS and Android surfaces already differ and mobile gestures/accessibility matter.
