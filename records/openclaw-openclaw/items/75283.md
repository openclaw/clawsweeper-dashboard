## Summary

Current main now implements the runtime-deps convergence behavior this report asks for: generated package-level manifests can satisfy later narrower lazy plugin requests, the lazy package-level path has an early materialization fast path before full bundled-manifest scans or installs, and regression tests cover the no-reinstall cases. The fix is only on main/Unreleased; it is not contained in the latest v2026.4.29 tag.

## What This Changes

The issue reports that OpenClaw 2026.4.29 reaches gateway readiness with Telegram enabled, then becomes event-loop starved while intermittently spawning gateway-owned runtime-deps `pnpm install` children under the packaged plugin runtime root.

## Best Possible Solution

Keep the current main implementation and ship it in the next release, while leaving broader runtime-mirror and gateway CPU investigations to their narrower open issues if symptoms remain after this convergence fix.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction against OpenClaw 2026.4.29 with gateway logs, process trees, repeated child PIDs, runtime-deps root details, and liveness warnings; this review verified the current-main source and regression tests rather than rerunning the live VPS scenario.

## Solution Assessment

Yes. For the reported runtime-deps reinstall loop, the current main solution is narrow and maintainable: it proves package-level convergence before lazy plugin activation can re-enter package-manager staging, without making Telegram-specific changes or broad package-manager policy changes.
