## What This Changes

The PR adds a Windows Scheduled Task fallback path when `schtasks /Run` is accepted but the task reports a clean stopped result and no gateway listener appears.

## Best Possible Solution

Rebase the stopped-success fallback onto current main's scheduler lifecycle code, fold stopped-success handling into the shared launch-observation state machine, and add native Windows proof that the gateway becomes reachable.

## Reproduction Assessment

Yes at source level: current main treats `Last Run Result=0` as stopped and `shouldFallbackScheduledTaskLaunch` currently falls through unless the observation is `not-yet-run`. Native Windows behavior proof has not been established in this read-only review.

## Solution Assessment

Likely the right layer, but not the best complete fix yet: the fallback belongs in `src/daemon/schtasks.ts`, but the patch should share the state-machine handling with the existing poll and prove the native Windows path.
