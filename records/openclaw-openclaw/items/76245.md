## What This Changes

The branch adds a Windows Scheduled Task stopped-success observation and test so an accepted `/Run` that exits cleanly with no listener launches the existing Startup-style detached fallback.

## Best Possible Solution

Rebase the stopped-success fallback onto current main's scheduler lifecycle code, handle stopped-success inside the shared launch-observation polling path, and add redacted native Windows proof that the gateway becomes reachable.

## Reproduction Assessment

Yes at source level: current main treats `Last Run Result=0` as stopped and `shouldFallbackScheduledTaskLaunch` currently falls through unless the observation is `not-yet-run`. Native Windows behavior proof has not been established in this read-only review.

## Solution Assessment

Likely the right layer, but not the best complete fix yet: the fallback belongs in `src/daemon/schtasks.ts`, but the patch should share stopped-success handling with the existing poll and prove the native Windows path.
