## Summary

Keep open. Current main has model fallback and run-level auth-profile rotation, but compaction still carries a single resolved auth profile into each compaction invocation instead of retrying the same compaction snapshot across same-provider profiles.

## What This Changes

The issue requests same-provider auth-profile rotation and failover for compaction when the selected profile is rate-limited or in cooldown.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with existing auth semantics, retries only classified failover failures, and commits exactly one successful compaction result across direct and context-engine paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path. A focused harness can configure multiple same-provider profiles, make the first compaction summarization call fail with a rate-limit/failover-classified error, and assert current main does not retry the same compaction snapshot with the second profile.

## Solution Assessment

Yes on direction, but not as a blind retry. The maintainable fix is an idempotent compaction attempt boundary that reuses existing same-provider auth ordering and applies only one successful result.
