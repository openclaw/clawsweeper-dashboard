## Summary

Keep open: current main and the latest release still carry one selected auth profile through direct compaction instead of retrying same-provider backup profiles, and the linked fixing PR is open but unmerged.

## What This Changes

The issue asks for compaction to retry same-provider backup auth profiles when the current profile is rate-limited or overloaded.

## Best Possible Solution

Land or replace the linked fixing PR with same-provider auto-profile rotation that preserves user-locked profiles and applies exactly one compaction result from a stable transcript snapshot.

## Reproduction Assessment

Yes at source level: current main uses one selected auth profile for direct compaction while normal embedded runs have explicit profile advancement on retryable failures. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

Yes on direction, but the final fix needs maintainer-approved idempotency semantics: rotate only auto-selected same-provider profiles, preserve user-locked profiles, and apply one result from a stable transcript snapshot.
