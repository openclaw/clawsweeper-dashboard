## Label Justifications

- `P2`: This is a valid long-session reliability bug with a focused surface, but it is not an emergency outage.
- `impact:session-state`: The failure happens during transcript compaction and can leave long-running sessions unable to shrink context safely.
- `impact:auth-provider`: The requested behavior is specifically about auth-profile routing, cooldown, and failover within one provider.

## Summary

Keep open. Current main still has normal-run auth-profile rotation, but compaction resolves and carries one auth profile per compaction invocation, so the same-provider failover requested here remains valid and needs maintainer agreement on the idempotent retry boundary before implementation.

## What This Changes

The issue asks compaction summarization to retry across multiple auth profiles for the same provider when the active profile is rate-limited or cooling down.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with existing auth semantics, retries only classified failover failures, and commits exactly one successful compaction result across direct and context-engine paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure multiple same-provider profiles, make the first compaction provider call fail with a failover-classified rate-limit or cooldown error, and observe that compaction does not retry the same snapshot with the next profile.

## Solution Assessment

Yes on direction, but not as an unconditional retry loop. The maintainable solution is to add an idempotent compaction attempt boundary and reuse the existing same-provider auth ordering so one successful summary is applied once.
