## Label Justifications

- `P2`: This is a valid compaction/auth failover bug with limited blast radius and a non-trivial session-state design requirement.
- `impact:session-state`: The reported failure leaves long-running sessions stuck around compaction and the fix must avoid duplicate compaction state changes.
- `impact:auth-provider`: The issue is specifically about auth-profile cooldown and rotation behavior within one provider.

## Summary

Keep open. Current main and v2026.5.18 still pass a single authProfileId through compaction, while normal agent runs build and advance an ordered auth-profile candidate list; the idempotent retry boundary raised in discussion remains the key design constraint.

## What This Changes

Request compaction to retry same-provider auth profiles when the current profile is rate-limited or in cooldown, matching normal run failover behavior.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with existing auth semantics, retries only classified failover failures, and commits exactly one successful compaction result across direct and context-engine paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure multiple same-provider profiles, make the first compaction summarization call fail with a failover-classified rate-limit error, and current main has no same-compaction retry path to the second profile.

## Solution Assessment

Yes on direction, but not as a blind retry. The maintainable fix needs an idempotent compaction attempt boundary that retries the same snapshot and applies at most one successful result.
