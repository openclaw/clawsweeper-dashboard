## Label Justifications

- `P2`: This is a valid long-session reliability bug with a focused surface, but it is not an emergency outage.
- `impact:session-state`: The failure occurs during transcript compaction and can leave long-running sessions unable to shrink context safely.
- `impact:auth-provider`: The requested behavior is specifically about auth-profile routing, cooldown, and same-provider failover.

## Summary

Keep open. Current main still has normal-run auth-profile rotation, but compaction carries a single authProfileId through both direct and context-engine paths, so a rate-limited/cooldowned compaction profile is not retried against the next same-provider profile; the discussion's idempotent retry boundary remains a real design requirement.

## What This Changes

The issue requests same-provider auth-profile rotation or failover for compaction when the selected profile is rate-limited or in cooldown.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with the existing auth semantics, retries only classified failover failures, and applies exactly one successful compaction result across direct and context-engine paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure multiple same-provider profiles, make the first compaction provider call fail with a failover-classified rate-limit or cooldown error, and observe that compaction has only one authProfileId rather than a candidate list to retry the same snapshot with the next profile.

## Solution Assessment

Yes on direction, but not as an unconditional retry loop. The maintainable solution needs an idempotent compaction attempt boundary and should reuse the existing same-provider auth ordering so one successful summary is committed once.
