## Summary

Keep open. Current main still passes a single effective auth profile into compaction and returns or fails from that compaction attempt; the same-provider candidate ordering, cooldown skipping, failure marking, and profile advancement used by normal turns are not reused inside direct or context-engine compaction. The issue discussion also correctly adds an idempotency/snapshot requirement, so this should not be closed as implemented or turned into a blind retry wrapper.

## What This Changes

Compaction should retry eligible same-provider auth profiles when the active compaction profile is rate-limited, unavailable, or in cooldown.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with existing auth semantics, retries only classified failover failures, updates cooldown/failure state consistently, and commits exactly one successful compaction result across both direct and context-engine paths.

## Reproduction Assessment

Yes. A focused harness can configure two same-provider profiles, make the first compaction request fail with a rate-limit/auth failover error, and assert current main does not retry that same compaction snapshot with the second profile.

## Solution Assessment

Yes on direction, unclear on final implementation. Same-provider profile failover is the narrow maintainable scope, but the implementation should be an idempotent compaction job rather than wrapping the current side-effectful `activeSession.compact()` call in a blind retry loop.
