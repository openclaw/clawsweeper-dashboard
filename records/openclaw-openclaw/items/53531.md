## Summary

Keep open. Current main still has normal-run auth-profile candidate rotation, but compaction paths carry a single `authProfileId`, so a rate-limited or cooldowned compaction call cannot retry the same provider with the next eligible profile; the discussion's idempotent retry boundary remains necessary.

## What This Changes

The issue requests same-provider auth-profile rotation/failover for compaction when the active profile is rate-limited or in cooldown.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with the existing auth semantics, retries only classified failover failures, and commits at most one successful compaction result across direct and context-engine paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure multiple same-provider profiles, force the active compaction provider call into a rate-limit/cooldown failure, and the compaction code has only one `authProfileId` rather than a candidate list to retry the same snapshot with the next profile.

## Solution Assessment

Yes on direction, but not as an unconditional retry loop. The maintainable fix needs a maintainer-approved idempotent compaction attempt boundary and should reuse the existing same-provider auth ordering so only one successful summary is applied.
