## Summary

Keep open: current main still uses one selected auth profile for a compaction summarization attempt, while normal embedded runs can rotate auth profiles on rate-limit or overload failures; no merged fix or canonical duplicate was found.

## What This Changes

The issue requests same-provider auth-profile rotation or failover for compaction summarization when the initially selected profile is rate-limited or cooling down.

## Best Possible Solution

Add a compaction job boundary that snapshots the transcript, retries eligible same-provider auth profiles against that snapshot, and applies exactly one successful summary/checkpoint.

## Reproduction Assessment

Yes at source level: current compaction preserves one auth profile for same-provider compaction attempts, while the normal embedded-run auth controller can advance to the next non-cooldown profile. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No final implementation shape is settled yet: same-provider rotation is the right direction, but the best fix must add an idempotent snapshot/apply-once boundary before retrying compaction across profiles.
