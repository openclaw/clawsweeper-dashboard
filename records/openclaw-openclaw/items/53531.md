## Summary

Keep open. Current main added compaction model fallback, but it still does not reuse the normal same-provider auth-profile candidate order during a single compaction job; compaction resolves one effective profile, performs one summarization call, and then returns failure or falls through to model fallback rather than marking and advancing to the next eligible profile.

## What This Changes

The issue requests same-provider auth-profile rotation/failover for compaction requests when the selected profile is rate-limited or cooling down.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with the existing auth semantics, retries only classified failover failures, updates cooldown/failure state consistently, and commits exactly one successful compaction result across direct and context-engine paths.

## Reproduction Assessment

Yes. A focused harness can configure multiple same-provider profiles, make the first compaction summarization call fail with a rate-limit/failover-classified error, and assert current main does not retry the same compaction snapshot with the second profile.

## Solution Assessment

Yes on direction, unclear on final implementation shape. Same-provider profile failover is the narrow maintainable scope, but it should be implemented as an idempotent compaction attempt rather than by blindly retrying the existing side-effectful compaction call.
