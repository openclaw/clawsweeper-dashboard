## Summary

Keep open: current main still routes compaction through one selected auth profile per compaction attempt, while normal embedded runs have explicit auth-profile advancement on rate-limit/overload failures. The discussion's idempotency concern is still the key design blocker before this becomes a safe fix PR.

## What This Changes

The issue asks for compaction summarization to reuse same-provider auth-profile rotation/failover when the active profile is rate-limited or cooling down.

## Best Possible Solution

Add an idempotent compaction job boundary that snapshots the transcript, retries eligible same-provider auth profiles against that snapshot, and applies exactly one successful summary/checkpoint.

## Reproduction Assessment

Yes, at source level: current compaction preserves one selected auth profile for same-provider compaction attempts, while normal embedded runs can advance to another non-cooldown profile on retryable failures. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No final implementation shape is settled yet. Same-provider rotation is the right target, but the maintainable fix needs an idempotent snapshot/apply-once compaction boundary before profile retries are added.
