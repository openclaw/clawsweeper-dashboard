## Summary

Keep open. Current main has compaction model fallback, but compaction still carries one resolved auth profile into each compaction attempt and does not reuse the normal same-provider auth-profile candidate rotation during the compaction job.

## What This Changes

The issue requests same-provider auth-profile rotation during compaction when the selected profile is rate-limited or cooling down.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with existing auth semantics, retries only classified failover failures, and commits exactly one successful compaction result across direct and context-engine paths.

## Reproduction Assessment

Yes. A focused harness can configure multiple same-provider profiles, make the first compaction summarization call fail with a rate-limit/failover-classified error, and assert current main does not retry the same snapshot with the second profile.

## Solution Assessment

Yes on direction, but the implementation shape needs maintainer agreement. Same-provider profile failover is the narrow scope, but it should be an idempotent compaction attempt rather than a blind retry of side-effectful compaction calls.
