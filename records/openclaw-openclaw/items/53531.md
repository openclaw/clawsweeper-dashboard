## Summary

Keep open. Current main and the latest release still route compaction through a single selected auth profile, while normal agent turns have candidate-list rotation; the remaining work is the compaction-specific retry/idempotency contract, not stale cleanup.

## What This Changes

The issue requests same-provider auth-profile rotation and failover for compaction when the currently selected profile is rate-limited, cooling down, or otherwise unavailable.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the compaction input once, orders eligible same-provider auth profiles with existing auth semantics, retries only classified failover failures, and commits at most one successful compaction result across direct and context-engine paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure multiple same-provider profiles, force the active compaction provider call into a rate-limit/cooldown failure, and current compaction has only one authProfileId rather than a candidate list to retry the same snapshot with the next profile.

## Solution Assessment

Yes on direction, but not as an unconditional retry loop. The maintainable fix needs a maintainer-approved idempotent compaction attempt boundary and should reuse existing same-provider auth ordering so only one successful summary is applied.
