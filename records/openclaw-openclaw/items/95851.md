## Summary

Keep open. Current main and v2026.6.10 still have the Gemini CLI re-onboarding gap: the Google plugin only writes a Gemini CLI runtime binding for one default model and the Google manifest still lacks the auth alias needed for `google-gemini-cli` profiles to canonicalize to `google`; the open linked PR at https://github.com/openclaw/openclaw/pull/95951 remains the candidate fix.

## What This Changes

The issue requests fixing Gemini CLI re-onboarding so Google OAuth setup produces a working Gemini chat response instead of leaving OpenClaw Control stuck loading.

## Best Possible Solution

Review and land, revise, or replace the focused Google-plugin-owned fix that adds the Gemini CLI auth alias and complete runtime binding coverage without duplicating model catalog policy.

## Reproduction Assessment

Yes, at source level rather than live: current main only binds the default Gemini CLI model during onboarding and lacks the manifest auth alias needed for `google-gemini-cli` profiles to match `google` provider routing. A live reproduction would re-onboard through Gemini CLI OAuth, select a Google model missing that runtime binding, and send a Control message.

## Solution Assessment

Yes for the direction: the maintainable fix belongs in the Google plugin manifest/onboarding config path, with regression coverage for auth alias and runtime resolution. The currently linked PR is the right candidate to review, but this issue should remain open until a reviewed fix merges.
