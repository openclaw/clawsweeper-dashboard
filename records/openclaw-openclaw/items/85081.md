## Summary

Keep open. Current source and the shipped v2026.5.19 Codex app-server path appear to drop the explicitly requested shell tool under a narrow allowlist, so this is not safe to close as stale, duplicate, or implemented.

## What This Changes

The issue reports that isolated cron `agentTurn` runs for the main agent lose `exec` when `lightContext: true` is combined with `toolsAllow: ["exec"]`, causing announce delivery to treat a no-exec assistant response as a successful run.

## Best Possible Solution

Honor explicit cron `toolsAllow` in the Codex app-server runtime by exposing a usable OpenClaw `exec` path for `exec` allowlists while keeping `lightContext` limited to bootstrap context reduction and preserving fail-closed behavior for unrelated tools.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection gives a clear Codex-provider reproduction path: cron forwards `toolsAllow`, Codex disables native shell for narrow allowlists, and dynamic `exec` is excluded before filtering.

## Solution Assessment

Yes, the requested direction is aligned with the documented `--tools` contract. The safer implementation is to make the Codex app-server path materialize `exec` only when explicitly allowlisted, plus add a visible warning or error for requested tools that still cannot be provided.
