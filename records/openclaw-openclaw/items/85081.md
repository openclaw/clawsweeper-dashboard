## Summary

Keep open: current source still has a plausible Codex app-server path where a narrow `toolsAllow: ["exec"]` disables Codex native shell access while OpenClaw's Codex dynamic-tool profile also excludes `exec`, so this is not implemented on main.

## What This Changes

The issue reports that isolated cron agent-turn jobs with `lightContext: true` and `toolsAllow: ["exec"]` on the `main` agent do not expose an executable shell tool and then announce the graceful no-exec response as a successful run.

## Best Possible Solution

Honor explicit cron `toolsAllow: ["exec"]` in the Codex app-server runtime by exposing a usable shell execution path or failing the run with a clear validation error, while keeping `lightContext` scoped to bootstrap-context reduction.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection gives a clear Codex-provider failure path: cron forwards `toolsAllow`, Codex disables native shell for narrow allowlists, and dynamic `exec` is excluded before filtering.

## Solution Assessment

Yes, the requested direction matches the documented `--tools` contract. The safest fix is to make the Codex app-server path satisfy explicit `exec` allowlists without widening unrelated narrow allowlists, plus add a clear warning or run error for requested tools that cannot be provided.
