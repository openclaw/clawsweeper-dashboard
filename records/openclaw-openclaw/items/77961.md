## Summary

Keep open. The branch is a plausible API direction, but it is not merge-ready because the latest diff can pause reconnect before clearing an existing tick timer, the proof is still mocked/unit-test only, and resolving the linked operational issue still needs caller-specific retry policy review.

## What This Changes

The PR adds an opt-in `GatewayClientOptions.maxReconnectAttempts` cap, resets the reconnect counter on stop/hello-ok, and adds GatewayClient reconnect tests.

## Best Possible Solution

Define the retry-cap semantics, clear lifecycle timers before any reconnect pause, document the new public option, and either wire caller-specific opt-ins or keep the linked issue open for that follow-up.

## Reproduction Assessment

Yes for source-level behavior: current main's ordinary close path reaches `scheduleReconnect()`, which reschedules `start()` without an attempt budget. The PR-specific timer leak is also source-reproducible with `maxReconnectAttempts: 0` after a successful hello-ok because the cap return precedes timer cleanup.

## Solution Assessment

No, not yet. An opt-in API may be the right maintainable direction, but this patch should clear timers before pausing and needs maintainer agreement on caller opt-ins before it can resolve the operational issue.
