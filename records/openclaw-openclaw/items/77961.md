Keep open. The PR addresses a real gateway reconnect concern, but it is not merge-ready because current main moved the runtime to `packages/gateway-client`, the option is not wired to any affected caller, and the proof remains mock-only.

## What This Changes

The branch adds an optional `maxReconnectAttempts` GatewayClient option with reconnect-attempt counting and two mocked reconnect tests in the pre-package `src/gateway` client.

## Best Possible Solution

Port the retry limit to `packages/gateway-client`, expose it through any intended SDK/CLI/caller surfaces, and choose per-caller defaults so long-lived node-host recovery and short-lived client fail-fast behavior are both intentional.

## Reproduction Assessment

Yes, from source inspection: current `packages/gateway-client/src/client.ts` schedules reconnects with capped backoff but no max-attempt stop condition. I did not run a live reconnect repro because this was a read-only review.

## Solution Assessment

No. An opt-in cap is safer than a global retry cap, but this PR targets the old implementation path and does not wire the option to affected callers; the maintainable fix is package-level implementation plus explicit caller policy.
