## Summary

Keep open. Current main still matches the reported Discord action-discovery failure: Discord message-tool discovery can pass a raw `channels.discord.token` SecretRef into strict account/token resolution, while the docs support SecretRef-backed Discord tokens and the adjacent send/reply/status fixes do not cover this discovery path.

## What This Changes

The issue asks for Discord runtime message-tool discovery and operations to honor resolved `channels.discord.token` SecretRefs when `openclaw secrets audit` reports them resolved.

## Best Possible Solution

Discord action discovery should use a config-only credential-presence accessor, or an already resolved runtime snapshot, so configured SecretInputs can advertise safe actions without reading token values; action execution should continue to use strict scoped SecretRef resolution.

## Reproduction Assessment

Yes. The issue gives concrete config and the current source has a direct path from Discord `describeMessageTool` to strict token normalization, with an existing test proving raw SecretRef objects throw there.

## Solution Assessment

No current solution on main. The prior send/reply/status fixes are adjacent but leave Discord action discovery using strict account resolution; the narrower maintainable fix is to split discovery credential-presence checks from runtime token reads.

## Repair Work Prompt

Fix the Discord message-tool discovery SecretRef regression tracked by https://github.com/openclaw/openclaw/issues/75324. Current main lets `discordMessageActions.describeMessageTool` call strict `resolveDiscordAccount`/`resolveDiscordToken`, so a raw `channels.discord.token` SecretRef can throw during action discovery even when the gateway runtime has resolved it. Keep runtime action execution strict and fail-closed; only the read-only discovery path should avoid dereferencing token values. Prefer a Discord-owned config-only credential-presence/accessor helper, similar in spirit to the read-only accessor in `extensions/discord/src/shared.ts`, or use an already resolved runtime snapshot when available. Add regression coverage for top-level and account-scoped Discord SecretRef tokens in discovery, including that actions are still gated by `actions` config and disabled/unconfigured accounts do not gain actions. Do not broaden core with Discord-specific ids, do not change SecretRef audit semantics, and do not make runtime sends tolerate unresolved tokens. Add a user-facing changelog fix entry with reporter credit if the PR changes behavior.
