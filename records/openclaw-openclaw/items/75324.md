## Summary

Keep open. Current main still has the Discord message-action discovery path that strict-resolves `channels.discord.token`, so the reported SecretRef failure remains observable in source. The issue is also explicitly paired with open PR #75424, so the right cleanup is to review and land or close that PR before closing this issue.

## What This Changes

The issue reports that Discord message-action/runtime paths can receive a raw `channels.discord.token` SecretRef at runtime even though `openclaw secrets audit` reports no unresolved secrets.

## Best Possible Solution

Keep this issue open until #75424 or an equivalent fix lands. The end state should make Discord action discovery use config-only credential inspection for SecretRef-backed tokens while preserving strict token resolution for actual runtime send/auth paths, with regression coverage for top-level and account-scoped Discord token SecretRefs.

## Reproduction Assessment

Yes. The issue gives concrete config and the current source path reproduces the failure statically: raw SecretRef config reaches `describeDiscordMessageTool` -> `resolveDiscordAccount` -> `resolveDiscordToken` -> `normalizeResolvedSecretInputString`.

Yes, keeping the issue open while reviewing the linked closing PR is the best current path. Closing before merge would lose the issue/PR pairing, and queuing a separate repair would duplicate the open implementation candidate.
