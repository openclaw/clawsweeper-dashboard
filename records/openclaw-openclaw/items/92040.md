## Summary

The branch addresses the reported custom Anthropic-provider regression, but its generic API-ref owner lookup is not safe as written: transport API names are shared by bundled plugins, and the resolver stops at the first matching public surface rather than the first surface that supplies a thinking profile. The PR should remain open for a focused repair or canonical selection against the narrower competing PR.

## What This Changes

The PR propagates selected model API metadata into thinking-policy resolution, expands bundled provider-policy lookup to API/catalog references, and adds Anthropic-compatible profile fallback coverage.

## Best Possible Solution

Keep one canonical API-aware thinking-profile path: either iterate deterministic eligible bundled owners until a profile is returned, with shared-API regression tests, or use the narrower existing thinking chokepoint and avoid expanding generic policy ownership.

## Reproduction Assessment

Yes. On current main, the catalog API is identified but not passed into provider thinking-policy dispatch, and exact provider-id lookup therefore misses the Anthropic profile for a renamed provider.

## Solution Assessment

No, not yet. Routing by selected API is the right fact, but returning the first bundled surface for a non-unique transport API is broader and less reliable than either exhaustive owner resolution or the narrower chokepoint fix.
