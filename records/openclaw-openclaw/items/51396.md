## Summary

Keep open: current main and v2026.5.22 still clear self-declared scopes for device-less shared-token WebSocket operator sessions, and open PRs already cover competing security-policy fixes that need maintainer reconciliation.

## What This Changes

The issue reports that non-local token-authenticated operator WebSocket clients lose requested operator scopes and then fail write-scoped calls such as chat.send.

## Best Possible Solution

Land one maintainer-approved WebSocket shared-token scope contract, preferably the bounded gateway.auth.tokenScopes path with privileged opt-in, docs, regression tests, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main accepts the relevant device-less shared-token operator path with cleared scopes, and checked-in tests expect a follow-up scoped RPC to fail with missing scope.

## Solution Assessment

Unclear for the blanket requested behavior. Preserving all client-declared scopes for remote token-auth clients changes the auth boundary; the safer maintainable path is an explicit configured token-scope or device-identity policy.
