## Summary

Keep open: current main and v2026.6.10 still clear device-less shared-token/password WebSocket operator scopes, checked-in Gateway tests assert the resulting missing-scope behavior, and the remaining fix needs maintainer security/product approval rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to retain operator scopes for non-local token-authenticated WebSocket operator clients so write-scoped calls such as chat.send succeed after authentication.

## Best Possible Solution

Resolve the canonical Gateway auth issue with a maintainer-approved bounded token-scope or device-token contract, then land source, tests, docs, and real Gateway proof before closing it.

## Reproduction Assessment

Yes, at source level: current main clears device-less shared-token/password WebSocket operator scopes and checked-in tests assert the resulting missing-scope behavior. I did not run a live LAN/tailnet Gateway scenario because this review is read-only.

## Solution Assessment

No for unconditional retention of client-declared scopes. The safer solution is a bounded maintainer-approved token-scope or device-token contract rather than trusting remote device-less operator scope claims by default.
