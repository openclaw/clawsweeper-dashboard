## Summary

Keep open. Current main still clears self-declared operator scopes for device-less shared-token/password WebSocket operator connections outside the narrow local backend exception, and checked-in auth tests assert the resulting missing-scope behavior; open closing PRs #51413 and #57889 should be reviewed before this issue is closed.

## What This Changes

The issue requests that non-local WebSocket clients authenticated with a valid gateway token retain requested operator scopes instead of failing write-scoped RPCs after hello-ok.

## Best Possible Solution

Keep the issue open while maintainers choose the WebSocket auth policy, then either land a scoped fix with regression coverage or update docs/tests to state that remote shared-token operator clients must use device identity.

## Reproduction Assessment

Yes, from source and checked-in tests. The current tests connect with a valid shared token and no device identity, then assert a scope-gated RPC fails with `missing scope`.

## Solution Assessment

Unclear. Preserving scopes for all valid remote shared-token/password clients would address the report, but current source and docs deliberately clear self-declared device-less scopes except narrow trust paths, so the safest next step is maintainer approval of the exact trust boundary.
