## Summary

Keep open: current main and v2026.6.8 still clear scopes for device-less shared-token/password WebSocket operator clients, and the linked implementation work is unmerged and not a safe covering fix for the regular token-auth LAN path.

## What This Changes

Non-local shared-token WebSocket operator clients should keep requested operator scopes after successful token auth so `chat.send` and other write-scoped RPCs do not fail.

## Best Possible Solution

Resolve the bounded Gateway auth policy first, then land a tested fix or document the required device-identity/device-token flow before closing this issue.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 clear device-less shared-token/password WebSocket operator scopes, and checked-in tests assert the resulting missing-scope behavior.

## Solution Assessment

No for unconditional scope retention. The safer solution is a maintainer-approved bounded token-scope or device-token contract with tests/docs for both retained-scope and still-cleared cases.
