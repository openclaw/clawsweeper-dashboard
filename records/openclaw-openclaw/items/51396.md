## Summary

## What This Changes

The issue asks remote, non-loopback WebSocket clients with valid gateway token auth to retain requested operator scopes instead of connecting with scopes cleared and later failing write-scoped RPCs.

## Best Possible Solution

Settle one maintainer-approved WebSocket shared-token scope policy, then align the handler, connect-policy tests, default-token/auth compatibility suites, security docs, and linked fix PR to that policy.

## Reproduction Assessment

Yes, at source level. Current `main` clears scopes for the relevant device-less shared-token WebSocket path, and checked-in tests assert a successful connect followed by `missing scope`.

## Solution Assessment

Unclear. A blanket token-auth exception is not clearly the best fix because current docs reserve device-less WebSocket exceptions to narrow trust paths; the maintainable fix needs an explicit maintainer security-policy choice.
