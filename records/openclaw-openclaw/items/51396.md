## Summary

Keep open. Current main and the latest release still encode the reported device-less shared-token WebSocket scope clearing, and open linked gateway auth PRs need maintainer/security-policy review before this can be resolved.

## What This Changes

The issue asks for non-local WebSocket clients with valid gateway token auth to retain requested operator scopes instead of losing `operator.write` after `hello-ok`.

## Best Possible Solution

Settle one maintainer-approved WebSocket shared-token scope policy, likely via configured token scopes or a documented device-token requirement, then land the chosen implementation with source, tests, security docs, and upgrade behavior aligned.

## Reproduction Assessment

Yes, at source level. Current main clears scopes for the relevant device-less shared-token WebSocket path, and checked-in tests assert empty authenticated scopes plus a follow-up `missing scope` rejection.

## Solution Assessment

No for a blanket token-auth exception. The maintainable solution needs maintainer security approval for the remote operator scope policy, with a configured scope path or explicit device-token requirement looking safer than trusting self-declared scopes unconditionally.
