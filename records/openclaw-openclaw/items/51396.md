## Summary

Keep open. Current main still clears requested operator scopes for device-less shared-token/password WebSocket operator connections outside the narrow local backend exception, and current tests assert the resulting missing-scope behavior. The issue also has open closing PRs (#51413 and #57889), so it should remain open until maintainers review, merge, or close those candidates.

## What This Changes

Remote non-local WebSocket clients authenticated with a valid gateway token or password should retain requested operator scopes so `chat.send` and other `operator.write` methods work without device identity.

## Best Possible Solution

Keep this issue open while maintainers reconcile the WebSocket auth security model and review #51413 and #57889. The end state should either be a scoped fix with regression coverage proving authenticated non-local operator clients retain scopes safely, or explicit documentation/tests that remote WebSocket operator scopes require device identity.

## Reproduction Assessment

Yes. Current-main regression tests already encode the failure path: a valid shared-token operator connect without device identity succeeds, then a scope-gated RPC fails with `missing scope`.

## Solution Assessment

Unclear. Preserving scopes for all valid shared-token/password remote WebSocket clients would address the report, but current main deliberately clears self-declared scopes without device identity, so maintainers should approve the exact trust boundary before either PR lands.
