## Summary

Keep open: current main still clears scopes for the reported device-less shared-token WebSocket operator path, v2026.6.6 contains the same behavior, and the linked open PR is not a safe covering fix for the regular token-auth LAN case.

## What This Changes

The issue reports that non-local WebSocket operator clients authenticated with a valid gateway token lose requested operator scopes and then fail operator.write-scoped calls.

## Best Possible Solution

Maintain one canonical gateway auth-policy path: either land a maintainer-approved, tested WebSocket scope-retention fix for trusted remote clients or document that remote clients must use device identity/device tokens with approved scopes.

## Reproduction Assessment

Yes from source inspection: current main clears scopes for the reported device-less shared-token operator WebSocket path, and adjacent auth tests assert the missing-scope behavior. I did not run a live Docker/LAN repro in this read-only checkout.

## Solution Assessment

No/unclear. Preserving non-local token-auth operator scopes is a plausible workflow fix, but the narrowest maintainable solution needs an explicit gateway trust-boundary decision plus tests for both retained-scope and still-cleared cases.
