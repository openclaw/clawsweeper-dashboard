## What This Changes

The issue reports that remote token-authenticated WebSocket operator clients lose requested operator scopes before write-scoped calls like chat.send.

## Best Possible Solution

Resolve the canonical issue with a maintainer-approved bounded token-scope or device-token contract, then update Gateway source, tests, docs, and real Gateway proof before closing it.

Yes at source level: current main clears device-less shared-token/password WebSocket operator scopes and checked-in tests assert the resulting missing-scope behavior. I did not run a live LAN/tailnet Gateway scenario because this review is read-only.

## Solution Assessment

No for blanket retention of client-declared scopes. The best direction is a bounded maintainer-approved token-scope or device-token contract rather than trusting remote device-less operator scope claims by default.
