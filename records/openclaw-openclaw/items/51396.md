## Label Justifications

- `P1`: The issue blocks a token-authenticated remote Gateway operator workflow for multi-agent/backend clients and remains source-reproducible.
- `impact:security`: The remaining fix changes whether remote device-less WebSocket clients can receive privileged operator scopes after shared-token authentication.

## Summary

Keep open: current main and v2026.5.22 still clear requested operator scopes for device-less shared-token WebSocket operator sessions, and the open linked PRs still need maintainer/security reconciliation before this can close.

## What This Changes

The issue requests that valid non-local shared-token WebSocket operator clients retain operator scopes so `chat.send` and other `operator.write` calls succeed.

## Best Possible Solution

Land one maintainer-approved WebSocket shared-token scope contract, likely bounded `gateway.auth.tokenScopes` with privileged opt-in, docs, regression tests, and redacted Docker/LAN proof.

## Reproduction Assessment

Yes, at source level. Current main accepts the relevant device-less shared-token operator path with cleared scopes, and checked-in tests expect a follow-up scoped RPC to fail with `missing scope`.

## Solution Assessment

Unclear for the blanket requested behavior. The safer maintainable solution is an explicit configured token-scope or device-identity policy rather than trusting remote client-declared scopes by default.
