## Summary

Keep open. Current main and the latest release still preserve the reported missing-scope behavior in source and tests, while open related PRs carry unresolved gateway auth/security policy decisions.

## What This Changes

The issue asks for non-local token-auth WebSocket operator clients to retain requested operator scopes instead of connecting with cleared scopes and failing write-scoped RPCs.

## Best Possible Solution

Define and land one maintainer-approved WebSocket token-scope policy, likely a bounded configured `gateway.auth.tokenScopes` path or a documented device-token requirement, with source, regression tests, protocol/security docs, and real gateway proof aligned.

## Reproduction Assessment

Yes, at source level. Current main accepts a valid shared-token operator without device identity, clears its requested scopes, and checked-in tests expect the follow-up scoped RPC to fail with `missing scope`.

## Solution Assessment

Unclear. Preserving client-declared scopes would address the symptom, but current docs and tests define narrow device-less exceptions, so the safer path is an explicit maintainer-approved auth policy/config surface rather than a blanket remote token exception.
