## Label Justifications

- `P1`: The report describes a broken remote gateway operator workflow with source-level reproduction and current tests that still encode the failure mode.
- `impact:auth-provider`: The behavior depends on gateway token authentication, requested operator scopes, and authorization decisions for WebSocket clients.

- `impact:message-loss`: The observable failure rejects `chat.send` and other write-scoped operator actions after a successful connection.

## Summary

Keep open. Current main and v2026.5.18 still encode the reported device-less shared-token WebSocket scope clearing, and the issue is still linked to open gateway auth PRs that need a maintainer security-policy decision before this can be resolved.

## What This Changes

The issue asks OpenClaw to preserve requested operator scopes for non-local WebSocket clients that authenticate with a valid gateway token.

## Best Possible Solution

Define and land one maintainer-approved WebSocket shared-token scope policy, likely through a bounded `gateway.auth.tokenScopes` config path or a documented device-token requirement, with source, tests, security docs, and upgrade behavior aligned.

## Reproduction Assessment

Yes, at source level. Current main accepts a valid shared-token operator without device identity, clears its requested scopes, and checked-in tests expect the follow-up scoped request to fail with `missing scope`.

## Solution Assessment

No for a blanket token-auth exception. Because the current behavior is also documented as a security hardening, the maintainable fix is an explicit maintainer-approved scope policy such as https://github.com/openclaw/openclaw/pull/81792 or a documented device-token requirement.
