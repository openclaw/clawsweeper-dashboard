## Summary

Keep open: current main and v2026.5.22 still clear requested scopes for device-less shared-token WebSocket operator sessions, and the open fix/policy PRs need maintainer security reconciliation before this can close.

## What This Changes

The issue reports that non-local token-authenticated operator WebSocket clients lose requested operator scopes and then fail write-scoped calls such as chat.send.

## Best Possible Solution

Land one maintainer-approved WebSocket shared-token scope contract, preferably centralized in gateway connect policy or explicit token-scope configuration, with docs and regression coverage for both default fail-closed and opt-in trusted paths.

## Reproduction Assessment

Yes, at source level. Current main accepts the relevant device-less shared-token operator path with cleared scopes, and checked-in tests expect follow-up scoped RPCs to fail with missing scope.

## Solution Assessment

Unclear for the blanket requested behavior. Preserving all self-declared scopes for any remote token-auth client changes the auth boundary; the safer path is a maintainer-approved centralized policy or explicit configured token scopes.
