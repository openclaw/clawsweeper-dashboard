## Label Justifications

- `P1`: The issue blocks a remote gateway operator workflow for token-authenticated multi-agent or backend clients and remains source-reproducible.
- `impact:security`: The remaining fix changes whether remote device-less WebSocket clients can receive privileged operator scopes after shared-token authentication.
- `impact:message-loss`: The observable failure rejects `chat.send` and other write-scoped operator actions after a successful connection.
- `impact:auth-provider`: The behavior depends on gateway token authentication, auth method classification, and operator scope resolution.

## Summary

Keep open: current `main` still clears requested operator scopes for the device-less shared-token WebSocket path, current tests assert the resulting missing-scope behavior, and open linked PRs still need a maintainer-owned security policy decision before this can close.

## What This Changes

The issue asks for valid non-local token-authenticated operator WebSocket clients to retain requested operator scopes instead of connecting successfully and then failing write-scoped calls.

## Best Possible Solution

Land one maintainer-approved WebSocket shared-token scope contract, likely a bounded configured token-scope path with privileged opt-in, docs, regression tests, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current `main` accepts the relevant device-less shared-token operator path with cleared scopes, and checked-in tests expect a follow-up scoped RPC to fail with `missing scope`.

## Solution Assessment

Unclear. A blanket token-auth exception is not clearly the safest fix; the maintainable solution should be an explicit maintainer-approved WebSocket scope policy, with tests and docs aligned to that contract.
