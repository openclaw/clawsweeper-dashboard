## Summary

Keep open. The PR is a plausible narrow fix for a source-proven debounce bug, but it is still draft and the supplied proof is only an API/source probe rather than real channel behavior after the fix.

## What This Changes

The PR changes the shared inbound debounce policy to call `isControlCommandMessage()` and adds regression assertions for bare abort phrases such as `stop`, `abort`, and `wait`.

## Best Possible Solution

Merge a focused shared-helper fix after real channel proof and targeted tests, then expand or separately track plugin-local debounce gates before broadly closing the linked issue if they remain affected.

## Reproduction Assessment

Yes for source reproduction: current main routes bare `stop` through `hasControlCommand()`, which does not consult the abort trigger set, so the shared debounce predicate treats it as debounceable. I did not run a live channel gateway in this read-only review.

## Solution Assessment

Yes for the shared debounce consumers: reusing `isControlCommandMessage()` is the narrowest maintainable code direction because it preserves slash commands and adds existing abort-trigger detection. It still needs real behavior proof and maintainer judgment on plugin-local debounce predicates before merge/closure.
