## Repair Work Prompt

Fix the embedded-run dispatch-prep performance regression tracked at https://github.com/openclaw/openclaw/issues/75999. Start by measuring and separating the request-prep costs for `core-plugin-tools`, `system-prompt`, and `stream-setup`, then implement the narrowest safe amortization: reuse existing gateway/plugin metadata or runtime snapshots for plugin tool prep, cache only stable system-prompt components keyed by skill/context/config inputs, and avoid rebuilding provider stream wrappers when inputs are unchanged. Keep dynamic per-turn inputs below the cache boundary: current channel/message/thread IDs, auth/runtime secret state, session state, provider/model/transport, tool allowlists, sandbox mode, user time, and abort/yield state. Do not treat PR #75503 as solving this issue; it is TUI-specific. Add focused regression tests that prove repeat embedded attempts do not rerun the expensive stable plugin/prompt setup when inputs are unchanged and do rerun when relevant plugin/skill/context/config inputs change. Add a user-facing `CHANGELOG.md` fix entry for the active version, crediting human reporters if known and not bot/maintainer handles. Do not change plugin manifest semantics, provider transport behavior, security policy, or broad worker-thread architecture unless the measured narrow fix cannot address the regression.
