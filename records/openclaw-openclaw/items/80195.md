## Summary

Keep open: the PR targets a real ACP Discord/Codex regression, but it is not merge-ready because the contributor proof is still test-only and the stream-completion patch can now treat tool-call progress as a successful turn without a terminal completion event.

## What This Changes

The PR relaxes ACP turn completion for output-only streams, moves ACP child session patching after initialization, reloads fresh session metadata, and adds focused ACP manager/spawn regression tests.

## Best Possible Solution

Repair ACP spawn ordering and output-only completion without treating tool progress as success, then require redacted live Discord/Codex ACP proof before merge.

## Reproduction Assessment

Yes for a high-confidence source-level path: current main patches the child session before ACP initialization and rejects output-bearing streams without `done`. I did not run the live Discord/WSL2 scenario in this read-only review.

## Solution Assessment

No as-is: moving `sessions.patch` after initialization is the right direction, but the stream-completion change should not accept tool-call progress/status/id alone as a completed turn. The PR also still needs real Discord/Codex behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Do not accept tool progress as turn completion:** `src/acp/control-plane/manager.turn-stream.ts:22-26`
  - body: The new output-evidence check treats a `tool_call` status/title/id as enough to bypass the terminal `done` gate. ACP tool-call events can represent an in-progress call before any result or final assistant text, so a stream that dies after starting a tool would now mark the turn idle/succeeded instead of failed. Limit the no-`done` success path to actual assistant output, or only to a completed tool result if the runtime can distinguish it.
  - confidence: 0.85
