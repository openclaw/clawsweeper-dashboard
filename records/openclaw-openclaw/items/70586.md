## Summary

Keep this PR open, but do not merge it as-is. The remaining Slack status-activity gap is plausible for accepted early-return branches, but the patch tracks before later filters and before the shared message handler, which already tracks accepted normal messages and app mentions once.

## What This Changes

The PR threads Slack `trackEvent` into `registerSlackMessageEvents`, calls it from `message` and `app_mention` handlers after the workspace mismatch guard, and adds Slack monitor tests for tracked and mismatched events.

## Best Possible Solution

Revise the Slack message registrar so `trackEvent` fires only for accepted early-return branches that bypass `createSlackMessageHandler`, keep the shared handler as the single tracking point for normal handled messages and app mentions, add tests proving each path tracks exactly once, and include the required Slack fix changelog entry.

## Reproduction Assessment

Yes. A static reproduction is enough: with the PR patch, a normal Slack `message` or channel `app_mention` calls `trackEvent` in the registrar and then reaches `createSlackMessageHandler`, which already calls the same tracker for accepted handled messages.

## Solution Assessment

No. The requested gap is narrow, but this patch tracks too early and too broadly; the safer fix is to track only authorized early-return system-event and DM/MPIM app-mention skip branches while preserving current handler-level tracking.
