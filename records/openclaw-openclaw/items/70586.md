## Summary

Keep open: the underlying Slack message-event tracking gap is real, and the discussion correctly ties it to #69833, but this PR tracks too early. Current main already tracks accepted normal Slack messages/app_mentions inside the shared handler, so the PR would double-update those paths and still needs the required changelog entry.

## What This Changes

The PR threads Slack `trackEvent` into `registerSlackMessageEvents`, calls it from `message` and `app_mention` handlers after the mismatch guard, and adds Slack monitor tests for tracked and mismatched events.

## Best Possible Solution

Keep the shared handler as the tracking point for normal handled messages/app_mentions, add registrar tracking only for authorized early-return branches that bypass it, and document the final Slack status/activity fix in the changelog.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: with the PR diff, a normal Slack `message` or channel `app_mention` calls `trackEvent` in the registrar and then reaches `createSlackMessageHandler`, which already calls the same tracker for accepted non-duplicate messages.

## Solution Assessment

No. The requested gap is narrow, but this patch tracks too early and too broadly; the safer solution is targeted registrar tracking only for accepted branches that return before `handleSlackMessage`.
