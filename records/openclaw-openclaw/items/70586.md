## Summary

Keep this PR open for revision. Current main still has a Slack message-event activity tracking gap for accepted branches that return before the shared message handler, but the proposed placement tracks too early, double-updates normal handled events, and relies on a stale-socket rationale that current health policy no longer uses.

## What This Changes

The PR threads `trackEvent` into Slack message-event registration, calls it near the start of `message` and `app_mention` handlers, and adds Slack monitor tests for tracked and mismatched events.

## Best Possible Solution

Revise the Slack message registrar so `trackEvent` fires only for accepted message-event branches that return before `handleSlackMessage`, keep the shared handler as the single tracking point for handled messages/app mentions, add tests proving each accepted path tracks exactly once, and add the required Slack fix changelog entry.

## Reproduction Assessment

Yes. A static reproduction path is enough here: apply the PR logic mentally to a normal Slack message and the registrar calls `trackEvent` before delegating, then `createSlackMessageHandler` calls the same tracker again on the accepted message path already covered by tests.

## Solution Assessment

No. The intended gap is real, but the best fix is narrower than this patch: track only accepted early-return branches that bypass the shared handler, and do not reframe app-event timestamps as socket liveness.
