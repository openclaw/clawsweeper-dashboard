## Summary

Keep open: current main and the latest release already skip gateway-token generation when a plaintext token resolves, but the report provides production audit hashes without redacted before/after config proving `gateway.auth.token` changed. Because the reported impact is credential-state and auth-sensitive, this needs maintainer/security review or concrete repro proof before a fix or close.

## What This Changes

The issue reports that interactive `openclaw doctor --fix` can regenerate a valid `gateway.auth.token`, silently disconnecting existing gateway consumers.

## Best Possible Solution

Obtain redacted current-main before/after config-audit proof, then fix the actual writer and add a focused regression that preserves valid `gateway.auth.token` while making intentional token generation visibly logged.

## Reproduction Assessment

Unclear: the issue has production audit hashes, but no redacted before/after config or changed-path proof tying the change to `gateway.auth.token`. I did not run a write-path repro because this review is read-only, and source inspection suggests valid plaintext tokens return before generation.

## Solution Assessment

Unclear: the suggested early return is already covered by the current `needsToken` gate for resolved plaintext tokens. The best fix depends on a current-main repro that identifies whether this function or another config writer is changing the token.
