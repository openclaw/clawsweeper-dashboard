## Summary

Keep open: the report describes a serious gateway auth-token rotation, but current source, docs, and tests only show doctor should generate a token when no effective token source exists; the missing piece is a redacted current-main repro or config-audit record tying the hash changes specifically to `gateway.auth.token`.

## What This Changes

The issue requests preventing interactive `openclaw doctor --fix` from silently regenerating an existing valid `gateway.auth.token` and warning when token generation occurs.

## Best Possible Solution

Add or verify a focused doctor regression proving `doctor --fix` preserves a valid `gateway.auth.token`; if the audit data identifies another writer, fix that canonical path and make intentional token generation visibly warn about consumer restarts.

## Reproduction Assessment

Unclear: source inspection shows token generation only when `resolveGatewayAuth` lacks an effective token, but the issue provides audit hashes rather than a redacted config/audit record proving `gateway.auth.token` changed.

## Solution Assessment

Unclear: an early return for a resolved token is already present through `needsToken`, so the best fix depends on a current-main repro showing which writer changes the token.
