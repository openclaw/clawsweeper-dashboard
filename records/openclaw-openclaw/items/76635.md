## Summary

Current main already fixes the reported source-only external WhatsApp package path: publishable plugins now build package-local JavaScript runtime artifacts and installed package discovery prefers those runtime entries before falling back to TypeScript source. The fix is on main only; no release tag contains it yet.

## What This Changes

Prevent externally installed `@openclaw/whatsapp` from loading source TypeScript in a way that can starve Windows cron model calls.

## Best Possible Solution

Keep the package-local JavaScript runtime publishing path and ship the `2026.5.3` WhatsApp plugin/OpenClaw release so installs no longer execute the source-only `index.ts` payload.

## Reproduction Assessment

No high-confidence current-main failing reproduction was run, because this needs a Windows WhatsApp account and cron/model setup. The release failure path is source-backed by the issue’s `index.ts` load log and by current main’s package-runtime fix.

## Solution Assessment

Yes. Building package-local `dist` artifacts and making installed packages resolve those runtime entries is the narrow maintainable fix for source-only external plugin loads; a cron/model workaround would not address the blocking plugin load.
