## Summary

Keep this PR open. Current main still lacks a Telegram busy-session bypass for ordinary immediate-mode messages, so the underlying behavior is not obsolete, but the proposed patch has blocking correctness and API-drift issues that should be fixed before merge.

## What This Changes

The PR adds a Telegram sequential-key wrapper for busy steer/steer-backlog/interrupt sessions, exports reply-run active lookup through the plugin SDK, and adds Telegram plus reply-run registry tests.

## Best Possible Solution

Land an equivalent Telegram-owned bypass after rebasing onto current main, reusing core queue-setting semantics, honoring the bot instance config, checking the same active-run state core uses for embedded runs, and updating the Plugin SDK API baseline if the new SDK export remains necessary.

## Reproduction Assessment

Yes. A focused unit test can create a busy Telegram topic with default queue settings and show current main keeps an ordinary follow-up on the topic lane; another can model an embedded-only active run and show the PR's reply-run-only check misses it.

## Solution Assessment

No. The direction is narrow and maintainable, but this patch should not merge until it mirrors core queue defaults, uses the per-instance config, covers embedded-only active runs, and fixes the SDK baseline drift.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match the core queue default:** `extensions/telegram/src/sequential-key.runtime.ts:95`
  - body: Core resolveQueueSettings defaults an unset queue mode to steer, but this wrapper falls back to collect. With default Telegram config, a busy session will stay on the ordinary topic lane even though core would treat the turn as immediate steering, so the main user path remains blocked.
  - confidence: 0.92
- **[P2] Treat embedded-only runs as busy:** `extensions/telegram/src/sequential-key.runtime.ts:173`
  - body: Core busy-state resolution covers embedded runs that may not have a reply-run registry entry, but this gate only calls isReplyRunActiveForSessionKey. In that embedded-only state, immediate Telegram turns still fall back to the topic lane and cannot reach core interrupt or steer handling promptly.
  - confidence: 0.88
- **[P2] Use the bot instance config for lane policy:** `extensions/telegram/src/bot-core.ts:551`
  - body: createTelegramBotCore can be constructed with opts.config, and webhook/polling callers pass that instance config through. The new wrapper reloads runtime/global config instead, so a bot created with an override can choose the wrong queue mode or command policy when deciding whether to bypass the topic lane.
  - confidence: 0.86
- **[P2] Regenerate the Plugin SDK API baseline:** `src/plugin-sdk/reply-runtime.ts:33`
  - body: Adding isReplyRunActiveForSessionKey to the public reply-runtime SDK subpath changes the generated SDK API surface, but this PR does not include the tracked docs/.generated/plugin-sdk-api-baseline.sha256 update. The plugin-sdk:api:check gate will report baseline drift until the hash is regenerated or the public export is removed.
  - confidence: 0.84
