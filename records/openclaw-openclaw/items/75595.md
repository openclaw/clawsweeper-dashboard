## Summary

Keep this PR open. The async direction is appropriate, but the current head still has a P2 correctness blocker in the queued engine-owned compaction path: it reaches the new async snapshot helper only after synchronously opening the full transcript.

## What This Changes

The PR adds an async/yielded `sessions.list` path, an async compaction checkpoint snapshot helper, switches Gateway and compaction callers toward those paths, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the sessions-list yielding and async checkpoint-copy approach, then revise queued engine-owned compaction so checkpoint metadata is obtained through bounded async transcript reads or already-available runtime metadata, without a full synchronous transcript open before the async helper runs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a Gateway `sessions.list` request with derived titles/previews over a large session store, plus engine-owned compaction over a large transcript; source and dependency inspection show the synchronous calls that can block the event loop.

## Solution Assessment

No. The proposed direction is mostly right, but the current PR is not the best complete fix because one changed compaction caller still performs a full synchronous transcript open before reaching the async checkpoint copy path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid synchronous session opens in queued checkpoint capture:** `src/agents/pi-embedded-runner/compact.queued.ts:118-120`
  - body: The queued engine-owned compaction path now awaits the async checkpoint helper, but it still builds the argument with `SessionManager.open(params.sessionFile)`. Pass/read only bounded checkpoint metadata asynchronously instead.
