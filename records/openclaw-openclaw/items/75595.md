## Summary

Keep open. The PR targets real current-main Gateway event-loop hot paths, but it has the protected `maintainer` label and the compaction part is not complete: the new async snapshot helper still calls `SessionManager.open()`, which synchronously reads the full transcript in the pinned dependency.

## What This Changes

The PR adds an async `sessions.list` store path with periodic event-loop yields, adds an async compaction checkpoint snapshot helper, and switches the gateway session-list handler plus embedded compaction callers to those async helpers.

## Best Possible Solution

Revise the compaction path so checkpoint capture obtains only the needed session id and leaf metadata with bounded async I/O, or through an upstream dependency API that avoids full synchronous transcript loads. Keep the sessions-list yield path if tests prove it preserves ordering, filtering, and result shape, then add the required changelog entry.

## Reproduction Assessment

Yes. Current main still has the synchronous `sessions.list` preview hydration and synchronous checkpoint copy/reopen paths, and the linked reports describe large session stores or 20MB+ transcripts causing event-loop starvation.

## Solution Assessment

No. The sessions-list yielding change is directionally narrow, but the compaction helper still reopens copied transcripts synchronously through `SessionManager.open()`, so it does not fully solve the large-transcript event-loop block.
