## Summary

Keep open: this PR targets valid gateway event-loop starvation reports and changes the right gateway/agent surfaces, but it is draft and maintainer-labeled, and the proposed async checkpoint helper still performs a synchronous full transcript reopen through `SessionManager.open()`. That leaves a central part of the reported compaction blocking path unresolved before merge.

## What This Changes

The PR adds an async `sessions.list` store path with periodic `setImmediate` yields, adds an async compaction checkpoint snapshot helper using `fs.promises`, and switches the gateway sessions handler plus embedded compaction callers to those async functions.

## Best Possible Solution

Revise this PR so every large transcript operation on the gateway compaction hot path is async or tightly bounded, especially avoiding synchronous `SessionManager.open()` on copied checkpoint files; keep the sessions-list yielding path if it preserves existing result ordering, add focused async regression coverage, and add an Unreleased `CHANGELOG.md` Fixes entry with contributor credit.

## Reproduction Assessment

Yes. The linked reports describe concrete large-session/large-transcript event-loop starvation paths, and the code-level reproduction is clear from current main plus dependency source: `sessions.list` hydrates transcript previews via synchronous reads, and compaction snapshot capture copies then reopens large JSONL files synchronously.

## Solution Assessment

No. The PR is directionally useful, but the compaction fix is incomplete because async `copyFile` is followed by a synchronous full-file `SessionManager.open()`; the safer solution is to avoid that reopen with async header/session-id extraction or a dependency API that does not synchronously load the whole transcript.

- **[P2] Avoid reopening the checkpoint snapshot synchronously:** `src/gateway/session-compaction-checkpoints.ts:173`
  - body: The new async checkpoint helper copies the transcript with `fs.promises`, but then immediately calls `SessionManager.open(snapshotFile, ...)`. In `@mariozechner/pi-coding-agent@0.70.6`, `open()` synchronously reads the whole JSONL file via `loadEntriesFromFile()` before constructing, and construction reloads it again through `setSessionFile()`. For the 20-64MB transcripts this PR is meant to handle, compaction can still block the Gateway event loop after the async copy. Read the snapshot session id/validate the header with async I/O, or add/use a dependency API that avoids full synchronous loads, before using this in the compaction hot path.
