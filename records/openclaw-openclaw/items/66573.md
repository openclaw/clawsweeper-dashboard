## Summary

Keep open. Current main and the latest release still allow `sessions_history` to forward `limit: 2000` while `chat.history` rejects values over 1000 before its handler clamp runs; the open implementation PR at https://github.com/openclaw/openclaw/pull/66650 references this issue with closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that `sessions_history` can request `chat.history` with `limit: 2000`, exceeding the Gateway maximum of 1000 and silently losing history context.

## Best Possible Solution

Review, update, and land https://github.com/openclaw/openclaw/pull/66650 or an equivalent caller-side 1000-limit clamp with regression coverage while keeping the Gateway `chat.history` cap at 1000.

## Reproduction Assessment

Yes. A high-confidence source-level path is to invoke `sessions_history` with `limit: 2000`: current main forwards 2000 to `chat.history`, whose active validator rejects limits above 1000 before the handler clamp runs.

## Solution Assessment

Yes. Clamping at the `sessions_history` tool boundary is the narrowest maintainable fix because it aligns the caller with the existing Gateway protocol instead of raising the protocol maximum or parsing validation errors for retry.
