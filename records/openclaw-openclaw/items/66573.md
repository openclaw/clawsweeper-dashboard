## Summary

Keep open. Current main still permits `sessions_history` callers to pass an oversized limit through to `chat.history`, where Gateway validation rejects values above 1000 before the handler's later hard cap can apply; an open PR already references this issue with closing syntax, so the issue should remain open until that fix or an equivalent one lands.

## What This Changes

The issue reports that `sessions_history` can call `chat.history` with `limit: 2000`, exceeding the Gateway maximum of 1000 and silently losing history context after `INVALID_REQUEST`.

## Best Possible Solution

Land PR #66650 or an equivalent narrow fix that caps `sessions_history.limit` at 1000 before calling `chat.history`, exposes the same maximum in the tool schema, and adds regression coverage for oversized caller-supplied limits.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: call `sessions_history` with `limit: 2000`, the tool forwards 2000 to `chat.history`, and the Gateway validator rejects anything above 1000 before the handler cap runs.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix: clamp at the `sessions_history` caller boundary and publish the same limit in the tool schema, rather than raising the Gateway maximum or parsing error strings for retry.
