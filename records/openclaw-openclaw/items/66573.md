## Summary

Keep open. Current main still allows `sessions_history` callers to pass an oversized limit through to `chat.history`, where Gateway validation rejects values above 1000 before the handler's own cap can apply; the issue is also paired with open PR #66650, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `sessions_history` can request `chat.history` with `limit: 2000`, exceeding the Gateway maximum of 1000 and losing history context without retry.

## Best Possible Solution

Land PR #66650 or an equivalent narrow fix that exposes `maximum: 1000` in the `sessions_history` tool schema, clamps caller-supplied limits to 1000 before the `chat.history` request, and adds regression coverage for oversized limits.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: call `sessions_history` with `limit: 2000`, and the tool forwards 2000 to `chat.history`, whose schema rejects values above 1000 before the handler cap runs.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable fix because it aligns the tool boundary with the existing Gateway protocol limit instead of increasing the protocol maximum or parsing validation error strings for retry.
