## Summary

Keep open. Current main and v2026.6.6 already have lower-level Slack Block Kit plumbing, but the requested producer-facing cron/report support boundary is still a maintainer product/API decision rather than an implemented behavior gap.

## What This Changes

The issue asks maintainers to decide and document whether Slack-native Block Kit blocks are a supported advanced reporting surface for trusted cron/report producers.

## Best Possible Solution

Maintainers should choose and document one Slack-plugin-owned boundary: either bless trusted native Block Kit report payloads with fallback and validation rules, or keep portable presentation as the only supported producer API and track richer report/table layout separately.

## Reproduction Assessment

Not applicable. This is explicitly a feature/product-boundary request; source inspection confirms the raw Slack block paths exist while the documented support boundary remains unresolved.

## Solution Assessment

Unclear until maintainers choose the boundary. A narrow docs or Slack-plugin update is best only if maintainers bless provider-specific blocks; otherwise the maintainable answer is to keep presentation canonical and design richer report layout separately.
