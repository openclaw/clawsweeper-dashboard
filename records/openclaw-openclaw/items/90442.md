## Summary

Keep open. Current main and v2026.6.8 already have lower-level Slack Block Kit delivery paths, but the requested trusted cron/report producer contract is still an explicit Slack plugin product/API decision rather than implemented cleanup.

## What This Changes

Decide and document whether trusted cron/report producers may rely on Slack-native Block Kit blocks with fallback text, or whether portable `presentation` remains the only supported producer API.

## Best Possible Solution

Maintainers should choose one Slack-plugin-owned boundary: either bless trusted native Block Kit report payloads with fallback and validation rules, or keep portable presentation as the only supported producer API and track richer report/table layout separately.

## Reproduction Assessment

Not applicable. This is explicitly a feature/product-boundary request; source, docs, tests, and the issue discussion confirm raw Slack block paths exist while the documented producer contract remains unresolved.

## Solution Assessment

Yes as a triage shape: asking for the maintainer support-boundary decision before a PR is the right path. A narrow docs or Slack-plugin API update is best only after maintainers choose whether provider-native blocks are supported for trusted report producers.
