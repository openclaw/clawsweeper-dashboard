## Summary

Keep open: current main does not implement the requested continuation surface, and the PR head still has source-verifiable lifecycle bugs in delegate tool gating and TaskFlow delegate consumption that should block merge.

## What This Changes

The PR adds an opt-in continuation runtime with `continue_work`, `continue_delegate`, `request_compaction`, response-token fallbacks, continuation config, context-pressure events, TaskFlow-backed delegate storage, protocol fields, docs, and tests.

## Best Possible Solution

Keep this PR open, fix the concrete scheduling and TaskFlow lifecycle blockers on the branch, then have maintainers review the opt-in continuation API, config, RFC, and UX before merge.

## Reproduction Assessment

Yes for the review findings: source inspection at PR head shows the missing delegate-drain flag propagation, ignored TaskFlow update results, and the `continue_work` zero-delay contract mismatch. I did not run runtime tests under the read-only constraints.

## Solution Assessment

No. The direction may be useful, but this PR is not yet the best merge state because it can expose scheduling tools in paths that do not drain them and can re-dispatch TaskFlow delegates after lifecycle conflicts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Forward the delegate-drain flag into embedded tools:** `src/agents/pi-tools.ts:646-647`
  - body: `runEmbeddedPiAgent` forwards `drainsContinuationDelegateQueue` into `runEmbeddedAttempt`, but the attempt/tool construction path never passes it into `createOpenClawTools`. Because the `continue_delegate` gate treats undefined as enabled, command and utility runs that explicitly set the flag false can still show a scheduling tool even though no post-run delegate drain will consume its queue.
  - confidence: 0.88
- **[P2] Check TaskFlow finalization before returning delegates:** `src/auto-reply/continuation-delegate-store-taskflow.ts:98-101`
  - body: `finishFlow` reports revision conflicts via `{ applied: false }` instead of throwing, but this consume path ignores the result and still returns the delegate to the runner. If finalization loses a revision race, the flow remains queued and can be consumed and dispatched again on a later pass.
  - confidence: 0.9
- **[P2] Keep continue_work delay semantics consistent:** `src/auto-reply/reply/agent-runner.ts:2624-2625`
  - body: The tool contract says omitted or zero `delaySeconds` is immediate, and the tool reports `delaySeconds: 0`, but the scheduler clamps that value up to `minDelayMs` (5000 ms by default). Either preserve explicit zero as immediate or update the tool contract/result so agents are not told an immediate continuation was scheduled when it will wait.
  - confidence: 0.82
