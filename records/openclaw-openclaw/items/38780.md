## Best Possible Solution

## Reproduction Assessment

No. The proposed direction may be useful, but this is not the best mergeable form while delegate dispatch is split across stale inline code and the new dispatcher, delayed delegates can wait twice, and disabled continuation configs can still inject continuation-specific model events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Route tool delegates through the single dispatcher:** `src/auto-reply/reply/agent-runner.ts:2557`
  - body: This inline `consumePendingDelegates()` drains and finishes TaskFlow rows before the later `dispatchToolDelegates()` call can see them. That bypasses the new dispatcher's failure accounting and hedge logic, so rejected or failed inline spawns can be lost after the flow was already marked released/succeeded.
  - confidence: 0.86
- **[P2] Skip the second timer for matured delegates:** `src/auto-reply/reply/agent-runner.ts:2784`
  - body: `consumePendingDelegates()` only returns rows whose original `delayMs` horizon has already matured, but this branch checks `delegate.delayMs > 0` and schedules another timeout. A delayed `continue_delegate` therefore waits once in TaskFlow and then waits the same clamped delay again before spawning.
  - confidence: 0.9
- **[P2] Gate context-pressure events on continuation enablement:** `src/auto-reply/reply/agent-runner.ts:1434-1436`
  - body: This block runs whenever a threshold is configured, even if `agents.defaults.continuation.enabled` is false. That lets disabled deployments still enqueue model-visible `[system:context-pressure]` guidance mentioning CONTINUE_DELEGATE, which violates the opt-in contract for this autonomy surface.
  - confidence: 0.84
