## Label Justifications

- `P1`: Routine heartbeat silence can multiply model calls, exhaust fallback chains, and persist an unintended model choice for subsequent runs.
- `impact:auth-provider`: The defect misclassifies successful model candidates and changes later provider/model routing through automatic fallback persistence.

## Summary

This issue remains necessary: current main still removes a text-only `NO_REPLY` before terminal payload counting, while intentional-silence handling only recognizes genuinely empty assistant text. That mismatch can route a valid silent heartbeat through empty-response retry, candidate failure, fallback, and persisted automatic model selection, and no open PR currently owns the repair.

## What This Changes

Treat a text-only `NO_REPLY` assistant turn as a successful silent completion instead of an empty failed turn that triggers retries and model fallback.

## Best Possible Solution

Carry intentional-silence state through payload construction and terminal classification, then complete the run without retry or fallback while preserving channel suppression, media-bearing silent payloads, and safeguards for genuinely ambiguous empty turns.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main removes a text-only `NO_REPLY` payload before counting terminal payloads, while its intentional-silence predicate rejects that attempt because the assistant text is non-empty. Existing tests cover an actually empty text block rather than the explicit silence token.

## Solution Assessment

Yes in direction: preserve an explicit intentional-silence signal before filtering. The best implementation should return a closed payload-build result that distinguishes deliberate silence from a genuinely empty response instead of duplicating token checks across retry and fallback paths.
