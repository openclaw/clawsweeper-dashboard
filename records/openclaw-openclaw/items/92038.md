## Label Justifications

- `P1`: Routine heartbeat runs can make repeated paid model calls and persist an unintended fallback model, affecting subsequent agent behavior.
- `impact:auth-provider`: The bug misclassifies a successful model result as a format failure, cascades model routing, and can persist an automatic model override.

## Summary

The report remains necessary: v2026.6.5 and current main both suppress a text-only `NO_REPLY` payload before counting terminal payloads, while existing silent-empty coverage exercises an actually empty text block rather than the explicit silence token described here. The source-backed gap can still trigger empty-response retry and fallback behavior for non-harness heartbeat models, so this should stay open for a narrow core fix.

## What This Changes

Treat a text-only `NO_REPLY` assistant turn as a successful silent completion so it does not retry, fail model fallback classification, or persist an unintended fallback model.

## Best Possible Solution

Preserve an explicit deliberate-silence fact from the final assistant message before delivery payload filtering, classify it as a successful terminal outcome, and add focused tests for exact `NO_REPLY` heartbeat output plus fallback and auto-override non-persistence.

## Reproduction Assessment

Yes, source-reproducible with high confidence: an exact `NO_REPLY` payload is filtered to zero, while the existing no-retry test covers an empty text block rather than the explicit token. The reporter also provides a concrete released-version heartbeat path and transcript evidence.

## Solution Assessment

Yes in direction, but the best fix is to carry deliberate-silence state into terminal outcome classification before payload filtering rather than add another fallback-only exception.
