## Summary

Keep open. Current main has fixed much of the provider-profile/UI option path, and `openai-codex/gpt-5.4-mini` now gets `xhigh` via the Codex GPT-5.4 profile match, but the reported display inconsistency is not fully settled because status and `/think` still surface explicit configured `thinkingDefault: "xhigh"` before any model-supported-level normalization.

## What This Changes

The issue asks for consistent thinking-level display and allowed options across `/think`, `/status`, TUI, and Web UI, plus broader `xhigh` support for Codex/GitHub Copilot model refs.

## Best Possible Solution

Decide and document whether status surfaces should show configured intent, effective model-supported level, or both, then make `/think`, `/status`, TUI, and Web UI share that same provider-profile-normalized display path while keeping model capability metadata provider-owned.

## Reproduction Assessment

Yes. A high-confidence source-level repro remains for a model profile that lacks `xhigh`: set a global or per-agent `thinkingDefault: "xhigh"`, then `/think` and status read the explicit default before supported-level normalization while session option lists can omit `xhigh`.

## Solution Assessment

Unclear. The provider-profile architecture is the right direction, but the remaining status/default semantics are a product decision rather than a safe mechanical code cleanup.
