## Summary

Keep open. Current main has addressed several related provider-profile and UI-option paths, including GitHub Copilot `gpt-5.4` and OpenAI Codex startup-lazy `xhigh`, but the broader issue is still not fully resolved: configured defaults can still be displayed before provider-supported-level normalization, and the requested GitHub Copilot `gpt-5.4-mini`/`gpt-5-mini` `xhigh` support is not present in the provider allowlist.

## What This Changes

Request consistent thinking-level display across `/think`, `/status`, TUI, Web UI, and broader `xhigh` support for GitHub Copilot GPT models.

## Best Possible Solution

Define the configured-vs-effective thinking display contract, document it, then route `/think`, `/status`, TUI, and Web UI through the same provider-profile-normalized display path while only adding Copilot `xhigh` entries for verified-capable models.

## Reproduction Assessment

Yes, source-level. Configure `thinkingDefault: "xhigh"` for a model whose profile omits `xhigh` such as current `github-copilot/gpt-5-mini`, then the provider option list omits `xhigh` while `/think` and status paths can still display the configured default before support normalization.

## Solution Assessment

Unclear. The provider-profile architecture is the right direction, but changing display semantics or expanding Copilot `xhigh` support needs maintainer/provider judgment before a safe mechanical fix.
