## Summary

Keep open: the report maps to a current source-level Discord cron delivery bug, not a stale or duplicate cleanup candidate. Current main still lets cron announce TTS produce an `audioAsVoice` Discord payload, and the CLI outbound dependency proxy can resolve `discordVoice` as a nonexistent channel adapter with the exact reported unavailable-adapter error.

## What This Changes

A Discord isolated cron announce with `messages.tts.auto="always"` fails delivery after the final text is converted into a Discord voice-note payload.

## Best Possible Solution

Preserve Discord auto-TTS voice-note behavior, but prevent the CLI/cron outbound dependency layer from treating `discordVoice` as a normal channel id and add a regression test for Discord cron TTS delivery.

## Reproduction Assessment

Yes at source level: current main has a path from cron announce auto-TTS to Discord `audioAsVoice`, then to `resolveOutboundSendDep(deps, "discordVoice")`, where the CLI proxy can create the exact unavailable-adapter sender. I did not run a live Discord cron job in this read-only sweep.

## Solution Assessment

Yes, but the narrow fix is dependency resolution, not disabling Discord TTS for cron. The maintainable solution is to keep the plugin-owned Discord voice sender available while ensuring pseudo-dependency keys such as `discordVoice` are not manufactured as channel adapters.
