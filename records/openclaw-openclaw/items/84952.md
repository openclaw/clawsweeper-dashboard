Cron announce delivery to Discord text channels should keep delivering text when automatic TTS marks the payload as `audioAsVoice` and the `discordVoice` helper is unavailable.

## Best Possible Solution

Land one canonical fix that preserves Discord voice-note TTS while preventing text-channel cron delivery from failing when `discordVoice` is unavailable, then close this issue and superseded fix PRs.

## Reproduction Assessment

Yes at source level: current main still applies cron TTS, can produce a Discord `audioAsVoice` payload, and can resolve `discordVoice` through the CLI proxy into the exact unavailable-adapter error. I did not run a live Discord cron job in this read-only review.

## Solution Assessment

Yes for the direction: keep Discord voice-note TTS behavior, but stop CLI/cron outbound deps from presenting `discordVoice` as a normal channel adapter or handle that unavailable path without masking real voice failures.
