## Summary

Keep open: current main and the latest release still have a source-level path from cron announce TTS to Discord `audioAsVoice` voice delivery, where the CLI outbound dependency proxy can manufacture `discordVoice` as an unavailable channel adapter. The issue is also already paired with open closing PRs, so it should stay open until one fix is reviewed and merged.

## What This Changes

Cron announce delivery to a Discord text channel should use normal Discord text/media delivery or gracefully bypass the unavailable voice route when `messages.tts.auto="always"`.

## Best Possible Solution

Land one narrow fix that preserves real Discord voice sends while preventing cron/CLI delivery from treating `discordVoice` helper lookup as a normal channel adapter, then close this issue after the merged PR ships.

## Reproduction Assessment

Yes at source level: current main applies cron TTS, can produce `audioAsVoice`, and routes Discord `audioAsVoice` payloads through `discordVoice`, where the CLI deps proxy can create the exact unavailable-adapter error. I did not run a live Discord cron job in this read-only review.

## Solution Assessment

Yes, the maintainable fix is narrow dependency routing or equivalent Discord fallback that preserves real voice sends while stopping missing helper keys from becoming fake channel adapters. Because open closing PRs already exist, the next step is review and land one of them rather than queue another repair.
