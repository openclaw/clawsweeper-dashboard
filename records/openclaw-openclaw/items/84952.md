Keep open: current main still has the source-level path from cron announce auto-TTS to Discord `audioAsVoice` delivery, and the issue is already paired with open fixing PRs that use closing syntax. This should not be closed before one canonical fix lands.

## What This Changes

The issue reports Discord isolated cron announce delivery failing when `messages.tts.auto="always"` routes a text-channel result through the Discord voice outbound helper.

## Best Possible Solution

Land one canonical fix that keeps Discord voice-note TTS working while preventing CLI cron outbound deps from manufacturing `discordVoice` as a normal channel adapter, then close this issue from the merged PR.

## Reproduction Assessment

Yes at source level: current main routes cron announce payloads through TTS, Discord marks auto TTS as `audioAsVoice`, and Discord then resolves `discordVoice` from the CLI outbound deps proxy that can generate the exact unavailable-adapter sender. I did not run a live Discord cron job in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow outbound dependency-contract repair rather than disabling Discord TTS or changing cron announce semantics. The issue should stay open while maintainers review and land one of the linked fixing PRs or close those PRs in favor of a cleaner canonical patch.
