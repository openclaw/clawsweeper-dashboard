## Summary

Keep open: current main still has a source-reproducible Discord cron/TTS delivery path that can synthesize a fake `discordVoice` outbound sender and fail before normal Discord text delivery succeeds. The issue is also paired with open closing PRs, so it should stay open until a linked fix lands.

## What This Changes

Discord cron announce delivery fails when `messages.tts.auto="always"` causes a text-channel announcement to be routed through an unavailable Discord voice sender.

## Best Possible Solution

Land a focused linked fix that prevents the CLI/outbound dependency layer from manufacturing `discordVoice` as a normal channel adapter, while preserving Discord voice-note delivery and adding regression coverage for cron TTS announcements.

## Reproduction Assessment

Yes at source level: current main still applies TTS before cron delivery, marks Discord TTS as `audioAsVoice`, and lets the CLI deps proxy synthesize a `discordVoice` unavailable adapter. I did not run a live Discord cron job in this read-only review.

## Solution Assessment

Yes, the best direction is a narrow dependency-routing repair, not globally disabling Discord TTS or downgrading all voice-note payloads. The linked PRs should be reviewed against that boundary, then the issue can close after the chosen fix merges.
