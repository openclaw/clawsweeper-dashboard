## What This Changes

The issue asks OpenClaw to deliver Discord text-channel cron announcements successfully when `messages.tts.auto="always"` marks the output as `audioAsVoice`.

## Best Possible Solution

Land one narrow fix that preserves Discord voice-note behavior but prevents isolated cron CLI deps from manufacturing `discordVoice` as a normal channel adapter, then close this issue with regression and real-behavior proof.

## Reproduction Assessment

Yes, at source level: current main can route cron announce TTS payloads into Discord `audioAsVoice`, resolve `discordVoice` through CLI outbound deps, and manufacture the exact unavailable-adapter error. I did not run a live Discord cron job in this read-only review.

## Solution Assessment

Yes: the best fix is at the dependency/Discord outbound seam, not by disabling Discord TTS for cron. The repair should preserve valid voice sends while letting text-channel cron announces fall back or resolve helpers correctly.
