## Summary

Current main no longer has the crashing synchronous chmod path for the reported credential/config writes; the active writer now delegates to fs-safe/private store helpers that treat chmod as best-effort.

## What This Changes

The issue requests that synchronous credential/config atomic writes tolerate EPERM from chmod on Windows-backed Docker volumes, matching the async writer’s best-effort chmod behavior.

## Best Possible Solution

Keep the fs-safe-backed secrets writer on main; only reopen narrower work if a new report shows a remaining Windows-mounted Docker rename or credential-permission failure after this change.

## Reproduction Assessment

Yes. The old v2026.5.7 source directly called fs.chmodSync(tempPath, mode), so forcing chmodSync to throw EPERM reproduces the reported crash path; current main no longer follows that implementation for default credential writes.

## Solution Assessment

Yes. Moving the active credential writer to fs-safe/privateFileStore preserves directory creation and private-file handling while making chmod best-effort, which is more maintainable than adding a disconnected try/catch helper.
