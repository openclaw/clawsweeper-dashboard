## Summary

Current main no longer uses the crash-prone direct `chmodSync` implementation for the affected writer. The secrets writer now routes through fs-safe private/atomic write helpers whose pinned upstream implementation treats chmod as best-effort on filesystems without POSIX mode support; the proof commit is on current main but not in a release tag I could identify.

## What This Changes

The issue requests that synchronous atomic text writes stop crashing when `chmod` returns `EPERM` on Windows-backed Docker volumes.

## Best Possible Solution

Keep the fs-safe-backed secrets writer on main, ship it in the next release, and handle the older open helper-only PR separately as superseded or needing replacement.

## Reproduction Assessment

No current-main failing reproduction path remains: the active writer no longer performs the direct unguarded `chmodSync` described in the report. The historical failure is source-plausible for the older implementation and for v2026.5.7, but current main has moved the path to fs-safe-backed writes.

## Solution Assessment

Yes. The current-main solution is more maintainable than a local one-line catch because secrets writes now use the shared private/atomic filesystem helper contract that already centralizes best-effort chmod behavior.
