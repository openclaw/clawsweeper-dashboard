## Summary

Current main fixes the reported chmodSync crash path by routing sync secrets/auth writes through fs-safe-backed writers that treat chmod as best-effort; the fix is on main via https://github.com/openclaw/openclaw/pull/77918, but I could not prove it shipped in v2026.5.7.

## What This Changes

Handle EPERM from synchronous atomic config/auth writes on Windows-mounted Docker volumes instead of crashing on chmodSync.

## Best Possible Solution

Keep the fs-safe-backed implementation on main and close this issue; the obsolete related contributor PR can be handled separately as superseded.

## Reproduction Assessment

Yes for the original bug: the old source had an unguarded sync chmod on the exact writer, and a focused mock of chmodSync throwing EPERM would reproduce it without a Windows bind mount. Current main no longer has that failing chmod path for the reported auth/config writes.

## Solution Assessment

Yes. Moving the affected sync writers onto fs-safe-backed private/JSON stores is narrower and more maintainable than adding a one-off unused helper, because it preserves existing directory and credential-permission semantics while making chmod best-effort on unsupported filesystems.
