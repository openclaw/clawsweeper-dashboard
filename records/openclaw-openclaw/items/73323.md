## Summary

Keep this open. Current main has useful mitigations for pricing opt-out, Telegram dispatcher lifecycle, HTTP/1-only dispatchers, transport rebuilds, and diagnostics, but the broad Gateway degradation is not source-proven fixed: pricing still starts from initial runtime setup when enabled, remote catalog fetches still use the reported 60s timeout, and the channel stop-timeout path still preserves the stale running task that the discussion ties to Telegram recovery failures.

## What This Changes

The issue reports chronic cross-subsystem Gateway degradation where model-pricing fetches time out, Telegram polling/sends stall or fail, and Gateway RPCs slow dramatically on long-running Windows Node 24/25 processes.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker. Review, land, or replace the narrow pricing-startup work in #73486, land an equivalent fix for the stopChannel timeout/zombie-task behavior tracked by #71412, then validate a long-lived Windows Gateway with Telegram enabled on current main or the next release using pricing enabled and disabled, diagnostics.stability, event-loop metrics, handle/socket counts, Telegram transport logs, and timings for the slow RPC methods before splitting any remaining behavior into narrower follow-ups.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path from detailed setup steps, repeated logs, multiple corroborating Windows/Linux/macOS reports, same-host direct-network counterexamples, and the pricing-skip A/B test; this review did not run a live Windows Gateway.

## Solution Assessment

No complete solution is currently in place. The maintainable path is targeted fixes for the source-backed pricing-ordering and channel timeout recovery gaps plus maintainer-led live validation before declaring the broader Gateway degradation fixed.
