## Summary

Keep this issue open. Current main has useful mitigations and diagnostics, but the central cross-subsystem Gateway degradation is not proven fixed: pricing refresh can still start from the initial runtime-services path, remote pricing fetches still use 60s timeouts, Telegram stall recovery is reactive, and the channel stop-timeout zombie behavior remains source-backed and test-backed.

## What This Changes

The issue requests a fix for chronic Gateway runtime degradation where pricing fetches time out, Telegram polling/send calls stall or fail, and Gateway RPCs slow down after startup.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Gateway. Land or replace the pricing-startup work, fix the channel stop-timeout recovery gap, then split any still-reproducing Telegram transport or RPC/event-loop root cause into narrower follow-ups.
