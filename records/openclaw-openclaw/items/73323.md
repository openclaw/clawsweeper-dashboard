## Summary

Keep this open. Current main has useful mitigations for pricing opt-out, Telegram transport lifecycle, and diagnostics, but the broad Gateway runtime degradation is not source-proven fixed: pricing still starts from the initial runtime-services path when enabled, remote pricing fetches still use the reported 60s timeout, and the channel stop-timeout path still preserves the stale running task that the discussion connects to Telegram recovery failures.

## What This Changes

The issue requests a fix for chronic Gateway runtime degradation where pricing fetches, Telegram polling/sends, and Gateway RPC calls stall or time out in long-running Gateway processes.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Windows Gateway, land or replace the focused pricing-startup deferral, fix the channel stop-timeout recovery gap, and split any still-reproducing RPC/event-loop or Telegram transport behavior into narrower canonical follow-ups.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path from detailed setup steps, repeated logs, multiple corroborating hosts, same-host direct-network counterexamples, and the pricing-skip A/B test; this review did not run a live Windows Gateway.

## Solution Assessment

No complete solution is currently in place. The narrow maintainable path is maintainer-led live validation plus targeted fixes for the still-source-backed pricing-ordering and channel timeout recovery gaps before considering closure.
