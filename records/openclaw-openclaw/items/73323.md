## Summary

Keep this open. Current main contains useful mitigations for pricing opt-out, Telegram transport lifecycle, request timeouts, and stability diagnostics, but the broad Gateway runtime degradation is not source-proven fixed: pricing still uses the reported 60s remote fetch timeout when enabled, initial runtime services still start pricing refresh, and the channel stop-timeout path still preserves the stale running task that the discussion ties to Telegram recovery failures.

## What This Changes

The issue reports chronic Gateway runtime degradation where pricing fetches, Telegram polling/sendMessage, and Gateway RPCs stall or time out across recent releases on Windows and corroborated hosts.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Windows Gateway, land or replace the focused pricing-startup work, fix the channel stop-timeout recovery gap, and split any remaining RPC/event-loop or Telegram transport behavior into narrower follow-ups.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path from detailed setup steps, repeated logs, corroborating Windows/Linux/macOS reports, same-host direct-network counterexamples, and the pricing-skip A/B test; this review did not run a live Windows Gateway.

## Solution Assessment

No complete solution is currently in place. Current main has partial mitigations, but the narrow maintainable path is maintainer-led live validation plus targeted fixes for pricing startup, channel timeout recovery, and any still-reproducing RPC/event-loop or Telegram transport degradation.
