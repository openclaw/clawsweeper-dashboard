## Summary

Keep open. Current main has useful mitigations and diagnostics, but source and test evidence still show unresolved paths that match the broad report: pricing refresh still starts from initial runtime services when enabled, the remote pricing timeout remains 60s, and the channel stop-timeout path still preserves the stale running task state. The open pricing PR and related channel-lifecycle issue cover subsets, not the full cross-subsystem Gateway degradation.

## What This Changes

The issue reports chronic Gateway runtime degradation where pricing fetches time out, Telegram polling/send paths stall or fail, and Gateway RPCs become slow on long-running Windows Node 24-25 processes despite fast same-host direct network probes.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker. Land or replace the narrow pricing-startup work in #73486, fix or consciously reject the stopChannel timeout recovery gap tracked by #71412, then validate current main or the next release on an affected long-lived Windows Gateway before splitting any residual RPC/event-loop or Telegram transport behavior into narrower follow-ups.
