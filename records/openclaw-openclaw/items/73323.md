## Summary

Keep this issue open. Current main contains useful mitigations for pricing opt-out, model-prewarm ordering, Telegram transport lifecycle, request timeouts, and event-loop diagnostics, but the broad cross-subsystem Gateway degradation is not proven fixed: pricing still starts from initial runtime services when enabled, the reported 60s pricing timeout remains, and the channel stop-timeout path still preserves the stale running task state tied to Telegram recovery failures.

## What This Changes

The issue reports chronic Gateway runtime degradation across pricing fetches, Telegram polling/sendMessage, and slow RPCs on Windows 11 with Node 24/25, with same-host standalone network probes remaining fast.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker. Review, land, or replace #73486 for the pricing-startup subset, fix or supersede the #71412 stopChannel timeout recovery gap, then run a long-lived Windows Gateway with Telegram enabled on current main or the next release and split any still-reproducing RPC/event-loop or Telegram transport behavior into narrower canonical follow-ups.

Yes. The issue has a high-confidence reproduction path from detailed setup steps, repeated logs, multiple corroborating hosts, same-host direct-network counterexamples, and the pricing-skip A/B test; this review did not run a live Windows Gateway.

## Solution Assessment

No complete solution is currently in place. The maintainable path is targeted fixes for the pricing ordering and channel timeout gaps, followed by live validation that can separate any remaining Telegram transport or Gateway event-loop/RPC root cause.
