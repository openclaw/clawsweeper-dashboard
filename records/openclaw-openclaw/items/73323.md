## Summary

Keep this open. Current main has useful mitigations for pricing opt-out, Telegram transport cleanup, request timeouts, and responsive model-list RPCs, but I did not find high-confidence proof that the broad chronic Gateway runtime degradation is fixed; the remaining symptoms still span pricing startup, Telegram recovery, channel stop-timeout state, and Gateway RPC/event-loop behavior.

## What This Changes

The issue reports chronic long-running Gateway degradation where pricing fetches time out, Telegram polling and outbound Bot API calls stall or fail, and Gateway RPCs become slow on Windows Node installs despite fast same-host standalone network probes.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Windows Gateway, land or replace the pricing-startup subset in #73486, fix or consciously split the stopChannel timeout recovery gap tracked by #71412, and then move any still-reproducing Telegram transport or RPC/event-loop root cause into narrower follow-ups.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path from detailed setup steps, repeated logs, multiple independent corroborating reports, same-host direct-network counterexamples, and a pricing-skip A/B test; this review did not run a live Windows Gateway.

## Solution Assessment

No complete solution is currently in place. Current main adds useful partial mitigations, but the narrowest maintainable path is maintainer live validation plus targeted fixes for the still-source-backed pricing-ordering and channel timeout recovery gaps before closing the broad tracker.
