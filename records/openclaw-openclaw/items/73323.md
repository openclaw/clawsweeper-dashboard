## Summary

Keep this open. Current main has several useful mitigations and diagnostics, but the broad reported failure is not proven fixed: pricing still starts from the initial runtime-services path when enabled, the remote pricing timeout remains 60s, and the channel stop-timeout zombie behavior implicated by the Telegram recovery reports is still asserted by source and tests.

## What This Changes

The issue requests a fix for chronic Gateway runtime degradation across pricing fetches, Telegram polling/sends, and slow Gateway RPCs on long-running Windows/Node installs, with corroborating reports across platforms.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Windows Gateway, land or replace the pricing-startup deferral work, fix the channel stop-timeout zombie recovery gap, and then split any still-reproducing Telegram transport or RPC/event-loop root cause into narrower follow-ups.
