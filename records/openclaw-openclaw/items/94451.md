## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Require inline data to be an object:** `packages/ard-core/src/validation.ts:103-104`
  - body: The ARD schema defines inline `data` as the embedded JSON object, but this accepts any defined value and returns it as a validated entry. A string or number would be treated as a valid descriptor, so later ingestion/search would trust malformed catalog data.
  - confidence: 0.93
- **[P2] Accept and preserve spec trust manifests:** `packages/ard-core/src/validation.ts:286-302`
  - body: Spec trust manifests require `identity` and use object-shaped attestation/provenance fields, but this validator only handles string arrays and drops identity/provenance/signature metadata. Spec-valid catalogs can therefore fail validation or lose their trust envelope.
  - confidence: 0.9
- **[P2] Preserve ARD host identity fields:** `packages/ard-core/src/validation.ts:322-330`
  - body: The upstream host object requires `displayName` when present and supports identity, documentation, logo, and host trust fields, but this validator accepts a host without `displayName` and drops those spec fields. Publisher identity metadata would disappear from a catalog that was just validated.
  - confidence: 0.84
- **[P2] Wire dist exports into the shared build:** `packages/ard-core/package.json:9-11`
  - body: This package declares `main` and exports under `./dist`, but the PR does not add `ard-core` to the repo's shared tsdown package output roots/config. A normal OpenClaw build will not emit these artifacts, so the first consumer can resolve missing package files unless it knows to run a separate filtered build.
  - confidence: 0.88
