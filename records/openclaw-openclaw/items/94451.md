## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Require inline data to be an object:** `packages/ard-core/src/validation.ts:103-104`
  - body: The ARD schema defines inline `data` as the embedded JSON object, but this validator accepts any defined value and then normalizes it into the catalog entry. That means `data: "not an object"` or a number is marked valid, so later ingestion/search would treat malformed resource descriptors as validated ARD entries.
  - confidence: 0.93
- **[P2] Accept spec trustManifest objects:** `packages/ard-core/src/validation.ts:286-302`
  - body: The upstream ARD schema requires a trust manifest `identity` and models `attestations`/`provenance` as arrays of objects, but this validator only accepts string arrays and drops the identity/signature fields. A spec-valid catalog using the documented trust metadata will fail validation or lose the trust envelope, which breaks the shared contract this package is meant to establish.
  - confidence: 0.9
- **[P2] Wire dist exports into the shared build:** `packages/ard-core/package.json:8-12`
  - body: This package declares `main`/exports under `./dist`, but the PR does not add `ard-core` to the repo's shared tsdown package config or output-root list. A normal OpenClaw build will not emit `packages/ard-core/dist`, so the first consumer of `@openclaw/ard-core` can resolve missing artifacts unless it knows to run a separate filter build.
  - confidence: 0.86
