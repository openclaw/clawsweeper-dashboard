## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Align MCP media types with the pinned ARD draft:** `packages/ard-core/src/types.ts:10-11`
  - body: The inspected upstream ARD draft uses `application/mcp-server+json` for MCP entries and warns intermediaries not to strictly verify pending media-type registration. This patch instead makes `application/mcp-server-card+json` the standard value and warns on `application/mcp-server+json`, so spec-valid MCP entries become legacy or nonstandard to OpenClaw. Please either pin evidence for the `-card` spelling or make the current draft value the standard contract.
  - confidence: 0.9
- **[P2] Use the ARD query envelope for search requests:** `packages/ard-core/src/types.ts:83-88`
  - body: The upstream draft defines Search and Explore requests around a nested `query` object with `text` and `filter`, while this exported type and validator use `query?: string` plus root-level `filters`. A future registry or gateway consumer built on this package would reject or ignore spec-shaped requests, so the exported contract should match the pinned draft or be renamed as an OpenClaw-local helper.
  - confidence: 0.86
- **[P2] Preserve spec-shaped trust manifests:** `packages/ard-core/src/validation.ts:286-303`
  - body: The ARD draft's `trustManifest` carries identity fields, attestation objects, provenance, and a signature, but `optionalTrustManifest` only accepts string arrays plus `transparencyLog` and returns that narrowed object. That rejects or drops spec-valid trust evidence even though the new docs say trust manifests are preserved as metadata, so model the draft shape or keep this field opaque until the spec settles.
  - confidence: 0.86
