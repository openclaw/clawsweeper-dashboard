Full review comments:

- **[P1] Keep ARD out of core packaging unless intended:** `extensions/ard/package.json:31-33`
  - body: This package declares `publishToClawHub`/`publishToNpm`, and the official catalog advertises `clawhub:@openclaw/ard-plugin`, but the root package `files` list has no `!dist/extensions/ard/**` exclude. Current inventory logic treats unexcluded extension dirs as core, so this would silently make ARD part of the core package while documenting it as an external install.
  - confidence: 0.9
- **[P2] Validate inline ARD data as an object:** `extensions/ard/src/validation.ts:109-110`
  - body: The upstream ARD catalog schema requires inline `data` to be an object, but this validator only checks that `data` is not `undefined`. A descriptor with `data: "not an object"` would pass and be preserved for later ingestion/search.
  - confidence: 0.92
- **[P2] Reject unsupported top-level manifest metadata:** `extensions/ard/src/validation.ts:187-199`
  - body: The upstream catalog manifest only allows `specVersion`, `host`, and `entries` at the top level, but this path accepts and returns top-level `metadata`. That lets schema-invalid manifests become valid OpenClaw discovery input.
  - confidence: 0.88
- **[P2] Regenerate the plugin inventory for ARD:** `extensions/ard/openclaw.plugin.json:2`
  - body: Adding `extensions/ard/package.json` plus `openclaw.plugin.json` creates a new plugin source entry, and the inventory generator validates that every source entry is represented. Without regenerated `docs/plugins/plugin-inventory.md`, `docs/plugins/reference.md`, and `docs/plugins/reference/ard.md`, the generated docs/check output stays stale.
  - confidence: 0.86
