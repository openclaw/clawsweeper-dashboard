## Summary

Keep this PR open, but it is not merge-ready: the default-agent direction is useful, yet the branch lets configured-main or empty-config scans fall through to ambient process state, and the contributor proof gate is still unmet.

## What This Changes

The PR updates secrets auth/model path discovery helpers and adds unit tests so configured default agent IDs are used instead of hard-coded `main` seeds.

## Best Possible Solution

Keep the configured-default-agent direction, but resolve fallback agent paths against the caller-provided `stateDir`/env, preserve configured custom `agentDir` support, and add negative tests proving no ambient store is scanned.

## Reproduction Assessment

Yes, source-level. With empty config or a configured `main` agent and a caller-supplied stateDir that differs from `process.env`, the patched loop falls through to `resolveAgentDir(config, agentId)` and can add an ambient process-state path.

## Solution Assessment

No as written. The narrow maintainable fix is to use configured default-agent identity while keeping path resolution owned by the explicit scan stateDir/env boundary, not ambient `process.env`.
