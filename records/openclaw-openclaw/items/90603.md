## Summary

Keep open: the direction matches the linked default-agent issue, but the patch is not merge-ready because the removed `main` branches can route scans through ambient `process.env` instead of the caller-provided state directory, and the external real-behavior proof gate is still unsatisfied.

## What This Changes

This PR changes two secrets storage discovery helpers to seed from `resolveDefaultAgentId(config)`, removes `main`-specific loop branches, and adds default-agent path unit tests.

## Best Possible Solution

Keep the configured-default-agent direction, but resolve fallback/default paths against the scanner's explicit `stateDir`/`env`, preserve configured custom `agentDir` support, and add negative tests proving no ambient state directory is scanned.

## Reproduction Assessment

Yes, source-level. With an empty config or configured `main` agent and a caller-provided `stateDir` that differs from `process.env.OPENCLAW_STATE_DIR`, the patched loop can add the process-global `main` agent store through `resolveAgentDir(config, agentId)`.

## Solution Assessment

No as written. The fix should use the configured default agent identity while preserving the explicit secrets scan stateDir/env boundary instead of falling back to ambient process state.
