## Summary

Keep open: current main still routes durable Telegram final replies through `sendMessageTelegram()`, which returns Telegram message ids and records activity but does not emit a self-contained journal success line. The related media issue and open PR cover legacy/media logging only, not the durable text success gap reported here.

## What This Changes

Request safe journal-facing Telegram outbound success logs for durable final replies after Telegram accepts text or media sends.

## Best Possible Solution

Add safe info-level Telegram outbound success logging in `send.ts` after text and media message-id resolution, with tests proving ids are present and message bodies, captions, URLs, and file paths are absent.

## Reproduction Assessment

Yes at source level. A mocked successful `sendMessageTelegram()` call can return a Telegram `message_id` while current main records the id/activity and emits no info-level success line; live `journalctl` proof remains needed after a patch.

## Solution Assessment

Yes. A small logger in `sendMessageTelegram()` after platform message-id resolution is narrower than relying on generic diagnostics, because the requested operator proof needs a compact journal line with the Telegram message id.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83196 by adding safe, journal-facing Telegram outbound success logging in `extensions/telegram/src/send.ts` for the modern durable path. Log only operational identifiers after Telegram returns a message id: account id, chat id, message id, operation/media kind, thread id, reply-to id, silent flag, and chunk count when available. Do not log message text, captions, quote text, sender names, chat titles, media URLs, local file paths, bot tokens, API roots, or raw request params. Add focused coverage in `extensions/telegram/src/send.test.ts` for successful text and media sends, including negative assertions for body text, caption, media URL, and filename/path. Be aware of https://github.com/openclaw/openclaw/issues/68770 and https://github.com/openclaw/openclaw/pull/68836, which cover legacy/media logging; avoid unrelated churn in `delivery.replies.ts` unless deliberately sharing a helper without conflict. Add a concise `CHANGELOG.md` fix entry if this is treated as operator-visible logging behavior.
