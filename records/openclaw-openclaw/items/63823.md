## Summary

Keep open. Current main still lacks the MiniMax manifest auth-alias mapping while the runtime still exposes the CN aliases and auth lookup is manifest-backed; the linked open PR #72798 remains the active implementation path, so closing before that patch or an equivalent fix lands would be premature.

## What This Changes

The issue asks for `extensions/minimax/openclaw.plugin.json` to declare `providerAuthAliases` mapping `minimax-cn` to `minimax` and `minimax-portal-cn` to `minimax-portal`.

## Best Possible Solution

Land #72798 or apply an equivalent maintainer patch that adds the MiniMax manifest aliases with focused regression coverage; the generic auth resolver should stay manifest-backed for the existing trust boundary.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `resolveProviderIdForAuth("minimax-cn")` has no MiniMax manifest alias to read and falls back to `minimax-cn`, while credential lookup then searches that unresolved provider id before profiles/env/config for the base MiniMax providers.

## Solution Assessment

Yes. The manifest-level alias map is the narrowest maintainable fix because the core resolver and docs already define `providerAuthAliases` as the contract for provider variants sharing auth.
