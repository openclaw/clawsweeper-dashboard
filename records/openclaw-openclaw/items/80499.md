## Summary

Keep open: this is a focused and useful Claude ACP bug fix, but the current patch still has blocking correctness gaps in the same config-control path it changes. It needs to normalize the advertised `effort` key and preserve `xhigh` separately from `max` before merge.

## What This Changes

The PR adds Claude ACP command detection and config filtering/remapping in the ACPX runtime, expands ACPX runtime tests, and updates two existing test assertions.

## Best Possible Solution

Keep the Claude ACP-specific filtering direction, but normalize all thinking aliases on both generic and advertised `effort` flows while preserving `xhigh` and `max` as distinct dependency-supported values.

## Reproduction Assessment

Yes. Current main's config-control path plus the published Claude ACP validation code show a high-confidence source reproduction for unsupported or invalid config forwarding, although I did not run the live ACP flow in this read-only review.

## Solution Assessment

No, not yet. The PR is aimed at the right runtime boundary, but the implementation misses the advertised `effort` key and collapses `xhigh` to `max`, so a narrow follow-up patch is needed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve xhigh instead of mapping it to max:** `extensions/acpx/src/runtime.ts:440`
  - body: Claude's SDK and ACP adapter treat `xhigh` and `max` as distinct effort values, and the adapter validates against the active model's advertised values. Mapping `xhigh`/`x-high` aliases to `max` can reject sessions on models that advertise `xhigh` but not `max`, or silently request a stronger effort than the user selected. Please map xhigh aliases to `xhigh` and reserve `max` for an explicit max input.
  - confidence: 0.9
- **[P2] Normalize advertised effort values for Claude ACP:** `extensions/acpx/src/runtime.ts:1008`
  - body: The control plane resolves stored `thinking` to the adapter-advertised `effort` key when Claude ACP exposes it, and explicit `/config effort ...` also reaches this branch as `key === "effort"`. This condition skips `effort`, so values like `x-high` bypass the new normalizer and are forwarded unchanged to an adapter that validates option values. Include `effort` in this path and add a regression test for the advertised-key flow.
  - confidence: 0.94
