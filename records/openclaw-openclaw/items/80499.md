## Summary

Keep this PR open: it targets a real Claude ACP config-control bug, the supplied live-output proof is sufficient, but the current patch still has blocking correctness issues in Claude effort normalization.

## What This Changes

The PR adds Claude ACP config filtering/remapping in the ACPX runtime, expands ACPX runtime tests, and adjusts two existing test assertions.

## Best Possible Solution

Keep the Claude ACP-specific filtering direction, but implement it on top of current main while normalizing both generic thinking aliases and advertised `effort` keys without collapsing `xhigh` into `max`.

## Reproduction Assessment

Yes. Current main's ACPX runtime and control-plane source show that Claude ACP can still receive unsupported model/thinking config values, and the published Claude ACP adapter rejects unknown config ids and invalid option values.

## Solution Assessment

No, not yet. The PR is aimed at the right runtime boundary, but it misses the advertised `effort` key path and maps `xhigh` aliases to `max`, so a narrow correction is needed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve xhigh as a Claude effort value:** `extensions/acpx/src/runtime.ts:436`
  - body: Claude's SDK and adapter treat `xhigh` and `max` as distinct supported effort values. Mapping `xhigh`/`x-high` aliases to `max` can reject models that advertise `xhigh` but not `max`, or silently request a stronger effort than the user selected.
  - confidence: 0.9
- **[P2] Normalize effort config keys for Claude ACP:** `extensions/acpx/src/runtime.ts:1008`
  - body: The control plane can resolve stored `thinking` to the adapter-advertised `effort` key, and explicit `/config effort ...` reaches this branch as `key === "effort"`. Include `effort` in this condition so aliases like `x-high` do not bypass normalization and get forwarded unchanged.
  - confidence: 0.94
