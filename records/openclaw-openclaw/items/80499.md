## Summary

Keep open: the PR is aimed at a real Claude ACP config-control bug, but the branch still lets advertised `effort` keys bypass its Claude normalization and it lacks required after-fix real behavior proof.

## What This Changes

The PR adds Claude ACP command detection plus ACPX runtime filters and mappings for timeout, model, and thinking config controls with focused runtime tests.

## Best Possible Solution

Land one focused ACPX runtime fix that filters unsupported Claude timeout/Codex-model values, normalizes all thinking aliases including advertised `effort`, preserves `xhigh` as distinct from `max`, and includes regression tests plus redacted live Claude ACP proof.

## Reproduction Assessment

Yes, from source inspection rather than a live run in this read-only review. Current main can send advertised `effort` keys, while the PR branch only normalizes other thinking-key spellings, so `effort=x-high` falls through to Claude ACP validation unchanged.

## Solution Assessment

No. The adapter-level filtering/mapping is the right place to fix this, but the proposed implementation needs to normalize `effort` itself and preserve `xhigh` instead of converting every x-high alias to `max`.

## Review Findings

  - body: The control plane rewrites persisted and explicit thinking controls to the backend's advertised key, so Claude sessions can call this runtime with `key === "effort"`.   - confidence: 0.84
