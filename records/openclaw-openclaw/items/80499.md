## Summary

Keep open. The Claude ACP direction is useful and the supplied live proof is sufficient, but the patch still has blocking correctness gaps around timeout handling, advertised `effort` normalization, and preserving Claude's distinct `xhigh` effort value.

## What This Changes

The PR adds Claude ACP command detection and ACPX config filtering/remapping, expands ACPX runtime tests, and updates two existing test assertions.

## Best Possible Solution

Keep the Claude ACP-specific filtering direction, but make it interoperate with current manager capability validation, normalize both generic and advertised effort keys, and preserve dependency-supported effort values exactly.

## Reproduction Assessment

Yes. Current main's runtime-control path plus the published Claude ACP validation code show a high-confidence source reproduction for invalid or unsupported config forwarding, although I did not run the live ACP flow in this read-only review.

## Solution Assessment

No, not yet. The PR is aimed at the right ACPX boundary, but timeout handling also has to happen before manager advertised-key rejection, and effort aliases must be normalized without collapsing `xhigh` to `max`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Filter Claude timeout before manager validation:** `extensions/acpx/src/runtime.ts:1000-1001`
  - body: `runTimeoutSeconds` is persisted as `runtimeOptions.timeoutSeconds`, and current main's `applyManagerRuntimeControls` builds a `timeout` config pair then rejects it when Claude advertises only keys like `mode`, `model`, and `effort`, before `AcpxRuntime.setConfigOption` is called. This runtime-level skip is therefore too late for the real pre-turn path, so `sessions_spawn(...runTimeoutSeconds...)` can still fail instead of treating the timeout as OpenClaw-owned.
  - confidence: 0.86
- **[P2] Normalize advertised effort values for Claude ACP:** `extensions/acpx/src/runtime.ts:1008`
  - body: Current main resolves persisted `thinking` through advertised config keys, so Claude sessions can call this branch with `key === "effort"`; explicit `/config effort ...` does the same. Because `effort` is not included here, aliases such as `x-high` bypass the normalizer and are forwarded to an adapter that validates option values. Include `effort` in the normalized Claude path and add a regression test for that advertised-key flow.
  - confidence: 0.94
- **[P2] Preserve xhigh instead of mapping it to max:** `extensions/acpx/src/runtime.ts:436`
  - body: Claude's SDK and ACP adapter treat `xhigh` and `max` as distinct effort values, and the adapter validates against the active model's advertised values. Mapping `xhigh`/`x-high` aliases to `max` can reject sessions on models that advertise `xhigh` but not `max`, or silently request a stronger effort than the user selected. Map xhigh aliases to `xhigh` and reserve `max` for an explicit max input.
  - confidence: 0.9
