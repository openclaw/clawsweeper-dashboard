## Summary

Keep open. The PR targets a real Claude ACP config-control failure, but the current patch still mishandles dependency-supported effort values and misses the advertised `effort` key path, so it should not merge as-is.

## What This Changes

The PR adds Claude ACP command detection and runtime config filtering/remapping in the ACPX plugin, expands ACPX runtime coverage, and updates two existing test assertions.

## Best Possible Solution

Keep the Claude ACP-specific ACPX guard as a narrow complement to current advertised-key mapping, but preserve `xhigh`, normalize `effort` as well as thinking aliases, and add regression coverage for both paths.

## Reproduction Assessment

Yes, by source inspection. Current main and the Claude ACP dependency show the relevant path: OpenClaw can deliver config controls to an adapter that rejects unknown ids or invalid values, and the PR body supplies after-fix live output, although I did not run a live ACP session here.

## Solution Assessment

No, not yet. The runtime boundary is the right place for the PR's Claude-specific filtering, but the implementation should preserve dependency-supported `xhigh` and normalize advertised `effort` values before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve xhigh instead of mapping it to max:** `extensions/acpx/src/runtime.ts:436`
  - body: Claude's SDK and ACP adapter treat `xhigh` and `max` as distinct effort values, and the adapter validates against the active model's advertised values. Mapping `xhigh`/`x-high` aliases to `max` can reject sessions on models that advertise `xhigh` but not `max`, or silently request a stronger effort than the user selected. Map xhigh aliases to `xhigh` and reserve `max` for explicit max input.
  - confidence: 0.9
- **[P2] Normalize advertised effort values for Claude ACP:** `extensions/acpx/src/runtime.ts:1008`
  - body: The control plane resolves stored `thinking` to the adapter-advertised `effort` key, and explicit `/config effort ...` reaches this branch as `key === "effort"`. Because this condition skips `effort`, values like `x-high` bypass the normalizer and are forwarded unchanged to an adapter that validates option values. Include `effort` in this path and add a regression test for the advertised-key flow.
  - confidence: 0.94
