## Summary

Keep open: the fix is aimed at the right ACPX runtime boundary and has sufficient live proof for the basic Claude ACP crash, but latest head still mishandles Claude effort values by collapsing `xhigh` to `max` and by skipping normalization when the wire key is already `effort`.

## What This Changes

The PR adds Claude ACP command detection and filters or remaps model, timeout, and thinking config controls in the ACPX runtime, with ACPX/control-plane/media test updates.

## Best Possible Solution

Keep the Claude ACP-specific runtime filtering direction, but repair effort normalization so dependency-supported values are preserved and both generic and advertised-key flows are covered.

## Reproduction Assessment

Yes. Current-main source plus the locked Claude ACP adapter show a high-confidence source reproduction for unsupported or invalid config forwarding, though I did not run the live ACP flow in this read-only pass.

## Solution Assessment

No, not yet. The PR uses a reasonable ACPX runtime boundary, but the latest diff must preserve `xhigh` and normalize the advertised `effort` key before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve xhigh effort instead of mapping it to max:** `extensions/acpx/src/runtime.ts:436`
  - body: Claude's SDK and ACP adapter treat `xhigh` and `max` as distinct effort values, and the adapter validates against the active model's advertised values. Mapping `xhigh`/`x-high` aliases to `max` can reject sessions on models that advertise `xhigh` but not `max`, or silently request a stronger effort than the user selected. Please return `xhigh` for xhigh aliases and reserve `max` for explicit max input.
  - confidence: 0.9
- **[P2] Normalize advertised effort values for Claude ACP:** `extensions/acpx/src/runtime.ts:1008`
  - body: The control plane resolves stored `thinking` to the adapter-advertised `effort` key, and explicit effort config updates can also arrive here as `key === "effort"`. This condition skips `effort`, so values like `x-high` bypass the new normalizer and are forwarded unchanged to an adapter that validates option values. Include `effort` in this path and add a regression test for the advertised-key flow.
  - confidence: 0.94
