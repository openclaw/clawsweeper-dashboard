## Summary

Keep open: this is a useful Claude ACP bug-fix PR, but it still has two correctness blockers in the Claude effort normalization path and is currently merge-conflicting with main.

## What This Changes

The PR adds Claude ACP command detection and config filtering/remapping in the ACPX runtime, expands ACPX runtime tests, and updates two existing test assertions.

## Best Possible Solution

Keep the Claude-specific filtering direction, but normalize all Claude effort aliases on both generic and advertised `effort` flows while preserving dependency-supported `xhigh` and `max` as distinct values.

## Reproduction Assessment

Yes. Current main plus the published Claude ACP package contract provide a high-confidence source reproduction for unsupported or invalid config forwarding; I did not run the live ACP flow in this read-only review.

## Solution Assessment

No, not yet. The PR is aimed at the right ACPX runtime boundary, but it skips the advertised `effort` key and collapses `xhigh` to `max`, so a narrow runtime/test update is needed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve xhigh as an effort value:** `extensions/acpx/src/runtime.ts:436`
  - body: Claude's SDK and ACP adapter treat `xhigh` and `max` as distinct effort values, and the adapter validates against the active model's advertised values. Mapping `xhigh`/`x-high` aliases to `max` can reject sessions on models that advertise `xhigh` but not `max`, or silently request a stronger effort than the user selected.
  - confidence: 0.9
- **[P2] Normalize the advertised effort key:** `extensions/acpx/src/runtime.ts:1008`
  - body: The control plane can call this runtime with `key === "effort"` when Claude ACP advertises that config option or when a user sets native effort directly. This condition skips `effort`, so values like `x-high` bypass the new normalizer and are forwarded unchanged to an adapter that validates option values.
  - confidence: 0.94
