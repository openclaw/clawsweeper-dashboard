## Summary

Keep open: this is a real Claude ACP config-control bug with sufficient live proof, but the latest diff still mishandles Claude effort values by collapsing xhigh aliases to max and letting advertised effort values bypass normalization.

## What This Changes

The PR adds Claude ACP command detection and config filtering/remapping in the ACPX plugin runtime, expands runtime tests, and adjusts two existing test assertions.

## Best Possible Solution

Keep the Claude-specific runtime guard, but align it with Claude's dependency-supported effort values and normalize both generic and advertised effort-key flows before merge.

## Reproduction Assessment

Yes. Source inspection shows current main forwards non-Codex ACP config values through AcpxRuntime while Claude ACP 0.33.1 rejects unknown config ids and invalid option values; I did not run a live ACP session in this read-only review.

## Solution Assessment

No, not yet. The runtime-boundary direction is right, but the patch must preserve xhigh and normalize advertised effort writes before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve xhigh instead of mapping it to max:** `extensions/acpx/src/runtime.ts:436`
  - body: Claude's SDK and ACP adapter treat xhigh and max as distinct effort values, and the adapter validates against the active model's advertised values. Mapping xhigh/x-high aliases to max can reject sessions on models that advertise xhigh but not max, or silently request a stronger effort than the user selected.
  - confidence: 0.9
- **[P2] Normalize advertised effort values for Claude ACP:** `extensions/acpx/src/runtime.ts:1008`
  - body: The control plane can resolve stored thinking to the adapter-advertised effort key, and explicit effort writes also reach this branch as key === "effort". Include effort in the Claude normalization path so values like x-high are converted before the adapter validates them.
  - confidence: 0.94
