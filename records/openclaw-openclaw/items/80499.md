## Summary

Keep open. The PR targets a real Claude ACP config-control crash and has sufficient live proof, but the current head still has two blocking normalization bugs: `xhigh` is collapsed to `max`, and advertised `effort` values bypass the new Claude normalizer.

## What This Changes

The PR adds Claude ACP command detection and config filtering/remapping in the ACPX runtime, plus related ACPX/control-plane/media test updates.

## Best Possible Solution

Keep the Claude ACP-specific filtering direction, but normalize both generic thinking aliases and advertised `effort` inputs while preserving dependency-supported `xhigh` and `max` as distinct values.

## Reproduction Assessment

Yes. Current main delegates non-Codex ACP config unchanged, the control plane can emit `effort` for stored thinking controls, and the published Claude ACP adapter rejects unknown config ids or invalid option values.

## Solution Assessment

No, not yet. The runtime boundary is the right place for this fix, but the PR must preserve `xhigh` and normalize `key === "effort"` before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve xhigh instead of mapping it to max:** `extensions/acpx/src/runtime.ts:436`
  - body: Claude's SDK and ACP adapter treat `xhigh` and `max` as distinct effort values. Mapping `xhigh`/`x-high` aliases to `max` can reject sessions on models that advertise `xhigh` but not `max`, or silently request a stronger effort than the user selected.
  - confidence: 0.9
- **[P2] Normalize advertised effort values for Claude ACP:** `extensions/acpx/src/runtime.ts:1008`
  - body: When Claude ACP advertises `effort`, persisted thinking controls and explicit `/config effort ...` calls can reach this branch as `key === "effort"`. Because that key is skipped here, values like `x-high` are forwarded unchanged to an adapter that validates option values.
  - confidence: 0.94
