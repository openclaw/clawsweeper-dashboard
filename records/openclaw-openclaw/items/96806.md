## Summary

Keep open: the core trajectory change is useful and real-behavior proof is sufficient for that path, but the same event-level truncation/usage-loss invariant remains in the Codex app-server trajectory recorder, so the linked accounting bug is only partially fixed.

## What This Changes

The PR modifies `src/trajectory/runtime.ts` to copy `data.usage` into the event-level truncation stub and fall back to the previous full-stub behavior if that preserved stub still exceeds the byte limit.

## Best Possible Solution

Preserve sanitized `usage` during event-level truncation in both trajectory recorders and add focused regression tests for oversized `model.completed` events in the core and Codex paths.

## Reproduction Assessment

Yes. On current main, a `model.completed` event with `data.usage` plus enough sanitized payload to exceed 262144 bytes follows `truncateOversizedTrajectoryEvent` and replaces `data` with only the truncation stub; the Codex sibling has the same source-reproducible path.

## Solution Assessment

No. The core change is the right narrow shape, but it is not the best complete fix until the sibling Codex trajectory recorder preserves the same accounting-critical field and both paths have regression coverage.
