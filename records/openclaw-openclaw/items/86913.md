## Summary

Keep open: the new Gateway pending-state RPC is useful, but the proposed response drops the restart reason during the preparing/active-deferral window, so it does not fully expose the pending state it promises.

## What This Changes

The PR adds an advertised `gateway.restart.pending` operator-read RPC that returns restart scheduler state, registers its lazy handler, and adds handler/runtime tests.

## Best Possible Solution

Carry the scheduled restart reason through preparing and active-deferral state before exposing the RPC, with focused coverage for scheduled, deferred, and emitted restart states.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the defect: the scheduler clears `pendingRestartReason` before active deferral, while the new response reads `reason` only from `pendingRestartReason` or `emittedRestartReason`.

## Solution Assessment

No. The API direction is narrow, but the implementation should preserve a preparing/deferral reason until the restart emits or is handled.
