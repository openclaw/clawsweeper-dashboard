## Summary

Keep this open: current main has the shared reply sanitizer and delivery normalization path, but it does not yet strip the reported current-message wrapper or bare `(no output)` placeholder before user-facing delivery.

## What This Changes

Suppress leaked current-message scaffolding and bare `(no output)` placeholders from user-facing reply delivery, skipping replies when no visible text remains.

## Best Possible Solution

Add a narrow shared sanitizer/normalizer guard for these exact internal-only strings, with regression tests that preserve real visible reply text and skip placeholder-only payloads.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main reproduction path: nonempty `(no output)` text reaches `sanitizeUserFacingText`, remains nonempty, and is therefore deliverable. The current-message marker is documented as injected context, but the sanitizer has no exact removal for that wrapper.

## Solution Assessment

Yes, the narrowest maintainable fix is at the shared reply sanitizer/normalizer boundary, matching the prior `[tool calls omitted]` fix. The broader text-between-tool-calls policy should remain with #25592 rather than expanding this bug fix.
