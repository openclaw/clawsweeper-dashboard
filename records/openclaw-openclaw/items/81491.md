## Summary

Keep open: current main still lacks the requested Slack process-restart escalation, and the underlying channel-restart failure remains only partially addressed by the merged recovery-stop work. The report overlaps the open generic channel lifecycle issue, but the requested self-SIGTERM behavior is a process policy/config decision rather than a safe cleanup close.

## What This Changes

The issue requests a configurable Slack reconnect threshold that self-terminates the gateway process after repeated failed reconnects so an external supervisor can recover a silent Slack outage.

## Best Possible Solution

Fix the generic gateway channel lifecycle so an unhealthy account can be force-retired or fenced without old-task side effects, then add gateway-wide supervisor escalation only if maintainers explicitly choose that recovery policy.

## Reproduction Assessment

No, not for the exact Slack Socket Mode outage: I did not run a live Slack transport failure and the issue log lacks the stop-timeout line. Source inspection does show a clear overlapping current-main path where a health-monitor stop timeout leaves the old task registered and startChannel no-ops until that task settles.

## Solution Assessment

Unclear: Slack-specific self-SIGTERM is not obviously the narrowest maintainable fix. The safer first step is the generic channel lifecycle repair tracked by https://github.com/openclaw/openclaw/issues/71412, with process self-termination considered as a separate gateway-wide policy.
