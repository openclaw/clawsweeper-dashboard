## Summary

Keep open: current main still lacks the requested self-SIGTERM/processRestartAfter behavior, and adding it would be a new gateway lifecycle policy that overlaps active Slack reconnect-cap and channel restart recovery work.

## What This Changes

The issue requests a configurable Slack/Gateway escalation that SIGTERMs the gateway after repeated failed Slack reconnect or restart attempts so a supervisor restarts the process.

## Best Possible Solution

Review or replace the narrow Slack reconnect-cap fix and generic channel lifecycle recovery first, then decide whether a documented gateway-wide supervisor escalation is acceptable for bounded in-process recovery failure.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection still shows the finite Slack reconnect cap, in-process-only health-monitor restart, and no `processRestartAfter` or SIGTERM path, while related threads provide field logs for this outage class.
