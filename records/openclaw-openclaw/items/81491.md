## Summary

Keep open: current main still lacks the requested Slack `processRestartAfter`/self-SIGTERM behavior, while related open Slack reconnect and channel lifecycle items cover only narrower recovery paths. The issue is real enough to keep, but solving it as written needs maintainer lifecycle policy judgment because process self-termination would restart unrelated gateway work.

## What This Changes

The issue asks for repeated failed Slack reconnects to escalate to a configurable gateway self-SIGTERM so a supervisor can respawn the process.

## Best Possible Solution

Consolidate the Slack reconnect-cap, stop-timeout, and no-event liveness fixes first, then decide whether a documented gateway supervisor escalation knob is still needed as an operator escape hatch.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the missing requested behavior and the relevant failure preconditions: finite Slack retries, in-process health restarts, and no process-level escalation path.

## Solution Assessment

Unclear. A Slack-specific self-SIGTERM knob may be a useful defense-in-depth escape hatch, but the safer maintainable path is to fix or consolidate the Slack reconnect cap and channel lifecycle recovery issues before adding process self-termination policy.
