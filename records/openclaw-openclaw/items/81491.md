## Summary

Keep open: current main and the latest shipped release still lack the requested self-SIGTERM/processRestartAfter behavior, and the remaining work is a gateway lifecycle/config policy that overlaps unresolved Slack reconnect-cap and channel stop-timeout recovery work.

## What This Changes

The issue requests a configurable Slack reconnect escalation that SIGTERMs the gateway after repeated failed in-process reconnects so an external supervisor can respawn it.

## Best Possible Solution

Resolve the direct Slack reconnect exhaustion and channel stop-timeout recovery paths first, then add a documented generic supervisor-escalation policy only if maintainers accept gateway-wide process termination as the right fallback.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection still shows the finite Slack retry cap, in-process-only health-monitor restart, no processRestartAfter/SIGTERM path, and linked field logs matching the silent Slack dead state.

## Solution Assessment

Unclear. A Slack-specific self-SIGTERM knob may be useful as an operator escape hatch, but the safer first path is to land or replace the open reconnect-cap and lifecycle recovery fixes before adding gateway-wide supervisor escalation.
