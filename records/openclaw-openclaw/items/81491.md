## Summary

Keep open: current main still has the Slack Socket Mode reconnect cap and in-process-only health-monitor recovery, while the requested self-SIGTERM behavior is a new lifecycle/config policy that overlaps active Slack reconnect and channel restart work.

## What This Changes

The issue asks for Slack Socket Mode recovery to escalate to a configurable gateway self-SIGTERM after repeated failed reconnect/restart attempts so a supervisor respawns the process.

## Best Possible Solution

Fix or land the Slack reconnect-cap and generic channel restart lifecycle work first, then add gateway-level fatal escalation only if maintainers explicitly want supervisor-driven recovery after bounded in-process repair fails.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: current source and tests show the stale-task restart blocker and finite Slack reconnect cap, but no live Slack Socket Mode outage was rerun for this review.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A Slack-specific self-SIGTERM knob may be a useful operator escape hatch, but the narrower maintainable path is to land the reconnect-cap and channel lifecycle fixes before adding process-level policy.
