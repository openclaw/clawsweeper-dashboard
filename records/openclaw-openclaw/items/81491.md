## Summary

Keep open: current main still has no Slack self-SIGTERM threshold or `channels.slack.reconnect.processRestartAfter` config, while the already-merged generic stop-timeout recovery only partially overlaps the newer v2026.5.7 outage report.

## What This Changes

Add a configurable Slack reconnect escalation that SIGTERMs the gateway after repeated failed Slack Socket Mode reconnects so the service supervisor restarts the process.

## Best Possible Solution

Keep the generic restart recovery in place, then define a gateway-wide fatal escalation and liveness contract for channels that remain unhealthy after bounded in-process recovery, adding Slack-specific configuration only if maintainers want per-channel control.

## Reproduction Assessment

Unclear. The production timestamps and current source show no process-level escalation path, but I did not reproduce a live Slack Socket Mode wedge on current main and the provided logs omit the key stop-timeout/status evidence.

## Solution Assessment

Unclear. A Slack-specific self-SIGTERM knob may be a useful operator escape hatch, but a gateway-wide fatal recovery or readiness/liveness contract is likely the cleaner maintainer decision after the generic stop-timeout fix.
