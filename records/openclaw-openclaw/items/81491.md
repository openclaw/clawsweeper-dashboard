## Summary

Keep this open: current main and v2026.6.6 appear to have fixed the v2026.5.7 health-monitor stop-timeout recovery failure, but the requested Slack process-level escalation/config is still absent and related Slack reconnect/liveness work remains unresolved.

## What This Changes

Request Slack/gateway recovery to SIGTERM the gateway after repeated failed Slack reconnects so an external supervisor can restart the process instead of leaving Slack silently down.

## Best Possible Solution

Keep this as the explicit process-escalation policy discussion or intentionally supersede it under the narrower Slack reconnect-cap and zombie-liveness items after maintainer decision; avoid adding a new Slack config/process-kill default without upgrade and operator proof.

Yes at source level for the missing process escalation/config and the remaining finite Slack reconnect/no-event liveness surfaces; no live current-main 15-hour outage was reproduced.

## Solution Assessment

Unclear: self-SIGTERM after K Slack reconnect failures may be the right operator policy for some deployments, but maintainers need to choose it against the existing channel recovery path, #73162's reconnect-cap removal, and #77249's transport-liveness fix.
