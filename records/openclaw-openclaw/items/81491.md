## Summary

Keep open: current main and v2026.5.12 still lack the requested Slack `processRestartAfter`/self-SIGTERM recovery policy, and the remaining change is a gateway lifecycle/config decision that overlaps active Slack reconnect-cap and channel recovery work.

## What This Changes

The issue asks OpenClaw to terminate the gateway process after a configurable number of failed Slack reconnect or recovery attempts so an external supervisor restarts it.

## Best Possible Solution

Consolidate the Slack reconnect-cap and channel recovery work first, then decide and document a generic supervisor-escalation policy if in-process recovery remains insufficient.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a medium-confidence path: Slack Socket Mode still has a 12-attempt cap and the health monitor still performs in-process recovery without any process SIGTERM fallback.

## Solution Assessment

Unclear. A Slack-specific self-SIGTERM knob may be useful for operators, but the safer maintainable direction is to finish the reconnect-cap and channel lifecycle recovery work before adding process termination policy.
