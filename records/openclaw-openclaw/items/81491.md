## Summary

Keep open: current main still has a finite Slack Socket Mode reconnect cap and only in-process health-monitor restarts, while the requested self-SIGTERM behavior would add a new process-lifecycle policy that overlaps active narrower reliability work.

## What This Changes

The issue asks OpenClaw to terminate the gateway after a configurable number of failed Slack reconnects so an external supervisor can restart the process.

## Best Possible Solution

Fix or land the generic channel restart and Slack reconnect-cap work first, then add a gateway-wide fatal escalation only if maintainers explicitly want supervisor-driven recovery after bounded in-process repair fails.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The failure is source-reproducible: a disconnected Slack account can enter the in-process health-monitor restart path, and a timed-out channel stop can leave an old task registered so the immediate replacement start no-ops.

## Solution Assessment

Unclear. A Slack-specific `processRestartAfter` self-SIGTERM knob may be a useful operator escape hatch, but the narrower maintainable first fix is the generic channel lifecycle recovery plus the open Slack reconnect-cap fix.
