## Summary

Keep open. Current main removed the former Slack reconnect-attempt cap, but it still does not implement the requested supervisor-visible SIGTERM escalation, and that process-exit/default-config policy overlaps the canonical Slack Socket Mode liveness work without being fully owned by it.

## What This Changes

The issue requests configurable Slack reconnect escalation that self-SIGTERMs the gateway after repeated failed recovery so an external supervisor can restart it.

## Best Possible Solution

Define a maintainer-approved Slack recovery policy that first fixes or proves the no-event Socket Mode liveness gap, then adds supervisor-visible SIGTERM only if in-process recovery cannot be made reliable and the default is upgrade-safe.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. Source inspection confirms current main lacks the requested SIGTERM/config escalation and still relies on SDK disconnect/error events plus in-process channel/account restart recovery.

## Solution Assessment

Unclear. Process SIGTERM may be a valid supervised recovery backstop, but current main changed reconnect behavior and a related report shows aggressive restarts can be harmful, so maintainers should choose the threshold, window, and default before implementation.
