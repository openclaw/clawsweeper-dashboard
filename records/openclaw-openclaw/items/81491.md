## Summary

Keep open. Current main and v2026.6.6 still keep Slack Socket Mode recoverable retries capped at 12, health-monitor recovery remains an in-process stop/start with logged failures only, and the related open reconnect-cap PR does not implement this issue's requested process-level escalation.

## What This Changes

The issue requests configurable process-level SIGTERM escalation after repeated failed Slack reconnect or channel recovery attempts.

## Best Possible Solution

Define a maintainer-approved channel recovery policy first, then either make Slack's in-process recovery reliable or add a generic supervisor-escalation path with focused Slack/gateway tests and docs.

## Reproduction Assessment

Yes at source level, but not from a live Slack run in this review. Current main still has the finite Slack retry cap, health-monitor restart failures are caught and logged without process escalation, and `/healthz` remains shallow even when readiness fails.

## Solution Assessment

Unclear: the exact Slack-only `processRestartAfter` knob is a plausible mitigation, but the best fix needs maintainer policy on fail-closed process restarts versus stronger in-process recovery or readiness exposure.
