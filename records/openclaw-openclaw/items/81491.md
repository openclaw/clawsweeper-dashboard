## Label Justifications

- `P1`: The issue reports a long Slack channel outage where the gateway stayed HTTP-healthy while Slack delivery was unavailable.
- `impact:crash-loop`: The requested recovery path is process-level termination/restart after repeated channel recovery failure.
- `impact:message-loss`: The reported Slack provider failure leaves Slack messages unavailable or silently ignored while process health probes still pass.

## Summary

Keep open. Current main and the latest release still keep the Slack Socket Mode retry cap at 12, health-monitor restart failures are only logged, `/healthz` is intentionally shallow, and the open related reconnect-cap PR does not implement this issue's requested process-level escalation.

## What This Changes

Request a gateway/Slack recovery policy that escalates repeated failed Slack reconnects to process termination so an external supervisor can respawn the gateway.

## Best Possible Solution

Define a maintainer-approved channel recovery policy: land the finite-reconnect-cap fix separately, then either make readiness/monitoring expose exhausted channel recovery or add a generic supervisor-owned restart escalation with focused Slack/gateway tests and docs.

## Reproduction Assessment

Yes at source level, but not from a live Slack run in this review. Current main still has the finite Slack retry cap, health-monitor restart failures are caught and logged without process escalation, and `/healthz` remains shallow even when readiness fails.

## Solution Assessment

No, the exact proposed Slack-only `processRestartAfter` knob is not yet proven to be the best fix. A safer end state is a maintainer-approved generic channel recovery policy, or readiness/monitoring exposure for exhausted recovery, while the open reconnect-cap PR handles finite retry exhaustion.
