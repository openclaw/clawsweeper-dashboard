## Summary

Keep open: current main and the latest release still do not implement the requested self-SIGTERM or `processRestartAfter` escalation, and the remaining decision is a gateway lifecycle policy rather than conservative cleanup. Related open work covers the Slack reconnect cap and channel restart recovery, but it does not settle whether OpenClaw should terminate the whole gateway after repeated Slack reconnect failure.

## What This Changes

Request to add configurable gateway process self-termination after repeated failed Slack reconnects so a supervisor can restart the gateway after silent Slack channel outages.

## Best Possible Solution

Consolidate the narrow Slack reconnect-cap and channel lifecycle fixes first, then make an explicit maintainer decision on a documented gateway supervisor-escalation contract if in-process recovery remains insufficient.

## Reproduction Assessment

Yes at source level for the missing requested behavior: current main still has finite Slack reconnect attempts, in-process health-monitor restarts, and no `processRestartAfter` or SIGTERM path. No live reproduction of the exact 15-hour Slack outage was run.

## Solution Assessment

Unclear. A configurable self-SIGTERM may be useful as an operator escape hatch, but it is broader than the narrow Slack reconnect-cap fix and channel lifecycle recovery work, and it needs explicit product/lifecycle approval because it terminates the whole gateway.
