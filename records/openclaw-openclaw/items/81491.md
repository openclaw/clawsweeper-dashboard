## Summary

Keep open. Current main has improved the Slack reconnect and channel-restart paths, including the merged reconnect-cap fix, but it still does not implement the requested supervisor-visible SIGTERM escalation and that policy remains a maintainer product decision with live-repro risk.

## What This Changes

The issue requests Slack gateway process-level SIGTERM escalation after repeated failed Slack reconnects so external supervisors can recover a silent channel outage.

## Best Possible Solution

Define a maintainer-approved Slack recovery policy that keeps normal in-process reconnects by default and adds supervisor-visible escalation only if current-main live proof shows the channel can still remain dead after bounded recovery.

## Reproduction Assessment

No high-confidence live reproduction was established against current main. Source inspection confirms current main lacks the requested SIGTERM/config escalation and still uses in-process Slack reconnect plus channel-account restart recovery, but the exact v2026.5.7 15-hour stuck provider state was not replayed.

## Solution Assessment

Unclear. Process SIGTERM may be a valid supervised recovery option, but current main changed the reconnect and stop-timeout paths and a related report shows aggressive restarts can be harmful, so maintainers should choose the threshold/window/default before implementation.
