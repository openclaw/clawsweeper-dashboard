Keep open: current main and the latest shipped release still lack the requested `processRestartAfter`/self-SIGTERM path, and the remaining behavior spans Slack reconnect-cap, zombie Socket Mode liveness, and gateway lifecycle policy work that needs maintainer judgment.

## What This Changes

Requests a Slack/gateway lifecycle policy that self-terminates the gateway after configurable repeated Slack reconnect failures so an external supervisor can recover silent Slack outages.

## Best Possible Solution

Resolve the narrow Slack reconnect-cap and Socket Mode liveness issues first, then design any process-level escalation as a documented gateway supervisor policy instead of a Slack-only hidden kill path.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the missing requested behavior on current main: Slack reconnects remain capped, health recovery stays in-process, and no `processRestartAfter`/SIGTERM path exists.

## Solution Assessment

Unclear. A process self-termination knob may be useful as an operator escape hatch, but the narrower maintainable path is to land or replace the Slack reconnect-cap fix and address the canonical Socket Mode liveness/recovery issues before adding gateway-wide termination policy.
