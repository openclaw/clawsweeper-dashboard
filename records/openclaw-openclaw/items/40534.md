## Summary

Keep open. Current main fixed the original Telegram cron-announce path through final-visible-text delivery, but it still does not carry or suppress `isReasoning` in the shared cron payload selectors; this PR is useful, but it is now conflicting with current resolver code and lacks after-fix real behavior proof.

## What This Changes

The PR propagates `isReasoning` from embedded-runner payload construction, adds it to the run-result type, filters reasoning payloads from cron announce selection and recovery paths, and adds helper coverage.

## Best Possible Solution

Land one canonical cron announce fix that preserves `isReasoning`, treats reasoning as non-user-visible in summary/output/delivery/recovery paths, adds regression tests for the current resolver and multi-payload selector, and includes real channel proof.

## Reproduction Assessment

Yes. Source inspection on current main shows a reasoning payload can be produced, lose its marker before delivery selection, and then flow through cron announce fallback/direct delivery as ordinary text.

## Solution Assessment

No as-is. The proposed direction is right, but the stale branch must be rebased into the current resolver and multi-payload selector, then proven with real cron announce output before it is the best merge candidate.
