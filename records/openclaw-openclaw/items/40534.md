## Summary

Keep this PR open: it targets a valid cron reasoning leak, but the branch is stale against current main and misses the shared cron outcome path plus multi-payload delivery selector that now own the behavior.

## What This Changes

This PR propagates `isReasoning` from embedded-runner payload construction, filters reasoning in cron helper selection and run-level fallback/recovery paths, and adds helper regression tests.

## Best Possible Solution

Land one rebased canonical fix in the shared cron outcome path that preserves `isReasoning`, keeps reasoning out of summary/output/delivery and error recovery, preserves real error fallback, and includes the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: reasoning payloads are created, the returned payload map drops the marker, and the shared cron outcome treats later non-error text as summary/output/delivery and recovery.

## Solution Assessment

No. The PR fixes useful older call sites, but current main centralizes final and interim cron behavior in `resolveCronPayloadOutcome()` and uses `pickDeliverablePayloads()` for multi-payload delivery, so the filtering must move there.
