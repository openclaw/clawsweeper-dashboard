## Summary

Keep open: the underlying reasoning leak is still source-reproducible on current main, and this PR contains useful source work, but the branch is stale against the current cron outcome path and would not fully fix delivery after rebase. The overlapping #73363 context is relevant, but it has not cleanly superseded this PR because the desired behavior and remaining repair points still need one canonical implementation.

## What This Changes

This PR propagates `isReasoning` from embedded-runner payload construction, filters reasoning payloads from older isolated-cron selection helpers and run fallback logic, and adds helper regression tests.

## Best Possible Solution

Land one rebased canonical fix that preserves `isReasoning`, filters reasoning from summary, output, delivery, and post-error recovery in the shared cron outcome path, preserves real error fallback, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: reasoning items are created, the returned payload map drops the marker, and shared cron outcome selection treats payload text as visible/deliverable without `isReasoning` checks.

## Solution Assessment

No. The PR fixes useful older call sites, but current main centralizes final/interim cron behavior in `resolveCronPayloadOutcome()` and multi-payload selection in `pickDeliverablePayloads()`, so the filtering must move there.
