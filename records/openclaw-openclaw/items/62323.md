## Summary

Keep open. The underlying phantom-provider bug still exists on current main, and this PR is a useful but incorrect implementation because it can drop legitimate implicitly discovered native providers and regresses provider-prefix normalization and defensive registry value handling.

## What This Changes

The PR adds a `loadModelCatalog()` post-filter and regression tests to remove native-provider catalog rows inferred from slashed aggregator model IDs while preserving explicitly configured providers.

## Best Possible Solution

Land one corrected model-catalog/status fix that suppresses only proven duplicate aggregator-prefix artifacts, preserves real native providers from explicit config and implicit auth/env/profile discovery, normalizes provider aliases consistently, keeps defensive registry value coercion, and adds focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence reproduction is to stub `ModelRegistry.getAll()` with `openrouter` + `google/gemini-3-flash-preview` and a split `google` + `gemini-3-flash-preview` row; current main returns both and `/model status` groups both by provider.

## Solution Assessment

No, not as written. Filtering in `loadModelCatalog()` is the right boundary, but this implementation should preserve implicitly discovered native providers, normalize split prefixes before matching, keep defensive coercion, and add the required user-facing changelog entry.
