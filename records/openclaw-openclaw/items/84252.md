## Summary

This should stay open: current main still has source-backed gaps around unresolved Codex OAuth sidecar status and gateway refresh, and the open closing PR https://github.com/openclaw/openclaw/pull/84266 plus companion https://github.com/openclaw/openclaw/pull/84358 are implementation candidates rather than closure proof.

## What This Changes

The issue requests fail-closed, operator-visible handling for legacy openai-codex OAuth sidecar profiles in models status, doctor repair output, and running gateway auth refresh behavior.

## Best Possible Solution

Review and land or supersede the linked repair PRs with one coherent fix that classifies unresolved sidecar OAuth profiles as broken, refreshes or clearly restarts gateway auth state after repair, and leaves any source-checkout CLI-error or profile-rebind remainder tracked explicitly.

Yes, from source inspection plus reporter evidence: current main can keep oauthRef-only Codex profiles in inventory while runtime eligibility rejects OAuth credentials without access or refresh material. I did not run a live macOS LaunchAgent reproduction.

## Solution Assessment

Yes for the requested direction, but not yet on current main: the maintainable fix is to make unresolved sidecar state part of the shared auth-health contract and force or clearly request gateway refresh after doctor auth repairs.
