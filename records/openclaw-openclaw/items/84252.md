## Summary

This should stay open: current main still supports the reported partial-repair shape, and an open closing PR is an implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue asks for legacy `openai-codex` OAuth sidecar profiles to be reported and repaired as broken auth instead of appearing configured while runtime execution still fails.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/84266, then explicitly cover or split follow-up work for `models auth login` gateway refresh and source-checkout service-error scoping.

## Reproduction Assessment

Yes, from source inspection and reporter evidence: an OAuth profile with `oauthRef` but no inline access/refresh can remain visible as stored profile inventory while runtime eligibility rejects it. The live macOS LaunchAgent timing has not been reproduced in this review.

## Solution Assessment

No, current main is not a complete solution because status, doctor gateway refresh, and login/config reload behavior are not aligned. The narrow maintainable path is to finish the linked auth-health/status repair and make the remaining refresh/restart behavior operator-visible or explicitly refreshed.
