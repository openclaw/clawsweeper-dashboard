## Summary

Keep open: the PR fixes the reported runtime-alias bug, but it still has a blocking compatibility edge case where runtime-injected aliases can override user-authored aliases that the docs say should win.

## What This Changes

The branch makes `models set` resolve aliases against runtime config while writing canonical model IDs back to the authored config, with focused model-command tests.

## Best Possible Solution

Keep the source-write/runtime-fallback design, but resolve aliases in two stages so authored aliases win before falling back to runtime aliases.

## Reproduction Assessment

Yes. The source path and linked issue transcript show `models list` can expose the runtime `sonnet` alias while current `models set` resolves only against source config and falls back to `openai/sonnet`.

## Solution Assessment

No. The PR uses the right source/runtime boundary, but resolving directly against full runtime config is not the best final fix because it can violate documented custom-alias precedence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve authored aliases before runtime aliases:** `src/commands/models/set.ts:10`
  - body: Resolving directly against `context.runtimeConfig` can let built-in aliases injected at runtime override aliases the user explicitly authored. For example, if source config maps `sonnet` to `openai/gpt-5.4` and runtime defaults also add `sonnet` for Anthropic, this patch can persist the Anthropic model even though the docs say configured aliases win. Please resolve against source aliases first, then fall back to runtime aliases.
  - confidence: 0.91
