## Summary

Keep this PR open: it fixes the reported runtime-only alias miss, but the current patch can let runtime-injected default aliases override a user-authored alias collision that the docs say should win.

## What This Changes

The PR adds runtime config context to `models set`, uses it for alias resolution before writing the source config, and adds focused regression tests for the model command path.

## Best Possible Solution

Keep the runtime fallback for aliases that only exist in effective config, but resolve authored/source aliases first and add a collision regression test proving custom aliases still win.

## Reproduction Assessment

Yes. The linked bug is source-reproducible because current main resolves `models set` against source config only while runtime defaults can add `sonnet`; the new review finding is also source-reproducible from runtime alias injection plus duplicate-alias overwrite order.

## Solution Assessment

No, not yet. Resolving directly against the full runtime config fixes the reported miss but is not the safest solution because it bypasses documented custom-alias precedence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve authored alias precedence before runtime aliases:** `src/commands/models/set.ts:11`
  - body: Passing only `context.runtimeConfig` into the write resolver means a runtime-injected default alias can shadow a user-authored alias with the same name. For example, if the source config maps `sonnet` to `openai/gpt-5.4` and also has the Anthropic Sonnet model without an alias, runtime defaults can add `alias: "sonnet"` to Anthropic; `buildModelAliasIndex` then overwrites the earlier alias entry and `models set sonnet` writes the wrong model, despite the docs saying configured aliases win. Please resolve against source aliases first, then fall back to runtime aliases.
  - confidence: 0.92
