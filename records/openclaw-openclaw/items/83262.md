## Summary

Keep open: the PR fixes the reported runtime-only alias path, but the branch can let a runtime-injected default alias override a user-authored alias, contradicting the documented configured-alias precedence and silently changing model choice.

## What This Changes

The PR adds a runtime-config context to model config updates, changes `models set` to resolve aliases against that runtime config, and adds focused regression tests.

## Best Possible Solution

Resolve `models set` aliases in source-first, runtime-fallback order, with regression coverage for both runtime-only default aliases and custom-alias collisions.

## Reproduction Assessment

Yes, source-reproducible. The linked issue includes a concrete release transcript for the original bug, and source inspection of the PR plus current alias materialization shows the new custom-alias collision path; I did not run mutating CLI commands in this read-only review.

## Solution Assessment

No. The source/runtime split is the right direction, but direct runtime-config alias resolution is not the safest fix because authored aliases must take precedence over injected defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve authored alias precedence:** `src/commands/models/set.ts:11`
  - body: Passing only `context.runtimeConfig` into the resolver means `models set` builds its alias index after runtime defaults have injected built-in aliases. If a user authored `openai/gpt-5.4` with `alias: "sonnet"` and also has Anthropic Sonnet configured without an alias, runtime materialization can add a second `sonnet`; `buildModelAliasIndex` then lets the later entry overwrite the first, so the command writes the built-in Anthropic model instead of the user's alias despite the docs saying configured aliases win. Resolve against source aliases first, then fall back to runtime aliases.
  - confidence: 0.92
