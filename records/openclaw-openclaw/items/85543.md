## Summary

Keep open: the bug and runtime proof are strong, but the fallback executable is selected after system.run approval, allowlist audit, durable trust, and completion metadata have already bound the original `/bin/sh` argv, so the security-boundary design needs maintainer review before merge.

## What This Changes

The PR adds a `/usr/bin/sh` retry for Linux node-host `system.run` when spawning `/bin/sh` fails with `ENOENT`, plus focused fallback tests.

## Best Possible Solution

Choose the fallback shell before approval, allowlist, durable-trust, audit, and completion records are built, or explicitly model both approved absolute shell paths in the system.run plan before refreshing the branch.

## Reproduction Assessment

Yes at source level: current main's node-host `runCommand` treats a spawn `ENOENT` as terminal, and the PR body adds a chroot proof for the same missing-`/bin/sh` condition. I did not reproduce the reporter's exact paired Linux node/systemd setup.

## Solution Assessment

No as currently shaped: retrying a trusted absolute shell is a plausible fix, but selecting it inside the execution helper is not the narrowest safe implementation because it bypasses the existing approval and audit binding point.
