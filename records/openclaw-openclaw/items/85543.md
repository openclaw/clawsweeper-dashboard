## Summary

Keep open: this is a focused fix for a real node-host execution failure with adequate runtime proof, but the fallback changes an approval-gated execution boundary and still needs maintainer acceptance before merge.

## What This Changes

The PR adds a Linux node-host retry from `/bin/sh` to trusted absolute `/usr/bin/sh` on ENOENT and adds regression coverage for that fallback path.

## Best Possible Solution

Keep the bounded absolute fallback only if maintainers accept `/usr/bin/sh` as an equivalent trusted shell after `/bin/sh` ENOENT; otherwise bind the fallback candidate in the approval/audit plan before execution.

## Reproduction Assessment

Yes for the code path: current main constructs `/bin/sh -lc` and finalizes on spawn ENOENT, and the PR body supplies live chroot output showing the patched runtime succeeds with `/bin/sh` absent and `/usr/bin/sh` present. The reporter's exact paired Linux node/systemd setup was not reproduced here.

## Solution Assessment

Unclear until maintainers decide the security tradeoff. The absolute `/usr/bin/sh` retry is narrow and avoids PATH resolution, but the safer version would bind the fallback in the approval/audit plan or record maintainer acceptance of the post-approval switch.
