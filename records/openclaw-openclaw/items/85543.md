## What This Changes

## Best Possible Solution

Keep shell selection bound before approval/policy, but handle or clearly diagnose `spawn ENOENT` when `/bin/sh` exists, including the cwd-vs-shell distinction, and refresh current-head real behavior proof before merge.

## Reproduction Assessment

Yes for source-level triage, but not as a fresh local end-to-end run. The linked issue has live Linux node evidence with `/bin/sh` present, and current main plus the latest release still route non-Windows node exec to `/bin/sh` and return spawn error messages.

## Solution Assessment

No. Moving shell selection before policy is the right layer, but the current predicate is too narrow for the reported hosts; the best fix needs a fallback or diagnostic that covers ENOENT with `/bin/sh` present while keeping metadata bound to the executed argv.
