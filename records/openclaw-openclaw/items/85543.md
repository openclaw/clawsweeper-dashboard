## Summary

Keep this PR open, but it should not merge as-is because the fallback is chosen inside execution after the node-host approval, durable trust, allowlist audit, and `exec.finished` command context are already bound to the original `/bin/sh` argv.

## What This Changes

The PR adds a Linux node-host `system.run` retry from `/bin/sh` to `/usr/bin/sh` on ENOENT and adds focused fallback tests.

## Best Possible Solution

Land a retry that binds the selected fallback shell before approval matching, durable trust, allowlist audit, and `exec.finished` reporting, with focused tests for those records.

## Reproduction Assessment

Yes. Current main has a single `/bin/sh` Linux node shell path, and the PR body supplies chroot proof for the missing-`/bin/sh` condition, though not the reporter's exact paired-node setup.

## Solution Assessment

No. Retrying inside `runCommand` fixes the spawn symptom but is not the safest solution because it changes the actual executable after the approval and audit context has already been finalized.
