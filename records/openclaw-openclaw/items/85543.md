## Summary

Keep open: the PR addresses a real Linux node exec blocker with sufficient terminal proof, but the current patch still retries a different shell below the approval/audit boundary, so it is not merge-ready.

## What This Changes

The PR adds a Linux node-host retry from `/bin/sh -lc` to `/usr/bin/sh -lc` on ENOENT and adds focused shell fallback coverage.

## Best Possible Solution

Keep the fallback only if the selected shell argv is bound or revalidated through approval, allowlist-use, durable trust, and `exec.finished` metadata before execution.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.6 build Linux node exec as `/bin/sh -lc` and directly spawn it, matching the linked issue's live ENOENT evidence. I did not run the reporter's full paired Docker plus user-systemd node setup.

## Solution Assessment

No: the fallback idea is useful, but this implementation chooses the fallback inside `runCommand` after security metadata is finalized. The safer fix is to bind the executable fallback before approval/audit metadata is recorded, or revalidate and record the fallback retry explicitly.
