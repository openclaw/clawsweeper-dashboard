## Summary

Keep open. The PR is aimed at the right node-host execution path for the linked Linux headless-node failure, but the submitted retry loop does not actually reach the final PATH-resolved `sh` fallback if `/usr/bin/sh` is also missing, and the supplied proof does not exercise the node-host retry branch itself.

## What This Changes

The PR adds node-host POSIX shell fallback argv generation and retries `system.run` shell execution after a `/bin/sh` ENOENT, with unit coverage for the fallback argv list.

## Best Possible Solution

Keep the fallback in the node-host execution boundary, but make every intended ENOENT fallback reachable and add focused coverage for the retry loop plus redacted real node-host proof.

## Reproduction Assessment

Yes for the PR defect: source inspection of the submitted loop shows `/usr/bin/sh` ENOENT returns before the final `sh` attempt. For the original issue, the linked report has live Linux headless-node evidence, but I did not rerun that environment in this review.

## Solution Assessment

No as submitted. The node-host fallback location is maintainable, but the retry predicate should treat ENOENT from each planned shell candidate as retryable until the last candidate and the proof should exercise that runtime branch.
