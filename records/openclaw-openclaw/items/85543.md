## Summary

Keep open: the fallback addresses a real linked Linux node exec failure, but the PR is not merge-ready because the fallback executable is selected after approval, allowlist-use, durable trust, and completion metadata have already been bound to the original argv.

## What This Changes

The PR adds a Linux node-host retry from `/bin/sh -lc` to `/usr/bin/sh -lc` after `/bin/sh` spawn ENOENT, plus focused fallback tests.

## Best Possible Solution

Keep the fallback behavior, but bind or revalidate the selected fallback argv before system.run approval, durable trust, allowlist-use, and completion metadata are finalized.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.6` still build Linux node exec as `/bin/sh -lc` and directly spawn it, matching the linked issue's live ENOENT evidence. I did not run the reporter's exact paired Docker plus user-systemd node setup.

## Solution Assessment

No: the fallback idea is useful, but this implementation chooses the fallback inside `runCommand` after security metadata is finalized. The safer fix is to make the argv that may execute the same argv that approval and audit surfaces bind to, or revalidate and record the fallback retry before execution.
