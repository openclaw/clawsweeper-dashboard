## Summary

Keep open. The branch is a plausible fix candidate, but it is not merge-ready because the fallback only activates when `/bin/sh` is absent while the canonical linked bug reports `spawn /bin/sh ENOENT` on hosts where `/bin/sh` and `/usr/bin/sh` both exist; the real-behavior proof also predates the current implementation shape.

## What This Changes

The PR adds a `/usr/bin/sh` canonicalization helper for node-host `system.run` planning when `/bin/sh` is absent, updates system.run parse/approval paths, exports `runCommand`, and adds focused tests.

## Best Possible Solution

Keep the approval-safe planning placement, but make the implementation cover or diagnose `ENOENT` when `/bin/sh` exists, with regression coverage for approval, allowlist, audit, and `exec.finished` parity plus current-head real behavior proof.

## Reproduction Assessment

Yes for source-level triage. The linked issue provides live Linux node evidence with `/bin/sh` present, and current main still routes Linux node exec through `/bin/sh -lc` before direct spawn.

## Solution Assessment

No. Moving shell selection before approval is the right layer, but the current predicate is too narrow for the reported failure and the proof is stale for the latest head.
