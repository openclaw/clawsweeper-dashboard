## Summary

Keep this PR open: it targets a real user-blocking node exec failure and has useful runtime proof, but the current patch moves shell fallback inside `runCommand` after approval, durable approval, audit, and event text have already been derived from the original `/bin/sh` argv.

## What This Changes

The PR retries node-host POSIX shell execution with `/usr/bin/sh` after `/bin/sh` spawn ENOENT and adds regression coverage for that retry path.

## Best Possible Solution

Bind any approved fallback executable into the system.run execution plan before approval, durable approval, audit recording, and completion events, or get explicit maintainer acceptance that `/bin/sh` and `/usr/bin/sh` are equivalent for this boundary.

## Reproduction Assessment

Yes at source level: current main builds Linux node shell commands as `/bin/sh -lc ...` and current `runCommand` treats a spawn error as terminal. I did not run the original paired-node systemd fleet in this read-only review.

## Solution Assessment

No, not yet. Retrying a trusted absolute shell path is plausible, but doing it inside `runCommand` after policy evaluation is not the narrowest maintainable fix for the approval boundary.
