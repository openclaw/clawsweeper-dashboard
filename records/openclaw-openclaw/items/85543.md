## Summary

Keep this PR open: it addresses a real Linux node-host failure with useful proof, but the current patch still moves execution from `/bin/sh` to `/usr/bin/sh` inside `runCommand` after the approval, durable trust, allowlist audit, and finished-event surfaces have already bound to the original command.

## What This Changes

The PR adds a Linux node-host `/usr/bin/sh` retry after `/bin/sh` spawn `ENOENT` and adds focused fallback tests.

## Best Possible Solution

Land the fallback only after the approved execution plan, durable trust/allowlist audit, and `exec.finished` output either bind to the actual fallback executable or explicitly prove and record the allowed `/bin/sh` to `/usr/bin/sh` equivalence, then refresh against current main.

## Reproduction Assessment

Yes from source inspection, though I did not run a live repro in this read-only review. Current main builds Linux node-host shell commands as `/bin/sh -lc` and has no retry path, matching the linked ENOENT failure mode.

## Solution Assessment

No as-is. The fallback is a narrow symptom fix with solid runtime proof, but the maintainable solution must bind the fallback executable before or through the approval and audit surfaces.
