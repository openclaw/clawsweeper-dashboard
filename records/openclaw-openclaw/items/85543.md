## Summary

Keep open: the fallback is still useful for the linked Linux node exec failure, but this patch is not merge-ready because it retries a different shell executable after system.run approval, durable trust, allowlist-use, and completion metadata have already been bound to the original argv.

## What This Changes

The PR adds a Linux node-host retry from `/bin/sh -lc` to `/usr/bin/sh -lc` on ENOENT and adds focused unit coverage for the retry chain.

## Best Possible Solution

Keep the fallback idea, but bind the fallback argv before system.run policy/approval metadata is finalized, or explicitly revalidate and record the selected fallback argv before retrying it.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.6` build Linux node exec as `/bin/sh -lc` and directly spawn it, matching the linked issue's live ENOENT evidence. I did not run the reporter's exact paired Docker plus user-systemd node setup.

## Solution Assessment

No: the fallback idea is useful, but this implementation chooses the fallback inside `runCommand` after security metadata is finalized. The safer fix is to make the argv that may execute the same argv that approval and audit surfaces bind to, or revalidate and record the fallback retry before execution.
