## Summary

Keep open: this PR addresses a still-missing node-host shell fallback and has solid chroot runtime proof, but the fallback changes the executed shell path after approval and audit context are already bound to `/bin/sh`, so maintainer security-boundary judgment is still needed before merge.

## What This Changes

The PR adds a POSIX node-host retry from `/bin/sh` to `/usr/bin/sh` on ENOENT and adds regression coverage for that retry chain.

## Best Possible Solution

Land only after maintainers explicitly accept `/usr/bin/sh` as a trusted equivalent fallback for approved `/bin/sh` node-host runs, or revise the implementation so the approved/audited argv identifies the actual shell path.

## Reproduction Assessment

Yes, source inspection shows current main returns the first `/bin/sh` spawn error without trying another shell, and the PR proof exercises the missing-shell condition through the patched runtime path. I did not reproduce the reporter's exact paired systemd node fleet.

## Solution Assessment

Unclear: the narrow fallback fixes the observable ENOENT symptom, but the safest merge path depends on whether maintainers accept `/usr/bin/sh` as equivalent after exact-argv approval binding. Binding the selected shell before approval/audit would reduce the security-boundary ambiguity.
