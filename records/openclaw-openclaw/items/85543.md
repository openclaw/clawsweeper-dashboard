## Summary

Keep this PR open: the fallback addresses a real node-host failure and the contributor supplied useful runtime proof, but the current diff changes the shell executable after the approval/audit path has already bound the command to `/bin/sh`. That needs a security-boundary fix or explicit maintainer approval before merge.

## What This Changes

The PR retries node-host POSIX shell execution with `/usr/bin/sh` after `/bin/sh` spawn ENOENT and adds regression tests for the retry path.

## Best Possible Solution

Keep the trusted absolute fallback, but bind the selected shell or approved fallback set into the `systemRunPlan`, approval matching, durable records, and audit/event text before execution.

## Reproduction Assessment

Yes, from source and contributor proof: current node-host POSIX exec is prepared as `/bin/sh -lc`, and the PR body demonstrates the patched runtime succeeding when `/bin/sh` is absent and `/usr/bin/sh` exists. I did not rerun the chroot proof locally in this read-only review.

## Solution Assessment

No, not yet: retrying a trusted absolute shell is a plausible fix, but doing it below approval and audit binding is not the safest implementation. The maintainable path is to make fallback shell selection part of the approved execution plan or get an explicit maintainer decision that the two shell paths are equivalent for this boundary.
