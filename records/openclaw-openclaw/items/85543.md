## Summary

Keep open: the fix is not on current main and the supplied chroot proof shows the patched node-host path solving the reported ENOENT condition, but the fallback changes the executable after approval analysis and needs maintainer security-boundary acceptance before merge.

## What This Changes

The PR adds a node-host retry from `/bin/sh` to absolute `/usr/bin/sh` on spawn ENOENT and adds regression coverage for that retry path.

## Best Possible Solution

Keep the bounded absolute fallback only if maintainers accept the approval-identity tradeoff; otherwise move fallback selection into the plan/audit path so the approved and executed shell identities stay aligned.

## Reproduction Assessment

Yes for the code path: current main constructs `/bin/sh -lc` and finalizes on spawn ENOENT, and the PR body supplies live chroot output showing the patched runtime succeeds with `/bin/sh` absent and `/usr/bin/sh` present. The reporter's exact paired Linux node/systemd setup was not reproduced.

## Solution Assessment

Unclear until maintainers decide the security tradeoff. The absolute `/usr/bin/sh` retry is narrow and avoids PATH resolution, but the safest version would either bind that fallback in the approval/audit plan or explicitly document maintainer acceptance of the post-approval switch.
