## Summary

Keep open: this is a focused, proof-positive fix for Linux node shell ENOENT, but it still changes the executed shell from `/bin/sh` to `/usr/bin/sh` after `system.run` approval, allowlist, durable approval, and audit text have been derived from `/bin/sh`. Maintainers need to explicitly accept that shell-path equivalence or ask for the chosen executable to be bound earlier.

## What This Changes

The PR changes node-host shell execution to retry `/usr/bin/sh` after `/bin/sh` spawn ENOENT and adds focused regression coverage.

## Best Possible Solution

Merge only after maintainers either accept `/bin/sh` to `/usr/bin/sh` as an explicit trusted node-transport equivalence or adjust the prepare/approval/run path so the actually executed shell path is the path approved and audited.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.5.27 build POSIX node exec as `/bin/sh -lc` and spawn that argv once, so the linked ENOENT report terminates before any fallback. I did not run the reporter's paired systemd node fleet.

## Solution Assessment

Unclear as proposed: the retry is narrow and has good runtime proof for missing `/bin/sh`, but doing the executable swap after approval/audit is not clearly the best security-boundary shape. The safer end state is to bind or explicitly approve the actual shell path before merge.
