## Summary

Keep open: the fallback is useful and the contributor supplied credible runtime proof, but the current patch retries a different shell after the approval, durable trust, allowlist-use, and completion metadata have already bound the original argv.

## What This Changes

The PR adds a Linux node-host fallback from `/bin/sh` to `/usr/bin/sh` on ENOENT and adds focused fallback tests.

## Best Possible Solution

Keep the fallback, but bind or revalidate the selected shell argv before approval, audit, allowlist-use, and `exec.finished` metadata are finalized.

## Reproduction Assessment

Yes, source-reproducible rather than locally live-reproduced: current main and v2026.6.6 build Linux node exec as `/bin/sh -lc` and directly spawn it, and the linked issue has live Linux headless node evidence for the ENOENT failure.

## Solution Assessment

No: the fallback idea is useful, but choosing the fallback inside `runCommand` happens after security metadata is finalized. The safer fix is to make the argv that may execute the same argv that approval and audit surfaces bind to, or to revalidate and record the fallback retry.
