## Summary

Keep open: the fallback targets a real Linux node exec failure and has useful runtime proof, but the PR still retries the executable below the system.run approval/audit layer, so it can execute a different shell than the one bound to metadata.

## What This Changes

The PR adds a Linux node-host `/bin/sh` ENOENT retry to `/usr/bin/sh`, exports `runCommand` for a new focused test, and adds fallback regression coverage.

## Best Possible Solution

Keep the fallback behavior, but bind or revalidate the final executable argv before system.run policy, approval, audit, durable trust, and completion metadata are finalized.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.6` still build Linux node exec as `/bin/sh -lc` and directly spawn it, matching the linked issue's live ENOENT evidence. I did not run the reporter's exact paired Docker plus user-systemd node setup.

## Solution Assessment

No: the fallback idea is useful, but this implementation chooses the fallback inside `runCommand` after security metadata is finalized. The safer fix is to make the argv that may execute the same argv that approval and audit surfaces bind to, or revalidate and record the fallback retry before execution.
