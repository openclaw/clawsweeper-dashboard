## Summary

Keep open: the bug is valid and the PR has useful proof, but the latest head still chooses the fallback shell inside `runCommand` after approval, audit, and completion metadata have already bound to the original `/bin/sh` argv.

## What This Changes

The PR adds a Linux node-host retry from `/bin/sh -lc` to `/usr/bin/sh -lc` on ENOENT, plus focused shell fallback tests.

## Best Possible Solution

Keep the fallback, but bind or revalidate the selected shell argv before approval, audit, allowlist-use, and `exec.finished` metadata are finalized.

## Reproduction Assessment

Yes, source-reproducible rather than locally live-reproduced: current main and v2026.6.6 build Linux node exec as `/bin/sh -lc` and directly spawn it, matching the linked issue's live ENOENT evidence. I did not run the reporter's exact paired Docker plus user-systemd node setup.

## Solution Assessment

No: the fallback idea is useful, but this implementation chooses the fallback inside `runCommand` after security metadata is finalized. The safer fix is to make the argv that may execute the same argv that approval and audit surfaces bind to, or explicitly revalidate and record the fallback retry.
