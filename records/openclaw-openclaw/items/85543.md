## Summary

Keep open: the Linux node exec fallback is still needed, but this branch is not merge-ready because fallback execution is chosen inside `runCommand` after `system.run` approval and audit metadata have already been bound to the original argv.

## What This Changes

The PR adds a node-host `system.run` retry from `/bin/sh` to absolute `/usr/bin/sh` on ENOENT and adds focused shell fallback tests.

## Best Possible Solution

Keep the fallback behavior, but bind or revalidate the selected fallback argv before system.run approval, durable trust, allowlist-use, and completion metadata are finalized.

## Reproduction Assessment

Yes for source-level triage: the linked issue has live Linux node evidence, and current main plus `v2026.6.8` still build `/bin/sh -lc` and directly spawn it. I did not rerun the reporter's exact paired Docker plus user-systemd setup.

## Solution Assessment

No: the fallback idea is useful, but this implementation chooses the fallback inside `runCommand` after security metadata is finalized. The safer fix is to make the argv that may execute the same argv that approval and audit surfaces bind to, or to revalidate and record the fallback retry before execution.
