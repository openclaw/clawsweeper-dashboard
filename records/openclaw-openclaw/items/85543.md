## Summary

Keep open. The PR targets a valid linked Linux node exec failure, but the submitted retry loop stops after `/usr/bin/sh` ENOENT and never reaches the advertised PATH `sh` fallback, so it needs a small correctness repair before merge.

## What This Changes

The branch adds node-host shell fallback argv for `/bin/sh` ENOENT and wires `system.run` to retry `/usr/bin/sh` and `sh`, with unit coverage for the fallback argv list.

## Best Possible Solution

Keep the node-host fallback in this execution path, but retry each fallback candidate on spawn ENOENT and cover the runtime retry sequence with focused regression proof.

## Reproduction Assessment

Yes, there is a strong source-level reproduction path and linked live repro steps for `exec host=node` returning `spawn /bin/sh ENOENT`; I did not run a Linux headless systemd node locally in this read-only review.

## Solution Assessment

No, not as submitted. The fix belongs in node-host shell execution, but the retry loop must continue through the full fallback list and proof should exercise the node-host retry behavior, not only the helper output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Retry the PATH shell after /usr/bin/sh ENOENT:** `src/node-host/invoke.ts:149`
  - body: The fallback list includes `sh`, but this loop returns immediately if the `/usr/bin/sh` attempt fails with ENOENT because `spawnMissing` is only true for the original `/bin/sh` attempt. That means a host where `/bin/sh` and `/usr/bin/sh` are unavailable but `sh` is resolvable from PATH will never reach the advertised PATH fallback; treat ENOENT from any non-final fallback attempt as retryable and add coverage for that sequence.
  - confidence: 0.94
