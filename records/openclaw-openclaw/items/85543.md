## Summary

Keep open. Current main and the latest release still lack a Linux node-host shell fallback, but this PR is not merge-ready because the fallback is selected below the system.run approval, durable trust, allowlist-use, and completion metadata layer.

## What This Changes

The PR adds a Linux node-host system.run retry from `/bin/sh` to trusted absolute `/usr/bin/sh` on ENOENT, with focused fallback tests.

## Best Possible Solution

Keep the fallback behavior, but bind or revalidate the selected fallback argv before system.run approval, durable trust, allowlist-use, and completion metadata are finalized.

## Reproduction Assessment

Yes for review: the linked issue has live Linux node evidence, and current main plus `v2026.6.8` still build `/bin/sh -lc` and directly spawn it. I did not rerun the reporter's exact paired Docker plus user-systemd setup.

## Solution Assessment

No: the fallback idea is useful, but this implementation chooses the fallback inside `runCommand` after security metadata is finalized. The safer fix is to make the argv that may execute the same argv that approval and audit surfaces bind to, or to revalidate and record the fallback retry before execution.
