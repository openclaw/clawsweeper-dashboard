## Summary

Keep open: the fallback addresses a real Linux node-host exec blocker, but the current patch is not merge-ready because it chooses `/usr/bin/sh` inside `runCommand` after approval, allowlist-use, and finished-event metadata have already bound the original `/bin/sh` argv.

## What This Changes

The PR adds a Linux node-host `system.run` retry from `/bin/sh` to `/usr/bin/sh` on ENOENT, with focused unit coverage for the fallback chain.

## Best Possible Solution

Keep the fallback only after making the selected executable part of the system.run policy plan, or revalidating and recording the fallback argv before approval/audit/completion metadata are finalized.

## Reproduction Assessment

Yes, source-reproducible with linked live evidence: current main and v2026.6.6 build Linux node exec as `/bin/sh -lc` and directly spawn it, while the linked issue reports live `spawn /bin/sh ENOENT` on connected Linux nodes. I did not rerun the reporter's exact paired Docker plus user-systemd setup.

## Solution Assessment

No: the fallback idea is useful, but this implementation chooses the fallback inside `runCommand` after security metadata is finalized. The safer fix is to bind or revalidate the fallback argv before approval, audit, allowlist-use, and completion metadata are written.
