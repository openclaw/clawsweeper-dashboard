## Summary

Keep open. The PR fixes a real Linux node exec failure mode, but it is not merge-ready because the fallback shell is selected inside `runCommand` after the `system.run` approval, allowlist-use, and completion metadata have already been bound to the original argv.

## What This Changes

The PR retries Linux node-host shell execution with trusted absolute `/usr/bin/sh` after `/bin/sh` spawn ENOENT and adds focused fallback tests.

## Best Possible Solution

Keep the trusted absolute fallback, but make the selected shell argv the same command bound to approval, allowlist-use, durable trust, and completion metadata before execution.

## Reproduction Assessment

Yes, with medium confidence as source-reproducible: the linked issue has live Linux node repro evidence and current source still builds `/bin/sh -lc` and directly spawns it. I did not rerun the reporter's paired Docker plus user-systemd node setup locally.

## Solution Assessment

No: the fallback idea is useful, but this implementation retries below the approval and audit boundary. The safer fix is to bind fallback selection before policy metadata or revalidate and record the fallback argv before execution.
