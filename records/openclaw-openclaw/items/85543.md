## Summary

Keep open: this is still the active fix path for an open Linux node-host exec blocker, but the current patch is not merge-ready because the fallback retry can run a different shell argv than the one bound to approval and audit metadata.

## What This Changes

The PR adds a Linux node-host retry from `/bin/sh -lc` to `/usr/bin/sh -lc` on ENOENT, plus focused regression tests for the retry chain.

## Best Possible Solution

Preserve the fallback only if the selected shell argv is bound to the same approval, allowlist, durable trust, and completion metadata as the executed process.

## Reproduction Assessment

Yes for source and reporter evidence: current main and v2026.6.6 build Linux node exec as `/bin/sh -lc` and directly spawn it, while the linked issue has live headless-node reports of `spawn /bin/sh ENOENT`. I did not run the reporter's exact paired node setup locally.

## Solution Assessment

No: the fallback idea is useful, but selecting it inside `runCommand` is too late for approval and audit parity. The safer fix is to bind fallback selection into planning/policy or revalidate and record fallback metadata before execution.
