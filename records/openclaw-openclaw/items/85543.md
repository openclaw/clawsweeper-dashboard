## Summary

Keep open. The PR is still the active candidate for the linked Linux node exec failure, but the current fallback predicate only switches when `/bin/sh` is absent, while the canonical report and related reports show `spawn /bin/sh ENOENT` with `/bin/sh` present.

## What This Changes

The PR adds a Linux node-host shell canonicalization path that rewrites `/bin/sh -lc ...` to `/usr/bin/sh -lc ...` before `system.run` approval/execution planning when `/bin/sh` is missing.

## Best Possible Solution

Keep the pre-approval binding shape, but make shell selection cover the reported ENOENT-with-`/bin/sh`-present case or prove a different root cause, with coverage for commandText, approval, allowlist, and `exec.finished` parity.

## Reproduction Assessment

Yes for source-level triage, though I did not run the exact paired-node setup. The linked issue has live Linux node evidence for `spawn /bin/sh ENOENT` with `/bin/sh` and `/usr/bin/sh` present, and current main/latest release still plan `/bin/sh -lc`.

## Solution Assessment

No. Moving fallback selection before approval is the right layer, but the current predicate is too narrow for the reported failure because it keeps `/bin/sh` whenever that path exists.
