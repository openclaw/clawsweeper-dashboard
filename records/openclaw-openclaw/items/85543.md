## Summary

Keep open: current main and v2026.5.27 still lack the `/usr/bin/sh` fallback, and the PR has credible runtime proof, but merging it still needs maintainer acceptance of the post-approval shell identity change.

## What This Changes

The PR adds a Linux node-host retry from `/bin/sh` to absolute `/usr/bin/sh` on spawn ENOENT and adds regression coverage for that retry chain.

## Best Possible Solution

Land the bounded absolute fallback only after maintainers accept `/usr/bin/sh` as an equivalent trusted shell, or revise the implementation so fallback selection is part of the approved/audited execution plan.

## Reproduction Assessment

Yes for the code path: current main constructs `/bin/sh -lc` and finalizes spawn ENOENT, while the PR body shows the patched path succeeding in a Linux chroot with `/bin/sh` absent and `/usr/bin/sh` present. The exact reporter node/systemd topology was not reproduced.

## Solution Assessment

Unclear until maintainers decide the security tradeoff. The absolute fallback is narrow and avoids PATH resolution, but the safer design would bind or audit the fallback candidate before execution.
