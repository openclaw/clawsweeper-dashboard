## Summary

Keep open: current main and v2026.6.6 still use the direct `/bin/sh -lc` node-host path, so the PR addresses a real linked bug, but the current patch retries `/usr/bin/sh` inside `runCommand` after approval/audit metadata has already been bound to the original argv.

## What This Changes

This PR adds a Linux node-host `system.run` retry from `/bin/sh -lc` to `/usr/bin/sh -lc` on ENOENT and adds focused fallback tests.

## Best Possible Solution

Preserve the trusted fallback, but bind or revalidate the selected shell argv before approval, allowlist-use, durable trust, and `exec.finished` metadata are finalized.

## Reproduction Assessment

Yes, source-reproducible rather than personally live-reproduced: the linked issue has live Linux systemd-node evidence, current main and v2026.6.6 build `/bin/sh -lc` and spawn it directly, and a Node probe confirms missing absolute executables emit the same `ENOENT` shape.

## Solution Assessment

No. The fallback idea is useful and in the right area, but this implementation applies it below the policy/metadata boundary; the safer fix binds or revalidates the selected argv before approval, audit, and completion records.
