## Summary

Keep open: this targets a real Linux node exec blocker and has useful runtime proof, but the patch still selects the fallback shell after system.run approval, allowlist-use, durable trust, and completion metadata have already been bound to the original argv.

## What This Changes

The PR adds a `/bin/sh` ENOENT retry to `/usr/bin/sh` in the node-host `runCommand` path and adds focused node shell fallback tests.

## Best Possible Solution

Keep the fallback behavior, but bind or revalidate the selected fallback argv before system.run approval, durable trust, allowlist-use, and completion metadata are finalized.

## Reproduction Assessment

Yes for source-level triage, but not as a fresh local run: the linked issue has live Linux node evidence, and current main plus `v2026.6.8` still build `/bin/sh -lc` and directly spawn it.

## Solution Assessment

No: the fallback idea is useful, but this implementation chooses the fallback inside `runCommand` after security metadata is finalized. The safer fix is to make the argv that may execute the same argv that approval and audit surfaces bind to, or to revalidate and record the fallback retry before execution.
