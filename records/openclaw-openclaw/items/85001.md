## Summary

Keep this PR open: it targets a real source-reproducible reply-run cleanup problem linked to https://github.com/openclaw/openclaw/issues/84903 and https://github.com/openclaw/openclaw/issues/84755, but the current fix clears no-backend operations too broadly and lacks after-fix real behavior proof.

## What This Changes

The PR changes reply-run abort cleanup to clear running operations with no attached backend and adds a registry unit test plus a QA flow for stalled visible-turn recovery.

## Best Possible Solution

Keep active registry state for legitimate no-backend in-flight phases, distinguish truly orphaned/detached runs or force-clear only after stuck recovery proves inactivity, then validate with focused registry coverage plus real qa-lab or live-channel recovery proof.

## Reproduction Assessment

No, not for the full live stalls in https://github.com/openclaw/openclaw/issues/84903 and https://github.com/openclaw/openclaw/issues/84755. The narrow registry subcase is source-reproducible: current main aborts a running no-backend reply operation without clearing it, so waiters can remain blocked until force-clear.

## Solution Assessment

No. The fix should not treat every no-backend operation as orphaned because valid runs can be active before backend attachment; the safer solution is explicit orphan tracking or recovery-scoped force-clear after inactivity proof.
