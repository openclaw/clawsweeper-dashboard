## What This Changes

The PR changes reply-run abort cleanup to immediately clear no-backend operations and adds a unit test plus a QA scenario for stalled visible-turn queue recovery.

## Best Possible Solution

Land a scoped registry fix that clears orphaned no-backend running turns immediately, keeps compaction and memory-flush state active until their cleanup finishes, and proves the recovery with focused tests plus redacted real QA or gateway output.

## Reproduction Assessment

Source-reproducible, not executed: current main leaves a `running` reply operation with no backend active after `abortReplyRunBySessionId` because `abortByUser` only clears queued operations. The broader Telegram/Feishu live stall still lacks a deterministic local reproduction in this review.

## Solution Assessment

No: the useful direction is narrow, but this implementation treats every no-backend phase as orphaned. The safer fix is to scope immediate clearing to the `running` phase and preserve compaction/memory-flush ownership until their async cleanup completes.
