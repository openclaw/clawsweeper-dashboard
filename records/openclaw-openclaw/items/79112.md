## Summary

Keep open: the reported runtime alias failure is valid and this PR targets the right postbuild surface, but the diff introduces a blocking resolver regression for `install.runtime.js` by bypassing its marker-based collision handling.

## What This Changes

This PR changes `runtime-postbuild` to ignore existing stable-alias compatibility wrappers when choosing a hashed root runtime alias, adds a regression test, and adds a changelog fix entry.

## Best Possible Solution

Keep the wrapper-filtering direction, but preserve marker-based resolution for `install.runtime.js` and add regression coverage with stale install compatibility wrappers plus a non-plugin install chunk.

## Reproduction Assessment

Yes, source-reproducible: current main fails the described shape when old `runtime-plugins.runtime-*` wrappers re-export the stable alias because they are counted as colliding candidates and no stable alias is written. I did not run a live TUI send in this read-only review.

## Solution Assessment

No, not yet: filtering stable-alias wrappers is the right direction, but the proposed generic single-candidate return must not bypass the special marker contract for `install.runtime.js`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve marker checks for install runtime aliases:** `scripts/runtime-postbuild.mjs:170-171`
  - body: `install.runtime.js` is intentionally resolved through plugin-install markers because that basename can collide with other install chunks. After filtering out stale wrappers, this generic single-candidate return runs before that marker check, so a dist tree with old `install.runtime-*` wrappers and one non-plugin install chunk would write the stable alias to the wrong module. Keep the marker-based path for `PLUGIN_INSTALL_RUNTIME_ALIAS` and cover this wrapper-collision case.
  - confidence: 0.87
