## Summary

Keep open: the PR is still useful and not superseded, but the current head still has a compatibility false positive in the proposal-marker count. A complete update for an existing skill whose preserved H1 contains "update proposal" can still be rejected if the body also contains one normal proposal-like section such as "Proposed changes".

## What This Changes

The branch adds Skill Workshop apply-time validation and regression tests so update proposals that look like delta notes, empty bodies, or identity swaps are rejected before replacing an existing SKILL.md.

## Best Possible Solution

Keep the shared apply-time guard, but make preserved identity headings neutral for proposal-marker counting while preserving rejection for non-preserved wrapper drafts.

## Reproduction Assessment

Yes. Source inspection shows current main writes the stripped update body directly, and PR head source shows the remaining false positive when a preserved update-proposal H1 contributes to the two-marker threshold.

## Solution Assessment

No, not quite yet. The shared service apply boundary is the right place for this guard, but preserved identity headings should not contribute to wrapper-note marker counts.
