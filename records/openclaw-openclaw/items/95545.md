## Summary

Keep open. The PR is aimed at the right shared apply boundary, but it still leaves one supported Skill Workshop update case exposed: a one-marker proposal-note heading can overwrite an H1-less existing skill, and the shipped fail-closed behavior remains compatibility-sensitive.

## What This Changes

The PR adds Skill Workshop apply-time validation for update proposals and regression tests around delta-note rejection, empty bodies, H1 identity handling, H1-less skills, and complete rewrites.

## Best Possible Solution

Add a narrow guard and regression test for proposal-like H1 drafts on H1-less skills, then land the shared apply-boundary guard once maintainers accept the fail-closed compatibility change.

## Reproduction Assessment

Yes from source: current main strips proposal frontmatter and writes the remaining body directly during update apply, so a delta-style draft can replace the live SKILL.md. The PR's remaining gap is also source-reproducible for H1-less skills because one-marker proposal headings are not rejected and the H1 identity check is skipped.

## Solution Assessment

No, not yet. The shared apply boundary is the best layer, but the guard should also reject proposal-like H1 headings for H1-less existing skills before this is the narrow maintainable fix.
