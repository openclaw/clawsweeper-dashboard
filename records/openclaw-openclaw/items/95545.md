## Summary

Keep this PR open: the apply-time guard is the right boundary and the proof is useful, but the current strong-marker ordering still creates a compatibility false positive for complete updates whose preserved skill H1 contains “update proposal.”

## What This Changes

The PR adds Skill Workshop apply-time validation and regression tests so update proposals that look like delta/proposal notes, empty bodies, or identity swaps are rejected before overwriting `SKILL.md` while complete rewrites remain allowed.

## Best Possible Solution

Keep the apply-time guard, but exempt identity-preserving H1 headings from the strong proposal-marker rejection and add the focused regression before merge.

## Reproduction Assessment

## Solution Assessment

No, not quite yet. The service apply boundary is the right place for this guard, but the marker ordering should preserve complete updates when the old and new H1 normalize to the same identity.
