## Summary

Keep open: current main still has single-target cron delivery, but this PR is not merge-ready because the patch still has blocking delivery-mode/session-routing bugs and lacks required after-fix real behavior proof.

## What This Changes

The PR adds `delivery.additionalTargets` cron fan-out across cron types, gateway schema, normalization, runtime dispatch, CLI edit flags, and tests, plus lazy channel-runtime access in `commands-session.ts`.

## Best Possible Solution

Land one rebased, documented cron fan-out implementation that keeps schema, normalization, delivery planning, dispatch, trace/status reporting, CLI/UI surfaces, and tests aligned, with explicit behavior for webhook/no-delivery modes and job-session `last` routing.

## Reproduction Assessment

Yes. The two blocking behaviors are source-reproducible from the PR head: webhook delivery accepts `additionalTargets` but never reaches the direct-send fan-out gate, and `channel: "last"` additional targets use a different session key than the primary target.

## Solution Assessment

No. The feature direction is plausible, but this patch should first rebase onto current main, define supported delivery modes, resolve `last` against the job delivery session, and add real runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/gateway/protocol/schema/cron.ts:191`
  - body: Adding `additionalTargets` to the shared delivery schema makes webhook-mode jobs accept the field, but the runtime only fans out when a primary direct send sets `sendOccurred`. Webhook delivery is posted later outside this path, so webhook jobs silently drop every additional target; reject the field for webhook mode or dispatch fan-out from the webhook completion path too.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets pass the cron run's `agentSessionKey` into `resolveDeliveryTarget`, while the primary delivery resolves from `params.job.sessionKey`. For `channel: "last"`, session-bound jobs can route fan-out from the wrong fresh cron session or fail closed; pass the job delivery session key through this helper and cover it with a regression test.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces one concrete privacy-sensitive routing risk in additional-target `last` resolution.

Concerns:

- **[medium] Wrong-session last target can disclose cron output:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Using the run `agentSessionKey` instead of the job delivery session key for additional targets can resolve `channel: "last"` from a different conversation context than the primary delivery, potentially sending cron output to the wrong chat or thread.
