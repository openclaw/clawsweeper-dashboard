## Summary

Keep open. Current main still has only single-target cron delivery, and this PR is the active implementation candidate, but the branch is not mergeable as-is because it accepts webhook fan-out without an execution path and can resolve `channel: "last"` additional targets from the wrong session.

## What This Changes

The PR adds cron `delivery.additionalTargets` fan-out across cron types, protocol schema, normalization, runtime dispatch, CLI edit support, and tests, with adjacent lazy channel-runtime cleanup.

## Best Possible Solution

Land one documented cron fan-out contract that keeps schema, normalization, delivery planning, runtime dispatch, trace/status reporting, CLI/UI surfaces, docs, and regression tests aligned, with explicit behavior for announce, webhook, no-delivery, primary-success gating, `channel: "last"`, and per-target failure isolation.

## Reproduction Assessment

Yes for the review findings. Static inspection of the PR head shows webhook-mode `additionalTargets` is accepted by the shared schema but fan-out only runs after direct primary sends, and additional `last` targets are resolved with the run session key.

## Solution Assessment

No. The feature direction is useful, but this branch is not the best mergeable solution until supported modes and job-session routing are fixed or explicitly rejected in the contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/gateway/protocol/schema/cron.ts:191`
  - body: `additionalTargets` is added to the shared delivery schema, so webhook-mode cron jobs can accept and persist extra targets. The runtime fan-out only runs when `deliveryResult.sendOccurred` is true, while webhook delivery is posted later through gateway notifications, so webhook jobs silently skip all additional targets.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets can use `channel: "last"`, but this call resolves them with `params.agentSessionKey`. Primary delivery resolves from the job delivery session, so session-bound cron jobs can fan out to the wrong conversation or fail unexpectedly; pass the job delivery session key through and cover it with a regression test.
  - confidence: 0.86
