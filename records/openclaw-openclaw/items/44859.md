## Summary

Keep open. Current main still has no cron `additionalTargets` implementation, but this external XL feature PR is not merge-ready because webhook-mode configs accept the new field without delivering fan-out, `channel: "last"` fan-out still resolves from the cron run session instead of the job delivery session, the branch is conflicting, and real behavior proof is missing.

## What This Changes

The PR adds a `delivery.additionalTargets` cron fan-out config/API/CLI/runtime path with normalization and tests, plus lazy channel-runtime access in `commands-session.ts`.

## Best Possible Solution

Land one canonical, documented cron fan-out implementation that keeps schema, normalization, delivery planning, runtime dispatch, trace/status reporting, CLI/UI surfaces, docs, changelog, and regression tests aligned, with explicit behavior for announce, webhook, no-delivery, primary-success gating, `channel: "last"`, and per-target failure isolation.

## Reproduction Assessment

Yes, the blocking review findings are source-reproducible: the PR schema accepts webhook additional targets while runtime fan-out is gated on direct-send state, and additional target resolution uses the generated run session key instead of the job delivery session. I did not run a live cron fan-out setup.

## Solution Assessment

No. The direction is useful, but the current branch should not merge until it either supports or rejects webhook fan-out, resolves additional `last` targets from the same delivery session as primary routing, rebases onto current main, and includes real after-fix behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/gateway/protocol/schema/cron.ts:191`
  - body: Adding `additionalTargets` to the shared delivery schema makes webhook-mode cron jobs accept and persist extra targets, but fan-out only runs when isolated direct delivery reports `sendOccurred`. Webhook delivery is posted later by the gateway notification path, so these accepted targets are silently skipped; reject the field outside supported modes or dispatch it from the webhook completion path too.
  - confidence: 0.9
- **[P2] Resolve fan-out targets with the delivery session:** `src/cron/isolated-agent/run.ts:784`
  - body: Primary delivery resolves against the job delivery session key, but fan-out passes the generated cron run session key into additional target resolution. For `channel: "last"` or session-derived account/thread metadata this can route from the wrong fresh cron session or fail unexpectedly; pass the same job delivery session key used for primary routing and cover it with a regression test.
  - confidence: 0.84
