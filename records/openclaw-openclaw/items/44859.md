## Summary

Keep open. Current main still lacks cron multi-target delivery, but this PR is not merge-ready because webhook-mode targets are accepted without a fan-out path, `last` additional targets resolve against the cron run session, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds cron `delivery.additionalTargets` across cron types, gateway schema, normalization, delivery dispatch, CLI edit flags, tests, and a lazy channel-runtime access change.

## Best Possible Solution

Land one canonical, documented cron fan-out implementation that keeps schema, normalization, delivery planning, runtime dispatch, trace/status reporting, CLI/docs/UI surfaces, upgrade behavior, and regression tests aligned for each accepted delivery mode.

## Reproduction Assessment

Yes for the PR findings: source inspection of PR head shows `additionalTargets` are accepted through the shared schema for webhook delivery, while fan-out only runs from the direct-send `sendOccurred` path, and additional `last` targets use the cron run session key. The requested multi-target delivery itself is a new feature rather than a failing current-main bug.

## Solution Assessment

No, not yet. The direction is plausible, but the implementation should first make delivery-mode support explicit, resolve session-derived targets from the same job delivery context as primary delivery, add real behavior proof, and rebase against current cron delivery-plan files.

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/gateway/protocol/schema/cron.ts:191`
  - body: `additionalTargets` is added to the shared delivery properties, so webhook-mode cron jobs can now accept and persist extra targets. The runtime fan-out is gated on `deliveryResult.sendOccurred`, which webhook delivery never sets because webhook completion is dispatched through gateway notifications, so those extra targets are silently skipped. Please either reject `additionalTargets` for unsupported modes or wire fan-out into the webhook completion path with tests.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:762-767`
  - body: Additional targets are resolved with `sessionKey: params.agentSessionKey`, which is the fresh cron run session. Primary delivery resolves against the job delivery session key, so `channel: "last"` or session-derived account/thread metadata can come from the wrong session or fail unexpectedly. Pass the job delivery session context into this helper and cover `last` fan-out with a regression test.
  - confidence: 0.86
