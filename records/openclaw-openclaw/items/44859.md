## Best Possible Solution

Land one canonical, documented cron fan-out/output-routing contract that aligns schema, normalization, delivery planning, runtime dispatch, trace/status reporting, CLI/UI surfaces, docs, regression tests, and real behavior proof.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/gateway/protocol/schema/cron.ts:191`
  - body: `additionalTargets` is added to shared delivery properties, so webhook-mode jobs can accept and persist extra targets. The fan-out path only runs after direct announce delivery reports `sendOccurred`, while webhook delivery is posted later by gateway notifications, so those accepted extra targets are silently skipped.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:762-767`
  - body: Additional targets can use `channel: "last"`, but this resolves them with the cron run session key. Primary delivery resolves against the job's configured session key, so session-bound jobs can route fan-out to the wrong chat/thread or fail unexpectedly.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Needs attention because session-derived additional targets can resolve against the wrong session and misroute scheduled output.

Concerns:

- **[medium] Wrong-session fan-out can misroute cron output:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets using `channel: "last"` are resolved with `params.agentSessionKey` instead of the job delivery session context, which can send scheduled output to an unintended chat/thread or fail closed depending on stored session state.
  - confidence: 0.78
