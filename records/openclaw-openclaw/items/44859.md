## Summary

Keep open. Current main still has single-target cron delivery, and this branch remains useful, but it is not mergeable yet because real behavior proof is missing and the latest PR head still has source-reproducible webhook and session-routing defects.

## What This Changes

The PR adds cron `delivery.additionalTargets` fan-out across cron types, gateway schema, normalization, dispatch, CLI edit flags, and tests, plus lazy channel-runtime access in session command handling.

## Best Possible Solution

Land one canonical, documented cron fan-out or output-routing contract that aligns schema, normalization, delivery planning, runtime dispatch, CLI/UI/docs, status tracing, webhook behavior, `channel: "last"`, and per-target failure isolation.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head `556d481b344faf49a2f1c08d789c6a34a663d259` shows webhook `additionalTargets` are accepted but have no fan-out path, and `channel: "last"` additional targets resolve with the run agent session key. The requested feature itself is not a bug repro because current main has no multi-target cron delivery contract.

## Solution Assessment

No. The branch is a plausible implementation direction, but it is not the best mergeable solution until delivery-mode semantics, session routing, current-main integration, documentation, and real behavior proof are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/gateway/protocol/schema/cron.ts:191`
  - body: `additionalTargets` is added to shared delivery properties, so webhook-mode cron jobs can now accept and persist extra targets. The runtime fan-out only runs after a direct announce send sets `sendOccurred`, while webhook delivery is posted later through gateway notifications, so webhook jobs silently skip every additional target. Please either reject `additionalTargets` for unsupported modes or trigger fan-out from the webhook completion path too.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets may use `channel: "last"`, but this path resolves them with `params.agentSessionKey`. Primary cron delivery resolves session-derived targets from the job delivery session, so session-bound jobs can route fan-out to the wrong chat/thread or fail unexpectedly. Pass the job delivery session key into this helper and add regression coverage.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete outbound-routing risk for session-derived additional cron targets.

Concerns:

- **[medium] Wrong-session fan-out can misroute cron output:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets using `channel: "last"` are resolved from the cron run's agent session key instead of the job's delivery session key, which can send scheduled output to an unintended chat/thread or fail depending on stored session state.
