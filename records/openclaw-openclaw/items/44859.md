## Summary

Keep open. Current main still has a single-target cron delivery contract, and this external XL feature PR is not mergeable yet because the branch still has concrete delivery-mode/session-routing defects plus missing after-fix real behavior proof.

## What This Changes

The PR adds cron `delivery.additionalTargets` fan-out across cron types, gateway schema, normalization, runtime delivery dispatch, CLI edit flags, and tests, plus lazy channel-runtime access in session command handling.

## Best Possible Solution

Land one canonical, documented cron output/fan-out contract that keeps schema, normalization, announce/webhook runtime dispatch, session-key routing, traces/status, CLI/UI/docs, tests, and real behavior proof aligned.

## Reproduction Assessment

Yes for the PR findings: source inspection of PR head `556d481b344faf49a2f1c08d789c6a34a663d259` shows webhook `additionalTargets` are accepted without a webhook fan-out path, and `channel: "last"` fan-out resolves with the run session key. The requested multi-target delivery capability itself is a feature request, not a current-main bug reproduction.

## Solution Assessment

No. The branch is a plausible implementation direction, but it is not the best mergeable solution until webhook semantics, job-session routing, trace/status/docs coverage, real behavior proof, and the relationship to the broader cron output-routing contract are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/gateway/protocol/schema/cron.ts:191`
  - body: `additionalTargets` is added to the shared delivery properties, so webhook-mode cron jobs can validate and persist extra targets. The runtime fan-out is gated on direct announce delivery's `sendOccurred`, while webhook delivery is posted later by gateway notifications, so webhook jobs silently skip all additional targets. Restrict this field to supported modes or trigger fan-out from the webhook completion path too.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets can use `channel: "last"`, but `deliverToAdditionalTargets` passes the cron run's `agentSessionKey` into `resolveDeliveryTarget`. Primary delivery resolves against the job's configured delivery session, so session-bound jobs can resolve additional targets from the wrong fresh cron session or fail closed. Pass the job delivery session key into this helper and cover it with a regression test.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff has no CI or dependency supply-chain changes, but it introduces a concrete outbound-routing risk for session-derived additional cron targets.

Concerns:

- **[medium] Wrong-session fan-out can misroute cron output:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets using `channel: "last"` are resolved with the cron run session key instead of the job delivery session context, which can send scheduled output to an unintended chat/thread or fail depending on stored session state.
