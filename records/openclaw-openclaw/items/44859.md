## Summary

Keep open. Current main still has single-target cron delivery, but this PR is dirty against main and still has merge-blocking fan-out semantics around webhook delivery, session-bound `last` routing, and current threaded-delivery support.

## What This Changes

The PR adds cron `delivery.additionalTargets` fan-out across cron types, gateway schema, normalization, runtime dispatch, CLI edit flags, and tests, plus lazy channel-runtime access in `commands-session.ts`.

## Best Possible Solution

Land one canonical, documented cron fan-out implementation that preserves current delivery-plan/threadId behavior and explicitly defines announce, webhook, no-delivery, primary-success, `channel: "last"`, status/trace, CLI/UI/docs, changelog, and failure-isolation semantics.

## Reproduction Assessment

Yes for the review findings. Static inspection of the PR head against current main is enough: shared schema accepts webhook additional targets, fan-out only runs after direct-send `sendOccurred`, and additional `last` targets resolve with the run session key rather than the job delivery session key.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the best mergeable solution until it is rebased onto current main, reconciled with #48194, and fixed for webhook semantics, session routing, threaded delivery preservation, docs, and changelog.

  - body: Current main accepts and tests `delivery.threadId` for cron add/update and carries it through `CronDeliveryPlan`, but this branch's delivery schema/plan/type surface omits `threadId` while adding `additionalTargets`.   - body: `additionalTargets` is added to the shared delivery schema, so webhook-mode jobs can persist extra targets. The runtime fan-out is gated on `deliveryResult.sendOccurred`, which only comes from direct announce delivery, while webhook completion is dispatched through the gateway notification path; webhook jobs will silently skip all configured additional targets.
  - confidence: 0.9

  - body: Additional targets may use `channel: "last"`, but this call resolves them using `params.agentSessionKey`. Primary cron delivery resolves against the job's delivery session key, so session-bound cron jobs can send fan-out to the wrong conversation or fail unexpectedly. Pass the job delivery session key into this helper and cover it with a regression test.
  - confidence: 0.87
