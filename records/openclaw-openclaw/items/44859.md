## Summary

Keep open. Current main still has single-target cron delivery, but this PR is not mergeable as-is because source inspection shows webhook-mode additional targets can be accepted but skipped, session-derived additional targets can resolve from the wrong session, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds cron `delivery.additionalTargets` fan-out across cron types, gateway schema, normalization, runtime delivery dispatch, CLI edit flags, and tests, plus lazy channel-runtime access in session command handling.

## Best Possible Solution

Land one canonical, documented cron output or fan-out contract that aligns schema, normalization, delivery planning, runtime dispatch, trace/status reporting, CLI/UI/docs, webhook behavior, `channel: "last"`, and per-target failure isolation.

## Reproduction Assessment

Yes for the PR findings: source inspection of PR head `556d481b344faf49a2f1c08d789c6a34a663d259` shows webhook `additionalTargets` are accepted but have no fan-out path, and `channel: "last"` additional targets resolve with the run agent session key. The requested multi-target delivery capability itself is a feature request, not a current-main bug reproduction.

## Solution Assessment

No. The branch is a plausible implementation direction, but it is not the best mergeable solution until webhook support or rejection, session routing, current-main integration, documentation, and real behavior proof are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Either support or reject webhook fan-out:** `src/gateway/protocol/schema/cron.ts:191`
  - body: `additionalTargets` is added to the shared delivery schema, so webhook-mode cron jobs can now accept and persist extra targets. The runtime fan-out, however, only runs when `deliveryResult.sendOccurred === true`, while webhook delivery is posted later by the gateway notification path, so webhook jobs silently skip every additional target. Please either reject `additionalTargets` outside supported modes or trigger fan-out from the webhook completion path too.
  - confidence: 0.9
- **[P2] Resolve last targets from the job session:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets can use `channel: "last"`, but `deliverToAdditionalTargets` passes the cron run's `agentSessionKey` into `resolveDeliveryTarget`. Session-bound jobs can therefore resolve fan-out from the wrong run session or fail unexpectedly; pass the job delivery session key into this helper and add regression coverage.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff has no dependency or CI supply-chain changes, but it introduces a concrete outbound-routing risk for session-derived additional cron targets.

Concerns:

- **[medium] Wrong-session fan-out can misroute cron output:** `src/cron/isolated-agent/delivery-dispatch.ts:766`
  - body: Additional targets using `channel: "last"` are resolved from the cron run's agent session key instead of the job delivery/session context, which can send scheduled output to an unintended chat/thread or fail depending on stored session state.
