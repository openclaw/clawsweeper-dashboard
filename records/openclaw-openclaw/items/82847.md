## Summary

Keep open: current main still has the reported owner-boundary bug. The primary reply path distinguishes CLI backends from embedded harnesses, but queued followup drains still call the embedded Pi runner directly, so a CLI runtime can reach harness selection as an unregistered harness id.

## What This Changes

The issue requests routing queued followup drains for CLI runtimes through the CLI runner instead of the embedded Pi harness path.

## Best Possible Solution

Add the same CLI-runtime dispatch used by the primary reply path to queued followup execution, with regression coverage for CLI and embedded runtimes.

## Reproduction Assessment

Yes from source inspection, though not from a live run: queued followups still always call `runEmbeddedPiAgent`, and harness selection fails closed for CLI runtime ids that are not embedded harnesses.

## Solution Assessment

Yes; mirroring the primary reply path's CLI dispatch in `createFollowupRunner` is the narrowest maintainable fix and preserves the documented split between CLI backends and embedded harnesses.
