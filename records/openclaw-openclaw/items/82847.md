## Summary

Close as duplicate/superseded. The reported queued-followup CLI dispatch failure is already covered by the broader open CLI-backed helper-path issue, and the open implementation candidate for that canonical item explicitly includes queued followup runs.

## What This Changes

The issue asks for queued followup drains to dispatch CLI-runtime selections through `runCliAgent` instead of the embedded harness path.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/57326 as the canonical tracker and land a rebased https://github.com/openclaw/openclaw/pull/72499 or equivalent replacement that routes queued followups through the shared CLI dispatch path.

## Reproduction Assessment

Yes, at source level. `createFollowupRunner` still sends queued runs through `runEmbeddedPiAgent`, while `agentRuntime.id: "claude-cli"` is documented as a CLI backend alias that must not enter AgentHarness selection, where unregistered runtime ids throw.

## Solution Assessment

Yes for the requested direction, but it should be handled under the canonical tracker. The maintainable fix is to reuse the primary `resolveCliRuntimeExecutionProvider` plus `isCliProvider` plus `runCliAgent` pattern while preserving queued-run auth, session, routing, and delivery semantics.
