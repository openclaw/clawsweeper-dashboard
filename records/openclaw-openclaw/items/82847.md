## Summary

Keep open: current main still has the reported queued-run routing bug, because followup drains call the embedded Pi runner directly while the primary reply path has a separate CLI-runtime dispatch branch.

## What This Changes

The issue asks OpenClaw to route queued followup drains for CLI runtimes through the CLI runner instead of embedded harness selection.

## Best Possible Solution

Route queued followup fallback attempts through the same CLI-runtime provider resolution and `runCliAgent` path used by primary replies, while preserving embedded Pi behavior for non-CLI runtimes.

## Reproduction Assessment

Yes, source-reproducible: configure a canonical Anthropic or Google model with `agentRuntime.id` set to a CLI backend, enqueue a followup while the active turn is running, and the drain reaches `runEmbeddedPiAgent` instead of `runCliAgent`. I did not run a live Signal/CLI scenario in this read-only pass.

## Solution Assessment

Yes; mirroring or sharing the primary reply path's CLI dispatch in `createFollowupRunner` is the narrowest maintainable fix, and it keeps CLI backends separate from embedded harness ids.
