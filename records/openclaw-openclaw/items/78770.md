## Summary

Keep this PR open: the bug and fix boundary are valid, and real WhatsApp proof is supplied, but the latest head still drops an entire deferred media-only payload when any URL overlaps a later captioned payload, which can lose unrelated attachments.

## What This Changes

The PR buffers WhatsApp media-only interim auto-reply payloads, coalesces them with a later captioned payload for the same media, updates dispatcher coverage, and adds a changelog entry.

## Best Possible Solution

Keep dispatcher-level coalescing, but split or filter deferred media by URL so only duplicated media is suppressed while unrelated attachments still flush normally, with focused mixed-overlap regression coverage.

## Reproduction Assessment

Yes for the underlying bug. Current main's dispatcher and test show the duplicate lifecycle path, and the linked report plus PR body provide redacted live WhatsApp logs and screenshots for the same behavior.

## Solution Assessment

No, not yet. The dispatcher is the right boundary, but the current PR implementation is too broad because it suppresses a whole pending media payload instead of only the duplicated URLs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve non-overlapping buffered media:** `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:171-174`

Filter or split the pending payload so only duplicated URLs are suppressed.
