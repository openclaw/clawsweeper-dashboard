## Summary

Keep open: the fix direction and supplied WhatsApp proof are useful, but the patch can drop unrelated buffered media when only one URL overlaps a later captioned payload, so it needs a small correctness repair before merge.

## What This Changes

The PR buffers non-final WhatsApp media-only auto-reply payloads, drops same-media interim payloads when a later captioned payload arrives, updates dispatcher coverage, and adds a changelog entry.

## Best Possible Solution

Keep the dispatch-level coalescing approach, but suppress only duplicated media URLs while preserving unrelated buffered media, then land with the supplied live WhatsApp proof and green checks.

## Reproduction Assessment

Yes. Current main source and the existing dispatcher test show tool media `/tmp/generated.jpg` delivered once as media-only and again as captioned block media; the linked issue also provides live WhatsApp duplicate-send logs.

## Solution Assessment

No, not quite. Dispatcher-level coalescing is the right boundary, but dropping an entire buffered payload on partial URL overlap can lose attachments; the safer repair filters or splits only the duplicated URLs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve non-overlapping buffered media:** `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:411-412`
  - body: When a pending media-only payload contains multiple `mediaUrls`, `flushPendingMediaOnlyPayloads` drops the whole payload if any URL overlaps the later captioned payload. A turn with `[a, b]` followed by captioned `[a]` will now lose `b`; filter or split the buffered payload so only duplicated URLs are suppressed.
