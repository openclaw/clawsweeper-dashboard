## Summary

Keep open: the WhatsApp duplicate-media bug is valid and the PR targets the right dispatcher boundary, but the current coalescer can suppress non-duplicated attachments when only one URL overlaps a later captioned payload.

## What This Changes

The PR buffers WhatsApp media-only interim auto-reply payloads, drops same-URL duplicates before a later captioned media payload, updates dispatcher coverage, adds a changelog entry, and includes a task-command test reset change.

## Best Possible Solution

Keep dispatcher-level dedupe, but make it URL-granular so only duplicated media URLs are suppressed while unrelated deferred media still flushes, with mixed-overlap regression coverage.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path through the dispatcher test and delivery code, and the linked report plus this PR provide redacted live WhatsApp logs and screenshots for the same duplicate-send behavior.

## Solution Assessment

No, not yet. The dispatcher is the right fix boundary, but the current implementation suppresses a whole deferred payload on any URL overlap instead of preserving non-overlapping media.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve non-overlapping buffered media:** `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:171-173`
  - body: When a deferred media-only payload contains multiple `mediaUrls`, this skips the whole payload if any URL overlaps the later captioned payload. A pending `[a, b]` followed by captioned `[a]` would drop `b`, even though multi-attachment replies are supported and WhatsApp delivery sends the full list. Filter or split the pending payload so only duplicated URLs are suppressed.
  - confidence: 0.87
