## Summary

Keep open: the underlying WhatsApp duplicate-send bug is real and the PR targets the right dispatcher boundary, but the latest patch still drops an entire deferred media-only payload when only one URL overlaps a later captioned payload.

## What This Changes

The PR adds a WhatsApp inbound auto-reply coalescer, updates dispatcher coverage, and adds a changelog entry so captioned `MEDIA:` replies send once instead of first sending empty media.

## Best Possible Solution

Keep dispatcher-level coalescing, but make it URL-granular so duplicate interim media is suppressed while unrelated deferred attachments still flush normally.

## Reproduction Assessment

Yes for the underlying bug. Current main's source and existing dispatcher test show the duplicate lifecycle delivery path, and the linked issue plus PR body provide redacted live WhatsApp logs and screenshots for the same behavior.

## Solution Assessment

No, not quite. The dispatcher boundary is the right place to fix the duplicate, but the current implementation is too broad because it suppresses an entire pending media payload instead of only duplicated URLs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve non-overlapping buffered media:** `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:171-174`
  - body: When a deferred media-only payload contains multiple `mediaUrls`, this skips the whole payload if any URL overlaps the later captioned payload. A pending `[a, b]` followed by captioned `[a]` would drop `b`, even though multi-attachment replies are supported and WhatsApp delivery sends the full list. Filter or split the pending payload so only duplicated URLs are suppressed.
  - confidence: 0.86
