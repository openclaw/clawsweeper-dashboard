## Summary

Keep open: current main still has the source-level mismatch described in the report, and the docs/manifest contract indicate startup-lazy channel plugins should remain correct when reached through a narrower activation trigger.

## What This Changes

The issue reports that isolated cron announce delivery to Slack fails after Slack plugin externalization because the cron delivery target path does not bootstrap startup-lazy channel plugins.

## Best Possible Solution

Add a targeted bootstrap path for runtime delivery target resolution, preserving startup laziness and avoiding eager bootstrap on unrelated outbound hot paths.

## Reproduction Assessment

Yes, from source inspection: Slack is startup-lazy, the channel resolver only bootstraps with `allowBootstrap: true`, and isolated cron delivery calls the target resolver without that option. A live Slack/Homebrew run was not executed in this read-only pass.

## Solution Assessment

Yes, the maintainable fix is to thread an explicit bootstrap allowance through the isolated runtime delivery target path rather than making Slack startup-eager globally.
