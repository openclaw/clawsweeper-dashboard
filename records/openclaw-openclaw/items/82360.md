## Summary

Keep open: current main still has a clear source-level regression where isolated cron announce delivery can reach an externalized, non-startup Slack plugin only if the plugin is already loaded. This is a narrow bug in the outbound target bootstrap path, not a stale duplicate of the older registry-pinning fixes.

## What This Changes

The issue asks OpenClaw to restore isolated cron `delivery.mode: announce` delivery to Slack after Slack was externalized into an `activation.onStartup: false` plugin.

## Best Possible Solution

Thread an explicit runtime bootstrap allowance through cron announce target resolution so configured external channel plugins can be lazily loaded, then add focused coverage for an unloaded `activation.onStartup: false` channel.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: with Slack not already loaded, isolated cron delivery calls `resolveOutboundTarget` without `allowBootstrap`, so the channel resolver will not bootstrap the externalized Slack plugin. I did not execute the live Slack/Homebrew flow in this read-only pass.

## Solution Assessment

Yes, the best fix is a narrow core bootstrap-threading change for runtime delivery paths, not making the Slack manifest always start up. That keeps externalization lazy while matching the existing opt-in bootstrap contract used by delivery recovery paths.
