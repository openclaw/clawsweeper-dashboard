## Summary

Current main still drops Baileys CTWA referral data before WhatsApp inbound messages reach plugin hooks or agent context, so the request remains useful; however, exposing the identifier to plugins, injecting it into prompts, and persisting it in session state are distinct privacy and public-contract decisions that need maintainer approval rather than an automatic fix.

## What This Changes

Expose Click-to-WhatsApp referral identifiers and related ad metadata from inbound WhatsApp messages to agent or plugin session context.

## Best Possible Solution

Approve a minimal typed CTWA referral contract—preferably an optional WhatsApp-owned structured field propagated to the existing plugin inbound hooks—then separately decide whether trusted agent prompts or SQLite-backed session state may consume it, with explicit redaction, retention, and first-message semantics.

## Reproduction Assessment

Yes, at source level: Baileys exposes ContextInfo.ctwaClid, while current OpenClaw WhatsApp inbound and canonical hook types have no field that can carry it, so it is dropped before plugin or agent access. A live CTWA-origin message was not exercised.

## Solution Assessment

No, not as one combined implementation: narrowly exposing typed referral metadata to plugin hooks is maintainable, but automatic prompt inclusion and session persistence require separate privacy and lifecycle decisions.
