## Summary

Keep open. Current main partially handles WhatsApp `externalAdReply` metadata for GIF/media context, but it still does not expose the requested CTWA click identifier or define the retention/prompt/plugin-hook contract needed for ad attribution metadata.

## What This Changes

Request to expose Click-to-WhatsApp referral and click-id metadata from inbound WhatsApp messages to agent or session context for Meta attribution workflows.

## Best Possible Solution

Define a WhatsApp-owned CTWA attribution payload that extracts `ctwaClid` and related source fields, exposes them through the existing untrusted structured-context or plugin-hook surface, and persists them only after an explicit security/retention decision.

## Reproduction Assessment

Yes, source-level: a WhatsApp Baileys message carrying `externalAdReply.ctwaClid` would pass through the current extractor path, which only returns title, sourceUrl, and body. I did not run a live Click-to-WhatsApp ad webhook reproduction.

## Solution Assessment

Unclear as proposed: extracting in the WhatsApp plugin is the right owner, but broad session metadata or preamble exposure is not automatically the safest contract. A typed untrusted structured-context or plugin-hook payload with explicit retention rules is the safer likely direction.
