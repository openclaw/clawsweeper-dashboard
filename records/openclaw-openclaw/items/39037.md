## Summary

Keep open: current main still extracts raw WhatsApp `contextInfo` but does not normalize or expose CTWA ad-click metadata, and the remaining decision is where that identifier may be exposed or stored. Related WhatsApp metadata threads cover GIF media metadata and forwarding signals, not the ad-attribution `ctwaClid` use case.

## What This Changes

Expose Click-to-WhatsApp ad referral/click metadata from inbound WhatsApp messages to session, prompt, or plugin metadata.

## Best Possible Solution

Add a WhatsApp-owned typed extractor for `externalAdReply.ctwaClid` and approved ad-referral fields, then expose them through the maintainer-approved untrusted context, hook, or session surface with focused tests.

## Reproduction Assessment

Yes, source-level. A synthetic Baileys inbound message with `contextInfo.externalAdReply.ctwaClid` would reach raw `extractContextInfo`, but current main has no carrier into queued messages, prompt metadata, hook metadata, or session metadata.

## Solution Assessment

Unclear as proposed. The direction is valid, but broadly copying referral data into session metadata or preamble is not the best fix until maintainers choose a narrow, typed, privacy-aware exposure surface.
