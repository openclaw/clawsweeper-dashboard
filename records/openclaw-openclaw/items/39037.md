## Summary

Keep open. Current main still exposes a path for untrusted WhatsApp structured metadata, but it never extracts CTWA fields from Baileys `contextInfo.externalAdReply`, so the reporter's attribution use case remains unimplemented; related #69374 is a neighboring contextInfo exposure request, not a replacement.

## What This Changes

Expose Click-to-WhatsApp ad referral fields, especially `ctwaClid`, from inbound WhatsApp messages to agent/session metadata.

## Best Possible Solution

Add a WhatsApp-owned normalizer for a minimal CTWA/referral payload from `contextInfo.externalAdReply` into `UntrustedStructuredContext`, with focused tests, and leave durable session storage out of scope for the first fix.

## Reproduction Assessment

Yes. A synthetic Baileys inbound message with `extendedTextMessage.contextInfo.externalAdReply.ctwaClid` can be traced through the current extractor, monitor, and context builder; the field is reachable but never normalized or forwarded.

## Solution Assessment

Yes. The narrow maintainable solution is to keep this inside the WhatsApp plugin and reuse the existing untrusted structured metadata path instead of adding core session schema or broad hook changes first.
