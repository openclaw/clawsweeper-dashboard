## Summary

Keep open. Current main and the latest release expose only partial WhatsApp external-ad-reply metadata, while the requested CTWA click identifier and its security/retention contract remain undefined.

## What This Changes

Expose Click-to-WhatsApp attribution fields from inbound WhatsApp metadata to agents or session context.

## Best Possible Solution

Define a WhatsApp-owned CTWA attribution payload, expose it through the existing untrusted structured-context or plugin-hook boundary, and persist it only after an explicit retention/security decision.

## Reproduction Assessment

Yes, source-level: a WhatsApp Baileys message carrying externalAdReply.ctwaClid would pass through the current extractor path, which returns only title, sourceUrl, and body. I did not run a live Click-to-WhatsApp ad webhook reproduction.

## Solution Assessment

Unclear as proposed: extraction belongs in the WhatsApp plugin, but broad session metadata or preamble exposure is not automatically the safest contract. The safer likely direction is a typed untrusted structured-context or plugin-hook payload with explicit retention rules.
