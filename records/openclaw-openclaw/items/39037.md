## Summary

Keep open. Current main still drops Click-to-WhatsApp ad metadata before it can reach agents, plugin hooks, or session metadata: the WhatsApp plugin can find Baileys `contextInfo`, but the normalized inbound message only carries replies, mentions, media, location, contacts, and generic routing fields. The pinned Baileys type surface includes `externalAdReply.ctwaClid` and related source fields, so this remains an implementation gap in the WhatsApp plugin rather than an unsupported dependency.

## What This Changes

The issue requests that the WhatsApp plugin extract Click-to-WhatsApp `ctwaClid` and related ad referral fields from inbound Baileys context and expose them through agent/session metadata.

## Best Possible Solution

Implement this inside the WhatsApp plugin by normalizing a minimal untrusted CTWA payload from inbound `contextInfo.externalAdReply` into the existing structured context path, then deliberately decide whether plugin hook metadata and session persistence should carry the same payload. Keep the change additive, WhatsApp-owned, and covered by focused tests.

## Reproduction Assessment

Yes. A synthetic Baileys message with `extendedTextMessage.contextInfo.externalAdReply.ctwaClid` can be traced through the current extractor, queueing, context builder, hook mapper, and session metadata code; the field is never read or propagated.

## Solution Assessment

Yes. A narrow WhatsApp plugin change using the existing untrusted structured metadata path is the best first fix; broad core session schema changes should be avoided unless maintainers explicitly want durable CTWA storage.
