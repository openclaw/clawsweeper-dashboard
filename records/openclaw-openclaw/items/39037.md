## Summary

Keep open: current main still extracts raw WhatsApp `contextInfo` but does not normalize or expose CTWA ad-click metadata, and the remaining decision is the approved privacy-aware exposure or storage surface. Related WhatsApp metadata items cover forwarding and GIF/ad-reply context, not the CTWA attribution identifier itself.

## What This Changes

Request to extract CTWA ad-click fields from inbound WhatsApp `contextInfo.externalAdReply` and make them available to agents or session metadata.

## Best Possible Solution

Add a WhatsApp-owned typed extractor for `externalAdReply.ctwaClid` and approved ad-referral fields, then expose them through the maintainer-approved untrusted context, hook, or session surface with durable storage only if explicitly approved.

## Reproduction Assessment

Yes, source-level. A synthetic Baileys inbound message with `contextInfo.externalAdReply.ctwaClid` would reach raw `extractContextInfo`, but current main has no carrier into queued messages, prompt metadata, hook metadata, or session metadata.

## Solution Assessment

Unclear as proposed. The direction is valid, but broadly copying referral data into session metadata or preamble is not the best fix until maintainers choose a narrow, typed, privacy-aware exposure surface.
