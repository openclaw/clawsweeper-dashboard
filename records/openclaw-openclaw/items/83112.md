## Summary

Keep open: the fix targets a real message-loss bug, but the PR introduces a stale-snapshot path where a prior turn's declined or failed native tool can become the current attempt's `lastToolError`. That can emit a misleading warning on a later turn.

## What This Changes

The PR records non-success Codex app-server native tool items as attempt-level `lastToolError` and adds projector regression tests.

## Best Possible Solution

Keep the intended no-silent-reply behavior, but make native error recording obey the same current-turn snapshot boundary as native progress before merge.

## Reproduction Assessment

Yes, at source level: the linked issue and PR proof show a declined native `commandExecution` currently leaves `lastToolError` unset, and the existing payload builder only emits the visible warning when that field is present. I did not rerun the live Codex/Telegram path in this read-only review.

## Solution Assessment

No as written. The core approach is narrow, but the snapshot-loop implementation must filter stale prior-turn native items before recording `lastToolError`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Filter stale snapshot items before recording native errors:** `extensions/codex/src/app-server/event-projector.ts:654`

Please apply the same current-turn guard, and add a regression with a prior-turn declined/failed native item.
