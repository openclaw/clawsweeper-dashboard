## Summary

Keep open: current main still has a source-visible WebChat auto-TTS delivery gap. The audio supplement is persisted through transcript/session-update state, but the agent-run final chat event remains text-only and the Control UI does not replay the deferred history reload after a normal renderable final message.

## What This Changes

Fix WebChat auto-TTS so generated audio media reaches the live browser delivery path instead of only durable transcript/history state.

## Best Possible Solution

Bridge the appended WebChat TTS media into the agent-run live final/update path, or make the Control UI reload and merge the deferred transcript supplement for renderable finals, with coverage proving audio appears without duplicate assistant transcript rows.

## Reproduction Assessment

No live current-main WebChat/TTS reproduction was run. Source inspection shows the current path clearly, and a May 18 report confirms the same symptom on v2026.5.12, but an end-to-end current-main proof is still missing.

## Solution Assessment

No, the paired PR at https://github.com/openclaw/openclaw/pull/81470 is not the best current fix because it edits the non-agent final assembly branch. The safer fix targets the agent-run final delivery or the deferred WebChat history reload/merge path.
