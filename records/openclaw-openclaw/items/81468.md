## Summary

Keep open: current main still has source-level evidence of an agent-run WebChat TTS delivery gap, and the same-author paired PR remains open but appears to target the non-agent branch. The merged command-reply fix at https://github.com/openclaw/openclaw/pull/81722 improves `/tts audio` rendering, but it explicitly did not cover agent-sent TTS end to end.

## What This Changes

The issue asks WebChat to include generated assistant TTS audio in the live final broadcast instead of only persisting it in transcript/history media state.

## Best Possible Solution

Fix the agent-run Gateway/WebChat final chat event so it carries the same resolved assistant-media attachment content as the transcript supplement, without duplicating transcript entries or changing non-agent command semantics.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection on current main shows the media supplement can be persisted while the agent-run final chat event is still assembled from buffered text only.

## Solution Assessment

No, the proposed raw-final-payload branch is not the best fix because that branch is non-agent-only while the reported media supplement requires an agent run. The safer solution is to bridge resolved media into the agent-run final broadcast path and keep transcript idempotency intact.
