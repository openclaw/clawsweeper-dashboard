## Summary

Keep open: current main still has a source-visible WebChat/Gateway TTS media delivery gap, and the same-author paired implementation PR is still open rather than merged. The related autoplay issue is separate; this issue is about the generated audio attachment being absent from the live final chat broadcast path.

## What This Changes

Reports that WebChat live final chat events omit generated TTS audio media even though the audio reaches the transcript/history path.

## Best Possible Solution

Fix the agent-run Gateway/WebChat final broadcast path so the same normalized audio content appended as a WebChat-managed media transcript supplement is also carried in the final chat event, without duplicating Pi transcript entries.

## Reproduction Assessment

Yes for source-level reproduction, not for live browser playback: an agent-run TTS media payload can append an assistant-media transcript supplement, while `emitChatFinal` broadcasts only buffered text. I did not run a real WebChat/TTS session.

## Solution Assessment

No, the issue's proposed raw-final-payload edit targets the non-agent final assembly branch. The safer fix is to bridge appended agent-run media into the final chat broadcast path while keeping transcript deduplication intact.
