## Summary

Keep open. Current main and latest v2026.5.12 still route Anthropic OAuth extra-usage-shaped messages into an API-key/top-up billing formatter, and the open closing PR https://github.com/openclaw/openclaw/pull/80895 should remain the fix path until merged.

## What This Changes

The issue requests auth-mode-aware billing error copy so Anthropic Max OAuth usage-cap failures do not tell users to top up an API key.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80895 or an equivalent auth-mode-aware formatter/context propagation fix that preserves hard API-key billing copy for API-key failures and uses neutral recovery wording for OAuth/token-backed Anthropic usage-cap errors.

Yes. Source inspection gives a high-confidence reproduction path: an Anthropic usage-cap string matching `out of extra usage` is classified as billing and then formatted by an auth-mode-blind API-key/top-up formatter on current main.

## Solution Assessment

Yes. Passing the already-known auth profile mode into the central formatter and downstream failover/payload synthesis points is the narrow maintainable fix; it does not require a new config option or product policy decision.
