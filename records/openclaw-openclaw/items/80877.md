## Summary

Keep open. Current main still has an auth-mode blind billing formatter: billing-classified Anthropic errors are shaped into API-key top-up copy even when the Anthropic route is Claude CLI/OAuth, and the related closed reports addressed response-sanitizer false positives or cooldown behavior rather than this specific message path.

## What This Changes

The issue asks OpenClaw to make billing-shaped Anthropic errors auth-mode aware so Claude CLI/Max OAuth users do not receive API-key top-up instructions.

## Best Possible Solution

Carry auth profile mode or credential source into the failover/error-formatting path, keep the existing API-key billing copy for API-key 402/balance failures, and use neutral retry/recovery wording for Claude CLI/OAuth usage-cap or transient errors.

## Reproduction Assessment

Yes for the source-level formatter path: once a Claude CLI/OAuth error is classified as billing, current main still emits API-key top-up copy. No live Max OAuth/Telegram run was performed during this read-only review.

## Solution Assessment

Yes, the best fix is auth-mode-aware error shaping rather than globally weakening billing detection. The implementation should preserve hard billing treatment for API-key credentials and only change the OAuth/Claude CLI branch.
