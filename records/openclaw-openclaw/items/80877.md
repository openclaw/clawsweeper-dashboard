## Summary

Keep open. Current `main` and the latest release still route billing-classified Anthropic OAuth/Claude CLI usage errors through API-key/top-up copy, and the issue is already paired with the open fix PR at https://github.com/openclaw/openclaw/pull/80895, so it should remain open until that PR lands or maintainers choose another fix path.

## What This Changes

Make billing-classified Anthropic Max OAuth/Claude CLI errors avoid API-key top-up copy and use neutral recovery wording while preserving API-key billing messages.

## Best Possible Solution

Land the narrow auth-mode-aware formatter/context propagation fix from https://github.com/openclaw/openclaw/pull/80895 or a maintainer replacement that preserves API-key billing copy and uses neutral recovery copy for OAuth/token-backed credentials.

## Reproduction Assessment

Yes at source level: once an Anthropic OAuth/Claude CLI usage-cap-shaped error is classified as billing, current `main` formats it with API-key/top-up copy because the formatter has no auth-mode input. I did not run a live Max OAuth or Telegram reproduction.

## Solution Assessment

Yes: threading the already-known auth profile mode into the central billing formatter and user-visible failover/payload surfaces is the narrowest maintainable fix. Weakening billing classification globally would risk regressing real API-key billing failures.
