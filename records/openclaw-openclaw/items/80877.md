## Summary

Keep open. Current main still formats every billing-classified assistant error with API-key/top-up wording, while the Anthropic plugin and docs support a Claude CLI/OAuth route with no API key; the related surfaced-error fix at https://github.com/openclaw/openclaw/pull/70848 made these failures visible but did not make the copy auth-aware.

## What This Changes

The issue asks for Anthropic Max OAuth transient or usage-cap errors to avoid API-key credit/top-up wording in user-facing channels.

## Best Possible Solution

Carry Anthropic auth mode into the failover/error formatting decision and emit neutral recovery copy for Claude CLI/OAuth usage-cap or token-transient cases while preserving API-key billing copy for real credit failures.

## Reproduction Assessment

Yes from source inspection, with medium confidence: billing-shaped text reaches `formatAssistantErrorText` and is always formatted as an API-key top-up problem, while Anthropic Claude CLI/OAuth is a supported no-API-key route. I did not run a live OAuth expiry or usage-cap repro in this read-only review.

## Solution Assessment

No, the current generic formatter is not the best solution because it lacks auth context. A narrower maintainable fix should classify or format Anthropic OAuth/Claude CLI failures before the generic API-key billing copy is chosen.

## Repair Work Prompt

Fix the misleading Anthropic Max OAuth billing copy reported in https://github.com/openclaw/openclaw/issues/80877. Current main routes billing-shaped assistant errors through `formatBillingErrorMessage`, which always mentions an API key and top-up; Anthropic supports Claude CLI/OAuth without an API key. Implement a narrow shared error-shaping fix that detects Anthropic Claude CLI/OAuth/no-API-key auth at the failover/error formatting boundary and uses neutral temporary/recovery-oriented copy for OAuth usage-cap/token-transient cases, while preserving the existing top-up/API-key copy for true API-key credit-balance failures. Prefer carrying known auth mode/profile type into the active error context or classifier over Telegram-specific handling. Do not change Telegram delivery behavior and do not suppress the surfaced-provider-error path from https://github.com/openclaw/openclaw/pull/70848. Add regression tests for Anthropic API-key insufficient credits still using billing copy, Anthropic OAuth/Claude CLI extra-usage or usage-limit text not mentioning API key/top-up, and surfaced `surface_error`/payload paths using the new copy. Add a changelog entry if this becomes a user-facing fix PR.
