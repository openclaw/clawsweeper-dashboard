## Summary

Current main applies routed agent tool policy; the observed tool schemas come from `tools.allow: []`, which current policy treats as non-restrictive rather than LLM-only, so this is a configuration/support mismatch rather than a source bug.

## What This Changes

The report asks voice-call embedded responses to treat a routed agent with `tools.allow: []` as LLM-only and avoid sending tool schemas to Ollama.

## Best Possible Solution

Keep the existing tool-policy contract and configure LLM-only voice agents with a deny-all policy such as `agents.list[].tools.deny: ["*"]`, or use a non-empty allowlist for the exact tools intended.

## Reproduction Assessment

No, not as a source bug: source inspection explains why `tools.allow: []` still leaves tools available under current policy semantics. The observed behavior is source-reproducible as expected configuration behavior, not a failing current-main contract.

## Solution Assessment

Yes: the narrow maintainable solution is to keep the current policy semantics and use `deny: ["*"]` or a restrictive non-empty allowlist for no-tool voice runs, rather than reinterpreting empty persisted allowlists only for Voice Call.
