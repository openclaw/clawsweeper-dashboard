## Summary

Keep open: current main still omits runtime tool-policy forwarding in the classic voice-call response path, and the issue is already paired with an open closing PR that should be reviewed or replaced before the issue closes.

## What This Changes

The issue asks for classic voice-call embedded responses to respect a routed agent's no-tool or tool-allow configuration so local Ollama response models are not sent unrelated OpenClaw tool schemas.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79508 or a narrow replacement that forwards the routed voice agent's effective tool policy into classic voice responses without changing global tool-policy semantics unexpectedly, with regression coverage and any needed docs/changelog updates.

## Reproduction Assessment

Yes. The source path is clear on current main and matches the reporter's redacted trajectory logs: classic voice-call does not pass toolsAllow, while the embedded runner only suppresses tools when an explicit runtime toolsAllow: [] reaches it.

## Solution Assessment

Unclear as an issue-only cleanup. Passing a routed allowlist into the embedded runner is the narrow implementation direction, but maintainers should decide how config-level tools.allow: [] should behave for classic voice before merging a fix.
