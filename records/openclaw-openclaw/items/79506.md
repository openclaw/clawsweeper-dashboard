## Summary

Keep open: the timeout report is source-plausible, but the requested `tools.allow: []` behavior conflicts with current tool-policy semantics, so this needs maintainer/product judgment instead of conservative cleanup.

## What This Changes

The issue asks voice-call classic embedded responses to avoid sending tool schemas when a routed voice agent is intended to run without tools, especially for low-latency Ollama response models.

## Best Possible Solution

Decide and document the supported no-tools path for classic voice responses, preferably preserving existing tool-policy semantics while adding a clear voice/Ollama configuration route if needed.

## Reproduction Assessment

Yes, source inspection shows the classic voice-call path does not pass runtime `toolsAllow` or `disableTools`, while config-level `tools.allow: []` currently does not disable tools. I did not perform a live Twilio/Ollama call in this read-only review.

## Solution Assessment

No, not as stated: treating `tools.allow: []` as no tools would contradict current documented and tested allowlist semantics. A safer path is maintainer-approved documentation or an explicit voice/model no-tools configuration path.
