## Summary

Keep open: current main still has a concrete Anthropic/Claude CLI auth-inspection mismatch, at least for `models auth list --provider anthropic`, and the status path overlaps an open but not-yet-merged status/probe repair.

## What This Changes

The issue asks OpenClaw to make Anthropic-facing auth list/status commands recognize the usable `anthropic:claude-cli` profile used by canonical `anthropic/*` model refs.

## Best Possible Solution

Keep the issue open and fix the inspection layer so Anthropic provider filters/status use the same provider-auth alias mapping and scoped CLI profile discovery as the runtime path.

## Reproduction Assessment

Yes, at source level for the auth-list half: a Claude CLI login stores an `anthropic:claude-cli` profile whose credential provider is `claude-cli`, while `models auth list --provider anthropic` filters by exact normalized provider. The status half is consistent with the same boundary and has an adjacent unmerged status/probe repair, but I did not live-run Claude CLI.

## Solution Assessment

Yes, the narrow maintainable fix is to make auth inspection consume the shared provider-auth alias/profile resolver rather than changing the stored profile shape. If the status/probe PR has not landed, the status fix should be coordinated with that branch.
