## Summary

Keep open: current main still hides the stored Claude CLI auth profile from `models auth list --provider anthropic` because the command filters by raw summarized provider equality. The linked fix PR at https://github.com/openclaw/openclaw/pull/79602 is closed unmerged, and I did not find another canonical open successor.

## What This Changes

The issue asks OpenClaw auth inspection to recognize Claude CLI auth profiles for canonical Anthropic model refs after `models auth login --provider anthropic --method cli`.

## Best Possible Solution

Make model auth inspection resolve provider aliases consistently so `anthropic:claude-cli` remains stored as a `claude-cli` runtime credential while Anthropic-facing list and status output treat it as usable Anthropic auth.

## Reproduction Assessment

Yes, at source level. Seed an auth store with `anthropic:claude-cli` whose credential provider is `claude-cli`, then run the current auth-list filter for `anthropic`; the direct equality check drops the profile.

## Solution Assessment

No for current main. The maintainable path is to make auth-list filtering use the existing canonical provider/auth alias contract or a shared helper over the runtime alias table, with regression coverage for the reported list/status flow.
