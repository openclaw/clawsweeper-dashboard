## Summary

Keep open: current main still silently filters the agent-facing `sessions_list` result by `tools.sessions.visibility` and returns only `count` plus `sessions`, so the requested scope disclosure remains missing. This is a narrow UX enhancement, not a backend correctness bug.

## What This Changes

The issue asks for session listing/status tools to disclose when results are scoped by `tools.sessions.visibility` so users do not mistake filtered results for the complete session store.

## Best Possible Solution

Add a non-breaking visibility/scope note or metadata field to agent-facing `sessions_list` results when effective visibility is not `all`, preserving current filtering and defaults.

## Reproduction Assessment

Yes, source-reproducible: `sessions_list` filters Gateway rows through the effective session visibility checker and then returns only `count` and `sessions`. I did not run a live store reproduction because this read-only review has direct source and test evidence for the missing scope indication.

## Solution Assessment

Yes, the narrow maintainable fix is an additive agent-tool visibility note/metadata field, not a default visibility change or new config option. If maintainers want Gateway clients to receive the same signal, that protocol surface should remain additive.
