## Summary

Keep open. Current main still applies session-tool visibility filtering silently: the default effective visibility is `tree`, `sessions_list` filters rows through the visibility guard, and the returned tool payload contains only `count` and `sessions` without a scope note or structured visibility metadata. The discussion comment’s link to #50621 and KeepMyClaw supports that session-state observability is a recurring operator pain point, but it does not supersede this narrower UX fix.

## What This Changes

The issue requests that session-related tools disclose when results are scoped by restricted `tools.sessions.visibility` settings instead of presenting filtered results as complete.

## Best Possible Solution

Add an additive runtime disclosure to session tool outputs when the effective session-tool visibility is not `all`, preserving the existing access policy. Prefer structured metadata such as configured visibility, effective visibility after sandbox clamp, requester scope, and a human-readable note; include filtered counts only when they can be computed without extra leakage or misleading semantics.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: with default config, `resolveSessionToolsVisibility` returns `tree`, `sessions_list` filters through `createSessionVisibilityGuard`, then returns only `count` and `sessions`. No live tool invocation or tests were run because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The requested additive disclosure is the narrowest maintainable fix because it improves observability without broadening access, changing the default visibility policy, or exposing hidden session identifiers.
