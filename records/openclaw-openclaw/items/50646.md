## Summary

Keep open. Current main still applies `sessions_list` visibility filtering silently: default/effective visibility resolves to `tree`, rows are skipped through the visibility guard, and the tool response still returns only `count` plus `sessions` with no scope note or structured visibility metadata. Ryce’s comment usefully links this to the broader session-state observability pain in #50621 and KeepMyClaw, but that is adjacent context rather than a merged fix or superseding tracker.

## What This Changes

The issue asks session listing tools to disclose when returned rows are scoped by `tools.sessions.visibility` instead of presenting a bare count as if it were complete.

## Best Possible Solution

Add an additive runtime disclosure to session tool outputs when effective visibility is not `all`, preserving the existing access policy and documenting the new metadata.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: default config resolves to `tree`, `sessions_list` filters rows through `createSessionVisibilityGuard`, then returns only `count` and `sessions`.

## Solution Assessment

Yes. The requested additive disclosure is the narrowest maintainable direction because it improves observability without broadening session access, changing default visibility, or exposing hidden session identifiers.

## Repair Work Prompt

Implement the session visibility disclosure requested by https://github.com/openclaw/openclaw/issues/50646. Add an additive field or note to `sessions_list` results when the effective session-tool visibility is not `all`; include configured visibility, effective visibility after sandbox clamp, and a human-readable note that results are scoped. Do not broaden access, change the default `tree` policy, expose hidden session identifiers, or promise a filtered-out count unless it is cheap and non-leaky under existing filters. Add focused regression coverage for default `tree` output and `all` output, update user-facing docs for the tool/config behavior, and add a changelog entry because this changes user-visible tool output.
