## Summary

Keep open: current main and the latest release still return `sessions_list` as a complete-looking `count`/`sessions` payload after visibility filtering, so the requested result-level scope disclosure is still missing. Prior closed reports document and accept scoped behavior, but they do not add the payload metadata requested here.

## What This Changes

Request an additive `sessions_list` disclosure when effective session-tool visibility scopes or filters the listed sessions.

## Best Possible Solution

Add additive scope metadata or a warning to `sessions_list` results when effective visibility is not `all`, while preserving current filtering and privacy boundaries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure restricted session visibility, have more gateway rows than the caller can see, and `sessions_list` returns only filtered `count` plus `sessions` without scope metadata; I did not run a live invocation in this read-only review.

## Solution Assessment

Yes. A narrow additive result-level disclosure is the best fit; changing default visibility or exposing hidden-session counts would be broader and riskier.
