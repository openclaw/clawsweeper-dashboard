## Label Justifications

- `P1`: The issue reports an active provider-routing regression where Anthropic primary turns can silently switch to a different fallback provider mid-turn.
- `impact:auth-provider`: The affected behavior is provider transport failure handling, model choice, and fallback routing across Anthropic primary and OpenAI fallback chains.

## Summary

Keep open: current main and the latest release still route Anthropic Messages through guarded fetch and rethrow the first pre-response transport failure into model fallback without a same-provider retry. The open related PR is still unmerged and previously reviewed as not proof-positive, so this issue remains necessary.

## What This Changes

The issue requests that transient Anthropic `UND_ERR_SOCKET` or similar pre-response fetch failures retry the primary provider once instead of silently falling through to an OpenAI/Codex fallback mid-turn.

## Best Possible Solution

Land a reviewed guarded-transport fix that retries the same provider once for clearly transient pre-response socket failures before model fallback, with focused regression coverage and affected-environment proof.

## Reproduction Assessment

Yes for the source-level path: current main routes Anthropic Messages through guarded fetch, rethrows the first pre-response transport failure, and failover classifies socket/network failures as timeout. I did not live-reproduce the multi-hour stale-socket trigger in this read-only review.

## Solution Assessment

Yes in direction: the narrow maintainable fix is a bounded same-provider retry in guarded model transport before model fallback runs. The open related PR should be reviewed or reworked rather than closing this issue before merge.
