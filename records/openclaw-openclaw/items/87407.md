## Label Justifications

- `P1`: The issue describes an active provider-routing regression where real Anthropic turns can silently switch to a different OpenAI/Codex fallback model mid-turn.
- `impact:auth-provider`: The affected behavior is provider routing and model choice across an Anthropic primary and OpenAI/Codex fallback chain.

## Summary

Keep open: current main and the latest release still send Anthropic Messages through the guarded fetch path without a same-key transient retry, so a first `fetch failed`/socket error can still enter the model fallback ladder; the related fix PR is open and not merged.

## What This Changes

The issue requests a narrow fix for Anthropic Undici socket/fetch failures so the primary Anthropic model is retried before cross-provider fallback can switch the turn to OpenAI/Codex.

## Best Possible Solution

Add a bounded same-key retry for clearly transient pre-response socket/fetch failures in the guarded model transport before model fallback runs, preserving the existing security dispatcher and adding focused fallback regression coverage.

## Reproduction Assessment

Yes for the source-level fallback behavior: current code and tests show bare transport failures can become timeout-class fallback attempts. I did not live-reproduce the multi-hour Anthropic stale-socket trigger in this read-only pass.

## Solution Assessment

Yes, the narrowest maintainable fix is a capped retry before fallback in the guarded model transport path. A dispatcher-only keep-alive tweak is only a mitigation and risks missing cold-start transient fetch failures or bypassing existing transport policy.
