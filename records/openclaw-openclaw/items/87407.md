## Label Justifications

- `P1`: The issue reports a provider-routing regression where real Anthropic turns can silently switch to a different OpenAI/Codex fallback model mid-turn.
- `impact:auth-provider`: The affected behavior is provider routing and model choice across Anthropic primary and OpenAI/Codex fallback chains.

## Summary

Keep open: current main and the latest release still send Anthropic Messages through guarded model fetch without a same-key transient retry, so a first `fetch failed` or `UND_ERR_SOCKET` can still enter the model fallback ladder. The related fix PR is open and unmerged, so it remains the active candidate rather than proof this issue is implemented.

## What This Changes

The issue reports Anthropic `UND_ERR_SOCKET`/`fetch failed` transport errors advancing the model fallback chain from Anthropic to OpenAI/Codex mid-turn instead of retrying the primary provider once.

## Best Possible Solution

Add a capped same-key retry for transient pre-response socket/fetch failures in the guarded model transport before model fallback runs, with focused fallback coverage and affected-environment proof.

## Reproduction Assessment

Yes for the source-level fallback behavior: current main routes Anthropic Messages through guarded fetch, guarded fetch rethrows transport failures, and existing fallback tests classify bare transport failures as timeout fallback attempts. I did not live-reproduce the multi-hour Anthropic stale-socket trigger.

## Solution Assessment

Yes: the narrow maintainable fix is a bounded retry in the guarded model transport before model fallback; the open dispatcher-only PR is a candidate but appears to be the wrong layer unless reworked.
