## Label Justifications

- `P1`: The issue reports a real regression where a configured Anthropic primary can silently fall through to a different provider during an agent turn.
- `impact:auth-provider`: The affected behavior is provider/model routing and fallback selection for Anthropic primary models with OpenAI fallback candidates.

## Summary

Keep open: current main and the latest release still let Anthropic transport fetch failures escape the guarded fetch path and enter model fallback, and the linked open fix has not merged.

## What This Changes

The issue requests a retry or transport fix so transient Anthropic `UND_ERR_SOCKET`/fetch failures do not silently switch a turn from Anthropic to an OpenAI fallback model.

## Best Possible Solution

Add a bounded same-provider retry for transient pre-response Anthropic transport failures in the guarded transport path before model fallback, with focused regression coverage and live/log proof from the affected setup.

## Reproduction Assessment

No high-confidence live reproduction was run here. Source inspection plus reporter logs show a plausible current path: Anthropic create fetch failure is rethrown by guarded fetch, classified as timeout/failover, and the fallback chain can move to another provider.

## Solution Assessment

No, the existing linked PR is only a plausible mitigation. The best fix should live at the guarded Anthropic transport/retry boundary so it protects the embedded Messages create path without bypassing OpenClaw’s SSRF/proxy/TLS dispatcher policy.
