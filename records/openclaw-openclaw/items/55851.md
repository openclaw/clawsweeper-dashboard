## Summary

Keep open: current main still lacks the requested provider/model/profile/trigger context for transient provider errors, but this PR is not merge-ready because it regresses transient api_error failover classification, exposes raw provider request IDs in chat-visible copy, and lacks real behavior proof.

## What This Changes

The PR enriches pi-embedded overload, rate-limit, and transport error text with provider/model/profile/trigger/session/request context and threads profile/trigger through runner payload formatting.

## Best Possible Solution

Land a rebased shared-formatter implementation that appends sanitized active provider/model/profile/trigger context, preserves broad transient failover classification, and keeps provider request IDs out of chat-visible copy.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows req= extraction and narrowed api_error detection, while current main shows request_id suppression and broad transient matching. I did not run a live provider/channel scenario, and the PR does not provide one.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best solution until it is rebased onto the shared formatter, preserves existing failover behavior, removes raw request IDs from chat copy, and includes real behavior proof.
