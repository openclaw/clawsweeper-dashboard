## Summary

Keep open: the feature direction is still valid, but this PR is not mergeable as-is because it exposes raw provider request IDs in chat-visible copy, narrows transient api_error failover behavior, is currently conflicting, and lacks real behavior proof.

## What This Changes

The PR threads provider/model/profile/trigger/session context into pi-embedded overload, rate-limit, and transport error messages and loosens related exact-string tests.

## Best Possible Solution

Land or replace this PR with a rebased shared-formatter change that adds sanitized active context without exposing provider request IDs or regressing failover classification.

## Reproduction Assessment

Yes for the PR blockers: PR-head source shows raw req= extraction and narrowed api_error matching, while current-main source and tests show request-id suppression and broader transient matching. I did not run a live provider or Telegram scenario, and the PR does not provide one.

## Solution Assessment

No, not as-is. The maintainable path is a rebased shared-formatter implementation that preserves broad transient failover, omits raw request IDs from channel-visible text, and includes real Telegram or equivalent runtime proof.
