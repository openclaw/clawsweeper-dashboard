## Summary

Keep open. The feature is still missing on current main, but this PR is not merge-ready because it regresses the existing transient api_error failover contract for known provider 5xx variants.

## What This Changes

The PR enriches pi-embedded overloaded, rate-limit, and transport error messages with provider/model/profile/trigger/session/request context and threads profile/trigger through runner payload construction and tests.

## Best Possible Solution

Land a rebased version of the context-enrichment feature on the current formatter split while preserving the broad transient api_error failover classifier, keeping diagnostic log assertions meaningful, and adding release-note coverage for the user-facing behavior.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the existing classifyFailoverReason coverage for JSON api_error payloads such as unknown error 520, Service temporarily unavailable, and unexpected-error; the PR’s narrowed predicate would stop classifying those as timeout.

## Solution Assessment

No. The direction is useful, but the current PR is not the best implementation because it mixes message enrichment with a failover classifier regression; the safer fix is to add the context suffix through the current helpers without removing API_ERROR_TRANSIENT_SIGNALS_RE.
