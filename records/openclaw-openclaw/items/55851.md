Keep open: current main still lacks the requested transient-error context, but this PR should not merge as-is because it regresses existing api_error failover handling and would expose raw provider request IDs in chat-visible error copy.

## What This Changes

The PR enriches pi-embedded overloaded, rate-limit, and transport error messages with provider/model/profile/trigger/session/request context and threads profile/trigger through runner payload construction and tests.

## Best Possible Solution

Land a rebased context-enrichment implementation that preserves broad transient failover classification, omits or hashes provider request IDs in chat-visible copy, restores focused regression coverage, and adds a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the requested profile/trigger/context plumbing, and the PR diff plus current tests give a high-confidence path for the api_error failover regression without executing the suite.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best solution until it preserves existing transient api_error classification and follows the current policy of keeping raw provider request IDs out of user-facing UI copy.
