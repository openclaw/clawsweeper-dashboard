## Summary

Keep open: current main still lacks provider/model/profile/trigger context for transient chat-visible errors, but this PR branch is not mergeable as-is because it exposes raw provider request IDs, narrows retryable api_error classification, and still lacks real behavior proof.

## What This Changes

The PR enriches pi-embedded overload, rate-limit, and transport error text with provider/model/profile/trigger/session/request context and threads profile/trigger through runner payload formatting.

## Best Possible Solution

Land a rebased shared-formatter implementation that appends sanitized active provider/model/profile/trigger context, preserves broad transient failover classification, omits raw request IDs from channel-visible copy, and includes real Telegram/runtime proof.

## Reproduction Assessment

Yes for the PR review blockers: PR-head source shows raw req= extraction and narrowed api_error matching, while current-main source/tests show request-id suppression and broader transient matching. I did not run a live provider or Telegram scenario, and the PR does not provide one.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best way to solve it until it is rebased onto the shared formatter, preserves failover behavior, removes raw request IDs from chat copy, and includes real behavior proof.
