The PR enriches pi-embedded overload, rate-limit, and transport error text with provider/model/profile/trigger/session/request context and threads profile/trigger through runner payload formatting and tests.

## Best Possible Solution

Land a rebased shared-formatter implementation that appends sanitized active provider/model/profile/trigger context, preserves broad transient failover classification, and keeps raw provider request IDs out of chat-visible copy.

## Reproduction Assessment

Yes for the review blockers: source inspection of PR head shows req= extraction and narrowed api_error matching, while current main shows request-id suppression and broad transient matching. I did not run a live provider/channel scenario, and the PR does not provide one.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best way to solve it until it is rebased onto the shared formatter, preserves failover behavior, removes raw request IDs from chat copy, and includes real behavior proof.
