## Summary

Keep open: the feature is still not on main, but this PR is not mergeable as-is because it regresses transient api_error failover, exposes raw provider request IDs in chat-visible copy, and lacks required real behavior proof.

## What This Changes

The PR enriches pi-embedded rate-limit, overloaded, and transport error messages with provider/model/profile/trigger/session/request context and threads profile/trigger through runner payload calls.

## Best Possible Solution

Rebase or replace this with a shared formatter change that shows sanitized active provider/model/task context, omits raw request IDs from channel text, preserves broad failover classification, and includes real Telegram or equivalent proof.

## Reproduction Assessment

Yes for the PR blockers: PR-head source shows req= extraction and narrowed api_error detection, while current-main source/tests keep raw request IDs out of payload text and classify broader api_error variants as timeout. I did not run a live provider or Telegram scenario in this read-only review.

## Solution Assessment

No, not as-is. The direction is valid, but the maintainable fix should keep request IDs in redacted diagnostics only and build on the current shared sanitizer without weakening failover behavior.

- **[P1] Preserve transient api_error failover variants:** `src/agents/pi-embedded-helpers/errors.ts:845`
  - body: This narrows JSON api_error retry detection to only messages containing internal server error.
