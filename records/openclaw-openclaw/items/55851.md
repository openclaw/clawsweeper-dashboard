## Summary

Keep this PR open: the feature is still useful and current main does not provide the requested provider/model/profile/trigger context, but the proposed patch needs changes before merge because it weakens transient failover classification and exposes raw provider request IDs in chat-visible copy.

## What This Changes

The PR enriches pi-embedded transient and transport error messages with provider/model/profile/trigger/session/request context and threads those fields through runner payload formatting while updating related tests.

## Best Possible Solution

Land a rebased implementation that adds safe transient-error context on the current sanitizer path, preserves existing failover classification, omits raw request IDs from chat text, restores focused regression coverage, and adds a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection of current main shows the transient message formatter and runner payload path do not have profile/trigger context, and the PR diff plus existing tests show the reviewed failover and request-id issues without needing to execute the suite.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best solution until it preserves broad `api_error` transient classification and follows the existing policy of keeping raw provider request IDs out of user-facing copy.

  - body: This narrows JSON `api_error` retry detection to internal-server-error text and drops existing transient variants such as unknown error 520, Service temporarily unavailable, and Anthropic unexpected-error.
