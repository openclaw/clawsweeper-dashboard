## What This Changes

The PR keeps Telegram's implicit `default` account listed when top-level `botToken`, SecretRef-like `botToken`, or `tokenFile` credentials coexist with named accounts, and adds a mixed-config regression test.

## Best Possible Solution

Land the focused Telegram account-listing fix with its regression test after maintainer review; decide separately whether env-only default credentials should be folded into this PR.

## Reproduction Assessment

Yes. Current main's source shows `default` is only injected when no configured or bound ids exist, and the linked before log demonstrates the top-level-token plus named-account case returning only `fusion`.

## Solution Assessment

Yes. Adding `default` at the Telegram account-listing boundary when top-level credential sources exist is narrow, keeps named-only configs unchanged, and matches the runtime token contract for configured top-level credentials.
