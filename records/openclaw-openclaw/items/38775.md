## Summary

Keep open: current main still lacks usage-aware auth-profile routing. Profile selection is based on configured order, credential/type preference, last-used rotation, optional provider preference, and reactive cooldowns; Anthropic usage windows are fetched for usage/status reporting but are not used to choose a request profile. Related #41135 is the closest umbrella for the generic pool-routing design, but this issue still carries useful Anthropic-specific usage-signal details for maintainers to preserve during consolidation.

## What This Changes

Request opt-in usage-aware routing for multi-account auth profiles, using Anthropic usage windows to choose the profile with the most headroom before reactive rate-limit cooldowns.

## Best Possible Solution

Design a generic opt-in provider-profile routing contract that consumes provider-owned per-profile headroom signals, preserves current defaults, and lets Anthropic contribute its 5h/week windows, thresholds, and reset-aware scoring as one provider implementation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current gap is high-confidence from source inspection of auth-profile ordering, embedded profile candidate construction, cooldown handling, provider usage fetching, config types, docs, and tests.

## Solution Assessment

Unclear. The Anthropic barrier-cost formula is useful design input, but the narrower maintainable direction is a generic opt-in provider-profile routing policy with provider-owned usage/headroom signals rather than hard-coding an Anthropic-specific core policy.
