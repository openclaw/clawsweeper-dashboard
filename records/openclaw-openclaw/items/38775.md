## Summary

Keep open: current main has provider usage snapshots and a provider-owned auth-profile preference hook, but request-time auth-profile selection still does not score Anthropic accounts by utilization/headroom or proactively circuit-break near quota windows. The related open items overlap the broader design space, but they do not fully supersede this Anthropic multi-account usage-routing request.

## What This Changes

Request adds opt-in usage-aware request routing across multiple auth profiles, especially Anthropic OAuth profiles, using utilization windows to pick the account with the most headroom instead of reactive cooldown-only failover.

## Best Possible Solution

Add an opt-in generic auth-profile routing contract that consumes provider-owned usage snapshots, preserves current session-sticky/default behavior, supports proactive thresholds and fallback for profiles without usage data, exposes routing decisions/status, and lets the Anthropic plugin provide its 5h/week utilization windows as one policy input.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current gap is high-confidence by tracing current main through auth-profile ordering, embedded profileCandidates, auth-controller cooldown checks, and usage-reporting surfaces.

## Solution Assessment

Unclear that the issue’s Anthropic-specific barrier formula should be the final implementation. The maintainable path is a generic opt-in provider-profile routing policy with Anthropic usage data as an input, rather than hard-coding one provider’s algorithm into core.
