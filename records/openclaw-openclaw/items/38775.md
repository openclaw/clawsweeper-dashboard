## Summary

Keep open: current main still selects auth profiles by configured order, credential type, last-used rotation, and reactive cooldown state, while Anthropic usage snapshots are only wired into reporting/status surfaces.

## What This Changes

Request to route multi-account Anthropic auth profiles by usage/headroom instead of only static order, round-robin, and reactive cooldowns.

## Best Possible Solution

Add an opt-in generic provider-profile routing contract that can consume provider-owned per-profile usage/headroom signals, preserve current defaults, support proactive thresholds and fallbacks, and let Anthropic supply its 5h/week windows as policy input.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current gap is high-confidence from source inspection of auth-profile ordering, embedded profileCandidates, cooldown handling, provider usage fetching, and config schema.

## Solution Assessment

Unclear that the issue’s Anthropic-specific barrier-cost formula is the best final implementation. A generic opt-in provider-profile routing policy with Anthropic usage snapshots as one input is the narrower maintainable direction.
