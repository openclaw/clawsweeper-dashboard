## Summary

Keep open: current main still lacks request-time, usage-aware auth-profile routing; it only has ordered/profile-type/last-used selection, session stickiness, and reactive cooldown/fallback behavior, while usage snapshots are wired to status/reporting surfaces.

## What This Changes

Requests opt-in routing across multiple auth profiles based on per-account usage/headroom, especially Anthropic OAuth usage windows, instead of static order and reactive cooldowns.

## Best Possible Solution

Add a generic opt-in provider-profile routing policy contract that consumes provider-owned per-profile headroom/usage signals, preserves current defaults and session stickiness, and lets Anthropic supply its 5h/week windows as one policy input.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection give high confidence that current main lacks proactive usage-aware routing and only has ordered selection, session stickiness, and reactive cooldowns.

## Solution Assessment

Unclear that the proposed Anthropic-specific barrier-cost formula is the best final shape. A generic opt-in provider-profile routing policy with provider-owned usage snapshots is the narrower maintainable direction.
