## What This Changes

The PR changes durable inbound final-reply fallback from context `ReplyToIdFull`/`ReplyToId` to current-message `MessageSidFull`/`MessageSid` and updates focused resolver tests.

## Best Possible Solution

Land the resolver change after redacted live Telegram proof confirms the visible final reply targets the current inbound message and maintainers accept this as the shared durable fallback invariant; keep the broader contract tracked in https://github.com/openclaw/openclaw/issues/87561.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.6.6` fall back to `ReplyToId*`, while Telegram context separately carries the current inbound `MessageSid*`. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the shared durable resolver is the best source layer for this fallback because Telegram final delivery routes through it and a Telegram-only workaround would split the invariant. The PR still needs real transport proof and maintainer acceptance of the shared fallback behavior before merge.
