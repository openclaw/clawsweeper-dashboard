Keep open. The code change appears to address a real source-visible Telegram reply-target bug, but it is not merge-ready because the supplied terminal proof only shows the patched bundle parses and does not demonstrate the changed Telegram delivery behavior.

## What This Changes

This PR changes durable inbound final reply fallback from context `ReplyToId*` ids to current-message `MessageSid*` ids and updates focused resolver tests.

## Best Possible Solution

Land the resolver change only after redacted live Telegram proof demonstrates current-message anchoring and maintainers accept `MessageSid*` as the shared implicit fallback invariant.

## Reproduction Assessment

Yes, as a source-level reproduction: current main and `v2026.6.8` fall back to `ReplyToIdFull`/`ReplyToId`, while Telegram keeps the current inbound message id separately as `MessageSid*`. I did not reproduce a live Telegram misreply in this read-only review.

## Solution Assessment

Yes for the code shape, but not merge-ready: the shared resolver is the right layer because Telegram durable final delivery calls it before outbound send. A Telegram-only workaround would split durable fallback semantics and leave the shared invariant unclear.
