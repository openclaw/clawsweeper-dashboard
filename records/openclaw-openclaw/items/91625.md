## Summary

The implementation is a focused and well-proven way to expose the existing nullable cron update through the CLI, but the new help and docs overstate the result: clearing the per-job model does not guarantee agent/default selection because a stored cron-session model override still has precedence.

## What This Changes

The PR adds `cron edit --clear-model`, emits `payload.model = null`, rejects conflicting model flags, adds focused CLI regression tests, and documents the new option.

## Best Possible Solution

Keep the additive null-clear CLI behavior and revise all three new public descriptions to say it removes the per-job override and restores normal cron model-selection precedence.

## Reproduction Assessment

Yes. Current `main` supports `model: null` deletion but `cron edit` cannot emit it, and the PR supplies live after-fix gateway output showing that the branch removes the stored payload key.

## Solution Assessment

Mostly yes. The additive flag and nullable patch are the narrowest maintainable fix, but the help and docs must describe the established model-precedence behavior accurately.
