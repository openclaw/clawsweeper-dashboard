## Label Justifications

- `P2`: This is a real documented-hook coverage bug on a default chat surface, but it has bounded impact and needs careful Gateway hook semantics rather than emergency remediation.
- `impact:security`: Pre-send guardrail, redaction, or policy plugins relying on `message_sending` can silently miss WebChat final delivery.

## Summary

Keep open. Current main still sends WebChat final assistant payloads through Gateway chat broadcasts without invoking `message_sending`, while the documented hook contract and shared outbound delivery path already support rewrite/cancel behavior; the related open reports are broader umbrella or sibling hook-coverage trackers, not replacements for this narrow WebChat gap.

## What This Changes

Run the existing `message_sending` plugin hook for WebChat/WebSocket final message delivery.

## Best Possible Solution

Wire `message_sending` into Gateway WebChat final assistant delivery with session/run context, rewrite/cancel handling, and focused regression coverage while keeping the universal outbound policy design in https://github.com/openclaw/openclaw/issues/50126 and https://github.com/openclaw/openclaw/issues/56349.

## Reproduction Assessment

Yes, at source level. Follow a WebChat `chat.send` assistant run to `emitChatFinal`; current main reaches the direct final `chat` broadcast without any `message_sending` hook call.

## Solution Assessment

Yes in direction: reusing the existing `message_sending` hook is narrower than adding a WebChat-specific hook. The remaining maintainer decision is how final rewrite/cancel semantics reconcile with already-streamed deltas.
