## Summary

Keep open. Current main still broadcasts WebChat final assistant payloads without invoking `message_sending`, while the shared outbound delivery path and docs already define rewrite/cancel behavior for that hook; the related open reports are broader or sibling hook-coverage trackers, not replacements for this narrow WebChat gap.

## What This Changes

Requests wiring the existing `message_sending` plugin hook into WebChat/WebSocket final assistant delivery so plugins can rewrite or cancel webchat output.

## Best Possible Solution

Wire `message_sending` into WebChat final assistant delivery with session/run context, rewrite and cancel handling, failure isolation, and focused regression coverage, while leaving universal outbound policy unification to the broader hook-coverage issues.

## Reproduction Assessment

Yes, at source level. A WebChat assistant run that reaches `emitChatFinal` on current main broadcasts the final `chat` payload directly, while the hook-aware outbound path is separate and not called there.

## Solution Assessment

Yes for this narrow issue. Reusing the existing `message_sending` hook in the WebChat final path is the smallest maintainable fix; broader unbypassable streaming policy should remain separate.
