## Summary

Keep open. Current main still sends WebChat final assistant output directly from the Gateway chat event path without invoking `message_sending`; the `sessionKey` context part already exists, but the hook wiring and rewrite/cancel behavior are still missing. The related open items are broader or sibling hook-coverage reports, not a replacement for this narrow WebChat parity gap.

## What This Changes

The issue requests firing the existing `message_sending` plugin hook for WebChat/WebSocket final assistant delivery.

## Best Possible Solution

Wire the existing `message_sending` hook into WebChat final assistant delivery with `channelId: "webchat"`, session/run context, failure isolation, rewrite/cancel handling, and focused regression coverage, while leaving cross-channel lifecycle unification to the umbrella issues.

## Reproduction Assessment

Yes, at source level. A WebChat run that reaches `emitChatFinal` with `jobState === "done"` broadcasts the final `chat` payload without calling the hook runner, while the shared outbound path does call `message_sending`.

## Solution Assessment

Yes. Reusing the existing `message_sending` hook in the WebChat final path is the narrowest maintainable direction; adding a WebChat-only hook or requiring a universal outbound boundary would be broader than this issue.
