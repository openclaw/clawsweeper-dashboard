## Summary

Keep open. Current main still sends WebChat final assistant output directly from the Gateway chat event path without invoking the existing `message_sending` hook, while the shared outbound delivery path already applies that hook; related open issues are broader or sibling hook-coverage trackers, not replacements for this narrow WebChat parity gap.

## What This Changes

The issue requests wiring the existing `message_sending` plugin hook into the WebChat/WebSocket final assistant delivery path.

## Best Possible Solution

Wire the existing `message_sending` hook into WebChat final assistant delivery with `channelId: "webchat"`, `sessionKey`, and run correlation, apply rewrite/cancel consistently to WebSocket and node-session payloads, and leave broader outbound-policy unification to the umbrella items.

## Reproduction Assessment

Yes, at source level. Register a WebChat run, emit assistant text, then emit lifecycle `end`; current main resolves the session and reaches `emitChatFinal`, which broadcasts the final chat payload without any `message_sending` hook call.

## Solution Assessment

Yes. Reusing the existing `message_sending` hook in the WebChat final path is the narrowest maintainable fix; adding a WebChat-only hook or attempting universal outbound policy unification would be broader than this issue.
