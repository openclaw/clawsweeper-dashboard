## Summary

Keep open. Current main still delivers WebChat final assistant output directly from the Gateway chat event path without invoking the existing `message_sending` hook, while the documented hook contract and shared outbound delivery path already support rewrite/cancel behavior; the related open issues are broader umbrella or sibling hook-coverage trackers, not replacements for this narrow WebChat parity gap.

## What This Changes

The issue requests firing the existing `message_sending` hook for WebChat/WebSocket final assistant delivery.

## Best Possible Solution

Wire the existing `message_sending` hook into WebChat final assistant delivery with `channelId: "webchat"`, session/run context, rewrite/cancel handling, and focused regression coverage, while leaving broader outbound lifecycle unification to the umbrella issues.

## Reproduction Assessment

Yes, at source level. Register a WebChat run, emit assistant text, then emit lifecycle `end`; current main resolves the session and reaches `emitChatFinal`, which broadcasts the final chat payload without any `message_sending` hook call.

## Solution Assessment

Yes. Reusing the existing `message_sending` hook in the WebChat final path is the narrowest maintainable fix; adding a WebChat-only hook or attempting a universal outbound policy boundary would be broader than this issue.
