- `impact:security`: Pre-send guardrail, DLP, or policy plugins relying on `message_sending` can silently miss WebChat final delivery.

## Summary

Keep open. Current main still broadcasts WebChat final assistant output directly from `emitChatFinal` without invoking `message_sending`, while the documented hook contract and shared outbound delivery path already support rewrite/cancel behavior.

## What This Changes

The issue asks WebChat/WebSocket final assistant delivery to invoke the existing `message_sending` plugin hook with session context and rewrite/cancel handling.

## Best Possible Solution

Wire the existing `message_sending` hook into WebChat final assistant delivery with session/run context, rewrite/cancel handling, and UI-safe cancellation semantics, while leaving universal outbound lifecycle redesign to the umbrella issues.

## Reproduction Assessment

Yes, at source level. Register a WebChat run that emits assistant text and a lifecycle end event; current main resolves the session and reaches `emitChatFinal`, which broadcasts the final chat payload without any `message_sending` hook call.

## Solution Assessment

Yes. Reusing the existing `message_sending` hook in the WebChat final path is the narrowest maintainable solution, provided the fix also handles final rewrite/cancel behavior without leaking already-streamed text.
