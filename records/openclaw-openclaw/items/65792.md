## Summary

Keep open. Current main and v2026.5.7 still bypass `message_sending` in WebChat final delivery, while the shared outbound and Telegram delivery paths invoke the existing hook; the related open reports are broader hook-coverage or policy umbrellas, not replacements for this narrow WebChat parity bug.

## What This Changes

Request that WebChat/WebSocket final assistant delivery run the existing `message_sending` hook so plugins can rewrite or cancel outbound content like channel delivery paths do.

## Best Possible Solution

Wire the existing plugin `message_sending` hook into WebChat final assistant delivery with `channelId: "webchat"`, `sessionKey`, and `runId`, apply rewrite/cancel consistently to WebSocket and node-session payloads, and keep the broader lifecycle policy work in the umbrella issues.

## Reproduction Assessment

Yes, at source level. Drive `createAgentEventHandler` with assistant text for a registered WebChat run followed by lifecycle `end`; current main reaches `emitChatFinal`, which broadcasts the final payload without `runMessageSending`, though I did not run a live WebSocket session in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse the existing `message_sending` hook rather than add a WebChat-only hook, while explicitly handling rewrite and cancel so stale streamed text is not persisted.
