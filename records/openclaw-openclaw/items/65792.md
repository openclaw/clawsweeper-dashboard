## Label Justifications

- `P2`: The issue is a source-proven hook coverage gap on a default chat surface, but it is not a core runtime outage.
- `impact:security`: Plugins may rely on `message_sending` to redact or cancel sensitive outbound content, and WebChat final delivery can bypass that pre-send guardrail.

## Summary

Keep open. Current `main` and the latest stable tag still send the WebChat final assistant payload directly from the Gateway chat event path without invoking `message_sending`, while the documented hook contract and shared outbound delivery path already support rewrite/cancel behavior.

## What This Changes

Wire the existing `message_sending` plugin hook into WebChat/WebSocket final assistant delivery so hook rewrite/cancel semantics apply before the final chat payload is delivered.

## Best Possible Solution

Add a focused Gateway/WebChat hook bridge for final assistant delivery with `channelId: "webchat"`, session/run context, rewrite/cancel handling, failure isolation, and regression coverage for streamed-final reconciliation.

## Reproduction Assessment

Yes, at source level. A WebChat assistant reply reaches lifecycle `end`, `finalizeLifecycleEvent` calls `emitChatFinal`, and current `main` broadcasts the final payload without any `message_sending` hook call.

## Solution Assessment

Yes for this narrow gap. Reusing the existing `message_sending` hook in the WebChat final path is more maintainable than adding a WebChat-only hook, as long as the implementation handles final rewrite/cancel semantics with streamed deltas.
