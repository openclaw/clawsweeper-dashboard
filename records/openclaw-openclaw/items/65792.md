## Summary

Keep open. Current main still has the WebChat final-delivery gap: `emitChatFinal` broadcasts the final chat payload directly, while the shared outbound path runs `message_sending`; the related open reports are broader hook-coverage/policy items, not replacements for this narrow parity bug.

## What This Changes

Request that WebChat/WebSocket assistant final delivery run the existing `message_sending` plugin hook before broadcasting to WebSocket and node-session clients.

## Best Possible Solution

Wire the existing plugin `message_sending` hook into the WebChat final assistant path with `channelId: "webchat"`, `sessionKey`, and run correlation, apply rewritten final content consistently to WebSocket and node-session payloads, and handle cancellation without preserving stale streamed text; leave the broader unbypassable outbound-policy work to #56349/#50126.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to drive `createAgentEventHandler` with assistant text followed by lifecycle `end`; current main emits the final WebChat payload from `emitChatFinal` without any hook-runner call.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing `message_sending` hook rather than adding a WebChat-only hook. The implementation should also account for the existing final/no-message stream fallback so cancellation is observable to WebChat clients.
