## Summary

Keep open. Current main still sends the WebChat final assistant payload from the Gateway chat path without invoking `message_sending`, while the documented hook contract and shared outbound delivery path already support rewrite/cancel behavior; the related open reports are broader umbrella or sibling hook-coverage trackers, not replacements for this narrow WebChat gap.

## What This Changes

The issue asks OpenClaw to fire the existing `message_sending` plugin hook for WebChat/WebSocket final assistant delivery.

## Best Possible Solution

Wire `message_sending` into Gateway WebChat final assistant delivery with session/run context, rewrite/cancel handling, and focused regression coverage while keeping universal outbound policy design in the broader trackers.

## Reproduction Assessment

Yes, at source level. Follow a WebChat `chat.send` assistant run to `emitChatFinal`; current main reaches the direct final `chat` broadcast without any `message_sending` hook call.

## Solution Assessment

Yes in direction: reusing the existing `message_sending` hook is narrower than adding a WebChat-specific hook. The remaining maintainer decision is how final rewrite/cancel semantics reconcile with already-streamed deltas.
