## Summary

Keep open. Current main still has the reported WebChat final-delivery gap: `emitChatFinal` constructs and emits the final chat payload directly, while the shared outbound and Telegram reply paths run `message_sending` before delivery. No open PR appears to wire this exact WebChat path.

## What This Changes

Request WebChat/WebSocket assistant final delivery to invoke the `message_sending` plugin hook so plugins can rewrite or cancel outbound content before the UI receives it.

## Best Possible Solution

WebChat final delivery should run the existing plugin `message_sending` hook before constructing visible assistant content, pass `channelId: "webchat"` with `sessionKey` and run correlation, apply rewritten text to both WebSocket and node-session payloads, and isolate hook failures so delivery still completes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit assistant text plus a lifecycle end through `createAgentEventHandler` and observe that the final WebChat payload is broadcast without any `runMessageSending` call; the outbound reference paths show the expected hook call.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing hook into the WebChat final path; broader streaming and unbypassable policy semantics should stay with the related architecture issues.
