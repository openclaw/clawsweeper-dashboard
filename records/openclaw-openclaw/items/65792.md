## Label Justifications

- `P2`: This is a real hook-coverage gap with limited blast radius to WebChat/plugin policy behavior, but it is not an emergency runtime outage.
- `impact:security`: The missing hook can bypass plugin-defined pre-send rewrite/cancel guardrails on the default WebChat delivery path.

## Summary

Keep open. Current main and the latest shipped tag still broadcast WebChat final assistant payloads directly, while the documented and shared outbound path already supports `message_sending` rewrite/cancel behavior; related open issues are broader hook-coverage or policy-boundary trackers, not replacements for this narrow WebChat gap.

## What This Changes

The issue asks OpenClaw to run the existing `message_sending` plugin hook for WebChat/WebSocket final message delivery.

## Best Possible Solution

Wire `message_sending` into WebChat final delivery through a single Gateway helper with explicit rewrite/cancel behavior, session/run context, failure isolation, and focused regression coverage for streamed and non-streamed final payloads.

## Reproduction Assessment

Yes, at source level. Following a WebChat assistant run to `emitChatFinal` reaches a direct final `chat` broadcast without any `message_sending` hook call, while the shared outbound path does call that hook.

## Solution Assessment

Yes in direction: reusing `message_sending` is narrower than adding a WebChat-specific hook. The maintainer decision is the exact final rewrite/cancel behavior after streaming deltas have already been emitted.
