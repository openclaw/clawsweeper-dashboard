## Summary

Keep this PR open. The target bug is valid and current main still lacks direct ACP run lifecycle cleanup, but the PR’s fixed 24-hour fallback can close an ACP run that is still legitimately running under the existing no-timeout or long-timeout contract.

## What This Changes

The PR adds ACP `mode: "run"` lifecycle cleanup, dispatch-failure runtime cleanup, removes ACP run timeout from backend runtime initialization options, and adds colocated regression tests.

## Best Possible Solution

Keep the lifecycle end/error cleanup and dispatch-failure cleanup, but make the fallback cleanup state-aware or timeout-derived; leave broader parent reset and orphan reaper work to the canonical ACP cleanup issue #68916.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an ACP `mode: "run"` spawn with `runTimeoutSeconds: 0` or an effective timeout over 24 hours where the lifecycle terminal event is missed; the PR head will close the runtime at the fixed fallback time despite the current no-timeout contract.

## Solution Assessment

No. The cleanup direction is narrow and maintainable, but the fixed wall-clock fallback is not the best fix; the safer implementation should check terminal task state or derive the cleanup delay from the explicit/effective run timeout plus grace.
