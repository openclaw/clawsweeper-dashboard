## Summary

## What This Changes

The PR branch adds ACP mode `run` lifecycle cleanup, a 24-hour fallback cleanup timer, dispatch-failure runtime-handle cleanup, and ACP spawn regression tests.

## Best Possible Solution

Keep the shipped task-maintenance and ACPX process cleanup path, and leave the reset-specific child ACP enumeration gap to the linked issue rather than merging this conflicting lifecycle-listener branch.

## Reproduction Assessment

Yes, source-level. The PR branch schedules an unconditional 24-hour runtime close if a lifecycle terminal event is missed and removes `runTimeoutSeconds` from ACP runtime metadata that current main still consumes.

## Solution Assessment

No. Current main's task-maintenance plus ACPX close/startup reaper path is the narrower maintainable solution; the PR's fixed fallback timer and timeout metadata removal are not safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve long ACP runs in fallback cleanup:** `src/agents/acp-spawn.ts:1030-1032`
  - body: The fallback timer always closes the ACP runtime after 24 hours when no lifecycle terminal event is observed. ACP runs can explicitly use `runTimeoutSeconds: 0`, and configured runs can exceed 24 hours, so this can terminate a still-running one-shot runtime; tie fallback cleanup to terminal task state or the effective timeout plus grace.
  - confidence: 0.9
- **[P2] Keep runTimeoutSeconds in ACP runtime options:** `src/agents/acp-spawn.ts:947-955`
  - body: This removes `runTimeoutSeconds` from `initializeSession` runtime options, so the ACP manager loses persisted `runtimeOptions.timeoutSeconds` used for its turn watchdog. Current main already gates unsupported timeout config emission in the runtime-options layer, so keep the metadata here instead of dropping it at spawn time.
  - confidence: 0.88
