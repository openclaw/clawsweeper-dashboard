## Summary

Keep open. The PR targets a real ACP run-session cleanup gap, but it is not ready as-is because the fixed 24-hour fallback can terminate legitimate long/no-timeout runs, the timeout cleanup commit drops ACP runtime timeout metadata, and the external PR body only lists formatter/test/typecheck proof rather than a real ACP run/process cleanup proof.

## What This Changes

The PR adds ACP `mode: "run"` lifecycle cleanup, dispatch-failure runtime-handle cleanup, removes `runTimeoutSeconds` from ACP runtime options, and adds `src/agents/acp-spawn.test.ts` regression coverage.

## Best Possible Solution

Keep the lifecycle end/error and dispatch-failure cleanup, make fallback cleanup respect terminal state or effective run timeout plus grace, preserve manager-side timeout metadata, and leave parent reset, PID fallback, and startup reaping to the broader orphan-cleanup issue.

## Reproduction Assessment

Yes, source-level. On the PR branch, an ACP `mode: "run"` with `runTimeoutSeconds: 0` or a timeout longer than 24 hours will still hit the fixed 24-hour fallback if the lifecycle terminal event is missed, and current docs/tests define those longer/no-timeout semantics.

## Solution Assessment

No. The cleanup direction is good, but the unconditional wall-clock fallback and timeout metadata removal are not the narrowest maintainable fix; repair those while keeping existing timeout semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve long ACP runs in fallback cleanup:** `src/agents/acp-spawn.ts:1030-1032`
  - body: The fallback timer always calls cleanup after 24 hours when no lifecycle terminal event is seen. ACP runs can explicitly use `runTimeoutSeconds: 0` for no-timeout behavior, and configured agent runs may legitimately exceed 24 hours, so this can close a still-running ACP runtime. Gate fallback cleanup on terminal task state or derive the delay from the effective run timeout plus grace.
  - confidence: 0.9
- **[P2] Keep runTimeoutSeconds in ACP runtime options:** `src/agents/acp-spawn.ts:953-957`
  - body: This removes `runTimeoutSeconds` from `initializeSession` runtime options, so the ACP manager no longer stores `runtimeOptions.timeoutSeconds` for its turn watchdog or supported backend config. Current docs and tests say the value applies to both the Gateway run and ACP runtime; keep the metadata here and solve unsupported timeout config in the runtime-control layer.
  - confidence: 0.86
