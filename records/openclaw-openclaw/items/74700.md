## Summary

Keep this PR open: it targets a real ACP run cleanup gap, but the proposed fixed 24-hour fallback can terminate legitimate no-timeout or longer-than-24-hour ACP runs when the lifecycle terminal event is missed.

## What This Changes

This PR adds ACP `mode: "run"` lifecycle-based runtime cleanup, dispatch-failure cleanup using the initialized runtime handle, and colocated regression tests.

## Best Possible Solution

Keep the lifecycle end/error cleanup and dispatch-failure handle cleanup, but make fallback cleanup depend on confirmed terminal state or the effective run timeout plus grace, and leave broader parent reset, PID fallback, and startup orphan recovery to #68916.

## Reproduction Assessment

Yes. A focused code-level reproduction is an ACP `mode: "run"` spawn with `runTimeoutSeconds: 0` or a timeout longer than 24 hours where the lifecycle terminal event is missed; the PR fallback still closes the runtime at the fixed 24-hour deadline.

## Solution Assessment

No. The cleanup direction is narrow and useful, but the fixed fallback policy should respect existing timeout semantics instead of imposing a 24-hour wall-clock cap.
