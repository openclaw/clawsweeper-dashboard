## Summary

Keep open: the PR targets a real ACP run cleanup gap, but the current head still introduces two P2 correctness regressions and lacks required after-fix real behavior proof.

## What This Changes

The PR closes ACP run runtimes on lifecycle end/error, adds a fixed 24-hour fallback cleanup, passes the initialized runtime close handle into dispatch-failure cleanup, and adds acp-spawn regression coverage.

## Best Possible Solution

Keep the lifecycle end/error and dispatch-failure cleanup, make fallback cleanup depend on terminal state or effective timeout plus grace, and preserve timeoutSeconds metadata while relying on the runtime-control layer to avoid unsupported backend config.

## Reproduction Assessment

Yes, source-level. The PR head schedules unconditional cleanup after 24 hours and removes timeoutSeconds metadata while current docs, tests, and manager code preserve no-timeout and long-timeout semantics.

## Solution Assessment

No. The cleanup direction is useful, but the current patch should preserve timeout metadata and avoid closing a still-running ACP runtime from a fixed wall-clock fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve long ACP runs in fallback cleanup:** `src/agents/acp-spawn.ts:1030-1032`
  - body: The fallback timer always calls cleanup after 24 hours when no lifecycle terminal event is seen. ACP runs can explicitly use `runTimeoutSeconds: 0` for no-timeout behavior, and configured agent runs may legitimately exceed 24 hours, so this can close a still-running ACP runtime. Gate fallback cleanup on terminal task state or derive the delay from the effective run timeout plus grace.
  - confidence: 0.9
- **[P2] Keep ACP timeout metadata in runtime options:** `src/agents/acp-spawn.ts:949-955`
  - body: Removing `runTimeoutSeconds` from `initializeSession` drops the `runtimeOptions.timeoutSeconds` metadata that the ACP manager uses for its turn watchdog. Current main already gates unsupported backend timeout config in the runtime-control layer, so keep this metadata and solve backend emission there.
  - confidence: 0.88
