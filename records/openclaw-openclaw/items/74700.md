## Summary

Keep open: this PR targets a real ACP run-cleanup gap, but it is not merge-ready because it lacks real behavior proof and introduces two timeout-semantics regressions.

## What This Changes

The PR adds ACP `mode: "run"` lifecycle cleanup, dispatch-failure runtime-handle cleanup, and regression tests in `src/agents/acp-spawn.ts` and `src/agents/acp-spawn.test.ts`.

## Best Possible Solution

Keep the lifecycle end/error and dispatch-failure cleanup direction, but preserve `runTimeoutSeconds` metadata and make fallback cleanup depend on terminal state or effective timeout plus grace.

## Reproduction Assessment

Yes, source-level. On the PR branch, a run with `runTimeoutSeconds: 0` or a timeout beyond 24 hours can still be closed by the fixed 24-hour fallback if the lifecycle terminal event is missed, while current docs and tests define longer/no-timeout behavior.

## Solution Assessment

No. The cleanup direction is useful, but the patch should be repaired to preserve existing timeout semantics and handle unsupported timeout config in the manager/runtime-control layer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve long ACP runs in fallback cleanup:** `src/agents/acp-spawn.ts:1030-1032`
  - body: The new fallback timer always calls cleanup after 24 hours when no lifecycle terminal event is seen. ACP runs can explicitly use `runTimeoutSeconds: 0`, and configured agent runs may legitimately exceed 24 hours, so this can close a still-running ACP runtime; derive the fallback from terminal task state or the effective timeout plus grace.
  - confidence: 0.9
- **[P2] Keep runTimeoutSeconds in ACP runtime options:** `src/agents/acp-spawn.ts:953-957`
  - body: This removes `runTimeoutSeconds` from `initializeSession` runtime options, so the ACP manager loses persisted `runtimeOptions.timeoutSeconds` used for its turn watchdog and supported backend config. Keep the metadata here and solve unsupported timeout config in the manager/runtime-control layer.
  - confidence: 0.86
