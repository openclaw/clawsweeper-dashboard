## Summary

Keep this PR open, but it should not merge as-is. The lifecycle end/error cleanup and dispatch-failure cleanup address a real ACP run leak, but the fixed 24-hour fallback can close a still-running ACP run when lifecycle events are missed, which conflicts with current no-timeout and long-timeout behavior.

## What This Changes

The PR adds lifecycle cleanup for ACP `mode: "run"` sessions, a 24-hour fallback cleanup timer, dispatch-failure runtime cleanup, and ACP spawn regression tests while removing `runTimeoutSeconds` from ACP runtime initialization options.

## Best Possible Solution

Keep the lifecycle end/error cleanup and dispatch-failure cleanup, but change the fallback to close only after confirmed terminal task state or after a grace period derived from the effective run timeout. Leave parent reset cleanup, PID-level orphan recovery, and startup reaping to the broader #68916 thread.

## Reproduction Assessment

Yes. A focused code-level reproduction is an ACP `mode: "run"` spawn with `runTimeoutSeconds: 0` or a configured/default timeout longer than 24 hours where the lifecycle terminal event is missed; the PR head still closes the runtime at the fixed fallback deadline.

## Solution Assessment

No. The cleanup direction is narrow and maintainable, but the fixed wall-clock fallback is not the best fix because it bypasses existing timeout semantics; terminal task-state cleanup or timeout-derived fallback timing is safer.
