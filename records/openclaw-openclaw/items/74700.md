## Summary

Keep open. The PR addresses a real ACP run cleanup leak, but the current branch still has blocking timeout-semantics regressions and lacks after-fix real ACP process cleanup proof.

## What This Changes

The PR adds ACP `mode: "run"` lifecycle cleanup on end/error events, a fixed 24-hour fallback cleanup timer, dispatch-failure runtime-handle cleanup, and regression tests.

## Best Possible Solution

Keep the lifecycle cleanup and dispatch-failure cleanup direction, but retain runtime timeout metadata, rely on the manager/runtime-options layer for unsupported backend timeout controls, and make fallback cleanup depend on terminal state or effective timeout plus grace.

## Reproduction Assessment

Yes, source-level. On the PR branch, a run with `runTimeoutSeconds: 0` or a timeout above 24 hours can still be closed by the fixed 24-hour fallback if the lifecycle terminal event is missed, and the branch removes runtime timeout metadata that current manager code consumes.

## Solution Assessment

No. The cleanup goal is right, but the current patch is not the best fix because it regresses existing timeout semantics instead of preserving metadata and tying fallback cleanup to the real run lifecycle or effective timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve long ACP runs in fallback cleanup:** `src/agents/acp-spawn.ts:1030-1032`
  - body: The new fallback timer always closes the ACP runtime after 24 hours when no lifecycle terminal event is observed. ACP runs can explicitly use `runTimeoutSeconds: 0`, and configured agent runs may legitimately exceed 24 hours, so this can terminate a still-running one-shot runtime; derive the fallback from terminal task state or the effective timeout plus grace.
  - confidence: 0.9
- **[P2] Keep runTimeoutSeconds in ACP runtime options:** `src/agents/acp-spawn.ts:949-955`
  - body: This removes `runTimeoutSeconds` from `initializeSession` runtime options, so the ACP manager loses persisted `runtimeOptions.timeoutSeconds` used for its turn watchdog. Current main already gates unsupported timeout config emission in the runtime-options layer, so keep the metadata here instead of dropping it at spawn time.
  - confidence: 0.88
