## Summary

Keep open: current main and v2026.6.8 still skip targeted diagnostics for non-empty `plugins.allow`, while this branch adds a narrow warning for populated allowlists that match no discovered plugin id. The related broader warning work at https://github.com/openclaw/openclaw/issues/68780 and https://github.com/openclaw/openclaw/pull/78105 overlaps but does not supersede this PR.

## What This Changes

The PR updates plugin loader provenance warnings to emit a targeted message for populated allowlists whose entries match no discovered plugin id, plus regression tests for mismatch, partial-match, and bundled-only allowlists.

## Best Possible Solution

Land this narrow diagnostic change after required checks are green, while leaving broader first-run warning remediation to https://github.com/openclaw/openclaw/issues/68780 and https://github.com/openclaw/openclaw/pull/78105.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 return before inspecting any non-empty `plugins.allow`, so a channel-style entry such as `feishu` with a discovered plugin id such as `openclaw-lark` gets no targeted mismatch warning.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix for this diagnostic path: it stays in the existing loader provenance warning helper, preserves empty-allowlist copy and allow/deny semantics, and leaves channel-id aliasing or broader first-run UX to separate product/security work.
