## Summary

Keep this PR open, but it is not merge-ready: the diagnostics/control-lane direction is useful, yet the updated Telegram debounce bypass can clear the keyed lane for ordinary private captioned-media DMs, and the supplied proof does not directly show the affected live Telegram behavior.

## Best Possible Solution

Preserve ordinary Telegram DM serialization, make the direct-control bypass an explicit control/abort decision rather than a generic debounce result, add captioned-media regression coverage, and land the diagnostics only with targeted live Telegram proof.

## Reproduction Assessment

Yes, by source inspection: a private Telegram message with both text and media passes hasMedia=true into shouldDebounceTextInbound, which returns false and causes the caller to set debounceKey to null. I did not run the runtime, but the current PR code path is direct and high confidence.

## Solution Assessment

No; the explicit direct-control lane is the right direction, but using the generic debounce predicate as the bypass decision is too broad. The safer fix is to keep the keyed lane unless an explicit control/abort or direct-control-priority predicate matches.
