Full review comments:

- **[P1] Keep private DMs on a keyed lane unless they are control messages:** `extensions/telegram/src/bot-handlers.runtime.ts:1950`
  - body: When Telegram queue mode resolves to `interrupt`, this branch nulls `debounceKey` for every private message. Combined with the new per-message private sequential key, ordinary DMs from the same chat no longer share the grammY sequential lane or the inboundDebouncer key, so two normal messages can enter the same session concurrently and race history/reply state. Limit the bypass to abort/direct-control messages or preserve a per-chat key while still interrupting the active run.
  - confidence: 0.88
- **[P2] Pass chat type before relying on direct-control debounce bypass:** `src/channels/inbound-debounce-policy.ts:33`
  - body: The new debounce policy only skips debounce when `chatType` normalizes to direct, but Telegram still calls `shouldDebounceTextInbound` with only text, config, and bot username. That means manual-override or urgent direct messages can still wait behind the inbound debounce window before the runtime sees the new control-lane priority.
  - confidence: 0.84
- **[P2] Document or keep internal the new SDK contract:** `src/plugin-sdk/channel-inbound.ts:172-173`
  - body: This export makes turn state part of the public plugin SDK, and the PR also exposes `resolveQueueSettings`/`QueueSettings` through `reply-runtime`, but the SDK docs and subpath docs do not define the new support contract. Either keep these helpers internal for the spike or add the public docs/support semantics maintainers want external plugins to rely on.
