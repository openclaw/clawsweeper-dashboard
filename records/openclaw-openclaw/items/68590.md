## Summary

Keep open: this PR still has blocking correctness issues already identified in prior review, and the branch has not been updated since the reviewed commit. The useful `index.yml` rewrite idea is not on current main, but merging this patch as-is would weaken the existing forced QMD repair path and it lacks after-fix real behavior proof.

## What This Changes

The PR adds a QMD `index.yml` rewrite during collection bootstrap and replaces forced managed-collection repair remove/add logic with `ensureCollections()` plus matching test expectation changes.

## Best Possible Solution

Keep the index rewrite concept, but serialize YAML safely and preserve forced repair reset semantics through an explicit force path or the existing remove/add loop, with focused tests for both YAML-special values and listed-collection repair.

## Reproduction Assessment

Yes for the blocking regression. A focused unit reproduction is to mock `qmd collection list --json` returning matching managed collections, make `qmd update` fail with `UNIQUE constraint failed: documents.collection, documents.path`, and verify the PR skips the remove/add reset before retrying.

## Solution Assessment

No. Rewriting the QMD index before reconciliation is a reasonable part of the fix, but replacing forced repair with the normal idempotent ensure path is not the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P1] Preserve forced repair rebuilds:** `extensions/memory-core/src/memory/qmd-manager.ts:940`
  - body: `rebuildManagedCollectionsForRepair()` now delegates to `ensureCollections()`, which skips remove/add when QMD lists matching path and pattern metadata.   - confidence: 0.86
