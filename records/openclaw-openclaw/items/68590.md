## Summary

Keep this PR open. The intent is useful, but the diff replaces a current forced QMD repair rebuild with the normal idempotent ensure path, which can skip remove/add when QMD still lists matching metadata; the PR also lacks real QMD repair proof.

## What This Changes

The PR writes QMD collection metadata to `index.yml`, delegates managed collection repair to `ensureCollections()`, and updates QMD manager tests to expect repair without removals.

## Best Possible Solution

Preserve repair-time forced remove/add semantics, add any index.yml rewrite through a YAML-safe path only if real QMD repair proof supports it, and cover both normal ensure and repair paths with focused tests.

## Reproduction Assessment

Yes for the blocking PR regression from source inspection: if QMD lists matching managed collections and qmd update fails with a duplicate-document constraint, the proposed ensureCollections() repair path skips remove/add before retrying. I did not run tests because this review is read-only.

## Solution Assessment

No. Rewriting index.yml may be a useful resilience step, but replacing forced repair rebuilds with normal idempotent collection ensure is not the narrowest maintainable fix without real QMD repair proof.
