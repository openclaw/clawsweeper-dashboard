## Summary

Keep open: the branch has a useful QMD index-rewrite idea, but the current diff regresses the forced collection reset used by repair and hand-writes unsafe YAML scalars; it also still lacks real behavior proof from a QMD repair run.

## What This Changes

The branch writes QMD `index.yml` before collection reconciliation and changes repair to call `ensureCollections()` instead of the explicit remove/add loop, with tests updated to expect no repair-time removals.

## Best Possible Solution

Keep the index rewrite only if needed, but serialize QMD config safely and preserve explicit forced remove/add repair semantics with focused coverage for listed collections and YAML-special values.

## Reproduction Assessment

Yes for the blocking PR regression from source inspection: mock `qmd collection list --json` returning matching managed collections, make `qmd update` fail with a duplicate-document constraint, and the proposed `ensureCollections()` repair path skips remove/add before retrying. I did not run tests because this cleanup review is read-only.

## Solution Assessment

No. Rewriting the QMD index may be a reasonable resilience step, but replacing the forced repair reset with the normal idempotent ensure path is not the narrowest maintainable fix.
