## Summary

Keep open: the patch is a plausible security hardening for a real dotenv precedence gap, but it intentionally breaks workspace `.env` provider-credential sourcing and still lacks after-fix real runtime proof outside the focused test fixture.

## What This Changes

The PR expands workspace `.env` blocking to registered provider credential env vars and adds dotenv regression coverage that trusted global provider auth values win over workspace values.

## Best Possible Solution

Land the registry-backed blocklist after maintainers accept the provider credential-source policy, after-fix real runtime proof is added, and user-facing migration wording makes shell env or trusted global dotenv the canonical credential source.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main loads cwd `.env` before global `.env`, and unblocked registered provider keys can therefore win over trusted global values.

## Solution Assessment

Yes for the code boundary, but not merge-ready yet. The registry-backed list is the maintainable fix, while the compatibility break and after-fix proof need maintainer handling before landing.
