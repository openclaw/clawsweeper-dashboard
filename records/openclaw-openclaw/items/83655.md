## Summary

Keep open: the PR is a focused and plausible credential-source hardening, but it still needs after-fix real behavior proof and maintainer acceptance of the workspace `.env` compatibility break before merge.

## What This Changes

The PR blocks registered provider credential environment variables from project-local `.env` files while preserving shell and trusted global dotenv credential sources.

## Best Possible Solution

Merge the denylist and registry-backed regression only after maintainers accept the credential-source compatibility break, the docs explain the new exception, and the contributor adds redacted after-fix runtime proof.

Yes. Source inspection shows current main loads cwd `.env` before trusted global dotenv, and the PR body provides shipped-version terminal output showing unblocked provider keys such as `GEMINI_API_KEY`, `XAI_API_KEY`, and `MISTRAL_API_KEY` winning from workspace dotenv.

## Solution Assessment

Mostly yes. Blocking registered provider credentials from untrusted workspace dotenv is the right security boundary, but the merge-ready solution should also update user-facing precedence docs and include after-fix runtime proof.
