## Summary

Keep open, but do not merge as-is. Current main and docs still treat cwd `.env` as a valid non-overriding source, while this PR unconditionally blocks provider credentials from that source and the current assignee has recommended against that policy because it breaks normal project-local API-key workflows.

## What This Changes

The PR blocks registered provider credential env vars from workspace `.env`, adds provider-registry-backed dotenv regression coverage, and documents trusted credential sources.

## Best Possible Solution

Preserve project-local provider credentials by default unless maintainers explicitly change the env-source policy; a warning or shadow-only warning is the safer path for the risk described here.

## Reproduction Assessment

Yes for the behavior change: source inspection shows current main loads cwd `.env` before the global fallback and only blocks a subset of provider credential names, and the PR body supplies before/after terminal output for that dotenv-loader path. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as-is. The implementation is focused and test-covered, but unconditional blocking is not the narrowest maintainable solution after the assignee’s compatibility objection; warning on shadowing or preserving project-local-only keys would better match current behavior.
