## What This Changes

The PR adds registered provider auth env var names to the workspace dotenv blocklist and adds a regression test that expects trusted global dotenv values to win over cwd dotenv values for those keys.

## Best Possible Solution

Merge the denylist plus registry regression once maintainers accept the workspace .env credential break and proof/docs reflect shell or trusted global dotenv as the canonical provider credential sources.

## Reproduction Assessment

Yes. Source inspection shows current main loads cwd .env before global .env and does not block many registered provider auth keys, so a cwd/global BRAVE_API_KEY or AI_GATEWAY_API_KEY fixture would reproduce the precedence problem.

## Solution Assessment

Yes, with review gates. The code shape is a narrow fix that keeps dotenv loading independent of plugin discovery while the regression test ties the hardcoded denylist to the provider registry; merge should wait on real behavior proof and explicit compatibility acceptance.
