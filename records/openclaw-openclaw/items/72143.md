## Label Justifications

- `P2`: This is a normal-priority exec approval follow-up reliability and messaging bug with limited but real channel/session impact.
- `impact:session-state`: The failure occurs when OpenClaw cannot resume the original agent session for an async exec follow-up and falls back outside the session path.
- `impact:message-loss`: The fallback path can distort or bypass the intended follow-up delivery, and related reports show exec approval follow-up results being lost or misdelivered.

## Summary

Keep open. Current main and the latest release still leave part of the reported behavior in place: successful `code 0` fallbacks are covered, but other direct fallback deliveries can still prepend the internal resume-failure prefix, and the session resume path still has only one attempt before fallback.

## What This Changes

The issue requests bounded retry/backoff for exec approval follow-up session resume, internal-only resume-failure logging, and prefix-free direct fallback delivery that preserves the command result.

## Best Possible Solution

Land a focused maintainer-reviewed fix in the exec approval follow-up helper that retries session resume briefly, logs resume failures internally with safe metadata, and never exposes the internal resume-failure prefix in direct fallback delivery.

## Reproduction Assessment

Yes at source level: mock `callGatewayTool("agent")` to fail with a deliverable route and a non-`code 0` or unparsed exec result, and current main can still prepend the internal resume-failure prefix. I did not run a live channel repro in this read-only review.

## Solution Assessment

Yes for the desired end state, but not as a new autonomous fix right now. The narrow code owner is `src/agents/bash-tools.exec-approval-followup.ts`; maintainers should first decide whether to split, replace, or coordinate with the broad overlapping PR.
