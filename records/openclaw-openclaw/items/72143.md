## Summary

Keep open. Current main still exposes the internal resume-failure prefix on denied and non-success direct fallback messages, and the session follow-up path still makes only one resume attempt before falling back; the prior focused PR is closed unmerged and the broader open PR is not a shipped fix.

## What This Changes

The issue requests retry/backoff for exec approval follow-up session resume failures, silent direct fallback delivery, and internal-only logging while preserving command result content.

## Best Possible Solution

Add bounded short retry/backoff around the existing session follow-up call, remove the resume-failure prefix from every direct fallback message, and log sanitized resume-failure metadata internally after retries are exhausted.

## Reproduction Assessment

Yes from source inspection. Mocking callGatewayTool rejection in the existing follow-up test exercises the direct fallback path, and current main still expects the internal prefix while the implementation shows only one session-resume attempt.

## Solution Assessment

Yes. The narrow maintainable fix is scoped to the exec approval follow-up path: retry the existing session resume call, keep direct fallback user-facing content clean, and leave broader delivery lifecycle issues in #76162 and #76233 separate.

In `src/agents/bash-tools.exec-approval-followup.ts`, add bounded short retry/backoff around the existing `callGatewayTool("agent", ...)` session follow-up attempt before direct fallback; keep tests fast by using a small injectable/testable delay seam rather than real multi-second sleeps. Remove the user-facing `Automatic session resume failed...` prefix from all direct fallback messages, including denied and non-success results, while preserving sanitized command result/status text. Check open PR #68129 first because it touches the same file, but do not pull in its broader prompt-policy changes unless they have already landed.
