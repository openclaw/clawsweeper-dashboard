- `impact:message-loss`: The issue concerns fallback delivery of async exec follow-up messages to chat surfaces after the normal session handoff fails.

## Summary

Keep open. Current main and the latest release still make only one session-followup attempt before direct fallback, and they still prepend the internal resume-failure prefix for denied or non-success direct fallback messages; the existing shipped fix only covers successful completions.

## What This Changes

The issue requests retry/backoff for async exec approval follow-up session resume and removal of the internal resume-failure prefix from user-facing direct fallback messages.

## Best Possible Solution

Add bounded retry/backoff around the session follow-up call, keep direct fallback user-clean for every outcome, and log sanitized resume-failure metadata internally after retry exhaustion.

## Reproduction Assessment

Yes, at source level. Mocking callGatewayTool("agent") rejection with a deliverable Telegram or Discord route exercises the direct fallback path, and the current denied test still expects the internal prefix.

## Solution Assessment

Yes. The requested bounded retry plus silent direct fallback is the narrow maintainable fix for the documented follow-up contract, while broader raw-output and late-output leakage reports should stay separate.
