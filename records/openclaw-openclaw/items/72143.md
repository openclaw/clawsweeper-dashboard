## Summary

Keep open. Current main still matches the reported bug: the async exec approval follow-up path makes one session-resume attempt, then direct fallback can prepend the internal resume-failure banner for denied and non-success results. A prior shipped fix only covers successful completions, while the closed partial PR is unmerged and the broad overlapping PR remains open rather than canonical for this issue.

## What This Changes

The issue requests bounded retry/backoff for async exec approval follow-up session resume, silent direct fallback without the internal prefix, preserved result content, and internal-only diagnostics.

## Best Possible Solution

Add a narrow agents fix that retries the existing session follow-up call with bounded short backoff, removes the resume-failure banner from every direct fallback delivery path, preserves sanitized command result/status text, and logs sanitized resume diagnostics only after retries are exhausted.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists by mocking `callGatewayTool` to reject in `src/agents/bash-tools.exec-approval-followup.test.ts`; the current denied fallback test expects the internal prefix in the outbound message.

## Solution Assessment

Yes. The requested approach is the narrow maintainable fix for this path: retry the existing `agent` follow-up call briefly, then silently direct-deliver sanitized content while keeping resume diagnostics internal. A broader delivery-policy rewrite is not needed for this issue.
