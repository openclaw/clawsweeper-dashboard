## Summary

Keep open. Current main still exposes the internal resume-failure prefix for denied and other non-success direct fallback messages, and the session follow-up path still makes one resume attempt before falling back; the related prefix-removal PR was closed unmerged and explicitly left retry/backoff out.

## What This Changes

The issue requests bounded retry/backoff for exec approval session follow-up fallback, removal of the internal fallback prefix from user-facing direct delivery, and internal logging of resume failures.

## Best Possible Solution

Add bounded short retry/backoff around the session follow-up call, remove the resume-failure prefix from every direct fallback message, and log sanitized resume-failure metadata internally after retries are exhausted.

## Reproduction Assessment

Yes. The focused reproduction is to mock callGatewayTool rejection in src/agents/bash-tools.exec-approval-followup.test.ts; the denied resume-failure test currently expects the internal prefix, and the source shows one session-follow-up attempt before fallback.

## Solution Assessment

Yes. The requested retry plus silent direct fallback is the narrow maintainable fix; it should reuse the existing idempotency key and avoid changing exec approval policy or command execution semantics.
