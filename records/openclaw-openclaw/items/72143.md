## Summary

Keep open. Current main still only partially addresses the report: successful completion fallback no longer shows the banner, but denied/non-success direct fallback still exposes the internal resume-failure prefix, the session follow-up path still makes a single agent call before falling back, and resume failures are not logged internally when direct fallback succeeds.

## What This Changes

The issue requests bounded retry/backoff for exec approval session follow-up resume, removal of the internal fallback prefix from user-facing direct delivery, internal-only resume-failure logging, and preservation of sanitized command status content.

## Best Possible Solution

Land a narrow agents fix that retries the existing session follow-up call with short bounded backoff, removes the resume-failure banner from every user-facing direct fallback path, preserves sanitized result or denial content, and emits sanitized internal diagnostics only after retry exhaustion.

## Reproduction Assessment

Yes. The focused unit reproduction is to mock `callGatewayTool` rejection in `src/agents/bash-tools.exec-approval-followup.test.ts`; the denied fallback test currently expects the internal prefix, and the source shows only one session follow-up attempt.

## Solution Assessment

Yes. The requested direction is the narrow maintainable solution: add bounded retry around the existing follow-up call, then silently direct-deliver sanitized content if retries fail. A broader delivery-policy rewrite is not required, but overlap with #68129 should be reconciled first.
