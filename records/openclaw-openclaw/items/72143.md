## Summary

Keep open. Current main still preserves the reported internal prefix on denied and non-success direct fallback paths, and the session follow-up path still performs a single gateway agent call before falling back. The successful-completion leak was partially handled, but this issue’s retry/backoff and all-fallback-prefix removal remain unresolved.

## What This Changes

The issue requests bounded retry/backoff for exec approval follow-up session resume failures, prefix-free user-facing direct fallback delivery, internal-only diagnostics, and preservation of the command result/status content.

## Best Possible Solution

Land a narrow agents fix that retries the existing session follow-up call with bounded backoff, removes the resume-failure banner from every direct fallback delivery path, preserves sanitized result/status text, and emits sanitized internal diagnostics only after retry exhaustion.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists in `src/agents/bash-tools.exec-approval-followup.test.ts` by mocking `callGatewayTool` to reject; the denied fallback test currently expects the internal prefix in the outbound message. The one-shot resume behavior is also visible directly in `sendExecApprovalFollowup`.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: wrap the existing `agent` follow-up call with short bounded retries, then silently direct-deliver sanitized content if needed. A broader delivery-policy rewrite is not required, though the fix must reconcile with open PR #68129.
