## Label Justifications

- `P2`: This is a bounded exec approval follow-up reliability bug with confusing user-facing output, but not a core outage or security emergency.
- `impact:session-state`: The failure is in resuming the original agent session for an exec approval follow-up.
- `impact:message-loss`: When resume fails, direct fallback can distort or misframe the command result visible to the user.

## Summary

Keep open. Current main and the latest release still have the one-shot exec approval session-followup attempt, and direct fallback can still prepend the internal resume-failure prefix for non-success fallback messages.

## What This Changes

The issue asks OpenClaw to retry exec approval session resume before direct fallback, hide the internal resume-failure prefix from users, preserve the command result, and log resume failures internally.

## Best Possible Solution

Land a narrow owner-path fix that retries transient session-followup failures with bounded backoff, logs resume failure metadata internally, and keeps direct fallback user copy free of internal session-resume wording, coordinated with the overlapping delivery PR.

## Reproduction Assessment

Yes. From source, a focused reproduction can mock `callGatewayTool("agent")` failing and pass a nonzero/error `Exec finished` result with a deliverable route to observe the prefix path; I did not run live channel proof in this read-only review.

## Solution Assessment

Yes. Bounded retry around the existing session-followup call, internal-only failure logging, and prefix-free direct fallback is the narrow maintainable fix in the owning helper.
