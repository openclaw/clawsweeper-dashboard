## Label Justifications

- `P2`: The issue is a normal-priority exec approval follow-up reliability and messaging bug with limited but real channel/session impact.
- `impact:session-state`: The failure occurs when OpenClaw cannot resume the original agent session for an async exec follow-up and falls back outside the session path.
- `impact:message-loss`: The fallback path can distort or bypass the intended follow-up delivery, and sibling reports show related exec approval follow-up results being lost or misdelivered.

## Summary

Keep open. Current main partially improved the successful `code 0` path, but the exec approval follow-up fallback still has a user-facing internal resume-failure prefix for other direct fallback deliveries and still attempts session resume only once before falling back.

## What This Changes

The issue asks exec approval follow-up fallback to retry session resume with short backoff, log resume failures internally, and deliver the command result without an internal prefix if direct fallback is needed.

## Best Possible Solution

Land a focused maintainer-reviewed fix in the exec approval follow-up helper that adds bounded resume retry/backoff, logs resume failures with follow-up/session/target metadata, and keeps all direct fallback delivery free of internal resume-failure wording while preserving sanitized command result content.

Yes, from source inspection: mock `callGatewayTool("agent")` to fail with a deliverable route and a non-`code 0` or unparsed exec result, and current main can still prepend the internal resume-failure prefix. I did not run a live channel repro in this read-only pass.
