## Summary

Keep open. Current main and the latest release still retain the central gaps: successful `code 0` fallbacks suppress the reported prefix, but other direct fallback deliveries can still prepend the internal resume-failure text, and the session resume path still makes a single attempt before fallback.

## What This Changes

The issue asks for async exec approval follow-up fallback to retry session resume, hide internal resume-failure prefixes from user messages, and log resume-failure metadata internally.

## Best Possible Solution

Land a focused maintainer-reviewed fix in the exec approval follow-up helper that retries session resume briefly, logs resume failures internally with safe metadata, and never exposes the internal resume-failure prefix in direct fallback delivery.

## Reproduction Assessment

Yes, at source level: mock `callGatewayTool("agent")` to fail with a deliverable route and a non-`code 0` or unparsed exec result, and current main can still prepend the internal resume-failure prefix. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Yes for the desired end state, but not as a new autonomous fix right now. The narrow code owner is `src/agents/bash-tools.exec-approval-followup.ts`; maintainers should first decide whether to split, replace, or coordinate with the broad overlapping PR.
