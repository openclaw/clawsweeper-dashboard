## Label Justifications

- `P2`: The issue reports a real exec approval follow-up delivery/session-state bug with user-visible confusing output, but it is bounded to async exec fallback behavior rather than a core outage.
- `impact:session-state`: The failing path is specifically session resume/follow-up after an async exec approval, and current source still falls back after a single failed resume attempt.
- `impact:message-loss`: The fallback path can bypass the intended agent continuation and deliver confusing direct output to the user when the follow-up handoff fails.

## Summary

Keep open. Current main and the latest release still retain the central gaps: the exec approval follow-up path attempts session resume only once before fallback, and non-success direct fallback can still prepend the internal resume-failure text to user-facing delivery.

## What This Changes

The issue asks OpenClaw to retry exec approval follow-up session resume with short backoff, hide the internal resume-failure prefix from direct fallback delivery, and keep resume failure details in logs.

## Best Possible Solution

Land a narrow owner-approved fix that retries only the session-resume handoff before any direct send, removes internal resume-failure wording from all user-facing fallback delivery, logs the resume failure with correlation metadata, and adds focused regression tests.

## Reproduction Assessment

Yes by source inspection: current main still makes one `agent` resume call and then direct-fallbacks, and the direct fallback still prepends the internal prefix for non-success completion payloads after a resume error. I did not run a live channel repro in this read-only review.

## Solution Assessment

Yes on the requested user-facing end state, but the exact retry schedule and duplicate-delivery guard need maintainer approval before a repair PR is safe.
