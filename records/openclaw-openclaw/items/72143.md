## Summary

Keep open. Current main still exposes the internal resume-failure prefix on denied and non-success direct fallback messages, and the session follow-up path still makes a single resume attempt before falling back.

## What This Changes

Request retry/backoff for async exec approval session follow-up, silent direct fallback without the internal prefix, and internal-only resume failure logging.

## Best Possible Solution

Add bounded retry/backoff around the existing agent session follow-up, remove the resume-failure prefix from all direct fallback messages, and log sanitized resume-failure metadata internally after retries are exhausted.

## Reproduction Assessment

Yes. The focused reproduction is to mock callGatewayTool rejection in src/agents/bash-tools.exec-approval-followup.test.ts; current source and tests show the denied fallback still emits the internal prefix after one failed resume attempt.

## Solution Assessment

Yes. The requested retry plus silent direct fallback is the narrow maintainable fix; #67181 only covered successful completions, and #72148 was closed unmerged without the retry portion.

Fix the async exec approval follow-up fallback reported in openclaw/openclaw#72143. In src/agents/bash-tools.exec-approval-followup.ts, add bounded short retry/backoff around the existing callGatewayTool("agent", ...) session follow-up path before direct fallback; keep the retry timing small and testable. Remove the user-facing `Automatic session resume failed...` prefix from all direct fallback messages, including denied and non-success completions, while preserving the command result/status content. Add internal-only sanitized logging after retries are exhausted with useful metadata such as approvalId, sessionKey or safe route identifiers, delivery target type, and the error cause; do not leak command output, secrets, or user-facing internal machinery.
