## Summary

Keep open. Current main still has the two reported surfaces: default `sessions.list {}` loads and merges full session stores before responding, while `pi-trajectory-flush` still falls back to the fixed 10000 ms cleanup timeout because the caller passes no override.

## What This Changes

Request to keep `sessions.list` responsive with 100-200 sessions and make the `pi-trajectory-flush` cleanup timeout configurable or adaptive.

## Best Possible Solution

Land a focused core fix that gives default `sessions.list {}` a read-only or summary fast path where safe and threads a bounded configurable trajectory-flush cleanup timeout through the runtime contract, with regression coverage and docs/changelog updates.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is directly reproducible from current code by making `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has production logs and matching hot-path code, but still needs a local synthetic timing proof.

## Solution Assessment

Yes. The narrow maintainable direction is to optimize the current read-only listing path and expose a cleanup timeout knob; a metadata index or persistent backend can remain future work unless measurements show the focused fix is insufficient.
