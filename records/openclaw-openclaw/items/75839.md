## Summary

Keep open. Current main has partial session-list performance mitigations, but it still contains the reported fixed 10000 ms cleanup timeout and enough synchronous/session-store work in the list path that the reported Raspberry Pi latency is plausible and not superseded by an existing fix.

## What This Changes

The issue reports slow `sessions.list` responses under moderate persisted-session load and fixed 10 second `pi-trajectory-flush` cleanup timeouts on a Docker/Raspberry Pi deployment.

## Best Possible Solution

Add a focused core fix that makes `sessions.list` avoid unnecessary full-store cloning/parsing and synchronous transcript work for dashboard-style reads, while adding a bounded configurable cleanup timeout for trajectory flushing with tests and user-facing docs/changelog.

## Reproduction Assessment

Partially yes. The exact `pi-trajectory-flush` timeout is reproducible from current code by making `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has concrete operator logs and a plausible code path, but needs a synthetic 100 to 200 session fixture or live Pi proof for high-confidence timing.

## Solution Assessment

The current implementation is not the best end state because cleanup timeout tuning is unavailable and list performance still depends on full store parsing/cloning plus per-row synchronous metadata work. A narrow measured optimization and bounded timeout knob is safer than a broad session-store redesign.
