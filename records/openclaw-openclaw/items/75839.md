## Summary

Keep open. Current main has useful partial mitigations and an open closing PR #77133, but the default list path still starts by loading combined session stores and `pi-trajectory-flush` still inherits the 10000 ms cleanup timeout without a config-backed override.

## What This Changes

The issue requests faster `sessions.list` behavior under 100-200 session stores and a configurable or adaptive `pi-trajectory-flush` cleanup timeout.

## Best Possible Solution

Keep this issue open until #77133 or a replacement lands, then finish any remaining measured `sessions.list` cold-path/indexing work and route transcript-loss fallout through #76990.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is source-reproducible from the 10000 ms default and call site with no override; the `sessions.list` latency has strong production logs and a matching full-store load path, but no synthetic current-main timing proof.

## Solution Assessment

Not yet. Current main’s lightweight rows, default limit, trajectory cap, and queued-writer yielding are good mitigations, but the linked PR or a replacement still needs review for the cleanup-timeout side and any remaining cold-store list latency.
