## Summary

Keep open. Current main includes partial mitigations for lightweight session rows, cache no-clone reads, session maintenance, and Control UI pressure, but the default `sessions.list {}` path still loads and merges full session stores before responding, and `pi-trajectory-flush` still uses the fixed 10000 ms cleanup default with no config-backed override at that call site.

## What This Changes

The issue asks for `sessions.list` to stay responsive on 100-200 session JSON stores and for `pi-trajectory-flush` cleanup timeout to become configurable or adaptive.

## Best Possible Solution

Keep this open and scope a first repair around a fast default `sessions.list {}` path for large JSON stores plus a documented cleanup-timeout policy for trajectory flushing, leaving persistent index/backend redesign as a measured follow-up.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is source-reproducible by forcing `trajectoryRecorder.flush()` past the default cleanup timeout; the `sessions.list` latency has strong production logs and matching full-store hot-path code, but no high-confidence synthetic timing run was established here.

## Solution Assessment

Unclear as written. The report's direction is sound, but maintainers should first choose a narrow default-list and cleanup-timeout scope before committing to broader session metadata indexing or a persistent backend.
