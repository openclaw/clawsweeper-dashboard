## Summary

Keep open. Current main has meaningful `sessions.list` and trajectory-write mitigations, but the issue is still paired with open PR #77133 via closing syntax, and current source still leaves `pi-trajectory-flush` on the default 10000 ms cleanup timeout with no config-backed override at the call site.

## What This Changes

The issue asks for `sessions.list` to stay responsive around 100-200 sessions and for `pi-trajectory-flush` cleanup timeout behavior to become configurable or adaptive.

## Best Possible Solution

Review and land or revise the linked trajectory-flush PR, then measure current `sessions.list {}` against a representative multi-agent store and open a narrower follow-up only if the remaining full-store load path still exceeds the Gateway budget.

## Reproduction Assessment

Partially yes. The fixed 10000 ms `pi-trajectory-flush` timeout is source-reproducible on current main, while the exact `sessions.list` latency still relies on the reporter and follow-up production logs rather than a local timing harness.

## Solution Assessment

The best current path is to keep this issue open while PR #77133 is reviewed because it is the active implementation candidate. Current main's lightweight list rows and trajectory write caps are useful partial fixes, but they do not yet prove the whole reported behavior is resolved.
