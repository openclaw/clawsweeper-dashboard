## Summary

Keep open. Current main contains several useful mitigations, but the source still matches the two reported remaining paths: default `sessions.list {}` builds lightweight rows only after loading and merging the full session stores, and `pi-trajectory-flush` still inherits the shared 10000 ms cleanup timeout without a config-backed override.

## What This Changes

The issue asks OpenClaw to keep `sessions.list` responsive around 100-200 sessions and make `pi-trajectory-flush` cleanup timeout configurable or adaptive.

## Best Possible Solution

Land a focused Gateway/session and agent-cleanup fix that makes default `sessions.list {}` list-safe on large JSON stores and exposes a documented trajectory-flush cleanup timeout, while coordinating broader indexing or trajectory-retention redesign through the related open reports.

## Reproduction Assessment

Partially yes. The 10000 ms `pi-trajectory-flush` timeout is source-reproducible by making `trajectoryRecorder.flush()` exceed the shared cleanup default; the `sessions.list` latency has production logs and matching full-store hot-path code, but no synthetic timing repro was run here.

## Solution Assessment

Yes for a narrow first repair, with one product caveat. Optimizing the existing list/cache path and adding a config-backed cleanup timeout is more maintainable than replacing the session backend immediately, but the timeout contract needs maintainer sign-off.
