## Summary

Keep open. Current main still loads and merges the full session store before `sessions.list` can answer, cache hits still deep-clone the store, and `pi-trajectory-flush` still uses the fixed 10000 ms cleanup fallback with no caller override.

## What This Changes

The issue requests lower latency for default dashboard-style `sessions.list` calls under 100-200 sessions and a configurable or adaptive timeout for `pi-trajectory-flush`.

## Best Possible Solution

Land a focused core fix that gives default `sessions.list {}` a read-only summary path that avoids unnecessary full-store cloning/row hydration where safe, and threads a bounded configurable trajectory-flush cleanup timeout through the runtime/config contract; keep the larger persistent store redesign in #55334 separate.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is source-reproducible by making `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has concrete production logs and matching hot-path code, but still needs a local synthetic timing proof.

## Solution Assessment

Yes for a focused fast-path and timeout-configuration repair. A persistent metadata index or SQLite-style backend may be the long-term direction, but it is broader than the narrow fix needed to address this report safely.
