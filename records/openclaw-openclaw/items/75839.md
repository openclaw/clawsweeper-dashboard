## Summary

Keep open. Current main has useful partial mitigations for Control UI defaults, session-store cache cloning, session write-lock timeout, and session cleanup, but the reported core surfaces still remain: default `sessions.list` loads and merges full stores before responding, and `pi-trajectory-flush` still inherits the fixed 10000 ms cleanup timeout with no config override.

## What This Changes

The issue asks for responsive `sessions.list` behavior under 100-200 sessions, better built-in session/trajectory lifecycle maintenance, and a configurable or adaptive trajectory-flush cleanup timeout.

## Best Possible Solution

Land a focused core/runtime fix that makes default `sessions.list {}` list-safe and responsive on large JSON stores, exposes a documented trajectory-flush cleanup timeout, and extends session lifecycle maintenance conservatively with tests and docs.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is directly reproducible from current code by making `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has strong production logs and matching hot-path code, but still needs a synthetic timing proof on current main.

## Solution Assessment

Yes for a focused first repair. Optimizing the existing list/cache path and adding a config-backed cleanup timeout is narrower and more maintainable than jumping straight to a new persistent backend, while broader metadata-index or SQLite work can remain future follow-up if measurements require it.
