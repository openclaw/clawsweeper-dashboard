## Summary

Keep open. Current main has useful partial mitigations for Control UI defaults, transcript hydration, write-lock waits, and disk-budget cleanup, but the reported core surfaces are not fully closed: default `sessions.list {}` still loads and merges session stores before responding, and `pi-trajectory-flush` still inherits the fixed 10000 ms cleanup timeout with no config override.

## What This Changes

The issue requests lower-latency session listing under moderate stored-session load and a configurable or adaptive trajectory flush cleanup timeout.

## Best Possible Solution

Keep the issue open until core has a measured sessions.list fast path for default dashboard polling and a documented, bounded timeout policy for trajectory flush cleanup.

## Reproduction Assessment

Partially yes. The cleanup timeout is directly reproducible from current code by letting trajectoryRecorder.flush() exceed runAgentCleanupStep's 10000 ms default; the sessions.list latency has production logs plus matching source hot paths, but still needs a synthetic 100-200 session timing regression proof on current main.

## Solution Assessment

No, current main is not yet the complete solution. The landed fixes are the right direction, but the remaining narrow path is to thread a cleanup-timeout knob into pi-trajectory-flush and prove or further reduce default sessions.list store-load/cache costs without a larger backend redesign.
