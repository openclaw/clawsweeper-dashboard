## Summary

Keep open. Current main has partial `sessions.list` performance work, but the reported `pi-trajectory-flush` path still falls back to the fixed 10000 ms cleanup timeout, so the issue is not fully implemented or superseded.

## What This Changes

The issue reports slow `sessions.list` responses under moderate session-store load and a fixed 10000 ms `pi-trajectory-flush` cleanup timeout on long-running Raspberry Pi Docker deployments.

## Best Possible Solution

Keep the current session-list/cache improvements, add a bounded configurable timeout for trajectory-flush cleanup, and treat deeper metadata-index or backend changes as a separate follow-up if measured Pi workloads remain slow.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is directly reproducible from current code by letting `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has strong operator logs plus matching hot-path code, but no live Pi timing proof in this read-only pass.

## Solution Assessment

Yes for the remaining narrow repair. A focused config-backed timeout for cleanup steps is safer than changing unrelated session retention behavior, while the broader list-performance architecture should be driven by measurements after the current main optimizations.
