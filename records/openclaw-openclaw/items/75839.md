## Summary

Keep open. Current main still has both reported paths: `sessions.list` synchronously loads and merges full session stores before responding, and `pi-trajectory-flush` still uses the default 10000 ms cleanup timeout with no caller override.

## What This Changes

The issue requests better `sessions.list` responsiveness under roughly 100 to 200 sessions and a configurable or adaptive timeout for `pi-trajectory-flush`.

## Best Possible Solution

Add a focused core improvement that avoids unnecessary full-store clone/merge overhead for read-only dashboard-style `sessions.list {}` calls, plus a bounded configurable cleanup timeout for trajectory flushing, with regression coverage and user-facing docs/changelog updates.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is directly reproducible from current code by making `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has concrete operator logs and matching hot-path code, but still needs a synthetic timing proof for high-confidence performance reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is to optimize the read-only list path and expose a cleanup timeout knob; a full metadata-index or persistent backend redesign can remain future work unless measurements show the focused fix is insufficient.
