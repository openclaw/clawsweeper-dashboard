## Summary

Keep open. Current main still has the reported fixed 10000 ms `pi-trajectory-flush` timeout and the `sessions.list` path still loads/merges full session stores before responding, so the reporter's Raspberry Pi latency remains plausible and not superseded by another tracked fix.

## What This Changes

The issue asks for `sessions.list` to stay responsive with roughly 100 to 200 sessions and for the `pi-trajectory-flush` cleanup timeout to become configurable or adaptive.

## Best Possible Solution

Add a focused core improvement that reduces avoidable full-store clone/merge overhead for dashboard-style `sessions.list {}` while preserving write-path isolation, and add a bounded configurable timeout for trajectory flushing with tests, docs, and a changelog entry.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is reproducible from current code by making `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has concrete operator logs and matching current code paths, but still needs a synthetic 100 to 200 session timing proof for high-confidence performance reproduction.

## Solution Assessment

Yes, the best near-term fix is a narrow measured optimization plus a configurable cleanup timeout. A broader session-store backend redesign may be useful later, but it is not required to address the reported stalls safely.

Fix https://github.com/openclaw/openclaw/issues/75839 with a focused core PR. Address two concrete symptoms: default `sessions.list {}` remains slow under roughly 100 to 200 sessions on lower-power hosts, and `pi-trajectory-flush` uses an unconfigurable 10000 ms cleanup timeout.
