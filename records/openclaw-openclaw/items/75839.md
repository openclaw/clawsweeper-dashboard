## Summary

Keep open. Current main and v2026.5.2 include useful mitigations for Control UI defaults, session cache reuse, and write-lock timeout policy, but the two central remaining surfaces are still present: default `sessions.list` can load and merge every session store on a cache miss, and `pi-trajectory-flush` still inherits the fixed 10000 ms cleanup timeout without a config-backed override.

## What This Changes

Request to keep `sessions.list` responsive with roughly 100-200 sessions and make the `pi-trajectory-flush` cleanup timeout configurable or adaptive.

## Best Possible Solution

Land a focused runtime/config fix that keeps the existing JSON-store path list-safe on cache misses and wires trajectory cleanup through a documented timeout policy, leaving SQLite or a new persistent backend for a separate design discussion.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is source-reproducible by making `trajectoryRecorder.flush()` exceed the cleanup default, while the `sessions.list` latency has production logs and matching hot-path source but no local timing proof on current main.

## Solution Assessment

Yes for a narrow repair direction. Optimizing the current list/cache path and adding a config-backed cleanup timeout is more maintainable than jumping directly to a new storage backend or broad `sessions.list` mode.
