## Summary

Keep open. Current main has useful session-list and maintenance mitigations, but the remaining reported paths are still present: default `sessions.list {}` still loads/parses/merges the session store before responding, and `pi-trajectory-flush` still falls back to the fixed 10000 ms cleanup timeout.

## What This Changes

The issue requests more responsive session listing under moderate JSON-store load and a configurable or adaptive trajectory-flush cleanup timeout.

## Best Possible Solution

Land a measured core/runtime fix that makes the default session list path list-safe for large JSON stores, adds a documented trajectory-flush cleanup timeout policy, and leaves backend replacement as a separate follow-up if profiling still shows bottlenecks.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is source-reproducible by making `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has strong production logs and matching full-store hot-path code, but no synthetic current-main timing proof was established.

## Solution Assessment

Unclear as written. The suggested direction is reasonable, but the maintainable first step should be scoped to measured list-store hot paths plus a bounded cleanup-timeout setting rather than committing to a full backend migration immediately.
