## Summary

Keep open. Current main has useful partial mitigations for lightweight `sessions.list` rows and trajectory writer yielding, but the issue is still paired with an open closing PR and current source still has the 10000 ms default cleanup timeout at the `pi-trajectory-flush` call site while default `sessions.list {}` still loads combined session stores before row limiting.

## What This Changes

The issue asks for responsive session listing around 100-200 sessions and a configurable or adaptive trajectory-flush cleanup timeout.

## Best Possible Solution

Review and land or adjust the linked trajectory-flush PR, then keep a focused follow-up for the remaining default `sessions.list` store-load/index path before closing this issue.

## Reproduction Assessment

Partially yes. The fixed cleanup timeout is source-reproducible from `runAgentCleanupStep` and the `pi-trajectory-flush` call site; the session-list latency has strong production logs and matching full-store source paths, but I did not establish a synthetic timing reproduction on current main.

## Solution Assessment

No complete solution is on main yet. The current mitigations are useful, but closing on them or on the still-open trajectory PR would leave the combined-store load path and timeout configuration question unresolved.
