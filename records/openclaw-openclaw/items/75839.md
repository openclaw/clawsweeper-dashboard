## Summary

Keep open. Current main has useful partial mitigations for session-list defaults, cache reuse, write-lock waits, and session cleanup, but the default `sessions.list` path still loads and merges full session stores before responding, and `pi-trajectory-flush` still uses the shared fixed 10000 ms cleanup timeout without a config override.

## What This Changes

The issue asks for responsive `sessions.list` behavior under 100-200 sessions, stronger session/trajectory lifecycle maintenance, and a configurable or adaptive trajectory-flush cleanup timeout.

## Best Possible Solution

Land a focused runtime/config fix that keeps the JSON store compatible, reduces default list work on cache hits and misses, documents a trajectory-flush cleanup timeout policy, and leaves larger metadata-index or SQLite work for measured follow-up.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is source-reproducible by making `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has strong reporter data and matching hot-path code but still needs timing proof on current main.

## Solution Assessment

Yes for a focused first repair. Optimizing the existing list/cache path and threading a documented cleanup-timeout config is narrower than replacing the session backend, while broader indexing can follow if measurements still justify it.
