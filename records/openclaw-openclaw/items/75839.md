## Summary

Keep open. Current main still exposes both reported paths: `sessions.list` builds its response after loading the combined session store, and `pi-trajectory-flush` still uses the fixed cleanup timeout when no caller override is passed. The reporter’s follow-up adds useful live Docker-on-ARM evidence, and the related title-search items do not supersede this narrower performance/timeout report.

## What This Changes

The issue requests faster `sessions.list` behavior for roughly 100-200 sessions and a configurable or adaptive `pi-trajectory-flush` cleanup timeout.

## Best Possible Solution

Add a focused core fix that reduces avoidable full-store clone/merge overhead for read-only dashboard-style `sessions.list {}` while preserving store mutation isolation, and add a bounded configurable timeout for trajectory flushing with tests, docs, and a changelog entry.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is reproducible directly from current code by letting `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has concrete operator logs and matching current code paths, but still needs a synthetic timing proof for high-confidence reproduction of the exact 10-16s Raspberry Pi behavior.

## Solution Assessment

Yes. A narrow measured optimization plus a configurable cleanup timeout is the best near-term path; a full metadata index or persistent backend redesign may be useful later but is broader than needed for this report.
