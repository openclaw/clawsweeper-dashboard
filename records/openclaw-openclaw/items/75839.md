## Summary

Keep open. Current main has useful partial mitigations, but the reported remaining paths still exist: default `sessions.list {}` still loads and merges full session stores before responding, and `pi-trajectory-flush` still inherits the fixed 10000 ms cleanup timeout with no config-backed override.

## What This Changes

The issue requests responsive `sessions.list` behavior with roughly 100-200 sessions and a configurable or adaptive `pi-trajectory-flush` cleanup timeout for long-running Docker/ARM deployments.

## Best Possible Solution

Land a focused core/runtime fix that makes default `sessions.list {}` list-safe on large JSON stores, exposes a documented trajectory-flush cleanup timeout, and leaves broader persistent-index or backend redesign for a measured follow-up.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is source-reproducible by making `trajectoryRecorder.flush()` exceed the default cleanup timeout; the `sessions.list` latency has strong production logs and matching full-store hot-path code, but no synthetic current-main timing proof was established.

## Solution Assessment

Yes for a focused first repair. Optimizing the existing list/cache path and adding a config-backed cleanup timeout is narrower than replacing the session backend, while broader metadata indexing can remain a measured follow-up if needed.
