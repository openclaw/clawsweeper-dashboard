## Summary

Keep open. Current main has useful partial mitigations for bounded/lightweight `sessions.list` rows and trajectory capture, but source still shows default `sessions.list {}` loading and merging session stores before row limiting, and `pi-trajectory-flush` still inherits the fixed 10000 ms cleanup timeout without a call-site or config override.

## What This Changes

The issue asks for `sessions.list` to stay responsive around 100-200 sessions and for `pi-trajectory-flush` cleanup timeouts to become configurable or adaptive under moderate load.

## Best Possible Solution

Keep the issue open until #77133 is reviewed or adjusted for the trajectory-flush side and a measured session-list fix proves cold/default `sessions.list {}` avoids full-store stalls on large stores.

## Reproduction Assessment

Partially yes. The fixed `pi-trajectory-flush` timeout is source-reproducible from `AGENT_CLEANUP_STEP_TIMEOUT_MS` plus the no-override call site, while the exact `sessions.list` latency still relies on production reports plus the current full-store load/parse path rather than a fresh timing run.

## Solution Assessment

No, not fully yet. Current main and #77133 address important trajectory/event-loop pieces, but the remaining maintainable solution also needs a measured default `sessions.list` hot-path fix and an explicit cleanup-timeout policy.
