## Summary

Keep this PR open. The current diff still conflicts with OpenClaw's shipped DeepSeek V4 plain-assistant replay behavior and existing SDK/plugin/live coverage, while the claimed older-endpoint 400 for passing plain `reasoning_content` has not been independently reproduced in the discussion.

## What This Changes

The PR changes the DeepSeek V4 OpenAI-compatible thinking wrapper so enabled-thinking payloads preserve or backfill `reasoning_content` only on assistant messages with `tool_calls` and strip it from other assistant messages.

## Best Possible Solution

Keep the enabled-thinking DeepSeek V4 replay path preserving or backfilling plain and tool-call assistant `reasoning_content` unless maintainers obtain live endpoint proof that plain replay must be stripped. If that split is real, implement it with updated SDK, plugin, and live fixtures plus a changelog entry while preserving the #71372 and #73417 tool-call/plain replay fixes.

## Reproduction Assessment

Yes for the regression introduced by this PR: existing SDK, DeepSeek plugin, and live fixtures assert blank `reasoning_content` on plain assistant replay, so the proposed helper would fail that coverage. No high-confidence reproduction is provided for the PR's older-endpoint 400 claim when plain `reasoning_content` is passed.

## Solution Assessment

No. The proposed helper preserves tool-call replay correctly, but stripping plain assistant `reasoning_content` is broader than the current OpenClaw evidence supports; the safer path is to keep the shipped helper unless live provider proof justifies a narrower conditional rule.
