## Summary

Keep open; the PR still conflicts with the shipped DeepSeek V4 replay behavior by stripping plain assistant `reasoning_content`, and the asserted older-endpoint failure is not backed by a reproducible endpoint/version while current tests and live fixtures cover the opposite payload shape.

## What This Changes

The PR changes the enabled DeepSeek V4 thinking wrapper to preserve or backfill `reasoning_content` only for assistant messages with `tool_calls` and remove it from plain assistant replay messages.

## Best Possible Solution

Keep the current enabled DeepSeek V4 backfill for all assistant replay messages unless maintainers can reproduce an endpoint-specific rejection; any accepted rule change should update source, tests, docs, and changelog together.

## Reproduction Assessment

Yes for the PR regression: current SDK, DeepSeek plugin, and live fixtures all assert or accept plain assistant replay with `reasoning_content: ""`. No for the claimed older-endpoint 400 because the discussion lacks endpoint/version/live proof and the official docs say the field is ignored when passed for non-tool turns.

## Solution Assessment

No. Stripping non-tool assistant replay is broader than the evidence supports and contradicts the shipped #73417 contract; the safer path is to keep main's helper until live provider proof justifies a narrower conditional rule.
