## Summary

Keep open, but this patch is not mergeable as written: it reverses the shipped DeepSeek V4 plain-assistant replay fix, and the asserted older-endpoint rejection is not backed by live proof in the PR discussion.

## What This Changes

The PR changes the DeepSeek V4 enabled-thinking wrapper to strip `reasoning_content` from non-tool-call assistant messages while preserving or backfilling it for assistant messages with tool calls.

## Best Possible Solution

Keep enabled DeepSeek V4 replay preserving or backfilling `reasoning_content` on all assistant messages unless maintainers get live provider proof for an endpoint-specific exception; any exception should be narrowly gated and update tests and docs together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the regression introduced by this patch: current main source and tests assert that plain assistant replay carries `reasoning_content: ""`; no high-confidence reproduction exists for the PR's claimed older-endpoint 400.

## Solution Assessment

Is this the best way to solve the issue? No. Stripping plain assistant replay is broader than the evidence supports and conflicts with the shipped DeepSeek V4 replay contract; the safer path is provider proof first, then a narrowly gated rule if needed.
