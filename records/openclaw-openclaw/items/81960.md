## Summary

Keep open. Current main still runs interactive provider/model onboarding as a single provider pass, and this fresh feature request needs maintainer UX/product scoping rather than conservative cleanup closure.

## What This Changes

Request to let interactive `openclaw onboard` configure multiple providers/models in one run, choose primary/fallback or role-specific models, persist them, and show a final summary.

## Best Possible Solution

Define a scoped repeat-provider onboarding stage that reuses existing provider auth, model allowlist, fallback, and role-specific config contracts while keeping explicit `--auth-choice` and non-interactive flows single-pass.

## Reproduction Assessment

Not applicable as a bug; this is a feature request. Source inspection gives high confidence that current interactive onboarding exits after one provider/model setup, confirming the requested feature gap.

## Solution Assessment

Unclear until maintainers scope the UX. A repeat prompt is a plausible small slice, but default/fallback/vision/embedding role assignment should reuse existing config and allowlist helpers rather than adding ad hoc state.
