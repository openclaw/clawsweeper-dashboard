## Summary

Keep open. Current `main` still treats onboarding model auth as a single provider/default-model pass, while the requested repeat-provider flow and role assignment would be a new onboarding UX/product capability rather than a broken existing contract.

## What This Changes

The issue requests expanding `openclaw onboard` so one run can configure multiple providers/models, choose defaults, optionally assign roles, persist them, and show a final summary.

## Best Possible Solution

Design a bounded onboarding provider stage that can repeat auth/custom-provider setup, then map selected model refs into existing primary, fallback, and role-specific config fields with a clear final summary.

## Reproduction Assessment

Not applicable: this is a feature request. Source inspection is enough to confirm current `openclaw onboard` does not provide the requested repeat-provider setup flow.

## Solution Assessment

Yes, the requested direction is plausible, but it should reuse the existing model config and model-picker/fallback machinery rather than adding an ad hoc parallel config surface.
