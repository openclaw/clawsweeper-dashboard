## Summary

Keep open. Current main still has a one-pass provider/default-model onboarding path, and the requested repeat-provider flow plus default/fallback/role handling is a new onboarding UX capability that needs maintainer scoping rather than cleanup closure.

## What This Changes

Allow interactive `openclaw onboard` to configure several providers/models in one run, choose a primary/default model, optionally assign model roles, and show a final summary.

## Best Possible Solution

Design a bounded interactive onboarding provider stage that can repeat existing auth/custom-provider setup, then map selected model refs into existing primary, fallback, and role-specific config fields with focused tests and docs.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing contract. Source inspection confirms current interactive onboarding exits after one provider/default-model path and has no repeat-provider prompt.

## Solution Assessment

Unclear until maintainer scope is chosen. The requested direction is plausible, but the maintainable path should reuse existing model picker, primary-model, fallback, and agent-default config contracts instead of adding a parallel config surface.
