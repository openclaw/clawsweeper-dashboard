## Label Justifications

- `P1`: A completed async media task can leave the originating chat without the artifact and present the result as a failed completion path.
- `impact:message-loss`: The core symptom is a valid generated media artifact not being delivered back to the original Slack/channel requester.
- `impact:session-state`: The failure depends on requester-session wake/settle state diverging from the underlying media task success.

## Summary

Keep open. Current main still converts a successful media-generation run into a failed task/receipt when completion delivery cannot be confirmed, and the existing generated-media direct fallback only covers inactive requester sessions or missing media after a completion-agent response.

## What This Changes

The issue asks OpenClaw to treat generated media artifacts as generation success even when the follow-up completion delivery to the original Slack/session times out.

## Best Possible Solution

Preserve the media-generation success state, surface completion delivery failure separately, and use the existing idempotent generated-media delivery path for missing artifacts without rerunning provider generation.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: stub a successful media job whose `wakeTaskCompletion` returns false and current main follows the success path into `failTaskRun`. I did not execute a live Slack run in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to split generation success from delivery failure and reuse the generated-media idempotent fallback for missing artifacts. A new broad retry UX should not be required for the first repair.
