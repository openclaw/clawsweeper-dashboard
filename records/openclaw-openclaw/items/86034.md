## Summary

Keep open: current main and v2026.5.22 still have a source-backed path where successful media generation becomes a failed task when active requester completion delivery cannot be confirmed. The focused remaining work appears to be owned by the open follow-up at https://github.com/openclaw/openclaw/pull/85489, so this issue should stay open until that path is reviewed, landed, or deliberately redirected.

## What This Changes

The issue asks OpenClaw to treat a completed `image_generate` artifact as generation success even when completion delivery back to the original chat/session fails, preserving artifact references and a retryable delivery path.

## Best Possible Solution

Review and land one canonical generated-media completion fix that separates generation success from delivery outcome, enables idempotent missing-media fallback after failed active wakes, and records retryable delivery failure without rerunning generation.

## Reproduction Assessment

Yes, at source level: a successful media `run()` followed by `wakeTaskCompletion` returning false currently fails the task and emits an error completion result. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

Mostly yes: the focused active-wake fallback in https://github.com/openclaw/openclaw/pull/85489 is the narrowest implementation path I found, but maintainers still need to choose the exact generation-success versus delivery-failure contract.
