## Label Justifications

- `P2`: The report describes a real user-visible media completion failure with bounded blast radius, not an emergency or core-runtime outage.
- `impact:message-loss`: The generated media/completion can fail to reach the originating chat even though the artifact exists.
- `impact:session-state`: The failure centers on requester-session wake/settle behavior after the media job has completed.

## Summary

Keep open: current main still has a source-backed path where successful media generation is converted into a task failure when completion delivery cannot be confirmed. Main has useful direct-fallback coverage for inactive/missed generated media, but it does not fully solve the reported active requester wake/delivery timeout case.

## What This Changes

The issue requests that successful `image_generate` artifacts be reported separately from final completion-delivery failures, with artifact-preserving messaging and a retryable delivery path.

## Best Possible Solution

Split media generation result state from delivery outcome: complete the generation with artifact references, record delivery failure separately, and provide an idempotent delivery retry or fallback that does not rerun generation.

## Reproduction Assessment

Yes at source level: a successful media `run()` followed by `wakeTaskCompletion` returning false currently fails the task and emits an error completion result. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

The requested direction is right, but the exact fix needs maintainer judgment because the current code intentionally treats completion delivery as required and avoids direct fallback while the requester session is active.
