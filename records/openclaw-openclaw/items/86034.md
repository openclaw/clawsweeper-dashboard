## Summary

Keep open: current main still has a source-backed path where successful media generation is turned into a failed media task when completion delivery cannot be confirmed. The merged generated-media fallback work in https://github.com/openclaw/openclaw/pull/84006 covers important inactive/missed-media cases, but the active requester wake or handoff timeout described here remains unresolved.

## What This Changes

The issue asks OpenClaw to separate media generation success from chat/session delivery failure, preserve the generated artifact reference, and provide a retryable delivery path without rerunning generation.

## Best Possible Solution

Define one shared completion contract where generated artifacts are recorded as successful independently from delivery, delivery failures are surfaced as retryable handoff failures with preserved media references, and any direct fallback remains idempotent across active and inactive sessions.

## Reproduction Assessment

Yes, at source level: a successful media `run()` followed by `wakeTaskCompletion` returning false currently fails the task and emits an error completion result. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

The requested direction is right, but the exact fix needs maintainer judgment because retryable delivery and active-session direct fallback semantics can change visible message delivery behavior.
