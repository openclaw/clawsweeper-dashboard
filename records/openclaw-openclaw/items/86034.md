## Label Justifications

- `P2`: This is a normal-priority but real media/channel reliability bug with limited scope and source-backed behavior.
- `impact:message-loss`: The user can miss the generated-media completion message even though the artifact exists.
- `impact:session-state`: The completion wake/settle path can leave the original requester session looking failed or stalled after a successful media run.

## Summary

Keep open: current main and the latest release still have a source-backed path where a successful media generation is converted into task failure when completion delivery cannot be confirmed. The existing inactive/missed-media fallback does not fully cover the reported active requester wake timeout case, and the open async-start PR explicitly does not claim completion delivery.

## What This Changes

The issue asks OpenClaw to distinguish generated-artifact success from completion-delivery failure, preserve the artifact reference, and provide a retryable delivery path without rerunning media generation.

## Best Possible Solution

Define a shared generated-media completion contract that records provider success and artifact paths separately from delivery status, surfaces a distinct retryable delivery failure, and covers active requester wake timeouts with idempotent regression tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.22 turn a false/throwing successful media completion delivery into task failure, and tests assert that branch. I did not run a live Slack/media provider reproduction in this read-only review.

## Solution Assessment

Partly: separating generation success from delivery failure is the right direction, but maintainers still need to choose the exact task-state and retry-delivery contract before an automated fix is safe.
