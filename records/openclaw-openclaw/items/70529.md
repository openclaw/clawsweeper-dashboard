## Summary

Keep open. Current main and the latest release still default the built-in `user` profile to Chrome MCP through `npx`, matching the packaged Desktop `spawn npx ENOENT` path, and the discussion explicitly keeps this Desktop packaging variant separate from the broader attach-diagnostics and login-handoff trackers.

## What This Changes

The issue reports that packaged Desktop cannot use existing Chrome sessions because EasyClaw OAuth fails and the built-in `user` browser profile tries to spawn `npx`, which is absent from the Desktop bundle.

## Best Possible Solution

Make packaged Desktop handle the built-in `user` profile without relying on ambient `npx`, either through an approved Chrome MCP runner resolution path or a structured Desktop diagnostic that points to the managed `openclaw` profile and documented override.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.12` still resolve the built-in `user` profile to Chrome MCP, default that launch to `npx -y chrome-devtools-mcp@latest`, and hand that command to stdio spawn, while the issue supplies the packaged Desktop environment where `npx` is absent.

## Solution Assessment

Unclear because there is no patch here. The existing `mcpCommand` override is a useful workaround, but the maintainable fix depends on the Desktop packaging and Chrome MCP launch policy maintainers choose.
