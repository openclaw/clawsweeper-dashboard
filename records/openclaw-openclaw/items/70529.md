## Summary

Keep open. Current main removed the legacy EasyClaw extension relay path, but the remaining Desktop `user` existing-session failure is still present: the built-in profile still resolves to Chrome MCP, the default runner is still `npx -y chrome-devtools-mcp@latest`, and the MCP stdio transport spawns that executable directly. Maintainer context also scoped this as separate from the broader browser attach and login handoff reports.

## What This Changes

The issue asks OpenClaw Desktop to support existing signed-in Chrome sessions without EasyClaw OAuth failure or an ambient `npx` dependency for the built-in `user` profile.

## Best Possible Solution

Keep the removed EasyClaw extension relay retired, and make the packaged Desktop/browser runtime handle the built-in `user` profile without relying on ambient `npx`: either resolve a pinned/bundled Chrome MCP runner through the app/package boundary, or fail early with a structured diagnostic that points to the managed `openclaw` profile and an explicit `mcpCommand` override. Update the browser docs, config schema/help, and Chrome MCP tests with the chosen contract.

## Reproduction Assessment

Yes for the remaining Desktop `user` profile failure: current main defaults the built-in profile to existing-session, builds `npx -y chrome-devtools-mcp@latest` args, and the MCP transport spawns that command directly. The EasyClaw relay half is obsolete on current main because the extension relay lifecycle is now a no-op.

## Solution Assessment

Unclear because there is no patch here and the narrowest maintainable fix depends on a maintainer packaging strategy. The existing `mcpCommand` override is a useful workaround, but it is not a complete solution for a built-in Desktop profile that users expect to work without installing `npx` separately.
