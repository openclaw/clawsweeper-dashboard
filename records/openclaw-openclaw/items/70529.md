## Summary

Keep open. Current main still leaves the packaged Desktop `user` browser profile on the Chrome MCP existing-session path that defaults to spawning `npx -y chrome-devtools-mcp@latest`; the EasyClaw relay half is obsolete on main, but the Desktop-specific `spawn npx ENOENT` failure remains actionable and was explicitly scoped by maintainer discussion as separate from the broader browser attach and login-handoff reports.

## What This Changes

The issue requests that packaged Desktop users be able to attach the built-in browser `user` profile to an existing Chrome session without failing when ambient `npx` is unavailable.

## Best Possible Solution

Keep the EasyClaw relay retired, and make the packaged Desktop/browser runtime handle the built-in `user` profile without relying on ambient `npx`, either by resolving a pinned or bundled Chrome MCP runner through the app boundary or by failing early with a structured Desktop diagnostic and documented `mcpCommand` workaround.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level reproduction for the remaining failure: resolve the built-in `user` profile, observe the default Chrome MCP command is `npx -y chrome-devtools-mcp@latest`, and follow the stdio launch path that spawns that executable directly in a Desktop bundle reported to lack `npx`.

## Solution Assessment

Unclear because there is no patch here and the best fix depends on maintainer packaging policy. The existing `mcpCommand` override is a useful workaround, but it is not a complete solution for a built-in Desktop profile that users expect to work without installing `npx` separately.
