## Summary

Keep open. Current main still routes Desktop's built-in `user` browser profile through the Chrome MCP existing-session path that defaults to spawning `npx -y chrome-devtools-mcp@latest`; the discussion explicitly preserves this as the Desktop packaged-app failure rather than a duplicate of broader browser attach or login handoff work.

## What This Changes

The issue asks Desktop to attach the built-in `user` browser profile to an existing Chrome session instead of failing when the packaged app lacks `npx`, while noting the retired EasyClaw relay sign-in failure.

## Best Possible Solution

Make packaged Desktop handle the built-in `user` profile without relying on ambient `npx`, either through an approved Chrome MCP runner resolution path or a structured Desktop diagnostic that points to the managed `openclaw` profile and documented override.

## Reproduction Assessment

Yes, at source level. Current main resolves the built-in `user` profile to Chrome MCP existing-session, defaults that launch to `npx -y chrome-devtools-mcp@latest`, and passes it to the MCP stdio transport; the issue supplies the packaged Desktop environment where `npx` is absent.

## Solution Assessment

Unclear because there is no patch here. The existing `mcpCommand` override is a workaround, but the maintainable product fix depends on Desktop packaging and Chrome MCP launch policy.
