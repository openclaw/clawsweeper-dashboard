## Summary

Keep open. Current main still defaults the built-in Desktop `user` browser profile to the Chrome MCP existing-session path that spawns `npx`, and the maintainer discussion explicitly keeps this as the Desktop-packaging variant rather than folding it into the broader attach or login-handoff issues.

## What This Changes

The issue asks OpenClaw Desktop to attach to existing Chrome sessions without failing when the packaged app lacks `npx`, while recognizing the old EasyClaw relay OAuth path is now obsolete.

## Best Possible Solution

Keep the EasyClaw relay retired and make packaged Desktop handle `profile="user"` without relying on ambient `npx`, either by resolving a supported Chrome MCP runner through the app boundary or by failing early with a structured diagnostic and documented override.

## Reproduction Assessment

Yes, at source level. Current main resolves the built-in `user` profile to existing-session, defaults Chrome MCP to `npx -y chrome-devtools-mcp@latest`, and passes that command to stdio; the issue supplies the packaged Desktop environment where `npx` is absent.

## Solution Assessment

Unclear because there is no patch here. The existing `mcpCommand` override is a useful workaround, but the maintainable product fix depends on Desktop packaging and Chrome MCP launch policy.
