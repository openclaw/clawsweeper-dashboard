## Label Justifications

- `P1`: The documented `profile="user"` browser workflow remains blocked for packaged Desktop users when `npx` is absent.
- `impact:session-state`: The issue is about reusing existing browser tabs, cookies, and logged-in session state through the built-in `user` profile.

## Summary

Keep open. Current main and v2026.5.18 still route the built-in `user` profile through Chrome MCP with a default `npx` launch, and the issue discussion explicitly scopes this as a Desktop-packaging failure separate from the broader attach-diagnostics and login-handoff trackers.

## What This Changes

The issue asks Desktop/browser existing-session attach to work without failing in a packaged app that has `node` but no `npx`, while also noting a separate EasyClaw OAuth callback failure.

## Best Possible Solution

Make packaged Desktop handle the built-in `user` profile without relying on ambient `npx`, either through an approved Chrome MCP runner resolution path or a structured Desktop diagnostic that points to the managed `openclaw` profile and documented `mcpCommand` override; treat the EasyClaw OAuth callback as a separate auth-handoff check.

## Reproduction Assessment

Yes, at source level. Current main resolves the built-in `user` profile to Chrome MCP, defaults the launch to `npx -y chrome-devtools-mcp@latest`, and hands that command to stdio spawn, which matches the reported Desktop bundle missing `npx`.

## Solution Assessment

Unclear because there is no patch here. The existing `mcpCommand` override is a useful workaround, but the durable fix depends on the Desktop packaging and Chrome MCP runner policy maintainers choose.
