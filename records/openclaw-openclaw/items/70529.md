## Summary

Keep open. Current main still routes the built-in `user` browser profile through Chrome MCP with a default `npx -y chrome-devtools-mcp@latest` launch, so a packaged Desktop environment without `npx` would still hit the reported spawn failure; related discussions keep this Desktop packaging variant separate from broader attach diagnostics and login handoff work.

## What This Changes

The issue asks OpenClaw Desktop to support existing Chrome sessions by addressing the EasyClaw relay OAuth failure and the built-in `user` profile path that tries to spawn `npx` in a packaged app environment without `npx`.

## Best Possible Solution

Make packaged Desktop handle the built-in `user` profile without relying on ambient `npx`, either through an approved Chrome MCP runner resolution path or a structured Desktop diagnostic that points to the managed `openclaw` profile and documented override.

## Reproduction Assessment

Yes at source level, but not by a live packaged Desktop run. Current main resolves the built-in `user` profile to Chrome MCP, defaults that launch to `npx -y chrome-devtools-mcp@latest`, and passes it to stdio spawn while the issue supplies the packaged Desktop environment where `npx` is absent.

## Solution Assessment

Unclear because there is no patch here. The existing `mcpCommand` override is a workaround, but the maintainable fix depends on the Desktop packaging and Chrome MCP launch policy maintainers choose.
