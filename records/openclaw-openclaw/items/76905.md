## Summary

Keep open: current main still lacks the foreground freshness guard needed to prevent older buffered final replies from delivering after a newer inbound message, and open PR #76970 is the linked implementation candidate rather than a reason to close before merge.

## What This Changes

A WhatsApp direct-message bug report asks for foreground auto-reply finals to stay bound to the originating inbound turn and be suppressed or reclassified when a newer same-session message arrives.

## Best Possible Solution

Land or replace #76970 with a shared auto-reply freshness guard plus regression coverage, then close this issue through the merged fix.

## Reproduction Assessment

No live WhatsApp/Codex reproduction was established here. Source inspection gives a medium-confidence path: active-run steering accepts newer inbound messages while final delivery lacks an origin freshness check.

## Solution Assessment

Yes, the narrow maintainable direction is a shared buffered-dispatch freshness guard like #76970 proposes. A WhatsApp-only workaround would miss the generic auto-reply delivery surface.
