## Summary

Keep open: current main still has a source-reproducible foreground reply race, and open PR #76970 already tracks the intended fix with closing syntax but has not merged.

## What This Changes

Requests suppressing or reclassifying stale foreground WhatsApp final replies when a newer same-session inbound message arrives before the older tool-backed run finishes.

## Best Possible Solution

Review and land or close PR #76970; the desired end state is a shared auto-reply foreground freshness guard with regression coverage, while keeping provider replay fixes separate.

## Reproduction Assessment

No live high-confidence reproduction path was established in this read-only review. Source inspection does show a credible current-main path where a newer message can steer an active run while the older run still has an unguarded final delivery path.

## Solution Assessment

Yes, directionally: a shared foreground final freshness check in the auto-reply delivery path is the narrow maintainable fix. The linked open PR is the right implementation candidate to review before this issue closes.
