## Summary

Keep open: current main still lacks a shared foreground freshness guard for older final replies, and the linked open PR is an implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue asks OpenClaw to prevent stale tool-backed foreground auto-reply finals from being delivered after a newer WhatsApp direct message arrives for the same chat/session.

## Best Possible Solution

Land or replace #76970 with a shared auto-reply freshness guard plus regression coverage, then let the merged fix close this issue.

## Reproduction Assessment

No live WhatsApp/Codex reproduction was established here. Source inspection gives a medium-confidence path: active-run steering accepts newer same-session messages while final delivery has no origin freshness check before channel-visible final dispatch.

## Solution Assessment

Yes. The narrow maintainable solution is a shared auto-reply buffered-dispatch freshness guard like #76970 proposes; a WhatsApp-only workaround would miss the generic reply delivery surface.
