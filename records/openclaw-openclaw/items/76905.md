## Summary

Keep open: this is a recent, concrete bug report and an open unmerged PR already uses closing syntax for it. Current main still shows active-run steering can accept a newer inbound message while final delivery is routed through the generic dispatcher without a shared origin freshness guard.

## What This Changes

The issue asks OpenClaw to keep foreground WhatsApp auto-reply finals bound to their origin turn and suppress old tool-run completions after newer messages arrive in the same chat.

## Best Possible Solution

Land or replace the linked shared auto-reply freshness guard so superseded foreground finals are suppressed before channel delivery, with regression coverage for same-session newer inbound turns.

## Reproduction Assessment

No, not from live execution in this review. The source path is still reproducible by inspection: newer inbound messages can steer an active run while the older run's final delivery path has no shared freshness check.

## Solution Assessment

Yes, the current proposed direction appears to be the narrowest maintainable fix. A shared foreground auto-reply freshness guard is preferable to a WhatsApp-specific workaround, and the issue should remain open while the linked PR is reviewed or replaced.
