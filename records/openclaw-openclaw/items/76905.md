## Summary

Keep open: current main still sends WhatsApp foreground replies through the shared buffered dispatch path without a stale-origin freshness check, and open PR #76970 is linked with closing syntax so this issue should wait for that PR to merge or be rejected.

## What This Changes

Prevent an older tool-backed WhatsApp foreground reply from being delivered after a newer inbound message in the same chat/session.

## Best Possible Solution

Land or replace #76970 with a shared buffered-dispatch freshness guard and regression coverage for same-session same-target stale finals.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence current-main path: steer mode can accept a newer inbound message while the older run still has a buffered final reply with no freshness check before delivery.

## Solution Assessment

Yes, the best direction is a shared auto-reply freshness guard before visible foreground delivery. A WhatsApp-only workaround would miss the generic buffered dispatch surface used by messaging channels.
