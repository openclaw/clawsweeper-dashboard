## Summary

Keep open: current main still lacks the shared foreground freshness guard for WhatsApp-style buffered replies, and open PR #76970 is the active implementation candidate rather than a reason to close before merge.

## What This Changes

The issue asks OpenClaw to suppress or re-route stale foreground WhatsApp final replies when a newer inbound message arrives in the same chat before the older run finishes.

## Best Possible Solution

Land or replace #76970 with a shared buffered-dispatch freshness guard plus regression coverage, then close this issue through the merged fix.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction from WhatsApp/Codex on current main. Source inspection gives a medium-confidence path: WhatsApp records newer inbound turns while older buffered final delivery lacks a same-session origin freshness check.

## Solution Assessment

Yes, a shared auto-reply freshness guard is the best direction because WhatsApp uses the generic buffered dispatch path. A WhatsApp-only workaround would leave the same foreground stale-finalization class available to other channel integrations.
