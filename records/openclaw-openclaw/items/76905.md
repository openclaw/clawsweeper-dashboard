## Summary

Keep open: the report is plausible against current `main`; the shared auto-reply path still returns or routes an older run's final foreground payload after newer same-session inbound messages are enqueued, and I did not find a freshness guard before delivery.

## What This Changes

Prevent stale foreground WhatsApp direct-message final replies from older tool-backed runs from being delivered after a newer inbound message in the same session.

## Best Possible Solution

Add a shared foreground-reply freshness guard before final auto-reply delivery, keyed by session/chat plus originating inbound message or run sequence, with regression coverage for an active run followed by a newer inbound turn.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The source path is clear enough to reproduce by simulating run A active, message B enqueued in the same WhatsApp/direct session, and run A later returning a final foreground payload without a freshness check.

## Solution Assessment

Yes, the narrowest maintainable fix is in the shared auto-reply finalization/delivery path rather than the WhatsApp adapter. Use the existing message/run origin metadata to suppress stale foreground finals, and keep any explicit background completion behavior separate.
