## Summary

Keep open: current main still lacks a shared WhatsApp foreground-final freshness guard, and open PR #76970 is already the implementation candidate that explicitly closes this issue.

## What This Changes

The issue reports stale WhatsApp foreground final replies from an older tool-backed run being delivered after a newer same-session inbound message.

## Best Possible Solution

Review and land #76970, or an equivalent shared auto-reply freshness guard, with regression coverage for same-session same-target stale final suppression and different-target non-collision.

## Reproduction Assessment

No live reproduction was established here. Source inspection gives a credible current-main path: active-run steering can accept a newer inbound message while the shared WhatsApp final delivery path still lacks an origin freshness check.

## Solution Assessment

Yes. A shared foreground-final freshness guard is the narrowest maintainable fix; a WhatsApp-only workaround would leave the same generic dispatcher race available to other channel surfaces.
