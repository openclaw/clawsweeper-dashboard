## Summary

Keep open: current main still has a credible shared reply-finalization gap where a WhatsApp turn can deliver a visible payload after another inbound message has already advanced the same session, and I did not find an implemented freshness guard for foreground channel replies.

## What This Changes

Prevent older tool-backed WhatsApp direct-message final replies from being delivered as visible answers after a newer inbound message arrives in the same chat.

## Best Possible Solution

Add a generic origin-message freshness guard for automatic foreground channel replies, then cover the WhatsApp DM race with a regression test while leaving explicit message-tool/background delivery paths intact.

## Reproduction Assessment

No high-confidence live reproduction was established. The source path is clear enough to explain the race, but a fix PR should add a deterministic test that records message B before message A's delayed final delivery.

## Solution Assessment

Yes for the bug boundary: the narrow maintainable fix is a shared freshness check before visible automatic delivery, not a WhatsApp route rewrite. If background completion notices are desired, that should be an explicit follow-up behavior rather than the default stale foreground final.

## Repair Work Prompt

Fix the stale foreground WhatsApp reply race reported at https://github.com/openclaw/openclaw/issues/76905. On current main, bind automatic visible channel reply delivery to the inbound message/run that produced it and suppress an older foreground final/block delivery when the same session/chat has recorded a newer inbound user message before delivery. Start in the shared auto-reply/channel turn path if possible, with WhatsApp coverage as the concrete regression. Do not add a user config option, do not change explicit message-tool sends, and do not turn this into a broad queue-mode redesign. Add a regression test that simulates message A starting a delayed/tool-backed run, message B being recorded for the same WhatsApp DM/session before A's final payload is delivered, and verifies A's stale visible final is not sent as the answer to B. Preserve normal delivery when no newer inbound exists. Add a CHANGELOG.md fix entry because this is user-facing channel behavior.
