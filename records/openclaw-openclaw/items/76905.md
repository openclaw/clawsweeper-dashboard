## Summary

Keep open: current main still lacks the shared foreground freshness guard needed for this stale WhatsApp final-reply race, and open PR #76970 is a linked implementation candidate rather than a reason to close before merge.

## What This Changes

The issue reports stale WhatsApp foreground replies from an older tool-backed run being delivered after a newer inbound message in the same chat.

## Best Possible Solution

Land or replace #76970 with a shared auto-reply freshness guard plus regression coverage, then close this issue through the merged fix.

## Reproduction Assessment

No: I do not have a high-confidence live WhatsApp/Codex reproduction from this review. Source inspection gives a medium-confidence path because active-run steering can accept a newer inbound message while the older run's final delivery lacks an origin freshness check.

## Solution Assessment

Yes: the smallest maintainable fix is the shared buffered-dispatch freshness guard already proposed by #76970. A WhatsApp-only patch would leave the generic auto-reply delivery surface exposed for other channels.
