## Summary

Keep open: current main still shows the default steering plus buffered final-delivery path without a shared origin freshness guard, and open PR #76970 is the linked implementation candidate rather than a reason to close this before merge.

## What This Changes

Prevent foreground WhatsApp direct-message final replies from an older tool-backed run from delivering after a newer inbound message in the same chat.

## Best Possible Solution

Review and land or replace #76970 with a shared auto-reply freshness guard plus regression coverage, then close this issue through the merged fix.

## Reproduction Assessment

No, there is not a high-confidence live reproduction from this review. Source inspection gives a medium-confidence path: default steering can accept newer inbound messages while an older final is still queued, and current final delivery has no origin freshness check.

## Solution Assessment

Yes, the best direction is a shared buffered-dispatch freshness guard in the generic auto-reply path. A WhatsApp-only guard would leave the same stale-final race open for other channel surfaces using the dispatcher.
