## Summary

Keep open: current main still appears to lack a shared freshness guard for buffered foreground final replies, and open PR #76970 is the linked implementation candidate rather than a reason to close before merge.

## What This Changes

The report asks OpenClaw to prevent an older WhatsApp foreground tool-turn final from being delivered after a newer inbound message has arrived in the same chat/session.

## Best Possible Solution

Land or replace #76970 with a shared auto-reply freshness guard plus regression coverage, then close this issue through the merged fix.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path: active-run steering accepts newer inbound messages while final delivery lacks an origin freshness check before sending the older final.

## Solution Assessment

Yes, the narrow maintainable direction is a shared buffered-dispatch freshness guard like #76970 proposes. A WhatsApp-only workaround would miss the generic auto-reply delivery surface.
