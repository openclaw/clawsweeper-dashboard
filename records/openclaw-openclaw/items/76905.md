## Summary

Keep open: current main still lacks a shared foreground-reply freshness guard, and open PR #76970 is already the linked implementation candidate for this reported race.

## What This Changes

The issue reports stale WhatsApp foreground replies when an older tool-backed run finalizes after a newer inbound message in the same chat.

## Best Possible Solution

Land or replace the shared auto-reply freshness guard from #76970, with regression coverage proving an older same-target final is suppressed after a newer inbound turn starts.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection on current main still shows a plausible path: a newer inbound message can steer the active run while the older run later delivers a final payload without freshness context.

## Solution Assessment

Yes. The narrow maintainable fix is a shared foreground final-delivery freshness check in auto-reply dispatch, not a WhatsApp-specific routing workaround; #76970 is already scoped that way.
