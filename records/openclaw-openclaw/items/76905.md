## Summary

Keep open: current main still lacks the shared foreground freshness guard for WhatsApp-style buffered auto-reply delivery, and open PR #76970 is the linked implementation candidate rather than a reason to close before merge.

## What This Changes

Suppress or clearly reroute stale foreground WhatsApp/auto-reply final responses when a newer same-session inbound message arrives before delivery.

## Best Possible Solution

Land or replace #76970 with a shared buffered-dispatch freshness guard plus regression coverage for same-session same-target stale finals.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection gives a medium-confidence path: WhatsApp uses shared buffered dispatch, active steering accepts newer inbound messages, and final delivery lacks a same-session origin freshness check.

## Solution Assessment

Yes, the best direction is a shared buffered-dispatch freshness guard like #76970 proposes. A WhatsApp-only workaround would miss the generic auto-reply delivery surface used by multiple channels.
