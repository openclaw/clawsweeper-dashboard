## Summary

Keep open: current `main` still has a source-level foreground final-delivery race, and open PR #76970 is already the closing implementation candidate that should be reviewed or replaced before this issue is closed.

## What This Changes

The issue reports that WhatsApp direct-message final replies from older tool-backed foreground runs can be delivered after a newer inbound message, making the older completion look like the latest answer.

## Best Possible Solution

Land or replace the shared auto-reply foreground freshness guard so final payload delivery checks origin run/message recency for the same channel, account, session, and target before sending.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was established, but source inspection shows a credible current-main path: newer input can steer an active run while the older run later enqueues a final reply without an origin freshness check.

## Solution Assessment

Is this the best way to solve the issue? Yes directionally: a shared foreground-final freshness check is narrower and more maintainable than a WhatsApp-specific workaround, but the linked fix remains open and unmerged.
