## Summary

Keep open: current main still lacks an origin/run freshness gate on foreground final auto-reply delivery, and #76970 is an open closing PR for this same defect. Adjacent #76832 covers native Codex replay state, but it does not replace the generic channel delivery ordering guard.

## What This Changes

Report requests that WhatsApp foreground auto-reply finals stay bound to their originating inbound message/run so stale tool-turn completions are not delivered after newer user messages.

## Best Possible Solution

Land or replace #76970 with a shared auto-reply foreground freshness guard and regression tests; keep #76832 scoped to provider replay state.

## Reproduction Assessment

No live WhatsApp/Codex repro was run here. Source inspection gives a credible current-main path: default steer can accept a newer inbound message while the older run's final delivery has no origin freshness check.

## Solution Assessment

Yes, directionally: a shared foreground final freshness guard in generic auto-reply dispatch is the narrowest maintainable fix. A WhatsApp-only workaround would miss the same delivery race on other messaging surfaces.
