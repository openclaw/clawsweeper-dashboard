## Summary

Keep open: current main still has a plausible foreground delivery race where active-run steering records a newer inbound message while the older run can still enqueue and deliver its final payload without a freshness check. Adjacent #76413/#76832 cover native Codex replay state, but they do not fully cover this WhatsApp foreground finalization path.

## What This Changes

The issue asks OpenClaw to bind WhatsApp foreground replies to their originating inbound message/run so older tool-turn completions cannot appear after a newer DM.

## Best Possible Solution

Add a generic foreground reply freshness guard in the shared auto-reply delivery path, with regression coverage proving stale finals are suppressed or clearly reclassified when a newer inbound message exists for the same session.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a credible current-main path: newer inbound messages can steer an active run while the older run still has an unguarded final delivery path.

## Solution Assessment

Yes, directionally: the narrowest maintainable fix is a shared foreground-final freshness check, not a WhatsApp-specific workaround. Keep #76832 separate because it addresses native Codex transport replay state rather than channel delivery ordering.
