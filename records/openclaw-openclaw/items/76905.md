## Summary

Keep open: current main still lacks a foreground freshness guard before final auto-reply delivery, and the linked fix PR is open rather than merged.

## What This Changes

Prevent stale foreground WhatsApp/auto-reply final responses from an older tool-backed turn from being delivered after a newer inbound message for the same session.

## Best Possible Solution

Land or replace #76970 with a shared auto-reply freshness guard and regression coverage, then close this issue through the merged fix.

## Reproduction Assessment

No live WhatsApp/Codex reproduction was established here. Source inspection gives a medium-confidence path: active-run steering accepts newer inbound messages while final delivery lacks an origin freshness check.

## Solution Assessment

Yes, the narrow maintainable fix is a shared auto-reply freshness guard before foreground final delivery. A WhatsApp-only workaround would leave the same buffered dispatch path exposed for other inbound channels.
