## Summary

Keep open: current main still lacks the shared foreground freshness guard needed to suppress an older final reply after a newer inbound message, and open PR #76970 is the linked implementation candidate rather than a reason to close before merge.

## What This Changes

The issue asks OpenClaw to suppress or explicitly reroute stale foreground WhatsApp final replies when a newer inbound message has arrived in the same chat/session.

## Best Possible Solution

Land or replace #76970 with a shared auto-reply freshness guard and regression coverage, then close this issue through the merged fix.

## Reproduction Assessment

No live WhatsApp/Codex reproduction was established here. Source inspection gives a medium-confidence path: active-run steering accepts a newer inbound message while final delivery lacks an origin freshness check before sending the older foreground final.

## Solution Assessment

Yes, the narrow maintainable direction is the shared buffered-dispatch freshness guard proposed by #76970. A WhatsApp-only workaround would miss the generic auto-reply delivery surface.
