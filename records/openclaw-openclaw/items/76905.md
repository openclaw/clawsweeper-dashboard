## Summary

Keep open: current main still has a credible source-level race in the shared foreground auto-reply finalization path, and open PR #76970 explicitly closes this issue but has not merged. The issue should stay open until that PR, or an equivalent shared freshness guard, lands.

## What This Changes

The issue requests that foreground WhatsApp auto-reply finals stay bound to their origin turn and be suppressed or reclassified when a newer same-chat inbound message arrives first.

## Best Possible Solution

Review and land PR #76970, or an equivalent shared auto-reply foreground-final freshness guard with regression coverage, then close this issue after merge.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection shows a credible current-main path: a newer inbound message can steer an active run while the older run can still enqueue or route its final payload without an origin freshness check.

## Solution Assessment

Yes, the linked direction is the narrow maintainable fix: add the guard in the shared auto-reply foreground delivery path rather than a WhatsApp-specific workaround. The issue should remain open while the closing PR is still open and unmerged.
