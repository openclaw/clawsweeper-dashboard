## Summary

Keep open: current main still lacks the shared foreground freshness guard needed for the reported stale WhatsApp final reply, and open PR #76970 is the linked implementation candidate rather than a reason to close before merge.

## What This Changes

The issue asks OpenClaw to prevent an older tool-backed foreground WhatsApp reply from being delivered after a newer inbound message in the same chat/session.

## Best Possible Solution

Land or replace #76970 with a shared auto-reply freshness guard plus regression coverage, then close this issue through the merged fix.

## Reproduction Assessment

No live WhatsApp/Codex reproduction was established here. Source inspection gives a medium-confidence path: active-run steering can accept a newer inbound message while the older run's final delivery path lacks an origin freshness check.

## Solution Assessment

Yes, the best direction is a shared buffered-dispatch freshness guard like #76970 proposes. A WhatsApp-only workaround would miss the generic auto-reply delivery surface used by multiple channels.
