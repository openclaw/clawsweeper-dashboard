## Summary

Keep open: current main still has a clear source path where WhatsApp foreground final delivery lacks a same-session freshness or supersession fence, while Telegram has analogous stale-finalization protection; the report is plausible but was not live-reproduced in this read-only pass.

## What This Changes

The issue reports that WhatsApp DM final replies from an older tool-backed turn can be delivered after a newer user message, making the old completion appear to answer the newer message.

## Best Possible Solution

Add a WhatsApp or shared foreground-reply freshness fence keyed by session/chat/message generation so final delivery from a superseded turn is suppressed or explicitly treated as background, with regression coverage modeled on the Telegram stale-finalization tests.

## Reproduction Assessment

Source-reproducible, but not live-reproduced. A focused test can start one WhatsApp dispatch, hold its final delivery, deliver a newer same-session dispatch, then release the first final and assert the stale final is not sent.

## Solution Assessment

Yes, keeping this open for a narrow repair is the best path. The maintainable fix is a delivery-time freshness fence, preferably shared where practical and modeled after Telegram's existing reply fence, rather than changing WhatsApp routing.
