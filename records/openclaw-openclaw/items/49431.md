## Summary

Keep open: the PR is still a useful restart-recovery direction, but the latest branch is not mergeable because pending-inbound replay has no production capture path, active-turn recovery is not visibly delivered, corrupt-store hardening remains incomplete, and real behavior proof is still absent.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay/recovery hooks, embedded-runner lifecycle writes, a system-event queue export, command-queue drain-state export, tests, and a changelog entry.

## Best Possible Solution

Fold the useful persistence pieces into the existing restart sentinel and session-delivery recovery architecture with a production capture path, explicit visible delivery or wake semantics, bounded state-file handling, and real Telegram/Discord restart proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the latest diff has no production writePendingInbound caller, active-turn recovery only queues a system event, and the store reads/backups corrupt state without byte/path hardening.

## Solution Assessment

Is this the best way to solve the issue? No. The store and tests are useful pieces, but the branch needs production capture, visible delivery or wake behavior, corrupt-state hardening, and alignment with current main's restart recovery paths before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Persist inbound messages before replaying them:** `src/gateway/server-startup-post-attach.ts:475-486`
  - body: The startup replay reads and clears pending-inbound.json, but this branch never calls writePendingInbound() from a production inbound or drain path. Messages arriving during the drain window therefore are not persisted, so restart replay has nothing real to restore.
  - confidence: 0.92
- **[P1] Wake or send the recovery notice:** `src/gateway/server-startup-post-attach.ts:725-729`
  - body: After resolving the channel and target, active-turn recovery only calls enqueueSystemEvent(). System events are prompt prefixes for a later turn, not direct channel sends or heartbeat wakeups, so an interrupted Telegram or Discord user can still see silence until another turn is triggered.
  - confidence: 0.9
- **[P1] Bound corrupt-store reads and backups:** `src/infra/pending-inbound-store.ts:333-352`
  - body: readPendingInboundFile() reads and parses the entire state file, then writes the full invalid payload to a predictable .bak path with fs.writeFile(). A corrupted or attacker-controlled state file can still cause local memory/disk amplification and backup-path overwrite risk in a writable state directory.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: Needs attention: the prompt-trust issue appears addressed, but the new pending-inbound store still has corrupt-file read and backup-write hardening gaps.

Concerns:

- **[medium] Unbounded pending-inbound state read:** `src/infra/pending-inbound-store.ts:333`
  - body: readPendingInboundFile() calls fs.readFile() and JSON.parse() on the whole store before applying entry caps, so a large or corrupted state file can force large allocations and parse work during startup or read-modify-write operations.
  - confidence: 0.86
- **[high] Predictable backup path follows existing links:** `src/infra/pending-inbound-store.ts:352`
  - body: On corrupt JSON, the store writes the raw payload to ${filePath}.bak with fs.writeFile(). In a writable or misconfigured state directory, that predictable backup path lacks exclusive-create or no-follow defenses and can overwrite an unintended target.
  - confidence: 0.84
