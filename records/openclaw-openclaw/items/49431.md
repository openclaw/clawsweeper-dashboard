## Summary

Keep open. Current main does not already implement this restart-recovery path, but the proposed patch is not merge-ready because its pending replay has no production capture path, replayed external content stays trusted, the new store has local state-file hardening gaps, and stale-turn recovery only queues a future prompt event instead of actually notifying the user.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay and recovery hooks, embedded run lifecycle persistence, JSON-file helper changes, and related tests.

## Best Possible Solution

Finish one coherent restart-resilience path: persist inbound work before drain rejection at the real ingress boundary, replay external content as untrusted, send stale-turn recovery notices through the actual delivery path, harden local state reads/backups, and explicitly align the design with or supersede #65650.

## Reproduction Assessment

Yes for the review defects: static inspection of the PR diff and current system-event contract is enough to reproduce them. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR is useful source work, but the safest solution needs real production capture, prompt-boundary fixes, bounded state-file handling, and a maintainer decision on the SQLite-vs-file store direction before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist inbound before rejecting during drain:** `src/process/command-queue.ts:230-231`
  - body: The new `isGatewayDraining()` helper is not used by any production inbound handler, and `writePendingInbound` appears only in the store and tests. When `enqueueCommandInLane` rejects during drain, the inbound message is still not serialized, so startup replay cannot recover real drain-time messages.
  - confidence: 0.91
- **[P2] Send stale-turn notices through delivery:** `src/gateway/server-startup-post-attach.ts:576-579`
  - body: This recovery path resolves a delivery target but only queues a system event. System events are consumed on a future agent turn, so if the restart killed the user's last turn and no new inbound arrives, the user never receives the resend notice.
  - confidence: 0.87
- **[P2] Mark replayed inbound previews untrusted:** `src/gateway/server-startup-post-attach.ts:428-431`
  - body: `eventText` includes stored sender/text fields from external inbound data, but the call omits `trusted: false`. The current system-event contract defaults this to trusted `System:` prompt content instead of the untrusted owner-downgraded form.
  - confidence: 0.9
- **[P2] Bound pending-store reads and backups:** `src/infra/pending-inbound-store.ts:333-352`
  - body: `readPendingInboundFile` reads and parses the whole store before any byte cap, then writes corrupt raw content to a predictable `.bak` path. A large or symlinked local state file can cause memory/disk amplification or redirect the backup write.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces security-sensitive local state-file and prompt-boundary behavior that still needs hardening before merge.

Concerns:

- **[medium] Trusted replay of user-controlled inbound preview:** `src/gateway/server-startup-post-attach.ts:428`
  - body: Pending replay formats sender and text preview from external inbound data and enqueues it without `trusted: false`, so prompt formatting treats it as trusted `System:` content.
  - confidence: 0.9
- **[high] Predictable corrupt-store backup can follow symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: The parse-failure path writes raw content to `${filePath}.bak` with `fs.writeFile`; if the state directory is writable or misconfigured, a preexisting symlink can redirect that overwrite.
  - confidence: 0.8
- **[medium] Oversized pending store can exhaust memory or disk:** `src/infra/pending-inbound-store.ts:333`
  - body: The pending store is read and parsed in full before validation, and corrupt content is backed up in full, so a large local state file can force high memory use and disk amplification during startup or recovery.
  - confidence: 0.84
