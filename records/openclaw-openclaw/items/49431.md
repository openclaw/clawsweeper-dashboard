## Summary

Keep open. The PR is useful source work, but it is not mergeable as-is because the branch still lacks a production inbound-capture path, queues recovery notices through prompt-only system events, and introduces security-sensitive state-file handling that needs hardening.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay/recovery hooks, embedded-runner lifecycle persistence, JSON helper changes, and related tests.

## Best Possible Solution

Finish one coherent restart-resilience path: persist inbound work before drain rejection at the real ingress boundary, replay external content as untrusted, send restart recovery notices through user-visible delivery, harden state-file reads/backups, and align the storage design with #65650.

## Reproduction Assessment

Yes for the review defects: static inspection of the PR diff and current main contracts reproduces the missing capture, prompt-trust, delivery, and state-file issues. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR is useful source work, but the current approach is not the best mergeable fix until ingress capture, real delivery, untrusted prompt formatting, bounded local-state handling, and the #65650 storage-direction decision are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist inbound before rejecting during drain:** `src/process/command-queue.ts:230-232`
  - body: The branch only exports `isGatewayDraining()` here; `writePendingInbound` is used by the store and tests but not by any production inbound handler. When `enqueueCommandInLane` rejects because the gateway is draining, the inbound message still has not been serialized, so startup replay cannot recover real drain-time messages.
  - confidence: 0.91
- **[P2] Mark replayed inbound previews untrusted:** `src/gateway/server-startup-post-attach.ts:428-431`
  - body: `eventText` includes stored sender/text fields from external inbound payloads, but this call omits `trusted: false`. Because system events default to trusted, the replayed preview is formatted as trusted `System:` prompt content instead of the downgraded untrusted form.
  - confidence: 0.9
- **[P2] Send stale-turn notices through delivery:** `src/gateway/server-startup-post-attach.ts:576-579`
  - body: This path resolves a channel delivery target but then only queues a system event. System events are consumed on a future agent turn, so if the interrupted turn was the user's last message, the affected channel never receives the restart/resend notice.
  - confidence: 0.87
- **[P2] Bound pending-store reads and backups:** `src/infra/pending-inbound-store.ts:333-352`
  - body: `readPendingInboundFile` reads and parses the whole store before any byte cap, then writes corrupt raw content to a predictable `.bak` path. A large or symlinked local state file can cause memory/disk amplification or redirect the backup write.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR introduces local state-file and prompt-boundary behavior that can still regress security before merge.

Concerns:

- **[medium] Trusted replay of user-controlled inbound preview:** `src/gateway/server-startup-post-attach.ts:428`
  - body: Pending replay formats sender and text preview from external inbound data and enqueues it without `trusted: false`, so prompt formatting treats it as trusted `System:` content.
  - confidence: 0.9
- **[high] Predictable corrupt-store backup can follow symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: The parse-failure path writes raw content to `${filePath}.bak` with `fs.writeFile`; if the state directory is writable or misconfigured, a preexisting symlink can redirect that overwrite.
  - confidence: 0.8
- **[medium] Oversized pending store can exhaust memory or disk:** `src/infra/pending-inbound-store.ts:333`
  - body: The pending store is read and parsed in full before validation, and corrupt content is backed up in full, so a large local state file can force high memory use and disk amplification during startup or recovery.
  - confidence: 0.84
