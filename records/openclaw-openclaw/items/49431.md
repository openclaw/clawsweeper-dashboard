## Summary

Keep this PR open, but it is not ready to merge: the branch adds useful restart-recovery pieces, yet source review still shows no production capture path for pending inbound messages, the active-turn recovery notice is queued for a later prompt rather than sent or woken visibly, corrupt-store handling has security gaps, and the external PR still lacks real restart proof.

## What This Changes

The PR adds a pending-inbound JSON store, active-turn persistence, startup replay/recovery hooks, system-event queue export/tests, JSON helper permission changes, and a changelog entry.

## Best Possible Solution

Finish the restart-recovery design end to end: capture inbound messages at the real drain rejection point, deliver or wake a visible recovery notice through the resolved route, harden state-file reads/backups, and prove the flow in a real restart setup.

## Reproduction Assessment

Yes for the PR defects at source level: diff inspection shows no production `writePendingInbound` caller, and the stale-turn recovery path queues only a system event. I did not establish a live restart reproduction because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

No. The store and tests are useful pieces, but this is not the best complete solution until production capture, visible send/wake semantics, corrupt-state hardening, and real restart proof are handled together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Persist inbound messages before replaying them:** `src/gateway/server-startup-post-attach.ts:422-433`
  - body: The startup replay reads and clears `pending-inbound.json`, but this branch never calls `writePendingInbound()` from a production inbound or drain path; diff search only finds the writer in the new store and tests. When a channel message hits the drain window, nothing is persisted, so restart replay has no real message to restore.
  - confidence: 0.92
- **[P1] Send or wake the recovery notice:** `src/gateway/server-startup-post-attach.ts:679-683`
  - body: After resolving the channel and target, this path only calls `enqueueSystemEvent()`. System events are prompt prefixes for the next turn, not direct channel sends, so an interrupted user still sees silence until they manually message again or another wake path runs.
  - confidence: 0.9
- **[P1] Bound corrupt-store reads and backup writes:** `src/infra/pending-inbound-store.ts:333-352`
  - body: `readPendingInboundFile()` loads and parses the whole state file, then writes the full invalid content to a predictable `.bak` path with `fs.writeFile()`, which follows pre-existing symlinks. A corrupted or attacker-controlled state file can still cause local memory/disk amplification and, with a writable state dir, overwrite an unintended target.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The system-event trust issue appears addressed in the latest head, but the new pending-inbound store still has corrupt-file read and backup-write hardening gaps.

Concerns:

- **[medium] Unbounded pending-inbound state read:** `src/infra/pending-inbound-store.ts:333`
  - body: `readPendingInboundFile()` calls `fs.readFile()` and `JSON.parse()` on the whole store before applying entry caps, so a large or corrupted state file can still force large allocations and parse work during startup or read-modify-write operations.
  - confidence: 0.86
- **[high] Predictable backup path follows symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: On corrupt JSON, the store writes the raw payload to `${filePath}.bak` with `fs.writeFile()`. That follows an existing symlink in a writable or misconfigured state directory and can overwrite an unintended target.
  - confidence: 0.84
