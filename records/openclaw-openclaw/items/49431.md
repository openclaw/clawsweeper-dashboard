## Summary

Keep this PR open, but the current patch is not mergeable: the PR head still lacks production ingress persistence, only queues recovery notices for a future prompt, and introduces durable local-state security risks.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay/recovery hooks, embedded-runner active-turn writes, JSON helper hardening, tests, and a changelog entry.

## Best Possible Solution

Land one coherent restart-resilience design, likely aligned with #65650, that persists inbound work before drain rejection, resumes or visibly notifies interrupted turns, and hardens durable state reads/backups.

## Reproduction Assessment

Yes. Static inspection of PR head 9049cf1 reproduces the blockers: no production writePendingInbound caller, recovery notices are prompt-only, and pending-store reads/backups are unbounded.

## Solution Assessment

No. The current PR is not the best mergeable solution until ingress capture, visible recovery delivery, local-state hardening, and the #65650 storage-direction decision are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Persist inbound before drain rejection:** `src/process/command-queue.ts:306-307`
  - body: The branch adds replay code, but no production caller writes pending inbound entries before enqueueCommandInLane rejects while the gateway is draining. Startup replay therefore has no durable inbound messages to recover for real drain-time events.
  - confidence: 0.93
- **[P2] Deliver recovery notices to the channel:** `src/gateway/server-startup-post-attach.ts:679-683`
  - body: This path resolves the originating channel and target, then only enqueues a system event. System events are consumed by a future prompt, so if the interrupted turn was the user's last message, the channel still receives no visible restart or resend notice.
  - confidence: 0.9
- **[P2] Bound pending-store reads and backups:** `src/infra/pending-inbound-store.ts:333-352`
  - body: readPendingInboundFile reads and parses the whole state file before any byte cap, then writes corrupt raw content to a predictable .bak path. A large or symlinked local state file can cause memory/disk amplification or redirect the backup overwrite during startup/recovery.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The PR adds durable local-state handling with concrete symlink and resource-exhaustion risks before merge.

Concerns:

- **[high] Predictable corrupt-store backup can follow symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: The parse-failure path writes raw content to `${filePath}.bak` with fs.writeFile; if the state directory is writable or misconfigured, a preexisting symlink can redirect that overwrite.
  - confidence: 0.82
- **[medium] Oversized pending store can exhaust memory or disk:** `src/infra/pending-inbound-store.ts:333`
  - body: The pending store is read and parsed in full before validation, and corrupt content is backed up in full, so a large local state file can force high memory use and disk amplification during startup or recovery.
  - confidence: 0.86
