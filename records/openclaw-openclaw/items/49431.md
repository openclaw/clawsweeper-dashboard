## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay/recovery hooks, runner lifecycle writes, JSON helper changes, system-event tests/export, a command-queue drain-state export, and a changelog entry.

## Best Possible Solution

Finish recovery end to end: capture drain-time inbound messages from production paths, deliver or wake a visible channel recovery notice, harden store I/O, and attach redacted real restart proof.

## Reproduction Assessment

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Persist inbound messages before replaying them:** `src/gateway/server-startup-post-attach.ts:422-433`
  - body: The startup replay reads and clears pending-inbound.json, but the branch never calls writePendingInbound() from a production inbound or drain path. A message that arrives during the drain window therefore is not persisted, so restart replay has nothing real to restore.
  - confidence: 0.92
- **[P1] Send a visible recovery notice:** `src/gateway/server-startup-post-attach.ts:679-683`
  - body: After resolving the channel and target, this path only calls enqueueSystemEvent(). System events are prompt prefixes for a future turn, not direct channel sends or wakeups, so an interrupted user still sees silence until they message again or some other wake path runs.
  - confidence: 0.9
- **[P1] Bound corrupt-store reads and backups:** `src/infra/pending-inbound-store.ts:333-352`
  - body: readPendingInboundFile() loads and parses the whole state file, then writes the full invalid content to a predictable .bak path with fs.writeFile(). A corrupted or attacker-controlled state file can still cause local memory/disk amplification and, with a writable state dir, overwrite an unintended target through a symlink.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: Needs attention: the system-event trust issue appears addressed, but the new pending-inbound store still has corrupt-file read and backup-write hardening gaps.

Concerns:

- **[medium] Unbounded pending-inbound state read:** `src/infra/pending-inbound-store.ts:333`
  - body: readPendingInboundFile() calls fs.readFile() and JSON.parse() on the whole store before applying entry caps, so a large or corrupted state file can force large allocations and parse work during startup or read-modify-write operations.
  - confidence: 0.86
- **[high] Predictable backup path follows symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: On corrupt JSON, the store writes the raw payload to ${filePath}.bak with fs.writeFile(). That follows an existing symlink in a writable or misconfigured state directory and can overwrite an unintended target.
  - confidence: 0.84
