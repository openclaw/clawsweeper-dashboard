## Summary

Keep open. The PR is useful source work, but the latest branch still does not persist real inbound work before drain rejection, treats replayed channel content as trusted prompt text, queues restart notices instead of delivering them visibly, and leaves pending-store local-state hardening gaps.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay/recovery hooks, embedded-runner lifecycle writes, JSON helper hardening, tests, and a changelog entry for gateway restart resilience.

## Best Possible Solution

Land one coherent restart-resilience path that persists inbound work at the ingress boundary before drain rejection, treats replayed channel content as untrusted, delivers recovery notices visibly, hardens durable state files, and resolves the file-versus-SQLite direction with #65650.

## Reproduction Assessment

Yes. Static inspection reproduces the review defects: the PR has no production writePendingInbound caller, the replay enqueue calls omit trusted: false, the recovery path only enqueues a system event, and the store reads/backups are unbounded.

## Solution Assessment

No. The current PR is not the best mergeable solution until ingress capture, prompt trust, visible recovery delivery, local-state hardening, and the #65650 storage-direction decision are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Persist inbound before drain rejection:** `src/process/command-queue.ts:306-307`
  - body: The branch adds a pending-inbound store and startup replay, but the production drain integration here only exposes the drain flag; `writePendingInbound` is not called from any inbound handler. When the gateway is draining and `enqueueCommandInLane` rejects new work, the original inbound message still has not been serialized, so the promised lossless replay path has nothing to recover.
  - confidence: 0.92
- **[P2] Mark replayed inbound previews untrusted:** `src/gateway/server-startup-post-attach.ts:519-522`
  - body: `eventText` includes stored sender and text preview fields from external channel payloads, but this enqueue call omits `trusted: false`. Since system events default to trusted, replayed user-controlled content is formatted as privileged `System:` prompt text instead of the downgraded untrusted form.
  - confidence: 0.91
- **[P2] Send stale-turn notices through delivery:** `src/gateway/server-startup-post-attach.ts:670-673`
  - body: This path resolves `resolvedChannel` and `resolvedTo`, then only queues a system event. System events surface on a future agent turn, so if the interrupted turn was the user's last message, the original channel still receives no visible restart/resend notice.
  - confidence: 0.89
- **[P2] Bound pending-store reads and backups:** `src/infra/pending-inbound-store.ts:333-352`
  - body: `readPendingInboundFile` reads and parses the whole state file before any byte cap, then writes corrupt raw content to a predictable `.bak` path. A large or symlinked local state file can cause memory/disk amplification or redirect the backup overwrite during startup/recovery.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces prompt-boundary and local state-file behavior with concrete security and supply-chain-adjacent hardening gaps before merge.

Concerns:

- **[medium] Trusted replay of user-controlled inbound preview:** `src/gateway/server-startup-post-attach.ts:519`
  - body: Pending replay formats sender and text preview from external inbound data and enqueues it without `trusted: false`, so prompt assembly treats it as trusted `System:` content.
  - confidence: 0.91
- **[high] Predictable corrupt-store backup can follow symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: The parse-failure path writes raw content to `${filePath}.bak` with `fs.writeFile`; if the state directory is writable or misconfigured, a preexisting symlink can redirect that overwrite.
  - confidence: 0.8
- **[medium] Oversized pending store can exhaust memory or disk:** `src/infra/pending-inbound-store.ts:333`
  - body: The pending store is read and parsed in full before validation, and corrupt content is backed up in full, so a large local state file can force high memory use and disk amplification during startup or recovery.
  - confidence: 0.84
