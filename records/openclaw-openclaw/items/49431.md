## Summary

Keep this PR open, but it is not merge-ready: the branch adds useful restart-recovery building blocks, yet the pending-inbound replay has no production capture path, stale-turn recovery is not actually user-visible, corrupt-store handling still has security hardening gaps, and contributor real behavior proof is missing.

## What This Changes

Adds a file-backed gateway pending-inbound and active-turn store, startup replay/recovery hooks, embedded-run active-turn writes, JSON helper hardening, tests, and a changelog entry.

## Best Possible Solution

Finish the restart-recovery design end to end: persist inbound messages at the real drain rejection point, wake or send a visible recovery/replay notification through the resolved channel route, harden state-file reads/backups, and prove the flow in a real restart setup.

Yes for the PR defects: source inspection shows the writer is not wired into production, and stale-turn recovery only queues a prompt event. I did not run a live restart reproduction because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

No. The proposed store and tests are useful, but the current branch is not the best complete solution until capture, visible delivery/wake semantics, and state-file hardening are implemented together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Persist inbound messages before replaying them:** `src/gateway/server-startup-post-attach.ts:422-433`
  - body: The startup replay reads and clears `pending-inbound.json`, but this PR never wires `writePendingInbound()` into any production inbound/drain path; it appears only in the new store and tests. When a channel message hits `GatewayDrainingError` during drain, nothing is persisted, so restart replay has no real message to restore.
  - confidence: 0.92
- **[P1] Send or wake the recovery notice instead of queueing it:** `src/gateway/server-startup-post-attach.ts:679-683`
  - body: After resolving a channel target, this path only calls `enqueueSystemEvent()`. System events are in-memory prompt prefixes for the next turn, and this code neither sends to `resolvedChannel`/`resolvedTo` nor requests a heartbeat, so an interrupted user still sees silence until they manually message again.
  - confidence: 0.9
- **[P1] Bound and harden corrupt-store backup reads:** `src/infra/pending-inbound-store.ts:333-352`
  - body: The pending-inbound reader loads and parses the entire state file, then writes the full invalid content to a predictable `.bak` path with `fs.writeFile()`, which follows pre-existing symlinks. A corrupted or attacker-controlled state file can still cause local memory/disk amplification and, with a writable state dir, overwrite another file.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The system-event trust issue appears addressed, but the new pending-inbound store still has concrete corrupt-file read and backup-write hardening gaps.

Concerns:

- **[medium] Unbounded pending-inbound state read:** `src/infra/pending-inbound-store.ts:333`
  - body: `readPendingInboundFile()` calls `fs.readFile()` and `JSON.parse()` on the whole store before applying entry caps, so a large or corrupted state file can still force large allocations and parse work during startup or read-modify-write operations.
  - confidence: 0.86
- **[high] Predictable backup path follows symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: On corrupt JSON, the store writes the raw payload to `${filePath}.bak` with `fs.writeFile()`. That follows an existing symlink in a writable or misconfigured state directory and can overwrite an unintended target.
  - confidence: 0.84
