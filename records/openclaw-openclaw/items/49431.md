## Summary

Keep open: this is useful restart-recovery work, but the PR is not mergeable as-is because the replay store has no production inbound writer, active-turn recovery only queues a future prompt event instead of notifying the channel, corrupt-store I/O is still security-sensitive, and no real restart proof is attached.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay/recovery hooks, embedded-runner lifecycle writes, system-event queue exports/tests, command-queue drain-state export, JSON helper changes, and a changelog entry.

## Best Possible Solution

Finish or replace this branch with an end-to-end, hardened restart-recovery design that captures accepted inbound work in production, sends or wakes a visible recovery notice when appropriate, preserves current main recovery semantics, and includes real restart proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level for the PR defects: diff inspection shows no production writePendingInbound caller, active-turn recovery only queues a prompt event, and corrupt-store handling reads/backups unbounded state.

## Solution Assessment

Is this the best way to solve the issue? No. The store and tests are useful pieces, but the solution is incomplete until production capture, visible delivery or wake semantics, corrupt-state hardening, and real restart proof are handled together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Persist inbound messages before replaying them:** `src/gateway/server-startup-post-attach.ts:422-433`
  - body: The startup replay reads and clears pending-inbound.json, but the branch never calls writePendingInbound() from a production inbound or drain path. A message that arrives during the drain window therefore is not persisted, so restart replay has nothing real to restore.
  - confidence: 0.92
- **[P1] Send a visible recovery notice:** `src/gateway/server-startup-post-attach.ts:679-683`
  - body: After resolving the channel and target, this path only calls enqueueSystemEvent(). System events are prompt prefixes for a future turn, not direct channel sends or wakeups, so an interrupted user still sees silence until they message again or another wake path runs.
  - confidence: 0.9
- **[P1] Bound corrupt-store reads and backups:** `src/infra/pending-inbound-store.ts:333-352`
  - body: readPendingInboundFile() loads and parses the whole state file, then writes the full invalid content to a predictable .bak path with fs.writeFile(). A corrupted or attacker-controlled state file can still cause local memory/disk amplification and, with a writable state dir, overwrite an unintended target through a symlink.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The system-event trust issue appears addressed on the latest head, but the new pending-inbound store still has corrupt-file read and backup-write hardening gaps.

Concerns:

- **[medium] Unbounded pending-inbound state read:** `src/infra/pending-inbound-store.ts:333`
  - body: readPendingInboundFile() calls fs.readFile() and JSON.parse() on the whole store before applying entry caps, so a large or corrupted state file can force large allocations and parse work during startup or read-modify-write operations.
  - confidence: 0.86
- **[high] Predictable backup path follows symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: On corrupt JSON, the store writes the raw payload to ${filePath}.bak with fs.writeFile(). That follows an existing symlink in a writable or misconfigured state directory and can overwrite an unintended target.
  - confidence: 0.84
