## Summary

Keep open. Current main does not include this restart-resilience change, and the PR is not merge-ready because pending inbound entries are never persisted from a production ingress path, stale-turn recovery only queues a next-turn system prompt, and the new store still has local-state hardening gaps; #65650 also keeps the durable storage direction open.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay/recovery hooks, embedded-runner active-turn writes, JSON file hardening, system-event queue tests, and a changelog entry.

## Best Possible Solution

Land one coherent restart-resilience path that persists inbound work at the ingress boundary before drain rejection, sends visible channel recovery notices or safely resumes interrupted turns, hardens durable state files, and resolves the file-versus-SQLite direction with #65650.

## Reproduction Assessment

Yes. Static inspection of the PR head reproduces the blockers: no production writePendingInbound caller, recovery notices are only queued as system events for a future prompt, and the pending store reads/backups are unbounded.

## Solution Assessment

No. The current PR is not the best mergeable solution until ingress capture, visible recovery delivery, local-state hardening, and the #65650 storage-direction decision are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Persist inbound before drain rejection:** `src/process/command-queue.ts:306-307`
  - body: The branch adds replay code, but the production changes only expose the drain flag; `writePendingInbound` is not called from any inbound handler. Once `enqueueCommandInLane` rejects during drain, the original message has not been serialized, so startup replay has nothing to recover.
  - confidence: 0.92
- **[P2] Deliver stale-turn recovery notices visibly:** `src/gateway/server-startup-post-attach.ts:679-683`
  - body: This path resolves `resolvedChannel` and `resolvedTo`, then only enqueues a system event. System events are prompt-prefix input for a future turn, so if the interrupted message was the user's last message, the original channel still receives no visible restart/resend notice.
  - confidence: 0.9
- **[P2] Bound pending-store reads and corrupt backups:** `src/infra/pending-inbound-store.ts:333-352`
  - body: `readPendingInboundFile` reads and parses the whole state file before any byte cap, then writes corrupt raw content to a predictable `.bak` path. A large or symlinked local state file can cause memory/disk amplification or redirect the backup overwrite during startup/recovery.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces durable local-state handling with concrete symlink and resource-exhaustion risks before merge.

Concerns:

- **[high] Predictable corrupt-store backup can follow symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: The parse-failure path writes raw content to `${filePath}.bak` with `fs.writeFile`; if the state directory is writable or misconfigured, a preexisting symlink can redirect that overwrite.
  - confidence: 0.82
- **[medium] Oversized pending store can exhaust memory or disk:** `src/infra/pending-inbound-store.ts:333`
  - body: The pending store is read and parsed in full before validation, and corrupt content is backed up in full, so a large local state file can force high memory use and disk amplification during startup or recovery.
  - confidence: 0.86
