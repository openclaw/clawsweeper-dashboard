## Summary

Keep open: current main still shows only effective node commands in `node.list` / `node.describe`, while the latest real-environment update shows this symptom can be a pending reapproval gate that needs diagnostics rather than automatic command exposure. The open fix path is https://github.com/openclaw/openclaw/pull/92547, which references this issue and is not merged.

## What This Changes

The issue requests that connected node-hosts with declared command surfaces not appear as `Commands: (none reported)` without a clear approval or reapproval diagnostic and repair command.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/92547 so `nodes status` and `nodes describe` show pending approval/reapproval state and the exact approval command while keeping effective commands restricted until approval.

## Reproduction Assessment

Yes for the revised core gap from source and maintainer real-environment evidence: v2026.6.6 showed a connected node with effective empty commands while the pending request held the declared surface, and current main still omits pending requests from `node.list` / `node.describe`. I did not personally run a current-main Windows node setup, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes: additive pending-approval diagnostics are the maintainable fix because they explain the operator action without weakening the command approval boundary. Automatically expanding approved commands would be the wrong solution because it would bypass the reapproval gate.
