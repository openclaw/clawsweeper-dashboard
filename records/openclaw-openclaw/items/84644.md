## Summary

Keep open: the report is a fresh, concrete regression report with a second user seeing the same commandless-node symptom on macOS, and current source still has a commandless connected state when the node command-pairing approval is missing or stale. I do not have enough evidence to close it as fixed or as pure support because the report does not include the `openclaw nodes pending` / node-pair approval state needed to distinguish a pairing-flow bug from an unclear approval/docs path.

## What This Changes

The issue reports that a paired Windows headless node-host connects to a Linux gateway but advertises no supported commands, blocking `system.which` and `system.run`.

## Best Possible Solution

Make the intended approval path unambiguous and reliable: after the required node command-pairing approval, `nodes describe` should advertise `system.which`, `system.run.prepare`, and `system.run`; if device pairing alone is intentionally commandless, CLI/docs should surface the pending node-pair approval and exact command to run.

## Reproduction Assessment

Partially. Source inspection reproduces a connected, device-paired node with empty effective commands when the separate gateway-owned node command pairing is missing or stale, but I do not have a high-confidence live reproduction of the reporter's claimed fully approved Windows path.

## Solution Assessment

Unclear. The maintainable fix should first verify whether the node-pair approval was pending/missing; if so, improve CLI/docs diagnostics, but if approval was completed then the repair belongs in gateway reconciliation or node command policy rather than a fallback that invokes undeclared commands.
