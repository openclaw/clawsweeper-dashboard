Keep open: this is a fresh, concrete regression report with a second macOS node confirmation, and current main still has the same node-host advertisement and gateway reconciliation surfaces that can leave a connected node commandless.

## What This Changes

The issue reports that a freshly paired headless node connects as `node-host/node` but advertises no commands, preventing `system.which` and `system.run` invocation from the gateway.

## Best Possible Solution

Fix the node-host pairing/reconnect path so freshly approved headless nodes retain their declared `system.which`, `system.run.prepare`, and `system.run` commands, with regression tests and real Windows or macOS node-to-gateway proof.

## Reproduction Assessment

No high-confidence live repro was run here. The reporter and a second commenter provide concrete cross-platform output, and source inspection shows exactly how an empty effective command list produces `Commands: (none reported)` and the `node did not declare commands` invoke rejection.

## Solution Assessment

Unclear until the live pairing trace identifies why commands are being dropped. The fix should preserve the declared-command and pairing approval contract rather than adding a gateway fallback that invokes undeclared commands.
