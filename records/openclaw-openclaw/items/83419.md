## Summary

Keep open: current main appears to have narrowed the active Discord metadata path by concatenating current-turn context into the prompt, but OpenRouter still explicitly uses a passthrough replay policy that disables Anthropic turn normalization, so the remaining provider-policy question and live proof are not settled.

## What This Changes

The issue asks OpenClaw to avoid sending separate user-role metadata and user-message turns for group chat context, especially on Anthropic models through OpenRouter.

## Best Possible Solution

Make OpenRouter replay policy model-ref-aware so Anthropic/Claude routes get Anthropic turn normalization while Gemini passthrough behavior stays intact, then prove it with focused replay-policy tests and a redacted live Discord/OpenRouter check.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction on current main. Source inspection gives a medium-confidence path: Discord topic metadata is still prompt context, active prompt assembly now concatenates it with the prompt, and OpenRouter still disables Anthropic turn normalization if adjacent user turns reach replay/provider dispatch.

## Solution Assessment

Partly. Merging metadata into the active prompt is already how current main handles the obvious first-turn surface; the safer remaining direction is scoped OpenRouter Anthropic/Claude replay normalization, not moving Discord topic metadata into trusted system context.
