## Summary

Keep open: current main only partially narrows the report. Active Discord metadata is now concatenated into the current prompt body, but OpenRouter still owns a passthrough replay policy that disables Anthropic turn normalization for Claude/OpenRouter routes, and the issue still needs maintainer policy plus live provider proof.

## What This Changes

The issue reports that Discord group-chat metadata and the actual inbound prompt can be emitted as adjacent user-role messages, causing Anthropic-compatible OpenRouter requests to fail role alternation.

## Best Possible Solution

Make OpenRouter replay policy model-ref-aware so Anthropic/Claude routes get Anthropic turn normalization while Gemini and other passthrough routes keep their existing behavior, then verify with focused replay-policy tests and a redacted live Discord/OpenRouter check.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path: Discord topic metadata enters untrusted context, active prompt assembly now concatenates that context with the current prompt, but OpenRouter disables Anthropic turn normalization if adjacent user-role turns reach replay/provider dispatch.

## Solution Assessment

Partly. Merging metadata into the active prompt is already how current main handles the obvious first-turn surface, so the safer remaining fix is scoped OpenRouter Anthropic/Claude replay normalization rather than moving Discord topic metadata into trusted system context.
