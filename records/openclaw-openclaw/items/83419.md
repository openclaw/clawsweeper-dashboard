## Summary

Keep open: current main still has a source-visible path where channel metadata is carried as current inbound context and prepended to the model prompt, while fresh discussion reports the same duplicate user-turn shape on OpenClaw 2026.5.18 across direct Anthropic, Telegram, and Control UI. The open runtime-context PR is the best current implementation candidate, but it is not merged and may not cover the OpenRouter replay-policy edge.

## What This Changes

The issue asks OpenClaw to stop injecting channel/group metadata as separate or duplicated user-role content so Anthropic-compatible providers and transcripts see one logical user turn per inbound message.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81521 or an equivalent embedded-runner fix, then separately decide and test any OpenRouter Claude replay-policy follow-up with redacted live provider proof.

## Reproduction Assessment

No high-confidence current-main live reproduction was run. Source inspection plus the 2026.5.18 Telegram/Control UI JSONL comment give a medium-confidence reproduction path through current inbound context, embedded prompt submission, and provider replay policy.

## Solution Assessment

Unclear as a complete fix until maintainers decide scope. Routing current-turn metadata out of user text is the best embedded-runner direction, but OpenRouter Claude replay normalization and non-embedded paths need explicit coverage before this can be closed.
