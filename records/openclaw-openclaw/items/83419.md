## Summary

Keep open: current main only partially addresses the reported first-turn shape by concatenating active inbound context into the submitted prompt, but OpenRouter still uses a passthrough replay policy that disables Anthropic turn normalization, so the remaining provider-policy question needs maintainer decision and live OpenRouter proof.

## What This Changes

The issue asks OpenClaw to avoid sending Discord group-chat metadata and the actual inbound message as adjacent user-role messages to Anthropic/OpenRouter models.

## Best Possible Solution

Make OpenRouter replay policy model-ref-aware so Claude/Anthropic-routed OpenRouter requests get Anthropic turn normalization while Gemini and other passthrough routes keep their existing behavior, then prove it with focused replay-policy tests and a redacted live Discord/OpenRouter run.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path: Discord topic metadata enters untrusted context, active prompt assembly now concatenates that context with the current prompt, but OpenRouter disables Anthropic turn normalization if adjacent user-role turns reach replay/provider dispatch.

## Solution Assessment

Partly. Merging metadata into the active prompt is already how current main handles the obvious first-turn surface, so the safer remaining fix is scoped OpenRouter Anthropic/Claude replay normalization rather than moving Discord topic metadata into trusted system context.
