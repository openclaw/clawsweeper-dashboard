## Summary

This should stay open: current prompt assembly appears to merge Discord metadata with the current user body before model submission, but OpenRouter's provider policy still explicitly disables Anthropic turn validation, leaving a credible provider-boundary path for consecutive user-role payloads on Anthropic-backed OpenRouter models.

## What This Changes

The issue reports that Discord group-chat metadata and the actual message can reach Anthropic/OpenRouter as consecutive user-role messages and cause fallback away from the configured primary model.

## Best Possible Solution

Add a provider-aware fix for OpenRouter Anthropic/Claude model refs that squashes or validates consecutive user turns before dispatch, with regression coverage, while keeping Discord topic metadata as fenced untrusted user context and leaving failover transcript persistence to the related issue.

## Reproduction Assessment

Yes at source level, with medium confidence: current OpenRouter policy disables Anthropic turn validation even though the shared helper can merge consecutive user turns. The review did not reproduce the reporter's live OpenRouter payload, and current Discord prompt assembly itself appears to concatenate context and body into one prompt.

## Solution Assessment

Yes, the provider-adapter direction is the best fit: OpenRouter Anthropic model refs should get Anthropic-compatible turn validation before dispatch. Moving Discord topic metadata out of untrusted user context would fight the documented prompt boundary and would not address same-role transcript history.
