## Summary

Keep open: current main partially narrows the original group-chat metadata concern, but it does not fully settle the reported provider/session failure. OpenRouter still uses a passthrough replay policy that disables Anthropic turn normalization for Claude-over-OpenRouter routes, and a later live comment reports the same metadata-wrapped plus stripped user-message duplication on direct Anthropic, Telegram, and Control UI surfaces.

## What This Changes

The issue asks OpenClaw to avoid creating duplicate or consecutive user-role turns when inbound metadata and the actual user message are assembled for Anthropic-compatible model calls.

## Best Possible Solution

Make inbound metadata/user-message assembly produce one authoritative current turn, then apply Claude-aware provider replay normalization only where needed while preserving Gemini passthrough behavior and proving it with redacted live Telegram, Discord/OpenRouter, and Control UI checks.

## Reproduction Assessment

No, not with high confidence against current main. The issue has concrete live captures from 2026.5.18 and current source shows a plausible remaining OpenRouter replay-policy gap, but this review did not run a live provider/channel reproduction on e42726204490fa6829105efcf9b54284a937fc85.

## Solution Assessment

Unclear as a single fix path. Merging metadata into the active prompt is already partly present on current main, so the maintainable solution likely needs both inbound-context deduplication and scoped Claude-aware replay normalization rather than only moving Discord metadata into system context.
