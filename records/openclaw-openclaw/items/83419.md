## Label Justifications

- `P1`: The report describes a configured primary Anthropic/OpenRouter model failing in a real group-chat workflow and falling through to fallback providers.
- `impact:session-state`: The affected behavior is prompt/replay role ordering in session context.

## Summary

Keep open: current main appears to have narrowed the active-turn metadata split by concatenating inbound context with the current prompt, but OpenRouter still uses a passthrough replay policy where Anthropic turn normalization is disabled, and the related fallback-duplication PR does not prove this OpenRouter/Claude role-shape report is fixed.

## What This Changes

The issue reports that Discord/group-chat metadata can be sent as a separate user-role message before the actual user message, causing Anthropic/OpenRouter requests to contain consecutive user messages.

## Best Possible Solution

Make OpenRouter replay behavior model-ref-aware so Claude/Anthropic routes receive Anthropic turn normalization while Gemini and other passthrough routes keep existing behavior, then verify with focused replay-policy tests and a redacted live Discord/OpenRouter check.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence path: Discord topic metadata enters untrusted context, active prompt assembly now concatenates it with the current prompt, and OpenRouter disables Anthropic turn normalization if adjacent user turns reach replay/provider dispatch.

## Solution Assessment

Partly. Merging metadata into the active prompt is already how current main handles the obvious first-turn surface, so the safer remaining fix is scoped OpenRouter Claude/Anthropic replay normalization rather than moving Discord topic metadata into trusted system context.
