## Summary

Keep open: current main reduces the active-turn metadata problem by joining inbound context into the prompt body, but OpenRouter still uses provider-owned replay hooks that disable Anthropic turn normalization, and tests currently lock that behavior. The report remains a plausible OpenRouter/Claude compatibility bug needing maintainer policy judgment and live provider proof.

## What This Changes

The issue asks OpenClaw to avoid sending separate consecutive `user` messages when Discord group-chat metadata and the actual user message are submitted to Anthropic models through OpenRouter.

## Best Possible Solution

Add model-ref-aware OpenRouter Anthropic/Claude turn normalization at the replay or provider payload boundary, preserve non-Anthropic OpenRouter passthrough behavior, and verify with focused unit coverage plus a redacted live Discord/OpenRouter check.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection supports a medium-confidence path because Discord topic metadata reaches untrusted context and OpenRouter currently disables Anthropic turn normalization, while the report supplies an intercepted payload.

## Solution Assessment

Partly. Merging active inbound metadata into the current prompt is already present on main, so the safer remaining fix is scoped OpenRouter Anthropic/Claude replay normalization rather than moving Discord metadata into trusted system context.
