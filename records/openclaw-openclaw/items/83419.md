## Summary

Keep open: current main has narrowed the original first-turn prompt concern, but it has not resolved the OpenRouter Anthropic replay policy. Discord metadata is now prompt-local in the active path, yet OpenRouter still uses a passthrough policy that disables Anthropic turn normalization, so maintainers need a policy decision plus live provider proof.

## What This Changes

The issue asks OpenClaw to prevent Discord group-chat metadata from creating adjacent user-role turns that Anthropic/OpenRouter rejects.

## Best Possible Solution

Make OpenRouter replay policy model-ref-aware so Anthropic/Claude routes get Anthropic turn normalization while Gemini and other passthrough routes keep their existing behavior, then verify with focused replay-policy tests and a redacted live Discord/OpenRouter check.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path: Discord topic metadata enters untrusted prompt context, current active prompts concatenate that context with the prompt, but OpenRouter disables Anthropic turn normalization if adjacent user-role turns reach replay/provider dispatch.

## Solution Assessment

Partly. Concatenating metadata into the active prompt already handles the obvious first-turn surface, so the better remaining fix is scoped OpenRouter Anthropic/Claude replay normalization rather than moving channel topic metadata into trusted system context.
