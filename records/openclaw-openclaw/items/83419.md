## Summary

Keep open: current main appears to have narrowed the first-turn prompt assembly path, but OpenRouter still uses a passthrough replay policy that disables Anthropic turn normalization for Claude/OpenRouter routes, so the provider-compatibility bug is not proven fixed.

## What This Changes

The issue reports that Discord group-chat metadata is sent as a separate user message before the real inbound message, causing OpenRouter-routed Anthropic models to reject non-alternating role sequences and fall back.

## Best Possible Solution

Make OpenRouter replay policy model-ref-aware so Claude/Anthropic routes get Anthropic user-turn normalization while Gemini and other passthrough routes keep their intended behavior, then verify with focused replay-policy tests plus a redacted live Discord/OpenRouter check.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence path: Discord topic metadata enters untrusted context, active prompt assembly now concatenates that context with the current prompt, but OpenRouter disables Anthropic turn normalization if adjacent user turns reach replay/provider dispatch.

## Solution Assessment

Partly. Merging metadata into the active prompt is already the current-main shape for the obvious first-turn surface, so the safer remaining fix is scoped OpenRouter Anthropic/Claude replay normalization rather than moving Discord topic metadata into trusted system context.
