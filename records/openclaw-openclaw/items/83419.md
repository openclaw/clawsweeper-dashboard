## Summary

Keep open: current main still has a credible OpenRouter Anthropic/Claude provider-boundary path where same-role user turns are not normalized, even though the current Discord prompt prefix itself is concatenated with the inbound message body.

## What This Changes

The issue reports that Discord group chat metadata can create consecutive user-role messages for Anthropic-backed OpenRouter requests, causing provider rejection and fallback.

## Best Possible Solution

Add model-ref-aware OpenRouter Anthropic/Claude turn normalization or validation before dispatch, preserve existing Gemini/OpenAI OpenRouter behavior, and coordinate with the broader Anthropic normalization issue plus the separate failover transcript fix.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence reproduction path: OpenRouter openai-completions currently disables Anthropic same-role validation while the issue supplies an intercepted payload showing adjacent user turns.

## Solution Assessment

Partly. Provider-boundary normalization for OpenRouter Anthropic/Claude model refs is the best fit; moving Discord topic metadata into trusted system context is not ideal because the prompt code intentionally treats channel metadata as user-role untrusted context and already concatenates the current prefix with the user body.
