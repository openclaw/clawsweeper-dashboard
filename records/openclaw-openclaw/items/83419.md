## Summary

Keep open: current main appears to have narrowed the active Discord prompt path by concatenating untrusted context into the current user prompt, but OpenRouter still explicitly uses a passthrough replay policy that disables Anthropic turn normalization for Claude/OpenRouter routes. The issue is recent, assigned, and already marked for maintainer/product review plus live repro, so this should not be auto-closed or queued as a new fix PR yet.

## What This Changes

The issue reports Discord group-chat untrusted context being sent as a separate user-role message before the actual message, causing Anthropic/OpenRouter rejection and fallback away from the configured primary model.

## Best Possible Solution

Make OpenRouter replay policy model-ref-aware so Anthropic/Claude routes normalize consecutive same-role turns while Gemini and non-Anthropic OpenRouter routes keep their existing passthrough behavior, then prove it with focused tests and a redacted live Discord/OpenRouter check.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction against current main. Source inspection gives a medium-confidence path: Discord topic metadata enters untrusted context and current active prompt assembly concatenates it with the prompt, while OpenRouter still disables Anthropic turn normalization if adjacent user-role turns reach replay/provider dispatch.

## Solution Assessment

Unclear. Folding metadata into the active prompt is already the current-main behavior for the obvious first-turn surface, so the cleaner remaining direction is a scoped OpenRouter Anthropic/Claude replay policy with live proof rather than moving untrusted channel topic text into trusted system context.
