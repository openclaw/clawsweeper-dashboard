## Summary

Keep open: current main narrows the active Discord metadata path and the downstream failover duplication issue is fixed, but OpenRouter still uses passthrough replay policy with Anthropic turn validation disabled for Claude/OpenRouter routes; the remaining behavior needs live provider proof and maintainer policy.

## What This Changes

The issue requests merging Discord group-chat metadata with the actual user message, or applying provider turn normalization, so Anthropic/OpenRouter requests do not contain consecutive user-role messages.

## Best Possible Solution

Make OpenRouter replay policy model-ref-aware so Anthropic/Claude routes get Anthropic turn normalization while Gemini and other passthrough routes keep existing behavior, then verify with focused replay-policy tests and a redacted live Discord/OpenRouter check.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path: Discord topic metadata enters untrusted context, active prompt assembly now concatenates that context with the prompt, but OpenRouter disables Anthropic turn normalization if adjacent user-role turns reach replay/provider dispatch.

## Solution Assessment

Partly. Merging metadata into the active prompt is already how current main handles the obvious first-turn surface, so the safer remaining fix is scoped OpenRouter Anthropic/Claude replay normalization after live proof.
