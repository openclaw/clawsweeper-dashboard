Active Discord metadata is now assembled into the current prompt body, but OpenRouter still uses passthrough replay hooks that disable Anthropic turn normalization, and the related failover PR only addresses transcript duplication after failures.

## What This Changes

The issue reports that Discord group chat metadata and the actual inbound message can reach OpenRouter Anthropic models as adjacent user-role messages, causing primary-model failure and fallback.

## Best Possible Solution

Add model-ref-aware OpenRouter Anthropic/Claude turn normalization at the provider replay boundary, preserve OpenRouter Gemini/OpenAI passthrough behavior, and verify with a redacted live Discord/OpenRouter repro plus focused unit coverage.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection plus the intercepted payload give a medium-confidence path because OpenRouter currently disables Anthropic turn normalization while the report shows adjacent user-role messages reaching that provider route.

## Solution Assessment

Partly. The current active-turn prompt code already concatenates metadata with the body, so the maintainable fix is scoped OpenRouter Anthropic/Claude replay normalization rather than moving Discord topic metadata into trusted system context.
