## Label Justifications

- `P2`: The issue is a normal-priority operational bug: expired provider auth can create noisy logs and disk I/O, but it does not make the core runtime unusable.
- `impact:auth-provider`: The reported behavior is triggered by expired provider tokens and the auth/fallback cooldown path.

## Summary

Keep open. Current main has auth cooldown handling and an opt-in auth skip cache, but the default path still emits warn-level `model-fallback/decision` records for each fallback skip/probe/failure decision, so the reported log spam remains source-reproducible.

## What This Changes

The issue reports excessive `model-fallback/decision` logs when an auth provider token expires and model fallback repeatedly evaluates the same failing provider path.

## Best Possible Solution

Add a bounded default coalescing or throttle layer for repeated auth-class fallback decision observations while preserving current fallback routing, auth cooldown semantics, and structured first-occurrence diagnostics.

## Reproduction Assessment

Yes by source inspection, but not by a fresh live expired-token run. Expired/invalid token errors classify as `auth`, auth cooldown state is recorded, and the fallback loop still feeds repeated skip/probe/failure decisions into an unthrottled warn-level logger while the skip cache is disabled by default.

## Solution Assessment

Yes, a logging-layer coalescer is the best narrow fix for this issue. The broader provider quarantine work in https://github.com/openclaw/openclaw/issues/47910 can remain separate because this report is specifically about repeated decision logs, not changing fallback routing policy.
