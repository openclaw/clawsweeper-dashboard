## Summary

Keep open. Current main has auth cooldown and cooldown-probe throttling, but normal non-probe `model-fallback/decision` warnings still flow through the logger without duplicate coalescing, including to file logs, so the reported log-spam behavior remains actionable.

## What This Changes

The issue asks OpenClaw to reduce repeated model fallback decision logs when an auth provider token expires and fallback retries continue during cooldown.

## Best Possible Solution

Add bounded duplicate coalescing in the model fallback observability path, keyed by stable decision/requested/candidate/reason/next/error fingerprint, emitting the first warning and periodic summary/count updates while leaving fallback selection, auth cooldown semantics, and diagnostic step forwarding unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a fallback chain with a cooldowned or failing auth profile and invoke `runWithModelFallback` repeatedly; current main reaches the direct decision warning path each time, and file logging happens before probe-only console suppression.

## Solution Assessment

No. Current main improves cooldown and fallback diagnostics but is not the best solution for this issue because it still logs each normal fallback decision; the narrow maintainable fix is coalescing in `src/agents/model-fallback-observation.ts` or a tightly scoped logging helper, not a change to provider fallback policy.
