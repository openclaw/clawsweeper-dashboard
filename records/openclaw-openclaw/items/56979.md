## What This Changes

The issue asks for cooldown-aware throttling or coalescing of repeated model-fallback decision logs when an auth provider token expires.

## Best Possible Solution

Add a bounded, candidate-aware auth-class coalescer in fallback decision logging with regression coverage, while leaving provider quarantine and status policy to https://github.com/openclaw/openclaw/issues/47910.

## Reproduction Assessment

Yes for a source-level reproduction: current main classifies HTTP 401/invalid-token failures as auth, routes failed and skipped candidates through the fallback decision logger, and that logger warns unconditionally. I did not run a live expired-token gateway scenario because this read-only review has no real provider credentials.

## Solution Assessment

Yes for the narrow logging bug: candidate-aware auth-class coalescing is the maintainable fix because it reduces duplicate diagnostics without changing provider routing. The broader provider quarantine direction should stay in https://github.com/openclaw/openclaw/issues/47910.
