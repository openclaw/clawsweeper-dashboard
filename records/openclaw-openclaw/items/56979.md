## Summary

Keep this issue open. Current main still emits every non-probe `model-fallback/decision` warning directly, while the existing cooldown and probe-throttle logic only reduces provider retry behavior, not duplicate fallback-decision log output.

## What This Changes

The issue asks OpenClaw to reduce repeated model fallback decision logs when an auth provider token expires and fallback runs repeatedly.

## Best Possible Solution

Add bounded duplicate coalescing in the model fallback observability layer, keyed by a stable decision/requested/candidate/reason/next/error fingerprint, with the first event emitted immediately and periodic summary/count emissions for repeats.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a cooldowned auth profile with model fallbacks and call `runWithModelFallback` repeatedly; current main reaches `logModelFallbackDecision` on each run and has no duplicate suppression for ordinary non-probe fallback warnings.

## Solution Assessment

No, current main is not the best solution yet because cooldown and probe throttling do not coalesce repeated fallback-decision logs. The safer fix is a narrow logging/observability change, not a change to fallback selection or auth cooldown policy.
