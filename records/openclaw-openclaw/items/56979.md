## Summary

Keep open. Current main still has auth cooldown and cooldown-probe throttling, but normal `model-fallback/decision` warnings are emitted directly without duplicate coalescing, so the reported fallback-log spam remains a valid narrow logging bug.

## What This Changes

The issue asks OpenClaw to reduce repeated model-fallback decision logs after an auth-provider token expires by adding cooldown-aware log throttling or coalescing.

## Best Possible Solution

Add bounded duplicate coalescing for repeated model fallback decision warnings, keyed by stable decision/requested/candidate/reason/status/code/next/error-fingerprint fields, while emitting the first event and periodic summary/count updates and preserving structured diagnostics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a cooldowned auth profile with model fallbacks and call `runWithModelFallback` repeatedly; current main reaches `logModelFallbackDecision` on each run and has no duplicate suppression for normal fallback warnings.

## Solution Assessment

No. Current main is not yet the best solution because it throttles cooldown probes and probe console output, but not repeated non-probe fallback-decision warnings; the narrow fix belongs in the observability/logging path, not the fallback policy.
