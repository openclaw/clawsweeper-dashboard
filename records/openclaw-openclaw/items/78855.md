## Summary

Keep open: current main still intentionally persists `gateway.controlUi.allowInsecureAuth=true` for QuickStart loopback setup, while runtime policy, docs, audit, and startup warnings narrow it to a warning-class local compatibility toggle. Changing that setup default or adding setup-specific remediation is a security/product decision, not a conservative cleanup close.

## What This Changes

The issue asks OpenClaw to stop leaving `gateway.controlUi.allowInsecureAuth` enabled after onboarding or to add stronger setup/runtime warnings for that persisted security-sensitive flag.

## Best Possible Solution

Have gateway/security owners decide whether QuickStart should stop persisting the compatibility flag, add an explicit setup warning/remediation, or keep the current audit/startup warning model while preserving local Control UI usability.

## Reproduction Assessment

Yes, by source inspection: QuickStart loopback setup writes `gateway.controlUi.allowInsecureAuth=true` when unset, and current tests assert that behavior. I did not run the interactive wizard in this read-only review.

## Solution Assessment

Unclear: defaulting the flag to false, adding a setup-specific warning, or relying on existing audit/startup warnings is a security and onboarding UX tradeoff. The narrow maintainable path should be chosen by gateway/security owners and then covered by setup, audit, and auth-policy tests.
