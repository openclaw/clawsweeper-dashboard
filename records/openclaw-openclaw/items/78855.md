## Summary

Keep open: current main still intentionally writes this flag for QuickStart loopback setup, while runtime/docs/audit code narrow its effect and treat it as a warning-class compatibility toggle. Changing the default or adding a setup-specific warning is a security/product decision, not a stale cleanup close.

## What This Changes

The issue asks OpenClaw to stop leaving `gateway.controlUi.allowInsecureAuth` enabled after setup, or to add stronger setup/log warnings when it is enabled.

## Best Possible Solution

Have gateway/security maintainers decide whether QuickStart should stop persisting this compatibility flag or add a specific setup warning/remediation while preserving local Control UI usability.

## Reproduction Assessment

Yes, by source inspection: QuickStart loopback setup writes `gateway.controlUi.allowInsecureAuth: true` when unset, and current tests assert that behavior. I did not run the interactive wizard in this read-only review.

## Solution Assessment

Unclear: defaulting the flag to false, adding a specific wizard warning, or relying on startup/audit warnings is a security/UX policy tradeoff. The narrow maintainable path should be chosen by the gateway/security owners and then covered by setup and auth-policy tests.
