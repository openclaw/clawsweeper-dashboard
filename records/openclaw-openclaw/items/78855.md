## Summary

Keep open: current main and the latest release still intentionally persist the flag for QuickStart loopback setup, while runtime/docs/audit code narrow its effect to local compatibility and warning-class posture. Changing the setup default or adding setup-specific warnings needs gateway/security policy judgment, not cleanup closure.

## What This Changes

The issue requests changing or warning about the QuickStart wizard persisting `gateway.controlUi.allowInsecureAuth=true` after setup.

## Best Possible Solution

Have gateway/security maintainers choose the intended setup posture, then implement either a safer default or an explicit setup remediation/warning while preserving local Control UI usability.

## Reproduction Assessment

Yes, by source inspection: QuickStart loopback setup writes `gateway.controlUi.allowInsecureAuth=true` when unset, and current plus v2026.5.7 tests assert that behavior. I did not run the interactive wizard in this read-only review.

## Solution Assessment

Unclear: defaulting the flag to false, adding a wizard warning, or relying on existing startup/audit warnings is a security and UX policy tradeoff. The maintainable path should be selected by gateway/security owners and covered by setup and auth-policy tests.
