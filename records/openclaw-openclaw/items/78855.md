## Summary

Keep open: current main still intentionally persists `gateway.controlUi.allowInsecureAuth=true` for QuickStart loopback setup, while runtime policy, audit warnings, and docs narrow the flag to local insecure-HTTP compatibility. Changing the setup default or adding setup-specific remediation is a security/onboarding policy decision.

## What This Changes

The issue asks OpenClaw onboarding to stop leaving `gateway.controlUi.allowInsecureAuth=true` after setup without warning or remediation.

## Best Possible Solution

Gateway/security owners should decide whether QuickStart should stop persisting the compatibility flag, add explicit setup remediation, or keep the existing audit/startup warning model while preserving local Control UI usability.

## Reproduction Assessment

Yes, by source inspection: QuickStart loopback setup writes `gateway.controlUi.allowInsecureAuth=true` when unset, and current tests assert that behavior. I did not run the interactive wizard in this read-only review.

## Solution Assessment

Unclear: defaulting the flag to false, adding setup-specific warnings, or relying on existing audit/startup warnings is a security and onboarding UX tradeoff. The narrow maintainable path should be chosen by gateway/security owners and then covered by setup, audit, and auth-policy tests.
