## Summary

Keep open: current main still intentionally writes `gateway.controlUi.allowInsecureAuth=true` for QuickStart loopback setup, while the auth policy, docs, audit, and startup warning paths treat it as a warning-class local compatibility toggle. Changing that default or adding setup-time remediation is a security and onboarding product decision, not a conservative cleanup close.

## What This Changes

The issue asks OpenClaw setup to stop leaving `gateway.controlUi.allowInsecureAuth=true` enabled after onboarding, or at least warn users about the security implication.

## Best Possible Solution

Gateway/security owners should choose one policy and align setup, docs, audit, startup logging, and regression tests around it: stop persisting the flag, add explicit setup remediation, or keep the current warning model.

## Reproduction Assessment

Yes, by source inspection: QuickStart loopback setup writes `gateway.controlUi.allowInsecureAuth=true` when unset, and current tests assert that behavior. I did not run the interactive wizard in this read-only review.

## Solution Assessment

Unclear: defaulting the flag to false, adding setup-specific warnings, or relying on existing audit/startup warnings are security and onboarding UX tradeoffs. The narrow maintainable solution should come from gateway/security owner policy and then be covered by setup, audit, and auth-policy tests.
