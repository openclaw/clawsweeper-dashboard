## Summary

Keep open: current main still intentionally persists this QuickStart loopback compatibility flag, while auth-policy, docs, audit, and startup warnings narrow its effect and treat it as warning-class security posture. Changing the default or adding a setup-specific warning needs gateway/security product judgment, not cleanup closure.

## What This Changes

The issue asks OpenClaw to stop leaving `gateway.controlUi.allowInsecureAuth` enabled after setup, or to warn users clearly about the security implication.

## Best Possible Solution

Have gateway/security maintainers decide whether QuickStart should stop persisting this compatibility flag or add a setup-specific warning/remediation, then cover the chosen policy in setup, auth-policy, audit, and startup-warning tests.

## Reproduction Assessment

Yes, by source inspection: QuickStart loopback setup writes `gateway.controlUi.allowInsecureAuth: true` when unset, and current tests assert that behavior. I did not run the interactive wizard in this read-only review.

## Solution Assessment

Unclear: defaulting the flag to false, adding a setup warning, or relying on existing audit/startup warnings is a security and UX policy tradeoff. The narrow maintainable path should be chosen by gateway/security owners and then backed by focused regression coverage.
