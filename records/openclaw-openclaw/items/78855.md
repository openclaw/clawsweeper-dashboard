## Summary

Keep open: current main still intentionally persists `gateway.controlUi.allowInsecureAuth=true` for QuickStart loopback setup, and tests assert that behavior. Runtime/docs/audit evidence narrows the risk to localhost compatibility rather than the broad remote bypass described in the report, so changing the default or adding setup-specific warnings is a security/product decision rather than stale cleanup.

## What This Changes

The issue requests that setup stop leaving `gateway.controlUi.allowInsecureAuth` enabled, or at least warn users about the security implications.

## Best Possible Solution

Have gateway/security maintainers define the QuickStart policy: either stop persisting the compatibility flag or add explicit setup warning/remediation while preserving local Control UI usability.

## Reproduction Assessment

Yes, by source inspection: QuickStart loopback setup still writes `gateway.controlUi.allowInsecureAuth=true` when unset, and current tests assert that behavior. I did not run the interactive wizard in this read-only review.

## Solution Assessment

Unclear: defaulting the flag to false, adding a setup warning, or relying on existing audit/startup warnings are security and UX policy tradeoffs. The narrow maintainable path should be chosen by gateway/security owners and covered by setup plus auth-policy tests.
