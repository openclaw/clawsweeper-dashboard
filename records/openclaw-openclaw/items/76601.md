## Summary

Keep this PR open: the reported `agents add` race is still present on current `main`, but the proposed retry patch has blocking correctness issues around stale retry outcomes and same-id concurrent adds, plus the required changelog entry is missing.

## What This Changes

The PR adds retry handling for non-interactive `openclaw agents add` config commit conflicts, reports wizard conflicts with a user-facing error, and adds unit coverage for retry and retry exhaustion.

## Best Possible Solution

Keep this PR as the implementation candidate, but preserve add semantics and fresh retry outcomes before landing it with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, at source level: current `main` reads a config base hash once and `replaceConfigFile` throws `ConfigMutationConflictError` when another writer changes the hash before commit. The PR author also provided a focused unit reproduction in discussion, but I did not run commands because this review is read-only.

## Solution Assessment

No, not as proposed. Retrying the config mutation is the right narrow direction, but the retry must recheck duplicate-agent semantics and return the successful attempt's binding parse/apply outcome before it can be considered correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recheck agent identity on retry:** `src/commands/agents.commands.add.ts:199-206`
  - body: After a conflict, `buildNextConfig` applies the requested agent to the refreshed config without repeating the existing-agent guard. If the concurrent writer already added the same normalized agent id, `applyAgentConfig` updates that entry, so this command can overwrite another add instead of reporting `Agent "..." already exists.`.
  - confidence: 0.9
- **[P2] Return retry binding outcomes:** `src/commands/agents.commands.add.ts:213-217`
  - body: The retry path recomputes bindings against the fresh config but returns only the config. After a conflict, the JSON payload, conflict warning, and bootstrap defaults still use the initial `bindingResult`, so a route claimed during the retry window can be committed as skipped/conflicted while the command reports it as added.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/commands/agents.commands.add.ts:54-55`
  - body: This is a user-facing fix for `openclaw agents add`, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fixes to be recorded before merge.
  - confidence: 0.83
