## Summary

Keep open. The PR targets a real current-main `agents add` config-write race, but the proposed retry path is not merge-ready because it keeps stale duplicate/binding state after a retry and the new test fixtures do not use the real agent-list schema.

## What This Changes

This PR adds retry handling around `openclaw agents add` config commits and adds unit coverage for `ConfigMutationConflictError` retries in non-interactive mode.

## Best Possible Solution

Land a narrow retry implementation that revalidates duplicate-agent and binding outcomes on the winning fresh snapshot, returns that committed state to output/bootstrap, tests `agents.list` preservation and conflicts, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: parallel `openclaw agents add` calls can race between the initial snapshot/base hash read and the later base-hash commit, where `replaceConfigFile` raises `ConfigMutationConflictError`.

## Solution Assessment

No. Retrying the non-interactive commit is the right boundary, but the PR must re-check fresh-snapshot duplicate and binding state instead of reporting the initial result after a conflict retry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Revalidate the retried agent add state:** `src/commands/agents.commands.add.ts:199-217`
  - body: The retry callback reapplies the add to a fresh config, but duplicate-agent and binding outcomes still come from the initial snapshot. If another concurrent add creates the same agent or claims the requested binding before this retry, `applyAgentConfig` updates that entry and `applyAgentBindings` records a conflict, yet the command can still exit successfully and report the initial binding additions. Return the winning attempt's binding/conflict state and re-run duplicate validation on the fresh snapshot.
  - confidence: 0.87
- **[P3] Use the real agents.list schema in retry tests:** `src/commands/agents.add.test.ts:181-182`
  - body: These fixtures put the concurrent agent under `agents.entries`, but the command reads and writes `agents.list`. As written, the test would still pass if the retry dropped the concurrently added agent, so it does not protect the reported race. Use `agents.list` and assert the second commit's `nextConfig` preserves both agents.
  - confidence: 0.91
- **[P3] Add the required changelog entry:** `src/commands/agents.commands.add.ts:244-248`
  - body: This changes user-visible `openclaw agents add` behavior for a reported config race. Repo policy requires user-facing fixes to be recorded in `CHANGELOG.md`, so add a single-line fix entry before merge.
  - confidence: 0.82
