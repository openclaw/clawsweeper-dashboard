## Summary

Keep open: the race is source-reproducible and retrying config conflicts is the right direction, but the current patch can report stale binding results after a retry, can retry from stale state in fresh-conflict cases, lacks the required changelog entry, and still lacks inspectable after-fix real CLI proof.

## What This Changes

This PR adds retry handling around `openclaw agents add` config commits on `ConfigMutationConflictError`, handles wizard conflicts, and adds unit tests for retry and retry exhaustion.

## Best Possible Solution

Keep the retry direction, but make retries operate on the fresh valid snapshot, re-run duplicate and binding checks, return the successful attempt outcome, add the changelog entry, and attach real parallel CLI proof.

## Reproduction Assessment

Yes, at source level. Current `main` reads a config base hash before the `agents add` mutation and the config writer throws `ConfigMutationConflictError` when another writer changes that hash before commit; I did not run a live race because this review is read-only.

## Solution Assessment

No, not yet. Retrying `ConfigMutationConflictError` is the narrow maintainable direction, but this patch must preserve fresh-attempt results and avoid retrying from stale or invalid snapshots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the retry attempt outcome:** `src/commands/agents.commands.add.ts:236-240`
  - body: After a conflict, `buildNextConfig` recomputes bindings against the fresh config, but the caller ignores that outcome and keeps using the pre-retry `bindingResult` for JSON/text output and bootstrap flags. If another process claims a requested route before the retry succeeds, this command can report the binding as added even though the committed config skipped it; return and use the successful attempt's binding result.
  - confidence: 0.9
- **[P2] Re-check existing agents on retry:** `src/commands/agents.commands.add.ts:191-198`
  - body: The retry transform applies `applyAgentConfig` to the fresh config without repeating the `already exists` guard. If a concurrent `agents add` creates the same normalized ID, this retry updates that existing agent's workspace/model instead of failing, so one add command can overwrite another command's agent.
  - confidence: 0.84
- **[P2] Abort retries when the fresh snapshot is invalid:** `src/commands/agents.commands.add.ts:86-90`
  - body: On conflict, this code records the invalid file's hash but keeps the previous config when `freshSnapshot.valid` is false. The next attempt can then pass the new base hash and overwrite the invalid current file with stale data; surface the invalid config instead of retrying from the old snapshot.
  - confidence: 0.78
- **[P3] Add the changelog entry:** `src/commands/agents.commands.add.ts:55-56`
  - body: This is a user-facing fix for `openclaw agents add`, but the PR changes only the command and tests. OpenClaw policy requires user-facing fixes to be recorded in `CHANGELOG.md`.
  - confidence: 0.82
