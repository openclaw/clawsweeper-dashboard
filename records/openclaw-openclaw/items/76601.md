## Summary

Keep this PR open: the retry direction is valid, but the patch is not merge-ready because the retry path can commit refreshed bindings while reporting the initial binding outcome, and the user-facing fix is missing a changelog entry.

## What This Changes

The PR adds retry handling around non-interactive `openclaw agents add` config commits, catches interactive wizard config conflicts, and adds unit tests for retry and retry exhaustion.

## Best Possible Solution

Keep the PR open and update it so the successful retry attempt returns the final binding parse/application outcome, adds focused regression coverage, records the user-facing fix in `CHANGELOG.md`, and treats #43367 as only partially addressed.

## Reproduction Assessment

Yes, at source level: current main reads a config `baseHash` before mutating `agents add` state, and `replaceConfigFile` throws `ConfigMutationConflictError` when another process changes the config before commit. I did not run a live parallel CLI reproduction because this review had to stay read-only.

## Solution Assessment

No, not as currently proposed. Retrying the config mutation is the narrow maintainable direction, but the implementation must preserve the successful retry attempt's binding result and refreshed parse errors instead of using stale pre-retry state.
