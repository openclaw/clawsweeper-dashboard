## Summary

Keep this PR open. It targets a real current-main race in `openclaw agents add`, but the proposed retry path can commit against a refreshed config while still reporting binding results computed from the stale pre-retry snapshot.

## What This Changes

This PR adds retry handling around non-interactive `openclaw agents add` config commits, catches interactive wizard config conflicts, and adds unit coverage for retry and retry exhaustion.

## Best Possible Solution

Keep the PR open, make the retry helper return the successful attempt's final binding outcome and parse errors, add a regression test for a binding conflict that appears only after refresh, and include the required changelog entry.

## Reproduction Assessment

Yes, at source level: current main reads a config base hash before the `agents add` mutation and `replaceConfigFile` throws on a changed base hash. I did not run a live parallel CLI repro because the checkout was required to stay read-only.

## Solution Assessment

No, not as proposed. Retrying the config mutation is the right narrow direction, but the PR must preserve the binding result and error semantics from the successful retry attempt instead of reporting stale pre-retry state.

## Repair Work Prompt

On PR #76601, keep the fix scoped to `src/commands/agents.commands.add.ts`, `src/commands/agents.add.test.ts`, and `CHANGELOG.md`. Update the non-interactive retry helper so each attempt returns both the committed config and the binding parse/apply outcome from the config snapshot that actually committed; do not report stale `added`, `updated`, `skipped`, or `conflicts` from the initial attempt, and fail rather than silently dropping bindings if parsing fails after a refresh. Add a regression test where the first commit raises `ConfigMutationConflictError` and the refreshed config already has the requested binding claimed by another agent; assert the command reports the final conflict/no-added-binding behavior. Add a single-line user-facing changelog entry for the `agents add` concurrency fix. Do not broaden this PR into the session-lock, OAuth, or detached-work portions of #43367.
