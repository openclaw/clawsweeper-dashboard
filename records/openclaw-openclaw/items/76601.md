## Summary

Keep open: the PR addresses a valid source-visible `agents add` config race, but it is not merge-ready because the retry path can commit refreshed binding results while still reporting and bootstrapping from the stale pre-retry result, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR adds retry handling around non-interactive `openclaw agents add` config commits, catches interactive wizard config conflicts, and adds unit coverage for retry and retry exhaustion.

## Best Possible Solution

Keep the narrow retry direction, return the successful retry attempt's full binding/parse outcome, add regression coverage and a changelog entry, and leave the broader #43367 session lifecycle work out of this patch.

## Reproduction Assessment

Yes, at source level: current main loads one config snapshot, writes with `baseHash`, and `replaceConfigFile` throws `ConfigMutationConflictError` if another process changed the file first. I did not run a live parallel CLI repro or tests because this review had to stay read-only.

## Solution Assessment

No, not as proposed. Retrying the config mutation is the right narrow fix, but the implementation must use the successful retry attempt's final binding result and parse errors instead of stale pre-retry state.

## Review Findings
