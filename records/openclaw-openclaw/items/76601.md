## Summary

Keep this PR open: it targets a valid source-reproducible `openclaw agents add` config race, but the proposed retry path is not merge-ready because it can commit refreshed bindings while reporting the stale pre-retry binding outcome and it lacks the required changelog entry.

## What This Changes

The PR adds retry handling for `ConfigMutationConflictError` in `agents add`, catches interactive wizard conflicts, and adds unit tests for retry and retry exhaustion.

## Best Possible Solution

Keep the retry approach, but make the successful retry return the final binding parse/result alongside the committed config, use that fresh result for output and bootstrap decisions, and add the required changelog entry.

## Reproduction Assessment

Yes, at source level: current `main` reads a config `baseHash`, later commits with that stale hash, and `replaceConfigFile` throws `ConfigMutationConflictError` if another process has written first. The linked issue and PR comments also provide a concrete parallel `openclaw agents add` scenario.

## Solution Assessment

No, not as proposed. Retrying the mutation is the right narrow direction, but the implementation must preserve the binding and parse outcome from the successful retry attempt rather than reporting stale pre-retry state.
