## Summary

Keep this open: current main still lets the store-targeted `/allowlist add|remove --store` path skip the shared `configWrites` authorization gate before mutating persistent pairing allowlist state, and the focused closing PR #72361 remains the right review path.

## What This Changes

The issue asks for `/allowlist add|remove --store` to honor channel/account `configWrites=false` before mutating the persistent pairing allowlist store.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that applies the existing config-write authorization before any pairing-store allowlist add/remove, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: with a pairing-capable channel, `commands.config=true`, and `configWrites=false`, `/allowlist add dm --store <id>` skips the authorization branch and reaches the store mutation.

## Solution Assessment

Yes. Reusing the existing `configWrites` authorization before store-targeted allowlist mutations is the narrowest maintainable fix; a separate store-write permission would require explicit schema and docs work.
