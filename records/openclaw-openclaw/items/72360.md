## Summary

Keep this open. Current main still has the reported store-write authorization gap, the docs still promise `configWrites` coverage for `/allowlist add|remove`, and the linked focused fix PR remains open and unmerged.

## What This Changes

Ensure `/allowlist add|remove --store` obeys channel/account `configWrites=false` before mutating persistent pairing allowlist state.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that runs the existing config-write authorization before every pairing-store allowlist add/remove, with regression tests for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches the store mutation call.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable fix; a separate permission model would need explicit schema and docs changes.
