## Summary

Keep this open. Current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` authorization branch before mutating the persistent pairing allowlist store, and the focused linked PR remains open and unmerged.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate persistent pairing allowlist state even when the relevant channel or account has `configWrites=false`.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that runs the existing config-write authorization before any pairing-store allowlist add/remove, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips the `resolveConfigWriteDeniedText(...)` branch, and then reaches the persistent pairing-store mutation path.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable fix; a separate permission model would need explicit schema, docs, and product review.
