## What This Changes

The issue requests applying channel `configWrites` authorization to `/allowlist --store` pairing-store mutations.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/72361 after preserving true command-origin authorization separately from the selected allowlist target, or apply an equivalent narrow fix with regression coverage for store writes.

## Reproduction Assessment

Yes. Source inspection on current main shows `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`; no live mutation was run because this review is read-only.

## Solution Assessment

Yes for the issue direction. Reusing the existing `configWrites` gate before any pairing-store mutation is the narrowest maintainable fix, provided the implementation keeps command origin separate from the selected target channel/account.
