## Summary

Keep this open: current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` authorization gate before mutating persistent pairing allowlist state, and the focused closing PR #72361 is still open and unmerged.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist store on pairing-capable channels even when channel or account `configWrites=false`.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that authorizes config writes before any pairing-store allowlist mutation and adds regression coverage for channel/account `configWrites=false` `--store` paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: with a pairing-capable channel, `commands.config=true`, and `configWrites=false`, `/allowlist add dm --store <id>` skips the config-write branch and reaches the store mutation.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable fix; a separate store-write permission would require explicit schema and docs work.
