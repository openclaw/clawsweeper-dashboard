## Summary

Keep open: current main still has the reported store-write authorization gap, the public docs still promise `configWrites` coverage for `/allowlist add|remove`, and the linked focused fix PR is open and unmerged.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist even when the channel or account has `configWrites=false`.

## Best Possible Solution

Land PR #72361 or an equivalent narrow patch that authorizes every pairing-store allowlist add/remove against the same source and target `configWrites` policy before mutation, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches the pairing-store mutation call on current main.

## Solution Assessment

Yes. The narrow maintainable fix is to run the existing config-write authorization before store mutation, using the real source channel/account as origin and the edited channel/account as target; a separate permission would need explicit schema and docs if maintainers intentionally want different semantics.
