## Summary

Keep open: current main still lets `/allowlist add|remove --store` skip the documented `configWrites` gate before mutating persistent pairing allowlist state, and the linked closing PR #72361 is open and unmerged.

## What This Changes

Ensure store-targeted `/allowlist add|remove --store` obeys channel/account `configWrites` before mutating the pairing allowlist store.

## Best Possible Solution

Land PR #72361 or an equivalent narrow patch that applies the existing `configWrites` authorization before any pairing-store allowlist add/remove, with regression coverage for channel-level and account-level disabled writes.

## Reproduction Assessment

Yes. Source inspection on current main shows `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`; no live store mutation was run because this review is read-only.

## Solution Assessment

Yes. Reusing the existing `configWrites` gate before store mutation is the narrowest maintainable fix; a separate store-write permission would be a new config/product decision.
