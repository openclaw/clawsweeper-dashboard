## Summary

Keep this issue open: current main still lets store-targeted `/allowlist add|remove --store` skip the `configWrites` authorization branch before mutating persistent pairing allowlist state, and the focused closing PR remains open and unmerged.

## What This Changes

Apply channel/account `configWrites` authorization to store-targeted `/allowlist add|remove` mutations before writing the pairing allowlist store.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that checks config-write authorization before any pairing-store allowlist add/remove, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable fix; a separate permission model would need explicit schema and docs work.
