## Summary

Keep this issue open: current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` authorization gate before mutating persistent pairing allowlist state, and the focused closing PR #72361 is still open and unmerged.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate pairing allowlist storage even when the relevant channel or account has `configWrites=false`.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that runs config-write authorization before any pairing-store allowlist add/remove, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable fix; a separate permission model would require explicit schema and docs work.
