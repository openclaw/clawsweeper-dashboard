## Summary

Keep open: current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` gate before mutating persistent pairing allowlist state, and the linked fix PR is still open.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist store without honoring a channel-level `configWrites=false` policy.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/72361 after correcting any source-vs-target authorization gap, or apply an equivalent narrow patch with regression coverage for store writes.

## Reproduction Assessment

Yes. Source inspection on current main shows `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`; no live store mutation was run because this review is read-only.

## Solution Assessment

Yes for the issue direction. Reusing the existing `configWrites` gate before any pairing-store mutation is the narrowest maintainable fix, provided the implementation keeps the command origin separate from the selected target channel/account.
