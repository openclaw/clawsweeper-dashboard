## Summary

Keep open: current main still lets store-targeted `/allowlist add|remove --store` skip the documented `configWrites` gate before mutating persistent pairing allowlist state, and the linked fix PR https://github.com/openclaw/openclaw/pull/72361 is still open.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the pairing allowlist store even when the originating channel has `configWrites=false`.

## Best Possible Solution

Apply a narrow guard that authorizes the originating channel/account and selected target scope before any config or pairing-store allowlist write, with same-channel and cross-channel regression coverage.

## Reproduction Assessment

Yes. On current main, `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips the only `resolveConfigWriteDeniedText(...)` call, and reaches `updatePairingStoreAllowlist(...)`; I did not run live mutation because the review is read-only.

## Solution Assessment

Yes for the issue direction. Reusing the existing `configWrites` policy for store writes is the narrowest maintainable fix, provided the implementation keeps the true command origin separate from the selected allowlist target.
