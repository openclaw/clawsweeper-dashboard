## Summary

Keep this issue open. Current main still lets `/allowlist add|remove --store` skip the `resolveConfigWriteDeniedText(...)` authorization branch and mutate the persistent pairing allowlist store, while the focused closing PR #72361 is still open and unmerged.

## What This Changes

The issue asks for store-targeted `/allowlist add|remove` mutations to honor channel and account `configWrites` policy before updating the persistent pairing allowlist store.

## Best Possible Solution

Land PR #72361 or an equivalent narrow patch that applies the existing config-write authorization to every pairing-store allowlist add/remove before mutation, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence on current main: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches the pairing-store mutation call.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing config-write policy for store mutations; a separate store-write permission would need an explicit schema and docs decision first.
