## Summary

Keep this open. Current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` authorization gate, and the focused closing PR #72361 is still open and unmerged, so this issue should remain tracked for maintainer/security review.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist store even when the channel or account has `configWrites=false`.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that runs the existing config-write authorization before any pairing-store allowlist add/remove, with regression tests for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, then reaches the pairing-store mutation path.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable solution; a separate permission model would require explicit schema and docs work.
