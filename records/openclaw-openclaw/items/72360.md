## Summary

Keep this issue open: current main still routes `/allowlist add|remove --store` around the shared `configWrites` authorization branch before mutating persistent pairing allowlist state, and the focused closing PR remains open and unmerged.

## What This Changes

The issue reports that store-targeted `/allowlist add|remove` can mutate the persistent pairing allowlist without honoring channel or account `configWrites` policy.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that applies config-write authorization before any pairing-store allowlist mutation, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: configure a pairing channel with `configWrites=false`, send `/allowlist add dm --store <id>` from an authorized sender, and the handler skips `resolveConfigWriteDeniedText(...)` before calling the store update helper.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable fix; a separate permission model would need explicit schema and docs work.
