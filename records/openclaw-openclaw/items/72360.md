## Summary

Keep open: current main still lets store-targeted `/allowlist add|remove --store` skip the documented `configWrites` gate before mutating persistent pairing allowlist state, and the linked closing PR remains open with a prior security review finding.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist store without honoring the channel `configWrites=false` policy.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/72361 after preserving true command-origin authorization separately from the selected allowlist target, or apply an equivalent narrow fix with regression coverage for store writes.

## Reproduction Assessment

Yes. Source inspection on current main shows `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`; no live mutation was run because this review is read-only.

## Solution Assessment

Yes for the requested remediation. Reusing the existing `configWrites` authorization before any pairing-store mutation is the narrowest maintainable fix, provided the implementation distinguishes the command origin from the selected target channel/account.
