- `P0`: The report describes a bypass of an operator-configured access-control boundary for persistent allowlist writes.
- `impact:security`: The issue affects authorization and persistent pairing allowlist state that controls future DM command access.

## Summary

Keep open: current main and latest release v2026.5.18 still let store-targeted `/allowlist add|remove --store` skip the documented `configWrites` authorization gate before mutating persistent pairing allowlist state, and the linked fix PR https://github.com/openclaw/openclaw/pull/72361 is still open with review/proof blockers.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist even when the initiating channel has `configWrites=false`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/72361 after fixing command-origin versus selected-target authorization and adding regression plus real behavior proof, or apply an equivalent narrow security fix.

## Reproduction Assessment

Yes. Source inspection shows `/allowlist ... --store` sets `shouldUpdateConfig=false`, skips the only `resolveConfigWriteDeniedText(...)` call, and then calls `updatePairingStoreAllowlist(...)` on current main.

## Solution Assessment

Yes for the issue direction: store mutations should reuse the existing `configWrites` authorization boundary. The linked PR should still preserve true command origin separately from the selected store target before merge.
