## Summary

Keep this issue open. Current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` authorization gate before mutating the persistent pairing allowlist store, and the focused closing PR #72361 remains open and unmerged.

## What This Changes

The issue asks OpenClaw to enforce the documented channel/account `configWrites` policy before `/allowlist add|remove --store` mutates persistent pairing allowlist state.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that runs the existing config-write authorization before any pairing-store allowlist add/remove, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and then reaches the pairing-store mutation path.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable solution; a separate permission model would require explicit schema and docs work.
