## Summary

Keep this open: current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` gate before mutating pairing allowlist state, and the focused closing PR #72361 is still open and unmerged.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist store without honoring channel or account `configWrites=false`.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that runs config-write authorization before any pairing-store allowlist add/remove, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: with a pairing-capable channel and `configWrites=false`, `/allowlist add dm --store <id>` skips the config-write authorization branch and reaches the store mutation.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable fix; a separate store-write permission would require explicit schema and docs work.
