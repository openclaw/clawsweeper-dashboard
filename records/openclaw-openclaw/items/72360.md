## Summary

Keep this issue open. Current main still has the reported store-only authorization gap, the docs still state that `/allowlist add|remove` honors channel `configWrites`, and the linked focused fix PR is open and unmerged, so the issue should remain open until that PR or an equivalent maintainer-reviewed fix lands.

## What This Changes

Ensure `/allowlist add|remove --store` mutations honor channel and account `configWrites` policy before updating the persistent pairing allowlist store.

## Best Possible Solution

Land PR #72361 or an equivalent narrow patch that runs the existing config-write authorization before every pairing-store allowlist mutation, with regression coverage for channel-level and account-level `configWrites=false` store writes.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence on current main: an owner-authorized `/allowlist add dm --store <id>` on a pairing-capable channel sets `shouldUpdateConfig` false, skips the `resolveConfigWriteDeniedText(...)` branch, and reaches the pairing-store mutation call.

## Solution Assessment

Yes. Reusing the existing `configWrites` authorization before store mutation is the narrowest maintainable fix; a separate permission model would require explicit schema and docs work if maintainers intentionally want store writes to differ from config writes.
