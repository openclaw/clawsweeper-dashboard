## Summary

Keep this issue open. Current main still has the reported `/allowlist --store` bypass: store-only edits skip the existing `configWrites` authorization check and then mutate the persistent pairing allowlist store, while the linked focused fix PR is open and unmerged.

## What This Changes

The issue asks OpenClaw to enforce channel/account `configWrites` policy on `/allowlist add|remove --store` pairing-store mutations.

## Best Possible Solution

Review and land PR #72361 or an equivalent narrow patch that runs the same config-write authorization before every pairing-store allowlist mutation, with regression coverage for store add and store remove when channel or account `configWrites=false`.

## Reproduction Assessment

Yes. The current source path is enough for high-confidence reproduction: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches the pairing-store mutation call.

## Solution Assessment

Yes. Reusing `resolveConfigWriteDeniedText` or the underlying `authorizeConfigWrite` policy before store mutation is the narrowest maintainable fix unless maintainers intentionally add and document a separate permission for pairing-store edits.
