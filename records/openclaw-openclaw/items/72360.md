## Summary

Keep open: current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` authorization before mutating persistent pairing allowlist state, and the linked closing pull request at https://github.com/openclaw/openclaw/pull/72361 is still open and unmerged.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist store even when the originating channel has `configWrites=false`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/72361 or an equivalent narrow patch that applies the existing `configWrites` authorization before every pairing-store allowlist add/remove, with regression coverage for same-channel, account-level, and cross-channel source-vs-target cases.

## Reproduction Assessment

Yes. Source inspection on current main shows `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`.

## Solution Assessment

Yes. Reusing the existing `configWrites` gate before store mutation is the narrowest maintainable fix; the implementation should keep the command origin separate from any selected allowlist target channel/account.
