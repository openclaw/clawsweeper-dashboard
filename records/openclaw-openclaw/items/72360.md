## Summary

Keep open: current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` authorization gate before mutating persistent pairing allowlist state, and the linked closing PR is still open.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist even when the originating channel has `configWrites=false`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/72361 after fixing true command-origin versus selected-target authorization and adding regression plus real behavior proof, or apply an equivalent narrow fix.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path. On current main, `/allowlist --store` sets `shouldUpdateConfig=false`, skips `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`.

## Solution Assessment

Yes for the issue direction: store mutations should reuse the existing `configWrites` authorization boundary. The fix should keep the true command origin separate from the selected target channel/account.
