## Summary

Keep open: current main still lets store-targeted `/allowlist add|remove --store` skip the shared `configWrites` check before mutating persistent pairing allowlist state, and #72361 is open and unmerged.

## What This Changes

This issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist even when the originating channel has `configWrites=false`.

## Best Possible Solution

Land #72361 or an equivalent narrow patch that applies the existing `configWrites` authorization before any pairing-store allowlist add/remove, with regression coverage for channel-level and account-level disabled writes.

## Reproduction Assessment

Yes. Source inspection on current main shows `--store` sets `shouldUpdateConfig` false, skips the branch containing `resolveConfigWriteDeniedText(...)`, then calls `updatePairingStoreAllowlist(...)`; no live mutation was run because this review is read-only.

## Solution Assessment

Yes. Reusing the existing `configWrites` gate before store mutation is the narrowest maintainable fix; a separate store-write permission would be a new config/product decision.
