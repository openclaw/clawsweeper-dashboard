## Summary

Keep open: current main still contains the reported security-relevant bypass, and the focused closing PR #72361 is still open and unmerged.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate persistent pairing allowlist entries without honoring channel or account `configWrites=false`.

## Best Possible Solution

Land PR #72361 or an equivalent narrow security patch that applies the existing `configWrites` authorization before any pairing-store allowlist add/remove, with regression coverage for channel-level and account-level disabled writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: with a pairing-capable channel and `configWrites=false`, `/allowlist add dm --store <id>` skips `resolveConfigWriteDeniedText(...)` and reaches `updatePairingStoreAllowlist(...)`.

## Solution Assessment

Yes. Reusing the existing config-write authorization for store-targeted allowlist mutations is the narrowest maintainable fix; a separate store-write permission would need explicit schema, docs, and product review.
