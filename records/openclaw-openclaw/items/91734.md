## Summary

Keep open: current main and the latest release still omit `--file` from `openclaw infer image generate`, while the shared runner and provider contract already have a file-backed `inputImages` path. Two open PRs use closing syntax for this issue, so the issue should stay open until maintainers land one canonical implementation or close the PRs.

## What This Changes

Add repeatable input-file support to `openclaw infer image generate` so CLI image generation can use reference/input images like the existing image edit flow.

## Best Possible Solution

Review and land one focused PR that adds repeatable `--file` to `image generate`, forwards through the existing `inputImages` runtime path, updates capability metadata/docs, and preserves explicit unsupported-provider errors.

## Reproduction Assessment

Yes. Source inspection shows current `image.generate` has no `--file` option while the shared runner and `image.edit` path already support file-backed `inputImages`; I did not run the CLI because this was a read-only review.

## Solution Assessment

Yes for the issue direction. The narrow maintainable solution is an additive CLI option that reuses the existing file-loading/input-image provider path, with the remaining maintainer choice being which open PR shape should become canonical.
