## Summary

Keep open: current `main` and the latest release still omit `--file` from `openclaw infer image generate`, while the shared image-generation runner and provider contract already carry `inputImages`. The two linked implementation PRs are both closed unmerged, and live search found no open fixing PR.

## What This Changes

Add repeatable input-file support to `openclaw infer image generate` so provided files are passed as image/reference inputs to capable providers.

## Best Possible Solution

Expose repeatable `--file` on the existing image generate CLI, forward it through the current `runImageGenerate` file/inputImages path, and keep provider-specific support or rejection in the existing image-generation provider contracts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `image generate` lacks the `--file` option and does not pass `file` to `runImageGenerate`, while `image edit` and the shared runner already support file-backed `inputImages`.

## Solution Assessment

Yes. The narrow maintainable fix is to add a repeatable `--file` option to `image.generate`, forward it into the existing runner, and cover capability metadata, forwarding, and unsupported-provider behavior without adding provider-specific CLI policy.
