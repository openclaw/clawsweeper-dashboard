## Summary

Keep open. Current main still installs the unscoped `openshell@0.1.0` package in the OpenShell plugin, the default resolver still prefers that package's bin over the PATH CLI, and the published npm package is a Telegram bridge that throws when `TELEGRAM_BOT_TOKEN` is absent; the related open PR is incomplete and does not resolve the runtime path.

## What This Changes

The issue reports that the OpenShell sandbox backend can resolve the unscoped npm `openshell` package, whose Telegram bot startup requires `TELEGRAM_BOT_TOKEN`, instead of the standalone OpenShell CLI.

## Best Possible Solution

Remove or stop preferring the unscoped npm package for the default OpenShell command, make the default use the documented standalone PATH CLI or a verified scoped package, and update the manifest, lockfile, resolver tests, docs, and packaged-install proof together.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: default `command: "openshell"` resolves the plugin-local npm package first, and that package's published startup code throws when `TELEGRAM_BOT_TOKEN` is absent.

## Solution Assessment

Yes. The narrow maintainable fix is to remove the unintended executable from the runtime path or change resolver precedence so the documented standalone CLI is used; the open PR's dependency-only deletion is not enough by itself.
