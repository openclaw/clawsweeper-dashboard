## Summary

Keep open. Current main and the latest stable release still declare and prefer the unscoped `openshell@0.1.0` package for the default OpenShell command, and the published package source confirms it is a Telegram bridge that throws when `TELEGRAM_BOT_TOKEN` is absent.

## What This Changes

The issue asks OpenClaw to stop bundled OpenShell sandbox resolution from launching an unrelated `openshell@0.1.0` Telegram bot package when the standalone OpenShell CLI is intended.

## Best Possible Solution

Stop resolving the unrelated unscoped npm package for the default OpenShell command, use the documented standalone PATH CLI or a verified upstream/scoped package, and update manifest, lockfile, resolver tests, docs, and packaged-install proof together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: the default `command: "openshell"` resolves the package-local `openshell@0.1.0` binary first, and that package throws when `TELEGRAM_BOT_TOKEN` is missing.

## Solution Assessment

Yes, the requested direction is the right fix boundary. The maintainable solution is to remove or stop preferring the unintended executable package while preserving the documented standalone CLI path and adding package/runtime regression proof.
