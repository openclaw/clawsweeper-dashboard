## Summary

Keep open. Current main and the latest stable release still declare and prefer the unscoped `openshell@0.1.0` executable package, and that published package is a Telegram bridge whose startup path requires `TELEGRAM_BOT_TOKEN`; the discussion also includes a later stable-release reproduction and a working explicit-command workaround.

## What This Changes

The issue reports that the OpenShell sandbox backend can execute the bundled unscoped `openshell@0.1.0` npm package, which requires `TELEGRAM_BOT_TOKEN`, instead of the standalone OpenShell CLI for sandboxed exec.

## Best Possible Solution

Stop resolving the unintended unscoped npm package for the default OpenShell command, use the documented standalone PATH CLI or a verified upstream/scoped package, and update the manifest, lockfile, resolver tests, docs, and packaged-install proof together.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: the default `command: "openshell"` resolves the package-local `openshell@0.1.0` binary before PATH, and that package throws when `TELEGRAM_BOT_TOKEN` is missing.

## Solution Assessment

Yes. The best fix boundary is to remove or stop preferring the unintended executable package while preserving the documented standalone CLI path and adding install/runtime regression proof; the related one-file PR is not sufficient by itself.
