## Summary

Keep open. Current main and the latest release still declare the unscoped `openshell@0.1.0` dependency, the OpenShell resolver still prefers that package-local binary for the default `openshell` command, and the dependency source confirms it is a Telegram bridge that requires `TELEGRAM_BOT_TOKEN`; the issue comments also include a later stable-release repro and a working command override workaround.

## What This Changes

The report asks the OpenShell sandbox backend to stop launching the bundled unscoped npm package and instead use the standalone OpenShell CLI or a verified package so sandbox exec works without Telegram credentials.

## Best Possible Solution

Stop resolving the unrelated unscoped npm package for the default OpenShell command, use the documented standalone PATH CLI or a verified upstream/scoped package, and update the manifest, lockfile, resolver tests, docs, and packaged-install proof together.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: the default `command: "openshell"` resolves the package-local `openshell@0.1.0` binary first, and that package throws when `TELEGRAM_BOT_TOKEN` is missing.

## Solution Assessment

Yes, the best fix boundary is clear. The maintainable solution is to remove or stop preferring the unintended executable package while preserving the documented standalone CLI path and adding install/runtime regression proof.
