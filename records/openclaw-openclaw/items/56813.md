## Label Justifications

- `P1`: The issue breaks sandboxed exec for real OpenShell users and has a high-confidence source-level root cause affecting current main and the latest release.
- `impact:security`: The affected path is a sandbox command-execution boundary that can launch an unintended npm package instead of the intended OpenShell CLI.

## Summary

Keep open. Current main and the latest release still declare the unintended unscoped `openshell@0.1.0` dependency, the OpenShell plugin still resolves that package-local binary before PATH for the default command, and the published package is a Telegram bridge whose startup config requires `TELEGRAM_BOT_TOKEN`.

## What This Changes

The issue asks OpenClaw’s OpenShell sandbox backend to stop launching an unrelated `openshell@0.1.0` Telegram package when users configure the standalone OpenShell CLI.

## Best Possible Solution

Stop resolving the unrelated unscoped npm package for the default OpenShell command, prefer the documented standalone PATH CLI or a verified upstream/scoped package, and update the manifest, lockfile, resolver tests, docs, and packaged-install proof together.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: the default `command: "openshell"` resolves the package-local `openshell@0.1.0` binary first, and that package throws when `TELEGRAM_BOT_TOKEN` is missing; the May 6 stable-release comment also reports a live repro and config-override workaround.

## Solution Assessment

Yes, the fix boundary is clear. The maintainable solution is to remove or stop preferring the unintended executable package while preserving the documented standalone CLI path and adding install/runtime regression proof; the related manifest-only PR is not sufficient.
