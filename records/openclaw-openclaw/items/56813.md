## Summary

Keep open. Current main still declares unscoped `openshell@0.1.0`, resolves the bundled package before the PATH CLI, and the npm artifact is a Telegram bridge that throws when `TELEGRAM_BOT_TOKEN` is absent; the related open PR does not yet fix the current package graph.

## What This Changes

Stop the OpenShell sandbox backend from executing the unrelated unscoped `openshell` npm package when the standalone OpenShell CLI should be used.

## Best Possible Solution

Remove unscoped `openshell@0.1.0` from the root and plugin package graph, make default `command: "openshell"` use the standalone PATH CLI unless a verified package is intentionally configured, update the resolver tests, and validate the packaged dependency contract plus OpenShell smoke path.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with default `command: "openshell"`, current main resolves the bundled npm package first, and that package throws before startup when `TELEGRAM_BOT_TOKEN` is absent.

## Solution Assessment

Yes. The narrow maintainable fix is to stop bundling or preferring the unrelated npm package and keep the documented standalone CLI resolution path, with package/lockfile tests covering the dependency contract.
