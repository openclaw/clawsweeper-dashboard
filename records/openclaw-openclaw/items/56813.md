## Summary

Keep open. Current `main` still declares and resolves unscoped `openshell@0.1.0` ahead of the PATH CLI, and the npm package source is a Telegram bridge that throws when `TELEGRAM_BOT_TOKEN` is missing.

## What This Changes

The issue requests that OpenClaw stop loading the unrelated bundled `openshell` npm package for the OpenShell sandbox backend and use the standalone OpenShell CLI instead.

## Best Possible Solution

Remove the unscoped `openshell@0.1.0` dependency from the root/plugin package graph, update the resolver and tests so default `command: "openshell"` uses the PATH CLI, and add a changelog entry plus package-contract validation.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: enable the OpenShell backend with the default `command: "openshell"`, let the resolver find the bundled npm package, and that package's startup path throws without `TELEGRAM_BOT_TOKEN`.

## Solution Assessment

Yes. The reporter's suggested direction is the narrow maintainable fix: stop bundling the unrelated npm package and keep the documented standalone CLI path unless a verified scoped OpenShell CLI package exists later.
