## Summary

Keep open. Current main still matches the reported OpenShell name-collision path: the bundled OpenShell plugin declares and locks unscoped `openshell@0.1.0`, the resolver prefers a bundled `openshell` package before the PATH command for the default config, and the upstream npm package is a Telegram bridge bot that throws when `TELEGRAM_BOT_TOKEN` is absent.

## What This Changes

The issue reports that OpenClaw’s bundled OpenShell integration installs an unrelated unscoped `openshell` npm package that requires `TELEGRAM_BOT_TOKEN`, causing sandboxed `exec` calls to fail when the standalone OpenShell CLI should be used instead.

## Best Possible Solution

Remove the unscoped `openshell@0.1.0` runtime dependency, update lockfile and package-manifest contracts, and make the default `command: "openshell"` resolve to the PATH CLI. Only add a packaged OpenShell CLI later if it is verified, scoped, documented, and intentionally supported.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: enable the OpenShell plugin with default `command: "openshell"`, let the resolver find the bundled `openshell` package, and the upstream npm package's config throws without `TELEGRAM_BOT_TOKEN`.

## Solution Assessment

Yes, the reporter's suggested direction is the narrow maintainable fix: stop bundling the unrelated npm package and preserve the documented standalone CLI path. Replacing it with another package should wait until there is a verified OpenShell CLI package contract.
