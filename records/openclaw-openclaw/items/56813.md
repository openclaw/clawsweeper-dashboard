## Summary

Keep open. Current main still bundles and prefers the unscoped `openshell@0.1.0` npm package, and the published package is a Telegram bridge whose startup path throws without `TELEGRAM_BOT_TOKEN`.

## What This Changes

The issue reports that the OpenShell sandbox backend can resolve the bundled unscoped `openshell@0.1.0` npm package instead of the standalone OpenShell CLI, breaking exec startup when `TELEGRAM_BOT_TOKEN` is unset.

## Best Possible Solution

Remove the unscoped `openshell@0.1.0` dependency from the plugin/root package graph, make the default command resolve to the PATH CLI, update the resolver tests, and add a changelog plus package-contract validation.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: enable the OpenShell backend with the default `command: "openshell"`, let the resolver find the bundled npm package, and the package startup path throws when `TELEGRAM_BOT_TOKEN` is missing.

## Solution Assessment

Yes. The reporter's suggested direction is the narrow maintainable fix: stop bundling the unrelated npm package and keep `plugins.entries.openshell.config.command` as a CLI path/name unless a verified scoped OpenShell CLI package exists later.
