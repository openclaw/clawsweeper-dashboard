## Summary

Keep open. Current main still has only the partial Feishu WebSocket startup-rejection retry; after `WSClient.start()` returns, the monitor waits only for abort and does not supervise SDK retry exhaustion or later silent stalls. Two open repository PRs already reference this issue with closing syntax, so cleanup should wait for one of those candidates or an equivalent replacement to land.

## What This Changes

The issue requests Feishu WebSocket recovery after network disruption or Lark SDK retry exhaustion, without requiring a manual gateway restart.

## Best Possible Solution

Keep the issue open and use PR #73739 or PR #73940 as the implementation candidate. The final fix should stay in the Feishu plugin transport, wire the SDK post-start failure/reconnect signals or reconnect-info polling into an OpenClaw-owned supervisor, recreate dead WSClient instances with bounded abortable backoff, preserve shutdown cleanup, and add focused regression tests for retry exhaustion and abort paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: after `WSClient.start()` resolves, current `monitorWebSocket()` waits only for abort while the SDK can later report retry exhaustion through callbacks that OpenClaw does not wire.

## Solution Assessment

Yes for keeping this open. Closing now would be premature because current main only solves startup failures, while the narrower maintainable fix is the existing Feishu transport supervisor work already represented by open PR candidates.
