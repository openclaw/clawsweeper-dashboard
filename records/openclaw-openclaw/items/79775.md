## Summary

Keep open: current source explains a plausible token-drift/caller-scope path for the reported error, but the report does not prove that the active approval session actually held `operator.admin` rather than only showing an approved pairing baseline.

## What This Changes

The issue asks for `openclaw devices approve <requestId>` to succeed when `devices list` shows the paired CLI device with `operator.admin` during a Synology DSM reverse-proxy deployment.

## Best Possible Solution

Clarify or fix the CLI/device approval flow so operators can distinguish approved baseline scopes from active token/session scopes, then ensure an admin-capable shared-token path can approve browser pairing without unsafe scope escalation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear: source and tests reproduce the same error when the paired record is admin-approved but the active caller session is pairing-only, but the report does not prove the DSM CLI session had an admin-scoped active token.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the current caller-scope guard is an intentional security boundary, so the maintainable fix may be better diagnostics/token rotation guidance or a carefully scoped shared-auth retry rather than relaxing approval authorization.
