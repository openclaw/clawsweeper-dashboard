## Label Justifications

- `P2`: The issue affects context visibility and compaction behavior for providers without usage data, but the blast radius is limited to token/session accounting surfaces.
- `impact:session-state`: The report is about context token state becoming unknown or stale across session storage, status, UI, and compaction decisions.

## Summary

Keep open. Current main partially covers transcript-backed compaction/session rows, but the requested local estimated context visibility is still incomplete: `/status` suppresses transcript context when `totalTokensFresh` is false, `/usage` and the Control UI hide stale context percentages, and a related open PR only covers the memory-flush fallback.

## What This Changes

Request local estimated context-token tracking when provider usage is absent so status, UI visibility, and compaction can use a labeled estimate.

## Best Possible Solution

Define a shared source-aware context usage snapshot for provider-fresh, transcript-estimated, and unknown values, then use it consistently in status, `/usage`, session rows, Control UI notices, preflight, and memory-flush without promoting cumulative session usage as current prompt size.

## Reproduction Assessment

Yes. Source inspection on current main shows stale/unknown totals still render as unknown or hidden in `/status`, `/usage`, and Control UI while preflight has a transcript estimator; I did not run a live MiniMax session.

## Solution Assessment

Unclear until maintainers accept the estimated-display semantics. The safest direction is a shared source-aware context snapshot rather than provider-specific MiniMax handling or separate per-surface heuristics.
