## Label Justifications

- `P2`: The issue affects context visibility and compaction accounting for providers without usage data, but the blast radius is limited to token/session accounting surfaces.
- `impact:session-state`: The report is about session context token state becoming stale, unknown, or unavailable across status, UI, and compaction decisions.

## Summary

Keep open: current main only partially addresses the request. Preflight compaction can estimate from transcripts, but `/status`, `/usage`, and the Control UI still suppress stale or missing context usage instead of showing an estimated value, and the remaining memory-flush fallback is covered by an open narrower PR rather than a merged fix.

## What This Changes

Request to locally estimate context token usage from session transcripts when providers do not return usable usage data, then show and use that estimate for status and compaction decisions.

## Best Possible Solution

Define one shared source-aware context usage snapshot with provider-fresh, transcript-estimated, and unknown states, then feed it consistently into status, `/usage`, session rows, Control UI notices, preflight compaction, and memory-flush without treating cumulative session usage as current prompt size.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the remaining gap: stale or unavailable `totalTokens` still renders unknown or hidden context in `/status`, `/usage`, and the Control UI while only preflight has a transcript estimator. I did not run a live MiniMax session in this read-only review.

## Solution Assessment

Unclear until maintainers decide the estimated-usage semantics. The safest direction is a shared source-aware context snapshot, not MiniMax-specific handling or separate heuristics in each display and compaction surface.
