## Label Justifications

- `P2`: The issue affects context visibility and compaction accounting for providers without usable usage data, but it is limited to token/session accounting behavior.
- `impact:session-state`: The reported failure is stale or missing context token state across session storage, status, UI, and compaction decisions.

## Summary

Keep open: current main partially addresses transcript-derived usage through session listing, but `/status`, `/usage`, the Control UI notice, and memory-flush still do not consistently expose or use an estimated context snapshot when provider usage is stale or missing.

## What This Changes

Request local transcript-derived token estimates for providers lacking usage data and use them in status visibility and automatic compaction decisions.

## Best Possible Solution

Define one shared source-aware context usage snapshot with provider-reported, transcript-estimated, and unknown states, then feed it consistently into status, `/usage`, Control UI notices, preflight compaction, and memory-flush while keeping cumulative usage separate from current prompt size.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: stale or unavailable `totalTokens` still renders unknown or hidden context in `/status`, `/usage`, and Control UI surfaces, while only some compaction/session-list paths estimate from transcripts. I did not run a live MiniMax session.

## Solution Assessment

Unclear as proposed because independent chars-per-token fallbacks in each surface would drift. The safer solution is a shared estimated-vs-reported context snapshot with explicit freshness/source semantics.
