## Summary

Keep open: current main has provider/transcript usage fallback in some paths, but it does not estimate tokens from transcript content when provider usage is absent or zero, and chat `/status` intentionally suppresses transcript context fallback when `totalTokensFresh` is false.

## What This Changes

Request local token estimation from session history when provider usage data is absent so status and compaction decisions can reflect approximate context size.

## Best Possible Solution

Add a clearly marked estimated-token path for stale or zero-usage transcripts, keep provider-reported usage distinct from estimates, and decide explicitly whether estimates may update persisted totals or only power display/diagnostics.

## Reproduction Assessment

Yes from source inspection: a session with `totalTokensFresh: false` is blocked from transcript context fallback, and the current test suite asserts `Context: ?/1.0m` for that path. I did not run tests because this review is read-only and `pnpm` is unavailable in the checkout.

## Solution Assessment

Unclear. The requested local estimate is a plausible direction, but the exact fix should separate estimated display from persisted provider usage and should not silently treat heuristic counts as authoritative compaction telemetry without maintainer agreement.
