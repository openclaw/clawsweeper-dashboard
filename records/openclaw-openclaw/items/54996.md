## Label Justifications

- `P2`: The issue affects context visibility and compaction accounting for providers without usage data, but the impact is limited to token/session accounting surfaces.
- `impact:session-state`: The report is about context token state becoming stale or unknown across session storage, status, UI, and compaction decisions.

## Summary

Current main only partially covers the requested behavior: preflight compaction can estimate from transcripts, but status, usage, Control UI notices, and memory-flush still treat stale or missing token totals as unknown or only use transcript usage when it is already reliable. The related open issues and PRs cover narrower slices, so this broader source-aware estimated-context behavior still needs maintainer direction instead of cleanup closure.

## What This Changes

The issue requests locally estimated context-token tracking and an estimated display/compaction path when provider API usage data is unavailable.

## Best Possible Solution

Define one shared source-aware context usage snapshot with provider-fresh, transcript-estimated, and unknown states, then feed that consistently into status, `/usage`, session rows, Control UI notices, preflight compaction, and memory-flush without promoting cumulative session usage as current prompt size.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: stale or unavailable `totalTokens` still renders unknown or hidden context in `/status`, `/usage`, and the Control UI, while preflight has a transcript estimator that is not shared across all requested surfaces. I did not run a live MiniMax session in this read-only review.

## Solution Assessment

Unclear until maintainers accept the estimated-display semantics. The safest direction is a shared source-aware context snapshot, not MiniMax-specific handling or independent heuristics in each surface.
