## Label Justifications

- `P2`: The issue affects context visibility and compaction accounting for providers without usable usage telemetry, with limited blast radius to session/status accounting.
- `impact:session-state`: The reported failure is stale or missing context token state across session storage, status display, and compaction decisions.

## Summary

Current main now covers the specific provider-missing-usage problem: it stores a pre-prompt estimated context budget, uses it for `/status` when fresh provider totals are unavailable, and uses the same local token-pressure estimate in the pre-prompt compaction decision path.

## What This Changes

Request local context-token estimation when provider usage is missing so context reporting and compaction decisions do not stay at zero or unknown.

## Best Possible Solution

Keep the current-main estimated context budget implementation for this specific missing-usage case, and route broader proactive compaction policy work to the existing runtime budget awareness thread.

## Reproduction Assessment

No current-main failure path remains from source inspection: stale or absent provider usage can now fall back to `contextBudgetStatus` in `/status`, and pre-prompt compaction already uses local token pressure estimates. I did not run a live MiniMax session in this read-only review.

## Solution Assessment

Yes for this issue's specific scope: the implementation avoids trusting stale provider totals, marks the display as estimated, persists/clears the estimate across session boundaries, and keeps broader proactive policy separate.
