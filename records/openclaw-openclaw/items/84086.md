## Label Justifications

- `P1`: The report describes a live Discord/Codex channel workflow repeatedly losing native thread continuity and cache state across ordinary multi-agent channel traffic.
- `impact:session-state`: The affected behavior abandons the native Codex thread binding and accumulated session/cache state while the OpenClaw session continues.

## Summary

Keep open: current `main` still changes the Codex `message` dynamic-tool shape when `room_event` forces `message_tool_only`, and that shape participates in the stored dynamic-tool fingerprint that can clear the native thread binding. The related owner-only tool PRs address a different tool-count flip and do not cover this direct-vs-deferred `message` schema path.

## What This Changes

The issue asks OpenClaw to preserve Codex native thread continuity in Discord multi-agent channel sessions when room-event delivery forces message-tool-only handling.

## Best Possible Solution

Stabilize Codex dynamic-tool identity across normal Discord channel turns and `room_event` message-tool-only turns, while keeping per-turn visible delivery requirements and unauthorized sends fail-closed, with a regression that alternates human and room-event turns in one session.

## Reproduction Assessment

Yes, at source level: a room-event turn forces `message_tool_only`, which makes `message` direct, removes `namespace`/`deferLoading` from that tool spec, and changes the fingerprint used to decide thread resume. I did not run the reporter's private Discord setup.

## Solution Assessment

Unclear for the exact patch shape. The issue boundary is right, but maintainers should choose whether to keep `message` in one stable registration mode or normalize direct/deferred loading fields out of the compatibility fingerprint after confirming the Codex app-server contract.
