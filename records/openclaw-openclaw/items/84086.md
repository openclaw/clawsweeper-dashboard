## Summary

Keep open. Current main still has the source-level path where Discord room events force `message_tool_only`, Codex makes `message` direct only for that turn, and the dynamic-tool fingerprint can invalidate the native thread binding; the open fixing PR at https://github.com/openclaw/openclaw/pull/84211 specifically references this issue and should be reviewed or closed before the issue is resolved.

## What This Changes

The issue asks OpenClaw to stop Codex app-server native thread churn in mixed Discord channel sessions when `room_event` / `message_tool_only` changes only the `message` dynamic-tool registration shape.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84211, or an equivalent fix that keeps the `message` dynamic-tool schema stable across automatic and `message_tool_only` turns while preserving message-tool-only delivery and authorization checks.

## Reproduction Assessment

Yes, source-level. Current main clearly alternates the `message` dynamic tool between direct and deferred specs when `sourceReplyDeliveryMode` flips, and that metadata is included in the binding fingerprint; the private production trajectory itself was not replayed.

## Solution Assessment

Yes, the best direction is a narrow Codex app-server fix that stabilizes the `message` tool contract across delivery-mode flips while keeping delivery and authorization checks in runtime policy; the open linked PR implements that approach but is not merged.
