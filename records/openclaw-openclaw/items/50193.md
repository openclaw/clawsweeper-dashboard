## Summary

Keep open: the provider-grouped picker is a useful Control UI feature, but this PR is not merge-ready because it edits the pre-split helper path and drops catalog-aware labels that current main relies on to keep model choices distinguishable.

## What This Changes

This PR changes the Control UI chat model picker to render provider `<optgroup>` sections with a hard-coded provider label table and alphabetically sorted model options.

## Best Possible Solution

Represent provider grouping in `ui/src/ui/chat-model-select-state.ts`, render it from `ui/src/ui/chat/session-controls.ts`, and derive labels from catalog or provider metadata while preserving submitted values and disambiguated display labels.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Source review gives a high-confidence current-main path showing the active picker is flat and that duplicate-label behavior is already protected by tests.

## Solution Assessment

No. The requested grouping is reasonable, but this patch targets the stale helper path and bypasses `buildChatModelOptionFromLookup`/`buildCatalogDisplayLookup`; rebasing into the current state/render split is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move grouping into the current selector split:** `ui/src/ui/app-render.helpers.ts:548-555`
  - body: Current main no longer builds or renders the chat model picker in this helper; it delegates model state to `chat-model-select-state.ts` and rendering to `chat/session-controls.ts`. Rebase the grouping into those files, otherwise this PR cannot land cleanly on the active UI path.
  - confidence: 0.88
- **[P2] Preserve catalog-aware option labels:** `ui/src/ui/app-render.helpers.ts:688`
  - body: This builds each catalog option label from `entry.name` only, dropping the existing display lookup that adds provider or model ID suffixes when friendly names collide. Duplicate catalog names can then render as identical options while submitting different values.
  - confidence: 0.91
- **[P3] Derive provider labels from catalog metadata:** `ui/src/ui/app-render.helpers.ts:640-641`
  - body: The fixed provider table only lowercases the raw provider ID, so real catalog IDs such as `openai-codex` and `azure-openai-responses` still render as slugs. Use existing catalog/provider metadata or a shared display helper with a fallback.
  - confidence: 0.78
