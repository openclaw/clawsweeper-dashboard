## Summary

Keep open. The proposal is a plausible UX feature, but the branch is stale against the current selector split, regresses catalog label disambiguation, and lacks required real Control UI proof.

## What This Changes

The PR changes the Control UI chat model selector helper to group models into provider `<optgroup>` sections with a hard-coded friendly provider-label map.

## Best Possible Solution

Decide the model-picker grouping and label policy first, then implement it in `ui/src/ui/chat-model-select-state.ts` and `ui/src/ui/chat/session-controls.ts` while preserving catalog metadata and duplicate-label disambiguation.

## Reproduction Assessment

Not applicable: this is a feature PR, not a bug report. For the review findings, source inspection of current main and the PR diff gives a high-confidence path showing the stale target file and label-contract regression.

## Solution Assessment

No. Grouping may be reasonable, but this implementation should move to the active selector state/render split and use catalog/provider metadata or an approved label policy instead of hard-coded labels.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move grouping into the active selector split:** `ui/src/ui/app-render.helpers.ts:716-755`
  - body: Current main renders the model picker from `ui/src/ui/chat/session-controls.ts` and state from `ui/src/ui/chat-model-select-state.ts`. Adding grouped rendering here would not update the active picker after resolving the branch against main, so users would still get the flat selector.
  - confidence: 0.94
- **[P2] Preserve catalog-backed option labels:** `ui/src/ui/app-render.helpers.ts:686-688`
  - body: The catalog loop labels entries with `entry.name` or the raw id instead of the current display lookup. That drops alias support and duplicate-name disambiguation covered by current tests, so different catalog entries can render with indistinguishable option text.
  - confidence: 0.9
- **[P3] Derive provider group labels from metadata:** `ui/src/ui/app-render.helpers.ts:566-640`
  - body: The hard-coded provider table will drift and already misses real provider IDs such as `openai-codex`, leaving those groups as raw slugs. Group labels should come from provider/catalog metadata or a shared formatter with slug fallback.
  - confidence: 0.82
