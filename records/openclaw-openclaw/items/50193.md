## Summary

Keep this PR open: the UX goal is valid, but current main still renders the chat model picker as a flat list and this branch targets the old helper after the selector was split into separate state/render modules. The patch also bypasses existing catalog-aware display labels and uses a hard-coded provider label map, which would regress ambiguous model labels and leave real provider IDs as raw slugs.

## What This Changes

The PR changes the Control UI chat model picker from flat options in `ui/src/ui/app-render.helpers.ts` to provider `<optgroup>` sections with a hard-coded provider label table and alphabetically sorted model options.

## Best Possible Solution

Represent provider grouping in the current `chat-model-select-state.ts` model-select state, render it from `chat/session-controls.ts`, and reuse the existing catalog display lookup or provider metadata so grouped labels improve scanning without losing unique model labels or changing submitted `sessions.patch` values.

## Reproduction Assessment

Yes. A high-confidence read-only reproduction path exists: current main exposes a flat `options` array and renders it directly as `<option>` children, while current tests demonstrate duplicate-name cases that the PR's `entry.name` label path would collapse.

## Solution Assessment

No. The requested grouping is a good direction, but this patch is not the best implementation because it targets the old owner file and bypasses the catalog display helper that already protects model-picker correctness.
