## Summary

Keep open: this is a plausible UI feature, but the branch is conflicting and the diff targets the old helper instead of the current active selector split. It also drops the current catalog-label contract and lacks real behavior proof, so it needs human product review before merge.

## What This Changes

The PR groups the Control UI chat model selector by provider with hard-coded friendly provider labels and per-provider model sorting.

## Best Possible Solution

Decide the model-picker display policy first, then rebuild provider grouping on ui/src/ui/chat-model-select-state.ts and ui/src/ui/chat/session-controls.ts while preserving catalog alias/name disambiguation and deriving group labels from provider/catalog metadata.

## Reproduction Assessment

Not applicable: this is a feature PR, not a bug report. Source inspection confirms current main still exposes a flat ChatModelSelectState.options list and the PR lacks runtime proof of the grouped UI.

## Solution Assessment

No. Grouping may be a reasonable direction, but this implementation should move to the active selector split, preserve current catalog-label contracts, and wait for product direction on grouped labels versus exact refs.
