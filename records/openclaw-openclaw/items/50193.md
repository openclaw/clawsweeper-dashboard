## Summary

Keep open: the requested grouped model picker is still useful, but this PR is not ready because it targets the old helper path and drops catalog-aware labels that current main relies on to keep model choices distinguishable.

## What This Changes

This PR changes the Control UI chat model picker to group options by provider with hard-coded friendly provider labels and sorted model entries.

## Best Possible Solution

Represent provider grouping in `ui/src/ui/chat-model-select-state.ts`, render it from `ui/src/ui/chat/session-controls.ts`, and preserve catalog-derived display labels and submitted values.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Source review gives a high-confidence current-main path showing the active picker is flat and duplicate-label behavior is covered by tests.

## Solution Assessment

No. The requested grouping is reasonable, but the patch should be rebased into the current state/render split and reuse catalog display lookup semantics instead of inlining labels in the old helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move grouping into the current selector split:** `ui/src/ui/app-render.helpers.ts:548-555`
  - body: Current main no longer builds or renders the chat model picker in this helper; it resolves state in `chat-model-select-state.ts` and renders in `chat/session-controls.ts`. Rebase the grouping there so the patch applies to the active picker path.
  - confidence: 0.88
- **[P2] Preserve catalog-aware option labels:** `ui/src/ui/app-render.helpers.ts:688`
  - body: This builds catalog option text from `entry.name` alone, dropping the existing display lookup that adds provider or model ID suffixes for collisions. Duplicate catalog names can then render as identical choices while submitting different values.
  - confidence: 0.91
- **[P3] Derive provider labels from catalog metadata:** `ui/src/ui/app-render.helpers.ts:640-641`
  - body: The fixed provider table misses real and plugin-owned IDs such as `openai-codex` and `azure-openai-responses`, so those groups still show slugs and the table will drift. Use catalog/provider metadata or a shared display helper with a fallback.
  - confidence: 0.78
