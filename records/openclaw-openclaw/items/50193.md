## Summary

Keep open: current main still renders the Control UI chat model picker as a flat list, so the requested UX is not implemented, but this PR needs rework because it targets the stale helper surface and drops catalog-aware label behavior.

## What This Changes

The PR changes the Control UI chat model selector to group model options by provider with friendly provider labels and sorted models.

## Best Possible Solution

Implement provider grouping in `ui/src/ui/chat-model-select-state.ts` and `ui/src/ui/chat/session-controls.ts`, preserving catalog display lookup semantics and adding focused state/render tests plus a changelog entry.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Source review gives a high-confidence current-main path showing the active picker is flat and covered by tests for value and label contracts.

## Solution Assessment

No. The requested grouping is reasonable, but this patch changes the stale helper path and bypasses the current catalog-aware state/render split; rebasing into the active files is the narrower maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move grouping into the current selector split:** `ui/src/ui/app-render.helpers.ts:548-555`
  - body: Current main no longer builds or renders the chat model picker in this helper; it delegates model state to `chat-model-select-state.ts` and rendering to `chat/session-controls.ts`. Rebase the grouping into those active files so the feature lands on the code path users actually run.
  - confidence: 0.88
- **[P2] Preserve catalog-aware option labels:** `ui/src/ui/app-render.helpers.ts:688`
  - body: The PR builds catalog option labels from `entry.name` directly, dropping the existing display lookup that adds provider or model ID suffixes when friendly names collide. Duplicate catalog names can then render as identical choices while submitting different values.
  - confidence: 0.91
- **[P3] Derive provider labels from catalog metadata:** `ui/src/ui/app-render.helpers.ts:640-641`
  - body: The fixed provider table still leaves real provider IDs outside the list, such as `openai-codex` or `azure-openai-responses`, as raw slugs. Use catalog or provider metadata with a fallback instead of baking a partial provider map into the UI helper.
  - confidence: 0.78
