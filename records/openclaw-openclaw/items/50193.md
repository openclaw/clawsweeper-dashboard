## Summary

Keep open: the PR still proposes unique model-picker grouping behavior, so it is not closeable as already implemented or superseded, but the current patch is stale against main and regresses the current catalog-backed label contract.

## What This Changes

The PR changes the Control UI model selector helper to group catalog models into provider `<optgroup>` sections using a hard-coded friendly provider-label table.

## Best Possible Solution

Decide the model-picker grouping and label policy, then implement grouping in `ui/src/ui/chat-model-select-state.ts` and `ui/src/ui/chat/session-controls.ts` while preserving catalog metadata and duplicate-label disambiguation.

## Reproduction Assessment

Not applicable as a user bug reproduction: this is a feature PR. The review findings are source-reproducible by comparing the PR head with current main's active model selector and label tests.

## Solution Assessment

No. Grouping may be reasonable, but this patch targets a stale helper path and bypasses the current catalog-backed label contract; the safer solution is to update the active selector split after a label-policy decision.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Move grouping into the active model selector:** `ui/src/ui/app-render.helpers.ts:716-755`
  - body: Current main renders the model picker from `ui/src/ui/chat/session-controls.ts` using `resolveChatModelSelectState`, while this patch adds grouped rendering back to `app-render.helpers.ts`. After the branch is reconciled with main, this code is not the active picker path, so users would still see the flat selector.
  - confidence: 0.94
- **[P2] Preserve catalog-backed option labels:** `ui/src/ui/app-render.helpers.ts:686-688`
  - body: The catalog loop now labels options with `entry.name` or the raw id instead of the lookup-backed option builder. That drops aliases and duplicate-label disambiguation covered by current tests, so different provider or same-provider entries can become indistinguishable.
  - confidence: 0.9
- **[P3] Derive provider labels from catalog metadata:** `ui/src/ui/app-render.helpers.ts:566-640`
  - body: The static `PROVIDER_LABELS` table creates a second provider naming policy and misses real provider IDs such as `openai-codex`. If grouping is accepted, group labels should come from provider/catalog metadata or a shared formatter with slug fallback.
  - confidence: 0.82
