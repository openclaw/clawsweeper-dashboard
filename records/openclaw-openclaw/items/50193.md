## Summary

Keep open: provider grouping is a plausible Control UI feature, but this branch is not mergeable as-is because it targets the stale pre-split helper, bypasses current catalog display-label contracts, and lacks external real-behavior proof.

## What This Changes

The PR changes the Control UI chat model selector helper to render catalog models under provider optgroups with hard-coded provider display labels and per-group sorting.

## Best Possible Solution

Implement grouping in `ui/src/ui/chat-model-select-state.ts` and `ui/src/ui/chat/session-controls.ts`, reuse existing catalog display disambiguation, derive provider labels through a shared/provider-owned path, and include focused tests plus redacted browser proof.

## Reproduction Assessment

Not applicable: this is a feature PR, not a bug report. Source inspection shows current main still exposes a flat `ChatModelSelectState.options` array rendered as repeated model `<option>` entries.

## Solution Assessment

No. Provider grouping can be a reasonable UX direction, but this diff should be rebuilt on the current selector state/render split and retain catalog-label contracts instead of hard-coding provider labels in the old helper.
