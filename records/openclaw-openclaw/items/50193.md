## Summary

Keep open: the requested provider-grouped model selector is not implemented on current main and no exact canonical replacement was found, but this PR should not merge as-is because it targets stale helper code, drops existing catalog-label disambiguation, and lacks after-fix real behavior proof.

## What This Changes

The PR changes the Control UI chat model selector helper to render provider-grouped optgroups with hard-coded provider labels and sorted model options.

## Best Possible Solution

Implement grouping in the current selector state/render split, preserve catalog display lookup disambiguation, derive provider names from catalog metadata or a shared formatter, and add focused state/render tests.

## Reproduction Assessment

Not applicable: this is a feature PR, not a bug report. Source inspection shows current main still exposes and renders a flat chat model option list in the active selector path.

## Solution Assessment

No. Provider grouping is a reasonable UI direction, but this patch should target the current selector split and keep the existing catalog display-label contract instead of rebuilding labels in a stale helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move grouping into the active selector split:** `ui/src/ui/app-render.helpers.ts:548-555`
  - body: Current main renders the chat model picker from `ui/src/ui/chat/session-controls.ts` using state from `ui/src/ui/chat-model-select-state.ts`, and `app-render.helpers.ts` no longer owns this model select code. Adding grouped state and optgroup rendering here will not update the live selector after rebase, so the grouping needs to move into the current state/render split.
  - confidence: 0.94
- **[P2] Preserve catalog display labels:** `ui/src/ui/app-render.helpers.ts:681-688`
  - body: The catalog loop now uses `entry.name` or the raw model id directly instead of the existing catalog display lookup. Current tests require duplicate friendly names across providers, and even within one provider, to keep provider/model-id context, so this can render indistinguishable option text for different submitted values.
  - confidence: 0.9
- **[P3] Derive provider labels from catalog metadata:** `ui/src/ui/app-render.helpers.ts:568-636`
  - body: The fixed provider label table will drift from plugin-owned provider ids and already misses real catalog ids such as `openai-codex` and `azure-openai-responses`. Prefer provider metadata or a shared formatter with fallback behavior so the UI improvement applies to actual catalog entries.
  - confidence: 0.82
