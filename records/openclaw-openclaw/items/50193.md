## Summary

Keep open: provider grouping is a useful UI direction, but this branch is not merge-ready because it edits the pre-split helper path, drops existing catalog display disambiguation, and lacks after-fix browser behavior proof.

## What This Changes

The branch changes the Control UI chat model selector helper to group catalog entries under provider optgroups with hard-coded provider labels and sorted model options.

## Best Possible Solution

Implement provider grouping in the current selector state/render split, preserve catalog display lookup disambiguation, derive provider names from catalog metadata or shared formatting, and add focused tests plus visual proof.

## Reproduction Assessment

Not applicable: this is a feature PR, not a bug report. Source inspection shows current main still renders a flat option list through the current selector state/render split.

## Solution Assessment

No. Provider grouping is reasonable, but this patch should be rebased onto the current selector split and keep the existing catalog display-label contract instead of hard-coding provider names in the stale helper.

Current tests require duplicate friendly names across providers, and even within one provider, to keep provider/model-id context, so this can render indistinguishable option text for different submitted values.
  - confidence: 0.9

  - body: The fixed provider label table will drift from plugin-owned provider IDs and already misses real catalog IDs such as `openai-codex` and `azure-openai-responses`. Prefer provider metadata or a shared formatter with fallback behavior so the UI improvement applies to actual catalog entries.
  - confidence: 0.82
