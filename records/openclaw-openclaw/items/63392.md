## Summary

Keep open. Current main still exposes whole-install `openclaw backup create` and `openclaw backup verify` only, with no restore subcommand, no per-agent `--agent` backup scope, and no `--as` clone flow. The request is concrete and distinct from the adjacent generic restore work because it needs single-agent scoping plus config/path rewrite behavior, but it is security-sensitive enough that maintainers should settle the restore safety and credential/session policy before an automated repair PR attempts it.

## What This Changes

Add `--agent` backup/restore scope and `--as` clone support so users can back up, restore, or duplicate one configured agent instead of the whole OpenClaw install.

## Best Possible Solution

Keep this as a valid feature request. The maintainable end state is a safe generic restore foundation first, followed by per-agent archive creation and restore with optional `--as`, scoped workspace plus `agents/<agentId>` payloads, config-entry rewrite for `id`/`name`/`workspace`/`agentDir`, collision handling, docs, and targeted safety tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main absence is high-confidence from source, docs, and CLI registration tests showing create/verify only with no restore, per-agent backup scope, or clone flags.

## Solution Assessment

Unclear as an immediate implementation path. The requested CLI shape is coherent, but the best maintainable solution should build on the generic restore safety model and explicitly define credential/session clone policy first.
