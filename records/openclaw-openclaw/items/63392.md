## Summary

Keep open: current main and v2026.6.10 still expose whole-install `backup create`/`backup verify` only, while the requested per-agent `--agent`/`--as` restore and clone workflow remains missing. Existing community backup/sync tools and the open encrypted-snapshot PR are useful context, but they do not settle the first-party per-agent config, SQLite state, workspace, credential, and session-history contract.

## What This Changes

Request per-agent backup, restore, and clone support through `openclaw backup create --agent`, restore `--agent`, and clone `--as` flows covering one agent's workspace, state, sessions, and config entry.

## Best Possible Solution

Keep this as the narrow per-agent recovery tracker and design a maintainer-sponsored backup/restore extension that reuses the manifest/SQLite snapshot model, defines secret and session portability, and adds guarded `--agent`/`--as` restore/clone tests plus docs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source, docs, and v2026.6.10 give a high-confidence feature-gap check because backup exposes only create/verify and no per-agent restore/clone flags.

## Solution Assessment

Unclear until maintainers decide the restore/clone policy. A first-party core extension is plausible, but the final design must settle archive contents, secret/session portability, path rewrites, rollback, and coordination with broader backup/restore work.
