## Label Justifications

- `P2`: This is a meaningful recovery and migration feature for multi-agent users, but it is not a current crash or urgent regression.
- `impact:data-loss`: The request is about avoiding whole-instance rollback when only one agent's workspace, config, or state needs recovery.
- `impact:session-state`: The requested per-agent archive explicitly includes sessions, transcripts, and cloned agent history/state.
- `impact:security`: Restore and clone would write credentials-adjacent auth profiles, archived state, workspaces, and sessions from an archive.

## Summary

Keep open. Current main and the latest release still provide whole-install backup create/verify only, while the requested per-agent archive, restore, and clone behavior needs explicit product and security policy before implementation.

## What This Changes

Add `openclaw backup create --agent`, `openclaw restore --agent`, and restore-as clone support for a single agent's workspace, config entry, agent state, and sessions.

## Best Possible Solution

Keep this as a focused native follow-up to the backup lifecycle work, with scoped manifests, `--agent`/`--as`, config rewrites, credential/session policy, docs, and destructive-restore safety tests.

## Solution Assessment

Unclear as an immediate implementation path. The proposed flags are coherent, but the maintainable route should settle generic restore safety plus credential and session clone policy before adding per-agent rewrites.
