## Summary

Keep open: current main and v2026.6.10 still expose whole-install `backup create` and `backup verify` only, while the requested per-agent `--agent` and `--as` restore/clone workflow remains missing. The open encrypted-snapshot PR overlaps backup/restore work, but it is not merged and does not settle the per-agent clone contract.

## What This Changes

The issue requests per-agent `openclaw backup` and restore/clone support, including `--agent <id>` and restore-as `--as <new-id>` behavior for workspaces, agent state, sessions, and config entries.

## Best Possible Solution

Keep this as the narrow per-agent recovery tracker and design a maintainer-sponsored backup/restore extension that reuses the manifest/SQLite snapshot model, defines secret and session portability, and adds guarded `--agent`/`--as` tests plus docs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source, docs, and v2026.6.10 give a high-confidence feature-gap check because backup exposes only create/verify and no per-agent restore/clone flags.

## Solution Assessment

Unclear until maintainers decide the restore/clone policy. A first-party core extension is plausible, but the final design must settle archive contents, secret and session portability, path rewrites, rollback, and coordination with broader backup/restore work.
