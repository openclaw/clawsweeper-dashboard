## Label Justifications

- `P1`: The issue is a high-confidence Feishu channel workflow break that can split active conversation context and misroute replies.
- `impact:session-state`: The central symptom is duplicate direct/group session keys and split history for the same Feishu `oc_...` chat.
- `impact:message-loss`: The split identity can route proactive delivery or replies away from the intended Feishu group conversation.

## Summary

Keep open: current main and the latest shipped release still source-reproducibly persist a direct Feishu session for a bare `oc_...` outbound target, while the Feishu inbound/group paths can use group session identity. The related inbound PR at https://github.com/openclaw/openclaw/pull/52342 is still open and does not cover outbound route construction or duplicate persisted session cleanup.

## What This Changes

Feishu OpenChat/topic chats should keep one session identity across proactive outbound sends, reply-mirror delivery, and inbound messages instead of splitting `oc_...` traffic between direct and group session keys.

## Best Possible Solution

Keep this as the canonical Feishu outbound/session-consistency tracker: share a verified Feishu group/direct classifier across inbound and outbound route construction, preserve explicit `chat:` and `dm:` semantics, and handle existing duplicate sessions through a deliberate doctor or migration path.

## Reproduction Assessment

Yes, source-reproducible. A bare Feishu `oc_...` outbound target still resolves to a direct route that `ensureOutboundSessionEntry` persists, and anomalous `p2p` OpenChat events are still not classified as group by current source inspection; I did not run a live Feishu account.

## Solution Assessment

No, not as the full suggested startup reconciliation. The safest solution is to fix the shared Feishu classifier and outbound route facts first, then make persisted duplicate cleanup an explicit doctor or migration decision.
