## Summary

Keep open: current main still has a source-visible Feishu session-key mismatch where `oc_` IDs are valid chat targets, but unprefixed outbound session routing builds a `direct` route. The related inbound/topic PRs cover adjacent Feishu routing bugs, not this exact duplicate group/direct session problem.

## What This Changes

The issue requests one Feishu OpenChat `oc_` group session across inbound, proactive outbound, cron, and reply delivery instead of split `group` and `direct` session keys.

## Best Possible Solution

Implement a Feishu-owned route/classifier that preserves explicit `user:`/`dm:` versus `chat:`/`group:` intent, uses verified Feishu OpenChat/topic evidence for ambiguous `oc_`/`p2p` cases, and records one canonical group session for each actual group chat.

## Reproduction Assessment

Yes, source-level for the raw outbound target path: a bare `oc_` Feishu target is deliverable as a chat ID, but the outbound route records it as `direct`. The broader reply-delivery mirror path is corroborated by the issue comments, but I did not run a live Feishu current-main reproduction.

## Solution Assessment

No for the broad suggested `oc_` to group heuristic as written. The maintainable fix needs a narrower Feishu-owned discriminator that aligns inbound, outbound, reply, and session-store metadata without weakening p2p/DM routing policy.
