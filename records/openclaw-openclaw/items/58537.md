## Summary

Keep this open: current main still has a source-reproducible Feishu session-key split where bare `oc_...` outbound targets default to direct sessions, while group-known Feishu conversations route as `chat:oc_...`; the related inbound PR remains open and explicitly does not cover this outbound/session-consistency follow-up.

## What This Changes

The issue asks OpenClaw to stop Feishu OpenChat/topic-group `oc_...` chats from creating separate direct and group sessions across outbound sends, normal replies, and persisted session metadata.

## Best Possible Solution

Keep this as the canonical Feishu outbound/session-consistency follow-up, fix the plugin-owned classifier and outbound session-route path to reuse the same verified group conversation identity, and handle any existing duplicate-session cleanup through a deliberate doctor/migration path.

## Reproduction Assessment

Yes, source-reproducible: a bare Feishu `oc_...` target still flows through the outbound session-route builder with `isGroup === false`, while group-known Feishu delivery uses `chat:oc_...`. I did not run a live Feishu account, but the current source and the recent runtime report line up.

## Solution Assessment

No, the full suggested startup reconciliation is not the safest first fix. The maintainable path is to make Feishu group/direct classification and outbound route construction share a verified conversation identity, then treat persisted duplicate cleanup as a separate doctor/migration decision.
