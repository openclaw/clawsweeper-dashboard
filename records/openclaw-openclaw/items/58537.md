## Summary

Keep open. Current main still has the unresolved Feishu classification split: outbound bare `oc_` targets are routed as direct sessions, inbound grouping is based only on Feishu `chat_type`, and session-store lookup does not collapse direct/group variants for the same chat id. A close would be unsafe because prior shipped work explicitly treats bare `oc_` as ambiguous, and the related open PR has review feedback that the prefix-only classifier can regress direct-message authorization.

## What This Changes

The issue asks OpenClaw to keep one Feishu session for `oc_` OpenChat/topic-group chats across proactive outbound sends and inbound webhook messages instead of splitting them into direct and group session keys.

## Best Possible Solution

Keep the issue open and settle the Feishu-owned chat-classification contract first. The maintainable end state is one shared classifier used by inbound routing, mention-forward detection, outbound session routing, and tests, with explicit `chat:`/`group:` targets preserved and any legacy duplicate-session cleanup handled through doctor/fix repair rather than startup migration.

## Reproduction Assessment

Unclear for the full live Feishu duplicate without a current reporter payload, but the code-level split risk is clear: bare outbound `oc_` targets route as direct while group-shaped Feishu sessions can be recorded separately. The missing piece is a verified discriminator for OpenChat topic groups that does not misclassify true p2p `oc_` chats.

## Solution Assessment

No. The suggested blanket `oc_` prefix correction is not the best fix because it conflicts with prior shipped behavior that treats bare `oc_` as ambiguous; a safer solution should use a verified Feishu signal and then apply it consistently across inbound, outbound, and session metadata paths.
