## Summary

Current main still has the reported Discord probe path: gateway startup launches a Discord enrichment probe and channel status probes call the same helper, which reaches Discord `/users/@me` through the shared AbortController timeout wrapper. The issue is new and plausible, but the provided logs and source inspection do not prove that the wrapper itself causes Node.js 24/Windows event-loop starvation, so this should stay open for targeted live validation rather than be swept closed.

## What This Changes

The issue asks OpenClaw to prevent Discord plugin `/users/@me` probe fetch timeouts from causing gateway liveness warnings on Node.js 24/Windows.

## Best Possible Solution

Validate the Discord probe on Windows with Node.js 24, then reduce or avoid nonessential startup `/users/@me` probes when token/config metadata is enough, while preserving useful status diagnostics with focused regression coverage.

## Reproduction Assessment

No, not with high confidence. Source inspection confirms the exact `/users/@me` timeout path on current main, but I did not live-reproduce Node.js 24/Windows event-loop starvation or prove the timeout wrapper is the cause.

## Solution Assessment

Unclear. The likely narrow direction is to reduce nonessential Discord startup/status REST probes, but worker-thread isolation, configurable timeouts, and retry policy are broader runtime/product choices.
