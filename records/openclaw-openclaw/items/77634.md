## Summary

Current main still contains the reported Discord probe path through `/users/@me` and the shared `fetchWithTimeout` AbortController wrapper, and the issue is new with concrete Windows/Node 24 logs. Source inspection supports keeping it open, but it does not prove that the timeout wrapper itself caused event-loop starvation, so this needs targeted live validation rather than cleanup closure.

## What This Changes

The issue requests that Discord bot identity/status probes stop causing Node.js event-loop liveness warnings when `/users/@me` fetches time out.

## Best Possible Solution

Live-validate the Discord probe on Windows with Node 24 and a bot token, then reduce or defer nonessential `/users/@me` startup/status REST probes while preserving useful diagnostics and regression coverage.

## Reproduction Assessment

No, not with high confidence. Source inspection confirms the current-main `/users/@me` timeout path, but I did not reproduce Windows/Node 24 event-loop starvation or prove the wrapper is the cause.

## Solution Assessment

Unclear. The narrow likely direction is to avoid or reduce nonessential Discord REST probes, but worker-thread isolation, configurable timeouts, and retry policy are broader choices that need live evidence and maintainer judgment.
