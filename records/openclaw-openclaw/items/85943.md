## Summary

Keep open: current main intentionally routes legacy Codex CLI refs and `codex-cli` runtime pins to the Codex app-server path, and the docs say OpenClaw no longer registers a bundled `codex-cli` backend. The requested first-class CLI-isolated runtime is not implemented on main, but choosing it over app-server isolation/backpressure fixes needs maintainer product and runtime-safety review.

## What This Changes

The issue asks OpenClaw to restore an opt-in Codex CLI runtime alongside the current Codex app-server route for stronger process isolation under concurrent high-context workloads.

## Best Possible Solution

Maintainers should decide whether the Codex plugin should reintroduce a first-class `codex-cli` CLI backend or instead fix app-server isolation, recovery, and backpressure while keeping CLI use unsupported.

## Reproduction Assessment

No high-confidence live reproduction of the multi-agent Telegram stall was performed. Source inspection does reproduce the missing first-class runtime surface: current docs, migration code, and runtime aliases intentionally route `codex-cli` to the Codex app-server path.

## Solution Assessment

Unclear: restoring CLI isolation may be useful, but it is not obviously the best fix versus improving app-server scheduling, recovery, and backpressure. The permanent direction needs maintainer runtime/product review.
