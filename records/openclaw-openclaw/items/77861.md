## Summary

Keep open: the PR targets a real context-window reporting bug, but it is not merge-ready because the capped helper result is overwritten by directive persistence and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR changes auto-reply model selection so `agents.defaults.contextTokens` caps, rather than expands, the resolved model context window and adds helper-level unit tests.

## Best Possible Solution

Land a complete fix that applies downward-only `contextTokens` cap semantics everywhere reply/status context budgets are computed, including directive persistence, with full-flow status regression coverage.

## Reproduction Assessment

Yes, from source inspection: current main returns the agent default before model lookup, while the guard contract already treats the agent value as a downward-only cap. I did not run a live `/status` turn under this read-only review, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. Capping the model-selection helper is the right direction, but the current PR is incomplete because directive persistence recomputes the old uncapped override before the status path consumes it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep the capped value through directive persistence:** `src/auto-reply/reply/model-selection.ts:331-346`
  - body: This helper now returns the capped model window, but `get-reply-directives` immediately replaces it with `persistInlineDirectives()`'s return value. That path still passes `agentCfg?.contextTokens` as `contextTokensOverride`, and the shared resolver returns overrides before model lookup, so the linked `/status` config can still report `1.0m`. Share the cap logic with the persistence path or avoid recomputing the uncapped override.
