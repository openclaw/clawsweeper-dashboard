## Summary

Keep open. Current main still contains the supervised duplicate-start retry window reported here when a systemd-managed gateway hits a duplicate lock or `EADDRINUSE` and the existing gateway health probe is false.

## What This Changes

The issue asks for read-only CLI commands such as `health`, `cron list`, and `tasks list` to avoid restarting or taking over a running systemd-managed Gateway.

## Best Possible Solution

Keep query commands on RPC or lean local paths, and make systemd-supervised duplicate lock or `EADDRINUSE` conflicts fail closed with exit 78 instead of entering the retry window.

## Solution Assessment

Yes. The narrow maintainable repair is to preserve read-only CLI RPC behavior and restore fail-closed exit-78 handling for supervised duplicate-start conflicts without adding a new mode or config option.

Start in `src/cli/gateway-cli/run.ts`, `src/cli/gateway-cli/run.supervised-lock.test.ts`, `src/gateway/server/http-listen.ts`, and `src/daemon/systemd-unit.ts`; only touch health, cron, or tasks command paths if tracing proves they enter gateway startup.
