## Summary

Keep open. Current main contains the earlier #75115 systemd/EADDRINUSE exit-78 hardening, but this report is against v2026.4.29 after that fix shipped, and the current supervised startup path still has the reported `waiting 5000ms before retrying startup` branch. The source also shows the named read-only CLI commands should query the running Gateway or local task state rather than intentionally start a Gateway, so the observed restart is not proven fixed.

## What This Changes

Read-only CLI commands such as `openclaw health --json`, `openclaw cron list --json`, and `openclaw tasks list` should not trigger a systemd-managed Gateway restart or take over the active Gateway port.

## Best Possible Solution

Add a narrow Linux/systemd guard so read-only CLI query commands cannot cause Gateway startup or restart, and so any systemd-supervised duplicate Gateway process that encounters an existing listener exits with sysexits 78 without retrying into port takeover. Preserve unmanaged duplicate-start behavior, add targeted regression coverage for health/cron/tasks CLI routing and supervised lock conflicts, and include a user-facing changelog entry crediting the reporter.

## Reproduction Assessment

Yes, there is a concrete product reproduction path in the report: run a systemd-managed local Gateway, invoke `openclaw health --json`, and watch the Gateway log and Telegram reconnect behavior. Code-level reproduction should also simulate a systemd-supervised lock or EADDRINUSE conflict where the health probe is false and assert that the process does not retry into takeover.

## Solution Assessment

Unclear until the exact restart trigger is isolated. The safest direction is narrower than a broad restart redesign: keep read-only CLI commands on RPC/local read paths and make systemd-supervised duplicate-start conflicts fail closed with exit 78 instead of waiting and retrying against an occupied port.
