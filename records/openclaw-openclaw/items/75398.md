## Summary

Keep open. Current main has a partial systemd exit-78 mitigation for healthy duplicate Gateway lock conflicts, but the exact `waiting ... before retrying startup` path still exists and current tests still assert systemd false-health retries for duplicate lock and `EADDRINUSE` cases.

## What This Changes

The issue reports that read-only CLI commands in v2026.4.29 can trigger a systemd-managed Gateway restart by entering a supervised duplicate-start wait-and-retry path.

## Best Possible Solution

Make read-only CLI commands unable to start or displace the Gateway, and make systemd-supervised duplicate lock or port conflicts fail closed with exit 78 instead of retrying into a takeover window.

## Reproduction Assessment

Yes. The issue gives concrete Linux/systemd steps, and current unit coverage reproduces the contested behavior by injecting a systemd supervisor, duplicate Gateway lock or `EADDRINUSE`, a false health probe, and asserted retry sleeps.

## Solution Assessment

No. Current main has a useful healthy-Gateway exit-78 guard, but the false-health systemd retry behavior remains; the narrower maintainable fix is to fail closed for systemd duplicate-start conflicts while keeping query commands on RPC/read-only paths.
