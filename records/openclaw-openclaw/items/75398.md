## Summary

Keep open. Current main still has the systemd supervised duplicate-lock path that waits and retries startup when the existing Gateway health probe is false, and current tests explicitly assert those retries, so the reported v29 behavior is not fully fixed.

## What This Changes

The issue asks for read-only CLI commands such as health, cron list, and tasks list to stop restarting or taking over a running systemd-managed Gateway.

## Best Possible Solution

Keep query commands on RPC or lean local paths, and make systemd-supervised duplicate lock or EADDRINUSE conflicts fail closed with exit 78 instead of entering a retry window that can take over the running port.

## Reproduction Assessment

Yes. The issue gives concrete Linux/systemd steps, and current source plus unit tests reproduce the contested branch by simulating a systemd supervisor, duplicate lock or EADDRINUSE conflict, and false health probe.

## Solution Assessment

Yes for the repair direction. The narrow maintainable fix is to preserve read-only CLI RPC behavior and make supervised duplicate-start conflicts return 78 without retry takeover; no new user-facing mode or config option is needed.
