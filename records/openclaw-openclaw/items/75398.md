## Summary

Keep open. Current main still has the supervised Gateway lock path that logs a 5s wait and retries startup when the existing systemd Gateway does not answer the health probe, and the report is recent, concrete, and not protected.

## What This Changes

The issue asks for read-only CLI commands to query the existing Gateway without triggering a systemd-managed Gateway restart or port takeover retry loop.

## Best Possible Solution

Make read-only CLI commands unable to start or take over the Gateway, and make systemd-supervised duplicate lock or occupied-port conflicts fail closed with sysexits 78 without retrying full startup through plugin/bootstrap.

## Solution Assessment

No. Current main has adjacent exit-78 handling, but it still preserves the wait/retry startup branch; the narrower maintainable fix is to fail closed for systemd duplicate conflicts while keeping CLI query commands on RPC/local paths.
