## Summary

Keep open. Current main still contains the supervised Gateway wait-and-retry path named in the report, and the existing tests preserve retries when the health probe is false; the prior #75115 exit-78 fix shipped in the reported v2026.4.29 release, so it is not enough to treat this post-release report as fixed.

## What This Changes

Read-only CLI commands such as `openclaw health --json`, `openclaw cron list --json`, and `openclaw tasks list` should not trigger a systemd-managed Gateway restart or take over the active Gateway port.

## Best Possible Solution

Add a narrow Linux/systemd guard so read-only CLI query commands cannot cause Gateway startup or restart, and systemd-supervised duplicate Gateway lock or EADDRINUSE conflicts fail closed with sysexits 78 instead of retrying into port takeover. Preserve unmanaged duplicate-start behavior, add targeted regression coverage for health/cron/tasks CLI routing plus supervised lock conflicts, and add a user-facing changelog entry crediting the reporter.

## Reproduction Assessment

Yes. The issue provides a concrete product reproduction path: run a systemd-managed local Gateway, invoke `openclaw health --json`, and watch the Gateway log and Telegram reconnect behavior; a code-level reproduction should simulate a systemd-supervised lock or EADDRINUSE conflict where the health probe is false.

## Solution Assessment

Unclear until the exact restart trigger is isolated. The safest direction is narrower than a broad restart redesign: keep read-only CLI commands on RPC/local read paths and make systemd-supervised duplicate-start conflicts exit 78 rather than wait and retry against an occupied port.
