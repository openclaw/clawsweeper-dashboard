## Summary

Keep open. Current main still contains and tests the systemd-supervised duplicate-start retry path that matches the reported log message, while the direct `health`, `cron list`, and `tasks list` command paths should be traced before changing broad CLI routing.

## What This Changes

The issue reports that read-only CLI commands can trigger a systemd-managed gateway restart/takeover instead of connecting to the running gateway or exiting safely.

## Best Possible Solution

Trace why the reported CLI invocations enter gateway startup, then adjust the systemd duplicate-start/health-probe policy so read-only CLI use cannot take over a supervised gateway while preserving intentional stale-process recovery.

## Reproduction Assessment

Yes at source level for the supervised false-health retry branch: current tests still assert the duplicate-start retry behavior. No live end-to-end reproduction shows `health`, `cron list`, or `tasks list` entering that branch on current main.

## Solution Assessment

Unclear. Failing closed with exit 78 for systemd duplicate-start conflicts is likely the narrow direction, but current docs/tests intentionally keep bounded unhealthy retries, so maintainers should choose that policy after tracing the CLI invocation path.
