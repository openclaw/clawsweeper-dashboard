## Summary

Keep open. Current main still contains the supervised Gateway lock branch that waits and retries startup when an existing systemd-managed Gateway does not answer the health probe, while the issue is recent, concrete, and not protected.

## What This Changes

Read-only CLI commands in v2026.4.29 are reported to restart or take over a systemd-managed running Gateway instead of only connecting to it.

## Best Possible Solution

Make read-only CLI query commands unable to start or take over the Gateway, and make systemd-supervised duplicate lock/port conflicts fail closed with sysexits 78 instead of waiting into a retry loop.

## Reproduction Assessment

Yes. The issue gives a concrete Linux/systemd path, and current tests can reproduce the contested behavior by simulating a supervised lock or EADDRINUSE conflict with a false health probe.

## Solution Assessment

Unclear for the full CLI trigger until the startup path is isolated, but the narrow maintainable direction is clear: keep query commands on RPC/read-only paths and make supervised duplicate-start conflicts exit 78 without unhealthy retry takeover.
