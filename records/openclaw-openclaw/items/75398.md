## Summary

Keep open. Current main still contains the systemd-supervised lock branch that waits and retries startup when the existing Gateway is not considered healthy, so this recent report is not proven fixed by the earlier systemd/EADDRINUSE work.

## What This Changes

Read-only CLI commands such as `openclaw health --json`, `openclaw cron list --json`, and `openclaw tasks list` should not restart or take over a running systemd-managed Gateway port.

## Best Possible Solution

Add a narrow Linux/systemd guard so read-only CLI query commands cannot cause Gateway startup or restart, and make systemd-supervised duplicate Gateway lock/EADDRINUSE conflicts fail closed with sysexits 78 instead of waiting into a possible port takeover. Preserve unmanaged duplicate-start behavior and cover the routing plus supervised conflict cases with focused tests.

## Reproduction Assessment

Yes. The issue gives a concrete product reproduction path: run a systemd-managed local Gateway, invoke `openclaw health --json`, and watch the Gateway log for the wait/retry startup branch and channel reconnects; code-level reproduction should simulate a systemd lock/EADDRINUSE conflict with a false health probe.

## Solution Assessment

Unclear until the exact restart trigger is isolated. The safest fix is narrower than a restart redesign: keep read-only commands on RPC/local read paths and make supervised duplicate-start conflicts exit 78 instead of retrying against an occupied port.
