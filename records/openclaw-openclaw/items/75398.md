## Summary

Keep open. Current main still contains the systemd-supervised duplicate-start branch that waits and retries when an existing Gateway does not answer the health probe, and the regression tests still assert that retry behavior for unhealthy lock and EADDRINUSE conflicts. The #75115 changelog entry covers the healthy/exit-78 case but does not prove this newer CLI-triggered restart report is fixed.

## What This Changes

Prevent read-only `openclaw` CLI commands on Linux/systemd from triggering Gateway restarts or taking over the running Gateway port.

## Best Possible Solution

Make read-only CLI query commands unable to enter Gateway startup, and make systemd-supervised duplicate lock or port conflicts fail closed with sysexits 78 instead of waiting into a retry loop. Preserve unmanaged duplicate-start behavior and cover both command routing and supervised conflict semantics with focused tests plus a changelog entry.

## Reproduction Assessment

Yes. The issue provides a concrete Linux/systemd path, and current unit-level code can reproduce the core failure mode by simulating a systemd-supervised lock or EADDRINUSE conflict with a failing health probe; current tests still assert retries in that condition.

## Solution Assessment

No, current main is not the best solution because it still retries supervised duplicate starts when the existing Gateway is unhealthy. The safer repair is narrow: keep read-only commands on RPC/local read paths and return exit 78 for systemd duplicate-start conflicts without a wait/retry takeover window.
