Keep open. Current main still contains the systemd supervised-lock retry path that logs the reported 5s retry message when the existing Gateway health probe is false, and tests explicitly assert repeated startup attempts in that case.

## Best Possible Solution

Keep the existing Gateway process in control: read-only CLI commands should stay query-only, and systemd duplicate lock or `EADDRINUSE` conflicts should fail closed with exit 78 instead of retrying into a takeover window.

The observable bug is that read-only CLI commands such as `openclaw health --json`, `openclaw cron list --json`, and `openclaw tasks list --json` must not start, restart, or displace an already running systemd-managed Gateway. Keep the fix scoped to Gateway startup/CLI routing: preserve query-only command routing, and make systemd-supervised duplicate Gateway lock or `EADDRINUSE` conflicts fail closed with sysexits 78 rather than waiting/retrying into a port takeover window. Add or update focused tests around `runGatewayLoopWithSupervisedLockRecovery`, EADDRINUSE handling, the generated systemd unit, and read-only command routing.
