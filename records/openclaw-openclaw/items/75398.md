## Summary

Keep open. Current main still contains the supervised Gateway lock branch that waits and retries startup when a systemd-managed duplicate start sees an unhealthy health probe, and the report is recent, concrete, and not protected.

## What This Changes

The issue asks that read-only CLI commands such as health, cron list, and tasks list query the existing Gateway without causing a systemd-managed Gateway restart or port takeover.

## Best Possible Solution

Make read-only CLI query commands unable to start or take over the Gateway, and make systemd-supervised duplicate lock or port conflicts fail closed with sysexits 78 instead of waiting into a retry loop.

## Reproduction Assessment

Yes. The issue gives concrete Linux/systemd steps, and current tests can reproduce the contested behavior by simulating a supervised lock or EADDRINUSE conflict with a false health probe.

## Solution Assessment

Unclear for the full CLI trigger until the startup path is isolated, but the narrow maintainable direction is clear: keep query commands on RPC/read-only paths and make supervised duplicate-start conflicts exit 78 without unhealthy retry takeover.
