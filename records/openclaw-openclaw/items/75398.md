## Summary

Close as implemented on current main. The central systemd duplicate-start path now exits with sysexits 78 for a healthy existing Gateway before the retry window can run, and the separate Google secrets-reloader crash-loop context is already tracked in #75797.

## What This Changes

The issue reports that v2026.4.29 read-only CLI commands can trigger a systemd-managed Gateway restart by entering a five-second supervised startup retry path instead of exiting with sysexits 78.

## Best Possible Solution

Keep the current main-only sysexits-78 duplicate-start guard, ship it in the next release, and leave the separate Google secrets-reloader startup regression to #75797.

## Reproduction Assessment

Yes. The issue's steps map to a systemd-supervised duplicate Gateway start while an existing /healthz responder is live, and current unit coverage now exercises that branch without entering the retry loop.

## Solution Assessment

Yes. The narrow maintainable fix is to keep read-only CLI commands on query paths and make healthy systemd lock or port conflicts fail closed with exit 78, while retaining only bounded retries for genuinely unhealthy startup races.
