## Summary

Keep open. Current main still contains the supervised systemd lock path that logs the reported wait-and-retry message when the existing Gateway health probe is false, and the regression tests intentionally preserve those retries. The read-only CLI surfaces named in the report are documented and implemented as Gateway RPC or local-store reads, so the remaining bug is to isolate why those invocations reach or provoke the supervised startup path rather than treating #75115 as a complete fix for this post-release report.

## What This Changes

The issue reports that read-only CLI commands in v2026.4.29 can trigger a systemd-managed Gateway restart or port takeover through the supervised startup wait-and-retry behavior.

## Best Possible Solution

Add a narrow Linux/systemd guard so read-only CLI query commands cannot cause Gateway startup or restart, and make systemd-supervised duplicate Gateway starts fail closed with sysexits 78 before retrying into an occupied or unhealthy listener. Preserve unmanaged duplicate-start behavior, keep Gateway query commands on RPC/local read paths, add focused regression tests, and include a user-facing changelog entry crediting the reporter.

## Reproduction Assessment

Yes. The report gives a concrete product repro: run a systemd-managed local Gateway, invoke `openclaw health --json`, and watch Gateway logs and Telegram reconnects. A code-level repro should simulate a systemd-supervised lock or EADDRINUSE conflict with an unhealthy health probe and assert that read-only CLI commands do not retry into startup or port takeover.

## Solution Assessment

Unclear because there is no proposed patch and the exact trigger still needs isolation. The safest repair direction is narrow: keep read-only CLI commands on RPC/local read paths, and fail closed with exit 78 for systemd duplicate-start conflicts instead of broadly redesigning Gateway restart behavior.
