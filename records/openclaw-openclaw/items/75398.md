## Summary

Keep open. Current main still contains the systemd supervised-lock path that waits and retries startup when the existing Gateway health probe is false, and the current tests explicitly assert those retries, so the reported v29 behavior is not implemented away.

## What This Changes

The issue asks OpenClaw to stop read-only CLI commands from triggering a systemd-managed Gateway restart or port takeover and to restore fail-closed exit 78 semantics for duplicate startup conflicts.

## Best Possible Solution

Make read-only CLI commands unable to start or displace the Gateway, and make systemd-supervised duplicate lock or port conflicts fail closed with sysexits 78 instead of waiting into a retry window that can take over the running port.

## Reproduction Assessment

Yes. The issue gives concrete Linux/systemd steps, and current unit coverage can reproduce the contested behavior by simulating a systemd supervisor, a duplicate Gateway lock or EADDRINUSE error, and a false health probe.

## Solution Assessment

Unclear for the full CLI trigger until the startup path is isolated, but the narrow maintainable fix is clear: keep query commands on RPC/read-only paths and make systemd duplicate-start conflicts return 78 without retry takeover.
