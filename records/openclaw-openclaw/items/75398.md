## Summary

Keep open. Current main still has the systemd-supervised duplicate-start retry path that logs the reported `waiting ... before retrying startup` message when `/healthz` is false, and the latest release still asserts that retry behavior; no open or merged PR now supersedes this issue.

## What This Changes

The issue asks that ordinary CLI commands against a systemd-managed gateway connect to the existing gateway without triggering duplicate-start retry, port takeover, or Telegram downtime.

## Best Possible Solution

Repair the systemd gateway lifecycle path so duplicate supervised starts cannot retry into a running gateway takeover, while preserving the existing read-only CLI RPC/local-query surfaces and non-systemd duplicate-start semantics.

## Reproduction Assessment

Yes, at source level: current tests simulate systemd duplicate lock and EADDRINUSE conflicts with a false health probe and assert the retry/sleep behavior. I did not run a live systemd gateway, so this is not a full environment reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is in the gateway/systemd lifecycle path: honor the exit-78 supervisor contract for duplicate-start conflicts and remove or constrain the takeover retry window, rather than adding a new CLI workaround or config flag.
