## Summary

Keep open. Current main still has the supervised Gateway lock path that waits and retries when the existing systemd-managed Gateway does not answer `/healthz`, and tests still lock in that retry behavior; the earlier #75115 changelog entry only proves healthy-systemd duplicate starts exit 78, not this newer CLI-triggered restart report.

## What This Changes

The issue reports that read-only CLI commands on Linux/systemd can trigger Gateway restart or port-takeover behavior in v2026.4.29, causing repeated channel downtime.

## Best Possible Solution

Make read-only CLI query commands unable to start or take over the Gateway, and make systemd-supervised duplicate lock or port conflicts fail closed with sysexits 78 instead of entering the unhealthy-health-probe retry loop. Keep unrelated provider/secret startup regressions on their own reports.

## Reproduction Assessment

Unclear for the complete end-to-end CLI trigger: the issue gives concrete Linux/systemd steps and current tests reproduce the supervised retry branch, but the inspected `health`, `cron list`, and `tasks list` paths do not obviously start the Gateway themselves.

## Solution Assessment

Unclear until the CLI trigger is isolated. The safest likely repair is narrow: keep read-only commands on RPC/local snapshot paths and change systemd duplicate-start conflict handling to return 78 rather than waiting into a retry loop when the port is already occupied.
