## Summary

Keep open. Current main and the latest release still retain the systemd-supervised duplicate-start retry branch when the existing gateway health probe is false, so the reported v29 restart/takeover behavior is not fully resolved.

## What This Changes

The issue reports that read-only CLI commands can trigger a systemd-managed gateway restart by entering a supervised retry path instead of exiting on an existing gateway.

## Best Possible Solution

Make systemd-supervised duplicate lock or EADDRINUSE conflicts fail closed with exit 78 before any retry takeover, while preserving the existing read-only CLI RPC/local query paths.

## Reproduction Assessment

Yes, at source level: current tests simulate the systemd duplicate-lock and EADDRINUSE branches with a false health probe and assert the retry behavior. I did not run a live systemd gateway in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable repair is to preserve query-only CLI behavior and remove the supervised retry window for duplicate-start conflicts under the existing exit-78 systemd contract.

## Repair Work Prompt

Fix the systemd-supervised gateway duplicate-start regression reported at https://github.com/openclaw/openclaw/issues/75398. Current main still retries after GatewayLockError/EADDRINUSE when a supervisor is detected and the existing gateway health probe is false; under systemd this should fail closed with exit 78 rather than waiting and retrying startup. Keep read-only commands such as `openclaw health --json`, `openclaw cron list --json`, and `openclaw tasks list --json` on their existing RPC/local paths; first trace how the reported CLI invocation reaches gateway startup, and do not add a new config option or broaden gateway startup policy. Coordinate with the related systemd timing report at https://github.com/openclaw/openclaw/issues/80696, but do not claim to fix it unless the patch also addresses that timing contract. Update focused tests around `runGatewayLoopWithSupervisedLockRecovery`, EADDRINUSE lock handling, and the systemd unit contract, plus a changelog fix entry if the behavior change is user-visible. Stop if the implementation would require a product decision beyond duplicate-start exit semantics.
