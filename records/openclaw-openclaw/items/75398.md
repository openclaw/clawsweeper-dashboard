## Summary

Keep open. Current main still contains the supervised Gateway lock path that waits and retries duplicate startup when the existing systemd-managed Gateway does not answer `/healthz`, and current tests still assert that unhealthy lock and `EADDRINUSE` conflicts retry. The shipped #75115 changelog/appcast entry only proves the healthy-conflict sysexits 78 case, while this newer report is specifically about the remaining wait/retry behavior and CLI-triggered churn.

## What This Changes

The issue reports that read-only CLI commands can trigger a systemd-managed Gateway restart/takeover loop on v2026.4.29.

## Best Possible Solution

Make read-only CLI query commands unable to start or take over the Gateway, and make systemd-supervised duplicate lock/port conflicts fail closed with sysexits 78 instead of entering the wait/retry loop. Preserve unmanaged duplicate-start behavior where still intentional, and cover the supervised conflict plus command-routing cases with focused tests and a changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Linux/systemd reproduction path, and current tests can simulate the core retry behavior by making the supervised lock health probe return false for lock and `EADDRINUSE` conflicts.

## Solution Assessment

Unclear until the CLI-to-startup trigger is isolated. The safest likely fix is narrow: keep read-only commands on RPC/local read paths and change supervised duplicate-start conflicts to exit 78 without retrying against an occupied port.
