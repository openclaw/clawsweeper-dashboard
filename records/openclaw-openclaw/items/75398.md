## Summary

Keep open. Current main still preserves the systemd-supervised duplicate Gateway startup path that waits and retries when the existing Gateway health probe is false, while the reported read-only commands are intended to query existing state rather than start or take over the Gateway.

## What This Changes

Prevent read-only CLI commands such as `openclaw health --json`, `openclaw cron list --json`, and `openclaw tasks list` from triggering or taking over a systemd-managed Gateway restart in v2026.4.29.

## Best Possible Solution

Add a narrow Linux/systemd guard so read-only CLI query commands cannot invoke Gateway startup, and make systemd-supervised duplicate Gateway lock/EADDRINUSE conflicts fail closed with sysexits 78 instead of waiting into a possible port takeover. Preserve unmanaged duplicate-start behavior and cover both routing and supervised-conflict cases with focused tests.

Yes. The issue gives a concrete product path with a systemd-managed local Gateway plus `openclaw health --json`; code-level reproduction should simulate `runGatewayLoopWithSupervisedLockRecovery` under `systemd` with a duplicate lock/EADDRINUSE error and a false health probe.

## Solution Assessment

The maintainable fix should be narrow: prove read-only commands never call foreground Gateway run, then fail systemd duplicate-start conflicts with 78 rather than redesigning restart behavior broadly.
