## Summary

Keep open. Current `main` still contains the supervised Gateway lock branch that waits and retries startup when the existing systemd-managed Gateway does not answer the health probe, and the regression tests still assert that retry behavior for unhealthy lock and `EADDRINUSE` conflicts. The issue is recent, concrete, and not protected, so it should become a focused repair candidate rather than be closed from the earlier #75115 changelog entry.

## What This Changes

The issue reports that read-only CLI commands such as `openclaw health --json`, `openclaw cron list --json`, and `openclaw tasks list` can trigger a systemd-managed Gateway restart/port takeover on v2026.4.29.

## Best Possible Solution

Make read-only CLI query commands unable to start or take over the Gateway, and make systemd-supervised duplicate lock/port conflicts fail closed with sysexits 78 instead of waiting into a retry loop. Preserve unmanaged duplicate-start behavior and cover the supervised conflict plus command-routing cases with focused tests and a changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete Linux/systemd path: run `openclaw-gateway.service`, invoke read-only CLI commands, and watch the Gateway log for the systemd wait/retry branch and channel reconnects. Code-level reproduction should simulate systemd-supervised lock and `EADDRINUSE` conflicts with a false health probe.

## Solution Assessment

Unclear until the read-only CLI to Gateway-startup trigger is isolated. The safest likely repair is narrow: keep these commands on RPC/local read paths and make supervised duplicate-start conflicts exit 78 without retrying against an occupied port.
