## Summary

Keep open. Current main has an adjacent systemd exit-78 fix for #75115, but it still retains the supervised lock branch that logs a 5s wait and retries startup when the health probe is false; the existing tests assert that retry behavior for both lock and EADDRINUSE conflicts.

## What This Changes

The issue reports that v2026.4.29 read-only CLI commands against a systemd-managed Gateway can start a duplicate Gateway, hit the supervised wait/retry path, and disrupt the existing Telegram-connected service.

## Best Possible Solution

Keep read-only CLI commands on RPC or local snapshot paths, and make systemd-supervised duplicate lock or occupied-port conflicts fail closed with sysexits 78 after a single classification instead of retrying full Gateway startup through the 5s loop.

## Reproduction Assessment

Yes at code level. The issue gives concrete Linux/systemd steps, and current tests can reproduce the contested wait/retry behavior by simulating a systemd lock or EADDRINUSE conflict with a false health probe.

## Solution Assessment

No, current main is not yet the best solution for this issue. The #75115 fix improved exit codes for systemd conflicts, but preserving the unhealthy-probe wait/retry branch leaves the reported restart/takeover surface actionable.
