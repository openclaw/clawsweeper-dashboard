## Summary

Keep open. Current main still has real, source-backed recovery gaps: cron task cancellation can hit the unsupported-runtime path, offline CLI maintenance is deliberately non-authoritative for cron liveness, TaskFlow cancellation still stops when child tasks remain active, and recent comments plus open focused PRs show the stale-running maintenance/reload-blocker problem is only partially addressed.

## What This Changes

The issue requests operator recovery for stuck running tasks through `tasks cancel`, `tasks maintenance --apply`, cron deletion/restart cleanup, and non-deadlocking TaskFlow cancellation.

## Best Possible Solution

Define one operator recovery contract: gateway-owned maintenance reconciles proven-stale cron and task rows, CLI cancellation and maintenance either coordinate with the live gateway or use a clearly bounded offline authority model, and TaskFlow cancellation has a non-deadlocking recovery path for unrecoverable children.

## Reproduction Assessment

Yes, source-backed. On current main, a running cron task still falls through a non-CLI unsupported-cancel path, offline CLI maintenance is non-authoritative for cron liveness, and TaskFlow cancel still returns when children remain active.

## Solution Assessment

No. The current implementation is a useful partial fix, but the remaining behavior needs a maintainer-approved recovery contract rather than an isolated age-only prune or another one-off cancel escape hatch.
