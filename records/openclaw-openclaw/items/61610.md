## Summary

Keep open. Current main has useful partial recovery for stale cron records and bounded restart deferral, but the reported operator recovery path is still incomplete: cron task cancellation still falls through the unsupported runtime branch, offline CLI maintenance is intentionally not cron-authoritative, and TaskFlow cancellation can still remain blocked by active child tasks.

## What This Changes

The issue requests a reliable recovery path for stuck running background tasks, including task cancellation, stale-running maintenance, cron orphan cleanup, and restart/update behavior that does not wedge on zombie task state.

## Best Possible Solution

Define and implement an explicit operator recovery model: Gateway-authoritative maintenance should reconcile proven-stale cron tasks, the CLI should expose a deliberate safe force or Gateway-backed cancel path for cron records, TaskFlow cancellation should have a non-deadlocking path for unrecoverable children, and restart/update flows should continue to bound active-work waits with actionable diagnostics.

## Reproduction Assessment

Yes. A source-backed reproduction exists on current main: a running cron task still reaches the unsupported non-CLI/non-ACP/non-subagent cancel branch, and a stale cron task under offline CLI maintenance remains running because the CLI is not cron-runtime-authoritative.

## Solution Assessment

No. The current implementation is a good partial fix for Gateway-owned cron reconciliation and restart deferral, but it is not the complete requested recovery path because cron cancellation and an explicit offline or Gateway-backed force mode remain unresolved.
