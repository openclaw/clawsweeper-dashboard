## Summary

Keep open. Current main has partial recovery for stale cron records and bounded restart deferral, but cron task cancellation still falls through the unsupported runtime path, offline CLI maintenance remains deliberately non-authoritative for cron liveness, and TaskFlow cancellation can still block on active children.

## What This Changes

The issue requests manual and automatic recovery for stuck running cron and TaskFlow task records, including CLI cancellation, maintenance reconciliation, cron deletion cleanup, and restart cleanup.

## Best Possible Solution

Define an explicit operator recovery model where Gateway-authoritative maintenance reconciles proven-stale cron records, the CLI exposes a deliberate Gateway-backed or force-mark terminal path for cron tasks, and TaskFlow cancellation has a non-deadlocking recovery path for unrecoverable children.

## Reproduction Assessment

Yes. A source-backed reproduction exists on current main: a running cron task still reaches the unsupported non-CLI/non-ACP/non-subagent cancel branch, and stale cron tasks under offline CLI maintenance are preserved because the CLI is not cron-runtime-authoritative.

## Solution Assessment

No. The current implementation is a useful partial fix for Gateway-owned cron reconciliation and bounded restart waits, but it does not provide the complete requested operator recovery path for cron cancellation or blocked TaskFlow child cancellation.
