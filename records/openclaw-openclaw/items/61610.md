## Summary

Keep open. Current main has shipped important partial fixes for CLI task cancellation, runtime-aware cron/CLI maintenance, stale cron session pruning, and bounded reload deferral, but the full operator recovery request is still not implemented: cron tasks still fall through the unsupported cancel path, offline CLI maintenance is intentionally not cron-runtime-authoritative, and TaskFlow cancel can still deadlock on children that remain active.

## What This Changes

The issue requests recovery for zombie background tasks by making task cancel, task maintenance, cron deletion/startup cleanup, and restart/reload drains handle stale running task and TaskFlow records.

## Best Possible Solution

Define a clear operator recovery contract: Gateway-owned maintenance reconciles proven-stale cron records, `tasks cancel` has an intentional cron or force-mark terminal path, and TaskFlow cancellation has a non-deadlocking recovery path for unrecoverable children.

## Reproduction Assessment

Yes, source-backed. On current main, cron task cancellation can still reach the unsupported runtime branch, and offline `openclaw tasks maintenance --apply` still lacks cron runtime authority for empty in-process active-job state.

## Solution Assessment

No. The current implementation is a useful partial fix, but the remaining request spans cron cancellation authority, offline recovery semantics, and TaskFlow child-cancel behavior, which need a narrower maintainer-approved design rather than an age-only prune.
