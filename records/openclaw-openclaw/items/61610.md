- `impact:session-state`: The core failure is persisted task and TaskFlow state drifting from actual runtime liveness.

## Summary

Keep open. Current main has several partial fixes and better diagnostics, but the central recovery contract is still incomplete: cron cancellation can still be unsupported, offline maintenance is intentionally non-authoritative for cron liveness, ordinary stale subagent rows with backing session entries can still be retained, TaskFlow cancellation can still stop on active children, and the linked TaskFlow freshness PR is still open.

## What This Changes

The issue asks for stuck `running` task recovery across `tasks cancel`, `tasks maintenance --apply`, gateway restart/reload drain, cron deletion cleanup, and TaskFlow cancellation.

## Best Possible Solution

Define one operator recovery contract: gateway-owned maintenance reconciles proven-stale cron/task rows, CLI cancellation either coordinates with the live gateway or states its offline limits clearly, TaskFlow cancellation gains a non-deadlocking recovery path, and reload/restart gates ignore or bound impossible stale blockers.

## Reproduction Assessment

Yes, source-backed. Current main still has paths where cron cancel is unsupported, offline cron maintenance intentionally retains rows, normal subagent rows with backing session entries can remain `running`, and TaskFlow cancel can return while children remain active.

## Solution Assessment

No. Current main is only partial; the best fix needs a maintainer-approved recovery contract instead of another isolated prune threshold, cancel escape hatch, or restart-drain tweak.
