## Summary

Keep open: current `main` still does not implement a `PAUSED` `HEARTBEAT.md` sentinel, and https://github.com/openclaw/openclaw/pull/81276 is the open implementation path with active maintainer product-boundary discussion. The documented controls today are `0m`, CLI disable, and empty-file skipping, so this should not be closed before that PR is either landed or explicitly rejected.

## What This Changes

The issue requests a lower-cost idle heartbeat path by respecting a `PAUSED` `HEARTBEAT.md` state, making heartbeat disable controls clearer, and preserving heartbeat disable configuration across restarts.

## Best Possible Solution

Use the linked PR to make one explicit heartbeat pause contract: either land a documented `PAUSED` sentinel with diagnostics and real runtime proof, or keep `0m`/CLI disable as canonical and ask for a separate source-level repro of any config persistence failure.

## Reproduction Assessment

Partially yes: source inspection gives a high-confidence path for `HEARTBEAT.md` containing `PAUSED` being treated as ordinary non-empty content and not skipped. The broader credit-burn number and hosted config-overwrite claim are not reproduced from the available evidence.

## Solution Assessment

Unclear: adding a `PAUSED` sentinel at preflight is technically plausible, but it is a new user-facing contract and maintainers are already weighing it against the existing `0m` and `openclaw system heartbeat disable` controls.
