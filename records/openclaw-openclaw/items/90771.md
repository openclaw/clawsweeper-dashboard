## Summary

Keep open: current main and v2026.6.1 already have generic restart-drain, pending-final, and Codex app-server failure-message machinery, but the reported v2026.6.1 Telegram/systemd SIGTERM window still matches a remaining source path where shutdown can close the Codex client before a channel-visible final reply is delivered. ## What This Changes

The issue reports that a gateway restart or SIGTERM during an active Telegram-backed Codex turn can drop the final channel reply after `model.completed`.

## Best Possible Solution

Make restart/shutdown handling durable for active channel-backed Codex turns: either drain until the final reply is delivered, or persist and deliver a clear interruption/final message after restart, with coverage for the SIGTERM/systemd path and Telegram/WebChat-style channels.

## Reproduction Assessment

Yes, source inspection gives a credible reproduction path, but I did not run it: an active Codex channel turn plus SIGTERM without a restart intent can close as `gateway stopping`, while the Codex client-close handler records the same promptError reported here. The live issue logs provide repeated real-environment evidence on v2026.6.1.

## Solution Assessment

No current-main close is justified. The existing generic restart-drain and failure-message paths are useful, but the best fix needs to cover the supervisor/SIGTERM restart window and durable channel delivery, not only in-process SIGUSR1 or already-delivered pending finals.
