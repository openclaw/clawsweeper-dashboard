## Summary

Keep open: current main still has a raw `innerHTML` toolbar helper, but source review did not find a current user-controlled icon path. This is a low-severity security-hardening task rather than a proven exploitable XSS.

## What This Changes

The issue asks OpenClaw to remove or constrain `createToolbarButton`'s raw `iconMarkup` string so future diff-viewer callers cannot turn it into an XSS sink.

## Best Possible Solution

Keep toolbar icons as a closed internal set, preferably by passing icon IDs or DOM-created SVG elements instead of raw strings, and update the bundled viewer runtime alongside focused coverage.

## Reproduction Assessment

No high-confidence exploitable reproduction path is present. Source inspection confirms the `innerHTML` sink, but current callers are hardcoded local icon factories and no user-controlled icon path was shown.

## Solution Assessment

Yes, the recommended direction is maintainable if it closes the helper boundary instead of sanitizing arbitrary future strings. A closed icon enum or DOM-created SVG factory is safer than preserving raw caller-supplied markup.
