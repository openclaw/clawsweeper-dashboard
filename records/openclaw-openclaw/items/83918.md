## Summary

Keep open: current main still has the reported raw toolbar `innerHTML` sink, and an open linked PR already proposes the narrow sealed-icon fix. The issue should close only after that PR lands or maintainers choose a different hardening path.

## What This Changes

The issue requests removing the diffs viewer toolbar helper’s raw `iconMarkup: string` to `button.innerHTML` path so future dynamic icon markup cannot become an XSS sink.

## Best Possible Solution

Review and land the linked sealed-icon PR, or replace it with an equivalent internal icon-name or DOM-created SVG implementation that removes raw caller-supplied markup from the helper boundary.

## Reproduction Assessment

No for a high-confidence exploitable XSS path; yes for the source-level sink. Current main still assigns `params.iconMarkup` through `innerHTML`, but source review found only hardcoded local icon factories reaching that parameter today.

## Solution Assessment

Yes, the best solution direction is to close the helper boundary with a sealed icon name or DOM-created SVG factory. The discussion is right that DOMParser alone would not be sufficient sanitization, and the linked PR is the active implementation candidate.
