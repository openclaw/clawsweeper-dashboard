## What This Changes

The issue requests hardening the diffs viewer toolbar helper so future callers cannot pass arbitrary icon markup into `button.innerHTML`.

## Best Possible Solution

Review and land or reject the linked sealed-icon implementation; if rejected, keep the helper boundary closed with known icon IDs or DOM-created SVG nodes rather than caller-supplied markup.

## Reproduction Assessment

No high-confidence exploitable reproduction path is established. The source-level sink is clear, but current call sites are hardcoded local icon factories and no user-controlled icon path was shown.

## Solution Assessment

Yes, the closed icon-name direction is the best maintainable shape because it removes the unsafe string boundary instead of sanitizing arbitrary future markup.
