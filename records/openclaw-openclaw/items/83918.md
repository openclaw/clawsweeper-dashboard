## Summary

Current main still contains the `createToolbarButton` raw `innerHTML` assignment, and the live issue has an open closing PR that specifically owns the proposed sealed-icon fix, so this should remain open until that PR lands or is rejected.

## What This Changes

The issue requests security hardening for the diffs viewer toolbar so `createToolbarButton` no longer accepts arbitrary SVG markup and writes it through `innerHTML`.

## Best Possible Solution

Review and land the linked sealed icon-name implementation, or replace it with an equivalent closed internal icon factory that removes raw caller-supplied markup from the toolbar helper and updates the served runtime artifact.

## Reproduction Assessment

Yes for the source-level sink: current main shows `iconMarkup: string` flowing into `button.innerHTML`. No for a live exploit path: current callers are hardcoded local icon factories and the payload schema has no icon markup field.

## Solution Assessment

Yes, the best direction is the proposed closed icon set or DOM-created SVG factory, not parsing arbitrary future strings. The issue should remain open while the linked implementation PR is reviewed because it uses closing syntax and has not merged.
