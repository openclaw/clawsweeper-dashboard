## Summary

This PR is useful Feishu observability work, but it is not merge-ready: it conflicts with current main, lacks required real Feishu behavior proof, and its new success log can misreport fallback direct sends as replies after current-main reply fallback changes.

## What This Changes

The PR adds structured Feishu outbound success logging for text, card, and media reply dispatches and tests two text logging paths.

## Best Possible Solution

Land a rebased Feishu observability change that logs each successful send using the actual delivery operation/result, with focused tests and redacted live Feishu runtime proof.

## Reproduction Assessment

Yes at source level for the PR defect: current main can fall back from a Feishu reply to a direct send for unavailable targets, while the PR logs `reply` solely from the requested reply target. I did not run tests because this review is read-only and must not create artifacts.

## Solution Assessment

No for the patch as-is: delivery-result logs should report the actual operation selected after fallback, not only the originally requested reply metadata. A narrow send-result field or dispatcher helper should carry that fact without changing routing semantics.
