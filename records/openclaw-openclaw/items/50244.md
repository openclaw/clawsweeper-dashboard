## Summary

Keep open: current main still lacks Feishu outbound success logging, but this PR is not merge-ready because the new log mode is based on the requested reply target rather than the actual post-fallback delivery operation, and real Feishu behavior proof is still missing.

## What This Changes

The PR adds Feishu reply-dispatcher success logging for outbound text, card, and media sends and updates dispatcher tests to assert some log output.

## Best Possible Solution

Land a rebased Feishu observability change that logs the actual send-vs-reply operation after fallback, with focused regression tests and redacted live Feishu log proof.

## Reproduction Assessment

Yes, at source level for the PR defect: sendReplyOrFallbackDirect can return a direct fallback result while the proposed log mode is derived only from sendReplyToMessageId. I did not run tests because this is a read-only review.

## Solution Assessment

No for this patch as-is: delivery-result logs should report the actual operation selected after fallback, not only the originally requested reply metadata. A narrow result field or dispatcher helper should carry that fact without changing routing semantics.
