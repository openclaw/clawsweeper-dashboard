Keep this PR open: the Feishu outbound success logging is still useful and not already implemented on current main, but the branch is conflicting, lacks real behavior proof, and has a diagnostic correctness bug where fallback direct sends can be logged as replies.

## What This Changes

The PR adds Feishu reply-dispatcher success logs for outbound text, card, and media sends plus focused tests for some text log output.

## Best Possible Solution

Land a rebased Feishu observability change that logs the actual successful delivery mode after fallback, with focused regression coverage and redacted live Feishu log proof.

## Reproduction Assessment

Yes at source level for the PR defect: current main can fall back from Feishu reply to direct send for withdrawn/not-found targets, while the PR derives log mode from the requested reply target. I did not run tests because this review is read-only.

## Solution Assessment

No for the patch as-is: delivery-result logs should report the actual operation selected after fallback, not only the originally requested reply metadata. A narrow result field or dispatcher helper should carry that fact without changing routing semantics.
