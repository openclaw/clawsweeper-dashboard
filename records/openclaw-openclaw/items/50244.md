## What This Changes

The PR adds Feishu reply-dispatcher success logs for outbound text/card/media deliveries and tests text send/reply log output.

## Best Possible Solution

Land a narrow Feishu change that logs actual dispatcher delivery outcomes for text, card, and media sends after fallback resolution, with messageId coverage, regression tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows non-thread Feishu replies can fall back to direct create, while the PR logs mode from sendReplyToMessageId, so the misleading fallback log path is high-confidence without live Feishu access.

## Solution Assessment

No for the PR as-is. The safer fix is to have the send helpers return or expose the actual delivery mode after fallback and have the dispatcher log that structured outcome instead of deriving mode from requested reply metadata.
