## Summary

Keep open. Current main still lacks the requested Feishu delivery-result observability, but this PR remains unsafe to merge because it can log a fallback direct send as a reply and the external PR only provides test-command proof.

## What This Changes

The PR adds Feishu reply-dispatcher success logging for outbound text, card, and media sends, plus tests for text send/reply log output.

## Best Possible Solution

Land a narrow Feishu change that logs actual dispatcher delivery outcomes after fallback resolution for text, card, and media paths, with focused fallback coverage.

## Reproduction Assessment

Yes for the blocking review finding. Source inspection and current tests show non-thread withdrawn replies can fall back to create, while the PR logs mode from requested reply metadata.

## Solution Assessment

No for the PR as-is. The maintainable fix is to expose or return the actual send mode from the send/fallback helpers and log that structured outcome, rather than deriving mode from replyTo metadata.
