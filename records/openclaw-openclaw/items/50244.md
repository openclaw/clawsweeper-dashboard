## Summary

Keep open. The PR addresses a real Feishu observability gap, but the live branch is conflicting, lacks contributor-provided real Feishu proof, and the new success log can misclassify a fallback direct send as a reply success.

## What This Changes

The PR adds Feishu outbound text/card/media delivery-result logging in the reply dispatcher and test assertions for text send/reply log output.

## Best Possible Solution

Land a rebased Feishu observability change that logs the actual post-fallback delivery operation, with focused regression coverage and redacted live Feishu runtime proof.

## Reproduction Assessment

Yes at source level for the PR defect: current main can fall back from a Feishu reply to a direct send for withdrawn/not-found targets, while the PR logs mode from the requested reply target. I did not run tests because this review is read-only and must not create artifacts.

## Solution Assessment

No for the patch as-is: delivery-result logs should report the actual operation selected after fallback, not only the originally requested reply metadata. A narrow result field or dispatcher helper should carry that fact without changing routing semantics.
