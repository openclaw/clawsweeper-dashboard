## Summary

Keep this PR open. Current main still lacks successful Feishu reply-dispatcher delivery logging, but the PR as written can misreport fallback direct sends as replies, so it needs a small rebase/revision before merge.

## What This Changes

The PR adds Feishu reply-dispatcher success logs for outbound text, card, and media deliveries and updates reply-dispatcher tests to assert text-path log output.

## Best Possible Solution

Add success-side delivery logging in the Feishu reply dispatcher, but make the logged delivery mode reflect the actual helper path after fallback, then cover direct send, reply/thread send, media success, card success, and withdrawn-target fallback in focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is the existing sendReplyOrFallbackDirect path: a non-thread reply to a withdrawn or unavailable target can fall back to direct send while the PR logs reply mode from the requested reply target.

## Solution Assessment

No for the PR as-is. The observability direction is right, but the maintainable fix should have send helpers or their returned result expose the actual delivery mode instead of inferring it only from replyToMessageId.
