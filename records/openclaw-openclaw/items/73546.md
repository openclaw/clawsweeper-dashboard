## Summary

Keep open. Current main still has the TUI continuity gap described in the report: a fresh launch with no explicit session resolves back to the configured main/global key, while `/new` intentionally moves the client to a unique `tui-<uuid>` key that is not remembered for the next launch. The older closed reconnect item covered main-session normalization, but this newer report identifies the remaining unique-TUI-session relaunch path.

## What This Changes

The issue asks for TUI reconnects or relaunches to resume the previous active conversation, or clearly warn/carry context when a fresh session is created.

## Best Possible Solution

Add a TUI-owned resume contract: persist the last active TUI session key per relevant gateway/agent/scope after successful session switches and sends, prefer it on startup only when no explicit session was supplied, validate it through existing session/history APIs, and show visible resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: launch TUI without `--session`, create or switch to a `tui-<uuid>` session with `/new`, exit, then relaunch without `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable repair belongs in the TUI startup/session-action boundary using existing `sessions.list` and `chat.history`; changing gateway reset/archive behavior would be broader than this continuity bug requires.
