## Summary

Keep open. Current main still starts a fresh `openclaw tui` without `--session` from the configured main/global key and then loads history for that selected key, while `/new` TUI conversations live under separate `tui-<uuid>` keys that are not persisted as the next startup target. #41168 overlaps, but it is broader and mixes in storage bloat; this issue remains the cleaner focused repair target.

## What This Changes

The issue requests that TUI reconnect or relaunch resume the previous active conversation session, carry context forward, or visibly warn when starting fresh.

## Best Possible Solution

Add a TUI-owned resume contract: persist the last active TUI session key in scoped local state after successful session switches and sends, read it on startup only when no explicit `--session` was supplied, validate it against the current agent/session scope, and show a clear resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start TUI without `--session`, create or switch to a `tui-<uuid>` session, exit, then relaunch without `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state; changing gateway reset/archive semantics would be broader than needed for the observed continuity gap.
