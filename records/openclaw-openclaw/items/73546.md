## Summary

Keep open. Current main still resolves a fresh TUI launch with no explicit session back to main/global, while `/new` creates an isolated `tui-<uuid>` session that is not remembered for the next process start, so the reported conversation-continuity gap remains valid.

## What This Changes

The issue asks the TUI to resume the previous active conversation after reconnect or relaunch, or visibly warn/carry context when a fresh session is started.

## Best Possible Solution

Add a TUI-owned resume contract: remember the last active TUI session key per relevant agent/scope after successful session switches and sends, prefer it on startup only when no explicit session was supplied and the key is still valid, and show a visible resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start TUI without `--session`, create or switch to a `tui-<uuid>` session, exit, then relaunch without `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state; changing gateway reset/archive behavior would be broader than this continuity bug requires.
