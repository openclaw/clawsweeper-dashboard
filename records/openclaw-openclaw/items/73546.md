## Summary

Keep open. Current main still resolves a fresh `openclaw tui` launch without `--session` to the configured main/global session and loads history only for that selected key, so a prior `tui-<uuid>` conversation remains hidden after relaunch. The older #41168 overlaps, but this issue is the cleaner focused repair candidate for the TUI continuity bug.

## What This Changes

The issue asks for TUI reconnects or relaunches to resume the prior conversation session, or clearly warn/carry context when starting a fresh session.

## Best Possible Solution

Add a TUI-owned resume contract: persist the last active TUI session key in scoped local state, update it after successful sends and explicit session switches, use it on startup only when no explicit `--session` was supplied and the key is valid for the current agent/session scope, and show a visible resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to launch TUI with no `--session`, switch or create a `tui-<uuid>` session, exit, then relaunch with no `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state; changing gateway reset/archive semantics would be broader than needed for the observed continuity gap.
