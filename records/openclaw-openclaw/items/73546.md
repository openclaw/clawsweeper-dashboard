## Summary

Keep open. Current main still resolves a fresh TUI launch without `--session` back to the configured main/global session, while a prior `/new` TUI conversation remains under a separate `tui-<uuid>` key and is not reselected on the next process start. #41168 overlaps, but it is broader; this issue is the cleaner focused repair target for the TUI continuity bug.

## What This Changes

The issue requests that TUI relaunch or reconnect resume the previous active conversation session, carry context forward, or visibly warn when starting fresh.

## Best Possible Solution

Add a TUI-owned resume contract: persist the last active TUI session key in scoped local state after successful session switches and sends, read it on startup only when no explicit `--session` was supplied, validate it against the current agent/session scope, and show a clear resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start TUI without `--session`, create or switch to a `tui-<uuid>` session, exit, then relaunch without `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state; changing gateway reset/archive semantics would be broader than needed.
