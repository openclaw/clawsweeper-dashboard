## Summary

Keep open. Current main still resolves a TUI launch without an explicit session from `opts.session` back to the configured main/global key, while `/new` creates a unique `tui-<uuid>` key that is only held in memory for that process. That leaves the reported relaunch path unresolved: the prior unique TUI conversation remains stored, but the next launch does not select it or warn that it fell back.

## What This Changes

The issue asks for TUI reconnects or relaunches to resume the previous active conversation, or to clearly warn/carry context when a fresh session is created.

## Best Possible Solution

Add a TUI-owned resume contract: persist the last active TUI session key after successful session switches and sends, prefer it on startup only when no explicit session was supplied, validate it through existing session/history APIs, and surface resumed/fallback/fresh-session status in the TUI.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to launch TUI without `--session`, create or switch to a `tui-<uuid>` session with `/new`, exit, then relaunch without `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing `sessions.list` and `chat.history`; changing gateway reset/archive behavior would be broader than this continuity bug requires.
