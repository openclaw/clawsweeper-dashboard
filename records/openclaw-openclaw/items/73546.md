## Summary

Keep open. Current `main` still defaults a fresh `openclaw tui` launch without `--session` to the configured main/global session, while `/new` and manual session switches can move the user to a unique `tui-<uuid>` key that is not persisted as the next launch target. The older #41168 overlaps, but this issue is the narrower TUI continuity repair candidate.

## What This Changes

The issue requests that `openclaw tui` resume the previous active TUI conversation after reconnect or warn/carry context when it starts a fresh session.

## Best Possible Solution

Land a TUI-owned resume contract: persist the last active TUI session key in scoped local state, validate it against the current agent/session scope on startup when `--session` is absent, load that transcript when valid, and show a visible resumed/fallback/fresh-session status without changing gateway reset/archive semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to launch TUI without `--session`, switch to or create a `tui-<uuid>` session, exit, then relaunch without `--session`; current startup resolves the empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing session list/history APIs plus scoped local state; changing gateway transcript reset/archive behavior would be broader than needed for the observed continuity gap.
