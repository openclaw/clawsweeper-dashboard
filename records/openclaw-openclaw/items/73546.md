## Summary

Keep open. Current main still resolves a fresh TUI launch from empty `opts.session` back to the configured main/global session, while `/new` still creates unique `tui-<uuid>` sessions without any persisted last-active TUI session readback. That means a relaunch can load history for a different selected key and hide the previous conversation transcript.

## What This Changes

The issue requests that the TUI resume the previous active conversation after reconnect or relaunch, or clearly warn/carry context when it starts a fresh session.

## Best Possible Solution

Add a TUI-owned resume contract: remember the last active TUI session key per relevant agent/scope after successful session switches and sends, prefer it on startup only when no explicit session was supplied and the key is still valid, and show visible resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: launch TUI without `--session`, create or switch to a `tui-<uuid>` session with `/new`, exit, then relaunch without `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing `sessions.list` and `chat.history` APIs; changing gateway reset/archive behavior would be broader than the observed continuity bug requires.
