## Summary

Keep open. Current main still has the TUI continuity gap: a fresh launch without an explicit session resolves to the configured main/global session, while a prior `tui-<uuid>` session selected by `/new` is not persisted or restored, so history remains tied to the wrong selected key.

## What This Changes

The issue asks the TUI to resume the previous conversation after reconnect or relaunch, or at least warn/carry context when starting a fresh session.

## Best Possible Solution

Add a TUI-owned resume contract: remember the last active TUI session key per relevant agent/scope after successful session switches and sends, prefer it on startup only when no explicit session was supplied and the key is still valid, and show visible resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: start TUI without `--session`, create or switch to a `tui-<uuid>` session, exit, then relaunch without `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable solution is in the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state; changing gateway reset/archive behavior would be broader than the observed continuity bug requires.
