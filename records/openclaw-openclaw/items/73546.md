## Summary

Keep open. Current main still has the TUI continuity gap: a launch without an explicit session derives the active key from `--session`/defaults, while `/new` moves the running client to a unique `tui-<uuid>` key that is only held in memory and therefore is not preferred on the next process start.

## What This Changes

The issue asks the TUI to resume the previous active conversation after reconnect or relaunch, or at least warn/carry context instead of silently starting with an empty session.

## Best Possible Solution

Add a TUI-owned resume contract: persist the last active TUI session key per relevant gateway/agent/scope after successful session switches and sends, prefer it on startup only when no explicit session was supplied, validate it through existing session/history APIs, and show clear resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to launch the TUI without `--session`, create or switch to a `tui-<uuid>` session with `/new`, exit, then relaunch without `--session`; current startup resolves empty input back to the default session before loading history.

## Solution Assessment

Yes. The best fix is narrow TUI session-selection persistence plus validation; changing gateway reset/archive behavior or reverting `/new` isolation would be broader and would risk regressing the multi-client isolation fix.
