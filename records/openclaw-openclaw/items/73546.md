## Summary

Keep open. Current main still resolves a no-argument TUI launch back to the configured main/global session and has no TUI-owned restore path for the last active `tui-<uuid>` session, so the reported silent continuity loss remains actionable.

## What This Changes

The issue asks the TUI to resume the previous conversation after reconnect or relaunch, or clearly warn/carry context when starting a fresh session.

## Best Possible Solution

Add a TUI-owned resume contract: remember the last active TUI session key per relevant agent/scope after successful session switches and sends, prefer it on startup only when no explicit session was supplied and the key is still valid, and surface resumed/fallback/fresh-session status in the TUI and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start the TUI without `--session`, create or switch to a `tui-<uuid>` session, exit, then relaunch without `--session`; current startup resolves empty input back to main/global before loading history.

## Solution Assessment

Yes. The narrow maintainable solution is in the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state; changing gateway reset/archive behavior would be broader than the observed continuity bug requires.
