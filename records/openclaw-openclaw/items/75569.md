## Summary

Keep open. Current main still defaults WhatsApp DM access to `pairing` and can send a pairing-code reply to a live unknown DM; the personal-phone setup prompt is only a guided-setup mitigation, not a fix for existing or unguided personal-number configurations.

## What This Changes

The issue asks OpenClaw to avoid using the WhatsApp `pairing` default for personal-number accounts and instead require an explicit allowlist-style personal account intent.

## Best Possible Solution

Define explicit WhatsApp account intent: personal-number accounts should use `allowlist`, owner `allowFrom`, and `selfChatMode`, while pairing remains available only for explicit dedicated/shared bot setups with doctor/docs guidance for existing configs.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction path: default/fallback `dmPolicy` is `pairing`, live unknown DMs take the pairing branch, and the test suite asserts a pairing reply is sent for recent DMs.

## Solution Assessment

No for current behavior. The guided personal-phone prompt is a useful mitigation, but the maintainable fix needs explicit intent/migration semantics so existing and unguided personal-number configs are not left on the pairing default.
