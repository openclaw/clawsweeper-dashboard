## Summary

Keep open. Current main still defaults WhatsApp DM access to `pairing`, and a live unknown DM can send the reported pairing-code reply; the personal-phone setup path is only a mitigation for guided/new configuration, not a fix for existing or unguided personal-number setups.

## What This Changes

The issue asks OpenClaw to avoid default WhatsApp pairing replies for personal-number accounts by using an owner-only allowlist or an explicit setup intent.

## Best Possible Solution

Define explicit WhatsApp account intent: personal-number accounts should persist allowlist/self-chat owner config and existing risky configs should get doctor/setup migration guidance, while dedicated bot accounts can keep explicit pairing.

## Reproduction Assessment

Yes. Current source and regression tests give a high-confidence reproduction path: no explicit WhatsApp `dmPolicy` resolves to `pairing`, and an unknown live DM takes the pairing branch that sends a pairing reply.

## Solution Assessment

No for current behavior. The guided setup prompt is useful, but the narrow maintainable fix needs an explicit intent/default/migration model rather than relying on the current global pairing fallback.
