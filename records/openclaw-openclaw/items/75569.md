## Summary

Keep open. Current main still defaults WhatsApp DM access to pairing and the live unknown-DM path still sends a pairing challenge reply; the setup flow now offers a personal-phone allowlist path, but that is only a guided-setup mitigation and does not resolve the reported default/runtime behavior.

## What This Changes

Request to change personal WhatsApp setups so unknown contacts do not receive pairing-code replies from the user's own number, either by defaulting personal accounts to owner allowlist mode or by requiring an explicit setup intent choice.

## Best Possible Solution

Define an explicit WhatsApp account intent in setup/config. Personal-number accounts should persist dmPolicy "allowlist", selfChatMode, and owner allowFrom; dedicated/shared numbers can use pairing only after an explicit choice, with doctor/docs guidance for existing personal-number configs.

## Reproduction Assessment

Yes. Current-main source and tests provide a high-confidence reproduction path: no WhatsApp dmPolicy resolves to pairing, an unknown live DM takes the pairing branch, and the no-config inbox test asserts that a pairing prompt is sent to the unknown sender.

## Solution Assessment

Unclear. The suggested automatic personal-account detection may not reliably distinguish a personal number from a dedicated OpenClaw number; an explicit account-intent config/setup path with migration guidance looks safer.
