## Summary

Keep open. Current main still defaults/falls back to WhatsApp `pairing` and sends pair-code replies for unknown direct messages; the personal-phone setup path is a mitigation, not a fix for unguided or existing configs.

## What This Changes

The issue asks OpenClaw to stop default WhatsApp pairing replies from personal-number accounts and prefer owner-only allowlist behavior or an explicit setup choice.

## Best Possible Solution

Define and persist WhatsApp account intent so personal-number accounts use owner-only allowlist/self-chat behavior while dedicated accounts explicitly opt into pairing.

## Reproduction Assessment

Yes. Source inspection on current main shows unset WhatsApp policy resolves to `pairing`, and the access-control test path asserts a recent unknown direct message sends a pairing reply when that policy is active.

## Solution Assessment

No. The guided personal-phone setup path helps new users who choose it, but the remaining default/fallback behavior needs an explicit maintainer product/security decision for personal versus dedicated account intent.
