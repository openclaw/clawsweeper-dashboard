## Summary

Keep open. Current main still defaults WhatsApp DM access to `pairing` and can send a live unknown-DM pairing reply; the personal-phone setup prompt is a useful guided-config mitigation, but it does not protect existing or unguided personal-number setups.

## What This Changes

The issue asks OpenClaw to avoid sending default WhatsApp pairing-code replies from personal accounts by making personal-number setups use explicit allowlist/self-owner intent.

## Best Possible Solution

Define explicit WhatsApp account intent in setup/config: personal-number accounts should use `allowlist` with the owner number in `allowFrom`, while dedicated bot accounts should use pairing only after that intent is explicit, with doctor/docs guidance for existing configs.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence reproduction path: empty/default WhatsApp config resolves to `pairing`, and the live non-self DM path creates a pairing request and calls `sendMessage`. I did not run tests because this review is read-only.

## Solution Assessment

No for current main. The guided setup prompt is useful, but the narrow maintainable solution still needs an explicit intent/default/migration policy for existing and unguided personal-number configurations.
