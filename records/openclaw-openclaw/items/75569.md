## Summary

Keep open. Current main still defaults WhatsApp DMs to `pairing` when no explicit policy is configured, and the live-DM access-control path still sends a pairing challenge reply to unknown non-self senders; guided setup has a personal-phone mitigation, but it does not eliminate the reported default/runtime behavior.

## What This Changes

The issue asks OpenClaw to avoid defaulting personal WhatsApp accounts to unsolicited DM pairing replies, either by defaulting personal-number setups to allowlist mode or prompting for personal versus dedicated-number intent during setup.

## Best Possible Solution

Define an explicit WhatsApp account intent in setup/config so personal-number accounts use `dmPolicy: "allowlist"` with the owner number in `allowFrom`, while dedicated/shared bot numbers only use pairing after an explicit choice; pair that with doctor/docs guidance for existing personal-number configs.

## Reproduction Assessment

Yes. Current-main source and tests provide a high-confidence reproduction: no WhatsApp config defaults to `pairing`, an unknown live DM takes the pairing branch, and the no-config inbox test asserts that a pairing prompt is sent to the unknown sender.

## Solution Assessment

Unclear. The setup prompt is a useful partial mitigation, but the proposed automatic personal-account detection needs maintainer judgment because the linked identity alone may not reliably encode personal versus dedicated bot-number intent.
