## Summary

Keep open. Current main still has the default/no-config WhatsApp path that sends a pairing challenge to unknown non-self DMs; guided setup now has a personal-number mitigation, but source and tests show the reported default behavior is not fully eliminated.

## What This Changes

The issue asks OpenClaw to avoid sending WhatsApp pairing codes by default from personal accounts, preferring allowlist plus the owner number or an explicit setup intent prompt while retaining pairing for dedicated bot numbers.

## Best Possible Solution

Define and persist WhatsApp account intent so personal-number setups never send pairing replies to arbitrary contacts, while pairing remains available only for explicit dedicated/shared-bot mode. Existing configs should get a clear setup/doctor/docs path rather than an ambiguous runtime guess.

## Reproduction Assessment

Yes. Current-main source and tests provide a high-confidence reproduction: unset WhatsApp `dmPolicy` resolves to `pairing`, non-self direct messages take the pairing branch and send a WhatsApp reply, and the no-config inbox test asserts a pairing prompt is sent to an unknown sender.

## Solution Assessment

Unclear. The existing guided setup prompt is a good partial mitigation, but it only helps users who run setup and choose personal mode; the proposed automatic connection-identity detection needs maintainer judgment because identity alone may not encode personal versus dedicated-number intent.
