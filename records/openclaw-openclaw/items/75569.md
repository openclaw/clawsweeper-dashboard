## Summary

Keep open. Current main still defaults no-config WhatsApp direct messages to pairing and sends a pairing challenge to unknown non-self senders; guided setup has a personal-number mitigation, but the runtime/default path described by the report remains. The remaining change is security-sensitive and can affect dedicated bot-number pairing behavior, so it needs maintainer product/security follow-up.

## What This Changes

The issue requests changing personal WhatsApp account setup/defaults so unknown contacts do not receive pairing-code replies from the owner’s own number, while preserving pairing for explicit dedicated bot-number use.

## Best Possible Solution

Add an explicit WhatsApp account-intent contract and persist it during setup or repair: personal-number mode should write/maintain allowlist plus self-chat settings and never emit pairing replies to arbitrary contacts, while dedicated/shared bot mode should retain pairing only when explicitly chosen. Cover no-config, skipped-setup, existing-config, and dedicated-number paths with focused tests plus docs/doctor guidance.

## Reproduction Assessment

Yes. Current-main source and tests give a high-confidence reproduction: omit WhatsApp config, receive a non-self direct message from an unknown sender, and the access-control path resolves pairing and sends a WhatsApp pairing prompt instead of silently blocking.

## Solution Assessment

Unclear. The existing guided setup prompt is a good partial mitigation, but automatic identity-based defaulting is not clearly the safest fix; persisting explicit personal-versus-dedicated intent is a narrower and more maintainable direction.
