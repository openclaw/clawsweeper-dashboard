## Summary

Keep open. Current main still defaults WhatsApp DM access to `pairing` when no explicit policy is configured, and the live unknown-DM path still creates a pairing challenge and sends the reply back over WhatsApp. The guided setup and quickstart personal-phone paths mitigate new guided setups, but they do not remove the reported runtime/default behavior for existing or unguided personal-number setups.

## What This Changes

The issue requests changing personal WhatsApp account defaults so unknown contacts are not sent pairing codes from the user's personal number, either by detecting personal-account intent or by prompting during setup and writing an owner-only allowlist baseline.

## Best Possible Solution

Define an explicit WhatsApp account intent in setup/config so personal-number accounts use `dmPolicy: "allowlist"` with the owner number in `allowFrom`, while dedicated/shared bot numbers use pairing only after that intent is explicit. Pair the behavior change with doctor/docs guidance for existing personal-number configs.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction path: no WhatsApp config resolves to `pairing`, an unknown live DM takes the pairing branch, and the no-config inbox test asserts that a pairing prompt is sent to the unknown sender.

## Solution Assessment

Unclear. The existing setup and quickstart allowlist paths are useful mitigations, but the requested automatic personal-account default is not clearly the safest durable fix without an explicit account-intent or migration policy decision.
