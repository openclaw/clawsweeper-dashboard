## Summary

Keep open. Current main still defaults WhatsApp DM access to pairing when no explicit policy is configured, and the live unknown-DM path still creates a pairing challenge and sends the reply back over WhatsApp. The guided setup personal-phone path is a partial mitigation for new prompted setups, but it does not remove the reported default/runtime behavior for existing or unguided personal-number configs.

## What This Changes

Request changing personal WhatsApp account defaults so unknown contacts do not receive pairing-code replies by default, either by detecting personal accounts or by prompting for explicit account intent.

## Best Possible Solution

Define an explicit WhatsApp account intent in setup/config so personal-number accounts use `dmPolicy: "allowlist"` with the owner number in `allowFrom`, while dedicated/shared bot numbers use pairing only after that intent is explicit. Pair the behavior change with doctor/docs guidance and regression coverage for existing personal-number configs.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction path: no WhatsApp config resolves to `pairing`, an unknown live DM takes the pairing branch, and the no-config inbox test asserts that a pairing prompt is sent to the unknown sender.

## Solution Assessment

Unclear. The suggested automatic personal-account detection may help, but the safer durable fix appears to be explicit account intent plus migration/doctor guidance rather than silently changing all WhatsApp defaults.
