## Summary

Keep open: current main and v2026.6.6 still only expose partial spawned-session lineage, not the requested persisted rootSessionKey/session-chain traceId contract. The direct implementation PR is closed unmerged, and the merged ACP work only serializes existing Gateway lineage fields.

## What This Changes

Add first-class session-chain metadata for spawned sessions across persistence, spawn writes, Gateway session listing, and the agent-facing sessions_list tool.

## Best Possible Solution

Land a focused additive core session-chain contract that persists optional rootSessionKey and traceId, derives them consistently for native and ACP child sessions, exposes them through Gateway and agent list projections, documents the fields, and preserves older entries as optional metadata.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 expose only partial spawned-session lineage in the checked model/schema/projection/spawn paths, while rootSessionKey and a session-chain traceId remain absent.

## Solution Assessment

Yes for the owner boundary: this belongs in the core session model, spawn write paths, and Gateway/tool projections because plugins cannot reliably invent canonical lineage after spawn. The remaining blocker is the stable traceId/root semantics and keeping the implementation narrower than the closed source PR.
