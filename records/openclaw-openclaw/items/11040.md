## Summary

Keep open: current main still provides only partial spawned-session lineage. The code exposes `spawnedBy`, `parentSessionKey`, `spawnDepth`, and `childSessions` in some paths, but the requested first-class `rootSessionKey` and session-chain `traceId` contract is still missing from the persistent model, Gateway protocol/projections, agent-facing `sessions_list`, and native/ACP spawn writes.

## What This Changes

The issue requests first-class persisted and exposed spawned-session chain metadata, including parent/root/trace/depth fields.

## Best Possible Solution

Approve a focused additive core session-chain contract that persists optional `rootSessionKey` and `traceId`, derives them consistently for native and ACP child sessions, exposes them through Gateway and agent session-list projections, documents the fields, and leaves traversal/UI work as follow-ups.

## Reproduction Assessment

Yes, source-level: current main’s session entry type, Gateway protocol schema, Gateway row projection, agent-facing `sessions_list` row, and native/ACP spawn writes can be inspected directly and still lack `rootSessionKey` plus a session-chain `traceId`. I did not run a live spawned-session flow because this is a read-only review.

## Solution Assessment

Unclear until maintainers choose trace semantics. A narrow additive session-chain metadata contract is the maintainable direction, but it should align with existing diagnostic trace context and adjacent resolver-lineage work.
