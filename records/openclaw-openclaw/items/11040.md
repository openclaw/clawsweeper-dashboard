## Summary

Current main still implements only partial lineage: parent/depth relationships are available, including ACP metadata, but there is no canonical root-session or chain-correlation identifier. The request remains useful and is not owned by a viable open implementation PR; however, persistent lineage lifecycle and whether chain correlation should be distinct from diagnostic tracing require maintainer product/API judgment before implementation.

## What This Changes

Add canonical persisted parent, root, trace, and depth metadata for spawned session chains and expose it through session APIs.

## Best Possible Solution

Define one lean canonical session-lineage contract with parent, root, depth, and an explicitly named chain-correlation ID; derive it in every native and ACP spawn path, clear or reject stale lineage at top-level lifecycle boundaries, and project the same optional fields through Gateway, agent-tool, and ACP surfaces with multi-hop tests.

## Reproduction Assessment

Yes, at source level. Current main directly shows parent/depth lineage in the persisted model and projections while lacking the requested canonical root and chain-correlation fields; no live runtime execution was necessary to establish that contract gap.

## Solution Assessment

Unclear as proposed: the canonical session model is the right ownership boundary, but maintainers should first decide whether the correlation field is a new session-chain ID or a bridge to another tracing contract, and define stale-lineage reset semantics.
