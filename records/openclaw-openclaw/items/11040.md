## Summary

Keep open: current main still only has partial spawned-session lineage. It stores and exposes spawnedBy, parentSessionKey, spawnDepth, and childSessions in some paths, but the requested rootSessionKey and session-chain traceId contract is still absent from the persistent model, Gateway schema/projections, agent-facing sessions_list rows, and native/ACP spawn writes; the direct implementation PR is closed unmerged, while the later ACP and session-resolve work covers different lineage surfaces.

## What This Changes

The issue requests additive first-class session-chain metadata for spawned sessions, including parent/root/trace/depth fields in storage, spawn writes, and list projections.

## Best Possible Solution

Land a focused additive core contract that persists optional rootSessionKey and traceId, derives them consistently for native and ACP child sessions, exposes them through Gateway and agent session-list projections, documents them, and proves multi-hop reconstruction without changing spawn policy.

## Reproduction Assessment

Yes, for the feature gap: source inspection gives a high-confidence reproduction because current main lacks rootSessionKey and session-chain traceId in the storage type, protocol schema, projections, tool rows, and spawn writes. I did not run a live spawn because the missing contract is directly visible in source.

## Solution Assessment

Yes. The proposed Phase 1 optional metadata contract is the narrow maintainable direction; https://github.com/openclaw/openclaw/pull/73458 only serializes existing ACP lineage fields, while https://github.com/openclaw/openclaw/pull/79962 and https://github.com/openclaw/openclaw/issues/79903 track rotated session-family discovery rather than spawn-chain root/trace metadata.
