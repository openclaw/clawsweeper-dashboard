## Summary

Keep open: current main and the latest release still provide only partial spawned-session lineage. The core model, protocol/schema, Gateway rows, agent-facing list rows, and native/ACP spawn write paths expose spawnedBy, parentSessionKey, and spawnDepth in places, but still lack the requested first-class rootSessionKey and session-chain traceId contract; the direct implementation PR is closed unmerged, and https://github.com/openclaw/openclaw/pull/73458 only covered ACP metadata over existing Gateway fields.

## What This Changes

Requested first-class session/task chain metadata so spawned sessions expose parent, root, trace, and depth information for reconstructing execution chains.

## Best Possible Solution

Land a focused additive core session-chain contract that persists optional rootSessionKey and traceId, derives them consistently for native and ACP child sessions, exposes them through Gateway and agent list projections, documents the fields, and preserves older entries as optional metadata.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 expose only spawnedBy, parentSessionKey, and spawnDepth in the checked session model/schema/projection/spawn paths, while rootSessionKey and a session-chain traceId remain absent.

## Solution Assessment

Mostly yes for the owner boundary: this belongs in the core session model, spawn write paths, and Gateway/tool projections because plugins cannot reliably invent canonical lineage after the fact. The unresolved part is the stable traceId semantics and keeping the implementation narrower than the closed source PR.
