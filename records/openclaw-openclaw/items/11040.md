## Summary

Current main still has only partial spawned-session lineage, so this issue remains necessary. `SessionEntry` persists `spawnedBy`, `parentSessionKey`, and `spawnDepth`, but not a canonical root key or durable multi-hop chain identifier; gateway/tool projections likewise cannot return the full requested contract. The merged ACP lineage work exposes the existing partial fields to ACP clients but does not implement root/trace persistence, and both direct implementation PRs are closed, leaving no open implementation owner.

## What This Changes

Add canonical persisted parent/root/trace/depth metadata for spawned sessions and expose it consistently through gateway, agent-tool, and ACP session projections.

## Best Possible Solution

Define one durable spawn-chain identity contract in the canonical session model, derive parent/root/chain/depth through a shared helper for every supported spawn path, and expose only authorized lineage consistently through gateway, agent-tool, and ACP projections with multi-hop reconstruction tests.

## Reproduction Assessment

Yes at the source-contract level: current main deterministically lacks `rootSessionKey` and a durable session-chain identifier while exposing only partial parent/depth lineage. This is a feature gap, not a failure of an already-defined behavior.

## Solution Assessment

The proposed additive metadata direction is maintainable, but not yet fully specified: maintainers should first define whether chain identity is distinct from diagnostic `traceId`, then require one shared derivation path rather than parallel native and ACP implementations.
