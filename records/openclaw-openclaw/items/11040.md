## Summary

Current main still provides only partial lineage: parent/depth data exists and ACP can project it, but there is no canonical persisted root-session or chain-trace contract across native and ACP spawn paths. The closed source PR is not a viable canonical implementation, and the remaining trace semantics and lineage ownership require maintainer product/API judgment.

## What This Changes

Add optional parent, root, trace, and depth metadata so spawned session chains can be reconstructed through session APIs.

## Best Possible Solution

Define one canonical optional lineage object with explicit root and trace semantics, derive it in a shared spawn-boundary helper for native and ACP children, persist it only on genuine spawned sessions, and expose it consistently through Gateway, sessions_list, and ACP with multi-hop and stale-lineage regression coverage.

## Reproduction Assessment

Yes. Current main directly demonstrates the gap: parent and depth lineage exist, but rootSessionKey and a stable session-chain trace identifier are absent from the persistent model and public session projections.

## Solution Assessment

Unclear. The optional metadata direction is maintainable, but the best implementation depends on a maintainer decision about trace identity, canonical write ownership, and stale-lineage clearing semantics.
