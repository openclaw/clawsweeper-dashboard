## Summary

Close as implemented: current main and the reported-release source preserve the original tagged text for final TTS while stripping it from visible streamed text, and the speech runtime prefers `directives.ttsText` over surrounding visible text before calling the provider.

## What This Changes

The issue reports that `[[tts:text]]...[[/tts:text]]` content is stripped from chat but not passed to the TTS engine when surrounded by normal text.

## Best Possible Solution

Keep the shipped streamed-directive implementation and ask for a new report only if a current install can still show a config- or package-specific failure with redacted effective TTS settings and CLI input logs.

## Reproduction Assessment

No. Current main and the reported `v2026.5.22` source both show the hidden tagged text is preserved for final TTS and preferred over surrounding visible text, and I found no current-main failing path for this exact handoff.

## Solution Assessment

Yes. The existing design is the narrow maintainable solution: strip hidden TTS blocks from visible chat while preserving the original tagged text for final TTS parsing and provider input.
