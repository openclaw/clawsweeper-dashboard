## Summary

Keep open: current main already surfaces the replay-unsafe warning and live tool/lifecycle events, but it does not provide the durable recovery receipt the report asks for after successful tool execution followed by failed continuation or delivery. The adjacent open reports cover restart UI recovery and stale reply locks, but neither is an exact canonical replacement for this field report.

## What This Changes

Request durable, user-visible tool lifecycle and recovery receipts when successful tool execution is followed by failed model continuation or response delivery.

## Best Possible Solution

Design a core durable recovery receipt that persists sanitized tool start/result and terminal continuation state in the session/state store, then exposes it through CLI/UI diagnostics without weakening the existing replay-safety warning.

## Reproduction Assessment

No high-confidence live repro of the reporter's Mac/Gemma failure is available from the issue alone. Source inspection does show the current incomplete-turn warning path and the absence of a durable per-tool recovery receipt on current main.

## Solution Assessment

Unclear until maintainers choose the recovery contract. The direction is plausible, but the best fix should be a designed, redacted core receipt/persistence surface rather than ad hoc logging or a provider-specific retry.
