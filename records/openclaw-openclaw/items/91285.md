## Summary

Keep open: current main still provides a replay-unsafe warning, live tool/item events, aggregate run metadata, restart-recovery markers, and pending-final-delivery state, but not the durable per-tool recovery receipt requested for successful tool execution followed by failed continuation or delivery. The newer restart-recovery fix is adjacent, not a complete replacement for this field report.

## What This Changes

The issue requests deterministic recovery and observability when tool calls complete successfully but the post-tool model continuation or response delivery fails.

## Best Possible Solution

Design a core durable, redacted tool/continuation recovery receipt in session or state storage and expose it through diagnostics/UI while preserving the replay-unsafe retry guard and handling provider empty-output failures separately.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The reporter supplied a smaller 2026.6.5 failure boundary, and source inspection confirms current main can surface the same replay-unsafe incomplete-turn warning without a durable recovery receipt.

## Solution Assessment

Unclear until maintainers choose the recovery contract. A durable receipt is a plausible direction, but the best fix should be a designed core persistence/diagnostic surface rather than ad hoc logging or a provider-specific retry.
