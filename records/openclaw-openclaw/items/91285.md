## Summary

Keep open: current main still only provides a generic replay-unsafe warning plus live tool events and restart-specific recovery, not the durable post-tool recovery receipt requested after successful tool execution followed by failed continuation or delivery.

## What This Changes

The issue reports successful filesystem tool execution followed by failed agent response generation and asks for deterministic recovery state for post-tool continuation or delivery failures.

## Best Possible Solution

Design a core, redacted durable recovery receipt for tool start/result and terminal continuation state, then expose it through session diagnostics/UI without weakening the existing replay-safety warning.

## Reproduction Assessment

No high-confidence current-main live reproduction was performed. The reporter supplied a reduced macOS/Ollama/Gemma sequence, and source inspection confirms the current incomplete-turn warning path plus missing durable receipt, but the exact local provider failure still needs logs or live repro.

## Solution Assessment

Unclear until maintainers choose the recovery contract. A durable, redacted core receipt is the right end state; ad hoc logging or provider-specific retry would not solve the operator recovery uncertainty cleanly.
