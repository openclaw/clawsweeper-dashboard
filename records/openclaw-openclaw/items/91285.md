## Summary

Keep open: current main and v2026.6.6 have the replay-unsafe warning, live tool events, terminal session status persistence, and a retry for replay-safe empty local-backend turns, but they still do not provide the durable per-tool recovery receipt requested after a successful tool action followed by failed continuation or delivery.

## What This Changes

The issue reports successful tool execution followed by failed agent continuation or delivery, leaving users unsure which actions completed before retrying.

## Best Possible Solution

Design a core recovery receipt that persists sanitized tool lifecycle/result and terminal continuation state in the session/state store, then expose it through CLI/UI diagnostics without weakening current replay-safety behavior.

## Reproduction Assessment

No high-confidence current-main live reproduction is available from this read-only review. Source inspection and the reporter's reduced sequence do establish the incomplete post-tool warning boundary and the missing durable receipt surface.

## Solution Assessment

Unclear until maintainers choose the recovery contract. A durable, redacted core receipt is a better direction than ad hoc logs or a provider-specific retry, but its storage/API/UI boundaries need design first.
