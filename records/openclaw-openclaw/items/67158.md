## Summary

Keep open. Current main has recent OpenAI Codex OAuth transport and catalog work, but it still does not implement or prove a fix for ChatGPT/Codex account entitlement rejection of older Codex model ids; the discussion also has a later data point showing the same provider-boundary family with gpt-5.5 before the current transport fixes, so this still needs a deliberate provider-boundary decision rather than cleanup closure.

## What This Changes

The issue requests that OpenAI Codex ChatGPT/OAuth runs stop hard-failing for non-gpt-5.4 Codex models such as gpt-5.1, gpt-5.2, and gpt-5.3, or at least degrade correctly when the backend rejects those model ids.

## Best Possible Solution

Keep this as an OpenAI Codex provider-boundary bug. The maintainable end state is provider-owned handling for this backend error: classify the unsupported ChatGPT-account model response as model unavailable or entitlement-limited for fallback/status, avoid implying synthesized or catalog rows are definitely runnable for the current account, and document the diagnostic path without hiding models that may work for other accounts.

## Reproduction Assessment

Yes, for affected accounts: authenticate with openai-codex ChatGPT/Codex OAuth, select one of the rejected model refs, and observe the backend unsupported-model response. I did not live-verify credentials in this read-only pass, but current source confirms OpenClaw sends the selected model id through to the ChatGPT/Codex Responses backend.

## Solution Assessment

Unclear; there is no proposed patch, and current main is not a terminal solution for the reported entitlement rejection. The safest direction is precise provider error classification and account-aware diagnostics/probing, not a broad hard-coded deletion of old Codex rows.
