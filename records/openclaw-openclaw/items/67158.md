## Summary

Keep open. Current main still lacks a terminal OpenClaw-side handling path for ChatGPT/Codex OAuth backend rejections of older Codex model ids; recent OpenAI Codex fixes cover adjacent OAuth transport and gpt-5.4-mini routing, but not this account-entitlement error family.

## What This Changes

The issue asks OpenClaw to avoid hard-failing when ChatGPT/Codex OAuth rejects openai-codex gpt-5.1, gpt-5.2, or gpt-5.3 variants and to degrade or route safely instead.

## Best Possible Solution

Keep this as OpenAI Codex provider-boundary work. The maintainable end state is account-aware handling of this backend response: classify or probe it so fallback/status can mark the selected model unavailable for the current ChatGPT/Codex account, avoid globally deleting models that may work for other accounts, and document the diagnostic path.

## Reproduction Assessment

Yes, for affected accounts: authenticate with openai-codex ChatGPT/Codex OAuth, select one of the rejected model refs, and observe the backend unsupported-model response. I did not run a live OAuth probe, but source inspection confirms OpenClaw sends the selected model id through to the Codex Responses backend.

## Solution Assessment

Unclear; there is no proposed patch, and current main's adjacent OAuth fixes are not the terminal solution for this entitlement rejection. A provider-owned classifier/probe is safer than hard-coding removal of older Codex rows or treating all ChatGPT-account errors as generic auth failures.
