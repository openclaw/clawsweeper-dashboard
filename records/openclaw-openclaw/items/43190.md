## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the channel-data tenant in archive metadata:** `extensions/msteams/src/monitor-handler/message-handler.ts:792`
  - body: Current Teams ingress treats `activity.channelData.tenant.id` as the reliable tenant source because channel activities often omit `conversation.tenantId`. Forwarding only `conversation?.tenantId` into `ProviderMetadata` can store archives without tenant metadata, so cleanup may use the default tenant token for cross-tenant channels and skip or prune the wrong archive.
  - confidence: 0.91
- **[P1] Declare the archive tools in the manifest:** `extensions/msteams-channel-archive/openclaw.plugin.json:2`
  - body: `registerArchiveTools` registers four agent tools, but this manifest has no `contracts.tools`. The current plugin registry emits a diagnostic and returns before registering tools when that contract is missing, so the archive search/get tools will not load or be discoverable.
  - confidence: 0.94
- **[P2] Resolve cleanup credentials through Teams auth:** `extensions/msteams-channel-archive/src/channel-cleanup.ts:282-292`
  - body: This resolver only succeeds with appPassword/client-secret credentials. Current Teams config supports federated certificate and managed-identity auth, so those supported deployments will silently skip deleted-channel cleanup even when the archive plugin is enabled.
  - confidence: 0.88
- **[P2] Route cleanup Graph calls through guarded helpers:** `extensions/msteams-channel-archive/src/channel-cleanup.ts:301-335`
  - body: These OAuth and Graph requests use raw `fetch` with secrets/bearer tokens instead of the existing Teams guarded fetch/token helpers. That bypasses the established SSRF guard, audit context, User-Agent, timeout, and provider response handling used by other Teams Graph paths.
  - confidence: 0.86
