## Summary

Keep open. Current main still has the two implicated diagnostic gaps: active reply runs are not mirrored into diagnostic active-work state, and `noop/no_active_work` recovery does not clear queued diagnostic state; the related fixes are open PRs rather than merged or shipped behavior.

## What This Changes

The issue reports that Telegram gateway lane diagnostics can remain queued after a completed, failed, or compacted run, causing later substantive messages to receive no-op replies until restart.

## Best Possible Solution

Coordinate the active-reply diagnostics and `noop/no_active_work` recovery fixes, then add a regression and live Telegram proof showing compacted or failed runs reconcile queue state and do not send silent no-op replies to substantive follow-ups.

## Reproduction Assessment

Yes, at source level: current main can leave queued diagnostic state after terminal run progress and can let the detector miss active reply work that recovery sees. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No, a separate automated fix lane is not the best next step right now. The maintainable path is to reconcile and land the existing related diagnostic PRs, then cover the full Telegram compaction/no-op case explicitly.
