## Summary

Keep open: the report is recent and credible, and current main still has a source path where `openclaw status` awaits daemon service summaries that can call Linux `systemctl` without a timeout despite the command exposing `--timeout`.

## What This Changes

The issue reports a Linux npm-global regression where `openclaw status` hangs and exits 124 in v2026.5.18/v2026.5.19 while `openclaw health --json` returns immediately.

## Best Possible Solution

Bound status service/process probes by the command timeout, return degraded/unknown service status on probe timeout, and preserve the fast read-only default status surface.

## Reproduction Assessment

No high-confidence live reproduction was run. The issue gives concrete Ubuntu/npm-global steps, and source inspection shows current `openclaw status` can wait on unbounded Linux `systemctl` service probes.

## Solution Assessment

Yes. A narrow timeout and fail-soft fix in the status service probe path is the maintainable direction; it should not add a config option or change status semantics beyond preventing hangs.
