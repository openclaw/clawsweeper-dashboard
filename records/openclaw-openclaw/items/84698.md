Keep open. The report is recent, externally filed, and matches a current-main source path where `openclaw status` still runs Linux service probes through unbounded `systemctl` subprocesses even though the command exposes `--timeout`.

## What This Changes

The issue reports that after upgrading from 2026.5.7 to 2026.5.18 or 2026.5.19, `openclaw status` hangs until an external timeout while `openclaw health --json` succeeds.

## Best Possible Solution

Bound status service/process probes by the command timeout and fail soft, preserving the fast read-only default status path with regression coverage for a hung Linux service probe.

## Reproduction Assessment

No high-confidence live reproduction was run. The issue gives concrete steps, and source inspection shows a credible unbounded Linux `systemctl` path inside current `openclaw status`.

## Solution Assessment

Yes, a narrow timeout/fail-soft fix in the status service probe path is the best direction. This should not add a config option or change the documented fast status surface beyond preventing hangs.
