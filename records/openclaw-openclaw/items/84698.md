## What This Changes

The issue reports a Linux regression where `openclaw status` hangs and exits 124 after 2026.5.18+, even when `--timeout` is supplied.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94149, or an equivalent fix that bounds Linux service-manager status probes and returns degraded status rows instead of hanging.

## Reproduction Assessment

Yes by source inspection: current `main` forwards `--timeout` into status runtime work but not into service summaries, and Linux service summaries can call unbounded `systemctl` subprocesses. The linked PR also supplies controlled wedged-`systemctl` before/after proof, but I did not independently execute it here.

## Solution Assessment

Yes: the best fix is to thread the existing status timeout through service-summary and systemd read probes while failing soft to degraded output. The open linked PR targets that owner boundary, so the issue should stay open until that PR lands or is replaced.
