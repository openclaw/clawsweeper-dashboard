## Summary

Keep open: this PR targets a real shipped/current OpenAI/Codex configure bug, but the proposed migration can still strand legacy per-model settings on the old model key and adds OpenAI-specific migration policy inside generic provider-auth code.

## What This Changes

The PR changes provider-auth default preservation so OpenAI OAuth/device-code auth can migrate matching legacy `codex/*` or `openai-codex/*` primary model refs to canonical `openai/*`, with regression tests and CI cleanup.

## Best Possible Solution

Land a lossless migration that carries legacy per-model metadata to the canonical key, with focused regression coverage, and place the OpenAI-specific rule behind a maintainer-approved provider-owned or generic migration seam.

## Reproduction Assessment

Yes at source level: configure auth preserves existing defaults, current provider auth preserves any different primary, and OpenAI/Codex auth adds canonical `openai/gpt-5.5`. The linked issue shows the resulting stale primary versus allowed-model mismatch on v2026.5.28.

## Solution Assessment

No. Migrating the primary is the right repair shape, but this branch should make the migration lossless for per-model settings and settle the provider-owned/core seam before merge.
