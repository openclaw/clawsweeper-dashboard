## Summary

Keep open: the PR targets the linked OpenAI/Codex configure mismatch with useful proof and regression coverage, but the current patch can also overwrite a still-supported Codex default when a user only adds OpenAI auth outside the ChatGPT/Codex migration path.

## What This Changes

The PR changes provider-auth default-model preservation so matching legacy Codex/openai-codex primaries are rewritten to canonical openai/* after OpenAI auth, and adds a focused regression test.

## Best Possible Solution

Land a scoped fix that migrates stale legacy Codex defaults only for the ChatGPT/Codex auth flow while preserving current Codex defaults for unrelated OpenAI setup paths, with regression coverage for both.

## Reproduction Assessment

Yes. Source inspection of current main shows preserveExistingDefaultModel keeps a different codex/gpt-5.5 primary while OpenAI auth adds openai/gpt-5.5, and the linked issue shows that mismatched default/allowlist state in a release install.

## Solution Assessment

No. The central migration idea is right, but the PR should scope it to the ChatGPT/Codex OAuth/device-code path or an explicit provider-owned migration signal so unrelated OpenAI setup still preserves current Codex defaults.
