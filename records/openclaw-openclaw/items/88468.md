## Summary

Keep open: the patch addresses the stale primary-model mismatch, but it can still strand existing per-model settings under the legacy Codex key, and the OpenAI-specific migration rule remains an owner-boundary decision before merge.

## What This Changes

The PR changes provider-auth default handling so OpenAI OAuth/device-code auth can migrate matching legacy Codex/OpenAI-Codex primary model refs to canonical OpenAI refs, with regression coverage and merge-refresh cleanups.

## Best Possible Solution

Make the legacy-to-canonical migration lossless for matching per-model settings, add focused coverage for those settings, and resolve whether the OpenAI-specific rule belongs in this core helper or a provider-owned seam.

## Reproduction Assessment

Yes. The linked issue supplies concrete release evidence, and current-main source preserves a different existing primary when `preserveExistingDefaultModel` is true, which explains how `codex/gpt-5.5` can remain primary after OpenAI auth adds `openai/gpt-5.5`.

## Solution Assessment

No. The proposed direction fixes the primary mismatch, but the implementation is not yet lossless for existing per-model settings and still needs maintainer agreement on the OpenAI-specific core/provider boundary.
