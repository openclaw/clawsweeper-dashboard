## Summary

Keep open. Current main still passes Matrix `/keys/upload` duplicate-key 400 responses through unchanged, and #74529 is the focused open fix candidate for the failure mode shown in the report.

## What This Changes

The issue reports Matrix E2EE recovery/setup failing when a recovery-key flow reaches a duplicate one-time-key `/keys/upload` 400.

## Best Possible Solution

Land #74529 or an equivalent narrow Matrix-owned guarded-fetch mitigation for duplicate `/keys/upload` OTK collisions, with tests proving non-collision Matrix errors still pass through.

## Reproduction Assessment

No, not as a live affected-account run. The reporter supplied exact logs, and source inspection shows a duplicate `/keys/upload` 400 would still flow through `createMatrixGuardedFetch` to the pinned SDK’s non-OK error path on current main.

## Solution Assessment

Yes, the narrow transport-boundary direction in #74529 appears to be the best maintainable fix because it can target the duplicate-OTK collision without changing recovery-key trust semantics or hiding unrelated Matrix errors.
