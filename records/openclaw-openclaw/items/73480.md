## Summary

Keep open. Current main still passes the reported Matrix `/keys/upload` one-time-key collision 400 through the guarded fetch path used by `matrix-js-sdk`, and the focused fix remains open in #74529.

## What This Changes

Recover Matrix encrypted accounts when recovery-key/device setup reaches usable backup material but encryption bootstrap fails on a `/keys/upload` one-time-key ID collision.

## Best Possible Solution

Keep this open until #74529 or an equivalent Matrix-owned transport mitigation lands, then verify recovery-key, backup-restore, and encryption-setup flows against an affected account.

## Reproduction Assessment

Yes at source level: a `POST` to `/keys/upload` returning the reported `signed_curve25519:<id> already exists` 400 is buffered and returned unchanged by current main, and `matrix-js-sdk` 41.4.0 throws on non-OK responses. No live affected homeserver/account was exercised in this read-only review.

## Solution Assessment

Yes for the linked direction: the narrow maintainable fix is Matrix plugin transport handling for this exact collision plus regression tests. It should preserve the existing distinction between backup recovery and full cross-signing identity trust.
