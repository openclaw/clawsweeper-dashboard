## Summary

Keep open. Current main and v2026.5.2 still pass the reported Matrix `/keys/upload` one-time-key collision 400 through the guarded fetch path, while open PR #74529 remains the focused implementation candidate; the separate recovery-key/nonzero identity-trust diagnostic is documented behavior.

## What This Changes

The issue reports that Matrix recovery-key verification and encryption setup cannot recover an account after a `/keys/upload` one-time-key ID collision.

## Best Possible Solution

Review and land PR #74529 or an equivalent Matrix-owned transport fix scoped to POST `/keys/upload` collision bodies, then verify recovery-key, backup-restore, and encryption-setup flows while preserving identity-trust diagnostics.

## Reproduction Assessment

Yes for source reproduction, not as a live affected-account run. A POST `/keys/upload` 400 matching the reporter's body is returned unchanged by current `createMatrixGuardedFetch`, and the pinned matrix-js-sdk throws on non-OK responses.

## Solution Assessment

Yes, the linked narrow transport-boundary mitigation appears to be the best current direction if it remains strictly limited to the OTK collision body and leaves other Matrix failures untouched.
