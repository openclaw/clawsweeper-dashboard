## Summary

Keep open. Current main still passes the reported Matrix `/keys/upload` one-time-key collision 400 through the guarded fetch path, and open PR #74529 remains unmerged as the focused fix candidate.

## What This Changes

The issue reports that Matrix recovery-key/device verification and encryption setup fail on an account with usable backup material when `/keys/upload` returns an existing one-time-key ID collision.

## Best Possible Solution

Review and land #74529 or an equivalent Matrix-owned transport fix, then verify recovery-key, backup-restore, and encryption-setup flows against an affected account.

## Reproduction Assessment

No, not as a live affected-account run in this review. The reporter supplied exact failing commands/logs, and source/dependency inspection makes the 400-pass-through path source-reproducible on current main.

## Solution Assessment

Yes, the narrow transport-boundary mitigation proposed by #74529 appears to be the maintainable direction, provided it stays strictly gated to POST `/keys/upload` collision bodies and preserves recovery-key versus identity-trust diagnostics.
