## Summary

Keep open. The documented recovery-key warning explains the incomplete owner trust part, but current main still lacks handling for the duplicate Matrix `/keys/upload` 400 shown in the report, and #74529 is an open unmerged fix candidate.

## What This Changes

The issue reports Matrix recovery-key recovery failing on a Windows Docker install, including a duplicate `signed_curve25519` one-time-key `/keys/upload` 400 during `openclaw matrix encryption setup`.

## Best Possible Solution

Land or replace the narrow Matrix transport mitigation for duplicate OTK key-upload collisions while preserving the documented recovery-key and identity-trust diagnostics.

## Reproduction Assessment

No, not as a live affected-account run. The reporter supplied exact commands/logs, and source inspection shows current main returns duplicate `/keys/upload` 400 responses unchanged through a dependency path that throws on non-OK responses.

## Solution Assessment

Yes, the best direction appears to be the narrow transport-boundary fix proposed in #74529 or an equivalent Matrix-owned patch. Broad recovery-key behavior changes would risk masking the documented distinction between backup usability and full owner trust.
