## Summary

Keep open. Current main still lacks the `/keys/upload` OTK-collision mitigation shown in the report, while Matrix recovery-key handling still intentionally separates backup usability from full cross-signing identity trust. The related open Matrix PRs are implementation candidates, not merged proof that the regression is fixed.

## What This Changes

The issue reports that Matrix recovery with `verify device --recovery-key-stdin` and `matrix encryption setup` cannot complete after a recovery-key attempt, with logs showing usable backup material, missing owner trust, and a `/keys/upload` one-time-key collision.

## Best Possible Solution

Keep this issue open as the Matrix recovery umbrella until maintainers land or replace the focused Matrix fixes, then verify the reported `verify device --recovery-key-stdin` and `matrix encryption setup` sequence against an affected or disposable account before closing it as fixed.

## Reproduction Assessment

Unclear for a full live reproduction: the issue provides concrete commands and logs, and current main still has code paths matching the diagnostic state, but this read-only pass did not run a live Matrix E2EE recovery against an affected homeserver.

## Solution Assessment

Unclear because there is no patch attached to this issue. PR #74529 appears to be the narrowest known fix for the `/keys/upload` OTK collision, while maintainers should decide whether #74509 or a follow-up is also required for the cross-signing trust side.
