## Summary

Keep open: the PR targets a real current-main bug in agent runtime prompt OS metadata, but it should not merge as submitted because the new Darwin helper would synchronously run `sw_vers` on every prompt construction and the contributor proof does not show the target macOS/Tahoe behavior.

## What This Changes

The PR adds `resolveRuntimeOsLabel()` in `src/infra/os-summary.ts` and switches the CLI, embedded attempt, and compaction runtime prompt paths to use it.

## Best Possible Solution

Land the runtime prompt fix with cached/reused macOS product-version resolution, focused Darwin coverage, and real macOS runtime prompt proof; keep or consolidate the sibling diagnostics-only PR through maintainer review.

## Reproduction Assessment

Yes, source-reproducible: current main passes `os.type()` plus `os.release()` into the CLI, embedded attempt, and compaction runtime prompt paths, and the Runtime line renders that value. I did not run a live macOS 26 host in this read-only review.

## Solution Assessment

No as submitted: using `sw_vers -productVersion` for Darwin runtime metadata is the right direction, but the helper should cache or reuse the existing cached product-version lookup before it is called from prompt hot paths.
