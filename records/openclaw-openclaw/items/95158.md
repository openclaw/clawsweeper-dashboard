## Summary

Keep open: the PR targets a real source-reproducible macOS runtime prompt bug, but as submitted it adds an uncached synchronous `sw_vers` call to prompt construction and the proof does not show the changed macOS Runtime line.

## What This Changes

The PR adds `resolveRuntimeOsLabel()` in `src/infra/os-summary.ts` and switches the CLI, embedded attempt, and compaction prompt builders from `os.type() + os.release()` to that helper.

## Best Possible Solution

Land the runtime prompt fix with cached or reused Darwin product-version resolution, focused coverage for the new helper, and redacted macOS terminal/log proof of the corrected Runtime line.

## Reproduction Assessment

Yes, source-reproducible: current main passes `${os.type()} ${os.release()}` into the CLI, embedded attempt, and compaction runtime prompt paths, and `buildRuntimeLine()` renders that value into the Runtime line. I did not run a live macOS 26 host in this read-only review.

## Solution Assessment

No as submitted: using `sw_vers -productVersion` for Darwin runtime metadata is the right direction, but the new helper should cache or reuse the existing cached product-version lookup before prompt hot paths call it.
