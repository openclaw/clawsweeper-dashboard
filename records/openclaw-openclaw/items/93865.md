## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Key backfill to the agent session:** `extensions/mattermost/src/mattermost/monitor.ts:1558-1563`
  - body: `threadsBackfilledThisSession` is a monitor-lifetime Set keyed only by `historyKey`. After the first recovery fetch, `/new` or session reset rotates the agent session but leaves the same history key in the Set, so the next accepted mention in that existing thread skips the only server backfill and can still reply without restored thread context. Use the current session id, or clear the marker on reset, rather than a monitor-lifetime key.
  - confidence: 0.9
- **[P1] Mark active first sightings as serviced:** `extensions/mattermost/src/mattermost/monitor.ts:1564-1575`
  - body: The Set is updated only after a fetch inside the empty-window branch. If an active thread is first seen with pending history, this block skips marking it; after the turn kernel clears that window, the next same-session follow-up sees empty history and performs the full server fetch/injection on a normal turn. Mark the key or session before skipping non-empty windows and add that regression case.
  - confidence: 0.89
- **[P2] Filter the current post before trimming:** `extensions/mattermost/src/mattermost/monitor.ts:1580-1583`
  - body: The thread endpoint can include the current reply, but the PR slices to `historyLimit` before skipping `post.id`. When the current post is among the newest entries, the seeded window can be one entry short and omit an older prior message that should fit the configured limit. Filter out the current post first, then take the last N prior posts before resolving users.
  - confidence: 0.82
