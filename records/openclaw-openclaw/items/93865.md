## Summary

This PR is still a useful candidate for the Mattermost thread-history recovery bug, but it is not merge-ready because the backfill gate still reads the base route session entry instead of the resolved thread session entry used by Mattermost thread turns.

## What This Changes

The PR adds a Mattermost thread fetch helper, backfills empty in-memory thread history before inbound context construction, gates backfills by a stored session id, and adds client/monitor tests.

## Best Possible Solution

Keep the Mattermost-owned server backfill, but derive the marker from the resolved thread session entry, update the monitor-path test to assert that key, and add redacted live or diagnostic proof for /new thread recovery plus same-session no-refetch.

## Reproduction Assessment

Yes, source-level. Current main keeps Mattermost thread history in a process-local map and records thread turns under ctxPayload.SessionKey, while the PR's reset gate can be checked against the thread-scoped session key path.

## Solution Assessment

No, not as patched. The Mattermost plugin is the right owner boundary and the server backfill shape is plausible, but the gate must read the thread session entry instead of the base route entry before this is the best fix.
