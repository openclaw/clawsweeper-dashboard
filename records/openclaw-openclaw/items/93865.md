## Summary

The branch is still not merge-ready: it moves the backfill to the right Mattermost plugin boundary and fixes the earlier admission/trim issues, but the gate is monitor-lifetime rather than session-aware, so `/new` or session clear can still skip recovery, and warm same-process first sightings can later refetch as if they were cold recovery.

## What This Changes

The PR adds a Mattermost client helper for `/posts/{postId}/thread`, uses it in the Mattermost monitor to seed empty thread history after admission gates, tracks one backfill per monitor lifecycle, and adds client helper tests.

## Best Possible Solution

Land one canonical Mattermost plugin fix that keys recovery backfill to session/reset state, marks warm accepted first sightings as serviced, filters the current post before trimming, and carries monitor-level regression coverage plus redacted real Mattermost proof.

## Reproduction Assessment

Yes. Source inspection shows current main keeps Mattermost thread history in a process-local map and builds context from that map, so restart or cleared session state leaves an existing thread with no local history to include.

## Solution Assessment

No. The Mattermost plugin is the right owner boundary, but this branch should use a reset-aware session gate, mark warm accepted sightings as serviced, and cover those sequences before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Key backfill to the agent session:** `extensions/mattermost/src/mattermost/monitor.ts:1558-1563`
  - body: `threadsBackfilledThisSession` is keyed only by `historyKey` for the monitor lifetime. After one recovery fetch, `/new` or another same-process session reset can rotate the agent session while this Set still suppresses the next accepted mention in the same thread, leaving the reported session-clear recovery path blind. Use a session/reset-aware marker or clear the marker when the session changes.
  - confidence: 0.9
- **[P1] Mark warm first sightings as serviced:** `extensions/mattermost/src/mattermost/monitor.ts:1564-1575`
  - body: The Set is updated only after a fetch inside the empty-window branch. If a thread is first accepted with local pending history, this block skips marking it; after the turn kernel clears that window, the next same-session follow-up sees empty history and performs the full server fetch/injection on a normal turn. Mark accepted non-empty first sightings as serviced and add that regression case.
  - confidence: 0.89
- **[P2] Filter the current post before trimming:** `extensions/mattermost/src/mattermost/monitor.ts:1580-1583`
  - body: The thread endpoint can include the current reply, but this slices to `historyLimit` before skipping `post.id`. When the current post is among the newest entries, the seeded window can be one entry short and omit an older prior message that should fit the configured limit. Filter out the current post first, then take the last N prior posts before resolving users.
  - confidence: 0.82
