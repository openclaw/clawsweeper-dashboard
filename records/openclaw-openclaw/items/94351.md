## Summary

Keep this PR open, but it is not merge-ready: the server-side rewrite is aimed at a real public-asset/proxy symptom, yet the configured-basePath path moves manifest/PWA behavior onto an unsupported namespace bootstrap URL and the supplied proof is only Gateway HTTP/mock test output.

## What This Changes

The PR rewrites Control UI public asset hrefs in gateway-served `index.html` to the internal namespace path and preserves those hrefs during client startup link sync.

## Best Possible Solution

Use one canonical asset URL strategy that prevents host-root proxy requests without moving configured-basePath manifest/PWA behavior onto an unsupported namespace bootstrap path.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: with `basePath: "/openclaw"`, the PR emits `/openclaw/__openclaw__/manifest.webmanifest`, while current manifest and bootstrap paths make `/openclaw/__openclaw__/control-ui-config.json` unsupported. I did not run a live browser/proxy deployment in this read-only review.

## Solution Assessment

No, not as written. Server-side rewriting is a reasonable layer for pre-JS requests, but configured-basePath deployments should keep manifest-relative URLs on a supported base path or add the matching namespace bootstrap/start-url contract.
