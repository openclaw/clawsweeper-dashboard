## Summary

Keep open: the revised branch is a useful move from document-relative links to a server-side rewrite, but it is not merge-ready because the browser startup code still re-syncs the same links back to root paths on root entry, and the PR proof is unit-test-only rather than real browser/proxy behavior.

## What This Changes

The PR adds a gateway-side rewrite of root-absolute Control UI public asset hrefs in served `index.html`, plus gateway HTTP regression tests for root, deep-route, and configured-base-path entrypoints.

## Best Possible Solution

Keep the server-side early rewrite only if the browser-side public-asset sync preserves the same namespace/base-path decision, then prove the root, deep-route, and base-path cases in a real browser or proxy-style setup.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the PR-introduced gap: load `/`, receive namespaced hrefs from the server, then `ui/src/main.ts` immediately sets the manifest link through `inferControlUiPublicAssetPath`, which returns `/manifest.webmanifest` for an empty base path.

## Solution Assessment

No. Server-side rewriting is a reasonable layer for the pre-JS window, but this PR is not the best complete fix until the client-side asset sync shares or preserves that decision.
