## Summary

Keep open: the prior client-side link-sync blocker is addressed, but the configured-basePath rewrite now points public asset hrefs at `/<basePath>/__openclaw__/...`, which can make manifest-relative PWA launch/bootstrap use an unsupported namespace path. Contributor proof is still mock/test output rather than a real browser or proxy network run.

## What This Changes

The PR rewrites gateway-served Control UI `index.html` public asset hrefs to the internal namespace, preserves those hrefs in client startup link sync, and adds Gateway HTTP regression tests.

## Best Possible Solution

Use one canonical asset URL strategy that prevents root-level proxy requests without moving configured-basePath manifest semantics onto an unsupported namespace bootstrap path.

## Reproduction Assessment

Yes for the PR defect by source inspection: with `basePath: "/openclaw"`, the PR emits `/openclaw/__openclaw__/manifest.webmanifest`, while the manifest's `./` start URL and client base-path inference lead to an unsupported bootstrap URL. I did not run a live browser/proxy deployment in this read-only review.

## Solution Assessment

No, not as written. Server-side rewriting is a reasonable layer for pre-JS requests, but configured-basePath deployments should keep manifest-relative URLs on a supported base path or add the matching namespace bootstrap/start-url contract.
