## Summary

Keep open. Current main has provider replay sanitizers, display/reasoning visibility controls, and a plugin hook that can mutate messages before persistence, but it still lacks a documented, schema-backed mode that lets thinking/reasoning run while excluding thinking blocks from durable session transcript/history surfaces.

## What This Changes

The issue requests a silent reasoning mode that decouples reasoning production from transcript persistence so thinking blocks are not stored or exposed to memory/history consumers.

## Best Possible Solution

Define a core transcript-persistence policy that is separate from model thinking effort and user/channel reasoning visibility. Durable transcript/history/memory-facing surfaces should be able to omit thinking blocks, while provider-required replay state is preserved through a tested internal path and documented across config, ACP, compaction, and session APIs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction path exists: persist an assistant message containing a `type: "thinking"` content block, then read it through `sessions.get` or ACP load-session replay; current main returns or projects that stored thinking content rather than applying a silent-reasoning transcript policy.

## Solution Assessment

Unclear as a concrete implementation. The product direction is sound, but the proposed unconditional strip-before-transcript step is not the safest full solution because provider replay may need a separate internal state path for signed or encrypted reasoning.
