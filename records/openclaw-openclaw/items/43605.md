## Label Justifications

- `P1`: The report describes a real multi-machine agent workflow where session messages cannot reach agents on other Gateways and the workaround bypasses native session history and routing.
- `impact:message-loss`: The central problem is that `sessions_send` cannot deliver messages to sessions on another Gateway.

## Summary

Keep open: current main and the latest release still keep session tools and `sessions.send` on local Gateway/session surfaces, and the linked implementation PR is open rather than merged.

## What This Changes

Request cross-gateway session messaging so agents on separate Gateway instances can address remote sessions without SSH workarounds.

## Best Possible Solution

Keep this issue as the concrete tracker until maintainers decide the cross-gateway trust/routing contract and either repair/land the linked PR or split the work into smaller protocol, security, and session-tool issues.

## Reproduction Assessment

Yes for source-level confirmation: current main lacks peer Gateway fields in `sessions_send`, `sessions_spawn`, and the `sessions.send` protocol schema, and the CLI validates agent IDs against local config. I did not run a live multi-machine reproduction.

## Solution Assessment

Unclear: the linked `gateway.peers` approach is a plausible implementation candidate, but the best fix depends on maintainer decisions about peer identity, URL allowlisting, token scope, session-state ownership, and whether this belongs in a narrower core protocol.
