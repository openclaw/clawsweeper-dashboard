## Summary

Keep open: current main still keeps `sessions_send` and `sessions_spawn` gateway-local, and the remaining work is already paired with an open implementation PR plus a broader federation/security design issue that needs maintainer review.

## What This Changes

The issue requests native cross-gateway session messaging so session tools can address agents on other gateway instances instead of relying on SSH workarounds.

## Best Possible Solution

Land or replace the linked implementation only after maintainers choose a secure federation or peer-routing contract that preserves gateway trust boundaries, session history semantics, and operator-controlled scopes.

## Reproduction Assessment

Yes, at source level: current `sessions_send` and `sessions_spawn` expose no remote gateway target parameters, and `sessions.resolve` reads the local gateway session store. I did not run a live multi-machine repro because the missing surface is clear from current main.

## Solution Assessment

Unclear as a final product shape: the linked `gateway.peers` PR is a plausible implementation candidate, but cross-gateway identity, scopes, and security posture need maintainer approval before it can be called the best fix.
