## Summary

Keep open: current main and latest release still lack native cross-gateway targeting for `sessions_send`/`sessions_spawn`, and the open linked implementation PR explicitly closes this issue but is unmerged, conflicting, and still needs product/security review.

## What This Changes

Request native cross-gateway session messaging so agents on separate OpenClaw gateways can send or spawn work without SSH wrappers.

## Best Possible Solution

Keep this issue as the narrow cross-gateway session-tool tracker, review or replace the linked PR only after maintainers settle the security/product contract, and coordinate any broader protocol decisions through https://github.com/openclaw/openclaw/issues/87733.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` expose only local `sessions_send`/`sessions_spawn` parameters, and the gateway config has a single `gateway.remote` target rather than peer routing. I did not run a live multi-machine repro in this read-only review.

## Solution Assessment

Unclear as an implementation: the linked `gateway.peers` PR is plausible but not clearly the best or safe final shape without a maintainer security/product decision. The safer path is to define the trust, URL, identity, scope, and delivery contract first, then repair or replace the PR narrowly.
