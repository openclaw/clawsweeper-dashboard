## Summary

Keep open. Current main still has local-only `sessions_send`, `sessions_spawn`, Gateway session RPC, and CLI agent targeting; https://github.com/openclaw/openclaw/pull/43656 explicitly closes this issue but is still open/conflicting, and the broader federation design remains in https://github.com/openclaw/openclaw/issues/46724. The `xihe-jianmu-ipc` MCP relay mentioned in related reports is a useful external workaround, but it does not replace the requested native session-tool behavior.

## What This Changes

The issue requests native cross-gateway session messaging so agents on different Gateway instances can communicate without SSH scripts or external relays.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/43656 lands with an accepted trust model and real two-Gateway proof, or maintainers intentionally fold the narrower session-tool request into https://github.com/openclaw/openclaw/issues/46724.

## Reproduction Assessment

Yes for confirming the limitation by source inspection: current `sessions_send`, `sessions_spawn`, Gateway session RPC, and CLI agent targeting all resolve local config/session state before sending. It is a feature gap rather than a broken documented contract.

## Solution Assessment

Unclear. The `gateway.peers` direction in the linked PR is plausible, but the maintainable solution still needs reviewed cross-Gateway trust, auth, URL policy, routing semantics, and live proof.
