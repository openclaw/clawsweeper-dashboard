## Summary

Keep open. Current main and the latest release still lack native cross-gateway session-tool routing, and the same-author closing PR remains open, conflicting, and blocked on product/security/proof questions.

## What This Changes

The issue requests native cross-gateway messaging so `sessions_send` and related tools can reach agents on other Gateway instances instead of only local sessions.

## Best Possible Solution

Define an authenticated Gateway-to-Gateway contract covering peer identity, authorization, URL policy, idempotency, delivery/failure semantics, and full session lifecycle, then repair or replace the linked PR against that contract.

## Reproduction Assessment

Yes at source level: current `sessions_send` and `sessions_spawn` expose no remote Gateway target and resolve sessions/agents through the local Gateway. I did not run a live multi-machine repro in this read-only review.

## Solution Assessment

No, not yet. Direct peer parameters are a plausible prototype, but the best fix needs a reviewed Gateway identity, authorization, addressing, and delivery contract before exposing model-facing session-tool routing.
