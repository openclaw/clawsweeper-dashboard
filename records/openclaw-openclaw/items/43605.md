## Summary

Keep open. Current main still has local-only session-tool and Gateway session RPC surfaces, and https://github.com/openclaw/openclaw/pull/43656 explicitly closes this issue but remains open and conflicting; the remaining work needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue requests native cross-Gateway routing for `sessions_send` and related session tools so agents on different OpenClaw Gateway instances can message or spawn work across machines.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/43656 lands with an accepted trust model and real two-Gateway proof, or maintainers intentionally fold the narrower session-tool request into https://github.com/openclaw/openclaw/issues/46724.

## Reproduction Assessment

Yes for source-level confirmation: current `sessions_send`, `sessions_spawn`, Gateway session RPC, and CLI agent targeting all resolve local config/session state before sending. This is a feature gap rather than a broken documented single-Gateway contract.

## Solution Assessment

Unclear. The `gateway.peers` direction in the linked PR is plausible, but the maintainable solution still needs reviewed cross-Gateway trust, auth, URL policy, routing semantics, and live proof.
