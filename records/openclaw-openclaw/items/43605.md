## Summary

Keep open. Current main still has local-only session send/spawn and CLI agent targeting, and the open linked implementation PR explicitly closes this issue, so cleanup closure before a reviewed merge would be premature.

## What This Changes

Add native support for agents on one OpenClaw Gateway to message or spawn sessions on agents hosted by another Gateway.

## Best Possible Solution

Keep this tracker open until https://github.com/openclaw/openclaw/pull/43656 lands with an accepted trust model, or until maintainers redirect the work into the broader federation design at https://github.com/openclaw/openclaw/issues/46724.

## Reproduction Assessment

Yes for the current limitation, though this is a feature gap rather than a broken existing contract. Source inspection on current main shows local-only tool schemas, local Gateway dispatch, and CLI validation against local agent IDs before the Gateway request.

## Solution Assessment

Unclear. The gateway.peers approach in https://github.com/openclaw/openclaw/pull/43656 is a plausible implementation direction, but it is not the best accepted solution until maintainers settle the cross-Gateway trust, auth, routing, and proof requirements.
