## Summary

Keep open. Current main still has local-only session send/spawn paths, and #43656 already links this issue for closure, so cleanup closure before a reviewed merge would be premature.

## What This Changes

Request cross-gateway `sessions_send`/`sessions_spawn` routing so agents on separate Gateway instances can message each other without SSH workarounds.

## Best Possible Solution

Keep this tracker open until #43656 lands with an accepted trust model, or maintainers redirect the work to the broader #46724 federation design.

## Reproduction Assessment

Yes for the current limitation, though it is a feature gap rather than a broken existing contract. Source inspection shows no remote target parameters in the session tool schemas and local CLI agent validation before the Gateway call.

## Solution Assessment

Unclear. The `gateway.peers` direction in #43656 is plausible, but it needs maintainer acceptance of the trust, auth, and routing boundaries before it is the best core solution.
