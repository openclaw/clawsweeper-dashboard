## Summary

Keep open. Current main still has only local session send/spawn routing, and an open same-author PR already links this issue for closure, so cleanup closure would be premature.

## What This Changes

The issue requests native cross-gateway messaging so `sessions_send` and related session workflows can target agents on other Gateway instances.

## Best Possible Solution

Keep this tracker open until maintainers land a reviewed cross-Gateway session-routing design or redirect the narrow session-tool request to a canonical Gateway federation design issue.

## Reproduction Assessment

Yes for the current limitation, by source inspection rather than a live multi-machine run. The relevant schemas and dispatch paths have no remote Gateway target, peer registry, or relay selection on current main.

## Solution Assessment

Unclear. The linked `gateway.peers` PR is a plausible implementation direction, but maintainer review is needed for trust boundaries, auth, transcript ownership, and whether this should be narrow session routing or broader Gateway federation.
