## Summary

Keep open. Current main still has local-only session send/spawn behavior and local CLI agent validation, and PR #43656 is an open implementation candidate with closing syntax for this issue, so cleanup closure would be premature.

## What This Changes

Request native cross-gateway `sessions_send` and `sessions_spawn` routing so agents on separate Gateway instances can communicate without SSH scripts.

## Best Possible Solution

Keep the tracker open until a reviewed, additive cross-Gateway session-routing design lands, likely by reviewing or replacing PR #43656 while preserving local defaults and explicit auth boundaries.

## Reproduction Assessment

Yes for the current limitation, though this is a feature gap rather than a broken existing contract. Source inspection on current main shows no remote Gateway parameters in the session tool schemas and local CLI agent-id validation before the Gateway call.

## Solution Assessment

Unclear. The `gateway.peers` direction in PR #43656 is a plausible narrow implementation, but it still needs maintainer review against the broader auth, trust, relay, and transcript-boundary questions also reflected in #46724.
