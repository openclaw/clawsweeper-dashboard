## Summary

Keep this issue open: current `main` still lacks native cross-gateway `sessions_send` or `sessions_spawn` tool routing, and the same-author implementation PR is still open with product/security/proof blockers rather than merged.

## What This Changes

Request native cross-gateway session messaging so agents on different OpenClaw gateway instances can communicate without SSH wrappers or ad hoc relays.

## Best Possible Solution

Keep this as the concrete cross-gateway session-tool tracker while maintainers either revise and land the linked PR or split the accepted design into narrower gateway identity, routing, and session-delivery work.

## Reproduction Assessment

Yes from source inspection for the current limitation: current `sessions_send` and `sessions_spawn` expose no peer or remote gateway target and route through local session resolution. I did not run a live multi-machine reproduction.

## Solution Assessment

Unclear. A native cross-gateway primitive is plausible, but the best fix requires maintainer product/security decisions about peer auth, URL policy, session ownership, and whether the narrower linked PR or the broader federation FRD should set the contract.
