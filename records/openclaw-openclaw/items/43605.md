## Label Justifications

- `P1`: The missing primitive blocks a reported real distributed-agent workflow and forces fragile SSH or external-channel workarounds for affected users.
- `impact:session-state`: The request is about routing messages, runs, replies, and history across gateway-owned sessions without losing continuity.
- `impact:security`: Cross-gateway messaging would add authentication, authorization, credential, and network trust boundaries between machines.
- `impact:message-loss`: The requested capability needs delivery, timeout, retry, dedupe, and disconnect semantics to avoid lost or duplicated inter-gateway messages.

## Summary

Keep open. Current main and v2026.6.6 still keep session tools gateway-local with no peer registry or peer Gateway role, and the open same-author closing PR is conflicting and blocked on product, security, and proof questions.

## What This Changes

Request native cross-gateway messaging so session tools can address agents on other Gateway instances without SSH workarounds.

## Best Possible Solution

Define an authenticated Gateway-to-Gateway contract for peer identity, authorization, URL policy, idempotency, delivery/failure semantics, and full session lifecycle before repairing or replacing the linked PR.

## Reproduction Assessment

Yes, at source level: current sessions_send and sessions_spawn expose no remote Gateway target and resolve sessions through the local Gateway. I did not run a live multi-machine reproduction in this read-only review.

## Solution Assessment

No, not yet. Direct peer parameters are a plausible prototype, but the best fix needs a reviewed Gateway identity, authorization, addressing, and delivery contract before exposing model-facing session-tool routing.
