## Summary

This issue is still necessary: current `main` and the latest release still expose only local session routing for `sessions_send`, while the linked implementation PR remains open and conflicting rather than merged.

## What This Changes

Request native cross-gateway session messaging so agents on separate gateway instances can send messages without SSH or ad hoc external relays.

## Best Possible Solution

Keep this issue open as the canonical native cross-gateway session-messaging tracker, then resolve it by narrowing or replacing the linked PR after explicit security, session-state, and delivery semantics are agreed.

## Reproduction Assessment

Yes, from source inspection: current `sessions_send` and Gateway `sessions.send` accept only local session identifiers and have no peer gateway selector or peer registry. I did not run a live multi-gateway setup because the missing capability is clear from current source and release inspection.

## Solution Assessment

Unclear at implementation level: `gateway.peers` is a plausible candidate, but the best solution needs maintainer decisions on trust, routing, history ownership, and failure semantics before code shape is final.
