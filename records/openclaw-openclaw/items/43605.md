## Label Justifications

- `P1`: The issue blocks a real distributed-agent workflow and currently forces fragile SSH/channel workarounds for cross-machine session messaging.
- `impact:session-state`: The requested feature is about preserving and addressing agent/session state across gateway boundaries.
- `impact:security`: Any native cross-gateway route must define trust, credentials, and authorization across separate gateway control planes.
- `impact:message-loss`: The reported failure is that `sessions_send` cannot deliver messages to sessions owned by another gateway.

## Summary

Keep open: current main and latest release still lack native cross-gateway targeting for `sessions_send`/`sessions_spawn`, and the linked implementation PR is open, conflicting, missing real behavior proof, and has security/session-delivery concerns.

## What This Changes

Add native cross-gateway messaging so `sessions_send` and related session tools can address agents or sessions on another OpenClaw gateway instead of requiring SSH wrappers.

## Best Possible Solution

Keep #43605 open as the canonical native session-tool tracker, resolve the product/security design with #87733 where needed, then either repair or replace PR #43656 with proof-backed implementation.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` session tools expose no peer/gateway target fields, and CLI `--agent` validates against local config before dispatch; no live multi-machine repro was run.

## Solution Assessment

Unclear for implementation: the linked `gateway.peers` PR is plausible but not the settled best fix because current evidence shows security, URL validation, session-state, and proof gaps.
