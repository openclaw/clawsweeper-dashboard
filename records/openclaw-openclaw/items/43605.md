## Summary

Keep open. Current main still has local-only session send/spawn behavior, and the issue is already paired with an open implementation PR that uses closing syntax, so cleanup closure would be premature.

## What This Changes

The issue requests native cross-gateway session messaging so agents on different Gateway instances can communicate without SSH workarounds.

## Best Possible Solution

Keep this tracker open until maintainers land a reviewed cross-Gateway session-routing design or explicitly fold it into a broader Gateway federation decision.

## Reproduction Assessment

Yes for the current limitation, though it is a feature gap rather than a broken existing contract. Source inspection shows no remote target parameters in the session tool schemas, no peer Gateway registry, and local CLI agent validation before the Gateway call.

## Solution Assessment

Unclear. The open gateway.peers PR is a plausible implementation candidate, but the best solution needs maintainer agreement on cross-Gateway trust, auth, URL guardrails, and routing semantics before implementation details are final.
