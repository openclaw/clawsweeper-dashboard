## Summary

Keep open for changes: the PR targets the right sandbox path area, but the patch is not safe or complete enough to merge because it reuses user-path alias normalization for a configured root and does not carry the normalized root through the actual Read/guard path.

## What This Changes

The PR expands tilde-prefixed sandbox roots in `resolveSandboxPath` and adds direct regression tests for that helper.

## Best Possible Solution

Normalize sandbox/workspace roots with a root-only home expansion helper at the shared boundary, carry that normalized root through alias and bridge checks, and add coverage for the actual Read/stat path.

## Reproduction Assessment

Yes at source-helper level: current main can falsely compare an expanded absolute file path against a literal `~/` root resolved under cwd. No high-confidence live Read-tool reproduction is present because the linked report was withdrawn and the PR proof uses a copied probe.

## Solution Assessment

No. The requested direction is reasonable, but this implementation should not reuse `expandPath` for roots and must normalize the root used by alias guards, Read wrappers, and sandbox bridge mount resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use root-only tilde expansion:** `src/agents/sandbox-paths.ts:75`
  - body: This changes the sandbox root to go through `expandPath`, but that helper also strips a leading `@` and normalizes Unicode spaces before handling `~`. Those are user-path conveniences, not configured-root semantics; a valid root such as `@workspace` would be checked as `workspace`, moving the sandbox boundary. Please use a small helper that only expands bare `~` and `~/` for roots.
  - confidence: 0.92
- **[P2] Normalize the root used by the real guard path:** `src/agents/sandbox-paths.ts:75`
  - body: After this line, `resolveSandboxPath` can accept a `~/` root, but `assertSandboxPath` still sends the raw root to `assertNoPathAliasEscape`, and the Read wrapper pre-resolves roots with `path.resolve(trimmedRoot)`. With `~/.openclaw/workspace`, those paths still resolve the root as `<cwd>/~/.openclaw/workspace`, so the actual Read/guard flow can keep rejecting the same in-workspace file. Carry the same normalized root through those callers and cover that path.
  - confidence: 0.87
