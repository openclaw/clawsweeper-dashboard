- **[P1] Add the new gateway settings to the validated config contract:** `src/config/types.gateway.ts:443-450`
  - body: The PR adds `gateway.ipAllowlist`, `gateway.ipBlocklist`, `gateway.requestRateLimit`, and `gateway.tls.terminatedUpstream` as TypeScript fields, but current main validates `openclaw.json` with strict zod objects and the matching schema/help/docs entries are absent. Operators following the PR body would either have these keys rejected or lack documented configuration support, so the controls are not a reliable public config surface yet.
  - confidence: 0.89
- **[P1] Do not throttle every gateway by default without an upgrade path:** `src/gateway/server.impl.ts:531`
  - body: `createRequestRateLimiter(cfgAtStart.gateway?.requestRateLimit)` always creates the limiter, so users who omit the new config still get the 120 req/min/IP default. That can turn previously working non-loopback HTTP or WebSocket gateway traffic into 429s after upgrade without an opt-in, migration, or maintainer-approved default policy.
  - confidence: 0.86
- **[P2] Thread the audit logger through real auth callers:** `src/gateway/server-http.ts:831-838`
  - body: The server now accepts `authAuditLogger`, but the normal HTTP auth stages still pass only the existing auth rate limiter into handlers that perform gateway authentication. As a result, the new `auth_success` and `auth_failure` hooks in `authorizeGatewayConnect` will not cover normal HTTP auth outcomes, leaving the advertised audit trail incomplete.
  - confidence: 0.82
