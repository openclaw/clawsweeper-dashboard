## Summary

Keep open, but do not merge as-is. Current main does not already contain the new Gateway IP ACL, request limiter, auth audit log, or credential-strength surface, and the PR branch still has blocking issues: the advertised config keys are not accepted by the strict config schema, auth success/failure audit events are not wired through normal Gateway auth callers, and the external PR lacks real behavior proof.

## What This Changes

The PR adds Gateway public-network hardening, including IP allow/block lists, per-IP request limiting, auth audit JSONL logging, credential/TLS warnings, auto-HSTS, and stricter bind resolution.

## Best Possible Solution

Keep the PR open only for maintainer security review after a rebase, schema/help/docs alignment, complete auth-audit wiring, and redacted real Gateway runtime proof.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough to reproduce the review findings: the strict Zod config schema lacks the new keys, and normal HTTP/WS auth callers do not pass an audit logger into the new auth audit hooks.

## Solution Assessment

No. The security direction is useful, but this branch is not the best merge shape until the config contract is complete, auth audit events are actually emitted on normal auth paths, and maintainer review settles the overlap with related request-limiter work.
