## Review Findings

Full review comments:

- **[P1] Add the new gateway settings to the validated config contract:** `src/config/types.gateway.ts:443-450`
  - body: The PR advertises `gateway.ipAllowlist`, `gateway.ipBlocklist`, `gateway.requestRateLimit.*`, and `gateway.tls.terminatedUpstream`, but only the TypeScript types include them. The strict `OpenClawSchema` gateway object still omits those keys, and the schema help/docs labels are missing too, so `openclaw config set` or a real `openclaw.json` using the new controls will fail validation instead of enabling the hardening.
  - confidence: 0.94
- **[P1] Do not throttle every gateway by default without an upgrade path:** `src/gateway/server.impl.ts:531`
  - body: `createRequestRateLimiter(cfgAtStart.gateway?.requestRateLimit)` always returns a limiter, and the HTTP/WS paths enforce it whenever present. That makes the new 120 req/min/IP behavior default-on for existing deployments that never opted into request throttling, so busy gateways can start returning 429s after upgrade unless maintainers explicitly approve the default and the PR proves fresh-install and upgrade behavior.
  - confidence: 0.9
- **[P2] Thread the audit logger through normal auth callers:** `src/gateway/server-http.ts:743`
  - body: The PR adds `auth_success` and `auth_failure` events in `authorizeGatewayConnect`, but `authAuditLogger` is only used at the top-level IP block/rate-limit checks. Normal auth callers such as tools, canvas, Control UI, and WebSocket auth still pass only the auth rate limiter, so the advertised credential-outcome audit trail is incomplete.
  - confidence: 0.86
