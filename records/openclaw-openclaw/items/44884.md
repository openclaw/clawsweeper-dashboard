Full review comments:

- **[P1] Add new gateway keys to the strict schema:** `src/config/types.gateway.ts:443-450`
  - body: The PR adds public `gateway.ipAllowlist`, `gateway.ipBlocklist`, `gateway.requestRateLimit`, and `gateway.tls.terminatedUpstream` types, but the strict zod schema does not accept those keys. Operators following the PR body can end up with config that strict validation rejects, so the schema/help/docs/config tests need to move with the new surface.
  - confidence: 0.93
- **[P1] Keep request throttling opt-in until default policy is approved:** `src/gateway/server.impl.ts:531`
  - body: `createRequestRateLimiter` is constructed even when `gateway.requestRateLimit` is omitted, so every upgraded non-loopback gateway gets a new 120 req/min/IP behavior and can start returning 429s without an operator opt-in. Preserve current behavior by default or add maintainer-approved upgrade proof for the breaking default.
  - confidence: 0.89
- **[P1] Cap the per-IP limiter state before exposing it publicly:** `src/gateway/request-rate-limit.ts:57`
  - body: The new limiter keeps a `Map` of per-IP timestamp arrays and creates entries for allowed requests, but there is no hard cap on unique IPs. A public-network flood that rotates source IPs can grow this state until the next prune tick, undermining the DoS hardening this PR is meant to add.
  - confidence: 0.9
- **[P2] Preserve an upgrade-safe bind fallback path:** `src/gateway/net.ts:253-255`
  - body: Changing loopback, tailnet, and custom bind resolution from 0.0.0.0 fallback to thrown startup errors can make existing gateways fail to start after upgrade. This needs either an explicit maintainer-approved breaking change with upgrade proof or a compatibility-safe default plus a separate strict mode.
  - confidence: 0.86
- **[P2] Thread the audit logger through real auth callers:** `src/gateway/auth.ts:85-94`
  - body: Adding `auditLogger` to `authorizeGatewayConnect` does not make auth outcome logging happen unless the HTTP and WebSocket auth call sites pass it through. The branch still omits it from normal gateway auth stages, so the advertised `auth_success` and `auth_failure` audit trail misses core credential outcomes.
  - confidence: 0.82
