## Summary

Keep this PR open. Current main does not implement the broad hardening request, and the PR still has concrete merge blockers: the advertised config keys are not registered in the strict config schema, auth audit outcomes are not wired through normal auth call sites, and tailnet exposure handling is inconsistent.

## What This Changes

The PR adds public-network gateway hardening for IP access control, request rate limiting, credential/TLS warnings, auth audit logging, HSTS defaults, and bind fallback behavior.

## Best Possible Solution

Repair the PR or a replacement branch by aligning the config contract and docs, wiring audit logging through every auth surface, making exposure semantics consistent for tailnet, adding a user-facing changelog entry, and then rerunning the gateway/config validation lanes.

## Reproduction Assessment

Yes for the PR findings: source inspection of the PR head shows the strict config schema lacks the new keys, auth call sites omit `auditLogger`, and startup exposure checks still classify Tailnet bind hosts as network-exposed. No runtime test was needed to establish these code-path defects.

## Solution Assessment

No. The hardening direction is useful, but this implementation is not yet the narrowest maintainable solution because the public config contract, audit coverage, and tailnet exposure policy are internally inconsistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Register the new gateway config keys in the schema:** `src/config/types.gateway.ts:443-450`
  - body: The PR advertises `gateway.ipAllowlist`, `gateway.ipBlocklist`, `gateway.requestRateLimit`, and `gateway.tls.terminatedUpstream`, but only adds them to TypeScript types. The gateway zod schema is strict and has no entries for these keys, so an operator who sets the documented options will fail config validation before the controls can run.
  - confidence: 0.92
- **[P2] Thread the audit logger into auth verification:** `src/gateway/auth.ts:85-94`
  - body: `authorizeGatewayConnect` now emits `auth_success` and `auth_failure` only when `auditLogger` is supplied, but the HTTP and WebSocket auth call sites still omit that parameter. Normal token/password/trusted-proxy handshakes therefore miss the advertised credential audit events and only log IP/rate-limit events.
  - confidence: 0.88
- **[P2] Keep tailnet out of startup exposure warnings:** `src/gateway/server-runtime-config.ts:177`
  - body: The security audit path special-cases `bind="tailnet"` as not public, but startup warnings still use `!isLoopbackHost(bindHost)`. When Tailnet resolves to a 100.x address, startup will still emit CRITICAL TLS/credential warnings for the tailnet-only case the PR says should be suppressed.
  - confidence: 0.86
