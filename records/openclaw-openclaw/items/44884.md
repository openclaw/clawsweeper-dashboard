Full review comments:

- **[P1] Add the new gateway settings to the runtime schema:** `src/config/types.gateway.ts:436-448`
  - body: These new operator-facing keys are added to the TypeScript type, but current main validates `gateway` with a strict Zod schema and the PR does not add matching schema/help/label entries. Config using `gateway.ipAllowlist`, `gateway.ipBlocklist`, `gateway.requestRateLimit`, or `gateway.tls.terminatedUpstream` can fail validation or be unavailable to config tooling, so the enforcement path is not reliably configurable.
  - confidence: 0.91
- **[P1] Do not enable request throttling as an implicit upgrade default:** `src/gateway/server.impl.ts:527-530`
  - body: This creates a request limiter for every gateway even when no operator config is present. Existing non-loopback deployments can start receiving 429s on ordinary HTTP/WebSocket traffic after upgrade, and the PR does not include an opt-in/default-policy decision, migration, or live upgrade proof for that behavior change.
  - confidence: 0.82
- **[P1] Keep bind fallback changes behind an upgrade-safe policy:** `src/gateway/net.ts:251-253`
  - body: Changing loopback, tailnet, and custom bind fallback paths from `0.0.0.0` fallback to startup throws can make existing deployments stop starting after upgrade. If this fail-closed behavior is desired, it needs an explicit maintainer policy, migration/doctor path, and upgrade proof; otherwise preserve current default behavior and add a strict opt-in.
  - confidence: 0.8
- **[P2] Thread the audit logger through real auth callers:** `src/gateway/auth.ts:87-96`
  - body: Adding `auditLogger` to `authorizeGatewayConnect` is not enough: the normal HTTP and WebSocket auth wrappers still call the auth helpers without that logger. As a result, the advertised `auth_success` and `auth_failure` events remain missing for the primary gateway credential paths.
  - confidence: 0.88
