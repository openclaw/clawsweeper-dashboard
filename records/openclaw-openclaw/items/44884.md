- **[P1] Add the new Gateway settings to the config contract:** `src/config/types.gateway.ts:443-450`
  - body: These new `gateway.*` fields are used by runtime code, but the PR does not add them to the strict zod config schema, schema labels/help, or user docs.   - confidence: 0.91
- **[P1] Gate default-on request throttling before upgrade:** `src/gateway/server.impl.ts:531`
  - body: `startGatewayServer` now always creates a request limiter with defaults and threads it into HTTP/WS handling. That means existing public Gateway clients can start receiving 429s after upgrade without opting in, without a migration, and without maintainer-approved default-change proof; preserve existing behavior by default or make the stricter policy explicit and upgrade-tested.
  - confidence: 0.88
- **[P1] Keep bind fallback changes upgrade-safe:** `src/gateway/net.ts:253-255`
  - body: Changing `loopback`, `tailnet`, and `custom` bind fallback paths from returning a fallback host to throwing can break shared callers beyond startup, including doctor/status/diagnostic paths that use the same resolver. This needs an explicit compatibility plan or strict-mode boundary instead of changing the helper behavior globally.
  - confidence: 0.82
- **[P2] Thread the audit logger into auth callers:** `src/gateway/auth.ts:85-86`
  - body: The new audit logger only records auth outcomes when `authorizeGatewayConnect` receives `auditLogger`, but normal HTTP and WS auth helper call chains still do not pass it through. As a result, the advertised `auth_success` and `auth_failure` JSONL audit trail will miss core credential outcomes for real Gateway traffic.
  - confidence: 0.86
