Full review comments:

- **[P1] Add the new Gateway keys to the config contract:** `src/config/types.gateway.ts:443-450`
  - body: The PR only adds the new hardening keys to the TypeScript `GatewayConfig` type, but current Gateway config is validated by a strict Zod schema and surfaced through schema help/labels/docs. As written, operators cannot safely persist or discover `gateway.ipAllowlist`, `gateway.ipBlocklist`, `gateway.requestRateLimit.*`, or `gateway.tls.terminatedUpstream` through the canonical config contract.
  - confidence: 0.93
- **[P1] Do not enable request throttling without upgrade policy:** `src/gateway/server.impl.ts:531`
  - body: `createRequestRateLimiter(cfgAtStart.gateway?.requestRateLimit)` always creates a limiter with the 120 req/min/IP default and the HTTP/WS paths enforce it even when users never opted in. That can make existing public Gateway clients start receiving 429 responses after upgrade, so this needs an explicit maintainer-approved default or an opt-in compatibility shape with upgrade proof.
  - confidence: 0.9
- **[P1] Keep bind fallback errors out of shared resolution:** `src/gateway/net.ts:253-255`
  - body: This shared resolver is used by startup, doctor, status, and display code, but the PR makes loopback/tailnet/custom resolution throw instead of returning a best-effort host. Current startup already rejects unsafe non-loopback fallback separately, so moving fail-closed behavior into the shared helper can break diagnostics and status paths that only need a display/probe address.
  - confidence: 0.84
- **[P2] Thread the audit logger into auth callers:** `src/gateway/server-http.ts:743`
  - body: The new `auth_success`/`auth_failure` events only fire when `auditLogger` reaches `authorizeGatewayConnect`, but the normal HTTP endpoint helper, WebSocket auth-context, canvas, and OpenAI/OpenResponses call chains still invoke auth without it. The advertised audit log therefore misses the core credential outcomes it claims to record.
  - confidence: 0.88
