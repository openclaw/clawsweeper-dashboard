- **[P1] Add the new Gateway keys to the config contract:** `src/config/types.gateway.ts:443-450`
  - body: The PR exposes new runtime keys in `GatewayConfig`, but current main's Gateway zod schema is strict and schema help/labels/docs do not include these keys. Operators may be unable to set or discover the advertised hardening options through the supported config path, so the schema/help/docs contract needs to land with the runtime code.
  - confidence: 0.91
- **[P1] Do not enable request throttling without upgrade policy:** `src/gateway/server.impl.ts:530-531`
  - body: `createRequestRateLimiter(cfgAtStart.gateway?.requestRateLimit)` always creates a limiter with defaults, and the HTTP/WS paths enforce it when present. That can make existing Gateway clients start receiving 429 responses after upgrade even if operators never opted into the new setting; make the default policy explicit and prove the upgrade behavior before merge.
  - confidence: 0.88
- **[P1] Keep bind fallback errors out of shared resolution:** `src/gateway/net.ts:253-255`
  - body: This changes `resolveGatewayBindHost` itself to throw for loopback/tailnet/custom fallback failures, but current main calls the helper from doctor, CLI health/status, and display paths as well as startup. Startup already rejects unsafe resolved fallbacks in `resolveGatewayRuntimeConfig`, so fail-closed startup behavior should stay at that boundary unless every non-startup caller is updated and tested for thrown errors.
  - confidence: 0.84
- **[P2] Thread the audit logger into auth callers:** `src/gateway/auth.ts:522-531`
  - body: The new `auth_success` and `auth_failure` events only fire when `authorizeGatewayConnect` receives `auditLogger`, but the PR's normal OpenAI/OpenResponses/Canvas/plugin auth stages still pass only the existing auth/rate-limit parameters. As written, the advertised audit JSONL log misses normal credential outcomes and records only IP block/rate-limit events for many Gateway paths.
  - confidence: 0.86
