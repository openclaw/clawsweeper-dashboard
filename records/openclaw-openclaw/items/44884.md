## Summary

Keep open. The PR is useful security work, but it is not merge-ready because the new gateway config knobs are not accepted by the strict config schema and the advertised auth outcome audit events are still not wired into normal HTTP/WS auth paths.

## What This Changes

The PR adds gateway public-network hardening with IP allow/block lists, per-IP request limiting, credential/TLS warnings, auth audit logging, HSTS defaults, and stricter bind fallback behavior.

## Best Possible Solution

Update the hardening patch so the config contract, docs/changelog, tests, and auth audit call chains all match the advertised behavior, then have gateway/security maintainers review the policy choices before merge.

## Reproduction Assessment

Yes. Source inspection is enough: the PR adds new config fields while the strict zod schema omits them, and the new audit logger parameter is not passed through normal auth call sites.

## Solution Assessment

No. The direction is plausible, but the current patch is not the narrowest maintainable solution until config validation/metadata/docs and auth outcome audit wiring are complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Add the new gateway keys to the config schema:** `src/config/types.gateway.ts:16`
  - body: The PR exposes `gateway.tls.terminatedUpstream`, `gateway.ipAllowlist`, `gateway.ipBlocklist`, and `gateway.requestRateLimit` in TypeScript types, but the runtime config schema is strict and still lacks these keys. Any user who sets the documented new controls will hit config validation before the gateway can read them, so the hardening controls are not actually configurable. Add the zod schema entries and the matching schema/help/label/generated docs coverage with the type additions.
  - confidence: 0.93
- **[P2] Wire audit logging into auth verification:** `src/gateway/auth.ts:82-91`
  - body: The new `auditLogger` parameter logs `auth_success` and `auth_failure` inside `authorizeGatewayConnect`, but this PR does not pass the logger through `authorizeGatewayHttpRequestOrReply`, canvas auth, or the WS auth context. As a result the server only logs IP blocks/rate limits and the advertised credential outcome audit trail is missing for normal auth. Thread the logger through the HTTP and WS auth call chains, or remove the auth outcome claim until it is implemented.
  - confidence: 0.88
