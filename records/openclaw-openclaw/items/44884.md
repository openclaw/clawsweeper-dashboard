  - body: The PR advertises `gateway.tls.terminatedUpstream`, `gateway.ipAllowlist`, `gateway.ipBlocklist`, and `gateway.requestRateLimit`, but PR-head `src/config/zod-schema.ts` keeps the gateway and tls objects strict without those keys.   - confidence: 0.92
- **[P1] Gate default-on request throttling before upgrade:** `src/gateway/server.impl.ts:531`
  - body: `createRequestRateLimiter` is created even when `gateway.requestRateLimit` is omitted, and the HTTP/WS paths enforce it before routing. That can make existing non-loopback clients start receiving 429s after upgrade without opt-in, so this needs an accepted default policy or a compatibility-safe gate.
  - confidence: 0.87
- **[P2] Thread auth audit logging into real auth callers:** `src/gateway/server-runtime-state.ts:63`
  - body: Adding `authAuditLogger` to runtime state does not pass it into inspected HTTP/WS auth callers, so the new `auth_success` and `auth_failure` logging in `authorizeGatewayConnect` does not fire for real gateway authentication paths. Pass the logger through the call chain or remove the promised event coverage.
  - confidence: 0.86
- **[P2] Make bind fallback changes upgrade-safe:** `src/gateway/net.ts:253-285`
  - body: Loopback, tailnet, and custom binds now throw instead of falling back when their target address is unavailable. That is a valid hardening direction, but existing deployments that relied on fallback can fail at startup, so the PR needs maintainer-approved upgrade handling or an explicit strict mode.
  - confidence: 0.8
