- **[P1] Add the new Gateway settings to the config contract:** `src/config/types.gateway.ts:443-450`
  - body: The PR only adds these keys to `GatewayConfig`, but the config parser is a strict Zod object and schema help/labels do not include them.   - confidence: 0.92
- **[P1] Gate default-on request throttling before upgrade:** `src/gateway/server.impl.ts:531`
  - body: This creates a request limiter for every Gateway start even when no operator configured one. Existing non-loopback HTTP or WebSocket clients can start receiving 429s after upgrade, so this needs an opt-in/default policy decision or explicit upgrade proof before merge.
  - confidence: 0.9
- **[P1] Make bind fallback changes upgrade-safe:** `src/gateway/net.ts:253-278`
  - body: These throws replace shipped fallback behavior for loopback, tailnet, and custom bind modes. That can stop an existing Gateway from starting where it previously recovered, so the PR needs a migration/deprecation path or explicit maintainer approval for the fail-closed change.
  - confidence: 0.88
- **[P2] Thread the audit logger into real auth callers:** `src/gateway/auth.ts:85-94`
  - body: The new `auditLogger` only records auth outcomes when callers pass it, but the HTTP/plugin/canvas and WebSocket auth paths still call their auth helpers without that logger. The advertised `auth_success` and `auth_failure` audit events will therefore miss normal Gateway auth traffic.
  - confidence: 0.84
