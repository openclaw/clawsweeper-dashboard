## Summary

Keep open: the PR is pointed at a real P1 systemd update handoff bug, but the current head still has transient-service contract problems, red CI, and no real Linux user-systemd proof after the fix.

## What This Changes

The PR passes the detected supervisor into the managed update handoff and launches systemd handoff helpers via a transient `systemd-run --user` service with mocked tests for systemd and non-systemd paths.

## Best Possible Solution

Keep the systemd transient-service direction, but pass the minimal handoff environment into the unit, require exec startup semantics, clear CI, and require redacted Linux user-systemd update proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main starts a detached helper inside a documented `KillMode=control-group` systemd unit, and the linked report includes concrete Linux/systemd symptoms with missing `handoff.log`. I did not run a live systemd update in this read-only review.

## Solution Assessment

No. The `systemd-run` service direction is the right fix shape, but this implementation still needs transient-unit environment propagation, exec startup confirmation, CI fixes, and real systemd proof before it is the safest maintainable solution.

Full review comments:

- **[P1] Pass handoff env into the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-345`
  - body: In service mode, `systemd-run` starts the helper in a clean service-manager environment; setting env on the `systemd-run` client does not give the helper's nested `openclaw update --yes --json` the control-plane sentinel metadata. Since the update command skips final sentinel writes when that metadata is absent, a successful package update can leave the UI/session stuck on the pending handoff. Pass the required env via `--setenv`/`Environment=` and assert it.
  - confidence: 0.91
- **[P2] Require exec startup before returning started:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-340`
  - body: The transient service still uses systemd's default `Type=simple`, which `systemd-run` can consider started after fork and before the helper executable is successfully exec'd. If exec fails, OpenClaw can return `handoff.status: started` and restart the gateway without a running helper or `handoff.log`; add `Type=exec` and test the expected argument.
  - confidence: 0.88

  - body: `check-test-types` is failing because this mocked process object sets `pid: undefined` even though the inferred mock return type requires a numeric `pid`. Adjust the mock shape or explicit type so the PR head typechecks.
