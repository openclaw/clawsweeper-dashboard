## Label Justifications

- `P1`: The PR targets a broken in-product managed update path for recommended Linux systemd-supervised installs.
- `merge-risk: 🚨 compatibility`: The patch changes update handoff behavior for existing systemd-supervised gateways and can still fail differently on upgrade.
- `merge-risk: 🚨 session-state`: The update handoff carries session continuation metadata through the restart sentinel, and losing the sentinel env can drop that continuation.
- `merge-risk: 🚨 availability`: A bad handoff can leave the gateway on the old version with a pending or misleading update sentinel after restart.

## Summary

Keep open: this is a focused fix for a real Linux systemd managed-update bug, but the current PR head is not merge-ready because the transient service still does not receive the restart-sentinel metadata and it still relies on systemd-run's default simple startup semantics.

## What This Changes

The PR passes the detected gateway supervisor into managed update handoff and changes the systemd branch to launch the helper with a transient `systemd-run --user` service plus mocked success and failure coverage.

## Best Possible Solution

Keep the systemd escape approach, but make the transient unit contract explicit with a narrow required-env allowlist, exec startup semantics, focused tests, and redacted Linux user-systemd update proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main launches a detached helper from inside the supervised gateway, and the linked Linux/systemd report provides concrete steps and logs showing the helper never writes `handoff.log`. I did not run a live user-systemd update in this read-only review.

## Solution Assessment

No: the proposed boundary is the right one, but the transient service needs explicit sentinel env propagation and exec startup semantics before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass the sentinel env into the transient unit:** `src/gateway/server-methods/update-managed-service-handoff.ts:345`
  - body: The systemd branch sets `env` only on the `systemd-run` client. Transient service units run in a clean detached environment, so without explicit `--setenv`/`Environment=` for `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META`, the CLI update cannot rewrite the pending sentinel or run the restart continuation after success.
  - confidence: 0.91
- **[P2] Use exec service startup semantics:** `src/gateway/server-methods/update-managed-service-handoff.ts:337-339`
  - body: `systemd-run` defaults to `Type=simple`, which reports startup success after fork and before `execve`. Add `--service-type=exec` or `--property=Type=exec` before returning `started`, otherwise a missing or rejected helper command can still schedule the gateway restart with no running helper.
  - confidence: 0.88
