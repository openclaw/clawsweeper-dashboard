## Summary

Keep open: the PR targets a real P1 systemd update bug, but the latest diff is not merge-ready because it can still report `started` when `systemd-run` fails and it weakens normal systemd child-process cleanup by switching the unit to `KillMode=mixed`.

## What This Changes

The PR passes supervisor detection into the managed-service update handoff, launches systemd handoff helpers through `systemd-run --user --scope`, changes systemd unit restart/kill settings and exit code handling, and adds focused mocked tests.

## Best Possible Solution

Keep the escaped systemd helper direction, but preserve normal control-group cleanup, use a startup contract that observes `systemd-run` failure before reporting `started`, keep non-systemd behavior unchanged, and prove the upgrade path in a real Linux user-systemd install.

## Reproduction Assessment

Yes for source-level reproduction: the linked Linux user-systemd report matches current main and `v2026.5.18`, where the helper is a detached child under a `KillMode=control-group` unit. I did not run a live systemd update in this read-only review.

## Solution Assessment

No: the escaped systemd helper is the right boundary, but this PR should not weaken normal cgroup cleanup and must observe `systemd-run` startup failures before declaring the handoff started. Real Linux systemd proof is still needed.

Full review comments:

- **[P1] Handle systemd-run failures before returning started:** `src/gateway/server-methods/update-managed-service-handoff.ts:351-365`
  - body: `spawn("systemd-run", ...)` can return a PID and then exit non-zero for a missing user bus or rejected transient unit, but this path ignores the exit code/stderr and still returns `started`. That preserves the same stale `managed-service-handoff-started` state for a failed handoff, so the systemd branch needs a startup contract that observes early `systemd-run` failure before scheduling the restart.
  - confidence: 0.9
- **[P1] Preserve control-group cleanup for systemd services:** `src/daemon/systemd-unit.ts:88`
  - body: `KillMode=mixed` sends the first termination signal only to the main gateway process and later SIGKILLs remaining cgroup children. The existing `control-group` setting intentionally keeps gateway workers in the normal service lifecycle, and the update helper should escape via its own unit instead of weakening cleanup for every normal stop/restart.
  - confidence: 0.84
