## Summary

Keep this PR open: the systemd cgroup escape is the right direction for the linked update bug, but the current head can still leave the gateway stopped after a helper/update failure and still trusts a predictable `/tmp` marker for startup delay.

## What This Changes

The branch passes supervisor state into update handoff, launches systemd helpers with `systemd-run --user --scope`, exits systemd update restarts with code 78, adds a per-UID `/tmp` marker guard to generated units, and extends tests.

## Best Possible Solution

Keep the `systemd-run --user --scope` escape, but add tested failure recovery that starts the recorded systemd user unit, move or owner-check the update marker, and prove the PR-head update lifecycle under user systemd.

## Reproduction Assessment

Yes at source/report level: current main combines a detached update helper with generated `KillMode=control-group`, and the linked report includes live systemd traces showing the helper never reaches its first log. I did not run a packaged OpenClaw update in this read-only review.

## Solution Assessment

No for the current PR head: `systemd-run --user --scope` is the narrow mechanism for the cgroup escape, but the implementation still needs failure recovery and a safer marker boundary before it is the best fix.

Full review comments:

- **[P1] Restart the systemd unit when the helper fails:** `src/cli/gateway-cli/run-loop.ts:227`
  - body: This exits with 78 so `RestartPreventExitStatus=78` intentionally leaves the gateway stopped while the scoped helper runs.   - confidence: 0.88
- **[P2] Keep the marker out of predictable tmp:** `src/daemon/systemd-unit.ts:77`
  - body: The new `ExecStartPre` trusts a predictable file under world-writable `/tmp`; any local user who can guess the service UID can create that filename and force generated systemd starts through the update delay. Use a private per-user runtime/state path or verify owner/mode before treating the marker as authoritative.
  - confidence: 0.78
