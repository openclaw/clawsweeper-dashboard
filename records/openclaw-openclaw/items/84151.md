## Summary

Keep open: the branch is aimed at the right systemd update-handoff root cause, but the current transient service launch still drops handoff/session metadata and can report success before the helper execs. The supplied proof is terminal output from Docker/process-group simulation, not a live user-systemd OpenClaw update.

## What This Changes

The PR routes systemd-supervised managed update handoffs through `systemd-run`, passes supervisor detection into the handoff helper, and adds mocked tests for systemd and non-systemd spawn behavior.

## Best Possible Solution

Keep the `systemd-run` direction, but make the transient unit contract explicit: pass required handoff/update environment into the unit, wait through helper exec, preserve non-systemd behavior, and prove an end-to-end user-systemd update.

## Reproduction Assessment

Yes at source level, with live evidence in the linked issue. Current main combines a detached helper spawn with a `KillMode=control-group` systemd unit, and the linked report shows the helper log never appears and the version stays unchanged.

## Solution Assessment

No, not in the current patch shape. The systemd-run approach is maintainable, but it needs explicit environment propagation, exec-level readiness, and real user-systemd proof before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Pass handoff env into the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:345`
  - body: `systemd-run` transient services run from the service manager environment, so the `env` option here only reliably configures the `systemd-run` client. The helper's `openclaw update --yes --json` can then miss `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META`, making `readControlPlaneUpdateSentinelMeta()` return null and dropping handoff/session/continuation metadata from the final restart sentinel. Add explicit unit environment, such as `--setenv`, and cover it in tests.
  - confidence: 0.89
- **[P2] Make systemd-run wait through exec:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-340`
  - body: The transient service is left at systemd's default `Type=simple`, where `systemd-run` may return success after fork but before the helper has actually exec'd. That lets the gateway report `handoff.status: "started"` and schedule restart even if the Node executable or script cannot start; use `Type=exec`/`--service-type=exec` or an equivalent readiness contract.
  - confidence: 0.86
