## Summary

Keep this PR open: the linked systemd update bug is real and the systemd-run direction is useful, but the current branch is not merge-ready because the transient service does not receive the required update sentinel environment and the supplied proof is still mock-only.

## What This Changes

The branch passes the detected supervisor into managed update handoff and, for systemd, starts the helper through `systemd-run --user` with mocked tests for systemd launch, failure, and non-systemd fallback.

## Best Possible Solution

Keep the systemd-run direction, but launch the helper with an explicit transient-service contract that propagates the required update/sentinel environment, waits through exec, preserves non-systemd behavior, and proves a real user-systemd update end to end.

## Reproduction Assessment

Yes at source level: current main combines a plain detached helper spawn with `KillMode=control-group`, and https://github.com/openclaw/openclaw/issues/84068 provides live Linux evidence with no `handoff.log` and an unchanged version. I did not run a live systemd update in this read-only review.

## Solution Assessment

No; the proposed boundary is plausible, but the current implementation loses the env contract needed for final sentinel writes and does not make `systemd-run` wait through exec. The safer solution is explicit transient-unit env plus `Type=exec` and live Linux proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass sentinel env into the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:345`
  - body: The `env` object here only configures the `systemd-run` client process. In service mode, `systemd-run` starts the command in a clean detached environment, while `openclaw update` reads `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META` from `process.env` before writing the final restart sentinel. On systemd this can complete or fail the package update without replacing the initial `managed-service-handoff-started` sentinel or preserving continuation metadata; pass the required env into the transient unit and cover it in tests.
  - confidence: 0.92
- **[P2] Make systemd-run wait through exec:** `src/gateway/server-methods/update-managed-service-handoff.ts:336-340`
  - body: Without `--service-type=exec` or `Type=exec`, `systemd-run` defaults to `Type=simple` and can return success after systemd forks, before `execve` of Node and the helper script succeeds. A bad executable path or inaccessible helper can still report `started`, trigger the gateway restart, and leave the update pending; use an exec/readiness contract that proves the helper command actually started.
  - confidence: 0.86
