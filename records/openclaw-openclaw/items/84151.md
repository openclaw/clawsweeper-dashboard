## Label Justifications

- `P1`: The PR targets a broken systemd-managed update workflow and still has availability-sensitive blockers before it can safely merge.
- `merge-risk: 🚨 compatibility`: The diff changes generated systemd unit behavior, including KillMode and restart-prevent exit codes, which can affect existing supervised installs.

## Summary

Keep this PR open: the reported systemd update bug is well supported and `systemd-run --user --scope` is a plausible direction, but the current patch still has merge-blocking availability and compatibility risks around failed handoff startup and changed systemd child cleanup semantics.

## What This Changes

The PR routes systemd-supervised managed update handoffs through `systemd-run --user --scope`, adds exit-code 80 restart handling, changes generated systemd unit restart/kill settings, and adds focused tests.

## Best Possible Solution

Keep the systemd scope handoff, but guard observable handoff startup failures, preserve existing child cleanup semantics unless separately proven safe, and require redacted live OpenClaw systemd update proof before merge.

## Reproduction Assessment

Yes at source/report level: current main uses a detached helper while generated systemd units use `KillMode=control-group`, and the linked issue includes live Linux/systemd failure traces. I did not run a live OpenClaw update in this read-only review.

## Solution Assessment

No. The `systemd-run --scope` direction is likely right, but the current patch should guard startup failures and avoid changing systemd child cleanup semantics unless that broader behavior is proved safe.

Full review comments:

- **[P1] Check systemd-run failure before reporting handoff started:** `src/gateway/server-methods/update-managed-service-handoff.ts:371-376`
  - body: This path suppresses `systemd-run` errors and only checks whether a pid was assigned. If `systemd-run` starts and then exits non-zero before the scope is usable, `update.run` still reports `managed-service-handoff-started` and the run loop can exit 80, leaving a systemd install stopped or stale instead of falling back with a real handoff failure.
  - confidence: 0.88
- **[P1] Keep control-group cleanup unless mixed shutdown is proven:** `src/daemon/systemd-unit.ts:88`
  - body: The scoped update helper already escapes the gateway cgroup, so changing the generated unit from `KillMode=control-group` to `mixed` is a separate compatibility change. systemd sends SIGTERM only to the main process in mixed mode and later kills remaining cgroup processes, which can remove the existing graceful cleanup path for gateway child workers on normal stops and restarts.
  - confidence: 0.84
