## Summary

Keep open: the underlying systemd update bug is real, but the latest PR head is not merge-safe because it forwards the full gateway environment through `systemd-run --setenv` and the supplied live proof still describes a different `--scope` implementation.

## What This Changes

The PR passes the detected supervisor into the managed update handoff and adds a systemd-specific `systemd-run` transient-unit launch path with focused handoff tests.

## Best Possible Solution

Use a systemd handoff that escapes the gateway cgroup without copying the full environment into argv or unit properties, then provide redacted live proof of the current-head OpenClaw update flow on a user-systemd install.

## Reproduction Assessment

Yes at source/report level: the related issue gives a concrete Linux user-systemd failure and current main still starts the helper as a detached child while the generated unit uses `KillMode=control-group`. I did not run the full live OpenClaw update in this read-only review.

## Solution Assessment

No. The cgroup-escape direction is right, but this PR's latest implementation trades it for full-environment `--setenv` exposure and the proof no longer exercises the current code path.
