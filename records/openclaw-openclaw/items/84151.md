## Label Justifications

- `P1`: The PR addresses a real broken systemd update workflow, but the current patch can still leave a managed gateway stopped or stuck during update handoff.
- `merge-risk: 🚨 compatibility`: The patch changes generated systemd unit defaults and exit-code handling, and existing installed units may not have the matching restart-prevention settings.

## Summary

Keep open: the `systemd-run --scope` direction matches the linked bug, but the patch still reports handoff success before early `systemd-run` failures can be observed and changes systemd child-cleanup semantics without proof.

## What This Changes

The PR routes systemd-supervised managed update handoff through `systemd-run --user --scope`, exits systemd update restarts with status 80, changes generated systemd unit kill/restart settings, and adds focused tests.

## Best Possible Solution

Keep the systemd scope handoff direction, but detect early `systemd-run` failure before exiting the gateway, preserve or explicitly prove the changed child-cleanup semantics, and require live OpenClaw user-systemd update proof before merge.

## Reproduction Assessment

Yes at source/report level: current main launches a detached helper while the generated systemd unit uses `KillMode=control-group`, and the linked issue includes live Linux/systemd failure traces. I did not run a live OpenClaw update in this read-only review.

## Solution Assessment

No: `systemd-run --scope` is the right root-cause direction, but this implementation needs early failure detection and a narrower or better-proved systemd unit change before it is the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Check systemd-run early failure before reporting start:** `src/gateway/server-methods/update-managed-service-handoff.ts:375-382`
  - body: In the systemd path, a successful `spawn()` PID only proves that `systemd-run` itself started. If the user bus is unavailable, the unit is rejected, or the command exits immediately, this code still returns `started`; `update.run` will write the handoff-started sentinel and the gateway can exit 80 with no helper running. Wait briefly for `error` or early non-zero `exit` before reporting success, and cover that path in tests.
  - confidence: 0.88
- **[P1] Keep child cleanup semantics unless the mixed mode is proved safe:** `src/daemon/systemd-unit.ts:84-88`
  - body: The helper already escapes into a separate scope, so switching the gateway unit from `KillMode=control-group` to `mixed` is not needed for the handoff. This changes normal stop/restart behavior from signalling all gateway children to signalling only the main process, then hard-killing remaining workers later, which can delay restarts or lose graceful ACP/runtime cleanup. Preserve the current kill mode or add proof that all existing child workers exit safely under the new mode.
  - confidence: 0.8

## Risks / Open Questions

- If `systemd-run` starts as a process and then exits non-zero before accepting the scope, this patch can still write `managed-service-handoff-started` and exit the gateway with status 80, leaving new systemd units stopped instead of updated.
- Changing `KillMode` from `control-group` to `mixed` removes the existing immediate SIGTERM fan-out to gateway child workers on normal stops/restarts; the scoped update helper does not require that broader change.
- Existing installed systemd units do not have `RestartPreventExitStatus=80` or the new `ExecStartPre` marker delay, so upgrade behavior from current released units still needs explicit proof.
- The posted proof validates the systemd cgroup mechanism, but not the full OpenClaw update flow with helper log creation, version change, healthy restart, and sentinel cleanup.
