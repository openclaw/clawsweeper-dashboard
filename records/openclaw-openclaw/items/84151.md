## Summary

Keep this PR open: it targets a real systemd update failure, but the current patch still has a P1 upgrade-safety blocker because the new exit-80 restart path depends on systemd unit metadata that existing installs do not have.

## What This Changes

The PR adds a systemd-specific managed update handoff that launches the helper with `systemd-run --user --scope`, passes supervisor context into update scheduling, and updates generated systemd unit restart settings and tests.

## Best Possible Solution

Keep the systemd scope handoff, but make the restart-stop behavior upgrade-safe for existing units and prove both old-unit and newly generated-unit paths in a real user-systemd OpenClaw update.

## Reproduction Assessment

Yes at source/report level: the linked issue gives a concrete systemd failure, and current main spawns a detached helper while the generated unit uses `KillMode=control-group`. I did not run the full live update path in this read-only review.

## Solution Assessment

No: `systemd-run --scope` is the right direction, but relying on a newly generated `RestartPreventExitStatus=78 80` unit is not upgrade-safe for existing systemd installs.
