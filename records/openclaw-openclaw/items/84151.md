## Summary

Keep open: the PR targets a real systemd managed-update bug, but the latest diff uses a transient systemd service with only two `--setenv` values and still lacks live user-systemd update proof. That can drop the managed service's profile/state/config/PATH/proxy environment during the handoff, so existing profiled or custom-state installs can still fail or restart the wrong gateway.

## What This Changes

The PR passes the detected supervisor into managed update handoff, routes systemd handoffs through `systemd-run` with exit/failure handling, and adds mocked gateway handoff tests.

## Best Possible Solution

Keep the systemd cgroup escape direction, but either use an environment-preserving scope with a proven start contract or explicitly forward the sanitized service environment into the transient service, then prove a real profiled Linux user-systemd update end to end.

## Reproduction Assessment

Yes at source/report level: current main uses a detached helper from a systemd unit with `KillMode=control-group`, and the linked report provides concrete failed-update artifacts with no `handoff.log`. I did not run a live systemd upgrade in this read-only review.

## Solution Assessment

No: `systemd-run` is the right class of fix, but this implementation has not preserved the managed-service environment or proven a live user-systemd update. The safer solution is to make that environment contract explicit and cover it with a profiled/custom-state test plus real systemd proof.
