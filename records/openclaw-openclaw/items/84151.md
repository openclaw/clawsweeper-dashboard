## Label Justifications

- `P1`: The PR targets a broken in-product managed update workflow for Linux systemd-supervised installs.
- `merge-risk: 🚨 compatibility`: The patch changes the update handoff mechanism for existing systemd-supervised gateways and can fail differently on upgrade.
- `merge-risk: 🚨 session-state`: The handoff carries session continuation metadata through the restart sentinel, and losing the sentinel env can drop that continuation.
- `merge-risk: 🚨 availability`: A bad handoff can leave the gateway on the old version with a pending or misleading update sentinel after restart.

## Summary

Keep open: the linked systemd update bug is source-reproducible, but this PR still misses required transient-service environment propagation, exec startup semantics, and real Linux systemd proof.

## What This Changes

This PR threads the detected supervisor into managed update handoff and adds a systemd-run transient-service branch plus mocked success and failure tests.

## Best Possible Solution

Use a transient user systemd service with a minimal required environment allowlist and exec startup semantics, keep non-systemd handoff behavior unchanged, and prove the full update/restart path on Linux.

## Reproduction Assessment

Yes for source-level reproduction: current main launches the helper as a detached child inside the systemd-supervised gateway, and the linked Linux report provides concrete logs where handoff.log is never created. I did not run a live user-systemd upgrade in this read-only review.

## Solution Assessment

No. The systemd escape is the right boundary, but the current diff omits explicit transient-unit environment propagation and exec startup semantics, so it is not yet the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass the sentinel env into the systemd unit:** `src/gateway/server-methods/update-managed-service-handoff.ts:345`
  - body: The systemd path passes `env` only to the `systemd-run` client. Transient service units run in a clean detached environment, so without explicit `--setenv` entries for the sentinel metadata, the CLI update cannot rewrite the pending sentinel or run the restart continuation after success.
  - confidence: 0.91
- **[P2] Require exec startup semantics:** `src/gateway/server-methods/update-managed-service-handoff.ts:336-340`
  - body: `systemd-run` defaults to `Type=simple`, which can report startup success after fork and before the helper command execs. Add `--service-type=exec` or `--property=Type=exec` before returning `started`, otherwise missing or rejected helper execution can still schedule a restart with no running helper.
  - confidence: 0.88
