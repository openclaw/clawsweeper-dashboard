## Summary

Keep open: this targets a real P1 updater bug, but the patch is not merge-ready because it can still report a systemd handoff as started before the escaped unit is accepted, it ignores early nonzero `systemd-run` failures, and it has no real Linux systemd proof.

## What This Changes

The PR passes the detected supervisor into the managed update handoff and uses `systemd-run --user --scope` for systemd-supervised handoffs, with focused unit tests.

## Best Possible Solution

Make the systemd handoff return `started` only after systemd has accepted an escaped unit, surface early failures as `managed-service-handoff-failed`, and prove the full update flow on a real Linux user-systemd install.

## Reproduction Assessment

Yes for source-level reproduction: the linked issue has concrete Linux user-systemd upgrade steps, and current main plainly starts a detached helper before scheduling the supervised restart. I did not run a live managed upgrade in this read-only review.

## Solution Assessment

No: escaping the systemd cgroup is the right direction, but this implementation returns before the escaped unit is proven to exist and misses early `systemd-run` exit failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Wait for systemd to accept the escaped handoff:** `src/gateway/server-methods/update-managed-service-handoff.ts:334-348`
  - body: `systemd-run --scope` execution is synchronous, but this function returns `started` immediately after spawning the `systemd-run` client and `update.run` then schedules the gateway restart for any started handoff. With `KillMode=control-group`, the restart can still kill that client before it has created or moved the helper into an escaped scope, leaving the sentinel stuck at `managed-service-handoff-started`; wait for systemd to accept/start the escaped unit before returning success.
  - confidence: 0.86
- **[P2] Report early systemd-run exit failures:** `src/gateway/server-methods/update-managed-service-handoff.ts:358-365`
  - body: A missing user bus or rejected transient unit can start the `systemd-run` process and then exit nonzero, so `child.pid` is still set and this code returns `started` while ignoring the failure. Capture an early exit/status from `systemd-run` and surface `managed-service-handoff-failed` instead of treating process spawn as handoff success.
  - confidence: 0.89
