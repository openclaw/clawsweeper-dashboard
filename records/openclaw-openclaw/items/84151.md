## Label Justifications

- `P1`: The PR targets an urgent broken managed update workflow for Linux/systemd users and still has merge-blocking correctness gaps in that path.
- `merge-risk: 🚨 compatibility`: Switching from a detached child to a systemd transient service changes the inherited execution environment for existing managed update handoffs.
- `merge-risk: 🚨 availability`: A bad handoff can leave the in-product update path reporting started while the helper or final restart status never completes.
- `merge-risk: 🚨 session-state`: Dropping the control-plane sentinel metadata can prevent the post-update continuation and leave update session state pending.

## Summary

Keep open: this PR targets a valid P1 systemd updater bug, and the second commit improves the earlier handoff-start check, but the transient service path still drops the control-plane sentinel environment needed to write the final update status/continuation, and real Linux user-systemd proof is still missing.

## What This Changes

The PR passes the detected supervisor into the managed update handoff and starts systemd-supervised update helpers through `systemd-run --user` with focused mocked tests.

## Best Possible Solution

Keep the systemd escape in the gateway handoff, but propagate the required update sentinel environment into the transient unit, use exec start semantics for the systemd handoff, preserve non-systemd behavior, and require redacted real user-systemd update proof before merge.

## Reproduction Assessment

Yes for source-level reproduction, but not live in this read-only review. Current main launches the handoff as a detached child under the supervised gateway, and the linked report gives concrete Linux user-systemd steps and logs showing the helper never reaches `handoff.log`.

## Solution Assessment

No. The direction is right, but the current PR path must preserve the control-plane sentinel environment in the transient systemd unit and use exec startup semantics before it is the narrow maintainable fix.

Full review comments:

- **[P1] Pass the sentinel environment into the systemd unit:** `src/gateway/server-methods/update-managed-service-handoff.ts:345`
  - body: In the transient service path, `env` only configures the `systemd-run` client process. The service itself runs in systemd's clean detached environment, so the helper's `openclaw update --yes --json` can start without `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META`; then `readControlPlaneUpdateSentinelMeta()` returns null and the CLI never writes the final restart sentinel or continuation. Pass the required environment into the unit explicitly and cover that contract.
  - confidence: 0.9
- **[P2] Use exec service type before reporting the handoff started:** `src/gateway/server-methods/update-managed-service-handoff.ts:336-340`
  - body: `systemd-run` defaults transient services to Type=simple, where systemd considers startup successful after fork and before execve, even if the command cannot be executed. This can still let `update.run` record `managed-service-handoff-started` before the helper command is actually running. Add `--property=Type=exec` or `--service-type=exec` and test an exec rejection path.
  - confidence: 0.82
