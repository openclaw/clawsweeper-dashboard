## Label Justifications

- `P1`: The linked bug breaks the in-product update workflow for systemd-supervised installs, and the PR still has a service-outage failure path.
- `merge-risk: 🚨 compatibility`: The PR changes generated systemd unit behavior, restart limits, and systemd update exit handling for existing managed installs.
- `merge-risk: 🚨 security-boundary`: The new generated unit trusts existence of a predictable path in world-writable `/tmp` to alter startup timing.
- `merge-risk: 🚨 availability`: A failed scoped helper or update command can leave the systemd-managed gateway stopped because exit 78 suppresses `Restart=always`.

## Summary

Keep open: the systemd scope escape is the right direction, but this PR still leaves a systemd-managed gateway stopped if the scoped helper or `openclaw update --yes` fails, and it adds a predictable `/tmp` marker gate. The supplied terminal proof is useful mechanism proof, but it still does not show the full PR-head OpenClaw update lifecycle.

## What This Changes

The PR passes supervisor state into update handoff, launches systemd-supervised handoff helpers with `systemd-run --user --scope`, exits systemd update restarts with code 78, adds a `/tmp` update marker guard to generated units, and extends tests.

## Best Possible Solution

Keep the `systemd-run --user --scope` escape, but add tested systemd failure recovery that restarts the recorded unit, move or remove the `/tmp` marker gate, and prove the PR-head OpenClaw update lifecycle on user systemd.

## Reproduction Assessment

Yes at source/report level: current main combines a detached update helper with generated `KillMode=control-group`, and the related report provides live Linux/systemd traces showing the helper never reaches its first log. I did not run a packaged OpenClaw update in this read-only review.

## Solution Assessment

No. The scope escape is maintainable, but the PR needs failure recovery and a safer marker design before it is the best fix for systemd update handoff.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restart the gateway when the scoped helper fails:** `src/cli/gateway-cli/run-loop.ts:226-228`
  - body: This exit-78 path tells systemd not to restart the gateway while the helper runs. But in the PR head, helper failures only update the sentinel and exit non-zero; if `openclaw update --yes` fails before restarting the unit, the gateway remains stopped until manual recovery. Pass the systemd unit into the helper and start/restart it on every helper or update failure path, with coverage for the failed-update case.
  - confidence: 0.92
- **[P2] Move the update marker out of world-writable tmp:** `src/daemon/systemd-unit.ts:77`
  - body: The generated unit now sleeps whenever `/tmp/openclaw-update-in-progress-$(id -u)` exists. Because `/tmp` is world-writable, another local user can pre-create the victim UID path and force every gateway start through this delay; `O_EXCL` only protects the helper write, not the unit's existence check. Use a private user runtime/state path or remove the guard.
  - confidence: 0.86
