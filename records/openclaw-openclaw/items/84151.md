## Summary

Keep open: the PR targets a real P1 updater bug and moves in the right direction, but the current patch still has a blocking transient-systemd-service contract bug and only supplies mocked macOS test proof rather than real Linux user-systemd update proof.

## What This Changes

This PR passes the detected supervisor into the managed-service update handoff and uses `systemd-run --user` for systemd-supervised update helpers, with focused handoff tests.

## Best Possible Solution

Keep the systemd transient-service direction, but pass the minimal handoff environment into the service unit, require exec startup semantics, cover both in tests, and require redacted live Linux user-systemd update proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main starts the handoff helper as a plain detached child inside a documented `KillMode=control-group` systemd unit, and the linked report provides concrete Linux user-systemd symptoms. I did not run a live systemd managed update in this read-only review.

## Solution Assessment

No. The proposed `systemd-run` direction is maintainable, but this implementation still needs service environment propagation, exec startup confirmation, and live systemd proof before it is the best safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Pass handoff env through the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-345`
  - body: In service mode, `systemd-run` starts the helper under the service manager rather than as the `systemd-run` client, but this call only puts the sentinel metadata env on the client process. The nested `openclaw update --yes --json` reads `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META` before writing the final restart sentinel, so a successful package update can leave the Control UI/session stuck on the pending handoff instead of delivering the continuation. Pass the required env with `--setenv`/`Environment=` and assert it in the systemd test.
  - confidence: 0.91
- **[P2] Require exec startup before returning started:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-345`
  - body: The transient service still uses systemd's default `Type=simple`, which `systemd-run` can consider started after fork and before the helper `execve` succeeds. If the helper binary cannot execute, OpenClaw can return `handoff.status: started` and schedule the gateway restart without a running helper or `handoff.log`; add `--property=Type=exec` or `--service-type=exec` and cover the expected args.
  - confidence: 0.87
