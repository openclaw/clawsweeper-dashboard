## Label Justifications

- `P1`: The PR targets a broken managed update workflow for systemd-supervised Linux gateways, which blocks real users from updating through the product UI.
- `merge-risk: 🚨 compatibility`: The diff changes generated systemd unit defaults and adds exit code 80 behavior that older installed units do not recognize.
- `merge-risk: 🚨 availability`: If the new systemd handoff fails after spawn or an old unit restarts at the wrong time, the gateway update can still leave the service unhealthy or on the old version.

## Summary

Keep this PR open: the systemd cgroup-escape direction addresses a real managed-update failure, but the current patch can still report a handoff as started without proving `systemd-run` accepted the scope, and the live user-systemd update path remains unproven.

## What This Changes

The PR routes systemd-supervised managed update handoff through `systemd-run --user --scope`, changes systemd unit restart/kill defaults, passes supervisor state into `update.run`, and adds focused tests.

## Best Possible Solution

Keep the systemd scope handoff direction, but make handoff startup fail loudly when `systemd-run` rejects the scope and prove the current-unit and newly-rendered-unit upgrade paths end to end.

## Reproduction Assessment

Yes at source/report level: current main launches a detached helper while the generated systemd unit uses `KillMode=control-group`, and the linked issue includes a live Linux/systemd failure trace. I did not run a live user-systemd update in this read-only review.

## Solution Assessment

No: the `systemd-run --scope` direction is plausible, but this patch should prove scope acceptance/failure before reporting success and should include live upgrade proof for current systemd users.
