## Label Justifications

- `P1`: The PR targets a broken managed update workflow for systemd-supervised installs and currently has a merge-blocking availability failure path.
- `merge-risk: 🚨 availability`: Merging this branch can leave the gateway stopped after failed scoped update handoffs because exit 78 prevents systemd from restarting the unit.
- `merge-risk: 🚨 security-boundary`: The generated systemd unit consults a predictable marker in world-writable `/tmp`, creating a concrete local-boundary startup-delay concern.

## Summary

Keep this PR open, but not merge-ready: the systemd scope escape addresses the reported root cause, while the latest branch still leaves the gateway stopped if the scoped update helper or inner update command fails.

## What This Changes

The PR routes systemd-managed update handoffs through `systemd-run --user --scope`, exits the gateway with status 78 for systemd update restarts, adds a temporary update marker guard to generated systemd units, and expands focused tests.

## Best Possible Solution

Keep the `systemd-run --user --scope` escape, add tested systemd failure recovery that restarts the recorded user unit on every helper/update failure path, move or ownership-check the marker, then prove success and failure recovery on real user systemd before merging.

## Reproduction Assessment

Yes at source/report level: current main combines a detached helper with generated systemd `KillMode=control-group`, and systemd documents that this kill mode terminates all remaining processes in the unit cgroup. I did not run a packaged OpenClaw update in this read-only review.

## Solution Assessment

No. The `systemd-run --user --scope` direction is the right core fix, but the current PR head still needs failure recovery and a trusted marker path before it is the safest maintainable solution.
