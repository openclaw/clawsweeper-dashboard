## Label Justifications

- `P1`: The PR targets a broken in-product managed update path for Linux systemd-supervised installs.
- `merge-risk: 🚨 compatibility`: The patch changes update handoff behavior for existing systemd-supervised gateways and can still fail differently on upgrade.
- `merge-risk: 🚨 session-state`: The update handoff carries session continuation metadata through the restart sentinel, and losing the sentinel env can drop that continuation.
- `merge-risk: 🚨 availability`: A bad handoff can leave the gateway on the old version with a pending or misleading update sentinel after restart.

## Summary

Keep open: the PR targets a real P1 managed-update bug, but the final systemd path still does not explicitly pass the control-plane sentinel environment into the transient service and still relies on systemd's default `Type=simple`, so it can report `started` while the helper cannot complete the documented sentinel/continuation contract. Real Linux user-systemd proof is also still missing.

## What This Changes

The PR passes the detected gateway supervisor into managed update handoff and changes systemd-supervised handoffs to launch the helper through `systemd-run` with mocked success and failure tests.

## Best Possible Solution

Keep the systemd escape approach, but make the transient unit contract explicit with a narrow sentinel env allowlist, exec startup semantics, focused tests, and redacted real systemd update proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main launches a detached helper from inside the supervised gateway, and the linked Linux/systemd report provides concrete steps and logs showing the helper never writes `handoff.log`. I did not run a live user-systemd update in this read-only review.

## Solution Assessment

No: the proposed boundary is the right one, but the transient service needs explicit sentinel env propagation and exec startup semantics before it is the narrow maintainable fix.

Full review comments:

- **[P1] Pass the sentinel env into the transient unit:** `src/gateway/server-methods/update-managed-service-handoff.ts:345`
  - body: The systemd path passes `env` only to the `systemd-run` client. Transient service units run in a clean detached environment, so without explicit `--setenv`/`Environment=` entries for `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META` and the handoff marker, the CLI update cannot rewrite the pending sentinel or run the restart continuation after success.
  - confidence: 0.91
- **[P2] Use exec service startup semantics:** `src/gateway/server-methods/update-managed-service-handoff.ts:339-340`
  - body: `systemd-run` defaults to `Type=simple`, which reports startup success after fork and before `execve`. Add `--service-type=exec` or `--property=Type=exec` before returning `started`, otherwise a missing or rejected helper command can still schedule the gateway restart with no running helper.
  - confidence: 0.88
