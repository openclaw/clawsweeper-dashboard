Keep open for maintainer review: the PR is a focused fix with sufficient real systemd proof and no blocking code-review finding, but the Linux/systemd self-update lifecycle change still carries compatibility and availability risk that maintainers should explicitly accept before merge.

## What This Changes

The PR routes systemd-managed update handoff helpers through `systemd-run --user --scope`, exits the supervised gateway with code 78, adds a marker guard to generated systemd units, and expands focused unit coverage.

## Best Possible Solution

Land the scoped systemd handoff once maintainers accept the upgrade-safety boundary, while preserving the non-systemd detached path and exit-78 compatibility for generated systemd units.

## Reproduction Assessment

Yes, at source/proof level: current main detached-spawns the helper while generated systemd units use `KillMode=control-group`, and the linked issue plus PR logs show the helper can be killed before it runs. I did not run a live systemd update in this read-only review.

## Solution Assessment

Yes at code-shape level: a transient systemd scope is the right boundary for a helper that must outlive the gateway service cgroup. The remaining question is maintainer acceptance of upgrade-safety risk, especially full version-swap and custom-unit behavior.
