## Summary

Keep open: the systemd scope handoff is a plausible fix direction, but this PR still needs full OpenClaw systemd update proof and has merge-blocking upgrade/security concerns around exit 80 and the global `/tmp` marker.

## What This Changes

The PR routes systemd-managed update handoffs through `systemd-run --user --scope`, passes supervisor context into update scheduling, and changes generated systemd restart settings and tests.

## Best Possible Solution

Keep the systemd scope handoff, but make the old-unit upgrade path explicit and safe, replace the global marker with a per-user safe marker or remove it, and prove the full OpenClaw update flow on user-systemd before merge.

Full review comments:

- **[P1] Make exit 80 safe for existing systemd units:** `src/cli/gateway-cli/run-loop.ts:226-227`
  - body: This exits 80 for every systemd update, but current and shipped user units only prevent restart on exit 78. Users upgrading from an existing unit can have `Restart=always` bring the old gateway back while the handoff is supposed to own the update, so gate exit 80 on a refreshed unit or add a tested old-unit path.
  - confidence: 0.87

  - body: The fixed `/tmp/openclaw-update-in-progress` path is global and `writeFileSync` follows pre-existing paths. Put the marker in a per-user runtime/state location and create it with exclusive/no-follow semantics, or remove the guard if it is only defensive.
  - confidence: 0.82
