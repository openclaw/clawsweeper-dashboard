## Summary

This focused gateway fix should stay open, but it is not merge-ready: the transient systemd service path still misses required service-environment propagation and exec-start confirmation, and the supplied proof is mocked macOS test output rather than live Linux user-systemd behavior.

## What This Changes

The PR passes the detected gateway supervisor into the managed-service update handoff and launches systemd-supervised helpers through `systemd-run --user` with mocked success and failure tests.

## Best Possible Solution

Keep the systemd transient-service direction, but pass only the minimal handoff/sentinel environment into the unit, use `Type=exec` startup semantics, preserve non-systemd behavior, and verify a real Linux user-systemd update end to end.

## Reproduction Assessment

Yes for source-level reproduction: current main starts the managed update helper as a detached child under the documented systemd `KillMode=control-group` unit, and the linked report gives concrete Linux/systemd symptoms. I did not run a live systemd update in this read-only review.

## Solution Assessment

No: the transient service approach is the right boundary, but this PR still needs explicit service environment propagation, `Type=exec`, and real Linux user-systemd proof before it is the safest fix.

- **[P1] Pass sentinel env into the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:333-345`
  - body: `systemd-run` transient services run in a clean detached execution environment, so the `env` passed to the `systemd-run` client is not enough for the helper's `openclaw update` process to read `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META`. Without passing the minimal handoff env via the unit, the package update can finish while the control UI/session remains on the pending handoff sentinel.
  - confidence: 0.9
- **[P2] Require exec startup before returning started:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-340`
  - body: The default transient service type is `simple`, where systemd can consider startup successful after fork and before `execve`. Add `--property=Type=exec` or `--service-type=exec` and assert it in the systemd test so `started` means the helper executable actually launched.
  - confidence: 0.86
