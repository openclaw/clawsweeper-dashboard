## Summary

Keep open for maintainer merge review: current main still has the systemd cgroup kill failure, the PR addresses the implicated path with real systemd proof, and I found no discrete blocking correctness or security finding. The remaining concern is maintainer acceptance of the Linux systemd update/recovery lifecycle risk.

## What This Changes

The PR routes systemd-supervised update handoff helpers through `systemd-run --user --scope`, exits with the existing systemd code 78 during update restarts, adds marker/recovery logic, and expands focused tests.

## Best Possible Solution

Land the scoped systemd handoff once maintainers accept the upgrade-safety boundary, while preserving the non-systemd detached path and exit-78 compatibility for shipped units.

## Reproduction Assessment

Yes at source/proof level: current main spawns a detached helper while generated systemd units use `KillMode=control-group`, and the linked issue plus PR logs show the helper can be killed before it runs. I did not run a live systemd update in this read-only review.

## Solution Assessment

Yes at code-shape level: a transient systemd scope is the right boundary for a helper that must outlive the gateway service cgroup. The remaining question is maintainer acceptance of upgrade-safety risk, especially full version swap and failure recovery.
