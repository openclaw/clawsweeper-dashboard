## Summary

Keep this PR open: the systemd handoff direction matches the reported failure, but the current patch still has merge-blocking correctness gaps and lacks real Linux/systemd proof.

## What This Changes

The PR passes the detected gateway supervisor into the managed-service update handoff and uses `systemd-run --user` for systemd-supervised update helpers, with mocked tests for success and failure paths.

## Best Possible Solution

Keep the systemd-run escape mechanism, but pass only the required handoff metadata env into the transient service, require exec startup confirmation, cover both in tests, and add redacted real Linux user-systemd update proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main uses a plain detached helper inside a documented `KillMode=control-group` systemd unit, and the linked report includes concrete Linux/systemd steps and missing `handoff.log` evidence. I did not run a live systemd update in this read-only review.

## Solution Assessment

No: systemd-run is the right boundary, but this patch does not yet preserve the sentinel env in the transient service or require exec startup confirmation. The safer implementation is a minimal explicit service env plus `Type=exec`, followed by real systemd proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass handoff env into the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:345`
  - body: The `env` option here only affects the `systemd-run` client. A transient service runs in a clean detached environment unless variables are passed with `--setenv`/`Environment=`, so the helper's `openclaw update` will not see `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META`; after a successful update the final sentinel and continuation can be skipped, leaving the UI/session stuck on the pending handoff state.
  - confidence: 0.91
- **[P2] Require exec startup before returning started:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-340`
  - body: This uses the default simple service type, where systemd can mark startup successful after fork but before the helper command is exec'd. Add `--property=Type=exec` or `--service-type=exec` so `handoff.status: started` is only returned once the helper executable has actually started, and cover that contract in the systemd-run tests.
  - confidence: 0.86
