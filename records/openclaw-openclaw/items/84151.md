## Summary

No blocking correctness finding after reviewing the diff against current main; keep this PR open for maintainer review because it fixes a real Linux/systemd update failure but still carries upgrade-lifecycle risk that CI cannot settle alone.

## What This Changes

The PR routes systemd-managed update handoff helpers through `systemd-run --user --scope`, adds exit-78 restart handling, marker/recovery logic, and focused tests for the managed handoff path.

## Best Possible Solution

Land the scoped systemd handoff once maintainers accept the upgrade-safety boundary, while preserving the non-systemd detached path and exit-78 compatibility for shipped generated units.

## Reproduction Assessment

Yes, at source/proof level: current main detached-spawns the helper while generated systemd units use `KillMode=control-group`, and the linked issue plus PR logs show the helper can be killed before it runs. I did not run a live systemd update in this read-only review.

## Solution Assessment

Yes at code-shape level: a transient systemd scope is the right boundary for a helper that must outlive the gateway service cgroup. The remaining question is maintainer acceptance of upgrade-safety risk, especially full version swap and failure recovery.
