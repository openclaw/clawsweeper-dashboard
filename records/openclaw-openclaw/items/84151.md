## Best Possible Solution

Use a transient user systemd service that explicitly passes only the required handoff/sentinel environment, uses exec startup semantics, preserves non-systemd behavior, and is backed by focused tests plus redacted real systemd update proof.

## Reproduction Assessment

Yes for source-level reproduction: current main launches a detached child inside the supervised gateway cgroup, and the linked Linux/systemd report includes concrete steps and logs showing the helper never writes `handoff.log`. I did not run a live user-systemd update in this read-only review.

## Solution Assessment

No. Moving the helper into a transient systemd service is the right boundary, but this patch still needs explicit service environment propagation, exec startup semantics, and real systemd proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Pass handoff env into the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-345`
  - body: The systemd branch builds `env` with `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META`, but only passes it to the `systemd-run` launcher process. A transient service runs in a clean service environment unless variables are set on the unit, while the CLI update only writes the final restart sentinel when that env var is present. Add explicit minimal `--setenv` entries for the required handoff variables and cover them in the systemd test.
  - confidence: 0.9
- **[P2] Use exec startup semantics for systemd-run:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-340`
  - body: This waits for `systemd-run` to exit, but the default transient service type is `simple`, which systemd treats as started after fork and before the helper binary has exec'd. That can still return `started` when the helper command cannot actually exec. Add `--service-type=exec` or the equivalent property and assert it in the systemd test.
  - confidence: 0.87
