## Label Justifications

- `P1`: The PR targets a broken in-product update workflow for systemd-supervised installs and still has upgrade-safety risk before merge.
- `merge-risk: 🚨 compatibility`: The new exit 80 behavior depends on regenerated systemd units, while existing installs only prevent restart on exit 78.
- `merge-risk: 🚨 availability`: If systemd restarts the gateway during the handoff, the service can run while its package tree is being updated or leave the update stale.

## Summary

Keep open: the systemd scope direction is plausible and the PR includes useful live systemd mechanism proof, but it is not merge-ready because the new exit-80 path is not upgrade-safe for already-installed units and the proof still does not run the actual OpenClaw update flow.

## What This Changes

The PR routes managed update handoffs under systemd through `systemd-run --user --scope`, adjusts systemd restart handling around exit 80, adds an in-progress marker guard, and expands handoff/unit tests.

## Best Possible Solution

Land a systemd handoff that escapes the helper into its own scope while preserving safe behavior for already-installed units, with tests and redacted live proof for both fresh and upgraded systemd service files.

## Reproduction Assessment

Yes at source/report level: current main uses a detached helper while generated systemd units use `KillMode=control-group`, and the linked issue includes live failure traces. I did not run a live current-main OpenClaw update in this read-only review.

## Solution Assessment

No. `systemd-run --scope` is the right direction, but the unconditional exit 80 path is not safe until existing installed units are detected, repaired, or otherwise handled.

- **[P1] Make exit 80 safe for existing systemd units:** `src/infra/process-respawn.ts:117-121`
  - body: `respawnGatewayProcessForUpdate()` now makes every systemd-supervised update exit 80, but only newly rendered units include `RestartPreventExitStatus=78 80`. Gate this on a detected/repaired unit or keep a compatibility path until the service is migrated.
