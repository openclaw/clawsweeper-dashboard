## Label Justifications

- `P1`: The PR targets a broken managed update workflow for systemd-supervised installs and currently has merge-blocking upgrade and availability risks.
- `merge-risk: 🚨 compatibility`: The live head relies on exit 80, but existing installed systemd units only contain `RestartPreventExitStatus=78`.
- `merge-risk: 🚨 availability`: A wrong restart-prevent status or failed helper recovery can leave update handoff behavior racing or the gateway stopped until manual recovery.
- `merge-risk: 🚨 security-boundary`: The helper recovery path builds a shell command from the systemd unit value instead of passing the unit as an argv element.

## Summary

Keep open: the systemd scope direction is useful, but the live PR head is not merge-ready because it switched the update handoff to exit 80 even though shipped units only suppress restarts for 78, it adds unrelated proof scripts under `src/`, and the helper recovery path shells out with an unescaped unit name.

## What This Changes

This PR routes systemd-supervised managed update handoffs through `systemd-run --user --scope`, adjusts systemd restart handling and generated units, expands updater tests, and currently adds two unrelated proof scripts under `src/`.

## Best Possible Solution

Keep the `systemd-run --user --scope` escape, use the already-shipped exit 78 contract, remove the unrelated proof scripts, restart systemd with argv-based APIs or validated unit names, and refresh current-head Linux user-systemd proof for success and failure recovery.

## Reproduction Assessment

Yes at source/report level: the linked issue gives a concrete Linux user-systemd update failure, and current main combines a detached helper with a generated systemd unit using `KillMode=control-group`. I did not run a packaged OpenClaw update in this read-only review.

## Solution Assessment

No for the current PR head. The `systemd-run --user --scope` direction is the right fix shape, but this branch needs the shipped exit-78 compatibility path, branch cleanup, safer systemctl invocation, and current-head proof before it is the best solution.

Full review comments:

- **[P1] Use the shipped exit 78 update handoff contract:** `src/infra/process-respawn.ts:121`
  - body: Existing systemd units in current releases only have `RestartPreventExitStatus=78`, while this PR exits 80 and only adds 80 to newly generated units. On the first upgrade from an installed release, systemd will restart the old gateway during the update handoff instead of leaving the scoped helper in control, so keep the update handoff on exit 78.
  - confidence: 0.93
- **[P1] Remove the unrelated proof scripts from src:** `src/proof-typecheck.ts:10-13`
  - body: This PR adds top-level proof scripts for `listChannelAgentTools`, which are unrelated to the systemd update fix and are included by `tsconfig.core.json`. `src/proof-typecheck.ts` also passes sender fields to a function whose PR-head signature still only accepts `{ cfg?: OpenClawConfig }`, so it will fail core typechecking; drop these files from the branch.
  - confidence: 0.91
- **[P2] Pass the systemd unit as argv instead of shell text:** `src/gateway/server-methods/update-managed-service-handoff.ts:173`
  - body: `params.systemdUnit` comes from `OPENCLAW_SYSTEMD_UNIT`, but the helper concatenates it into `execSync("systemctl --user start " + params.systemdUnit)`. Use `spawnSync`/`execFileSync` with `['--user', 'start', unit]`, or validate/escape the unit name, so a custom unit value cannot be interpreted by `/bin/sh`.
  - confidence: 0.78
