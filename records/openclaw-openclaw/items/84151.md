## Label Justifications

- `P1`: The PR targets a broken in-product update flow for systemd-supervised gateway installs.
- `merge-risk: 🚨 compatibility`: The PR changes the managed Linux update handoff path and can fail existing systemd installs closed when `systemd-run` or the user bus is unavailable.
- `merge-risk: 🚨 security-boundary`: The diff changes how gateway environment values, including possible secrets, are transferred into a supervisor-managed transient unit.
- `merge-risk: 🚨 availability`: The diff changes restart/update process orchestration for the gateway and can determine whether managed updates complete or leave the service on the old version.

## Summary

Keep open: the PR addresses a real systemd update-handoff bug, but the latest head still forwards the full gateway environment as `systemd-run --setenv` command-line arguments and the posted proof does not exercise a patched OpenClaw update under a live user-systemd service.

## What This Changes

The PR passes the detected supervisor into managed update handoff and routes systemd-supervised gateway updates through `systemd-run` with additional tests.

## Best Possible Solution

Keep the systemd cgroup escape, but transfer the helper environment through a secret-safe contract such as a 0600 transient environment file or narrow allowlist, then prove a redacted live user-systemd OpenClaw update end to end.

## Reproduction Assessment

Yes at source/report level: current main starts a detached helper while the generated systemd unit uses `KillMode=control-group`, and the linked issue reports a live Linux/systemd failure. I did not run a live user-systemd update in this read-only review.

## Solution Assessment

No: the systemd cgroup escape direction is appropriate, but forwarding every environment value through `systemd-run` argv is not the best safe implementation. A secret-safe environment handoff plus live systemd proof is the maintainable path.
