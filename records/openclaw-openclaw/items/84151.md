## Summary

Keep this PR open: the cgroup-escape direction is plausible, but the latest patch still has a merge-blocking upgrade-safety problem around exit 80 and the real-behavior proof does not show the actual OpenClaw update flow succeeding.

## What This Changes

The PR routes systemd-supervised managed update handoffs through `systemd-run --user --scope`, changes systemd restart directives, passes the detected supervisor into the handoff path, and adds focused tests.

## Best Possible Solution

Land the scoped systemd handoff only with an upgrade-safe exit-80 path, service audit or migration coverage for stale units, updated operator-facing systemd guidance, and redacted live proof for both fresh and upgraded systemd installs.

## Reproduction Assessment

Yes at source/report level: current main spawns a detached helper while generated systemd units use `KillMode=control-group`, and the linked issue includes live traces where helper files were created but `handoff.log` never appeared. I did not run a live current-main OpenClaw update in this read-only review.

## Solution Assessment

No. `systemd-run --user --scope` is the right direction for cgroup escape, but this PR is not the best complete fix until it handles already-installed units that lack exit-80 prevention and proves the real OpenClaw update path end to end.
