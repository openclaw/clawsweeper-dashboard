## Label Justifications

- `P1`: The PR targets a broken systemd update workflow and currently has an introduced availability risk in that same gateway update path.
- `merge-risk: 🚨 compatibility`: The exit-80 behavior depends on regenerated systemd unit semantics and still needs old-unit and refreshed-unit upgrade proof.
- `merge-risk: 🚨 security-boundary`: The patch writes and reads a predictable marker in world-writable `/tmp` without exclusive/no-follow handling.
- `merge-risk: 🚨 availability`: A helper failure after the scope is accepted can leave systemd from restarting the gateway because the new unit prevents restart on exit 80.

## Summary

This PR is a plausible fix for the reported systemd cgroup handoff bug, but it is not merge-ready: the new exit-80 path can leave the gateway stopped if the helper fails after the scope is accepted, the new global /tmp marker is not a safe boundary, and the supplied proof still does not exercise the actual OpenClaw update flow.

## What This Changes

The PR routes systemd-supervised managed update handoffs through `systemd-run --user --scope`, passes supervisor context into update scheduling, exits systemd update restarts with status 80, and updates generated systemd unit settings and tests.

## Best Possible Solution

Keep the systemd scope escape, but add a recovery-safe exit-80 design, remove or harden the marker, and prove the actual OpenClaw update flow on user-systemd before merge.

## Reproduction Assessment

Yes at source/report level: current main combines a detached helper spawn with generated `KillMode=control-group`, and the linked report gives concrete systemd symptoms. I did not run the full live OpenClaw update path in this read-only review.

## Solution Assessment

No. `systemd-run --user --scope` is the right systemd primitive, but the current PR needs safer failure recovery and marker handling before this is the narrow maintainable fix.

Full review comments:

- **[P1] Restart the gateway if the handoff helper fails:** `src/cli/gateway-cli/run-loop.ts:227`
  - body: After `systemd-run` accepts the scope, this branch exits the gateway with status 80 and the new unit prevents systemd from restarting it. If the helper then fails before the update command reaches its own service restart path, the helper only marks the sentinel failed and exits, leaving the managed gateway stopped; keep supervisor restart behavior until recovery is guaranteed, or make the helper restart the service on failure.
  - confidence: 0.86
- **[P2] Move the update marker out of global /tmp:** `src/gateway/server-methods/update-managed-service-handoff.ts:180`
  - body: This writes a predictable path in world-writable `/tmp` without exclusive/no-follow handling, while the unit checks the same global path. A local user can pre-create that name to delay unrelated starts, or use a symlink to make the OpenClaw user truncate/remove a user-writable target; put the marker under a per-user state/private temp path with safe open flags, or remove the guard.
  - confidence: 0.9
