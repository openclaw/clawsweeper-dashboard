## Label Justifications

- `P1`: The underlying bug breaks the in-product update workflow for systemd-supervised users, and the PR changes core restart behavior for that path.
- `merge-risk: 🚨 compatibility`: The patch relies on new generated unit settings while existing installed units do not yet have `RestartPreventExitStatus=80` or the new kill-mode semantics.
- `merge-risk: 🚨 session-state`: A failed or premature handoff can leave update restart sentinels in pending/skipped states and restart the gateway without completing the update.
- `merge-risk: 🚨 availability`: The change touches systemd restart exits and kill behavior, where a wrong transition can leave the gateway stopped, immediately restarted on old code, or unable to complete updates.

## Summary

Keep open: this is a plausible fix for a real P1 systemd update bug, but the current head is not merge-ready because it returns before `systemd-run --scope` proves the helper escaped, assumes existing installed units already understand exit code 80, and only supplies mocked macOS proof.

## What This Changes

The PR changes managed gateway update handoff behavior for systemd installs, including generated unit defaults, handoff spawning, respawn exit handling, and related tests.

## Best Possible Solution

Keep the systemd escape direction, but launch through a contract that confirms the transient unit started, handles old installed unit semantics, updates doctor/docs as needed, and proves a real Linux user-systemd update end to end.

## Reproduction Assessment

Yes at source level: current main combines a plain detached helper spawn with generated systemd `KillMode=control-group`, and the linked issue gives live Linux evidence that the helper never reaches its first log write. I did not run a live systemd update in this read-only review.

## Solution Assessment

No; the direction is maintainable, but the current patch does not prove `systemd-run` acceptance before returning, does not preserve upgrades from old installed units, and leaves manual systemd docs on the broken unit shape.

## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P1] Wait for systemd-run to accept the handoff:** `src/gateway/server-methods/update-managed-service-handoff.ts:351-365`
  - body: This branch returns `started` immediately after spawning `systemd-run --scope`, but systemd-run's scope mode is a synchronous wrapper and this code never observes unit acceptance or a non-zero systemd-run exit. If the user bus rejects the scope or the gateway exits before the scope is accepted, the update sentinel can remain pending with no helper; use a launch contract that confirms startup and covers failure exits.
  - confidence: 0.9
- **[P1] Keep exit 80 compatible with installed units:** `src/cli/gateway-cli/run-loop.ts:226-228`
  - body: Existing released systemd units only have `RestartPreventExitStatus=78`, while this path now exits 80 unconditionally for any detected systemd supervisor. Users upgrading with an old unit will restart immediately on the previous gateway while the handoff is supposed to run, so detect/migrate the installed unit or preserve a compatible exit path until the unit is updated.
  - confidence: 0.88
- **[P2] Update the manual systemd unit examples:** `src/daemon/systemd-unit.ts:88`
  - body: The generated unit now depends on `KillMode=mixed`, exit 80, and an update guard, but the public Linux gateway examples still show `KillMode=control-group` and no exit-80 handling. Users following the documented custom-unit path would keep the same cgroup-kill failure this PR is trying to fix.
  - confidence: 0.78
