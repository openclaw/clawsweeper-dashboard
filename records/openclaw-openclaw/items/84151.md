## Summary

Keep this PR open for maintainer review: it targets a real systemd update-handoff bug with a focused implementation and no blocking code finding, but the Linux service lifecycle change still carries compatibility and availability risk that maintainers should explicitly accept before merge.

## What This Changes

The PR routes systemd-supervised managed update handoffs through `systemd-run --user --scope`, exits the gateway with code 78, adds marker/recovery handling, and expands focused daemon/gateway tests.

## Best Possible Solution

Land the scoped systemd handoff once maintainers accept the upgrade-safety boundary, preserving the non-systemd detached path and exit-78 compatibility for generated systemd units.

## Reproduction Assessment

Yes, at source/proof level: current main detached-spawns the helper while generated systemd units use `KillMode=control-group`, and the linked report plus PR body include real systemd logs showing the helper killed without a scope and surviving with one. I did not run a live systemd update in this read-only review.

## Solution Assessment

Yes at code-shape level: a transient systemd scope is the right boundary for a helper that must outlive the gateway service cgroup while preserving the non-systemd path. The remaining question is upgrade safety for custom or stale units, not a discrete patch defect I can point to in the diff.
