## Label Justifications

- `P1`: The PR targets a broken managed gateway update workflow for systemd-supervised Linux installs and still has merge-blocking runtime risks.
- `merge-risk: 🚨 compatibility`: The systemd path changes existing update failure behavior by failing closed when `systemd-run` or the user bus is unavailable.
- `merge-risk: 🚨 session-state`: Missing handoff metadata in the transient service can leave the restart sentinel pending and lose the requested continuation.
- `merge-risk: 🚨 availability`: Returning `handoff.status: started` without confirmed helper execution can restart the gateway while no update helper is actually running.

## Summary

Keep this PR open: the systemd-run direction matches the linked gateway update bug, but the latest patch still has a transient-service contract bug that can leave the control-plane update sentinel pending, and the external PR still lacks real Linux user-systemd proof.

## What This Changes

This PR passes the detected gateway supervisor into the managed-service update handoff and uses `systemd-run --user` for systemd-supervised handoff helpers, with mocked tests for success and failure paths.

## Best Possible Solution

Keep the systemd transient-service approach, but pass the minimal handoff environment into the unit, use `Type=exec` startup semantics, cover both in tests, and require redacted live Linux user-systemd update proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main starts a detached helper inside a documented `KillMode=control-group` systemd unit, and the linked report includes concrete Linux/systemd symptoms with missing `handoff.log`. I did not run a live systemd update in this read-only review.

## Solution Assessment

No. The `systemd-run` service approach is the right fix direction, but this implementation needs explicit transient-unit environment, `Type=exec` startup confirmation, and real systemd proof before it is the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Pass handoff env through the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-340`
  - body: In service mode, `systemd-run` starts the helper in a clean service-manager environment, but this call only sets env on the `systemd-run` client. The nested `openclaw update --yes --json` reads `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META` before it writes the final restart sentinel, so a successful package update can leave the Control UI/session stuck on the pending handoff instead of delivering the continuation. Pass the minimal required env with `--setenv` or `Environment=` and assert it.
  - confidence: 0.91
- **[P2] Require exec startup before returning started:** `src/gateway/server-methods/update-managed-service-handoff.ts:336-340`
  - body: The transient service still uses systemd's default `Type=simple`, which `systemd-run` can consider started after fork and before the helper `execve` succeeds. If the helper binary cannot execute, OpenClaw can return `handoff.status: started` and schedule the gateway restart without a running helper or `handoff.log`; add `--property=Type=exec` and cover the expected args.
  - confidence: 0.87
