## Label Justifications

- `P1`: The PR fixes an urgent broken managed update workflow for Linux/systemd installs, but still has merge-blocking correctness gaps.
- `merge-risk: 🚨 compatibility`: Merging changes the Linux managed-update startup path from detached spawn to systemd transient units and can fail closed when the user bus or unit contract is wrong.
- `merge-risk: 🚨 session-state`: The handoff carries restart sentinel and continuation metadata, and the current PR can lose the metadata before the CLI writes the final sentinel.
- `merge-risk: 🚨 availability`: A false handoff start or failed transient unit can leave users on the old version with a stale update/restart state.

## Summary

Keep open: this PR targets a real systemd-managed update bug, but the current patch is not merge-ready because the transient service contract still can lose the handoff sentinel metadata and can report success before the helper executable has actually exec'd.

## What This Changes

The PR passes the detected supervisor into managed-service update handoff and routes systemd handoffs through a transient `systemd-run` service with mocked success and failure tests.

## Best Possible Solution

Use systemd-run as the escape mechanism, but make the transient unit explicit: pass only the required handoff metadata env, use exec startup semantics, keep non-systemd behavior unchanged, and require live Linux/systemd update proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main launches the helper with detached spawn inside the supervised gateway cgroup, while the linked Linux/systemd report includes concrete steps and logs showing no `handoff.log`. I did not run a live systemd update in this read-only review.

## Solution Assessment

No, not as currently written. `systemd-run` is the right direction, but the patch must explicitly pass the required handoff env and use exec service startup semantics before reporting the handoff as started.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass handoff env into the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:345`
  - body: The systemd branch passes `env` only to the `systemd-run` client process. In transient service mode the helper runs under the user service manager's environment unless the unit is given explicit env, so the CLI update may not see `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META` and then cannot rewrite the final restart sentinel or deliver the continuation. Pass the minimal required env through `--setenv`/unit properties instead of relying on `spawn` options.
  - confidence: 0.9
- **[P2] Require exec startup before returning started:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-340`
  - body: The args omit `--property=Type=exec` or `--service-type=exec`, so systemd-run's default simple service can exit successfully after fork but before the helper binary/script is actually exec'd. That can still make `update.run` return `handoff.status: started` and schedule a restart even when the helper cannot start; add exec startup semantics and assert the flag in the systemd test.
  - confidence: 0.89
