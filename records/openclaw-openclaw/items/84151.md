## Summary

Keep open: this targets the right gateway update bug, but the patch still has a process-lifetime race and does not observe early `systemd-run` failures. It also has no after-fix Linux user-systemd proof, which is required before merge.

## What This Changes

The PR passes the detected supervisor into the managed update handoff and changes the systemd path to spawn the helper through `systemd-run --user --scope`, with focused unit coverage.

## Best Possible Solution

Make the systemd handoff return `started` only after the user manager has accepted an escaped unit, report early failures as `managed-service-handoff-failed`, and prove the full update flow on a real Linux user-systemd install.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source-level reproduction: the linked issue has concrete Linux user-systemd upgrade steps, and current main plainly starts a detached helper before scheduling the supervised restart; I did not run a live managed upgrade in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No: escaping the systemd cgroup is the right direction, but this implementation returns before the escaped unit is proven to exist and misses early `systemd-run` exit failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Wait for systemd to accept the escaped handoff:** `src/gateway/server-methods/update-managed-service-handoff.ts:334-348`
  - body: `systemd-run --scope` is synchronous and keeps `systemd-run` as the parent until the helper exits, but this function returns `started` immediately after spawning that client and `update.run` schedules a gateway restart for any started handoff. With `KillMode=control-group`, the restart can still kill the `systemd-run` client before it has created or moved the helper into an escaped scope, leaving the sentinel stuck at `managed-service-handoff-started`; wait for systemd to accept/start the escaped unit, or use an async transient service path that provides that handshake before returning.
  - confidence: 0.86
- **[P2] Report early systemd-run exit failures:** `src/gateway/server-methods/update-managed-service-handoff.ts:358-365`
  - body: A missing user bus or rejected transient unit starts the `systemd-run` process and then exits non-zero, so `child.pid` is still set and this code returns `started` while ignoring the failure. That leaves the gateway to restart with no helper running; capture an early exit/status from `systemd-run` and surface `managed-service-handoff-failed` instead of treating spawn success as handoff success.
  - confidence: 0.89
