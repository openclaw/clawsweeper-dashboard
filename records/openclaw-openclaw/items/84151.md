## Summary

Keep open: this targets a real systemd-supervised update bug, but the current PR head still has availability and local-boundary blockers. The latest discussion says helper failure recovery was added, but the live `b3ad15c` patch still exits the gateway with status 78 without restarting it on helper/update failure.

## What This Changes

The PR routes systemd-supervised managed update handoffs through `systemd-run --user --scope`, exits systemd update restarts with code 78, adds an update marker guard to generated systemd units, and extends focused tests.

## Best Possible Solution

Keep the `systemd-run --user --scope` escape, add tested failure recovery that starts the recorded systemd user unit on every helper failure path, move or validate the marker in a private path, and then prove success and failure recovery on user systemd.

## Reproduction Assessment

Yes at source/report level: current main combines a detached update helper with generated systemd `KillMode=control-group`, matching the linked issue's traces where the helper never writes its first log. I did not run a packaged OpenClaw update in this read-only review.

## Solution Assessment

No. `systemd-run --user --scope` is the right mechanism for the cgroup escape, but the exit-78 failure mode and `/tmp` marker boundary need repair before this is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Restart the gateway when the scoped helper fails:** `src/cli/gateway-cli/run-loop.ts:226-227`
  - body: This exits the systemd-supervised gateway with 78, and generated units intentionally do not restart on that status. The helper's failure paths only mark the update sentinel and exit non-zero, so a failed update command, signal, parent timeout, or helper exception after handoff acceptance leaves the gateway service stopped until someone starts it manually. Please pass the unit into the helper and start it on every failure path before returning/exiting.
  - confidence: 0.9
- **[P2] Keep the update marker out of predictable tmp:** `src/daemon/systemd-unit.ts:77`
  - body: The generated unit now sleeps whenever `/tmp/openclaw-update-in-progress-$(id -u)` exists, but `/tmp` is world-writable and this check does not verify owner, mode, or a token. A stale file, or a file pre-created by another local user, can delay every start. Store the guard in a user-private runtime/state path or validate it before trusting it.
  - confidence: 0.82
