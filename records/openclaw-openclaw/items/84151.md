- `P1`: The PR targets a broken in-product update workflow for recommended Linux systemd-supervised installs and still has merge-blocking runtime correctness gaps.

## Summary

Keep open: the PR targets a real P1 updater bug, but it is not merge-ready because the systemd path reports the handoff as started before proving that systemd accepted the escaped unit, ignores early `systemd-run` failures after spawn, and still lacks real Linux user-systemd proof.

## What This Changes

The PR passes the detected supervisor into the managed update handoff and uses `systemd-run --user --scope` for systemd-supervised update helpers with focused unit tests.

## Best Possible Solution

Make the systemd handoff use an observable transient-unit start handshake, or a transient service mode whose client success means the helper has begun outside the gateway cgroup, while preserving the non-systemd paths and adding real upgrade proof.

## Reproduction Assessment

Yes for source-level reproduction: the linked issue gives concrete Linux user-systemd upgrade steps and current main plainly starts a detached helper before scheduling the supervised restart. I did not run a live managed upgrade in this read-only review.

## Solution Assessment

No. Escaping the systemd cgroup is the right direction, but this implementation treats spawning the `systemd-run` client as success and misses early failures, so it is not yet the best maintainable fix.
