## Label Justifications

- `P1`: The PR targets a broken managed update workflow for recommended Linux/systemd installs and still has merge-blocking correctness gaps.
- `merge-risk: 🚨 compatibility`: Merging changes Linux managed-update startup from a detached child to a systemd transient unit that can fail closed or lose custom service identity if the unit contract is wrong.
- `merge-risk: 🚨 session-state`: The handoff carries restart sentinel and continuation metadata, and the current transient service can drop that metadata before the CLI writes the final sentinel.
- `merge-risk: 🚨 availability`: A false handoff start or failed transient unit can leave users on the old version with stale update/restart state.

## Summary

Keep this PR open, but it is not merge-ready: the systemd-run direction matches the reported Linux cgroup failure, yet the current transient service still omits required service environment and exec startup confirmation, and no real Linux/systemd update proof has been provided.

## What This Changes

The PR passes the detected supervisor into managed-service update handoff and routes systemd handoff helpers through `systemd-run --user`, with mocked tests for systemd, non-systemd, and failure paths.

## Best Possible Solution

Use systemd-run as the cgroup escape, but make the transient unit explicit: pass only the required handoff/service-identity environment, require exec startup confirmation, preserve non-systemd behavior, and verify with a real Linux user-systemd update.

## Reproduction Assessment

Yes for source-level reproduction: current main launches the helper as a detached child inside the supervised gateway cgroup, and the linked Linux/systemd report includes concrete logs showing no `handoff.log`. I did not run a live systemd update in this read-only review.

## Solution Assessment

No, not as currently written. `systemd-run` is the right direction, but the PR needs explicit transient-service env and exec startup semantics before it is the maintainable fix.

- **[P1] Pass handoff env into the transient service:** `src/gateway/server-methods/update-managed-service-handoff.ts:345`
  - body: The systemd branch sets `env` only on the `systemd-run` client process. Transient services run under the service manager's environment, so the helper can miss `OPENCLAW_CONTROL_PLANE_UPDATE_SENTINEL_META` and custom service identity; pass the minimal required values with `--setenv` or unit properties before the command.
  - confidence: 0.91
- **[P2] Require exec startup before reporting started:** `src/gateway/server-methods/update-managed-service-handoff.ts:335-340`
  - body: The `systemd-run` args leave the transient service at the default simple type, which can return success after fork but before the helper executable has actually exec'd. Add `--property=Type=exec` or `--service-type=exec` and assert it in the systemd test so `handoff.status: started` means the helper command really started.
  - confidence: 0.9
