## Label Justifications

- `P1`: The underlying issue breaks systemd-supervised in-product updates, and the PR currently risks leaving those gateways stopped on helper failure.
- `merge-risk: 🚨 compatibility`: The PR changes supervised update exit behavior and generated systemd unit defaults that existing Linux service installs depend on.
- `merge-risk: 🚨 security-boundary`: The new marker guard uses a predictable path in world-writable `/tmp`, which is a local file-boundary concern.
- `merge-risk: 🚨 availability`: A helper or package-update failure after exit 78 can turn an update failure into gateway downtime.

## Summary

Keep open: the systemd scope direction addresses the linked bug, but the current PR head still changes systemd update restarts to exit 78 without a reliable helper-failure recovery path, and the new marker remains a predictable `/tmp` boundary.

## What This Changes

The PR routes managed update handoff helpers through `systemd-run --user --scope` under systemd, passes supervisor state into update scheduling, exits systemd update restarts with 78, adds a `/tmp` update marker guard to generated units, and extends tests.

## Best Possible Solution

Keep the systemd scope escape, but add tested recovery that starts the exact user unit on helper/update failure, move or remove the marker guard, and prove the full PR-head OpenClaw update lifecycle on user systemd.

## Reproduction Assessment

Yes at source/report level: current main combines detached helper spawn with generated `KillMode=control-group`, and the linked report shows the helper never writes its first log under systemd. I did not run a full live OpenClaw update in this read-only review.

## Solution Assessment

No, not yet. `systemd-run --user --scope` looks like the right direction, but the current exit-78 and marker design need recovery and file-boundary fixes before this is the maintainable solution.

## Review Findings

Full review comments:

- **[P1] Restart the gateway when the scoped helper fails:** `src/cli/gateway-cli/run-loop.ts:227`
  - body: This branch exits the systemd-managed gateway with 78, which generated units treat as `RestartPreventExitStatus`, so systemd will not bring the gateway back. The helper can still fail after that point; current update CLI recovery only restarts services it stopped itself, so when the service is already stopped it can leave no gateway running to surface the failure sentinel. Please add a tested failure path that starts or restarts the recorded user unit before using exit 78.
  - confidence: 0.88
- **[P2] Put the update marker in a private per-user directory:** `src/daemon/systemd-unit.ts:77`
  - body: The new generated `ExecStartPre` checks a predictable file under world-writable `/tmp`, and the helper writes the same path. Even with `wx`, another local user can pre-create `/tmp/openclaw-update-in-progress-<uid>` and force every start through the delay; stale files can do the same. Use a user-private runtime/state directory or remove this guard.
  - confidence: 0.83
