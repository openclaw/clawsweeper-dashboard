## Label Justifications

- `P1`: The linked bug breaks the documented in-product update workflow for systemd-supervised installs, and the PR touches the restart path that keeps those gateways available.
- `merge-risk: 🚨 compatibility`: The PR changes systemd unit defaults and introduces exit code 80 while existing installed units only prevent exit 78 until refreshed.
- `merge-risk: 🚨 session-state`: A false `managed-service-handoff-started` result can leave the restart/update sentinel pending and lose continuation metadata for the update handoff.
- `merge-risk: 🚨 availability`: The diff changes gateway service stop/restart behavior on Linux and can leave systemd installs unable to update or restart cleanly if the handoff fails.

## Summary

Keep open: this targets a real P1 systemd update failure, but the current PR head still reports the handoff as started when `systemd-run` itself exits non-zero, and the supplied proof is mocked macOS Vitest output rather than live Linux user-systemd proof.

## What This Changes

The PR passes the detected supervisor into the managed update handoff, launches systemd handoffs through `systemd-run --user --scope`, exits supervised update restarts with code 80, and changes generated systemd unit defaults/tests.

## Best Possible Solution

Keep the systemd escape direction, but use a contract that confirms transient-unit startup or surfaces failure, preserves old-unit upgrade behavior, and proves a real Linux user-systemd update end to end.

## Reproduction Assessment

Yes for source-level reproduction: current main launches the helper as a detached child under a generated systemd unit with `KillMode=control-group`, and the linked report gives concrete Linux/systemd symptoms. I did not run a live systemd update in this read-only review.

## Solution Assessment

No. Escaping the helper from the gateway cgroup is the right direction, but this implementation should confirm or fail the transient systemd launch and handle old-unit upgrade behavior before merge.

## Review Findings

Full review comments:

- **[P1] Handle systemd-run exit failures before returning started:** `src/gateway/server-methods/update-managed-service-handoff.ts:378-380`
  - body: This branch only checks whether spawning `systemd-run` produced a pid. If `systemd-run` itself starts and then exits non-zero, for example because the user bus is unavailable or the transient unit is rejected, the handler still returns `started`, the gateway exits, and the helper may never run. Capture the transient launch result and surface stderr/failure instead of treating pid creation as success.
  - confidence: 0.9
- **[P2] Keep exit 80 compatible with installed systemd units:** `src/cli/gateway-cli/run-loop.ts:226-228`
  - body: Released systemd units only have `RestartPreventExitStatus=78`, but this path exits with 80 before proving the installed unit has been refreshed to prevent 80. On upgrade from an existing unit, systemd can restart the gateway while the handoff is trying to update it; gate the new exit code on a compatible unit or add upgrade proof and handling.
  - confidence: 0.78
