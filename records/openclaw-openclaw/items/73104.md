## Summary

Keep open. The PR is scoped to the right Telegram bug and the dummy-timer nit was addressed, but the latest head still reuses the original update context during the delayed flush, so production middleware can mark the update complete before the debounced dispatch runs and then skip that dispatch.

## What This Changes

The PR adds a debounced Telegram `edited_message` handler, focused edited-message tests, a test timing option, and a test-harness idle cleanup.

## Best Possible Solution

Repair this branch so the accepted edited update schedules the debounce, but the final content dispatch does not re-check the completed `update:<id>` key; add a production-middleware regression test and an Unreleased Telegram fix entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to run an `edited_message` update through the real Telegram middleware, let the handler return after scheduling the timer, and observe the later flush reuse a context whose `update:<id>` key has already been completed and cached.

## Solution Assessment

No. The handler location and debounce shape are appropriate, but the delayed dispatch must separate update acceptance from final content processing instead of reusing the completed update context.
