## Summary

Keep open. The latest PR head fixes the earlier dummy-timer cleanup, but the production debounce path still reuses the completed Telegram update context, so the final edited message can be skipped by the update tracker; the user-facing fix also needs a changelog entry before merge.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-chat/message debounce, focused unit coverage, a test timing knob, and a test-harness idle callback cleanup.

## Best Possible Solution

Land the Telegram-owned edited-message debounce after separating middleware update acceptance from delayed final-content dispatch, with regression coverage through the real middleware/update-tracker path and an Unreleased Telegram fix entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an `edited_message` update accepted by middleware returns after scheduling the debounce, `finishUpdate` records `update:<id>`, and the timer later reuses that same context so `shouldSkipUpdate` can drop the dispatch.

## Solution Assessment

No. The handler location and debounce shape are appropriate, but the delayed dispatch must not re-check the already completed update context, and the PR needs the required changelog coverage.
