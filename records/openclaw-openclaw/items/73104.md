## Summary

Keep open. The latest PR head still has a production-path blocker: the delayed edited-message flush reuses the original Telegram update context after middleware has already completed and cached that update key, so the dispatch can be skipped before it reaches the inbound pipeline; the user-facing fix also still needs a changelog entry.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-chat/message debounce, focused edited-message tests, a test timing option, and a test-harness idle callback cleanup.

## Best Possible Solution

Repair this PR so middleware accepts/dedupes the edited update once while the delayed final-content dispatch bypasses the completed update key, then cover that path with a production-middleware regression test and add the Telegram changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to run an `edited_message` update through the real Telegram middleware, let the handler return after scheduling the debounce timer, and observe the later flush call `shouldSkipUpdate` on a completed `update:<id>` key.

## Solution Assessment

No. The handler location and debounce shape are appropriate, but the delayed flush should not reuse the completed update context for dedupe; separating update acceptance from delayed content dispatch is the safer narrow fix.
