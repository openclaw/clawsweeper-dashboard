## Summary

Keep open. Current main still lacks a Telegram `edited_message` handler, so the PR is not obsolete, and the latest PR head did address Greptile's dummy-timer cleanup. However, the proposed delayed flush still reuses the original Telegram update context after the middleware has finalized that update, so production `edited_message` dispatches can still be skipped by the existing update tracker; the PR also needs the required user-facing changelog entry.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-`chatId:messageId` debounce, a test-only debounce timing option, focused edited-message tests, and a test-harness idle callback cleanup.

## Best Possible Solution

Repair this PR so update-level dedupe is completed synchronously while the delayed final edited-message dispatch does not re-check the same completed update key. Add a middleware-path regression test and an Unreleased Telegram fix changelog entry, then land this PR or an equivalent narrow replacement to close #72873.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: run an `edited_message` update through the real Telegram middleware, let the handler return after scheduling the debounce timer, and the later timer flush reuses a context whose `update:<id>` key was already completed and cached.

## Solution Assessment

No. The handler location and debounce idea are the right shape, but reusing the original update context for delayed dispatch is not the best implementation because it collides with existing update tracking; the safer fix is to separate update acceptance/dedupe from delayed final content dispatch and test that production path.
