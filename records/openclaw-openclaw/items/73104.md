## Summary

Keep open. The PR targets a real Telegram gap, but the latest head still appears to drop debounced edited-message dispatches in the production middleware path because the timer reuses an update context that has already been completed and cached by the update tracker. It also still lacks the required user-facing changelog entry.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-chat/message debounce, a test-only debounce timing option, focused unit tests, and a Telegram test-harness idle callback cleanup.

## Best Possible Solution

Land a repaired Telegram-specific implementation that debounces `edited_message` updates through the shared inbound path without re-checking a completed update key, with middleware-path regression coverage and a changelog entry. Broader Discord and Signal edit semantics should stay in their existing related issues.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: run an `edited_message` update through the real Telegram middleware, let the handler return after scheduling the debounce timer, and the later timer flush reuses a context whose `update:<id>` key has already been completed and cached.

## Solution Assessment

No. The handler location and debounce shape are appropriate, but the delayed dispatch should be separated from update-level dedupe or use a dispatch context/key that cannot collide with the already completed Telegram update.
