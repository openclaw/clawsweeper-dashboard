## What This Changes

The PR adds a debounced Telegram `edited_message` handler that redispatches the final edited message into inbound processing, plus focused tests and a test-harness idle callback.

## Best Possible Solution

Keep current main's edit-cache implementation; any future edit-to-new-agent-turn behavior should be proposed separately with explicit Telegram UX semantics and live Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction of the PR-branch blocker: the timer fires after Telegram middleware has completed the same update key, so the delayed dispatch reuses a context that the update tracker can now skip.

## Solution Assessment

No. The branch's debounce-and-redispatch approach is superseded by current main's cache-update path, which fixes the streaming-context problem without starting a new agent turn and has focused regression coverage plus real proof in the merged PR.
