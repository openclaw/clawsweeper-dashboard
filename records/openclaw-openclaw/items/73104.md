## Summary

Keep open. The bug is valid and current main still lacks a Telegram `edited_message` handler, but the latest PR diff still routes the delayed debounce flush through the original update context after the production middleware has finalized that update, so real edited-message updates can still be deduped away. The author already addressed the Greptile dummy-timer cleanup in `d0f22b63`, leaving a narrow correctness repair plus the required changelog entry.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-chat/message debounce, exposes a test timing knob, updates the Telegram test harness idle callback, and adds focused edited-message unit tests.

## Best Possible Solution

Keep the debounce-based Telegram handler, but separate update-level dedupe from the delayed final-message dispatch so the middleware claims each Telegram update once while the timer can still process the final edited text. Add a production-path regression test that runs through the update-tracker middleware, then add the changelog entry and land this PR or an equivalent narrow replacement for #72873.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static: a real grammY update enters `bot.use`, the handler schedules a timer and returns, `finishUpdate` records `update:<id>` as recent, and the timer later calls the shared inbound path with that same completed update context.

## Solution Assessment

No. The handler shape and debounce are a good fit, but reusing the original update context for delayed dispatch is not the best implementation because it collides with the existing update tracker; the safer repair is to use a dedupe strategy that does not re-check the completed update during flush.
