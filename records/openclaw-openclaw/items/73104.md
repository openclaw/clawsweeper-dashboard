## Summary

Keep open. The PR addresses a real Telegram gap and the Greptile dummy-timer concern was fixed, but current head still reuses a completed update context during the delayed flush, so production middleware can drop the debounced edit; it also still needs the required changelog entry.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-`chatId:messageId` debounce, focused tests, a test timing option, and a test-harness idle cleanup.

## Best Possible Solution

Repair the PR so edited updates are accepted and deduped at the middleware update level while the delayed final content dispatch does not reuse the completed update context, then add middleware-path coverage and an Unreleased Telegram fix entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to run an `edited_message` update through the real Telegram middleware, let the handler return after scheduling the debounce timer, and observe the later flush reuse a context whose `update:<id>` key has already been completed and cached.

## Solution Assessment

No. The handler location and debounce shape are appropriate, but the delayed dispatch must avoid rechecking the completed update context; an edit-specific synthetic dedupe key or a split between update acceptance and final content dispatch is safer.
