## Summary

Keep open. The PR targets a real Telegram gap, but the latest head still reuses a completed Telegram update context during the delayed debounce flush, so edited messages will be dropped on the production middleware path; it also still lacks the required user-facing changelog entry.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-`chatId:messageId` debounce, focused edited-message tests, a test timing option, and a test-harness idle callback cleanup.

## Best Possible Solution

Repair this PR so edited updates are accepted/deduped at the middleware update level while the delayed final content dispatch does not re-check the same completed `update:<id>` key, then add a production-middleware regression test and an Unreleased Telegram fix entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to run an `edited_message` update through the real Telegram middleware, let the handler return after scheduling the debounce timer, and observe the later flush reuse a context whose `update:<id>` key was already completed and cached.

## Solution Assessment

No. The handler location and debounce shape are appropriate, but the delayed dispatch must not reuse the completed update context; a safer implementation separates update acceptance from delayed final content dispatch and tests that middleware path.
