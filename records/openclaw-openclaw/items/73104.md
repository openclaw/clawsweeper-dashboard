## Summary

Keep open. This PR addresses a real Telegram bug, but the current patch is not mergeable because the delayed edited-message flush reuses the completed Telegram update context and can be dropped by the production update tracker; it also still needs the required user-facing changelog entry.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-`chatId:messageId` debounce, focused edited-message tests, a test timing knob, and a test-harness idle callback cleanup.

## Best Possible Solution

Keep the Telegram-owned fix, but separate middleware update acceptance from delayed final-content dispatch, cover that through the real `createTelegramBotCore` middleware/update-tracker path, and add an Unreleased Telegram fix entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to run an `edited_message` update through the real Telegram middleware, let the handler return after scheduling the debounce timer, then observe the delayed flush re-check the same completed `update:<id>` key.

## Solution Assessment

No. The handler location and debounce shape are appropriate, but the delayed dispatch must not reuse a context whose update key has already been completed and cached by the middleware.
