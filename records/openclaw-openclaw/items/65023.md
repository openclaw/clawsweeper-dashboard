## Summary

Keep open, but this PR should not merge as-is. The latest head still has blocking import, runtime, config-plumbing, and security issues, lacks inspectable real behavior proof, and overlaps ongoing Google Vertex direction work in https://github.com/openclaw/openclaw/pull/60860 and https://github.com/openclaw/openclaw/issues/60736.

## What This Changes

The branch adds a core `google-genai` provider and `google-genai-sdk` auth mode backed by `@google/genai`, plus Vertex project/location config, embedded-runner routing, live tests, and an auth guide.

## Best Possible Solution

Consolidate Google Vertex/GenAI support through the Google plugin's guarded transport or an explicit provider SDK seam, preserving auth, tools, run options, cancellation, and network-policy parity before exposing new config.

## Reproduction Assessment

Yes for the blocking review findings: PR-head source and current-main source show import-scope, guarded-transport, option-forwarding, auth-surface, and config-plumbing gaps without live Google credentials. The claimed end-to-end Google behavior still lacks inspectable real-run proof.

## Solution Assessment

No. A parallel core direct-SDK provider is not the narrowest maintainable solution while the existing Google plugin owns guarded transport, auth evidence, and Vertex routing; the safer path is plugin-owned transport work or an explicit reviewed SDK seam.

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the current Pi package scope:** `src/agents/google-genai-stream.ts:2-3`
  - body: Current main depends on `@earendil-works/pi-agent-core` and `@earendil-works/pi-ai`, but this new file imports the old `@mariozechner` packages. A rebase or merge onto current main will fail to resolve these modules until the imports are updated.
  - confidence: 0.93
- **[P2] Keep Google calls on the guarded transport:** `src/agents/google-genai-stream.ts:40-54`
  - body: Constructing `GoogleGenAI` here sends model traffic through the SDK's internal network path instead of OpenClaw's guarded provider fetch. That loses the proxy, capture, timeout, cancellation, and network-policy controls used by the current Google plugin transport.
  - confidence: 0.88
- **[P2] Forward tools and run options to Google GenAI:** `src/agents/google-genai-stream.ts:96-100`
  - body: The `generateContentStream` request only sends model, contents, and optional system instruction. Tool declarations, tool choice, max tokens, temperature, headers, and cancellation are dropped, so tool-dependent and bounded runs will not match other provider transports.
  - confidence: 0.91
- **[P2] Allow the keyless auth mode in all runtime callers:** `src/agents/model-auth.ts:422-423`
  - body: This branch can return `google-genai-sdk` without an API key, but only embedded-runner checks were relaxed. Other runtime surfaces such as `/btw` still call `requireApiKey` for every no-key mode except `aws-sdk`, so this auth mode will fail outside the patched runner paths.
  - confidence: 0.88
- **[P2] Plumb Vertex config into runtime models:** `src/config/types.models.ts:79-82`
  - body: Adding `models.providers.*.vertexai` to config does not make it available to the stream: runtime model construction does not copy that provider-level object into the model passed to `createGoogleGenAiStreamFnForModel`. Users who configure only `openclaw.json` can still hit the missing-project path.
  - confidence: 0.87
