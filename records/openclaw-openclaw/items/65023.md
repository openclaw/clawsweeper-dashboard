## Summary

Keep open for maintainer/provider-owner review: the branch is a meaningful Google provider feature, but it is not mergeable as-is because the new stream path lacks agent-tool and generation-option parity, the new keyless auth mode is only allowed in some runtime callers, and the PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds a core `google-genai` provider path with unified Google AI Studio API-key and Vertex AI ADC auth, a new `google-genai-sdk` auth mode, config schema fields, docs, and live tests.

## Best Possible Solution

Consolidate the Google GenAI/Vertex work in the Google plugin-owned provider path, with full agent stream parity, one accepted config/auth surface, focused regression coverage, and redacted live proof before merge.

## Reproduction Assessment

Yes for the review blockers by source inspection: a `google-genai` run with tools or generation controls builds a request that omits them, and a `/btw` run with `auth: "google-genai-sdk"` reaches `requireApiKey` because only `aws-sdk` is exempt there. I did not run live Google credentials in this read-only review.

## Solution Assessment

No, not as-is. A unified Google SDK path may be useful, but this branch duplicates provider logic outside the current Google plugin boundary and misses runtime parity that the maintained Google transport already has.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass tools through the Google GenAI request:** `src/agents/google-genai-stream.ts:96-100`
  - body: The new stream builds the SDK request without `context.tools`, so `google-genai` models are never told about available agent functions and cannot produce tool calls. Agent runs that depend on filesystem/web/tool workflows will behave unlike the existing Google transport, which serializes tools and tool choice.
  - confidence: 0.91
- **[P2] Allow keyless auth mode in every runtime caller:** `src/agents/model-auth.ts:422-423`
  - body: `resolveApiKeyForProvider` can now return `google-genai-sdk` with no `apiKey`, but only the embedded runner checks were updated. Other callers such as `/btw` still exempt only `aws-sdk` before calling `requireApiKey`, so configured keyless Google GenAI runs fail before reaching the new stream.
  - confidence: 0.88
- **[P2] Forward generation controls to Google GenAI:** `src/agents/google-genai-stream.ts:96-100`
  - body: The SDK call only forwards `systemInstruction`, ignoring `SimpleStreamOptions` such as `maxTokens`, `temperature`, and request cancellation. Callers that set token limits or sampling controls will get provider defaults instead, which can change behavior and cost compared with other transports.
  - confidence: 0.84
