## Summary

Keep open. Current main still does not contain the proposed `google-genai` provider or `google-genai-sdk` auth mode, so the PR is not obsolete, but the current patch has blocking provider-boundary, runtime parity, dependency, and security-sensitive network-policy issues before it can merge.

## What This Changes

The PR adds a core `google-genai` provider backed by `@google/genai`, a `google-genai-sdk` auth mode, Vertex project/location config, embedded-runner routing, live tests, and a Google GenAI auth guide.

## Best Possible Solution

Revise the integration so unified Google GenAI and Vertex behavior stays within the bundled Google provider/plugin architecture, or narrow the work to remaining `google-vertex` ADC/proxy failures while preserving existing transport controls, dependency ownership, config contracts, docs, and regression coverage.

## Reproduction Assessment

Yes. The blocking findings are reproducible by static inspection against current main and the PR diff: the runtime model builders do not forward `vertexai`, `/btw` only allows keyless `aws-sdk`, and the root package does not declare `@google/genai`. Live Google credentials are not required to verify those defects.

## Solution Assessment

No. The current PR direction adds a core direct-SDK provider path that duplicates and weakens the existing Google plugin transport; the safer solution is a plugin-owned implementation or narrower repairs to the current `google-vertex` path with the same guarded transport semantics.
