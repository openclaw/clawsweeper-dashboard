## Summary

Keep open. The PR is a useful attempt at unified Google GenAI and Vertex support, but current head is not mergeable because it is stale against the current Pi package scope, bypasses guarded provider transport, drops tools/run options, incompletely wires keyless auth and Vertex config, lacks inspectable real-run proof, and overlaps active Google Vertex work.

## What This Changes

Adds a core `google-genai` provider and `google-genai-sdk` auth mode backed by `@google/genai`, plus Vertex project/location config, embedded-runner routing, docs, and live tests.

## Best Possible Solution

Consolidate Google GenAI and Vertex support through the Google plugin's guarded transport, or a deliberately reviewed SDK seam, while preserving auth, config, tools/options, cancellation, diagnostics, and redacted live proof before exposing new provider config.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head and current main shows the package-scope mismatch, direct SDK transport path, dropped tools/options, keyless-auth surface gap, and missing Vertex config plumbing without live Google credentials. The claimed successful live run itself is not reproducible from attached evidence because no proof artifact is provided.

## Solution Assessment

No. A parallel core direct-SDK provider is not the narrowest maintainable solution while current main keeps Google provider behavior in the Google plugin's guarded transport; the safer path is plugin-owned Vertex/GenAI transport work or an explicit reviewed SDK seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the current Pi package scope:** `src/agents/google-genai-stream.ts:2-3`
  - body: Current main depends on `@earendil-works/pi-agent-core` and `@earendil-works/pi-ai`, but this new file imports the old `@mariozechner` packages. A rebase or merge onto current main will fail to resolve these modules until the imports are updated.
  - confidence: 0.94
- **[P2] Keep Google model requests on guarded transport:** `src/agents/google-genai-stream.ts:40-54`
  - body: Constructing `GoogleGenAI` here sends model traffic through the SDK's internal network path instead of OpenClaw's guarded model fetch. That loses proxy/TLS policy, capture, timeout, cancellation, and network controls used by the current Google plugin transport.
  - confidence: 0.9
- **[P2] Forward tools and run options to Google GenAI:** `src/agents/google-genai-stream.ts:96-100`
  - body: The `generateContentStream` request only sends model, contents, and optional system instruction. Tool declarations, tool choice, max tokens, temperature, headers, cached content, and cancellation are dropped, so tool-dependent and bounded runs will not match other provider transports.
  - confidence: 0.91
- **[P2] Allow the keyless auth mode in all runtime callers:** `src/agents/model-auth.ts:422-423`
  - body: This branch can return `google-genai-sdk` without an API key, but only embedded-runner checks were relaxed. Other runtime surfaces such as `/btw` still call `requireApiKey` for every no-key mode except `aws-sdk`, so this auth mode fails outside the patched runner paths.
  - confidence: 0.88
- **[P2] Plumb Vertex config into runtime models:** `src/config/types.models.ts:79-82`
  - body: Adding `models.providers.*.vertexai` to config does not make it available to the stream: runtime model construction copies known provider fields but does not carry this new object into the model passed to `createGoogleGenAiStreamFnForModel`. Users who configure only `openclaw.json` can still hit the missing-project path.
  - confidence: 0.87
