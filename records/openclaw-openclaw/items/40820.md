## Summary

Keep open: the feature may be useful, but this branch is not mergeable as-is because it targets the pre-plugin Telegram source tree, writes a legacy/destructive streaming config path, and still lacks inspectable real Telegram behavior proof.

## What This Changes

This PR adds a Telegram DM `/settings` inline keyboard for viewing and changing `dmPolicy`, `groupPolicy`, and streaming settings, plus a read-only allowlist view.

## Best Possible Solution

Port a smaller current-main implementation into `extensions/telegram/src`, reuse the current Telegram command-auth/config-write helpers, write canonical nested streaming mode values, and require live Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows current main has moved Telegram runtime code to `extensions/telegram/src`, while the PR edits `src/telegram`, and the PR maps streaming writes to the legacy scalar key. I did not run a live Telegram reproduction because this is a conflicting external PR without real-behavior proof.

## Solution Assessment

No. A Telegram settings panel could be a good solution, but this branch needs to be ported to the current plugin architecture and updated for the current streaming config contract before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Port the handler to the Telegram plugin:** `src/telegram/bot-native-commands.ts:788-792`
  - body: Current main no longer has `src/telegram`; Telegram registers from `extensions/telegram/src`. Adding `/settings` to the old `src/telegram/bot-native-commands.ts` path leaves the feature outside the active runtime on current main, and the PR is already conflicting, so the command, callback handling, and tests need to be ported to the extension tree before merge.
  - confidence: 0.94
- **[P1] Write streaming.mode instead of replacing streaming:** `src/telegram/settings-panel.ts:30-34`
  - body: The stream setting maps to `channels.telegram.streaming` and `accounts.*.streaming`, but current Telegram config treats scalar `streaming` as legacy and uses `streaming.mode`. On a current config with `streaming.preview` or `streaming.progress`, tapping this button would replace the whole object with a string and discard nested streaming settings.
  - confidence: 0.9
