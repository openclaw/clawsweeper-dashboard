## Summary

Keep open. The feature direction is useful, but the branch is not merge-ready against current main: Telegram runtime wiring now lives in the bundled Telegram plugin, while this PR still adds and modifies the old core Telegram path, and its streaming write target conflicts with the current nested streaming config shape.

## What This Changes

The PR adds a Telegram DM `/settings` inline keyboard panel for access-control and streaming settings, with callback parsing, config mutation/auth handling, native command registration, provider filtering, and tests under `src/telegram`.

## Best Possible Solution

Port the panel into `extensions/telegram/src` using the Telegram plugin's existing command, callback, auth, runtime-config, and config-write seams; write streaming changes to `streaming.mode`; then add plugin-local tests plus docs and changelog coverage before merge.

## Reproduction Assessment

Yes. Static current-main checks reproduce the integration blockers: `src/telegram` is absent, active Telegram command/callback wiring is plugin-local, and current schema treats `channels.telegram.streaming` as a nested object rather than a scalar setting.

## Solution Assessment

No. A Telegram settings panel is a reasonable product direction, but this branch is not the best implementation as-is; the safer maintainable path is a current-main plugin port with canonical nested streaming writes and renewed auth review.
