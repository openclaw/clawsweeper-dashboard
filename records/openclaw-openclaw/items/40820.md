## Summary

The feature direction is plausible, but this PR is not merge-ready: it is conflicting, implements the Telegram handler under the old core `src/telegram` path instead of the active plugin runtime, can clobber nested streaming config, and still lacks real Telegram proof for a privileged config-write surface.

## What This Changes

This PR adds a Telegram `/settings` inline-keyboard panel, callback parser, config mutation flow, command registration changes, authorization helpers, and tests for DM policy, group policy, streaming, and allowlist display.

## Best Possible Solution

Rebase and implement the panel inside `extensions/telegram/src`, preserve nested streaming config and `progress` mode, and prove the privileged callback write path in a real Telegram session before merge.

Yes for the review blockers: source inspection shows current main runs Telegram from `extensions/telegram/src`, while the PR adds the handler under `src/telegram` and writes scalar `streaming` values. The proof gap is also clear from the PR discussion because no inspectable after-fix Telegram artifact is attached.

## Solution Assessment

No, not in this branch shape. A Telegram settings panel could be maintainable if it lives in the active plugin runtime, preserves canonical config subkeys, and has maintainer approval for callback-based config mutation.
