## Summary

Keep open. The feature is not implemented on main, but this patch is not merge-ready because it targets the retired Telegram core path, mismatches current streaming/group-policy contracts, and lacks required real Telegram proof for a privileged config-write UI.

## What This Changes

Adds a Telegram `/settings` native inline-keyboard control panel under legacy `src/telegram` files, plus shared native-command provider filtering and tests.

## Best Possible Solution

Port the panel into `extensions/telegram/src` using plugin-local native command, callback, and config-mutation contracts; preserve `streaming.mode` including `progress`, keep group access fail-closed, add focused extension tests, and attach redacted real Telegram proof.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head against current main shows the handler is in the retired `src/telegram` path and the streaming/default contracts mismatch. I did not run a live Telegram session because after-fix real behavior proof is missing and the blockers are visible in source.

## Solution Assessment

No. A Telegram settings panel may be useful, but this implementation needs to move to the current Telegram plugin runtime and align with current access-control and streaming contracts before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Port the settings handler into the Telegram plugin:** `src/auto-reply/commands-registry.data.ts:540-546`
  - body: This registry entry advertises `/settings`, but the dedicated command and callback handlers are added under `src/telegram`, while current main runs Telegram from `extensions/telegram/src`. After rebase this would not open the panel from the active runtime.
  - confidence: 0.95
- **[P1] Write the nested streaming mode value:** `src/telegram/settings-panel.ts:30-34`
  - body: `stream` maps to `streaming`, and `applySettingChange` writes the selected string at that path. Current config stores streaming as an object with `streaming.mode` plus sibling preview/progress/block settings, so this would replace the object and discard nested settings.
  - confidence: 0.92
- **[P2] Keep the group-policy default fail-closed:** `src/telegram/settings-buttons.ts:113-115`
  - body: The menu falls back missing `groupPolicy` to `open`, but current Telegram group access defaults missing provider config to `allowlist`. The settings panel must not present unset group access as more permissive than the runtime policy.
  - confidence: 0.88
- **[P2] Preserve the progress streaming mode:** `src/telegram/settings-buttons.ts:24-28`
  - body: Current Telegram supports `progress`, but the panel omits it from selectable stream values and normalizes existing `progress` to `partial`. Users with progress mode would see the wrong state and can lose that mode on save.
  - confidence: 0.88
