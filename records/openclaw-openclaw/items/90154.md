## Summary

This PR is a plausible narrow security hardening for the backend self-pairing bypass, and the branch includes live after-fix WebSocket proof plus focused regression coverage. I found no line-level correctness defect, but the change is compatibility-sensitive because shared token/password direct-local backend clients lose the self-pairing scope-preservation path.

## What This Changes

The PR narrows gateway backend self-pairing so only explicit no-auth local backend connects and device-token backend connects skip pairing, while shared token/password backend connects clear self-declared scopes and the docs/baseline tests are updated accordingly.

## Best Possible Solution

Merge the narrow hardening only after maintainers accept the shared token/password backend compatibility break and keep the dedicated local no-auth/device-token paths documented and tested.

## Reproduction Assessment

Yes for source reproduction: current main's helper preserves shared token/password backend self-pairing, and the caller skips clearing scopes when that exception is true. I did not run a live WebSocket exploit in this read-only review, but the PR body supplies live after-fix probe output for the same path.

## Solution Assessment

Yes, if maintainers accept the compatibility break: removing shared token/password from the backend self-pairing exception is the narrowest code fix for the reported device-less backend impersonation path. A transition path would be safer only if shipped device-less backend clients rely on the old behavior.
