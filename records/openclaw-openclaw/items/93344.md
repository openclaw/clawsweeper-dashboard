## Label Justifications

- `P1`: The bug deterministically breaks direct SSH sandbox upload/cleanup for real sandbox users and affects workspace/materialized-skill setup before an agent can run reliably.
- `impact:security`: The validated remote path is not the path used by appended upload/cleanup commands, weakening the intended sandbox path boundary.
- `impact:data-loss`: The same preflight is followed by cleanup commands using `find "$1" ... rm -rf`, so a rewritten relative `$1` can target the wrong directory.
- `impact:session-state`: The failure corrupts or mis-associates remote sandbox workspace/materialized-skill state for affected SSH sandbox sessions.

## Summary

Keep open: current main and `v2026.6.6` still mutate shell positional parameters inside the SSH/OpenShell remote-directory preflight, and a current-main source repro returns `target=workspace` / `root=.openclaw` instead of preserving the caller-supplied paths.

## What This Changes

The issue requests fixing SSH sandbox remote-directory preflight so validation does not overwrite caller shell positional parameters before appended upload and cleanup commands run.

## Best Possible Solution

Fix the shared preflight shape so component validation iterates without changing positional parameters, update both SSH and OpenShell sibling scripts, and add regression coverage that appended commands still see the original `$1` and `$2`.

## Reproduction Assessment

Yes. A current-main source repro using the exported SSH preflight under `/bin/sh` prints rewritten positional parameters (`target=workspace`, `root=.openclaw`) instead of the caller-provided target/root paths.

## Solution Assessment

Yes. The maintainable fix is narrow: keep the current validation behavior but stop using shell positional parameters as loop scratch space, and apply the same invariant to the OpenShell sibling copy.
