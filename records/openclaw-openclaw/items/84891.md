## Label Justifications

- `P2`: This is a user-visible Control UI session picker regression with a clear source path and limited blast radius.
- `impact:session-state`: The issue concerns valid persisted sessions becoming hidden from session selection after restart or upgrade, not deleted from disk.

## Summary

Keep open. Current main still loads the chat picker from a bounded 120-minute, 100-row, agent-scoped `sessions.list` result; agent switching only chooses among those already-loaded rows before falling back to the agent main session, while the gateway filters `activeMinutes` by `updatedAt`, matching the reported hidden-but-intact stale session behavior.

## What This Changes

The issue asks Control UI to keep existing non-dashboard channel sessions visible after gateway restart or upgrade and to prefer the selected agent's most recently active valid session instead of falling back to the dashboard session.

## Best Possible Solution

Keep the picker fast by default, but add an owner-approved path that deliberately recovers the latest durable session for the selected agent outside the recency window, with any show-all or retention control made explicit and tested.

## Reproduction Assessment

Yes, from source inspection: create an agent with a channel session whose `updatedAt` is older than 120 minutes, let the chat picker refresh with `activeMinutes: 120`, then switch to that agent; the stale row is filtered before `resolvePreferredSessionForAgent` can select it. I did not run a live gateway/UI replay in this read-only review.

## Solution Assessment

Unclear. Removing the bound or picking an arbitrary larger window is not clearly the best solution; maintainers should preserve the bounded fast path while intentionally recovering stale-but-valid sessions or exposing a deliberate show-all path.
