## Label Justifications

- `P2`: The report describes a user-visible session picker regression with a clear source path and limited blast radius.
- `impact:session-state`: The issue is about persisted sessions being hidden from UI selection after restart or upgrade, not lost on disk.

## Summary

Keep open. Current main and v2026.5.19 still use a 120-minute, 100-row chat session refresh; `sessions.list` filters by `updatedAt`, so older channel sessions are excluded before the agent picker can select them and the UI can fall back to the agent main session. This looks like a real session-state UX regression candidate, but the exact fix needs maintainer choice around recency bounds and large session stores.

## What This Changes

The issue asks Control UI to keep older persisted non-dashboard channel sessions visible in the chat session picker after gateway restart or upgrade, and to choose the selected agent's most recently active session instead of falling back to dashboard/main.

## Best Possible Solution

Keep the chat picker fast for normal use, but add an owner-approved path that surfaces the latest durable session for the selected agent outside the recency window and makes any show-all behavior explicit.

## Reproduction Assessment

Yes, from source inspection: create an agent with an older channel session row and a main/dashboard row, let the chat path request `sessions.list` with `activeMinutes: 120`, then switch to that agent; the older row is filtered before `resolvePreferredSessionForAgent` runs. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Unclear. Removing all bounds or choosing an arbitrary larger window is not obviously the best fix; a maintainable solution should preserve a bounded fast path while deliberately recovering the latest valid persisted session or offering a clear show-all path.
