## Label Justifications

- `P2`: The issue describes a user-visible session picker regression with a clear source path and limited blast radius.
- `impact:session-state`: Persisted sessions remain on disk but drift out of the UI's selectable session state after restart or upgrade.

## Summary

Keep open. Current main and the latest shipped release still use a recency-bound chat session refresh, so older valid channel sessions can be filtered out before the agent picker chooses a session and falls back to the agent main session.

## What This Changes

The issue asks the Control UI chat session picker to surface existing non-dashboard channel sessions for an agent after restart or upgrade instead of hiding older persisted sessions.

## Best Possible Solution

Keep the normal picker fast, but add an owner-approved path that can recover the latest valid persisted session for the selected agent outside the recent-session window and make any broader show-all behavior explicit.

## Reproduction Assessment

Yes, from source inspection: create an agent with an older channel session row and a main/dashboard row, let the chat path request sessions.list with activeMinutes: 120, then switch to that agent. The older row is filtered before resolvePreferredSessionForAgent runs, so no live gateway run was needed to identify the path.

## Solution Assessment

Unclear. The report points at the right failure mode, but removing all bounds or choosing an arbitrary larger window is not obviously the best fix; maintainers should choose the intended bounded recovery or explicit show-all behavior.
