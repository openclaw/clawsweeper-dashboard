## Label Justifications

- `P1`: The PR addresses a reported MiniMax OAuth regression that can add 30-50 seconds of auth delay on affected requests.
- `merge-risk: 🚨 auth-provider`: The diff changes how MiniMax OAuth expiry is interpreted, which affects credential freshness and refresh decisions for that provider.

## Summary

Keep open: current main still stores MiniMax token `expired_in` directly as `expires` while shared auth treats `expires` as an absolute timestamp, and this PR is a focused provider-local fix with sufficient terminal proof and no blocking review findings.

## What This Changes

The branch normalizes MiniMax OAuth token expiry values before storing them, adds focused expiry-format tests, and updates the raw-fetch allowlist for shifted MiniMax OAuth callsites.

## Best Possible Solution

Land the narrow provider-local expiry normalization if maintainers accept the OAuth-only scope, and make any broader MiniMax OAuth2 or refresh work reuse the same expiry contract.

## Reproduction Assessment

Yes, by source inspection: current main stores MiniMax `expired_in` directly as `expires`, while shared auth subtracts `Date.now()` from that field. I did not run a live MiniMax OAuth reproduction in this read-only review, but the PR body supplies after-fix terminal proof.

## Solution Assessment

Yes, normalizing MiniMax token expiry at the provider boundary is the narrowest maintainable fix for this OAuth path. Any recovery for already-bad stored profiles should be handled as separate refresh or re-login proof rather than by broadening core auth semantics.
