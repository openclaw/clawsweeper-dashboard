## Label Justifications

## Summary

Keep open: the expiry normalization may be useful, but this PR is not merge-ready because it lacks after-fix real behavior proof and adds no-op MiniMax `prepareRuntimeAuth` hooks that change auth-error retry/failover behavior without proving they solve the linked per-message delay.

## What This Changes

The PR normalizes MiniMax OAuth `expired_in` before storing credentials and adds MiniMax provider `prepareRuntimeAuth` hooks that return the resolved key or token unchanged.

## Best Possible Solution

Keep the expiry normalization only if it is backed by MiniMax endpoint proof and tests, and avoid a no-op runtime-auth hook unless it implements a real cross-run auth optimization without changing failover semantics.

## Reproduction Assessment

No high-confidence current-main reproduction of the reported 30-50s MiniMax auth delay was established. Source inspection does show the PR's no-op runtime-auth hook would create `runtimeAuthState` and consume auth errors as refresh attempts before normal profile rotation.

## Solution Assessment

No. Expiry normalization may be the right narrow fix if MiniMax returns relative seconds, but returning the same key from `prepareRuntimeAuth` is not the best way to solve a per-message cache problem and changes auth failure behavior.
