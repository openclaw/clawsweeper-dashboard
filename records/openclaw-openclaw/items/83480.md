## Summary

Keep open: this is a plausible fix for a real MiniMax auth regression, but the PR is not merge-ready because it lacks real behavior proof and the added no-op runtime-auth hooks duplicate the default direct-credential path while changing auth retry behavior.

## What This Changes

The PR normalizes MiniMax OAuth token expiry values during login and adds MiniMax `prepareRuntimeAuth` hooks that return the resolved credential for runtime auth state caching.

## Best Possible Solution

Keep the expiry normalization with focused regression coverage, remove or replace the identity runtime-auth hooks with a real provider-owned exchange only if needed, and require redacted live MiniMax auth-stage proof before merge.

## Reproduction Assessment

Partly yes by source inspection: current main stores MiniMax token endpoint `expired_in` directly into OAuth `expires`, while the auth store expects an absolute millisecond timestamp. I did not establish a live Windows/Feishu/MiniMax CN reproduction, and the API-key mode slowdown remains unproven by source here.

## Solution Assessment

No, not as written. The expiry normalization is the right direction for OAuth-login tokens, but the no-op `prepareRuntimeAuth` hooks duplicate existing direct credential storage and should be removed or replaced with a real exchange/cache design plus tests.
