## Summary

Keep open: the MiniMax OAuth expiry normalization is source-plausible and now has focused regression tests, but this external PR still lacks redacted after-fix MiniMax OAuth behavior proof and the linked slowdown evidence is broader than this OAuth-only path.

## What This Changes

The PR adds a MiniMax OAuth expiry normalizer, uses it when storing token endpoint `expired_in` values, adds unit coverage for relative seconds/Unix seconds/millisecond timestamps, and updates the raw-fetch guard line allowlist.

## Best Possible Solution

Land this as a MiniMax OAuth-only expiry fix after redacted real MiniMax Portal OAuth proof is added and the broader performance reports remain scoped to their own issues.

## Reproduction Assessment

Partly yes by source inspection: current main stores MiniMax token `expired_in` directly while the auth store expects an absolute millisecond timestamp. I did not establish a live MiniMax OAuth reproduction, and the linked production slowdown also contains non-OAuth evidence.

## Solution Assessment

Mostly yes for the MiniMax OAuth-token path: normalizing at the plugin-owned OAuth boundary is the narrow maintainable fix. It is not enough to claim the broader Feishu/MiniMax performance regression is solved without live OAuth proof and separate handling for API-key/session-loader evidence.
