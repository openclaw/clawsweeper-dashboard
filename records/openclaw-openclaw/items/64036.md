## Summary

Keep this issue open. Current main still has the reported final residual chunk behavior, the latest inspected release tag still contains it, and open draft PR #72491 already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix is merged or closed.

## What This Changes

The issue requests trimming and skipping the final residual chunk in `chunkTextByBreakResolver` so over-limit text does not return a trailing-whitespace final chunk.

## Best Possible Solution

Land PR #72491 or an equivalent narrow patch that trims the final residual chunk only after the split loop, skips it if empty, preserves existing early-return behavior, and adds regression coverage for the shared resolver plus any copied runtime chunker that must stay aligned.

## Reproduction Assessment

Yes. Current main has a deterministic source-level reproduction: when over-limit input leaves `remaining` as `! ` or `b ` after the loop, `src/shared/text-chunking.ts:30-31` pushes that value without `trimEnd()`.

## Solution Assessment

Yes. The narrow maintainable direction is to normalize the final residual chunk the same way loop-emitted chunks are normalized, then cover the reported counterexamples and the Matrix runtime copy without changing unrelated channel defaults or delivery APIs.
