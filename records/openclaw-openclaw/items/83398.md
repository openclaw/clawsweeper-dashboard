## Summary

Keep open: the report is recent and describes Feishu/Lark message loss where model output exists but no visible reply is queued. Current main has a source-level path that can produce the same `queuedFinal=false, replies=0` shape before Feishu delivery runs, but the exact LongCat beta trigger is not yet proven and the related open work does not cover ordinary non-command DMs.

## What This Changes

Feishu/Lark DMs should visibly reply when the model produces final text instead of completing with `replies=0` on v2026.5.16-beta.6.

## Best Possible Solution

Make an explicit maintainer policy call for Feishu/LongCat direct-chat visible replies, then implement a narrow fix or diagnostic in the shared source-reply policy or Feishu context boundary with regression coverage and redacted live Feishu proof.

## Reproduction Assessment

Partly: there is a high-confidence source-level reproduction for the `queuedFinal=false, replies=0` shape when direct/source delivery resolves to `message_tool_only`. The exact v2026.5.16-beta.6 LongCat Feishu setup has not been live-reproduced from the available issue data.

## Solution Assessment

Unclear as an automated fix: restoring automatic visible replies broadly would override documented tool-only behavior, while the safe narrow solution depends on confirming whether LongCat is entering the same source-reply suppression path or exposing a distinct Feishu dispatch regression.
