## Summary

Keep open: this is a recent Feishu/Lark message-loss regression report with independent rollback evidence in a related open issue, and current main still has a source-level path that can suppress final assistant text before Feishu delivery and produce `queuedFinal=false, replies=0`. The ordinary non-command DM case is not covered by the open text-command PR, and the remaining fix needs maintainer policy plus live Feishu proof.

## What This Changes

The issue reports Feishu/Lark direct messages on `v2026.5.16-beta.6` where the model generates a final assistant reply but the channel dispatch finishes with `queuedFinal=false, replies=0` and no visible reply.

## Best Possible Solution

Decide the Feishu/Lark direct-chat visible-reply compatibility policy, then land either a narrow fallback/diagnostic or migration guidance around the existing `messages.visibleReplies: "automatic"` workaround with regression tests and redacted live Feishu proof.

## Reproduction Assessment

No for the exact live beta.6 Feishu/LongCat setup: I did not run a credentialed Feishu DM reproduction. Yes at source level for the symptom shape: current code and tests show successful assistant output can be suppressed before `sendFinalReply` when direct/source delivery resolves to `message_tool_only`, yielding `queuedFinal=false` with no final reply queued.

## Solution Assessment

Unclear as an automated fix: setting direct replies back to automatic may restore the reporter's workflow, but it would override documented tool-only behavior for some harness/provider paths. The maintainable path is a maintainer policy decision plus a narrow code or docs change proven against real Feishu/Lark DMs.
