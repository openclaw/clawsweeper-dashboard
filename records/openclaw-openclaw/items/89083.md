## Summary

Keep open: this PR addresses a real current-main Feishu streaming final-overwrite path, but the head is not merge-ready because an exact duplicate final before `onIdle` can still be emitted as a second static card, and the visible Feishu/Lark client proof is still insufficient.

## What This Changes

The branch adds Feishu reply-dispatcher state and tests to commit the first streaming final, route later distinct final-shaped text through static delivery, and suppress duplicate finals.

## Best Possible Solution

Keep the Feishu dispatcher-layer fix, but add pre-idle duplicate suppression without marking unclosed content as delivered, decide the temporary proof logs, and require visible Feishu/Lark client proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, at source level: on PR head, deliver final text A and then the same final text A again before `onIdle`; `finalReplyCommitted` disables streaming reuse but `deliveredFinalTexts` does not contain A until close, so the duplicate falls through to static delivery. Current main also has a source-level reproduction for the original overwrite, with final B replacing `streamText` before idle close.

## Solution Assessment

No, not yet. The Feishu dispatcher is the right owner boundary, but the current patch needs a committed-final-text duplicate guard and visible client proof before it is the best safe fix.
