## Summary

This should stay open: current `main` still has a source-level path where `openclaw status --deep` enters the security audit and the sandbox-browser audit launches `docker ps` without its own timeout or abort budget, matching the reported hang.

## What This Changes

The issue asks for `openclaw status --deep` subprocess probes to complete within a bounded timeout and return partial diagnostics instead of hanging indefinitely.

## Best Possible Solution

Bound the Docker-backed sandbox-browser audit probes with a timeout or `AbortSignal`, keep the audit best-effort on timeout, and add regression coverage for a never-settling Docker subprocess.

## Reproduction Assessment

Yes at source level: a never-settling `execDockerRawFn` in `collectSandboxBrowserHashLabelFindings` would block the status security audit because current code does not add a timeout. I did not run the hanging command live in this read-only review.

## Solution Assessment

Yes; the narrowest maintainable fix is to add a bounded timeout/abort path to the existing Docker audit probe and preserve the current best-effort diagnostics, not add a new user-facing mode or config option.
