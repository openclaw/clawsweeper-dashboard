## Summary

Keep open: current main and v2026.5.19 still route `openclaw status --deep` through the security audit's sandbox-browser Docker collector, where `docker ps`/`inspect`/`port` calls are awaited without a signal or timeout despite the command's `--timeout` option.

## What This Changes

The issue asks for `openclaw status --deep` to bound subprocess probes and return partial diagnostics instead of hanging indefinitely.

## Best Possible Solution

Thread a bounded timeout or AbortSignal into the sandbox-browser Docker audit probes and surface a skipped-or-timeout diagnostic while leaving status output and sandbox runtime semantics intact.

## Reproduction Assessment

Yes. Source inspection shows `status --deep` enters the security audit path and awaits sandbox-browser Docker probes without passing a timeout or AbortSignal, so a test double that never settles from `docker ps` should reproduce the hang deterministically.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing timeout budget or a small default for the Docker audit calls and keep returning partial diagnostics; no new config option or product decision is needed.
