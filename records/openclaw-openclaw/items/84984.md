## Summary

Keep open: current main still has a source-proven unbounded Docker subprocess path in text `status --deep`, and the reported hang is a valid narrow availability bug rather than stale cleanup.

## What This Changes

The issue requests that `openclaw status --deep` stop hanging indefinitely when its Docker-backed subprocess probe does not return.

## Best Possible Solution

Keep the deep status diagnostics, but bound the sandbox-browser Docker audit probes and return partial status/audit output with a clear skipped-or-timeout diagnostic when Docker does not respond.

## Reproduction Assessment

Yes, by source inspection rather than live execution: `status --deep` enters the security audit path, which reaches a Docker `ps` audit probe without passing a timeout or AbortSignal. A test double that never resolves from that Docker probe should reproduce the same hang deterministically.

## Solution Assessment

Yes: the narrow maintainable fix is to bound the Docker audit calls using the existing timeout budget or a small default and keep returning partial diagnostics. This should not need a new config option or a change to sandbox runtime semantics.
