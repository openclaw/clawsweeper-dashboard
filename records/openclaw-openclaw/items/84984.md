## Summary

Keep open: current `main` still has a source-proven unbounded Docker subprocess path in text `status --deep`, and the issue is a valid narrow CLI availability bug rather than cleanup.

## What This Changes

`openclaw status --deep` should complete within a bounded time or return partial diagnostics instead of hanging on a subprocess probe.

## Best Possible Solution

Keep the deep status diagnostics, but bound the sandbox-browser Docker audit probes and return partial status/audit output with a clear skipped-or-timeout diagnostic when Docker does not respond.

## Reproduction Assessment

Yes, by source inspection rather than live execution: `status --deep` awaits a security audit that reaches Docker commands without a deadline. A test double that never resolves from the Docker probe should reproduce the hang deterministically.

## Solution Assessment

Yes, the narrow maintainable fix is to bound the Docker audit calls using the existing timeout budget or a small default and keep returning partial diagnostics. This should not need a new config option or a change to sandbox runtime semantics.
