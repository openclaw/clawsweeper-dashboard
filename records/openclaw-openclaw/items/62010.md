## Summary

Keep open. Current main does not implement the requested WhatsApp-scoped media root config, and this PR is still a plausible feature, but the current head remains incorrect: configured roots are passed only through the fallback `mediaLocalRoots` path, `file://` roots validate without normalization, WhatsApp policy is added in core outbound delivery, and real after-fix WhatsApp proof is still missing.

## What This Changes

The PR adds `channels.whatsapp.mediaLocalRoots` to WhatsApp config/types/schema, generated config baseline hashes, and outbound delivery tests for channel, account, and default-account roots.

## Best Possible Solution

Keep the feature WhatsApp-scoped, implement the root merge in the WhatsApp-owned outbound path or a generic channel contract, normalize configured roots into filesystem paths, feed them into the canonical `mediaAccess.localRoots` consumed by the loader, and keep generated config baselines plus channel/account/default-account regression coverage aligned.

## Reproduction Assessment

Yes, for the PR defects by source inspection. Configure normal outbound `mediaAccess.localRoots` plus `channels.whatsapp.mediaLocalRoots`; PR head sends the configured root only as fallback `mediaLocalRoots`, while the WhatsApp loader prefers `mediaAccess.localRoots` and rejects files under the new root.

## Solution Assessment

No. The feature direction is reasonable, but the implementation should not split roots across two option channels or add WhatsApp-specific policy in core delivery; the safer path is plugin-owned normalized merging into canonical media access.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Merge configured roots into mediaAccess:** `src/infra/outbound/deliver.ts:304-305`
  - body: The patch builds a merged WhatsApp root list but still returns `mediaAccess: params.mediaAccess`. The WhatsApp loader prefers `mediaAccess.localRoots` whenever it is present and only falls back to `mediaLocalRoots` when that list is empty, so files under the new configured root can still be rejected in normal outbound sends. Merge the normalized WhatsApp roots into the `mediaAccess.localRoots` object passed to the loader, or make the loader combine both inputs consistently.
  - confidence: 0.93
- **[P2] Normalize or reject file URL roots:** `src/config/zod-schema.providers-whatsapp.ts:42`
  - body: This schema accepts `file://` roots, but media URLs are converted to filesystem paths before allowlist checks while root entries are compared as raw strings. A configured root like `file:///tmp/wa-media` validates but will not match `/tmp/wa-media/photo.png`, so either normalize root entries before delivery or reject `file://` roots here.
  - confidence: 0.89
- **[P2] Keep WhatsApp root policy in the plugin:** `src/infra/outbound/deliver.ts:273-291`
  - body: This adds a WhatsApp-specific branch to core outbound delivery and reads `channels.whatsapp` directly. The repo architecture keeps owner-specific channel policy and defaults in the owning plugin, and the WhatsApp adapter already receives `cfg`, `accountId`, and `mediaAccess`, so the configured-root merge should live in the WhatsApp outbound path or behind a generic channel contract.
  - confidence: 0.86
