## Summary

Keep this PR open. Current main still lacks the requested WhatsApp-scoped outbound media roots, and the PR is a useful implementation attempt, but the latest branch still has correctness blockers: configured roots are not merged into the canonical media access object used by real media loading, `file://` roots validate but do not match the filesystem-path allowlist check, and the user-facing feature is missing the required changelog entry.

## What This Changes

The PR adds WhatsApp `mediaLocalRoots` config/types/schema and generated config baseline updates, then attempts to merge channel and account roots into outbound WhatsApp media delivery with focused config and outbound tests.

## Best Possible Solution

Revise the PR so WhatsApp-configured roots are normalized once and become part of the same allowlist that real media loaders consume, preferably in the WhatsApp-owned outbound path or through a generic media-access merge seam. Keep the feature WhatsApp-scoped, add a real-loader regression for scoped `mediaAccess.localRoots`, handle `file://` roots consistently, add the changelog entry, and rebase onto current main before normal review.

## Reproduction Assessment

Yes. Configure a WhatsApp media send where `mediaAccess.localRoots` is present and `channels.whatsapp.mediaLocalRoots` points outside that list; the PR passes the extra root only as `mediaLocalRoots`, while the WhatsApp loader selects `mediaAccess.localRoots` first, so the configured root is ignored. A second deterministic path is `mediaLocalRoots: ["file:///tmp/wa-media"]`, which validates but cannot match a media path converted to `/tmp/wa-media/...` by the local media loader.

## Solution Assessment

No. The feature direction is reasonable, but the current patch splits roots across two channels instead of updating the canonical media access consumed by the loader, and it accepts `file://` roots without normalizing them for root comparison.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Merge configured roots into mediaAccess:** `src/infra/outbound/deliver.ts:305`
  - body: The PR only adds configured WhatsApp roots to `mediaLocalRoots` while forwarding the original `mediaAccess`. In paths where `mediaAccess.localRoots` is present without a read override, `loadOutboundMediaFromUrl` prefers that old list and never consults `mediaLocalRoots`, so files under the configured WhatsApp root are still rejected. Merge the roots into the `mediaAccess.localRoots` list that the loader actually consumes, or normalize them inside the WhatsApp media loader before choosing roots.
  - confidence: 0.93
- **[P2] Normalize or reject file URL roots:** `src/config/zod-schema.providers-whatsapp.ts:42-43`
  - body: The schema accepts `file://` entries as roots, but media URLs are converted from `file://` to filesystem paths before allowlist checks, while root entries are compared as raw path strings. A root like `file:///tmp/wa-media` validates but will not match `/tmp/wa-media/photo.png`; either normalize roots to filesystem paths before matching or reject `file://` roots in config.
  - confidence: 0.89
- **[P3] Add the required changelog entry:** `src/config/types.whatsapp.ts:80`
  - body: This adds a user-facing WhatsApp config option and outbound behavior, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing `feat` changes to include a changelog entry, so add one before merge.
  - confidence: 0.94
