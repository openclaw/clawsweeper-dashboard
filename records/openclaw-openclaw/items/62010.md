## Summary

Keep open: current main does not include the WhatsApp `mediaLocalRoots` config surface, and the PR is a plausible feature, but the current patch still misses the canonical `mediaAccess` path, accepts nonfunctional `file://` root entries, crosses the core/plugin boundary, and lacks real behavior proof.

## What This Changes

The PR adds a WhatsApp `mediaLocalRoots` config field, schema validation, generated config baseline hashes, and outbound delivery tests/root merging for channel and account scopes.

## Best Possible Solution

Define one configured local-media root contract, validate and normalize roots into canonical `mediaAccess.localRoots`, keep WhatsApp-specific account policy out of shared delivery unless a generic seam owns it, and require redacted real WhatsApp media-send proof.

## Reproduction Assessment

Yes, for the PR defects by source inspection. In normal outbound media sends current main builds `mediaAccess.localRoots`, and the PR adds configured roots only to `mediaLocalRoots` while the WhatsApp loader prefers `mediaAccess.localRoots`.

## Solution Assessment

No. The direction may be useful, but the current patch should normalize configured roots into the canonical media-access path and either keep WhatsApp account resolution in the WhatsApp plugin or move the feature into a generic configured-root resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Merge configured roots into mediaAccess:** `src/infra/outbound/deliver.ts:305`
  - body: The patch computes the merged WhatsApp roots in `mediaLocalRoots`, but still returns `mediaAccess: params.mediaAccess`. WhatsApp media loading uses `mediaAccess.localRoots` whenever present and only falls back to `mediaLocalRoots`, so normal sends with existing media access can still reject files under the configured root.
  - confidence: 0.93
- **[P2] Normalize or reject file URL roots:** `src/config/zod-schema.providers-whatsapp.ts:42`
  - body: The schema accepts `file://` root entries, but `loadWebMedia` converts media URLs to filesystem paths before checking the allowlist. A root like `file:///tmp/wa-media` validates but will not match `/tmp/wa-media/photo.png`, so normalize those roots before delivery or reject them in the schema.
  - confidence: 0.89
- **[P2] Keep WhatsApp policy out of shared delivery:** `src/infra/outbound/deliver.ts:273-288`
  - body: `createChannelOutboundContextBase` now reads `channels.whatsapp` and resolves the default WhatsApp account inside generic outbound delivery. That makes shared delivery own channel-specific account policy; move this into the WhatsApp outbound path or a generic configured-root resolver.
  - confidence: 0.84
