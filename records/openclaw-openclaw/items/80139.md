## Summary

Keep open: the PR addresses a valid source-reproducible cron status gap and includes sufficient after-fix proof, but latest head still has a QA wait predicate regression that can match stale non-ok terminal rows before the newly triggered run finishes.

## What This Changes

The PR adds a first-class cron `warning` status for delivery-only failures across cron dispatch/state, gateway schema, CLI/UI, docs, hooks, QA helpers, and tests, plus small run-log append and OpenAI stream type fixes.

## Best Possible Solution

Fix the QA wait predicate and add regression coverage for stale non-ok terminal rows, then land the warning-status model if maintainers accept the public cron status contract.

## Reproduction Assessment

Yes, by source inspection. Current main can return `status: "error"` from successful isolated cron execution when final delivery target resolution or direct send fails, and the linked issue plus PR body provide the concrete scenario and after-fix output.

## Solution Assessment

No, not merge-ready yet. The additive `warning` status is a plausible maintainable direction, but the QA wait helper must keep the `afterTs` guard around every terminal status before this patch is correct.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Preserve the timestamp guard for all terminal statuses:** `extensions/qa-lab/src/cron-run-wait.ts:45-48`

Keep the timestamp check outside a parenthesized terminal-status test.
