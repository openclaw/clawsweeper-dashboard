## Summary

Keep open: current main still records delivery-only cron failures as `error`, and this PR addresses the linked warning-status request, but the PR head is not merge-ready because it can let stale non-ok QA run-log entries satisfy the completion wait.

## What This Changes

The PR adds a first-class cron `warning` status across cron dispatch/state, gateway schema, CLI/UI, docs, plugin hook types, QA helpers, and tests, plus small run-log append and OpenAI stream type fixes.

## Best Possible Solution

Fix the QA wait predicate and stale non-ok regression coverage, then land the warning-status contract only after maintainer acceptance and a refresh onto current cron state handling.

## Reproduction Assessment

Yes. Current main is source-reproducible: delivery-target failures still return `status: "error"`, and the cron status contract still lacks `warning`; the linked issue and PR body provide the concrete delivery-failure scenario and after-fix output.

## Solution Assessment

No, not merge-ready yet. The additive warning status is a plausible maintainable direction, but the QA helper must keep the timestamp guard around every terminal status before this patch is correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve the timestamp guard for all terminal statuses:** `extensions/qa-lab/src/cron-run-wait.ts:45-48`

Keep the timestamp check outside a parenthesized terminal-status test.
