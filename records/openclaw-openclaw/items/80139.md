## Summary

Keep open: current main still lacks the requested warning status and this PR is the active replacement for the closed earlier copy, but the patch has a concrete QA wait predicate regression that should be fixed before merge.

## What This Changes

The PR adds a first-class cron `warning` status for successful isolated runs with failed final delivery across cron dispatch/state, gateway schema, CLI/UI, docs, hooks, QA helpers, and tests, plus small run-log append and OpenAI stream type fixes.

## Best Possible Solution

Fix the QA wait predicate and add regression coverage for stale non-ok rows, then land the additive warning-status model if maintainers accept the public cron status contract.

## Reproduction Assessment

Yes. Current main is source-reproducible: successful isolated cron execution can flow into delivery-target or direct-send failure paths that return `status: "error"`, and the linked issue plus PR body provide the concrete scenario and after-fix output.

## Solution Assessment

No, not merge-ready yet. The additive `warning` status is a plausible maintainable direction, but the QA wait helper must keep the `afterTs` guard around every terminal status before this patch is correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve the timestamp guard for all terminal statuses:** `extensions/qa-lab/src/cron-run-wait.ts:45-48`
  - body: This predicate now applies `entry.ts >= params.afterTs` only to `ok` runs because `&&` binds more tightly than `||`. Any older `error`, `skipped`, or `warning` row can satisfy the wait before the newly triggered run finishes, so qa-lab scenarios can pass or fail against stale run-log data. Keep the timestamp check outside a parenthesized terminal-status test.
  - confidence: 0.94
