## Summary

Keep open. The cron warning-status direction matches the linked report and the supplied runtime proof is sufficient, but the PR introduces a QA-lab wait regression that can make cron validation finish against stale history.

## What This Changes

This PR adds a cron `warning` status for successful isolated cron execution with failed final delivery, updates protocol/UI/docs/tests, and includes run-log and OpenAI stream compatibility adjustments.

## Best Possible Solution

Keep the first-class warning status approach, but restore the timestamp guard for every terminal status in the QA helper before normal maintainer review and landing.

## Reproduction Assessment

Yes. The original behavior is source-reproducible: current main returns `error` for unresolved delivery targets, and the PR body supplies after-fix output from real cron modules showing the new warning path.

## Solution Assessment

No, not as submitted. The core warning-status design is a maintainable additive solution, but the QA helper predicate needs a small fix so validation waits for the current run instead of stale terminal entries.
