## Summary

Keep open: this is meaningful memory-lancedb work, but it is not merge-ready because the replace path still deletes before the replacement is committed, the audit writer has a concrete unsafe-preexisting-path issue, public docs contradict the preserved-ID implementation, and the latest head still needs after-fix real behavior proof.

## What This Changes

The branch adds a memory-lancedb `memory_refresh` tool with search and replace modes, per-ID locking for refresh/forget, an audit JSONL write path, manifest/docs updates, and extensive refresh tests.

## Best Possible Solution

Implement refresh on a no-delete-first LanceDB update or merge-insert path, keep ID-preserving semantics, harden the audit writer, align public/package docs, remove release-owned changelog edits, and add redacted after-fix live proof before merge.

## Solution Assessment

No. A refresh tool is plausible, but the current delete-then-insert design is not the best fix for a data-loss window when pinned LanceDB exposes update and merge-insert APIs that can avoid deleting the original row first.
