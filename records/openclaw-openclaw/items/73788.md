- **[P1] Replace without deleting the row first:** `extensions/memory-lancedb/index.ts:2052-2062`
  - body: The refresh path still deletes the existing row before inserting the replacement.   - confidence: 0.86
- **[P1] Always tighten the audit log mode:** `extensions/memory-lancedb/index.ts:2133-2136`
  - body: This branch only calls `chmod(0o600)` when the audit file did not already exist. A preexisting `refresh-audit.jsonl` created with loose permissions stays readable by other local users even though it now receives memory ids and timestamps.
  - confidence: 0.9
- **[P2] Validate refresh importance like memory_store:** `extensions/memory-lancedb/index.ts:1930-1931`
  - body: `importance` is declared as a bare number and then written directly into the replacement row, so values outside the documented 0-1 range can be persisted. Mirror `memory_store`'s `optionalFiniteNumberSchema` plus `readFiniteNumberParam(..., { min: 0, max: 1 })` handling before storing.
  - confidence: 0.82
- **[P2] Keep refresh docs aligned with preserved ids:** `docs/plugins/memory-lancedb.md:259-281`
  - body: The new docs still say the tool atomically replaces a row and that the replacement gets a fresh id, but the implementation is explicitly best-effort and preserves the original id. Users will rely on the wrong contract unless the docs match the actual behavior.
  - confidence: 0.92
