## Summary

Keep open. Current main still lacks `memory_refresh`, and the earlier related PR was closed unmerged, so this PR is not obsolete. The current force-pushed diff is a plausible plugin-owned implementation and no longer includes the stale session-file changes from the body, but it is not merge-ready because the new audit log uses default filesystem permissions and the new user-facing tool has no docs or changelog coverage.

## What This Changes

The PR adds a `memory_refresh` tool to the bundled `memory-lancedb` plugin with search-only conflict preview, per-ID serialized replace/forget operations, rollback/audit behavior, and mocked tests for the new paths.

## Best Possible Solution

Keep the implementation in the `memory-lancedb` plugin if maintainers accept the feature, but tighten audit-log persistence to private permissions, document the new tool and audit behavior, add the required changelog entry, and then validate the targeted plugin tests plus the changed gate before merge.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. The current-main gap is high-confidence from source and docs: only three memory-lancedb tools are registered and documented, while the PR adds a fourth with mocked coverage for its main branches.

## Solution Assessment

Unclear as submitted. A plugin-owned `memory_refresh` tool is the right boundary, but the branch needs the permission fix plus docs/changelog follow-through, and maintainers should confirm that delete/insert with a new ID is the desired replace contract.
