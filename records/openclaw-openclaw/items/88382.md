## Summary

Keep open: current main still compares the next canonical config write against the raw previous file bytes, so the linked Windows/PowerShell false-positive remains unresolved until this protected maintainer draft or an equivalent fix lands.

## What This Changes

The PR changes config write anomaly detection to compare size-drop protection against canonicalized previous config bytes for valid existing configs, while retaining raw-byte fallback behavior for invalid configs and adding regression coverage.

## Best Possible Solution

Land the writer-level canonical-size fix after maintainer review, preserving raw-byte fallback for invalid configs and destructive-write rejection while allowing format-only JSON/BOM rewrites to persist.

## Reproduction Assessment

Yes, from source inspection: current main compares raw previous config bytes against the next canonical write, so a valid verbose/BOM JSON file can look like a destructive size drop. I did not run a live Windows/PowerShell reproduction in this read-only review.

## Solution Assessment

Yes, this appears to be the narrow writer-level fix rather than an auth-only workaround: canonicalizing the previous valid config before size comparison targets format-only false positives, while raw-byte fallback keeps invalid configs protected.
