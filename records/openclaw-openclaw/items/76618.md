## Summary

Keep this PR open: it is a plausible narrow Comfy Cloud polling fix, but it is not merge-ready because the reported `success` status still lacks real Cloud proof and the new `error` terminal path drops Comfy Cloud's documented `error_message` detail.

## What This Changes

The PR changes the bundled Comfy workflow runtime to treat Comfy Cloud `success` as complete and `error` as terminal failure during status polling.

## Best Possible Solution

Preserve Comfy Cloud `error_message`, add focused status coverage for `success`, `error`, and `error_message`, and land only after redacted real Comfy Cloud success/error proof is added.

## Reproduction Assessment

Partly: source inspection shows current main would keep polling until timeout for the official `error` status, and the same path would also keep polling for the reported `success` value. I did not run a live Comfy Cloud job, and `success` is not present in the official status enum I checked.

## Solution Assessment

No: the polling function is the right boundary, but this patch should preserve `error_message`, add focused coverage, and include real Comfy Cloud proof before it is the best merge path.
