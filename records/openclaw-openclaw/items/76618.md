## Summary

Keep open: the PR targets a real Comfy plugin polling bug, but the patch is incomplete because the new `error` path drops the documented `error_message` failure detail and the external PR has no after-fix real behavior proof. ## What This Changes

The PR changes Comfy Cloud polling to treat `success` as completed and `error` as a terminal failure in `extensions/comfy/workflow-runtime.ts`.

## Best Possible Solution

Land a small Comfy plugin fix that accepts the relevant terminal statuses, preserves `error_message`, adds focused status regression coverage, and is backed by redacted Comfy Cloud runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main polls until timeout for Cloud status values outside `completed`, `failed`, and `cancelled`, while the status API documents `error` as terminal. I did not live-run Comfy Cloud. ([docs.comfy.org](https://docs.comfy.org/api-reference/cloud/job/get-job-status?utm_source=openai))

## Solution Assessment

Mostly yes: the Comfy plugin polling path is the right boundary, but this branch should also parse `error_message`, add focused status tests, and prove the Cloud behavior before merge.
