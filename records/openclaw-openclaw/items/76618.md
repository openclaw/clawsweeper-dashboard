## Summary

Keep this PR open: current main still treats Comfy Cloud `error` and the reported `success` status as non-terminal, but the PR needs a small repair before merge because the documented failure detail field is `error_message` and the user-facing fix lacks a changelog entry.

## What This Changes

The branch updates Comfy Cloud polling to return on `success` as well as `completed` and throw on `error` as well as `failed`/`cancelled` in `extensions/comfy/workflow-runtime.ts`.

## Best Possible Solution

Land a narrow Comfy-plugin fix that treats the real terminal statuses as terminal, preserves Comfy Cloud `error_message`, tolerates the reported `success` alias if needed, and adds regression coverage plus a changelog entry.

## Reproduction Assessment

Source-reproducible, not live-reproduced: current main polls forever for any status other than `completed`, `failed`, or `cancelled`, so documented `error` and the PR-reported `success` state do not terminate the wait loop.

## Solution Assessment

Mostly yes, but not complete: keeping this in the Comfy plugin is the right boundary, while the PR should also parse `error_message`, add focused status tests, and add the required changelog entry.
