## Summary

Keep this PR open, but it is not merge-ready: the Comfy plugin is the right boundary for this timeout/error fix, while the patch still misses the documented `error_message` field and the required changelog coverage.

## What This Changes

The PR changes Comfy Cloud polling to treat `success` as complete and `error` as terminal failure in `extensions/comfy/workflow-runtime.ts`.

## Best Possible Solution

Land a narrow Comfy plugin fix that recognizes the real terminal status values, preserves provider failure text, adds regression coverage for status polling, and records the user-facing timeout/error fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Source inspection shows current main only returns on `completed` and only throws on `failed` or `cancelled`, so a mocked Comfy Cloud `error` response follows the polling loop until timeout; I did not live-run Comfy Cloud.

## Solution Assessment

Mostly yes, but incomplete. The Comfy plugin runtime is the correct owner boundary, but the patch should also parse `error_message`, add focused regression tests, and add the required changelog entry before merge.
