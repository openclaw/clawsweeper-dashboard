## Summary

Keep open: current main still has a source-provable heartbeat/pending-final-delivery path where stored heartbeat text is replayed before the normal heartbeat prompt can run, and the related fixed delivery-telemetry issue does not cover this session-state loop.

## What This Changes

The issue requests preventing heartbeat acknowledgements and no-target heartbeat output from becoming stuck `pendingFinalDelivery` state, plus surfacing stale pending-delivery diagnostics.

## Best Possible Solution

Land a focused heartbeat/session-state fix that prevents heartbeat ack text from entering durable final-delivery replay, safely clears or ignores stale no-target heartbeat pending state, records meaningful failure state where dispatch really fails, and adds regression coverage plus a changelog entry.

## Reproduction Assessment

Yes, at source level: seed a main heartbeat session with `pendingFinalDeliveryText`, then a heartbeat run will replay that stale text and update pending-attempt state before the normal heartbeat prompt executes. I did not run a live gateway fixture during this read-only review.

## Solution Assessment

Yes, a narrow heartbeat-specific normalization and replay fix is the maintainable path. A broader durable-delivery redesign is not needed, but the fix must avoid changing successful real-channel pending delivery semantics.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79258. Current main can persist heartbeat acknowledgement text or no-target heartbeat output into `pendingFinalDeliveryText`; later heartbeat runs replay that stale text before the normal heartbeat prompt, increment attempt state, keep `pendingFinalDeliveryLastError` null, and leave the heartbeat stuck. Keep the fix narrowly scoped: prevent heartbeat output that strips to empty under the existing `HEARTBEAT_OK` contract from being written to pending final delivery; safely clear or ignore stale heartbeat/no-target pseudo-target pending state during heartbeat replay; record useful failure state where dispatch actually fails; preserve durable pending-final-delivery recovery for real user/channel targets. Add focused regression coverage for bare `HEARTBEAT_OK`, short preamble/suffix ack text, no-target heartbeat delivery, and stale pending replay. Add a single-line user-facing `CHANGELOG.md` fix entry. Do not add a config option, do not redesign the durable delivery queue, and do not change real-channel success clearing semantics.
