## Summary

Keep open: current main still has a source-provable heartbeat/pending-final-delivery loop path, and the related reports are broader or sibling symptoms rather than a merged fix for this issue.

## What This Changes

The issue reports that heartbeat runs can leave the main agent session with pending final delivery to the heartbeat pseudo-target, causing repeated silent retries that block future heartbeat work.

## Best Possible Solution

Keep the issue open and land a focused heartbeat/session-state fix that prevents heartbeat ack text from entering pending-final-delivery replay, clears or safely ignores stale heartbeat/no-target pending state, and adds regression coverage plus a changelog entry.

## Reproduction Assessment

Yes: seed a main heartbeat session with `pendingFinalDeliveryText` and run `runHeartbeatOnce`; source shows heartbeat replay updates attempt state and returns the stale pending text before the normal heartbeat prompt can execute. I did not execute that fixture in this read-only review.

## Solution Assessment

Yes, the best fix is narrow heartbeat-specific normalization and replay handling, not a broad durable-delivery redesign. Add failure visibility only where dispatch actually fails and avoid changing real-channel delivery semantics.
