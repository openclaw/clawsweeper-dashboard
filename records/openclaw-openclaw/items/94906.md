## Label Justifications

- `P1`: The report describes long-lived embedded sessions becoming repeatedly unusable after one provider-rejected image block.
- `impact:session-state`: The affected image block remains in session history and poisons later prompt replay until reset or manual transcript repair.
- `impact:message-loss`: Subsequent user messages in the same session can repeatedly fail with only a generic channel-visible error.

## Summary

Keep open: current main still has only age-window image pruning and corrupted-image file repair, while recent structurally valid image blocks remain replayed after a provider rejection and structured provider errors still collapse to generic copy.

## What This Changes

The issue requests a recovery path that removes or downgrades provider-rejected image blocks from session history while preserving text context.

## Best Possible Solution

Add a narrow, maintainer-approved image-rejection recovery contract that detects proven provider image-block failures, downgrades only the offending prompt-history image data while preserving text/media references, and surfaces actionable retry guidance.

## Reproduction Assessment

Yes at source level: current main preserves recent image/toolResult blocks and has no targeted strip command or post-provider-rejection image recovery. No live MiniMax/Feishu repro was established in this read-only pass.

## Solution Assessment

No, the current old-history pruning and corrupted-image repair paths are not sufficient for this issue. The best solution needs maintainer direction on automatic recovery versus a user command, plus live proof for the provider-specific rejection shape.
