## Label Justifications

- `P1`: The report describes a real long-lived channel session becoming repeatedly unusable after a provider rejects one recent image block.
- `impact:session-state`: The poisoned recent image block remains in session history and is replayed on every subsequent turn.
- `impact:message-loss`: Later user messages in the same session can fail without useful assistant replies until reset or manual transcript surgery.

## Summary

Keep open: current main still only prunes older already-processed image history, preserves recent image/toolResult blocks, and genericizes provider schema rejections; related open PRs cover adjacent failover or stale-image contamination but not targeted recovery for a recently rejected image block.

## What This Changes

The issue requests a recovery path that preserves text context while removing or downgrading provider-rejected recent image blocks from session history.

## Best Possible Solution

Add a targeted, attempt-scoped recovery path that recognizes provider image-block rejection, preserves text context, removes or downgrades only the likely offending recent image replay block, and surfaces actionable user copy without broadly disabling native multimodal images.

## Reproduction Assessment

Source-reproducible, but not live-reproduced: current main and v2026.6.8 preserve recent image/toolResult blocks and genericize provider schema rejection text. I did not run a MiniMax/Feishu live rejection path in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product shape. The best direction is targeted recovery for recent image-block rejection, while generic content-policy failover or broad image stripping would only partially address the poisoned-history problem.
