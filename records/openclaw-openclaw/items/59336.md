## Summary

Keep open: this PR addresses a real current-main Config Raw mode regression, but the proposed safe-bin normalization change introduces a security-sensitive policy regression and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes config raw-redaction round-trip comparison to use the source config with recursive undefined-key stripping, and changes exec safe-bin normalization to delete empty safe-bin fields.

## Best Possible Solution

Land a revised fix that compares raw round-trip output against the source config while preserving safe-bin policy shadows or sanitizing prototypes, with focused regression tests and redacted real Control UI or terminal proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main writes undefined safe-bin fields, compares restored raw text against materialized `snapshot.config`, and the UI disables Raw mode when the fallback sets `raw` to `null`. I did not run the UI because this review is read-only.

## Solution Assessment

No. Comparing against source config and stripping undefined values in the round-trip check is the right direction, but deleting safe-bin fields is not the narrowest maintainable fix because it changes exec policy semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve safe-bin shadows when clearing fields:** `src/config/normalize-exec-safe-bin.ts:28-34`
  - body: Deleting these fields removes the own-property shadow that previously masked inherited `safeBinProfiles` or `safeBinTrustedDirs`. Runtime policy later reads those properties through ordinary property access, so a polluted or inherited config object can regain policy values after normalization; keep the shadow or strip prototype data and limit undefined removal to the raw comparison path.
  - confidence: 0.84
