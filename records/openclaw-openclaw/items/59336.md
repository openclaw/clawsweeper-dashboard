## Summary

Keep this PR open. Current main still has the raw-mode fallback behavior the PR targets, but the patch needs maintainer review before merge because it changes exec safe-bin normalization semantics and omits the required user-facing changelog entry.

## What This Changes

The PR changes config raw redaction to compare restored raw snapshots against source config with undefined-key stripping, and changes exec safe-bin normalization to delete empty safe-bin profile/trusted-dir fields.

## Best Possible Solution

Land a maintainer-reviewed fix that keeps Raw mode available for source-vs-materialized snapshot differences while preserving exec safe-bin prototype-hardening semantics, adding regression tests for undefined safe-bin keys, source-config comparison, and the `file` substring redaction case, plus the required changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main creates own `undefined` safe-bin keys during snapshot materialization and then compares restored JSON5 raw output against `snapshot.config`, which cannot deep-equal JSON-parsed source for those keys or for materialization-only defaults.

## Solution Assessment

No. Comparing against source config and stripping undefined during the round-trip check is the right direction, but changing materialized exec config by deleting safe-bin keys is not the safest fix; keep runtime policy semantics intact and limit the compatibility normalization to the redaction comparison path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Preserve safe-bin shadows when clearing fields:** `src/config/normalize-exec-safe-bin.ts:28-34`
  - body: Deleting these fields changes more than the raw-mode comparison: if an `exec` object has an own empty `safeBinProfiles` or `safeBinTrustedDirs` value while its prototype has one, this delete reveals the inherited value. Runtime safe-bin policy later reads these properties with normal optional chaining, so a normalization pass can unintentionally re-enable inherited exec policy data where current main kept an own `undefined` shadow. Please keep the runtime object shadowed or sanitize to own plain data, and strip `undefined` only for the raw round-trip comparison.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `CHANGELOG.md:12`
  - body: This is a user-facing Control UI/config editing fix, but the PR does not add an Unreleased `CHANGELOG.md` Fixes entry. OpenClaw policy requires one for user-visible fixes, with contributor credit that is safe to publish.
  - confidence: 0.9
