## Summary

Keep open: the Raw-mode regression is still present on current main and v2026.5.12, but this PR still has a blocking safe-bin policy regression and lacks after-fix real behavior proof.

## What This Changes

The PR changes config raw-redaction round-trip comparison to use a source config with recursive undefined-key stripping and changes exec safe-bin normalization to delete empty safe-bin fields.

## Best Possible Solution

Land a revised fix that compares raw round-trip output against a JSON-serializable authored/source config shape while preserving safe-bin policy clearing semantics, with focused regression tests and redacted real UI or terminal proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: safe-bin normalization writes undefined own fields, raw redaction compares restored raw text against materialized `snapshot.config`, and Control UI disables Raw when fallback returns `raw: null`; I did not run the UI in this read-only review.

## Solution Assessment

No. Comparing against a source-shaped config and stripping undefined values is the right direction, but deleting safe-bin fields is not the narrowest maintainable fix because it changes exec policy semantics; preserve own sanitized shadows or strip prototype data instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve safe-bin shadows when clearing fields:** `src/config/normalize-exec-safe-bin.ts:28-34`

Runtime policy later reads `safeBinProfiles` and `safeBinTrustedDirs` through ordinary property access, so an exec object with inherited or polluted policy values can regain safe-bin profiles or trusted dirs after normalization; preserve an own shadow or sanitize prototype data instead.
