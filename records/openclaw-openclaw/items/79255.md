## Summary

Close as not actionable in OpenClaw source: Telegram's native read-check state is platform-controlled, current docs explicitly say the Bot API has no read-receipt support, and OpenClaw already exposes typing/progress-draft signals for agent work.

## What This Changes

Request native send, received, and read status for Telegram and other chat messages so users can distinguish gateway receipt from agent processing.

## Best Possible Solution

Keep native read receipts channel-owned only where the platform exposes them, and use OpenClaw typing/progress drafts for Telegram in-progress visibility.

## Reproduction Assessment

Not applicable. This is a feature request, and source/docs inspection shows Telegram receive acknowledgements, agent progress, and native read receipts are separate surfaces.

## Solution Assessment

No. Mapping gateway receipt or agent processing into Telegram blue checks is not a maintainable OpenClaw fix because the Bot API does not expose that control; existing progress and typing signals are the supported path.
