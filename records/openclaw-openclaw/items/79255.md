## Summary

Keep open: this is a real feature request for user-visible delivery/read state, but current evidence shows Telegram normal bot chats cannot use native read receipts and OpenClaw only has adjacent ack/progress/provider-acceptance surfaces today.

## What This Changes

Request user-visible sent, received, and read status distinctions for Telegram and other message channels.

## Best Possible Solution

Define a channel lifecycle/status UX that separates provider acceptance, OpenClaw agent dispatch, and platform read support per channel, reusing existing ack reactions and live previews where possible.

## Reproduction Assessment

Not applicable: this is an enhancement request rather than a broken existing contract. Source and docs confirm the current Telegram lifecycle and feedback surfaces, but there is no current-main failure to reproduce.

## Solution Assessment

Unclear: using Telegram blue checkmarks for ordinary bot chats is not a viable implementation path. The maintainable path is a product/API design for channel-specific lifecycle states, with Telegram Business read support treated separately if desired.
