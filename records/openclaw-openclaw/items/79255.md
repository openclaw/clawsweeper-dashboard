## Summary

Close as implemented on current main: native Telegram read receipts remain unavailable for normal bot chats, but OpenClaw now has opt-in Telegram/Slack/Discord lifecycle status reactions that provide the requested user-visible processing cue; the implementation is current-main-only after v2026.5.7.

## What This Changes

Request a Telegram and cross-channel status cue that distinguishes gateway receipt, agent processing, and read/received state for slow or stuck turns.

## Best Possible Solution

Keep the current opt-in lifecycle status reactions as the channel-safe status cue and leave native read-receipt semantics out of normal Telegram bot chats.

## Reproduction Assessment

Not applicable: this is an enhancement request rather than a broken existing contract. Source, docs, upstream Bot API docs, and tests show the current-main status cue behavior and the native read-receipt limit.

## Solution Assessment

Yes: lifecycle reactions are a narrower maintainable solution than overloading Telegram blue checkmarks or adding a rejected SDK-wide providerAccepted/read surface.
