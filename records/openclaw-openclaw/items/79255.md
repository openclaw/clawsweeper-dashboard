## Summary

Keep open: this is a real feature request for user-visible message lifecycle state, and current main only has adjacent internal ack/progress/receipt surfaces while Telegram normal bot chats still cannot provide the requested native read-checkmark distinction.

## What This Changes

The issue asks OpenClaw to distinguish sent, gateway/agent received, and read states for Telegram and similar channels.

## Best Possible Solution

Define an explicit channel lifecycle/status model that separates provider acceptance, OpenClaw dispatch/durability, typing/progress, and platform read support per channel.

## Reproduction Assessment

Not applicable: this is an enhancement request, not a broken existing contract. Current source and docs show the available Telegram lifecycle surfaces and the platform read-receipt limitation, but there is no failing current-main behavior to reproduce.

## Solution Assessment

Unclear: using Telegram blue checkmarks is not a viable general solution for ordinary bot chats. The safer path is a maintainer-designed status model that can expose channel-specific capabilities without reusing internal ack policies as user-visible read receipts.
