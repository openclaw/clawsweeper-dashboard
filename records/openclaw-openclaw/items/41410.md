## Summary

Keep this PR open because current main still lacks Telegram forum-topic deletion, but the proposed patch is not merge-ready. The remaining blocker is concrete: the new `strictInteger` validation still accepts non-decimal strings through `Number(...)`, which can feed malformed IDs into destructive Telegram delete paths.

## What This Changes

The PR adds Telegram `topic-delete` and `deleteForumTopic` routing, a `deleteForumTopicTelegram` transport helper, a `topicId` tool-schema alias, docs, tests, and stricter numeric parsing for destructive IDs.

## Best Possible Solution

Keep the focused Telegram plugin feature, refresh it onto current main, and require decimal-only raw string validation for `strictInteger` plus targeted parser/action/runtime/transport/docs coverage before merge. Leave #11414 as the broader forum-topic lifecycle umbrella.

## Reproduction Assessment

Yes. The remaining issue can be reproduced without live Telegram by passing `0x10` or `1e2` through the PR-head `readNumberParam(..., { strict: true, strictInteger: true })` path; JavaScript converts those strings to finite integers before the new integer check.

## Solution Assessment

No. The feature direction is narrow and fits the Telegram plugin boundary, but the current patch is not the best mergeable solution until strict integer parsing rejects non-decimal raw strings and the branch is refreshed on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject non-decimal strict integer inputs:** `src/agents/tools/common.ts:154-157`
  - body: The new `strictInteger` mode validates only `Number.isInteger` after `Number(trimmed)`. Strings like `0x10` and `1e2` become integer IDs, so the destructive `delete` and `topic-delete` paths can accept malformed input instead of failing. Validate the raw string as a decimal integer before coercion when `strictInteger` is enabled.
  - confidence: 0.86
