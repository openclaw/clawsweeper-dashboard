## Summary

Keep open. The feature direction is useful, but this PR is not mergeable as-is: it is conflicting, still carries a stale core Telegram action runtime path, lacks live Telegram proof, and leaves a concrete destructive-ID parsing issue.

## What This Changes

The branch adds Telegram `topic-delete` and `deleteForumTopic` routing, a transport helper, a `topicId` schema alias, docs, and tests for deleting Telegram forum topics.

## Best Possible Solution

Port or replace the useful deletion behavior in the current Telegram plugin runtime, enforce decimal-only destructive IDs, align it with the canonical lifecycle tracker, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature PR. Current-main source inspection gives high-confidence proof that Telegram exposes topic create/edit only and has no `topic-delete` or `deleteForumTopic` runtime path.

## Solution Assessment

No. The feature direction is valid, but this branch needs to be ported to the current extension runtime, hardened for decimal-only destructive IDs, coordinated with https://github.com/openclaw/openclaw/issues/11414, and backed by live Telegram proof.
