## Summary

Keep open: the current head fixed the earlier guard-order and retry-boundary blockers, but one merge-blocking session-state cleanup bug remains. A quarantined session can keep stale retry/quarantine metadata after normal user progress, so a later unrelated restart can skip the intended fresh recovery budget.

## What This Changes

The PR adds persistent main-session restart-recovery attempt/quarantine fields, uses them to quarantine over-budget recovery rows, reserves the new session keys for plugin slots, and adds focused recovery tests.

## Best Possible Solution

Keep the durable main-session retry budget, but clear the retry/quarantine metadata when normal user-driven session progress makes the quarantine obsolete, with focused regression coverage before merge.

## Solution Assessment

No, not yet. The durable retry budget is the right layer, but the implementation must reset the new state when a user-driven turn makes the quarantine obsolete.
