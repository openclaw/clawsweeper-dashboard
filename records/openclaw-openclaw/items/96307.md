## Label Justifications

- `P1`: The PR addresses a real restart-recovery loop that can exhaust supervisor restarts and break active channel workflows.
- `merge-risk: 🚨 compatibility`: The diff adds new persistent SessionEntry fields and changes upgrade behavior for restart-aborted sessions without complete reset semantics.

## Summary

Keep open: this PR targets the right P1 recovery gap, but it is not merge-ready because the new persistent budget can quarantine recoverable sessions too early and across unrelated future incidents, and the contributor proof is unit-test-only.

## What This Changes

Adds persistent main-session restart-recovery attempt/quarantine fields, gates automatic recovery on that budget, reserves the new session-entry keys from plugin slots, and adds focused recovery tests.

## Best Possible Solution

Keep the durable main-session budget approach, but make the counter incident-scoped, reset it after successful recovery and fresh user-driven session initialization, align the threshold with the intended retry count, and add proof for that behavior before merge.

## Reproduction Assessment

Yes, at source level: seed a running main-session row with abortedLastRun true, make recovery fail across fresh startup contexts, and current main has no persisted ceiling. I did not run the full Telegram/systemd reproduction in this read-only review.

## Solution Assessment

No, not as submitted. A durable main-session budget is the right layer, but the PR needs incident-scoped reset semantics and threshold alignment before it is the best fix.
