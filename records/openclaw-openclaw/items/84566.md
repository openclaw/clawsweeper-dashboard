## Label Justifications

- `P1`: The PR touches credential persistence for model providers and can break existing provider auth while addressing a security-sensitive leak path.
- `merge-risk: 🚨 compatibility`: The diff changes the documented `models.json` merge precedence and can remove data existing users currently rely on after upgrade.
- `merge-risk: 🚨 auth-provider`: Deleting provider `apiKey` fields from existing `models.json` entries can make custom provider authentication fail at runtime.

## Summary

Keep this PR open: the security goal is valid and source-reproducible, but the current diff strips every provider-level `apiKey` after merge, including documented `models.json`-owned credentials that existing custom providers may rely on. It also still needs real behavior proof before merge.

## What This Changes

The PR adds a serialization-time sanitizer that removes provider-level `apiKey` fields from generated `models.json` output and updates models-config tests to expect omission for env, config, and existing-provider cases.

## Best Possible Solution

Keep the security fix, but narrow the sanitizer or migration so generated/source-managed secrets are not persisted while documented user-owned `models.json` credentials are either preserved or intentionally migrated with docs and upgrade proof.

## Reproduction Assessment

Yes. Source inspection shows current main serializes `finalProviders` into `models.json`, and the merge/docs path preserves non-SecretRef-managed `apiKey` values from existing agent catalogs.

## Solution Assessment

No. The serialization boundary is a good place to avoid generated secret persistence, but deleting every `apiKey` also removes documented user-owned credentials; the safer solution needs a narrower sanitizer or an explicit migration.
