## Summary

Keep this PR open. The Signal container-mode direction is still useful and current main does not implement it, but the latest branch still has a concrete timestamp contract bug and lacks inspectable real container proof.

## What This Changes

This PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, `channels.signal.apiMode`, docs, tests, a changelog fragment, config baseline updates, and a plugin-local `ws` dependency.

## Best Possible Solution

Update this branch to normalize bbernhard response timestamps to OpenClaw's numeric contract, clean up container stream abort listeners, keep docs/config aligned, and attach redacted live container proof before maintainer merge review.

## Reproduction Assessment

Yes, at a source-contract level. Current main only uses native Signal `/api/v1/rpc`, `/api/v1/check`, and `/api/v1/events`, while bbernhard documents `/v2/send` plus WebSocket `/v1/receive/<number>`; I did not run a live container in this read-only review.

## Solution Assessment

No, not yet. A plugin-local adapter is the right direction, but the branch should fix the timestamp contract issue, remove the reconnect listener leak, and include inspectable live container proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Normalize container send timestamps before returning them:** `extensions/signal/src/client-container.ts:358-365`
  - body: The bbernhard `/v2/send` response returns `timestamp` as a string, but this path types it as a number and passes it through to `SignalSendResult.timestamp` and the receipt raw timestamp. Convert the response value to a finite number, or intentionally widen the downstream contract, and update the test to mock the upstream string response.
  - confidence: 0.93
- **[P3] Remove abort listeners when container streams close:** `extensions/signal/src/client-container.ts:300-309`
  - body: `runSignalSseLoop` reconnects with the same abort signal after each WebSocket close, but this listener is only removed when abort fires. Repeated reconnects retain old WebSocket closures; have cleanup remove the listener on close/error/abort like the native SSE client does.
  - confidence: 0.82
