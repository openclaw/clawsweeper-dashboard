## Summary

Keep open. Current main is still native Signal JSON-RPC/SSE only, and this PR remains the active implementation path for bbernhard REST/WebSocket container support. The latest head is close, but still has a merge-blocking generated channel metadata drift: `channels.signal.apiMode` is added to types/schema/docs while the committed generated Signal channel metadata does not expose it.

## What This Changes

The PR adds a Signal native/container transport adapter, a bbernhard REST/WebSocket container client, Signal caller rewiring, `channels.signal.apiMode` config/docs/tests/changelog coverage, and a direct `ws` runtime dependency for the Signal plugin.

## Best Possible Solution

Land the plugin-local Signal adapter after regenerating all committed config metadata artifacts so `channels.signal.apiMode` is exposed consistently, then gate the branch with targeted Signal tests, config generation checks, changed-gate proof, and live container smoke or an explicit maintainer waiver.

## Reproduction Assessment

Yes. The original problem is source-verifiable on current main because Signal still calls only native `/api/v1/*` endpoints, while bbernhard container mode requires `/v2/send` and `/v1/receive/{number}`. The remaining PR blocker is statically reproducible on latest head: `apiMode` is in zod/types/docs but absent from the generated bundled Signal channel metadata.

## Solution Assessment

No. The adapter design is a plausible narrow solution, and the latest head appears to address prior transport/security findings, but it is not the best mergeable solution until generated config metadata is aligned and checked.
