## Summary

Keep open. Current main still lacks Signal container REST/WebSocket support, and this PR remains the active implementation candidate, but the latest head is dirty/unmergeable and still has blocking transport, config-boundary, schema, and raw Signal payload logging issues.

## What This Changes

The PR adds a Signal native/container adapter and bbernhard REST/WebSocket client, rewires Signal send/monitor paths, adds `channels.signal.apiMode` docs/types/tests/changelog, and adds a direct `ws` runtime dependency.

## Best Possible Solution

Ship a plugin-local Signal transport adapter that preserves native JSON-RPC/SSE behavior, adds documented bbernhard REST/WebSocket mode through accepted SDK runtime/config seams, aligns strict schema/generated metadata/docs, removes sensitive logging, and includes targeted tests plus live container proof or an explicit maintainer waiver.

## Reproduction Assessment

Yes. The original gap is reproducible from source because current main uses `/api/v1/rpc`, `/api/v1/check`, and `/api/v1/events` while the bbernhard container documents `/v2/send` and WebSocket `/v1/receive/<number>`; the PR regressions are source-level checks on the latest head.

## Solution Assessment

No. The adapter direction is plausible, but this branch is not the best mergeable form until the native SSE timeout forwarding, startup retry behavior, raw logging, SDK boundary imports, strict config schema, dirty merge state, and live-proof gaps are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Forward the native SSE timeout through the adapter:** `extensions/signal/src/client-adapter.ts:118-143`
  - body: Current main opens the long-lived native Signal SSE monitor with `timeoutMs: 0`. The adapter signature omits `timeoutMs` and the native call does not forward it, so native mode falls back to the default timeout and can reintroduce idle reconnect churn for healthy quiet streams.
  - confidence: 0.92
- **[P1] Keep startup Signal checks retryable:** `extensions/signal/src/client-adapter.ts:103-111`
  - body: `signalCheck()` now resolves auto mode before returning a `{ ok: false }` result. While an auto-started native daemon is still booting, both probes can fail and `detectSignalApiMode()` throws; `waitForTransportReady()` does not catch check callback exceptions, so startup can fail immediately instead of polling until the daemon is ready.
  - confidence: 0.9
- **[P1] Remove raw Signal payload logging:** `extensions/signal/src/client-container.ts:202`
  - body: `streamContainerEvents` logs the first 200 characters of every inbound WebSocket frame before parsing it. Those frames can contain private Signal message text plus sender, group, and attachment metadata, so normal container receive would write sensitive content to gateway logs.
  - confidence: 0.95
- **[P2] Use the accepted config seam:** `extensions/signal/src/client-adapter.ts:9`
  - body: This bundled plugin production file imports the broad `openclaw/plugin-sdk/config-runtime` barrel and calls ambient `loadConfig()`. Repo guards reject that broad subpath and ambient config access in bundled plugin production code; pass the loaded config or use the accepted runtime config seam instead.
  - confidence: 0.91
- **[P2] Import fetch helpers from the narrow SDK subpath:** `extensions/signal/src/client-container.ts:11`
  - body: The container client imports `resolveFetch` from the broad `openclaw/plugin-sdk/infra-runtime` barrel. The bundled-plugin import guard rejects that broad subpath, and existing plugins use `openclaw/plugin-sdk/fetch-runtime` for this helper.
  - confidence: 0.88
- **[P2] Add apiMode to the strict config schema:** `src/config/types.signal.ts:68`
  - body: `apiMode` is added to the public Signal config type and docs, but the strict `SignalConfigSchema` still does not accept this key. A user setting `channels.signal.apiMode: "container"` can fail config validation before the adapter sees it.
  - confidence: 0.92
