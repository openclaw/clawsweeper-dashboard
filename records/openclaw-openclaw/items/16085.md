## Summary

Keep open: current main still lacks Signal container REST/WebSocket support, and this PR remains the active implementation candidate, but the latest branch still has a narrow credential-leak regression before merge.

## What This Changes

This PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, `channels.signal.apiMode` config/docs/tests, a changelog fragment, config baseline updates, and a plugin-local `ws` dependency.

## Best Possible Solution

Ship the plugin-owned Signal adapter after the container client shares the native URL credential guard and the Signal plugin, config, docs, and live bbernhard smoke are green.

## Reproduction Assessment

Yes. Source inspection shows current main only supports native `/api/v1/*` Signal endpoints, and the PR’s credential leak is source-reproducible with a credentialed `httpUrl` in container mode.

## Solution Assessment

No, not yet. The adapter direction is maintainable and plugin-local, but the branch should reject credentialed Signal base URLs in container mode before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject credentialed Signal URLs in container mode:** `extensions/signal/src/client-container.ts:170-174`
  - body: The native Signal client rejects `httpUrl` values containing username/password, but the new container path accepts the same base URL and logs `redactedWsUrl` built from it. Parse the container base URL like the native client and add coverage for credential rejection/redaction.
