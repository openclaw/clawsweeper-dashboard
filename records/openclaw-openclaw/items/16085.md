## Summary

Keep open: current main still has only the native Signal JSON-RPC/SSE transport, and this PR remains the canonical container REST/WebSocket implementation candidate, but latest head still has a container group-typing bug and lacks inspectable live bbernhard container proof.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, `channels.signal.apiMode`, docs/tests/config baseline/changelog updates, and a plugin-local `ws` dependency.

## Best Possible Solution

Ship the plugin-local adapter after fixing the remaining branch issues, keeping docs/config/tests aligned, and attaching redacted live bbernhard container proof for the changed transport paths.

## Reproduction Assessment

Yes, at a source-contract level. Current main only uses Signal `/api/v1/rpc`, `/api/v1/check`, and `/api/v1/events`, while upstream bbernhard exposes `/v2/send` and WebSocket `/v1/receive/{number}` for the container path.

## Solution Assessment

No, not yet. A plugin-local adapter is the right direction, but the branch should format container group typing targets, clean up reconnect abort listeners, and provide redacted real container proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Format group targets for container typing:** `extensions/signal/src/client-container.ts:667-670`
  - body: `sendTypingSignal` can pass a group target through `p.groupId`, but container mode forwards that raw group id as `recipient`. bbernhard recognizes group recipients by the `group.` prefix and decodes them before calling signal-cli, so group typing indicators will fail in container mode until this fallback uses `formatGroupIdForContainer`.
  - confidence: 0.88
- **[P3] Remove abort listeners when streams close:** `extensions/signal/src/client-container.ts:361-370`
  - body: Each reconnect reuses the same abort signal, but normal WebSocket closes resolve without removing the abort listener added here. Reconnect churn will retain stale closures until abort; store the handler and remove it in the shared cleanup path.
  - confidence: 0.81
