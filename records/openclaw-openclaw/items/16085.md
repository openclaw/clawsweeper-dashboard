## Summary

Keep open. Current main still lacks bbernhard REST/WebSocket Signal support, and this PR remains the active implementation path, but the branch still has a credentialed URL logging defect and lacks inspectable real container proof.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, `channels.signal.apiMode`, docs, tests, a changelog fragment, config baseline updates, and a plugin-local `ws` dependency.

## Best Possible Solution

Ship the plugin-owned Signal adapter after the container client shares native URL credential validation, docs/config stay aligned, targeted tests pass, and redacted live bbernhard container proof is attached.

## Reproduction Assessment

Yes. Source inspection shows current main only uses native `/api/v1/*` Signal endpoints, and upstream bbernhard docs show the container uses `/v2/send` plus WebSocket `/v1/receive/<number>`; I did not run a live container in this read-only review.

## Solution Assessment

No, not yet. The adapter boundary is a maintainable plugin-local direction, but the branch should reject credentialed container base URLs and include inspectable live container proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject credentialed Signal container base URLs:** `extensions/signal/src/client-container.ts:170`
  - body: The native Signal client rejects `httpUrl` values containing username/password, but the new container path accepts them and logs `redactedWsUrl` from the full authority. With `http://user:pass@host`, the connection log still includes `user:pass@host`; parse the container base URL like the native client and add credential rejection/redaction coverage.
  - confidence: 0.91
