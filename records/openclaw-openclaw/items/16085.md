## Summary

Keep open: current main still lacks bbernhard REST/WebSocket Signal support, and this PR is the active implementation path, but it needs real behavior proof and maintainer review before merge.

## What This Changes

The PR adds a Signal native/container adapter, bbernhard REST/WebSocket client, `channels.signal.apiMode` config/schema/docs/tests, a changelog fragment, and a plugin-local `ws` dependency.

## Best Possible Solution

Ship a plugin-local Signal transport adapter that preserves native JSON-RPC/SSE behavior while adding documented bbernhard REST/WebSocket mode with schema/docs/tests and redacted live container proof.

## Reproduction Assessment

Yes, at source level: current main always uses `/api/v1/rpc`, `/api/v1/check`, and `/api/v1/events`, while upstream bbernhard exposes `/v2/send` and `/v1/receive/{number}` for this deployment mode. I did not run a live container smoke in this read-only review.

## Solution Assessment

Yes, directionally: keeping the transport split inside the Signal plugin adapter is the right owner boundary. Merge should still wait for redacted real behavior proof and maintainer review of the new transport/config/dependency surface.
