## Summary

Keep open. Current main still has only the native Signal JSON-RPC/SSE transport, so the container REST/WebSocket work is still relevant, but the latest PR head is not mergeable and introduces at least one concrete send-result regression plus a low-severity logging concern that should be fixed before merge.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket container client, Signal caller rewiring, `channels.signal.apiMode` config/docs/tests/changelog coverage, and a direct `ws` runtime dependency for the Signal plugin.

## Best Possible Solution

Land a plugin-local Signal transport adapter that preserves native JSON-RPC/SSE behavior, parses bbernhard REST responses correctly, avoids logging account identifiers in normal logs, keeps config/schema/docs aligned, and ships after rebase, targeted Signal tests, changed-gate proof, and live container smoke or maintainer waiver.

## Reproduction Assessment

Yes. The user problem is source-verifiable against current main and upstream bbernhard docs: OpenClaw uses native `/api/v1/*` endpoints while the container path needs `/v2/send` and WebSocket `/v1/receive/{number}`. The PR regression is also statically reproducible because 201 send responses are discarded before parsing.

## Solution Assessment

No. The adapter direction is plausible and most prior findings appear addressed, but the branch is not yet the best mergeable form until it preserves the `/v2/send` timestamp response, redacts the WebSocket account URL log, resolves mergeability, and has maintainer-visible validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse the send Created response:** `extensions/signal/src/client-container.ts:118-120`
  - body: The bbernhard `/v2/send` contract returns `201 Created` with a JSON `timestamp`, but `containerRestRequest` returns `undefined` for every 201 before reading the body. Container sends will therefore make `sendMessageSignal` report `messageId: "unknown"` even when the API returned the real timestamp; only no-content statuses should bypass parsing.
  - confidence: 0.88
- **[P3] Redact the WebSocket receive URL:** `extensions/signal/src/client-container.ts:173`
  - body: This logs the full receive URL in normal gateway logs, including the Signal account number embedded in `/v1/receive/{account}`. Please log a redacted endpoint or move the full URL behind verbose/debug logging so normal container startup does not expose account identifiers.
  - confidence: 0.82
