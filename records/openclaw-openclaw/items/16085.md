## Summary

Keep open. Current main still lacks bbernhard REST/WebSocket Signal support, and related issues were centralized on this PR, but the branch still needs a credential-logging fix plus inspectable live container proof before merge.

## What This Changes

The branch adds a Signal adapter and bbernhard REST/WebSocket container client, `channels.signal.apiMode`, docs/tests/config baselines, a changelog fragment, and a plugin-local `ws` dependency.

## Best Possible Solution

Ship a plugin-local Signal transport adapter that preserves native JSON-RPC/SSE behavior while adding bbernhard REST/WebSocket mode, sanitized URL handling, aligned config/docs/tests, and redacted live container proof.

## Reproduction Assessment

Yes. Source inspection shows current main only uses `/api/v1/rpc`, `/api/v1/check`, and `/api/v1/events`, while upstream bbernhard docs use `/v2/send` and WebSocket `/v1/receive/<number>`; I did not run a live container in this read-only review.

## Solution Assessment

No, not ready. The plugin-local adapter is the right boundary, but this branch should reject or sanitize credentialed container URLs, include inspectable redacted live container proof, and then get maintainer review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject credentialed Signal container URLs:** `extensions/signal/src/client-container.ts:170`
  - body: The native Signal client rejects `httpUrl` values with usernames/passwords, but the new container path accepts them and builds the logged receive endpoint from the full URL authority. A config such as `http://user:pass@host` can put credentials in runtime logs; parse and reject credentials or build logs from a sanitized URL.
  - confidence: 0.91
