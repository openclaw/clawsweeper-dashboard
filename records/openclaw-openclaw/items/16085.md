## Summary

Keep this PR open: current main still lacks Signal container REST/WebSocket support, and the PR is the active implementation candidate, but the proposed container client introduces a concrete credential-redaction regression that should block merge.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard signal-cli-rest-api REST/WebSocket client, `channels.signal.apiMode` config/docs/tests, a changelog fragment, and a plugin-local `ws` dependency.

## Best Possible Solution

Ship the plugin-owned adapter after making the container client share the native Signal URL credential guard, preserving native JSON-RPC/SSE behavior and validating the Signal plugin plus live bbernhard container smoke.

## Reproduction Assessment

Yes. Source inspection shows current main only uses native `/api/v1/*` Signal endpoints, so a bbernhard container exposing `/v2/send` and `/v1/receive/{account}` is unsupported; the PR’s credential leak is also source-reproducible with a credentialed `httpUrl` in container mode.

## Solution Assessment

No, not yet. The adapter direction is maintainable and plugin-local, but the branch should first reject credentialed Signal base URLs in container mode and keep credentials out of logs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject credentialed Signal URLs in container mode:** `extensions/signal/src/client-container.ts:153`
  - body: The native Signal client rejects `httpUrl` values that include username/password, but the new container path accepts the same value and builds `redactedWsUrl` from it. For `http://user:pass@host`, the connection log at this line still includes `user:pass@host`, so container mode can leak credentials and weakens an existing guard. Parse the base URL like the native client and reject credentials before constructing REST/WebSocket URLs.
  - confidence: 0.9
