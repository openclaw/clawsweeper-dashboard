## Summary

Keep open. Current main is still native Signal JSON-RPC/SSE only, and this PR addresses a real bbernhard container endpoint mismatch, but the branch still has a narrow container group-typing bug and the new transport/config/dependency behavior needs maintainer review before merge.

## What This Changes

The PR adds a Signal adapter and bbernhard REST/WebSocket container client, switches Signal runtime callers to the adapter, adds global `channels.signal.apiMode` config/docs/tests, updates config baselines, adds a changelog fragment, and declares a Signal plugin `ws` dependency.

## Best Possible Solution

Ship the plugin-local Signal transport adapter after fixing group typing formatting, confirming the global `apiMode` scope, and validating the documented bbernhard send/receive/attachment/group/reaction/receipt paths.

## Reproduction Assessment

Yes. Source inspection shows current main only calls native `/api/v1/rpc` and `/api/v1/events`, while upstream bbernhard docs expose `/v2/send` and WebSocket `/v1/receive/{number}`; I did not run a live container in this read-only pass.

## Solution Assessment

Mostly yes. A plugin-local adapter is the right boundary for preserving native Signal behavior while adding bbernhard mode, but the group typing formatting bug should be fixed and maintainers should accept the channel-global mode scope before merge.
