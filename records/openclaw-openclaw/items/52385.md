## Summary

Current main still does not satisfy the requested internal-only behavior. The first target-agent turn is routed with `deliver:false` on the internal channel, but the follow-up A2A announce step can still call gateway `send` for the resolved channel/thread whenever the announce reply is non-empty and not `ANNOUNCE_SKIP`; existing tests continue to assert that visible delivery path.

## What This Changes

Request that inter-agent `sessions_send` traffic reach the target agent session without rendering visible Slack thread replies or top-level channel messages.

## Best Possible Solution

Enforce an internal-only routing contract for inter-agent `sessions_send`: preserve the target session transcript and any returned tool result, but prevent the A2A announce path from invoking user-facing channel `send` for Slack-like channel-bound and thread-bound targets. If maintainers prefer opt-in behavior, add a typed silent/internal-only flag and update protocol docs, config docs, and regressions around that contract.

## Reproduction Assessment

Yes. Current tests and source provide a high-confidence reproduction path: make the announce step return any text other than `ANNOUNCE_SKIP`, and `runSessionsSendA2AFlow` calls gateway `send` with the resolved channel/thread target.

## Solution Assessment

No. The current solution relies on model compliance with an `ANNOUNCE_SKIP` prompt and still has tests proving visible delivery; the safer fix is an enforced routing/API contract that cannot leak internal coordination text to Slack.
