## Summary

Keep open: current main still models the affected phone and sender identifier fields as plain strings, numbers, or E.164 strings, and PR #73822 is the open unmerged implementation lane.

## What This Changes

Allow channel phone-number and equivalent sender identifier fields to accept SecretRef objects while preserving existing plain string configs.

## Best Possible Solution

Land or replace PR #73822 with a backward-compatible channel/plugin-owned SecretRef implementation for the affected identifier fields, including runtime resolution, metadata/docs, redaction, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured SecretRef object in channels.whatsapp.allowFrom or plugins.entries.voice-call.config.fromNumber; current main schemas accept only strings, numbers, or E.164 strings there.

## Solution Assessment

Yes. The narrow maintainable path is additive support in the owning channel/plugin schemas, types, metadata, docs, and runtime resolution path rather than a startup migration or core-only bypass.
