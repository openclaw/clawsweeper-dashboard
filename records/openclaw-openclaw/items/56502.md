## Summary

Keep this open: current main and the latest release still keep the reported phone-number and sender-identifier fields as plain strings/E.164 strings, and https://github.com/openclaw/openclaw/pull/73822 is the open implementation lane that closes this issue.

## What This Changes

Allow channel phone-number and sender-identifier config fields to accept SecretRef values while preserving existing plaintext string configs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73822, or an equivalent replacement, with additive SecretRef support for the affected identifier fields and the existing public-open access-control semantics preserved.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a structured SecretRef object in an affected field such as channels.whatsapp.allowFrom or plugins.entries.voice-call.config.fromNumber; current main accepts only string/number/E.164 shapes there.

## Solution Assessment

Yes in direction. The maintainable fix is channel/plugin-owned SecretRef support through schemas, contracts, runtime resolution, metadata, and docs while keeping plain strings backward-compatible and preserving the literal wildcard guard for public-open DM policy.
