## Summary

Keep open: current main still rejects SecretRef objects for the reported phone-number and sender-identifier fields, and https://github.com/openclaw/openclaw/pull/73822 is the open implementation lane with closing syntax for this issue.

## What This Changes

Allow channel phone-number and sender-identifier config fields to accept SecretRef values while preserving existing plaintext string configs.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73822 or an equivalent replacement that adds backward-compatible channel/plugin-owned SecretRef support while preserving visible access-control opt-ins.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place a structured SecretRef object in channels.whatsapp.allowFrom or plugins.entries.voice-call.config.fromNumber; current schemas accept only string/number/E.164 shapes there.

## Solution Assessment

Yes in direction. The maintainable solution is additive SecretRef support in the owning channel/plugin schemas, contracts, runtime resolution, metadata, docs, and tests, with the open-policy wildcard guard preserved.
