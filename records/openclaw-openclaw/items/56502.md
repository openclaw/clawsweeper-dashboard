## Summary

Keep this open: current main still does not accept SecretRef objects for the reported phone-number and sender-identifier channel fields, and PR #73822 is the open, unmerged implementation lane.

## What This Changes

Allow phone-number and sender-identifier channel config fields to accept SecretRef objects while preserving existing plaintext string configs.

## Best Possible Solution

Keep this issue open until PR #73822 or an equivalent replacement lands a backward-compatible channel/plugin-owned SecretRef implementation for the affected identifier fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a structured SecretRef object in an affected field such as channels.whatsapp.allowFrom or plugins.entries.voice-call.config.fromNumber; the current schemas accept only strings, numbers, or E.164 strings there.

## Solution Assessment

Yes. The narrow maintainable direction is additive support in the owning channel/plugin schemas and contracts, with SecretRefs resolved before allowlist/outbound/runtime checks and reflected in generated metadata, docs, and tests.
