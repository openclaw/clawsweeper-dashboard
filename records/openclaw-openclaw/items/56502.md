## Summary

Keep this open: current main and the latest release still model the reported phone-number and sender-identifier fields as plain strings/E.164 strings, and https://github.com/openclaw/openclaw/pull/73822 is the open unmerged implementation lane.

## What This Changes

Allow phone-number and sender-identifier channel config fields to accept SecretRef values while preserving existing plain-string configs.

## Best Possible Solution

Keep this open until https://github.com/openclaw/openclaw/pull/73822 or an equivalent replacement lands backward-compatible SecretRef support for the affected channel/plugin identifier fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a structured SecretRef object in an affected field such as channels.whatsapp.allowFrom or plugins.entries.voice-call.config.fromNumber; current main accepts only string/number/E.164 shapes there.

## Solution Assessment

Yes in direction. The maintainable solution is additive channel/plugin-owned SecretRef support, but the active PR must preserve existing access-control semantics such as requiring a visible literal "*" for open policy.
