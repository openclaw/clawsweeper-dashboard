## Summary

Keep this issue open. Current main still requires the reported phone-number and sender-identifier channel fields to be plain strings or string/number arrays, the canonical SecretRef target surface does not include those paths, and PR #73822 is open and unmerged as the active implementation lane.

## What This Changes

Allow phone-number and equivalent channel identifier fields to accept SecretRef values while preserving backward-compatible plain string configs.

## Best Possible Solution

Keep this issue open while maintainers review PR #73822 or an equivalent replacement. The desired end state is a backward-compatible, channel/plugin-owned implementation that accepts plain strings and SecretRef objects for phone-number and sender-identity fields, resolves refs before allowlist/outbound checks, and updates config metadata, docs, status/redaction behavior, and tests without doing a startup migration.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a SecretRef object in an affected field such as channels.whatsapp.allowFrom or plugins.entries.voice-call.config.fromNumber; current schemas accept only strings, numbers, or E.164 strings there, so the config cannot parse as requested.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: extend the owning channel/plugin schemas and contracts additively, then resolve SecretRefs through the existing secrets pipeline before consumers read those values. Closing this issue before the linked implementation lands would be premature.
