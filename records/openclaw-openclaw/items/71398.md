## Summary

Keep open. Current main still leaves the reported QQ Bot SecretRef failure path actionable: literal `secretref:/...` values are accepted as ordinary strings, QQ Bot still lacks a `secret-contract-api.ts` runtime materialization contract, the canonical SecretRef surface still omits QQ Bot client secrets, and PR #74772 is open rather than merged.

## What This Changes

The issue requests fixing QQ Bot `clientSecret` handling so SecretRef-backed secrets, especially `secretref:/QQBOT_...` after upgrade, resolve before QQ token requests.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and an explicit policy for raw SecretRef-looking marker strings.

Yes. A high-confidence source-level reproduction is to configure QQ Bot `clientSecret` as `secretref:/QQBOT_...`; current main accepts it as a string, returns it as an available secret, and can pass it to the QQ token endpoint.
