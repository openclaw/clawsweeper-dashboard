## Summary

Keep open. Current main still leaves the reported QQ Bot SecretRef path actionable: literal `secretref:/...` strings are accepted as plaintext, QQ Bot still lacks a bundled secret-contract surface, and the active fix path is the still-open PR #74772 rather than shipped main code.

## What This Changes

Fix QQ Bot credential handling so SecretRef-backed `channels.qqbot.clientSecret` and account-level `clientSecret` values are resolved or rejected clearly instead of being sent to QQ as raw marker strings.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and explicit handling for raw SecretRef-looking marker strings.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure QQ Bot `clientSecret` as `secretref:/QQBOT_...`; current main treats it as an available string and can pass it to the QQ token endpoint.

## Solution Assessment

Yes. The narrowest maintainable solution is in the QQ Bot plugin plus the existing SecretRef contract/policy surface, not a broad core rewrite.
