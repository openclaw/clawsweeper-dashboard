## Summary

Keep open. Current main still leaves the reported QQ Bot SecretRef path actionable: `secretref:/...` is accepted as an ordinary non-empty string, QQ Bot has no bundled channel secret-contract artifact for runtime materialization, and the related implementation PR #74772 is still the appropriate path to review or land.

## What This Changes

The issue requests fixing QQ Bot `clientSecret` SecretRef handling after v2026.4.23 so `secretref:/...` values backed by `secrets.json` are resolved instead of being sent to the QQ OpenAPI as credentials.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and a deliberate policy for raw SecretRef-looking marker strings.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` as `secretref:/QQBOT_...`; current main accepts it as a non-empty string and can post that value as the QQ `clientSecret`.

## Solution Assessment

Yes. Keeping this issue open is the best current action because the missing behavior belongs in the QQ Bot plugin plus the shared SecretRef credential surface, and the related PR has not merged yet.
