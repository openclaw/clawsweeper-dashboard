## Summary

Keep open. Current `main` still leaves the reported QQ Bot SecretRef path actionable: a raw `secretref:/...` value is accepted as a normal string, QQ Bot has no secret-contract artifact or canonical credential-surface entry, and the linked fix PR is still open rather than merged.

## What This Changes

The issue reports that QQ Bot `clientSecret` values using `secretref:/...` stopped resolving after upgrade and are sent to QQ as raw credentials, causing `invalid appid or secret`.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and explicit handling for raw SecretRef-looking marker strings.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure QQ Bot `clientSecret` as `secretref:/QQBOT_...`; current `main` accepts it as a string, returns it as an available secret, and can pass it to the QQ token endpoint.

## Solution Assessment

Yes. Keeping this issue open while reviewing or landing #74772 is the narrowest maintainable path because the remaining work belongs in the QQ Bot plugin plus shared SecretRef policy coverage, not a broad core rewrite.
