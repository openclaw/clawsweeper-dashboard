## Summary

Keep open. Current `main` still contains the reported QQ Bot SecretRef failure mode: raw `secretref:/...` strings are accepted as plaintext, QQ Bot does not expose a secret-contract entry on `main`, and the exact remediation remains in open PR #74772 rather than shipped code.

## What This Changes

The issue asks for QQ Bot `clientSecret` values configured as `secretref:/...` to resolve correctly after the v2026.4.23 upgrade instead of being sent to QQ as an invalid secret.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and explicit handling for raw SecretRef-looking marker strings.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure QQ Bot `clientSecret` as `secretref:/QQBOT_...`; current `main` accepts it as an available string and can pass it to the QQ token endpoint.

## Solution Assessment

Yes. The narrowest maintainable solution is in the QQ Bot plugin plus the existing SecretRef contract and policy surface, not a broad core rewrite.
