## Summary

Keep open. Current main still leaves the reported QQ Bot SecretRef failure path actionable: literal `secretref:/...` values are accepted as ordinary strings, QQ Bot still has no channel secret-contract artifact or credential-surface listing, and the related fix PR #74772 is open rather than merged.

## What This Changes

The issue requests fixing QQ Bot `clientSecret` SecretRef resolution after upgrades so `secretref:/...` or supported SecretRef inputs are not sent raw to QQ OpenAPI.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and an explicit policy for raw SecretRef-looking marker strings.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` as `secretref:/QQBOT_...`; current main accepts it as a non-empty string and can post it as QQ `clientSecret`.

## Solution Assessment

Yes. Keeping this issue open is the best current action because the narrow implementation belongs in the QQ Bot plugin plus shared SecretRef policy coverage, and the linked PR has not merged yet.
