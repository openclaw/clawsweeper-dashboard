## Summary

Close as implemented on main: PR #74772 and follow-up hardening are merged, current QQ Bot code resolves structured `clientSecret` SecretRefs and rejects raw legacy marker strings before QQ token exchange; no release tag contains the proof commit yet.

## What This Changes

The report asks QQ Bot to stop sending a raw `secretref:/...` clientSecret to QQ OpenAPI after upgrade and support the configured secret reference path.

## Best Possible Solution

Keep the QQ Bot-owned SecretRef contract and legacy-marker rejection on main, ship it in the next release, and have users migrate `secretref:/...` values to structured SecretRef objects.

## Reproduction Assessment

No for current main. The historical failure path is clear, but current source and regression tests show `secretref:/...` is rejected before QQ token exchange and structured env SecretRefs resolve through QQ Bot.

## Solution Assessment

Yes. The merged QQ Bot-owned fix is the narrow maintainable route: plugin-local SecretRef contract/materialization plus explicit legacy marker rejection, not a broad core secret parser change.
