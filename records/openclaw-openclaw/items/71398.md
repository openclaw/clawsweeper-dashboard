## Summary

Keep open. Current main still leaves the reported QQ Bot `secretref:/...` failure mode unresolved: raw marker strings are accepted as normal strings, QQ Bot lacks secret-contract registry coverage, and the relevant fix remains an open PR rather than shipped code.

## What This Changes

The issue asks for QQ Bot `clientSecret` SecretRef handling so `secretref:/...` values do not reach the QQ token endpoint as the client secret.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and explicit legacy marker handling.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` as `secretref:/QQBOT_...`; current main accepts it as a non-empty string and can pass it into the QQ token request.

## Solution Assessment

Yes. The narrowest maintainable solution is in the QQ Bot plugin plus the existing SecretRef contract and policy surface, with centralized legacy-marker validation only if maintainers choose to reject that syntax.
