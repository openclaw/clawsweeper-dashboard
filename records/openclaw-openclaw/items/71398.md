## Summary

Keep open. Current `main` still leaves the reported QQ Bot SecretRef path actionable: `secretref:/...` is accepted as a normal non-empty string, QQ Bot has no bundled channel secret-contract artifact for runtime materialization, and the related fix PR #74772 is open rather than merged.

## What This Changes

The issue requests that QQ Bot resolve `clientSecret` SecretRefs from config instead of passing a literal `secretref:/...` value to QQ OpenAPI after upgrade.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and an explicit policy for raw SecretRef-looking marker strings.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` as `secretref:/QQBOT_...`; current `main` accepts it as a non-empty string and can post that literal value as QQ `clientSecret`.

## Solution Assessment

Yes. Keeping this open is the best current action because the narrow implementation belongs in the QQ Bot plugin plus shared SecretRef policy coverage, and the linked fix PR has not merged yet.
