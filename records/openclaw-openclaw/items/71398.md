## Summary

Keep open. Current main still leaves the reported QQ Bot SecretRef failure actionable, and the active fix path is the still-open PR #74772 rather than shipped code on main.

## What This Changes

The issue asks for QQ Bot `clientSecret` SecretRef resolution to work after the v2026.4.23 upgrade, including the reported legacy `secretref:/...` value stored through `secrets.json`.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and explicit handling for raw SecretRef-looking marker strings.

## Reproduction Assessment

Yes. A source-level reproduction is to configure `channels.qqbot.clientSecret` as `secretref:/QQBOT_...`; current main treats that non-empty string as an available plaintext secret and can pass it to the QQ token endpoint.

## Solution Assessment

Yes. The narrowest maintainable solution belongs in the QQ Bot plugin plus the existing SecretRef contract/policy surface, with only a small policy decision for the legacy marker format.
