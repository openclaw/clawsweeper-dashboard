## Summary

Keep open. Current main still contains the reported QQ Bot SecretRef failure mode, and the exact QQ Bot contract/runtime/docs fix remains in open PR #74772 rather than shipped main.

## What This Changes

QQ Bot should resolve or reject `secretref:/...` client secrets instead of passing the raw marker to the QQ OpenAPI after upgrade.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds secret-contract coverage, runtime materialization, credential-surface docs, regression tests, and explicit handling for raw SecretRef-looking marker strings.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` as `secretref:/QQBOT_...`; current main accepts it as a non-empty string and can pass it to the token endpoint.

## Solution Assessment

Yes. The narrowest maintainable direction is in the QQ Bot plugin plus existing SecretRef contract and policy surfaces, with a small compatibility decision for the literal marker syntax.
