## Summary

Keep open. Current main still accepts raw `secretref:/...` as a plaintext QQ Bot `clientSecret`, QQ Bot still lacks a secret-contract entry, and the related fix remains in open PR #74772 rather than shipped code.

## What This Changes

The issue requests QQ Bot to resolve or reject `secretref:/...` clientSecret values instead of passing them to QQ OpenAPI.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix, with explicit tests and validation/doctor guidance for raw `secretref:/...` markers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to set `channels.qqbot.clientSecret` to `secretref:/QQBOT_...`; current main treats it as an available string and can pass it to the QQ token request.

## Solution Assessment

Unclear. Adding QQ Bot SecretRef contract/runtime coverage is the right narrow direction, but maintainers still need to decide whether the raw marker should be migrated, resolved, or rejected with actionable diagnostics.
