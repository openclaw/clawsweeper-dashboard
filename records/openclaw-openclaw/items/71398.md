## Summary

Keep open. Current main still has the QQ Bot SecretRef gap: `secretref:/...` is accepted as a normal non-empty string, QQ Bot has no channel secret-contract/runtime materialization surface, and open PR #74772 is the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue asks for QQ Bot `clientSecret` SecretRef values to resolve after upgrade instead of sending a raw `secretref:/...` marker to the QQ API.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract coverage, runtime materialization, credential-surface docs, and explicit validation/doctor policy so SecretRef-looking marker strings cannot be sent to QQ as credentials.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` as `secretref:/QQBOT_...`; current main treats that as an available plaintext string and can post it as QQ `clientSecret`.

## Solution Assessment

Yes. Keeping this open while reviewing PR #74772 is the narrow maintainable path because the missing behavior belongs in the QQ Bot plugin's secret contract/runtime resolution plus shared marker policy, not in a broad unrelated core change.
