## Summary

Keep open. Current main and the latest release still persist OpenRouter streaming usage through the shared OpenAI-compatible parser and local cost calculation, with no OpenRouter `/generation` reconciliation for authoritative billed `total_cost`; the adjacent tiered-pricing and usage-aggregation work does not fix this transcript write path.

## What This Changes

The issue reports that OpenRouter streaming usage costs persisted in assistant transcripts can undercount billed cost for tier-priced models because OpenRouter’s authoritative generation record reports a higher total cost.

## Best Possible Solution

Add OpenRouter-scoped post-stream billed-cost reconciliation using the response id and a real OpenRouter credential, validate numeric `total_cost`, persist it as the authoritative `usage.cost.total`, and preserve the existing estimate when reconciliation is unavailable.

## Reproduction Assessment

Yes, at source level: a final OpenRouter-style stream chunk with a response id is parsed into locally calculated `usage.cost.total`, and current main has no `/generation` lookup before the assistant record is finalized. I did not run a live billed OpenRouter request in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is provider-owned OpenRouter reconciliation after streaming, with validation and fallback, not a dashboard aggregation change or unrelated `agent.wait` metadata work.
