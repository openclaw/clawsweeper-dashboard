## Summary

Keep open: current main still has only subsystem-specific SSRF opt-ins, no gateway-level SSRF policy, and the reported MS Teams lookup plus OpenAI OAuth refresh paths still reach guarded network calls without a configurable inherited policy. The request is valid, but it needs maintainer security/product direction because a global SSRF default would change a cross-cutting security boundary.

## What This Changes

The issue requests a gateway-wide SSRF policy such as `gateway.ssrfPolicy` with per-subsystem overrides, plus at least MS Teams parity for fake-IP or private-network opt-in.

## Best Possible Solution

Define a default-off global SSRF policy contract, decide which subsystems inherit it and how local overrides compose, then wire MS Teams Graph/Bot Framework and provider auth refresh through that shared policy with focused tests and docs while preserving strict defaults.

## Reproduction Assessment

Yes at source level: the strict gateway schema has no `ssrfPolicy`, and MS Teams Graph plus OpenAI OAuth refresh call `fetchWithSsrFGuard` without a configurable policy. No live fake-IP proxy reproduction was run for this review.

## Solution Assessment

Unclear for immediate implementation: subsystem parity is plausible, but a gateway-wide inherited SSRF policy is a security/product boundary and needs maintainer-approved semantics before code generation.
