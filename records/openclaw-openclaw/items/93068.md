## Summary

Keep open: current main has per-surface SSRF policy plumbing, but no gateway-level default, and source inspection confirms plausible uncovered Teams and OpenAI OAuth refresh paths. The requested global policy is security-sensitive config/product work, not a safe cleanup close or autonomous fix job.

## What This Changes

Support a gateway-level SSRF policy so subsystems can inherit allow-list behavior for fake-IP/private-network environments while preserving per-subsystem overrides.

## Best Possible Solution

Design an additive, default-deny gateway SSRF policy or approved equivalent that only applies to guarded outbound paths, preserves subsystem overrides, documents the trust boundary, and adds focused tests for web fetch, provider transport/auth refresh, and affected channel paths.

## Reproduction Assessment

Partly: source inspection gives a high-confidence reproduction of the missing configuration surface and uncovered guarded-fetch paths, but I did not run the reported fake-IP DNS environment live. The runtime symptom is therefore source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear as an implementation path: a gateway-level policy is plausible and aligned with the report, but maintainers still need to decide whether the durable fix is global inheritance, narrower per-subsystem knobs, or a shared policy resolver with explicit security semantics.
