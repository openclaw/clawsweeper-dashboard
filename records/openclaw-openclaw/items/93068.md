## Summary

Keep open: current main and v2026.6.8 still lack a gateway-level SSRF policy, MS Teams has no user-facing network SSRF config, and provider OAuth refresh can still use guarded fetches without an inherited policy. The request is valid, but it is security-sensitive feature/config work that needs maintainer product review.

## What This Changes

The issue requests a gateway-level SSRF policy default, with subsystem overrides, so fake-IP/private-network deployments can opt in once instead of configuring each transport separately.

## Best Possible Solution

Define a maintainer-approved, default-off SSRF policy model that composes global defaults with subsystem overrides, covers Teams/provider auth paths intentionally, preserves host-scoped fake-IP safeguards, and lands with docs plus focused security tests.

## Reproduction Assessment

Source-reproducible, but not live reproduced: strict schemas omit the requested global and Teams keys, and current guarded fetch call sites do not receive a global policy. The reported fake-IP DNS failure still needs a live Teams/OpenAI auth setup to mark fully reproduced.

## Solution Assessment

Unclear: a global default may be the right owner for repeated fake-IP policy gaps, but maintainers need to decide whether broad inheritance is safer than narrower per-surface fixes such as channels.msteams.network plus provider-auth policy hooks.
