## Summary

Keep open: current main and v2026.6.8 still have per-surface SSRF policy knobs, no gateway-level SSRF default, and MS Teams/auth paths that use guarded fetch without a user-facing policy override, so this needs maintainer security/product direction rather than cleanup closure.

## What This Changes

Request a default-off gateway-level SSRF policy, or at least MS Teams network-policy parity, so fake-IP/private-range environments can opt in while preserving per-subsystem overrides.

## Best Possible Solution

Define a maintainer-approved, default-off SSRF policy contract that either composes a gateway default into audited per-subsystem policies or intentionally chooses targeted MS Teams/auth/provider parity instead, with secure defaults, docs, schema tests, and fake-IP regression coverage.

## Reproduction Assessment

Yes at source level: current main has no gateway.ssrfPolicy, MS Teams has no network policy field, and several MS Teams guarded fetches run without policy. I did not run a live fake-IP DNS/MS Teams/OAuth environment, so this is not marked fully reproduced.

## Solution Assessment

Unclear until maintainers choose the policy boundary. A default-off inherited policy could be the right direction, but an unconditional global bypass would be too broad for OpenClaw's SSRF security model.
