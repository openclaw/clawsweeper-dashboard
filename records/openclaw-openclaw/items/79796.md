## Summary

Keep open: current main still has a status/health gap where a failing iMessage `imsg rpc` readiness probe can be hidden and the channel can fall back to `configured`/OK during startup or restart grace.

## What This Changes

The issue asks for iMessage channel status to report degraded/error when the underlying `imsg rpc` process cannot start because macOS Full Disk Access is missing.

## Best Possible Solution

Carry a sanitized `imsg rpc` probe/startup failure into both `openclaw status --deep` Channels and Health output so iMessage cannot render OK while receive/send handling is unusable.

## Reproduction Assessment

Yes, source-reproducible: deny `imsg` access to `~/Library/Messages/chat.db` so `probeIMessage` returns `ok:false` while the account is in startup/restart grace; current health/status can hide that probe and fall back to `configured`. I did not run a live macOS FDA repro in this read-only Linux checkout.

## Solution Assessment

No, the current behavior is not the best solution because it clears or hides the only failure signal before status rendering. The narrower maintainable fix is to preserve a sanitized failed-probe/startup state through the existing iMessage status and generic health formatting paths.
