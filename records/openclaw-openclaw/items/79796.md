## Summary

This should stay open: current `main` can run an iMessage probe that returns `ok: false`, then hide that failed probe from non-admin health/status output and still render the configured iMessage account as OK/configured.

## What This Changes

Report that iMessage status and health output can show OK/configured while `imsg rpc` is failing because macOS Full Disk Access blocks `chat.db`.

## Best Possible Solution

Make the status/health path treat failed iMessage probes or RPC startup errors as degraded, with a sanitized Full Disk Access remediation, and add regression coverage for `openclaw status --deep` and health formatting.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: iMessage probing returns `ok: false` on RPC failure, but normal health/status output drops that probe before formatting and falls back to `configured`. I did not run the macOS FDA denial live in this environment.

## Solution Assessment

Yes, a narrow fix in the existing iMessage/status health path is the best direction; it should surface sanitized probe/runtime failures rather than add a new mode or config switch.
