## Summary

Keep open. Current main still has QR/setup-code and direct mobile pairing paths, but not the requested `openclaw invite` command, short cloud-backed invite codes, invite register/resolve API, or `cloud` gateway config; the related cloud relay request remains open design context, not a merged implementation.

## What This Changes

The issue requests an `openclaw invite` command that registers short cloud-backed invite codes for mobile Gateway pairing.

## Best Possible Solution

Design this with the open cloud relay request, then either implement a coherent first-party invite/relay flow that preserves bootstrap-token and device-approval guarantees or explicitly document setup codes as the supported non-QR path.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken existing behavior. Source, docs, mobile app, and live related-issue inspection give a high-confidence verification path that the requested command, config, and invite register/resolve flow are absent on current main.

## Solution Assessment

Unclear; the proposed direction is plausible, but it is not yet proven to be the safest or narrowest maintainable solution. Maintainers should decide the invite-code and cloud-relay trust model together with https://github.com/openclaw/openclaw/issues/55915 before implementation.
