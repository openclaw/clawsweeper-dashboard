## Summary

Keep open. Current `main` still has QR/setup-code mobile pairing but not the requested `openclaw invite` command, short `CLAW-xxxx` discovery codes, cloud invite register/resolve endpoints, `gateway.cloud` config, or mobile invite resolver; the related cloud-relay design item is also still open.

## What This Changes

Add an `openclaw invite` command that generates short shareable invite codes, registers them with a configurable cloud backend, and lets mobile clients resolve the code before using the existing device approval flow.

## Best Possible Solution

Keep this as a maintainer product/security design item alongside #55915. The end state should either add a documented invite-code flow that reuses the existing bootstrap-token and `openclaw devices approve` contracts, or explicitly document that `openclaw qr --setup-code-only` is the supported non-QR pairing path and decline the cloud invite scope.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence verification path: current `main` can be inspected for the CLI catalog, registration table, setup-code resolver, gateway config type, docs, and mobile UI, and those surfaces show setup-code pairing but no invite-code implementation.

## Solution Assessment

Unclear. The requested user problem is real, but the proposed cloud registration/resolution path is not yet proven to be the narrowest maintainable solution because it depends on the broader cloud relay and security model in #55915.
