## Summary

Keep open. Current main does not fully supersede this PR because the existing visible New session control still dispatches the older `/new` reset path, while this PR adds a separate named `sessions.create` workflow. The branch is useful but should not merge as-is because the new agent resolver can trust stale session-list state during reconnect and create the session under the wrong agent namespace.

## What This Changes

The PR adds a Control UI chat-header plus button, a named New Session modal, dashboard session-key helpers/tests, and a `sessions.create` RPC flow that labels and selects the new session.

## Best Possible Solution

Land a direct named session-creation UI only after the agent id is derived from fresh current-connection context, the existing `/new` reset behavior remains separate, and the user-facing feature is documented in the changelog with focused regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: keep an old `sessionsResult` containing `agent:old:...`, reconnect so `connected` becomes true before fresh sessions load, then create a new session while the PR resolver accepts the stale row and sends `sessions.create` with the old agent id.

## Solution Assessment

No. The PR's product direction is reasonable, but the current implementation is not the safest solution until session-list freshness is tied to the active connection and the required changelog/test coverage is added.
