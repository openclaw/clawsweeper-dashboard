## Summary

Keep open. The cron JSON/quiet-helper problem is still present on current main, but this PR is not merge-ready because its gateway retry is method-agnostic and the branch carries unrelated security-sensitive SSRF and broad CI-repair changes; #54475 is the narrower gateway-side fix for the transient pre-hello close path.

## What This Changes

This PR changes cron helper status calls, adds CLI-side gateway transport retry logic and tests, and currently also carries status, plugin registry, SSRF, build-entrypoint, and channel/plugin test changes.

## Best Possible Solution

Split the work: review or land #54475 or an equivalent structured pre-hello gateway fix, then apply a small current-main cron patch that skips the helper in JSON mode and uses probe-mode options with `expectFinal: false`.

## Reproduction Assessment

Yes. Static current-main inspection shows `cron add --json` can print the RPC result and then run the secondary `cron.status` helper, while #54475 provides a focused unit-level reproduction for the transient pre-hello gateway close class.

## Solution Assessment

No. The cron helper direction is narrow and appropriate, but the PR's method-agnostic CLI retry is not the safest fix; a structured gateway-layer pre-hello tolerance plus a separate cron helper patch is more maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid retrying mutating gateway RPCs:** `src/cli/gateway-rpc.ts:56-68`
  - body: The retry loop retries any method when `callGateway` rejects with matching close/timeout text. Since `callGateway` sends the RPC after `hello-ok`, a close before the response can still happen after the gateway processed a mutating command like `cron.add`, so this can duplicate jobs or repeat side effects. Keep retry scoped to a structured pre-request/pre-hello signal, or use the focused gateway-layer fix in #54475.
  - confidence: 0.84
