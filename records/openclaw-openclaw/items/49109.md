## Summary

Keep open. Current main still lacks the cron JSON/quiet helper change and the CLI gateway retry behavior, while this PR remains too broad and now has a concrete correctness risk in the CLI-side retry loop plus an unrelated security-sensitive SSRF change that should not ride along with the cron fix.

## What This Changes

This PR changes cron CLI helper behavior and CLI gateway transport retry handling, adds cron/gateway tests, and currently also carries broad unrelated status, plugin registry, SSRF, build-entrypoint, generated-baseline, and channel/plugin test changes.

## Best Possible Solution

Split the work. Land #54475, or an equivalent gateway-layer pre-hello fix, for the transient close path, then apply the cron helper change as a small separate patch that skips the helper in JSON mode and uses a quiet probe with `expectFinal: false` outside JSON mode. The unrelated SSRF, registry, status, build, baseline, and channel-test changes should be dropped or reviewed separately.

## Reproduction Assessment

Yes. Static current-main inspection reproduces the cron issue path: `cron add --json` prints JSON and then still invokes `warnIfCronSchedulerDisabled`, and the helper still performs a plain `cron.status` call. The gateway half also has a clear unit-level reproduction path in related #54475 for a transient pre-hello close, though I did not run tests in this read-only review.

## Solution Assessment

No. The cron helper change is directionally right, but the PR's CLI-level retry is not the best fix because it retries by error-message text for every method and can duplicate mutating RPCs; the safer direction is a structured gateway-layer pre-hello tolerance plus a separate cron helper quieting patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid retrying mutating gateway RPCs:** `src/cli/gateway-rpc.ts:50-68`
  - body: The new retry loop retries any method when `callGateway` rejects with a matching close/timeout message. `callGateway` sends the RPC after `hello-ok`, so a socket close before the response can still happen after the gateway processed a mutating command like `cron.add`; retrying here can create duplicate jobs or repeat other side effects. Keep this retry scoped to a structured pre-request/pre-hello signal, or use the focused gateway-layer fix instead.
  - confidence: 0.78
