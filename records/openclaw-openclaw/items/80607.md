## Summary

Keep open: current main and the latest release still route Feishu inbound replies through the shared reply runner into `runEmbeddedAgent`, and the embedded runner still performs the per-run prep stages named in the report. The issue discussion also shows a later stable benchmark where the original gap improved but non-default routed agents remained slower, so this is not a cleanup-close candidate.

## What This Changes

The issue requests lower latency for non-default multi-agent routing by reusing a warm runtime/session path or otherwise avoiding repeated per-message embedded-run setup.

## Best Possible Solution

Keep this issue open for an owner decision on whether routed non-default agents should share a warm direct-session path, use a reusable embedded-run cache, or keep isolation while optimizing the specific prep stages.

## Reproduction Assessment

No high-confidence live current-main timing reproduction was run in this read-only review. Source inspection does reproduce the route shape: Feishu routed replies still enter the shared reply runner, and the OpenClaw runtime branch still calls `runEmbeddedAgent` with the reported prep stages.
