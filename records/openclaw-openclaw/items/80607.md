## Summary

Keep open: the updated reporter traces and current source still support a real per-turn embedded-run setup cost for channel-routed non-default agents, and no merged fix or canonical duplicate fully owns the warm-session/direct-session parity question.

## What This Changes

The issue requests reducing extra per-message setup latency for Feishu/channel-routed non-default agents by reusing a warm runtime/session path instead of paying embedded-run startup on every turn.

## Best Possible Solution

Define the supported runtime contract for channel-routed non-default agents and amortize repeated model/tool/session setup on warm sessions without weakening per-agent workspace, auth, plugin, or transcript isolation.

## Reproduction Assessment

Yes, at source-plus-trace level: reporter-provided Gateway CLI and trace logs on v2026.5.7 show the repeated embedded-run stages, and current main still routes Feishu-bound non-default sessions through dispatchReplyFromConfig into runEmbeddedPiAgent. I did not independently run a live Feishu setup in this checkout.

## Solution Assessment

Unclear: direct-session parity may be the right end state, but the safer maintainable path is to first decide whether to reuse warm embedded session objects or optimize the repeated model/tool/resource stages individually.
