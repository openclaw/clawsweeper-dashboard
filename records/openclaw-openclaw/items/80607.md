## Summary

Keep open: current source supports the reported routing shape for Feishu-bound non-default agents, and the embedded/plugin-tool setup path remains plausibly tied to the reported latency, but this needs runtime design/measurement rather than cleanup closure.

## What This Changes

Request that non-default agents routed through Feishu or multi-agent bindings avoid repeated embedded-run startup latency by reusing a warm/direct session path or equivalent.

## Best Possible Solution

Define the supported runtime contract for channel-routed non-default agents and remove repeated per-turn embedded/plugin-tool setup on warm sessions while preserving per-agent workspace, auth, and session isolation.

## Reproduction Assessment

No, not as a live timing repro. Source inspection shows the Feishu bound-group route enters the generic reply dispatch and embedded runtime path, but the exact latency requires a real Feishu/gateway run with trace logs.

## Solution Assessment

Unclear: direct-session parity may be the right product direction, but a safer first step is to measure the routed session's embedded prep stages and decide whether to optimize plugin-tool setup, runtime reuse, or expose a documented policy knob.
