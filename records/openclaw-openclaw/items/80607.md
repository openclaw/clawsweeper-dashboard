## Summary

Keep open: current main still routes Feishu-bound non-default agent turns through the normal reply dispatch and embedded-run tool-prep path, while the reporter added plausible Gateway-mode timing data; this needs runtime design and current-main measurement, not cleanup closure.

## What This Changes

The issue asks OpenClaw to avoid the repeated embedded-run latency paid by non-default multi-agent sessions, especially for Feishu-routed Gateway turns.

## Best Possible Solution

Define the supported runtime contract for channel-routed non-default agents and eliminate repeated per-turn embedded preparation on warm Gateway sessions while preserving per-agent workspace, auth, tool, and session isolation.

## Reproduction Assessment

Yes, with caveats: the reporter supplied a concrete Gateway-mode CLI benchmark on v2026.5.7 comparing `openclaw agent --agent main` against a non-default agent, and source inspection shows that path enters embedded-run preparation. I did not run the live Feishu/Gateway timing path on current main, so the structured status is source-reproducible rather than reproduced.

## Solution Assessment

Unclear: switching non-default agents to a direct-session path may be right, but the narrower maintainable first step is to capture current-main `[trace:embedded-run]` and `[trace:plugin-tools]` timings and decide whether to reuse embedded runtime state, optimize tool/resource prep, or change routing policy.
