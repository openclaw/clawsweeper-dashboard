## Summary

Keep open: the current source still routes channel-bound non-default agent turns through the generic reply pipeline and embedded runner, and the reporter has added latest-release timing plus trace-stage data after the prior review. This is not a cleanup-close candidate because the remaining question is the runtime contract for warm non-default agent sessions, not whether the issue is obsolete.

## What This Changes

The issue requests reducing non-default multi-agent latency by avoiding repeated per-message embedded_run setup for channel/CLI-routed non-default agents.

## Best Possible Solution

Define a supported warm-session contract for channel-routed non-default agents and remove repeated embedded setup on warm turns while preserving per-agent workspace, auth, plugin, and session isolation.

## Reproduction Assessment

No, not as a high-confidence current-main live reproduction by this review. The reporter provided latest-release gateway timing and trace logs, and current main still shows the same embedded-run source shape, but a current-main Feishu/Gateway timing run is needed to quantify the delta.

## Solution Assessment

Unclear: direct-session parity may be the right end state, but choosing between direct warm sessions, embedded runtime reuse, and targeted prep-stage caching needs maintainer runtime-design judgment. A safer next step is current-main trace comparison for default and non-default sessions before changing the contract.
