## Summary

Keep open: current main still routes the ordinary gateway reply path through `runEmbeddedAgent` and the embedded runner still rebuilds the reported startup/prep stages per run; related cleanup has also treated this issue as a canonical narrower tracker alongside the broad Gateway starvation issue.

## What This Changes

The issue asks OpenClaw to reduce non-default multi-agent message latency by reusing a warm session/runtime path instead of paying embedded-run startup work on every routed message.

## Best Possible Solution

Keep this as the focused non-default multi-agent latency tracker and solve it with profiled runtime/session reuse or prep caching that preserves per-agent isolation, while leaving broad event-loop isolation work on the canonical Gateway starvation tracker.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection confirms the reported embedded-run path and prep-stage markers remain, and the reporter provided stable-release timing data showing the remaining non-default-agent gap.

## Solution Assessment

Unclear whether the requested direct-session reuse is the best fix. A maintainable solution needs profiling first, then a narrow runtime/session reuse or prep-caching design that preserves per-agent workspace, auth, tools, and delivery-context isolation.
