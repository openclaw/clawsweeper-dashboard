## Summary

Keep open: the issue is recent, has fresh reporter Gateway/trace evidence, and current main still shows channel-routed non-CLI turns resolving the routed agent and then falling through to `runEmbeddedPiAgent` with per-run model and tool setup.

## What This Changes

The issue requests reducing extra per-message latency for channel-routed non-default agents by avoiding fresh embedded-run setup where a direct or warm session path should be reusable.

## Best Possible Solution

Define a supported warm-runtime or session-reuse contract for channel-routed non-default agents that amortizes model, tool, and prompt setup while preserving per-agent workspace, auth, and tool-policy isolation.

## Reproduction Assessment

Yes, source/log-reproducible: use a Gateway with two configured agents, route a Feishu/group binding to the non-default agent, and compare repeated `openclaw agent --agent <id> -m ...` or channel turns with trace logging. I did not run a live Feishu/Gateway reproduction in this checkout.

## Solution Assessment

Unclear. Direct-session parity is plausible, but the maintainable fix should first decide the supported warm-runtime/session-reuse boundary rather than adding a Feishu-specific or provider-specific bypass.
