The PR expands read-tool start warning detection to treat `file` and `filePath` as path aliases and adds focused unit coverage for those aliases.

## Best Possible Solution

Align the read-tool parameter contract across schema, execution validation, diagnostics, display, mutation fingerprinting, docs, and tests, then rebase this PR or replace it with a focused fix that includes real behavior proof and changelog coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current main, calling `handleToolExecutionStart` for `read` with only `file` or `filePath` still reaches the warning branch because the guard checks only `path` and `file_path`.

## Solution Assessment

No. As of current main, `createOpenClawReadTool` rejects alias-only read calls, so suppressing the warning for `file` and `filePath` without changing validation would make invalid calls quieter; the maintainable fix is contract alignment.
