## Summary

Keep open. The PR fixes the earlier test assertion gap, but it still suppresses warnings for `file`/`filePath` alias-only read calls that current execution validation and the pinned upstream read schema do not accept, so the patch risks hiding invalid calls rather than aligning the contract.

## What This Changes

The PR adds `file` and `filePath` to the embedded read-tool start-warning alias list and adds handler tests for those aliases.

## Best Possible Solution

Define one coherent read-tool parameter contract across schema, validation, diagnostics, display/fingerprint helpers, docs, and tests; then implement either end-to-end alias support or canonical-`path` enforcement consistently.

## Reproduction Assessment

No for a high-confidence reproduction of the PR's claimed successful-read false-positive path. Source inspection shows `file` or `filePath` reaches the warning branch on current main, but current execution validation rejects alias-only reads before a successful read.

## Solution Assessment

No. Warning-only alias suppression is not the best standalone fix because it makes unsupported alias-shaped calls look valid; the safer direction is maintainer-approved contract alignment.
