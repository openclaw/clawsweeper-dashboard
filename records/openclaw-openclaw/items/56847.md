## Summary

Keep open: this PR is not merge-ready because it suppresses missing-path warnings for `file` and `filePath` even though current main's read tool validation and regression tests still define canonical `path` as the executable contract, and the external contributor proof is only test-level.

## What This Changes

The PR adds `file` and `filePath` to read-tool start warning alias detection and adds unit tests that those aliases suppress the missing-path warning.

## Best Possible Solution

Align the whole read-tool contract: either support these aliases in schema/runtime/validation and warning logic together, or keep canonical `path` only and leave unsupported aliases visible as bad input.

## Reproduction Assessment

No. Source inspection shows the handler can warn on `file` or `filePath`, but current main's read execution path rejects those aliases, so the reported false-positive warning for a successful valid read is not established on current main.

## Solution Assessment

No. Suppressing the warning alone is not the best fix because the read tool would still reject the same alias-shaped call; the maintainable fix must update the runtime contract and tests together if alias support is intended.
