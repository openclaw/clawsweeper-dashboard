## Summary

Keep open. The branch still has a blocking correctness issue: it suppresses missing-path diagnostics for alias-only `read` calls even though current execution validation and regression coverage require canonical `path`; the supplied proof also does not exercise the changed behavior in a real OpenClaw run.

## What This Changes

This PR adds `file` and `filePath` to embedded read-tool start-warning alias handling and adds handler tests for no warning plus block-reply flushing.

## Best Possible Solution

Define one coherent read parameter contract across schema, validation, diagnostics, display/fingerprint helpers, docs, and tests, then either support aliases end to end or keep diagnostics canonical.

## Reproduction Assessment

Mixed: source inspection gives a high-confidence path for `file` or `filePath` without `path` to reach the handler warning branch. The full successful-read false-positive claim is not high-confidence because current execution validation rejects alias-only reads.

## Solution Assessment

No. Warning-only alias suppression is not the best standalone fix because it can hide diagnostics for reads that execution still rejects; the safer direction is maintainer-approved contract alignment.
