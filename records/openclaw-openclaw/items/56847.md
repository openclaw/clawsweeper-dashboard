## Summary

Keep open. The current branch addresses the earlier flush-test review note, but it still only suppresses diagnostics for aliases that current main and the pinned upstream read tool do not accept at execution time; it needs contract alignment and stronger real behavior proof before merge.

## What This Changes

The PR adds `file` and `filePath` to embedded read-tool start-warning alias handling and adds handler tests asserting no warning and block-reply flushing for those aliases.

## Best Possible Solution

Define one coherent read parameter contract across schema, validation, diagnostics, display/fingerprint helpers, docs, and tests, then either support aliases end to end or keep diagnostics aligned with canonical `path`.

## Reproduction Assessment

Mixed: source inspection on current main shows `file` or `filePath` reaches the missing-path warning branch, but the claimed successful-read false-positive path is not high-confidence because current validation rejects alias-only reads.

## Solution Assessment

No. Warning-only alias suppression is not the best standalone fix because it can hide diagnostics for reads that execution still rejects; the safer direction is maintainer-approved contract alignment.
