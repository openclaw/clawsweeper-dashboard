## Summary

Keep open, but do not merge as-is: the branch is a focused warning-path patch, yet current main now enforces canonical `path` for read execution, so suppressing warnings for `file`/`filePath` would make invalid alias-only calls look accepted without restoring alias support end to end.

## What This Changes

The PR adds `file` and `filePath` to read-tool missing-path warning detection and adds focused tests for those aliases.

## Best Possible Solution

Settle the read-tool parameter contract first, then make schema, validation, diagnostics, display, fingerprinting, docs, and tests match that contract in a rebased patch.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `handleToolExecutionStart` reaches the warning branch for `file` or `filePath`, while current read execution validation accepts only `path`. I did not run tests because this review is read-only and `pnpm` is unavailable in the checkout environment.

## Solution Assessment

No. A warning-only alias expansion is not the best current-main fix because read execution now rejects alias-only params; the safer path is a maintainer-approved contract alignment, either canonical-only everywhere or aliases restored end to end.
