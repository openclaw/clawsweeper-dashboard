## Summary

Keep open. The PR handles the narrow warning branch, but current main still treats `read` execution as canonical `path` only and has tests rejecting alias-only reads, so this patch would suppress a diagnostic for calls that execution still rejects unless maintainers choose alias compatibility end to end.

## What This Changes

This PR adds `file` and `filePath` to embedded read-tool start-warning alias handling and adds regression tests for no warning plus block-reply flushing.

## Best Possible Solution

Define one coherent read parameter contract across schema, validation, diagnostics, display/fingerprint helpers, docs, and tests, then either support aliases end to end or keep diagnostics canonical.

## Reproduction Assessment

Mixed: source inspection shows `file` or `filePath` without `path` reaches the handler warning branch. The full successful-read false-positive claim is not high-confidence because current execution validation rejects alias-only reads.

## Solution Assessment

No. Warning-only alias suppression is not the best standalone fix because it can hide diagnostics for reads that execution still rejects; the safer direction is maintainer-approved contract alignment.
