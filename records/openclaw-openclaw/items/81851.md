## Summary

Keep open: the feature is useful and has real proof, but the current head is not merge-ready because it regresses current-main auth-profile runtime routing, broadens native filesystem-read permission on overflow prompts, needs explicit security/product approval for local TLS interception, and is currently conflicting with main.

## What This Changes

The PR adds an opt-in Anthropic `claude-cli-interactive` backend that runs Claude through a Bun loopback TLS proxy, reconstructs stream-json reasoning output, wires the backend through CLI/runtime gates, and adds tests/static assets plus a Bun type dependency.

## Best Possible Solution

Keep the opt-in backend only if it rebases onto current auth routing, removes or tightly bounds native `Read` overflow recovery, documents or internalizes the new CLI backend inheritance hook, and receives explicit maintainer security approval for the TLS proxy design.

## Reproduction Assessment

Yes for the review blockers by source inspection: PR head omits `authProfileId` in runtime selection while current main passes it, and the overflow branch appends `--allowedTools Read`. I did not run tests because this is a read-only review.

## Solution Assessment

No. The opt-in backend direction may be useful, but this implementation should first preserve current auth routing, avoid permission expansion on prompt overflow, and get an explicit maintainer decision on the TLS proxy boundary.
