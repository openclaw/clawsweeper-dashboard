## Summary

Keep open. Current main still only has the existing `claude-cli` backend, while this draft PR adds a new security-sensitive interactive Claude backend; the branch still needs a setup-registration fix, stronger current-head real behavior proof, and maintainer/security acceptance of the local TLS proxy design.

## What This Changes

The PR adds an experimental Anthropic `claude-cli-interactive` backend that runs Claude through Bun and a loopback TLS proxy, streams `thinking_delta` into CLI/auto-reply reasoning, and updates CLI backend config inheritance, plugin metadata, tests, and static package assets.

## Best Possible Solution

Keep this as a draft until setup metadata and runtime registration are aligned, current-head live proof is attached, and maintainers either accept the child-scoped TLS proxy trust model or choose a less invasive backend design.

## Reproduction Assessment

Yes for the review finding: source inspection shows the new backend id is declared in the manifest while `setup-api` still registers only the legacy backend, and setup lookup requires a matching setup-registered backend. I did not run a live Claude subscription or Telegram flow.

## Solution Assessment

No in its current shape: an opt-in backend may be a viable direction, but the setup registration gap, current-head proof gap, and TLS proxy trust model need to be resolved before merge.
