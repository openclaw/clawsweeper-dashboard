## Summary

Keep this PR open, but it still needs changes before merge: the opt-in backend is useful and has real Windows proof, yet the current diff leaves proxy key material outside OpenClaw's configured state root and the long-prompt overflow path can require an unattended Claude permission prompt.

## What This Changes

Adds an experimental Anthropic `claude-cli-interactive` backend that runs Claude Code through a Bun loopback TLS proxy, reconstructs Claude stream-json events, extends Claude CLI routing/auth gates, packages proxy support files, and adds focused tests.

## Best Possible Solution

Land this only as an explicit opt-in experimental backend after the proxy key lifecycle honors OpenClaw state isolation, overflow prompt recovery is non-interactive or fails fast, and maintainers accept the local TLS interception boundary with upgrade proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: the cert path is hard-coded outside `resolveStateDir()`, and the overflow branch asks Claude to read a temp file while the backend only auto-allows OpenClaw MCP tools. I did not run a live long-prompt failure path.

## Solution Assessment

No, not yet. The opt-in backend is a plausible direction, but the mergeable shape should preserve state/profile isolation and make overflow recovery deterministic before maintainers decide whether the local MITM boundary is acceptable.
