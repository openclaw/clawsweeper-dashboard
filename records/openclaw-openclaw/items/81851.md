## Summary

Keep open. The feature is useful and has credible Windows live-proof, but the current head is not merge-ready because it drops current-main auth-profile-aware CLI runtime routing, changes the overflow prompt path to auto-allow native Claude `Read`, and needs explicit maintainer/security approval for local TLS interception.

## What This Changes

Adds an opt-in Anthropic `claude-cli-interactive` backend that runs Claude Code through a Bun loopback TLS proxy, reconstructs stream-json reasoning output, wires the backend through Anthropic and CLI runtime paths, and adds tests/static assets.

## Best Possible Solution

Keep the opt-in backend only if it rebases cleanly onto current main, preserves auth-profile runtime routing, uses a permission-preserving overflow prompt path or approved security model, and gets explicit owner approval for the TLS proxy boundary.

## Reproduction Assessment

Yes for the review blockers by source inspection: current main passes `authProfileId` into CLI runtime selection while PR head omits it, and PR head appends `--allowedTools Read` for overflow prompts. I did not run tests because this is a read-only review.

## Solution Assessment

No. The backend direction is plausible and the proof is meaningful, but this implementation needs a rebase-safe auth routing fix, a narrower overflow delivery path, and explicit maintainer approval of the TLS interception boundary before it is the best solution.
