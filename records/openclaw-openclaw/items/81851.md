## Summary

Keep this PR open, but it is not merge-ready. The backend is useful and has real Windows runtime proof, but the current head is conflicting with main and still has blocking auth-routing and security-boundary issues.

## What This Changes

The PR adds an opt-in Anthropic `claude-cli-interactive` backend that runs Claude through a Bun loopback TLS proxy, reconstructs stream-json output, wires shared Claude CLI routing gates, and adds tests/static runtime assets.

## Best Possible Solution

Land the opt-in backend only after a clean rebase preserves auth-profile-aware runtime selection, overflow recovery stays within an approved permission model, and maintainers explicitly accept the local TLS proxy boundary with cross-platform proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: current main passes `authProfileId` into CLI runtime selection while the PR head omits it, and the PR head appends `--allowedTools Read` for overflow prompts. I did not run tests because this is a read-only review.

## Solution Assessment

No. The backend idea is plausible and the supplied proof is useful, but this implementation needs rebase-safe auth routing, a narrower or explicitly approved overflow permission path, and maintainer approval of the TLS interception boundary.

Even with `--add-dir`, that changes the approval boundary from the backend's MCP-only tool allowlist to a native filesystem-read tool. Use a permission-preserving handoff, or get explicit maintainer security approval for this exact native `Read` model with focused tests.
