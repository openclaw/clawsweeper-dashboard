## Label Justifications

- `P1`: The PR targets a near-term Anthropic CLI billing/runtime workflow problem that could materially affect Claude subscription users, but it still has merge-blocking safety work.
- `merge-risk: 🚨 compatibility`: The new backend inherits existing CLI config and writes persistent proxy state, so state-dir/profile isolation and upgrade behavior can change for users who opt in.

## Summary

Keep open: the feature is useful and has real Windows proof, but the patch is not merge-ready because proxy key storage, long-prompt overflow handling, and the new local TLS interception boundary still need repair or explicit maintainer acceptance.

## What This Changes

Adds an opt-in Anthropic `claude-cli-interactive` backend that runs Claude Code through a Bun loopback TLS proxy, reconstructs Claude stream-json events, extends Claude CLI routing/auth gates, packages proxy assets, and adds focused tests.

## Best Possible Solution

Land only after proxy state uses the configured state dir, overflow handling is non-interactive or fail-fast, and maintainers explicitly accept the experimental local TLS interception boundary with cross-platform proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: the cert manager hard-codes `homedir()/.openclaw`, and the overflow path asks Claude to `Read` a temp file while only OpenClaw MCP tools are auto-allowed. I did not run a live failing long-prompt turn.

## Solution Assessment

No, not yet. The proposed backend may be the right product direction, but the current implementation should first fix state isolation and overflow behavior, then get explicit security acceptance for the local TLS proxy.
