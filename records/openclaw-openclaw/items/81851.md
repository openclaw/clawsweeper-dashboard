## Summary

Keep open: this is a substantial draft feature, not obsolete, and current-head source review found blocking tool-turn and terminal-stop handling bugs plus a concrete TLS proxy key-hardening concern. The supplied proof shows the wrapper starting in a real install, but not the visible reasoning output this backend is meant to deliver.

## What This Changes

The PR adds an opt-in `claude-cli-interactive` Anthropic backend with a Bun loopback TLS proxy, Claude stream JSONL reconstruction, config inheritance, parser/reasoning bridge wiring, packaging/tests, and doctor warning changes.

## Best Possible Solution

Keep the backend experimental and opt-in, fix the wrapper state machine for tool continuations and terminal stop reasons, fail closed on MITM key hardening, rebase away unrelated current-main changes, then require visible Telegram or stdout JSONL proof plus maintainer/security review.

## Reproduction Assessment

Yes for the review blockers: source inspection shows later `message_start` streams are suppressed, finalization only runs on `end_turn`, and key-hardening failures are ignored. I did not run a live Claude subscription or Telegram turn locally.

## Solution Assessment

No in its current form: an opt-in backend may be a maintainable direction, but the wrapper must preserve Claude's multi-request tool flow and terminal stop reasons, and the TLS key/trust boundary needs a fail-closed implementation and maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve post-tool message streams:** `extensions/anthropic/interactive-proxy/wrapper.ts:301-319`
  - body: Claude tool use can end one stream with `stop_reason: "tool_use"` and then continue through another `/v1/messages` stream after the CLI runs the tool. This code treats every `message_start` after the first as auxiliary and drops it, so any tool-using turn can lose the continuation and never emit the final `result`.
  - confidence: 0.91
- **[P2] Finish terminal non-end_turn stops:** `extensions/anthropic/interactive-proxy/wrapper.ts:350-362`
  - body: The Messages API also uses terminal stop reasons like `max_tokens`, `stop_sequence`, and `refusal`. When one of those arrives, this branch skips result emission and never kills the interactive child, turning a valid final response into a watchdog timeout.
  - confidence: 0.86
- **[P2] Fail closed when key hardening fails:** `extensions/anthropic/interactive-proxy/cert-manager.ts:123-132`
  - body: `hardenKeyPermissions()` still swallows `icacls`/`chmod` failures and `ensureCerts()` returns the reusable MITM signing keys anyway. In the broad-permission state shown by the proof artifact, a failed repair keeps trusting readable proxy private keys instead of stopping or regenerating safely.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a local TLS MITM backend, and current-head key hardening still returns reusable signing keys when ACL repair fails.

Concerns:

- **[medium] Fail closed on proxy key hardening failures:** `extensions/anthropic/interactive-proxy/cert-manager.ts:130`
  - body: The certificate manager ignores `icacls`/`chmod` failures and continues with cached or newly generated CA/leaf private keys. For a reusable local MITM signing key, failed ACL repair should prevent use rather than preserving broader access.
  - confidence: 0.9
- **[medium] Review the TLS interception trust boundary:** `extensions/anthropic/interactive-proxy/wrapper.ts:191`
  - body: The wrapper forces Claude through a loopback HTTPS proxy with child-scoped `HTTPS_PROXY` and `NODE_EXTRA_CA_CERTS`, decrypting Anthropic API traffic locally to recover thinking events. Even opt-in and child-scoped, this needs explicit maintainer/security acceptance and user-facing documentation.
  - confidence: 0.88
