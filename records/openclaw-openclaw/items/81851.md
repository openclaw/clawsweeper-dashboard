## Summary

Keep open: the proof is strong, but this head is not merge-ready because it regresses current auth-profile-aware CLI runtime selection and expands the native Claude filesystem-read permission path for overflow prompts.

## What This Changes

The PR adds an opt-in Anthropic `claude-cli-interactive` backend that runs Claude Code through a Bun loopback TLS proxy, reconstructs Claude stream-json reasoning output, wires the new backend through shared Claude CLI gates, and adds tests/static runtime assets.

## Best Possible Solution

Keep the opt-in backend only if it preserves current auth/profile routing, keeps overflow prompt recovery inside a narrow approved tool boundary, and receives explicit maintainer security/provider approval for local TLS interception.

## Reproduction Assessment

Yes for the review blockers by source inspection: current main passes `authProfileId` into CLI runtime selection while PR head omits it, and PR head appends `--allowedTools Read` for overflow prompts. I did not run tests because this is a read-only review.

## Solution Assessment

No. The backend idea is plausible and the proof is meaningful, but this implementation needs a rebase-safe auth routing fix, a narrower overflow delivery path, and explicit approval of the TLS interception boundary before it is the best solution.
