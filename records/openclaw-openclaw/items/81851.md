## Summary

Keep open: the PR is still draft, source inspection finds a remaining proxy-key permission bug, the current proof does not show the promised stream-event/thinking output or visible Telegram reasoning, and the local TLS interception design needs explicit security/product review.

## What This Changes

The PR adds an opt-in Anthropic `claude-cli-interactive` backend that runs Claude through a Bun wrapper and loopback TLS proxy to reconstruct stream-json reasoning events.

## Best Possible Solution

Keep the backend experimental in the Anthropic plugin boundary until key-permission handling, current-head live reasoning proof, a careful rebase, and maintainer security/product acceptance are resolved.

## Reproduction Assessment

Yes for the review blocker: source inspection shows existing unexpired proxy keys are accepted without permission repair, and the supplied cert artifact shows readable key files. I did not reproduce a live Claude subscription or Telegram turn locally.

## Solution Assessment

No in its current shape: an opt-in backend may be a maintainable direction, but this patch still needs the key-permission fix, stronger real behavior proof, and a maintainer decision on local TLS interception.

## Security Review

Status: needs_attention

Summary: The PR introduces local TLS interception and persisted signing keys, so the trust model and key handling need explicit security review before merge.

Concerns:

- **[medium] Review the TLS interception trust boundary:** `extensions/anthropic/interactive-proxy/wrapper.ts:122`
  - body: The wrapper forces the Claude child through a loopback HTTPS proxy and adds a generated CA via `NODE_EXTRA_CA_CERTS`, decrypting Anthropic API traffic locally to extract thinking events. Even with child-scoped trust, this changes the provider traffic trust boundary and should be an explicit documented opt-in decision.
  - confidence: 0.92
- **[medium] Fail closed or repair existing proxy key permissions:** `extensions/anthropic/interactive-proxy/cert-manager.ts:122`
  - body: The certificate manager accepts existing unexpired CA and leaf key files before permission hardening, while the supplied proof artifact shows key files readable beyond the owner. Reusable signing material should be verified or repaired on every fast path, or rejected if it cannot be made private.
  - confidence: 0.84
