## Summary

Keep this PR open: current main does not have the interactive Claude backend, but the branch is still draft, needs current-head real behavior proof and maintainer/security approval for the local TLS interception design, and has a setup-only config inheritance bug that should block merge.

## What This Changes

The PR adds an opt-in Anthropic `claude-cli-interactive` backend that runs Claude through a Bun loopback TLS proxy to reconstruct stream-json reasoning output, plus parser/reasoning-lane wiring, backend config inheritance, static assets, tests, and doctor warning changes.

## Best Possible Solution

Keep the backend experimental and opt-in only if setup-only resolution inherits documented `claude-cli` overrides, the TLS trust boundary is accepted and documented, and current-head proof demonstrates JSONL reasoning plus Telegram-visible behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature PR adding a new backend. Source inspection confirms current main lacks the backend, and the review is about the proposed runtime behavior and trust boundary.

## Solution Assessment

Unclear in its current form: a plugin-owned opt-in backend is a plausible location, but setup-only config inheritance, current-head proof, and maintainer/security acceptance need to be resolved first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Inherit sibling config in setup-only paths:** `src/agents/cli-backends.ts:221`
  - body: The inherited `claude-cli` override is only applied when the runtime registry returns a registered backend. This PR also registers `claude-cli-interactive` for cold setup/live-test/fallback paths, where `registered` is absent and the fallback policy ignores a documented `cliBackends.claude-cli.command` override, so minimal-PATH setups fall back to spawning `claude` from PATH instead of the configured binary. Thread `inheritUserConfigFrom` through setup fallback resolution or apply sibling inheritance before both paths.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Needs attention: the PR introduces a local TLS interception backend and still needs current-head proof for the reusable CA/key hardening path.

Concerns:

- **[medium] Review TLS interception trust boundary:** `extensions/anthropic/interactive-proxy/wrapper.ts:242`
  - body: The backend forces the Claude child through a loopback HTTPS proxy and trusts a generated CA through `NODE_EXTRA_CA_CERTS`, so Anthropic API traffic is decrypted locally to recover reasoning events. Even though this is child-scoped and opt-in, it needs explicit maintainer/security acceptance and user-facing documentation.
  - confidence: 0.9
- **[medium] Re-prove current-head proxy key hardening:** `extensions/anthropic/interactive-proxy/cert-manager.ts:164`
  - body: The submitted certificate artifact shows broad private-key permissions from an earlier branch state, while current head now attempts fail-closed key hardening. Because this backend creates reusable MITM CA material, the PR should include current-head proof showing repaired permissions or the fail-closed path before merge.
  - confidence: 0.84
