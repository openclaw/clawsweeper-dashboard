## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pin every proxy env alias to the loopback proxy:** `extensions/anthropic/interactive-proxy/wrapper.ts:590-595`
  - body: `claudeEnv` preserves the parent environment and sets only `HTTPS_PROXY`. If the gateway already has lowercase proxy variables, `HTTP_PROXY`, `ALL_PROXY`, `NO_PROXY`, or lowercase `no_proxy`, the Claude child can bypass or override the intended loopback proxy, so the wrapper may never see the SSE events that drive reasoning/tool progress. Clear proxy/no-proxy aliases and set the supported proxy variables consistently for this child.
  - confidence: 0.92
- **[P1] Normalize and fail closed before opening CONNECT tunnels:** `extensions/anthropic/interactive-proxy/mitm-server.ts:422-426`
  - body: The CONNECT router compares the raw host to `api.anthropic.com` and opens `net.connect` for every other target. Case/format variants can bypass capture, and any local client that reaches this loopback proxy can tunnel arbitrary non-API egress while the proxy is alive. Normalize host/port first, route only the approved Anthropic target to the TLS server, and reject or explicitly allowlist everything else.
  - confidence: 0.94
- **[P1] Declare the Bun type package used by the proxy tsconfig:** `extensions/anthropic/interactive-proxy/tsconfig.json:4`
  - body: The new proxy tsconfig asks TypeScript for `bun-types`, and the PR adds this project to `tsgo:prod`, but the owning Anthropic package does not declare that type package. A fresh production typecheck should not depend on undeclared ambient types, so add the owning devDependency or remove the type reference.
  - confidence: 0.9
- **[P1] Preserve Claude skills on the interactive backend:** `extensions/anthropic/cli-backend-interactive.ts:103`
  - body: Adding `claude-cli-interactive` as a Claude CLI backend is not enough for selected OpenClaw skills: at this head, `prepareClaudeCliSkillsPlugin` still returns no plugin args unless the backend id is exactly `claude-cli`. Users who switch to the new backend will silently lose their selected Claude skills unless the helper and tests use the Claude CLI family predicate.
  - confidence: 0.88
- **[P1] Carry the current Claude CLI capability flags onto the variant:** `extensions/anthropic/cli-backend-interactive.ts:116`
  - body: The new backend declares `nativeToolMode` but not the current `claude-cli` `sideQuestionToolMode: "disabled"` and `ownsNativeCompaction: true` semantics. Because it spawns the same Claude CLI and uses resumable Claude transcripts, side-question and compaction paths will treat the variant differently unless the backend either carries the same flags or documents and tests a deliberate difference.
  - confidence: 0.84
- **[P1] Document or keep private the inheritance hook:** `src/plugins/cli-backend.types.ts:201-204`
  - body: `CliBackendPlugin` is exported through the plugin SDK, so adding `inheritUserConfigFrom` creates a public plugin API. The PR head's CLI backend plugin docs do not define this hook, which leaves third-party plugin authors without a contract for inheritance precedence, arg filtering, or compatibility. Either keep the mechanism internal to bundled backends or update the public docs/API coverage.
  - confidence: 0.86
