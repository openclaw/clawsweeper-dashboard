## Summary

Keep open: this active draft adds a security-sensitive experimental backend, and current head still has a blocking fail-open proxy-key hardening issue plus insufficient live JSONL/Telegram proof.

## What This Changes

This draft PR adds an opt-in Anthropic `claude-cli-interactive` backend that runs Claude through a Bun loopback TLS proxy, converts captured SSE thinking events into CLI JSONL/reasoning streams, registers the backend/config inheritance/static assets, and adjusts Doctor warning logic.

## Best Possible Solution

Keep the backend experimental and opt-in, but require fail-closed proxy-key handling, updated live JSONL/Telegram proof, and explicit maintainer security approval before merge.

## Reproduction Assessment

Yes for the review blocker: PR-head source shows key-hardening failures are swallowed and `ensureCerts()` still returns the signing keys. I did not reproduce a live Claude subscription or Telegram turn locally.

## Solution Assessment

No in its current form: the opt-in backend may be a maintainable direction, but reusable proxy signing keys must be verified or rejected when permission hardening fails.

Concerns:

- **[medium] Fail closed on proxy key hardening failures:** `extensions/anthropic/interactive-proxy/cert-manager.ts:130`
  - body: The certificate manager ignores failures from the permission-hardening step and still returns reusable CA/leaf private keys. For a local MITM signing key, failed repair should prevent reuse rather than preserving broader ACLs.
