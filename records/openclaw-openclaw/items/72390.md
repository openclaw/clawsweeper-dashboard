## Summary

Keep open. Current main still defines top-level `env.vars` as string-only across schema, public types, generated schema, runtime collection, and tests; adding SecretRef support is a real feature request with security-sensitive runtime versus durable-service env semantics to decide.

## What This Changes

Request support for SecretRef objects in top-level `openclaw.json` `env.vars` so externally managed secrets can populate environment variables at runtime.

## Best Possible Solution

Approve and document a top-level `env.vars` SecretInput contract, then update schema/types/generated docs, target registry, resolver coverage, and regression tests with explicit plaintext persistence boundaries.

## Reproduction Assessment

Yes, from source inspection. A structured object under `env.vars` is outside the accepted schema/type shape and is skipped by `collectConfigRuntimeEnvVars` if raw JSON reaches runtime.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable direction is to reuse the existing SecretInput resolver and target registry while explicitly separating runtime env injection from durable service env persistence.
