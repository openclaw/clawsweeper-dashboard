- `impact:security`: The request changes how externally managed secrets may be resolved and materialized into OpenClaw environment handling.
- `impact:auth-provider`: The affected behavior is SecretRef resolution for credential-like environment variables used by providers, plugins, or skills.

## Summary

Keep open: current main and the latest release still make top-level `env.vars` a literal string-only surface, while SecretRef resolution is limited to explicitly registered credential paths. This remains a product/security decision because resolving SecretRefs into environment variables needs clear runtime versus durable service-env semantics.

## What This Changes

Support `SecretRef` object values in `openclaw.json` `env.vars` so env vars can be resolved from configured secret providers at runtime.

## Best Possible Solution

Define a focused SecretInput-compatible `env.vars` contract with explicit runtime and durable-service-env semantics, then update schema, types, resolution, redaction, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection shows `env.vars` accepts strings only, runtime collection skips object values, and current tests document that skip behavior; this is a missing feature rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the contract. Reusing the existing SecretInput/SecretRef runtime pipeline is the plausible direction, but a schema-only change would be the wrong layer without service-env and redaction decisions.
