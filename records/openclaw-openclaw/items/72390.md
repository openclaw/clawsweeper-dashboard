## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment surface and skips structured SecretRef objects, while the existing SecretRef target registry covers specific credential paths such as ACPX MCP server env values but not top-level env injection.

## What This Changes

Allow `openclaw.json` `env.vars` entries to use SecretRef objects resolved at runtime, including exec-backed sources such as 1Password.

## Best Possible Solution

Approve the exact `env.vars` SecretInput boundary first, then update schema/types/generated docs/target metadata and resolve through the existing SecretRef assignment pipeline with explicit runtime-only versus durable service-env semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put `{ "source": "exec", "provider": "op-my-secret", "id": "value" }` under `env.vars`; schema/types require strings and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The maintainable approach would reuse the existing SecretInput resolver, target registry, and config collectors rather than adding a provider-specific or exec-only shortcut.
