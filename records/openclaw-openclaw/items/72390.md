## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only config surface, while SecretRef resolution is limited to enumerated credential targets; the related MCP env SecretRef work does not cover this global runtime/service environment path.

## What This Changes

The issue requests SecretRef object support in top-level `openclaw.json` `env.vars` so external secret providers can populate environment variables at runtime.

## Best Possible Solution

Approve and implement a top-level `env.vars` SecretInput contract with explicit runtime-versus-durable-service semantics, then update schema/types/generated docs, SecretRef target metadata, collectors, redaction, and focused tests.

## Reproduction Assessment

Yes. Put a structured `{source, provider, id}` value under `env.vars`; current schema/types require strings, and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The maintainable path is to reuse the existing SecretInput assignment/resolution pipeline and explicitly block or document any durable service-env persistence of resolved plaintext.
