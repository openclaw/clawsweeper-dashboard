## Summary

Keep open. Current main still exposes top-level `env.vars` as a string-only environment surface, while SecretRef resolution is limited to enumerated credential paths; this feature is valid but needs maintainer/security approval before resolving external secrets into global runtime or durable service environment handling.

## What This Changes

Support structured SecretRef values in top-level `openclaw.json` `env.vars` so environment variables can be resolved dynamically from providers such as 1Password.

## Best Possible Solution

Approve a top-level `env.vars` SecretInput contract, then update schema/types/generated docs/target metadata and resolve through the existing SecretRef assignment pipeline with explicit runtime-versus-durable service-env semantics.

## Reproduction Assessment

Yes. Put a structured `{source, provider, id}` SecretRef under `env.vars`; schema/types require strings, and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable path is to reuse the existing SecretInput resolver, target registry, and config collectors, with new tests proving plaintext is not persisted where it should not be.
