## Summary

Keep open. Current main still defines `env.vars` as a string-only config surface, skips non-string values at runtime, and routes config env into durable service env collection, so SecretRef support here needs an explicit maintainer/security policy decision.

## What This Changes

Request adding SecretRef/SecretInput support to top-level `openclaw.json` `env.vars` so external secret providers can supply runtime environment variables.

## Best Possible Solution

Define and document a top-level `env.vars` SecretInput contract with explicit runtime versus durable service-env semantics, then update schema/types/generated docs/target registry and regression tests together.

## Reproduction Assessment

Yes, from source inspection. A structured SecretRef under `env.vars` is outside the schema/type contract and would be skipped by `collectConfigRuntimeEnvVars` if raw JSON reached runtime.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable path is to reuse the existing SecretInput schema, resolver, target registry, generated docs, and service-env safety model rather than adding ad hoc object expansion.
