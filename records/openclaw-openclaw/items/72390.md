## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface and skips structured SecretRef objects at runtime, while the supported SecretRef registry/docs cover specific credential paths such as ACPX MCP env but not global env vars; the request is valid but crosses a security-sensitive process/service environment boundary that needs maintainer/secops design before implementation.

## What This Changes

Support structured SecretRef values in the top-level `openclaw.json` `env.vars` map so environment variables can be resolved dynamically at runtime from configured secret providers such as exec/1Password.

## Best Possible Solution

Extend the generic SecretInput/SecretRef pipeline for this core config surface after maintainer/secops design: widen `env.vars` schema/types to approved SecretInput values, collect refs under an explicit target path, resolve them before applying env values, preserve existing-env precedence plus dangerous-env and unresolved-placeholder protections, define runtime vs durable service behavior, and update generated schema/docs, target registry metadata, redaction, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`: schema/type validation accepts only strings today, and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable solution should use the existing generic SecretInput resolver and registry surfaces with explicit security boundaries rather than adding a 1Password-specific or exec-only shortcut.
