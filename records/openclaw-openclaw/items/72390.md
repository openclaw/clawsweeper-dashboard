## Summary

Keep open. Current main still treats `env.vars` as static string-only configuration and applies only static strings before substitution, while the supported SecretRef target registry/docs do not include `env.vars`; the related plugin-source work at https://github.com/openclaw/openclaw/pull/72548 explicitly defers broader env/config field widening rather than superseding this request.

## What This Changes

Request support for structured SecretRef values in `openclaw.json` `env.vars` so environment variables can be resolved from external secret providers at runtime.

## Best Possible Solution

Define a first-class SecretInput-aware `env.vars` path that resolves configured refs through the existing secrets runtime, preserves dangerous-var filtering, avoids persisting resolved values, and updates schema, target registry, audit/configure behavior, docs, and focused tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing contract. Source inspection is high-confidence that current `env.vars` accepts/applies static strings only and does not resolve structured SecretRefs.

## Solution Assessment

Unclear until maintainers set the security/product boundary. The maintainable direction is to reuse the existing SecretInput resolver and target-registry model, not add an ad hoc `env.vars`-only exec object path.
