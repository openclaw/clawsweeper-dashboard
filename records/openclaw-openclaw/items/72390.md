## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only config/service environment surface, and the SecretRef runtime collectors do not collect or resolve SecretRef objects there. The request is valid, but it crosses global runtime and durable service environment materialization, so it needs maintainer/secops design before implementation.

## What This Changes

Add SecretRef support to the top-level `openclaw.json` `env.vars` map so environment variables can be resolved from external secret providers such as 1Password at runtime.

## Best Possible Solution

Add this as an end-to-end SecretInput extension only after the environment boundary is approved: widen schema and public types, collect and resolve `env.vars.*` through the existing runtime secret pipeline, define durable service-env persistence semantics, update generated schema/docs/audit metadata, and cover the behavior with focused config, secrets-runtime, registry, and daemon-environment tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`; current schema/types require strings, and the runtime env collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable fix should reuse the existing SecretInput resolver, target registry, and service-env policy rather than adding a provider-specific or exec-only shortcut.
