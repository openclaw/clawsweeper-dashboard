## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface, and the related open plugin-provider work explicitly defers broader env/config SecretRef field coverage, so this remains a valid security-sensitive feature request.

## What This Changes

Request support for SecretRef values under top-level `openclaw.json` `env.vars` so environment variables can be resolved from external secret providers at runtime.

## Best Possible Solution

Approve a top-level `env.vars` SecretInput contract with explicit runtime-only versus durable-service-env semantics, then update schema/types/docs, runtime target collection, resolution, and regression coverage without broadening unrelated env maps by accident.

## Reproduction Assessment

Yes for the current limitation, from source inspection: a structured SecretRef object under `env.vars` is outside schema/types and would be skipped by runtime collection if raw JSON reached that code path. This is not a broken existing contract; it is a requested expansion of supported SecretRef surfaces.

## Solution Assessment

Unclear because there is no patch yet. The maintainable solution is to reuse the existing SecretInput assignment and target-registry model, but only after maintainers define the security boundary for resolved env values.
