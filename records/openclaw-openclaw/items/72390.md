## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface, and resolving SecretRefs there needs a maintainer decision because those values also feed durable service environment collection.

## What This Changes

Allow `openclaw.json` `env.vars` entries to use SecretRef objects such as exec-backed 1Password values instead of only literal strings.

## Best Possible Solution

Define a top-level `env.vars` SecretInput contract with explicit runtime-only versus durable-service-env semantics, then update schema/types/docs, runtime collection/resolution, and focused regression coverage.

## Reproduction Assessment

Yes, via source inspection: zod/types accept strings only and config env collection drops non-string `env.vars` values. This is a feature expansion request rather than broken existing behavior.

## Solution Assessment

Unclear because there is no patch yet. Reusing the existing SecretInput resolution pipeline is plausible, but maintainers first need to decide whether resolved env secrets may enter durable service metadata.
