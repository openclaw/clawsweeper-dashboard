## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only env injection surface: schema/types/generated schema reject SecretRef objects, runtime skips non-strings, and the supported SecretRef target surface does not include `env.vars`; adding it needs maintainer/security policy because config env values also feed durable service env collection.

## What This Changes

Requests support for SecretRef values in top-level `openclaw.json` `env.vars` so external secret managers can supply environment variables at runtime.

## Best Possible Solution

Approve and document a top-level `env.vars` SecretInput contract with explicit runtime versus durable service env semantics, then update schema/types/generated docs/target registry and regression tests.

## Reproduction Assessment

Yes, from source inspection. Put a structured SecretRef object under `env.vars`; the schema/public type accept only strings, and `collectConfigRuntimeEnvVars` skips non-string values if raw JSON reaches runtime.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable path is to reuse the existing SecretInput resolver, target registry, generated schema/docs pipeline, and tests rather than adding ad hoc env expansion.
