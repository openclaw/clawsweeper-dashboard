## Summary

Keep open: current main and v2026.6.10 still treat top-level `env.vars` as a literal string-only config surface, while SecretRef resolution remains allowlisted to supported credential paths. The request is useful and source-reproducible, but it needs maintainer product/security direction because it changes how secrets materialize into runtime and service environments.

## What This Changes

Requested SecretRef support for `openclaw.json` top-level `env.vars` so environment variables can resolve from dynamic secret providers such as 1Password-backed exec providers.

## Best Possible Solution

Define a focused SecretInput-compatible `env.vars` contract with explicit runtime-versus-service-env semantics, then update schema, types, resolution, redaction, docs, and regression coverage together.

## Reproduction Assessment

Yes, by source inspection: current main declares `env.vars` as string-only and skips non-string values at runtime, which matches the reported SecretRef object gap.

## Solution Assessment

Unclear until maintainers choose the contract; reusing the existing SecretInput/SecretRef pipeline is the likely maintainable direction, but the safe fix must define service-env persistence and redaction behavior.
