## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment surface and does not register, collect, or resolve structured SecretRef values there; the requested capability is valid but crosses a security-sensitive runtime/service environment boundary.

## What This Changes

Issue requests SecretRef support for top-level `openclaw.json` `env.vars` values so external secret providers can populate runtime environment variables.

## Best Possible Solution

Approve the `env.vars` SecretInput boundary first, then extend schema/types/generated docs/target metadata and route resolution through the existing SecretRef assignment pipeline with explicit runtime versus service-env semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`; schema/types require strings, and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The maintainable path is to reuse the existing SecretInput resolver, target registry, and config collectors rather than adding a provider-specific or exec-only shortcut.
