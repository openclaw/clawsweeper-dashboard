- `impact:security`: The request changes how externally managed secrets may be resolved and materialized into OpenClaw environment handling.

## Summary

Keep open: current main and v2026.6.8 still treat top-level `env.vars` as a literal string-only config surface, while SecretRef resolution remains limited to supported credential paths. This is a useful security/auth-provider feature, but it needs maintainer product and security direction because it would change how secret material is resolved into runtime and managed service environments.

## What This Changes

Allow `openclaw.json` `env.vars` values to use SecretRef objects so external secret managers can populate environment variables at runtime.

## Best Possible Solution

Define a focused SecretInput-compatible `env.vars` contract with explicit runtime versus durable-service-env semantics, then update schema, types, resolution, redaction, docs, and regression coverage together.

## Reproduction Assessment

Yes, by source inspection. Current main declares `env.vars` as string-only, skips non-string values at runtime, and has a regression test for that skip behavior.

## Solution Assessment

Unclear until maintainers choose the contract. Reusing the existing SecretInput/SecretRef pipeline is the maintainable direction, but the safe fix must also define service-env persistence and redaction behavior.
