## Summary

Keep open. Current main still models top-level `env.vars` as a string-only global environment injection surface and skips non-string values at runtime; existing SecretRef support covers targeted credential paths and plugin-declared MCP env surfaces, not `env.vars`. The request is valid, but it changes how externally managed secrets enter runtime and durable service environment handling, so it needs maintainer/security design before implementation.

## What This Changes

Allow `openclaw.json` `env.vars` entries to use SecretRef/SecretInput values resolved at runtime from external secret providers instead of only static strings.

## Best Possible Solution

Extend the generic SecretInput/SecretRef pipeline for `env.vars` only after deciding the environment boundary: widen schema and public types, collect and resolve refs through the existing secrets runtime and target registry, preserve dangerous-env and unresolved-placeholder protections, avoid durable plaintext persistence unless explicitly approved, and update generated schema/docs plus focused config and secrets tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`: schema/type validation accepts only strings, and if raw JSON reaches runtime the collector skips non-string values.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable fix should extend the existing generic SecretInput resolver and registry surfaces rather than adding a 1Password-specific or exec-only path.
