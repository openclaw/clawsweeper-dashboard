## Summary

Keep open: current main and v2026.6.8 still treat top-level `env.vars` as a literal string-only config surface, while the shipped SecretRef plugin-provider seam only helps fields that already accept SecretRefs. The request remains useful but needs maintainer product/security direction because it changes config value shapes and where resolved secret material can flow.

## What This Changes

The issue requests allowing `openclaw.json` `env.vars` entries to use SecretRef objects instead of only static strings.

## Best Possible Solution

Define a focused SecretInput-compatible `env.vars` contract with explicit runtime versus managed-service-env semantics, then update schema, types, resolution, redaction, docs, and regression coverage together.

## Reproduction Assessment

Yes, by source inspection. Current main declares `env.vars` as string-only, skips non-string runtime values, and has regression coverage for that skip behavior.

## Solution Assessment

Unclear until maintainers choose the contract. Reusing the existing SecretInput/SecretRef pipeline is the maintainable direction, but the safe fix must also define service-env persistence and redaction behavior.
