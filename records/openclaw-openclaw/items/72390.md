## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface, while SecretRef support is limited to registered credential targets such as provider keys and plugin-scoped MCP env values; adding this needs maintainer/security policy because the same env collector feeds runtime and durable service environments.

## What This Changes

The issue requests SecretRef support for top-level `openclaw.json` `env.vars` values so environment variables can resolve dynamically from configured secret providers.

## Best Possible Solution

Approve and document a top-level `env.vars` SecretInput contract with explicit runtime versus durable service semantics, then update schema/types/generated docs/target registry/runtime and regression coverage together.

## Reproduction Assessment

Yes, from source inspection. Put a structured SecretRef object under `env.vars`: schema/types/generated schema accept only strings, and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable path is to reuse the existing SecretInput resolver, target registry, generated schema/docs pipeline, and activation tests rather than adding ad hoc env expansion.
