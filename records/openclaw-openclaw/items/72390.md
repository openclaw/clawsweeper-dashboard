## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment surface and does not collect or resolve structured SecretRef values there; the requested capability is valid but needs maintainer/security approval because it would feed external secrets into global runtime and service environment handling.

## What This Changes

Support SecretRef/SecretInput values in top-level `openclaw.json` `env.vars` so environment variables can be resolved at runtime from external secret providers such as 1Password.

## Best Possible Solution

Approve the `env.vars` SecretInput boundary first, then update schema/types/generated docs/target metadata and resolve through the existing SecretRef assignment pipeline with explicit runtime versus durable service-env semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`; schema/types require strings, and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The maintainable path is to reuse the existing SecretInput resolver, target registry, and config collectors rather than adding a provider-specific or exec-only shortcut.
