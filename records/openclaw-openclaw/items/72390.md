## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface, while SecretRef collection, target registry, and docs cover other credential paths such as ACPX MCP env but not the global env block. The request is valid, but it crosses a secrets-sensitive runtime/service environment boundary and should get maintainer/security design before implementation.

## What This Changes

Add SecretRef/SecretInput support for top-level `openclaw.json` `env.vars` so dynamically resolved external secrets can be supplied as runtime environment variables.

## Best Possible Solution

Extend the generic SecretInput/SecretRef pipeline for this core config surface: widen the approved `env.vars` contract, resolve refs before applying env values, keep resolved values runtime-only unless durable service behavior is explicitly approved, preserve dangerous-env and unresolved-placeholder protections, and update schema/types/generated docs, target registry metadata, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`: schema/type validation accepts only strings, and if raw JSON reaches runtime the env collector skips non-string values.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable solution should use the existing generic SecretInput resolver and registry surfaces rather than adding a 1Password-specific or exec-only path.
