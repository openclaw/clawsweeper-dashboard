## Label Justifications

- `impact:security`: The request changes how externally managed secrets may be resolved and materialized into OpenClaw environment handling.

## Summary

Keep open: current main and the latest release still treat top-level `env.vars` as string-only, while the related pluggable SecretRef provider work does not cover this field-widening request.

## What This Changes

The issue requests allowing `openclaw.json` `env.vars` values to be SecretRef objects so runtime environment variables can come from external secret providers.

## Best Possible Solution

Keep this open until maintainers define a SecretInput-compatible `env.vars` contract with explicit runtime and durable-service-env semantics, then update schema, resolution, docs, and focused regression coverage.

## Reproduction Assessment

Yes, via source inspection: schema/types accept top-level `env.vars` strings only, runtime collection skips non-string values, and tests lock in that skip behavior. This reproduces the missing capability, but it is a feature expansion rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the contract. Reusing the existing SecretInput resolution pipeline is plausible, but the durable service-env and plaintext materialization boundaries need an explicit security/product decision first.
