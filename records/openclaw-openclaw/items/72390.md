## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only config env surface, while adding SecretRef support would expand the secrets/config contract and needs maintainer security policy for runtime versus durable service env handling.

## What This Changes

Support SecretRef-style dynamic secret values in top-level `openclaw.json` `env.vars`.

## Best Possible Solution

Approve and document a top-level `env.vars` SecretInput contract with explicit runtime-only versus durable service env semantics, then update schema/types/generated docs/target registry/runtime coverage together.

## Reproduction Assessment

Yes, from source inspection. Put a structured SecretRef object under `env.vars`; schema and public types accept strings only, and `collectConfigRuntimeEnvVars` skips non-strings if raw JSON reaches runtime.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable path is to reuse the existing SecretInput resolver, target registry, generated schema/docs pipeline, and tests rather than adding ad hoc env expansion.
