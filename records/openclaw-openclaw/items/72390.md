## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface: schema/types reject SecretRef objects, runtime skips non-string values, and the documented SecretRef target surface does not include `env.vars`. This is a real feature request, not implemented-on-main, and it needs maintainer/security policy because these values also feed durable service environment collection.

## What This Changes

Add SecretRef/SecretInput support to top-level `openclaw.json` `env.vars` so external secret providers can supply environment variables at runtime.

## Best Possible Solution

Approve and document `env.vars` as a SecretInput-backed config surface only with explicit non-leakage semantics, then update the existing schema/types/target-registry/runtime resolver/docs/tests pipeline rather than adding ad hoc env expansion.

## Reproduction Assessment

Yes, from source inspection. Put a structured SecretRef object under `env.vars`: current schema/public types accept only strings, and the runtime collector skips non-string values if raw JSON reaches that path.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable solution is to reuse the existing SecretInput resolver and target registry, but maintainer/security policy should first decide where resolved top-level env secrets may flow.
