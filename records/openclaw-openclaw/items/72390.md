## Summary

Keep open. Current main still exposes top-level env.vars as a string-only environment injection surface, while SecretRef resolution is limited to registered credential targets such as model/provider credentials and ACPX MCP server env values. The request is valid, but it changes a global secrets-to-environment materialization boundary and should get maintainer/security design before implementation.

## What This Changes

The issue requests allowing structured SecretRef values in top-level openclaw.json env.vars so runtime environment variables can be resolved from configured secret providers.

## Best Possible Solution

Add env.vars to the generic SecretInput/SecretRef contract only after maintainer/security approval: widen schema/types/generated docs, register the target surface, resolve refs through the existing secrets runtime before env application, keep plaintext strings working, preserve dangerous-env filtering and existing process-env precedence, and decide explicitly whether service-env materialization is runtime-only or allowed for durable install metadata.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured SecretRef object under env.vars: schema/type validation accepts only strings, and if raw JSON reaches runtime the env collector skips non-string values.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable fix should reuse the existing SecretInput resolver and target-registry pipeline rather than adding a 1Password-specific or exec-only special case.
