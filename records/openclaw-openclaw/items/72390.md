## Summary

Keep open: current main still treats `env.vars` as string-only, and the documented SecretRef credential surface does not include it. The related work in https://github.com/openclaw/openclaw/issues/71593 and https://github.com/openclaw/openclaw/pull/72548 covers adjacent SecretRef provider/source architecture, but it explicitly leaves broader env/config field coverage as follow-up work.

## What This Changes

The issue requests SecretRef support in `openclaw.json` `env.vars` so runtime environment variables can be resolved from external secret providers such as 1Password.

## Best Possible Solution

Add `SecretInput` support for `env.vars` only after maintainer agreement on the activation boundary, with aligned type/schema/docs/target-registry coverage, blocked-key filtering, redaction, and exec-provider policy.

## Reproduction Assessment

Not applicable; this is a feature/API request, not a broken existing behavior report. Source inspection gives high confidence that current main does not support object-form SecretRefs in `env.vars`.

## Solution Assessment

Unclear; the requested direction is plausible, but resolving arbitrary env SecretRefs directly into process env needs a maintainer decision on security and runtime boundaries. The maintainable path should reuse the existing SecretInput/runtime snapshot model rather than adding an ad hoc resolver in config loading.
