## Summary

Keep open. Current main still treats top-level `env.vars` as string-only config and runtime/service env input, so structured SecretRef values are neither accepted nor resolved; the requested capability is valid but needs a maintainer security boundary decision before implementation.

## What This Changes

Request SecretRef support for top-level `openclaw.json` `env.vars` so external secret providers can populate environment variables at runtime.

## Best Possible Solution

Approve the `env.vars` SecretInput boundary first, then extend schema, public types, generated docs, target metadata, and runtime collectors through the existing SecretRef resolution pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`; schema/types require strings, and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The maintainable path is to reuse the existing SecretInput resolver and target registry rather than adding a provider-specific or exec-only shortcut.
