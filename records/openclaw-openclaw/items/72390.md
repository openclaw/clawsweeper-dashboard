- `impact:security`: The request changes how externally managed secrets may be resolved and materialized into OpenClaw environment handling.
- `impact:auth-provider`: The affected behavior is SecretRef resolution for credential-like environment variables used by providers, plugins, or skills.

## Summary

Keep open: current main and latest release still define top-level `env.vars` as string-only and runtime skips non-string values, while the related pluggable SecretRef provider PR explicitly defers widening env/config fields.

## What This Changes

Request support for structured SecretRef values inside the top-level `openclaw.json` `env.vars` block so environment variables can be resolved from configured secret providers such as exec-backed 1Password.

## Best Possible Solution

Define a focused SecretInput-compatible `env.vars` contract with explicit runtime and durable-service-env semantics, then update schema, resolution, docs, and regression tests together.

## Reproduction Assessment

Yes, via source inspection: schema/types accept top-level `env.vars` strings only, runtime collection skips non-string values, and tests lock in that skip behavior. This is a missing feature rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the contract. Reusing the existing SecretInput resolution pipeline is plausible, but the runtime versus durable service-env boundary needs explicit product and security review first.
