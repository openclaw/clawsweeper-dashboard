## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface, and the related SecretRef provider/plugin work does not add this field-widening; the change needs a maintainer security/product decision because these values also feed durable service environment collection.

## What This Changes

Request to allow top-level `openclaw.json` `env.vars` values to be SecretRef/SecretInput objects resolved at runtime, such as through 1Password.

## Best Possible Solution

Define a top-level `env.vars` SecretInput contract with explicit runtime-only versus durable-service-env semantics, then update schema/types/docs, runtime collection/resolution, and focused regression coverage.

## Reproduction Assessment

Yes for the current limitation via source inspection: schema/types accept strings only and the env collector skips non-string values. This is a feature expansion request, not broken existing behavior.

## Solution Assessment

Unclear because there is no patch yet. Reusing the existing SecretInput resolution pipeline is plausible, but maintainers first need to define where resolved env secrets may flow and whether they may persist.
