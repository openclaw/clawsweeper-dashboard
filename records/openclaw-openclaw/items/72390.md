## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface and does not collect or resolve SecretRef objects there; the request is valid, but it crosses a security-sensitive process/service environment boundary that should get maintainer design before implementation.

## What This Changes

The issue requests allowing `openclaw.json` top-level `env.vars` values to reference secrets dynamically, such as via an exec-backed 1Password provider.

## Best Possible Solution

Extend the generic SecretInput/SecretRef pipeline for this core config surface after maintainer/security design: widen schema and public types, register an explicit `env.vars.*` target if approved, resolve through the existing runtime secret resolver before env application, and update generated schema/docs, audit/target metadata, redaction, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`; current schema/types require strings, and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the narrow maintainable solution should reuse existing SecretInput resolver and target-registry contracts rather than adding a 1Password-specific or exec-only shortcut.
