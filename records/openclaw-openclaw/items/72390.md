## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only config/runtime surface, while SecretRef resolution is limited to registered credential paths; adding this needs maintainer/security approval because it changes how external secrets materialize into global runtime and durable service environments.

## What This Changes

The issue requests SecretRef support for top-level `openclaw.json` `env.vars` values so dynamic external secrets can populate environment variables at runtime.

## Best Possible Solution

Approve a top-level `env.vars` SecretInput contract, then update schema/types/generated docs/target metadata and resolve through the existing SecretRef assignment pipeline with explicit runtime-versus-durable service-env semantics.

## Reproduction Assessment

Yes. Put a structured `{source, provider, id}` SecretRef under `env.vars`; schema/types require strings, and the runtime collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable path is to reuse the existing SecretInput resolver, target registry, and config collectors after maintainers define persistence and redaction rules.
