- `impact:security`: The request changes how externally managed secrets may be resolved and materialized into OpenClaw environment handling.

## Summary

Keep open: current main and v2026.6.10 still treat top-level `env.vars` as a literal string-only config surface, while SecretRef resolution remains allowlisted to supported credential paths. The request is valid and source-reproducible, but it needs maintainer product/security direction because it changes how secrets materialize into runtime and service environments.

## What This Changes

Support SecretRef objects in the `openclaw.json` `env.vars` block so environment variables can be resolved from external secret providers such as 1Password-backed exec providers.

## Best Possible Solution

Define a focused SecretInput-compatible `env.vars` contract with explicit runtime-versus-service-env semantics, then update schema, types, resolution, redaction, docs, and regression coverage together.

## Reproduction Assessment

Yes, by source inspection: current main declares `env.vars` as string-only and skips non-string values at runtime, which matches the reported SecretRef object gap.

## Solution Assessment

Unclear until maintainers choose the contract; reusing the existing SecretInput/SecretRef pipeline is the likely maintainable direction, but the safe fix must define service-env persistence and redaction behavior.
