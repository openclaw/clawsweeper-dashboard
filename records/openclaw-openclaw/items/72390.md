## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface, while related SecretRef provider/plugin work leaves broader env/config field widening out of scope and this feature needs a security/product boundary before implementation.

## What This Changes

Request support for SecretRef object values in `openclaw.json` `env.vars` so external secret providers can populate environment variables at runtime.

## Best Possible Solution

Define a top-level `env.vars` SecretInput contract with explicit runtime-only versus durable-service-env semantics, then update schema/types/docs, runtime collection/resolution, and focused regression coverage.

## Reproduction Assessment

Yes for the current limitation via source inspection: schema/types accept strings only and the env collector skips non-string values. This is a feature expansion request, not broken existing behavior.

## Solution Assessment

Unclear because there is no patch yet. The maintainable direction is to reuse existing SecretInput resolution only after maintainers define where resolved env secrets may flow and whether they may persist.
