## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only config/service environment surface and does not collect or resolve SecretRef objects there; the issue remains valid, but the requested change crosses a security-sensitive global environment materialization boundary that needs maintainer/secops design before implementation.

## What This Changes

The issue requests SecretRef support for top-level `openclaw.json` `env.vars` so environment variables can be resolved dynamically from configured secret providers such as exec-backed 1Password at runtime.

## Best Possible Solution

Add an end-to-end SecretInput extension only after the environment boundary is approved: widen schema and public types, register an explicit `env.vars.*` SecretRef target if desired, collect and resolve through the existing runtime secret pipeline, define service-env persistence semantics, update generated schema/docs/audit metadata, and cover the behavior with focused config, secrets-runtime, registry, and daemon-environment tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`; current schema/types require strings, and the runtime env collector skips non-string values if raw JSON reaches it.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable fix should reuse the existing SecretInput resolver, target registry, and service-env policy rather than adding a provider-specific or exec-only shortcut.
