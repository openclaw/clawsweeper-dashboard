## Summary

Keep open. Current main still treats `env.vars` as a string-only env injection surface, while the related SecretRef-provider work explicitly leaves broader env/config field widening out of scope; this remains a security-sensitive feature request.

## What This Changes

Add SecretRef support for `openclaw.json` top-level `env.vars` so env values can resolve from configured secret providers at runtime.

## Best Possible Solution

Define a top-level `env.vars` SecretInput contract with explicit runtime-only versus durable-service-env semantics, then update schema/types/docs, runtime target collection, resolution, and regression coverage.

## Reproduction Assessment

Yes for the current limitation via source inspection: `env.vars` is typed and schematized as string-only, and runtime collectors skip non-string values. This is a requested expansion of supported SecretRef surfaces, not a broken existing contract.

## Solution Assessment

Unclear because there is no patch yet. The maintainable direction is to reuse the existing SecretInput resolver and target-collection model after maintainers define where resolved env secrets may flow and whether they may persist.
