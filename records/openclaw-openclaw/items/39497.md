## Summary

Keep this issue open. Current main still maps only the configured container workdir through the read-tool workspace-root guard, while the documented and mounted read-only agent workspace path is `/agent`; current regression coverage still expects `/agent/secret.txt` to be rejected under the workspace-only guard, so the reported behavior remains valid.

## What This Changes

The issue asks for sandboxed read-tool path resolution to accept the documented read-only agent workspace mount at `/agent/...` when `workspaceAccess` is `ro`.

## Best Possible Solution

Make the sandboxed read workspace-root guard validate through the sandbox mount table, or pass a narrowly scoped `/agent` mapping only when the active sandbox context has a real `workspaceAccess: "ro"` agent workspace mount. Add regression coverage for the guard helper and `createOpenClawCodingTools` proving `/agent/...` reads succeed while write/edit/apply_patch boundaries remain blocked.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction path: the read guard only maps `/workspace` via `containerWorkdir`, and the existing workspace-only mounted-path test still expects `/agent/secret.txt` to fail with `Path escapes sandbox root`.

## Solution Assessment

Partly. Fixing the read guard is the right direction, but the best implementation should be mount-aware or ro-scoped rather than an unconditional `/agent` rewrite.
