## Summary

Keep this open: current main still guards sandboxed `read` with only `containerWorkdir` remapping, while the documented read-only workspace mount is `/agent` and existing coverage still expects `/agent` reads to be rejected under `tools.fs.workspaceOnly`.

## What This Changes

Allow sandboxed `read` calls in `workspaceAccess: "ro"` sessions to resolve valid `/agent/...` container paths back to the read-only agent workspace instead of rejecting them as escaping the sandbox.

## Best Possible Solution

Make the read workspace-root guard mount-aware, or pass a ro-scoped `/agent` mapping only when the active sandbox context has the real read-only agent workspace mount, while preserving write/edit/apply_patch and arbitrary bind rejections.

## Reproduction Assessment

Yes, from source inspection. With `tools.fs.workspaceOnly` enabled, sandboxed `read` is guarded using only `containerWorkdir`, so `/agent/...` reaches `assertSandboxPath` unchanged and current coverage still expects that rejection.

## Solution Assessment

No, not as an unconditional `/agent` rewrite. The safer path is mount-aware or ro-scoped validation that proves `/agent` is the active read-only workspace mount and keeps other sandbox escape protections intact.
