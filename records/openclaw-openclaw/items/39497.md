## Summary

Keep this open: current main still remaps only the configured container workdir before the workspace-only read guard, while the sandbox docs and mount resolver define `/agent` as the read-only agent workspace mount.

## What This Changes

The issue requests that sandboxed `read` workspace-root validation accept the read-only `/agent` workspace mount when `workspaceAccess: "ro"` instead of rejecting it as outside the sandbox root.

## Best Possible Solution

Make the read workspace-root guard mount-aware, or pass a narrowly scoped read-only `/agent` mapping from the active sandbox context, while leaving `/tmp` and arbitrary bind reads as separate policy decisions.

## Reproduction Assessment

Yes, from source inspection rather than a runtime test. The current guard maps only `containerWorkdir` before `assertSandboxPath`, and current workspace-only sandbox coverage still asserts `/agent` reads are rejected.

## Solution Assessment

No, not as an unconditional `/agent` rewrite. The maintainable fix is a mount-aware or ro-scoped validation path that proves the active sandbox mount and preserves write/edit/apply_patch and arbitrary bind protections.
