## Summary

Keep this open: current main still routes sandboxed read paths through a guard that only remaps the configured container workdir, while the sandbox mount table and docs define `/agent` as the read-only agent workspace mount.

## What This Changes

Allow the read tool to resolve `/agent/...` paths in `workspaceAccess: "ro"` sandboxed sessions while preserving read-only workspace isolation.

## Best Possible Solution

Make the read workspace-root guard mount-aware, or pass a narrowly scoped read-only `/agent` mapping from the active sandbox context, while leaving `/tmp` and arbitrary bind reads as separate policy decisions.

## Reproduction Assessment

Yes. Source inspection is high-confidence: `/agent` passes through the guard unmodified, current coverage still asserts the rejection, and I did not need a live sandbox run to identify the failing path.

## Solution Assessment

No, not as an unconditional `/agent` rewrite. The maintainable fix should be mount-aware or explicitly scoped to `workspaceAccess: "ro"` so read-only workspace access is restored without broadening writes, apply_patch, `/tmp`, or arbitrary bind reads.
