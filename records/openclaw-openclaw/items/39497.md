## Summary

Keep this open: current main still guards sandboxed read paths by remapping only the configured container workdir, while the read-only agent workspace is mounted at `/agent` and existing workspace-only coverage still rejects `/agent` paths.

## What This Changes

Allow sandboxed read-tool calls in `workspaceAccess: "ro"` sessions to resolve `/agent/...` paths against the read-only agent workspace instead of failing the workspace-root guard.

## Best Possible Solution

Make the read workspace-root guard mount-aware, or pass a narrowly scoped `/agent` mapping only when the active sandbox context has a real read-only agent workspace mount, while preserving write/edit/apply_patch and arbitrary bind rejections.

## Reproduction Assessment

Yes, from source inspection. With `tools.fs.workspaceOnly` enabled, sandboxed `read` is guarded with only `containerWorkdir`, so `/agent/...` is not remapped before `assertSandboxPath`; I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not as an unconditional `/agent` rewrite. The safer solution is mount-aware or ro-scoped validation that proves `/agent` is the active read-only workspace mount and keeps other sandbox escape protections intact.
