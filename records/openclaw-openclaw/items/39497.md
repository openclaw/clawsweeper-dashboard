## Summary

Keep this open: current main still only remaps the configured container workdir before the read-tool workspace-root guard, while the read-only agent workspace is documented and mounted at `/agent`; existing coverage still expects `/agent` to be rejected under the workspace-only guard.

## What This Changes

Allow sandboxed `read` calls using `/agent/...` paths to succeed for `workspaceAccess: "ro"` sessions without widening write access.

## Best Possible Solution

Make the read workspace-root guard mount-aware, or pass a narrowly scoped `/agent` mapping only when the active sandbox context has a real read-only agent workspace mount, while preserving write/edit/apply_patch and arbitrary bind rejections.

## Reproduction Assessment

Yes, from source inspection. With `tools.fs.workspaceOnly` enabled, sandboxed `read` is guarded with only `containerWorkdir`, so `/agent/...` reaches `assertSandboxPath` unchanged and current coverage still expects that rejection.

## Solution Assessment

No, not as an unconditional `/agent` rewrite. The safer solution is mount-aware or ro-scoped validation that proves the `/agent` path is the active read-only workspace mount and keeps other sandbox escape protections intact.
