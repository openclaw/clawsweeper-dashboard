## Summary

Keep this open: current main still leaves the sandboxed read-tool workspace-root guard unaware of the read-only `/agent` workspace mount, and existing coverage still expects `/agent` reads to be rejected when `tools.fs.workspaceOnly` is enabled.

## What This Changes

Allow sandboxed read-tool calls to resolve `/agent/...` when `workspaceAccess: "ro"` instead of rejecting them as outside the workspace root.

## Best Possible Solution

Make the read workspace-root guard mount-aware, or add a narrowly scoped `/agent` mapping only when the active sandbox context proves it is the read-only agent workspace mount, while preserving write/edit/apply_patch and arbitrary bind-mount rejections.

## Reproduction Assessment

Yes, from source inspection. With sandboxed `read` plus `tools.fs.workspaceOnly=true`, `/agent/...` is not remapped by the guard, so `assertSandboxPath` compares it against the sandbox workspace root and rejects it before the sandbox FS bridge can resolve the managed `/agent` mount.

## Solution Assessment

No, not as an unconditional `/agent` rewrite. The maintainable fix should be mount-aware or explicitly ro-scoped so it only allows the real agent workspace mount and keeps the rest of the sandbox escape protection intact.
