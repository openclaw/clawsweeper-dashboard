## Summary

Keep open: current main still validates sandboxed `read` paths after only remapping the configured container workdir, while the sandbox mount table and docs define `/agent` as the read-only agent workspace mount.

## What This Changes

Allow sandboxed `read /agent/...` calls in `workspaceAccess: "ro"` sessions to resolve to the read-only agent workspace instead of failing the workspace-root guard.

## Best Possible Solution

Make the read workspace-root guard mount-aware, or pass a narrowly scoped active `/agent` mapping from the sandbox context, while leaving `/tmp` and arbitrary bind reads as separate policy decisions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `/agent/...` is not remapped by the guard, current test coverage still asserts rejection, and the docs/mount table define `/agent` as the read-only workspace mount.

## Solution Assessment

No, not as an unconditional `/agent` rewrite. The maintainable fix should be mount-aware or explicitly scoped to `workspaceAccess: "ro"` so read-only access is restored without broadening write/edit/apply_patch, `/tmp`, or bind-mount access.
