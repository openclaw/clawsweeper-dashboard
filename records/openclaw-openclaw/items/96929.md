## Summary

Keep open: current main and the latest release still have a source-backed Feishu SecretRef activation bug where the top-level default account is preserved for startup, but the secrets collector can mark its top-level appSecret inactive when all named accounts override their own appSecret.

## What This Changes

The issue reports that a top-level Feishu appSecret SecretRef is skipped when configured named Feishu accounts have their own appSecret values, causing the default Feishu account to fail startup.

## Best Possible Solution

Align Feishu SecretRef activation with Feishu account listing so the implicit top-level default account keeps channels.feishu.appSecret active, without adding config or weakening inactive-surface filtering for other channels.

## Reproduction Assessment

Yes, at source level: current main preserves an implicit Feishu default account when top-level credentials are configured, but the secrets collector marks that top-level appSecret inactive when all explicit named accounts override appSecret. I did not run a live gateway startup in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to make the Feishu secret contract recognize the active implicit default account for top-level appSecret resolution, while preserving strict runtime SecretRef behavior and inactive-surface warnings elsewhere.
