## What This Changes

The issue reports that top-level Feishu `channels.feishu.appSecret` SecretRefs are skipped as inactive when every named Feishu account overrides `appSecret`.

## Best Possible Solution

Review and land a narrow Feishu-scoped fix, preferably preserving shared inactive-surface semantics, then close this issue after the merged PR proves the default account startup path.

## Reproduction Assessment

Yes, at source level: Feishu still creates an implicit default account from top-level credentials while the shared collector can mark that same top-level `appSecret` inactive when every explicit account overrides it. I did not run a live gateway startup in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow Feishu appSecret activation change that mirrors Feishu's implicit default account while preserving shared inactive-surface behavior for sibling channels. Closing this issue before one of the linked PRs merges would be premature.
