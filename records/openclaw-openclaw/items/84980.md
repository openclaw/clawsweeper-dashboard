## Summary

Close: current main and the latest release still expose `openclaw/plugin-sdk`, and the documented/source contract scopes SDK peer-link repair to managed plugin install flows while local TypeScript plugin directories own their dependency and link setup.

## What This Changes

The issue reports that an external local TypeScript plugin using `openclaw/plugin-sdk` no longer loads on OpenClaw 5.x because Node cannot resolve the SDK package export from the plugin directory.

## Best Possible Solution

Keep the current managed-install peer-link contract and have this plugin use `openclaw plugins install`/`openclaw doctor --fix` for managed installs or maintain its own local `openclaw` peer dependency link when loaded as a developer-controlled local plugin.

## Reproduction Assessment

Yes, at source-contract level: a local plugin directory without a plugin-local `node_modules/openclaw` link is outside the managed npm repair path, so Node resolution can fail for `openclaw/plugin-sdk`. I did not execute the private plugin package, but the reported install shape matches the documented boundary.

## Solution Assessment

Yes. The maintainable solution is to keep SDK exports plus managed peer-link repair for supported install flows, while local plugins install or link their own dependencies; repairing arbitrary local plugin directories would be a new product boundary.
