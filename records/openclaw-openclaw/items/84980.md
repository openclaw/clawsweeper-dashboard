## Summary

This is a support/setup issue for an externally managed local plugin, not an OpenClaw source change: the SDK export still exists, managed installs have a peer-link repair path, and local plugins are documented as owning their own dependency/link setup.

## What This Changes

The issue reports that a locally loaded raw TypeScript plugin importing `openclaw/plugin-sdk` fails on OpenClaw 5.x because Node cannot resolve the host SDK package from the plugin install location.

## Best Possible Solution

Keep the current managed-install peer-link contract and have this plugin use a supported install flow or maintain its own local `openclaw` peer link/dependency setup.

## Reproduction Assessment

Yes, at source-contract level: a local plugin directory without a plugin-local `node_modules/openclaw` link is outside the managed npm repair path, so Node resolution can fail for `openclaw/plugin-sdk`. I did not run the private plugin package, but the report matches the documented local-plugin dependency boundary.

## Solution Assessment

Yes. The maintainable solution is to keep SDK exports plus managed peer-link repair for installed packages, while local plugins either install/link their own dependencies or use `openclaw plugins install`; repairing arbitrary local plugin directories would be a new product boundary.
