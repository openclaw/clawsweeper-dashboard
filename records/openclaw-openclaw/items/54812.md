## Summary

Keep open. Current main and the latest release still authorize channel-originated config writes only at channel/account scope, so the requested per-DM/group/context permission model remains a valid security-sensitive feature/design item.

## What This Changes

The issue requests per-context `configWrites` permissions so trusted DMs or admin groups can write config while client groups on the same channel cannot.

## Best Possible Solution

Add a generic context-scoped config-write policy model in the shared channel/plugin contract, then let each multi-context channel map DM, group, topic, or thread identities into that policy with docs and regression tests.

## Reproduction Assessment

Yes for the missing-capability claim: source inspection shows the authorization scope, command gate, schemas, docs, and tests only support channel/account `configWrites`. No live channel run is needed to prove that no per-DM/group override surface exists.

## Solution Assessment

Unclear as proposed. The security need is valid, but the exact config shape should be chosen by maintainers because it adds cross-channel policy, config, and SDK surface.
