## Summary

Keep open. Current main still authorizes config writes at channel/account scope only, and the requested conversation-level DM/group/topic policy remains absent from the policy model, command gates, bundled channel schemas, docs, and tests.

## What This Changes

The issue requests context-scoped config-write permissions so DMs, groups, topics, or conversations can grant or deny config changes independently of the whole channel/account.

## Best Possible Solution

Add a backward-compatible generic context-scoped config-write policy with normalized conversation identity, inherited defaults, SDK helpers, schema/docs coverage, and contract tests for allow, deny, and inheritance.

## Reproduction Assessment

Yes, by source inspection. Current main accepts and evaluates configWrites only at channel/account scope, so a per-DM/group/topic override cannot be configured or authorized.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and policy contract rather than one-off dmConfigWrites and channel-specific group fields.
