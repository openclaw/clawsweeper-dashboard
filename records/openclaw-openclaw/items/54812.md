## Summary

Keep open. Current main and the latest release still model config-write permission at channel/account scope only, so the requested per-DM/group/conversation policy remains a real cross-cutting config and authorization design gap rather than a stale or implemented item.

## What This Changes

Add per-context `configWrites` permissions so DMs, groups, topics, or conversations can allow or deny config writes independently within the same channel.

## Best Possible Solution

Add a backward-compatible generic config-write policy model that carries normalized conversation identity through channel/plugin boundaries, defines inheritance rules, updates schemas/docs, and adds contract tests for allow, deny, and inheritance cases.

## Reproduction Assessment

Yes, by source inspection. Current schemas, command gates, and authorization helpers only accept channel/account scope, so a per-DM/group/topic override cannot be configured or evaluated.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and policy contract rather than one-off `dmConfigWrites` and repeated `groups.*.configWrites` fields per channel.
