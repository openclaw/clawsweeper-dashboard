## Summary

Keep open. Current main and the latest release still gate channel-initiated config writes at channel/account scope only, while DM/group context schemas and docs do not expose per-context configWrites; the request remains a valid security-sensitive feature/design item rather than stale cleanup.

## What This Changes

The issue requests per-context config-write permissions so a multi-context channel can allow config writes in admin or DM contexts while denying them in client groups.

## Best Possible Solution

Define one generic context-scoped config-write policy model, including normalized conversation identity, inheritance behavior, SDK helpers, schemas, docs, and regression tests across bundled multi-context channels.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the gap: the current policy, command gates, schemas, docs, and tests expose config-write authorization at channel/account level only. I did not run a live channel scenario because the missing config and authorization surface is visible in current main and v2026.6.6 source.

## Solution Assessment

Unclear. The requested direction is valid, but the exact proposed config shape is not proven as the best permanent solution until maintainers choose the generic context identity and inheritance semantics.
