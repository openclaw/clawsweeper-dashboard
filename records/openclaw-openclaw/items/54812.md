## Summary

Keep open. Current main still resolves config-write authorization at channel/account scope only, while schemas, docs, and contract tests expose only channel/account gates; the requested per-conversation policy remains valid and is not superseded by the provided related items.

## What This Changes

Request to add per-context config-write permissions for DMs, groups, topics, threads, or equivalent conversations across multi-context channels.

## Best Possible Solution

Add a backward-compatible generic config-write policy model that carries normalized conversation identity through channel/plugin boundaries, defines inheritance from channel/account defaults into context overrides, updates bundled multi-context schemas and docs, and adds contract tests for allow, deny, and inheritance behavior.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the strict context schemas do not accept per-context configWrites, and the /config and /allowlist authorization paths pass only channel/account scope into the shared policy.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable solution should be a generic core/SDK policy and context identity model rather than a channel-by-channel literal dmConfigWrites/groups shape.
