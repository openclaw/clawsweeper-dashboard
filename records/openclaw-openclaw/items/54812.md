## Summary

Keep open. Current main still has only channel/account-scoped config-write authorization, so the requested DM/group/topic/conversation-scoped policy remains a real feature and security-design gap rather than stale cleanup.

## What This Changes

Add per-context configWrites permissions so DMs, groups, topics, or conversations can override channel/account defaults across channel types.

## Best Possible Solution

Define a backward-compatible generic context-scoped config-write policy with normalized conversation identity, inheritance rules, SDK helpers, schemas, docs, and contract tests.

## Reproduction Assessment

Yes, by source inspection. Current main accepts and evaluates configWrites only at channel/account scope, and the command gates, schemas, and docs have no per-context override path.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and inheritance contract rather than one-off dmConfigWrites and channel-specific group fields.
