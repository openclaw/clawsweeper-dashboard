## Summary

Keep open. Current main still models config-write permission at channel/account scope only, and the requested per-DM/group/topic conversation policy is absent from the authorization model, command gates, bundled channel schemas, docs, and tests.

## What This Changes

Request per-context config-write permissions so multi-context channels can allow trusted DMs or admin groups while denying client groups under the same channel/account.

## Best Possible Solution

Add a backward-compatible generic context-scoped config-write policy with normalized conversation identity, inherited defaults, SDK helpers, schema/docs coverage, and contract tests for allow, deny, and inheritance cases.

## Reproduction Assessment

Yes, by source inspection. Current main accepts and evaluates configWrites only at channel/account scope, so a per-DM/group/topic override cannot be configured or authorized.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and policy contract rather than one-off dmConfigWrites and channel-specific group fields.
