## Summary

Keep open; current main and the latest shipped release still model config-write authorization only at channel/account scope, so the requested per-conversation policy remains absent and needs maintainer product/security design.

## What This Changes

Request a per-context `configWrites` policy so DMs, groups, topics, threads, or conversations can allow or deny config writes independently under the same channel/account.

## Best Possible Solution

Define a backward-compatible generic context-scoped config-write policy with normalized conversation identity at the channel/plugin boundary, inheritance from channel/account defaults, SDK helpers, schema/docs updates, and cross-channel contract tests.

## Reproduction Assessment

Yes, by source inspection. Current main and `v2026.5.20` accept and evaluate `configWrites` only at channel/account scope, and the command gates, schemas, docs, and tests have no per-context override path.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable solution is a generic normalized context identity and inheritance contract rather than one-off channel-specific keys like `dmConfigWrites`.
