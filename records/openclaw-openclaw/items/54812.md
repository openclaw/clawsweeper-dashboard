## Summary

Keep open; current main and v2026.5.20 still implement `configWrites` only at channel/account scope, so the requested per-DM/group/topic/conversation policy remains absent and needs product/security/API design before implementation.

## What This Changes

Request a generic per-context `configWrites` policy so DMs, groups, topics, and similar channel contexts can allow or deny config writes independently of channel/account defaults.

## Best Possible Solution

Define a backward-compatible generic config-write policy model with normalized conversation identity, clear inheritance rules, SDK helpers, schema/docs updates, and cross-channel contract tests.

## Reproduction Assessment

Yes, by source inspection. Current main and v2026.5.20 accept and evaluate `configWrites` only at channel/account scope, while command gates, schemas, docs, and tests have no per-context override path.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and inheritance contract rather than one-off channel-specific keys.
