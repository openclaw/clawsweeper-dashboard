## Summary

Keep open. Current main still models config-write permission at channel/account scope only, so the requested per-DM/group/topic/conversation policy remains absent from authorization, command gates, schemas, docs, and tests.

## What This Changes

The issue requests per-context config-write permissions so trusted DMs or admin groups can edit configuration while other conversations on the same channel cannot.

## Best Possible Solution

Add a backward-compatible generic context-scoped config-write policy with normalized conversation identity, inherited defaults, SDK helpers, schemas/docs, and contract tests for allow, deny, and inheritance behavior.

## Reproduction Assessment

Yes, by source inspection. Current main accepts and evaluates configWrites only at channel/account scope, so a per-DM/group/topic override cannot be configured or authorized.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and policy contract rather than one-off dmConfigWrites and channel-specific group fields.
