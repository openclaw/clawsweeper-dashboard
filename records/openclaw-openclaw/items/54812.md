## Summary

Keep open. Current main still models config-write permission at channel/account scope only, so the requested per-DM/group/topic/conversation policy remains absent from authorization, command gates, schemas, docs, and tests.

## What This Changes

Request per-DM/group/conversation `configWrites` overrides across multi-context channels instead of only channel/account-level permissions.

## Best Possible Solution

Add a backward-compatible generic context-scoped config-write policy with normalized conversation identity, SDK support, docs, schemas, and contract tests for allow, deny, and inheritance.

Yes, by source inspection. Current main accepts and evaluates `configWrites` only at channel/account scope, so the requested per-DM/group/topic override cannot be configured or authorized.

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and policy contract rather than one-off `dmConfigWrites` and channel-specific group fields.
