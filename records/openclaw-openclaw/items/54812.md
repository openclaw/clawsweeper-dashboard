## Summary

Keep open. Current main still models `configWrites` at channel/account scope only, so per-DM/group/topic/conversation config-write policy remains a real cross-channel access-control design gap rather than implemented or stale.

## What This Changes

Request per-DM, per-group, and per-conversation `configWrites` overrides across channel types instead of only channel/account-level permission.

## Best Possible Solution

Add a backward-compatible generic config-write policy model that carries normalized conversation identity through channel/plugin boundaries, defines inheritance, updates SDK/config schemas/docs, and adds contract tests for allow, deny, and inheritance cases.

## Reproduction Assessment

Yes, by source inspection. Current schemas, command gates, and authorization helpers only accept channel/account scope, so a per-DM/group/topic override cannot be configured or evaluated on current main.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and policy contract rather than one-off `dmConfigWrites` plus repeated channel-specific group fields.
