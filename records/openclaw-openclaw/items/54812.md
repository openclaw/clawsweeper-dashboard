## Summary

Keep open. Current main and the latest release still model channel-originated config writes at channel/account scope only, so the requested per-DM/group/topic/conversation permission model remains a valid security-sensitive product/API design item.

## What This Changes

The issue requests per-DM, group, topic, or conversation `configWrites` authorization instead of only channel/account-level config write permissions.

## Best Possible Solution

Add a generic context-scoped config-write policy model in the shared channel/plugin contract, then let each multi-context channel map DM, group, topic, or thread identities into that policy with docs and regression tests.

## Reproduction Assessment

Yes for the missing-capability claim: current source, schemas, docs, and tests only support channel/account `configWrites`. No live channel run is needed to prove that no per-DM/group/topic override surface exists.

## Solution Assessment

Unclear as proposed. The security need is valid, but the exact cross-channel config shape, inheritance rules, and SDK contract need maintainer design before implementation.
