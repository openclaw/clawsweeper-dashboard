## Summary

Keep open; current main and v2026.5.22 still authorize config writes only at channel/account scope, while the requested DM/group/topic/conversation overrides are absent and need product/security design.

## What This Changes

The issue requests per-context `configWrites` permissions so multi-context channels can allow config writes in trusted DMs or admin groups while denying them in client groups.

## Best Possible Solution

Define a backward-compatible generic context-scoped config-write policy with normalized conversation identity at the channel/plugin boundary, inheritance from channel/account defaults, narrow SDK helpers, schema/docs updates, and cross-channel contract tests.

## Reproduction Assessment

Yes, by source inspection. Current main and v2026.5.22 evaluate `configWrites` only at channel/account scope, and the command gates, schemas, docs, and tests have no per-context override path.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable solution is a generic normalized context identity and inheritance contract rather than one-off keys like `dmConfigWrites`.
