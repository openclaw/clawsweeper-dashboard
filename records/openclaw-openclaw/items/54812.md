Keep open because current main and the latest release still scope config-write authorization only to channel/account, while the request is a valid cross-channel security feature that needs maintainer design before implementation.

## What This Changes

The issue asks for `configWrites` overrides scoped to individual DMs, groups, topics, or conversations across multi-context channels.

## Best Possible Solution

Define a backward-compatible generic context-scoped config-write policy with normalized conversation identity, inheritance rules, SDK helpers, schemas, docs, and contract tests.

## Reproduction Assessment

Yes, by source inspection. Current main accepts and evaluates `configWrites` only at channel/account scope, and the command gates, schemas, docs, and tests have no per-context override path.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and inheritance contract rather than one-off channel-specific keys.
