## Summary

Keep open. Current main still models `configWrites` at channel/account scope only, so the requested per-DM/group/topic/conversation policy is absent; the related open issue and PR cover a narrower `/allowlist --store` bypass, not this per-context feature.

## What This Changes

Request per-context `configWrites` permissions so multi-context channels can allow config edits in admin DMs/groups while denying them in client groups or other conversations.

## Best Possible Solution

Add a backward-compatible generic config-write policy model that carries normalized conversation identity through channel/plugin boundaries, defines inheritance rules, updates schemas/docs, and adds contract tests for allow, deny, and inheritance cases.

## Reproduction Assessment

Yes, for the reported gap by source inspection. Current schemas, command gates, and authorization helpers only accept channel/account scope, so a per-DM/group/topic override cannot be configured or evaluated.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable fix should be a generic context identity and policy contract rather than one-off `dmConfigWrites`/`groups` fields repeated per channel.
