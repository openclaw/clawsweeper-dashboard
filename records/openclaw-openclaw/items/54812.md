## Summary

Keep open. Current main still authorizes config writes only at channel/account scope, so the requested per-DM/group/topic/conversation policy remains absent from the authorization model, command gates, schemas, docs, and tests.

## What This Changes

The issue requests per-context configWrites permissions so trusted DMs or admin groups can write config while less-trusted groups on the same channel cannot.

## Best Possible Solution

Define a backward-compatible generic context-scoped config-write policy with normalized conversation identity, inheritance rules, SDK helpers, schemas, docs, and contract tests.

## Reproduction Assessment

Yes, by source inspection. Current main accepts and evaluates configWrites only at channel/account scope, and the command gates, schemas, docs, and tests have no per-context override path.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and inheritance contract rather than one-off channel-specific fields.
