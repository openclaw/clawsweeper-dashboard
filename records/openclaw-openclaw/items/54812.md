## Summary

Keep open; current main and v2026.5.20 still authorize config writes at channel/account scope only, so the requested per-DM/group/topic/conversation policy remains absent and needs maintainer product/security design.

## What This Changes

Add per-conversation configWrites permissions so trusted DMs or admin groups can write config while client groups or other contexts under the same channel/account cannot.

## Best Possible Solution

Define a backward-compatible generic context-scoped config-write policy with normalized conversation identity at the channel/plugin boundary, inheritance from channel/account defaults, narrow SDK helpers, schema/docs updates, and cross-channel contract tests.

## Reproduction Assessment

Yes, by source inspection. Current main and v2026.5.20 accept and evaluate configWrites only at channel/account scope, and the command gates, schemas, docs, and tests have no per-context override path.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable solution is a generic normalized context identity and inheritance contract rather than one-off keys like dmConfigWrites.
