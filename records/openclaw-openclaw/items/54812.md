## Summary

Keep open. Current main and the latest release still authorize config writes at channel/account scope only, so the requested per-DM/group/topic/conversation policy remains valid but needs maintainer product and security design before implementation.

## What This Changes

The issue requests per-DM, per-group, or per-conversation `configWrites` authorization overrides across channel plugins instead of only channel/account-level settings.

## Best Possible Solution

Adopt a backward-compatible generic context-scoped config-write policy with normalized conversation identity, inheritance rules, SDK helpers, schemas, docs, and contract tests.

## Reproduction Assessment

Yes, by source inspection. Current main and `v2026.5.18` carry only channel/account IDs through config-write authorization, and the schemas/docs do not expose context-scoped overrides.

## Solution Assessment

Unclear for the exact proposed field names. The maintainable direction is a generic normalized context identity and inheritance contract rather than one-off `dmConfigWrites` or channel-specific group fields.
