## Summary

Keep open. Current main and v2026.6.10 still gate channel-initiated config writes by channel/account only; the requested per-DM/group/context permission model remains a security-sensitive config/API design item that needs maintainer product direction rather than cleanup closure.

## What This Changes

Request per-DM/group/conversation config-write permissions so multi-context channel deployments can allow config writes in admin contexts while denying them elsewhere.

## Best Possible Solution

Define one generic context-scoped config-write policy model before implementation, then update core authorization, SDK helpers, bundled channel schemas, docs, and contract tests together.

## Reproduction Assessment

Yes. Source inspection shows the requested per-context permission surface is missing: config-write scopes and targets carry channel/account only.

## Solution Assessment

No. The requested outcome is valid, but the proposed exact keys are only one possible shape; maintainers should first choose the normalized context identity and inheritance model for all channel plugins.
