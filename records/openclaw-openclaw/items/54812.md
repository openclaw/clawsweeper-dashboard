## Summary

Keep open: current main and the latest release still authorize channel-originated config writes only by channel/account, while this request needs a security-sensitive per-conversation policy design across channel runtime, SDK, schemas, docs, and tests.

## What This Changes

Request per-context `configWrites` permissions so operators can allow config writes in trusted DMs or admin groups while denying them in client/business groups on multi-context channels.

## Best Possible Solution

Design a generic config-write policy model at the channel/plugin boundary: carry normalized conversation identity through origins and targets, define inheritance from channel/account defaults into DM/group/topic/thread overrides, expose narrow SDK helpers, update bundled schemas/docs, and cover allow, deny, and inheritance cases with contract tests.

## Reproduction Assessment

Yes, at source level. Current main has no conversation identity in the config-write policy inputs, write gates, context schemas, or docs, so the requested per-DM/group/topic override behavior cannot be configured today.

## Solution Assessment

Unclear as proposed. The nested example captures the need, but the maintainable solution should be a generic policy/SDK design rather than one-off per-channel keys added independently.
