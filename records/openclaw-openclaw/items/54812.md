## Summary

Keep open. Current main and the latest release still model channel-originated config-write permission only at channel/account scope, while the requested per-DM/group/topic policy would add security-sensitive public config and SDK semantics that need maintainer design.

## What This Changes

The issue requests per-DM/group/topic `configWrites` authorization instead of the current channel/account-only setting.

## Best Possible Solution

Add a generic context-scoped config-write policy at the shared channel/plugin contract, then let each multi-context channel map DM, group, topic, or thread identity into that policy with docs, schema coverage, and regression tests.

## Reproduction Assessment

Yes for the missing-capability claim: current source, schema, docs, and tests expose channel/account `configWrites` only, not per-DM/group/topic policy. No live channel run is needed to prove the config surface is absent.

## Solution Assessment

Unclear as proposed. The security need is valid, but the exact cross-channel config shape, inheritance rules, deny precedence, migration behavior, and SDK contract need maintainer design before implementation.
