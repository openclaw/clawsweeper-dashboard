## Summary

Keep open. Current main still authorizes config writes only at channel/account scope and does not model originating DM/group/topic/thread identity, so the requested per-context permission remains a valid security-sensitive feature/design item rather than a stale, duplicate, or implemented report.

## What This Changes

Request per-context `configWrites` controls so multi-context channels can allow config edits in operator/admin contexts while denying them in client groups or other conversations.

## Best Possible Solution

Add a backward-compatible generic config-write policy model that carries normalized conversation identity through channel/plugin boundaries, defines channel/account/context inheritance, updates bundled multi-context schemas and docs, and adds contract tests for allow, deny, and inheritance behavior.

## Reproduction Assessment

Yes, by source inspection. The strict config schemas reject per-context `configWrites`, and the `/config` and `/allowlist --config` authorization paths pass only channel/account scope into the shared policy.

## Solution Assessment

Unclear for the exact proposed field names. The durable fix should be a generic core/SDK policy and normalized context identity model rather than a literal channel-by-channel `dmConfigWrites`/`groups` shape.
