## Summary

Keep open. Current main and v2026.6.10 still authorize channel-originated config writes only by channel/account, so per-DM/group/topic policy remains missing and needs maintainer security/product/API design rather than conservative cleanup.

## What This Changes

Request adding per-conversation configWrites permissions across multi-context channels so DMs, admin groups, and client groups can have different write access.

## Best Possible Solution

Add a generic context-scoped config-write policy model in the shared channel/plugin contract, then let each multi-context channel map DM, group, topic, or thread identity into that policy with docs and regression tests.

## Reproduction Assessment

Yes for the missing-capability claim: current source, schemas, docs, and tests only support channel/account configWrites. No live channel run is needed to prove that no per-DM/group/topic override surface exists.

## Solution Assessment

Unclear as proposed. The security need is valid, but the exact cross-channel config shape, inheritance rules, and SDK contract need maintainer design before implementation.
