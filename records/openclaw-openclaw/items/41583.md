## Summary

Keep open. Current main still unconditionally builds both the system-level inbound metadata prompt and user-role metadata blocks, while shared, Telegram, and Discord config surfaces expose no disable/minimize mode; related PR #41299 is an open separator/stripping proposal and explicitly leaves API/config changes out of scope.

## What This Changes

Request to add a config option to disable or minimize OpenClaw inbound channel message metadata injection for trusted channels or senders, especially Telegram/Discord DMs.

## Best Possible Solution

Add a documented shared reply-runtime metadata mode that defaults to current metadata-on behavior, supports narrow per-channel/account or trusted-DM scoping, applies consistently to both system metadata and user-role metadata prefixes, and includes regression coverage for the default-on path plus the opt-in minimized/disabled path.

## Reproduction Assessment

Yes for the current behavior: a direct external-channel context with channel/sender fields reaches buildInboundUserContextPrefix and emits Conversation info/Sender metadata, and runPreparedReply unconditionally prepends both system and user-role metadata. This is a feature request rather than a crash-style bug, so the reproduction is source/test inspection against current main.

## Solution Assessment

No current solution exists in main, and a simple global off switch would not be the best maintainable fix because it would bypass provenance metadata too broadly. The safer direction is a default-on, scoped, documented mode with security review and tests.
