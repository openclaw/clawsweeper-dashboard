## Summary

Keep this PR open for the automerge/repair path. The feature direction matches the linked WebChat server-side STT request, but the PR introduces one small recorder error-path bug that should be fixed before a clean re-review.

## What This Changes

Adds a `chat.transcribeAudio` Gateway RPC, WebChat composer dictation control, tests, docs, and changelog coverage for server-side STT.

## Best Possible Solution

Land the server-side dictation path after the small MediaRecorder error handling fix and exact-head checks, keeping STT on the existing media-understanding pipeline rather than adding a separate transcription stack.

## Reproduction Assessment

Yes for the review finding: start recording and trigger a MediaRecorder error after data collection begins; the spec then fires `dataavailable` and `stop`, and the PR stop handler can still submit the collected blob. For the original feature request, current-main inspection is enough to show the server-side dictation path is absent.

## Solution Assessment

Mostly yes: an authenticated Gateway RPC that reuses `transcribeAudioFile` is the narrow maintainable design and keeps provider credentials server-side. The safer final form should cancel the subsequent stop handler after recorder errors so failed recordings are not transcribed as partial drafts.
