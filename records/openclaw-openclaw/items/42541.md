## Summary

Keep open: current main has shipped partial OAuth fixes, but the remaining Gemini CLI OAuth report is still source-visible across diagnostics, provider refresh behavior, cooldown policy, and projectId handling rather than fully implemented.

## What This Changes

The issue asks OpenClaw to stop silently swallowing Gemini CLI OAuth refresh failures and keep refreshed token/project metadata usable in the running gateway.

## Best Possible Solution

Keep this open and split the remaining work into focused fixes: surface the first OAuth refresh failure in normal diagnostics/status, preserve upstream refresh causes through the Google path, and track retry/cooldown/projectId policy separately where needed.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: an expired OAuth profile whose refresh throws is caught in provider-profile iteration, logged only at debug level, and then discarded while resolution continues. I did not perform a live Google OAuth refresh in this read-only review.

## Solution Assessment

No. Current main is a partial solution: it has debug logging, better wrapping, snapshot refresh, and refresh serialization, but it does not fully settle the user-visible Gemini refresh diagnostics/cooldown/projectId behavior described by later comments.
