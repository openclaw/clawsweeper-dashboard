## Summary

Keep open. Current main still leaves the central Gemini OAuth refresh visibility problem unresolved in automatic provider resolution and provider-usage auth, even though direct profile refresh error wrapping and runtime auth snapshot updates now address parts of the original report. The multi-version user comments, the closed-unmerged attempted fix PR, and current source all point to remaining work rather than a safe cleanup close.

## What This Changes

The issue requests fixing Gemini CLI OAuth refresh so expired tokens are refreshed or fail visibly instead of being hidden by swallowed auth-resolution errors, stale runtime snapshots, and cooldown behavior.

## Best Possible Solution

Keep this open until Gemini CLI OAuth has regression coverage for expired profiles through automatic embedded provider resolution and provider-usage auth, refresh failures are surfaced at an operator-visible level, and the Gemini-owned projectId, refresh hook, doctor guidance, and cooldown policy are explicitly settled.

## Reproduction Assessment

Yes for the source-level failure path: an expired google-gemini-cli OAuth profile that makes resolveApiKeyForProfile throw will still be downgraded to debug-only logging in automatic provider auth and ignored in provider-usage auth. Live OAuth reproduction was not attempted because this was a read-only review.

## Solution Assessment

Unclear. Logging the swallowed catches is necessary but likely not sufficient; the maintainable fix should also address Gemini provider-owned refresh/projectId guidance and decide whether cooldown behavior should differ for Gemini refresh failures.
