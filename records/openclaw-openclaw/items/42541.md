## Summary

Keep open: current main has partial OAuth hardening, but Gemini CLI OAuth refresh is still not implemented end-to-end and the auto-profile path can still hide the refresh failure behind debug-only logging and a generic missing-auth error.

## What This Changes

Restore Gemini CLI OAuth auto-refresh and diagnostics so expired `google-gemini-cli` profiles refresh with the stored refresh token and project context instead of failing silently or falling into generic auth/cooldown errors.

## Best Possible Solution

Add a provider-owned `google-gemini-cli` OAuth refresh implementation with regression coverage and preserve actionable refresh errors; treat cooldown-bypass policy as a separate follow-up unless owners explicitly want it in the same change.

## Reproduction Assessment

Yes, by source inspection. An expired `google-gemini-cli` OAuth profile reaches the generic refresh path, but the Google provider has no `refreshOAuth` hook and the auto-profile route still demotes caught failures to debug-only logging before a generic auth error.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is in the Google provider/plugin auth surface, with core diagnostics adjusted only enough to preserve the first refresh failure.
