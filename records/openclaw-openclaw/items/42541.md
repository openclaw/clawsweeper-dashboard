## Summary

Keep open: current main fixed some surrounding auth-store mechanics, but the observable Gemini OAuth auto-refresh problem still has a source-level path where expired `google-gemini-cli` credentials cannot be provider-refreshed.

## What This Changes

The issue asks OpenClaw to stop silently losing Gemini CLI OAuth refresh failures and to refresh expired `google-gemini-cli` auth-profile tokens reliably.

## Best Possible Solution

Add Google-plugin-owned OAuth refresh for `google-gemini-cli` using the existing provider `refreshOAuth` seam, preserve HTTP error context, keep core extension-agnostic, and cover expired-token refresh plus failure logging in focused tests.

## Reproduction Assessment

Yes, source-level. An expired `google-gemini-cli` OAuth profile reaches the generic refresh path, but the Google provider has no `refreshOAuth` hook and the shared pi-ai OAuth registry does not provide a Google refresher, so current main cannot rotate that credential from source inspection alone.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-owned refresh in `extensions/google`, not a Google-specific core fallback or broad cooldown policy change.
