## Summary

Keep open. Current main still advertises Feishu `read`, but the handler only performs a single-message `messageId` lookup and no Feishu chat-container history branch exists; the broader pagination issue #71452 does not supersede this Feishu-specific gap.

## What This Changes

Feishu `message read` should support paginated chat history retrieval by chatId with limit, page token, and time filters instead of only single-message reads.

## Best Possible Solution

Add a Feishu-owned chat-history read mode that preserves existing `messageId` reads, calls Feishu message list with `container_id_type=chat`, returns items plus pagination metadata, and documents the new surface.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `read` with chatId/limit/pageToken but no `messageId` either fails shared target validation for chatId-only input or reaches the Feishu handler's `messageId` guard when a target is supplied.

## Solution Assessment

Yes. A narrow Feishu plugin fix is the maintainable route; core should only need small generic alias or normalization coverage if `chatId` must satisfy shared `read` target validation.
