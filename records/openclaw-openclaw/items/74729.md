## Summary

Keep open. Current main still advertises Feishu `read`, but the implemented handler only accepts `messageId` and calls the single-message lookup path; `chatId`, `limit`, `before`/`after`, and `pageToken` do not reach a Feishu chat-history list call. Open PR #68443 is a relevant implementation candidate, but it is still open and has a needs-changes review; #71452 is broader cross-channel pagination work and does not supersede this Feishu-specific gap.

## What This Changes

Feishu `message read` should support paginated chat-history retrieval by chat/chatId with limit, cursor, and time-filter parameters instead of only reading one message by messageId.

## Best Possible Solution

Land a Feishu-owned paginated chat-history path that preserves the existing single-message `messageId` read, uses Feishu `im.message.list` with `container_id_type=chat` for chat/group targets, returns pagination metadata, maps `limit`/`pageToken` and time filters predictably, and updates Feishu docs/changelog plus focused tests. #68443 is the closest current branch if maintainers want to repair it rather than start a replacement.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: `action=read` with Feishu `chatId` but no `messageId` either fails target validation before plugin dispatch or, when a target is supplied, reaches the Feishu handler and fails the `messageId` guard; no chat-history list helper exists on current main.

## Solution Assessment

Yes, the narrow maintainable solution belongs in the Feishu plugin because the missing behavior is Feishu API wrapping, not core pagination alone. The open PR direction is plausible, but it is not yet the best implementation until its review findings and action-name/docs decisions are resolved.
