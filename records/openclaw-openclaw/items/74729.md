## Summary

Keep open. Current main still matches the report: Feishu advertises `read`, but the handler only accepts `messageId` and there is no chat-container history read path; the broader open pagination item #71452 does not supersede this Feishu-specific wrapper gap.

## What This Changes

Request Feishu `message read` to fetch paginated chat history by `chatId` with limit/time/page-token parameters instead of only reading one `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path using the Feishu message-list API for chat containers while preserving the existing single-`messageId` read behavior.
