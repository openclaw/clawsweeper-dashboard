## Summary

Keep open. Current main and the latest shipped tag still only support Feishu `read` by `messageId`; Feishu chat-history listing with pagination is not implemented, and the closest implementation candidate remains an open PR with review blockers.

## What This Changes

The issue requests Feishu message history reads by chat/container with multiple results and pagination instead of only single-message lookup by `messageId`.

## Best Possible Solution

Implement the Feishu-owned chat-history path in the Feishu plugin, preserving single-message reads while adding normalized limit/cursor handling, pagination metadata, correct chat-versus-thread IDs, tests, docs for required scopes, and redacted live Feishu proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Feishu `read` without `messageId` throws, and `read` with `messageId` calls the single-message getter; I did not run a live Feishu tenant in this read-only review.

## Solution Assessment

Unclear. The capability belongs in the Feishu plugin, but maintainers still need to decide whether the open `list` action PR is the desired public action shape or whether `read` should grow paginated chat-history semantics.
