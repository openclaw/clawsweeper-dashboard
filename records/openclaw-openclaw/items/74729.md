## Summary

Keep open. Current main still advertises Feishu `read`, but the implementation only supports single-message lookup by `messageId`; the existing thread-history helper is thread-only and the broader pagination issue #71452 does not supersede this Feishu chat-history wrapper gap.

## What This Changes

The issue asks Feishu `message read` to support paginated chat history retrieval with `chatId`, `limit`, `before`/`around`, and page-token style parameters instead of only returning one message by `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that uses the Feishu message list endpoint for chat containers when a chat target is supplied, while preserving the existing `messageId` single-message read behavior.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `action="read"` with `chatId`, `limit`, `before`, `after`, or `pageToken` but no `messageId` either fails shared target handling for Feishu `read` or reaches the handler's `messageId` guard, and no Feishu chat-history list path exists.

## Solution Assessment

Yes. A narrow Feishu plugin fix is the maintainable path; core should only need generic alias/normalization coverage if `chatId` must satisfy shared `read` target validation.
