## Summary

Keep open. Current main still matches the report and the existing ClawSweeper comment: Feishu advertises a `read` action, but the implemented path only resolves a single `messageId` and never routes `chatId`, `limit`, `before`, `after`, or `pageToken` to a chat-history list call. The related #71452 is broader shared pagination work and does not supersede this Feishu-specific missing wrapper.

## What This Changes

The issue requests Feishu `message read` support for paginated historical chat messages by `chatId`/limit/cursor/time filters instead of only single-message lookup by `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that preserves the existing single-message `messageId` lookup and uses the Feishu message list endpoint for chat containers when `chatId` or equivalent channel target parameters are provided. Keep #71452 as the broader cross-channel pagination follow-up rather than making core block this plugin-specific repair.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `action=read` with `chatId`, `limit`, `before`, `after`, or `pageToken` but no `messageId` cannot reach any Feishu chat-history list path and fails the handler's `messageId` guard.

## Solution Assessment

Yes. A narrow Feishu plugin fix is the best path because the shared runner already has plugin-owned action alias plumbing; core should only need small generic alias/normalization coverage if `chatId` must satisfy `read` target validation.
