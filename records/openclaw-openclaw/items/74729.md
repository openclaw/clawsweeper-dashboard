## Summary

Keep open. Current main still has no Feishu chat-history pagination path: the advertised Feishu `read` action requires `messageId` and calls the single-message getter, while the closest open implementation PR is still unmerged and has review blockers.

## What This Changes

The issue requests paginated Feishu chat-history reads using chatId/limit/before/after/page_token instead of only single-message `messageId` reads.

## Best Possible Solution

A Feishu-owned chat-history path should preserve single-message reads, use correct chat versus thread IDs, normalize limit/cursor inputs, return pagination metadata, and document required scopes and CLI/tool behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main advertises Feishu `read`, but `read` without `messageId` throws and `read` with `messageId` calls the single-message Feishu getter; no live Feishu tenant was run in this read-only review.

## Solution Assessment

Unclear. The capability belongs in the Feishu plugin, but the closest open PR adds a new `list` action rather than extending `read` and still needs routing, pagination-normalization, and real-behavior proof before it is the maintainable fix.
