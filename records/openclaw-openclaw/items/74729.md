## Summary

Keep open. Current main still matches the report: Feishu advertises a `read` message action, but the implementation only accepts `messageId` and has no chat-container history list path; the broader open pagination item #71452 does not supersede this Feishu-specific wrapper gap.

## What This Changes

Feishu `message read` should support paginated chat-history reads using `chatId`, `limit`, time filters, and `pageToken`, while preserving single-message reads by `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that uses the Feishu message list endpoint for chat containers when a chat target is supplied, while keeping the existing `messageId` single-message read intact.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `action="read"` with `chatId`, `limit`, `before`, `after`, or `pageToken` but no `messageId` either lacks a recognized `read` target alias or reaches the handler's `messageId` guard, and no Feishu chat-history list path exists.

## Solution Assessment

Yes. A narrow Feishu plugin fix is the maintainable path; core should only need small generic alias/normalization coverage if `chatId` must satisfy shared `read` target validation.
