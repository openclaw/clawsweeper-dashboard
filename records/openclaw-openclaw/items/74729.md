## Summary

Keep open. Current main still exposes Feishu `read`, but the implementation only fetches one `messageId` and has no paginated chat-history path; the broader #71452 pagination issue does not supersede this Feishu-specific wrapper gap.

## What This Changes

Request Feishu `message read` support for paginated chat history using chat targets and pagination/filter parameters instead of only single-message lookups.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path while preserving the existing single-`messageId` read behavior, then document and test the new surface.

## Reproduction Assessment

Yes. Source inspection on current main shows `read` with `chatId`, `limit`, or `pageToken` but no `messageId` cannot reach a chat-history path, while `read` with `messageId` calls the single-message Feishu getter.

## Solution Assessment

Yes. A narrow Feishu plugin change is the maintainable solution; core should only need minimal shared alias or normalization support if `chatId` must satisfy generic `read` target validation.
