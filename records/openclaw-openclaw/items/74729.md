## Summary

Keep open. Current main still advertises Feishu `read`, but the handler requires `messageId` and calls the single-message getter; there is no chat-history/page-token read path, and the broader #71452 pagination item does not supersede this Feishu-specific wrapper gap.

## What This Changes

Request Feishu `message read` to fetch paginated chat history with `chatId`, `limit`, `page_token`, and before/after filters instead of only reading one `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history branch to `read` while preserving existing single-`messageId` reads, then cover it with plugin tests, shared alias tests if needed, docs, and changelog.

## Reproduction Assessment

Yes, source-reproducible on current main: Feishu `read` without `messageId` throws before dispatch, while `read` with `messageId` calls the single-message getter. I did not run a live Feishu tenant check.

## Solution Assessment

Yes. A narrow Feishu plugin enhancement is the maintainable path; core should only change if `chatId` target alias or normalization support is needed for the new read mode.
