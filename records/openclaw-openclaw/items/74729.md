## Summary

Keep open. Current main still has the Feishu gap described in the report: the plugin advertises `read`, but the Feishu handler only accepts `messageId` and calls the single-message lookup path; `chatId`, `limit`, `pageToken`, `before`, and `after` cannot reach a paginated chat-history list call. The related #71452 is broader shared pagination work and does not supersede this Feishu-specific missing API wrapper.

## What This Changes

The issue requests Feishu `message read` support for paginated chat history using `chatId`, limit, page token, and time filters instead of only fetching a single `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that preserves the existing single-message `messageId` lookup and uses the Feishu message list endpoint for chat containers when `chatId` or equivalent channel target parameters are provided. Keep #71452 as the broader cross-channel pagination follow-up rather than making core block this plugin-specific repair.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `action=read` with `chatId`, `limit`, `before`, `after`, or `pageToken` but no `messageId` cannot reach any Feishu chat-history list path and either fails target normalization or the Feishu handler's `messageId` guard.

## Solution Assessment

Yes. A narrow Feishu plugin fix is the best path because the shared runner already supports plugin-owned action aliases and dispatch; core should only need small generic alias/normalization coverage if `chatId` must satisfy `read` target validation.
