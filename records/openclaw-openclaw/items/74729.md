## Summary

Keep open. Current main and the latest release still advertise Feishu `read`, but the Feishu handler only accepts `messageId` and routes to a single-message getter; `chatId`, `limit`, and `pageToken` do not reach a paginated chat-history path, and the broader #71452 pagination issue does not supersede this Feishu-specific wrapper gap.

## What This Changes

Request Feishu `message read` to list chat history with pagination and time filters instead of only reading one message by `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history branch to `read`, preserve the existing single-message `messageId` path, and cover the new surface in tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Feishu `read` with `chatId`, `limit`, or `pageToken` but no `messageId` reaches the handler and throws, while `read` with `messageId` calls the single-message getter.

## Solution Assessment

Yes. A narrow Feishu plugin enhancement is the best fit; core should only change if a generic target-alias seam is needed to let `chatId` satisfy `read` validation.
