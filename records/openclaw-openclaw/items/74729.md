## Summary

Keep open. Current main still advertises Feishu `read`, but the Feishu handler only accepts `messageId` and fetches one message; the broader formatter pagination issue at https://github.com/openclaw/openclaw/issues/71452 and its open PR do not cover this Feishu provider wrapper gap.

## What This Changes

Request Feishu `message read` to support paginated chat-history reads with `chatId`, `limit`, page tokens, and before/after filters instead of only single-message lookup.

## Best Possible Solution

Add a Feishu-owned paginated chat-history `read` path that preserves existing single-`messageId` reads and returns normalized messages plus `hasMore`/`pageToken` metadata.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `read` without `messageId` cannot reach a chat-history list, and `read` with `messageId` calls the single-message Feishu getter.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin, with only minimal shared alias/normalization changes if needed for `chatId` targeting.
