## Summary

Keep open. Current main still matches the reported Feishu gap: the plugin advertises `read`, but the implementation only accepts a `messageId` and calls the single-message lookup path, while chat-history parameters such as `chatId`, `limit`, `pageToken`, `before`, and `after` have no Feishu chat-list route. The related #71452 is broader shared pagination work and does not supersede this Feishu-specific missing wrapper.

## What This Changes

Request Feishu `message read` support for paginated chat history using `chatId` plus limit/cursor/time filters instead of only single-message `messageId` reads.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that preserves the existing single-message `messageId` lookup, uses Feishu `im.message.list` with `container_id_type: "chat"` when a chat target is provided, returns messages plus pagination metadata, and updates focused tests, CLI/channel docs, and the changelog.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `action=read` with `chatId`, `limit`, `before`, `after`, or `pageToken` but no `messageId` cannot reach any Feishu chat-history list path and either fails target normalization or the Feishu handler's `messageId` guard.

## Solution Assessment

Yes. A narrow Feishu plugin fix is the best maintainable path because the shared runner already allows plugin-owned action aliases and dispatch; core should only need small generic alias/normalization coverage if `chatId` must satisfy `read` target validation.
