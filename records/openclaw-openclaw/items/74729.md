## Summary

Keep open. Current main still advertises Feishu `read`, but the Feishu handler requires `messageId` and calls the single-message lookup path; `chatId`, `limit`, `pageToken`, `before`, and `after` cannot reach a paginated chat-history request. The broader open pagination issue #71452 is related but does not supersede this Feishu-specific API gap.

## What This Changes

Requests Feishu `message read` to support paginated chat-history reads by `chatId` with limits, page tokens, and time filters instead of only fetching one `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that uses `client.im.message.list` with `container_id_type: "chat"`, preserves the existing `messageId` lookup behavior, returns messages plus pagination metadata, and updates Feishu action aliases, tests, docs, and changelog. Keep #71452 as the broader cross-channel pagination/rendering follow-up.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: Feishu `action=read` with `chatId`, `limit`, `before`, `after`, or `pageToken` but no `messageId` throws before dispatching any list request; with `messageId` it only calls the single-message `client.im.message.get` path.

## Solution Assessment

Yes. A narrow Feishu plugin change is the maintainable fix because the shared runner already passes action params through to channel plugins; core should only need alias or target-normalization coverage if `chatId` must satisfy generic read target validation.
