## Summary

Keep open. Current main still advertises Feishu `read`, but the implemented path only resolves a single `messageId`; `chatId`, `limit`, `before`, `after`, and `pageToken` do not reach a Feishu chat-history list call. The related #71452 is broader shared pagination work and does not supersede this Feishu-specific missing wrapper.

## What This Changes

Request Feishu `message read` support for paginated chat history using `chatId` plus limit/cursor/time filters instead of only single-message lookup by `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that preserves the existing `messageId` lookup and returns multiple messages plus `hasMore`/`pageToken` when a chat container is supplied. Update Feishu action aliases, focused tests, docs, and changelog; keep broader cross-channel pagination tracked separately by #71452.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `action=read` with `chatId`, `limit`, `before`, `after`, or `pageToken` but no `messageId` cannot reach a Feishu chat-history list path and fails before or inside the Feishu handler.

## Solution Assessment

Yes. A narrow Feishu plugin fix is the best path because this is Feishu-owned API behavior; core should only need minimal generic alias/normalization coverage if `chatId` must satisfy shared `read` target validation.
