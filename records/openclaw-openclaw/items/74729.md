## Summary

Keep open. Current main still advertises Feishu `read`, but the handler only accepts a message ID and dispatches to Feishu single-message lookup; `chatId`, `limit`, `pageToken`, and time-filtered history reads do not reach a chat-history list call. The related #71452 is broader shared pagination work and does not supersede this Feishu-specific API gap.

## What This Changes

The issue requests paginated Feishu `message read` support for chat history using `chatId`, `limit`, `page_token`, and time filters instead of returning only one `messageId` lookup.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path using `client.im.message.list` with `container_id_type: "chat"`, while preserving the existing `messageId` lookup. Update Feishu action aliases/normalization, docs, tests, and changelog; leave #71452 as the broader cross-channel pagination follow-up.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `action=read` with only `chatId`, `limit`, `before`, `after`, or `pageToken` cannot reach a Feishu chat-history list path and fails before dispatching any Feishu history request.

## Solution Assessment

Yes. The best fix is a narrow Feishu plugin change because the shared runner already passes action params through to channel plugins; core should only need alias/normalization coverage if `chatId` must satisfy generic read target validation.
