## Summary

Keep open. Current main still implements Feishu `read` as a `messageId`-only lookup and has no chat-container history path; open PR #68443 is an unmerged implementation candidate, not proof this is fixed.

## What This Changes

Allow Feishu message reads to return paginated chat history by `chatId` with pagination and time filters instead of only fetching one `messageId`.

## Best Possible Solution

Land a Feishu-owned paginated chat-history path using `im.message.list` with `container_id_type=chat`, preserving the existing single-message lookup and coordinating with #68443 or a replacement PR.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `read` with `chatId`, `limit`, or `pageToken` but no `messageId` either fails shared target validation or reaches the Feishu handler's `messageId` guard, and no chat-history list path exists on main.

## Solution Assessment

Yes. The maintainable fix belongs in the Feishu plugin, with only small shared action/alias metadata changes if maintainers choose a new `list` action instead of extending `read`.
