## Summary

Keep open. Current main still exposes Feishu `read`, but it only supports a single `messageId` lookup; chat-history reads with `chatId`, `limit`, `pageToken`, `before`, or `after` remain unimplemented, and the broader pagination issue #71452 does not supersede this Feishu-specific wrapper gap.

## What This Changes

Add Feishu message read support for paginated chat history instead of only single-message reads.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path while preserving the existing single-message `messageId` behavior.

Yes. Static current-main reproduction is high-confidence: a Feishu `read` call without `messageId` cannot reach chat history listing, and `chatId` is not accepted as a read target alias.

## Solution Assessment

Yes. A narrow Feishu plugin fix is the maintainable route; core should only need small generic alias or normalization coverage if `chatId` must satisfy shared `read` target validation.
