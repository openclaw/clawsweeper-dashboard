## Summary

Current main still lacks the requested read-only `openclaw doctor` diagnostic for catalog rows that exist in the runtime/provider-augmented catalog but are omitted by a non-empty `agents.defaults.models` allowlist. The report remains valid and narrow: current doctor only checks the hooks-specific model path, while OpenAI/OpenAI Codex provider hooks still add forward-compatible catalog rows and the allowlist builder still filters catalog availability without warning about omitted rows.

## What This Changes

Add an `openclaw doctor` warning for available provider-augmented or synthetic model catalog rows that are missing from an active `agents.defaults.models` allowlist, without modifying the allowlist.

## Best Possible Solution

Keep this open and implement a warn-only doctor health contribution backed by a stable catalog-origin contract. The check should load the runtime catalog, skip absent or empty allowlists, compare provider-augmented/forward-compatible rows against configured allowlist keys, and print concrete model refs plus `openclaw models allowlist add <model-id>` guidance without mutating config.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a non-empty `agents.defaults.models` allowlist while OpenAI/OpenAI Codex `augmentModelCatalog` contributes GPT-5.4/GPT-5.5 rows, then current `openclaw doctor` has no general catalog allowlist-gap check beyond `hooks.gmail.model`.

## Solution Assessment

Yes, the requested warn-only doctor diagnostic is the narrow maintainable solution, provided it first adds or reuses a stable catalog-source marker instead of provider-name heuristics. Auto-adding models would be worse because the allowlist is an operator control boundary.
