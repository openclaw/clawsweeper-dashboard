## Summary

Keep open: current main and v2026.6.10 define Chat Completions `tool_calls` as caller-supplied client tools, so returning internal OpenClaw tool telemetry would be a new API/security decision rather than a confirmed regression.

## What This Changes

The issue asks whether OpenClaw can return internal tool-call information through the OpenAI-compatible `/v1/chat/completions` API when using a supplied base URL, API key, and model.

## Best Possible Solution

Define the API contract explicitly: keep OpenAI-compatible `tool_calls` limited to caller-supplied tools, or design a separate opt-in, redacted diagnostic/internal-tool telemetry mode with tests and docs.

## Reproduction Assessment

No high-confidence live reproduction. Source review confirms current `/v1/chat/completions` only returns `tool_calls` for pending client tools, but the issue lacks a concrete qwen request/response showing an internal OpenClaw tool turn.

## Solution Assessment

No as a bug fix. The narrowest maintainable path is a maintainer product/security decision on whether internal tool telemetry belongs in this API or in a separate diagnostic/Gateway surface.
