## Summary

Keep open: current main and latest release document and implement `/v1/chat/completions` `tool_calls` for caller-supplied client tools, not every internal OpenClaw tool. Exposing internal tool telemetry through this API would be a new product and security contract, and the report lacks a concrete qwen request/response proving a regression.

## What This Changes

The issue asks whether OpenClaw's `/api/chat/completions` agent interaction can return internal tool-call information when using a supplied base URL, API key, and model name.

## Best Possible Solution

Define the product contract: keep Chat Completions `tool_calls` limited to caller-supplied tools and point internal-tool debugging to trajectory/Gateway diagnostics, or design an explicit redacted opt-in diagnostic/API mode for internal tool telemetry.

## Reproduction Assessment

No high-confidence live bug repro. Source review confirms current `/v1/chat/completions` only returns OpenAI `tool_calls` for pending client-tool calls, but the issue lacks a concrete qwen request/response showing the claimed internal-tool turn.

## Solution Assessment

No, treating this as a bug fix is not the best path yet. Current docs define the Chat Completions tool contract for caller-supplied tools, so maintainers first need to decide whether internal tool telemetry belongs in this API, a distinct Responses/diagnostic contract, or existing trajectory surfaces.
