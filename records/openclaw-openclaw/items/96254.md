## Summary

Keep open for maintainer product/security direction: current main and v2026.6.10 define Chat Completions `tool_calls` as caller-supplied client tools, while returning internal OpenClaw tool telemetry would be a new API contract with sensitive-data implications.

## What This Changes

The issue asks OpenClaw to return internal agent tool-call information through the OpenAI-compatible Chat Completions API when using a supplied base URL, API key, and model.

## Best Possible Solution

Define a redacted, opt-in diagnostic or Gateway API contract if internal tool telemetry is wanted; otherwise keep Chat Completions `tool_calls` limited to caller-supplied tools and point tool inspection to existing diagnostics surfaces.

## Reproduction Assessment

No high-confidence live reproduction exists because the issue lacks the actual HTTP request and response. Source review does confirm current main only returns OpenAI `tool_calls` for pending caller-supplied client tools, not arbitrary internal OpenClaw tool telemetry.

## Solution Assessment

No, treating this as a straightforward bug fix is not the best path. The maintainable solution first needs a product and security decision about whether internal tool telemetry belongs in Chat Completions or in an explicit redacted diagnostic/Gateway surface.
