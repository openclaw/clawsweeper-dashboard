## Summary

Keep open for maintainer product/API direction, but treat it as a feature/support request rather than a confirmed bug. Current `main` deliberately returns OpenAI-compatible `tool_calls` only for caller-supplied client tools; exposing already-executed internal OpenClaw tool calls over an HTTP response surface is a separate audit/trace contract with compatibility and security implications.

## What This Changes

Request that OpenClaw’s OpenAI-compatible chat-completions flow return internal OpenClaw tool-call information when the agent uses built-in tools.

## Best Possible Solution

Keep the current Chat Completions default limited to caller-supplied client tools, and have maintainers decide whether internal tool visibility belongs in the ongoing `/v1/responses` audit work or needs a separate opt-in Chat Completions trace contract.

## Reproduction Assessment

Yes, source-level for the missing capability: current `main` documents and implements Chat Completions `tool_calls` for caller-supplied client tools, while internal OpenClaw tools are not exposed as chat-completions tool calls. I did not run a live Gateway/OpenWebUI repro because the report lacks concrete request details and credentials.

## Solution Assessment

No, not as a bug fix. Returning internal OpenClaw tool calls through `/v1/chat/completions` would be a new API/audit feature that needs an explicit item shape, redaction policy, default-off behavior, and compatibility review.
