## Summary

Keep open: current main and v2026.6.10 document Chat Completions `tool_calls` as caller-supplied client tools, not every internal OpenClaw tool. Exposing internal tool telemetry would be a new API/security decision, and the report lacks a concrete qwen request/response to treat it as a confirmed regression.

## What This Changes

Request that `/v1/chat/completions` return OpenClaw internal tool-call information when interacting with an agent via a base URL, API key, and model name.

## Best Possible Solution

Define the product contract: keep OpenAI-compatible `tool_calls` limited to caller-supplied tools and point internal-tool debugging to Gateway/trajectory surfaces, or design an explicit redacted diagnostic/API mode for internal tool telemetry.

## Reproduction Assessment

No high-confidence live repro. Source review confirms current `/v1/chat/completions` only returns OpenAI `tool_calls` for pending client-tool calls, but the issue lacks a concrete request/response showing an internal OpenClaw tool turn.

## Solution Assessment

No, treating this as a bug fix is not the best path yet. Current docs define the Chat Completions tool contract for caller-supplied tools, so maintainers first need to decide whether internal tool telemetry belongs in this API or in diagnostic/Gateway surfaces.
