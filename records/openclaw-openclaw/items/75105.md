## Summary

Keep open. Current main still has the requested compatibility gap: custom OpenAI-compatible models can carry `compat.thinkingFormat: "deepseek"`, and the shared DeepSeek V4 payload wrapper exists, but the generic stream wrapper path does not install that wrapper from model compat for proxied or provider-prefixed DeepSeek V4 model ids.

## What This Changes

The issue requests a per-model configuration path so DeepSeek V4 `reasoning_content` replay handling works for third-party or custom OpenAI-compatible providers with non-native provider ids or differently prefixed model ids.

## Best Possible Solution

Add a generic model-compat-driven DeepSeek V4 path that reuses `compat.thinkingFormat: "deepseek"`, recognizes DeepSeek V4 Pro/Flash including provider-prefixed ids such as `deepseek/deepseek-v4-pro`, installs the shared replay wrapper for custom OpenAI-compatible providers, and documents the custom provider configuration without hard-coding Requesty, OpenRouter, Wisgate, or another proxy provider.

## Reproduction Assessment

Yes. Configure a custom `openai-completions` provider with a model id such as `deepseek/deepseek-v4-pro` and `compat.thinkingFormat: "deepseek"`; source inspection shows current main uses that value for rendering but does not install the DeepSeek V4 request wrapper in the generic stream chain.

## Solution Assessment

Partially. The per-model compatibility direction is the right boundary, but the narrowest maintainable solution should reuse the existing `compat.thinkingFormat` key unless maintainers explicitly choose a public alias named `reasoningFormat`.
