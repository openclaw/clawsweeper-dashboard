## Summary

Keep open. Current main now covers DeepSeek V4 proxy routes by model-id suffix, but it still does not use a model-level compat opt-in when deciding whether to install the DeepSeek reasoning_content replay wrapper.

## What This Changes

The issue requests per-model configuration so custom OpenAI-compatible DeepSeek routes can opt into DeepSeek-style reasoning_content replay.

## Best Possible Solution

Reuse the existing `compat.thinkingFormat: "deepseek"` model-level contract to trigger the generic DeepSeek V4 replay wrapper, with focused tests and docs, instead of adding provider ids or another overlapping config key.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` model with `compat.thinkingFormat: "deepseek"`, thinking enabled, replayed assistant history missing `reasoning_content`, and a model id that does not end in the two hard-coded DeepSeek V4 suffixes.

## Solution Assessment

Yes. The narrow maintainable solution is to consume the existing model compat key at the wrapper-install decision and keep the current model-id fallback for known DeepSeek V4 routes.
