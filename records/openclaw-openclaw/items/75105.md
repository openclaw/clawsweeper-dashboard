## Summary

Keep open. Current main has named-provider DeepSeek V4 replay fixes and an existing model-level `compat.thinkingFormat` field, but arbitrary OpenAI-compatible providers still have no generic path that uses model compat metadata to install the DeepSeek `reasoning_content` replay wrapper.

## What This Changes

Allow model configuration to opt custom OpenAI-compatible DeepSeek routes into the DeepSeek `reasoning_content` replay fix.

## Best Possible Solution

Wire the existing model-level `compat.thinkingFormat === "deepseek"` opt-in into the generic OpenAI-compatible stream wrapper path, document the supported config, and avoid hard-coding proxy provider aliases.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: configure a custom `openai-completions` model with `compat.thinkingFormat: "deepseek"`, enable thinking, and replay assistant history missing `reasoning_content`; current main accepts the compat value but does not install the DeepSeek replay wrapper from it.

## Solution Assessment

Yes. A model-level opt-in via existing compat metadata is narrower and more maintainable than adding Requesty, Wisgate, or other proxy aliases to provider-specific gates.
