## Summary

Keep open. Current main has the model-level `compat.thinkingFormat` surface and a shared DeepSeek V4 replay wrapper, but custom OpenAI-compatible providers still do not get that wrapper from model compat alone, so the reported proxy-provider gap remains actionable.

## What This Changes

Allow a per-model compatibility setting to trigger DeepSeek-style `reasoning_content` replay handling for OpenAI-compatible proxy or custom providers.

## Best Possible Solution

Wire the existing model-level `compat.thinkingFormat: "deepseek"` opt-in into the generic OpenAI-compatible stream-wrapper path, with docs and tests, instead of adding proxy provider IDs.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider with a DeepSeek V4 model, `compat.thinkingFormat: "deepseek"`, thinking enabled, and replayed assistant history missing `reasoning_content`; current main renders the DeepSeek thinking format but does not install the DeepSeek payload wrapper unless a provider plugin does it.

## Solution Assessment

Yes. The model-level opt-in direction is the narrow maintainable fix, preferably by reusing the shared DeepSeek V4 wrapper and avoiding provider-name hardcoding.
