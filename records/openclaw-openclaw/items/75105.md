## Summary

Keep open. Current main covers many DeepSeek V4 proxy routes by model-id suffix and the shared wrapper now backfills every replayed assistant message, but the generic wrapper decision still does not honor a model-level compat opt-in for non-matching custom model ids.

## What This Changes

Request per-model configuration to enable DeepSeek-style `reasoning_content` replay handling for custom OpenAI-compatible proxy models.

## Best Possible Solution

Reuse the existing `compat.thinkingFormat: "deepseek"` model-level contract to trigger the generic DeepSeek V4 replay wrapper, with focused tests and docs, instead of adding another config key.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` model with `compat.thinkingFormat: "deepseek"`, thinking enabled, replayed assistant history missing `reasoning_content`, and a model id whose final segment is not `deepseek-v4-flash` or `deepseek-v4-pro`.

## Solution Assessment

Yes. The narrowest maintainable fix is to consume the existing model compat key at the wrapper-install decision and keep the current model-id fallback for known DeepSeek V4 routes.
