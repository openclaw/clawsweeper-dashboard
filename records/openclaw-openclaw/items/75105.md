## Summary

Keep open. Current main has native DeepSeek plus provider-specific OpenRouter, Venice, and opencode-go replay fixes, but it still lacks the requested generic model-compat path for arbitrary OpenAI-compatible providers such as Requesty.

## What This Changes

Request a per-model compatibility setting that triggers DeepSeek-style `reasoning_content` replay handling for third-party or custom providers with DeepSeek V4 models.

## Best Possible Solution

Add a focused model-compat-driven DeepSeek V4 replay path for custom OpenAI-compatible providers, preferably reusing `compat.thinkingFormat: "deepseek"` and the existing shared wrapper instead of adding more provider IDs.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` model with `compat.thinkingFormat: "deepseek"`, thinking enabled or passed through, and replayed assistant history missing `reasoning_content`; current main preserves the compat metadata but never installs the shared DeepSeek wrapper from that value.

## Solution Assessment

Yes. The model-level opt-in is the narrow maintainable direction; the safest implementation is to reuse or document the existing `compat.thinkingFormat` surface, adding an alias only if maintainers intentionally want the requested name.
