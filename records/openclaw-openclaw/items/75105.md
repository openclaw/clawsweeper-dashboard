## Summary

Keep open. Current main has the shared DeepSeek V4 replay wrapper and model-level `compat.thinkingFormat`, but the generic OpenAI-compatible stream path still does not install that wrapper from model compat for custom/proxy providers. The only current wrapper entry points are provider-owned DeepSeek, Venice, and OpenCode Go paths, and the linked implementation PR #75619 is closed without being merged.

## What This Changes

The issue requests a per-model compatibility setting that lets custom or proxy OpenAI-compatible providers opt into DeepSeek-style `reasoning_content` replay handling.

## Best Possible Solution

Add a schema/docs-supported model-compat opt-in that makes custom OpenAI-compatible models install the shared DeepSeek replay wrapper without adding Requesty/OpenRouter-specific provider ids, while preserving the existing DeepSeek, Venice, and OpenCode Go provider-owned behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a custom `openai-completions` provider model with `compat.thinkingFormat: "deepseek"`, thinking enabled, and replayed assistant history missing `reasoning_content`; current main renders DeepSeek thinking format but never installs the DeepSeek payload wrapper on the generic stream path.

## Solution Assessment

Yes on the model-level opt-in direction, but not by hard-coding more proxy provider ids. The narrow maintainable fix is to reuse `compat.thinkingFormat: "deepseek"` or add a small alias mapped to it, then wire that metadata into the generic stream-wrapper chain with focused tests.
