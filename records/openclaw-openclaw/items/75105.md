## Summary

Keep open. Current main has a model-level `compat.thinkingFormat` value and the shared DeepSeek V4 payload wrapper, but the generic stream-wrapper path still does not install that wrapper from model compat for custom OpenAI-compatible providers. The native DeepSeek plugin remains provider and exact-model-id gated, and the related fix PR was closed without merging.

## What This Changes

The issue requests a per-model compatibility setting that lets custom/proxy OpenAI-compatible providers opt into the DeepSeek `reasoning_content` replay fix without hard-coding provider IDs or exact model IDs.

## Best Possible Solution

Add a model-compat-driven DeepSeek reasoning replay path for OpenAI-compatible custom providers, preferably reusing existing `compat.thinkingFormat: "deepseek"`. An explicit model config should install the shared payload wrapper without provider hard-coding, while preserving provider-specific wrappers for cases such as Venice.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider with a model such as `deepseek/deepseek-v4-pro` and `compat.thinkingFormat: "deepseek"`; current main accepts/uses the compat value for rendering but never installs the DeepSeek payload wrapper on the generic stream path.

## Solution Assessment

Yes, model-level opt-in is the maintainable boundary, but the narrowest fix should reuse the existing compat field unless maintainers explicitly want a new public alias. Hard-coding Requesty, OpenRouter, or more proxy provider IDs would only repeat the current failure mode.
