## Summary

Keep open. Current main has the shared DeepSeek V4 payload wrapper and already backfills `reasoning_content` on every replayed assistant message, but the generic OpenAI-compatible provider path still does not install that wrapper from model-level compat. Native DeepSeek remains provider/exact-id gated, and the only linked fix PR is closed unmerged.

## What This Changes

Request model-level configuration that lets custom or proxy OpenAI-compatible providers opt into DeepSeek-style `reasoning_content` request/replay handling for DeepSeek V4 or similar models.

## Best Possible Solution

Add a model-compat-driven opt-in for the DeepSeek reasoning replay wrapper on OpenAI-compatible custom providers, avoiding provider ID hard-coding. Prefer reusing or clearly documenting the existing compat surface if it is meant to imply request payload behavior; otherwise add a narrowly named compat key and wire schema, normalization, docs, tests, and changelog together.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider with a model like `deepseek/deepseek-v4-pro` and DeepSeek compat configured: current main can consume the compat value for rendering, but the generic stream path never installs the DeepSeek payload wrapper.

## Solution Assessment

Yes, model-level opt-in is the maintainable direction because proxy providers and model IDs vary. Hard-coding Requesty, OpenRouter, or more provider aliases would repeat the current failure mode.
