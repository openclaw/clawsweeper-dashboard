## Summary

Keep open. Current main has the shared DeepSeek V4 payload wrapper and model-level `compat.thinkingFormat: "deepseek"`, but the generic custom-provider stream path still does not install that wrapper from model compat, so proxy providers can render DeepSeek thinking format while still sending replay payloads without the required `reasoning_content` backfill. The earlier linked implementation PR was closed unmerged, so no merged fix supersedes the report.

## What This Changes

Request a per-model DeepSeek reasoning-content compatibility opt-in so custom or proxy OpenAI-compatible providers can replay DeepSeek V4 `reasoning_content` without hard-coded provider IDs.

## Best Possible Solution

Wire the existing model compat metadata into the generic OpenAI-compatible stream wrapper path so a configured model can opt into the shared DeepSeek V4 replay wrapper without adding Requesty, OpenRouter, or other proxy-provider hard-codes.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider model such as `deepseek/deepseek-v4-pro` with `compat.thinkingFormat: "deepseek"`, thinking enabled, and prior assistant messages missing `reasoning_content`; current main consumes the compat for rendering but does not install the DeepSeek payload wrapper on that generic path.

## Solution Assessment

Yes, the model-level opt-in direction is the narrowest maintainable fix. The safer implementation is to reuse the existing `compat.thinkingFormat` surface or add a small alias to it, then install the shared wrapper from generic stream composition rather than adding proxy-provider IDs.
