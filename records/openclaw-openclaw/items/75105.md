## Summary

Keep open. Current main covers native DeepSeek, OpenRouter, Venice, and opencode-go DeepSeek V4 replay cases, but it still does not expose a generic model-compat-driven replay path for arbitrary OpenAI-compatible providers such as Requesty.

## What This Changes

Request a model-level compat setting that applies DeepSeek-style `reasoning_content` replay handling to DeepSeek V4 models behind custom or proxy OpenAI-compatible providers.

## Best Possible Solution

Add a focused model-compat-driven DeepSeek V4 replay path, preferably using `compat.thinkingFormat: "deepseek"`, and document the custom-provider configuration.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider/model with `compat.thinkingFormat: "deepseek"`, thinking enabled, and replayed assistant history missing `reasoning_content`; current main preserves that compat value but does not install the replay wrapper from it.

## Solution Assessment

Yes. A model-level opt-in is the narrow maintainable direction, but the safest implementation should reuse the existing compat contract rather than adding more provider ids or an overlapping public key without maintainer approval.
