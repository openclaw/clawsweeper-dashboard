## Summary

Keep open. Current main has the shared DeepSeek V4 replay helper, model-level `compat.thinkingFormat` parsing, and named-provider fixes for routes such as native DeepSeek, OpenRouter, Venice, and opencode-go, but arbitrary OpenAI-compatible proxy providers still cannot opt into the DeepSeek replay wrapper from model-level compat; the closed #75619 PR was not merged, and #76018 only covered verified OpenRouter routes.

## What This Changes

Allow a model-level compatibility setting to trigger DeepSeek-style `reasoning_content` replay for DeepSeek V4 models exposed through non-DeepSeek or proxy providers.

## Best Possible Solution

Implement a generic post-plugin OpenAI-completions wrapper that applies the existing DeepSeek V4 wrapper when a model opts into DeepSeek thinking format, with docs/schema tests and no proxy-provider hardcoding.

## Reproduction Assessment

Yes, at source level. A custom `openai-completions` model can carry `compat.thinkingFormat: "deepseek"`, but current main only uses that compat value for rendering while the generic stream wrapper path never installs the DeepSeek replay wrapper.

## Solution Assessment

Yes. Reusing the existing `compat.thinkingFormat === "deepseek"` model surface is the narrowest maintainable direction; adding Requesty or more proxy provider IDs would leave the same class of issue behind.
