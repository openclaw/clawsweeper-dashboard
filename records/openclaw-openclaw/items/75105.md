## Summary

Keep open. Current main has DeepSeek V4 replay fixes for the native DeepSeek plugin and named proxy routes such as OpenRouter/Venice, and it accepts model-level `compat.thinkingFormat`, but it still does not use that model compat value to install the DeepSeek replay wrapper for arbitrary OpenAI-compatible providers such as Requesty.

## What This Changes

Request a per-model compatibility setting so arbitrary OpenAI-compatible DeepSeek V4 routes can opt into `reasoning_content` replay/backfill behavior.

## Best Possible Solution

Wire model-level `compat.thinkingFormat === "deepseek"` into the generic OpenAI-compatible stream wrapper path, document the supported config, and avoid hard-coding individual proxy provider IDs.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` model with `compat.thinkingFormat: "deepseek"`, thinking enabled, and replayed assistant history missing `reasoning_content`; current main accepts that compat value but never installs the DeepSeek replay wrapper from it.

## Solution Assessment

Yes. A model-level opt-in is the narrow maintainable direction; the safer implementation should reuse the existing `compat.thinkingFormat` surface or add a deliberate alias instead of hard-coding Requesty, Wisgate, or other proxy providers.
