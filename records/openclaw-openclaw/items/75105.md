## Summary

Keep open. Current main has the model-level `compat.thinkingFormat` surface and a shared DeepSeek V4 replay wrapper, but custom OpenAI-compatible providers still do not get that wrapper installed from model compat; the linked implementation PR was closed unmerged.

## What This Changes

This issue requests per-model configuration so proxy or custom providers can opt into DeepSeek V4 `reasoning_content` replay compatibility.

## Best Possible Solution

Add a generic model-compat-driven DeepSeek V4 replay path for OpenAI-compatible custom providers, using the shared wrapper without adding Requesty/OpenRouter-specific provider IDs.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider with a DeepSeek V4 model, `compat.thinkingFormat: "deepseek"`, thinking enabled, and replayed assistant history missing `reasoning_content`; current main renders DeepSeek thinking format but does not install the DeepSeek payload wrapper on the generic stream path.

## Solution Assessment

Yes, the model-level opt-in direction is the narrow maintainable fix. Prefer wiring the existing `compat.thinkingFormat === "deepseek"` surface into the generic stream-wrapper path, with an alias only if maintainers want the requested `reasoningFormat` spelling.
