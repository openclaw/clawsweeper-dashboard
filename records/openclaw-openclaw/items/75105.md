## Summary

Keep open. Current main has the shared DeepSeek V4 replay helper and provider-specific fixes for native DeepSeek, OpenRouter, Venice, and opencode-go, but it still lacks the requested generic model-compat-driven path for arbitrary OpenAI-compatible providers such as Requesty.

## What This Changes

Request a model-level compat setting that applies DeepSeek-style `reasoning_content` replay handling to DeepSeek V4 models behind custom or proxy OpenAI-compatible providers.

## Best Possible Solution

Add a focused model-compat-driven DeepSeek V4 replay path for custom OpenAI-compatible providers, reusing the shared wrapper and preserving existing provider-specific wrappers.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider/model with `compat.thinkingFormat: "deepseek"`, thinking enabled, and replayed assistant history missing `reasoning_content`; current main will render DeepSeek thinking format but does not install the DeepSeek replay wrapper from that compat value.

## Solution Assessment

Yes. The model-level opt-in is the narrow maintainable direction; the safest implementation is to reuse or alias the existing `compat.thinkingFormat === "deepseek"` surface rather than adding more provider ids.
