## Summary

Keep open. Current main has fixed several named DeepSeek V4 routes and the shared helper now backfills every assistant replay message, but arbitrary OpenAI-compatible providers still cannot opt into that wrapper from model-level compat; the linked implementation PR was closed unmerged.

## What This Changes

Request a per-model compat setting that lets custom OpenAI-compatible providers opt into the DeepSeek V4 `reasoning_content` replay fix instead of relying on hard-coded provider/model IDs.

## Best Possible Solution

Implement and document a model-level DeepSeek replay opt-in, preferably by reusing `compat.thinkingFormat: "deepseek"`, while preserving the existing provider-specific wrappers and avoiding new proxy-provider ID lists.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a custom `openai-completions` provider/model with `compat.thinkingFormat: "deepseek"` reaches DeepSeek rendering compat but never installs the DeepSeek replay wrapper on the outgoing stream payload.

## Solution Assessment

Yes. Reusing the existing model compat key as the trigger is the narrow maintainable path; adding more hard-coded Requesty/OpenRouter/Venice-style provider IDs would keep the same class of gap open.

The implementation should not add Requesty, Wisgate, or other proxy provider IDs, should preserve existing provider-owned DeepSeek/OpenRouter/Venice/opencode-go behavior, and should not take on the separate Z.AI visible-response behavior from #76048.
