## Summary

Keep open. Current main has the shared DeepSeek V4 replay wrapper and already supports model-level `compat.thinkingFormat`, but the generic OpenAI-compatible stream path still does not install that wrapper from model compat, so custom providers such as Requesty or OpenRouter remain outside the fix unless they have their own provider wrapper.

## What This Changes

The issue requests a model-level compatibility setting so DeepSeek-style `reasoning_content` handling can be enabled for non-DeepSeek providers and differently named DeepSeek model IDs.

## Best Possible Solution

Add a model-compat-driven DeepSeek reasoning replay path for generic OpenAI-compatible providers, preferably by reusing `compat.thinkingFormat: "deepseek"`; keep provider-specific wrappers such as native DeepSeek, Venice, and opencode-go intact and avoid hard-coding proxy provider ids.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider with a model such as `deepseek/deepseek-v4-pro` and `compat.thinkingFormat: "deepseek"`; current main accepts and renders the compat value but never installs the DeepSeek payload wrapper on the generic stream path.

## Solution Assessment

Yes, model-level opt-in is the maintainable direction. The narrowest repair should reuse the existing compat field unless maintainers explicitly want a public alias, because hard-coding Requesty/OpenRouter-style provider ids would repeat the current failure mode.
