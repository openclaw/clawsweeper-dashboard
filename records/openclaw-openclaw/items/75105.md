## Summary

Keep open. Current main has provider-specific DeepSeek V4 replay fixes and accepts model-level `compat.thinkingFormat`, but it still lacks the requested generic model-level path that lets arbitrary OpenAI-compatible providers such as Requesty opt into DeepSeek-style `reasoning_content` replay.

## What This Changes

Request a per-model compatibility setting/path that applies DeepSeek-style `reasoning_content` replay for non-native OpenAI-compatible provider models instead of only hard-coded provider/model IDs.

## Best Possible Solution

Wire a model-level DeepSeek replay opt-in into the generic OpenAI-compatible stream wrapper path, preferably by consuming existing `compat.thinkingFormat === "deepseek"` or a deliberate alias, and document the supported config.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` model with `compat.thinkingFormat: "deepseek"`, thinking enabled, and replayed assistant history missing `reasoning_content`; current main accepts and renders that compat value but never installs the DeepSeek replay wrapper from it.

## Solution Assessment

Yes. A model-level opt-in is the narrow maintainable direction; the safer implementation should reuse the existing compat surface or intentionally alias the requested spelling instead of hard-coding Requesty, Wisgate, or other proxy providers.
