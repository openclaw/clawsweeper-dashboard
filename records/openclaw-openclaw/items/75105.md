## Summary

Keep open. Current main has a model-level DeepSeek thinking compat value and a shared payload repair wrapper, but custom/OpenAI-compatible providers still lack a generic stream-wrapper path that uses that model compat to replay or strip DeepSeek `reasoning_content`.

## What This Changes

Request model-level configuration that can trigger the DeepSeek `reasoning_content` compatibility behavior for DeepSeek-family models served through custom or proxy providers with non-native provider IDs or prefixed model IDs.

## Best Possible Solution

Use the existing model compat contract as the control point: when a resolved OpenAI-completions model explicitly declares `compat.thinkingFormat: "deepseek"`, install the shared DeepSeek V4 OpenAI-compatible replay wrapper in the generic stream composition, document the custom provider config, and keep proxy-specific behavior separate where a provider rejects native DeepSeek controls.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure a custom `openai-completions` provider model with `compat.thinkingFormat: "deepseek"`; rendering will see the value, but the generic stream chain never installs the shared DeepSeek replay wrapper.

## Solution Assessment

Yes, with one adjustment. The model-level compat direction is the right boundary, but the narrowest maintainable fix should reuse `compat.thinkingFormat: "deepseek"` unless maintainers explicitly want a public `reasoningFormat` alias.
