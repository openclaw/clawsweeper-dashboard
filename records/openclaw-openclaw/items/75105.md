## Summary

Keep open. Current main has the shared DeepSeek V4 replay wrapper and the existing model-level `compat.thinkingFormat` config surface, but custom OpenAI-compatible providers still do not get that wrapper installed from model compat; the remaining gap is exactly the per-model opt-in path requested here.

## What This Changes

Request a model-level compatibility setting so custom OpenAI-compatible providers serving DeepSeek V4 can opt into `reasoning_content` replay behavior without hard-coded provider or model IDs.

## Best Possible Solution

Add a generic model-compat-driven DeepSeek replay path for OpenAI-compatible custom providers. The end state should let a configured model opt into DeepSeek V4 `reasoning_content` replay without provider hard-coding, while keeping existing DeepSeek, Venice, and OpenCode Go provider-owned behavior intact.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider with a model such as `deepseek/deepseek-v4-pro`, `compat.thinkingFormat: "deepseek"`, thinking enabled, and a replayed assistant message missing `reasoning_content`; current main renders DeepSeek thinking format but does not install the DeepSeek payload wrapper on the generic stream path.

## Solution Assessment

Yes on the model-level opt-in direction. The narrowest maintainable fix is likely to reuse the existing `compat.thinkingFormat` surface or add a small alias mapped to it, then wire the shared wrapper into the generic stream-wrapper path without adding Requesty/OpenRouter-specific IDs.
