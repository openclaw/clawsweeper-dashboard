## Summary

Keep open. Current main has the shared DeepSeek V4 replay helper and several named-provider fixes, but it still does not expose the requested model-level path for arbitrary OpenAI-compatible providers such as Requesty.

## What This Changes

Allow model-level compatibility metadata to opt DeepSeek-style OpenAI-compatible models, including proxy/custom providers, into `reasoning_content` replay handling.

## Best Possible Solution

Document and implement a model-level DeepSeek replay opt-in that preserves existing provider-specific behavior and avoids adding more provider IDs.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider/model with `compat.thinkingFormat: "deepseek"`, thinking enabled, and replayed assistant history missing `reasoning_content`; current main renders DeepSeek thinking format but does not install the DeepSeek replay wrapper from that compat value.

## Solution Assessment

Yes. A model-level opt-in is the narrow maintainable direction; the safest implementation should reuse the existing `compat.thinkingFormat === "deepseek"` surface or intentionally alias the requested spelling rather than hard-coding Requesty or other proxy providers.

## Repair Work Prompt

Implement issue #75105 as a focused model-compat-driven DeepSeek V4 `reasoning_content` replay fix. Current main already accepts and preserves `compat.thinkingFormat: "deepseek"` and uses it for OpenAI-completions rendering, but the generic PI stream wrapper path does not install `createDeepSeekV4OpenAICompatibleThinkingWrapper` for custom OpenAI-compatible providers. Add the smallest generic wrapper path, likely in `src/agents/pi-embedded-runner/extra-params.ts`, that applies the existing shared DeepSeek V4 OpenAI-compatible replay wrapper when a model explicitly opts in via model compat. Prefer reusing/documenting `compat.thinkingFormat: "deepseek"`; add a `compat.reasoningFormat` alias only if needed for backward-compatible user ergonomics. Do not add Requesty/OpenRouter/Venice-specific provider IDs in core, and preserve existing native DeepSeek, OpenRouter, Venice, and opencode-go behavior. Add focused tests for a custom provider model with prefixed DeepSeek IDs and replayed assistant messages with and without tool calls, update schema/catalog/docs only if the public config surface changes, and add a user-facing changelog entry if behavior changes.
