## Summary

Keep open. Current main has the model-level `compat.thinkingFormat` surface and the shared DeepSeek wrapper now backfills all assistant replay messages, but custom OpenAI-compatible providers still do not get that wrapper installed from model compat; the linked implementation PR #75619 was closed unmerged.

## What This Changes

Allow a model-level compat setting to apply the DeepSeek `reasoning_content` replay fix for DeepSeek V4 models served through custom or proxy OpenAI-compatible providers.

## Best Possible Solution

Add an explicit model-compat-driven DeepSeek V4 replay path for custom OpenAI-compatible providers, reusing the shared wrapper without hard-coding Requesty, OpenRouter, or other proxy provider IDs.

## Reproduction Assessment

Yes. A source-level reproduction is a custom `openai-completions` provider/model with `compat.thinkingFormat: "deepseek"`, thinking enabled, and replayed assistant history missing `reasoning_content`; current main renders DeepSeek thinking format but does not install the DeepSeek payload wrapper on the generic custom-provider path.

## Solution Assessment

Yes, the model-level opt-in direction is the narrow maintainable solution. Prefer wiring the existing compat surface into the generic stream-wrapper path, with an alias only if maintainers want the requested `reasoningFormat` spelling.
