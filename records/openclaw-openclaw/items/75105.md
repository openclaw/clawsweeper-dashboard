## Summary

Keep open. Current main has the model-level `compat.thinkingFormat` field and the shared DeepSeek V4 payload wrapper, but the generic OpenAI-compatible provider path still does not use model compat to install that wrapper. The native DeepSeek path remains provider/exact-ID gated, and the related fix PR was closed without merging.

## What This Changes

The issue requests a per-model configuration knob that lets custom OpenAI-compatible providers opt into DeepSeek-style `reasoning_content` handling without hard-coded provider or model IDs.

## Best Possible Solution

Add a model-compat-driven DeepSeek replay path for custom OpenAI-compatible providers, preferably by reusing existing `compat.thinkingFormat: "deepseek"` to install the shared DeepSeek V4 payload wrapper without adding provider IDs. Keep native DeepSeek, Venice, and OpenCode Go provider-specific wrappers intact and document the per-model opt-in.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a custom `openai-completions` provider model such as `deepseek/deepseek-v4-pro` with `compat.thinkingFormat: "deepseek"`; current main accepts and renders the compat value but never installs the DeepSeek payload wrapper on the generic stream path.

## Solution Assessment

Yes, model-level opt-in is the narrow maintainable fix. Hard-coding Requesty, OpenRouter, or more proxy provider IDs would repeat the same failure mode; a new public alias should only be added if maintainers prefer that naming over the existing field.
