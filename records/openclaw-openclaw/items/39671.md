## Summary

Keep open: current main does not already include this repo linter, but the PR head still has source-reproducible config bugs and lacks after-fix real behavior proof, so it is not ready to merge.

## What This Changes

The PR adds a root `repotype.yaml`, a `SKILL.md` frontmatter JSON Schema, and canvas skill frontmatter to validate OpenClaw repository structure.

## Best Possible Solution

Adopt repotype only after a maintainer-approved config models OpenClaw skill and plugin layouts with folder-level required-file checks, removes unsupported section assertions, and includes redacted real validation output.

## Reproduction Assessment

Yes. The PR blockers are source-reproducible from PR head `repotype.yaml` and repotype@0.1.5 source: `requiredSections: "#"` maps to a literal `## #`, and missing required files are checked through folder rules rather than file globs.

## Solution Assessment

No. The maintainable path is to use `folders[].requiredFiles` for real skill roots, remove or replace the literal section check, then get maintainer approval and real validation proof before adoption.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Remove the literal section requirements:** `repotype.yaml:20-30`
  - body: `requiredSections: ["#"]` does not prove that a skill has an H1. repotype@0.1.5 extracts only `##`-or-deeper headings and then reports missing `## #`, so this will fail normal `SKILL.md` files instead of validating them.
  - confidence: 0.96
- **[P2] Require skill files with folder rules:** `repotype.yaml:15-25`
  - body: These `files` globs only apply after `SKILL.md` already exists. repotype emits missing-file diagnostics from `folders[].requiredFiles`, so a new skill directory without `SKILL.md` bypasses the advertised structure check.
  - confidence: 0.94
