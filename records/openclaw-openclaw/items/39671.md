## What This Changes

The PR adds a root `repotype.yaml`, a `SKILL.md` frontmatter JSON schema, and canvas skill frontmatter to introduce optional repository structure validation.

## Best Possible Solution

Adopt repotype only if maintainers approve the policy, using folder-level `requiredFiles`, valid section rules or no section rule, OpenClaw skill/plugin path coverage, and recorded validation output.

## Reproduction Assessment

Yes. Source inspection of the PR config plus repotype 0.1.5 shows the file-rule and section-rule defects without running a repo-mutating validation command.

## Solution Assessment

No. The direction may be useful, but this PR is not the best implementation until it follows repotype's folder-rule contract and maintainers approve the new lint policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Use folder rules for required files:** `repotype.yaml:15-42`
  - body: The `files[]` globs only validate paths that already exist. A new skill, package, or plugin directory missing its required file will not match these rules, so the advertised structure enforcement does not happen; model those directories with `folders[].requiredFiles` instead.
  - confidence: 0.94
- **[P2] Remove the literal section requirements:** `repotype.yaml:20-30`
  - body: repotype extracts markdown sections from `##`-or-deeper headings, so `requiredSections: ["#"]` requires a literal `## #` section rather than checking for an H1. Drop this rule or require actual section names present in the matched `SKILL.md` files.
  - confidence: 0.96
