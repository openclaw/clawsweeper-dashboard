## Summary

Keep open: this is a useful repo-lint proposal, but the latest head still has blocking correctness issues in the repotype config and lacks contributor real behavior proof plus maintainer approval for introducing a new repository policy.

## What This Changes

The PR adds a root `repotype.yaml`, a `SKILL.md` frontmatter JSON Schema, and frontmatter for `skills/canvas/SKILL.md` to start optional repository structure validation.

## Best Possible Solution

Adopt repotype only after maintainer policy approval, using folder-level `requiredFiles`, valid section rules or no section rule, coverage for OpenClaw skill/plugin path contracts, and redacted validation output.

## Reproduction Assessment

Yes. Source inspection of PR head plus repotype 0.1.5 shows the file-rule and section-rule defects without needing a repo-mutating validation run.

## Solution Assessment

No. The direction may be useful, but the current config is not the narrowest maintainable solution until it follows repotype's folder-rule contract and maintainers approve the new lint policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Use folder rules for required files:** `repotype.yaml:15-42`
  - body: The `files[]` globs only validate paths that already exist. A new skill, package, or plugin directory missing its required file will not match these rules, so the advertised structure enforcement does not happen; model those directories with `folders[].requiredFiles` instead.
  - confidence: 0.94
- **[P2] Remove the literal section requirements:** `repotype.yaml:20-30`
  - body: repotype extracts only `##`-or-deeper headings, so `requiredSections: ["#"]` requires a literal `## #` section rather than checking for an H1. Drop this rule or require actual section names present in the matched `SKILL.md` files.
  - confidence: 0.96
