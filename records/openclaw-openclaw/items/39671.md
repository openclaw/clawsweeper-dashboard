## Summary

Keep open. The PR is a plausible tooling feature, but the latest head still has merge-blocking repotype config defects and lacks after-fix real behavior proof.

## What This Changes

The branch adds a root `repotype.yaml`, a `schemas/skill.frontmatter.schema.json` schema, and frontmatter for `skills/canvas/SKILL.md` to introduce opt-in repository structure validation.

## Best Possible Solution

Adopt repotype only after a maintainer-approved config models OpenClaw’s actual skill, package, and plugin layouts and passes validation on a clean checkout.

## Reproduction Assessment

Yes. Source inspection of PR head plus repotype@0.1.5 shows the current config treats `#` as a literal required `## #` section and uses file globs where folder `requiredFiles` rules are needed.

## Solution Assessment

No. The safer path is to use folder-level `requiredFiles` rules for real OpenClaw roots, remove or replace the literal section checks, and leave CI adoption as a separate maintainer decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Remove the literal section requirements:** `repotype.yaml:20-21`
  - body: `requiredSections` is compared to headings extracted from `##`-or-deeper headings, so `"#"` requires a literal `## #` section. Existing skill files use an H1 title and normal sections, so validation will report missing sections instead of proving that a heading exists.
  - confidence: 0.96
- **[P2] Require skill roots with folder rules:** `repotype.yaml:15-16`
  - body: This `files` glob only validates `SKILL.md` files that already exist, so a new skill directory without `SKILL.md` bypasses the advertised structure check. Use folder `requiredFiles` rules for skill roots, including the documented grouped layout.
  - confidence: 0.94
