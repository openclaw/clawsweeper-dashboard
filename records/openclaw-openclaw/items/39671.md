## What This Changes

This PR adds a root repotype configuration, a SKILL.md frontmatter JSON Schema, and canvas skill frontmatter to validate OpenClaw repository structure.

## Best Possible Solution

Adopt repotype only after a maintainer-approved config uses folder-level required-file checks, removes the unsupported literal section assertions, and includes redacted real validation output.

## Reproduction Assessment

Yes. The PR blockers are source-reproducible: repotype@0.1.5 applies `requiredSections` to extracted `##`-level headings and missing-file checks come from `folders[].requiredFiles`, while this PR uses file globs for `SKILL.md`.

## Solution Assessment

No. The direction may be useful, but this is not the best mergeable version while it uses literal section requirements and existing-file globs for missing-file enforcement; folder rules plus real proof and maintainer approval are the safer path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Remove the literal section requirements:** `repotype.yaml:20-30`
  - body: `repotype@0.1.5` extracts only `##`-or-deeper headings and reports missing sections as `## ${section}`, so `requiredSections: ["#"]` makes normal `SKILL.md` files fail with missing `## #` instead of checking for an H1. Drop this rule or require a real section name.
  - confidence: 0.96
- **[P2] Require skill files with folder rules:** `repotype.yaml:15-25`
  - body: These `files` globs only run for existing `SKILL.md` files. A new `skills/foo/` or plugin skill directory without `SKILL.md` will not hit this rule, because repotype emits missing-file diagnostics from `folders[].requiredFiles`; model the skill roots as folder rules before claiming missing skills are enforced.
  - confidence: 0.94
