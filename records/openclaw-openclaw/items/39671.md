## Summary

Keep open: current main does not already contain this repo-structure linter, and the PR still has source-reproducible correctness blockers plus missing real behavior proof before maintainers can consider adopting the tool.

## What This Changes

The PR adds a root `repotype.yaml`, a skill frontmatter JSON schema, and canvas skill frontmatter to introduce opt-in repository structure validation.

## Best Possible Solution

Adopt repotype only after the config models OpenClaw’s actual skill and plugin layouts, passes real validation on a clean checkout, and keeps CI enforcement as a separate maintainer decision.

## Reproduction Assessment

Yes. The blockers are source-reproducible from PR head `repotype.yaml` and repotype@0.1.5: `requiredSections: "#"` requires a literal `## #`, and missing skill files must be enforced through folder `requiredFiles`, not existing-file globs.

## Solution Assessment

No. The direction may be useful, but the mergeable solution should remove or replace the literal section checks, use folder-level required-file rules for real skill roots, cover grouped skills, and then provide real validation output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Remove the literal section requirements:** `repotype.yaml:20-30`
  - body: `requiredSections` is compared with H2-or-deeper heading text, so `"#"` requires a literal `## #` section. Normal skill files use `# Title` plus real `## Overview` sections, so validation reports `missing_section` instead of proving that a heading exists.
  - confidence: 0.96
- **[P2] Require skill files with folder rules:** `repotype.yaml:15-25`
  - body: The advertised check that every skill has `SKILL.md` is not enforced by a `files` glob; repotype only validates files that already exist, and missing files are reported from `folders[].requiredFiles`. A new skill directory without `SKILL.md` can bypass this rule.
  - confidence: 0.94
- **[P2] Cover grouped workspace skills:** `repotype.yaml:16`
  - body: Current OpenClaw supports `skills/<group>/<skill>/SKILL.md`, but this schema rule only matches `skills/*/SKILL.md`. Valid grouped workspace skills would skip the frontmatter and structure checks this config is meant to enforce.
  - confidence: 0.9
