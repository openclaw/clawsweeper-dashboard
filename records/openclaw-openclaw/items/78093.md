## Summary

Keep open: current main still has the nested non-default workspace fallback, and this issue is paired with an open same-author closing PR plus a related canonical workspace-default PR that should be reviewed, refreshed, or superseded before closure.

## What This Changes

The issue asks to keep `agents add` from defaulting non-default agent workspaces inside `agents.defaults.workspace`, because nested paths can make TUI workspace-to-agent routing ambiguous.

## Best Possible Solution

Land one compatibility-reviewed fix that preserves the default agent's configured workspace, gives brand-new non-default agents an unambiguous sibling workspace default, updates resolver/command/TUI regression coverage, and then closes this issue with the related workspace-default thread.

## Reproduction Assessment

Yes. Source inspection on current main shows that configuring `agents.defaults.workspace` and running `openclaw agents add newbot` will seed the prompt from the resolver branch that appends `newbot` under that workspace; TUI workspace inference then sees containment matches for both paths.

## Solution Assessment

Yes in direction, but this issue should stay open until one implementation path lands. The maintainable fix should choose the compatibility boundary, update focused coverage, and avoid running a parallel repair while open PRs already target the same bug.
