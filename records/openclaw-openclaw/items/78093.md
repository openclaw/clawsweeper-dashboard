## Summary

Keep open: current main still has the nested non-default workspace fallback, and this issue is linked to open implementation PRs that should be reviewed, refreshed, or superseded before the issue closes.

## What This Changes

The issue reports that `openclaw agents add` suggests nested non-default workspaces under `agents.defaults.workspace`, causing ambiguous TUI workspace routing.

## Best Possible Solution

Land one compatibility-reviewed fix that preserves the default agent's configured workspace, gives new non-default agents an unambiguous sibling workspace default, updates resolver/command/TUI coverage, and then closes this issue with the related workspace-default thread.

## Reproduction Assessment

Yes. Source inspection on current main shows that configuring `agents.defaults.workspace` and running `openclaw agents add newbot` will seed the prompt from the resolver branch that appends `newbot` under that workspace; TUI workspace inference can then see containment matches for both paths.

## Solution Assessment

Yes in direction, but the issue should stay open until one implementation path lands. The maintainable fix should choose the compatibility boundary, update focused coverage, and avoid a parallel repair while open PRs already target the same bug.
