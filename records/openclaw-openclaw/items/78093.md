## Summary

Keep open: current main still derives an interactive non-default agent workspace as a child of `agents.defaults.workspace`, which conflicts with the documented `workspace-<agentId>` default and creates the reported routing overlap.

## What This Changes

Restore sibling default workspaces for non-default agents while preserving `agents.defaults.workspace` for the configured default agent.

## Best Possible Solution

Make the shared workspace resolver return `agents.defaults.workspace` only for the configured default agent, and keep non-default implicit workspaces on the sibling `workspace-<agentId>` convention across core and the memory-host SDK mirror.

## Reproduction Assessment

Yes from source inspection: with `agents.defaults.workspace` set, `agents add` uses `resolveAgentWorkspaceDir` as the wizard default, and that helper currently returns a child path for non-default agents. A path inside that child also satisfies the parent workspace containment check used by TUI agent inference.

## Solution Assessment

Yes; the narrow maintainable fix is to correct the resolver fallback and update the regression tests, including the memory-host SDK copy of the same helper. No new config or product policy is needed.
