## Summary

Keep open: current main still has the reported nested default path for non-default agents when `agents.defaults.workspace` is configured, and `agents add` uses that resolver as the interactive default. The TUI path resolver also treats nested workspaces as multiple matches ordered by specificity, so the ambiguity is source-reproducible even though the most-specific agent currently wins.

## What This Changes

Restore non-default agent workspace defaults to a sibling `workspace-<id>` path while preserving `agents.defaults.workspace` for the configured default agent.

## Best Possible Solution

Keep the issue open and change workspace defaulting so only the configured default agent uses `agents.defaults.workspace` directly, while newly added non-default agents default to a sibling state-dir `workspace-<id>` path with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `agents.defaults.workspace`, run interactive `agents add newbot`, and the prompt default comes from `resolveAgentWorkspaceDir`, which currently nests `newbot` under the default workspace.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust the non-default fallback in `resolveAgentWorkspaceDir`, update the #59789 regression to cover default-agent behavior only, and add coverage for the `agents add` default suggestion and TUI inference boundary.
