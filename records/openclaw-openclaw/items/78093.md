## Summary

Keep open: current main still has the reported nested non-default workspace fallback, and the issue is paired with an open closing PR that should be reviewed, refreshed, or superseded before the issue closes.

## What This Changes

The issue asks for non-default agents created by `openclaw agents add` to default to sibling `workspace-<id>` directories instead of nesting under `agents.defaults.workspace`.

## Best Possible Solution

Land one focused fix that preserves the default agent’s configured workspace while making newly added non-default agents use unambiguous peer workspaces, with resolver and wizard coverage aligned to the chosen compatibility boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows `agents.defaults.workspace` is joined with the non-default agent id, and `agents add` uses that resolver value as the prompt default; no runtime CLI command was run in this read-only review.

## Solution Assessment

Yes for the target behavior, but the implementation boundary needs maintainer choice. The safe end state is sibling defaults for newly added non-default agents while preserving explicit workspaces and the configured default-agent workspace.
