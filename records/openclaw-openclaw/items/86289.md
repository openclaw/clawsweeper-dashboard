## Summary

Keep open: this PR is the active linked fix path for the Control UI session picker bug, but it is not merge-ready because the shared override helper still reads the Agents tab selection and the PR lacks an inspectable real behavior proof artifact.

## What This Changes

The PR filters Control UI chat session-list and picker RPC requests by an agentId so the chat session picker is scoped to the active agent.

## Best Possible Solution

Before merge, make every chat session-list refresh derive the filter from the active chat session key, keep Agents-tab state out of the shared helper, add focused regression coverage, and attach redacted real behavior proof.

## Reproduction Assessment

Yes from source: current main builds chat session-list overrides without agentId, loadSessions forwards agentId when supplied, and the picker renders the loaded sessionsResult rows directly. I did not run a live Control UI instance in this read-only review.

## Solution Assessment

No; the PR moves in the right direction, but the best fix is to centralize or pass a chat-session-derived agentId for every chat session-list load and avoid using agentsSelectedId as a fallback source.

Full review comments:

- **[P2] Derive the helper filter from the chat session:** `ui/src/ui/app-chat.ts:108-110`
  - body: `createChatSessionsLoadOverrides()` now reads `agentsSelectedId`, but callers such as `refreshChat()` and gateway event refreshes still use this helper directly. If the Agents tab is on agent B while Chat is showing an agent A session, those refreshes will load B's sessions into `sessionsResult`; the picker falls back to `sessionsResult`, so the active chat's sessions can still disappear or show the wrong scope. Remove this Agents-tab source or pass a chat-session-derived `agentId` for every helper caller.
