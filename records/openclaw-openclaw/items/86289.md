## Summary

Keep this PR open: it fixes the right Control UI session-picker path, but the branch still adds `agentId` from the Agents tab selection to a shared chat session-load helper, which can scope unrelated chat refreshes to the wrong agent and hide the active session from UI state.

## What This Changes

The branch adds active-agent `agentId` filtering to Control UI chat session picker and refresh requests, exports `resolveChatAgentFilterId`, and updates focused UI tests.

## Best Possible Solution

Keep the shared chat session-load helper agent-agnostic, add `agentId` from `resolveChatAgentFilterId(state, state.sessionKey)` only at active chat picker/refresh call sites, and cover the divergent Agents-tab versus active-chat state.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current picker params omit `agentId`, while `sessions.list` already supports `agentId` and `loadSessions` forwards it. I did not run a live UI reproduction in this read-only review.

## Solution Assessment

No, not yet. Passing the active chat agent to the picker/refresh call sites is the right direction, but putting `agentsSelectedId` into the shared helper is not the narrowest maintainable fix.
