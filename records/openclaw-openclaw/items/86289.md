## Summary

This PR should stay open: it fixes the right Control UI bug shape and now uses the active chat session for the picker paths, but the branch still leaves an `agentsSelectedId` filter in the shared chat session-load helper, which can reintroduce the same wrong-source-of-truth problem in non-picker refresh paths.

## What This Changes

The PR adds `agentId` filtering to Control UI chat session-list and picker `sessions.list` requests and updates focused UI tests for the new active-agent parameter.

## Best Possible Solution

Keep the picker and explicit chat-session refresh paths filtered by the active `sessionKey` agent, but remove the `agentsSelectedId` branch from the shared helper so unrelated refresh paths do not inherit Agents tab state.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: current main builds chat picker `sessions.list` params without `agentId`, while the backend and UI controller already support that filter; the linked issue provides a concrete multi-agent Control UI path, though I did not live-run it in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not quite as submitted: deriving picker filters from `resolveChatAgentFilterId(state, state.sessionKey)` is the right direction, but the shared helper should not derive scope from `agentsSelectedId`.

- **[P2] Remove the agents-panel filter from the shared helper:** `ui/src/ui/app-chat.ts:108-111`
  - body: This helper is used by `refreshChat`, gateway event refreshes, and other non-picker session refreshes.
