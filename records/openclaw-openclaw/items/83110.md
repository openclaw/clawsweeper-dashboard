## Label Justifications

- `P2`: This is a focused cron reliability bug with real user impact but limited surface area around isolated agent-turn watchdog timing.
- `impact:crash-loop`: The reported failure aborts otherwise healthy cron runs early and can make scheduled automation repeatedly fail on modest hardware.

## Summary

Keep this issue open: current main still contains the half-timeout pre-execution watchdog formula, and the same author has an open paired PR for that exact change. Source inspection narrows the likely failure mode: embedded tool starts already clear the watchdog, but CLI-backed cron startup can remain in the pre-execution state until `process_spawned`, so the remaining fix needs maintainer review rather than cleanup closure.

## What This Changes

The issue asks cron to stop computing the isolated-agent pre-execution watchdog with `Math.floor(jobTimeoutMs / 2)` because long silent tool or CLI work can be aborted before the configured `timeoutSeconds`.

## Best Possible Solution

Keep this issue open until maintainers resolve the paired PRs or choose a narrower cron fix that proves active CLI and embedded execution clear the pre-execution watchdog while true pre-execution stalls still fail fast.

## Reproduction Assessment

Partly yes by source inspection: an isolated CLI-backed cron run can enter `waiting_for_execution` after `onExecutionStarted()` and stay there until `process_spawned`, so the pre-execution formula/cap can abort early. I did not live-reproduce the reporter's `rclone` job, and the embedded post-`tool_execution_started` path appears to clear the watchdog.

## Solution Assessment

Unclear. Removing `/2` may be one piece, but the safer maintainable solution should coordinate the linked formula and cap/config work and add regression coverage for silent CLI startup and long-running execution milestones.
