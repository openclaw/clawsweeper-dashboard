## Summary

Keep open: current main still has the half-timeout/capped pre-execution watchdog formula, and source inspection shows a plausible CLI-backed isolated-cron path that can remain under that watchdog after runner start. The embedded tool path does emit execution milestones, so the exact root needs maintainer review with the paired open PR and overlapping watchdog-config work rather than cleanup closure.

## What This Changes

The issue requests changing cron's pre-execution watchdog calculation so silent long-running isolated agent/tool executions are not aborted before the configured `timeoutSeconds`.

## Best Possible Solution

Preserve fast setup/pre-execution stall detection, but ensure real CLI/tool execution emits or maps an execution-start milestone before long silent work; if maintainers choose timeout tuning instead, land it through the reviewed config/defaults path with regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection shows a plausible CLI-backed isolated-cron path that can remain in `waiting_for_execution`, while the embedded tool path already clears the watchdog on `tool_execution_started`.

## Solution Assessment

No. Removing `/2` alone is not the best proven fix because the 60s cap still dominates many reported jobs and embedded tools already emit a clearing milestone; a narrower execution-milestone fix or an approved config/cap change needs maintainer review.
