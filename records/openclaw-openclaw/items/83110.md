## What This Changes

The issue asks for the cron pre-execution watchdog formula to stop aborting silent long-running work before the configured `timeoutSeconds` budget.

## Best Possible Solution

Resolve the paired PRs as one cron-watchdog decision: either land a complete fix with formula/cap or missing-phase coverage, or close the PRs and this issue together with maintainer rationale for preserving the current pre-execution cap.

## Reproduction Assessment

Yes for the source-level pre-execution boundary, but not for the exact embedded silent-tool claim. Current code can remain in `waiting_for_execution` before a CLI/process execution milestone and then uses the half-timeout/capped resolver; embedded tool starts already clear that watchdog.

## Solution Assessment

No, the one-line formula change alone is not the best complete fix. The cap/configuration behavior and execution milestone coverage need to be handled with regression tests so true pre-execution stalls still fail quickly while active work is bounded by the job timeout.
