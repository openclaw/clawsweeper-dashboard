## Summary

Keep open for maintainer/security design: current main and the latest release still have prompt-level external-content wrappers and a blocking `before_tool_call` hook, but they do not carry an `external_content` action source through tool-call events or enforce the proposed source-aware execution gate. The Liquefy and Colors comments are useful prior-art signals, and https://github.com/openclaw/openclaw/pull/91800 is a partial implementation candidate, but the remaining API, policy, and compatibility choices are not settled.

## What This Changes

The issue proposes source-aware instruction tagging so external content can inform responses without being allowed to authorize tool execution.

## Best Possible Solution

Define a coordinated provenance and tool-policy contract that tags external content at ingress, preserves enough source context through model/tool decisions, exposes stable hook or approval fields, and enforces risky content-derived actions with explicit operator-controlled exceptions.

## Reproduction Assessment

Yes at source level, not as a live exploit reproduction. Current main's hook type, event builder, input provenance helper, and docs show no external-content action-source field or execution gate.

## Solution Assessment

Unclear as a direct implementation. The security direction is aligned, but a single `action.source` deny check is too narrow until maintainers define the provenance model, plugin API, approval behavior, and compatibility path.
