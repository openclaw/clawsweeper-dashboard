## Summary

Keep open for maintainer and security architecture review: current main and the latest release harden external content with prompt-level wrappers and limited input provenance, but they do not carry external-content provenance through tool-call events or block tool execution based on an action source. The Liquefy and Colors comments are useful prior-art signals, but they strengthen the case for design review rather than making this already implemented or safely outsourceable.

## What This Changes

The issue requests source-aware instruction provenance and an execution-gate policy that blocks tool calls derived from external content.

## Best Possible Solution

Design one core provenance and tool-policy contract that tags external content at ingress, preserves that provenance through model/tool decisions, exposes stable plugin/security hook fields, and enforces high-risk actions with tests, docs, and operator-controlled exceptions.

## Reproduction Assessment

No, not as a live exploit reproduction. Source inspection does give high confidence that current main and `v2026.6.6` lack the requested action-source provenance and external-content execution gate.

## Solution Assessment

Unclear as a direct implementation. The security direction is aligned, but a single `action.source` gate is probably too narrow for the existing wrappers, gateway invocation path, plugin hooks, approvals, and Codex/OpenClaw tool runtimes.
