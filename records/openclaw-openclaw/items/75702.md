## Summary

Keep open. The PR is a focused and mostly correct browser snapshot timeout propagation fix with sufficient terminal proof, but it still leaves the CDP fallback/direct aria snapshot branches on the helper default timeout instead of the requested/default snapshot timeout.

## What This Changes

The PR threads browser snapshot `timeoutMs` through the agent action, browser client/proxy request, snapshot route plan, Chrome MCP and Playwright helpers, focused tests, and changelog.

## Best Possible Solution

Finish the same timeout propagation by forwarding `plan.timeoutMs` to the CDP snapshot fallback branches, then land the focused browser plugin fix and keep the broader Linux all-action timeout work tracked separately.

## Reproduction Assessment

Yes, source-level. Current main accepts `timeoutMs` but drops it before the snapshot client, plan, route, Chrome MCP, and several helper calls; I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No, not quite. Threading one normalized timeout through the snapshot stack is the right approach, but the route also needs to pass it to `snapshotRoleViaCdp` and direct `snapshotAria` CDP branches before the fix is complete.
