## Summary

Keep open: current main does not contain the proposed scanner rules, but the PR still has blocking scanner-accuracy issues and lacks real behavior proof for a security-sensitive install scanner change.

## What This Changes

The PR adds four regex-based skill scanner rules and unit cases for network listeners, prototype pollution, symlink calls, and credential-file reads.

## Best Possible Solution

Revise the core scanner rules with mutation-aware prototype matching, scoped Node API/context checks, credential paths tied to readFile arguments, regression tests, and security-owner review, or fold the work into a canonical scanner-hardening PR.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the current scanner checks contexts source-wide, and the PR regexes can miss direct prototype writes while warning on unrelated credential or listener context.

## Solution Assessment

No. The core scanner is the right surface, but the proposed source-wide regexes and overbroad/incomplete prototype rule are not the narrowest maintainable security fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict prototype detection to mutations:** `src/security/skill-scanner.ts:184-185`
  - body: The critical prototype-pollution rule matches any __proto__ or constructor["prototype"] access, including read-only checks. Because critical scanner findings can block installs, require an assignment or mutation shape before emitting this rule.
  - confidence: 0.89
- **[P2] Cover direct prototype-pollution writes:** `src/security/skill-scanner.ts:184-185`
  - body: The prototype-pollution pattern catches bracket notation but misses common direct writes such as obj.__proto__.polluted = true and Object.setPrototypeOf(obj, payload). Broaden the mutation patterns or narrow the rule scope and tests.
  - confidence: 0.82
- **[P2] Match credential paths in readFile arguments:** `src/security/skill-scanner.ts:230-231`
  - body: SOURCE_RULES checks the credential path and readFile context independently across the whole file, so a help string mentioning .env plus an unrelated readFileSync("README.md") can warn. Match the credential path inside the file-read expression or same-line argument.
  - confidence: 0.9
- **[P2] Constrain listener detection to real Node server APIs:** `src/security/skill-scanner.ts:178-179`
  - body: The network-listener context regex matches http, https, net, http2, or tls anywhere in the source, including URL strings or comments. Require a real Node import/binding or a qualified call like http.createServer before warning.
  - confidence: 0.86
- **[P3] Tie symlink detection to fs calls:** `src/security/skill-scanner.ts:194-195`
  - body: The symlink rule has the same source-wide context problem: any fs or node:fs token elsewhere enables warnings for local functions named readlink or symlink. Require a real fs import/binding or a qualified fs call before reporting it.
  - confidence: 0.78
