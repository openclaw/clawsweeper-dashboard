## Summary

Keep open: current main does not contain the proposed scanner rules, but the PR changes install-time security scanning and still has blocking false-positive/false-negative issues plus missing real behavior proof.

## What This Changes

This PR adds four regex-based skill scanner rules and tests for network listeners, prototype pollution, symlink calls, and credential-file reads.

## Best Possible Solution

Revise the scanner hardening into context-scoped, mutation-aware rules with positive and false-positive regression tests plus secops review before merging.

## Reproduction Assessment

Yes. Source inspection shows the current scanner applies broad source-level context checks, and focused scanSource cases can demonstrate the proposed false positives and false negatives.

## Solution Assessment

No. The core scanner is the right surface, but these regex rules need tighter API/mutation matching and coordination with the related scanner-hardening PRs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict prototype detection to mutations:** `src/security/skill-scanner.ts:184-185`
  - body: The critical prototype-pollution rule matches any __proto__ or constructor["prototype"] access, including read-only checks. Because critical scanner findings can block installs, require an assignment or mutation shape before emitting this rule.
  - confidence: 0.89
- **[P2] Cover direct prototype-pollution writes:** `src/security/skill-scanner.ts:184-185`
  - body: The prototype-pollution pattern catches bracket notation but misses common direct writes such as obj.__proto__.polluted = true and Object.setPrototypeOf(obj, payload). Broaden the mutation patterns or narrow the advertised rule scope and tests.
  - confidence: 0.83
- **[P2] Match credential paths in readFile arguments:** `src/security/skill-scanner.ts:227-228`
  - body: SOURCE_RULES checks credential paths and readFile context independently across the whole file, so a string mentioning .env plus an unrelated readFileSync("README.md") can warn. Tie the credential path to the file-read argument or same expression.
  - confidence: 0.9
- **[P2] Constrain listener detection to Node server APIs:** `src/security/skill-scanner.ts:177-178`
  - body: The network-listener context regex matches http, https, net, http2, or tls anywhere in the source, including URL strings or comments. Require a real Node import/binding or a qualified call like http.createServer before warning.
  - confidence: 0.86
- **[P3] Tie symlink detection to fs calls:** `src/security/skill-scanner.ts:192-193`
  - body: The symlink rule has the same source-wide context problem: any fs or node:fs token elsewhere enables warnings for local functions named readlink or symlink. Require a real fs binding or a qualified fs call before reporting it.
  - confidence: 0.78
