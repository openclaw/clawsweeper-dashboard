## Summary

Keep open. Current main does not implement the requested scanner rule IDs, so this is not an implemented-on-main cleanup candidate; the proposed patch also has concrete security-scanner accuracy problems that should be revised before merge. Prior review discussion already identified the prototype-pollution gap, and the later Codex review expands that into actionable false-positive and false-negative findings that are still relevant.

## What This Changes

This PR adds four regex-based skill scanner rules for network listeners, prototype pollution, symlink calls, and credential-file access, plus positive and negative scanner tests.

## Best Possible Solution

Keep the work in core scanner hardening, but revise it into a tighter detection change: bind network and symlink detections to real Node imports or qualified calls, match credential paths inside the actual file-read argument or expression, make prototype-pollution detection mutation-oriented while covering common dot-notation and Object.setPrototypeOf forms, and add regression tests for each false-positive and false-negative shape before merge.

## Reproduction Assessment

Yes. The reproduction path is source-level: scanSource applies line-rule requiresContext checks and source-rule pattern/context checks against the whole source, so small snippets with unrelated http/readFile/fs context reproduce the proposed false positives, while dot-notation prototype writes reproduce the false negative.

## Solution Assessment

No. The existing core scanner is the right surface, but the current PR's source-wide regex implementation is not the narrowest maintainable solution for security scanning because it reports unrelated context and misses common mutation forms.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Restrict prototype detection to mutations:** `src/security/skill-scanner.ts:187-188`
  - body: This pattern reports any __proto__ or constructor["prototype"] access as critical, including read-only defensive checks. Since critical scanner findings block gateway-backed installs by default, require an assignment or other mutation shape before emitting this rule.
  - confidence: 0.89
- **[P2] Cover direct prototype-pollution writes:** `src/security/skill-scanner.ts:187-188`
  - body: The new prototype-pollution rule only catches bracket notation, so direct writes like obj.__proto__.polluted = true or Object.setPrototypeOf(obj, payload) still bypass the scanner. Broaden the mutation patterns or narrow the rule scope and tests.
  - confidence: 0.82
- **[P2] Tie credential paths to file-read arguments:** `src/security/skill-scanner.ts:233-234`
  - body: SOURCE_RULES checks the credential path pattern and readFile context independently across the whole file, so documentation text mentioning .env plus an unrelated readFileSync("README.md") can warn. Match the credential path in the actual readFile/readFileSync argument or same expression.
  - confidence: 0.9
- **[P2] Constrain network listener context to real APIs:** `src/security/skill-scanner.ts:180-181`
  - body: The network-listener context regex matches http, https, net, http2, or tls anywhere in the source, including URL strings and comments, so a local createServer helper can be flagged as a listener. Require an import/require binding or a qualified call like http.createServer.
  - confidence: 0.86
- **[P3] Tie symlink detection to fs calls:** `src/security/skill-scanner.ts:193-194`
  - body: The symlink rule has the same source-wide context problem: any fs or node:fs token elsewhere enables warnings for local functions named readlink or symlink. Require a real fs import/binding or qualified fs call before reporting this rule.
  - confidence: 0.78
