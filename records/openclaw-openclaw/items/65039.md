## Summary

Keep open. Current main still preserves the reported DAVE-disabled pre-Ready join failure boundary, and the shipped timeout/reconnect fix only addressed the broader slow-Ready cluster while referencing this issue rather than resolving its non-DAVE handshake policy.

## What This Changes

The issue reports Discord `/vc join` aborting after voice WebSocket Hello but before Ready/UDP handshake when `channels.discord.voice.daveEncryption` is disabled.

## Best Possible Solution

Make the Discord voice join policy explicit: either support DAVE-disabled joins with tested Hello-to-Ready behavior, or fail early with a clear diagnostic and docs that say non-DAVE joins are unsupported.

## Reproduction Assessment

Yes, with source-level confidence. The issue includes live repro steps and a detailed trace, and current main still has a path where a pre-Ready `entersState()` rejection returns the same `/vc join` failure.

## Solution Assessment

Unclear. A focused code change is possible, but the maintainable direction depends on whether maintainers want `daveEncryption=false` to remain a supported Discord voice join mode or become an intentional early rejection.
