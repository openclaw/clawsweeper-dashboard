## Summary

Keep open. Current main still matches the reported DAVE-disabled pre-Ready join abort path: OpenClaw forwards `daveEncryption=false`, waits for `Ready`, destroys on a pre-Ready rejection, and only has DAVE receive recovery after the connection is already ready.

## What This Changes

The issue asks OpenClaw to fix Discord `/vc join` aborting after the voice WebSocket Hello but before Ready/UDP handshaking when `channels.discord.voice.daveEncryption=false`.

## Best Possible Solution

Make the Discord voice join policy explicit: either support DAVE-disabled joins with tested Hello-to-Ready recovery, or reject/warn on unsupported non-DAVE joins before starting the voice connection and update docs/tests accordingly.

## Reproduction Assessment

Yes, with source-level confidence. The issue includes live repro steps and a detailed trace, and current main still has a path where a pre-Ready `entersState()` rejection returns the same `/vc join` failure.

## Solution Assessment

Unclear. The shipped timing/config fix is a useful partial repair for slow Ready or reconnect paths, but it does not decide or test what OpenClaw should do when `daveEncryption=false` closes after Hello before Ready.
