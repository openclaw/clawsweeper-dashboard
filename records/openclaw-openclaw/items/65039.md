## Summary

Keep open. Current main still leaves the reported DAVE-disabled pre-Ready join path unresolved: OpenClaw forwards `daveEncryption=false`, waits for `Ready`, destroys on a pre-Ready rejection, and only has DAVE receive recovery after the connection is already ready.

## What This Changes

The issue reports a Discord `/vc join` failure where the DAVE-disabled voice WebSocket closes after Hello and before the Ready/UDP handshake stage.

## Best Possible Solution

Make the Discord voice join policy explicit: either support DAVE-disabled joins with tested Hello-to-Ready recovery, or reject/warn on unsupported non-DAVE joins before starting the voice connection and update docs/tests accordingly.

## Reproduction Assessment

Yes, with source-level confidence. The issue includes live repro steps and a detailed trace, and current main still has the same pre-Ready `entersState()` rejection path that returns the reported `/vc join` failure.

## Solution Assessment

Unclear. The shipped timing/config fix is useful for slow Ready or reconnect paths, but it does not decide or test what OpenClaw should do when `daveEncryption=false` closes after Hello before Ready.
