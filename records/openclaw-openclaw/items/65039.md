## Summary

Keep open. Current main partially addressed the broader slow-handshake cluster with configurable Discord voice timing, but this issue's DAVE-disabled Hello-before-Ready close is still unresolved because the plugin still forwards `daveEncryption=false`, waits for Ready, destroys on pre-Ready failure, and only has DAVE receive recovery after Ready.

## What This Changes

Fix Discord `/vc join` aborting before Ready on the DAVE-disabled path after the voice WebSocket Hello and before UDP handshaking.

## Best Possible Solution

Make the Discord voice join policy explicit in the Discord plugin: either support DAVE-disabled joins with tested handshake recovery, or reject/warn on unsupported non-DAVE joins before starting the voice connection.

## Reproduction Assessment

Yes. The issue includes concrete live steps and a high-signal trace, and current main has a source-level reproduction path where a pre-Ready `entersState()` rejection returns the same user-facing join failure.

## Solution Assessment

Unclear. The current #63098 timing fix is a useful partial repair for slow Ready/reconnect paths, but it does not prove that a DAVE-disabled WebSocket close after Hello can now reach Ready or fail with an intentional unsupported-mode diagnostic.
