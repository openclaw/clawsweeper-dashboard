## Summary

Keep open. Current main still has the reported pre-Ready Discord voice join failure surface: the Discord plugin forwards `daveEncryption=false`, waits for a fixed Ready transition, destroys the connection on `entersState` failure, and only has DAVE recovery paths after the connection is already Ready. The linked receive/reply recovery PR fixed a later pipeline stage, while #39825 and #63098 remain overlapping open reports rather than a clearly better canonical replacement.

## What This Changes

The issue reports that Discord `/vc join` aborts before the voice connection reaches Ready when `daveEncryption` is disabled, after the voice WebSocket Hello and before UDP handshaking.

## Best Possible Solution

Make the Discord voice join policy explicit in the Discord plugin: either support DAVE-disabled joins reliably with a tested handshake/timeout/retry path, or reject/warn on unsupported non-DAVE joins before starting the voice connection. Keep post-Ready receive recovery separate and coordinate the final behavior with #39825 and #63098.

## Reproduction Assessment

Yes. The issue includes concrete live steps and a high-signal trace, and current main has a source-level reproduction path where a pre-Ready `entersState()` abort produces the same user-facing `Failed to join voice channel` response.

## Solution Assessment

Unclear. There is no patch to assess, and a timeout knob alone may help #63098 without resolving the DAVE-disabled Hello-before-Ready close; maintainers should first decide the supported non-DAVE join behavior.
