## Summary

Keep open. Current main still preserves the reported DAVE-disabled pre-Ready failure path: OpenClaw passes `daveEncryption=false` to `@discordjs/voice`, waits for `Ready`, and destroys the connection on a pre-Ready rejection, while existing recovery only applies after the connection is already ready.

## What This Changes

The issue reports Discord `/vc join` aborting after voice WebSocket Hello and before Ready/UDP handshaking when `channels.discord.voice.daveEncryption=false`.

## Best Possible Solution

Make the Discord voice join policy explicit: either support DAVE-disabled joins through Ready with regression coverage, or reject/warn on unsupported non-DAVE joins before starting the voice connection and update docs/tests accordingly.

## Reproduction Assessment

Yes, with source-level confidence. The reporter supplied live steps and a high-signal trace, and current main still has a `daveEncryption=false` pre-Ready rejection path that returns the same `/vc join` failure.

## Solution Assessment

Unclear. The shipped timing/config changes are useful for slow Ready and reconnect cases, but they do not decide or test the non-DAVE Hello-before-Ready failure mode.
