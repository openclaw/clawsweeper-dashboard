## Summary

Keep open: the PR targets a real P1 Windows media/image event-loop stall and includes convincing packaged Windows probe output, but the current head still broadens local media attachment access by treating the agent workspace as a fallback media workspace/root.

## What This Changes

The PR adds a fast image-model resolution path, threads workspaceDir through media-understanding/provider auth, adjusts provider stream lookup, and adds tests plus a changelog entry.

## Best Possible Solution

Land the fast image-resolution path after separating provider/auth workspaceDir from media attachment authorization and adding a regression that relative MediaPath without MediaWorkspaceDir remains blocked.

## Reproduction Assessment

Yes, source/probe reproducible enough for PR review: current main performs broad image model preparation before payload construction, and the linked issue plus PR body provide concrete Windows stall/probe evidence. I did not rerun the Windows VM probe in this read-only pass.

## Solution Assessment

No, not as-is. The fast image-model path is the right direction, but media attachment authorization must stay limited to MediaWorkspaceDir, default media roots, and channel-configured roots while workspaceDir is used only for provider/auth scoping.
