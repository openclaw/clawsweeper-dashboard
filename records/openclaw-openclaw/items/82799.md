## Summary

Keep open. The PR targets a real Windows image-prep event-loop stall and includes sufficient Windows packaged-install proof, but the protected `maintainer` label and a concrete media-root authorization regression make it not ready to merge.

## What This Changes

The branch adds a fast static image-model resolution path, threads `workspaceDir` through media-understanding/provider auth, updates provider stream lookup plumbing, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the fast image-model resolution path, but keep provider/auth `workspaceDir` separate from media attachment authorization so local reads remain limited to `MediaWorkspaceDir`, default media roots, and channel-configured roots.

## Reproduction Assessment

Yes, source-reproducible enough for review: current main does broad model/plugin preparation before image payload construction, and the linked issue provides concrete Windows Telegram event-loop starvation logs. I did not rerun the Windows VM probe in this read-only pass.

## Solution Assessment

No, not as-is. The fast static image-model path is the right direction, but the agent workspace must not become a fallback media attachment root when `MediaWorkspaceDir` is absent.

## Review Findings

Overall correctness: patch is incorrect
