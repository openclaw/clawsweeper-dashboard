## Summary

This PR should stay open because the proposed gateway early-return is too broad for the current session resolution contract and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds an early `key === "current"` INVALID_REQUEST response in the gateway session resolver and a unit test asserting that no session-store lookup occurs.

## Best Possible Solution

Resolve `current` before the WS key lookup when requester context is available, while preserving literal `current` key/sessionId behavior and adding regression coverage for both paths.

## Reproduction Assessment

Yes from source inspection: `resolveSessionReference` sends `sessions.resolve` with `key: "current"` first on the unrestricted path, and current gateway resolver returns `No session found: current` when no literal key exists. I did not run live reproduction because this review is read-only.

## Solution Assessment

No. A global gateway rejection for `key="current"` conflicts with existing literal-key/sessionId behavior and still returns a failed WS response; the narrower fix is wrapper-side or context-aware alias resolution.
