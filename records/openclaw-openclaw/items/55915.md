## Summary

Keep open. Current main still does not implement the requested cloud relay product surface; it has direct Gateway URLs, setup codes, Tailscale/Serve remote access, and an APNs push relay, but no `cloud` config, OAuth account login, Gateway cloud registration/heartbeat, mobile gateway listing, or transparent WebSocket relay transport.

## What This Changes

Add an opt-in cloud relay flow with `openclaw login`, Gateway cloud registration/heartbeat, and relayed mobile WebSocket access while preserving device approval.

## Best Possible Solution

Keep this open for maintainer design review, ending in either a first-party cloud relay design across CLI, Gateway, config/schema, mobile apps, docs, and security tests, or a documented plugin/API seam that lets an optional relay own the backend without bypassing device approval.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main verification path is high-confidence source and docs inspection showing the requested commands, config, heartbeat registration, gateway listing, and relay transport are absent.

## Solution Assessment

Unclear. The proposed architecture is coherent, but it is too broad and security-sensitive to accept as a narrow cleanup fix without maintainer design review, especially alongside #55914.
