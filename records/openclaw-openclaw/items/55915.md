## Summary

Keep open. Current main still supports direct mobile-to-Gateway connectivity, QR/setup-code pairing, Tailscale/Serve guidance, Talk-specific gateway-relay sessions, and iOS APNs push relay, but it does not implement the requested cloud account login, cloud config block, Gateway registration/heartbeat, mobile cloud Gateway listing, or transparent relay transport for remote mobile access.

## What This Changes

The issue requests a cloud relay product surface with `openclaw login`, OAuth token storage, Gateway registration/heartbeat, mobile Gateway listing, and relayed mobile-to-Gateway transport.

## Best Possible Solution

Keep this open for maintainer design, ending in either a coherent first-party cloud relay across CLI, config/schema, Gateway, mobile apps, backend protocol, docs, and security tests, or an explicit plugin/API seam if core should not own the relay.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Current-main source and docs inspection give a high-confidence verification path showing the requested commands, config, heartbeat registration, mobile Gateway listing, and transparent relay transport are absent.

Unclear. The proposal is coherent, but the safest path is maintainer product and security design first because it adds cloud accounts, stored tokens, backend registration, and remote Gateway transport, and it should be coordinated with https://github.com/openclaw/openclaw/issues/55914.
