## Summary

Keep open. Current main still lacks the requested cloud relay product surface: mobile clients connect directly to a reachable Gateway URL, remote QR/setup code still depends on `gateway.remote.url` or Tailscale Serve/Funnel, the Gateway config/schema have no `cloud` OAuth/API/heartbeat/relay block, and the root CLI has no `login`, `logout`, `cloud`, or `invite` command for this flow. This is also too broad and security-sensitive to queue as an automated repair without maintainer product review.

## What This Changes

The issue requests first-party cloud relay support with OAuth login/logout/status, gateway registration and heartbeat, configurable cloud endpoints, and transparent WebSocket relay transport for remote mobile Gateway access.

## Best Possible Solution

Keep this issue open for maintainer design review. The end state should either be a first-party cloud relay design across Gateway, CLI, config/schema, iOS/Android UX, docs, and security tests, or a documented public plugin/API seam that lets a relay plugin own the optional backend without bypassing device approval or hardcoding a provider.

Not applicable as a bug reproduction; this is a feature request. The current-main verification path is high-confidence source and docs inspection showing the requested commands, config, heartbeat registration, and relay transport are absent.

## Solution Assessment

Unclear. The proposed architecture is coherent, but it is broad and security-sensitive; the maintainable path should be decided by maintainers first, likely coordinated with #55914 so cloud config and mobile discovery/connectivity stay aligned.
