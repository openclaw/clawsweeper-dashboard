## Summary

Keep open: current main and the latest release still resolve session-extension patching/projection through the mutable active plugin registry, and a clean open PR with closing syntax already owns the fix path. Per issue/PR policy, this issue should stay open until that PR merges.

## What This Changes

The issue reports that documented plugin session extensions registered at gateway startup are lost after later plugin registry churn, causing `sessions.pluginPatch` and `pluginExtensions` projection to miss them.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/91208 as the canonical fix; after it merges, close this issue with the merge commit and release status.

## Reproduction Assessment

Yes from source: current main's patch/projection functions read `getActivePluginRegistry().sessionExtensions` while startup/gateway lifecycle pins only HTTP-route and channel registries. I did not run a live WebSocket repro in this read-only review.

## Solution Assessment

Yes. A pinned session-extension registry matching the existing HTTP/channel lifecycle is the narrow owner-boundary fix, and the open linked PR already implements that path; closing this issue before that PR merges would be premature.
