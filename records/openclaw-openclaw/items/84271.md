## Summary

Keep open: current main and v2026.5.20 still let a post-execution tool-result middleware failure replace a delivered Slack `message.send` result with the generic middleware error, and the linked closing PR remains open with requested changes.

## What This Changes

The issue asks OpenClaw to preserve successful Slack `message.send` delivery receipts when tool-result post-processing fails after the side effect.

## Best Possible Solution

Land a narrow fix that preserves a sanitized delivered `message.send` success result, including nested Slack/core receipt identifiers, when post-processing fails while keeping generic fail-closed behavior for non-delivered results.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a successful Slack `message.send` raw result followed by thrown or invalid tool-result middleware still returns the generic middleware error on current main. I did not run a live Slack send in this read-only pass.

## Solution Assessment

Yes, the best fix is a narrow side-effect-safe fallback for delivered `message.send` results, not broad pass-through of rejected middleware output. The linked PR is the right implementation lane but still needs review changes before it can close this issue.
