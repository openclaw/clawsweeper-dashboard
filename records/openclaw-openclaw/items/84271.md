## What This Changes

The issue asks OpenClaw to preserve successful Slack `message.send` delivery identifiers when tool-result post-processing fails after Slack already accepted the message.

## Best Possible Solution

Land a corrected fix in https://github.com/openclaw/openclaw/pull/84292 or a replacement that preserves sanitized delivered `message.send` identifiers while keeping generic middleware fail-closed behavior for unknown or unsafe results.

## Reproduction Assessment

Yes, at source level: a successful Slack `message.send` result is produced before middleware runs, and current middleware returns the generic post-processing failure when a handler throws or returns invalid output. I did not run a live Slack send in this read-only review.

## Solution Assessment

Yes, the best direction is a narrow side-effect-aware fallback for delivered `message.send` results, not a broad pass-through of rejected middleware output. The open PR needs to handle the actual nested Slack/core delivery shape before it can resolve this issue.
