## Label Justifications

- `P2`: This is a normal-priority Slack delivery reporting bug with limited blast radius but real duplicate-send or retry confusion risk.
- `impact:message-loss`: The issue affects message delivery semantics by making a successful Slack send look failed to the model.

## Summary

Keep open: current main and the latest tagged release still replace thrown or invalid tool-result middleware output with the generic post-processing error after the Slack send side effect can already have returned a delivery receipt. The linked fix PR references this issue but is still open and has review feedback that it does not yet cover the actual nested Slack result shape.

## What This Changes

Preserve successful Slack `message.send` delivery receipts when tool-result middleware fails after the Slack side effect, instead of reporting a generic post-processing failure.

## Best Possible Solution

Repair and land https://github.com/openclaw/openclaw/pull/84292, or a replacement, with a minimal sanitized success fallback for actual `message.send` delivery receipts while preserving fail-closed middleware behavior for unknown or non-send results.

## Reproduction Assessment

Yes for a source-level path: current main executes the tool before middleware, Slack returns delivery identifiers, and middleware failure then substitutes the generic error result. I did not run a live Slack send in this read-only review.

## Solution Assessment

Yes, the maintainable solution is a narrow `message.send` side-effect-safe fallback that preserves only verified delivery identifiers and warnings. A broad pass-through of middleware-rejected output would weaken the middleware safety contract.
