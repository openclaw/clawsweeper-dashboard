## What This Changes

The issue asks OpenClaw to preserve a successful Slack `message.send` delivery result when tool-result middleware fails after the Slack side effect has already happened.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/84292 or a successor that recognizes the actual nested Slack/core delivery receipt shape and keeps generic fail-closed behavior for non-delivered or unsafe results.

## Reproduction Assessment

Yes, as a source-reproducible path. Current main still executes the tool before middleware, Slack returns delivery identifiers in the action payload, and thrown or invalid middleware output still replaces that result with the generic failure sentinel.

## Solution Assessment

Yes, the narrow maintainable direction is to preserve a sanitized delivered-message success result only when there is concrete delivery evidence. The linked PR is the right implementation path, but review notes show it needs to handle nested Slack/core receipt data before it can close the issue.
