Full review comments:

- **[P1] Wire external CLI sync through async readers:** `src/agents/cli-credentials.ts:837-843`
  - body: The PR adds async cached readers, but `resolveExternalCliAuthProfiles` still calls the sync Codex and Claude cached readers in `src/agents/auth-profiles/external-cli-sync.ts`. That overlay is reachable from runtime external auth/profile discovery, so Gateway auth can still hit sync keychain/file I/O after merge.
  - confidence: 0.9
- **[P2] Use async readers by default in auth epoch:** `src/agents/cli-auth-epoch.ts:184-192`
  - body: With default deps, `hasAsyncDepOverride` is false, so production takes the sync cached reader and only async test overrides use the async reader. Invert this so the normal runtime path awaits the async reader and only sync-only test overrides fall back.
  - confidence: 0.92
- **[P2] Use async file reads in async credential readers:** `src/agents/cli-credentials.ts:617`
  - body: `readClaudeCliCredentialsAsync` still calls `loadJsonFile`, and the Codex async reader has the same sync file read plus `fs.statSync` fallback and mtime fingerprint. File-backed credentials still block the event loop when keychain is unavailable or disabled.
  - confidence: 0.84
- **[P2] Finish wiring async config command resolution:** `src/agents/sessions/resolve-config-value.ts:193-209`
  - body: The PR adds async config resolvers, but `SessionModelRegistry.getApiKeyAndHeaders` still uses the synchronous `resolveConfigValueOrThrow` and header helpers for provider/model `!command` values, and the Windows async path still calls the `spawnSync` configured-shell helper. Those config-backed auth paths can still block the event loop.
  - confidence: 0.82
