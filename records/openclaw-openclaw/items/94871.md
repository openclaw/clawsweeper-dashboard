Full review comments:

- **[P1] Wire external CLI sync through async readers:** `src/agents/cli-credentials.ts:837-843`
  - body: The PR adds async cached readers, but the external CLI overlay still uses the sync Codex and Claude cached readers in `src/agents/auth-profiles/external-cli-sync.ts`. That reachable provider overlay can still hit sync keychain/file I/O, so the linked Gateway auth stall can remain after merge.
  - confidence: 0.9
- **[P2] Use async readers on the default auth-epoch path:** `src/agents/cli-auth-epoch.ts:187-192`
  - body: With default deps, `hasAsyncDepOverride` is false, so production calls the sync cached reader and only uses the async reader when tests override the async dependency. Invert this so normal runtime uses the async reader and only sync-only test overrides fall back.
  - confidence: 0.92
- **[P2] Remove sync file reads from async credential helpers:** `src/agents/cli-credentials.ts:617`
  - body: `readClaudeCliCredentialsAsync` still calls `loadJsonFile`, and the Codex async reader has the same sync file read plus `fs.statSync` fallback and cached fingerprint. File-backed credentials still block the event loop when keychain is unavailable or disabled.
  - confidence: 0.84
- **[P2] Make Windows configured-shell execution async:** `src/agents/sessions/resolve-config-value.ts:151-154`
  - body: The async config resolver still calls `executeWithConfiguredShell` on Windows, but that helper uses `spawnSync`. A `!command` API key or header on Windows can therefore still block the event loop on the new async path.
  - confidence: 0.8
