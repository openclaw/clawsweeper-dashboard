## Label Justifications

- `P2`: This is a bounded plugin tool resolver bug affecting users who rely on plugin tools under `group:plugins`.
- `merge-risk: 🚨 compatibility`: The PR changes runtime plugin id scoping in a way that can hide existing `plugins.load.paths` tools when another selected plugin is already loaded.
- `merge-risk: 🚨 security-boundary`: The resolver decides which plugin-owned tools become model-callable under allowlist policy.

## Summary

Keep open: the PR fixes the loaded/unloaded bundled sibling case, but the new loaded-or-explicit filter can drop selected config-origin `plugins.load.paths` tools whenever an unrelated active/channel registry already satisfies the filtered subset.

## What This Changes

The PR changes `resolvePluginTools` to probe loaded registries with loaded or explicitly configured plugin ids while preserving broader ids for standalone cold-load, and adds regression tests for unloaded siblings and cold-load behavior.

## Best Possible Solution

Preserve the unloaded-sibling fix while keeping config-origin/path plugins eligible for standalone cold-load before a partial loaded-registry probe is accepted.

## Reproduction Assessment

Yes, source-level. Combine an active registry containing one selected plugin with a selected config-origin `plugins.load.paths` tool plugin that is not active/channel-loaded; the PR filter can keep only the loaded id and skip the full cold-load path for the path plugin.

## Solution Assessment

No. Filtering only for the active/channel probe is the right direction, but the implementation must preserve selected config-origin/path plugin ids or otherwise force full cold-load before accepting a partial loaded-registry result.
