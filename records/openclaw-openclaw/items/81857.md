## Summary

Keep open: current main still has the all-or-nothing plugin registry lookup that can drop loaded plugin tools, and this PR contains a meaningful, proof-backed fix. It still needs maintainer handling because GitHub reports the branch as dirty/conflicting and the change affects the model-callable plugin tool boundary.

## What This Changes

The PR changes `resolvePluginTools` to probe loaded or explicitly configured plugin ids before falling back to full standalone cold-load, and adds two optional-tool resolver regression tests.

## Best Possible Solution

Rebase onto current main, preserve the request-local registry behavior from https://github.com/openclaw/openclaw/pull/93276, and land only after plugin-runtime owner confirmation of the loaded-or-explicit probe plus full cold-load semantics.

## Reproduction Assessment

Yes. Current main still probes plugin registries with the full manifest-derived runtime id set, and the inspected before/after Telegram screenshots show Active Memory changing from `status=unavailable` to `status=ok` after the patch.

## Solution Assessment

Yes, with caveats. Filtering only the active/channel probe while preserving full cold-load ids is the narrowest source-level fix I found, but it needs rebase and plugin-runtime owner confirmation before merge.
