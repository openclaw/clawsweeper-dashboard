## Summary

Keep this PR open: it addresses a real plugin-tool lookup bug, but the current patch narrows plugin IDs before the standalone cold-load path and would regress config/path plugin tool discovery.

## What This Changes

The PR filters plugin-tool runtime IDs to plugins present in active/channel registries before resolving `group:plugins` tools and adds a regression test for a loaded plugin with an unloaded sibling.

## Best Possible Solution

Keep the loaded-registry scoping idea, but use it only for active/channel registry probes while preserving full manifest-resolved `runtimePluginIds` for load options and standalone cold-load, with regression coverage for both loaded siblings and cold config/path plugins.

## Reproduction Assessment

Yes. The original Active Memory failure has before/after Telegram proof in the PR body, and the introduced cold-load regression is source-reproducible from the diff plus current `tools.optional.test.ts` cold-registry coverage.

## Solution Assessment

No. The direction is right, but this implementation scopes the ID set too early; the maintainable fix is to separate loaded-registry probe IDs from the full IDs used for standalone loading.
