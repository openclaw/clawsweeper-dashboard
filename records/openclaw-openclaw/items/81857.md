## Summary

Keep open for maintainer review. Current main still uses the full manifest-resolved plugin id set for registry lookup, while this PR appears to fix the loaded/unloaded sibling failure with focused tests and inspected real Telegram proof; the remaining concern is maintainer confirmation of the plugin-tool exposure boundary before merge.

## What This Changes

The PR changes `resolvePluginTools` to probe loaded registries with loaded or explicitly configured plugin ids while preserving full manifest-resolved ids for standalone cold-load, and adds regression tests for unloaded siblings and slot-selected cold-load.

## Best Possible Solution

Use this PR as the landing path after plugin-runtime owner confirmation that the resolver scoping contract is intended and CI remains green on the current head.

## Reproduction Assessment

Yes, source-level. Current main still applies an every-id registry check to manifest-selected plugin ids, and the inspected before/after Telegram screenshots plus added tests show the Active Memory path changing from unavailable to ok after the patch.

## Solution Assessment

Yes, conditionally. Filtering only the active/channel registry probe while retaining full manifest-resolved ids for standalone cold-load is the narrow fix shape, but the plugin-runtime owner should confirm this exposure contract before merge.
