## Label Justifications

- `P2`: This is a bounded plugin tool resolver bug affecting users who rely on plugin tools under `group:plugins`.
- `merge-risk: 🚨 compatibility`: The PR changes runtime plugin id scoping in a way that can hide existing `plugins.load.paths` tools during upgrade or normal runtime use.
- `merge-risk: 🚨 security-boundary`: The changed resolver gates which plugin-owned tools become model-callable under explicit tool policy, so incorrect scoping is security-boundary-sensitive even when the observed blocker is compatibility.

## Summary

Keep open. The intended loaded-sibling bug fix is useful and well-proven, but the latest branch can still drop selected config-origin `plugins.load.paths` tools when a partial active/channel registry satisfies the narrowed probe before the full standalone cold-load path runs.

## What This Changes

The PR changes plugin tool resolution to probe loaded registries with loaded or explicitly configured plugin ids and adds optional-tool resolver tests for unloaded bundled siblings and slot-selected cold-load behavior.

## Best Possible Solution

Preserve the unloaded-sibling fix while keeping config-origin/load-path plugins eligible for standalone cold-load before accepting a partial loaded-registry result.

## Reproduction Assessment

Yes, source-level. Combine an active registry containing one selected plugin with a selected origin=`config` `plugins.load.paths` tool plugin that is not active/channel-loaded; the PR's filtered probe can succeed for the active plugin and skip the full `runtimePluginIds` cold-load.

## Solution Assessment

No. Filtering only for the active/channel lookup is the right direction, but the implementation must preserve selected config-origin/path plugin ids or force full standalone cold-load before accepting a partial registry result.
