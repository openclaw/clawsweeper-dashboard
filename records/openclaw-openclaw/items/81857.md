## Label Justifications

- `P2`: This is a normal-priority plugin tool resolver bug with real but bounded impact on users relying on `group:plugins` and loaded plugin tools.
- `merge-risk: 🚨 compatibility`: Changing runtime plugin id scoping can alter existing plugin tool availability across loaded, explicitly configured, slot-selected, and cold-loaded plugins.
- `merge-risk: 🚨 security-boundary`: The diff changes which plugin-owned tools become model-callable under allowlist policy, even though it appears restorative and keeps deny checks.

## Summary

Keep open. Current main and the latest release still have the full manifest-id registry check that can drop loaded plugin tools, and the PR now appears clean and focused, but the callable plugin-tool scoping boundary still deserves plugin-runtime owner review before merge.

## What This Changes

The PR changes `resolvePluginTools` to probe loaded registries with loaded or explicitly configured plugin ids while preserving full manifest-resolved ids for standalone cold-load, with regression tests for unloaded siblings and cold-load behavior.

## Best Possible Solution

Merge after plugin-runtime owner review confirms the loaded-or-explicit registry probing contract and full-ID standalone cold-load fallback.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.6.9` still use the full manifest-selected id set with an every-id registry check, and the PR body supplies before/after Telegram proof for the Active Memory path.

## Solution Assessment

Yes, conditionally. Filtering only the active/channel registry probe while preserving full runtime plugin ids for standalone cold-load is the narrow fix shape, but it should land only after owner confirmation of the plugin-tool exposure boundary.
