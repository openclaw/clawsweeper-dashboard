## Summary

Keep this PR open: it addresses a real plugin tool-resolution bug with sufficient proof, but the current patch drops manifest ids before standalone cold-load and would regress path/config plugin tool discovery.

## What This Changes

The PR filters plugin tool resolution to loaded active/channel plugin ids and adds a regression test for a loaded plugin with an unloaded manifest sibling.

## Best Possible Solution

Keep the loaded-registry scoping idea, but apply it only to active/channel registry probes while preserving the full manifest-resolved plugin id set for standalone cold-load and its load options.

## Reproduction Assessment

Yes. The PR includes before/after Telegram proof for the reported Active Memory failure, and the PR-introduced cold-load regression is source-reproducible from the early return before standalone loading.

## Solution Assessment

No. The direction is right, but the patch narrows the id set too early; the safer fix is to separate active/channel probe ids from full manifest ids used for standalone loading.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep standalone cold loads on the full plugin set:** `src/plugins/tools.ts:967-968`
  - body: The new `presentPluginIds` list only contains ids already visible in active/channel registries. Use the filtered list only for active/channel probes, and keep the full manifest-resolved ids for standalone loading.
