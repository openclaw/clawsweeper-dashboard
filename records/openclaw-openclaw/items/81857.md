## Summary

Keep this PR open: the reported loaded-vs-unloaded sibling bug is plausible and the supplied proof is useful, but the patch currently regresses config-origin/path plugin tools by returning before the existing standalone cold-load path can run.

## What This Changes

The branch filters manifest-resolved plugin tool ids to ids present in active or channel registries before resolving plugin tools, with a regression test for a loaded plugin plus an unloaded sibling.

## Best Possible Solution

Keep the loaded-registry scoping idea, but apply it only to active/channel registry probes while preserving full manifest-resolved ids for standalone cold-load and its load options.

## Reproduction Assessment

Yes: the PR body includes before/after Telegram proof for the reported Active Memory failure, and current source plus existing tests make the introduced cold-load regression source-reproducible.

## Solution Assessment

No: the direction is right, but the patch narrows the id set too early; the safer fix is to separate loaded-registry probe ids from standalone cold-load ids.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Keep the full `runtimePluginIds` for standalone loading and use the filtered ids only for loaded-registry probes.
