## Summary

Keep open: current main and the latest release still have the strict full-plugin-id registry lookup that can drop loaded plugin tools when an unloaded manifest sibling is also selected, and this PR remains a useful focused fix. It is not cleanup-closeable because GitHub currently reports the branch as dirty/conflicting and the resolver boundary needs plugin-runtime owner review before merge.

## What This Changes

The PR filters active/channel plugin tool registry lookup to loaded or explicitly configured plugin ids while preserving full manifest-resolved ids for standalone cold-load, with regression tests for unloaded siblings and cold-load behavior.

## Best Possible Solution

Rebase the branch onto current main, preserve current main's runtimeRegistry fast path plus this PR's filtered active/channel probe and full-id cold-load path, then merge after plugin-runtime owner review.

## Reproduction Assessment

Yes. Current main and v2026.6.9 still apply an every-id registry check to manifest-selected plugin ids, and the inspected before/after Telegram screenshots show the affected Active Memory path changing from status=unavailable to status=ok after the patch.

## Solution Assessment

Yes, conditionally. Filtering only the active/channel registry probe while preserving full runtimePluginIds for standalone cold-load is the narrow fix shape, but the branch must be rebased without losing current main's runtimeRegistry fast path.
