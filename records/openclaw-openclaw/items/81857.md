## Summary

Keep this PR open: it addresses the earlier review concern by preserving full `runtimePluginIds` for the standalone cold-load retry, but the new loaded/explicit filter still misses slot-based explicit activation and can drop a selected plugin before cold-load gets a chance to materialize it.

## What This Changes

The branch narrows plugin tool registry lookup to loaded or explicitly configured plugin ids, keeps full runtime ids for cold-load fallback, and adds optional-tool resolver coverage.

## Best Possible Solution

Keep the loaded-registry narrowing, but derive the preserved explicit ids from the normalized activation model, including slot selections, normalized allow aliases, and enabled entries, while retaining full `runtimePluginIds` for standalone cold-load.

## Reproduction Assessment

Yes. The original bug is source-reproducible from current main's all-id registry check plus the PR's before/after screenshots; the new PR regression is also source-reproducible from the filter omitting slot-based explicit activation.

## Solution Assessment

No. The proposed shape is close, but the best fix should reuse or mirror the normalized activation contract instead of hand-reading only raw `plugins.allow` and `plugins.entries` keys.
