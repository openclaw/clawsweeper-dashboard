## Summary

Keep open: this is a plausible core doctor fix, but the proposed implementation can leave non-bundled configured-channel plugins without the explicit activation signal current startup planning requires, and the PR still lacks real doctor-plus-gateway proof.

## What This Changes

The PR changes doctor plugin auto-enable to skip writing `plugins.entries` for global/workspace manifest-channel plugins with configured channels, updates the test registry helper to accept plugin origin, and adds focused channel auto-enable tests.

## Best Possible Solution

Fix doctor’s duplicate materialization for external manifest-channel plugins while keeping doctor output, allowlist/activation trust, and startup planning on one explicit contract, with coverage for both config output and gateway startup ownership.

## Reproduction Assessment

Yes, by source inspection: the new `alreadyEnabled` path can skip both entry creation and allowlisting, while current startup code and tests reject non-bundled configured-channel owners without explicit selection. I did not run a writable doctor/gateway setup because this review is read-only.

## Solution Assessment

No. Doctor is the right layer, but this branch treats configured global/workspace channels as already safe without preserving the explicit activation or allowlist trust that current startup planning requires.
