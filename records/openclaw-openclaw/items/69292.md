## Summary

Keep open: this is a core doctor/plugin activation fix, but the PR still has a blocking correctness issue. The new skip avoids writing `plugins.entries`, yet current startup and channel-presence code only treats non-bundled channel owners as explicitly trusted through `plugins.allow` or `plugins.entries`, so a configured global/workspace channel plugin can be left without a startable owner.

## What This Changes

The PR changes doctor plugin auto-enable to treat global/workspace manifest-channel plugins as already enabled when their channel is configured, updates the test helper origin support, and adds channel skip tests.

## Best Possible Solution

Keep the root-cause fix in core doctor/plugin activation, but preserve an explicit trust/activation signal for non-bundled channel owners while preventing duplicate config-selected registry materialization.

## Reproduction Assessment

Yes, by source inspection: apply the PR's auto-enable branch to a configured global/workspace manifest-channel plugin, then evaluate startup/channel presence with current main policy. The skipped entry leaves no `plugins.entries` or `plugins.allow` trust signal, and those paths require explicit enablement for non-bundled channel owners.

## Solution Assessment

No. The PR fixes the right layer, but configured channel presence is not currently equivalent to explicit non-bundled plugin trust; the safer solution must separate duplicate-entry avoidance from trust preservation.
