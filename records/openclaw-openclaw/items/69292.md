## Summary

Keep open. The duplicate-plugin warning target is a real core doctor/plugin activation problem and the related PRs are closed unmerged, but this PR should not merge as-is because it skips the config materialization that currently creates explicit trust for non-bundled configured channel owners.

## What This Changes

The PR changes doctor/plugin auto-enable logic to skip writing `plugins.entries` for global or workspace manifest-channel plugins, extends the registry test helper with origins, and adds global, workspace, and bundled channel auto-enable tests.

## Best Possible Solution

Keep the fix in core doctor/plugin activation logic, but revise it so duplicate plugin registration is avoided without losing explicit trust. The landed version should cover global, workspace, bundled, restrictive-allowlist, and startup-planning behavior together.

## Reproduction Assessment

Yes. A focused reproduction can compare `applyPluginAutoEnable` for `channels.acme-chat` plus a global/workspace manifest record: current main writes `plugins.entries`/allowlist state, while the PR skips it, after which configured-channel startup planning still requires explicit trust.

## Solution Assessment

No. The PR is aimed at the right layer, but using discovery plus configured channel presence as an `alreadyEnabled` proxy is not the safest solution because it bypasses the trust materialization required by current startup policy.
