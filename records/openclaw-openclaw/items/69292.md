## Summary

Keep open; the PR targets the right core doctor bug, but the current patch is not merge-ready because it skips the explicit entry/allowlist trust signal that current startup and channel-presence policy require for non-bundled channel owners.

## What This Changes

The branch changes doctor plugin auto-enable logic to treat global/workspace manifest-channel plugins as already enabled, extends the registry test helper with origins, and adds channel auto-load skip tests.

## Best Possible Solution

Keep the fix in core doctor/plugin activation, but change it so duplicate registry materialization is avoided while an explicit trust/activation signal is still preserved for global and workspace channel owners.

## Reproduction Assessment

Yes, by source inspection: applying the PR path to a configured global/workspace manifest-channel plugin skips entry and allowlist materialization, then current startup/presence policy does not treat that owner as explicitly trusted.

## Solution Assessment

No. The PR fixes the right layer, but configured channel presence is not currently equivalent to explicit plugin trust, so the safer solution must separate duplicate-entry avoidance from trust preservation.
