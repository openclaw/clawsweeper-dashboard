## Summary

Keep open. Current main still writes `plugins.entries` for non-built-in configured channel plugins, and the related prior PRs are closed unmerged, so this work is not obsolete. The branch should not merge as-is because it treats global/workspace discovery plus channel config as enough to skip both entry materialization and allowlist preservation, while current startup/channel planning requires explicit trust for non-bundled channel owners.

## What This Changes

The PR changes doctor plugin auto-enable logic to skip `plugins.entries` for global or workspace manifest-channel plugins, extends the test registry helper with plugin origins, and adds global, workspace, and bundled channel tests.

## Best Possible Solution

Keep the fix in core doctor/plugin activation logic, but separate trust preservation from duplicate registry materialization. The landed path should avoid duplicate `plugins.entries` registration while still making global/workspace channel owners explicitly eligible at startup, with coverage for no allowlist, restrictive allowlist, global, workspace, bundled, and duplicate-diagnostic cases.

## Reproduction Assessment

Yes. A focused reproduction can call `applyPluginAutoEnable` with `channels.acme-chat` and a global/workspace manifest record for `acme-chat`, then verify the resulting config against `resolveConfiguredChannelPluginIds` or startup planning; current tests already show untrusted non-bundled channel owners are filtered.

## Solution Assessment

No. The PR is aimed at the right core layer, but configured channel presence is not a safe proxy for explicit plugin trust under current startup policy; the safer fix must preserve trust while avoiding duplicate registration.
