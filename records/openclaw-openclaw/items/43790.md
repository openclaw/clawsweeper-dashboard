## Summary

Keep open: the routing goal is valid and current main still lacks the explicit message-tool agent route, but the PR still makes a fallback-derived agent id authoritative in one lazy-config path and needs a changelog credit fix before merge.

## What This Changes

The PR threads an `agentId` option through OpenClaw tool construction into the message tool, adjusts effective tool-policy fallback resolution, adds focused routing coverage, and adds a changelog bullet.

## Best Possible Solution

Land a rebased fix that passes only a trusted active route agent id into the message tool, leaves session/default fallback resolution config-aware at message execution, covers the no-config lazy-load fallback path, and credits the changelog entry.

## Reproduction Assessment

Yes. Build `createOpenClawCodingTools` without `config` for an opaque session key while persisted runtime config has a non-`main` default agent, then execute a message send and inspect the `runMessageAction` input; the PR's existing tests do not cover that lazy-config path.

## Solution Assessment

No. Preferring an explicitly known active agent id is the right direction, but this patch also forwards a policy/session fallback as if it were explicit; the safer boundary is to pass only caller-supplied trusted route ids and let `runMessageAction` keep config-aware fallback ownership.
