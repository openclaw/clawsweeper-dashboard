## Summary

Keep open: current main already covers the central explicit message-tool agent-routing idea, but this external PR still has an actionable routing-edge bug, lacks real behavior proof, and is conflicting with newer main-line routing work.

## What This Changes

The PR threads an explicit/effective agent id into message-tool routing, changes non-agent session-key tool-policy fallback, adds focused tests, and adds a changelog entry.

## Best Possible Solution

Refresh against main, preserve only authoritative explicit or config-aware agent ids in message routing, keep the configured-default regression coverage, and add redacted real-run or terminal proof of send attribution.

## Reproduction Assessment

Yes for focused source-level paths: construct message-tool routing with conflicting explicit and session-derived agent ids, and construct createOpenClawCodingTools without config using an opaque session key plus a non-main persisted default. I did not run tests in this read-only review.

## Solution Assessment

No as-is: preferring an authoritative explicit agent id is the right direction, but passing a derived fallback main as explicit can still bypass the configured default. The safer fix is to preserve lazy/config-aware session fallback when the caller did not supply a true explicit agent id.
