## Summary

This PR is useful but not ready: the current head still has a routing regression for opaque session keys when config is unavailable, and the external PR only reports focused Vitest coverage. It needs a branch update plus real behavior proof before merge; redacted terminal output, logs, screenshots, recordings, or linked artifacts are acceptable, and updating the PR body should trigger a fresh ClawSweeper review.

## What This Changes

The PR threads an explicit agent id through Pi/OpenClaw tool construction into the message tool, prefers it over session-derived routing, and adds focused routing/default-agent coverage plus a changelog entry.

## Best Possible Solution

Rebase onto current main, thread only a truly explicit requester agent id into message routing, preserve config-aware fallback resolution after config load, and cover both explicit-agent and config-unavailable opaque-session cases before adding redacted runtime proof.

## Reproduction Assessment

Yes, from source inspection. A focused regression can construct tools with an explicit active agent and a conflicting session key, then assert the message action receives the explicit agent; the remaining PR bug is likewise source-reproducible with an opaque session key and no config at construction time.

## Solution Assessment

No, not yet. The direction is right, but the current patch turns a session-derived fallback into an authoritative explicit id when config is unavailable; the safer fix is to pass only caller-explicit agent ids or defer fallback resolution until the message tool has loaded config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve config-aware fallback for opaque sessions:** `src/agents/pi-tools.ts:494`
  - body: Passing the policy-derived `agentId` into `createOpenClawTools` makes the message tool treat it as explicit. When `createOpenClawCodingTools` is called without `options.config` and with an opaque/non-agent session key, policy resolution still produces `main`; the message tool then bypasses its lazy config-aware fallback, so installs whose configured default agent is not `main` can send or bind accounts as the wrong agent. Only pass a caller-supplied explicit agent id through, or resolve the fallback after config is loaded.
  - confidence: 0.88
