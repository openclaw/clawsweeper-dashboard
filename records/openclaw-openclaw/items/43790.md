## Summary

Keep open. Current main still does not expose a separate explicit message-tool routing agent, so the intended bug fix is not already implemented; however, the PR should not merge as-is because the diff can promote a fallback-derived agent id into an authoritative outbound route and the changelog entry is missing required attribution.

## What This Changes

The PR threads an `agentId` option through OpenClaw tool construction into the message tool, changes effective tool-policy session fallback resolution, adds focused regression tests, and adds a changelog note.

## Best Possible Solution

Land a rebased routing fix that passes a distinct trusted active/message-routing agent id only from runtime call sites that truly know it, keeps fallback resolution config-aware at message execution time, adds coverage for config-less opaque-session default-agent behavior, and credits the changelog entry.

## Reproduction Assessment

Yes. A high-confidence static/unit reproduction is to build coding tools without config for an opaque session key while runtime config has a non-`main` default agent, then execute a message send and verify the PR can make the no-config fallback id authoritative before the message route is resolved from loaded config.

## Solution Assessment

No. The direction is valid, but this implementation conflates trusted explicit routing with policy/session fallback inference; the narrower maintainable fix is a separate active-agent route input plus fallback coverage at the message action boundary.
