## Summary

Keep open. The underlying explicit-agent routing bug still appears valid on current main, but this PR is not ready to merge because its routing change can make a fallback-derived agent id authoritative for outbound sends, and the user-facing changelog entry is missing required credit.

## What This Changes

The PR threads an `agentId` option through OpenClaw tool construction into the message tool, updates tool-policy fallback resolution, adds focused routing tests, and adds a changelog bullet.

## Best Possible Solution

Land a rebased routing fix that passes a trusted active message-routing agent id only from runtime call sites that truly know it, keeps session/default fallback resolution config-aware at message execution, adds coverage for opaque-session/no-config fallback behavior plus explicit override behavior, and credits the changelog entry.

## Reproduction Assessment

Yes. A high-confidence unit/static reproduction is to build tools without config for an opaque session key while persisted config has a non-`main` default agent, then execute a message send and verify a precomputed fallback id can bypass the runtime config-aware route. The PR covers the explicit-agent happy path but not this fallback path.

## Solution Assessment

No. The direction is valid, but the patch conflates trusted explicit routing with policy/session fallback inference; the safer solution is a separate trusted active-agent route input and fallback coverage at the message action boundary.
