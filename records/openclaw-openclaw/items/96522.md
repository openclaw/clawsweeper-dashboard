## Label Justifications

- `P2`: This is a bounded cron/session-state persistence bug that can make delivered cron output invisible to later destination-session turns.
- `merge-risk: 🚨 compatibility`: Existing cron jobs with `session:` values may route differently after upgrade when the suffix matches a persisted session ID.

## Summary

Keep open: the patch plausibly fixes the linked cron/session-history bug, but the external PR lacks after-fix real behavior proof and still needs a maintainer-visible scope call for related notification paths before merge.

## What This Changes

The PR adds cron `session:` target resolution against persisted session IDs and uses the resolved key for gateway isolated cron runs and command announcement delivery.

## Best Possible Solution

Land a resolver-centered fix after redacted real behavior proof shows the reported cron output appears in the destination session history, with an explicit maintainer decision on whether notification paths should share the same session-id lookup.

## Reproduction Assessment

Yes. Current main treats `session:<sessionId>` as a literal session key, and the linked issue gives the cron command flow where that suffix only exists as a stored `sessionId`, so the destination session history is missed by source inspection.

## Solution Assessment

Likely yes for the primary runtime path: gateway is the caller that chooses the session key for isolated cron execution, and direct cron delivery mirrors custom targets to that caller-selected key. The remaining scope question is whether failure notification siblings should also move to the store-backed resolver before merge.
