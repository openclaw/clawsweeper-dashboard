## Summary

Keep open: current main still has the gateway-rpc timeout coercion bug, but this branch is not merge-ready because it broadens shared integer parsing while leaving `tasks audit --limit` as a silent no-limit fallback, and the external PR still lacks real CLI behavior proof.

## What This Changes

The PR validates gateway-rpc CLI timeout input, tightens shared positive-integer parsing, validates cron timeout-seconds, and adds focused tests plus changelog text.

## Best Possible Solution

Land a rebased CLI-boundary timeout validation that preserves the 30s omitted-option default, validates or scopes every shared-parser caller, keeps the newer gateway-rpc runtime fields, and includes redacted real CLI output.

## Reproduction Assessment

Yes, by source inspection. Current main advertises a 30000 ms gateway-rpc timeout default, but malformed explicit values still flow through `Number(...)` and then the gateway fallback path treats non-finite timeouts as unset.

## Solution Assessment

No as-is. CLI-boundary validation is the right fix direction, but the shared parser change must either be scoped to timeout callers or paired with explicit validation for remaining callers such as `tasks audit --limit`, and the PR still needs real CLI proof.
