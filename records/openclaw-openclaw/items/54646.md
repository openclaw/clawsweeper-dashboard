## Summary

Keep this PR open: the reported gateway-rpc timeout bug is still present on current main and the PR is directionally useful, but the latest patch still introduces a shared-parser regression and lacks required real behavior proof.

## What This Changes

The PR validates gateway-rpc `--timeout`, tightens shared positive-integer parsing, validates cron `--timeout-seconds`, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land this branch or a canonical successor after validating every affected shared-parser caller, preserving the gateway-rpc 30s omitted-option default, and adding redacted real CLI proof.

## Reproduction Assessment

Yes by source inspection. Current main advertises a 30000 ms gateway-rpc timeout default, but the runtime still coerces malformed explicit input with `Number(...)`, and the gateway fallback converts non-finite values to 10000 ms.

## Solution Assessment

No as-is. CLI-boundary validation is the right fix direction, but the shared parser change must be paired with explicit validation for remaining callers such as `tasks audit --limit`, and the contributor still needs real CLI behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate every caller of the tightened integer parser:** `src/cli/program/helpers.ts:17-18`
  - body: This PR changes `parsePositiveIntOrUndefined` globally, not only for timeout parsing. Add explicit invalid-input handling for that caller or keep strict parsing scoped to timeout paths.
