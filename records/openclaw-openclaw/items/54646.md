## Summary

Keep open: current main still has the gateway-rpc timeout bug and this PR is pointed at the right surface, but the branch is not merge-ready because it changes a shared parser without handling an affected caller and still has only test/mock proof.

## What This Changes

This PR adds gateway-rpc timeout validation, tightens shared positive-integer parsing, validates cron add timeout seconds, and adds focused tests plus changelog text.

## Best Possible Solution

Land a narrow rebase that validates gateway-rpc timeout at the CLI boundary, preserves the 30s omitted-option default, either scopes strict parsing or validates every affected shared-helper caller, and includes redacted real CLI output.

## Reproduction Assessment

Yes, by source inspection. Current main advertises a 30000 ms gateway-rpc timeout default, but malformed explicit values still flow through `Number(...)` and then the gateway fallback path treats non-finite timeouts as unset.

## Solution Assessment

No as-is. CLI-boundary validation is the right fix direction, but the shared parser change must be paired with explicit handling for remaining callers such as `tasks audit --limit`, and the PR still needs real CLI proof.
