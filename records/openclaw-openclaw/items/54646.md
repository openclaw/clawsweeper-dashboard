## Summary

Keep open: current main still has the gateway-rpc timeout coercion bug, and this PR remains a useful fix path, but the latest branch is not merge-ready because it still changes the shared integer parser without validating every affected caller and lacks required real CLI behavior proof.

## What This Changes

The PR validates gateway-rpc `--timeout`, tightens shared positive-integer parsing, validates cron `--timeout-seconds`, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rebase and land a narrow successor that validates gateway-rpc timeout at the CLI boundary, preserves the 30s omitted-option default, explicitly handles every affected shared-parser caller, and includes redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection rather than a live CLI run. Current main advertises a `30000` ms gateway-rpc timeout default, but the runtime still coerces explicit malformed input with `Number(...)`, and the gateway fallback converts non-finite values to `10000` ms.

## Solution Assessment

No as-is. CLI-boundary validation is the right fix direction, but the shared parser change must be paired with explicit validation for remaining callers such as `tasks audit --limit`, and the PR still needs a rebase plus real CLI behavior proof.
