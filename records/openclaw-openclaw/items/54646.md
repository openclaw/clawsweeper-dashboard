## Summary

Keep open: current main still has the reported timeout mismatch, but this PR is not merge-ready because the shared parser change still leaves an unvalidated `tasks audit --limit` path and the PR body only provides unit-test proof.

## What This Changes

The PR validates gateway-rpc `--timeout`, tightens shared positive-integer parsing, validates cron `--timeout-seconds`, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land this branch or a canonical successor after scoping strict parsing or validating every affected shared-helper caller, then add redacted real CLI proof for valid, omitted, and malformed gateway-rpc timeout input.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI advertises a `30000` timeout default, but malformed explicit gateway-rpc timeout input can reach the gateway as `NaN` and fall back to `10000`.

## Solution Assessment

No as-is. CLI-boundary validation is the right direction, but the shared parser tightening should either be scoped to timeout paths or paired with explicit invalid-input handling for remaining callers such as `tasks audit --limit`.
