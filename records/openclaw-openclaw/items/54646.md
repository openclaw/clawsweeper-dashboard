## Summary

Keep this PR open, but do not merge it as-is: current main still has the reported gateway-rpc timeout bug, and the branch fixes that path, but the shared parser tightening leaves another explicit CLI integer option able to silently drop malformed input.

## What This Changes

The PR validates gateway-rpc `--timeout`, tightens the shared positive-integer parser, validates cron add `--timeout-seconds`, and adds focused CLI tests plus a changelog entry.

## Best Possible Solution

Land this branch or a canonical successor after validating all explicit callers affected by the stricter integer helper, preserving the gateway-rpc default and malformed-timeout rejection behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows malformed gateway-rpc `--timeout` values flow through `Number(...)` as `NaN` and then hit the gateway call's `10000` ms fallback despite the CLI option defaulting to `30000` ms.

## Solution Assessment

No as-is. CLI-boundary validation is the right direction, but the shared helper change also needs explicit validation for remaining call sites such as `tasks audit --limit`, or the gateway-rpc fix should use a narrower timeout parser.
