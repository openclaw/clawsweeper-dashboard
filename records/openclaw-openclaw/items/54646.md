## Summary

Keep open: the underlying CLI timeout bug is real and the PR is in the right area, but the current branch still has a P2 regression from tightening a shared parser without validating every caller, lacks external real behavior proof, and is currently merge-conflicting.

## What This Changes

The PR validates gateway-rpc `--timeout`, tightens shared positive-integer parsing, validates cron add `--timeout-seconds`, adds focused CLI tests, and adds a changelog entry.

## Best Possible Solution

Land this branch or a canonical successor after adding explicit invalid-input handling for every affected shared-helper consumer, resolving the merge conflict, and posting redacted real CLI proof for valid, omitted, and malformed gateway-rpc timeout behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: a gateway-rpc CLI command with `--timeout nope` reaches `Number(opts.timeout ?? 10_000)` as `NaN`, then the gateway call falls back to `10_000` instead of failing at the CLI boundary.

## Solution Assessment

No as-is. Validating at the CLI boundary is the right solution direction, but tightening the shared integer helper needs matching validation for remaining explicit callers such as `tasks audit --limit`, or the strict parser should be scoped to timeout paths.
