## Summary

Keep open. Current main still treats a removed versioned OpenClaw package path in `plugins.load.paths` as a config error, and the existing doctor repair only removes aliases under the current package root, so the merged related work at https://github.com/openclaw/openclaw/pull/55054 remains narrower than this report.

## What This Changes

Repair upgrade-stale plugin config paths so `openclaw gateway start` and/or `openclaw doctor --fix` recover from absolute bundled plugin paths pointing at removed pnpm/npm OpenClaw package versions.

## Best Possible Solution

Extend `doctor --fix` with a conservative bundled-plugin repair for missing old-version OpenClaw package paths, removing redundant bundled load-path entries only when the current bundled plugin exists and preserving strict errors for custom or ambiguous paths.

## Reproduction Assessment

Yes, by source inspection. A missing old-version path in `plugins.load.paths` still flows through `discoverFromPath()` as `plugin path not found` and is surfaced as a config validation error; I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Yes, the narrow doctor repair direction is the best fit. Startup-time config mutation or weakening all missing-path validation would be riskier than teaching the existing bundled load-path repair to recognize old OpenClaw package roots.
