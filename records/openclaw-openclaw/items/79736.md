## Summary

Close as implemented: current main and v2026.5.7 already expose `api.on()` typed-hook registration to non-bundled plugins, with prompt-injection hooks default-enabled and raw conversation hooks separately gated by the existing `allowConversationAccess` policy.

## What This Changes

Allow third-party plugins loaded outside the bundled tree to register typed lifecycle and prompt hooks with `api.on()`.

## Best Possible Solution

Keep the shipped `api.on()` typed-hook implementation; new reports should focus on a minimal plugin fixture and runtime diagnostics if a specific third-party plugin still does not activate or dispatch.

## Reproduction Assessment

No. Current main and v2026.5.7 source/tests show non-bundled prompt-injection hooks registering into `registry.typedHooks` by default, so the reported missing `api.on()` capability does not reproduce from the repository evidence.

## Solution Assessment

Yes. The current implementation is the narrow maintainable solution: `api.on()` registers typed hooks for plugin entries, prompt mutation remains enabled unless explicitly disabled, and raw conversation access stays behind the existing operator-controlled config gate.
