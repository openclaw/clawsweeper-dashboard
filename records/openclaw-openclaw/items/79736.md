## Summary

Current main already supports this: non-bundled plugins get `api.on(...)` during full registration, prompt-injection typed hooks are registered into `registry.typedHooks`, and explicit hook-capable/policy plugins are included in the Gateway startup scope. The startup-scope bug matching this report was fixed by the merged hook-plugin startup PR and shipped before v2026.5.7.

## What This Changes

Allow third-party plugins to register typed prompt and lifecycle hooks such as `before_prompt_build` through `api.on(...)`.

## Best Possible Solution

Keep the shipped typed-hook implementation; any remaining local failure should be filed as a narrow bug with the plugin manifest, `plugins.entries.<id>` config, runtime inspect output, and Gateway turn logs.

## Reproduction Assessment

No: current-main source and regression tests show non-bundled `api.on("before_prompt_build", ...)` registers into the live typed-hook registry, and the matching startup-scope bug is already fixed. The provided report lacks the manifest/runtime inspect output needed to reproduce a remaining local v2026.5.7 failure.

## Solution Assessment

Yes: the existing typed-hook API plus startup planning from hook capability or explicit hook policy is the narrow maintainable solution. A separate third-party-only hook API is not needed.
