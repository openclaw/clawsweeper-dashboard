## Summary

The issue is still necessary: OpenClaw previously shipped a targeted fix for one-shot memory-manager exit hangs, but the detailed 2026.6.1 large-corpus report shows the same observable failure can persist through the existing cleanup path, indicating a regression or an additional retained worker owner.

## What This Changes

Fix `openclaw agent --local --json` so completed large-memory runs exit promptly and do not leave descendant workers alive.

## Best Possible Solution

Extend the existing one-shot teardown architecture to every runtime instantiated by local memory recall, then add real process proof that final JSON is followed by prompt natural exit and zero surviving descendants without disrupting process-stable gateway reuse.

## Reproduction Assessment

Yes, from source and report evidence: the issue provides an exact released image, generated corpus procedure, command, scale control, timing threshold, and survivor observations. This read-only review did not execute the expensive 10,000-entry scenario against current main, so the status is source-reproducible rather than directly reproduced.

## Solution Assessment

Yes in architecture: repair the existing resource-owner teardown and prove descendant cleanup. Unconditional forced exit or output-based termination inside OpenClaw would conceal the leak rather than solve it.
