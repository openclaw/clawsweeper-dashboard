## Summary

Keep open. Current main still lacks `cron.defaults.delivery` or an equivalent primary cron announce delivery default, and the remaining work is a new config/product surface rather than an already-implemented cleanup.

## What This Changes

The issue requests a `cron.defaults.delivery` config option that supplies a global default cron delivery mode/channel/target when a cron job does not set its own delivery.

## Best Possible Solution

Add an approved cron-owned `defaults.delivery` contract across config types, strict/generated schema, docs/help, Gateway/service normalization, CLI, Control UI, and regression tests with precedence: explicit job delivery, then cron default, then existing fallback.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence verification that current main has no accepted `cron.defaults.delivery` config surface and still falls back to implicit announce/`last` behavior.

## Solution Assessment

Unclear pending maintainer design review. The requested precedence is plausible, but the narrowest maintainable solution may need to align with broader #57638 and secops review for scheduled delivery defaults.
