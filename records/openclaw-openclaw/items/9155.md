## Summary

Keep open: current main still lacks `cron.defaults.delivery` or an equivalent primary cron delivery default, and the issue discussion already keeps this as the focused tracker while broader cron-defaults work remains unsettled.

## What This Changes

Request to add `cron.defaults.delivery` so isolated cron jobs can inherit a default announce channel and target instead of repeating per-job delivery config.

## Best Possible Solution

Add a documented, schema-backed cron delivery default, or the chosen broader `cron.defaults` design, that applies after per-job delivery and before the existing implicit-last fallback, with CLI/UI/API support and regression coverage.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection verifies `cron.defaults.delivery` is absent from current main rather than exposing a failing runtime path.

## Solution Assessment

Unclear. `cron.defaults.delivery` is a focused and plausible solution, but maintainers should settle its public config shape and precedence alongside broader #57638 and existing failure-destination semantics.
