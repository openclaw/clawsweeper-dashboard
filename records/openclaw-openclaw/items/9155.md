## Summary

Keep open: current main still lacks `cron.defaults.delivery` or an equivalent primary cron delivery default, and the prior discussion already treats this as the focused tracker rather than a duplicate of the broader cron-defaults proposal.

## What This Changes

The issue requests a new `cron.defaults.delivery` config option so cron jobs can inherit a default delivery mode, channel, and target before falling back to `last`.

## Best Possible Solution

Add a documented, schema-backed primary cron delivery default or fold the same precedence into an approved unified `cron.defaults` design, with explicit per-job delivery winning before any config default and existing fallback behavior last.

## Reproduction Assessment

Not applicable as a bug reproduction. This is a request for a new public config option, and source inspection verifies the requested path is absent from current main.

## Solution Assessment

Unclear until maintainers choose the public config contract. `cron.defaults.delivery` is focused and plausible, but it should be reconciled with the broader cron defaults proposal before implementation.
