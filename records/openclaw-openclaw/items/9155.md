## Label Justifications

- `P2`: This is a bounded cron configuration enhancement with delivery and privacy impact, but it is not an urgent runtime outage.
- `impact:security`: A default cron recipient controls where scheduled output may be sent, so it affects sensitive-data routing.
- `impact:message-loss`: The request is motivated by cron output being misrouted or not delivered when jobs fall back to unstable `last` routing.

## Summary

Keep open: current main and v2026.6.5 still lack `cron.defaults.delivery` or an equivalent primary cron delivery default, and the broader open cron-defaults issue does not safely replace this focused product/security-sensitive tracker.

## What This Changes

The issue requests a `cron.defaults.delivery` config option so cron jobs can inherit a default delivery channel and target instead of repeating per-job delivery settings.

## Best Possible Solution

Approve one cron-defaults contract that preserves explicit per-job delivery first, keeps failure notifications separate, and covers config schema/help/docs, gateway/service behavior, CLI and Control UI support, and regression tests together.

## Reproduction Assessment

Yes, at source level: current `CronConfig`, the strict zod schema, job creation, and delivery planning do not expose or consume `cron.defaults.delivery`. I did not run a live runtime repro because this is a read-only feature/config review.

## Solution Assessment

Unclear until maintainers choose the public cron-defaults contract. The requested path is focused and plausible, but it should be reconciled with https://github.com/openclaw/openclaw/issues/57638 and reviewed for recipient-routing safety.
