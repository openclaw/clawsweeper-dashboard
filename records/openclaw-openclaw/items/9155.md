## Summary

Keep open. Current main and the latest release still lack `cron.defaults.delivery` or an equivalent primary cron delivery default; the remaining work is still owned by this focused tracker because https://github.com/openclaw/openclaw/issues/41164 was closed back here and https://github.com/openclaw/openclaw/issues/57638 is closed stale.

## What This Changes

The issue requests a `cron.defaults.delivery` configuration default so cron jobs can inherit a stable delivery channel and target instead of repeating per-job delivery settings or falling back to `last`.

## Best Possible Solution

Define one maintainer-approved primary cron delivery-default contract, then update config types/schema/help/docs plus cron creation/CLI/UI behavior and tests so explicit per-job delivery wins before the config default and only then falls back to `last`.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `cron.defaults.delivery` config path, and omitted isolated cron delivery deterministically defaults to announce delivery with a `last` channel fallback.

## Solution Assessment

Unclear pending maintainer direction. `cron.defaults.delivery` is a plausible narrow solution, but maintainers still need to choose creation-time versus run-time semantics and the exact precedence/migration behavior before implementation.
