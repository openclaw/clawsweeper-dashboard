## Summary

Keep open. Current main and the latest tagged release still lack a primary cron delivery default; the existing global cron delivery-like setting is failure-notification-only, while job creation and delivery planning still fall back to announce/last when no explicit target is supplied.

## What This Changes

The issue requests a `cron.defaults.delivery` config option so cron jobs can inherit a default delivery channel and target instead of repeating per-job delivery config.

## Best Possible Solution

Define a maintainer-approved cron primary delivery default that wires the config type, strict schema, generated/help docs, service creation, CLI/UI behavior, and precedence tests together.

## Reproduction Assessment

Yes, source inspection reproduces the current gap: current main has no `cron.defaults.delivery` config field, and omitted detached cron delivery still resolves to announce/last rather than a configured default target. I did not run a live CLI repro because this read-only review forbids commands that may write artifacts.

## Solution Assessment

No current implementation solves this yet. The best path is a maintainer-approved config/default design, not a strict bug-fix automation lane, because the change adds a new recipient-default config surface.
