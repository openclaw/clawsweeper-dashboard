## Summary

Keep open. Current main still lacks `cron.defaults.delivery` or an equivalent primary cron announce delivery default; only failure-notification defaults exist, and the related discussion keeps this as the focused tracker while broader #57638 remains open.

## What This Changes

Request adding a `cron.defaults.delivery` config default so cron jobs can inherit a shared delivery mode/channel/target instead of repeating per-job delivery fields.

## Best Possible Solution

Add an approved first-class cron-owned default delivery surface across config types, strict schema, generated schema/help, Gateway cron normalization, CLI, Control UI, docs, and tests, with precedence of explicit job delivery, then cron default delivery, then existing fallback behavior.

## Reproduction Assessment

Not applicable as a feature request. Source verification is high confidence: current main's strict schema lacks the requested key and cron job creation only uses explicit delivery or the built-in announce fallback.

## Solution Assessment

Unclear pending maintainer design review. The requested precedence is plausible, but it overlaps #57638 and scheduled recipient defaults are privacy-sensitive, so the best fix should be an approved cron-owned default rather than repurposing `cron.failureDestination`.
