## Summary

Current main still has no primary cron delivery default: omitted isolated-job delivery becomes announce-only and ultimately resolves through the last-route fallback. This issue remains the focused canonical tracker, while the broader cron-defaults proposal is open and does not replace the maintainer decision needed here.

## What This Changes

Add a global cron delivery default whose channel and target apply when a job omits delivery settings, while preserving explicit per-job precedence and the existing final fallback.

## Best Possible Solution

Define a narrow primary cron delivery-default contract with explicit per-job precedence, a documented creation-versus-execution persistence rule, strict schema/help/docs, and tests covering fresh jobs, existing jobs, channel/account/target resolution, and the final last-route fallback.

## Reproduction Assessment

Yes at source level. Current main deterministically stores announce-only delivery for an omitted isolated-job delivery and then falls back to the last route; no live multi-channel delivery run was performed.

## Solution Assessment

Unclear. A configured default is a plausible and narrow remedy, but the proposed shape is not ready for autonomous implementation until maintainers decide whether defaults are copied into persisted jobs at creation or resolved dynamically on every run.

## Risks / Open Questions

- Applying the default dynamically at execution would let later config changes silently retarget already-persisted jobs, while applying it only at creation would not change existing jobs; maintainers must choose and document that persistence contract.
- A global recipient default is security-sensitive because incorrect scoping, precedence, or account handling could redirect many scheduled announcements to the wrong destination.
- The broader cron-defaults proposal also includes model and context policy; merging these decisions without a narrow design could create an oversized configuration surface and competing semantics.
