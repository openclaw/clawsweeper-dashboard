## Summary

Keep open: current main and the latest release still lack `cron.defaults.delivery` or an equivalent primary cron announce delivery default, and this focused request is not fully replaced by the broader open cron-defaults proposal.

## What This Changes

Request to add `cron.defaults.delivery` so cron jobs can inherit a default delivery mode, channel, and target when they do not set delivery explicitly.

## Best Possible Solution

Approve one cron-defaults contract that preserves explicit per-job delivery first, keeps failure notifications separate, and covers config schema, docs, CLI/UI, and regression tests together.

## Reproduction Assessment

Yes, at source level: current `CronConfig`, the strict zod schema, job creation, and delivery planning do not expose or consume `cron.defaults.delivery`. I did not run a runtime repro because this is a read-only feature/config review.

## Solution Assessment

Unclear until maintainers choose the public cron-defaults contract. The requested path is focused and plausible, but it should be reconciled with https://github.com/openclaw/openclaw/issues/57638 and reviewed for recipient-routing safety.
