## Label Justifications

- `P2`: This is a normal-priority feature request with real privacy and routing impact but no current core outage by itself.
- `impact:security`: The requested default is meant to prevent scheduled cron output from falling back to unintended chat targets and leaking sensitive content.
- `impact:message-loss`: Without a stable default target, cron output can route to the wrong destination or fail when the implicit last route is stale or unsuitable.

## Summary

Keep open. Current main and the latest release still lack `cron.defaults.delivery` or an equivalent primary cron delivery target default, while related issues confirm this remains the focused canonical tracker rather than a duplicate.

## What This Changes

The issue requests a `cron.defaults.delivery` config option that supplies a global default cron announce delivery channel and target when individual jobs omit delivery routing.

## Best Possible Solution

Decide the cron output-default policy explicitly, then implement the chosen `cron.defaults.delivery` shape across config types, strict schema, cron creation/runtime paths, CLI/UI/docs, and precedence tests.

## Reproduction Assessment

Not applicable as a failing-bug repro; this is a feature request. The source-level gap is high confidence because current main and `v2026.6.8` expose no `cron.defaults.delivery` path and still use explicit delivery or implicit announce/last behavior.

## Solution Assessment

Yes, keeping this focused tracker open is the best current triage outcome. A narrow fallback-mode tweak would not solve the requested config default; maintainers need to approve the default-recipient policy and then update schema, runtime, CLI/UI, docs, and tests together.
