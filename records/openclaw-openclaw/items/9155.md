Keep open. Current main and v2026.6.6 still lack `cron.defaults.delivery` or an equivalent primary cron delivery default; related reports are duplicates, broader closed proposals, or adjacent delivery-default issues rather than a merged implementation.

## What This Changes

The issue requests a `cron.defaults.delivery` config path that supplies a default cron delivery mode/channel/target before falling back to the existing `last` route.

## Best Possible Solution

Define one maintainer-approved primary cron delivery-default contract, update schema/types/help/docs and job creation semantics together, and preserve explicit per-job delivery before any config default or `last` fallback.

Yes, at source level. Current main has no accepted `cron.defaults.delivery` config path, and omitted isolated cron delivery deterministically defaults to announce delivery with a `last` channel fallback.

Unclear pending maintainer direction. A cron config default is a plausible narrow solution, but maintainers need to choose creation-time versus run-time semantics and recipient precedence before implementation.
