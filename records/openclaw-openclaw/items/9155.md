## Summary

Keep open. Current `main` and latest release still lack `cron.defaults.delivery` or an equivalent primary cron delivery default, and related cleanup confirms this issue is the focused canonical tracker rather than a duplicate to close.

## What This Changes

The issue requests a `cron.defaults.delivery` config path so cron jobs can inherit a default announce delivery channel and target instead of repeating per-job delivery fields.

## Best Possible Solution

Decide the cron output-default policy explicitly, then implement the chosen `cron.defaults.delivery` shape across config types, strict schema/generated schema, cron creation/runtime paths, CLI/UI/docs, and precedence tests.

## Reproduction Assessment

Not applicable as a failing-bug repro; this is a feature request. The source-level gap is high confidence because current `main` and `v2026.6.8` expose no `cron.defaults.delivery` config path and still use explicit delivery or implicit announce/last behavior.

## Solution Assessment

Yes, keeping this open is the best current issue-triage outcome. A narrow fallback tweak would not be the best fix; maintainers need to approve the config/default policy and then update schema, runtime, UI/CLI, docs, and tests together.
