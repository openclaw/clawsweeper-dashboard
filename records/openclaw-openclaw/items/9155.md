## Summary

Keep open. Current main and the latest release still lack `cron.defaults.delivery` or an equivalent primary cron output default; the related issues are closed duplicates or partial overlaps, and the June 3 comment adds another default-target/privacy use case.

## What This Changes

Requested config support for a default cron delivery target so cron announce jobs do not need repeated per-job `delivery.channel` and `delivery.to` values.

## Best Possible Solution

Define one narrow primary cron announce delivery-default contract with explicit per-job precedence, documented persisted-job semantics, schema/help/docs updates, and regression coverage for the final `last` fallback.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `cron.defaults.delivery` config path, and omitted isolated cron delivery deterministically becomes announce delivery with a `last` channel fallback.

## Solution Assessment

Unclear. A config default is a plausible narrow remedy, but maintainers first need to decide the supported config scope and whether defaults are copied into persisted jobs or resolved at run time.
