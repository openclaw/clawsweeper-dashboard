## Summary

Keep this issue open. Current main still lacks `cron.defaults.delivery` or an equivalent primary cron announce delivery default, and the related reports either duplicate this tracker, solve adjacent routing/account cases, or are stale-closed broader proposals.

## What This Changes

The issue requests a `cron.defaults.delivery` config path that supplies a global channel/target default for cron job announcement delivery.

## Best Possible Solution

Define one narrow primary cron announce delivery-default contract with explicit per-job precedence, a documented creation-versus-execution persistence rule, schema/help/docs updates, and regression coverage for fresh jobs, existing jobs, channel/account/thread targets, and the final `last` fallback.

## Reproduction Assessment

Yes, at source level. Current main has no accepted `cron.defaults.delivery` config path, and omitted isolated cron delivery deterministically becomes announce delivery with a `last` channel fallback.

## Solution Assessment

Unclear. A config default is a plausible narrow remedy, but maintainers first need to decide the supported config scope and whether defaults are copied into persisted jobs or resolved at run time.

## Risks / Open Questions

- Applying a default dynamically at execution could silently retarget already-persisted jobs after config changes, while copying it only at creation would not fix existing jobs; maintainers need to choose and document that contract.
- A global recipient default is privacy-sensitive because a schema, precedence, account, or thread handling mistake could redirect many scheduled announcements to the wrong destination.
- The existing `cron.failureDestination` path is failure-notification-specific, so a primary delivery default should avoid reusing that name or semantics in a way that confuses operators.
