## Summary

Keep this issue open: current main and v2026.6.5 still lack a fleet-wide primary cron announcement destination, and recent routing improvements preserve active-chat context without satisfying the requested default for CLI-created or otherwise context-free jobs. The broader cron-defaults proposal remains open and bundles separate model and context-token policy, while an equivalent later report was explicitly closed in favor of this focused tracker.

## What This Changes

Requests a global `cron.defaults.delivery` setting that supplies a default cron announcement channel and recipient beneath explicit per-job delivery.

## Best Possible Solution

Add one canonical primary cron delivery-default surface with explicit per-job precedence, channel-owned target validation, clear creation-versus-execution semantics, and aligned config schema, CLI, Control UI, docs, and regression coverage.

## Reproduction Assessment

Yes at the source level: current config and strict schema lack the requested setting, omitted service-level delivery becomes announce-only, and the CLI retains `last` as its default channel. A live run is unnecessary to establish that the requested capability is missing.

## Solution Assessment

The requested outcome is appropriate, but the proposal is not implementation-ready until maintainers choose creation-time versus execution-time application and recipient-safety semantics. A focused primary-delivery default is more maintainable than coupling it to unrelated model and context-token defaults.
