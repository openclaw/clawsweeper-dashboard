## Summary

Keep open. Current main still lacks `cron.defaults.delivery` or an equivalent primary cron announce delivery default; strict config/schema surfaces reject the requested key, and job creation still preserves explicit delivery or falls back only to `{ mode: "announce" }`. The related #57638 is broader and still open, while #41164 was closed back to this focused tracker.

## What This Changes

Add `cron.defaults.delivery` so cron jobs can inherit a default delivery mode/channel/target instead of repeating per-job delivery config.

## Best Possible Solution

Add an approved cron-owned default delivery surface across config types, strict/generated schema/help/docs, Gateway/service job creation, CLI, Control UI, and regression tests with precedence: explicit job delivery, then cron default, then existing fallback.

## Reproduction Assessment

Not applicable as a feature request. Source verification is high confidence: current main's strict schema has no `cron.defaults`, and cron job creation only uses explicit delivery or the built-in announce fallback.

## Solution Assessment

Unclear pending maintainer design review. The requested precedence is plausible, but it adds a privacy-sensitive global scheduled-output recipient default and overlaps #57638, so the narrowest maintainable solution needs product approval first.
