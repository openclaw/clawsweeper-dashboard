## Summary

Keep open: current main still lacks `cron.defaults.delivery`, and the discussion already treats this as the focused delivery-default tracker while the broader cron-defaults design remains open.

## What This Changes

Add a `cron.defaults.delivery` config option so cron jobs can inherit a default announce channel and target when per-job delivery is omitted.

## Best Possible Solution

Add a documented, schema-backed cron delivery default, or fold the same precedence into the approved unified `cron.defaults` design, with regression coverage for per-job delivery before config default before `last`.

## Reproduction Assessment

Not applicable as a bug reproduction. This is a request for a new public config option, and source inspection verifies that the requested path is absent from current main.

## Solution Assessment

Unclear until maintainers choose the public config contract. `cron.defaults.delivery` is a plausible focused API, but the safer path is to settle its precedence and overlap with the broader cron defaults proposal before implementation.
