## Summary

Keep this open. Current main still has no `cron.defaults.delivery` or equivalent primary cron announce delivery default: the config type/schema only expose scheduler settings plus failure-notification defaults, and cron job creation still preserves explicit job delivery or falls back to announce without a configured channel/target. The broader #57638 remains open and overlaps this request, but the provided discussion already kept this issue as the focused delivery-default tracker.

## What This Changes

Add a `cron.defaults.delivery` configuration default that cron jobs use when per-job delivery is omitted, before falling back to the current `last` behavior.

## Best Possible Solution

Add a first-class `cron.defaults.delivery` surface across config types, strict schema, generated schema/help, Gateway cron job normalization, CLI, Control UI, docs, and regression tests for precedence: explicit per-job delivery, then cron default delivery, then existing fallback behavior. Keep it separate from `cron.failureDestination`, which is failure-notification-specific.

## Reproduction Assessment

Not applicable as a feature request. We do have a high-confidence source verification path: current main rejects `cron.defaults.delivery` through the strict config schema and no cron creation path reads a global delivery default.

## Solution Assessment

Unclear pending maintainer design review. The requested precedence is plausible and narrowly scoped, but it overlaps #57638 and recipient-default privacy behavior, so the best fix should be an approved cron-owned default rather than repurposing failure-notification config.
