## Summary

Keep this issue open: current main and v2026.6.5 still lack a stable fleet-wide destination for ordinary cron announcements. The broader https://github.com/openclaw/openclaw/issues/57638 adds separate model and context policy and remains unresolved, while the equivalent later report https://github.com/openclaw/openclaw/issues/41164 was closed in favor of this focused tracker.

## What This Changes

Add a global cron delivery default for mode, channel, and recipient, with explicit per-job delivery taking precedence.

## Best Possible Solution

Define one canonical cron delivery-default contract with explicit job-level precedence, validated recipient semantics, and aligned config types, strict schema, help, docs, CLI, Control UI, and regression coverage for fresh and existing jobs.

## Reproduction Assessment

Yes at source level: create an isolated agent-turn cron without explicit delivery and current main stores announce mode without a stable channel or recipient. The same missing capability is present in v2026.6.5.

## Solution Assessment

Unclear: a dedicated cron default is the narrowest apparent capability, but maintainers must decide its canonical shape, creation-versus-execution timing, and recipient-safety rules.
