## Summary

Keep open. Current main still has the reported exec-safety gap: host exec defaults to `security="full"` plus `ask="off"`, the approval predicate does not prompt in that mode, and the hard-coded exec control guard only blocks `/approve` and interactive `openclaw channels login`, not package-manager self-removal such as `npm uninstall -g openclaw`. The related open #54416 is adjacent self-management exec-safety work, but it does not fully supersede this specific self-uninstall report.

## What This Changes

The issue requests safeguards so agent-issued destructive package-manager commands do not misinterpret a similarly named package and uninstall OpenClaw itself without explicit confirmation.

## Best Possible Solution

Keep this open until OpenClaw either ships a shared exec approval/analysis boundary for high-risk OpenClaw self-management commands or explicitly documents that trusted-operator YOLO mode permits this class of self-removal. A maintainable fix should cover gateway and node host exec consistently, at minimum requiring explicit approval or denial for package-manager removal of OpenClaw itself and coordinating with the broader self-management command policy tracked by #54416.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially yes: current source shows `npm uninstall -g openclaw` would not hit a hard approval rule under host `full/off`, though I did not execute the destructive command. The exact `opencli` to `openclaw` language confusion remains provider-specific.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide the YOLO policy boundary; prompt-only guidance is not sufficient because the raw exec gate still permits the command. The safer direction is a shared exec approval/analysis rule for OpenClaw self-management and package-manager self-uninstall commands, plus docs/tests for the intended exception behavior.
