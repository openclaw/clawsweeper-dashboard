## Summary

Keep open. Current `main` still has no deterministic self-uninstall approval class: host exec remains `security="full"` plus `ask="off"` by default, the approval predicate does not prompt in that mode, and the only hard OpenClaw CLI shell block I found is for `/approve` and interactive channel login.

## What This Changes

The issue asks OpenClaw to avoid conflating `opencli` with `openclaw` and to require explicit confirmation before executing destructive self-uninstall commands.

## Best Possible Solution

Define a shared hard approval boundary for OpenClaw package-manager self-removal and related self-management commands, with docs and regression tests that state how this exception composes with full/off host exec.

## Reproduction Assessment

Yes at source level: current `security="full"` plus `ask="off"` host exec will not prompt for a raw `npm uninstall -g openclaw` command, though I did not run the destructive command. The exact `opencli` to `openclaw` model confusion is not independently reproducible from source alone.

## Solution Assessment

Unclear until maintainers define the YOLO-mode exception. The maintainable fix is likely a deterministic exec approval/preflight rule for OpenClaw self-uninstall and other high-risk self-management commands, not a prompt-only mitigation.
