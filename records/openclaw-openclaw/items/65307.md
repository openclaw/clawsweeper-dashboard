## Summary

Keep open. Current `main` and the latest shipped tag still lack a deterministic approval or preflight class for OpenClaw package-manager self-removal; the related open trackers cover config/restart safety and a broader tool-confirmation design, but neither fully supersedes this self-uninstall case.

## What This Changes

The issue asks OpenClaw to avoid confusing `opencli` with `openclaw` and to require explicit confirmation before model-issued host exec can uninstall OpenClaw itself.

## Best Possible Solution

Define a deterministic exec approval or preflight boundary for OpenClaw self-management commands, especially package-manager self-removal, and document how that exception composes with intentional full/off host automation.

## Reproduction Assessment

Yes at source level: current full/off host exec does not require approval for a raw package-manager command that removes OpenClaw. I did not run a destructive uninstall command, and the exact model-name confusion is not independently reproducible from source alone.

## Solution Assessment

Unclear as an unattended repair. The maintainable direction is likely a deterministic exec approval/preflight class for OpenClaw self-management commands, but maintainers need to define the exception to documented full/off semantics first.
