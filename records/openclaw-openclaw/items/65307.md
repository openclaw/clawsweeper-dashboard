## Label Justifications

- `P1`: The report describes a concrete model-issued host exec path that can remove the installed OpenClaw runtime without deterministic confirmation.
- `impact:security`: The issue is about the raw host-exec authorization boundary for sensitive OpenClaw self-management commands.
- `impact:crash-loop`: A package-manager self-uninstall can make the local OpenClaw runtime unavailable even though the process failure is caused by removal rather than a code crash.

## Summary

Keep open. Current main and v2026.5.18 still lack a deterministic approval or preflight class for OpenClaw package-manager self-removal through raw host exec; the related confirmation and config/restart issues are adjacent but do not fully supersede this self-uninstall case.

## What This Changes

Request a safeguard so model-issued exec cannot confuse an unrelated opencli package with OpenClaw and remove the global OpenClaw install without explicit confirmation.

## Best Possible Solution

Define a deterministic raw-exec approval or preflight boundary for OpenClaw self-management package-removal commands, then document how that exception composes with intentional full/off automation.

## Reproduction Assessment

Yes at source level: current gateway/node full/off host exec does not require approval for a raw package-manager command that removes OpenClaw. I did not run a destructive live uninstall, and the exact model-name confusion is not independently reproducible from source alone.

## Solution Assessment

Unclear as an unattended repair. The maintainable direction is a deterministic self-management approval/preflight class, but maintainers need to define the exception to documented full/off semantics first.
