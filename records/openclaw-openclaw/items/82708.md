## Summary

Keep open. Current main still has a source-level path where a heartbeat run can receive a generic foreground request failure payload and deliver it as a heartbeat alert when alerts are enabled.

## What This Changes

Request to classify heartbeat/background agent-run failures separately from foreground Discord request failures so the generic `/new` guidance is not posted for heartbeat timeouts.

## Best Possible Solution

Classify agent-runner failures with heartbeat context before delivery, emit failed heartbeat diagnostics by default, and only send heartbeat-specific Discord text under the chosen heartbeat visibility policy.

## Reproduction Assessment

Yes, source-level: a heartbeat run with `showAlerts` enabled that receives the generic final error payload from `getReplyFromConfig` follows the normal heartbeat alert delivery path. I did not run a live Discord/Codex timeout in this read-only review.

## Solution Assessment

Unclear pending maintainer policy. The narrow technical boundary is heartbeat-aware failure classification, but the visible behavior needs a decision between suppression and heartbeat-specific alert copy.
