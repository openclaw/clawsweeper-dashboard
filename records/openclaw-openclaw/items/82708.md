## Summary

Keep open. Current main still has a source-visible path where heartbeat runs use the normal agent reply path and a generic agent-runner failure payload can be normalized and delivered as heartbeat alert content when `showAlerts` is enabled.

## What This Changes

The issue asks OpenClaw to classify Discord heartbeat/background failures separately from foreground user-request failures so generic `/new` guidance is not posted for heartbeat timeouts.

## Best Possible Solution

Add heartbeat-aware failure classification before outbound delivery or generic failure-text construction, preserving real heartbeat diagnostics while replacing or suppressing foreground user-request copy.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only pass. The source path is clear: a heartbeat `getReplyFromConfig` result containing generic failure copy is normalized and delivered by `runHeartbeatOnce` when alerts are visible.

## Solution Assessment

Yes. The narrow maintainable fix is heartbeat-aware failure mapping in the agent-runner or heartbeat delivery path, not the broader persistent task-status feature or a new channel-wide error-policy option.
