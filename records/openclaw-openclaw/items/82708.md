## Summary

Keep open. Current main still has a source-visible path where heartbeat runs can receive the generic foreground agent-runner failure payload and `runHeartbeatOnce` can deliver that payload as a heartbeat alert when `showAlerts` is enabled; no linked PR or duplicate now owns this exact heartbeat/background classification bug.

## What This Changes

Ensure Discord heartbeat/background failures are classified separately from foreground user-request failures so generic `/new` error copy is not posted for heartbeat lanes.

## Best Possible Solution

Add heartbeat/background-aware failure classification before outbound delivery, preserving existing `showAlerts` semantics while replacing or suppressing foreground `/new` copy for heartbeat runner failures.

## Reproduction Assessment

Yes at source level, not as a live Discord timeout: stub `getReplyFromConfig` to return the generic agent-runner failure payload during `runHeartbeatOnce` with a Discord target and `showAlerts` enabled, and current main will normalize and send it as heartbeat output.

## Solution Assessment

Yes. The narrow maintainable solution is to branch on heartbeat/background context at failure-text construction or heartbeat delivery, without changing foreground user-request errors or adding a new config surface.
