## Summary

Keep open. Current main still has a source-visible path where a heartbeat run can receive the generic agent-runner failure payload and the heartbeat runner then treats that payload as a normal heartbeat alert for Discord delivery, so the report is valid and not implemented or superseded.

## What This Changes

The issue asks OpenClaw to classify Discord heartbeat/background failures separately from foreground user-request failures so generic `/new` guidance is not posted for heartbeat timeouts.

## Best Possible Solution

Implement a small heartbeat failure-classification branch that preserves heartbeat diagnostics while preventing generic foreground request copy from being delivered to chat.

## Reproduction Assessment

No live Discord/Codex timeout reproduction was run, but the source path is clear: a heartbeat `getReplyFromConfig` result containing the generic failure payload is normalized and sent by `runHeartbeatOnce` when alerts are visible.

## Solution Assessment

Yes. The narrow maintainable fix is to branch on heartbeat/background context before outbound delivery or failure-text construction, keep existing heartbeat visibility semantics, and avoid folding this into the broader persistent task-status request.
