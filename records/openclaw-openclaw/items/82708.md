## Summary

Keep open. Current main still has a source-level path where heartbeat runs can receive the generic foreground request failure payload and deliver it as a heartbeat alert when `showAlerts` is enabled; the related Codex timeout fix does not cover the background-vs-foreground classification decision.

## What This Changes

The issue requests heartbeat/background failures to avoid surfacing the generic foreground Discord request failure text when the failed run was not user-initiated.

## Best Possible Solution

Add heartbeat-aware failure classification at the agent-runner or heartbeat-runner boundary so generic foreground failures become heartbeat failed diagnostics by default, with any visible Discord copy explicitly heartbeat-specific and governed by heartbeat visibility policy.

## Reproduction Assessment

Yes, source-level: force a heartbeat run with `showAlerts` enabled to receive the generic final error payload from `getReplyFromConfig`, and current main will normalize and send it through heartbeat alert delivery. A live Discord/Codex timeout repro remains unrun.

## Solution Assessment

Unclear pending maintainer policy. The narrow code boundary is clear, but maintainers should choose silent-by-default versus heartbeat-specific visible copy before changing Discord-facing behavior.
