## Summary

Keep open. Current main still has a source-level path where a heartbeat run can convert an embedded-agent failure into the generic foreground request failure payload and then deliver it through heartbeat chat delivery when alerts are enabled; related fixes cover Codex liveness and duplicate message-tool timeout payloads, but not this background/foreground classification gap.

## What This Changes

The issue asks OpenClaw to classify Discord heartbeat/background failures separately from foreground user-request failures so generic `/new` recovery text is not posted for heartbeat timeouts.

## Best Possible Solution

Add heartbeat-aware failure classification at the agent-runner or heartbeat-runner boundary so generic foreground request failures become heartbeat failed diagnostics by default, with any visible Discord copy explicitly heartbeat-specific and governed by heartbeat visibility policy.

## Reproduction Assessment

Yes, source-level. A heartbeat run with `showAlerts` enabled that receives the generic final error payload from `getReplyFromConfig` will follow the normal heartbeat alert delivery path; a live Discord/Codex timeout still needs validation.

## Solution Assessment

Unclear pending maintainer policy. The narrow technical fix is clear, but maintainers should decide whether the default is suppression or heartbeat-specific visible copy before changing user-facing Discord behavior.
