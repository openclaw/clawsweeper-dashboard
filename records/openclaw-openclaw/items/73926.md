## Summary

Keep open. Current main has partial fixes for bare reset markers and blank visible user prompts, but the runtime-only hidden-context path still exempts empty prompts from the guard and submits `prompt: ""`; that leaves the provider-safe empty-input contract unresolved for memory, active-memory, heartbeat, and Anthropic-compatible provider paths.

## What This Changes

The issue requests preventing `/new` or `/reset` with active-memory/reset races from dispatching an empty prompt and empty message list to Anthropic-compatible or Responses providers.

## Best Possible Solution

Define a provider-safe runtime-only contract at the embedded-runner or provider-adapter boundary: hidden runtime context should stay out of visible transcript turns, but provider calls should never contain no model-visible input. The design should explicitly cover memory flush, active-memory prompt context, heartbeat/system events, and Anthropic-compatible providers.

## Reproduction Assessment

Yes, at source level. A non-empty `effectivePrompt` with blank `transcriptPrompt` becomes `prompt: ""` and `runtimeOnly: true`, the skip helper exempts it, and the runner submits that empty prompt; existing tests assert the empty runtime-only submission shape.

## Solution Assessment

No. The local patch direction of injecting a visible greeting everywhere is not clearly the narrowest maintainable fix because it can change hidden runtime-context and transcript semantics; a provider-safe runtime-only sentinel or adapter contract is safer.
