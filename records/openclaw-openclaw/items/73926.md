## Summary

Keep open. Current main covers the simple bare `/new` transcript marker path, but it still preserves a runtime-only path where hidden context can produce `prompt: ""`, bypass the empty-submission guard, and call the embedded session with that empty visible prompt; Anthropic-compatible transport then builds `messages` from the session message list without a fallback for an empty list.

## What This Changes

Reports that `/new` or `/reset`, especially during active-memory rebuild races, can dispatch an embedded agent run with empty prompt/messages to Anthropic-compatible providers and leave the session stuck after provider 400s.

## Best Possible Solution

Define a provider-safe runtime-only input contract at the embedded runner or provider-adapter boundary: hidden runtime context should stay out of persisted visible user turns, but provider calls must never contain no model-visible messages. The fix should cover active-memory startup, memory flush, context-engine runtime context, and Anthropic/MiniMax-compatible transports with focused regression tests.

## Reproduction Assessment

Yes, at source level. A run with non-empty `effectivePrompt` and blank `transcriptPrompt` becomes `prompt: ""` plus `runtimeOnly: true`, the guard permits it, and current tests assert the empty runtime-only submission shape.

## Solution Assessment

No, current main is only a partial solution. A broad visible greeting injection like the local workaround could change transcript semantics; a sentinel/runtime-only transport contract or provider-boundary guard is safer.
