## Summary

Keep open. Current main and v2026.4.29 partially address the bare `/new` marker and blank visible-prompt cases, but the runtime-only hidden-context path still exempts empty prompts from the guard and can submit `prompt: ""` to the embedded session, so the provider-safe empty-input contract is not terminally fixed.

## What This Changes

The issue requests preventing `/new` or `/reset` races with active-memory/reset handling from dispatching an empty prompt and empty message list to Anthropic-compatible or Responses providers.

## Best Possible Solution

Define a provider-safe runtime-only contract at the embedded-runner or provider-adapter boundary so hidden runtime context can stay out of visible transcript turns while provider calls never contain no model-visible input. The design should explicitly cover memory flush, active-memory prompt context, heartbeat/system events, and Anthropic-compatible transports.

## Reproduction Assessment

Yes, at source level. A non-empty `effectivePrompt` with blank `transcriptPrompt` becomes `prompt: ""` and `runtimeOnly: true`; the skip helper exempts it and the runner submits that empty prompt, with tests asserting the empty runtime-only shape.

## Solution Assessment

No. Current main is only a partial solution, and the local patch idea of injecting a visible greeting everywhere is not clearly the best fix because it can alter hidden runtime-context and transcript semantics; a sentinel or transport/provider contract is safer.
