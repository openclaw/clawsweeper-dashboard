## Summary

Keep open. Current main covers the plain bare-reset marker and blank visible-user-prompt cases, but it still intentionally exempts runtime-only prompt submissions from the empty-input guard and then calls the embedded session with `prompt: ""`; that leaves the reported active-memory/runtime-context path unresolved for strict Anthropic-compatible transports.

## What This Changes

The issue asks OpenClaw to prevent `/new` or `/reset` active-memory races from dispatching an empty prompt and empty message list to providers such as Anthropic or OpenAI Responses.

## Best Possible Solution

Define a provider-safe runtime-only contract at the embedded-runner or provider-adapter boundary: hidden active-memory and memory-flush context should remain out of visible transcript turns, while every provider dispatch must contain model-visible input acceptable to Anthropic-compatible and OpenAI Responses transports. Coordinate that with PR #74731 or an equivalent transport guard, then add regression coverage for runtime-only empty visible prompts.

## Reproduction Assessment

Yes, at source level. A non-empty runtime prompt with `transcriptPrompt: ""` becomes `prompt: ""` and `runtimeOnly: true`; the skip helper allows it, the runner submits it, and current tests assert the empty runtime-only shape.

## Solution Assessment

No, the local patch suggested in the issue is not clearly the best fix because injecting a visible greeting at the runner boundary can change hidden runtime-context and transcript semantics. A narrower provider-safe sentinel or final transport guard, paired with explicit runtime-only semantics, is safer.
