## Summary

Keep open for maintainer review: the reported Windows PID/uptime mismatch is source-reproducible, but this PR reverses a shipped unmanaged-Windows safety branch and exits into a detached child without normal-restart health proof.

## What This Changes

The branch changes `restartGatewayProcessWithFreshPid()` so unmanaged Windows `/restart` attempts detached respawn instead of returning disabled for the in-process fallback.

## Best Possible Solution

Maintainers should decide the unmanaged Windows restart contract; if fresh PID is desired, implement a health-checked Windows handoff with live Windows proof, otherwise keep the current in-process fallback and clarify status/docs expectations.

## Reproduction Assessment

Yes, at source level: current main returns `disabled` for win32 without Scheduled Task markers, and the run loop then takes the in-process restart fallback. I did not run a live Windows gateway in this read-only review.

## Solution Assessment

No. The patch is a plausible way to force a fresh PID, but it is not the best safe fix as-is because it reverses a shipped safety decision without a health-checked handoff or live Windows proof.
