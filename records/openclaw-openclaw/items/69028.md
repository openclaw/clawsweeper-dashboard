## Summary

Keep open: current main enforces the Feishu media size cap earlier, but the Feishu inbound handler still logs download failures and can continue into a normal agent turn with only the media placeholder, so the reported user-visible silent-failure behavior remains.

## What This Changes

The issue requests a user-facing Feishu chat error when inbound media exceeds `mediaMaxMb` instead of silently omitting the file from the agent turn.

## Best Possible Solution

Teach the Feishu plugin to surface inbound media-download failures to the triggering chat or thread and stop the normal agent turn when required media was dropped, without adding a new config option.

## Reproduction Assessment

Yes, from source inspection: force `saveMessageResourceFeishu` to throw an over-limit error and `resolveFeishuMediaList` will log it, return no media, and let `handleFeishuMessage` dispatch the placeholder-bearing turn. I did not run a live Feishu account in this read-only pass.

## Solution Assessment

Yes. The narrow fix is to propagate media-download failure state from the Feishu media resolver to the message handler, send a visible Feishu reply, and return before agent dispatch; this matches the existing Telegram failure pattern and avoids new policy or config surface.
