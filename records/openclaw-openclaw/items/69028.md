## Summary

Current main fixed earlier Feishu over-limit buffering and shared media cap enforcement, but it still treats an oversized inbound Feishu media download as a logged failure with an empty media list. The related pull request and EPERM issue are useful context, yet neither added the requested user-facing Feishu failure reply or agent-turn block.

## What This Changes

The issue requests a user-facing Feishu notification and workflow guard when an inbound media file is dropped because it exceeds `mediaMaxMb`.

## Best Possible Solution

Add a Feishu-owned media failure result that sends a reply to the source chat and prevents a normal agent turn when required inbound media is rejected by the configured cap.

## Reproduction Assessment

Yes, from source inspection. Oversized downloads throw in `extensions/feishu/src/media.ts`, `resolveFeishuMediaList` catches and logs them, and `extensions/feishu/src/bot.ts` continues the normal agent dispatch path unless the message has no text and no media.

## Solution Assessment

Yes. The narrow maintainable fix is to return explicit Feishu media failure metadata from `resolveFeishuMediaList`, send a user-facing reply with the configured limit, and block the agent turn for messages whose media was not actually delivered.

## Repair Work Prompt

Fix the Feishu inbound media silent-failure bug for https://github.com/openclaw/openclaw/issues/69028. Current main enforces `mediaMaxMb` during Feishu downloads, but `resolveFeishuMediaList` catches over-limit/download failures, logs them, returns an empty media list, and `bot.ts` can continue to a normal agent turn. Keep the fix inside `extensions/feishu`: return explicit media failure metadata from `resolveFeishuMediaList` or an equivalent typed helper, send a user-facing Feishu reply to the same chat/thread with the configured size limit for oversize media and a clear download-failed message for other download failures, and prevent normal agent execution when the inbound message's required media was dropped. Preserve successful media handling, Feishu `type=file` to `type=media` fallback behavior, rich post parsing, and the shared media-store/download helper contracts. Add focused regression tests in Feishu bot/media tests proving an over-limit media download notifies the user and does not dispatch to the agent, plus a successful under-limit media case still dispatches. Do not add a config option or move this into core unless a missing SDK seam is proven.
