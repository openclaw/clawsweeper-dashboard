## Summary

Keep open: current main still turns Feishu media download and size-limit failures into logs while preserving an agent-facing media placeholder, so the silent user-facing failure remains.

## What This Changes

The issue requests a user-facing Feishu chat error when inbound media exceeds `mediaMaxMb` instead of only logging the dropped attachment.

## Best Possible Solution

Feishu should notify the same conversation that the attachment was not accepted or downloaded and suppress a normal agent turn that implies the missing file was processed.

## Reproduction Assessment

Yes, from source inspection: mock `saveMessageResourceFeishu` or the media store to reject with `Media exceeds 30MB limit` for a Feishu media message and the handler will log the failure while returning no media. I did not run a live Feishu account reproduction in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is plugin-local: classify Feishu inbound media download/size failures, send a concise Feishu reply using the existing send helper, and avoid dispatching the normal agent response for the dropped attachment.
