## Summary

Keep open. Current main still catches Feishu inbound media download/save failures as log-only errors, returns an empty attachment list, and can continue normal agent dispatch with a nonblank media placeholder and no Feishu-visible warning.

## What This Changes

The issue asks Feishu to notify the source chat and avoid success-looking agent processing when inbound media is dropped because it exceeds `mediaMaxMb`.

## Best Possible Solution

Add a Feishu-owned media resolution result that reports dropped media to the source chat/thread and prevents normal agent turns when all requested media was undelivered, while preserving successful attachments on partial failures.

## Reproduction Assessment

Yes, at source level. A focused test can mock Feishu media download/save so `saveMediaBuffer` throws for a file event with a filename, then assert current main logs the failure and still reaches dispatch because parsed placeholder content is nonblank.

## Solution Assessment

Yes. A plugin-local structured failure plus Feishu-visible warning is the narrowest maintainable fix because the Feishu plugin owns the reply target, account config, media parsing, and download behavior.
