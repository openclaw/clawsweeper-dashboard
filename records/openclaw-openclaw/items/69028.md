- `impact:message-loss`: The issue is about inbound media being silently dropped from the channel message flow.
- `impact:session-state`: The agent can continue a turn and produce transcript state as if the missing media had been received.

## Summary

Keep open. Current main and the latest release still log Feishu inbound media download failures without exposing failure state to the handler, so captioned or placeholder-bearing messages can continue into a normal agent turn after required media was dropped.

## What This Changes

The issue requests a user-facing Feishu reply and blocked agent turn when inbound media exceeds the configured media download limit.

## Best Possible Solution

Add a Feishu-owned media failure result that sends a user-facing reply with the configured limit or generic download error, then blocks the normal agent turn when required inbound media was rejected.

## Reproduction Assessment

Yes, from source inspection. Oversized downloads throw in Feishu media helpers, `resolveFeishuMediaList` catches and logs the failure, and the handler continues whenever the message still has non-empty content.

## Solution Assessment

Yes. The narrow maintainable fix is to return explicit Feishu media failure metadata, send a user-facing reply from the Feishu handler, and prevent a normal agent turn for media that was not delivered.
