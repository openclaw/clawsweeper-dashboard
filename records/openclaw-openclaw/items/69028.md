## Summary

Keep open. Current main enforces the Feishu media byte cap earlier, but over-limit or failed inbound media is still handled as a log-only drop; placeholder text can keep the agent turn nonblank, so normal dispatch remains reachable without a Feishu-visible warning.

## What This Changes

The issue requests a user-facing Feishu warning and agent-dispatch guard when inbound media is dropped because it exceeds the configured media size cap.

## Best Possible Solution

Add a Feishu-owned media resolution result that reports dropped media to the same chat/thread and prevents normal agent turns when all requested media was undelivered, while preserving successful attachments on partial failures.

## Reproduction Assessment

Yes, at source level. A focused test can mock Feishu media download/save to reject for an over-limit file with a filename, then assert the current path logs the failure, retains placeholder text, and still reaches normal dispatch with no media attachment.

## Solution Assessment

Yes. A plugin-local structured failure plus Feishu-visible warning is the narrow maintainable fix because the Feishu plugin owns the reply target, media parsing, account config, and inbound dispatch decision.
