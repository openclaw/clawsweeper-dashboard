Keep this PR open for human review: the linked logs timestamp problem is still present on current `main`, but the PR still needs stronger real behavior proof, docs/test alignment, and maintainer acceptance of the changed CLI contract.

## What This Changes

This PR changes `openclaw logs` to render human-readable timestamps in local time by default, keeps `--local-time` accepted, adds `--utc`, and includes type-only image parser cleanup.

## Best Possible Solution

Land or replace this with a logs-only change that defaults human-readable `openclaw logs` output to local time, preserves compatibility, documents and tests `--utc`, and leaves image cleanup or console logging scope to separate work.

## Reproduction Assessment

Yes, at source level. Current `main` and `v2026.5.12` parse the local-offset `time` field but render human-readable output as UTC unless `--local-time` or a valid `TZ` enables local mode.

## Solution Assessment

No, not as submitted. The core `localTime = !opts.utc` direction is narrow, but the maintainable PR still needs docs, focused default-local and `--utc` coverage, stronger real behavior proof, and removal or separation of unrelated cleanup.
