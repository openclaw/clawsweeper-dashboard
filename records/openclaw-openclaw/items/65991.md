## Summary

Keep open. Current main still lacks a `failureAlert.silent` schema, config, UI, and runtime forwarding path, while the shared message delivery and Telegram adapter already support silent sends; this is a narrow but public config feature that needs maintainer acceptance.

## What This Changes

The issue requests an optional `failureAlert.silent` cron setting so Telegram failure alerts can be delivered without sound or push notification.

## Best Possible Solution

Add a backward-compatible `silent?: boolean` failure-alert option and thread it through cron config, Gateway/tool schemas, runtime alert resolution, announce delivery, CLI/UI editing, docs, and focused tests while preserving omitted/false behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows strict cron config/Gateway schemas do not accept `failureAlert.silent`, and the runtime alert path has no value to forward to durable delivery.

## Solution Assessment

Yes, if maintainers accept the feature. The maintainable direction is to pass a generic `failureAlert.silent` option into the existing outbound silent-delivery path, not to add Telegram-specific branching in cron core.
