## Summary

Keep open. Current main still lacks a `failureAlert.silent` cron alert contract and runtime path, while shared outbound delivery and the Telegram plugin already support quiet sends; this is a narrow but public feature/config addition rather than a closeable cleanup item.

## What This Changes

Request to add a `failureAlert.silent` boolean so Telegram cron failure alerts can be delivered without sound or push notification.

## Best Possible Solution

Add a backward-compatible `silent?: boolean` failure-alert option and thread it through cron config, Gateway schemas, runtime alert resolution, announce delivery, CLI/UI editing, docs, and focused tests while leaving webhook alerts and omitted/false values unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows the strict Gateway schema rejects `failureAlert.silent`, and the runtime/UI/CLI delivery paths have no field that could preserve or send it.

## Solution Assessment

Yes, the requested additive field is the narrowest maintainable direction if maintainers want the feature. It should pass the existing generic outbound `silent` option through cron delivery rather than adding Telegram-specific branching in core.
