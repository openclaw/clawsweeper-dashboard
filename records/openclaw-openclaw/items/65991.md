## Summary

Keep open. Current main still lacks a `failureAlert.silent` contract and runtime forwarding path, while the existing generic message delivery stack already supports quiet sends; this is a narrow public feature/config addition that needs maintainer acceptance.

## What This Changes

Add a `silent: boolean` option to cron job `failureAlert` so Telegram failure-alert messages can be delivered without sound or push notification.

## Best Possible Solution

Add a backward-compatible `silent?: boolean` failure-alert option and thread it through cron config, Gateway schemas, runtime alert resolution, announce delivery, CLI/UI editing, docs, and focused tests while leaving omitted/false values and webhook delivery unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows strict cron config/Gateway schemas do not accept `failureAlert.silent`, and the runtime announce path has no value to forward to durable delivery; I did not run a live Telegram send.

## Solution Assessment

Yes, if maintainers accept the feature. The narrow maintainable solution is to pass the existing generic outbound `silent` option through cron failure-alert announce delivery, not to add Telegram-specific branching in core.
