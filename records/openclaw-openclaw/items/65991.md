## Summary

Keep open. Current main still lacks `failureAlert.silent` across the strict cron contract, runtime alert resolution, Gateway announce delivery, Control UI, CLI/tool surfaces, and docs, while shared outbound delivery and the Telegram plugin already support quiet sends.

## What This Changes

Add an additive `silent` option to cron `failureAlert` so Telegram failure alerts can be delivered without sound or push notification.

## Best Possible Solution

Add a backward-compatible `silent?: boolean` failure-alert field and thread it through cron config/protocol, runtime resolution, Gateway announce delivery, UI, CLI/tool schemas, docs, and tests while keeping omitted or false alerts audible.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `cron.add` or `cron.update` payload containing `failureAlert.silent` is outside the strict Gateway schema, and no current runtime/UI path can preserve or deliver it.

## Solution Assessment

Yes. The requested additive field is the narrowest maintainable direction because shared outbound delivery and Telegram already support `silent`; cron should pass a generic delivery option rather than add Telegram-specific branching.
