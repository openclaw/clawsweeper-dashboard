## Summary

Keep open. Current main still lacks a `failureAlert.silent` contract and runtime forwarding path, while existing outbound and Telegram delivery already support silent sends; this is a narrow public feature/config addition that needs maintainer acceptance.

## What This Changes

Request to add a `silent: boolean` option to cron `failureAlert` so Telegram failure alerts can be delivered without a push or sound notification.

## Best Possible Solution

Add a backward-compatible `silent?: boolean` failure-alert option and thread it through cron config, Gateway/tool schemas, runtime alert resolution, announce delivery, CLI/UI editing, docs, and focused tests while leaving omitted/false values and webhook delivery unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows strict cron config/Gateway schemas do not accept `failureAlert.silent`, and the runtime alert path has no value to forward to durable delivery; I did not run a live Telegram send.

## Solution Assessment

Yes, if maintainers accept the feature. The narrow maintainable solution is to pass a generic `failureAlert.silent` option into the existing outbound silent-delivery path, not to add Telegram-specific branching in cron core.
