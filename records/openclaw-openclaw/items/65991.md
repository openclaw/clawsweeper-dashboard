## Summary

Keep open. Current main and the latest release still lack a `failureAlert.silent` contract and runtime path, while the existing outbound Telegram delivery primitive already supports quiet sends; this is a narrow public feature/config addition that needs maintainer acceptance.

## What This Changes

The issue requests a `silent` boolean option on cron `failureAlert` so Telegram failure alerts can be delivered without notification sound while still appearing in chat.

## Best Possible Solution

Add a backward-compatible `silent?: boolean` failure-alert option and thread it through cron config, Gateway schemas, runtime alert resolution, announce delivery, CLI/UI editing, docs, and focused tests while leaving webhook alerts and omitted/false values unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows strict schemas reject `failureAlert.silent`, and the runtime delivery/UI/CLI paths have no value that could preserve or forward it.

## Solution Assessment

Yes, if maintainers accept the feature. The narrow technical solution is to pass the existing generic outbound `silent` option through cron announce failure-alert delivery, not to add Telegram-specific branching in core.
