## Summary

Keep open: current main and v2026.6.10 still reject remote non-localhost HTTP Control UI connections without browser device identity, and docs/tests codify that as a security boundary. The remaining request is a security/product decision for homelab remote-HTTP ergonomics, with narrower warning/default PRs still open rather than a merged fix.

## What This Changes

Request to make Control UI usable for Docker, VPS, and NAS remote HTTP deployments that hit the secure-context device-identity requirement.

## Best Possible Solution

Keep the secure default while maintainers decide the remote-HTTP identity policy; land narrow warning/docs/default improvements if accepted, and only support plaintext browser identity with explicit security approval, docs, tests, and real LAN proof.

## Reproduction Assessment

Yes. Source inspection and current tests show remote non-localhost Control UI without device identity is rejected even when `allowInsecureAuth` is configured; I did not run a live Docker/LAN browser repro in this read-only review.

## Solution Assessment

Unclear. Warning and copy improvements are the safest narrow mitigation, but enabling persistent device identity or a new opt-in mode over plaintext non-localhost HTTP is only the best fix if maintainers intentionally accept that security model.
