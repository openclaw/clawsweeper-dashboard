## Summary

Keep open. Current main still enforces the documented HTTPS-or-localhost Control UI device-identity boundary; the open linked pull request only clarifies messaging, while the broader remote plain-HTTP identity behavior remains a security/product decision tracked in a canonical follow-up.

## What This Changes

The issue asks whether Control UI device identity should work from non-local plain-HTTP Docker or homelab deployments instead of requiring HTTPS or localhost.

## Best Possible Solution

Keep the secure default, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and resolve remote plain-HTTP browser device identity in https://github.com/openclaw/openclaw/issues/53274 with explicit threat-model coverage.

## Reproduction Assessment

Yes, source-backed. A non-local plain-HTTP browser lacks the WebCrypto secure-context path used for Control UI device identity, so the UI does not attach device identity and the gateway rejects the device-less session.

## Solution Assessment

Unclear. Clarifying the current error is narrow and maintainable, but supporting remote plain-HTTP browser device identity changes a documented security boundary and needs maintainer approval.
