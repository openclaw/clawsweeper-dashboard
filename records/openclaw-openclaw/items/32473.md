## Summary

Keep open. Current main still intentionally rejects remote plain-HTTP Control UI sessions that cannot present device identity, the issue is paired with an open messaging PR that uses closing syntax, and the broader LAN HTTP identity policy remains a security/product decision in a separate open issue.

## What This Changes

The issue asks for Control UI to connect from Docker, VPS, or LAN remote HTTP deployments without failing on the device-identity secure-context requirement.

## Best Possible Solution

Keep the secure default, review or replace the open messaging PR for clearer guidance, and handle any remote HTTP device-identity redesign in the canonical security/product issue.

## Reproduction Assessment

Yes for the current failure path. Source inspection shows remote plain HTTP skips browser device identity and the gateway rejects non-local Control UI without a device; I did not run a live Docker/browser scenario in this read-only review.

## Solution Assessment

Unclear. Clarifying messaging is a narrow safe improvement, but accepting device identity or operator auth over remote plain HTTP would change the documented security model and needs maintainer threat-model approval.
