## Best Possible Solution

Keep the secure default while maintainers decide the remote-HTTP policy; land narrow warning/copy improvements if accepted, and only implement plaintext browser device identity after explicit security approval, docs, tests, and real LAN proof.

Unclear. Warning and copy improvements are the safest narrow mitigation, but enabling persistent device identity over plaintext non-localhost HTTP is only the best fix if maintainers intentionally accept that security model.
