## Summary

Keep open. Current main and the latest release still enforce the HTTPS-or-localhost Control UI device-identity boundary, the open linked PR only clarifies messaging, and the remaining remote plain-HTTP behavior is already security/product-sensitive work tracked by the canonical follow-up.

## What This Changes

The issue asks for Docker/VPS or homelab Control UI access over non-local plain HTTP to connect despite the current secure-context device-identity requirement.

## Best Possible Solution

Keep the secure default, review or replace the copy-only guidance PR, and resolve remote plain-HTTP device identity through `https://github.com/openclaw/openclaw/issues/53274` with explicit threat-model coverage.

## Reproduction Assessment

Yes, source-backed rather than live-run. In a non-secure browser context the UI skips device identity because `crypto.subtle` is absent, and current gateway policy rejects non-local device-less Control UI connects unless trusted-proxy or break-glass bypass applies.

## Solution Assessment

Unclear. Clarifying the current error is a narrow safe improvement, but supporting remote plain-HTTP browser device identity changes a documented security boundary and needs maintainer approval before implementation.
