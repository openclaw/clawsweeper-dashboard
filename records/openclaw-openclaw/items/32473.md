## Summary

Keep this open for an explicit security/product decision: current main intentionally rejects remote plain-HTTP Control UI sessions without device identity, and the linked PR only clarifies that policy rather than providing the requested homelab access mode.

## What This Changes

Allow remote plain-HTTP Control UI access while retaining browser device identity and authentication.

## Best Possible Solution

Keep HTTPS and localhost as the secure defaults, then either approve a narrowly threat-modeled remote-HTTP device-identity design with focused gateway/UI tests and explicit warnings, or formally retain the current policy and track only diagnostics in the narrower linked items.

## Reproduction Assessment

Yes, at source level with high confidence: remote plain HTTP lacks `crypto.subtle`, the UI therefore sends no device identity, and the gateway rejects that remote device-less Control UI connection unless device auth is fully bypassed.

## Solution Assessment

Unclear. Moving the identity fingerprint to a pure-JavaScript hash could remove the browser capability limitation, but it is only the best solution if a security review confirms that remote HTTP does not undermine the intended authentication and interception protections.
