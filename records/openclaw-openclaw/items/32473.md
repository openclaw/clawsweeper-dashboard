## Summary

Keep open: current main and v2026.6.10 still preserve the HTTPS-or-localhost device-identity boundary for remote plain-HTTP Control UI, and the remaining request is a security/product decision rather than implemented cleanup.

## What This Changes

The issue asks for a supported way to use Control UI device authentication from remote Docker/VPS or homelab HTTP URLs that are not HTTPS or localhost.

## Best Possible Solution

Keep the secure default while maintainers decide the plaintext-browser identity policy; land narrow warning/copy improvements separately, and only add a remote HTTP identity mode if it has explicit security approval, docs, tests, and real LAN proof.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction path: non-secure remote HTTP lacks browser WebCrypto/device identity, and the Gateway rejects remote device-less Control UI unless a trust path or break-glass bypass is used.

## Solution Assessment

Unclear as a product/security matter. A simple bypass switch is not the best fix by default; the maintainable end state is either clearer guidance around the current secure boundary or an explicitly approved narrow plaintext-LAN identity design.
