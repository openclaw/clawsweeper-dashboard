## Summary

Keep open. Current main still enforces the HTTPS-or-localhost Control UI device-identity boundary and still emits the reported error for remote plain-HTTP browser sessions without device identity; docs and tests describe that as intentional security policy. Open PR #56830 references this issue with closing syntax but only changes messaging, while #53274 and #71669 track the deeper policy and diagnostics questions.

## What This Changes

Remote Docker/VPS/LAN Control UI users want non-localhost HTTP access to stop failing with `control ui requires device identity (use HTTPS or localhost secure context)`, ideally without disabling device auth entirely.

## Best Possible Solution

Keep the issue open until maintainers decide the product/security boundary. If remote HTTP support is accepted, implement a reviewed device-identity path that does not depend on browser WebCrypto secure-context availability, with gateway/UI tests and docs; otherwise land clearer messaging and route the policy request to #53274.

## Reproduction Assessment

Yes. The source and tests provide a high-confidence reproduction path: remote Control UI, no device identity, `allowInsecureAuth: true`, and `isLocalClient: false` reaches `reject-control-ui-insecure-auth`, while the browser client skips device identity when `crypto.subtle` is unavailable.

## Solution Assessment

Unclear. A broad bypass switch is not the best fix because it weakens the current security boundary; the safer split is #56830 for clearer messaging and #53274 for any reviewed device-identity design change.
