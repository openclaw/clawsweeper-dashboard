## Summary

Keep open. Current main still enforces the documented HTTPS-or-localhost Control UI device-identity boundary, this issue is paired with an open PR using closing syntax, and the broader remote plain-HTTP identity behavior remains an open security/product decision.

## What This Changes

The issue asks OpenClaw to connect the Control UI from Docker/VPS or LAN HTTP without failing device identity on non-localhost insecure contexts.

## Best Possible Solution

Keep the secure default for now, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and decide remote plain-HTTP device-identity support in https://github.com/openclaw/openclaw/issues/53274.

## Reproduction Assessment

Yes, source-backed. A remote plain-HTTP browser skips device identity because `crypto.subtle` is unavailable, then the gateway rejects the non-local Control UI session without device identity.

## Solution Assessment

Unclear. Clarifying the message is a narrow maintainable step, but supporting browser device identity or operator auth over remote plain HTTP changes a documented security boundary and needs maintainer threat-model approval.
