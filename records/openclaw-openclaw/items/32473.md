## Summary

Keep open. Current main still enforces the documented HTTPS-or-localhost Control UI device-identity boundary, an open PR still references this issue with closing syntax for the narrower error-copy fix, and the broader remote plain-HTTP identity behavior is tracked in a canonical open policy issue.

## What This Changes

The issue asks OpenClaw to connect the Control UI from Docker/VPS remote plain HTTP instead of failing with the device-identity secure-context error.

## Best Possible Solution

Keep the secure default for now, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and decide any remote plain-HTTP device-identity support in https://github.com/openclaw/openclaw/issues/53274.

## Reproduction Assessment

Yes, source-backed. A remote plain-HTTP browser lacks `crypto.subtle`, the UI skips device identity, and the gateway rejects the non-local Control UI connect without a device identity.

## Solution Assessment

Unclear. Clarifying the message is a narrow maintainable step, but accepting browser device identity or operator auth over remote plain HTTP changes a documented security boundary and needs maintainer threat-model approval.
