## Summary

Keep open. Current main still enforces the documented HTTPS-or-localhost Control UI device-identity boundary, https://github.com/openclaw/openclaw/pull/56830 is an open closing PR for the narrower messaging problem, and https://github.com/openclaw/openclaw/issues/53274 remains the canonical open policy issue for remote plain-HTTP browser identity support.

## What This Changes

The issue asks for VPS/Docker/NAS-style remote HTTP Control UI access to stop failing with the device-identity secure-context error, or to provide clearer guidance for that failure mode.

## Best Possible Solution

Keep the secure default, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and decide any remote plain-HTTP device-identity support in https://github.com/openclaw/openclaw/issues/53274.

## Reproduction Assessment

Yes, source-backed. A remote plain-HTTP browser lacks crypto.subtle, the UI skips device identity, and the gateway rejects the non-local Control UI connection without device identity.

## Solution Assessment

Unclear. Clarifying the message is a narrow maintainable step, but accepting browser device identity or operator auth over remote plain HTTP changes a documented security boundary and needs maintainer threat-model approval.
