## Summary

Keep open: current main and v2026.6.10 still preserve the secure-context/device-identity boundary for non-localhost HTTP Control UI access, and the remaining request is a security/product decision rather than already-implemented cleanup.

## What This Changes

The issue asks OpenClaw to provide a supported path for Docker/VPS or homelab Control UI access when non-localhost HTTP cannot generate browser device identity.

## Best Possible Solution

Keep the secure default while maintainers decide the plaintext-browser identity policy; land narrow warning/copy improvements, and only add a remote HTTP identity mode if it has explicit security approval, docs, tests, and live LAN proof.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: non-secure remote HTTP lacks crypto.subtle/device identity, and the gateway rejects remote device-less Control UI without the break-glass bypass. I did not run a live Docker or LAN browser repro in this read-only review.

## Solution Assessment

Unclear as a product/security matter. A simple bypass switch is not the best fix by default; the maintainable end state is either clearer guidance around the current secure boundary or an explicitly approved narrow plaintext-LAN identity design.
