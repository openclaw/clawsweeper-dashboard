## Summary

Keep open. Current main and the latest release still skip Control UI browser device identity when WebCrypto is unavailable and still reject remote non-localhost plain-HTTP Control UI sessions without device identity; the remaining behavior needs maintainer security/product review, while related open items own the messaging and startup-warning slices.

## What This Changes

Request to make Control UI Docker/VPS or homelab access work, or be better guided, when non-local plain HTTP cannot satisfy the current HTTPS-or-localhost device-identity requirement.

## Best Possible Solution

Keep the secure default for now, resolve the remote plain-HTTP device-identity policy through https://github.com/openclaw/openclaw/issues/53274 with threat-model coverage, and let the linked messaging and startup-warning items land or close on their own merits.

## Reproduction Assessment

Yes, source-backed rather than live-run. In a non-secure browser context the UI skips device identity because crypto.subtle is absent, and current gateway policy rejects remote device-less Control UI connects unless trusted-proxy, localhost compatibility, or break-glass bypass applies.

## Solution Assessment

Unclear. Replacing the WebCrypto fingerprint with a browser-safe hash is technically plausible, but accepting device identity over remote plain HTTP changes a documented security boundary and needs maintainer approval before implementation.
