## Summary

Keep open. Current main still intentionally rejects remote plain-HTTP Control UI sessions without device identity, the open messaging PR at https://github.com/openclaw/openclaw/pull/56830 still references this issue with closing syntax, and the broader LAN HTTP identity policy remains security/product work tracked by https://github.com/openclaw/openclaw/issues/53274.

## What This Changes

Remote Docker/VPS users want Control UI to connect over non-localhost HTTP without hitting the device-identity secure-context rejection.

## Best Possible Solution

Keep the secure default, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer messaging, and route any remote HTTP device-identity redesign through https://github.com/openclaw/openclaw/issues/53274 with explicit security review.

## Reproduction Assessment

Yes. Source inspection on current main shows remote plain HTTP lacks crypto.subtle so the UI omits device identity, then the gateway returns reject-control-ui-insecure-auth and sends the reported close reason; I did not run a live Docker/browser scenario in this read-only review.

## Solution Assessment

Unclear. Clarifying messaging is a narrow safe improvement, but accepting remote plain HTTP for Control UI device identity would change the documented security model and needs maintainer threat-model approval.
