## Summary

Keep open. Current main still enforces the documented HTTPS-or-localhost Control UI device-identity boundary, and the linked open implementation only clarifies messaging while broader remote plain-HTTP identity support remains a security/product decision tracked in https://github.com/openclaw/openclaw/issues/53274.

## What This Changes

The issue asks for remote Control UI access to work when device identity is blocked by the HTTPS-or-localhost secure-context requirement.

## Best Possible Solution

Keep the secure default, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer user guidance, and decide remote plain-HTTP browser device identity in https://github.com/openclaw/openclaw/issues/53274 with explicit threat-model coverage.

## Reproduction Assessment

Yes, source-backed. A remote plain-HTTP browser lacks `crypto.subtle`, the UI skips device identity, and the gateway rejects the non-local device-less Control UI session.

## Solution Assessment

Unclear. Clarifying the error is narrow and maintainable, but supporting browser device identity or operator auth over remote plain HTTP changes a documented security boundary and needs maintainer approval.
