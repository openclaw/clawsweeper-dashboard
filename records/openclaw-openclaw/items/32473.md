## Summary

Keep open: current main and the latest release intentionally preserve the HTTPS-or-localhost device-identity boundary for remote Control UI HTTP, while the remaining work is a maintainer security/product decision around homelab ergonomics and linked mitigation PRs that are still open.

## What This Changes

Request to make or explain Control UI access from Docker/VPS/NAS hosts when remote plain HTTP fails with the secure-context/device-identity requirement.

## Best Possible Solution

Keep the secure-context/device-identity boundary, then land or reject the linked diagnostics/onboarding work through maintainer security review with HTTPS, Tailscale Serve, trusted-proxy, localhost tunneling, and explicitly warned break-glass as the supported paths.

## Reproduction Assessment

Yes, from source and regression tests: remote Control UI without browser device identity is rejected even when `allowInsecureAuth` is configured, while local insecure compatibility remains allowed. I did not run a live Docker/VPS browser repro in this read-only review.

## Solution Assessment

No, making `allowInsecureAuth` a remote HTTP bypass would not be the best fix because it conflicts with the security fix in https://github.com/openclaw/openclaw/pull/20684. The safer path is maintainer-reviewed diagnostics/onboarding or a clearly scoped secure remote-access mode.
