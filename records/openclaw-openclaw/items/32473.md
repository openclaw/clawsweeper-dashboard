## Summary

Keep open. Current main and the latest release still enforce the documented HTTPS-or-localhost Control UI device-identity boundary, an open PR still references this issue with closing syntax for the narrower messaging fix, and the broader remote-HTTP identity behavior is already tracked as a security/product decision in https://github.com/openclaw/openclaw/issues/53274.

## What This Changes

The issue requests a way for Control UI device identity to work from Docker/VPS or LAN HTTP deployments without requiring HTTPS or localhost.

## Best Possible Solution

Keep the secure default for now, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and decide any remote plain-HTTP device-identity support in https://github.com/openclaw/openclaw/issues/53274.

## Reproduction Assessment

Yes, source-backed. A remote plain-HTTP browser lacks `crypto.subtle`, the UI omits device identity, and the gateway rejects the non-local Control UI connect without a device.

## Solution Assessment

Unclear. Clarifying the message is a narrow maintainable step, but accepting browser device identity or operator auth over remote plain HTTP changes a documented security boundary and needs maintainer threat-model approval.
