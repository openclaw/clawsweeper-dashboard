## Summary

Keep open. Current main still intentionally rejects remote plain-HTTP Control UI sessions that cannot present device identity, while open PR #56830 only clarifies the messaging and the deeper remote-HTTP device-identity policy remains tracked by #53274 and #71669.

## What This Changes

The issue asks OpenClaw to support or better explain Control UI device identity for Docker/VPS or LAN HTTP deployments that are not HTTPS or localhost.

## Best Possible Solution

Keep the secure default until maintainers approve a reviewed remote-HTTP device-identity design; land or close #56830 for copy, and use #53274/#71669 for the policy and warning follow-ups.

## Reproduction Assessment

Yes. Source and tests give a high-confidence path: a non-localhost HTTP Control UI browser without `crypto.subtle` sends no device identity, and the gateway returns `reject-control-ui-insecure-auth` with the reported close reason.

## Solution Assessment

Unclear. A simple bypass switch is not the best fix because it weakens a documented security boundary; the safer direction is clearer messaging plus a maintainer-reviewed design if remote HTTP device identity should be supported.
