## Summary

Keep open. Current main still intentionally rejects non-localhost plain-HTTP Control UI sessions without device identity, and open PR #56830 references this issue with closing syntax while only addressing messaging.

## What This Changes

Users are asking for Docker/VPS or homelab Control UI access over non-localhost HTTP to connect, or at least fail with clearer device-identity guidance.

## Best Possible Solution

Keep the secure default unless maintainers approve a reviewed remote-HTTP device-identity design; review #56830 for copy and use #53274/#71669 for the remaining policy and warning work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: non-localhost HTTP Control UI cannot create browser device identity and the gateway rejects the device-less operator connection with the reported reason.

## Solution Assessment

Unclear. A simple bypass switch is not the best fix because it would weaken a documented security boundary; the maintainable path is clearer messaging plus a reviewed design for remote HTTP device identity if maintainers want to support it.
