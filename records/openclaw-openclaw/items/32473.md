## Summary

Keep open. Current main still implements the secure-context policy that rejects non-localhost plain-HTTP Control UI sessions without device identity, and the open closing PR only clarifies messaging rather than changing that policy. The discussion also has recent affected-user confirmations and narrower related items for the actual design and diagnostics work.

## What This Changes

The issue asks for Docker/VPS or LAN Control UI access to work despite the `control ui requires device identity` HTTPS-or-localhost secure-context requirement.

## Best Possible Solution

Resolve the product direction explicitly: either keep remote plain HTTP rejected and land clearer diagnostics/docs, or design a narrow secure-context-independent device identity flow with threat-model review and regression coverage.
