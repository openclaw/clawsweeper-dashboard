## Summary

Keep open. Current main and v2026.5.12 still enforce the documented HTTPS-or-localhost Control UI device-identity boundary, this issue has an open closing PR, and broader remote plain-HTTP identity support remains a security/product decision tracked at https://github.com/openclaw/openclaw/issues/53274.

## What This Changes

The issue requests a workable Control UI path for Docker/VPS or homelab deployments where non-localhost HTTP currently cannot satisfy browser device identity.

## Best Possible Solution

Keep the secure default, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and decide any remote plain-HTTP device-identity support in https://github.com/openclaw/openclaw/issues/53274 with explicit threat-model coverage.

## Reproduction Assessment

Yes, source-backed. A remote plain-HTTP browser lacks crypto.subtle, the UI skips device identity, and the gateway rejects the non-local device-less Control UI session.

## Solution Assessment

Unclear. Clarifying the error is narrow and maintainable, but supporting browser device identity or operator auth over remote plain HTTP changes a documented security boundary and needs maintainer approval.
