## Summary

Keep open: current main and latest release still enforce the HTTPS/localhost device-identity boundary for remote Control UI, and the linked copy-clarification PR is open, unmerged, and does not change the underlying policy. The deeper behavior request is security/product-sensitive and is also tracked in https://github.com/openclaw/openclaw/issues/53274, so this should not be auto-closed or sent to a new fix lane.

## What This Changes

The issue asks for a better supported path when Docker/VPS/NAS Control UI access over non-local HTTP hits the browser secure-context device-identity requirement.

## Best Possible Solution

Review or replace the linked copy-clarification PR, and keep the broader behavior decision in https://github.com/openclaw/openclaw/issues/53274 until maintainers choose either HTTPS/localhost-only as policy or an explicit plaintext-LAN identity design with security tests and docs.

## Reproduction Assessment

Yes. Source inspection shows the browser skips device identity without `crypto.subtle`, and the gateway rejects remote Control UI sessions without identity unless a trusted-proxy or explicit break-glass path applies; I did not run a live Docker/browser repro in this read-only review.

## Solution Assessment

Unclear. Clarifying copy is a safe mitigation, but supporting non-local plaintext HTTP device authentication is only the best solution if maintainers explicitly accept that security model; otherwise the secure transport requirement should remain.
