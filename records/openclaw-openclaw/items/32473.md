## Summary

Keep open: current main and v2026.6.10 still intentionally reject remote non-localhost HTTP Control UI sessions that cannot present secure-context device identity. The remaining work is a security/product decision across warning UX, setup defaults, and possible plaintext browser identity support, with related open items still under review.

## What This Changes

The issue asks for Docker/VPS or homelab Control UI access to work when remote HTTP cannot complete secure-context device identity.

## Best Possible Solution

Keep the secure default, land or replace narrow warning/setup-default improvements after review, and only support remote plaintext browser identity if maintainers explicitly approve that threat model with docs, tests, and real LAN proof.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.10 still skip browser device identity without secure-context WebCrypto and reject remote device-less Control UI; I did not run a live Docker/LAN browser reproduction in this read-only review.

## Solution Assessment

Unclear until maintainer decision. Warning and copy improvements are the safest narrow mitigation, but enabling persistent device identity or a new bypass for plaintext non-localhost HTTP is only best if maintainers explicitly accept that security model.
