## Summary

Keep open. Current main and v2026.5.12 still enforce the documented HTTPS-or-localhost Control UI device-identity boundary, this issue has an open closing PR for the narrower messaging fix, and broader remote plain-HTTP identity support remains a security/product decision tracked separately.

## What This Changes

Requests that Control UI users on Docker/VPS or homelab remote HTTP be able to connect without hitting the device-identity secure-context error.

## Best Possible Solution

Keep the secure default, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and decide remote plain-HTTP device-identity support in https://github.com/openclaw/openclaw/issues/53274 with explicit threat-model coverage.

## Reproduction Assessment

Yes, source-backed. A remote plain-HTTP browser lacks crypto.subtle, the UI skips device identity, and the gateway rejects the non-local device-less Control UI session.

## Solution Assessment

Unclear. Clarifying the error is narrow and maintainable, but supporting browser device identity or operator auth over remote plain HTTP changes a documented security boundary and needs maintainer approval.
