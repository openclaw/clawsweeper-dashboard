## Summary

Keep open: current main and the latest release still intentionally reject remote plain-HTTP Control UI sessions without device identity, while the remaining request is a security/product decision with open, unmerged follow-up PRs.

## What This Changes

The issue asks for a workable Docker/VPS or homelab path when Control UI device authentication fails over remote plain HTTP because the browser is not on HTTPS or localhost.

## Best Possible Solution

Preserve the secure default, have maintainers rebase/land or replace the focused warning or messaging PR, and only add a remote plain-HTTP bypass or new config surface after explicit security/product approval.

## Reproduction Assessment

Yes at source level: a browser on remote plain HTTP cannot build device identity, and the gateway rejects that device-less Control UI connection unless the local exception or break-glass bypass applies. I did not run a live Docker/VPS browser reproduction in this read-only review.

## Solution Assessment

No for an unconditional remote plain-HTTP bypass: current docs and tests make the secure-context rejection intentional. A narrower warning/messaging improvement is the safer maintainable path unless maintainers explicitly choose a new security mode.
