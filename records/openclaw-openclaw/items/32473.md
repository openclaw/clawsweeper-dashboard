## Summary

Keep open. Current main still intentionally rejects non-localhost plain-HTTP Control UI sessions without browser device identity, and open PR #56830 references this issue with closing syntax while only addressing clearer messaging.

## What This Changes

The issue asks OpenClaw to connect the Control UI from Docker/VPS or LAN HTTP deployments where the browser cannot provide secure-context device identity.

## Best Possible Solution

Keep the secure default, merge or replace the messaging work if useful, and handle any remote-HTTP device-identity support through the narrower design discussion in #53274 plus warning UX in #71669.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: non-localhost HTTP Control UI cannot create browser device identity and the gateway rejects the device-less operator connection with the reported reason.

## Solution Assessment

Unclear. A simple bypass switch is not the best fix because it weakens a documented security boundary; a maintainable solution needs reviewed remote-HTTP device identity semantics or clearer operator warnings.
