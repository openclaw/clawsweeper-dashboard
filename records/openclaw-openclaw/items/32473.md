## Summary

Keep open. Current main and the latest release still skip Control UI device identity when browser WebCrypto is unavailable and still reject remote device-less plain-HTTP Control UI sessions; the open copy-only PR does not implement the requested remote HTTP behavior, and the deeper policy work remains tracked in related open items.

## What This Changes

The issue asks OpenClaw to support or better guide Docker/VPS/NAS Control UI access from non-local plain-HTTP browser URLs without forcing users to disable device auth.

## Best Possible Solution

Keep the secure default for now, land or close the copy-only messaging PR on its own merits, and resolve remote plain-HTTP browser device identity through https://github.com/openclaw/openclaw/issues/53274 with explicit threat-model coverage; https://github.com/openclaw/openclaw/issues/71669 covers the separate startup-warning UX.

## Reproduction Assessment

Yes, source-backed rather than live-run. In a non-secure browser context the UI skips device identity because `crypto.subtle` is absent, and current gateway policy rejects remote device-less Control UI connects unless trusted-proxy or break-glass bypass applies.

## Solution Assessment

Unclear. Replacing the WebCrypto fingerprint with a browser-safe hash is technically plausible, but accepting device identity over remote plain HTTP changes a documented security boundary and needs maintainer approval before implementation.
