## Summary

Keep open. Current main still enforces the documented HTTPS-or-localhost Control UI device-identity boundary; https://github.com/openclaw/openclaw/pull/56830 is an open closing PR for the narrow messaging fix, while broader remote plain-HTTP device identity support remains a security/product decision tracked in https://github.com/openclaw/openclaw/issues/53274.

## What This Changes

Users want the Control UI to connect from remote Docker/VPS/LAN plain-HTTP deployments without the HTTPS-or-localhost device-identity rejection.

## Best Possible Solution

Keep the secure default, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and decide remote plain-HTTP browser device identity in https://github.com/openclaw/openclaw/issues/53274 with explicit threat-model coverage.

## Reproduction Assessment

Yes, source-backed. A remote plain-HTTP browser lacks `crypto.subtle`, the UI skips device identity, and the gateway rejects the non-local device-less Control UI session.

## Solution Assessment

Unclear. The messaging-only PR is a narrow maintainable UX fix, but supporting remote plain-HTTP device identity or an opt-in bypass changes a documented security boundary and needs maintainer approval.
