## Summary

Keep open. Current main and latest v2026.5.12 still enforce the documented HTTPS-or-localhost Control UI device-identity boundary, this issue has an open closing PR, and broader remote plain-HTTP identity support remains a security/product decision tracked at https://github.com/openclaw/openclaw/issues/53274.

## What This Changes

The issue asks OpenClaw to connect the Control UI from Docker/VPS remote HTTP setups that hit the HTTPS-or-localhost device-identity secure-context requirement.

## Best Possible Solution

Keep the secure default, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and decide any remote plain-HTTP device-identity support in https://github.com/openclaw/openclaw/issues/53274.

## Reproduction Assessment

Yes, source-backed. A remote plain-HTTP browser lacks crypto.subtle, the UI skips device identity, and the gateway rejects the non-local device-less Control UI session.

## Solution Assessment

Unclear. Clarifying the error is a narrow maintainable step, but supporting browser device identity or operator auth over remote plain HTTP changes a documented security boundary and needs maintainer threat-model approval.
