## Summary

Keep open. Current main and v2026.5.20 still enforce the documented HTTPS-or-localhost Control UI device-identity boundary; the open linked pull request only clarifies messaging, while the remaining remote plain-HTTP identity behavior is a security/product decision also tracked in https://github.com/openclaw/openclaw/issues/53274.

## What This Changes

The issue asks for Control UI on Docker/VPS or homelab remote HTTP deployments to connect without failing the device-identity secure-context requirement.

## Best Possible Solution

Keep the secure default, review or replace https://github.com/openclaw/openclaw/pull/56830 for clearer guidance, and resolve remote plain-HTTP browser device identity in https://github.com/openclaw/openclaw/issues/53274 with explicit security coverage.

## Reproduction Assessment

Yes, source-backed rather than live-run. A non-local plain-HTTP browser skips device identity when `crypto.subtle` is absent, and the gateway rejects remote device-less Control UI sessions unless a trusted-proxy or break-glass path applies.

## Solution Assessment

Unclear. Clarifying the current error is a narrow safe improvement, but supporting remote plain-HTTP browser device identity changes a documented security boundary and needs maintainer approval before implementation.
