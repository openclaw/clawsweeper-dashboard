## Summary

Keep this issue open because https://github.com/openclaw/openclaw/pull/56830 explicitly closes it and remains open. That PR only clarifies messaging, while https://github.com/openclaw/openclaw/issues/53274 is the stronger canonical design discussion for preserving device identity on LAN HTTP.

## What This Changes

The issue asks OpenClaw to support or better handle remote Control UI access over plain HTTP where browser secure-context device identity is unavailable.

## Best Possible Solution

Review the messaging-only PR independently, and use https://github.com/openclaw/openclaw/issues/53274 for the security and product decision on whether browser device identity can safely work on LAN HTTP without disabling device auth.

## Reproduction Assessment

Yes. Current source and docs give a high-confidence path: a non-loopback plain-HTTP browser lacks the expected device identity, so the gateway rejects it unless device auth is explicitly disabled.

## Solution Assessment

No. Clearer error text is useful but does not solve the requested workflow; the best technical direction is the narrower canonical design review of secure device identity generation outside WebCrypto secure contexts.
