## Summary

Keep open: current main and the latest release still route Slack file uploads through the shared optimizing media loader and expose no Slack-specific raw-upload or optimization opt-out; current PNG/JPEG/WebP preservation only partially reduces the symptom.

## What This Changes

The issue requests a Slack media-upload fidelity control so generated images can be sent without unconditional OpenClaw image optimization or format conversion.

## Best Possible Solution

Choose a maintainer-approved Slack media-fidelity contract, then implement it at the Slack outbound/message-tool boundary with tests for raw delivery, optimized fallback, and docs/config behavior.

## Reproduction Assessment

Yes, at source level: Slack uploads call loadOutboundMediaFromUrl, which uses the optimizing loadWebMedia path by default, and Slack config has no opt-out. I did not run a live Slack upload for this read-only review.

## Solution Assessment

Unclear. A Slack opt-out is plausible, but maintainers should first choose between raw-upload default behavior, a new config option, message-tool enforcement, or a shared media-delivery contract.
