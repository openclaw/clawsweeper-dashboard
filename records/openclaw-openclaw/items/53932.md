## Summary

Keep open: current main and the latest release still route Slack media uploads through the shared image-optimizing media loader with no Slack-owned opt-out. The linked implementation attempt is closed unmerged and inert, and the newer workaround comment is useful context but does not match current source because Slack `upload-file` still calls the same default optimizing loader path.

## What This Changes

Allow Slack file uploads to opt out of shared image optimization so original image bytes, such as PNGs, can be preserved.

## Best Possible Solution

Add a default-compatible Slack config option, thread it through account resolution and the existing outbound-media facade, and preserve current size, local-read, and SSRF guardrails with focused tests and docs.

## Reproduction Assessment

Yes, at source level. Slack `mediaUrl` and `upload-file` sends reach `uploadSlackFile`, which calls `loadOutboundMediaFromUrl` without any way to pass `optimizeImages: false`, while the shared loader defaults image optimization on.

## Solution Assessment

Yes in direction. A Slack-owned opt-out threaded through the existing media loader is narrower and safer than bypassing the loader or forcing broad agent tool-routing behavior, because it keeps existing size, local-read, and SSRF protections intact.
