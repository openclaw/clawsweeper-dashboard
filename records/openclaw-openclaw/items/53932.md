## Summary

Keep open: current main and v2026.5.18 still route Slack media uploads through the shared image-optimizing loader with no Slack config or plugin-SDK option to disable optimization. The newer workaround discussion is useful, but source inspection shows `message(action="upload-file")` still reaches the same Slack send/upload path, and the only related Slack opt-out PR was closed unmerged with inert code.

## What This Changes

Request to let Slack uploads preserve original image bytes or opt out of the shared image optimization that can compress or convert PNG uploads.

## Best Possible Solution

Add a default-compatible Slack config option that flows through account resolution, the plugin-SDK outbound-media facade, and the shared loader, while preserving existing size, local-read, and SSRF guardrails.

## Reproduction Assessment

Yes, at source level. Slack `mediaUrl` and `uploadFile` sends flow through `loadOutboundMediaFromUrl`, while current main exposes no Slack or SDK option to pass `optimizeImages: false`.

## Solution Assessment

Yes. A Slack-owned opt-out threaded through the existing outbound-media loader is the narrow maintainable path; bypassing the loader or relying on prompt/tool choice would risk losing existing media guardrails.
