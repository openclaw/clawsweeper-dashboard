- `impact:data-loss`: The reported behavior loses image fidelity and format information during Slack upload where users expect the original file to be shared.

## Summary

Keep open: current main and the latest release still route Slack file uploads through the shared outbound-media loader without any Slack-owned way to disable image optimization. The later `upload-file` workaround comment is useful context, but source inspection shows that explicit Slack `upload-file` still maps into the same optimized upload path, and the related PR was closed unmerged as an inert config stub.

## What This Changes

Request a Slack-owned option or delivery mode to disable image optimization so uploaded images can preserve original fidelity.

## Best Possible Solution

Add a default-compatible Slack preserve-original upload setting and thread it through the plugin SDK outbound-media facade into the existing media loader, with tests for both the optimized default and the preserve-original Slack path.

## Reproduction Assessment

Yes, at source level. Slack `sendMessage` and `upload-file` flows reach `uploadSlackFile`, which calls the shared outbound-media loader without a path to set `optimizeImages: false`.

## Solution Assessment

Yes, the narrow maintainable direction is a Slack-owned preserve-original option wired through the existing media loader. Bypassing the loader would risk dropping current size, local-file-read, and SSRF protections.
