- `impact:data-loss`: The reported behavior can lose image fidelity and format information when Slack recipients expect the original generated file.

## Summary

Keep open: current main and v2026.5.18 still route Slack media uploads through the shared outbound-media loader without any Slack-owned way to disable image optimization. The reporter's later `upload-file` workaround is useful context, but source inspection shows Slack `upload-file` also reaches the same upload path, and the linked implementation PR was closed unmerged as an inert stub.

## What This Changes

Request a Slack upload option or path that preserves original image files instead of always sending them through image optimization.

## Best Possible Solution

Introduce a default-compatible Slack preserve-original upload option, expose it in config/schema/docs, and forward `optimizeImages: false` through the plugin SDK outbound-media facade into the existing media loader.

## Reproduction Assessment

Yes, at source level. Slack `sendMessage` and `upload-file` flows reach `uploadSlackFile`, which calls the shared outbound-media loader without any path to set `optimizeImages: false`.

## Solution Assessment

Unclear until maintainers choose the public config semantics, but the implementation boundary is clear: Slack config should drive the existing media loader rather than bypassing it.
