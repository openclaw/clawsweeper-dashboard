## Summary

Keep open: current main still forwards Slack read bounds without timestamp normalization or format validation, but accepting ISO 8601 strings is not clearly part of the current documented message-tool contract.

## What This Changes

The issue requests that message-tool read time filters such as `after`/`oldest` accept ISO 8601 timestamp strings or fail clearly instead of returning misleading empty results.

## Best Possible Solution

Define the message-tool read timestamp contract, then add a small shared parser that preserves Slack Unix timestamp strings and either converts ISO 8601 bounds or rejects unsupported formats before calling Slack.

## Reproduction Assessment

Yes for the source path: current main forwards ISO strings directly into Slack `oldest`/`latest` without conversion or validation. No live Slack workspace reproduction was performed.

## Solution Assessment

Unclear; ISO 8601 acceptance would improve the tool UX, but Slack's upstream contract is Unix timestamp strings, so the maintainable fix should start by clarifying the OpenClaw-level contract.
