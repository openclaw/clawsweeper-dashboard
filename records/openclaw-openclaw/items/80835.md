## Summary

Keep open: current main and the latest shipped release still pass Slack message read `after`/`before` strings straight through to Slack `oldest`/`latest`, so ISO 8601 inputs are not normalized or rejected before the API call.

## What This Changes

The issue asks for Slack message-tool `after`/`oldest` timestamp filters to handle ISO 8601 strings correctly instead of silently returning empty or misleading read results.

## Best Possible Solution

Normalize Slack message read `before`/`after` values at the Slack boundary to valid Slack timestamp strings, accept ISO 8601 inputs, and throw a clear error for unrecognized formats before calling Slack.

## Reproduction Assessment

Yes, from source inspection: an ISO 8601 `after` string flows from the message tool into Slack `oldest` without parsing, while Slack expects Unix timestamp-style boundaries. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to parse and validate Slack read boundary timestamps in the Slack message-read path, with focused tests for ISO strings, numeric Slack timestamps, and invalid input. Client-side filtering or a config flag would leave the misleading API call behavior in place.
