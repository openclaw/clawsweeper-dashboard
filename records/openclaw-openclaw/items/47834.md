## Summary

Keep open. Current main still does not expose the current browser tab URL to `before_tool_call` before browser `act`/`type` execution; the reporter's `openclaw-credential-vault v1.0.0-beta.5` workaround is useful and fail-secure, but it confirms a remaining plugin API gap rather than closing the core request.

## What This Changes

Request an additive browser/plugin hook capability that supplies the current browser page URL, or an equivalent query seam, before browser `act`/`type` actions for domain-pinned credential injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests for blocked/private URLs and managed plus existing-session profiles.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a browser `act` call reaches `before_tool_call` with the original params before browser execution, while active-tab URL resolution happens later in the browser route response path.

## Solution Assessment

No for copying the raw `_currentUrl` sketch into params as-is. The maintainable fix should expose a typed, policy-safe current URL or query seam that reuses the existing safe route URL policy.
