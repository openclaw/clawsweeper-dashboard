## Summary

Keep open: current main and the latest release still do not expose a policy-safe active browser URL to `before_tool_call` handlers before `browser` `act` or `type` runs, and the reporter's credential-vault workaround confirms this remains a security-sensitive plugin API gap rather than stale cleanup.

## What This Changes

Request to provide browser `before_tool_call` hooks with the current page URL, or an equivalent query seam, so credential plugins can enforce domain pinning before password/text injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser security hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests for blocked URLs and supported browser profiles.

## Reproduction Assessment

Yes, at source level. Current main sends the pending browser `act`/`type` params into `before_tool_call` before browser route code resolves and returns the active tab URL.

## Solution Assessment

Unclear as an exact implementation. The requested capability is valid, but maintainers should choose whether the permanent contract is hook param enrichment, a typed hook context/query API, or another browser-owned seam that preserves URL policy and plugin compatibility.
