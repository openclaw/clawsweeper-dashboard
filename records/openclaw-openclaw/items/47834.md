## Summary

Keep open. Current main and the latest release still do not provide a policy-safe current browser tab URL to `before_tool_call` hooks before browser `act`/`type` runs, and the reporter’s credential-vault workaround confirms a real security-sensitive plugin API gap rather than a resolved core behavior.

## What This Changes

Request to expose a policy-safe current browser page URL to `before_tool_call` hooks before browser `act`/`type` actions so credential plugins can domain-pin secret injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and regression tests for blocked/private URLs and supported browser profiles.

## Reproduction Assessment

Yes, at source level. The generic adapter calls the hook before browser execution, the hook runner forwards only existing params, and the browser route resolves policy-filtered URLs later in the response path.

## Solution Assessment

No for copying the raw `_currentUrl` sketch into params as-is. The maintainable path is a typed, additive, policy-safe hook-time URL contract or query seam that preserves existing hook ordering and URL filtering semantics.
