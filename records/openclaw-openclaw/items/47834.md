## Summary

Keep open. Current main still does not expose the current browser tab URL to `before_tool_call` before browser `act`/`type` execution; browser route code can add a policy-checked `url` only to post-execution responses. The credential-vault workaround in `openclaw-credential-vault v1.0.0-beta.5` is useful and fail-secure, but it confirms a remaining plugin API gap.

## What This Changes

Request to expose a policy-safe current browser page URL, or equivalent query seam, to `before_tool_call` hooks for browser `act`/`type` actions.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests for blocked/private URLs and managed plus existing-session profiles.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. A browser `act` call reaches `before_tool_call` with the original params before browser execution, while active-tab URL resolution happens later in the browser route response path.

## Solution Assessment

No for copying the raw `_currentUrl` sketch into params as-is. The maintainable fix should expose a typed, policy-safe current URL or query seam that reuses the existing safe route URL policy.
