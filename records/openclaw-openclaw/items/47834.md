## Summary

Keep open. Current main still does not provide browser `act`/`type` calls with hook-time current-page URL context, and the reporter's `openclaw-credential-vault` workaround confirms the remaining plugin API gap rather than a resolved core behavior. The broader credential-firewall discussion at https://github.com/openclaw/openclaw/issues/18245 is related, but it does not supersede this narrower hook contract question.

## What This Changes

Request to expose a policy-safe current browser page URL, or an equivalent query seam, to `before_tool_call` hooks for browser `act`/`type` actions so credential plugins can enforce domain pins before injection.

## Best Possible Solution

Add a narrow, additive browser/plugin hook contract that gives pre-action policy hooks a policy-safe active-tab URL for browser `act`/`type`, or an equivalent supported query seam, with docs and regression coverage for private/blocked URL handling.

## Reproduction Assessment

Yes, at source level. The adapter calls `runBeforeToolCallHook` before browser execution, the hook runner forwards only existing params, and the browser route resolves a safe current URL only later in the response path.

## Solution Assessment

No for copying the raw `_currentUrl` sketch into params as-is. The maintainable solution is a typed, additive, policy-safe hook-time URL contract or query seam that preserves existing hook ordering and URL filtering semantics.
