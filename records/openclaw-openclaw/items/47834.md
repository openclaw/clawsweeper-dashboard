## Summary

Keep open. Current main still does not give `before_tool_call` hooks a policy-safe current browser URL for `browser` `act`/`type` before execution, and the reporter's credential-vault workaround confirms a real plugin API gap rather than a resolved core behavior.

## What This Changes

The issue requests hook-time access to the browser's current page URL for `before_tool_call` handling of browser `act`/`type` actions so security plugins can domain-check credential injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that provides policy-safe current tab URL context for pre-action browser hooks, or an equivalent supported query seam, with docs and regression tests for blocked/private URLs and supported browser profiles.

## Reproduction Assessment

Yes, at source level. Current main forwards only supplied or policy-adjusted tool params into `before_tool_call` before browser execution, while current browser URL resolution happens later inside route execution and response helpers.

## Solution Assessment

No for copying the raw `_currentUrl` sketch into params as-is. The maintainable solution is a typed, additive, policy-safe hook-time URL contract or query seam that preserves existing hook ordering and browser URL filtering semantics.
