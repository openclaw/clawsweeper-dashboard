## Summary

Keep open. Current main still does not expose the browser tab URL to `before_tool_call` hooks before browser `act`/`type` execution, and the reporter's credential-vault workaround confirms this remains a real security contract gap rather than obsolete cleanup.

## What This Changes

Allow browser `before_tool_call` hooks for `act`/`type` actions to receive or query the current page URL so credential plugins can enforce domain pins before text injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests proving blocked/private URLs are omitted consistently across supported browser profiles.

## Reproduction Assessment

Yes. The source path is enough to reproduce the contract gap: `before_tool_call` receives the original browser act params before `tool.execute`, while active-tab URL resolution happens later inside the browser route and response construction.

## Solution Assessment

No for the issue's raw `_currentUrl` sketch as-is. The maintainable solution should expose a policy-safe current URL through a typed hook context or explicit query seam rather than copying arbitrary tab URLs into every hook params object.
