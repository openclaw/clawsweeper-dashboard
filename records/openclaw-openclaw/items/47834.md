## Summary

Keep open. Current main still passes `before_tool_call` only the prepared browser tool params before browser `act` execution, while the policy-checked current tab URL is resolved inside the browser route and returned after execution; the credential-vault workaround remains an external fail-secure mitigation, not a core plugin contract.

## What This Changes

Request adds a policy-safe current browser page URL, or equivalent query seam, to browser `act`/`type` `before_tool_call` hook data for domain-pinned credential injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action hooks a policy-safe current tab URL for browser `act`/`type`, or an equivalent supported query seam, with tests and docs for blocked/private URLs and profile variants.

## Reproduction Assessment

Yes, at source level. Current main passes browser `act`/`type` params into `before_tool_call` before browser route execution, and the policy-safe active URL is resolved only inside or after the browser route.

## Solution Assessment

Unclear as an exact implementation. The requested capability is valid, but maintainers should choose whether the permanent contract is a hook param, typed hook context/query API, or another browser-owned seam that preserves URL policy and plugin compatibility.
