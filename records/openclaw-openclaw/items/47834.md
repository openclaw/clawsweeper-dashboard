## Summary

Keep open. Current main and the latest release still do not give `before_tool_call` handlers a policy-safe active browser URL before `browser` `act`/`type` executes, and the reporter's credential-vault workaround confirms this is still a security-sensitive plugin API gap rather than stale cleanup.

## What This Changes

Request to expose the current browser page URL, or an equivalent query seam, to plugin `before_tool_call` handlers for browser `act`/`type` actions.

## Best Possible Solution

Add a narrow additive browser/plugin contract that gives pre-action browser security hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with tests and docs for blocked URLs and profile variants.

## Reproduction Assessment

Yes, at source level. Current main sends prepared browser `act`/`type` params into `before_tool_call` before the browser route resolves or returns the active tab URL.

## Solution Assessment

Unclear as an exact implementation. The requested capability is valid, but maintainers should choose whether the permanent contract is a hook param, typed hook context/query API, or another browser-owned seam that preserves URL policy and plugin compatibility.
