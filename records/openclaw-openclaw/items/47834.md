## Summary

Keep open. Current main and the latest release still run `before_tool_call` with the original browser `act`/`type` params before browser execution, while safe current-tab URL resolution exists only inside the browser route response path after the hook; the reporter's credential-vault workaround is useful but confirms the remaining plugin API gap.

## What This Changes

Request a policy-safe current browser tab URL, or an equivalent query seam, for `before_tool_call` handling of browser `act`/`type` actions so credential plugins can enforce domain pins before injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests covering blocked/private URLs and supported browser profiles.

## Reproduction Assessment

Yes at source level. Current main passes the original browser `act`/`type` params into `before_tool_call` before browser execution, and the only safe current-tab URL helper is used in the later browser route response path.

## Solution Assessment

Unclear as an exact implementation. The requested capability is valid, but maintainers should choose whether the best permanent seam is an additive hook param, a typed hook context/query API, or another browser-owned contract that preserves URL policy and plugin compatibility.

## Risks / Open Questions

- The API shape is still a maintainer decision: enriching `event.params` versus adding a hook-time query/context seam changes the plugin hook contract and rewrite semantics.
- Any exposed URL must be policy-checked consistently across managed browser, node proxy, and existing-session profiles so plugins do not receive private or local browser targets by accident.
- The current external workaround is fail-secure but in-memory, so gateway restarts or cold caches can block legitimate credential injection until a new URL source is observed.
