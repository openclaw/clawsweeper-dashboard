## Summary

Keep open. Current main still routes unowned CLI roots through managed proxy startup and lazy plugin CLI loading, and the issue now has an open closing PR (#76379) that should be reviewed or landed before this issue closes.

## What This Changes

Unknown CLI roots such as `openclaw foo` should fail fast without provider/runtime plugin loading, surviving hot child processes, or avoidable gateway catalog pressure.

## Best Possible Solution

Review and land #76379 or an equivalent narrow fix that gates unowned CLI roots before proxy/plugin runtime loading, then close this issue after the merge verifies `openclaw foo` exits cleanly.

## Reproduction Assessment

Yes, at source level. The reporter supplied process-group repro logs for `openclaw foo`, and current main still contains the proxy and broad plugin-registration path for unowned command roots; I did not run a live CLI smoke in this read-only review.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix: validate builtin or manifest/descriptor command ownership before managed proxy startup and plugin runtime loading, while leaving the separate `models.list` mitigation in place.
