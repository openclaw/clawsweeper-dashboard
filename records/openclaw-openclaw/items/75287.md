## Summary

Keep open. Current main still lets arbitrary non-builtin CLI roots enter managed proxy startup and lazy plugin CLI registration; the merged models-list mitigation in #75326 helps one gateway hot path but does not fix the invalid-command lifecycle bug.

## What This Changes

Unknown or unavailable CLI commands should fail before provider/runtime plugin loading and exit without leaving hot child processes, while gateway log/model/session surfaces avoid repeated catalog pressure.

## Best Possible Solution

Add a manifest/descriptor-backed CLI root ownership gate before managed proxy startup and runtime plugin loading, with regression tests proving unowned roots fail promptly without plugin loads or surviving child processes.

## Reproduction Assessment

Yes. The reporter supplied high-confidence process-group repros for `openclaw foo`, and current-main source/tests still deterministically route unknown non-reserved roots into proxy/plugin-registration paths. I did not execute the live repro because this review was required to stay read-only.

## Solution Assessment

Not applicable as a patch. The narrow maintainable fix is to validate root ownership from builtins plus plugin manifest/descriptor metadata before starting proxy or runtime plugin CLI registration; adding more catalog timeouts would not fix the invalid-command lifecycle.

## Repair Work Prompt

Fix the invalid CLI root lifecycle bug tracked by https://github.com/openclaw/openclaw/issues/75287. On current main, an unowned root such as `openclaw foo` can start managed proxy routing and lazy plugin CLI registration, then fall back to broad runtime plugin loading before failing. Add a pre-runtime ownership gate so only builtin roots and manifest/descriptor-owned plugin CLI roots can start plugin CLI registration or managed proxy startup. Unknown/unowned roots should fail quickly with a normal unknown/unavailable-command error, without provider/runtime plugin load bursts and without leaving child processes. Keep the fix generic: do not hard-code bundled plugin ids, do not move owner-specific behavior into core, and do not broaden gateway catalog timeout work in this PR. Update the focused CLI/plugin tests and add a user-facing changelog fix entry if behavior changes are user-visible.
