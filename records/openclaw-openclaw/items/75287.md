## Summary

Keep open. Current main still allows an unowned CLI root such as `foo` to enter lazy plugin CLI registration and default proxy startup before normal command failure, so the central invalid-command lifecycle bug remains even though related gateway catalog paths now have partial mitigations.

## What This Changes

The issue asks OpenClaw to reject unknown CLI roots before provider/runtime plugin loading and to prevent invalid commands from leaving hot child processes while reducing gateway catalog pressure.

## Best Possible Solution

Add a pre-runtime CLI ownership gate so only builtin roots and manifest/descriptor-owned plugin CLI roots can start proxy/plugin CLI registration; unowned roots should fail promptly without provider/runtime plugin loading or surviving child processes.

## Reproduction Assessment

Yes. The reporter supplied high-confidence process-group repros for `openclaw foo`, and current-main source inspection still shows deterministic proxy/plugin-registration routing for non-built-in, non-owned roots. I did not execute the live repro because this review was required to stay read-only.

## Solution Assessment

Not applicable as a patch. The narrow maintainable solution is a manifest/descriptor-backed CLI root ownership check before managed proxy startup and runtime plugin loading, not another timeout around gateway catalog discovery.
