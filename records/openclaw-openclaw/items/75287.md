## Summary

Keep open. Current main still lets an arbitrary non-builtin primary command reach lazy plugin CLI registration, and when activation planning finds no matching plugin owner the CLI registry loader still falls back to broad runtime plugin loading. The merged gateway-side work for slow model catalog responses only covers part of the reported gateway symptom, not the invalid-command fail-fast and clean-exit problem.

## What This Changes

The issue requests that unknown or unavailable CLI commands fail before provider/runtime plugin loading, exit promptly, and avoid leaving a hot child process while keeping gateway control-plane requests responsive.

## Best Possible Solution

Keep the issue focused on the invalid-command startup boundary: classify unknown and unavailable primary roots from core descriptors plus lightweight plugin manifest/metadata, avoid full runtime plugin loading when there is no matching plugin-owned CLI surface, and ensure any CLI startup helper process or proxy state is cleaned up on startup errors. Treat the already-merged `models.list` timeout work as a related mitigation, not as closure for this report.

## Reproduction Assessment

Yes. The reporter supplied a high-confidence `openclaw foo` process-group reproduction, and current-main source inspection still shows the deterministic path that can load broad plugin CLI runtime for arbitrary non-builtin primary roots.

## Solution Assessment

Not applicable as a patch. The narrow maintainable fix is a lightweight unknown-command/plugin-command ownership gate before runtime CLI registration, with targeted cleanup coverage for startup-error child/process lifecycle; a broad provider-catalog redesign should not be required for the first fix.
