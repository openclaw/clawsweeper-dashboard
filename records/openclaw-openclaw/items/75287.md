## Summary

Keep open. Current main still routes arbitrary unknown CLI roots through managed proxy startup and lazy plugin CLI registration before normal command failure, while the merged gateway/model-list work only mitigates one related saturation surface.

## What This Changes

The issue asks for unknown or unavailable CLI commands to fail promptly without provider/runtime plugin loading, surviving hot child processes, or gateway hot-path catalog saturation.

## Best Possible Solution

Add a pre-runtime CLI ownership gate so only builtin roots and manifest/descriptor-owned plugin CLI roots can start proxy/plugin CLI registration; unowned roots should fail promptly without provider/runtime plugin loading or surviving child processes.

## Reproduction Assessment

Yes. The reporter supplied process-group reproductions for `openclaw foo`, and current-main source plus tests still show the same deterministic proxy and broad plugin-registration route for non-built-in, non-owned roots.

## Solution Assessment

Not applicable as a patch. The narrow maintainable fix is a manifest/descriptor-backed CLI root ownership check before managed proxy startup and runtime plugin loading, while leaving the existing `models.list` timeout mitigation intact.
