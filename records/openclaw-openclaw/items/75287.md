## Summary

Keep open. Current main has partial gateway-side mitigations for slow model/session surfaces, but the central invalid CLI command path still reaches plugin CLI registration and can fall back to broad runtime plugin loading for arbitrary non-builtin roots before Commander reports an unknown or unavailable command.

## What This Changes

The issue reports that a non-existent CLI command loads provider/runtime plugins, leaves a hot child process alive, and coincides with gateway event-loop saturation from repeated catalog-heavy control-plane calls.

## Best Possible Solution

Keep this open until unknown or unavailable primary CLI roots are classified from core command metadata plus lightweight plugin manifest/alias ownership before any runtime plugin loading. An unowned root such as `foo` should skip broad plugin CLI registration, avoid unnecessary proxy/runtime startup, print a normal unknown/unavailable-command error, exit promptly, and leave no child process behind; gateway catalog-pressure fixes should remain separate unless the CLI smoke exposes a shared lifecycle bug.

## Reproduction Assessment

Yes. The reporter supplied a high-confidence process-group reproduction for `openclaw foo`, and current-main source inspection still shows the deterministic unscoped plugin CLI loading path for arbitrary non-builtin primary roots. I did not run the live CLI repro because this review is read-only and the command can load local plugins or leave processes behind.

## Solution Assessment

Not applicable as a patch. The narrow maintainable solution is a pre-runtime CLI ownership gate backed by manifest/descriptor metadata, with cleanup coverage for the proxy/runtime lifecycle, rather than another broad provider-catalog or gateway-session mitigation.
