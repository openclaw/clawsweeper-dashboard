## Summary

Keep open. Current main still allows arbitrary non-builtin CLI roots to enter lazy plugin CLI registration and broad runtime plugin loading before the command is rejected; the merged gateway mitigations for `models.list` and `sessions.list` address adjacent saturation symptoms but not the reported `openclaw foo` invalid-command lifecycle bug.

## What This Changes

The issue requests that unknown or unavailable CLI commands fail before provider/runtime plugin loading, exit promptly, and avoid leaving hot child processes, while related gateway catalog-heavy polling should not saturate the event loop.

## Best Possible Solution

Add a lightweight pre-runtime ownership gate for primary CLI roots using core command metadata plus manifest/alias ownership. Unknown unowned roots such as `foo` should skip broad plugin CLI registration and unnecessary proxy/runtime startup, print a normal unknown/unavailable-command error, exit promptly, and preserve legitimate manifest-owned plugin command activation and existing `plugins.allow` diagnostics.

## Reproduction Assessment

Yes. The reporter provided a high-confidence process-group reproduction for `openclaw foo`, and current-main source inspection still shows the deterministic unscoped plugin-registration path for arbitrary non-builtin roots.

## Solution Assessment

Not applicable as a patch. The narrow maintainable direction is a pre-runtime CLI ownership gate with lifecycle cleanup coverage, while the merged gateway catalog/session mitigations should remain separate unless a CLI smoke exposes a shared lifecycle bug.
