## Summary

Keep open. Current main still routes arbitrary unknown CLI roots through managed proxy startup and lazy plugin CLI registration, and the merged models-list work only mitigates one related gateway catalog surface.

## What This Changes

The issue requests that unknown or unavailable CLI commands fail quickly without provider/runtime plugin loading, surviving hot child processes, or gateway event-loop saturation from repeated catalog loads.

## Best Possible Solution

Add a pre-runtime CLI ownership gate so only builtin roots and manifest/descriptor-owned plugin CLI roots can start proxy/plugin CLI registration; unowned roots should fail promptly without provider/runtime plugin loading or lingering child processes.

## Reproduction Assessment

Yes. The reporter supplied process-group reproductions for `openclaw foo`, and current-main source plus tests still show unknown roots taking the proxy and broad plugin-registration path.

## Solution Assessment

Not applicable as a patch. The narrow maintainable fix is a manifest/descriptor-backed CLI root ownership check before managed proxy startup and runtime plugin loading, while keeping the existing models-list timeout mitigation.

## Repair Work Prompt

Fix the unknown CLI root lifecycle bug from https://github.com/openclaw/openclaw/issues/75287. Add a pre-runtime ownership gate so unowned roots such as `openclaw foo` fail promptly without managed proxy startup, runtime/provider plugin loading, or lingering child processes. Preserve builtin roots and plugin CLI roots that are owned by manifest/descriptor metadata; do not solve this by loading all plugin runtimes. Likely implementation is in CLI command-path/proxy policy, plugin command registration policy, and the plugin CLI registry loader. Add regression coverage for an unowned root that asserts no proxy startup and no plugin registration/runtime load, and keep existing owned plugin CLI behavior working. Update CHANGELOG.md because this is user-facing. Do not change provider catalog semantics or the existing #75326 models-list timeout behavior.
