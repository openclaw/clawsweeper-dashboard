## Summary

Keep open. Current main has partial gateway-side mitigation for slow `models.list` and targeted provider hook caching, but the reported invalid-command path is still present: arbitrary non-builtin primary commands still reach lazy plugin CLI registration, and the plugin CLI loader still falls back to broad runtime plugin loading when manifest activation finds no owner. That means the central `openclaw foo` fail-fast and clean-exit bug is not yet implemented away.

## What This Changes

The issue asks for unknown or unavailable CLI commands to fail fast without provider/runtime plugin loading or leftover hot child processes, and for gateway log/model/session/control polling to avoid repeated provider plugin reloads and event-loop saturation.

## Best Possible Solution

Keep this issue open until the remaining invalid-command behavior is fixed: unknown or unavailable primary CLI roots should be classified from core descriptors plus lightweight plugin manifest/metadata before runtime plugin loading, exit promptly with a normal unknown/unavailable-command error, and clean up any proxy/runtime child process. The broader gateway catalog pressure should stay aligned with #75297 and the already-merged #75326 work.

## Reproduction Assessment

Yes. The reporter supplied a high-confidence process-group reproduction for `openclaw foo`, and current-main source inspection still shows the deterministic path that can load plugin CLI runtimes for arbitrary non-builtin primary roots; the repair PR should add a focused test plus a live process smoke.

## Solution Assessment

Not applicable as a patch. The narrow maintainable fix is to add a lightweight unknown-command/plugin-command ownership gate before full runtime CLI registration, while preserving legitimate plugin CLI activation and existing allowlist/disabled diagnostics.
