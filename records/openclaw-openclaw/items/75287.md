## Summary

Keep open. Current main has partial gateway-side mitigations for slow model/session catalog discovery, but the central invalid CLI lifecycle bug is still present: arbitrary non-builtin roots can still enter plugin CLI registration and managed proxy startup before Commander reports the command error.

## What This Changes

Make unknown or unavailable CLI commands fail before broad plugin/provider loading, exit cleanly without leaving a hot child process, and reduce gateway catalog-load pressure on model/session/channel surfaces.

## Best Possible Solution

Add a pre-runtime CLI ownership gate that classifies primary roots from core command metadata plus lightweight plugin manifest/alias descriptors before managed proxy startup or runtime plugin loading. Unowned roots such as `foo` should skip broad plugin CLI registration, print a normal unknown or unavailable-command error, exit promptly, and leave no child process behind; existing gateway catalog timeouts should remain as separate hot-path protection.

## Reproduction Assessment

Yes. The reporter supplied a high-confidence process-group reproduction for `openclaw foo`, and current-main source inspection still shows deterministic unscoped plugin CLI loading and default proxy routing for arbitrary non-builtin primary roots.

## Solution Assessment

Not applicable as a patch. The narrow maintainable fix is to gate CLI root ownership before runtime/plugin/proxy startup, rather than adding another gateway catalog timeout or broad provider-loading mitigation.
