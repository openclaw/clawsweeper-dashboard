## Summary

Keep open. Current main has partial gateway-side mitigations for slow catalog/session surfaces, but the central invalid-command path remains: a non-builtin root can still start proxy/bootstrap work, skip route-first handling, enter lazy plugin CLI registration, and fall back to broad runtime plugin loading when manifest activation finds no owner.

## What This Changes

The issue requests that unknown or unavailable CLI commands fail fast without provider/runtime plugin loading, without leaving a hot child process, and that related gateway hot paths avoid repeated provider catalog load bursts.

## Best Possible Solution

Keep this issue open until unknown or unavailable primary CLI roots are classified from core command metadata plus lightweight plugin manifest ownership before any proxy/provider/runtime plugin loading, then exit promptly with the existing useful unknown/unavailable-command diagnostics and reliable cleanup of any started child process.

## Reproduction Assessment

Yes. The reporter supplied a high-confidence process-group reproduction for `openclaw foo`, and current-main source inspection still shows the deterministic path that can broad-load plugin CLI runtimes for arbitrary non-builtin roots.

## Solution Assessment

Not applicable as a patch. The narrow maintainable direction is a lightweight unknown-command/plugin-command ownership gate before broad runtime CLI registration, plus explicit cleanup proof for proxy/runtime child handles.
