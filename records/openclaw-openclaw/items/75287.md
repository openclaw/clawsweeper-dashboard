## Summary

Keep open. Current main has partial gateway-side mitigations for slow `models.list` and `sessions.list`, but the central invalid CLI path still treats arbitrary non-builtin roots as possible plugin commands, can start the default proxy path, and then performs lazy plugin CLI registration before Commander reports the unknown/unavailable command.

## What This Changes

The issue requests that invalid CLI commands such as `openclaw foo` fail fast without provider/runtime plugin loading, without leaving a hot child process, and that related gateway/control-plane surfaces avoid repeated provider catalog pressure.

## Best Possible Solution

Keep this open until unknown or unavailable primary CLI roots are classified from core command metadata plus lightweight plugin manifest/alias ownership before any proxy startup or runtime plugin loading. An unowned root such as `foo` should skip broad plugin CLI registration, avoid unnecessary proxy/runtime startup, print a normal unknown/unavailable-command error, exit promptly, and leave no child process behind; gateway catalog-pressure fixes should remain separate unless the CLI lifecycle smoke exposes a shared bug.
