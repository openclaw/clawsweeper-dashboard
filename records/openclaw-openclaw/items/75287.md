## Summary

Keep open. Current `main` still has the invalid-root CLI path entering lazy plugin CLI registration and default proxy startup for non-builtin roots, while the merged gateway work only partially mitigates related `models.list`/`sessions.list` pressure.

## What This Changes

The issue reports that unknown CLI roots such as `openclaw foo` load provider/runtime plugins, print the unavailable-command error late, and can leave a hot child process while related gateway catalog paths saturate the event loop.

## Best Possible Solution

Reject unowned primary CLI roots before proxy startup and broad plugin CLI/runtime registration, using the core command catalog plus lightweight manifest/alias ownership metadata. Known plugin-owned roots should still lazy-load only their owning plugin, while an unowned root should print a normal unknown/unavailable-command error, exit promptly, and leave no child process behind.

## Reproduction Assessment

Yes. The reporter supplied a high-confidence process-group reproduction for `openclaw foo`, and current source inspection still shows deterministic routing from arbitrary non-builtin roots into proxy startup and broad plugin CLI registration.

## Solution Assessment

Not applicable as a patch. The narrow maintainable fix is an early lightweight CLI ownership gate plus lifecycle cleanup coverage, not another gateway catalog timeout or broad plugin-loading workaround.
