## Summary

Keep open. Current main still lets an unowned CLI root enter managed proxy startup and lazy plugin CLI registration before normal command failure, while the related gateway model-list work only mitigates part of the broader saturation report.

## What This Changes

The issue asks for unknown or unavailable CLI commands to fail before provider/runtime plugin loading, leave no hot child process, and avoid repeated gateway catalog work on control-plane paths.

## Best Possible Solution

Add a pre-runtime CLI ownership gate so only builtin roots and manifest/descriptor-owned plugin CLI roots can start proxy/plugin CLI registration; unowned roots should fail promptly without provider/runtime plugin loading or surviving child processes.

## Reproduction Assessment

Yes. The reporter supplied process-group reproductions for `openclaw foo`, and current-main source inspection still shows the same deterministic proxy/plugin-registration route for non-built-in, non-owned roots.

## Solution Assessment

Not applicable as a patch. The narrow maintainable fix is a manifest/descriptor-backed CLI root ownership check before managed proxy startup and runtime plugin loading, not another timeout around gateway catalog discovery.
