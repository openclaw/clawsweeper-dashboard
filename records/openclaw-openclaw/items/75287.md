## Summary

Keep open. Current main still sends unrecognized CLI roots through managed proxy startup and lazy plugin CLI registration, and the runtime CLI loader still broad-loads plugins when no manifest owner is found.

## What This Changes

The issue requests that invalid CLI commands fail quickly without provider/runtime plugin loading, surviving child processes, or related gateway hot-path saturation.

## Best Possible Solution

Add a pre-runtime CLI ownership gate so only builtin roots and manifest/descriptor-owned plugin CLI roots can start proxy/plugin CLI registration; unowned roots should fail promptly without provider/runtime plugin loading or surviving child processes.

## Reproduction Assessment

Yes. The reporter supplied controlled process-group reproductions for `openclaw foo`, and current-main source inspection still shows the same deterministic proxy/plugin-registration route for non-built-in, non-owned roots.

## Solution Assessment

Not applicable as a patch. The narrow maintainable fix is a manifest/descriptor-backed CLI root ownership check before managed proxy startup and runtime plugin loading, not another timeout around gateway catalog discovery.
