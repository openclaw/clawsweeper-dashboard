## Summary

Keep open. Current main still lets arbitrary non-builtin primary CLI roots reach lazy plugin CLI registration, and when manifest activation finds no owning plugin the loader falls back to broad runtime plugin loading; the unknown-command path also still uses default proxy routing. The merged models-list mitigation reduces one gateway symptom, but it does not fix the central `openclaw foo` fail-fast and clean-exit bug.

## What This Changes

The issue asks for unknown or unavailable CLI commands to fail before provider/runtime plugin loading, exit cleanly without hot child processes, and reduce related gateway catalog pressure on hot control-plane paths.

## Best Possible Solution

Keep this issue open until unknown or unavailable primary CLI roots are classified from core descriptors plus lightweight plugin manifest/metadata before runtime plugin loading, skip managed proxy/runtime startup for truly unknown roots, preserve legitimate plugin CLI activation, and guarantee prompt cleanup on startup errors. Treat the `models.list` work from #75326 and the broader gateway saturation in #75297 as related but not sufficient to close this issue.

## Reproduction Assessment

Yes. The reporter supplied a concrete `openclaw foo` process-group repro, and current-main source inspection shows the deterministic path that can broad-load plugin CLI runtimes and start proxy routing for arbitrary non-builtin roots.

## Solution Assessment

Yes for the repair direction, not applicable as a proposed patch. The narrow maintainable fix is an early lightweight owner/unknown-command gate plus startup-error child cleanup, rather than a broad provider catalog redesign in the same PR.
