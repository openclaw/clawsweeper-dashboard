## Label Justifications

- `P1`: The issue reports a real channel workflow failure where stale final text can be delivered repeatedly over multiple days from persisted session state.
- `impact:message-loss`: The affected behavior duplicates stale outbound messages on heartbeat ticks and can also suppress fresh heartbeat work behind the replay path.
- `impact:session-state`: The root state is a persisted `pendingFinalDelivery` session entry whose retry metadata advances indefinitely without a terminal state.

## Summary

Keep open: current `main` still has a source-provable unbounded heartbeat pending-final replay path, and the related open items cover specific causes rather than this general retry/TTL fail-safe.

## What This Changes

Request to add an attempt cap or age-based expiry for heartbeat replays of stale `pendingFinalDeliveryText` so orphaned sessions stop re-emitting the same cached final message indefinitely.

## Best Possible Solution

Add a narrow heartbeat replay fail-safe in `get-reply.ts` that clears stale pending final delivery after a conservative attempt or age bound, logs the reason, and preserves normal replay behavior below the limit.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: seed a session entry with `pendingFinalDelivery: true` and substantive `pendingFinalDeliveryText`, then invoke the heartbeat reply path and observe the attempt count update plus returned replay text. I did not run a writable fixture because this review is read-only.

## Solution Assessment

Yes, a bounded retry/expiry check in the existing heartbeat pending replay branch is the narrowest maintainable fix; the clear-on-success and `target:none` PRs should remain separate cause-specific repairs.
