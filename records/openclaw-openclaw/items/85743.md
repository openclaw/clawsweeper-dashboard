## Label Justifications

- `P1`: The issue reports a real user-facing heartbeat/channel workflow that can repeatedly resend stale final text until session state is manually repaired.
- `impact:session-state`: The root condition is persisted session state with `pendingFinalDelivery` metadata that advances without a terminal state.
- `impact:message-loss`: The stale replay path can duplicate obsolete outbound text and block fresh heartbeat behavior behind cached delivery state.

## Summary

Keep open: current main still has the unbounded heartbeat `pendingFinalDelivery` replay path, but https://github.com/openclaw/openclaw/pull/85770 is an open closing PR that directly implements the requested retry/TTL fail-safe.

## What This Changes

The issue requests a bounded retry/TTL fail-safe for heartbeat `pendingFinalDelivery` replay so orphaned session state cannot resend stale final-delivery text indefinitely.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85770, or replace it with an equivalent narrow cap/TTL fix, then close this issue as implemented after merge.

## Reproduction Assessment

Yes. Source inspection shows a current-main path: seed a session with `pendingFinalDelivery: true` and substantive `pendingFinalDeliveryText`, invoke the heartbeat reply path, and the branch increments attempt metadata and returns the cached replay text with no cap or TTL.

## Solution Assessment

Yes. A narrow heartbeat-only retry/TTL gate in `get-reply.ts`, with focused persisted-session regression tests, is the maintainable fix; the existing open PR follows that shape and should be reviewed rather than starting another repair lane.
