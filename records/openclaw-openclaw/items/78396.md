## Summary

Keep open: current main still contains the forced Matrix bootstrap path that can recreate secret storage and then retry `setupNewCrossSigning`, and current tests assert that duplicate reset sequence instead of preventing it.

## What This Changes

The issue reports that Matrix `--force-reset-cross-signing` can perform a second cross-signing reset after secret-storage repair, corrupting E2EE recovery state.

## Best Possible Solution

Patch the Matrix plugin bootstrap flow so one forced reset invocation performs at most one destructive cross-signing reset while still exporting keys to consistent secret storage after the reset succeeds.

## Reproduction Assessment

Yes for source-level reproduction: current main has a deterministic path where a repairable secret-storage error during forced reset recreates SSSS and calls `setupNewCrossSigning` again. I did not establish a live homeserver reproduction in this read-only review.

## Solution Assessment

Unclear whether the reporter's exact two-line patch is the safest fix, because current code deliberately avoids secret-storage mutation before UIA succeeds. The maintainable fix should prevent repeated destructive resets while preserving that partial-mutation guard.
