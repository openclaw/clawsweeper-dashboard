## Summary

Keep open: current main and the latest release still contain the forced Matrix bootstrap path that can recreate secret storage and then retry `setupNewCrossSigning`, and the current unit tests assert that duplicate-call behavior rather than guarding against it.

## What This Changes

The issue asks OpenClaw to stop `openclaw matrix verify bootstrap --force-reset-cross-signing` from performing a second forced Matrix cross-signing reset after secret-storage repair.

## Best Possible Solution

Patch the Matrix plugin bootstrap flow so one command performs at most one forced cross-signing reset, repairs or recreates secret storage before relying on it for exported keys, and keeps the UIA failure path from leaving partial secret-storage mutations.

## Reproduction Assessment

Yes for source-level reproduction: current main has a deterministic path where a repairable secret-storage error during forced reset recreates SSSS and calls `setupNewCrossSigning` again. I did not establish a live homeserver reproduction in this read-only review.

## Solution Assessment

Unclear whether the exact two-line patch in the report is the safest fix, because current code deliberately avoids secret-storage mutation before UIA succeeds. The maintainable fix should prevent repeated destructive resets while preserving that partial-mutation guard.
