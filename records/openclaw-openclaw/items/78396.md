## Summary

Keep open: current main and v2026.5.7 still contain the Matrix forced-reset path that can recreate secret storage and then retry a destructive cross-signing reset, and the focused unit test still asserts that duplicate-reset behavior.

## What This Changes

The issue asks for the Matrix `--force-reset-cross-signing` bootstrap path to avoid running a second destructive cross-signing reset after secret-storage repair.

## Best Possible Solution

Patch the Matrix plugin bootstrap flow so one forced bootstrap can perform at most one destructive cross-signing reset, repairs secret storage without creating mismatched recovery material, and preserves the existing UIA no-partial-mutation guard.

## Reproduction Assessment

Yes, at source level: current main deterministically skips initial secret-storage bootstrap, then retries `setupNewCrossSigning` after repairable stale SSSS failure. I did not establish a live homeserver reproduction in this read-only pass.

## Solution Assessment

Unclear whether the exact two-line patch in the report is the safest fix, because current code intentionally avoids secret-storage mutation before required cross-signing UIA succeeds. The best fix should prevent repeated destructive resets while preserving that guard and recovery-key staging semantics.
