## Summary

Keep open: current main and the latest release still contain the reported Matrix forced-reset repair path, and no merged related PR supersedes it. Source, tests, and upstream SDK contracts show a repairable secret-storage failure can recreate SSSS and immediately call `setupNewCrossSigning` again.

## What This Changes

The issue reports that Matrix `--force-reset-cross-signing` can perform two destructive cross-signing resets and leave recovery-key, secret-storage, and backup state inconsistent.

## Best Possible Solution

Patch the Matrix bootstrapper so one forced bootstrap can perform at most one destructive cross-signing reset while keeping recovery-key staging, server secret storage, backup creation, and the existing UIA no-partial-mutation guard coherent.

## Reproduction Assessment

Yes, at source level: current main skips initial secret-storage bootstrap during forced reset, allows SSSS recreation after a repairable failure, and then calls setup-new cross-signing again. I did not establish a live homeserver reproduction in this read-only review.

## Solution Assessment

Unclear whether the exact two-line patch in the issue is the safest final shape. The maintainable fix should prevent the repeated destructive reset while preserving UIA handling, recovery-key staging, and backup consistency.
