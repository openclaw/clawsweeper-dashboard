## Summary

Keep open: current main and v2026.5.5 still contain the forced Matrix bootstrap path that can recreate secret storage after a repairable error and then retry `setupNewCrossSigning`, and the focused tests still assert that repeated reset behavior.

## What This Changes

The issue requests fixing Matrix verification bootstrap so `--force-reset-cross-signing` does not perform a second destructive cross-signing reset after secret-storage repair.

## Best Possible Solution

Patch the Matrix plugin bootstrap flow so a forced reset performs at most one destructive `setupNewCrossSigning` attempt per command while keeping secret storage, `recovery-key.json`, and room-key backup state coherent.

## Reproduction Assessment

Yes, at source level: a repairable secret-storage error during forced reset reaches a branch that recreates SSSS and calls `setupNewCrossSigning` again. I did not establish a live homeserver reproduction in this read-only pass.

## Solution Assessment

Unclear for the exact two-line patch from the report, because current code deliberately delays secret-storage mutation until cross-signing UIA succeeds. The right solution should prevent repeated destructive resets while preserving that guard.
