## Summary

Keep open: current main and v2026.5.12 still contain the Matrix forced-reset path that can recreate secret storage and then retry `setupNewCrossSigning`, and the focused test still asserts that duplicate-reset behavior. The related MAS PR is open and useful context, but it does not close this exact double-reset/data-loss report.

## What This Changes

The issue reports that Matrix `--force-reset-cross-signing` can perform a second destructive cross-signing reset after secret-storage repair, leaving recovery material inconsistent with server state.

## Best Possible Solution

Patch the Matrix plugin bootstrap flow so one forced bootstrap can perform at most one destructive cross-signing reset, keeps recovery-key staging/server secret storage coherent, and preserves the current UIA no-partial-mutation guard.

## Reproduction Assessment

Yes, at source level: current main skips initial secret-storage bootstrap during forced reset, recreates SSSS on a repairable failure, then calls `setupNewCrossSigning` again. I did not establish a live homeserver reproduction in this read-only pass.

## Solution Assessment

Unclear whether the exact two-line patch in the issue is the safest final shape. The best fix should prevent the repeated destructive reset while preserving the existing guard around UIA and secret-storage mutation.
