## Summary

Keep open: current main and the latest shipped release still contain the Matrix forced-reset path that can recreate secret storage after a repairable failure and immediately retry `setupNewCrossSigning`, matching the reported E2EE state-corruption risk.

## What This Changes

The issue asks OpenClaw to prevent Matrix `--force-reset-cross-signing` from performing a second destructive cross-signing reset after secret-storage repair.

## Best Possible Solution

Patch the Matrix bootstrapper so one forced bootstrap can perform at most one destructive cross-signing reset, while keeping recovery-key staging, server secret storage, backup creation, and the existing UIA no-partial-mutation guard coherent.

## Reproduction Assessment

Yes, at source level: current main still skips initial secret-storage bootstrap during forced reset, allows secret-storage recreation on a repairable failure, and then calls `setupNewCrossSigning` again. I did not establish a live homeserver reproduction in this read-only review.

## Solution Assessment

Unclear whether the exact two-line patch in the issue is the safest final shape. The maintainable fix should prevent the repeated destructive reset while preserving Matrix UIA handling and recovery/backup consistency.
