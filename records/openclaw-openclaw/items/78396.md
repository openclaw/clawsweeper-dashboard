## Summary

Keep open: current main and the latest release still contain the forced Matrix cross-signing repair path that can recreate secret storage and retry `setupNewCrossSigning`, and the exact double-reset issue is not owned by a merged fix or a safe canonical PR.

## What This Changes

The issue reports that Matrix `--force-reset-cross-signing` can run a second forced cross-signing reset after recreating secret storage, leaving Matrix E2EE recovery state inconsistent.

## Best Possible Solution

Change the Matrix crypto bootstrapper so forced reset prepares or repairs secret storage before the destructive reset path and cannot retry `setupNewCrossSigning` after SSSS recreation, with regression coverage proving one forced reset and consistent recovery-key handling.

## Reproduction Assessment

Yes, at source level. Current main still skips initial secret-storage bootstrap for forced reset, enables repair-without-recovery-key, recreates SSSS on a repairable error, and retries `setupNewCrossSigning`; I did not run a live Matrix account reproduction.

## Solution Assessment

Yes directionally: the best fix belongs in `MatrixCryptoBootstrapper` rather than CLI/docs, because that is where forced reset, SSSS repair, and SDK cross-signing calls are sequenced. The implementation should be narrower than the open MAS PR and must preserve the earlier passwordless-bot guardrail.
