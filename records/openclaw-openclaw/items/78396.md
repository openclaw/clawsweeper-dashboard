## Summary

Keep open: current main still contains the Matrix forced cross-signing reset path that can recreate secret storage after a repairable failure and immediately retry `setupNewCrossSigning`, matching the reported double destructive reset risk. The related MAS PR is useful context, but it is still open and does not provide a merged fix for this exact data-loss path.

## What This Changes

The issue requests changing Matrix crypto bootstrap so `--force-reset-cross-signing` cannot perform two destructive cross-signing resets and leave recovery material inconsistent.

## Best Possible Solution

Patch the Matrix bootstrapper so one forced bootstrap can perform at most one destructive cross-signing reset, keeps recovery-key staging and server secret storage coherent, and preserves the existing UIA no-partial-mutation guard.

## Reproduction Assessment

Yes, at source level: current main skips initial secret-storage bootstrap during forced reset, allows secret-storage recreation on a repairable failure, then calls `setupNewCrossSigning` again. I did not establish a live homeserver reproduction in this read-only review.

## Solution Assessment

Unclear whether the exact two-line patch in the issue is the safest final shape. The best fix should prevent the repeated destructive reset while preserving Matrix UIA handling, recovery-key staging, and backup consistency.
