## Summary

Keep open: current main and the latest release still contain the Matrix forced-reset flow that can recreate secret storage inside the cross-signing reset error path and then call cross-signing bootstrap again; the related MAS/cross-signing PR is still open, so this user-facing E2EE recovery bug is not implemented or safely superseded.

## What This Changes

The issue asks OpenClaw to fix Matrix `--force-reset-cross-signing` so stale secret-storage repair cannot trigger repeated cross-signing resets and leave recovery or backup state incoherent.

## Best Possible Solution

Keep this issue open until the Matrix bootstrap flow prevents stale SSSS forced reset from doing repeated reset/repair cycles, with focused regression coverage and real Matrix E2EE or MAS proof before closing against a merged PR and release.

## Reproduction Assessment

Yes, at source level: current main's forced-reset branch still skips upfront secret-storage bootstrap, permits SSSS recreation inside cross-signing repair, and has a test asserting three cross-signing bootstrap calls for stale server SSSS. I did not run a destructive live Matrix repro.

## Solution Assessment

Unclear on the exact proposed two-line patch: the current code deliberately delays SSSS mutation for passwordless UIA safety, and the open MAS PR changes nearby auth behavior. The best fix should be maintainer-reviewed in the Matrix E2EE bootstrap path with regression tests and live proof.
