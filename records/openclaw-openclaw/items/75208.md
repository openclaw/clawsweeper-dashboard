## Summary

Keep open: current main already has the base artifact RPC implementation from the merged hardening work, but this PR still proposes unique cursor/type filtering and mixes in a Hyprland browser path that has correctness, security, and proof blockers before it can be merged.

## What This Changes

The branch adds Gateway and SDK artifact RPC schemas/handlers with cursor pagination and type filtering, changes artifact provenance filtering and SDK helper signatures, regenerates docs/Swift models, and adds an opt-in Hyprland screenshot capture path.

## Best Possible Solution

Keep current main's artifact RPC hardening as the base, then land cursor/type filtering as a focused Gateway/SDK protocol change with stable cursor semantics and handle Hyprland capture through the dedicated browser fix path.

## Reproduction Assessment

Yes. Source inspection gives high-confidence paths for the blockers: a run/task query with an untagged same-session artifact can leak through the PR filter, a message with more artifacts than the page limit loses leftovers after a messageSeq cursor, and the Hyprland route reads a resolved config field that the PR does not fully wire.

## Solution Assessment

No. The PR is not the best merge vehicle until it preserves current SDK signatures, keeps scoped artifact queries provenance-safe, uses cursor tie-breakers, and separates or fully wires the Hyprland feature.
