## Summary

Keep open for maintainer review: current main already has the base artifact RPC implementation from the merged hardening work, but this PR still contains unique cursor/type-filter and Hyprland work that is not safely superseded and has concrete correctness/security blockers plus missing real behavior proof.

## What This Changes

The PR adds and revises Gateway/SDK artifact RPC schemas, handlers, SDK/docs/generated models, cursor/type filtering, and a Hyprland browser screenshot capture path.

## Best Possible Solution

Keep the shipped artifact RPC implementation as the base, then split cursor/type filtering into a focused Gateway/SDK protocol follow-up with stable cursor semantics and handle Hyprland capture separately under browser-plugin ownership.

## Reproduction Assessment

Yes. Source inspection gives high-confidence reproduction paths for the PR blockers: scoped run/task queries can include untagged artifacts, messageSeq-only cursors skip same-message artifacts after a limited page, and the Hyprland toggle is read from a resolved config field that the PR does not populate.

## Solution Assessment

No. The PR overlaps the shipped artifact RPC base and is not the best merge vehicle until it preserves current SDK signatures, keeps scoped queries safe, uses a cursor tiebreaker, and separates or fully wires the Hyprland feature.

Full review comments:

- **[P2] Keep run and task artifact queries provenance-scoped:** `src/gateway/server-methods/artifacts.ts:233-237`
  - body: This keeps messages that lack explicit run/task provenance for scoped queries, so a runId or taskId request can return unrelated untagged artifacts from the resolved session. Keep the current fail-closed behavior or add a proven mapping before returning those artifacts.

  - body: Current main documents and tests `oc.artifacts.get(id, params)` and `download(id, params)`, but this branch exposes param-object calls. Reconcile with the merged SDK shape so rebasing this PR does not break current consumers.
  - confidence: 0.84
- **[P2] Wire hyprlandCapture into resolved browser config:** `extensions/browser/src/browser/routes/agent.snapshot.ts:496-497`
  - body: The route checks `ctx.state().resolved.hyprlandCapture`, but this PR only adds schema/type surface and does not populate the resolved browser config through the normal resolver. In a real gateway the new path remains disabled.
