## Summary

Keep open for maintainer reconciliation. Current main has the core artifact RPC surface from the merged #74926 path, but this PR still proposes different cursor, type-filter, and provenance semantics plus an unrelated Hyprland capture path; the patch has blocking correctness and low-severity security findings, so it is not a cleanup-close candidate.

## What This Changes

The PR adds Gateway and SDK artifact list/get/download RPCs with cursor pagination and type filtering, changes run/task artifact provenance filtering, updates docs/tests/generated Swift models, and adds a Hyprland screenshot capture path and config surface.

## Best Possible Solution

Keep the shipped #74926 artifact API as the base, then decide cursor/type filtering as a focused follow-up with a stable cursor tiebreaker and explicit provenance rules. Split the Hyprland capture work into a separate browser-owned PR if it is still wanted.

## Reproduction Assessment

Yes. The PR cursor bug reproduces with one transcript message containing two artifact blocks, `artifacts.list` with `limit: 1`, then the returned cursor; the second artifact is skipped. The provenance issue reproduces by querying a runId on a multi-run session with untagged transcript artifact messages, and the Hyprland wiring issue is visible by following `browser.hyprlandCapture` through `resolveBrowserConfig`.

## Solution Assessment

No. The PR direction overlaps a merged maintainer implementation, but this branch is not the best final solution because it changes provenance semantics, adds message-sequence-only pagination, and mixes browser capture work into the artifact API patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep run and task artifact queries provenance-scoped:** `src/gateway/server-methods/artifacts.ts:232-235`
  - body: This changes runId/taskId filtering to keep messages that lack explicit provenance, so a run-scoped query can return unrelated untagged artifacts from the same session. The merged maintainer path intentionally requires matching transcript provenance for run/task scopes; keep that fail-closed behavior or add a separate proven mapping before returning these artifacts.
  - confidence: 0.9
- **[P2] Preserve artifacts that share a message sequence:** `src/gateway/server-methods/artifacts.ts:388`
  - body: The cursor only encodes `messageSeq`. If one transcript message contains more artifacts than the page limit, the next request filters with `messageSeq > cursorSeq` and permanently skips the remaining artifacts from that message. Include a tiebreaker such as content index or artifact id in the cursor.
  - confidence: 0.95
- **[P2] Wire the Hyprland toggle through resolved config:** `extensions/browser/src/browser/routes/agent.snapshot.ts:493`
  - body: The route checks `ctx.state().resolved.hyprlandCapture`, but the normal config resolver never copies the raw browser setting into the returned resolved config. In a real gateway this condition remains falsey, so the new capture path cannot be enabled outside test setup.
  - confidence: 0.9
- **[P3] Resolve Hyprland tools from trusted paths:** `extensions/browser/src/browser/hyprland-capture.ts:8`
  - body: The new helper spawns `hyprctl` and `grim` by command name through `PATH`. Since this is a Gateway subprocess path, resolve those binaries through the existing trusted system-binary helper before spawning so an enabled feature cannot run a shadowed command.
  - confidence: 0.78
- **[P3] Add a changelog entry for the public API change:** `packages/sdk/src/client.ts:786`
  - body: This PR exposes new user-facing Gateway and SDK artifact APIs but does not update `CHANGELOG.md`. Repo policy requires a changelog entry for public `feat` changes so the release notes capture the new API surface.
  - confidence: 0.86
