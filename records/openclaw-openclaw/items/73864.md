## Summary

Keep open. Current main still lacks a supported `openclaw mcp serve` scope selector and the bridge continues to request the fixed read/write/approvals scope set; this issue is paired with open PR #74428, so cleanup should wait for that PR to be reviewed, landed, or closed.

## What This Changes

Request a supported way for `openclaw mcp serve` to request narrower operator scopes or explicit least-privilege MCP modes.

## Best Possible Solution

Land #74428 or an equivalent narrow change that validates MCP bridge-relevant scopes, threads them through CLI serve options into `GatewayClient`, preserves the existing default when omitted, and updates docs/tests for explicit, invalid, default, and cached-token cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: inspect `mcp serve`, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()` to see there is no configurable scope input and the bridge always supplies the fixed scope list.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to the MCP bridge's least-privilege operator scopes is the narrowest maintainable fix because the gateway client already accepts requested scopes.
