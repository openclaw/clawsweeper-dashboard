## Summary

Keep open. Current main still lacks a configurable `openclaw mcp serve` operator-scope request path, and the issue is already paired with open implementation PR #74428 by the same reporter, so it should be reviewed or landed rather than closed or duplicated. citeturn0view0

## What This Changes

Request a supported `openclaw mcp serve` scope or mode selector so separate MCP bridge identities can ask for narrower operator-scope sets.

## Best Possible Solution

Land #74428 or an equivalent narrow implementation that validates repeatable MCP bridge scopes, threads an `OperatorScope[]` through CLI, server, bridge, and `GatewayClient`, preserves the default read/write/approvals request when omitted, documents the contract, and covers explicit/default/cached-token behavior with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `mcp serve` has no scope option or serve option field, and the bridge hardcodes read/write/approvals when constructing `GatewayClient`.

## Solution Assessment

Yes. A validated repeatable scope option limited to MCP bridge-relevant operator scopes is the narrow maintainable direction; static `tools/list` filtering can remain documented or be a follow-up if invocation-time gateway auth remains canonical.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the requested least-privilege bridge profiles affect effective operator authorization.

Concerns:

- **[medium] MCP bridge cannot request narrower scopes:** `src/mcp/channel-bridge.ts:124`
  - body: Current main always requests read, write, and approvals for the MCP bridge, so separate bridge identities cannot reduce blast radius by choosing read-only or approval-only privileges.
  - confidence: 0.9

  - body: `GatewayClient.resolveConnectScopes()` reuses stored device-token scopes before constructor-provided scopes, so an implementation must define how explicit MCP scopes interact with cached broader tokens.
