## Summary

Keep open. Current main still lacks configurable operator scopes for `openclaw mcp serve`: the CLI has no `--scope` or mode selector, the MCP serve options do not carry scopes, and the bridge still requests the fixed read/write/approvals set. The issue is paired with open replacement PR #74428, so the maintainer path is to review or land that PR rather than close this issue or queue duplicate repair work.

## What This Changes

Request a supported way for `openclaw mcp serve` to request a narrower operator scope set for least-privilege MCP bridge identities.

## Best Possible Solution

Land #74428 or an equivalent narrow change that validates MCP bridge-relevant operator scopes, threads an `OperatorScope[]` through the CLI, serve options, bridge construction, and `GatewayClient`, preserves the existing read/write/approvals default when omitted, and updates docs/tests for explicit, default, invalid, and cached-token cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main by inspecting the `mcp serve` command, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()` to confirm there is no configurable scope input and the bridge always requests the fixed read/write/approvals set.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to the MCP bridge's read/write/approvals scopes is the narrowest maintainable fix because the gateway client already accepts requested scopes; dynamic `tools/list` filtering can stay documented or follow later if maintainers want it.
