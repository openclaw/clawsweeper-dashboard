## Summary

Keep open. Current `main` and the latest release still have no `openclaw mcp serve` scope or mode selector, and the bridge still requests the fixed read/write/approvals scope set; https://github.com/openclaw/openclaw/pull/74428 is the active implementation candidate while https://github.com/openclaw/openclaw/pull/74030 has already been closed as superseded.

## What This Changes

The issue requests a supported way for `openclaw mcp serve` to request narrower operator scopes than the current fixed read/write/approvals set.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74428 or an equivalent narrow implementation that validates MCP bridge scopes, preserves current defaults when omitted, documents the contract, and adds focused CLI/server coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows no `--scope` or mode option, no scope field in `OpenClawMcpServeOptions`, and a fixed MCP bridge scope list passed to `GatewayClient`.

## Solution Assessment

Yes, with maintainer review. A repeatable, allowlisted `--scope` flag threaded through the MCP serve options to `GatewayClient` is the narrow maintainable path; broader agent tool-policy mapping remains separate from this issue.
