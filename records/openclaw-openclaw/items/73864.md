## Summary

Keep open. Current main still lacks configurable MCP bridge operator scopes, and the issue now has an open focused replacement PR (#74428) after the earlier implementation PR (#74030) was closed unmerged, so the maintainer path is to review/rebase/land the linked PR rather than close this issue or queue duplicate work.

## What This Changes

Request a supported `openclaw mcp serve` scope or mode selector so MCP bridge processes can request least-privilege subsets of `operator.read`, `operator.write`, and `operator.approvals`.

## Best Possible Solution

Land #74428 or an equivalent narrow implementation that validates repeatable MCP bridge scopes, threads an `OperatorScope[]` through the CLI, server, bridge, and `GatewayClient`, preserves the default read/write/approvals request when omitted, updates docs/changelog, and covers explicit/default/cached-token behavior with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `mcp serve` has no scope option or serve option field, and the bridge hardcodes read/write/approvals when constructing `GatewayClient`.

## Solution Assessment

Yes. A validated repeatable scope option limited to MCP bridge-relevant operator scopes is the narrow maintainable direction; static `tools/list` filtering can remain a documented or follow-up behavior if invocation-time gateway auth is canonical.
