## Summary

Keep open. Current main still has the MCP least-privilege gap: `openclaw mcp serve` has no scope or mode selector, the serve option type cannot carry requested scopes, and the bridge still requests the fixed read/write/approvals set. Open PR #74428 is the active same-author implementation candidate, while #74030 is already closed unmerged, so this issue should stay open until the replacement PR lands or maintainers choose a different path.

## What This Changes

Add a supported way for `openclaw mcp serve` to request narrower operator scopes, such as repeatable `--scope` values or explicit bridge modes.

## Best Possible Solution

Land #74428 or an equivalent narrow change that validates MCP bridge-relevant scopes, threads them through CLI/server/bridge into `GatewayClient`, preserves the current default when omitted, and updates docs/tests for explicit, default, invalid, and cached-token cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main by inspecting `mcp serve`, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()`; no live gateway is needed to see that the requested scope set is fixed.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to the MCP bridge's read/write/approvals scopes is the narrow maintainable fix; dynamic `tools/list` filtering can remain documented or follow separately.
