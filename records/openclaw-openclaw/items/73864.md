## Summary

Keep open. Current main still has the reported MCP least-privilege gap: `openclaw mcp serve` exposes no scope or mode selector, the serve options cannot carry scopes, and the bridge still constructs its Gateway client with the fixed read/write/approvals scope list. The issue is also paired with open implementation PR #74428 by the same author, while PR #74030 was closed unmerged, so the maintainer-safe path is to review or land the linked replacement PR rather than close this issue or queue a duplicate repair.

## What This Changes

The issue requests configurable operator scopes for `openclaw mcp serve` so separate MCP bridge identities can request least-privilege read, write, or approvals scope sets.

## Best Possible Solution

Land #74428 or an equivalent narrow change that validates MCP bridge-relevant operator scopes, threads an `OperatorScope[]` through the CLI, serve options, bridge construction, and `GatewayClient`, preserves the existing read/write/approvals default when omitted, documents the behavior, and covers explicit/default/cached-token cases with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: inspect the `mcp serve` command, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()` to see there is no configurable scope input and the bridge always requests the fixed read/write/approvals set.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to the MCP bridge’s read/write/approvals scopes is the narrowest maintainable solution; dynamic `tools/list` filtering can remain documented or be handled as a follow-up if maintainers want it.
