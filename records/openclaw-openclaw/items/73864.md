## Summary

Keep open. Current main still lacks configurable `openclaw mcp serve` operator scopes: the CLI, serve options, docs, and bridge construction all still use the fixed read/write/approvals request. The issue is also paired with open implementation PR #74428, while #74030 was closed unmerged, so the next step is PR review rather than closing or queueing a duplicate repair.

## What This Changes

Request a supported way for `openclaw mcp serve` to request narrower operator scopes, such as repeatable `--scope`, config/env support, or explicit read/write/approvals modes.

## Best Possible Solution

Land #74428 or an equivalent narrow implementation that validates MCP bridge-relevant operator scopes, threads an `OperatorScope[]` through CLI, server, bridge, and `GatewayClient`, preserves the default read/write/approvals request when omitted, documents the behavior, and covers explicit/default/cached-token cases with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: inspect `mcp serve` registration, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()` to see there is no configurable scope input and the bridge still passes a fixed scope list.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to the MCP bridge's read/write/approvals scopes is the narrowest maintainable fix; filtering `tools/list` can remain documented or follow-up behavior if gateway invocation auth is canonical.
