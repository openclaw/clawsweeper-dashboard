## Summary

Keep open. Current main still lacks configurable operator scopes for `openclaw mcp serve`, and the issue is paired with open implementation PR #74428 while earlier PR #74030 was closed unmerged.

## What This Changes

Add a supported way for `openclaw mcp serve` to request narrower operator scopes or explicit read/write/approvals modes for least-privilege MCP bridge identities.

## Best Possible Solution

Land #74428 or an equivalent narrow implementation that validates MCP bridge-relevant operator scopes, threads an `OperatorScope[]` through CLI, server, bridge, and `GatewayClient`, preserves the current read/write/approvals default when omitted, documents the behavior, and covers explicit/default/cached-token cases with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: inspect `mcp serve` registration, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()` to see there is no configurable scope input and the bridge still passes a fixed scope list.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to the MCP bridge's read/write/approvals scopes is the narrowest maintainable path; static `tools/list` filtering can remain a documented follow-up if maintainers want it.
