## Summary

Keep open. Current main still has the reported MCP least-privilege gap: `openclaw mcp serve` has no scope or mode selector, its serve options cannot carry scopes, and the bridge still requests the fixed read/write/approvals scope set. The issue is already paired with open replacement PR #74428, while #74030 is closed unmerged, so the maintainer path is to review or land the linked PR rather than close this issue or queue duplicate repair work.

## What This Changes

Request a supported `openclaw mcp serve` scope or mode selector so MCP bridge processes can request narrower operator scope sets than the fixed read/write/approvals default.

## Best Possible Solution

Land #74428 or an equivalent narrow change that validates MCP bridge-relevant operator scopes, threads an `OperatorScope[]` through the CLI, serve options, bridge construction, and `GatewayClient`, preserves the existing read/write/approvals default when omitted, and updates docs/tests for explicit, default, invalid, and cached-token cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: inspect the `mcp serve` command, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()` to see there is no configurable scope input and the bridge always requests the fixed read/write/approvals set.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to the MCP bridge's read/write/approvals scopes is the narrowest maintainable fix because the gateway client already accepts requested scopes; dynamic `tools/list` filtering can stay documented or follow later if maintainers want it.
