## Summary

Keep open. Current main still has the MCP least-privilege gap: the CLI exposes no scope or mode selector, the serve option type cannot carry requested scopes, and the bridge still requests the fixed read/write/approvals set; #74428 is the active replacement implementation while #74030 is closed unmerged. citeturn1view4turn1view5

## What This Changes

Request a supported way for `openclaw mcp serve` to request narrower operator scopes or MCP modes instead of always asking for read/write/approvals.

## Best Possible Solution

Land #74428 or an equivalent narrow change that validates MCP bridge-relevant scopes, threads them through CLI/server/bridge into `GatewayClient`, preserves the current default when omitted, and updates docs/tests/changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main by inspecting `mcp serve`, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()`; no live gateway is needed to see that the requested scope set is fixed.

## Solution Assessment

Yes. A fail-closed repeatable `--scope` option limited to `operator.read`, `operator.write`, and `operator.approvals` is the narrow maintainable fix; static `tools/list` behavior can be documented or handled separately.
