## Summary

Keep open. Current main still has the reported MCP least-privilege gap: `openclaw mcp serve` exposes no scope or mode selector, the MCP serve options cannot carry scopes, and the bridge still constructs every Gateway client with the fixed read/write/approvals scope list. The issue is valid and already paired with open implementation PR #74428 by the same author, while earlier PR #74030 was closed unmerged, so the maintainer path is to review or land the paired PR rather than close this issue or queue duplicate repair work.

## What This Changes

The issue requests a supported way for `openclaw mcp serve` to request configurable narrower operator scopes, such as repeatable `--scope`, config/env support, or explicit read/write/approvals modes.

## Best Possible Solution

Land #74428 or an equivalent narrow change that validates MCP bridge-relevant operator scopes, threads an `OperatorScope[]` through the CLI, serve options, bridge construction, and `GatewayClient`, preserves the existing default when omitted, updates MCP docs/changelog, and covers explicit/default/cached-token behavior with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: inspect the `mcp serve` command, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()` to see there is no configurable scope input and the bridge always requests the fixed read/write/approvals set.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to the MCP bridge’s read/write/approvals scopes is the narrowest maintainable fix; dynamic `tools/list` filtering can remain documented or be handled as a separate follow-up if maintainers want it.
