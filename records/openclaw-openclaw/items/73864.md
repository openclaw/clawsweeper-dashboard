## Summary

Keep open. Current main still has no `openclaw mcp serve` scope or mode selector and the MCP bridge still requests the fixed read/write/approvals scope set; https://github.com/openclaw/openclaw/pull/74428 is the active same-author implementation candidate, while https://github.com/openclaw/openclaw/pull/74030 was closed as the broader superseded attempt.

## What This Changes

The issue requests a supported `openclaw mcp serve` scope or mode selector for least-privilege MCP bridge identities.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/74428 or an equivalent narrow change that validates MCP bridge-relevant scopes, preserves the current default when omitted, documents the contract, and adds focused CLI/server coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows no `--scope` or mode option, no scopes field in `OpenClawMcpServeOptions`, and fixed scopes passed to `GatewayClient`; a live gateway is not needed to verify this gap.

## Solution Assessment

Yes, with maintainer review. A repeatable `--scope` flag constrained to `operator.read`, `operator.write`, and `operator.approvals` is the narrow maintainable path; dynamic `tools/list` filtering can remain separate if maintainers want it later.
