## Summary

Keep open. Current main still has the reported MCP least-privilege gap: `openclaw mcp serve` exposes no scope or mode selector, its serve options cannot carry requested scopes, and the bridge still requests the fixed read/write/approvals set. Open PR #74428 already targets this feature while #74030 is closed unmerged, so the maintainer path is to review or land the open PR rather than close this issue or queue duplicate repair work.

## What This Changes

Request a supported way for `openclaw mcp serve` to request narrower operator scopes for separate least-privilege MCP bridge identities.

## Best Possible Solution

Land #74428 or an equivalent narrow change that validates MCP bridge-relevant operator scopes, threads them through the CLI/server/bridge into `GatewayClient`, preserves the existing default when omitted, and updates docs/tests for explicit, default, invalid, and cached-token cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main by inspecting the `mcp serve` command, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()` fixed `GatewayClient` scope list.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to the MCP bridge's read/write/approvals scopes is the narrowest maintainable fix; dynamic `tools/list` filtering can remain documented or follow as a separate enhancement.
