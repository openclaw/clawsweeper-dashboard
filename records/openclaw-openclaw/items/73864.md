## Summary

Keep open. Current main still has the MCP least-privilege gap: `openclaw mcp serve` has no scope or mode selector, the serve options cannot carry requested scopes, and the bridge still requests the fixed read/write/approvals set; open PR #74428 is the active implementation candidate while #74030 is closed unmerged.

## What This Changes

Request a supported `openclaw mcp serve` scope, mode, config, or env selector so separate MCP bridge identities can use narrower operator scope sets.

## Best Possible Solution

Land #74428 or an equivalent narrow change that validates MCP bridge-relevant scopes, threads them through CLI/server/bridge into `GatewayClient`, preserves the existing default when omitted, and updates docs/tests for explicit, invalid, and default cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main by inspecting `mcp serve`, `OpenClawMcpServeOptions`, and `OpenClawChannelBridge.start()`; no live gateway is needed to see that the requested scope set is fixed.

## Solution Assessment

Yes. A validated repeatable `--scope` option limited to `operator.read`, `operator.write`, and `operator.approvals` is the narrow maintainable fix; dynamic `tools/list` filtering can remain documented or follow separately.
