## Summary

Keep open. Current main preserves existing external route state so webchat/dashboard access does not overwrite it, but it does not add the requested opt-in webchat-to-last-external-channel mirroring feature.

## What This Changes

Add a default-off session configuration option that mirrors complete Control UI/WebChat replies to the session's last established external channel route.

## Best Possible Solution

Add a default-false `session.mirrorWebchatToLastExternalChannel` config surface that keeps WebChat as the primary delivery path and sends a secondary complete reply through the existing external `routeReply` path only when the session has a valid persisted external route.

## Reproduction Assessment

Yes. The current-main source and tests provide a high-confidence static reproduction: seed a session with an external last route, send from a webchat client, and the dispatch context remains `webchat` with no external destination.

## Solution Assessment

Yes, the requested opt-in flag is the narrow maintainable direction if maintainers want this workflow. It avoids restoring the old implicit mirroring bug globally while reusing the preserved last-route state already present in current main.

## Repair Work Prompt

Implement #44302 as a narrow opt-in feature on current main. Add `session.mirrorWebchatToLastExternalChannel?: boolean` (singular `session`) with default false in config types, zod schema, generated schema/help/docs surfaces, and user docs. When enabled, a Control UI/WebChat-originated turn should still deliver normally to WebChat, and after successful WebChat delivery should also mirror complete agent reply payloads to the session's last established external route using existing persisted `lastChannel`/`lastTo` or delivery context state and the existing `routeReply` delivery path. Do not restore unconditional implicit mirroring. Do not mirror when no routable external route exists, when the origin is already external, or when the target would be webchat/internal. Mirror failures should be logged or surfaced as non-blocking diagnostics and must not block WebChat delivery. Avoid streaming/partial duplicate delivery unless the existing final/block delivery contracts already make it safe. Use #52068 only as context/source inspiration; rebuild against current main. Update docs and add a user-facing changelog entry with reporter credit, not bot credit.
