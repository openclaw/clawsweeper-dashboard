## Label Justifications

- `P2`: This is a normal-priority feature for a real multi-device workflow, but it needs policy review before implementation.
- `impact:message-loss`: Users relying on mobile continuity can miss replies because current WebChat-originated replies remain in WebChat rather than reaching the expected external channel.
- `impact:security`: The requested cross-channel mirroring can send internal WebChat content to a prior external recipient unless privacy and route-freshness rules are explicit.

## Summary

Keep open. Current main still intentionally keeps ordinary WebChat/Control UI replies internal unless an explicit delivery route is present, and the requested default-off mirroring config does not exist on main or the latest release.

## What This Changes

Add an opt-in setting that mirrors Control UI/WebChat replies to a session's last active external channel for multi-device workflows.

## Best Possible Solution

Define a default-off mirroring policy with explicit stale-route, no-route, and privacy safeguards; then implement it through the core reply-routing/config surface with focused tests, docs, and live channel proof.

## Reproduction Assessment

Yes, at source level for the current behavior. Current routing code and tests intentionally keep internal WebChat turns local unless explicit delivery is set, and the requested mirroring config key is absent.

## Solution Assessment

Unclear until maintainers choose the policy. A default-off explicit setting is plausible, but the implementation must define stale-route, no-route, and privacy behavior before a repair PR is safe.
