## Label Justifications

- `P2`: This is a normal-priority feature for a real multi-device workflow, but it requires product/security review before implementation.
- `impact:message-loss`: Users relying on mobile continuity can miss replies because WebChat-originated replies remain in WebChat instead of reaching the expected external channel.
- `impact:security`: The requested cross-channel mirroring can send internal WebChat content to a previous external recipient unless privacy and route-freshness rules are explicit.

## Summary

Keep open. Current main and the latest release still keep ordinary WebChat/Control UI replies internal unless explicit external delivery is requested, and the requested default-off mirroring config is not implemented; this remains a real feature/security policy decision rather than stale cleanup.

## What This Changes

Add a default-off session setting that mirrors WebChat or Control UI replies to the session's last established external channel.

## Best Possible Solution

Define a default-off mirroring policy with explicit stale-route, no-route, authorization, and privacy safeguards, then implement it in the core reply-routing/config surface with focused tests, docs, and live channel proof.

## Reproduction Assessment

Yes, at source level. Current routing code and tests intentionally keep WebChat-originated replies local unless ExplicitDeliverRoute is set, and the requested mirroring config key is absent.

## Solution Assessment

Unclear until maintainers choose the policy. A default-off explicit setting is plausible, but the implementation must define stale-route, authorization, no-route, and privacy behavior before a repair PR is safe.
