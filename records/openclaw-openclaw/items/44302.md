## Label Justifications

- `P2`: This is a normal-priority channel workflow feature with real but bounded multi-device delivery impact.
- `impact:message-loss`: Users relying on mobile continuity can miss external-channel replies when a response is authored from WebChat only.
- `impact:security`: The requested mirroring can intentionally copy WebChat-authored content into an external transport, so stale or unintended routes are a sensitive-data boundary.

## Summary

Keep open. Current main and the latest release still intentionally keep ordinary WebChat/Control UI sends on WebChat unless an explicit external delivery route is requested, the requested default-off mirroring config is absent, and the linked implementation attempts are closed unmerged rather than safe canonical fixes.

## What This Changes

Request an opt-in config setting that mirrors Control UI/WebChat replies to the session's last active external channel for multi-device workflows.

## Best Possible Solution

Keep this as a maintainer-approved core routing feature only if the design is default-off, defines stale-route and permission safeguards, updates schema/docs, and adds focused coverage for WebChat-only, established external route, stale route, and explicit-delivery paths.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` route ordinary WebChat turns as internal and the config schema lacks the requested mirroring key. I did not run a live Discord or Telegram WebChat flow in this read-only review.

## Solution Assessment

Unclear. A default-off mirroring flag is safer than restoring implicit fan-out, but maintainers still need to choose the product/security model versus explicit per-send delivery or the broader force-origin policy.
