## Summary

Keep open. Current main preserves established external route metadata, but it still lacks the requested opt-in WebChat-to-external mirroring config and intentionally keeps normal Control UI/WebChat replies internal unless an explicit delivery route is set.

## What This Changes

Request an opt-in session config flag that mirrors Control UI/WebChat replies to the session's last active external channel while leaving default routing unchanged.

## Best Possible Solution

Add a documented default-false session-level mirroring or operator-reply policy that mirrors only complete WebChat replies through the existing `routeReply` path when a valid persisted external route exists, with stale-route and privacy safeguards.

## Reproduction Assessment

Yes, at source level. Seed a session with external `lastChannel`/`lastTo`, send through the Control UI/WebChat `chat.send` path, and current main resolves the turn as internal WebChat with no external delivery route unless explicit delivery is set.

## Solution Assessment

Unclear as a merge path. A default-off flag is a plausible narrow direction, but maintainers still need to settle the public config shape and whether the broad open implementation candidate should be split before landing.
