## Label Justifications

- `P2`: This is a meaningful multi-device workflow improvement with bounded impact, but it is an additive feature rather than an urgent regression under the current contract.
- `impact:security`: Automatic mirroring would send content from an internal Control UI surface to a persisted external recipient and therefore crosses a privacy and authorization boundary.
- `impact:message-loss`: The motivating workflow treats replies that remain only in webchat as missed messages on the expected mobile channel, while an unsafe implementation could also duplicate or misroute them.

## Summary

Current main intentionally keeps ordinary Control UI turns on webchat unless delivery is explicitly requested, and no supported config enables automatic mirroring. The related implementation PRs were closed unmerged, while the broader open routing issue covers origin-delivery failures rather than this focused dual-delivery feature, so this request remains necessary but needs an explicit product and security decision.

## What This Changes

Add an opt-in mode that delivers Control UI replies to webchat and also mirrors them to the session’s most recent valid external channel target.

## Best Possible Solution

Keep webchat-only delivery as the default, then add an explicitly approved per-turn or tightly scoped session delivery choice that shows the external target and enforces route freshness, authorization/send policy, account and thread pairing, deduplication, loop prevention, transcript behavior, and visible partial-failure reporting.

## Reproduction Assessment

Not applicable as a bug reproduction: current main intentionally keeps ordinary webchat replies on webchat, and source plus focused tests establish that behavior with high confidence. This issue asks to introduce a new opt-in dual-delivery contract.

## Solution Assessment

No, the proposed global last-external-channel boolean is not yet proven to be the best solution. An explicit per-turn or narrowly scoped session control is safer because it can expose the selected recipient and avoid silently reusing stale routing state.

- A broad global flag may conflict with per-channel send policies, account selection, topic/thread targeting, and sessions that have moved between multiple external channels.
