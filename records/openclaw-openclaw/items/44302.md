## Label Justifications

- `P2`: This is a normal-priority channel workflow feature with real but bounded multi-device delivery impact.
- `impact:message-loss`: Users relying on mobile continuity can miss replies on the external channel when a response is authored from webchat.
- `impact:security`: Mirroring webchat content to external channels can expose sensitive conversation content across channel boundaries if the route is stale or unintended.

## Summary

Keep open. Current main preserves established external routes for some handoff/event paths, but it still intentionally keeps ordinary Control UI/webchat sends on webchat and has no opt-in mirroring config; the exact feature PR is closed unmerged, and the broader force-origin issue does not replace this narrower product decision.

## What This Changes

Request an opt-in setting that mirrors Control UI/webchat-authored replies to the session’s last active external channel while preserving the default webchat-only behavior.

## Best Possible Solution

Keep this as the canonical maintainer product decision for explicit opt-in webchat-to-external mirroring, with privacy/stale-route safeguards, config schema/docs, and focused route tests if maintainers approve the behavior.

## Reproduction Assessment

Yes, at source level: current main has tests and gates showing ordinary webchat/UI turns do not inherit external delivery routes, and the requested config key is absent. I did not run a live Discord/Telegram/Web UI flow in this read-only review.

## Solution Assessment

Unclear. An opt-in config flag is safer than restoring implicit mirroring, but maintainers still need to choose the exact product/security model versus alternatives such as explicit per-send delivery or the broader force-origin policy.
