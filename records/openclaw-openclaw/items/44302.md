## Label Justifications

- `P2`: This is a meaningful cross-channel workflow gap with limited blast radius, but it is an additive feature and policy decision rather than an urgent outage.
- `impact:message-loss`: Replies can remain visible only in WebChat when the operator expected Discord, Telegram, WhatsApp, or another external channel to receive them.
- `impact:security`: The requested feature intentionally bridges internal WebChat replies to external chat targets and needs explicit privacy boundaries.

## Summary

Keep open: current main and the latest release still intentionally keep ordinary WebChat replies on the internal WebChat path unless explicit external delivery is requested, and the proposed opt-in mirroring key is absent. The closed related PRs are useful prior art, but none merged this feature, so the remaining work is a maintainer product/privacy decision rather than conservative cleanup.

## What This Changes

Add a default-off session config option that mirrors Control UI/WebChat-originated agent replies to the session's last established external channel.

## Best Possible Solution

Define a narrow, documented, default-off session policy that mirrors only completed WebChat replies to a valid persisted external route after privacy, stale-route, duplicate-delivery, failure logging, and sendPolicy semantics are approved.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/config request. Source, tests, current main, latest release, and linked issue context give high confidence that ordinary WebChat replies intentionally stay in WebChat today.

## Solution Assessment

Unclear as a final implementation path. A default-off session policy is plausible, but maintainers still need to approve cross-channel mirroring, stale-route, duplicate-delivery, sendPolicy, and privacy semantics.
