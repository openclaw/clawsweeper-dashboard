## Summary

Keep open: current main and v2026.5.22 still lack the requested opt-in WebChat-to-external-channel mirroring, and the related implementation PRs were closed unmerged. The remaining work is a product/security decision because it deliberately bridges internal Control UI replies to external chat targets.

## What This Changes

Request a default-off config option that mirrors Control UI/WebChat replies to the session’s last active external channel when an external route exists.

## Best Possible Solution

Define a narrow, documented, default-off session policy that mirrors only completed WebChat replies to a valid persisted external route after privacy, stale-route, duplicate-delivery, failure, and sendPolicy semantics are approved.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an additive feature/config request. Source, tests, release-tag inspection, and linked issue context give high confidence that ordinary WebChat replies intentionally stay in WebChat today.

## Solution Assessment

Unclear as a final implementation path. A default-off policy is plausible, but maintainers still need to approve cross-channel mirroring, stale-route, duplicate-delivery, failure, sendPolicy, and privacy semantics.
