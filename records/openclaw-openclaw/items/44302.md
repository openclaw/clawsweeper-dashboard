## Label Justifications

- `P2`: This is a normal-priority channel workflow feature with real but bounded multi-device delivery impact.
- `impact:message-loss`: Users relying on mobile continuity can miss external-channel replies when a response is authored from WebChat only.
- `impact:security`: The requested mirroring can intentionally copy WebChat content into an external transport, so stale or unintended routes are a sensitive-data boundary.

## Summary

Keep open. Current main and the latest release still keep ordinary WebChat/Control UI sends on WebChat unless explicit external delivery is requested, and there is no opt-in mirroring config; related mirroring PRs are closed unmerged while https://github.com/openclaw/openclaw/issues/54531 remains a broader overlapping policy discussion.

## What This Changes

Request an optional config-controlled mode that mirrors WebChat-authored replies to the session's last established external channel while leaving default delivery unchanged.

## Best Possible Solution

Keep this as a maintainer-approved core routing feature only if the design specifies default-off config, route freshness and permission safeguards, schema/docs, and focused tests for webchat-only, established external route, stale route, and explicit-delivery paths.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` route ordinary WebChat turns as internal and the config schema lacks the requested mirroring key. I did not run a live Discord or Telegram WebChat flow in this read-only review.

## Solution Assessment

Unclear. A default-off opt-in flag is safer than restoring implicit mirroring, but maintainers still need to choose the exact product/security model versus explicit per-send delivery or the broader force-origin policy.
