## Summary

Keep open. Current main still treats ordinary WebChat/Control UI replies as internal unless explicit external delivery is requested, the proposed default-off mirroring config key is absent, and the related implementation attempts are closed unmerged; this remains a product and security policy decision.

## What This Changes

The issue requests an opt-in config flag that mirrors WebChat/Control UI-originated replies to the session’s last established external channel for multi-device workflows.

## Best Possible Solution

Define a default-off mirroring policy with explicit stale-route, no-route, authorization, and privacy safeguards, then implement it in the core reply-routing/config surface with focused tests, docs, and live channel proof.

## Reproduction Assessment

Yes, at source level. Current routing code and tests intentionally keep WebChat-originated replies local unless explicit delivery is set, and the requested config key is absent.

## Solution Assessment

Unclear until maintainers choose the policy. A default-off explicit setting is plausible, but the implementation must define stale-route, authorization, no-route, and privacy behavior before a repair PR is safe.
