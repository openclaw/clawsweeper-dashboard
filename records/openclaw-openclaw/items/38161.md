## Summary

Keep this PR open: current main does not implement the requested hook/session-memory controls, and prior reviewer comments from darfaz and dknoodle support the runtime hardening direction, but the current head is still not merge-ready because the public docs show a save-blocking pattern that runs too late, the changelog is missing, and a few comments/docs overstate filename uniqueness semantics.

## What This Changes

This PR adds `postHookActions`, session-memory save suppression/replacement controls, random-suffixed memory filenames, slug-generation tool disabling, related docs, and tests.

## Best Possible Solution

Repair the PR branch in place with corrected hook-ordering docs, precise filename-uniqueness wording, a changelog entry, and stale comment cleanup, then proceed to maintainer review and validation.

## Reproduction Assessment

Yes. Source inspection of the PR head shows session-memory queues its post-action before the docs example queues its own, and the shared drainer executes the queued actions in push order.

## Solution Assessment

No, not as written. The runtime direction is plausible and additive, but the public docs need to tell hook authors to set save flags before session-memory's post-action runs, and the changelog/comment polish should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Set save flags before session-memory's post-action runs:** `docs/automation/hooks.md:207-211`
  - body: This example queues `blockSessionSave` inside a later post-hook action. Since session-memory queues its own post-action while its handler runs and the drainer executes actions in push order, session-memory checks the flag before this example sets it and leaves the file in place. Set `event.context.blockSessionSave` directly in the handler, or document the stricter ordering requirement.
  - confidence: 0.94
- **[P2] Add the required changelog entry:** `docs/automation/hooks.md:180`
  - body: This PR adds user-facing hook/session-memory behavior and public docs, but the branch does not update `CHANGELOG.md`. Repo policy requires user-facing `feat`, `fix`, and `perf` changes to add an active-version changelog entry.
  - confidence: 0.9
- **[P3] Avoid promising collision-free random filenames:** `docs/automation/hooks.md:180`
  - body: The docs say the 8-hex suffix guarantees uniqueness, but the implementation is a random 32-bit suffix without exclusive-create plus retry. Collisions are very unlikely, not impossible, so describe this as probabilistic collision resistance instead of a guarantee.
  - confidence: 0.91
- **[P3] Clarify sessionSaveContent filename semantics:** `docs/automation/hooks.md:189`
  - body: The table says `sessionSaveContent` still derives the filename slug from the original transcript, but the pre-set path skips transcript loading and LLM slug generation and falls back to the timestamp slug. Split the pre-set and late-set semantics so the docs match the implementation.
  - confidence: 0.87
- **[P3] Update the suffix-width comment:** `src/hooks/bundled/session-memory/handler.test.ts:268-270`
  - body: The test comment still says filenames include a 4-char suffix, while the assertion below requires eight hex characters and the implementation slices eight. Update the comment so the tested entropy budget is clear.
  - confidence: 0.97
