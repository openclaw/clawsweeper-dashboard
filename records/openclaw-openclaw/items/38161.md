## Summary

Keep open. Current main does not contain this work, and the PR still appears useful, but the latest branch is not merge-ready because its public hook docs teach a save-blocking pattern that runs too late, and the user-facing feature lacks the required changelog entry. Prior discussion is valuable: reviewers had converged on the runtime/security direction after several fixes, and #38162 and #64181 are complementary rather than replacements.

## What This Changes

The PR adds internal-hook post actions, session-memory save blocking/content replacement, random-suffixed memory filenames, slug-generator hardening with disabled tools/random run IDs, and related docs/tests with a few incidental formatting-only test changes.

## Best Possible Solution

Repair the PR branch in place: make the hook docs match the actual push-order drain contract, soften random-suffix uniqueness wording, update the stale suffix-width comment, add the required Unreleased changelog entry with contributor credit, then rerun targeted hook/session-memory validation before maintainer review decides merge ordering with #38162 and #64181.

## Reproduction Assessment

Yes. Source inspection reproduces the docs bug: session-memory queues its post-action while its handler runs, a later managed hook queues after it, and drainPostHookActions executes the snapshot in push order, so setting blockSessionSave inside the later post-action happens after session-memory already checked the flag.

## Solution Assessment

No, not as written. The runtime direction is additive and mostly consistent with the discussion, but the public docs and changelog must be corrected before this is the narrowest maintainable solution for hook authors and users.

- **[P2] Set save flags before the session-memory post-action runs:** `docs/automation/hooks.md:207-210`
  - body: This example queues `blockSessionSave` inside a later post-hook action.   - confidence: 0.92

  - confidence: 0.88

  - body: The docs say the 8-hex suffix guarantees uniqueness, but it is a random 32-bit suffix without exclusive-create plus retry. That makes collisions very unlikely, not impossible; describe it as probabilistic collision resistance instead.
