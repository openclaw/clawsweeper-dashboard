## Summary

Keep this PR open. Current main does not implement the requested hook post-actions, session-memory save controls, random-suffixed filenames, or slug-generator tool hardening, so the PR is not obsolete; however, the branch still has a blocking docs correctness issue around post-hook ordering plus two small wording nits before merge.

## What This Changes

The PR adds optional internal-hook post actions, session-memory `blockSessionSave`/`sessionSaveContent` handling, random-suffixed memory filenames, slug-generator `disableTools: true`, and docs/tests for those behaviors.

## Best Possible Solution

Fix the docs so late-registered save policy hooks mutate `event.context` during their normal handler, not from a later `postHookActions` callback, then soften the random-suffix wording and stale test comment before maintainer review decides whether to land this PR as one unit or coordinate it with #38162 and #64181.

## Reproduction Assessment

Yes. Source inspection reproduces both the current-main gap and the PR docs bug: the PR drains post-hook callbacks in push order, while the docs example queues the save-block mutation after session-memory has already queued its own callback.

## Solution Assessment

No, not as written. The runtime design is broadly additive and maintainable, but the public docs must match the drain ordering contract so security/privacy hook authors do not copy a pattern that fails to retract session-memory writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Set save flags before the session-memory post-action runs:** `docs/automation/hooks.md:207-210`
  - body: This example queues `blockSessionSave` inside a later post-hook action, but session-memory queues its own post-hook callback when its handler runs. Since `drainPostHookActions` executes callbacks in push order, a managed hook registered after session-memory will set the flag after the session-memory callback has already checked it, so the file is not retracted. Show direct `event.context.blockSessionSave = true` mutation in the handler, or change the runtime contract if post-action policy mutation is intended to work.
  - confidence: 0.9
- **[P3] Avoid promising collision-free random filenames:** `docs/automation/hooks.md:180`
  - body: The docs say the 8-hex suffix guarantees filename uniqueness, but it is a 32-bit random suffix without exclusive-create plus retry. That makes collisions very unlikely, not impossible; please describe it as probabilistic collision resistance and align the nearby implementation comment if needed.
  - confidence: 0.9
- **[P3] Update the suffix-width comment:** `src/hooks/bundled/session-memory/handler.test.ts:268-270`
  - body: The comment still says the filename includes a 4-char suffix, while the assertion below requires eight hex characters and the implementation slices eight. Update the comment so the tested entropy budget is clear.
  - confidence: 0.97
