## Summary

Keep this PR open: current main does not implement the proposed hook/session-memory behavior, and the PR still has narrow merge blockers in its public docs/changelog surface. The runtime direction looks additive, and prior discussion includes positive security review context, but the documented post-hook example would not actually block the session-memory write as written.

## What This Changes

The PR adds `postHookActions` internal-hook draining, session-memory save blocking/content replacement with random-suffixed filenames, disables tools for LLM slug generation, and updates related docs/tests with some incidental formatting-only changes.

## Best Possible Solution

Repair the PR branch in place: make the hook docs match the actual push-order drain contract, add the required Unreleased changelog entry, soften random-suffix uniqueness wording in docs/comments, fix the stale suffix-width test comment, then rerun targeted validation before maintainer review decides merge ordering with the related hook/slug PRs.

## Reproduction Assessment

Yes. Diff inspection is enough to reproduce the main docs bug: session-memory queues its post-action while its handler runs, the docs example queues a later post-action, and the exported drainer runs the snapshot in push order, so the example sets `blockSessionSave` too late.

## Solution Assessment

No, not as written. The runtime implementation is a plausible additive direction, but the public docs and changelog must be corrected before this is the narrowest maintainable solution for hook authors and users.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Set save flags before session-memory's post-action runs:** `docs/automation/hooks.md:207-211`
  - body: This example queues `blockSessionSave` inside a later post-hook action. Since session-memory queues its own post-action while its handler runs, `drainPostHookActions` executes that callback first, so it checks the save flags before this example sets them and leaves the file in place. Mutate `event.context.blockSessionSave` directly in the handler, or change the runtime contract if post-action policy mutation is intended to work.
  - confidence: 0.93
- **[P2] Add the required changelog entry:** `docs/automation/hooks.md:180`
  - body: This PR adds user-facing hook/session-memory behavior and public docs, but the diff does not update `CHANGELOG.md`. Repo policy requires user-facing `feat`, `fix`, and `perf` changes to add an active-version changelog entry; credit the contributor handle rather than a bot or maintainer placeholder.
  - confidence: 0.9
- **[P3] Avoid promising collision-free random filenames:** `docs/automation/hooks.md:180`
  - body: The docs say the 8-hex suffix guarantees uniqueness, but the implementation is a random 32-bit suffix without exclusive-create plus retry. Collisions are very unlikely, not impossible, so describe this as probabilistic collision resistance instead of a guarantee.
  - confidence: 0.9
- **[P3] Update the suffix-width comment:** `src/hooks/bundled/session-memory/handler.test.ts:265-267`
  - body: The test comment still says filenames include a 4-char suffix, while the assertion below requires eight hex characters and the implementation slices eight. Update the comment so the tested entropy budget is clear.
  - confidence: 0.97
