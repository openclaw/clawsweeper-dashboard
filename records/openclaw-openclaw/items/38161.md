## Summary

Keep open. Current main still does not implement the hook post-action mechanism, session-memory save controls, random-suffixed memory filenames, or slug-generator tool disabling, so this PR is not obsolete; however, the PR branch still has narrow docs/comment/changelog blockers before merge.

## What This Changes

The PR adds optional internal hook post actions, session-memory save blocking/content replacement with random-suffixed filenames, slug-generator hardening, hooks documentation, and related tests with a few incidental formatting-only changes.

## Best Possible Solution

Repair the PR branch in place: make the hook docs match the push-order post-action contract, describe random filename suffixes as probabilistic collision resistance, correct the stale suffix-width comment, add the required Unreleased changelog entry with contributor credit, then rerun targeted hook/session-memory validation before maintainer review decides merge order with #38162 and #64181.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction for the docs bug: session-memory queues its post action while its handler runs, a later managed hook queues after it, and drainPostHookActions executes the snapshot in push order, so setting blockSessionSave inside the later post action happens after session-memory already checked the flag.

## Solution Assessment

No, not as written. The runtime direction is additive and appears largely aligned with prior review, but the public docs and changelog need correction before this is the narrowest maintainable change for hook authors and users.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Set save flags before session-memory post-actions run:** `docs/automation/hooks.md:207-210`
  - body: This example queues `blockSessionSave` inside a later post-hook action. A managed hook registered after `session-memory` will enqueue this callback after session-memory's own post action, and `drainPostHookActions` runs the snapshot in push order, so the session-memory callback checks before the flag is set and leaves the file in place. Mutate `event.context.blockSessionSave` directly in the handler, or document that post actions cannot change policy for earlier post actions.
  - confidence: 0.92
- **[P2] Add the required changelog entry:** `docs/automation/hooks.md:180`
  - body: This PR adds user-facing hook/session-memory behavior and public docs, but the diff does not update `CHANGELOG.md`. Repo policy requires user-facing `feat`, `fix`, and `perf` changes to add an active-version changelog entry before merge; credit the contributor handle rather than a bot or maintainer placeholder.
  - confidence: 0.88
- **[P3] Avoid promising collision-free random filenames:** `docs/automation/hooks.md:180`
  - body: The docs say the 8-hex suffix guarantees uniqueness, but it is a random 32-bit suffix without exclusive-create plus retry. That makes collisions very unlikely, not impossible, so describe it as probabilistic collision resistance.
  - confidence: 0.9
- **[P3] Clarify pre-set custom-content filename behavior:** `docs/automation/hooks.md:188-189`
  - body: The table says `sessionSaveContent` keeps the filename slug derived from the original transcript, but the handler skips transcript loading and LLM slug generation when custom content is pre-set, then falls back to the timestamp slug plus suffix. Split the pre-set and late-set behavior so privacy-critical users do not expect a transcript-derived slug in the pre-set path.
  - confidence: 0.86
- **[P3] Update the suffix-width comment:** `src/hooks/bundled/session-memory/handler.test.ts:268-270`
  - body: The comment still says the filename includes a 4-char suffix, while the assertion below requires eight hex characters and the implementation slices eight. Update the comment so the tested entropy budget is clear.
  - confidence: 0.97
