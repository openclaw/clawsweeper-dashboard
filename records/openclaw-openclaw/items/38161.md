## Summary

Keep open: current main does not implement the new hook/session-memory controls, and the PR is still useful but not merge-ready because PR-head source/docs still show a policy-ordering bug and missing real behavior proof.

## What This Changes

The PR adds an optional postHookActions queue, session-memory block/content controls, random-suffixed memory filenames, LLM slug tool disabling, docs/API baseline updates, and tests.

## Best Possible Solution

Repair the PR branch so session-memory observes the final documented save policy, make the docs/logging/changelog precise, and require real OpenClaw behavior proof before maintainer review proceeds.

## Reproduction Assessment

Yes. Source inspection of PR head shows session-memory queues its post-action before later handlers can queue their own, and the shared drainer executes queued actions in push order.

## Solution Assessment

No, not as written. The additive runtime direction is plausible, but the save-policy ordering contract, privacy warning, docs, changelog, and real behavior proof need repair before this is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Run save reconciliation after policy post-actions settle:** `src/hooks/bundled/session-memory/handler.ts:411`
  - body: `session-memory` queues its reconciliation callback while its handler runs, but the drainer executes `postHookActions` in push order. A later hook that queues a callback to set `blockSessionSave` or `sessionSaveContent` runs after this callback, so the file is left written despite the documented policy. Add a final policy phase/priority or require direct handler-time mutation and update docs/tests.
  - confidence: 0.94
- **[P2] Gate the LLM-egress warning on an actual slug call:** `src/hooks/bundled/session-memory/handler.ts:421-441`
  - body: The late-block privacy warning fires whenever `sessionContent` exists, but slug generation is skipped when tests are active, config is missing, or `llmSlug` is false. In those common configs the transcript was not sent to a provider, so this logs a false privacy incident. Track whether `generateSlugViaLLM` actually ran before claiming provider egress.
  - confidence: 0.9
- **[P2] Add the required changelog entry:** `docs/automation/hooks.md:181`
  - body: This branch adds user-facing hook/session-memory behavior and public docs, but the PR file list does not include `CHANGELOG.md`. Repo policy requires user-facing `feat`, `fix`, and `perf` changes to add an active-version changelog entry.
  - confidence: 0.9
- **[P3] Avoid promising impossible filename uniqueness:** `docs/automation/hooks.md:181`
  - body: The docs say the 8-hex random suffix guarantees uniqueness, but the implementation does not use exclusive create plus retry. A 32-bit suffix makes collisions unlikely, not impossible, so describe this as probabilistic collision resistance or implement exclusive-create retry semantics.
  - confidence: 0.91
- **[P3] Clarify pre-set sessionSaveContent filename semantics:** `docs/automation/hooks.md:190`
  - body: The table says `sessionSaveContent` still derives the filename slug from the original transcript, but the pre-set path skips transcript loading and LLM slug generation and falls back to the timestamp slug plus suffix. Split pre-set and late-set semantics so plugin authors know when transcript-derived slugs are retained.
  - confidence: 0.87
- **[P3] Update the suffix-width comment:** `src/hooks/bundled/session-memory/handler.test.ts:411-413`
  - body: The test comment still says filenames include a 4-char suffix, while the assertion below requires eight hex characters and the implementation slices eight. Update the comment so the tested entropy budget is clear.
  - confidence: 0.97
