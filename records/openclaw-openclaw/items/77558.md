## Summary

Keep open: the PR targets a valid config RPC persistence bug, but it is not mergeable yet because the normal public writer still returns void, include-backed writes still fall back to an unpersisted value, and release/test coverage is incomplete.

## What This Changes

The PR adds `persistedConfig` to config mutation results and changes `config.set`, `config.patch`, and `config.apply` responses to redact that persisted value instead of the submitted in-memory config.

## Best Possible Solution

Propagate `{ persistedHash, persistedConfig }` through the exported writer and include fast path, have all gateway write responses use the canonical persisted value, then add focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, from source inspection: current main writes through transforms that can change persisted disk state while the three gateway write handlers respond from `parsed.config` or `validated.config`. The PR’s root-writer failure is also source-reproducible because the public writer still returns `Promise<void>`.

## Solution Assessment

No: the direction is the narrow maintainable fix, but this implementation stops one layer too early. The safer solution is to make the persisted write result a real contract through `src/config/io.ts`, the include fast path, mutation helpers, and gateway response tests.

  - body: `replaceConfigFile` now dereferences `writeResult.persistedConfig`, but the default exported `writeConfigFile` is still typed as `Promise<void>` and does not return the low-level write result. That can still include managed keys stripped before the include file is written, so include-backed responses can remain non-canonical; return the refreshed source config from the fast path.
  - confidence: 0.84

  - body: This changes user-facing config RPC response payloads, but the PR does not touch `CHANGELOG.md`. Add a single-line active-version fix entry so release notes cover the behavior change.
