## Summary

Keep open: the linked bug is real, and the PR is pointed at the right behavior, but the patch is not mergeable because it consumes `persistedConfig` from a public writer that still returns `void` and it leaves the include fast path reporting the unpersisted input.

## What This Changes

The PR captures config write results through the mutate and gateway config write flow, then redacts `persistedConfig` in `config.set`, `config.patch`, and `config.apply` responses.

## Best Possible Solution

Propagate `{ persistedHash, persistedConfig }` from the exported config writer and the include fast path, then have the three gateway write responses use that canonical persisted value with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes from source inspection: current main writes through transforms that can change the persisted config, while the gateway handlers respond from `parsed.config` or `validated.config`. The proposed patch also has a source-level failure path because the normal root writer currently returns `void`.

## Solution Assessment

No: the direction is the narrow maintainable fix, but this implementation stops one layer too early and leaves include-backed writes non-canonical. The safer fix is to make the persistence result a real contract through `src/config/io.ts`, `src/config/mutate.ts`, and `commitGatewayConfigWrite`.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Return the persisted result from the public writer:** `src/config/mutate.ts:274`
  - body: `replaceConfigFile` normally calls the exported `writeConfigFile`, but that wrapper still returns `void` and falls off after using the low-level result for notifications. This added read will therefore see `undefined` for ordinary root config writes, so `config.set`/`config.patch`/`config.apply` can throw before responding. Return `{ persistedHash, persistedConfig }` from the public writer before consuming it here.
  - confidence: 0.95
- **[P2] Return persisted config from include writes:** `src/config/mutate.ts:266`
  - body: When `tryWriteSingleTopLevelIncludeMutation` handles an isolated `$include`, this default leaves `persistedConfig` as the unpersisted `nextConfig`. That path strips managed keys and refreshes a canonical snapshot, so include-backed `config.patch` or `config.apply` responses can still report values that were not written. Return the refreshed source config from the include fast path instead.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/config.ts:343`
  - body: This is a user-facing fix to config RPC response payloads and the PR metadata does not include `CHANGELOG.md`. OpenClaw policy requires a single-line active-version changelog entry for user-facing fixes.
  - confidence: 0.9
