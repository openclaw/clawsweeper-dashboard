## Summary

This PR is aimed at a real config API bug, but it should not merge as written because the normal production path would read `persistedConfig` from a writer wrapper that still returns `void`.

## What This Changes

The PR threads a `persistedConfig` value from config mutation writes through gateway config write handlers and uses it in `config.set`, `config.patch`, and `config.apply` responses.

## Best Possible Solution

Preserve the PR direction, but return the real persisted write result through the production wrapper and include path, then cover gateway responses for persisted/read-back config differences.

## Reproduction Assessment

Yes, this is source-reproducible: current main handlers return pre-write config while the write path separately computes `persistedConfig`. I did not run live tests because this review was constrained to read-only inspection.

## Solution Assessment

No. The direction is right, but the patch consumes a persisted result from an exported writer that still returns `void`, and the include fast path still defaults to the requested config instead of read-back persisted state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Return the persisted result from the exported writer:** `src/config/mutate.ts:268-276`
  - body: `replaceConfigFile` normally calls the exported `writeConfigFile` from `src/config/io.ts`, but that wrapper is still typed as `Promise<void>` and never returns the inner `{ persistedHash, persistedConfig }`. After a normal write, `writeResult` is therefore `undefined`, so `config.set`/`config.patch`/`config.apply` can throw after persisting instead of responding. Return the inner write result from the exported wrapper or route this code through a writer that already returns it.
  - confidence: 0.92
- **[P2] Use the include readback as persistedConfig:** `src/config/mutate.ts:266`
  - body: When `tryWriteSingleTopLevelIncludeMutation` succeeds, this initializer leaves `persistedConfig` as the requested config even though the include writer refreshed a snapshot from disk and may have stripped managed keys such as `plugins.installs`. That keeps the response inaccurate for isolated top-level `$include` writes, which is the same class of bug this PR is fixing. Return the refreshed source config from the include path instead.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/config.ts:343`
  - body: This changes user-visible gateway config API responses for `config.set`, `config.patch`, and `config.apply`, so repo policy expects a `CHANGELOG.md` entry for the fix. Add a concise single-line fix entry in the active release section.
  - confidence: 0.82
