## Summary

This PR targets a real config write-response bug, but it should stay open because the new `persistedConfig` fallback is still wrong when the single-top-level include writer handles the write and skips `writeConfigFile`.

## What This Changes

The PR threads `writeConfigFile().persistedConfig` through config mutation helpers and gateway `config.set`, `config.patch`, and `config.apply` responses.

## Best Possible Solution

Keep the PR open and finish it by returning the actual persisted logical config for both root writes and include writes, then add focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main responds with the in-memory parsed/validated config while the writer returns a transformed persisted config, and the existing include-write test proves one write path can differ from the submitted `nextConfig`.

## Solution Assessment

No, not as submitted. Threading `writeConfigFile().persistedConfig` is the right direction, but the include fast path must return the refreshed persisted config instead of defaulting to the untransformed input.
