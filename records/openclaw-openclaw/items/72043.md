## Summary

Keep open: current main still lacks the requested Gateway recovery-notice policy, and the recovery paths still enqueue prompt-facing system events and can send the same notice out-of-band. The linked items do not supersede this because they cover a channel-specific WhatsApp filter, narrower recovery trigger work, or the opposite request for more visible context-reset signals.

## What This Changes

Request adds an operator-controlled Gateway config knob to suppress last-known-good recovery notices from prompt and outbound channel surfaces while preserving recovery and restart continuations.

## Best Possible Solution

Add a Gateway-owned, documented config policy with current behavior as the default that gates only recovery-notice delivery to prompt and outbound channel surfaces, while preserving recovery, restart continuations, wake behavior, and operator logs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by triggering startup or reload last-known-good recovery, or by consuming a `config-auto-recovery` restart sentinel with routing; current main then queues the recovery message into the agent session and may deliver it outbound.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice`-style policy is the narrowest maintainable fix because it centralizes operator intent for this event class; channel-specific filters such as #49579 would not cover both config recovery and restart-sentinel delivery.
