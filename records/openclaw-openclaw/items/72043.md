## Summary

Keep open: current `main` still lacks a Gateway recovery-notice policy, and last-known-good recovery can still queue the technical recovery notice into the main agent session and deliver restart-sentinel notices back to the originating channel.

## What This Changes

Add a documented `gateway.systemEvents.recoveryNotice` config option to control whether config-recovery notices reach prompt-facing and outbound channel surfaces.

## Best Possible Solution

Add a Gateway-owned, documented policy that preserves current behavior by default while allowing operators to suppress only recovery-notice prompt and outbound delivery, leaving recovery mechanics and restart continuations unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by triggering last-known-good startup/reload recovery or consuming a config auto-recovery restart sentinel with routing; current code then queues the notice into the main session and may deliver it outbound.

## Solution Assessment

Yes. A central `gateway.systemEvents.recoveryNotice` policy is the narrowest maintainable fix because channel-specific prompt filters do not cover both config-recovery notice paths and restart-sentinel delivery.
