## Summary

Keep open. Current main still lacks an operator-controlled Gateway recovery-notice policy: last-known-good recovery notices are still queued into the main agent prompt path and routed restart-sentinel notices still go out over the originating channel. Recent recovery-detail work improved the wording and diagnostics, but did not add the requested suppression knob.

## What This Changes

Request a `gateway.systemEvents.recoveryNotice` Gateway config option to choose whether recovery notices are user-visible, log-only, or fully suppressed.

## Best Possible Solution

Add a Gateway-owned config policy with a current-behavior default that gates both prompt-visible recovery system events and out-of-band restart-sentinel recovery notices, while leaving last-known-good recovery and restart continuations unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger a non-plugin last-known-good recovery and current main will enqueue a main-session recovery system event; for routed restart sentinels, it also sends the same notice through outbound channel delivery.

## Solution Assessment

Yes. A single `gateway.systemEvents.recoveryNotice` policy is the narrow maintainable fix because channel-specific prompt filters such as #49579 would not cover startup/reload recovery notices or the out-of-band restart-sentinel path.
