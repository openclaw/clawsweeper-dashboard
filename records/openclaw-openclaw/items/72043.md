## Summary

Keep open: current main still lacks the requested Gateway recovery-notice policy, and both config recovery paths can still feed technical recovery text into the main agent turn or originating channel.

## What This Changes

Request adds a `gateway.systemEvents.recoveryNotice` knob to control prompt, outbound, and log handling for Gateway config-recovery notices.

## Best Possible Solution

Add a Gateway-owned policy with current behavior as the default, gating only recovery-notice prompt and outbound delivery while preserving recovery, restart continuations, wake behavior, and operator diagnostics.

## Reproduction Assessment

Yes. A code-level reproduction exists by triggering last-known-good startup/reload recovery or consuming a `config-auto-recovery` restart sentinel with routing; current main then queues the notice into the main session and may deliver it outbound.

## Solution Assessment

Yes. A central Gateway config policy is the narrowest maintainable solution because channel-specific prompt filters like #49579 do not cover both startup/reload recovery and restart-sentinel delivery.
