## Summary

Keep open. Current main still has no operator-controlled recovery-notice policy, and the recovery notice remains visible through both the prompt system-event path and the restart-sentinel outbound channel path; related #49579 and #39307 do not supersede this narrower Gateway config request.

## What This Changes

The issue requests a new `gateway.systemEvents.recoveryNotice` config setting that preserves current behavior by default while allowing operators to suppress config-recovery notices from prompt-visible and out-of-band channel surfaces.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates config-recovery prompt events and restart-sentinel outbound recovery notices, while leaving last-known-good recovery, restart continuations, and operator logs intact.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger last-known-good config recovery and current startup/reload code queues a main-session system event; for routed config-auto-recovery restart sentinels, startup also sends the notice outbound.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice` knob is the narrow maintainable solution because it centralizes operator policy for this recovery event class; channel-specific filters like #49579 would not cover the outbound restart-sentinel notice path.
