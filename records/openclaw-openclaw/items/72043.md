## Summary

Keep open: current main still lacks the requested Gateway recovery-notice policy, while startup/reload recovery and config-auto-recovery restart sentinels still put technical recovery text on prompt-facing and outbound channel surfaces.

## What This Changes

Request a Gateway config knob that lets operators keep, log-only, or suppress config-recovery notices for customer-facing bots.

## Best Possible Solution

Add a documented Gateway-owned policy with current behavior as the default, gating only recovery-notice delivery to prompt and outbound channel surfaces while preserving recovery, restart continuations, and operator logs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by triggering last-known-good startup/reload recovery or consuming a `config-auto-recovery` restart sentinel with routing; current main then queues the notice into the main session and may deliver the same message outbound.

## Solution Assessment

Yes. A central `gateway.systemEvents.recoveryNotice`-style policy is the narrowest maintainable fix because channel-specific filters do not cover both config-recovery paths and restart-sentinel delivery.
