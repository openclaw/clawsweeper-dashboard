## Summary

Keep open: current main still has no Gateway recovery-notice policy knob, and the recovery paths still queue operator-facing notices into the main session and restart-sentinel outbound flow. This is a valid but public config/product-policy feature request, so it should get maintainer direction before implementation.

## What This Changes

Request adding `gateway.systemEvents.recoveryNotice` modes to control whether recovery notices are injected into prompts, delivered out-of-band, or only logged.

## Best Possible Solution

Add a documented Gateway-owned policy, defaulting to current behavior, that gates only recovery-notice prompt and outbound delivery while preserving recovery, restart continuations, and diagnostic logs.

## Reproduction Assessment

Yes for the current behavior: source inspection shows last-known-good startup/reload recovery queues the recovery notice into the main session, and restart-sentinel startup queues and can deliver the notice outbound. I did not run a live Telegram scenario, but the current-main code path is clear.

## Solution Assessment

Unclear until maintainers approve the public config surface. If accepted, a central Gateway policy is narrower and more maintainable than channel-specific prompt filters such as the related WhatsApp banner PR.
