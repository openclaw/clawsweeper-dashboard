## Summary

Keep open. Current main still has no operator-controlled recovery-notice policy, and the same recovery paths still queue prompt-facing system events or deliver restart-sentinel notices when routing survives. The related open items do not supersede this because they cover channel-specific banner filtering or the opposite user-visible reset-signaling direction.

## What This Changes

Request to add a `gateway.systemEvents.recoveryNotice` config mode that can suppress last-known-good recovery notices from prompt and outbound channel surfaces while preserving recovery and restart continuations.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates only recovery notice delivery to prompt and outbound channel surfaces, while preserving recovery mechanics, restart continuations, session wake behavior needed for continuations, and operator logs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger startup/reload last-known-good recovery or consume a `config-auto-recovery` restart sentinel on current main, and the source plus tests show the recovery message is queued into the main session and may be delivered outbound when a route resolves.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice`-style policy is the narrowest maintainable direction because it centralizes operator intent for this event class; a channel-specific filter like #49579 would not cover both startup/reload recovery and restart-sentinel outbound delivery.
