## Summary

Keep open. Current `main` still has no `gateway.systemEvents.recoveryNotice` config surface, while both recovery paths still expose recovery notices through prompt-facing system events and the restart-sentinel outbound notice path. Related #49579 and #39307 are useful context but do not supersede this narrower Gateway policy knob.

## What This Changes

Add a Gateway config policy to control whether last-known-good config recovery notices are prompt/channel-visible, logged only, or disabled, while preserving recovery and restart continuations.

## Best Possible Solution

Add a documented Gateway-owned policy with current behavior as the default, gate config-recovery prompt events and config-auto-recovery restart-sentinel outbound notices through that policy, and leave last-known-good recovery, continuation delivery, and existing recovery mechanics intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger last-known-good startup or reload recovery and `enqueueConfigRecoveryNotice` queues a main-session system event; trigger gateway-run config auto-recovery and the restart-sentinel startup task enqueues a wake event and sends an outbound notice when routing resolves.

## Solution Assessment

Yes. A central `gateway.systemEvents.recoveryNotice` policy is the narrow maintainable fix because channel-specific prompt filters like #49579 do not cover the restart-sentinel outbound path, while #39307 tracks a different audience that wants more visible recovery context.

Implement the Gateway recovery-notice policy requested by https://github.com/openclaw/openclaw/issues/72043. Add `gateway.systemEvents.recoveryNotice` with modes `"user"` (default/current behavior), `"log"`, and `"off"` in the Gateway config schema/type/help/labels/docs surfaces. Gate the last-known-good config recovery notice so `"user"` preserves prompt and outbound behavior, `"log"` suppresses prompt/outbound notices and emits one `gateway/restart-sentinel` info log, and `"off"` suppresses both prompt/outbound notices and the extra policy log. Cover both direct startup/reload `enqueueConfigRecoveryNotice` and config-auto-recovery restart-sentinel wake/outbound delivery. Do not suppress or change `payload.continuation`, last-known-good recovery itself, session-delivery queues, generic system events, or channel-specific filtering. Add focused tests for the three modes, update `docs/gateway/configuration.md` and related troubleshooting copy if needed, regenerate/check config docs metadata if required, and add a user-facing `CHANGELOG.md` entry under the active `### Changes` section with reporter attribution.
