## Summary

Keep open. Current main still has no `gateway.systemEvents.recoveryNotice` policy, and the config-recovery notice remains user-visible through the main-session system-event path and the restart-sentinel outbound delivery path. The related open items track different behavior: #49579 is channel-specific WhatsApp banner filtering, and #39307 asks for more user-visible context reset disclosure rather than suppressing recovery internals.

## What This Changes

Add a Gateway config knob to control whether last-known-good config recovery notices are sent to the prompt, sent out-of-band to the originating channel, logged only, or suppressed.

## Best Possible Solution

Add a Gateway-owned, documented config policy with current behavior as the default that gates config-recovery prompt events and restart-sentinel outbound recovery notices, while leaving last-known-good recovery, restart continuations, and operator diagnostics intact.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger last-known-good config recovery and current main queues a main-session system event; for routed restart sentinels it also delivers the notice out-of-band to the originating channel.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice` enum is the narrow maintainable direction because it centralizes operator policy for this recovery event class; channel-specific filters like #49579 would not cover the restart-sentinel outbound path.
