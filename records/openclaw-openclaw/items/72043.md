## Summary

Keep open: current main still lacks a Gateway recovery-notice policy knob, and the recovery paths still enqueue the technical notice into the main session and can deliver it outbound to the originating channel.

## What This Changes

Requests a `gateway.systemEvents.recoveryNotice` config policy so operators can keep recovery notices user-visible, log-only, or silent for customer-facing bots.

## Best Possible Solution

Add a documented Gateway-owned recovery-notice policy with the current behavior as the default, gating only prompt and outbound notice delivery while preserving recovery, restart continuations, and operator diagnostics.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by triggering last-known-good startup/reload recovery or consuming a `config-auto-recovery` restart sentinel with routing; current main then queues the notice into the main session and may deliver it outbound.

## Solution Assessment

Yes. A central `gateway.systemEvents.recoveryNotice`-style policy is the narrowest maintainable fix because channel-specific prompt filters like #49579 do not cover both config recovery and restart-sentinel delivery surfaces.
