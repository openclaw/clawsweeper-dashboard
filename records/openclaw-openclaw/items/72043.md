## Summary

Keep open. Current main still lacks an operator-controlled recovery-notice policy, and the relevant recovery paths still queue prompt-facing system events and can deliver the same restart-sentinel notice out-of-band. The related open items do not supersede this because they cover a channel-specific WhatsApp filter and the opposite user-visible context-reset request.

## What This Changes

The issue requests a Gateway config knob to suppress last-known-good recovery notices from prompt-facing and out-of-band customer channel surfaces while preserving recovery and restart continuations.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates only recovery notice delivery to prompt and outbound channel surfaces, while preserving last-known-good recovery, restart continuations, wake behavior, and operator logs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger startup or reload last-known-good recovery, or consume a `config-auto-recovery` restart sentinel with routing, and current main queues the recovery message into the agent session and may deliver it outbound.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice`-style policy is the narrowest maintainable fix because it centralizes operator intent for this event class; channel-specific filters such as #49579 would not cover both config recovery and restart-sentinel delivery.
