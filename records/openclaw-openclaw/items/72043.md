## Summary

Keep open. Current main still lacks an operator-controlled recovery-notice policy, while the startup, reload, and restart-sentinel paths still put recovery text into prompt-facing system events and can deliver the restart-sentinel notice out-of-band. The related open items do not supersede this because they cover a WhatsApp-specific banner filter and the opposite request for more visible context-reset signaling.

## What This Changes

The issue requests a Gateway config policy to control whether last-known-good config recovery notices are injected into agent prompts and delivered out-of-band.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates recovery notice delivery to prompt and outbound channel surfaces while preserving recovery mechanics, restart continuations, wake behavior, routing, and operator logs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger startup or reload last-known-good recovery, or consume a config-auto-recovery restart sentinel with routing, and current main queues the recovery message into the agent session and may deliver it outbound.

## Solution Assessment

Yes. A gateway.systemEvents.recoveryNotice-style policy is the narrowest maintainable fix because it centralizes operator intent for this event class; a channel-specific filter such as #49579 would not cover both config recovery notices and restart-sentinel delivery.
