## Summary

Keep open: current main still lacks the requested Gateway recovery-notice policy, and the relevant recovery paths still enqueue prompt-facing system events and can send the same notice out-of-band. The linked items do not supersede this because they cover narrower recovery-trigger work, a channel-specific WhatsApp filter, or the opposite request for more visible context-reset signals.

## What This Changes

The issue requests `gateway.systemEvents.recoveryNotice` so operators can keep, log-only, or suppress last-known-good recovery notices for customer-facing bots.

## Best Possible Solution

Add a Gateway-owned, documented config policy with current behavior as the default that gates only recovery-notice delivery to prompt and outbound channel surfaces, while preserving recovery and restart continuation semantics.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by triggering startup or reload last-known-good recovery, or by consuming a `config-auto-recovery` restart sentinel with routing; current main then queues the recovery message into the agent session and may deliver it outbound.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice` policy is the narrowest maintainable fix because it centralizes operator intent for this event class; channel-specific filters such as #49579 would not cover both config recovery and restart-sentinel delivery.

## Repair Work Prompt

Implement the requested `gateway.systemEvents.recoveryNotice` policy for openclaw/openclaw#72043. Add a public config field under `gateway.systemEvents.recoveryNotice` with enum values `"user" | "log" | "off"`, defaulting to `"user"` for current behavior. Gate only the last-known-good recovery notice surfaces: startup/reload `enqueueConfigRecoveryNotice`, config-auto-recovery restart-sentinel prompt enqueue/wake, and routed outbound restart-sentinel notice delivery. Preserve config recovery itself, rejected `.clobbered.*` handling, durable restart continuations, sentinel cleanup, continuation delivery, and any wake needed for an actual continuation. For `"log"`, suppress prompt/outbound notice delivery and emit a single operator log line from the recovery/restart-sentinel path; for `"off"`, suppress prompt/outbound notice delivery and extra notice logging. Update schema/types/help/labels/generated docs as required by the repo, add focused tests for all three modes, document the operator knob in Gateway configuration docs, and add a single-line `CHANGELOG.md` entry under the active `### Changes` section. Do not add channel-specific filters or change unrelated system-event behavior.
