## Summary

Keep open: current main still lacks the requested Gateway recovery-notice policy, and both recovery paths still put operator-facing text into prompt/outbound surfaces. This is a valid but public config/product-policy request, so it needs maintainer direction before an automated fix PR.

## What This Changes

Request adding a Gateway `systemEvents.recoveryNotice` config knob to control whether config recovery notices reach the prompt, outbound channel, or only logs.

## Best Possible Solution

Add a documented Gateway-owned policy, defaulting to current behavior, that gates only recovery-notice prompt/outbound delivery while preserving recovery, restart continuations, and diagnostic logs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: trigger startup/reload last-known-good recovery or consume a `config-auto-recovery` restart sentinel with routing, and current main queues the notice into the session and can deliver it outbound.

## Solution Assessment

Unclear until maintainers approve the public config surface. If the direction is accepted, a central Gateway policy is narrower and more maintainable than channel-specific prompt filters.
