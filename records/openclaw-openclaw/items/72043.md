## Summary

Keep open. Current main still lacks the requested Gateway recovery-notice policy, and the two user-visible surfaces in the report remain hardwired: last-known-good config recovery queues a main-session system event, while config-auto-recovery restart sentinels can wake the session and send an outbound channel notice. The related open items do not supersede this because #49579 is a channel-specific banner filter and #39307 asks for more visible reset context rather than operator-controlled suppression.

## What This Changes

The issue requests a `gateway.systemEvents.recoveryNotice` config policy that preserves current recovery-notice behavior by default while allowing operators to log-only or suppress recovery notices for customer-facing bots.

## Best Possible Solution

Add a Gateway-owned, documented recovery-notice policy with a current-behavior default that gates config-recovery prompt events and config-auto-recovery restart-sentinel outbound notices, while leaving last-known-good recovery, restart continuations, and operator logs intact.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger last-known-good config recovery and current main queues the main-session recovery notice; for CLI config-auto-recovery sentinels, startup also wakes the session and sends the notice outbound when a route resolves.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice`-style policy is the narrowest maintainable fix because it covers both current surfaces centrally; channel-specific filters like #49579 would not cover restart-sentinel outbound delivery and would keep adding one-off suppressions.
