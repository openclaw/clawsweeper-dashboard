## Summary

Keep open: current main still has no `gateway.systemEvents` or `recoveryNotice` config surface, and Gateway recovery paths still emit the technical recovery notice into prompt-facing and outbound channel delivery paths.

## What This Changes

Request to add a `gateway.systemEvents.recoveryNotice` policy with `user`, `log`, and `off` modes so customer-facing bots can suppress Gateway recovery notices without disabling recovery.

## Best Possible Solution

Add a documented Gateway-owned policy with current behavior as the default, gating only recovery-notice prompt and outbound delivery while preserving last-known-good recovery, restart continuations, and operator logs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by triggering startup/reload last-known-good recovery or consuming a routed `config-auto-recovery` restart sentinel; current main then queues the notice into the main session and may deliver it outbound.

## Solution Assessment

Yes. A central Gateway config policy is the narrowest maintainable solution because channel-specific filters such as #49579 do not cover both config recovery paths and restart-sentinel delivery.
