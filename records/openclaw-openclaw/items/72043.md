## Summary

Keep open: current main still lacks the requested recovery-notice policy, and source inspection shows recovery notices still enter both prompt-facing and outbound restart-sentinel surfaces. The fix would add a public Gateway config option and operator policy, so it needs maintainer direction before implementation.

## What This Changes

Request to add a Gateway `gateway.systemEvents.recoveryNotice` config policy that can suppress last-known-good recovery notices from model prompts and outbound channel messages while preserving recovery behavior.

## Best Possible Solution

Add a documented Gateway-owned policy, defaulting to current behavior, that gates only recovery-notice prompt and outbound delivery while preserving recovery, restart continuations, and diagnostic logs.

## Reproduction Assessment

Yes by source inspection: startup/reload last-known-good recovery calls `enqueueConfigRecoveryNotice`, and restart-sentinel recovery calls `enqueueRestartSentinelWake` plus `deliverRestartSentinelNotice`. I did not run a live Telegram recovery because this is read-only triage, but the current-main path is clear.

## Solution Assessment

Unclear until maintainers approve the public config surface and mode names. If accepted, a central Gateway policy is narrower and more maintainable than channel-specific prompt filters such as the open WhatsApp-only PR.
