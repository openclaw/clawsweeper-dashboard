## Summary

Keep open: current main still lacks the requested Gateway recovery-notice policy, and the recovery notice remains queued into the main session plus delivered outbound by restart-sentinel recovery when routing is available.

## What This Changes

Add a `gateway.systemEvents.recoveryNotice` config policy to control whether config auto-recovery notices are injected into agent prompt context, sent out-of-band to the channel, logged only, or suppressed.

## Best Possible Solution

Add a documented Gateway-owned policy with current behavior as the default, gating only recovery-notice delivery to prompt and outbound channel surfaces while preserving recovery, restart continuations, and operator diagnostics.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by triggering last-known-good startup/reload recovery or consuming a `config-auto-recovery` restart sentinel with routing; current main then queues the notice into the main session and may deliver the same message outbound.

## Solution Assessment

Yes. A central `gateway.systemEvents.recoveryNotice`-style policy is the narrowest maintainable fix because channel-specific prompt filters do not cover both config-recovery paths and restart-sentinel delivery.
