## Summary

Keep open. Current main still has no operator-controlled recovery-notice policy, and the recovery paths described in the report still enqueue prompt-facing system events and, for restart sentinels with a route, deliver the same notice out-of-band. The related open items do not supersede this: #49579 is a channel-specific banner filter, while #39307 asks for more user-visible reset signaling rather than suppressing customer-facing recovery internals.

## What This Changes

Request to add a `gateway.systemEvents.recoveryNotice` config policy that can keep current recovery notices, log them only, or suppress them for customer-facing bots.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates only recovery notice delivery to prompt and outbound channel surfaces, while preserving last-known-good recovery, restart continuations, wake behavior, and operator logs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger last-known-good config recovery or a `config-auto-recovery` restart sentinel on current main, and the source/tests show the recovery message is queued into the main session and may be delivered outbound when routing survives restart.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice`-style policy is the narrowest maintainable fix because it centralizes operator intent for this event class; channel-specific filters such as #49579 would not cover both startup/reload recovery and restart-sentinel outbound delivery.
