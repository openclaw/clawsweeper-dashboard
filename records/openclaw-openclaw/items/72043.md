## Summary

Keep open. Current main still lacks an operator-controlled recovery-notice policy, and the two user-visible recovery paths described in the report are still hardwired: config recovery queues a main-session system event, while config-auto-recovery restart sentinels can both wake the session and send an outbound channel notice. The related open items do not supersede this because #49579 is channel/banner-specific and #39307 asks for a different, more visible context-reset behavior.

## What This Changes

Add a `gateway.systemEvents.recoveryNotice` configuration knob to control whether last-known-good recovery notices are injected into prompts, delivered out-of-band, or only logged/suppressed.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates config-recovery prompt events and config-auto-recovery restart-sentinel outbound notices, while leaving actual last-known-good recovery, restart continuations, and operator logs intact.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger last-known-good config recovery and current main queues the main-session recovery notice; for CLI config-auto-recovery sentinels, startup also wakes the session and sends the notice outbound when a route exists.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice`-style policy is the narrowest maintainable fix because it centralizes operator intent for this event class; channel-specific filters like #49579 would not cover the restart-sentinel outbound path and would keep adding one-off suppressions.
