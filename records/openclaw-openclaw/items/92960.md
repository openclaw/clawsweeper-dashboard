## Summary

Keep open: current main still cannot preserve an explicit agent scope through `sessions.describe`, and the latest release already exposes this method, so the reported scoped-global session-state bug remains actionable.

## What This Changes

The issue reports that Gateway `sessions.describe` drops caller `agentId` scope, letting global session descriptions resolve through the default store agent instead of the caller's scoped agent.

## Best Possible Solution

Add optional `agentId` support to `sessions.describe`, thread it into the existing scoped store-target path for global sessions, and cover default-agent plus non-default-agent behavior with focused Gateway protocol tests.

## Reproduction Assessment

Yes from source inspection: `sessions.describe` cannot currently accept `agentId`, while the handler's target helper already supports scoped global lookup when `agentId` is provided. I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

Yes. Optional protocol and handler threading is the narrowest maintainable fix because it reuses existing scoped store lookup behavior and matches sibling session paths that already preserve agent scope.
