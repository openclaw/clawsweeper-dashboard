## Summary

Keep open: current main and the latest release still leave `sessions.describe` without a safe agent-scoped contract, and the live maintainer closeout explicitly says the wrong-agent report remains valid while param-only fixes are incomplete.

## What This Changes

The issue asks Gateway `sessions.describe`/higher-level describe paths to preserve caller agent scope so `global` sessions resolve against the requested agent instead of the default store agent.

## Best Possible Solution

Keep this as the canonical tracker; land a shared scoped-session resolver and durable requester-agent provenance plan before adding a public `sessions.describe.agentId` contract.

## Reproduction Assessment

Yes. Source inspection on current main shows the describe schema rejects extra `agentId` fields and the handler loads entries without agent scope; I did not run a live multi-agent Gateway repro in this read-only review.

## Solution Assessment

No. A param-only public `agentId` addition is not the best fix because maintainer discussion and current source show the durable repair needs resolver semantics and requester-agent provenance first.
