## Summary

Keep open: current main and v2026.6.8 still leave `sessions.describe` without a safe agent-scoped contract, and maintainer discussion explicitly rejected the recent param-only fixes as incomplete for legacy aliases, retired stores, and global child provenance.

## What This Changes

The issue requests preserving caller `agentId` scope for Gateway `sessions.describe` so `global` sessions resolve against the requested agent instead of the default store agent.

## Best Possible Solution

Keep this as the canonical tracker; land a shared scoped-session resolver and requester-agent provenance/migration plan, then expose `sessions.describe.agentId` once the contract can be honored.

## Reproduction Assessment

Yes. Source inspection shows current main still rejects `agentId` on `sessions.describe` and then loads `global` session entries without requested agent scope; I did not run a live multi-agent Gateway repro in this read-only review.

## Solution Assessment

No, this is not solved by current main or the closed param-only PRs. The best fix needs maintainer-owned resolver/provenance design before adding the public describe `agentId` contract.
