## Label Justifications

- `P2`: The issue is a bounded Gateway session-state correctness bug with wrong-agent data risk, not a core outage or emergency.
- `impact:session-state`: The affected path can mis-associate an agent-scoped caller with another agent's `global` session state.

## Summary

Keep open: current main and the latest release still leave `sessions.describe` without a scoped `agentId` contract, and the handler still loads `global` sessions without caller agent scope. Maintainer discussion also says the recent param-only PRs were incomplete because aliases, retired stores, and global child provenance need a canonical resolver/provenance design first.

## What This Changes

This issue asks Gateway `sessions.describe` to preserve caller `agentId` scope for `global` sessions instead of resolving through the default store agent.

## Best Possible Solution

Keep this as the canonical tracker and land a shared scoped-session resolver plus requester-agent provenance/migration strategy before exposing a public `sessions.describe.agentId` contract.

## Reproduction Assessment

Yes. Source inspection shows the current describe schema rejects `agentId` and the handler loads `global` sessions without passing agent scope, which falls back to the default store agent.

## Solution Assessment

No, a small public `agentId` parameter alone is not the best fix. The safer path is an internal canonical scoped resolver and persisted requester-agent provenance before exposing the protocol promise.
