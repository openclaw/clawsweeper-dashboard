## Label Justifications

- `P2`: This is a bounded Gateway session-state correctness bug with wrong-agent data risk, but not a core outage or security emergency.
- `impact:session-state`: The affected path can mis-associate an agent-scoped caller with another agent's `global` session state.

## Summary

Keep open: current main and v2026.6.8 still leave `sessions.describe` without a safe agent-scoped contract, and maintainer follow-up already closed the simple `agentId` forwarding PRs as incomplete for global child provenance, aliases, and retired stores.

## What This Changes

The issue requests a safe agent-scoped `sessions.describe` path so a caller scoped to a non-default agent does not receive the default agent's `global` session data.

## Best Possible Solution

Keep this issue as the canonical tracker; design and land a shared scoped-session resolver and requester-agent provenance path first, then add the public `sessions.describe.agentId` contract with focused upgrade and regression proof.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main because the describe schema rejects `agentId` and the handler loads the session target without passing any agent scope; no live multi-agent Gateway repro was run in this read-only review.

## Solution Assessment

No. The simple param-forwarding fix is not the best current solution because maintainer testing found unresolved legacy alias, retired-store, and global child-provenance failures; the safer path is a shared scoped resolver plus provenance before a public protocol promise.
