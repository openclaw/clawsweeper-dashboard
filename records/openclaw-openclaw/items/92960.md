## Summary

Keep open: current `main` and the latest release still lack an agent-scoped `sessions.describe` contract, and the linked open PR is an implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue requests that Gateway `sessions.describe` preserve a caller-provided `agentId` when resolving global session state.

## Best Possible Solution

Add optional `agentId` support to `sessions.describe` at the protocol/schema boundary, thread it through the handler to the existing scoped global store-target helper, and cover scoped plus unscoped behavior with focused Gateway tests.

## Reproduction Assessment

Yes from source inspection: current `main` rejects `agentId` in `sessions.describe` params and then resolves the target without any requested agent scope. I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

Yes. Threading `agentId` through the existing protocol, handler, and scoped store-target helper is the narrow owner-boundary fix; a handler-only diff should also cover the schema and regression tests.
