- `impact:session-state`: The affected path can mis-associate an agent-scoped caller with another agent's `global` session state.

## Summary

Keep open: current main and the latest release still leave `sessions.describe` without a safe agent-scoped contract, and maintainer discussion explicitly rejected the recent param-only fixes as incomplete for legacy aliases, retired stores, and global child provenance.

## What This Changes

The issue reports that Gateway `sessions.describe` drops caller agent scope, so `global` session descriptions can resolve through the default store agent instead of the requested agent.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers land a shared scoped-session resolver and durable requester-agent provenance plan, then add the public `sessions.describe.agentId` contract only after those cases are safe.

## Reproduction Assessment

Yes, source-reproducible: current main's describe schema rejects caller `agentId` and the handler loads session entries without passing agent scope. I did not run a live multi-agent Gateway repro because this review is read-only.

## Solution Assessment

No, a param-only public `agentId` contract is not the best current fix. Maintainer discussion closed those attempts because the durable solution needs canonical resolver behavior and requester-agent provenance before the protocol promise is added.
