## Summary

Keep open: current main and the latest release still have the scoped `sessions.describe` gap, and a maintainer explicitly kept this as the canonical issue after closing the param-only PR attempts as insufficient.

## What This Changes

The issue reports that Gateway `sessions.describe` loses requested agent scope and can resolve a non-default caller's `global` session through the default agent store.

## Best Possible Solution

Keep this canonical issue open until core has a storage/provenance-aware scoped-session resolver shared by describe/list and child-link persistence; only then add a public describe `agentId` contract with focused tests and real Gateway proof.

## Reproduction Assessment

Yes. Source inspection shows current main rejects `agentId` on `sessions.describe`, then loads and builds the `global` row without requested agent scope while sibling methods already preserve that scope.

## Solution Assessment

No. The closed PRs show that a param-only `agentId` fix is not the best solution; the safer fix is a canonical resolver/provenance path first, followed by the public protocol field.
