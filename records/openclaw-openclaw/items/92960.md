## Summary

Keep open: current main and the latest release still lack an `agentId` field on `sessions.describe` and the handler still resolves through the default agent path; the issue should wait for the open canonical fix PR to merge.

## What This Changes

The issue reports that Gateway `sessions.describe` does not preserve a requested `agentId`, so global session lookups can resolve against the default agent instead of the caller-scoped agent.

## Best Possible Solution

Merge the canonical scoped `sessions.describe` fix that adds the optional protocol field, threads validated agent scope into lookup, and keeps regression coverage for default and non-default global sessions.

## Reproduction Assessment

Yes, source-reproducible: current main rejects `agentId` in the describe schema and the handler omits agent scope when loading global sessions. I did not run a live multi-agent Gateway repro because this review is read-only.

## Solution Assessment

Yes: adding optional protocol support and routing the handler through the existing scoped session lookup is the narrow maintainable fix. The issue should remain open until the linked implementation PR merges.
