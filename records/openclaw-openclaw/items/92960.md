## Label Justifications

- `P2`: The report is a bounded Gateway session-state correctness problem with wrong-agent data risk, not a core outage or emergency.
- `impact:session-state`: The affected path can mis-associate an agent-scoped caller with another agent's `global` session state.

## Summary

Keep open: current main and the latest release still leave `sessions.describe` without an agent-scoped request contract, and the maintainer discussion shows the param-only fixes were closed because they could still misattribute legacy aliases, retired stores, and `global` children without requester-agent provenance.

## What This Changes

The issue reports that Gateway `sessions.describe` can ignore a requested agent scope and return `global` session state from the default agent store.

## Best Possible Solution

Keep this issue as the canonical tracker; implement a shared scoped-session resolver plus persisted requester-agent provenance and migration strategy, then add the public `sessions.describe.agentId` contract once the resolver can safely honor it.

## Reproduction Assessment

Yes. Current source shows `sessions.describe` rejects `agentId` at schema validation and then loads `global` sessions without passing caller agent scope, which falls back to the default store agent; I did not run a live multi-agent Gateway repro in this read-only review.

## Solution Assessment

No. A small public `agentId` field by itself is not the best fix; the safer path is an internal canonical scoped resolver and requester-agent provenance before exposing the public describe promise.
