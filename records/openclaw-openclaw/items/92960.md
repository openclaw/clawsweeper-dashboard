## Label Justifications

- `P2`: This is a bounded Gateway session-state correctness bug with wrong-agent session association, not an outage or emergency.
- `impact:session-state`: The report is about `sessions.describe` returning or associating the wrong agent's global session state.

## Summary

Keep open: current main and the latest release still have the wrong-agent `sessions.describe` gap, and maintainer discussion says the safe replacement needs a canonical scoped-session resolver/provenance design rather than the closed one-line fixes.

## What This Changes

The issue requests that Gateway `sessions.describe` preserve caller agent scope instead of resolving `global` sessions through the default store agent.

## Best Possible Solution

Define one canonical scoped-session resolver, including requester-agent provenance where needed, then expose the `agentId` describe contract with regression coverage for default, non-default, child, alias, and retired-store cases.

## Reproduction Assessment

Yes, from source inspection: current main rejects `agentId` in `sessions.describe` params and the handler resolves without agent scope. I did not run a live multi-agent Gateway repro because this is a read-only cleanup review.

## Solution Assessment

No, the previously proposed one-line/protocol-only fixes are no longer the best way to solve the issue after maintainer review. The maintainable path is a canonical scoped-session resolver and/or persisted requester-agent provenance before adding the public describe contract.
