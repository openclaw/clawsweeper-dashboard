## Summary

Keep open: current main and the latest release still leave `sessions.describe` without a safe agent-scoped contract, and maintainer discussion confirms the recent param-only fixes were closed because resolver/provenance gaps remain.

## What This Changes

The issue requests that Gateway `sessions.describe` preserve a caller’s requested agent scope so `global` session lookups do not fall back to the default agent store.

## Best Possible Solution

Keep this issue as the canonical tracker until a shared scoped-session resolver and durable requester-agent provenance story are designed, tested, and only then exposed through the public describe contract.

## Reproduction Assessment

Yes, source-reproducible: current main’s describe schema rejects extra `agentId` fields and the handler loads session entries without passing agent scope. I did not run a live multi-agent Gateway repro because this review is read-only.

## Solution Assessment

No, a param-only public `agentId` describe contract is not the best current fix. The safer path is a shared scoped-session resolver plus requester-agent provenance before making the protocol promise.
