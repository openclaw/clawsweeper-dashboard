## Label Justifications

- `P2`: The report is a bounded Gateway session-state correctness bug with wrong-agent data risk, not a core outage or emergency.
- `impact:session-state`: The affected path can mis-associate an agent-scoped caller with another agent's `global` session state.

## Summary

Keep open: current main and v2026.6.8 still leave `sessions.describe` without an agent-scoped contract, the handler still loads the target without caller agent scope, and maintainer discussion says the recent param-only fixes were incomplete for aliases, retired stores, and global-child provenance.

## What This Changes

The issue asks for Gateway `sessions.describe` and higher-level callers to preserve requested agent scope so global session lookup cannot return another agent's session state.

## Best Possible Solution

Keep this issue as the canonical tracker; land a shared scoped-session resolver plus requester-agent provenance and migration strategy, then add the public `sessions.describe.agentId` contract once those paths are safe.

## Reproduction Assessment

Yes, source-reproducible: current main's describe schema rejects `agentId` and the handler loads `global` sessions without passing agent scope, which falls back to the default store agent. I did not run a live multi-agent Gateway repro in this read-only review.

## Solution Assessment

No, a small public `agentId` parameter alone is not the best fix. The safer path is an internal canonical scoped resolver and persisted requester-agent provenance before exposing the protocol promise.
