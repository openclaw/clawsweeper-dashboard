## Label Justifications

- `P2`: This is a bounded Gateway session-state correctness bug with wrong-agent data risk, not a core outage or security emergency.
- `impact:session-state`: The affected path can mis-associate an agent-scoped caller with another agent's `global` session state.

## Summary

Keep open: current main and the latest release still leave `sessions.describe` without a safe agent-scoped contract, and maintainer discussion explicitly rejected the recent param-only fixes as incomplete for aliases, retired stores, and global child provenance.

## What This Changes

The issue asks for Gateway session description to preserve agent scope so `global` session rows resolve for the requested agent instead of the default store agent.

## Best Possible Solution

Keep this as the canonical tracker; land a shared scoped-session resolver and durable global-parent provenance plan first, then add the public `sessions.describe.agentId` contract once aliases, retired stores, and child attribution are safe.

## Reproduction Assessment

Yes, source-reproducible: current main's describe schema rejects extra `agentId` fields and the handler loads session entries without passing agent scope. I did not run a live multi-agent Gateway repro because this review is read-only.

## Solution Assessment

No, a param-only public `agentId` contract is not the best current fix. Maintainer discussion closed those attempts because the durable solution needs canonical resolver behavior and requester-agent provenance before the protocol promise is added.
