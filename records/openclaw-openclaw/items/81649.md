## Summary

Keep open: the report is fresh and user-impacting, but current source points to an invalid runtime/activation misunderstanding rather than proof that the Anthropic provider plugin fails to load.

## What This Changes

The issue reports that OpenClaw 2026.5.7 fails with `Requested agent harness "anthropic" is not registered` when using an Anthropic API-key profile and runtime `anthropic`.

## Best Possible Solution

Confirm the reporter’s effective config, then either guide them to remove the `anthropic` runtime for API-key use or use `claude-cli` for Claude CLI, and consider a targeted doctor/config diagnostic for this misconfiguration.

## Reproduction Assessment

Yes for the error string when current main resolves an explicit runtime policy to `anthropic`: source shows that unregistered forced harness ids throw that message. The exact packaged release flow and the legacy config path named in the report were not live-reproduced.

## Solution Assessment

No, registering a new `anthropic` harness does not look like the narrowest fix because current docs use no runtime override for API-key Anthropic and `claude-cli` for Claude CLI. The safer path is maintainer triage around config cleanup or diagnostics.
