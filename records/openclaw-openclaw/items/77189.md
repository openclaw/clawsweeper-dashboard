## Summary

Keep open: this PR targets a valid gateway startup warning-noise bug, but the current predicate would also downgrade peer-supplied pre-auth 1013 closes that were not server-initiated startup closes.

## What This Changes

The PR downgrades `closed before connect` logging for 1013 `gateway starting` WebSocket closes to debug, adds startup regression coverage, and adds a changelog entry.

## Best Possible Solution

Narrow the downgrade to the existing `startup-sidecars-pending` close cause plus the 1013/reason check, with positive startup coverage and a negative peer-supplied close test.

## Reproduction Assessment

Yes, at source level: current main marks the startup close cause and then closes 1013, while the close logger currently routes that pre-client close to WARN. The PR regression is also source-clear because its predicate trusts close code and reason without checking the server-owned close cause.

## Solution Assessment

No. Downgrading the expected startup close is the right direction, but the maintainable fix should key on the existing `startup-sidecars-pending` close cause so unrelated pre-connect failures keep WARN visibility.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Restrict 1013 downgrade to startup close cause:** `src/gateway/server/ws-connection.ts:349-350`
  - body: The new predicate trusts only the close code and reason, so any pre-auth peer can close with 1013 and text containing `gateway starting` and have an unexpected non-startup close logged at debug. The startup path already sets the `startup-sidecars-pending` close cause; require that server-owned marker too so other pre-connect closes stay WARN.
  - confidence: 0.9
