## Summary

Keep open: the underlying startup WARN inflation is valid, but the PR is not mergeable as written because it downgrades based only on close code/reason instead of the gateway-owned startup close cause.

## What This Changes

The PR downgrades gateway WebSocket 1013 `gateway starting` close-before-connect logging to debug, adds startup regression coverage, and adds a changelog entry.

## Best Possible Solution

Gate the debug downgrade on `closeCause === "startup-sidecars-pending"` as well as the 1013/reason check, while preserving the existing SwiftPM downgrade and WARN behavior for other pre-connect closes.

## Reproduction Assessment

Yes, at source level: current main marks `startup-sidecars-pending`, sends retryable UNAVAILABLE, then closes 1013 `gateway starting`, while the close-before-connect path logs non-clients at WARN. The PR regression is also source-clear because the added downgrade trusts only peer-visible code and reason.

## Solution Assessment

No. Downgrading the expected startup close is the right direction, but the maintainable fix should require the existing server-owned startup close cause so unrelated pre-connect failures keep WARN visibility.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Restrict 1013 downgrade to startup close cause:** `src/gateway/server/ws-connection.ts:349-350`
