Full review comments:

- **[P1] Warn before sending memory text to completion models:** `src/config/schema.help.ts:1053-1054`
  - body: The new `completionModel` help enables lumber/hichunk but does not warn that memory file text is placed into prompts and sent to the configured completion model during indexing.   - confidence: 0.88
- **[P2] Bound LLM chunking completions:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:651-658`
  - body: This completion call passes only model/auth/context. The current helper supports `options.maxTokens` and `options.signal`, but this path supplies neither, so a slow or verbose provider can stall memory indexing or create unbounded cost; wire a bounded max token and cancellation/deadline into lumber/hichunk indexing.
  - confidence: 0.84
- **[P2] Avoid splitting every tied semantic distance:** `src/memory-host-sdk/host/chunking/semantic.ts:72-74`
  - body: When all adjacent semantic distances tie, the percentile threshold equals that same value and `d >= threshold` marks every boundary as a breakpoint. Identical or flat embeddings would produce one chunk per sentence instead of one stable semantic segment; add a tie/min-spread guard or use a strict above-threshold comparison.
  - confidence: 0.9
- **[P3] Fix semantic option help to match the implementation:** `src/config/schema.help.ts:1041-1050`
  - body: The config help says `maxTokens` is used by semantic chunking, `bufferSize` is measured in tokens, and higher percentile thresholds yield more breakpoints. The code does not read `maxTokens` for semantic chunking, treats `bufferSize` as adjacent sentence count, and higher percentiles generally select fewer breakpoints, so users would tune the wrong knobs.
  - confidence: 0.86
