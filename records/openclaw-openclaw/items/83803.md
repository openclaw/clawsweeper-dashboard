## What This Changes

## Best Possible Solution

## Reproduction Assessment

Yes from source inspection: current main recovery looks for status running plus abortedLastRun true, while the PR deletes those fields from normal main/topic sessions during store normalization. I did not run tests because this review is read-only and the relevant tests create files.

## Solution Assessment

No, not as implemented. The safer fix is to distinguish stale channel busy/fallback pins from durable restart-recovery lifecycle state, then add coverage proving both the Telegram/topic wedge fix and restart recovery still work.
