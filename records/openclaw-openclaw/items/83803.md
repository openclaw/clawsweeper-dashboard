## Summary

Keep open: the PR targets a real session-state bug, but the current patch should not merge because the new normalizer strips lifecycle fields that restart recovery still depends on. The supplied live output is useful context, but it does not prove this patch running in a real gateway.

## What This Changes

The PR adds session-store normalization and doctor repair to remove transient lifecycle, reply-turn, fallback, and auth metadata from normal sessions while preserving it for internal execution sessions.

## Best Possible Solution

Keep the transient fallback cleanup, but scope it so restart-recovery lifecycle state is preserved or moved to an explicit durable recovery marker with regression coverage.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR normalizes every store entry through a helper that deletes `status` and `abortedLastRun`, while current restart recovery depends on those fields. I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction is right, but the implementation needs a narrower persistence boundary or a replacement recovery marker so it does not break restart recovery while clearing unsafe fallback state.
