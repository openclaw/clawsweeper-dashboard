## Summary

Keep open. Current main still does not notify WhatsApp senders after approval: the CLI has a generic opt-in `--notify` path, but notification is adapter-gated and the WhatsApp plugin only registers an `idLabel`, so the notifier returns without sending. This remains a narrow fix because WhatsApp already has outbound send machinery and other channels show the intended text-pairing adapter pattern.

## What This Changes

Notify a WhatsApp sender when their DM pairing request is approved, with a possible notify flag/default confirmation and optional blocked-message replay follow-up.

## Best Possible Solution

Implement WhatsApp support for the existing pairing approval notification contract so `openclaw pairing approve whatsapp <CODE> --notify` sends the standard approval message through the correct active WhatsApp account, with focused tests for the adapter and account scoping. Keep default-on behavior and blocked-message replay as separate follow-up decisions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: WhatsApp creates and sends pairing challenges, `pairing approve --notify` reaches the generic notifier, and the notifier exits because the WhatsApp pairing adapter has no `notifyApproval` callback.

## Solution Assessment

Yes for a narrow repair. Reusing the existing `--notify` pairing adapter path is the smallest maintainable fix; changing defaults, adding new config, or replaying the blocked message should be separate maintainer/product work.
