## Summary

Keep open. Current main and the latest release still expose the generic pairing approval notification contract, but WhatsApp does not register an approval notifier and the CLI/generic notifier path still drops account scope, so the approved sender receives no WhatsApp confirmation even when `--notify` is used.

## What This Changes

The issue requests a WhatsApp confirmation message to the sender after their pairing request is approved.

## Best Possible Solution

Implement WhatsApp support for the existing account-aware `pairing approve --notify` contract, using WhatsApp outbound delivery for the standard approval text and leaving default-on notification or blocked-message replay to separate product decisions.

## Reproduction Assessment

Yes, at source level. `openclaw pairing approve --channel whatsapp --account <id> <code> --notify` reaches `notifyPairingApproved`, which returns when the WhatsApp adapter lacks `notifyApproval`, and the current notify path also drops account scope.

## Solution Assessment

Yes for the narrow repair. Reusing the existing pairing adapter notification contract, forwarding account scope, and sending the standard approval message through WhatsApp outbound is the smallest maintainable fix; default-on notification and replay are separate decisions.
