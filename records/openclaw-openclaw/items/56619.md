## Summary

Keep open: current main and v2026.5.22 still expose `pairing approve --notify`, but WhatsApp lacks a `notifyApproval` pairing adapter and the CLI does not pass account scope into the notifier, so an approved WhatsApp sender still receives no confirmation.

## What This Changes

Notify a WhatsApp sender after their DM pairing request is approved, with optional future support for configurable copy or blocked-message replay.

## Best Possible Solution

Implement WhatsApp support for the existing account-aware `pairing approve --notify` contract, using the standard approval message through WhatsApp outbound while leaving default-on behavior, configurable copy, and blocked-message replay as separate product decisions.

## Reproduction Assessment

Yes, at source level. `openclaw pairing approve --channel whatsapp --account <id> <code> --notify` reaches a generic notifier that returns because WhatsApp has no `notifyApproval`, and the CLI currently drops `accountId` before notifying.

## Solution Assessment

Yes for the narrow repair. Reusing the existing pairing adapter notification contract, forwarding account scope, and sending the standard approval message through WhatsApp outbound is the maintainable path; default-on behavior and replay are separate decisions.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/56619 narrowly. Current main documents `openclaw pairing approve ... --notify`, but WhatsApp lacks a `notifyApproval` pairing adapter and the CLI drops `--account` before calling `notifyPairingApproved`. Reuse the existing `ChannelPairingAdapter.notifyApproval` contract: thread optional `accountId` from `src/cli/pairing-cli.ts` through `src/channels/plugins/pairing.ts`, add a WhatsApp notifier in `extensions/whatsapp/src/channel.ts` that sends the standard approval message through the existing WhatsApp outbound/send path with the correct account, and add focused tests. Do not implement default-on notification, new config keys, owner-notification hooks, approved-sender list/revoke commands, or blocked-message replay. Expect a user-facing fix changelog entry during maintainer landing, not as an external reporter action.
