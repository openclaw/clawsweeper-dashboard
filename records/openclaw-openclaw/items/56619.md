## Summary

Keep open: current main and the latest release still expose `pairing approve --notify`, but WhatsApp has no approval notifier and the CLI notification path still drops `--account`, so the approved sender does not receive the promised confirmation.

## What This Changes

Notify an approved WhatsApp pairing requester after `openclaw pairing approve` succeeds, with optional discussion of default-on behavior, configurable copy, or replaying the blocked message.

## Best Possible Solution

Implement WhatsApp support for the existing account-aware `pairing approve --notify` contract, using the standard approval message through WhatsApp outbound and leaving default-on notification, custom copy, and replay as separate decisions.

## Reproduction Assessment

Yes, at source level. `openclaw pairing approve --channel whatsapp --account <id> <code> --notify` reaches a generic notifier that returns because WhatsApp has no `notifyApproval`, and the CLI currently drops `accountId` before notifying.

## Solution Assessment

Yes for the narrow repair. Reusing the existing pairing adapter notification contract, forwarding account scope, and sending the standard approval message through WhatsApp outbound is the maintainable path; default-on behavior and replay are separate decisions.
