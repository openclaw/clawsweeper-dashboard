- `impact:message-loss`: The issue is about an expected channel confirmation message being silently omitted after pairing approval.

## Summary

Keep open: current main still documents `pairing approve --notify`, but WhatsApp pairing has no `notifyApproval` adapter and the generic notify path still drops account scope, so approval notification is still a source-proven no-op for WhatsApp senders.

## What This Changes

The issue requests notifying a WhatsApp sender after their DM pairing request is approved, with optional default-on/configurable behavior or replay of the blocked message.

## Best Possible Solution

Implement WhatsApp support for the existing account-aware `pairing approve --notify` contract, keeping default-on notification, configurable copy, and blocked-message replay as separate product decisions.

## Reproduction Assessment

Yes, at source level. `openclaw pairing approve --channel whatsapp --account <id> <code> --notify` reaches `notifyPairingApproved`, which returns because WhatsApp lacks `notifyApproval`, and the current notify call also drops `accountId`.

## Solution Assessment

Yes for the narrow repair. Reusing the existing pairing adapter notification contract, forwarding account scope, and sending through WhatsApp outbound is the smallest maintainable solution; default-on behavior and replay are separate decisions.
