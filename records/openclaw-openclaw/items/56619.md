- `impact:message-loss`: The issue is about an expected channel confirmation message being silently omitted after pairing approval.

## Summary

Keep open: current main still documents `pairing approve --notify`, but WhatsApp pairing has no approval notifier and the CLI notification helper does not forward account scope, so the source path still no-ops for an approved WhatsApp sender.

## What This Changes

Request a WhatsApp sender-facing confirmation after pairing approval, preferably through the existing pairing approval notification path.

## Best Possible Solution

Implement WhatsApp support for the existing account-aware `pairing approve --notify` contract, and leave default-on notification, configurable message text, and blocked-message replay as separate product decisions.

## Reproduction Assessment

Yes, at source level. `openclaw pairing approve --channel whatsapp --account <id> <code> --notify` reaches `notifyPairingApproved`, which returns because WhatsApp lacks `notifyApproval`, and the current notify call also drops `accountId`.

## Solution Assessment

Yes for the narrow repair. Reusing the existing pairing adapter notification contract, forwarding account scope, and sending the standard approval message through WhatsApp outbound is the smallest maintainable solution; default-on behavior and replay are separate decisions.
