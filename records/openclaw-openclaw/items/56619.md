## Label Justifications

- `P2`: This is a normal-priority WhatsApp pairing UX bug with limited blast radius and a clear adapter-level repair path.
- `impact:message-loss`: An approved WhatsApp sender receives no confirmation and the originally blocked message remains unhandled until they retry.

## Summary

Keep open: current main still has the generic `pairing approve --notify` path, but WhatsApp declares only the pairing id/normalizer and has no `notifyApproval` hook, so the sender-confirmation gap remains source-reproducible.

## What This Changes

The issue requests notifying a WhatsApp sender after their pairing request is approved, with optional follow-up ideas for default notification, configurable text, or replaying the blocked message.

## Best Possible Solution

Add a WhatsApp `pairing.notifyApproval` implementation for the existing `openclaw pairing approve ... --notify` path using the shared approval message and existing WhatsApp outbound send route, without adding config or replay behavior in the first fix.

## Reproduction Assessment

Yes, at source level: `pairing approve --notify` dispatches through `notifyPairingApproved`, and WhatsApp’s pairing adapter lacks `notifyApproval`. I did not run a live WhatsApp transport repro in this read-only review.

## Solution Assessment

Yes for the narrow repair: implement the missing WhatsApp adapter hook using the existing outbound send path and shared approval message. Making notification the default or replaying the blocked message should be separate maintainer product work.
