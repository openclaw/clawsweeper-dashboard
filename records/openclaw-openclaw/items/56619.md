## Summary

Keep open: current main and v2026.5.20 still document `pairing approve --notify`, but WhatsApp does not provide a pairing approval notifier and the CLI notification path does not forward account scope, so an approved WhatsApp sender still gets no confirmation.

## What This Changes

Request a WhatsApp pairing-approval confirmation to the approved sender, with optional ideas for a flag, default notification, configurable text, and replaying the blocked message.

## Best Possible Solution

Implement WhatsApp support for the existing account-aware `pairing approve --notify` contract, and leave default-on notification, configurable copy, and blocked-message replay as separate product decisions.

## Reproduction Assessment

Yes, at source level. `openclaw pairing approve --channel whatsapp --account <id> <code> --notify` reaches `notifyPairingApproved`, which returns because WhatsApp lacks `notifyApproval`, and the current notify call also drops `accountId`.

## Solution Assessment

Yes for the narrow repair. Reusing the existing pairing adapter notification contract, forwarding account scope, and sending the standard approval message through WhatsApp outbound is the smallest maintainable solution; default-on behavior and replay are separate decisions.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/56619 on current main. The existing `openclaw pairing approve --notify` contract is documented, but WhatsApp does not notify the approved sender because `src/cli/pairing-cli.ts` does not pass `accountId` into `notifyPairingApproved`, `src/channels/plugins/pairing.ts` omits it from the adapter callback payload, and `extensions/whatsapp/src/channel.ts` has no `notifyApproval` implementation. Implement the narrow repair only: add accountId forwarding through the generic notifier, add a WhatsApp `notifyApproval` callback that sends `PAIRING_APPROVED_MESSAGE` to the approved WhatsApp sender through the channel-owned outbound/send path with the selected account, and add regression coverage. Do not make notification default-on, add new config flags, change approval semantics, or replay the blocked message in this PR. Preserve plugin boundaries: WhatsApp code should use plugin SDK/local WhatsApp APIs, and core should only carry the generic account-aware callback contract.
