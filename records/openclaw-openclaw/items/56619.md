## Label Justifications

- `P2`: The issue is a normal-priority WhatsApp pairing UX bug with limited blast radius and a clear implementation boundary.
- `impact:message-loss`: An approved WhatsApp sender receives no confirmation and the original blocked message remains unhandled, so the communication can be effectively lost until they retry.

## Summary

Keep open: current `main` has the generic `pairing approve --notify` path, but WhatsApp still does not implement the channel `notifyApproval` hook, so the reported sender-confirmation gap remains for the requested channel.

## What This Changes

Request that WhatsApp DM pairing approval notify the newly approved sender after owner approval, optionally through a CLI flag or configurable default.

## Best Possible Solution

Add a WhatsApp `pairing.notifyApproval` implementation for the existing `openclaw pairing approve ... --notify` path, without adding a new config option or original-message replay in the first fix.

## Reproduction Assessment

Yes, at source level: `pairing approve --notify` dispatches through `notifyPairingApproved`, and WhatsApp’s pairing adapter lacks `notifyApproval`. I did not run a live WhatsApp transport repro in this read-only review.

## Solution Assessment

Yes for the narrow repair: implement the missing WhatsApp adapter hook using the existing outbound/send path and shared approval message. Making notification the default or replaying the blocked message should be separate maintainer product work.
