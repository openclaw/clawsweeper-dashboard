## Summary

Keep open: the spawned-helper isolation is still useful and well proven, but the current diff still lets the isolated startup path bypass the resolved Codex startup timeout floor, so it needs a narrow fix before merge.

## What This Changes

The PR routes spawned Codex helper app-server startup through isolated clients, closes those clients during cleanup, and adds Codex app-server regression coverage for helper auth failures.

## Best Possible Solution

Preserve the spawned-helper isolation while keeping the existing resolved startup-timeout semantics identical for shared and isolated Codex app-server startup paths.

## Reproduction Assessment

Yes. The linked issue gives concrete invalid-token helper-spawn steps, and current main source shows spawnedBy is not used to select isolated clients while shared-client cleanup can clear the current singleton; I did not run a live repro in this read-only review.

## Solution Assessment

No, not as-is. The spawnedBy isolation boundary is the right narrow fix, but the implementation should pass the resolved startupTimeoutMs into isolated startup rather than raw params.timeoutMs.
