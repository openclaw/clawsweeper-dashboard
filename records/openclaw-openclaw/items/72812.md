## Label Justifications

- `P1`: The linked bug can tear down an active Codex/OpenClaw session after a helper auth failure, breaking a real agent workflow until recovery.
- `merge-risk: 🚨 auth-provider`: The patch changes how spawned helper runs acquire and clean up Codex app-server clients around auth-profile startup failures.
- `merge-risk: 🚨 session-state`: The patch deliberately changes shared-client ownership for spawned helper sessions so parent session state is no longer coupled to child startup cleanup.

## Summary

Keep open for maintainer merge review: the patch targets the linked Codex helper-run disconnect in https://github.com/openclaw/openclaw/issues/72574, has no blocking code findings from this review, and has sufficient terminal/live app-server proof, but it intentionally changes spawned-helper auth and session lifecycle.

## What This Changes

The branch routes Codex app-server runs with non-empty `spawnedBy` through isolated clients, closes isolated clients during failure/abort/final cleanup, adds focused regressions, and updates the changelog.

## Best Possible Solution

Codex helper runs with `spawnedBy` should fail and clean up their own app-server client without clearing the active parent shared client, while normal and subagent-key-only runs keep existing shared-client behavior.

## Reproduction Assessment

Yes, at source level. Current main starts helper runs through the shared client factory and clears the current startup client on failure, while the linked report supplies the invalid-token disconnect symptom and this PR adds focused regression coverage for that path.

## Solution Assessment

Yes. Gating isolation on non-empty `spawnedBy` is a narrow fix for helper runs, avoids over-isolating subagent-shaped session keys, and preserves existing shared-client behavior outside the reported boundary.
