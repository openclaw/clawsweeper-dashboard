## Summary

Keep open for normal maintainer review: the patch is a focused fix for the linked Codex helper-spawn disconnect bug, current main still uses the shared startup client path, and I did not find a blocking correctness issue in the diff.

## What This Changes

The PR routes `spawnedBy` Codex app-server attempts through isolated clients, closes those clients during startup failure/abort/final cleanup, and adds focused regression coverage plus a changelog entry.

## Best Possible Solution

Land this narrow spawned-helper isolation fix, or an equivalent implementation, while keeping ordinary non-spawned Codex runs on the existing shared-client path.

Yes, source-backed: the linked issue gives concrete invalid-token steps and logs, and current main shows a failed helper startup can still flow through shared-client cleanup. I did not rerun a live invalid-token gateway session in this read-only review.

## Solution Assessment

Yes, the `spawnedBy`-gated isolated startup path is the narrow maintainable fix because it isolates helper failures while preserving shared-client reuse for normal runs. The remaining question is maintainer acceptance of the extra per-helper client lifecycle cost.
