## Summary

Keep this PR open for maintainer review: current main still uses the shared Codex app-server startup path for spawned helper runs, while this patch is a focused fix with regression coverage and sufficient terminal proof. I found no blocking code-review findings; the remaining question is whether maintainers accept the auth/session lifecycle tradeoff of isolating helper runs.

## What This Changes

The PR routes Codex app-server attempts with non-empty `spawnedBy` through isolated startup clients, adds abort/error/final cleanup for those clients, adds focused tests, and updates the changelog.

## Best Possible Solution

Land the scoped `spawnedBy` isolation fix once maintainers accept the helper-run lifecycle boundary, preserving the guard that session-key shape alone does not trigger isolation.

## Reproduction Assessment

Yes, at source level: current main unconditionally routes startup through the shared client factory and clears that startup client on failure, while the linked report gives concrete invalid-token helper-run disconnect steps. I did not run a live reproduction because this review is read-only.

## Solution Assessment

Yes: gating isolated startup on non-empty `spawnedBy` is a narrow maintainable fix that reuses the existing isolated-client seam and avoids over-isolating by session-key shape. The remaining concern is maintainer acceptance of lifecycle risk, not a definite code defect.
