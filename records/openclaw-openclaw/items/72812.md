## Summary

Keep open for maintainer merge review: the PR is a focused Codex app-server session-stability fix, the current patch appears correct, and I did not find a blocking code or security finding. The remaining question is maintainer acceptance of the spawned-run lifecycle boundary and its auth/session ownership risk.

## What This Changes

The PR routes Codex app-server attempts with non-empty `spawnedBy` through isolated startup clients, adds abort/error/final cleanup for those clients, adds focused regression tests, and adds a changelog entry.

## Best Possible Solution

Land the scoped `spawnedBy` isolation fix once maintainers accept the helper-run lifecycle boundary, preserving the guard that session-key shape alone does not trigger isolation.

## Reproduction Assessment

Yes, at source level: current `main` routes helper-shaped runs through the shared startup factory and clears that startup client on failure, while the linked report supplies the invalid-token disconnect symptom. I did not run a live reproduction because this review is read-only.

## Solution Assessment

Yes: gating isolated startup on non-empty `spawnedBy` is a narrow maintainable fix that reuses the existing isolated-client seam and avoids over-isolating by session-key shape. The remaining issue is lifecycle-risk acceptance, not a definite code defect.
