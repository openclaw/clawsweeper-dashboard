## Label Justifications

- `P1`: A permanently non-durable Codex native subagent completion can re-enter the parent agent every five minutes indefinitely in real channel-backed workflows.
- `impact:message-loss`: The child completion never reaches a durable parent delivery state when the parent handoff remains non-durable.
- `impact:session-state`: The monitor leaves completion state pending forever and repeatedly wakes the parent session instead of settling delivery.

## Summary

Keep open: current main still retries Codex native subagent completion delivery forever on a permanently non-durable parent handoff, v2026.6.10 has the same retry scheduler, and the open related PRs cover adjacent delivery visibility or sessions_spawn parent-wake behavior rather than this monitor cap/deadline bug.

## What This Changes

Bound Codex native subagent completion delivery retries so permanently non-durable parent handoffs settle instead of re-running the parent indefinitely.

## Best Possible Solution

Add a bounded retry/deadline policy in the Codex native subagent monitor, record a terminal failed or blocked delivery state on give-up, and cover permanent non-durable plus eventual durable success without adding config surface.

## Reproduction Assessment

Yes, at source level: current main keeps the pending completion, clears the in-flight key, and re-arms deliverPendingCompletion on every non-durable result with no cap or deadline. I did not run a live channel outage in this read-only review.

## Solution Assessment

Yes: a bounded retry/deadline with terminal failed or blocked delivery in the Codex native monitor is the narrowest maintainable fix. Adding retry config knobs is not needed for the bug and would add public surface area.
