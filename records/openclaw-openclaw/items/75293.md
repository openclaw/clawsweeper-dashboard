## Summary

Keep open: the intended Parallels harness fix is still useful, but this PR's merge ref passes `macosExecArgs` before it is declared, so the macOS update smoke would fail before reaching the actual update path. The existing ClawSweeper comment already identified this blocker and a repair worker has been queued, but the PR head has not changed yet.

## What This Changes

The PR routes macOS Parallels npm-update guest script writing and chmod through the resolved desktop-user transport with mode `700`, and adds a regression test for that transport shape.

## Best Possible Solution

Land a repaired version that resolves macOS exec args before writing the guest script, keeps Linux on the existing root-owned path, and validates the macOS npm-update lane before merge.

## Reproduction Assessment

Yes for the PR blocker: the merge ref has a direct source-level temporal-dead-zone path in `guestMacos()`. The original Parallels permission behavior still needs live macOS Parallels validation.

## Solution Assessment

No for the current PR head: the same-transport writer approach is narrow and maintainable, but the declaration order must be fixed before the patch can run. The safer path is a tiny repair plus the focused script test and live macOS lane.
