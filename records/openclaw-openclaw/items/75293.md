## Summary

Keep open: the intended Parallels harness fix is still useful, but the current GitHub merge ref is incorrect after current main's fallback-user changes because it passes `macosExecArgs` before that const is declared.

## What This Changes

The PR routes macOS Parallels temp-script writing and chmod through the resolved guest exec args, restores `chmod 700`, and adds a regression test for that transport shape.

## Best Possible Solution

Repair the stale merge by resolving `macosExecArgs` before writing the macOS guest script, keep Linux on the root-owned path, preserve current fallback-user behavior, and validate the macOS Parallels lane before merge.

## Reproduction Assessment

Yes for the PR regression: the GitHub merge ref shows `macosExecArgs` used before declaration in `guestMacos()`, which is a source-level runtime failure path. The original Parallels permission behavior still needs live validation on a macOS Parallels host.

## Solution Assessment

No as merged: the intended same-identity writer approach is narrow and maintainable, but the stale merge order must be fixed first. The safer path is a tiny rebase/repair that moves `resolveMacosUpdateExecArgs(ctx)` above `writeGuestScript()` and then runs the focused script test plus live macOS lane.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Resolve macOS exec args before writing the script:** `scripts/e2e/parallels/npm-update-smoke.ts:665`
