## Summary

Keep open for maintainer handling: current main mitigates the original macOS permission failure with a broader `755` script plus fallback `chown`, but it does not implement the PR’s tighter writer/runner identity alignment for credential-bearing temp scripts.

## What This Changes

The PR routes macOS guest script writes and chmods through the resolved Parallels exec transport, restores `700` permissions, and adds a regression assertion for that transport shape.

## Best Possible Solution

Land a rebased version if maintainers want the tighter permission model: macOS script write, chmod, and execution should use the same resolved transport while Linux keeps the root-owned path.

## Reproduction Assessment

Unclear against current main: the original root-owned `700` permission-denied path is mitigated by the current `755` plus fallback `chown` workaround. The remaining owner-only credential exposure concern is source-verifiable, and live proof would be the macOS npm-update Parallels lane.

## Solution Assessment

Yes after rebase: the PR direction is narrower than the current broad-permission workaround because it aligns writer and runner identity while keeping credential-bearing scripts owner-readable only. The rebase should retain the current sudo fallback semantics.
