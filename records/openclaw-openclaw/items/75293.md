## Summary

Keep this PR open. The patch still addresses a real macOS Parallels npm-update smoke harness issue, but the exact head is now dirty against current main and needs maintainer-controlled rebase plus live Parallels validation before merge.

## What This Changes

The PR changes the macOS Parallels npm-update smoke harness to write and chmod guest scripts through the resolved macOS exec transport with `700` permissions, and adds a regression test for that transport shape.

## Best Possible Solution

Land a rebased version that writes, chmods, and executes macOS npm-update scripts through the same resolved desktop-user transport, keeps Linux on the root-owned path, preserves the sudo fallback semantics, and validates both the changed gate and the macOS npm-update Parallels lane.

## Reproduction Assessment

Unclear: current main no longer cleanly reproduces the original permission-denied failure because it uses `chmod 755` and a sudo-fallback `chown`. The remaining problem is source-verifiable as a broader-permission workaround, with live reproduction or validation through `timeout --foreground 75m pnpm test:parallels:npm-update -- --platform macos --json` on a Parallels host.

## Solution Assessment

Yes, after rebase: using the resolved macOS exec transport for `tee`, `chmod`, and `/bin/bash` is the narrowest maintainable fix because it aligns script ownership with execution while keeping credential-bearing temp scripts owner-readable only.
