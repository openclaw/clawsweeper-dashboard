## Summary

Keep open: current main avoids the original macOS permission failure with a broader `chmod 755` plus sudo-fallback `chown` workaround, but it does not implement this PR's tighter same-transport `chmod 700` fix. The PR itself is focused and has no discrete correctness finding, but GitHub reports it as dirty against current `main`, so it needs maintainer rebase and live Parallels validation before merge.

## What This Changes

The PR changes the Parallels npm-update smoke harness so macOS guest scripts are written and chmodded through the resolved macOS exec transport with `700` permissions, and adds a regression test for that transport shape.

## Best Possible Solution

Land a rebased version of this focused repair that resolves the macOS exec transport before writing the guest script, uses that same transport for `tee`, `chmod 700`, and `/bin/bash`, leaves the Linux root-owned path separate, and validates both the changed gate and the macOS Parallels npm-update lane.

## Reproduction Assessment

Unclear: source inspection verifies the original identity/permission concern and the PR's targeted behavior, but this environment cannot run `prlctl` or the macOS VM smoke. Current `main` may no longer reproduce the exact permission-denied failure because it uses `chmod 755` and a sudo-fallback `chown` workaround.

## Solution Assessment

Yes, after rebase: writing and chmodding through the same resolved macOS transport is narrower and more maintainable than making credential-bearing temp scripts world-readable. The safer integration path is to preserve current fallback intent while replacing the broad-permission workaround with the PR's owner-readable script handling.
