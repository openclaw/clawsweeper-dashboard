## Summary

This PR should stay open for maintainer review. Current `main` has a later workaround that makes the original permission-denied shape less direct, but it still writes the macOS update script through plain root `prlctl exec` and makes it `755`; the PR’s intended same-transport `700` write/chmod approach is still a focused improvement for this smoke harness.

## What This Changes

The PR changes the Parallels npm-update smoke harness so macOS guest scripts are written and chmodded through the resolved macOS exec transport with `700` permissions, and adds a regression test for that transport shape.

## Best Possible Solution

Land a rebased version of this focused repair that writes and chmods macOS npm-update scripts through the same resolved desktop-user transport used to execute them, keeps Linux on its root-owned path, and validates the changed gate plus the macOS npm-update Parallels lane.

## Reproduction Assessment

Unclear: current main no longer cleanly reproduces the original permission-denied failure because it uses `chmod 755` and a sudo-fallback `chown`. The remaining problem is source-verifiable as a broader-permission workaround, with live reproduction/validation through `timeout --foreground 75m pnpm test:parallels:npm-update -- --platform macos --json` on a Parallels host.

## Solution Assessment

Yes, after rebase: using the resolved macOS exec transport for `tee`, `chmod`, and `/bin/bash` is narrower than the current `755` plus fallback `chown` workaround because it aligns file ownership with execution while keeping credential-bearing temp scripts owner-readable only.
