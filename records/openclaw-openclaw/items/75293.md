## Summary

Keep open: current main still uses the broader `755` workaround for the macOS Parallels update script, while this PR preserves the tighter owner-only transport model and still needs maintainer rebase plus live Parallels validation before merge.

## What This Changes

The PR passes the resolved macOS exec arguments into the guest script writer, uses them for `tee` and `chmod`, restores `700` permissions, and adds a regression test for that transport shape.

## Best Possible Solution

Land a rebased version of this focused repair after preserving current fallback-user resolution, restoring owner-only temp-script permissions, and validating the changed gate plus the macOS npm-update Parallels lane.

## Reproduction Assessment

No high-confidence live reproduction was established: current main's `755` workaround means the original permission-denied symptom is not expected from source alone. The remaining owner-only transport gap is source-verifiable, and the live reproduction/validation path is `pnpm test:parallels:npm-update -- --platform macos --json` on a Parallels host.

## Solution Assessment

Yes, after rebase: using the same resolved macOS exec transport for `tee`, `chmod`, and `/bin/bash` is narrower than the current broad-readability workaround. The rebase should keep Linux on the root-owned path and retain current fallback-user resolution semantics.
