## Summary

Keep this PR open. Current `main` mitigates the original permission failure with broader `755` permissions and a sudo-fallback `chown`, but it has not implemented the narrower same-transport, owner-only script handling that this PR proposes for the macOS Parallels npm-update lane.

## What This Changes

The PR writes and chmods macOS Parallels npm-update scripts through the resolved macOS exec transport, restores `700` permissions, and adds regression coverage for that transport shape.

## Best Possible Solution

Land a rebased version of this focused fix that writes, chmods, and executes the macOS update script through the same resolved transport with owner-only permissions, while leaving the Linux root-owned path separate and validating the macOS npm-update lane on a Parallels host.

## Reproduction Assessment

Unclear for the exact permission-denied failure on current main because current main now uses `755` plus a fallback `chown`. The remaining concern is source-verifiable, and the live reproduction/validation path is `timeout --foreground 75m pnpm test:parallels:npm-update -- --platform macos --json` on a Parallels host.

## Solution Assessment

Yes. Passing the resolved macOS exec args into the script writer is narrower and safer than the current broader-permission workaround, especially because the temp script can carry provider credential material.
