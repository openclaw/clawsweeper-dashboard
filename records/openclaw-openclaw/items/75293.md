## Summary

Keep open: current main fixes the original permission-denied failure with a broader `755`/fallback `chown` workaround, but it does not implement this PR's tighter same-transport, owner-only script handling for the macOS Parallels npm-update lane.

## What This Changes

The PR changes the Parallels npm-update smoke harness so macOS temp scripts are written and chmodded through the resolved macOS exec transport with `700` permissions, plus a regression test for that shape.

## Best Possible Solution

Land a rebased version of this focused repair if maintainers want the tighter permission model: macOS `tee`, `chmod`, and `/bin/bash` should use the same resolved desktop-user transport while Linux keeps its root-owned path.

## Reproduction Assessment

Unclear: the exact original root-owned `700` permission-denied path no longer matches current main because main uses `755` plus fallback `chown`. The remaining gap is source-verifiable, with high-confidence live validation available via `timeout --foreground 75m pnpm test:parallels:npm-update -- --platform macos --json` on a Parallels host.

## Solution Assessment

Yes, after normal maintainer validation: the PR's approach is narrower than the current `755` workaround because it aligns writer and runner identity while keeping credential-bearing temp scripts owner-readable only.
