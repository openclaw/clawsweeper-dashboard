## Summary

Keep this PR open. Current main has a broader workaround for the original macOS permission failure, but it does not implement the PR’s intended tighter fix: writing and chmodding the credential-bearing macOS guest script through the same desktop-user transport that later executes it with owner-only permissions. The existing PR remains useful but needs a careful rebase over current main plus live Parallels validation before merge.

## What This Changes

The PR updates the Parallels npm-update smoke harness to write and chmod macOS guest scripts through the resolved macOS exec transport with `700` permissions, and adds a regression test for that transport shape.

## Best Possible Solution

Land a rebased version of this focused repair if maintainers want the tighter permission model: macOS `tee`, `chmod`, and `/bin/bash` should use the same resolved desktop-user transport, Linux should keep its root-owned path, and validation should include the changed gate plus the macOS npm-update Parallels lane.

## Reproduction Assessment

Unclear: the original root-owned `700` permission-denied path no longer exactly matches current main because main now uses `chmod 755` plus a sudo-fallback `chown`. The remaining issue is source-verifiable as a broader-permission workaround, with high-confidence live validation available through `timeout --foreground 75m pnpm test:parallels:npm-update -- --platform macos --json` on a Parallels host.

## Solution Assessment

Yes, after rebase: the PR’s approach is narrower than current main’s `755` workaround because it aligns writer and runner identity while keeping credential-bearing temp scripts owner-readable only. The safer rebase should retain the current fallback-user resolution semantics rather than reverting that work wholesale.
