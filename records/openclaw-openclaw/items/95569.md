## Summary

Keep open: the linked session/cron confusion is real and current main plus v2026.6.9 still have the old tool description, but this PR's wording is not merge-ready because it calls a main-session wait primitive "subagent-only" and the external PR has no real behavior proof.

## What This Changes

The PR changes the `sessions_yield` tool description to clarify that isolated cron completion does not resume a yielded session and suggests cron polling or `sessions_spawn` instead.

## Best Possible Solution

Revise the description to say `sessions_yield` waits for spawned sub-agent completion events and that isolated cron completion will not resume it; track any true cron wait/resume API as a separate maintainer-approved contract.

## Reproduction Assessment

Yes. Source inspection shows `sessions_yield` records a yielded turn without triggering a new turn, while isolated cron completion follows the detached cron path rather than the main-session heartbeat path; I did not run a live cron scenario in this read-only review.

## Solution Assessment

No. A docs/tool-text clarification is the right low-risk direction, but this exact wording is not the best fix because "subagent-only" conflicts with valid main-session use after `sessions_spawn`.
