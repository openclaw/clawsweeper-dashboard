## Summary

Keep open. The feature is plausible, but the patch still has a source-level correctness bug: when `preserveText` sends the transcript before voice media, the unchanged media fallback path can send that same transcript again on degradation or upload failure; the public config surface also needs maintainer approval.

## What This Changes

The PR adds a Feishu `tts.preserveText` boolean, updates reply delivery to send text with voice media when enabled, and adds dispatcher tests for enabled, default, and native-opus cases.

## Best Possible Solution

Land a backward-compatible TTS text-delivery policy only after fallback/degradation behavior is correct, docs/config are aligned, and maintainers accept Feishu-specific versus shared TTS policy scope.

## Reproduction Assessment

Yes for the PR blocker, at source level: the changed `shouldDeliverText` branch can send preserved text before media, then the unchanged fallback path still receives the same transcript. The requested preserve-text behavior itself is a new opt-in feature, not a broken existing contract.

## Solution Assessment

No, not yet. The opt-in can be reasonable, but the patch should track whether fallback text was already delivered and avoid duplicating it while preserving link-only media failure fallback where appropriate.

## Review Findings

Overall correctness: patch is incorrect
