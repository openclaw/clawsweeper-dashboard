## Summary

Keep open: the request is plausible and real behavior proof was supplied, but the patch introduces a duplicate-text fallback when preserved voice text has already been sent.

## What This Changes

The PR adds a Feishu `tts.preserveText` option, changes voice-media text delivery when enabled, and adds dispatcher coverage for enabled/default/native-voice cases.

## Best Possible Solution

Land a backward-compatible TTS text-preservation policy only after fallback/degradation behavior is correct and maintainers accept the Feishu-specific versus generic policy scope.

## Reproduction Assessment

Yes, at source level: current main suppresses Feishu voice text, and the PR's duplicate fallback follows from the changed `shouldDeliverText` gate plus the unchanged `fallbackText` path. I did not run live Feishu or Vitest in this read-only review.

## Solution Assessment

No, not yet. The opt-in can be reasonable, but the patch needs to stop treating already-delivered `preserveText` text as skipped fallback text.
