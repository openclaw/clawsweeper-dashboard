## Summary

Keep open because the PR still introduces a source-level duplicate-text bug in Feishu voice fallback paths, while the requested preserve-text feature is not implemented on current main and the linked feature request remains open.

## What This Changes

The PR adds a Feishu `tts.preserveText` config flag, changes the reply dispatcher to send text when voice media is present, and adds dispatcher tests for enabled, default, and native-opus cases.

## Best Possible Solution

Land a backward-compatible TTS text-delivery policy only after fallback/degradation behavior avoids duplicate transcripts and the chosen Feishu-specific or generic config surface is accepted and documented.

## Reproduction Assessment

Yes, at source level: current main suppresses Feishu text for voice media and preserves it as fallback, and the PR’s changed delivery gate leaves the fallback path able to resend already-delivered text. I did not run live Feishu or Vitest in this read-only review.

## Solution Assessment

No, not yet. The opt-in direction is plausible, but the patch must gate fallback text when `preserveText` already delivered the transcript, and maintainers still need to accept the public config shape.
