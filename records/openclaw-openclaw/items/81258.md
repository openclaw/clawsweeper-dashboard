## Summary

Keep open: the PR is a focused implementation candidate with sufficient real-behavior proof, but it still needs a fallback-path fix because enabling preserved text would duplicate that text when Feishu voice media degrades or upload fallback runs.

## What This Changes

The PR adds a Feishu `tts.preserveText` config option, lets the reply dispatcher send text alongside voice media when it is enabled, and adds dispatcher tests for enabled/default/native-voice behavior.

## Best Possible Solution

Land the feature only after preserved text is made idempotent across success, degradation, and upload-failure paths, and after maintainers accept the Feishu-specific config surface or replace it with a shared TTS delivery policy.

## Reproduction Assessment

Yes, at source level: current main suppresses Feishu text for voice media, and the PR's duplicate fallback follows from changing `shouldDeliverText` while leaving `fallbackText` active. I did not run live Feishu or Vitest in this read-only review.

## Solution Assessment

No, not yet. The opt-in approach is plausible, but the patch must stop treating already-delivered preserved text as skipped fallback text and maintainers should confirm the final config policy.
