## Summary

Keep open. The Feishu opt-in is plausible and the PR now supplies after-fix Feishu proof, but the patch still treats the transcript as skipped fallback text after `preserveText` has already sent it, so degraded or failed voice sends can duplicate the text.

## What This Changes

The PR adds a Feishu `tts.preserveText` config option, changes reply delivery to send text alongside voice media when enabled, and adds dispatcher tests for enabled, default, and native-voice behavior.

## Best Possible Solution

Land a backward-compatible TTS text-delivery policy only after fallback/degradation behavior is correct, the chosen config/docs surface is aligned, and maintainers accept Feishu-specific versus generic policy scope.

## Reproduction Assessment

Yes, at source level: current main and tests show Feishu suppresses text for voice media, and the PR's fallback duplicate follows from the changed `shouldDeliverText` branch into the unchanged `fallbackText` path. I did not run live Feishu or Vitest in this read-only review.

## Solution Assessment

No, not yet. The opt-in can be reasonable, but the patch must stop treating already-delivered `preserveText` text as skipped fallback text, and maintainers should confirm whether the policy is Feishu-specific or generic.

## Review Findings
