## What This Changes

The PR adds a Feishu `tts.preserveText` boolean to the TTS config schema, allows text delivery alongside voice media when enabled, and adds dispatcher tests for enabled/default/native-voice cases.

## Best Possible Solution

Land the opt-in Feishu preserve-text behavior after making media fallback conditional on whether text was already delivered, adding preserveText failure/degrade tests, and documenting the accepted config key.

## Reproduction Assessment

Yes, by source inspection: current main suppresses text whenever `hasVoiceMedia` is true, and the PR's merge result then sends text before still passing `fallbackText` into the voice-media fallback path. A focused test can reproduce the review finding by enabling `tts.preserveText` and making `sendMediaFeishu` reject or return `voiceIntentDegradedToFile`.

## Solution Assessment

No, not as submitted; the success path is narrow, but the media fallback path still assumes voice text was skipped. The safer solution is to carry whether text was already delivered into `sendMediaReplies` and only provide fallback text for skipped-text cases.
