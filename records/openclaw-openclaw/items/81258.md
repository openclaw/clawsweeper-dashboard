## Summary

Keep open. This is a plausible Feishu TTS feature PR, but the current patch has blocking source-level issues and the external PR only provides mock/unit-test proof, not real Feishu behavior proof.

## What This Changes

The PR adds a Feishu `tts.preserveText` boolean, uses it to allow text delivery alongside voice media, and adds dispatcher unit coverage.

## Best Possible Solution

Fix the PR so the enabled config is actually exercised, preserve fallback text only when text was not already delivered, then have maintainers decide whether this belongs as Feishu-specific `tts.preserveText` or a generic TTS delivery policy.

## Reproduction Assessment

Yes. Source inspection reproduces the PR defects: the new enabled tests overwrite their own `preserveText` config, and the production fallback path can resend text after preserveText already delivered it; I did not run tests because this review is read-only.

## Solution Assessment

No. The proposed direction may be acceptable after maintainer API review, but this patch needs test harness repair and fallback gating before it is a safe implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep the preserveText config in the enabled tests:** `extensions/feishu/src/reply-dispatcher.test.ts:1527`
  - body: The enabled preserveText tests set `resolveFeishuAccountMock` with `tts: { preserveText: true }`, but then call `setupNonStreamingAutoDispatcher()`, which immediately replaces the mock account config with one that has no `tts` block. These tests therefore do not exercise the enabled path and should fail once run; pass the config through the helper or use `createDispatcherHarness()` after setting the mock.
  - confidence: 0.97
- **[P2] Avoid resending fallback text after preserveText sends it:** `extensions/feishu/src/reply-dispatcher.ts:555`
  - body: When `preserveText` makes `shouldDeliverText` true, the dispatcher sends the text before media, but the later `sendMediaReplies()` call still receives `fallbackText` for every voice-media payload. If Feishu degrades the voice to a file or the media send errors, that fallback path sends the same text again. Gate fallback text on text not already being delivered, or track that it was already sent.
  - confidence: 0.9
