## Summary

Current main and the shipped v2026.5.2 release no longer match the reported 2026.4.29 behavior: Discord typing starts through the shared nonblocking typing callback, hung typing requests are bounded, and the Discord REST scheduler has follow-up coverage for queue stalls that could delay channel REST work.

## What This Changes

Fix a Discord plugin regression where the typing indicator could appear 20-35 seconds after an inbound message on 2026.4.29 while the reply followed normally.

## Best Possible Solution

Keep the shipped v2026.5.2 behavior and ask for a new current-version report with Discord REST scheduler metrics if a similar delay is still observable.

## Reproduction Assessment

No high-confidence failing current-main reproduction remains. Source comparison shows the 2026.4.29 path awaited typing startup, while current main starts typing asynchronously and has regression coverage for that behavior.

## Solution Assessment

Yes. The current solution fixes the shared typing lifecycle and Discord REST queue behavior rather than adding a Discord-only workaround, which is the narrower maintainable direction for this regression.
