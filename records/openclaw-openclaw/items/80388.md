## Summary

Keep open: this is a useful, well-proven feature direction, but the current patch is not merge-ready because the in-app plugin iframe bridge can hand the parent-mediated plugin fetch capability to a redirected or navigated external iframe document.

## What This Changes

The PR adds plugin-owned Control UI app navigation entry points across the plugin SDK, registry, gateway protocol/RPC/auth path, Control UI rendering, docs, protocol models, and tests.

## Best Possible Solution

Harden the iframe bridge so only the intended plugin-owned document receives the plugin fetch capability, then merge only after maintainers explicitly accept the plugin SDK/protocol API shape and browser auth model on the exact head.

## Reproduction Assessment

Yes for the review finding: source inspection shows the parent posts the plugin UI bridge port to the iframe with "*" on load, based on the frame element rather than proof of a plugin-owned loaded document. The original PR is a feature, so there is no current-main bug reproduction to run.

## Solution Assessment

No, not yet. The generic plugin registry/gateway/Control UI seam is the right layer, but the iframe bridge should be document-bound before this is the safest maintainable solution.

Full review comments:

- **[P1] Gate the iframe bridge to plugin-owned documents:** `ui/src/ui/app.ts:1049`
  - body: The parent installs the plugin UI MessagePort into whatever document is currently loaded in the iframe using `postMessage(..., "*")`.
