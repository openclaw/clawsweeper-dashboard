## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Export the entry-point type from SDK barrels:** `src/plugins/types.ts:2603`
  - body: The new public API method accepts `PluginControlUiEntryPoint`, but `src/plugin-sdk/plugin-entry.ts` still exports only `PluginControlUiDescriptor`. Plugin authors using documented `openclaw/plugin-sdk/*` entrypoints cannot import the named contract for the new method, so add it to the SDK barrels and refresh the API baseline.
  - confidence: 0.9
- **[P2] Preserve user activation for new-window launches:** `ui/src/ui/app-render.helpers.ts:277-284`
  - body: For `openMode: "new-window"`, this awaits `plugins.uiEntryPointLaunch` before eventually calling `window.open` through `openExternalUrlSafe`. That call is no longer in the direct click activation, so normal popup blockers can drop plugin UIs that request a new window; reserve or open the window synchronously, then navigate it after the launch token arrives.
  - confidence: 0.86
- **[P2] Align the iframe sandbox with authenticated plugin fetches:** `ui/src/ui/app-render.ts:227`
  - body: The in-app iframe is sandboxed without `allow-same-origin`, but the launch flow relies on same-origin plugin routes and an HttpOnly session cookie for follow-up plugin requests. Scripted plugin apps in that frame get an opaque origin, so relative fetch/XHR calls to `/plugins/<id>/...` will not use the intended same-origin credentials path; add a host-mediated request channel or adjust the iframe/auth model.
  - confidence: 0.78
