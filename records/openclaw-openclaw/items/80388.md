## Summary

Keep open: this is a plausible new plugin UI seam, but the proposed path validation is incomplete and the external proof does not show the visible Control UI behavior. It needs a small canonicalization fix and contributor-supplied browser proof before merge.

## What This Changes

The PR adds plugin-registered Control UI app-nav entry points, gateway launch-token plumbing, protocol types, Control UI rendering, docs, and tests.

## Best Possible Solution

Land a canonicalized plugin UI entry-point contract that rejects normalized path escapes, documents the new SDK/protocol surface, and includes visible real-browser proof.

## Reproduction Assessment

Yes for the blocking review finding: URL parsing normalizes dot-segment plugin paths outside the plugin root while the new source accepts the raw prefix. The requested entry-point capability itself is a feature, not a current-main bug reproduction.

## Solution Assessment

No. The feature direction is plausible, but the implementation should canonicalize or reject dot segments before trusting plugin-owned paths, and the PR should include visible proof for the browser UI path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject dot segments before accepting plugin UI paths:** `src/plugins/registry.ts:1726-1728`
  - body: `normalizeControlUiEntryPointPath` approves any raw path that starts with `/plugins/${pluginId}/`, so `/plugins/notes-plugin/../other-plugin/` passes even though browsers and `new URL()` normalize it to `/plugins/other-plugin/`. That can mint a launch URL for a different route than the registry approved, or fail auth after navigation; reject `.`/`..` segments, including encoded dot segments, or canonicalize before the prefix check and mirror the guard in `pathForPluginUiEntryPoint`.
  - confidence: 0.9
