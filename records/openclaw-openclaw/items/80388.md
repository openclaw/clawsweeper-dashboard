## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Export the entry-point type from SDK barrels:** `src/plugins/types.ts:2603`
  - body: The new public API method takes `PluginControlUiEntryPoint`, but the plugin SDK barrels still only expose `PluginControlUiDescriptor`. Plugin authors using the documented `openclaw/plugin-sdk/*` entrypoints cannot import the named contract for the new method, so please export it anywhere the descriptor type is exported and refresh the SDK API baseline.
  - confidence: 0.9
- **[P2] Preserve user activation for new-window launches:** `ui/src/ui/app-render.helpers.ts:277-284`
  - body: For `openMode: "new-window"`, this awaits `plugins.uiEntryPointLaunch` before eventually calling `window.open`. That call is no longer in the direct click activation, so normal popup blockers can drop plugin UIs that request a new window. Reserve/open the window synchronously on click, then navigate it after the launch token arrives.
  - confidence: 0.86
- **[P2] Align the iframe sandbox with authenticated plugin fetches:** `ui/src/ui/app-render.ts:227`
  - body: The in-app iframe is sandboxed without `allow-same-origin`, but the launch flow relies on same-origin plugin routes and an HttpOnly session cookie for follow-up plugin requests. Scripted plugin apps in that frame get an opaque origin, so relative fetch/XHR calls to `/plugins/<id>/...` are treated as cross-origin and will not use the intended same-origin credentials/CORS path. Either add a host-mediated request channel or adjust the iframe/auth model so authenticated plugin UIs can actually call their routes.
  - confidence: 0.78
