## Summary

Keep open. The latest head appears to address the earlier scoped-plugin cookie boundary concern, but it still has a root-path session bug and the external proof does not directly show the browser Control UI navigation and launch path working.

## What This Changes

The PR adds generic plugin Control UI entry-point registration, gateway listing and launch-token/session-cookie auth, protocol types, Control UI navigation/rendering, SDK docs, tests, and generated Swift protocol models.

## Best Possible Solution

Keep the generic seam, but fix root-path session matching, keep the scoped-id hardening, align the public contract/docs, and require direct browser proof before merge.

## Reproduction Assessment

Yes for the review finding: source inspection shows the registry accepts /plugins/<id>, while the session helper stores and checks /plugins/<id>/ only. A focused regression should launch /plugins/notes-plugin, consume the token, then reload /plugins/notes-plugin with the issued cookie.

## Solution Assessment

No. The overall seam is plausible, but the path handling should either normalize accepted entry paths to the slash form or safely match both the exact plugin root and descendants before this is a reliable plugin UI contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle root plugin entry paths in session cookies:** `src/gateway/plugin-ui-entry-launch-tokens.ts:82`
  - body: `registerControlUiEntryPoint` accepts an entry path equal to `/plugins/<id>`, but the launch session always stores `/plugins/<id>/` as the cookie/session prefix. After the one-time launch token is consumed, a reload or revisit of the exact root path will not send or accept that session cookie, so valid root entry points can fall back to 401. Normalize accepted entry paths to the slash form or make session matching cover both the exact root and descendants, with a regression for `/plugins/notes-plugin`.
  - confidence: 0.86
