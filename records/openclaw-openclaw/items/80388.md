## Summary

Keep open. This is useful feature work, not cleanup already present on main, but the current PR still has a blocking iframe bridge security issue and the visible Control UI launch flow needs stronger real behavior proof before merge.

## What This Changes

The PR adds plugin-owned Control UI app-navigation entry points across the plugin SDK/registry, gateway protocol, launch-token auth, generated clients, docs, and Control UI navigation/rendering.

## Best Possible Solution

Land the generic plugin UI entry-point contract after the iframe bridge is bound to the original plugin document or an unguessable channel, the auth/session model is reviewed, generated docs/types stay aligned, and real Control UI proof is attached.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature PR. Source inspection confirms current main lacks the entry-point API while the PR introduces the new contract and runtime path.

## Solution Assessment

No. The overall direction is maintainable, but the current bridge needs a stronger message-auth/navigation boundary before it is the safest implementation of authenticated in-app plugin UIs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Bind plugin bridge requests to the original document:** `ui/src/ui/app.ts:774-779`
  - body: The parent bridge trusts any message from the iframe's current WindowProxy and then fetches authenticated `/plugins/<activePluginId>/...` responses. A sandboxed plugin frame can navigate itself to another document while keeping that WindowProxy, letting the new page post `openclaw.pluginUi.request` messages and use the plugin session bridge. Bind requests to an unguessable per-launch token or MessagePort, or otherwise reject messages after cross-origin/frame navigation before proxying.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces an authenticated iframe request bridge that can be reused by a navigated iframe document unless the bridge is bound to the original plugin page.

Concerns:

- **[medium] Iframe bridge can survive navigation:** `ui/src/ui/app.ts:774`
  - body: `pluginUiMessageHandler` only checks `event.source` against the iframe WindowProxy, then `proxyPluginUiFrameRequest` performs same-origin credentialed fetches for plugin paths and replies with `targetOrigin: "*"`. Because a sandboxed iframe can navigate itself while preserving the WindowProxy, a non-OpenClaw document loaded in that frame could use the bridge for the active plugin's authenticated routes.
  - confidence: 0.78
