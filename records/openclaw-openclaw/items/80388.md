## Summary

Keep this PR open, but it is not ready to merge: the generic plugin UI seam is plausible, yet the default in-app frame/auth path has a blocking browser-runtime issue and the external-PR real behavior proof does not actually demonstrate the Control UI browser path.

## What This Changes

The PR adds generic plugin-owned Control UI entry points across the plugin SDK, registry, gateway protocol/auth, generated Swift protocol models, docs, and web UI navigation/rendering.

## Best Possible Solution

Keep the generic plugin UI entry-point direction, but define a browser-compatible in-app auth model for sandboxed plugin pages, split unrelated Codex normalization work, and require real browser proof of the entry appearing, launching, and making authenticated plugin calls.

## Reproduction Assessment

Yes, at source level. The PR renders default in-app plugin UIs in a sandboxed iframe without allow-same-origin while the new launch path relies on a same-site cookie for subsequent plugin-route auth.

## Solution Assessment

No. The seam is a reasonable direction, but this patch needs a maintained in-app iframe/auth design instead of relying on browser behavior that makes default sandboxed plugin pages unable to use their authenticated same-origin routes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Make in-app plugin frames usable for authenticated calls:** `ui/src/ui/app-render.ts:227`
  - body: The default `openMode` is `in-app`, but this iframe sandbox omits `allow-same-origin`. Sandboxed documents without that token get a special origin that fails same-origin checks and may block cookie/storage access, while fetch credentials default to same-origin and cross-origin credentialed reads need CORS; the new plugin UI session auth is an HttpOnly cookie under `/plugins/<id>/`, so a default in-app plugin page cannot reliably call its own authenticated plugin routes after the initial load. Please add a sandbox-safe authenticated bridge/CORS contract or change the in-app framing/auth model, with a browser test that proves an in-frame plugin page can make an authenticated request. ([developer.mozilla.org](https://developer.mozilla.org/docs/Web/HTML/Reference/Elements/iframe))
