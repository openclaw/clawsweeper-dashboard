## Summary

Keep open: the follow-up head appears to address the earlier trusted-operator scope-header issue, but the new launch-session cookie prefix is too broad for scoped plugin IDs and can cross-authenticate sibling plugins under the same scope.

## What This Changes

The PR adds plugin-owned Control UI entry point registration, gateway listing and launch-token/session-cookie auth, Control UI navigation/iframe rendering, SDK docs, tests, and generated Swift protocol models.

## Best Possible Solution

Keep the generic entry-point seam, but bind launch/session auth to the exact owning plugin ID/path for scoped and unscoped plugins, add a scoped sibling-plugin regression, and require direct browser proof before merge.

## Reproduction Assessment

Yes for the blocking finding: current source supports scoped plugin IDs, while the PR derives the new session prefix from only the first /plugins/ path segment. A focused regression would launch /plugins/@team/one/ and assert the resulting cookie cannot authenticate /plugins/@team/two/.

## Solution Assessment

No. The overall seam is plausible, but the launch/session auth should carry or derive the full owning plugin route root before this becomes a safe generic plugin UI contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Bind session cookies to the full plugin path:** `src/gateway/plugin-ui-entry-launch-tokens.ts:66-71`
  - body: Scoped plugin IDs are valid today, but this regex only captures the first segment after `/plugins`. A launch for `/plugins/@team/one/` therefore creates a session cookie scoped to `/plugins/@team/`, and `resolvePluginUiEntryPointSessionCookie` will accept that cookie for sibling routes such as `/plugins/@team/two/`. Store/derive the full owning plugin path prefix and add a scoped sibling-plugin regression.
  - confidence: 0.9
