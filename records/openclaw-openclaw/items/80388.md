## Summary

Keep open: current main does not have this plugin Control UI entry-point seam, and the PR is still an active feature/API change that needs review before merge. The latest head appears to address the earlier root-path session bug, but the in-app iframe auth model still has a blocking browser-runtime concern and the external proof is not direct enough for the Control UI launch path.

## What This Changes

The PR adds a plugin SDK Control UI entry-point API, gateway protocol/list/launch methods with token/session auth, Control UI navigation and iframe rendering, docs, tests, and generated client models.

## Best Possible Solution

Keep the generic plugin entry-point seam, but settle the browser-frame isolation and auth model, then require direct browser proof before merge.

## Reproduction Assessment

Yes for the review finding from source inspection: the PR combines cookie-backed plugin UI sessions with an in-app sandboxed iframe that omits allow-same-origin. A focused browser check should launch a real plugin entry and fetch or load a protected nested plugin asset/API route after the one-time token is consumed.

## Solution Assessment

No. The generic seam is plausible, but the default in-app launch path needs a coherent browser isolation/auth design before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Make in-app plugin frames compatible with launch auth:** `ui/src/ui/app-render.ts:227`
  - body: The launch flow authenticates the first plugin page with a one-time URL token, then relies on the path-scoped `openclaw_plugin_entry` cookie for subsequent plugin UI requests. But the default in-app iframe is sandboxed without `allow-same-origin`, so the framed plugin document gets an opaque origin and default same-origin asset/API requests back to `/plugins/<id>` will not use that cookie after the token is consumed. A real plugin UI can open once and then hit 401s for its own JS/API/navigation; choose an isolation/auth model that supports in-app plugin pages and cover it with a browser regression.
  - confidence: 0.78
