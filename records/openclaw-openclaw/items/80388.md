## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use an opaque-frame targetOrigin for the bridge:** `ui/src/ui/app.ts:1055`
  - body: The iframe is sandboxed without `allow-same-origin`, so the plugin document is treated as a special origin; sending the initial `MessagePort` with `window.location.origin` will not reach that opaque frame in real browsers. The earlier proofed head used `*`, but current head changed this line, so in-app plugin UIs can fail before they can issue bridged requests. ([developer.mozilla.org](https://developer.mozilla.org/en-US/docs/Web/API/Window/postMessage)) ([developer.mozilla.org](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/iframe))
  - confidence: 0.94
- **[P2] Export and validate the entry-point result contracts:** `packages/gateway-protocol/src/index.ts:895-897`
  - body: The PR defines `PluginsUiEntryPointsResultSchema` and `PluginsUiEntryPointLaunchResultSchema`, but the package entrypoint only imports/exports param schemas and only exposes param validators. That leaves clients without the same public result validators that `plugins.uiDescriptors` already has, and the new handlers also skip result validation before responding.
  - confidence: 0.9
