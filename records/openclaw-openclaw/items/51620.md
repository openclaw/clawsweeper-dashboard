## Summary

Keep open: current main still keeps followup queues process-local, drain-window enqueues still reject without durable inbound replay, and open implementation PRs are still under review rather than merged. ## What This Changes

The issue asks OpenClaw to persist and replay queued or in-flight inbound messages across gateway restarts.

## Best Possible Solution

Land one maintainer-approved durable restart recovery contract that persists accepted followups and drain-window arrivals, replays them through typed untrusted dispatch paths with idempotent dedupe, and leaves provider history catch-up/ACK behavior to narrower provider-owned work.

## Reproduction Assessment

Yes, source-level: a pending item in FOLLOWUP_QUEUES is process-local and disappears across a real process restart, while markGatewayDraining makes new command enqueues reject with GatewayDrainingError and current main has no durable pending-message replay path. I did not run a live channel restart reproduction in this read-only review.

## Solution Assessment

Unclear for the full request: persisting accepted followups is the narrow first repair, but drain-window arrivals, provider ACK/catch-up, replay ordering, and local-state security need a typed maintainer-approved contract rather than ad hoc replay.
