## Summary

Keep this issue open. Current main still keeps followup queue state in process memory, still rejects drain-window command enqueues with `GatewayDrainingError`, and has no durable pending inbound/followup replay path; open PRs #51623 and #51803 remain the linked implementation candidates rather than a reason to close the issue early.

## What This Changes

Request to persist queued and in-flight inbound or followup messages across gateway restarts and replay them after startup.

## Best Possible Solution

Keep the issue open until maintainers choose and land one durable Phase 1 followup handoff, with typed serialization, deletion/TTL, dedupe, and privacy handling. Then explicitly split or resolve the drain-window and provider catch-up pieces through narrower trackers such as #55792 and #74424.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: enqueue a followup item in `FOLLOWUP_QUEUES`, restart before it drains, and the process-local map is lost; during restart drain, `markGatewayDraining()` makes new command enqueues reject with `GatewayDrainingError`, which the current tests assert.

## Solution Assessment

Unclear for the full issue. Persisting followup queues is the right first step, but the complete solution should use a typed durable handoff with dedupe/TTL and channel-specific ACK or catch-up hooks where provider contracts support them.
