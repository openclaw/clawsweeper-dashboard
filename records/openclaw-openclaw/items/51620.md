## Summary

Keep open: current main still lacks durable followup-queue persistence/replay, and the linked closing PR plus the newer SQLite-based repair PR are both open and unmerged.

## What This Changes

The issue asks OpenClaw to preserve queued, in-flight, and restart-window inbound messages across gateway restarts instead of silently dropping them.

## Best Possible Solution

Land a focused shared-SQLite followup-queue recovery path, then keep broader offline channel catch-up tracked by https://github.com/openclaw/openclaw/issues/55792.

## Reproduction Assessment

Yes at source level: current main keeps followup queues in a process-global map, rejects new enqueues while draining, and has no durable followup replay store. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No, the issue's JSON sidecar sketch is not the best current shape because repository policy requires SQLite for runtime state. The newer shared-SQLite PR direction is closer, but maintainers still need to settle scope for drain-window replay and channel catch-up.
