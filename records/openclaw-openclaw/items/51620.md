## Summary

Keep open: current main still has process-local followup queues, drain-window command enqueues still reject without durable inbound replay, and two open PRs explicitly reference this issue as their closing target. The existing discussion also treats this as a real reliability problem, so the canonical issue should stay open until a maintainer-approved recovery path lands or the linked PRs are closed.

## What This Changes

Request durable recovery for queued followup and drain-window inbound messages across gateway restarts.

## Best Possible Solution

Land one maintainer-approved durable restart recovery contract that persists accepted followups and drain-window arrivals, replays them through typed untrusted dispatch paths, and leaves broader provider catch-up/ACK behavior to narrower follow-up work.

## Reproduction Assessment

Yes, source-level. A pending item in FOLLOWUP_QUEUES is process-local and disappears across a real process restart, while markGatewayDraining makes new command enqueues reject with GatewayDrainingError and current main has no durable pending-message replay path.

## Solution Assessment

Unclear for the full request. Persisting accepted followups is the narrow first repair, but drain-window arrivals and provider ACK/catch-up need a typed, idempotent, untrusted recovery contract rather than ad hoc system-event replay.
