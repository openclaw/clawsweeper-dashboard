## Summary

Keep open: current main still keeps followup queues process-local, still rejects new enqueues during gateway drain without durable inbound replay, and the open PRs that reference this issue are implementation candidates rather than merged proof.

## What This Changes

The issue requests durable handling for queued, in-flight, and restart-window inbound messages across gateway restarts.

## Best Possible Solution

Keep this issue open until one durable queued-work restart recovery design lands, then split or resolve the remaining drain-window, provider ACK/catch-up, and continuation edge cases through narrower trackers and PRs.

## Reproduction Assessment

Yes, at source level. A pending item in FOLLOWUP_QUEUES is process-local and disappears across a real process restart, while markGatewayDraining makes new command enqueues reject with GatewayDrainingError and no durable inbound replay path is present on current main.

## Solution Assessment

Unclear for the full request. Persisting followup queues or draining accepted buffers is the narrow first repair, but drain-window arrivals, trusted replay boundaries, idempotency, and provider ACK/catch-up behavior need a maintainable typed recovery contract rather than a generic pending file alone.
