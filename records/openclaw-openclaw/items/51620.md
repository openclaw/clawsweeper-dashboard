## Summary

Keep open: current main still keeps followup queues process-local, still rejects drain-window enqueues without durable inbound replay, and the linked Phase 1 PRs are open rather than merged proof.

## What This Changes

Persist and replay queued or in-flight inbound gateway messages across restarts instead of silently dropping them.

## Best Possible Solution

Land one maintainer-approved durable queued-work restart recovery path, then track provider catch-up, ACK, and coalesced-continuation edge cases through narrower issues or PRs.

## Reproduction Assessment

Yes, at source level. A pending item in FOLLOWUP_QUEUES is process-local and disappears across a real process restart, while markGatewayDraining makes new command enqueues reject with GatewayDrainingError and no durable inbound replay path is present on current main.

## Solution Assessment

Unclear for the full request. Persisting accepted followup queue items is the narrow first repair, but rejected drain-window arrivals and provider ACK/catch-up behavior need a typed, idempotent recovery contract rather than an ad hoc generic pending file alone.
