## Summary

Keep open: current main still does not durably persist followup queue items or drain-window inbound messages across a gateway restart, and an open PR with closing syntax is still the implementation candidate rather than a reason to close the issue early.

## What This Changes

The issue asks OpenClaw to prevent silent inbound message loss across gateway restarts by persisting pending followup/inbound queue state and replaying it after startup.

## Best Possible Solution

Land a maintainer-reviewed durable queue design, preferably the SQLite-backed followup persistence direction, then track drain-window inbound capture and channel ACK or catch-up behavior as explicit follow-up work instead of closing this issue early.

## Reproduction Assessment

Yes, at source level. Current main keeps followup queues in a process-global map, rejects new lane work with GatewayDrainingError once restart draining starts, and only persists restart sentinel notices or continuations rather than queued inbound messages.

## Solution Assessment

Unclear as originally proposed. Persisting to an ad hoc JSON file conflicts with current storage guidance; the cleaner direction is a SQLite-backed queue/recovery path with separate maintainer decisions for drain-window capture and provider ACK semantics.
