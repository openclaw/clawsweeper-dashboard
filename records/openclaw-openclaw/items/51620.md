## Summary

Keep this open: current main still keeps followup queue items in process memory, still rejects new command enqueues during restart drain, and the linked Phase 1 PRs remain open implementation candidates rather than a reason to close the issue early.

## What This Changes

The issue asks OpenClaw to durably preserve and replay queued, in-flight, and restart-window inbound messages across gateway restarts.

## Best Possible Solution

Keep this issue open until maintainers land one durable Phase 1 followup handoff, then explicitly split or resolve drain-window and provider catch-up behavior through narrower trackers such as #55792 and #74424.

## Reproduction Assessment

Yes, at source level. A pending item in FOLLOWUP_QUEUES is process-local and disappears across process restart, while markGatewayDraining makes new command enqueues reject with GatewayDrainingError; I did not run a live provider restart test in this read-only pass.

## Solution Assessment

Unclear for the full issue. Persisting followup queues is the narrow first repair, but drain-window arrivals and provider downtime catch-up need typed durable handoff plus channel-specific ACK or catch-up contracts rather than only a generic pending file.
