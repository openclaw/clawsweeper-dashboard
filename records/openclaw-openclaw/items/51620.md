## Summary

Keep this open: current main still has process-local followup queues, still rejects new command enqueues during restart drain without durable inbound replay, and the open Phase 1 PRs are implementation candidates rather than a reason to close this issue before merge.

## What This Changes

Persist and replay inbound followup work across gateway restarts so queued, drain-window, and downtime messages are not silently lost.

## Best Possible Solution

Keep this issue open until one durable Phase 1 followup handoff lands, then split or resolve the broader drain-window and provider catch-up behavior through narrower trackers such as #55792 and #74424.

## Reproduction Assessment

Yes, at source level. A pending item in FOLLOWUP_QUEUES is process-local and disappears across a real process restart, while markGatewayDraining makes new command enqueues reject with GatewayDrainingError; I did not run a live provider restart test in this read-only pass.

## Solution Assessment

Unclear for the full issue. Persisting followup queues is the narrow first repair, but drain-window arrivals and provider downtime catch-up need a typed durable handoff plus channel-specific ACK or catch-up contracts rather than only a generic pending file.
