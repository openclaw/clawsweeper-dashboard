## Summary

Keep open: current main still keeps followup queue state process-local, still rejects drain-window command enqueues during restart, and the durable recovery work is still in open pull requests rather than merged code.

## What This Changes

The issue asks OpenClaw to preserve queued, in-flight, and drain-window inbound messages across gateway restarts and replay them after startup.

## Best Possible Solution

Land or replace the open work with a maintainer-approved SQLite-backed durable followup/inbound replay path, then split offline catch-up and channel ACK semantics into narrower follow-ups if needed.

## Reproduction Assessment

Yes, at source level: current main keeps followup queues in a process-global Map and rejects new command-lane enqueues once restart draining starts. I did not run a live channel restart in this read-only review.

## Solution Assessment
