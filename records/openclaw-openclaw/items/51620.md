## Summary

Keep open: current main and v2026.5.22 still keep followup queues in process memory and reject drain-window enqueues without durable replay. The linked implementation PRs remain open, so the issue should stay available until maintainers choose and land one restart-recovery contract.

## What This Changes

The issue requests durable recovery for queued, in-flight, and drain-window inbound messages across gateway restarts.

## Best Possible Solution

Land one maintainer-approved restart recovery contract that persists accepted followups and drain-window arrivals, replays them through typed untrusted dispatch paths with dedupe, and leaves provider history catch-up or ACK policy to narrower provider-owned work.

## Reproduction Assessment

Yes, source-level: FOLLOWUP_QUEUES is process-local, enqueueFollowupRun appends to memory only, and markGatewayDraining makes new command enqueues reject with GatewayDrainingError without durable replay.

## Solution Assessment

Unclear for the full request: persisting accepted followups is the narrow first repair, but drain-window arrivals, provider ACK/catch-up, replay ordering, dedupe, and local-state security need a typed maintainer-approved contract.
