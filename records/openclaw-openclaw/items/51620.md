## Summary

Keep open: current main and the latest release still keep followup queues in process memory, append accepted followups to that memory-only queue, and reject drain-window command enqueues with GatewayDrainingError rather than durably replaying them. The direct closing PR for this issue and adjacent restart-recovery PRs remain open, so this issue is still needed until maintainers choose and land one durable replay and ACK contract.

## What This Changes

The issue requests durable preservation and replay of queued, in-flight, and drain-window inbound messages across gateway restarts.

## Best Possible Solution

Land one maintainer-approved restart recovery contract that persists accepted followups and drain-window arrivals, replays them through typed dispatch with bounded dedupe, and leaves provider history catch-up or ACK policy to explicit provider-owned follow-up where needed.

## Reproduction Assessment

Yes, source-level: current main still stores followup queue items in a process-local Map, appends accepted followups only to memory, and rejects new command enqueues during gateway draining. I did not run a live channel restart because this review is read-only and the source/test contract is already clear.

## Solution Assessment

Unclear for the final design: persisting accepted followups is a necessary slice, but the full issue needs a maintainer-owned replay and ACK contract covering drain-window and downtime arrivals without duplicate or stale message delivery.
