## Summary

Keep open: current main and v2026.6.8 still keep follow-up queue state in process memory and reject drain-window enqueues, while linked implementation PRs remain open and the broader drain-window/channel-ack behavior still needs maintainer direction.

## What This Changes

Persist and replay inbound or follow-up messages around gateway restarts so queued, drain-window, and downtime messages are not silently dropped.

## Best Possible Solution

Review and land a SQLite-backed durable follow-up queue recovery path, then track drain-window arrivals and channel catch-up or ACK semantics as explicit follow-up work if they are not included.

## Reproduction Assessment

Yes, at source level: current main keeps follow-up queues in process memory and rejects new enqueues after gateway draining starts. I did not run a live channel restart repro in this read-only review.

## Solution Assessment

No, the issue's JSON sidecar sketch is not the best final shape because repository policy prefers SQLite-backed runtime state and doctor-owned migration. The active SQLite PR is closer, but it still needs maintainer review and proof before this issue can close.
