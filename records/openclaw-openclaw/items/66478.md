## Summary

Keep open: the PR addresses a valid Feishu reaction bug, but the current head still needs changes because several normalized values do not match Feishu's documented emoji_type vocabulary, the channel import bypasses the existing lazy runtime boundary, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR adds Feishu reaction emoji normalization for Unicode/pass-through inputs, wires it into add/list/react paths, and adds reaction helper tests.

## Best Possible Solution

Repair this PR with exact documented Feishu emoji_type values, keep normalization behind a runtime-safe Feishu helper path, add focused regression coverage plus a changelog entry, then land it to close the linked Feishu reaction bug.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: a Unicode message action react emoji is passed unchanged into Feishu reaction_type.emoji_type, which Feishu documents as requiring a supported emoji_type value and rejects with 231001 when invalid.

## Solution Assessment

No. Normalizing inside the Feishu plugin is the right direction, but this patch should use Feishu's exact documented values, avoid eager runtime imports from channel.ts, and include the required changelog entry.
