## Summary

Keep open: current main still has a source-reproducible Feishu reaction failure path, but this PR should not merge yet because PR head still maps several emoji to non-documented Feishu values, crosses the Feishu lazy runtime boundary, and does not show full-path Feishu reaction proof.

## What This Changes

The PR adds a Feishu emoji normalizer, applies it before add/list reaction calls and the channel react handler, and adds focused Feishu reaction tests.

## Best Possible Solution

Move Feishu emoji normalization into a runtime-free Feishu helper that uses exact documented emoji_type values, share it from channel and reaction helpers, keep focused tests, and add redacted full-path Feishu reaction proof.

## Reproduction Assessment

Yes, at source level. Current main passes raw `ctx.params.emoji` through to `reaction_type.emoji_type`, and Feishu documents 231001 when that value is not a supported emoji_type.

## Solution Assessment

No. Normalizing inside the Feishu plugin is the right direction, but this patch needs exact documented values and a runtime-free helper before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Use Feishu's exact emoji_type strings:** `extensions/feishu/src/reactions.ts:131-194`
  - body: The normalizer invents several values that Feishu does not document, including `THUMBSDOWN`, `LAUGHING`, `GRINNING`, `FIRE`, `CHECK`, and `CROSS`. Feishu's reaction API is validated against exact emoji_type values such as `ThumbsDown`, `LAUGH`, `BLUSH`, `Fire`, `CheckMark`, and `CrossMark`, so these mapped inputs can still return 231001 instead of fixing the reaction failure.
  - confidence: 0.97
- **[P2] Keep emoji normalization out of the runtime module:** `extensions/feishu/src/channel.ts:72`
  - body: Importing `normalizeFeishuEmoji` from `reactions.ts` at channel module load also evaluates `client.ts`, whose module body rewrites the Lark SDK HTTP interceptor. Extract the pure emoji map into a runtime-free helper so Feishu plugin loading does not cross the existing lazy runtime boundary before an API action actually needs it.
  - confidence: 0.91
