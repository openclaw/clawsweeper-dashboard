## Summary

Keep this PR open: it targets a real Feishu reaction bug, but the current patch still has blocking correctness issues around Feishu's documented emoji_type vocabulary and the plugin's lazy runtime boundary. The supplied proof is useful, but it does not yet show the full OpenClaw Feishu reaction path after the fix.

## What This Changes

The PR adds Feishu reaction emoji normalization in the channel and reaction helpers, plus tests for Unicode conversion, pass-through, trimming, and add/list request serialization.

## Best Possible Solution

Land a Feishu-local, runtime-free emoji normalizer that preserves exact Feishu emoji_type spelling, normalizes add/list/remove reaction paths, and includes focused tests plus redacted full-path proof.

## Reproduction Assessment

Yes, at source level. The linked issue gives concrete Feishu react steps and current main shows raw Unicode emoji reaches reaction_type.emoji_type unchanged, matching Feishu's documented 231001 invalid-type failure.

## Solution Assessment

No. Normalizing inside the Feishu plugin is the right direction, but this patch needs exact documented Feishu values and a runtime-free helper before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Return only documented Feishu emoji_type values:** `extensions/feishu/src/reactions.ts:163-190`
  - body: The normalizer treats values like `THUMBSDOWN`, `FIRE`, `CHECK`, `CROSS`, `LAUGHING`, and `GRINNING` as valid Feishu types, but the official table uses different exact spellings such as `ThumbsDown`, `Fire`, `CheckMark`, `CrossMark`, `LAUGH`, and `BLUSH`. Those Unicode paths can still send invalid `emoji_type` values and keep returning 231001.
  - confidence: 0.97
- **[P2] Keep emoji normalization behind the lazy runtime boundary:** `extensions/feishu/src/channel.ts:72`
  - body: Importing `normalizeFeishuEmoji` from `reactions.ts` at channel module load pulls in `client.ts`, whose module body rewrites the Lark SDK HTTP interceptor. Move the pure emoji map into a runtime-free helper, or keep it behind the existing lazy runtime path, so static Feishu plugin loading does not eagerly evaluate client runtime side effects.
  - confidence: 0.91
