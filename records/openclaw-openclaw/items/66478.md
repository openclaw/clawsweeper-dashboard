## Summary

Keep open. The PR targets a real Feishu reaction bug, but the current patch is not merge-ready because several normalized values do not match Feishu's documented `emoji_type` table and the new top-level import crosses the existing lazy runtime boundary.

## What This Changes

This PR adds Feishu reaction emoji normalization for add/list/generic `react` paths and adds tests for unicode conversion and pass-through behavior.

## Best Possible Solution

Use one Feishu-local lightweight normalizer with exact official `emoji_type` values, preserve already-valid Feishu strings, keep runtime-heavy client imports behind the lazy boundary, and align any adjacent Feishu reaction-tool work to the same helper.

## Reproduction Assessment

Yes, at source level. Current main passes raw emoji strings into Feishu `reaction_type.emoji_type`, and Feishu's official table requires documented type strings instead of unicode emoji.

## Solution Assessment

No, not as currently written. Normalizing at the Feishu reaction API boundary is the right direction, but the PR must use exact documented values and avoid pulling runtime-heavy client code into channel module load.
