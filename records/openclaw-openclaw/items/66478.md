## Summary

Keep open: this PR is the active canonical fix path for a real Feishu reaction bug, but the current patch still maps several Unicode inputs to undocumented Feishu emoji_type values and crosses the existing lazy runtime boundary.

## What This Changes

The PR adds Feishu reaction emoji normalization in `channel.ts` and `reactions.ts`, plus focused tests for Unicode-to-Feishu emoji_type conversion.

## Best Possible Solution

Move the Feishu emoji normalizer into a runtime-free Feishu helper, use exact documented emoji_type values, share it from the channel and reaction helpers, and land it with focused tests plus redacted full-path reaction proof.

## Reproduction Assessment

Yes, at source level. The linked issue gives concrete `message(action="react", emoji="👍")` steps, current main forwards that raw value into `reaction_type.emoji_type`, and Feishu documents 231001 for invalid reaction types.

## Solution Assessment

No. Feishu-owned normalization is the right direction, but this patch needs exact Feishu emoji_type values and a runtime-free helper before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Use Feishu's documented emoji_type strings:** `extensions/feishu/src/reactions.ts:130-186`
  - body: The new map normalizes several Unicode inputs to values Feishu does not document, such as `THUMBSDOWN`, `LAUGHING`, `GRINNING`, `FIRE`, `CHECK`, `CROSS`, and `PRAY`. Feishu validates exact emoji_type strings, so these normalized calls can still return 231001 instead of fixing the reaction failure.
  - confidence: 0.97

  - body: Importing `normalizeFeishuEmoji` from `reactions.ts` at channel module load also evaluates `client.ts`, whose module body rewrites the Lark SDK HTTP interceptor. Extract the pure emoji map into a runtime-free helper and import that from both `channel.ts` and `reactions.ts` so the existing lazy runtime boundary stays intact.
  - confidence: 0.91
