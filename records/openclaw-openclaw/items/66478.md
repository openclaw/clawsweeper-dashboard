## Summary

Keep open: the linked Feishu reaction bug is valid and this PR is the canonical fix path, but the current diff still sends several values that do not match Feishu's documented case-sensitive emoji_type table and also pulls the reaction runtime into the channel module at load time.

## What This Changes

This PR adds Feishu reaction emoji normalization in channel.ts and reactions.ts plus tests so Unicode reaction inputs are converted before add/list API calls.

## Best Possible Solution

Land a Feishu-local, runtime-free normalizer that maps Unicode inputs to exact documented Feishu emoji_type values at the reaction API boundary, with regression tests for the reported Unicode case and mixed-case documented values.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: the linked issue passes emoji="👍", current main forwards that string into reaction_type.emoji_type, and Feishu documents 231001 for invalid reaction types.

## Solution Assessment

Is this the best way to solve the issue? No. A Feishu-owned normalization layer is the right direction, but this PR should use exact documented emoji_type values and keep the pure normalizer out of the runtime-heavy reactions module before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Use documented Feishu emoji_type values:** `extensions/feishu/src/reactions.ts:134-150`
  - body: The new table treats several reactions as all-uppercase, but Feishu's emoji table is case-sensitive and lists values such as `ThumbsDown`, `CheckMark`, `CrossMark`, and `Fire`, while the list API example uses `LAUGH`. Because these mapped values are sent directly, inputs like `👎`, `🔥`, or an already-valid `Fire` can still return 231001 instead of being fixed.
  - confidence: 0.97
- **[P2] Keep the normalizer out of the runtime module:** `extensions/feishu/src/channel.ts:72`
  - body: Importing `normalizeFeishuEmoji` from `./reactions.js` at channel module load evaluates `reactions.ts`, which imports `client.ts`; `client.ts` mutates the Lark SDK HTTP interceptor during module load. That bypasses the existing `loadFeishuChannelRuntime()` split for this channel, so move the pure emoji map into a lightweight helper imported by both `channel.ts` and `reactions.ts`.
  - confidence: 0.92
