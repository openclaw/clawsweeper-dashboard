## Summary

Keep open: the PR targets a real Feishu reaction bug that is still present on current main and the latest release, but the patch is not merge-ready because several mappings do not match Feishu's documented `emoji_type` values and the helper import bypasses the existing lazy runtime boundary.

## What This Changes

This PR adds Feishu reaction emoji normalization in `channel.ts` and `reactions.ts`, plus focused tests for normalization and reaction helper request bodies.

## Best Possible Solution

Land a Feishu-local normalizer that maps Unicode inputs to exact documented Feishu `emoji_type` values from a runtime-free helper shared by the action handler and reaction helpers, with focused tests and redacted full-path Feishu proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, at source level: the linked report gives `message(action="react", emoji="👍")`, current main forwards that string into `reaction_type.emoji_type`, and Feishu documents code 231001 for invalid values. I did not run a live tenant repro in this read-only review.

## Solution Assessment

No. A Feishu-owned normalization layer is the right direction, but this patch should use exact documented Feishu values and keep the pure normalizer out of the runtime-heavy module before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Map Unicode reactions to documented Feishu types:** `extensions/feishu/src/reactions.ts:132-148`
  - body: Feishu's official emoji table is not an all-uppercase vocabulary: it lists values such as `ThumbsDown`, `LAUGH`, `Fire`, `CheckMark`, and `CrossMark`, and does not list several of the new uppercase entries. Because these normalized values are sent directly, Unicode inputs like thumbs-down, laughing, fire, check, and cross can still return 231001 instead of being fixed.
  - confidence: 0.97
- **[P2] Keep normalization behind a runtime-free boundary:** `extensions/feishu/src/channel.ts:72`
  - body: Importing `normalizeFeishuEmoji` from `./reactions.js` at channel module load evaluates `reactions.ts`, which imports `client.ts`; `client.ts` mutates the Lark SDK HTTP interceptor at module load. That regresses the existing lazy runtime split, so move the pure emoji map into a lightweight helper imported by both `channel.ts` and `reactions.ts`.
  - confidence: 0.92
