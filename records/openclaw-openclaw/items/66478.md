## Summary

Keep open for maintainer review/merge rather than cleanup close: current main still sends raw Feishu reaction emoji values, while this PR is a focused request-boundary fix with regression coverage and no line-level correctness findings.

## What This Changes

The PR adds a Feishu emoji_type normalizer, applies it to reaction add/list and the channel react handler, and adds regression tests for Unicode, documented casing, GLANCE, and request payloads.

## Best Possible Solution

Land this narrow Feishu-owned normalizer if maintainers accept the proof level, then require the broader reaction-tool branch to reuse this canonical mapping instead of creating a second source of truth.

## Reproduction Assessment

Yes, source-reproducible: current main trims `ctx.params.emoji` in `extensions/feishu/src/channel.ts` and passes it through to `addReactionFeishu`/`listReactionsFeishu`, while `extensions/feishu/src/reactions.ts` serializes that value directly into Feishu `emoji_type`. The linked issue reports Feishu code 231001 for raw Unicode reactions with valid reaction scopes.

## Solution Assessment

Yes, this is the best narrow fix shape: the Feishu plugin owns the API vocabulary, the pure map avoids the channel runtime import side effect, and the runtime helpers normalize at the request-builder boundary so direct and channel callers share behavior. The broader reaction-tool PR should consume this same helper rather than replace this bug fix.
