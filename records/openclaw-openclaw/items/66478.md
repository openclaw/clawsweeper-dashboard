## Summary

Keep open: this PR targets a valid Feishu reaction bug, but it still emits undocumented `emoji_type` values and imports reaction runtime code at Feishu channel module load. The supplied proof is useful, but it does not exercise the full OpenClaw Feishu reaction path or a live Feishu/Lark API result after the fix.

## What This Changes

The PR adds Feishu reaction emoji normalization in the channel and reaction helpers, plus tests for Unicode conversion and request/list serialization.

## Best Possible Solution

Land a Feishu-local normalizer that uses the official emoji_type vocabulary from a runtime-free helper shared by channel and reaction helpers, with focused tests and redacted full-path proof.

## Reproduction Assessment

Yes, at source level. Current main passes raw Unicode emoji into `reaction_type.emoji_type`, and Feishu documents 231001 when that value is not a valid emoji_type.

## Solution Assessment

No. Normalizing inside the Feishu plugin is the right direction, but this patch needs exact documented values and a runtime-free helper before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Use documented Feishu emoji_type values:** `extensions/feishu/src/reactions.ts:131-194`

  - confidence: 0.97

Extract the pure emoji map into a runtime-free helper so static Feishu channel loading does not eagerly evaluate client runtime code.
  - confidence: 0.91
