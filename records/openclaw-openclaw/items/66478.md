## Summary

Keep this PR open: it targets a real Feishu reaction bug, but the current head still has blocking correctness issues in the documented emoji mapping and the Feishu lazy runtime boundary, and the supplied proof does not exercise the full reaction path.

## What This Changes

The PR adds Feishu Unicode emoji normalization in `extensions/feishu/src/reactions.ts`, wires it into the Feishu reaction action path, and adds focused reaction helper tests.

## Best Possible Solution

Land a Feishu-local normalizer that maps only to exact documented `emoji_type` values from a runtime-free helper shared by `channel.ts` and `reactions.ts`, with focused tests and redacted full-path Feishu reaction proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, at source level. The linked report gives a concrete `message(action="react", emoji="👍")` path, current main forwards that raw value into Feishu `reaction_type.emoji_type`, and Feishu documents 231001 for invalid reaction types.

## Solution Assessment

No. A Feishu-owned normalization layer is the right direction, but this PR should use exact documented Feishu values and keep the pure normalizer out of the runtime-heavy module before it is the narrow maintainable fix.

## Review Findings

- **[P2] Map only to documented Feishu emoji types:** `extensions/feishu/src/reactions.ts:134-152`
  - body: The new lookup marks values such as `THUMBSDOWN`, `LAUGHING`, `FIRE`, `CHECK`, `CROSS`, and `PRAY` as canonical, but Feishu's emoji table uses exact strings such as `ThumbsDown`, `LAUGH`, `Fire`, `CheckMark`, and `CrossMark` and does not list several of these uppercase values.   - confidence: 0.97

That bypasses the existing `loadFeishuChannelRuntime` lazy boundary, so extract the pure emoji map into a runtime-free helper and import that from both files.
  - confidence: 0.92
