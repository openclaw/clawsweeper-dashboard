## Summary

Keep open: the PR targets a valid Feishu reaction bug, but the current branch still maps several inputs to emoji_type values that are not in Feishu's official catalog and it pulls the runtime reaction helper into channel module load. The supplied proof is a terminal smoke of an extracted normalizer slice, not a real Feishu/Lark reaction API success path.

## What This Changes

The PR adds Feishu emoji normalization for reaction add/list/react paths and adds tests for Unicode conversion, type pass-through, trimming, fallback, and request-body helpers.

## Best Possible Solution

Normalize in the Feishu plugin with exact official emoji_type values in a runtime-free helper shared by channel and reaction helpers, keep the lazy runtime boundary intact, and verify with focused tests plus redacted real Feishu/Lark behavior proof.

## Reproduction Assessment

Yes, at source level: current main passes a user-provided Unicode emoji unchanged into Feishu reaction_type.emoji_type, and Feishu documents 231001 for invalid reaction type values. I did not establish a live failing Feishu account run in this read-only review.

## Solution Assessment

No. Normalizing inside the Feishu plugin is the right direction, but this branch should use Feishu's exact documented emoji_type values and avoid importing client-backed reaction helpers during channel module load.
