## Summary

Keep open. Current main still forwards raw Feishu reaction emoji unchanged, and this PR is the right narrow area, but the branch is not merge-ready because the patch still bypasses the Feishu lazy runtime boundary, uses several undocumented emoji_type values, and is now dirty against current main.

## What This Changes

The PR adds Feishu reaction emoji normalization, applies it to add/list/react paths, adds reaction helper tests, and records the user-facing fix in CHANGELOG.md.

## Best Possible Solution

Land a Feishu-plugin-only fix on current main that normalizes Unicode reactions to documented Feishu emoji_type values inside a runtime-safe helper path, with focused tests and the existing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, message action=react with a Unicode emoji reaches reaction_type.emoji_type unchanged, matching the linked 231001 reports.

## Solution Assessment

No, not as currently patched. Normalizing in the Feishu plugin is the right direction, but the final patch should use Feishu's documented vocabulary, preserve the lazy runtime boundary, and be rebased onto current main.

Move the normalizer to a runtime-free helper or rely on normalization inside the lazy add/list helpers.
  - confidence: 0.9

  - body: Current main no longer has the FeishuEmoji table in reactions.ts, while this branch still builds knownFeishuEmojiTypes from it and GitHub reports the branch as dirty.
