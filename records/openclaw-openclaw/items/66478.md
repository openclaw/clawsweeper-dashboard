## Summary

Keep this PR open. Current main still has the Feishu reaction bug, and this PR is aimed at the right plugin boundary, but the current patch still sends several undocumented Feishu emoji_type values and pulls the reaction runtime onto the channel module import path.

## What This Changes

The PR adds Feishu reaction emoji normalization, applies it to the channel react action plus add/list reaction helpers, adds Feishu reaction tests, and records the user-facing fix in CHANGELOG.md.

## Best Possible Solution

Land a Feishu-plugin-only fix that normalizes user-facing Unicode reactions through Feishu's documented emoji_type vocabulary, preserves valid mixed-case Feishu values, keeps SDK/client imports behind the existing lazy runtime boundary, and closes the linked Feishu bug after the repaired branch is rebased and validated.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, `message action=react emoji="👍"` reaches `reaction_type.emoji_type` unchanged, and Feishu documents 231001 for unsupported emoji_type values.

## Solution Assessment

No, not as currently patched. Normalizing inside the Feishu plugin is the right direction, but the implementation should use the official Feishu vocabulary and avoid importing the runtime reaction module from `channel.ts`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use documented Feishu emoji_type values:** `extensions/feishu/src/reactions.ts:165-190`
  - body: The new normalizer maps several Unicode inputs to values Feishu's current emoji reference does not list: thumbs-down, fire, check marks, and cross become `THUMBSDOWN`, `FIRE`, `CHECK`, and `CROSS`, while the documented values are `ThumbsDown`, `Fire`, `CheckMark`, and `CrossMark`. Those inputs can still produce 231001, and valid mixed-case values like `Fire` are now uppercased through the known-type set. Please build the table from the documented vocabulary or limit it to verified values.
  - confidence: 0.94
- **[P2] Keep normalization off the channel import path:** `extensions/feishu/src/channel.ts:66`
  - body: Importing `normalizeFeishuEmoji` from `./reactions.js` makes `channel.ts` load the reaction runtime module during channel module evaluation. That module imports `client.ts`, which imports the Lark SDK and rewrites its HTTP interceptor at module load, bypassing the existing `loadFeishuChannelRuntime` lazy boundary. Move the normalizer to a runtime-free helper or rely on the normalization already added inside the lazy add/list helpers.
  - confidence: 0.9
