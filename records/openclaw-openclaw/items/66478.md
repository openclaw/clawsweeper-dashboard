## Summary

The PR targets a real Feishu reaction bug, but the current patch is not merge-ready because it still emits undocumented emoji_type values, crosses the existing lazy runtime boundary, lacks the required changelog entry, and has only mock/unit proof rather than a real Feishu/Lark run.

## What This Changes

The PR adds Feishu reaction emoji normalization, applies it to add/list/react paths, and adds mocked reaction helper tests.

## Best Possible Solution

Repair this PR with exact documented Feishu/Lark emoji_type values in a runtime-safe Feishu helper path, add the changelog and regression coverage, then require real Feishu/Lark proof before merge.

## Reproduction Assessment

Yes, at source level: current main passes a Unicode emoji unchanged into Feishu reaction_type.emoji_type, and the official API docs say invalid emoji_type values return 231001. I did not establish a live failing Feishu account run, so this is source_reproducible rather than reproduced.

## Solution Assessment

No. Normalizing inside the Feishu plugin is the right direction, but this patch needs documented emoji_type values, a lazy-boundary-safe placement, a changelog entry, and real behavior proof before it is the best maintainable fix.
