## Summary

Keep open: this is the canonical focused Feishu reaction bug-fix PR, but the current head still has merge-blocking correctness and runtime-boundary issues that should be repaired before landing.

## What This Changes

The PR normalizes Feishu reaction emoji inputs before calling Feishu reaction APIs and adds regression tests for Unicode and type-string cases.

## Best Possible Solution

Use a Feishu-local lightweight emoji normalizer with exact documented `emoji_type` values, preserve already-valid Feishu strings, and keep the helper import free of Feishu client runtime side effects.

## Reproduction Assessment

Yes, at source level: current main forwards `message action=react emoji="👍"` through to `reaction_type.emoji_type` unchanged, and the issue reports Feishu returning code 231001 for that path. I did not run a live Feishu roundtrip in this read-only review.

## Solution Assessment

No. Normalizing at the Feishu reaction API boundary is the right direction, but this implementation should use Feishu's exact documented values and keep the pure normalizer outside the runtime-heavy `reactions.ts` import path.

## Review Findings

- **[P1] Preserve documented mixed-case emoji types:** `extensions/feishu/src/reactions.ts:204-213`
  - body: Current main passes caller-supplied Feishu `emojiType` values through unchanged, so documented values like `Fire`, `ThumbsDown`, `CheckMark`, and `CrossMark` can work today. The channel already routes runtime helpers through `loadFeishuChannelRuntime()`, so move the pure emoji map into a lightweight helper imported by both paths instead of pulling the reaction client into startup.
  - confidence: 0.91
