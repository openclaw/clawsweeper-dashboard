## Summary

Keep this PR open but do not merge the current head: it fixes the reported raw-Unicode Feishu reaction path, but the normalizer can still break documented Feishu emoji values and it imports reaction runtime code across the existing lazy boundary.

## What This Changes

The PR adds Feishu reaction emoji normalization, wires it into generic react/add/list paths, and adds regression tests for Unicode, pass-through, trimming, and unknown inputs.

## Best Possible Solution

Use a Feishu-local lightweight emoji normalizer with exact documented `emoji_type` values, preserve already-valid Feishu strings, and keep the helper import free of Feishu client runtime side effects.

## Reproduction Assessment

Yes, at source level: current main forwards `message action=react emoji="👍"` through to `reaction_type.emoji_type` unchanged, and Feishu documents invalid reaction type code 231001 for bad reaction types. I did not run a live Feishu roundtrip in this read-only review.

## Solution Assessment

No, not on the current head. The right fix boundary is the Feishu reaction API edge, but the implementation should use exact documented values and a lightweight helper that does not import runtime/client modules eagerly.
