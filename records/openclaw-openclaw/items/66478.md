## Summary

Keep this PR open: the Feishu reaction normalization is the right direction, but the latest branch still emits several emoji_type values that do not match Feishu's documented vocabulary and it pulls reaction runtime code into the eager channel module. The supplied proof is useful terminal smoke output for the normalizer, but it does not show the full OpenClaw Feishu reaction path or a live Feishu/Lark API success after the fix.

## What This Changes

The PR adds Feishu reaction emoji normalization in the channel and reaction helpers, plus tests for Unicode-to-emoji_type conversion and request-body behavior.

## Best Possible Solution

Keep this contributor branch as the canonical path, but repair it with exact documented Feishu emoji_type values, runtime-free normalization shared by channel and reaction helpers, focused tests, and redacted live/full-path Feishu proof before merge.

## Reproduction Assessment

Yes at source level: current main passes the raw action emoji into reaction_type.emoji_type, and Feishu documents 231001 for invalid reaction types. I did not run a live Feishu workspace reproduction, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. Normalizing inside the Feishu plugin is the right owner boundary, but this patch needs exact documented values and a runtime-free helper before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Use Feishu's documented emoji_type values:** `extensions/feishu/src/reactions.ts:134-190`
  - body: The normalizer still emits values Feishu does not document. The official reference lists values such as `ThumbsDown`, `Fire`, `CheckMark`, `CrossMark`, `LAUGH`, and `BLUSH`, while this table returns `THUMBSDOWN`, `FIRE`, `CHECK`, `CROSS`, `LAUGHING`, and `GRINNING`; the reporter's `👀` case also still falls through unchanged. Those inputs can still return 231001 instead of fixing the reaction failure.
  - confidence: 0.95
- **[P2] Keep normalization out of the eager channel module:** `extensions/feishu/src/channel.ts:72`
  - body: Importing `normalizeFeishuEmoji` from `./reactions.js` at the top of `channel.ts` makes the Feishu plugin eagerly evaluate `reactions.ts` and then `client.ts`, whose module body rewrites the Lark SDK HTTP interceptor. That bypasses the existing `loadFeishuChannelRuntime()` lazy boundary for startup/static plugin paths, so put the pure mapping in a runtime-free helper used by both files.
  - confidence: 0.9
