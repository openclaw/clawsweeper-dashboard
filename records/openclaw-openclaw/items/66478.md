## Summary

Keep open: the PR targets the right Feishu reaction bug, but the current patch is not merge-ready because it can rewrite documented Feishu emoji_type values into invalid strings and it pulls the reaction client across the channel lazy-runtime boundary.

## What This Changes

The PR adds Feishu emoji normalization for reaction add/list paths, wires it into the generic Feishu react action, and adds focused reaction helper tests.

## Best Possible Solution

Keep the Feishu-owned normalization boundary, but move the pure emoji mapping into a lightweight helper, preserve exact documented emoji_type values, and test both Unicode aliases and already-valid canonical strings.

## Reproduction Assessment

Yes, at source level: the linked issue gives a concrete message(action="react", emoji="👍") path, current main passes that raw value into reaction_type.emoji_type, and Feishu documents named emoji_type values plus 231001 for invalid reaction types. I did not run a live Feishu tenant roundtrip in this read-only review.

## Solution Assessment

No. Normalizing at the Feishu reaction API boundary is the right direction, but this patch should use the exact Feishu emoji_type table and keep the pure normalizer outside runtime-heavy imports before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve Feishu's documented emoji_type strings:** `extensions/feishu/src/reactions.ts:131-153`
  - body: The official Feishu table is case-sensitive and includes values such as `ThumbsDown`, `CheckMark`, `CrossMark`, and `Fire`; current main passes caller-supplied values through unchanged. This table and the later uppercase match turn valid inputs or Unicode aliases into `THUMBSDOWN`, `CHECK`, `CROSS`, and `FIRE`, which are not the documented values, so existing valid calls and those aliases can still fail with 231001.
  - confidence: 0.96
- **[P2] Keep normalization behind the lazy runtime boundary:** `extensions/feishu/src/channel.ts:73`
  - body: Importing `normalizeFeishuEmoji` from `./reactions.js` at channel module load also evaluates `client.ts`, which mutates the Lark SDK request interceptor during module evaluation. The channel already routes runtime helpers through `loadFeishuChannelRuntime()`, so move the pure emoji map into a lightweight helper imported by both paths instead of pulling the reaction client into startup.
  - confidence: 0.91
