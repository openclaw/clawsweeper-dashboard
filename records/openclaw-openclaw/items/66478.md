## Summary

Keep open: the Feishu reaction bug is real and this is the canonical fix path, but the current patch is not merge-ready because it maps several documented Feishu emoji values to invalid strings and bypasses the existing lazy runtime boundary.

## What This Changes

This PR adds Feishu reaction emoji normalization and tests, wiring unicode/type-string inputs through `channel.ts` and `reactions.ts` before add/list API calls.

## Best Possible Solution

Keep this canonical Feishu reaction fix path, but land it only after the normalizer uses Feishu's official `emoji_type` values exactly and remains a lightweight helper outside runtime-heavy modules.

## Reproduction Assessment

Yes, at source level: current main passes `message action=react emoji="👍"` through to Feishu as the raw `emoji_type`, and the linked issue reports Feishu returning code 231001 for that path. I did not run a live Feishu HTTP roundtrip in this read-only review.

## Solution Assessment

No, not yet: normalizing at the Feishu reaction API boundary is the right direction, but the table must match Feishu's official values exactly and the helper should not cross the channel lazy-runtime boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Use Feishu's exact emoji_type values:** `extensions/feishu/src/reactions.ts:134-150`
  - body: Feishu's official table does not list several values this PR emits: for example it uses `ThumbsDown`, `CheckMark`, `CrossMark`, and `Fire`, while the PR emits `THUMBSDOWN`, `CHECK`, `CROSS`, and `FIRE`. That means existing valid calls like `emoji="Fire"` get rewritten to an invalid value, and the new unicode aliases for those reactions can still fail with the same API validation error.
  - confidence: 0.91
- **[P2] Keep the normalizer out of the runtime module:** `extensions/feishu/src/channel.ts:73`
  - body: This top-level import pulls in `reactions.ts`, which imports `client.ts`; `client.ts` mutates the Lark SDK HTTP interceptor at module load. That bypasses the existing `loadFeishuChannelRuntime()` boundary, so simply loading the Feishu plugin now evaluates reaction/client runtime side effects even when no reaction or API action runs.
  - confidence: 0.86
