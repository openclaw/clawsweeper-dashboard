## Summary

Keep this PR open as the canonical fix for the Feishu reaction bug, but it still needs changes before merge: the normalization direction is right, while the current table sends several non-documented `emoji_type` values and the channel import crosses the existing lazy runtime boundary.

## What This Changes

This PR adds Feishu emoji normalization for reaction add/list/remove paths and adds Feishu reaction helper tests for Unicode conversion and pass-through behavior.

## Best Possible Solution

Land a Feishu-local, runtime-light normalizer using exact documented Feishu `emoji_type` strings at the reaction API boundary, with regression coverage for the reported Unicode input and mixed-case documented values.

## Reproduction Assessment

Yes, at source level: the linked issue passes `emoji="👍"`, current main forwards that value into `reaction_type.emoji_type`, and Feishu documents 231001 for invalid reaction types.

## Solution Assessment

No. Normalizing at the Feishu reaction boundary is the right fix, but this PR should use exact documented `emoji_type` values and keep the pure normalizer out of the runtime-heavy reactions module before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Use documented Feishu emoji_type values:** `extensions/feishu/src/reactions.ts:134-150`
  - body: The new table treats several reactions as all-uppercase, but Feishu's emoji table is case-sensitive and lists values such as `ThumbsDown`, `CheckMark`, `CrossMark`, and `Fire`, while the list API example uses `LAUGH`. Since these mapped values are sent directly, inputs like `👎`, `🔥`, `✅`, or an already-valid `Fire` can still return 231001 instead of being fixed.
  - confidence: 0.97
- **[P2] Keep emoji normalization behind the lazy runtime boundary:** `extensions/feishu/src/channel.ts:72`
  - body: Importing `normalizeFeishuEmoji` from `./reactions.js` at channel module load evaluates `reactions.ts`, which imports `client.ts`; `client.ts` mutates the Lark SDK HTTP interceptor during module evaluation. That bypasses the existing `loadFeishuChannelRuntime()` split for this channel, so move the pure emoji map into a lightweight helper imported by both `channel.ts` and `reactions.ts`.
  - confidence: 0.92
