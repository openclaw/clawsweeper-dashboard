## Summary

Keep open: current main still returns a text-only no-argument `/reasoning` status reply, while Feishu only renders clickable buttons when a reply carries typed `presentation` controls. The related args-menu issue and open Telegram/native-menu PR partially overlap, but they do not own the remaining Feishu command-status presentation gap.

## What This Changes

The issue requests clickable Feishu card buttons for command options such as `/reasoning` instead of text-only option lists.

## Best Possible Solution

Emit typed portable presentation command actions for selectable no-argument command status replies, preserve the current text fallback, and let Feishu render those controls through its existing card-button adapter.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: no-argument `/reasoning` returns only a text payload with an `Options:` line, while Feishu only creates clickable buttons when a reply includes typed `presentation` blocks.

## Solution Assessment

Yes, if solved at the shared typed presentation boundary. A Feishu-only parser that infers buttons from `Options:` text would be the wrong layer and would conflict with the repository's typed command-action guidance.
