## Summary

Keep open: current main and v2026.6.6 still produce text-only no-argument `/reasoning` status replies with an `Options:` line, while Feishu only renders clickable buttons when a reply already carries typed `presentation` controls. The related native-menu issue and PR partially overlap, but they do not own the Feishu text-command presentation gap.

## What This Changes

Render selectable command options such as `/reasoning on/off/stream` as clickable Feishu card buttons instead of only an `Options:` text line.

## Best Possible Solution

Emit typed portable presentation command actions for no-argument selectable command status replies, preserve the current text fallback, and let Feishu render those controls as native card buttons.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: no-argument `/reasoning` returns only a text reply with an `Options:` line, while Feishu renders buttons only when the reply includes structured `presentation` blocks. I did not run a live Feishu chat reproduction in this read-only review.

## Solution Assessment

Yes, if solved at the shared typed presentation boundary. A Feishu-only parser that infers buttons from `Options:` text would be the wrong layer and would conflict with the repository's command-action guidance.
