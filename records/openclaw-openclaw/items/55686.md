## Summary

Keep open. Current main still builds the reported `/reasoning` status as a plain `Options:` text reply, and Feishu's normal inbound reply dispatcher does not render payload presentation buttons for that path. The related Feishu interactive-card work supplies reusable button/callback primitives, while the open callback-field work is adjacent but does not close this command-option UX gap.

## What This Changes

The issue requests rendering Feishu command option choices such as `/reasoning on/off/stream` as clickable interactive-card buttons instead of requiring typed replies.

## Best Possible Solution

Add a focused command-option presentation path for known directive/menu replies and have Feishu's normal inbound reply pipeline render those buttons as native cards through the existing shared `presentation` and `ocf1` quick-action callback path, while retaining readable text fallback for channels or contexts without buttons.

## Reproduction Assessment

Yes. Current source and tests reproduce the issue: `/reasoning` without an argument returns a plain `Options: on, off, stream.` text payload, and the Feishu direct reply dispatcher does not render presentation buttons for that reply path.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing shared presentation and Feishu quick-action primitives, not add Feishu-specific markdown parsing or broaden this into the separate callback-field preservation work.
