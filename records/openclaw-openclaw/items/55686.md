## Summary

Keep open: current main still has a Feishu-specific gap for clickable command options. `/reasoning` now has shared `argsMenu: "auto"`, but Feishu does not consume `resolveCommandArgMenu`; the no-arg command reply still produces a plain `Options: on, off, stream.` line, while Feishu only renders buttons when a payload already contains presentation blocks.

## What This Changes

The issue requests clickable Feishu card buttons for command options such as `/reasoning` instead of plain text option lists.

## Best Possible Solution

Add a Feishu-owned option-presentation path that turns trusted simple command option replies into `MessagePresentation` buttons with safe command values, while preserving the existing text fallback and keeping the separate inbound callback-field issue separate.

## Reproduction Assessment

Yes, source-level: current main still returns a plain `/reasoning` `Options:` line, and Feishu has no `resolveCommandArgMenu` consumer or outbound `normalizePayload` transform to turn that line into card buttons.

## Solution Assessment

Yes directionally: the maintainable path is a Feishu adapter presentation/normalization path, or a small shared command-option presentation producer, rather than Feishu-specific UI logic inside the directive handler.
