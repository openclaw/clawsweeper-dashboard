## Summary

Keep open: current main still returns command-option status replies such as `/reasoning` as plain `Options:` text, and Feishu's normal reply pipeline does not currently turn those options into native presentation/card buttons. Existing Feishu card support is useful infrastructure, while the adjacent callback-field work does not close this command-option UX gap.

## What This Changes

Request to render Feishu command option replies such as `/reasoning` as clickable interactive card buttons instead of plain text options.

## Best Possible Solution

Add a focused command-option presentation path for known directive/menu replies and have Feishu render those buttons through the existing `presentation` and `ocf1` quick-action plumbing, with readable text fallback for channels or contexts without buttons.

## Reproduction Assessment

Yes, source-level: `/reasoning` without an argument returns a plain `Options: on, off, stream.` text payload, and Feishu does not currently synthesize presentation buttons for that normal reply path. I did not run a live Feishu replay in this read-only review.

## Solution Assessment

Yes at the direction level: the maintainable fix is to reuse the shared message presentation and Feishu quick-action primitives rather than add Feishu-specific markdown parsing. The repair should be limited to known command-option/menu replies and preserve text fallback.
