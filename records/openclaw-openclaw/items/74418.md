The PR normalizes per-model `params.thinking` values `false`, `"disabled"`, and `"none"` to `off` in the thinking resolver and adds focused model-selection tests.

## Best Possible Solution

Add one single-line `CHANGELOG.md` entry under `Unreleased` / `Fixes`, then merge the resolver normalization with focused tests while leaving the broader `extra_body` collision to #75040.

## Reproduction Assessment

Yes. On current main, configuring `agents.defaults.models["deepseek/deepseek-v4-pro"].params.thinking` as `false`, `"disabled"`, or `"none"` leaves the resolver on the provider fallback path, which resolves DeepSeek V4 thinking to `high`.

## Solution Assessment

Yes, after the changelog addition. Normalizing disabled aliases before provider fallback is the narrow maintainable fix because the DeepSeek wrapper already handles the disabled replay path once it receives `off` or `none`.
