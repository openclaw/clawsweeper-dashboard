## Summary

Keep open. Current main still has the resolver gap this PR targets, and the patch is a narrow, plausible fix for the linked DeepSeek disabled-thinking regression; the remaining blocker I found is the required changelog entry plus normal targeted validation before merge.

## What This Changes

The PR normalizes per-model `params.thinking` values `false`, `"disabled"`, and `"none"` to `"off"` in `resolveThinkingDefault` and adds three DeepSeek v4-pro resolver tests.

## Best Possible Solution

Merge a narrow resolver-level normalization with focused tests and a user-facing changelog entry, while keeping the broader framework-level `extra_body` collision discussion separate.

## Reproduction Assessment

Yes. On current main, a config with `agents.defaults.models["deepseek/deepseek-v4-pro"].params.thinking` set to `false`, `"disabled"`, or `"none"` exercises the resolver path and falls through because only canonical string levels are accepted.

## Solution Assessment

Yes, with the changelog addition. Normalizing the disabled aliases before provider fallback is the narrowest maintainable fix because the DeepSeek stream wrapper already implements the correct disabled replay behavior once it receives `off` or `none`.
