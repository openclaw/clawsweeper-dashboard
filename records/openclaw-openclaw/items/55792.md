## Label Justifications

- `P1`: Silent missed inbound messages across routine gateway restarts can break active channel workflows for real users.
- `impact:message-loss`: The reported failure is inbound channel messages being silently missed and never routed to the agent after restart.

## Summary

Keep open. Current main still lacks a shared, configurable missed-inbound catch-up contract after gateway restart; iMessage has an opt-in provider-local implementation, Telegram polling has offset/spool recovery for a narrower mode, and Discord/Slack startup still do not replay channel history after process downtime.

## What This Changes

The issue asks for configurable post-restart inbound message catch-up so channel messages sent during gateway downtime are not silently lost.

## Best Possible Solution

Design an additive opt-in channel catch-up contract with provider-owned cursors/history fetches, bounded replay, durable dedupe, and docs, then land provider-specific slices without treating Discord reconnect backfill or Telegram polling recovery as the whole solution.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main has no shared catch-up hook, and Discord/Slack startup paths register live handlers without startup history replay; I did not run a live credentialed restart E2E.

## Solution Assessment

Unclear as proposed. Anchoring to the last outbound message may work for some providers, but the maintainable solution needs provider-owned cursor semantics, opt-in config, replay limits, ACK behavior, and durable dedupe rather than a core loop assuming uniform history APIs.
