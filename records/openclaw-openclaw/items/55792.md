## Summary

Keep open. Current main still has provider-specific recovery pieces, but not the requested configurable post-startup catch-up contract for Discord/Slack and other channel plugins; this remains a cross-provider reliability feature that needs maintainer API/design work.

## What This Changes

Request an opt-in post-startup channel catch-up mechanism that replays inbound messages sent while the gateway was down.

## Best Possible Solution

Add an additive, opt-in channel catch-up contract with provider-owned cursors/history fetches, durable per-message dedupe, self-message filtering, replay windows, and docs, then implement bounded Discord and Slack support first while keeping Telegram webhook durability and queued/in-flight restart persistence as related work.

## Reproduction Assessment

Yes, by source inspection: Discord and Slack startup paths register live event handlers and do not run a startup history replay, so messages sent while those providers are offline have no generic catch-up path. I did not run a live credentialed Discord or Slack restart E2E in this read-only review.

## Solution Assessment

Unclear as proposed. Anchoring to last outbound is a useful candidate, but the maintainable solution needs an explicit provider opt-in contract, per-provider cursor semantics, durable dedupe, and replay limits rather than a core loop that assumes uniform history APIs.
