## Summary

Keep open. Current main now has opt-in iMessage catch-up, but Discord/Slack and the shared channel contract still lack the requested configurable post-startup missed-inbound replay, so the cross-provider reliability feature remains unresolved and needs maintainer design.

## What This Changes

The issue requests a configurable post-restart channel catch-up mechanism that replays inbound messages sent while the gateway was offline.

## Best Possible Solution

Add an additive, opt-in channel catch-up contract with provider-owned cursors/history fetches, durable per-message dedupe, replay bounds, and docs, then implement Discord and Slack support while keeping Telegram webhook durability and queued/in-flight restart persistence tracked separately.

## Reproduction Assessment

Yes, source-reproducible: Discord and Slack startup paths register live event handlers and current channel SDK/gateway contracts have no missed-inbound catch-up hook. I did not run a live credentialed restart E2E in this read-only review.

## Solution Assessment

Unclear as proposed. Anchoring to the last outbound message is a useful candidate, but the maintainable solution needs explicit provider-owned cursor semantics, opt-in config, replay limits, ACK behavior, and durable dedupe rather than a core loop assuming uniform history APIs.
