## Label Justifications

- `P1`: Silent missed inbound messages across routine gateway restarts can break active channel workflows for real users.
- `impact:message-loss`: The issue is about inbound channel messages being silently missed and never routed to the agent after restart.

## Summary

Keep open. Current main still lacks a shared, configurable missed-inbound catch-up contract after gateway restart; existing iMessage, Telegram, and Discord work only covers narrower provider-local slices.

## What This Changes

Request a configurable post-startup catch-up mechanism that replays inbound channel messages missed during gateway downtime or restart.

## Best Possible Solution

Design an additive opt-in channel catch-up contract with provider-owned cursors/history fetches, bounded replay, durable dedupe, and docs, then land provider-specific slices separately.

## Reproduction Assessment

Yes, at source level. Current main has no shared catch-up hook, while Discord and Slack startup still register live handlers without startup history replay; I did not run a live credentialed restart E2E.

## Solution Assessment

Unclear as proposed. Anchoring on the last outbound message is plausible for some providers, but the maintainable solution needs provider-owned cursor semantics, opt-in config, replay limits, ACK behavior, and durable dedupe rather than a uniform core loop.
