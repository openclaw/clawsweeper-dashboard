## Label Justifications

- `P1`: The issue describes real inbound user messages being silently missed across gateway restarts, which is an urgent channel workflow reliability gap.
- `impact:message-loss`: The affected behavior is lost or unreplayed inbound channel messages during restart or downtime windows.

## Summary

Keep open: current main has provider-specific recovery for iMessage, but the broader requested restart catch-up behavior remains missing for Discord, Slack, Telegram webhook delivery, and other inbound channels, and the right shape needs maintainer design across provider APIs, durable receive state, and config scope.

## What This Changes

The issue requests configurable cross-provider catch-up or durable journaling for inbound messages missed while the gateway is down or restarting.

## Best Possible Solution

Keep this as the canonical design tracker and land provider-owned durable receive or history-replay implementations behind narrow shared channel/plugin contracts once maintainers choose the default and config shape.

## Reproduction Assessment

Yes, source-reproducible: current code shows live-only Discord and Slack paths plus Telegram webhook ACK-before-processing behavior, while only iMessage has provider-specific catch-up.

## Solution Assessment

Unclear as an implementation, but the best direction is provider-owned replay/durable receive behavior with shared contracts rather than a single generic core queue or ad hoc per-channel config.
