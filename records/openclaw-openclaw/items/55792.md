## Label Justifications

- `P1`: Silent missed inbound messages during routine gateway restarts can break active channel workflows for real users.
- `impact:message-loss`: The reported failure is inbound channel messages being silently missed and never routed to the agent after restart.

## Summary

Keep open. Current main and the latest release still do not provide a shared, configurable missed-inbound catch-up contract after gateway restart; the existing work is provider-local or narrower, with iMessage opt-in catch-up, WhatsApp durable pending replay, Telegram polling spool recovery, and no Discord/Slack startup history replay on main.

## What This Changes

The issue asks OpenClaw to catch up on inbound channel messages missed during gateway downtime, ideally with configurable replay bounds and provider support across Discord, Telegram, Slack, and similar channels.

## Best Possible Solution

Design an additive, opt-in channel catch-up contract with provider-owned cursors/history fetches, bounded replay, durable dedupe, and docs, then land provider-specific slices without treating existing iMessage, WhatsApp, Telegram polling, or Discord reconnect work as the whole solution.

## Reproduction Assessment

Yes, at source level. Current main has no shared catch-up hook, Discord and Slack startup paths are live-event driven without history replay, and Telegram webhook mode acknowledges before durable processing; I did not run a live credentialed restart E2E.

## Solution Assessment

Unclear as proposed. Anchoring to the last outbound message may work for some providers, but the maintainable solution needs provider-owned cursor semantics, opt-in config, replay limits, ACK behavior, and durable dedupe rather than a core loop assuming uniform history APIs.
