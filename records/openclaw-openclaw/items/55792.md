## Label Justifications

- `P1`: The issue describes silently lost inbound channel messages during gateway restart, which can break active real-user conversations.
- `impact:message-loss`: The requested work is specifically about recovering or replaying inbound messages that otherwise disappear during restart or reconnect windows.

## Summary

Keep open. Current main still lacks the requested shared, configurable inbound catch-up behavior after gateway restart: iMessage now has provider-local downtime recovery and Telegram polling has durable offset handling, but Discord and Slack remain live-event/reconnect driven without startup history replay, and the related Discord backfill PR was closed unmerged.

## What This Changes

The issue requests configurable replay or catch-up of inbound channel messages missed while the gateway is restarting or offline.

## Best Possible Solution

Define provider-owned catch-up hooks and durable cursors/dedupe with explicit ACK, scope, rate-limit, and config semantics, then land bounded per-channel slices behind maintainer-approved defaults.

## Reproduction Assessment

Yes at source level. Discord and Slack current startup paths register live listeners or reconnect loops without startup history replay, Telegram webhook ACKs before async processing, and no live credentialed restart E2E was run.

## Solution Assessment

Unclear as proposed. Last-outbound anchoring is useful for some channels, but the Slack discussion and current provider differences show the best fix needs provider-owned replay contracts rather than a uniform core pending-message file.
