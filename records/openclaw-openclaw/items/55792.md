## Summary

Keep open: current main has automatic iMessage downtime recovery and durable primitives, but the broader requested restart catch-up remains missing for Discord, Slack, Telegram webhook delivery, and other channels; related open issues and PRs cover partial slices rather than a merged canonical fix.

## What This Changes

Request cross-channel post-restart inbound catch-up so messages sent while the gateway is down are replayed instead of silently missed.

## Best Possible Solution

Keep this as the broad design tracker and land provider-owned durable receive or history-replay paths through a narrow shared channel/plugin contract once maintainers choose the default, config, and replay semantics.

## Reproduction Assessment

Yes. Source inspection shows current main has provider-specific iMessage recovery but live-only Discord and Slack startup paths plus Telegram webhook ACK-before-processing behavior, so the cross-channel missed-message gap is source-reproducible without a live restart run.

## Solution Assessment

Unclear as a specific implementation sketch. The best direction is provider-owned replay or durable receive behavior over a shared channel/plugin contract, not a single generic history fetch loop or ad hoc sidecar queue.
