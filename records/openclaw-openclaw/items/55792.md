- `impact:message-loss`: The report is specifically about inbound channel messages being silently dropped or never replayed after gateway downtime.

## Summary

Keep open: current main has iMessage-specific restart recovery and generic drain gates, but it still does not provide the requested cross-channel missed-inbound catch-up for Slack/Discord-style downtime windows. The issue is also the canonical tracker for the closed Slack-specific duplicate, and the remaining behavior needs maintainer product/owner direction rather than cleanup closure.

## What This Changes

Request cross-channel post-startup catch-up for inbound messages sent while the gateway was restarting or down, with configurable replay bounds and idempotency.

## Best Possible Solution

Keep this as the canonical design tracker and land provider-owned recovery paths, using shared SDK/core durability seams only where needed, after maintainers settle replay and idempotency semantics.

## Reproduction Assessment

Yes for a source-level reproduction path: current main has iMessage recovery but no Slack/Discord catch-up surface, and restart drain still rejects new command-lane work. I did not perform a live Discord or Slack restart test in this read-only review.

## Solution Assessment

Unclear as a single implementation. The last-outbound anchor is plausible, but the maintainable solution should be channel-owned per provider with shared durable receive/replay seams only where the plugin API needs them.
