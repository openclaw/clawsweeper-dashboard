## Label Justifications

- `P1`: Silent missed inbound messages across routine gateway restarts can break active channel workflows for real users.
- `impact:message-loss`: The reported failure is inbound channel messages being silently missed and never routed to the agent after restart.

## Summary

Keep open. Current main still lacks the requested cross-provider, configurable post-startup missed-inbound replay: iMessage has a provider-local opt-in catch-up implementation, and https://github.com/openclaw/openclaw/pull/81226 covers a partial Discord path, but the shared channel contract plus Discord/Slack mainline startup paths still have no equivalent catch-up hook.

## What This Changes

Request to add configurable post-startup recovery for inbound channel messages sent while the gateway is restarting or offline.

## Best Possible Solution

Design an additive, opt-in channel catch-up contract with provider-owned cursors/history fetches, bounded replay, durable dedupe, and docs, then land provider slices without folding Telegram webhook ACK persistence or generic followup queue recovery into this umbrella.

## Reproduction Assessment

Yes, source-reproducible: current main's shared channel adapter has only start/stop and Discord/Slack startup paths register live handlers without startup history replay. I did not run a live credentialed restart E2E.

## Solution Assessment

Unclear as proposed. Anchoring to the last outbound message is a plausible provider strategy, but the maintainable solution needs provider-owned cursor semantics, opt-in config, replay limits, ACK behavior, and durable dedupe rather than a core loop assuming uniform history APIs.
