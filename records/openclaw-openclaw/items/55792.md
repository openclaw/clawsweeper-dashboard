## Summary

Keep open. Current main has opt-in iMessage catch-up, but the requested cross-provider, configurable post-startup missed-inbound replay is still absent for Discord/Slack and not represented in the shared channel contract.

## What This Changes

Request configurable post-startup catch-up for inbound channel messages missed while the gateway was down or restarting.

## Best Possible Solution

Add an additive, opt-in channel catch-up contract with provider-owned cursors/history fetches, durable per-message dedupe, bounded replay, and docs, then implement provider slices without merging Telegram webhook durability or in-flight queue persistence into this umbrella.

## Reproduction Assessment

Yes, source-reproducible: Discord and Slack startup paths register live handlers/start socket mode without a missed-inbound catch-up hook, and the shared channel gateway contract has no replay surface. I did not run a live credentialed restart E2E.

## Solution Assessment

Unclear as proposed. Anchoring to the last outbound message is a useful candidate, but the maintainable solution needs provider-owned cursor semantics, opt-in config, replay limits, ACK behavior, and durable dedupe rather than a core loop assuming uniform history APIs.
