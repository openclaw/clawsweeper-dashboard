## Summary

Keep open. Current main still does not implement the requested cross-provider post-startup inbound catch-up: BlueBubbles has a provider-specific startup replay path and Telegram polling has offset persistence, but Discord and Slack are still live-event driven and the channel plugin contract has no generic history replay/catch-up adapter for core to call.

## What This Changes

Request a configurable post-startup catch-up pass that replays inbound messages sent while the gateway was offline, ideally anchored around the last outbound bot message or another safe cursor.

## Best Possible Solution

Add an additive, opt-in channel catch-up contract that lets capable providers enumerate recent inbound messages for known conversations after startup, then implement bounded provider-specific replay for Discord and Slack using durable per-message dedupe, bot/self-message filtering, replay windows, and clear docs for supported channels.

## Reproduction Assessment

Yes. Send a Discord or Slack DM while the gateway is stopped or restarting, then let the channel reconnect; current source shows those providers only process delivered live events after startup and do not scan recent channel history.

## Solution Assessment

Unclear. The last-outbound anchor is a reasonable design input, but the maintainable solution should be an explicit provider opt-in contract with per-provider cursor/history semantics, bounded replay, and durable dedupe rather than a core loop assuming every channel can fetch history the same way.
