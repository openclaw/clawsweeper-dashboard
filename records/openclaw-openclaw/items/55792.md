## Summary

Keep open: current main has solid iMessage downtime recovery, but the requested cross-provider post-restart inbound catch-up remains missing for Discord, Slack, and the broader channel framework, with related queued/in-flight restart work still open and only partially overlapping.

## What This Changes

The issue requests configurable post-restart inbound message catch-up across channel providers so messages sent while the gateway is down are not silently missed.

## Best Possible Solution

Keep this as the canonical tracker for post-start channel-history catch-up, design a provider-owned replay seam with shared idempotency rules, and leave queued/in-flight restart persistence to the narrower open restart-recovery cluster.

## Reproduction Assessment

Yes at source level: current main has iMessage-specific recovery, but Slack and Discord startup/reconnect paths do not replay missed channel history, and gateway draining still rejects new enqueues. I did not run a live provider restart in this read-only review.

## Solution Assessment

No, not as one generic history-fetch sketch copied across providers; the best fix is a provider-owned/plugin-aware replay design that reuses existing admission and dedupe paths and splits queued in-process recovery from offline channel-history catch-up.
