## Summary

Keep open: current main has provider-specific recovery for Telegram and iMessage, but the requested restart catch-up remains missing for live-event providers such as Discord and Slack and needs maintainer design across provider APIs, durable state, dedupe, and defaults.

## What This Changes

The issue requests durable catch-up of missed inbound channel messages after gateway restart across providers such as Discord, Slack, BlueBubbles/iMessage, and Telegram.

## Best Possible Solution

Define a reusable provider/plugin recovery contract, then implement provider-owned catch-up with durable cursors, dedupe, and upgrade-safe defaults per transport.

## Reproduction Assessment

Yes from source inspection: Discord and Slack rely on live event handlers without a startup catch-up path, while existing iMessage and Telegram recovery paths show the requested behavior is provider-specific rather than globally solved.

## Solution Assessment

Unclear as a code change until maintainers choose the provider contract; the best direction is provider-owned durable cursors and replay semantics behind a shared API.
