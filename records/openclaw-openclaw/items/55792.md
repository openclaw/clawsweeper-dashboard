- `P1`: Silent missed inbound messages across routine gateway restarts can break active channel workflows for real users.
- `impact:message-loss`: The reported failure is inbound channel messages being silently missed and never routed to the agent after restart.

## Summary

Keep open. Current main still does not implement the requested cross-provider, configurable missed-inbound replay: iMessage has an opt-in provider-local catchup path, and https://github.com/openclaw/openclaw/pull/81226 covers a partial Discord reconnect path, but the shared channel contract plus Discord/Slack startup paths still have no general post-startup history replay hook.

## What This Changes

The issue requests configurable post-startup replay of inbound messages that arrived while channel providers were offline during a gateway restart.

## Best Possible Solution

Design an additive, opt-in channel catchup contract with provider-owned cursors/history fetches, bounded replay, durable dedupe, and docs, then land provider slices without folding Telegram webhook ACK persistence or generic followup queue recovery into this single umbrella.

## Reproduction Assessment

Yes, source-reproducible: current main's shared channel adapter has only start/stop and Discord/Slack startup paths register live handlers without startup history replay. I did not run a live credentialed restart E2E.

## Solution Assessment

Unclear as proposed. Anchoring to the last outbound message is plausible for some providers, but the maintainable solution needs provider-owned cursor semantics, opt-in config, replay limits, ACK behavior, and durable dedupe rather than a core loop assuming uniform history APIs.
