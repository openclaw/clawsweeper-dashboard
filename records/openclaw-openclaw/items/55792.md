## Summary

Keep open. Current main has provider-specific restart recovery for BlueBubbles and Matrix, but it does not provide the requested configurable cross-provider startup catch-up contract; Discord and Slack still rely on live event delivery without a startup history replay.

## What This Changes

The issue requests configurable startup catch-up for inbound channel messages missed while the gateway is down or restarting.

## Best Possible Solution

Add an additive provider opt-in catch-up capability with provider-owned history cursors, bounded replay windows, durable per-message dedupe, self-message filtering, and docs, then implement Discord and Slack first while keeping Telegram webhook durability and queued/in-flight replay tracked with the related issues.

## Reproduction Assessment

Yes, by source inspection: Discord and Slack startup paths register live event handlers and do not run a startup history replay, while Telegram webhook mode acknowledges before durable processing. I did not run a live credentialed restart E2E in this read-only review.

## Solution Assessment

Unclear as proposed. Anchoring to the last outbound message is a useful candidate, but the maintainable solution should be an explicit provider opt-in contract with cursor semantics, replay limits, ordering, and durable dedupe rather than a core loop that assumes uniform history APIs.
