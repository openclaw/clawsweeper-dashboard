## Summary

Keep open. Current main still lacks the requested cross-provider, configurable post-startup missed-inbound replay; iMessage now has a provider-local opt-in catch-up implementation, but Discord/Slack and the shared channel/gateway contract do not have equivalent replay hooks.

## What This Changes

Request an opt-in, bounded catch-up mechanism that replays inbound messages missed while the gateway was restarting across Discord, Telegram, Slack, and other channel providers.

## Best Possible Solution

Add an additive, opt-in channel catch-up contract with provider-owned cursors/history fetches, durable per-message dedupe, bounded replay, and docs, then implement provider slices without folding Telegram webhook durability or in-flight queue persistence into this umbrella.

## Reproduction Assessment

Yes, source-reproducible: Discord and Slack startup paths still connect live handlers without a missed-inbound replay hook, and the shared channel gateway contract has no catch-up surface. I did not run a live credentialed restart E2E.

## Solution Assessment

Unclear as proposed. Anchoring to the last outbound message is a useful candidate, but the maintainable solution needs provider-owned cursor semantics, opt-in config, replay limits, ACK behavior, and durable dedupe rather than a core loop assuming uniform history APIs.
