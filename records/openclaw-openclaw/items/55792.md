## Label Justifications

- `P1`: Silent missed inbound messages during gateway restarts can break live agent/channel workflows for real users.
- `impact:message-loss`: The central impact is user messages being dropped or never delivered to the agent after gateway downtime.

## Summary

Keep open. Current main solves this only for iMessage; Discord and Slack still lack equivalent post-restart inbound catch-up, Telegram webhook still acknowledges before durable processing, and related provider-specific work remains open or unmerged.

## What This Changes

Request configurable post-startup recovery for inbound channel messages missed while the gateway is down or restarting.

## Best Possible Solution

Keep this as the cross-provider design tracker and land provider-owned catch-up or durable receive-journal implementations through approved SDK/storage seams with explicit replay windows, dedupe, ACK, policy, and config/default semantics.

## Reproduction Assessment

Yes, at source level. Current main has iMessage recovery, but Discord and Slack do not show equivalent startup replay, and Telegram webhook acknowledges before durable handling; no live restart E2E was run.

## Solution Assessment

Unclear as proposed. Last-outbound anchoring is useful input, especially for Slack, but the maintainable fix needs provider-owned replay or journaling with durable dedupe, policy filtering, rate limits, storage, and approved defaults.
