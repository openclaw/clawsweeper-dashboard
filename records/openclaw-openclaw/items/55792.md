## Label Justifications

- `P1`: The issue describes silent loss of inbound channel messages during gateway restart or reconnect windows, which can break real agent/channel workflows.
- `impact:message-loss`: The central reported impact is user messages being missed or dropped without a response after gateway downtime or restart.

## Summary

Keep open. Current main has iMessage downtime recovery, but Discord and Slack still depend on live event delivery, Telegram webhook still acknowledges before durable processing, and related channel/gateway restart work remains open or unmerged.

## What This Changes

Request cross-channel catch-up of inbound messages missed while the gateway is offline or restarting, anchored by last outbound activity with configurable replay behavior.

## Best Possible Solution

Keep this as the cross-provider design tracker and land provider-owned catch-up or receive-journal implementations through shared SDK seams with approved replay windows, dedupe, ACK, storage, and config/default semantics.

## Reproduction Assessment

Yes, at source level. Current main shows iMessage recovery, but Discord registers live MESSAGE_CREATE listeners, Slack Socket Mode registers live event handlers/reconnect backoff without startup replay, and Telegram webhook returns 200 before durable processing; no live restart E2E was run.

## Solution Assessment

Unclear as proposed. Last-outbound anchoring is useful, especially for Slack-style conversation catch-up, but the durable end state needs provider-owned replay or receive journaling with explicit dedupe, ACK, authorization, rate-limit, storage, and default semantics.
