## Label Justifications

- `P1`: Silent missed inbound messages across routine gateway restarts can break active channel workflows for real users.
- `impact:message-loss`: The reported failure is inbound channel messages being silently missed and never routed to the agent after restart.

## Summary

Keep open. Current main still lacks a shared, configurable missed-inbound catch-up contract after gateway restart; iMessage has a provider-local opt-in path, Telegram polling and the open Discord PR cover narrower slices, and Discord/Slack startup still rely on live event delivery.

## What This Changes

Request an opt-in or opt-out channel catch-up mechanism that replays inbound messages missed while the gateway was restarting, with bounded provider history and dedupe.

## Best Possible Solution

Design an additive opt-in channel catch-up contract with provider-owned cursors/history fetches, bounded replay, durable dedupe, and docs, then land provider-specific slices without merging Telegram webhook ACK persistence or generic follow-up queue recovery into the same change.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main has no shared catch-up hook, and Discord/Slack startup paths register live handlers without startup history replay; I did not run a live credentialed restart E2E.

## Solution Assessment

Unclear as proposed. Anchoring to the last outbound message is plausible for some providers, but the maintainable solution needs provider-owned cursor semantics, opt-in config, replay limits, ACK behavior, and durable dedupe rather than a core loop assuming uniform history APIs.
