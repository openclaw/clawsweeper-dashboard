## Label Justifications

- `P1`: The issue describes silent loss of real user messages during routine gateway or provider restart windows in live channel workflows.
- `impact:message-loss`: The central impact is inbound Discord, Slack, Telegram, and other channel messages never reaching the agent after downtime.

## Summary

Keep open. Current main and the latest release still solve restart-time inbound recovery only for iMessage, while Discord and Slack remain live-event driven and Telegram webhook mode still acknowledges before durable processing; linked PRs cover adjacent in-memory drain or follow-up queue persistence but not the requested cross-provider post-startup history catch-up.

## What This Changes

The issue requests configurable recovery of inbound messages sent while the gateway or channel provider is offline, using channel history/replay with durable dedupe and replay bounds.

## Best Possible Solution

Keep this as the canonical maintainer design tracker for cross-provider missed-inbound recovery, then land provider-owned replay or durable receive-journal implementations that preserve live channel policy and durable idempotency.

## Reproduction Assessment

Yes. Source inspection shows iMessage has downtime recovery, while Discord and Slack lack equivalent startup replay and Telegram webhook mode still ACKs before durable processing; no live provider restart E2E was run.

## Solution Assessment

Unclear as proposed. The last-outbound anchor is useful input, especially for Slack, but the best maintainable solution needs provider-owned replay or journaling with durable dedupe, policy filtering, rate limits, storage, and approved defaults.
