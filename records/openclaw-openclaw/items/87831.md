## What This Changes

The PR threads an optional manifest-normalization opt-out through persisted model resolvers, passes it from status summary model parsing, and adds a Vitest regression test that mocks manifest/runtime normalization.

## Best Possible Solution

Land the narrow opt-out only after maintainers accept the status display contract, while preserving existing resolver defaults for all non-status callers.

## Reproduction Assessment

Yes. Source inspection shows current main still calls parseModelRef from status summary without allowManifestNormalization:false, and the parser defaults to manifest normalization unless explicitly disabled.

## Solution Assessment

Yes, with a maintainer decision. Threading the existing parse option keeps default resolver behavior intact and localizes the performance opt-out to status; if normalized status labels are contractual, the safer alternative is a narrower cache or preserving normalization for that display path.
