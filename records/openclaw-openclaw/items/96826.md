## Summary

Keep open: current main and the latest release still use loose parsing for `cron add --output-max-bytes`, while this PR is a focused fix with regression coverage and terminal proof. I found no blocking patch defect; maintainers only need to accept the intentional fail-fast CLI compatibility change.

## What This Changes

The PR switches `cron add --output-max-bytes` to strict positive-integer validation and adds regression coverage for invalid values.

## Best Possible Solution

Land the strict validation if maintainers accept fail-fast behavior for explicitly invalid output caps, while keeping omitted and valid values unchanged.

## Reproduction Assessment

Yes, source-reproducible: current main parses invalid `cron add --output-max-bytes` through the optional positive-int helper and can omit the cap while still sending `cron.add`. I did not run the CLI because this review is read-only, but the PR body includes terminal proof after the patch.

## Solution Assessment

Yes. The patch is the narrowest maintainable fix because it reuses the existing strict positive-int helper on the one inconsistent `cron add` flag and mirrors `cron edit` behavior with focused regression coverage.
