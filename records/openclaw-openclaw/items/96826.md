## Label Justifications

- `P2`: This is a normal-priority CLI correctness bug with limited blast radius: invalid cron output cap input can silently create a job without the requested cap.
- `merge-risk: 🚨 compatibility`: The PR deliberately changes a public CLI flag from silently defaulting on invalid explicit input to failing the command.

## Summary

Keep open: this is a narrow, useful CLI bug fix that current main and the latest release do not already contain. Source shows `cron add` still silently drops an invalid explicit output cap while `cron edit` rejects the same invalid flag, and the PR aligns the add path with focused regression coverage and terminal proof.

## What This Changes

The PR makes `cron add --output-max-bytes` use strict positive-integer validation and adds cron CLI regression tests that invalid values exit before `cron.add`.

## Best Possible Solution

Land the strict validation if maintainers accept the fail-fast behavior for explicitly invalid output caps, keeping omitted and valid values unchanged.

## Reproduction Assessment

Yes from source inspection: current main parses invalid `cron add --output-max-bytes` through the optional positive-int helper and can omit the cap while still sending `cron.add`. I did not run the CLI because this review is read-only, but the PR body includes terminal proof after the patch.

## Solution Assessment

Yes. The patch is the narrowest maintainable fix: reuse the existing strict positive-int helper on the one inconsistent `cron add` flag and mirror the already-strict `cron edit` behavior with regression coverage.
