## Summary

Keep this PR open for maintainer review: the patch is a narrow, correct CLI bug fix with sufficient terminal proof, and the remaining decision is whether maintainers accept the intentional fail-fast behavior for an explicitly invalid public CLI flag.

## What This Changes

The branch makes `cron add --output-max-bytes` use strict positive-integer validation and adds cron CLI tests that invalid values exit before `cron.add`.

## Best Possible Solution

Land the strict validation if maintainers accept fail-fast handling for explicitly invalid output caps, while keeping omitted and valid values unchanged.

## Reproduction Assessment

Yes from source inspection: current main parses invalid `cron add --output-max-bytes` through the optional positive-int helper and can omit the cap before sending `cron.add`. The PR body also includes after-fix terminal output showing rejection.

## Solution Assessment

Yes. The patch is the narrowest maintainable fix for this flag: reuse the existing strict positive-int helper on the inconsistent `cron add` option and mirror `cron edit`; adjacent no-output timeout validation is separately tracked by https://github.com/openclaw/openclaw/pull/96516.
