## Summary

Keep open. The security direction is plausible, but the current PR introduces a high-confidence bypass by treating raw marker text in tool output as an existing wrapper, and the external PR still lacks contributor-owned real behavior proof.

## What This Changes

The PR adds transport-boundary wrapping of text `toolResult` content with `wrapExternalContent` and updates transport transform tests to expect wrapped outputs.

## Best Possible Solution

Revise the PR so raw tool-result text is always sanitized or only skipped after validating a full internally generated wrapper, add marker-spoof regression coverage, and include contributor-owned real transport proof.

## Reproduction Assessment

Yes, for the PR-introduced bypass. Source inspection shows the marker-name substring guard passes crafted raw tool output through unchanged, and an existing PR comment reports a local transform/payload harness confirming the bypass on the PR head.

## Solution Assessment

No. Wrapping at the shared transport boundary is a reasonable direction, but trusting attacker-controlled marker substrings is not safe; the narrower fix is to wrap and sanitize raw `toolResult` text or validate an internally generated wrapper before skipping.
