## Summary

Keep open: the shared transport-boundary hardening direction is useful, but the current branch introduces a security bypass by trusting raw marker substrings and still lacks contributor-owned real behavior proof.

## What This Changes

The PR wraps text toolResult blocks in transformTransportMessages with external-content markers and adds focused transport-message tests.

## Best Possible Solution

Wrap text tool results at the shared transport boundary by sanitizing raw tool-result text even when marker strings appear, add spoofed-marker regression coverage, and require redacted real payload proof before merge.

## Reproduction Assessment

Yes, for the PR-introduced bypass. Source inspection shows the substring guard skips wrapExternalContent, and a maintainer-side harness comment against PR head shows marker-injected tool output reaching the payload without a security notice.

## Solution Assessment

No. Wrapping at the shared transport boundary is the right direction, but the implementation must not trust attacker-controlled marker substrings; it should always sanitize raw tool output or validate a full internally generated wrapper before skipping.
