## Summary

Keep open: the PR is a plausible security-hardening direction, but the current patch has a blocking marker-bypass because it treats raw marker substrings in tool output as already wrapped, and the external contributor has not provided real behavior proof.

## What This Changes

The PR wraps text `toolResult` content in `transformTransportMessages` with external-content markers and updates focused transport-message tests.

## Best Possible Solution

Keep the shared transport-boundary hardening direction, but sanitize raw tool-result text even when marker strings appear, add spoofed-marker regression coverage, and require redacted real transport/payload proof before merge.

## Reproduction Assessment

Yes, for the PR-introduced bypass. Source inspection shows the substring guard skips `wrapExternalContent`, and a maintainer-side harness comment reproduces marker-injected tool output reaching the payload without a security notice.

## Solution Assessment

No. Wrapping at the shared transport boundary is a reasonable direction, but the implementation must not trust attacker-controlled marker substrings; it should always sanitize raw tool output or validate a full internally generated wrapper before skipping.
