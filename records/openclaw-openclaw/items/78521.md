## Summary

Keep open: the transport-boundary hardening direction is useful, but this PR currently trusts raw boundary-marker substrings in untrusted tool output, creating a concrete wrapper bypass, and it still lacks contributor-owned real behavior proof.

## What This Changes

The PR adds external-content wrapping for text tool-result blocks in `transformTransportMessages` and updates focused transport-message tests.

## Best Possible Solution

Keep the shared transport-boundary hardening, but sanitize raw tool-result text even when marker strings appear, add spoofed-marker regression coverage, and require redacted real transport/payload proof before merge.

## Reproduction Assessment

Yes, for the PR-introduced bypass. Source inspection shows the substring guard skips `wrapExternalContent`, and the discussion includes a maintainer-side harness reproduction of marker-injected tool output reaching the payload without a security notice.

## Solution Assessment

No. Wrapping at the shared transport boundary is the right direction, but the implementation must not trust attacker-controlled marker substrings; raw text should always be sanitized or only a validated internally generated wrapper should be preserved.
