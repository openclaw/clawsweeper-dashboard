## Summary

This PR should stay open because the proposed transport hardening is directionally useful but currently introduces a security bypass: raw marker text in a tool result is treated as already safe and skips the existing sanitizer. It also still lacks contributor-owned real behavior proof for the changed transport path.

## What This Changes

The PR wraps `toolResult` text in `transformTransportMessages` with external-content markers and adds regression tests for wrapped and already-wrapped tool results.

## Best Possible Solution

Always treat raw tool-result text as untrusted content unless it can be validated as an internally generated wrapper, add regression coverage for marker-spoofed tool output, and include real transport/payload proof in the PR body.

## Reproduction Assessment

Yes, for the PR-introduced bypass. Source inspection and a reviewer-local harness show that a `toolResult` containing raw boundary marker strings skips wrapping and sanitizer output on the PR head.

## Solution Assessment

No. Wrapping at the shared transport boundary is a reasonable direction, but trusting attacker-controlled marker substrings is not a safe implementation; the safer path is to always sanitize raw tool output or validate a full internally generated wrapper shape before skipping.
