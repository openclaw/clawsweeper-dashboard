## Summary

Keep open: current main still has only the generic task failure bucket, but this PR’s implementation has blocking consistency issues where summaries classify a different task set than the rows/operators can still see, and the contributor has not provided real CLI behavior proof.

## What This Changes

The PR adds recent and historical task failure counts to task/status/flow summaries and related tests.

## Best Possible Solution

Land a revised split that computes recent and historical failures from the same task record set each command renders, keeps the flows table aligned, and includes redacted real CLI proof.

## Reproduction Assessment

Yes, source-level. A cleanupAfter-expired failed task can remain in reconcileInspectableTasks/listTasksForFlowId until maintenance pruning, while the PR’s new summaries reuse buildTaskStatusSnapshot, which filters that task out.

## Solution Assessment

No. The recent/historical split is useful, but this patch is not the best implementation until registry, tasks, and flows classify failures from the same record set they display and the flows column width is fixed.

  - confidence: 0.9

  - body: `flows show` reports recent/historical counts through `taskSummary`, while the linked-task section still renders every `listTasksForFlowId` record. Use the same filtered or unfiltered basis so visible failed linked tasks are not hidden from the headline.
  - confidence: 0.89

  - body: Rows now pad task summaries to `TASKS_PAD`, but the header still pads `Tasks` to 14 characters. That shifts the `Goal` header left of every row value in `flows list`, a visible table regression.
