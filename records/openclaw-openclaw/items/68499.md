## Summary

Keep open. The PR targets a useful founder-visible task summary improvement, but the current head still has source-level P2 regressions where failure aggregates can disagree with rendered task rows, plus a visible flows table alignment bug and missing real behavior proof.

## What This Changes

The branch adds recentFailures and historicalFailures task summary counts and updates tasks, status, flows CLI output, and targeted tests.

## Best Possible Solution

Land the split after registry, tasks, and flows compute displayed summaries from the same record basis, the flows table width is fixed, and redacted real CLI proof is added.

## Reproduction Assessment

Yes, source-level. At PR head, a failed, timed_out, or lost task whose cleanupAfter has passed can remain listed until maintenance pruning, while buildTaskStatusSnapshot filters it out before the new summary counts are rendered.

## Solution Assessment

No. The recent/historical split is useful, but this implementation is not the best fix until all affected summaries and rendered rows use one consistent record basis and the visible flows table alignment regression is corrected.
