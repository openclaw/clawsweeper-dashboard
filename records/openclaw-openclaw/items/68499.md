## Summary

Keep open: current main still lacks the requested split, and the PR remains useful, but the head introduces cleanup-basis mismatches in task/flow summaries, a flow table alignment regression, and lacks required real behavior proof.

## What This Changes

The PR adds recent/historical task failure counts to task registry/status summaries, updates tasks/status/flows command output, and adds targeted tests.

## Best Possible Solution

Land the recent/historical split after making every summary classify failure recency from the same record set it renders, fixing the flows table width, and adding redacted real CLI behavior proof.

## Reproduction Assessment

Yes, source-level. A failed, timed_out, or lost task with cleanupAfter already in the past but still present before maintenance pruning will be rendered by tasks/flows while snapshot-derived summaries omit it.

## Solution Assessment

No. The split is useful, but using cleanup-aware buildTaskStatusSnapshot as the aggregate source for unfiltered registry, tasks-list, and flow-list/show summaries is not the narrowest maintainable implementation.
