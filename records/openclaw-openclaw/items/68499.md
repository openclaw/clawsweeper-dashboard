## Summary

Keep open. Current main still has one generic task failure bucket, so the PR is not obsolete, but the branch still introduces inconsistent summary bases for failure counts and lacks required after-fix CLI proof.

## What This Changes

The PR adds recentFailures/historicalFailures task summary fields and updates tasks, status, and flows CLI output plus targeted tests.

## Best Possible Solution

Land the split only after registry, tasks, and flows classify failures from the same record set they render, the flows table width is fixed, and redacted real CLI proof is added.

## Reproduction Assessment

Yes, source-level. In the PR, a failed/timed_out/lost task whose cleanupAfter has passed but remains before maintenance pruning is excluded by buildTaskStatusSnapshot while tasks/flows can still render that record.

## Solution Assessment

No. The recent/historical split is useful, but the current patch is not the best implementation because it mixes cleanup-filtered snapshot counts with unfiltered registry and CLI row sets.
