## Summary

Keep open: current main still only exposes aggregate task failure counts, but this PR needs changes before merge because it mixes cleanup-aware snapshots with unfiltered registry/list/flow summaries and leaves a visible flow table alignment regression. The remaining work is narrow and repairable.

## What This Changes

The PR adds recent versus historical task failure counts to task status/registry summaries, updates tasks/status/flows command output, and adds targeted command and task tests.

## Best Possible Solution

Land the recent/historical split through a shared failure-recency classifier that uses the same task record set as each rendered summary, keep cleanup-aware snapshot behavior limited to status-card views, align the flows table header with the widened task column, and update user-facing docs/changelog with focused tests.

## Reproduction Assessment

Yes. A failed, timed_out, or lost task whose cleanupAfter is already in the past but has not yet been pruned reproduces the PR-introduced mismatch: the row can still render in tasks or flows output while summary counts derived from buildTaskStatusSnapshot omit it.

## Solution Assessment

No. The requested split is the right product direction, but using the cleanup-aware status snapshot as the aggregate source for unfiltered registry, tasks-list, and flow-list/show summaries is not the narrowest maintainable implementation; classify recency over the exact displayed records instead.
