## Summary

Keep open: the split is useful, but the current branch still has correctness regressions from mixing cleanup-filtered snapshot counts with unfiltered task records, and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds recentFailures/historicalFailures task summary fields and updates tasks, status, flows CLI summaries, and targeted tests to show recent versus historical failures.

## Best Possible Solution

Land the split after registry, tasks, and flows classify failures from the same task-record basis they render, the flows table width is fixed, and redacted real CLI proof is added.

## Reproduction Assessment

Yes, source-level. A failed, timed_out, or lost task whose cleanupAfter has passed but remains in the registry before maintenance pruning is excluded by buildTaskStatusSnapshot while tasks and flows can still render or count that record.

## Solution Assessment

No. The recent/historical split is useful, but the current implementation is not the best fix because it mixes cleanup-filtered snapshot counts with unfiltered registry and CLI row sets; compute the split from the same records each command/API renders, or filter both surfaces consistently.
