## Summary

Keep open, but this PR is not merge-ready: the recent/historical split is aimed at a real task-status problem, but the patch mixes cleanup-filtered snapshot counts with unfiltered registry, task-list, and flow rows, and it still lacks real CLI behavior proof.

## What This Changes

The PR adds recent and historical task failure counts to task snapshots/registry summaries, updates tasks/status/flows CLI output to display the split, and adds focused tests.

## Best Possible Solution

Land a maintainer-approved current/retained task-health model, or a corrected version of this PR, that computes each rendered summary from the same record set, updates JSON/text contracts deliberately, fixes the flows table alignment, and includes real CLI proof.

## Reproduction Assessment

Yes, source-level. A cleanupAfter-expired failed task can remain in reconcileInspectableTasks/listTasksForFlowId until maintenance pruning, while the PR reuses buildTaskStatusSnapshot, which filters that task out of the new summaries.

## Solution Assessment

No. The split is useful, but the patch is not the best implementation until registry, task-list, and flow summaries classify failures from the same record set they display and the broader retained-task semantics are aligned with the canonical issue.
