## Summary

Keep open: current main still has one failure bucket and the PR’s direction is useful, but current head is not mergeable because the new counts mix cleanup-filtered snapshots with unfiltered task rows, the flows table alignment regresses, and external contributor real behavior proof is still missing.

## What This Changes

The PR adds recentFailures/historicalFailures task summary fields, updates tasks/status/flows CLI output to display recent versus historical failures, and adds targeted Vitest coverage.

## Best Possible Solution

Land the recent/historical split only after every summary classifies failures from the same record set it renders, the flows table width is fixed, and redacted real CLI proof is added.

## Reproduction Assessment

Yes, source-level. A failed, timed_out, or lost task with cleanupAfter already in the past but still present before maintenance pruning will be rendered by tasks/flows while the PR’s snapshot-derived summaries omit it.

## Solution Assessment

No. The split is useful, but using cleanup-aware buildTaskStatusSnapshot as the aggregate source for unfiltered registry, task-list, and flow-list/show summaries is not the narrowest maintainable implementation.
