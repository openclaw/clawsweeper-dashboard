## Summary

## What This Changes

The PR adds recent and historical task failure counts to task registry summaries and updates `tasks`, `status`, `flows`, and focused tests to display the split.

## Best Possible Solution

Land a revised task-health split that uses one record basis per rendered surface, keeps CLI/JSON contracts and docs aligned, resolves the current conflicts, and fits the broader semantics in https://github.com/openclaw/openclaw/issues/83253.

## Reproduction Assessment

Yes, source-level: a cleanupAfter-expired failed task can remain in `reconcileInspectableTasks` or `listTasksForFlowId`, while the PR’s summaries use `buildTaskStatusSnapshot`, which filters expired rows. I did not run a command reproduction because this review is read-only.

## Solution Assessment

No. The split is directionally useful, but this patch is not the best implementation until each rendered surface uses a consistent task-record basis and maintainers settle the broader current versus retained task-health semantics.
