## Summary

Keep this PR open. Current main still uses a single generic task-failure bucket, and the PR is a focused candidate for the missing founder-visible split, but the proposed patch has concrete count-alignment and table-alignment regressions that should be fixed before merge.

## What This Changes

The PR adds recent-versus-historical task failure counts to task summaries and updates the tasks, status, and flows command output plus targeted tests to show that split.

## Best Possible Solution

Land the recent/historical split only after the summaries and rendered task rows use the same record basis, the flows table header matches the widened task-summary column, and the user-facing CLI change has the expected changelog/docs treatment.

## Reproduction Assessment

Yes. A failed/timed_out/lost task with cleanupAfter already in the past, before maintenance pruning removes it, reproduces the PR-introduced mismatch: the row can still be listed while summary counts derived from buildTaskStatusSnapshot omit it.

## Solution Assessment

No. The requested split is maintainable, but reusing the cleanup-aware status snapshot for unfiltered registry, task-list, and flow-list summaries is not the best implementation; a shared failure-recency classifier over the exact displayed record set is safer.
