## Label Justifications

## Summary

Keep open: current main still has the manifest account-listing/account-resolution mismatch for raw external channel account keys, and the linked open pull request is a candidate fix rather than merged implementation.

## What This Changes

The issue reports that an external openclaw-weixin account with a raw ID like `59000514e8ad@im.bot` is listed but not resolved back to its config entry, so Gateway never reaches a working Weixin session startup.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93686, or an equivalent generic normalized manifest account resolver fix, then close this issue as implemented.

## Reproduction Assessment

Yes, source-level: current main normalizes manifest account keys for listing but resolves by direct lookup of the normalized ID, and Gateway uses that resolved account before the startup handoff. I did not run a live Weixin instance in this read-only review.

## Solution Assessment

Yes, the best fix is to align generic manifest account listing and resolution so route-safe account IDs map back to raw config keys; a Weixin-specific fallback would be the wrong boundary.
