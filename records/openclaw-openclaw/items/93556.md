## Summary

Keep open: current main and the latest release still have the manifest account-resolution mismatch, while https://github.com/openclaw/openclaw/pull/93686 is the active proof-positive fix candidate. Closing this issue before that PR lands would be premature.

## What This Changes

The issue reports that a configured external openclaw-weixin account whose raw provider ID contains characters like `@` is listed as a route-safe account but cannot be resolved back to its account config, so Gateway never reaches `startAccount` for that session.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93686 if maintainers accept the compatibility hardening, then close this issue as implemented by that merge.

## Reproduction Assessment

Yes, at source level: current main lists raw manifest account keys after `normalizeAccountId` but resolves account config by exact normalized key, so a raw key like `59000514e8ad@im.bot` becomes `59000514e8ad-im-bot` and loses its account config before `startAccount`. I did not run a live Weixin repro in this read-only review.

## Solution Assessment

Yes, the best current fix path is the open PR at https://github.com/openclaw/openclaw/pull/93686 because it aligns manifest account resolution with the same normalized account lookup contract used for listing, instead of adding Weixin-specific logic. The issue should stay open until that PR merges or maintainers choose a different fix.
