## Summary

Keep open: this is a current Feishu runtime bug, not a duplicate of the older Windows loader reports. Current main still builds Feishu as an ESM plugin with the pinned Lark SDK path that can inline the SDK ESM bundle containing a bare `__dirname` reference.

## What This Changes

The issue reports that the Feishu channel crashes under Node 24 because the bundled Lark SDK ESM path references `__dirname`.

## Best Possible Solution

Make Feishu load a Node-compatible Lark SDK entrypoint, or upgrade to an SDK release without the ESM `__dirname` bug, and add packaged-runtime regression coverage so startup cannot inline the broken ESM bundle again.

## Reproduction Assessment

Yes from source inspection, though not from a live account run. Current main still statically imports the pinned SDK from Feishu runtime code, and the pinned SDK tarball still contains the ESM `__dirname` crash site reported by the user.

## Solution Assessment

No for the suggested `globalThis.__dirname` workaround; it papers over one symptom and changes process-global semantics. The maintainable fix should be Feishu-local or dependency-local: avoid the SDK ESM bundle, externalize/load the CJS SDK entrypoint, or move to a fixed SDK version with regression coverage.
