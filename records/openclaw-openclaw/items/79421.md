## Summary

Keep open: current main and the latest release still block non-bundled conversation hooks with a registry warning diagnostic, route that warning through gateway `log.info`, and omit warning-level diagnostics from `openclaw plugins doctor`. The hooks docs, config schema, and type help now mention `allowConversationAccess`, so the remaining work is the operator warning/doctor surface rather than the policy gate itself.

## What This Changes

The issue asks OpenClaw to make default-denied non-bundled conversation-hook registration discoverable through logs, doctor output, and nearby plugin-author docs/type guidance.

## Best Possible Solution

Keep the default-deny policy and existing docs/type hints, then add a maintainer-approved warning path for blocked conversation hooks and a focused doctor report that points operators to `allowConversationAccess`.

## Reproduction Assessment

Yes, source-reproducible: current main still returns before registering non-bundled conversation hooks unless `allowConversationAccess` is explicitly true, and the remaining warning/doctor surfaces are directly visible in source. I did not run a live gateway repro because this review was read-only.

## Solution Assessment

No, current main is not the complete solution yet. The best fix is a narrow diagnostic-surfacing change that preserves the security gate, with maintainer choice on whether doctor reports all warnings or only targeted conversation-hook denials.
