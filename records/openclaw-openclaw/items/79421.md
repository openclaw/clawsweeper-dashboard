## Summary

Keep open: current main now documents `allowConversationAccess` in the hooks reference and gateway startup logs plugin diagnostics, but the conversation-hook block still only creates a warn diagnostic, warn diagnostics are routed through `log.info`, and `plugins doctor` still filters them out.

## What This Changes

The issue asks OpenClaw to keep the non-bundled conversation-hook access gate but make blocked hook registration visible through operational warning surfaces instead of only runtime inspection diagnostics.

## Best Possible Solution

Preserve the default-deny policy and plugin inspect diagnostics, but route warn-level plugin diagnostics to a real warning surface and include relevant warn diagnostics in `openclaw plugins doctor`; keep the hooks-reference docs that are already on main.

## Reproduction Assessment

Yes, at source level. The existing loader test and current registry code show that a non-bundled plugin registering conversation hooks without `allowConversationAccess=true` gets no typed hooks and only warn diagnostics, while the latest issue comment adds live operator evidence on 2026.5.19.

## Solution Assessment

Yes. The narrow maintainable fix is to improve warning/doctor visibility while leaving the security gate and explicit opt-in semantics unchanged.
