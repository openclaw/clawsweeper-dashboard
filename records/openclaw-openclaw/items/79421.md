## Summary

Keep open: current main still blocks non-bundled conversation-hook registration without `allowConversationAccess` and records only a warn diagnostic; the Gateway startup diagnostic path routes that warning through `info`, and `openclaw plugins doctor` still filters warn diagnostics out.

## What This Changes

The issue asks OpenClaw to make default-denied conversation-hook registration visible through warning-level operational logs, while preserving the existing default-deny policy and inspect diagnostics.

## Best Possible Solution

Keep the default-deny policy, preserve the inspect diagnostic, and promote denied conversation-hook registration to warning-level operational output during plugin load; surfacing warn diagnostics in `openclaw plugins doctor` is a compatible focused follow-up.

## Reproduction Assessment

Yes. Current source plus the existing loader test show the hook registration is blocked, no typed hook is registered, and the only direct signal is a warn diagnostic; Gateway then logs warn diagnostics as info and plugin doctor filters them out.

## Solution Assessment

Yes. The narrow maintainable fix is to route the existing warn-level denial to warning-level startup logging, with focused regression coverage; optionally include warn diagnostics in `openclaw plugins doctor` without changing hook access policy.
