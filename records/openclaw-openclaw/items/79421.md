## Summary

Keep open: current main improves discoverability by documenting `allowConversationAccess` in the hooks reference and logging plugin diagnostics during gateway startup, but the blocked conversation-hook path still only emits a warn diagnostic, gateway startup still routes warn diagnostics through `log.info`, and `openclaw plugins doctor` still filters regular warn diagnostics out.

## What This Changes

The issue asks OpenClaw to make default-denied non-bundled conversation-hook registrations visible through warning-level operational signals and preferably plugin doctor, while preserving the explicit `allowConversationAccess` gate.

## Best Possible Solution

Keep the default-deny security policy, preserve runtime inspect diagnostics, route blocked conversation-hook diagnostics to a warning-level startup signal, and surface relevant warn diagnostics in plugin doctor without weakening `allowConversationAccess`.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the current loader test shows non-bundled conversation hooks register zero typed hooks without `allowConversationAccess=true` and only produce warn diagnostics. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Yes, the narrow fix remains the best path: keep the trust gate unchanged and promote the existing warn diagnostic to warning-level gateway/doctor surfaces. The current docs improvement helps authoring discoverability but does not fully solve operational severity or doctor visibility.
