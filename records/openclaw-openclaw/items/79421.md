## Summary

Keep open: current `main` still blocks non-bundled conversation hooks without `allowConversationAccess` and keeps the denial in warning diagnostics, but the normal operational paths do not promote those warnings to a warning-level signal. The two exact fix PRs, https://github.com/openclaw/openclaw/pull/79430 and https://github.com/openclaw/openclaw/pull/79433, were closed without merging, so there is no implemented canonical fix yet.

## What This Changes

The issue asks for blocked conversation-hook registrations to be visible in gateway logs or plugin doctor instead of only in runtime inspect diagnostics.

## Best Possible Solution

Keep the default-deny hook policy, but promote blocked conversation-hook registration to a warning-level operational signal while preserving the existing inspect diagnostic.

## Reproduction Assessment

Yes. Current main has a focused loader test that reproduces the blocked-registration state in source: the non-bundled plugin registers conversation hooks without `allowConversationAccess`, `typedHooks` remains empty, and only warn diagnostics are recorded.

## Solution Assessment

Yes. The narrow maintainable fix is to keep authorization unchanged and surface the existing denial through `logger.warn` or warning-level gateway diagnostics, with focused regression coverage for both the log signal and preserved diagnostics.
