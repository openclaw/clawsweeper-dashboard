- `impact:session-state`: Dropped conversation hooks can prevent memory/context plugins from observing agent lifecycle events and maintaining session-adjacent state.

## Summary

Keep open: current main and the latest release still block non-bundled conversation hooks with only a warn diagnostic, route non-error gateway plugin diagnostics through the info logger, and omit warn diagnostics from `openclaw plugins doctor`. The docs/type discoverability pieces are now improved, but no merged PR or narrower open item owns the remaining warning/doctor behavior.

## What This Changes

The issue asks OpenClaw to make blocked non-bundled conversation-hook registrations discoverable beyond a plugin-scoped diagnostic.

## Best Possible Solution

Keep the default-deny policy and the improved docs/type hints, then make blocked conversation-hook registrations show up as warning-level startup output and decide whether `plugins doctor` should report warn diagnostics broadly or for this hook-policy class only.

## Reproduction Assessment

Yes, source-based. Current `src/plugins/registry.ts` still returns before registration and records only a warn diagnostic when a non-bundled conversation hook lacks `allowConversationAccess=true`; I did not run a live gateway repro under read-only constraints.

## Solution Assessment

Partially. Current docs and config type surfaces are better, but the narrow runtime/operator fix is still to promote the existing warn diagnostic to the right visible warning path and decide whether `plugins doctor` should include warn diagnostics.
