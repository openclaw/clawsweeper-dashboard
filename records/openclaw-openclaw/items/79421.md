## Summary

Keep open: current main still fail-closes non-bundled conversation hooks without `allowConversationAccess`, and the actionable signal is not promoted to a warning-level operational path; `plugins doctor` also filters out warn-level diagnostics.

## What This Changes

Request an operator-visible warning when non-bundled plugin conversation-hook registrations are denied for missing `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Best Possible Solution

Keep the default-deny policy, but make denied conversation-hook registrations visible through warning-level startup logging and, if desired, plugin doctor output while preserving the existing inspect diagnostics.

## Reproduction Assessment

Yes. The source path and existing loader test show that a non-bundled plugin registering conversation hooks without `allowConversationAccess` gets zero registered typed hooks and only warn diagnostics in the registry report.

## Solution Assessment

Yes. The narrow maintainable fix is to promote the existing warn-level diagnostic to `logger.warn` during gateway plugin bootstrap, with focused tests; surfacing warn diagnostics in `openclaw plugins doctor` would be a compatible follow-up within the same area.
