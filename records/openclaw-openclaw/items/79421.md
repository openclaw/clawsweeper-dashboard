## Label Justifications

- `P2`: The issue is a normal-priority plugin developer/operator diagnostics bug with a narrow affected surface and no core runtime outage.
- `impact:security`: The affected gate controls whether non-bundled plugins may access raw conversation content.
- `impact:session-state`: The reported production failure prevented a memory-continuity conversation hook from running, which can leave plugin-maintained memory or session state stale.

## Summary

Keep open: current main now documents `allowConversationAccess` in the hooks docs and exported config type, and gateway startup does emit plugin diagnostics, but the remaining operational gap is still source-backed because warn diagnostics are routed through `info` and `openclaw plugins doctor` still filters diagnostics down to errors only.

## What This Changes

The issue asks OpenClaw to make non-bundled conversation-hook denials discoverable through normal operational surfaces instead of only plugin-scoped runtime diagnostics.

## Best Possible Solution

Keep the default-deny security policy, preserve runtime diagnostics, route warn-level plugin diagnostics to `log.warn`, and have `openclaw plugins doctor` surface actionable warn diagnostics such as blocked conversation hooks.

## Reproduction Assessment

Yes. Current source and `src/plugins/loader.test.ts` give a high-confidence source-level reproduction: blocked non-bundled conversation hooks produce warn diagnostics, zero typed hook registrations, info-level gateway logging, and no doctor warning.

## Solution Assessment

Yes. The maintainable fix is to honor existing diagnostic severity in gateway logging and doctor output while leaving the conversation-access gate and documented opt-in unchanged.
