## Summary

Keep open: current main still blocks non-bundled conversation hooks without `allowConversationAccess` and the warning remains poorly surfaced, with gateway startup routing non-error plugin diagnostics through info-level logging and `plugins doctor` filtering warn diagnostics out.

## What This Changes

The issue asks OpenClaw to emit an operational warning when a non-bundled plugin conversation hook is blocked by missing `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Best Possible Solution

Keep the default-deny security policy, but promote denied conversation-hook registrations to warning-level startup logging and include warn diagnostics in the plugin doctor path while preserving `plugins inspect --runtime --json`.

## Reproduction Assessment

Yes. The source path and existing loader test show that a non-bundled plugin registering conversation hooks without `allowConversationAccess` gets zero registered typed hooks and only warn diagnostics in the registry report; I did not run a live gateway in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to keep the security gate unchanged and surface the existing warn diagnostic through `logger.warn`/plugin doctor output, not to default-allow hooks or fold in the separate manifest-permission RFC.
