## Summary

Keep open: current main still default-denies non-bundled conversation hooks without `allowConversationAccess`, preserves the denial only as plugin diagnostics, routes non-error startup diagnostics to `info`, and omits warn-level diagnostics from `plugins doctor`; the two directly linked warning PRs are closed and unmerged.

## What This Changes

The issue asks for denied conversation-hook registrations to surface through operational warning logs, and optionally plugin doctor, instead of only plugin inspect diagnostics.

## Best Possible Solution

Keep the default-deny policy, preserve plugin inspect diagnostics, and emit a warning-level startup signal for blocked conversation-hook registrations; plugin doctor can also surface warn diagnostics if maintainers want the broader diagnostic path.

## Reproduction Assessment

Yes. Current source and the existing loader test give a high-confidence reproduction path: a non-bundled plugin registering conversation hooks without `allowConversationAccess=true` gets zero registered typed hooks and only warn diagnostics.

## Solution Assessment

Yes. Promoting the existing warn diagnostic to `logger.warn` at load time is the narrowest maintainable fix because it keeps the security gate unchanged and only improves operational visibility.
