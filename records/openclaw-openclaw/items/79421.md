## Summary

Keep open: current main still drops non-bundled conversation-hook registrations without `allowConversationAccess` and does not promote that warn diagnostic to a warning-level operational signal.

## What This Changes

The issue asks for blocked conversation-hook registrations to surface as operational warnings instead of only plugin-scoped diagnostics.

## Best Possible Solution

Preserve the default-deny policy, keep the inspect diagnostic, and add warning-level startup logging for denied conversation-hook registrations; surfacing warn diagnostics in `openclaw plugins doctor` is a compatible same-area improvement.

## Reproduction Assessment

Yes. Current source and existing loader coverage show a non-bundled plugin registering conversation hooks without `allowConversationAccess=true` gets zero typed hooks and only warning diagnostics in the registry report.

## Solution Assessment

Yes. The narrow maintainable fix is to promote the existing denial message to `logger.warn` during registration while preserving the diagnostic buffer, with focused gateway/CLI coverage if doctor output is included.
