## Summary

Keep open: current main still drops non-bundled conversation hooks without `allowConversationAccess` and leaves the actionable signal below warning-level operational paths; the two related fix PRs were closed unmerged.

## What This Changes

The issue asks for blocked conversation-hook registrations to be visible in normal warning/doctor output instead of only in runtime inspect diagnostics.

## Best Possible Solution

Keep the default-deny policy, promote denied conversation-hook registrations to warning-level startup diagnostics, preserve inspect diagnostics, and add focused regression coverage; doctor warning output can be included if scoped cleanly.

## Reproduction Assessment

Yes, source-level. Current code and existing loader coverage show a non-bundled plugin registering conversation hooks without `allowConversationAccess` gets no registered typed hooks and only warn diagnostics in the plugin registry report.

## Solution Assessment

Yes. The narrow maintainable fix is to keep the access policy unchanged and route the existing warn diagnostic to a warning-level operational surface during plugin load, with doctor warning surfacing as a compatible same-area improvement.
