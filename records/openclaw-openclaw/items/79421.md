## Summary

Keep open: current main has fixed much of the discoverability gap by documenting `allowConversationAccess` in the hooks reference and config type comments, and gateway startup now emits plugin diagnostics into the normal plugin log path. The remaining issue is still source-backed: warn-level conversation-hook denials are routed through `log.info`, and `openclaw plugins doctor` still filters plugin diagnostics down to errors, so operators using severity filters or doctor can still miss the denial.

## What This Changes

The report asks OpenClaw to make non-bundled conversation-hook denials visible in normal operator surfaces instead of only plugin inspect diagnostics.

## Best Possible Solution

Preserve the default-deny security gate and existing inspect diagnostics, while treating warn-level plugin diagnostics as operator-visible warnings in gateway startup and plugin doctor.

## Reproduction Assessment

Yes, at source level. The existing loader test reproduces the default-deny behavior and current source shows the warn diagnostic still does not reach `log.warn` or `plugins doctor`; I did not run a live gateway under the read-only constraints.

## Solution Assessment

Yes, the narrowest maintainable repair is to keep the gate unchanged and promote the existing warn diagnostic through gateway warning logs and plugin doctor. The current docs/type additions should be kept, not replaced with a new config or hook policy.
