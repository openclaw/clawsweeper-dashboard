## Summary

Keep open: current main still blocks non-bundled conversation hooks without `allowConversationAccess` and preserves the denial primarily as registry diagnostics; gateway startup routes warn diagnostics through info-level logging, `plugins doctor` filters warning diagnostics out, and the two direct fix PRs were closed unmerged.

## What This Changes

The issue asks OpenClaw to make default-denied conversation-hook registrations visible in operational warnings instead of only plugin-scoped diagnostics.

## Best Possible Solution

Keep the default-deny policy, preserve runtime inspection diagnostics, and promote blocked conversation-hook registrations to warning-level startup logging; warn-level `plugins doctor` surfacing can be included if kept narrowly diagnostic.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current `registerTypedHook` returns before registering non-bundled conversation hooks without `allowConversationAccess`, and the existing loader test asserts zero typed hooks plus warn diagnostics.

## Solution Assessment

Yes. The best fix is narrow: keep the access gate intact, emit the existing denial reason through warning-level operational logging, and add focused tests for loader/gateway or doctor visibility.
