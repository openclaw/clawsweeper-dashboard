## Summary

Keep open: current main still blocks non-bundled conversation hooks without `allowConversationAccess=true` by appending warn diagnostics and returning before hook registration, while the operational surfaces still do not promote that condition to warning-level visibility.

## What This Changes

The issue requests visible warning/log/doctor output when plugin-loader conversation-hook registration is denied for missing `plugins.entries.<id>.hooks.allowConversationAccess=true`.

## Best Possible Solution

Keep the default-deny policy, but promote the existing warn diagnostic to warning-level gateway startup logging and consider surfacing warn diagnostics in `openclaw plugins doctor` while preserving `plugins inspect` output.

## Reproduction Assessment

Yes. Source inspection and existing loader coverage provide a high-confidence path: a non-bundled plugin registering conversation hooks without `allowConversationAccess` gets zero typed hooks and only warn diagnostics in the registry report.

## Solution Assessment

Yes. The narrow maintainable repair is to keep the access gate unchanged and route this already-classified warn diagnostic to an operational warning path, with focused regression coverage for loader and gateway diagnostic severity.
