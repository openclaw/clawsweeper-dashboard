## Summary

Keep open. Current main partially supports custom memory embedding provider IDs for owner adapters like Ollama, but the reported OpenAI-compatible provider path still cannot route `api: "openai-completions"` through the OpenAI memory adapter while preserving the custom provider's endpoint and auth.

## What This Changes

Make `memory_search` use a configured custom Bailian/DashScope embedding provider instead of returning a provider fetch error.

## Best Possible Solution

Preserve the requested custom provider id while routing OpenAI-compatible memory embeddings through the OpenAI adapter, with regression tests for adapter lookup, endpoint/auth selection, and gateway/memory-search behavior.

## Reproduction Assessment

No live high-confidence reproduction was run because Bailian credentials and gateway logs are unavailable. Source inspection gives a clear path: a custom memory provider using `api: "openai-completions"` cannot currently resolve to the OpenAI memory adapter while preserving the custom provider endpoint/auth.

## Solution Assessment

Yes, a narrow generic fix in memory embedding provider resolution and OpenAI-compatible remote client plumbing is the best direction. It should not add Bailian-specific core policy or a new config knob.
