## Summary

Keep open. Current main documents custom memory embedding provider IDs, but the OpenAI-compatible route still appears unable to honor a custom provider such as `bailian-embedding` without losing its endpoint/auth or failing adapter lookup.

## What This Changes

The issue asks `memory_search` to use a configured Bailian/DashScope OpenAI-compatible embedding provider instead of returning `fetch failed` and unavailable metadata.

## Best Possible Solution

Honor custom OpenAI-compatible memory embedding providers generically by resolving `openai-completions` to the OpenAI embedding adapter while preserving the requested provider id for baseUrl, API key, headers, and model selection, with focused tests and docs.

## Reproduction Assessment

No live high-confidence reproduction was run because no Bailian credentials or gateway logs were available. Source inspection gives a clear reproducible path: a custom `memorySearch.provider` whose `models.providers.<id>.api` is `openai-completions` cannot currently map cleanly to the OpenAI embedding adapter while preserving that custom provider's endpoint/auth.

## Solution Assessment

Yes, a narrow generic fix is the best direction. The repair should stay in memory embedding provider resolution/OpenAI-compatible client plumbing, not add Bailian-specific core policy or a new config knob.
