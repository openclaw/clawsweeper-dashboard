## Summary

Keep open. Current main and the latest shipped release still force the OpenAI embedding lookup key in the memory search path, while the linked open fix PR has closing linkage and targets the exact affected adapter/resolver path.

## What This Changes

The issue reports that `memory_search` fails for a custom OpenAI-compatible embedding provider because the configured provider endpoint and credentials are not used.

## Best Possible Solution

Land the focused provider-ID preservation in https://github.com/openclaw/openclaw/pull/81170 with regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection on current main shows a custom `memorySearch.provider` can select the OpenAI adapter through `models.providers.<id>.api`, but the adapter and resolver then force the provider lookup key back to `openai`.

## Solution Assessment

Yes. Preserving `options.provider ?? "openai"` at both OpenAI embedding boundaries is the narrow fix because adapter identity stays `openai` while endpoint, auth, and cache lookup use the caller's configured provider ID.
