## Summary

Keep open. Current main still does not carry an OpenAI-compatible custom memory provider id through adapter lookup and remote endpoint/auth resolution, so the docs-backed custom-provider memory_search path remains broken.

## What This Changes

The issue requests that memory_search use a configured OpenAI-compatible custom embedding provider instead of failing with a provider fetch error.

## Best Possible Solution

Add a narrow memory embedding fix that maps OpenAI-compatible custom provider configs to the OpenAI embedding transport while preserving the requested provider id for endpoint, auth, headers, cache identity, and model handling.

## Reproduction Assessment

Yes, by source inspection rather than a live vendor call: a provider configured with api: "openai-completions" cannot resolve a memory adapter, and the OpenAI adapter would resolve endpoint/auth through openai instead of the custom id.

## Solution Assessment

Yes, the best fix is a focused adapter-alias and provider-id preservation repair with regression coverage; a Qwen-only special case or a memorySearch.remote workaround would leave the documented custom-provider contract broken.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/47884. Current main only partially resolves custom memory embedding provider ids: api: "openai-completions" does not map to the OpenAI memory adapter, and the OpenAI memory adapter overwrites the requested custom provider id with "openai" before endpoint/auth resolution. Implement a narrow fix that treats OpenAI-compatible custom provider configs as using the OpenAI embedding transport while preserving the requested provider id for models.providers.<id> baseUrl, apiKey, headers, model normalization/cache identity, and runtime metadata. Add regression coverage for a custom provider such as bailian-embedding with api: "openai-completions" and for api: "openai" preserving the custom provider id. Do not add a new config option, do not special-case only Qwen/Bailian, and do not move provider-specific auth/catalog policy into core. Update memory docs and CHANGELOG.md if the user-facing behavior changes.
