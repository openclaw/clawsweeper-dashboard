## Summary

Keep open. Current main still only partially handles custom memory embedding provider IDs: it can look up a configured API owner, but the reported `api: "openai-completions"` path does not reach the OpenAI memory adapter while preserving the custom provider's endpoint and auth.

## What This Changes

The issue asks for `memory_search` to use a configured custom OpenAI-compatible embedding provider instead of failing with `fetch failed`.

## Best Possible Solution

Preserve the requested custom provider ID while selecting the correct OpenAI-compatible memory adapter, with regression coverage for adapter lookup, endpoint/auth selection, and gateway/memory-search behavior.

## Reproduction Assessment

Yes, at source level. A custom provider with `api: "openai-completions"` either misses the OpenAI adapter or reaches OpenAI code that forces `provider: "openai"`, so the configured custom endpoint/auth is not preserved.

## Solution Assessment

Yes. A narrow generic fix in memory embedding provider resolution and OpenAI-compatible remote client plumbing is the maintainable path; it should not add Bailian-specific core logic or a new config knob.
