## Summary

Keep open. Current main still drops the custom memory embedding provider id on the OpenAI-compatible path, and the linked open fix PR uses closing syntax for this issue but has not merged yet.

## What This Changes

The issue asks OpenClaw to make `memory_search` use a configured custom OpenAI-compatible embedding provider instead of failing with `fetch failed`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81170, or an equivalent narrow fix that preserves `options.provider` through the OpenAI memory adapter and resolver, then close this issue after merge.

## Reproduction Assessment

Yes, at source level. A custom `memorySearch.provider` can select the OpenAI-compatible adapter through `models.providers.<id>.api`, but current main then overwrites the provider id with `openai` before endpoint and auth lookup.

## Solution Assessment

Yes. Preserving the caller's provider id at the OpenAI adapter and resolver is the narrow maintainable fix; it keeps adapter identity stable and avoids Bailian-specific core logic or a new config option.
