- `impact:auth-provider`: The bug makes memory embeddings resolve endpoint and API key data for openai instead of the configured custom provider.

## Summary

Keep open. Current main and v2026.5.22 still force the OpenAI provider lookup key in the OpenAI memory embedding path, while the matching fix at https://github.com/openclaw/openclaw/pull/81170 is open and unmerged.

## What This Changes

memory_search should use the configured custom OpenAI-compatible embedding provider instead of returning an embedding/provider fetch failure.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81170 or an equivalent focused provider-id preservation fix with regression coverage, then close this issue once the fix reaches main.

## Reproduction Assessment

Yes, at source level. A custom memorySearch.provider can reach the OpenAI-compatible adapter, but current main then overwrites the provider id before the remote client reads config.models.providers[params.provider].

## Solution Assessment

Yes. Preserving options.provider ?? "openai" at both OpenAI embedding boundaries is the narrow maintainable fix because adapter identity stays OpenAI while endpoint, auth, and cache lookup use the configured provider id.
