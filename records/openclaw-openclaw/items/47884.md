## Summary

Keep open. Current main still loses the custom provider ID in the OpenAI-compatible memory embedding path, and the linked open fix PR uses closing linkage for this issue, so cleanup should wait for that PR to merge or be closed.

## What This Changes

The issue reports that `memory_search` fails with `fetch failed` when a custom OpenAI-compatible embedding provider such as `bailian-embedding` is configured.

## Best Possible Solution

Preserve the custom provider ID through the OpenAI-compatible memory adapter and resolver with regression coverage, then close this issue after the linked fix PR merges.

## Reproduction Assessment

Yes, at source level. Current main can select the OpenAI adapter through a custom provider's `api`, but then forces the provider lookup key back to `openai` before endpoint and auth resolution.

## Solution Assessment

Yes. Preserving `options.provider ?? "openai"` at both OpenAI embedding boundaries is the narrow maintainable fix because adapter identity stays `openai` while provider-config lookup uses the caller's custom ID.
