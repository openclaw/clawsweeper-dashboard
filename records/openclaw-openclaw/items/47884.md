## What This Changes

The issue asks memory_search to honor a configured custom OpenAI-compatible embedding provider instead of failing with fetch failed after routing through the default OpenAI provider.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81170 or an equivalent focused fix that preserves the caller's provider id at both OpenAI embedding boundaries, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level. Current main shows custom provider lookup can select the OpenAI adapter, but the adapter and resolver then force provider: "openai" before the remote client reads endpoint and auth config.

## Solution Assessment

Yes. Preserving options.provider ?? "openai" at the adapter and resolver is the narrow maintainable fix because adapter identity stays openai while endpoint, auth, and cache lookup use the configured provider id.
