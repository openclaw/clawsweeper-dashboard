## Summary

Keep open. Current main and the latest release still force the OpenAI provider lookup key in the memory embedding path, and the open linked PR is the right active fix path for this issue.

## What This Changes

Memory search should preserve a configured custom OpenAI-compatible embedding provider ID through adapter selection, endpoint lookup, and credential resolution.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81170 or an equivalent focused fix that preserves the requested provider ID at both OpenAI embedding boundaries, then close this issue after it reaches main.

## Reproduction Assessment

Yes. Source inspection on current main shows a custom memorySearch.provider can select the OpenAI adapter through models.providers.<id>.api, but the adapter and resolver then force the provider lookup key back to openai.

## Solution Assessment

Yes. Preserving options.provider ?? "openai" at both OpenAI embedding boundaries is the narrow maintainable fix because adapter identity stays openai while endpoint, auth, and cache lookup use the caller's configured provider ID.
