## Summary

Keep open. Current main still drops the custom provider ID in the OpenAI memory embedding path, and the linked open fix PR uses closing syntax for this issue, so the issue should stay open until that PR is reviewed, merged, or closed.

## What This Changes

A configured custom `memorySearch.provider` should be preserved for endpoint and auth lookup when semantic memory search uses an OpenAI-compatible embedding adapter.

## Best Possible Solution

Preserve the custom provider ID through the OpenAI-compatible memory adapter and remote client, cover that behavior with focused regression tests, then close this issue after the linked fix PR merges.

## Reproduction Assessment

Yes, at source level. Current main can route a custom provider to the OpenAI memory adapter via `api`, but the adapter and resolver force the lookup key back to `openai` before endpoint/auth resolution.

## Solution Assessment

Yes. Preserving `options.provider ?? "openai"` at the OpenAI adapter and resolver boundaries is the narrow maintainable fix because it keeps adapter identity stable while correcting provider-config lookup.
