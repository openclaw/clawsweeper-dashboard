## Summary

Keep open. Current main and the latest release still hardcode the OpenAI provider lookup inside the OpenAI memory embedding path, while the linked closing PR that preserves the custom provider id remains open.

## What This Changes

The issue reports that memory_search returns fetch failed instead of using a configured custom OpenAI-compatible embedding provider's endpoint and credentials.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/81170 or an equivalent focused fix that preserves the caller's provider id at both OpenAI embedding boundaries, then close this issue after it reaches main.

## Reproduction Assessment

Yes, at source level. Current main can select the OpenAI adapter for a custom OpenAI-compatible provider, but the adapter and resolver then force provider: "openai" before the remote client reads endpoint and auth config.

## Solution Assessment

Yes. Preserving options.provider ?? "openai" at the adapter and resolver is the narrow maintainable fix because adapter identity stays openai while endpoint, auth, and cache lookup use the configured provider id.
