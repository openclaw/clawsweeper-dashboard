## Summary

Keep open. Current main still has only partial custom memory-provider lookup: a configured custom provider can point at an adapter owner, but the reported `api: "openai-completions"` path is not registered as a memory embedding adapter and the OpenAI memory adapter still resolves endpoint/auth through `openai` instead of the custom provider id.

## What This Changes

The issue requests that `memory_search` work with a configured Bailian/DashScope OpenAI-compatible embedding provider instead of returning `fetch failed`.

## Best Possible Solution

Implement a focused memory embedding provider fix that treats OpenAI-compatible custom provider configs as using the OpenAI embedding transport while preserving the requested provider id for `models.providers.<id>` endpoint, headers, auth, cache identity, and model handling; alternatively, emit a clear unsupported-adapter diagnostic and document the `memorySearch.remote` workaround.

## Reproduction Assessment

Yes. The issue gives a concrete config shape, and current source inspection reproduces the failure path: `bailian-embedding` resolves toward `openai-completions`, but no such memory adapter exists and the OpenAI adapter would read `models.providers.openai` instead of the custom entry.

## Solution Assessment

No. The current partial lookup is not the best solution because it resolves an adapter owner without preserving the custom provider id through OpenAI-compatible endpoint and auth resolution; the safer repair is a narrow alias/preservation fix with regression coverage.
