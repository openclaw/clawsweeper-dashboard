- `impact:auth-provider`: The bug makes memory embeddings resolve endpoint and API key data for openai instead of the configured custom provider.

## Summary

Keep open. Current main and the latest release still force the OpenAI provider lookup key inside the OpenAI memory embedding path, while the linked fix at https://github.com/openclaw/openclaw/pull/81170 is open and not merged.

## What This Changes

The issue reports that memory_search returns fetch failed when memorySearch.provider uses a custom OpenAI-compatible embedding provider such as Bailian/DashScope.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81170 or an equivalent focused provider-id preservation fix, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level. Current main can select the OpenAI-compatible adapter through models.providers.<id>.api, but the adapter and resolver then force provider: "openai" before the remote client reads endpoint and auth config.

## Solution Assessment

Yes. Preserving options.provider ?? "openai" at both OpenAI embedding boundaries is the narrow maintainable fix because adapter identity stays OpenAI while endpoint, auth, and cache lookup use the configured provider id.
