## Summary

Keep open: current main still scans deep plugin code-safety directories without passing `excludeTestFiles`, and the active implementation path is still an open PR rather than merged code.

## What This Changes

The issue reports false critical `env-harvesting` findings from bundled plugin test files because deep plugin code-safety scans include non-runtime tests and heuristic matching can see string literals.

## Best Possible Solution

Review and land one focused implementation that excludes non-runtime test/spec/mock paths from deep plugin and skill code-safety scans while preserving manifest-entry and runtime-reachable coverage; track string-literal masking separately if still needed.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main calls the scanner without `excludeTestFiles`, the scanner default includes tests, and the cited Discord test files still contain the trigger patterns. I did not run a live audit because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable direction is to reuse the existing scanner test-exclusion option for deep audits while preserving explicit/runtime-reachable scan coverage. The issue should stay open until one linked fixing PR is merged or the remaining scope is otherwise resolved.
