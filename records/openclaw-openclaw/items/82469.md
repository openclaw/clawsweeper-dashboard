## Summary

Keep open: current `main` still routes deep plugin code-safety scans through the scanner without `excludeTestFiles`, so plugin test files remain eligible for false-positive critical findings; the focused fix PR is still open, not merged.

## What This Changes

The issue reports false-positive `plugins.code_safety` critical env-harvesting findings from bundled plugin test files because deep plugin scans include test files and can still match fixture `process.env` plus network-send patterns.

## Best Possible Solution

Review and land one focused PR, such as https://github.com/openclaw/openclaw/pull/82535, that excludes non-runtime test/spec/mock paths from deep plugin and skill code-safety scans while preserving forced runtime entry coverage; track string-literal masking separately if still needed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: deep plugin audits call `scanDirectoryWithSummary` without `excludeTestFiles`, while the scanner default includes test files and still flags adjacent `process.env` plus network-send patterns as critical.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing scanner test-file exclusion for deep audit scans and keep explicit runtime-entry coverage; the open focused PR should be reviewed or replaced before closing this issue.
