## Summary

Keep open: current main and the latest release still run deep plugin code-safety scans without passing `excludeTestFiles`, while the scanner default includes test files. An open focused fix PR exists, but it has not merged, so this issue is not implemented or superseded yet.

## What This Changes

The issue asks for `plugins.code_safety` deep audits to stop raising critical `env-harvesting` false positives from bundled plugin test/spec/mock files and string-literal network words.

## Best Possible Solution

Land one focused implementation that excludes non-runtime test/spec/mock paths from deep plugin and skill code-safety scans while preserving explicit manifest/runtime-entry scan coverage; split broader string-literal heuristic hardening only if still needed.

## Reproduction Assessment

Yes. Source inspection shows current main still scans plugin deep-audit directories without `excludeTestFiles`, while the scanner default includes test files; I did not run a live audit in this read-only review.

## Solution Assessment

Yes. Reusing the existing scanner `excludeTestFiles` option at the deep-audit helper is the narrow maintainable fix for the reported false positives, as long as forced manifest/runtime entries remain scanned.
