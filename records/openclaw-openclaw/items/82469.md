## What This Changes

The issue asks OpenClaw to stop reporting critical `plugins.code_safety` false positives from bundled plugin test files and string-literal-only env-harvesting context.

## Best Possible Solution

Review and land one canonical PR that excludes non-runtime test/spec/mock paths from deep plugin and skill code-safety scans while preserving forced manifest-entry scanning; track string-literal masking separately if it still reproduces in production files.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main still calls the scanner without `excludeTestFiles`, while the scanner defaults that option to false and only skips test paths when it is true.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing scanner option for deep audits, keep explicit/runtime entry files force-scanned, and leave broader string-literal heuristic changes to a separate focused follow-up if needed.
