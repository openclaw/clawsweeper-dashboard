## What This Changes

The PR changes memory-core SQLite reindex publishing, adds an `allowCreate` flag to database opening, opts post-swap reopens into no-create mode, and adds POSIX swap/read-only probe tests.

## Best Possible Solution

Repair the publish sequence so the target main file and sidecars are never mismatched, apply no-create behavior to the affected existing-index open path without breaking first-run creation, and add live gateway/concurrent search proof.

## Reproduction Assessment

Yes from source and reporter evidence, but I did not run a live gateway repro in this read-only review. Current main uses a create-enabled `DatabaseSync` open and a swap path that can separate the live gateway handle from the on-disk index, and the linked issue includes deleted-descriptor proof.

## Solution Assessment

No. The PR is directionally right, but the current implementation deletes target sidecars after publishing and leaves the affected gateway open path create-enabled during fallback absent windows.

- **[P2] Do not delete newly published WAL sidecars:** `extensions/memory-core/src/memory/manager-atomic-reindex.ts:147-148`
  - body: `moveMemoryIndexFiles(tempPath, targetPath, options)` renames the temp main DB and any temp `-wal`/`-shm` files into the target names before this cleanup runs. These lines then remove `targetPath-wal` and `targetPath-shm` unconditionally, so a temp DB that still has WAL frames after close/checkpoint would lose its newly published sidecars instead of only clearing stale old ones.
  - confidence: 0.89
- **[P2] Guard the gateway open path against auto-create:** `extensions/memory-core/src/memory/manager-db.ts:14`
  - body: The new no-create mode defaults to `true`, and the only false call sites are the reindex process's own post-swap reopens. `openDatabase()` still calls this helper without the flag, so on the Windows fallback path, where the target is still moved away before temp is moved in, a queued gateway/lazy open can still auto-create an empty DB and reproduce the same missing-metadata race.
  - confidence: 0.82
