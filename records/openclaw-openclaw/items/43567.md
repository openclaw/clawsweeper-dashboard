## Summary

Keep open. Current main fixes the narrow pre-compaction memory-flush prompt leak, but it still does not define or render intentionally visible runtime/system messages as a distinct display class across chat history projection, Control UI, and TUI replay.

## What This Changes

Request to show internal/runtime system messages in a clearly distinct UI treatment instead of normal chat bubbles.

## Best Possible Solution

Define a structural display contract for intentionally visible runtime/system messages, carry it through chat.history/session projection, render it with a dedicated Control UI system treatment, and make TUI live/history display use its system notice path while continuing to suppress artifacts that should never be user-visible.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by projecting or rendering a visible role system or internal_system-provenance row on current main: Control UI falls through to generic other styling, and TUI history replay does not route ordinary system rows to addSystem.

## Solution Assessment

Unclear for an automated fix lane. Distinct rendering is the right end state for intentionally visible runtime messages, but maintainers should first resolve the hide-versus-label policy with the open chat.history filtering work.
