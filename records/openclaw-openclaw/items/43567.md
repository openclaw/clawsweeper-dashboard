## Summary

Keep open. Current main has important partial fixes: the pre-compaction memory-flush prompt is now runtime-only for transcript display, and compaction checkpoints render as a distinct divider, but the broader request still needs a shared product/API decision for intentionally visible runtime/system messages rather than normal chat bubbles.

## What This Changes

Request that Control UI/TUI render internal system/runtime messages in a visually distinct block instead of normal user or assistant chat bubbles.

## Best Possible Solution

Define one shared runtime/system event display contract: internal-only prompts stay out of chat history, while intentionally visible runtime notices carry structured provenance/audience so Gateway projection, Control UI, and TUI can render them in a labeled system block instead of user chat.

## Reproduction Assessment

Yes, at source level. Restart continuations still become internal_system-provenance agentTurn text in normal body fields, and the shared display/UI path has no generic internal_system or runtime/system block branch; no live UI/TUI run was performed.

## Solution Assessment

Unclear until maintainers choose the product boundary. A distinct block is appropriate for intentionally visible runtime notices, but classes like memory flush and some continuations may be better solved as hidden/runtime-only events with shared provenance.
