## Summary

Keeping this open: the specific pre-compaction memory-flush transcript leak is fixed, but current main still lacks a general product/API contract for rendering runtime-originated messages as a distinct non-user block. Generic SystemEvent text is still flattened into prompt text, and restart continuations still record internal-system provenance without a display projection that treats them differently in chat history.

## What This Changes

The issue requests that internal runtime/system messages remain visible when useful but render as clearly system-originated instead of normal user or assistant chat bubbles.

## Best Possible Solution

Define one shared runtime/system audience or provenance contract, then project it through gateway history and Control UI/TUI so system-originated items are hidden or labeled consistently across surfaces.

## Reproduction Assessment

Yes, at source level. Generic SystemEvent entries still have no audience/display metadata and flatten into plain System prompt text, while restart continuations carry `internal_system` provenance without a user-facing display projection that treats them as a distinct runtime/system block.

## Solution Assessment

Unclear pending maintainer product/API direction. A shared provenance or audience contract with one display projection is safer than per-surface styling or string-prefix filters.
