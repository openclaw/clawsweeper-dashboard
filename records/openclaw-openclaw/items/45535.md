## Summary

Keep this PR open, but it is not mergeable as-is: current main has separately fixed the linked Bailian/DashScope usage reports, while this branch still carries a narrower alias-precedence idea, is conflicting, lacks real behavior proof, and introduces an all-zero JSONL usage overwrite regression.

## What This Changes

The PR changes token usage normalization to prefer positive token alias values over earlier zero-valued fields and adds OpenAI `prompt_tokens`/`completion_tokens` aliases to the CLI runner usage helper.

## Best Possible Solution

If maintainers still want the positive-over-zero alias rule, rework it against current `normalizeUsage` and `toCliUsage`, preserve all-zero JSONL records as absent, add focused regression tests, and include redacted real behavior proof from an affected provider or CLI output path.

## Reproduction Assessment

Yes for the PR-introduced regression: the diff returns a `CliUsage` object for all-zero records, and JSONL parsing overwrites prior usage with `toUsage(parsed.usage) ?? usage`. The original provider symptom is also source-explainable from alias precedence, but the linked Bailian reports were fixed elsewhere on main.

## Solution Assessment

No. The intended normalization may still be useful, but this patch should not merge because it targets stale code, conflicts with current main, and regresses JSONL usage preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve prior usage when parsing all-zero JSONL events:** `src/agents/cli-runner/helpers.ts:149`
  - body: The new picker falls back to zero and the guard now only rejects `undefined`, so an end-of-stream record like `usage: { input_tokens: 0, output_tokens: 0, total_tokens: 0 }` returns a usage object. Since JSONL parsing keeps doing `usage = toUsage(parsed.usage) ?? usage`, that later bookkeeping record can replace a real earlier token total with zeros; keep all-zero usage absent or only overwrite when the parsed usage has a positive value.
  - confidence: 0.93
