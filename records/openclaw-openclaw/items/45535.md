Keep this PR open. Current main still has the mixed zero/positive token-alias gap the PR targets, but the submitted patch is stale against the active CLI parser and can regress JSONL usage retention, so it should be refreshed or replaced rather than merged as-is.

## What This Changes

The PR changes token usage normalization and an older CLI runner helper to prefer positive token aliases before zero values and to add OpenAI-style prompt/completion aliases for CLI usage parsing.

## Best Possible Solution

Implement the positive-first alias resolver on current main in `src/agents/usage.ts`, update the active `toCliUsage` path in `src/agents/cli-output.ts`, preserve zero-only stream events as non-updates, and add focused regression tests plus a changelog entry if required.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve prior nonzero JSONL usage:** `src/agents/cli-runner/helpers.ts:141-148`
  - body: The new finite fallback returns a usage object when every parsed token field is `0`. JSONL parsing keeps the latest parsed usage with `usage = toUsage(parsed.usage) ?? usage`, so a trailing zero-only bookkeeping event can replace earlier real totals and make the run look like it used no tokens. Keep zero-only usage returning `undefined` before updating stream state.
  - confidence: 0.91
- **[P2] Update the active CLI usage parser:** `src/agents/cli-runner/helpers.ts:130-138`
  - body: Current main no longer parses CLI usage from this helper; CLI execution imports `parseCliOutput` and streaming parsing from `src/agents/cli-output.ts`. Adding prompt/completion aliases here will not fix CLI usage reporting unless the active `toCliUsage` implementation is updated too.
  - confidence: 0.89
