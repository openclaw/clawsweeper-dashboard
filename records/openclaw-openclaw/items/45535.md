## Summary

Keep open: current main fixed the linked Bailian/DashScope zero-usage reports through other merged PRs, but this PR still proposes a distinct positive-over-zero alias rule that is not on main and the branch should not merge as-is.

## What This Changes

The PR changes agent usage normalization and the older CLI runner usage helper to prefer positive token alias values over earlier zero-valued fields and add OpenAI prompt/completion aliases.

## Best Possible Solution

Rework the positive-over-zero alias normalization against current `src/agents/usage.ts` and `src/agents/cli-output.ts`, preserve all-zero records as absent, add focused regression tests, and include redacted real-provider or CLI output proof.

## Reproduction Assessment

Yes for the PR-introduced regression: from the diff, all-zero usage can become a real object and overwrite prior JSONL usage. The original Bailian symptom is source-explainable, but the linked reports were fixed elsewhere on main.

## Solution Assessment

No. The intended alias rule may still be useful, but this branch targets stale code and introduces an all-zero JSONL overwrite; a current-main rewrite with tests and proof is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve prior usage when parsing all-zero JSONL events:** `src/agents/cli-runner/helpers.ts:142-148`
  - body: The new picker falls back to zero and the guard now only rejects `undefined`, so an end-of-stream record like `usage: { input_tokens: 0, output_tokens: 0, total_tokens: 0 }` returns a usage object. Since JSONL parsing keeps doing `usage = toUsage(parsed.usage) ?? usage`, that later bookkeeping record can replace a real earlier token total with zeros; keep all-zero usage absent or only overwrite when the parsed usage has a positive value.
  - confidence: 0.93
