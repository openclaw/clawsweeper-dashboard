## Summary

Keep open: the useful direction is real, but this PR is not safe to merge because it patches an obsolete CLI helper, can turn zero-only bookkeeping records into usage updates, and does not update the active CLI parser. The linked Bailian issues were fixed on main by narrower WebSocket/status changes, while the remaining active CLI alias gap is separately tracked by https://github.com/openclaw/openclaw/pull/78085.

## What This Changes

The PR changes shared usage normalization and an older CLI runner helper to prefer positive OpenAI-compatible token aliases over zero-valued aliases.

## Best Possible Solution

Apply a current-main replacement that fixes positive-first alias selection in normalizeUsage, updates or coordinates with the active cli-output parser work, preserves zero-only records as non-updates, and adds focused regressions plus changelog coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main selects input_tokens: 0 before a later positive prompt_tokens alias in normalizeUsage, and the active CLI parser drops prompt_tokens/completion_tokens entirely. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No. The positive-first normalization direction is reasonable, but this PR updates an obsolete CLI helper and can overwrite prior nonzero stream usage with zero-only records; the safer fix belongs in current src/agents/usage.ts and src/agents/cli-output.ts or the canonical CLI follow-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

  - confidence: 0.89
