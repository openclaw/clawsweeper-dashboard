## Summary

Keep open: the branch is aimed at a real usage-accounting bug, but this PR head is not mergeable as-is because it can replace prior nonzero CLI usage with a zero-only usage object and it targets an older parser surface. The linked Bailian issues have since been fixed through narrower merged changes, while the remaining mixed-zero and CLI-alias cleanup still needs a current-main rework plus real behavior proof.

## What This Changes

The branch changes usage normalization to prefer positive token alias values over earlier zero aliases and adds prompt/completion/cache aliases to the legacy CLI runner usage helper.

## Best Possible Solution

Rework the fix on current main so shared normalization safely prefers positive aliases, CLI usage parsing handles OpenAI-style aliases in `src/agents/cli-output.ts`, zero-only events do not erase prior usage, and focused regression coverage plus real proof back the behavior.

## Reproduction Assessment

Yes, at source level: current main still lets `input_tokens: 0` win over a later positive `prompt_tokens`, and the CLI parser still drops `prompt_tokens`/`completion_tokens`. I did not establish live Bailian, llama.cpp, or other provider proof in this read-only review.

## Solution Assessment

No. The direction is useful, but this branch has a zero-only CLI overwrite bug, is stale against the current parser surface, and needs a current-main implementation that preserves cache-safe normalization semantics.
