## Summary

Keep open: the direction is still relevant, but this stale PR lacks real behavior proof and its current patch can regress CLI usage reporting by letting zero-only JSONL events overwrite earlier nonzero usage. Current main fixed the linked Bailian WebSocket/status path, while the narrower CLI alias subset is separately tracked by https://github.com/openclaw/openclaw/pull/78085.

## What This Changes

The PR adds positive-first token-field selection to shared usage normalization and legacy CLI runner usage parsing, including prompt/completion aliases and cached token support.

## Best Possible Solution

Land a current-main rework that prefers positive aliases only where safe, prevents zero-only events from replacing nonzero usage, adds focused regression coverage, and uses https://github.com/openclaw/openclaw/pull/78085 for the narrower CLI alias subset if that lands first.

## Reproduction Assessment

Yes for source-level reproduction: current main still lets `input_tokens: 0` win over a later positive `prompt_tokens`, and the CLI parser still drops prompt/completion aliases. I did not establish live Bailian, llama.cpp, or other provider proof in this read-only review.

## Solution Assessment

No. The proposed direction is useful, but the branch can turn zero-only CLI events into real usage objects, is stale against the current CLI parser surface, and needs cache-safe normalization semantics before merge.
