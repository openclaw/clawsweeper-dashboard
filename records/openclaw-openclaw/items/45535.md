## Summary

Keep open, but do not merge as-is. The positive-first alias direction is useful, yet this branch is stale against current usage normalization, patches an old CLI helper instead of the active CLI parser, can overwrite stream usage with all-zero records, and still lacks after-fix real behavior proof.

## What This Changes

The PR changes usage normalization to prefer positive token aliases over zero-valued OpenAI-compatible aliases and adds prompt/completion token aliases to a CLI runner helper.

## Best Possible Solution

Port the positive-first alias selection into current `normalizeUsage()` while preserving cache/timing/clamping semantics, handle CLI aliases in `src/agents/cli-output.ts` or the newer focused PR, and add regressions plus real behavior proof or an explicit proof override.

## Reproduction Assessment

Yes for source-level reproduction: current main still chooses `input_tokens` before `prompt_tokens` in `normalizeUsage()`, and the active CLI parser still lacks `prompt_tokens`/`completion_tokens` aliases. I did not run a live provider reproduction or tests in this read-only review.

## Solution Assessment

No. The positive-first normalization idea is reasonable, but this stale branch patches an obsolete CLI helper and would need to be rebuilt on current `src/agents/usage.ts` and `src/agents/cli-output.ts` without losing newer normalization behavior.
