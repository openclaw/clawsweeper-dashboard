## Summary

Keep this PR open, but do not merge the branch as-is: the alias-precedence idea is still useful, while the current diff can overwrite prior nonzero CLI usage with all-zero usage events and lacks real behavior proof.

## What This Changes

The PR changes token usage normalization to prefer positive alias values over zero-valued fields and adds OpenAI prompt/completion aliases to the CLI usage helper.

## Best Possible Solution

Reapply the useful alias-precedence behavior in the current usage and CLI parser paths with all-zero overwrite guards, focused regression tests, and redacted provider or CLI proof.

## Reproduction Assessment

Yes for the PR-introduced regression from source inspection: the fallback pass can return 0 and JSONL parsing keeps the latest defined usage, so an all-zero event can replace earlier nonzero usage. The original mixed zero/positive alias symptom is source-explainable on current main, while the linked Bailian reports were fixed separately for their documented path.

## Solution Assessment

No. The proposed direction is useful, but the branch should be reworked against current main with all-zero guards, current CLI parser coverage, and real behavior proof before merge.
