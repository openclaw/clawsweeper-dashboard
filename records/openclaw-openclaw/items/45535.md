## Summary

Keep open: the PR addresses a real token-usage gap, but current main still has source-level alias gaps and this branch is stale, conflicting, missing real behavior proof, and introduces a zero-only CLI overwrite bug.

## What This Changes

The PR changes shared agent usage normalization and legacy CLI runner usage parsing to prefer positive OpenAI-compatible token aliases over zero fields and adds prompt/completion/cache aliases.

## Best Possible Solution

Rework this on current main so shared normalization safely prefers positive aliases, CLI usage parsing handles OpenAI-style aliases in `src/agents/cli-output.ts`, zero-only events do not erase prior usage, and focused regression coverage plus real proof back the change.

## Reproduction Assessment

Yes at source level: current main still lets `input_tokens: 0` win over a later positive `prompt_tokens`, and the CLI parser still drops `prompt_tokens`/`completion_tokens`. I did not establish live Bailian, llama.cpp, or other provider proof in this read-only review.

## Solution Assessment

No. The direction is useful, but this branch has a zero-only CLI overwrite bug, is stale against the current parser surface, and needs a current-main implementation that preserves cache-safe normalization semantics.
