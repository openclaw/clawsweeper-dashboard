## Summary

Keep open: the original Bailian reports were closed by later merged usage work, but this stale branch still contains a distinct mixed-zero alias fix that current main does not fully cover, and the branch is not merge-ready because it targets an older CLI helper and can overwrite session usage with zero-only events.

## What This Changes

The PR changes `src/agents/usage.ts` and the older `src/agents/cli-runner/helpers.ts` usage helper to prefer positive OpenAI-compatible token aliases such as `prompt_tokens` and `completion_tokens` over zero-valued fields.

## Best Possible Solution

Rework current main's shared normalizer and CLI output parser to prefer positive aliases without accepting zero-only overwrites, preserving existing cache accounting and letting https://github.com/openclaw/openclaw/pull/78085 own the narrower CLI-only subset if it lands first.

## Reproduction Assessment

Yes, at source level: current main still lets `input_tokens: 0` win over later positive prompt-token aliases, and the current CLI parser still drops `prompt_tokens` / `completion_tokens`. I did not establish live Bailian, llama.cpp, or provider proof in this read-only review.

## Solution Assessment

No. The direction is useful, but this stale branch introduces a zero-only overwrite bug and must be reworked against current main's `src/agents/usage.ts` and `src/agents/cli-output.ts` surfaces.
