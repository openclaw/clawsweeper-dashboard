## Summary

Keep open: the reported token-usage problem is still source-reproducible on current main, but this PR is not merge-ready because it has a zero-only usage overwrite bug, is stale against the current CLI parser surface, and still lacks real behavior proof.

## What This Changes

The branch changes usage normalization and the old CLI runner helper to prefer positive token aliases over zero fields and add OpenAI-style prompt/completion/cache aliases.

## Best Possible Solution

Rework this on current main so `normalizeUsage()` and `toCliUsage()` handle mixed zero/positive aliases safely, preserve prior nonzero CLI usage snapshots, and add focused regression tests plus real behavior proof or maintainer override.

## Reproduction Assessment

Yes, source-level: current main still lets zero-valued first aliases mask later positive OpenAI-style aliases, and the current CLI parser still drops `prompt_tokens`/`completion_tokens`. I did not run a live Bailian, llama.cpp, or other provider setup in this read-only review.

## Solution Assessment

No. The direction is useful, but this branch has a zero-only CLI overwrite bug, targets a stale parser surface, and needs a current-main implementation with proof before it is the maintainable fix.
