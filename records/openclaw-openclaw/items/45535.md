## Summary

Keep open: the PR targets a real usage-accounting gap, but it is not merge-ready because it lacks real behavior proof, is conflicting against the current CLI parser surface, and introduces a zero-only usage overwrite regression.

## What This Changes

The branch changes shared usage normalization and the legacy CLI runner usage helper to prefer positive token aliases over zero fields and add prompt/completion/cache aliases.

## Best Possible Solution

Rework this on current main so shared normalization prefers positive aliases safely, CLI usage parsing handles OpenAI-style aliases in `src/agents/cli-output.ts`, zero-only events do not erase prior usage, and focused regression coverage plus real proof back the change.

## Reproduction Assessment

Yes for source-level reproduction: current main still lets `input_tokens: 0` win over a later positive `prompt_tokens`, and the CLI parser still drops `prompt_tokens`/`completion_tokens`. I did not establish live Bailian, llama.cpp, or other provider proof in this read-only review.

## Solution Assessment

No. The direction is useful, but this branch has a zero-only CLI overwrite bug, is stale against the current parser surface, and needs a current-main implementation that preserves cache-safe normalization semantics.
