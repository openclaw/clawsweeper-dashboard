## Summary

Keep open: current main still has the source-level usage alias gaps this PR targets, but the branch is stale/conflicting, introduces a zero-only CLI usage regression, and lacks required real behavior proof.

## What This Changes

The PR changes shared usage normalization and the legacy CLI runner usage helper to prefer positive token aliases over zero-valued fields and add prompt/completion/cache aliases.

## Best Possible Solution

Rework the fix on current main so shared usage normalization prefers positive aliases safely, CLI usage parsing handles prompt/completion aliases in `src/agents/cli-output.ts`, zero-only events do not erase prior usage, and focused regression coverage protects both paths.

## Reproduction Assessment

Yes for source-level reproduction: current main still lets `input_tokens: 0` win over a later positive `prompt_tokens`, and the CLI parser still drops `prompt_tokens`/`completion_tokens`. I did not establish live Bailian, llama.cpp, or other provider proof in this read-only review.

## Solution Assessment

No. The proposed direction is useful, but this branch can turn zero-only CLI events into real usage objects, is stale against the current parser surface, and needs cache-safe current-main normalization semantics before merge.
