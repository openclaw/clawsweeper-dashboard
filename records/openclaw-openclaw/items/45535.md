## Summary

Keep open: the linked Bailian reports were closed by later WebSocket/status fixes, but this PR still targets a source-visible mixed-zero alias gap and an older CLI alias gap now represented by https://github.com/openclaw/openclaw/pull/78085. The branch should not merge as-is because it can overwrite prior CLI usage with zero-only records and it still lacks real behavior proof.

## What This Changes

This PR changes token usage normalization to prefer positive aliases over zero values and adds OpenAI prompt/completion aliases to the legacy CLI runner usage helper.

## Best Possible Solution

Implement a current-main fix that prefers positive aliases only where safe, prevents zero-only events from replacing nonzero usage, and lands with focused regression tests plus redacted runtime proof; let https://github.com/openclaw/openclaw/pull/78085 handle the prompt/completion-only CLI subset if it lands first.

## Reproduction Assessment

Yes for source-level reproduction: current main picks input_tokens before prompt_tokens, so input_tokens: 0 with prompt_tokens: N normalizes input as 0. I did not establish live Bailian or CLI provider proof in this read-only review.

## Solution Assessment

No. The direction is useful, but the branch can turn zero-only CLI events into real usage objects, is stale against the current CLI parser surface, and needs cache-safe normalization semantics before merge.
