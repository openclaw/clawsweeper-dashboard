## Summary

Keep open for human handling: current main still has a source-level mixed-alias normalization gap, but this stale branch is not the right merge shape because it updates an obsolete CLI helper, must preserve newer usage/cache/timing normalization, lacks tests and real behavior proof, and overlaps the active CLI follow-up.

## What This Changes

The PR changes token usage normalization to prefer positive OpenAI-compatible token aliases over zero-valued aliases and adds prompt/completion token aliases to a CLI runner helper.

## Best Possible Solution

Replace the stale branch with a current-main fix that adds positive-first alias selection without dropping current normalization semantics, coordinates with the active `cli-output.ts` CLI alias work, adds focused regressions, and includes real behavior proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes for source-level reproduction: current main still selects `input_tokens: 0` before a later positive `prompt_tokens` alias in `normalizeUsage`, and the active CLI parser still omits `prompt_tokens`/`completion_tokens`. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No. The positive-first normalization direction is reasonable, but this branch updates an obsolete CLI helper and predates current-main cache/timing normalization; the safer fix belongs in current `src/agents/usage.ts` and `src/agents/cli-output.ts` or the focused CLI follow-up.
