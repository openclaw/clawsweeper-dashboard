## Summary

Keep open: the positive-first token alias direction is useful, but this PR is stale and incorrect against current main because it updates an obsolete CLI helper and can turn zero-only usage events into overwrites of prior nonzero JSONL usage.

## What This Changes

The PR changes usage normalization and an older CLI-runner usage helper to prefer positive token aliases over zero-valued aliases for OpenAI-compatible usage payloads.

## Best Possible Solution

Implement positive-first alias selection in current normalizeUsage and cli-output parsing, preserve cache/timing semantics and zero-only no-op behavior, then add focused regressions and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: pass normalizeUsage a payload with input_tokens: 0 plus positive prompt_tokens, or feed the active CLI parser prompt_tokens/completion_tokens-only usage.

## Solution Assessment

No. The direction is right, but the PR edits a stale CLI helper and needs to be rebuilt against current usage.ts and cli-output.ts while keeping zero-only usage events from overwriting prior totals.
