## Summary

Keep open: the linked Bailian/OpenAI-compatible WS/status bugs have since been addressed on main, but this branch still has a blocking JSONL usage regression, lacks real behavior proof, and is not fully aligned with the current parser layout.

## What This Changes

The PR changes usage normalization and the CLI usage helper to prefer positive token aliases over zero-valued aliases and to recognize prompt/completion/cache token fields.

## Best Possible Solution

Rework the useful normalization idea against current `src/agents/usage.ts` and `src/agents/cli-output.ts`, preserve all-zero JSONL events as non-overwriting, add focused regression tests, and include redacted real run proof.

## Reproduction Assessment

Yes for the branch regression: source inspection shows `toUsage` can return all-zero usage and `parseCliJsonl` then overwrites the prior value. No real Bailian/OpenAI-compatible runtime proof is present for the proposed fix itself.

## Solution Assessment

No. The current branch is not the best fix as-is because it can drop prior JSONL usage and needs to be rebased onto the current parser files; a narrow rework with tests is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve prior JSONL usage when a later event is all zero:** `src/agents/cli-runner/helpers.ts:149-153`
  - body: With the new undefined-only guard, a usage payload containing only zero-valued fields now returns a `CliUsage` object instead of `undefined`. `parseCliJsonl` assigns that over prior usage on every line, so a later bookkeeping/status event can erase real token totals and make session/cost reporting look empty; keep all-zero parsed usage non-overwriting.
  - confidence: 0.91
