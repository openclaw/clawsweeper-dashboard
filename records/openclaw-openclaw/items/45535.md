## Summary

Keep open: merged work fixed the linked Bailian/DashScope prompt/completion-only usage path, but the exact mixed zero/positive alias case is still source-reproducible on current main, and this stale branch has a blocking all-zero overwrite bug plus no real behavior proof.

## What This Changes

The PR changes usage normalization to prefer positive token aliases over zero values and adds OpenAI prompt/completion aliases to the older CLI runner usage helper.

## Best Possible Solution

Rework the remaining normalization on current main with zero-record overwrite guards, cache-subtraction-safe alias precedence, focused regression tests, and redacted provider or CLI proof; use the narrower CLI PR for that subset if it lands first.

## Reproduction Assessment

Yes for source-level reproduction: current normalizeUsage nullish-selects input_tokens before prompt_tokens, so a payload with input_tokens: 0 and prompt_tokens: N normalizes input as 0. I did not establish live Bailian or CLI provider proof in this read-only review.

## Solution Assessment

No. The branch direction is useful, but this implementation can overwrite prior nonzero CLI usage with zero-only events and needs to be rebased onto the current parser surface before it is a maintainable fix.
