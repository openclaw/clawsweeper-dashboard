## Summary

Keep open: the change is aimed at a real usage-normalization gap, but this branch is not merge-ready because it has a zero-only CLI usage overwrite bug, lacks after-fix real behavior proof, and is stale against the current CLI parser surface.

## What This Changes

The PR changes shared and CLI usage normalization to prefer positive token aliases over zero values and adds OpenAI-style prompt/completion/cache aliases.

## Best Possible Solution

Rework this on current main so shared normalization safely prefers positive aliases without breaking cache accounting, handle CLI aliases in the current parser or defer that subset to https://github.com/openclaw/openclaw/pull/78085, and back it with focused regression tests plus real behavior proof.

## Reproduction Assessment

Yes at source level: current main still lets `input_tokens: 0` win over a later positive `prompt_tokens`, and the CLI parser still drops `prompt_tokens`/`completion_tokens`. I did not establish live Bailian, llama.cpp, or other provider proof in this read-only review.

## Solution Assessment

No. The direction is useful, but this branch has a zero-only CLI overwrite bug, is stale against the current parser surface, and needs a current-main implementation that preserves cache-safe normalization semantics.
