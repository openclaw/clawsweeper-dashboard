## Summary

Keep this PR open: the user-visible bug is real and the proposed direction is useful, but the submitted formatter can double-prefix already canonical model refs and the PR’s TUI PTY lane is failing on exactly that case.

## What This Changes

The PR changes the TUI `/model` success message to display the gateway-resolved canonical model ref when available and adds regression tests for canonical and fallback display behavior.

## Best Possible Solution

Use the existing canonical model-ref formatter for the resolved confirmation path so aliases display the applied provider/model while already-prefixed model ids are not duplicated, and keep the raw-input fallback when resolved data is absent.

## Reproduction Assessment

Yes, from source and PR proof: current main emits `model set to ${args}` after `/model`, and the before screenshot shows that raw alias differing from the resolved status-bar model. I did not run a local TUI reproduction in this read-only review.

## Solution Assessment

No as submitted: using `result.resolved` is the right fix location, but manual string concatenation is not the best formatter because resolved model values can already contain the provider prefix. The narrower safe fix is to reuse `modelKey` or an equivalent shared formatter for the display string.
