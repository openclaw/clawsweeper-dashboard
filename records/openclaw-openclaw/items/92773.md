Keep open. The fix direction and real TUI proof are useful, but the current patch is not merge-ready because it can render already-qualified resolved models as `provider/provider/model`, and the live TUI PTY check failed on that exact path.

## What This Changes

The PR changes the TUI `/model` confirmation to use the gateway-returned resolved model ref, with tests for alias resolution and missing-resolved fallback.

## Best Possible Solution

Keep the PR open and repair the display normalization so aliases show the resolved canonical ref while already-qualified resolved models remain unchanged.

## Reproduction Assessment

Yes. The PR body supplies before/after local TUI screenshots for the original `/model llama3.2` confirmation mismatch, and the introduced qualified-ref regression is proven by the live TUI PTY failure on the changed path.

## Solution Assessment

No. Using `resolved` is the right source, but the PR should normalize or reuse a shared model-ref display helper instead of blindly prefixing `resolved.modelProvider` when `resolved.model` may already be provider-qualified.
