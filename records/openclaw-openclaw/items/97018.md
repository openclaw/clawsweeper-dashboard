## Summary

Keep open: this PR targets a real source-backed LM Studio embedding preload bug and is paired with https://github.com/openclaw/openclaw/issues/97016, but the current head is not merge-ready because it still ignores the lower `contextTokens` cap, the new tests do not exercise the production preload call, and real LM Studio behavior proof is only code/test-level.

## What This Changes

The branch looks up the configured LM Studio embedding model, passes its `contextWindow` as `requestedContextLength` to the model load helper, and adds a small test file for that lookup.

## Best Possible Solution

Resolve LM Studio embedding preload context with the same `contextTokens`-then-`contextWindow` rule used by inference preload, assert the actual factory-to-load-helper call, and require redacted LM Studio load-request proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main omits `requestedContextLength` in the LM Studio embedding preload and the callee falls back to the default load context; I did not reproduce the CUDA OOM live.

## Solution Assessment

No. The provider-local fix location is right, but the implementation should mirror `stream.ts` by preferring `contextTokens` before `contextWindow` and should test the actual preload call.
