## Summary

Keep open: current main does not have a sessions explain command, and the PR still has source-reproducible CLI resolution bugs plus missing required real behavior proof.

## What This Changes

The PR adds an `openclaw sessions explain <key>` subcommand with text/JSON output, optional `--tool` diagnostics, and sessions command tests.

## Best Possible Solution

Rebase on current main, implement explain through canonical normalized session target/load helpers using the inspected store entry, keep diagnostics lazy, document the CLI surface, and add redacted real CLI proof.

## Reproduction Assessment

Yes for the review findings: the PR head source shows parent `--json` is not forwarded, raw `--store` is loaded, parent `--agent` is not applied to unqualified keys, first-key matching diverges from current freshest-entry lookup, and reason chains do not mirror resolver precedence. I did not run the command in this read-only review.

## Solution Assessment

No. The feature direction is useful, but the current patch should be tightened around existing session resolution contracts instead of duplicating lookup and explanation logic that can drift.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep explain diagnostics behind a lazy path:** `src/commands/sessions.ts:1-8`
  - body: These static imports are used only by explain mode, but they load agent/tool-policy/runtime helpers for ordinary `openclaw sessions` listing. Current sessions code already uses a lazy loader for cold-path context lookup, so explain-only diagnostics should stay behind a lazy boundary.
  - confidence: 0.84
- **[P2] Honor the parent JSON flag:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The subcommand passes only `Boolean(opts.json)`, so `openclaw sessions --json explain <key>` falls back to text even though the parent command accepts `--json`. Merge the parent and child JSON flags like sibling sessions commands do.
  - confidence: 0.9
- **[P2] Normalize explicit store paths before loading:** `src/commands/sessions.ts:118`
  - body: Explain mode calls `loadSessionStore(opts.store)` directly, skipping the normal session store target normalization for values such as home-relative or templated store paths. Resolve the target first and load from the normalized store path.
  - confidence: 0.91
- **[P2] Apply parent --agent to unqualified explain keys:** `src/commands/sessions.ts:119-123`
  - body: The parent `--agent` option reaches `sessionsCommand`, but `resolveGatewaySessionStoreTarget` receives only `opts.explain`. `openclaw sessions --agent work explain main` can therefore inspect the default agent's session instead of the requested agent.
  - confidence: 0.88
- **[P2] Select the freshest matching session entry:** `src/commands/sessions.ts:126-127`
  - body: This uses the first key in `target.storeKeys`, while current canonical lookup compares `updatedAt` across canonical, legacy, and case-variant keys. If stale and fresh variants coexist, explain can report the wrong provider, model, and workspace.
  - confidence: 0.9
- **[P2] Use the inspected store for subagent capabilities:** `src/commands/sessions.ts:145`
  - body: After selecting an entry from a concrete store, capabilities are recomputed from `target.canonicalKey` with config-derived stores. Custom stores or legacy-key matches can therefore show subagent role/control from a different persisted entry.
  - confidence: 0.86
- **[P2] Resolve tools from the inspected session context:** `src/commands/sessions.ts:162-165`
  - body: The effective tool inventory receives `target.canonicalKey` without the selected store context, so subagent tool gating can be derived from a different config-derived entry. Use the matched key/store context or precomputed capabilities from the inspected entry.
  - confidence: 0.84
- **[P2] Report runtime model reasons only when runtime wins:** `src/commands/sessions.ts:217-223`
  - body: `resolveSessionModelRef` prefers explicit overrides before persisted runtime model fields, but this reason chain always adds `persisted-runtime-model-ref` whenever runtime fields exist. Sessions with both fields will claim runtime participated when it did not.
  - confidence: 0.87
- **[P2] Mark overrides active only when a model override participates:** `src/commands/sessions.ts:218-222`
  - body: `usesOverrides` and the reason chain treat a lone `providerOverride` as active, but the resolver ignores provider-only overrides without `modelOverride`. Legacy/manual entries can therefore report an override path while falling back to runtime/default selection.
  - confidence: 0.83
