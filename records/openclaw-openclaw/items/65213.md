Full review comments:

- **[P2] Honor the parent JSON flag:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The subcommand passes only `Boolean(opts.json)`, so `openclaw sessions --json explain <key>` falls back to text even though sibling sessions flows honor parent options. Merge the parent JSON option here so existing CLI flag placement keeps working.
  - confidence: 0.9
- **[P2] Normalize the store before loading explain data:** `src/commands/sessions.ts:118`
  - body: Explain mode calls `loadSessionStore(opts.store)` directly, skipping the normal `resolveStorePath` handling for home-relative and `{agentId}` store paths. Store values that work in sibling sessions flows can miss the intended file here.
  - confidence: 0.91
- **[P2] Apply parent agent selection to explain lookup:** `src/commands/sessions.ts:119-123`
  - body: The target resolver is called without `opts.agent`, so unqualified keys can resolve against the default agent rather than the parent `--agent` selection. In multi-agent setups this can explain the wrong session or report not found.
  - confidence: 0.88
- **[P2] Pick the freshest matching session entry:** `src/commands/sessions.ts:126-127`
  - body: This lookup takes the first existing candidate from `target.storeKeys`, but current gateway helpers choose the freshest match by `updatedAt` when canonical and legacy/case-variant keys coexist. Explain can therefore print stale provider/model/workspace data.
  - confidence: 0.89
- **[P2] Resolve diagnostics from the inspected store:** `src/commands/sessions.ts:145-165`
  - body: The command loads a concrete entry from one store, then recomputes subagent capabilities and effective tools from `target.canonicalKey` without passing the matched store context. Custom stores or legacy key matches can produce role/tool diagnostics for a different session than the one being explained.
  - confidence: 0.86
- **[P2] Report model reasons from the winning branch:** `src/commands/sessions.ts:217-224`
  - body: The reason flags are based on field presence, not on the branch `resolveSessionModelRef` actually selected. Sessions with explicit overrides plus runtime fields, or provider-only overrides, can get a misleading explanation of why the final model won.
  - confidence: 0.84
- **[P2] Keep explain-only diagnostics lazy:** `src/commands/sessions.ts:1-8`
  - body: These imports are used only by explain mode, but they load tool-policy/runtime helpers for ordinary `openclaw sessions` listing. Current main keeps cold-path context lookup lazy, so explain-only diagnostics should stay behind a lazy boundary during the rebase.
  - confidence: 0.8
