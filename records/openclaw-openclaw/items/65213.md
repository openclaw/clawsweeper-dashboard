Full review comments:

- **[P2] Keep explain-only diagnostics lazy:** `src/commands/sessions.ts:1-8`
  - body: These imports are used only by explain mode, but they load agent/tool-policy helpers for ordinary `openclaw sessions` listing. Current main keeps cold-path context lookup lazy, so explain-only diagnostics should stay behind a lazy boundary.
  - confidence: 0.84
- **[P2] Honor the parent JSON flag:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The subcommand passes only `Boolean(opts.json)`, so `openclaw sessions --json explain <key>` falls back to text even though sibling sessions flows merge parent and child flags. Include the parent JSON option here.
  - confidence: 0.9
- **[P2] Normalize explicit store paths before loading:** `src/commands/sessions.ts:118`
  - body: Explain mode calls `loadSessionStore(opts.store)` directly, skipping normal `resolveStorePath` handling for home-relative and `{agentId}` store paths that work in sibling sessions flows.
  - confidence: 0.91
- **[P2] Apply parent agent selection to unqualified keys:** `src/commands/sessions.ts:119-123`
  - body: The parent `--agent` option reaches `sessionsCommand`, but explain canonicalizes only the key. Unqualified keys like `main` can therefore inspect the default agent instead of the requested agent.
  - confidence: 0.88
- **[P2] Select the freshest matching entry:** `src/commands/sessions.ts:126-127`
  - body: This takes the first existing key from `target.storeKeys`, while canonical lookup compares `updatedAt` across canonical, legacy, and case-variant keys. If stale and fresh variants coexist, explain can report the wrong effective state.
  - confidence: 0.9
- **[P2] Read capabilities from the inspected store:** `src/commands/sessions.ts:145`
  - body: After selecting an entry from a concrete store, capabilities are recomputed from config-derived storage. Custom stores or legacy-key matches can show a role/control scope from a different entry than the one being explained.
  - confidence: 0.86
- **[P2] Resolve tools from the inspected session context:** `src/commands/sessions.ts:162-166`
  - body: The effective tool inventory receives only the canonical session key, so subagent tool gating can be derived from config-derived storage instead of the custom or legacy-key store entry that explain loaded.
  - confidence: 0.84
- **[P2] Report runtime reasons only when runtime wins:** `src/commands/sessions.ts:217-224`
  - body: `resolveSessionModelRef` prefers explicit overrides before persisted runtime model fields, but this reason chain always adds `persisted-runtime-model-ref` whenever runtime metadata exists, making mixed entries misleading.
  - confidence: 0.87
- **[P2] Mark overrides active only for real model overrides:** `src/commands/sessions.ts:218-223`
  - body: `usesOverrides` treats a lone `providerOverride` as active, but the resolver ignores provider-only overrides without `modelOverride`. Legacy or manual entries can claim an override path that did not participate.
  - confidence: 0.83
