## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep explain-only diagnostics lazy:** `src/commands/sessions.ts:1-8`
  - body: These static imports are used only by explain mode, but they load agent/tool-policy/runtime helpers for ordinary `openclaw sessions` listing. Current sessions code already uses lazy loading for cold-path context lookup, so explain-only diagnostics should stay behind a lazy boundary.
  - confidence: 0.84
- **[P2] Honor the parent JSON flag:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The subcommand passes only `Boolean(opts.json)`, so `openclaw sessions --json explain <key>` falls back to text even though sibling sessions subcommands merge parent and child JSON flags.
  - confidence: 0.9
- **[P2] Normalize explicit store paths before loading:** `src/commands/sessions.ts:118`
  - body: Explain mode calls `loadSessionStore(opts.store)` directly, skipping normal session-store target normalization for home-relative or `{agentId}` store paths that work in sibling sessions flows.
  - confidence: 0.91
- **[P2] Apply parent --agent to unqualified keys:** `src/commands/sessions.ts:119-123`
  - body: The parent `--agent` option reaches `sessionsCommand`, but the explain lookup canonicalizes only `opts.explain`, so `openclaw sessions --agent work explain main` can inspect the default agent instead of the requested one.
  - confidence: 0.88
- **[P2] Select the freshest matching entry:** `src/commands/sessions.ts:126-127`
  - body: This takes the first existing key from `target.storeKeys`, while the canonical lookup compares `updatedAt` across canonical, legacy, and case-variant keys. If stale and fresh variants coexist, explain can report the wrong effective state.
  - confidence: 0.9
- **[P2] Read subagent capabilities from the inspected store:** `src/commands/sessions.ts:145`
  - body: After selecting an entry from a concrete store, capabilities are recomputed from `target.canonicalKey` using config-derived stores, so custom stores or legacy-key matches can show a role/control scope from a different entry.
  - confidence: 0.86
- **[P2] Resolve tools from the inspected session context:** `src/commands/sessions.ts:162-165`
  - body: The effective tool inventory receives `target.canonicalKey` without the selected store context, so subagent tool gating can be derived from a different config-derived entry than the one being explained.
  - confidence: 0.84
- **[P2] Only report runtime reasons when runtime wins:** `src/commands/sessions.ts:223`
  - body: `resolveSessionModelRef` prefers explicit overrides before persisted runtime model fields, but this reason chain always adds `persisted-runtime-model-ref` whenever runtime metadata exists, making mixed entries misleading.
  - confidence: 0.87
- **[P2] Mark overrides active only for model overrides:** `src/commands/sessions.ts:218-223`
  - body: `usesOverrides` treats a lone `providerOverride` as active, but the resolver ignores provider-only overrides without `modelOverride`. Legacy or manual entries can therefore claim an override path that did not participate.
  - confidence: 0.83
