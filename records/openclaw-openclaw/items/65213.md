## What This Changes

The PR adds `openclaw sessions explain <key>` with text/JSON output, optional `--tool` diagnostics, effective model/workspace/tool/capability output, and sessions command tests.

## Best Possible Solution

Keep the PR open and rework it around canonical session target/load helpers, lazy explain-only dependencies, docs, and real CLI proof before merge.

## Reproduction Assessment

Yes. The source-level reproduction path is high-confidence: parent `--json`, raw `--store` with `~` or `{agentId}`, `--agent work explain main`, and canonical/legacy key collisions all follow changed lines that can report text, miss a store, or inspect the wrong entry.

## Solution Assessment

No. The feature direction is useful, but this implementation should be revised to use canonical session lookup semantics, preserve the sessions cold path, update the docs, and include real CLI proof.

Full review comments:

- **[P2] Keep explain-only imports off the sessions list path:** `src/commands/sessions.ts:1-8`
  - body: These new static imports are used only by explain mode, but they load agent/tool-policy/runtime helpers for ordinary `openclaw sessions` list usage. Current main moved sessions command work toward lazy imports for cold-path performance; keep the explain diagnostics behind a lazy/runtime boundary so the list command does not pay this cost.
  - confidence: 0.84
- **[P2] Honor the parent JSON flag:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The subcommand only passes `Boolean(opts.json)`, so `openclaw sessions --json explain <key>` falls back to text even though the parent command accepts `--json` and sibling subcommands merge parent and child JSON flags. Include `json?: boolean` in `parentOpts` and merge it here.
  - confidence: 0.9
- **[P2] Load explicit stores after path normalization:** `src/commands/sessions.ts:118`
  - body: Explain mode calls `loadSessionStore(opts.store)` before session-store target resolution can normalize `~` and `{agentId}` paths. A `--store` value that works for list or cleanup can miss here; load from the resolved target path instead.
  - confidence: 0.91
- **[P2] Apply parent --agent to unqualified explain keys:** `src/commands/sessions.ts:119-123`
  - body: The parent `--agent` option reaches `sessionsCommand`, but the explain branch resolves `opts.explain` without using it. `openclaw sessions --agent work explain main` can canonicalize against the default agent and inspect or miss the wrong session.
  - confidence: 0.88
- **[P2] Choose the freshest matching session entry:** `src/commands/sessions.ts:126-127`
  - body: This uses the first key in `target.storeKeys`, while the canonical gateway helper chooses the freshest entry across canonical, legacy, and case-variant keys. If stale and fresh variants coexist, explain can report the wrong provider, model, and workspace state.
  - confidence: 0.9
- **[P2] Resolve capabilities from the inspected store:** `src/commands/sessions.ts:145`
  - body: After selecting an entry from a concrete store, capabilities are recomputed from `target.canonicalKey` with config only. Custom stores or legacy-key matches can therefore show subagent role/control from a different persisted entry; pass the selected store and matched key through.
  - confidence: 0.86
- **[P2] Resolve tools from the inspected session:** `src/commands/sessions.ts:162-165`
  - body: The effective tool inventory receives the canonical key without the selected store context, so subagent tool gating can be derived from config-derived stores rather than the entry being explained. That makes tool groups and `--tool` output inconsistent for explicit-store or legacy-key sessions.
  - confidence: 0.84
- **[P2] Only report runtime model reasons when they win:** `src/commands/sessions.ts:217-223`
  - body: The reason chain always includes persisted runtime metadata whenever runtime fields exist, but `resolveSessionModelRef` prefers explicit model overrides first. Sessions with both fields will claim runtime participated in the final model even when it did not.
  - confidence: 0.87
- **[P2] Report override usage only when a model override exists:** `src/commands/sessions.ts:218-222`
  - body: `usesOverrides` and the reason chain mark a lone `providerOverride` as active, but the persisted override resolver returns null without an override model. Legacy or manual entries with only a provider override will show a misleading override reason while falling back to runtime/default selection.
  - confidence: 0.83
- **[P3] Document the sessions explain command:** `src/cli/program/register.status-health-sessions.ts:168-172`
  - body: This adds a user-facing CLI command, but the sessions CLI docs still cover list, export, and cleanup only. Add usage, options, and the JSON shape so published docs match the binary.
  - confidence: 0.82
