## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor parent JSON for explain:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The action only forwards `opts.json`, so `openclaw sessions --json explain <key>` will still emit text even though sibling session subcommands merge parent JSON options. This can break scripts that place shared flags before the subcommand.
  - confidence: 0.93
- **[P2] Load the normalized session store:** `src/commands/sessions.ts:118-125`
  - body: Explain mode loads `opts.store` directly and reports the raw value as the store path. That bypasses the existing `resolveStorePath` behavior used by session target resolution, so values with `~`, relative paths, or `{agentId}` can read or report the wrong store compared with the rest of `openclaw sessions`.
  - confidence: 0.92
- **[P2] Scope unqualified keys to the selected agent:** `src/commands/sessions.ts:119-123`
  - body: `resolveGatewaySessionStoreTarget` is called without the parent `--agent` context, so an unqualified key such as `main` is resolved through the configured default agent rather than the selected agent. In multi-agent setups this can explain the wrong stored session or report not found.
  - confidence: 0.9
- **[P2] Pick the freshest matched session entry:** `src/commands/sessions.ts:126-127`
  - body: This takes the first existing key from `target.storeKeys`, but that list is lookup-order, not freshness-order. When canonical and legacy or case-variant entries coexist, explain can choose stale metadata while other session paths prefer the current stored session state.
  - confidence: 0.86
- **[P2] Resolve tool state from the inspected session store:** `src/commands/sessions.ts:145-165`
  - body: The command loads one `entry`, but then recomputes subagent capabilities and effective tools from `target.canonicalKey` plus config-derived stores. With `--store <custom>` or a legacy `matchedKey`, the reported role, control scope, and tool availability can describe a different session than the one being explained.
  - confidence: 0.91
- **[P2] Report only model sources that affect resolution:** `src/commands/sessions.ts:217-223`
  - body: The reason flags append runtime and provider-override sources whenever the fields exist, but the resolver only treats an override as effective when a model override is present and it prefers explicit overrides before runtime metadata. The diagnostic chain can therefore claim a source affected the final model when it did not.
  - confidence: 0.89
