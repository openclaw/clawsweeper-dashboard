## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor the inherited JSON flag:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: `sessions explain` only forwards the child `opts.json`, so `openclaw sessions --json explain <key>` will still produce text even though sibling session commands merge parent flags. This breaks CLI consistency and scripts that put shared flags before the subcommand.
  - confidence: 0.91
- **[P2] Load the normalized store path:** `src/commands/sessions.ts:118`
  - body: Explain mode reads `opts.store` directly before target normalization, so store values that require normal resolution can miss sessions that list/cleanup would find. Resolve the target first and load the resolved `storePath`.
  - confidence: 0.9
- **[P2] Pass the selected agent into key resolution:** `src/commands/sessions.ts:119-123`
  - body: The target resolver supports `agentId`, but this call omits `opts.agent`; unqualified keys such as `main` can therefore resolve against the default agent instead of the requested one in multi-agent setups.
  - confidence: 0.91
- **[P2] Choose the freshest matching session entry:** `src/commands/sessions.ts:126`
  - body: Taking the first candidate from `target.storeKeys` can select a stale canonical or legacy entry because that list is not newest-first. Use the same freshest matching semantics as the session lookup path so explain reports the active stored state.
  - confidence: 0.85
- **[P2] Resolve tools from the inspected store context:** `src/commands/sessions.ts:145-166`
  - body: The command loads a concrete entry but recomputes subagent capabilities and tools from `target.canonicalKey` and config-derived stores. For custom stores or legacy key matches, the tool/profile diagnostics can describe a different session than the one being explained.
  - confidence: 0.87
- **[P2] Report only model sources that affect resolution:** `src/commands/sessions.ts:217-224`
  - body: The reason flags add runtime and provider override causes whenever fields are present, but the resolver requires an actual model override for explicit overrides and prefers overrides before runtime metadata. The explanation can therefore claim a source participated when it did not.
  - confidence: 0.88
