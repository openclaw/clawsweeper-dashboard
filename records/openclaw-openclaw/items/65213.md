## Summary

Keep open. Current main still does not provide a `sessions explain` command, so the PR is not obsolete, but the proposed branch has multiple correctness issues in option propagation, store/key resolution, and diagnostic reason reporting that should block merge until revised.

## What This Changes

The PR adds `openclaw sessions explain <key>` with text/JSON output, optional `--tool`, effective model/workspace/tool diagnostics, and sessions command tests.

## Best Possible Solution

Revise the PR so `sessions explain` is built around the canonical session-store/model/tool helpers: normalize explicit stores, honor parent flags and agent scope, select the freshest matched entry, pass the inspected store/key into capability and tool resolution, report reason chains that match actual resolver precedence, and add docs/changelog coverage.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: `openclaw sessions --json explain <key>`, `openclaw sessions --store '~/.openclaw/agents/{agentId}/sessions/sessions.json' explain <key>`, `openclaw sessions --agent work explain main`, and stores with canonical/legacy key collisions follow code paths that drop flags or inspect/report the wrong entry.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best implementation as-is because it bypasses or only partially reuses the canonical lookup and resolution semantics already present on current main.

Full review comments:

- **[P2] Honor the parent JSON flag:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The subcommand only passes `Boolean(opts.json)`, so `openclaw sessions --json explain <key>` falls back to text even though the parent command accepts `--json` and sibling subcommands merge parent and child JSON flags. Include `json?: boolean` in `parentOpts` and merge it here.
  - confidence: 0.88
- **[P2] Resolve explicit stores before loading them:** `src/commands/sessions.ts:118`
  - body: Explain mode calls `loadSessionStore(opts.store)` before the session-store target resolver can normalize `~`, `{agentId}`, and relative paths. A `--store` value that works for list/cleanup can miss here; load from the resolved target path instead.
  - confidence: 0.91
- **[P2] Choose the freshest matching store entry:** `src/commands/sessions.ts:126-127`
  - body: This uses the first key in `target.storeKeys`, but the canonical gateway helper chooses the freshest entry across canonical, legacy, and case-variant keys. If stale and fresh variants coexist, explain can report the wrong provider/model/workspace state.
  - confidence: 0.9
- **[P2] Apply --agent to unqualified explain keys:** `src/commands/sessions.ts:119-123`
  - body: The parent `--agent` option is passed into `sessionsCommand`, but the explain branch resolves `opts.explain` without using it. `openclaw sessions --agent work explain main` can canonicalize against the default agent and inspect or miss the wrong session.
  - confidence: 0.86
- **[P2] Resolve capabilities from the inspected store:** `src/commands/sessions.ts:145`
  - body: After selecting an entry from a concrete store, capabilities are recomputed from `target.canonicalKey` with config only. Custom stores or legacy-key matches can therefore show subagent role/control from a different persisted entry; pass the selected store and matched key through.
  - confidence: 0.86
- **[P2] Resolve tool inventory from the inspected session:** `src/commands/sessions.ts:165`
  - body: The effective tool inventory receives the canonical key without the selected store context, so subagent tool gating can be derived from config-derived stores rather than the entry being explained. That makes tool groups and `--tool` output inconsistent for explicit-store or legacy-key sessions.
  - confidence: 0.84
- **[P2] Only mark runtime model reasons when they win:** `src/commands/sessions.ts:217-224`
  - body: The reason chain always includes persisted runtime metadata whenever runtime fields exist, but `resolveSessionModelRef` prefers explicit model overrides first. Sessions with both fields will claim runtime participated in the final model even when it did not.
  - confidence: 0.87
- **[P2] Do not treat provider-only overrides as active:** `src/commands/sessions.ts:218-223`
  - body: `usesOverrides` and the reason chain mark a lone `providerOverride` as active, but the persisted override resolver returns null without an override model. Legacy or manual entries with only a provider override will show a misleading override reason while falling back to runtime/default selection.
  - confidence: 0.83
- **[P3] Document the new sessions explain command:** `src/cli/program/register.status-health-sessions.ts:169-174`
  - body: This adds a user-facing CLI command but does not update the sessions CLI docs. Add the `explain` syntax, JSON/text behavior, `--tool`, and scope interactions to `docs/cli/sessions.md` so the public CLI reference stays complete.
  - confidence: 0.82
