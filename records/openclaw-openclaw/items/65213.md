## Summary

Keep this PR open: current main still lacks a `sessions explain` command, but the branch is not merge-ready because the implementation can misreport session state and the external PR still lacks after-fix real CLI proof.

## What This Changes

The PR adds an `openclaw sessions explain <key>` subcommand with text/JSON output, optional `--tool` diagnostics, and sessions command tests.

## Best Possible Solution

Rebase on current main, implement explain through canonical normalized session target/load helpers using the inspected store entry, keep explain-only diagnostics lazy, document the CLI surface, and add redacted terminal proof for a real stored session.

## Reproduction Assessment

Yes for the review findings: the PR head source shows parent `--json` is not forwarded, raw `--store` is loaded, parent `--agent` is ignored for unqualified keys, and first-key matching diverges from current canonical freshest-entry lookup. I did not run the command because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but the current patch duplicates session resolution logic instead of reusing the canonical target/load semantics, and it needs lazy explain-only imports, docs, and real CLI proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep explain diagnostics behind a lazy path:** `src/commands/sessions.ts:1-8`
  - body: These static imports are used only by explain mode, but they load agent/tool-policy/runtime helpers for ordinary `openclaw sessions` listing. Current sessions code has moved toward lazy imports for cold-path performance, so keep explain diagnostics behind a lazy boundary.
  - confidence: 0.84
- **[P2] Honor the parent JSON flag:** `src/cli/program/register.status-health-sessions.ts:184`
  - body: The subcommand passes only `Boolean(opts.json)`, so `openclaw sessions --json explain <key>` falls back to text even though the parent command accepts `--json` and sibling subcommands merge parent and child JSON flags.
  - confidence: 0.9
- **[P2] Normalize explicit store paths before loading:** `src/commands/sessions.ts:118`
  - body: Explain mode calls `loadSessionStore(opts.store)` directly, skipping `resolveStorePath` normalization for `~` and `{agentId}` values that work in the normal sessions flows. Load from the resolved target path instead of the raw option string.
  - confidence: 0.91
- **[P2] Apply parent --agent to unqualified keys:** `src/commands/sessions.ts:119-123`
  - body: The parent `--agent` option reaches `sessionsCommand`, but explain resolves `opts.explain` without using it. `openclaw sessions --agent work explain main` can inspect the default agent's session instead of the requested agent.
  - confidence: 0.88
- **[P2] Select the freshest matching session entry:** `src/commands/sessions.ts:126-127`
  - body: This uses the first key in `target.storeKeys`, while current canonical lookup chooses the freshest entry across canonical, legacy, and case-variant keys. If stale and fresh variants coexist, explain can report the wrong provider, model, and workspace.
  - confidence: 0.9
- **[P2] Use the inspected store for subagent capabilities:** `src/commands/sessions.ts:145`
  - body: After selecting an entry from a concrete store, capabilities are recomputed from `target.canonicalKey` with config only. Custom stores or legacy-key matches can therefore show subagent role/control from a different persisted entry.
  - confidence: 0.86
- **[P2] Resolve tools from the inspected session context:** `src/commands/sessions.ts:162-165`
  - body: The effective tool inventory receives the canonical key without the selected store context, so subagent tool gating can be derived from config-derived stores rather than the entry being explained. That makes tool groups and `--tool` output inconsistent for explicit-store or legacy-key sessions.
  - confidence: 0.84
- **[P2] Only report runtime model reasons when runtime wins:** `src/commands/sessions.ts:217-223`
  - body: `resolveSessionModelRef` prefers explicit overrides before persisted runtime model fields, but this reason chain always includes `persisted-runtime-model-ref` whenever runtime fields exist. Sessions with both fields will claim runtime participated even when it did not.
  - confidence: 0.87
- **[P2] Only mark overrides active when a model override participates:** `src/commands/sessions.ts:218-222`
  - body: `usesOverrides` and the reason chain treat a lone `providerOverride` as active, but the persisted override resolver ignores provider-only overrides without `modelOverride`. That makes legacy/manual entries report an override path while falling back to runtime/default selection.
  - confidence: 0.83
- **[P3] Document the new sessions explain command:** `src/cli/program/register.status-health-sessions.ts:168-172`
  - body: This adds a user-facing CLI command, but the sessions CLI docs still cover list, export-trajectory, and cleanup only. Add usage, options, and the JSON shape so the published docs match the binary.
  - confidence: 0.82
