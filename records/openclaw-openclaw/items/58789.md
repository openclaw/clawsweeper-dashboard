## Summary

Keep open. Current main still documents Discord bot-loop risk and has bot-message gating, replay dedupe, and `allowBots="mentions"`, but it does not implement the requested bot-pair sliding-window/cooldown guard; the related RFC, general safeguard issue, and Telegram issue are adjacent open work rather than superseding fixes.

## What This Changes

Add Discord bot-to-bot loop detection/rate limiting for `allowBots=true`, with per-bot-pair window/cooldown behavior and config.

## Best Possible Solution

Add a Discord-owned deterministic guard before agent dispatch: track bot sender, mentioned/target bot, account, and channel in a small sliding window with cooldown, expose the documented config/help/schema surface, and keep semantic/progress scoring out of the first fix.

## Reproduction Assessment

Yes. A focused preflight or handler test can configure Discord with `allowBots=true`, send alternating bot-authored messages that mention the receiving bot, and show that current main accepts them because no per-pair window or cooldown state is consulted.

## Solution Assessment

Yes for the narrow safety shape. A deterministic Discord inbound guard is the smallest maintainable first step; the broader syntactic/semantic/progress loop scoring suggested in comments should remain separate unless maintainers choose that product direction.
