## Summary

Keep open. Current main still has Discord bot-message safeguards, mention gating, replay dedupe, and docs warnings, but it does not implement the requested bot-pair sliding-window/cooldown guard for `allowBots=true`; the linked teleconference RFC treats this as a dependency rather than a superseding implementation, and the production follow-up corroborates the runaway-loop impact.

## What This Changes

Request a Discord gateway-level bot-to-bot loop detector for `allowBots=true`, with per sender/receiver/channel rate limits and cooldown suppression before agent dispatch.

## Best Possible Solution

Add a Discord-owned `botLoopProtection` policy and deterministic bot-pair/window/cooldown guard in the inbound path before an agent turn is dispatched, with docs, changelog, schema/help/UI updates, and focused tests for `allowBots=true` plus mention-gated bot messages. Keep embedding or progress-scoring detection as later product work unless maintainers explicitly choose that broader design.

## Reproduction Assessment

Yes. A focused preflight/handler test can configure Discord with `allowBots=true`, send alternating bot-authored messages that mention the gateway bot, and observe that current main accepts them because no pair/window/cooldown state is consulted.

## Solution Assessment

Yes for the narrow requested shape. A deterministic Discord inbound guard keyed by sender bot, target/mentioned bot, account, and channel is the most maintainable first step; semantic similarity or progress scoring should not be required for the initial fix.
