## Summary

Keep open: current main still retries the Telegram `getFile` step but not the subsequent file download, and the related retry PR is closed unmerged.

## What This Changes

Telegram inbound media downloads should retry transient network/proxy fetch failures before returning the user-facing media download error.

## Best Possible Solution

Add bounded retry/backoff around the Telegram file download for transient `fetch_failed` network errors, preserving proxy/SSRF policy and permanent-error behavior, with focused regression tests.

## Reproduction Assessment

Yes, source-level: make `getFile` succeed and `fetchRemoteMedia` reject, and current `resolveMedia` rejects after one download failure. The existing retry test already encodes that current behavior, though I did not run a live proxy test.

## Solution Assessment

Yes for the retry-only direction: the narrow maintainable fix is to retry transient download failures after `getFile` succeeds. Automatic direct fallback would be a policy change and should not be part of this bug fix.

## Repair Work Prompt

Implement a narrow fix for https://github.com/openclaw/openclaw/issues/45799: in Telegram inbound media resolution, add bounded retry/backoff around the file download step after `getFile` succeeds. Use the existing `retryAsync` seam from `delivery.resolve-media.runtime.ts` or plugin runtime, retry only transient fetch/media failures such as `MediaFetchError` code `fetch_failed` or equivalent network errors, and keep `http_error`, `max_bytes`, file-too-big, and local-path policy failures non-retryable. Do not add direct-connect fallback or weaken SSRF/proxy policy. Add/update focused tests in `extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts` for transient download retry success, exhausted transient failure, and non-retryable failures. Use the closed PR #46013 only as reference, not as a blind cherry-pick. Add a single-line `CHANGELOG.md` fix entry with human attribution if known.
