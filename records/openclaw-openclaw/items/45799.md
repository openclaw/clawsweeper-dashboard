## Summary

Keep open: current main still retries Telegram `getFile()` but not the subsequent media file download, and the focused regression-shaped test still expects a `fetchRemoteMedia` failure to propagate without retry. Related Telegram proxy/SSRF fixes exist, but they do not implement the requested same-download retry/backoff path.

## What This Changes

Request bounded retry/backoff for Telegram inbound media downloads when a configured proxy or transient network failure interrupts the file fetch after `getFile()` succeeds.

## Best Possible Solution

Add a bounded Telegram media download retry for transient failures while preserving caller transport, proxy dispatch, SSRF checks, size limits, and permanent-failure behavior.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main has a focused mocked path where `fetchRemoteMedia` rejects once and `resolveMedia` propagates the failure without retrying. I did not run a live proxy outage test in this read-only review.

## Solution Assessment

Yes, the narrow maintainable solution is a same-policy bounded retry around transient Telegram media download failures. Automatic direct fallback is not the best default because it could bypass an operator-configured proxy policy.
