## Summary

Valid bug; current main still retries the Telegram Bot API `getFile()` step but not the subsequent media file download, and the linked retry PR was closed unmerged. The safe scope is a narrow Telegram media-download retry, not automatic direct/proxy bypass.

## What This Changes

The issue asks for Telegram inbound media downloads to retry transient proxy/network failures instead of failing on the first failed fetch.

## Best Possible Solution

Add a Telegram-local retry around transient `fetchRemoteMedia` download failures with bounded exponential backoff, preserve SSRF/proxy policy, add focused regression tests, and include a user-facing changelog fix entry.

## Reproduction Assessment

Yes, source-reproducible. Current main has a test proving `fetchRemoteMedia` download errors reject immediately while `getFile` is retried, and the explicit-proxy transport path can have only one download dispatcher attempt.

## Solution Assessment

Yes. The narrowest maintainable fix is to retry transient download fetch failures in the Telegram media resolver while avoiding automatic direct fallback that could bypass an operator's proxy policy.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/45799 by adding bounded retry/backoff for transient Telegram inbound media download failures after `getFile()` succeeds. Work in the Telegram plugin only unless a shared helper is already the established local pattern. The likely insertion point is `downloadAndSaveTelegramFile` / the `fetchRemoteMedia` call in `extensions/telegram/src/bot/delivery.resolve-media.ts`. Retry only transient network/download failures such as `MediaFetchError` with `code === "fetch_failed"` or equivalent fetch envelopes; do not retry HTTP errors, `max_bytes`, trusted-local-file read failures, absolute-path policy failures, or SSRF guard/policy failures. Do not add automatic direct/no-proxy fallback because a configured proxy may be an operator policy. Add focused tests in `extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts` proving transient download failure retries and succeeds, persistent transient failure exhausts attempts, and permanent HTTP/max-bytes/policy failures do not loop. Add a single-line `CHANGELOG.md` fix entry because this is user-facing. Use #46013 only as context/source work; do not broaden into the related proxy-propagation reports.
