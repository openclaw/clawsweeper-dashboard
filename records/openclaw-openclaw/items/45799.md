## Summary

Keep open: current main still retries the Telegram `getFile()` metadata step but not the subsequent Bot API media file download, and the existing retry test still locks in that single-shot download behavior.

## What This Changes

The issue requests bounded retry/backoff for Telegram inbound media file downloads when a configured proxy temporarily fails.

## Best Possible Solution

Add a bounded Telegram plugin retry around only the remote Bot API media download stage, retrying transient fetch/transport failures while preserving the configured transport, SSRF guard, and permanent-failure behavior.

## Reproduction Assessment

Yes. The current unit test mocks a successful `getFile()` followed by a failed `fetchRemoteMedia()` and asserts that `resolveMedia()` rejects without retrying, which directly exercises the missing download-stage retry path.

## Solution Assessment

Yes. A small retry wrapper around the remote `fetchRemoteMedia()` download call is the narrowest maintainable fix; direct fallback is not the right default because it can bypass configured proxy policy.
