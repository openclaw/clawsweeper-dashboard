## Summary

Close: current main now retries Telegram media file downloads through the shared SSRF-guarded media fetch retry path while preserving proxy transport, policy blocks, 4xx, abort, and maxBytes fail-fast behavior.

## What This Changes

The issue requests bounded retry/backoff for Telegram inbound media downloads when a configured HTTP proxy drops during the file fetch step.

## Best Possible Solution

Keep the current same-policy retry implementation on main and avoid automatic direct/no-proxy fallback, which could bypass an operator's configured network policy.

## Reproduction Assessment

No current-main reproduction remains: Telegram media downloads now pass a three-attempt retry policy into the shared guarded media save path. I did not run a live proxy outage test, but the source path that used to be one-shot is now retry-enabled.

## Solution Assessment

Yes. The current implementation is the maintainable fix because it retries transient same-policy media fetch/read failures while preserving SSRF checks, proxy dispatch, size limits, abort handling, and permanent HTTP failure behavior.
