## Summary

Keep open: the adjacent Telegram proxy/SSRF propagation bugs have shipped fixes, but the specific same-download retry/backoff requested here is still missing on current main. The current Telegram media test still asserts that `fetchRemoteMedia` failures are not caught by `resolveMedia`, so a transient proxy drop can still fail the download path without a bounded retry.

## What This Changes

The issue requests bounded retry/backoff for transient Telegram inbound media download failures when a configured proxy drops or resets during the file fetch.

## Best Possible Solution

Add a bounded Telegram media download retry around the file fetch for transient failures, while keeping proxy dispatch, SSRF checks, size limits, and permanent HTTP/policy failures non-retryable.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main has a focused mocked path where `fetchRemoteMedia` rejects once and `resolveMedia` propagates that failure without retrying. I did not run a live proxy disruption test in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to retry transient failures in the Telegram download step without bypassing the configured proxy. Automatic direct fallback would be a worse fit because it could violate operator network policy.
