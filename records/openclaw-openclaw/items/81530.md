## Summary

Keep open: current main still has a source-reproducible Telegram forum-topic ingress lane mismatch, and the latest issue discussion points to the same current code path. This is a broken documented Telegram topic behavior, not a new feature request.

## What This Changes

Telegram supergroup forum-topic inbound messages should create or update the topic-scoped session and reply in that same topic instead of showing typing with no response.

## Best Possible Solution

Align Telegram sequential/spool lane classification with the handler’s resolved forum metadata or a `message_thread_id`-aware forum fallback, then add regression coverage and live Telegram topic proof.

## Reproduction Assessment

No for the full live symptom: I did not run a real Telegram supergroup-topic probe. Yes at source level: a supergroup update with `message_thread_id` but omitted `chat.is_forum` and `is_topic_message` is keyed as `telegram:<chatId>` before the handler later resolves it as a forum topic.

## Solution Assessment

Yes for a narrow sequential-key/forum metadata alignment. The broader `allowed_updates` idea is not the best fix because current polling already requests message updates and the mismatch is in topic lane classification before handler processing.
