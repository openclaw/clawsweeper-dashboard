## Summary

Keep open: current main still has a plausible Telegram forum-topic ingress gap in the pre-handler sequential/spool lane key when a supergroup update carries `message_thread_id` but lacks both `chat.is_forum` and `is_topic_message`. The full live symptom is not reproduced on current main, but the source path is concrete enough for a focused fix/proof pass rather than cleanup closure.

## What This Changes

The issue reports Telegram supergroup forum-topic inbound messages showing typing but not creating or updating the expected topic session or reply.

## Best Possible Solution

Add a narrow Telegram plugin fix so pre-handler lane selection can safely recognize forum-topic updates when Telegram omits the usual forum flags, while preserving regular supergroup reply-thread behavior and adding regression plus live or redacted spool proof.

## Reproduction Assessment

No, not for the full live symptom on current main. Source inspection does show a reproducible keying gap for a delivered supergroup update with `message_thread_id` but without `chat.is_forum` or `is_topic_message`, which matches the latest diagnostic comment but still needs live raw-update proof.

## Solution Assessment

Unclear for the exact implementation, but closing as already implemented would be premature. The safest direction is a focused Telegram lane-key/forum-flag repair with tests that prove the missing-field case while keeping ordinary supergroup reply threads chat-wide.
