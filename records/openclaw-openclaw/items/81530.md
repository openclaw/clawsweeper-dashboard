## Summary

Keep open: current main still has a plausible Telegram forum-topic ingress lane mismatch. The sequential key can route a `message_thread_id` supergroup update to the chat-wide lane when forum metadata is omitted, while the inbound handler later resolves forum metadata for topic sessions; I found no open or merged PR that supersedes this issue.

## What This Changes

The issue reports Telegram supergroup forum-topic inbound messages showing typing but not creating/updating the expected topic session or producing a reply.

## Best Possible Solution

Align Telegram sequential/spool lane classification with the handler’s resolved forum metadata, add regression coverage for metadata-omitted forum-topic updates, and prove the fix with a redacted live Telegram topic run before landing.

## Reproduction Assessment

No for the complete live Telegram symptom; I did not run a real supergroup-topic probe. Source inspection does reproduce the key mismatch for a supergroup update that has `message_thread_id` but omits `chat.is_forum` and `is_topic_message`.

## Solution Assessment

Yes for a narrow sequential/spool-lane alignment; no for the broad `allowed_updates` proposal. Current main already requests `message` updates, so the maintainable fix is to share or match the inbound handler’s resolved forum metadata in the key builder.
