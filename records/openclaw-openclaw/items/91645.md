## Summary

Keep open: current main and v2026.6.1 still have a Gemma 4-specific OpenAI-completions sanitizer path that can strip real `reasoning_content` before the request is sent, while the issue describes a narrower in-turn tool replay case that is not already solved.

## What This Changes

The issue requests preserving real in-turn `reasoning_content` for Gemma 4/vLLM-style OpenAI-compatible reasoning models during multi-tool replay while still avoiding completed-turn reasoning replay.

## Best Possible Solution

Keep completed-turn Gemma reasoning stripped, but preserve real `reasoning_content` only for the active in-turn assistant tool-call replay path when the provider emitted a replayable reasoning field.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live vLLM/Gemma run was performed, but source inspection gives a medium-confidence reproduction path: preserved in-turn thinking can be converted to `reasoning_content` and then stripped by the Gemma 4 transport sanitizer before the request is sent.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the exact policy boundary; the safe direction is a narrow same-turn tool-call replay exception, not a blanket removal of the Gemma 4 stripping rule.
