## Summary

Keep open. Current main and the latest release still treat accepted Telegram reaction updates as queued system events without a wake path, while the remaining fix requires a maintainer choice between the open shared opt-in policy and narrower Telegram-only wake PRs.

## What This Changes

The issue requests that inbound Telegram reaction context events wake an agent turn instead of remaining passive until another message or heartbeat occurs.

## Best Possible Solution

Pick one canonical default-safe reaction wake contract, land it with schema/docs/tests and redacted live Telegram proof, then close this issue as implemented.

## Reproduction Assessment

Yes, at source level. An authorized Telegram message_reaction with an added emoji reaches enqueueSystemEvent on current main and then returns without a heartbeat or agent-turn path, matching the live reports.

## Solution Assessment

Unclear until maintainers choose the API. The safest direction is an opt-in, queue-safe wake policy after existing Telegram authorization and notification filters pass, not an unconditional wake for every reaction notification.
