## Summary

Keep open. Current main still treats Telegram `message_reaction` updates as authorized context/system-event enqueue only; it does not call an agent dispatch or heartbeat wake path, and the related implementation PR #47687 is still open rather than shipped.

## What This Changes

Telegram inbound reaction events should trigger an agent turn when configured, instead of only being recorded as later prompt context.

## Best Possible Solution

Keep this issue open until a Telegram-owned fix lands that preserves passive reaction notifications by default, adds a documented opt-in wake policy for meaningful reactions, and regression-tests both wake-on and wake-off behavior for authorized DM and group reaction events.

## Reproduction Assessment

Yes. Multiple live repro comments confirm the behavior through v2026.4.29, and the current-main static path is high-confidence: `message_reaction` reaches the Telegram handler, queues a system event, and never calls a wake or dispatch path.

## Solution Assessment

No, current main is not the best solution because it records reactions for later context without waking the agent. The narrow maintainable fix is to call the existing heartbeat wake primitive after authorized, configured reaction enqueue, with defaults that avoid waking on every generic reaction.
