## Summary

Keep open. Current main still receives and enqueues Telegram reaction updates without starting an agent turn, and the relevant implementation PRs remain open rather than merged.

## What This Changes

The issue asks Telegram inbound reaction events to trigger an agent turn when reaction handling is enabled, instead of only appearing later as passive context.

## Best Possible Solution

Review and land or supersede the narrow Telegram wake fix in https://github.com/openclaw/openclaw/pull/75899, while keeping broader reaction semantics in https://github.com/openclaw/openclaw/pull/47687 or https://github.com/openclaw/openclaw/issues/47677.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path. An authorized Telegram message_reaction with an added emoji reaches enqueueSystemEvent and returns without any agent-turn wake, matching multiple live reports.

## Solution Assessment

No, current main is not the best solution because it leaves reaction events passive. The maintainable fix is a Telegram-owned wake path with regression coverage, while semantic emoji mapping should stay in the broader reaction-semantics thread.
