## Label Justifications

- `P2`: The issue blocks Telegram reaction-only workflows but is limited to reaction notification wake behavior rather than a core runtime outage.
- `impact:message-loss`: Incoming Telegram reaction updates are consumed and queued but not delivered into an agent turn until another wake occurs, suppressing reaction-only inputs.

## Summary

Keep open. Current main and latest release still enqueue accepted Telegram reaction updates as passive system events without a wake/agent-turn path, and the remaining fix needs maintainer selection between overlapping open wake-policy PRs.

## What This Changes

The issue asks for inbound Telegram reaction events to trigger an agent turn instead of remaining queued until another message or heartbeat wakes the session.

## Best Possible Solution

Adopt or supersede the open wake-policy PRs with one default-safe policy that wakes only accepted opt-in reaction notifications after successful enqueue, with schema/docs updates, focused regression tests, and live Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an authorized Telegram message_reaction with an added emoji reaches enqueueSystemEvent and then returns without a wake call, matching the live reports.

## Solution Assessment

Unclear until maintainers choose the contract. The requested behavior is coherent, but current main does not expose the reported triggerAgentTurn setting, so a default-off, maintainer-approved wake policy is safer than unconditional reaction wakes.
