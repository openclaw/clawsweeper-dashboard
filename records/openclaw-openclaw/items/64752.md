## Summary

Keep open. Current main still records Telegram `message_reaction` updates as system events without starting an agent turn, and the two related implementation PRs are still open rather than merged.

## What This Changes

The issue asks for inbound Telegram emoji reactions to wake the agent when reaction-triggered turns are enabled, instead of remaining passive context until another turn starts.

## Best Possible Solution

Land a Telegram-owned wake fix with regression coverage for authorized reaction enqueue plus immediate agent processing, and keep broader reaction semantics tracked separately if maintainers prefer #47687’s product direction.

## Reproduction Assessment

Yes. Multiple live reports describe the same steps, and current-main source inspection shows the reaction update reaches the handler and queues a system event without any wake or message-processing call.

## Solution Assessment

No, current main is not the best solution because it leaves reaction events passive. The narrow maintainable repair is an authorized Telegram reaction wake path with tests, while any semantic emoji mapping should remain a separate product decision.
