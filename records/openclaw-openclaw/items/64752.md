## Label Justifications

- `P2`: The issue blocks Telegram reaction-only workflows but is scoped to reaction notifications rather than a crash, security incident, or whole-channel outage.
- `impact:message-loss`: Incoming Telegram reaction updates are consumed and queued but not delivered into an agent turn until another wake occurs, so reaction-only inputs are effectively suppressed.

## Summary

Keep open. Current main and the latest v2026.5.18 release still enqueue Telegram `message_reaction` updates as passive system events without requesting a wake, while the active fixes are still open PRs that need maintainer choice and proof.

## What This Changes

Telegram inbound reaction updates should wake the routed agent turn when configured instead of only adding passive session context.

## Best Possible Solution

Land a maintainer-approved notification wake policy, likely through https://github.com/openclaw/openclaw/pull/83611 or a superseding narrow PR, with tests and real Telegram proof that accepted reactions can opt in to waking after successful enqueue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an authorized Telegram `message_reaction` with an added emoji reaches `enqueueSystemEvent` and then returns without any wake call, and multiple live issue comments report the same behavior.

## Solution Assessment

Unclear. The narrow synthetic-session approach fixes the symptom, but the shared `notificationWake.reactions` policy is likely safer because it can preserve queued-only behavior unless maintainers explicitly opt reactions into waking.
