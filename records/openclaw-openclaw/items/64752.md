## Label Justifications

- `P2`: The issue blocks Telegram reaction-only workflows but is scoped to reaction notifications rather than a crash, security incident, or whole-channel outage.
- `impact:message-loss`: Incoming Telegram reaction updates are consumed and queued but are not delivered into an agent turn until another wake occurs, so reaction-only inputs are effectively suppressed.

## Summary

Keep open. Current main and the latest release still treat accepted Telegram reactions as queued system events without a wake path, and the active fix paths are still open pending maintainer product choice and live Telegram proof.

## What This Changes

Request that inbound Telegram reaction events trigger an agent turn when configured, instead of remaining queued until another wake occurs.

## Best Possible Solution

Adopt or supersede the open shared notification wake policy with an opt-in Telegram reaction wake mode that preserves queued-only defaults, covers authorization/session routing, updates docs/schema/tests, and includes live Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an authorized Telegram message_reaction with an added emoji reaches enqueueSystemEvent and then returns without a wake call, matching multiple live reports.

## Solution Assessment

Unclear. The requested direction is coherent, but current main only documents queued reaction notifications; the maintainable fix is a default-safe opt-in wake policy rather than unconditional reaction-triggered turns.
