## Summary

Keep open. Current main still accepts Telegram `message_reaction` updates and enqueues reaction system events, but there is no wake/processMessage path afterward; the narrow and broad implementation paths are still open PRs rather than merged fixes.

## What This Changes

The issue asks for received Telegram reaction context events to start an agent turn instead of remaining passive until a later message.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/75899 for the narrow wake gap, while keeping broader reaction semantics in https://github.com/openclaw/openclaw/pull/47687 or https://github.com/openclaw/openclaw/issues/47677.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path for the observed gap: an authorized Telegram `message_reaction` with an added emoji reaches `enqueueSystemEvent` and then returns without a wake/processMessage call. Multiple live comments report the same behavior in real Telegram DMs.

## Solution Assessment

Unclear. The narrow synthetic wake PR is plausible, but maintainers still need to decide whether reaction notifications should wake by default or whether explicit reaction semantics should own that behavior.
