## Label Justifications

- `P2`: The issue blocks Telegram reaction-only workflows but is scoped to reaction notifications rather than a crash, security incident, or whole-channel outage.
- `impact:message-loss`: Incoming Telegram reaction updates are consumed and queued but not delivered into an agent turn until another wake occurs, so reaction-only inputs are effectively suppressed.

## Summary

Keep open. Current main and the latest release still enqueue accepted Telegram reactions as passive system events without waking an agent turn, and the relevant fix paths remain open pending maintainer product choice and live Telegram proof.

## What This Changes

The issue requests that inbound Telegram reaction context events trigger an agent turn when reaction wake behavior is configured.

## Best Possible Solution

Adopt or supersede the open reaction wake PRs with one default-safe policy that wakes only authorized opt-in reaction inputs after successful enqueue, with schema/docs, focused regression tests, and live Telegram proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for current behavior: an authorized Telegram message_reaction with an added emoji reaches enqueueSystemEvent and then returns without any wake call, matching multiple live reports.

## Solution Assessment

Unclear. The requested direction is coherent, but the current repo does not expose the reported triggerAgentTurn contract; the safer solution is a maintainer-approved opt-in wake policy rather than unconditional reaction wakes.
