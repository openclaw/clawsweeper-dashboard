## Summary

Keep open. Current main still receives Telegram `message_reaction` updates and stores them as session-scoped system events, but the Telegram handler does not request a heartbeat or otherwise start an agent turn; the linked open PR remains the active implementation path rather than a shipped fix.

## What This Changes

Telegram inbound reaction events should trigger an agent turn when configured instead of only being recorded as later prompt context.

## Best Possible Solution

Land or replace the Telegram-owned implementation so authorized, configured reactions enqueue structured context and request a session-scoped heartbeat/agent turn, while preserving passive notification behavior by default and covering wake-on plus wake-off behavior in targeted tests and docs.

## Reproduction Assessment

Yes. Multiple live comments document the reproduction, and the current-main static path is high-confidence: `message_reaction` reaches the Telegram handler, queues a system event, and never calls a wake or dispatch path.

## Solution Assessment

No. Current main is not the best solution because it records reactions for a later prompt without waking the agent; the maintainable fix is an explicit Telegram reaction wake policy that calls the existing heartbeat wake primitive after authorized enqueue.
