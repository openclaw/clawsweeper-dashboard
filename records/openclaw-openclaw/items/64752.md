## Summary

Keep open. Current main still handles Telegram `message_reaction` updates by authorizing/routing them and enqueueing system events only; it does not request a heartbeat/agent wake, and the related implementation PR #47687 remains open rather than shipped.

## What This Changes

Telegram inbound reaction events should trigger an agent turn when reaction wake behavior is enabled instead of only being recorded as context events.

## Best Possible Solution

Keep this issue open until a Telegram-owned fix lands that preserves passive reaction notifications by default, provides a documented opt-in wake path, and requests a session-scoped heartbeat/agent turn after an authorized reaction event is enqueued. The fix should include regression tests for wake-on and wake-off behavior, with broader custom emoji/semantic mapping either handled by #47687 or tracked in #47677.

## Reproduction Assessment

Yes. The live repro path is documented in multiple comments, and the current-main static path is high-confidence: a `message_reaction` update reaches `extensions/telegram/src/bot-handlers.runtime.ts`, queues a system event, and never calls a wake or dispatch path.

## Solution Assessment

No, current main is not the best solution because it records reactions as next-prompt context without waking the agent. The narrow maintainable fix is to add a supported Telegram reaction wake policy and call the existing heartbeat wake primitive after authorized enqueue, while keeping queue-only behavior available.
