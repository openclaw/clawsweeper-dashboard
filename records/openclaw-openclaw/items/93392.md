## Best Possible Solution

Land the event-kind guard after redacted real behavior proof shows direct or mentioned `user_request` fallback `NO_REPLY` surfaces the warning while ambient `room_event` silence still behaves as intended.

## Reproduction Assessment

Yes, from source inspection: current main can enable empty-reply silence while `buildSilentFallbackFailurePayload` suppresses the fallback warning without checking `currentInboundEventKind`. I did not run a live forced model-fallback reproduction in this read-only review.

## Solution Assessment

Yes, threading the existing inbound event kind into the two empty-reply suppression decisions is the narrowest maintainable fix for this bug; changing silent-policy generation would risk breaking ambient `room_event` silence.
