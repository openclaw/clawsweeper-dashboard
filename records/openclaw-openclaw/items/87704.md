## Summary

Keep open: current main and v2026.6.6 still suppress standalone Discord tool-progress messages in progress mode, while the shared progress draft can remain unstarted for a fast one-tool turn, so the reported blind spot still has a source-backed path and needs maintainer UX direction.

## What This Changes

Discord progress streaming should keep fast tool-call activity visible instead of suppressing standalone tool messages before any preview draft appears.

## Best Possible Solution

Pick and test one default fallback: either start the Discord progress draft on the first visible fast tool event or suppress standalone tool messages only after a preview draft has actually appeared.

## Reproduction Assessment

No live Discord/Codex current-main run was executed, but source inspection gives a high-confidence path where the first tool event schedules a delayed draft, standalone progress is already suppressed, and final delivery can cancel the unstarted draft.

## Solution Assessment

Unclear until maintainers choose the UX contract; the proposed unsuppression is plausible but can duplicate progress in longer turns, so the best fix should encode the fallback policy with focused Discord and shared-channel tests.
