## Summary

Keep open: current main and latest release still suppress standalone Discord tool-progress messages in progress mode, while the progress draft intentionally waits before first work and can remain unstarted for a fast single-tool turn.

## What This Changes

The issue requests a Discord progress-mode fallback so fast tool calls show either per-tool messages or a visible progress draft instead of only typing plus the final answer.

## Best Possible Solution

Pick a default fallback contract, then encode it in shared streaming plus Discord regression tests: either start the progress draft on the first visible fast tool event or suppress standalone tool messages only after a preview draft has actually rendered.

## Reproduction Assessment

Yes for source-level reproduction: current main suppresses standalone progress in progress mode, delays the first progress draft for five seconds, and only finalizes a progress preview after it has started. I did not run a live Discord/Codex repro in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the UX contract. The reporter's unsuppression proposal is plausible, but a safer final fix may be to suppress standalone progress only after the preview has actually appeared or to start the preview immediately for the first meaningful fast tool event.
