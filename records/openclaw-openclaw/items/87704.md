## Summary

Keep open: current main and the latest release still contain the progress-mode standalone-suppression path, while the Discord progress draft can remain unstarted for short work. The report is source-reproducible, but the exact fix needs a maintainer UX choice because current tests and docs also encode suppression of older standalone tool-progress messages during preview streaming.

## What This Changes

The issue requests visible Discord per-tool progress when `streaming.mode: "progress"` does not produce a live preview for fast tool calls.

## Best Possible Solution

Choose and test one fallback policy that preserves progress-mode visibility without adding config: either make the first fast tool event start a Discord progress preview before final delivery, or defer standalone-message suppression until the preview has actually appeared.

## Reproduction Assessment

No live high-confidence current-main repro was run. Source inspection supports the failure shape: progress mode suppresses standalone tool messages while the progress draft can be delayed or canceled before it becomes visible.

## Solution Assessment

Unclear. The report's desired visibility is valid, but the safest implementation should be chosen deliberately because current docs/tests encode preview-mode suppression of older standalone progress messages.
