## Summary

Keep open: current main and the latest release still suppress standalone tool-progress delivery in progress mode before proving a Discord progress draft is visible, while the first work event can remain behind the delayed draft gate. This is a valid Discord visibility bug and UX-policy decision, not a cleanup-close candidate.

## What This Changes

The issue asks OpenClaw to keep Discord operator-visible tool progress from disappearing in progress streaming mode when a fast tool completes before the preview draft is rendered.

## Best Possible Solution

Choose the progress fallback contract, then update shared streaming and Discord regression coverage so a fast single-tool turn emits at least one visible progress signal without duplicating longer progress drafts unnecessarily.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path because progress mode suppresses standalone progress and the first draft work event can remain delayed. I did not run a live Discord/Codex reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the fallback UX contract. The reporter's unsuppression proposal is plausible, but the safer final shape may be to start the preview immediately for the first meaningful fast-tool event or suppress standalone messages only after the preview is actually visible.
