## Summary

Keep open: current main and v2026.6.8 still suppress standalone Discord tool-progress messages in progress mode, while the progress draft may stay unstarted for a fast single-tool turn. The remaining work is a maintainer UX decision about the fallback contract, not a cleanup-close candidate.

## What This Changes

The issue asks OpenClaw to restore visible Discord tool-progress feedback when `streaming.mode="progress"` suppresses standalone tool messages but no preview draft is shown for fast tools.

## Best Possible Solution

Choose a progress fallback contract, then update shared streaming and Discord regression coverage so fast tool activity produces at least one visible Discord progress signal without duplicating longer-running draft progress unnecessarily.

## Reproduction Assessment

Yes for source-level reproduction: current main still suppresses standalone progress in progress mode, and the first work event may only schedule a delayed draft instead of rendering immediately. I did not run a live Discord/Codex repro in this read-only pass.

## Solution Assessment

Unclear until maintainers pick the fallback UX contract. The reporter's unsuppression proposal is plausible, but the safer final shape may be to start the preview immediately for the first meaningful fast-tool event or suppress standalone messages only after the preview is actually visible.
