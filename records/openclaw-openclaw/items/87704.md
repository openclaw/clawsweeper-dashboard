## Summary

Keep open: current main and latest release still suppress standalone Discord tool-progress messages in progress mode while the progress draft can remain unstarted for a fast single-tool turn; the remaining choice is the fallback UX contract, not whether the source path exists.

## What This Changes

The issue requests restoring visible Discord tool-progress feedback for fast tool calls when `streaming.mode: "progress"` suppresses standalone tool messages but no preview draft update appears.

## Best Possible Solution

Choose a progress fallback contract, then update shared streaming and Discord regression coverage so standalone tool progress is suppressed only after a preview has actually rendered, or the first meaningful fast-tool event starts the draft immediately.

## Reproduction Assessment

Yes for source-level reproduction: current main suppresses standalone progress in progress mode, delays the first progress draft, and finalizes a progress preview only after it has started. I did not run a live Discord/Codex repro in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the UX contract. The reporter's unsuppression proposal is plausible, but the safer final shape may be to start the preview immediately for the first meaningful fast-tool event or suppress standalone messages only after the preview is actually visible.
