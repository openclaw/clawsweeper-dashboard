## Summary

Keep open: current main still uses one task failure bucket for failed, timed_out, and lost records, and `openclaw status` renders that bucket plus audit warnings as operational task health. A related open PR attempts part of the split, but it is not merged and does not settle the requested retained/soft-retained cleanup semantics.

## What This Changes

The issue asks OpenClaw to separate live task failures from retained historical and soft-retained task records in status, audit, and cleanup surfaces.

## Best Possible Solution

Define a maintainer-approved task health model that keeps historical task records inspectable while making default status alert on live or unreviewed operational failures, then align CLI JSON, text output, audit, maintenance, docs, and focused tests.

## Reproduction Assessment

Yes, source-level. A retained failed, timed_out, or lost task passed through `getInspectableTaskRegistrySummary` is counted in `failures`, and `openclaw status` renders that count as task issues; I did not run a live SQLite state reproduction because this review is read-only.

## Solution Assessment

Unclear. The requested split is directionally maintainable, but the exact retained/soft-retained and cleanup or acknowledgement semantics need maintainer approval before implementation; the existing related PR is not sufficient as-is.
