## Summary

Keep open: current main still treats retained terminal task failures as status issues, while the requested current/retained/soft-retained split would change the CLI/JSON status and cleanup model rather than close as already implemented.

## What This Changes

Request `openclaw status` and task audit/maintenance to distinguish live task failures from retained historical task records and provide a supported cleanup or acknowledgement path.

## Best Possible Solution

Define a maintainer-approved task health model that keeps historical task rows inspectable while making the default CLI status alert only on live or unreviewed operational failures, then align JSON types, text output, audit, docs, and focused tests.

## Reproduction Assessment

Yes, from source inspection: a retained failed, timed_out, or lost task passed through `getInspectableTaskRegistrySummary` is counted in `failures` and rendered by `openclaw status` as task issues. I did not run a live state-database reproduction because this review is read-only.

## Solution Assessment

Unclear: the requested split is directionally sound, but the exact current/retained/soft-retained semantics and cleanup or acknowledgement UX need maintainer approval before implementation.
