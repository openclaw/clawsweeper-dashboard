## Summary

Keep open: current main and the latest shipped docs still aggregate retained terminal task failures into task issue and audit-warning counts, so the reported status-noise gap is real. This is a product/UX improvement rather than an already-fixed bug.

## What This Changes

The issue requests a task health model that separates live task failures from retained historical terminal task records and adds a supported cleanup or acknowledgement path.

## Best Possible Solution

Define a native task-health contract that preserves task history and auditability while making normal status primarily alert on unresolved live operational failures.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear reproduction path: retained `failed`/`timed_out`/`lost` task records still flow into aggregate issue and audit-warning counts, though I did not run a live stateful task DB repro.

## Solution Assessment

Is this the best way to solve the issue? The direction is sound, but the exact current/retained/soft-retained contract and cleanup workflow should be decided before implementation.
