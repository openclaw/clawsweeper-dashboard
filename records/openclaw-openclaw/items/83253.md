## Summary

Keep open: current main has a narrower retained-lost mitigation, but it still lacks the requested current/retained/soft-retained task-health model for retained `failed`, `timed_out`, delivery-failed, and reviewed historical task records.

## What This Changes

The issue asks for task status and audit output to distinguish live task failures from retained historical task records and to provide a supported cleanup or acknowledgement path.

## Best Possible Solution

Define one maintainer-approved task health model across status, tasks list/audit JSON, maintenance, docs, and tests so default status alerts only on live or unreviewed operational failures while historical records stay inspectable.

## Reproduction Assessment

Yes, at source level: retained `failed` or `timed_out` task records still increment aggregate `failures`, and `tasks audit` still reports a combined finding set. I did not run a live state-database reproduction because this review is read-only.

## Solution Assessment

Unclear: the requested split is directionally maintainable, but the exact soft-retained or acknowledged semantics and JSON compatibility need maintainer approval before a fix PR should define them.
