## Summary

Keep open: current main still uses one task `failures` bucket for retained terminal rows, and the related implementation PR is still open, conflicting, and narrower than the requested current/retained/soft-retained health model.

## What This Changes

The issue requests that task status distinguish live task failures from retained historical and soft-retained task records so normal status output does not stay unhealthy forever.

## Best Possible Solution

Define and land a maintainer-approved task-health contract that keeps historical task rows inspectable while default status alerts on live or unreviewed operational failures, then align CLI/JSON output, audit, maintenance, docs, and tests.

## Reproduction Assessment

Yes, from source inspection: retained `failed`, `timed_out`, and `lost` task records passed through `getInspectableTaskRegistrySummary` are counted in `failures` and rendered by `openclaw status` as task issues. I did not mutate a live state database or run tests because this review is read-only.

## Solution Assessment

Unclear: the requested split is directionally maintainable, but the exact current/retained/soft-retained semantics and cleanup or acknowledgement UX need maintainer approval. The related open PR is a plausible source of work but is currently conflicting and has correctness/proof blockers.
