## Summary

Keep open. Current main has a beta-first stable promotion flow and release validation, but it still exposes only stable/beta/dev update channels and does not provide the pinned LTS-style production release line requested here; the Windows Telegram regression context makes this a real release-policy discussion, not a narrow cleanup close.

## What This Changes

Request to provide a stricter production-stable or LTS release path instead of having production users follow each normal latest update.

## Best Possible Solution

Define the release support model first, then align release docs, dist-tags, update-channel semantics, and Windows Telegram release gates with that policy if maintainers choose an LTS-style lane.

## Reproduction Assessment

Not applicable for the release-policy feature request. The linked Windows Telegram reports provide stability context, but this item itself is not a single source-level bug reproduction.

## Solution Assessment

Unclear as a product decision. The current beta-first validation flow is useful but not equivalent to a pinned LTS release line, so the maintainable path is release-manager policy design before code or docs changes.
