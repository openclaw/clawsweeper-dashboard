## Summary

Keep open. Current main documents and implements stable/beta/dev update lanes plus beta-first release validation, but it does not define a separate LTS or pinned production-stable policy; the added Windows Telegram regression context makes this a maintainer release-policy decision, not a closeable source bug.

## What This Changes

Request a real stable/LTS OpenClaw release lane or policy for production use instead of treating each normal update as the production target.

## Best Possible Solution

Maintainers should decide whether to add a distinct LTS/prod-stable channel or stricter latest-promotion policy, then encode it in release docs, workflows, update-channel semantics, and targeted Windows Telegram release gates.

## Reproduction Assessment

Not applicable for the release-policy feature request. The related Telegram regression claims have concrete issue references, especially open #76640, but this issue does not provide a single current-main gateway crash reproduction path.

## Solution Assessment

Unclear until maintainers make a product/release-policy decision. Existing beta-first promotion and package validation are a useful base, but a real LTS or production-stable lane would need intentional docs, workflow, and update-channel design.
