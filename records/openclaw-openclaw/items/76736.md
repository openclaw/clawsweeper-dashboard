## Summary

Keep open. Current main has stable/beta/dev release lanes, auto-update controls, and release validation, but it does not provide a separate LTS or explicitly validated production-stable lane; the added Windows Telegram regression examples make this a release-policy and quality-gate decision, not a narrow bug fix.

## What This Changes

Request an explicitly validated stable or LTS release path separate from the normal update stream for production use.

## Best Possible Solution

Maintainers should decide whether OpenClaw needs a distinct LTS/validated channel or a stricter stable-promotion policy, then align release docs, update-channel semantics, workflows, and validation gates around that decision.

## Reproduction Assessment

Not applicable for the release-policy feature request. The linked regressions provide supporting examples, but this issue does not give one current-main gateway crash reproduction path with logs, config, and steps.

## Solution Assessment

Unclear as a product decision. A stricter production-stable or LTS path may be the right outcome, but it should be designed through release policy, dist-tag/update-channel semantics, and validation requirements rather than treated as a small source bug fix.
