## Summary

Keep open. Current main defines stable/beta/dev update lanes and a beta-first promotion flow, but it does not provide the requested LTS-style or stricter production-stable release policy, so this needs maintainer release-policy judgment rather than conservative cleanup closure.

## What This Changes

Request a separate real stable/LTS release path for production use because normal updates are reported to introduce gateway instability.

## Best Possible Solution

Maintainers should decide whether stable/latest is sufficient or define an LTS/production-stable policy, then align release docs, update-channel semantics, workflow dist-tags, and rollback guidance.

## Solution Assessment

Unclear as a product decision. A stricter production-stable or LTS policy could be valid, but it should be designed across release docs, update semantics, and publish automation before any narrow code change is attempted.
