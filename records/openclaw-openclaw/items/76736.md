## Summary

Keep open. Current main has a beta-first stable promotion flow, release validation workflows, package acceptance, and stable auto-update rollout delay, but it still defines only stable/beta/dev and does not provide the requested LTS-style or separately trusted production-stable channel; the added regression context makes this a maintainer release-policy decision, not a closeable cleanup item.

## What This Changes

The issue requests a real production-stable or LTS-style OpenClaw release lane because normal updates have repeatedly broken gateway and Telegram-on-Windows usage.

## Best Possible Solution

Maintainers should decide whether the existing beta-first latest promotion is enough or whether OpenClaw needs a documented LTS/production-stable policy with explicit dist-tag/update-channel semantics and required Windows Telegram validation before promotion.

## Reproduction Assessment

Not applicable for the release-policy feature request. The issue and follow-up comment cite real regressions, but this item does not provide a high-confidence current-main gateway crash reproduction path beyond the separately open #76640 Telegram media report.

## Solution Assessment

Unclear as a product decision. The current beta-first release validation is the closest existing mechanism, but a separate LTS or production-stable lane would need a maintainer-owned policy and update-channel design rather than a narrow bug fix.
