## Summary

Keep open: the PR is aimed at a real Feishu local-media failure path, but current main has newer streaming closeout and thread-fallback behavior that this branch does not preserve, and the external PR still lacks real Feishu behavior proof.

## What This Changes

The PR adds Feishu-specific detection for blocked local `MEDIA:` paths, a one-time final fallback message, dispatcher tests, and MEDIA path documentation.

## Best Possible Solution

Port the Feishu blocked-local-media hint and one-time final fallback onto current main while preserving streaming closeout, thread fallback, and current MEDIA policy docs.

## Reproduction Assessment

Yes, at source level: blocked local paths still throw `LocalMediaAccessError` with `path-not-allowed`, and current Feishu `onError` only logs and closes streaming. I did not run a live Feishu account reproduction in this read-only review.

## Solution Assessment

No, not as-is. The Feishu dispatcher is the right boundary, but the branch must be ported onto current streaming closeout and top-level reply fallback behavior before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve streaming error closeout state:** `extensions/feishu/src/reply-dispatcher.ts:399`
  - body: The new local-media path calls plain `closeStreaming()`. Current main marks the reply as error-closed and calls `closeStreaming({ markClosedForReply: false })`; without that, a media failure can mark the streamed text as a successful final and suppress later recovery delivery.
  - confidence: 0.9
- **[P2] Carry top-level fallback into the recovery notice:** `extensions/feishu/src/reply-dispatcher.ts:403-410`
  - body: The fallback notice sends through `sendMessageFeishu` without `allowTopLevelReplyFallback`. Current Feishu quoted-thread replies can fall back to a top-level message when the quoted target disappeared, but this notice would fail in that same recovery case.
  - confidence: 0.84
