## Summary

Keep open. Current main and v2026.6.10 still use a Docker-managed sandbox-browser lifecycle, and the requested external CDP/noVNC endpoint support remains a security-sensitive product/API decision rather than implemented or fully covered by the broader browser-driver issue.

## What This Changes

The issue requests externally managed CDP/noVNC endpoints so sandboxed browser automation can work without requiring the OpenClaw Gateway to control Docker.

## Best Possible Solution

Define one maintainer-approved sandbox-browser backend or endpoint contract that supports externally managed browsers while preserving CDP auth, SSRF policy, noVNC observer tokens, lifecycle semantics, docs, and tests.

## Reproduction Assessment

Yes, at source level. Current main still routes sandbox browser setup through Docker lifecycle and rejects browser support for non-browser-capable backends; I did not run a live Kubernetes, Docker Compose, or external CDP sidecar deployment in this read-only review.

## Solution Assessment

Unclear as proposed. Optional endpoint fields are a plausible unblocker, but the best fix needs maintainer agreement on backend ownership, auth, SSRF/noVNC policy, and config contract rather than a one-off shortcut.
