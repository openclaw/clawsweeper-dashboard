## Summary

Keep open. Current main still models sandbox browser lifecycle as a Docker-managed Chromium/noVNC container, and the existing backend abstraction does not expose a backend-owned CDP/noVNC endpoint contract, so this is a real feature/product-design request rather than stale cleanup.

## What This Changes

The issue requests configurable external CDP/noVNC browser sandbox endpoints so browser automation can run without Docker-managed browser lifecycle.

## Best Possible Solution

Add a deliberate browser sandbox endpoint/backend contract, with config/schema/docs/tests and equivalent security guardrails for CDP ingress, bridge auth, noVNC observer access, and externally managed lifecycle.

## Reproduction Assessment

Not applicable; this is a feature request, not a broken existing contract. Source inspection does give high confidence that current main lacks external sandbox browser endpoints and still routes browser sandbox startup through Docker-managed Chromium.

## Solution Assessment

Unclear. The proposed endpoint config is a plausible narrow unblocker, but maintainers need to choose between a direct external-endpoint config and a backend-owned browser capability contract because this touches sandbox, browser-control, and CDP/noVNC security boundaries.
