## Summary

Keep open. Current main and v2026.6.10 still keep the sandbox-browser path on Docker-managed container lifecycle, while the requested external CDP/noVNC endpoint support remains a security-sensitive product/API decision rather than an implemented feature or a duplicate of the broader browser-driver registry issue.

## What This Changes

The issue requests externally managed CDP/noVNC endpoints for the sandbox browser so browser automation can run without a Docker daemon managing the browser container.

## Best Possible Solution

Define a maintainer-approved sandbox-browser backend or endpoint contract that supports externally managed browsers while preserving relay auth, SSRF policy, noVNC observer tokens, and explicit operator-controlled configuration.

## Reproduction Assessment

Yes, at source level: current main still routes sandbox browser setup through Docker lifecycle and rejects browser support for non-browser-capable backends. I did not run a live Kubernetes, Docker Compose, or external CDP sidecar deployment in this read-only review.

## Solution Assessment

Unclear as proposed. The endpoint fields are a plausible unblocker, but the best solution needs maintainer agreement on backend ownership, auth, SSRF/noVNC policy, and config contract rather than an automatic one-off config shortcut.
