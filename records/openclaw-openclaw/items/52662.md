## Summary

Keep open. Current main and v2026.6.10 still route sandbox-browser startup through Docker-managed container lifecycle and Docker port mappings, while the requested external CDP/noVNC endpoint support remains a security-sensitive endpoint/backend API decision rather than implemented behavior or a duplicate of the broader browser-driver registry issue.

## What This Changes

Requested support for configuring externally managed browser CDP/noVNC endpoints so the browser sandbox can work without Docker-managed browser lifecycle.

## Best Possible Solution

Define a maintainer-approved sandbox-browser endpoint/backend contract that supports externally managed CDP/noVNC targets while preserving relay auth, SSRF policy, noVNC observer tokens, explicit operator configuration, docs, and upgrade-safe validation.

## Reproduction Assessment

Yes, at source level: current main routes sandbox browser startup through Docker lifecycle and Docker port mapping, and the strict sandbox browser schema has no external CDP/noVNC endpoint fields. I did not run a live Kubernetes, Docker Compose, or external sidecar deployment in this read-only review.

## Solution Assessment

No, not as the proposed minimal two-field shortcut without maintainer design. The maintainable solution needs an explicit endpoint/backend contract that preserves the existing CDP relay auth, SSRF, noVNC, lifecycle, and upgrade semantics.
