## Summary

Keep open. Current main and v2026.6.8 still keep sandbox-browser lifecycle on the Docker-managed path, while external CDP/noVNC support would add security-sensitive browser-control and configuration surface that needs maintainer design.

## What This Changes

The issue requests configured external CDP/noVNC endpoints so sandbox browser automation can use an externally managed browser instead of requiring Docker lifecycle management.

## Best Possible Solution

Define a reviewed sandbox-browser backend or endpoint contract that supports externally managed browsers while preserving relay auth, SSRF policy, noVNC observer tokens, and explicit operator-controlled configuration.

## Reproduction Assessment

Yes, at source level: current main still routes sandbox browser setup through Docker lifecycle and rejects browser support for non-browser-capable backends. I did not run a live Kubernetes, Docker Compose, or external CDP sidecar deployment in this read-only review.

## Solution Assessment

Unclear. The proposed endpoint fields are a plausible unblocker, but the best solution needs maintainer agreement on backend ownership, auth, SSRF/noVNC policy, and config contract rather than an automatic shortcut.
