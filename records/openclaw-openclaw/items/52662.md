## Summary

Keep open: current main still lacks externally managed CDP/noVNC support for sandbox browsers, and the useful comments describing a minimal endpoint mode and Lightpanda use case remain unmet. The existing implementation is still Docker-lifecycle based, while global remote CDP profiles are a separate browser-control feature and do not solve sandbox browser/noVNC support for non-Docker backends.

## What This Changes

Allow the sandbox browser configuration to use externally managed CDP/noVNC endpoints so browser automation can work with non-Docker sandbox backends without requiring Docker daemon lifecycle management.

## Best Possible Solution

Add an explicit, validated sandbox browser seam: either a backend-owned browser capability or narrowly scoped external CDP/noVNC settings under sandbox browser config. It should skip Docker lifecycle only in that mode, preserve current Docker defaults, update config types/schema/help/docs, and add regression coverage for non-Docker sandbox browser use plus security checks around endpoint validation and noVNC/auth handling.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. There is high-confidence current-main verification by static inspection: the config/schema do not accept external endpoint keys, non-Docker backends are rejected for browser sandboxes, and ensureSandboxBrowser still requires Docker lifecycle and Docker port mapping.

## Solution Assessment

Unclear. The proposed endpoint fields are a plausible narrow unblocker, but the safest maintainable solution should include validated endpoint semantics and a backend/browser capability boundary rather than a raw passthrough that bypasses existing SSRF, auth, noVNC, and cleanup assumptions.
