## Summary

Keep open: the user-visible iMessage placeholder bug is valid and independently reproduced, but this patch still removes the media-only echo key that current iMessage dedupe relies on when IDs are missing or mismatched. The supplied after-fix proof is an instrumented RPC helper run, not live after-fix iMessage behavior.

## What This Changes

This PR removes iMessage outbound media placeholder synthesis so attachment-only sends pass empty text to imsg, and adds send tests for media-only sends plus literal placeholder text.

## Best Possible Solution

Keep RPC text empty for real media-only iMessage sends, but separately record a non-visible media echo key or equivalent metadata for dedupe, with coverage for missing or mismatched IDs and live after-fix proof.

## Reproduction Assessment

Yes. Current main source rewrites empty image sends to `<media:image>`, and the PR discussion includes an independent live reproduction on v2026.5.12-beta.3 that matches this path.

## Solution Assessment

No. The visible-send direction is right, but the current patch deletes the internal echo signal instead of decoupling delivered RPC text from dedupe metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve media echo matching while clearing RPC text:** `extensions/imessage/src/send.ts:183`
  - body: This clears `message` for media-only sends, but `message` is also persisted for echo matching and returned as `sentText` for the in-memory echo cache. When imsg returns `ok`/`unknown` or the inbound row has a different ID shape, current echo detection falls back to the media placeholder text; with empty text no cache key is recorded, so OpenClaw can reprocess its own media as inbound. Keep the RPC text empty, but store a non-visible media echo key or add equivalent coverage for that case.
  - confidence: 0.86
