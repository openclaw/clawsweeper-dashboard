## Summary

Keep open: the PR fixes the user-visible placeholder at the iMessage RPC boundary, but it also removes the internal media placeholder text that the iMessage echo/dedupe path still uses as a fallback for attachment-only echoes.

## What This Changes

The PR changes iMessage media-only sends to pass an empty text body with the file path, and adds regression coverage for media-only sends plus literal placeholder text without an attachment.

## Best Possible Solution

Keep the outbound RPC text empty for media-only sends, but preserve a separate internal echo/dedupe key for attachment-only media so no-id or compatibility responses still suppress self-chat echoes.

## Reproduction Assessment

Yes from source inspection: current main calls `sendMessageIMessage` with empty text and an image attachment, then assigns `<media:image>` before sending. I did not execute the path because this review is read-only.

## Solution Assessment

No as written: removing the visible placeholder is the right direction, but the patch should decouple user-visible RPC text from the internal echo/dedupe placeholder instead of deleting the placeholder path entirely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Keep a media echo key for attachment-only sends:** `extensions/imessage/src/send.ts:183`
  - body: This leaves media-only sends with an empty `sentText`. The monitor caches `sent.sentText || undefined`, persisted echo storage drops empty text and `ok`/`unknown` ids, and inbound attachment-only rows still synthesize `bodyText` as `<media:image>`. For compatibility sends that do not return a usable message id, self-chat media echoes can miss the cache and be dispatched back to the agent. Please keep an internal echo/dedupe text or equivalent while leaving the outbound RPC `text` empty.
