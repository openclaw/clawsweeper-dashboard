## Summary

Keep this PR open: the visible-placeholder fix points in the right direction, but the current patch also removes the non-visible media marker used as an echo-cache fallback for attachment-only iMessage sends.

## What This Changes

This PR removes iMessage media placeholder synthesis from `sendMessageIMessage` and updates send tests to expect empty outbound text for attachment-only sends while preserving literal placeholder text without attachments.

## Best Possible Solution

Keep RPC `text` empty for real media-only sends, but preserve a non-visible media echo key or equivalent cache signal so inbound media echoes still dedupe when ids are missing or mismatched.

## Reproduction Assessment

Yes. Current main source and tests show that a media-only iMessage send with an image attachment rewrites empty text to `<media:image>` and passes that value through the RPC/caching path.

## Solution Assessment

No. The narrow direction is correct for visible delivery, but the implementation should decouple delivered RPC text from internal echo matching instead of deleting the media marker everywhere.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve media echo matching when clearing RPC text:** `extensions/imessage/src/send.ts:183`
  - body: This removes the only text key cached for attachment-only sends. `sentText` is also what the persisted and in-memory iMessage echo caches receive; when the bridge returns `ok`/`unknown` or the inbound row has a different id shape, current echo detection falls back to the `<media:image>` body text. Keep the RPC text empty, but still cache a non-visible media echo key or add equivalent coverage for the id-missing/mismatched case.
