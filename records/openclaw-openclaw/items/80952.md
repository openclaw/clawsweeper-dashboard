## Summary

Keep open: the unrelated localized docs were removed, but this PR still exposes a shared plugin command reply flag while only Telegram honors it, and the external contributor has provided only test output rather than real Telegram after-fix proof.

## What This Changes

The PR adds an optional `suppressReply` field to the shared reply payload type and makes Telegram native plugin commands clean up progress state and skip the empty-response fallback when that flag is true.

## Best Possible Solution

Land a clear silent-result contract at the right boundary, either Telegram-scoped or consistently implemented across native adapters, with focused regression coverage, documentation of the semantics, and real Telegram proof that the duplicate fallback is gone.

## Reproduction Assessment

Yes, source-reproducible on current main: undefined plugin command returns normalize to `{}`, Telegram native commands treat that as non-renderable, and an existing test asserts the empty-response fallback. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not yet: a silent plugin-command result is a plausible fix, but putting it on shared `ReplyPayload` should either be honored consistently by native adapters or scoped to Telegram-specific data before merge.
