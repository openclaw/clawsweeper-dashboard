## Summary

Keep open: current main has Slack duplicate-completion state after dispatch, but no pending receive journal or startup replay before the volatile Slack message path, so the requested restart-window durability remains real work. The configurable spool-path part needs maintainer direction because repository policy strongly prefers SQLite/plugin runtime state over new sidecar paths or config surfaces.

## What This Changes

The issue requests Slack inbound receive durability by persisting and replaying Slack socket-mode events with parity to Telegram/WhatsApp durable ingress behavior.

## Best Possible Solution

Implement a Slack-owned durable receive journal using the existing plugin runtime channel ingress queue, accepting normal message/app_mention payloads before volatile debounce/dispatch and replaying pending entries at monitor startup; avoid or defer a new configurable file spool path unless maintainers explicitly approve that config surface.

## Reproduction Assessment

No live current-main restart reproduction was established. Source inspection does show Slack lacks a pending durable receive journal before the handler/debounce path, which is enough to keep the issue open but not enough to call the runtime loss window fully reproduced.

## Solution Assessment

Unclear as requested. A Slack-plugin durable receive journal using the existing channel ingress queue looks like the right implementation direction, but the requested configurable spool path is not automatically the best fit under OpenClaw storage and config policy.
