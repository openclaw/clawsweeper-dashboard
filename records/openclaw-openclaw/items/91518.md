## Summary

Keep open: current main and v2026.6.8 still lack a Slack pending receive journal and startup replay path, while the existing Slack state is only a delivered-message dedupe ledger written after dispatch. The work is real but needs maintainer storage/config direction and live Slack restart proof before it is a safe fix lane.

## What This Changes

Request a Slack-owned durable inbound receive journal and replay path so Socket Mode messages survive gateway restarts, with documented and configurable spool behavior.

## Best Possible Solution

Add an account-scoped Slack durable receive/replay path through the existing SQLite-backed channel ingress queue or approved plugin state primitive, and avoid a new configurable sidecar spool unless maintainers explicitly choose that surface.

## Reproduction Assessment

No live Slack Socket Mode restart run was performed. Source inspection shows Slack has only post-dispatch delivery tombstones and no pending receive replay, so the missing durability is source-reproducible but not runtime-reproduced here.

## Solution Assessment

Unclear as written. The durable journal direction is plausible, but the best maintainable fix should use the existing SQLite-backed queue/plugin state and needs maintainer approval before adding a configurable filesystem spool path.
