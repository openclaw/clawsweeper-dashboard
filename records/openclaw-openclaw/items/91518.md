## Summary

Close as duplicate/superseded by the open canonical restart-catch-up tracker at https://github.com/openclaw/openclaw/issues/55792. Current main still lacks the requested Slack pending receive journal, but the remaining user problem is already covered there, including Slack-specific catch-up notes; the proposed local journal is only a partial shape because it cannot recover events Slack never delivered while Socket Mode was disconnected.

## What This Changes

Add Slack-owned durable inbound receive/replay so Socket Mode events are persisted before dispatch and replayed after restarts, mirroring Telegram-style ingress durability.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/55792 as the canonical design tracker, then implement Slack replay/durable receive there using the SQLite-backed channel ingress queue and approved config/default behavior.

## Reproduction Assessment

No live Slack restart repro was run in this read-only review. Source inspection does show the narrower gap with medium confidence: Slack has no pre-dispatch pending receive journal or startup replay path, while the broader restart-catch-up reproduction is tracked at https://github.com/openclaw/openclaw/issues/55792.

## Solution Assessment

No, not as a standalone issue. A Slack-local journal inside the current Bolt event handlers is only a partial solution because Events API payloads are acknowledged before listener middleware and disconnected-window events need history catch-up or a lower-level receiver design.
