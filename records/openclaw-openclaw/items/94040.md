## What This Changes

A Windows Companion/node onboarding approval flow asks the operator to run `openclaw nodes approve <requestId>`, but the command fails with an opaque unknown-requestId gateway error.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/94452 as the focused nodes CLI recovery fix; treat gateway idempotency semantics as separate maintainer-reviewed work.

## Reproduction Assessment

Yes, at source level: current main returns `unknown requestId` from `node.pair.approve` when the pending ID is missing, and the nodes CLI wrapper can render that raw gateway error. I did not live-run the full Windows Companion plus Kubernetes port-forward setup.

## Solution Assessment

Yes for the immediate operator-facing bug: translating the exact stale/unknown gateway response in the nodes CLI is narrower than changing token issuance or approval semantics. The broader idempotency approach in https://github.com/openclaw/openclaw/pull/94532 needs separate maintainer review.
