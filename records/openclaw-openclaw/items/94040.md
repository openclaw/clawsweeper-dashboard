## What This Changes

The issue reports that `openclaw nodes approve <requestId>` can fail during Windows Companion onboarding with an opaque `GatewayClientRequestError: unknown requestId` error.

## Best Possible Solution

Land or reject https://github.com/openclaw/openclaw/pull/94452 as the focused CLI recovery fix; evaluate gateway idempotency separately only with explicit auth/token contract tests.

## Reproduction Assessment

Yes, at source level: current main returns `unknown requestId` from `node.pair.approve` when the pending ID is missing, and the nodes CLI wrapper can render that raw gateway error. I did not live-run the full Windows Companion plus Kubernetes port-forward setup.

## Solution Assessment

Yes for the immediate operator-facing bug: translating the exact gateway stale-request response in the nodes CLI is narrower than changing gateway approval or token semantics. The broader idempotency idea should stay separate unless maintainers explicitly choose that contract.
