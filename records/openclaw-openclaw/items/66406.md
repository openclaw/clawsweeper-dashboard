## Summary

Keep open because current main still forwards raw Feishu reaction emoji unchanged into the Feishu reaction API, and the canonical fixing PR remains open with closing syntax for this issue.

## What This Changes

Feishu reactions should normalize user-provided Unicode emoji to Feishu-supported `emoji_type` values before add/list/remove reaction API calls.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/66478 with a Feishu-local normalizer using the documented `emoji_type` vocabulary, focused regression tests, and redacted full-path proof.

## Reproduction Assessment

Yes, at source level. The issue gives concrete `message(action="react", emoji="👍")` steps, and current main passes that raw emoji into Feishu `reaction_type.emoji_type`, which Feishu documents as requiring supported type strings and returning 231001 for invalid values.

## Solution Assessment

Yes, the best direction is a Feishu-owned normalization layer at the reaction API boundary. The open canonical PR is the right implementation path to review, repair if needed, and land before closing this issue.
