## Summary

Keep open. Current main still seeds the interactive add-agent workspace prompt through the resolver path that can nest a new agent under the main workspace, while the docs still describe peer-level workspaces; the open implementation PRs with closing syntax should be reviewed or superseded rather than closing the issue before a fix merges.

## What This Changes

The issue asks for `openclaw agents add <id>` to pre-fill a peer-level workspace path such as `~/.openclaw/workspace-<id>` instead of `~/.openclaw/workspace/<id>`.

## Best Possible Solution

Land one narrow existing PR, or an equivalent maintainer patch, that changes the new-agent wizard default to the documented peer-level workspace while preserving explicit and legacy configured workspace paths, with focused resolver/wizard tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The source path is deterministic on current main: setup can persist `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir(cfg, agentId)` as the prompt default, and the resolver appends the non-default agent id under that fallback.

## Solution Assessment

Yes. Changing the add-agent wizard default to a peer-level path is the narrow maintainable fix, provided existing agents with explicit or legacy nested workspaces are not silently relocated.
