## Label Justifications

- `P2`: This is a real multi-agent workspace-default bug with limited blast radius and open targeted fix paths, but it is not a crash loop or unusable core runtime.
- `impact:data-loss`: Accepting the nested default can put peer-agent workspaces under the main workspace, so recursive cleanup of the main workspace can delete peer-agent files.
- `impact:security`: The nested layout can let main-workspace operations include peer-agent files under OpenClaw's soft workspace-boundary model.

## Summary

Keep open: current main and the latest release still seed the interactive add-agent workspace prompt from the resolver branch that nests non-default agents under `agents.defaults.workspace`, while the multi-agent docs still describe sibling `workspace-<agentId>` workspaces. The open closing PR at https://github.com/openclaw/openclaw/pull/71941 and overlapping resolver PR at https://github.com/openclaw/openclaw/pull/78113 need maintainer resolution before this issue can close.

## What This Changes

The issue requests that `openclaw agents add <id>` prefill the workspace prompt with sibling `~/.openclaw/workspace-<id>` instead of nesting under the main workspace.

## Best Possible Solution

Land one compatibility-reviewed fix, likely by refreshing or superseding the open PRs, so new agent defaults use the documented sibling workspace while explicit workspaces, existing-agent updates, and intentional shared-base layouts remain stable.

## Reproduction Assessment

Yes. Source inspection on current main shows setup writes `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and that resolver appends the non-default id while docs expect a sibling workspace.

## Solution Assessment

Yes in direction, but not by closing this issue now. The maintainable path is to land or supersede the open implementation work after maintainers choose the wizard-only versus resolver-level compatibility boundary.
