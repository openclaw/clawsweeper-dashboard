## Summary

Keep open. Current main still seeds the interactive add-agent workspace prompt from a resolver path that nests a new non-default agent under `agents.defaults.workspace`, while the docs describe a peer-level workspace; https://github.com/openclaw/openclaw/pull/71941 is still open with closing syntax and should be reviewed, refreshed, or superseded before this issue closes.

## What This Changes

The issue asks for `openclaw agents add <id>` to pre-fill the documented peer-level `workspace-<id>` path instead of `workspace/<id>` under the main workspace.

## Best Possible Solution

Review and land one compatibility-preserving fix that gives brand-new add-agent prompts the documented sibling workspace default while preserving explicit per-agent workspaces, existing-agent updates, state-dir isolation, and intentional shared-base configurations.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: setup stores `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and the resolver appends a non-default id while docs expect a sibling workspace.

## Solution Assessment

Yes, the best direction is a narrow compatibility-preserving fix through https://github.com/openclaw/openclaw/pull/71941 or an equivalent replacement, with regression coverage for the prompt default and the shared-base/default-agent boundary.
