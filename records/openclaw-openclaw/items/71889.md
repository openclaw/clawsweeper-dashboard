## Summary

Keep open. Current main still seeds the add-agent wizard from the runtime workspace resolver, which appends a new non-default agent id under `agents.defaults.workspace` even though the docs describe a peer-level workspace; #71941 has closing syntax for this issue and #71899 covers the same fix area.

## What This Changes

The issue asks for `agents add <id>` to pre-fill a peer-level `workspace-<id>` directory instead of nesting under the main `workspace/<id>` directory.

## Best Possible Solution

Review and land #71941 or an equivalent narrow fix that uses the documented peer-level default for brand-new agents while preserving explicit and custom shared-base workspace resolution.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: setup stores the main workspace in `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and the resolver returns `<defaults.workspace>/<id>` for a new non-default agent.

## Solution Assessment

Yes. A narrow wizard/default-helper fix is the maintainable path because it avoids blessing the nested default while preserving legacy and intentionally shared workspace-base configurations.

Status: needs_attention
