## Summary

Keep open. Current main still seeds the interactive add-agent workspace prompt from the resolver branch that appends a non-default agent id under `agents.defaults.workspace`, while the multi-agent docs still describe peer-level `workspace-<agentId>` workspaces. The linked implementation PRs should be reviewed or superseded rather than closing the issue before a fix merges.

## What This Changes

The issue asks for `openclaw agents add <id>` to prefill the documented peer-level workspace form `workspace-<id>` instead of nesting the new agent under the main workspace.

## Best Possible Solution

Land one narrow existing PR, or an equivalent maintainer patch, that changes the new-agent wizard default to the documented peer-level workspace for brand-new agents while preserving explicit and legacy configured workspace paths, with focused resolver/wizard tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The deterministic source path is enough to reproduce: setup can persist `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir(cfg, agentId)` as the prompt default, and the resolver appends the non-default id under that fallback.

## Solution Assessment

Yes. A focused wizard-default helper is the narrowest maintainable fix if it preserves existing explicit and legacy workspace paths; changing the runtime resolver wholesale could silently relocate existing agents.
