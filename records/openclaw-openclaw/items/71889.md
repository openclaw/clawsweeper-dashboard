## Summary

Current `main` still has the reported source path: setup stores the main workspace in `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and that resolver nests non-default agents under the fallback workspace. The issue should stay open because https://github.com/openclaw/openclaw/pull/71941 still has closing syntax for this issue, and related open PRs disagree on the compatibility boundary.

## What This Changes

The issue reports that the interactive add-agent wizard defaults new agent workspaces to a nested `workspace/<id>` path instead of the documented peer-level `workspace-<id>` path.

## Best Possible Solution

Review and land one compatibility-preserving fix that gives new `agents add` prompts the documented sibling workspace default while preserving explicit per-agent workspaces, existing-agent updates, state-dir isolation, and any intentionally shared-base configuration contract maintainers keep.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: setup stores `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and the resolver appends a non-default id while docs expect a sibling workspace.

## Solution Assessment

Yes in direction, but not by closing this issue now. The best path is to review, refresh, or supersede the open implementation PRs, especially https://github.com/openclaw/openclaw/pull/71941 because it has closing syntax for this issue.
