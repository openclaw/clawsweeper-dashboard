## Summary

Keep open. Current main still uses the resolver result as the interactive add-agent workspace prompt default, and that resolver still nests non-default agents under `agents.defaults.workspace`; the issue also has open implementation PRs with closing syntax, so it should remain open until one fix lands.

## What This Changes

The issue asks for `openclaw agents add <id>` to prefill a peer-level workspace such as `~/.openclaw/workspace-<id>` instead of nesting the new agent under the main workspace.

## Best Possible Solution

Land one narrow fix PR, or an equivalent maintainer patch, that changes the new-agent interactive default to a peer-level workspace while preserving explicit existing workspace paths and any intentionally configured shared-base behavior. Add focused resolver/wizard regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. The reproduction is deterministic from source: after setup persists `agents.defaults.workspace`, `agents add testbug` seeds the prompt from `resolveAgentWorkspaceDir(cfg, 'testbug')`, which currently resolves to `<workspace>/testbug`.

## Solution Assessment

Yes. The requested peer-level add-wizard default is the narrow maintainable direction, as long as the fix does not silently relocate existing agents or custom shared workspace bases.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because current defaults can place a secondary agent workspace under the main workspace, crossing OpenClaw's soft workspace boundary during recursive workspace operations.

Concerns:

- **[medium] Keep new agent workspaces outside the main workspace:** `src/agents/agent-scope-config.ts:172`
  - body: When the add wizard persists the accepted default, the resolver can place a secondary workspace at `<main workspace>/<agentId>`. Workspace skill paths are then rooted under the main workspace, so recursive reads, skill discovery, or deletes can include peer-agent files unexpectedly.
