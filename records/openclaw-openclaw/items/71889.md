## Summary

Keep open. Current main still seeds the interactive `agents add` workspace prompt from the resolver path that nests a new non-default agent under `agents.defaults.workspace`, while the docs describe a sibling `workspace-<agentId>` layout; an open closing PR, https://github.com/openclaw/openclaw/pull/71941, should be reviewed or superseded before this issue closes.

## What This Changes

The issue reports that `openclaw agents add <id>` defaults new agent workspaces to nested `workspace/<id>` paths instead of the documented peer-level `workspace-<id>` layout.

## Best Possible Solution

Land or supersede one compatibility-preserving fix that gives brand-new `agents add` prompts the documented sibling workspace default while preserving explicit per-agent workspaces, existing-agent updates, and any intentional shared-base configurations.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: setup stores `agents.defaults.workspace`, the interactive add wizard uses `resolveAgentWorkspaceDir` as the prompt default, and the resolver appends the new non-default id under that workspace.

## Solution Assessment

Unclear between the open PRs. The narrow wizard-only direction in https://github.com/openclaw/openclaw/pull/71941 is a plausible safe fix, but maintainers still need to choose it or a broader resolver fallback change from the related open PRs.
