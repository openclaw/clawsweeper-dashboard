## Summary

Keep open. Current main still seeds the `agents add` workspace prompt from the runtime resolver, which appends a new non-default agent id to `agents.defaults.workspace`; the issue is also paired with open implementation PRs, including #71941 with closing syntax.

## What This Changes

The issue reports that `agents add <id>` pre-fills a nested `~/.openclaw/workspace/<id>` workspace instead of the documented peer-level `~/.openclaw/workspace-<id>` path.

## Best Possible Solution

Review and land #71941 or an equivalent narrow fix that changes the new-agent wizard default while preserving legacy and custom runtime workspace resolution.

## Reproduction Assessment

Yes. With setup-created `agents.defaults.workspace` set to the main workspace and a new non-default agent id, current source deterministically resolves `<workspace>/<id>` and uses that as the wizard prompt default.

## Solution Assessment

Yes. A targeted wizard/default-helper fix is the safest path; changing the runtime resolver broadly could relocate existing agents or break intentionally shared workspace-base configurations.
