## Label Justifications

- `P2`: This is a normal-priority agents workspace default bug with a workaround, open fix PRs, and limited but real workspace-boundary impact.
- `impact:data-loss`: Accepting the nested default can put peer-agent workspaces under the main workspace, making broad cleanup of the main workspace delete peer data.
- `impact:security`: The reported layout can cause recursive main-workspace operations to read peer-agent files that users expected to keep separate.
- `impact:session-state`: The report specifically calls out peer agent memory, identity, and workspace state being reachable through the main workspace tree.

## Summary

Keep open: current main and the latest release still expose the nested `workspace/<id>` default through the add-agent wizard, while the docs still describe the sibling `workspace-<agentId>` layout. There are open implementation PRs for this same bug, so the issue should stay open until maintainers land or supersede one fix path.

## What This Changes

The issue asks for `openclaw agents add <id>` to prefill a sibling workspace path instead of nesting the new agent under the main workspace.

## Best Possible Solution

Land one compatibility-reviewed fix that gives brand-new non-default agents a sibling workspace default while preserving explicit per-agent workspaces, existing-agent updates, and any intentionally shared-base contracts maintainers choose to keep.

## Reproduction Assessment

Yes. Source inspection on current main shows setup writes `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and that resolver appends the non-default agent id while docs expect a sibling workspace.

## Solution Assessment

Yes in direction, but the issue should not close until one implementation path lands. Maintainers need to choose the wizard-only compatibility boundary or the broader resolver-level change and update focused coverage accordingly.
