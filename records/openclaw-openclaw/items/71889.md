## Summary

Keep open. Current main still uses the runtime workspace resolver as the `agents add` wizard default, and that resolver still nests non-default agents under `agents.defaults.workspace`; open PR #71941 has closing syntax and open PR #71899 also covers the bug, so this issue should remain open until one implementation lands or is superseded.

## What This Changes

The issue asks to change the interactive `openclaw agents add <id>` workspace default from `<main workspace>/<id>` to the documented peer-level `workspace-<id>` layout.

## Best Possible Solution

Land #71941 or an equivalent narrow fix that pre-fills peer-level workspaces for brand-new agents, preserves legacy/custom runtime resolution, adds focused tests, and includes a user-facing changelog entry.

## Reproduction Assessment

Yes. With `agents.defaults.workspace` populated by setup and a non-default agent id, the current resolver deterministically returns `<fallback>/<id>`, and the add-agent wizard uses that value as the prompt default while docs expect `workspace-<agentId>`.

## Solution Assessment

Yes, the best path is a narrow new-agent wizard/default-helper fix. A broad runtime resolver change may be too risky without maintainer review because existing nested and custom shared-base configurations currently rely on that fallback.
