## Label Justifications

- `P2`: This is a real multi-agent workspace-default bug with limited blast radius and open targeted fix paths, but it is not a crash loop or unusable core runtime.
- `impact:data-loss`: Accepting the nested default can put peer-agent workspaces under the main workspace, so recursive cleanup of the main workspace can delete peer-agent data.
- `impact:security`: The nested layout can let recursive main-workspace operations include peer-agent files under OpenClaw's soft workspace-boundary model.

## Summary

Keep open: current main and latest release still prefill `agents add` from the resolver path that nests a non-default agent under `agents.defaults.workspace`, while docs still describe peer-level `workspace-<agentId>` defaults. The issue should not close while an open closing PR and an overlapping resolver-level PR still need maintainer resolution.

## What This Changes

The issue requests that `agents add <id>` pre-fill `~/.openclaw/workspace-<id>` instead of `~/.openclaw/workspace/<id>`.

## Best Possible Solution

Land one compatibility-reviewed fix, likely by refreshing or superseding the open PRs, so new agent defaults use the documented sibling workspace while explicit workspaces, existing-agent updates, and intentional shared-base layouts remain stable.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.19` shows setup stores `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and that resolver returns `<defaults.workspace>/<id>` for non-default agents while docs expect the sibling form.

## Solution Assessment

Yes in direction, but not by closing this issue now. The maintainable path is to land or supersede the open implementation work after maintainers choose the wizard-only versus resolver-level compatibility boundary.
