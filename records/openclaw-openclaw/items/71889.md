## Summary

Keep open. Current main still uses the runtime workspace resolver as the interactive add-agent prompt default, and that resolver still nests non-default agents under `agents.defaults.workspace` when setup has stored the main workspace there; the docs still describe peer-level `workspace-<agentId>` workspaces. Because open PR #71941 includes closing syntax and open PR #71899 covers the same bug, the issue should remain open until maintainers land or supersede one implementation.

## What This Changes

The issue requests changing the interactive `agents add` wizard default from nested `workspace/<id>` to peer-level `workspace-<id>` to match the multi-agent docs.

## Best Possible Solution

Land one narrow existing PR, or an equivalent maintainer patch, that changes the new-agent wizard prefill to the documented peer-level workspace for brand-new agents while preserving explicit per-agent workspaces, existing nested legacy paths, and intentional custom shared workspace bases; include focused resolver/wizard coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. The deterministic source path is enough to reproduce: setup can persist `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir(cfg, agentId)` as the prompt default, and the resolver appends the non-default id under that fallback while docs expect `workspace-<agentId>`.

## Solution Assessment

Yes, if scoped to the wizard default for brand-new agents. A broader runtime resolver change may be acceptable only with maintainer approval because it could alter legacy agents or custom shared-root configurations.
