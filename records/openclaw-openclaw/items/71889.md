## Label Justifications

- `P2`: This is a normal-priority agents workspace default bug with a documented workaround and limited but real workspace-boundary impact.
- `impact:data-loss`: Accepting the nested default can place peer workspaces under the main workspace, making recursive deletion of the main workspace delete peer-agent files too.
- `impact:security`: The nested default can let main-workspace recursive operations traverse peer-agent workspace files under OpenClaw's soft workspace boundary.

## Summary

Keep open: current main and the latest stable tag still prefill `agents add` from a resolver fallback that nests non-default agents under `agents.defaults.workspace`, while the docs still describe sibling workspaces. The issue is also tied to open implementation PRs, so it should not be closed before one lands or is explicitly superseded.

## What This Changes

Fix the `agents add <id>` wizard so new agents default to the documented sibling workspace path instead of nesting under the main workspace.

## Best Possible Solution

Land or supersede the open fix PRs with tests and real CLI proof, while preserving explicit per-agent workspaces and existing shared-base configs unless maintainers intentionally change the resolver contract.

## Reproduction Assessment

Yes. Source inspection on current main shows setup writes `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and that resolver appends the non-default agent id while docs expect a sibling workspace.

## Solution Assessment

Yes in direction, but not by closing this issue now. The final fix should either preserve runtime compatibility with a wizard-only default or make a deliberate resolver-level contract change after maintainer review.
