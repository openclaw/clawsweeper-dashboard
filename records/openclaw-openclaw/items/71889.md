## Summary

Keep open. Current main still seeds the interactive add-agent workspace prompt from the runtime resolver that nests non-default agents under `agents.defaults.workspace`, while the multi-agent docs still describe peer-level workspaces; the linked open PRs are implementation candidates and should be reviewed rather than closing the issue before merge.

## What This Changes

The issue asks for the interactive `openclaw agents add <id>` workspace prompt to default to the documented peer-level `workspace-<agentId>` layout instead of nesting under the main workspace.

## Best Possible Solution

Land one narrow fix that changes the new-agent wizard default to a peer-level workspace, preserves explicit and legacy configured workspace paths, adds focused resolver/wizard regression tests, and includes a user-facing changelog entry.

## Reproduction Assessment

Yes. The source path is deterministic on current main: setup can persist `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir(cfg, agentId)` as the prompt default, and the resolver appends a non-default agent id under that fallback.

## Solution Assessment

Yes. Changing the add-agent wizard default for new agents while leaving explicit and legacy runtime workspace resolution intact is the narrow maintainable fix; documenting the nested layout instead would preserve the reported data-loss and workspace-leakage footgun.
