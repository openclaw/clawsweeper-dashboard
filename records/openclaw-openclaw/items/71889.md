## Label Justifications

- `P2`: This is a real multi-agent workspace-default bug with limited blast radius and open targeted fix paths, but it is not a crash loop or unusable core runtime.
- `impact:data-loss`: Accepting the nested default can place peer-agent workspaces under the main workspace, so recursive cleanup of the main workspace can delete peer-agent data.
- `impact:security`: The nested layout can let recursive main-workspace operations include peer-agent files under OpenClaw's soft workspace-boundary model.

## Summary

Keep open: current main and `v2026.5.19` still have the nested default path, and the issue has an open closing PR plus an overlapping resolver-level PR that need maintainer resolution before the issue can close.

## What This Changes

The issue reports that `openclaw agents add <id>` defaults the workspace prompt to `<main workspace>/<id>` instead of the documented sibling `workspace-<id>` path.

## Best Possible Solution

Land or supersede one compatibility-reviewed fix that makes new-agent defaults sibling-level while preserving explicit per-agent workspaces, existing-agent updates, and intentional shared-base layouts.

## Reproduction Assessment

Yes. Source inspection on current main shows setup stores `agents.defaults.workspace`, `agents add` feeds `resolveAgentWorkspaceDir` into the prompt, and the resolver appends non-default ids under the default workspace while docs expect a sibling path.

## Solution Assessment

Unclear. The sibling default is the right direction, but maintainers still need to choose the wizard-only prefill boundary versus a broader resolver-level fallback change.
