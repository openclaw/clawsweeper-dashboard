## Summary

Keep open: current main still has the reported source path, the docs still describe the peer-level workspace convention, and https://github.com/openclaw/openclaw/pull/71941 remains an open implementation PR with closing syntax for this issue.

## What This Changes

The issue asks for the `agents add` wizard to pre-fill new agents with the documented peer-level `workspace-<id>` path instead of nesting under the main `workspace/<id>` directory.

## Best Possible Solution

Land or supersede the existing compatibility-preserving fix so only brand-new wizard defaults move to the documented sibling workspace layout while explicit workspaces, existing-agent updates, and intentional shared-base configs keep working.

## Reproduction Assessment

Yes. Source inspection on current main shows setup writes `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and that resolver appends the non-default agent id while docs expect a sibling workspace.

## Solution Assessment

Yes in direction, but not by closing this issue now. A wizard-only default fix for new agents is the narrow maintainable path, and the linked implementation PR should be reviewed, refreshed, landed, or superseded.
