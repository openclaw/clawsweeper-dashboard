## Summary

Keep open. Current main still drives the add-agent wizard default through the resolver path that nests non-default agents under `agents.defaults.workspace`, while docs still describe peer-level workspaces; the open PRs with closing syntax are implementation candidates and should be reviewed rather than closing the issue before merge.

## What This Changes

The issue asks for the `agents add <id>` wizard to default new agent workspaces to the documented peer-level `workspace-<id>` path instead of nesting under the main `workspace/<id>` directory.

## Best Possible Solution

Land one narrow existing PR, or an equivalent maintainer patch, that changes only the new-agent wizard default to a peer-level workspace while preserving explicit and legacy configured workspace paths, with focused resolver/wizard tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The source path is deterministic on current main: setup can persist `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir(cfg, agentId)` as the prompt default, and the resolver appends the non-default agent id under that fallback.

## Solution Assessment

Yes. The requested peer-level wizard default is the narrow maintainable fix, provided runtime resolution does not silently relocate existing agents with explicit or legacy nested workspace paths.
