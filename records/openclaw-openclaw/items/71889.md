## Label Justifications

- `P2`: This is a real multi-agent workspace-default bug with limited blast radius and an open targeted fix path, but it is not a crash loop or core-runtime outage.
- `impact:data-loss`: Accepting the nested default can put peer-agent workspaces under the main workspace, making common recursive cleanup of the main workspace delete peer-agent data.
- `impact:security`: The nested layout can let recursive main-workspace operations include peer-agent files under OpenClaw's soft workspace-boundary model.

## Summary

Keep open: current main and the latest release still seed the `agents add` workspace prompt from the runtime resolver that nests non-default agents under `agents.defaults.workspace`, while the docs still describe peer-level `workspace-<agentId>` defaults. An open implementation PR with closing syntax already targets this issue, so the issue should stay open until that PR lands or is superseded.

## What This Changes

Fix the `agents add` wizard default workspace prefill so new agents use the documented sibling `workspace-<agentId>` path instead of nesting under the main workspace.

## Best Possible Solution

Land or supersede the existing compatibility-preserving fix so new `agents add` defaults use the documented sibling workspace while explicit workspaces, existing-agent updates, and custom shared-base configs stay stable.

## Reproduction Assessment

Yes. Source inspection on current main shows setup writes `agents.defaults.workspace`, `agents add` passes `resolveAgentWorkspaceDir` as the prompt default, and that resolver returns `<defaults.workspace>/<id>` for non-default agents while docs expect the sibling form.

## Solution Assessment

Yes in direction, but not by closing this issue now. A wizard-only default fix for brand-new agents that preserves explicit workspaces, existing-agent updates, and custom shared roots is the narrow maintainable path; https://github.com/openclaw/openclaw/pull/71941 should be reviewed, refreshed, landed, or superseded.
