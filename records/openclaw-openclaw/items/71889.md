## Summary

Keep open. Current main still seeds the interactive `agents add` workspace prompt from the runtime resolver, which nests a new non-default agent below `agents.defaults.workspace`; open PR #71941 has closing syntax for this issue and #71899 covers the same fix area, so this should wait for one implementation to land.

## What This Changes

Change the interactive `agents add <id>` wizard so its default workspace path is the documented peer-level `workspace-<id>` form rather than `workspace/<id>` under the main workspace.

## Best Possible Solution

Land #71941 or an equivalent narrow wizard-default fix that keeps runtime resolution compatible while new agents get the documented peer-level workspace default.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: setup writes `agents.defaults.workspace`, the resolver appends the new non-default id beneath it, and the wizard uses that resolver result as the prompt default.

## Solution Assessment

Yes. A narrow wizard/default-helper fix is the maintainable direction; updating docs to bless the nested default would preserve the workspace-boundary and reset-footgun behavior, while a broad runtime resolver change risks moving existing agents.
