## Summary

Keep open. Current main still derives the interactive add-agent workspace prompt from the resolver path that nests a new non-default agent under `agents.defaults.workspace`, while the docs describe a peer-level workspace; open fix PRs, especially https://github.com/openclaw/openclaw/pull/71941, should be reviewed or superseded before closing this issue.

## What This Changes

The issue requests that `openclaw agents add <id>` pre-fill the documented peer-level `~/.openclaw/workspace-<id>` workspace instead of `~/.openclaw/workspace/<id>`.

## Best Possible Solution

Review and land one compatibility-preserving fix that gives newly added agents the documented sibling workspace default while preserving explicit per-agent workspaces and any intentional shared-base configurations.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: setup stores the main workspace in `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and the resolver returns `<defaults.workspace>/<id>` for a new non-default agent.

## Solution Assessment

Yes. The best direction is a narrow compatibility-preserving fix through https://github.com/openclaw/openclaw/pull/71941 or an equivalent replacement, with regression coverage for the prompt default and the shared-base/default-agent boundary.
