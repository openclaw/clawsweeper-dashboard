## Summary

Keep open: current main still only reports the count of truncated skills, and this PR targets a real skills discoverability gap, but the latest diff has blocking correctness issues around prompt budgeting and recoverable identifiers.

## What This Changes

The PR modifies workspace skills prompt truncation to append sanitized names of omitted skills and a `openclaw skills info <name>` recovery hint.

## Best Possible Solution

Land an equivalent patch that keeps the final prompt within the configured cap, exposes exact recoverable skill identifiers as bounded prompt data, adds focused regression tests, and includes the required changelog entry.

## Reproduction Assessment

Yes, at source level. Building a skills prompt with `maxSkillsInPrompt` below the eligible count or a tight `maxSkillsPromptChars` follows the current-main path that emits only `included N of M`, and the PR’s remaining bugs follow directly from appending names after limit selection and rewriting exact lookup identifiers.

## Solution Assessment

No. Listing omitted skills is the right narrow direction, but this diff does not keep the final prompt inside the configured cap and does not preserve the exact identifiers needed by the advertised lookup command.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Budget omitted-skill guidance before emitting the prompt:** `src/agents/skills/workspace.ts:916-919`
  - body: The omitted-name sentence is appended after `applySkillsPromptLimits()` has already selected a catalog sized against `maxSkillsPromptChars`. With many omitted skills, the final snapshot can exceed the configured cap, so reserve or trim this note inside the final budget.
  - confidence: 0.93
- **[P2] Preserve exact lookup keys for omitted skills:** `src/agents/skills/workspace.ts:826`
  - body: `sanitizeSkillName()` removes characters from the advertised name, but `openclaw skills info <name>` resolves by exact raw name or `skillKey`. An omitted skill named `foo<bar` would be shown as `foobar`, making the suggested lookup fail.
  - confidence: 0.89

  - body: This changes runtime prompt behavior for users, but the latest diff only touches the skills workspace code. Add a single-line `CHANGELOG.md` entry under the active fixes section before merge.
