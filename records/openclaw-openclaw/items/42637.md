Full review comments:

- **[P2] Sample across the omitted skill range:** `src/skills/loading/workspace.ts:1310-1312`
  - body: When the omitted set is larger than the note budget, this formatter only ever emits `skillKeys.slice(0, included)`. That still hides late omitted skills from both `<available_skills>` and `<omitted_skill_keys>`, so a large alphabetically sorted workspace can still lose the exact kind of late skill this PR is meant to recover. Please sample from across the omitted range, or otherwise guarantee late omitted keys can be discovered.
  - confidence: 0.9
- **[P2] Prefer lookup keys over audit copy under tight budgets:** `src/skills/loading/workspace.ts:1417-1423`
  - body: `buildTruncationNote()` only tries the omitted-key segment after the full audit sentence fits, and falls back to count-only text when that base does not fit. Under a tight `maxSkillsPromptChars`, the prompt can therefore say skills were truncated but provide no exact key even when a shorter key-first recovery note could fit one. Since lookup keys are the recovery path, reserve budget for them before `skills check` copy.
  - confidence: 0.78
