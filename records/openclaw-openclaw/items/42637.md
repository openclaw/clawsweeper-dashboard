## Best Possible Solution

Keep the exact JSON lookup-key approach, but reserve minimal recovery budget, expose non-prefix omitted keys under the cap, and require refreshed current-head terminal proof before merge.

## Reproduction Assessment

## Solution Assessment

No. The JSON exact-key direction is maintainable, but the current implementation should reserve minimal omitted-key recovery and surface non-prefix omitted keys before it is the best fix.

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reserve room for omitted-key recovery under tight budgets:** `src/agents/skills/workspace.ts:1140-1142`
  - body: When the notice budget is too small for the full base warning, this branch returns a shorter count-only warning and never tries to include `<omitted_skill_keys>`. That means a tight `maxSkillsPromptChars` can still drop skills without any exact lookup key, so the fix should reduce the skill body or shorten guidance before giving up on recovery keys.
  - confidence: 0.88
- **[P2] Include non-prefix omitted skill keys:** `src/agents/skills/workspace.ts:1033-1035`
  - body: The omitted-key segment only serializes `skillKeys.slice(0, included)`, so once the payload cap is hit it advertises only the earliest omitted skills. In large alphabetically sorted skill sets, later omitted skills remain undiscoverable behind `(+N more)`, which preserves the failure this change is meant to address for those skills.
  - confidence: 0.86
