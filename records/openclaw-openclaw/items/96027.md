## Summary

Keep open for maintainer review/landing: current main and the latest release still leave the two Ollama discovery success bodies unbounded, while this PR uses the existing provider HTTP bounded JSON helper with focused proof and no blocking correctness findings. The only maintainer-visible risk is the intentional 16 MiB cap changing behavior for an unusually large but legitimate Ollama discovery response.

## What This Changes

The PR routes Ollama `/api/tags` and `/api/show` discovery JSON responses through the shared bounded provider JSON reader and adds an oversized-stream regression test.

## Best Possible Solution

Land the plugin-local bounded-reader change after maintainers accept the 16 MiB Ollama discovery cap, keeping fail-soft behavior and avoiding a new Ollama-specific parsing path.

## Reproduction Assessment

Yes by source path: current main uses unbounded `response.json()` for successful `/api/tags` and `/api/show` responses, and those functions are called from Ollama setup/model discovery. I did not execute a local repro in this read-only review, but the PR body provides live-output proof from a real local HTTP server and the code path is direct.

## Solution Assessment

Yes: using `readProviderJsonResponse` from `openclaw/plugin-sdk/provider-http` is the narrow owner-boundary fix and follows sibling provider patterns. The only tradeoff is the inherited 16 MiB cap, which should be an explicit maintainer acceptance rather than a code-review blocker.
