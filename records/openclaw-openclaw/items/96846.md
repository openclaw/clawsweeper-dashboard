## Summary

Keep open: current main and v2026.6.10 still have a Telegram album prompt-context hydration gap where album media is collected for the current turn, but only the primary album message id is mapped to a model-readable `media://inbound/...` path, leaving sibling album cache entries able to surface as raw `telegram:file/...` refs.

## What This Changes

The issue reports that Telegram media-group albums pass only one photo as model-readable image content while sibling album photos surface as raw `telegram:file/...` references.

## Best Possible Solution

Keep the existing album buffer and current-turn media payload, then add message-id-aware hydration for each resolved album item so prompt context never exposes sibling photos as raw Telegram file refs.

## Reproduction Assessment

Yes at source level: current main records album messages and collects all media, but prompt-context hydration only maps the primary message id while sibling cache entries retain raw `telegram:file/...` refs. I did not run a live Telegram transport reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to carry each resolved album item’s originating message id into the existing Telegram prompt-context hydration map, not to add a gateway batching feature or new config option.
