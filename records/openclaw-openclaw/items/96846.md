## Label Justifications

- `P1`: A common Telegram album workflow can leave successfully received photos unreadable to the model for real users.
- `impact:message-loss`: Album photos are received by the channel but sibling entries can reach model-facing context as unusable raw references instead of usable media.

## Summary

Keep open: current main and v2026.6.10 still have the Telegram album prompt-context hydration gap, and the active closing fix is https://github.com/openclaw/openclaw/pull/97045 but it is still open and unmerged.

## What This Changes

Fix Telegram inbound album handling so every photo reaches model-facing context as usable media instead of raw `telegram:file/...` references.

## Best Possible Solution

Land the focused Telegram plugin fix in https://github.com/openclaw/openclaw/pull/97045, then close this issue as implemented and treat the narrower draft as superseded if maintainers accept that PR.

## Reproduction Assessment

Yes, at source level: current main buffers the album and collects all media, but prompt-context hydration only maps the first media item to the primary message id while cached sibling messages can still expose raw `telegram:file/...` refs. I did not run a live Telegram transport repro in this read-only review.

## Solution Assessment

Yes, the best fix is the current narrow Telegram plugin handoff: carry each successfully resolved album item’s source message id into prompt-context hydration and keep failed siblings out of model-facing context. A gateway batching feature or new config option would be the wrong layer for this bug.
