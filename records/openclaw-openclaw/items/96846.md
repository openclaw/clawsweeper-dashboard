## Label Justifications

- `P1`: This breaks a common Telegram workflow where multi-photo albums can leave model-facing image content unreadable for real users.
- `impact:message-loss`: Album photos are received by the Telegram channel but sibling photos can be represented to the model as unreadable raw refs instead of usable media.

## Summary

Keep open: current main and the latest release still have the Telegram album prompt-context gap, while an open live-proven fix PR already uses closing syntax for this issue and should be reviewed before the issue closes.

## What This Changes

Telegram albums should expose every successfully received photo to model-facing context as readable media instead of leaving sibling photos as raw `telegram:file/...` references.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/97045, then close this issue only after the chosen fix reaches main and supersede the weaker overlapping draft.

## Reproduction Assessment

Yes, at source level: current main collects album media but only maps the first media item into prompt context while cached siblings can fall back to `telegram:file/...`. I did not run the live Telegram transport path in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow Telegram plugin handoff repair that carries successful album sibling message ids into prompt-context media hydration. The open candidate PR follows that boundary and adds partial-download exclusion behavior.
