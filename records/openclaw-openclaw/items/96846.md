## Label Justifications

- `P1`: This breaks a common Telegram channel workflow where multi-photo albums lose model-visible image content for real users.
- `impact:message-loss`: Album photos are received by the channel but can be represented to the model as unreadable raw references instead of usable media.

## Summary

Keep open: current main and the latest release still have a source-level Telegram album context gap where all album media is collected for the turn, but only the primary message id is hydrated for model-facing prompt context, leaving sibling album cache entries able to surface as raw `telegram:file/...` refs.

## What This Changes

Telegram media group albums should pass every photo to the model as usable image media instead of leaving non-primary photos as unreadable `telegram:file/...` references.

## Best Possible Solution

Keep the existing Telegram album buffer, but carry each resolved album item’s originating message id into prompt-context hydration so every album photo gets a `media://inbound/...` path and no sibling photo leaks as `telegram:file/...`.

## Reproduction Assessment

Yes at source level: current main records album messages and collects all media, but prompt-context hydration only maps the first media item to the primary message id while cached sibling messages can still expose raw `telegram:file/...` refs. I did not run a live Telegram transport reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is in the Telegram plugin’s existing media-group to prompt-context handoff, not a new gateway batching feature or a new config option.
