## Label Justifications

- `P1`: The report describes a broken Telegram channel workflow where common multi-photo albums lose model-visible image content for real users.
- `impact:message-loss`: Album photos are received by the channel but can be represented to the model as unreadable raw references instead of usable image media.

## Summary

Keep open: current main already buffers Telegram albums and carries all resolved media into the current turn, but the model-facing conversation-context path can still project sibling album messages as raw `telegram:file/...` refs because only the primary album message ID is hydrated with one media path.

## What This Changes

Fix Telegram inbound media-group handling so every album photo reaches model-facing context as resolved image media instead of leaving non-primary album items as `telegram:file/...` references.

## Best Possible Solution

Preserve the existing Telegram album buffer, but carry each resolved album item’s message ID into prompt-context hydration so every cached album message gets a `media://inbound/...` path and no non-primary photo leaks as `telegram:file/...`.

## Reproduction Assessment

Yes at source level: current main records each Telegram album update in the cache, but prompt-context hydration only associates the first resolved album media item with the primary message ID. I did not run live Telegram transport in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is in the Telegram plugin’s existing media-group and prompt-context handoff, not a new gateway batching feature or new config option.
