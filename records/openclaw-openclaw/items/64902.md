## Summary

## What This Changes

Request a structured, sanitized session transcript event that records the primary model failure that triggers fallback before the subsequent `model-snapshot` entry.

## Best Possible Solution

Persist one bounded, sanitized fallback audit custom transcript event at the fallback/session boundary before `model-snapshot`, reusing existing fallback metadata and leaving routing, retry, cooldown, and user-visible notice policy unchanged.

Yes, at source level. A primary failure followed by fallback success flows through `recordFailedCandidateAttempt`, `logModelFallbackDecision`, and `onFallbackStep`, while `model-snapshot` is appended later in replay-history with no fallback custom-entry path.

## Solution Assessment

Yes. The narrow maintainable fix is to persist a sanitized transcript audit record from existing fallback-step or failed-attempt metadata; changing fallback selection, cooldown policy, or visible channel notices would solve different problems.
