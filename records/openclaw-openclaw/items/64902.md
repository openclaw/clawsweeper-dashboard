## Summary

Keep this open. Current main has adjacent fallback observability through attempt arrays, structured subsystem logs, trajectory sidecars, fallback notice state, and docs, but it still does not persist the requested ordered session JSONL audit event before the fallback `model-snapshot`.

## What This Changes

The issue requests a structured persistent session transcript event recording the failed primary model attempt and fallback target when model fallback succeeds.

## Best Possible Solution

Add a small sanitized transcript audit event at the fallback/reply-runner boundary that records the failed selected provider/model, chosen fallback provider/model, reason/class, status/code when available, chain position, and bounded sanitized or hashed error detail before the fallback model snapshot or retry, while preserving existing routing, lifecycle events, status state, trajectory sidecars, and user-visible notices.

## Reproduction Assessment

Yes. A focused reproduction can mock a primary failure followed by successful fallback, then inspect the session manager entries; current source inspection shows attempts/logs/notices plus the later `model-snapshot`, not the requested ordered transcript audit entry.

## Solution Assessment

Yes. The narrowest maintainable fix is a sanitized session custom event wired to existing fallback step data; changing fallback eligibility, model routing, or user notice behavior would be broader than this issue requires.
