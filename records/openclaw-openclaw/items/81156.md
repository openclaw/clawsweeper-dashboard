Keep open: current main and the latest release already implement the proposed count-field inversion, but the fresh report includes live MiniMax dashboard/endpoint disagreement that is not resolved by changing `current_interval_usage_count` semantics alone.

## What This Changes

The issue reports that MiniMax quota counts are being interpreted with the wrong used-versus-remaining semantics in Telegram usage display.

## Best Possible Solution

Verify MiniMax's current token-plan/coding-plan endpoint and auth contract with live credentials, then update only the fetch URL/parser/tests needed to make OpenClaw match the provider's authoritative quota source.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the reported `model_remains` shape deterministically flows through the MiniMax parser, but the dashboard-versus-endpoint mismatch needs a live MiniMax contract check.

## Solution Assessment

No: adding `current_interval_usage_count` to `REMAINING_KEYS` is already on main and in v2026.5.7. The narrower maintainable path is to confirm which MiniMax endpoint/auth mode is authoritative before changing the shipped parser behavior.
