## Summary

Keep open: current main and v2026.6.10 still do not cover the latest MiniMax Token Plan shape, where remaining-percent fields are ignored and `model_remains` can collapse general/weekly/video quota data into a wrong `0% left` status line.

## What This Changes

The issue asks OpenClaw to fix MiniMax provider usage parsing so status output matches actual remaining quota instead of showing inverted or dropped MiniMax usage windows.

## Best Possible Solution

Keep this as the canonical tracker, verify the live MiniMax Token Plan response, then land a parser-level fix with tests for remaining-percent aliases, mixed model_remains rows, and the shared `% left` display contract.

## Reproduction Assessment

Yes, source-reproducible: current main ignores the reported `current_interval_remaining_percent` and `current_weekly_remaining_percent` fields, can choose a non-chat `model_remains` count row, and then formats the resulting normalized value as `% left`. The June 26, 2026 live comment corroborates the same symptom on v2026.6.10, but this review did not have live MiniMax credentials.

## Solution Assessment

No final fix is established yet. The best path is a MiniMax parser-level repair backed by live API contract proof, not a formatter-only `% used` exception or a global reversal of all `current_interval_usage_count` semantics.
