## Summary

Keep open: current main and the latest release already preserve the original MiniMax remaining-count interpretation, but the newer live evidence still exposes unsupported remaining-percent fields and a model_remains selection path that can drop the general/weekly quota rows.

## What This Changes

The issue reports that MiniMax usage counts and remaining percentages are interpreted with the wrong semantics, causing incorrect `% left` usage status for MiniMax Token Plan users.

## Best Possible Solution

Keep this as the canonical tracker until maintainers verify the live MiniMax Token Plan response and land a parser/test update for remaining-percent aliases and model_remains window selection while preserving the shared `% left` display contract.

## Reproduction Assessment

Yes, source-reproducible: current main does not recognize the reported current_interval_remaining_percent/current_weekly_remaining_percent fields, and model_remains selection can choose a non-chat count row when the chat/general row lacks current_interval_total_count. Live same-account MiniMax verification is still missing.

## Solution Assessment

No final fix is established yet. The best direction is a MiniMax parser-level repair with live contract proof, not reversing current_interval_usage_count semantics or adding a MiniMax-only `% used` formatter exception.
