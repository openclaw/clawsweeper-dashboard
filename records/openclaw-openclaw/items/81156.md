## Summary

Keep open: current main and the latest release already preserve MiniMax remaining-count semantics, but the fresh live evidence adds remaining-percent and model_remains selection behavior that current parser/tests do not fully cover.

## What This Changes

The issue asks OpenClaw to correct MiniMax usage-status math when MiniMax token-plan fields are interpreted as used versus remaining quota.

## Best Possible Solution

Keep this canonical issue open until maintainers verify the live token_plan contract and land a parser/test update for the reported remaining-percent and model_remains windows without undoing proven remaining-count semantics.

## Reproduction Assessment

Yes, source-reproducible: the latest reported remaining-percent field names are not recognized, and current model_remains selection can fall back to a count-based video entry when the chat entry lacks current_interval_total_count. A live MiniMax account was not available in this read-only review.

## Solution Assessment

Unclear until live contract proof: current main already fixes the original remaining-count interpretation, but the likely remaining fix is an endpoint-specific parser/test update for the newer remaining-percent and window-selection shape rather than a global count semantic reversal.
