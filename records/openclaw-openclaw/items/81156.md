## Summary

Keep open: current main and v2026.5.7 still poll MiniMax `/v1/token_plan/remains` while treating `current_interval_usage_count` as remaining quota, and the reporter supplied live token-plan/coding-plan payloads that conflict with the dashboard and the older coding-plan assumption.

## What This Changes

The issue reports that MiniMax token-plan usage counts are interpreted with the wrong used-versus-remaining semantics, causing the status display to show the wrong `% left` value.

## Best Possible Solution

Verify `/v1/token_plan/remains` and `/v1/api/openplatform/coding_plan/remains` with real MiniMax OAuth/token-plan credentials against the dashboard, then encode endpoint-specific handling or endpoint selection with regression tests.

## Reproduction Assessment

No high-confidence live reproduction is available here. The supplied payload has a clear current-main source path, but confirming the dashboard mismatch requires MiniMax token-plan credentials or upstream confirmation for the token-plan endpoint.

## Solution Assessment

Unclear. The suggested blanket remaining-count interpretation is already present on current main, so the safer solution is endpoint-specific verification and tests before changing parser semantics or switching endpoints.
