## Summary

Keep open: this draft, maintainer-labeled PR is still merge-dirty and the latest head still introduces approval prompt integrity regressions around existing command analysis, sanitized highlight offsets, carrier inline-eval coverage, and generated risk text.

## What This Changes

The PR adds parser-backed exec approval risk hints and highlights through agent approval registration, Gateway schema/events, forwarded messages, view models, Control UI rendering, tests, changelog, and an `.npmrc` registry pin.

## Best Possible Solution

Rebase onto current main, preserve existing commandAnalysis, map highlights onto sanitized display text, keep carrier inline-eval coverage, generate useful fallback risk lines, then rerun focused approval and UI checks.

## Reproduction Assessment

Yes. Source inspection of current main and PR head is enough to reproduce the review findings: compare the current commandAnalysis request payload/test with the PR request payload, then inspect the raw-offset, carrier-detection, and empty-line formatter paths.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best mergeable solution until it preserves existing approval-analysis behavior and keeps risk display accurate across UI and forwarded approval surfaces.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve command analysis in approval requests:** `src/gateway/server-methods/exec-approval.ts:257-258`
  - body: Current main stores `commandAnalysis` on exec approval requests and forwarded consumers use its `warningLines`. This request object adds the new explanation fields but omits `commandAnalysis`, so merging would drop existing inline-eval warning payloads.
  - confidence: 0.93
- **[P2] Remap highlights onto sanitized command text:** `src/gateway/server-methods/exec-approval.ts:258`
  - body: The highlights are computed from raw command text, but the Gateway stores sanitized display text and the UI slices that string by the supplied offsets. Redaction, invisible escaping, or truncation before a risky fragment can highlight the wrong text or hide the intended risk.
  - confidence: 0.9
- **[P2] Keep carrier inline-eval risks for find and xargs:** `src/infra/command-explainer/extract.ts:1033-1036`
  - body: This branch disables inline-eval detection for `find` and `xargs`, while the recursive carrier parser only covers shell-wrapper payloads. Direct carriers such as `find -exec python -c` or `xargs node -e` would lose the inline-code risk line and only show the generic carrier warning.
  - confidence: 0.86
- **[P2] Generate risk lines for non-highlight approval surfaces:** `src/infra/command-explainer/format.ts:72-73`
  - body: `formatCommandExplanationLines` always returns an empty array, so automatically generated approvals never populate `commandExplanationLines`. Forwarded approval messages and clients that cannot render highlights will not receive the parser-backed risk text this PR is trying to surface.
  - confidence: 0.87
