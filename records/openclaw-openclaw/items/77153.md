## Summary

Keep this PR open: it is a draft, maintainer-labeled, merge-dirty feature PR, and the latest head still has approval-prompt regressions that should be fixed before merge.

## What This Changes

The branch adds parser-backed command risk lines/highlights to host exec approval registration, gateway payloads, forwarded approval messages, Control UI rendering, tests, changelog, and a root `.npmrc` registry pin.

## Best Possible Solution

Keep the feature on the author branch, rebase onto current main, and render both targeted highlights and concise risk lines while preserving sanitized command-analysis behavior across UI and forwarded approvals.

## Reproduction Assessment

Yes. Source inspection of PR head shows generated approvals can carry both lines and highlights, while the Web modal suppresses lines whenever highlights exist; the xargs parser gap is also visible from the added option-scanning code.

## Solution Assessment

No. The feature direction is reasonable and previous commandAnalysis/sanitization blockers are mostly addressed, but the current implementation should show the plain-language risks alongside highlights and parse common xargs option-value forms before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Show risk lines alongside highlights:** `ui/src/ui/views/exec-approval.ts:86-89`
  - body: Generated approval explanations can include both `commandExplanationHighlights` and `commandExplanationLines`, but this branch passes `undefined` to `renderCommandExplanation` whenever any highlight exists. In the normal generated path, risky commands therefore show colored spans without the `Risks:` bullets shown in the PR proof and needed to explain why the highlighted fragments matter.
  - confidence: 0.9
- **[P2] Skip xargs option values before parsing payload:** `src/infra/command-explainer/extract.ts:923-929`
  - body: `xargsCommandArgv` treats the first non-option token as the carried command after skipping only a few options with values. Common forms like `xargs -n 1 sh -c ...` will start parsing at `1`, so the recursive shell-wrapper parser misses the real `sh -c` payload and its nested risk details.
  - confidence: 0.82
