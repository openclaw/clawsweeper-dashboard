## Summary

Keep open: this draft, maintainer-labeled PR is a useful feature direction, but current source review finds merge-blocking issues around preserving existing approval analysis, highlighting sanitized command text, and retaining direct carrier inline-eval risk detection.

## What This Changes

The PR adds parser-backed exec approval risk lines and command highlights across host approval registration, Gateway payloads, forwarding, view models, Web UI rendering, command-carrier parsing, tests, changelog, and an npm registry pin.

## Best Possible Solution

Rebase onto current main, preserve the existing commandAnalysis contract, remap or recompute highlight spans against sanitized display text, and keep carrier inline-eval coverage before rerunning focused approval/UI checks.

## Reproduction Assessment

Yes. Source inspection of current main plus PR head shows the commandAnalysis preservation gap, raw parser offsets being applied to sanitized display text, and find/xargs direct inline-eval detection being skipped.

## Solution Assessment

No. The feature direction is reasonable, but this is not the best mergeable solution until it preserves current-main analysis behavior, maps highlights to the rendered command, and keeps direct carrier inline-eval warnings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve command analysis in approval requests:** `src/gateway/server-methods/exec-approval.ts:256-257`
  - body: Current main attaches commandAnalysis to exec approval request payloads and tests that inline-eval warning lines survive. This branch adds the new explanation fields from an older base without carrying that existing field, so rebasing or merging as-is would drop the current warning payload used by approval consumers.
  - confidence: 0.91
- **[P2] Remap highlights onto sanitized command text:** `src/gateway/server-methods/exec-approval.ts:257`
  - body: The highlights are computed from raw command text, but the Gateway stores a sanitized display command and forwards the original offsets. When sanitization redacts secrets, escapes invisibles, or truncates text before a risky fragment, the UI can highlight the wrong span or omit the intended risk highlight.
  - confidence: 0.88
- **[P2] Keep carrier inline-eval risks for find and xargs:** `src/infra/command-explainer/extract.ts:1033-1036`
  - body: This branch nulls inline-eval detection for find and xargs, while the new recursive carrier parser only handles carried shell-wrapper payloads. Direct carriers such as find ... -exec python -c ... or xargs python -c ... would lose the inline-eval risk line and only show the generic carrier warning.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Approval prompt integrity needs attention because raw highlight offsets and skipped direct carrier inline-eval risks can make risky commands look safer than they are; the .npmrc npmjs.org registry pin is not a concrete supply-chain concern here.

Concerns:

- **[medium] Raw highlight offsets can misrepresent sanitized commands:** `src/gateway/server-methods/exec-approval.ts:257`
  - body: Approval prompts sanitize command text before display, but this PR forwards highlight offsets computed against the raw command. Redaction, invisible-character escaping, or truncation can make the UI highlight the wrong fragment or miss the intended risky fragment.
  - confidence: 0.88
