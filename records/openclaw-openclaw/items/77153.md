## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve command analysis in approval payloads:** `src/gateway/server-methods/exec-approval.ts:256-257`
  - body: Current main computes `commandAnalysis` for every `exec.approval.request` and forwards those parser warning lines through existing approval surfaces. This head only carries the new `commandExplanation*` fields in the request object, so callers that do not send the new metadata and current `commandAnalysis` consumers lose those warnings. Carry `commandAnalysis` forward alongside the new fields when rebasing.
  - confidence: 0.92
- **[P2] Map highlights after command sanitization:** `src/gateway/server-methods/exec-approval.ts:257`
  - body: The new highlights are produced from raw command text, but the request stores `sanitizeExecApprovalDisplayText(effectiveCommandText)` and forwards the raw offsets. If sanitization escapes invisible characters, redacts secrets, or truncates text before a risky fragment, the Control UI slices the sanitized string with stale offsets and can mark the wrong text or drop the risk highlight. Recompute or remap highlights against the sanitized display command.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The npm registry pin is not a concrete supply-chain concern, but approval prompt highlight integrity needs attention because raw offsets are applied after command sanitization.

Concerns:

- **[medium] Raw highlight offsets can misrepresent sanitized commands:** `src/gateway/server-methods/exec-approval.ts:257`
  - body: Approval prompts sanitize command text to expose invisible characters, redact secrets, and cap oversized text, but this PR forwards highlight offsets computed against the raw command. That can make a security-sensitive approval prompt highlight the wrong fragment or omit the intended risky fragment in spoofing/redaction cases.
  - confidence: 0.86
