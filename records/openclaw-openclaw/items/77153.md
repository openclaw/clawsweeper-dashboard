## Best Possible Solution

Rebase onto current main, preserve commandAnalysis, map highlights onto sanitized display text, keep direct carrier inline-eval coverage, generate useful risk lines, then rerun focused approval/UI checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve command analysis in approval requests:** `src/gateway/server-methods/exec-approval.ts:257-258`
  - body: Current main stores commandAnalysis on exec approval requests and forwarded consumers use its warningLines. This request object adds the new explanation fields but omits commandAnalysis, so merging would drop existing inline-eval warning payloads.
  - confidence: 0.93
- **[P2] Remap highlights onto sanitized command text:** `src/gateway/server-methods/exec-approval.ts:258`
  - body: The highlights are computed from raw command text, but the Gateway stores sanitized display text and the UI slices that string by the supplied offsets. Redaction, invisible escaping, or truncation before a risky fragment can highlight the wrong text or hide the intended risk.
  - confidence: 0.9
- **[P2] Keep carrier inline-eval risks for find and xargs:** `src/infra/command-explainer/extract.ts:1033-1036`
  - body: This branch disables inline-eval detection for find and xargs, while the recursive carrier parser only covers shell-wrapper payloads. Direct carriers such as find -exec python -c or xargs node -e would lose the inline-code risk line and only show the generic carrier warning.
  - confidence: 0.86
- **[P2] Generate risk lines for non-highlight approval surfaces:** `src/infra/command-explainer/format.ts:56-57`
  - body: formatCommandExplanationLines always returns an empty array, so automatically generated approvals never populate commandExplanationLines. Forwarded approval messages and clients that cannot render highlights will not receive the parser-backed risk text this PR is trying to surface.
  - confidence: 0.87
