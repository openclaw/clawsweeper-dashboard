## Summary

Keep open. The docs PR addresses a real trusted-proxy configuration gap that is distinct from the open runtime-fix PRs, but the new nginx/troubleshooting guidance omits configured Control UI basePath asset URLs, so path-prefixed deployments can still follow the docs and keep seeing the same PWA asset 403s.

## What This Changes

The PR adds trusted-proxy documentation telling operators to bypass auth for Control UI PWA static assets such as manifest.webmanifest, sw.js, and favicons.

## Best Possible Solution

Update the trusted-proxy docs so the public-asset bypass guidance covers bare root, default /__openclaw__ namespace, and configured /<gateway.controlUi.basePath>/<asset> URLs while keeping the bypass scoped to the known public asset names.

## Reproduction Assessment

Yes, source inspection is enough for the review finding: current docs and runtime support configured Control UI base paths, and the browser public-asset helper resolves those assets under the configured base path.

## Solution Assessment

No, not as currently written. Documentation is the right layer for proxy-owned 403s, but the patch needs to include configured basePath asset paths before it is the narrow maintainable fix.
