## Summary

Keep open: this is a useful maintainer PR for a real Telegram streaming bug, but the current diff is not ready because the new long-final path can corrupt streamed long replies and bypass existing fresh-final rules for stale or overwritten previews.

## What This Changes

The PR changes Telegram final delivery to reuse the active preview for split long text finals, adds Telegram live QA/mock coverage, updates docs and changelog, and handles diagnostics-otel session recovery events as no-ops.

## Best Possible Solution

Merge after the long-final preview reuse path preserves exact text, respects the existing fresh-final fallback for stale or overwritten previews, and adds regression coverage for those cases.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main falls back to standard send for oversized text finals after a preview, matching the transient extra bubble this PR targets. The PR-introduced regression is also source-reproducible from the new long-final branch and draft-stream max-length behavior.

## Solution Assessment

No. Reusing the preview is the right narrow direction, but this implementation should only do it under the same conditions that allow preview final edits and must not send remainders after a regressive-skipped first-chunk edit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Handle regressive skips before sending long-final remainders:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:389-402`
  - body: When the streamed draft has already advanced past `firstChunk`, `tryUpdatePreviewForLane` can return `regressive-skipped`; this branch then sends `remainingChunks.join("")` as if the preview was edited to exactly `firstChunk`. Because the draft stream stops once rendered text exceeds Telegram's max, the visible preview can still be shorter than `lane.lastPartialText`, so a normal long streamed final can lose the middle of the answer while the QA only checks begin/end markers. Handle `regressive-skipped` by editing, falling back, or computing the remainder from the actual visible chunk.
  - confidence: 0.9
- **[P2] Keep fresh-final fallback for stale or overwritten previews:** `extensions/telegram/src/lane-delivery-text-deliverer.ts:675-681`
  - body: This oversized-final branch runs before the normal fallback and does not check `shouldUseFreshFinalForLane(lane)`. That regresses the existing Telegram behavior for previews older than about a minute or previews followed by visible non-preview output: those should be replaced by a fresh final and cleanup, not edited into the first long-final chunk. Bail out to the standard send/cleanup path when the fresh-final guard is true.
  - confidence: 0.86
