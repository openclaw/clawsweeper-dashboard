## Summary

Current main and the v2026.5.7 release already cover the central startup contention problem: startup WebSocket handshakes are retry-gated while sidecars settle, readiness stays red until sidecars/channels/hooks are ready, channel startup fanout is capped, and restart retries already use exponential backoff with jitter.

## What This Changes

The issue requests explicit readiness gating before sidecar/plugin connection plus randomized reconnect jitter/backoff to avoid startup connection storms.

## Best Possible Solution

Keep the shipped startup retry/readiness/fanout/backoff behavior, and ask for a new report with current v2026.5.7-or-newer logs if startup timeouts still reproduce.

## Reproduction Assessment

No high-confidence current-main failure path remains. The report targets OpenClaw 2026.4.15, while current main and v2026.5.7 source show retry-gated startup handshakes, readiness gating, capped channel startup, and jittered restart backoff.

## Solution Assessment

Yes. The shipped fix addresses the maintainable core boundary by gating clients/readiness during sidecar startup and bounding channel lifecycle retries, without adding plugin-specific dependency policy to core.
