## Summary

Keep open: current main still accepts only exact cron job keys or a few known concatenated-key repairs, so `schedule `, `sessionTarget `, `payload `, and `enabled ` reach strict `cron.add` validation as unexpected properties. The prior flat-params fix overlaps but does not cover this non-empty `job` object key-whitespace shape.

## What This Changes

The issue reports that the cron tool `add` action fails when model/tool-call extraction adds trailing spaces to several top-level keys inside the `job` parameter.

## Best Possible Solution

Normalize only known cron tool job/patch property names with harmless trailing whitespace in the agent cron-tool canonicalizer, add focused regression coverage, and keep the strict Gateway cron schema unchanged.

## Reproduction Assessment

Yes, source-reproducible: invoke the cron tool with `action: "add"` and a `job` object whose known top-level keys have trailing spaces. Current source passes those keys through canonicalization and then strict `cron.add` validation rejects them as missing canonical fields plus unexpected properties.

## Solution Assessment

Yes: the narrowest maintainable fix is in the agent cron-tool canonicalization layer, which already owns model/tool-call argument repairs. Relaxing the Gateway protocol schema would be the wrong layer because RPC clients should continue to use canonical cron job keys.
