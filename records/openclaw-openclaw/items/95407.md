## Summary

Keep open: current `main` and latest release still lack whitespace-padded cron job key recovery, and multiple open linked PRs are already candidate fixes for the same reported failure.

## What This Changes

The issue asks for `cron.add` to accept otherwise valid cron job input when local model tool-call parsing leaves trailing spaces on top-level job keys such as `schedule`, `sessionTarget`, `payload`, and `enabled`.

## Best Possible Solution

Land one narrow cron canonicalization fix that recovers known whitespace-padded cron keys while preserving strict gateway schema validation and rejecting ambiguous duplicates.

## Reproduction Assessment

Yes, source-level: current `canonicalizeCronToolObject` has no padded-key repair, while `cron.add` routes its `job` through that canonicalizer before strict gateway validation. I did not run the live qwen35b/llamacpp route in this read-only review.

## Solution Assessment

Yes, a narrow cron canonicalization fix is the best boundary because prior local-parser cron recovery already lives there and the gateway schema should remain strict. The linked open PRs are the right implementation path to review rather than starting another fix job.
