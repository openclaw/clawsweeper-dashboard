## Summary

Keep open: current main still only recovers the earlier concatenated cron-key corruption, while this report is about whitespace-padded job keys and is now paired with multiple open fix PRs.

## What This Changes

The issue reports that cron.add rejects job parameters from affected local model tool-call parsing when top-level keys such as schedule, sessionTarget, payload, and enabled arrive with trailing spaces.

## Best Possible Solution

Review and land one narrow cron tool canonicalizer fix that recovers known whitespace-padded cron job keys before strict gateway validation, then close the duplicate candidate PRs.

## Reproduction Assessment

Yes, source-level: current main leaves `schedule `, `sessionTarget `, `payload `, and `enabled ` unchanged, and the strict cron.add schema requires exact keys. I did not run the live qwen35b/llamacpp serializer path in this read-only review.

## Solution Assessment

Yes. The existing cron tool canonicalizer is the narrowest maintainable fix layer because prior local-parser recovery already lives there, while loosening gateway schemas or globally trimming JSON keys would broaden the behavior change.
