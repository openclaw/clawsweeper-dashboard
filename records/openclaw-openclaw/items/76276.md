## Summary

Keep open. Current main still drops the manual `cron.run` acknowledgement ID before finished cron events and JSONL-backed history, and the linked open PR is the right implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Request that manual cron run IDs returned by `openclaw cron run` survive into `cron.runs` history for reliable correlation.

## Best Possible Solution

Land an additive runId propagation fix through cron finished events, run-log entries/readers, protocol/UI types, docs, and a focused regression; PR #76288 is the current candidate.

## Reproduction Assessment

Yes. Static inspection of current main is enough: `cron.run` returns a generated `manual:` ID, while the event, JSONL append, parser, schema, UI type, and existing gateway assertion omit it from history.

## Solution Assessment

Yes. The narrow maintainable fix is to persist the existing acknowledgement ID through the cron runtime/history contract rather than correlating by timestamp or changing only CLI output.
