## Summary

Keep open. Current main returns a `manual:...` run ID from `cron.run`, but that ID is not carried into cron finished events, the JSONL run log, the protocol schema, or the history reader, so this is a distinct follow-up from earlier fixes that only made manual runs appear in history.

## What This Changes

Expose a stable run ID from `openclaw cron run <id>` in cron run history so manual acknowledgments can be correlated with `cron.runs` entries.

## Best Possible Solution

Add an additive `runId` field to cron finished events, run-log entries, protocol schema, readers, and UI types, and thread the existing manual acknowledgment ID into the finished history entry while preserving old JSONL compatibility.

## Reproduction Assessment

Yes. The source path is enough to reproduce with high confidence: `cron.run` returns the `manual:...` ID from `enqueueRun()`, but the execution, event, log, schema, and reader paths omit it; a targeted regression can assert the latest `cron.runs` entry has the returned ID.

## Solution Assessment

Yes. The narrowest maintainable fix is to pass the existing acknowledgment ID through the runtime event/log contract and test it, rather than relying on timestamp matching or changing only CLI display output.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76276. Thread the `manual:...` runId returned by `cron.run` through queued manual execution into cron finished events and JSONL run history, then expose it from `cron.runs`. Keep the change additive and backward compatible: old JSONL entries without runId must still parse, scheduled runs should not be broken, and do not redesign cron lane concurrency or dedupe behavior. Likely updates: add optional `runId` to cron event/run-log/protocol/UI types, preserve parsed `runId`, include it when appending run logs, pass the manual ack ID from `enqueueRun()` into the actual execution/finish path, and add a gateway regression asserting the returned runId equals the latest `cron.runs` entry runId. Update docs/changelog if the user-facing history contract changes. Do not touch unrelated cron scheduling semantics.
