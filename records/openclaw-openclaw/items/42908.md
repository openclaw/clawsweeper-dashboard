## Summary

Keep open: current main and the latest release still drop these Feishu callback sibling fields, and this branch addresses the right parser/fallback boundary, but it still lacks redacted live Feishu proof and overlaps broader input_value/name contract work.

## What This Changes

The PR extends Feishu card-action parsing and fallback serialization to preserve `option`, `options`, and `form_value`, with focused tests and a changelog entry.

## Best Possible Solution

Land or replace one canonical Feishu plugin patch that preserves `option`, `options`, and `form_value` from raw parsing through fallback dispatch with regression coverage and redacted live Feishu proof, while explicitly deciding whether broader fields belong in the same contract.

## Reproduction Assessment

Yes, at source level. A raw `card.action.trigger` payload with sibling `action.option`, `action.options`, or `action.form_value` is normalized into an event containing only `value` and `tag`, then fallback text serializes only `action.value`.

## Solution Assessment

Yes for the narrowed fields: preserving them at the parser/event boundary and merging them into fallback JSON after legacy `text`/`command` precedence fits the current design. Maintainers still need to settle whether `input_value` and `name` are included in this PR or left to the broader linked follow-up.
