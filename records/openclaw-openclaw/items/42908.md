## Summary

Keep open: current main still drops Feishu `option`, `options`, and `form_value` before fallback dispatch, and this PR is a focused implementation path. It should not merge until it has real Feishu behavior proof and fresh landing validation.

## What This Changes

The PR preserves Feishu card-action `option`, `options`, and `form_value` fields through parsing and legacy fallback dispatch, with focused tests and a changelog entry.

## Best Possible Solution

Land this PR or an equivalent Feishu plugin patch after real Feishu behavior proof, rebase/mergeability cleanup, and targeted Feishu plus changed checks; keep broader `input_value`/`name` support in the linked follow-up.

## Reproduction Assessment

Yes, source-level. A raw `card.action.trigger` payload with `action.option`, `action.options`, or `action.form_value` is parsed into an event that keeps only `value` and `tag`, then fallback text only serializes `action.value`.

## Solution Assessment

Yes for the narrow scope. The PR preserves the fields at the parser/event boundary, carries them through fallback serialization, and keeps legacy `value.text`/`value.command` precedence; broader callback-field support should remain separate.
