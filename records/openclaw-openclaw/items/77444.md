## Summary

Keep open: current main still has a source-reproducible FAL image-generation dispatch bug where `model: "fal-ai/flux/dev"` is parsed as provider `fal-ai`, so the request fails before FAL auth resolution can use `FAL_KEY`.

## What This Changes

The issue asks for FAL image generation to resolve `FAL_KEY` or a valid auth profile and generate images for the reported FAL model instead of rejecting it as an unregistered provider.

## Best Possible Solution

Normalize FAL model selection so tool/runtime requests for the advertised FAL model reach provider `fal`, then add regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. From source inspection, passing `model: "fal-ai/flux/dev"` to `image_generate` is parsed as provider `fal-ai`, producing the reported registered-provider error before FAL auth resolution runs.

## Solution Assessment

Yes. A narrow fix in image-generation model-ref normalization, selected-provider resolution, or tool model listing is the maintainable path; changing auth resolution alone would not address the observed provider lookup failure.

Fix https://github.com/openclaw/openclaw/issues/77444. Current main parses an explicit `image_generate` model override of `fal-ai/flux/dev` as provider `fal-ai`, so generation fails before the bundled `fal` provider can resolve `FAL_KEY`. Keep the fix narrow: route the advertised FAL image model to provider `fal` without weakening exact override behavior for unrelated providers, and preserve the documented canonical config ref `fal/fal-ai/flux/dev`. Likely approaches are to qualify provider model listings, normalize known provider-owned model ids in image-generation selection, or adjust FAL image model metadata so agents/tool callers use provider-qualified refs. Add/update regression tests for runtime candidate parsing and the image_generate tool FAL path, include a CHANGELOG.md fix entry, and do not move FAL auth policy into core.
