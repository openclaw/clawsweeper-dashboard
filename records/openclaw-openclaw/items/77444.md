Keep open: current main still misroutes the reported FAL model override as provider `fal-ai`, so the request fails before FAL auth resolution can run.

## What This Changes

The issue requests that FAL image generation work with `FAL_KEY` or a valid auth profile when using the FAL model `fal-ai/flux/dev`.

## Best Possible Solution

Normalize FAL image model selection so the FAL model shown by `image_generate` routes to provider `fal`, then add regression coverage and a user-facing changelog entry.

Yes from source inspection: pass `model: "fal-ai/flux/dev"` to `image_generate`, and current main parses provider `fal-ai` before emitting the reported registered-provider error. I did not run a live FAL request.

## Solution Assessment

Yes: a narrow fix in image-generation model-ref/tool normalization is the maintainable path because an auth-only change would not reach FAL for the reported model override.
