## Summary

Keep open: current main still has a source-reproducible FAL image-generation dispatch problem where `model: "fal-ai/flux/dev"` is parsed as provider `fal-ai`, so the request fails before the bundled `fal` provider can resolve `FAL_KEY`.

## What This Changes

The issue requests that the bundled FAL image-generation provider accept the reported FAL model/auth setup and resolve `FAL_KEY` or an auth profile instead of failing with an unregistered-provider error.

## Best Possible Solution

Keep the core parser extension-agnostic, but make FAL image model selection/listing robust so the provider-advertised FAL model id cannot be misrouted as provider `fal-ai`, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes from source inspection: pass `model: "fal-ai/flux/dev"` to `image_generate` and current main parses it as provider `fal-ai`, then emits the reported unregistered-provider error before auth resolution. I did not run a live FAL request.

## Solution Assessment

Yes: the narrowest maintainable fix is image-generation model-ref normalization or display/selection handling around provider-owned model ids, not changing FAL auth resolution alone.
