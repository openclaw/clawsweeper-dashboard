## Summary

Keep open: current main and the latest release still forward inline image attachment data unchanged before provider payload assembly, and the focused repair remains an open, unmerged PR.

## What This Changes

The issue reports that inline or replayed history image attachments can carry raw non-ASCII byte strings into provider base64 fields, causing Anthropic/OpenAI request rejection and no reply.

## Best Possible Solution

Refresh, review, and land https://github.com/openclaw/openclaw/pull/88112 or an equivalent inline-boundary normalization repair with regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes, source-reproducible: current main forwards inline image data unchanged while the shared image contract and provider mappers consume that field as base64. I did not run a live channel/provider turn in this read-only review.

## Solution Assessment

Yes, the best fix direction is to normalize at the inline/replayed attachment boundary so the shared image-data base64 contract is restored before Anthropic and OpenAI sinks. The open PR still needs refresh and review before it can be the mergeable form.
