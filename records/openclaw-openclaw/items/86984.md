## Summary

Keep open: current main and the latest release still pass inline image attachment data through unchanged before provider payload assembly, and the focused repair remains an open unmerged PR rather than landed code.

## What This Changes

The issue reports that replayed inline image attachments can reach provider request mappers as raw binary/latin1 strings instead of ASCII base64, causing provider request rejection and silent channel non-response.

## Best Possible Solution

Refresh, review, and land https://github.com/openclaw/openclaw/pull/88112 or an equivalent inline-boundary normalization repair with regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes, source-reproducible: current main forwards inline image `data` unchanged while provider sinks consume that field as base64. I did not run a live channel/provider turn in this read-only review.

## Solution Assessment

Yes, normalizing at the inline/replayed attachment boundary is the narrow maintainable fix because it restores the shared `ImageContent.data` base64 contract before Anthropic and OpenAI sinks. Provider-layer guards could be defense in depth, but they should not replace the source-boundary repair.
