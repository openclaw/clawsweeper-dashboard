## Summary

Keep open: current main and the latest release still pass inline image attachment data through unchanged before provider payload assembly, while the focused repair is an open, unmerged PR rather than landed code.

## What This Changes

The issue reports that replayed or inline image attachments can carry raw latin1/binary data into Anthropic and OpenAI image payloads instead of ASCII base64, causing follow-up turns to fail silently.

## Best Possible Solution

Refresh, review, and land https://github.com/openclaw/openclaw/pull/88112 or an equivalent inline-boundary normalization repair with regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes, source-reproducible: current main forwards inline image `data` unchanged and provider mappers consume that field as base64. I did not run a live channel/provider turn in this read-only review.

## Solution Assessment

Yes, the best fix is to normalize at the inline/replayed attachment boundary so the shared `ImageContent.data` base64 contract is restored before Anthropic and OpenAI sinks. Provider-layer guards could be defense in depth, but they should not replace the source-boundary repair.
