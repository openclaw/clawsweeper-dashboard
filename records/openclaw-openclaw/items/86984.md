## Summary

Keep open: current main and the latest release still pass inline image attachment data through unchanged before Anthropic/OpenAI provider payload assembly, and the focused fix is currently an open, unmerged PR rather than landed code.

## What This Changes

The issue reports replayed or inline image attachments reaching Anthropic/OpenAI image payloads with raw non-base64 data, causing provider rejection and silent follow-up failures.

## Best Possible Solution

Refresh, review, and land https://github.com/openclaw/openclaw/pull/88112 or an equivalent inline-boundary normalization repair with regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes, source-reproducible: current main forwards inline image `data` unchanged and the provider mappers consume that field as base64. I did not run a live Discord/Anthropic or OpenAI Responses turn in this read-only review.

## Solution Assessment

Yes: normalizing at the inline/replayed attachment boundary is the narrow maintainable fix because it restores the shared `ImageContent.data` base64 contract before both Anthropic and OpenAI sinks. Provider-layer guards could be added later as defense in depth, but they should not replace the source-boundary repair.
