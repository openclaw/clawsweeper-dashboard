## Summary

Keep open. Current main still has the reported no-fallback `web_fetch` transport-error rethrow, so the headline fix remains useful, but this PR is not safe to merge as-is because it bundles unrelated changes and has concrete blockers in dependency metadata, Feishu key validation, and Telegram allowlist semantics.

## What This Changes

The PR wraps `web_fetch` network failures with URL and cause context while also changing Feishu external-key validation, package overrides, model/input handling, tool-result text checks, agent model schema, and Telegram allowlist parsing.

## Best Possible Solution

Replace or rework this PR into a narrow `web_fetch` patch that wraps only no-fallback network-level failures, extracts `error.cause.message` or `error.cause.code`, preserves the original error as `cause`, includes safe requested-URL context, and adds a regression test for a direct fetch rejection with provider fallback unavailable. Leave the Feishu, Telegram, config/schema, model/table, tool-result, and package override changes to separate reviewed PRs.

## Reproduction Assessment

Yes. A focused unit test can make the guarded fetch reject with a Node-style `TypeError("fetch failed")` plus `cause`, disable provider fallback, and observe that current main rethrows the original message without URL or cause context.

## Solution Assessment

No. The web_fetch wrapper itself is the right direction, but this PR is not the best solution because unrelated bundled changes introduce blocking install, Feishu, and Telegram regressions; the safer path is a web_fetch-only replacement patch.

  - confidence: 0.9

  - body: Allowing negative IDs makes Telegram group chat IDs look like valid `allowFrom`/`groupAllowFrom` sender entries, but runtime authorization compares this list with `from.id`. Users still will not authorize the group, and the invalid-config warning disappears.
  - confidence: 0.88
