## Summary

Keep open. The underlying `web_fetch` diagnostic bug is still valid on current `main`, but this PR is not mergeable as-is because unrelated bundled changes introduce install, Feishu API path, and Telegram authorization regressions.

## What This Changes

The PR wraps direct `web_fetch` network failures with URL/cause context and also changes Feishu key validation, dependency overrides, model/input handling, tool-result checks, agent model schema, and Telegram allowlist parsing.

## Best Possible Solution

Land a narrow replacement that only wraps direct `web_fetch` network failures after provider fallback is unavailable, adds focused regression coverage, and leaves Feishu, Telegram, package override, schema, and model/tool-result changes to separate reviews.

## Reproduction Assessment

Yes. Source inspection shows `runWebFetch` still rethrows the original direct network error after provider fallback returns no payload; a focused unit test can mock a Node-style `TypeError("fetch failed")` with `cause` and disabled fallback.

## Solution Assessment

No. The web_fetch wrapper is the right fix direction, but this PR is not the best way to solve it because unrelated package, Feishu, and Telegram changes introduce regressions outside the stated bug.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Regenerate the lockfile after removing overrides:** `package.json:427`
  - body: Removing the `node-domexception` override from `package.json` without the matching `pnpm-lock.yaml` update leaves frozen installs out of sync. CI runs `pnpm install --frozen-lockfile`, so fresh validation can fail before tests run.
  - confidence: 0.95
- **[P2] Keep rejecting Feishu key separators:** `extensions/feishu/src/external-keys.ts:18-19`
  - body: This check only rejects path segments equal to `.` or `..`, so values like `a/b` and `a\b` pass normalization. Those keys are later used as `image_key`/`file_key` path parameters, regressing the current separator guard.
  - confidence: 0.9
- **[P2] Keep group chat IDs out of sender allowlists:** `src/telegram/bot-access.ts:47-51`
  - body: Allowing negative IDs makes Telegram group chat IDs look valid in `allowFrom`/`groupAllowFrom`, but runtime authorization compares those lists with `from.id`. Users still will not authorize the group, and the invalid-config warning disappears.
  - confidence: 0.88
