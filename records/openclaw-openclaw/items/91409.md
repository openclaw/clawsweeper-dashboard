## Summary

Keep open: the PR addresses a real gateway/CLI asymmetry, but it is not merge-ready because the RPC finalizer loses the effective default `dev` channel for unconfigured git/source updates, so plugin convergence can run on `stable` instead of matching the core update.

## What This Changes

The PR adds a post-core `openclaw update finalize` subprocess after successful gateway `update.run` git/source updates, with tests for finalizer selection, env/argv handling, timeout behavior, and restart blocking.

## Best Possible Solution

Mirror the CLI’s split between effective post-core channel and requested persisted channel, then prove both default no-config source updates and configured-channel updates converge plugins before restart.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current gateway `update.run` calls `runGatewayUpdate` and then restarts on `result.status === "ok"`, while the runner doctor pass can defer configured-plugin repair. I did not run a live over-the-wire update in this read-only review.

## Solution Assessment

No, not as submitted: spawning the existing finalizer is the right general boundary, but it must preserve the effective git update channel separately from config persistence. The safer fix is the CLI-style channel context split plus focused proof for default source installs.
