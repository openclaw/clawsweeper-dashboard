## Summary

Current main still appears to have the reported bug: `/btw` accepts a Codex OAuth profile for the default OpenAI Codex runtime, but the side-question runner keeps the resolved model on the `openai` provider and can fall back to the plain OpenAI Responses stream path. The documented contract says `openai/gpt-5.5` with omitted runtime uses the Codex app-server harness and `openai-codex` auth, so this should remain open for a focused fix.

## What This Changes

The issue requests that `/btw` honor the active Codex/OpenAI runtime and auth transport when `openai/gpt-*` is backed by `openai-codex` OAuth.

## Best Possible Solution

Teach `/btw` to resolve and execute side questions through the active runtime/auth transport for `openai/gpt-*` Codex-auth sessions, with regression coverage proving it does not send Codex OAuth tokens to plain OpenAI Responses.

## Reproduction Assessment

Yes, at source level: configure an active `openai/gpt-5.5` session with an `openai-codex` OAuth profile and send `/btw ...`; current main resolves the Codex auth profile but keeps the side call on the `openai` model/provider stream path. I did not execute the live Telegram/OAuth request.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: fix `/btw` runtime/auth resolution rather than patching Telegram formatting or only the PI helper path. The side call should remain lightweight and tool-less while using the same effective Codex transport semantics as the active session.
