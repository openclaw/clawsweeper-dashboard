## Summary

Keep open: current main still lacks the requested mutable hook behavior, but this PR is not mergeable as-is because it bundles a security-sensitive plugin API expansion with adapter-side tool-result mutation and still has source-proven correctness/security blockers plus missing real behavior proof.

## What This Changes

This PR changes plugin hook contracts, hook runners, embedded agent handling, adapter-side tool execution, and tests so `llm_input`, `llm_output`, and `after_tool_call` can return blocking or payload-modifying results.

## Best Possible Solution

Land or supersede this with a focused, hardened mutable `llm_input`/`llm_output` contract, then design pre-model tool-result mutation separately with exact-once dispatch, result validation, docs, changelog, and live proof.

## Reproduction Assessment

Yes, for the PR review findings by source inspection: PR head applies prompt rewrites after image detection, uses truthy override checks, leaves original assistant data reachable after output replacement, and changes `after_tool_call` dispatch ownership against current-main guards.

## Solution Assessment

No. The feature direction is useful, but this PR bundles mutable LLM hooks with adapter-side tool-result mutation before prompt-policy, prompt-artifact, result-validation, exact-once dispatch, docs, conflict, and live-proof concerns are settled.

Full review comments:

- **[P1] Recompute prompt artifacts after llm_input rewrites:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: `detectAndLoadPromptImages` and related prompt-image state are computed before the new `llm_input` result is applied, but the model call still sends `imageResult.images`. If a plugin removes or changes prompt-local image references, stale images from the original prompt can still reach the model; move the hook earlier or recompute and roll back all derived prompt state after applying the override.

  - body: The new result type lets `llm_input` handlers return `prompt` and `systemPrompt`, but the existing prompt-injection policy does not cover this mutation path. A plugin configured with `allowPromptInjection=false` could still rewrite model instructions unless the policy gate strips or blocks these fields.
  - confidence: 0.9
- **[P2] Honor empty prompt and system prompt overrides:** `src/agents/pi-embedded-runner/run/attempt.ts:2839-2844`
  - body: The call site uses truthy checks for `llmInputResult.prompt` and `systemPrompt`, so intentional empty-string rewrites are ignored. Redaction or guardrail plugins need field-presence checks so `""` can remove sensitive prompt text or clear a system prompt deliberately.
  - confidence: 0.84
- **[P2] Strip original assistant payload after output replacement:** `src/agents/pi-embedded-runner/run/attempt.ts:3177-3178`
  - body: When `llm_output` replaces `assistantTexts`, the returned attempt still exposes the original `lastAssistant`. That can leave unredacted assistant text, reasoning, or error metadata reachable after a plugin requested response replacement.
  - confidence: 0.84
- **[P2] Preserve exact-once after_tool_call dispatch:** `src/agents/pi-tool-definition-adapter.ts:173-178`
  - body: The adapter now invokes `after_tool_call` before returning the tool result while the subscription handler still emits the same hook after the result is committed. That can call the same plugin hook twice for embedded tool executions, contradicting the current exact-once regression guard.
  - confidence: 0.86
- **[P2] Validate replacement tool results before returning them:** `src/agents/pi-tool-definition-adapter.ts:185-186`
  - body: The adapter returns `hookResult.result` with an `as AgentToolResult` cast and no normalization. A malformed plugin result can enter the tool pipeline without the shape checks applied to ordinary tool output, so replacement results should be validated or normalized before they are returned.
  - confidence: 0.79
- **[P2] Keep injected after_tool_call runners compatible:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:595`
  - body: The subscription handler now calls `runAfterToolCallVoid`, but existing injected runner shapes may only provide `runAfterToolCall`. Supplying that runner will throw before `.catch(...)` can handle it; update the injected runner contract or add a guarded fallback while migrating.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff introduces new plugin authority over model prompts, system prompts, assistant output, and tool results while leaving concrete policy, artifact, dispatch, and validation gaps.

Concerns:

- **[high] Mutable llm_input bypasses prompt policy:** `src/plugins/types.ts:1563`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields, but the prompt-injection policy is not extended to this new mutation path, so `allowPromptInjection=false` would not prevent these rewrites.
  - confidence: 0.9
- **[high] Prompt override can retain stale image artifacts:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: Prompt rewrites happen after prompt-local image detection, so a redaction plugin can change the prompt while images from the original prompt still go to the model.
  - confidence: 0.91
- **[medium] Output override can leave original assistant data reachable:** `src/agents/pi-embedded-runner/run/attempt.ts:3197`
  - body: Replacing `assistantTexts` does not clear or replace `lastAssistant`, leaving original assistant content or metadata available downstream after a plugin requested response redaction.
  - confidence: 0.84
- **[medium] Tool-result replacement is unvalidated:** `src/agents/pi-tool-definition-adapter.ts:185`
  - body: The new adapter path trusts an `unknown` plugin-returned result via a type cast, which can bypass the normal tool-result normalization boundary before the model sees the result.
