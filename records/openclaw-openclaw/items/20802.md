## Summary

Keep open: the mutable-hook feature is not implemented on current main, and this PR still needs maintainer API/security review, contributor real-behavior proof, and fixes for blocking patch findings before it can land.

## What This Changes

The PR changes plugin hook contracts/runners and embedded agent/tool call sites so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return blocking or payload-modifying results.

## Best Possible Solution

Land or supersede this with a focused, hardened mutable `llm_input`/`llm_output` contract, then handle pre-model tool-result mutation as a separate plugin API design with exact-once dispatch, validation, docs, changelog, and live proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR head applies `llm_input` rewrites after prompt image detection, exposes prompt/systemPrompt result fields without policy enforcement, and changes `after_tool_call` dispatch ownership while current main guards exact-once dispatch.

## Solution Assessment

No. The feature direction is useful, but this PR couples mutable LLM hooks with adapter-side tool-result mutation before prompt policy, redaction cleanup, result validation, dispatch ownership, docs, and live proof are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Recompute prompt artifacts after llm_input rewrites:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: `llm_input` prompt overrides are applied after `detectAndLoadPromptImages` has already built `imageResult`, but the later model call still sends `imageResult.images`. If a plugin removes or changes prompt-local image references, stale images from the original prompt can still reach the model; move the hook earlier or recompute and roll back all derived prompt state after applying the override.
  - confidence: 0.9
- **[P1] Enforce prompt policy on llm_input rewrites:** `src/plugins/types.ts:1563-1565`
  - body: The new result type lets `llm_input` handlers return `prompt` and `systemPrompt`, but the registry only constrains the existing prompt-injection hook names when `allowPromptInjection=false`. A plugin without prompt-injection permission can still rewrite model instructions through this new path.
  - confidence: 0.9
- **[P2] Strip original assistant payload after output replacement:** `src/agents/pi-embedded-runner/run/attempt.ts:3177-3197`
  - body: When `llm_output` replaces `assistantTexts`, the returned attempt still exposes the original `lastAssistant`. That can leave unredacted assistant text, reasoning, or error metadata reachable after a plugin requested response replacement.
  - confidence: 0.82
- **[P2] Keep exact-once hook runner compatibility:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:595`
  - body: The subscribe handler now calls `runAfterToolCallVoid`, but current exact-once embedded-run tests inject a hook runner with only `runAfterToolCall`. That path will call an undefined method when `ctx.hookRunner` is supplied, so update the regression test and guard or migrate injected legacy runners with the new method.
  - confidence: 0.78
- **[P3] Add a changelog entry for the plugin API feature:** `src/plugins/types.ts:1563-1565`
  - body: This changes the user-facing plugin hook contract and adds security-sensitive mutation behavior, but the PR does not update `CHANGELOG.md`. Repo policy requires an active-version changelog entry before this lands.
  - confidence: 0.86
