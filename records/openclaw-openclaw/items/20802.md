## Summary

Keep open: current `main` still lacks the requested mutable hook behavior, and this PR remains a security-sensitive plugin API change with concrete correctness findings, missing real behavior proof, and unresolved scope around adapter-side `after_tool_call` mutation.

## What This Changes

The PR changes `llm_input`, `llm_output`, and adapter-side `after_tool_call` plugin hooks from observation-only hooks into result-returning modifying hooks.

## Best Possible Solution

Land or supersede the mutable `llm_input`/`llm_output` work through a focused, hardened contract, then design pre-model tool-result mutation separately with exact-once dispatch, result validation, docs, and live proof.

## Reproduction Assessment

Yes, for the PR review findings by source inspection: the PR head applies prompt rewrites after prompt-image artifacts are derived, uses truthy override checks, leaves original assistant data reachable after output replacement, and changes `after_tool_call` dispatch ownership against current-main guards.

## Solution Assessment

No. The feature direction is useful, but this PR bundles mutable LLM hooks with adapter-side tool-result mutation before prompt-policy, artifact recomputation, result validation, exact-once dispatch, docs/changelog, and real proof concerns are settled.

## Security Review

Status: needs_attention

Summary: The diff is security-sensitive and has concrete policy-bypass and data-retention concerns around prompt and output mutation.

Concerns:

- **[high] Mutable llm_input bypasses prompt policy:** `src/plugins/types.ts:1563`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields without extending the existing prompt-injection policy gate, so `allowPromptInjection=false` would not prevent these rewrites.
  - confidence: 0.9
- **[high] Prompt override can retain stale image artifacts:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: Prompt rewrites happen after prompt-local image detection, so a redaction plugin can change prompt text while images from the original prompt still reach the model.
  - confidence: 0.91

  - confidence: 0.84

  - body: The adapter accepts `PluginHookAfterToolCallResult.result` as `unknown` and casts it to `AgentToolResult`, so malformed plugin output can cross the tool-result boundary without schema validation.
