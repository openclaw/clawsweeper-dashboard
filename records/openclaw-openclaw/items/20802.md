## Summary

Keep this PR open for maintainer/API review. The feature direction is useful and current main still has observation-only hooks, but this PR remains broader than the focused mutable LLM-hook path and introduces blocking security/correctness issues around prompt policy, stale prompt artifacts, output redaction, and `after_tool_call` dispatch ownership.

## What This Changes

The PR changes plugin hook types/runners and embedded agent/tool call sites so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return blocking or payload-modifying results, with new tests.

## Best Possible Solution

Ship a focused, hardened mutable `llm_input`/`llm_output` contract, then handle pre-model tool-result mutation as a separate plugin API design with exact-once dispatch, validation, docs, changelog, and policy constraints.

## Reproduction Assessment

Yes for the review findings by source inspection: current main still runs these hooks as void, and the PR head applies modifications after derived prompt state and outside current prompt policy while adding adapter-side dispatch alongside existing subscribe dispatch. I did not run tests because this task is read-only and `pnpm` is unavailable in the checkout.

## Solution Assessment

No. The direction is useful, but this PR is broader than the safer path because it couples mutable LLM input/output hooks with adapter-side `after_tool_call` result mutation before the prompt-policy, validation, and dispatch-ownership contracts are settled.

  - body: `llm_input` prompt overrides are applied after prompt-local image detection and other derived prompt state have already been built. If a plugin removes or changes image references, stale images or prompt metadata can still be sent with the rewritten prompt, so the hook needs to run earlier or recompute and roll back all derived prompt state after applying the result.
  - confidence: 0.9

  - body: Adding `prompt` and `systemPrompt` result fields makes `llm_input` a prompt-mutating hook, but `allowPromptInjection=false` currently constrains only the existing prompt-injection hook names. A plugin with conversation access but prompt injection disabled could still rewrite model instructions through this new result type.
  - confidence: 0.9

  - body: The adapter now dispatches `runAfterToolCall`, while the subscription handler still dispatches the same plugin hook through the audit path. In embedded runs both paths are active, so plugin side effects can fire twice and current exact-once regression coverage is violated.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds prompt, response, and tool-result mutation surfaces without fully carrying current prompt-policy, derived-payload rollback, and redaction cleanup guarantees forward.

Concerns:

- **[high] Mutable llm_input bypasses prompt-injection policy:** `src/plugins/types.ts:1563`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields, but current policy only treats existing prompt-building hooks as prompt-injection surfaces. Without a matching registry constraint, a plugin configured with `allowPromptInjection=false` can still rewrite model instructions.
  - confidence: 0.9
- **[high] Prompt override can retain stale prompt artifacts:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: The PR applies prompt overrides after prompt-local image detection and related prompt state are computed. For a redaction plugin, stale image artifacts can still be sent unless the implementation moves the hook earlier or recomputes all derived prompt payload state.
  - confidence: 0.9
- **[medium] Output override can leave original assistant text reachable:** `src/agents/pi-embedded-runner/run/attempt.ts:3177`
  - body: The proposed `llm_output` override changes `assistantTexts` but leaves the original `lastAssistant` object and message snapshots intact, so sensitive text that a plugin intended to redact can remain available to downstream handling.
  - confidence: 0.78
