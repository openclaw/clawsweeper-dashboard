## Summary

Keep open for maintainer review: current main still lacks mutable `llm_input`/`llm_output` behavior, but this broader PR also changes adapter-side `after_tool_call` result mutation and still has blocking correctness, security, and proof gaps.

## What This Changes

The branch widens plugin hook contracts and runners so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can block or modify prompts, assistant output, and tool results, with added hook tests.

## Best Possible Solution

Land or supersede the LLM-hook subset with a focused hardened implementation, then split pre-model tool-result mutation into a separate plugin API design with exact-once dispatch, result validation, docs, changelog, and live proof.

## Reproduction Assessment

Yes, for the review findings by source inspection. The PR diff applies prompt rewrites after prompt-derived image artifacts, uses truthy override checks, leaves original assistant data reachable after output replacement, and changes `after_tool_call` dispatch ownership against current-main guards.

## Solution Assessment

No, not as-is. The feature direction is useful, but this PR combines mutable LLM hooks with adapter-side tool-result mutation before the prompt policy, artifact consistency, result validation, exact-once dispatch, docs, and real proof concerns are resolved.
