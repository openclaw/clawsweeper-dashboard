## Summary

Keep open for maintainer review: current main still lacks the requested mutable hook capability, but this PR introduces prompt-safety and tool-dispatch contract issues that should block merge in its current shape.

## What This Changes

The PR changes plugin hook contracts and embedded/adapter call sites so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return blocking or mutation results.

## Best Possible Solution

Ship a focused, hardened mutable `llm_input`/`llm_output` contract, likely via #61485 or equivalent, and split pre-model tool-result mutation into a separate API design with exact-once dispatch and result validation.

## Reproduction Assessment

Yes. Source inspection of current main shows the hooks are still void/fire-and-forget, and static inspection of the PR diff gives high-confidence reproduction paths for the prompt-policy, prompt-derived-state, output-redaction, and duplicate-dispatch risks.

## Solution Assessment

No. The direction is useful, but this PR combines LLM prompt/output mutation with adapter-side tool-result mutation before policy enforcement, derived prompt state, redaction cleanup, and dispatch ownership are settled.
