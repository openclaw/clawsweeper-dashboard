## Summary

Close as superseded: the focused mutable `llm_input`/`llm_output` work is now tracked by a narrower PR, while the pre-model tool-result mutation question is tracked as a separate security/API item.

## What This Changes

This PR changes plugin hook contracts and agent/tool call sites so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return blocking or payload-modifying results.

## Best Possible Solution

Review or land a focused hardened mutable `llm_input`/`llm_output` PR, and handle pre-model tool-result mutation separately with explicit API, security policy, result validation, docs, and live proof.

## Reproduction Assessment

Yes for the PR review findings by source inspection; the branch applies prompt rewrites after image detection, uses truthy override checks, leaves original assistant data reachable after output replacement, and changes `after_tool_call` dispatch ownership against current-main guards. Not applicable as a user bug reproduction because this is a feature PR.

## Solution Assessment

No. The feature direction is useful, but the maintainable path is the focused LLM-hook work in https://github.com/openclaw/openclaw/pull/61485 plus a separate tool-result security/API design tracked by https://github.com/openclaw/openclaw/issues/79168.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Recompute prompt artifacts after llm_input rewrites:** `src/agents/pi-embedded-runner/run/attempt.ts:2839-2840`
  - body: `detectAndLoadPromptImages` runs before the new `llm_input` result is applied, but the model call still uses the earlier `imageResult.images`. If a plugin removes or changes prompt-local image references, stale images from the original prompt can still reach the model; move the hook earlier or recompute and roll back all derived prompt state after applying the override.
  - confidence: 0.91
- **[P1] Enforce prompt policy on llm_input rewrites:** `src/plugins/types.ts:1563-1565`
  - body: The new result type lets `llm_input` handlers return `prompt` and `systemPrompt`, but the existing prompt-injection policy is not extended to this mutation path. A plugin configured with `allowPromptInjection=false` could still rewrite model instructions unless this result is constrained before use.
  - confidence: 0.9
- **[P2] Honor empty prompt and system prompt overrides:** `src/agents/pi-embedded-runner/run/attempt.ts:2839-2844`
  - body: The call site checks `llmInputResult.prompt` and `systemPrompt` by truthiness, so intentional empty-string rewrites are ignored. Redaction or guardrail plugins need field-presence checks so `""` can deliberately remove sensitive prompt text or clear a system prompt.
  - confidence: 0.84
- **[P2] Strip original assistant payload after output replacement:** `src/agents/pi-embedded-runner/run/attempt.ts:3177-3178`
  - body: When `llm_output` replaces `assistantTexts`, the returned attempt still exposes `lastAssistant` with the original assistant content and metadata. That can leave unredacted text, reasoning, or error details reachable after a plugin requested response replacement.
  - confidence: 0.84
- **[P2] Preserve exact-once after_tool_call dispatch:** `src/agents/pi-tool-definition-adapter.ts:177-186`
  - body: The adapter now invokes `after_tool_call` before returning the tool result while the subscription handler still emits the same hook after the result is committed. Embedded tool executions can therefore call the same plugin hook twice, contradicting the current exact-once regression guard.
  - confidence: 0.86
- **[P2] Validate replacement tool results before returning them:** `src/agents/pi-tool-definition-adapter.ts:185-186`
  - body: The adapter returns `hookResult.result` with an `as AgentToolResult` cast and no normalization. A malformed plugin result can enter the tool pipeline without the shape checks applied to ordinary tool output, so replacement results should be validated or normalized before returning.
  - confidence: 0.79
- **[P2] Keep injected after_tool_call runners compatible:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:595`
  - body: The subscription handler now calls `runAfterToolCallVoid`, but existing injected runner shapes may only provide `runAfterToolCall`. Supplying that runner will throw before `.catch(...)` can handle it; update the injected runner contract or add a guarded fallback while migrating.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff adds plugin authority over prompts, system prompts, assistant output, and tool results without carrying current policy, artifact, dispatch, and validation guarantees forward.

Concerns:

- **[high] Mutable llm_input bypasses prompt policy:** `src/plugins/types.ts:1563`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields, but the prompt-injection policy is not extended to this new mutation path, so `allowPromptInjection=false` would not prevent these rewrites.
  - confidence: 0.9
- **[high] Prompt override can retain stale image artifacts:** `src/agents/pi-embedded-runner/run/attempt.ts:2839`
  - body: Prompt rewrites happen after prompt-local image detection, so a redaction plugin can change the prompt while images from the original prompt still go to the model.
  - confidence: 0.91
- **[medium] Output override can leave original assistant data reachable:** `src/agents/pi-embedded-runner/run/attempt.ts:3197`
  - body: Replacing `assistantTexts` does not clear or replace `lastAssistant`, leaving original assistant content or metadata available downstream after a plugin requested response redaction.
  - confidence: 0.84
- **[medium] Tool-result replacement is unvalidated:** `src/agents/pi-tool-definition-adapter.ts:185`
  - body: The new adapter path trusts an `unknown` plugin-returned result via a type cast, which can bypass the normal tool-result normalization boundary before the model sees the result.
  - confidence: 0.79
