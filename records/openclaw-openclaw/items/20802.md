## Summary

Keep open for maintainer review. Current main still leaves the requested mutable LLM-hook capability unimplemented, but this PR is broader than the safer remaining work and has concrete security/API blockers around prompt mutation, derived prompt state, output redaction, and adapter-side `after_tool_call` dispatch ownership.

## What This Changes

The PR changes plugin hook contracts and embedded/tool call sites so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can block or modify prompt, response, and tool-result payloads.

## Best Possible Solution

Ship a focused mutable `llm_input`/`llm_output` contract with policy enforcement, payload cloning, prompt image re-resolution or rollback, output cleanup, docs, changelog, and regression tests; split pre-model tool-result mutation into a separate plugin API design with exact-once dispatch ownership and result validation.

## Reproduction Assessment

Yes. Static inspection of current main shows the requested hooks remain void observation hooks, and PR diff inspection gives high-confidence reproduction paths for stale prompt images, prompt-policy bypass, output replacement leakage, and duplicate `after_tool_call` dispatch.

## Solution Assessment

No. The direction is useful, but this PR combines LLM prompt/output mutation with tool-result mutation before the security policy, derived prompt state, redaction cleanup, and dispatch ownership problems are resolved; a focused #61485-like LLM implementation plus a separate `after_tool_call` design is safer.

## Security Review

Status: needs_attention

Summary: The diff introduces prompt and response mutation surfaces without fully carrying existing prompt-injection and redaction safeguards forward.

Concerns:

- **[high] Mutable llm_input bypasses prompt-injection policy:** `src/plugins/types.ts:1563`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields, but current policy only treats existing prompt-building hooks as prompt-injection surfaces. Without matching registry constraints, a plugin configured with `allowPromptInjection=false` can still rewrite model instructions.
  - confidence: 0.9
- **[high] Prompt override can retain stale prompt artifacts:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: The PR applies prompt overrides after prompt-local image detection. For a redaction plugin, stale image artifacts can still be sent unless the implementation moves the hook earlier or recomputes all derived prompt payload state.
  - confidence: 0.9
