## Label Justifications

- `P2`: This is a bounded media-understanding bug affecting scanned PDF attachments across chat channels without making the core runtime unusable.
- `impact:data-loss`: User-provided PDF page image data is extracted and then silently dropped before model delivery on the chat path.
- `impact:message-loss`: The affected content originates from chat-channel attachments and is suppressed from the agent turn rather than delivered as usable message context.

## Summary

Keep open: current main and v2026.6.10 still convert extracted scanned-PDF page images on the chat media-understanding path into a placeholder text block, while the sibling OpenResponses path forwards the same extracted images to the model. No merged or open canonical fix PR owns the chat-path gap.

## What This Changes

The issue requests that chat-channel media understanding forward rendered page images from scanned or image-only PDFs to the model instead of replacing them with a text placeholder.

## Best Possible Solution

Carry extracted PDF page images from chat file extraction into the existing agent inline-image payload path while preserving text wrapping, MIME/size limits, and OpenResponses behavior.

## Reproduction Assessment

Yes, at source level. Current main has a clear path where extractFileContentFromSource can return PDF images and extractFileBlocks converts them to a placeholder string[] appended to ctx.Body; no live channel transport was run.

## Solution Assessment

Yes. The narrow maintainable fix is to pass PDF-derived image payloads into the existing chat agent image plumbing, not to change every channel adapter or alter the document extractor fallback itself.
