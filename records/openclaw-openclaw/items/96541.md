## Label Justifications

- `P2`: The bug silently drops scanned PDF content for chat users but is bounded to media-understanding file forwarding rather than a total runtime outage.
- `impact:data-loss`: Extracted page-image data from user documents is discarded before reaching the model.
- `impact:message-loss`: Attachment content from chat-channel messages is suppressed from the model input even though the message itself is received.

## Summary

Keep open: current main and v2026.6.10 still drop rendered PDF page images on the chat media-understanding path, and the issue should close only after a linked fix PR merges.

## What This Changes

The issue requests forwarding scanned/image-only PDF page rasters from chat-channel media understanding to the model instead of replacing them with a placeholder text block.

## Best Possible Solution

Land the typed current-turn image forwarding fix from https://github.com/openclaw/openclaw/pull/96690, or an equivalent canonical PR, then close this issue after merge.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make extractFileContentFromSource return empty text with at least one image, and current applyMediaUnderstanding appends only the placeholder text block while OpenResponses forwards the image list.

## Solution Assessment

Yes. The maintainable fix is to carry extracted PDF page images through the existing typed ImageContent/current-turn image path for normal and ACP replies, rather than embedding data URIs in prompt text.
