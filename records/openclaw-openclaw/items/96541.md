## Label Justifications

- `impact:data-loss`: User-provided PDF page image data is extracted and then silently dropped before model delivery on the chat path.

## Summary

Keep open: current main and v2026.6.10 still turn extracted scanned-PDF page images on the chat media-understanding path into a placeholder text block, while the sibling OpenResponses path forwards the same extracted images to the model. No merged or open PR found in live GitHub search owns this chat-path gap.

## What This Changes

The issue reports that scanned or image-only PDF page rasters are extracted on chat-channel media-understanding paths but dropped before the model receives them.

## Best Possible Solution

Carry extracted PDF page images from chat file extraction into the existing agent inline-image payload path while preserving text context, extraction limits, and sibling OpenResponses behavior.

## Reproduction Assessment

Yes, at source level. Current main has a clear path where extractFileContentFromSource can return PDF images and extractFileBlocks converts them to a placeholder string[] appended to ctx.Body; no live channel transport was run.

## Solution Assessment

Yes. The narrow maintainable fix is to pass PDF-derived image payloads into the existing chat agent image plumbing, not to change every channel adapter or alter the document extractor fallback itself.
