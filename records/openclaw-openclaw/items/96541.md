## Label Justifications

- `P2`: This is a bounded media-understanding bug affecting scanned PDF attachments across chat channels without making the core runtime unusable.
- `impact:data-loss`: PDF page image data is extracted from the user attachment and then silently dropped before model delivery on the chat path.
- `impact:message-loss`: The affected content originates from chat-channel attachments and is suppressed from the agent turn rather than delivered as usable message context.

## Summary

Keep open: current main and the latest release still drop scanned-PDF page images on the chat media-understanding path, and the issue is now owned by open linked fix PRs rather than a cleanup close.

## What This Changes

The issue asks OpenClaw to forward page images extracted from scanned or image-only PDFs into chat-channel model turns instead of replacing them with a placeholder text block.

## Best Possible Solution

Land the typed current-turn image handoff from the strongest linked PR, then let this issue close from the merged fix.

## Reproduction Assessment

Yes, at source level. Current main can receive PDF-rendered images from extractFileContentFromSource and then emit only a placeholder text block on the chat media-understanding path; I did not run a live chat send in this read-only review.

## Solution Assessment

Yes. The best fix is to carry PDF-derived images through the existing chat agent image plumbing, not to change every channel adapter or alter the document extractor fallback itself.
