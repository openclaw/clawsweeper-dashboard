## Label Justifications

- `P2`: This is a bounded media-understanding bug affecting scanned PDF attachments across chat channels without an outage or security emergency.
- `impact:data-loss`: Extracted PDF page image payloads are silently discarded before model delivery on the chat path.
- `impact:message-loss`: The lost content originates from chat-channel attachments and is suppressed from the agent turn rather than delivered as usable message context.

## Summary

Keep open: current main and the latest release still replace extracted scanned-PDF page images on the chat media-understanding path with a placeholder text block. The work is now owned by open closing PRs, with https://github.com/openclaw/openclaw/pull/96690 currently the stronger proof-positive candidate, so this issue should close only after a fix merges.

## What This Changes

Forward rendered page images from scanned or image-only PDF attachments on chat channels to vision-capable model input instead of replacing them with placeholder text.

## Best Possible Solution

Land the typed current-turn image forwarding path, preferably through the proof-positive candidate if maintainer review agrees, then close the overlapping PR and let this issue close from the merged fix.

## Reproduction Assessment

Yes, at source level. Current main can receive PDF-rendered images from extractFileContentFromSource and then emit only a placeholder text block from extractFileBlocks; I did not run a live channel send in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to carry extracted PDF page images into the existing current-turn image payload path for normal reply and ACP dispatch, rather than embedding base64 data URIs in text or changing every channel adapter.
