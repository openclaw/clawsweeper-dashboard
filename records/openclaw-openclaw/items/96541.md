## Label Justifications

- `P2`: This is a normal-priority media-understanding bug affecting scanned PDF attachments across chat channels without making the core runtime unusable.
- `impact:data-loss`: User-provided PDF page image data is extracted and then silently dropped before model delivery on the chat path.
- `impact:message-loss`: The affected content originates from chat-channel attachments and is suppressed from the agent turn rather than delivered as usable message context.

## Summary

Keep open: current main and the latest release still extract PDF page images on the chat media-understanding path and replace them with a text-only placeholder, while the sibling OpenResponses path forwards the same extracted images to the model.

## What This Changes

The issue asks for scanned or image-only PDF page rasters extracted from chat-channel attachments to be forwarded as model-visible inline images instead of being discarded.

## Best Possible Solution

Carry extracted PDF page images from chat file extraction into the existing agent inline-image payload path while preserving file text wrapping, extraction limits, MIME checks, and OpenResponses behavior.

## Reproduction Assessment

Yes, this is source-reproducible on current main: the extractor can return PDF images, but `extractFileBlocks` only returns text blocks and substitutes the image-present placeholder. I did not run a live channel send in this read-only review.

## Solution Assessment

Yes, the maintainable fix is to carry the extracted images through the existing agent image plumbing, not to change the PDF extractor or every channel adapter. The fix should also update the media-understanding docs that currently describe the limitation.
