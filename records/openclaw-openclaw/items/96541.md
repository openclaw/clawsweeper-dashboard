## Label Justifications

- `P2`: This is a bounded media-understanding bug affecting scanned PDF attachments across chat channels without making the core runtime unusable.
- `impact:data-loss`: User-provided PDF page image data is extracted and then silently dropped before model delivery on the chat path.
- `impact:message-loss`: The affected content originates from chat-channel attachments and is suppressed from the agent turn rather than delivered as usable message context.

## Summary

Keep open: current main and the latest release still drop extracted scanned-PDF page images on the chat media-understanding path, and the only active structural fix is still an unmerged PR with a real-behavior-proof blocker.

## What This Changes

The issue asks for scanned/image-only PDF page rasters extracted by media understanding to be forwarded to chat-channel model turns instead of being replaced by a placeholder.

## Best Possible Solution

Keep this issue open until the structural image-handoff PR merges, or until maintainers choose and land a different typed-image forwarding path for normal chat and ACP turns.

## Reproduction Assessment

Yes, at source level. Current main can receive extracted PDF page images from extractFileContentFromSource, but extractFileBlocks turns them into a placeholder-only text block on the chat media-understanding path; no live channel test was run in this read-only review.

## Solution Assessment

Yes, the issue's requested direction is the best fix: carry extracted PDF page images into typed current-turn image payloads, as the OpenResponses path does, rather than embedding data URIs in prompt text. The active structural PR follows that direction but still needs proof before merge.
