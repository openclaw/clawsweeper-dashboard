- `P2`: This is a clear media-understanding bug affecting scanned PDF attachments across chat channels, with bounded blast radius and no core-runtime outage signal.
- `impact:data-loss`: User-provided PDF page image content is extracted and then silently discarded before model delivery on the chat path.

- `maturity:stable`: The bug explicitly affects Discord chat attachments, and `qa/maturity-scores.yaml` marks the Discord surface as M4 Stable with Media and Rich Content supported.

## Summary

Keep open: current main and the latest release still drop scanned-PDF page images on the chat media-understanding path, and two open PRs now use closing syntax for this issue. The issue should remain the canonical tracker until maintainers land one candidate and close the duplicate.

## What This Changes

Issue reports that scanned or image-only PDF page images extracted for chat-channel messages are replaced with placeholder text instead of being forwarded to vision-capable models.

## Best Possible Solution

Review and land the typed current-turn image handoff from the stronger linked PR, then close the duplicate candidate and let this issue close from the merged fix.

## Reproduction Assessment

Yes, source-reproducible. Current main can receive PDF-rendered images from `extractFileContentFromSource` and then emits only a placeholder text block on the chat media-understanding path; I did not run a live channel send in this read-only review.

## Solution Assessment

Yes. The maintainable fix is to carry extracted PDF page images through the existing typed current-turn image path for normal replies and ACP dispatch, matching the sibling OpenResponses behavior without embedding data URIs in prompt text.
