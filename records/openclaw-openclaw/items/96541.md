## Label Justifications

- `P2`: This is a bounded media-understanding bug affecting scanned PDF attachments across chat channels without making the runtime unusable.
- `impact:data-loss`: User-provided PDF page image data is extracted and then silently discarded before it reaches the model on the chat path.
- `impact:message-loss`: The affected content originates from chat-channel attachments and is suppressed from the agent turn instead of being delivered as usable message context.

## Summary

Keep open: current main and v2026.6.10 still drop rendered scanned-PDF page images on the chat media-understanding path, and the issue now has open closing PR candidates that should be reviewed or selected before this issue closes.

## What This Changes

Scanned or image-only PDF page images extracted for chat-channel attachments should be forwarded to vision-capable model turns instead of being replaced by a placeholder text block.

## Best Possible Solution

Land one typed current-turn image handoff for extracted PDF page images, preferably the best-reviewed candidate if maintainers agree, then close the overlapping PR and let this issue close from the merged fix.

## Reproduction Assessment

Yes, at source level. Current main can receive PDF-rendered images from extractFileContentFromSource and then emits only a placeholder text block from extractFileBlocks on the chat media-understanding path; no live chat send was run in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to carry extracted page images through existing current-turn image plumbing for normal reply and ACP paths, while preserving OpenResponses behavior and existing attachment ordering.
