## Label Justifications

- `P2`: This is a normal-priority media-understanding bug affecting scanned PDF attachments across chat channels without making the core runtime unusable.
- `impact:data-loss`: User-provided PDF page image data is extracted and then silently dropped before model delivery on the chat path.
- `impact:message-loss`: The affected content originates from chat-channel attachments and is suppressed from the agent turn rather than delivered as usable message context.

## Summary

Keep open: current main and the latest release still replace extracted scanned-PDF page images on the chat media-understanding path with a text-only placeholder, while the sibling OpenResponses path forwards the same extractor images to the model.

## What This Changes

The issue requests forwarding rendered page images from scanned/image-only PDF attachments on chat channels instead of silently discarding them behind a placeholder.

## Best Possible Solution

Carry extracted PDF page images from chat file extraction into the existing agent inline-image payload path, while preserving text wrapping, extraction limits, MIME checks, and updating media-understanding docs/tests.

## Reproduction Assessment

Yes, this is high-confidence source-reproducible on current main: the PDF extractor can return images, but the chat media-understanding path only returns text blocks and substitutes the image-present placeholder. I did not run a live channel send in this read-only review.

## Solution Assessment

Yes. The best fix is to carry the already-extracted images through the existing agent inline-image plumbing, not to change the PDF renderer or add per-channel special cases.
