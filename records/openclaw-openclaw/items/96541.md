## Label Justifications

- `P2`: This is a normal-priority media-understanding bug affecting scanned PDF attachments across chat channels without making the core runtime unusable.
- `impact:data-loss`: User-provided PDF page image data is extracted and then silently dropped before model delivery on the chat path.

## Summary

Keep open: current main and the latest release still drop PDF fallback page images on the chat media-understanding path, while the sibling OpenResponses path forwards the same extracted images to the model. No merged or open canonical fix PR owns the remaining work.

## What This Changes

The issue reports that scanned or image-only PDFs attached through chat channels are rasterized into page images but those images are discarded before reaching the model.

## Best Possible Solution

Carry extracted PDF page images from chat file extraction into the existing agent inline-image payload path while preserving file text wrapping, extraction limits, MIME checks, and OpenResponses behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible because extracted PDF images trigger the chat placeholder branch and only text blocks are appended to the chat body; no live channel send was run.

## Solution Assessment

Yes. The narrow fix is to carry extractor-produced images through the existing chat agent image plumbing, not to change every channel adapter or add a new PDF extraction mode.
