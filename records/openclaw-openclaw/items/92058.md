## Summary

Keep open: current main still has source-visible failure-alert gaps around best-effort cron delivery and persisted notification status, so the central operator-alert problem is not safely implemented or superseded. The thread-routing sub-ask is separately tracked by https://github.com/openclaw/openclaw/issues/89848 and its open PR, but this issue still needs maintainer direction for the failureAlert delivery/status contract.

## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on OpenClaw 2026.6.5 record errored runs as `delivery_status: not-requested` and never request failure notifications.

## Best Possible Solution

Define `failureAlert` as an independent failure-notification path, persist its actual attempted/delivered/failed result, and leave the threadId routing sub-ask to the existing related issue and PR.

## Reproduction Assessment

Yes, at source level with medium confidence: current main still skips `sendCronFailureAlert` for jobs whose delivery is `bestEffort`, and it persists failure-notification status before any async alert send can report a result. I did not run the reporter's live macOS gateway and channel-delivery scenario.

## Solution Assessment

Unclear as a final contract: the narrow bug boundary is cron/gateway failure-alert delivery, but maintainers need to decide whether `failureAlert` intentionally overrides `bestEffort` and how public delivery statuses should represent attempted sends.
