## Summary

Keep open: current main partially improves failure-alert emission, but the durable failure-notification status is still not tied to the per-job failureAlert request path, so the reported operator-alert/status problem is not safely implemented or superseded.

## What This Changes

The issue reports cron failure alerts configured with after=1 staying at not-requested after errored runs on v2026.6.5.

## Best Possible Solution

Define failureAlert as an independent failure-notification path and persist durable job/run-log status from the actual alert request outcome, while leaving Telegram thread routing to the related threadId work.

## Reproduction Assessment

Yes, source-level for the remaining status gap: current main can emit a per-job failureAlert while the durable status resolver still records not-requested when primary delivery is not requested and no failureDestination exists. No; current main's alert emission path is useful but not the full fix because durable status still does not follow the per-job failureAlert request outcome.
