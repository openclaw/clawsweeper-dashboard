## Summary

Keep open: current main and v2026.6.9 still persist per-job failure-alert status from the primary delivery/failureDestination resolver, not from the actual `failureAlert` send result, and the linked closing PR is still open as the candidate fix.

## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on OpenClaw 2026.6.5 never request or deliver failure alerts, leaving errored runs at `delivery_status: "not-requested"`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95129 if maintainers accept its status semantics; otherwise keep this issue as the canonical tracker for a narrower replacement fix.

## Reproduction Assessment

Yes, at source level. Current main emits per-job failure alerts asynchronously while persisting failure-notification status from the primary delivery/failureDestination resolver, which leaves the reported no-primary-delivery path at `not-requested`.

## Solution Assessment

No, current main is only a partial solution because it can emit an alert without persisting the actual alert request result. The linked PR is the best current candidate if maintainers accept the bounded wait; otherwise a post-finish persistence update would be the safer design.
