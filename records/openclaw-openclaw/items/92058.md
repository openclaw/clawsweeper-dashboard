## Label Justifications

- `P1`: The latest release can silently suppress configured operational alerts across repeated cron failures, leaving operators unaware of broken scheduled work.
- `impact:message-loss`: The issue is specifically about configured failure notifications that are never requested or delivered.

## Summary

This issue remains necessary because current main still contains a source-level path that suppresses configured failure alerts and records `not-requested`; no merged fix or current-main change removes that behavior. The report also adds ordinary timeout evidence beyond the pre-execution stall case tracked elsewhere.

## What This Changes

Fix cron jobs configured with `failureAlert.after = 1` so terminal errors request and deliver the configured notification instead of persisting `not-requested`.

## Best Possible Solution

Define explicit failure alerts independently from the primary delivery's best-effort preference, then add a migrated-store regression proving an ordinary timeout invokes the alert sender and persists an attempted delivery status.

## Reproduction Assessment

Yes, with medium confidence: configure primary delivery with `bestEffort: true`, set `failureAlert.after: 1`, force an error, and current main returns before requesting the alert. Confirming that flag on the reported jobs and reproducing through a migrated SQLite row would make the path high-confidence.

## Solution Assessment

No, silently inheriting the primary delivery's best-effort setting as complete suppression of an explicitly configured failure alert is not the clearest maintainable contract. Maintainers should confirm the intended policy, then remove the coupling or introduce a separately owned alert policy rather than adding a runtime fallback.
