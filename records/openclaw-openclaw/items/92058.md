## Summary

Keep open: current main and the latest release partially cover failure-alert emission, but durable run/job failure-notification status is still resolved before the per-job `failureAlert` send and can remain `not-requested`, so the reported audit/status problem is not implemented on main.

## What This Changes

The issue reports that cron jobs configured with `failureAlert.after = 1` on v2026.6.5 never request or deliver failure alerts, leaving errored runs and job state at `not-requested`.

## Best Possible Solution

Define `failureAlert` as its own failed-run notification path and persist job/run-log status from the actual alert request outcome, while leaving Telegram `threadId` routing to the linked narrower work.

## Reproduction Assessment

Yes, at source level: current `main` can emit a per-job alert after an error while still persisting failure-notification status from `resolveDeliveryState`, which returns `not-requested` when primary delivery is not requested and no failure destination exists. I did not run a live gateway/channel reproduction.

## Solution Assessment

No, current `main` is only a partial solution: it covers alert emission for some ordinary errors, but not the durable request/status contract the reporter used to identify the failure.
