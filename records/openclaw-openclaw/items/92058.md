## Summary

Keep open: current main still has a source-visible gap between per-job `failureAlert` emission and the persisted/run-log failure-notification state, and no merged related PR or release proves the latest-release report is fixed.

## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on v2026.6.5 never request or deliver failure-alert notifications and leave errored runs at `delivery_status = 'not-requested'`.

## Best Possible Solution

Keep this open as the broader latest-release failure-alert tracking item, fix the cron/gateway alert request-result persistence path, and coordinate the runtime-stall and thread-routing pieces through https://github.com/openclaw/openclaw/issues/89847 and https://github.com/openclaw/openclaw/issues/89848.

## Reproduction Assessment

Yes from source for the request-state mismatch: current main can emit failure alerts through a fire-and-forget sender while persisted failure-notification state is resolved earlier and can remain `not-requested`. No live cron/Telegram reproduction was run in this read-only review.

## Solution Assessment

No as one bundled change: the core bug belongs in cron/gateway failure-alert state tracking, while explicit delivery-attempt enums, `failureAlert.threadId`, and cron-list highlighting are public schema/config/UX decisions that should be split or approved.
