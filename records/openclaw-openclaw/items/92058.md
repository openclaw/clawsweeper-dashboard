## Summary

Keep open: current main and v2026.6.6 still have source-visible gaps where configured cron failure alerts can be skipped for best-effort jobs and the persisted failure-notification status is computed before the async alert send can report any outcome. The related Telegram thread-routing work is tracked by https://github.com/openclaw/openclaw/issues/89848 and https://github.com/openclaw/openclaw/pull/90887, but this issue still owns the broader failureAlert delivery/status contract.

## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on v2026.6.5 never request or deliver alerts after errored runs, leaving run logs and job state at `not-requested`.

## Best Possible Solution

Define `failureAlert` as an independent failure-notification path, persist the actual attempted/delivered/failed outcome, and leave the threadId routing sub-ask to https://github.com/openclaw/openclaw/issues/89848 and its open PR.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The source path is reproducible with medium confidence because current main still skips best-effort failure alerts and persists status before any async alert result is known.

## Solution Assessment

Unclear as a final contract. The narrow fix belongs in cron/gateway failure-alert delivery and state persistence, but maintainers should first decide whether `failureAlert` intentionally overrides `bestEffort` and what public status vocabulary should mean.
