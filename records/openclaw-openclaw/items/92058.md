## Summary

Keep open: current main and v2026.6.6 still have source-visible paths where best-effort cron jobs suppress `failureAlert` sends and persist failure-notification state before any async alert result can be known. Related work covers runtime-stall alerts and thread routing, but no canonical issue or merged fix owns this broader `not-requested` failure-alert contract.

## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on v2026.6.5 never request or deliver failure alerts and persist `not-requested` statuses after errored runs.

## Best Possible Solution

Define `failureAlert` as an independent operator-alert path, make persisted failure-notification state reflect the actual requested/sent/failed result, and keep thread routing under the existing related issue and PR.

## Reproduction Assessment

Yes at source level with medium confidence: current main still suppresses failure alerts for best-effort cron jobs and persists failure-notification status before the async alert path can report a result. I did not run the reporter's macOS launchd plus real channel reproduction.

## Solution Assessment

Unclear as a final contract: the fix likely belongs in cron/gateway failure-alert delivery and state persistence, but maintainers need to decide whether `failureAlert` intentionally overrides `bestEffort` and whether the status enum should gain attempted/not-attempted vocabulary.
