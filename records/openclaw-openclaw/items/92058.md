## Label Justifications

- `P1`: Multiple scheduled jobs can fail repeatedly without their configured operator notification, leaving active workflows silently broken.
- `impact:message-loss`: The reported failure is suppression of configured operational alert messages.

## Summary

Current main and v2026.6.5 both contain a real path that silently suppresses cron failure alerts when primary delivery is best-effort, but the issue omits that decisive job field and treats `lastFailureNotificationDeliveryStatus` as proof for a separate legacy alert path; keep open for a config-complete live reproduction and maintainer decision on the intentionally shipped best-effort policy.

## What This Changes

Ensure per-job cron failure alerts configured with `after = 1` are requested and delivered after terminal errors, with accurate persisted delivery status.

## Best Possible Solution

Converge threshold alerts and failure destinations on one scheduler-owned delivery-result contract, migrate any legacy best-effort placement into the canonical delivery shape, and persist an explicit suppression or delivery outcome instead of leaving parallel behavior reported as `not-requested`.

## Reproduction Assessment

Partially yes: current source gives a deterministic reproduction when an errored job has `delivery.bestEffort: true`, but the issue's job snapshots omit that field and no current-main live run confirms the broader ordinary-error claim.

## Solution Assessment

No, the status-field hypothesis alone is not the best diagnosis because two notification mechanisms are involved; the maintainable fix is to unify their delivery result and make the intended best-effort policy explicit.
