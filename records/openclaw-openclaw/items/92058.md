## Label Justifications

- `P1`: The current policy can leave operators unaware of repeated failed scheduled jobs by suppressing all configured failure notifications.
- `impact:message-loss`: The issue concerns operational alert messages that are suppressed before any delivery request is made.

## Summary

Current main and v2026.6.5 both intentionally suppress failure alerts when a cron job’s primary delivery is `bestEffort`, which explains the reported `not-requested` state but is not yet proven for these exact jobs because their complete delivery blocks are absent. The issue remains necessary because no canonical fix exists and changing this tested behavior requires an explicit compatibility decision.

## What This Changes

Make configured cron failure alerts request and deliver a notification after the first errored run instead of remaining `not-requested`.

## Best Possible Solution

Treat an explicitly configured failure alert as an independent notification path, preserving its threshold, cooldown, destination, and explicit disable controls regardless of primary-delivery best-effort mode, unless maintainers intentionally document the current coupling.

## Reproduction Assessment

Yes at source level for a job with `delivery.bestEffort: true`: configure `failureAlert.after: 1`, force an error, and the code and focused test show that no alert is sent and the alternate notification is not requested. The exact reported setup still needs its full delivery blocks to confirm that branch was active.

## Solution Assessment

Unclear pending maintainer policy judgment: decoupling explicit failure alerts from primary-delivery best-effort is the narrowest maintainable design, but it changes deliberate tested semantics and needs upgrade-focused confirmation.
