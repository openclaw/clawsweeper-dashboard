## Summary

This issue is still necessary: v2026.6.5 contains the reported alert/status split, and current main still suppresses an explicit failure alert when the job’s primary delivery is best-effort while deriving persisted failure-notification status from a separate destination mechanism. The related report at https://github.com/openclaw/openclaw/issues/89847 covers a narrower pre-execution stall case, so it does not supersede ordinary timeout failures reported here.

## What This Changes

Fix cron failure alerts that are never requested or delivered after the configured first errored run.

## Best Possible Solution

Use one scheduler-owned failure-notification path to resolve policy, request and observe delivery, and persist that same outcome across ordinary timeout and pre-execution terminal errors, with focused best-effort and migrated-store coverage.

## Reproduction Assessment

Yes, from source: a failing job with `failureAlert.after = 1` and primary `delivery.bestEffort = true` deterministically exits before alert emission, while the separate status resolver can record `not-requested`. The reporter’s exact migrated macOS setup still needs a live run to establish whether this explains every observation.

## Solution Assessment

No. Splitting alert eligibility, fire-and-forget sending, and persisted request status across different policy mechanisms is not the best solution; they should share one scheduler-owned result path.
