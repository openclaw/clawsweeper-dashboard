## Summary

Keep open: current main still only partially covers failure-alert emission, while durable run/job failure-notification status is still resolved from primary delivery or `failureDestination` rather than from the `failureAlert` request outcome.

## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on v2026.6.5 never request or deliver failure notifications and leave run/job delivery status at `not-requested`.

## Best Possible Solution

Define `failureAlert` as an independent failed-run notification path whose persisted run/job status comes from the actual alert request outcome, while leaving Telegram thread routing to the narrower linked work.

## Reproduction Assessment

Yes, at source level. Current main can emit a per-job failure alert while persisting `not-requested` or `unknown` from primary delivery/failureDestination state rather than the actual `failureAlert` send outcome.

## Solution Assessment

No, current main is only a partial solution. The maintainable fix is to persist failure-alert request/delivery status from the actual alert path and split status semantics from the separate threadId routing work.
