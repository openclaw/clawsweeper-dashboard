## Summary

Keep open: current main and the latest release still derive durable failure-notification status from primary delivery/failureDestination state rather than the actual per-job `failureAlert` request outcome, and the linked closing PR is still open as the candidate fix.

## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on v2026.6.5 never request failure alerts, leaving errored runs and job state at `not-requested`.

## Best Possible Solution

Review and land the linked candidate PR if maintainers accept its status semantics; otherwise keep this issue as the canonical tracker for a narrower replacement fix.

## Reproduction Assessment

Yes, at source level. Current main can emit a per-job failure alert asynchronously while persisting failure-notification status from primary delivery/failureDestination state instead of the alert request result.

## Solution Assessment

No, current main is only a partial solution. The best fix is to persist failure-alert delivery status from the actual alert request outcome while keeping Telegram thread routing in the separate linked work.
