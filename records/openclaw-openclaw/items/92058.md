## Summary

Current main does not clearly resolve the report. It still deterministically suppresses failure alerts when the job’s primary delivery uses `bestEffort: true`, while documentation presents failure notifications as a separate destination path; however, the reporter’s complete job delivery configuration is unavailable, so that branch is a strong explanation rather than a confirmed root cause for every affected job.

## What This Changes

Restore configured cron failure-alert requests after eligible job errors instead of leaving their delivery status as `not-requested`.

## Best Possible Solution

Define explicit failure alerts independently from primary-output delivery tolerance, then ensure every eligible terminal error reaches that scheduler-owned path and persists the actual alert attempt result without changing the original run outcome.

## Reproduction Assessment

Partially. Current main deterministically reproduces `not-requested` behavior for failing jobs whose primary delivery has `bestEffort: true`, but the issue lacks the full stored configuration needed to prove that this explains every reported job.

## Solution Assessment

No definitive fix is established yet. The narrowest likely solution is to separate explicit failure-alert policy from primary best-effort delivery, but a SQLite-backed reproduction must also rule out migrated-config or pre-execution outcome loss.
