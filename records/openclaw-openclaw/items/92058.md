## Label Justifications

- `P1`: Multiple real cron failures reportedly produced no configured operator notification, leaving an important scheduled-work workflow silently broken.
- `impact:message-loss`: The affected behavior suppresses configured cron failure-alert messages before delivery is requested.

## Summary

The report remains necessary: both v2026.6.5 and current main deliberately skip a configured failure alert when the job’s primary delivery uses `bestEffort: true`, which matches the reported persistent `not-requested` state, but the issue omits the affected jobs’ full delivery blocks so the exact live trigger is not yet confirmed.

## What This Changes

Fix cron jobs with `failureAlert.after = 1` recording failed runs as `not-requested` without sending the configured alert on v2026.6.5.

## Best Possible Solution

Make failure-alert suppression an explicit failure-alert policy rather than inheriting it silently from primary delivery, persist a distinguishable suppression or delivery result, and cover ordinary timeouts plus pre-execution failures with upgrade-safe tests.

## Reproduction Assessment

Yes at source level for jobs with `delivery.bestEffort: true`: force an errored run with `failureAlert.after = 1`, and current code deterministically skips the sender while retaining `not-requested`. The issue needs the full affected job JSON or a live run to confirm that exact condition on the reporter’s installation.

## Solution Assessment

Unclear pending a product decision. Removing the implicit coupling is the narrowest maintainable direction, but maintainers must decide whether existing `bestEffort` users intentionally opted out of failure alerts and what upgrade behavior is acceptable.
