## Label Justifications

- `P1`: Configured cron failure alerts can be suppressed for real errored runs, breaking an urgent scheduled-workflow operator signal.
- `impact:message-loss`: The reported behavior is loss or suppression of configured failure-alert channel messages for cron failures.

## Summary

Keep open: current main and the latest release still encode failure-alert suppression for best-effort cron jobs and do not persist an actual async failure-alert send result, so the reported `not-requested` operator-alert gap is not implemented or superseded.

## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on OpenClaw 2026.6.5 never request failure notifications after errored runs, leaving persisted delivery status at `not-requested`.

## Best Possible Solution

Define `failureAlert` as an independent failure-notification path for errored runs that meet the threshold, persist the actual requested/delivered/failed result, and keep thread routing plus runtime-stall root cause work in their related items.

## Reproduction Assessment

Yes at source level with medium confidence: current main still has a bestEffort early return and tests that preserve `not-requested` failure-notification state. I did not run the reporter's live macOS gateway and channel-delivery scenario.

## Solution Assessment

No, current main is not the best final behavior because it can suppress a configured alert and persist status before the alert send has a result. The safer fix needs a maintainer-approved failureAlert delivery/status contract before code changes.
