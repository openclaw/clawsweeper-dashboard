## Label Justifications

- `P2`: Configured operational failure alerts can be silently suppressed, but the known impact is bounded to cron notification visibility rather than cron execution itself.
- `impact:message-loss`: The issue concerns configured failure notifications that remain unrequested and never reach their channel target.

## Summary

Current main still contains a deterministic suppression path that can explain the reported `not-requested` state: configured failure alerts are skipped when the job’s primary delivery is marked best-effort. The issue should remain open because no merged fix or canonical replacement was found, although the reporter’s affected job configuration needs one confirmation before treating this as a fully reproduced broad regression.

## What This Changes

Fix cron jobs whose configured `failureAlert.after = 1` never requests or delivers an alert after an errored run.

## Best Possible Solution

Make an explicitly configured failure alert independent of the primary delivery’s best-effort setting, then cover ordinary execution timeout and pre-execution failure outcomes while preserving threshold and cooldown behavior.

## Reproduction Assessment

Yes at source level: a qualifying failure with `delivery.bestEffort: true` deterministically returns before requesting its configured failure alert and leaves the persisted status `not-requested`. A live current-main run or the reporter’s full job JSON is still needed to confirm this explains every affected job.

## Solution Assessment

Yes, subject to confirming intended best-effort semantics: removing the unrelated primary-delivery suppression from the existing scheduler-owned alert path is narrower and more maintainable than adding migration logic or a second notification mechanism.
