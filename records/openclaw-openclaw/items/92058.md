## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on OpenClaw 2026.6.5 never request or deliver failure alerts and persist failure-delivery state as `not-requested`.

## Best Possible Solution

Keep this open until maintainers decide the failure-alert delivery-state contract, then land a focused cron/gateway fix with regression coverage for failureAlert-only errored runs through job state and SQLite run logs.

## Reproduction Assessment

Yes for the persisted status gap from source: current main computes and stores failure-notification state before the async failure-alert send result is known. No for live channel delivery: this review did not run a real cron/channel reproduction.

## Solution Assessment

Unclear. A narrow code fix is plausible, but maintainers should first decide whether failureAlert sends should be awaited and persisted as delivered/not-delivered/unknown or represented with a new explicit requested/attempted state.
