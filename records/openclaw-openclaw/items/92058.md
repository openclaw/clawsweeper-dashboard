- `impact:message-loss`: The affected behavior is loss or suppression of configured failure-alert channel messages for cron failures.

## Summary

Keep open: current main partially covers per-job failure-alert emission, but the durable run/job failure-notification status is still resolved from primary delivery or `failureDestination`, not from the `failureAlert` request path, so the reported `not-requested` audit/status problem remains.

## What This Changes

The issue reports that cron jobs with `failureAlert.after = 1` on v2026.6.5 never request failure-alert delivery, leaving errored runs at `delivery_status: "not-requested"`.

## Best Possible Solution

Define `failureAlert` as its own failed-run notification path and persist job/run-log status from the actual alert request outcome, while leaving Telegram `threadId` routing to the linked narrower work.

## Reproduction Assessment

Yes, at source level: current main can emit a per-job failure alert while the durable status resolver still records `not-requested` when primary delivery is not requested and no `failureDestination` exists. I did not run a live macOS gateway/channel reproduction in this read-only review.

## Solution Assessment

No, current main is only a partial solution: it covers alert emission for some ordinary errors, but not the durable request/status contract the reporter used to identify the failure.
