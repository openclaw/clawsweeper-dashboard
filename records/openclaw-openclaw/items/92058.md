## Label Justifications

- `P1`: Configured operational alerts can be silently suppressed across repeated cron failures, leaving real broken workflows undiscovered.
- `impact:message-loss`: The issue concerns failure notifications that are neither requested nor delivered to the configured channel.

## Summary

The issue remains necessary: current main still intentionally suppresses failure alerts for jobs with `delivery.bestEffort: true`, while the public configuration docs describe `failureAlert.after` without warning that this primary-delivery setting disables alerts.

## What This Changes

Investigate why cron jobs configured with `failureAlert.after = 1` record errored runs as `not-requested` and never send the configured alert after upgrading to v2026.6.5.

## Best Possible Solution

Make failure-alert policy independent and explicit: either deliver configured alerts regardless of primary `delivery.bestEffort`, or add a dedicated alert-level control, with upgrade-safe tests and documentation for migrated and fresh jobs.

## Reproduction Assessment

Yes from source with high confidence: set `failureAlert.after = 1`, set primary `delivery.bestEffort = true`, force an error, and the tested current path returns before requesting the alert. A live migrated v2026.6.5 store was not reproduced during this read-only review.

## Solution Assessment

No. Primary-delivery tolerance silently overriding an independently configured alert policy is not the clearest solution; the maintainable direction is an explicit failure-alert contract with focused fresh-install and upgrade coverage.
