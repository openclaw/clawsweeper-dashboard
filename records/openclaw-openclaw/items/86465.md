## Summary

Keep open: current main still routes Windows host exec through PowerShell `-Command`, and the docs describe a shell command without explaining the reported `$word` expansion behavior. The bare-variable part is source-plausible, but the quoted and escaped cases need real Windows proof before deciding between a behavior fix and documentation.

## What This Changes

The issue reports that the exec tool removes `$variable`-style tokens from shell command strings before execution, and asks for verbatim command delivery or documentation of the substitution behavior.

## Best Possible Solution

Add focused Windows proof for literal-dollar handling, then either preserve literal `$` tokens through the exec path with regression coverage or document PowerShell expansion and the supported workaround.

## Reproduction Assessment

Partially: current source gives a high-confidence path for bare `$name` behavior on Windows because host exec uses PowerShell `-Command`. The quoted and escaped variants still need a real Windows run against current main.

## Solution Assessment

Unclear: preserving literal command text would be the cleanest user-facing result, but maintainers should first decide whether Windows exec should keep PowerShell semantics and document them or change the execution strategy.
