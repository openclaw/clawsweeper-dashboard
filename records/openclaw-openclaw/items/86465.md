## Summary

Keep open: current main still routes Windows gateway exec through PowerShell `-Command`, and the newer dollar-token handling rejects expansion-sensitive strings for allowlist enforcement rather than proving the model-facing exec command reaches PowerShell verbatim. The issue is plausible and recurring, but it still needs live Windows proof and a maintainer decision on whether to preserve literal payloads, document the limitation, or solve it through the broader shell-selection design.

## What This Changes

The issue asks OpenClaw to stop silently stripping dollar-variable patterns in `exec()` command strings or document the substitution/expansion behavior.

## Best Possible Solution

Define the Windows exec command-fidelity contract, then either preserve literal PowerShell payloads in the default path with focused Windows proof and docs or explicitly document the limitation and route richer shell selection through the maintainer-approved shell-override design.

## Reproduction Assessment

No high-confidence live Windows reproduction was established. Source inspection and related reports show the likely path through PowerShell `-Command`, but current-main behavior for each quoted and escaped dollar pattern still needs real Windows proof.

## Solution Assessment

Unclear. A narrow runtime fix may be possible, but choosing between verbatim payload preservation, script-file execution, documentation, or a broader shell-selection design changes the command-execution contract and needs maintainer direction.
