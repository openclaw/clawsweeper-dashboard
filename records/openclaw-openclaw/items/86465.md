## Summary

Keep open: current main and latest v2026.6.6 still route the model-facing agent `exec` tool through the shell-command path on Windows, while the related argv/plugin command fix does not cover this surface. The discussion and older related reports support a real Windows command-fidelity problem, but the exact quoted/escaped behavior still needs Windows proof before maintainers choose runtime behavior versus documentation.

## What This Changes

The issue reports that the model-facing `exec` tool drops dollar-prefixed variable patterns from command strings on Windows.

## Best Possible Solution

Keep this open as the canonical remaining agent-exec Windows command-fidelity report, then use Windows proof to decide between a runtime shell/script execution change and explicit Windows command-contract documentation.

## Reproduction Assessment

Partly; current source clearly routes the model-facing Windows agent exec tool through PowerShell `-Command`, and related discussion has multiple Windows reports. The single-quote, backtick, double-dollar, and `cmd /c` claims still need current-main Windows proof.

## Solution Assessment

Unclear; preserving literal command text is best if OpenClaw is rewriting arguments, but adding a script-file/shell mode or changing the Windows default requires a compatibility/product decision because PowerShell routing was introduced deliberately.
