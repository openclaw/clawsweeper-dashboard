## Summary

Keep open: current main and latest release v2026.6.8 still route Windows gateway exec through PowerShell `-Command`, while the newer Windows allowlist work rejects dollar-expansion tokens for policy enforcement rather than proving the model-facing exec command reaches PowerShell verbatim. The related reports show recurring Windows `$` command-fidelity pain, but a live Windows proof and maintainer choice are still needed before deciding between runtime behavior, a documented limitation, or a broader shell-selection design.

## What This Changes

The issue asks OpenClaw to stop silently stripping dollar-variable patterns from Windows exec command strings, or to document the substitution behavior.

## Best Possible Solution

Define the Windows exec command-fidelity contract, then either preserve literal PowerShell payloads in the default exec path with focused Windows tests/docs or explicitly document the limitation and route richer shell selection through a maintainer-approved design.

## Reproduction Assessment

No high-confidence live Windows reproduction was established in this review. Source and related reports show the likely path: Windows gateway exec still uses PowerShell `-Command`, which can consume `$...` before nested PowerShell payloads see it, but the exact current-main behavior matrix needs live proof.

## Solution Assessment

Unclear. A narrow runtime fix may be possible, but choosing between verbatim payload preservation, `.ps1`/file execution, documentation, or a broader shell override changes the command-execution contract and needs maintainer direction.
