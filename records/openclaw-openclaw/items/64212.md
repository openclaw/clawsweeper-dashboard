## Summary

Keep open. Current main has generic custom-provider image routing and configurable image timeouts, but there is still no proof that NVIDIA NIM Kimi K2.5 image requests succeed; the bundled NVIDIA catalog remains text-only and the shared image path can still abort slow completions at the resolved timeout.

## What This Changes

The issue asks OpenClaw to make the image tool work for a custom NVIDIA NIM Kimi K2.5 provider configured as image-capable instead of aborting after about 70 seconds.

## Best Possible Solution

Keep this open until maintainers verify the exact NVIDIA NIM Kimi K2.5 image path, then document the required timeout/provider metadata, adjust timeout policy, fix OpenAI-compatible image payload handling, or return a clear unsupported-provider error.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection shows a plausible current-main timeout-abort path for a provider that takes longer than the resolved image timeout, but the exact NVIDIA NIM Kimi K2.5 image request was not verified.

## Solution Assessment

Unclear. The existing configurable image timeout is a reasonable generic mitigation, but the best fix for this issue depends on live/provider triage to separate timeout defaults from image payload handling or unsupported-model behavior.
