## Summary

Keep open: the requested descriptive avatar-tail parsing is still absent from current main and v2026.6.9, but this PR is not merge-ready because the new delimiter heuristic can silently rewrite existing direct avatar paths and the external PR still lacks real behavior proof.

## What This Changes

The PR adds avatar-specific parsing that extracts a trailing path, URL, or data URI from descriptive `IDENTITY.md` avatar lines and adds one parser regression test.

## Best Possible Solution

Keep the parser change only after rebasing and making extraction compatibility-safe: preserve full valid avatar values first, then apply constrained descriptive-tail parsing with regression tests and real Control UI proof.

## Reproduction Assessment

Yes, by source inspection for the PR-introduced regression: a value such as `avatars/hawk - final.png` or `avatars/hawk—final.png` is a direct workspace-relative avatar path today, but the PR can return only `final.png`. I did not run tests because this review is read-only.

## Solution Assessment

No. Parsing descriptive avatar text is a plausible place to solve the useful bug, but the implementation should first preserve any full valid avatar source and only then attempt a constrained descriptive-tail extraction.
