## Summary

Keep open: the parser tolerance is plausible, but the current patch can truncate existing direct avatar values containing spaced hyphens or en/em dashes, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds `extractAvatarValue` to parse trailing avatar URL/path/data URI candidates from descriptive `IDENTITY.md` avatar lines and adds a parser regression test.

## Best Possible Solution

Implement avatar extraction so existing direct URL, data URI, and local path values are preserved while descriptive prose is parsed only when the trailing source is clearly separated, with regression coverage for both prose and dash-containing filenames.

## Reproduction Assessment

Yes, at source level. Current main preserves the whole post-colon avatar string, while the PR’s separator scan can return `final.png` from a direct value like `avatars/hawk - final.png` because that tail passes `looksLikeAvatarPath`.

## Solution Assessment

No. The requested parser tolerance can be useful, but this implementation applies the tail heuristic too broadly and should be narrowed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve direct avatar values before extracting tails:** `src/agents/identity-file.ts:45`
  - body: The new helper scans dash separators on every avatar value and returns the last candidate that looks avatar-like. Existing direct values such as `avatars/hawk - final.png` or `avatars/hawk—final.png` would be parsed as `final.png`, because `looksLikeAvatarPath` accepts image filename tails, breaking avatar resolution for valid workspace-relative paths.
  - confidence: 0.92
