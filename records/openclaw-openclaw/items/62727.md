## Summary

Keep open: current main does not implement descriptive-tail avatar parsing, but this PR is not merge-ready because it can truncate existing direct avatar paths with dashes and lacks real behavior proof.

## What This Changes

The PR adds avatar-specific IDENTITY.md parsing that extracts a trailing URL/path/data URI from descriptive Avatar lines and adds one parser regression test.

## Best Possible Solution

If maintainers want this tolerance, preserve direct URL, data URI, and local path values before descriptive-tail parsing, then cover both descriptive tails and dash-containing direct filenames with proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: the PR scans dash separators on every avatar value, and current looksLikeAvatarPath accepts final.png, so a direct avatars/hawk - final.png value can be truncated. I did not run PR-head tests in this read-only pass.

## Solution Assessment

No. The useful direction is to preserve already-valid avatar values before applying any descriptive-tail heuristic, then prove both descriptive-tail and dash-containing filename cases.
