## Summary

Keep open. The requested parser behavior is still useful and is not implemented on current main, but this branch should not merge as-is because the separator extraction can truncate valid direct avatar paths containing dash-like filename characters, and the user-facing fix also needs a changelog entry.

## What This Changes

The PR updates `parseIdentityMarkdown` to extract a trailing avatar path, URL, or data URI from descriptive `Avatar:` lines and adds regression coverage for one descriptive local-avatar path.

## Best Possible Solution

Revise the parser so direct URL, data URI, and valid direct avatar path values are returned unchanged before any descriptive-tail extraction. Then extract a trailing avatar source only from clearly descriptive text, with focused tests for descriptive lines and dash-containing direct filenames.

## Reproduction Assessment

Yes. The PR helper scans every avatar value for dash separators, so `avatars/hawk - final.png` or `avatars/hawk—final.png` can produce the candidate `final.png`, which `looksLikeAvatarPath` accepts; current main would preserve the full value.

## Solution Assessment

No. The direction is reasonable, but the proposed implementation should preserve direct avatar values first and only apply delimiter extraction to descriptive prose; that is the narrower maintainable fix and keeps the existing resolver as the security boundary.
