## Summary

Keep open: the intended parser tolerance is useful, but the PR is not merge-ready because it applies dash-tail extraction to all avatar values, which can break existing direct avatar paths, and it still lacks real behavior proof.

## What This Changes

The PR adds avatar-specific IDENTITY.md parsing to extract a trailing URL/path/data URI from descriptive Avatar lines and adds parser regression coverage.

## Best Possible Solution

Preserve already-valid avatar URLs, data URIs, and workspace paths before any descriptive-tail heuristic, then cover both descriptive tails and dash-containing direct filenames with regression tests and real Control UI proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: the PR scans dash separators on every avatar value, and current looksLikeAvatarPath accepts final.png, so a direct avatars/hawk - final.png value can be truncated. I did not execute tests in this read-only review.

## Solution Assessment

No. The useful direction is to preserve already-valid avatar values before applying any descriptive-tail heuristic, then prove both descriptive-tail and dash-containing filename cases.
