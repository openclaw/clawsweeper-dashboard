- `P1`: A stale Telegram ingress claim can block a released channel workflow and stop user-visible replies from being delivered.
- `impact:message-loss`: Telegram updates can be accepted into the ingress spool but remain queued without replies while a stale claim blocks the lane.
- `impact:session-state`: The failure is caused by persisted Telegram spool claim state surviving gateway recreation and drifting from the current process identity.

## Summary

Keep open. Current `main` still has the stale-claim gap: a fresh `.json.processing` claim with a different process id and a PID that exists in the recreated container is treated as live-owned, and the drain path then leaves later same-lane Telegram updates queued behind it.

## What This Changes

The issue requests recovery for Telegram isolated ingress `.json.processing` spool claims that survive gateway recreation and block later updates.

## Best Possible Solution

Fix the Telegram plugin’s spool claim ownership and recovery model so restart-stale `.processing` claims are recovered during startup/drain while genuinely live other-process claims remain protected, with focused regression coverage for mismatched process identity and PID reuse.

## Reproduction Assessment

Yes, at source level. A `.json.processing` claim with a mismatched `processId`, `processPid === process.pid`, and fresh `claimedAt` is treated as live-owned, and the existing test shows a later same-lane `.json` remains pending behind it.

## Solution Assessment

Yes, the maintainable fix belongs in Telegram’s plugin-local spool recovery and ownership check. The repair should distinguish current gateway instance identity from stale PID reuse rather than changing core gateway routing or adding a config knob.
