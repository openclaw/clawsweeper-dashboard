## Summary

Keep open. Current main and v2026.6.8 still treat a fresh different-process Telegram spool claim with a live reused PID as owned by another process, and the direct fix PR is still open and conflicting rather than merged.

## What This Changes

The issue reports that a stale Telegram isolated ingress processing claim can survive gateway recreation and block later Telegram updates in the same spool.

## Best Possible Solution

A merged focused fix should make prior-process/current-PID Telegram spool claims recoverable while preserving protection for genuinely live sibling owners, with regression coverage and Docker/Telegram proof.

## Reproduction Assessment

Yes. Current source and tests show that a Telegram spool claim with a different owner id, current process PID, and age under the six-hour freshness window is still treated as live-owned, which blocks lane recovery without needing a live Docker run to see the code path.

## Solution Assessment

No current-main solution is complete. The narrow Telegram owner-identity predicate change in the direct PR is the best-shaped fix for this issue; the broader SQLite ingress startup sweep is adjacent recovery work, not a replacement.
