- `impact:message-loss`: Telegram updates are accepted and queued but replies are not delivered while later same-lane updates remain blocked.

## Summary

Keep open. Current main still has the reported Telegram stale-claim gap: a fresh `.json.processing` entry with a different gateway `processId` and a reused live PID is treated as live-owned, so later same-lane updates can remain queued behind it. The linked fix PR is the right implementation candidate, but it is still open and unmerged.

## What This Changes

The issue requests recovery for Telegram isolated polling ingress `.json.processing` claims left behind after gateway recreation so later queued Telegram updates are not blocked indefinitely.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84690 or an equivalent focused Telegram plugin fix that recovers current-PID/different-process-id restart claims while preserving live other-process protection.

## Reproduction Assessment

Yes, at source level. Current main has a clear ownership predicate and test fixture where a fresh `.json.processing` claim with `processPid === process.pid` and a different `processId` remains unrecovered, leaving later same-lane updates pending.

## Solution Assessment

Yes. The maintainable fix belongs in the Telegram plugin’s spool ownership and recovery path, distinguishing restart-stale current-PID claims from genuinely live other-process claims without changing core gateway routing.
