## Label Justifications

- `P1`: A stale Telegram ingress claim can block a real channel workflow and stop replies from being delivered in current released Docker setups.
- `impact:message-loss`: Telegram updates can be accepted into the spool while later same-lane updates remain queued and no replies are delivered.
- `impact:session-state`: The failure is caused by stale persisted ingress claim state surviving gateway recreation.

## Summary

Keep open. Current main still has the reported stale-claim gap: a fresh `.json.processing` claim with a different process id and a live or reused PID is treated as owned by another live process, so recovery skips it and later same-lane Telegram updates can remain queued behind it.

## What This Changes

Recover Telegram isolated ingress `.json.processing` claims left by a gateway recreate so stale claims cannot indefinitely block later updates.

## Best Possible Solution

Fix the Telegram plugin’s spool claim ownership model so restart-stale `.processing` claims are recovered on startup or drain while genuinely live peer-process claims remain protected.

## Reproduction Assessment

Yes, at source level. A fresh `.json.processing` claim with a mismatched `processId` and `processPid === process.pid` is treated as live-owned, and the existing test leaves a later same-lane `.json` pending behind it.

## Solution Assessment

Yes, the maintainable fix belongs in Telegram’s plugin-local spool recovery and ownership check. It should distinguish current gateway instance identity from stale PID reuse rather than changing core gateway routing.
