## Summary

Keep open. Current main and the latest release still treat a fresh Telegram claim from a different process identity as live-owned when the recorded PID exists, which leaves the reported container PID-reuse/gateway-recreate case unresolved; the direct closing PR is still open and conflicting, not merged.

## What This Changes

Request automatic Telegram isolated polling ingress recovery for stale claimed or `.processing` updates left by a prior gateway so later queued updates do not stay blocked.

## Best Possible Solution

Land a recovery path that handles Telegram PID reuse/current-PID different-process identity claims and reconcile it with the shared SQLite ingress recovery before closing this issue.

## Reproduction Assessment

Yes, at source level. Current main has a test where `${process.pid}:other-process` stays claimed and blocks the later same-lane update, which matches the stale recreated-gateway identity shape reported here.

## Solution Assessment

Yes, the maintainable fix belongs in Telegram or shared ingress claim recovery rather than provider/model routing. It should distinguish gateway instance identity/PID reuse while preserving live-owner protection.
