## Label Justifications

- `P1`: A stale Telegram ingress claim can block a real channel workflow and stop replies from being delivered in current released Docker setups.
- `impact:message-loss`: The issue is about Telegram updates being accepted but replies not being delivered while later updates remain queued.
- `impact:session-state`: The failure is caused by stale persisted spool claim state surviving gateway recreation.

## Summary

Keep open. Current main still has the reported stale-claim gap: isolated ingress recovery runs during drain, but a fresh `.processing` claim with a different process id and a PID that exists in the recreated container is treated as owned by another live process, so later same-lane Telegram updates can remain queued behind it.

## What This Changes

Recover Telegram isolated ingress `.json.processing` spool claims left behind by a recreated gateway so later Telegram updates do not remain blocked.

## Best Possible Solution

Fix the Telegram plugin’s spool claim ownership model so restart-stale `.processing` claims are recovered on startup/drain while genuinely live other-process claims remain protected.

## Reproduction Assessment

Yes, at source level. A fresh `.json.processing` claim with a mismatched `processId` and `processPid === process.pid` is currently treated as live-owned, and the existing test shows a later same-lane `.json` remains pending behind it.

## Solution Assessment

Yes, the maintainable fix belongs in Telegram’s plugin-local spool recovery and ownership check. The repair should distinguish current gateway instance identity from stale PID reuse rather than changing core gateway routing.
