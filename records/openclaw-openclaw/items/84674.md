## Summary

Keep open. Current main and the latest shipped tag still have the source-level stale-claim gap, and the linked open fix PR is the right path to review before closing this issue.

## What This Changes

The issue reports that Telegram isolated polling can leave a stale `.json.processing` spool claim after gateway recreation, blocking later same-lane updates until manual recovery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84690, or an equivalent Telegram plugin fix, so current-PID/different-processId claims are recovered while different-PID live claims remain protected; keep the distinct dispatch-idempotency work in https://github.com/openclaw/openclaw/issues/84886 separate.

## Reproduction Assessment

Yes, at source level. Current main's predicate treats a fresh different-processId claim with `processPid === process.pid` as other-live, and the current polling-session fixture expects recovery `0` with the later update left pending.

## Solution Assessment

Yes. The maintainable fix belongs in the Telegram plugin's spool ownership predicate and drain recovery tests; the linked PR follows that shape but still needs review and runtime proof before the issue can close.
