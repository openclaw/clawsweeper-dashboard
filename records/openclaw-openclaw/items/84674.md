## Summary

Keep open. Current main and the latest release still have the Telegram stale-claim recovery gap, and the issue is already paired with an open closing PR rather than being ready for cleanup.

## What This Changes

The issue reports that Telegram isolated polling can leave a stale ingress `.json.processing` claim after gateway recreation, blocking later queued updates.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/84690 lands or is replaced, and keep the separate durable dispatch idempotency work in https://github.com/openclaw/openclaw/issues/84886.

## Reproduction Assessment

Yes, at source level. Current main treats a fresh `.json.processing` claim with a different `processId` and an existing/current PID as live-owned, and the existing focused test leaves the later same-lane update queued behind it.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Telegram plugin’s spool ownership and recovery predicate, with focused tests for current-PID/different-process-id claims and no core gateway policy change.
