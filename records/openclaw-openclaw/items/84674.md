## What This Changes

Recover Telegram isolated-ingress `.json.processing` spool claims left behind by a recreated gateway so later same-lane updates are not blocked indefinitely.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/84690 with proof that restart-stale claims recover while different live-PID claims remain protected, then close this issue.

Yes, at source level. A `.json.processing` claim with a mismatched `processId`, `processPid === process.pid`, and a fresh `claimedAt` is treated as live-owned on current main, and the existing test leaves a later same-lane `.json` pending behind it.

## Solution Assessment

Yes. The maintainable fix belongs in the Telegram plugin's spool ownership and drain recovery logic, not in core gateway routing; the linked PR is the right narrow implementation path to review.
