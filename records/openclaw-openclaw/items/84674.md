## Label Justifications

- `P1`: A stale Telegram ingress claim can block a real channel workflow and stop replies from being delivered in current released Docker setups.
- `impact:message-loss`: The issue is about Telegram updates being accepted but later replies not being delivered while the spool remains blocked.
- `impact:session-state`: The failure is caused by persisted processing-claim state surviving gateway recreation and blocking later session progress.

## Summary

Keep open. Current main and the latest shipped release still have the source-level stale-claim gap, and https://github.com/openclaw/openclaw/pull/84690 is an open closing PR for this issue rather than a merged fix.

## What This Changes

Recover Telegram isolated ingress `.json.processing` claims from older gateway identities so one stale processing file cannot block later same-lane updates after gateway recreation.

## Best Possible Solution

Land a narrow Telegram plugin fix that treats current-PID/different-process-id claims as restart-stale while preserving protection for genuinely live different-PID owners, with focused regression tests and redacted real Telegram or Docker proof.

## Reproduction Assessment

Yes, at source level. Current main's predicate treats a fresh mismatched-processId claim with `processPid === process.pid` as live-owned, and the existing test expects the earlier update to remain `.json.processing` while a later same-lane update stays pending.

## Solution Assessment

Yes, the best fix belongs in Telegram's spool ownership/recovery predicate rather than core gateway routing. The open PR is a narrow implementation path, but this issue should stay open until that or another fix merges.
