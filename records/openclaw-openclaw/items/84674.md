Keep open. Current main still has the Telegram spool stale-claim gap at source level, and https://github.com/openclaw/openclaw/pull/84690 is an open closing PR for the same bug rather than a merged fix.

## What This Changes

The issue reports that Telegram isolated polling ingress can leave a stale `.json.processing` claim after gateway recreation, blocking later same-lane updates until manual spool cleanup.

## Best Possible Solution

Land a narrow Telegram plugin fix that recovers current-PID/different-processId restart-stale claims while preserving protection for genuinely live different-PID claims.

## Reproduction Assessment

Yes, at source level. Current main's live-owner predicate treats fresh different-processId/current-PID claims as live-owned, and the drain path uses that result before later same-lane updates can be claimed.

## Solution Assessment

Yes, the maintainable fix belongs in the Telegram plugin's spool ownership predicate and focused drain tests. The linked PR appears to target that boundary, with runtime proof still outstanding before merge.
