## Summary

Keep open: current main only handles Markdown tables and a limited iMessage emphasis-to-formatting path; it still does not implement the requested general Markdown-to-plain-text policy for headings, code spans/fences, lists, or shared plain-text delivery.

## What This Changes

Request capability-aware Markdown-to-plain-text conversion for iMessage/plain-text outbound delivery so raw Markdown markers are not sent to users.

## Best Possible Solution

Land or replace the shared outbound normalization work with a capability-aware policy that handles iMessage/SMS/IRC plain text while preserving channel renderers and covering direct sends, monitor replies, and media captions.

## Reproduction Assessment

Yes, for source-level reproduction: text with headings or code spans reaches the current iMessage send path without a general Markdown stripping pass, and the shared outbound path also lacks one. I did not establish a live Messages.app send reproduction.

## Solution Assessment

No complete solution is on main yet. The partial typed-formatting path is useful, but the maintainable fix is a capability-aware outbound text strategy or explicit transport fallback policy rather than treating this as fully implemented.
