## Summary

Keep open: current main now handles some iMessage emphasis markers through typed formatting runs, but it still does not provide the requested general Markdown-to-plain-text handling for backticks, headings, lists, or shared plain-text delivery; the broader implementation candidates remain unmerged.

## What This Changes

Convert or strip Markdown before sending iMessage replies so recipients do not see raw Markdown syntax.

## Best Possible Solution

Land or replace the open shared outbound normalization work with a capability-aware policy that handles iMessage/SMS/IRC plain text, preserves channel renderers, and covers direct sends, monitor replies, and media captions.

## Reproduction Assessment

Yes, for source-level reproduction. Sending text with code spans or headings through the current iMessage path still reaches sendMessageIMessage without a general Markdown stripping pass, while only emphasis-style markers are converted to formatting runs.

## Solution Assessment

No complete solution is on main yet. The partial iMessage formatter is useful, but the maintainable fix is a capability-aware outbound text strategy or explicit transport fallback policy rather than treating the issue as fully implemented.
