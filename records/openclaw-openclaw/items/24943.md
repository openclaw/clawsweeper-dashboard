## Summary

Keep open: current main still only handles tables plus simple iMessage style markers, while general Markdown such as headings, links, list markers, and code spans can still reach iMessage/plain-text delivery. The shared implementation candidate remains open at https://github.com/openclaw/openclaw/pull/56283.

## What This Changes

The issue requests converting or stripping Markdown before sending iMessage replies so raw Markdown syntax is not shown to recipients.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56283 with a shared outbound Markdown policy that strips only for true plain-text delivery after existing sanitization, preserves renderer-owned channels, and covers direct sends, monitor replies, captions, and account-scoped config.

## Reproduction Assessment

Yes, source-level. A reply containing headings, Markdown links, list markers, code spans, or fenced blocks can still pass through the current iMessage send path without a general Markdown-to-plain-text pass.

## Solution Assessment

No complete solution is on main yet. The current iMessage typed-run converter is useful but partial; the maintainable solution is the shared outbound policy represented by the open implementation candidate or an equivalent replacement.
