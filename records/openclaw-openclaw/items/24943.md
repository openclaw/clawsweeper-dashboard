## Summary

Keep open: current main still handles only Markdown tables plus limited iMessage style-run extraction, while headings, links, lists, code spans/backticks, and shared plain-text-channel stripping remain unresolved. The active implementation path is https://github.com/openclaw/openclaw/pull/56283, which is still open and conflicting, so closing this tracker would be premature.

## What This Changes

Requests converting or stripping Markdown in iMessage outbound replies so recipients do not see raw Markdown syntax.

## Best Possible Solution

Land a rebased or replacement shared outbound Markdown policy that strips only for true plain-text iMessage delivery after existing sanitization, preserves Markdown-capable and lightweight-format renderers, and covers account/config behavior with focused regression tests.

## Reproduction Assessment

Yes. Source inspection shows headings, links, lists, and code/backticks can still reach sendMessageIMessage without a general Markdown-to-plain-text pass; only tables and simple emphasis-style runs are handled on current main.

## Solution Assessment

No complete solution is on main yet. The shared-policy direction in https://github.com/openclaw/openclaw/pull/56283 is the right shape, but it needs rebase or replacement work that preserves renderer-owned channels and settles the formatting policy.
