## Summary

Keep open: current main and the latest shipped release still only convert Markdown tables plus simple iMessage style markers, while headings, links, lists, code spans/fences, and shared plain-text Markdown stripping remain unresolved. The active implementation path is https://github.com/openclaw/openclaw/pull/56283, but it is still open and needs rebase/policy work before this tracker should close.

## What This Changes

The issue requests that outbound iMessage replies convert or strip Markdown syntax before delivery to plain-text iMessage recipients.

## Best Possible Solution

Land a rebased or replacement shared outbound Markdown policy that strips only for true plain-text iMessage delivery after existing sanitization, preserves renderer-owned channels, and covers direct sends, monitor replies, media captions, account/config behavior, and empty-after-strip cases.

## Reproduction Assessment

Yes, source-level. A reply containing a heading, Markdown link, list marker, or backtick code can still pass through sendMessageIMessage without a general Markdown-to-plain-text pass; only tables and simple style markers are handled.

## Solution Assessment

No complete solution is on main yet. The shared-policy direction in https://github.com/openclaw/openclaw/pull/56283 is the right shape, but it needs rebase or replacement work that preserves renderer-owned channels and settles the config/default policy.
