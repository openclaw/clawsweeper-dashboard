## Summary

Keep open: current main only handles iMessage tables and emphasis-style formatting runs, while headings, backticks/code, links, list markers, and a shared non-Markdown-channel stripping policy are still unresolved. The active implementation path is the open outbound Markdown stripping PR, not a cleanup close.

## What This Changes

The issue requests converting or stripping Markdown before sending iMessage replies so recipients do not see raw Markdown syntax.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/56283 with a capability-aware outbound plain-text policy that keeps Markdown-capable and lightweight-format channels intact while cleaning iMessage and other true plain-text delivery paths.

## Reproduction Assessment

Yes, for source-level verification: text containing headings, links, list markers, or backticks still reaches sendMessageIMessage without a general Markdown-to-plain-text pass. I did not run a live iMessage send in this read-only review.

## Solution Assessment

No complete solution is on main yet. The narrow iMessage formatting-run support is useful, but the maintainable fix is a shared capability-aware outbound policy with account/config precedence and channel renderer preservation.
