## Summary

Keep open: current main only handles a subset of iMessage Markdown via typed bold/italic/underline/strikethrough runs, while general headings, links, lists, code/backticks, and shared plain-text-channel policy remain unresolved.

## What This Changes

Request that iMessage and other non-Markdown delivery paths convert or strip outbound Markdown before sending so recipients do not see raw Markdown syntax.

## Best Possible Solution

Land a rebased or replacement shared outbound Markdown policy that strips for true plain-text iMessage delivery after existing sanitization while preserving Markdown-capable and lightweight-format channel renderers.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: headings, links, list markers, and backticks still reach direct and monitor iMessage delivery without a general Markdown-to-plain-text pass; I did not run a live iMessage send.

## Solution Assessment

No complete solution is on main yet. The current typed-run support is useful for emphasis, but the maintainable fix is a capability-aware outbound policy with account/config precedence and renderer preservation.
