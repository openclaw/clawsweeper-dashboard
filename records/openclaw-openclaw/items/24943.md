## Summary

Keep open: current main and v2026.5.12 still only handle Markdown tables and simple iMessage style markers, not general Markdown-to-plain-text conversion. The shared implementation candidate remains open at https://github.com/openclaw/openclaw/pull/56283 and is not merged, so this tracker should stay open for maintainer follow-up.

## What This Changes

The issue requests converting or stripping Markdown before iMessage sends so recipients do not see raw Markdown syntax.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56283 with a shared outbound Markdown policy that strips only for true plain-text iMessage delivery after existing sanitization and covers direct sends, monitor replies, captions, config behavior, and empty-after-strip cases.

## Reproduction Assessment

Yes, source-level. A reply containing a heading, Markdown link, list marker, code span, or fenced block can still pass through sendMessageIMessage without a general Markdown-to-plain-text pass; only tables and simple style markers are handled.

## Solution Assessment

No complete solution is on main yet. The shared-policy direction in https://github.com/openclaw/openclaw/pull/56283 is the right shape, but it needs rebase or replacement work that preserves renderer-owned channels and settles the default policy.
