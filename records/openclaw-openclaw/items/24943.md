## Summary

Keep open: current main still only strips/converts a narrow subset of iMessage Markdown, while general headings, links, lists, code/backticks, and shared plain-text-channel policy remain unresolved. The active path is to review, rebase, or replace https://github.com/openclaw/openclaw/pull/56283 rather than close this tracking issue.

## What This Changes

The issue requests stripping or converting Markdown formatting before iMessage sends so raw Markdown markers are not delivered to recipients.

## Best Possible Solution

Land a rebased or replacement shared outbound Markdown policy that strips for true plain-text iMessage delivery after existing sanitization while preserving Markdown-capable and lightweight-format channel renderers.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: text with headings, links, list markers, or backticks still reaches sendMessageIMessage without a general Markdown-to-plain-text pass. I did not run a live iMessage send.

## Solution Assessment

No complete solution is on main yet. The current typed-run support is useful for emphasis, but the maintainable solution is a capability-aware outbound policy with account/config precedence and renderer preservation.
