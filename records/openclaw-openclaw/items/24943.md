## Summary

Keep open: current main and the latest release only partially handle iMessage Markdown by converting tables and a limited bold/italic/underline/strikethrough subset, while the requested general Markdown-to-plain-text policy remains unmerged and is still represented by https://github.com/openclaw/openclaw/pull/56283.

## What This Changes

The issue requests converting or stripping Markdown before sending iMessage/plain-text channel replies so users do not see raw formatting markers.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56283 with a shared outbound Markdown policy that strips only for true plain-text delivery while preserving current iMessage typed formatting, Markdown-capable channels, and account-scoped configuration behavior.

## Reproduction Assessment

Yes, source-level: headings, code spans, links, list markers, and fenced blocks can still reach iMessage/plain-text delivery without a general Markdown stripping pass. I did not run a live iMessage send in this read-only review.

## Solution Assessment

Yes in shape: a shared outbound plain-text Markdown policy is the maintainable direction. The current path should continue through or replace the open implementation candidate rather than queueing a competing narrow fix.
