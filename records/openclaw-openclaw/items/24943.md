## Summary

Keep open: current main only partially handles iMessage Markdown by converting tables and turning simple emphasis markers into iMessage formatting runs. General headings, links, lists, inline/fenced code, and the shared plain-text-channel policy are still unresolved, with https://github.com/openclaw/openclaw/pull/56283 remaining the active but conflicting implementation path.

## What This Changes

The issue requests converting or stripping Markdown before sending replies through iMessage so raw Markdown syntax is not shown in Messages.

## Best Possible Solution

Land a rebased or replacement shared outbound Markdown policy that strips only for true plain-text iMessage delivery after existing sanitization, preserves renderer-owned channels, and adds focused tests for direct sends, monitor replies, media captions, and edge cases.

## Reproduction Assessment

Yes. Source inspection on current main shows headings, links, lists, inline code, and fenced code can still reach sendMessageIMessage because direct and monitor paths only convert tables and simple emphasis-style runs.

## Solution Assessment

No complete solution is on main yet. The shared-policy direction in https://github.com/openclaw/openclaw/pull/56283 is the right shape, but it needs rebase or replacement work that preserves renderer-owned channels and settles the formatting policy.
