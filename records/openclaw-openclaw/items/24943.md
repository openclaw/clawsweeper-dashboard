## Summary

Keep open: current main still lacks a general Markdown-to-plain-text policy for iMessage/plain-text outbound delivery. iMessage now strips simple emphasis into private-API formatting runs, but headings, Markdown links, list markers, code spans, and fenced blocks remain outside that partial path, while the broader implementation is still open at https://github.com/openclaw/openclaw/pull/56283.

## What This Changes

The issue requests converting or stripping Markdown from iMessage outbound replies so plain-text iMessage recipients do not see raw Markdown punctuation.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56283 with a shared outbound Markdown policy that strips only for true plain-text delivery after existing sanitization, preserves renderer-owned channels, and covers direct iMessage sends, monitor replies, media captions, and account-scoped config.

## Reproduction Assessment

Yes, source-level: a reply containing headings, Markdown links, list markers, code spans, or fenced blocks can still pass through sendMessageIMessage without a general Markdown-to-plain-text pass. I did not run a live macOS/iMessage send in this read-only review.

## Solution Assessment

No complete solution is on main yet. The shared outbound policy direction is the maintainable shape, but it still needs maintainer decision around defaults/config and channel capability boundaries.
