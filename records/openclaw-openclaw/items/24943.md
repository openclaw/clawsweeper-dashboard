## Summary

Keep open: current main only strips/converts a subset of iMessage emphasis markers and Markdown tables; it still lacks the requested general Markdown-to-plain-text policy for code spans/fences, headings, lists, links, and shared plain-text delivery. The existing open implementation candidates should be reviewed or replaced rather than closing the user-visible tracker now.

## What This Changes

The issue requests converting or stripping Markdown syntax before sending replies through iMessage/plain-text channels.

## Best Possible Solution

Land or replace the shared outbound text policy with capability-aware plain-text handling that covers direct iMessage sends, monitor replies, media captions, and non-Markdown channels while preserving Markdown-capable renderers.

## Reproduction Assessment

Yes, for source-level reproduction. Sending text containing backticks, headings, links, or list markers through the current iMessage send path reaches `sendMessageIMessage` without a general Markdown stripping pass, while only tables and emphasis-style markers are handled.

## Solution Assessment

No complete solution is on main yet. The current iMessage formatting-run support is useful but narrower than the requested behavior; the maintainable fix is a channel-capability-aware outbound text strategy reviewed against the open implementation candidates.
