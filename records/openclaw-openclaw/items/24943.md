## Summary

Keep open. Current main still lacks a general Markdown-to-plain-text step for legacy iMessage delivery; the code only handles Markdown tables, internal directive cleanup, and HTML-focused plain-text sanitization. Open PR #56283 remains the relevant implementation candidate rather than merged current-main behavior.

## What This Changes

The issue requests that iMessage outbound replies strip or convert Markdown syntax to plain text before sending because iMessage does not render Markdown.

## Best Possible Solution

Land PR #56283 or an equivalent shared outbound normalization change that strips Markdown after existing HTML/plain-text sanitization for non-Markdown-capable channels, while preserving Markdown-capable channels and covering direct iMessage sends, monitor replies, captions, override behavior, and empty-after-strip cases.

## Reproduction Assessment

Yes. A source-level reproduction is straightforward: send text like `**bold**` through sendMessageIMessage with a mocked client and current main will pass that text through table conversion and directive cleanup without removing Markdown markers before `client.request("send", params)`.

## Solution Assessment

Yes, a shared outbound stage is the best direction because fixing only `extensions/imessage/src/send.ts` would miss shared delivery normalization concerns and other plain-text channels; the existing open PR is the right canonical implementation path to review rather than duplicating work.
