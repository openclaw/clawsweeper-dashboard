## Summary

Keep open. Current main still drops the auth profile for cross-provider fallback attempts and still invalidates CLI session reuse on stored/current auth-profile mismatch; the merged Gemini auth-epoch work in #71076 helps account-identity reuse but does not fix this fallback profile-scoping path.

## What This Changes

The issue asks OpenClaw to stop resetting `google-gemini-cli` CLI sessions when that OAuth-backed provider is reached through model fallback rather than as the primary model.

## Best Possible Solution

Fix the CLI fallback auth path so `google-gemini-cli` fallback runs bind a stable Gemini-specific auth identity while preserving cross-provider auth isolation, with regression coverage for fallback-created and stale session bindings.

## Reproduction Assessment

Yes. The issue provides live logs, and current source shows a stored Gemini CLI binding with an auth profile will be invalidated as `reason=auth-profile` when the same session reaches Gemini CLI through a cross-provider fallback whose current auth profile resolves to `undefined`.

## Solution Assessment

Unclear. The report identifies the right boundary, but the best repair should resolve or bind a provider-appropriate Gemini CLI auth identity rather than making all missing-versus-present auth profile bindings reusable.
