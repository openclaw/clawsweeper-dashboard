## Summary

Keep open. Current main still drops the run auth profile for cross-provider fallback, the auto-reply CLI path passes that scoped value into `runCliAgent`, and CLI session reuse can still reject missing-versus-present auth profile bindings unless the auth epoch also matches; the merged Gemini auth-epoch work is adjacent but does not resolve provider-appropriate fallback auth selection.

## What This Changes

The issue asks OpenClaw to preserve Google Gemini CLI conversation context when `google-gemini-cli` is reached as a fallback provider with OAuth auth profiles.

## Best Possible Solution

Keep this open and fix the auto-reply CLI fallback auth path so a `google-gemini-cli` fallback binds a stable, provider-appropriate auth identity while preserving cross-provider auth isolation, then add regression coverage for fallback-created and stale Gemini CLI bindings.

## Reproduction Assessment

Yes, at source level. The issue provides live reset logs, and current main still has a path where cross-provider fallback drops the Gemini CLI auth profile while CLI session reuse can reject missing-versus-present auth profile ids.

## Solution Assessment

Unclear for the broad suggested missing-versus-present compatibility rule. The safer direction is to resolve or bind a provider-appropriate Gemini CLI auth identity for fallback runs without weakening cross-provider auth isolation.
