## Summary

Keep open. Current main still drops cross-provider fallback auth to `undefined`, passes that value into the CLI runner, and still treats stored/current auth profile mismatches as `reason=auth-profile`; the later Gemini auth-epoch fix covers restart/account identity, not this fallback profile-scoping path.

## What This Changes

Fix `google-gemini-cli` fallback sessions so provider fallback does not reset CLI conversation context because of auth-profile mismatch.

## Best Possible Solution

Keep this open and fix the auto-reply CLI fallback auth path so Gemini CLI fallback runs bind a stable, provider-appropriate auth identity while preserving cross-provider auth isolation, with stale-binding regression tests and a changelog entry.

## Reproduction Assessment

Yes. The issue provides live logs, and source inspection on current main shows a stored `google-gemini-cli` auth profile id is invalidated when the same session is invoked as a cross-provider fallback with current auth resolving to `undefined`.

## Solution Assessment

Unclear. The report identifies the right boundary, but the safest repair should resolve a provider-appropriate CLI auth profile before session binding rather than making all missing-versus-present auth profile bindings reusable.
