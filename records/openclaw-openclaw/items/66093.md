## Summary

Keep open. Current main has a partial same-epoch auth-profile drift fix, but the google-gemini-cli fallback path still drops the provider auth profile and the CLI auth epoch still changes when local and auth-profile inputs differ, so this fallback-specific reset path is not clearly fixed.

## What This Changes

Prevent google-gemini-cli fallback runs with OAuth auth profiles from resetting CLI session context on each turn.

## Best Possible Solution

Fix the auto-reply CLI fallback auth path so google-gemini-cli fallback runs bind a stable provider-appropriate auth identity while preserving the cross-provider auth isolation fixed for the older fallback bug, with regression coverage for fallback-created and stale bindings.

## Reproduction Assessment

Yes, at source level. The issue provides live reset logs, and current main still has a path where cross-provider fallback drops the Gemini auth profile, CLI auth epoch composition can differ when the profile is later present, and session reuse invalidates on profile mismatch unless the epoch is identical.

## Solution Assessment

Unclear for the suggested broad compatibility rule. The safer repair should resolve or bind a provider-appropriate Gemini CLI auth identity for fallback runs, rather than treating all missing-versus-present auth profile ids as reusable across providers.
