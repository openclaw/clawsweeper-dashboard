## Summary

Keep open. Current main still scopes cross-provider fallback auth to undefined, passes that scoped value into the CLI runner, and invalidates CLI reuse whenever stored and current auth profile ids differ, matching the reported `google-gemini-cli` fallback reset path.

## What This Changes

Requests fixing a `google-gemini-cli` fallback auth-profile regression that resets CLI sessions on every turn and loses conversation context.

## Best Possible Solution

Fix the auto-reply CLI fallback path so `google-gemini-cli` fallback runs bind a stable Gemini-appropriate auth profile identity, preserve cross-provider auth isolation, and add regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete live setup and logs, and current main still has a source-level path where fallback auth becomes undefined while CLI reuse invalidates missing-versus-present auth ids.

## Solution Assessment

Unclear. The requested direction identifies the right boundary, but the exact repair should be chosen carefully; resolving a provider-appropriate CLI auth profile before session binding is safer than a broad stale-binding compatibility rule.
