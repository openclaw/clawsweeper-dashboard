## Summary

Keep open. Current main still has the reported source-level shape: cross-provider fallback scopes the run auth profile to undefined, the auto-reply CLI path forwards that scoped value into `runCliAgent`, and CLI session reuse still invalidates on missing-versus-present auth profile ids. The issue discussion includes concrete Gemini CLI OAuth fallback logs plus a stale `cliSessionBindings` case, and I found no current-main path that resolves a Gemini-appropriate fallback auth profile before session binding/reuse.

## What This Changes

The issue asks OpenClaw to stop resetting `google-gemini-cli` CLI sessions on every fallback call by keeping auth-profile identity stable for CLI session binding and reuse.

## Best Possible Solution

Fix the auto-reply CLI fallback path so a `google-gemini-cli` fallback run resolves and stores a stable, provider-appropriate Gemini CLI auth profile identity for session binding and reuse, while preserving cross-provider auth isolation. Add regression coverage for both fallback-created bindings and stale `cliSessionBindings` where the Gemini CLI auth profile identity is missing on one side.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path and logs, and current main still has a source-level path that can produce the same missing-versus-present auth-profile mismatch; I did not run a live OAuth fallback session in this read-only review.

## Solution Assessment

Unclear. Resolving a provider-appropriate Gemini CLI auth profile before binding looks safer than accepting any missing auth profile id as a match, because the latter could erode the profile-isolation boundary protected by #1156.
