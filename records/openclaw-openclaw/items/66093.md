## Summary

Keep open. Current main still drops the run auth profile on cross-provider fallback before the CLI path, and the Gemini CLI backend has no provider-owned default profile repair, so the reported fallback-created and stale missing/present auth-profile binding cases are not fully resolved by the adjacent auth-epoch work.

## What This Changes

The issue reports google-gemini-cli fallback runs losing CLI session continuity because cross-provider fallback drops the provider auth profile and stale CLI bindings can invalidate on auth-profile mismatch.

## Best Possible Solution

Fix the auto-reply CLI fallback auth path so google-gemini-cli fallbacks bind a stable provider-appropriate auth identity while preserving https://github.com/openclaw/openclaw/issues/1156 cross-provider isolation, with regression coverage for fallback-created and stale bindings.

## Reproduction Assessment

Yes, at source level. The issue provides live reset logs, and current main still drops cross-provider fallback auth before CLI session binding/reuse; I did not run a live Gemini CLI OAuth fallback.

## Solution Assessment

Unclear for the broad missing-versus-present compatibility rule. The safer fix is provider-appropriate fallback auth identity plus a narrow stale-binding upgrade path that preserves cross-provider auth isolation.
