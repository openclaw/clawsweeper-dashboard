## Summary

Keep open. Current main has adjacent fixes for Gemini CLI auth epochs and same-identity OAuth profile drift, but the fallback path still drops the provider auth profile and the session reuse contract still depends on a matching epoch, so the reported Google Gemini CLI fallback/stale-binding reset path is not fully proven fixed.

## What This Changes

The issue reports that google-gemini-cli fallback runs can lose CLI session continuity because fallback auth scoping drops the provider auth profile while CLI session bindings compare auth identity across turns.

## Best Possible Solution

Resolve or bind a stable, provider-appropriate Gemini CLI auth identity for fallback runs while preserving cross-provider auth isolation, then add regression coverage for fallback-created and stale missing/present `cliSessionBindings`.

## Reproduction Assessment

Yes, at source level. The issue provides live reset logs, and current main still has a path where cross-provider fallback drops the Gemini auth profile while CLI session reuse rejects missing-versus-present auth profile ids unless the versioned auth epoch also matches.

## Solution Assessment

Unclear for the broad suggested compatibility rule. The safer solution is to resolve or bind a provider-appropriate Gemini CLI auth identity on fallback runs, not to make all missing-versus-present auth profile ids reusable across providers.
