## Summary

Keep open. Current main has isolated multi-agent routing, sub-agent runs, Broadcast Groups, and Control UI agent/session selectors, but it still does not implement the requested chat-app product model with agents as contacts, per-agent availability, first-class chat windows, and user-created multi-agent rooms with invitations.

## What This Changes

Add a chat-style multi-agent UI where agents appear as contacts, conversations behave like chat windows, and group chats coordinate invited agents.

## Best Possible Solution

Define a first-class Control UI and Gateway protocol model for multi-agent chat: contact/sidebar data from agent identity/status/capabilities, per-agent conversation switching and history, room membership and invitations, permission boundaries, and transcript attribution, reusing existing sessions, Broadcast Groups, and sub-agent primitives where they fit.

## Reproduction Assessment

Not applicable: this is a feature request rather than a bug. Source and docs inspection of current main gives a high-confidence capability gap in the Control UI, `agents.list`, `sessions.create`, Broadcast Groups, and sub-agent surfaces.

## Solution Assessment

Unclear: the chat paradigm is plausible and supported by production comments, but the maintainable solution needs maintainer-scoped product and protocol design before a narrow implementation can be judged best.
