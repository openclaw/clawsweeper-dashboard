## Summary

Keep open. Current main has isolated multi-agent routing, sub-agent runs, Broadcast Groups, and a grouped session selector, but it still does not implement the requested chat-app product model: agents as contacts, per-agent availability, first-class per-agent chat windows, and user-created multi-agent group rooms with invitations.

## What This Changes

Request a chat-style multi-agent UI where agents appear as contacts, each agent has preserved conversations, and users can create group chats for multiple agents to collaborate.

## Best Possible Solution

Define a first-class Control UI and Gateway protocol model for multi-agent chat: contact/sidebar data from agent identity/status/capabilities, per-agent conversation switching/history, room membership and invitations, and transcript attribution, while reusing existing isolated routing, session listing, Broadcast Groups, and sub-agent primitives where they fit.

## Reproduction Assessment

Not applicable: this is a feature request rather than a bug report. The current-main verification path is source and docs inspection of Control UI chat/session selection, `agents.list`, session schemas, Broadcast Groups, and sub-agent docs, which shows the requested surfaces are still absent.

## Solution Assessment

Unclear: the chat paradigm is plausible and the production comment supports it, but the exact core protocol, budget/status semantics, and room model need maintainer design before a narrow implementation can be judged best.
