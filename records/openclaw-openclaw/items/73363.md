## Summary

Keep open. The branch fixes the important `isReasoning` propagation gap and the mixed-payload picker path, but it still preserves reasoning-only fallback behavior, which can leave model thinking content user-visible in cron announce delivery. The broader open PR #40534 tracks the same bug with fallback, first-text, and error-recovery coverage, so maintainers should choose one canonical branch before merge.

## What This Changes

The PR propagates `isReasoning` from embedded-runner payload construction, filters cron summary/output/deliverable pickers in their primary pass, adds focused tests, and updates the changelog.

## Best Possible Solution

Land one canonical cron announce fix that propagates `isReasoning` and treats reasoning payloads as non-user-visible across summary selection, output selection, deliverable selection, raw fallback, first-text fallback, and post-error success detection, while preserving fallback behavior for real error payloads.

## Reproduction Assessment

Yes. A high-confidence static reproduction is enough: after this PR, the added tests expect a payload list containing only `{ text: "thinking", isReasoning: true }` to be returned by the pickers, and cron direct delivery sends selected nonempty payloads.

## Solution Assessment

No as-is. The marker propagation and primary-pass filters are the right direction, but reasoning-only fallback should not surface thinking content, and the `firstText` plus post-error recovery paths should be aligned with the same policy.

## Review Findings
