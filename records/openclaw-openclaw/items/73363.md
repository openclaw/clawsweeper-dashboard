## Summary

Keep open. This PR fixes the marker propagation and mixed-payload primary filtering path, including the Feishu deliverable picker after review, but it still intentionally falls back to reasoning-only payloads and leaves first-text/error-recovery success detection treating reasoning as visible text.

## What This Changes

The PR propagates `isReasoning` from embedded-runner payload construction, filters reasoning in cron summary/output/deliverable primary selection, adds focused tests, and updates the changelog.

## Best Possible Solution

Land one canonical cron announce fix that propagates `isReasoning` and treats reasoning as non-user-visible across summary, output, delivery, fallback, first-text, and post-error success detection while preserving fallback behavior for real error payloads.

## Reproduction Assessment

Yes. A static reproduction is high confidence: on the PR head, a payload list containing only `{ text: "thinking", isReasoning: true }` is expected by new tests to be returned for delivery, and direct cron delivery sends nonempty selected payloads.

## Solution Assessment

No as-is. Marker propagation and primary-pass filtering are the right direction, but the reasoning-only fallback and error-recovery paths need to be aligned before this fully solves the reported leak.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Stop delivering reasoning-only cron payloads:** `src/cron/isolated-agent/helpers.test.ts:169-173`
  - body: The new tests deliberately keep reasoning-only payloads deliverable when no visible answer exists. Since `resolveCronPayloadOutcome()` can pass those selected/synthesized payloads to direct cron delivery, and Feishu has no downstream `isReasoning` guard on that path, a reasoning-only run can still expose thinking text. Please make reasoning payloads non-deliverable in fallback paths too, and align the first-text/error-recovery checks with that policy.
  - confidence: 0.86
