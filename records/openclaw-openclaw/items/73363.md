## Summary

Keep open. The `isReasoning` propagation is the right direction, but the PR still codifies reasoning-only fallback for cron summary/output/deliverable selection and does not align post-error success detection, so it is incomplete for the privacy-sensitive leak reported in #73186.

## What This Changes

The PR propagates `isReasoning` from embedded-runner payload construction, filters reasoning in cron picker primary passes, adds focused tests, and updates the changelog.

## Best Possible Solution

Land one canonical cron announce fix that preserves `isReasoning`, excludes reasoning from summary/output/deliverable fallbacks and post-error success checks, and preserves real error fallback behavior.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: on the PR head, new tests expect reasoning-only payloads to be returned, and current `resolveCronPayloadOutcome()` promotes picker results into cron outcome fields and success detection.

## Solution Assessment

No as-is. Marker propagation and primary-pass filtering are correct, but reasoning should be treated as non-user-visible in fallback and error-recovery paths rather than as a last-resort answer.
