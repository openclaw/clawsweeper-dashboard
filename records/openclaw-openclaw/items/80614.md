## Summary

Keep open: the PR addresses a valid status performance bug, but the current patch regresses shared session-store aggregation by reusing per-agent rows for the global recent-session list.

## What This Changes

The PR threads manifest-normalization opt-outs through status model resolution, caches CLI-provider checks during status summaries, reuses session rows, and adds focused status tests.

## Best Possible Solution

Land the status cold-path improvements after changing global session aggregation to build rows once per unique store without agentIdOverride and adding shared-store regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-backed: the linked report and PR body provide timing commands/profiles, and current main still shows status summary paths that can enter manifest normalization and per-row CLI-provider discovery; I did not run the live timing command in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the manifest-normalization and CLI-provider caching direction is narrow, but the global session reuse must cache/build rows per unique store without agentIdOverride so shared stores remain correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep shared session stores deduplicated:** `src/commands/status.summary.ts:262`
