## Summary

Keep open: the status performance bug is valid and the PR contains useful cold-path fixes, but the current diff still introduces a shared session-store regression in global status aggregation.

## What This Changes

The PR disables manifest normalization for status-only model parsing, caches CLI-provider checks during status rendering, reuses session rows for status summaries, and adds stale task health fields across CLI, gateway, SDK, and docs.

## Best Possible Solution

Land the status cold-path optimization after fixing shared-store aggregation and adding multi-agent shared-store regression coverage; decide separately whether the task-health API/docs surface belongs in this PR.

## Reproduction Assessment

Yes, source-backed: the PR body and linked issue provide before/after timing for 2026.5.7, and current main still shows status model parsing only disables plugin normalization while manifest normalization can load metadata. I did not run live timing in this read-only review.

## Solution Assessment

No, not as submitted: the normalization and CLI-provider-cache pieces are narrow, but the session aggregation change reuses per-agent-overridden rows for the global list. The safer repair is to reuse loaded stores while building global rows once per unique store without `agentIdOverride`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Deduplicate shared session stores:** `src/commands/status.summary.ts:264`
  - body: When `session.store` lacks `{agentId}`, every agent resolves to the same store file. This new push appends rows already built with `agentIdOverride` for each agent, so global `sessions.count` and `sessions.recent` repeat the same stored sessions and label them as each configured agent. Build the global rows once per unique store without the override, or dedupe by store/key while still reusing the loaded store.
