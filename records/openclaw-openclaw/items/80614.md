## Summary

Keep open: the status cold-path fix direction is source-backed and supported by live timing proof, but the PR introduces a shared-session-store aggregation regression that should block merge.

## What This Changes

The PR threads a status-only manifest-normalization opt-out through model-ref resolution, caches status CLI-provider checks, changes session-row aggregation, and adds stale task health fields to CLI, gateway, SDK, and docs surfaces.

## Best Possible Solution

Land the status optimization after building global recent rows once per unique store without `agentIdOverride`; split the task-health surface unless maintainers want it here.

## Reproduction Assessment

Yes, source-backed: the PR and related comments give concrete live timing data, and current main still has the status parsing path that can manifest-normalize without an explicit opt-out. I did not rerun the timing locally in this read-only review.

## Solution Assessment

No, not as submitted: the manifest-normalization and CLI-provider-cache pieces are narrow, but the global session aggregation change breaks shared-store semantics. The safer fix is to reuse loaded stores while building global rows once per unique store without per-agent overrides.
