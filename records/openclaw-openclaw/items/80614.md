## Summary

Keep open: the status performance fix is useful, but the latest head still introduces a shared session-store aggregation regression and includes unrelated task-health API/CLI work.

## What This Changes

The PR threads status model parsing around manifest normalization, caches status runtime provider checks, changes global session aggregation, and adds task health output to CLI/gateway task listings.

## Best Possible Solution

Land the status cold-path optimization after rebuilding global session rows once per unique store without `agentIdOverride`, with shared-store regression coverage and the task-health work split or explicitly accepted separately.

## Reproduction Assessment

Yes, source-backed: the reported timing commands against 2026.5.7 plus current-main status model parsing show how non-deep status can still enter manifest normalization. I did not rerun the live timing locally in this read-only review.

## Solution Assessment

No, not as submitted: disabling manifest normalization and caching CLI-provider checks are narrow, but global session aggregation must preserve unique shared-store rows. The safer fix is to reuse loaded stores without reusing per-agent-overridden rows for the global list.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep shared session stores deduplicated:** `src/commands/status.summary.ts:264`

Build the global rows once per unique store without the override, or dedupe by store/key.
