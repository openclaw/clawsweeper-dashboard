## Summary

Keep open: the status performance fix is still useful and tied to the open linked bug, but the current PR head introduces a shared-session-store aggregation regression and includes unrelated task-list behavior, so it should not be merged as-is.

## What This Changes

The branch threads `allowManifestNormalization` through persisted model-ref resolution, caches CLI-provider checks during status rendering, reuses session rows for status summaries, and adds stale active task health to `openclaw tasks list`.

## Best Possible Solution

Land the status cold-path optimization after restoring unique-store global session aggregation, adding shared-store regression coverage, and splitting or dropping the task-list change unless it is reviewed separately.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source/report level: the linked issue and PR body provide timing/profiling proof, and current main still has the status summary model/runtime paths that can re-enter manifest normalization and repeated CLI-provider discovery; I did not run live timing locally.

## Solution Assessment

Is this the best way to solve the issue? Not yet: disabling manifest normalization and caching CLI-provider checks is a narrow direction, but global recent sessions must still be built per unique store without `agentIdOverride`, and the task-list change should be split or explicitly retained.

## Review Findings
