## Summary

Keep open: the status cold-path fix is useful and backed by real timing output, but the latest PR still introduces a shared-session-store regression in the global recent-session aggregation and is draft. The current head also carries unrelated task-list behavior that should be split or deliberately reviewed outside the status performance fix.

## What This Changes

The PR threads manifest-normalization opt-outs through status model resolution, caches CLI-provider checks, reuses status session rows, and also adds stale task health display to `openclaw tasks list`.

## Best Possible Solution

Land the status cold-path optimization after restoring unique-store global session aggregation, adding shared-store regression coverage, and splitting or dropping the unrelated task-list change.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-backed yes for the root cause: the linked report and PR body provide timing/profiling proof, and current main shows status summary model/runtime paths that can re-enter manifest normalization and repeated CLI-provider discovery; I did not run the live timing repro locally.

## Solution Assessment

Is this the best way to solve the issue? Not yet: disabling manifest normalization and caching CLI-provider checks is a narrow direction, but the global recent-session reuse must be rebuilt per unique store without `agentIdOverride`, and the unrelated tasks change should not ride in this status PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep shared session stores deduplicated:** `src/commands/status.summary.ts:264`
  - body: When `session.store` omits `{agentId}`, every agent resolves to the same `sessions.json`. Build/cache the global rows once per unique store without the override, or dedupe by store/key.
