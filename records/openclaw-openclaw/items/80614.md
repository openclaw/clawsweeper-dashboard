## Summary

Keep open: the PR targets a valid status cold-path performance bug and has sufficient real-output proof, but the current head still introduces a shared-session-store regression and mixes in unrelated task-health API/docs work.

## What This Changes

The PR disables manifest normalization and caches CLI-provider checks for status session rendering, reuses session row data, and also adds stale task health to CLI, gateway, SDK, and docs surfaces.

## Best Possible Solution

Land the status cold-path optimization after preserving unique-store global session aggregation, with regression coverage; split or separately approve the task-health surface.

## Reproduction Assessment

Yes, source-backed: the PR body gives before/after timing output from an affected 2026.5.7 install, a follow-up comment corroborates the same cold path, and current main still has the manifest-normalization and CLI-provider discovery paths. I did not rerun live timing in this read-only review.

## Solution Assessment

No, not as submitted: the status normalization/cache pieces are narrow, but reusing per-agent-overridden rows for the global list breaks shared-store semantics. The safer fix is to reuse loaded stores while building global rows once per unique store without agentIdOverride, and split task-health changes unless maintainers want them here.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Deduplicate shared session stores:** `src/commands/status.summary.ts:264`
  - body: When session.store lacks {agentId}, every configured agent resolves to the same store file. Appending rows already built with agentIdOverride makes global sessions.count and sessions.recent include each stored session once per agent, stamped as that agent, so shared-store configs get inflated global status. Build the global rows once per unique store without the override, or dedupe by store/key.
  - confidence: 0.92
