## Summary

Keep open: the PR addresses a valid status cold-path performance bug, but the current patch introduces a shared-session-store aggregation regression that should block merge until repaired.

## What This Changes

The PR threads manifest-normalization opt-out through status model parsing, caches CLI-provider detection during status rendering, reuses session rows, and adds focused status summary tests.

## Best Possible Solution

Land the cold-path status improvements after replacing the global recent-session reuse with a unique-store row cache that preserves shared-store counts and adding regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-backed yes: the linked report and PR body provide timing commands/profiles, and current source shows the status summary paths that can re-enter manifest normalization; I did not run a live timing reproduction in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the manifest-normalization and CLI-provider caching direction is narrow, but global session reuse should cache rows per unique store without `agentIdOverride` so shared stores remain correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep shared session stores deduplicated:** `src/commands/status.summary.ts:263`
  - body: When `session.store` omits `{agentId}`, multiple agents resolve to the same `sessions.json`. This line appends the per-agent row list for every agent into the global list; because those rows are built with `agentIdOverride`, a shared store containing `agent:main:*` and `agent:ops:*` will make `sessions.count` and `sessions.recent` show each stored session once per agent. Cache/build global rows once per unique store without the override, or dedupe by store/key.
  - confidence: 0.9
