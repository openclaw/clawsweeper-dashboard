## Summary

Keep open: this PR addresses a real `openclaw status` cold-path performance bug, but the proposed session-row reuse introduces a shared-session-store aggregation regression that should block merge until repaired.

## What This Changes

The branch threads manifest-normalization opt-outs through persisted model parsing, caches status runtime CLI-provider detection, reuses session rows for global recent-session aggregation, and adds focused status summary tests.

## Best Possible Solution

Land the manifest and CLI-provider cold-path improvements after replacing the global recent-session reuse with a unique-store, no-override row cache and adding shared-store regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-backed yes: the linked issue and PR body give timing commands, profiles, and copied live output, and current main still shows the status summary paths that can re-enter manifest normalization and per-row CLI-provider checks.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the manifest opt-out and cached CLI-provider direction are narrow, but global recent sessions must be built from unique stores without `agentIdOverride` to preserve shared-store counts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep shared session stores deduplicated:** `src/commands/status.summary.ts:264`
  - body: When `session.store` omits `{agentId}`, multiple agents resolve to the same `sessions.json`. This line appends per-agent rows for every agent into the global list; because those rows are built with `agentIdOverride`, a shared store containing entries for multiple agents will duplicate and relabel stored sessions in `sessions.count` and `sessions.recent`. Build/cache global rows once per unique store without the override, or dedupe by store/key.
  - confidence: 0.9
