## Summary

Keep open: current main has source-backed evidence that the no-override UI still renders override/default labels rather than a true effective inherited state, and lightweight session rows can omit the resolved per-session default that the runtime uses.

## What This Changes

The issue asks for Control UI thinking selectors to show the effective inherited thinking level separately from explicit per-session overrides instead of showing no override as off/default/inherit.

## Best Possible Solution

Have sessions.list or the UI expose the effective inherited thinking value for no-override sessions and render that state distinctly from explicit overrides while preserving the existing clear-override value and runtime behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a no-override session uses the blank select value while current UI labels it as `Default (...)` or `inherit`, and lightweight sessions.list rows can omit the runtime-resolved thinking default. I did not run the browser UI, so this is source_reproducible rather than executed.

## Solution Assessment

Yes, the narrow fix is to keep this cosmetic/UI-facing: resolve or carry the effective default for rows and update the select label helper, without changing sessions.patch, model routing, reply execution, or runtime thinking resolution.
