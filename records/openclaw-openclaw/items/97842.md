## Summary

Keep open. The PR targets a valid source-proven memory status bug, but the patch reimplements session file enumeration with a canonical-directory scan instead of the corpus listing used by indexing, so it can miss or overreport supported session stores; after-fix real behavior proof is also mock-only.

## What This Changes

The branch adds a synchronous memory-core status-mode check that marks sessions dirty when usage-counted transcript files lack `memory_index_sources` rows, plus a startup-catchup regression test.

## Best Possible Solution

Keep the status-mode dirty check, but make it use the same session corpus/listing as startup catch-up and add custom-store coverage plus real after-fix status proof.

## Reproduction Assessment

Yes, at source level. Current main uses a transient status manager that skips session startup catch-up, and `status()` reports only stored manager dirty flags rather than comparing usage-counted session files with indexed `sessions` rows.

## Solution Assessment

No. The direction is right, but the patch should reuse the existing session corpus listing contract instead of scanning the canonical directory directly, and it needs real after-fix CLI/status proof.
