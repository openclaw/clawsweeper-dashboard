## Summary

Keep open: current main has edit-tool plumbing and an optional diffs plugin, but the requested always-visible unified diff after a successful edit is not implemented as a core/operator display behavior and needs a UX/product decision before automation changes it.

## What This Changes

Request that the `edit` tool show a unified diff after successful file modifications.

## Best Possible Solution

Define a bounded edit-result presentation contract that surfaces successful edit diffs where operators see tool output, while preserving the existing optional `diffs` plugin for richer viewer/file artifacts.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection shows current OpenClaw display paths do not automatically promote edit diffs into visible tool output.

## Solution Assessment

Unclear until maintainers choose the desired UX. The narrow implementation should likely reuse upstream `details.diff` with size limits rather than inventing a separate edit-diff generator.
