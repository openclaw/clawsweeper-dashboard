## Summary

Keep open. Current main still sends the bundled Claude CLI backend’s bare `claude` command through the supervised child-spawn path, while that Windows shim resolver only special-cases npm-family commands.

## What This Changes

Fix the Windows Claude CLI runtime so npm-installed Claude Code can launch from `claude.cmd`/`claude.exe` instead of failing with `spawn claude ENOENT`.

## Best Possible Solution

Use a shared Windows spawn resolver for CLI backend launches so `claude` resolves to the actual Claude Code executable or Node entrypoint on Windows while preserving no-shell safety and Unix behavior.

## Reproduction Assessment

Yes, source-reproducible: current main passes bare `claude` to child spawn and the Windows child adapter does not resolve that npm shim. A real Windows gateway run would be useful final proof, but the reported path is coherent and code-backed.

## Solution Assessment

Partly: fixing Windows CLI command resolution is the right direction, but a blanket `shell: true` fix is not the safest approach for prompt-bearing argv. Reusing or extending OpenClaw’s existing Windows spawn program resolver is the narrower maintainable solution.

On Windows, the bundled `claude-cli` backend defaults to `command: "claude"`, but npm global installs expose `claude.cmd` on PATH and the current supervised child-spawn path does not resolve that shim.
