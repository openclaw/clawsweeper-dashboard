## Summary

This should stay open: current `main` already has an internal runtime `preserveKeys` hook, but there is no public `session.maintenance.preserveKeys` config contract. Adding it is a new config surface with matching-policy choices, so it needs maintainer design review rather than cleanup closure.

## What This Changes

Add a `session.maintenance.preserveKeys` configuration field so selected session keys, such as the main WebUI session, are exempt from stale-entry and count pruning.

## Best Possible Solution

Define the public config contract first, then wire resolved preserve keys through all session-maintenance paths and document the exact matching semantics with focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a request for a new public config field. Source inspection does confirm the current gap because runtime helpers accept `preserveKeys`, while the schema/types/docs expose no `session.maintenance.preserveKeys` setting.

## Solution Assessment

Unclear pending maintainer design review. The proposed field is plausible, but the maintainable solution needs explicit decisions on exact-key versus pattern matching and whether main sessions should be protected by default instead of requiring user config.
