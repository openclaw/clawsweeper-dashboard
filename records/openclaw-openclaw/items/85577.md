## What This Changes

Telegram should avoid treating transient pairing-store read failures as an empty allowlist that sends bogus pairing prompts to already authorized direct-message senders.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85555, or an equivalent Telegram authorization fix that preserves configured `allowFrom`, stops treating transient store-read errors as empty state, and includes focused regression coverage plus Telegram proof.

Yes, at source level: a rejected Telegram pairing-store read is caught as `[]`, then pairing-mode direct-message access can reach the pairing-challenge path. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the linked fix direction is the right class of solution: avoid presenting transient store-read failures as empty authorization state while preserving configured `allowFrom` behavior. The exact PR still needs maintainer review and real Telegram proof before merge.
