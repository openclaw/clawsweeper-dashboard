## What This Changes

Reports that transient Telegram pairing-store read errors can be treated as an empty paired-user allowlist and produce misleading DM pairing prompts for already paired senders.

## Best Possible Solution

Review and land or explicitly reject the linked Telegram authorization fix after required live Telegram proof or owner acceptance, then close this issue when the fix merges.

## Reproduction Assessment

Yes at source level: current main catches Telegram pairing-store read failures as [] before DM authorization, and the pairing path can then send a challenge. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes on direction: the maintainable fix is to stop treating store-read failures as empty allowlists and avoid pairing-store reads when configured allowFrom already authorizes the sender. Current main has not landed that fix yet.
