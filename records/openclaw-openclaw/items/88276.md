## Summary

Keep this PR open, but it is not merge-ready: the underlying Feishu streaming-card bug is supported by current source and docs, while the patch also routes tool deliveries into a streaming-card branch that does not preserve their text, and the submitted proof is mock-only.

## What This Changes

The PR changes the Feishu reply dispatcher so streaming-enabled auto-mode plain replies start and close a Feishu streaming card, with dispatcher tests for final text, assistant-start, partial-snapshot, and non-streaming paths.

## Best Possible Solution

Constrain forced streaming-card delivery to final assistant replies, preserve tool summaries and errors as normal separate messages, add a regression test for that boundary, then attach redacted live Feishu/CardKit proof.

## Reproduction Assessment

Yes from source, not from a live Feishu run: current main sends auto-mode plain final text through the normal message path even though the Feishu docs describe `streaming: true` as interactive-card output. The PR’s introduced tool-summary drop is also source-reproducible from the dispatch-kind union and active streaming branch.

## Solution Assessment

No as written. The final-reply direction matches the documented behavior, but the non-block predicate also catches tool deliveries; the safer repair is to target `info?.kind === "final"` or explicitly preserve tool text.
