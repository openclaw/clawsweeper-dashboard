## Summary

Keep open: this is the right Feishu dispatcher surface and the PR has useful focused tests plus real Feishu/CardKit proof for the final-reply card path, but current head still creates a CardKit session on assistant-message start before knowing whether any deliverable reply will survive normalization, which can leak empty cards for silent/no-output turns.

## What This Changes

The PR changes Feishu reply dispatching so streaming-enabled auto-mode final text uses CardKit streaming cards, starts card sessions from assistant/partial events, and extends dispatcher coverage for final, partial, disabled-streaming, tool, media, and thread cases.

## Best Possible Solution

Keep the Feishu final-text streaming-card fix, but make CardKit creation depend on deliverable non-silent text and add a regression for assistant-start followed by NO_REPLY/no deliverable final.

## Reproduction Assessment

Yes. Source inspection shows current main documents Feishu streaming cards as the default path but plain auto-mode final text still falls through to normal messages unless it is markdown/table/card/block content; I did not run a live Feishu repro in this read-only review.

## Solution Assessment

No. The owner boundary is right, but the current implementation starts CardKit too early; the safer fix is to defer card creation until non-silent deliverable text exists or otherwise guarantee empty sessions are never sent.
