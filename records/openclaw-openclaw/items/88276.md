## Summary

This PR is useful and now has credible live Feishu/CardKit proof, but it still has a blocking message-delivery bug: starting an auto-mode streaming session before final text can suppress later non-final tool summary text instead of sending it on the normal Feishu message path.

## What This Changes

The PR changes the Feishu reply dispatcher so streaming-enabled auto-mode final plain text uses CardKit streaming cards, and adds dispatcher tests for final, partial/start, disabled-streaming, and tool-summary paths.

## Best Possible Solution

Keep the final plain-text streaming-card behavior, but before merge preserve normal Feishu delivery for non-final tool/error summaries even when an auto-mode streaming session has already started.

## Reproduction Assessment

Yes for the blocking PR bug from source: on the PR branch, call `onAssistantMessageStart`, then deliver `{ text: "tool summary" }` with `{ kind: "tool" }`; the newly active streaming session reaches the existing active-session branch and returns before `sendMessageFeishu`. I did not run tests because this review is read-only.

## Solution Assessment

No. Streaming final auto-mode plain text through CardKit matches the documented Feishu behavior, but this implementation must also preserve normal delivery for non-final tool/error summaries when a streaming session is already active.
