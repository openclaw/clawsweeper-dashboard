## What This Changes

Adds an `interrupt` steering mode to embedded-agent queue message options, cancels the active handle before queueing, and adds unit tests for cancel/no-cancel handles.

## Best Possible Solution

Model interrupt as a separate queue action or intent, keep agent-core queue drain modes on the existing `QueueMode` contract, and prove the replacement-run behavior on real long-running embedded sessions before merge.

## Solution Assessment

No. The current patch is a plausible sketch, but the maintainable solution is to keep interrupt intent separate from steering queue drain mode and to use the existing abort/replacement-run lifecycle rather than queueing through a cancelled handle.

  - body: Widening `EmbeddedAgentQueueMessageOptions.steeringMode` means `run/attempt.ts` can now assign `"interrupt"` to `activeSession.agent.steeringMode`, but agent-core still types that setter as `QueueMode = "all" | "one-at-a-time"`.   - confidence: 0.94
- **[P2] Do not queue through the run you just cancelled:** `src/agents/embedded-agent-runner/runs.ts:433`
  - body: The new path calls `handle.cancel("user_abort")` and then proceeds to `handle.queueMessage(...)` on that same active-run handle. For the Pi handle, transcript-wait queueing rejects when the run ends before the queued user message commits, so this can return `runtime_rejected` instead of the documented interrupt behavior: abort the active run and start the newest message as the next run.
  - confidence: 0.86
