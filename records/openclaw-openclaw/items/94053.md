- `impact:message-loss`: The affected path misroutes text that should remain private into a user-facing Telegram delivery path.

## Summary

Keep open: current main and the latest release still lack a heartbeat-runner guard for the no-`heartbeat_respond` path, while several linked PRs are open fix candidates but none has merged.

## What This Changes

The issue reports that heartbeat runs can deliver private final text to Telegram despite `messages.visibleReplies: "message_tool"`.

## Best Possible Solution

Review and land one narrow fix PR that suppresses no-tool-response heartbeat text only for explicit message-tool visibility while preserving marked failure notices and relayable exec-completion payloads, then let the issue close from the merged PR.

## Reproduction Assessment

Yes. The current source path is high-confidence: configure `messages.visibleReplies: "message_tool"`, return heartbeat final text without `heartbeat_respond`, and the heartbeat runner normalizes that payload for delivery because no no-tool-response privacy guard exists.

## Solution Assessment

Yes. The narrow maintainable fix is in `src/infra/heartbeat-runner.ts`: treat explicit message-tool visibility plus missing heartbeat tool response as a silent heartbeat, while preserving payloads that are explicitly marked for delivery and exec-completion events.
