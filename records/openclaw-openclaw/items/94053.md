## Summary

Keep open: current main and v2026.6.10 still let a non-empty heartbeat final reply with no `heartbeat_respond` call fall through toward normal durable delivery after forcing `message_tool_only`, and open linked PRs with closing syntax already own the repair path.

## What This Changes

The issue asks OpenClaw to keep heartbeat final text private in `messages.visibleReplies: "message_tool"` mode when the model does not call `heartbeat_respond`.

## Best Possible Solution

Repair and land the scoped canonical PR, then close this issue and retire the duplicate competing heartbeat-leak branches.

## Reproduction Assessment

Yes at source level. A non-empty heartbeat final reply with no heartbeat tool response skips the current silent branches and reaches durable delivery while the run is in `message_tool_only` mode.

## Solution Assessment

Yes, the heartbeat runner is the right fix boundary because it has the tool-response state, visibility mode, delivery exceptions, and heartbeat bookkeeping. The current best path is the scoped open PR after its preview leak blocker is fixed.
