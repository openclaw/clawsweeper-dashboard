## Label Justifications

- `P2`: This is a confirmed current-release privacy/message-delivery bug, but it is limited to users who opted heartbeat visible replies into `message_tool` mode.
- `impact:message-loss`: Private final replies are routed to the Telegram/source channel despite tool-only delivery mode, which is a concrete message-delivery mismatch.
- `impact:security`: The bug bypasses an explicit privacy/silent-delivery control and can expose private assistant final text to channel users.

## Summary

Keep open: current main and v2026.6.8 still allow a no-tool heartbeat text payload to fall through to durable channel delivery, and several open closing PRs already target the bug. The strongest path is to review and land one canonical fix PR, then close this issue after merge.

## What This Changes

The issue requests that heartbeat final text stay private in `messages.visibleReplies: "message_tool"` mode unless `heartbeat_respond` is called with `notify: true`.

## Best Possible Solution

Land a scoped heartbeat-runner guard with regression coverage from the canonical open PR, then close this issue and redirect the duplicate fix PRs.

## Reproduction Assessment

Yes. Current source shows a no-tool, non-empty heartbeat reply in `message_tool_only` mode is normalized and can be sent through durable delivery, and the reporter supplied logs from v2026.6.8 showing the same leaked Telegram send.

## Solution Assessment

Yes. The maintainable fix belongs at the heartbeat runner's final delivery decision, where it can see the forced response-tool mode, reply payload, marked-notice exceptions, exec-completion state, and target delivery context.
