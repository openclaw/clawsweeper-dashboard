## What This Changes

The issue asks OpenClaw to keep heartbeat final text private in `messages.visibleReplies: "message_tool"` mode when the model does not call `heartbeat_respond`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/94088 or an equivalent scoped heartbeat-runner fix, then close this issue and redirect the weaker duplicate PRs.

## Reproduction Assessment

Yes. Source inspection shows that with `messages.visibleReplies: "message_tool"`, a non-empty heartbeat final text without `heartbeat_respond` still enters normal heartbeat reply delivery on current `main`; I did not run live Telegram proof in this read-only review.

## Solution Assessment

Yes. The best fix is a narrow guard in `src/infra/heartbeat-runner.ts` before raw reply normalization, preserving marked failure notices and relayable exec completions; https://github.com/openclaw/openclaw/pull/94088 already implements that path.
