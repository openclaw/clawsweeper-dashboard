- `impact:message-loss`: The affected behavior is incorrect channel message delivery from the heartbeat runner to Telegram.
- `impact:security`: The report is about private heartbeat final text being exposed to a user-facing Telegram channel despite the configured privacy mode.

## Summary

Keep open: current main and v2026.6.8 still have the heartbeat no-tool-response fallthrough into normal text delivery, and multiple open linked PRs are already candidate fixes. This should close only after one focused fix lands.

## What This Changes

The issue reports that heartbeat final text can be delivered to Telegram despite `messages.visibleReplies: "message_tool"` when the heartbeat response tool is not called.

## Best Possible Solution

Land one focused linked PR that suppresses no-tool heartbeat final text in `message_tool_only` mode, preserves explicitly deliverable failure notices, and adds regression coverage in the heartbeat tool-response tests.

## Reproduction Assessment

Yes. Source inspection shows that with `messages.visibleReplies: "message_tool"`, a non-empty heartbeat final text without `heartbeat_respond` reaches heartbeat reply normalization instead of a quiet no-tool branch; I did not run live Telegram in this read-only review.

## Solution Assessment

Yes on direction. The maintainable fix belongs in `runHeartbeatOnce` before normal reply delivery, with a narrow exception for payloads explicitly marked deliverable despite source-reply suppression.
