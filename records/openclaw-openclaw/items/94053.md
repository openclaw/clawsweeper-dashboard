## Summary

Keep open: current main and v2026.6.9 still have the no-tool heartbeat final-text fallthrough, and open linked PRs with closing syntax already own the repair path.

## What This Changes

The issue asks OpenClaw to keep heartbeat final text private when `messages.visibleReplies` is `message_tool` and the model does not call `heartbeat_respond`.

## Best Possible Solution

Land a scoped heartbeat-runner privacy guard that suppresses explicit `message_tool` no-tool final text while preserving marked notices and relayable exec completions, with regression coverage; then close the issue.

## Reproduction Assessment

Yes, at source level. Configure `messages.visibleReplies: "message_tool"`, return a non-empty heartbeat final payload without `heartbeat_respond`, and current main skips the empty-only silent branch before normalizing and sending the payload.

## Solution Assessment

Yes. The heartbeat runner is the right fix boundary because it has the tool-response state and delivery context; blanket no-tool suppression is weaker because response-tool mode can also be entered for runtime defaults and marked notices still need delivery.
