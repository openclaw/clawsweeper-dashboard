## Summary

Keep open for explicit maintainer security and product follow-up. Current code still contains the exact transcript-takeover failure reported after Stop, and the existing abort-partial persistence behavior predates the incident; meanwhile, documented exec approvals reduce accidental execution risk but are not a per-agent filesystem boundary. No canonical duplicate or merged fix was found.

## What This Changes

Prevent Dashboard Stop from leaving session transcripts corrupted during active tool execution and add guardrails against one agent destructively modifying another agent’s persistent state.

## Best Possible Solution

Split this into two owned workstreams: prove and fix Stop-time transcript integrity with restart/parseability coverage, and separately design an agent-scoped host-exec authority policy that protects other agents’ state while preserving an explicit trusted-power mode.

## Reproduction Assessment

Partly yes: current source exposes the Dashboard-to-`chat.abort` path and the exact session-takeover exception, while the host-exec trust boundary is documented. No live current-main run in this read-only review reproduced the claimed truncation or cross-agent deletion sequence.

## Solution Assessment

No. A UI grace delay, command substring blacklist, recovery-only repair, or preinstalled trash utility would not establish the underlying cancellation and filesystem-authority invariants; the two boundaries need separate focused designs and proof.
