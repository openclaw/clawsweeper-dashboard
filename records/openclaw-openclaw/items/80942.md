## Summary

Keep open: current main and the latest release still use a static Control UI browser title, the direct dynamic-title PR was closed unmerged, and the open static-title PR is adjacent rather than a replacement. The related ClawHub skill linked from the closed PR is useful as a workaround, but it does not settle the core Control UI title policy requested here.

## What This Changes

The issue requests a dynamic Control UI browser title that reflects the active agent for multi-agent workflows when no static gateway title is configured.

## Best Possible Solution

Define one Control UI title resolver where a configured static title has explicit precedence if accepted, otherwise the active assistant name or agent id is used, with tests for identity changes, fallback/reset behavior, and static-title composition.

## Reproduction Assessment

Yes for the observable current behavior: source and latest-release inspection show a static `OpenClaw Control` title and no lifecycle title writer. This is not a bug reproduction; it is a feature/UX request with a source-confirmed baseline.

## Solution Assessment

Unclear as a product decision: the closed implementation PR showed a plausible narrow path, but the durable fix should centralize title precedence instead of creating competing static and agent-derived writers.
