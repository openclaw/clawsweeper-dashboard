## Summary

Keep open. Current main has meaningful Android chat support, image attachments, assistant-launch routing into Chat, and Tailscale-aware connection guidance, but the official Android app still defaults and documents itself as a Connect-first companion node; the remaining request is Android product direction plus selective upstreaming, and #70993 only covers part of that work.

## What This Changes

The issue asks maintainers to consider an official chat-first Android surface or focused upstreaming of validated Android UX, media, and connectivity improvements from external Android forks.

## Best Possible Solution

Android/mobile maintainers should choose the official UX shape first: default to Chat, add an explicit chat-first mode, or keep Connect-first while inviting focused PR slices for chat UX, media playback, and Tailscale/connectivity improvements.

## Reproduction Assessment

Not applicable. This is a product-direction and upstreaming discussion, not a reproducible defect; source inspection shows partial current support rather than broken behavior against an existing contract.

## Solution Assessment

No for automation. A single ClawSweeper fix PR is not the best route because the remaining decision spans Android launch behavior, pairing/security UX, media scope, and review of the already-open broad chat-parity PR.
