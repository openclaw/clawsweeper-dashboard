## Summary

Keep open. Current main already has meaningful Android chat, image attachment, assistant-launch, voice/Talk, and Tailscale-aware connection support, but the official app still defaults into an Overview/Home shell and is documented as a Gateway companion node; the linked `clawchat2-public`, AndroidForClaw, and open Android chat-parity PR are useful product references, not a settled maintainer decision.

## What This Changes

The issue asks maintainers whether OpenClaw should adopt a chat-first Android surface or selectively upstream validated Android UX and media improvements from an independent fork.

## Best Possible Solution

Android/mobile maintainers should decide the official product shape first, then invite narrow PR slices for launch/navigation, media handling, and connectivity rather than importing either external fork wholesale.

## Reproduction Assessment

Not applicable; this is a product-direction and possible-upstreaming discussion, not a broken current behavior report. Source and docs inspection verify the current Android app shape, but there is no expected-vs-actual defect to reproduce.

## Solution Assessment

No concrete best fix exists yet; broad fork upstreaming is not the maintainable path. The best route is a maintainer Android product decision followed by focused PR slices, with the open Android chat-parity PR treated as one possible slice rather than the whole answer.
