## Summary

Keep open. Current main has meaningful Android chat support, assistant-launch routing into Chat, image attachments, and Tailscale-aware pairing docs, but the official app still implements and documents a Connect-first companion-node flow; the remaining question is Android product direction plus focused upstreaming from external chat-first work, not a completed core fix.

## What This Changes

The issue asks whether OpenClaw should support a chat-first Android surface or accept focused Android UX/media improvements from external Android fork work.

## Best Possible Solution

Android/mobile maintainers should choose the official shape first: keep Connect-first, default to Chat after pairing, or add an explicit chat-first mode; then review #70993 or narrower PR slices for chat UX, media playback, and connectivity improvements without importing a whole fork.

Not applicable as a bug reproduction. The current product state is high-confidence from source and docs inspection: onboarding/post-onboarding remains Connect-first, Chat exists as a tab and assistant destination, and chat media handling is currently image-oriented.
