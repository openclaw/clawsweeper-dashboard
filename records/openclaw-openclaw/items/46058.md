## Summary

Keep open: current main and v2026.6.6 already have meaningful Android chat, assistant-entry, image-attachment, and Gateway pairing support, but they still implement and document Android as a Connect-first companion node rather than a decided chat-first official surface. The linked broad Android chat PR closed unmerged, and narrower open Android chat items do not replace the maintainer product decision requested here.

## What This Changes

The issue requests discussion on whether OpenClaw should support a chat-first official Android surface or accept smaller Android UX/media improvements from an external fork.

## Best Possible Solution

Android/mobile maintainers should decide the official launch and navigation shape first, then invite narrow PR slices for chat-first entry, media playback, and connectivity improvements while preserving explicit Gateway pairing and security posture.

## Reproduction Assessment

Not applicable: this is a product-direction and possible-upstreaming discussion, not a broken current behavior report. Source, docs, tests, release state, and related PR checks show the current Android app shape, but there is no expected-versus-actual defect to reproduce.

## Solution Assessment

No for automation: the external forks and broad closed PR are useful references, but importing a broad fork or silently changing the default launch surface is not the maintainable fix. The best route is a maintainer Android product decision followed by focused PR slices with their own proof.
