## Summary

Keep open. Current main has a functional Android Chat tab, image attachments, assistant-launch routing into Chat, and Tailscale-aware pairing docs, but the official app still defaults to and documents a Connect-first companion-node flow. The remaining request is Android product direction plus possible focused upstreaming from external chat-first forks, and related PR #70993 is still open rather than merged.

## What This Changes

The issue asks whether OpenClaw maintainers want an official chat-first Android mode or small focused upstream PRs for Android chat UX, media handling, and connectivity improvements from an independent fork.

## Best Possible Solution

Android/mobile maintainers should choose the official shape first: keep Connect-first, make Chat the default after pairing, or add an explicit chat-first mode. After that decision, focused PR slices for chat UX, media playback, and Tailscale/connectivity can be reviewed without importing a whole fork; #70993 is the current related chat-UX candidate.

## Reproduction Assessment

Not applicable as a bug reproduction. The current product state is high-confidence from source and docs inspection: onboarding requires a connected node, post-onboarding defaults to Connect, Chat exists as a tab and assistant destination, and chat media handling is currently image-oriented.

## Solution Assessment

Unclear because this issue has no proposed patch and the remaining decision is product direction. The maintainable path is maintainer review of the desired Android surface, then narrow implementation PRs or review of #70993 for the chat-parity slice.
