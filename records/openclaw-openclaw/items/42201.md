## Summary

Keep open: current main still rejects unknown config keys during strict config validation and startup can still abort on that failure, so the requested behavior is not implemented. The PR remains useful source work, but the current head is stale against newer config validation APIs and also carries unrelated secrets-baseline churn, so it should not merge as-is.

## What This Changes

The PR adds warning-based unknown config key stripping across config validation/startup/write/migration paths, preserves unknown raw keys on selected write-back paths, expands legacy fail-closed checks, adds config regression tests, and changes `.secrets.baseline`.

## Best Possible Solution

Land a narrow current-main implementation that strips and sanitizes unknown config keys only where startup/read resilience needs it, preserves unknown raw keys on write/migration paths only when intended, keeps legacy/known-field/SecretRef/plugin validation fail-closed, and avoids unrelated secrets-baseline churn.

## Reproduction Assessment

Yes. Current main has a source-backed reproduction path: `validateConfigObject({ agents: { list: [{ id: "pi" }] }, customUnknownField: { nested: "value" } })` is still expected to return `ok: false`, and `loadConfig` throws after `validateConfigObjectWithPlugins` fails.

## Solution Assessment

No for the current PR head as-is. The behavior direction is reasonable, but the branch must be rebased or replaced so it preserves current validation params, plugin metadata snapshot reuse, SecretRef policy checks, and security-baseline handling.
