## Summary

Keep open. Current main still rejects unknown config keys at the strict base-validation layer, so the central startup-resilience bug is not implemented on main; however, this PR branch is stale against current main and would drop newer validation APIs/contracts if merged as-is.

## What This Changes

The PR adds warning-based unknown config key stripping across config validation, startup/write/migration paths, preserves unknown keys on write-back, expands legacy fail-closed checks, adds config regression tests, and changes `.secrets.baseline`.

## Best Possible Solution

Land a narrow current-main implementation that strips and sanitizes unknown keys only on the read/startup paths, preserves unknown keys on write/migration where intended, and keeps the newer config validation parameters, SecretRef policy checks, plugin metadata reuse, and plugin/channel validation contracts intact.

Yes. Current main has a direct source/test reproduction path: `validateConfigObject({ agents: { list: [{ id: "pi" }] }, customUnknownField: { nested: "value" } })` is still expected to return `ok: false`, and startup throws after `validateConfigObjectWithPlugins` fails.
