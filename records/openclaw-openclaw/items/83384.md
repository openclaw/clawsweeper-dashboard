## Summary

Close as a duplicate: the same reporter opened an identical live issue that already carries the active ClawSweeper reproduction and queueable-fix labels, so the remaining BM25/CJK work should stay consolidated there rather than split across two open issues.

## What This Changes

The issue reports that multi-term Chinese memory searches get zero BM25 text scores because the FTS query builder joins FTS tokens with AND.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/83386 as the canonical tracker and land a targeted memory search fix there that broadens zero-hit multi-term CJK/default-tokenizer keyword recall without globally replacing established AND semantics.

## Reproduction Assessment

Yes at source level: current main builds the reported strict AND MATCH expression, and a focused in-memory SQLite FTS5 probe reproduced the zero-row AND query while individual/OR terms matched. I did not run a full persisted memory_search user flow in this read-only review.

## Solution Assessment

Yes for closing this duplicate and consolidating work under the labeled canonical issue. For the code fix itself, a blanket AND-to-OR change is not the best path; a bounded zero-hit or CJK-aware fallback is safer.
