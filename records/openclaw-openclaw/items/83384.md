## Summary

Keep open: current main still builds multi-token memory FTS queries with `AND`, and hybrid mode does not broaden keyword recall when that exact query returns no rows, so the reported CJK text-score failure is credible and narrow enough to fix.

## What This Changes

The issue requests changing builtin memory BM25/FTS keyword query construction so multi-term Chinese/CJK searches can produce keyword-score matches instead of requiring every term in one chunk.

## Best Possible Solution

Add a targeted builtin memory search fix that preserves the exact AND query first, then broadens recall for empty multi-term CJK/default-tokenizer keyword results, likely by mirroring the existing FTS-only fallback in hybrid mode.

## Reproduction Assessment

Yes at source level: current code and a focused in-memory SQLite FTS5 probe reproduce the strict AND query returning no keyword rows when terms span chunks. I did not run a full live `memory_search` agent flow, so this is source-reproducible rather than end-to-end reproduced.

## Solution Assessment

No for an unconditional OR replacement; that would broaden all keyword searches and conflict with existing AND-focused tests. The maintainable fix is to keep strict matching as the first pass and add a narrow fallback for empty multi-term/CJK keyword results in the hybrid path.
