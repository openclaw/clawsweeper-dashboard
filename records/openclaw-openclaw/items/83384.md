## What This Changes

The issue reports that builtin memory BM25/FTS keyword scoring returns no matches for multi-term Chinese queries because query terms are joined with AND.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/83386 as the canonical work item and fix the memory keyword fallback there, while closing this duplicate to avoid split repair lanes.

## Reproduction Assessment

Yes at source level: current main still builds the reported strict AND FTS expression, and an in-memory unicode61 FTS5 probe reproduced individual-term recall but zero rows for the combined AND query. I did not run a full persisted memory_search agent flow in this read-only review.

## Solution Assessment

Yes for cleanup: closing this duplicate is the best repository action because the canonical issue remains open with the same report and fix-path labels. For the code fix, a zero-hit or CJK-aware fallback is safer than changing every multi-token query from AND to OR.
