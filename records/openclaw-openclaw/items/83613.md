## Label Justifications

- `P2`: This is a normal-priority Docker packaging/runtime bug affecting the default OpenAI agent path with a mitigation already on main.
- `impact:auth-provider`: The issue concerns provider/model runtime harness selection and whether the Codex-owned OpenAI path is available in Docker.

## Summary

Keep open: current main still allows official Docker builds to prune the Codex plugin, while the docs and runtime contract treat OpenAI agent turns as Codex-backed by default. The merged fallback in https://github.com/openclaw/openclaw/pull/82532 mitigates the 500 path by using PI when implicit Codex is unavailable, but it does not make the official image carry the default harness owner.

## What This Changes

Official Docker images should include the Codex plugin by default so the default OpenAI agent harness is present in the runtime image.

## Best Possible Solution

Keep the PI fallback as defense, but update official Docker packaging so Codex is retained as the default OpenAI harness owner and add regression proof for the image contents and runtime inspection.

## Reproduction Assessment

Yes: source inspection gives a high-confidence reproduction path because current main excludes `dist/extensions/codex/**`, prunes excluded plugin dirs unless kept, and official Docker release jobs keep only `diagnostics-otel`. I did not build the full image during this read-only review.

## Solution Assessment

Yes: retaining Codex in official Docker packaging is the narrow maintainable fix because docs already make Codex the default OpenAI agent harness; #82532 should remain as fallback defense, not become the normal official-image behavior.
