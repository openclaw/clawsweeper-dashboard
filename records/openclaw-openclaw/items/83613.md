## Label Justifications

- `P1`: The official Docker release path can omit the default OpenAI agent harness owner and break or silently reroute real chat-completions deployments.
- `impact:auth-provider`: The issue concerns OpenAI model routing and Codex agent harness registration in packaged Docker runtime images.

## Summary

Keep open: current main and the shipped v2026.5.12 release still have a source-backed path where official Docker builds prune the package-excluded Codex plugin because the release workflow only keeps `diagnostics-otel`. https://github.com/openclaw/openclaw/pull/82532 fixed the request-time PI fallback, but it did not make official images retain the default Codex harness owner.

## What This Changes

Official Docker images should include the Codex plugin by default so the default OpenAI agent harness is present in the runtime image.

## Best Possible Solution

Treat Codex as a retained first-party runtime asset for official Docker images, with a focused prune/package test and packaged-image smoke proving the Codex plugin manifest and `codex` agent harness are present.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.5.12 both exclude Codex from package dist, run the Docker prune script, and build official images with only `OPENCLAW_EXTENSIONS=diagnostics-otel`. I did not build or run a Docker image in this read-only review.

## Solution Assessment

Yes: retaining Codex in official Docker runtime packaging is the narrowest maintainable fix because docs define Codex as the default OpenAI agent harness owner while the existing PI fallback remains defense-in-depth.
