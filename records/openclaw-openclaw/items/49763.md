## Summary

Keep open. Current main fixes blocked active sends, but Ctrl+O still replaces a busy TUI activity status with a non-busy tools-expanded/collapsed notice; `renderStatus` stops the Loader for non-busy statuses, so the spinner-loss path remains source-reproducible.

## What This Changes

The issue asks for the TUI activity spinner to keep animating when a user submits or toggles tool output during an active run.

## Best Possible Solution

Preserve the current busy Loader while showing transient notices such as tool-output expansion/collapse, and add focused regression coverage for Ctrl+O during streaming/running states without changing queue policy or adding config.

## Reproduction Assessment

Yes, at source level. Put the TUI in a busy `activityStatus` such as `streaming` or `running`, press Ctrl+O, and the handler stores a non-busy tools status that makes `renderStatus` stop the Loader.

## Solution Assessment

Yes. The narrow maintainable fix is to keep busy run status and Loader ownership separate from transient UI notices, rather than changing message queue semantics or adding a setting.
