## Summary

Keep open. Current main still has a source-backed path where pressing Ctrl+O during a busy TUI run replaces the busy status with non-busy tool-toggle text, causing renderStatus to stop the Loader animation; the queued-message path is partially hardened but still needs regression coverage around the active-run status model.

## What This Changes

The issue requests that the TUI loading spinner keep animating when a user sends another message or toggles tool output during an active agent run.

## Best Possible Solution

Preserve the active busy Loader through transient tool-output notices and queued sends, then add focused TUI regression coverage for active-run Ctrl+O and queued-message behavior.

## Reproduction Assessment

Yes, via high-confidence source inspection: put the TUI in a busy activityStatus, press Ctrl+O, and the handler writes tools expanded/collapsed, which renderStatus treats as non-busy and uses to stop the Loader.

## Solution Assessment

Yes. The narrow maintainable fix is to keep transient tool-toggle messaging from replacing the busy run status, or otherwise preserve the prior busy state while still toggling chatLog.setToolsExpanded().
