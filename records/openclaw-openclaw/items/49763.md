## Summary

Keep open. Current main and the latest release still let Ctrl+O replace a busy TUI activity status with non-busy tool-toggle text, which drives the status renderer down the path that stops the loader.

## What This Changes

Fix the TUI so the active-run loading spinner keeps animating when a user queues input or toggles tool output during an active run.

## Best Possible Solution

Keep busy run loader state separate from transient tool-output notices, add focused regression coverage for Ctrl+O during streaming/running, and leave the existing busy-submit guard intact.

## Reproduction Assessment

Yes, at source level. Put the TUI in a busy activityStatus such as streaming or running, trigger Ctrl+O, and the handler stores a non-busy tools status that makes renderStatus stop the loader.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve busy loader ownership while showing transient tool-toggle feedback, rather than changing message queue semantics or adding a setting.
