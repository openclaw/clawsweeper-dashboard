## Summary

Keep open: current main still has a narrow TUI status-rendering bug where tool-output toggles during an active run replace the busy spinner with a non-busy status, and the related queueing work on main/open PRs does not fully own that spinner-preservation behavior.

## What This Changes

The issue asks for the TUI activity spinner to keep animating when a user queues a message or toggles tool output while an agent run is active.

## Best Possible Solution

Preserve the TUI busy spinner while an active or pending run exists across transient UI actions such as Ctrl+O and queued-submit acknowledgement, with focused unit/PTX coverage and no queue-policy changes.

## Reproduction Assessment

Yes, source-level: Ctrl+O during an active run calls `setActivityStatus("tools expanded" | "tools collapsed")`, and those values are not busy statuses, so `renderStatus()` stops the loader. I did not run a live Windows terminal reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow TUI status-preservation change with regression coverage. It should not change backend queue semantics or broaden the open gateway delivery PR unless maintainers intentionally combine the work.
