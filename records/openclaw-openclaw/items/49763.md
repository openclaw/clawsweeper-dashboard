## Summary

The current source still has the remaining Ctrl+O path from the report: toggling tool output writes a non-busy status, and the status renderer stops the Loader for any non-busy status. The active normal-message send path has been improved, but this visible spinner-loss case is still a narrow source-backed TUI bug.

## What This Changes

The issue reports that the TUI loading spinner freezes or disappears when message submission or Ctrl+O changes UI state during an active run.

## Best Possible Solution

Preserve the busy Loader while showing transient tool-output expansion/collapse notices, and add focused regression coverage for Ctrl+O during streaming/running without changing queue policy or adding config.

## Reproduction Assessment

Yes, at source level. Put the TUI in a busy `activityStatus` such as `streaming` or `running`, press Ctrl+O, and the handler stores a non-busy tools status that makes `renderStatus` stop the Loader.

## Solution Assessment

Yes. The narrow maintainable fix is to keep busy run status and Loader ownership separate from transient UI notices, rather than changing message queue semantics or adding a setting.
