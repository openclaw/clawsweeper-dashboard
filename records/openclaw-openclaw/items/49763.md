## Summary

Keep open. Current main still has a source-backed TUI path where Ctrl+O/tool-output toggling during a busy run replaces the busy activity status with a non-busy notice, which makes the status renderer stop the Loader; the ordinary active-send path is partly hardened, so the remaining fix should preserve busy status through transient UI notices.

## What This Changes

The issue reports that the TUI loading spinner freezes when a user sends another message or toggles tool output while an agent run is active.

## Best Possible Solution

Preserve the current busy activity status and Loader while showing transient notices such as tool-output toggles or rejected sends, then add focused TUI regression coverage for Ctrl+O and active-run submit attempts.

## Reproduction Assessment

Yes, via high-confidence source inspection: put the TUI in a busy activityStatus such as streaming or running, press Ctrl+O, and the handler stores a non-busy tools expanded/collapsed status that renderStatus uses to stop the Loader. I did not run a live Windows terminal reproduction in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to stop transient UI notices from overwriting an active busy status, rather than changing queue policy or adding a new configuration surface.
