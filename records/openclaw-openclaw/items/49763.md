## Summary

Keep open. Current main and the latest release still have a source-backed path where Ctrl+O during a busy TUI run replaces the busy status with non-busy tool-toggle text, which makes `renderStatus` stop the Loader; the queued-message path is partially hardened but still needs regression coverage around active-run status preservation.

## What This Changes

The issue asks for the TUI loading spinner to keep animating when another message is queued or tool output is toggled during an active run.

## Best Possible Solution

Preserve the active busy Loader through transient tool-output notices and queued sends, then add focused TUI regression coverage for active-run Ctrl+O and queued-message behavior.

## Reproduction Assessment

Yes, via high-confidence source inspection: put the TUI in a busy `activityStatus`, press Ctrl+O, and the handler writes `tools expanded` or `tools collapsed`, which `renderStatus` treats as non-busy and uses to stop the Loader.

## Solution Assessment

Yes. The narrow maintainable fix is to keep transient tool-toggle messaging from replacing the busy run status, or otherwise preserve the prior busy state while still toggling `chatLog.setToolsExpanded()`.
