## Label Justifications

- `P2`: This is a source-backed TUI behavior bug with visible user confusion but limited blast radius and no data-loss, security, or core availability failure.
- `impact:session-state`: The visible TUI activity state can drift from the active run state when Ctrl+O replaces a busy status with a non-busy notice.

## Summary

Keep open. Current main and the latest release still let Ctrl+O replace a busy TUI activity status with a non-busy tool-toggle notice, and `renderStatus` stops the loader for non-busy statuses; the normal busy-message submit path is already partially fixed by blocking ordinary sends while an active run is in flight.

## What This Changes

Fix the TUI busy indicator so it keeps animating when a user submits or toggles tool output during an active run.

## Best Possible Solution

Keep the busy run loader state separate from transient tool-output notices, preserving the existing blocked-submit behavior and adding a focused regression test for Ctrl+O during `streaming` or `running`.

## Reproduction Assessment

Yes, at source level. Put the TUI in a busy `activityStatus` such as `streaming` or `running`, trigger Ctrl+O, and the handler stores a non-busy tool-toggle status that makes `renderStatus` stop the loader.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve busy loader ownership while showing transient tool-toggle feedback, rather than changing message queue semantics or adding a setting.
