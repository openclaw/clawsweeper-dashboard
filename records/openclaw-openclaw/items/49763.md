## Summary

Keep open. Current main still has a source-backed path where Ctrl+O during an active busy TUI run replaces the busy status with non-busy tool-toggle text, which stops the Loader animation; the queued-message path now uses busy statuses, so the remaining repair should cover both paths rather than close the report.

## What This Changes

The issue asks for the TUI loading spinner to keep animating when a message is queued or tool output is toggled during an active run.

## Best Possible Solution

Preserve the active busy Loader through transient tool-output notices and queued sends, then add a focused TUI regression test for the active-run Ctrl+O and queued-message behavior.

## Reproduction Assessment

Yes, via high-confidence source inspection rather than a live terminal run: put the TUI in a busy activityStatus, press Ctrl+O, and the handler writes tools expanded/collapsed, which renderStatus treats as non-busy and stops the Loader.

## Solution Assessment

Yes. The narrowest maintainable fix is to decouple transient tool-toggle messaging from the busy status model, or preserve the prior busy state while still toggling chatLog.setToolsExpanded().
