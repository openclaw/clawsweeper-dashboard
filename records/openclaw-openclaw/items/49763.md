## Summary

Keep open. Current main still has a source-backed path where Ctrl+O during a busy TUI run replaces the busy status with non-busy tool-toggle text, causing the status renderer to stop the Loader animation; the ordinary second-send path is partly hardened, so the remaining fix should preserve busy status through transient UI notices and add regression coverage.

## What This Changes

Requests that the TUI loading spinner keep animating when a user tries to send during an active run or toggles tool output.

## Best Possible Solution

Preserve the current busy activity status and Loader when transient notices such as tool-output toggles or rejected sends occur during an active run, then add focused TUI regression tests for Ctrl+O and active-run send attempts.

## Reproduction Assessment

Yes, via high-confidence source inspection: put the TUI in a busy activityStatus such as streaming, press Ctrl+O, and the handler stores tools expanded/collapsed, which renderStatus treats as non-busy and uses to stop the Loader. I did not run a live Windows terminal repro in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to prevent transient tool-toggle or busy-send notices from overwriting an active busy status, rather than changing queue policy or adding a new configuration surface.
