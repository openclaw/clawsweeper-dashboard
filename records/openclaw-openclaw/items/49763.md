## Summary

Keep open. Current main still has a source-traceable Ctrl+O path that can replace an active busy status with non-busy tool-toggle text and stop the Loader; the queued-message paths have been partially hardened, but this reported trigger remains actionable and untested.

## What This Changes

The issue requests that the TUI loading spinner continue animating when a user queues another message or toggles tool output during an active run.

## Best Possible Solution

Keep the issue open and make a narrow TUI status-model fix so transient tool-output toggle notices do not replace an active busy Loader. The fix should preserve the active spinner whenever activeChatRunId or pendingOptimisticUserMessage indicates work is still running, while still toggling chatLog.setToolsExpanded and preserving existing queued-message safeguards.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: set the TUI to an active busy state, press Ctrl+O, and the handler writes tools expanded/tools collapsed, which renderStatus treats as non-busy and uses to stop the Loader interval.

## Solution Assessment

Yes. The narrowest maintainable solution is in the TUI status layer, not the backend or pi-tui dependency: keep busy rendering active during transient status notices and add a focused regression test for Ctrl+O during an active run.
