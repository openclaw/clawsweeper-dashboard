## Summary

Keep open: the underlying Discord/WebP media-loss bug is still source-reproducible on current main and this PR is the canonical linked fix path, but the latest head is conflicting with current main, misses the active native-reply helper after refactors, leaves a preserved-image fetch-cap gap, changes Plugin SDK API without baseline follow-through, and lacks required real behavior proof.

## What This Changes

The PR threads detected media MIME types into Discord upload blobs, preserves WebP/AVIF/APNG-like images in outbound media loading, adds animated-image detection, and updates Discord/WhatsApp/media tests plus a changelog entry.

## Best Possible Solution

Land a rebased canonical fix that threads MIME types through standard, component, and native Discord uploads, preserves animated/WebP/AVIF media without over-buffering remote fetches, and either updates SDK baseline artifacts or keeps the preservation knobs internal.

## Reproduction Assessment

Yes at source level. Current main still wraps Discord media buffers in typeless Blobs and still sends WebP/APNG-like images through the shared optimizer; I did not establish a live Discord reproduction.

## Solution Assessment

No. The direction is right, but the branch needs a current-main rebase/port to `native-command-reply.ts`, bounded preserved-image fetching, SDK baseline follow-through, and real behavior proof before it is the maintainable fix.
