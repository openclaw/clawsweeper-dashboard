## Summary

Keep open. Current main still has the reported Discord/media behavior, and this PR remains a relevant implementation candidate for the linked open bug, but the branch is not merge-ready against current main because the native interaction reply code has since moved and the diff fails to apply cleanly; it also hardcodes an old 8 MB Discord media fallback where current main and docs use 100 MB.

## What This Changes

The PR threads detected media content types into Discord upload blobs, adds WebP/AVIF preservation and APNG/WebP animation detection in shared media loading, updates Discord standard/component/native send paths, and adds tests plus a changelog entry.

## Best Possible Solution

Land one rebased, tested fix, using this PR if repaired, that updates the active Discord standard send, component send, and native interaction reply paths, preserves detected MIME types and animated/WebP/APNG bytes intentionally, keeps current Discord account/default media caps, and updates any public SDK/docs surface if preserveWebp/preserveAvif remains plugin-facing.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: Discord media sends load media.contentType but do not pass it into Blob construction, while shared media loading still optimizes non-GIF images through PNG/JPEG conversion; a live Discord token would only confirm the already-traced behavior.

## Solution Assessment

No, not as currently proposed against main. The implementation direction is right, but the branch must be rebased onto the current native-command-reply.ts split and its new Discord media-cap fallback must align with the current 100 MB default before it is the narrowest maintainable fix.
