## Summary

Keep open: the reported Discord/WebP media bug still exists on current main and this PR is the active closing path, but the branch needs repair before merge because it is stale against current main, has blocking review findings, and lacks real after-fix Discord proof.

## What This Changes

The branch threads detected media MIME types into Discord upload blobs, preserves WebP/AVIF and animated PNG/WebP media in shared media loading, updates Discord and WhatsApp tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased equivalent fix that threads content types through every active Discord upload path, preserves WebP and animated media without weakening fetch caps or SDK contracts, and includes focused tests plus redacted Discord proof.

## Reproduction Assessment

Yes, at source level. Current main wraps Discord media buffers in untyped Blobs and sends WebP through the image optimizer; I did not run a live Discord upload in this read-only review.

## Solution Assessment

No. The direction is right, but this branch must be rebased onto the current native reply helper, update the Plugin SDK baseline, harden preserved-image fetch caps, and provide real Discord proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Port native media replies to the active helper:** `extensions/discord/src/monitor/native-command.ts:1370-1378`
  - body: Current main no longer uses the in-file helper this PR edits; `native-command.ts` imports `deliverDiscordInteractionReply` from `native-command-reply.ts`. Move the media-preservation and typed-Blob changes into that active helper, otherwise native command media replies will still load without preserve flags and wrap untyped Blobs after the branch is rebased.
  - confidence: 0.91
- **[P2] Update the Plugin SDK API baseline:** `src/plugin-sdk/outbound-media.ts:13-16`
  - body: Adding `preserveWebp` and `preserveAvif` to exported `OutboundMediaLoadOptions` changes the Plugin SDK surface, but the generated API baseline files are not in the PR. The plugin SDK API check is expected to fail until the baseline is regenerated for the intentional contract change.
  - confidence: 0.9
- **[P2] Clamp preserved remote images by detected MIME:** `src/media/web-media.ts:500-505`
  - body: The new fetch-cap adjustment only looks at the URL extension. Extensionless or signed URLs that return WebP/AVIF still download up to the 100MB document cap before the later Discord-size rejection, adding avoidable bandwidth and memory pressure; enforce the lower cap when the response MIME identifies a preserved format.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Needs attention: preserved image passthrough changes resource bounds and can retain raw container metadata.

Concerns:

- **[medium] Preserved remote images can bypass early fetch caps:** `src/media/web-media.ts:500`
  - body: For extensionless or signed WebP/AVIF URLs, the PR can fetch up to the document cap before rejecting against the Discord media cap after MIME detection, increasing bandwidth and memory exposure for a request that should fail earlier.
  - confidence: 0.84
- **[low] Raw preserved images may keep metadata:** `src/media/web-media.ts:456`
  - body: Skipping re-encode for WebP/AVIF/APNG can preserve EXIF/XMP-style container metadata that the optimizer path often removed, so the final fix should make that privacy tradeoff explicit or strip metadata where preservation does not require it.
  - confidence: 0.66
