## Label Justifications

## Summary

Keep open: current main and the latest release still treat WhatsApp socket-open as login success, and the issue is already paired with two open closing PRs that target the same durability boundary. The right cleanup is to review and land one of those PRs, then close this issue after merge.

## What This Changes

The issue asks OpenClaw to make WhatsApp login success wait until the selected WhatsApp auth directory is durably persisted and readable from the persisted state directory.

## Best Possible Solution

Review and land one canonical fix PR that gates the shared WhatsApp login waiter on the existing disk-backed auth decision, then close this issue after merge and restart-oriented proof where feasible.

## Reproduction Assessment

Yes from source: current main's shared login waiter reports `connected` after socket-open without a post-open disk auth read, while existing helpers can classify persisted auth as unstable or missing. I did not run the live WhatsApp plus Docker rebuild reproduction.

## Solution Assessment

Yes. Moving the success boundary into the shared WhatsApp login waiter and reusing `readWebAuthExistsForDecision` is the narrowest maintainable fix because CLI login and QR login both flow through that controller path.
