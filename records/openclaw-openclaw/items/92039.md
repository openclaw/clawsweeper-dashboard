## Label Justifications

- `P1`: A routine Docker rebuild can turn an apparently successful WhatsApp login into a workflow-blocking relink requirement.
- `impact:data-loss`: The issue reports authentication state that appears linked but is not durably recoverable from the persisted state volume.
- `impact:message-loss`: When persisted authentication is missing, the WhatsApp listener becomes inactive and channel delivery stops.
- `impact:auth-provider`: The affected behavior is durable channel authentication state and linked-session recovery.

## Summary

Current `main` still reports WhatsApp login success at socket-open without confirming that credentials are stable and linked on disk. Two open PRs already target this exact gap, so this issue should remain open until maintainers select and merge one canonical implementation.

Likely related people: @mcaxtr (high confidence, recent WhatsApp login/auth work), @steipete (medium confidence, recent auth-store refactor and area maintenance), and @itsuzef (medium confidence, recent connection-controller work).

## What This Changes

Require WhatsApp login to confirm stable, linked credentials in the selected persisted auth directory before returning success.

## Best Possible Solution

Land one plugin-local shared-waiter fix that waits for the existing credential persistence barrier, verifies stable linked auth from the same directory, and covers delayed persistence plus restart-style directory reuse; then close the competing PR as superseded.

## Reproduction Assessment

Yes, at source level with high confidence: current `main` returns success at socket-open while queued credential persistence and stable disk-auth readback are separate operations. This review did not rerun the live WhatsApp Docker rebuild cycle.

## Solution Assessment

Yes in fix shape: placing the gate in the shared login waiter and reusing `readWebAuthExistsForDecision` is the narrowest maintainable solution. Maintainers still need to choose between the two overlapping PR implementations.
