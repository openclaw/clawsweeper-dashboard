## Label Justifications

- `P1`: The issue describes a real channel workflow that appears successfully linked but becomes unusable after a routine Docker rebuild and requires QR relinking.
- `impact:message-loss`: Loss of the active WhatsApp authentication state suppresses subsequent channel delivery until the account is relinked.
- `impact:auth-provider`: The reported failure is specifically about durable channel authentication state and recovery from persisted credentials.

## Summary

Current main still reports WhatsApp login success when the socket opens, before the separately queued credentials write is guaranteed to have completed or produced stable linked state on disk. The report is actionable and aligns with existing durability safeguards already used before auth reads and reconnects, so it should remain open for a focused regression fix.

## What This Changes

Ensure WhatsApp login reports success only after the selected account’s credentials are durably persisted and recoverable from the reused auth directory.

## Best Possible Solution

At the WhatsApp login owner boundary, await the existing per-auth-directory credential queue with a bounded timeout, re-read stable linked auth from disk, and return success only when that durable state is confirmed; cover delayed saves and reuse of the same auth directory after socket teardown.

## Reproduction Assessment

No, not at live-runtime confidence: no controlled Docker before/after artifact was provided or run. Source inspection does expose the reported race with medium confidence because connection success and queued credential persistence are independent, and the existing test accepts connection-open alone.

## Solution Assessment

Yes in direction. Reusing the existing credential-queue barrier and stable disk-auth decision at the login completion boundary is the narrowest fix; a Docker-specific workaround, new config option, or fallback store would be worse.

## Repair Work Prompt

Fix the WhatsApp login durability race tracked by https://github.com/openclaw/openclaw/issues/92039. In the WhatsApp plugin login completion path, do not return CLI or QR success solely because `waitForWaConnection` resolved. Reuse the existing per-`authDir` credential-save queue barrier and stable linked-auth disk decision with a bounded timeout; return a clear retryable failure when the queue times out, auth remains unstable, or disk state is not linked. Add focused coverage where socket-open precedes a delayed credential save, plus restart-oriented coverage that tears down the socket and reuses the same temporary auth directory. Keep ownership inside `extensions/whatsapp`; do not add config or environment options, Docker-specific runtime branches, alternate/fallback storage, compatibility shims, or `CHANGELOG.md` edits. Preserve the existing code-515 and timeout restart behavior and use the same canonical auth-state definition rather than duplicating it.
