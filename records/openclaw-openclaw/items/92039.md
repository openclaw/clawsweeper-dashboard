## Label Justifications

- `P1`: Current main can report successful WhatsApp authentication that does not survive a Docker rebuild, blocking a real channel workflow.
- `impact:data-loss`: The reusable linked-auth state may be absent after restart despite the earlier successful-login result.
- `impact:message-loss`: When authentication does not survive, the WhatsApp listener becomes inactive and message delivery fails.
- `impact:auth-provider`: The defect concerns persistence and verification of WhatsApp authentication credentials.

## Summary

Current main still treats socket-open as sufficient for WhatsApp login success even though credential persistence is queued and the documented reusable state is the on-disk auth directory. The issue should remain open because two unmerged PRs claim it; the clean, narrower candidate should be reviewed as the canonical fix rather than closing the issue before merge.

## What This Changes

Require WhatsApp login success to mean the selected account’s linked credentials are stably persisted and recoverable after restart or Docker rebuild.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92108 if its narrow shared-waiter implementation and restart-oriented proof remain acceptable, then close the dirty competing PR and let the merge close this issue.

## Reproduction Assessment

Yes at source level: current main returns connected after socket-open and then claims credentials were saved, while persistence is a separate queued operation with an existing stable disk-read decision. This review did not independently run the full live WhatsApp Docker rebuild cycle.

## Solution Assessment

Yes. Enforcing the invariant once in the shared login waiter through the existing auth-store decision is narrower and more maintainable than adding caller-specific checks or another persistence mechanism.
