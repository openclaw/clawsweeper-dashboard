## Label Justifications

- `P1`: A routine Docker rebuild can turn an apparently successful WhatsApp login into a workflow-blocking relink requirement.
- `impact:data-loss`: The issue concerns authentication state that appears saved but may not be durably present for the next process.
- `impact:message-loss`: The resulting inactive WhatsApp listener prevents messages from being delivered.
- `impact:auth-provider`: The failure is specifically about persisted linked-session credentials and authentication recovery.

## Summary

Current main still has the reported false-positive success boundary: WhatsApp login returns connected when the socket opens, without waiting for the asynchronous credential write or confirming that linked auth is stable on disk. The existing persistence and auth-state helpers make this a narrow, plugin-local repair rather than a new product capability.

## What This Changes

Make WhatsApp login report success only after the selected account’s credentials are durably persisted and confirmed as stably linked on disk.

## Best Possible Solution

Make the shared login-result function drain the selected auth directory’s credential queue, verify canonical disk auth is stably linked, and return a clear retryable failure on timeout, unstable state, or missing linked credentials.

## Reproduction Assessment

Yes at source level: resolve the connection waiter while the queued credential write remains pending, and current main returns connected without checking disk stability. A real Docker restart run is still needed to prove the reporter’s exact deployment path.

## Solution Assessment

Yes; applying the existing per-account queue barrier and stable-auth reader at the shared login completion boundary is the narrowest maintainable fix and avoids duplicate persistence logic or new configuration.

## Repair Work Prompt

Fix the WhatsApp login durability race reported at https://github.com/openclaw/openclaw/issues/92039. In the shared login completion path, do not return or publish connected merely because `waitForWaConnection` resolved. Reuse the existing per-auth-directory credential persistence barrier and canonical stable/unstable auth decision; return a clear retryable failure when the queue times out, auth remains unstable, or disk auth is not linked. Add focused regression coverage where socket-open occurs before credential persistence stabilizes, and provide restart-oriented real behavior proof using the same persisted auth directory or Docker volume. Keep the change inside the WhatsApp plugin, add no config or environment surface, preserve atomic/fsync-backed writes, and do not edit CHANGELOG.md.
