## Summary

Keep open. The reporter supplied a focused live reproduction and a control RPC, and current main still has the same scope-blind Gateway device-identity decision that can drop device identity before the `agent` call asks for `operator.write`.

## What This Changes

The issue reports native `sessions_spawn` failing with `missing scope: operator.write` even when the local operator device token already has the full operator scope set.

## Best Possible Solution

Add a targeted regression for native `sessions_spawn -> Gateway agent` under token auth with a full-scope operator device token, then preserve the necessary device identity/scope propagation only for the scoped internal call path without widening remote auth or owner/admin privileges.

## Reproduction Assessment

Yes, with caveat: the reporter provided a concrete redacted live repro plus a direct `agent` control RPC, and current source still has the implicated identity/scope-selection shape. I did not execute that live Gateway setup against current main.

## Solution Assessment

Unclear. The reporter's local hotfix points at the right boundary, but current tests intentionally expect device identity to be omitted for scoped backend shared-auth loopback calls, so the maintainable fix should be regression-led and security-reviewed.
