## Summary

Keep open. Current main and the latest release still choose the backend loopback shared-auth device-identity omission without considering the resolved per-call scopes, matching the reporter's live sessions_spawn failure and direct agent RPC control.

## What This Changes

The issue reports that native sessions_spawn fails with missing operator.write even when the local operator device token already has full operator scopes.

## Best Possible Solution

Add a regression for native sessions_spawn to Gateway agent under token auth, then preserve approved device identity for explicitly scoped trusted internal calls without changing remote or untrusted auth behavior.

## Reproduction Assessment

Yes. The reporter provided a concrete live sessions_spawn failure plus a direct Gateway agent control, and current main still has the same source-level identity/scope-selection shape; I did not run the live Gateway setup in this read-only pass.

## Solution Assessment

Unclear. The local hotfix points at the right seam, but current tests intentionally assert deviceIdentity null for scoped backend shared-auth calls, so the final fix needs regression coverage and maintainer auth-policy approval.
