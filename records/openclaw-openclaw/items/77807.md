## Summary

Keep open. Current main still has the Gateway call/device-identity boundary implicated by the report, the reporter added focused live proof plus a local hotfix explanation, and I did not find a merged or open PR that resolves this exact `sessions_spawn -> agent` `operator.write` failure.

## What This Changes

The issue reports that native `sessions_spawn` fails with `missing scope: operator.write` even when the local operator device token has the full operator scope set.

## Best Possible Solution

Add a targeted regression for token-auth native `sessions_spawn -> Gateway agent` when the internal call is explicitly write-scoped, then adjust Gateway call/device-identity selection so the request arrives with `operator.write` without broadening device-less auth outside the intended local backend path.

## Reproduction Assessment

Yes. The reporter provided live `sessions_spawn` failure plus direct `agent` RPC control logs, and current main still contains the source path where resolved scopes are not considered before omitting device identity; I did not run a local Gateway myself.

## Solution Assessment

Unclear. The reporter's hotfix points at the right boundary, but the final patch should be test-led and reviewed as an auth-policy change rather than blindly preserving device identity for every internal call.
