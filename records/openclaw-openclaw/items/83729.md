## Summary

Keep open: this is useful security hardening with sufficient terminal proof, but it is not merge-ready because common `$HOME`/`${HOME}` path references can bypass a home-relative denied path entry.

## What This Changes

The PR adds `tools.exec.deniedPaths` config/schema/docs wiring, runtime exec preflight matching, path normalization, and regression tests for direct, shell-wrapped, command-substitution, sandbox-relative, and home-relative path operands.

## Best Possible Solution

Keep the new exec-owned denylist approach, but make command candidate normalization cover common shell-expanded path prefixes or fail those forms closed before exposing the config as a security control.

## Reproduction Assessment

Yes, from source inspection: a command like `cat "$HOME/.openclaw/credentials/provider.key"` leaves an env-prefixed token, while candidate resolution only handles literal absolute, relative, and `~/` paths. I did not run the repro because this review is read-only.

## Solution Assessment

No, not yet. The exec preflight/config boundary is the right place for this defense-in-depth feature, but the matcher needs env-prefixed path handling or fail-closed semantics before the new security contract is safe to merge.
