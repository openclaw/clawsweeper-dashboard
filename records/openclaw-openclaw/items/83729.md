## Summary

Keep this PR open: the latest head fixes the prior host=node HOME trust gap, but the new denied-path normalizer still mishandles Windows drive-root recursive patterns, so a configured security boundary can fail open for patterns like `C:\**`.

## What This Changes

The PR adds an optional `tools.exec.deniedPaths` config surface, exec preflight path matching across gateway/sandbox/node, schema/docs wiring, config normalization changes, and regression tests.

## Best Possible Solution

Fix Windows drive-root pattern normalization, add a focused regression, and document the host=node home-relative fail-closed semantics before maintainer security review.

## Reproduction Assessment

Yes, source-reproducible. The PR head strips `C:\**` to `C:`, and Node's win32 resolver treats `C:` as drive-relative to the workdir, so drive-root denied patterns can miss absolute operands elsewhere on that drive.

## Solution Assessment

No, not yet. The exec preflight layer and fail-closed node HOME direction are reasonable, but Windows drive roots need to be preserved or rejected fail-closed before the security knob is safe to land.
