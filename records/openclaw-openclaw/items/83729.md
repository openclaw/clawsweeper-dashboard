The PR adds `tools.exec.deniedPaths` config, docs, schema wiring, runtime exec preflight checks, and regression tests for gateway, sandbox, and node path blocking.

## Best Possible Solution

Keep the denied-path preflight, but preserve authored denied-path patterns until the exec guard can resolve them in the selected execution namespace, with config-through-runtime tests for gateway, sandbox, and node.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load config with `tools.exec.deniedPaths: ["~/.openclaw/credentials/**"]`, let `normalizeConfigPaths` expand it against the gateway home, then run sandbox or node exec where the guard resolves candidates against a different HOME/namespace.

## Solution Assessment

No. The runtime guard direction is maintainable, but the config normalizer should not consume home-relative denied patterns before the namespace-aware exec guard sees them.
