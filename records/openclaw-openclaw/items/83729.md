## Summary

Keep open: this is a useful security-hardening direction with supplied live-output proof, but the new denylist still misses env-expanded path operands, so a configured denied path can reach exec through a tool-provided environment variable.

## What This Changes

Adds a `tools.exec.deniedPaths` config key, wires it through exec config/docs/schema, and adds gateway/sandbox/node preflight deny-path matching with regression tests.

## Best Possible Solution

Resolve or fail closed on variable-backed path operands in the denied-path preflight across gateway, sandbox, and node exec, while keeping broader secret isolation tracked in https://github.com/openclaw/openclaw/issues/11829.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: the exec schema accepts `env`, while the new candidate resolver only handles literal paths and HOME-prefixed forms, so `cat "$SECRET"` with `SECRET` set to a denied path is not matched before execution.

## Solution Assessment

No. A pre-exec denied-path guard is a reasonable defense-in-depth feature, but it needs env-expanded operand handling before it is a reliable security boundary.
