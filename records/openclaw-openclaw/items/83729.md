## Summary

Keep open: the PR is useful and the shell-wrapper plus sandbox namespace fixes appear to address the earlier ClawSweeper findings, but the new security config still accepts home-relative denied path entries without normalizing them, which can leave common credential paths unprotected.

## What This Changes

The PR adds `tools.exec.deniedPaths` config/schema/docs wiring, an exec preflight path guard, and regression coverage for direct, shell-wrapped, command-substitution, and sandbox-relative path operands.

## Best Possible Solution

Merge after `deniedPaths` has explicit, tested path-normalization semantics across config read/write and runtime matching, or after unsupported `~` patterns are rejected with a clear validation error.

## Reproduction Assessment

Yes. Source inspection shows `deniedPaths` is a new path-list config array, but the config read normalizer does not include that key and the runtime resolves non-absolute deny patterns against `workdir`, so `~/.openclaw/credentials/**` will not match the actual home directory.

## Solution Assessment

No, not yet. The guard is close, but the maintainable solution needs explicit normalization or fail-closed validation for authored home-relative denied paths before this becomes a reliable security boundary.
