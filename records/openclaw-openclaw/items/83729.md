## Summary

Keep open: the latest follow-up fixes cover the earlier gateway, sandbox, shell-wrapper, and HOME-prefixed cases, but the denylist still runs node checks with gateway path semantics before the node target/platform is resolved, leaving cross-platform node denied paths ineffective.

## What This Changes

The PR adds optional `tools.exec.deniedPaths` config/schema/docs wiring plus exec preflight checks for commands that reference configured sensitive paths.

## Best Possible Solution

Enforce denied paths in the execution namespace: gateway paths locally, sandbox paths in the container namespace, and node paths after resolving the selected node platform, or explicitly reject/document node as out of scope.

## Reproduction Assessment

Yes, by source inspection for the blocking review finding: the PR checks `host=node` commands with `namespace: "host"` before node target/platform resolution, while current node exec deliberately avoids gateway cwd/path assumptions for remote nodes.

## Solution Assessment

No. The gateway and sandbox direction looks maintainable, but node support should either use resolved node platform semantics or be explicitly scoped out instead of silently using gateway path semantics.

- **[P1] Enforce node denied paths with node path semantics:** `src/agents/bash-tools.exec.ts:1757`
