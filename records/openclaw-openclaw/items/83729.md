## Summary

Keep open. The exec denylist guard is useful and the latest head fixes the prior shell, sandbox, HOME, and Windows drive-root cases, but host=node home-relative denied paths now fail closed for all node exec while the docs describe only path-specific blocking.

## What This Changes

The PR adds an optional `tools.exec.deniedPaths` config key, exec preflight path matching for gateway/sandbox/node hosts, schema/docs metadata, generated config hash updates, and regression tests.

## Best Possible Solution

Keep the denylist as opt-in defense-in-depth, but make the supported pattern language and host=node HOME behavior explicit or add a trusted node HOME contract before merge.

## Reproduction Assessment

Yes, source-reproducible. PR head throws for any host=node `~...` denied path pattern before command matching, while the docs describe deniedPaths as path-reference blocking only.

## Solution Assessment

No, not quite. The guard placement is broadly maintainable, but the final merge should either document the host=node home-relative fail-closed limitation or add a trusted node HOME contract.
