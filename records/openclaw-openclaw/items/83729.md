## Summary

Keep open. The PR is useful security hardening for the broader secret-protection roadmap, but the latest patch still has a host=node home-relative denied-path gap: node matching can resolve `~` from the gateway-side env instead of a trusted node runtime home, so the advertised `~/.openclaw/credentials/**` protection is not reliable for node hosts.

## What This Changes

The PR adds an optional `tools.exec.deniedPaths` config key, exec preflight denied-path matching for gateway/sandbox/node, docs/schema metadata, config normalization changes, and regression tests.

## Best Possible Solution

Land the config surface only after host=node home-relative patterns either resolve from a trusted node runtime home or fail closed as unsupported, with production-path regression coverage through `createExecTool`.

## Reproduction Assessment

Yes, at source level. The production call chain shows node denied-path matching uses gateway-side `params.env` for `~` pattern resolution while the selected node target does not expose a trusted runtime HOME.

## Solution Assessment

No. The exec preflight is the right general layer, but host=node home-relative matching must use a trusted node namespace or fail closed instead of relying on env values that are not the node's verified home.
