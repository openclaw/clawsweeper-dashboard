## Summary

Keep open. The PR is a useful security-hardening direction, but it is not merge-ready because sandbox denied-path checks resolve relative operands against the host workdir while execution happens in the container workdir, leaving the mounted-secret case bypassable.

## What This Changes

The PR adds `tools.exec.deniedPaths` config wiring, exec preflight path scanning, docs/config metadata, generated config baseline hashes, and regression tests for direct and shell-wrapped denied path operands.

## Best Possible Solution

Keep the config surface only if denied-path matching is performed in the actual execution namespace for each host mode, with sandbox regression coverage for relative traversal to mounted denied paths.

## Reproduction Assessment

Yes, by source inspection: with sandbox `containerWorkdir` separate from host `workdir`, a relative operand can resolve to `/run/secrets/...` inside the container while the PR checks a different host path. I did not run a live Docker repro because this review kept the checkout read-only.

## Solution Assessment

No. The config surface and preflight location are reasonable, but the implementation needs namespace-aware path normalization or a narrower documented contract before it is a safe security boundary.
