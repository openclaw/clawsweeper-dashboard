## Summary

Keep open: the follow-up addresses the earlier shell-wrapper bypass for gateway-style commands, but the new guard still appears to resolve relative operands in the host namespace while sandbox execution runs them in the container namespace. That leaves a plausible bypass for the mounted-secret scenario this PR is trying to harden.

## What This Changes

The PR adds `tools.exec.deniedPaths` wiring, exec preflight scanning, docs/config metadata, generated baseline hashes, and regression tests for direct and shell-wrapped denied path operands.

## Best Possible Solution

Make denied-path matching use the actual execution namespace for each host mode, especially sandbox container paths, and add regression coverage that proves relative traversal to a mounted denied path is blocked.

## Reproduction Assessment

Yes, by source inspection rather than by executing the PR branch: with sandbox `containerWorkdir` separate from host `workdir`, a relative path can resolve to `/run/secrets/...` inside the container while the PR checks a different host path. The checkout stayed read-only, so this was not run as a live Docker repro.

## Solution Assessment

No. The config surface and preflight location are reasonable, but the current implementation should normalize paths in the execution namespace or explicitly scope the feature to hosts where it can prove that namespace.
