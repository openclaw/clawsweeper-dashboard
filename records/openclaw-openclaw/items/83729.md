## Summary

Keep open: the denied-path feature is useful and the PR includes credible live-output proof, but the current implementation lets sandbox/node callers shift home-relative deny patterns with a request-supplied HOME value, leaving a security-boundary bypass before merge.

## What This Changes

The PR adds an optional `tools.exec.deniedPaths` config surface, exec preflight matching for gateway/sandbox/node commands, schema/docs metadata, generated config baseline hashes, and regression tests.

## Best Possible Solution

Land the config surface only after home-relative patterns are resolved from a trusted runtime home, or HOME overrides are rejected for sandbox/node denied-path evaluation, with regressions for malicious HOME overrides.

## Reproduction Assessment

Yes, at source level: configure `tools.exec.deniedPaths` with `~/.openclaw/credentials/**`, run sandbox or host=node exec with `env.HOME=/tmp`, then reference the literal protected credential path. I did not execute it because this review is read-only, but the source path shows the guard normalizes the deny root from caller-controlled HOME.

## Solution Assessment

No. The config surface is a reasonable defense-in-depth direction, but the implementation should not derive security pattern roots from model-controlled env values for sandbox/node runtimes.
