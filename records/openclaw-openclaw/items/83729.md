## Summary

Keep this PR open: the config direction is useful security hardening and the supplied proof covers the direct literal operand case, but the current implementation is bypassable through common shell-wrapper payloads before it can be treated as a documented security boundary.

## What This Changes

Adds `tools.exec.deniedPaths` config, exec preflight path scanning, config schema/help/docs/baseline updates, and a regression test for direct literal denied paths.

## Best Possible Solution

Keep the config surface only if denied paths are enforced through existing command-carrier and shell-wrapper analysis, with fail-closed behavior for unparseable expansion and regression coverage for wrapper/substitution bypasses.

## Reproduction Assessment

Yes for the PR defect by source inspection: a command such as `bash -lc 'cat /run/secrets/provider.key'` is scanned as an outer argv payload, so the inner denied path is not matched. I did not execute the PR branch because this review kept the checkout read-only.

## Solution Assessment

No. The config direction may be useful for the open security roadmap, but this implementation needs nested command analysis or fail-closed parser behavior before it is the right security-boundary fix.
