## Summary

Keep open. Current main still leaves `session_status` automation with only optional human-readable usage text, so the requested stable structured window payload and explicit unavailable/reset metadata remain unimplemented.

## What This Changes

Request stable structured `session_status` usage data for 5h/day quota windows, availability state, source metadata, and reset explanation instead of cron parsing optional status text.

## Best Possible Solution

Add an additive `session_status.details.usage` contract that exposes provider/source, availability or unavailable reason, plan when known, normalized windows such as 5h/day/week with percent and resetAt, and reset explanation only when upstream data supports it.

## Reproduction Assessment

Partially yes. The missing structured usage fields and silent omission path are deterministic in current source; the intermittent Day-window jumps need live OAuth provider monitoring to reproduce fully.

## Solution Assessment

Yes. An additive structured `details.usage` payload is the narrowest maintainable path because it avoids breaking existing text consumers while giving cron automation a stable availability/window contract.
