## Summary

Keep open. Current main still exposes provider quota windows to `session_status` only through optional human-readable status text, while the tool result has no structured `details.usage`, explicit availability state, source metadata, or reset explanation. Related #8635 is broader token/cost API work, and #45342 is scoped to Anthropic usage caching rather than this cron-safe `session_status` contract.

## What This Changes

Request stable, structured `session_status` usage data for provider quota windows, including 5h/day percentages, reset times, source, unavailable state, and reset metadata instead of intermittently omitted text lines.

## Best Possible Solution

Add an additive structured provider-quota payload under `session_status.details.usage` while keeping the existing rendered status card. The contract should expose provider/source, availability or unavailable reason, plan when known, normalized windows such as 5h/day/week with percent and resetAt, and reset explanation only when upstream data supports it.

## Reproduction Assessment

Partially yes. The intermittent Day-window jumps need a live OAuth provider and long-running cron monitor to reproduce, but the missing structured usage fields and silent omission path are deterministically visible in current source.

## Solution Assessment

Yes, an additive `details.usage` contract is the narrowest maintainable fix. It avoids breaking existing text consumers while giving automation a stable availability/window payload instead of parsing optional status text.
