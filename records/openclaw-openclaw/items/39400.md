## Summary

Keep open. Current main has fixed important Codex usage-routing and weekly-window labeling pieces, but `session_status` still exposes usage only as optional rendered text and still omits provider usage on timeout/error/no-window paths, so the stable structured availability/window contract requested here remains unresolved.

## What This Changes

The issue asks for stable machine-readable `session_status` provider usage windows, reset metadata, and explicit unavailable/source states for cron automation.

## Best Possible Solution

Define a documented additive `session_status.details.usage` contract, or an equivalent supported agent-facing tool, that reports normalized provider windows plus explicit availability/source metadata while preserving the existing rendered `statusText` path.

## Reproduction Assessment

Partially yes. Source inspection on current main shows there is no structured usage payload and provider usage is silently omitted from status text on timeout/error/no-window paths, but the reported live Day-window jumps still need credentialed Codex OAuth cron monitoring to reproduce end to end.

## Solution Assessment

Unclear until maintainers choose the public API shape. An additive `details.usage` payload is the narrowest likely direction, but the schema, source/unavailable semantics, and privacy boundary overlap the broader open usage API discussion.
