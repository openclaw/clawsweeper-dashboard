## Summary

Keep open. Current main has restored important Codex usage-window display behavior, but `session_status` still exposes provider quota data only as optional rendered status text and does not provide the stable structured availability/window metadata requested here.

## What This Changes

Request stable structured `session_status` usage-window fields, source/availability metadata, and reset context for cron-based usage monitoring.

## Best Possible Solution

Define a documented additive `session_status.details.usage` contract, or an equivalent supported agent-facing usage tool, with normalized provider windows plus explicit availability, source, and reset metadata while preserving the rendered `statusText` path.

## Reproduction Assessment

Partially yes. Source inspection on current main shows there is still no structured usage payload and provider-window data can be omitted from the rendered status line; the reported live Day-window jumps still need credentialed Codex OAuth cron monitoring to reproduce end to end.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy boundary. An additive structured usage payload is the narrowest likely direction, but schema, source/unavailable semantics, and reset explanations overlap the broader open usage API discussion.
