## Summary

Keep open. Current main partially guards Codex weekly-window labeling, but `session_status` still exposes provider usage only as optional rendered text and still drops usage on timeout/error/no-window paths, so the structured availability/window contract requested here remains missing.

## What This Changes

Request stable structured `session_status` provider usage data with explicit unavailable and reset metadata instead of parsing optional usage text.

## Best Possible Solution

Add an additive `details.usage` payload for `session_status` that reports provider/source, availability or unavailable reason, normalized windows with percent/resetAt, and reset explanation only when upstream data supports it.

## Reproduction Assessment

Partially yes. Source inspection deterministically shows missing structured usage fields and silent omission on provider usage failure, while the intermittent Day-window jump still needs live OpenAI Codex OAuth monitoring to reproduce end to end.

## Solution Assessment

Yes. An additive `details.usage` contract is the narrowest maintainable solution because it avoids breaking existing rendered `statusText` consumers while giving cron automation a stable machine-readable path.
