## Summary

Keep this open. Current main still matches the prior ClawSweeper review: `session_status` exposes provider quota windows only through optional human-readable status text, while successful tool details do not include a structured `usage` payload, explicit availability state, source metadata, or reset explanation. Related #8635 remains broader token/cost API work, and open PR #45342 is scoped to Anthropic usage caching for #45332 rather than this cron-safe `session_status` contract.

## What This Changes

Request stable, structured `session_status` usage data for provider quota windows, including 5h/day percentages, reset times, source, unavailable state, and reset metadata instead of intermittently omitted text lines.

## Best Possible Solution

Add an additive structured provider-quota payload under `session_status.details.usage` while keeping the existing rendered status card. The contract should expose provider/source, availability or unavailable reason, plan when known, normalized windows such as 5h/day/week with percent and resetAt, and reset explanation only when upstream data supports it.
