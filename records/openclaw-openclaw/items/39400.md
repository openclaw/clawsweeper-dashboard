## Summary

Keep open. Current main still exposes provider usage to `session_status` as optional human-readable status text and suppresses unavailable, errored, or timed-out usage windows, so the stable structured availability/window metadata remains absent.

## What This Changes

The issue requests stable structured `session_status` usage-window fields plus explicit unavailable/reset metadata for cron-style automation.

## Best Possible Solution

Add an additive `session_status.details.usage` contract with provider/source, availability or unavailable reason, plan when known, normalized windows, resetAt values, and reset explanations only when upstream data supports them.

## Reproduction Assessment

Partially yes. Source inspection deterministically shows the missing structured usage payload and silent omission path; the intermittent Day-window jumps require live OpenAI Codex OAuth monitoring over time.

## Solution Assessment

Yes. An additive structured `details.usage` payload is the narrowest maintainable solution because it preserves existing status text while giving automation a stable availability/window contract.

## Repair Work Prompt

Implement this issue by adding an additive structured usage payload to `session_status` without changing the existing human-readable status card. Preserve `content` and `details.statusText`; add `details.usage` with provider/source, availability or unavailable reason, plan when known, normalized windows such as 5h/day/week with percent and resetAt, and resetReason only when upstream data supports it. Keep the boundary in `session_status` and provider-usage/status helpers; do not add a config option, change provider auth policy, or attempt the broader token-usage API from #8635. Add regression coverage for successful Codex windows, no auth/no windows, provider error, and timeout/unavailable behavior, then update usage docs and CHANGELOG for the user-facing structured tool contract.
