## Summary

Keep open: current main still explains the reported root-vs-canvas difference, but changing it would require maintainer auth/security judgment rather than conservative cleanup. The dashboard imports legacy URL query tokens at `/`, while canvas capability routes require bearer auth or scoped capability URLs and ignore the `token` query parameter.

## What This Changes

The issue asks for token-in-URL onboarding to work at `/__openclaw__/canvas/` over an SSH-tunneled gateway, matching the root dashboard behavior.

## Best Possible Solution

Keep canvas on bearer headers and scoped capability URLs unless maintainers explicitly approve a broader auth mode; if the distinction is intentional, clarify operator-facing startup or dashboard guidance so users do not treat the canvas mount as the dashboard entrypoint.

## Reproduction Assessment

Yes, at source level. A request to `/__openclaw__/canvas/?token=...` reaches plugin-node capability auth, which ignores `token` and requires bearer auth or an active scoped capability, while the root dashboard explicitly imports legacy `?token=`.

## Solution Assessment

Unclear. Making canvas accept `?token=` is not clearly the best fix because canvas serves untrusted HTML/JS; maintainers should choose between clearer guidance and a deliberately designed canvas auth flow.
