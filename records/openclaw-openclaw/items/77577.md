## Summary

Keep open: current main behaves as reported, but making canvas accept URL query tokens would broaden an auth-sensitive untrusted-content surface and needs maintainer/security judgment rather than conservative cleanup.

## What This Changes

The issue asks for the same token-in-URL onboarding path that works at the Control UI root to also work at `/__openclaw__/canvas/` over an SSH tunnel.

## Best Possible Solution

Keep canvas access on bearer headers and scoped `oc_cap` capability URLs unless maintainers deliberately design a safer onboarding flow; clarify dashboard-versus-canvas URL guidance if that is the actual operator problem.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `GET /__openclaw__/canvas/?token=...` reaches canvas auth, which reads bearer auth or a scoped canvas capability and ignores the `token` query parameter; existing tests cover the 401 path without bearer/capability.

## Solution Assessment

No. Treating `?token=` as canvas auth is not clearly the best fix because canvas is documented as arbitrary untrusted HTML/JS; a maintainer should choose between clearer operator guidance and a deliberately designed canvas auth change.
