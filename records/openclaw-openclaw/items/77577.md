## Summary

Close: current main fixes the operator-facing problem through the supported dashboard root flow and safe fragment-token fallback, while direct query-token auth on the canvas route remains intentionally unsupported.

## What This Changes

The issue reports that a gateway token in the URL works on the Control UI root path but not on the `/__openclaw__/canvas/` path over an SSH tunnel.

## Best Possible Solution

Keep the supported dashboard bootstrap on `/` with fragment-token/manual-auth guidance, and preserve canvas bearer/capability auth instead of adding shared-token query auth to the canvas route.

## Reproduction Assessment

No, not as a current-main bug: source and tests show the supported dashboard path is `/` with fragment-token fallback, while direct `/__openclaw__/canvas/?token=...` access is expected to be unauthorized without bearer auth or a node capability.

## Solution Assessment

Yes, the shipped solution is the best current shape because it fixes the operator recovery path without putting shared gateway tokens into the untrusted canvas URL surface.
