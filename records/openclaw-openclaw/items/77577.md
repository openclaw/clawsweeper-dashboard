## Summary

Keep open: current main explains the observed split, but closing would be premature because the remaining choice is auth-sensitive product/docs work. The Control UI still has legacy one-time query-token bootstrap, while the canvas host is gated by bearer auth or short-lived canvas capabilities.

## What This Changes

The issue asks for tokenized URL onboarding to behave consistently between the Control UI root path and `/__openclaw__/canvas/` over an SSH-tunneled Docker gateway.

## Best Possible Solution

Keep canvas auth on bearer headers and scoped `oc_cap` capability URLs unless maintainers explicitly approve a broader canvas auth mode; improve startup/log/docs guidance if operators are being sent to the canvas mount as a dashboard URL.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `GET /__openclaw__/canvas/?token=...` reaches canvas auth, which ignores `token` query params and requires bearer auth or an active scoped capability; existing tests cover the 401 behavior without bearer/capability.

## Solution Assessment

No. Making canvas accept `?token=` is not clearly the best fix because canvas is documented as untrusted arbitrary HTML/JS; the safer path is maintainer review of docs/log guidance or a deliberately scoped auth design.
