## Summary

Keep open: current main has a high-confidence source reproduction, but making canvas accept URL tokens would change an untrusted-content auth surface and needs maintainer/security direction.

## What This Changes

The issue asks for token-in-URL gateway auth to work for `/__openclaw__/canvas/` the same way it works for the Control UI root URL.

## Best Possible Solution

Keep direct canvas access on bearer headers and scoped canvas capabilities unless maintainers approve a deliberate new auth mode; clarify logs or docs if the canvas mount URL is being mistaken for the dashboard entry point.

## Reproduction Assessment

Yes. Source inspection shows `GET /__openclaw__/canvas/?token=...` enters canvas auth, which only accepts bearer auth or an active scoped capability; existing coverage asserts 401 without those credentials.

## Solution Assessment

No. Accepting `?token=` on canvas is not clearly the best fix because canvas is documented as arbitrary HTML/JS; the safer path is maintainer-approved auth design or clearer operator guidance.
