## Summary

Keep open: current `main` still explains the reported difference, but changing canvas to accept URL query tokens would cross an auth/security product boundary rather than a conservative cleanup close.

## What This Changes

The issue asks for the gateway shared token in the URL query string to work on `/__openclaw__/canvas/` the same way it bootstraps the Control UI at `/`.

## Best Possible Solution

Keep canvas on bearer headers and scoped capability URLs unless maintainers explicitly approve a broader auth mode; otherwise improve the dashboard-vs-canvas guidance that led the operator to the canvas mount.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `GET /__openclaw__/canvas/?token=...` reaches canvas auth, which ignores `token` query params and requires bearer auth or an active scoped canvas capability.

## Solution Assessment

No clear best fix yet. Making canvas accept `?token=` is not obviously safe because canvas is documented as untrusted arbitrary HTML/JS; a safer default is maintainer review for docs/log guidance or a deliberately designed auth flow.
