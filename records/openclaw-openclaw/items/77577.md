## Summary

Keep open: current main still explains the root-vs-canvas difference, but the remaining decision is auth/security policy rather than conservative cleanup. The dashboard intentionally imports legacy URL query tokens at `/`, while canvas capability routes require bearer auth or scoped capability URLs and ignore the `token` query parameter.

## What This Changes

The issue asks whether a gateway shared-secret token in the URL should authenticate `/__openclaw__/canvas/` the same way it can bootstrap the root dashboard.

## Best Possible Solution

Keep canvas on bearer headers and scoped capability URLs unless maintainers explicitly approve a broader auth mode; otherwise clarify operator-facing dashboard-vs-canvas guidance where the misleading URL is produced.

## Reproduction Assessment

Yes, at source level. A request to `/__openclaw__/canvas/?token=...` reaches plugin node capability auth, which ignores `token` and requires bearer auth or an active scoped capability, while `/` runs the dashboard token bootstrap.

## Solution Assessment

Unclear. Making canvas accept `?token=` is not clearly the best fix because canvas serves untrusted HTML/JS; maintainers should choose between clearer guidance and a deliberately reviewed canvas auth change.
