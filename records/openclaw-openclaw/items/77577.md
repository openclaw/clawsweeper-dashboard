## Summary

Keep open: current main still explains the reported mismatch, but changing it would broaden auth behavior on the untrusted canvas surface. The Control UI root imports legacy URL tokens, while canvas routes require bearer auth or a scoped canvas capability.

## What This Changes

Request that a gateway token supplied in the URL authenticate both the Control UI root and `/__openclaw__/canvas/` over an SSH tunnel.

## Best Possible Solution

Keep canvas on bearer headers and scoped capabilities unless maintainers approve a new auth design; clarify dashboard-vs-canvas URLs in logs/docs if this was operator confusion.

## Reproduction Assessment

Yes. Source inspection on current main shows `GET /__openclaw__/canvas/?token=...` reaches canvas auth, where `token` query params are ignored and the request needs a bearer header or valid scoped canvas capability.

## Solution Assessment

No, making canvas accept `?token=` is not clearly the best fix without maintainer/security approval. The safer direction is to preserve the scoped canvas auth model and improve operator guidance, or design a deliberate new auth flow.
