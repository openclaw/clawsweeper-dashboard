## Summary

Keep open: current `main` explains the observed difference, but the right fix is a maintainer/security decision rather than a conservative cleanup close. The Control UI SPA has a legacy one-time `?token=` bootstrap, while the canvas host is a separate untrusted-content surface gated by bearer auth or short-lived canvas capabilities.

## What This Changes

The issue reports that a gateway token in the query string loads the root Control UI over an SSH tunnel but fails at the canvas URL under `__openclaw__/canvas/`.

## Best Possible Solution

Keep canvas auth on bearer headers and scoped `oc_cap` URLs unless maintainers explicitly approve a broader auth mode; improve operator-facing logs/docs if the canvas mount message is being mistaken for the dashboard URL.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `GET /__openclaw__/canvas/?token=...` reaches canvas auth, which ignores `token` query params and requires bearer auth or an active scoped canvas capability. I did not run a live gateway because this review is read-only and the existing test covers the 401 behavior.

## Solution Assessment

No, making canvas accept `?token=` is not clearly the best fix because canvas is documented as untrusted arbitrary HTML/JS. A safer resolution is maintainer review to either clarify the dashboard-vs-canvas entry points or approve a deliberately designed canvas auth flow.
