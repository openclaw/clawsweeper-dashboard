## Summary

Keep open: current main still explains the observed difference, but deciding whether to change canvas auth or clarify operator-facing guidance is an auth/security product decision. The Control UI intentionally imports legacy `?token=` once, while canvas is guarded by bearer auth or scoped `oc_cap` capability URLs.

## What This Changes

The issue reports that a gateway token in the URL query string works for the root Control UI but not for `/__openclaw__/canvas/` over an SSH-forwarded Docker gateway.

## Best Possible Solution

Keep canvas on bearer headers and scoped capabilities unless maintainers explicitly approve a broader auth mode; clarify logs/docs if the canvas mount URL is being mistaken for the dashboard entry point.

## Reproduction Assessment

Yes, at source level: `GET /__openclaw__/canvas/?token=...` reaches canvas capability auth, which ignores the `token` query parameter and requires bearer auth or an active `oc_cap` capability. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear. Making canvas accept `?token=` is not the narrowest safe fix because canvas serves untrusted HTML/JS; maintainer review should choose between documentation/log clarity and a deliberately designed canvas auth flow.
