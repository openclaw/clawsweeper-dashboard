## Summary

Keep open: current main already has plugin-side subagent runtime APIs that cover part of the report, but it still lacks a single opt-in hard-routing contract that can consume or redirect the original turn without relying on model compliance or ad hoc hook combinations.

## What This Changes

The issue requests a gateway/plugin API or configuration mode that can force selected tasks into subagent sessions instead of relying on prompt-based delegation guidance.

## Best Possible Solution

Document the existing plugin subagent runtime as the partial current answer, then design one explicit opt-in hard-routing contract that preserves sessions_spawn policy and auditability while leaving prompt-only delegation unchanged by default.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an additive feature/API request. Source inspection gives a high-confidence state check: direct plugin subagent runs exist, but enforced initial-turn routing is not a current contract.

## Solution Assessment

Unclear until maintainers choose the API and security model. The best path is not more prompt injection; it is either documenting the existing plugin runtime workaround as sufficient or adding a first-class opt-in delegate/handled routing result.

## Risks / Open Questions

- A new hard-routing surface could let plugins or config trigger session orchestration without the same operator intent, tool-policy, sandbox, depth, child-limit, and audit boundaries that currently protect sessions_spawn.
- The existing api.runtime.subagent.run path covers direct plugin-created background runs, but using it together with block/claim hooks is not the same as a documented, user-facing dispatch contract for consuming the original turn.
- The reporter's subsession-router workaround is useful evidence of demand, but it does not settle whether the product should expose hook results, config rules, a gateway endpoint, or dispatch_mode semantics.
