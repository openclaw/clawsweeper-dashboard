## Label Justifications

- `P3`: This is a speculative multi-agent ergonomics enhancement rather than a regression or broken documented behavior.
- `impact:session-state`: The request would automatically move work between a main session and subagent sessions, affecting transcript ownership and routing context.
- `impact:security`: The proposed hard-routing API would control `sessions_spawn`, which current gateway policy treats as security-sensitive session orchestration.

## Summary

Keep open: current main still only offers prompt guidance, pass/block gates, plugin-owned background subagent runs, and post-spawn lifecycle hooks; none of those atomically force the initial user task into `sessions_spawn` before the main model turn. The reported `subsession-router` workaround is useful evidence of a real API gap, but the requested hard routing needs maintainer product and security design.

## What This Changes

Request a plugin or gateway-level hard-routing mechanism that can force selected tasks into subagent sessions instead of relying on prompt-only delegation guidance.

## Best Possible Solution

Design an explicit opt-in pre-model routing contract that preserves existing `sessions_spawn` allowlists, sandbox and depth limits, records auditable routing decisions, and leaves prompt-only delegation as the default compatibility path.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The capability gap is source-verifiable from current hook result types, delegation docs, config schemas, and `sessions_spawn` policy tests.

## Solution Assessment

Unclear until maintainers choose the product and security model. The current prompt-injection workaround is not a durable hard-routing solution, but unconditional forced routing also needs an explicit opt-in trust boundary.
