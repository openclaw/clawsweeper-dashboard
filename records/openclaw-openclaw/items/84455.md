## Label Justifications

- `P3`: This is a useful but speculative multi-agent ergonomics feature rather than a regression or broken documented behavior.
- `impact:session-state`: The request would automatically move work from a main session into child sessions, affecting context ownership and routing state.
- `impact:security`: A hard-routing API would give plugin or config logic authority over `sessions_spawn`, a security-sensitive session orchestration surface.

## Summary

Keep open: current main still offers prompt-only delegation guidance, pass/block gate hooks, and lifecycle hooks around already-created subagents; the reported `subsession-router` workaround highlights a real API gap for forced pre-model routing, but that gap needs product and security design before implementation.

## What This Changes

Requests a gateway/plugin API or config surface that can force selected turns into `sessions_spawn` subagent sessions instead of relying on prompt guidance.

## Best Possible Solution

Design an explicit opt-in pre-model routing contract that preserves existing `sessions_spawn` policy boundaries, audits routing decisions, and leaves prompt-only delegation unchanged by default.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an additive feature request. Source inspection confirms current main has prompt guidance, pass/block gates, and post-spawn lifecycle hooks, but no forced initial routing result.

## Solution Assessment

Unclear until maintainers choose the API and security model. The maintainable direction is an explicit opt-in routing contract, not prompt-injection workarounds or unconditional hard-routing defaults.
