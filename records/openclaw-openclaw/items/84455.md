## Label Justifications

- `P3`: This is a useful but speculative multi-agent/plugin API enhancement rather than a documented regression or outage.
- `impact:session-state`: The request is about automatically moving a user task between the main session and child subagent sessions.
- `impact:security`: The requested hard-routing API would expand who can trigger `sessions_spawn`, which must preserve tool policy, sandbox, and Gateway trust boundaries.

## Summary

Keep open: current main has prompt-only delegation and plugin-owned subagent runtime, but no pre-model hard-routing contract that can claim an ordinary user turn and re-dispatch it as a subagent while preserving `sessions_spawn` policy.

## What This Changes

The issue requests a hard routing mechanism that lets plugins or gateway configuration force selected tasks into subagent sessions instead of relying on prompt-injected guidance.

## Best Possible Solution

Design an explicit opt-in routing contract that runs before the main model call, preserves existing `sessions_spawn` allowlists, sandbox checks, depth limits, and auditability, and keeps current prompt-only delegation unchanged by default.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. The capability gap is source-verifiable from the current hook result types, prompt-only delegation docs, cron-scoped reply hook, and `sessions_spawn` policy tests/docs.

## Solution Assessment

Unclear until maintainers choose the API and trust model. The current prompt-injection workaround is not a durable hard-routing mechanism, but unconditional forced delegation would be the wrong shape unless it is explicit, opt-in, auditable, and policy-preserving.
