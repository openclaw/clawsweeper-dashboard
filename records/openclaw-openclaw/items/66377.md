## Summary

Keep open. Current main only partially addresses deleted-agent session fallout: it purges some core session-store entries and rejects stale deleted-agent session targets, but agent deletion still does not run the broader session/plugin cleanup lifecycle, prune cron jobs for the agent, or provide an agent-scoped reset path.

## What This Changes

The issue requests reliable cleanup or reset of all agent-owned state when an agent is deleted or recreated, including session files, plugin persistence, cron references, and clearer corruption repair UX.

## Best Possible Solution

Keep this open until OpenClaw has a coherent agent deletion or agent reset lifecycle: enumerate and clean affected sessions through the existing mutation cleanup path, remove or clearly report cron jobs for the agent, and expose a generic plugin/context-engine cleanup hook rather than special-casing `lcm.db`. Add regression coverage for CLI deletion, Gateway deletion, cron references, plugin cleanup, and stale corrupted-session recovery messaging.

## Reproduction Assessment

Yes for the remaining cleanup gap: current source shows agent deletion does not prune cron jobs or invoke the session/plugin cleanup lifecycle. The original corruption trigger is still unclear because the report says the trigger is unknown and no corrupt fixture/log was provided.

## Solution Assessment

No, current main is not the best complete solution because it only purges/rejects some core session-store state. The maintainable fix should extend a generic agent deletion/reset lifecycle across cron and plugin-owned state instead of adding ad hoc database surgery.
