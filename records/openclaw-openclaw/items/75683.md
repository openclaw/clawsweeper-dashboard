## Summary

Keep open. Current main still returns `image_generate` outputs as OpenClaw-managed media-store paths, while sandbox filesystem follow-up tools only resolve workspace, agent-workspace, and configured bind mounts; the existing managed-media allowance is for media-source resolution, not exec/read-style workspace access.

## What This Changes

The issue requests making generated image outputs available to sandboxed follow-up workflows through workspace materialization, dual paths, export tooling, or direct media-reference consumption.

## Best Possible Solution

Keep the existing `MEDIA:` delivery path and add a bounded generated-output export or workspace materialization path that sandboxed follow-up tools can read, with regressions proving unrelated host paths and unmanaged media remain blocked.

## Reproduction Assessment

Yes. The code-level reproduction is to run `image_generate`, observe the returned `.openclaw/media/tool-image-generation` path, then resolve that host path through the sandbox fs bridge or an exec-style workflow where it is outside the workspace/agent/bind mounts.

## Solution Assessment

Yes. The requested direction is the right fix if implemented as a narrow generated-output export or workspace copy, not by loosening global sandbox filesystem policy.
