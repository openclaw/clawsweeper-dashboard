## Summary

Keep open. Current main still returns `image_generate` outputs only as OpenClaw-managed media-store paths for reply delivery, while sandboxed filesystem access is limited to the sandbox workspace, optional agent workspace mount, and configured bind mounts. The report has a concrete workflow and the code still matches the observed failure mode.

## What This Changes

The issue requests that `image_generate` outputs be materialized in a sandbox/workspace-readable location, exposed through an export/copy handle, or accepted directly by follow-up tools.

## Best Possible Solution

Preserve the existing `MEDIA:` delivery path for chat attachments, and add a bounded generated-artifact path or explicit export handle that sandboxed agents can read for follow-up processing. Cover both the new accessible output and the unchanged sandbox boundary with tests, then document the behavior.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: run `image_generate`, observe that the result paths are under the managed media store, then pass that host path to sandboxed read/exec-style tooling where the mount resolver rejects paths outside the sandbox mounts.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented as a generated-output copy/export path, not by loosening the global sandbox path policy or granting broad access to managed media.
