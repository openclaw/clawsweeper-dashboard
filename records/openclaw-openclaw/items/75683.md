## Summary

Keep open. Current main still saves `image_generate` outputs as managed media-store files and returns those host paths for delivery, while sandbox filesystem follow-up tools only resolve workspace, agent-workspace, and configured bind mounts; the newer media-source helper carve-out does not make those files visible to exec-style workflows.

## What This Changes

Make `image_generate` outputs usable by sandboxed follow-up workflows through a workspace-readable file, dual path, export/copy handle, or directly consumable `MEDIA:` artifact reference.

## Best Possible Solution

Keep the existing `MEDIA:` delivery path and add a bounded generated-output export or workspace copy that follow-up sandbox tools can read, with regression tests proving arbitrary host paths and unrelated managed media remain blocked.

## Reproduction Assessment

Yes. The code-level reproduction is to run `image_generate`, observe the returned `.openclaw/media/tool-image-generation` path, then pass that host path to a sandbox fs bridge or exec-style tool where it is outside the workspace/agent/bind mounts.

## Solution Assessment

Yes. The requested direction is the right fix if implemented as a narrow generated-output export or workspace materialization rather than by loosening global sandbox path policy.

## Repair Work Prompt

Fix openclaw/openclaw#75683. `image_generate` currently saves generated images under the managed media store and returns `MEDIA:`/`details.media.mediaUrls` paths, but sandboxed follow-up tools such as exec-style file operations cannot read those host paths because the sandbox fs bridge only maps workspace, agent workspace, and configured binds. Add a narrow generated-output export path or workspace copy for `image_generate` outputs while preserving existing `MEDIA:` delivery. Do not mount or expose the full managed media store, do not loosen arbitrary host-path policy, and keep unrelated managed media outside the sandbox. Update focused tests for image generation output details and sandbox path boundaries, document the new follow-up workflow in image-generation docs, and add a user-facing changelog fix entry.
