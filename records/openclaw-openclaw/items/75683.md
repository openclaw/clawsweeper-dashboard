## Summary

Keep open. Current main still returns `image_generate` outputs only as managed delivery media paths, while sandbox file access resolves only mounted workspace, agent workspace, and configured bind paths, so the reported follow-up workflow remains blocked.

## What This Changes

The issue requests that `image_generate` outputs be usable by sandboxed follow-up tools through a workspace-readable artifact, export handle, or direct media-reference input support.

## Best Possible Solution

Keep managed `MEDIA:` delivery and add a bounded sandbox-readable generated artifact or export contract for `image_generate` outputs, with regression tests proving unrelated managed media and host paths remain denied.

## Reproduction Assessment

Yes, at source level. `image_generate` returns managed `.openclaw/media/tool-image-generation` paths, while sandbox bridge file resolution only accepts workspace, agent workspace, and bind mounts, so the managed path is not reachable by read/stat/exec-style file access.

## Solution Assessment

Yes. The maintainable direction is a narrow generated-output materialization/export path; loosening the global sandbox filesystem resolver for all managed media would be the riskier alternative.

## Repair Work Prompt

Fix the generated-media follow-up workflow for https://github.com/openclaw/openclaw/issues/75683. Current `image_generate` saves outputs only under managed `media/tool-image-generation` and returns those paths, while sandboxed follow-up tools can only access workspace/agent/bind-mounted paths. Implement a bounded workspace-readable generated artifact or first-class export contract for sandboxed `image_generate` outputs while preserving managed `MEDIA:` delivery. Do not globally mount or allow all managed media or arbitrary host state paths through the sandbox fs resolver. Add/update regression coverage for image tool output details/text, sandbox media/path behavior, and fs bridge denial of unrelated outside paths. Update `docs/tools/image-generation.md` if the user-visible output contract changes, and add a single-line `CHANGELOG.md` entry if this ships as user-facing behavior.
