## Summary

Keep open. Current main still makes `image_generate` outputs available as managed media delivery paths, but it does not materialize a workspace-readable generated artifact or export handle for follow-up sandbox workflows such as exec-style local processing or third-party upload tools.

## What This Changes

The issue requests that generated `image_generate` media be accessible from the sandbox workspace, returned with a workspace path, exportable through a first-class handle, or consumable directly by follow-up tools.

## Best Possible Solution

Keep the existing `MEDIA:` delivery path for channel rendering, and add a bounded generated-artifact copy/export path that sandboxed follow-up tools can read, with tests proving unrelated managed media and arbitrary host paths remain outside the sandbox.

## Reproduction Assessment

Yes. The code-level reproduction is to run `image_generate`, observe the returned `.openclaw/media/tool-image-generation` path, then pass that host path to sandbox exec or a workspace-based tool where it is outside the configured mounts.

## Solution Assessment

Yes. A narrow generated-output export or workspace copy is the maintainable direction; loosening global sandbox path policy would be a less safe fix.

## Repair Work Prompt

Fix openclaw/openclaw#75683. `image_generate` currently writes generated images only to `.openclaw/media/tool-image-generation/...` and returns that managed `MEDIA:` path, which channel delivery can render but sandboxed follow-up tools cannot read from `/workspace`. Implement a narrow generated-output export path or workspace copy so sandboxed follow-up workflows can inspect/process/upload generated images. Keep the existing `MEDIA:` delivery path for channel rendering. Do not mount or expose the entire managed media store, do not loosen arbitrary host-path sandbox guards, and preserve existing inbound/outbound media security behavior. Likely update `src/agents/tools/image-generate-tool.ts`, sandbox/media path helpers if needed, focused tests, `docs/tools/image-generation.md`, and `CHANGELOG.md` if the behavior is user-facing. Add regression tests proving generated outputs become workspace-readable while unrelated managed media and arbitrary host paths remain blocked.
