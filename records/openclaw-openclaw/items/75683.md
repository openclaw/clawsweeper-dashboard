## Summary

Keep open. Current main still returns `image_generate` outputs as managed `.openclaw/media/tool-image-generation` host paths, while normal sandbox filesystem tools resolve only workspace, agent-workspace, and configured bind mounts. Media-aware loaders have a narrow managed-media allowance, but that does not make generated files workspace-addressable for `exec`, `read`, Basecamp-style attachment tools, or local post-processing.

## What This Changes

The issue asks for `image_generate` outputs to be available to sandboxed follow-up workflows through a workspace-readable copy, export handle, or direct artifact consumption path.

## Best Possible Solution

Keep the existing `MEDIA:` delivery path for channel rendering, and add a bounded generated-artifact path or export handle that sandboxed follow-up tools can read. The implementation should prove unrelated managed media and arbitrary host paths remain outside the sandbox.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to run or mock `image_generate`, observe the returned `.openclaw/media/tool-image-generation` path, then resolve that path through the sandbox filesystem bridge, where it is outside the mounted workspace/agent/bind roots.

## Solution Assessment

Yes. The requested direction is the right fix if implemented as a narrow workspace copy or export handle, rather than by loosening global sandbox filesystem policy.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75683. `image_generate` currently saves outputs under the managed media store and returns only those host paths, so sandboxed follow-up tools cannot read the generated file from `/workspace`. Add a narrow workspace-readable artifact path or first-class export/copy handle for generated image outputs while preserving the existing `MEDIA:` delivery path for channel rendering. Prefer materializing only the images produced by the current tool call into the active sandbox/workspace context when available; do not mount or expose the entire managed media store, do not relax arbitrary host-path checks, and respect read/write sandbox policy. Update structured details with a clear field for the follow-up path/handle, adjust tests for `image_generate` and sandbox path boundaries, document the workflow in `docs/tools/image-generation.md`, and add a single-line user-facing `CHANGELOG.md` fix entry.
