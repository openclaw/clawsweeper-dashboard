## Summary

Keep open. Current main still saves `image_generate` outputs to the managed media store and returns those host media paths for delivery, while the sandbox filesystem bridge only exposes workspace, optional agent-workspace, and configured bind mounts to sandboxed tools. That leaves the reported follow-up workflow gap valid.

## What This Changes

The issue requests that `image_generate` outputs become usable by sandboxed follow-up workflows through a workspace-readable generated-media path, dual returned paths, an export/copy handle, or first-class consumption of prior media artifacts.

## Best Possible Solution

Keep the existing `MEDIA:` delivery behavior, but add a narrow generated-artifact path that sandboxed agents can read for follow-up work. The safest direction is to materialize the generated image bytes into a deterministic workspace-visible generated-media directory or return an explicit export path/handle, with tests preserving the sandbox boundary for unrelated media and host paths.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: `image_generate` returns managed media-store paths, and the sandbox filesystem bridge rejects host paths outside its mounts, which matches the reported read/exec failure path.

## Solution Assessment

Yes, the requested direction is the right product fix, but the implementation should be narrow: preserve the managed delivery path and add a workspace-readable generated artifact or export handle instead of loosening sandbox path policy globally.
