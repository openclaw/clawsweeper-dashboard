## Summary

Keep open. Current main still saves `image_generate` results only as managed media-store files while generic sandbox file tools resolve only workspace, agent-workspace, and configured bind mounts, so the reported follow-up workflow remains blocked.

## What This Changes

The issue requests workspace-readable or exportable `image_generate` outputs so sandboxed follow-up tools can inspect, modify, upload, or archive generated images.

## Best Possible Solution

Keep existing `MEDIA:` delivery and add a bounded workspace materialization or export handle for generated outputs that follow-up tools can read without weakening sandbox path policy.

## Reproduction Assessment

Yes. The code-level reproduction is to run `image_generate`, take the returned managed `MEDIA:` path under `.openclaw/media/tool-image-generation`, and pass it to a sandbox bridge or exec-style file operation, where it misses all allowed mounts and is rejected as outside the sandbox root.

## Solution Assessment

Yes. The requested direction is appropriate if implemented as a narrow copy/export path or first-class handle; the safer alternative is not to loosen the generic sandbox resolver for all managed media paths.

## Repair Work Prompt

Fix the image generation follow-up workflow reported in https://github.com/openclaw/openclaw/issues/75683. Current main saves generated images with `saveMediaBuffer(..., "tool-image-generation", ...)` and returns only managed media-store paths, while sandbox file tools only resolve workspace, agent-workspace, and bind mounts. Add a bounded generated-output materialization or export path that sandboxed follow-up tools can read, preserving the existing `MEDIA:` managed path for delivery. Prefer a workspace-readable generated media location or explicit exported path in the tool result; do not add broad `~/.openclaw/media` access to `resolveSandboxFsPathWithMounts` or weaken host-path denial. Add focused tests for the image_generate result, sandbox bridge/path behavior, and the denial of unrelated managed/host paths. Add docs and a user-facing changelog entry if behavior changes.
