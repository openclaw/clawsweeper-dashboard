## Summary

Keep open. Current main still saves image_generate outputs only as managed media-store files and returns those paths, while generic sandbox filesystem follow-up tools resolve only workspace, agent-workspace, and configured bind mounts.

## What This Changes

The issue requests a workspace-readable or first-class export path for images generated by image_generate so later sandboxed workflow steps can inspect, modify, upload, or archive them.

## Best Possible Solution

Keep the existing MEDIA delivery path and add a bounded generated-output materialization/export handle that produces a sandbox-readable workspace path for image_generate results.

## Reproduction Assessment

Yes. The code-level reproduction is to generate an image, observe that the tool returns a managed media-store path, then pass that host path to sandbox bridge or exec-style file access where it misses all configured mounts and is rejected as outside the sandbox root.

## Solution Assessment

Yes. The requested direction is appropriate if implemented narrowly as a generated-output export or workspace copy; the safer alternative to avoid is loosening the global sandbox fs resolver for all managed media.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75683. image_generate currently saves generated images under managed media storage and returns only MEDIA/details paths that sandboxed follow-up tools cannot read. Add a bounded workspace-readable export/materialization path or first-class handle for image_generate outputs while preserving the existing MEDIA delivery path. Do not loosen resolveSandboxFsPathWithMounts to allow arbitrary ~/.openclaw/media, managed media, or host paths. Update focused tests to prove generated outputs can be read by sandbox follow-up paths and that unrelated managed media/host paths remain denied. Update docs if the user-visible tool contract changes, and add a CHANGELOG.md fix entry because this is user-facing.
