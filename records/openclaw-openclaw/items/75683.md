## Summary

Keep open. Current main still returns `image_generate` outputs as managed delivery media paths while sandbox file tools resolve only workspace, agent-workspace, and configured bind mounts, so the reported follow-up workflow remains blocked.

## What This Changes

Make generated `image_generate` outputs available to sandboxed follow-up tools through a workspace-readable artifact, export handle, or first-class media reference path.

## Best Possible Solution

Keep the managed `MEDIA:` delivery path and add a bounded workspace-visible generated artifact or export handle, with regression tests proving unrelated managed media and host paths remain denied.

## Reproduction Assessment

Yes. The code-level reproduction is to run `image_generate` in a sandboxed session, then pass the returned managed media path to a sandbox fs bridge or exec-style file operation; the resolver has no media mount and rejects it as outside the sandbox root.

## Solution Assessment

Yes. A narrow generated-output materialization/export path is the maintainable fix; loosening the global sandbox fs resolver for all managed media would be the riskier alternative.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75683. `image_generate` currently returns only managed `MEDIA:` paths under `tool-image-generation`, which sandboxed follow-up tools cannot read. Add a narrow generated-output follow-up path: either materialize generated image buffers into a workspace-relative location such as `media/generated/...` when a sandbox bridge is available, return both managed delivery paths and workspace-readable paths, or add an equivalent first-class export handle. Keep the existing managed `MEDIA:` delivery path. Do not add a blanket fs resolver allowlist for all managed media or host state paths. Update focused tests for `src/agents/tools/image-generate-tool.ts`, sandbox fs/path behavior, and docs/changelog if the user-facing result contract changes.
