## Summary

Keep open. Current main still returns generated image outputs as managed media-store paths for delivery, while sandboxed follow-up tools can only see the sandbox workspace, optional agent workspace mount, and configured bind mounts; the reported workflow gap remains valid.

## What This Changes

The issue requests that `image_generate` outputs be usable by sandboxed follow-up workflows through a workspace-readable generated-media path, export/copy handle, or equivalent media reference.

## Best Possible Solution

Keep the existing `MEDIA:` delivery behavior, but add a bounded generated-artifact path that sandboxed agents can read for follow-up work. The safest end state is a workspace-visible generated-media copy or explicit export path/handle, with tests proving unrelated managed media and arbitrary host paths remain outside the sandbox.

## Reproduction Assessment

Yes. The high-confidence code-level reproduction is to run `image_generate`, observe that it returns managed media-store paths, then pass that host path to sandboxed read/exec-style tooling where the sandbox bridge rejects paths outside its mounts.

## Solution Assessment

Yes. The requested direction is the right maintainable fix, provided it preserves managed delivery paths and adds a narrow workspace-readable artifact or export handle instead of loosening sandbox path policy globally.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75683. Current `image_generate` writes generated image buffers to the managed `tool-image-generation` media store and returns those paths for delivery, but sandboxed follow-up tools cannot read those host media paths. Implement a narrow fix so generated image outputs are also accessible from the current sandbox/workspace for follow-up workflows. Preserve existing `MEDIA:` delivery behavior and `details.media.mediaUrls` for channel delivery. Add a workspace-readable generated artifact path in tool output/details, or an explicit export/copy handle, using the sandbox/workspace context already passed into `createImageGenerateTool`. Do not mount or expose the full managed media store, do not allow arbitrary `MEDIA:` refs into exec/read paths, do not change provider APIs, and do not broaden video/music generation unless a small shared helper is clearly needed and tested. Add focused tests for sandboxed `image_generate` output accessibility and for preserving current delivery paths plus sandbox rejection of unrelated host paths.
