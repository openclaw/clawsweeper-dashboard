## Summary

Keep open. Current main still saves `image_generate` outputs only as managed media-store paths and returns those paths in the tool result, while sandbox filesystem resolution only exposes mounted workspace, agent workspace, and configured bind paths to follow-up sandbox tools.

## What This Changes

The issue requests that `image_generate` outputs be exposed through a sandbox/workspace-readable path, export handle, or equivalent artifact reference for follow-up agent workflows.

## Best Possible Solution

Keep the existing `MEDIA:` delivery path for channel rendering, and add a bounded generated-artifact path or export handle that sandboxed follow-up tools can read, with tests proving unrelated managed media and arbitrary host paths remain outside the sandbox.

## Reproduction Assessment

Yes. The code-level reproduction is to run `image_generate`, observe the returned `.openclaw/media/tool-image-generation` path, then pass that host path to sandbox bridge or exec-style tooling where it is outside the configured mounts.

## Solution Assessment

Yes. The requested direction is the right fix if implemented as a narrow generated-output export or workspace copy rather than by loosening global sandbox path policy.

## Repair Work Prompt

Fix openclaw/openclaw#75683. Current `image_generate` saves outputs only in `.openclaw/media/tool-image-generation` and returns those managed media paths, which remain inaccessible to sandboxed follow-up tools such as exec/plugin attach workflows. Preserve the existing `MEDIA:` delivery behavior, but add a bounded generated-artifact path or explicit export handle that is readable from the sandbox/workspace for generated images.
