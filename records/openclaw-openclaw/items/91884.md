## Summary

Current documentation defines opt-in Gateway startup as a one-shot QMD refresh that avoids the long-lived watcher, with `onBoot` gating that refresh. This PR instead retains the full manager for interval/embed maintenance, creating a compatibility-changing lifecycle redesign; it also lacks real Gateway proof.

## What This Changes

The PR retains full QMD managers at Gateway startup for configured interval/embed maintenance when boot sync is disabled, fixes collection-root-relative watcher ignores, and adds focused tests.

## Best Possible Solution

Preserve the documented one-shot Gateway startup path and fix background refresh activation at the established long-lived manager ownership boundary, or obtain an explicit product decision with updated config/docs and upgrade proof before changing startup semantics.

## Reproduction Assessment

Yes at source level: current main closes the transient startup manager after boot sync, so it cannot keep interval/embed maintenance armed. The exact user-visible symptom has not been demonstrated in a real Gateway run.

## Solution Assessment

No. Retaining the full manager during Gateway startup contradicts the documented one-shot ownership boundary; the repair should preserve that contract or be handled as an explicitly approved product and upgrade change.
