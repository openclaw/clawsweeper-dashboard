## Summary

Keep this PR open, but the current patch should not merge as-is because deleting the shared repo-root cache parent can race with another isolated retry between `mkdirSync` and `mkdtempSync`. Current `main` still has the reported residue behavior, so the contribution is directionally useful but needs a small concurrency-safe revision.

## What This Changes

The PR adds best-effort removal of `.openclaw-config-doc-cache` after isolated config-surface retry cleanup and adds a regression assertion that the repo-root cache directory is absent after a successful retry.

## Best Possible Solution

Keep the cleanup narrowly scoped, but make it concurrency-safe: preserve repo-root module resolution for the isolated retry while avoiding deletion of a shared parent that another retry may be about to use, then keep the regression coverage and skip a changelog entry as internal tooling hygiene.

## Reproduction Assessment

Yes. Current `main` can be reproduced with the existing isolated-package retry path in `src/config/load-channel-config-surface.test.ts`, then checking that `repoRoot/.openclaw-config-doc-cache` remains; the PR finding is reproducible by interleaving another process between shared-parent `mkdirSync` and `mkdtempSync`.

## Solution Assessment

No. The requested direction is right, but the current PR’s `rmdirSync(tempParent)` is not the safest implementation because it can remove a shared directory during another retry; a per-run repo-root temp parent or a robust recreate-and-retry around `mkdtempSync` would be safer.

## Review Findings

- **[P2] Avoid deleting the shared temp parent during another retry:** `scripts/load-channel-config-surface.ts:191`
  - body: The new cleanup removes the fixed `.openclaw-config-doc-cache` parent after deleting its child. Use a per-run repo-root temp directory or retry `mkdtempSync` after recreating the parent.
