## Summary

Keep open: current main and v2026.6.10 have the lower Gateway `plugin.approval.*` RPC and hook-owned `requireApproval` path, but they still do not expose the requested standalone Plugin SDK `api.approval.request()` method. The remaining work is a public plugin API and security-boundary decision, not a cleanup close.

## What This Changes

Requests a public Plugin SDK `api.approval.request()` method so plugins can ask for user approval outside `before_tool_call`.

## Best Possible Solution

Keep this open for maintainer API/security design, then implement a narrow additive SDK facade over the existing Gateway plugin approval flow with explicit context binding, fail-closed behavior, docs, baselines, and focused tests.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.10 have Gateway `plugin.approval.request` plus hook `requireApproval`, but `OpenClawPluginApi`, API construction, and public docs do not expose a standalone `api.approval.request()` method.

## Solution Assessment

Unclear pending maintainer design. The existing Gateway RPC is the right lower layer, but the public SDK facade needs an approved shape for context binding, cancellation, timeout, no-route, and security semantics.
