## Label Justifications

- `P2`: This is a useful public Plugin SDK feature with meaningful security design implications, but it is not an urgent regression in an existing workflow.
- `impact:security`: The requested API would let plugins initiate human approval prompts, which is an authorization and safety boundary.
- `impact:session-state`: The requested standalone approval flow would pause and resume agent workflows outside tool-call hooks, affecting run/session state handling.

## Summary

Keep open: current main and v2026.6.10 already have the lower Gateway plugin approval RPC and hook-owned `requireApproval` path, but not the requested standalone Plugin SDK `api.approval.request()` method. This remains a plausible plugin API extension, but it needs maintainer and security design before implementation.

## What This Changes

Request to add a standalone `api.approval.request()` Plugin SDK method so plugins can pause workflows for user approval outside `before_tool_call`.

## Best Possible Solution

Define a narrow additive SDK facade over the existing Gateway plugin approval flow, with explicit context binding, cancellation/no-route behavior, docs, API baseline updates, and focused Gateway/SDK/hook tests.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.10 have Gateway `plugin.approval.request` plus hook `requireApproval`, but `OpenClawPluginApi`, API construction, and public docs do not expose a standalone `api.approval.request()` method.

## Solution Assessment

Unclear pending maintainer design. The existing Gateway RPC is the right lower layer, but the public SDK facade needs an approved API shape for context binding, cancellation, timeout, no-route, and security semantics.
