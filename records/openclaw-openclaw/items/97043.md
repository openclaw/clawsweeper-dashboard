## Label Justifications

- `P2`: This is a useful plugin API feature with security-sensitive design implications, but it is not an urgent regression in an existing workflow.
- `impact:security`: The requested API controls human approval before plugin-owned actions, which is an authorization and safety boundary.
- `impact:session-state`: The requested standalone approval flow would pause and resume agent workflows outside tool-call hooks, affecting run/session state handling.

## Summary

Keep open: current main and the latest checked release already have Gateway-level plugin approval RPCs and the `before_tool_call.requireApproval` path, but they do not expose the requested standalone `api.approval.request()` method on `OpenClawPluginApi`. The request is a plausible plugin API extension, but it needs maintainer/security design because it changes a public approval boundary and run pause semantics.

## What This Changes

The issue requests an additive Plugin SDK `api.approval.request()` method for plugin-owned approval prompts outside `before_tool_call` hooks.

## Best Possible Solution

Design a narrow additive `api.approval.request` facade over the existing Gateway approval flow with explicit run/session/turn-source binding, cancellation and timeout semantics, docs, SDK baseline updates, and focused Gateway/SDK/hook tests.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.10 have Gateway `plugin.approval.request` plus hook `requireApproval`, but `OpenClawPluginApi` has no `approval` namespace or standalone request method.

## Solution Assessment

Unclear pending maintainer design: the existing Gateway RPC is the right lower layer, but the public SDK facade needs an approved API shape for context binding, cancellation, timeout, and security semantics.
