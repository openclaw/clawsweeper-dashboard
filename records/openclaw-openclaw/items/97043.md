## Summary

Keep open: current main and v2026.6.10 have the lower Gateway `plugin.approval.request` RPC and hook-owned `before_tool_call.requireApproval`, but they still do not expose the requested standalone Plugin SDK `api.approval.request()` facade. This is a plausible plugin API extension, but it needs maintainer API/security design before implementation.

## What This Changes

The issue requests an additive `api.approval.request()` method on `OpenClawPluginApi` so plugins can request approval outside `before_tool_call` tool execution.

## Best Possible Solution

Define a narrow additive SDK facade over the existing Gateway plugin approval flow, with explicit context binding, cancellation/no-route behavior, docs, API baseline updates, and focused Gateway/SDK/hook tests.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.10 have Gateway `plugin.approval.request` plus hook `requireApproval`, but `OpenClawPluginApi`, API construction, facades, and public SDK docs do not expose a standalone `api.approval.request()` method.

## Solution Assessment

Unclear pending maintainer design. The existing Gateway RPC is the right lower layer, but the public SDK facade needs an approved API shape for context binding, cancellation, timeout, no-route, and security semantics.
