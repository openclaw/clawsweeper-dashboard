## Summary

Keep open: current main and v2026.6.10 have the lower Gateway plugin approval RPC and hook-owned `requireApproval` path, but not the requested standalone Plugin SDK `api.approval.request()` facade. Adjacent approval/verifier issues overlap but do not replace this API request, and the public security boundary needs maintainer design.

## What This Changes

The issue requests a new Plugin SDK `api.approval.request()` method so plugins can request user approval outside `before_tool_call` hooks.

## Best Possible Solution

Define a narrow additive SDK facade over the existing Gateway plugin approval flow only after maintainers approve context binding, cancellation, timeout, no-route, `allow-always`, and security semantics, then update docs, API baselines, and focused Gateway/SDK tests together.

## Reproduction Assessment

Yes, at the source-contract level: current main has Gateway `plugin.approval.request` plus hook `requireApproval`, but `OpenClawPluginApi` and API construction do not expose `api.approval.request()`.

## Solution Assessment

Unclear pending maintainer design: the lower Gateway RPC is the right reusable infrastructure, but the public SDK facade needs an approved shape for context binding, cancellation, timeout, no-route, and security semantics.
