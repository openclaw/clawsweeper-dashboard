## Summary

Keep open: the feature is useful, but the latest patch still routes arbitrary-file validation through the normal config snapshot reader, which can perform recovery and observation writes. That contradicts the advertised side-effect-free validation behavior and should be fixed before merge.

## What This Changes

The PR adds `openclaw config validate --file <path>`, threads a custom config path into snapshot reads, adds CLI tests, and updates config docs plus the changelog.

## Best Possible Solution

Add an explicit side-effect-free config snapshot mode for `config validate --file` that suppresses shipped-plugin install persistence, suspicious-read recovery, config-health observation, audit writes, and `.clobbered` artifacts while preserving current recovery behavior for active config load/read paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to validate a prepared config file whose contents look suspicious and that has an adjacent valid `.bak`; the current PR path reaches `maybeRecoverSuspiciousConfigRead`, which can copy that backup over the file before validation returns.

## Solution Assessment

No. Passing `configPath` through the existing snapshot reader is too broad for arbitrary-file validation; the safer fix is a no-side-effects snapshot/read option used only by the `--file` validation path.
