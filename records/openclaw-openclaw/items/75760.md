## Summary

Keep open: the feature is useful and current main does not already provide it, but the PR still has a concrete correctness blocker. The advertised arbitrary-file validation path is not side-effect free because it routes custom files through the normal config snapshot reader, which can run recovery, observation, audit, health-state, and backup-restore writes.

## What This Changes

The PR adds `openclaw config validate --file <path>`, threads a custom config path through config snapshot IO, and updates CLI tests, config docs, and the changelog.

## Best Possible Solution

Add an explicit side-effect-free snapshot/read mode for arbitrary-file validation that parses, resolves, and validates the selected file without suspicious-read recovery, config-health observation, audit writes, `.clobbered.*` artifacts, or plugin-install persistence, while preserving the existing default config read behavior for startup, reload, and active-config validation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a custom config file with suspicious content plus an adjacent valid `.bak` or matching health baseline; the current PR path can invoke read recovery and overwrite or record artifacts before reporting validation.

## Solution Assessment

No. Passing a custom path through the existing full snapshot reader is too broad for the documented arbitrary-file validation behavior; a narrower no-side-effects read mode is the safer maintainable fix.
