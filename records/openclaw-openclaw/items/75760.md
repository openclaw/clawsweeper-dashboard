## Summary

Keep open: the PR adds the intended CLI option, tests, docs, and changelog follow-through, but the current implementation can make a supposedly read-only arbitrary-file validation persist legacy plugin-install migration side effects for that file/state. The changelog update also inserted a duplicate `### Changes` heading inside the active release section.

## What This Changes

The PR adds `openclaw config validate --file <path>`, threads the path into `readConfigFileSnapshot`, adds two config CLI tests, and documents the new flag with a changelog entry.

## Best Possible Solution

Keep the CLI option, but make `--file` validation non-mutating for arbitrary paths before merge, then place the changelog bullet under the existing active-release `### Changes` section and keep the docs examples.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to validate a custom config containing legacy `plugins.installs`: the PR routes that path through the default snapshot reader, whose current migration path can persist plugin install records and rewrite the config file.

## Solution Assessment

No, not yet. Passing a `configPath` through the existing IO seam is the right shape, but arbitrary-file validation needs a no-persist snapshot mode so the command remains read-only for prepared files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep custom-file validation read-only:** `src/config/io.ts:2376`
  - body: `--file` is documented and described as a read-only validation path for arbitrary prepared configs, but this call uses the default snapshot reader. That reader persists shipped `plugins.installs` migrations by default and can write the plugin index plus replace the config file, so `openclaw config validate --file /tmp/staged.json` can mutate the staged file/state instead of only validating it. Add a no-persist snapshot option for this path or otherwise use a read-only reader when `file` is provided.
  - confidence: 0.88
- **[P3] Use the existing changelog Changes section:** `CHANGELOG.md:25`
  - body: This inserts a second `### Changes` heading inside the active `2026.4.30` release block instead of adding the bullet to the existing section. Remove the extra heading/blank line so the new CLI/config entry stays under the first `### Changes` heading before `### Fixes`.
  - confidence: 0.93
