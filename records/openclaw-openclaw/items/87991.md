## Summary

Keep open: current main still routes expected generated-media completions through the requester-agent/message-tool path before direct fallback, while this PR moves image/music/video completion delivery to the direct idempotent sender first and has focused regression coverage plus live Discord proof. I found no blocking code findings; the remaining maintainer decision is whether to accept the intentional terminal direct-send failure tradeoff across generated-media channels.

## What This Changes

The PR changes generated image/music/video completion delivery to try direct idempotent media sends before requester-agent handoff, makes attempted media-send failures terminal, and updates delivery/dispatch tests.

## Best Possible Solution

Land direct-first generated-media delivery with terminal attempted-send failures if maintainers accept one-send/no-duplicate semantics; keep any future recovery in direct sender idempotency or retry rather than requester-agent handoff.

## Reproduction Assessment

Yes. The linked issue and PR discussion provide live Discord reproduction on 2026.5.28-beta.1, and current main source still shows expected generated media can go through requester-agent/message-tool handoff before direct fallback.

## Solution Assessment

Yes, with maintainer acceptance of the tradeoff. Direct-first idempotent media delivery is the narrowest fix for the duplicate-send loop, and terminal attempted-send failures prevent partial platform sends from duplicating through fallback.
