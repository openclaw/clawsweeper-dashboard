## Summary

Keep open: the PR moves generated-media completions in the right direction, but it still falls back to requester-agent/message-tool handoff after an attempted direct media send fails, preserving a duplicate-send path when the channel send outcome is ambiguous.

## What This Changes

The PR changes generated image, music, and video completion delivery to try direct media delivery before requester-agent/message-tool handoff and updates completion-delivery tests for Discord, Slack, Telegram topics, and legacy routes.

## Best Possible Solution

Keep the direct-first generated-media delivery design, but make an attempted direct media send terminal or artifact-idempotent so requester-agent handoff cannot resend the same media after an ambiguous failure.

## Reproduction Assessment

Yes. The linked report at https://github.com/openclaw/openclaw/issues/87995 includes live Discord evidence from 2026.5.28-beta.1 with 12 sends for one completion, and current-main source still has the requester-wake-first path; I did not rerun the live Discord flow in this read-only review.

## Solution Assessment

No. Direct-first delivery is the right direction, but the current patch keeps an unsafe fallback after an attempted direct media failure, so the narrow fix is to stop or idempotently guard that handoff before merge.
