## Summary

No blocking correctness findings remain on the latest head; this PR is still necessary because current main can route generated-media completions through requester-agent/message-tool delivery, but merge needs maintainer acceptance of the direct-send failure tradeoff.

## What This Changes

The PR moves image/music/video completion media delivery to a direct idempotent send before requester-agent handoff, makes attempted direct media failures terminal, and updates agent delivery tests.

## Best Possible Solution

Land the direct-first generated-media delivery path if maintainers accept terminal attempted-send failures; any future recovery should stay inside direct sender idempotency/retry rather than re-entering requester-agent handoff.

## Reproduction Assessment

Yes. The linked issue provides live Discord reproduction on 2026.5.28-beta.1, and current main source still routes active generated-media completions through requester wake before direct fallback while explicit-route message sends bypass the terminal guard.

## Solution Assessment

Yes, with maintainer acceptance of the tradeoff. Direct-first idempotent media delivery is the narrowest fix for the duplicate-send loop, and terminal attempted-send failures prevent partial platform sends from duplicating through fallback.
