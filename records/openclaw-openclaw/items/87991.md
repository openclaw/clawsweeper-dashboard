## Summary

Keep open: the direct-first approach addresses the reported duplicate completion loop and has strong live proof, but the patch still falls back to requester-agent handoff after a failed primary media send, which can duplicate an artifact when the direct send failed after partial or ambiguous outbound delivery.

## What This Changes

The PR changes generated image, music, and video completion delivery to attempt direct media sending before requester-agent/message-tool handoff, with updated completion-delivery tests for DMs, groups, Telegram topics, active sessions, and fallback behavior.

## Best Possible Solution

Keep direct-first generated-media delivery, but return a terminal failed direct result after an attempted primary media send; only fall back to requester-agent handoff when direct delivery was impossible because there was no target or media to send.

## Reproduction Assessment

Yes. The linked reports provide live duplicate-send evidence on recent beta builds, and current-main source still routes active generated-media completion through requester wake before direct fallback.

## Solution Assessment

No. Direct-first delivery is the right fix direction, but this patch should not wake the requester agent after an attempted primary direct media send fails unless it can prove no visible send occurred.
