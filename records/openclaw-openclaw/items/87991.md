## Summary

Keep this PR open, but do not merge it yet: the happy-path direct-first fix has strong live Discord proof, but the new failed-direct-send fallback can still duplicate generated media after a partial outbound send.

## What This Changes

The PR makes generated image, music, and video completions try direct media delivery before requester-agent handoff and updates completion-delivery tests around that behavior.

## Best Possible Solution

Keep direct-first generated-media completion delivery, but make a failed primary direct attempt terminal or partial-send-aware before any requester-agent fallback can run.

## Reproduction Assessment

Yes from source and linked live evidence: the linked report shows one completion producing 12 Discord posts, and current main still hands active generated-media completions to the requester path before fallback. I did not run a live repro during this read-only review.

## Solution Assessment

No, not yet. Direct-first delivery is the right shape for the reported bug, but the failed-direct fallback must not re-enter requester-agent delivery after a send attempt that may already have posted media.
