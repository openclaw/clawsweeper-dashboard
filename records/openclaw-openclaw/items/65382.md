## Summary

Keep open: this is still the narrow Cron Jobs formatter PR, but the current patch should not merge because it humanizes cron wildcard steps that Croner rejects and it still lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds a Control UI formatter and unit tests so common five-field cron schedules render as `Every ...` labels instead of raw cron text.

## Best Possible Solution

Keep the presenter-level formatter, bound minute and hour wildcard steps to Croner-compatible ranges, add invalid-step fallback coverage, and require redacted after-fix Control UI or terminal proof.

## Reproduction Assessment

Yes for source-level review: current main returns raw cron text, and the PR diff accepts any safe positive wildcard step before humanizing it. Croner 10.0.1 rejects steps greater than the minute/hour field length, so the invalid-label path is clear from source and dependency contract inspection.

## Solution Assessment

No, not as currently proposed. The presenter helper is the right narrow boundary, but it should validate minute/hour step ranges before humanizing and keep invalid schedules on the raw fallback.
