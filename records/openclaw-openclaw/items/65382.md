## Summary

Keep open: current main and the latest release still render cron schedules as raw text, so the formatter remains a plausible Control UI improvement, but this branch should not merge as-is because it can mislabel uneven cron steps and still has no real behavior proof.

## What This Changes

The PR adds Control UI formatting for common cron minute/hour step expressions and unit coverage for hourly, minute, timezone, and raw fallback cases.

## Best Possible Solution

Keep the improvement only if the formatter is exact for Croner-supported simple intervals, falls back to raw cron for uneven or invalid patterns, and the author supplies real Control UI proof.

## Reproduction Assessment

Yes for the PR-introduced formatter bug: source inspection plus a Croner 10.0.1 probe show that `*/40` minutes and `*/7` hours have uneven gaps while the PR would label them as uniform intervals. Current main itself still uses raw cron text, so this is not a current-main regression.

## Solution Assessment

No as-is. The presenter is a reasonable narrow UI boundary, but the formatter needs to mirror Croner cadence semantics and keep unsupported or uneven schedules on the raw fallback before it is the best fix.

- **[P2] Keep uneven cron steps on the raw fallback:** `ui/src/ui/presenter.ts:63`
  - body: `parsePositiveCronStep` accepts any positive `*/N` and the caller renders it as a uniform interval, but Croner does not treat all wildcard steps that way. Please only humanize steps that evenly divide the field, and leave the rest on the raw cron fallback with tests.
