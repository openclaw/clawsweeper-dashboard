## What This Changes

The PR adds a Control UI presenter helper that formats common 5-field cron minute/hour intervals as `Every ...` strings and adds focused presenter tests for hourly, minute, timezone, and fallback display cases.

## Best Possible Solution

Keep the UI presenter boundary, but make the formatter Croner-aware for minute and hour wildcard steps, leave invalid or unsupported cron expressions on the raw `Cron ...` fallback, add regression tests for out-of-range steps, and record the visible UI fix in `CHANGELOG.md` under the active release section.

Yes. On current main, a cron schedule like `0 */6 * * *` still renders as raw `Cron 0 */6 * * *`; on the PR diff, source inspection shows `*/70 * * * *` and `0 */25 * * *` would be rendered as valid-looking intervals even though Croner rejects those ranges.

## Solution Assessment

No, not as written. The presenter helper is the right narrow place to solve the display issue, but it should validate step ranges against Croner's accepted minute/hour bounds and include the required changelog entry before merge.
