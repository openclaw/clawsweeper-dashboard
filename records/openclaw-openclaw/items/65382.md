## Summary

Keep open: current main still renders cron schedules as raw `Cron ...` text, and the related broader cron-humanization PR was closed in favor of this narrower Control UI formatter, but this PR still has a correctness finding and lacks required real behavior proof.

## What This Changes

The PR adds a Control UI presenter helper and tests to render common five-field cron schedules as `Every ...` labels while preserving raw fallback and timezone suffixes.

## Best Possible Solution

Keep the presenter-level formatter, validate minute/hour step bounds against Croner before humanizing, add invalid-step coverage, and require redacted after-fix Control UI or terminal proof.

## Reproduction Assessment

Yes for source-level review: current main returns raw cron text, and PR head would humanize `*/70 * * * *` or `0 */25 * * *` even though Croner 10.0.1 rejects wildcard steps greater than the minute/hour field length. I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently proposed. The presenter helper is the right narrow boundary, but it should validate minute/hour step ranges before humanizing and keep invalid schedules on the raw fallback.
