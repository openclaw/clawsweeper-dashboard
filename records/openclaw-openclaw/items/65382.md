## Summary

Keep open. Current main and v2026.4.29 still display cron schedules through the raw `Cron ...` fallback, so the PR addresses a real Control UI gap, but the patch should not merge as-is because it humanizes out-of-range Croner step values and misses the required user-facing changelog entry.

## What This Changes

The PR adds a Control UI schedule formatter for common five-field cron minute/hour intervals plus presenter tests for hourly, minute, timezone, and fallback cases.

## Best Possible Solution

Merge a focused Control UI formatter that humanizes only Croner-valid common five-field minute/hour interval patterns, preserves raw fallback and timezone annotations for unsupported or invalid expressions, and includes tests for valid and out-of-range cases plus the required changelog entry.

## Reproduction Assessment

Yes. On current main, a cron job with `schedule: { kind: "cron", expr: "0 */6 * * *", tz: "UTC" }` follows `formatCronSchedule` to the raw `Cron ... (UTC)` fallback; in the PR diff, `*/70 * * * *` would be accepted by `parsePositiveCronStep` and displayed as `Every 70 minutes`.

## Solution Assessment

No, not as written. The presenter is the right boundary for this display-only change, but the formatter should validate minute/hour step values against Croner's field limits and add the missing release-note entry before merge.
