## Summary

This should stay open: current main still truncates raw multiline slash command bodies in the shared normalizer, and channel paths can pass that truncated `CommandBody` into downstream skill-command dispatch.

## What This Changes

The issue requests preserving multiline slash skill command arguments after the command head instead of dropping everything after the first newline.

## Best Possible Solution

Fix the shared command normalization/dispatch path so the command head is normalized but multiline argument tails remain available to skill commands and other argument-accepting slash commands, with regression coverage for direct and `/skill` forms plus a channel path that currently pre-normalizes `CommandBody`.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction: pass `/demo_skill\nline1\nline2` through `normalizeCommandBody()` before skill dispatch and the returned body is reduced to the command head or first line. I did not execute the test because this review environment lacks `pnpm`.

## Solution Assessment

Yes, the maintainable direction is to preserve the multiline tail at the shared command normalization or skill-dispatch boundary while keeping command-token normalization, bot-mention stripping, and no-arg command behavior covered by tests.
