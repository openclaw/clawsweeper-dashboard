## Summary

Keep open. Current main has the supported persistent wrapper path and SecretRef-aware status warnings, but it still unconditionally rewrites the generated macOS env wrapper and does not clearly explain when diagnostics are running outside an external secret-provider wrapper context.

## What This Changes

The issue requests safer macOS gateway reinstall/update diagnostics around generated service env wrapper overwrites, secret-provider wrapper context, and related status/doctor warnings.

## Best Possible Solution

Keep generated service-env wrappers managed, warn when an existing generated wrapper has been customized before overwrite, and make status/doctor point secret-provider users to `--wrapper` or `OPENCLAW_WRAPPER` when the CLI is outside the service wrapper context.

## Solution Assessment

Yes. The maintainable direction is generic diagnostics and warnings around the existing wrapper contract, not Infisical-specific support or preserving edited generated files as a new compatibility path.
