## Summary

Keep open. Current main and the latest release have a useful static foreground/background queue priority layer, but they still do not implement this issue's broader gateway fairness contract: configurable priority classes, cooperative preemption/yield policy, priority-aware liveness/head-of-line diagnostics, or doctor guidance.

## What This Changes

Requests an in-process gateway fairness policy so human-facing sessions can outrank background and cron agents under shared event-loop contention, with priority classes, preemption/yield points, diagnostics, and doctor warnings.

## Best Possible Solution

Define one maintainer-approved gateway fairness contract on top of the existing lane priority system, then add diagnostics and any config or doctor behavior only after the starvation and preemption policy is settled.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. Source inspection confirms the remaining gaps: current main only has trigger-based foreground/background priority and lacks configurable priority classes, head-of-line priority diagnostics, and starvation/preemption policy.

## Solution Assessment

Unclear as an implementation path. The existing command-lane priority code is the right boundary to build on, but the requested config, diagnostics, doctor, and preemption behavior need maintainer policy direction first.
