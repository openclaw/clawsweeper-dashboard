## Summary

Keep open. The PR addresses a real current-main fail-closed gap, but the proposed ordering would reject correctly resolved Google Chat SecretRef snapshots because sibling `serviceAccountRef` markers are retained while the resolved credential is written to `serviceAccount`.

## What This Changes

The PR changes Google Chat account credential resolution so an unresolved `serviceAccountRef` errors before falling back to legacy inline `serviceAccount`, and adds regression coverage for that mixed configuration.

## Best Possible Solution

Keep the fail-closed behavior for unresolved `serviceAccountRef`, but implement it in a way that preserves resolved snapshots, either by scrubbing sibling refs when applying resolved Google Chat service-account assignments or by making the account resolver able to distinguish unresolved source config from a resolved runtime snapshot. Add regression coverage for both mixed unresolved config and resolved `serviceAccountRef` snapshots, then add the required changelog entry before merge.

## Reproduction Assessment

Yes. Current main has a direct unit-level reproduction: `resolveGoogleChatAccount` with both inline `serviceAccount` and `serviceAccountRef` returns inline credentials because `accounts.ts` parses inline first; the PR’s added test captures that. The PR regression also has a high-confidence source-level path: runtime SecretRef assignment writes `serviceAccount` while retaining `serviceAccountRef`, then the new guard throws before reading the resolved value.

## Solution Assessment

No. The intended fail-closed direction is right, but checking `serviceAccountRef` before any resolved `serviceAccount` conflates unresolved source config with resolved runtime snapshots. The safer fix keeps `serviceAccountRef` authoritative without breaking the existing resolved-snapshot contract.
