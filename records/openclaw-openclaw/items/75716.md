## Summary

Keep this PR open: the fail-closed precedence is directionally aligned with the docs, but the patch is incomplete because command-resolved Google Chat configs can still retain `serviceAccountRef` beside a resolved `serviceAccount`, causing the reordered resolver to throw on a resolved snapshot.

## What This Changes

The PR reorders Google Chat credential resolution to reject sibling `serviceAccountRef` before inline credentials, removes the sibling ref when the plugin SecretRef assignment applies, and adds regression tests plus a changelog entry.

## Best Possible Solution

Make every resolved Google Chat SecretRef snapshot that can reach `resolveGoogleChatAccount` unambiguous by stripping or otherwise neutralizing the sibling ref after successful resolution, while preserving fail-closed behavior for truly unresolved refs.

## Reproduction Assessment

Yes, by source inspection. The command-gateway path can leave both `serviceAccountRef` and resolved `serviceAccount` in the config, and `openclaw message` then reaches the reordered Google Chat resolver with that mixed shape.

## Solution Assessment

No, not yet. Checking `serviceAccountRef` before inline credentials is the right fail-closed direction, but the fix must also align non-plugin resolved snapshot paths before this can safely merge.
