## Summary

Keep open: the PR targets a real gap in current `status --all` degraded output, but the proposed patch should not merge as-is because it introduces unsafe provider interpolation in a copy-paste auth command and has two concrete recovery-guidance correctness issues.

## What This Changes

The PR adds recovery guidance classifiers and tests for degraded `status --all` diagnosis output in `src/commands/status-all/diagnosis.ts` and `src/commands/status-all/diagnosis.test.ts`.

## Best Possible Solution

Land a rebased status diagnosis patch that keeps the structured cause/fix recovery guidance, reuses the existing OAuth refresh classifier and login-command sanitizer, tightens HTTP-status matching, emits unreachable recovery guidance once, and adds the required user-facing changelog entry.

## Reproduction Assessment

Yes. The defects are reproducible at unit level by calling `appendStatusAllDiagnosis` with crafted `health.error` values: an unsafe OAuth provider string shows the unsanitized command, `ECONNREFUSED 127.0.0.1:4030` hits the broad auth branch, and `gateway unreachable` with `gatewayReachable: false` prints duplicate guidance.

## Solution Assessment

No. The direction is the right narrow surface, but the current implementation should reuse the existing sanitized OAuth helper and make the classifiers precise before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reuse the sanitized OAuth refresh helper:** `src/commands/status-all/diagnosis.ts:64-67`
  - body: The new helper parses the provider from `health.error` and interpolates it into a login command directly. Current main already has `classifyOAuthRefreshFailure` and `buildOAuthRefreshFailureLoginCommand`, which sanitize provider IDs and fall back to the generic command for unsafe text; reuse that path so crafted health errors cannot produce unsafe copy-paste guidance.
  - confidence: 0.9
- **[P2] Match auth status codes precisely:** `src/commands/status-all/diagnosis.ts:123-128`
  - body: This branch treats any text containing `401` or `403` as an auth failure. Transport errors can include those digits in ports or durations, such as `127.0.0.1:4030`, so `status --all` can suggest token/password repair when the gateway is actually unreachable.
  - confidence: 0.88
- **[P2] Avoid repeating unreachable recovery guidance:** `src/commands/status-all/diagnosis.ts:386-398`
  - body: When `health.error` is `gateway unreachable` and `gatewayReachable` is false, the skipped-channel block emits unreachable cause/fix guidance and the health block emits the same diagnosis again. Collapse this to one section or make the health section refer back to the skipped-channel diagnosis.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/commands/status-all/diagnosis.ts:215-222`
  - body: This changes user-facing `status --all` recovery output, so repo policy expects a single-line `CHANGELOG.md` entry before merge. The PR currently changes only the diagnosis implementation and tests.
  - confidence: 0.77

## Security Review

Status: needs_attention

Summary: The diff introduces a command-guidance injection risk by formatting provider-specific OAuth login commands from parsed health error text without the existing sanitizer.

Concerns:

- **[medium] Sanitize provider IDs before printing login commands:** `src/commands/status-all/diagnosis.ts:66`
  - body: `buildOAuthRefreshLoginCommand` receives a provider parsed from `health.error` and formats `openclaw models auth login --provider ...` without normalization or safety checks. Reuse `buildOAuthRefreshFailureLoginCommand` from `src/agents/auth-profiles/oauth-refresh-failure.ts` so malformed provider text falls back to the generic login command.
  - confidence: 0.86
