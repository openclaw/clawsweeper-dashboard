## Summary

Keep this PR open. Current main still lacks the structured degraded-state recovery guidance the branch adds, but the proposed patch is not ready to merge because it can print unsafe OAuth re-auth commands from parsed error text and can misclassify or duplicate gateway guidance.

## What This Changes

The PR adds `status --all` recovery guidance helpers and tests for invalid config, gateway log failures, skipped channel inspection, gateway health failures, bind errors, unreachable gateways, and OAuth re-auth cases.

## Best Possible Solution

Land a rebased status-all guidance patch that keeps the PR's structured cause/fix output, reuses the existing sanitized OAuth refresh helper, avoids duplicate unreachable guidance, matches HTTP auth status codes precisely, and adds the required user-facing changelog entry.

## Reproduction Assessment

Yes. The proposed behavior and the defects are reproducible with unit-level `appendStatusAllDiagnosis` parameter objects: `health.error = "gateway unreachable"` duplicates guidance, and strings such as `ECONNREFUSED 127.0.0.1:4030` trigger the broad auth-status branch.

## Solution Assessment

No, not as currently implemented. The direction is the right narrow surface, but the maintainable fix should reuse the existing OAuth helper and tighten the classifiers before this is merged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reuse the sanitized OAuth refresh helper:** `src/commands/status-all/diagnosis.ts:61-66`
  - body: The PR parses the provider out of `health.error` and interpolates it into a login command directly. Current main already has `classifyOAuthRefreshFailure` and `buildOAuthRefreshFailureLoginCommand`, which sanitize provider IDs before printing provider-specific commands; use that helper here so crafted error text cannot produce unsafe copy-paste guidance.
  - confidence: 0.9
- **[P2] Match auth status codes precisely:** `src/commands/status-all/diagnosis.ts:123-133`
  - body: The health classifier treats any text containing `401` or `403` as an auth failure. That will misclassify transport errors containing those digits, such as `127.0.0.1:4030`, and suggest token/password fixes when the gateway is actually unreachable.
  - confidence: 0.88
- **[P2] Avoid repeating unreachable recovery guidance:** `src/commands/status-all/diagnosis.ts:386-398`
  - body: When `health.error` is `gateway unreachable` and `gatewayReachable` is false, the channel-skip block and health block both emit the same cause/fix guidance. Collapse that to one place or make the health section refer back to the skipped-channel diagnosis.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/commands/status-all/diagnosis.ts:260-265`
  - body: This changes user-facing `status --all` recovery output, so repo policy requires a single-line `CHANGELOG.md` entry with contributor credit before merge.
  - confidence: 0.77

## Security Review

Status: needs_attention
