## Summary

Keep open: the PR is a useful focused improvement, and current main still lacks the proposed recovery guidance, but the branch is not merge-ready because it introduces actionable command-guidance defects and has no real after-fix CLI proof.

## What This Changes

The PR adds `Cause:`/`Fix:` recovery guidance and tests for degraded `status --all` config, gateway log, gateway health, unreachable, and OAuth re-auth diagnostics.

## Best Possible Solution

Land a revised status diagnosis patch that reuses the existing sanitized OAuth helper, bounds HTTP status matching, de-duplicates unreachable guidance, and includes redacted real CLI output.

## Reproduction Assessment

Yes, for the PR-introduced defects: source inspection gives concrete paths through `appendStatusAllDiagnosis` with crafted `health.error` values for unsafe OAuth provider text, numeric substrings like `127.0.0.1:4030`, and `gateway unreachable`. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is narrow and useful, but the implementation should reuse the sanitizer-backed OAuth helper, match HTTP statuses precisely, and avoid duplicate unreachable guidance before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reuse the sanitized OAuth refresh helper:** `src/commands/status-all/diagnosis.ts:64-67`
  - body: The new helper parses the provider from `health.error` and interpolates it into a copy-paste login command directly. Current main already has `classifyOAuthRefreshFailure` and `buildOAuthRefreshFailureLoginCommand`, which sanitize provider IDs and fall back to the generic command for unsafe text; reuse that path so crafted health errors cannot produce unsafe command guidance.
  - confidence: 0.9
- **[P2] Match HTTP auth statuses precisely:** `src/commands/status-all/diagnosis.ts:123-128`
  - body: This branch treats any text containing `401` or `403` as an auth failure before checking transport failures. Errors such as `ECONNREFUSED 127.0.0.1:4030` can therefore suggest token/password repair when the gateway is actually unreachable, so the status-code match needs HTTP-status-specific bounds or transport checks first.
  - confidence: 0.88
- **[P2] Emit unreachable recovery guidance only once:** `src/commands/status-all/diagnosis.ts:393-398`
  - body: When `health.error` is `gateway unreachable` and `gatewayReachable` is false, the skipped-channel block emits unreachable cause/fix guidance and the health block emits the same diagnosis again. Collapse this to one section or make the health section refer back so the degraded report does not repeat the same recovery advice.
  - confidence: 0.84
