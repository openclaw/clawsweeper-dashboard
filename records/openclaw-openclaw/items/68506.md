## Summary

Keep open: current main has not implemented the PR’s structured degraded-state recovery guidance, and the PR remains useful, but the patch still needs fixes for OAuth command sanitization, HTTP status matching, duplicate unreachable guidance, and missing real behavior proof.

## What This Changes

The PR adds Cause/Fix recovery guidance to `status --all` degraded diagnosis output and tests config, gateway log, health, unreachable, and OAuth refresh cases.

## Best Possible Solution

Revise the PR branch to reuse the sanitizer-backed OAuth helper, tighten HTTP status matching, emit unreachable recovery guidance once, and include redacted real `status --all` proof.

## Reproduction Assessment

Yes, for the PR-introduced defects: source inspection gives direct paths through `appendStatusAllDiagnosis` with crafted `health.error` values for unsafe OAuth provider text, numeric substrings such as `127.0.0.1:4030`, and `gateway unreachable`. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is useful and narrow, but this implementation should reuse the existing OAuth sanitizer, match HTTP auth statuses precisely, and collapse duplicate unreachable guidance before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reuse the sanitized OAuth refresh helper:** `src/commands/status-all/diagnosis.ts:64-67`
  - body: The new helper parses the provider from `health.error` and interpolates it into a copy-paste login command directly. Current main already has `classifyOAuthRefreshFailure` and `buildOAuthRefreshFailureLoginCommand`, which sanitize provider IDs and fall back to the generic command for unsafe text; reuse that path so malformed health errors cannot produce unsafe or invalid command guidance.
  - confidence: 0.9
- **[P2] Match HTTP auth statuses precisely:** `src/commands/status-all/diagnosis.ts:123-128`
  - body: This branch treats any text containing `401` or `403` as an auth failure before checking transport failures. Errors such as `ECONNREFUSED 127.0.0.1:4030` can therefore suggest token/password repair when the gateway is actually unreachable, so the status-code match needs HTTP-status-specific bounds or transport checks first.
  - confidence: 0.88
- **[P2] Emit unreachable recovery guidance only once:** `src/commands/status-all/diagnosis.ts:393-398`
  - body: When `health.error` is `gateway unreachable` and `gatewayReachable` is false, the skipped-channel block emits unreachable cause/fix guidance and the health block emits the same diagnosis again. Collapse this to one section or make the health section refer back so the degraded report does not repeat the same recovery advice.
  - confidence: 0.84
