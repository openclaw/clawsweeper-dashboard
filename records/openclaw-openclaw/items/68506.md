- **[P1] Reuse the sanitized OAuth refresh helper:** `src/commands/status-all/diagnosis.ts:64-67`
  - body: This builds a login command from the provider text parsed out of `health.error`. Current main already has `classifyOAuthRefreshFailure` / `buildOAuthRefreshFailureLoginCommand`, which sanitizes provider ids and avoids unsafe or stale provider strings; using a local parser here can tell users to copy a malformed or legacy provider command during auth recovery.
  - confidence: 0.89
- **[P2] Match auth status codes precisely:** `src/commands/status-all/diagnosis.ts:123-128`
  - body: The health classifier treats any error containing `401` or `403` as an auth failure. A transport error with those digits in a port, timeout, or id can now produce credential-token advice instead of gateway recovery guidance, so this should use bounded HTTP-status matching or classify transport failures first.
  - confidence: 0.87
- **[P3] Avoid repeating unreachable-gateway guidance:** `src/commands/status-all/diagnosis.ts:393-398`
  - body: When `health.error` is `gateway unreachable` and `gatewayReachable` is false, the channel-skip block prints the same cause/fix guidance that the gateway-health block prints again. The new test only checks `toContain`, so it would not catch the duplicate user-facing diagnosis.
  - confidence: 0.81
