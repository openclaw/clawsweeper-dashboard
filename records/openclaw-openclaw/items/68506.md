## Summary

Keep open: current main still lacks the structured cause/fix guidance, and this PR targets the right status surface, but the current diff should not merge because it bypasses the existing OAuth command sanitizer, loosely matches 401/403 text, repeats unreachable guidance, and misses the required changelog entry.

## What This Changes

The PR adds recovery classifiers and tests for degraded `status --all` output covering invalid config, gateway log/health failures, and OAuth re-auth hints.

## Best Possible Solution

Land a rebased status diagnosis patch that uses the existing sanitized OAuth recovery helper, precise health-status classification, single-source unreachable guidance, and changelog coverage.

## Reproduction Assessment

Yes. The remaining defects are reproducible at unit level on the PR head by calling `appendStatusAllDiagnosis` with crafted `health.error` values for unsafe OAuth provider text, `ECONNREFUSED 127.0.0.1:4030`, and `gateway unreachable` with `gatewayReachable: false`.

## Solution Assessment

No. The product direction is narrow and maintainable, but the current implementation should reuse the shipped OAuth sanitizer, tighten status-code matching, avoid duplicated recovery output, and rebase over current main before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reuse the sanitized OAuth refresh helper:** `src/commands/status-all/diagnosis.ts:64-67`
  - body: The new helper parses the provider from `health.error` and interpolates it into a copy-paste login command. `formatCliCommand` only rewrites CLI prefixes/profile hints; it does not sanitize arbitrary provider text, while current main already has `classifyOAuthRefreshFailure` and `buildOAuthRefreshFailureLoginCommand` for this exact path.
  - confidence: 0.9
- **[P2] Match auth status codes precisely:** `src/commands/status-all/diagnosis.ts:123-128`
  - body: This branch treats any health error containing `401` or `403` as an auth failure. Transport errors can include those digits in ports or timings, such as `127.0.0.1:4030`, so `status --all` can suggest token/password repair when the gateway is actually unreachable.
  - confidence: 0.88
- **[P2] Avoid repeating unreachable recovery guidance:** `src/commands/status-all/diagnosis.ts:386-398`
  - body: When the health error is `gateway unreachable` and `gatewayReachable` is false, the skipped-channel block emits unreachable cause/fix guidance and the health block emits the same diagnosis again. Collapse this to one section or make the health section refer back to the earlier diagnosis.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/commands/status-all/diagnosis.ts:218-225`
  - body: This changes user-facing `status --all` recovery output, so repo policy expects a single-line `CHANGELOG.md` entry before merge. The PR currently changes only the diagnosis implementation and tests.
  - confidence: 0.77
