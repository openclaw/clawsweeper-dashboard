## Summary

Keep open: the PR is useful, but the current diff still has blocking correctness/security issues in the new recovery classifiers and lacks after-fix real behavior proof.

## What This Changes

The PR adds recovery-guidance classifiers and tests for degraded `status --all` diagnosis output, covering config problems, gateway logs, health failures, unreachable gateways, and OAuth re-auth cases.

## Best Possible Solution

Land a revised status diagnosis patch that keeps the structured recovery guidance, reuses the existing sanitized OAuth helper, tightens HTTP-status matching, emits unreachable recovery guidance once, and includes real redacted CLI proof.

## Reproduction Assessment

Yes, for the PR-introduced defects: source inspection gives concrete unit-level paths through `appendStatusAllDiagnosis` with crafted `health.error` values for unsafe OAuth provider text, `127.0.0.1:4030`, and `gateway unreachable`. I did not run tests because this review was read-only.

## Solution Assessment

No. The feature direction is narrow and useful, but the current implementation should reuse the sanitizer-backed OAuth helper, classify HTTP statuses precisely, and avoid duplicate unreachable guidance before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reuse the sanitized OAuth refresh helper:** `src/commands/status-all/diagnosis.ts:64-67`
  - body: The new helper parses the provider from `health.error` and interpolates it into a copy-paste login command directly. Current main already has `classifyOAuthRefreshFailure` and `buildOAuthRefreshFailureLoginCommand`, which sanitize provider IDs and fall back to the generic command for unsafe text; reuse that path so crafted health errors cannot produce unsafe command guidance.
  - confidence: 0.9
- **[P2] Match auth status codes precisely:** `src/commands/status-all/diagnosis.ts:123-128`
  - body: This branch treats any text containing `401` or `403` as an auth failure before checking transport failures. Errors such as `ECONNREFUSED 127.0.0.1:4030` can therefore suggest token/password repair when the gateway is actually unreachable, so the status-code match needs HTTP-status-specific bounds or transport checks first.
  - confidence: 0.88
- **[P2] Avoid repeating unreachable recovery guidance:** `src/commands/status-all/diagnosis.ts:386-398`
  - body: When `health.error` is `gateway unreachable` and `gatewayReachable` is false, the skipped-channel block emits unreachable cause/fix guidance and the health block emits the same diagnosis again. Collapse this to one section or make the health section refer back so the degraded report does not repeat the same recovery advice.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive command guidance that formats provider-specific OAuth login commands from parsed health error text without the existing sanitizer.
