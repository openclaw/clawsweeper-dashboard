Full review comments:

- **[P1] Reuse the sanitized OAuth refresh helper:** `src/commands/status-all/diagnosis.ts:56-65`
  - body: This reimplements OAuth refresh parsing and builds the displayed login command from provider text parsed out of a gateway health error. Current main already has `classifyOAuthRefreshFailure` and `buildOAuthRefreshFailureLoginCommand`, which sanitize provider ids and preserve the canonical `openai` recovery path for legacy Codex aliases; duplicating that path can show unsafe or stale auth guidance.
  - confidence: 0.86
- **[P2] Match HTTP auth status codes precisely:** `src/commands/status-all/diagnosis.ts:127-128`
  - body: The health classifier treats any error text containing `401` or `403` as an auth failure, so values like `127.0.0.1:4030` or `1401ms` can bypass the transport-failure branch and tell users to repair credentials when the gateway is actually unreachable. Match bounded HTTP status contexts or prioritize transport errors first.
  - confidence: 0.83
- **[P3] Avoid printing unreachable guidance twice:** `src/commands/status-all/diagnosis.ts:386-398`
  - body: For `health.error = "gateway unreachable"` with `gatewayReachable = false`, the new channel-skip block and the new gateway-health recovery block both emit the same cause/fix. That makes one degraded state look like two independent problems, so emit the recovery hint in only one section or make the health section refer back to the channel-skip diagnosis.
  - confidence: 0.78
