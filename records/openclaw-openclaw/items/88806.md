## Summary

Keep this PR open, but it should not merge yet: the bug-fix direction is valid, while the current patch can corrupt direct-message memory text, adds an undocumented public SDK subpath, and only supplies mock/unit-test proof.

## What This Changes

The PR adds memory-lancedb filtering and sanitization for inbound envelope metadata, adds a public `plugin-sdk/chat-channel-ids` subpath, updates SDK export metadata, and expands memory-lancedb tests.

## Best Possible Solution

Land a narrow memory-lancedb fix that strips only envelope metadata, preserves direct-message body text, uses or documents a supported SDK seam, and includes real after-fix gateway/plugin proof.

## Reproduction Assessment

Yes, from source inspection: current `formatInboundEnvelope` leaves direct-message bodies unprefixed, while the proposed sanitizer strips the first colon-prefixed token after any envelope bracket. A direct envelope like `[telegram alice] TODO: remember this` would store `remember this`.

## Solution Assessment

No; rejecting marker-free envelope sludge is the right bug target, but this patch is too broad for direct-message bodies and the new public SDK subpath needs a clearer contract or a narrower existing seam.
