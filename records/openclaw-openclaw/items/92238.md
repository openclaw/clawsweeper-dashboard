## Summary

The report is a current, source-reproducible regression and current main still lacks a shell tool in the direct, sandbox-off Codex path. A clean open PR specifically owns the focused repair, so this issue should remain open until that PR merges or is otherwise resolved.

## What This Changes

Restore shell execution for isolated OpenAI/Codex cron runs when policy allows `exec` but the Codex app-server has no execution environment.

## Best Possible Solution

Land the linked focused repair so existing OpenClaw `exec` and `process` tools are restored only when Codex has no native execution environment, while preserving sandbox-native execution, memory-flush restrictions, and explicit dynamic-tool exclusions.

## Reproduction Assessment

Yes, at high confidence from source: current main filters OpenClaw shell tools in the direct Codex path, while the bundled Codex 0.139.0 does not advertise native shell tools without a turn environment. The reporter also supplied a same-host Google control that isolates the failure to the Codex runtime path.

## Solution Assessment

Yes. Restoring the already-authorized OpenClaw shell tools at the Codex dynamic-tool owner boundary is narrower and more maintainable than changing cron, sandbox policy, or adding configuration, and the linked PR covers the sibling side-question path.
