## Summary

Keep open: the PR targets a real cron delivery bug, but the patch clears fatal state on final assistant text without ensuring that same final text is what gets delivered on channels that do not opt into final-text preference.

## What This Changes

The PR adds a finalAssistantVisibleText-based recovery gate in `resolveCronPayloadOutcome` and updates helper tests for recovered cron agentTurn error classification.

## Best Possible Solution

Land one canonical cron helper fix that keeps recovered tool errors non-fatal only when the final assistant output is also selected for delivery, while preserving run-level and explicit fatal-signal precedence.

## Reproduction Assessment

Yes at source level: current main still routes fatal structured-error classification into the pre-dispatch return path, and the canonical issue has a real reduced scheduled-cron repro. I did not run a live cron job in this read-only review.

## Solution Assessment

No: the final-answer recovery idea is plausible, but this implementation does not keep fatal-state recovery and delivery-payload selection coupled across channel policies. The safer fix is to select final assistant text whenever it is the recovery proof, or keep the gate scoped to final-text-preferring channels.
