## Summary

Keep open: the PR targets a real cron delivery/status bug, but the current patch is too broad because any non-empty `finalAssistantVisibleText` can downgrade generic/plain error payloads to a successful cron outcome.

## What This Changes

The PR adds a `finalAssistantVisibleText` recovery gate to cron isolated-agent outcome classification and updates helper tests so recovered error payloads can be classified and delivered as final assistant text.

## Best Possible Solution

Narrow the recovery predicate so final assistant text only clears fatality for known non-terminal/tool-warning payloads, keep generic/plain provider errors fatal, and land one canonical fix for the open cron recovery issue.

## Reproduction Assessment

Yes, source-level: current main keeps generic trailing errors and all-error payloads fatal even when final assistant text exists, while this PR flips those controls. I did not run a live cron/channel reproduction in this read-only review.

## Solution Assessment

No. The helper is the right layer, but this implementation is too broad; the safer fix is to use final assistant text only when the error payload itself is known non-terminal or tool-warning-shaped.
