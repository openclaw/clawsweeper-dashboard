## Summary

Keep open. Current main still has the Gateway event primitives needed for session/message observation, but the bundled TUI does not subscribe to or handle `session.message` / `sessions.changed` events and has no opt-in observer or unified inbox mode. Related #51691 was closed into this broader tracker, while #40678 and #73402 remain adjacent but do not fully implement this request.

## What This Changes

The issue requests unified cross-channel message visibility and reply sync across TUI, Telegram, and iMessage surfaces.

## Best Possible Solution

Keep a canonical product tracker for an explicit opt-in TUI observer or unified-inbox mode that consumes the existing Gateway session subscription primitives, clearly labels channel/session origin, separates read-only observation from interactive reply routing, and preserves current session isolation by default.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has no observe/mirror TUI option, the TUI backend has no subscription methods, TUI dispatch ignores `session.message` and `sessions.changed`, and visible chat remains filtered to the active session key.

## Solution Assessment

No for unconditional full sync as stated. The safer maintainable solution is an explicit opt-in observer/unified-inbox mode built on scoped Gateway subscriptions with origin labeling and read authorization; #73402 should remain a separate narrow active-session fix path.
