## Summary

Keep open: current main has a password fallback and same-host proxy opt-in, but the reported headerless internal loopback path still fails without configured password credentials, and the open RFC/PR context shows the final trust model is unresolved.

## What This Changes

The issue asks OpenClaw to support manager/subagent/backend self-connections when Gateway runs in trusted-proxy mode behind Traefik/authentik without proxy identity headers on internal loopback calls.

## Best Possible Solution

Keep the issue open until maintainers choose the intended internal-auth model and either land a narrow trusted-proxy loopback/service-identity implementation or document the password fallback as the supported deployment answer.

## Reproduction Assessment

Yes. Source inspection and current tests show a local loopback trusted-proxy request without a configured/supplied password still returns trusted_proxy_loopback_source or trusted_proxy_user_missing instead of authenticating as an internal caller.

## Solution Assessment

No. The current password fallback is a useful workaround, but it does not implement inherited user identity or a separate internal service-auth path for the reported no-password configuration.
