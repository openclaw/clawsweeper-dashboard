## Summary

Keep open: current main has a documented password fallback and same-host proxy-header opt-in, but the reported no-password headerless internal loopback path still fails closed and the related RFC/open implementation candidate need maintainer trust-model review.

## What This Changes

Configure trusted-proxy Gateway deployments so manager, subagent, and backend loopback self-connections authenticate instead of failing with trusted_proxy_user_missing or websocket 1008.

## Best Possible Solution

Keep this open until maintainers choose and land the intended internal-auth model, while preserving the documented password fallback as the supported workaround.

## Reproduction Assessment

Yes. Source inspection and current regression tests show a loopback trusted-proxy request without a configured/supplied password still fails with trusted_proxy_loopback_source or trusted_proxy_user_missing instead of authenticating as an internal caller.

## Solution Assessment

No. Current main's password fallback is useful, but it does not implement the requested inherited-user or separate internal service-auth path for the reported no-password configuration.
