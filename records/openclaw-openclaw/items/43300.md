## Summary

Keep open. Current main has a documented password-fallback mitigation, but the exact headerless trusted-proxy self-connection shape remains source-reproducible without `gateway.auth.password`, and linked auth-model work should be reviewed or explicitly superseded before cleanup.

## What This Changes

The issue asks for trusted-proxy Gateway deployments to support manager/subagent/backend self-connections that currently connect over loopback without proxy identity headers.

## Best Possible Solution

Resolve the auth-model path through maintainer review of #69066 and the linked implementation work, keeping trusted-proxy user auth fail-closed while either documenting password fallback as the supported internal path or adding first-class internal service identity.

## Reproduction Assessment

Yes, from source inspection rather than a live container run. Current main still routes local Gateway calls to loopback and rejects trusted-proxy loopback requests without either proxy identity headers or a configured password fallback.

## Solution Assessment

Unclear. The documented password fallback is a narrow mitigation, but the open RFC argues for separate internal service identity, so the best permanent fix needs maintainer auth/security judgment before this issue is closed.
