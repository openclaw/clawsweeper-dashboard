## Summary

Keep open: current main has a documented password fallback and same-host proxy opt-in, but the originally reported headerless loopback self-connection still fails unless a local password is configured. No loopbackUser or first-class service-identity path has landed, and the related RFC/PR context shows this still needs maintainer trust-model judgment.

## What This Changes

The issue asks OpenClaw to support manager, subagent, and backend self-connections when Gateway uses trusted-proxy auth behind Traefik/authentik.

## Best Possible Solution

Keep the issue open until maintainers choose the intended trusted-proxy internal-auth model: either document the password fallback as the supported answer for this deployment, land a narrow explicit loopback/service identity mechanism, or make #69066 the canonical design and close this only after that path ships.

## Reproduction Assessment

Yes. The source-level path is high-confidence: a local loopback trusted-proxy request without a configured/supplied password still returns trusted_proxy_loopback_source or trusted_proxy_user_missing, and current tests lock in that no-password failure behavior.

## Solution Assessment

No. The current password fallback is a useful workaround for configured-password deployments, but it does not implement inherited user identity or a separate internal service-auth path for the reported config; a safer final answer needs maintainer/security agreement on that boundary.
