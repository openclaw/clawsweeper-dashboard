## Summary

Keep open: current main ships a password fallback and same-host proxy opt-in, but the reported headerless internal loopback path still fails without configured password credentials, and the linked RFC/open implementation context shows the remaining trust model is not settled.

## What This Changes

Issue requests that trusted-proxy Gateway deployments support manager/subagent/backend self-connections that bypass the reverse proxy and therefore lack proxy identity headers.

## Best Possible Solution

Keep this issue open until maintainers choose the intended internal-auth model and either land a narrow service-identity/loopback implementation or explicitly document password fallback as the supported answer for these deployments.

## Reproduction Assessment

Yes. Source inspection and current tests show a trusted-proxy local loopback request without proxy identity headers or a supplied password still returns `trusted_proxy_loopback_source` or `trusted_proxy_user_missing`.

## Solution Assessment

No. The current password fallback is a useful shipped workaround, but it does not implement inherited user identity or a separate internal service-auth path for the no-password configuration described here.
