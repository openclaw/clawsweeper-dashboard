## Summary

Keep open: current main still has a source-reproducible path where recurring `main` cron `systemEvent` jobs wake the target session and proceed through the normal transcript-backed model path. The useful fix is not a conservative cleanup close because it requires a maintainer decision about cron session semantics, context budgeting, and model-call recovery policy.

## What This Changes

This issue requests a safer cron context and timeout policy so recurring `systemEvent` jobs targeted at existing sessions do not repeatedly submit long transcripts to the model.

## Best Possible Solution

Define an explicit cron context policy that keeps deliberate session reuse available while giving lightweight recurring jobs a fresh or bounded-context mode and a clear timeout/recovery story for long model calls.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create a long-lived main session, schedule a recurring `main` cron `systemEvent`, and the cron-triggered heartbeat path reopens that session's transcript-backed messages before model submission; I did not run a live multi-day token-spend reproduction.

## Solution Assessment

Unclear as proposed. Defaulting all recurring system events to ephemeral sessions could break users who intentionally target shared session context, so the maintainable path is an explicit cron context/budget and watchdog policy chosen by maintainers.
