## Summary

Keep open: current `main` still has a source-reproducible path where a `main` cron `systemEvent` targets an existing session and the normal heartbeat/reply runner opens that session's transcript-backed state before model submission. The remedy would change cron/session policy or add new budgeting and watchdog behavior, so it is not a conservative auto-close or narrow bug-fix lane.

## What This Changes

The issue reports that recurring cron `systemEvent` jobs targeted at long-lived sessions resend the accumulated session transcript on each tick, increasing cost and runtime.

## Best Possible Solution

Define an explicit cron context policy that keeps intentional session reuse available while giving recurring lightweight jobs an isolated or bounded-context default and a timeout/recovery story for long model calls.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: create a long-lived main session, schedule a recurring `main` cron `systemEvent`, and the cron-triggered heartbeat/reply path will reopen the same transcript-backed session before model submission. I did not run a live multi-day transcript reproduction in this read-only review.

## Solution Assessment

Unclear as proposed: defaulting all recurring system events to ephemeral sessions could break users who intentionally target the host session. A safer maintainable direction is an explicit cron context/budget policy with opt-in session reuse and bounded watchdog behavior.
