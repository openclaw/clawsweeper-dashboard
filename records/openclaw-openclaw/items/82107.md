## Summary

Keep open. Source inspection supports the reported cost path: recurring main-session cron `systemEvent` jobs enqueue into an existing session and wake heartbeat on that same session, while heartbeat runs only avoid prior transcript cost when `isolatedSession` is explicitly enabled and preflight compaction is skipped for heartbeats. The remaining fix needs product/API judgment because changing main-session cron defaults could alter jobs that intentionally wake the host session.

## What This Changes

The issue asks OpenClaw to prevent recurring cron `systemEvent` jobs targeting long-lived sessions from repeatedly sending accumulated session history to the model.

## Best Possible Solution

Define a documented safe recurring-cron mode that bounds or isolates context for low-cost reminders while preserving explicit main-session reuse for jobs that intentionally need host-session continuity.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear path: create a recurring main `systemEvent` against an existing large session, trigger a cron wake, and observe the heartbeat/reply run on that same session. I did not run a live multi-day token-cost reproduction, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Is this the best way to solve the issue? Unclear until owners choose the product contract: default ephemeral execution, per-job context budgets, and watchdog cancellation are all plausible but have different compatibility costs. The maintainable endpoint should make recurring low-cost reminders explicit without silently changing intentional main-session jobs.
