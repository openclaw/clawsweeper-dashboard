## Summary

Keep open: current `main` still routes recurring `main` cron `systemEvent` jobs into the selected long-lived session, so the transcript-cost mechanism remains source-reproducible. Current code has a generic cron timeout safety net, but solving the main cost issue requires a maintainer decision about cron session/context policy.

## What This Changes

The issue requests a cron context and timeout policy so recurring main-session `systemEvent` jobs do not resend an accumulated session transcript or stall per tick.

## Best Possible Solution

Define a cron context contract that gives lightweight recurring jobs isolated or bounded context by default while preserving explicit session reuse and documenting timeout behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction is available: create a long-lived main session, schedule a recurring `main` `systemEvent`, and the cron-triggered heartbeat path reuses that session key before model submission; I did not run a live multi-day token measurement.

## Solution Assessment

Unclear as proposed. Making all recurring system events ephemeral could break intentional main-session reminders, so the maintainable fix should be an explicit cron context/budget policy with clear timeout behavior.
