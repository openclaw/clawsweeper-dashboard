## Summary

Keep open: current main still has a source-reproducible path where recurring `main` cron `systemEvent` jobs target the same transcript-backed session, and the practical fixes require cron/session policy, context-budget, or watchdog decisions rather than conservative cleanup.

## What This Changes

The issue requests mitigation for recurring cron `systemEvent` jobs that target long-lived sessions and repeatedly submit the accumulated session transcript to the model.

## Best Possible Solution

Define an explicit cron context policy that keeps intentional session reuse available while giving recurring lightweight jobs an isolated or bounded-context default and a timeout/recovery story for long model calls.

## Reproduction Assessment

Yes. A high-confidence source path exists: create a long-lived main session, schedule a recurring main `systemEvent`, and current cron/heartbeat code reopens and prompts against that same transcript-backed session.

## Solution Assessment

Unclear as proposed. Defaulting all recurring system events to ephemeral sessions could break users who intentionally target the host session, so a maintainer should choose an explicit context/budget/watchdog contract first.
