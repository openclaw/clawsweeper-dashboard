## Summary

Keep open: current main still has a source-backed path where recurring main-session cron `systemEvent` jobs reuse the target session's transcript-backed heartbeat run, and fixing that safely requires a cron/session context policy or budget/watchdog decision rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to prevent recurring main-session cron `systemEvent` jobs from paying full long-lived session transcript cost on each tick.

## Best Possible Solution

Define an explicit cron context policy that preserves intentional persistent-session work while giving lightweight recurring jobs an isolated or bounded-context default plus a clear timeout/recovery story for long model calls.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a recurring `main` cron `systemEvent` enqueues onto the target session key, wakes the heartbeat runner with that key, and the embedded runner opens the transcript-backed session before model submission; I did not run a live multi-day token-spend reproduction.

## Solution Assessment

Unclear as proposed. Defaulting all recurring system events to ephemeral sessions could break users who intentionally target host-session context, so the better solution is an explicit maintainer-approved cron context, budget, and watchdog policy.
