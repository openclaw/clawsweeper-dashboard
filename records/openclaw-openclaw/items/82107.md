## Summary

Keep open: current main still has a source-reproducible path where recurring main-session cron system events reopen the target session's transcript-backed state before model submission. The fix would change cron/session context policy, budgeting, or watchdog semantics, so this needs maintainer product review rather than cleanup closure.

## What This Changes

Requests a safer, lower-cost cron `systemEvent` policy for recurring jobs targeting long-lived sessions so each tick does not resend the full transcript or stall model calls.

## Best Possible Solution

Define an explicit cron context policy that preserves intentional session reuse while giving lightweight recurring jobs an isolated or bounded-context mode and a clear timeout/recovery story for long model calls.

## Reproduction Assessment

Yes, at source level. A long-lived main-session cron `systemEvent` still targets the same session key, enters the heartbeat/model path, and opens transcript-backed session state before prompting; I did not run a live multi-day transcript reproduction.

## Solution Assessment

Unclear as proposed. Defaulting every recurring system event to an ephemeral session could break users who intentionally target the host session, so the better direction is an explicit context/budget policy with compatibility tests and docs.
