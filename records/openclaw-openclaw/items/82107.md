## Summary

Keep open: current main still has a source-reproducible cron path where main-session systemEvent jobs enqueue into the selected session and the heartbeat runner opens the transcript-backed session before model submission. The remaining fix is not a conservative cleanup close because it requires a cron/session policy choice, and related open issues only cover adjacent budget or stalled-session pieces, not the cron context default itself.

## What This Changes

The issue reports that recurring cron systemEvent jobs targeting a long-lived session reuse the full transcript on each tick, causing repeated high-cost model calls.

## Best Possible Solution

Define an explicit cron context policy that keeps intentional session reuse available while giving recurring lightweight jobs an isolated or bounded-context default and a model-call timeout/recovery story.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: create a long-lived main session, schedule a recurring main cron systemEvent, and the cron-triggered heartbeat path reopens that transcript-backed session before model submission. I did not run a live multi-day token-spend reproduction in this read-only review.

## Solution Assessment

Unclear. Defaulting all recurring system events to ephemeral sessions could break users who intentionally target the host session; a safer maintainable direction is an explicit cron context/budget policy with opt-in session reuse and bounded watchdog behavior.
