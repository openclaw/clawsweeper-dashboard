## Summary

Keep open: current main still has a source-reproducible path where recurring main-session cron `systemEvent` jobs enqueue into and wake the same long-lived session, so the next model turn can include that session's accumulated transcript. The remedy changes cron/session context policy, adds budgeting, or adds model-call recovery semantics, so it needs maintainer product judgment instead of conservative cleanup or an automated bug-fix lane.

## What This Changes

The issue requests mitigation for recurring cron `systemEvent` jobs that target long-lived sessions and repeatedly send accumulated session context to the model.

## Best Possible Solution

Define a cron context policy that keeps intentional session reuse available while giving lightweight recurring jobs an isolated or bounded-context path and a clear timeout/recovery story for long model calls.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: create a long-lived main or explicit session, schedule a recurring cron `systemEvent` that targets it, and the wake path reopens and prompts the same transcript-backed session. I did not run a live multi-day cost reproduction.

## Solution Assessment

Unclear as proposed: defaulting recurring system events to ephemeral sessions would conflict with documented intentional session reuse. A safer direction is an explicit maintainer-approved context/budget policy plus bounded model-call recovery semantics.
