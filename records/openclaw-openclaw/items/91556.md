## Summary

Keep open: current main already implements the standard MCP tool-list notification refresh, but the fallback for non-notifying servers remains unmerged and is owned by an open closing PR that still needs contract repair, maintainer approval, and real-environment proof.

## What This Changes

Support immediate MCP tool-catalog refresh and add an authenticated per-server reload mechanism for upstream MCP servers that do not emit tool-list change notifications.

## Best Possible Solution

Keep the standard notification invalidation as the canonical path, and land one narrow operator-authorized reload contract that coherently targets active sessions, invalidates the named server transport and catalog, exposes only implemented gateway methods, and includes redacted real-environment proof.

## Reproduction Assessment

Yes, from source with high confidence: current main refreshes compliant servers on tool-list notifications, but exposes no per-server operator reload path for an upstream that does not emit them. A live Composio reproduction was not independently run during this read-only review.

## Solution Assessment

The notification handler is already the narrowest spec-correct default. The linked HTTP fallback is plausible for non-notifying servers, but its gateway descriptor, authorization, targeting, and lifecycle contracts must be corrected and approved before it is the best maintainable solution.
