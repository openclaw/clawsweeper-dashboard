## Summary

Keep open: current main and v2026.6.1 already handle MCP tool-list change notifications, but the requested authenticated reload endpoint/core-tool fallback for servers that do not emit those notifications is still missing and needs maintainer API/security direction.

## What This Changes

Request support for MCP tool-list change notifications and an authenticated way to reload configured MCP server runtimes after external integration changes.

## Best Possible Solution

Keep the shipped notification invalidation path and design one narrow operator-authenticated MCP reload surface that reuses the existing session runtime retirement machinery without broadly allowing mcp.* config mutation through the gateway tool.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main has notification-driven catalog invalidation but no dedicated authenticated MCP server reload endpoint or core reload tool.

## Solution Assessment

Unclear until maintainers choose the surface. The spec-correct notification path is already the best default, while the fallback reload path should be a narrow operator-controlled API rather than a broad mcp.* config mutation escape hatch.
