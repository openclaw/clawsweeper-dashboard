## Summary

Keep open: the PR is a useful implementation direction and the author rebased/validated it locally, but the latest head still has a concrete low-severity security validation bug and is missing the required user-facing changelog entry before merge.

## What This Changes

The PR adds a Fluid `ack`/`executed` bridge-result parser in `src/chat/canvas-render.ts` and tool-card tests that render validated `regenerate.embed` shortcodes as assistant-message canvas previews.

## Best Possible Solution

Keep the PR open and land it after the bridge ref validation mirrors the existing canvas document-id rules, adds negative coverage for `.` and `..`, and includes a single-line Unreleased changelog entry for the Control UI rendering fix.

## Reproduction Assessment

Yes. Against the latest PR head, a Fluid bridge result with `ref: ".."` and matching `[embed ref=".." /]` passes the new regex and builds a URL that normalizes to `/__openclaw__/canvas/index.html`; current main does not reproduce because it lacks the bridge parser.

## Solution Assessment

No. Reusing the existing assistant-message canvas preview path is the right narrow design, but the proposed validator needs to be tightened to the existing canvas document-id contract before this is the best implementation.
