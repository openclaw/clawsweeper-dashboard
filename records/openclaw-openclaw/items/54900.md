## Summary

## What This Changes

Adds a `maxEntryAgeMs` recovery option, `skippedStale` summary accounting, and tests that move old first-attempt delivery-queue entries to `failed/` instead of replaying them on startup.

## Best Possible Solution

Rebase onto current main and implement an explicit, backward-compatible outbound expiry policy shared by startup recovery and reconnect drains, with tests for default no-expiry behavior, opt-in expiry, budget deferral, active-claim re-read, and unknown-send reconciliation.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

No. The recovery boundary is relevant, but the PR's default cutoff and startup-only first-attempt predicate are not the safest maintainable solution without a rebased, configurable policy that preserves current recovery semantics by default.
