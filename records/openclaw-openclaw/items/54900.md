## Summary

Keep open. Current main still lacks an outbound delivery queue age/expiry policy, and this PR remains a plausible focused fix, but the current patch is not merge-ready because it misses budget-deferred never-sent entries, needs rebase/test updates, and lacks contributor real behavior proof.

## What This Changes

The PR adds a default 10-minute stale cutoff for first-attempt pending outbound deliveries during startup recovery, reports `skippedStale`, exposes a `maxEntryAgeMs` recovery option, and adds recovery tests.

## Best Possible Solution

Land or replace this with a rebased outbound delivery expiry policy that covers startup recovery, reconnect drain, and recovery-budget deferrals while preserving legitimate recent crash recovery and coordinating operator-facing TTL semantics through https://github.com/openclaw/openclaw/issues/16555.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

No, not as-is. The shared outbound recovery boundary is the right place, but the first-attempt-only predicate misses budget-deferred never-sent entries and the TTL default/action/config policy still needs maintainer alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The stale gate only matches `retryCount === 0` entries with no `lastAttemptAt`, but current recovery budget deferral calls `failDelivery(...)` for unprocessed entries. That increments `retryCount` and sets `lastAttemptAt` without a send, so an old never-sent entry can bypass this guard on the next startup and still replay out of context.
  - confidence: 0.92
- **[P2] Keep reconnect-drain fixtures within the TTL:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The new default cutoff applies during startup recovery, but current reconnect-drain coverage backdates entries to `enqueuedAt = 1` and `2` only to force ordering. After rebase, those entries would be moved to `failed/` before the reconnect ack/skip path under test unless the fixtures use recent timestamps or disable the cutoff for that scenario.
  - confidence: 0.86
