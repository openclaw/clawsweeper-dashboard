## Summary

Keep open. Current main still has the stale startup-replay behavior, and the linked user report remains open, but this PR is not merge-ready because its stale predicate can be bypassed after recovery-budget deferral and it lacks the required user-facing changelog entry.

## What This Changes

The PR adds a delivery-queue recovery max-age guard, skipped-stale accounting, and recovery tests for old first-attempt queued entries.

## Best Possible Solution

Preserve stale protection based on true attempted-delivery state, add regression coverage for budget-deferred never-sent entries, and include the required changelog entry without expanding into unrelated delivery-queue or channel behavior.

## Reproduction Assessment

Yes. A focused unit test can enqueue an old retryCount=0 entry, run recovery with maxRecoveryMs=0 so it is deferred through failDelivery, then run recovery again after the TTL; the PR predicate can replay it instead of skipping it as stale.

## Solution Assessment

No, not as-is. The stale-guard direction is good, but retryCount and lastAttemptAt are not reliable attempted-send markers because recovery-budget deferral mutates both without sending.
