## Summary

Keep open: the stale-replay bug is real and this PR targets the right shared recovery path, but the branch is not merge-ready because the guard misses budget-deferred never-sent entries, likely breaks adjacent reconnect-drain coverage, and lacks external real-behavior proof.

## What This Changes

Adds a default max-age guard to delivery-queue startup recovery for first-attempt entries and updates recovery tests/accounting.

## Best Possible Solution

Repair the shared outbound recovery policy so old never-attempted entries stay protected even after budget deferral, refresh adjacent recovery/reconnect coverage, and land with contributor-provided real gateway restart proof.

## Reproduction Assessment

Yes, at source level. Current main writes queue entries before sending and startup recovery treats retryCount 0 entries with no lastAttemptAt as eligible without comparing enqueuedAt; the PR still has a source-level bypass through maxRecoveryMs deferral.

## Solution Assessment

No, not as-is. The shared recovery boundary is right, but retryCount and lastAttemptAt are not reliable attempted-send markers because recovery-budget deferral mutates both without sending, and adjacent reconnect coverage needs to be updated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:218-219`
  - body: The stale gate only matches `retryCount === 0` with no `lastAttemptAt`, but recovery-budget deferral calls `failDelivery(...)`, which increments both fields even though no send happened. A stale never-attempted entry deferred by `maxRecoveryMs` can bypass this guard on the next startup and still replay out of context.
  - confidence: 0.92
- **[P2] Refresh stale-sensitive reconnect fixtures:** `src/infra/outbound/delivery-queue-recovery.ts:218-219`
  - body: This default TTL applies to every startup recovery call. Existing reconnect-drain coverage backdates entries to `enqueuedAt = 1` and `2` only to force ordering, so those entries will be moved to `failed/` before the test expects delivery unless the fixtures use recent timestamps or explicitly disable the TTL for that path.
  - confidence: 0.86
