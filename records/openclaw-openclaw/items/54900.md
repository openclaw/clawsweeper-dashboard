## Summary

Keep open: the stale replay bug is valid, but the PR is not correct as-is because recovery-budget deferral can mutate never-sent entries out of the new stale predicate, the default TTL breaks stale-sensitive reconnect coverage, and the required changelog entry is missing.

## What This Changes

The PR adds a max-age guard to startup delivery-queue recovery so old first-attempt pending entries move to `failed/` instead of replaying, with recovery-summary accounting and tests.

## Best Possible Solution

Preserve stale protection based on true attempted-delivery state, keep recovery/reconnect tests using timestamps that match the intended scenario, and document the user-facing fix in the active changelog without adding a public config knob unless maintainers request one.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: startup recovery replays old first-attempt queue entries, and the PR can still replay an old never-sent entry after `maxRecoveryMs` defers it through `failDelivery`; I did not run tests because this review was read-only and `pnpm` is unavailable here.

## Solution Assessment

No, not as-is. The stale-guard direction is narrow, but `retryCount` and `lastAttemptAt` are not reliable attempted-send markers after recovery-budget deferrals, and the adjacent reconnect coverage and changelog still need to be handled.
