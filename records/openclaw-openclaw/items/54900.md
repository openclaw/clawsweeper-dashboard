## Summary

Keep open: the underlying stale replay bug is valid and the PR is close, but the current diff still lets old never-sent entries bypass the new guard after recovery-budget deferral and misses adjacent stale-sensitive coverage plus the required changelog entry.

## What This Changes

The PR adds a default max-age guard to delivery-queue startup recovery so old first-attempt queued entries are moved to `failed/`, adds skipped-stale accounting, and extends recovery tests.

## Best Possible Solution

Use a channel-agnostic attempted-delivery marker or non-mutating budget deferral so stale never-sent entries remain discardable across restarts, then cover the recovery and reconnect paths and document the fix in the changelog.

## Reproduction Assessment

Yes. Source inspection of current main shows startup recovery calls `recoverPendingDeliveries`, queued entries are persisted before send with `retryCount: 0` and no `lastAttemptAt`, and current eligibility replays that state without an age cutoff; the PR-specific regression follows from budget deferral calling `failDelivery` without a send.

## Solution Assessment

No, not as-is. The stale-TTL direction is narrow, but basing attempted delivery solely on `retryCount` and `lastAttemptAt` is unreliable because recovery-budget rollover mutates both fields before any send attempt.
