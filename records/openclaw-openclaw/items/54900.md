## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Make stale expiry opt-in before changing recovery defaults:** `src/infra/outbound/delivery-queue-recovery.ts:216-217`
  - body: This default makes every startup quarantine first-attempt entries older than 10 minutes, but the canonical TTL policy asks to preserve current no-TTL behavior unless operators opt in. Existing users with long downtime or slow recovery can lose queued messages into `failed/` during upgrade without choosing that policy.
  - confidence: 0.88
- **[P1] Run expiry after the recovery claim and re-read:** `src/infra/outbound/delivery-queue-recovery.ts:219-225`
  - body: The new stale check acts on the initial `pending` snapshot. Current main re-reads the queue file after claiming so startup and reconnect drains do not act on entries that were already acked or whose state changed; the expiry check needs to run on that current entry after the claim.
  - confidence: 0.84
- **[P2] Preserve expiry for budget-deferred first attempts:** `src/infra/outbound/delivery-queue-recovery.ts:218-219`
  - body: The predicate only matches `retryCount === 0` with no `lastAttemptAt`, but current budget deferral uses `failDelivery`, which increments retry state even when no send was attempted. An old never-sent entry deferred by the recovery budget will bypass this stale gate on the next startup.
  - confidence: 0.8
