## Summary

Keep this PR open for maintainer handling: the stale-replay problem still exists on current main, but this branch is conflicting, lacks contributor real-behavior proof, and the proposed age gate misses important recovery states on the current delivery-queue path.

## What This Changes

The PR adds a default 10-minute stale cutoff for first-attempt outbound delivery recovery, reports skipped stale entries, exposes a `maxEntryAgeMs` recovery option, and adds recovery tests.

## Best Possible Solution

Land or replace this with a rebased outbound delivery expiry policy that handles startup recovery, budget deferrals, reconnect drains, and uncertain-send reconciliation while preserving legitimate recent crash recovery.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

No, not as-is. The shared outbound recovery boundary is right, but the first-attempt-only predicate misses budget-deferred never-sent entries, can bypass uncertain-send reconciliation, and still needs TTL/default/action policy alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The stale gate only matches `retryCount === 0` entries with no `lastAttemptAt`, but recovery-budget deferral calls `failDelivery(...)` for unprocessed entries. That increments `retryCount` and sets `lastAttemptAt` without a send, so an old never-sent entry can bypass this guard on the next startup and still replay out of context.
  - confidence: 0.92
- **[P2] Keep reconnect-drain fixtures within the TTL:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The new default cutoff applies during startup recovery, but current reconnect-drain coverage backdates entries to `enqueuedAt = 1` and `2` only to force ordering. After rebase, those entries would be moved to `failed/` before the reconnect ack/skip path under test unless the fixtures use recent timestamps or disable the cutoff for that scenario.
  - confidence: 0.86
- **[P2] Reconcile uncertain sends before applying the age cutoff:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: This first-attempt predicate also matches old entries with `recoveryState` such as `send_attempt_started` or `unknown_after_send`. On current main those entries go through adapter reconciliation before any replay; moving them to `failed/` solely by age can skip a successful already-sent reconciliation and its commit hooks.
  - confidence: 0.84
