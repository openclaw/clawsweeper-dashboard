Keep open: the stale replay bug is still valid on current main, but this branch needs repair before merge because its stale predicate misses budget-deferred never-sent entries, leaves stale-sensitive reconnect coverage unadjusted, and lacks the required changelog entry.

## What This Changes

The PR adds a `maxEntryAgeMs` stale-entry guard to delivery-queue startup recovery, exposes skipped-stale accounting, and extends recovery tests.

## Best Possible Solution

Rebase or replace the branch, preserve stale protection based on true attempted-delivery state, refresh reconnect coverage, add the changelog entry, and validate the outbound changed lane before merge.

## Reproduction Assessment

Yes. Source inspection shows current main writes queue entries before sending, startup recovery replays pending entries, and first-attempt entries have no age cutoff; tests were not run because `pnpm` is unavailable.

## Solution Assessment

No, not as-is. The max-age direction is sound, but using `retryCount` and `lastAttemptAt` as attempted-send markers misses recovery-budget deferrals that mutate both without sending.
