## Summary

Keep open: the stale replay bug is real, but the patch is not merge-ready because the stale guard relies on `retryCount`/`lastAttemptAt` even though recovery-budget deferral mutates both without a send, and the new default TTL leaves adjacent tests and changelog coverage unaddressed.

## What This Changes

The branch adds a startup recovery max-age guard for old first-attempt delivery-queue entries, exposes skipped-stale accounting, and extends recovery tests.

## Best Possible Solution

Preserve stale protection based on true attempted-delivery state, update stale-sensitive recovery/reconnect coverage, and add the required changelog entry without expanding into channel-specific behavior or a new user config surface unless maintainers ask for one.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main replays old first-attempt queue entries on startup, and the PR can still replay an old never-sent entry after `maxRecoveryMs` defers it through `failDelivery`. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No, not as-is. The stale-guard direction is good, but `retryCount` and `lastAttemptAt` are not reliable attempted-send markers because recovery-budget deferral mutates both without sending, and the adjacent reconnect test fixtures/changelog still need to be handled.

  - confidence: 0.86

  - body: This changes user-visible gateway delivery behavior by dropping stale queued messages instead of replaying them after restart, but the PR only changes recovery code and tests. OpenClaw policy requires a `CHANGELOG.md` entry for user-facing fixes before merge.
