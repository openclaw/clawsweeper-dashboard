## Summary

Keep open. The PR targets a real outbound recovery problem still present on current main, but the patch is not correct as-is because the stale predicate can be bypassed after recovery-budget deferral, adjacent reconnect-drain fixtures are stale-sensitive, and contributor real-behavior proof is still missing.

## What This Changes

Adds a default 10-minute max-age guard to startup delivery-queue recovery for first-attempt entries and updates recovery summary/tests.

## Best Possible Solution

Repair or replace this PR with a shared outbound recovery policy that distinguishes stale never-sent entries from legitimate retries, covers budget deferral and reconnect drain, and leaves any operator-facing TTL/config semantics aligned with the canonical TTL issue.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and the gateway startup path calls recovery with the real outbound delivery function.

## Solution Assessment

No, not as-is. The shared recovery boundary is the right place, but the attempted-send predicate must survive budget deferral and adjacent reconnect coverage must be refreshed before this can land.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:218-219`
  - body: The new stale gate only matches retryCount === 0 with no lastAttemptAt, but recovery-budget deferral calls failDelivery(...), which increments both fields even though no send happened. A stale never-attempted entry deferred by maxRecoveryMs can bypass this guard on the next startup and still replay out of context.
  - confidence: 0.92
- **[P2] Refresh stale-sensitive reconnect fixtures:** `src/infra/outbound/delivery-queue-recovery.ts:218-219`
  - body: The default TTL applies to every startup recovery call. Existing reconnect-drain coverage backdates entries to enqueuedAt = 1 and 2 only to force ordering, so those entries will be moved to failed/ before the test expects delivery unless the fixtures use recent timestamps or explicitly disable the TTL for that path.
  - confidence: 0.86
