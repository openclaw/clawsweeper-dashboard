## Summary

Keep open. The PR targets a real, source-proven stale replay problem, but the branch is now conflicting with current main, the patch still has correctness blockers around budget-deferred entries and reconnect-drain coverage, and contributor real behavior proof is still missing.

## What This Changes

The PR adds a default 10-minute stale cutoff for first-attempt pending delivery entries during startup recovery, moves matching entries to `failed/`, reports `skippedStale`, and adds recovery tests.

## Best Possible Solution

Land or replace this with a shared outbound recovery expiry policy that works with current recovery state, preserves legitimate recent crash recovery, survives budget deferral and reconnect-drain paths, and coordinates default/config semantics with https://github.com/openclaw/openclaw/issues/16555.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as immediately eligible and gateway startup calls that recovery path.

## Solution Assessment

No, not as-is. The shared recovery boundary is the right place, but the first-attempt-only predicate misses budget-deferred never-sent entries, the branch needs a current-main rebase, and the TTL/default policy needs maintainer alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The stale gate only matches `retryCount === 0` with no `lastAttemptAt`, but the recovery budget path defers unprocessed entries through `failDelivery(...)`, which increments `retryCount` and sets `lastAttemptAt` without a send. A stale never-sent entry can be deferred once and then bypass this guard on the next startup, so the replay problem still survives large-backlog recovery.
  - confidence: 0.92
- **[P2] Keep reconnect-drain fixtures within the TTL:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The new default cutoff applies to startup recovery, but current reconnect-drain coverage backdates queued entries to `enqueuedAt = 1` and `2` only to force ordering. After rebase, those entries would be moved to `failed/` before the reconnect ack/skip path under test, so update the fixtures or explicitly disable the cutoff for that scenario.
  - confidence: 0.86
