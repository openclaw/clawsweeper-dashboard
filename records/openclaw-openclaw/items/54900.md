## Summary

Keep open. Current main still lacks an age cutoff for stale first-attempt delivery-queue recovery, but this PR is not merge-ready because its predicate misses budget-deferred never-sent entries, it needs rebase-aware reconnect coverage, and contributor real behavior proof is missing.

## What This Changes

The PR adds a default 10-minute startup recovery cutoff for first-attempt pending delivery entries and updates recovery summary/tests.

## Best Possible Solution

Land or replace this with a shared outbound recovery expiry policy that preserves legitimate recent crash recovery, survives budget deferral and reconnect-drain paths, and coordinates config/default semantics with https://github.com/openclaw/openclaw/issues/16555.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as immediately eligible and gateway startup calls that recovery path.

## Solution Assessment

No, not as-is. The shared recovery boundary is the right place, but the stale predicate needs to survive budget deferrals, reconnect-drain coverage needs updating after rebase, and TTL/default semantics need maintainer alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The stale gate only matches entries with retryCount === 0 and no lastAttemptAt, but deferRemainingEntriesForBudget calls failDelivery(...) for unprocessed entries when maxRecoveryMs is hit. That increments retryCount/lastAttemptAt without a send, so a stale never-sent entry deferred once can bypass this guard and replay on the next startup.
  - confidence: 0.92
- **[P2] Keep reconnect-drain fixtures within the TTL:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The new default cutoff applies to all startup recovery calls. Current reconnect-drain coverage backdates queued entries to enqueuedAt = 1 and 2 only to force ordering, so after this PR is rebased those entries will be moved to failed before the reconnect ack/skip path the test is asserting; update those fixtures or explicitly disable the TTL for that scenario.
  - confidence: 0.86
