## Summary

Keep open. The stale-replay problem is still source-provable on current main and this PR is a focused attempt, but the patch is outdated/conflicting, has blocking recovery-policy bugs, and still lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds a default 10-minute stale-entry cutoff to startup delivery-queue recovery for first-attempt pending entries and updates recovery summary/tests.

## Best Possible Solution

Land or replace this with a shared outbound recovery expiry policy that preserves legitimate retry/reconnect recovery, survives recovery-budget deferral, and coordinates any operator-facing TTL config with https://github.com/openclaw/openclaw/issues/16555.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

No, not as-is. The shared recovery boundary is the right place, but the stale predicate fails after budget deferral, stale-sensitive reconnect coverage must be updated, and TTL/default semantics need maintainer alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The stale gate only matches retryCount === 0 with no lastAttemptAt, but recovery-budget deferral calls failDelivery(...) on unprocessed entries, which increments both fields without a send attempt. A stale never-sent entry deferred by maxRecoveryMs can later bypass this guard and replay out of context.
  - confidence: 0.92
- **[P2] Refresh reconnect fixtures for the default TTL:** `src/infra/outbound/delivery-queue-recovery.ts:219`
  - body: The default TTL applies to every startup recovery call. Current reconnect-drain coverage backdates queue entries to enqueuedAt = 1 and 2 only to force ordering, so those entries would be moved to failed before the test expects delivery unless the fixtures use recent timestamps or explicitly disable the TTL for that path.
  - confidence: 0.86
