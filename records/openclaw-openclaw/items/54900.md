## Summary

Keep open. Current main still has the stale startup-replay behavior, and the linked user report remains open, but this PR is not merge-ready because its stale predicate can be bypassed after recovery-budget deferral and it lacks the required user-facing changelog entry.

## What This Changes

The PR adds a `maxEntryAgeMs` stale-entry guard, `skippedStale` recovery accounting, and recovery tests for old first-attempt delivery-queue entries.

## Best Possible Solution

Preserve stale protection based on true attempted-delivery state, not only `retryCount` and `lastAttemptAt`. The durable fix should keep never-sent budget-deferrals distinguishable from real failed sends, add regression coverage for that path, and include the required changelog entry while leaving broader TTL configuration to an explicit maintainer decision if needed.

## Reproduction Assessment

Yes. A focused unit test can enqueue an old `retryCount=0` entry, run recovery with `maxRecoveryMs=0` so it is deferred through `failDelivery`, then run recovery again after the TTL; the PR predicate will see `retryCount > 0`/`lastAttemptAt` and can replay it instead of skipping it as stale.

## Solution Assessment

No, not as-is. The stale-guard direction is good, but the proposed predicate uses fields that current recovery-budget deferral mutates without a send attempt; the safer solution needs an attempted-send marker or a deferral path that preserves never-attempted status.
