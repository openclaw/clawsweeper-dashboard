## Summary

Keep open. The PR targets a real source-reproducible stale replay path, but it is not merge-ready because real behavior proof is missing and the stale predicate misses never-sent entries that were mutated by recovery-budget deferral.

## What This Changes

Adds a default 10-minute max-age guard to startup delivery-queue recovery for first-attempt entries and updates recovery summary/tests.

## Best Possible Solution

Land or replace this with a shared outbound recovery stale policy that distinguishes never-sent stale entries from legitimate retries, preserves reconnect/live-send ownership behavior, and aligns any operator-facing TTL semantics with the canonical TTL issue.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

No, not as-is. The shared recovery boundary is the right place, but the attempted-send predicate must survive budget deferral and adjacent reconnect/startup coverage must be refreshed before this can land.

  - body: The new stale gate only matches retryCount === 0 with no lastAttemptAt, but recovery-budget deferral calls failDelivery(...) for unprocessed entries, incrementing both fields without any send attempt. A stale never-sent entry deferred by maxRecoveryMs can bypass this guard on the next startup and still replay out of context.
  - confidence: 0.92

  - confidence: 0.86
