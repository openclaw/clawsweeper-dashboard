## Summary

Keep open: the linked cron timestamp bug still appears valid on current main, but this PR should not merge as-is because it is stale against the split-store cron contract and mixes unrelated secret-scanning/iOS Fastlane changes into the cron fix.

## What This Changes

The PR adds load-time legacy cron timestamp backfills and extends the cron regression test, while also changing secret-scanning baseline and iOS Fastlane allowlist metadata.

## Best Possible Solution

Land a narrow current-main cron repair that backfills finite non-negative `createdAtMs` in job definitions/load output and `updatedAtMs` in the runtime sidecar/load output, keeps `cron.list` and `listPage` decodable, and leaves secret-scanning/iOS release files untouched.

## Reproduction Assessment

Yes, source inspection gives high confidence: a legacy job without timestamps reaches `cron.list`, current `loadCronStore` only backfills `updatedAtMs`, and both the gateway schema and macOS model require `createdAtMs` and `updatedAtMs`. No, not as currently proposed. Load-time repair is the right direction, but this branch asserts runtime timestamps against raw `jobs.json` despite the current sidecar contract and includes unrelated secret-scanning/Fastlane changes; a split-store-aware replacement is safer.
