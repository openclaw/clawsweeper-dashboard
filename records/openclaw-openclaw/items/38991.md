## Summary

Keep open: the underlying cron timestamp contract gap is still source-reproducible on current main, but this branch is not mergeable as-is because it is stale against the split-store contract, includes unrelated secret-handling metadata, and lacks after-fix real behavior proof.

## What This Changes

The PR backfills missing legacy cron job timestamps during cron service store load, extends a cron regression test, and refreshes secret-scanning/Fastlane metadata.

## Best Possible Solution

Land a narrow current-main cron repair that backfills finite non-negative `createdAtMs` for legacy jobs and keeps `updatedAtMs` in the runtime sidecar/load output, with regression coverage and no iOS or secret-scanning changes.

## Reproduction Assessment

Yes, by source inspection. A legacy job can still load without `createdAtMs`, `cron.list` returns loaded jobs, and the gateway/macOS models require both timestamp fields; this read-only review did not execute a live repro.

## Solution Assessment

No, not as currently proposed. Load-time repair is the right direction, but this branch is stale against the `jobs-state.json` sidecar contract, asserts runtime timestamps in raw `jobs.json`, and includes unrelated secret-handling metadata.
