## Summary

Keep open: the legacy cron timestamp bug is still source-reproducible at the gateway contract boundary, but this PR is not currently mergeable because it patches the service loader instead of the canonical split-store loader, asserts raw `jobs.json` state that current main moved to a sidecar, includes unrelated Fastlane/secret baseline edits, and lacks real behavior proof.

## What This Changes

The branch adds service-load timestamp backfill and regression assertions for legacy cron jobs, plus unrelated Fastlane secret-scanner and secret baseline edits.

## Best Possible Solution

Rework this as a narrow cron-store fix in `src/cron/store.ts` that synthesizes finite non-negative missing `createdAtMs` and `updatedAtMs`, persists config/state in the split-store shape, and drops the unrelated iOS/secret-scanning edits.

## Reproduction Assessment

Yes. By source inspection, a legacy job missing `createdAtMs` can pass through `loadCronStore()` with only `updatedAtMs` repaired, `cron.list` returns the loaded jobs, and the protocol schema requires both timestamps.

## Solution Assessment

No, not as proposed. Load-time repair is the right direction, but it should live in the canonical split-store loader with sidecar-aware assertions and no unrelated release secret-scanning changes.
