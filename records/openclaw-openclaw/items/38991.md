## What This Changes

The PR backfills missing legacy cron job timestamps during cron service load, extends a cron regression test, and also changes Fastlane and secret-scanning metadata.

## Best Possible Solution

Land a narrow current-main cron repair that backfills finite non-negative `createdAtMs` in the store loader, keeps `updatedAtMs` and runtime state in `jobs-state.json`, and removes unrelated secret metadata changes.

## Reproduction Assessment

## Solution Assessment

No, not as currently proposed. Load-time repair is the right direction, but the branch is stale against sidecar persistence and bundles unrelated secret-handling changes; the safer fix is a narrow rebase onto the current store contract.
