## Summary

Keep open: the cron compatibility bug is source-reproducible, but this PR is stale against the current split-store implementation, includes unrelated iOS/secret-scanning changes, and still lacks contributor real behavior proof.

## What This Changes

The PR backfills missing legacy cron job timestamps during service load, extends a cron regression test, and refreshes secret-scanning/Fastlane metadata.

## Best Possible Solution

Rework this into a narrow current-main cron repair in `loadCronStore()` and the split-state path: synthesize finite non-negative `createdAtMs`, keep runtime fields in `jobs-state.json`, and drop the unrelated iOS/secret metadata.

## Reproduction Assessment

Yes, by source inspection. A legacy job missing `createdAtMs` can pass through the current load/list path while the public cron schema and macOS model require that field; this read-only review did not run a live repro.

## Solution Assessment

No, not as currently proposed. Load-time repair is the right direction, but the patch targets the old service-store shape, asserts runtime fields in raw `jobs.json`, and includes unrelated secret-handling changes.
