## Summary

Keep open. The underlying cron timestamp repair is still meaningful because current main does not backfill a missing `createdAtMs` before `cron.list`, but this PR is not mergeable as-is: it targets stale pre-split store assumptions, includes unrelated secret-scanning/Fastlane changes, and lacks real behavior proof from the contributor.

## What This Changes

The PR backfills missing legacy cron job timestamps during cron service store load, extends a regression test, and also changes Fastlane secret-scan allowlist metadata plus the secrets baseline.

## Best Possible Solution

Rework this into a narrow current-main cron repair that backfills finite non-negative `createdAtMs` in `loadCronStore()`, keeps `updatedAtMs` and runtime state in `jobs-state.json`, removes the iOS/secret-scanning changes, and includes focused regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection. A legacy job without `createdAtMs` can flow through `loadCronStore()` and `cron.list` while the protocol and macOS models require that field; this read-only review did not run a live repro.

## Solution Assessment

No, not as currently proposed. Load-time repair is the right direction, but it should target the current split-store loader and sidecar contract and drop the unrelated Fastlane/baseline changes.
