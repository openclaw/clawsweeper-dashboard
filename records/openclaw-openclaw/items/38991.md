## Summary

Keep open. The reported cron timestamp bug is still source-reproducible in the gateway path, but this PR targets the wrong layer for current main, asserts the old monolithic store shape, lacks real after-fix proof, and mixes unrelated iOS Fastlane/secret-scanning changes into the cron fix.

## What This Changes

The branch adds service-load timestamp backfill and regression assertions for legacy cron jobs, plus unrelated Fastlane secret-scanner and secret baseline edits.

## Best Possible Solution

Rework this as a narrow cron-store fix in `src/cron/store.ts` that synthesizes finite non-negative missing `createdAtMs` and `updatedAtMs`, persists config/state in the split-store shape, and keeps unrelated iOS/secret-scanning edits out.

## Reproduction Assessment

Yes, by source inspection: a legacy job missing `createdAtMs` passes through `loadCronStore()` with only `updatedAtMs` repaired, `cron.list` returns those jobs, and the protocol schema requires both timestamps. I did not run a live repro in this read-only review.

## Solution Assessment

No, not as proposed. Load-time repair is the right direction, but it should live in the canonical split-store loader with sidecar-aware assertions and no unrelated release secret-scanning changes.
