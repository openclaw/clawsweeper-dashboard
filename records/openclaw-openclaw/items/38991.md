## Summary

Keep open: current main mitigates the linked macOS whole-tab decode failure, but source inspection still shows a legacy cron row without `createdAtMs` can pass through the cron loader while the protocol and Swift models require it. This PR points at a real compatibility gap, but the branch is not mergeable as-is because the repair sits at the wrong boundary, the persistence assertion conflicts with the split-store contract, real behavior proof is missing, and unrelated Fastlane secret-scanning metadata is included.

## What This Changes

The PR backfills missing cron job timestamps during service load, extends a cron regression test, and also changes iOS Fastlane secret-scanner metadata plus the generated secret baseline.

## Best Possible Solution

Revise or replace this branch with a narrow cron store-loader repair that synthesizes finite `createdAtMs` and `updatedAtMs` at the canonical load/migration boundary, persists according to the split-store contract, and drops the unrelated secret-scanning changes.

## Reproduction Assessment

Yes for source-level reproduction: a legacy `jobs.json` row without `createdAtMs` can still load because current main only backfills `updatedAtMs`, while `CronJobSchema` and the Swift `CronJob` model require both timestamps. I did not run a live macOS UI repro in this read-only review.

## Solution Assessment

No. Load-time timestamp repair is the right direction, but this branch puts it in `service.ensureLoaded()` instead of the canonical store loader, asserts runtime state in the wrong persistence file, and bundles unrelated secret-scanning metadata.
