## Summary

Keep this PR open. Current main still has the legacy cron timestamp gap the PR is trying to address, but the branch is stale against the split cron store and mixes the cron fix with unrelated security-sensitive Fastlane/secret-baseline changes, so it needs maintainer follow-up rather than cleanup closure.

## What This Changes

The branch adds cron load-time timestamp backfill and regression assertions for legacy jobs missing `createdAtMs`/`updatedAtMs`, while also modifying `.secrets.baseline` and an iOS Fastlane secret-scanner allowlist comment.

## Best Possible Solution

Land a split-store-aware cron repair that backfills finite non-negative `createdAtMs` for legacy job definitions and `updatedAtMs` for runtime state during load, keeps `cron.list`/`listPage` decodable, persists only actual legacy repairs, and asserts `jobs.json` versus `jobs-state.json` separately.

## Reproduction Assessment

Yes. A legacy cron job without timestamp fields is already modeled by the regression fixture, and current main’s load/list path still returns stored jobs without creating `createdAtMs` while the gateway protocol and macOS model require it.

## Solution Assessment

No, not as currently proposed. Load-time repair is the right direction, but the PR’s persisted-store assertions target the old monolithic layout and the branch includes unrelated secret-scanning changes; the safer fix is a narrow rebase or replacement against the current split store.
