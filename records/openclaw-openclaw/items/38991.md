## Summary

Keep open: the cron timestamp repair is still useful on current main, but this PR is not merge-ready because it is stale against the split cron store contract, lacks after-fix real behavior proof, and includes unrelated secret-handling changes.

## What This Changes

The PR adds load-time timestamp backfill for legacy cron jobs, extends a cron regression test, and also changes secret-scanning/Fastlane metadata.

## Best Possible Solution

Land a narrow current-main cron repair that backfills finite non-negative `createdAtMs` in the config path, keeps `updatedAtMs` in the state sidecar/load output, and drops unrelated secret metadata changes.

## Reproduction Assessment

Yes for the remaining gateway contract gap by source inspection: legacy jobs missing `createdAtMs` can still load, `cron.list` returns loaded jobs, and the protocol requires that field. The exact original whole-tab macOS crash is now mitigated by lossy decoding, but malformed jobs are skipped rather than repaired.

## Solution Assessment

No, not as currently proposed. Load-time repair is the right direction, but the PR must be rebased onto the split-store contract, assert runtime timestamps through `loadCronStore()` or the sidecar, and remove the unrelated secret-handling changes.
