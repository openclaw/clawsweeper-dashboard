The PR also needs real behavior proof and a clearer analytics/privacy contract before maintainers can accept the feature.

The branch adds an in-memory image-generation usage recorder, wires it into `image_generate`, adds unit tests, and commits a root Argus provenance file.

## Best Possible Solution

Record image-generation metadata exactly once from the actual completed job path, preferably through the existing diagnostics or session-usage contract with bounded/redacted fields and focused coverage.

Yes for the PR defects by source inspection: the async branch adds a second job execution path, and the added test contains `await import` in a non-async callback. I did not run tests because this review is read-only.

## Solution Assessment

No: this implementation is not the best current solution because it changes execution ordering and creates a parallel in-memory analytics store. A safer path is to record once after completion through a bounded diagnostics or session-usage contract.
