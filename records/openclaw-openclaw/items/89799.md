## Summary

The reported Windows Node 24.1 startup hang is valid, current main still exposes the affected packaged path, and the PR applies a coherent Windows-only mitigation across both entry layers with strong native proof. It should remain open for maintainer review because the patch re-enables compile cache at Node 24.15 based on upstream feature stabilization and release-CI practice, not evidence that the Windows deadlock is fixed at that version.

## What This Changes

The PR disables Node’s module compile cache for packaged Windows startup on Node 24.0–24.14, clears inherited cache through a one-shot respawn, and adds entry-helper and packaged-launcher regression coverage.

## Best Possible Solution

Keep one identical Windows packaged-startup mitigation in both entry layers, but re-enable compile cache only at a version proven by native packed CLI boundary testing or a specific upstream fix; otherwise disable it for all Windows Node 24 releases and revisit later.

## Reproduction Assessment

Yes. The report gives a deterministic Windows npm-global Node 24.1 path, and the PR supplies after-fix packed-install proof for the affected CLI, doctor, and gateway commands.

## Solution Assessment

Not fully. Disabling compile cache at the packaged startup boundary is the narrow root-cause mitigation, but Node 24.15 is not yet proven as the earliest safe re-enable point.
