## Summary

Keep this PR open, but block merge on the Node 24.15 cutoff: the supplied native Windows evidence strongly proves the mitigation on Node 24.1, while neither the proof nor the cited upstream changes establishes that the startup deadlock stops at Node 24.15.

## What This Changes

The PR disables or clears Node’s compile cache during Windows packaged CLI startup on Node 24.0–24.14 and adds focused entry and launcher regression coverage.

## Best Possible Solution

Keep one equivalent launcher and TypeScript-entry rule that disables compile cache for the proven affected Windows packaged runtime family, and re-enable it only after native packed-CLI or deadlock-specific upstream evidence establishes a safe version boundary.

## Reproduction Assessment

Yes: the PR supplies a native Windows npm-global Node 24.1 packed-install path and after-fix output for version, doctor, and gateway startup. It does not reproduce the proposed safe re-enable boundary at Node 24.15.

## Solution Assessment

No in its current cutoff form: disabling inherited and OpenClaw-enabled compile cache at the shared startup boundary is appropriate, but Node 24.15 should not be treated as safe without deadlock-specific runtime or upstream evidence.
