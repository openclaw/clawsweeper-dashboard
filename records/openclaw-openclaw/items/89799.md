- `P1`: The PR targets a user-facing Windows startup hang that can block `openclaw --version`, doctor, and gateway startup for affected users.
- `merge-risk: 🚨 compatibility`: The patch intentionally changes the packaged compile-cache default for an existing Windows Node 24 user class and still needs cutoff acceptance.
- `merge-risk: 🚨 availability`: If inherited compile cache remains active, the affected CLI startup path can still hang before command dispatch.

## Summary

Keep open: the patch is the right narrow direction and now includes native Windows proof, but it still misses inherited `NODE_COMPILE_CACHE` or an already-active Node compile cache, so some affected packaged Windows Node 24 users can still hit the startup hang before the runtime import.

## What This Changes

The PR disables OpenClaw's automatic packaged compile-cache startup path for Windows Node 24.0-24.14 in the launcher and typed entry helper, with focused regression tests and Windows npm-global proof.

## Best Possible Solution

Fully enter a no-cache startup mode for Windows packaged Node 24.0-24.14, including inherited `NODE_COMPILE_CACHE` and active `module.getCompileCacheDir()`, while preserving compile cache everywhere else and documenting maintainer acceptance of the 24.15 boundary.

## Reproduction Assessment

No full local Windows hang repro was run here, but the source path is high-confidence: current main enables compile cache before loading the CLI, the linked report and PR body describe the native Windows Node 24.1.0 hang, and a local Node probe confirms inherited `NODE_COMPILE_CACHE` keeps compile cache active.

## Solution Assessment

No, this is a good narrow mitigation but not yet the best complete fix. It should also clear or respawn away inherited active compile cache on affected Windows Node 24 before importing the runtime, and maintainers should explicitly accept the 24.15 cutoff.
