## Summary

Current main already supports the central request through session labels: `sessions.patch` accepts and persists `label`, the SDK documents `session.patch({ label })`, and the Control UI display resolver prefers `label` over generated session-key text. This appears current-main-only rather than shipped in `v2026.5.27`.

## What This Changes

The issue requests a custom session label or rename surface via CLI, Control UI, or API so multiple conversations are easier to distinguish.

## Best Possible Solution

Keep the current `sessions.patch` label API and UI label display behavior; any separate inline rename UI or CLI sugar should be tracked as new UX work if maintainers still want it.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug report. Source and test inspection show current main already supports labels through `sessions.patch` and displays them in the Control UI.

## Solution Assessment

Yes. The current implementation solves the requested “via CLI, Control UI, or API” path through the API/SDK and uses those labels in the UI; CLI or inline UI rename affordances can be separate polish.
