## Summary

Keep open: current main still only allows CSV/Markdown validated-text host reads, and this PR still leaves an XML-sniffed path that can bypass full text validation before merge.

## What This Changes

The PR extends host-read media validation and tests to accept validated HTML, XML, and CSS document attachments, adds outbound HTML coverage, and updates the changelog.

## Best Possible Solution

Keep HTML/XML/CSS/XML on the validated-text alias path only, remove text/XML MIME values from the buffer-verified document allowlist, and add a regression for XML-prologue content with a binary tail on a .txt or extensionless host-read path.

## Reproduction Assessment

Yes. On the PR head, a host-local .txt or extensionless file beginning with '<?xml ' can be sniffed as application/xml, classified as a document, and accepted by the new allowlist before isValidatedHostReadText checks the full buffer.

## Solution Assessment

No. The requested behavior is reasonable, but the maintainable fix is narrower: declared HTML/XML/CSS/XML documents should pass only after sniff compatibility and full-buffer text validation, not through the buffer-verified document allowlist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep XML out of the buffer-verified allowlist:** `src/media/web-media.ts:106`
  - body: Adding `application/xml` here lets XML-sniffed host reads return from the document allowlist before `isValidatedHostReadText` runs when the filename is not one of the validated aliases, such as `.txt` or extensionless content. `file-type@22.0.1` sniffs a `<?xml ` prefix as `application/xml`, so a file with an XML prologue and binary tail can bypass the text guard this PR is trying to enforce.
