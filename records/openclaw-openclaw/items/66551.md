## Summary

Keep open. Current main does not implement the requested host-read HTML/XML/CSS behavior, and the latest PR head still has a concrete security/correctness blocker in the host-read guard plus only test/CI-style proof from the contributor.

## What This Changes

The PR extends the shared host-read media guard and tests so validated HTML, XML, and CSS local document attachments can be sent while disguised binary payloads remain rejected.

## Best Possible Solution

Keep HTML/XML/CSS/XML on the declared validated-text path only, remove text/XML MIME values from the buffer-verified document allowlist, and add a regression for XML-prologue content with a binary tail on a .txt or extensionless host-read path.

## Reproduction Assessment

Yes by source inspection. Current main only allows CSV/Markdown validated text aliases, and on the PR head a .txt or extensionless host-local file beginning with an XML declaration can be sniffed as application/xml and accepted before full text validation.

## Solution Assessment

No. The requested direction is plausible, but the latest PR is not the safest implementation because plain-text XML is also placed in the buffer-verified document allowlist instead of being limited to the validated-text branch.
