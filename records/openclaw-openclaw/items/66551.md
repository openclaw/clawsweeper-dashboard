## Summary

Keep this PR open: current `main` and the latest shipped release still lack the requested HTML/XML/CSS host-read allowance, but the current PR head still has a host-read policy/security blocker and lacks after-fix real behavior proof.

## What This Changes

The PR extends host-read media validation to allow HTML, XML, and CSS local document attachments, updates related media/outbound tests, and adds a changelog entry.

## Best Possible Solution

Land a narrowed host-read text-document policy that keeps HTML/XML/CSS/XML on the validated-text branch, removes XML from the buffer-verified document allowlist, adds non-`.xml` XML-prologue coverage, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main maps HTML/XML/CSS MIME types but the host-read validated-text aliases only include CSV and Markdown, so host-local `.html`, `.xml`, and `.css` reads hit the rejection path. I did not run tests because this review is read-only.

## Solution Assessment

No. The PR's direction is plausible, but `application/xml` should not be in the buffer-verified document allowlist because `file-type` can sniff XML on non-alias paths before full text validation runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Keep XML out of the buffer-verified allowlist:** `src/media/web-media.ts:106`
  - body: Adding `application/xml` here lets any host-local path that starts with `<?xml `, including `.txt` or extensionless files, return from the sniffed document allowlist before `isValidatedHostReadText` runs. Keep XML only on the validated-text alias path and add a non-`.xml` XML-prologue regression so `.txt` remains blocked as promised.
  - confidence: 0.92
