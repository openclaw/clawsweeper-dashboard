## Summary

Keep open: current main still lacks this behavior and the PR is useful, but the current head is not merge-ready because `application/xml` remains in the buffer-verified allowlist and the external PR still lacks artifact-backed real behavior proof.

## What This Changes

The PR changes the shared host-read media guard, media/outbound tests, and changelog to allow validated HTML/XML/CSS document attachments while rejecting renamed binary payloads.

## Best Possible Solution

Land a narrowed host-read text-document policy that keeps HTML/XML/CSS/XML on the validated-text path only, adds XML-prologue non-alias regression coverage, and coordinates the overlapping text-document policy before merge.

## Reproduction Assessment

Yes, by source inspection: current main maps these extensions but the host-read guard only validates CSV/Markdown text aliases, so local HTML/XML/CSS document sends remain rejected. The PR-head XML bypass is also source-reproducible from the allowlist and return order.

## Solution Assessment

No. The direction is plausible, but `application/xml` should not be in the buffer-verified allowlist; XML and the other text document types should stay behind `isValidatedHostReadText()` with focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Keep XML on the validated-text path:** `src/media/web-media.ts:106`
  - body: Adding `application/xml` to `HOST_READ_ALLOWED_DOCUMENT_MIMES` lets a host-local `.txt` or extensionless file beginning with an XML prologue return from the sniffed document allowlist before `isValidatedHostReadText()` checks the full buffer. Remove XML from the buffer-verified set and cover the non-`.xml` XML-prologue case so XML only passes through the validated-text branch.
  - confidence: 0.92
