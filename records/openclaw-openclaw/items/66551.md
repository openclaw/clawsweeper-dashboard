## Summary

Keep open. The PR addresses a real host-read media gap on current main, but the current head is not mergeable as-is because XML-prologue content on non-alias paths can still bypass the intended full-buffer text validation, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR extends the host-read media guard and tests so validated local HTML, XML, and CSS attachments can be sent while renamed binary payloads remain blocked.

## Best Possible Solution

Land a narrowed host-read text-document policy that keeps HTML/XML/CSS/XML on the validated-text branch only, adds XML-prologue non-alias coverage, and coordinates with the overlapping text-document PR before landing.

## Reproduction Assessment

Yes, by source inspection. Current main lacks HTML/XML/CSS in the validated host-read text aliases, and the PR head lets XML-prologue content on non-alias paths reach the document allowlist before full text validation.

## Solution Assessment

No. The direction is plausible, but application/xml should stay out of the buffer-verified allowlist and go through the validated-text path with focused regression coverage.
