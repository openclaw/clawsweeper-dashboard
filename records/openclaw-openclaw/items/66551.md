## Summary

Keep this PR open. The intended validated-text direction is reasonable, but the latest head still puts `application/xml` in the buffer-verified document allowlist, which lets XML-prologue host reads bypass the full text validator outside the declared HTML/XML/CSS alias path.

## What This Changes

The PR expands host-read media validation to allow validated HTML, XML, and CSS documents from host-local attachment reads, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the validated-text alias approach, but remove XML/text formats from the buffer-verified document allowlist so HTML/XML/CSS/XML are accepted only after declared-alias compatibility and full-buffer text validation. Add a regression for extensionless or `.txt` XML-prologue content with a binary tail, then rerun the targeted media/outbound tests and changed gate.

## Reproduction Assessment

Yes. A host-local extensionless or `.txt` file starting with `<?xml ` can be reasoned through the PR head: `file-type` sniffs `application/xml`, `mediaKindFromMime` marks it as a document, and the new allowlist returns before `isValidatedHostReadText` validates the full buffer.

## Solution Assessment

No. The current PR direction is close, but the safest maintainable solution is to keep text document MIME types out of the buffer-verified allowlist and accept them only through the validated text alias branch.
