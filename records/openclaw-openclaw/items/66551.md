Keep open: current main still rejects the requested host-read HTML/XML/CSS document path, and this PR is useful, but the current head still has a concrete XML allowlist defect in a security-sensitive host-read guard and lacks external real behavior proof.

## What This Changes

The branch extends host-read media validation and tests to accept validated HTML, XML, and CSS local document attachments while rejecting renamed binary payloads, with a changelog note.

## Best Possible Solution

Keep HTML/XML/CSS/XML on the validated-text path only, remove text/XML MIME values from the buffer-verified document allowlist, add a non-alias XML-prologue regression, and coordinate with the related text-document PR before landing.

## Reproduction Assessment

Yes by source inspection. Current main lacks HTML/XML/CSS in the validated host-read text aliases, and the PR head lets non-alias XML-prologue content reach the document allowlist before full text validation.

## Solution Assessment

No. The requested direction is plausible, but the safest implementation keeps plain-text document MIME types out of HOST_READ_ALLOWED_DOCUMENT_MIMES and relies on the validated-text branch instead.
