## Summary

Keep open: the PR addresses a real Gateway config recovery problem, but the submitted startup gate still misses parse-failure snapshots and the user-facing config option needs docs, changelog, and focused tests before merge.

## What This Changes

The PR adds a `gateway.reload.recovery` config enum plus schema/help metadata and uses it to disable last-known-good config recovery during hot reload and startup.

## Best Possible Solution

Keep the opt-out as a core Gateway setting, but make startup honor it for parse failures too and document the exact recovery policy across config reference and troubleshooting docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a startup snapshot whose raw config has a JSON5 parse error: current snapshot construction sets `sourceConfig` to `{}`, so the PR's startup gate cannot see `gateway.reload.recovery: "off"`.

## Solution Assessment

No. The hot-reload approach is narrow, but startup should derive the recovery policy from a parse-independent signal or an accepted last-known-good policy before calling `recoverConfigFromLastKnownGood`.
