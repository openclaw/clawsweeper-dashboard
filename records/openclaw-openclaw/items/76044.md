## Summary

Current main already resolves the central destructive auto-revert problem by rejecting invalid startup and hot-reload config without rewriting the user file, with docs and regression coverage pointing repair to doctor instead. The PR’s new recovery config knob is therefore obsolete, and the branch still has narrow startup/test blockers if it were considered for merge.

## What This Changes

The branch adds a `gateway.reload.recovery` config/schema/docs/changelog surface and gates hot-reload/startup last-known-good recovery on that setting.

## Best Possible Solution

Keep the current fail-closed Gateway config behavior and only revisit a recovery policy knob if maintainers explicitly want configurable automatic restore again.

## Reproduction Assessment

Yes. Source inspection shows current main rejects invalid hot-reload and startup config without rewriting, and the PR branch blocker is statically reproducible from its startup recovery check reading only `recovery` instead of the full reload policy.

## Solution Assessment

No. Adding `gateway.reload.recovery` is no longer the best fix because current main removed automatic invalid-config restore from the runtime path and made doctor own repair; the PR branch would also still need startup policy and test-mock repairs before merge.

  - body: The PR says `gateway.reload.mode: "off"` makes recovery default to off, but startup only checks `gateway.reload.recovery` from the rejected snapshot or LKG file. A config with `mode: "off"` and no explicit recovery can still reach last-known-good recovery, so the branch does not preserve the stated no-recovery contract.
  - confidence: 0.9

Without updating the mock or injecting the resolver, Vitest will fail module loading before the startup recovery tests run.
  - confidence: 0.9
