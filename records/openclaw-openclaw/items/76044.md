## Summary

Keep open: the opt-out targets a real core Gateway recovery behavior, but the startup implementation still auto-recovers JSON5 parse failures because those rejected snapshots have empty `sourceConfig`, and the PR also needs the required user-facing docs/changelog follow-through.

## What This Changes

The branch adds a `gateway.reload.recovery` config enum, schema/help metadata, and hot-reload/startup gates for disabling last-known-good config restore.

## Best Possible Solution

Keep the opt-out as a core Gateway config setting, but make startup honor it even when the rejected file cannot be parsed, then document and test the exact recovery policy.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a startup snapshot whose raw config has a JSON5 parse error: current snapshot construction sets `sourceConfig` to `{}`, so the PR's startup gate cannot see `gateway.reload.recovery: "off"` and proceeds to last-known-good recovery.

## Solution Assessment

No. The hot-reload direction is narrow, but startup should derive the recovery policy from a parse-independent source, or from last-known-good when the rejected file cannot provide a parsed setting, before calling `recoverConfigFromLastKnownGood`.

## Review Findings

Please derive the startup policy from a parse-independent source, or from last-known-good when the rejected file cannot provide a parsed setting, before calling recovery.
