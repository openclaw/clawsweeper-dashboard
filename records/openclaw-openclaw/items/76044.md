## Summary

Keep open: current main does not implement the recovery opt-out, and the PR still has a startup bug because parse-failure snapshots cannot carry `gateway.reload.recovery`; the user-facing docs/changelog work is also incomplete.

## What This Changes

The PR adds a `gateway.reload.recovery` config enum with schema/help metadata and gates hot-reload plus startup last-known-good recovery on that setting.

## Best Possible Solution

Keep the opt-out as a core Gateway setting, but make startup recovery policy parse-independent for malformed files and land it with focused tests, docs, schema checks, and a changelog entry.

## Reproduction Assessment

Yes. A static reproduction is a startup snapshot for a malformed JSON5 config: current snapshot construction sets `sourceConfig` to `{}`, so the PR's startup check treats recovery as enabled and can still restore last-known-good.

## Solution Assessment

No. The hot-reload direction is narrow, but startup should derive the opt-out from a parse-independent source or a known last-good policy before calling recovery for parse failures.
