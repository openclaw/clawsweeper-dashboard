## What This Changes

The PR changes Mattermost reply target resolution so non-DM messages with an existing `root_id` keep that thread even when `replyToMode` is `off`, and updates focused tests for threaded and top-level cases.

## Best Possible Solution

Land the focused guard removal with regression coverage after a maintainer accepts the documented-default restoration and the PR has redacted live Mattermost proof for threaded and top-level `off` replies.

## Reproduction Assessment

Yes by source inspection, though I did not run a real Mattermost server: current main returns `undefined` for an existing non-DM `threadRootId` when `replyToMode` is `off`, while the docs say default `off` should keep already-threaded inbound posts in their thread.

## Solution Assessment

Yes, this is the narrowest maintainable fix: preserve an existing Mattermost thread root before using `replyToMode` to decide whether to auto-start a top-level thread. The remaining gap is after-fix live proof, not the code shape.
