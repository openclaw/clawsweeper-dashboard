## Summary

Keep open: the report is fresh, concrete, and current main still has a plausible MiniMax Portal local-inference path where provider text can be lost before the CLI collector sees it. Related open issues cover broader MiniMax/reasoning failures, but this report adds distinct Anthropic-compatible `thinking`-then-`text` content-block evidence that should be preserved in the fix.

## What This Changes

The issue reports that MiniMax Portal non-streaming Anthropic-compatible responses with a leading `thinking` block followed by `text` are treated as empty output.

## Best Possible Solution

Add focused regression coverage for MiniMax Portal responses containing `thinking` before `text`, then normalize the local and gateway model-run paths so visible text blocks are preserved without exposing thinking as user text or adding a config knob.

## Reproduction Assessment

No, not at high confidence from this review alone. The reporter gives concrete live commands and current source matches the exact empty-output error path, but a maintainer still needs live MiniMax credentials or a fake-server test to reproduce the dropped `thinking`-then-`text` response on current main.

## Solution Assessment

Yes, the suggested direction is the right narrow fix: collect valid `text` blocks regardless of preceding `thinking` blocks. The safer implementation is at the transport/simple-completion normalization boundary with regression tests, not a CLI-only fallback or a new user setting.
