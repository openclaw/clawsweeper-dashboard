## Label Justifications

- `P2`: The PR targets a normal-priority iMessage channel startup replay bug with limited blast radius.
- `merge-risk: 🚨 compatibility`: The diff changes startup behavior for existing configured iMessage `dbPath` setups by adding a new direct local database probe before subscription.
- `merge-risk: 🚨 availability`: If the new pre-subscribe database probe throws, the iMessage monitor can fail before `watch.subscribe` starts.

## Summary

Keep this PR open for maintainer review: the ROWID watermark approach is directionally good, but the implementation makes a new direct local chat.db read part of iMessage startup and can block supported wrapper setups before the existing RPC path subscribes. The PR also has the protected maintainer label and lacks after-fix real iMessage behavior proof.

## What This Changes

The PR adds a pre-subscribe iMessage chat.db ROWID watermark, passes it as `watch.subscribe` `since_rowid`, drops notifications at or below that watermark, and adds unit coverage for stale and live rows.

## Best Possible Solution

Make the startup watermark best-effort or derive it through the `imsg` RPC path, keep the prior subscribe behavior when the direct local read is unavailable, and require redacted real iMessage proof before merge.

## Reproduction Assessment

Yes from source inspection: current main subscribes without a startup ROWID cursor, while the PR adds one by reading the configured chat.db before subscription. I did not run a live iMessage reproduction because this review is read-only and no real proof artifact was provided.

## Solution Assessment

No, not as currently written: the ROWID watermark is a good direction, but making direct local database access mandatory before the existing RPC startup path is not the safest implementation. The safer path is a best-effort watermark with fallback to current subscribe behavior when the source cannot be proven local and readable.
