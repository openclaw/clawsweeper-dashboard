## Summary

## What This Changes

The PR adds a best-effort local `chat.db` ROWID watermark for iMessage watch startup, passes it as `watch.subscribe` `since_rowid`, drops notifications at or below that watermark, and tests stale, live, and unreadable-db paths.

## Best Possible Solution

Merge only after maintainers accept the startup watermark semantics and real iMessage proof shows stale startup rows are suppressed, new live rows still deliver, and unreadable or wrapper-backed DB paths fall back to no watermark.

## Reproduction Assessment

We have a source-level reproduction, not a live one: current main subscribes without a startup ROWID cursor, and the PR tests the stale `message.id <= maxRowid` replay case. No real iMessage reproduction artifact was provided.

## Solution Assessment

Mostly yes: the latest branch keeps the watermark best-effort and falls back when direct DB access fails, which addresses the previous availability concern. It still needs real startup proof because message suppression depends on the local DB watermark matching the live watch stream.
