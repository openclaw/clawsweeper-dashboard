## Summary

Keep open: the branch is aimed at the right iMessage startup replay problem, but it still leaves default local `imsg` setups without a startup watermark and lacks redacted real iMessage startup proof before merge.

## What This Changes

The branch reads a startup max iMessage `message.ROWID` from a configured local `chat.db`, passes it as `since_rowid` to `watch.subscribe`, drops notifications at or below that watermark, and adds monitor tests.

## Best Possible Solution

Resolve the watermark from the same local DB source OpenClaw already uses for default local `imsg`, keep remote or unreadable wrapper setups on the tolerant no-watermark path, and add redacted live iMessage proof before merge.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection does show that `dbPath`-unset local `imsg` still reaches `watch.subscribe` without `since_rowid`, but the actual stale-row replay needs redacted live iMessage proof.

## Solution Assessment

No. The configured-DB watermark path is useful, but the maintainable fix should also cover the existing default local `chat.db` source while preserving tolerant behavior for remote or unreadable setups.
