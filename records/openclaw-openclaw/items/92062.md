## Summary

Keep open. Current main only adds a narrow same-session reset-archive fallback when the active transcript is missing; it still cannot return prior reset/family fragments when a visible WebChat conversation has rotated across multiple session IDs, and the open family-history PR is still the canonical implementation path.

## What This Changes

Request that WebChat `sessions_history` expose or flag prior reset/archive/family transcript fragments for the same visible dashboard conversation instead of returning only the current active transcript.

## Best Possible Solution

Land or rework the open family-history path into one bounded, explicit, permission-conscious Gateway/tool contract that returns ordered segment/source metadata or a clear “older fragments available” signal.

## Reproduction Assessment

Yes. Source inspection shows `sessions_history` and `chat.history` still have no family-history selector, and the active-first reset fallback cannot include older generated session IDs when the current active transcript exists.

## Solution Assessment

No. Current main's same-session reset fallback is a useful partial repair, but the maintainable solution for this report is an explicit family/logical history API with bounded reads, lineage ordering, visibility checks, and metadata.
