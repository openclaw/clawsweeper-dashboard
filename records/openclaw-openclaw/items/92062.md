## Summary

Keep open. Current main and the latest release still lack a `sessions_history` or `chat.history` family/archive selector, and the existing reset fallback only helps when the active transcript for the same session id is missing, not when one visible WebChat conversation spans multiple session ids.

## What This Changes

The issue asks for WebChat/session history retrieval to include or disclose prior reset/archive/family transcript fragments for the same visible dashboard conversation.

## Best Possible Solution

Land one maintainer-approved, bounded, header-checked logical-history API that preserves the current same-session fallback and returns ordered fragment/source metadata or an explicit older-history-available indicator.

## Reproduction Assessment

Yes, at source level: current main exposes no family/archive selector, stores lineage metadata separately, and reads only the active session id unless that active transcript is missing. I did not run a live WebChat reset in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract: the open `includeFamily` implementation is the closest current path, but the archive access boundary and selector naming need maintainer approval before this can close.
