## Summary

Keep open. Current main and the latest release still lack a logical/family history selector for `sessions_history`/`chat.history`; main only adds same-session reset-archive fallback when the active transcript is missing. The closest implementation path is still https://github.com/openclaw/openclaw/pull/90239, but that PR is open/conflicting and needs maintainer API and transcript-visibility review before this issue can close.

## What This Changes

The issue asks for WebChat/session history to expose or flag older reset/archive/family transcript fragments for the same visible dashboard conversation.

## Best Possible Solution

Land one canonical, opt-in logical-history API backed by `usageFamilySessionIds` and bounded/header-checked reset archive lookup, with documented visibility limits and fragment/source metadata.

## Reproduction Assessment

Yes, at source level: current `sessions_history` forwards only `sessionKey` and `limit`, and `chat.history` reads only the current mapped `sessionId` with same-session reset fallback. A visible conversation split across prior generated session IDs will omit those older fragments unless a new family/logical history path is added.

## Solution Assessment

Unclear as final: the open family-history PR is the most relevant direction, but maintainers still need to choose the canonical public history selector, archive visibility boundary, ordering, and upgrade proof before this is the best landable solution.
