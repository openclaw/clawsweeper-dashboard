## Label Justifications

- `P2`: A supported core history tool hides retained prior conversation state after reset, causing misleading results with bounded but meaningful user impact.
- `impact:session-state`: The issue concerns logical session lineage and transcript continuity across reset/archive fragments.

## Summary

Current main and v2026.6.5 still expose only the active transcript through `sessions_history`; the requested logical-history capability remains unmerged, and the issue should stay open while maintainers consolidate the overlapping archive and family-history implementations.

## What This Changes

Expose prior reset/archive/family transcript fragments for the same visible WebChat conversation through a supported session-history interface.

## Best Possible Solution

Consolidate the useful work into one opt-in, read-only history contract that returns ordered active, reset-archive, and family fragments with source session/file metadata, preserves default `chat.history` and `chat.startup` behavior, and uses canonical persisted lineage rather than trajectory-file inference.

## Reproduction Assessment

Yes. A high-confidence source path is to map a visible WebChat key to a new active transcript while retaining older reset files; current `sessions_history` delegates to `chat.history`, which reads only the mapped active session transcript.

## Solution Assessment

The requested recovery direction is useful, but separate `includeArchived`, `includeFamily`, and alias surfaces are not yet proven to be the best permanent API; maintainers should consolidate them around one lineage model and compatibility-tested contract.
