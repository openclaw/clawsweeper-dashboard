## Summary

Keep open. Current main and the latest release still do not expose a family/archive history selector for `sessions_history`, and the closest implementation remains open and conflicting, so this issue still needs maintainer API direction rather than cleanup closure.

## What This Changes

Expose a reliable `sessions_history` or WebChat history path for prior reset/archive/family transcript fragments that belong to one visible dashboard conversation.

## Best Possible Solution

Land one rebased canonical opt-in logical-history API that uses persisted session lineage, returns ordered source metadata, preserves existing redaction/visibility/budget limits, and clearly signals when older archives exist but are not included.

## Reproduction Assessment

Yes, at source level. Current main has no family/archive request parameter and reads the current mapped session fragment; this review did not run a live WebChat reset scenario.

## Solution Assessment

Unclear as a final implementation path. The open family-history PR is the closest direction, but maintainers still need to settle the public API shape and rebase it over current reset-fallback behavior.
