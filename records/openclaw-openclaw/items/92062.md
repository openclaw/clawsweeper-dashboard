## Summary

Keep open. Current main and v2026.6.8 still lack a `sessions_history` or `chat.history` selector that can traverse prior reset/archive/family transcript fragments for the same visible WebChat conversation, and the closest fix path is still the open family-history PR that needs maintainer API and archive-access approval.

## What This Changes

Request that WebChat/session history retrieve or clearly signal prior reset/archive/family transcript fragments for the same visible dashboard conversation.

## Best Possible Solution

Land one maintainer-approved, bounded, header-checked logical-history API that settles selector naming and returns ordered fragment/source metadata or an explicit older-history-available indicator.

## Reproduction Assessment

Yes, at source level: current main exposes no family/archive selector, stores lineage metadata separately, and reads only the active session id unless that active transcript is missing. I did not run a live WebChat reset in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract: the open `includeFamily` implementation is the closest current path, but selector naming and archived-transcript exposure need approval before this can close.
