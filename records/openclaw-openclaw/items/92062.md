- `P2`: The issue causes misleading history retrieval for affected conversations but does not crash or disable the core runtime.

## Summary

Current main still returns only the active transcript, and the closest family-history implementation remains open but conflicting, so this issue should stay open until that PR is rebased, reviewed, and either landed or replaced.

## What This Changes

Expose prior WebChat history across active, reset/archive, and related session-family transcript fragments.

## Best Possible Solution

Rebase and narrow the existing family-history implementation so sessions_history can explicitly follow persisted reset lineage for WebChat, return ordered source-fragment metadata, and signal when older archives exist while leaving default active-session hydration unchanged.

## Reproduction Assessment

Yes, at high confidence from source and the reporter’s concrete storage structure: current sessions_history resolves the active session only, while reset code preserves prior IDs and archive files outside that retrieval path. A live current-main scenario was not run during this read-only review.

## Solution Assessment

The family-lineage approach in https://github.com/openclaw/openclaw/pull/90239 is the best available direction because it uses persisted session metadata instead of trajectory-file inference, but the public parameter contract and current merge conflicts still require maintainer resolution.
