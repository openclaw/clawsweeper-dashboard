## Summary

Keep open. Current main and the latest release still lack a `sessions_history` or `chat.history` family/archive selector for one visible WebChat conversation split across multiple session ids, and the active related implementation remains an open maintainer-review/product-decision path.

## What This Changes

The issue reports that a visible WebChat/dashboard conversation split across reset/archive session files is not reachable through `sessions_history`.

## Best Possible Solution

Land one maintainer-approved, bounded, header-checked logical-history API that preserves the current same-session fallback and returns ordered fragment/source metadata or an explicit older-history-available indicator.

## Reproduction Assessment

Yes, at source level: current main exposes no family/archive selector, stores lineage metadata separately, and reads only the active session id unless that same active transcript is missing. I did not run a live WebChat reset in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract: the open `includeFamily` implementation is the closest current path, but selector naming, archive access boundaries, and WebChat lineage proof need maintainer approval before this can close.
