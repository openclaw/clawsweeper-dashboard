## Summary

Keep open. Current main and v2026.6.6 only have a missing-active reset-archive fallback; they still do not expose prior reset/archive/family fragments when the current active WebChat transcript exists, and the closest implementation path remains the open, conflicting family-history PR at https://github.com/openclaw/openclaw/pull/90239.

## What This Changes

The issue asks for WebChat/session history retrieval to surface prior reset/archive/family transcript fragments for the same visible dashboard conversation instead of only the active transcript.

## Best Possible Solution

Land one maintainer-approved, opt-in logical-history contract that uses canonical session-family lineage, preserves current visibility and byte limits, and returns ordered fragment/source metadata plus an explicit older-history-available signal.

## Reproduction Assessment

Yes, at source level. Current main has no family/archive request parameter, and the active transcript path returns the current file first, so a visible conversation split across older session IDs is not reachable through sessions_history without a new mode or endpoint.

## Solution Assessment

Unclear until maintainers choose the public contract. The open family-history PR is the closest direction, but the best fix must be rebased and bounded, permission-conscious, and explicit about reset/archive semantics before it can be considered landable.
