## Summary

Keep open: current main only added a narrow missing-active reset-archive fallback, while the reported WebChat case still has an active current transcript plus older reset/family fragments under prior session IDs. The canonical remaining implementation work is still open in https://github.com/openclaw/openclaw/pull/90239 and needs maintainer API/security review before this issue can close.

## What This Changes

Make WebChat `sessions_history` expose or flag prior reset/archive/family transcript fragments for the same visible dashboard conversation.

## Best Possible Solution

Review or replace https://github.com/openclaw/openclaw/pull/90239 with one bounded, explicit family-history contract that returns ordered segment/source metadata and preserves visibility/security limits.

## Reproduction Assessment

Yes, at source level: current main has no family/archive request parameter and the Gateway reads the current mapped session transcript first. I did not run a live WebChat reset because the source path and related real-user reports already establish the gap.

## Solution Assessment

No, current main is not sufficient: the missing-active archive fallback helps one edge case but does not retrieve older fragments when the current active transcript exists. The best path is a maintainer-approved, bounded family-history API rather than another narrow fallback.
