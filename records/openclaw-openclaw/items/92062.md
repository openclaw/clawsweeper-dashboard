## Summary

This issue remains necessary because current `main` still reads only the active transcript, while the exact family/archive retrieval implementation remains open in https://github.com/openclaw/openclaw/pull/90239. The issue should stay paired with that PR until maintainers settle the additive API and session-lineage semantics.

## What This Changes

Make prior reset/archive fragments discoverable for the same visible WebChat conversation through `sessions_history` or equivalent metadata.

## Best Possible Solution

Land one additive, opt-in lineage-history path that uses durable family metadata, discovers relevant reset archives, preserves active-only defaults, and reports fragment boundaries and source session IDs.

## Reproduction Assessment

Yes at source level: current `chat.history` resolves the current session entry and reads only its active transcript, while rollover records older IDs separately and reset files sit outside that read path. No live current-main WebChat run was performed during this read-only review.

## Solution Assessment

The requested outcome is valid, but the exact API needs maintainer judgment; an additive opt-in family/archive query with explicit fragment boundaries is safer than silently changing default history semantics.
