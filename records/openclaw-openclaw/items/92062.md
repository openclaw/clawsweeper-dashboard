## Summary

Keep open. Current main and the latest release still lack a family/archive selector for `sessions_history`, and the adjacent same-session reset fallback only helps when the active transcript is missing, not when one visible WebChat conversation spans multiple session IDs.

## What This Changes

The issue requests a reliable `sessions_history` or `chat.history` path for retrieving prior WebChat conversation fragments stored in reset/archive/family session files.

## Best Possible Solution

Land one maintainer-approved, bounded, header-checked logical-history API that preserves current reset fallback and returns ordered fragment/source metadata for prior reset/archive family sessions.

## Reproduction Assessment

Yes at source level: the current tool/protocol expose no family/archive selector, and the Gateway reads the currently mapped session id with only a same-id missing-active reset fallback. I did not run a live WebChat reset because this review is read-only.

## Solution Assessment

No current-main fix is complete. The open family-history PR is the closest path, but the best solution still needs maintainer agreement on one public API and a rebase that preserves the reset fallback.
