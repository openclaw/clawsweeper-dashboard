## Summary

Keep open: the underlying bug still exists on current main and this PR is the paired implementation candidate, but the branch is incomplete for the current reload contract and lacks required real behavior proof.

## What This Changes

The PR adds account-scoped gateway reload planning for `channels.<channel>.accounts.<accountId>.*` paths and routes those targets through account-specific channel start/stop calls.

## Best Possible Solution

Rebase or replace the branch with account-scoped restart targets carried through planner, hot reload deferral, plugin reload, and `secrets.reload` rollback paths, then add redacted real gateway proof before landing it for https://github.com/openclaw/openclaw/issues/43935.

## Reproduction Assessment

Yes, at source level. `channels.telegram.accounts.*` matches Telegram's broad `channels.telegram` reload prefix and current main stores channel reload matches only in `restartChannels`; the linked issue adds runtime logs showing a whole-channel restart and Telegram delivery failures.

## Solution Assessment

No. Account-scoped reload planning is the right maintainable direction, but this patch must preserve active-work deferral and update every reload-plan consumer, especially `secrets.reload`, before it is the narrow safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:120-126`
  - body: The new account restart loop calls `stopChannel` and `startChannel` immediately. Whole-channel reloads on main wait for queued operations, pending replies, embedded runs, and active tasks before restarting, so account-scoped restarts can still interrupt the active traffic this fix is meant to protect.
  - confidence: 0.88
- **[P2] Update all reload-plan consumers for account targets:** `src/gateway/config-reload-plan.ts:16`
  - body: Adding `restartChannelAccounts` changes the reload-plan contract, but consumers such as `secrets.reload` still inspect only `restartChannels` for skip handling, failure reporting, and rollback. Account token changes can activate a new snapshot while the affected runtime keeps stale credential state.
  - confidence: 0.9
