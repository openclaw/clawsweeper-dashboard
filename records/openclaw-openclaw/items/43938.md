## Summary

Keep open: the linked bug is still present on current main and the PR is aimed at the right boundary, but this patch is incomplete because account restarts bypass current channel-reload deferral and are not carried through `secrets.reload` restart/rollback handling. The external PR also still lacks real after-fix gateway/channel proof.

## What This Changes

Adds a `restartChannelAccounts` reload-plan target, plans `channels.<channel>.accounts.<accountId>.*` paths as account-only restarts, and updates gateway reload tests.

## Best Possible Solution

Rebase or replace the branch with account-scoped restart targets carried through planner, hot reload, plugin reload, and `secrets.reload` restart/rollback paths while preserving current active-work deferral, then add redacted real reload proof.

## Reproduction Assessment

Yes. Source inspection shows `channels.telegram.accounts.botA.token` matches Telegram's broad `channels.telegram` reload prefix and current main adds only `telegram` to `restartChannels`; the linked issue also provides matching runtime logs.

## Solution Assessment

No. Account-scoped reload planning is the right maintainable direction, but this implementation is not complete until account targets share current deferral behavior and every reload-plan consumer handles them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:120-126`
  - body: The new account restart loop calls `stopChannel` and `startChannel` immediately. Current main defers channel reloads while queued operations, replies, embedded runs, or active tasks are still running, so account-scoped restarts need the same wait path or they can still interrupt the active traffic this fix is meant to protect.
  - confidence: 0.88
- **[P2] Update all reload-plan consumers for account targets:** `src/gateway/config-reload-plan.ts:16`
  - body: Adding `restartChannelAccounts` changes the reload-plan contract, but consumers such as `secrets.reload` still inspect only `plan.restartChannels` for skip handling, restart failure reporting, and rollback. Account token changes could activate a new snapshot while leaving the affected account runtime on stale credentials.
  - confidence: 0.9
