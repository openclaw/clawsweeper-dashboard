## Summary

This PR should stay open because it is a focused fix for a valid gateway reload bug, but it is not merge-ready: the branch narrows account reload planning while missing real behavior proof and leaving existing active-work deferral plus secrets reload rollback contracts unhandled.

## What This Changes

The branch adds account-scoped gateway reload targets for `channels.<channel>.accounts.<accountId>.*` changes and tests account-only versus full-channel restart planning.

## Best Possible Solution

Land or replace this with an account-scoped reload implementation that carries account targets through active-work deferral, plugin reload interactions, `secrets.reload` rollback semantics, focused tests, and redacted runtime proof.

## Reproduction Assessment

Yes, at source level: `channels.telegram.accounts.<id>.*` matches Telegram's broad reload prefix, current main records only whole-channel restart targets, and the linked issue includes matching runtime logs with Telegram send failures. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Account-scoped planning is the right direction, but this patch needs to preserve active-work deferral, plugin reload behavior, and `secrets.reload` rollback before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:123-126`
  - body: The new account restart loop calls `stopChannel`/`startChannel` directly. Current whole-channel hot reloads first wait for queued operations, pending replies, embedded runs, and active tasks, so this can still interrupt active traffic during the window the fix is trying to narrow.
  - confidence: 0.88
- **[P2] Keep secrets reload aware of account targets:** `src/gateway/config-reload-plan.ts:181-185`
  - body: Moving `channels.<channel>.accounts.*` token changes out of `restartChannels` means `secrets.reload` no longer sees those targets for skip handling, restart failure reporting, or rollback, so a new secrets snapshot can activate without restarting or recovering the affected account runtime.
  - confidence: 0.9
