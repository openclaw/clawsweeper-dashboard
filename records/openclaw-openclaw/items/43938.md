## Summary

Keep this PR open as the active fix candidate for the linked gateway reload bug, but do not merge it as-is: current main still has the source-level issue, while this branch is conflicting, lacks real behavior proof, and bypasses current reload deferral and secrets reload contracts.

## What This Changes

The PR adds account-scoped gateway reload planning for `channels.<channel>.accounts.<accountId>.*` changes and tests targeted channel-account restarts.

## Best Possible Solution

Rebase or replace this with account-target support that flows through current reload deferral, plugin reload pre-stop/active-channel handling, secrets.reload restart and rollback, focused tests, and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current main still maps `channels.telegram.accounts.<id>.*` through Telegram's broad reload prefix and stores only whole-channel restart targets, and the linked issue includes matching runtime send-failure logs. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as currently proposed: account-scoped planning is the right direction, but the patch must be updated for current active-work deferral, plugin reload, and secrets.reload restart/rollback semantics before it is the best maintainable fix.
