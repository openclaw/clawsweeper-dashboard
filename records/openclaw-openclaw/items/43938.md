## Label Justifications

- `P1`: The PR targets a broken gateway/channel hot-reload path that can interrupt active Telegram traffic and drop in-flight replies.
- `merge-risk: 🚨 message-delivery`: Changing channel account stop/start during hot reload can still drop or misroute messages if the account target, drain, or restart order is wrong.
- `merge-risk: 🚨 auth-provider`: The affected paths include account credentials and tokens, so a missed or mis-scoped restart could leave stale or wrong account auth active.
- `merge-risk: 🚨 availability`: The stale branch must be reconciled with current channel reload failure handling so a stop/start failure does not leave channel accounts unavailable.

## Summary

Keep open: the account-scoped reload fix is still useful and current main does not implement it, but this external PR is conflicting with current gateway reload code and only provides test/CI evidence rather than real reload proof.

## What This Changes

The branch adds account-scoped channel restart planning so `channels.<channel>.accounts.<accountId>.*` changes restart only the affected account while full-channel restarts still win for channel-global or mixed changes.

## Best Possible Solution

Rebase the account-scoped restart plan onto the current reload handler, preserve the active-work drain and plugin-reload semantics, then add focused tests plus redacted real hot-reload proof before merging.

## Reproduction Assessment

Yes, at source level: current main maps `channels.telegram.accounts.<id>.*` through the broad Telegram reload prefix into `restartChannels`, and the handler restarts the whole channel. I did not run a live Telegram bot repro in this read-only review.

## Solution Assessment

Yes as a direction, but not as a merge-ready branch. Carrying account ids through the generic reload planner is the narrow maintainable fix, but the PR must be rebased onto the current reload handler and proven in a real hot-reload setup.
