## Summary

Keep open. The PR targets a real current-main gateway reload gap, but the branch is conflicting and the proposed account restart path still misses current hot-reload deferral plus `secrets.reload` restart/rollback handling; external real behavior proof is also still missing.

## What This Changes

This PR adds account-scoped gateway reload planning for `channels.<channel>.accounts.<accountId>.*` changes and routes those targets through account-aware channel stop/start calls with tests.

## Best Possible Solution

Land a rebased account-scoped reload fix that carries account targets through planner, plugin reload interactions, active-work deferral, `secrets.reload` rollback, focused tests, and redacted runtime proof.

## Reproduction Assessment

Yes, at source level. `channels.telegram.accounts.engineer.dmPolicy` matches Telegram's broad `channels.telegram` reload prefix and current main records only a whole-channel restart, while https://github.com/openclaw/openclaw/issues/43935 includes matching runtime logs.

## Solution Assessment

No. Account-scoped planning is the right direction, but this patch is not the best mergeable form until it preserves current active-work deferral and `secrets.reload` skip/failure/rollback semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:123-126`
  - body: The new account restart loop calls `stopChannel` and `startChannel` directly. Current whole-channel reloads first wait for queued operations, pending replies, embedded runs, and active tasks, so account-scoped reloads need the same deferral or they can interrupt active traffic during the window this fix is meant to narrow.
  - confidence: 0.88
- **[P2] Keep secrets reload aware of account targets:** `src/gateway/config-reload-plan.ts:181-185`
  - body: Moving `channels.<channel>.accounts.*` token changes into `restartChannelAccounts` means `secrets.reload` no longer sees them in `plan.restartChannels`, so it can activate a new secrets snapshot without restarting or rolling back the affected account runtime.
  - confidence: 0.9
