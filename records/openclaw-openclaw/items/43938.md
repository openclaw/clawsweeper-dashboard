## Summary

Keep open: the PR targets a real current-main reload granularity problem, but the proposed patch is not mergeable because the new account-scoped plan bypasses existing reload deferral and leaves `secrets.reload` restart/rollback consumers unaware of account restarts.

## What This Changes

The PR adds account-scoped Gateway reload planning for channel account config paths, wires hot reload to account-specific channel stop/start calls, and adds focused reload tests.

## Best Possible Solution

Land a corrected account-scoped reload implementation on current main: keep full-channel precedence, preserve active-work deferral, extend `secrets.reload` restart and rollback handling to account IDs, add regression coverage, and include a changelog entry.

## Reproduction Assessment

Yes, by source inspection. Current main routes `channels.telegram.accounts.botA.token` through Telegram's broad `channels.telegram` reload prefix into `restartChannels`, while account-scoped channel lifecycle methods already exist.

## Solution Assessment

No. The requested direction is narrow and maintainable, but this PR is incomplete until account restarts share deferral semantics and every reload-plan consumer handles the new account map.
