## Summary

Keep open: the PR targets a valid current-main gateway reload bug, but it is not mergeable as-is because account restarts bypass current deferral semantics, `secrets.reload` remains on the old whole-channel contract, and real behavior proof is missing.

## What This Changes

Adds `restartChannelAccounts` planning for `channels.<channel>.accounts.<accountId>.*` changes, routes those hot reloads through account-aware channel stop/start calls, and adds planner/server reload tests.

## Best Possible Solution

Land a rebased account-scoped reload fix that carries account targets through planner, hot-reload deferral, plugin reload interactions, and `secrets.reload` rollback, with focused tests and redacted runtime proof.

## Reproduction Assessment

Yes, at source level. `channels.telegram.accounts.*` matches Telegram's broad reload prefix and current main records only a whole-channel restart, while the linked issue includes matching runtime logs for full Telegram channel restart and send failures.

## Solution Assessment

No. Account-scoped planning is the right maintainable direction, but this patch is not yet the best safe fix because it bypasses reload deferral and leaves `secrets.reload` on the old whole-channel contract.
