## Summary

Keep open: the account-scoped reload direction is valid and still needed on current main, but this PR is conflicting, lacks after-fix real behavior proof, and misses current deferral plus `secrets.reload` reload-plan consumers.

## What This Changes

The PR adds account-scoped gateway reload planning for channel account config paths and routes those targets through channel account start/stop calls.

## Best Possible Solution

Rebase or replace the branch with account-scoped restart targets carried through planner, hot reload, plugin reload, and `secrets.reload` restart/rollback paths while preserving current active-work deferral.

## Reproduction Assessment

Yes. Source inspection shows `channels.telegram.accounts.*` matches Telegram's broad `channels.telegram` reload prefix and current main plans only a whole-channel restart; the linked issue adds matching runtime logs.

## Solution Assessment

No. Account-scoped reload planning is the right maintainable direction, but this implementation is incomplete until account targets share current deferral behavior and every reload-plan consumer handles them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:120-126`
  - body: The new account restart loop calls `stopChannel` and `startChannel` immediately. Whole-channel reloads wait for queued operations, replies, embedded runs, and active tasks, so account-scoped restarts need the same wait path or they can still interrupt the traffic this fix is meant to protect.
  - confidence: 0.88
- **[P2] Update all reload-plan consumers for account targets:** `src/gateway/config-reload-plan.ts:16`
  - body: Adding `restartChannelAccounts` changes the reload-plan contract, but consumers such as `secrets.reload` still inspect only `restartChannels` for skip handling, failure reporting, and rollback. Account token changes can activate a new snapshot while the affected runtime keeps stale credentials.
  - confidence: 0.9
