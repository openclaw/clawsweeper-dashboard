## Summary

## What This Changes

The PR adds account-scoped Gateway reload planning for `channels.<channel>.accounts.<accountId>.*` paths and tests that those paths call account-specific channel stop/start instead of full channel restarts.

## Best Possible Solution

Rebase the account-scoped reload model onto current main so account targets share the same deferral, plugin replacement, skip/failure handling, and secrets rollback contracts as full-channel reloads, then prove it in a redacted real gateway run.

## Reproduction Assessment

Yes, at source level: `channels.telegram.accounts.<id>.*` matches Telegram's broad reload prefix, current main records only a whole-channel restart, and the linked issue includes matching runtime delivery-loss logs. I did not run a live Telegram reproduction in this read-only pass.

## Solution Assessment

No, not as currently implemented. Account-scoped reload planning is the right direction, but this branch must be rebased and integrated with current deferral, plugin reload, and `secrets.reload` rollback contracts before it is the maintainable fix.
