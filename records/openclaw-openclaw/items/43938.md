## Summary

Keep open: the PR is aimed at a real gateway reload bug, but the branch is now conflicting and the submitted account-restart path misses current-main reload deferral, plugin-reload, and secrets-reload safety contracts.

## What This Changes

The branch adds account-scoped reload planning so `channels.<channel>.accounts.<accountId>.*` hot reloads call account-specific channel stop/start instead of restarting the whole channel.

## Best Possible Solution

Rebase the account-scoped reload model onto current main so account targets share the same deferral, plugin replacement, skip/failure handling, and secrets rollback contracts as full-channel reloads, then prove it with redacted real gateway output.

## Reproduction Assessment

Yes, at source level: current main maps `channels.telegram.accounts.<id>.*` through Telegram's broad `channels.telegram` prefix into `restartChannels`, and the linked issue includes runtime logs showing send failures during the full restart. I did not run a live reproduction in this read-only pass.

## Solution Assessment

No, not as submitted. Account-scoped planning is the right direction, but this branch needs to be rebased and routed through current reload deferral, plugin replacement, failure collection, and secrets rollback before it is the maintainable fix.
