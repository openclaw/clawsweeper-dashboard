## Summary

Keep open. Current main still plans account-only channel config changes as whole-channel reloads, so the PR is addressing a real gateway reload limitation, but the proposed patch is not mergeable as-is because it does not preserve existing channel-reload deferral semantics and it changes the reload-plan contract without updating secrets.reload/account restart handling.

## What This Changes

The PR adds account-scoped restart state to GatewayReloadPlan, detects `channels.<channel>.accounts.<accountId>.*` hot-reload paths, dispatches account-specific channel stop/start calls, and adds gateway reload tests for scoped and full-channel restart planning.

## Best Possible Solution

Land a corrected account-scoped reload implementation on current main: keep full-channel precedence for global or mixed changes, apply the same active-work deferral before account and whole-channel restarts, update secrets.reload and reload-plan helpers/fixtures for the new state, and cover both hot reload and secrets reload with focused regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: on current main, `channels.telegram.accounts.botA.token` matches Telegram's broad `channels.telegram` reload prefix and becomes a `restartChannels` entry even though the channel manager already supports account-specific start/stop.

## Solution Assessment

No. The requested direction is the narrow maintainable fix, but this PR is not the best mergeable solution until account restarts preserve reload deferral semantics and every reload-plan consumer, especially secrets.reload, handles the new account-scoped state.
