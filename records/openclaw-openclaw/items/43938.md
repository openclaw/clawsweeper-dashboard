## Summary

Keep open: the account-scoped reload direction is useful, but this PR is conflicting, lacks real behavior proof, and the diff bypasses current reload deferral plus `secrets.reload` restart/rollback semantics.

## What This Changes

The branch adds `restartChannelAccounts` planning for `channels.<channel>.accounts.<accountId>.*`, restarts targeted channel accounts, and adds focused gateway reload tests.

## Best Possible Solution

Rebase or replace the PR with account-scoped reload support that carries account targets through active-work deferral, plugin reload, and `secrets.reload` rollback, plus focused tests and redacted live proof.

## Reproduction Assessment

Yes, at source level: current main maps `channels.telegram.accounts.<id>.*` through Telegram's broad reload prefix and stores only whole-channel restart targets, while the linked issue includes matching send-failure logs. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Account-scoped planning is the right direction, but this patch is not mergeable until it preserves active-work deferral, plugin reload interactions, and `secrets.reload` restart/rollback semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:123-126`
  - body: The new account restart loop calls `stopChannel`/`startChannel` directly. Current whole-channel reloads first wait for queued operations, pending replies, embedded runs, and active tasks, so account-scoped reloads can still interrupt active traffic during the window this fix is meant to narrow.
  - confidence: 0.88

  - confidence: 0.9
