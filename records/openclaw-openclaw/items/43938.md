## Summary

Keep open: the underlying gateway reload bug is still present on current main, and this PR is a useful direction, but the branch is conflicting, lacks required real behavior proof, and bypasses established reload deferral plus secrets.reload restart/rollback contracts.

## What This Changes

The branch adds account-scoped channel restart targets to the gateway reload plan, routes account config changes through account-aware stop/start calls, and adds focused reload planner/server tests.

## Best Possible Solution

Rebase or replace the PR with account-scoped reload support that carries account targets through reload deferral, plugin reload interactions, secrets.reload restart/rollback, focused tests, and redacted runtime proof.

## Reproduction Assessment

Yes, at source level: channels.telegram.accounts.<accountId>.* matches Telegram's broad reload prefix, current main stores only whole-channel restart targets, and the linked issue includes matching runtime logs with Telegram send failures.

## Solution Assessment

No. Account-scoped reload planning is the right direction, but this patch is not mergeable until it preserves active-work deferral, plugin reload interactions, and secrets.reload restart/rollback semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:123-126`
  - body: The new account restart loop calls stopChannel/startChannel directly. Current whole-channel reloads first wait for queued operations, pending replies, embedded runs, and active tasks, so account-scoped reloads can still interrupt active traffic during the window this fix is meant to narrow.
  - confidence: 0.88
- **[P2] Keep secrets reload aware of account targets:** `src/gateway/config-reload-plan.ts:181-185`
  - body: Moving channels.<channel>.accounts.* token changes into restartChannelAccounts means secrets.reload no longer sees them in plan.restartChannels, so it can activate a new secrets snapshot without restarting or rolling back the affected account runtime.
  - confidence: 0.9
