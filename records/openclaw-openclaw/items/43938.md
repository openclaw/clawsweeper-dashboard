## Summary

Keep open: the PR targets a real gateway reload limitation, but the current patch is not mergeable because account restarts bypass current channel-reload deferral and do not preserve secrets.reload restart/rollback semantics; it also still lacks after-fix real behavior proof.

## What This Changes

The branch adds account-scoped gateway hot-reload planning for channel account config changes and tests the planner plus handler calls.

## Best Possible Solution

Rework the account-scoped reload plan on current main so account targets flow through active-work deferral, plugin reload interactions, and secrets.reload rollback, then add focused tests plus redacted live proof.

## Reproduction Assessment

Yes, at source level: current main maps channels.telegram.accounts.<id>.* through Telegram's broad reload prefix into restartChannels, and the linked issue includes matching disruption logs. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Account-scoped planning is the right direction, but this patch needs to preserve active-work deferral, plugin reload behavior, and secrets.reload rollback before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:123-126`
  - body: The new account restart loop calls stopChannel/startChannel directly. Current channel hot reloads first wait for queued operations, pending replies, embedded runs, and active tasks, so this can still interrupt active traffic during the window the fix is trying to narrow.
  - confidence: 0.88
- **[P2] Keep secrets reload aware of account targets:** `src/gateway/config-reload-plan.ts:181-185`
  - body: Moving channels.<channel>.accounts.* token changes out of restartChannels means secrets.reload no longer sees those targets for skip handling, restart failure reporting, or rollback, so a new secrets snapshot can activate without restarting or recovering the affected account runtime.
  - confidence: 0.9
