## Summary

Keep open: the underlying gateway reload bug is still present on current main, but this branch is not merge-ready because it predates current reload deferral and secrets reload contracts and still lacks real behavior proof.

## What This Changes

The PR adds `restartChannelAccounts` to gateway reload plans and routes `channels.<channel>.accounts.<accountId>.*` changes through account-scoped channel stop/start calls with planner and handler tests.

## Best Possible Solution

Rebase the account-scoped reload model onto current main so account targets flow through active-work deferral, plugin reload interactions, and `secrets.reload` rollback, then prove it with focused tests and redacted real runtime evidence.

## Reproduction Assessment

Yes, at source level: current main maps `channels.telegram.accounts.<id>.*` through Telegram's broad reload prefix into `restartChannels` only, while the lower-level channel manager already supports account-scoped start/stop. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No for this branch as written. Account-scoped planning is the right direction, but the implementation must be rebased onto current main's reload deferral, plugin reload, and `secrets.reload` rollback contracts before merge.
