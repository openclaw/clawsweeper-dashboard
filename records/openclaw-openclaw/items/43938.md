## Summary

Keep open. The PR is a useful direction for the linked gateway reload bug, but the current patch bypasses existing channel-reload deferral, does not update `secrets.reload` consumers for account targets, and has no real after-fix gateway proof.

## What This Changes

The PR adds account-scoped channel reload targets for `channels.<channel>.accounts.<accountId>.*` and routes them through account-id-aware channel stop/start calls with tests.

## Best Possible Solution

Carry account-scoped restart targets through planner, hot-reload deferral, plugin reload, skip handling, and `secrets.reload` rollback, then add redacted real gateway proof before landing it for https://github.com/openclaw/openclaw/issues/43935.

## Reproduction Assessment

Yes, at source level. `channels.telegram.accounts.*` matches Telegram's broad `channels.telegram` reload prefix and current main stores the resulting restart only in `restartChannels`, while the linked issue includes runtime logs for whole-channel restart disruption.

## Solution Assessment

No. Account-scoped reload planning is the right maintainable direction, but this patch is not yet the narrow safe fix because it bypasses active-work deferral and leaves `secrets.reload` on the old whole-channel contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:120-126`
  - body: The new account restart loop calls `stopChannel` and `startChannel` immediately. Whole-channel reloads on main wait for queued operations, pending replies, embedded runs, and active tasks before restarting, so account-scoped restarts can still interrupt the active traffic this fix is meant to protect.
  - confidence: 0.88
- **[P2] Update all reload-plan consumers for account targets:** `src/gateway/config-reload-plan.ts:16`
  - body: Adding `restartChannelAccounts` changes the reload-plan contract, but consumers such as `secrets.reload` still inspect only `restartChannels` for skip handling, failure reporting, and rollback. Account token changes can activate a new snapshot while the affected runtime keeps stale credential state.
  - confidence: 0.9
