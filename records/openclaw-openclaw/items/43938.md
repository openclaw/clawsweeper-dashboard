## Summary

Keep open. The account-scoped reload direction is valid, but this PR is not mergeable as-is because the new account restart path bypasses existing active-work deferral and does not update the secrets.reload consumer/rollback contract for the new reload-plan state.

## What This Changes

The PR adds account-scoped channel restart planning for `channels.<channel>.accounts.<accountId>.*`, routes those changes through account-specific channel stop/start calls, and adds gateway reload tests for scoped and full-channel restart behavior.

## Best Possible Solution

Land a corrected account-scoped reload implementation on current main: keep full-channel precedence for global or mixed channel changes, apply the same active-work deferral before account and whole-channel restarts, update secrets.reload helpers and rollback tracking for account restarts, and cover hot reload plus secrets.reload with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: on current main, `channels.telegram.accounts.botA.token` matches Telegram’s broad `channels.telegram` reload prefix and becomes a whole-channel `restartChannels` entry even though the channel manager already accepts account-specific start/stop calls.

## Solution Assessment

No. The requested direction is the narrow maintainable fix, but this PR is not the best mergeable solution until account restarts preserve existing deferral semantics and every reload-plan consumer handles the new account-scoped state.
