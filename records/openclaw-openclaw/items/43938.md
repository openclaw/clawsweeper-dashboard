## Summary

Keep this PR open, but it is not merge-ready. The account-scoped reload direction matches the linked bug, yet the branch does not preserve current channel-reload deferral, does not update every reload-plan consumer, and still lacks required real behavior proof.

## What This Changes

The PR adds a `restartChannelAccounts` reload-plan target, plans `channels.<channel>.accounts.<accountId>.*` changes as account-only channel restarts, and adds gateway reload tests.

## Best Possible Solution

Rebase or replace the branch with account-scoped restart targets carried through planner, hot reload, plugin reload, and `secrets.reload` restart/rollback paths while preserving current active-work deferral semantics.

## Reproduction Assessment

Yes. Source inspection shows `channels.telegram.accounts.botA.token` matches Telegram’s broad reload prefix and current main adds only `telegram` to `restartChannels`; the linked issue also includes matching runtime logs.

## Solution Assessment

No. Account-scoped reload planning is the right maintainable direction, but this implementation is incomplete until every reload-plan consumer handles account targets and account restarts share the current deferral behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route account restarts through reload deferral:** `src/gateway/server-reload-handlers.ts:120-126`
  - body: The new account restart loop calls `stopChannel` and `startChannel` immediately. Current main waits for queued operations, replies, embedded runs, and task blockers before whole-channel reloads, so account-scoped restarts would reintroduce the active-traffic disruption this PR is trying to avoid.
  - confidence: 0.88
- **[P2] Update all reload-plan consumers for account targets:** `src/gateway/config-reload-plan.ts:16`
  - body: Adding `restartChannelAccounts` changes the reload-plan contract, but consumers such as `secrets.reload` still inspect only `restartChannels` for skip handling, restart failure reporting, and rollback. Account token changes can activate a new snapshot while leaving the affected runtime on stale credentials.
  - confidence: 0.9
