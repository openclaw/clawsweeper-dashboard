## Summary

Keep open. The underlying bug is source-reproducible and this PR is pointed in the right direction, but the branch is stale against current gateway reload code and misses current-main deferral plus `secrets.reload` consumers; it also has only test/CI proof, not real behavior proof.

## What This Changes

The branch adds `restartChannelAccounts` planning for `channels.<channel>.accounts.<accountId>.*`, wires hot reload to account-id stop/start calls, and adds gateway reload tests.

## Best Possible Solution

Rebase or replace the PR with account-scoped restart targets carried through planner, hot reload, plugin reload, and `secrets.reload` restart/rollback paths while preserving current deferral semantics.

## Reproduction Assessment

Yes. Source inspection shows `channels.telegram.accounts.botA.token` matches Telegram's broad `channels.telegram` reload prefix, then current main adds `telegram` to `restartChannels`; the linked issue also includes matching runtime logs.

## Solution Assessment

No. Account-scoped reload planning is the right narrow direction, but this implementation is not merge-ready until account restarts share current deferral behavior and every reload-plan consumer handles the new account map.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve reload deferral for account restarts:** `src/gateway/server-reload-handlers.ts:120-126`
  - body: The new account restart path calls `stopChannel` and `startChannel` immediately. Current main defers whole-channel reloads until queued operations, replies, embedded runs, and task blockers drain, so account-scoped reloads need to go through the same waiting path after rebase.
  - confidence: 0.88
- **[P2] Handle account restarts in every reload-plan consumer:** `src/gateway/config-reload-plan.ts:15-16`
  - body: Adding `restartChannelAccounts` changes the reload-plan contract, but consumers such as `secrets.reload` still inspect only `restartChannels` for skip checks, restart failures, and rollback. Secret-backed account paths moved into the new map could leave the affected account running with stale runtime state.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: No supply-chain change was found, but the incomplete reload-plan contract can leave secret-backed account runtimes on stale credentials after `secrets.reload`.

Concerns:

  - body: Because `secrets.reload` currently restarts and rolls back only `plan.restartChannels`, moving account token paths into `restartChannelAccounts` without updating that handler can activate a new secrets snapshot while the affected account keeps running with stale credential-backed runtime state.
