## Summary

Keep open: this is a MEMBER-authored, maintainer-labeled PR, and the current head still has a blocking source-visible performance regression in the runtime loader.

## What This Changes

The PR threads an optional shared plugin discovery result through plugin loader, manifest registry, installed-index, config-contract, metadata, gateway startup, and runtime registry loading paths.

## Best Possible Solution

Keep the additive discovery plumbing, but make runtime shared discovery lazy or only compute it after the existing active-registry fast paths actually need it.

## Reproduction Assessment

Yes from source inspection, not from an executed test: current main can skip loading when an active registry satisfies explicit plugin IDs, while the PR computes discovery before that skip can run.

## Solution Assessment

No: the optional plumbing is maintainable, but the runtime orchestration should compute discovery lazily or only after the existing skip checks need it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Defer discovery until a load path needs it:** `src/plugins/runtime/runtime-registry-loader.ts:139-143`
  - body: This computes `sharedDiscovery` for every `ensurePluginRegistryLoaded` call before `scopedLoad` is known and before either `activeRegistrySatisfiesScope` return path can run. For calls with explicit `onlyPluginIds` that are already loaded, current main can return from the active registry without a discovery walk; this PR adds a synchronous filesystem scan on that hot path, which is the performance work this branch is trying to avoid. Please make discovery lazy or move it behind the branches that actually pass it into scope resolution or `loadOpenClawPlugins`, and cover the already-loaded explicit-scope case.
  - confidence: 0.86
