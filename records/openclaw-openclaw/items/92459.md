## Summary

The reported v2026.6.5 lock-retention path is addressed in both current main and v2026.6.6: after the bounded sessions_yield abort-settle wait, the runner explicitly releases the retained session lock before any follow-up event drain or persistence. The broader durable subagent-delivery concern remains tracked separately.

## What This Changes

Fix a sessions_yield abort-settle timeout that retained a session write lock, blocked later turns, and contributed to dropped subagent completion announcements.

## Best Possible Solution

Keep the released canonical abort-path lock release, and address durable completion persistence and later delivery through https://github.com/openclaw/openclaw/issues/86538 rather than reopening a second lock-lifecycle implementation.

## Reproduction Assessment

No, not on current main or v2026.6.6: both explicitly release the retained lock after the sessions_yield abort-settle wait. The detailed logs provide high-confidence evidence that the failure existed on v2026.6.5.

## Solution Assessment

Yes. Releasing through the canonical embedded-run lock controller immediately after the bounded settle wait is narrower and safer than force-deleting locks or teaching announcements to break same-process locks.
