## Summary

Keep open: current main still exposes raw unclassified assistant error text on the channel path, but this PR is not merge-ready because it would genericize newer actionable diagnostics, adds a log preview without full redaction/sanitization, and lacks inspectable real behavior proof.

## What This Changes

The branch changes `formatAssistantErrorText` so unrecognized assistant errors return generic user-facing copy while logging a redacted preview, with formatter regression tests for the fallback.

## Best Possible Solution

Refresh or replace this with a narrow fallback hardening that genericizes only truly unclassified assistant errors, preserves actionable diagnostics, logs redacted and sanitized previews, and includes redacted live channel proof.

## Reproduction Assessment

Yes, by source inspection. An assistant message with `stopReason: "error"` and an unclassified `errorMessage` flows through `formatAssistantErrorText`, then `buildEmbeddedRunPayloads` can push that text as a channel-visible `isError` reply.

## Solution Assessment

No. The direction is right, but this patch should preserve current diagnostic carve-outs, redact and pass log previews through `sanitizeForLog` before truncating, and provide inspectable real behavior proof before merge.
