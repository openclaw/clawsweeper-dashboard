## Summary

Keep open. Current main still has a raw unclassified formatter fallback on the channel-visible assistant-error path, but this PR is not merge-ready because it conflicts with newer formatter work, lacks inspectable real behavior proof, and still has correctness/security blockers in the fallback and log-preview handling.

## What This Changes

This PR changes `formatAssistantErrorText` to replace its final unrecognized-error fallback with generic `/new` copy, adds a warning preview plus carve-outs, and adds formatter tests for orphaned tool-call and long raw errors.

## Best Possible Solution

Refresh or replace this with a current-main formatter/payload fix that genericizes truly internal unclassified provider errors, preserves actionable diagnostics, logs only redacted and sanitized previews, and includes redacted live channel proof.

## Reproduction Assessment

Yes, source-level. An assistant message with `stopReason: "error"` and an unclassified `errorMessage` flows through `formatAssistantErrorText`, and `buildEmbeddedRunPayloads` can push the returned text as a channel-visible `isError` reply.

## Solution Assessment

No. The direction is useful, but the current patch must be refreshed against current main, preserve existing diagnostic carve-outs, fix log redaction/sanitization order, and provide inspectable real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve current diagnostic carve-outs before the fallback:** `src/agents/pi-embedded-helpers/errors.ts:659`
  - body: Current main intentionally keeps proxy/config and request-validation diagnostics actionable. This final generic fallback would convert any unclassified string after its limited carve-outs into generic `/new` copy, so a refresh needs explicit classifiers or carve-outs to keep those existing diagnostics.
  - confidence: 0.87
- **[P2] Redact and sanitize before truncating the preview:** `src/agents/pi-embedded-helpers/errors.ts:657`
  - body: The warning preview slices `raw` before redaction and only removes CR/LF, so token-like values cut at 500 chars can miss boundary-based redaction and ANSI/control bytes can still reach logs. Redact, pass through `sanitizeForLog`, then truncate.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch adds warning logging for suppressed raw errors, but the preview is not redacted or sanitized safely before logging.

Concerns:

- **[medium] Preview truncates before redaction:** `src/agents/pi-embedded-helpers/errors.ts:657`
  - body: `redactSensitiveText(raw.slice(0, 500))` can leave a secret fragment unredacted when a bearer token, header, JSON field, or token-like value is cut before the complete sensitive pattern can match.
  - confidence: 0.82
- **[low] Preview leaves non-CRLF control characters:** `src/agents/pi-embedded-helpers/errors.ts:657`
  - body: The preview only replaces CR/LF, so tabs, ANSI escapes, DEL, and other control characters can still be interpolated into warning logs instead of going through `sanitizeForLog`.
  - confidence: 0.86
