## Summary

Keep open. Current main still exposes unrecognized assistant/provider error strings through the embedded delivery path, so the PR addresses a real user-facing leak, but the branch is not merge-ready because it can collapse newer actionable diagnostics into generic copy and its new warning preview is not log-safe enough.

## What This Changes

The PR changes `formatAssistantErrorText` to return a generic message for unrecognized assistant errors, log an operator preview, and add formatter tests for orphaned tool-result and long unknown error strings.

## Best Possible Solution

Refresh the PR against current main so only the true unrecognized catch-all returns generic user copy, existing actionable classifier and diagnostics branches remain covered, the operator preview is redacted before truncation and sanitized for logs, and the user-visible fix has a changelog entry.

## Reproduction Assessment

Yes. The source path is high-confidence: an unrecognized error such as `No tool call found for function call output with call_id ...` reaches the raw fallback in `formatAssistantErrorText`, and `buildEmbeddedRunPayloads` can send that formatted value as a user-facing error reply.

## Solution Assessment

No, not in its current form. The intended direction is right, but the patch should preserve current-main diagnostics and harden the log preview before merge.
