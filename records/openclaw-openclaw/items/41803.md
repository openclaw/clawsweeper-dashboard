## Summary

Keep this PR open, but do not merge it as-is: the user-facing fallback direction is valid, yet the patch still needs real behavior proof and fixes for diagnostic preservation and log safety.

## What This Changes

The PR changes `formatAssistantErrorText` so unrecognized assistant errors return generic user-facing copy while logging a preview, with formatter tests for orphaned tool-call and long raw errors.

## Best Possible Solution

Refresh this PR or replace it with a narrow fallback hardening that genericizes only truly unclassified assistant errors, preserves actionable diagnostics, logs redacted and sanitized previews, and includes redacted live proof.

## Reproduction Assessment

Yes, by source inspection. Current main can send an assistant message with `stopReason: "error"` through `formatAssistantErrorText`, return an unclassified raw `errorMessage`, and deliver it as an `isError` reply.

## Solution Assessment

No. The direction is right, but the patch should preserve current diagnostic carve-outs, redact and sanitize before truncating log previews, and provide inspectable real behavior proof before merge.

  - body: Current main has formatter coverage that keeps config/proxy unsupported text and non-streaming JSON/request-validation diagnostics actionable.
