## Summary

Keep open. Current main still has the raw/truncated unrecognized assistant-error fallback, so the PR targets a real user-facing leak, but the branch is not ready to merge because it is stale against newer formatter behavior and its new warning preview still needs log-safety hardening.

## What This Changes

The PR changes `formatAssistantErrorText` so unrecognized assistant errors return a generic user-facing message, logs a preview for operators, and adds formatter tests for orphaned tool-result and long unknown error strings.

## Best Possible Solution

Refresh the PR against current main so only the true unrecognized catch-all gets generic user copy, all current actionable classifier branches remain covered, and the operator preview is sanitized as redacted-then-truncated log text with regression tests for the linked orphan-call cases and newer diagnostics.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: an unrecognized error such as `No tool call found for function call output with call_id ...` reaches the raw fallback in `formatAssistantErrorText`, and `buildEmbeddedRunPayloads` sends that formatted value as an error reply.

## Solution Assessment

No, not in its current form. The intended direction is right, but the patch should be refreshed to preserve current-main diagnostics and harden the new log preview before merge.

## Security Review

Status: needs_attention

Summary: The diff avoids CI, dependency, lockfile, release, and script changes, but its new logging path needs concrete log-safety fixes before merge.

Concerns:

- **[medium] Preview truncates before redaction:** `src/agents/pi-embedded-helpers/errors.ts:654`
  - body: `redactSensitiveText(raw.slice(0, 500))` can log an unredacted secret fragment when a Bearer/API token begins near the 500-character boundary and the truncated fragment no longer satisfies the default redact pattern length or shape. Redact full text, or a sufficiently overlapped window, before truncating the redacted preview.
