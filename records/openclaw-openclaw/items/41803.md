## Summary

Keep open. The reported raw-error leak is valid and this PR is a plausible focused fix path, but the current head still has blocking correctness and log-safety issues and lacks inspectable real behavior proof.

## What This Changes

The PR changes `formatAssistantErrorText` so the final unrecognized-error fallback logs a redacted preview and returns generic `/new` guidance, with formatter regression tests.

## Best Possible Solution

Refresh or replace this with narrow fallback hardening that genericizes only truly unclassified assistant errors, preserves existing actionable diagnostics, logs redacted and sanitized previews before truncation, and includes redacted live channel proof.

## Reproduction Assessment

Yes, source-level. An assistant message with `stopReason: "error"` and an unclassified `errorMessage` flows through `formatAssistantErrorText`, and `buildEmbeddedRunPayloads` can push that text as a channel-visible `isError` reply.

## Solution Assessment

No. The direction is right, but this patch should preserve current diagnostic carve-outs, redact and pass log previews through `sanitizeForLog` before truncating, refresh against main, and provide inspectable real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

  - body: Current main keeps non-transport proxy/config text and non-streaming JSON/request-validation diagnostics actionable.   - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds a warning log for suppressed raw errors, but the preview is truncated before redaction and does not use the existing log sanitizer.
