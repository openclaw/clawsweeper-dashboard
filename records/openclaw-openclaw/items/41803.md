## Summary

Keep open. The underlying raw-error delivery bug is still source-reproducible on current main, but this PR is not merge-ready because it lacks inspectable real behavior proof and still has correctness/security blockers around broad fallback behavior and log preview handling.

## What This Changes

The branch makes unrecognized assistant errors return generic /new recovery copy, logs a diagnostic preview, preserves a few specific error messages, and adds formatter tests.

## Best Possible Solution

Refresh or replace this with a current-main formatter/payload fix that genericizes truly internal unclassified provider errors, preserves actionable diagnostics, logs only redacted and sanitized previews, and includes redacted live channel proof.

## Reproduction Assessment

Yes, source-level. An assistant message with stopReason "error" and an unclassified errorMessage flows through formatAssistantErrorText, and buildEmbeddedRunPayloads can push that returned text as a channel-visible isError reply.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable shape until it preserves current diagnostic carve-outs, redacts and sanitizes logs before truncation, and supplies inspectable real behavior proof.

## Security Review

Status: needs_attention

Summary: The patch adds logging for suppressed raw errors, but the preview is not redacted or sanitized safely before log interpolation.

Concerns:

- **[medium] Preview truncates before redaction:** `src/agents/pi-embedded-helpers/errors.ts:657`
  - body: redactSensitiveText(raw.slice(0, 500)) can leave a secret fragment unredacted when a bearer token, header, JSON field, or token-like value is cut before the complete sensitive pattern can match.

  - body: The preview only replaces CR/LF, so tabs, ANSI escapes, DEL, and other control characters can still be interpolated into warning logs instead of going through sanitizeForLog.
