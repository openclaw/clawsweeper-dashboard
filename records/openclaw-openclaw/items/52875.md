## Summary

Keep open. Current main still rejects missing target sessions instead of starting or bootstrapping them, no `sessions.start`/`sessions_start` surface is registered, and linked PR #52921 remains the open implementation candidate with closing syntax.

## What This Changes

The issue requests that `sessions_send` recover when a configured target agent has no active session, ideally by starting or prewarming a session and returning clear delivery metadata instead of `No session found` or ambiguous timeouts.

## Best Possible Solution

Review PR #52921 as the current implementation candidate. If it is correct and safe, land it and let GitHub close this issue; otherwise keep this as the canonical report and implement a focused replacement covering idempotent session start, `sessions_send` auto-bootstrap for known agent configs, structured not-found errors, delivery metadata on accepted/timeout paths, regression tests, and docs updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is calling `sessions.send` with a session key whose store entry has no `sessionId`: current `handleSessionSend` returns `INVALID_REQUEST` with `session not found` before any creation path can run. The agent label path similarly fails at `sessions.resolve` before start/bootstrap logic exists.

## Solution Assessment

No for current main; it still lacks the requested recovery path. The intended direction is maintainable if implemented as an additive idempotent start/auto-bootstrap path with tests and protocol compatibility review, while richer active/waiting/read states stay scoped separately if they complicate the bug fix.
