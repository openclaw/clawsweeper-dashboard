## Label Justifications

- `P2`: The original report broke agent-to-agent communication for configured agents without active sessions, but the affected surface is contained to session send/recovery.
- `impact:session-state`: The issue is about configured agents missing active session state after upgrade/model changes.
- `impact:message-loss`: The reported failure prevented cross-agent messages from reaching targets without active sessions.

## Summary

Current main and the latest shipped release now cover the central failure: first-contact sends to a configured agent main session bootstrap that session before sending, and the agent tool can target a configured agent by `agentId`.

## What This Changes

The issue asks for agent-to-agent `sessions_send`/`sessions.send` to recover when a configured target agent has no active session instead of returning “No session found.”

## Best Possible Solution

Keep the shipped bootstrap behavior from https://github.com/openclaw/openclaw/pull/81365; track broader active/waiting/read-state telemetry as a separate enhancement if maintainers want that product surface.

## Reproduction Assessment

No current-main failure repro remains: source and regression tests show configured agent main sessions are created before send. I did not rerun the original Ubuntu/MiniMax setup, but the shipped tests cover the reported missing-target shape.

## Solution Assessment

Yes. Reusing `sessions.create` before first send for configured agent main keys is the narrow maintainable fix, while broader read/active/waiting message-state telemetry should stay separate.
