## Label Justifications

- `P2`: This was a normal-priority agent-to-agent messaging regression with a bounded gateway/agent-session fix that is now shipped.
- `impact:message-loss`: The issue is about inter-agent messages failing or lacking useful delivery feedback when the target session is missing.
- `impact:session-state`: The root symptom was configured agents existing without active session state, causing `sessions_send` routing failures.

## Summary

Close as implemented: current main now creates configured agent main sessions before `sessions.send`/`sessions_send`, so the reported `No session found` dead end is fixed and shipped in `v2026.5.18`. The later ACK/read-state discussion is separate feature work, not a blocker for closing the original missing-session regression.

## What This Changes

The issue asks for agent-to-agent `sessions_send` to recover when a configured target agent has no active session instead of returning `No session found`.

## Best Possible Solution

Keep the shipped `sessions.create`-based bootstrap for missing configured-agent sessions, and handle richer handoff acknowledgements as a separate API/product decision if maintainers want that contract.

## Reproduction Assessment

No for the failure on current main: source and regression tests show missing configured-agent main sessions are created before send. The prior failing behavior was source-reproducible before the merged bootstrap fix.

## Solution Assessment

Yes: the merged solution reuses the existing `sessions.create` contract before send, which is narrower than duplicating session-row setup or adding a separate lifecycle API for this bug. Explicit handoff/read-state semantics are a separate feature track.
