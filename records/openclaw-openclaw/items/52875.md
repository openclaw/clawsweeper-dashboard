## Summary

Keep open. Current main still does not implement the requested recovery path: missing target sessions are rejected before any bootstrap flow, there is no registered session-start method/tool, and the linked implementation PR remains open with closing syntax, so this issue should stay open until that PR is reviewed, landed, replaced, or closed.

## What This Changes

The issue asks for `sessions_send` to recover when a configured target agent has no active session, ideally through an idempotent session-start path and clearer delivery metadata for timeout/accepted sends.

## Best Possible Solution

Review PR #52921 as the active implementation candidate. If it is correct and safe, land it and let the issue close through that linked PR; otherwise keep this as the canonical report and implement a focused replacement covering idempotent session start, `sessions_send` auto-bootstrap for known agent configs, structured not-found errors, delivery metadata on accepted/timeout paths, regression tests, and docs updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is calling `sessions_send` or Gateway `sessions.send` for a target with no stored `sessionId`: current main returns a not-found error before any start flow can run, and label targeting fails through `sessions.resolve` before bootstrapping.

## Solution Assessment

Unclear until PR #52921 is reviewed. The direction is likely right if implemented as an additive, idempotent start/auto-bootstrap path that preserves existing visibility, permissions, and protocol compatibility; active/waiting/read receipts should be separated if they broaden the fix.
