## Summary

Keep open: current main still has the source path that can carry detailed bootstrap truncation diagnostics into WebChat hidden runtime context, and current docs intentionally describe the warning as agent-visible, so the remaining decision is the intended diagnostic surface rather than cleanup closure.

## What This Changes

Requests preventing detailed bootstrap truncation diagnostics from being injected into WebChat runtime context when startup files exceed bootstrap limits.

## Best Possible Solution

Define one canonical truncation diagnostic surface: keep at most concise, non-sensitive agent guidance in runtime context, move raw counts/config-key diagnostics to operator/debug surfaces such as `/context`, `/status`, doctor, or logs, then align docs and regression tests.

## Reproduction Assessment

Yes, from source inspection: use a workspace with a bootstrap file over `bootstrapMaxChars`, send a WebChat `chat.send` turn where agent and transcript prompts differ, and the appended truncation warning can be split into hidden runtime context. I did not perform a live WebChat run in this read-only review.

## Solution Assessment

Unclear: the code path is narrow, but the best fix depends on whether maintainers want detailed truncation diagnostics agent-visible by default. A safer path is to settle the warning surface first, then update runner behavior, tests, and docs together.
