## Summary

Keep open: the traced handoff direction matches the reported startup diagnostic bug, but the PR currently introduces a stop-window race before `startAccount` runs.

## What This Changes

The PR bounds gateway channel account startup tracing to a short handoff span, yields one macrotask before traced channel startup, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Keep the handoff-span approach, but add an abort/manual-stop guard immediately after the handoff wait and cover that race with a focused gateway channel test.

## Reproduction Assessment

Yes for source-level reproduction: current main wraps long-lived `startAccount` in `startupTrace.measure`, and that path keeps the diagnostic phase active until the channel task settles. I did not run a live QQ Bot login, so the live disconnect symptom remains covered by the linked report rather than local E2E proof.

## Solution Assessment

No, not yet: bounding the trace span is the right narrow direction, but the added macrotask handoff must re-check stop/abort state before starting provider work.
