## Summary

Keep open: current main still violates the documented mutable hook context contract, and this PR is a focused, source-backed fix with regression coverage and sufficient real behavior proof. The remaining blocker is mergeability/landing state, not a reason to close the active implementation candidate.

## What This Changes

The PR changes the plugin registry `registerHook` wrapper to pass the original event context with scoped `pluginConfig` cleanup, adds loader regression coverage, and updates the changelog.

## Best Possible Solution

Land this PR or an equivalent narrow registry wrapper fix after resolving mergeability, preserving mutable hook context while restoring or deleting per-handler `pluginConfig` after each plugin hook call.

## Reproduction Assessment

Yes, at source level: a plugin loaded through `api.registerHook('agent:bootstrap')` can assign `event.context.bootstrapFiles`, but current main passes a cloned context while bootstrap resolution later reads the original event context. I did not execute tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. Passing the original context through the generic wrapper and restoring or deleting `pluginConfig` in `finally` is the narrow compatible fix for the existing mutable hook contract; an immutable return-value API would be a larger SDK change.
