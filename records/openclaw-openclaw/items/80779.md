## Summary

Keep open: the underlying current-main bug is real and the PR is close, but the latest patch still suggests a direct approval command for a single uncorrelated pending request, which can steer an operator toward the wrong device after TTL expiry or profile/state mismatch.

## What This Changes

The PR changes `openclaw devices approve <requestId>` local fallback to throw an actionable stale-request diagnostic with pending-request hints and adds CLI fallback regression tests.

## Best Possible Solution

Keep the stale-request diagnostic, but route uncorrelated alternatives through `openclaw devices list` or a preview that shows device/scopes unless the replacement can be correlated to the stale request.

## Reproduction Assessment

Yes, at source level. Current main returns `null` and prints `unknown requestId` when the local fallback misses the supplied ID and the pairing-required gateway error has no requestId, while pairing reconciliation and TTL pruning can remove the original pending ID.

## Solution Assessment

No. The CLI fallback is the right layer, but exactly one other pending request is not enough proof that it replaced the stale request; inspection or correlation is needed before suggesting approval.
