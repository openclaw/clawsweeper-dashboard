The PR changes `src/cli/devices-cli.runtime.ts` to throw an actionable stale request-id diagnostic that routes operators to `openclaw devices list`, with regression tests in `src/cli/devices-cli.test.ts`.

## Best Possible Solution

Merge the CLI-layer diagnostic after standard maintainer validation, preserving explicit operator inspection through `openclaw devices list` and leaving pairing-state semantics unchanged.

## Reproduction Assessment

Yes, at source level. On current main, a bare pairing-required gateway error plus a missing local pending id makes the local approve fallback return `null`, and `runDevicesApproveCommand` emits only `unknown requestId`.

## Solution Assessment

Yes. Improving the CLI fallback diagnostic is the narrowest maintainable fix; changing request-id reconciliation or auto-approving a different pending id would be a broader behavior change.
