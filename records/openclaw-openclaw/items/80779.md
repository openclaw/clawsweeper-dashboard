## Summary

## What This Changes

The PR changes the devices approve local fallback to throw an actionable stale request-id diagnostic that routes operators to `openclaw devices list`, with regression tests for zero, one, and multiple visible pending requests.

## Best Possible Solution

Keep the CLI-layer diagnostic approach, but sanitize the stale requestId before merge so operators get actionable routing without weakening terminal/log safety.

## Reproduction Assessment

Yes, at source level. Current main returns `null` from the local approve fallback when a bare pairing-required gateway error meets a missing local requestId, and the caller reports only `unknown requestId`.

## Solution Assessment

Yes in direction: the CLI fallback diagnostic is the narrowest maintainable solution and avoids changing pairing semantics. The implementation should sanitize the echoed requestId before merge.
