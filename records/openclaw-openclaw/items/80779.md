## Summary

Keep open. The reported stale-requestId path is a real current-main dead end, but this patch should not emit a copyable approval command for the newest pending request without verifying that it is the request the operator meant to approve; the PR also still lacks after-fix proof from a real setup.

## What This Changes

The PR changes the devices approval fallback to throw an actionable stale-requestId error with a fresh pending request hint and adds regression coverage in the devices CLI tests.

## Best Possible Solution

Keep the stale-requestId diagnostic, but route recovery through a verified preview/list flow or include correlated request details before showing an exact approval command; then add real after-fix proof from a patched runtime.

## Reproduction Assessment

Yes. Current main has a clear source path where a missing local approval plus a bare pairing-required gateway error returns `null` and prints `unknown requestId`, and the PR body includes live before-fix output from OpenClaw 2026.5.7; I did not execute a live repro in this read-only review.

## Solution Assessment

No. The CLI-layer diagnostic is the right boundary, but the current implementation should not recommend approving a global newest pending request unless the operator has verified the request details or the code can correlate it to the stale request.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid approving an unverified latest pending request:** `src/cli/devices-cli.ts:198-202`
  - body: This selects the newest pending request across the whole profile and prints a copyable `openclaw devices approve` command when the stale id is missing. If another device has a newer pending request, the operator can be led to approve an unrelated requester, which cuts against the current preview-only safety flow that shows request details before exact approval. Prefer sending the operator through `devices approve --latest`/`devices list`, or include/correlate the pending request details before emitting an exact approval command.
  - confidence: 0.86
