## Summary

This PR addresses a real CLI diagnostic gap, but it should not merge as-is because the new fallback can recommend approving the newest pending request in the whole profile without proving it is the superseding request, and the external PR still lacks after-fix proof from a real patched setup.

## What This Changes

The PR changes `openclaw devices approve` local fallback errors so a stale missing request ID reports likely causes and a suggested current pending approval command, with two regression tests.

## Best Possible Solution

Keep the stale-request diagnostic, but only suggest a direct approve command when the replacement request is unambiguous; otherwise route the operator to preview/list output that shows the request details.

## Reproduction Assessment

Yes. Source inspection shows the no-requestId fallback miss returns `null` and the caller prints `unknown requestId`; the PR body also includes a live before-fix ECS transcript for that path.

## Solution Assessment

No. The CLI fallback is the right layer, but selecting the global freshest pending request and printing a direct approval command is not the safest fix when multiple pending requests may exist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid suggesting an unverified latest pairing request:** `src/cli/devices-cli.ts:200-205`
  - body: The new helper selects the newest pending request in the whole profile and prints a direct `openclaw devices approve <id>` command. If another device created a newer pending request, the operator can copy this command and approve an unrelated requester; keep the direct rerun hint only for an unambiguous/correlated request, or point to the preview/list flow instead.
  - confidence: 0.87
