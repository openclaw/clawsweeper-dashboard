## What This Changes

The PR changes `devices approve` local fallback to throw an actionable stale requestId diagnostic that routes operators through `openclaw devices list`, with focused CLI regression tests.

## Best Possible Solution

Merge the CLI-layer diagnostic after the stale inline comment is aligned with the final list-only behavior and CI is cleared, preserving explicit operator inspection before approving any alternate request.

## Reproduction Assessment

Yes, at source level. Current main returns `null` and prints only `unknown requestId` when the local approve fallback misses the supplied id and the gateway pairing error has no requestId, while current pairing code can replace pending ids when approval snapshots diverge.

## Solution Assessment

Yes. Improving the CLI fallback diagnostic is the narrowest maintainable fix; changing reconciliation ids or auto-approving an alternate pending id would alter approval semantics and carry more operator-safety risk.
