## Summary

Keep this PR open: the reported CLI fallback bug is source-reproducible and the diagnostic is the right shape, but current head still has a P1 security-boundary blocker because it renders the raw stale requestId in terminal-facing output.

## What This Changes

This PR changes the devices approve local fallback to throw an actionable stale-request diagnostic that routes operators to `openclaw devices list`, with regression tests for visible pending-count cases.

## Best Possible Solution

Land the stale-request diagnostic after the displayed requestId is sanitized with `sanitizeForLog`, raw ids remain unchanged for state comparisons, and a control-character regression is added.

## Reproduction Assessment

Yes. Current main returns `null` from the local approve fallback when the supplied id is absent locally and the gateway error has no matching requestId, after which the caller prints only `unknown requestId`.

## Solution Assessment

No, not yet. Routing stale explicit approvals through `openclaw devices list` is the right approval-safety posture, but the rendered requestId must be sanitized before this is merge-safe.
