## Summary

Keep open: the PR addresses a real stale-request dead end, and the updated PR body now has after-fix CLI proof, but the branch still prints a direct approve command for a pending request that is not correlated to the stale ID.

## What This Changes

The PR changes `openclaw devices approve <requestId>` so the local fallback throws an actionable stale-request diagnostic and adds regression tests for zero, one, and multiple visible pending requests.

## Best Possible Solution

Keep the stale-request diagnostic, but route uncorrelated alternatives through `openclaw devices list` or a preview that shows device/scopes before suggesting approval.

## Reproduction Assessment

Yes, at source level. Current main returns `null` and prints `unknown requestId` when local fallback misses the supplied ID and the pairing-required gateway error has no requestId, and the pairing code can replace stale pending IDs during scope-set changes.

## Solution Assessment

No. The CLI fallback is the right layer, but exactly one other pending request is not enough proof that it is the replacement request; TTL/profile mismatch can leave one unrelated pending request and the PR still prints a direct approval command.

## Review Findings
