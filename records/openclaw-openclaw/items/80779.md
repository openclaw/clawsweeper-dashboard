## Summary

Keep this PR open: it fixes a real `openclaw devices approve` stale-request diagnostic gap, but the current patch still renders the operator-supplied request ID without terminal/log sanitization in a device-approval path.

## What This Changes

The PR changes `openclaw devices approve` local fallback so a stale explicit request ID throws an actionable diagnostic that routes the operator to `openclaw devices list`.

## Best Possible Solution

Land the stale-request diagnostic after the displayed request ID is sanitized with the existing terminal/log sanitizer and covered by a focused control-character regression.

## Reproduction Assessment

Yes. Source inspection shows current main returns null when the local fallback cannot approve a request ID and the gateway did not include a requestId, causing the caller to print only `unknown requestId`; I did not run the CLI because this review is read-only.

## Solution Assessment

No, not yet. Routing stale explicit approvals through `openclaw devices list` is the right approval-safety posture, but the rendered request ID must be sanitized before this is merge-safe.
