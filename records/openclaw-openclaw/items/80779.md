## Summary

Keep open: current main still has the dead-end fallback path, while this PR is focused, tested, and has sufficient terminal proof; I found no blocking correctness or security issue.

## What This Changes

The PR replaces the stale local approval fallback's `null` result with an actionable diagnostic that routes operators to `openclaw devices list`, and adds regression tests for zero, one, and multiple visible pending requests.

## Best Possible Solution

Merge the CLI-layer diagnostic after standard maintainer validation, preserving explicit operator inspection through `openclaw devices list` and leaving pairing-state semantics unchanged.

## Reproduction Assessment

Yes, at source level. On current main, a bare pairing-required gateway error plus a missing local pending id makes the local approve fallback return `null`, and `runDevicesApproveCommand` emits only `unknown requestId`.

## Solution Assessment

Yes. Improving the CLI fallback diagnostic is the narrowest maintainable fix; preserving request-id semantics and avoiding auto-approval of a different pending id is safer than changing reconciliation behavior.
