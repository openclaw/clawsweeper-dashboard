## Summary

Keep open: current main still treats an explicit missing `plugins.slots.memory` plugin as a fatal config issue, and the Memory LanceDB docs still present the plugin as bundled/configurable before showing the npm install step. The docs portion is straightforward, but changing gateway startup from fail-closed to fail-soft conflicts with current config recovery docs/tests and needs maintainer policy review first.

## What This Changes

Handle an uninstalled `memory-lancedb` selected in `plugins.slots.memory` more safely at gateway startup and document that the plugin must be installed before configuring the slot.

## Best Possible Solution

Align the Memory LanceDB docs and install flow with its external npm distribution, then make a maintainer-approved startup policy for explicit missing official external memory slots: either actionable warning with no active memory slot, or fail-closed with clearer Docker-safe recovery guidance.

## Reproduction Assessment

Yes, source-reproducible: current main hard-fails an explicit missing `plugins.slots.memory` value and gateway startup throws on invalid snapshots. I did not run the Docker loop, but the validator/startup path matches the reported error.

## Solution Assessment

Unclear. The docs install warning is clearly the right direction, but the gateway soft-fail proposal changes the current fail-closed config policy and should be chosen by maintainers before implementation.
