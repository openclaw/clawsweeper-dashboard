## Summary

Keep open: current main still hard-fails an explicit missing `plugins.slots.memory` reference, while the memory-lancedb docs and package/catalog metadata still support the reporter's install-step mismatch. The related memory issues in the body cover different post-install runtime/tool exposure paths, so they do not supersede this boot-time validation/docs problem.

## What This Changes

The issue requests softer gateway handling when a memory slot names an uninstalled `memory-lancedb` plugin and clearer memory-lancedb installation docs.

## Best Possible Solution

Reuse the existing official-external install-hint path for configured-but-uninstalled memory slot owners, let the gateway boot without binding the missing slot, and update the memory-lancedb docs to require `openclaw plugins install @openclaw/memory-lancedb` before slot config.

## Reproduction Assessment

Yes from source: a config with explicit `plugins.slots.memory: "memory-lancedb"` and no discovered plugin hits the hard-error branch in `src/config/validation.ts`, and `src/config/io.ts` throws that invalid config during startup. I did not run a live Docker loop repro in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to extend the existing official-external warning/install-hint behavior to the memory slot while preserving hard failures for unknown slot ids, plus add the missing install-first docs.
