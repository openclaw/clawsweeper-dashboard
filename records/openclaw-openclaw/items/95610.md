## Summary

Keep open: current main and the latest release still contain the reported `message_tool_only` prompt and tool-schema churn, and the related OpenAI cache-TTL PRs address a different idle-gap pruning problem rather than this within-window prefix-stability bug.

## What This Changes

The issue asks OpenClaw to stop turn-scoped `message_tool_only` system/tool prompt changes on OpenAI models so automatic prefix caching can reuse stable bytes.

## Best Possible Solution

Make provider-visible tool schemas byte-stable and move turn-scoped visible-reply guidance into a late, per-turn surface for OpenAI/Codex paths, with focused regression tests and redacted live OpenAI cache proof.

## Reproduction Assessment

Yes for the source-level churn path: current main still changes both the message tool description and system-prompt messaging section when `sourceReplyDeliveryMode` flips. I did not independently run a live current-main OpenAI session to remeasure the cache-hit rate.

## Solution Assessment

Yes directionally: stable tool schemas plus moving turn-scoped delivery guidance out of OpenAI's cache-sensitive prefix is the narrowest maintainable fix. The exact destination for that guidance needs maintainer review because it affects visible reply delivery semantics.
