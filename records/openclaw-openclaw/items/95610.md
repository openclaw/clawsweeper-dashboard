## Summary

Keep open: current main still conditionally appends the reported per-turn visible-reply hint to the `message` tool description, and current prompt assembly also injects runtime delivery guidance into the system prompt, so the core cache-stability problem is not implemented or obsolete.

## What This Changes

The issue asks OpenClaw to keep OpenAI request prefixes byte-stable by removing per-turn message-tool description churn and relocating volatile runtime prompt sections outside the OpenAI cached prefix.

## Best Possible Solution

Make the `message` tool schema byte-stable across turns, move per-turn visible-reply guidance into turn-specific user/runtime content for OpenAI-family calls, and add focused cache-stability coverage plus live provider cache proof before closing.

## Reproduction Assessment

Yes for the reported message-tool schema mutation: current main conditionally changes the `message` tool description for `message_tool_only` turns. The full provider cache-miss impact was not live-reproduced in this read-only review.

## Solution Assessment

Yes for the tool-schema part: stabilizing tool descriptions and moving per-turn delivery hints out of the OpenAI cached prefix is the narrowest maintainable direction. The wider system-prompt boundary should be designed carefully so message-tool-only visible replies continue to work.
