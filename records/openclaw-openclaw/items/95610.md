## Summary

Keep open: current main and the latest release still contain the two reported churn surfaces, so this is not implemented or obsolete. Source shows the `message` tool description changes for `message_tool_only` turns, and OpenAI transport strips the internal cache-boundary marker while still sending the dynamic system-prompt suffix as normal prompt text.

## What This Changes

The issue asks OpenClaw to keep OpenAI prompt-cache prefixes byte-stable by moving per-turn message-tool and runtime guidance out of cached OpenAI prompt/tool prefixes.

## Best Possible Solution

Make `message` tool schemas byte-stable across turns, move per-turn visible-reply/runtime guidance into non-prefix turn content for OpenAI/Codex paths, and add focused cache-stability coverage plus live provider cache proof before closing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection shows the reported message-tool schema and dynamic system-prompt churn paths on current main, but the full OpenAI cache-miss rate still needs live telemetry against current main.

## Solution Assessment

Is this the best way to solve the issue? Yes at the direction level: stable tool schemas plus moving volatile OpenAI/Codex guidance out of the cacheable prefix is the narrowest maintainable fix, but it should be reviewed with message-delivery and live cache proof.
