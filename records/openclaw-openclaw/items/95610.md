## Summary

Keep open: current main still makes `message_tool_only` a per-turn tool-schema and system-prompt input, and OpenAI/Codex request construction treats tools and instructions as cache-relevant request content, so the reported cache-stability problem is not implemented on main.

## What This Changes

The issue asks OpenClaw to stop per-turn message delivery guidance from changing OpenAI/Codex cached prompt prefixes and tool schemas while preserving visible reply delivery.

## Best Possible Solution

Keep the message-tool-only reply contract, but make the OpenAI/Codex cached tool schema and stable prompt prefix byte-stable across delivery-mode toggles, with regression coverage for both provider payload shape and visible reply delivery.

## Reproduction Assessment

Yes, source inspection shows current main still changes provider-visible tool descriptions and messaging guidance when `message_tool_only` toggles; the review did not run a live model session.

## Solution Assessment

Unclear without maintainer direction: the best repair should stabilize provider-visible cache inputs while preserving source-reply delivery semantics, rather than simply deleting the dynamic guidance.
