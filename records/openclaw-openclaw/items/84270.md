## Summary

Keep open: current main and the latest shipped release still account completion delivery from the later parent announce result, so a parent `NO_REPLY` can still drive retry bookkeeping even when the child already delivered the Slack completion reply.

## What This Changes

The issue requests that `expectsCompletionMessage` subagent cleanup credit a prior route/thread-scoped Slack completion delivery instead of retrying after a later parent `NO_REPLY`.

## Best Possible Solution

Persist or derive requester route/thread-scoped committed completion-delivery evidence and have cleanup credit that evidence before retrying a parent announce that intentionally returns `NO_REPLY`.

No high-confidence live reproduction was run. Source inspection gives a medium-confidence path: prior Slack delivery is not represented in the run record, the later parent `NO_REPLY` returns `delivered: false`, and cleanup retries from that result.

## Solution Assessment

Yes, the proposed direction is the right boundary if it uses a narrow route/thread-scoped committed-delivery signal. Prompt-only `NO_REPLY` changes or generic dedupe would not solve the registry accounting gap by themselves.
