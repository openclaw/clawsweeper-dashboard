## Label Justifications

- `P2`: This is a normal-priority Slack/subagent delivery-accounting bug with a clear but limited affected workflow.
- `impact:message-loss`: The issue is about a completion reply that was already delivered to Slack being treated as undelivered by the completion announce path.
- `impact:session-state`: The subagent registry can leave `announceRetryCount` and pending-final-delivery state inconsistent with the actual requester thread state.

## Summary

Keep open: current main still treats the later parent announce result as the completion-delivery source of truth, so a parent `NO_REPLY` after an already-delivered Slack reply can still drive retry/give-up bookkeeping instead of crediting prior committed delivery evidence.

## What This Changes

The issue requests that subagent completion cleanup account for earlier committed Slack completion delivery when a later parent-mediated announce returns `NO_REPLY`.

## Best Possible Solution

Persist or derive route/thread-scoped committed completion-delivery evidence and have cleanup credit it before retrying a parent announce that intentionally returns `NO_REPLY`.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection shows a medium-confidence repro path: parent announce delivery returns false after `NO_REPLY`, and cleanup retries because prior child/requester delivery evidence is not consulted.

## Solution Assessment

Yes, the proposed direction is the right boundary: credit committed completion delivery before retrying, while keeping retries when no route/thread-scoped evidence exists.
