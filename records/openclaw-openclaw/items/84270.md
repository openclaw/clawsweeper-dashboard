## Label Justifications

- `P2`: This is a bounded Slack/subagent completion accounting bug with limited blast radius but real retry and delivery-status impact.
- `impact:message-loss`: The issue is about a completion reply that reached Slack being treated as undelivered by the announce path.
- `impact:session-state`: The registry can leave retry count, pending-final-delivery, and task delivery status inconsistent with the actual requester thread state.

## Summary

Keep open: current main still bases completion cleanup on the later parent announce delivery result, so a parent `NO_REPLY` after an already-delivered Slack completion can still be treated as undelivered and retried.

## What This Changes

The issue asks OpenClaw to credit already-committed requester-channel completion delivery before retrying a later parent-mediated `NO_REPLY` announce.

## Best Possible Solution

Persist or derive route/thread-scoped committed completion-delivery evidence and let cleanup finalize delivered before retrying a later parent `NO_REPLY`, while preserving retries when no matching evidence exists.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an expected completion whose child already delivered to Slack can still run a parent announce that returns no visible/message-tool output, causing `delivery.delivered` to be false and cleanup to retry. I did not run a live Slack replay because the original route artifacts are redacted and this review is read-only.

## Solution Assessment

Yes, the requested direction is the right boundary: credit committed, route/thread-scoped completion delivery before treating a parent `NO_REPLY` as failure. The safer alternative is not to treat all `NO_REPLY` as success, but to require matching delivery evidence for the requester route.
