## Label Justifications

- `P2`: The bug affects a bounded Slack/subagent completion-delivery accounting path, with visible retry noise and state drift but not a whole-runtime outage.
- `impact:message-loss`: The issue is about a completion reply that was visible in Slack being treated as undelivered by the announce path.
- `impact:session-state`: The registry can keep pending-final-delivery and retry-count state inconsistent with the requester thread's actual delivered completion.

## Summary

Current `main` still has the reported accounting gap: completion cleanup follows the later parent announce result, and no route/thread-scoped prior-delivery proof is recorded or consulted before retrying a parent `NO_REPLY`. This should stay open as a narrow subagent delivery-accounting bug, not be closed as already fixed by the stale-wake recovery work.

## What This Changes

The issue asks OpenClaw to credit an already committed Slack subagent completion reply so a later parent `NO_REPLY` does not cause announce retries and retry-limit give-up state.

## Best Possible Solution

Persist or derive requester route/thread-scoped completion-delivery proof and let cleanup credit that proof before retrying a parent announce that intentionally returns `NO_REPLY`, while preserving retries when no matching proof exists.

## Reproduction Assessment

No live high-confidence reproduction was run because the Slack run artifacts are redacted and this is a read-only review. Source inspection gives a medium-confidence reproduction path: parent `NO_REPLY` produces `delivery.delivered=false`, and cleanup retries because prior route/thread delivery evidence is not recorded or consulted.

## Solution Assessment

Yes, the proposed direction is the right boundary: credit committed completion delivery before retrying the parent announce, while keeping existing retry behavior for runs without matching delivery proof.
