## Label Justifications

- `P2`: This is a normal-priority Slack/subagent delivery-accounting bug with a bounded affected workflow.
- `impact:message-loss`: The issue is about a completion reply already delivered to Slack being treated as undelivered by completion announce accounting.
- `impact:session-state`: The retry count and pending-final-delivery state can diverge from the actual requester thread state.

## Summary

Keep open: current main still credits only the later parent announce result, so a parent `NO_REPLY` with no new message-tool delivery can drive retry bookkeeping even when the child already delivered the Slack completion reply.

## What This Changes

The issue reports that a Slack-thread subagent completion already delivered by the child can still be treated as an undelivered parent announce when the parent responds `NO_REPLY`, causing retry-limit state.

## Best Possible Solution

Persist or derive route/thread-scoped committed completion-delivery evidence and let cleanup credit it before retrying a parent announce that intentionally returns `NO_REPLY`, while preserving retries when no such evidence exists.

## Reproduction Assessment

Yes at source level: make a completion-mode run with no `completionAnnouncedAt`, have the parent announce return no messaging-tool evidence after `NO_REPLY`, and cleanup follows the false `didAnnounce` path into retry bookkeeping. I did not run a live Slack replay in this read-only review.

## Solution Assessment

Yes: the proposed boundary is the narrow maintainable fix, provided the evidence is tied to the requester route/thread and the existing retry behavior stays intact when no committed delivery evidence exists.
