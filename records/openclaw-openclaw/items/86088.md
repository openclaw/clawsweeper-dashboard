## Summary

## What This Changes

The PR changes task maintenance to recover terminal `lost` cron rows with a backing-session-missing error from durable cron history, clear stale terminal errors when recovery has no error, and add focused regression coverage.

## Best Possible Solution

Land the narrowed recovery only after task/session-state owners accept the persisted-state rewrite boundary and the seeded CLI proof as sufficient for this bug.

## Reproduction Assessment

Yes, from source inspection: current main only runs durable cron recovery for active cron tasks, so a seeded terminal `lost` cron row with matching durable history is not recovered on main. The PR body supplies after-fix CLI proof for the same state shape.

## Solution Assessment

Yes, subject to owner acceptance: the patch keeps recovery scoped to cron rows that were marked lost by the backing-session-missing path and adds boundary tests. The remaining question is policy/ownership around rewriting terminal persisted state, not a narrower code repair.
