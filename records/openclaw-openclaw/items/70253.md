## Summary

Keep open: current main partially fixed Mattermost streaming config support, but inbound Mattermost dispatch still forces generic block streaming off and Mattermost channel-level bot/agent-to-agent routing remains unresolved rather than implemented by `tools.agentToAgent`.

## What This Changes

The issue requests that Mattermost honor disabled streaming config and that configured bot/agent-to-agent messages in shared channels reach the intended agents instead of being silently dropped.

## Best Possible Solution

Split the work: finish the Mattermost streaming-mode/config fixes through the existing streaming follow-ups, then design explicit Mattermost bot-authored channel ingress with reliable bot identity, `allowBots` semantics, and loop protection before changing A2A routing.

## Reproduction Assessment

Yes, source-level reproduction is high confidence for the remaining behavior: inbound Mattermost still forces `disableBlockStreaming: true`, mention gating drops unmentioned shared-channel posts, and routing returns one selected agent. I did not run a live Mattermost or custom WebSocket scenario.

## Solution Assessment

No complete current-main solution exists. The streaming portion has narrow Mattermost follow-ups already open, while the channel-level A2A portion needs a product and trust-model decision before adding Mattermost `allowBots` or fan-out behavior.
