## Summary

Keep open. Current main still has the source-backed duplicate-emission path described by the reporter: cache-ttl metadata can trail the canonical embedded assistant row, and the CLI gap-fill dedupe only checks the newest physical transcript row before appending a second api: "cli" assistant message.

## What This Changes

The issue asks OpenClaw to stop the CLI cache-ttl completion path from adding duplicate assistant messages to persistent session history.

## Best Possible Solution

Update the CLI embedded assistant gap-fill dedupe to look past trailing metadata such as openclaw.cache-ttl while still stopping at real user or different assistant turns, then add focused regression coverage for that transcript order.

## Reproduction Assessment

Yes, at source level: cache-ttl appends a custom row after the canonical assistant turn, and the CLI gap-fill dedupe only inspects the newest physical JSONL row before appending an api: "cli" assistant. I did not run a live multi-turn model reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the existing embedded assistant gap-fill dedupe skip trailing metadata rows, rather than broadly filtering persisted CLI rows from every future model snapshot.
