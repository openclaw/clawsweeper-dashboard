## Summary

Keep open. Current main still has a source-backed path where cache-TTL custom metadata can sit after the canonical embedded assistant turn, defeat the embedded CLI gap-fill dedupe, and persist a duplicate `api: "cli"` assistant row into session history.

## What This Changes

The issue reports that CLI multi-turn sessions duplicate assistant transcript rows after cache-TTL metadata, poisoning later context and causing repeated model output.

## Best Possible Solution

Teach embedded assistant gap-fill to look past trailing metadata such as `openclaw.cache-ttl` while stopping at real user or assistant boundaries, and add focused regression coverage for that custom-tail case.

## Reproduction Assessment

Yes, at source level: current main appends cache-TTL custom metadata after the canonical embedded assistant turn, then gap-fill reads only the physical tail and appends `api: "cli"` when that tail is not assistant text. I did not execute the live multi-turn model flow in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to make embedded assistant gap-fill skip trailing metadata rows for dedupe without broadly filtering all CLI transcript rows from replay or changing cache-TTL semantics.
