## Summary

Keep open. Current main still has a source-backed path where the cache-ttl custom row lands after the canonical embedded assistant turn, then the CLI gap-fill dedupe only inspects that physical tail and appends a duplicate `api: "cli"` assistant row.

## What This Changes

The issue reports duplicate CLI assistant transcript rows after cache-ttl metadata in repeated CLI agent sessions, which can poison later model context and cause repeated outputs.

## Best Possible Solution

Teach embedded CLI transcript gap-fill to look past trailing cache-ttl/custom metadata for the immediately preceding assistant turn, while stopping at real user turns and preserving legitimate repeated replies.

## Reproduction Assessment

Yes, at source level: cache-ttl appends a custom row after the canonical embedded assistant turn, then gap-fill reads only the physical tail and appends `api: "cli"` when that tail is not an assistant. I did not run the live multi-turn model flow in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to make the gap-fill dedupe skip trailing metadata rows, not to add a new mode or broadly filter all CLI transcript rows from model context.
