## Summary

Keep open. Current main still has a source-backed path where cache-ttl metadata after the canonical embedded assistant turn defeats the CLI gap-fill dedupe and persists a duplicate assistant row into session history.

## What This Changes

The issue requests preventing cache-ttl bookkeeping plus CLI delivery mirroring from duplicating assistant turns in persisted session history.

## Best Possible Solution

Teach embedded assistant gap-fill to dedupe across trailing cache-ttl/custom metadata while stopping at real user or assistant turns, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: cache-ttl appends a custom row after the embedded assistant turn, then gap-fill reads only the physical tail and appends `api: "cli"` when that tail is not an assistant. I did not execute the live multi-turn model flow in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow gap-fill dedupe change that skips trailing metadata such as `openclaw.cache-ttl` without broadly filtering all CLI transcript rows from model snapshots.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83427. Current main appends `openclaw.cache-ttl` custom entries after embedded Pi attempts; `agent-command.ts` then calls `persistCliTurnTranscript` with `embeddedAssistantGapFill`, and `persistTextTurnTranscript` only compares `readTailAssistantTextFromSessionTranscript`. Because that helper returns undefined when the newest JSONL row is `type: "custom"` / `openclaw.cache-ttl`, the gap-fill appends a duplicate assistant row with `api: "cli"`. Implement a narrow fix that skips trailing metadata/custom rows for embedded assistant gap-fill but still appends a repeated reply after an intervening user message. Prefer source-level helper/test changes in `src/config/sessions/transcript.ts` and `src/agents/command/attempt-execution.ts`; do not add a config option or broad snapshot filtering unless the narrow fix proves insufficient. Add a user-facing `CHANGELOG.md` fixes entry if the repair PR owns release notes.
