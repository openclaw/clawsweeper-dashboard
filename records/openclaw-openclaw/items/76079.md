## Summary

Keep open. The latest head fixed the earlier standalone export and cache-eviction blockers, but the cached descriptor path can still return fewer non-optional tools than the uncached path for the same allowlisted request, so the PR needs a narrow repair before merge.

## What This Changes

The PR replaces plugin tool factory-result caching with a descriptor cache captured from registered tools, updates plugin/tool docs and tests, and adds a missing-runtime provider fallback.

## Best Possible Solution

Keep the descriptor cache, but make warmed cached results match the uncached runtime path for non-optional multi-tool plugins while preserving the bounded cache and standalone preload export.

## Reproduction Assessment

Yes. A plugin with manifest tools ["a", "b"] whose non-optional factory returns both tools and is requested with allowlist ["a"] will return both tools on the first uncached call but only "a" after the descriptor cache warms.

## Solution Assessment

No. Descriptor caching is a reasonable direction, but the current cached-path filtering changes observable tool visibility; the safer fix is to align cached filtering with the existing runtime path or skip cache hits for partial non-optional allowlists.

## Review Findings
