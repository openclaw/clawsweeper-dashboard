## Summary

Keep open. Current main still lacks a first-class Project model, Projects dashboard surface, and Gateway project API, and the recent discussion keeps this as the canonical core/product tracker rather than a duplicate of narrower session or runtime-skill requests.

## What This Changes

The issue requests first-class Projects in the Gateway dashboard with project/session/workspace binding, scoped tools/skills, and runtime constraint metadata or warnings.

## Best Possible Solution

Add an explicit Project domain model with additive Gateway project protocol/storage and a Control UI project list/detail/switching surface that reuses existing agents, sessions, workspaces, skill filters, and sandbox policies where appropriate.

## Reproduction Assessment

Not applicable: this is a feature request, not a runtime bug. The current-main gap is source-verifiable because Projects are absent from navigation, dashboard render paths, Gateway method discovery, session schemas, tests, and Control UI docs.

## Solution Assessment

Unclear until maintainer design. A first-class core Project model is plausible, but storage shape, protocol methods, UI hierarchy, and advisory-versus-enforced constraint semantics need product/API/security decisions before implementation.
