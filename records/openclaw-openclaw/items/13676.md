## Summary

Current main now covers meaningful adjacent ground through the Workboard plugin—named board metadata, default/card workspaces, linked sessions, suggested skills, and runtime-duration metadata—but it still does not provide the requested first-class Projects dashboard, durable home-session/workstream model, or project-level tool and execution-placement enforcement. The item remains useful as the product-design request for deciding whether Workboard boards should evolve into Projects and what guarantees that model provides.

## What This Changes

Add a first-class Project domain to the Gateway dashboard with durable session and workspace bindings, scoped tools/skills, and runtime constraints.

## Best Possible Solution

Evolve Workboard boards into the canonical project-capable model, adding explicit project/home-session and workstream relationships plus workspace/doc metadata, while designing enforceable tool and runtime policy as a separate reviewed security contract.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken defined behavior. Current-main source inspection provides high-confidence evidence that only adjacent Workboard primitives exist.

## Solution Assessment

No, not as a separate parallel Project subsystem. The narrowest maintainable direction is to decide whether Workboard boards become the durable project owner, then add the missing UI, relationship, and explicitly defined enforcement contracts there.
