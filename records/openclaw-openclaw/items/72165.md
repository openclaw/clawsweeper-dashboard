## Summary

Keep open. Current main has already mitigated the ordinary sessions.list path by filtering, sorting, limiting, and capping transcript-field hydration before row construction, but two central lookup paths from the report remain: sessionId resolution still builds unbounded session rows before filtering, and MCP conversation_get still reuses a 500-row conversations list with transcript-backed fields enabled.

## What This Changes

Request to make session listing and exact session lookup metadata-first so search/filter/limit and single-session queries avoid materializing every row or synchronously reading transcripts.

## Best Possible Solution

Land a narrow metadata-first lookup path: exact sessionId/key resolution should scan store metadata without building Gateway session rows, and MCP conversation_get should hydrate at most the requested conversation descriptor. Keep the already-shipped bounded list behavior, and leave any persistent summary index or transcript storage redesign to a separate storage-performance track.

## Reproduction Assessment

Yes. Static reproduction against current main is high confidence: call sessions.resolve with sessionId or MCP conversation_get in a large store, and the code still routes through unbounded listSessionsFromStore/listConversations before filtering to one target.

## Solution Assessment

No for the full SessionSummaryIndex as the immediate repair; that is a larger storage redesign. The best near-term solution is a focused exact-lookup helper or additive RPC that filters metadata first and hydrates only the matching row while preserving existing resolver semantics.
