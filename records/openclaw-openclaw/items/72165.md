## Summary

Keep open. Current main has mitigated the ordinary bounded `sessions.list` path, but exact `sessionId` resolution and MCP `conversation_get` still reuse list-style row materialization before selecting one session.

## What This Changes

The issue requests avoiding full Gateway session row materialization and transcript reads before filtering, limiting, or resolving a single session/conversation.

## Best Possible Solution

Land a narrow metadata-first lookup path for exact key/sessionId and MCP conversation lookup, hydrating at most the matching row while leaving a persistent `SessionSummaryIndex` redesign to a separate storage-performance track.

## Reproduction Assessment

Yes. Static reproduction is high confidence: in a large store, `sessions.resolve` with `sessionId` still calls unbounded `listSessionsFromStore`, and MCP `conversation_get` still requests up to 500 conversations before filtering to one.

## Solution Assessment

No for the full `SessionSummaryIndex` as the immediate repair; that is broader than needed. The safer near-term fix is a focused metadata-first resolver/conversation lookup that preserves existing selection semantics.
