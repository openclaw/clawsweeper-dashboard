## Summary

Keep open. Current main mitigates the ordinary sessions.list pagination path, but exact sessionId resolution still builds unbounded Gateway session rows before filtering, and MCP conversation_get still broad-lists conversations before selecting one target.

## What This Changes

The issue requests metadata-first session listing and lookup so filters, limits, and exact session selection avoid unnecessary transcript-backed row hydration.

## Best Possible Solution

Land a narrow metadata-first exact lookup path and have MCP conversation_get hydrate at most the requested descriptor, while leaving persistent summary indexing to a separate storage-performance design.

## Reproduction Assessment

Yes. Static reproduction is high confidence: on current main, sessions.resolve with sessionId enters the unbounded listSessionsFromStore branch, and MCP conversation_get enters listConversations with limit 500 before selecting one session.

## Solution Assessment

No for the full SessionSummaryIndex as the immediate repair; that is a larger storage redesign. A focused metadata-first resolver and MCP direct descriptor lookup is the narrowest maintainable fix for the remaining proven behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72165 with a focused exact-lookup performance PR. In `src/gateway/sessions-resolve.ts`, avoid resolving sessionId by calling unbounded `listSessionsFromStore`; scan session-store metadata first, build/hydrate only matching rows if needed, and preserve duplicate sessionId errors, includeGlobal/includeUnknown, agentId, spawnedBy visibility, legacy canonicalization, and deleted-agent rejection semantics. In `src/mcp/channel-bridge.ts`, change `getConversation` so it does not fetch a 500-row conversations list to find one session; use a direct metadata-first lookup or a bounded sessions.list query that hydrates at most the requested descriptor. Add regression tests that fail if these paths call broad row materialization. Do not implement a persistent `SessionSummaryIndex`, split sessions.json storage, or broaden #69451 memory work in this PR. Add a `CHANGELOG.md` fix entry because this is user-facing performance behavior.
