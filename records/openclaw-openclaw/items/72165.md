## Summary

Keep open. Current main has partially fixed the normal `sessions.list` path by filtering and limiting before row construction, but the report still has valid residual work: sessionId resolution still materializes list rows before filtering, and MCP `conversation_get` still reuses a 500-row conversation list with transcript-backed fields enabled.

## What This Changes

Request to avoid O(N) session list and lookup latency by filtering before transcript hydration, resolving session IDs directly, and avoiding MCP single-conversation lookup through a large sessions list.

## Best Possible Solution

Keep the issue open and land a narrow metadata-first lookup optimization: resolve exact session IDs without building Gateway session rows, make MCP `conversation_get` use an exact or narrowed descriptor path, and leave any persistent `SessionSummaryIndex` or transcript storage redesign to a separate storage-performance decision.

## Reproduction Assessment

Yes. Static reproduction against current main is high confidence: `sessions.resolve` with `sessionId` still calls an unbounded `listSessionsFromStore`, and MCP `conversation_get` still asks for up to 500 conversations with derived title/last-message hydration enabled.

## Solution Assessment

Unclear for the proposed full `SessionSummaryIndex` as the immediate fix. The narrowest maintainable next step appears to be metadata-first exact lookup plus a narrowed MCP descriptor path, while the persistent index should be scoped separately.

Do not implement a persistent SessionSummaryIndex, split sessions.json, or change the session storage format.
