## Summary

Keep open. Current main has mitigated the normal session list path by filtering, sorting, and applying limits before building rows, but the issue's exact lookup paths remain valid: sessionId resolution still builds session rows before filtering, and MCP conversation_get still performs a 500-row sessions.list request with transcript-backed fields enabled.

## What This Changes

The issue requests avoiding full session-row materialization and synchronous transcript reads for session list and exact lookup paths, including direct sessionId lookup and MCP getConversation behavior.

## Best Possible Solution

Land a narrow metadata-first lookup optimization: resolve exact session IDs without building Gateway session rows, make MCP conversation_get use an exact or narrowed descriptor path that hydrates at most one conversation, and leave any persistent SessionSummaryIndex or transcript storage redesign to the broader storage-performance track.

## Reproduction Assessment

Yes. Static reproduction against current main is high confidence: sessions.resolve with sessionId still calls unbounded listSessionsFromStore before filtering, and MCP getConversation still calls listConversations with limit 500 while transcript-backed fields default on.

## Solution Assessment

Unclear for the full proposed SessionSummaryIndex as the immediate fix. The narrowest maintainable next step is direct metadata lookup for exact session IDs plus an MCP exact descriptor path, preserving existing visibility, duplicate-match, and deleted-agent semantics.

## Repair Work Prompt

Fix the remaining performance issue in https://github.com/openclaw/openclaw/issues/72165. Current main already applies sessions.list filtering/limit before row hydration, so do not implement a persistent SessionSummaryIndex or change the session storage format. Instead, make exact session lookup metadata-first: update sessions.resolve by sessionId so it scans or indexes the combined session store entries directly, preserves includeGlobal/includeUnknown, agentId, spawnedBy visibility, duplicate-match errors, key-as-sessionId fallback, deleted-agent validation, and legacy key behavior, but does not call listSessionsFromStore/buildGatewaySessionRow for the exact lookup. Then update MCP conversation_get so it no longer calls listConversations({ limit: 500 }) to find one session; use an exact gateway lookup or narrowed descriptor path that hydrates at most the matched conversation. Add focused tests proving exact sessionId resolution and MCP conversation_get avoid broad transcript-backed list hydration, and keep existing session resolution semantics intact. Add a user-facing CHANGELOG.md fix entry crediting Thanks @6tizer. Do not broaden this into the storage/memory redesign tracked by #69451.
