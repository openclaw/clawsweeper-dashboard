## Summary

Keep open. Current main bounds the ordinary sessions.list pagination path enough to explain the related #57715 cleanup, but exact sessionId resolution and MCP conversation_get still build transcript-backed session rows before selecting one target.

## What This Changes

The issue requests a faster session summary and lookup path that filters or resolves sessions before transcript-backed row hydration.

## Best Possible Solution

Add a metadata-first exact lookup path for sessionId/session key resolution and have MCP conversation_get hydrate only the requested descriptor, leaving persistent summary indexing to a separate storage-performance design.

## Reproduction Assessment

Yes. Static reproduction against current main is high confidence: sessions.resolve with sessionId enters an unbounded listSessionsFromStore path before equality filtering, and MCP conversation_get calls listConversations(limit: 500) before selecting one session key.

## Solution Assessment

No for the full SessionSummaryIndex as the immediate fix; a focused direct lookup and descriptor hydration path is the narrower maintainable repair for the verified remaining behavior.

Add a metadata-first exact lookup path so sessions.resolve with sessionId scans cheap session-store metadata and only hydrates/builds rows for the matched session(s), preserving includeGlobal/includeUnknown, agentId, spawnedBy visibility, duplicate sessionId errors, key equality fallback, and deleted-agent validation.
