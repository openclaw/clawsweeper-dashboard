## Summary

Keep open. Current main has partial session-list mitigations, but the reported exact-lookup paths remain: sessionId resolution still materializes Gateway session rows before filtering, and MCP single-conversation lookup still goes through a broad 500-row sessions list with transcript-backed fields enabled.

## What This Changes

The issue asks OpenClaw to avoid O(N) session list and lookup costs by filtering before transcript hydration, resolving session IDs directly, and avoiding MCP single-conversation lookup through a large sessions list.

## Best Possible Solution

Keep the issue open and land a narrow exact-lookup optimization: resolve sessionId matches from store metadata before row hydration, make MCP conversation_get hydrate at most the requested conversation, and leave persistent summary indexing or storage-format redesign to the broader session-storage performance work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: sessions.resolve with sessionId calls an unbounded listSessionsFromStore path, and MCP conversation_get requests up to 500 list rows with transcript-backed fields enabled. Runtime tests were not run because this pass was read-only.

## Solution Assessment

Unclear for the full SessionSummaryIndex as the immediate fix. The safer first step is a metadata-first exact lookup plus a narrow MCP descriptor path; persistent indexing is a larger storage design decision.
