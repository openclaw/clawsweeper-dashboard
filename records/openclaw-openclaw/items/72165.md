## Summary

Keep open. Current main has useful mitigations for normal session listing, but the reported lookup paths are still valid: sessionId resolution still builds session rows before filtering, and MCP single-conversation lookup still goes through a broad 500-row list request with transcript-backed fields enabled.

## What This Changes

The issue requests avoiding O(N) session list and lookup costs by filtering before transcript hydration, resolving session IDs directly, and avoiding MCP single-conversation lookup through a large sessions list.

## Best Possible Solution

Keep the issue open and land a narrow metadata-first lookup optimization: resolve exact session IDs without building Gateway session rows, make MCP conversation_get use an exact or narrowed descriptor path, and track any persistent SessionSummaryIndex or transcript storage redesign separately with the broader storage-performance work.

## Reproduction Assessment

Yes. Static reproduction against current main is high confidence: sessions.resolve with sessionId still calls an unbounded listSessionsFromStore path, and MCP conversation_get still requests up to 500 sessions with derived title and last-message hydration enabled.

## Solution Assessment

Unclear for the full proposed SessionSummaryIndex as the immediate fix. The best next step is narrower: metadata-first exact lookup plus an MCP descriptor path that hydrates at most the returned conversation, leaving persistent indexing as a separate design decision.
