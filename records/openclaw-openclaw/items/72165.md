## Summary

Keep open. Current main partially addressed the list path by applying filters and limit before row construction, but the reported direct lookup and MCP conversation paths still reuse hydrated session rows, and returned rows can still synchronously read transcript usage, derived title, and last-message data.

## What This Changes

The issue requests avoiding full session-row materialization and synchronous transcript reads for session listing and direct session/conversation lookup.

## Best Possible Solution

Keep the current limit-before-enrichment behavior, add metadata-only key/sessionId lookup plus a direct MCP conversation descriptor path that avoids list-row hydration, and treat persistent write-time summary indexing as a separate storage design if maintainers choose it.

## Reproduction Assessment

Yes. Static reproduction against current main is high-confidence: sessionId resolution still calls listSessionsFromStore without a limit, MCP conversation_get still calls sessions.list with limit 500, and hydrated rows still synchronously read transcripts when stored summary fields are missing or derived preview fields are requested.

## Solution Assessment

Unclear. The proposed SessionSummaryIndex may be the right longer-term storage direction, but the narrowest maintainable next step is likely direct metadata lookup plus an MCP descriptor path that avoids transcript-backed row hydration.
