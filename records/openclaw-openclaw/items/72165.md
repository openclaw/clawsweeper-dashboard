## Summary

Keep open. Current main has the safer limit-before-row-construction optimization for normal session listing, but the remaining report is still valid: sessionId resolution still builds full session rows, MCP conversation lookup still calls a 500-row sessions.list with transcript-derived fields enabled, and returned rows can still synchronously hydrate transcript-backed usage/title/preview data.

## What This Changes

The issue requests making session list and single-session lookup paths avoid full session-row materialization and synchronous transcript reads before filtering, pagination, or direct lookup.

## Best Possible Solution

Keep the current limit-before-enrichment behavior, add metadata-only lookup/descriptor paths for sessionId resolution and MCP conversation_get that preserve existing visibility and duplicate-resolution semantics, and treat a persistent write-time summary index as a separate storage design decision that can also coordinate with #69451.

## Reproduction Assessment

Yes. Static reproduction against current main is high-confidence: sessionId resolution still calls listSessionsFromStore without a limit, and MCP conversation_get still makes a 500-row sessions.list request with transcript-derived fields enabled by default.

## Solution Assessment

Unclear. The requested SessionSummaryIndex may be the right long-term storage direction, but the narrowest maintainable next step appears to be direct metadata lookup plus an MCP descriptor path that avoids transcript-backed row hydration.
