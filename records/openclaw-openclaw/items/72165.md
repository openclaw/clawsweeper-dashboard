## Summary

Keep open. Current main has additional mitigations for normal session listing, including limit-before-row-construction, bounded usage reads, capped preview/title hydration, and indexed child links, but the report is still valid for direct lookup paths: sessionId resolution still materializes all session rows, and MCP conversation_get still reuses a 500-row sessions.list call with transcript-derived fields enabled.

## What This Changes

The issue requests making session list and lookup paths avoid full session-row materialization and synchronous transcript reads before filtering, pagination, or direct lookup.

## Best Possible Solution

Keep the current bounded list mitigations, add metadata-only direct lookup or descriptor paths for sessions.resolve and MCP conversation_get that preserve existing visibility and duplicate-resolution semantics, and coordinate any persistent write-time summary index with the broader session-storage performance work in #69451.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Static inspection on current main shows sessionId resolution still calls listSessionsFromStore without a limit, and MCP conversation_get still asks for a 500-row sessions.list response with transcript-derived fields enabled.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The narrowest maintainable next step appears to be direct metadata lookup plus an MCP descriptor path, while the proposed SessionSummaryIndex is a broader storage design that should be sequenced with #69451.
