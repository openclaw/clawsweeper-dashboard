Keep open. Current main still has the reported static coupling in the embedded attempt runner and Discord monitor, and the related merged harness-seam work did not complete a prepared runner context or narrower Discord process/preflight/provider split.

The issue requests reducing import coupling in the embedded attempt runner and Discord monitor by introducing prepared context/service bundles and narrower typed boundaries.

## Best Possible Solution

Keep this as the canonical staged refactor: introduce behavior-preserving prepared service/context bundles around runEmbeddedAttempt, then split Discord preflight/process/provider responsibilities behind narrower local helpers and tests.

## Solution Assessment

Yes directionally, but not as one broad dependency-injection rewrite. A staged prepared-runner context followed by narrower Discord helper/context types is the safer maintainable path.
