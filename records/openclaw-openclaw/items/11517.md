## Summary

Keep open. Current main still reproduces the reported static coupling: the embedded attempt runner and Discord monitor files remain large/import-heavy, the runner still resolves major services inline, the harness seam still forwards the same attempt params, and Discord processing still consumes a broad preflight context. The related merged #70743/#70772 work added adjacent runtime seams but did not complete this prepared-context or Discord monitor decomposition.

## What This Changes

The issue requests a behavior-preserving architecture refactor to reduce import coupling in the embedded attempt runner and Discord monitor by introducing prepared run/service contexts and narrower monitor boundaries.

## Best Possible Solution

Keep this as the canonical staged architecture refactor: first introduce behavior-preserving prepared context/service bundles around runEmbeddedAttempt, then narrow Discord preflight/process/provider responsibilities behind typed local helpers with focused regression coverage before the changed gate.

## Reproduction Assessment

Not applicable as a runtime bug, but yes for the static architecture signal. Current-main file sizes, import counts, runEmbeddedAttempt’s inline setup, harness param aliasing, and Discord context/process signatures reproduce the reported coupling.

## Solution Assessment

Yes directionally, but only as a staged maintainer-led refactor. A prepared runner context plus narrower Discord helper/context types is safer than one broad dependency-injection rewrite spanning both surfaces at once.
