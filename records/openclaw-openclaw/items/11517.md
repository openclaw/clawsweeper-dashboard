## Summary

Keep open. Current main still has the reported import-heavy embedded attempt runner and Discord monitor boundaries, and the linked harness-seam work explicitly left the pure split/import cleanup as follow-up.

## What This Changes

Reduce static import coupling in the embedded attempt runner and Discord monitor by introducing prepared context/service bundles and narrower local boundaries.

## Best Possible Solution

Keep this as the canonical staged refactor: add behavior-preserving prepared context/service bundles around runEmbeddedAttempt, then narrow Discord preflight/process/provider responsibilities behind typed local helpers and targeted tests.

## Reproduction Assessment

Yes for the reported static architecture signal, though not applicable as a runtime bug. Current-main file sizes, import counts, runner setup flow, harness parameter forwarding, and Discord context/process signatures reproduce the coupling.

## Solution Assessment

Yes directionally, but not as one broad DI rewrite. A staged prepared-runner context followed by narrower Discord helper/context types is the safer maintainable path.
