## Summary

Keep open. Current main still routes native Windows Gateway install through Scheduled Tasks, creates new tasks with logon/interactive flags, and keeps the per-user Startup-folder fallback; the related PowerShell ScheduledTasks implementation in #68149 is still open.

## What This Changes

The issue asks native Windows `openclaw gateway install` to keep the Gateway connected after the launching PowerShell window is closed.

## Best Possible Solution

Define the supported native Windows persistence contract, then either make the managed task survive terminal closure with clear privilege tradeoffs or document native Windows as login-scoped and direct headless users to WSL2/systemd.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main creates native Windows installs with logon/interactive Scheduled Task flags and keeps fallback persistence as a per-user Startup-folder login item. Live Windows confirmation is still needed for the exact close-PowerShell disconnect behavior.

## Solution Assessment

No, current main is not the best solution for the reported expectation because the native Windows install remains session/logon-oriented. #68149 is a plausible implementation candidate, but it is still open and changes privilege-sensitive behavior.
