## Summary

Keep open. Current main still uses a logon-oriented native Windows Scheduled Task path with a per-user Startup-folder fallback, and the related PowerShell ScheduledTasks implementation in #68149 remains open and needs maintainer review before this can be treated as resolved.

## What This Changes

Windows users report that a Gateway installed with `openclaw gateway install` disconnects after closing the launching PowerShell window instead of continuing in the background.

## Best Possible Solution

Define and implement the supported native Windows persistence contract: either make the managed task service-like enough to survive terminal closure with documented privilege tradeoffs, or document native Windows as login-scoped and direct headless users to the WSL2/systemd path.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main creates native Windows installs with logon/interactive Scheduled Task flags and keeps fallback persistence as a per-user Startup-folder login item. Live Windows confirmation is still needed for the exact close-PowerShell disconnect behavior.

## Solution Assessment

No, current main is not the best solution for the reported expectation because the native Windows install remains session/logon-oriented. #68149 is a plausible implementation candidate, but it is still open and changes privilege-sensitive behavior.
