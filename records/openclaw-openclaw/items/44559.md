## Label Justifications

- `impact:crash-loop`: The reported failure is that the Gateway process disconnects or stops when the Windows launching window/session is closed.

## Summary

Keep open: current main and the latest shipped tag still use the native Windows `gateway.cmd` plus interactive `ONLOGON` Scheduled Task model, and the related PowerShell ScheduledTasks implementation is open but conflicting and not merge-ready. The duplicate window/persistence report at https://github.com/openclaw/openclaw/issues/81017 was already closed into this issue, so this remains the canonical thread for the Windows Gateway persistence decision.

## What This Changes

The issue reports that native Windows `openclaw gateway install` disconnects or stops the Gateway when the PowerShell or visible task window is closed.

## Best Possible Solution

Define the supported native Windows Gateway persistence contract, then land or replace the Scheduled Task implementation so it survives terminal/window closure, preserves or explicitly migrates fallback behavior, updates docs/tests, and includes real Windows proof.

## Reproduction Assessment

Yes for source-level reproduction: current main and `v2026.5.20` still stage `gateway.cmd` and register an `ONLOGON` task with `/IT` when a user is available, and Microsoft documents `/IT` as interactive-only for a logged-on user. I did not run a live Windows close-PowerShell test in this Linux read-only review. ([learn.microsoft.com](https://learn.microsoft.com/en-us/windows/win32/taskschd/schtasks))

## Solution Assessment

No shipped solution currently satisfies the requested background persistence. The related PowerShell ScheduledTasks PR may be directionally useful, but the best path must preserve or deliberately migrate current fallback behavior, fix its review blockers, and include real Windows proof.
