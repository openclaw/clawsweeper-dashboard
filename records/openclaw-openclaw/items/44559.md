## Summary

Keep open. Current main and the latest release still create native Windows gateway installs through the interactive/logon Scheduled Task path and retain a per-user Startup-folder fallback, while the related PowerShell ScheduledTasks PR remains open with correctness, policy, and real-Windows-proof blockers.

## What This Changes

The issue asks for Windows gateway installs to keep the Gateway connected after the launching PowerShell window is closed.

## Best Possible Solution

Define the supported native Windows persistence contract, then land a Windows Scheduled Task or hidden-launcher design that survives terminal/window closure, preserves or explicitly replaces the no-admin fallback, updates docs/tests, and includes real Windows proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still registers the gateway with logon/interactive Scheduled Task flags and a `gateway.cmd`-based fallback. A live Windows run is still needed to prove the exact close-PowerShell disconnect symptom and any eventual fix.

## Solution Assessment

No, current main is not the best solution for the reported expectation because native Windows managed startup remains session/logon-oriented. The PowerShell ScheduledTasks PR may be a direction, but it still needs correctness fixes, policy approval, and real Windows behavior proof.
