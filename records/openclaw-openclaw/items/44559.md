## Summary

Keep open: current main and the latest shipped release still use the Windows `gateway.cmd` launcher through an interactive `ONLOGON` Scheduled Task, so the reported Gateway disconnect/window-lifetime behavior is not resolved. The remaining fix needs maintainer review because changing this path affects Windows persistence policy, privileges, fallback behavior, and upgrade proof.

## What This Changes

The issue asks for Windows `openclaw gateway install` to keep the Gateway running after the launching PowerShell or visible console window is closed.

## Best Possible Solution

Define the supported native Windows Gateway persistence contract, then land or replace the Scheduled Task implementation so it survives terminal/window closure, preserves or explicitly migrates fallback behavior, updates docs/tests, and includes real Windows proof.

## Reproduction Assessment

Yes for source-level reproduction: current main and `v2026.5.19` still stage `gateway.cmd` and register an `ONLOGON` interactive Scheduled Task with `/IT` when a user is available. I did not run a live Windows close-PowerShell test in this Linux read-only review.

## Solution Assessment

No shipped solution currently satisfies the requested background persistence. The existing PowerShell ScheduledTasks PR may be the right direction, but the best merge path must preserve or deliberately migrate current fallback behavior and include real Windows proof.
