## Summary

Current main still uses the native Windows `schtasks` gateway install path as a logon-oriented Scheduled Task (`/SC ONLOGON`, `/RL LIMITED`, and `/RU <user> /NP /IT` when a task user is available) with a per-user Startup-folder fallback. That matches the reported session-scoped behavior rather than fixing it. Related PR #68149 is open and relevant, but it is unmerged and changes the Windows command-execution, ScheduledTasks, and admin privilege model, so this should stay open for maintainer review.

## Best Possible Solution

Keep this issue open for maintainer product/design handling. The best path is to decide the supported native Windows persistence model, then either land a tested service-like Scheduled Task or PowerShell ScheduledTasks implementation that survives closing PowerShell/session with documented privilege and security tradeoffs, or explicitly document native Windows gateway install as login-session scoped and point headless users to WSL2/systemd. Review #68149 as the current implementation candidate before using it to close this issue.
