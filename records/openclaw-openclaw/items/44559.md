## Summary

Keep open. Current main still uses the native Windows Scheduled Task path with logon/interactive creation flags and a per-user Startup-folder fallback, while the related PowerShell ScheduledTasks implementation remains open with policy and correctness questions.

## What This Changes

The issue asks for native Windows `openclaw gateway install` to keep Gateway connected after the launching PowerShell window is closed.

## Best Possible Solution

Define the supported native Windows persistence contract, then either make managed startup survive terminal closure with clear privilege tradeoffs or document native Windows as login-scoped and direct headless users to WSL2/systemd.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main creates native Windows installs with logon/interactive Scheduled Task flags and keeps fallback persistence as a per-user Startup-folder login item. Live Windows confirmation is still needed for the exact close-PowerShell disconnect behavior.

## Solution Assessment

No, current main is not the best solution for the reported expectation because the native Windows install remains session/logon-oriented. The related PowerShell ScheduledTasks PR may be a direction, but it is still open and changes privilege-sensitive behavior.
