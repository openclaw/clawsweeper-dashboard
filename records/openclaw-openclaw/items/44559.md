## Summary

Keep open. Current main still creates the native Windows gateway task as a logon-oriented Scheduled Task and keeps the fallback as a per-user Startup-folder login item, so it does not satisfy the reported expectation that the Gateway survive closing the launching PowerShell/session. Related PR #68149 is relevant but still open and changes Windows privilege and ScheduledTasks behavior, so it should be reviewed rather than treating this as resolved.

## What This Changes

The issue asks for native Windows `openclaw gateway install` persistence that keeps the Gateway connected after the launching PowerShell window or user-session context closes.

## Best Possible Solution

Define the supported native Windows persistence contract clearly. Either add a tested service-like Scheduled Task or PowerShell ScheduledTasks mode that survives terminal/session closure with documented privilege tradeoffs, or explicitly document native Windows gateway install as login-scoped and direct headless users to the WSL2/systemd path.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main routes native Windows install to `schtasks` and creates new tasks with `/SC ONLOGON`, `/RL LIMITED`, and `/IT` when a user is available. Live Windows confirmation is still needed for the exact close-PowerShell behavior.

## Solution Assessment

No, current main is not the best solution for the reported expectation because it keeps the native install session/logon-oriented. PR #68149 is a plausible implementation candidate, but it is not yet proven or merged and needs maintainer review for privilege and security tradeoffs.
