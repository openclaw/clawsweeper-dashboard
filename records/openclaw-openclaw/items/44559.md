## Summary

Keep open. Current main and latest release v2026.5.12 still use the native Windows interactive/logon Scheduled Task path with a gateway.cmd Startup-folder fallback, while the related PowerShell ScheduledTasks PR remains open, conflicting, and blocked on correctness/policy/real-Windows proof.

## What This Changes

The issue asks native Windows gateway installs to keep the Gateway running after the launching PowerShell window or session is closed.

## Best Possible Solution

Define the supported native Windows persistence contract, then land a Windows Scheduled Task or hidden-launcher design that survives terminal/window closure, preserves or explicitly replaces the no-admin fallback, updates docs/tests, and includes real Windows proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence because current main still registers the gateway with logon/interactive Scheduled Task flags and a gateway.cmd-based fallback; live Windows proof is still needed for the exact close-PowerShell symptom and any eventual fix.

## Solution Assessment

No. Current main is not the best solution for the reported expectation because native Windows managed startup remains session/logon-oriented; the PowerShell ScheduledTasks PR may be a direction, but it still needs correctness fixes, policy approval, and real Windows behavior proof.
