## Summary

Keep open. Current main still routes native Windows gateway install through logon/interactive Scheduled Task creation with a per-user Startup-folder fallback, and the direct PowerShell ScheduledTasks replacement remains open with correctness and policy blockers.

## What This Changes

This issue asks for native Windows gateway install to keep the Gateway connected after the launching PowerShell window is closed.

## Best Possible Solution

Define the native Windows persistence contract, then either make managed startup survive terminal closure with clear privilege tradeoffs or document native Windows as login-scoped and direct headless users to WSL2/systemd.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main creates native Windows installs with logon/interactive Scheduled Task flags and keeps fallback persistence as a per-user Startup-folder login item. Live Windows confirmation is still needed for the exact terminal-close disconnect behavior.

## Solution Assessment

No, current main is not the best solution for the reported expectation because native Windows install remains session/logon-oriented. The PowerShell ScheduledTasks PR may be the right direction, but it still needs correctness fixes and maintainer policy judgment.
