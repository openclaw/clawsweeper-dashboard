## Summary

Keep open: the sanitizer-at-boundary direction is right, but this PR still needs stronger OpenClaw/plugin-path proof and has a remaining redirect header path that can still hit the Symbol-keyed `Headers` crash.

## What This Changes

The PR adds a `sanitizeHeadersInit` helper and applies it to proxy fetch, runtime fetch, redirect-header retention, and debug proxy capture paths to strip non-string header keys before native header handling.

## Best Possible Solution

Normalize `HeadersInit` at every OpenClaw-to-native `Headers` or fetch boundary, avoid mutating caller-owned init objects, add a single-line changelog entry and focused regression coverage, then merge after real OpenClaw/plugin proof on Node 22.

## Reproduction Assessment

Yes, source-reproducible with high confidence: native `Headers` throws on Symbol-keyed header records, and current main passes `init.headers` unchanged through common proxy/runtime fetch paths. I did not establish a full live Discord plugin run in this checkout.

## Solution Assessment

No, not yet. Sanitizing at the network boundary is the right fix direction, but the patch should also cover redirect body-header rewrites, avoid mutating debug-proxy caller init, and satisfy the changelog and real-behavior proof gates.

## Review Findings

  - body: The PR sanitizes proxy/runtime and cross-origin retention paths, but `fetch-guard.ts` still calls `new Headers(headers)` in `dropBodyHeaders`.
