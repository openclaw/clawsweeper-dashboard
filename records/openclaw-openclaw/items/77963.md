## Summary

Keep open: the underlying Symbol-keyed `HeadersInit` crash is real, and the sanitizer direction is plausible, but this diff still leaves ordinary non-FormData runtime/proxy fetch calls able to pass Symbol-keyed headers directly to undici.

## What This Changes

This PR adds a `sanitizeHeadersInit` helper and applies it in selected core fetch, redirect-header, and proxy-capture paths to strip Symbol keys before native header handling.

## Best Possible Solution

Normalize `HeadersInit` at every OpenClaw-to-undici/global-fetch boundary for all init shapes, avoid mutating caller-owned init objects, and add regression coverage for Symbol-keyed records across runtime fetch, proxy fetch, redirect handling, and debug capture.

## Reproduction Assessment

Yes, source inspection plus a Node one-liner give a high-confidence reproduction of the native `Headers` crash when a header record contains a Symbol key. I did not run the full Discord.js plugin flow, but current main has direct `new Headers(init.headers)` and undici dispatch paths that can receive that shape.

## Solution Assessment

No, not as written. The sanitizer should run before all early returns or immediately before every undici/global fetch dispatch, rather than only inside the FormData normalization branches.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Sanitize proxy fetch headers before dispatch:** `src/infra/net/proxy-fetch.ts:51`
  - body: `normalizeInitForUndici` still returns the original `init` for every non-FormData request, so `makeProxyFetch` and env-proxy fetch can pass a Symbol-keyed `init.headers` straight to undici. That leaves normal JSON/header-only requests able to throw the same ByteString error this PR is meant to fix; normalize headers for every init shape before dispatch.
  - confidence: 0.88
- **[P2] Sanitize runtime fetch headers before early returns:** `src/infra/net/runtime-fetch.ts:63`
  - body: `normalizeRuntimeRequestInit` still returns `init` unchanged when there is no body or the body is already runtime-compatible, so `fetchWithRuntimeDispatcher` can still call undici with Symbol-keyed headers. Move header sanitization ahead of those returns, or build a sanitized init independently of FormData conversion.
  - confidence: 0.86
