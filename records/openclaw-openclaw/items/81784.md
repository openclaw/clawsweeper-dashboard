## Summary

Keep open: this is a useful feature PR, but the current patch can expose the wrong inbound turn identifiers when same-session CLI turns overlap because the shared session-scoped file is rewritten before the queued CLI execution window.

## What This Changes

The PR adds an inbound-turn context, writes it to a session-scoped JSON file for CLI agent runs, exposes that file through `OPENCLAW_INBOUND_TURN_FILE`, wires it from auto-reply execution, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the env-file contract, but refresh the file inside the serialized CLI turn lifecycle and add regression coverage proving queued same-session turns cannot clobber each other's identifiers.

## Reproduction Assessment

For the requested feature, reproduction is not applicable because this adds a new runtime contract. For the blocking review finding, yes: source inspection shows two same-session CLI calls can both run prepare before the first queued execution reads the shared session-scoped file.

## Solution Assessment

No: a stable env var pointing to a rewritten file is a reasonable shape for live sessions, but the current write location is too early. The safer implementation writes or refreshes the file within the queued execution window for the active turn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move the turn-file write into the run queue:** `src/agents/cli-runner/prepare.ts:253`
  - body: `prepareCliRunContext` runs before `executePreparedCliRun` enters `enqueueCliRun`, but this new path is session-scoped. If two turns for the same CLI session arrive close together, the later prepare call can overwrite the JSON file before the earlier queued or running turn's wrapper reads it, so the wrapper may reply with the wrong message/thread identifiers. Refresh the file inside the serialized execution window, or otherwise make the file lifecycle per-active-turn safe.
  - confidence: 0.9
