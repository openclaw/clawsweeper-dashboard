## Summary

Keep open. Current main still has the `update.run` continuation producer gap: the restart continuation consumer exists, but `update.run` neither accepts `continuationMessage` nor writes `payload.continuation`, and the tests still assert that update sentinels omit it. Open PR #74362 is the active implementation candidate, so this issue should remain open until that or an equivalent fix lands.

## What This Changes

The issue reports silent Telegram/Discord reply loss when `openclaw update` restarts the gateway mid-agent-turn and asks for update-run continuation, graceful drain or fallback notice, post-reset bridge behavior, and agent guidance.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix that threads `continuationMessage` through the update protocol, restart request parsing, agent gateway tool, and successful update restart sentinel. Then split remaining graceful-drain, post-reset bridge, or coalesced-restart recovery work into focused follow-ups if maintainers want those tracked separately.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, a session-scoped `update.run` writes a restart sentinel without `continuation`, and existing tests assert the missing field.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge handling are adjacent recovery work, not substitutes for the missing producer field.
