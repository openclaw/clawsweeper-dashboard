## Summary

Keep open. Current main still has the `update.run` continuation producer gap: restart continuations are consumed after boot when present, but `update.run` still cannot accept `continuationMessage`, does not persist `payload.continuation`, and the current tests still assert update sentinels omit it. Open PR #74362 is the active implementation candidate, not a merged fix.

## What This Changes

The issue reports that running `openclaw update` during an in-flight Telegram or Discord turn can restart the gateway before buffered assistant output is delivered, causing silent message loss and no recovery bridge.

## Best Possible Solution

Review and land PR #74362, or an equivalent narrow additive fix, so `continuationMessage` flows through the update protocol, restart-request parsing, `update.run` sentinel creation, and the agent gateway tool. After that, track graceful drain, post-reset bridge behavior, and coalesced continuation loss in narrower follow-ups such as #74424.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is to call `update.run` from an agent session while expecting a continuation: current main rejects or drops `continuationMessage`, writes no `payload.continuation`, and tests assert the field remains absent. I did not run a live Telegram update because this review is read-only.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge behavior are adjacent recovery work rather than substitutes for the missing producer field.
