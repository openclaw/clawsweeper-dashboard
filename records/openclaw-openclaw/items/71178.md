## Summary

Keep open. Current main still has the `update.run` continuation producer gap: the restart sentinel consumer can drain a continuation when one exists, but the update RPC does not accept `continuationMessage`, does not persist `payload.continuation`, and its tests still assert the field is absent. Open PR #74362 remains the active implementation candidate rather than proof that the issue is fixed.

## What This Changes

The issue asks OpenClaw to avoid losing Telegram/Discord replies when `openclaw update` restarts the gateway while an agent turn is still in progress.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix that threads `continuationMessage` through the update protocol, restart request parsing, agent gateway tool, and successful update restart sentinel. Then split any remaining graceful-drain, post-reset bridge, or coalesced-restart recovery work into focused follow-ups.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, an `update.run` request cannot carry `continuationMessage`, writes a restart sentinel without `continuation`, and tests assert that missing field for session-scoped update sentinels.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge handling are adjacent recovery work, not substitutes for the missing producer field.
