## Summary

Keep open. Current `main` still has the `update.run` continuation producer gap: restart continuations can be consumed after boot when present, but `update.run` still cannot accept `continuationMessage`, does not persist `payload.continuation`, and tests still assert that update sentinels omit it. Open PR #74362 is the active implementation candidate, not proof of a merged fix. https://github.com/openclaw/openclaw/issues/71178

## What This Changes

The issue reports that running `openclaw update` while an agent turn is in progress can restart the gateway before buffered Telegram/Discord replies are delivered, causing the turn to be silently lost.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix that threads `continuationMessage` through the update protocol, restart request parsing, `update.run` sentinel creation, and the agent gateway tool. After that, split any remaining graceful-drain, post-reset bridge, or coalesced-restart recovery work into narrower follow-ups.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is that current `main` rejects `continuationMessage` for `update.run`, writes an update restart sentinel without `continuation`, and has tests asserting the field stays absent. I did not run a live Telegram update because this review is read-only.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge behavior are adjacent recovery work rather than substitutes for the missing producer field.
