## Summary

Keep open. Current main still lacks the `update.run` continuation producer needed for the reported message-loss path: restart continuations can be consumed after boot when present, but `update.run` cannot accept `continuationMessage`, does not write `payload.continuation`, and tests still assert that update sentinels omit it. Open PR #74362 is the active implementation candidate, not a merged fix.

## What This Changes

Prevent `openclaw update` invoked during an agent turn from silently losing Telegram or Discord replies by preserving or resuming the pending response across the gateway restart.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix that threads `continuationMessage` through the update protocol, restart request parsing, `update.run` sentinel creation, and the agent gateway tool. Then split any remaining graceful-drain, post-reset bridge, or coalesced-restart recovery work into focused follow-ups if those are still desired.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is that current main rejects `continuationMessage` for `update.run`, writes an update restart sentinel without `continuation`, and has tests asserting that the field stays absent; I did not run a live Telegram update because this review is read-only.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge behavior are adjacent recovery work, not substitutes for the missing producer field.
