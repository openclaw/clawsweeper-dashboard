## Summary

Keep open. Current main still has the `update.run` continuation producer gap: the restart sentinel consumer can deliver a continuation when present, but the update protocol does not accept `continuationMessage`, the update handler does not persist `payload.continuation`, and the agent tool does not forward the field on `update.run`. Open PR #74362 remains the active implementation candidate rather than proof of a merged fix.

## What This Changes

The issue reports that running `openclaw update` or `update.run` during an active Telegram/Discord agent turn restarts the gateway and loses buffered assistant output without a continuation, fallback, or bridge message.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix that threads `continuationMessage` through the update protocol, restart request parsing, the agent gateway tool, and successful update restart sentinels, with tests changed from expecting an absent continuation to expecting the restart-continuation contract.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main cannot accept or forward `continuationMessage` for `update.run`, persists no `payload.continuation`, and has tests asserting that the field is absent.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract is the smallest maintainable fix; graceful drain and post-reset bridge handling are adjacent recovery work, not substitutes for this missing producer field.
