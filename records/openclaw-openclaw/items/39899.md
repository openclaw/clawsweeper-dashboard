## Summary

Keep open: current main does not yet have surrogate-specific recovery, and this PR addresses a plausible production loop. It still needs maintainer review and rebase because existing text sanitization only partially overlaps the bug, and one fallback path would still expose raw provider text.

## What This Changes

The PR adds a surrogate-encoding error detector, exports it through the embedded helpers barrel, formats those errors with reset guidance, and resets the active session transcript/store when the runner detects that failure before reply.

## Best Possible Solution

Rebase this PR onto current main, add focused coverage for the detector, friendly formatting, runner reset, and structured replay payloads, then land the narrowest recovery path that prevents repeated provider rejection without hiding useful diagnostics.

Yes. A focused unit reproduction can place a lone surrogate in replayed session content or OpenAI Responses function-call arguments and verify that current main has no surrogate-specific reset or friendly formatter path; live provider reproduction was not run in this read-only review.

## Solution Assessment
