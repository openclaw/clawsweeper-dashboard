Keep open: current main still lacks the surrogate-specific detector/recovery path, and the PR addresses a plausible source-reproducible infinite-loop bug. The branch needs real behavior proof and code fixes before merge.

## What This Changes

The PR exports a surrogate-encoding error detector, formats friendly copy for surrogate JSON failures, and resets the active session transcript/store when the reply runner catches that error.

## Best Possible Solution

Land a refreshed branch or replacement on current main that repairs surrogate-bearing request fields where practical, detects provider surrogate JSON failures, and rotates the session through resetReplyRunSession with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible from current main: OpenAI replay sanitizes assistant text but forwards string tool-call arguments as-is, so a stored lone surrogate can enter the outbound provider payload shape reported in the PR. I did not run a live provider request in this read-only review.

## Solution Assessment

No, not as-is. The detector and recovery direction are useful, but the reset should route through resetReplyRunSession, the reset-unavailable fallback needs friendly copy, tests need updating, and contributor real behavior proof is missing.
