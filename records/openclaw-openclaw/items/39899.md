## Summary

Keep this PR open. Current main has partial surrogate mitigation for ordinary text payloads, but it still lacks the PR's surrogate-specific error detector, friendly invalid-JSON formatting, and runner-level session reset path; unsanitized replay fields such as OpenAI Responses function-call arguments remain a plausible way to hit the reported loop.

## What This Changes

The PR adds a surrogate-encoding error detector, exports it through the pi embedded helpers barrel, formats those invalid-JSON errors with user-facing reset guidance, and resets the active session transcript/store when the runner detects such a failure before reply.

## Best Possible Solution

Rebase this PR onto current main, add focused tests for the detector, friendly formatting, runner reset, OpenAI Responses function-call arguments, and Anthropic tool_use input replay, then choose the narrow final fix: either keep the detector/reset recovery path or pair it with recursive replay sanitization where provider payloads can still carry lone surrogates.

## Reproduction Assessment

Yes. A focused unit reproduction can build session history with a lone surrogate in replayed content or OpenAI Responses function-call arguments, then verify current main reaches provider payload construction without surrogate-specific recovery; a live provider loop was not run in this read-only review.

## Solution Assessment

Mostly yes. The PR follows existing runner recovery patterns and addresses the user-visible loop, but it should also cover the missing fallback-copy path and prove whether reset recovery alone is preferable to recursively sanitizing replayed structured payloads.
