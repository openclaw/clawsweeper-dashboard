## Summary

Keep open: this is a focused PR, but the proposed ordering conflicts with the Matrix poll processing contract and would let entries outside the max_selections truncation window affect displayed totals.

## What This Changes

The PR changes Matrix poll result parsing so duplicate answer IDs are deduped before the max selection cap is applied, with a regression test for that ordering.

## Best Possible Solution

Preserve Matrix truncate-before-dedupe semantics unless maintainers explicitly approve and document a deliberate product-level divergence from MSC3381.

## Reproduction Assessment

Yes from source inspection: current main truncates the filtered answer array before `Set` deduping, so `["a1", "a1", "a2"]` under `max_selections=2` produces only `a1`. The Matrix MSC text indicates that behavior is the expected processing contract rather than a confirmed bug.

## Solution Assessment

No: moving `slice` after `Set` is a plausible user-friendly interpretation, but it is not the best fix while OpenClaw tracks MSC3381 semantics. A spec-locking test or maintainer-approved divergence would be safer.
