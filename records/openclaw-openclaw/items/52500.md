## Summary

Keep open. Current main still lacks a central numeric-bound hint for Zod `too_big`/`too_small` config issues, and the active closing PR remains unmerged, so closing this now would drop a valid diagnostics follow-up.

## What This Changes

The issue requests explicit accepted maximum/minimum values in config validation errors for schema-enforced numeric ceilings and floors.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73220 or an equivalent central mapper change that adds numeric bound hints with focused regression coverage and correct origin/inclusive handling.

## Reproduction Assessment

Yes. Source inspection shows bounded config fields still flow through `mapZodIssueToConfigIssue()`, and the current mapper has no `too_big`/`too_small` branch to normalize maximum or minimum guidance.

## Solution Assessment

Yes. The central mapper is the right place to solve this once for schema-generated config errors; per-field custom messages would miss other numeric constraints and duplicate policy across schemas.
