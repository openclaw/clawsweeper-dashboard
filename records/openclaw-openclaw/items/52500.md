## Summary

Keep open. Current main still has the reported numeric ceiling and the central config validation mapper still only enriches allowed-value cases, while open PR #73220 explicitly targets this issue and should be reviewed or closed before the issue is resolved.

## What This Changes

Request that schema validation errors for numeric ceiling-constrained config values explicitly state the maximum allowed value, starting with `session.agentToAgent.maxPingPongTurns` and applying generically.

## Best Possible Solution

Keep this issue open until PR #73220 lands or an equivalent central `mapZodIssueToConfigIssue()` change adds generic numeric maximum/minimum hints with focused regression coverage for both ceiling and floor constraints.

## Reproduction Assessment

Yes. The source-level reproduction is validating a config object with `session.agentToAgent.maxPingPongTurns: 10`: the schema caps it at `5`, and current main has no central `too_big`/`too_small` enrichment beyond returning the native Zod message.

## Solution Assessment

Yes. A generic mapper enhancement with focused validation tests is the narrowest maintainable fix; per-field custom schema messages would not cover all schema-enforced ceilings and floors.
