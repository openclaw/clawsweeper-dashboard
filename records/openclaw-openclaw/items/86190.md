## Summary

Keep open. Current main and the latest release still have a source-backed gap for sandboxed plugin skills: plugin skills are published through generated symlinks, while the sandbox sync copies skill roots without dereferencing those symlink roots into real sandbox-local directories.

## What This Changes

The issue asks OpenClaw to make plugin-provided skills readable inside sandboxed agent sessions by syncing or remapping their skill files into the sandbox workspace like other advertised skills.

## Best Possible Solution

Materialize validated plugin-generated skill roots into the sandbox workspace as real directories and ensure the prompt for sandboxed runs points at the sandbox-local skill copy, while preserving symlink escape protections for arbitrary skill contents.

## Reproduction Assessment

Yes, source-reproducible but not live-run: enable a plugin-declared skill, run a non-rw sandbox so syncSkillsToWorkspace runs, and the current copy path handles a generated plugin skill symlink without dereferencing or supplying the validated real target. A focused regression test can assert the sandbox target prompt uses target-workspace skill files rather than generated plugin-skill host paths.

## Solution Assessment

Yes, the best fix is a narrow sandbox-sync repair for generated plugin skill roots. Broadly allowing host plugin-skill paths or dereferencing every copied symlink would be less maintainable and riskier for containment.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/86190. Plugin-declared skills are published as generated plugin-skill symlinks, but sandbox skill sync currently copies entry.skill.baseDir with fs.promises.cp without dereferencing, so plugin skill roots can remain host-backed or be filtered out instead of becoming sandbox-readable workspace skills. Implement a narrow fix that materializes validated generated plugin skill roots into targetWorkspaceDir/skills as real directories and makes the sandbox prompt use those target-workspace paths. Do not broadly dereference arbitrary symlinks inside skills, do not weaken path containment, and preserve existing managed symlink behavior. Add regression coverage with a fake plugin skill symlink: sync into a target workspace, build the target prompt, assert the target skills/<name>/SKILL.md exists as a real directory copy and the prompt does not expose the generated plugin-skill host path. Add a CHANGELOG.md fix entry because this is user-visible.
