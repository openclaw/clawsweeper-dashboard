## Summary

Keep open: current main fixed several Xiaomi/MiMo 400 paths, but the exact custom `openai-completions` Token Plan shape from this issue is not proven fixed, and source inspection shows the Xiaomi empty-array-items compatibility flag is still provider-owned rather than endpoint-class-wide for arbitrary custom provider ids.

## What This Changes

The issue reports a 2026.5.12 upgrade regression where a custom OpenAI-compatible Xiaomi Token Plan provider fails with `400 Param Incorrect` and fallback auth reports a missing API key.

## Best Possible Solution

Apply the Xiaomi empty-array-items compatibility by detected Xiaomi endpoint/custom-provider shape, keep the existing SecretRef credential contract, and leave the narrower auth display/runtime question to https://github.com/openclaw/openclaw/issues/82020 if it still reproduces.

## Reproduction Assessment

No, not as a high-confidence live current-main repro. The source path is reproducible enough to see that arbitrary custom `openai-completions` provider ids do not obviously inherit Xiaomi's `omitEmptyArrayItems` patch, but the exact Token Plan request was not replayed with credentials.

## Solution Assessment

Unclear. The already-merged Xiaomi fixes are the right direction, but the best remaining fix is likely endpoint-class-based schema compatibility for custom Xiaomi Token Plan routes rather than a global rollback or new user-facing config switch.

## Repair Work Prompt

Fix the remaining custom Xiaomi Token Plan schema compatibility gap from https://github.com/openclaw/openclaw/issues/81923. Current main has Xiaomi-owned `omitEmptyArrayItems` handling from https://github.com/openclaw/openclaw/pull/82575, but arbitrary custom providers using `api: "openai-completions"` and a `*.xiaomimimo.com` base URL do not obviously inherit that compat. Add a narrow regression that builds an OpenAI-completions tool payload for a custom provider id pointed at `https://token-plan-cn.xiaomimimo.com/v1` with an array property missing `items`, and ensure the final payload omits only synthetic empty `items: {}` while preserving typed/boolean/null/literal items. Prefer extending endpoint-class/model-compat detection or provider compat hooks; do not globally remove array-items normalization, do not change stock OpenAI behavior, do not add a user-facing config option, and do not change SecretRef/auth precedence. If the auth portion still needs work, leave it to https://github.com/openclaw/openclaw/issues/82020 unless the new regression directly exposes it. Add a changelog entry only if the final patch is user-facing.
