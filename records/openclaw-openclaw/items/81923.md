## Summary

Keep open: current main has guardrails that avoid forcing native OpenAI strict tools onto custom OpenAI-compatible providers and still accepts literal configured provider keys, but this recent report now has two affected users and current source still injects permissive array `items` schemas before provider submission. The exact current-main Xiaomi/MiMo outbound payload and live provider rejection are still missing, so this is not safe to close or auto-repair without maintainer direction.

## What This Changes

The issue reports a 2026.5.12 custom-provider regression where Xiaomi/MiMo OpenAI-compatible requests fail with `400 Param Incorrect` and fallback auth then surfaces missing-key errors.

## Best Possible Solution

Preserve the current non-native strict-tool guard and credential-safety contract, then add a targeted provider/tool-schema compatibility fix or documented workaround once maintainers confirm the exact rejected Xiaomi/MiMo payload.

## Reproduction Assessment

No, we do not have a high-confidence current-main reproduction of the full failure. Source inspection confirms the relevant `items: {}` injection path and auth/fallback paths, and user comments provide release logs, but the exact current-main provider payload and live rejection are still missing.
