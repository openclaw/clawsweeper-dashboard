## Summary

This should stay open. Current main still globally adds permissive `items: {}` to array tool schemas before provider submission, the auth path still intentionally rejects unresolved broad env-marker-style keys, and the report is recent enough that maintainer triage should decide the compatibility boundary rather than cleanup closing it.

## What This Changes

The issue reports that upgrading to 2026.5.12 broke a custom OpenAI-compatible provider via array `items` schema injection, then surfaced an Ollama fallback auth error.

## Best Possible Solution

Keep the SecretRef/auth hardening, add or document a targeted per-model/provider schema-compat path for backends that reject injected `items`, and improve fallback diagnostics so the primary schema failure remains visible when a fallback also fails.

## Reproduction Assessment

No. The custom provider endpoint, redacted config, and rejected tool payload are missing; source inspection does confirm the current-main schema injection and auth fallback paths named in the report.

## Solution Assessment

Unclear. A global rollback is not the narrowest maintainable fix because it would risk the OpenAI validation bug fixed by https://github.com/openclaw/openclaw/pull/81217; a targeted compat/documentation path plus clearer diagnostics is safer.
