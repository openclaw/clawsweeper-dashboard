## Summary

This should stay open: current main still contains the implicated schema-normalization and auth-resolution paths, the report is recent, and the exact custom provider/config needed to verify the regression has not been supplied.

## What This Changes

The issue reports that OpenClaw 2026.5.12 breaks a custom `openai-completions` provider through array `items` tool-schema injection, fallback Ollama auth failure, and misleading cascade diagnostics.

## Best Possible Solution

Keep the issue open for maintainer triage, preserve the SecretRef auth contract, add or document a targeted schema-compat path for providers that reject injected `items`, and improve fallback diagnostics so the primary schema failure remains visible.

## Reproduction Assessment

No high-confidence live reproduction exists because the custom provider endpoint and redacted config/tool payload were not supplied. Source inspection does confirm the current schema normalization and auth fallback paths that could produce the reported cascade.

## Solution Assessment

Unclear: a global rollback or opt-in switch would be too broad without provider evidence. The safer direction is targeted provider/model compatibility plus clearer diagnostics, reviewed by maintainers because auth and schema normalization are cross-provider contracts.
