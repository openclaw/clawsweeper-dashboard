## Summary

Keep open: current main has fixed the non-native `strict: true` part and still supports literal configured provider keys, but the `items: {}` array-schema normalization implicated by this report still runs globally and the fallback/auth symptom is not fully narrowed.

## What This Changes

The issue reports that v2026.5.12 broke custom OpenAI-compatible provider upgrades through tool schema normalization, fallback auth, and misleading missing-key errors.

## Best Possible Solution

Have maintainers choose the provider-compatibility policy, then add targeted regression coverage for Xiaomi/custom OpenAI-compatible schema payloads while preserving the current SecretRef credential contract.

## Reproduction Assessment

No, we do not have a high-confidence live current-main reproduction. Source inspection proves current main still adds `items: {}` to array schemas missing `items`, and comments provide v2026.5.12 failures, but the exact current-main payload and credentials are missing.

## Solution Assessment

Unclear. A global rollback or new opt-out knob is not proven to be the best fix; the safer path is targeted provider compatibility using the existing compat/schema boundary plus focused regression coverage.
