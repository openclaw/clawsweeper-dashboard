## Summary

This should stay open: the report is recent, plausible, and current main still contains both implicated behavior paths, while the exact custom-provider failure needs maintainer triage and a redacted repro before changing schema normalization or auth policy.

## What This Changes

The issue reports an upgrade regression where a custom OpenAI-compatible provider rejects injected array `items` schemas and fallback auth then reports a missing API key.

## Best Possible Solution

Keep the SecretRef credential boundary, add or document a targeted schema-compat path for providers that reject injected `items`, and preserve the primary schema failure in fallback diagnostics.

## Reproduction Assessment

No high-confidence live reproduction is available because the custom provider endpoint and redacted config were not provided. Source inspection does confirm that current main still injects `items: {}` before provider submission and can report missing API key for unresolved marker-style custom provider auth.

## Solution Assessment

Unclear: a global rollback or opt-in switch is probably too broad, while the maintainable path is targeted provider/model compatibility plus clearer failover diagnostics. Maintainers need a redacted repro before choosing the exact contract change.
