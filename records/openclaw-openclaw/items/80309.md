## Summary

Keep open. Live Actions data confirms repeated scheduled `startup_failure` runs with zero jobs, and current `main` still has a likely caller-job permission mismatch for the reusable live/E2E workflow.

## What This Changes

Investigate daily scheduled live/E2E workflow startup failures that occur before any GitHub Actions jobs are created.

## Best Possible Solution

Add `actions: read` under `jobs.live_and_openwebui_checks.permissions`, keep the reusable workflow least-privilege, run workflow sanity checks, and prove the caller creates jobs before checking secrets or runner limits.

## Reproduction Assessment

Yes. Live Actions API data shows repeated scheduled `startup_failure` runs with zero jobs, and current-main source still has the caller job permission ceiling that can prevent the reusable workflow from receiving `actions: read`.

## Solution Assessment

Yes, the narrow first fix is to grant `actions: read` at the caller job permissions block, not only at workflow scope; secrets and runner settings should be escalated only if a corrected caller run still fails.
