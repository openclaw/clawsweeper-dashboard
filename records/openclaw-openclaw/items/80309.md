## Summary

Keep open. The scheduled workflow is still failing before job creation on live GitHub Actions, and current main still has a likely permission-ceiling mismatch between the caller job and the reusable workflow that needs a real rerun proof after correction.

## What This Changes

The issue asks maintainers to investigate repeated scheduled live/E2E workflow startup failures with zero jobs created.

## Best Possible Solution

Add `actions: read` under `jobs.live_and_openwebui_checks.permissions`, keep the reusable workflow least-privilege, and close only after a scheduled or manual caller run creates jobs.

## Reproduction Assessment

Yes. Live Actions API data shows repeated scheduled `startup_failure` runs with zero jobs, including 2026-05-11, and source inspection shows the caller job still cannot grant the `actions: read` permission requested by the reusable workflow.

## Solution Assessment

Yes for the first code-level fix. The narrow maintainable path is to pass `actions: read` from the caller job and rerun the caller workflow before escalating to secrets, runner limits, or repository settings.
