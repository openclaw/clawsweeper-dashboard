## Summary

Keep open. The report matches live Actions API evidence: the scheduled workflow is ending in `startup_failure` with zero jobs, and current `main` still has a likely permission-ceiling mismatch between the caller job and the reusable workflow. A related open PR already targets the same workflow, but it only adds the permission at workflow scope and has review notes saying the fix needs to move to the caller job permissions.

## What This Changes

The issue asks maintainers to investigate daily scheduled live/E2E workflow startup failures and verify repository secrets, runner/concurrency limits, or workflow syntax/configuration.

## Best Possible Solution

Add `actions: read` under `jobs.live_and_openwebui_checks.permissions`, keep the reusable workflow least-privilege, run workflow sanity checks, and confirm a scheduled or manual caller run creates jobs before escalating to secrets or runner settings.

## Reproduction Assessment

Yes. Live Actions API data shows repeated scheduled `startup_failure` runs with zero jobs, and source inspection on current `main` shows the caller job still cannot grant the `actions: read` permission requested by the reusable workflow.

## Solution Assessment

Unclear for the admin-only hypotheses, but source evidence points to a narrower maintainable fix first: pass `actions: read` from the caller job and rerun the caller workflow. Secrets or runner/concurrency settings should be checked only if that corrected workflow still fails.
