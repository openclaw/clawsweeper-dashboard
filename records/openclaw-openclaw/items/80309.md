## Summary

Keep open. Current main still has the caller-job permission ceiling that likely causes the reusable workflow startup failure, and the latest scheduled Actions runs still fail before creating jobs; the related open PR is incomplete rather than a reason to close.

## What This Changes

The issue reports that the scheduled live/E2E workflow fails with `startup_failure` before creating any jobs and asks maintainers to check secrets, runner limits, or workflow syntax.

## Best Possible Solution

Add `actions: read` under `jobs.live_and_openwebui_checks.permissions`, keep the reusable workflow least-privilege, run workflow sanity checks, and prove that a caller run creates jobs before escalating to secrets or runner settings.

## Reproduction Assessment

Yes. Live Actions API data shows repeated scheduled `startup_failure` runs with zero jobs through May 12, 2026, and current source still lacks the caller-job `actions: read` permission needed by the called workflow.

## Solution Assessment

Yes for the first code fix. The narrow maintainable path is to pass `actions: read` from the caller job and only investigate secrets, runner limits, or repository settings if a corrected caller run still fails.
