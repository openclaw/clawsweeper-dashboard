## Summary

Keep open. The scheduled workflow is still failing before job creation, current main still omits `actions: read` from the reusable-workflow caller job, and the related open PR only changes workflow-level permissions rather than the job-level permission boundary.

## What This Changes

The issue asks maintainers to investigate recurring scheduled live/E2E workflow startup failures before any GitHub Actions jobs are created.

## Best Possible Solution

Add `actions: read` under `jobs.live_and_openwebui_checks.permissions`, keep the reusable workflow least-privilege, then verify that a scheduled or caller workflow run creates jobs before closing this as fixed.

## Reproduction Assessment

Yes. Live Actions API data shows repeated scheduled `startup_failure` runs with zero jobs, including a 2026-05-14 run on `main`, and current source still has the caller permission gap.

## Solution Assessment

Yes for the narrow source-first direction: pass `actions: read` from the caller job and rerun the caller workflow. The existing related PR is not the best fix as reviewed because changing only workflow-level permissions does not cover the job-level permission block.
