## Summary

Keep this PR open for maintainer review: the patch is a narrow, correct CLI validation fix, but it intentionally changes invalid `status` and `health --timeout` inputs from successful default-timeout execution to a validation error.

## What This Changes

The PR changes the route-first `status` and `health` parsers to defer present-but-invalid `--timeout` values to Commander validation and adds route-args regression coverage.

## Best Possible Solution

Land the narrow parser fix and regression test if maintainers accept that invalid timeout values are outside the CLI contract and should fail consistently across the fast route and Commander path.

## Reproduction Assessment

Yes from source inspection, although I did not run commands in this read-only review. Current main treats invalid present `--timeout` values as absent on the fast path, while the full Commander path rejects the same inputs.

## Solution Assessment

Yes. The PR uses the existing sibling raw-token strict parser pattern and lets Commander own the shared error path, avoiding a broader helper contract change that could affect other callers.
