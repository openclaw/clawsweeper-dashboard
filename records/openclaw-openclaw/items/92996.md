## Summary

Keep open for maintainer review: source inspection confirms the bug and the PR's narrow fix/test look correct, but changing invalid `--timeout` values from exit-0 defaulting to exit-1 validation is a compatibility-sensitive CLI behavior change.

## What This Changes

The PR changes the route-first `status` and `health` CLI parsers so present-but-invalid `--timeout` values defer to Commander validation, with regression coverage for invalid, valid, and absent timeout cases.

## Best Possible Solution

Land the narrow parser fix and regression test if maintainers accept that invalid timeout values are outside the CLI contract and should fail consistently across the fast route and Commander path.

## Reproduction Assessment

Yes from source inspection: current main treats invalid present `--timeout` values as absent on the fast path, while the full Commander path rejects the same values. I did not run commands because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The PR uses the existing sibling raw-token strict-parser pattern and lets Commander own the shared error path, avoiding a broader helper contract change that could affect other callers.
