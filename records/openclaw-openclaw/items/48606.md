## Summary

Keep open: current main still has the macOS Edge no-app-path fallback behavior, and this PR is a plausible narrow fix, but it is not merge-ready without regression tests, real macOS behavior proof, rebase/conflict resolution, and fresh validation.

## What This Changes

The PR changes the browser plugin's macOS default Chromium detection to try known bundle-ID executable paths, including system and user Applications paths, when osascript/defaults resolution cannot produce an executable.

## Best Possible Solution

Land a rebased browser-plugin fix that keeps dynamic macOS app resolution first, falls back only for recognized Chromium bundle IDs with existing system or user Applications executables, and adds focused Edge-present and Edge-missing regression tests plus redacted macOS proof.

## Reproduction Assessment

Yes, by source inspection: current main reads com.microsoft.edgemac, returns null when osascript/defaults cannot produce an app path/name, then checks Chrome before Edge in the generic macOS fallback. The existing no-app-path Edge test also expects Chrome.

## Solution Assessment

Yes in direction, but not merge-ready. A known bundle-ID path fallback is the narrow maintainable fix for this failure mode, but this branch needs regression coverage, contributor real behavior proof, rebase, and fresh validation.
