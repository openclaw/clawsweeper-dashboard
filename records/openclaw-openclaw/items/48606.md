## Summary

Keep open: current main still drops the macOS default-browser path when `osascript`/`defaults` resolution fails, so Chrome can still win over a configured Edge default. The PR is a plausible narrow fix, but it still needs regression coverage for the new fallback path and external after-fix behavior proof before merge.

## What This Changes

The PR adds a macOS known bundle-ID-to-executable fallback in browser executable detection, checking `/Applications` and `~/Applications` after dynamic app resolution fails.

## Best Possible Solution

Land a rebased browser-plugin fix that keeps dynamic macOS app resolution first, falls back only for recognized Chromium bundle IDs with existing system or user Applications executables, and adds focused Edge-present and Edge-missing regression tests; handle any changelog entry during maintainer landing.

## Reproduction Assessment

Yes, by source inspection: mock LaunchServices as `com.microsoft.edgemac`, make `osascript`/`defaults` fail, and make both Edge and Chrome paths exist; current main returns `null` from default detection and then selects Chrome from the generic fallback. I did not run the test because this was a read-only review.

## Solution Assessment

Yes in direction, but not merge-ready. A known bundle-ID path fallback is the narrow maintainable fix, but this PR still needs regression coverage, contributor real behavior proof, and fresh validation.
