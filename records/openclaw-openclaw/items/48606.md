## Summary

Keep this PR open: current main still falls through to Chrome when the Edge LaunchServices ID is recognized but `osascript`/`defaults` resolution fails, so the known-path fallback remains relevant. It is not merge-ready because contributor real behavior proof is missing, the branch is conflicting/stale, and maintainer review asked for focused regression tests.

## What This Changes

The PR adds a macOS bundle-id-to-executable fallback map so recognized Chromium defaults can use `/Applications` or `~/Applications` paths when `osascript`/`defaults` path resolution fails.

## Best Possible Solution

Land a small browser-plugin fix that preserves dynamic macOS resolution, adds known-path fallback for the Edge/Chromium LaunchServices mismatch, and covers both Edge-selected and missing-known-path fallback cases.

## Reproduction Assessment

Yes. Source inspection plus the existing mocked test show a current-main path where `com.microsoft.edgemac` with empty `osascript` output falls through to Chrome before Edge; I did not run a real macOS reproduction in this read-only review.

## Solution Assessment

Yes, directionally. Trying dynamic macOS resolution first and then known Edge/Chromium paths when that fails is a narrow maintainable fix, but it still needs regression coverage and real macOS proof before merge.
