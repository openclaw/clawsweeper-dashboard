## Summary

Keep open: current main still falls back to Chrome when Edge's LaunchServices ID is recognized but app-path lookup fails, and this PR is a plausible narrow fix. It is not merge-ready because it lacks fallback-branch regression tests, real macOS behavior proof, and fresh non-conflicting validation.

## What This Changes

The branch adds a macOS known bundle-ID-to-executable fallback in the browser plugin for `/Applications` and `~/Applications` when `osascript/defaults` resolution fails.

## Best Possible Solution

Land a rebased browser plugin fix that preserves dynamic resolution first, falls back to known system/user app paths for recognized Chromium bundle IDs, and adds Edge-present and Edge-missing regression tests plus redacted macOS status proof.

## Reproduction Assessment

## Solution Assessment

Yes in direction, but not merge-ready. A known bundle-ID path fallback is the narrow maintainable fix for this failure mode, but the branch needs tests, proof, rebase/conflict resolution, and fresh validation.
