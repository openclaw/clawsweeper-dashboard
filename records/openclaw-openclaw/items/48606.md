## Summary

Keep open: current main still has the no-app-path Edge-to-Chrome fallback and this PR remains useful, but it is not merge-ready because it lacks regression tests, lacks real macOS proof, and the branch is currently conflicting.

## What This Changes

The PR updates the browser plugin's macOS default Chromium detection to fall back from recognized bundle IDs to known system or user Applications executable paths when osascript/defaults cannot resolve the app.

## Best Possible Solution

Land a rebased browser plugin fix that preserves dynamic app resolution first, falls back to known existing system/user Applications executables for recognized Chromium bundle IDs, and includes focused regression tests plus redacted macOS status proof.

## Reproduction Assessment

Yes, by source inspection: current main reads `com.microsoft.edgemac`, returns `null` when osascript/defaults resolution fails, then checks Chrome before Edge in the generic macOS fallback. The existing no-app-path test also encodes Chrome as the current result.

## Solution Assessment

Yes in direction, but not merge-ready. A known bundle-ID path fallback is a narrow maintainable fix for this failure mode, but this branch still needs regression coverage, contributor real behavior proof, and a clean rebase.
