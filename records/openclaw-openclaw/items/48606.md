## What This Changes

The PR adds a macOS known bundle-ID-to-executable fallback in the browser plugin when LaunchServices reports a Chromium default that osascript/defaults cannot resolve.

## Best Possible Solution

Land a rebased browser plugin fix that keeps dynamic app resolution first, falls back to known existing system/user Applications executables for recognized Chromium bundle IDs, and includes focused regression tests plus redacted macOS status proof.

## Reproduction Assessment

Yes, by source inspection: current main reads `com.microsoft.edgemac`, returns null when osascript/defaults resolution fails, then checks Chrome before Edge in the generic macOS fallback. The existing no-app-path test encodes that current-main path.

## Solution Assessment

Yes in direction, but not merge-ready. A known bundle-ID path fallback is a narrow maintainable fix for this failure mode, but this branch still needs regression coverage, contributor real behavior proof, and a clean rebase.
