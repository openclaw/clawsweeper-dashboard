## Summary

Keep open: the macOS Edge fallback bug is still present on current main, and this PR is a plausible fix path but is not merge-ready because it lacks regression coverage and required real macOS behavior proof.

## What This Changes

The PR changes the browser plugin’s macOS default Chromium detection to fall back from recognized bundle IDs to known system and user Applications executable paths when dynamic app resolution fails.

## Best Possible Solution

Land a browser plugin fix that preserves dynamic app resolution first, then falls back to known existing system/user Applications executables for recognized Chromium bundle IDs, with focused regression tests and redacted macOS `openclaw browser status --json` proof.

## Reproduction Assessment

Yes, by source inspection: current main reads `com.microsoft.edgemac`, returns `null` when osascript produces no app path, then checks Chrome before Edge in the generic macOS fallback. The existing no-app-path test also encodes Chrome as the current result.

## Solution Assessment

Yes in direction, but not merge-ready. A known bundle-ID path fallback is a narrow maintainable fix for this failure mode, but this branch needs regression coverage, contributor real behavior proof, and fresh validation.
