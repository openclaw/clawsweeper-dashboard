## Summary

Keep open: current main still has the macOS Edge no-app-path fallback bug, and this PR is a plausible narrow fix, but it is not merge-ready because it lacks regression tests, real macOS behavior proof, and a clean merge/validation state.

## What This Changes

The PR changes the browser plugin’s macOS default Chromium detection to fall back from recognized LaunchServices bundle IDs to known system/user Applications executable paths when dynamic app resolution cannot produce an executable.

## Best Possible Solution

Land a rebased browser plugin fix that preserves dynamic app resolution first, then falls back to existing known system/user Applications executables for recognized Chromium bundle IDs, with Edge-present and Edge-missing regression tests plus redacted macOS `openclaw browser status --json` proof.

## Reproduction Assessment

Yes, by source inspection: current main reads `com.microsoft.edgemac`, returns `null` when `osascript` cannot produce an app path, and then checks Chrome before Edge in the generic macOS fallback. The existing no-app-path test also encodes Chrome as the current result.

## Solution Assessment

Yes in direction, but not merge-ready. A known bundle-ID path fallback is the narrow maintainable fix for this failure mode, but this branch needs regression coverage, contributor real behavior proof, rebase, and fresh validation.
