## Summary

Keep open: the latest PR still leaves gateway stop matching one-way for raw/canonical session aliases, and its supplied proof does not cover the full live Telegram/gateway stop path required for this behavior.

## What This Changes

The PR updates gateway `chat.send` stop handling to consider canonical session aliases and routes targeted Telegram `/stop@bot` updates through the isolated-ingress control lane, with focused regression tests.

## Best Possible Solution

Make gateway stop matching symmetric across raw and canonical session keys, keep the Telegram lane fallback, and land only after focused regression coverage plus redacted live Telegram/gateway stop proof.

## Reproduction Assessment

Yes, at source level: current main stores active `chat.send` abort controllers under the raw session key and session abort matching is exact, while Telegram isolated spool lane selection can run without bot metadata. I did not run a live provider or Telegram turn in this read-only review.

## Solution Assessment

No, not yet. The Telegram fallback is plausible, but the gateway alias fix remains one-directional and the proof does not yet demonstrate the full after-fix runtime stop path.
