## Summary

Keep open: this PR is a plausible bug fix, but it still has a blocking gateway session-key alias gap and the external real-behavior proof is only test output.

## What This Changes

The PR adds a shared active-run abort helper for stop commands, broadens gateway stop handling for raw/canonical session keys, and adds focused abort regression tests.

## Best Possible Solution

Make gateway stop matching symmetric or register abort controllers canonically, keep the active-run abort helper, add regression coverage for both alias directions, and attach redacted live stop proof.

## Reproduction Assessment

Yes, at source level: current main can register active gateway abort controllers under a raw session key while later stop handling may look up only the canonical key, and the related Telegram report gives a concrete active-turn `/stop` flow. I did not run a live provider turn in this read-only review.

## Solution Assessment

No. The auto-reply active-run helper is a sound part of the fix, but the gateway alias handling should be symmetric or canonicalized, and the PR still needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle reverse session-key aliases when stopping runs:** `src/gateway/server-methods/chat.ts:2058`
  - body: `chat.send` registers active abort controllers with the raw requested session key, but this alias list only adds the canonical key when the stop request used a raw alias.
