## Summary

Keep open: this PR still has useful gateway and Telegram stop-command work beyond the merged auto-reply fix, but the latest head still misses the reverse raw/canonical gateway stop path and does not yet provide sufficient after-fix live active-stop proof.

## What This Changes

The PR routes stop commands through a shared active-run abort helper, broadens gateway stop matching across raw/canonical session keys, routes Telegram `/stop@bot` to the control lane without cached bot metadata, and adds focused regression tests.

## Best Possible Solution

Rebase on current main, keep the merged auto-reply helper, make gateway stop matching symmetric across raw and canonical keys, keep the Telegram control-lane fix, and land with focused tests plus redacted live active-stop proof.

## Reproduction Assessment

Yes, at source level: current main registers active gateway abort controllers under the raw requested key while canonicalizing session storage, and the PR head only aliases raw-to-canonical stop requests. A focused gateway regression can reproduce the remaining canonical-to-raw miss without needing a full provider run.

## Solution Assessment

No. The shared auto-reply helper and Telegram control-lane direction are reasonable, but the gateway alias fix is still one-directional and the PR needs stronger after-fix real behavior proof before merge.
