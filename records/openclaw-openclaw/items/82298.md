## Summary

Keep open: the Telegram lane change looks narrowly useful and the duplicate auto-reply work was removed after https://github.com/openclaw/openclaw/pull/82342 merged, but the gateway stop fix still handles raw/canonical aliases in only one direction and can miss a raw-key active run when the later stop request is canonical. The supplied terminal proof is helpful for the resolver path, but it still does not show an active gateway or Telegram turn being stopped after the patch.

## What This Changes

The PR changes Telegram isolated-ingress lane selection for targeted `/stop@bot`, gateway `chat.send` session-scoped stop matching across raw/canonical keys, and focused tests for those paths.

## Best Possible Solution

Use one canonical identity for gateway abort registration/lookup, or carry a complete raw/canonical alias set in both directions while preserving requester authorization, then land the Telegram control-lane fallback with redacted live stop proof.

## Reproduction Assessment

Yes, at source level: current main registers `chat.send` abort controllers under the raw requested key and session-scoped abort matching is exact, while Telegram isolated ingress chooses the spool lane before full stop handling. I did not run a live provider turn in this read-only review.

## Solution Assessment

No. The Telegram fallback is a reasonable narrow fix, but the gateway alias handling should be symmetric or canonical at registration time; the current PR still leaves one alias direction uncovered.
