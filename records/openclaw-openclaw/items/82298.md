## Summary

Keep open: this is a focused bug-fix PR for the active-turn stop regression, but the draft patch still misses a raw/canonical gateway alias edge case and only provides test-based proof.

## What This Changes

The PR refactors stop handling to abort active embedded runs by session key and updates gateway `chat.send` stop handling plus regression tests for raw/canonical session-key aborts.

## Best Possible Solution

Finish the PR by matching raw and canonical session keys symmetrically without weakening requester authorization, then add focused regression coverage and redacted real stop proof.

## Reproduction Assessment

Yes, at source level: current main registers active chat abort controllers under the raw request key and the existing stop helper matches exact session keys. I did not execute a live Telegram or WebChat run in this read-only review.

## Solution Assessment

No. The auto-reply direction is sound, but the gateway fix should cover raw-to-canonical and canonical-to-raw stop requests and the external PR still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle reverse session-key aliases when stopping runs:** `src/gateway/server-methods/chat.ts:2058`
  - body: `chat.send` registers active abort controllers with the raw requested session key, but this alias list only adds the canonical key when the stop request itself used a raw alias. If a run starts under an alias such as `alias-main` and a later stop is sent with canonical `main`, the active controller is still registered under the alias and will not be matched. Include both raw and canonical keys for the active registration path, or register controllers canonically so stop works in both directions.
  - confidence: 0.87
