## Summary

Keep open: this is a valid direction for the stop-command regression, but the current patch still misses the reverse raw/canonical gateway session-key case and lacks required real behavior proof from an active streaming turn.

## What This Changes

The PR adds a shared active-run abort helper, updates gateway stop handling for raw/canonical session-key aborts, and adds focused abort regression tests.

## Best Possible Solution

Make gateway stop matching cover raw and canonical session keys symmetrically without weakening requester authorization, keep the active-run abort helper, and add focused tests plus redacted live stop proof.

## Reproduction Assessment

Yes, at source level: current main registers active chat abort controllers under the raw request key and the session abort helper matches exact keys. The related open report gives a concrete Telegram `/stop` reproduction, but I did not run a live provider turn in this read-only review.

## Solution Assessment

No. The auto-reply helper is a sound part of the fix, but the gateway alias handling should be symmetric and the external PR still needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle reverse session-key aliases when stopping runs:** `src/gateway/server-methods/chat.ts:2058`
