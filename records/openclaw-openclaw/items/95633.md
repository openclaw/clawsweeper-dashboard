## Summary

Keep open for maintainer direction. Current main intentionally exports placeholder OTLP log bodies unless broad telemetry content capture is enabled; the requested always-use-the-message behavior remains missing and is privacy/security-sensitive rather than a safe automatic bug fix.

## What This Changes

The issue asks for diagnostics-otel OTLP log records to put the rendered log message in `LogRecord.body` instead of placeholder bodies like `"log"` or `"openclaw.security.event"`.

## Best Possible Solution

Keep the safe default until maintainers either reaffirm broad `captureContent: true` as the only log-body path or approve a narrow documented log-body opt-in with redaction and tests.

## Reproduction Assessment

Yes for source-level reproduction: current main explicitly emits `"log"` for normal OTLP log bodies unless broad `diagnostics.otel.captureContent: true` is configured. I did not run the live Loki collector path in this read-only review.

## Solution Assessment

No as a default behavior change. The safer path is a maintainer-approved privacy decision, likely either retaining the broad content-capture policy or adding a narrow opt-in for OTLP log bodies.
