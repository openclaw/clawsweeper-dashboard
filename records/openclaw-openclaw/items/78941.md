## Summary

Keep open: this PR is not merge-ready because it changes only the WebSocket payload builder, while the linked LiteLLM/custom-baseUrl route normally stays on HTTP/SSE and still emits `store:false`. The related discussion usefully points toward an explicit `compat.responsesStore`-style opt-in, which is safer than letting every proxy-like endpoint inherit provider persistence defaults.

## What This Changes

The PR changes `buildOpenAIWebSocketResponseCreatePayload` so proxy-like OpenAI Responses WebSocket payloads omit `store:false` while native endpoints keep it.

## Best Possible Solution

Implement a shared, explicit Responses persistence opt-in, likely a provider/model compat flag, across HTTP/SSE and any WebSocket path, with LiteLLM docs and proxy/native regression tests.

## Reproduction Assessment

No live high-confidence reproduction is present because the failing LiteLLM proxy is private. Source inspection does reproduce the OpenClaw-side behavior: proxy/custom routes stay on HTTP/SSE and the HTTP Responses path still applies `store:false`.

## Solution Assessment

No. This is not the best fix because it changes only the WebSocket builder and makes proxy persistence implicit; the safer solution is an explicit shared Responses store policy with docs and tests.
