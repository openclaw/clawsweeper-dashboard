## Summary

Keep open: current main still drops the agent-layer `replace` signal before Gateway chat merging, so the underlying fix is useful, but this branch is conflicting, clears buffered chat text on empty replacements, and lacks real behavior proof.

## What This Changes

The branch threads an assistant stream `replace` flag through Gateway chat delta merging, bypasses throttling for replacements, adds a Gateway unit test, and adds a changelog entry.

## Best Possible Solution

Rebase onto current main, implement replace support in the live-chat projector/Gateway path, preserve existing buffers for empty replacements unless a distinct clear semantic is added, and add redacted live proof for the vLLM stream path.

## Reproduction Assessment

Yes. Source inspection shows a clear failing path: after prior text, a media-visible assistant update can emit `replace: true` with `text: ""`, and the PR branch turns that into an empty stored buffer.

## Solution Assessment

No. A deterministic replace signal is the right direction, but the patch should be rebased onto the current live-chat merge helper and should ignore or preserve empty textual replacements instead of clearing chat text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve buffered text for empty replacements:** `src/gateway/server-chat.ts:116-117`
  - body: When `replace` is true, this returns an empty string for empty incoming text, and the later empty guard is bypassed for replacements. Media-visible assistant updates can carry no text after previous streamed text, so this clears the stored chat buffer and makes the assistant response disappear; keep the prior buffer or skip the delta unless the replacement has non-empty text.
