## Summary

Keep open. The underlying streaming bug is still plausible because current main can emit assistant `replace` metadata that the Gateway does not consume, but this PR should not merge as-is because empty replacement payloads can clear existing chat text and the contributor has not provided inspectable real behavior proof.

## What This Changes

The PR adds Gateway handling for assistant stream `replace` metadata, bypasses the chat delta throttle for replacements, adds a Gateway regression test, and records a changelog entry.

## Best Possible Solution

Implement replace support in the current live-chat-projector/Gateway path so non-empty replacements update immediately, empty replacement payloads preserve prior text unless an explicit clear semantic exists, and the fix has focused tests plus redacted live vLLM proof.

## Reproduction Assessment

Yes, from source inspection. Current main can emit assistant events with `replace: true` from the agent layer while the Gateway ignores that field, and the PR branch has a deterministic empty-replacement path that clears the buffer.

## Solution Assessment

No. Passing deterministic replacement metadata through the Gateway is the right direction, but this patch should preserve prior text for empty replacement payloads and be rebased onto the current shared live chat projector.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve buffered text for empty replacements:** `src/gateway/server-chat.ts:116-117`
  - body: When `replace: true` arrives with empty text, this branch returns an empty string and the later empty guard is bypassed for replacements. Media or audio-visible assistant updates can carry replacement metadata without textual content, so this can clear the stored chat buffer and broadcast blank text after prior assistant output; preserve the previous buffer or ignore empty replacements unless a separate explicit clear semantic is introduced.
