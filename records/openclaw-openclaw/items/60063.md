## Summary

Keep open: the PR targets a real chat streaming gap, but the branch is stale against current main and still allows empty replacement events to clear buffered assistant text before it is safe to merge.

## What This Changes

The PR threads an assistant `replace` flag through Gateway chat delta handling, bypasses throttling for replacements, adds a regression test, and updates the changelog.

## Best Possible Solution

Implement replace support in the current live-chat projector/server-chat path so non-empty replacements update immediately while empty textual replacements preserve the prior buffer unless an explicit clear semantic is added.

## Reproduction Assessment

Yes. A source-level reproduction is clear: start with a non-empty chat buffer, then process an assistant event with `replace: true` and `text: ""`; the PR returns an empty merge and bypasses the empty guard, clearing the stored assistant text.

## Solution Assessment

No. A deterministic replace signal is the right direction, but the branch should be rebased onto the current `live-chat-projector.ts` shape and should guard empty replacements before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve buffered text for empty replacements:** `src/gateway/server-chat.ts:114-115`
  - body: `replace: true` returns an empty string when the incoming text is empty, and the later empty-merge guard is disabled for replacements. Media/audio-visible assistant updates can carry empty cleaned text after prior text, so this can clear the stored chat buffer and make streamed assistant text disappear. Preserve the previous buffer or skip the delta unless the replacement carries non-empty text.
