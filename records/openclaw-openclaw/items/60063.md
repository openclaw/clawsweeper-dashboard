## Summary

Keep open. Current main has some adjacent streaming protections, but it still does not consume the assistant `replace` signal in the Gateway chat path, so the PR is not obsolete. The branch is directionally useful, but it should not merge as-is because its empty replacement path can clear already-buffered assistant text.

## What This Changes

The PR threads an assistant `replace` flag through Gateway chat delta handling, bypasses the 150ms throttle for replacements, adds a server-chat regression test, and adds a changelog entry.

## Best Possible Solution

Keep the formal `replace` protocol support, but implement it in the current shared live-chat merge path so non-empty replacements update immediately while empty textual replacements preserve the existing buffer unless there is an explicit, tested reason to clear it.

## Reproduction Assessment

Yes. A source-level reproduction is clear: start with a non-empty chat buffer, then process an assistant event with `replace: true`, `text: ""`, and media/audio visibility; the proposed merge returns an empty buffer and bypasses the empty guard.

## Solution Assessment

No. A deterministic replace signal is the right direction, but the PR needs an empty-replacement guard and should be applied to the current `live-chat-projector.ts`/`server-chat.ts` shape instead of clearing buffered text on no-text updates.

- **[P2] Preserve buffered text for empty replace events:** `src/gateway/server-chat.ts:114-115`

Preserve the previous buffer or skip the delta unless the replacement carries non-empty text.
