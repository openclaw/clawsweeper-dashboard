## Summary

Keep open: current main still lacks Markdown/SKILL.md scanning and the latest PR addresses the earlier regex and changelog blockers, but the diff still has a security-relevant file-budget regression that should be fixed before merge.

## What This Changes

The PR adds Markdown/.md handling to the skill scanner, Markdown-specific hidden Unicode/data URI/download/encoded-payload rules, focused tests, and a changelog entry.

## Best Possible Solution

Keep the Markdown scanner work in the existing security scanner, but collect or prioritize files so JS/TS code scanning cannot be starved by Markdown entries under the file limit, with regression coverage for both paths.

## Reproduction Assessment

Yes. Main is source-reproducible because .md is absent from the scanner extension set and tests expect Markdown to be non-scannable; the remaining PR blocker is also source-reproducible because .md joins the same maxFiles-limited collection path as JS/TS.

## Solution Assessment

No, not as-is. The existing scanner is the right boundary and the PR now covers the earlier bypass variants, but it should preserve code-file coverage by prioritizing code files or separating Markdown/code budgets before merge.
