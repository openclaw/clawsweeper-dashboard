## Label Justifications

- `P1`: A live fallback turn reportedly duplicated completed work and pushed unwanted commits onto an active pull-request branch.
- `impact:session-state`: The failure is a session-context error where prior completed-task state is reinterpreted as new executable work.

## Summary

The report is credible and still necessary: current main has prompt guidance for subagent completion reports but no shared enforced state preventing a fallback turn from treating those reports as new work. The linked draft pull request remains open and is the correct place to refine the fix, but maintainer review already established that its CLI-only prompt injection does not cover embedded execution or Codex-native spawning.

## What This Changes

Prevent fallback-model turns from treating completed background-task announcements as new work and re-executing or re-spawning those tasks.

## Best Possible Solution

Represent fallback handling of completion announcements as a closed shared run state, prevent spawn or execution dispatch for work already marked completed across every applicable harness, and add focused cross-harness tests plus one redacted live fallback transcript proving no duplicate task or branch mutation occurs.

## Reproduction Assessment

No, not yet at high confidence: the live session evidence is concrete and current source exposes the plausible prompt/context gap, but there is no controlled current-main fallback scenario that reliably reproduces the duplicate spawn or execution.

## Solution Assessment

No: the linked CLI identity-prompt approach is not the best complete solution because the invariant belongs at a shared fallback/run boundary and must account for embedded execution and Codex-native `spawn_agent`.
