## Label Justifications

- `P1`: The reported fallback behavior can replay completed operations and create unwanted external side effects such as duplicate commits and pushes.
- `impact:session-state`: The failure is semantic drift in how a fallback turn interprets prior task-completion state in the session transcript.

## Summary

Keep this issue open. The reported semantic replay remains plausible on current main, and the paired draft PR is still open while being rewritten after its CLI-only prompt approach was acknowledged as unreachable and incomplete across the embedded, CLI, and Codex-native spawn surfaces.

## What This Changes

Prevent fallback turns from treating completed background-task announcements as new directives and repeating already-finished work.

## Best Possible Solution

Represent completed background work as typed, consumed, non-directive state at the shared run boundary, then enforce or explicitly document that invariant for embedded OpenClaw, CLI/MCP, and Codex-native spawning with focused regression tests and a live fallback trace.

## Reproduction Assessment

No, not at high confidence yet. The issue contains concrete historical live evidence and current source exposes the implicated fallback and spawn surfaces, but there is no deterministic current-main reproduction or focused failing test.

## Solution Assessment

No, the submitted CLI-only fallback-identity prompt is not the best solution. The narrower maintainable direction is a shared typed completion-state invariant with explicit enforcement at every spawn-capable owner boundary.
