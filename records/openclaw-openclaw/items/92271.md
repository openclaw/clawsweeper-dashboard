## Summary

This issue is still necessary: current main does not contain the fallback-completion spawn guard, and the linked closing PR remains open with a Codex-native/runtime-boundary decision still unresolved.

## What This Changes

The issue reports that fallback-model turns can treat completed-work announcements as new instructions and spawn duplicate work against already-finished tasks.

## Best Possible Solution

Keep this issue open until the linked PR or a replacement lands a shared fallback-completion runtime mode with mechanical enforcement or proven non-exposure across embedded, ACP, CLI/MCP, and Codex-native spawn surfaces.

## Reproduction Assessment

No high-confidence standalone current-main live reproduction was run. Source inspection shows the implicated paths are still present on main: fallback attempts are not marked for downstream tools, and `sessions_spawn` has no fallback-completion rejection.

## Solution Assessment

No, current main is not the solution, and the linked PR is not complete until the Codex-native and real fallback transition paths are settled. The maintainable fix is a shared runtime state with mechanical enforcement or explicit non-exposure for every spawn-capable harness.
