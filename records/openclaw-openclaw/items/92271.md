## Summary

This issue is still necessary: current main does not contain the fallback-completion spawn guard, and the linked closing PR is open with an unresolved Codex-native/runtime-boundary question rather than merged implementation.

## What This Changes

The issue asks OpenClaw to prevent fallback turns triggered by completed-work announcements from re-spawning or re-executing work that was already reported as complete.

## Best Possible Solution

Land the linked PR or a replacement that represents fallback-completion handling as one per-turn runtime mode, mechanically blocks or removes every work-spawning surface for that mode, and proves embedded, ACP, CLI/MCP, and Codex-native behavior.

## Reproduction Assessment

No high-confidence standalone current-main reproduction was run. The issue includes detailed live-session evidence, and current source plus the linked PR review show the relevant fallback, completion-handoff, sessions_spawn, and Codex-native spawn paths remain plausible.

## Solution Assessment

No, current main is not the solution and the linked PR is not complete until the Codex-native and real fallback transition paths are settled. The maintainable fix is a shared runtime state with mechanical enforcement or proven non-exposure for every spawn-capable harness.
