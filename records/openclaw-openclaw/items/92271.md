## Label Justifications

- `P1`: The reported fallback behavior can repeat completed commands and push unwanted commits to active work branches in real sessions.
- `impact:session-state`: Fallback turns are reported to reinterpret prior completion results and drift the session's task-execution state.

## Summary

This issue remains necessary and is correctly paired with an open draft implementation PR; current main has structural spawn-depth controls and prompt-level evidence guidance, but no shared fallback-turn invariant that prevents completed work from being semantically re-executed across OpenClaw and Codex spawning surfaces.

## What This Changes

Prevent fallback model turns from treating completed background-task announcements as new directives and duplicating already-finished work.

## Best Possible Solution

Represent completion-announcement fallback turns as a runtime-owned closed state, carry that state through fallback attempts, and enforce or explicitly suppress spawning in embedded, CLI/MCP, and Codex-native harnesses with focused tests and same-provider live proof.

## Reproduction Assessment

No, not yet at high confidence. The issue provides a concrete live sequence and current source exposes the relevant completion and fallback paths, but the duplicate mutation has not been reproduced on current main under the same runtime and tool conditions.

## Solution Assessment

No, the current draft is not the best complete solution. Its CLI-specific identity prompt does not own embedded sessions_spawn or Codex-native spawn_agent, so the shared runtime state and cross-harness enforcement proposed in the discussion is the safer direction.
