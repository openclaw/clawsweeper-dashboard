## Summary

This issue remains necessary: current `main` still permits completion-announcement turns to continue work and exposes no fallback/provenance guard at the `sessions_spawn` boundary, while the linked closing PR remains open. That PR now supplies a concrete OpenClaw repair and runtime proof, but maintainers still need to decide whether native Codex `spawn_agent` enforcement is required before landing or should become an explicit upstream follow-up.

## What This Changes

Prevent fallback turns from treating reports of completed background work as instructions to repeat or redelegate that work.

## Best Possible Solution

Enforce the invariant for OpenClaw-owned spawn tools at the shared fallback/completion boundary, then either cover native Codex `spawn_agent` through the integration or link a concrete upstream Codex follow-up before treating the cross-harness problem as resolved.
