## Summary

Current main still has separate model-fallback, completion-announcement, and spawn-capability paths without a shared invariant preventing a fallback turn from treating a completed child result as fresh work. The linked draft PR remains the active implementation vehicle, but its author and a repository member agree the existing CLI-only prompt patch is unreachable and must be rewritten across embedded, CLI/MCP, and Codex-native spawning surfaces, so this issue remains necessary and should stay open.

## What This Changes

Prevent fallback turns from re-executing or re-spawning work that completion announcements already report as finished.

## Best Possible Solution

Represent completed child work as explicit shared run state, enforce idempotent handling at the canonical fallback/announcement boundary, and add a harness matrix proving embedded, CLI/MCP, and Codex-native spawn behavior or documented non-exposure.

## Reproduction Assessment

No, not yet at high confidence: the report provides a concrete live sequence and current source exposes the implicated fallback, announcement, and spawn paths, but no controlled current-main reproduction with the same provider, transcript, and tool surface is attached.

## Solution Assessment

No, the existing linked patch is not the best solution: its fallback prompt was unreachable and only covered CLI backends; the agreed rewrite toward a shared state invariant with cross-harness enforcement is the maintainable path.
