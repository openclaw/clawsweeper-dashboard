## Summary

Keep this open: the report describes a real post-release recurrence, and current main still has a source-visible path where an unstructured `thread not found` preflight-compaction result would bypass the structured stale/missing binding recovery and throw before the Telegram turn runs.

The issue asks OpenClaw to recover from stale or missing Codex threads during Telegram preflight compaction instead of surfacing a generic failed dispatch.

## Best Possible Solution

Normalize Codex missing-thread preflight failures at the real compaction producer or boundary into the existing stale/missing binding failure shape, with regression coverage that keeps non-binding compaction failures fail-closed.

No live deterministic reproduction was run in this read-only review. Source inspection shows a clear failing path when preflight compaction returns a raw `thread not found` reason without structured binding failure metadata, matching the sanitized logs.

## Solution Assessment

Yes, the best direction is a narrow bug fix in the compaction result classification path. The fix should preserve the existing fail-closed behavior for auth/profile and other non-binding compaction failures.
