## Summary

Keep open. The cross-provider accountId leak is still present on current main and this PR is a useful starting point, but it cannot merge as-is because it checks only root cross-context settings, would weaken the current exec-approval source contract if resolved literally, mixes unrelated skill cleanup, and lacks real behavior proof.

## What This Changes

The PR suppresses implicit defaultAccountId on cross-provider message sends, adds a regression test, broadens exec-approval allowlist source metadata, and edits two skill descriptions.

## Best Possible Solution

Land or replace with a focused current-main fix that uses the effective message policy, preserves explicit accountIds and same-provider defaults, omits only implicit source defaults for allowed cross-provider sends, and leaves approval metadata plus skill cleanup to their existing threads.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: call runMessageAction with defaultAccountId, a source currentChannelProvider, a different target channel, and effective cross-provider policy enabled, then assert the target plugin does not receive the implicit source account id.

## Solution Assessment

No. The shared runner fix is the right direction, but this branch must compute suppression from the same effective policy used by enforcement and drop the now-obsolete approval-schema plus unrelated skill-description hunks.
