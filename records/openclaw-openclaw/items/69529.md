## Summary

Keep open. The cross-provider account-id leak is still a valid bug, but this PR is not mergeable as-is because the outbound fix bypasses the effective per-agent message policy, the approval-schema hunk weakens the current literal-only contract, the branch bundles unrelated skill edits, and the external PR still lacks real behavior proof.

## What This Changes

The branch suppresses implicit `defaultAccountId` for cross-provider message sends, adds a regression test, accepts exec approval `source` metadata as a string, and edits two skill descriptions.

## Best Possible Solution

Land or replace this with a focused current-main fix that computes account-id suppression from the effective message policy, preserves explicit and same-provider account ids, keeps approval `source` literal-only, and leaves skill cleanup to its own thread.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction` with a source `defaultAccountId`, a different target provider, and effective cross-provider policy enabled, then assert the target plugin does not receive the implicit source account id.

## Solution Assessment

No. The shared outbound runner is the right place to fix the bug, but this branch must use the same effective policy as enforcement and drop the now-obsolete approval-schema and unrelated skill-description hunks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the effective cross-context policy:** `src/infra/outbound/message-action-runner.ts:723-730`
  - body: The suppression condition reads only `cfg.tools?.message?.crossContext?.allowAcrossProviders`, while enforcement uses `resolveEffectiveMessageToolsConfig` with the resolved agent id, including agent overrides and `allowCrossContextSend`. A send allowed by those effective settings would still pass the source `defaultAccountId` to the target plugin, so base suppression on the same effective config enforcement uses.
  - confidence: 0.9
- **[P2] Keep approval source validation literal-only:** `src/gateway/protocol/schema/exec-approvals.ts:13`
  - body: Changing the allowlist entry schema to `Type.String()` accepts arbitrary `source` metadata. Current main intentionally accepts only `"allow-always"` and rejects unknown source values, so carrying this hunk forward would weaken the gateway approval contract; keep the literal schema or drop this obsolete hunk.
  - confidence: 0.9
