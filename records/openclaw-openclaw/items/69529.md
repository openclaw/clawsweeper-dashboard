## Summary

Keep open. The central cross-provider account leak is still present on current main and the PR contains useful source work, but the branch is conflicting, lacks after-fix real behavior proof, and bundles obsolete or unrelated changes that should not merge as-is.

## What This Changes

The PR suppresses implicit defaultAccountId on cross-provider message sends, adds a regression test, broadens exec-approval allowlist source metadata, and edits two bundled skill descriptions.

## Best Possible Solution

Revise or replace this branch with a focused current-main fix that preserves explicit accountId values, honors effective root/agent cross-context policy, suppresses only the implicit source default for allowed cross-provider sends, and leaves approval schema plus skill cleanup to their canonical items.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register source and target channel plugins, call runMessageAction with defaultAccountId, a source currentChannelProvider, a different target channel, currentChannelId, and allowAcrossProviders true, then assert the target plugin does not receive the implicit source account id.

## Solution Assessment

No. The shared outbound-runner fix is the right direction, but this branch should use the same effective message policy as enforcement and drop or split the now-obsolete approval-schema and skill-description hunks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Constrain approval source metadata:** `src/gateway/protocol/schema/exec-approvals.ts:13`
  - body: The PR changes exec approval allowlist entries to accept any `source` string. Current main accepts only the literal `"allow-always"` and has validator coverage rejecting unknown sources, so this would weaken the fail-closed gateway policy contract. Keep the literal schema or drop this hunk.
  - confidence: 0.9
- **[P2] Use the effective cross-context policy:** `src/infra/outbound/message-action-runner.ts:723`
  - body: The suppression check reads only `cfg.tools?.message?.crossContext?.allowAcrossProviders`, but current main resolves message policy through `resolveEffectiveMessageToolsConfig`, including agent-level overrides. If an agent enables cross-provider sends, a literal conflict resolution of this patch would still leak the source default account. Compute this from the effective policy used by enforcement.
  - confidence: 0.84
