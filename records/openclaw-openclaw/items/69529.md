## Summary

Keep this PR open. The outbound account-id fix direction is valid and current main still has the reported cross-provider leak, but the patch should not merge as-is because it broadens an already-fixed exec-approvals protocol field and bundles unrelated skill-description edits.

## What This Changes

The PR suppresses `input.defaultAccountId` for allowed cross-provider message sends, adds a regression test, widens exec-approval allowlist `source` metadata, and updates two bundled skill descriptions.

## Best Possible Solution

Revise or replace the branch with a focused outbound fix on current main: suppress only the source session default account for allowed cross-provider sends, preserve explicit account IDs and target agent-binding fallback, add a strong regression in the current outbound plugin-dispatch tests, add the required changelog entry, and leave exec-approvals plus skill-description work to their own tracked items.

Yes. A high-confidence unit reproduction can call `runMessageAction` with `defaultAccountId`, `tools.message.crossContext.allowAcrossProviders: true`, `toolContext.currentChannelProvider` set to a source provider, and a different target channel, then assert the target plugin receives no source default account id.

## Solution Assessment

No, not as currently bundled. The outbound account suppression is the narrow maintainable fix, but the PR should drop the broad exec-approvals schema hunk and split the unrelated skill edits before merge.
