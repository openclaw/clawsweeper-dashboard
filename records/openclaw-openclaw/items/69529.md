## Summary

Keep open: the outbound fix direction is valid and current main still has the account leak, but this PR should not merge as-is because it broadens the exec-approval source schema, bundles unrelated skill edits, lacks contributor real-behavior proof, and is currently conflicting.

## What This Changes

The PR suppresses implicit `defaultAccountId` on allowed cross-provider message sends, adds a regression test, widens exec-approvals allowlist source metadata, and updates two bundled skill descriptions.

## Best Possible Solution

Revise or replace the branch with a focused current-main fix that suppresses only the implicit source default account for allowed cross-provider sends, preserves explicit account IDs and target binding fallback, and leaves exec-approvals plus skill cleanup to their own tracked items.

## Reproduction Assessment

Yes. Source inspection of current main shows `runMessageAction` copies `input.defaultAccountId` into `params.accountId` before allowed cross-provider policy returns, and dispatch then forwards that account id; a focused unit test can exercise a source provider sending to a different target provider with `allowAcrossProviders` enabled.

## Solution Assessment

No. The outbound suppression is the right shared-runner fix, but this PR should drop the broad exec-approvals schema hunk or keep it literal-only and split the unrelated skill edits before merge.
