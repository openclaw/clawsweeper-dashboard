## Summary

Keep open. Current main and the latest release still have the Discord `thread-reply` attachment alias gap, and the open closing PR #43574 remains the focused implementation path, so closing this issue before merge would be premature.

## What This Changes

Discord thread replies should honor single-attachment aliases `media`, `path`, and `filePath` with the same precedence as Discord send.

## Best Possible Solution

Land #43574 or an equivalent Discord-owned patch that maps `thread-reply` attachments with `media` first, then `path`, then `filePath`, and add focused regression coverage for alias handling and precedence.

## Reproduction Assessment

Yes. The source trace is high-confidence: `thread-reply` reaches the Discord plugin handler, shared hydration does not normalize it, and the handler only reads `media`, so a call with only `filePath` leaves `mediaUrl` undefined.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Discord `thread-reply` handler because the downstream runtime already sends the attachment once `mediaUrl` is populated.
