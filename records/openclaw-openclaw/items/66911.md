## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Force the main migration trigger through disk:** `src/commands/models/auth-clean.ts:267`
  - body: The main-agent pre-lock migration trigger calls `ensureAuthProfileStore`, which can return a live runtime snapshot before touching disk. With a running gateway snapshot and legacy `auth.json` still present, this skips the write-enabled migration; then the locked writer can create an empty `auth-profiles.json` before loading, leaving the stale legacy profiles unpruned. Use the disk/local loader for this migration trigger and cover the runtime-snapshot + legacy-store case.
  - confidence: 0.86
- **[P2] Emit one JSON document for --json:** `src/commands/models/auth-clean.ts:197-213`
  - body: `--json` logs the cleanup plan before mutating and then logs a separate `{ ok, removed }` result after the locked update. A successful cleanup therefore writes two top-level JSON values to stdout, which breaks callers that parse command output as a single JSON document; build and emit one final result payload instead.
  - confidence: 0.94
- **[P3] Remove the nonexistent force hint:** `src/commands/models/auth-clean.ts:173-174`
  - body: The dry-run JSON warning tells users to pass `--force`, but the command only registers `--agent`, `--dry-run`, and `--json`. Replace the warning with supported remediation text, or add a real force flag only if that destructive behavior is intentionally designed and tested.
  - confidence: 0.96
- **[P3] Document the clean subcommand:** `src/cli/models-cli.ts:432-438`
  - body: This adds a user-facing auth command, but the public models/auth docs still omit `models auth clean` and do not explain dry-run, JSON, or the destructive prune behavior. Add the command to the models CLI docs so operators do not have to infer the safety contract from `--help`.
  - confidence: 0.84
