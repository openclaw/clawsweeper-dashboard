## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Load disk before creating the auth store placeholder:** `src/commands/models/auth-clean.ts:267`
  - body: This pre-lock migration trigger still goes through `ensureAuthProfileStore`, whose ensure path returns a runtime auth snapshot before reading from disk. With a main-agent legacy `auth.json` and a live runtime snapshot, cleanup can skip the disk migration before the locked writer has already created `auth-profiles.json`, so the legacy credentials are not available to prune or preserve correctly. Use a disk-backed write-enabled load that bypasses runtime snapshots for this migration step.
  - confidence: 0.9
- **[P2] Emit one JSON payload for successful cleanup:** `src/commands/models/auth-clean.ts:197-213`
  - body: In non-dry-run JSON mode this logs a plan object here, continues to mutate the store, and then logs a second `{ ok, removed }` object after the write. That makes `openclaw models auth clean --json` produce two top-level JSON documents, so scripts cannot parse stdout as the command result. Defer JSON output until after the write or include the final status in one object.
  - confidence: 0.92
- **[P3] Remove the unsupported force hint:** `src/commands/models/auth-clean.ts:173-174`
  - body: The dry-run JSON warning tells users to pass `--force`, but the new `clean` subcommand does not register a `--force` option. Following this recovery hint would fail with an unknown option, so either implement the flag with tests or change the warning to the supported recovery path.
  - confidence: 0.89
- **[P3] Document the new auth-clean subcommand:** `src/cli/models-cli.ts:432-438`
  - body: This registers a new user-facing auth command, but the Models CLI reference still enumerates auth subcommands without `clean`. Update the CLI docs/index alongside the new command so users can discover the supported cleanup workflow outside the changelog.
  - confidence: 0.82
