## Summary

Keep open. Current main still has an actionable hot-restart plugin-registry/handler-coherence gap: SIGUSR1 restarts reset task and restart state, but not the plugin loader cache or active runtime registry, while live WS dispatch still depends on the handler map assembled from the startup plugin registry.

## What This Changes

The issue reports that plugin gateway methods such as `bncr.connect` can be advertised after a SIGUSR1 in-process Gateway restart but later fail live WS dispatch as `unknown method`.

## Best Possible Solution

Add a generic hot-restart lifecycle fix so a SIGUSR1 in-process Gateway restart starts with a coherent plugin runtime registry and WS handler map, backed by a synthetic plugin regression test.

## Reproduction Assessment

Unclear. The report gives a strong manual reproduction path with `OPENCLAW_NO_RESPAWN`, SIGUSR1, advertised method listing, and a fresh WS call, but the `bncr` plugin and affected host logs are outside this checkout.

## Solution Assessment

No current fix is proven complete for this variant. The narrow maintainable direction is to reset or bypass stale plugin loader/runtime registry state at the in-process restart boundary, or otherwise make WS dispatch derive from the same live registry used for advertised methods.
