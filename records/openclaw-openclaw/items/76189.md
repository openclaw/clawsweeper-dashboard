## Summary

Keep open. Current main still has a plausible hot-restart plugin registry/cache split: the SIGUSR1 iteration reset does not reset plugin loader/runtime registry state, while Gateway WS dispatch depends on the handler map attached for the current server lifecycle.

## What This Changes

Ensure plugin gateway methods such as bncr.connect remain callable over Gateway WS after a SIGUSR1 in-process hot restart instead of becoming advertised but unknown.

## Best Possible Solution

Add a generic hot-restart lifecycle fix that keeps the active plugin runtime registry and WS handler map coherent after SIGUSR1, covered by a synthetic plugin gateway method regression test.

## Reproduction Assessment

Unclear. The report gives a strong repro path with OPENCLAW_NO_RESPAWN, SIGUSR1, and fresh WS calls to bncr.connect, but the external bncr plugin and exact host logs are not present in this checkout.

## Solution Assessment

No current fix is proven complete for this variant. The narrow maintainable direction is to reset or bypass stale plugin loader/runtime registry state at the in-process restart boundary, or make WS dispatch refresh from the same live registry used for advertised methods.
