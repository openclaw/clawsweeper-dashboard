## Summary

Keep open. Current main still has the same generic hot-restart coherence gap: SIGUSR1 in-process restart resets task/restart state but not plugin loader cache or active runtime registry state, while live WS dispatch still depends on the handler map attached from the startup plugin registry.

## What This Changes

The issue reports `bncr` plugin gateway methods becoming advertised but unknown to live Gateway WS dispatch after a SIGUSR1 in-process hot restart.

## Best Possible Solution

Add a generic SIGUSR1 in-process restart lifecycle fix so the next Gateway start uses a coherent plugin runtime registry and WS handler map, with no `bncr`-specific policy in core.

## Reproduction Assessment

No high-confidence live reproduction was established in this checkout because the `bncr` plugin and affected logs are external. Source inspection plus the issue logs give a clear reproduction shape: `OPENCLAW_NO_RESPAWN`, SIGUSR1 hot restart, cached/reused plugin registry state, and a fresh WS request to a plugin method.

## Solution Assessment

No current implementation proves this variant fixed. The maintainable fix is to reset or bypass stale plugin loader/runtime registry state at the in-process restart boundary, or otherwise ensure advertised methods and WS dispatch derive from the same fresh registry.
