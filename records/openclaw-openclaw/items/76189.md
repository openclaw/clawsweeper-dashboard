## Summary

Keep open. Current main still has a plausible generic SIGUSR1 hot-restart lifecycle gap: the in-process restart path resets task/restart state, but not plugin loader cache or active runtime registry state, while live WS dispatch depends on an attached plugin handler map.

## What This Changes

Reported that `bncr` plugin gateway methods can be registered or listed but return `unknown method` after a SIGUSR1 in-process Gateway restart.

## Best Possible Solution

Add a generic hot-restart lifecycle fix so each SIGUSR1 in-process Gateway restart starts with coherent plugin runtime registry state and WS handler bindings, backed by a synthetic plugin regression test.

## Reproduction Assessment

Unclear. The issue gives a strong manual reproduction path with `OPENCLAW_NO_RESPAWN`, SIGUSR1, fresh WS calls, and concrete logs, but the `bncr` plugin and host log bundle are outside this checkout.

## Solution Assessment

Yes for the repair direction. The maintainable fix should be generic at the restart/cache/registry boundary, not `bncr`-specific, and should prove that advertised plugin gateway methods and WS dispatch stay aligned after hot restart.

Fix the SIGUSR1 in-process Gateway restart plugin-registry/handler coherence bug reported in https://github.com/openclaw/openclaw/issues/76189. Keep the fix generic: do not hardcode `bncr` or any bundled plugin id. Investigate the restart boundary in `src/cli/gateway-cli/run-loop.ts` / `src/infra/restart.ts`, plugin loader/runtime cache state in `src/plugins/loader.ts` and `src/plugins/runtime.ts`, and Gateway handler attachment/reload behavior in `src/gateway/server.impl.ts` / `src/gateway/server-plugin-bootstrap.ts`. The expected outcome is that after an in-process SIGUSR1 restart, plugin gateway methods that are advertised/listed are backed by the live WS dispatch table, with stale loader/runtime registry state cleared, bypassed, or rebound safely. Preserve existing legitimate runtime cache reuse outside the restart boundary and avoid broad product or plugin API changes unless necessary.
