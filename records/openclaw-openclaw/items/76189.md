## Summary

Keep open. Current main has the earlier WS handler-refresh work, but I found no source or test evidence that the SIGUSR1 in-process restart boundary clears plugin loader cache or forces a fresh plugin gateway handler registry before the next gateway startup.

## What This Changes

The issue reports that `bncr` plugin gateway methods can be registered and listed but still return `unknown method` after a SIGUSR1 / `OPENCLAW_NO_RESPAWN` in-process hot restart.

## Best Possible Solution

Add a generic hot-restart lifecycle reset or fresh-load path so plugin gateway handler registries cannot be reused stale across SIGUSR1 in-process restarts, without adding `bncr`-specific core behavior.

## Reproduction Assessment

Unclear. The issue gives a concrete live sequence, and current source supports the cache-lifetime hypothesis, but I did not find an in-repo regression test or fixture that exercises plugin gateway methods across a SIGUSR1 in-process restart.

## Solution Assessment

Unclear. The current deferred-reload handler mutation is useful but not sufficient for this hot-restart variant; the safer solution is a generic restart-boundary cache/runtime reset or forced fresh gateway plugin load in core.

## Repair Work Prompt

Fix the hot-restart plugin gateway handler drift reported in https://github.com/openclaw/openclaw/issues/76189. Current evidence: after SIGUSR1 / `OPENCLAW_NO_RESPAWN` in-process restart, `run-loop.ts` resets command/restart/task state but does not invalidate plugin loader cache or reset plugin runtime registry state; `loadOpenClawPlugins` can reuse and reactivate a cached registry; WS dispatch uses the gateway startup handler map. Implement the smallest generic core fix so plugin gateway handlers are freshly bound after an in-process restart. Likely approaches: clear the plugin registry loader cache or force a fresh gateway plugin load at the restart iteration boundary, while preserving existing deferred plugin reload handler-map mutation in `server.impl.ts`. Do not add `bncr`-specific ids, defaults, migrations, or recovery policy. Add a focused regression test with a small plugin fixture or mocked registry that proves a plugin gateway method remains dispatchable after a SIGUSR1 in-process restart/cache-reuse scenario. Add a single-line `CHANGELOG.md` fix entry because this is user-facing plugin/gateway behavior.
