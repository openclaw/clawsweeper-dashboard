## Summary

Keep open. Current main still leaves an actionable hot-restart coherence gap: SIGUSR1 in-process restart does not invalidate plugin loader/runtime registry state, while the loader can reactivate cached registries and WS dispatch still depends on an attached handler map.

## What This Changes

Report that `bncr` gateway methods can be logged or listed as registered but still return `unknown method` after a SIGUSR1 / `OPENCLAW_NO_RESPAWN` in-process restart.

## Best Possible Solution

Add a generic in-process restart boundary that keeps plugin runtime registry, plugin loader cache, advertised methods, and WS handler dispatch coherent, backed by a synthetic plugin hot-restart regression test.

## Reproduction Assessment

Unclear. The issue has a strong manual reproduction path using `OPENCLAW_NO_RESPAWN`, SIGUSR1, advertised registration, and fresh WS calls, but the `bncr` plugin and host logs are outside this repository.

## Solution Assessment

Unclear because there is no proposed patch yet. The narrow maintainable direction is to reset or bypass stale plugin loader/runtime registry state at the in-process restart boundary, or make WS dispatch derive from the same live registry used for advertised methods.

## Repair Work Prompt

Fix the generic gateway hot-restart plugin handler coherence bug tracked by https://github.com/openclaw/openclaw/issues/76189. Current main still lets SIGUSR1 / OPENCLAW_NO_RESPAWN in-process restart proceed without clearing or refreshing plugin loader/runtime registry state, while WS dispatch depends on the attached gateway handler map. Implement the smallest generic fix so a hot restart cannot leave plugin gateway methods advertised or logged as registered while live WS dispatch returns `unknown method`. Likely approaches: clear the plugin registry load cache and reset/rebind active plugin runtime state at the in-process restart boundary, or make gateway WS dispatch/listing share a coherent live gateway-handler registry. Do not add bncr-specific logic or bundled plugin ids in core. Add a synthetic plugin regression test that simulates a hot restart or restart-boundary reload and proves plugin gateway handlers remain callable/listed coherently. Update `CHANGELOG.md` with a single-line user-facing fix entry if runtime behavior changes.
