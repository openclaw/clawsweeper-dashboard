## Summary

Close as implemented on current main. The gateway now refreshes the attached WS plugin handler map and advertised gateway method list when startup/deferred plugin runtime loading completes, closing the stale snapshot gap behind the reported hot-restart failure; no release tag contains the proof commit yet.

## What This Changes

Prevent bncr/plugin gateway methods from becoming advertised while live Gateway WS dispatch returns `unknown method` after a SIGUSR1 in-process hot restart.

## Best Possible Solution

Keep the generic Gateway runtime attachment fix on main; reopen or file a narrower plugin-local issue only if fresh current-main logs still show bncr diverging after this handler refresh path.

## Reproduction Assessment

No live current-main reproduction was run because the bncr plugin and host logs are external to this checkout. The issue logs plus source inspection give a clear stale handler-map reproduction shape, and current main directly covers that path.

## Solution Assessment

Yes. Refreshing the generic attached WS handler map and advertised method list from the same loaded plugin registry is the narrow maintainable fix; a bncr-specific core workaround or blanket startup cache wipe is not the better first solution.
