## Summary

Keep open. Current main still leaves the in-process restart boundary without a clear plugin loader/cache or active runtime registry reset, and the issue discussion gives a narrow hot-restart failure sequence that is not covered by the earlier generic registry-reuse fix.

## What This Changes

The issue requests fixing a SIGUSR1/OPENCLAW_NO_RESPAWN hot-restart path where plugin gateway methods can appear registered or listed but later fail live WS dispatch as unknown.

## Best Possible Solution

Add a generic in-process restart lifecycle fix so the next gateway start cannot reuse stale plugin registry/runtime state for advertised plugin gateway methods, with regression coverage that stays bncr-agnostic.

## Reproduction Assessment

No high-confidence live reproduction was established in this checkout because bncr and the affected logs are external. Source inspection plus the issue logs give a coherent reproduction path: run under OPENCLAW_NO_RESPAWN, trigger SIGUSR1, reuse plugin loader/runtime state, then call a plugin gateway method over a fresh WS connection.

## Solution Assessment

Yes, the best direction is a generic lifecycle fix at the in-process restart cache/registry boundary, not bncr-specific policy in core. The fix should either clear or bypass stale loader/runtime state before the new gateway plugin load, or otherwise prove WS dispatch and advertised methods derive from the same fresh registry.
