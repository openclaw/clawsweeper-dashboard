## Summary

Keep open. Current main has improved deferred plugin handler attachment, but the SIGUSR1 in-process restart boundary still does not clearly reset or bypass plugin loader/runtime registry cache state, while the issue discussion gives a narrow fresh-connection failure sequence after hot restart.

## What This Changes

The issue reports that plugin gateway methods such as `bncr.connect` can be registered and listed but still return `unknown method` after a SIGUSR1 in-process gateway restart.

## Best Possible Solution

Add a generic SIGUSR1 in-process restart lifecycle fix so the next gateway start uses a coherent fresh plugin runtime registry and WS handler map, without adding `bncr`-specific behavior to core.

## Reproduction Assessment

No, not as a live repo-only reproduction because the affected `bncr` plugin and logs are external. Source inspection plus the issue logs provide a clear reproducible shape around `OPENCLAW_NO_RESPAWN`, SIGUSR1 hot restart, cached plugin registry reuse, and a fresh WS call to a plugin method.

## Solution Assessment

No current implementation proves this variant fixed. The narrow maintainable solution is to reset or bypass stale plugin loader/runtime registry state at the in-process restart boundary, or otherwise prove WS dispatch and advertised plugin methods derive from the same refreshed registry.
