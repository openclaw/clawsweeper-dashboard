## Summary

Keep open. Current main still has a plausible hot-restart-specific registry/cache gap: the SIGUSR1 loop resets task/restart state but does not clear or force-refresh plugin loader/runtime registry state, while WS dispatch still depends on the handler table captured from the startup plugin registry. The related #54783 fix covered deferred reload and request-time registry churn, but this report adds newer v2026.4.26 evidence for the in-process restart boundary.

## What This Changes

This issue reports that after an OPENCLAW_NO_RESPAWN/SIGUSR1 in-process Gateway restart, a plugin method such as bncr.connect can be logged or advertised as registered while live WS dispatch returns unknown method.

## Best Possible Solution

Add a generic hot-restart lifecycle fix that makes the next Gateway start use a coherent plugin runtime registry and WS handler map, with a synthetic plugin regression proving an advertised method still dispatches after SIGUSR1.

## Reproduction Assessment

Unclear for a byte-for-byte local repro because the bncr plugin and exact host logs are outside this checkout. The report gives a strong repro shape: run with OPENCLAW_NO_RESPAWN, trigger SIGUSR1, then compare advertised plugin methods with a fresh WS call to bncr.connect.

## Solution Assessment

No current fix is proven complete for this variant. The narrowest maintainable direction is to reset or bypass stale plugin loader/runtime registry state at the in-process restart boundary, or otherwise make WS dispatch derive from the same live registry used for advertised methods.
