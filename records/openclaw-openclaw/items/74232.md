## Summary

Keep this PR open, but it is not ready to merge: the branch adds useful diagnostics, yet the synthetic feedback path still conflicts with current transcript-policy and replay-repair contracts, and the external contributor has not provided real after-fix runtime proof.

## What This Changes

The branch adds dropped-tool-call diagnostics, a report-returning sanitizer export, and synthetic error toolResult persistence/tests for assistant tool calls rejected by validation or allowedToolNames.

## Best Possible Solution

Keep bounded structured drop diagnostics, but redesign unavailable-tool feedback so it is provider-safe, replay-visible, and governed by the existing transcript policy.

## Reproduction Assessment

Yes for the blocking review issues: source inspection of the PR head shows synthetic toolResult writes without an originating assistant turn and without the existing synthetic-result policy gate, while current main replay repair drops free-floating toolResults. I did not run tests because this was a read-only review.

## Solution Assessment

No. Structured logging/reporting is a reasonable direction, but the proposed model-visible feedback should not persist orphan toolResults or bypass allowSyntheticToolResults; a safer design needs to fit the current replay pairing contract.
