## Summary

Keep this PR open for maintainer review. The latest head fixes the earlier mixed valid/invalid ordering problem, but it still persists standalone synthetic tool results that current replay repair treats as invalid/orphaned, bypasses the provider synthetic-result policy, and adds uncapped model-controlled log text.

## What This Changes

This PR adds structured dropped-tool-call diagnostics/logging and changes the session write guard/tests to persist synthetic error toolResult entries when a malformed or non-allowlisted assistant tool-call turn is fully dropped.

## Best Possible Solution

Keep the useful structured drop-reporting/observability idea, but do not persist orphan toolResult entries. Any model-visible unavailable-tool feedback should be provider-safe, honor transcriptPolicy.allowSyntheticToolResults, cap/sanitize rejected names before logging, and include replay-level tests proving the feedback survives sanitizeSessionHistory without creating invalid transcript spans.

## Reproduction Assessment

Yes. A high-confidence reproduction is an assistant message containing only a non-allowlisted or malformed tool call: the PR persists a lone toolResult for the dropped call, while current replay repair drops free-floating toolResult entries before provider replay. A second focused reproduction is the PR's own allowSyntheticToolResults=false test, which still expects an injected synthetic error result.

## Solution Assessment

No. Logging/report export is a reasonable direction, but the PR's synthetic toolResult feedback is not the safest fix because it lacks a preceding assistant tool call and bypasses the existing provider opt-out. A safer alternative is to keep diagnostics first, then add a provider-approved feedback surface with replay tests.

  - body: This path calls persistToolResult(... isSynthetic: true) without checking allowSyntheticToolResults. Gate this path on the policy or split the behavior explicitly.
  - confidence: 0.87
- **[P3] Cap raw tool names before logging:** `src/agents/session-tool-result-guard.ts:416-420`
  - body: droppedToolCallDetails stores the rejected raw name, and guardedAppend interpolates it directly into the WARN message. Names rejected for invalid tokens or length are model-controlled, so malformed output can write very long or newline-bearing values to logs. Cap and sanitize the display value before logging or storing it in details.
  - confidence: 0.8
