## Summary

Keep this PR open: the structured diagnostics/logging direction is useful and the mixed-message ordering issue was addressed, but the latest head still persists orphan synthetic tool results on full-drop paths, bypasses the provider synthetic-result opt-out, and logs model-controlled rejected names without a display cap.

## What This Changes

This PR adds dropped-tool-call diagnostics/logging, exports a report-producing sanitizer, and changes the session write guard/tests to persist synthetic error toolResult entries when an assistant tool-call turn is fully dropped.

## Best Possible Solution

Keep the structured drop-reporting/logging idea, but make rejected-name logging bounded/sanitized and route model-visible unavailable-tool feedback through a provider-safe assistant-text rewrite or the existing unknown-tool loop guard instead of orphan toolResult persistence; add focused replay tests and a changelog entry.

## Reproduction Assessment

Yes. A focused reproduction is a full-drop assistant message containing one unknown or malformed tool call: the PR persists a lone toolResult, while current replay repair treats free-floating toolResults as invalid and drops them for strict providers. A second focused check is installing the guard with `allowSyntheticToolResults: false`, which the new full-drop injection path does not honor.

## Solution Assessment

No. Logging/report export is a reasonable narrow improvement, but synthetic toolResult feedback without a persisted originating assistant tool call is not the safest provider transcript shape. A safer solution is observability first, plus provider-safe assistant-text feedback or the existing unknown-tool loop guard where model-visible correction is needed.
