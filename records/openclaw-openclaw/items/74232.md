## Summary

Keep open: the observability goal is plausible, but the proposed synthetic-feedback path writes a toolResult after dropping the assistant turn that introduced the tool call, which conflicts with OpenClaw's existing pairing repair. The PR also lacks required after-fix real behavior proof.

## What This Changes

The PR exports dropped-tool-call diagnostics, logs session-guard drops, and persists synthetic error toolResults for fully dropped unknown or malformed assistant tool calls with updated tests.

## Best Possible Solution

Keep the diagnostic/reporting direction, but rework model feedback so it never creates orphan toolResults; use the existing plain-text unknown-tool guard pattern or another paired, provider-safe contract, then prove it in a real session.

## Reproduction Assessment

Yes. Source inspection shows the PR persists a lone synthetic toolResult when the assistant message is fully dropped, and current main's repair path explicitly drops orphan toolResults. I did not run tests because this review is read-only.

## Solution Assessment

No. Persisting a toolResult after deleting the matching assistant tool-call turn is not the best fix; logging can stay, but model-visible feedback needs a provider-safe transcript shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

- **[P2] Do not persist orphaned synthetic tool results:** `src/agents/session-tool-result-guard.ts:429-435`
  - body: When `report.messages.length === 0`, the assistant turn containing `detail.id` has been dropped, so this loop writes a `toolResult` with no preceding tool call. Rework this to use a paired transcript shape or a plain-text diagnostic path.
