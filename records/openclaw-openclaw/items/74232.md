## Summary

Keep open: the observability/reporting direction is useful, but the latest diff still persists orphan synthetic tool results in the full-drop path, bypasses `allowSyntheticToolResults`, logs raw rejected names, and lacks the required changelog entry.

## What This Changes

The PR adds dropped-tool-call diagnostics/reporting and changes the session guard/tests to persist synthetic error `toolResult` records when an assistant tool-call turn is fully dropped.

## Best Possible Solution

Keep the structured drop-reporting and bounded WARN logging, but route model-visible unavailable-tool feedback through a provider-safe mechanism that respects `allowSyntheticToolResults`, then add focused replay/guard tests and a changelog entry.

## Reproduction Assessment

Yes. A focused check is a fully dropped assistant tool call: the PR persists a lone `toolResult`, while current replay repair drops free-floating tool results; a second check installs the guard with `allowSyntheticToolResults: false` and observes the new path still persists one.

## Solution Assessment

No. Logging/report export is a reasonable narrow improvement, but orphan `toolResult` feedback bypasses current transcript invariants and provider policy; observability plus a provider-safe feedback path is the safer design.
