## Summary

Keep open: the later commits fixed the mixed-message ordering concern, but the PR still writes a synthetic toolResult for a fully dropped assistant turn, leaving a tool result with no persisted assistant tool call. Current main’s pairing repair and tests treat those as orphaned and unsafe, and the external PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds dropped-tool-call diagnostics to transcript repair and changes the session guard to log dropped assistant tool calls and persist synthetic error toolResult messages for fully dropped invalid or unknown tool calls.

## Best Possible Solution

Keep the transcript pairing invariant intact; add redacted diagnostics and any model feedback through a channel that does not persist orphan toolResult entries for dropped assistant turns.

## Reproduction Assessment

Yes. The PR head source shows a full-drop path that writes a toolResult and then returns without writing the matching assistant turn, while current main’s repair code and tests classify that shape as an orphan toolResult.

## Solution Assessment

No. Logging dropped tool calls can be useful, but persisting a model-visible toolResult for a dropped assistant turn is not the narrow maintainable fix because it violates the existing pairing contract or gets stripped before the model can use it.
