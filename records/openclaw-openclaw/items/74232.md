## Summary

This PR should stay open because the diagnostics direction is useful, but the latest branch still persists synthetic toolResult messages for fully dropped assistant turns in a shape that conflicts with current strict replay repair and bypasses an existing synthetic-result policy option.

## What This Changes

The PR adds dropped-tool-call diagnostics, exports a report-producing sanitizer, logs rejected assistant tool calls, and persists synthetic error toolResult messages when a rejected assistant tool-call turn is fully dropped.

## Best Possible Solution

Keep the structured drop-reporting and bounded WARN logging, but make model-visible unavailable-tool feedback provider-safe and policy-aware, preferably through assistant-text feedback or the existing loop guard rather than orphan toolResult persistence.

## Reproduction Assessment

Yes. A focused reproduction is a fully rejected assistant message with one unknown or malformed tool call on the PR head: the guard persists a lone toolResult, while current replay repair drops toolResults without a matching accepted assistant tool-call span.

## Solution Assessment

No. The logging and report export are narrow improvements, but persisting synthetic toolResults for dropped assistant calls is not the safest transcript shape and should be replaced or gated before merge.
