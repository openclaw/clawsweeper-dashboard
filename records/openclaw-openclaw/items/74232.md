## Summary

Keep open: the latest commits fixed the first mixed-message ordering problem, but the PR still writes provider-sensitive synthetic feedback without the existing opt-out, can persist orphan toolResults, has a stale helper-export conflict with current main, and lacks real after-fix behavior proof.

## What This Changes

The PR adds dropped-tool-call diagnostics plus synthetic error toolResult feedback when the session guard rejects unknown or malformed assistant tool calls.

## Best Possible Solution

Keep bounded structured diagnostics, but redesign any model-visible unavailable-tool feedback so it is replay-visible, provider-safe, and governed by the existing synthetic-result policy.

## Reproduction Assessment

Yes for the blocking review issues: source inspection of PR head shows full-drop synthetic toolResults without an originating assistant turn or allowSyntheticToolResults gate, while current main drops orphan toolResults during replay repair. I did not run tests because this review is read-only.

## Solution Assessment

No. Structured logging/reporting is a reasonable direction, but the proposed feedback write should not bypass the synthetic-result opt-out, should not persist orphan toolResults, and should be rebased onto current helper contracts.
