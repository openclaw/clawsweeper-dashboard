## Summary

Keep this PR open; it is not safe to merge as written. The structured diagnostics direction is useful and the latest head addresses the earlier mixed-message ordering complaint, but it still persists standalone synthetic toolResult entries for fully dropped assistant turns, bypasses the existing synthetic-result policy, and logs rejected model-controlled tool names without a cap.

## What This Changes

This PR adds dropped-tool-call reporting to transcript repair and changes the session write guard/tests to warn about dropped assistant tool calls and persist synthetic error toolResult entries when an assistant tool-call turn is fully dropped.

## Best Possible Solution

Keep the structured drop report and observability pieces, but redesign unavailable-tool feedback so it is provider-safe: honor allowSyntheticToolResults, avoid standalone toolResult entries, and add regression coverage across the write guard, replay repair, and attempt-level history sanitization. If maintainers want immediate model-visible feedback, it should use an approved transcript shape rather than an orphan tool result.

## Reproduction Assessment

Yes. Append an assistant message containing only a malformed or non-allowlisted tool call: the PR head persists a standalone toolResult, while current replay repair drops free-floating toolResult entries before model context. A second focused reproduction is the PR's allowSyntheticToolResults=false test, which still expects a synthetic error result.

## Solution Assessment

No. Logging and structured diagnostics are a maintainable direction, but persisting orphan synthetic tool results is not the safest feedback surface and the path must respect the existing provider policy gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid persisting orphan synthetic tool results:** `src/agents/session-tool-result-guard.ts:429-446`
  - body: When the sanitized assistant turn is fully dropped, this branch persists only a toolResult for detail.id. There is no preceding assistant tool call for that id; current replay repair drops free-floating toolResult entries before model context, so the advertised feedback will not reliably reach the model and raw history contains an invalid span. Use a provider-safe feedback shape instead of an orphan toolResult.
  - confidence: 0.9
- **[P2] Respect synthetic-result opt-out:** `src/agents/session-tool-result-guard.ts:433-441`
  - body: This path calls persistToolResult(... isSynthetic: true) without checking allowSyntheticToolResults. Real runs pass transcriptPolicy.allowSyntheticToolResults into the guard, and that policy defaults false for providers that should not receive synthetic results; the new test even expects a synthetic error with the flag disabled. Gate this path or split the option explicitly.
  - confidence: 0.86
- **[P3] Cap raw tool names before logging:** `src/agents/session-tool-result-guard.ts:416-420`
  - body: droppedToolCallDetails keeps the rejected raw name, and guardedAppend interpolates it into a WARN line. Names rejected for length or invalid tokens are model-controlled text, so a malformed stream can write very long or newline-bearing values to logs. Cap and sanitize the display value before logging or storing details.
  - confidence: 0.78
