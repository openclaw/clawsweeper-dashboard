## Summary

Keep open; this PR is not safe to merge because the full-drop path persists orphan synthetic tool results, bypasses the existing synthetic-result opt-out, and logs rejected model-controlled tool names without a display cap. The reporting/logging direction is useful, but the feedback mechanism needs a provider-safe design and a changelog entry.

## What This Changes

The branch adds dropped-tool-call reporting/logging, exports a report-producing sanitizer, and changes the session guard/tests to persist synthetic error toolResult messages for fully dropped assistant tool-call turns.

## Best Possible Solution

Keep the structured drop-reporting/logging improvement, but avoid orphan `toolResult` persistence; if model-visible feedback is required, implement it through a provider-safe path that honors `allowSyntheticToolResults`, bounds logged names, and includes a changelog entry.

## Reproduction Assessment

Yes. On the PR head, append an assistant message with a single malformed or not-allowed tool call and the guard persists a lone `toolResult`; current replay repair then treats that shape as an orphan for strict providers. A second focused check is installing the guard with `allowSyntheticToolResults: false`, where the PR still injects the rejected-call error result.

## Solution Assessment

No. Logging/report export is a reasonable narrow improvement, but synthetic feedback as an orphan persisted `toolResult` is not the safest transcript shape and bypasses an existing provider policy switch.
