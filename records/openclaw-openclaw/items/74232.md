## Summary

Keep open and do not merge as-is: the PR improves observability, but the latest patch still persists orphan synthetic toolResult messages in the full-drop path, ignores the provider synthetic-result policy, and logs model-controlled rejected tool names without a display cap.

## What This Changes

The PR adds dropped-tool-call diagnostics/reporting, WARN logging, and synthetic error toolResult persistence for fully dropped assistant tool-call turns.

## Best Possible Solution

Keep the structured drop-reporting/logging direction, but make logging bounded and route unavailable-tool feedback through a provider-safe design such as assistant-text feedback or the existing unknown-tool loop guard instead of orphan toolResult persistence.

## Reproduction Assessment

Yes. A focused reproduction is a fully dropped assistant message containing one unknown or malformed tool call: the PR persists a lone toolResult, and current replay repair then drops that free-floating result before provider replay. A second check is installing the guard with `allowSyntheticToolResults: false`, where the PR still injects the full-drop synthetic result.

## Solution Assessment

No. The reporting/export part is a reasonable narrow improvement, but orphan synthetic toolResult persistence conflicts with current transcript repair semantics and provider policy. A safer fix should first preserve valid transcript ordering and policy boundaries.
