## Summary

Keep this PR open: the provider direction is plausible and the supplied live proof is useful, but the current patch does not actually enforce two configuration guarantees it documents for the new provider.

## What This Changes

The PR adds a bundled `openai-compatible` memory embedding provider for self-hosted OpenAI-compatible HTTP embedding servers, plus memory-lancedb docs, tests, lockfile, and changelog updates.

## Best Possible Solution

Keep the PR open, fix the provider-specific config validation and header wiring claims, add the new plugin labeler entry, then let maintainers decide whether this belongs as a bundled provider.

## Reproduction Assessment

Yes, for the review findings: source inspection shows memory-lancedb defaults an omitted model before the adapter sees it, and its config/runtime path cannot accept or forward `embedding.headers`. I did not run a local reproduction because this was a read-only review.

## Solution Assessment

No. The provider design is close, but this patch is not the best mergeable form until the documented fail-fast model requirement and optional headers behavior are either implemented through memory-lancedb or removed from the public contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate the required local model before defaulting:** `extensions/openai-compatible-embeddings/embedding-provider.ts:84-88`
  - body: This check never detects an omitted `embedding.model` when the provider is used through memory-lancedb: that plugin defaults a missing model to `text-embedding-3-small` before calling the adapter. With `embedding.provider: "openai-compatible"` plus only `baseUrl`, OpenClaw will request the OpenAI default model from a local server instead of failing fast as documented, and may initialize LanceDB with the wrong dimensions. Add provider-specific validation before the default is applied, or preserve model absence for this provider.
  - confidence: 0.9
- **[P2] Wire configured headers through memory-lancedb:** `extensions/openai-compatible-embeddings/embedding-provider.ts:97-100`
  - body: The provider advertises optional `headers` and reads `options.remote.headers`, but memory-lancedb's schema rejects `embedding.headers` and its adapter call only forwards `apiKey`/`baseUrl`. A reverse-proxied or internal server that needs custom headers will not work as promised. Add `embedding.headers` to the memory-lancedb config type/schema/runtime forwarding, or remove the advertised header support.
  - confidence: 0.88
- **[P3] Add a labeler rule for the new plugin:** `extensions/openai-compatible-embeddings/openclaw.plugin.json:2`
  - body: This adds a new bundled plugin surface under `extensions/openai-compatible-embeddings`, but `.github/labeler.yml` has no matching area label rule. Future changes to this plugin will miss the expected extension label, which the repo policy requires for new plugin surfaces.
  - confidence: 0.82
