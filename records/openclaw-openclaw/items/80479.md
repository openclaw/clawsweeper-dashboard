## Summary

Keep this PR open: the provider direction is plausible and the PR includes live adapter proof, but the patch does not yet satisfy the config contract it advertises. In the documented memory-lancedb path, missing models are defaulted before the new provider can fail fast, and custom headers are neither accepted nor passed through.

## What This Changes

Adds a bundled `openai-compatible` memory embedding provider plugin, documentation, tests, changelog text, and lockfile metadata for self-hosted OpenAI-compatible HTTP embedding servers.

## Best Possible Solution

Align the memory-lancedb config contract with the new provider before merge, then let maintainers make the bundled-provider product decision with the docs and tests matching the accepted behavior.

## Reproduction Assessment

Yes for the review findings by source inspection: current memory-lancedb code defaults missing models and rejects/drops `embedding.headers`, while the PR advertises fail-fast model validation and header support. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, not yet. A dedicated provider is a reasonable implementation boundary, but the current patch should either wire the advertised model/header behavior through memory-lancedb or narrow the docs and changelog before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve missing model state before claiming fail-fast validation:** `extensions/openai-compatible-embeddings/embedding-provider.ts:84-88`
  - body: The new provider promises a clear error when `embedding.model` is missing, but memory-lancedb currently defaults an absent model to `text-embedding-3-small` before this code runs. A user who configures `provider: "openai-compatible"` without a model will send the OpenAI default to their local server instead of getting the documented fail-fast error; make memory-lancedb require an explicit model for this provider or carry the missing/defaulted state into the adapter.
  - confidence: 0.88
- **[P2] Wire custom headers through memory-lancedb:** `extensions/openai-compatible-embeddings/embedding-provider.ts:97-100`
  - body: This provider advertises per-plugin `headers`, but memory-lancedb currently rejects unknown embedding keys and only passes `apiKey`/`baseUrl` into `remote`, so `options.remote?.headers` is never populated in the documented workflow. Add `headers` to the memory-lancedb config parser/pass-through/tests, or remove the advertised option so reverse-proxy setups do not configure a field that cannot work.
  - confidence: 0.84
