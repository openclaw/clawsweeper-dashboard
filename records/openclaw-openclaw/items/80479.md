## Summary

Keep open. This is an active, same-author feature PR with supplied adapter-level proof and a linked open feature issue, but the diff has a source-proven config/docs mismatch around advertised custom embedding headers that should be fixed before merge.

## What This Changes

Adds a bundled `openai-compatible-embeddings` plugin registering an explicit memory embedding provider for self-hosted OpenAI-compatible HTTP servers, plus docs, changelog, lockfile, and targeted adapter tests.

## Best Possible Solution

Accept the dedicated provider only if maintainers want this bundled boundary, with config, docs, and tests aligned so every advertised setting works through memory-lancedb.

## Reproduction Assessment

Yes for the review finding: source inspection shows memory-lancedb currently rejects and does not forward `embedding.headers` while the PR advertises that setting. The broader request is a new feature rather than a current-main bug reproduction.

## Solution Assessment

No, not as currently written. The dedicated-provider direction is plausible, but the patch should either wire advertised custom headers through memory-lancedb config and forwarding or remove the claim before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Wire advertised embedding headers through config:** `CHANGELOG.md:9`
  - body: The changelog tells users they can set optional `headers` under `plugins.entries.memory-lancedb.config.embedding`, but that config currently only allows `provider`, `apiKey`, `model`, `baseUrl`, and `dimensions`, and `ProviderAdapterEmbeddings` only forwards `apiKey`/`baseUrl`. Users following this will hit an unknown-key config error or never send their custom headers. Please either add `headers` to the memory-lancedb config/schema and forward it into `remote.headers`, with coverage, or remove the advertised support.
