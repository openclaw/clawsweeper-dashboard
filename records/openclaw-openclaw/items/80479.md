## Summary

Keep open: the provider addition is plausible and has live adapter proof, but the PR currently documents optional `embedding.headers` for memory-lancedb without adding that key to memory-lancedb's parser, manifest schema, or adapter forwarding path.

## What This Changes

Adds a bundled `openai-compatible` memory embedding provider plugin, tests, memory-lancedb docs, lockfile metadata, a Vitest config, and a changelog entry for self-hosted OpenAI-compatible HTTP embedding servers.

## Best Possible Solution

Ship the provider only if maintainers accept core bundling, with memory-lancedb docs, config schema, parsing, forwarding, and tests aligned to the exact supported options.

## Reproduction Assessment

Not applicable for the feature request itself. For the review finding, the mismatch is source-reproducible: current memory-lancedb rejects unknown `embedding.headers` while the PR advertises headers in the changelog/body.

## Solution Assessment

No as submitted. A dedicated adapter is a reasonable technical shape through `registerMemoryEmbeddingProvider`, but the branch should either implement memory-lancedb header support end to end or stop documenting it, and maintainers still need to decide the bundling boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Remove or implement documented embedding headers:** `CHANGELOG.md:9`
