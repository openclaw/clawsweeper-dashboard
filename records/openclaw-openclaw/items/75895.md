## Summary

Keep open. Current `main` still has no whole-command timeout for `openclaw message send`, no message-scoped SIGTERM/SIGINT abort path, and only bounds the Gateway RPC after dispatch is reached.

## What This Changes

The issue requests a bounded internal timeout and prompt SIGTERM/SIGINT handling for `openclaw message send`.

## Best Possible Solution

Add a message-CLI-owned timeout and AbortController that starts before plugin loading, converts SIGTERM/SIGINT into prompt cancellation/exit behavior, and threads the signal through `messageCommand` into outbound message actions while preserving existing send, media, presentation, and Gateway dispatch contracts.

## Reproduction Assessment

Yes. A focused unit test can make `messageCommand` or pre-dispatch plugin loading never settle and observe that `runMessageAction` has no timeout; a process-level regression should cover SIGTERM because current source has no message-scoped signal abort path.

## Solution Assessment

Yes. A narrow CLI wrapper timeout/signal path is the maintainable fix because the existing Gateway RPC timeout is too late in the flow and outbound/local paths already expose abort checkpoints once a signal is threaded in.
