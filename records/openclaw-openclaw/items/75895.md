## Summary

Keep open. Current main still has the reported reliability gap: the message CLI wrapper bounds only the gateway-stop hook, while plugin loading, config/secret resolution, message dispatch, and process signals are not covered by a whole-command timeout or message-scoped abort path.

## What This Changes

The issue requests a bounded internal timeout and prompt SIGTERM/SIGINT handling for `openclaw message send`.

## Best Possible Solution

Add a message-CLI-owned timeout and AbortController before plugin loading, convert SIGTERM/SIGINT into prompt cancellation, and thread the signal through `messageCommand` into outbound message actions without changing unrelated Telegram gateway-dispatch behavior.

## Reproduction Assessment

Yes. A focused unit test can make `messageCommand` or pre-dispatch plugin loading never settle and observe that `runMessageAction` has no internal timeout; a process-level regression should cover SIGTERM handling.

## Solution Assessment

Yes. The narrow maintainable fix is a CLI wrapper timeout and signal path that feeds the existing outbound `abortSignal` seams while leaving the existing gateway RPC timeout as a downstream guard.
