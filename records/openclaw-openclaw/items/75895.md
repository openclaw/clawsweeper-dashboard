## Summary

Keep open. Current main still lacks a whole-command timeout or message-scoped SIGTERM/SIGINT abort path for `openclaw message send`; only the downstream Gateway RPC is bounded after dispatch is reached.

## What This Changes

Add an internal timeout and prompt signal handling for `openclaw message send` so stuck deliveries fail clearly instead of hanging indefinitely.

## Best Possible Solution

Add a message-CLI-owned timeout and AbortController that starts before plugin loading, converts SIGTERM/SIGINT into prompt cancellation, and threads the signal through `messageCommand` into outbound message actions while preserving existing Gateway dispatch behavior.

## Reproduction Assessment

Yes. A focused test can make `messageCommand` or pre-dispatch plugin loading never settle and observe that `runMessageAction` has no internal timeout; the external SIGTERM case should be covered with a process-level regression because current source has no message-scoped signal wiring.

## Solution Assessment

Yes. A narrow CLI wrapper timeout/signal boundary is the maintainable fix because the existing Gateway RPC timeout is too late in the flow and the outbound runner already has abort-signal checkpoints once a signal is threaded in.
