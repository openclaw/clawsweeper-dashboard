## Summary

Keep open. Current main still has no `message send` command-level timeout or message-scoped SIGTERM/SIGINT abort path, and the open closing PR #76077 is still unmerged.

## What This Changes

Add an internal timeout and signal-aware cancellation to `openclaw message send` so hung deliveries fail with a clear CLI error instead of spinning indefinitely.

## Best Possible Solution

Review and repair #76077 or a replacement so the timeout starts before plugin loading, SIGTERM/SIGINT cancellation is scoped and disposed, timeout input is validated, and docs/changelog/tests cover the behavior.

## Reproduction Assessment

Yes for the missing internal timeout: a focused unit test can hold `ensurePluginRegistryLoaded` or `messageCommand` pending and observe that current `runMessageAction` has no timer. The external SIGTERM symptom should be covered with a process-level CLI regression because current main has no message-scoped signal handler.

## Solution Assessment

Yes, the requested CLI-level timeout and signal-abort direction is the narrow maintainable fix. Relying only on the gateway RPC timeout is insufficient because current main still leaves pre-dispatch CLI work unbounded.
