## Summary

Keep open. Current main only bounds the Gateway RPC after dispatch; the `openclaw message send` CLI still loads plugins and awaits the message command without a whole-command timeout or signal-driven abort.

## What This Changes

Add a bounded internal timeout and prompt SIGTERM/SIGINT handling for `openclaw message send` so stuck deliveries fail clearly instead of hanging.

## Best Possible Solution

Add a message-CLI-owned timeout and AbortController that starts before plugin loading, converts SIGTERM/SIGINT into prompt abort/exit behavior, and threads the signal through `messageCommand` into outbound message actions while preserving Gateway dispatch semantics.

## Reproduction Assessment

Yes for the missing internal timeout: a focused unit test can make `messageCommand` or pre-dispatch plugin loading never settle and observe that `runMessageAction` does not time out. The external SIGTERM symptom should be covered by a process-level regression because current source has no message-scoped signal abort path.

## Solution Assessment

Yes. A narrow CLI wrapper timeout/signal path is the maintainable fix because the existing Gateway RPC timeout is too late in the flow and local/plugin fallback paths already accept abort signals once one is threaded in.
