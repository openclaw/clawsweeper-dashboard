## Summary

Keep open. Current main still lacks a `message send` command-level timeout and message-scoped SIGTERM/SIGINT cancellation; the open closing PR #76077 is unmerged, so this issue should remain open until a fix lands.

## What This Changes

The issue requests a default internal timeout, prompt SIGTERM handling, and a clear timeout error for `openclaw message send`.

## Best Possible Solution

Review and repair #76077, or a replacement, so the timeout starts before plugin loading, signal handlers are scoped and disposed, timeout input is validated, and tests/docs/changelog cover the behavior.

## Reproduction Assessment

Yes for the missing internal timeout: hold `ensurePluginRegistryLoaded` or `messageCommand` pending and current `runMessageAction` has no timer. The external SIGTERM symptom should be covered with a process-level CLI regression because current main has no message-scoped signal wiring.

## Solution Assessment

Yes, the requested CLI-level timeout plus signal-abort path is the narrow maintainable fix. The existing gateway RPC timeout is not sufficient because pre-dispatch CLI work and plugin loading sit outside it.
