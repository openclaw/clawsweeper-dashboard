## Summary

Keep open. Current main still has no command-level timeout or message-scoped signal abort for `openclaw message send`; only the delegated Gateway RPC has a timeout once the action reaches that path.

## What This Changes

The issue asks for `openclaw message send` to fail clearly after a bounded internal timeout and respond promptly to SIGTERM from supervisors such as GNU `timeout`.

## Best Possible Solution

Add a message-CLI-owned cancellation boundary that starts before channel/plugin setup, applies a default bounded timeout, maps SIGTERM/SIGINT to prompt abort/exit, and reuses the existing outbound abort seams without changing channel delivery semantics.

## Reproduction Assessment

Yes. A focused unit test can hold `messageCommand` or pre-dispatch plugin loading pending and show the CLI wrapper never times out; a process-level test should cover SIGTERM behavior from an external supervisor.

## Solution Assessment

Yes. The narrowest maintainable fix is a CLI-level timeout and signal-abort wrapper around the whole message command, coordinated with the existing Telegram gateway-dispatch work rather than a channel-only workaround.
