## Summary

Keep open. Current main still lacks a `message send` command-level timeout and a message-scoped SIGTERM/SIGINT abort path, and #76077 is an open closing PR that should be reviewed or repaired before this issue closes.

## What This Changes

Add an internal timeout and prompt external signal handling to `openclaw message send` so stuck delivery attempts fail with a clear terminal error.

## Best Possible Solution

Land a repaired #76077 or replacement that starts the timeout before plugin registry loading, scopes and disposes SIGTERM/SIGINT cancellation, validates timeout input, and adds focused CLI tests plus any required docs/changelog entry.

## Reproduction Assessment

Yes for the missing internal timeout: a focused unit test can mock `ensurePluginRegistryLoaded` or `messageCommand` to never settle and observe that `runMessageAction` has no timer or abort path. The external-supervisor symptom should be covered with a process-level CLI regression because this review did not run the live SIGTERM path.

## Solution Assessment

Yes, the requested CLI-level timeout and signal-abort direction is the narrow maintainable fix. Current main does not implement it, and the open PR should be reviewed or repaired rather than closing the issue early.
