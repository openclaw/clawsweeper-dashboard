## Summary

Keep open. Current main still has no whole-command timeout or message-scoped SIGTERM/SIGINT abort path for `openclaw message send`; the only timeout I found is the narrower Gateway RPC budget once gateway dispatch is reached, and Telegram dispatch itself is still entangled with the related open gateway-dispatch lane.

## What This Changes

The issue asks for `openclaw message send` to enforce a default internal timeout, terminate promptly on SIGTERM, and report a clear timeout error.

## Best Possible Solution

The end state should be a CLI-owned cancellation budget that starts before plugin loading, converts SIGTERM/SIGINT into prompt aborts, reuses existing outbound abort seams, and preserves the Gateway dispatch contract.

## Reproduction Assessment

Yes. A focused test can hold `messageCommand` or pre-dispatch plugin loading pending and observe that the CLI helper never times out; a process-level regression can cover SIGTERM because current source has no message-action signal wiring.

## Solution Assessment

Yes. A narrow CLI wrapper timeout/signal implementation is the best fix because it bounds the full user command while allowing the existing outbound runner and Gateway RPC timeout paths to remain generic.
