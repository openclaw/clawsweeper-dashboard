## Summary

Keep open. Current main still lacks a `message send` command-level timeout and message-scoped SIGTERM/SIGINT abort path; the only bounded path is the gateway RPC once dispatch reaches `message.action`, and the linked closing PR #76077 is still open with required review fixes.

## What This Changes

Add a bounded timeout and prompt SIGTERM/SIGINT handling for `openclaw message send` so cron deliveries cannot hang indefinitely.

## Best Possible Solution

Review and repair #76077 or a replacement so the timeout starts before plugin loading, SIGTERM/SIGINT cancellation is scoped and disposed, timeout input is validated, and docs/changelog/tests cover the new behavior.

## Reproduction Assessment

Yes for the missing internal timeout: a focused unit test can hold `ensurePluginRegistryLoaded` or `messageCommand` pending and observe that current `runMessageAction` has no timer. The exact external SIGTERM symptom should be covered with a process-level CLI regression because current main has no message-scoped signal wiring.

## Solution Assessment

Yes, the requested CLI-level timeout and signal-abort direction is the narrow maintainable fix. The open PR is directionally aligned but is not yet the best implementation until it cleans up signal listeners and adds the required validation, docs, changelog, and tests.
