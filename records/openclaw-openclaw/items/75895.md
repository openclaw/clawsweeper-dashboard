## Summary

Keep open. Current main has a bounded Gateway RPC path for Telegram only after dispatch reaches `message.action`, but the CLI still has no overall `openclaw message send` timeout, no message-scoped SIGTERM/SIGINT abort path, and still performs pre-dispatch plugin loading outside any timeout.

## What This Changes

Add a bounded internal timeout and prompt SIGTERM handling for `openclaw message send`, especially Telegram sends run from cron.

## Best Possible Solution

Add a message-CLI-owned AbortController that starts before plugin loading, enforces a default bounded timeout, converts SIGTERM/SIGINT into prompt abort plus bounded exit, and preserves existing presentation/media/send contracts.

## Reproduction Assessment

Yes for the missing internal timeout: a focused test can hold `messageCommand` or pre-dispatch plugin loading pending and observe that `runMessageAction` never times out. The exact external SIGTERM symptom should be covered with a process-level regression because current source has no message-action abort wiring.

## Solution Assessment

Yes. A narrow CLI wrapper timeout/signal path is the right fix because it bounds the whole user command while reusing the existing outbound `abortSignal` seams and the already-bounded Gateway request path.
