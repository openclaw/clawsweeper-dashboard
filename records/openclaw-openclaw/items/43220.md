## Summary

Keep open. Current main still does not provide a privacy-safe OpenClaw-to-OpenClaw agent/package/profile export-import surface; the shipped backup and migrate paths solve adjacent recovery and external-provider import cases, while the overlapping PRs/issues show the product and security boundary is still unsettled.

## What This Changes

The issue requests a standardized, privacy-safe packaging model for sharing, migrating, and reproducing OpenClaw agents without copying secrets or runtime state.

## Best Possible Solution

Define a maintainer-approved portability model separate from full backup: versioned manifest/archive, strict secret and runtime-state exclusion, dependency metadata for skills/plugins, preview-first non-destructive import, and docs that distinguish profile sharing from disaster recovery.

## Reproduction Assessment

Not applicable as a runtime bug. The feature-gap check is high-confidence from current source and docs: no shipped `profile export/import`, no OpenClaw package export/import, `migrate` is provider import, and `backup` is broad local-state create/verify.

## Solution Assessment

Unclear for the exact Docker-like `agent-image` proposal. The user problem is valid, but the best implementation needs maintainer product/security choice between profile portability, safer backup/restore, provider migration extensions, and ClawHub/plugin distribution.

## Risks / Open Questions

- A native package/import workflow can leak secrets or private memory unless the archive schema excludes or transforms credentials, sessions, logs, local paths, and runtime state by default.
- Any restore/import path that writes OpenClaw state needs explicit containment, symlink, conflict, rollback, dry-run, and trusted-source rules before automation is safe.
- There are overlapping open trackers and an active narrower PR, so a drive-by fix risks duplicating or conflicting with the eventual maintainer-approved portability model.
