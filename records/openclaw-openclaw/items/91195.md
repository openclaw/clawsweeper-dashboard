## Summary

Keep open: current main still closes Feishu streaming cards only through the dispatcher idle/error settlement path, and the reported stuck case is specifically the path where the Feishu turn never reaches `dispatch complete` or card close. Current tests cover empty-card close and no-visible-reply fallback after idle, but they do not prove recovery for an already-started CardKit session whose dispatch never settles.

## What This Changes

The issue reports a Feishu DM regression where a streaming CardKit reply starts after a delayed agent turn but never closes, leaving an empty open card in the Feishu client.

## Best Possible Solution

Add a bounded Feishu streaming-card recovery path that closes or visibly falls back when a started card is abandoned, while preserving normal long-running turns and the existing `channels.feishu.streaming=false` workaround.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. The reporter supplied concrete v2026.6.1 logs and current source shows the card close depends on dispatcher settlement, but this review did not recreate the delayed Feishu turn against v2026.6.6.

## Solution Assessment

Unclear. A Feishu-owned lifecycle recovery path is the likely right layer, but the timeout/fallback semantics need maintainer and live Feishu validation so normal long turns are not closed prematurely.
