## Summary

Keep open. Current main still lacks the requested explicit ordered fallback chain, retryable-error contract, fallback-attempt logging, and aggregated all-provider error; the only existing retry behavior is the current auto-selected-provider path, while explicit provider failures are still intentionally thrown.

## What This Changes

The issue requests an ordered `web_search` provider fallback chain for recoverable quota, rate-limit, and network failures, including auto-mode fallbacks and aggregated diagnostics.

## Best Possible Solution

Define a backward-compatible fallback config and normalized provider error contract, then implement ordered retries with clear logging and aggregated failure reporting while preserving single-provider behavior by default.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main confirms the requested fallback fields are absent and explicit-provider failures are intentionally not retried today.

## Solution Assessment

No current implementation fully solves the request. The narrow maintainable path is to settle the config shape and provider error taxonomy before changing runtime retry behavior.
