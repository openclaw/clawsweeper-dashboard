## Summary

Keep open. Current main and v2026.6.10 still only support implicit auto-detected web_search fallback; the requested explicit fallback chain, recoverable-error taxonomy, DuckDuckGo-final ordering, fallback logging, and aggregated final error remain unresolved, and https://github.com/openclaw/openclaw/pull/63571 is still only an open conflicting candidate.

## What This Changes

The issue requests a configurable web_search provider fallback chain for recoverable quota, rate limit, network, and temporary provider failures.

## Best Possible Solution

Define and land a documented opt-in fallback contract that validates fallback provider ids, activates credentials deliberately, retries only approved recoverable failures, logs attempts, aggregates final errors, and preserves current single-provider behavior when no chain is configured.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection gives high confidence that current main lacks the explicit fallback-list contract and keeps configured provider failures fail-fast.

## Solution Assessment

No current implementation fully solves the issue. The requested direction is plausible, but the retry taxonomy, config shape, credential activation semantics, and provider trust boundaries should be settled before landing a refreshed implementation.
