## Summary

Keep open. Current main and v2026.6.6 only provide auto-selected provider fallback; the requested explicit `web_search` fallback chain, retryable-error taxonomy, logging, and aggregated failure behavior are still missing, and the related open PR is conflicting and not yet proof-ready.

## What This Changes

The issue asks for an opt-in `web_search` provider fallback chain that retries recoverable runtime failures and preserves current single-provider behavior by default.

## Best Possible Solution

Define and land a documented opt-in fallback contract with retryable-error classification, explicit credential activation semantics, fallback-attempt logging, aggregated final errors, and focused runtime/config/docs tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main ignores the requested explicit fallback-list fields and deliberately keeps explicit provider failures fail-fast.

## Solution Assessment

No current implementation fully solves the issue. The requested direction is plausible, but the config shape, retryable-error taxonomy, and credential activation rules need maintainer approval before a replacement or refreshed PR should land.
