## Summary

Keep open. Current main still supports single-provider web_search selection plus auto-detection fallback, but not the requested explicit fallback chain, retryable-error taxonomy, attempt logging, or aggregated all-provider diagnostics; prior related discussion includes useful design notes and a manual Exa/Tavily/DDG workaround, but those did not land as a core fallback contract.

## What This Changes

Request an ordered web_search provider failover chain for quota, rate-limit, timeout, network, and provider-side temporary failures, with DuckDuckGo only as the final fallback.

## Best Possible Solution

Define a backward-compatible web_search fallback config and normalized provider failure contract, then update runtime ordering, diagnostics, schema, docs, and regression tests while preserving single-provider behavior by default.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that the requested config fields and explicit-provider retry behavior are absent on current main.

## Solution Assessment

Unclear until maintainers choose the public config shape and error taxonomy. The requested direction is reasonable, but implementing it safely should start with a documented fallback/error contract rather than a narrow runtime retry loop only.
