## Summary

Keep open. Current main and the latest stable release still support a single configured `web_search` provider plus implicit auto-mode fallback, but they do not implement the requested ordered fallback chain, retryable-error contract, fallback logging, or aggregated all-provider diagnostics.

## What This Changes

The issue requests an ordered `web_search` fallback chain across configured providers for recoverable quota and network failures, with DuckDuckGo only as the final fallback.

## Best Possible Solution

Define a backward-compatible web-search fallback contract, normalize provider errors for retry classification, preserve current single-provider behavior by default, and document/test explicit and auto fallback modes.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection gives high confidence that the requested fallback config is absent and explicit-provider failures still throw today.

## Solution Assessment

No current implementation fully solves the issue. The existing auto-selected fallback path is useful, but the requested explicit ordered chain needs maintainer decisions on config shape, retryable error normalization, and credential behavior before implementation.
