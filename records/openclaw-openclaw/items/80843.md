## Summary

Keep open. Current main and the latest shipped web-search docs still only support single explicit provider selection plus limited auto-selected fallback behavior; the requested explicit fallback chain, retryable-error contract, logging, and aggregated failure reporting remain unimplemented and require maintainer product/API design.

## What This Changes

The issue requests an ordered web_search fallback chain across configured providers for quota, rate-limit, and transient network failures.

## Best Possible Solution

Define a documented, backward-compatible web-search fallback contract that preserves current single-provider behavior by default, normalizes provider failure reasons, keeps credential activation explicit, and adds focused runtime/config/docs coverage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that current main lacks the requested fallback config fields and deliberately does not retry explicit provider failures.

## Solution Assessment

No current implementation fully solves the request. The existing auto-selected fallback path is useful, but the requested explicit ordered chain and retryable-error policy need a maintainer-approved public contract before implementation.
