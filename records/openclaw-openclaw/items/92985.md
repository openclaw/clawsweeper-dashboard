## Label Justifications

- `P1`: The issue describes a recent Discord multi-account regression that can take a previously working channel account offline for real users.
- `impact:message-loss`: When the default Discord account stays offline and guild channels remain unresolved, users lose visible Discord channel delivery.
- `impact:auth-provider`: The suspected failure is SecretRef-backed Discord token resolution against the active runtime snapshot.
- `impact:crash-loop`: The reported default account repeatedly restarts after startup throws on `configured_unavailable`.

## Summary

Keep open: current main and v2026.6.6 still contain the Discord inspect-mode token path that can turn unresolved SecretRefs into `configured_unavailable`, then fail startup and enter account restart recovery. The related open send-path issue is adjacent, but it does not fully cover this default-account hot-reload outage.

## What This Changes

The issue reports that adding a second file-SecretRef-backed Discord account via hot reload can take the existing default Discord account offline in an auto-restart loop.

## Best Possible Solution

Add focused regression coverage for default-plus-second Discord file SecretRef accounts through startup and hot reload, then repair runtime snapshot materialization or token selection so each account starts from the resolved active snapshot without adding ad hoc fallback paths.

## Reproduction Assessment

No live current-main Discord repro was run in this read-only review. Source inspection gives a high-confidence failure branch: unresolved Discord SecretRefs become `configured_unavailable`, Discord startup throws, and the gateway schedules account restart attempts.

## Solution Assessment

Unclear whether the reporter's suggested fix location is the final best layer. The maintainable direction is to fix runtime snapshot materialization or account-token selection for Discord startup/hot reload, not add a new fallback or config option.
