## Label Justifications

- `P1`: The report describes a supported Discord multi-account workflow taking the existing Discord channel account offline.
- `impact:message-loss`: The default Discord account never reconnects, leaving channels unresolved and Discord delivery unavailable.

## Summary

Keep open: current main still has a source-backed path where Discord account startup fails closed on `configured_unavailable` if the hot-reload path restarts the channel before the new file-backed SecretRefs are materialized into the active runtime snapshot. Related closed SecretRef issues cover earlier startup/status or send paths, not this default-account outage when adding a second account by filesystem hot reload.

## What This Changes

The issue reports that adding a second Discord account by hot reload can make the existing default Discord account loop in auto-restart when account tokens use file-backed SecretRefs.

## Best Possible Solution

Ensure filesystem hot reload activates a fully resolved secrets runtime snapshot before restarting Discord accounts, then add a regression for default plus second file-SecretRef accounts and follow with live Discord proof.

## Reproduction Assessment

Yes, at source level with medium confidence: the watcher reload path can restart channels before a fresh resolved secrets snapshot is promoted, and Discord startup throws on `configured_unavailable`. I did not run a live gateway with real Discord accounts.

## Solution Assessment

Unclear as an implementation choice: the likely best fix is at the gateway reload/secrets activation boundary, not an ad hoc Discord token fallback, but that boundary is compatibility-sensitive and already marked for maintainer review.
