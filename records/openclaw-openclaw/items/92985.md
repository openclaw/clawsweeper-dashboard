## Label Justifications

- `P1`: The report describes a recent supported Discord multi-account regression that can take an existing channel account offline for real users.
- `impact:message-loss`: When the default Discord account remains offline, Discord guild channels stay unresolved and messages cannot be delivered through that channel.
- `impact:auth-provider`: The suspected failure is SecretRef-backed Discord token resolution against the active runtime snapshot.
- `impact:crash-loop`: The default account repeatedly enters auto-restart attempts after startup throws on `configured_unavailable`.

## Summary

Keep open: this is a recent, concrete Discord startup/hot-reload outage, and current main still has the inspect-mode Discord token path that can turn unresolved SecretRefs into `configured_unavailable` before channel startup throws and auto-retries. The open related send-path issue appears adjacent, but it does not fully cover this default-account startup outage.

## What This Changes

Adding a second Discord account via hot reload should not make the existing default SecretRef-backed account resolve unavailable and restart-loop offline.

## Best Possible Solution

Add a focused regression around Discord default-plus-named SecretRef accounts during hot reload, then repair the runtime snapshot/account-token boundary so each account starts from the resolved active snapshot.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection does reproduce the core failure branch: unresolved Discord SecretRef input becomes `configured_unavailable`, startup throws, and the gateway schedules account restart attempts.

## Solution Assessment

Unclear whether the reporter's suggested fix location is the final best fix, but the direction is correct: repair snapshot materialization or account-token selection, not add a config workaround or close as the send-path issue.
