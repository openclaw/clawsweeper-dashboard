## Summary

Keep open: current main still contains the Discord inspect-mode SecretRef branch that can surface `configured_unavailable`, startup still throws on that status, and the gateway still auto-retries failed channel starts. The adjacent open send-path issue is related but does not cover this default-account provider-startup outage.

## What This Changes

The issue reports that adding a second Discord account with a file-backed SecretRef can make the existing default Discord account resolve its token as unavailable and loop through auto-restarts.

## Best Possible Solution

Keep this open as the Discord startup/hot-reload SecretRef outage, add a focused regression for default plus named file-backed accounts, and repair the runtime snapshot/account-token boundary without adding a new config fallback.

## Reproduction Assessment

Yes for the source branch, no for live behavior: current main still maps inspect-mode unresolved Discord SecretRefs to `configured_unavailable`, startup throws on that status, and the gateway retry loop explains the reported logs. I did not run a live Discord gateway with file-backed tokens in this read-only review.

## Solution Assessment

Unclear whether the reporter's suggested fix location is the final best fix. The maintainable direction is to make Discord startup consume the active resolved snapshot consistently, but maintainers should choose whether that belongs in snapshot materialization, account-token selection, or both.
