## Label Justifications

- `P1`: The report describes a supported Discord multi-account regression that can take an existing live channel account offline.
- `impact:message-loss`: The default Discord account never reconnects, leaving guild channels unresolved and visible message delivery offline.
- `impact:auth-provider`: The failure path is SecretRef-backed Discord token materialization and account authentication against the active runtime snapshot.
- `impact:crash-loop`: The default Discord account repeatedly enters bounded auto-restart attempts after startup throws on an unavailable configured token.

## Summary

Keep open: current main and latest release still have a source-level path where adding explicit Discord accounts can leave the inherited default top-level token SecretRef unmaterialized, and Discord startup still fails closed into the reported restart loop when that token is `configured_unavailable`.

## What This Changes

The issue requests a fix for Discord multi-account hot reload so adding a second SecretRef-backed account does not knock the existing default account offline.

## Best Possible Solution

Repair shared channel SecretRef runtime collection so the implicit default Discord account's inherited top-level token is materialized when a named account is added, with a focused regression test and live hot-reload proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a top-level Discord token SecretRef plus an `accounts` map containing only a token-owning second account can skip default-token materialization, then Discord startup sees `configured_unavailable`. No live Discord gateway hot-reload run was performed in this read-only review.

## Solution Assessment

No patch is proposed yet. The best fix is at the shared SecretRef runtime snapshot/account-surface boundary, not an ad hoc Discord token fallback that would create a second credential source of truth.
