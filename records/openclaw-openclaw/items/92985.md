## Label Justifications

- `P1`: The issue describes a recent Discord multi-account regression that can take an existing channel account offline for real users.
- `impact:message-loss`: When the default Discord account stays offline, guild channels remain unresolved and Discord delivery is unavailable.
- `impact:auth-provider`: The suspected failure is Discord token SecretRef resolution against the active runtime snapshot.
- `impact:crash-loop`: The default account repeatedly enters auto-restart attempts after startup throws on configured_unavailable.

## Summary

Keep open: current main and v2026.6.8 still contain the Discord inspect-mode token path and fail-closed startup branch that can turn an unresolved account SecretRef into repeated account restarts. Related SecretRef reports cover status, send, or external contract loading paths, but not this default-account hot-reload outage.

## What This Changes

Fix Discord multi-account hot reload so adding a second SecretRef-backed account does not take the existing default account offline.

## Best Possible Solution

Add a focused regression for default-plus-second Discord SecretRef accounts during hot reload, then repair the snapshot materialization or account-token selection path so each account starts from the resolved active snapshot.

## Reproduction Assessment

No high-confidence live current-main repro was run. Source inspection does reproduce the central failure branch: unresolved Discord SecretRef input becomes configured_unavailable, startup throws, and gateway channel management schedules restart attempts.

## Solution Assessment

Unclear whether the final fix belongs in Discord token selection or runtime snapshot materialization. The maintainable direction is to keep SecretRefs resolved through the active snapshot and add a hot-reload regression, not a new fallback or config workaround.
