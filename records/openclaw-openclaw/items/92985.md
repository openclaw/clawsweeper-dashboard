## Label Justifications

- `P1`: The report describes a recent supported Discord multi-account regression that can take an existing channel account offline.
- `impact:message-loss`: When the default Discord account remains offline, Discord guild channels stay unresolved and messages cannot be delivered through that channel.
- `impact:auth-provider`: The suspected failure is SecretRef-backed Discord token resolution against the active runtime snapshot.
- `impact:crash-loop`: The default account repeatedly enters auto-restart attempts after startup throws on configured_unavailable.

## Summary

Keep open: current main still has a source-reproducible Discord SecretRef/runtime-snapshot mismatch that can leave the implicit default account reading an unresolved top-level token after explicit accounts are added. No merged PR or current-main evidence shows the hot-reload default-account outage is fixed.

## What This Changes

The issue asks for adding a second Discord account with SecretRef-backed tokens to leave the existing default account connected during hot reload or restart.

## Best Possible Solution

Repair the Discord runtime snapshot/account-token boundary so the implicit default account's top-level token is still resolved or promoted when explicit non-default accounts are added, without adding a new fallback source of truth.

## Reproduction Assessment

No, not as a live end-to-end repro. Source inspection does give a clear current-main failure path: the runtime collector can skip resolving the top-level Discord token while Discord still synthesizes a default account that reads it, which then throws and auto-retries.

## Solution Assessment

Unclear which owner boundary is best without maintainer review. The right fix should align the generic channel secret surface or Discord's implicit-default account contract, not add direct file/env fallback reads in the Discord startup path.
