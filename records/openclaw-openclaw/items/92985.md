## Summary

Keep open: current main still has a source-reproducible path where Discord keeps an implicit default account from the top-level token, but the SecretRef runtime snapshot collector can leave that top-level token unresolved once explicit named accounts exist.

## What This Changes

The issue reports that adding a second Discord account with a file-backed SecretRef causes the existing default Discord account to resolve its token as unavailable and enter an auto-restart loop.

## Best Possible Solution

Make SecretRef runtime snapshot activation use an account surface that matches Discord's implicit default-account contract, then add regression coverage for a top-level file SecretRef plus a named account added through reload.

## Reproduction Assessment

Yes by source inspection: Discord lists an implicit default account when a top-level token exists, while the SecretRef collector can leave that top-level token unmaterialized after named accounts are introduced. I did not run a live Discord hot-reload repro in this read-only review.

## Solution Assessment

The reporter's direction is broadly right, but the maintainable fix should align snapshot materialization with the account-listing contract rather than adding per-call token fallback behavior. That keeps runtime reads on the active snapshot and avoids another SecretRef resolution path.
