## Summary

Keep open: the report is recent, specific, and describes a current Discord multi-account startup failure that is not covered by the open send-path SecretRef issue. Current main and v2026.6.6 still rely on the same Discord token/runtime-snapshot and channel restart paths, and I did not find a merged fix or direct canonical duplicate.

## What This Changes

The issue reports that adding a second Discord account with a file-backed SecretRef during hot reload causes the existing default account to enter a permanent restart loop and leaves Discord offline.

## Best Possible Solution

Add a focused regression for Discord hot reload with default plus named SecretRef accounts, fix the runtime snapshot/account-token boundary so each account starts with its own resolved token, then verify with mocked provider identity and live Discord proof if available.

## Reproduction Assessment

No, not yet with high confidence. The issue includes concrete redacted logs and source confirms the hot-reload restart and auto-retry paths, but the exact default-to-second identity cross-resolution still needs a focused fixture or live Discord repro.

## Solution Assessment
