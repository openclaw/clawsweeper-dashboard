## Label Justifications

- `P1`: The report describes a recent supported Discord multi-account regression that can take an existing channel account offline for real users.
- `impact:message-loss`: When the default Discord account stays offline, Discord guild channels remain unresolved and user-visible delivery through that channel can stop.
- `impact:auth-provider`: The suspected failure is SecretRef-backed Discord token resolution against the active runtime snapshot.
- `impact:crash-loop`: The default account repeatedly enters auto-restart attempts after startup throws on `configured_unavailable`.

## Summary

Keep open: current main and the latest release still have the Discord inspect-mode token path that can turn an unresolved account SecretRef into `configured_unavailable`, and Discord startup still throws on that status before the account can connect. The open related send-path issue is adjacent, but it does not cover this default-account startup/hot-reload outage.

## What This Changes

The issue reports that adding a second Discord account with a file-backed SecretRef token can take the existing default Discord account offline in an auto-restart loop.

## Best Possible Solution

Add a focused regression for default-plus-named Discord accounts with file-backed SecretRefs during hot reload, then repair the snapshot/account-token boundary so every enabled account starts from the resolved active runtime config.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection and the reporter's real logs do show the core failure path: unresolved Discord SecretRef input becomes `configured_unavailable`, startup throws, and the gateway schedules repeated account restart attempts.

## Solution Assessment

Unclear whether the reporter's exact materialization location is the best final fix. The maintainable direction is to repair runtime snapshot materialization or Discord account-token selection, not add a new fallback or config workaround.
