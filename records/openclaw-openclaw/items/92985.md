## Label Justifications

- `P1`: The report describes a recent supported Discord multi-account regression that can take an existing channel account offline for real users.
- `impact:message-loss`: When the default Discord account remains offline, guild channels stay unresolved and Discord delivery is unavailable.
- `impact:auth-provider`: The failing surface is SecretRef-backed Discord token resolution against the active runtime snapshot.
- `impact:crash-loop`: The default account repeatedly restarts after startup throws on `configured_unavailable`.

## Summary

Keep open: current main still has a source-reproducible mismatch where Discord’s implicit default account can remain listed while the SecretRef runtime collector treats the top-level default token as inactive once only a second named account is present.

## What This Changes

Fix Discord hot reload so adding a second named account with a SecretRef token does not make the existing default account resolve its token as `configured_unavailable` and restart-loop.

## Best Possible Solution

Materialize the top-level Discord token for the implicit default account when explicit named accounts also exist, then add regression coverage for default-plus-second-account runtime snapshot refresh and startup.

## Reproduction Assessment

Yes, by source inspection: with a top-level Discord SecretRef token and only `accounts.second.token`, account listing still includes the implicit default but the secret collector does not materialize that top-level token. No live Discord run was executed.

## Solution Assessment

Unclear which exact code boundary is best until maintainers choose shared helper versus Discord-specific collector, but the fix should repair runtime-snapshot materialization rather than add a token fallback in the Discord resolver.
