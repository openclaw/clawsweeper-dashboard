## Label Justifications

- `P1`: The report describes a recent Discord multi-account regression that can take an existing channel account offline for real users.
- `impact:message-loss`: When the default Discord account remains offline, Discord guild channels stay unresolved and channel delivery is unavailable.
- `impact:auth-provider`: The failure is tied to SecretRef-backed Discord bot token resolution against the active runtime snapshot.
- `impact:crash-loop`: The default Discord account repeatedly enters gateway auto-restart attempts after startup throws on an unavailable token.

## Summary

Keep open: current main still has a source-level path where Discord lists an implicit default account from the top-level token while SecretRef runtime collection can leave that top-level token unresolved after explicit named accounts are added. The adjacent open send-path issue is related, but it does not fully cover this default-account startup/restart outage.

## What This Changes

A Discord multi-account hot-reload bug report says adding a second SecretRef-backed account can make the existing default account resolve its token as unavailable and enter repeated restart attempts.

## Best Possible Solution

Align the SecretRef runtime snapshot/account surface with Discord's implicit default account so the top-level default token remains resolved when named accounts are added, then cover fresh startup and hot reload with a focused regression.

## Reproduction Assessment

Yes at source level: top-level Discord SecretRef credentials imply a default account, but the SecretRef active-surface collector only sees explicit accounts once `accounts` exists. I did not run a live Discord gateway repro.

## Solution Assessment

Unclear whether the reporter's exact suggested location is the best fix. The maintainable direction is to repair the runtime snapshot/account-boundary invariant, not add a second token source or a per-call fallback.
