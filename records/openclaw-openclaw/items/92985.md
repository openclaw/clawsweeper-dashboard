## Label Justifications

- `P1`: The report describes a real Discord channel outage where the previously working default account can be forced into repeated restarts.
- `impact:message-loss`: When the default Discord account stays offline, inbound or outbound Discord messages for that account cannot be delivered.
- `impact:auth-provider`: The failure centers on Discord account token SecretRef resolution and account selection.
- `impact:crash-loop`: The reported symptom is an auto-restart loop for the default Discord account after hot reload.

## Summary

Keep open: current main and v2026.6.8 still have the Discord inspect-mode token path that can surface unresolved account SecretRefs as `configured_unavailable`, and the related fixes found so far cover adjacent send-path or default-env-fallback cases rather than this default-account hot-reload startup outage.

## What This Changes

The issue reports that adding a second Discord account via hot reload can make the existing default Discord account's file-backed SecretRef token unavailable, leaving the Discord channel in an auto-restart loop.

## Best Possible Solution

Keep this issue open and fix the Discord hot-reload path so each account resolves token SecretRefs from the active runtime snapshot without weakening the fail-closed behavior for genuinely unresolved secrets.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection supports the reported mechanism: unresolved Discord account SecretRefs become `configured_unavailable`, startup throws, and gateway recovery schedules repeated restarts.

## Solution Assessment

Unclear as an implementation choice: the desired behavior is defined, but the safest fix likely belongs at the runtime-snapshot/materialization boundary rather than as an ad hoc env or token fallback in the Discord resolver.
