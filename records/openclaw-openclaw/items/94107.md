## Summary

Keep open. The latest PR head still has a blocking explicit Telegram delivery bug: the new reserved-literal error can be discarded before Telegram’s session-route parser turns raw `current` into `telegram:current`, so the fail-closed path is not reliable and the supplied proof is not enough to clear the change.

## What This Changes

The PR changes Telegram/outbound target resolution to preserve directory matches for reserved words, reject reserved literal misses, propagate target-resolution errors through gateway delivery, add an optional channel-plugin SDK field, and document/test the behavior.

## Best Possible Solution

Use one directory-capable explicit delivery path: resolve configured directory hits before session-route parsing, reject reserved literals only after a directory miss, and land the `reservedLiterals` SDK seam only after maintainer acceptance of the compatibility contract.

## Reproduction Assessment

Yes, from source inspection. Current main lacks the reserved-target guard, and the PR head still has a source-reproducible explicit delivery path where raw `current` can be parsed into `telegram:current` instead of failing closed after a directory miss.

## Solution Assessment

No. The directory-first resolver is the right direction, but explicit gateway/agent delivery needs to use that resolver before Telegram session-route parsing, and the SDK/compatibility policy needs maintainer acceptance.
