## Summary

Keep this PR open: the directory-first resolver/gateway direction is useful, but the patch still lets heartbeat/check-in Telegram delivery continue after a reserved-literal directory miss, leaving a reported wrong-recipient path unfixed.

## What This Changes

The PR adds a channel plugin `reservedLiterals` target-resolver contract, marks Telegram `current`/`self`/`this`/`me` as reserved, changes resolver and gateway delivery planning around reserved target errors, and updates focused tests/docs.

## Best Possible Solution

Keep the directory-first resolver design, but make every Telegram delivery mode that can send messages, especially heartbeat/check-ins, fail closed on reserved-literal directory misses before session-route/default fallback; then add regression proof and get maintainer acceptance for the SDK field.

## Reproduction Assessment

Yes, source-reproducible: current main lacks reserved-literal protection, and PR head still lets heartbeat/check-in delivery ignore a reserved directory miss and route raw `current`. I did not run a live Telegram send in this read-only review.

## Solution Assessment

No, not yet. Directory-first reserved-target resolution is the right owner-boundary shape, but the heartbeat exemption needs to carry reserved miss errors through instead of allowing raw target fallback.
