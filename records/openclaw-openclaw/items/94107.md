## What This Changes

Adds an optional channel plugin `reservedLiterals` target-resolver contract, configures Telegram to reserve `current`, `self`, `this`, and `me`, rejects those literals in shared outbound resolution, propagates errors through agent delivery, and updates docs/tests.

## Best Possible Solution

Land the plugin-owned fail-closed resolver fix only after maintainer API signoff and current-head real gateway or Telegram proof, then let the linked canonical issue close through the merged PR.

## Reproduction Assessment

Yes by source inspection: current main and v2026.6.8 have no reserved-literal guard, and Telegram normalization treats username-shaped bare strings such as `current` as handle-like targets. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Mostly yes: prior maintainer feedback rejected a global guard, and this PR moves the rule to a plugin-owned boundary that covers both directory and direct fallback paths. The remaining blocker is maintainer acceptance of the new SDK field plus real behavior proof.
