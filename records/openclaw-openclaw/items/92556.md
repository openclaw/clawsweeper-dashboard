Full review comments:

- **[P1] Preserve Basic auth for Inworld requests:** `extensions/inworld/index.ts:81-85`
  - body: This catalog returns an `openai-completions` provider with only the raw Inworld key, so the generic OpenAI-compatible transport will send `Authorization: Bearer ...`. Inworld's chat-completions docs and the existing TTS code require `Authorization: Basic ...`, and the added live test bypasses OpenClaw by calling `fetch` with Basic directly, so an onboarded OpenClaw model request can fail auth.
  - confidence: 0.94
- **[P2] Read Inworld's documented pricing fields:** `extensions/inworld/models.ts:135-136`
  - body: Official Inworld list-models docs show `pricing.prompt` and `pricing.completion`, but this parser and its tests only read `promptToken` and `completionToken`. Documented live catalog rows would therefore get zero input/output cost metadata.
  - confidence: 0.9
- **[P2] Use a supported endpoint class for Inworld routing:** `extensions/inworld/openclaw.plugin.json:19-20`
  - body: Current core only accepts the closed endpoint classes in `src/agents/provider-attribution.ts`; unknown classes are skipped. Declaring `endpointClass: "inworld"` means this metadata is ignored and `api.inworld.ai` falls back to custom routing instead of the intended known endpoint classification.
  - confidence: 0.88
