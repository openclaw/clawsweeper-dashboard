## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Send Inworld LLM credentials with Basic auth:** `extensions/inworld/index.ts:81-85`
  - body: `catalog.run` returns the Inworld key as a generic OpenAI SDK `apiKey`, but OpenAI Node emits `Authorization: Bearer <apiKey>` from that field while Inworld documents `Authorization: Basic <apiKey>`. The PR's live proof bypasses this by calling fetch/curl with Basic auth directly, so configured OpenClaw LLM requests will authenticate differently from the proof.
  - confidence: 0.92
- **[P2] Read Inworld's documented pricing fields:** `extensions/inworld/models.ts:135-136`
  - body: The live catalog parser reads `pricing.promptToken` and `pricing.completionToken`, but Inworld's current list-models contract documents `pricing.prompt` and `pricing.completion`. As written, live catalog rows silently get zero input/output cost metadata even when Inworld returns pricing.
  - confidence: 0.91
- **[P2] Use a supported endpoint class for Inworld routing:** `extensions/inworld/openclaw.plugin.json:19-20`
  - body: The manifest declares `endpointClass: "inworld"`, but core only accepts endpoint classes in `MANIFEST_PROVIDER_ENDPOINT_CLASSES` and skips unknown rows. That leaves `api.inworld.ai` attributed as a custom/proxy-like endpoint instead of the intended known provider endpoint, which changes OpenAI-compatible request defaults.
  - confidence: 0.9
- **[P2] Bound the live catalog response before parsing JSON:** `extensions/inworld/models.ts:176`
  - body: `discoverInworldModels` buffers the authenticated provider catalog with `response.json()`. Live model catalogs are provider-controlled; the repository's shared catalog helper caps response bodies with `readResponseWithLimit`, so this path should use the bounded helper or an equivalent cap before parsing.
  - confidence: 0.84
