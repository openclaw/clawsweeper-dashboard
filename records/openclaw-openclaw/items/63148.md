## Summary

Keep open: current main still has the Feishu DM-topic session gap and this PR is the active implementation path, but the latest branch still has blocking code issues and lacks inspectable real Feishu proof.

## What This Changes

The PR scopes Feishu DM messages with root_id/thread_id to topic-specific peers, threaded replies, parent routing, per-topic sequential/debounce keys, and focused tests.

## Best Possible Solution

Land or replace a focused Feishu plugin change that isolates topic-bearing DMs, preserves group/topic_group queue semantics, uses a DM-topic-safe delivery/parser path, and includes redacted real Feishu proof.

## Reproduction Assessment

Yes, source-level: current main parses Feishu root_id/thread_id into the message context, but direct messages still route to ctx.senderOpenId with parentPeer null, replyInThread false, and a chat-scoped sequential key.

## Solution Assessment

No, not yet. The Feishu plugin is the right surface, but the branch should preserve topic_group group semantics, fix outbound delivery for DM-topic conversation IDs, and settle the default-on versus opt-in product shape before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Treat topic_group chats as group queues:** `extensions/feishu/src/sequential-key.ts:19-21`
  - body: The new queue branch only treats `chat_type === "group"` as grouped. Current Feishu code classifies `topic_group` as a group chat, so topic_group events with `root_id` or `thread_id` now take the DM-style `:topic:` queue key despite the PR's group-unchanged contract. Use the shared classifier or include `topic_group`, with coverage.
  - confidence: 0.9
- **[P2] Route DM-topic IDs before direct user IDs:** `extensions/feishu/src/bot.ts:711-715`
  - body: This encodes DM topics as `ou_*:topic:<id>`, but Feishu delivery resolves direct IDs before parsing topic IDs and accepts any `ou_` prefix. Sends or subagent replies addressed to the topic session can therefore target `user:ou_*:topic:<id>` instead of the original chat/thread. Add a DM-topic-safe encoding or parser path with tests.
  - confidence: 0.84
