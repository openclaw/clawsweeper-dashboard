## Summary

Keep open: current main still has the Feishu DM topic isolation gap and this PR is the focused implementation candidate, but the patch needs a small correctness fix for `topic_group` queue classification plus real behavior proof before merge.

## What This Changes

The PR changes Feishu DM topic handling to derive topic-scoped direct peers, thread replies, parent routing, per-topic sequential keys, and topic-aware debounce behavior with regression tests.

## Best Possible Solution

Land or replace a Feishu plugin change that isolates topic-bearing DMs while preserving group queue behavior, after maintainer confirmation on default-on versus opt-in behavior, real Feishu proof, and green relevant checks.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction path for the reported gap: Feishu p2p events can carry `root_id`/`thread_id`, but direct routing and queue keys stay flat; the PR-introduced `topic_group` queue regression is also source-reproducible from the changed check and the existing chat-type helper.

## Solution Assessment

Not quite yet. The approach is appropriately scoped to the Feishu plugin, but the sequential-key change needs to treat `topic_group` as group, and merge still needs real behavior proof plus maintainer confirmation on the product shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Treat topic_group chats as group queues:** `extensions/feishu/src/sequential-key.ts:19`
  - body: Feishu also emits `chat_type: "topic_group"`, and the bot's shared predicate treats that as a group chat. This new check only excludes `"group"`, so a topic-group event with `root_id` or `thread_id` now gets a DM-style `:topic:` sequential key and can run in parallel, changing group behavior despite the PR's group-unchanged contract. Use the shared group-chat predicate or explicitly include `topic_group`, and add coverage for it.
  - confidence: 0.9
