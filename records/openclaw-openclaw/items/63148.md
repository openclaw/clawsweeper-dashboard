## Summary

Keep open. The PR targets a real Feishu DM-topic isolation gap in the right plugin surface, but it still has source-reproducible regressions around Feishu topic-group queues and DM-topic conversation IDs, and it lacks inspectable real Feishu behavior proof.

## What This Changes

The branch scopes Feishu DM messages with root_id/thread_id to topic-specific peers, thread replies, parent routing, thread-aware debounce keys, and per-topic sequential queue keys with tests.

## Best Possible Solution

Land or replace a Feishu plugin change that isolates topic-bearing DMs while preserving group/topic_group queue semantics, handling outbound delivery for DM-topic conversation IDs, and documenting the chosen default/config behavior.

## Reproduction Assessment

Yes, source-level. Current main routes Feishu p2p root/thread events from the same sender through the same direct peer and chat-scoped queue, and the PR regressions are visible from the changed branch checks plus current parser/delivery contracts.

## Solution Assessment

No, not yet. The Feishu plugin is the right surface, but the patch should reuse the shared group-chat predicate, add a DM-topic-safe delivery/parser path, provide real Feishu proof, and get maintainer agreement on default-on versus opt-in behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Treat topic_group chats as group queues:** `extensions/feishu/src/sequential-key.ts:19-21`
  - body: Feishu topic groups use chat_type "topic_group", and the shared helper classifies that as a group chat. This literal check only excludes "group", so a topic_group event with root_id or thread_id now gets a DM-style :topic: queue key and can run in parallel, changing group behavior despite the PR's group-unchanged contract. Use the shared predicate or include topic_group, with coverage.
  - confidence: 0.9
- **[P2] Add a DM-topic-safe delivery path:** `extensions/feishu/src/bot.ts:713-717`
  - body: The new DM topic peer is shaped as ou_*:topic:<id>, but outbound delivery still checks the direct-id parser before the topic parser and accepts any ou_ prefix as a direct user id. Sends or subagent replies addressed to the current topic session can therefore skip the threaded route or target an invalid user id. Add a DM-topic-specific encoding/parser path and tests before exposing these session ids.
  - confidence: 0.74
