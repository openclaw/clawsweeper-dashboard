## Summary

Keep open: the Feishu DM-topic feature is useful, but this branch is not merge-ready because it is default-on for existing users, has concrete delivery and topic_group queue regressions, still lacks inspectable real Feishu proof, and is currently conflicting with failing checks.

## What This Changes

The branch scopes Feishu DM messages with root_id/thread_id into topic-specific session IDs, threaded replies, parent routing, per-topic sequential/debounce keys, and adds focused Feishu tests.

## Best Possible Solution

Add an explicit Feishu DM topic/session option that preserves flat DM sessions by default, model DM topic delivery with a valid parent/target contract, keep topic_group group semantics, then land with live proof and focused tests.

## Reproduction Assessment

Yes, at source level: current main parses Feishu root_id/thread_id but direct messages still route to ctx.senderOpenId with replyInThread false and a chat-scoped sequential key. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No. The implementation proves the rough direction, but the safer maintainable path is opt-in DM-topic scoping plus an explicit delivery-target contract and topic_group-safe queue classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep DM topic scoping behind an explicit setting:** `extensions/feishu/src/bot.ts:707-724`
  - body: The linked request asked for optional DM topic scoping with current flat per-sender DMs preserved by default, but this code enables topic sessions and threaded replies for every DM carrying root_id/thread_id. Existing users who already used Feishu DM topics would silently move to new sessions and lose flat-session continuity, so this needs a config-gated default plus tests for both modes.
  - confidence: 0.92
- **[P1] Preserve a valid delivery target for DM topic sessions:** `extensions/feishu/src/bot.ts:713-716`
  - body: Creating a direct peer id like `ou_x:topic:<id>` without also giving the delivery layer a real parent chat target makes later sends fragile: `resolveDeliveryTarget` currently checks direct IDs before parsing topic ids, so this shape can become `user:ou_x:topic:<id>` instead of a threaded chat reply. The DM topic model needs a delivery contract that carries the real chat/user target and thread id separately.
  - confidence: 0.88
- **[P2] Treat topic_group as a group queue:** `extensions/feishu/src/sequential-key.ts:19`
  - body: Current Feishu code treats `topic_group` as a group chat, but this new queue classifier only excludes `chat_type === "group"`; a `topic_group` event with root_id/thread_id will get the new DM-style topic key. That changes group-topic serialization even though the PR promises group behavior is untouched, so include `topic_group` in the group check or reuse `isFeishuGroupChatType`.
  - confidence: 0.9
