## Summary

Keep open. Current main still lacks Feishu DM topic isolation and this PR is a useful implementation candidate, but the latest head still has a P2 `topic_group` queue regression and lacks contributor-supplied real Feishu proof.

## What This Changes

The PR changes Feishu direct-message topic handling to use topic-scoped peers, thread replies, parent routing, thread-aware debounce keys, and per-topic sequential queue keys.

## Best Possible Solution

Land or replace a Feishu plugin change that isolates topic-bearing DMs while preserving existing group and topic-group queue behavior, with focused regression tests, real Feishu proof, and maintainer agreement on default-on versus opt-in behavior.

## Reproduction Assessment

Yes. Source inspection shows current main parses Feishu `root_id` and `thread_id`, but direct-message routing, thread-reply flags, debounce, and sequential keys remain flat for the sender/chat; the PR-introduced `topic_group` queue regression is also source-reproducible from the changed check and existing chat-type helper.

## Solution Assessment

Not yet. The PR is in the right Feishu plugin surface, but the sequential-key change should use the shared group-chat predicate or explicitly include `topic_group`, and merge still needs real behavior proof plus maintainer confirmation on the product shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Treat topic_group chats as group queues:** `extensions/feishu/src/sequential-key.ts:19-21`
  - body: Feishu topic groups use `chat_type: "topic_group"`, and the plugin's shared helper classifies that as a group chat. This new check only excludes literal `"group"`, so a topic-group event with `root_id` or `thread_id` now gets a DM-style `:topic:` sequential key and can run in parallel, changing group behavior despite the PR's group-unchanged contract. Use the shared group-chat predicate or include `topic_group`, and add coverage for it.
  - confidence: 0.9
