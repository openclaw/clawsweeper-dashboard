## Summary

Keep open: the Feishu DM-topic direction is useful and source-reproducible, but this branch still needs a product decision, real Feishu proof, and fixes for default-on session changes, delivery targeting, and topic-group queue classification before merge.

## What This Changes

The branch changes Feishu DM root_id/thread_id handling to create topic-scoped sessions, threaded replies, parent routing, per-topic sequential/debounce keys, and focused tests.

## Best Possible Solution

Add an explicit Feishu DM topic/session option that preserves flat DM sessions by default, model DM-topic delivery with a valid chat/thread target contract, keep `topic_group` group semantics, and land with live proof plus focused upgrade tests.

## Reproduction Assessment

Yes, at source level: current main and `v2026.5.19` parse Feishu root_id/thread_id but direct messages still route to `ctx.senderOpenId` with `replyInThread` false and a chat-scoped sequential key. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No. The branch proves the rough direction, but the maintainable solution is opt-in DM-topic scoping plus an explicit delivery-target contract and topic_group-safe queue classification.

Full review comments:

- **[P1] Keep DM topic scoping behind an explicit setting:** `extensions/feishu/src/bot.ts:707-724`
  - body: The linked request asked for optional DM topic scoping with current flat per-sender DMs preserved by default, but this code enables topic sessions and threaded replies for every DM carrying root_id/thread_id. Existing users who already use Feishu DM topics would silently move to new sessions and lose flat-session continuity, so this needs a config-gated default plus tests for both modes.
  - confidence: 0.92
- **[P1] Preserve a valid delivery target for DM topic sessions:** `extensions/feishu/src/bot.ts:713-716`
  - body: Creating a direct peer id like `ou_x:topic:<id>` without a real parent chat/thread delivery contract makes later sends fragile: `resolveDeliveryTarget` checks direct IDs before parsing topic ids, so this shape can become `user:ou_x:topic:<id>` instead of a threaded chat reply. Carry the real Feishu target and topic id separately before enabling this session shape.
  - confidence: 0.88
- **[P2] Treat topic_group as a group queue:** `extensions/feishu/src/sequential-key.ts:19-25`
  - body: Current Feishu code treats `topic_group` as a group chat, but this new queue classifier only excludes `chat_type === "group"`; a `topic_group` event with root_id/thread_id will get the new DM-style topic key. That changes group-topic serialization even though the PR promises group behavior is untouched, so include `topic_group` in the group check or reuse `isFeishuGroupChatType`.
  - confidence: 0.9
