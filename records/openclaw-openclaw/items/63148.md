## Summary

Keep open: current main still lacks Feishu DM topic isolation, and this PR remains a focused implementation candidate, but the current head still changes topic-group queue behavior and lacks contributor-supplied real Feishu proof.

## What This Changes

The PR changes Feishu DM topic handling to use topic-scoped direct peers, thread replies, parent routing, per-topic sequential keys, topic-aware debounce keys, and regression tests.

## Best Possible Solution

Land or replace a Feishu plugin change that isolates topic-bearing DMs while preserving group and topic_group queue behavior, after maintainer product review and real Feishu proof.

## Reproduction Assessment

Yes, source-level. Current main carries root_id/thread_id into the Feishu context, but direct routing still uses ctx.senderOpenId and the sequential key is account/chat scoped, so different DM topics from the same sender share routing and queue state.

## Solution Assessment

Not yet. The plugin-local direction is appropriate, but the PR needs to preserve topic_group queue semantics, provide real Feishu proof, and get maintainer confirmation on default-on versus opt-in behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve group queueing for topic_group chats:** `extensions/feishu/src/sequential-key.ts:19`
  - body: Feishu also emits `chat_type: "topic_group"`, and the shared predicate treats that as a group chat. This new check only excludes `"group"`, so a topic-group event with `root_id` or `thread_id` now gets a DM-style `:topic:` sequential key and can run in parallel, changing group behavior despite the PR's group-unchanged contract. Use the shared group-chat predicate or explicitly include `topic_group`, and add coverage for it.
  - confidence: 0.9
