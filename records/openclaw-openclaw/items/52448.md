## Summary

Keep open. The PR addresses a real sandbox context gap, but it changes a security boundary and still copies allowlisted symlink targets without a trusted-root policy or hardlink rejection, which conflicts with current main's restrictive seed contract.

## What This Changes

The PR adds sandbox workspace hydration for allowlisted top-level authority symlinks such as `README.md`, `CLAUDE.md`, `docs`, and `voro-docs`, plus unit tests for materialization and nested symlink skipping.

## Best Possible Solution

Define a documented sandbox authority-link model before merging: supported names, trusted resolved-target roots, symlink and hardlink rejection throughout copied trees, refresh behavior for reused sandboxes, Windows-safe tests, and docs for hydrated authority files.

## Reproduction Assessment

Yes. At PR head, point an allowlisted `README.md` or `docs` symlink at an outside path, or place a same-filesystem hardlink inside an allowed authority tree, then call `ensureSandboxWorkspace`; code inspection shows the copied sandbox will contain content the current seed path rejects.

## Solution Assessment

No. Copying resolved symlink targets with a custom recursive copier is not the narrowest maintainable solution for this sandbox boundary; it should reuse boundary validation or add an explicit maintainer-approved authority-root policy with hardlink checks.

  - body: `fs.realpath(src)` resolves an allowlisted name and then copies that resolved path without checking it is inside the seed workspace or another trusted authority root. A workspace can point `README.md` or `docs` at private host content and have it materialized into a non-rw sandbox.

  - confidence: 0.9

  - body: The copy helper returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later changes to authority docs or edits left inside the sandbox are not refreshed for later sessions.
  - confidence: 0.86
