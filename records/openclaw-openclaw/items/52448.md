## Summary

Keep open. Current main does not implement this feature, and the proposed branch still changes a sandbox trust boundary with unresolved security/copy-semantics findings plus missing real behavior proof.

## What This Changes

The PR adds sandbox workspace hydration for allowlisted top-level authority symlinks such as `README.md`, `CLAUDE.md`, `docs`, and `voro-docs`, with unit tests for materialization, allowlisting, and nested symlink skipping.

## Best Possible Solution

Define the sandbox authority-link contract first, then implement it with explicit supported names, resolved-target validation, symlink and hardlink rejection throughout copied trees, deliberate refresh semantics, focused tests, and matching docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review findings: source inspection of the PR head and generated merge result shows the realpath copy path, hardlink copy path, and existing-destination early return; the original request is a new capability rather than a current-main bug reproduction.

## Solution Assessment

Is this the best way to solve the issue? No. Hydrating authority links may be useful, but this patch copies resolved targets before the sandbox trust root, alias rejection, and refresh contract are defined and enforced.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority targets before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted name and then copies that resolved path without checking it is inside the seed workspace or another explicitly trusted root. A workspace can point `README.md` or `docs` at private host content and have it materialized into a non-`rw` sandbox.
  - confidence: 0.93
- **[P1] Reject hardlinks while copying authority trees:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `lstat()` plus `copyFile()` treats hardlinked files as ordinary files. Current bootstrap seeding rejects hardlink aliases through the boundary-file path, but this path would copy a same-filesystem hardlink inside an allowed authority tree into the sandbox as readable content.
  - confidence: 0.9
- **[P2] Refresh existing authority copies:** `src/agents/sandbox/workspace.ts:40-42`
  - body: `copyAuthorityTreeIfMissing` returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later source changes or previous-session edits in copied authority files can leave later sessions reading stale authority content.
  - confidence: 0.86
