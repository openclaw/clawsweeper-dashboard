## Summary

Keep open: current main does not implement this sandbox authority-link hydration, but the PR is not merge-ready because it broadens sandbox host-content exposure without a defined trust-root contract, still has source-level security/correctness findings, and lacks after-fix real behavior proof.

## What This Changes

The PR adds sandbox workspace hydration for allowlisted top-level symlinked authority entries such as `README.md`, `CLAUDE.md`, `docs`, and `voro-docs`, plus focused unit tests for materialization and exclusions.

## Best Possible Solution

Define the sandbox authority-link contract first, then implement explicit target-root validation, symlink and hardlink rejection through copied trees, deliberate refresh semantics for reused sandboxes, docs, focused tests, and redacted real sandbox proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows the unvalidated realpath copy, hardlink-copy path, and existing-destination early return. The original request is a new sandbox capability rather than a current-main bug reproduction.

## Solution Assessment

No, not as-is. Hydrating authority links may be useful, but this patch needs a documented trust model and enforcement before copying external host content into a sandbox.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority link targets before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted workspace symlink and then copies that resolved path without checking it is inside the seed workspace or another documented trusted root. A workspace can point `README.md` or `docs` at private host content and have it materialized into a non-`rw` sandbox.
  - confidence: 0.93
- **[P1] Reject hardlinks while copying authority trees:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `lstat()` plus `copyFile()` treats hardlinked files as ordinary files. Current bootstrap seeding rejects hardlink aliases through the root-file path, but this new path would copy a same-filesystem hardlink inside an allowed authority tree into the sandbox as readable content.
  - confidence: 0.9
- **[P2] Refresh existing authority copies:** `src/agents/sandbox/workspace.ts:40-42`
  - body: `copyAuthorityTreeIfMissing` returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later source changes or previous-session edits in copied authority files can leave later sessions reading stale authority content.
  - confidence: 0.86
- **[P2] Guard symlink tests on Windows:** `src/agents/sandbox.workspace.test.ts:25-34`
  - body: The new tests create file and directory symlinks unconditionally, while the existing sandbox symlink and hardlink tests are skipped on Windows because runners may lack symlink privileges. This can fail the Windows lane before reaching the sandbox assertions.
  - confidence: 0.84
