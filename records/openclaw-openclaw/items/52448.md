## Summary

Keep open: the PR is a plausible feature but still changes a sandbox trust boundary, and the current head leaves unresolved target-validation, hardlink-alias, refresh-semantics, and real-behavior-proof blockers before merge.

## What This Changes

The PR materializes allowlisted top-level workspace authority symlinks into sandbox workspaces as real copied files/directories and adds unit coverage for materialization, allowlisting, and nested symlink skipping.

## Best Possible Solution

Define a documented authority-link model first, then implement it with explicit supported names, resolved-target validation, hardlink rejection, deliberate refresh semantics for reused sandboxes, focused tests, and matching sandbox docs.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows the allowlisted symlink and hardlink copy paths. The original request is a new capability, so there is no current-main bug reproduction to run in this read-only review.

## Solution Assessment

No. Hydrating explicit authority links may be useful, but the current patch copies resolved targets before the sandbox trust boundary and refresh behavior are defined and enforced.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority targets before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted name and then copies that target without checking it is inside the seed workspace or another explicitly trusted root. A workspace can point `README.md` or `docs` at private host content and have it materialized into a non-rw sandbox, bypassing the current seed contract.
  - confidence: 0.93
- **[P1] Reject hardlinks while copying authority trees:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `lstat()` plus `copyFile()` treats hardlinked files as ordinary files. Current bootstrap seeding rejects hardlink aliases through the boundary-file path, but this new copier would copy a same-filesystem hardlink inside an allowed authority tree into the sandbox as readable content.
  - confidence: 0.9
- **[P2] Refresh existing authority copies:** `src/agents/sandbox/workspace.ts:40-42`
  - body: `copyAuthorityTreeIfMissing` returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later changes to the source authority docs or previous-session edits in the copied sandbox files can leave later sessions reading stale authority content.
  - confidence: 0.86
