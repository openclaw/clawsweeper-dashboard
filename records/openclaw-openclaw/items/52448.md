## Summary

Keep this PR open for maintainer/security review rather than closing it as obsolete. Current main still does not materialize README/docs authority symlinks into sandbox workspaces, but the proposed patch changes the sandbox filesystem trust boundary and has confirmed blocking issues around outside-root symlink targets and hardlinked files.

## What This Changes

The PR adds sandbox workspace hydration for allowlisted top-level authority symlinks such as `README.md`, `CLAUDE.md`, `docs`, and `voro-docs`, plus unit coverage for materialization, allowlisting, nested symlink skipping, and non-symlink exclusion.

## Best Possible Solution

Define and document an authority-link model before merging: supported names, allowed roots, refresh semantics for reused sandboxes, symlink and hardlink rejection throughout copied trees, Windows-safe tests, and docs that explain exactly which authority files can be hydrated.

## Reproduction Assessment

Yes. At PR head, point an allowlisted `README.md` or `docs` symlink at an outside path, or place a same-filesystem hardlink inside an allowed authority tree; the new copy path materializes content that the current `openBoundaryFile` seed contract rejects.

## Solution Assessment

No. Copying resolved symlink targets directly is not the safest maintainable solution for this sandbox boundary; a safer approach should reuse boundary validation or implement an explicitly approved authority-root policy with hardlink checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority links before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted name and then copies that resolved path without checking it is inside the seed workspace or another explicitly trusted root. A workspace can point `README.md` or `docs` at a credential directory or sibling checkout and have that content materialized into a `workspaceAccess: "none"`/`"ro"` sandbox, bypassing the current seed contract.
  - confidence: 0.93
- **[P1] Reject hardlinks while copying authority trees:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `lstat()` plus `copyFile()` treats hardlinked files as ordinary files. Current bootstrap seeding rejects hardlink aliases through `openBoundaryFile`, but this path would copy a same-filesystem hardlink inside an allowed authority tree into the sandbox as readable content.
  - confidence: 0.9
- **[P2] Refresh existing authority copies:** `src/agents/sandbox/workspace.ts:40-42`
  - body: `copyAuthorityTreeIfMissing` returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later changes to `README.md` or `docs` symlink targets, or edits left in copied sandbox files, will not be refreshed and later sessions can read stale authority content.
  - confidence: 0.86
- **[P2] Guard symlink tests on Windows:** `src/agents/sandbox.workspace.test.ts:25-34`
  - body: The new tests create file and directory symlinks unconditionally. Existing sandbox symlink and hardlink tests are skipped on `win32`, so this added file can fail Windows lanes before reaching the sandbox assertions.
  - confidence: 0.88
