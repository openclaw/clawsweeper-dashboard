## Summary

Keep open: the PR addresses a real local-first context gap, but the current branch changes a sandbox filesystem boundary and still has concrete security and correctness blockers before it can merge.

## What This Changes

The PR adds sandbox workspace hydration for allowlisted top-level authority symlinks such as `README.md`, `CLAUDE.md`, `docs`, and `voro-docs`, plus unit tests for materialization, allowlisting, and nested symlink skipping.

## Best Possible Solution

Define and document a sandbox authority-link model first, then implement hydration through the existing boundary-safe validation path with explicit root policy, hardlink rejection, refresh semantics, and Windows-safe tests.

## Reproduction Assessment

Yes, source-reproducible. At PR head, an allowlisted `README.md` or `docs` symlink can point outside the seed workspace, or an allowed authority tree can contain a hardlinked file; `ensureSandboxWorkspace` will copy that content into the sandbox.

## Solution Assessment

No. The current PR is not the best fix because it introduces a custom realpath-copy path around the existing sandbox boundary checks; a safer solution should reuse or extend boundary validation under a maintainer-approved authority-root policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority links before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted name and then copies that resolved path without checking it is inside the seed workspace or another explicitly trusted root. In `workspaceAccess: "none"` or `"ro"`, a workspace entry like `README.md -> ~/.ssh/config` or `docs -> ../private-repo` can be materialized under `/workspace`, bypassing the current seed contract.
  - confidence: 0.93
- **[P1] Reject hardlinks while copying authority trees:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `lstat()` plus `copyFile()` treats hardlinked files as ordinary files. Current bootstrap seeding rejects hardlink aliases through `openBoundaryFile`, but this path would copy a same-filesystem hardlink inside an allowed authority tree into the sandbox as readable content.
  - confidence: 0.9
- **[P2] Refresh existing authority copies:** `src/agents/sandbox/workspace.ts:40-42`
  - body: `copyAuthorityTreeIfMissing` returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later changes to `README.md` or `docs` symlink targets, or edits left in the copied sandbox files, will not be refreshed for later sessions.
  - confidence: 0.86
- **[P2] Guard new symlink tests on Windows:** `src/agents/sandbox.workspace.test.ts:25-34`
  - body: The new test cases create file and directory symlinks unconditionally, while existing sandbox symlink/hardlink tests are skipped on Windows because runners may lack symlink privileges. This can fail the Windows lane before reaching the sandbox assertions.
  - confidence: 0.88
