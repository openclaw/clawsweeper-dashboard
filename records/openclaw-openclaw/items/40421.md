## Summary

Keep open. The PR contains useful source work, but it is not mergeable against current main because it targets an obsolete Telegram path, still has concrete lifecycle/config bugs, and overlaps the open product decision in #6946 about the desired Telegram waiting-feedback UX.

## What This Changes

The PR adds a waiting-tip controller and tests, `messages.waitingTips` config metadata, and Telegram dispatch wiring to send and delete bilingual tip messages while replies are being generated.

## Best Possible Solution

Decide the Telegram waiting-feedback UX against #6946, then implement the accepted behavior in `extensions/telegram` using a plugin-local helper or a deliberately exposed SDK seam, with race-safe send/delete handling, honest config metadata, focused regression coverage, and docs/changelog if user-facing configuration ships.

## Reproduction Assessment

Yes. The integration mismatch is reproducible by checking that current main has no `src/telegram` directory and that the active dispatcher is `extensions/telegram/src/bot-message-dispatch.ts`; the cleanup race follows from calling `cleanup()` while `sendTip` is already in flight.

## Solution Assessment

No. This PR is useful as source material, but the current patch is not the best merge path because it targets an obsolete path, can leave tip messages behind, and exposes incomplete custom-tip config; a maintainer-approved Telegram plugin implementation is safer.
