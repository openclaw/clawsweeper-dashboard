## Summary

Keep open for maintainer decision: the feature is not on main, but the branch is currently stale/incorrect because it targets the removed Telegram path, has lifecycle/config bugs, lacks real Telegram proof, and overlaps the open processing-indicator issue.

## What This Changes

The PR adds a configurable waiting-tip controller, shared message config/schema/help/labels, tests, and Telegram dispatch wiring for temporary bilingual tip messages while replies are pending.

## Best Possible Solution

Choose one opt-in Telegram waiting UX, then implement it inside extensions/telegram using existing delivery/thread helpers with only implemented config, focused lifecycle tests, docs/changelog, and real Telegram proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows stale Telegram wiring, a cleanup race, an exposed-but-unimplemented customTipsPath, and bypassed thread normalization. As a user request, this is a feature rather than a bug reproduction.

## Solution Assessment

No. This PR is not the best current solution because it targets stale structure, bypasses current Telegram delivery semantics, and broadens the UX/config surface before the related product decision is settled.

  - confidence: 0.91

  - body: The tip sender calls bot.api.sendMessage with a raw message_thread_id, bypassing the current helper behavior that omits General-topic IDs and preserves Telegram topic semantics. Waiting tips can fail in chats where normal replies already work.
  - confidence: 0.86
