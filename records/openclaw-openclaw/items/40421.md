## Summary

Keep open: current main does not implement waiting tips, and this PR remains useful source work, but it is stale against the plugin-owned Telegram dispatch and has correctness/config issues that block merge.

## What This Changes

The branch adds a core waiting-tip controller with hard-coded bilingual tips, `messages.waitingTips` config metadata/schema, and Telegram dispatch code to send, cancel, and delete a temporary tip while the model replies.

## Best Possible Solution

Settle the UX against #6946, then implement the accepted waiting/processing indicator inside the current Telegram plugin flow using existing delivery/thread helpers, exposing only implemented config and adding focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Static comparison against current main reproduces the stale path problem, and the cleanup race is reproducible by letting the timer fire, keeping `sendTip` in flight, then calling `cleanup()` before the send resolves.

## Solution Assessment

No. The proposed patch is not the best current solution because it targets a removed path, bypasses current Telegram delivery/thread helpers, and exposes unimplemented config; a rebased plugin-owned implementation would be safer.

Full review comments:

- **[P1] Rebase the Telegram wiring onto the plugin dispatch:** `src/telegram/bot-message-dispatch.ts:14`
  - body: This import wires waiting tips into `src/telegram/bot-message-dispatch.ts`, but current main handles Telegram replies under `extensions/telegram/src/bot-message-dispatch.ts`. As written, the PR is stale against the current runtime path and must be rebased into the plugin dispatch before it can work.
  - confidence: 0.93
- **[P2] Preserve sent tip handles when cleanup races send:** `src/channels/waiting-tips.ts:214-216`
  - body: `cleanup()` calls `cancel()` before awaiting the in-flight send, so a tip that finishes sending after cancellation skips handle assignment and cannot be deleted. Record the returned message id even after cancellation, then let `deleteAfterReply` decide whether cleanup removes it.
  - confidence: 0.94
- **[P2] Wire custom tips before exposing the setting:** `src/config/schema.help.ts:1450-1451`
  - body: The new help/schema/type surface advertises `messages.waitingTips.customTipsPath`, but the PR never reads that directory or passes loaded tips to the controller. Operators can set the option and get no behavior change, so either implement loading or remove the setting until it works.
  - confidence: 0.92
- **[P2] Use Telegram thread-aware send helpers for tips:** `src/telegram/bot-message-dispatch.ts:501-504`
  - body: The tip send passes `message_thread_id` directly whenever `threadSpec.id` is present. Current Telegram delivery omits General-topic ids and retries DM thread failures; bypassing those helpers can make waiting tips fail in threads where normal replies work.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/config/types.messages.ts:130-131`
  - body: This is a user-facing Telegram feature with a new admin config surface, but the PR does not update `CHANGELOG.md`. Add a single-line entry in the active release section before merge.
  - confidence: 0.86
