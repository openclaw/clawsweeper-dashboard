## Summary

Keep open, but the PR is not mergeable as-is: it targets the removed core Telegram path, has source-reproducible lifecycle bugs, exposes unimplemented config, and overlaps an open product/config discussion for Telegram processing indicators.

## What This Changes

This PR adds a configurable bilingual waiting-tip controller, message config schema/help/labels, tests, and Telegram dispatch wiring to send and delete temporary tips while replies are pending.

## Best Possible Solution

Choose the intended Telegram waiting UX in the related issue, then implement it inside extensions/telegram with existing delivery helpers, only implemented config, focused lifecycle tests, and docs/changelog if accepted.

## Reproduction Assessment

Yes for the PR blockers: source inspection shows the branch wires a removed src/telegram path, can lose the waiting-tip handle during cleanup, and exposes customTipsPath without runtime loading. As a user request, this is a feature rather than a bug reproduction.

## Solution Assessment

No. This PR is not the best current solution because it targets stale structure, bypasses current Telegram delivery semantics, and broadens the UX/config surface before the related product decision is settled.

Full review comments:

- **[P1] Rebase the wiring onto the Telegram plugin dispatcher:** `src/telegram/bot-message-dispatch.ts:17`
  - body: This change edits src/telegram/bot-message-dispatch.ts, but current main has no src/telegram tree and handles Telegram replies from extensions/telegram/src/bot-message-dispatch.ts. As written, the feature misses the active Telegram runtime path.
  - confidence: 0.92
- **[P2] Preserve handles when cleanup races send:** `src/channels/waiting-tips.ts:215-216`
  - body: If cleanup() runs after the timer fires but before sendTip resolves, cancel() makes the later messageId assignment skip, so the sent tip has no handle and cannot be deleted. Record the handle whenever sendTip succeeds, then let deleteAfterReply control cleanup.
  - confidence: 0.94
- **[P2] Remove or implement custom tips path before exposing it:** `src/config/schema.help.ts:1451-1452`
  - body: messages.waitingTips.customTipsPath is typed, validated, labeled, and documented, but the patch never reads that path or passes loaded tips into the controller. Operators can configure it and see no behavior change.
  - confidence: 0.91
- **[P2] Use Telegram thread-aware delivery helpers for tip sends:** `src/telegram/bot-message-dispatch.ts:501-505`
  - body: The tip sender calls bot.api.sendMessage with a raw message_thread_id, bypassing the current helper behavior that omits General-topic IDs and handles thread failures. Waiting tips can fail in chats where normal replies already work.
  - confidence: 0.86
