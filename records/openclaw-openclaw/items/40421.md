## Summary

Keep open. The requested Telegram waiting-feedback feature is not implemented on current main, but this PR is not merge-ready because it targets the removed `src/telegram` path, has a send/cleanup race, exposes an unimplemented `customTipsPath`, bypasses current Telegram thread normalization, and still lacks real Telegram behavior proof.

## What This Changes

This PR adds a shared waiting-tip controller, `messages.waitingTips` config/schema/help/labels, tests, and Telegram dispatch wiring to send and delete bilingual tip messages while replies are pending.

## Best Possible Solution

Choose one opt-in Telegram waiting UX, then implement it inside `extensions/telegram` with existing delivery/thread helpers, only implemented config, focused lifecycle tests, docs/changelog if accepted, and real Telegram proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the diff and current main shows the stale Telegram path, cleanup race, unimplemented `customTipsPath`, and thread-helper bypass. As a user request, this is a feature rather than a current-main bug reproduction.

## Solution Assessment

No. This PR is not the best current solution because it targets stale structure, bypasses current Telegram delivery semantics, and broadens user-facing UX/config before the related product decision is settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Rebase the wiring onto the Telegram plugin dispatcher:** `src/telegram/bot-message-dispatch.ts:17`
  - body: This PR wires the feature into `src/telegram/bot-message-dispatch.ts`, but current main removed that tree and handles Telegram replies from `extensions/telegram/src/bot-message-dispatch.ts`. As written, the feature misses the active runtime path.
  - confidence: 0.94
- **[P2] Preserve handles when cleanup races send:** `src/channels/waiting-tips.ts:215-216`
  - body: If `cleanup()` runs after the timer fires but before `sendTip` resolves, `cancel()` makes the later `messageId` assignment skip, so the sent tip has no handle and cannot be deleted. Record the handle whenever send succeeds, then let `deleteAfterReply` control cleanup.
  - confidence: 0.94
- **[P2] Remove or implement custom tips path before exposing it:** `src/config/schema.help.ts:1451-1452`
  - body: `messages.waitingTips.customTipsPath` is typed, validated, labeled, and documented, but the patch never reads that path or passes loaded tips into the controller. Operators can configure it and see no behavior change.
  - confidence: 0.91
- **[P2] Use Telegram thread-aware delivery helpers for tip sends:** `src/telegram/bot-message-dispatch.ts:501-505`
  - body: The tip sender calls `bot.api.sendMessage` with a raw `message_thread_id`, bypassing current helper behavior that omits General-topic IDs and preserves Telegram topic semantics. Waiting tips can fail in chats where normal replies already work.
  - confidence: 0.86
- **[P3] Correct the built-in tip count comment:** `src/channels/waiting-tips.ts:54`
  - body: The comment says there are 75 built-in tips, but the test and array expect 74. Keeping the comment wrong makes the test expectation look suspicious during later edits.
  - confidence: 0.82
