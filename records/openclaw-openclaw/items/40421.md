## Summary

Keep this PR open, but do not merge it as-is. Current main does not implement waiting tips, and this branch is stale against the Telegram plugin layout while still carrying concrete correctness/config blockers already noted in prior review comments.

## What This Changes

This PR adds a core waiting-tip controller with hard-coded bilingual tips, exposes `messages.waitingTips` config metadata, and wires Telegram dispatch to send and later delete a temporary tip while the agent replies.

## Best Possible Solution

Settle the UX against #6946, then implement the accepted waiting/processing indicator inside the current Telegram plugin flow using existing delivery/thread helpers, exposing only implemented config and adding focused tests plus a changelog entry.

## Reproduction Assessment

Yes. The main correctness issue is reproducible by letting the timer fire, keeping `sendTip` in flight, then calling `cleanup()`: `cancelled` becomes true before the handle assignment, so the sent message cannot be deleted. The no-op `customTipsPath` is also statically reproducible because no diff code reads files or passes custom tips to the controller.

## Solution Assessment

No. The proposed branch is not the best current solution because it targets the old Telegram path, bypasses current Telegram delivery/thread helpers, and exposes unimplemented config; a rebased plugin-owned implementation should be narrower and match current main’s lifecycle seams.
