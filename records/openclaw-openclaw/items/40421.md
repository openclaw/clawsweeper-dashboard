## Summary

Keep open for maintainer handling: current main does not implement the requested Telegram waiting-tip lifecycle, but this PR is not merge-ready because it targets stale Telegram structure, has concrete lifecycle/config bugs, lacks real Telegram proof, and overlaps the canonical Telegram waiting UX decision.

## What This Changes

The PR adds a configurable bilingual waiting-tip controller, shared message config fields, tests, and Telegram dispatch wiring to send and delete a temporary tip while a reply is pending.

## Best Possible Solution

Choose the Telegram waiting UX in https://github.com/openclaw/openclaw/issues/6946, then implement it inside extensions/telegram with existing delivery/thread helpers, only implemented config, lifecycle tests, docs/changelog, and real Telegram proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the diff and current main shows stale src/telegram wiring, a cleanup/send race, and a customTipsPath config surface without runtime loading. As the user-facing request, this is a feature/config proposal rather than a bug reproduction.

## Solution Assessment

No. This is not the best current solution because it targets stale Telegram structure, bypasses current thread-helper semantics, and expands UX/config before maintainers settle the canonical Telegram waiting-message direction.

  - confidence: 0.91
- **[P2] Use Telegram thread-aware delivery helpers for tip sends:** `src/telegram/bot-message-dispatch.ts:501-505`
  - body: The tip sender calls bot.api.sendMessage with a raw message_thread_id, bypassing current helper behavior for DM topics and General-topic message sends. Waiting tips can fail in chats where normal replies already use the supported delivery path.
  - confidence: 0.86
