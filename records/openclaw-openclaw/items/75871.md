## Summary

Keep open: this PR is useful source work and is not implemented on current main, but it needs small follow-up before merge. The main functional shape is conservative and opt-in, but the new prompt mode is not fully aligned with the system-prompt diagnostic path and the user-facing config key lacks required docs/changelog coverage.

## What This Changes

The PR adds opt-in `skills.promptMode` progressive skill disclosure, new `skill_view` and `skill_search` tools, telemetry, schema/type updates, and focused tests for skill prompt formatting and tool behavior.

## Best Possible Solution

Keep the opt-in, default-legacy design, but align every prompt-reporting path that can render progressive skill instructions with the matching tool list, document the new config mode and pilot expectations, add the required changelog entry, and then require the exact head CI/changed gate to pass before merge.

## Reproduction Assessment

Yes for the review findings: with `skills.promptMode` set to `view` or `search`, `resolveCommandsSystemPromptBundle` can build a progressive `skillsPrompt` from `buildWorkspaceSkillSnapshot` while constructing tools without passing that snapshot. For the overall PR behavior, the author supplied focused test and typecheck/build claims, but I did not run tests in this read-only review.

## Solution Assessment

Mostly yes: an opt-in global mode backed by the already resolved prompt-visible skill snapshot is a narrow maintainable phase-one approach. It is not complete until diagnostics, docs/config help, changelog, and CI are aligned with the new public config and tool surface.
