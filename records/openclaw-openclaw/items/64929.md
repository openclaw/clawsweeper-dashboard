## Summary

Keep open. Current main still routes loopback local-managed browser profiles through the raw CDP/json tab-management path, and the report is recent, concrete, corroborated, and not covered by a linked merged fix or current-main benchmark proof.

## What This Changes

The issue asks OpenClaw to make local managed Brave/Chrome browser actions faster by reducing repeated CDP attach, discovery, and target-selection overhead, with benchmarks or timing diagnostics to prevent regressions.

## Best Possible Solution

Add focused browser action phase timing or a loopback benchmark first, then optimize local managed tab selection/open paths to avoid unnecessary repeated discovery or short-lived socket setup while preserving SSRF policy, stable tab aliases, profile lifecycle, and cleanup semantics.

## Solution Assessment

Unclear for the exact implementation. Persistent connection reuse or target caching is a plausible direction, but the narrowest maintainable fix should be chosen after phase timing confirms the bottleneck and browser/CDP lifecycle contracts are reviewed.
