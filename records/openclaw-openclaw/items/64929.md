## Summary

Keep open. Current main still routes local-managed loopback browser profiles through raw CDP/json tab discovery rather than persistent Playwright, and the issue has concrete, corroborated timing reports with no current-main benchmark or merged lifecycle change proving the slowdown is gone.

## What This Changes

The issue asks OpenClaw to investigate and reduce slow local-managed browser CLI actions by avoiding repeated CDP attach/discovery, adding regression timing coverage, and documenting expected browser performance.

## Best Possible Solution

Add focused browser phase timing or a loopback action benchmark, then optimize local-managed tab discovery/open paths to avoid unnecessary repeated discovery and short-lived sockets while preserving SSRF policy, stable tab aliases, profile lifecycle, and cleanup semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: run the reported managed Brave or default managed Chrome CLI commands against https://example.com and compare per-command wall time. I did not run the macOS browser benchmark here, but current main still contains the repeated local-managed CDP discovery/attach path that matches the reports.

## Solution Assessment

Unclear for the exact implementation. Persistent Playwright routing, connection reuse, and target-state caching are plausible, but phase timing should guide the narrowest maintainable fix before changing this SSRF-sensitive browser lifecycle path.
