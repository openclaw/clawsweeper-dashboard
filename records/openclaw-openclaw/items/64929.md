## Summary

Keep open. Current main still routes local managed loopback browser profiles through raw CDP/json tab management instead of the persistent Playwright path, and the issue has concrete, corroborated timing evidence with no merged fix or benchmark proof showing the regression is gone.

## What This Changes

The issue asks OpenClaw to make local managed Brave/Chrome browser actions faster by reducing repeated CDP attach, discovery, and target-selection overhead and adding timing diagnostics or benchmark coverage.

## Best Possible Solution

Add focused phase timing or a loopback browser-action benchmark first, then optimize the local managed tab selection/open path to avoid unnecessary discovery and short-lived socket setup while preserving SSRF policy, profile lifecycle, stable tab aliases, and cleanup semantics. Coordinate with #67966 if the chosen fix changes when local-managed profiles use Playwright.

## Reproduction Assessment

Yes. A maintainer can reproduce with the listed managed Brave or default managed Chrome commands against https://example.com and compare per-command wall time; current source inspection also confirms the repeated local-managed CDP discovery/attach path that matches the report.

## Solution Assessment

Unclear for the exact implementation. Persistent connection reuse, cached target state, or selective Playwright routing are plausible, but the narrowest maintainable fix should be chosen after phase timing confirms the bottleneck and browser/CDP lifecycle contracts are reviewed.
