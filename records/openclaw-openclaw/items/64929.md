## Summary

Keep open. Current main and the latest shipped release still route local-managed browser profiles through raw CDP, repeated tab discovery, and short-lived CDP sockets, and the related Playwright/SSRF issue does not cover the broader performance and long-lived-session request.

## What This Changes

The issue asks OpenClaw to reduce local-managed Brave/Chrome browser action latency by avoiding repeated CDP attach and target discovery, with benchmark or trace coverage.

## Best Possible Solution

Add browser phase timing or a loopback action benchmark, then optimize local-managed connection and tab reuse inside the browser plugin while preserving SSRF policy, tab aliases, profile lifecycle, and cleanup behavior.

## Reproduction Assessment

Yes. The high-confidence path is to run the reported managed Brave or default managed Chrome CLI actions against https://example.com; I did not run that macOS benchmark here, but current source still contains the repeated discovery and short-lived CDP socket path that matches the reports.

## Solution Assessment

Unclear for the exact implementation. Long-lived CDP or Playwright session reuse is plausible, but phase timing should guide the narrowest maintainable change and it should be coordinated with https://github.com/openclaw/openclaw/issues/67966 before changing lifecycle behavior.
