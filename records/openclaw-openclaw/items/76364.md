## What This Changes

The PR treats ENOENT/EACCES/EPERM `lsof` spawn errors during the initial Unix stale-PID scan as unavailable, reuses that classification in the port polling path, and updates adjacent unit coverage.

## Best Possible Solution

Refresh the branch against current main, add redacted real no-`lsof` gateway startup proof, and land this narrow runtime/test fix if the merge result remains clean.

## Reproduction Assessment

Yes, at source level: current main can resolve to bare `lsof`, Node returns ENOENT/status null for a missing command, and the initial scan logs the reported warning for any spawn error. I did not run a full minimal-container gateway startup in this read-only review.

## Solution Assessment

Yes, this is the narrow maintainable fix because it aligns the initial scan with the existing unavailable-lsof classification used by the later polling path. A separate preflight binary lookup would add another resolution path without handling the spawn-error boundary as directly.
