## Summary

Keep open. Current main still logs the expected startup-sidecars 1013 pre-connect close through WARN, even though the startup path is modeled as retryable and clients already suppress terminal errors for it.

## What This Changes

Downgrade the gateway WebSocket startup-sidecars 1013 close-before-connect log from WARN to a lower expected-startup severity.

## Best Possible Solution

Use the existing close cause to treat `startup-sidecars-pending` 1013 pre-connect closes as expected startup churn while preserving WARN for unexpected handshake, auth, protocol, timeout, and payload failures.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a connect frame, and the handler marks `startup-sidecars-pending` before closing 1013; the close handler then routes that non-client close to WARN.

## Solution Assessment

Yes. The narrowest maintainable fix is to branch the pre-connect close logger on the existing startup close cause and add a regression assertion, without changing retry timing, close code, readiness, or client reconnect behavior.
