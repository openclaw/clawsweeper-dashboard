## Summary

Keep open. Current main still logs the expected startup-sidecars 1013 pre-connect close through WARN, and the related open health-probe PR covers a different code=1000 path.

## What This Changes

Downgrade expected gateway startup WebSocket pre-connect 1013 close logging from WARN to debug or info while preserving WARN for unexpected pre-handshake closes.

## Best Possible Solution

Use the existing `startup-sidecars-pending` close cause to log expected startup retry closes below WARN, with regression coverage, while leaving handshake, auth, protocol, timeout, and payload failures unchanged.

## Reproduction Assessment

Yes. Source inspection gives a concrete path: force `isStartupPending()` true, send a connect frame, the handler marks `startup-sidecars-pending` and closes 1013, then the pre-connect close handler routes it to WARN.

## Solution Assessment

Yes. The narrowest maintainable fix is to branch the pre-connect close logger on the existing startup close cause and add a focused regression assertion without changing retry timing or client reconnect behavior.
