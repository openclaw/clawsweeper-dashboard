## Summary

Keep open. Current main still routes the expected startup-sidecars 1013 pre-connect close through the generic WARN path, while the protocol and clients already model that startup response as retryable connection churn.

## What This Changes

The issue asks to downgrade or suppress WARN logging for expected Gateway WebSocket pre-connect closes with code 1013 and reason `gateway starting` during startup.

## Best Possible Solution

Use the existing close cause to treat `startup-sidecars-pending` 1013 pre-connect closes as expected startup churn while preserving WARN for unexpected handshake, auth, protocol, timeout, and payload failures.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a connect frame, and the handler marks `startup-sidecars-pending` before closing 1013; the close handler then routes that non-client close to WARN. I did not run a live gateway restart because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to branch the pre-connect close logger on the existing startup close cause and add a regression assertion, without changing retry timing, close code, readiness, or client reconnect behavior.
