## Summary

Keep open. Current main still routes the expected startup-sidecars 1013 pre-connect close through WARN, and the linked open PR #77189 should be reviewed or landed before this issue closes.

## What This Changes

Downgrade the expected gateway startup WebSocket pre-connect 1013 close log from WARN to debug or info without masking unexpected pre-connect failures.

## Best Possible Solution

Review and land #77189 or an equivalent narrow fix that downgrades only the startup-sidecars 1013 pre-connect close while preserving WARN for unexpected pre-connect failures.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a connect frame, and the handler marks `startup-sidecars-pending` before closing 1013; the close handler then routes that non-client close to WARN. I did not run a live gateway because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable solution is to branch the pre-connect close logger on the existing startup-sidecars 1013 cause or equivalent code/reason check, with a regression assertion, and leave retry behavior unchanged.
