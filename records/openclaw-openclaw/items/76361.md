## Summary

Keep open. Current main still routes the expected startup-sidecars 1013 pre-connect close through `logWsControl.warn`, while the protocol, server branch, docs, and clients all model that path as retryable startup churn.

## What This Changes

Downgrade the gateway WebSocket WARN for expected startup-sidecars 1013 close-before-connect events during Control UI/WebChat reconnects.

## Best Possible Solution

Key the pre-connect close logger off the existing `startup-sidecars-pending` cause for the 1013 `gateway starting` close, add a focused regression assertion, and leave retry timing, close codes, readiness, and reconnect behavior unchanged.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a valid connect frame, and the handler sends retryable startup-unavailable before closing 1013. Because no client is registered yet, the current close handler falls through to WARN unless it matches the unrelated SwiftPM helper exception.

## Solution Assessment

Yes. The narrow maintainable fix is to special-case the existing startup close cause in the server logger and add regression coverage, without changing the startup protocol, client retry behavior, or readiness sequencing.
