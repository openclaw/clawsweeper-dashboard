## Summary

Keep open. Current main still treats the expected startup-sidecars 1013 pre-connect close as a WARN, while the surrounding protocol and client code model that startup response as retryable connection churn.

## What This Changes

The issue asks to downgrade expected gateway startup WebSocket 1013 pre-connect close logs from WARN to a lower severity while preserving warnings for unexpected pre-connect closes.

## Best Possible Solution

Use the existing startup close cause to log `startup-sidecars-pending` 1013 pre-connect closes below WARN, while preserving WARN for unexpected handshake, auth, protocol, timeout, payload, and unknown pre-connect failures.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a connect frame, and the handler marks `startup-sidecars-pending` before closing 1013; the close handler then routes that non-client close to WARN. I did not run the test suite because `pnpm` is unavailable in this read-only environment.

## Solution Assessment

Yes. The narrowest maintainable fix is to branch the pre-connect close logger on the existing startup close cause and add a regression assertion, without changing retry timing, close code, readiness, or client reconnect behavior.
