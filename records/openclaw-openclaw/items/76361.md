## Summary

Keep open. Current main still routes the expected startup-sidecars 1013 pre-connect WebSocket close through the generic WARN path, and the related open health-probe PR covers a different code-1000 case rather than this startup 1013 path.

## What This Changes

The issue asks to downgrade expected gateway startup WebSocket close-before-connect 1013 logs from WARN to a lower log level.

## Best Possible Solution

Treat `startup-sidecars-pending` 1013 pre-connect closes as expected startup churn at debug or info while preserving WARN for unexpected handshake, auth, protocol, timeout, payload, and load failures.

## Reproduction Assessment

Yes, at source level: with `isStartupPending()` true, a connect frame is marked `startup-sidecars-pending`, closed with 1013, and then routed through the generic pre-connect WARN logger. I did not run Vitest in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to branch the pre-connect close logger on the existing startup close cause and add a regression assertion, without changing retry timing, close code, readiness, or client reconnect behavior.
