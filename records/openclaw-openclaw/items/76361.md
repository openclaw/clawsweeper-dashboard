## Summary

Keep open. Current main still sends the expected startup-sidecars 1013 pre-connect close through the generic Gateway WebSocket WARN path, while docs and client tests model the same startup response as retryable and non-terminal.

## What This Changes

Requests lowering the Gateway WebSocket pre-connect 1013 `gateway starting` startup log from WARN to debug/info, or warning only after repeated failures.

## Best Possible Solution

Classify `startup-sidecars-pending` 1013 pre-connect closes as expected startup churn in the server logger while preserving WARN for unexpected handshake, auth, protocol, timeout, and payload failures.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a valid `connect` frame, and the handler returns retryable startup `UNAVAILABLE` before closing 1013. Because no client is registered yet, the close handler reaches the generic pre-connect WARN logger.

## Solution Assessment

Yes. The narrow maintainable fix is to key the pre-connect close logger off the existing `startup-sidecars-pending` close cause and add a regression assertion, without changing retry timing, close code, readiness, or client reconnect semantics.
