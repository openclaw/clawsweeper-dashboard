## Summary

Keep open. Current main still treats the expected startup-sidecars pre-connect 1013 close as a generic non-client close and logs it through WARN, while the protocol and clients model that startup response as retryable.

## What This Changes

Downgrade expected gateway WebSocket startup 1013 pre-connect close logging from WARN to a lower severity while preserving WARN for unexpected pre-connect failures.

## Best Possible Solution

Use the existing close cause to classify `startup-sidecars-pending` 1013 pre-connect closes as expected startup churn, with regression coverage that preserves WARN for unexpected handshake failures.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a connect frame, and the handler marks `startup-sidecars-pending` before closing 1013; the close handler then routes that non-client close to WARN.

## Solution Assessment

Yes. The narrowest maintainable fix is to branch the pre-connect close logger on the existing startup close cause and add a regression assertion, without changing retry timing, close code, readiness, or client reconnect behavior.
