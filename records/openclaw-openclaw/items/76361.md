## Summary

Keep open. Current main still routes the expected startup-sidecars 1013 pre-connect close through the generic WARN path, even though the same startup condition is modeled as retryable client behavior.

## What This Changes

Issue requests lowering the startup-sidecars-pending WebSocket close-before-connect 1013 log level because it represents expected gateway startup retry behavior.

## Best Possible Solution

Use the existing startup-sidecars close cause to log this expected 1013 pre-connect close at debug or info while preserving WARN for unexpected handshake, auth, protocol, timeout, and payload failures.

## Reproduction Assessment

Yes, at source level: force isStartupPending() true, send a connect frame, and the handler marks startup-sidecars-pending before closing 1013; the close handler then routes that non-client close to WARN.

## Solution Assessment

Yes. The narrow maintainable fix is to branch the pre-connect close logger on the existing startup close cause and add a regression assertion, without changing retry timing, close codes, readiness, or client reconnect behavior.
