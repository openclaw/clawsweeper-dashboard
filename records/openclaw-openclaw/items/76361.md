## Summary

Keep open. Current main still sends the expected startup-sidecars `1013` close through the generic pre-connect WARN path, while the protocol and clients already model this as retryable startup churn.

## What This Changes

Request to downgrade expected gateway startup-sidecars WebSocket `1013` pre-connect closes from WARN to debug/info, or warn only after repeated failures.

## Best Possible Solution

Use the existing startup close cause to log `startup-sidecars-pending` `1013` pre-connect closes as expected startup churn while preserving WARN for unexpected handshake, auth, protocol, timeout, and payload failures.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a connect frame, and the handler marks `startup-sidecars-pending` before closing `1013`; the close handler then routes that non-client close to WARN.

## Solution Assessment

Yes. The narrowest maintainable fix is to branch the pre-connect close logger on the existing startup close cause and add a regression assertion, without changing retry timing, close code, readiness, or client reconnect behavior.
