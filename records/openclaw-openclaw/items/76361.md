## Summary

Keep open. Current main still routes the expected startup-sidecars 1013 pre-connect close through WARN, and open PR #77189 explicitly references this issue with closing syntax, so this issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests downgrading the expected gateway startup WebSocket 1013 close-before-connect log from WARN to debug/info while preserving WARN for unexpected pre-connect closes.

## Best Possible Solution

Review and land a narrow fix that recognizes the gateway's own startup 1013 close as expected startup churn while leaving other pre-connect failures at WARN and covering the case with a focused regression test.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a connect frame, and the server marks `startup-sidecars-pending`, closes 1013 with `gateway starting`, then the pre-connect close handler routes that close through WARN on current main. I did not run a live gateway because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to branch the pre-connect logger on the existing startup 1013 close cause, add a regression assertion, and avoid changing retry timing, close code, readiness, or client reconnect behavior.
