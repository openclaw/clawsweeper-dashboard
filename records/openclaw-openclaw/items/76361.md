## Summary

Keep open. Current main still routes the expected startup-sidecars 1013 pre-connect close through WARN, and the issue is now paired with open closing PR #77189, so it should stay open until that PR lands or is closed.

## What This Changes

The issue asks to downgrade expected gateway startup WebSocket 1013 close-before-connect logs from WARN to debug/info, or only warn after repeated failures.

## Best Possible Solution

Land a narrow fix that recognizes the existing startup-sidecars 1013 close as expected startup churn and logs it below WARN while preserving WARN for other pre-connect failures.

## Reproduction Assessment

Yes, at source level: set `isStartupPending()` true, send a connect frame, and the handler closes 1013 with `gateway starting`; the current close handler then logs that pre-connect close at WARN unless it is the SwiftPM helper case.

## Solution Assessment

Yes. The narrowest maintainable solution is to branch the close-before-connect logger on the existing startup 1013 state/reason and add a regression assertion, without changing retry timing, protocol shape, or client reconnect behavior.
