## Summary

Keep open. Current main still sends the expected startup 1013 close through the pre-connect WARN path, and the linked fix PR is open but not merged.

## What This Changes

The issue asks to downgrade expected startup WebSocket close-before-connect logs for code 1013 with reason `gateway starting` from WARN to a lower severity while preserving warnings for unexpected pre-connect closes.

## Best Possible Solution

Land or revise #77189 so the server's pre-connect close logger treats the existing startup-sidecars 1013 close as expected startup churn and adds regression coverage.

## Reproduction Assessment

Yes, source-level: `isStartupPending()` leads the connect handler to send retryable `UNAVAILABLE`, mark `startup-sidecars-pending`, and close with 1013/`gateway starting`; the close handler then routes the pre-connect close to WARN unless the SwiftPM helper exception matches. I did not run a live gateway under the read-only review constraints.

## Solution Assessment

Yes. The narrow maintainable fix is to branch the pre-connect logger on the existing startup code/reason or close cause, add a regression assertion, and leave retry timing, close code, readiness, and client reconnect behavior unchanged.
