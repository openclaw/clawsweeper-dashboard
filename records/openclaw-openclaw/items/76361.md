## Summary

Keep open. Current main still routes the expected startup 1013 pre-connect close through `logWsControl.warn`, while the same path is explicitly modeled as retryable startup-unavailable and clients already retry without terminal errors. This is a narrow logging cleanup, not implemented or superseded by the supplied related context.

## What This Changes

Downgrade the expected gateway startup WebSocket close-before-connect log for a 1013 `gateway starting` close from WARN to a lower severity.

## Best Possible Solution

Treat `startup-sidecars-pending` 1013 pre-connect closes as expected startup churn in the server logger while preserving WARN for unexpected handshake, auth, protocol, timeout, and payload failures.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a connect frame, and the handler returns retryable startup-unavailable before closing 1013. The current close handler then logs any non-client close through WARN unless it matches the unrelated SwiftPM helper exception.

## Solution Assessment

Yes. The narrow maintainable fix is to key the pre-connect close logger off the existing `startup-sidecars-pending` close cause and add a regression assertion, without changing retry timing, close code, readiness, or client reconnect semantics.
