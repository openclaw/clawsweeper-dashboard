## Summary

Keep open. Current main still routes the expected startup-sidecars 1013 pre-connect close through WARN, while the protocol/docs/tests already model that startup response as retryable; related PR #60536 covers a different internal-probe code 1000 case.

## What This Changes

The issue asks to stop logging expected startup WebSocket close-before-connect code 1013 as a warning during gateway restart.

## Best Possible Solution

Use the recorded `startup-sidecars-pending` close cause to treat expected startup 1013 pre-connect closes as normal startup churn while preserving WARN for unexpected pre-connect failures.

## Reproduction Assessment

Yes, at source level. Force `isStartupPending()` true and send a connect frame; the handler marks `startup-sidecars-pending`, closes 1013 before client registration, and the shared pre-connect close handler currently logs that through WARN.

## Solution Assessment

Yes. The narrow maintainable fix is to branch the pre-connect close logger on the existing startup close cause and add a regression assertion without changing retry timing, close code, readiness, or client reconnect behavior.
