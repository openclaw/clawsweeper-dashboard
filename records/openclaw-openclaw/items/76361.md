## Summary

Keep open. Current main still routes the expected startup-sidecars 1013 pre-connect close through the generic WARN logger, while the protocol and clients already model that startup response as retryable churn.

## What This Changes

Downgrade the expected gateway startup WebSocket 1013 close-before-connect log from WARN to a lower severity or bounded warning policy.

## Best Possible Solution

Treat the existing `startup-sidecars-pending` 1013 pre-connect close as expected startup churn while preserving WARN for unexpected preauth, auth, protocol, timeout, and payload failures.

## Reproduction Assessment

Yes, at source level: with `isStartupPending()` true, a connect frame marks `startup-sidecars-pending`, sends retryable `UNAVAILABLE`, and queues `close(1013, "gateway starting")`; the `!client` close handler still chooses `logWsControl.warn`.

## Solution Assessment

Yes. The narrowest maintainable fix is to branch the pre-connect close logger on the existing startup close cause/code and add a regression assertion, without changing retry timing, readiness, or client reconnect behavior.
