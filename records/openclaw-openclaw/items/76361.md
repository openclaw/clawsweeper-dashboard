Current main still routes the retryable startup-sidecars pre-connect close through `logWsControl.warn`, so this remains a valid narrow logging cleanup rather than a close candidate.

## What This Changes

Downgrade the expected gateway startup WebSocket 1013 close-before-connect log from WARN while preserving WARN for unexpected pre-connect failures.

## Best Possible Solution

Use the existing close cause to treat `startup-sidecars-pending` 1013 pre-connect closes as expected startup churn while leaving unexpected handshake, auth, protocol, timeout, and payload failures at WARN.

## Repair Work Prompt

Fix the gateway WebSocket startup 1013 close-before-connect warning reported in https://github.com/openclaw/openclaw/issues/76361. On current main, `isStartupPending()` makes the connect handler mark `startup-sidecars-pending`, send retryable `UNAVAILABLE`, and close with 1013 `gateway starting`, but `src/gateway/server/ws-connection.ts` still logs the pre-connect close at WARN unless it is the unrelated SwiftPM helper case. Make only this expected startup-sidecars close log at debug or another non-warning level; preserve WARN for unexpected preauth closes including handshake timeout, auth failure, protocol mismatch, payload limit, and other non-startup 1013/1008 cases. Add focused regression coverage in `src/gateway/server/ws-connection.startup.test.ts` asserting that startup-sidecars 1013 avoids `logWsControl.warn` and still records useful debug/context. Do not change retry timing, close code/reason, readiness semantics, or client reconnect behavior. Add a single-line `CHANGELOG.md` fix entry if this is user-facing per repo policy. Related context: PR #60536 is adjacent but only covers internal health probes with close code 1000 and should not be treated as covering this issue.
