## Summary

Keep open. Current main still logs the retryable startup-sidecars WebSocket close through WARN; the existing startup protocol/client code treats this as expected reconnect churn, and the related health-probe PR covers a different 1000 close path.

## What This Changes

The issue requests downgrading expected gateway startup WebSocket 1013 close-before-connect logs from WARN to a lower log level.

## Best Possible Solution

Use the existing startup-sidecars close cause to log this expected startup 1013 close at debug or info while preserving WARN for unexpected handshake, auth, protocol, timeout, and payload failures.

## Reproduction Assessment

Yes, at source level: force `isStartupPending()` true, send a connect frame, and current main marks `startup-sidecars-pending` before closing 1013. The close handler then sees no registered client and routes the close to WARN.

## Solution Assessment

Yes. The narrowest maintainable solution is to branch the pre-connect close logger on the existing startup close cause and add a regression assertion, without changing retry timing, close code, readiness, or client reconnect behavior.
