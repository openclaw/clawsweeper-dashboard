## Label Justifications

- `P2`: The issue affects external integrations that map Gateway replies to original IM questions, but the fix is a bounded protocol/API improvement rather than an emergency outage.
- `impact:message-loss`: The reported failure mode is reply/question miscorrelation when collect-mode processing merges queued chat messages.

## Summary

Keep open: current main and v2026.5.22 still do not expose original queued chat.send run IDs on Gateway agent events. Source shows collect mode can coalesce queued messages into one follow-up execution with a fresh runId, so the remaining work is an additive Gateway/SDK protocol decision rather than conservative cleanup.

## What This Changes

The issue requests that merged queued messages expose the original chat.send run IDs on the agent execution event so gateway clients can correlate replies to source messages.

## Best Possible Solution

Add an additive lineage field that carries all source chat.send run IDs through collect/follow-up execution into the Gateway protocol and SDK while preserving the actual execution runId semantics.

## Reproduction Assessment

Yes, at source level: collect mode can merge queued messages into one follow-up execution, that execution gets a fresh runId, and the Gateway/SDK event surfaces expose only a single runId with no original run-id lineage. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive lineage field is the safest apparent direction; selecting the first or last queued runId would conflate source-message identity with execution-run identity.
