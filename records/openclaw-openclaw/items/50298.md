## Summary

Keep open: current main still exposes only singular execution/client run IDs, while collect/follow-up handling can merge queued messages and execute under a fresh run ID without a public original-run lineage field.

## What This Changes

Request to expose the original chat.send run IDs on agent/WebSocket events when queued steer/collect messages are merged into a later execution run.

## Best Possible Solution

Add an additive, documented original client-run lineage field while preserving the actual execution runId, then cover chat.send, collect/follow-up drain, gateway projection, SDK normalization, and protocol docs with focused tests.

## Reproduction Assessment

Yes. Source inspection shows chat.send returns the idempotency key as the client run ID, collect/follow-up processing can merge queued items and allocate a fresh execution run ID, and public agent events expose no original-run lineage field.

## Solution Assessment

Unclear until maintainers choose the public event contract. An additive lineage field is likely safer than selecting the first or last queued run ID because it preserves execution identity while exposing the original client IDs.
