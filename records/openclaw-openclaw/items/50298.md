- `impact:message-loss`: Clients can mis-correlate or effectively lose replies when merged queued execution cannot be mapped back to the original submitted messages.
- `maturity:stable`: The affected Gateway runtime RPC/event surface is listed as M4 Stable, with Gateway RPC APIs and Events covering chat APIs and request/event envelopes.

## Summary

Keep open: current main still exposes only singular `runId`/`clientRunId` in Gateway agent/chat event contracts, while the internal steering queue preserves source `runIds` only for lease/ack; this is valid core protocol work that needs maintainer API direction.

## What This Changes

The issue requests that Gateway websocket agent events for merged queued message execution carry the original submitted run ID set so clients can map replies back to source messages.

## Best Possible Solution

Add a maintainer-approved additive lineage field to Gateway agent/chat event contracts and propagate queued source run IDs from steering lease state into emitted events, with regression coverage and docs if the public protocol changes.

## Reproduction Assessment

Yes. Source inspection shows internal queued steering batches retain source `runIds`, while current main and `v2026.6.10` public Gateway agent/chat event schemas and projections expose only singular run identifiers.

## Solution Assessment

Unclear until maintainers choose the public API shape. An additive lineage field looks safer than overloading `runId`, while selecting the first or last queued ID would be a semantic product decision.
