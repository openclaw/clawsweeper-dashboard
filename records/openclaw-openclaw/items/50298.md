## Summary

Keep open. Current main still does not expose the original queued `chat.send` run IDs on agent events: queued/collected followups retain only message identity, collect mode collapses queued items, and the drained followup creates a fresh execution `runId` while the agent event schema has no lineage field. The earlier #50296 thread points here, so this is the canonical remaining report rather than a duplicate.

## What This Changes

Expose the original `chat.send` run ID set on WebSocket agent events when queued or merged messages are processed under modes such as steer or collect.

## Best Possible Solution

Add an additive, stable, deduplicated lineage field for queued/collected executions so WebSocket clients can map a merged agent run back to every original `chat.send` run ID, while preserving the current primary `runId` semantics for direct runs and internal execution identity.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to send a `chat.send` while a session is already busy so it enters the followup queue, let collect/drain process it, and observe that `followup-runner` creates a fresh execution `runId` while the agent event schema cannot carry the original queued IDs.

## Solution Assessment

Yes, the requested additive lineage direction is the best maintainable fix. Reusing the first or last queued run ID as the execution `runId` would lose batch information and risks breaking existing consumers that treat `runId` as the actual execution identity.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/50298. Current `chat.send` returns the request `idempotencyKey` as `runId`, but when an active-session followup is queued/collected, the later drain in `src/auto-reply/reply/followup-runner.ts` creates a fresh execution `runId` and WebSocket agent events have no lineage back to the queued `chat.send` IDs. Add an additive protocol field, such as `originalRunIds` or `sourceRunIds`, with stable ordering and dedupe, to the agent event payloads emitted for queued/collected followup runs. Propagate the `chat.send` client run ID/`MessageSid` through `FollowupRun`, enqueue/get-reply-run, collect batching, followup execution, and gateway event broadcast as needed. Keep existing primary `runId` semantics unchanged; do not replace the execution run ID with the first or last queued ID. Update schemas/types/docs and add focused tests for collect-mode merging multiple chat sends plus ordinary direct `chat.send` behavior. Add a user-facing changelog entry if this repo’s current changelog section is open for gateway protocol changes, crediting `Thanks @yourse007`.
