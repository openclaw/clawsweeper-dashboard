## Summary

Keep open: current main still lacks a run-lineage field on gateway agent events, while collect/followup queue draining can coalesce queued messages into a fresh execution run ID. This remains protocol/API feature work, not cleanup-closeable stale noise.

## What This Changes

Request carrying the original queued chat run IDs on the actual agent event when followup or collect queue processing creates a merged execution run.

## Best Possible Solution

Add an additive run-lineage contract from FollowupRun through AgentRunContext/AgentEventPayload and the gateway protocol/generated clients, while preserving current runId semantics.

## Reproduction Assessment

Yes, by source inspection rather than a live run. chat.send returns a client runId, queued followup execution creates a fresh random UUID, collect can merge queued prompts, and AgentEventSchema has no field for the original queued run IDs.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive lineage field is likely safer than changing runId to the first or last queued ID because existing clients already use runId as the execution/chat event identifier.
