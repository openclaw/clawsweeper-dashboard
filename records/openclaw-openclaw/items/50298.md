## Label Justifications

- `P2`: The issue affects gateway users who need reliable reply/message correlation, but it is a bounded event-contract improvement rather than an emergency outage.
- `impact:message-loss`: The reported failure mode is replies becoming unmatchable to the original queued messages after run ids diverge.

## Summary

Keep open: current main and the latest release still expose only singular run/client-run mapping, while collect/follow-up processing can create a fresh execution run without carrying the original chat.send run ids into public agent events.

## What This Changes

The issue requests that merged queued-message processing expose the original chat.send runId values on the subsequent agent event.

## Best Possible Solution

Add an additive, documented lineage field for original client run ids while preserving the actual execution runId, then cover chat.send, collect/follow-up drain, gateway projection, and release-upgrade behavior with focused tests/docs.

## Reproduction Assessment

Yes from source: `chat.send` returns the idempotency key as `runId`, queued follow-up execution can allocate a fresh UUID, and the public event schema has no origin-run set. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public contract; additive lineage is likely safer than reusing a first or last queued runId because it preserves execution identity while exposing the original client ids.
