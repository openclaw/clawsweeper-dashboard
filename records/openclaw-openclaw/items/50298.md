## Label Justifications

- `P2`: The issue affects gateway clients that need reliable reply/message correlation, but the remaining work is a bounded protocol improvement rather than an emergency outage.
- `impact:message-loss`: Merged queued messages can make replies unmatchable to the original submitted message IDs, which is a message correlation and delivery-risk class.

## Summary

Keep open: current main still exposes singular execution/client run IDs, while collect-mode queued messages can merge into one follow-up without any public original-run lineage field.

## What This Changes

The issue requests that merged queued messages expose their original chat.send run IDs on the later agent execution event so gateway clients can correlate replies to original questions.

## Best Possible Solution

Add a documented additive original-run lineage field while preserving execution runId, then cover chat.send, collect/follow-up drain, gateway projection, SDK/protocol validation, and docs with focused tests.

## Reproduction Assessment

Yes. Source inspection shows chat.send returns one client run ID, collect-mode drain can merge queued items into one later follow-up, and public agent events still expose no original-run lineage field on current main.

## Solution Assessment

Unclear until maintainers choose the public event shape. An additive original-run-IDs field appears safer than selecting the first or last queued run ID because it preserves execution identity while exposing correlation IDs.
