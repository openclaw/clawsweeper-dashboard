## Label Justifications

- `P2`: The request affects Gateway message/reply correlation for queued runs but is a bounded enhancement rather than an emergency.
- `impact:message-loss`: The reported user impact is reply/question miscorrelation when queued IM messages are collected into a later execution run.

## Summary

Keep open: current main and the latest release still expose only a single run/client-run mapping, while the requested multi-origin lineage field for merged queued messages is still missing and needs a maintainer API decision.

## What This Changes

The issue requests Gateway/agent events to carry the original chat.send run ID set when queued messages are merged into a later execution run.

## Best Possible Solution

Define an additive lineage field on the appropriate Gateway event surfaces and propagate collect/follow-up source IDs without changing execution runId semantics.

## Reproduction Assessment

Yes, at source level: collect mode coalesces queued messages into one follow-up turn, chat.send acknowledges one client run ID, and agent/session event payloads expose at most one clientRunId mapping rather than a set of original run IDs. I did not run a live Gateway scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive lineage field is safer than selecting the first or last queued run ID because it preserves execution identity while giving external clients correlation data.
