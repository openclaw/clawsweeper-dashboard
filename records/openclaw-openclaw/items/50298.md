## Summary

Keep open: current main still does not expose original queued chat run IDs on gateway agent events, and the remaining work is additive gateway protocol/API design rather than cleanup-closeable stale noise.

## What This Changes

The issue requests that merged queued-message executions carry the original chat.send run IDs on websocket agent events.

## Best Possible Solution

Add an additive run-lineage contract from queued FollowupRun metadata through AgentRunContext/AgentEventPayload and the gateway protocol, while preserving current runId semantics.

## Reproduction Assessment

Yes, by source inspection. chat.send returns a client runId, collect/followup can run later under a fresh execution runId, and current AgentEventSchema has no field for the original queued run IDs.

## Solution Assessment

Unclear until maintainers choose the exact protocol shape. An additive lineage field is likely the maintainable path; changing runId to the first or last queued ID would risk breaking clients that already treat runId as the execution/chat event identifier.
