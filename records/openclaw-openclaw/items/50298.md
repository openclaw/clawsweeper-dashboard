## Label Justifications

- `P2`: The issue affects external Gateway integrations that correlate replies to original IM questions, but the remaining work is a bounded protocol/API improvement rather than an emergency outage.
- `impact:message-loss`: The reported failure mode is reply/question miscorrelation when queued chat messages are collected into a later execution run.

## Summary

Keep open: current main still lacks a public Gateway event lineage field for original queued chat.send run IDs when collect/follow-up processing executes under a separate run ID. The existing code exposes one execution/client run mapping at a time, while the requested multi-origin correlation contract still needs maintainer API direction.

## What This Changes

The issue requests that Gateway agent events carry the original queued chat.send run ID set when merged or collected messages later execute under a different run ID.

## Best Possible Solution

Define an additive Gateway/SDK lineage contract, such as originalRunIds or sourceRunIds on the relevant agent/chat/session events, and propagate it through collect/follow-up execution without changing actual execution runId semantics.

## Reproduction Assessment

Yes, at source level: chat.send acknowledges one client run ID, collect/follow-up processing can later execute under a fresh UUID, and the current event schema/projection has no field for the original run ID set. I did not run a live Gateway scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive lineage field is safer than reusing the first or last queued run ID because it preserves execution identity while giving clients correlation data.
