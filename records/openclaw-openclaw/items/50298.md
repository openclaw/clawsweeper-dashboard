## Summary

Keep open. Current main still has no way for public agent events to carry the original queued `chat.send` run ID set after collect/followup merges; the closed #50296 duplicate points here as the canonical remaining report.

## What This Changes

The issue requests that queued or merged message execution events expose the original `chat.send` run IDs so clients can map replies back to submitted messages.

## Best Possible Solution

Add an additive, stable, deduplicated queued-run lineage field to agent events while keeping the existing primary `runId` as the actual execution identity.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to send `chat.send` while a session is busy, let collect/followup drain merge the queued item, and observe that the later followup execution uses a fresh `runId` while the event schema cannot carry the original queued client run IDs.

## Solution Assessment

Yes. The additive lineage direction is the narrowest maintainable fix; reusing the first or last queued run ID would lose batch membership and blur client acknowledgement IDs with execution IDs.
