## Summary

Keep open: current main and the latest release still do not expose the original queued chat run IDs on gateway agent events, and the remaining work is an additive gateway protocol/product decision rather than a cleanup-closeable shipped fix.

## What This Changes

Add a way for agent events from merged queued messages to carry the original chat.send runId values so clients can correlate replies to submitted messages.

## Best Possible Solution

Add an additive ordered run-lineage contract from queued FollowupRun metadata through AgentRunContext or AgentEventPayload and the Gateway protocol, while preserving current execution runId semantics.

## Reproduction Assessment

Yes, by source inspection. chat.send returns the idempotency-derived client runId, queued followups can later execute under a fresh UUID, and current agent event schemas do not expose the original queued run IDs.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive lineage field is likely the maintainable path; changing runId to a first or last queued ID would risk breaking clients that already treat runId as the execution/event identifier.
