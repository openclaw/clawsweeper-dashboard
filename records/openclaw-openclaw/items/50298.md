## Summary

Keep open: current main and v2026.6.10 still expose only singular run identifiers in public gateway agent/chat events; internal queued steering batches preserve source run IDs only for lease/ack, not as a websocket event lineage field.

## What This Changes

Request that merged queued-message execution events carry the original `chat.send` run IDs, or reuse one original ID, so clients can correlate replies to submitted questions.

## Best Possible Solution

Add a maintainer-approved additive lineage field to gateway agent/chat event contracts and propagate queued source run IDs from queue lease/dispatch into the emitted events with regression coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current queued steering code has internal source `runIds`, while public gateway agent/chat event schemas and projections do not expose those IDs.

## Solution Assessment

Unclear until maintainers choose the public API shape; an additive lineage field looks safer than overloading `runId`, while choosing first/last original ID would change correlation semantics.
