## Label Justifications

- `P2`: The issue affects gateway clients that need reliable reply/message correlation, but the remaining work is a bounded protocol improvement rather than an emergency outage.
- `impact:message-loss`: Merged queued messages can make replies unmatchable to the original submitted message IDs, which is a message correlation and delivery-risk class.

## Summary

Keep open: current main and the latest release still expose only singular execution/client run IDs, while collect-mode queued messages can merge into one later follow-up without any public original-run lineage field.

## What This Changes

The issue requests original queued chat run IDs to be carried through later merged agent execution events so gateway clients can map replies back to submitted messages.

## Best Possible Solution

Add a documented additive original-run lineage field while preserving execution runId, then cover chat.send, collect/follow-up drain, gateway projection, SDK/protocol validation, and docs with focused tests.

## Reproduction Assessment

Yes. Source inspection shows chat.send returns one client run ID, collect drain can merge multiple queued items, and public agent events expose no original-run lineage field on current main or v2026.6.8.

## Solution Assessment

Unclear until maintainers choose the public event shape. An additive original-run-IDs field is likely safer than selecting the first or last queued run ID because it preserves execution identity while exposing client correlation IDs.
