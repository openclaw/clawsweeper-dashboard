## Label Justifications

- `P2`: The issue affects gateway users who need reliable reply/message correlation, but it is a bounded event-contract improvement rather than an emergency outage.
- `impact:message-loss`: The reported failure mode makes replies unmatchable to the original queued messages after run IDs diverge.

## Summary

Keep open: current main and v2026.6.8 still expose singular run/client-run IDs, while collect/follow-up processing can merge queued messages and execute under a fresh run ID without carrying the original chat.send run IDs into public agent events.

## What This Changes

Expose original chat.send run ID lineage on agent/gateway events when queued messages are merged and processed under a new execution run.

## Best Possible Solution

Add an additive, documented lineage field for original client run IDs while preserving the actual execution runId, then cover chat.send, collect/follow-up drain, gateway projection, and release-upgrade behavior with focused tests/docs.

## Reproduction Assessment

Yes. Source inspection shows chat.send returns the idempotency key as runId, collect/follow-up can merge queued messages and create a fresh execution runId, and the public agent event schema has no original-run lineage field.

## Solution Assessment

Unclear until maintainers choose the public event contract. An additive lineage field is likely safer than selecting a first or last queued runId because it preserves execution identity while exposing the original client IDs.
