## Summary

Keep open: current main and the latest release still lack a public Gateway event lineage field for the original queued `chat.send` run IDs when queued messages are merged into a later execution run. The source path shows a real correlation gap, but the additive protocol shape needs maintainer product/API direction before implementation.

## What This Changes

The issue requests Gateway websocket agent events to carry the original queued `chat.send` run IDs when queue modes merge messages into a later execution run.

## Best Possible Solution

Define an additive Gateway/protocol event lineage contract, such as `originalRunIds` or `sourceRunIds`, and propagate it through merged queued-message execution while preserving actual execution `runId` semantics.

## Reproduction Assessment

Yes, at source level: `chat.send` acknowledges one client run ID, queue modes can merge follow-up work, and the public Gateway agent event projection has no original-run ID set. I did not run a live Gateway websocket scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive lineage field is safer than selecting the first or last queued run ID because it preserves actual execution identity while exposing correlation data.
