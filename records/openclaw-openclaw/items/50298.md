## Label Justifications

- `P2`: The issue affects gateway clients that need reliable reply/message correlation, but the remaining work is a bounded protocol improvement rather than an emergency outage.
- `impact:message-loss`: Merged queued messages can make replies unmatchable to the original submitted message IDs, which is a message correlation and delivery-risk class.

## Summary

Keep open: current main and latest v2026.6.10 still expose only singular execution/client run IDs, while collect-mode queued messages can merge into one later follow-up without any public original-run lineage field. The remaining work needs maintainer product/API direction because it changes the gateway agent-event contract.

## What This Changes

The issue requests that merged queued agent runs expose the original client run IDs in gateway/agent events so clients can correlate replies to submitted messages.

## Best Possible Solution

Add a documented additive original/client run lineage field while preserving execution runId, then cover chat.send, collect/follow-up drain, gateway projection, protocol validation, and docs with focused tests.

## Reproduction Assessment

Yes. Source inspection shows chat.send returns one client run ID, collect/follow-up drain can execute later under a fresh UUID, and public gateway agent events expose no original run ID set.

## Solution Assessment

Unclear until maintainers choose the public event shape. An additive lineage field is likely safer than selecting the first or last queued run ID because it preserves execution identity while exposing client correlation IDs.
