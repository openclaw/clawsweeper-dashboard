## Summary

Keep open: current main and the latest release still do not expose original queued chat run IDs on gateway agent events, and the remaining work needs an additive Gateway protocol decision rather than conservative cleanup closure.

## What This Changes

Expose original queued chat run IDs when queued or merged messages later execute under a different agent run ID.

## Best Possible Solution

Add an additive ordered queued-run lineage contract through FollowupRun metadata, AgentRunContext or AgentEventPayload, Gateway AgentEventSchema, docs, and focused tests while preserving current execution runId semantics.

## Reproduction Assessment

Yes, by source inspection. chat.send returns the idempotency-derived client runId, queued followups can later execute under a fresh UUID, and current agent event schemas do not expose original queued run IDs.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive lineage field is likely the maintainable path; changing runId to a first or last queued ID would be compatibility-sensitive for clients that already treat runId as the execution/event identifier.
