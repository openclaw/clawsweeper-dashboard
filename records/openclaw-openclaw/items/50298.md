## Summary

Keep open. Current main still acknowledges a `chat.send` client run ID, but queued/collected followups can execute under a fresh run ID and the public agent event contract has no field for the original queued run ID set; the closed #50296 thread points here, so this remains the canonical report.

## What This Changes

Expose the original queued `chat.send` run ID set on WebSocket agent events when queued messages are merged into a later execution.

## Best Possible Solution

Add an additive, stable, deduplicated queued-run lineage field to agent events while preserving the current primary `runId` as the actual execution identity.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to send `chat.send` while the session is busy, let collect/followup drain process the queued item, and observe that the followup runner creates a fresh execution `runId` while the event schema cannot carry the original queued client run IDs.

## Solution Assessment

Yes. The additive lineage direction is the narrowest maintainable solution; reusing the first or last queued run ID would lose batch membership and blur the distinction between client acknowledgement IDs and execution IDs.
