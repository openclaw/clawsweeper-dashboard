## Label Justifications

- `P2`: The issue affects external integrations that map Gateway replies to original IM questions, but the fix is a bounded protocol/API improvement rather than an emergency outage.
- `impact:message-loss`: The reported failure mode is reply/question miscorrelation when collect-mode processing merges queued chat messages.

## Summary

Keep open: current main and v2026.5.22 still do not expose original queued chat.send run IDs on Gateway agent events, and the remaining work is an additive protocol/API design choice rather than conservative cleanup.

## What This Changes

Request adding Gateway event correlation for the original chat.send run IDs when queued messages are merged into a later execution run.

## Best Possible Solution

Add an additive Gateway/SDK lineage field that carries all queued chat.send run IDs through collect/follow-up execution while preserving the actual execution runId semantics.

## Reproduction Assessment

Yes, at source level: collect mode can merge queued messages into one follow-up execution run, that run gets a fresh runId, and the Gateway agent event schema/projection lacks original run-id lineage. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive lineage field is safer than selecting the first or last queued runId because it preserves execution identity while giving clients correlation data.
