## Label Justifications

- `P2`: The issue affects external integrations that map Gateway replies to original IM questions, but the fix is a bounded protocol/API improvement rather than an emergency runtime outage.
- `impact:message-loss`: The reported failure mode is reply/question miscorrelation when collect-mode processing merges queued chat messages.

## Summary

Keep open: current main and v2026.5.22 still do not expose the original queued chat.send run IDs on Gateway agent events, and the remaining work is an additive Gateway protocol/API decision rather than conservative cleanup.

## What This Changes

The issue requests that merged or collect-processed chat messages carry the original chat.send run ID set on the actual execution agent event.

## Best Possible Solution

Add an additive Gateway/SDK lineage field that carries all queued chat.send run IDs through collect/follow-up execution while preserving the actual execution runId semantics.

## Reproduction Assessment

Yes, at source level: chat.send returns a clientRunId, collect follow-up execution creates a fresh runId, and the agent broadcaster only rewrites runId when the registry already has the execution runId. I did not run a live Gateway repro because this review was read-only.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive lineage field is safer than selecting the first or last queued runId, because it preserves execution identity while giving clients correlation data.
