## Summary

Keep open. Current main still has no public agent-event field for original queued chat.send run IDs after collect/followup merging, and the closed duplicate at https://github.com/openclaw/openclaw/issues/50296 points here as the canonical remaining request.

## What This Changes

Add original queued run ID lineage to websocket agent events when queued messages are merged into a later agent execution.

## Best Possible Solution

Add an additive, documented, stable, deduplicated queued-run lineage array to agent events while keeping runId as the actual execution identity.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to send chat.send while a session is busy, let collect/followup drain merge queued items, and observe that the later execution uses a fresh runId while the event contract cannot carry the original client run IDs.

## Solution Assessment

Yes. An additive lineage field is the narrowest maintainable direction; reusing the first or last queued runId would lose batch membership and conflate client acknowledgement IDs with execution identity.
