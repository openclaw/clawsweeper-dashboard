## Summary

Keep open: current main still creates a separate queued follow-up execution run and the gateway agent event schema has no field for the original queued chat run ids, so this is real remaining protocol/API work rather than cleanup-closeable stale noise.

## What This Changes

The issue requests that merged queued-message executions expose the original chat.send runId values in websocket agent events, or otherwise preserve a stable original-message-to-executed-run mapping.

## Best Possible Solution

Add an explicit, additive run-lineage contract for queued/collect executions, carry it from FollowupRun through AgentRunContext/AgentEventPayload, update gateway protocol schema and generated clients, and cover normal, collect, and split-batch queue cases with tests.

## Reproduction Assessment

Yes by source inspection, not by a live run: chat.send returns a client runId, collect/follow-up queue draining creates a fresh execution runId, and the current agent event schema cannot expose the original queued ids. A live repro would send a second collect/follow-up message while a session run is active and observe the later agent event runId.

## Solution Assessment

Unclear until maintainers choose the protocol shape. An additive sourceRunIds/originalRunIds field looks safer than changing runId to the first or last queued id, because existing clients already treat runId as the execution/chat event identifier.
