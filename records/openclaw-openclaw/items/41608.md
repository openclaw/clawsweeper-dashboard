## Summary

Keep open. Current main still passes raw Slack channel/user IDs into generic route matching while configured binding peer IDs are only trim-normalized and exact-matched, so target-style Slack IDs such as `channel:C...` can miss and fall back to the default `main` agent. PR #42162 directly targeted this but is closed and unmerged, leaving a narrow repair still valid.

## What This Changes

Slack inbound channel messages should construct session keys with the agent matched by Slack route bindings instead of falling back to `agent:main` when the binding uses target-style peer IDs.

## Best Possible Solution

Add Slack-owned route-binding peer canonicalization, or an additive generic hook that Slack owns, so Slack bindings accept raw IDs and target-style forms such as `channel:C123`, `user:U123`, and `<@U123>` before generic matching while keeping canonical raw IDs in session keys. Cover channel, group/channel compatibility, and direct-message cases with focused tests plus config help or warnings if raw Slack IDs remain the preferred documented form.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a Slack binding with `peer.id: "channel:C123"` and route an inbound room event whose runtime peer is raw `C123`; current normalization keeps those strings different and exact peer matching falls through to the default agent.

## Solution Assessment

Yes, the narrow maintainable path is to normalize Slack binding peer IDs at the Slack boundary, or through a generic plugin-provided route-normalization seam, before invoking exact route matching. Changing generic core matching to understand Slack target syntax directly would be broader and less compatible with third-party channels.
