## Summary

Keep open: current main has adjacent stale-reply and wedged-lane recovery fixes, but observed reply delivery is still not the lifecycle point that releases the dispatch-owned reply operation or the embedded session lane during failover cleanup.

## What This Changes

The issue reports same-session lane occupancy after a visible reply is delivered during provider failover, causing later inbound messages to stall.

## Best Possible Solution

Define the post-delivery lifecycle contract, then add a narrow handoff that releases visible same-session admission after durable delivery while preserving failover cleanup, transcript state, and queued follow-up ordering.

## Reproduction Assessment

Yes, source-reproducible: current main records observed delivery without using it as a reply-operation or session-lane release point, and command-lane tasks remain serialized until task completion or timeout. I did not run a live provider-outage reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers decide the lifecycle contract; the existing stale-reply and stuck-lane mitigations are useful, but the likely fix belongs at the dispatch-owned reply-operation handoff rather than only diagnostic recovery.
