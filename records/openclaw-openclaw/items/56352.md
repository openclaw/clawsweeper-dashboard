## Summary

Keep this open. Current main still couples Telegram delivery to the awaited channel turn, model-fallback execution, reply finalization, and the per-session command lane; I did not find a post-visible-delivery boundary that would release same-session follow-ups before cleanup settles.

## What This Changes

The issue requests releasing a Telegram same-session lane after a successful reply is delivered during failover, instead of keeping follow-up messages blocked until fallback cleanup or timeout completes.

## Best Possible Solution

Define an explicit post-delivery lifecycle boundary for channel turns: keep the user-visible reply path responsive, but move only the cleanup that is proven safe into bounded or detached finalization while preserving transcript consistency, fallback selection persistence, compaction accounting, dispatcher idle tracking, and follow-up steering semantics.

## Reproduction Assessment

Unclear. The production report is detailed and current source confirms same-session serialization until the awaited run settles, but I did not find a deterministic Telegram/failover regression test that reproduces visible delivery followed by delayed cleanup and a blocked follow-up.

## Solution Assessment

No. Releasing the session lane immediately after the first visible reply is not proven to be the safest fix; a bounded finalization or explicit cleanup-lane design is the narrower maintainable direction because the remaining code still mutates session, transcript, and fallback state.
