## Label Justifications

- `P1`: The report describes a real Telegram workflow where provider failover can leave same-session follow-up messages blocked for minutes despite a visible prior reply.
- `impact:message-loss`: Follow-up inbound messages can receive only status feedback or sit queued without an agent response while the occupied lane drains.
- `impact:session-state`: The failure is centered on when a session lane, reply-run state, and turn finalization are considered idle.
- `impact:auth-provider`: The blocking window is triggered by provider overload, auth-profile rotation, and fallback escalation behavior.

## Summary

Keep open. Current main still appears to serialize same-session inbound work through the embedded run and channel turn lifecycle, so the requested release-after-visible-reply behavior is not implemented and the issue has recent ClawSweeper-maintainer triage labels indicating product/design review is still needed.

## What This Changes

The issue asks for Telegram same-session lanes to become available again once a reply has been visibly delivered during provider failover, instead of staying occupied until failover cleanup or timeout completes.

## Best Possible Solution

Add an explicit post-visible-delivery boundary for interactive channel replies that releases same-session intake only after the reply is safely delivered, while preserving transcript/finalization ordering and keeping timeout/provider escalation policy in the relevant fallback configuration work.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection and the prior review still show the same-session lane wraps the awaited embedded run and Telegram turn dispatch/finalization path, so the requested release point is absent on current main.

## Solution Assessment

Unclear as an immediate implementation, because releasing the lane at first visible reply may break transcript and finalization invariants. The better fix is an explicit, tested post-delivery boundary with maintainer approval for ordering and fallback semantics.
