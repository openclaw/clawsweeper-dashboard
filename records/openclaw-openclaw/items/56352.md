## Summary

Keep open: current main and the latest release still serialize visible turns behind the active reply/run lifecycle, while recovery is delayed stale-work handling rather than an immediate release after channel delivery.

## What This Changes

The issue reports that a Telegram session lane stays occupied after a failover reply is delivered, delaying later inbound messages until cleanup or timeout finishes.

## Best Possible Solution

Define a durable reply/lane lifecycle contract that allows later inbound turns after safe visible delivery while preserving transcript, failover, and terminal run state correctness.

## Reproduction Assessment

Yes from source for the blocking shape: visible turns wait for an active reply operation, and the embedded run owns the session lane until its task completes. No live Telegram/provider-outage reproduction was run in this read-only review.

## Solution Assessment

No, the literal release-on-any-reply rule is not yet proven as the best fix. The safer direction is a maintainer-owned lifecycle contract that preserves transcript and failover consistency while preventing stale cleanup from blocking later turns.
