## Summary

Keep this open. Current main still serializes the same session around the awaited embedded run and channel dispatch lifecycle; steering and stalled-session recovery reduce some impact, but they do not add the requested post-visible-delivery lane boundary for Telegram failover cleanup.

## What This Changes

The issue asks OpenClaw to release a same-session lane after a Telegram reply is visibly delivered during provider failover, plus consider faster per-provider/model timeout and provider-escalation behavior.

## Best Possible Solution

Add an explicit post-visible-delivery/finalization boundary that keeps Telegram follow-up input responsive while preserving session serialization for transcript, fallback, compaction, and delivery cleanup invariants.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection shows the same-session lane and Telegram dispatch remain coupled until awaited run/dispatch completion, but I did not reproduce the outage-style Telegram sequence against current main.

## Solution Assessment

No, not as the proposed immediate lane release at first visible reply. The maintainable path needs an explicit post-delivery/finalization boundary rather than dropping the lane while cleanup, fallback state, transcript updates, and Telegram finalization may still be in flight.
