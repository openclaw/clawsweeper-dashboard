## Summary

Keep this open. Current main has stronger provider fallback/cooldown handling, but the same-session lane is still held around the awaited embedded run/dispatch lifecycle while Telegram can already have emitted visible block or final output, so the central post-visible-delivery blocking problem is not resolved.

## What This Changes

The issue asks OpenClaw to stop blocking subsequent Telegram inbound messages after a visible reply has been emitted during model failover, with additional faster timeout/provider-escalation behavior.

## Best Possible Solution

Add a supported post-visible-delivery/finalization boundary that keeps Telegram follow-up input responsive while preserving session serialization for transcript, fallback, compaction, and delivery cleanup invariants.

## Reproduction Assessment

No, not as a high-confidence live reproduction. The source path shows visible Telegram/block delivery can occur inside an awaited embedded failover run while the per-session lane remains active, but I did not reproduce the outage-style Telegram sequence against current main.

## Solution Assessment

No, not as the proposed immediate lane release at first visible reply. The maintainable fix needs an explicit post-delivery/finalization boundary rather than dropping the lane while cleanup, fallback state, transcript updates, and Telegram finalization may still be in flight.
