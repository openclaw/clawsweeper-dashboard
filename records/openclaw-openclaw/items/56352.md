## Summary

Keep open. Current main has adjacent Telegram reply-admission and stale-lane recovery work, but the central request is still not implemented: embedded agent execution still holds the per-session lane around the full run/failover lifecycle, and recovery only releases lanes after timeout/stuck-session gates rather than immediately after a visible reply is delivered.

## What This Changes

The issue reports that after a Telegram reply is delivered during provider failover, the session lane remains occupied until failover cleanup or timeout completes, delaying later inbound messages.

## Best Possible Solution

Define a maintainer-approved reply/lane lifecycle boundary that releases or aborts session ownership only after durable delivery and terminal transcript state are safe, then prove it with targeted tests plus live Telegram/provider-failover evidence.

## Reproduction Assessment

No high-confidence live reproduction was run. The issue has detailed incident evidence and current source shows the plausible blocking path, but the exact Telegram failover outage sequence still needs live or synthetic current-main proof.

## Solution Assessment

No, the literal proposed release-on-reply rule is not yet proven as the best fix. The safer direction is a maintainer-owned lifecycle contract that preserves transcript/fallback consistency while preventing queued inbound messages from waiting on stale cleanup.
