## Summary

Keep this open. Current main has improved fallback, timeout, and stuck-lane recovery paths, but the Telegram reply path still awaits the full inbound dispatch/run lifecycle and the command lane still frees the session only when that queued task settles, so the requested post-visible-delivery release boundary is not implemented.

## What This Changes

The issue requests releasing a Telegram/session lane after a visible reply is delivered during failover so subsequent inbound messages are not held until the whole run settles.

## Best Possible Solution

Add an explicit post-visible-delivery finalization boundary that keeps transcript/fallback cleanup bounded while allowing later same-session input to steer, enqueue, or drain responsively.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection shows the same-session lane remains tied to the awaited queued run and dispatch lifecycle, but I did not run a deterministic Telegram failover test for the exact incident interleaving.

## Solution Assessment

No. Releasing the session lane immediately after the first visible reply is not proven to be the safest fix; the maintainable direction is a designed post-delivery/finalization boundary with regression coverage across Telegram, shared channel turns, and reply-run cleanup.
