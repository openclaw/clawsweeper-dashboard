## Summary

Keep this open. Current main has improved overloaded cross-provider fallback and followup-drain cleanup, but Telegram visible delivery still sits inside the awaited channel turn and the per-session command lane still releases only when the queued run settles, so the central post-visible-delivery lane-release problem remains unresolved.

## What This Changes

Requested behavior is to release a Telegram same-session lane once a failover reply is visibly delivered, with optional shorter per-provider/model timeouts and faster provider escalation.

## Best Possible Solution

Add a supported post-delivery lifecycle boundary that lets follow-up input steer or enqueue responsively after visible delivery while keeping session, transcript, fallback, and compaction cleanup bounded and consistent.

## Reproduction Assessment

No, not as a high-confidence live reproduction. The production report is detailed and current source shows same-session work remains serialized until the awaited run and dispatch settle, but I did not establish a deterministic Telegram failover test that emits a visible reply and then delays cleanup.

## Solution Assessment

No. Releasing the session lane immediately after the first visible reply is not yet proven to be the safest fix; the narrower maintainable path is an explicit post-delivery/finalization boundary with bounded cleanup and preserved session state.
