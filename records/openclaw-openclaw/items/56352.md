## Summary

Keep open: current main and v2026.6.10 still keep the embedded session lane tied to task completion/timeout, while observed reply delivery is only recorded and does not release same-session admission during failover cleanup.

## What This Changes

The issue reports that Telegram follow-up messages stay blocked after a failover reply is delivered because the session lane remains occupied until failover cleanup or timeout finishes.

## Best Possible Solution

Define a maintainer-approved visible-delivery handoff that admits same-session follow-ups after durable channel delivery while preserving transcript, cleanup, failover, and Telegram ordering semantics.

## Reproduction Assessment

Yes at source level, not as a live run: current main records observed reply delivery but the embedded session lane is still held until the queued task settles or times out.

## Solution Assessment

Unclear as an implementation direction: releasing after visible delivery is the right goal, but the exact handoff needs maintainer choice because cleanup and transcript ownership may still be active.
