## Label Justifications

- `P1`: The report describes a user-facing Telegram workflow where provider failover can block same-session follow-up messages for minutes after a visible reply.
- `impact:message-loss`: Follow-up inbound messages can receive only status feedback or remain queued without an agent response while the lane is occupied.
- `impact:session-state`: The failure depends on when the session lane, reply fence, transcript persistence, and cleanup state are considered idle.
- `impact:auth-provider`: The blocking window is triggered by provider overload, auth-profile rotation, fallback escalation, and model timeout behavior.

## Summary

Keep open. Current main still ties same-session occupancy to the embedded run and Telegram dispatch task settling, not to the first durable visible reply, and the related open Telegram PR addresses overlapping DM delivery rather than this failover cleanup boundary.

## What This Changes

Release the Telegram/interactive session lane after a visible reply is delivered during provider failover so same-user follow-up messages are not blocked by remaining cleanup, retries, or timeouts.

## Best Possible Solution

Define and test a post-durable-visible-delivery boundary for interactive replies that allows same-session intake to proceed after confirmed delivery while preserving transcript, cleanup, reply-fence, and failover-state ordering.

## Reproduction Assessment

No live high-confidence current-main reproduction was established. Source inspection still shows the same-session lane and Telegram dispatch task remain active until the embedded run and cleanup settle, so the requested release boundary is absent at source level.

## Solution Assessment

Unclear as an immediate patch. The best direction is an explicit durable-delivery release boundary rather than opportunistically freeing the lane at the first final payload or changing provider timeout policy alone.
