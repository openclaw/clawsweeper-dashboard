## Summary

Keep open: current main has adjacent stale-reply and wedged-lane recovery mitigations, but observed reply delivery still is not the lifecycle point that releases the dispatch-owned reply operation or embedded session lane during failover cleanup.

## What This Changes

The issue reports that a Telegram session lane remains occupied after a reply is delivered during provider failover, blocking later inbound messages until failover cleanup or timeout completes.

## Best Possible Solution

Define the durable visible-delivery handoff contract, then release same-session admission after that handoff while preserving failover cleanup, transcript consistency, delivery ordering, and Telegram-visible behavior.

## Reproduction Assessment

Source-reproducible at medium confidence: current main records observed delivery without using it as a reply-operation or embedded session-lane release point. I did not run a live provider-outage or Telegram reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers decide the lifecycle contract; the existing stale-reply and stuck-lane mitigations are useful, but the remaining fix likely belongs at the dispatch-owned reply-operation handoff rather than diagnostic recovery only.
