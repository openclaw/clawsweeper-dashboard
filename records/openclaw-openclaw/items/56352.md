## Label Justifications

- `P1`: The report describes a broken real user Telegram workflow where same-session follow-up messages can receive only status feedback for minutes during failover cleanup.
- `impact:message-loss`: Follow-up inbound messages in the same Telegram session can be queued without an agent response while the active lane remains occupied.
- `impact:session-state`: The reported failure centers on when a session lane and reply-run state are considered idle after visible delivery.
- `impact:auth-provider`: The blocking window is triggered by provider overload, auth-profile rotation, and fallback escalation behavior.

## Summary

Keep open. Current main still couples the same-session lane to the awaited embedded run and Telegram dispatch/finalization path; recent Telegram follow-up/progress changes mitigate adjacent problems but do not add the requested post-visible-delivery lane boundary.

## What This Changes

The issue requests that a Telegram session lane stop blocking new inbound messages after a successful visible reply during provider failover.

## Best Possible Solution

Add an explicit post-visible-delivery/finalization boundary for inbound replies that keeps Telegram follow-up input responsive while preserving session serialization for transcript, fallback, compaction, and delivery cleanup invariants; leave timeout-map policy to the separate per-channel timeout item.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection still shows the same-session lane wraps the awaited embedded run and Telegram turn dispatch through finalization, so the requested release point is not present on current main.

## Solution Assessment

No, not as an immediate lane release on first visible reply. The maintainable solution needs an explicit post-delivery/finalization boundary, and the requested per-channel/per-provider timeout configuration is better handled through the existing timeout-config discussion.
