## Summary

Keep open. Current main has a newer Telegram reply-admission fix, but the session lane still stays occupied until the embedded run and its cleanup settle; the requested durable-reply release boundary is still not present.

## What This Changes

The issue asks OpenClaw to release a Telegram session lane once a failover-produced reply has been visibly delivered, instead of blocking later inbound messages until failover, timeout, and cleanup finish.

## Best Possible Solution

Define a durable visible-delivery boundary for interactive replies, then let same-session intake proceed after confirmed delivery while preserving transcript, reply-fence, failover, and cleanup ordering.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the requested release point is absent: the session lane and reply operation still settle at embedded-run completion, not at first durable visible delivery.

## Solution Assessment

Unclear as an immediate patch. The durable-delivery release boundary is the right direction, but maintainers need to define exactly what cleanup and transcript work may continue after the lane admits the next turn.
