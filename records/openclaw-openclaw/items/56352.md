## Summary

Keep open: current main has adjacent stale-reply and wedged-lane recovery plus documented provider timeout configuration, but it still admits visible same-session turns by waiting behind active reply ownership and does not implement the requested immediate post-delivery lane release while failover cleanup continues.

## What This Changes

The issue requests that a lane be released once a reply has been emitted to the channel so later inbound messages are not blocked by failover or cleanup work after delivery.

## Best Possible Solution

Define the reply lifecycle contract first, then implement a narrow post-delivery ownership handoff that preserves final delivery, transcript durability, failover cleanup, and queued follow-up ordering while reusing the existing provider timeout knob.

## Reproduction Assessment

Yes, source-reproducible: current admission and tests show visible same-session turns wait behind active reply ownership, while the recovery code only releases lanes after watchdog-style stale recovery; no live provider outage was run in this read-only review.

## Solution Assessment

Unclear until maintainers decide lifecycle semantics; the existing stale and stuck-session recovery fixes are useful mitigations, but the likely best fix belongs at reply-operation or delivery ownership rather than only in diagnostic recovery.
