## Summary

Keep open. Current main still couples Telegram visible delivery to the awaited channel turn and session lane; diagnostics and recovery paths reduce some stuck-lane cases, but they do not add the requested post-visible-delivery lane boundary.

## What This Changes

The issue reports that Telegram follow-up messages can be blocked while a failover turn keeps the same session lane occupied after a visible reply was already delivered.

## Best Possible Solution

Design an explicit post-visible-delivery/finalization boundary that keeps Telegram follow-up input responsive while preserving session serialization for transcript, fallback, compaction, diagnostics, and delivery cleanup invariants.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection does show the same-session lane and Telegram dispatch remain coupled until the awaited turn/dispatch completes, but the outage-style Telegram failover sequence was not reproduced against current main.

## Solution Assessment

No, not as an immediate lane release at first visible reply. The safer maintainable solution is an explicit finalization boundary that keeps follow-up input responsive without breaking transcript, fallback, compaction, and Telegram delivery invariants.
