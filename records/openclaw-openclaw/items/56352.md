## Summary

Keep open. Current main includes the adjacent Telegram reply-admission work from https://github.com/openclaw/openclaw/pull/85709, but the central failover/session-lane release boundary requested here is still not present: source still holds the per-session queue for the full embedded run, and a visible block reply can be flushed before later cleanup without freeing that lane.

## What This Changes

The issue requests that OpenClaw release or re-admit the same-session lane after a visible Telegram reply is durably emitted during provider failover instead of blocking follow-up messages until failover cleanup or timeout completes.

## Best Possible Solution

Define a durable visible-delivery boundary for interactive replies, then let same-session intake proceed after confirmed delivery while preserving transcript order, reply-fence ownership, failover attribution, and cleanup guarantees.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the requested release point is absent: the session lane remains active until the embedded run promise settles, while block replies can flush before later cleanup.

## Solution Assessment

Unclear as an immediate patch. The durable-delivery release boundary is likely the right direction, but maintainers need to decide exactly which transcript, failover, and cleanup work may continue after the lane admits the next turn.
