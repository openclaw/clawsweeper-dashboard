## Summary

Keep open: current main has adjacent reply-operation, stale-recovery, and wedged-lane mitigations, but the embedded session lane is still task-scoped and there is no proven post-delivery release contract for failover cleanup.

## What This Changes

The issue requests releasing the session lane after a successful visible Telegram reply during model failover so later inbound messages from the same user can run promptly.

## Best Possible Solution

Define the durable visible-delivery handoff contract, then release same-session admission after that handoff while preserving failover cleanup, transcript consistency, delivery ordering, and Telegram-visible behavior.

## Reproduction Assessment

Yes at source level: current main records observed delivery but does not use it to release the embedded session lane, and the command queue keeps lane work active until task completion or timeout. I did not run a live Telegram/provider-outage repro in this read-only review.

## Solution Assessment

Unclear as an implementation direction: immediate release after visible delivery is the right user-facing goal, but maintainers still need to choose the lifecycle handoff that preserves failover cleanup and transcript correctness.
