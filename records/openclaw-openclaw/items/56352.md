## Summary

Keep open: current main still holds the same-session command lane until the embedded task completes or timeout recovery runs; observed reply delivery is recorded in the reply path but does not release the command-queue session lane for follow-up messages.

## What This Changes

The issue requests that failover cleanup stop occupying the session lane after a reply has already been emitted, so subsequent Telegram messages are not blocked until model timeout.

## Best Possible Solution

Define a maintainer-approved handoff where durable visible delivery can admit later visible turns while failover cleanup continues under explicit transcript, ordering, and timeout-recovery invariants, then prove it with focused source tests and a live Telegram/provider-failover scenario.

## Reproduction Assessment

Yes at source level, but not live: current main records observed delivery while the command-queue session lane still remains held until the embedded task settles or timeout recovery runs.

## Solution Assessment

Unclear: releasing on visible delivery is directionally right, but the exact fix needs a maintainer decision on ordering, transcript durability, failover cleanup, and Telegram live proof.
