## Summary

Keep open: current main and v2026.6.8 still keep same-session visible turns behind active reply/run ownership until completion or delayed recovery, so the reported post-delivery lane occupancy is not implemented away.

## What This Changes

The issue asks OpenClaw to release or unblock a same-session lane after a Telegram-visible reply is emitted during provider failover, instead of blocking later inbound messages until failover or timeout cleanup finishes.

## Best Possible Solution

Define and test a durable reply/lane lifecycle contract that admits later inbound turns after safe visible delivery while preserving transcript, failover, terminal-run, and channel-delivery consistency.

## Reproduction Assessment

Yes from source: visible turns wait behind active reply operations, and embedded runs own the same-session lane until completion or recovery. No live Telegram/provider outage reproduction was run in this read-only review.

## Solution Assessment

No: releasing the lane on any visible reply is plausible but not yet proven as the safest fix. The better path is a maintainer-owned lifecycle contract with regression coverage for reply delivery, failover cleanup, transcript durability, and queued inbound ordering.
