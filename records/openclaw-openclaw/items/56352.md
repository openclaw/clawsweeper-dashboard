## Label Justifications

- `P1`: The report describes a real agent/channel workflow where follow-up Telegram messages can be blocked during provider failover after the user has already seen a reply.
- `impact:message-loss`: Follow-up inbound messages are delayed or effectively suppressed behind the occupied session lane, with only status feedback visible to the user.
- `impact:session-state`: The core failure mode is session-lane ownership and active-run state remaining occupied after visible reply delivery.
- `impact:auth-provider`: The reported trigger is provider/auth-profile failover during overloaded Anthropic and OpenAI-style model paths.

## Summary

Keep open: current main still serializes the session lane around the whole embedded run, while visible block delivery and stale recovery are later/gated paths rather than an immediate delivered-reply release boundary. Related failover and timeout behavior has improved, but the central lane lifecycle request is still unresolved and needs maintainer design.

## What This Changes

The issue requests that Telegram/session lanes release after a successful visible reply during provider failover, with tighter provider/model timeout and fallback escalation behavior.

## Best Possible Solution

Define a durable shared reply/lane lifecycle contract that lets follow-up inbound turns drain after safe visible delivery without corrupting transcript, fallback, terminal run, or Telegram delivery state.

## Reproduction Assessment

No high-confidence live reproduction was run. Current source shows the plausible blocking path, but the exact Telegram failover outage sequence still needs live or synthetic current-main proof.

## Solution Assessment

No, the literal release-on-any-reply rule is not yet proven as the best fix. The safer direction is a maintainer-owned lifecycle contract that preserves transcript and failover consistency while preventing stale cleanup from blocking later turns.
