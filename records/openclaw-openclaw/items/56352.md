## Summary

Keep this open. Current main still holds the same-session lane until the embedded run and channel dispatch promise settle; Telegram visible delivery happens inside that awaited path, so the post-visible-delivery blocking problem is not implemented as a supported lifecycle boundary.

## What This Changes

Release the same-session lane after a Telegram-visible reply is delivered during failover so later inbound messages are not blocked by remaining cleanup, retries, or timeouts.

## Best Possible Solution

Define a supported post-visible-delivery finalization boundary that lets Telegram follow-up input proceed after durable visible output while preserving transcript, fallback, compaction, and dispatch cleanup invariants.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection shows Telegram visible delivery can occur inside an awaited embedded run while the per-session lane remains active, but I did not reproduce the outage-style Telegram failover sequence against current main.

## Solution Assessment

No, not as the proposed immediate lane release at first visible reply. The maintainable fix needs an explicit post-delivery/finalization boundary rather than dropping the session lane while cleanup, transcript updates, fallback state, and Telegram finalization may still be in flight.
