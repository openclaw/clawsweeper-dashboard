## Summary

Keep open: current main still holds the per-session lane around the full embedded run, while visible reply delivery and stale recovery are later/gated paths rather than an immediate delivered-reply release boundary.

## What This Changes

The issue requests that Telegram/provider-failover replies stop blocking later inbound messages once a visible reply has been delivered, with additional faster timeout/fallback controls.

## Best Possible Solution

Define a durable reply/lane lifecycle contract that lets later inbound turns drain after safe visible delivery without corrupting transcript, fallback, or terminal run state.

## Reproduction Assessment

No high-confidence live reproduction was run. Current source shows the plausible blocking path, but the exact Telegram failover outage sequence still needs live or synthetic current-main proof.

## Solution Assessment

No, the literal release-on-any-reply rule is not yet proven as the best fix. The safer direction is a maintainer-owned lifecycle contract that preserves transcript and failover consistency while preventing stale cleanup from blocking later turns.
