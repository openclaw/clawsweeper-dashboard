## Summary

Keep this open. Current main still keeps Telegram-visible delivery inside the awaited channel turn, fallback execution, reply finalization, and per-session command lane; I did not find a supported post-visible-delivery boundary that would let the next same-session inbound turn start before cleanup settles.

## What This Changes

The issue requests releasing a Telegram same-session lane after a successful fallback reply is delivered, instead of blocking later inbound messages until fallback cleanup or timeout completion.

## Best Possible Solution

Define a supported post-delivery lifecycle boundary for channel turns: keep the visible reply path responsive, but move only proven-safe cleanup into bounded or detached finalization while preserving transcript consistency, fallback selection persistence, compaction accounting, dispatcher idle tracking, and follow-up steering semantics.

## Reproduction Assessment

Unclear. The production report is detailed and current source confirms same-session serialization until the awaited run settles, but I did not find a deterministic Telegram/failover test that reproduces visible delivery followed by delayed cleanup and a blocked follow-up.

## Solution Assessment

No. Releasing the session lane immediately after the first visible reply is not proven to be the safest fix; a bounded finalization or explicit cleanup-lane design is the narrower maintainable direction because remaining code still mutates session, transcript, and fallback state.
