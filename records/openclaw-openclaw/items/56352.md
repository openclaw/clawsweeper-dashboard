## Summary

Keep open. Current main and v2026.6.6 still keep the per-session lane around the full embedded run and only steer follow-up messages while the active backend is streaming, so the reported non-streaming failover/cleanup window is not solved; related open issues and PRs cover adjacent steering, fairness, and idle-drain races but do not fully replace this Telegram/provider-failover report.

## What This Changes

The issue asks OpenClaw to stop blocking same-session inbound messages after a visible Telegram reply has already been delivered during provider failover cleanup, with additional timeout and provider-escalation improvements requested.

## Best Possible Solution

Define a maintainer-approved lifecycle boundary for delivered replies, active non-streaming failover cleanup, and session ownership, then prove it with targeted queue/steer tests plus live Telegram/provider-failover evidence.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. The current source makes the failure path plausible: active non-streaming runs cannot be steered and same-session work can remain queued until the active run/failover task completes.

## Solution Assessment

No, the literal release-on-any-visible-reply rule is not yet proven as the best fix. A safer solution needs a maintainer-owned lifecycle contract that preserves transcript and fallback consistency while preventing new inbound messages from waiting on stale cleanup.
