Keep open: current main and v2026.5.12 still have no code-fence lint/write-time Markdown rewrite or config surface. The linked implementation PRs are closed unmerged after parser correctness and write-time mutation concerns, so this remains a feature/product decision rather than cleanup.

## What This Changes

Request adding write-time linting and auto-fix for Markdown code fences in workspace files, with language inference, warn mode, and per-write skip support.

## Best Possible Solution

Keep workspace writes verbatim unless maintainers approve a bounded Markdown rewrite policy; if accepted, implement parser-backed linting with opt-in or warn-first semantics, explicit skips, schema/docs, and regression coverage for nested fence examples.

## Reproduction Assessment

No; reproduction is not applicable because this is a request for new write-time Markdown rewriting/config behavior. Source inspection shows no existing code-fence lint contract whose current behavior can be reproduced as broken.

## Solution Assessment

Unclear; a write-path helper could address the request, but auto-fix-by-default and heuristic language inference require maintainer approval because they mutate user-authored Markdown. A safer direction may be opt-in or warn-first behavior backed by a real Markdown parser.
