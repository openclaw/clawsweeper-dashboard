## Summary

Keep open: the complete-line cap is useful but not merge-ready because it only checks newline-delimited lines; the buffered tail and EOF final frame still bypass the new guard, and the fail-closed 1 MiB proxy event boundary still needs maintainer acceptance.

## What This Changes

The PR adds a 1 MiB byte-length guard for completed proxy SSE lines in `streamProxy` and tests oversized plus coalesced newline-delimited lines.

## Best Possible Solution

Apply one byte-accurate proxy SSE size policy to complete lines, the buffered tail, and the EOF final frame, then coordinate with the total-body guard and the maintainer-accepted cap value.

## Reproduction Assessment

Yes, by source inspection. After this patch, an oversized final SSE frame without a trailing newline still reaches `processSseLine(buffer)` with no byte check, and an open no-newline tail can keep growing before any complete-line cap runs.

## Solution Assessment

No. The complete-line check is a plausible mitigation, but the best fix must guard the shared line/tail drain path or coordinate with the total-byte guard so all proxy SSE buffering paths are bounded.
