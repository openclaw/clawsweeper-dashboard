## Summary

Keep open: the reported Tavily SecretRef failure is real from source, but this patch is too broad because it lets any unresolved configured SecretRef fall through to ambient `TAVILY_API_KEY`, which can silently use the wrong credential source.

## What This Changes

The PR changes Tavily API-key resolution to catch strict SecretRef resolution failures and then use the existing `TAVILY_API_KEY` environment fallback, with a new focused unit test.

## Best Possible Solution

Keep the Tavily env fallback, but implement it with SecretRef-aware guardrails so explicit file, exec, or non-matching env refs do not silently fall through to ambient credentials.

## Reproduction Assessment

Yes, from source inspection: current main strictly resolves the configured Tavily SecretRef before evaluating the existing `TAVILY_API_KEY` fallback, so an unresolved configured ref throws first. I did not run the CLI because this review is read-only.

## Solution Assessment

No; the PR is a plausible mitigation but not the best fix because it catches all configured-ref failures. The better fix is inspect-mode or typed-error handling that only falls back for the supported env ref and preserves failures for other configured refs.
