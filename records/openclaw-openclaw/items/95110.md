## Summary

Keep this PR open, but do not merge it as-is: it fixes a real Tavily SecretRef/env fallback bug, but the catch-all fallback can silently swap an explicit configured credential source for ambient `TAVILY_API_KEY`.

## What This Changes

The PR changes Tavily API-key resolution to catch strict SecretRef normalization failures and adds a focused unit test for falling back to `process.env.TAVILY_API_KEY`.

## Best Possible Solution

Keep the Tavily env fallback, but implement it with inspect-mode SecretRef guardrails so only the allowed matching env ref falls through and other configured refs remain blocked.

## Reproduction Assessment

Yes, from source inspection: current main strictly resolves the configured Tavily SecretRef before evaluating the existing `TAVILY_API_KEY` fallback, so an unresolved configured ref can throw first. I did not run the CLI because this review is read-only.

## Solution Assessment

No; this is a plausible mitigation but not the best fix because it catches all configured-ref failures. The safer fix is inspect-mode or typed-error handling that only falls back for the supported Tavily env ref and preserves failures for other configured refs.
