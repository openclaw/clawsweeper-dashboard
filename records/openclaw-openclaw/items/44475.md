## Summary

Keep open: current main still does not implement this config shortcut, and this PR is not merge-ready because its passthrough path bypasses current model-ref normalization, blurs runtime-policy boundaries, weakens validation for known model metadata, and lacks contributor real behavior proof.

## What This Changes

The PR widens `agents.defaults.models` entries to accept passthrough fields, merges those fields into provider model definitions in `applyModelDefaults()`, broadens the type, and adds config-default tests.

## Best Possible Solution

Land a maintainer-approved config contract that adds concise per-model metadata overrides without weakening normalization, validation, or runtime-policy boundaries.

## Reproduction Assessment

Yes for the PR review defects by source inspection: the PR's raw lookup and known-key filter differ from current-main normalization and runtime-policy contracts, and `.passthrough()` bypasses provider metadata validation. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction may be useful, but the patch should normalize lookup keys, keep runtime policy scoped, validate known metadata, update docs/help if the contract changes, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Normalize defaults keys before passthrough lookup:** `src/config/defaults.ts:265-266`
  - body: The new merge indexes `agents.defaults.models` with raw `${providerId}/${raw.id}` before the current-main normalization used by `normalizeAgentModelMapForConfig()`. Valid normalized refs such as retired Gemini ids or provider aliases can keep working for model selection but silently miss the new metadata override, so normalize the map or lookup key before extracting overrides.
  - confidence: 0.9
- **[P2] Keep scoped runtime keys out of provider models:** `src/config/defaults.ts:49`
  - body: The known-key filter excludes only `alias`, `params`, and `streaming`, but current main supports `agents.defaults.models.*.agentRuntime` as scoped runtime policy. On rebase this can be copied into the provider model definition and create a second source of truth for runtime selection, so filter all known agent-model entry fields before forwarding provider metadata.
  - confidence: 0.87
- **[P2] Validate known model metadata before forwarding passthrough:** `src/config/zod-schema.agent-defaults.ts:35`
  - body: Switching the entry schema directly to `.passthrough()` accepts bad known metadata such as `contextWindow: "large"` or negative `maxTokens`, while provider model schema rejects those fields. `applyModelDefaults()` can then fall back to defaults instead of surfacing a config error, so explicitly validate supported metadata before forwarding truly unknown fields.
  - confidence: 0.82
