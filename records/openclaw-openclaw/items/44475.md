## What This Changes

The PR changes `agents.defaults.models` entries to accept arbitrary passthrough keys and merges those keys into provider model definitions during `applyModelDefaults()`, with tests for `contextWindow`, `maxTokens`, and filtered known keys.

## Best Possible Solution

Rebase onto current main and implement a validated, normalized override path that works against implicit provider catalogs without requiring a `models.providers` entry, while preserving `params`, `agentRuntime`, wildcard, and provider-model schema semantics.

## Solution Assessment

No: the current branch is not the best solution because it adds broad passthrough at validation time instead of a normalized, typed override path at the provider catalog/defaulting boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Apply overrides without requiring models.providers:** `src/config/defaults.ts:257`
  - body: The advertised concise config only sets `agents.defaults.models`, but this merge code runs while iterating `cfg.models.providers`. If the user has no provider entry, there is no provider model to mutate and the override is ignored, so the PR still requires the verbose provider config it is meant to avoid.
  - confidence: 0.9
- **[P2] Normalize model-default refs before lookup:** `src/config/defaults.ts:265-266`
  - body: This exact `${providerId}/${raw.id}` lookup skips the normalization used by the rest of `agents.defaults.models` (`normalizeAgentModelMapForConfig` / provider aliases / normalized model ids). Valid config keys that current main canonicalizes can silently miss their passthrough overrides.
  - confidence: 0.88
- **[P2] Validate supported model metadata instead of passthrough:** `src/config/zod-schema.agent-defaults.ts:35`
  - body: Switching the entry schema to `.passthrough()` accepts known model-definition fields without their existing provider-model validation, so values like a string `contextWindow` validate and then fall back during defaulting instead of producing a config error. Add explicit typed fields or validate passthrough values against the model-definition contract before merging.
  - confidence: 0.82
