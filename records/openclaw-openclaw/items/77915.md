## Summary

Keep open: this is a plausible fix path for the related 0-byte session-store bug, but the latest head still has a blocking recovery-candidate ordering bug and the external PR proof does not show an after-fix Gateway or CLI recovery run.

## What This Changes

This PR adds load-time session-store recovery from `.bak` and temp files, validates candidates, self-heals `sessions.json`, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the recovery-only design, but order tmp candidates by recovery priority before capping/evaluation and require a real Gateway or CLI recovery transcript showing `.bak` or `.tmp` recovery and self-heal before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main returns the initialized empty session store after a zero-byte, malformed, or non-object `sessions.json` read failure when there is no cache hit, and the related Linux report provides a concrete 0-byte incident.

## Solution Assessment

No. The recovery-only direction is narrow and maintainable, but this implementation does not honor its own fresh-as-fallback ordering and still lacks sufficient real after-fix behavior proof.
