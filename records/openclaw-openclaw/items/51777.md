## Summary

Keep open: the Feishu dmPolicy persistence bug is still source-reproducible, but this PR is stale against current main because it patches the old command implementation while the active onboarding flow lives under `src/flows`; it also still lacks after-fix real behavior proof.

## What This Changes

The PR adds a Feishu-only fallback `setPolicy` call in the old onboarding command loop when the selected DM policy matches the fallback current value and top-level `channels.feishu.dmPolicy` is absent.

## Best Possible Solution

Patch `src/flows/channel-setup.prompts.ts` with a presence-aware Feishu write based on the resolved policy key, preserve no-op behavior for non-idempotent channels, and add focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a fresh Feishu config with no explicit `dmPolicy`, selecting `pairing`, Feishu `getCurrent` returning fallback `pairing`, the active same-value guard skipping `setPolicy`, and the writer not materializing runtime defaults.

## Solution Assessment

No. The intent is right, but this PR patches the old command body instead of the active flow helper, and a broad unconditional setter is unsafe for channels with non-idempotent setters such as Twitch.
