## Summary

Keep open. The Feishu persistence bug still appears source-reproducible on current main, but this PR is not mergeable as-is because it patches a stale command implementation instead of the active onboarding flow and still lacks inspectable real behavior proof.

## What This Changes

The PR adds a Feishu-specific fallback write so an explicit `pairing` DM-policy choice during onboarding is persisted when `channels.feishu.dmPolicy` is missing.

## Best Possible Solution

Move a presence-aware Feishu write into `src/flows/channel-setup.prompts.ts` using the resolved policy key, preserve the no-op guard for non-idempotent channel setters, and add focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: fresh Feishu config, opt into DM-policy setup, select `pairing`, Feishu `getCurrent` returns fallback `pairing`, the active guard skips `setPolicy`, and the writer does not materialize runtime defaults.

## Solution Assessment

No. The intended Feishu fix is valid, but this PR patches the obsolete command body instead of the active flow helper; a global unconditional setter is also unsafe because some channel setters are not idempotent.
