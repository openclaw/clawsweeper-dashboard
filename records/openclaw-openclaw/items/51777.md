## Summary

Keep this PR open. The Feishu explicit-default persistence bug is still present on current main, but this patch is stale because it changes the old onboard-channel implementation while current main routes through the channel setup flow.

## What This Changes

The PR adds a Feishu-specific missing-key setter call in the old onboarding DM-policy loop so selecting `pairing` on a fresh config persists `channels.feishu.dmPolicy`.

## Best Possible Solution

Rewrite the fix against `src/flows/channel-setup.prompts.ts` with a presence-aware path for Feishu, or a small generic persisted-presence contract that individual channel setup surfaces can implement safely. Add a regression test for the fresh Feishu `pairing` selection and keep non-idempotent setters protected.

## Reproduction Assessment

Yes. A source-level reproduction is clear: start with a Feishu config where `channels.feishu.dmPolicy` is absent, select `pairing` in `maybeConfigureDmPolicies`, and the current `nextPolicy !== current` guard skips the Feishu setter because `getCurrent` returns the fallback `pairing`.

## Solution Assessment

No. The current PR direction is useful, but the patch is not the best current fix because it targets an obsolete implementation location; the maintainable fix belongs in the active channel setup prompt flow with tests and without globally invoking non-idempotent setters.
