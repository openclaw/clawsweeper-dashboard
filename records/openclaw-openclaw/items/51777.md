## Summary

Keep open. The underlying Feishu onboarding persistence bug is source-reproducible, but this PR is stale against current main: it patches the old command loop while the live setup path still contains the guarded write, and it also lacks the required changelog/test coverage.

## What This Changes

The PR adds a Feishu-specific fallback `setPolicy` call in the old onboard DM-policy loop when `channels.feishu.dmPolicy` is absent and the selected policy matches the current fallback.

## Best Possible Solution

Patch the active DM-policy prompt to persist an explicit Feishu default when the key is absent while preserving the changed-value guard for non-idempotent channel setters, then add focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with fresh Feishu config missing `channels.feishu.dmPolicy`, selecting `pairing` makes Feishu `getCurrent` return fallback `pairing`, so the active guard skips `setPolicy` and the config writer does not persist validation defaults.

## Solution Assessment

No. The PR addresses the right user-visible bug, but it patches the obsolete command implementation instead of the active setup flow; a presence-aware write in `src/flows/channel-setup.prompts.ts` is the narrower maintainable fix.
