## Summary

Keep open: the PR addresses a real trusted-proxy auth fallback and includes live after-fix proof, but it still leaves env password credential resolution and docs inconsistent with the fail-closed policy.

## What This Changes

The PR removes the trusted-proxy local-direct password fallback, updates local credential planning/tests, trusted-proxy docs, and the changelog.

## Best Possible Solution

Land a coordinated fail-closed trusted-proxy policy change after suppressing all implicit password credential sources, removing fallback guidance from docs, and getting gateway/secops approval.

## Reproduction Assessment

Yes for source-level reproduction. Current main has the fallback branch in `src/gateway/auth.ts`, and existing tests assert successful password fallback for the reported local-direct trusted-proxy path.

## Solution Assessment

No. Removing the server fallback is a plausible policy direction, but this patch still resolves env password credentials and leaves public docs describing the removed fallback.

## Review Findings

Overall correctness: patch is incorrect

  - body: This only clears configured fallback passwords. Because the resolver still passes `params.env` through, `OPENCLAW_GATEWAY_PASSWORD` can still be returned for local trusted-proxy configs, so `callGateway` may keep sending password auth that the server now rejects. Add coverage for the env-password path and suppress password resolution for this mode.
  - confidence: 0.9

That gives operators contradictory security guidance for the changed auth policy.
