## Summary

Keep open: the timing goal is still relevant, but this PR is not merge-ready because it can spend the entire Codex hook timeout on direct-bridge polling, starving the gateway fallback, and it lacks real after-fix proof.

## What This Changes

The PR changes `openclaw hooks relay` to wait for the direct native-hook bridge for the configured timeout before falling back to `nativeHook.invoke`, with CLI coverage for direct bridge and fallback paths.

## Best Possible Solution

Rework the PR around a bounded direct-bridge wait that reserves fallback budget, rebase it onto current main's relay generation handling, and require real Codex hook relay proof before merge.

## Reproduction Assessment

Yes from source: with a missing direct bridge, the PR waits the full hook timeout before attempting `nativeHook.invoke`, while Codex enforces that same timeout around the hook subprocess. ## Solution Assessment

No. The best fix is a bounded direct-bridge startup wait that still reserves gateway fallback budget, not spending the entire Codex hook timeout before fallback starts.

- **[P1] Reserve fallback budget before waiting on the bridge:** `src/cli/native-hook-relay-cli.ts:54-55`
