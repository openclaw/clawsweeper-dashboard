## Summary

Keep open. The direct-bridge startup race appears valid, but this PR should not merge as written because it can spend the whole hook timeout before gateway fallback and it still lacks real after-fix behavior proof.

## What This Changes

The PR changes `openclaw hooks relay` to wait for the direct native relay bridge for the configured hook timeout before falling back to `nativeHook.invoke`, with CLI tests for direct bridge and fallback paths.

## Best Possible Solution

Keep the direct bridge preference, but implement a bounded readiness wait that reserves gateway fallback and fail-closed output budget, with focused tests and redacted real self-hosted proof.

## Reproduction Assessment

Yes, from source inspection. Current main gives the direct bridge only a short fixed registration wait, and the PR-introduced fallback starvation is visible because it passes the whole hook timeout as the bridge registration window.

## Solution Assessment

No, not as written. The maintainable fix should cap direct-bridge readiness/polling and reserve fallback budget, similar to the direction explored in https://github.com/openclaw/openclaw/pull/80860, rather than spending the whole hook timeout before gateway fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve time for the gateway fallback:** `src/cli/native-hook-relay-cli.ts:54`
  - body: Setting `registrationTimeoutMs` to the whole hook timeout means a missing direct bridge can make `invokeNativeHookRelayBridge` wait until the command budget is exhausted before `nativeHook.invoke` runs. In fallback-only or older bridge environments, cap the direct-bridge readiness wait and reserve budget for gateway fallback and provider-compatible fail-closed output.
