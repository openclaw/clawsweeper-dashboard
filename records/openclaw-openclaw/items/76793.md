## Summary

Keep open: the PR targets a real source-reproducible native hook relay race, but it should not merge as written because the direct-bridge wait can consume the whole hook timeout before the gateway fallback runs, and the external PR lacks after-fix real behavior proof. The prior changelog-only bot finding should not be carried forward under current policy.

## What This Changes

The PR makes `openclaw hooks relay` use the normalized hook timeout as the direct bridge registration wait and adds CLI tests for direct bridge and gateway fallback paths.

## Best Possible Solution

Keep the direct bridge preference, but implement it as a bounded readiness wait that reserves gateway fallback and fail-closed output budget, with focused tests and redacted real self-hosted proof.

## Reproduction Assessment

Yes, from source inspection. Current main gives the direct bridge only 100 ms before fallback, and the PR-introduced fallback starvation is also reproducible from the bridge loop when `registrationTimeoutMs` equals the full hook timeout.

## Solution Assessment

No, not as written. The narrow maintainable fix should use a capped direct-bridge readiness wait and reserve fallback budget, similar to the direction explored in https://github.com/openclaw/openclaw/pull/80860, rather than spending the entire hook timeout on bridge registration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve time for the gateway fallback:** `src/cli/native-hook-relay-cli.ts:54`
  - body: Setting `registrationTimeoutMs` to the whole hook timeout means an unavailable direct bridge makes `invokeNativeHookRelayBridge` wait until the command budget is exhausted before `nativeHook.invoke` gets a chance. In fallback-only or older bridge environments this delays every hook by the full timeout and can leave no time for provider-compatible fail-closed output; cap the direct-bridge readiness wait and reserve fallback budget instead.
  - confidence: 0.88
