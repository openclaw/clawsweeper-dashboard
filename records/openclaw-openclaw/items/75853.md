## Summary

Keep open. The reporter’s follow-up narrows this from a core compatibility regression to a guardrail/diagnostic gap: current main intentionally rejects `hooks.transformsDir` outside the canonical hook transforms root, but that check still happens during Gateway runtime config resolution and can make a supervised Gateway repeatedly fail instead of giving a targeted doctor/status recovery hint.

## What This Changes

The issue requests clearer handling when `hooks.transformsDir` is configured to a workspace skill directory instead of the canonical hook transform directory.

## Best Possible Solution

Keep the fail-closed containment rule for transform modules, but add a narrow diagnostic path that flags `hooks.transformsDir` outside the canonical transforms root before or during doctor/status checks, with docs that point users to move transforms under `~/.openclaw/hooks/transforms` or use the relevant plugin webhook flow instead of a workspace skill directory.

## Reproduction Assessment

Yes. A config with hooks enabled, a transform mapping, and an absolute `hooks.transformsDir` outside the config hooks transform root reaches `resolveHooksConfig` during Gateway startup and throws the same `Hook transformsDir` containment error shown in the report.

## Solution Assessment

Yes for the guardrail direction: preserving the containment check is the right security posture. The maintainable fix is a friendlier preflight/doctor/status diagnostic and documentation, not compatibility with arbitrary workspace skill paths for dynamic imports.
