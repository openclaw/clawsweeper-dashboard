## Summary

Keep open. The reporter narrowed this from a core compatibility regression to a guardrail/diagnostic gap, and current main still fails closed during Gateway startup without a targeted doctor/status recovery hint for an invalid `hooks.transformsDir`.

## What This Changes

The issue asks OpenClaw to handle `hooks.transformsDir` values outside the canonical hook transform directory with migration, earlier validation, or clearer recovery diagnostics.

## Best Possible Solution

Keep the fail-closed transform containment rule, then add a focused doctor/status/preflight diagnostic plus docs that tell users to place transforms under the canonical hooks transform root or use plugin-owned webhook flows.

## Reproduction Assessment

Yes. A config with hooks enabled, a mapping transform, and an absolute `hooks.transformsDir` outside `<configDir>/hooks/transforms` reaches `resolveHooksConfig` during Gateway startup and throws `Hook transformsDir`; existing tests cover that rejected shape.

## Solution Assessment

Yes for the diagnostic direction. The maintainable fix is clearer validation and recovery guidance, not compatibility with arbitrary workspace skill directories for dynamic imports.
