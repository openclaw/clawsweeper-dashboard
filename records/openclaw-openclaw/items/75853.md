## Summary

Keep open. Current main still rejects an absolute `hooks.transformsDir` outside the canonical hooks transform root during gateway runtime configuration, and that happens before the HTTP/WebSocket server is started, so the reported startup failure remains plausible and actionable.

## What This Changes

Detect and recover from stale `hooks.transformsDir` values outside the canonical hooks transform directory so the gateway does not crash-loop on startup.

## Best Possible Solution

Keep the strict transform-module containment rule, but add a targeted recovery/diagnostic path for stale external `hooks.transformsDir` values: detect them before opaque gateway crash loops, point users to the canonical hooks transform directory, and let `doctor --fix` remove or quarantine the invalid setting without trusting or importing code from the external path.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: an enabled hooks config with at least one mapping and an absolute `hooks.transformsDir` outside `<configDir>/hooks/transforms` reaches `resolveGatewayRuntimeConfig` -> `resolveHooksConfig` -> `resolveHookMappings` and throws before the gateway starts.

## Solution Assessment

Unclear for the migration option. The safest maintainable solution is not to restore compatibility with arbitrary workspace skill paths; it is to preserve containment and add clear startup/config/doctor recovery for stale values.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75853 with a narrow recovery/diagnostic change. Preserve the current rule that hook transform modules must resolve under `<configDir>/hooks/transforms`; do not allow or import modules from arbitrary workspace skill directories. Add detection for configured `hooks.transformsDir` values that escape the canonical transforms root when hook mappings are present, so users get an actionable recovery path instead of an opaque gateway crash loop. Prefer a `doctor`/config/startup preflight that reports the invalid path and canonical target; `doctor --fix` may remove or quarantine the invalid `hooks.transformsDir` only if it does not trust external code and tells the user to move reviewed transform modules into `~/.openclaw/hooks/transforms`. Add focused tests for an absolute workspace path such as `~/.openclaw/workspace/skills/linear-webhook`, accepted in-root subdirectories, and the startup/config/doctor message. Update docs and `CHANGELOG.md` if the change is user-facing.
