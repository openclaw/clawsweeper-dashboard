## Summary

Keep open. The reporter’s correction narrows this from a Linear runtime regression to a user-configuration guardrail, but current main still lets `hooks.transformsDir` pass schema validation and then throws during gateway runtime config resolution before the server is up. That leaves a focused repo-side improvement: clearer diagnostics and doctor/docs recovery while preserving the transform containment boundary.

## What This Changes

The issue requests that invalid `hooks.transformsDir` values pointing at workspace skill paths stop producing opaque gateway crash loops and instead get a clear recovery path.

## Best Possible Solution

Add a small, security-preserving guardrail around stale or invalid external `hooks.transformsDir` values: detect the bad path early, show a precise recovery hint, and let `openclaw doctor --fix` remove or quarantine the invalid setting without ever importing code from the external directory. Documentation should also steer Linear-style webhook workflows toward plugin/webhook setup rather than custom hook transform roots.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `hooks.enabled=true` with a token, at least one mapping transform, and `hooks.transformsDir` set to an absolute workspace skill path outside `<configDir>/hooks/transforms`; startup reaches `resolveGatewayRuntimeConfig` -> `resolveHooksConfig` -> `resolveHookMappings` and throws before the gateway can serve.

## Solution Assessment

Yes for the narrow guardrail direction, no for restoring compatibility with arbitrary workspace skill paths. The maintainable fix is to preserve containment and improve validation, status, doctor, and docs recovery around this configuration error.
