## Summary

Keep open. Current main still rejects the reported `hooks.transformsDir` shape during Gateway runtime config resolution, while normal config validation/preflight only treats `hooks.transformsDir` as a string, so the service can still reach a startup-time throw instead of an actionable config/doctor repair path.

## What This Changes

The issue requests recovery or clear diagnostics when `hooks.transformsDir` points outside the canonical hook transform directory and causes Gateway startup crash loops.

## Best Possible Solution

Keep the secure transform-module boundary, but catch invalid `hooks.transformsDir` before Gateway startup reaches runtime config construction. A maintainable fix should add config validation/preflight and doctor guidance or repair for out-of-root transform directories, with explicit tests for the reported absolute workspace-skill path.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: configure hooks with a mapping and an absolute `hooks.transformsDir` outside `<configDir>/hooks/transforms`; current `resolveHookMappings` throws, and startup calls it through `resolveHooksConfig`.

## Solution Assessment

Yes, keeping this open with a focused repair path is the right outcome. The safest solution is early validation and doctor remediation, not relaxing dynamic transform module loading to arbitrary workspace skill directories.
