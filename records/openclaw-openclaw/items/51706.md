## Summary

Keep open. The original runtime-model display bug shipped via #55425, but this PR’s latest fallback-preservation path is stale against the current `src/status` split and still has a concrete fallback-display bug for channel/group-selected sessions.

## What This Changes

The PR updates `session_status` and status rendering tests so persisted runtime session models can appear in the status card while attempting to keep active fallback details visible.

## Best Possible Solution

Land a focused current-main replacement that separates display-only runtime identity from selected-model/fallback semantics in `src/status`, resolves fallback selected refs from persisted fallback notice state or the actual selected route, preserves unknown-provider display, and leaves routing, auth, and persistence unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists for the PR bug: create a session with fallback notice fields where the selected model came from a channel/group route rather than the global default, then the PR’s `preserveFallbackStatus` path selects the wrong ref and can suppress the fallback line.

## Solution Assessment

No. The PR’s current approach is not the best fix because it hardcodes the global configured model for fallback preservation and targets a stale status surface; the safer fix is a current-main `src/status` API split for selected versus display/runtime identity.
