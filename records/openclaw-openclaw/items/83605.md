## Label Justifications

- `P2`: The bug affects cron task-status and cleanup/reporting linkage but is scoped to isolated cron task rows.
- `impact:session-state`: Detached cron task records can lose the association to their backing cron session key.

## Summary

Keep open: current main and the latest release still copy only `job.sessionKey` into cron task ledger rows, while isolated cron execution derives its backing session key separately. The linked open fix PR has closing syntax for this issue, so the issue should stay open until that PR lands or is rejected.

## What This Changes

The issue reports that isolated cron job task ledger rows can be created without `childSessionKey` when the job has no explicit `sessionKey`.

## Best Possible Solution

Review and land or explicitly close https://github.com/openclaw/openclaw/pull/83606; the desired end state is for isolated cron ledger rows to use the stable backing cron session key while preserving per-run transcript session suffixes.

## Reproduction Assessment

Yes from source inspection: an isolated `agentTurn` cron job can omit `sessionKey`, and current main copies only `job.sessionKey` into `childSessionKey` before isolated execution returns its derived session key. I did not run a live cron reproduction because this review is read-only.

## Solution Assessment

Yes in direction: deriving the ledger `childSessionKey` from the same `resolveCronAgentSessionKey` path used by isolated execution is the narrow maintainable fix. The linked PR is the active implementation candidate and should be reviewed rather than queueing a second repair.
