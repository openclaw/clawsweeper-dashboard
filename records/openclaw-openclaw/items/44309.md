## Summary

Keep open. Current main and v2026.5.12 still expose `sessions_send` as conversation-style A2A delivery with no public one-way dispatch/private mode; the related private-delivery PR is open and not a merged implementation, so this needs maintainer API direction rather than cleanup closure.

## What This Changes

Request a first-class one-way dispatch/handoff mode for `sessions_send` that enqueues work in the target session without reply-back ping-pong or announce delivery.

## Best Possible Solution

Define a backward-compatible dispatch intent in the A2A handoff contract, with explicit tool schema, optional delivery ack, docs, and regression tests that preserve existing conversation mode.

## Reproduction Assessment

Yes, for the current behavior by source inspection: `sessions_send` has no dispatch parameter and ordinary async or waited sends still enter the A2A follow-up flow. This is a feature request for new semantics, not a failure of an already documented contract.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should align with broader A2A handoff work instead of relying on prompts or `timeoutSeconds`.
