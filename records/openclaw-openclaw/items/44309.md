## Summary

Keep open. Current main and v2026.5.20 still expose conversation-oriented `sessions_send` behavior without a caller-selected dispatch-only mode, while the related private-delivery and A2A handoff protocol work remains open rather than merged.

## What This Changes

Requests a first-class one-way A2A dispatch or handoff mode for `sessions_send`, or a new primitive, that persists work into the target session without requester reply-back or announce lifecycle.

## Best Possible Solution

Define a backward-compatible A2A dispatch intent or primitive with explicit tool schema, optional delivery acknowledgement semantics, docs, and regression tests while preserving the existing conversation mode.

## Reproduction Assessment

Yes for source-level behavior: current main has no dispatch parameter or primitive, and ordinary peer sends still enter the A2A reply/announce path. I did not run a live multi-agent transport repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should be aligned with the open private-delivery, structured-handoff, acknowledgement, and duplicate-reply work rather than added as prompt-only behavior.
