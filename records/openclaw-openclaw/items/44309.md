## Summary

Keep open. Current main and the latest release still expose `sessions_send` as a conversation-style A2A tool: there is a narrow parent-owned child skip, but no public one-way dispatch, reply-back, or announce suppression contract for arbitrary controller/worker handoffs.

## What This Changes

The issue requests a first-class one-way A2A dispatch mode for `sessions_send` that persists work into the target session without reply-back ping-pong or requester announce behavior.

## Best Possible Solution

Define a backward-compatible dispatch intent in the A2A handoff contract, with explicit tool schema, optional delivery ack, docs, and regression tests that preserve existing conversation mode.

## Reproduction Assessment

Yes for the current behavior by source inspection: `sessions_send` has no dispatch parameter and normal async or waited sends still enter the A2A follow-up flow. This is a feature request for new semantics, not a failure of an already documented contract.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should align with the broader A2A handoff work instead of relying on prompts or `timeoutSeconds` behavior.
