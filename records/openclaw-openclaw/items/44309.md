## Summary

Keep open. Current main still implements `sessions_send` as a conversation-style A2A lifecycle: the public schema has no dispatch/ack mode, normal sends still start the reply-back/announce flow, and tests intentionally assert ping-pong plus target announce. Related reports cover narrower duplicate guards and broader handoff protocol work, but none has shipped the explicit one-way handoff semantic.

## What This Changes

Request an explicit dispatch-only `sessions_send` mode or separate primitive that enqueues work in a target session without reply-back or announce.

## Best Possible Solution

Fold a backward-compatible dispatch intent into the A2A handoff contract, with a public tool/API shape, optional structured ack, docs, and regression tests preserving existing conversation mode.

## Reproduction Assessment

Yes for current behavior by source inspection: `sessions_send` lacks a dispatch parameter and starts the A2A flow for both async and waited sends. This is still a feature request rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the API shape. A first-class dispatch mode or primitive is the right class of solution, but it should align with https://github.com/openclaw/openclaw/pull/63789 and https://github.com/openclaw/openclaw/issues/33478 instead of relying on prompts or `maxPingPongTurns`.
