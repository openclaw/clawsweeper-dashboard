## Summary

Keep open: current main still lacks an explicit one-way dispatch contract for ordinary A2A `sessions_send`; related open work covers narrower duplicate/reverse-call and announce problems without replacing the reply-back/announce lifecycle.

## What This Changes

The issue requests an explicit one-way A2A dispatch/handoff mode for `sessions_send` that persists work into the target session without requester reinjection, reply-back ping-pong, or final announce.

## Best Possible Solution

Define one canonical delivery intent for `sessions_send` or a separate one-way primitive, preserving the current conversation mode while adding tested dispatch behavior that never reinjects the target reply or runs announce unless explicitly requested.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: ordinary cross-session sends still call `runSessionsSendA2AFlow`, which can run reply steps and an announce step after the target reply. I did not execute a live A2A handoff in this read-only review.

## Solution Assessment

Unclear as an implementation choice: a first-class delivery intent is better than prompt-only workarounds, but maintainers still need to choose whether the permanent API is a `sessions_send` mode/flag or a separate one-way primitive.
