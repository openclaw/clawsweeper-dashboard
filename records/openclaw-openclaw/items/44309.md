## Summary

Keep open: current `main` and `v2026.6.10` still implement ordinary peer `sessions_send` as conversation-style A2A delivery with bounded reply-back turns plus announce, while this issue asks for a distinct one-way dispatch contract. The open reverse-call PR overlaps duplicate-message symptoms but explicitly leaves result delivery unchanged, so it does not supersede the dispatch-mode product/API decision.

## What This Changes

The issue requests an explicit dispatch-only agent-to-agent handoff mode for `sessions_send` or a sibling primitive that avoids reply-back reinjection and requester announce behavior.

## Best Possible Solution

Keep this open for maintainer API direction, then implement a narrow first-class dispatch path that writes the target-session message and returns optional delivery acknowledgement without starting reply-back or announce follow-up.

## Reproduction Assessment

Yes at source level: current peer `sessions_send` still has no dispatch parameter and starts the A2A reply/announce flow for ordinary peer sends. I did not run a live multi-agent replay in this read-only cleanup pass.

## Solution Assessment

Yes at the product level: a first-class framework mode or primitive is the maintainable alternative to prompt conventions or `timeoutSeconds: 0`. The exact model-facing API shape still needs maintainer direction.
