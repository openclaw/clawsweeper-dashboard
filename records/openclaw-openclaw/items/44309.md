## Summary

Keep open. Current main still has no dispatch-only `sessions_send` mode or one-way primitive, and normal peer sends still enter the reply-back plus announce A2A lifecycle except for narrow parent-owned child skips. Related open items cover adjacent duplicate, structured-callback, and handoff-contract work, but they do not make this requested dispatch semantic obsolete.

## What This Changes

The issue requests an explicit one-way A2A dispatch or handoff mode that delivers work to the target session without reply-back reinjection or requester announce behavior.

## Best Possible Solution

Define a backward-compatible A2A delivery-intent contract that separates conversation from dispatch, document it, and cover peer, persistent, async, and announce-suppression behavior with focused tests while preserving intentional peer follow-up workflows.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on `origin/main` shows the missing capability: `sessions_send` has no dispatch/deliver-only parameter and normal peer sends still run the A2A reply/announce flow.

## Solution Assessment

Unclear. A per-call dispatch mode or a separate one-way primitive is plausible, but maintainers should settle the A2A handoff semantics alongside https://github.com/openclaw/openclaw/issues/62872, https://github.com/openclaw/openclaw/issues/33478, and https://github.com/openclaw/openclaw/pull/63789 before an automated fix.
