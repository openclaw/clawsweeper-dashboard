## Summary

Keep open: current `main` and `v2026.6.10` still implement ordinary peer `sessions_send` as conversation-style A2A delivery with bounded reply-back turns plus announce, while this issue asks for a distinct one-way dispatch contract. The open reverse-call PR overlaps duplicate-message symptoms but explicitly leaves result delivery unchanged, so it does not supersede the dispatch-mode product/API decision.

## What This Changes

The issue requests an explicit one-way A2A dispatch/handoff mode for `sessions_send` or a separate delivery primitive that avoids reply-back reinjection and announce behavior.

## Best Possible Solution

Keep this open for maintainer API direction, then implement a narrow first-class dispatch path that persists the target-session message and returns optional delivery acknowledgement without starting reply-back or announce follow-up.

## Reproduction Assessment

Yes at source level: current `sessions_send` lacks a dispatch parameter and still starts a follow-up A2A flow for ordinary peer sends. I did not run a live multi-agent replay in this read-only sweep.

## Solution Assessment

Yes at the product level: a first-class framework mode or primitive is the maintainable alternative to prompt conventions or `timeoutSeconds: 0`. The exact model-facing API shape still needs maintainer direction.
