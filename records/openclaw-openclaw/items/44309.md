## Summary

Keep open. Current main still only supports `sessions_send` as conversation-style delivery: the tool schema has no per-call dispatch intent, fire-and-forget still schedules the A2A follow-up path, and the helper still performs requester/target reply-back plus announce behavior. The related open items and PRs are adjacent lifecycle or hook work, not a merged canonical replacement for this one-way handoff semantic.

## What This Changes

Add an explicit one-way dispatch or handoff mode for `sessions_send` so a message can be delivered to a target session without requester-side reply-back reinjection or announce behavior.

## Best Possible Solution

Add a backward-compatible delivery intent to `sessions_send`, with current behavior as the default conversation mode and a dispatch mode that persists the inter-session message into the target session, starts or enqueues the target run as appropriate, returns a minimal delivery acknowledgement, and does not invoke requester-side ping-pong or announce follow-up.

## Reproduction Assessment

Yes. The current behavior is reproducible from source and focused tests: `timeoutSeconds: 0` still returns pending announce metadata and schedules `runSessionsSendA2AFlow`, while the A2A helper runs reply-back and announce steps.

## Solution Assessment

Yes, a first-class per-call delivery intent is the narrow maintainable direction. Prompt conventions, global `maxPingPongTurns`, and the narrow ACP skip do not solve one-way handoffs without changing unrelated conversation semantics.
