## Summary

Keep open. Current main and the latest shipped release still lack a one-way dispatch mode for `sessions_send`; ordinary sends continue to use the A2A reply-back and announce lifecycle, and the related private-delivery and handoff-protocol items are still open and only cover parts of the requested semantic.

## What This Changes

Add a one-way A2A dispatch/handoff mode or primitive that delivers work to a target session without reply-back ping-pong or announce-back behavior.

## Best Possible Solution

Define a backward-compatible dispatch intent in the A2A handoff contract, with explicit tool schema, optional delivery acknowledgement semantics, docs, and regression tests that preserve existing conversation mode.

## Reproduction Assessment

Yes. Source inspection shows `sessions_send` has no dispatch parameter or primitive, and ordinary peer sends still enter the A2A reply/announce flow; this is a feature/API gap rather than broken documented behavior.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should align with the open private-delivery and handoff-protocol work instead of relying on prompts or `timeoutSeconds`.
