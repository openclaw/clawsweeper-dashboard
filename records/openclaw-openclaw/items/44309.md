## Summary

Keep open. Current `main` still has conversation-oriented `sessions_send` semantics with no dispatch intent, and the related open proposals are either broader, stale, or docs-only rather than a merged implementation.

## What This Changes

The issue requests an explicit one-way `sessions_send` dispatch/handoff mode that persists work to the target session without requester reply-back ping-pong or announce-back behavior.

## Best Possible Solution

Add a backward-compatible per-call dispatch intent in the A2A handoff contract that enqueues or starts the target session with a minimal acknowledgement while skipping requester reply-back and target announce only when explicitly requested.

## Reproduction Assessment

Yes for the current behavior: source inspection shows normal and fire-and-forget peer sends still enter the A2A helper, which can inject reply-back turns and run announce delivery. This is a feature gap rather than a broken documented contract.

## Solution Assessment

Unclear until maintainers choose the public API shape. A per-call dispatch intent looks like the narrowest maintainable direction, but the exact contract should align with the open handoff protocol and private-delivery discussions.
