## Summary

Keep open. Current main still exposes the conversation-oriented `sessions_send` path with hard-coded announce delivery and the A2A reply/announce flow; the related private-delivery PR is open, not merged, and does not yet settle the requested dispatch-only public API.

## What This Changes

Request a first-class one-way A2A dispatch/handoff mode for `sessions_send` or a sibling primitive that persists the target message without reply-back or announce behavior.

## Best Possible Solution

Define a backward-compatible A2A dispatch intent with explicit tool schema or primitive, optional delivery acknowledgement semantics, docs, and regression tests while preserving the existing conversation mode.

## Reproduction Assessment

Yes for source-level current behavior: current main has no dispatch parameter or primitive, and normal peer sends still enter the A2A reply/announce path. I did not run a live multi-agent transport repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should be aligned with https://github.com/openclaw/openclaw/pull/50720, https://github.com/openclaw/openclaw/issues/33478, and https://github.com/openclaw/openclaw/issues/39476 rather than patched as prompt-only behavior.
