## Summary

Keep open: current main and v2026.6.6 still expose conversation-style `sessions_send` semantics for ordinary peer A2A sends, and the open related PR only guards reverse-calls without adding the requested one-way dispatch/no-announce contract.

## What This Changes

Requested feature: add one-way A2A dispatch/handoff semantics for `sessions_send` or a new primitive so target work can be delivered without reply-back reinjection or final announce.

## Best Possible Solution

Define one canonical A2A delivery intent that preserves existing conversation mode while adding tested dispatch behavior that persists the target message, returns an optional acknowledgement, and skips reply-back and announce unless explicitly requested.

## Reproduction Assessment

Yes, source-level: current `sessions_send` has no dispatch selector, ordinary sends still call the A2A helper, and the helper still performs reply-back plus announce work. I did not run a live two-agent handoff in this read-only review.

## Solution Assessment

Unclear as an implementation choice: a first-class delivery intent is the right level compared with prompt-only workarounds, but maintainers still need to choose the permanent public tool/API shape.
