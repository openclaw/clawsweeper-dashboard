## Label Justifications

- `P2`: The request addresses a meaningful multi-agent workflow problem, but it is a bounded feature/API decision rather than an emergency regression.
- `impact:session-state`: The unwanted reply-back loop can pull requester and worker sessions into unintended transcript and context state.

## Summary

Keep open: current main and v2026.6.6 still implement ordinary `sessions_send` as conversation-style A2A delivery with reply-back and final announce, while this issue asks for an explicit one-way dispatch/handoff contract. Related open PRs address acknowledgement or reverse-call duplicate behavior, but none supersede the requested dispatch-only mode.

## What This Changes

Request adds an explicit one-way A2A dispatch mode or primitive for `sessions_send`-style handoffs without reply-back ping-pong or requester announce.

## Best Possible Solution

Keep this as the canonical design item for a maintainer-approved typed one-way A2A dispatch contract, with compatible default conversation behavior and focused tests for peer sends, dispatch-only delivery, reply-back suppression, announce suppression, and channel/session provenance.

## Reproduction Assessment

Yes, source-level: current main has no dispatch selector and ordinary peer sends still enter the A2A reply-back plus announce helper. I did not run a live two-agent handoff in this read-only review.

## Solution Assessment

Unclear on exact API shape: a typed per-send dispatch mode or separate primitive is the right owner boundary, but maintainers need to choose the permanent contract and compatibility story.
