## Summary

Keep open: current main and the latest release still implement ordinary `sessions_send` as conversation-style A2A delivery with reply-back and announce, while this issue asks for a new explicit one-way dispatch contract. The open reverse-call fix for https://github.com/openclaw/openclaw/pull/91161 overlaps with duplicate-message symptoms, but its own body says result delivery is unchanged, so it does not supersede the broader dispatch-mode product/API decision.

## What This Changes

The issue requests an explicit one-way A2A dispatch mode for `sessions_send`-style handoffs so target-session work is delivered without reply-back or requester-session reinjection.

## Best Possible Solution

Keep this open for maintainer API direction, then implement a narrow first-class dispatch path that persists the target-session message and returns optional delivery acknowledgement without starting reply-back or announce follow-up.

## Reproduction Assessment

Yes from source inspection: current `sessions_send` starts the A2A follow-up flow for ordinary peer sends, and current tests assert ping-pong plus announce behavior. I did not run a live multi-agent repro in this read-only sweep.

## Solution Assessment

Yes at the product level: a framework-level one-way semantic is the maintainable alternative to prompt conventions or `timeoutSeconds: 0`. The exact API shape still needs maintainer direction because it changes a model-facing session-tool contract.
