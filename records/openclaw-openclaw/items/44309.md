## Summary

Keep open. Current main and v2026.5.12 still expose `sessions_send` as conversation-style A2A: there is a narrow parent-owned child skip, but no public dispatch mode for arbitrary controller/worker handoffs, and the request overlaps open private-delivery and handoff-protocol work.

## What This Changes

Request to add a public one-way dispatch/handoff mode for `sessions_send` or a sibling primitive that persists a task to the target session without reply-back ping-pong or announce.

## Best Possible Solution

Define a backward-compatible dispatch intent in the A2A handoff contract, with explicit tool schema, optional delivery ack, docs, and regression tests that preserve existing conversation mode.

## Reproduction Assessment

Yes, by source inspection: `sessions_send` has no dispatch parameter and normal async or waited sends still enter the A2A follow-up flow. This is a feature/API gap, not a failure of an existing documented contract.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should align with the broader A2A handoff and private-delivery work instead of relying on prompts or `timeoutSeconds`.
