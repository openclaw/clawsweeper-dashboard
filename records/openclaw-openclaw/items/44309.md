## Label Justifications

- `P2`: The issue describes a meaningful multi-agent workflow problem, but it is a product/API feature decision rather than an emergency regression.
- `impact:session-state`: The requested dispatch mode is specifically about preventing requester and worker sessions from being pulled into unintended A2A state loops.
- `impact:message-loss`: The reported behavior involves duplicated, misrouted, and suppressed inter-session messages during A2A handoffs.

## Summary

Keep open: current main and v2026.6.6 still expose ordinary `sessions_send` as a conversation-style A2A flow with reply-back and announce behavior, while this issue asks for a first-class one-way dispatch contract.

## What This Changes

Request an explicit one-way A2A dispatch/handoff mode for `sessions_send`-style delivery that persists work into the target session without reply-back reinjection or requester announce ping-pong.

## Best Possible Solution

Keep this as the canonical design item for a maintainer-approved typed one-way A2A dispatch contract, implemented with default-compatible conversation behavior plus focused regression tests for peer sends, fire-and-forget, reply-back suppression, announce suppression, and channel delivery.

## Reproduction Assessment

Yes, source-level: current `sessions_send` has no dispatch selector and ordinary peer sends still enter the A2A reply-back plus announce helper. I did not run a live two-agent handoff in this read-only review.

## Solution Assessment

Unclear on exact API shape: a typed per-send dispatch mode or separate primitive is the right boundary, but maintainers need to choose the permanent contract and compatibility story.
