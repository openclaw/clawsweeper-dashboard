## Summary

Keep open. Current main and latest release still expose conversation-oriented `sessions_send` semantics without a caller-selected dispatch mode, and the overlapping private-delivery and handoff-protocol work is still open or closed unmerged.

## What This Changes

The issue requests an explicit A2A dispatch/handoff mode that persists a task into a target session without reply-back ping-pong, requester reinjection, or final announce.

## Best Possible Solution

Define a backward-compatible dispatch intent or separate primitive with explicit tool schema, optional delivery acknowledgement semantics, docs, and regression tests while preserving current conversation mode by default.

## Reproduction Assessment

Yes for source-level behavior: current main has no dispatch parameter or primitive, and ordinary peer sends still enter the A2A reply/announce path. I did not run a live multi-agent transport repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should be aligned with the related private-delivery, structured-handoff, acknowledgement, and duplicate-reply work.
