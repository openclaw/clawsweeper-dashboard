## Summary

Keep open. Current main still lacks a first-class one-way dispatch mode for `sessions_send`; the public tool schema has no dispatch/reply-back/announce control, and the implementation still starts the A2A ping-pong plus announce lifecycle for normal async and waited sends. Related issues and the draft handoff PR cover adjacent or broader work, but they do not supersede this specific product decision.

## What This Changes

Request an explicit A2A dispatch-only handoff mode that persists a message into the target session without reply-back reinjection or announce back to the requester.

## Best Possible Solution

Define a backward-compatible dispatch intent in the A2A handoff contract, with explicit public schema, optional structured delivery ack, docs, and regression tests that preserve existing conversation mode.

## Reproduction Assessment

Yes for the current behavior by source inspection: `sessions_send` has no dispatch parameter and enters the A2A flow for normal async and waited sends. This is a feature request for new semantics, not a failure of an existing documented contract.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should align with the open handoff protocol discussion instead of relying on prompt conventions or `timeoutSeconds` behavior.
