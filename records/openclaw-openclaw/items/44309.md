## Summary

Keep open. Current main still has only conversation-oriented `sessions_send` behavior: no dispatch intent in the tool schema, fire-and-forget still schedules A2A follow-up, and the A2A helper still performs requester reply-back plus target announce for normal peer sends.

## What This Changes

Request a first-class one-way dispatch/handoff mode for `sessions_send` that delivers to the target session without requester reply-back or final announce.

## Best Possible Solution

Add a backward-compatible per-call delivery intent to `sessions_send` so dispatch mode enqueues or starts the target session with a minimal acknowledgement while skipping requester reply-back and target announce.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `sessions_send` with `timeoutSeconds: 0` or a normal peer send, and current main still schedules `runSessionsSendA2AFlow` unless the narrow parent-owned skip applies.

## Solution Assessment

Yes. A per-call delivery intent is the narrow maintainable solution because prompt conventions, the global ping-pong limit, and the ACP parent-owned skip do not express clean one-way handoff semantics.

## Repair Work Prompt

Implement a backward-compatible dispatch-only mode for `sessions_send` for https://github.com/openclaw/openclaw/issues/44309. Add an explicit per-call delivery intent, preferably `mode: "dispatch"` while preserving the current conversation behavior as the default. Dispatch mode should persist/inject the message into the target session and start the target run, return a minimal accepted/ack result, and skip requester reply-back, ping-pong reinjection, and target announce. Do not convert `session.agentToAgent.maxPingPongTurns: 0` into dispatch semantics, do not change global defaults, and preserve the existing parent-owned ACP/native subagent skip behavior. Add focused tests for default conversation behavior, `timeoutSeconds: 0` dispatch behavior, waited dispatch behavior if supported, and no announce/send/reply-back side effects in dispatch mode. Update user docs for `sessions_send` and add a user-facing CHANGELOG entry.
