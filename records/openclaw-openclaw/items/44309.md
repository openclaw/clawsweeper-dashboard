## Summary

Keep open. Current main and v2026.5.12 still expose `sessions_send` as conversation-style A2A with reply-back and announce behavior, while the linked private-delivery PR is still open and only covers part of the requested dispatch semantic.

## What This Changes

Request adding an explicit one-way `sessions_send` dispatch or handoff mode that delivers work to a target session without reply-back ping-pong or announce-back behavior.

## Best Possible Solution

Define a backward-compatible dispatch intent in the A2A handoff contract, with explicit tool schema, optional delivery ack semantics, docs, and regression tests that preserve existing conversation mode.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `sessions_send` has no dispatch parameter and ordinary peer sends still enter the A2A reply/announce flow. This is a feature/API gap rather than a broken existing documented contract.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should align with the open private-delivery and handoff-protocol work instead of relying on prompts or `timeoutSeconds`.
