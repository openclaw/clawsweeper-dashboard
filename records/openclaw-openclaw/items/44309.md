## Summary

Keep open. Current main and v2026.5.20 still expose conversation-oriented `sessions_send` semantics without a caller-selected dispatch mode, and the overlapping private-delivery and handoff-protocol work is still open rather than merged.

## What This Changes

The issue requests a first-class one-way A2A dispatch or handoff mode for `sessions_send` so a task can be delivered to a target session without reply-back ping-pong or announce behavior.

## Best Possible Solution

Define a backward-compatible A2A dispatch intent or primitive with explicit tool schema, optional delivery acknowledgement semantics, docs, and regression tests while preserving the existing conversation mode.

## Reproduction Assessment

Yes for source-level behavior: current main has no dispatch parameter or primitive, and ordinary peer sends still enter the A2A reply/announce path. I did not run a live multi-agent transport repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape. A first-class dispatch mode or primitive is the right class of solution, but it should be aligned with https://github.com/openclaw/openclaw/pull/50720, https://github.com/openclaw/openclaw/pull/63789, and the related A2A issues rather than added as prompt-only behavior.
