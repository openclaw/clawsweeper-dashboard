## Summary

Current main still couples cross-session delivery to the reply-back and announce lifecycle, and the related open reports address narrower duplicate, announce, or routing problems rather than providing the requested one-way contract. Keep this issue open for a maintainer product decision on the canonical model-facing API.

## What This Changes

Add an explicit one-way agent-to-agent dispatch semantic that persists work in the target session without reply-back ping-pong or requester announce behavior.

## Best Possible Solution

Define one canonical typed A2A delivery intent at the existing session-tool boundary, preserve conversational delivery as the default, and add focused coverage proving that dispatch writes only to the target session without requester reinjection or an announce turn.

## Reproduction Assessment

Yes, at source level: current main has no per-call one-way delivery intent, and a cross-session reply enters the reply-back lifecycle before the announce turn. A live run was not needed to establish that the requested contract is absent.

## Solution Assessment

The desired separation between conversation and dispatch is sound, but the issue intentionally leaves multiple API shapes open. The narrowest maintainable solution is one typed owner-boundary contract rather than prompt tokens, magic strings, or overloading timeout behavior.
