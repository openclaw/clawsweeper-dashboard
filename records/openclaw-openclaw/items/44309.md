## Summary

Keep open: current main and the latest release still implement ordinary `sessions_send` as conversation-style A2A delivery with reply-back and announce, while this issue asks for a new explicit one-way dispatch contract. The related reverse-send bug and cron-session delivery request overlap, but neither supersedes this broader product/API decision.

## What This Changes

Request one-way dispatch semantics for A2A `sessions_send` handoffs so target sessions can receive work without reply-back reinjection or final announce.

## Best Possible Solution

Keep this as the broader design item for an explicit typed one-way A2A dispatch contract that preserves current conversation behavior by default and coordinates with the reverse-call and cron-session follow-ups.

## Reproduction Assessment

Yes, source-level: current main has no dispatch selector and ordinary peer sends still enter the A2A reply-back plus announce helper. I did not run a live two-agent handoff in this read-only review.

## Solution Assessment

Unclear on the exact API shape: a typed per-send dispatch mode or separate primitive is likely the right owner boundary, but maintainers need to choose the permanent contract and compatibility story.
