## Summary

Keep this PR open for maintainer review: current main still does not provide iOS ElevenLabs STT, but this branch has blocking realtime STT reliability issues, missing real-device proof, and a privacy-sensitive client-side microphone streaming design that needs explicit direction.

## What This Changes

The PR adds direct ElevenLabs realtime and batch STT clients to iOS Talk Mode, routes TalkModeManager audio through them when an ElevenLabs key is present, and adds Settings controls plus Swift source-list entries.

## Best Possible Solution

Implement iOS realtime STT through the gateway-owned Talk transcription/provider contract or an explicitly tokenized client flow, with robust readiness, commit, fallback, and PTT handling plus real-device proof.

## Reproduction Assessment

Yes for the review findings: PR-branch source inspection, current-main iOS Talk code, and the ElevenLabs/current-main provider contract show the commit framing, PTT teardown, readiness, and fallback problems. No real iOS device reproduction was run.

## Solution Assessment

No. A direct client-side API-key path that automatically streams microphone audio to ElevenLabs is not the narrowest maintainable solution without maintainer approval; the gateway relay/provider contract or client tokens are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Send manual commits as audio chunk frames:** `apps/ios/Sources/Voice/ElevenLabsRealtimeSTTClient.swift:71`
  - body: The realtime API publish shape and current main's provider use `input_audio_chunk` with `commit: true`; sending a standalone `message_type: "commit"` can leave PTT waiting for a committed transcript that never arrives. ([elevenlabs.io](https://elevenlabs.io/docs/api-reference/speech-to-text/v-1-speech-to-text-realtime?utm_source=openai))
  - confidence: 0.9
- **[P1] Wait for the committed PTT transcript:** `apps/ios/Sources/Voice/TalkModeManager.swift:397-398`
  - body: After `commit()`, this sleeps for 300 ms, then clears listening and stops recognition. A committed transcript that arrives after that window is ignored by `handleTranscript`, so PTT can submit an empty or truncated partial.
  - confidence: 0.9
- **[P1] Request Speech permission before Apple fallback:** `apps/ios/Sources/Voice/TalkModeManager.swift:618-620`
  - body: After realtime STT hits the retry limit, the code switches to Apple Speech and restarts recognition without requesting Speech authorization. Users who started on ElevenLabs may never have granted that permission, so fallback can fail instead of recovering input.
  - confidence: 0.86
- **[P1] Recover push-to-talk after realtime errors:** `apps/ios/Sources/Voice/TalkModeManager.swift:620`
  - body: The realtime error handler always calls `restartRecognitionAfterError()`, but that helper returns unless capture mode is continuous. In push-to-talk, socket or auth failures can leave the active capture with no recovered transcript or fallback.
  - confidence: 0.84
- **[P2] Mark the socket ready only after server confirmation:** `apps/ios/Sources/Voice/ElevenLabsRealtimeSTTClient.swift:47-48`
  - body: `isConnected` flips true immediately after `task.resume()`, before the WebSocket upgrade or `session_started` event confirms readiness. Audio and commits can be sent into a socket that never became usable. ([elevenlabs.io](https://elevenlabs.io/docs/developers/guides/cookbooks/speech-to-text/realtime/event-reference?utm_source=openai))
  - confidence: 0.84
