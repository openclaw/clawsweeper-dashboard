## Summary

Keep this PR open for maintainer review: it is a real iOS Talk feature proposal, but the current branch has blocking realtime-STT correctness issues, missing real-device proof, a live merge conflict, and a privacy/architecture decision around direct client-side ElevenLabs microphone streaming.

## What This Changes

The PR adds iOS ElevenLabs realtime and batch STT clients, a WAV encoder, TalkModeManager routing, Settings language/model controls, and Swift source-list entries.

## Best Possible Solution

Land a maintainer-approved iOS Talk STT design that uses the gateway/provider boundary or explicit client tokens, handles commit/readiness/fallback robustly, and includes focused tests plus redacted real-device proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR branch, current main's Talk path, and the realtime STT/provider contracts show the commit framing, PTT teardown, fallback-permission, PTT recovery, and readiness blockers. No real-device reproduction was run.

## Solution Assessment

No. The direct client-side API-key path with automatic microphone streaming is not the best maintainable shape without maintainer approval; a gateway relay/provider contract or tokenized native flow would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Send manual commits as audio chunk frames:** `apps/ios/Sources/Voice/ElevenLabsRealtimeSTTClient.swift:71`
  - body: The realtime client sends a standalone `message_type: "commit"` frame. Manual PTT can wait for a committed transcript that the provider never emits if the API expects commit on an audio-chunk frame, so release can return an empty or stale partial.
  - confidence: 0.9
- **[P1] Wait for the committed PTT transcript:** `apps/ios/Sources/Voice/TalkModeManager.swift:397-398`
  - body: After `commit()`, the code sleeps for 300 ms and then clears listening state and stops recognition. If the committed transcript arrives after that window, `handleTranscript` ignores it and PTT can submit an empty or truncated transcript.
  - confidence: 0.9
- **[P1] Request Speech permission before Apple fallback:** `apps/ios/Sources/Voice/TalkModeManager.swift:618-620`
  - body: After realtime STT exhausts retries, the code switches to Apple Speech and restarts recognition without first requesting Speech authorization. Users who started on ElevenLabs may not have granted that permission, so fallback can fail instead of recovering input.
  - confidence: 0.86
- **[P1] Recover push-to-talk after realtime errors:** `apps/ios/Sources/Voice/TalkModeManager.swift:620`
  - body: The realtime error handler always calls `restartRecognitionAfterError()`, but that helper returns unless capture mode is continuous. In push-to-talk, socket or auth failures can leave the active capture with no recovered transcript or fallback.
  - confidence: 0.84
- **[P2] Mark the socket ready only after server confirmation:** `apps/ios/Sources/Voice/ElevenLabsRealtimeSTTClient.swift:47-48`
  - body: `isConnected` flips to true immediately after `task.resume()`, before the WebSocket upgrade or first server event succeeds. Audio and commits can be sent into a socket that never became usable, delaying failure and losing speech input.
  - confidence: 0.84
