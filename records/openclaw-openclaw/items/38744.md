## Summary

Keep open. Current main has the Gateway/plugin ElevenLabs realtime transcription path, but this PR still proposes unique native iOS wiring and has blocking source-visible runtime, security/privacy, and proof gaps before it is mergeable.

## What This Changes

The PR adds native iOS ElevenLabs realtime and batch Speech-to-Text clients, Settings STT model/language selectors, and TalkModeManager wiring for ElevenLabs audio streaming and Apple Speech fallback.

## Best Possible Solution

Land a maintainer-approved iOS Talk design that either reuses the Gateway-owned transcription/realtime contracts or adds an explicit-opt-in native STT path with tokenized/client-safe auth, robust readiness and PTT commit handling, fallback recovery, docs, tests, and redacted real-device proof.

## Reproduction Assessment

Yes for the PR review blockers from source inspection: the diff shows the wrong manual commit message shape, fixed 300 ms PTT wait, optimistic ready flag, fallback without Speech permission, and PTT error recovery routed through a continuous-only helper. No live iOS runtime proof is attached.

## Solution Assessment

No. The automatic direct native vendor WebSocket path is not the best merge shape until maintainers approve the privacy and architecture boundary; current Gateway relay/provider contracts point to a server-owned or explicit-opt-in design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Send ElevenLabs commits as input audio chunks:** `apps/ios/Sources/Voice/ElevenLabsRealtimeSTTClient.swift:65-66`
  - body: The realtime API commits manual transcription by sending an `input_audio_chunk` with `commit: true`; this method sends a separate `message_type: "commit"` frame, so PTT release can wait forever for a committed transcript that the provider never produces. ([elevenlabs.io](https://elevenlabs.io/docs/api-reference/speech-to-text/v-1-speech-to-text-realtime))
  - confidence: 0.9
- **[P1] Wait for committed PTT transcripts before teardown:** `apps/ios/Sources/Voice/TalkModeManager.swift:397-398`
  - body: The ElevenLabs PTT stop path sends commit, sleeps a fixed 300 ms, then clears listening state and stops recognition. If the committed transcript arrives later, `handleTranscript` ignores it and PTT can submit an empty or truncated partial transcript.
  - confidence: 0.9
- **[P1] Request Speech permission before Apple fallback:** `apps/ios/Sources/Voice/TalkModeManager.swift:618-620`
  - body: When realtime STT reaches the retry limit, this path flips to Apple Speech and restarts recognition without requesting Speech permission. Users who started on ElevenLabs may never have granted that permission, so fallback can fail instead of recovering input.
  - confidence: 0.86
- **[P1] Recover push-to-talk after realtime STT errors:** `apps/ios/Sources/Voice/TalkModeManager.swift:620`
  - body: The realtime error handler always calls `restartRecognitionAfterError()`, but that helper returns unless capture mode is continuous. In push-to-talk mode, socket or auth failures can leave the active capture without a recovered transcript or immediate Apple fallback.
  - confidence: 0.84
- **[P2] Mark the WebSocket ready after server confirmation:** `apps/ios/Sources/Voice/ElevenLabsRealtimeSTTClient.swift:48`
  - body: `isConnected` becomes true immediately after `task.resume()`, before the WebSocket upgrade or first ElevenLabs server message succeeds. Callers can send audio and commits to a socket that never became usable, delaying failure and losing speech input.
  - confidence: 0.84
