## Summary

Keep open: current main does not implement native iOS ElevenLabs realtime STT, and the PR still has blocking reliability findings plus an unresolved product/security choice around direct vendor microphone streaming despite existing gateway/plugin realtime surfaces.

## What This Changes

The branch adds native iOS ElevenLabs realtime and batch STT clients, PCM/WAV conversion, TalkModeManager integration, STT settings, and Swift source-list entries.

## Best Possible Solution

Choose the Talk architecture first: prefer the existing Gateway/provider realtime relay, or explicitly approve native iOS ElevenLabs STT with opt-in, robust readiness/fallback handling, docs, changelog, and focused iOS coverage.

## Reproduction Assessment

Yes for the PR findings from source inspection: the provided diff shows the fixed 300 ms PTT commit window, premature websocket ready flag, and fallback paths. Real-device iOS audio behavior still needs live validation before merge.

## Solution Assessment

No. The proposed direct native vendor STT path is a useful prototype, but it is not the narrowest maintainable solution until maintainers choose it over the existing Gateway realtime relay and harden privacy and failure behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Wait for committed PTT transcripts before stopping:** `apps/ios/Sources/Voice/TalkModeManager.swift:392-395`
  - body: The ElevenLabs PTT stop path sends `commit()` and then waits a fixed 300 ms before setting listening false and tearing recognition down. If `committed_transcript` arrives after that window, the final transcript is ignored or the receive task is already gone, so PTT can queue an empty or truncated partial.
  - confidence: 0.9
- **[P1] Recover Apple fallback after realtime STT failures:** `apps/ios/Sources/Voice/TalkModeManager.swift:623-627`
  - body: After realtime STT errors, the fallback path flips toward Apple Speech but restarts through logic that only reliably handles continuous capture and may not request Apple Speech permission for users who started on ElevenLabs. Invalid keys or realtime access failures can leave speech input broken instead of degrading to the existing Apple path.
  - confidence: 0.86
- **[P2] Mark the WebSocket ready only after the server handshake:** `apps/ios/Sources/Voice/ElevenLabsRealtimeSTTClient.swift:47-48`
  - body: `isConnected` is set immediately after `task.resume()`, before the WebSocket upgrade or ElevenLabs `session_started` message succeeds. With an invalid key or unavailable service, callers can enqueue audio and commits onto a socket that has not become usable, delaying failure and losing speech input.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `apps/ios/Sources/Settings/SettingsTab.swift:339`
  - body: This adds a user-facing iOS feature and settings surface, but the PR file list does not include `CHANGELOG.md`. Repo policy requires user-facing `feat` changes to be represented in release notes.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Needs attention because the diff auto-enables direct iOS microphone streaming to ElevenLabs from an existing gateway API key rather than an explicit opt-in or approved gateway-relay design.

Concerns:

- **[medium] Add explicit opt-in before streaming mic audio to ElevenLabs:** `apps/ios/Sources/Voice/TalkModeManager.swift:2128`
  - body: The PR enables ElevenLabs STT whenever the gateway Talk API key is configured, so a key intended for voice output can also cause live microphone audio to be sent directly from iOS to ElevenLabs. That sensitive data-flow change needs explicit user-facing opt-in or a maintainer-approved Gateway relay design before merge.
