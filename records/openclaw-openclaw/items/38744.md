## Review Findings

Overall confidence: 0.91

Full review comments:

- **[P1] Route iOS STT through the Gateway Talk contract:** `apps/ios/Sources/Voice/TalkModeManager.swift:588-600`
  - body: This adds a second ElevenLabs transcription stack inside the iOS app and automatically selects it when an ElevenLabs key exists, bypassing the current Gateway-owned Talk session/provider contract. That creates conflicting credential, model, readiness, fallback, and session-event behavior for the same Talk feature.
  - confidence: 0.9
- **[P1] Wait for the committed PTT transcript before teardown:** `apps/ios/Sources/Voice/TalkModeManager.swift:396-402`
  - body: After commit(), the code sleeps for a fixed 300 ms and then clears isListening and stops recognition. If the committed transcript arrives later, handleTranscript ignores it or the receive loop is already canceled, so PTT can submit an empty or stale partial transcript.
  - confidence: 0.92
- **[P1] Recover active captures after realtime STT failures:** `apps/ios/Sources/Voice/TalkModeManager.swift:613-620`
  - body: Realtime WebSocket or auth failures only call the continuous restart path, so push-to-talk captures do not reliably recover in place. A user with an expired or unsupported realtime key can lose the active speech attempt instead of degrading to Apple Speech.
  - confidence: 0.86
- **[P1] Request Apple Speech permission before fallback:** `apps/ios/Sources/Voice/TalkModeManager.swift:618-620`
  - body: The ElevenLabs-first start path skips SFSpeechRecognizer authorization, but the retry-limit fallback flips back to Apple Speech and restarts recognition without requesting speech permission. First-time users can therefore fail both the realtime path and the fallback path.
  - confidence: 0.84
- **[P2] Do not mark the WebSocket connected before readiness:** `apps/ios/Sources/Voice/ElevenLabsRealtimeSTTClient.swift:47-48`
  - body: isConnected is set immediately after task.resume(), before the WebSocket handshake or ElevenLabs session_started message. Audio and commit sends can therefore run against a socket that is still connecting or already rejected, causing speech frames to be silently dropped until receiveLoop reports an error.
  - confidence: 0.82
