## Summary

Keep open: the transport hook is the right place for this Telegram bug, but the patch only handles config-sourced non-ipv4first DNS and still misses other explicit IPv6-capable override paths; it also has only mocked/build proof and would close the wrong sibling issue from the PR body.

## What This Changes

The PR gates Telegram sticky IPv4 fallback when `dnsResultOrder` comes from config and is not `ipv4first`, and adds a fetch unit test for `dnsResultOrder: "verbatim"`.

## Best Possible Solution

Land a narrow Telegram transport fix that treats all explicit IPv6-capable config/env overrides as opt-outs from sticky IPv4 fallback, preserves default and `ipv4first` recovery, corrects the closing reference, and includes live or diagnostic proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 build IPv4-only fallback attempts after resolving explicit Telegram network decisions, and media downloads consume those attempts. I did not run a live IPv4-broken, IPv6-working Telegram host in this read-only review.

## Solution Assessment

No. The transport policy hook is the right layer, but the proposed guard is too narrow until it covers autoSelectFamily-only config and documented env DNS overrides, while preserving default and `ipv4first` recovery.
