## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not require the optional compose override:** `scripts/docker/openai-codex-auth.sh:7`
  - body: The helper always includes `-f docker-compose.extra.yml`, but that file is not in the repo and the Docker docs describe it as an optional override only when it exists. In a normal checkout this exits before the auth flow can start, so the new Docker Codex OAuth helper is unusable by default.
  - confidence: 0.94
- **[P1] Use the canonical OpenAI auth provider:** `scripts/docker/openai-codex-auth.sh:8`
  - body: Current OpenClaw docs route ChatGPT/Codex OAuth through `--provider openai` and say `openai-codex` is legacy migration input, but this new helper creates a fresh live login path for the legacy provider. That can write stale profiles that doctor later has to repair and does not match the supported Codex auth route.
  - confidence: 0.92
- **[P2] Match history responses by request session:** `extensions/whatsapp/src/inbound/monitor.ts:701`
  - body: Baileys returns a request/session id from `fetchMessageHistory` and includes it on `messaging-history.set`, but this code ignores it and resolves any pending request for the same chat JID. Concurrent requests or unrelated history sync chunks for the same chat can satisfy the wrong promise and return stale or unrelated messages.
  - confidence: 0.88
- **[P2] Preserve upload-file when rebasing WhatsApp actions:** `extensions/whatsapp/src/channel.ts:140-142`
  - body: Current main already supports `upload-file` as a WhatsApp gateway action, but the PR head replaces the action predicate with only `react` and `history`. Because the branch is conflicting, the rebase must compose history with current `upload-file` support instead of dropping WhatsApp media uploads.
  - confidence: 0.84
