## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make WhatsApp history opt in before advertising it:** `extensions/whatsapp/src/channel-actions.ts:83-84`
  - body: `createActionGate` defaults missing action keys to enabled, and this PR adds `history` as an optional config key. Existing WhatsApp configs would advertise recent-history access to agents unless users explicitly disable it, so this needs an explicit opt-in/default-false policy before exposure.
  - confidence: 0.94
- **[P2] Correlate history responses with Baileys request ids:** `extensions/whatsapp/src/inbound/monitor.ts:679`
  - body: Baileys `fetchMessageHistory` returns a request id and `messaging-history.set` includes `peerDataRequestSessionId`, but this code stores one pending request per chat JID. Concurrent requests or unrelated sync chunks for the same chat can resolve the wrong caller; key pending history by the returned request id and match the event id before resolving.
  - confidence: 0.89
- **[P2] Use the canonical OpenAI provider in the Docker helper:** `scripts/docker/openai-codex-auth.sh:8`
  - body: Current main and docs treat `openai-codex` as legacy migration input, while bare `--provider openai` starts ChatGPT/Codex OAuth. A new Docker helper should not create fresh legacy-profile/auth-order state; route through `--provider openai` and any current method flag needed for the intended OAuth flow.
  - confidence: 0.91
- **[P2] Make the Docker OAuth callback actually reachable:** `docker-compose.yml:32`
  - body: The PR publishes host port 1455, but the OpenAI OAuth callback code binds a loopback-only `localhost` listener and rejects non-loopback callback hosts. A host browser hitting `localhost:1455` is not proven to reach that container listener; either document the manual-paste/device flow or add a tested host-reachable callback path without widening the auth listener unsafely.
  - confidence: 0.78
