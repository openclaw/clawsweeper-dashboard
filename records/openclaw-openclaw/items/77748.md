## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Make WhatsApp history opt in before advertising it:** `extensions/whatsapp/src/channel-actions.ts:83`
  - body: `createActionGate` defaults missing action keys to enabled, and this PR adds `history` as an optional config key. That means existing WhatsApp configs would advertise recent-history access to agents unless users explicitly disable it, which is too sensitive for an upgrade default; gate this with an explicit opt-in/default-false policy before exposing the action.
  - confidence: 0.94
- **[P2] Correlate history responses with Baileys request ids:** `extensions/whatsapp/src/inbound/monitor.ts:679`
  - body: Baileys `fetchMessageHistory` returns a request id and `messaging-history.set` includes `peerDataRequestSessionId`, but this code stores one pending request per chat JID and resolves the first matching chat batch. Concurrent requests or unrelated sync chunks for the same chat can satisfy the wrong caller; key pending history by the returned request id and match the event id before resolving.
  - confidence: 0.89
- **[P2] Use the canonical OpenAI provider in the Docker helper:** `scripts/docker/openai-codex-auth.sh:8`
  - body: Current main and docs treat `openai-codex` as legacy migration input, while bare `--provider openai` starts ChatGPT/Codex OAuth. A new Docker helper should not create fresh legacy-profile/auth-order state; route through `--provider openai` and any current method flag needed for the intended OAuth flow.
  - confidence: 0.91
