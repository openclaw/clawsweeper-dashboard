## Summary

Keep open: current main still renders the Chat page header session picker without the switch callback, so picker option clicks close the popover but route through the default no-op handler instead of changing the active chat.

## What This Changes

The issue requests that the ControlUI Chat page’s right-side conversation selector switch the active conversation like the left sidebar recent list does.

## Best Possible Solution

Wire the Chat header session selector to the existing `switchChatSession` path and add regression coverage at the app-header wiring level, without changing picker search/filter semantics.

## Reproduction Assessment

Yes, source-reproducible: the current Chat header omits the switch callback while the picker option click depends on that callback, which defaults to a no-op. I did not run a live browser reproduction in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to pass the existing `switchChatSession` handler into the header’s `renderChatSessionSelect` call and cover that app-level wiring. A new session-switching path or config option would be unnecessary.
