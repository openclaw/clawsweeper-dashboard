## Summary

Keep open: the PR is useful but still does not fix the reported default `openclaw status` path, because that path disables live channel status and setup fallback, and the new merge only runs when a gateway `channels.status` payload exists. The synthesized row logic also treats an omitted optional `connected` field as disconnected/warn.

## What This Changes

The PR adds a status-table merge helper that synthesizes missing Channels rows from the gateway `channels.status` payload, wires it into status overview runtime, adds focused tests, and updates the changelog.

## Best Possible Solution

Land a revised PR that keeps default status fast while still providing configured channel rows, uses gateway synthesis for live/deep payloads, and treats optional connection state as unknown unless explicitly false.

## Reproduction Assessment

Yes. Related #73525 and #75318 provide command-level reproductions, and current main has source/test evidence showing plain text status disables both live channel status and setup fallback.

## Solution Assessment

No. The gateway merge is a useful live/deep-path addition, but it is not the narrow complete fix for default `openclaw status` and it misreads optional connection state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Cover the default status path:** `src/commands/status.scan-overview.ts:258-260`
  - body: The merge only runs when `channelsStatus` is non-null, but plain `openclaw status` currently passes `includeLiveChannelStatus: false` and `includeChannelSetupRuntimeFallback: false`. That means the reported fast-status case can still render a header-only Channels table while `--deep` or `channels status` has data, so the fix needs a command-level regression for the default path and a deliberate fast row source.
  - confidence: 0.92
- **[P2] Treat omitted connected as unknown:** `src/commands/status-all/channels.ts:236`
  - body: `ChannelAccountSnapshot.connected` is optional, and the existing `openclaw channels status` formatter only reports disconnected when the field is explicitly `false`. Requiring every account to have `connected === true` turns running/configured snapshots that omit the field into warn/disconnected rows even though the gateway did not report a disconnect.
  - confidence: 0.88
