## Summary

Keep open: this is a focused fix for a real status regression, but the PR head is not merge-ready because the new synthesis still does not run through the plain `openclaw status` caller and it treats an omitted optional connection field as disconnected. The supplied proof uses a real gateway, but the gateway snapshot is empty and the rendered Channels table remains header-only, so it does not demonstrate the non-empty fixed behavior.

## What This Changes

The branch adds a gateway-snapshot merge helper for status Channels rows, wires it into status overview runtime, adds focused tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Make the status Channels table contract explicit, then either wire a bounded live snapshot into the public default path or narrow the change to `--deep`/`--all`; preserve optional snapshot-field semantics and add scanStatus-level coverage plus non-empty real proof.

## Reproduction Assessment

Yes for the PR code path: source inspection shows plain `scanStatus` still disables live channel status, so the new merge does not run for default `openclaw status`. I did not establish a live configured Telegram reproduction in this read-only review.

## Solution Assessment

No. Gateway-backed synthesis may be useful, but this patch misses the public default command path named by the linked reports and misclassifies snapshots that omit optional connection state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire synthesis into the public status path:** `src/commands/status.scan-overview.ts:259-260`
  - body: The new merge only runs when `channelsStatus` exists, but plain `openclaw status` still passes `includeLiveChannelStatus: false` unless `--all` or `--deep` is used. That leaves the default-status regression from https://github.com/openclaw/openclaw/issues/73525 on the same read-only table path whenever discovery returns no rows, so this needs a public `scanStatus`-level fix/test or a narrower claim that it only changes deep/all status.
  - confidence: 0.91
- **[P2] Treat absent connection state as unknown:** `src/commands/status-all/channels.ts:298`
  - body: `ChannelAccountSnapshot.connected` is optional, and the existing `channels status` formatter only reports connection state when the field is boolean. Requiring every enabled account to have `connected === true` makes a running/configured snapshot that omits connection state synthesize a WARN/`disconnected` row, so only explicit `connected: false` should count as disconnected.
  - confidence: 0.88
