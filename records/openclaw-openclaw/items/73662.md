## Summary

Keep open; the PR is useful but not mergeable because it only augments live/deep gateway status rows, while the linked regression and current command path are plain fast `openclaw status`, and the synthesized state misclassifies snapshots that omit optional `connected`.

## What This Changes

The PR adds a status-table merge helper that fills missing Channels rows from the gateway `channels.status` snapshot, wires it into status overview collection, adds tests, and updates the changelog.

## Best Possible Solution

Land a revised PR that keeps plain status fast/read-only, makes configured channels render in the default Channels table, preserves live gateway augmentation for `--all`/`--deep`, and treats missing `connected` as unknown.

## Reproduction Assessment

Yes. The linked reports give a command-level reproduction, and current main confirms it structurally: default `scanStatus({ json: false })` disables both live `channels.status` and setup fallback while `--deep` enables them.

## Solution Assessment

No. The gateway merge is a useful live/deep augmentation, but it does not solve the default fast command path and it treats omitted optional connection state as a warning; a safer repair should use a fast read-only/setup row source for default status and only treat explicit `connected: false` as disconnected.
