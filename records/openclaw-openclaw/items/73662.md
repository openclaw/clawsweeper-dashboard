## Summary

Keep open: the latest head still misses the default `openclaw status` path and misclassifies snapshots with omitted connection state, and the supplied terminal proof does not show the non-empty synthesis behavior after the fix.

## What This Changes

The PR adds a Channels-table merge helper that synthesizes rows from `channels.status`, wires it into status overview/runtime, adds merge/overview tests, and updates the changelog.

## Best Possible Solution

Preserve the fast default status contract while ensuring configured/running channels populate consistently, either through the current setup fallback or a bounded gateway snapshot path with default-status and optional-connection coverage.

## Reproduction Assessment

Yes for the PR-head code path: default `openclaw status` still disables live channel status, so the new merge cannot run there. I did not verify a live configured Telegram/Discord node in this read-only review.

## Solution Assessment

No. Gateway-backed synthesis may be useful, but this patch does not cover the default command path named by the report and treats omitted optional connection state as disconnected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire synthesis into the default status path:** `src/commands/status.scan-overview.ts:259-261`
  - body: The new merge only runs when `channelsStatus` exists, but plain `openclaw status` still passes `includeLiveChannelStatus: false` unless `--all` or `--deep` is used. That leaves the default-status regression from https://github.com/openclaw/openclaw/issues/73525 on the same unmerged read-only rows whenever discovery returns empty, so add a `scanStatus`-level fix/test or narrow this PR's behavior claim to `--deep`/`--all`.
  - confidence: 0.91
- **[P2] Treat missing connection state as unknown:** `src/commands/status-all/channels.ts:298`
  - body: `ChannelAccountSnapshot.connected` is optional, and `openclaw channels status` only reports connection state when the field is boolean. Requiring every enabled account to have `connected === true` makes a running/configured snapshot that omits connection state synthesize WARN/`disconnected`, so only explicit `connected: false` should count as disconnected.
  - confidence: 0.88
