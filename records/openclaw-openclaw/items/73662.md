## Summary

Keep this PR open, but the current head should not merge yet: it adds a useful gateway-snapshot synthesis seam, but it does not reach the reported default `openclaw status` path and it treats omitted `connected` fields as disconnected.

## What This Changes

The PR adds `mergeChannelsTableWithGatewayStatus`, wires it into status overview collection, adds focused tests, and updates the changelog for the Channels table regression.

## Best Possible Solution

Revise the PR so plain `openclaw status` gets populated Channels rows for configured/running accounts while preserving the fast-status contract, and align synthesized health with the existing optional snapshot semantics. Add a command-level `scanStatus({ json: false })` regression test, not only an overview seam test.

## Reproduction Assessment

Yes. #73525 and #75318 give a command-level reproduction, and current main has a regression guard showing plain text status does not request live channel status or setup fallback.

## Solution Assessment

No. The gateway merge helper is a reasonable ingredient for reachable-gateway paths, but as wired it misses the reported default command path and is stricter about `connected` than the current snapshot/formatter contract.

Requiring every account to have `connected === true` makes running/configured snapshots that omit the field synthesize as `warn` with `disconnected` even though the gateway did not report a disconnect.
