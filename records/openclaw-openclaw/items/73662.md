## Summary

Keep open: this is a focused bug-fix PR, but it is not mergeable as written because the new synthesis only runs when live channel status is requested, leaving the default `openclaw status` path named by the linked report unchanged, and the helper treats an omitted optional `connected` field as disconnected. The supplied real behavior proof also does not show the non-empty channel snapshot/rendered row that the PR is meant to fix.

## What This Changes

The PR adds a CLI status Channels-table merge helper, wires it into status overview runtime, adds merge/overview tests, and updates the changelog.

## Best Possible Solution

Make the default-status contract explicit, then either wire the synthesis into the public default path with a bounded latency-safe call and `scanStatus` coverage, or narrow the PR to `--deep`/`--all`; in either case, treat absent connection state as unknown and require non-empty live proof.

## Reproduction Assessment

Yes for the code path: the PR head still disables live channel status for plain `openclaw status`, and the new merge only runs when `channelsStatus` exists. I did not establish a live configured Telegram reproduction in this read-only review.

## Solution Assessment

No. Gateway-backed synthesis may be useful, but this patch misses the default command path named by the report and misclassifies snapshots that omit optional connection state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire synthesis into the public status path:** `src/commands/status.scan-overview.ts:259-260`
  - body: The new merge only runs when `channelsStatus` exists, but plain `openclaw status` still passes `includeLiveChannelStatus: false` unless `--all` or `--deep` is used. That leaves the default-status regression from https://github.com/openclaw/openclaw/issues/73525 on the same read-only table path whenever discovery returns no rows, so this needs a public `scanStatus`-level fix/test or a narrower claim that it only changes deep/all status.
  - confidence: 0.91
- **[P2] Treat absent connection state as unknown:** `src/commands/status-all/channels.ts:298`
  - body: `ChannelAccountSnapshot.connected` is optional, and the existing `channels status` formatter only reports connection state when the field is boolean. Requiring every enabled account to have `connected === true` makes a running/configured snapshot that omits connection state synthesize a WARN/`disconnected` row, so only explicit `connected: false` should count as disconnected.
  - confidence: 0.88
