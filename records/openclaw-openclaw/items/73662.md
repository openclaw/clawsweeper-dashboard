## Summary

Keep open: this PR still has a P2 correctness issue around optional channel connection state and lacks the required after-fix proof from a real gateway setup. Current main now keeps default status on the setup-fallback row source, so the branch's remaining useful scope is live/deep gateway augmentation rather than the original default-status regression.

## What This Changes

The branch adds a status-table merge helper, wires it into status overview collection, adds regression tests, and updates the changelog so missing Channels rows can be synthesized from `channels.status`.

## Best Possible Solution

Revise the PR to treat absent gateway connection state as unknown, keep default `openclaw status` on the current fast setup-fallback path, and add redacted real gateway proof before merge.

## Reproduction Assessment

Yes, for the narrow source path: current main can receive a live `channels.status` payload but still builds visible rows only through `buildChannelsTable`. I did not establish a real Telegram gateway reproduction on this checkout.

## Solution Assessment

No, not yet. The live/deep augmentation is a reasonable direction, but the patch should not turn missing optional `connected` data into a warning and it still needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Treat missing connection state as unknown:** `src/commands/status-all/channels.ts:298`
  - body: `ChannelAccountSnapshot.connected` is optional, and the existing `channels status` formatter only reports connection state when the field is boolean.
