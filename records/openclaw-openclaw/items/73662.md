## Summary

Keep open. Current main does not implement the PR's intended fix, and the PR still has correctness blockers: the synthesized rows are only used when a live gateway snapshot exists, while the reported plain `openclaw status` path still disables that snapshot, and the helper treats an omitted optional `connected` field as a disconnect.

## What This Changes

The PR adds a Channels table merge helper that synthesizes missing rows from the live `channels.status` gateway snapshot, wires it into status overview collection, adds focused tests, and updates the changelog.

## Best Possible Solution

Update this PR or replace it with a narrower fix that makes plain `openclaw status` show configured/running channels without regressing the fast path, aligns synthesized health with optional snapshot fields, and adds a command-level `scanStatus({ json: false })` regression test for the related reports.

## Reproduction Assessment

Yes. #73525 and #75318 give command-level reproductions, and current main has a test that preserves the exact default path where live channel status and setup fallback are disabled.

## Solution Assessment

No. The gateway merge helper is a useful ingredient for live/deep paths, but as wired it misses the reported default command path and is stricter about `connected` than the existing snapshot and formatter contract.

## Review Findings

Overall correctness: patch is incorrect

  - body: This merge only runs when `channelsStatus` is non-null, but plain `openclaw status` still disables live channel status and setup fallback. That is the reported path in #73525/#75318, so the command can still render a header-only Channels table unless the default path gets a row source or a command-level regression test proves otherwise.
  - confidence: 0.94

Requiring `connected === true` makes otherwise running/configured snapshots that omit the field synthesize as warn/disconnected without the gateway reporting a disconnect.
  - confidence: 0.88
