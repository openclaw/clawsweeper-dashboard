## Summary

Keep open: the PR is relevant but not mergeable because it only augments rows when live `channels.status` is queried, while the reported fast `openclaw status` path keeps both live channel status and setup fallback disabled; it also mislabels snapshots that omit optional `connected` state.

## What This Changes

The branch adds a status-table helper that augments read-only channel rows from the live gateway `channels.status` snapshot, wires it into status overview collection, adds tests, and updates the changelog.

## Best Possible Solution

Land a revised status fix that keeps default status fast, populates configured channel rows from a non-live/setup-aware source, and uses the gateway snapshot as an additive live/deep augmentation.

## Reproduction Assessment

Yes. The related reports give command-level steps, and current main structurally reproduces the gap: default `scanStatus({ json: false })` disables both live channel status and setup fallback, while `--deep` enables them.

## Solution Assessment

No. The gateway merge is a useful augmentation for live/deep status, but the narrow maintainable fix also needs a fast default row source and should treat omitted `connected` as unknown rather than disconnected.
