## Summary

Keep open: the PR is a plausible direction, but it still does not fix the default `openclaw status` path described by the linked report, and the supplied proof uses an empty gateway snapshot rather than showing a non-empty synthesized channel row.

## What This Changes

The PR adds a gateway-status merge helper for the CLI Channels table, wires it into status overview runtime, adds merge/overview tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep the fast default contract explicit while making visible Channels rows match configured/running accounts, with public `scanStatus` coverage and redacted live proof for a non-empty channel snapshot.

## Reproduction Assessment

Yes, source-reproducible: current main keeps default text status off `channels.status`, while the PR only synthesizes rows when that payload exists. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No. Gateway-backed synthesis can help `--deep`/`--all`, but this patch misses the default command path named by the report and treats an absent optional `connected` field as disconnected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire synthesis into the public status path:** `src/commands/status.scan-overview.ts:257-259`
  - body: The new merge only runs when `channelsStatus` exists, but plain `openclaw status` still passes `includeLiveChannelStatus: false` unless `--all` or `--deep` is used. That leaves the linked default-status regression on the same read-only `buildChannelsTable` path whenever discovery returns no rows, so the PR needs a public `scanStatus`-level fix/test or a narrower claim that this only changes deep/all status.
  - confidence: 0.91
- **[P2] Treat absent connection state as unknown:** `src/commands/status-all/channels.ts:298`
  - body: `ChannelAccountSnapshot.connected` is optional, and the existing `channels status` formatter only reports connection state when the field is boolean. Requiring every enabled account to have `connected === true` makes a running/configured snapshot that omits connection state synthesize a WARN/`disconnected` row, so only explicit `connected: false` should count as disconnected.
  - confidence: 0.88
