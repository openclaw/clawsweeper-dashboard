## Summary

Keep open: this PR targets a real `openclaw status` Channels regression, but the current patch should not merge because it misses the default fast command path and misclassifies omitted gateway connection state.

## What This Changes

The PR adds a helper that augments status Channels rows from `channels.status`, wires it into status overview collection, adds focused tests, and updates the changelog.

## Best Possible Solution

Land a revised status fix that keeps plain status fast while still showing configured/running channels consistently, uses current option names, treats optional snapshot fields as unknown, and adds command-level regression coverage.

## Reproduction Assessment

Yes. The linked reports give command-level steps, and current main has a focused test showing plain `scanStatus({ json: false })` disables live channel status and setup fallback.

## Solution Assessment

No. The gateway merge helper is useful for live/deep paths, but the current PR does not solve the reported default fast path and uses too-strict optional `connected` semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the current setup fallback option:** `src/commands/status.scan-overview.ts:260`
  - body: Current main wires this call through `includeSetupFallbackPlugins`; this branch still carries the older `includeSetupRuntimeFallback` name in the overview path. When the PR is rebased, keep the current option name so the fast-path false setting is actually honored.
  - confidence: 0.86
- **[P2] Cover the default status command path:** `src/commands/status.scan-overview.ts:262-264`
  - body: The merge only runs when `channelsStatus` is non-null, but plain `openclaw status` passes `includeLiveChannelStatus: false`, so the reported fast path still never synthesizes rows. Add a `scanStatus({ json: false })` regression and make the chosen fast row source explicit.
  - confidence: 0.91
- **[P2] Treat missing connected state as unknown:** `src/commands/status-all/channels.ts:236`
  - body: `ChannelAccountSnapshot.connected` is optional, and existing status output only reports disconnected when the field is explicitly `false`. Requiring `connected === true` makes running/configured snapshots that omit the field synthesize a warning without a reported disconnect.
  - confidence: 0.89
