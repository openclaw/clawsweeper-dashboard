## Summary

Keep open. The patch looks directionally correct and CI is green, but the external-PR real behavior proof is still insufficient because the shown live gateway payload has empty `channelAccounts` and the rendered Channels table remains header-only, so it does not prove the non-empty synthesis path after the fix.

## What This Changes

The PR adds a status Channels-table merge helper that synthesizes missing rows from `channels.status.channelAccounts`, wires it into `collectStatusScanOverview`, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the focused gateway-snapshot synthesis if maintainers want live/deep status parity, but require after-fix proof with a real configured channel before merge.

## Reproduction Assessment

Yes, source-level. On current main, a live status path can receive `channels.status` while the visible Channels rows still come from `buildChannelsTable`; a gateway payload with accounts plus empty read-only rows would stay empty without the PR's merge helper.

## Solution Assessment

Mostly yes. Synthesizing only missing rows from the existing gateway snapshot is a narrow fix for the live-status mismatch, but the PR still needs real proof that the non-empty path works on an actual configured channel.
