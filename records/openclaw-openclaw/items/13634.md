## Summary

Keep open: current main and the latest release cover the workspace-file part through shared Project Context, but Slack still lacks a supported account/global `systemPrompt` contract. Prior Slack prompt PR history confirms the need was considered valid, but the exposed API shape was rejected rather than implemented.

## What This Changes

The issue asks for Slack-wide prompt inheritance plus workspace bootstrap file context parity so Slack sessions do not need repeated per-channel `systemPrompt` files.

## Best Possible Solution

Define the supported Slack prompt inheritance contract first; if accepted, implement it through typed config, strict schema, generated metadata, docs, normal-message handling, and native slash-command handling while keeping shared Project Context as the workspace-file mechanism.

## Reproduction Assessment

Yes for the missing requested capability: current source and the latest release show no Slack account/global `systemPrompt` field in the types, strict schema, docs, or runtime prompt resolution. The workspace-file half is already covered by shared Project Context, so a live Slack repro is not needed for that part.

## Solution Assessment

Unclear. The requested key is plausible, but a prior exact Slack global/DM prompt PR was closed as the wrong API shape and current main already has a partial room-channel wildcard prompt path, so maintainers should choose the public inheritance contract before implementation.
