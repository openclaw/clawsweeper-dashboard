## Summary

Current main still does not expose the requested Slack account/global `systemPrompt`; the workspace-file part is already handled by shared Project Context, but the remaining request is a user-facing config/API enhancement that needs maintainer-approved semantics.

## What This Changes

Add Slack-wide prompt inheritance, such as `channels.slack.systemPrompt`, and make Slack sessions receive workspace context with parity to Telegram.

## Best Possible Solution

Keep shared Project Context as the workspace-file mechanism, and add a documented Slack account/global prompt field with explicit inheritance and stacking semantics once maintainers approve the config/API shape.

## Reproduction Assessment

Yes for the missing feature surface: source inspection on current main shows Slack accepts `systemPrompt` only under `channels.slack.channels.*`, while account/global types, schema, docs, and prompt resolution do not expose `channels.slack.systemPrompt`. Live Slack reproduction is not needed to prove this config/API gap.

## Solution Assessment

Unclear until maintainer API review: shared Project Context is already the right workspace-file solution, but the remaining global Slack prompt should not reuse the previously closed local prompt-stacking patch without agreed inheritance and prompt-order semantics.
