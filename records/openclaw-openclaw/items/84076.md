## Summary

Keep open: current main still fail-fasts the missing `turn/completed` case, but it does not implement the user-visible recovery/status and safe resume semantics reported here; recent comments showing later healthy runs only show a wedged lane was cleared, not that the underlying recovery gap is fixed.

## What This Changes

The issue requests accurate visible recovery/status and safe retry or resume handling when Codex app-server goes quiet after `item/completed` without `turn/completed`.

## Best Possible Solution

Add a bounded Codex app-server recovery path that releases the lane, emits accurate visible status, preserves replay metadata, and only offers or performs retry when side-effect safety is known.

## Reproduction Assessment

Yes, as a source/log reproduction: reporter and follow-up logs show `item/completed` followed by missing `turn/completed`, `turn_completion_idle_timeout`, and `recovery=none`, and current main still has that timeout path. I did not run the live provider/channel scenario in this read-only review.

## Solution Assessment

Unclear as a complete fix: visible timeout/status is a straightforward bug fix, but automatic resume or retry after completed tools requires a maintainer decision around replay safety and side effects.
