## Summary

Keep open: current main still has the Codex app-server timeout/fail-fast path, but the reported recovery/status gap is not fully implemented and is now the canonical issue for the active app-server recovery work.

## What This Changes

The issue requests visible recovery/status and retry-safe handling when a Codex app-server turn times out after `item/completed` without `turn/completed`.

## Best Possible Solution

Keep this issue as the canonical tracker and land a bounded recovery path that releases the lane, emits accurate user-visible timeout status, preserves replay metadata, and retries only when the attempt is known to be side-effect safe.

## Reproduction Assessment

Yes. Source inspection plus live reporter logs give a high-confidence reproduction shape: Codex app-server emits `item/completed`, omits `turn/completed`, and OpenClaw aborts via `turn_completion_idle_timeout`; I did not run the live provider/channel repro in this read-only review.

## Solution Assessment

Unclear as a full fix. A visible timeout/status path is a narrow bug fix, but automatic retry or resume after completed tools needs a maintainer decision around replay safety and side effects.
