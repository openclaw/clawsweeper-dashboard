## Summary

Keep open: current main has the prior fail-fast watchdogs for missing Codex `turn/completed`, but the remaining recovery/status behavior reported here is not implemented or clearly superseded by the closed adjacent fixes.

## What This Changes

The issue asks OpenClaw to recover or visibly report Codex app-server turns that stall after `item/completed` without a following `turn/completed`.

## Best Possible Solution

Add a bounded recovery path for Codex app-server completion-idle timeouts after completed current-turn items: release the lane, emit an accurate user-visible failure/status, preserve replay metadata, and only offer or perform retry when side-effect safety is known.

## Reproduction Assessment

Yes, source inspection and the reporter's live 2026.5.18 logs give a high-confidence reproduction shape: Codex app-server emits `item/completed`, omits `turn/completed`, and OpenClaw times out via `turn_completion_idle_timeout`. I did not run the live provider/channel repro in this read-only review.

## Solution Assessment

Unclear as a full fix: explicit visible timeout status is a straightforward bug fix, but automatic retry/resume after completed tools needs a maintainer decision around replay safety and side effects.
