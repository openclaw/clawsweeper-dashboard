## Summary

Keep open: current main still implements the fail-fast timeout for missing Codex `turn/completed`, but the reported recovery/status gap after productive `item/completed` work is not implemented or fully superseded by the adjacent closed fixes.

## What This Changes

The issue asks OpenClaw to recover visibly when Codex app-server emits `item/completed` but never `turn/completed`, instead of aborting without useful status or resume guidance.

## Best Possible Solution

Add a bounded recovery path for completion-idle timeouts after completed current-turn items: release the lane, emit an accurate visible failure/status, preserve replay metadata, and retry or resume only when side-effect safety is known.

## Reproduction Assessment

Yes. Source inspection and live reporter logs give a high-confidence reproduction shape: Codex app-server emits `item/completed`, omits `turn/completed`, and OpenClaw times out via `turn_completion_idle_timeout`; I did not run a live provider/channel repro in this read-only review.

## Solution Assessment

Unclear as a complete implementation path. Visible timeout status is a narrow bug fix, but automatic retry or resume after completed tools needs a maintainer decision around replay safety and side effects.
