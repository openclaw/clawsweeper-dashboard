## Summary

Keep open: current main still implements the prior fail-fast Codex app-server timeout, but the reported recovery/status gap after a productive `item/completed` stall remains unsolved and was confirmed again in issue discussion.

## What This Changes

The issue asks OpenClaw to handle Codex app-server turns that go quiet after `item/completed` without `turn/completed` by surfacing accurate status and preserving a safe retry or recovery path.

## Best Possible Solution

Add a bounded recovery path for Codex app-server completion-idle timeouts after completed current-turn items: release the lane, emit an accurate user-visible failure/status, preserve replay metadata, and only offer or perform retry when side-effect safety is known.

## Reproduction Assessment

Yes. Reporter logs, a second confirmation, and current source give a high-confidence reproduction shape: Codex app-server emits `item/completed`, omits `turn/completed`, and OpenClaw times out via `turn_completion_idle_timeout`; I did not run the live provider/channel path.

## Solution Assessment

Unclear as a complete implementation. Accurate visible timeout status is a narrow bug fix, but automatic retry or resume after completed tools needs maintainer agreement on replay safety and side effects.
