- `P1`: The report affects live Discord and Telegram Codex app-server workflows where user-visible work can stall, abort, and leave users without a useful recovery signal.
- `impact:message-loss`: The issue is about chat lanes appearing silent and losing the requested work after a completed tool item.

## Summary

Keep open: current main still implements the prior fail-fast Codex app-server watchdog, but it does not clearly provide the structured recovery/status/resume behavior requested here, and fresh comments point to multiple plausible runtime causes rather than a shipped fix.

## What This Changes

The issue asks OpenClaw to recover visibly when Codex app-server emits `item/completed` but never sends `turn/completed`, instead of aborting silently and losing work.

## Best Possible Solution

Add a bounded recovery contract for this timeout class: release the lane, emit an accurate user-visible failure/status, preserve replay metadata, and gate any retry/resume on explicit side-effect safety; separately investigate `logs_2.sqlite` diagnostics if the new evidence holds.

## Reproduction Assessment

Yes, source inspection plus reporter and commenter logs give a high-confidence reproduction shape: Codex app-server emits `item/completed`, omits `turn/completed`, and OpenClaw reaches `turn_completion_idle_timeout`. I did not run the live provider/channel repro in this read-only pass.

## Solution Assessment

Unclear as a full fix: visible timeout status and replay metadata are straightforward bug-fix territory, but automatic retry/resume after completed tools needs maintainer judgment around side effects and upstream Codex behavior.
