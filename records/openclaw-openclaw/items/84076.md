## Summary

Keep open: the prior shipped fixes make this Codex app-server dead-air case time out instead of hanging indefinitely, but current main still does not implement the reported structured recovery/status and safe retry/resume behavior after productive work is abandoned.

## What This Changes

The issue requests recovery/status handling for Codex app-server runs that abort after an `item/completed` notification without a following `turn/completed`.

## Best Possible Solution

Add a bounded recovery path for completion-idle timeouts after current-turn item completion: release the lane, emit accurate visible status, preserve last-item/tool replay metadata, and only offer or perform retry when side-effect safety is known.

## Reproduction Assessment

Yes, at source level: the current code path after final `item/completed` arms the completion-idle timeout and then aborts with `turn_completion_idle_timeout`; the reporter also supplied live Discord and Telegram logs from v2026.5.18. I did not run the live provider/channel repro in this read-only review.

## Solution Assessment

Unclear as a complete fix: visible timeout status is a narrow bug fix, but automatic retry or resume after completed tool work needs a maintainer decision around replay safety and side effects.
