## Label Justifications

- `P1`: The report affects live Discord and Telegram agent workflows where productive Codex app-server work can stall, abort, and leave the user without useful recovery.
- `impact:message-loss`: The user-facing symptom is silent or missing chat output after the assistant has already started work through a completed tool item.
- `impact:session-state`: The affected lane remains in `processing` until timeout and lacks a clear persisted recovery/resume state after abort.

## Summary

Keep open: current main still treats the Codex app-server `item/completed` without `turn/completed` case as a timeout/abort path, while the reported missing user-visible recovery and safe retry/resume behavior remains unresolved. The adjacent merged fixes covered fail-fast liveness and account-update noise, not this recovery/status gap.

## What This Changes

Request recovery and status handling for Codex app-server turns that receive `item/completed` but no `turn/completed`, so user-facing chat lanes do not silently lose work.

## Best Possible Solution

Add a bounded recovery path for Codex app-server completion-idle timeouts after completed current-turn items: release the lane, emit an accurate user-visible failure/status, preserve replay metadata, and only offer or perform retry when side-effect safety is known.

## Reproduction Assessment

Yes. Source inspection and two live reporter/commenter log sets give a high-confidence reproduction shape: Codex app-server emits `item/completed`, omits `turn/completed`, and OpenClaw times out through `turn_completion_idle_timeout`; I did not run the live provider/channel path in this read-only review.

## Solution Assessment

Unclear as a full fix. Explicit visible timeout status is a straightforward bug fix, but automatic retry or resume after completed tools needs a maintainer decision around replay safety and side effects.
