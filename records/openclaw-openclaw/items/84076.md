## Label Justifications

- `P1`: The report affects live Discord and Telegram agent workflows where Codex app-server work can stall, abort, and fail to produce useful user-visible recovery.
- `impact:message-loss`: The user-facing requested work can disappear after a completed tool item without a useful channel result.
- `impact:session-state`: The session lane remains processing until timeout and lacks a durable recovery or resume state after the abort.

## Summary

Keep open: current main contains the prior fail-fast watchdog behavior, but the reported recovery/status gap after Codex app-server `item/completed` without `turn/completed` is still not implemented or superseded by the adjacent fixes.

## What This Changes

Codex app-server should recover or clearly report a user-facing failure after a completed current-turn item when `turn/completed` never arrives, instead of silently aborting Discord or Telegram work.

## Best Possible Solution

Add a bounded Codex app-server recovery path that releases the lane, emits an accurate visible timeout status, records replay and side-effect safety, and only offers or performs retry/resume under an explicit maintainer-approved policy.

## Reproduction Assessment

Yes. Source inspection plus the reporter's live 2026.5.18 logs give a high-confidence path: Codex app-server emits `item/completed`, omits `turn/completed`, and OpenClaw aborts through `turn_completion_idle_timeout`.

## Solution Assessment

Unclear as a full fix. A visible timeout status is a straightforward bug fix, but automatic retry or resume after completed tools needs a product decision around side-effect safety and replay semantics.
