## Summary

Keep open: current main still has the compat-mode runtime snapshot path that can recompute the installed plugin inventory fingerprint, and this focused PR appears to address the linked CPU hot loop with sufficient live proof and no blocking review findings.

## What This Changes

The PR adds a cached `inventoryFingerprint` to plugin metadata snapshots, reuses it for current-snapshot control-plane fingerprint checks, adds a no-restat regression test, and updates the changelog.

## Best Possible Solution

Land the narrow snapshot-fingerprint handoff after maintainer validation, keeping freshness tied to explicit snapshot replacement and installed-index invalidation rather than adding a broad discovery cache.

## Reproduction Assessment

Yes, source-reproducible: current main passes `snapshot.index` into repeated current-snapshot fingerprint checks, which falls through to `resolveInstalledManifestRegistryIndexFingerprint` and `fs.statSync`. The linked issue also provides concrete compat-mode idle Gateway profiling output.

## Solution Assessment

Yes. Carrying the already-computed inventory fingerprint on the explicit `PluginMetadataSnapshot` is the narrowest maintainable fix I see and matches the documented snapshot boundary.
