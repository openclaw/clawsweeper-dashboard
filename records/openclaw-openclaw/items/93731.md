## Summary

Keep open: the PR targets a real Telegram allowed_updates gap, but as written it accepts `message_reaction_count` updates without adding any handler for that distinct update type, so count updates would still be ignored. It also lacks after-fix proof from a real Telegram setup.

## What This Changes

The PR adds `message_reaction_count` to Telegram polling/webhook allowed updates and updates Telegram reaction docs/tests.

## Best Possible Solution

Pair the allowed_updates change with a tested `message_reaction_count` handling path and real Telegram proof, or leave count updates out until the product behavior for anonymous aggregate reaction counts is defined.

## Reproduction Assessment

Yes by source inspection: Telegram/grammY expose `message_reaction_count` as a separate update field, while OpenClaw only registers `message_reaction`. A mocked count update would pass through `handleUpdate` without producing the promised reaction event; no live Telegram run was performed in this read-only review.

## Solution Assessment

No. Requesting the count update is only a plausible first step; the maintainable fix needs a count-update handler/tests/docs or should avoid subscribing to an unhandled event type.
