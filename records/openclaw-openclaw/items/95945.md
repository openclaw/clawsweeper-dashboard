## Summary

Keep open: this is a focused candidate for the linked media wake-false bug, but the current patch is too broad because it retries direct delivery for every false wake result and bypasses existing no-direct-send semantics for requester abandonment and generic handoff failures.

## What This Changes

The PR adds a direct media-delivery retry when media wake completion returns false after successful generation, plus a regression test for that path.

## Best Possible Solution

Make media completion fallback reason-aware inside `wakeMediaGenerationTaskCompletion`, or return structured failure metadata to the outer scheduler, so only explicitly safe generated-media loss reasons retry direct delivery while abandonment and generic handoff failures remain blocked.

## Reproduction Assessment

Yes at source level: the actual lifecycle can return false for `requester_abandoned`, and current tests assert that this must not call direct `sendMessage`. I did not run a live or local failing test in this read-only review.

## Solution Assessment

No. The symmetric boolean-level retry is plausible, but it is not the best fix because it discards the failure reason that current code uses to keep abandonment and generic handoff failures from direct-sending.
