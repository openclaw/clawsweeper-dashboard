## Summary

Keep open: the PR targets a real media wake-false delivery asymmetry, but it is not merge-ready because the new fallback runs after any false wake result and bypasses the existing requester-abandoned no-delivery contract.

## What This Changes

The PR adds a direct media delivery retry when `scheduleMediaGenerationTaskCompletion` sees `wakeTaskCompletion` return false, plus a mocked regression test for successful direct fallback.

## Best Possible Solution

Make the recovery reason-aware so generated media is retried only for confirmed delivery-loss false returns while preserving the abandoned-session stop and proving the final path in a real channel run.

## Reproduction Assessment

Yes. Source inspection shows `requester_abandoned` returns false without direct send on current main, while the PR's new outer false branch would direct-send whenever a handle has a deliverable origin.

## Solution Assessment

No. Symmetric fallback is plausible, but this implementation is too broad because the outer handler only sees a boolean; the maintainable fix should preserve reason-specific delivery contracts where the delivery result is still known.
