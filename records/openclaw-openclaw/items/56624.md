## Solution Assessment

No. Direct runner-owned status writes duplicate lifecycle semantics and happen too early; the safer solution is lifecycle-owned finalization after delivery with best-effort persistence.

- **[P2] Persist status after the final delivery outcome:** `src/cron/isolated-agent/run.ts:569-570`

  - body: The catch path awaits `persistSessionEntry()` before returning the structured run error. If the session-store write fails, callers get a rejection and lose the original timeout, model, or setup failure result.
  - confidence: 0.87
- **[P2] Avoid persisting running from the pre-run helper:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: Adding `status = "running"` to `markCronSessionPreRun` means the pre-run persist can store a running session before all setup and execution are protected by terminal lifecycle handling. If a later setup step fails, the row can remain stuck at `running`.
  - confidence: 0.78
