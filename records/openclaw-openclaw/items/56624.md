## Summary

Keep open. The PR targets a plausible isolated-cron session-state bug, but the current patch is not mergeable as-is: it lacks real behavior proof and introduces correctness regressions around delivery failures, session-store write failures, and pre-run setup failures.

## What This Changes

The PR sets isolated cron session status to `running`, persists `done`/`failed`/`timeout` from isolated-run paths, and adds focused session-status regression tests.

## Best Possible Solution

Land a rebased or replacement fix that finalizes isolated cron session lifecycle after the final delivery outcome, reuses the existing gateway session lifecycle semantics where applicable, and includes focused regression tests plus real run evidence.

## Reproduction Assessment

Yes, source-reproducible. Current main still has a clear isolated-cron finalization path that persists session metadata without direct terminal status assignment, and the linked report gives concrete session-store versus run-log inspection steps, but I did not execute a live cron run in this read-only review.

## Solution Assessment

No. The PR direction is useful, but this patch writes terminal status before delivery can still fail, lets session-store write failures mask original run outcomes, and persists `running` before later setup can throw.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist status after delivery finishes:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The PR writes `done` or `failed` before `dispatchCronDelivery` runs. A non-best-effort direct delivery failure can still return `status: "error"`, leaving the persisted session status as `done` even though the cron run failed.
  - confidence: 0.9
- **[P2] Keep terminal status writes best-effort:** `src/cron/isolated-agent/run.ts:716-717`
  - body: The catch path awaits `persistSessionEntry()` before returning the structured run error. If the session-store write fails, the runner rejects and loses the original timeout or model failure result callers expect.
  - confidence: 0.87
- **[P2] Avoid persisting running before setup can fail:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: `markCronSessionPreRun` now sets `status` to `running`, and the PR persists it before auth-profile resolution. If later setup throws, no terminal status is written and the session row can remain stuck at `running`.
  - confidence: 0.83
