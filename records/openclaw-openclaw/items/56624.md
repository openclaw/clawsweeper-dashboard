## Summary

Keep open. The PR targets a real isolated-cron session status gap, but the current branch is merge-conflicting, lacks after-fix real behavior proof, and still has correctness issues around when and how status writes are persisted.

## What This Changes

The branch sets isolated cron session status to running/done/failed/timeout in the isolated runner and adds focused session-status regression tests.

## Best Possible Solution

Land a rebased fix that records isolated-cron terminal status after the final delivery outcome, keeps status persistence best-effort and observable, and proves the session-store update with focused tests plus real run evidence.

## Reproduction Assessment

Yes, source-reproducible. Current main has isolated-run terminal return paths that persist metadata but do not assign terminal sessionEntry.status in the runner, and the linked report gives concrete sessions.json versus run-log inspection steps.

## Solution Assessment

No. The PR direction is useful, but this patch persists done/failed before delivery can still fail, lets status-store write failures mask the original run result, and persists running before later setup can throw.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist status after delivery finishes:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The PR writes done/failed before dispatchCronDelivery runs. A non-best-effort direct delivery failure can still return status: "error", leaving the persisted session status as done even though the cron run failed.
  - confidence: 0.88
- **[P2] Keep status writes best-effort:** `src/cron/isolated-agent/run.ts:716-717`
  - body: The catch path awaits persistSessionEntry before returning the structured run error. If the session-store write fails, the runner rejects and loses the original timeout or model failure result callers expect.
  - confidence: 0.84
- **[P2] Avoid persisting running before setup can fail:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: markCronSessionPreRun now sets status to running, and run.ts persists it before auth-profile resolution. If later setup throws, no terminal status is written and the session row can remain stuck at running.
  - confidence: 0.8
