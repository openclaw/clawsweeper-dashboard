## Summary

Keep open. Current main still does not persist an isolated cron session terminal status, so the PR addresses a real gap, but the branch is dirty against main and its current patch can store the wrong status or mask the original run result when persistence fails.

## What This Changes

The PR sets isolated cron session status to running at pre-run time, persists done/failed/timeout in terminal paths, and adds focused session-status regression tests.

## Best Possible Solution

Land a rebased isolated-cron status persistence fix that records running only inside a protected execution window, derives the terminal stored status after final run and delivery outcome classification, and treats status-store writes as logged best-effort side effects.

## Reproduction Assessment

Yes. Current main source inspection shows isolated cron session metadata is persisted, but no pre-run, success, fatal-error, abort, or thrown-error path assigns a terminal sessionEntry.status; the linked sessions.json versus run-log report matches those paths.

## Solution Assessment

No. The PR targets the right owner module, but it persists done/failed before delivery can still fail and awaits terminal persistence in error paths, which can mask the original structured cron result. A safer repair should classify after delivery and guard persistence failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist status after delivery outcome is known:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The new done/failed write happens before fallback/direct delivery runs. If dispatchCronDelivery returns an error result, the function returns status: "error" while the session store has already been persisted as "done" for a non-fatal payload, so status-based monitoring will misclassify failed cron runs.
  - confidence: 0.88
- **[P2] Guard running status until setup is protected:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: Adding status = "running" to the pre-run helper makes that value persist before later setup work can throw. If a setup step fails before the run-level try/catch, the stored cron session can stay stuck as running.
  - confidence: 0.82
- **[P2] Keep terminal status persistence best-effort:** `src/cron/isolated-agent/run.ts:717`
  - body: The catch path now awaits persistSessionEntry() before returning the original run error. If the session store write fails, runCronIsolatedAgentTurn rejects and masks the structured { status: "error" } result callers currently receive.
  - confidence: 0.8
