## Summary

Keep open. Current main still does not persist terminal isolated-cron session status, but this PR is not safe to merge because it records success/failure before delivery can still fail and adds unguarded terminal persistence that can mask the original run result.

## What This Changes

The PR sets isolated cron session status to running before execution, persists done/failed/timeout on terminal paths, and adds focused status regression tests.

## Best Possible Solution

Land a rebased isolated-cron fix that marks running only inside the protected execution window, derives the stored terminal status after final delivery outcome classification, and treats status-store writes as logged best-effort side effects.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: an isolated cron run can return ok/error/timeout while the session entry is only persisted with metadata, not a terminal status; the linked sessions.json versus run-log report matches those paths.

## Solution Assessment

No. The PR targets the right owner module, but it persists done/failed before delivery can still fail and awaits terminal store writes in ways that can mask the original structured cron result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist status after delivery outcome is known:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The new done/failed write happens before fallback/direct delivery runs. If dispatchCronDelivery later returns an error result, the function returns status: "error" while the session store has already been persisted as "done" for a non-fatal payload, so status-based monitoring will misclassify failed cron runs.
  - confidence: 0.88
- **[P2] Guard running status until setup is protected:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: Adding status = "running" to the pre-run helper makes that value persist before later setup work can throw. If a setup step fails before the run-level try/catch, the stored cron session can stay stuck as running.
  - confidence: 0.82
- **[P2] Keep terminal status persistence best-effort:** `src/cron/isolated-agent/run.ts:717`
  - body: The catch path now awaits persistSessionEntry() before returning the original run error. If the session store write fails, runCronIsolatedAgentTurn rejects and masks the structured { status: "error" } result callers currently receive.
  - confidence: 0.8
