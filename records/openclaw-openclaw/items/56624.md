## Summary

Keep open. The underlying cron session-state bug is credible, but this branch can persist the wrong terminal status before delivery finishes, can mask the original run error if the session-store write fails, and still lacks contributor real behavior proof.

## What This Changes

The PR adds isolated cron run session status writes for `running`, `done`, `failed`, and `timeout`, plus focused regression tests for those paths.

## Best Possible Solution

Land a rebased or replacement fix that finalizes isolated cron session lifecycle after the final delivery outcome, preferably through the existing gateway lifecycle persistence path, with focused regression coverage and redacted real run proof.

## Reproduction Assessment

Yes, source-level. The linked report gives concrete `sessions.json` versus cron run-log inspection steps, and the current/PR code paths show where isolated cron session metadata can be persisted without a final lifecycle-consistent terminal state; I did not execute a live cron run in this read-only review.

## Solution Assessment

No. Direct status writes in this patch are useful intent, but the maintainable fix needs to derive terminal status after delivery settles and keep session-store persistence best-effort or lifecycle-owned so it cannot diverge from the actual run result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist status after delivery finishes:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The PR writes `done` or `failed` before `dispatchCronDelivery` runs. A non-best-effort direct delivery failure can still return `status: "error"`, leaving the stored session status as `done` even though the cron run failed.
  - confidence: 0.9
- **[P2] Keep terminal status writes best-effort:** `src/cron/isolated-agent/run.ts:716-717`
  - body: The catch path awaits `persistSessionEntry()` before returning the structured run error. If the session-store write fails, callers get a rejection and lose the original timeout or model/setup failure result.
  - confidence: 0.87
- **[P2] Avoid persisting running before setup can fail:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: `markCronSessionPreRun` now sets `status` to `running`, and the PR persists it before auth-profile resolution. If later setup throws, the session row can remain stuck at `running`.
  - confidence: 0.83
