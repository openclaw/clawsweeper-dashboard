## Summary

Keep open. The underlying isolated-cron session-status bug is still source-reproducible on current main, but this PR head still has correctness blockers, is merge-conflicting, and lacks after-fix real behavior proof.

## What This Changes

The PR adds isolated cron session status updates for running/done/failed/timeout paths and adds focused regression tests for success, fatal-error, thrown-error, and timeout outcomes.

## Best Possible Solution

Land a rebased fix that records isolated-cron terminal status after the final delivery outcome, keeps persistence best-effort and observable, and proves the session-store update with focused tests plus real run evidence.

## Reproduction Assessment

Yes, source-reproducible. Current main has direct isolated-run terminal return paths that persist metadata but do not assign terminal sessionEntry.status, and the linked report gives concrete sessions.json versus run-log steps.

## Solution Assessment

No. The PR direction is useful, but the current patch persists done/failed before delivery can still fail, lets status-store write failures mask the original run result, and persists running before remaining setup can throw.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist terminal status after delivery settles:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The PR writes `done`/`failed` before `dispatchCronDelivery` runs. A non-best-effort direct delivery failure can still return `status: "error"`, leaving the persisted session status as `done` even though the cron run failed.
  - confidence: 0.88
- **[P2] Keep status persistence from masking run errors:** `src/cron/isolated-agent/run.ts:716-717`
  - body: The catch path awaits `persistSessionEntry()` before returning the structured error. If the session-store write fails, the runner rejects and loses the original timeout/run failure result that callers expect as `{ status: "error" }`.
  - confidence: 0.84
- **[P2] Move running persistence after fallible setup:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: `markCronSessionPreRun` now persists `running` before auth-profile resolution. If that later setup throws, no terminal status is written and the session row can remain stuck at `running`.
  - confidence: 0.8
