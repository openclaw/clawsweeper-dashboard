## Summary

Keep open, but this PR is not merge-ready. The branch tries to fix a real isolated-cron session-state bug, but the current patch writes session status through a direct runner-side path that can diverge from the final delivery/run outcome, lacks required real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds direct isolated cron session status mutations and persistence calls plus a new focused Vitest file for done, failed, thrown-error, and timeout paths.

## Best Possible Solution

Land a rebased or replacement fix that finalizes isolated cron session state through the existing lifecycle persistence path after the final run and delivery outcome, with focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Yes, source-level. The linked report gives concrete `sessions.json` versus cron run-log steps, and the current/PR code paths show how direct session writes can diverge from the actual final run or delivery outcome.

## Solution Assessment

No. Direct runner-side status writes before final delivery and awaited persistence in error paths are not the best fix; the safer direction is lifecycle-owned finalization with best-effort persistence that cannot mask the original cron result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist status after delivery finishes:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The new status write runs before `dispatchCronDelivery`. A non-best-effort delivery failure can still return `status: "error"`, leaving the session store persisted as `done` even though the cron run failed.
  - confidence: 0.9
- **[P2] Keep terminal status persistence best-effort:** `src/cron/isolated-agent/run.ts:716-717`
  - body: The new catch/timeout paths await `persistSessionEntry()` before returning the structured run error. If the session-store write fails, callers get a rejection and lose the original timeout, model, or setup failure result.
  - confidence: 0.87
- **[P2] Avoid persisting running before setup can fail:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: Setting `status = "running"` in `markCronSessionPreRun` is picked up by the existing pre-run persist before later setup, such as auth-profile resolution, has completed. If that setup throws, the row can remain stuck at `running`.
  - confidence: 0.83
