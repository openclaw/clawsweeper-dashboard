## Summary

Keep open. The underlying isolated-cron session-status bug is source-reproducible on current main, but this PR is still merge-conflicting, lacks required after-fix real behavior proof, and its head can persist misleading terminal status or mask the original run result.

## What This Changes

The PR sets isolated cron sessions to `running`, persists `done`/`failed`/`timeout` in terminal runner paths, and adds focused session-status regression tests.

## Best Possible Solution

Land a rebased fix that records isolated-cron terminal status after the final delivery outcome, keeps status persistence best-effort and observable, and proves the session-store update with focused tests plus real run evidence.

## Reproduction Assessment

Yes, source-reproducible. Current main persists isolated cron session metadata but does not set terminal `sessionEntry.status`, and the linked report gives concrete `sessions.json` versus run-log inspection steps.

## Solution Assessment

No. The PR direction is useful, but this patch persists `done`/`failed` before delivery can still fail, lets session-store write failures mask the original result, and persists `running` before later setup can throw.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist status after delivery finishes:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The PR writes `done`/`failed` before `dispatchCronDelivery` runs. A non-best-effort direct delivery failure can still return `status: "error"`, leaving the persisted session status as `done` even though the cron run failed.
  - confidence: 0.9
- **[P2] Keep terminal status writes best-effort:** `src/cron/isolated-agent/run.ts:716-717`
  - body: The catch path awaits `persistSessionEntry()` before returning the structured run error. If the session-store write fails, the runner rejects and loses the original timeout or model failure result callers expect.
  - confidence: 0.86
- **[P2] Avoid persisting running before setup can fail:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: `markCronSessionPreRun` now sets `status` to `running`, and the PR head persists it before auth-profile resolution. If later setup throws, no terminal status is written and the session row can remain stuck at `running`.
  - confidence: 0.82
