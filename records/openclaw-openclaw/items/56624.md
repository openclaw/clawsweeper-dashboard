## Summary

Keep open. The PR targets a real isolated cron session-finalization bug, but the branch is not merge-ready: it persists terminal status before delivery has finished, awaits terminal persistence in error paths, can leave setup failures stuck at running, lacks real behavior proof, and is currently conflict-marked against main.

## What This Changes

The PR adds direct isolated cron session status writes in the isolated runner and adds focused tests for done, failed, thrown-error, and timeout status mutations.

## Best Possible Solution

Land a rebased or replacement fix that finalizes isolated cron session state through the existing lifecycle persistence path after the final run and delivery outcome, with focused regression coverage and redacted real-run proof.

## Reproduction Assessment

Yes, source-level. The linked report gives concrete `sessions.json` versus cron run-log steps, and the current/PR code paths show how direct session writes can diverge from the actual final run or delivery outcome.

## Solution Assessment

No. Direct status writes before final delivery and awaited persistence in error paths are not the best fix; the safer direction is lifecycle-owned finalization with best-effort persistence that cannot mask the original cron result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Persist status after delivery finishes:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The PR writes `done` or `failed` before `dispatchCronDelivery` runs. A non-best-effort direct delivery failure can still return `status: "error"`, leaving the persisted session status as `done` even though the cron run failed.
  - confidence: 0.9
- **[P2] Keep terminal status persistence best-effort:** `src/cron/isolated-agent/run.ts:716-717`
  - body: The catch path awaits `persistSessionEntry()` before returning the structured run error. If the session-store write fails, callers get a rejection and lose the original timeout, model, or setup failure result.
  - confidence: 0.87
- **[P2] Avoid persisting running before setup can fail:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: Adding `status = "running"` here means the pre-run persist stores a running session before later setup, such as auth-profile resolution, has completed. If that setup throws, the session row can remain stuck at `running`.
  - confidence: 0.83
