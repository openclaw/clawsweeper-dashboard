## Summary

Keep open. Current main still lacks isolated cron terminal session-status persistence, but this PR should not merge as-is because it can persist the wrong status before delivery finishes and can let session-store write failures mask structured cron results.

## What This Changes

The PR sets isolated cron session status to running, persists done/failed/timeout on terminal paths, and adds focused regression tests for successful, fatal-error, thrown-error, and timeout outcomes.

## Best Possible Solution

Land a rebased isolated-cron fix that records running only once execution is protected, derives terminal status from the final delivered run outcome, and treats status persistence as a logged best-effort side effect.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: isolated cron runs can return ok/error/timeout while the session entry is persisted only with metadata, not a terminal `sessionEntry.status`.

## Solution Assessment

No. The PR targets the right owner module, but it persists done/failed before delivery can still fail and awaits status-store writes in ways that can mask the original structured cron result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist status after delivery outcome is known:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The new done/failed write happens before fallback/direct delivery runs. If `dispatchCronDelivery` later returns an error result, the function returns `status: "error"` while the session store has already been persisted as `"done"` for a non-fatal payload, so status-based monitoring will misclassify failed cron runs.
  - confidence: 0.88
- **[P2] Keep terminal status writes best-effort:** `src/cron/isolated-agent/run.ts:717`
  - body: The catch path now awaits `persistSessionEntry()` before returning the original run error. If the session-store write fails, `runCronIsolatedAgentTurn` rejects and masks the structured `{ status: "error" }` result callers currently receive.
  - confidence: 0.82
- **[P2] Move running status inside protected execution:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: Adding `status = "running"` to the pre-run helper makes that value persist before later setup work can throw. If a setup step fails before the run-level try/catch, the stored cron session can stay stuck as running.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/cron/isolated-agent/run.ts:572-573`
  - body: This is a user-facing cron fix, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing fixes before merge.
  - confidence: 0.9
