## Summary

Keep open. Current main still does not persist an isolated cron run's terminal `sessionEntry.status`, so the useful bug remains, but the proposed patch is not safe to merge because it records `done`/`failed` before delivery can still fail and lets session-store write failures mask the structured cron result. The linked report was later folded into #70619 and closed from triage, not by a source change that implements this PR's behavior.

## What This Changes

The PR sets isolated cron session entries to `running` before execution, persists `done`/`failed`/`timeout` on completion paths, and adds focused session-status regression tests.

## Best Possible Solution

Land a current-main scoped isolated-cron fix that derives the stored terminal session status from the final run and delivery result, records it as a logged best-effort session-store side effect, and avoids persisting `running` until setup failures can no longer strand the row.

## Reproduction Assessment

Yes. Source inspection is enough for the main bug and PR regressions: current main has no terminal `sessionEntry.status` writes, while the PR head writes `done` before `dispatchCronDelivery` can still return an error.

## Solution Assessment

No. The PR targets the right owner module, but the write is too early and not failure-isolated; the safer solution is to classify after delivery and guard status persistence so it cannot change the run-result contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist status after delivery outcome is known:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The new `done`/`failed` write happens before `dispatchCronDelivery` runs. If non-best-effort direct delivery fails, the function returns `status: "error"` while the session store has already been saved as `done` for a non-fatal payload, so status-based monitoring can misclassify failed cron runs.
  - confidence: 0.88
- **[P2] Avoid stranding setup failures as running:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: Adding `status = "running"` to the pre-run helper makes that value persist before later setup work such as auth-profile resolution runs outside the protected execution try/catch. If setup throws there, no terminal status is written and the row can stay stuck as `running`.
  - confidence: 0.82
- **[P2] Keep status persistence from masking run errors:** `src/cron/isolated-agent/run.ts:717`
  - body: The catch path now awaits `persistSessionEntry()` before returning the original structured run error. If the session-store write fails, `runCronIsolatedAgentTurn` rejects instead of returning `{ status: "error" }`, changing the caller contract and hiding the real run failure.
  - confidence: 0.8
