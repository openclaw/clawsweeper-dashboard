## Summary

Keep open. Current main still has the isolated-cron session-status gap, but this PR is conflicting, lacks after-fix real behavior proof, and its current diff can persist a terminal status that diverges from the final cron result.

## What This Changes

The PR marks isolated cron sessions as running and writes done, failed, or timeout status to the session store, with focused regression tests for terminal paths.

## Best Possible Solution

Land a rebased fix that records isolated-cron terminal state after the final delivery outcome, keeps status persistence best-effort and observable, and proves the session-store update with focused tests plus real run evidence.

## Reproduction Assessment

Yes, source-reproducible. Current main still returns terminal isolated cron outcomes without assigning terminal sessionEntry.status, and the linked report gives concrete sessions.json versus run-log steps.

## Solution Assessment

No. The direction is useful, but this patch persists done/failed before delivery can still fail and lets unguarded status-store writes mask the original run result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist terminal status after delivery completes:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The PR stores `done`/`failed` before `dispatchCronDelivery` runs. Non-best-effort direct delivery can still return an error result afterward, so a run can return `status: "error"` while the session store was already persisted as `done`.
  - confidence: 0.88
- **[P2] Make terminal status persistence best-effort:** `src/cron/isolated-agent/run.ts:717`
  - body: This catch path awaits `persistSessionEntry()` before returning the structured run error. If the session-store write fails, the runner rejects and masks the original timeout or run failure instead of returning the expected `{ status: "error" }` result.
  - confidence: 0.84
- **[P2] Move running persistence inside protected execution:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: Setting and persisting `running` during pre-run setup happens before later setup work such as auth-profile resolution. If that later setup throws, no terminal status is written and the session row can remain stuck as `running`.
  - confidence: 0.8
