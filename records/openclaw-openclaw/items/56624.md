## Summary

Keep open: this PR targets a real isolated-cron session-status bug, but the current head is outdated and still has correctness blockers around when status is persisted, how persistence failures are handled, and real behavior proof.

## What This Changes

The PR sets isolated cron session entries to running/done/failed/timeout during isolated cron execution and adds focused session-status regression tests.

## Best Possible Solution

Land a rebased fix that records isolated-cron lifecycle status after the final delivery outcome, keeps status persistence best-effort and logged, and proves the session-store update with focused tests plus real run evidence.

## Reproduction Assessment

Yes, source-reproducible. Current main returns isolated cron timeout/error/ok results without assigning terminal sessionEntry.status, and the linked report gives concrete sessions.json versus run-log steps.

## Solution Assessment

No. The direction is right, but this patch persists done/failed before delivery can still fail and lets terminal status-store writes mask the original cron result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist terminal status after delivery completes:** `src/cron/isolated-agent/run.ts:572-573`
  - body: The PR writes `done`/`failed` before `dispatchCronDelivery` runs. Non-best-effort direct delivery can still return an error result, so a run can return error while the session store was already persisted as done for a successful payload.
  - confidence: 0.88
- **[P2] Make terminal status persistence best-effort:** `src/cron/isolated-agent/run.ts:717`
  - body: The new catch path awaits `persistSessionEntry()` before returning the structured run error. If `updateSessionStore` fails, the runner rejects and masks the original timeout or run failure instead of returning `{ status: "error" }`.
  - confidence: 0.84
- **[P2] Set running status only inside protected execution:** `src/cron/isolated-agent/run-session-state.ts:74`
  - body: Adding `status = "running"` to `markCronSessionPreRun` lets the pre-run persist store running before later setup steps can throw outside the protected execution catch, leaving failed setup stuck as running.
  - confidence: 0.8
