## Summary

Keep open. The linked proxy bug is still real on current main, but this branch should not merge as-is because it is broad, stale against the current plugin/SDK layout, and introduces proxy-routing plus validation regressions already called out in review.

## What This Changes

The PR routes Gemini CLI OAuth fetches through environment proxy mode, changes dispatcher teardown/keep-alive handling, and also edits Discord proxy fallback, LINE setup, Nostr bus/key handling, and plugin-sdk tests.

## Best Possible Solution

Replace or narrow this to a current-main Gemini CLI OAuth fix tied to #46184: use the existing guarded-fetch trusted env-proxy path without extension-to-core internals, preserve target-aware HTTP(S)/ALL_PROXY/NO_PROXY semantics, and split Discord, LINE, Nostr, and plugin-sdk cleanup into separate reviewed work.

## Reproduction Assessment

Yes. #46184 gives concrete macOS HTTP(S) proxy steps for google-gemini-cli OAuth, and current main inspection shows the OAuth helper still does not request trusted env-proxy routing. I did not run a live OAuth flow in this read-only review.

## Solution Assessment

No. The Gemini/proxy fix direction is plausible, but this PR is not the best way to solve it because it mixes unrelated channel rewrites with SSRF-adjacent proxy changes and weakens existing validation/account-scoping behavior.
