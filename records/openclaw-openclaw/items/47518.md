## Summary

Keep open. The underlying Gemini CLI proxy bug is real and still source-reproducible on main, but this PR is not mergeable as-is because it mixes the narrow OAuth/dispatcher fix with unrelated channel and SDK changes that introduce proxy-policy and validation regressions.

## What This Changes

This PR routes Gemini CLI OAuth through trusted env-proxy mode, changes proxy dispatcher teardown/keepAlive behavior, and updates Discord proxy handling, LINE setup, Nostr bus code, and plugin-SDK tests.

## Best Possible Solution

Replace or refresh this as a narrow Gemini CLI OAuth fix using the plugin-SDK proxy seam, target-aware env-proxy behavior, dispatcher cleanup coverage, and a changelog entry; split Discord, LINE, Nostr, and SDK cleanup into separate reviewed PRs.

## Reproduction Assessment

Yes. #46184 provides concrete macOS HTTP(S) proxy steps, and current-main source still shows Gemini OAuth fetches without trusted env-proxy mode plus dispatcher cleanup that awaits close() first.

## Solution Assessment

No. The core Gemini/proxy direction is plausible, but this PR is too broad and regresses unrelated channel behavior; the maintainable fix is a focused current-main patch around OAuth fetch mode, proxy selection semantics, and dispatcher cleanup tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Honor NO_PROXY before applying Discord env proxy:** `extensions/discord/src/monitor/provider.ts:438`
  - body: The PR turns HTTPS_PROXY/HTTP_PROXY into an explicit Discord proxy for REST and gateway paths without checking whether discord.com is excluded by NO_PROXY. In environments that intentionally bypass Discord, startup REST calls and gateway traffic will now be forced through the proxy and can fail or violate operator routing policy; use the target-aware proxy gate for both paths.
  - confidence: 0.9
- **[P2] Use target-aware proxy selection for OAuth:** `extensions/google/oauth.http.ts:11`
  - body: Gating Gemini OAuth on hasProxyEnvConfigured() is broader than the EnvHttpProxyAgent target path: ALL_PROXY-only environments still go direct, while NO_PROXY and HTTP(S) proxy selection are already modeled by shouldUseEnvHttpProxyForUrl(). Use the target-aware helper or explicitly support the intended proxy options so the fix matches the documented env behavior.
  - confidence: 0.86
- **[P1] Reinstate strict Nostr pubkey validation:** `extensions/nostr/src/nostr-bus.ts:35`
  - body: The PR's normalizePubkey fallback accepts malformed non-npub values by lowercasing them instead of rejecting them. Callers rely on early validation for sender and allowlist IDs, so invalid keys can reach encryption, profile, and allowlist paths before failing unpredictably.
  - confidence: 0.92
- **[P2] Write LINE allowlist changes to the active account:** `extensions/line/src/setup-surface.ts:164`
  - body: The onboarding allowlist update hardcodes the default LINE account, so configuring a named account writes the allowlist to the wrong account. Preserve the active accountId so multi-account setup updates the account being edited.
  - confidence: 0.84
