## What This Changes

The PR attempts to route Gemini OAuth through env-proxy mode and change dispatcher teardown, while also modifying Discord proxy handling, LINE setup, Nostr bus/key utilities, networking tests, and plugin SDK tests.

## Best Possible Solution

Keep the canonical issue open and land a narrow current-main PR that changes only the Google OAuth env-proxy path with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main's Gemini OAuth helper does not opt into trusted env-proxy mode, while guarded fetch only uses EnvHttpProxyAgent for that explicit mode. The canonical issue also includes reporter proxy steps and a local patch that resolved the failure.

## Solution Assessment

No. The central fix direction is valid, but this PR is too broad and introduces unrelated regressions; the maintainable solution is a narrow target-aware trusted env-proxy change in the Google OAuth helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor NO_PROXY for Discord env proxy fallbacks:** `extensions/discord/src/monitor/provider.ts:440`
  - body: This derives a Discord proxy from HTTPS_PROXY/HTTP_PROXY and passes it as an explicit ProxyAgent, so NO_PROXY=discord.com is ignored and Discord REST traffic can be forced through a proxy the operator excluded. Use target-aware env-proxy handling or skip the fallback when NO_PROXY matches Discord hosts.
  - confidence: 0.88
- **[P2] Gate Gemini proxy mode on a usable HTTP(S) proxy:** `extensions/google/oauth.http.ts:13`
  - body: hasProxyEnvConfigured() is also true for ALL_PROXY, but this trusted-env path relies on EnvHttpProxyAgent without passing explicit ALL_PROXY-derived httpProxy/httpsProxy options. In ALL_PROXY-only environments the PR can select proxy mode while still sending OAuth calls direct, leaving the advertised proxy-mandatory case broken.
  - confidence: 0.84
- **[P2] Reinstate strict Nostr pubkey validation:** `extensions/nostr/src/nostr-bus.ts:35`
  - body: normalizePubkey now falls back to lowercasing any non-npub input, so malformed pubkeys are silently accepted instead of rejected. Nostr target and allowlist normalization rely on this throwing early; invalid IDs would now fail later in send or pairing paths.
  - confidence: 0.9
- **[P2] Write LINE allowlist updates to the selected account:** `extensions/line/src/setup-surface.ts:162`
  - body: The onboarding allowFrom handler hardcodes DEFAULT_ACCOUNT_ID, so configuring a non-default LINE account writes the allowlist to the wrong account. Multi-account setup can appear successful while enforcing the old allowlist on the account being edited.
  - confidence: 0.87
