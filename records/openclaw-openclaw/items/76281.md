## Summary

Keep open: current main has adjacent session tools, labels, Control UI session lists, and plugin-owned session extension state, but it does not provide the requested skill-facing project context API, native project display/switching, project-scoped history filter, or cross-project query contract. This needs maintainer product/API/security design before implementation.

## What This Changes

Request a first-class project/thread context API for skills, with Control UI display/switching, project-scoped history, and cross-project lookup.

## Best Possible Solution

Define an additive project-context contract that reuses plugin session extension seams where appropriate, then wire SDK/skills, Gateway protocol, Control UI display/switching, and project-filtered history with explicit visibility rules.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Current-main source and docs inspection gives a high-confidence comparison showing adjacent session/plugin primitives exist while the requested project API and project filters do not.

## Solution Assessment

Unclear until maintainer design. The proposed session.set_project sketch is plausible, but it should not land as a one-off helper before protocol, UI, history, and cross-project visibility semantics are chosen.
