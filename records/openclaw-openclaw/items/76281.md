## Summary

Keep open: current main has generic sessions, labels, cross-session tools, and plugin-owned session extension state, but it still lacks a first-class skill-facing project context API plus native project-scoped Control UI/history behavior.

## What This Changes

Request first-class project context for skills, including project declaration, Control UI display and thread switching, project-scoped history, and cross-project queries.

## Best Possible Solution

Design an additive project-context contract if maintainers want first-class project semantics, starting from existing session/plugin extension seams and then wiring Gateway protocol, SDK or skill-facing access, Control UI display, and history filtering consistently.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source and docs inspection gives a high-confidence current-main comparison showing generic sessions and plugin extensions exist, but the requested project API/UI/history contract does not.

## Solution Assessment

Unclear. The proposed session.set_project sketch is a useful direction, but maintainers should first choose whether project state is plugin-owned session extension data or a versioned core session metadata field before implementation.
