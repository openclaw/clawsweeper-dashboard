## Summary

Keep open: current main has generic sessions, labels, cross-session tools, and plugin-owned session extensions, but it does not provide a first-class skill-facing project context API or native project-scoped UI/history contract for the requested workflow.

## What This Changes

Skills would be able to declare active project metadata, expose it in the Control UI, switch between project threads, scope history by project, and query other projects.

## Best Possible Solution

Define an additive project-context contract around sessions that plugins/skills can use, then wire consistent Control UI display, session filtering, and cross-session history access if maintainers want first-class project semantics.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source inspection gives a high-confidence current-main comparison showing generic sessions exist but the requested skill project API does not.

## Solution Assessment

Unclear. The proposed `session.set_project` shape is useful as a sketch, but the maintainable path should first choose whether project context is a plugin-owned session extension or a versioned core session metadata field.
