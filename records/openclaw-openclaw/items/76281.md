## Summary

Keep open: current `main` has generic sessions, labels, session tools, plugin-owned session extension state, and recent plugin session slot projection, but it still lacks the requested skill-facing project context API plus native project display, switching, project-scoped history, and cross-project query semantics.

## What This Changes

The issue requests first-class project/thread awareness for skills, including project metadata on sessions, Control UI display/switching, project-filtered history, and cross-project lookup.

## Best Possible Solution

Define an additive project-context contract that aligns with the broader Projects dashboard work, reuses plugin session extension seams where appropriate, and then wires Gateway protocol, skill/SDK access, Control UI display/switching, and project-filtered history consistently.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source and docs inspection gives a high-confidence current-main comparison showing adjacent generic session/plugin primitives exist, while the requested project API/UI/history contract does not.

## Solution Assessment

Unclear until maintainer design. The proposed `session.set_project` sketch is plausible, but maintainers should first choose whether project state belongs in plugin-owned extension data, a first-class Project/session metadata model, or the broader Projects dashboard protocol.
