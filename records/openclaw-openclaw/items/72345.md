## Summary

Keep open. Current main still routes operational `voicecall` CLI commands through a process-local runtime that starts the webhook listener, while the Gateway already exposes `voicecall.*` RPC methods the CLI could delegate to; the related in-process runtime-sharing fix does not cover this separate CLI/Gateway process collision.

## What This Changes

The issue asks for `openclaw voicecall call` and related voice-call CLI actions to avoid starting a second webhook runtime when the Gateway already owns the voice-call listener.

## Best Possible Solution

Make the voice-call plugin CLI Gateway-aware: delegate operational call/control/status commands through the existing `voicecall.*` Gateway RPC methods when a Gateway is available or explicitly selected, preserve a deliberate standalone fallback, and keep diagnostic/read-only commands from starting or retaining long-lived voice runtime resources.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: Gateway service startup calls `ensureRuntime()`, a separate CLI `voicecall call` process also calls `ensureRuntime()`, runtime creation starts the webhook server, and the default listener is `127.0.0.1:3334`.

## Solution Assessment

Yes. The Gateway-delegation direction described in the report is the narrowest maintainable fix because the plugin already registers `voicecall.*` Gateway methods and the SDK already exposes a CLI Gateway RPC helper; a broad core runtime registry is not needed first.

Implement a plugin-local Gateway-aware CLI path using the existing `openclaw/plugin-sdk/gateway-runtime` helper and the registered `voicecall.*` Gateway methods.
