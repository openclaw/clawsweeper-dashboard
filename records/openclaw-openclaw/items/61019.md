## Summary

Keep open. Current main still updates the configured default model only, while persisted session runtime/override metadata can keep active or reused sessions on the prior model until an explicit reset, patch, or user model change occurs.

## What This Changes

Requests a way for active or persisted sessions to pick up changed global model defaults instead of staying pinned to stored session model metadata.

## Best Possible Solution

Choose one explicit session model refresh policy, then implement it where default-model updates and session metadata meet, preserving `modelOverrideSource: "user"` selections and documenting the `models set` behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: persist a session entry with old `model`/`modelProvider` or override metadata, change the default with `models set`, and session resolution/listing will still prefer the stored session state until reset or patch.

## Solution Assessment

Unclear. A refresh or migration mechanism is the right direction, but maintainers need to choose automatic propagation, an opt-in config, or an explicit migrate/reset action before this can be a narrow maintainable fix.
