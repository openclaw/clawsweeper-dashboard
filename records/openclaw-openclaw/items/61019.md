- `P2`: The issue affects model-selection usability and routing correctness for existing sessions, but it is not an outage, data-loss emergency, or security bypass.
- `impact:session-state`: The reported behavior is persisted session model state staying active after global default-model config changes.
- `impact:auth-provider`: The issue concerns provider/model choice precedence and stale model routing state.

## Summary

Keep open. Current main still updates only the global default model config while session resolution and session listing continue to prefer persisted session model state; related closed issues and the closed stale PR route the remaining policy decision back to this issue rather than resolving it.

## What This Changes

The issue asks for active or resumed sessions to inherit global default model changes, or for an explicit refresh/migration path after `openclaw models set`.

## Best Possible Solution

Keep this as the canonical active-session default-model refresh tracker, then choose a source-aware policy that preserves explicit user selections while adding either opt-in propagation, an explicit migrate/reset command, or clear status/doctor guidance.

## Reproduction Assessment

Yes, at source level. Persist a session with old `model`/`modelProvider` or override metadata, run `openclaw models set`, and current session resolution/listing still prefers the stored session state before falling back to the new configured default; I did not run a live CLI session because this review is read-only.

## Solution Assessment

Unclear. A refresh or migration mechanism is plausible, but the best fix depends on a maintainer policy choice between automatic propagation, opt-in config, explicit reset/migrate commands, and status/doctor guidance.
