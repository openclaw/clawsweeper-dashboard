## Summary

Current main still has the reported CLI validation gap, and the issue should stay open because two open PRs already use closing syntax for the fix. The right cleanup path is to review and land one of those PRs, then let this issue close after merge.

## What This Changes

The issue reports that invalid explicit `openclaw node run --port` values silently fall back to the saved/default gateway port instead of failing with the existing invalid-port diagnostic.

## Best Possible Solution

Land one focused fix that rejects invalid explicit `node run --port` values with the shared invalid-port diagnostic while preserving config/default fallback when the option is omitted.

## Reproduction Assessment

Yes. Source inspection on current main shows invalid explicit `opts.port` becomes `null` in `parsePort`, then `parsePortWithFallback` turns it into the configured/default port before `runNodeHost` is called.

## Solution Assessment

Yes. The narrow maintainable fix is to split omitted-port fallback from invalid explicit-port validation and reuse the existing `formatInvalidPortOption("--port")` behavior already used by adjacent commands.
