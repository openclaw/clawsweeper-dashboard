## Summary

Keep open: current main still has the silent fallback path, and multiple open PRs already reference this issue with closing syntax, so the issue should stay open until one canonical fix lands.

## What This Changes

Reject invalid explicit `openclaw node run --port` values instead of silently falling back to the configured or default port.

## Best Possible Solution

Land one canonical fix that preserves fallback when `--port` is omitted, rejects invalid explicit values with the existing `--port` diagnostic before `runNodeHost`, and then close this issue as implemented.

## Reproduction Assessment

Yes, from source inspection: invalid explicit `opts.port` becomes null in `parsePort`, then `parsePortWithFallback` converts that null to the configured/default port before `runNodeHost` is called.

## Solution Assessment

Yes. The narrow fix is to split omitted-port fallback from invalid explicit-port validation and reuse the existing `formatInvalidPortOption("--port")` behavior already used by adjacent commands.
