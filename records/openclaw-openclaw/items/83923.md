## Label Justifications

- `P2`: This is a normal CLI correctness bug with limited blast radius: a mistyped node-host port can launch against an unintended gateway address without a diagnostic.

## Summary

Current `main` still has the reported CLI bug: explicit invalid `openclaw node run --port` input is parsed as `null` and then silently replaced with the configured/default port. The issue should stay open because three open PRs already reference it with closing syntax, so the right cleanup is to review and land one canonical fix or close the competing PRs after that fix merges.

## What This Changes

Reject invalid explicit `openclaw node run --port` values instead of silently falling back to the configured or default gateway port.

## Best Possible Solution

Land one focused fix that validates explicit `--port` before fallback, keeps omitted-port config/default behavior, and adds regression coverage for invalid, valid, and omitted ports.

## Reproduction Assessment

Yes, from source inspection: invalid explicit `opts.port` becomes `null` in `parsePort`, then `parsePortWithFallback` turns it into the config/default port before calling `runNodeHost`. I did not start a foreground node host during this read-only review.

## Solution Assessment

Yes, the maintainable fix is to split omitted-option fallback from invalid explicit-option validation and reuse the existing `formatInvalidPortOption("--port")` behavior already used by adjacent commands.
