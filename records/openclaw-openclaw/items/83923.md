## Summary

Keep open: current `main` and the latest release still have the silent `node run --port` fallback, and the active fix is the open linked PR with closing syntax, so this issue should close only after that PR lands or is otherwise resolved.

## What This Changes

The issue reports that `openclaw node run --port <invalid>` silently falls back to the configured/default gateway port instead of rejecting the explicit invalid port.

## Best Possible Solution

Land or close the linked fix PR, then close this issue only after `main` rejects invalid explicit `node run --port` values while preserving omitted-port fallback.

## Reproduction Assessment

Yes from source inspection: invalid explicit `opts.port` becomes `null` in `parsePort`, then `parsePortWithFallback` turns that into the config/default port before `runNodeHost` is called. I did not start a foreground node host process during this read-only review.

## Solution Assessment

Yes: the narrow maintainable solution is to split missing-port fallback from invalid explicit-port validation and reuse the existing invalid-port diagnostic already used by adjacent CLI paths.
