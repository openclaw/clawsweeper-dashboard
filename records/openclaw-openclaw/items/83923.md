## Summary

Keep open: current main and the latest stable release still silently fall back for an explicit invalid `node run --port`, while the adjacent install and gateway run paths fail fast.

## What This Changes

The issue reports that `openclaw node run --port <invalid>` silently falls back to the configured/default port instead of rejecting the invalid option.

## Best Possible Solution

Validate explicit `--port` in `node run` with the same fail-fast contract used by `node install` and `gateway run`, while preserving config/default fallback only when the option is omitted.

## Reproduction Assessment

Yes, source-reproducible: `openclaw node run --port abc` flows through `parsePortWithFallback`, where `parsePort` returns `null` and the helper substitutes the configured/default port. I did not start the runtime process during this read-only review.

## Solution Assessment

Yes: the narrow fix is to reject an invalid explicit `--port` before `runNodeHost`, matching the already-defined install and gateway run validation behavior.
