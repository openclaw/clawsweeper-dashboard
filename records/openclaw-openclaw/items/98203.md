## What This Changes

The issue asks for Gateway `res` frames to preserve the optional response metadata already passed by server method handlers, especially cached/dedupe metadata.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/98206, or an equivalent narrow fix that adds optional response-frame `meta` to the protocol schema and forwards it from the WebSocket responder.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current main accepts `meta` in `RespondFn`, handlers pass cached/run metadata, but the `res` frame builder and schema omit it. I did not run a live gateway/client repro because this review is read-only.

## Solution Assessment

Yes, the linked PR's proposed shape is the narrow maintainable fix: add optional response-frame `meta` to the additive protocol schema and forward the existing `meta` argument. The issue should stay open until that PR merges or maintainers choose a replacement fix.
