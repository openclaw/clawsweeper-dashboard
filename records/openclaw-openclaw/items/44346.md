## Summary

Keep open: the feature is plausible, but this PR is not merge-ready because it conflicts with current main, lacks real behavior proof, changes existing `references/` semantics, and only materializes reference content in copied sandbox sync paths.

## What This Changes

This PR adds `references` skill metadata/frontmatter parsing, loads selected skill-local markdown into `SkillEntry.referencesContent`, appends it during skill sync, and documents/tests `references/` auto-load behavior.

## Best Possible Solution

Define a shared materialized-skill-content contract that works for prompt, snapshot, local, read-write, and copied-sandbox paths, make automatic reference loading opt-in or migrate intended bundled skills explicitly, and then add real behavior proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the new content is only appended in `syncSkillsToWorkspace`, and current main still tells agents to read the original `SKILL.md` location. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No: the current implementation is not the best way to solve it because it creates a new default for existing `references/` directories and does not centralize materialized skill content. A safer path is shared content resolution plus explicit opt-in for auto-loaded references.
