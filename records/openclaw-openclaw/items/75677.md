## Summary

Keep open: the i18n goal is useful, but this patch is incorrect because it hardcodes zh-CN into the default TUI and replaces semantic activity state keys that current recovery paths and tests depend on; it also lacks after-fix real behavior proof.

## What This Changes

The PR replaces English TUI status strings, activity status values, disconnect/Ctrl+C/tool status text, and default waiting phrases with hardcoded Simplified Chinese across four `src/tui` files.

## Best Possible Solution

Add a TUI localization layer at the display/message boundary that maps typed semantic activity states to localized copy, keeps English as the default fallback, and documents/tests the chosen locale-selection contract.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows `setActivityStatus` feeds the same `activityStatus` state read by exact-key waiting and streaming recovery branches. I did not run tests because this review is read-only, but the current source and tests give a high-confidence reproduction path.

## Solution Assessment

No. Hardcoding zh-CN over existing literals is not the best solution; the maintainable path is a locale-aware display layer that leaves control state semantic and preserves English fallback.
