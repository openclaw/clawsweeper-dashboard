## Summary

Keep open. Current main does not add zh-CN support for the TUI, but this PR should not merge as-is: it replaces English globally and also localizes internal activity-state sentinel strings that current rendering, recovery logic, and tests treat as semantic states.

## What This Changes

The PR replaces TUI status messages, activity-state strings, disconnect prompts, and default waiting phrases in `src/tui` with Simplified Chinese literals.

## Best Possible Solution

Add a real TUI i18n boundary: keep activity state as typed semantic keys, translate only at render/copy boundaries, preserve English as the default, add an explicit locale selection or documented auto-detection rule, and cover zh-CN labels plus state-machine behavior in focused TUI tests.

## Reproduction Assessment

Yes. Applying the PR and running the focused TUI tests would exercise the changed `setActivityStatus` literals, and static inspection already shows the mismatch between localized state values and current `"waiting"`/`"streaming"` comparisons.

## Solution Assessment

No. Hardcoding zh-CN over the existing TUI strings is not the narrowest maintainable solution because it removes the English default and conflates display labels with state-machine keys; a render-time translation layer is safer.

  - body: `setActivityStatus` updates the shared TUI activity state, and current recovery logic compares that state to semantic literals such as `"streaming"`. Keep the stored state typed/semantic and translate only when rendering text.
  - confidence: 0.9

  - body: This initializes every TUI session with Chinese status text regardless of user locale or config. The existing TUI docs and tests describe English status labels, and there is no TUI locale setting in this patch, so English users would lose the current default rather than gaining zh-CN support. Please make the translation opt-in or locale-aware.
  - confidence: 0.86
