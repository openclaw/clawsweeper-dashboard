## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact task-flow tag values before Prometheus export:** `extensions/diagnostics-prometheus/src/service.ts:506`
  - body: Task-flow tags are arbitrary plugin/user metadata, but this path writes accepted values directly into Prometheus labels. That bypasses the existing `lowCardinalityLabel` redaction boundary and can expose tokens, paths, hostnames, or other private tag values in telemetry.
  - confidence: 0.91
- **[P1] Redact task-flow tag attributes before OTel export:** `extensions/diagnostics-otel/src/service.ts:2934`
  - body: The OTel exporter has the same raw tag-value path, so arbitrary task-flow tags become metric attributes without the redaction used by the rest of diagnostics. Please route values through the redaction/low-cardinality helper and cover secret-like values in exporter tests.
  - confidence: 0.9
- **[P2] Sanitize Prometheus tag label names:** `extensions/diagnostics-prometheus/src/service.ts:506`
  - body: `LOW_CARDINALITY_VALUE_RE` allows dots, dashes, and colons, but `formatLabels` emits those keys unquoted as Prometheus label names. A tag like `team.name` becomes `tag_team.name`, which is not compatibility-safe for Prometheus labels and can break scraping or downstream tooling.
  - confidence: 0.86
- **[P2] Carry sync_mode through deletion diagnostics:** `src/tasks/task-flow-registry.ts:251-257`
  - body: The deleted-flow diagnostic is built from the previous flow record but drops `previous.syncMode`, so the Prometheus deleted counter can only emit `sync_mode="unknown"` even though the docs advertise `sync_mode` for that metric. Include the previous flow's sync mode in the diagnostic event or remove the documented label.
  - confidence: 0.82
