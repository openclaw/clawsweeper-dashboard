## Summary

Keep open: the feature direction is plausible, but the patch is not mergeable because it adds native media contracts that current runtime helpers do not consume, routes unsupported pi-ai audio/video blocks through `complete()`, and bypasses established media transport controls.

## What This Changes

The PR adds native `understandAudio`/`understandVideo` provider hooks, audio/video-capable config auto-registration, new media result kinds, and native-first runner paths with fallback to existing transcription/description.

## Best Possible Solution

Define a typed native audio/video media contract or provider-owned transport that preserves existing request policy, update consumers/docs/changelog/tests, and validate native/fallback paths with redacted runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this PR adds a new feature. Source inspection gives a high-confidence way to verify the patch defects through the current runtime output contract, pi-ai content types, and media transport code paths.

## Solution Assessment

No. The current patch casts unsupported audio/video blocks through `complete()` and misses transport-policy plumbing; the safer approach is a typed native media contract or provider-owned transport with the runtime consumers updated together.
