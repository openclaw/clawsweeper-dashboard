## Summary

Keep open: this is a broad user-facing feature PR that is not implemented on current main, and the proposed patch has blocking media contract, transport, and proof issues before it can be merged.

## What This Changes

The branch adds native `understandAudio` and `understandVideo` runtime paths, auto-registers audio/video-capable configured providers, and introduces new media-understanding result kinds with fallback to existing transcription/description flows.

## Best Possible Solution

Define a supported native audio/video media contract first, preserve existing media egress/request controls, add focused regression coverage and docs/changelog updates, then re-review the feature as a deliberate API expansion.

## Reproduction Assessment

Not applicable as a bug reproduction because this PR adds a new feature. Source inspection does provide a high-confidence way to verify the patch defects through the `pi-ai` content types and the existing media transport contract.

## Solution Assessment

No. Wrapping generic `complete()` with ad hoc audio/video blocks is not the narrow maintainable solution until the upstream/provider payload contract and OpenClaw transport semantics are explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Use a supported message content contract:** `src/media-understanding/audio-understanding.ts:103-109`
  - body: The new native audio path builds a `Context` with `{ type: "audio" }` and then casts the user message to `Record<string, unknown>`. `@mariozechner/pi-ai@0.73.0` only types user content as text or image blocks, so this bypasses TypeScript and gives provider transports a block they do not know how to convert. The video helper repeats the same pattern; please add a real supported media contract or provider-owned request path before enabling this.
  - confidence: 0.91
- **[P2] Preserve proxy/request controls on native media calls:** `src/media-understanding/audio-understanding.ts:200-204`
  - body: The runner passes `fetchFn`, headers, and request transport overrides into `understandAudio`, but this implementation resolves a model and calls `complete()` without forwarding any of those controls. Native audio/video understanding would bypass the documented media provider proxy/request path that existing transcription and video description providers honor.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/media-understanding/types.ts:4-9`
  - body: This PR adds user-visible media behavior and new result kinds, but it does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing `feat` changes before merge.
  - confidence: 0.83
