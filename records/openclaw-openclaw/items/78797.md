## Summary

Keep open for maintainer review: the feature direction is plausible, but this PR still has blocking media-contract, dependency, consumer-output, and transport/auth issues, and the supplied proof does not demonstrate the native runtime path clearly enough for an external PR.

## What This Changes

The PR adds native `understandAudio`/`understandVideo` media-understanding hooks, new audio/video result kinds, native-first runner paths with fallback, and config-provider auto-registration for audio/video-capable models.

## Best Possible Solution

Define a typed native audio/video media contract or provider-owned transport that preserves existing auth/request policy and consumer output semantics, then update docs/tests and prove native/fallback behavior with redacted runtime diagnostics.

## Reproduction Assessment

Not applicable as a bug reproduction because this PR adds a feature. Source inspection gives a high-confidence way to verify the patch blockers through current media consumers, the pi-ai dependency contract, and the native helper transport path.

## Solution Assessment

No. The requested direction is plausible, but this patch is not the best solution because it uses unsupported content blocks, leaves existing consumers keyed to legacy result kinds, imports the old pi-ai package name, and bypasses configured transport/auth policy.

  - body: Current main depends on `@earendil-works/pi-ai` and no longer has `@mariozechner/pi-ai` in `package.json` or the lockfile. These added files will fail module resolution after rebasing onto main unless they use the current package name.

  - body: This casts `type: "audio"` blocks into `Context`, but pi-ai's user-message type only supports text and image blocks, and its OpenAI-completions converter serializes every non-text block as `image_url`.   - confidence: 0.91

  - body: A native success returns `audio.understanding` or `video.understanding`, but `applyMediaUnderstanding` and `runMediaUnderstandingFile` still look only for `audio.transcription` and `video.description`. Native audio will not set the transcript/applied flags, and file callers can get `text: undefined` despite a successful result.
  - confidence: 0.92
- **[P2] Reuse the resolved media transport context:** `src/media-understanding/audio-understanding.ts:200-205`
  - body: The runner resolves rotated API keys, base URL, headers, request overrides, and proxy-aware fetch before calling this helper, but the helper re-resolves auth and calls `complete()` with only its own API key, token limit, signal, and timeout. Native media can therefore leave through different auth/network policy than the configured media provider path; the video helper repeats this.
  - confidence: 0.88
