- **[P1] Use a typed native media content path:** `src/media-understanding/audio-understanding.ts:103-106`
  - body: `buildAudioContext` inserts `{ type: "audio" as any }` into an old pi-ai `Context`, but current main's LLM content and provider transports only define text/image content, and OpenAI-compatible adapters serialize non-text blocks as image input.   - confidence: 0.93
- **[P1] Honor provider transport and auth overrides:** `src/media-understanding/audio-understanding.ts:185-205`
  - body: `runProviderEntry` passes `baseUrl`, `headers`, `request`, `fetchFn`, profile data, and rotated keys into `understandAudio`, but this runner ignores those transport controls and calls `complete()` directly after registering a stream. That can bypass configured provider routing, proxy/private-network policy, headers, and auth behavior; wire native media through the same provider execution path as existing media providers. The video runner repeats this.
  - confidence: 0.91
- **[P2] Wire the new output kinds through media application:** `src/media-understanding/types.ts:4-9`
  - body: Adding `audio.understanding` and `video.understanding` expands the output union, but `applyMediaUnderstanding` and the shared formatter still special-case only `audio.transcription`, `image.description`, and `video.description`. Native audio outputs will not set transcript/body/skip-state behavior correctly, and unknown kinds fall into the video-description formatting branch; update the consumers and tests before returning these kinds.
  - confidence: 0.9
