Full review comments:

- **[P1] Use a typed native media content contract:** `src/media-understanding/audio-understanding.ts:103-106`
  - body: The new runner injects `{ type: "audio" as any }` into a pi-ai context that only types user content as text or image blocks, and pi-ai 0.73 maps non-text blocks on the OpenAI-completions path as `image_url`. Define or use a real native audio/video media contract before sending these bytes.
  - confidence: 0.95
- **[P1] Route native calls through provider transport controls:** `src/media-understanding/audio-understanding.ts:200-205`
  - body: `runProviderEntry` passes rotated auth plus `baseUrl`, `headers`, `request`, and `fetchFn`, but the native runner resolves auth again and calls `complete()` without those controls. This can bypass custom endpoints, headers, proxy/private-network policy, no-auth providers, and key-rotation behavior.
  - confidence: 0.92
- **[P1] Gate raw-media egress before auto-registering providers:** `src/media-understanding/provider-registry.ts:60-67`
  - body: Auto-registering native audio handlers from `models.providers.*.models[].input` lets existing capability metadata start sending original audio bytes to a provider without explicit native-media opt-in, payload/retention policy, or audit story. Keep raw media egress behind a maintainer-approved policy gate.
  - confidence: 0.88
- **[P2] Wire native output kinds through existing consumers:** `src/media-understanding/types.ts:6-8`
  - body: Adding `audio.understanding` and `video.understanding` expands the output union, but current consumers and shared media-understanding contracts still exact-match `audio.transcription` and `video.description` for transcript/body/runtime behavior. Update those paths and tests before returning the new kinds.
  - confidence: 0.88
