## Summary

Keep open: the feature direction is useful, but the PR is not mergeable as submitted. The current branch introduces new native audio/video result kinds and provider hooks without updating all consumers, relies on an unsupported pi-ai audio/video content shape, and the author has acknowledged that maintainer guidance is needed on the dependency/transport contract.

## What This Changes

The PR adds native `understandAudio`/`understandVideo` provider hooks, audio/video-capable config-provider auto-registration, new media result kinds, and native-first media-understanding runner paths with fallback to transcription/description.

## Best Possible Solution

Define a typed native audio/video media contract or provider-owned transport that preserves existing request policy, update all media consumers/docs/tests together, and validate native/fallback behavior with redacted runtime proof.

## Reproduction Assessment

Not applicable as a bug reproduction because this PR adds a new feature. Source inspection gives a high-confidence way to verify the patch defects through the media output contract, pi-ai content types, and transport plumbing.

## Solution Assessment

No. The requested direction is plausible, but this patch is not the best solution because it casts unsupported audio/video blocks through `complete()` and misses existing transport-policy plumbing; a typed native media contract or provider-owned transport is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Normalize native result kinds before returning:** `src/media-understanding/runner.entries.ts:666`
  - body: A successful native audio/video call returns `audio.understanding` or `video.understanding`, but existing consumers still look for `audio.transcription` and `video.description`. `runMediaUnderstandingFile` can return `text: undefined`, and chat media application will not set the audio transcript/applied flags for native audio results.
  - confidence: 0.92
- **[P1] Use a supported native media content contract:** `src/media-understanding/audio-understanding.ts:103-106`
  - body: This builds `type: "audio"` and `type: "video"` blocks through `as any`, but the pinned pi-ai context only accepts text/image user blocks and the OpenAI-completions converter serializes non-text blocks as `image_url`. Add a real audio/video contract or provider-owned transport before routing native media through `complete()`.
  - confidence: 0.91
- **[P2] Preserve media transport controls in native calls:** `src/media-understanding/audio-understanding.ts:200-205`
  - body: The runner resolves `baseUrl`, headers, request overrides, rotated API keys, and proxy-aware `fetchFn`, but this `complete()` call forwards only its own API key, token limit, signal, and timeout. Native media requests can bypass configured auth overrides, proxy/TLS/private-network policy, and API-key rotation; the video helper repeats the same omission.
  - confidence: 0.88
