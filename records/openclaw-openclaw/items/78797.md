## Summary

Keep open: the feature direction is useful, but the patch is not merge-ready because it sends unsupported audio/video blocks through `pi-ai`, drops existing media transport controls, and needs stronger real behavior proof.

## What This Changes

The PR adds native `understandAudio` and `understandVideo` media-understanding paths, new result kinds, auto-registration for audio/video-capable configured providers, and fallback to existing transcription/description flows.

## Best Possible Solution

Define a supported native audio/video media contract first, preserve existing media egress/request controls, add focused regression coverage plus docs/changelog updates, then validate with redacted real provider output.

## Reproduction Assessment

Not applicable as a bug reproduction because this PR adds a new feature. Source inspection gives a high-confidence way to verify the patch defects through the `pi-ai` content types and media runner transport contract.

## Solution Assessment

No. The current approach uses casts and generic `complete()` calls before the provider payload and transport contracts support native audio/video; the maintainable path is to formalize that contract and preserve request controls first.
