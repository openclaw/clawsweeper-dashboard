## Review Findings

Full review comments:

- **[P1] Use a typed native media content contract:** `src/media-understanding/audio-understanding.ts:103-106`
  - body: The new runner injects `{ type: "audio" as any }` into a pi-ai Context contract that only supports text/image content, and pi-ai 0.73 serializes non-text OpenAI-compatible blocks as `image_url`. The video sibling has the same problem, so define or use a typed native media/provider contract before sending these bytes.
  - confidence: 0.95
- **[P1] Honor provider transport and auth controls:** `src/media-understanding/audio-understanding.ts:200-205`
  - body: `runProviderEntry` passes rotated auth plus `baseUrl`, `headers`, `request`, and `fetchFn`, but the native runner resolves auth again and calls `complete()` without those controls. This can bypass custom endpoints, headers, proxy/private-network policy, no-auth providers, and key-rotation behavior.
  - confidence: 0.92
- **[P1] Gate raw-media egress explicitly:** `src/media-understanding/provider-registry.ts:60-67`
  - body: Auto-registering native audio handlers from `models.providers.*.models[].input` lets existing capability metadata start sending original audio bytes to a provider without an explicit native-media opt-in, payload/retention policy, or audit story. Keep raw media egress behind a maintainer-approved policy gate.
  - confidence: 0.88
- **[P2] Wire native output kinds through consumers:** `src/media-understanding/types.ts:4-9`
  - body: Adding `audio.understanding` and `video.understanding` expands the output union, but the application, file runtime, and formatting paths still exact-match `audio.transcription` and `video.description` for transcript/body behavior. Update those consumers and tests before returning the new kinds.
  - confidence: 0.88
