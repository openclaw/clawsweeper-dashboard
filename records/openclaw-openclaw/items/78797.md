The PR adds native audio/video media-understanding hooks, new result kinds, auto-registration for audio/video-capable configured models, and native-first runner paths with fallback to existing transcription/description flows.

## Best Possible Solution

Define a supported native media contract or provider-owned native transport first, preserve existing request/proxy controls, update docs/changelog and regression coverage, then validate native and fallback paths with redacted real runtime proof.

Not applicable as a bug reproduction because this PR adds a new feature rather than fixing broken current behavior. Source inspection gives a high-confidence way to verify the patch defects through the pi-ai content types and media runner transport contract.

## Solution Assessment

No. The current approach casts unsupported audio/video blocks through complete() before the provider payload and transport contracts support native media; a safer direction is a formal media content contract or provider-specific native transport that preserves request controls.
