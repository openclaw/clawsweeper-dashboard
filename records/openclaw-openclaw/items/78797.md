## Summary

Keep open: this is a broad feature/API expansion, current main does not already implement it, and the proposed patch has blocking contract and transport issues that the author has already asked maintainers to resolve.

## What This Changes

The PR adds native `understandAudio` and `understandVideo` media-understanding hooks, new `audio.understanding` and `video.understanding` result kinds, auto-registration for configured audio/video-capable providers, and fallback to existing transcription/description flows.

## Best Possible Solution

Define a supported native audio/video media contract first, preserve existing provider request and proxy controls, update docs/changelog and regression coverage, then validate with redacted real provider output that shows the native result and fallback paths.

## Reproduction Assessment

Not applicable as a bug reproduction because this PR adds a new feature rather than fixing broken current behavior. Source inspection gives a high-confidence way to verify the patch defects through the pi-ai content types and media runner transport contract.

## Solution Assessment

No. The current approach uses casted audio/video blocks through `complete()` before the provider payload and transport contracts support native media; a safer direction is a formal media content contract or provider-specific native transport that preserves request controls.
