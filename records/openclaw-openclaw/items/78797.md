Full review comments:

- **[P1] Use a typed native media content path:** `src/media-understanding/audio-understanding.ts:103-106`
  - body: `buildAudioContext` inserts `{ type: "audio" as any }` into a text/image-only content contract, and the OpenAI-compatible adapters serialize non-text blocks as image payloads. The video runner repeats this pattern, so the advertised native media path cannot work reliably until audio/video are first-class typed content blocks with provider-specific serialization.
  - confidence: 0.93
- **[P1] Honor provider stream and auth controls:** `src/media-understanding/audio-understanding.ts:185-205`
  - body: The native runner accepts execution-context fields from `runProviderEntry`, but then resolves auth again and calls `complete` directly instead of using the provider stream wrapper, rotated key, baseUrl, headers, request overrides, and proxy fetch. This can break configured providers and bypass transport controls; the video runner has the same issue.
  - confidence: 0.91
- **[P2] Wire the new output kinds through media consumers:** `src/media-understanding/types.ts:4-9`
  - body: Adding `audio.understanding` and `video.understanding` expands the output union, but current apply/format consumers only special-case `audio.transcription`, `image.description`, and `video.description`. Native audio outputs would miss transcript, echo, appliedAudio, and file-extraction skip behavior, while shared formatting can label unknown output as video.
  - confidence: 0.9
