## Summary

Keep open: current main still has timer-based inbound debounce and active-run collect queueing, but not the requested user-controlled continuation marker that holds marked fragments until an unmarked final message arrives.

## What This Changes

The issue requests a configurable continuation marker that buffers multi-part inbound messages and invokes the agent only when a final unmarked message arrives.

## Best Possible Solution

Keep this open for a product decision on whether marker-based batching belongs in shared inbound handling, a documented plugin/API seam, or should be declined in favor of existing debounce and collect modes.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested marker config, schema, policy parsing, marker stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear. The requested marker workflow is plausible and backward-compatible, but maintainers should first decide how it should interact with existing inbound debounce, active-run collect mode, media/control-command behavior, and typing/status semantics.
