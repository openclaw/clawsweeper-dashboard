## Summary

Keep open. Current main still lacks the requested marker-controlled continuation batching; existing queue collect and inbound debounce behavior only provide timing or active-run batching, and two recent issue comments reached the same keep-open conclusion.

## What This Changes

The issue requests configurable continuation markers that buffer rapid user message fragments until a later unmarked message is received, reducing redundant agent turns and context loads.

## Best Possible Solution

Define the shared continuation-batching contract first, then add typed config/schema/docs plus marker parsing, marker stripping, timeout flush behavior, queue/typing integration, and focused inbound coverage if core ownership is accepted.

## Reproduction Assessment

Not applicable. This is an enhancement request, and source inspection on current main shows the requested marker config and parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The marker workflow is plausible, but maintainers need to decide whether this belongs in shared core config, a channel/plugin API seam, or a documented non-goal in favor of existing debounce/collect modes.
