## Summary

Keep open: current main and the latest release still have only timer-based inbound debouncing and active-run queue collect behavior, not the requested configurable marker that buffers marked fragments until a later unmarked message. The remaining work is a product/API decision because it would add new cross-channel message grouping semantics and public config.

## What This Changes

Request a configurable continuation marker and timeout that batch quick successive user messages into one agent turn, with optional typing suppression for continuation fragments.

## Best Possible Solution

Decide the cross-channel product/API contract first, then either document timer-based batching as the intended answer or implement an opt-in shared inbound or SDK-level continuation-marker flow with schema, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing or stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers choose the product shape. The marker proposal is plausible, but the maintainable solution needs explicit cross-channel semantics rather than a narrow local queue tweak.
