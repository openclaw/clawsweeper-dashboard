## Summary

Keep open: current main and the latest release still provide timer-based inbound debouncing plus active-run collect queueing, not the requested marker-controlled batching that buffers marked fragments until a later unmarked message.

## What This Changes

Request a configurable message continuation marker and timeout so multiple inbound message fragments can be buffered into one agent turn before context loading.

## Best Possible Solution

Choose the product boundary first, then either add a minimal typed continuation/timeout contract in shared inbound handling or expose a plugin/API seam; keep timer-based debounce as the current default until that decision is made.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior report. Source inspection confirms current main has timer-based inbound debounce and active-run collect queueing, not marker-controlled batching.

## Solution Assessment

No, not as specified: the marker, timeout, queue mode, and typing mode are proposed implementation choices that add core config/session semantics. Maintainers should first choose whether this belongs in core inbound handling, a narrow plugin/API seam, or remains covered by timer-based batching.
