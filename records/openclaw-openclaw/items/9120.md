## Summary

Keep open. Current main still has fixed timer-based inbound debouncing and active-run collect queueing, but not the requested user-controlled continuation marker that holds marked fragments until a later unmarked message; prior review comments and current labels also route this to maintainer/product follow-up.

## What This Changes

Request a configurable continuation marker such as `(cont)` that buffers marked inbound message fragments and invokes the agent once when an unmarked final message arrives.

## Best Possible Solution

Define the semantic batching contract first; if accepted, implement shared inbound continuation config, marker stripping, timeout/media/control-command behavior, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. The high-confidence check is a negative current-main source/config/docs review showing no marker-based hold-and-flush support.

## Solution Assessment

Unclear until maintainers decide the product shape. If accepted, a shared inbound-buffering contract with typed config/schema/docs/tests is the maintainable path, not channel-specific marker patches.
