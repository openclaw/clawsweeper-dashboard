## Summary

Keep open. Current main still provides timer-based inbound debouncing and active-run collect/steer queue behavior, but it does not implement the requested user-controlled continuation marker that holds marked fragments until a later unmarked message arrives; the issue is also already labeled for maintainer review and product decision.

## What This Changes

The issue requests a configurable message continuation marker that buffers marked inbound fragments and invokes the agent only after an unmarked final message arrives.

## Best Possible Solution

Define an explicit product contract for semantic continuation batching before implementation, then add typed config/schema/docs plus shared inbound buffering tests if maintainers want it in core.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection confirms current main has timer-based inbound debouncing and active-run queue collection, but no marker-based hold-and-flush path.

## Solution Assessment

Unclear until maintainers decide the product shape. A marker can be implemented cleanly only after defining timeout, marker stripping, media/control-command behavior, typing behavior, and interaction with existing inbound debounce and queue modes.
