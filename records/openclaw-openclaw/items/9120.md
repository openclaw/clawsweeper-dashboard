## Summary

Keep open: current main and the latest release provide timer-based inbound debouncing and active-run collect queueing, but they still do not implement a user-controlled continuation marker that holds marked fragments until an unmarked final message arrives. Related open issues cover broader semantic aggregation and channel-specific batching, but none fully supersedes this marker-specific product/API decision.

## What This Changes

The issue requests a configurable continuation marker that buffers marked inbound message fragments and invokes the agent once when a later unmarked message arrives.

## Best Possible Solution

Decide the cross-channel product/API contract first, then implement an opt-in shared inbound or SDK-level continuation-marker flow with config schema/types/docs and focused behavior tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing/stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. The marker proposal is plausible, but the maintainable solution needs explicit cross-channel semantics rather than a narrow local queue tweak.
