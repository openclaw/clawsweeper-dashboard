## Summary

Keep open. Current main has fixed timer-based inbound debouncing and active-run collect queueing, but no user-controlled continuation marker that holds marked fragments until a later unmarked message; existing reviews and labels also route this to maintainer/product follow-up.

## What This Changes

The issue requests configurable continuation markers that buffer marked multi-message input and invoke one batched agent turn when a final unmarked message arrives.

## Best Possible Solution

Define the shared semantic batching contract first; if accepted, implement typed config/schema/docs, marker stripping, timeout and edge-case behavior, and focused regression tests in shared inbound handling.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. The high-confidence check is a negative current-main source/config/docs review showing no marker-based hold-and-flush support.

## Solution Assessment

Unclear until maintainers decide the product shape. If accepted, a shared inbound-buffering contract with typed config/schema/docs/tests is the maintainable path, not channel-specific marker patches.
