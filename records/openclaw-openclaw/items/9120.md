## Summary

Keep open: current main and v2026.6.10 still provide fixed timer-based inbound debouncing plus active-run collect queueing, but not the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message.

## What This Changes

The issue requests a configurable continuation marker that buffers multi-message input until a final unmarked message to reduce repeated agent turns and context loads.

## Best Possible Solution

Keep this open as the focused marker-controlled continuation design item until maintainers either approve a shared inbound/plugin API contract or explicitly fold it into a broader inbound batching policy.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior report. Source inspection gives high-confidence evidence that current main supports timer-based inbound debounce and active-run collect batching, but not marker-controlled continuation buffering.

## Solution Assessment

No, not as an automatic fix: the proposed marker, timeout, queue mode, and typing mode are plausible but add core config and session turn semantics. The best path is a maintainer-approved shared inbound or plugin/API continuation contract, or an explicit product decision that existing timer-based batching is the intended answer.
