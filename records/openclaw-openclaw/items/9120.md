## Summary

Current main still lacks user-controlled continuation semantics: inbound debounce only batches by elapsed time, while collect mode applies after an agent run is already active. The request is still meaningful, but it introduces a new configuration and deferred-message-state contract, and the existing discussion already identifies unresolved media, commands, timeout, typing, and ownership choices that require maintainer product judgment.

## What This Changes

Add a configurable continuation suffix and timeout that buffer marked inbound text fragments until an unmarked message completes one combined agent turn.

## Best Possible Solution

Define one typed pre-agent message-assembly contract—core only if this is a supported cross-channel workflow, otherwise through a documented plugin hook—with marker stripping, bounded timeout/state, sender and thread isolation, immediate control-command handling, explicit media behavior, and composition tests for debounce, collect queueing, transcript order, and typing status.

## Reproduction Assessment

Not applicable: this is a clearly specified request for new behavior, not a report that an existing documented workflow is broken. Source inspection does confirm that current main has no marker-driven batching path.

## Solution Assessment

Unclear: a suffix marker can solve the user workflow, but the proposed parallel queue and typing modes are not yet the best proven architecture. Maintainers should first select the canonical pre-agent assembly boundary and decide whether the behavior is a core cross-channel contract or requires a new plugin seam.
