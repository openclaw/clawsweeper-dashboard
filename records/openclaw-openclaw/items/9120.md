## Summary

Keep open. Current main and the latest release still provide timer-based inbound debouncing plus active-run queue collection, but not the requested user-controlled continuation marker that holds marked fragments until a later unmarked message; existing labels and prior review comments also route this to maintainer/product follow-up.

## What This Changes

The issue requests a configurable continuation marker that buffers marked inbound message fragments and invokes the agent only after a final unmarked message arrives.

## Best Possible Solution

Decide the semantic batching contract first; if accepted, implement typed messages continuation config, marker stripping, flush/timeout/media/control-command behavior, docs, and focused tests in the shared inbound path.
