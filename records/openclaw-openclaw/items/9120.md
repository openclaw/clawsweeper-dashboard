## Summary

Keep open: current main still has timer-based inbound debouncing and active-run collect queueing, but not the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message arrives. The live discussion already contains two prior reviews keeping it open for the same missing product/config behavior, and no closing pull request or canonical duplicate was found.

## What This Changes

Requested support for a configurable continuation marker that batches multiple inbound user messages into one agent turn before invoking the model.

## Best Possible Solution

Decide the cross-channel product contract first, then implement an opt-in shared inbound or SDK-level continuation-marker flow with schema/types/docs and focused tests for marker stripping, timeout flush, media/control-command bypass, queue interaction, and typing/status behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing/stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. The marker proposal is plausible, but the maintainable solution needs explicit cross-channel semantics rather than a narrow local queue tweak.
