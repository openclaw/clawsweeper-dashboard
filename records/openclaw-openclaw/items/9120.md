## Summary

Keep open. Current main still has timer-based inbound debouncing and active-run queue collection, but not the requested user-controlled continuation marker that buffers marked fragments until an unmarked final message; the issue also has two recent review comments reaching the same keep-open conclusion.

## What This Changes

The issue requests configurable marker-based batching so messages ending with a continuation marker are buffered until a later unmarked message triggers one combined agent turn.

## Best Possible Solution

Define the shared continuation-batching contract first, then implement config/schema/docs, marker parsing and stripping, timeout flush behavior, queue and typing interactions, and focused inbound coverage.

## Reproduction Assessment

Not applicable. This is an enhancement request, and source inspection on current main shows the requested marker config and parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The marker workflow is plausible, but maintainers still need to decide whether this belongs in shared core config, a channel/plugin API seam, or an explicit non-goal in favor of existing debounce/collect modes.
