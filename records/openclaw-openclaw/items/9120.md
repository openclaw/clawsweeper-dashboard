## Summary

Keep open: current main still has timer-based inbound debouncing and active-run collect batching, but it does not implement the requested user-controlled continuation marker, marker config, marker stripping, or continuation-specific typing behavior.

## What This Changes

Add configurable continuation-marker batching so marked inbound message fragments are buffered and sent to the agent only when a later unmarked message arrives.

## Best Possible Solution

Define the cross-channel continuation-batching contract first; if accepted, implement typed config/schema/docs plus marker parsing, marker stripping, timeout flushing, queue/typing integration, and focused inbound coverage.

## Reproduction Assessment

Not applicable. This is an enhancement request, and current-main source inspection shows the requested marker config and parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The marker workflow is plausible for cost savings, but maintainers need to choose whether shared core config, a plugin/channel seam, or existing debounce/collect modes should be the supported direction.
