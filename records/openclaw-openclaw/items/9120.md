## Summary

Keep open. Current main still provides timer-based inbound debouncing and active-run queue collection, but it does not implement the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message. The issue discussion already includes an Apr 26, 2026 maintainer-side automated review keeping it open for the same missing behavior, and the current code/docs inspection still matches that conclusion.

## What This Changes

The issue requests a configurable message continuation marker, defaulting to `(cont)`, that buffers marked inbound fragments without starting an agent turn and flushes the combined content when a later unmarked message arrives.

## Best Possible Solution

Keep the issue open for an explicit product/API decision. A complete implementation would add typed config and docs for continuation marker behavior, buffer only supported inbound text fragments before agent invocation, strip the marker from model-visible text, flush on an unmarked message or timeout, and define interactions with media, control commands, queue modes, and typing/status indicators.

## Reproduction Assessment

Not applicable as a feature request, but the absence is verifiable on current main: the config schema lacks the proposed fields, inbound debounce flushes by timer, and the inbound policy does not inspect message suffix markers.

## Solution Assessment

Unclear until maintainers choose the product shape. The narrowest maintainable solution is likely a shared inbound helper or documented plugin/API seam rather than channel-specific ad hoc buffering.
