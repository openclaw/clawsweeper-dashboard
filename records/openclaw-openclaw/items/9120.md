## Summary

Keep open: current main still has timer-based inbound debouncing and active-run collect batching, but not the requested user-controlled continuation marker, marker config, or continuation-specific typing mode.

## What This Changes

Requests configurable marker-based buffering so marked inbound message fragments are batched into one later agent turn.

## Best Possible Solution

Define the cross-channel continuation-batching contract first; if accepted, implement typed config/schema/docs plus marker parsing, marker stripping, timeout flushing, queue/typing integration, and focused inbound coverage.

Not applicable. This is an enhancement request, and current-main source inspection shows the requested marker config and parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The marker workflow is plausible, but maintainers need to choose whether shared core config, a plugin/channel seam, or existing debounce/collect modes should be the supported direction.
