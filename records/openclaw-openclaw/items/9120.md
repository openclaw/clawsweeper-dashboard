## Summary

Keep open: current main and the latest release still provide timer-based inbound debounce and active-run collect queueing, but not the requested configurable continuation marker that buffers marked fragments until a later unmarked message arrives. This remains a product/API decision rather than a cleanup close.

## What This Changes

Request a configurable message continuation marker that buffers marked inbound message fragments, strips the marker, and invokes one batched agent turn when the final unmarked message arrives.

## Best Possible Solution

Decide the cross-channel product/API contract first, then either document timer-based batching as the intended answer or implement an opt-in shared inbound or SDK-level continuation-marker flow with schema, docs, and behavior tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing or stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers choose the product shape. The marker proposal is plausible, but the maintainable solution needs explicit shared semantics rather than a narrow local queue tweak.
