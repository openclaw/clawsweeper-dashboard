## Summary

Keep open: current main still lacks the requested marker-based batching flow. Existing inbound debouncing and queue collect behavior are timer/active-run based, while this request needs a new user-controlled config/API contract for holding marked fragments until a later unmarked message.

## What This Changes

Add an opt-in configurable continuation marker, such as `(cont)`, that buffers multiple inbound messages and invokes the agent once when an unmarked final message arrives.

## Best Possible Solution

Decide the cross-channel product contract first, then implement an opt-in shared inbound or SDK-level continuation-marker flow with config schema/types/docs and focused behavior tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing/stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. The marker proposal is plausible, but the maintainable solution needs explicit cross-channel semantics rather than a narrow local queue tweak.
