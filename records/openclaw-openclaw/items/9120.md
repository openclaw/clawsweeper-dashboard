## Summary

Keep open: current main and the latest release still provide fixed timer-based inbound debouncing plus active-run collect queueing, not a user-controlled continuation marker that holds marked fragments until a later unmarked final message.

## What This Changes

Request to add marker-driven inbound message batching, with configurable marker/timeout behavior and optional typing suppression, to reduce redundant agent turns for multi-part user input.

## Best Possible Solution

Keep the current timer-based batching as the default, and only add a minimal marker/timeout contract in shared inbound handling or a typed plugin hook after maintainers choose the product boundary.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior report. Source inspection confirms current main follows documented timer debounce and active-run queue behavior, not marker-controlled batching.

## Solution Assessment

No, not yet: the requested marker, queue mode, and typing mode are implementation options rather than an approved canonical contract. Maintainers should first choose whether this belongs in core inbound handling, a narrow plugin/API seam, or remains covered by timer-based batching.
