## Summary

Keep open. Current main still has timer-based inbound debouncing and active-run collect queueing, but it does not implement a user-controlled continuation marker that buffers marked fragments until a later unmarked message; the issue discussion also contains a prior maintainer-side review keeping this open for the same gap.

## What This Changes

The issue requests configurable message continuation markers, timeout handling, marker stripping, and typing/queue interactions for batching multi-part user input into one agent turn.

## Best Possible Solution

Define the shared continuation contract first, then implement typed config/schema/docs, marker parsing and stripping, timeout flushing, queue and typing interactions, and focused regression coverage in the inbound/queue path.

## Reproduction Assessment

Not applicable. This is an enhancement request, and source inspection on current main shows the requested marker config and marker parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The proposed direction is plausible, but the marker syntax, timeout behavior, marker stripping, media/control-command handling, and typing semantics need maintainer product/API decisions before a narrow implementation should proceed.
