## Summary

Keep open: current main has fixed/renewable inbound debouncing and active-run collect/steer queueing, but it does not implement the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message arrives.

## What This Changes

The issue requests configurable marker-based batching, such as `(cont)`, so multiple quick user messages are held and submitted as one agent turn only after a final unmarked message.

## Best Possible Solution

Decide the product contract first, then implement an opt-in shared inbound or SDK-level continuation-marker flow with schema/types/docs and focused tests for marker stripping, timeout flush, media/control-command bypass, queue interaction, and typing/status behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing/stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. The proposed marker is plausible, but the durable solution needs explicit rules for config, channel ownership, media/control commands, timeout behavior, and status indicators.
