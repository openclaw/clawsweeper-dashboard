## Summary

Current main and v2026.6.5 provide timer-based inbound debouncing and active-run collect queueing, but neither implements the requested user-controlled continuation marker that delays the first agent invocation. The feature remains meaningful, while its new configuration, routing, deferred-state, timeout, media, command, and typing semantics require maintainer product judgment.

Likely related people: Vincent Koc (high confidence, recent inbound debounce and queue contributor), Agustin Rivera (high confidence, collect-queue contributor), and steipete (medium confidence, shared inbound policy contributor and prior reviewer).

## What This Changes

Add an explicit continuation marker and timeout so marked inbound fragments wait for a later final message before invoking the agent.

## Best Possible Solution

Define one canonical shared pre-agent continuation contract with explicit routing, persistence, buffer limits, timeout, marker stripping, command/media, and typing behavior; implement only that bounded path if maintainers prefer explicit continuation over documenting timed inbound debounce as the supported solution.

## Reproduction Assessment

Not applicable: this issue proposes new user-controlled continuation behavior rather than reporting a failure of an established contract. Current-main source inspection clearly establishes that the proposed behavior is absent.

## Solution Assessment

Unclear: a continuation marker directly represents user intent, but the proposed extra queue and typing modes are not yet shown to be better than one shared pre-agent buffering contract or the existing timed debounce option.
