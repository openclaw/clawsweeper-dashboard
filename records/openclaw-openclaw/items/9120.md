## Summary

Keep open. Current main still provides timer-based inbound debouncing and active-run queue collection, but not the requested user-controlled continuation marker that holds marked fragments until an unmarked final message; the issue discussion already has two prior reviews reaching the same conclusion.

## What This Changes

Request a configurable message continuation marker like `(cont)` that buffers marked inbound message fragments until the first unmarked message so one batched agent turn handles the combined input.

## Best Possible Solution

Define the shared continuation-batching contract, then implement typed config/schema/docs, marker parsing and stripping, timeout flushing, queue and typing interactions, and focused coverage in the inbound/queue path.

## Reproduction Assessment

Not applicable. This is an enhancement request, and source inspection on current main shows the requested marker config and marker parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The marker approach is plausible, but maintainers still need to decide whether this should be a shared core contract, a channel/plugin seam, or an explicit non-goal in favor of existing debounce/collect modes.
