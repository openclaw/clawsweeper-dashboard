## Summary

Keep open: current main and the latest release still have timer-based inbound debounce and active-run collect queueing, but not the requested configurable continuation marker that buffers marked fragments until a later unmarked message arrives. The live discussion and existing labels also point to a maintainer product/API decision rather than a cleanup close.

## What This Changes

The issue requests an opt-in message continuation marker, such as `(cont)`, that batches multiple inbound fragments into one later agent turn to reduce redundant context loads.

## Best Possible Solution

Decide the cross-channel product/API contract first, then either document timer-based batching as the intended answer or implement an opt-in shared inbound or SDK-level continuation-marker flow with config schema, docs, and focused behavior tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing or stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers choose the product shape. The marker proposal is plausible, but the maintainable solution needs explicit cross-channel semantics rather than a narrow local queue tweak.
