## Summary

Keep open: current main and v2026.6.5 support timer-based inbound debounce and active-run collect batching, but neither provides the requested user-controlled hold-until-final-fragment behavior. This remains a low-priority product/API decision because it introduces cross-channel grouping semantics and new public configuration.

## What This Changes

Add a configurable continuation marker and timeout that buffer marked inbound message fragments until an unmarked final message arrives.

## Best Possible Solution

Define a single shared pre-dispatch message-grouping contract with per-conversation isolation, explicit flush and timeout behavior, and focused cross-channel tests; add public config only if maintainers approve the marker convention over improving the existing debounce workflow.

## Reproduction Assessment

Not applicable: this issue requests new user-controlled message-grouping behavior rather than reporting a failure of an existing documented contract. Source inspection does provide a high-confidence comparison showing that current debounce and collect modes are not equivalent.

## Solution Assessment

No, not as currently proposed: channel-specific buffering or a parallel queue mode would duplicate shared inbound behavior. The maintainable direction is a shared pre-dispatch grouping contract after maintainers decide its public semantics and whether new configuration is justified.
