## Summary

Keep open: current main and the latest release still provide timer-based inbound debounce plus active-run collect queueing, not the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message.

## What This Changes

The issue requests configurable marker-driven batching for multi-message user input before invoking an agent.

## Best Possible Solution

Keep this as a design-reviewed enhancement until maintainers either declare timer-based batching the intended answer or approve a minimal typed continuation-buffer contract in shared inbound handling or a plugin/API seam.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior report. Source inspection gives high-confidence evidence that current main lacks marker-controlled batching while supporting timer-based debounce and active-run collect behavior.

## Solution Assessment

No, not exactly as specified: the marker, timeout, queue mode, and typing mode are proposed implementation choices that add core config/session semantics. The maintainable path is a maintainer-approved shared inbound or plugin/API contract, or an explicit decision that existing timer-based batching is the product answer.
