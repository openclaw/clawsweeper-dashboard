## Summary

Keep open. Current main still only provides timer-based inbound debouncing and active-run queue collection, not the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message; the existing issue discussion also has prior maintainer-side automated reviews keeping it open for the same gap.

## What This Changes

The issue requests a configurable message continuation marker that buffers marked inbound fragments and invokes one agent turn after an unmarked final message.

## Best Possible Solution

Keep this open for a maintainer product/API decision, then implement shared inbound continuation handling with typed config, marker stripping, timeout flushing, and documented interactions with media, control commands, queue modes, and typing/status indicators.

## Reproduction Assessment

Not applicable as a feature request. The absence is high-confidence from source inspection: the schema lacks the proposed fields, inbound debounce flushes by timer, and inbound policy does not inspect suffix markers.

## Solution Assessment

Unclear until maintainers choose the product shape. The narrowest maintainable direction is a shared inbound helper or documented plugin/API seam, not channel-specific ad hoc buffering.
