## Summary

Keep open. Current main still lacks the requested user-controlled continuation marker; existing inbound debounce is timer-based and collect queueing only batches active-run followups, and the live discussion already kept this enhancement open for that missing behavior.

## What This Changes

The issue requests configurable marker-based buffering so marked multi-message input is batched into one later agent turn.

## Best Possible Solution

Define the shared continuation-batching contract first; if accepted, add typed config/schema/docs plus marker parsing, marker stripping, timeout flush behavior, queue/typing integration, and focused inbound coverage.

## Reproduction Assessment

Not applicable. This is an enhancement request, and current-main source inspection shows the requested marker config and parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The marker workflow is plausible, but maintainers need to decide whether shared core config, a channel/plugin seam, or the existing debounce/collect modes should be the supported product direction.
