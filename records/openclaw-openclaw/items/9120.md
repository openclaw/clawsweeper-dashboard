## Summary

Keep open: current main and v2026.6.8 still provide timer-based inbound debouncing plus active-run collect queueing, not the requested user-controlled continuation marker that buffers fragments until an unmarked final message.

## What This Changes

Request a configurable `(cont)`-style marker that buffers multiple inbound fragments and invokes one agent turn when the final unmarked message arrives.

## Best Possible Solution

Keep this as a design-reviewed enhancement until maintainers either declare timer-based batching the intended answer or approve a minimal shared inbound or plugin/API continuation-buffer contract with schema, docs, and focused tests.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior report. Source inspection gives high-confidence evidence that current main lacks marker-controlled batching while supporting timer-based debounce and active-run collect behavior.

## Solution Assessment

No, not exactly as specified: the marker, timeout, queue mode, and typing mode are proposed implementation choices that add core config and session semantics. The maintainable path is a maintainer-approved shared inbound or plugin/API contract, or an explicit decision that existing timer-based batching is the product answer.
