## Summary

Keep open. Current main still lacks the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message; existing inbound debounce and collect queue behavior are timing/active-run batching, and recent issue comments already kept this open on the same basis.

## What This Changes

Request a configurable message continuation marker, defaulting to `(cont)`, that buffers marked inbound fragments and sends them to the agent as one batched turn when an unmarked final message arrives.

## Best Possible Solution

Define the shared continuation-batching contract first; if accepted, add typed config/schema/docs plus marker parsing, marker stripping, timeout flush behavior, queue/typing integration, and focused inbound coverage.

## Reproduction Assessment

Not applicable. This is an enhancement request, and current-main source inspection shows the requested marker config and parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The marker workflow is plausible, but maintainers need to decide whether it belongs in shared core config, a plugin/API seam, or as a documented non-goal in favor of existing debounce and collect modes.
