## Summary

Keep open. Current main still has timer-based inbound debouncing and active-run collect queueing, but not the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message; recent issue comments reached the same keep-open conclusion.

## What This Changes

The issue requests a configurable message continuation marker that buffers multi-part inbound messages until an unmarked final message, reducing repeated context loads.

## Best Possible Solution

Define the shared continuation-batching contract first, then implement typed config/schema/docs plus marker parsing, marker stripping, timeout flush behavior, queue/typing interactions, and focused inbound coverage if core ownership is accepted.

## Reproduction Assessment

Not applicable. This is an enhancement request, and source inspection on current main shows the requested marker config and parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The marker workflow is plausible, but maintainers still need to choose whether it belongs in shared core config, a channel/plugin API seam, or an explicit non-goal in favor of existing debounce/collect modes.
