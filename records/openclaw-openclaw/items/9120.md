## Summary

Keep open. Current main still has timer-based inbound debouncing and active-run queue collection, but not the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message; prior issue comments also kept it open for the same missing behavior.

## What This Changes

Request adding a configurable message continuation marker such as `(cont)` to buffer marked inbound fragments and invoke the agent once when an unmarked final message arrives.

## Best Possible Solution

Define the shared continuation batching contract, then implement typed config/schema/docs, marker parsing and stripping, timeout flushing, queue and typing interactions, and focused regression coverage in the inbound/queue path.

## Reproduction Assessment

Not applicable. This is an enhancement request, and source inspection on current main shows the requested marker config and marker parser are absent rather than a broken existing contract.

## Solution Assessment

Unclear. The marker approach is plausible, but maintainers still need to decide whether it should be a shared core contract, a channel/plugin seam, or an explicit non-goal in favor of existing debounce/collect modes.
