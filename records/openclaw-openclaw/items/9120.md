## Summary

Keep open: current main still provides timer-based inbound debounce and active-run collect queueing, but not the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message arrives.

## What This Changes

The issue requests a configurable message continuation marker that batches multi-part user input before invoking the agent to reduce redundant context loads.

## Best Possible Solution

Keep this open for a product decision on whether marker-based batching belongs in shared inbound handling, a documented plugin/API seam, or should be declined in favor of existing debounce and collect modes.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested marker config, schema, policy parsing, marker stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear. The requested marker workflow is plausible and backward-compatible, but maintainers should first decide how it should interact with existing inbound debounce, collect queue mode, media/control-command behavior, and typing/status semantics.
