## Summary

Keep open: current main still lacks the requested user-controlled continuation marker, marker stripping, and unmarked-message flush behavior; existing support is timer-based inbound debouncing and active-run queue collection, so this remains a product/API decision rather than conservative cleanup.

## What This Changes

The issue requests an opt-in continuation marker that buffers multiple inbound user messages until an unmarked final message arrives, reducing redundant agent turns and context loading.

## Best Possible Solution

Decide the cross-channel product contract first, then implement an opt-in shared inbound or SDK-level continuation-marker flow with config schema/types/docs and focused behavior tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing/stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. The marker proposal is plausible, but the maintainable solution needs explicit cross-channel semantics rather than a narrow local queue tweak.
