## Summary

Keep open: current main has timer-based inbound debounce and active-run collect queueing, but not the requested user-controlled continuation marker with marker stripping and flush-on-unmarked-message behavior. Prior issue discussion also kept this open for the same product/config gap.

## What This Changes

Request adds a configurable continuation marker such as `(cont)` to buffer marked inbound message fragments and flush them as one batched agent turn when an unmarked message arrives.

## Best Possible Solution

Decide the product contract first, then implement an opt-in shared inbound or SDK-level continuation-marker flow with schema/types/docs and tests for marker stripping, timeout flush, media/control-command bypass, queue interaction, and typing/status behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing/stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. The proposed marker is plausible, but the durable solution needs explicit rules for config, channel ownership, media/control commands, timeout behavior, and status indicators.
