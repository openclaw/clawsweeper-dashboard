## Summary

Keep open: current main still has timer-based inbound debouncing and active-run collect queueing, but not the requested user-controlled continuation marker with marker stripping and flush-on-unmarked-message behavior. The live issue is open, has no closing PR, and exact-title/capability search did not find a canonical duplicate that owns the marker semantics.

## What This Changes

The issue requests an opt-in message continuation marker that buffers marked inbound fragments and sends them as one batched agent turn after a final unmarked message.

## Best Possible Solution

Decide the product contract first, then implement an opt-in shared inbound or SDK-level continuation-marker flow with schema/types/docs and focused tests for marker stripping, timeout flush, media/control-command bypass, queue interaction, and typing/status behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing/stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers decide the cross-channel product shape. The marker proposal is plausible, but the maintainable solution needs explicit shared semantics rather than a narrow queue tweak.
