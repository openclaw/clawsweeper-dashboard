## Summary

Keep open. Current main still has timer-based inbound debouncing and active-run queue collection, but not the requested user-controlled continuation marker; the live discussion and labels also point to maintainer/product follow-up rather than cleanup closure.

## What This Changes

The issue requests a configurable marker such as `(cont)` that buffers multiple inbound user messages and invokes the agent only when a final unmarked message arrives.

## Best Possible Solution

Define the semantic batching contract first, then add typed config/schema/docs and shared inbound buffering tests if maintainers want marker-based continuation in core.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection on current main confirms existing debounce and collect paths, but no marker-based hold-and-flush behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. A clean implementation needs decisions on marker stripping, timeout, media/control-command handling, typing behavior, and interaction with existing inbound debounce and queue modes.
