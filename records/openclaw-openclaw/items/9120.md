## Summary

Keep open: current main and v2026.6.6 still provide timer-based inbound debouncing plus active-run collect queueing, not the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message.

## What This Changes

The issue requests a configurable message continuation marker and timeout so multiple marked inbound messages are buffered into one agent turn before dispatch.

## Best Possible Solution

Keep timer-based batching as the default, then add a minimal marker/timeout contract in shared inbound handling or a typed plugin/API seam only after maintainers choose the product boundary.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior report. Source inspection confirms current main has timer-based inbound debounce and active-run collect queueing, not marker-controlled batching.

## Solution Assessment

No, not as specified: the marker, timeout, queue mode, and typing mode are proposed implementation choices that add core config/session semantics. Maintainers should first choose whether this belongs in core inbound handling, a narrow plugin/API seam, or remains covered by timer-based batching.
