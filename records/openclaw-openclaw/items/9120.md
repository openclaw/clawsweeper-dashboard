## Summary

Keep open: current main still has timer-based inbound debouncing and active-run collect queueing, but not the requested user-controlled continuation marker that buffers marked fragments until a later unmarked message. The prior issue discussion reached the same conclusion, and the remaining work is a product/config decision rather than conservative cleanup closure.

## What This Changes

Add an opt-in message continuation marker, such as `(cont)`, that buffers marked inbound message fragments and invokes the agent only after an unmarked final message arrives.

## Best Possible Solution

Decide the public message contract first, then implement an opt-in shared inbound or SDK-level continuation-marker flow with schema/types/docs and tests for marker stripping, timeout flush, media/control-command bypass, queue interaction, and typing/status behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing/stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. The proposed marker is plausible, but the durable solution needs explicit rules for config, channel ownership, media/control commands, timeout behavior, and status indicators.
