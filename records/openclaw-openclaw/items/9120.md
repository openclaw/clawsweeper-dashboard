## Summary

Keep open: current main still has timer-based inbound debounce and active-run collect queueing, but not the requested user-controlled continuation marker with marker stripping and flush-on-unmarked-message behavior. The live issue has no closing PR, and prior review discussion already kept this open for the same product/config gap.

## What This Changes

The issue requests an opt-in message continuation marker that buffers multiple marked inbound messages and flushes them as one agent turn when a final unmarked message arrives.

## Best Possible Solution

Decide the product contract first, then implement an opt-in shared inbound or SDK-level continuation-marker flow with schema/types/docs and focused tests for marker stripping, timeout flush, media/control-command bypass, queue interaction, and typing/status behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks marker config, marker parsing/stripping, and unmarked-message flush behavior.

## Solution Assessment

Unclear until maintainers decide the product shape. The marker proposal is plausible, but the maintainable solution needs explicit cross-channel semantics rather than a narrow local queue tweak.
