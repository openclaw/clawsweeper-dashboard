## Summary

Keep open. Current main does not implement the PR's full intended runtime-recovery surface, but the PR head is not correct enough to merge: raw head inspection reproduces duplicate Feishu declarations, a missing imported module, and a heartbeat path that can drop drained system events. The remaining useful work needs maintainer-directed rebase, splitting, or replacement because it is broad, overlaps open Feishu work in #39489, and changes a security-sensitive ACPX executable dependency path.

## What This Changes

The PR branch changes heartbeat system-event consumption, Feishu runtime health status propagation, exec/process terminal-status semantics, ACPX/Codex ACP launch behavior, ACP spawn guidance, first-visible reply diagnostics/config/docs, and related tests/lockfile entries.

## Best Possible Solution

Replace or split the branch into current-main-sized changes: reconcile Feishu health reporting with #39489, keep heartbeat event consumption on the current prefix-consumption model or add a narrowly tested equivalent, decide exec non-zero exit semantics with compatibility-focused tests, review ACPX launch behavior against the current package baseline, and land first-visible diagnostics separately if still desired.

Yes for the PR defects. Read-only raw-source inspection of the PR head reproduces the duplicate declarations, missing imported module, and drained heartbeat events followed by an alerts-disabled return path; local main inspection also confirms the intended broad behavior is not already implemented.

## Solution Assessment

No. This PR is not the best way to solve the runtime recovery issues as-is because it combines multiple subsystems, stale dependency changes, overlapping Feishu work, docs, diagnostics, and blocking static defects in one XL branch; smaller rebased PRs are safer.
