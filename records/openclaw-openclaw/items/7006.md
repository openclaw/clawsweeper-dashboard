## Summary

Keep open. Current main still does not expose the concrete model selected by `openrouter/auto` or OpenRouter's provider-reported cost through the OpenAI-compatible transport, session metadata, or response usage footer; the prior related PR is closed unmerged and the related open issues track adjacent cost/model visibility work rather than a shipped fix.

## What This Changes

The issue requests exposing the concrete model selected by `openrouter/auto` and its usage cost so response footers can show the resolved model and actual cost.

## Best Possible Solution

Capture OpenRouter's returned model and provider usage cost at the OpenAI-compatible transport boundary, persist selected-versus-resolved model metadata through assistant/session/transcript records, and render the resolved auto label plus accurately labeled provider cost on response usage surfaces with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to feed `processOpenAICompletionsStream` a final OpenRouter-style chunk containing a top-level resolved `model` and numeric `usage.cost`; current code ignores those fields and the footer formatter has no parameters that could display them.

## Solution Assessment

Yes. A focused transport-to-session-to-footer repair is the narrowest maintainable path; ClawRouter is a different routing service, PR #43026 did not land, and #9016/#68066 should be treated as related cost context rather than replacements for this resolved-model request.
