## Summary

Keep open: current main and v2026.6.6 partially address OpenRouter billed-cost capture, but they still do not expose the resolved OpenRouter `auto` model and provider-reported cost through agent metadata, `session_status`, or the reply footer requested here.

## What This Changes

The issue asks OpenClaw to expose the concrete model and cost returned by OpenRouter when `openrouter/auto` routes a request.

## Best Possible Solution

Define one provider-neutral selected-versus-resolved model and provider-reported cost contract, then propagate it from OpenRouter/OpenAI-compatible transport through assistant records, agent metadata, session/status APIs, transcripts, and response footers with targeted `openrouter/auto` coverage.

## Reproduction Assessment

Yes, at source level: current main reconciles some OpenRouter billed cost into assistant message usage, but agent metadata, `session_status`, and the reply footer still lack resolved-model and provider-reported cost fields. I did not run a live OpenRouter call in this read-only review.

## Solution Assessment

No, a footer-only OpenRouter patch is not the best complete solution. The safer path is a generic provider metadata contract that keeps selected model, resolved model, local estimates, and provider-reported charges distinct.
