## Summary

Keep open: current main still stores the selected `openrouter/auto` model and local estimated-cost path, but it does not preserve OpenRouter's returned routed model or provider-reported cost for agent, session, or response-footer use.

## What This Changes

Expose the actual OpenRouter model selected for `openrouter/auto` and its returned cost so agents can report the resolved model and cost in response usage surfaces.

## Best Possible Solution

Define additive selected-versus-resolved model and provider-reported-cost metadata at the OpenAI-compatible/OpenRouter response boundary, then surface it consistently through assistant/session records, response footers, transcripts, status/usage tools, and focused OpenRouter auto tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `openrouter/auto` is stored as `model.id`, streamed `chunk.model` is not captured, OpenRouter cost fields are ignored, and the response footer has no resolved-model or provider-cost inputs.

## Solution Assessment

Unclear until maintainers choose the API and UX contract. The safest direction is additive metadata that preserves selected model, resolved model, local estimate, and provider-reported cost separately instead of overwriting existing fields or adding only a footer special case.
