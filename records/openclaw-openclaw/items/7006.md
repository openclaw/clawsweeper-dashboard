## Summary

Keep open: current main still does not expose OpenRouter's routed `openrouter/auto` model or provider-reported cost through the OpenAI-compatible transport, agent metadata, session/status state, or response usage footer. The related open items cover adjacent resolved-model and cost work, but none is a merged replacement for this specific OpenRouter-auto transparency request.

## What This Changes

The issue requests exposing the concrete model and cost returned when OpenRouter routes `openrouter/auto`, so agents and response footers can show the resolved model and usage cost.

## Best Possible Solution

Define additive selected-versus-resolved model and provider-reported-cost metadata at the OpenAI-compatible/OpenRouter boundary, preserve local estimates separately, and surface it consistently through assistant/session records, response footers, status/usage tools, transcripts, and focused `openrouter/auto` tests.

## Reproduction Assessment

Yes, source-level. Current main still stores the configured `openrouter/auto` model id, parses normalized token usage plus local estimated cost, and formats response usage footers without resolved-model or provider-reported-cost inputs.

## Solution Assessment

Unclear until maintainers choose the public metadata and UX contract. The safest direction is additive metadata that preserves selected model, resolved model, local estimate, and provider-reported cost separately instead of overwriting existing fields or adding only a footer special case.
