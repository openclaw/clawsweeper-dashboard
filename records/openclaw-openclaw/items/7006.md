## Summary

Keep open: current main now captures the concrete model returned by an OpenAI-compatible stream, but it still does not expose that routed model to the agent/session/footer path or preserve OpenRouter’s provider-reported billed cost; the related open issues divide the broader model and cost work without replacing this footer-focused outcome.

## What This Changes

Expose the concrete model selected by `openrouter/auto` and its provider-reported cost to the agent, including a resolved-model label and cost in response usage footers.

## Best Possible Solution

Define one generic selected-versus-resolved model and authoritative-versus-estimated cost contract, propagate it from assistant messages into run/session metadata, and let the configured usage footer, status, transcript, and agent-facing surfaces render that canonical data consistently.

## Reproduction Assessment

Yes at source level: current main captures a concrete `responseModel`, while the response footer accepts neither that field nor provider-reported billed cost. A live OpenRouter call was not performed during this read-only review.

## Solution Assessment

No, a footer-only OpenRouter special case is not the best solution. The footer should consume a generic canonical metadata path shared with agent, transcript, session, and status surfaces, with explicit semantics for billed versus estimated cost.
