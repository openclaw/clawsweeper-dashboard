## Summary

Keep open: current main has adjacent OpenRouter generation-cost reconciliation and a richer usage-footer/template contract, but it still does not expose OpenRouter's resolved `openrouter/auto` response model or source-aware provider cost details through agent/session/footer metadata.

## What This Changes

The issue requests exposing the resolved OpenRouter `openrouter/auto` backend model and provider-reported cost to agents and response footers.

## Best Possible Solution

Define a provider-neutral selected-versus-resolved model and source-aware cost metadata contract, then capture OpenRouter response `model` plus provider cost details at the OpenAI-compatible transport/OpenRouter wrapper boundary and thread them through agent metadata, session/status projections, transcripts, and response usage rendering.

## Reproduction Assessment

Yes, at source level. Current main still initializes OpenAI-compatible assistant messages from the configured model, parses only token-oriented usage plus local cost, and formats response footers without OpenRouter response-model or cost-detail metadata.

## Solution Assessment

Unclear pending maintainer product/API direction. Capturing OpenRouter response fields is the right technical source, but the best durable fix needs one selected-versus-resolved model and cost-source contract across runtime/session/footer surfaces instead of a one-off footer suffix.
