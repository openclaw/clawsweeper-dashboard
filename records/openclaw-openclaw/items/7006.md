## Summary

Keep open: current main still treats `openrouter/auto` as the selected catalog model, does not capture OpenRouter's returned resolved model or raw provider cost fields in the OpenAI-compatible stream path, and formats usage footers from local estimates only. The related open items are useful context, but none fully replaces this combined resolved-model plus cost transparency request.

## What This Changes

Expose the concrete model selected by `openrouter/auto` and OpenRouter provider-reported usage cost so agent/runtime and response footer surfaces can report them accurately.

## Best Possible Solution

Define an additive selected-versus-resolved model and provider-reported cost contract, capture OpenRouter response `model` and raw usage cost fields at the OpenAI-compatible/OpenRouter boundary, and thread them through transcripts, session/runtime metadata, status/usage tools, response footers, docs, and targeted tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main does not expose OpenRouter returned model metadata or provider-reported cost on the requested runtime/footer surfaces.

## Solution Assessment

Unclear until maintainers choose the public metadata and cost-authority contract. The safer direction is additive resolved-model and provider-cost metadata, not overloading existing estimated USD fields or applying a footer-only patch.
