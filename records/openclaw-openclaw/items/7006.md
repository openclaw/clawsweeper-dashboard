## Summary

Keep open: current main still keeps `openrouter/auto` as the selected catalog model, ignores OpenRouter returned model and provider-reported cost fields in the shared OpenAI-compatible transport, and renders response usage footers from local token estimates only. Related open items cover adjacent model or cost observability, but none is a merged/current replacement for this combined resolved-model plus provider-cost transparency request.

## What This Changes

Expose the actual model resolved by `openrouter/auto` and its provider-reported usage cost to agents and response footers.

## Best Possible Solution

Define an additive resolved-model and provider-reported cost contract, capture OpenRouter response `model` and raw usage cost fields at the OpenAI-compatible/OpenRouter boundary, and thread them through agent metadata, transcripts, session/status surfaces, response footers, docs, and targeted tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug reproduction because this is a feature request; source inspection on current main gives high-confidence proof that the requested OpenRouter returned model and provider-cost metadata are not exposed on the runtime and footer surfaces.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the public metadata and cost-authority contract; the safer direction is additive resolved-model and provider-cost metadata, not overloading existing estimated USD fields or applying a footer-only patch.
