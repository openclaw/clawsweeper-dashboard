## Summary

Keep open. Current main still keeps `openrouter/auto` as the selected model, drops the OpenRouter-returned routed model from the chat-completions stream, computes local estimated cost, and formats response usage footers without a resolved-model label or provider-reported cost.

## What This Changes

The issue requests exposing the model OpenRouter actually selected for `openrouter/auto` and the provider-reported usage cost to agents and response footers.

## Best Possible Solution

Define an additive selected-versus-resolved OpenRouter metadata contract, capture `response.model` plus provider-reported usage cost at the OpenAI-compatible transport boundary, and thread it through assistant/session records, agent-visible metadata, response footers, status/usage surfaces, transcripts, and focused OpenRouter regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the feature gap: current main never carries OpenRouter `response.model` or `usage.cost` from the shared transport into agent/session/footer metadata. I did not run a live OpenRouter turn in this read-only review.

## Solution Assessment

Yes on the requested direction, but the exact API/UX shape needs maintainer choice. A footer-only patch is not sufficient unless it also captures provider-returned model/cost metadata and preserves selected-versus-resolved state.
