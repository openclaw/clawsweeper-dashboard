## Summary

Keep open: current main and the latest release still do not expose OpenRouter's routed `openrouter/auto` model or provider-reported cost through the OpenAI-compatible transport, agent metadata, session/status state, or response usage footer. The related open issues cover adjacent cost and resolved-backend metadata work, but no merged fix or single canonical item fully replaces this OpenRouter-auto transparency request.

## What This Changes

The issue requests exposing the actual model selected by OpenRouter for `openrouter/auto` plus the provider-reported usage cost so agents and response footers can show the resolved model and cost.

## Best Possible Solution

Define additive selected-versus-resolved model and provider-reported-cost metadata at the OpenAI-compatible/OpenRouter boundary, preserve local estimates separately, and surface the data consistently through assistant/session records, response footers, status/usage tools, transcripts, and targeted `openrouter/auto` tests.

## Reproduction Assessment

Yes, source-level. Current main still stores the configured `openrouter/auto` model id, parses only normalized token usage plus local cost, and formats response usage footers without resolved-model or provider-reported-cost inputs.

## Solution Assessment

Unclear until maintainers choose the public metadata and UX contract. The safest direction is additive metadata that preserves selected model, resolved model, local estimate, and provider-reported cost separately instead of overwriting existing fields or adding only a footer special case.
