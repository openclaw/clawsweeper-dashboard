## Summary

Keep open: current main still preserves the selected `openrouter/auto` catalog model, does not capture OpenRouter's returned model or provider-reported cost fields in the shared OpenAI-compatible transport, and renders response footers from local token/cost estimates only. Related open items cover adjacent model visibility and cost accounting, but none is a merged replacement for this combined OpenRouter auto transparency request.

## What This Changes

Expose the actual model selected by OpenRouter for `openrouter/auto` and its provider-reported cost to agents and response footers.

## Best Possible Solution

Define an additive resolved-model and provider-reported cost contract, capture OpenRouter response `model` plus raw usage cost fields at the OpenAI-compatible/OpenRouter boundary, and thread them through agent metadata, transcripts, session/status surfaces, response footers, docs, and targeted tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug reproduction because this is a feature request; source inspection on current main gives high-confidence proof that the requested OpenRouter returned model and provider-cost metadata are not exposed on runtime and footer surfaces.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the public metadata and cost-authority contract; the safer direction is additive selected-versus-resolved model and provider-cost metadata, not overloading existing estimated-cost fields or applying a footer-only patch.
