## Summary

Keep open: current main still treats `openrouter/auto` as the selected model and local estimated-cost path, without carrying OpenRouter's routed response model or provider-reported cost into agent/session/status/footer surfaces. Related open issues cover broader resolved-model and cost work, but none is a merged replacement for this specific OpenRouter auto transparency request.

## What This Changes

Requests exposing the OpenRouter auto routed model name and provider-reported cost to agents and response footers.

## Best Possible Solution

Define additive selected-versus-resolved model and provider-reported-cost metadata at the OpenAI-compatible/OpenRouter boundary, preserve local estimates separately, and surface the data consistently through assistant/session records, response footers, status/usage tools, transcripts, and focused OpenRouter auto tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `openrouter/auto` is stored as `model.id`, streamed provider `model` is not captured, OpenRouter cost fields are ignored, and the response footer has no resolved-model or provider-cost inputs.

## Solution Assessment

Unclear until maintainers choose the public metadata and UX contract. The safest direction is additive metadata that preserves selected model, resolved model, local estimate, and provider-reported cost separately instead of overwriting existing fields or adding only a footer special case.
