## Summary

Keep open: current main still misses the reported Feishu fallback shape and this PR remains the paired fix candidate, but the head has a blocking parser control-flow regression and lacks inspectable real Feishu proof.

## What This Changes

The PR updates the Feishu message parser and tests to extract title, text, plain-text, and link content from interactive-card post-format fallback payloads with nested `elements` arrays.

## Best Possible Solution

Revise and land a narrow Feishu-plugin parser/test fix that handles top-level nested fallback `elements` and title/link text while preserving existing empty-array fallthrough across candidate element arrays.

## Reproduction Assessment

Yes. A mocked `getMessageFeishu()` response with `msg_type: "interactive"` and `body.content` set to `{"title":"...","elements":[[{"tag":"text","text":"..."}]]}` follows current main to `[Interactive Card]` because arrays are ignored and post fallback requires `content` or `post`; I verified this by source inspection, not by running tests.

## Solution Assessment

No, not as currently patched. The narrow parser direction is right, but the branch should collect usable element text across later candidate arrays before falling back to a title-only result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve later element-array fallthrough:** `extensions/feishu/src/send.ts:335-337`
  - body: `readInteractiveElementArrays()` checks `parsed.elements` before `body.elements` and `i18n_elements`. Because this loop joins `titleTexts` into every iteration, a card with a header/top-level title plus an empty first `elements` array returns the title here and never reaches later nonempty arrays, regressing the current empty-array fallback behavior. Collect nonempty element text across candidates before combining the title, or only return title-only after all arrays are exhausted.
  - confidence: 0.87
