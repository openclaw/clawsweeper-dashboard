## Summary

Keep open: current main and the latest release still use pi-tui 0.78.0, and TUI streaming updates still repaint assistant markdown through that renderer without a TUI-specific viewport anchoring or auto-follow policy. A member explicitly clarified that the merged WebChat auto-scroll work does not cover TUI, and the open pi-tui scrollback-clearing report only partially covers this broader TUI viewport issue.

## What This Changes

The issue requests stable TUI chat viewport behavior so new input or output does not cause disruptive scroll jumps, possibly via anchoring or an auto-scroll control.

## Best Possible Solution

Keep this as the TUI tracking issue and resolve it with a maintainer-approved viewport anchoring or auto-follow policy, backed by PTY proof and coordinated with the pi-tui scrollback-clearing work in https://github.com/openclaw/openclaw/issues/78017.

## Reproduction Assessment

No high-confidence live current-main reproduction was established in this read-only review. Source inspection shows the current TUI streaming path still repaints through pi-tui, and the pinned renderer still has a content-driven full-redraw path that can clear scrollback.

## Solution Assessment

Unclear. The implicated surface is the TUI renderer and viewport path, but maintainers still need to choose whether the end state is anchoring repair, an explicit TUI auto-follow preference, a pi-tui dependency fix or bump, or a combination.
