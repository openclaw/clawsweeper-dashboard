## Summary

Current main still uses a renderer path that can full-redraw when streamed content changes above the viewport, and no merged TUI anchoring fix supersedes this report; the separate WebChat setting and narrower scrollback-loss issue do not resolve the broader TUI viewport behavior.

## What This Changes

Preserve a stable, predictable TUI transcript viewport while messages are submitted and assistant output streams.

## Best Possible Solution

Define one default invariant—follow new output only while the user is at the live tail, otherwise preserve the reading position—then fix it at the owning layer and keep destructive scrollback clearing tracked by the narrower related issue.

## Reproduction Assessment

Yes, at source level: current main repeatedly updates streaming Markdown while pi-tui 0.78.0 full-renders when content changes above the viewport. A live terminal capture is still needed to distinguish the exact jump variant and terminal-specific behavior.

## Solution Assessment

No, the issue’s menu of possible toggles, paging, and anchoring changes is broader than the best fix. First correct the default follow-versus-reading invariant at the proper OpenClaw or pi-tui owner boundary; consider an optional mode only as a separate product decision.
