## Summary

Keep open. Current main still does not have a TUI transcript viewport or scroll policy: ChatLog is an append/prune container, the root layout mounts it directly, streaming/final events update it without checking user scroll position, and history reloads rebuild it. The related TUI scrolling PR is still open and unmerged, so it is an implementation candidate rather than proof that this report is fixed.

## What This Changes

Request TUI behavior that preserves a stable transcript viewport during sends and streaming, with controlled auto-follow, scroll anchoring, and a jump-to-latest or auto-scroll control.

## Best Possible Solution

Add a TUI-owned transcript viewport around ChatLog that tracks user position, preserves anchors across renders/history reloads, auto-follows only when already at latest, and exposes keyboard or mouse scrolling plus a clear jump-to-latest affordance. Review #51128 first and either land it as part of that path or supersede it with a broader TUI transcript fix, keeping the separate Control UI/WebChat request in #7648 scoped to the web surface.

## Reproduction Assessment

Unclear: the report gives a plausible live path with `openclaw tui` on Linux during repeated sends and streaming, but this read-only sweep did not run an interactive terminal reproduction. Static inspection does give high-confidence evidence that the requested viewport, anchor, and scroll controls are missing on current main.

## Solution Assessment

Yes for keeping this open: the narrow maintainable solution belongs in the TUI ChatLog/root input/event path, not in the web chat scroll code. The current main implementation is not the best end state because it renders the transcript as an append-only container without user-scroll policy.
