## Summary

Keep open. The wrapper-forwarding part is still useful, but the PR as written broadens Discord gateway execution to every message action while current main’s adapter and tests narrow gateway mode to read/search only; current main also still lacks the registered wrapper forwarding needed for CLI dispatch.

## What This Changes

The branch adds a Discord message-action execution-mode resolver and forwards that resolver through the registered Discord channel wrapper.

## Best Possible Solution

Rebase or replace the branch so the registered Discord wrapper forwards the existing read/search-only resolver, add focused wrapper/runner coverage, and keep non-read/search Discord actions on local execution.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main’s registered Discord wrapper lacks `resolveExecutionMode`, and the shared runner defaults a missing resolver to local execution. The PR regression is also source-level reproducible because its resolver is unconditional while current tests require non-read/search actions to stay local.

## Solution Assessment

No, not as-is. Forwarding the wrapper resolver is the narrow maintainable fix, but replacing the adapter behavior with unconditional gateway execution is broader than the reported read/search hang and conflicts with the current resolver contract.
