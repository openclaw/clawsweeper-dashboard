## Summary

Keep open: current main is not fixed for the shared outbound metadata leak, even though the direct WhatsApp pairing-template path itself is clean. The visible `Sender (untrusted metadata):` block matches the unresolved outbound echo-sanitization bug already tracked by https://github.com/openclaw/openclaw/issues/39847 and https://github.com/openclaw/openclaw/pull/50520.

## What This Changes

The issue reports that WhatsApp `dmPolicy: pairing` replies can expose internal `Sender (untrusted metadata)` JSON to the external WhatsApp user.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/50520 so leading inbound metadata is stripped at the shared outbound payload boundary, then add/keep WhatsApp-visible regression proof for this symptom.

## Reproduction Assessment

Yes for the shared leak: source inspection shows reply payload text beginning with inbound metadata reaches `parseReplyDirectives` unstripped in `normalizeReplyPayloadsForDelivery`. The exact direct WhatsApp pairing auto-reply path appears clean, so live WhatsApp proof would still help confirm the reported surface.

## Solution Assessment

Yes. The narrow maintainable fix is to strip leading inbound metadata once in the shared outbound payload planner before directive parsing; per-channel WhatsApp-only filtering would leave the same leak open on other delivery paths.
