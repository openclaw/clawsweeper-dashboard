## Summary

Keep open. Current main still auto-recovers restart-aborted subagent sessions solely from `abortedLastRun` plus the rapid re-wedge gate, with no task-age cutoff or `agents.defaults.taskRecovery.maxAgeMinutes` config, and existing regression coverage still expects a three-hour stale aborted run to be recovered.

## What This Changes

The issue requests an age-bound gateway recovery policy so old interrupted CLI/subagent work is marked `lost` instead of being re-dispatched after restart.

## Best Possible Solution

Add a configurable stale-recovery cutoff in subagent orphan recovery and task/session reconciliation, mark skipped stale work `lost` without replaying the prompt, and update docs, schema, changelog, and regression coverage.

## Reproduction Assessment

Yes. Persist an unended subagent run older than the intended cutoff with a child session entry marked `abortedLastRun: true`, restart the registry/gateway, and current main will attempt recovery; the existing three-hour persistence test asserts that path.

## Solution Assessment

Yes. An age check at the orphan-recovery decision point, paired with terminalizing the matching run/task state and config/docs/tests, is the narrowest maintainable fix; filtering task restore alone would leave synthetic resume behavior intact.
