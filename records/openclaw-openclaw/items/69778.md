- `impact:session-state`: The issue is rooted in persisted task/session recovery state causing stale subagent context to resume.
- `impact:message-loss`: The discussion includes stale recovered work sending an unexpected WhatsApp message after restart.

## Summary

Keep open. Current main and the latest release still let restart-aborted subagent runs bypass the stale-run age check and recover old prompts, and the live discussion includes both the original 24-hour replay and a second 16-day WhatsApp replay report.

## What This Changes

Request to bound gateway restart recovery for interrupted CLI/subagent tasks by age and mark stale work lost instead of auto-executing old prompts.

## Best Possible Solution

Define a stale-age policy for restart-aborted subagent recovery, terminalize too-old work as lost before synthetic resume, and align config, docs, and regression tests with that guarantee.

## Reproduction Assessment

Yes, at source level. Seed a restored subagent run older than the intended cutoff with a child session entry marked `abortedLastRun: true`; current code and tests keep it eligible for restart recovery instead of marking it lost.

## Solution Assessment

Unclear until maintainers choose the recovery policy. The narrow technical fix point is an age gate before synthetic resume, but the cutoff, config surface, and side-effectful-tool behavior need owner judgment.
