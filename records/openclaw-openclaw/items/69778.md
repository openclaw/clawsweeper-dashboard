## Summary

Keep open. Current main and v2026.5.18 still let restart-aborted subagent sessions with `abortedLastRun: true` reach synthetic resume without a stale-age cutoff; the existing guard only bounds rapid re-wedges, and the issue discussion includes a second live report of a 16-day-old stale WhatsApp send after restart.

## What This Changes

The issue requests a bounded gateway-restart recovery policy so stale interrupted CLI/subagent work is marked `lost` instead of auto-executing old prompts.

## Best Possible Solution

Define a restart-aborted recovery freshness policy, terminalize too-old work as `lost` before synthetic resume, and align config, docs, and regression coverage with that guarantee.

## Reproduction Assessment

Yes, at source level. Seed an active restored subagent run older than the intended cutoff with a child session entry marked `abortedLastRun: true`; current code and tests keep it eligible for restart recovery instead of marking it lost.

## Solution Assessment

Unclear until maintainers choose the recovery policy. The narrow technical fix point is an age gate before synthetic resume, but the cutoff, config surface, and side-effectful-tool behavior need owner judgment.
