## Summary

Needs changes before merge: the duplicate progress-line bug and supplied proof are credible, but the current patch gates `typingSignals.signalToolStart()` along with `onToolStart`, so tool update events would no longer refresh typing keepalive for long-running tools.

## What This Changes

The PR changes tool event handling so `onToolStart` fires only for `phase: "start"` and adds a unit test for a start/update/update/end sequence.

## Best Possible Solution

Keep channel progress `onToolStart` start-only while preserving tool update typing keepalive, with regression coverage for both callback counts.

## Reproduction Assessment

Yes. Source inspection shows update events are emitted without args and forwarded to `onToolStart`, where shared channel progress formatting renders args-less tool events as bare labels; the PR body also includes redacted live before/after Telegram and Discord output.

## Solution Assessment

No, not as written. The progress callback should be start-only, but the update branch should still refresh typing TTL instead of gating away both behaviors together.
