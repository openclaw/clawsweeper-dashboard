## Summary

Keep open: the PR targets a real duplicate progress-draft bug and the main fix direction is right, but the patch also stops tool update events from refreshing the typing keepalive because `typingSignals.signalToolStart()` is inside the same gated branch.

## What This Changes

The PR gates tool `onToolStart` callbacks to `phase === "start"` and adds a regression test for a start/update/update/end tool-event sequence.

## Best Possible Solution

Split the handling so update phases still refresh typing keepalive while only start phases call channel progress `onToolStart`, with the focused regression test extended to cover both behaviors.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main forwards args-less tool update events to `onToolStart`, and shared draft formatting renders those as bare tool lines that adjacent-only dedupe cannot collapse when interleaved.

## Solution Assessment

No, not as written. The progress callback should be start-only, but the typing keepalive side effect should still run for update phases instead of being gated away with the duplicate-line fix.
