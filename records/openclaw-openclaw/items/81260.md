## Summary

Keep open: the duplicate progress-line fix is valid and the supplied live proof is enough, but the patch also stops tool-update events from refreshing typing keepalive because `signalToolStart()` remains under the same gated condition.

## What This Changes

The PR gates tool-stream `onToolStart` callbacks to `phase: "start"` and adds a focused unit test for a start/update/update/end tool event sequence.

## Best Possible Solution

Split tool progress from typing: call `onToolStart` only for start phases, keep typing refreshes on start and update phases, and cover both callback counts in the focused runner test.

## Reproduction Assessment

Yes. Source inspection shows args-less tool update events are forwarded into shared progress formatting, and the PR body includes redacted live before/after Telegram and Discord output showing the duplicate-line symptom and the intended result.

## Solution Assessment

No, not as written. The start-only progress callback is the right direction, but the condition should not also remove update-phase typing TTL refreshes.
