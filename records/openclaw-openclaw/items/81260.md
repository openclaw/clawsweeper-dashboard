## Summary

Keep open with changes requested: the reported duplicate progress line bug is real and the supplied live proof is adequate, but the patch also stops update-phase tool events from refreshing the typing TTL. A narrow split can keep `onToolStart` start-only while preserving typing keepalive on start and update phases.

## What This Changes

The PR gates tool-stream `onToolStart` callbacks to `phase=start` and adds a runner unit test for a start/update/update/end tool event sequence.

## Best Possible Solution

Split tool progress from typing: call `onToolStart` only for tool start phases, keep `signalToolStart()` on start and update phases, and leave item/command update streams unchanged.

## Reproduction Assessment

Yes. Source inspection shows args-less tool update events are forwarded into shared progress formatting, and the PR body includes redacted live before/after Telegram and Discord output showing the duplicate-line symptom and intended result.

## Solution Assessment

No, not as written. The start-only progress callback is the right direction, but the patch should not also remove update-phase typing TTL refreshes.
