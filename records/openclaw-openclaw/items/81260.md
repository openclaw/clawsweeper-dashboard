## Summary

Keep open: the duplicate progress-line bug is real and the PR direction is right, but the current diff also stops update-phase typing TTL refreshes for long-running tools.

## What This Changes

The PR gates the shared tool-start progress callback to tool-stream start phases and adds a runner unit test for start/update/update/end tool events.

## Best Possible Solution

Split tool progress from typing: call `onToolStart` only for start phases, keep `signalToolStart()` on start and update phases, and extend the runner test to assert both callback counts.

## Reproduction Assessment

Yes. Source inspection shows args-less tool update events are forwarded into shared progress formatting on current main, and the PR body includes redacted live Telegram/Discord before-and-after output plus screenshots.

## Solution Assessment

No, not as written. The start-only progress callback is the narrow fix for duplicate lines, but typing refresh must remain on update phases instead of being gated by the same condition.
