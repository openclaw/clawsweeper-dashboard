## Summary

Keep this PR open: the duplicate progress-line bug is real and the supplied Telegram/Discord proof is useful, but the patch is not merge-ready because it gates typing refreshes with visible progress filtering, misses the queued follow-up bridge on current main, and is now conflicting.

## What This Changes

The branch changes `runAgentTurnWithFallback` so tool `onToolStart` callbacks fire only for `phase: "start"`, with a regression test for a start/update/update/end sequence.

## Best Possible Solution

Land a rebased fix that suppresses only duplicate visible progress rows while preserving update-phase typing refreshes and applying the same policy to both main and queued follow-up paths.

## Reproduction Assessment

Yes. Source inspection shows start events include args while update events do not, and current main forwards both phases into the progress callback; the PR body also supplies before/after live chat proof.

## Solution Assessment

No, not as-is. The narrow maintainable fix should separate typing keepalive from visible progress rendering, refresh the branch, and cover the follow-up bridge; id-keyed merging is the safer alternative if maintainers want to preserve update frames.
