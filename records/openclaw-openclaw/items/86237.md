## Summary

Keep open: current main and the latest checked release still expose the scheduler as `cron`, so the requested rename is not implemented. The request is actionable in core, but it is a broad public API/config/protocol migration that needs maintainer product and compatibility decisions before code work.

## What This Changes

Request renaming OpenClaw's public and internal `cron` scheduler surfaces to `scheduler` with a compatibility alias, deprecation warning, and lossless state migration.

## Best Possible Solution

Decide the canonical naming and compatibility contract first, then implement an additive `scheduler` surface with tested `cron` aliases, config/state migration, docs updates, and explicit upgrade behavior across CLI, Gateway methods, tools, plugin SDK, and session/state handling.

## Reproduction Assessment

Not applicable as a rename/product request. Source inspection does confirm the reported current state: the CLI, agent tool, Gateway methods, docs, state path, and session-key format still use `cron`, and `cron.list` defaults to the requester agent unless an agentId is explicit.

## Solution Assessment

Unclear until maintainers choose the naming and compatibility policy. If the rename is accepted, the proposed alias-first migration shape is the maintainable direction, but it is too broad and compatibility-sensitive for an autonomous cleanup patch.
