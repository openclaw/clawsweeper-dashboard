## Summary

Keep open: current main and the latest release still expose the scheduler as `cron`, so the requested rename is not implemented. The request is actionable in core, but it is a broad public API/config/protocol/docs/state migration that needs maintainer product and compatibility direction before code work.

## What This Changes

The issue requests renaming OpenClaw's public and internal `cron` scheduler surfaces to a less ambiguous name such as `scheduler`, including CLI, tool, config, state, session-key, and docs migration.

## Best Possible Solution

Choose the canonical name and compatibility contract first, then implement an additive `scheduler` surface with tested `cron` aliases, doctor-backed migration, docs updates, and explicit upgrade behavior across CLI, Gateway methods, tools, plugin SDK, and session/state handling.

## Reproduction Assessment

Yes for the current naming surface: source and docs still expose the subsystem as `cron` across CLI, tool, config, state path, protocol, and docs. The reported confusion incident is discussion evidence rather than a local executable bug repro.

## Solution Assessment

Unclear until maintainers choose the naming and compatibility policy. If accepted, the alias-first migration shape discussed in comments is the maintainable path, but it is too broad and compatibility-sensitive for an autonomous cleanup patch.
