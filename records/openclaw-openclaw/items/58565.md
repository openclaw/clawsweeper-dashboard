## Summary

Keep open. Current main still treats ordinary 401/403 invalid-key shapes as generic auth and persists them through cooldown state, while only high-confidence revoked/deactivated auth enters the disabled lane; the merged 429/529 rotation work is adjacent but does not settle the dead-key credential policy or Retry-After persistence.

## What This Changes

The issue requests error-specific auth profile health handling so invalid credentials are disabled while transient 429/529 provider failures use retry-aware or short cooldowns.

## Best Possible Solution

Define a maintainer-approved auth-health policy for high-confidence dead credentials, then align classifier coverage, usage persistence, doctor/status guidance, logs, tests, and docs while preserving the existing 429/529 rotation-cap behavior unless owners choose otherwise.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a 401 invalid_api_key resolves to auth, markAuthProfileFailure records auth through cooldownUntil, and tests explicitly preserve that behavior; I did not run a live invalid-key gateway request.

## Solution Assessment

Unclear. The requested direction is valid, but disabling all 401s until operator action may be too broad; the safer maintainable path is a high-confidence dead-key classifier with explicit recovery semantics and Retry-After plumbing for rate limits.
