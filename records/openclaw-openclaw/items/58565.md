## Summary

Keep open. Current main still sends ordinary 401/403 invalid-key shapes through the generic auth cooldown lane, while merged adjacent work only covers narrow auth_permanent cases plus 429/529 rotation caps; changing dead-key disable and recovery semantics needs owner policy review.

## What This Changes

Request differentiated auth-profile failover so dead 401 credentials are skipped or disabled, 429 uses Retry-After-aware cooldowns, 529 fails over quickly, and 402 keeps extended billing backoff.

## Best Possible Solution

Define an owner-approved credential-health policy for high-confidence dead credentials and Retry-After propagation, then align classifier coverage, persistence, doctor/models-auth guidance, docs, and regression tests while preserving transient-auth safeguards.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: 401 invalid_api_key classifies as auth, the runner records that reason via markAuthProfileFailure, and usage persistence stores it in cooldownUntil; I did not run a live invalid-key gateway request.

## Solution Assessment

Unclear. The direction is useful, but disabling all 401 or invalid_api_key responses until operator action is broader than the recent auth_permanent policy; the safer path is a narrow dead-credential classifier plus explicit recovery and Retry-After semantics.
