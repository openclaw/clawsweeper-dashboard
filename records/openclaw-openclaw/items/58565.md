## Summary

Keep open. Current main still matches the reported gap: ordinary 401/403 responses such as `invalid_api_key` remain generic `auth`, and that lane persists as `cooldownUntil` rather than a disabled credential state. The related merged auth_permanent backoff work is adjacent, but it does not implement the requested dead-key policy or wire provider Retry-After timing into auth-profile cooldown persistence.

## What This Changes

The issue requests auth-profile failover to treat dead credentials differently from transient 429/529 responses so invalid primary profiles are skipped or disabled instead of repeatedly cooling down.

## Best Possible Solution

Define a conservative auth-health policy that promotes only high-confidence dead credentials out of transient cooldown, preserves retryable handling for ambiguous provider incidents, threads 429 retry timing where available, keeps 529 overloads short and rotation-friendly, and aligns classifier coverage, persistence, doctor/status guidance, logs, tests, and docs.

## Reproduction Assessment

Yes. The issue’s invalid-primary/valid-secondary scenario is reproducible at the code level by resolving a 401 `invalid_api_key` to `auth`, recording that failure through `markAuthProfileFailure`, and observing `cooldownUntil` rather than `disabledUntil`; current tests already encode the classifier behavior.

## Solution Assessment

Unclear as written. The requested direction is valid, but disabling all 401s until explicit operator action may be too broad; the safer path is a maintainer-approved high-confidence dead-key classifier with explicit recovery semantics.
