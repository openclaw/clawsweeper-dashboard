## Summary

Keep open. Current main still does not implement the requested dead-key policy: ordinary 401/403 provider responses, including `invalid_api_key`, remain in the generic `auth` lane, and generic `auth` is persisted as a cooldown rather than an operator-disabled credential state. The adjacent merged #60404 work shortened `auth_permanent` backoff, but it did not make ordinary dead-key 401s persistently disabled or connect provider `Retry-After` values to auth-profile cooldown timing.

## What This Changes

The issue requests auth-profile failover to distinguish invalid or expired credentials from transient rate limits, overloads, and billing failures so dead keys are skipped immediately instead of repeatedly entering cooldown.

## Best Possible Solution

Define a conservative auth-health policy that promotes only high-confidence dead credentials out of transient cooldown, preserves retryable handling for ambiguous provider incidents, wires 429 retry timing where the provider exposes it, keeps 529 overloads short and rotation-friendly, and aligns classifier coverage, persistence, doctor/status guidance, logs, and docs.

## Reproduction Assessment

Yes. The issue’s invalid-primary/valid-secondary scenario is reproducible at the code level by resolving a 401 `invalid_api_key` to `auth`, recording that failure through `markAuthProfileFailure`, and observing `cooldownUntil` rather than `disabledUntil`; current tests already encode the first half of that path.

## Solution Assessment

Unclear as written. The requested direction is right, but disabling all 401s until explicit operator action may be too broad; the safer solution is a maintainer-approved high-confidence dead-key classifier plus clear re-enable and status semantics.
