## Summary

Keep open. Current main still treats ordinary `401 invalid_api_key` as generic auth cooldown rather than a dead-credential disable, and the remaining behavior is a credential-health policy decision rather than cleanup closure.

## What This Changes

The issue asks for auth-profile failover to distinguish dead credentials from transient rate-limit, overload, and billing failures.

## Best Possible Solution

Define an owner-approved credential-health policy for high-confidence dead credentials and Retry-After propagation, then align classifier coverage, persistence, doctor/models-auth guidance, docs, and regression tests.

## Reproduction Assessment

Yes, source-level. Current main classifies `401 invalid_api_key` as generic `auth`, records that failure, and persists it in `cooldownUntil`; I did not run a live invalid-key gateway request.

## Solution Assessment

Unclear. The direction is useful, but permanently disabling all 401 or `invalid_api_key` responses is broader than the current high-confidence `auth_permanent` policy; a narrower dead-credential classifier plus explicit recovery and Retry-After semantics is safer.
