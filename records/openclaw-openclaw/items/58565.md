## Summary

Keep open. Current main and v2026.5.22 still classify ordinary `401 invalid_api_key` as generic `auth` and persist it through the short cooldown lane; the remaining dead-credential behavior needs a maintainer policy decision rather than cleanup closure.

## What This Changes

The issue requests error-specific auth-profile failover so dead credentials are skipped differently from transient provider throttling or overload failures.

## Best Possible Solution

Define an owner-approved credential-health policy for high-confidence dead credentials and Retry-After propagation, then align classifier coverage, persistence, doctor/models-auth guidance, docs, and regression tests.

## Reproduction Assessment

Yes, source-level. Current main classifies `401 invalid_api_key` as generic `auth`, then `markAuthProfileFailure` persists generic auth through `cooldownUntil`; I did not run a live invalid-key gateway request.

## Solution Assessment

Unclear. The direction is useful, but permanently disabling broad 401 or `invalid_api_key` responses is wider than the current high-confidence `auth_permanent` policy; maintainers should choose the dead-credential and recovery boundary first.
