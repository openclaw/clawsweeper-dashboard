## Summary

Keep open. Current main still routes ordinary `401 invalid_api_key` payloads through the generic auth cooldown lane, while adjacent merged work only covers high-confidence `auth_permanent` patterns and 429/529 rotation/backoff policy.

## What This Changes

The issue requests auth-profile failover semantics that treat dead 401 credentials differently from transient 429/529 provider conditions for scheduler and HTTP API jobs.

## Best Possible Solution

Define an owner-approved credential-health policy for high-confidence dead credentials and Retry-After propagation, then align classifier coverage, persistence, doctor/models-auth guidance, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `401 invalid_api_key` classifies as generic `auth`, the runner records that reason, and usage persistence stores it in `cooldownUntil`; I did not run a live invalid-key gateway request.

## Solution Assessment

Unclear. The direction is useful, but permanently disabling all 401 or `invalid_api_key` responses is broader than the current high-confidence `auth_permanent` policy; a narrower dead-credential classifier plus explicit recovery and Retry-After semantics is safer.
