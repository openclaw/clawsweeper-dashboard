## Summary

Keep open: current main still routes ordinary 401/403 `invalid_api_key` cases to generic `auth` cooldowns, while the related merged work only shortened `auth_permanent` lockouts. The requested dead-key disable policy and Retry-After-backed rate-limit persistence remain unresolved and need maintainer/secops judgment.

## What This Changes

Auth profile failover should classify dead credentials separately from transient rate-limit or overload failures so requests skip unusable keys faster.

## Best Possible Solution

Define a maintainer/secops-approved auth-health policy that promotes only high-confidence dead credentials to disabled state, preserves retryable handling for ambiguous provider incidents, threads Retry-After into rate-limit cooldowns, keeps overloads rotation-friendly, and updates classifier coverage, persistence, doctor/status guidance, logs, tests, and docs.

## Reproduction Assessment

Yes, source-reproducible: a 401 `invalid_api_key` is still classified as `auth`, and `markAuthProfileFailure` persists `auth` as `cooldownUntil` rather than `disabledUntil`. I did not run tests because this review is read-only, but current tests also pin that classifier behavior.

## Solution Assessment

Unclear as written. Differentiating high-confidence dead keys is the right direction, but disabling all 401/403 or `invalid_api_key` responses until manual intervention is too broad without an approved classifier and recovery policy.
