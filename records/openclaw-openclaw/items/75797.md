## Summary

Keep open. Current main still routes configured Google/Gemini web-search secrets through the bundled public artifact loader, and that loader still rejects hardlinked bundled artifacts with the same wrapped error reported here; the latest shipped release I could inspect has the same public-surface behavior and no linked closing PR fixes this path.

## What This Changes

The issue asks OpenClaw to avoid a Google/Gemini startup crash loop when the secrets reloader loads the bundled Google web-search contract from a hardlink-based global install, and to preserve useful diagnostics/recovery for that path.

## Best Possible Solution

Align bundled public-surface loading with the existing bundled plugin hardlink policy, add a hardlinked bundled public-artifact regression test, and keep broader Google disable/uninstall or secondary-provider policy changes out of the narrow fix unless maintainers explicitly scope them in.

## Reproduction Assessment

Yes for the source-level path: current main still routes configured Google/Gemini web-search secrets to the bundled public artifact loader, which rejects hardlinked artifacts and then startup secret activation fails fast. No for the full macOS/Homebrew LaunchAgent crash loop, which was not live-run in this read-only review.

## Solution Assessment

Yes for the narrow repair direction. The maintainable fix is in src/plugins/public-surface-loader.ts or a shared hardlink-policy helper so bundled public surfaces follow existing bundled hardlink rules without changing SecretRef startup semantics in the same PR.
