## Summary

Keep open: current main still routes configured Google/Gemini web-search secrets through the bundled public-surface loader, the loader still rejects hardlinked public artifacts, and startup secret activation still fails the Gateway on startup errors. The issue-specific main change I found improves diagnostic stability bundles, but it does not prove the v2026.4.29 crash-loop root path is fixed.

## Best Possible Solution

Fix https://github.com/openclaw/openclaw/issues/75797 with a narrow packaged public-surface loader repair. Current `src/plugins/public-surface-loader.ts` opens bundled public artifacts with `rejectHardlinks: true`, so valid package-manager/global-install hardlinked artifacts can fail as `Unable to open bundled plugin public surface google/web-search-contract-api.js` even though direct Node import works. Update the loader to accept valid OpenClaw-owned bundled public artifacts after boundary-root and same-file-identity validation, following the existing bundled-plugin policy that allows bundled hardlinked entry files while preserving hardlink rejection for untrusted external/custom plugin surfaces. Do not broaden plugin install/discovery trust policy, do not move Google-specific logic into core, and do not relax startup SecretRef fail-fast semantics. Add a regression test in `src/plugins/public-surface-loader.test.ts` that loads a hardlinked bundled public artifact from a trusted bundled-dir fixture, and keep the post-validation identity race check. Validate with targeted tests for the public-surface loader and the Google/Gemini secrets path.
