## Summary

Keep open: the linked bug is valid, but this PR is not merge-ready because the new regression test does not set up a real queued idle stall and the runtime change bypasses the documented diagnostic event backoff instead of separating recovery scheduling from warning emission.

## What This Changes

The PR changes `logSessionAttention` so recovery-eligible stuck or long-running classifications bypass warning backoff and adds a Vitest regression test for that path.

## Best Possible Solution

Revise the PR so warning/event throttling remains intact while recovery requests still run, then prove it with a real heartbeat-path regression test and redacted after-fix terminal output.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current `main` returns from `logSessionAttention` during warning backoff before the heartbeat can call `requestStuckSessionRecovery`; I did not run a live QQ gateway reproduction in this read-only review.

## Solution Assessment

No. The direction is right, but this patch is not the best fix because it bypasses warning/event throttling instead of separating recovery scheduling from diagnostic emission, and the new test does not exercise the intended runtime path.

- **[P1] Use a real queued idle stall in the regression test:** `src/logging/diagnostic.test.ts:510`
  - body: `logSessionStateChange` does not accept or apply `queueDepth`, so this object literal should fail type checking and at runtime leaves the session with `queueDepth === 0`.   - confidence: 0.96

  - body: Adding `!recoveryEligible` here lets recovery-eligible sessions bypass the warning throttle entirely, so `session.stuck` warnings/events can emit every heartbeat. The docs and telemetry contract say repeated `session.stuck` diagnostics back off while unchanged; the fix should let recovery scheduling continue without also unthrottling log/event emission.
  - confidence: 0.9
