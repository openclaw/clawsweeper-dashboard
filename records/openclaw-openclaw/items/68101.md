## Summary

Keep open: both current checked-out main and a read-only remote-head source check still swallow Lobster workflow-file resolution errors and fall back to inline pipeline execution. The latest release tag still has the same behavior, and the open closing PR is the right owner for the remaining fix path.

## What This Changes

The issue asks Lobster to treat relative workflow-looking paths as workflow files or clear path errors instead of falling back to inline pipeline parsing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/68106, or replace it with the same narrow Lobster runner fix plus regression coverage if the branch cannot be made mergeable.

## Reproduction Assessment

Yes from source inspection: current `detectWorkflowFile` catches `resolveWorkflowFile` failures and returns `null`, which allows fallback to inline pipeline execution. I did not run a live tool invocation because this is a read-only review.

## Solution Assessment

Yes, failing closed for workflow-looking unresolved paths inside the Lobster runner is the narrow maintainable fix; the docs already define workflow files as `pipeline` file paths, and normal inline pipelines can remain on the existing fallback path.
