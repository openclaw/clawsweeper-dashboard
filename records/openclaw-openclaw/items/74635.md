## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open PR #74974 explicitly closes this issue with the narrow schema/test fix, so the issue should stay open until that canonical fix merges or is superseded.

## What This Changes

Allow Paperclip's root-level `paperclip` metadata on Gateway `agent` requests without relaxing unrelated unknown-field rejection.

## Best Possible Solution

Review and land one additive Gateway protocol compatibility fix through PR #74974 or an explicitly superseding PR, keeping schema, validator tests, generated client models, and any required changelog/docs aligned.

## Reproduction Assessment

Yes. A minimal `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the report and deterministically fails current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. The reporter's `paperclip: Type.Optional(Type.Unknown())` workaround is the narrowest immediate fix and matches PR #74974, but maintainers may prefer a generic opaque metadata namespace before growing the public wire contract with a vendor-named field.
