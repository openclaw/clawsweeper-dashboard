## Summary

Keep open. Current main still rejects the reported root-level `paperclip` field before the agent request reaches runtime dispatch, and open PR #74974 plus related PR #69139 are implementation candidates rather than a reason to close this issue before one canonical fix merges.

## What This Changes

The issue requests Gateway `agent` params compatibility with Paperclip payloads that include a root-level `paperclip` metadata field.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix that accepts Paperclip metadata, preserves strict rejection for unrelated unknown fields, and updates schema tests, docs if needed, and generated client artifacts together.

## Reproduction Assessment

Yes. A minimal `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the report and deterministically fails current `validateAgentParams` because the closed schema omits that field.

## Solution Assessment

Unclear. The reporter's `paperclip: Type.Optional(Type.Unknown())` workaround is the narrowest immediate compatibility fix, but protocol owners should decide whether a generic opaque metadata namespace is the better public contract before merging one of the open PRs.
