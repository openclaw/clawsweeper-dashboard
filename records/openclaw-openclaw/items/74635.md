## Summary

Keep open. Current main still rejects root-level Paperclip metadata on Gateway `agent` requests, and open PR #74974 explicitly tracks the same compatibility fix, so this issue should remain open until that PR lands or is replaced by a canonical protocol change.

## What This Changes

The issue asks for Gateway agent invocation validation to tolerate Paperclip's root-level `paperclip` metadata so Paperclip-backed agents can run instead of receiving `INVALID_REQUEST`.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix through PR #74974 or a maintainer-selected replacement, preserving strict rejection for unrelated typos while updating schema, regression tests, generated client models, and any required changelog/docs together.

## Reproduction Assessment

Yes. A minimal `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object deterministically fails current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. The reporter's `paperclip: Type.Optional(Type.Unknown())` workaround is the narrowest immediate compatibility fix and matches the active PR, but maintainers may prefer a generic opaque metadata namespace before extending the wire contract.
