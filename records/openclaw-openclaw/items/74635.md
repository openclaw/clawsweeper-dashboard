## Summary

Keep open. Current main still rejects root-level Paperclip metadata on Gateway `agent` requests, and open PR #74974 explicitly closes this issue while #69139 covers the same root cause, so the issue should remain open until one canonical protocol change merges.

## What This Changes

Allow Paperclip's root-level `paperclip` metadata on Gateway `agent` invocation payloads, or route that context through an existing supported field, so Paperclip `openclaw_gateway` calls are not rejected.

## Best Possible Solution

Review and land one additive Gateway protocol compatibility fix, preferably through the canonical open PR selected by maintainers, while preserving strict rejection for unrelated typos and keeping schema, validator tests, docs, and generated client artifacts in sync.

## Reproduction Assessment

Yes. A minimal `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the report and deterministically fails current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. The reporter's `paperclip: Type.Optional(Type.Unknown())` workaround is the narrowest immediate compatibility fix, but the Gateway protocol boundary may be better served by a generic opaque metadata namespace if maintainers want to avoid a vendor-named protocol field.
