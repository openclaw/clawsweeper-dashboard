## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` requests, the latest stable release has the same strict schema, and https://github.com/openclaw/openclaw/pull/74974 is the open linked implementation candidate, so cleanup should wait for that PR or a superseding protocol fix to land.

## What This Changes

The issue asks OpenClaw Gateway to accept Paperclip `openclaw_gateway` agent invocations that include root-level `paperclip` metadata instead of rejecting them as unknown AgentParams fields.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74974 or a superseding maintainer-approved additive AgentParams metadata contract with schema, generated client model, focused validator coverage, and changelog aligned, then close the linked Paperclip reports.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: send a Gateway `agent` request with valid `message` and `idempotencyKey` plus root `paperclip`, and the closed AgentParams validator rejects it before dispatch.

## Solution Assessment

Unclear until protocol owners approve the public contract. The open PR's generic `adapterMeta` namespace plus a temporary `paperclip` alias is the strongest current implementation candidate and is more maintainable than a permanent vendor-only root field.
