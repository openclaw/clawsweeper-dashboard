## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` requests, and open PR #74974 explicitly closes this issue, so cleanup should wait for that PR or a superseding fix to land.

## What This Changes

The issue asks OpenClaw Gateway to tolerate Paperclip root `paperclip` metadata on `agent` invocation payloads instead of rejecting them as unexpected properties.

## Best Possible Solution

Land an additive Gateway protocol compatibility change with schema, tests, generated client artifacts, and changelog aligned while preserving strict rejection for unrelated root fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an `agent` RPC with valid `message` and `idempotencyKey` plus a root `paperclip` property, and `validateAgentParams` rejects before dispatch.

## Solution Assessment

Unclear. The linked PR's Paperclip-specific tolerance is the narrow unblocker, but a generic adapter metadata namespace may be more maintainable and needs maintainer judgment during #74974 review.
