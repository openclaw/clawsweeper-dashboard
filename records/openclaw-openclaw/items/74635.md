## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` params, and open PR #74974 explicitly closes this issue, so the issue should remain open until that PR or a superseding fix lands.

## What This Changes

The issue reports that Paperclip's `openclaw_gateway` adapter sends a root-level `paperclip` field in Gateway `agent` params, causing OpenClaw to reject the invocation.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix for adapter metadata in `agent` params, preserving strict rejection for unrelated root fields and keeping schema, tests, generated clients, and changelog aligned.

## Reproduction Assessment

Yes. A minimal `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` object follows the report and reaches `validateAgentParams`, where current main rejects it before dispatch.

## Solution Assessment

Unclear. Adding optional opaque `paperclip` metadata is the narrowest immediate unblocker and matches PR #74974, but a generic adapter metadata namespace may be the more maintainable public protocol surface.
