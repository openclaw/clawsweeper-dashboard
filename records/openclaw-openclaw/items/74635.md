## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open PR #74974 explicitly closes this issue, so the issue should remain open until that PR or a superseding fix lands.

## What This Changes

Paperclip sends a root-level `paperclip` property in Gateway agent invocation params, and the report asks OpenClaw to accept or route that metadata instead of rejecting the request.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix for adapter metadata in `agent` params, preserving strict rejection for unrelated root fields and keeping schema, tests, generated clients, docs, and changelog aligned.

## Reproduction Assessment

Yes. A minimal valid `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` object follows the report and fails current `validateAgentParams` before dispatch because the schema omits that property and rejects extras.

## Solution Assessment

Unclear. Adding `paperclip: Type.Optional(Type.Unknown())` is the narrowest immediate unblocker, but a generic adapter metadata namespace may be the more maintainable public protocol surface if maintainers want to avoid a service-named root field.
