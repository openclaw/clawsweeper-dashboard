## Summary

Keep open. Current main still rejects root-level Paperclip metadata on Gateway `agent` requests, and open PR #74974 explicitly closes this issue, so the issue should remain open until one canonical protocol/schema change merges.

## What This Changes

Allow Paperclip's root-level `paperclip` metadata on Gateway `agent` requests so Paperclip-invoked agents are not rejected by strict `AgentParamsSchema` validation.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix through the canonical PR path, preserving strict rejection for unrelated typos while updating schema, validator tests, generated client models, and any required changelog/docs together.

## Reproduction Assessment

Yes. A minimal `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the report and deterministically fails current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. The reporter's `paperclip: Type.Optional(Type.Unknown())` workaround is the narrowest immediate fix and matches the open PR, but maintainers may prefer a generic opaque metadata namespace before making the wire contract public.
