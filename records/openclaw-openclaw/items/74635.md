## Summary

Keep open. Current `main` still rejects root-level `paperclip` metadata on Gateway `agent` requests, and the provided GitHub context shows open PR #74974 explicitly closes this issue, so the correct cleanup path is to review and land or decline the canonical protocol fix before closing.

## What This Changes

The issue requests Gateway compatibility for Paperclip agent invocation payloads that include a root-level `paperclip` metadata field.

## Best Possible Solution

Land one canonical additive Gateway protocol compatibility PR, likely #74974 or a maintainer-approved replacement, with focused validator coverage and generated protocol artifacts updated if the wire contract changes.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` field follows the report and will fail current `validateAgentParams` because the closed schema omits that field.

## Solution Assessment

Unclear. The reporter's `paperclip: Type.Optional(Type.Unknown())` shim is narrow and likely compatible, but a generic opaque metadata namespace may be a cleaner long-term protocol contract if maintainers want to avoid encoding an external integration name in core.
