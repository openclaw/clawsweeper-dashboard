## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open PR #74974 explicitly closes this issue with the narrow schema/test fix, so this issue should remain open until that PR or a superseding fix lands.

## What This Changes

Allow Paperclip's root-level `paperclip` metadata on Gateway agent invocation payloads without disabling strict validation for other unknown fields.

## Best Possible Solution

Land an additive Gateway protocol compatibility fix that accepts opaque Paperclip metadata on agent calls while keeping unknown-field rejection for other names, with schema, tests, and generated clients aligned.

## Reproduction Assessment

Yes. A minimal `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the report and deterministically fails the current closed `AgentParamsSchema` before dispatch.

## Solution Assessment

Unclear. The reporter's `paperclip: Type.Optional(Type.Unknown())` workaround and PR #74974 are the narrowest immediate unblocker, but maintainers may prefer a generic opaque metadata namespace for the public Gateway protocol.
