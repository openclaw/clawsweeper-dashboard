## Summary

Keep open. Current `main` still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open implementation PRs already reference this issue, so the right maintainer cleanup is to review and consolidate the canonical protocol fix rather than close the issue early.

## What This Changes

The issue reports that OpenClaw Gateway rejects Paperclip agent invocation payloads because the strict Gateway `AgentParamsSchema` does not allow a root-level `paperclip` metadata field.

## Best Possible Solution

Land one canonical additive Gateway protocol compatibility change, with focused validator coverage and generated protocol follow-through, then close or supersede the duplicate PRs once that fix merges.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` object follows the report and will fail current `validateAgentParams` because the closed schema omits that field.

## Solution Assessment

Unclear. The reporter's `paperclip: Type.Optional(Type.Unknown())` shim is the narrowest compatibility fix, but a generic adapter metadata field may be the more maintainable protocol shape if maintainers want to avoid encoding one external integration name in core.
