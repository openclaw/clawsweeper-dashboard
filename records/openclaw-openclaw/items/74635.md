## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open PR #74974 explicitly closes this issue, so the issue should remain open until that PR or a superseding fix lands.

## What This Changes

Allow Paperclip's `openclaw_gateway` agent payload metadata to pass Gateway `AgentParams` validation instead of being rejected as an unexpected root property.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix for adapter metadata in `agent` params, preserving strict rejection for unrelated root fields and keeping schema, tests, generated clients, and changelog aligned.

## Reproduction Assessment

Yes. A minimal valid `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` object follows the report and fails current `validateAgentParams` before dispatch.

## Solution Assessment

Unclear. Adding `paperclip: Type.Optional(Type.Unknown())` is the narrowest immediate unblocker, but a generic adapter metadata namespace may be the more maintainable public protocol surface.
