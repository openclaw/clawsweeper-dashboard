## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` requests, and https://github.com/openclaw/openclaw/pull/74974 is still an open linked implementation candidate, so this issue should remain open until that PR or a superseding protocol fix lands.

## What This Changes

Request that Gateway agent invocations accept or correctly route Paperclip integration metadata instead of rejecting a root-level `paperclip` payload field.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74974 or a superseding maintainer-approved additive AgentParams metadata contract with schema, generated client model, focused validator coverage, and changelog, then close the linked Paperclip reports.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: send a Gateway `agent` request with valid `message` and `idempotencyKey` plus root `paperclip`, and the closed AgentParams validator rejects it before dispatch.

## Solution Assessment

Unclear until protocol owners approve the public contract. The open PR's generic `adapterMeta` namespace plus a temporary `paperclip` alias is more maintainable than a permanent vendor-only root field, but it is still an API/product decision.
