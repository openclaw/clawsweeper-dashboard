## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` requests, and #74974 is still the open implementation candidate that explicitly closes this issue.

## What This Changes

Allow Paperclip's root-level `paperclip` metadata on Gateway agent invocation payloads without relaxing strict validation for unrelated fields.

## Best Possible Solution

Land or replace #74974 with an additive Gateway protocol compatibility change that keeps unrelated unknown fields rejected and updates schema, tests, generated client artifacts, and changelog together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an `agent` RPC with valid `message` and `idempotencyKey` plus root `paperclip`, and `validateAgentParams` rejects before dispatch.

## Solution Assessment

Unclear. #74974 is the narrow unblocker, but a generic adapter metadata namespace may be the more maintainable protocol shape and needs maintainer judgment.
