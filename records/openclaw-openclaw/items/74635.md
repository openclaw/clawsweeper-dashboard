## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` requests, and open PR #74974 explicitly closes this issue, so cleanup should wait for that PR or a superseding protocol fix to land.

## What This Changes

The issue requests that Gateway `agent` invocation payloads from Paperclip stop failing validation when they include a root-level `paperclip` metadata field.

## Best Possible Solution

Review and land #74974, or replace it with an additive protocol compatibility fix that updates schema, tests, generated client artifacts, and changelog while preserving strict rejection for unrelated root fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an `agent` RPC with valid `message` and `idempotencyKey` plus root `paperclip`, and `validateAgentParams` rejects it before dispatch.

## Solution Assessment

Unclear. The linked PR's Paperclip-specific tolerance is the narrow unblocker, but a generic adapter metadata namespace may be more maintainable and needs maintainer judgment during #74974 review.
