## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` requests, and https://github.com/openclaw/openclaw/pull/74974 is open, mergeable, and linked to close this issue, so cleanup should wait for that PR or a superseding fix to land.

## What This Changes

The issue asks for Gateway agent requests from Paperclip to stop failing when the payload includes root-level `paperclip` metadata.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/74974 or a superseding maintainer-approved additive AgentParams metadata contract with schema, generated client model, tests, and changelog aligned, then close the linked Paperclip reports.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: send a Gateway `agent` request with valid `message` and `idempotencyKey` plus root `paperclip`, and the closed AgentParams validator rejects it before dispatch.

## Solution Assessment

Unclear until protocol owners approve the public contract. The open PR's `adapterMeta` namespace plus temporary `paperclip` alias is the best current implementation candidate, but the namespace shape and alias lifecycle are protocol decisions.
