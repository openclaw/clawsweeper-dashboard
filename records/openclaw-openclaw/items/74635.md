## Summary

Keep open. Current main still rejects root-level Paperclip metadata on Gateway `agent` requests, and open implementation PRs already reference this compatibility fix, so the issue should remain open until one canonical protocol change merges.

## What This Changes

Paperclip's `openclaw_gateway` adapter needs OpenClaw Gateway `agent` params to accept root-level `paperclip` metadata or route that context through a supported metadata field.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix for adapter metadata in `agent` params, preserve strict rejection for unrelated typos, add focused regression coverage, regenerate protocol artifacts, and add the required user-facing changelog entry before closing this issue and consolidating duplicates.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` object follows the report and will fail current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. The reporter's optional `paperclip: Type.Unknown()` field is the narrowest immediate compatibility shim, but a generic adapter metadata namespace may be the more maintainable protocol shape unless maintainers approve a vendor-named exception.
