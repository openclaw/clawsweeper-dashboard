## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open PR #74974 explicitly closes this issue, so the issue should remain open until that PR or a superseding fix lands.

## What This Changes

Paperclip agent invocations send a root-level `paperclip` metadata field that OpenClaw Gateway's closed `AgentParamsSchema` rejects.

## Best Possible Solution

Land or supersede the Gateway protocol compatibility fix so `agent` requests accept the agreed opaque Paperclip metadata while preserving strict rejection for unrelated unknown fields and keeping tests/generated clients aligned.

## Reproduction Assessment

Yes. A minimal `agent` request containing valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the report and will fail the current closed schema before dispatch.

## Solution Assessment

Unclear. The narrow `paperclip: Type.Optional(Type.Unknown())` fix is the fastest unblocker and already has PR coverage, but maintainers may prefer a generic metadata namespace for the public Gateway protocol.
