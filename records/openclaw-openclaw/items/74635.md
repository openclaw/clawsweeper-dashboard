## Summary

Keep open. Current main still rejects root-level `paperclip` on Gateway `agent` requests, and PR #74974 is the open implementation candidate that targets this issue, so cleanup should wait for that PR or a superseding fix to land.

## What This Changes

Allow Paperclip's root-level `paperclip` metadata on Gateway `agent` invocation payloads so Paperclip tasks do not fail schema validation.

## Best Possible Solution

Land an additive Gateway protocol compatibility change with schema, regression tests, generated client artifacts, and changelog aligned while preserving strict rejection for unrelated root fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an `agent` RPC with valid `message` and `idempotencyKey` plus a root `paperclip` property, and `validateAgentParams` rejects before dispatch.

## Solution Assessment

Unclear. Accepting `paperclip` exactly matches the reporter's workaround and PR #74974, but maintainers should decide whether top-level Paperclip tolerance or a generic adapter metadata namespace is the right protocol shape.
