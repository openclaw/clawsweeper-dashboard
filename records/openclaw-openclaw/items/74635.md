## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open PR #74974 explicitly closes this issue, so it should stay open until that fix merges or is replaced.

## What This Changes

Allow Gateway agent invocation payloads from Paperclip to include root-level `paperclip` metadata without rejecting the request.

## Best Possible Solution

Review and land PR #74974 or a superseding additive Gateway protocol change that accepts the needed opaque metadata while keeping schema, tests, docs, and generated clients aligned.

## Reproduction Assessment

Yes. A minimal `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the reported path and is rejected by current main's closed `AgentParamsSchema` validator.

## Solution Assessment

Unclear. Accepting `paperclip: Type.Optional(Type.Unknown())` is the smallest compatible unblocker, but protocol owners may prefer a generic opaque metadata namespace before growing the public wire contract with a vendor-named field.
