## Summary

Keep open. Current main still rejects root-level Paperclip metadata on Gateway `agent` requests, and open PR #74974 explicitly tracks this issue while #69139 covers the same schema/test repair, so the issue should remain open until one canonical protocol change merges.

## What This Changes

Allow Paperclip's root-level `paperclip` metadata on Gateway `agent` invocation payloads, or route that context through an existing supported field, so Paperclip openclaw_gateway requests are accepted.

## Best Possible Solution

Review and land one additive Gateway protocol compatibility fix through the selected canonical PR, preserving strict rejection for unrelated typos while keeping schema, validator coverage, generated clients, docs if needed, and changelog aligned.

## Reproduction Assessment

Yes. A minimal `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the report and deterministically fails current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. The reporter's `paperclip: Type.Optional(Type.Unknown())` workaround is the narrowest immediate compatibility fix, but the Gateway protocol boundary may be better served by a generic opaque metadata namespace if maintainers want to avoid a vendor-specific root field.
