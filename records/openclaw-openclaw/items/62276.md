## Summary

Keep open. Current main has fixed several gateway port-conflict classes, but it still does not fully close the two signals in this thread: the original Windows/Dreaming startup path lacks current-main proof, and the Linux noninteractive onboarding path still installs a user service without the system-level service guard that doctor already uses.

## What This Changes

Avoid OpenClaw gateway startup failing with port 18789 already in use after configuring Dreaming, including duplicate gateway-service collisions.

## Best Possible Solution

## Solution Assessment

Unclear. Current main is a partial solution with better lock diagnostics, systemd exit-78 handling, macOS stale-PID cleanup, and Dreaming cron retry, but it does not yet prove or fix the remaining mixed Windows/Dreaming plus Linux onboarding concerns.
