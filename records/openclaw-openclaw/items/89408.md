## Summary

Keep open: current main and v2026.6.8 still keep bundled MCP runtimes session-scoped and await catalog materialization during agent/session runtime setup. The related open PRs are partial and unmerged, so the reported session-switch latency remains necessary to track.

## What This Changes

The issue reports multi-second session-switch delay when OpenClaw rebuilds a session and synchronously materializes MCP tools via `getCatalog()` and `createRuntime`.

## Best Possible Solution

Land or replace a maintainer-approved MCP runtime latency fix that preserves session/workspace isolation, adds focused upgrade and lifecycle tests, and includes benchmark proof for session switching with a heavy MCP catalog.

## Reproduction Assessment

Yes at source level: current main still awaits session runtime recreation, MCP runtime lookup, catalog loading, and tool materialization on the setup path. No for the exact measured 5-8 second delay, because I did not run the reporter's Windows/vibe-trading workload.

## Solution Assessment

Unclear as a final product choice. Shared runtime pooling most directly targets repeated cross-session cold starts, but it needs an explicit maintainer decision on the isolation boundary; catalog/LSP parallelization is useful but only a partial mitigation.
