## Summary

Keep open: current main and the latest release still keep bundled MCP runtimes session-scoped and await catalog materialization during session replacement, so the reported session-switch latency remains actionable. There are active related PRs, but they are unmerged and cover different parts of the solution space.

## What This Changes

This issue reports multi-second session-switch delay because session replacement awaits bundled MCP runtime creation, catalog fetching, and tool materialization.

## Best Possible Solution

Land or replace the active shared-runtime direction with explicit isolation docs and upgrade-safe tests, while treating parallel catalog/LSP initialization as a complementary latency mitigation rather than the full session-switch fix.

## Reproduction Assessment

No for the exact measured 5-8 second delay: I did not run a live Windows/vibe-trading current-main reproduction. Yes at source level: session replacement awaits runtime creation, embedded setup awaits MCP materialization, and materialization awaits getCatalog/connect/listTools.

## Solution Assessment

Unclear as a final product choice. Shared runtime scope is the strongest fit for repeated cross-session cold starts, but it needs explicit maintainer acceptance of the session-isolation boundary; parallel loading PRs are useful partial mitigations.
