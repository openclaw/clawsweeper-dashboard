## Summary

Keep open: current main and v2026.6.9 still have the session-scoped, synchronously materialized MCP catalog path, and the newer related work splits the problem across runtime sharing and cold-start parallelization rather than providing one merged canonical fix.

## What This Changes

The issue asks OpenClaw to stop making session switches wait on per-session MCP runtime creation, catalog loading, and tool materialization for large MCP tool sets.

## Best Possible Solution

Review and land or replace a focused maintainer-approved MCP startup fix that addresses session-switch latency while preserving config invalidation, runtime isolation, active leases, idle sweep, disposal, and tool-list refresh semantics.

## Reproduction Assessment

Yes for the source-level blocking shape: current main awaits session MCP runtime creation and catalog materialization before bundled tools are available, and the runtime/cache key remains session-scoped. I did not run a live 70-tool stdio MCP workload, so the exact 5-8s number remains issue-provided.

## Solution Assessment

Unclear as final product direction. Cross-session pooling, lazy materialization, and parallel catalog loading address overlapping causes, but they carry different lifecycle and isolation tradeoffs; the best fix needs maintainer approval of that runtime contract.
