## Summary

Keep open: current main still awaits bundled MCP catalog materialization during embedded-run setup and the runtime cache is keyed by session id, so the session-switch performance complaint remains actionable. The related open MCP issues are adjacent, but none is a clean canonical replacement for this session-switch-specific report.

## What This Changes

The issue reports that switching sessions in the TUI/agent runtime waits for bundled MCP runtime creation and full tool catalog materialization, causing multi-second delays with large stdio MCP servers.

## Best Possible Solution

Design an owner-approved MCP runtime warm/reuse path that keeps session replacement responsive while preserving workspace/config/session isolation, tool readiness semantics, and focused performance tests.

## Reproduction Assessment

No measured live reproduction was run, so we do not have a high-confidence timed repro in this review. Source inspection does show the reported path: session switch awaits createRuntime, attempt setup awaits MCP materialization, and materialization awaits getCatalog/connect/listTools.

## Solution Assessment

Unclear whether the suggested cross-session pool is the best fix as written. The best path needs maintainer design around session isolation, tool readiness, and whether pooling, background warmup, or catalog caching should own the improvement.
