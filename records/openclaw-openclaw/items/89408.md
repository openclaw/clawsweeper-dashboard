## Summary

Keep open: current main and the latest release still create/session-cache bundled MCP runtimes by session id and await catalog materialization on the embedded run setup path, so the reported session-switch latency remains actionable. The open shared-runtime PR is relevant and proof-positive, but it is not merged and still needs maintainer acceptance of the cross-session MCP isolation boundary.

## What This Changes

Reduce session-switch latency by avoiding synchronous per-session MCP runtime/catalog materialization during runtime creation.

## Best Possible Solution

Review and land the shared-runtime direction or an equivalent owner-approved MCP warm/reuse design with isolation docs, regression tests, and before/after session-switch proof.

## Reproduction Assessment

No high-confidence measured current-main reproduction was run in this read-only pass. Source inspection does reproduce the blocking path: session replacement awaits runtime creation, embedded setup awaits MCP materialization, and materialization awaits getCatalog/connect/listTools.

## Solution Assessment

Unclear. The open shared-runtime PR is a plausible high-impact fix, but the best solution depends on maintainer choice around session isolation, readiness semantics, and whether the reuse mode should be default, opt-in, or replaced by a narrower warmup/cache design.
