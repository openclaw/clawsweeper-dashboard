## Summary

Keep open: current main and v2026.6.10 still intentionally allow outside-workspace `apply_patch` writes when the dangerous opt-out is set, while the audit class remains `warn`; the live issue also now has the protected `security` label, so this needs maintainer security/product review rather than cleanup close.

## What This Changes

The issue requests stronger handling of the `tools.exec.applyPatch.workspaceOnly=false` dangerous config flag because it permits patch writes outside the workspace.

## Best Possible Solution

Security maintainers should decide the permanent policy for this opt-out, then update audit severity/catalog, schema help, docs, and regression tests together.

## Reproduction Assessment

Yes at source level: current code and tests show explicit `workspaceOnly=false` permits outside-workspace `apply_patch` writes and the audit reports it as warn. I did not run a live mutating patch in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security posture. A narrow severity/docs/test update is straightforward, but removal or secondary confirmation would be a broader behavior change.
