## Summary

Keep open: current main and the latest release still intentionally allow outside-workspace `apply_patch` writes when the dangerous opt-out is set, and the audit class remains `warn`; the remaining question is a security/product decision, not a cleanup close.

## What This Changes

The issue requests stronger security treatment for `tools.exec.applyPatch.workspaceOnly=false`, including higher audit severity, clearer docs, possible removal, or extra confirmation for outside-workspace patches.

## Best Possible Solution

Security maintainers should decide the permanent policy for this opt-out, then update audit severity/catalog, schema help, docs, and regression tests together.

## Reproduction Assessment

Yes at source level: current code and tests show explicit `workspaceOnly=false` permits outside-workspace `apply_patch` writes and the audit reports it as warn. I did not run a live mutating patch in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security posture. A narrow severity/docs/test update is straightforward, but removal or secondary confirmation would be a broader behavior change.
