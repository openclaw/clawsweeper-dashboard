## Summary

Keep open: this is a plausible focused behavior change, but it changes a documented default from opt-in to opt-out and still lacks after-fix real behavior proof from a cron alert run.

## What This Changes

The PR changes cron failure-alert resolution so recurring jobs alert by default when no global or per-job failure-alert config is set, and adds a Vitest case for that path.

## Best Possible Solution

Maintainers should decide whether cron failure alerts should become opt-out; if approved, align the config docs and land the resolver/test change with real cron alert proof, otherwise keep the documented opt-in default.

## Reproduction Assessment

Yes for source-level current behavior: current main returns null for unset job/global failure-alert config, so alerts do not fire by default. I did not run a live cron job, and the docs currently define that opt-in behavior as expected.

## Solution Assessment

No, not as submitted. The resolver and test change are narrow, but changing a documented default needs maintainer approval, docs alignment, and after-fix real behavior proof before merge.
