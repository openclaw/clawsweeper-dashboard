## Summary

Keep open. Current main still keeps cron failure alerts opt-in, and the PR's runtime/test direction is narrow, but the branch would make a user-visible default change while the public configuration reference still says the default is disabled and the active changelog has no matching entry.

## What This Changes

The PR changes cron failure-alert resolution so unset global/per-job alert config alerts by default, and adds a unit test for repeated failures with unset alert config.

## Best Possible Solution

If maintainers want default-on failure alerts, land a revised branch that keeps the narrow runtime guard and regression test while aligning the public config reference, relevant cron docs/generated docs artifacts if required, and changelog. If default-on notifications are not desired, keep the existing opt-in gate and close the branch without merging.

Yes. On current main, create or simulate an isolated recurring cron job with announce delivery and no job-level failureAlert plus no global cron.failureAlert.enabled=true; resolveFailureAlert returns null, so repeated failures cannot emit the alert path.

## Solution Assessment
