## Summary

Keep open. Current main still has the opt-in failure-alert gate, so the PR is not obsolete, but the submitted patch changes a product-visible default without aligning the public config docs and changelog.

## What This Changes

The PR changes cron failure alerts with no explicit job or global alert config from opt-in to opt-out and adds a regression test for the default alert path.

## Best Possible Solution

Land a revised version only after maintainers confirm opt-out failure alerts are the intended product default, with runtime behavior, regression coverage, public config docs, generated config artifacts if required, changelog, and changed-gate proof aligned.

## Reproduction Assessment

Yes. On current main, a recurring job with no job-level `failureAlert` and no global `cron.failureAlert.enabled: true` follows the `resolveFailureAlert` guard and cannot emit repeated-failure alerts by default; the PR's added test is the focused reproduction for that path.

## Solution Assessment

No, not as submitted. The runtime guard and test are narrow, but changing the default without updating the documented config default and changelog is incomplete; the safer path is to align docs/changelog after maintainer confirmation or keep the opt-in default.
