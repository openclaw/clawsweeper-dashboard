## Summary

This PR should stay open for maintainer handling: it changes cron failure alerts from documented opt-in behavior to default-on alerts, which is a product/upgrade decision and still lacks real behavior proof.

## What This Changes

The PR changes cron failure-alert resolution so recurring jobs alert by default when no job or global failure-alert config is set, and adds a unit test for that default path.

## Best Possible Solution

Either keep the documented opt-in default, or if maintainers approve opt-out semantics, align docs and upgrade expectations before landing the resolver/test change with real cron alert proof.

## Reproduction Assessment

Yes for the current source-level behavior: current main returns null for unset job/global failure-alert config, so alerts do not fire by default. That behavior also matches the current configuration reference, so the PR is changing policy rather than fixing a documented bug.

## Solution Assessment

No, not as submitted. The resolver and test are narrow, but changing a documented notification default needs maintainer approval, docs alignment, upgrade-safety acceptance, and after-fix real behavior proof.
