## Summary

Keep open: the PR changes cron failure alerts from opt-in to default-on, while current source and public docs still define opt-in behavior. It needs a maintainer default-policy decision, docs alignment, and contributor real behavior proof before merge.

## What This Changes

The PR changes `resolveFailureAlert` so unset global/job alert config sends recurring cron failure alerts by default and adds a Vitest case for the unset-config path.

## Best Possible Solution

Decide whether cron failure alerts should become opt-out; if approved, land the resolver/test change with docs and config expectations aligned, otherwise keep the existing opt-in guard.

## Reproduction Assessment

Yes for the described current behavior: source inspection shows unset job/global failure-alert config returns `null`, so alerts do not fire by default. That proves opt-in behavior, but not a bug against the current docs because the config reference says the default is `false`.

## Solution Assessment

No, not as submitted. The runtime change and regression test are narrow, but changing a documented default needs maintainer approval and docs alignment before merge.
