## Summary

This PR is a plausible policy change, but it is not a straight bug fix against the current documented contract: it changes cron failure alerts from opt-in to opt-out, so it needs maintainer approval, docs alignment, and contributor real-behavior proof before merge.

## What This Changes

The PR changes cron failure-alert resolution so omitted global/job alert config sends alerts by default, and adds a regression test for that unset-config path.

## Best Possible Solution

Decide whether cron failure alerts should become opt-out; if yes, land the narrow resolver/test change with public config docs updated to match, otherwise keep the current opt-in guard.

## Reproduction Assessment

Yes for the described current behavior: source inspection shows unset job/global `failureAlert` config returns `null`, so alerts do not fire by default. That proves the opt-in behavior, but not a contract bug because the config reference says the default is `false`.

## Solution Assessment

No, not as submitted. The runtime change and regression test are narrow, but changing a documented notification default needs maintainer approval and docs alignment before merge.
