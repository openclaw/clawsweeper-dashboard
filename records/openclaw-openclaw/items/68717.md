## What This Changes

The PR flips cron failure-alert resolution from opt-in to opt-out for jobs without job-level failureAlert config and adds a focused Vitest case for that default path.

## Best Possible Solution

Maintainers should explicitly choose opt-in versus opt-out for cron failure alerts; if opt-out is approved, align docs/config examples and capture redacted real cron alert proof before merge.

## Reproduction Assessment

Yes for source-level behavior: current main returns null when both job.failureAlert and cron.failureAlert.enabled are unset, so alerts do not fire by default. I did not run a live cron job, and the docs define that opt-in behavior as expected.

## Solution Assessment

No, not as submitted. The resolver and test change are narrow, but changing a documented default needs maintainer approval, docs and upgrade alignment, and after-fix real behavior proof.
