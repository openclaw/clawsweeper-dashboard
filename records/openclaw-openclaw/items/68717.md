## Summary

Keep this PR open, but it should not merge as submitted: it changes cron failure alerts from documented opt-in behavior to opt-out behavior without updating the public config contract, and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR changes `resolveFailureAlert` so unset global/job failure-alert config sends recurring cron failure alerts by default, and adds a unit test for that unset-config path.

## Best Possible Solution

Decide whether cron failure alerts should become opt-out; if yes, land the narrow resolver/test change with public config docs aligned, otherwise keep the existing opt-in guard.

## Reproduction Assessment

Yes for the described current behavior: source inspection shows unset job/global `failureAlert` config returns `null`, so alerts do not fire by default. That proves the opt-in behavior, but not a bug against the current docs because the config reference says the default is `false`.

## Solution Assessment

No, not as submitted. The runtime change and regression test are narrow, but changing a documented default needs maintainer approval and docs alignment before merge.
