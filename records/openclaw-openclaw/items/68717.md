## What This Changes

The PR changes cron failure-alert resolution so unset global/job alert config sends alerts by default, and adds a unit test for that unset-config path.

## Best Possible Solution

Decide whether cron failure alerts should become opt-out; if yes, land the resolver/test change with public docs and config expectations aligned, otherwise keep the existing opt-in guard.

## Reproduction Assessment

Yes for the described current behavior: source inspection shows unset job/global failure-alert config returns `null`, so alerts do not fire by default. That proves opt-in behavior, but not a bug against the current docs because the config reference says the default is `false`.
