## Summary

Keep open: this PR is a focused runtime/test change, but it changes a documented cron alert default from opt-in to opt-out and still needs product signoff, docs alignment, and real behavior proof before merge.

## What This Changes

The PR changes cron failure-alert resolution so recurring jobs alert when global and job-level `failureAlert` config is unset, and adds a unit test for that default path.

## Best Possible Solution

Decide whether cron failure alerts should become opt-out; if yes, land the narrow resolver/test change with config docs aligned, otherwise keep the documented opt-in behavior.

## Reproduction Assessment

Yes. On current main, a recurring job with no job-level `failureAlert` and no `cron.failureAlert.enabled=true` reaches `resolveFailureAlert` and returns `null`, so the alert path cannot fire; this was verified by source inspection rather than by running tests in the read-only sweep.

## Solution Assessment

No, not as submitted. The one-line guard and regression test are the narrow runtime shape, but default-on alerts contradict the current public config reference and need maintainer product approval plus docs alignment before merge.
