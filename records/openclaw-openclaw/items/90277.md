## Summary

Keep this issue open: current main has generic managed-env machinery, but the latest release and current main differ only by comments in the relevant files, and the source path still has a plausible managed-key provenance gap that can drop a `.env` value from the macOS service-env render.

## What This Changes

The issue reports that `openclaw gateway install --force` regenerates the macOS LaunchAgent service-env file without exporting `MINIMAX_API_KEY` even though the key is present in `.env` and tracked by `OPENCLAW_SERVICE_MANAGED_ENV_KEYS`.

## Best Possible Solution

Fix the generic service-env planning/render policy so every non-empty `.env` value in the managed-key set is rendered for macOS LaunchAgent service-env files, without MiniMax-specific special casing or a new config surface.

## Reproduction Assessment

No live high-confidence reproduction was run, but the report has concrete steps and the source path shows a plausible managed-key loss after the macOS render policy scrubs keys and only restores final `state-dotenv` entries. A focused test should reproduce the exact `MINIMAX_API_KEY` shape before patching.

## Solution Assessment

Yes, the best fix is a generic service-env planner/rendering correction that preserves managed `.env` values by normalized key; adding MiniMax-specific handling or asking users to manually edit service-env would be the wrong layer.
