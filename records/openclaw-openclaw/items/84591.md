## Summary

Keep open: current main has a generic reasoning-only retry path, but the reported DeepSeek shape can still bypass it when an earlier tool-use assistant message already produced visible text in the same attempt. The corrected issue evidence matches a final signed `thinking`-only `stop` message after a visible pre-tool message, and current tests/source explicitly treat “visible assistant text already exists” as a no-retry case.

## What This Changes

Detect DeepSeek V4 high-thinking final turns that contain only reasoning content and prevent a blank channel-visible reply, preferably by retrying for visible text rather than exposing hidden thinking.

## Best Possible Solution

Extend the existing reasoning-only recovery so it evaluates the final/current assistant message's visible text, not only aggregate earlier assistant text, then retry with the visible-answer continuation or surface a non-deliverable error without exposing hidden thinking blocks.

## Reproduction Assessment

Yes, at source level. A current-main attempt with a visible pre-tool assistant text plus a final signed thinking-only assistant stop matches the reporter's corrected evidence and is explicitly not retried by the current helper.

## Solution Assessment

The direct promotion fallback is not the best default because it risks exposing hidden reasoning. The narrower maintainable fix is to reuse the existing visible-answer retry path for final/current assistant messages that have no visible text, even when earlier assistant text exists in the attempt.
