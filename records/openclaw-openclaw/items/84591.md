## Summary

Keep open. Current main has a general reasoning-only retry path, but the reported shape can still bypass it because earlier visible pre-tool text makes `assistantTexts` non-empty while the final assistant message has no visible answer.

## What This Changes

The issue reports DeepSeek V4 high-thinking final turns producing only `reasoning_content`/thinking content after a visible tool-use prelude, leaving the channel with no final answer.

## Best Possible Solution

Extend the existing embedded-runner non-visible final-turn recovery so it evaluates the latest assistant message's visible text separately from earlier pre-tool progress text, then retries with the visible-answer instruction or surfaces the existing incomplete-turn error without exposing raw reasoning.

## Reproduction Assessment

Yes, at source level. The supplied logs describe the exact assistant-message sequence, and current main shows a path where prior `assistantTexts` can suppress the reasoning-only final-turn retry even when the latest assistant has no visible text.

## Solution Assessment

No, promoting DeepSeek `reasoning_content` directly to visible text is not the best default because reasoning is treated as internal. The safer fix is to refine the existing retry/incomplete-turn guard around the final assistant message and keep replay side-effect protections intact.
