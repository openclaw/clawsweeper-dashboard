## Summary

Keep open. Current main still has the reported ACP initialize validation path, and this PR is a useful narrow fix attempt, but the submitted guard does not match the ACP SDK's uint16 protocolVersion contract and needs porting, regression coverage, and changelog coverage before merge.

## What This Changes

The PR modifies `src/acp/server.ts` to wrap the ACP stdio stream and coerce non-numeric `initialize.params.protocolVersion` values to the ACP SDK `PROTOCOL_VERSION`.

## Best Possible Solution

Land or replace this PR with a current-main ACP stream-boundary compatibility layer that normalizes only SDK-invalid initialize protocol versions, preserves valid messages unchanged, and adds focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes, by source inspection. An `initialize` request with `protocolVersion: "2025-11-25"` reaches the ACP SDK's numeric `zInitializeRequest` validation before OpenClaw's translator can respond.

## Solution Assessment

No, not as submitted. A stream-boundary transform is the right narrow seam, but the predicate should match the SDK integer/range validation and the patch needs current-main tests and changelog coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the SDK protocolVersion contract:** `src/acp/server.ts:34`
