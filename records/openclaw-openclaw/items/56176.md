## Summary

Keep open. Current main still has the ACP initialize compatibility bug, and this PR targets the right stream-boundary seam, but the submitted guard still lets SDK-invalid numeric protocol versions through and the external real-behavior proof gate is unmet.

## What This Changes

The PR adds a `coerceAcpStream()` wrapper in `src/acp/server.ts` to coerce non-number initialize `protocolVersion` values to the ACP SDK `PROTOCOL_VERSION` before `AgentSideConnection`.

## Best Possible Solution

Refresh this as a current-main ACP stream-boundary normalizer before `AgentSideConnection` that coerces every SDK-invalid initialize `protocolVersion` value to `PROTOCOL_VERSION`, with focused tests and redacted VS Code/Cursor proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: VS Code/Cursor send `protocolVersion: "2025-11-25"`, current main passes initialize directly into SDK 0.21.0, and that SDK schema requires a numeric uint16 before `AcpGatewayAgent.initialize()` runs.

## Solution Assessment

No, not as submitted. A stream-boundary normalizer before `AgentSideConnection` is the right seam, but the predicate should match the SDK integer and 0..65535 contract and the PR still needs regression coverage plus real IDE proof.
