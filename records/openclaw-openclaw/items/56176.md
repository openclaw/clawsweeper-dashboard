## Summary

Keep open: current main still has the reported ACP initialize rejection path, and this PR is a plausible focused fix, but it needs a current-main refresh, a tighter SDK-contract predicate, focused regression coverage, and external real behavior proof before merge.

## What This Changes

The PR wraps the ACP server NDJSON stream so initialize requests with non-number `protocolVersion` values are rewritten to the SDK `PROTOCOL_VERSION` before `AgentSideConnection` validates them.

## Best Possible Solution

Normalize ACP initialize `protocolVersion` before SDK validation on current main, matching the SDK integer 0..65535 contract, with regression coverage and redacted VS Code or Cursor proof that the date-string handshake succeeds.

## Reproduction Assessment

Yes, by source inspection. Current main sends initialize params directly into SDK 0.21.0 validation, and that SDK requires an integer 0..65535 `protocolVersion`, so a VS Code/Cursor date string like `"2025-11-25"` fails before OpenClaw's initialize handler runs.

## Solution Assessment

No, not as submitted. A stream-boundary normalizer before `AgentSideConnection` is the right narrow seam, but the predicate should match the SDK range/integer contract and the PR still needs regression coverage plus real IDE proof.
