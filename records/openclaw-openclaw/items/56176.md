## Summary

Keep open. The linked ACP compatibility bug is still source-reproducible on current main, and this PR is the active closing implementation candidate, but the proposed shim is incomplete against the SDK uint16 contract and lacks after-fix real behavior proof.

## What This Changes

The PR wraps the ACP server's NDJSON stream to rewrite non-numeric initialize `protocolVersion` values to the ACP SDK `PROTOCOL_VERSION` before creating `AgentSideConnection`.

## Best Possible Solution

Rebase the ACP stream normalizer onto current main, match the SDK `zProtocolVersion` integer/range contract, add focused regression coverage for string and invalid numeric initialize values, and verify with VS Code or Cursor before merging.

## Reproduction Assessment

Yes, by source inspection: current main passes `ndJsonStream` directly to `AgentSideConnection`, and SDK 0.21.0 parses `initialize.protocolVersion` as a uint16 number before `AcpGatewayAgent.initialize()` runs. I did not run a live VS Code/Cursor session.

## Solution Assessment

No, not as submitted. A stream-boundary normalizer is the right narrow seam, but the predicate should match the SDK integer/range schema and the PR still needs regression coverage plus real after-fix proof.
