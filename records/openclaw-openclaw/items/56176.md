## Summary

Keep open. Current main still feeds the ACP SDK the raw initialize payload, so the linked VS Code/Cursor date-string bug remains real; this PR is useful source work but is not merge-ready because its predicate does not match the SDK's uint16 protocolVersion contract and the branch patch is stale against the current ACP server bootstrap.

## What This Changes

The PR wraps the ACP server SDK stream to coerce non-numeric `initialize.params.protocolVersion` values to the ACP SDK `PROTOCOL_VERSION` before `AgentSideConnection` validates requests.

## Best Possible Solution

Land a current-main ACP server compatibility fix at the `ndJsonStream` boundary that normalizes only invalid initialize `protocolVersion` values before SDK validation, preserves valid ACP numeric versions and all other messages, adds focused regression coverage for VS Code/Cursor-style date-string input, and records the user-facing fix in the changelog with contributor credit.

## Reproduction Assessment

Yes. The linked bug logs show VS Code/Cursor sending `protocolVersion: '2025-11-25'`, and current main plus the pinned SDK contract show that value reaches a numeric uint16 parser before OpenClaw's initialize handler runs.

## Solution Assessment

No, not as submitted. A stream-boundary transform is the right narrow seam, but the implementation should use the SDK-equivalent integer/range predicate, be ported to current main's server bootstrap, and include regression coverage plus a changelog entry.
