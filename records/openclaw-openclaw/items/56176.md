## Summary

Keep open: current main still passes the raw ACP stdio stream into the SDK, and this PR fixes the right boundary but is not merge-ready because its predicate is narrower than the pinned SDK validation contract and it lacks required coverage/changelog follow-through.

## What This Changes

The PR wraps the ACP server stdio stream to rewrite non-numeric `initialize.params.protocolVersion` values to the ACP SDK `PROTOCOL_VERSION` before constructing `AgentSideConnection`.

## Best Possible Solution

Land a current-main ACP stream-boundary normalizer that matches the SDK uint16 validation contract, preserves valid messages unchanged, and adds focused regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes, by source inspection. An `initialize` request with `protocolVersion: "2025-11-25"` reaches the ACP SDK's numeric `zInitializeRequest` validation before OpenClaw's `initialize()` handler can return the SDK protocol version.

## Solution Assessment

No, not as submitted. A stream-boundary transform is the narrow maintainable fix, but the predicate should match `z.number().int().gte(0).lte(65535)` and the PR needs current-main tests plus changelog coverage.
