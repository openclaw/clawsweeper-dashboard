## Summary

Keep open: this is a focused fix for a real media-delivery bug with supplied runtime log proof, but the patch needs one guard so media-only `text_end` replies are not replayed again at `message_end`.

## What This Changes

The PR flushes final streaming media directives, parses phase-aware final-answer text at `text_end`, keeps structured tool media queued for outbound delivery, and adds focused regression tests.

## Best Possible Solution

Land this after the final empty flush records media-only `text_end` delivery or otherwise prevents the later `message_end` fallback from resending the same directive, with a regression test covering `MEDIA:`-only text_end plus message_end.

## Reproduction Assessment

Yes. Current main has a clear source reproduction: a final `MEDIA:` line can remain in `pendingTail` until an empty final flush, and the existing text_end path returns before consuming it; the PR tests exercise that path, though I did not run them in this read-only review.

## Solution Assessment

Mostly yes. Flushing the directive accumulator and preserving structured tool media is the narrow right direction, but the patch also needs to mark media-only text_end delivery so message_end fallback cannot send the same attachment again.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid replaying media-only text_end replies at message_end:** `src/agents/pi-embedded-subscribe.ts:739-755`
  - body: The new final-empty flush can emit a media-only payload while leaving `lastBlockReplyText` unset. For an assistant answer that is only `MEDIA:/tmp/final.png`, the later `message_end` fallback sees a text_end channel with no delivered-text marker and calls `consumeReplyDirectives(text, { final: true })`, sending the same attachment again. Mark the final flush as delivered or have the message_end guard recognize media-only text_end delivery.
  - confidence: 0.82
