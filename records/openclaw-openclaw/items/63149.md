## Summary

Keep open. Current main has meaningful mitigations for heartbeat load, but the default path still does not fully close the reported multi-agent heartbeat fanout: subagent/nested lane deferral remains opt-in, and accepted heartbeat-spawned children still enter the shared subagent lane without heartbeat-origin admission policy.

## What This Changes

The issue reports Gateway Local CPU saturation and timeouts during five-agent heartbeat peaks and asks for gateway-level resource control or serialization of heartbeat-triggered subagent work.

## Best Possible Solution

Define a default-safe heartbeat resource policy that accounts for shared subagent and nested lane pressure around heartbeat-driven work, preserves normal interactive subagents, and backs it with a focused multi-agent heartbeat stress regression plus diagnostics.

## Reproduction Assessment

No. The issue has concrete setup steps and logs, and source inspection shows a plausible remaining default fanout path, but I did not live-run the five-agent macOS Local plus Feishu load scenario against current main.

## Solution Assessment

No. The suggested direction is plausible, but the narrow maintainable fix needs maintainer choice on default heartbeat backpressure versus opt-in skipWhenBusy and proof from a stress/regression test.
