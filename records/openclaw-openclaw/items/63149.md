## Summary

Keep open. Current main still has only partial mitigations for the reported heartbeat load spike: heartbeat scheduling is phased and lanes are capped, but subagent/nested lane deferral remains opt-in and accepted heartbeat-spawned children still enter shared Gateway lanes without heartbeat-origin admission policy.

## What This Changes

The issue reports macOS Local Gateway CPU saturation and timeout cascades when several heartbeat-enabled agents trigger concurrent heartbeat and subagent work.

## Best Possible Solution

Add or validate a default-safe heartbeat resource policy that accounts for shared subagent and nested lane pressure before and during heartbeat-driven work, while preserving interactive subagent behavior and adding focused multi-agent stress coverage.

## Reproduction Assessment

Unclear. The issue has concrete setup steps and logs, and source inspection shows a plausible remaining default heartbeat-spawned subagent pressure path, but this review did not live-run the five-agent macOS Local plus Feishu scenario.

## Solution Assessment

No. Current main’s phased scheduling, generic lane caps, and opt-in `skipWhenBusy` are useful mitigations, but they are not a complete default fix for heartbeat-origin fanout under shared Gateway load.
