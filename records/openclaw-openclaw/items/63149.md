## Summary

Keep open. Current main has useful mitigations for heartbeat load, including phased scheduling, generic lane caps, cron-lane deferral, per-session subagent child limits, and opt-in `heartbeat.skipWhenBusy`, but the reported default multi-agent heartbeat fanout is not fully closed: subagent/nested pressure is still only an opt-in heartbeat skip, and accepted heartbeat-spawned children still enter the shared subagent lane without heartbeat-origin admission.

## What This Changes

The issue requests a fix for macOS Local Gateway CPU saturation and timeout cascades when multiple heartbeat-enabled agents start concurrent heartbeat and subagent work under load.

## Best Possible Solution

Add or validate a default-safe heartbeat resource policy that accounts for shared subagent and nested lane pressure before and during heartbeat-driven work, preserves normal interactive subagent behavior, exposes diagnostics for heartbeat-origin backlog, and includes a focused multi-agent heartbeat stress or regression test.

## Reproduction Assessment

Unclear. The issue has concrete setup steps and logs, but this review did not live-run the five-agent macOS Local plus Feishu scenario; code inspection shows a plausible remaining path for default heartbeat-spawned subagent pressure.

## Solution Assessment

No. Current main’s phased scheduling, generic lane caps, and opt-in `skipWhenBusy` are useful mitigations, but they are not a complete default fix for heartbeat-origin fanout under shared Gateway load.
