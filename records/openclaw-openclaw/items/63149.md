## Summary

Keep this issue open. Current main has useful generic controls, including lower default agent/subagent lane concurrency, per-session child caps, and the newer opt-in heartbeat `skipWhenBusy` mitigation, but I did not find a default heartbeat-origin admission policy that prevents several heartbeat-enabled agents from starting turns and spawning shared-lane subagents during the same load spike.

## What This Changes

The report asks for Gateway scheduling and backpressure so multi-agent heartbeat-driven subagent fanout does not saturate one CPU core or delay notifications under high load.

## Best Possible Solution

Add or validate a default-safe heartbeat resource policy that accounts for shared subagent/nested lane pressure before and during heartbeat-driven work, preserves normal interactive subagent behavior, exposes enough diagnostics to identify heartbeat-origin backlog, and includes a focused regression or stress-style test for multiple heartbeat-enabled agents spawning background subagents while channel responsiveness remains bounded.

## Reproduction Assessment

Unclear. The issue gives concrete steps and logs, but this review did not run the five-agent macOS Local plus Feishu stress scenario; code inspection only establishes a plausible remaining path.

## Solution Assessment

No. The current opt-in `skipWhenBusy` path is a useful mitigation, but it is not the narrowest complete fix for the reported default multi-agent heartbeat fanout because it does not create heartbeat-origin admission or spawn-time backpressure.
