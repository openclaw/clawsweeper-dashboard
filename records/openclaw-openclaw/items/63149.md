## Summary

Keep open. Current main has heartbeat-load mitigations, but it still has a source-visible multi-agent heartbeat fanout path: interval broadcasts await each agent sequentially, retryable busy skips can return before later agents run, and subagent/nested-lane deferral remains opt-in; an active open PR now appears to be the right implementation vehicle for this remaining work.

## What This Changes

Ensure the Gateway remains responsive under multi-agent heartbeat and subagent fanout instead of saturating a single CPU core and delaying notifications.

## Best Possible Solution

Review and land a narrowed, approved version of https://github.com/openclaw/openclaw/pull/80470 or equivalent code that makes multi-agent heartbeat dispatch independent, scopes busy checks per agent, and adds focused stress/regression proof before closing this issue.

## Reproduction Assessment

No. There is a clear current-main source path and related live proof from an open PR, but no high-confidence live reproduction of the exact five-agent macOS Local plus Feishu load scenario against current main.

## Solution Assessment

Unclear. The active PR is the strongest current fix path, but it changes heartbeat scheduling and busy-lane semantics, so maintainers should review that policy choice and require focused multi-agent load proof before treating the issue as solved.
