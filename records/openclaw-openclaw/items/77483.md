## Summary

Keep open: current main has partial doctor repair and official catalog metadata, but config validation still tells operators to remove missing configured plugins and package postinstall intentionally avoids installing plugin dependencies, so the direct package-upgrade outage and misleading UX are not fully resolved.

## What This Changes

Request a migration path, warning, or documentation for the 2026.4.x built-in channel to 2026.5.x external `@openclaw/*` plugin change so configured channels do not silently disappear after upgrade.

## Best Possible Solution

Keep channels externalized, but make missing official configured plugins impossible to misread: preserve config, point users to `openclaw doctor --fix` or `openclaw plugins install @openclaw/<id>`, and document the 5.x migration path.

## Reproduction Assessment

Yes for the misleading current-main behavior: source inspection shows a missing configured plugin still emits the stale/remove warning even when the official catalog knows the `@openclaw/<id>` package. I did not live-run the historical package upgrade in this read-only review.

## Solution Assessment

No, the current path is not sufficient: doctor repair is useful, but gateway/status/config validation should not tell operators to remove valid externalized channel config. The narrow maintainable fix is to improve the missing official plugin diagnostic and docs without adding automatic network installs to startup/postinstall.

## Repair Work Prompt

Fix the misleading migration warning for missing official externalized plugins. When `plugins.entries.<id>` or relevant channel-configured state refers to an official external plugin such as `whatsapp`, `discord`, or `bluebubbles` that is absent from the registry, do not call it stale config or tell the operator to remove it. Instead, preserve the entry and surface a clear repair hint such as `run openclaw doctor --fix` or `openclaw plugins install @openclaw/<package>` using the official external plugin/channel catalog. Keep the fix generic for official external catalog entries; do not hard-code only WhatsApp and do not add network installs to package postinstall or gateway startup. Add/update regression coverage around config validation/status or doctor preview as appropriate, update user-facing docs if the command text changes, and add a single-line changelog fix entry. Related refs: #77483, #77266, #77424. Validate with targeted tests for config plugin validation and missing configured plugin install repair.
