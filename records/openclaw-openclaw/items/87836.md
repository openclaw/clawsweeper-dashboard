## Summary

Keep open: the branch is useful but not merge-ready because `--webhook` can still be applied to main/systemEvent cron jobs whose current execution path does not produce the documented finished webhook payload, and the PR still lacks live Gateway webhook proof.

## What This Changes

The branch adds positional `cron add/create` schedule and prompt parsing, `--webhook` delivery on cron add/create/edit, cron delivery cleanup tests/docs, schedule identity widening, and formatting-only dependency-guard changes.

## Best Possible Solution

Resolve main-session webhook semantics before merge by either rejecting that combination or implementing real finished-output webhook dispatch, then add docs/tests and redacted live Gateway POST proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `cron create` or `cron edit` can store webhook delivery on a main/systemEvent job, while current main-session cron execution returns the system-event text as the run summary. I did not run a live Gateway reproduction in this read-only review.

## Solution Assessment

No. The isolated cron ergonomics are useful, but exposing main/systemEvent webhook delivery is not the narrowest maintainable solution until it is either rejected or backed by an actual finished-output delivery path.
