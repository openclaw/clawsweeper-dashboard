## Summary

Keep this PR open for maintainer review. The cron CLI ergonomics are useful, but the branch exposes webhook delivery for main-session cron jobs while current main-session execution only enqueues a system event/heartbeat and never posts a finished webhook payload; the PR also still lacks live webhook proof.

## What This Changes

The branch adds positional cron add/create parsing, `--webhook` support for cron add/create/edit, webhook delivery cleanup, cron schedule identity type widening, docs/tests, and formatting-only dependency-guard changes.

## Best Possible Solution

Align the new CLI and docs with runtime semantics: either implement finished-payload webhook dispatch for main-session cron jobs or reject webhook for main/systemEvent create/edit, then add redacted live Gateway webhook proof.

## Reproduction Assessment

Yes from source inspection: create a main `systemEvent` cron job with `--webhook` or edit an existing main job with `--webhook`, then run it; current main-session execution returns without any webhook delivery path.

## Solution Assessment

No. The PR should not expose or document webhook delivery for main-session cron jobs until runtime either supports that finished-payload POST path or rejects the unsupported combination with tests and docs.
