## Summary

Keep this PR open for maintainer review. The ergonomics are useful, but the branch still exposes `--webhook` for main-session/systemEvent cron jobs even though the main-session execution path does not dispatch final webhook delivery, and the PR body still lacks live Gateway webhook proof.

## What This Changes

Adds positional cron add/create schedule and prompt parsing, exposes `--webhook` on cron add/create/edit, clears delivery fields across webhook/chat mode changes, updates cron docs, and widens schedule identity input types.

## Best Possible Solution

Ship the CLI ergonomics only after webhook delivery is either limited to isolated agentTurn jobs or main-session jobs gain a tested final-output webhook dispatch path, with redacted live Gateway webhook proof.

## Reproduction Assessment

Yes. Source inspection shows `cron add/create --webhook --system-event` and `cron edit --webhook` can store webhook delivery for main jobs, while `executeMainSessionCronJob` only queues a system event and never calls webhook dispatch.

## Solution Assessment

Unclear as submitted. Adding a first-class `--webhook` CLI is a maintainable direction for isolated jobs, but main-session webhook semantics need either rejection or a real final-output dispatch path before this is the best solution.
