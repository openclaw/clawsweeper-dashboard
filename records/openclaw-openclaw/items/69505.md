## Summary

Keep open. Current main still lacks the requested `assistantPrefix` contract on cron payloads, direct agent runs, agent defaults, CLI/SDK surfaces, docs, and provider runtime forwarding; the existing outbound `messages.responsePrefix` and provider prefill cleanup helpers are adjacent but do not force the model's generated reply to begin with fixed text.

## What This Changes

Request to add an optional assistant-prefix field across cron `agentTurn` payloads, agent defaults, direct agent invocations, CLI/API surfaces, and provider runtimes so fixed-template replies can start with guaranteed text.

## Best Possible Solution

If maintainers accept the direction, add one backwards-compatible, typed assistant-prefix field that flows consistently from cron payloads, direct agent runs, SDK calls, and agent defaults into a provider-facing continuation hook; document unsupported-provider behavior and cover streaming/fallback delivery so fixed-template automations cannot publish preamble before the required first text.

## Reproduction Assessment

Yes for the missing contract: current main has no matching field, strict cron and agent schemas reject unknown properties, and the runtime has no parameter to forward into providers. The specific production WhatsApp/model incident from the report was not live-reproduced in this read-only review.

## Solution Assessment

Unclear until maintainer API review. The requested provider-prefill direction is plausible and `messages.responsePrefix` is not equivalent, but the exact schema name, default precedence, thinking interaction, and provider no-op semantics should be designed centrally before implementation.
