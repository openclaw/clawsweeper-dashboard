## Summary

Keep open. Current main still lacks the requested assistant-prefix contract across cron payloads, direct agent runs, agent defaults, CLI flags, runtime forwarding, docs, and provider adapters; the existing outbound `messages.responsePrefix` and provider prefill cleanup are adjacent but do not force the generated assistant text to start with fixed text.

## What This Changes

Add an optional assistant-prefix field to cron agent-turn payloads, agent defaults, and CLI/agent invocation surfaces so model replies can be forced to begin with fixed text.

## Best Possible Solution

If maintainers accept the direction, add one backwards-compatible typed field that flows from cron payloads, direct agent runs, SDK/config defaults, and CLI flags into a provider-facing continuation hook with documented fallback/no-op behavior.

## Reproduction Assessment

Yes for the missing contract: current strict schemas and typed payload/config surfaces have no accepted assistant-prefix field, and runtime forwarding has no value to pass into providers. The production WhatsApp/model fallback incident itself was not live-reproduced.

## Solution Assessment

Unclear until maintainer API review. Provider prefill is a plausible implementation strategy and outbound `messages.responsePrefix` is not equivalent, but the exact public field name, precedence rules, and provider behavior should be chosen centrally.
