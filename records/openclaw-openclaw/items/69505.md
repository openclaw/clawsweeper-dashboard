## Summary

Keep open. Current main still does not expose an assistant-prefix contract for cron payloads, direct agent calls, agent defaults, CLI flags, or provider-facing execution; adjacent outbound response prefixes and prefill-stripping code do not guarantee generated assistant text starts with a fixed string.

## What This Changes

Request adds an optional assistant-prefix field for cron agent turns, agent defaults, and CLI invocations so provider calls can force replies to begin with fixed text.

## Best Possible Solution

If maintainers accept the direction, add one additive typed field that flows from cron payloads, direct agent runs, SDK/config defaults, and CLI flags into a documented provider continuation hook with clear precedence and no-op behavior.

## Reproduction Assessment

Yes for the missing contract: current strict schemas and typed config/payload surfaces have no accepted assistant-prefix field, and runtime forwarding has no value to pass to providers. The reported production WhatsApp incident itself was not live-reproduced.

## Solution Assessment

Unclear until maintainer API review. Provider prefill is a plausible implementation strategy and outbound messages.responsePrefix is not equivalent, but field naming, precedence, thinking interaction, and provider fallback behavior should be chosen centrally.
