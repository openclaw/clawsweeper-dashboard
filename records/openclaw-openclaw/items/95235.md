## Summary

Keep open: the report describes a real Slack activation bug with source-backed behavior, and current main does not show a canonicalization fix for recording bot thread participation under the parent `thread_ts`.

## What This Changes

Slack thread follow-ups without a fresh mention can be dropped when bot participation is recorded under a child message timestamp instead of the canonical root `thread_ts`.

## Best Possible Solution

Normalize the Slack participation key at the Slack dispatch boundary, preferring the canonical root `message.thread_ts` for genuine thread replies, and add a regression that proves both the persisted key and the next unmentioned follow-up activation.

## Reproduction Assessment

Yes for a source-level reproduction: configure Slack with mention-required channels, `replyToModeByChatType.channel = "all"`, and `thread.requireExplicitMention = false`, then record participation under a child `ts` and observe that the later root `thread_ts` lookup misses. I did not run a live Slack or Vitest repro because this review was read-only.

## Solution Assessment

Yes, the narrowest fix is to make `recordSlackThreadParticipation` receive the same canonical thread key that `hasSlackThreadParticipationWithPersistence` later checks. Changing mention gating to search alternate child keys would preserve stale state semantics and risk broader activation behavior.
