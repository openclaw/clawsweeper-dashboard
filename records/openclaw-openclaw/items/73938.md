## Summary

This PR should stay open: current main still lacks HTTP-level timeout-budget threading for remote embedding batch requests, and this branch is a useful narrow port of https://github.com/openclaw/openclaw/pull/49981. It is not merge-ready because the proposed shared timeout resolver can turn an expired supplied budget into no timeout.

## What This Changes

The branch adds memory SDK timeout-budget helpers and threads `remote.batch.timeoutMinutes` through remote HTTP helpers plus OpenAI, Gemini, and Voyage batch upload/create/poll/error/download paths, with docs, changelog, and focused tests.

## Best Possible Solution

Keep the timeout-budget plumbing, but make explicitly supplied exhausted budgets fail closed before any request or retry can proceed without a timeout, then rebase and validate the provider matrix.

## Reproduction Assessment

Yes, from source inspection. Current main forwards the configured remote batch timeout only as provider execution data, not into memory remote HTTP calls; PR head can also be source-reproduced by passing a timeout function that returns 0 before `postJsonWithRetry` runs.

## Solution Assessment

No, not as-is. The implementation direction is the right owner-boundary fix, but the shared resolver/retry guard must distinguish an omitted timeout from an exhausted supplied budget and fail closed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Fail closed when supplied timeout budgets expire:** `packages/memory-host-sdk/src/host/remote-http.ts:19-20`
  - body: `resolveRemoteHttpTimeoutMs` maps a supplied timeout function or number that resolves to `0` or less to `undefined`. In this PR, `undefined` means no timeout, so an exhausted supplied budget can still start provider HTTP work through `postJsonWithRetry`/`postJson` without the configured batch budget.
  - confidence: 0.88
