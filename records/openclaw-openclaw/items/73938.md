## Summary

Keep open; the branch addresses a real current-main remote batch timeout gap, but review found a PR-introduced fail-open timeout path in the shared SDK helper, matching the existing Aisle and Codex security comments.

## What This Changes

The branch threads remote embedding batch timeout budgets through memory-host SDK HTTP helpers and OpenAI, Gemini, and Voyage provider batch upload/create/status/error-read/download paths, with docs, changelog, exports, and focused tests.

## Best Possible Solution

Keep the timeout-budget plumbing, but make explicitly supplied expired budgets fail closed before any request or retry proceeds without a timeout, and add regression coverage for zero/negative budget functions.

## Reproduction Assessment

Yes from source, though I did not execute a live repro in this read-only review. Current main passes the configured remote batch timeout into provider batch execution without forwarding it to upload/create/status/error-read/download HTTP calls, and PR head is source-reproducible by supplying `postJsonWithRetry` a timeout function that returns `0`.

## Solution Assessment

No, not as-is. The implementation direction is maintainable, but the shared resolver/retry path must distinguish an omitted timeout from an exhausted supplied timeout and fail closed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fail closed when supplied timeout budgets expire:** `packages/memory-host-sdk/src/host/remote-http.ts:18-20`
  - body: `resolveRemoteHttpTimeoutMs` maps a supplied timeout function or number that resolves to `0` or less to `undefined`. In this PR, `undefined` means no timeout: `postJsonWithRetry` can skip the budget-aware loop and `postJson` then calls remote HTTP without a timeout, so an exhausted supplied budget can still start an unbounded provider request.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces a shared fail-open timeout-budget path that can turn an exhausted supplied budget into an unbounded remote HTTP request.

Concerns:

- **[medium] Expired supplied budget can disable HTTP timeout:** `packages/memory-host-sdk/src/host/remote-http.ts:18`
  - body: A supplied timeout function returning `0` or a numeric value `<= 0` resolves to `undefined`; callers then pass no timeout to guarded fetch or fall back to generic retry, which can let stalled provider requests run without the configured batch budget.
  - confidence: 0.86
