## Summary

Keep open. Current main still lacks remote embedding batch HTTP timeout threading, so the PR addresses a real bug, but the proposed shared resolver can turn an explicitly exhausted timeout budget into no timeout and should not merge as-is.

## What This Changes

The PR adds shared remaining-time timeout budget helpers, threads them through memory-host SDK remote HTTP helpers and OpenAI/Gemini/Voyage batch phases, and updates docs, changelog, exports, and focused tests.

## Best Possible Solution

Keep the narrow budget-threading design and #49981 credit, but make explicitly supplied expired budgets fail closed before any request or retry proceeds without a timeout.

## Reproduction Assessment

Yes. Current-main source shows `remote.batch.timeoutMinutes` reaches provider batch execution but is not forwarded into upload/create/status/error-read/download HTTP calls; the PR-specific regression is reproducible by passing `postJsonWithRetry` a timeout function that returns `0`.

## Solution Assessment

No, not as-is. The implementation direction is maintainable, but the shared resolver/retry path should distinguish an omitted timeout from an exhausted supplied budget and fail closed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fail closed when supplied timeout budgets expire:** `packages/memory-host-sdk/src/host/remote-http.ts:18-20`
  - body: `resolveRemoteHttpTimeoutMs` maps a supplied timeout function returning `0` or a numeric `<= 0` value to `undefined`. `postJsonWithRetry` then skips the budget-aware retry path, and `postJson` passes no timeout to remote HTTP, so an exhausted remote batch budget can still issue an unbounded request instead of failing closed.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete fail-open timeout-budget path in shared remote HTTP helpers.

Concerns:

- **[medium] Expired supplied budget can disable HTTP timeout:** `packages/memory-host-sdk/src/host/remote-http.ts:18`
  - body: `resolveRemoteHttpTimeoutMs` maps non-finite or `<= 0` supplied values to `undefined`; downstream helpers interpret `undefined` as no timeout, so an exhausted remote batch budget can allow a stalled provider request to run unbounded.
  - confidence: 0.86
