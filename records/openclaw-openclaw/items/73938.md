## Summary

Keep open: current main still does not thread the configured remote embedding batch timeout into the shared memory-host HTTP helpers, but the proposed patch should not merge as-is because an explicitly supplied exhausted budget can be converted into an unbounded HTTP request.

## What This Changes

The PR threads the configured remote embedding batch timeout budget through memory-host SDK HTTP helpers and OpenAI, Gemini, and Voyage batch upload, create, poll, error-read, retry, and download paths, with docs, changelog, SDK exports, and focused tests.

## Best Possible Solution

Keep the narrow remote-batch timeout threading direction, but make the shared resolver and retry path fail closed when a supplied budget is exhausted, then land this PR or an equivalent replacement after focused SDK/provider regression tests and the changed gate pass.

## Reproduction Assessment

Yes. Current main can be verified by source inspection: the configured batch timeout reaches provider `batchEmbed`, but upload/create/status/error-read/download HTTP calls do not receive it; the PR regression is reproducible by passing a timeout function returning `0` to `postJsonWithRetry`.

## Solution Assessment

No, not as-is. The implementation direction is the narrow maintainable fix, but converting an explicitly supplied exhausted budget to `undefined` is not the best fail-closed behavior for shared guarded HTTP calls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fail closed when a supplied timeout budget is exhausted:** `packages/memory-host-sdk/src/host/remote-http.ts:18-19`
  - body: `resolveRemoteHttpTimeoutMs` returns `undefined` for `<= 0` even when the caller supplied a budget. In `postJsonWithRetry`, that can skip the budget-aware retry loop and fall through to `retryAsync`; `postJson` then forwards no timeout to guarded fetch, so an expired remote batch budget can hang instead of failing closed.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete fail-open timeout-budget path in shared remote HTTP helpers.

Concerns:

- **[medium] Expired supplied budget can disable HTTP timeout:** `packages/memory-host-sdk/src/host/remote-http.ts:18`
  - body: `resolveRemoteHttpTimeoutMs` maps non-finite or `<= 0` supplied values to `undefined`; downstream helpers treat `undefined` as no timeout, so an exhausted remote batch budget can allow a stalled provider request to run unbounded.
  - confidence: 0.86
