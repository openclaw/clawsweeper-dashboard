## Summary

Keep open, but do not merge as-is. Current main still has the reported remote batch timeout propagation gap, and this PR is the right narrow area, but the patch introduces a fail-open path where an explicitly supplied expired timeout budget can become an unbounded HTTP request in the shared memory-host SDK helper.

## What This Changes

The PR adds a shared embedding batch timeout budget, threads it through memory-host SDK remote HTTP helpers and OpenAI/Gemini/Voyage batch phases, and adds docs, changelog, SDK exports, and focused tests.

## Best Possible Solution

Keep the narrow budget-threading design and #49981 credit, but make the shared resolver/retry path distinguish omitted timeout from an exhausted supplied budget and fail closed before any request or retry proceeds without a timeout.

## Reproduction Assessment

Yes. Source inspection on current main shows the configured batch timeout reaches provider batch execution but is not forwarded into upload/create/status/error-read/download HTTP calls; the PR-specific regression is reproducible by passing `postJsonWithRetry` a timeout function that returns `0`.

## Solution Assessment

No, not as-is. The implementation direction is maintainable, but the shared timeout resolver should throw or otherwise fail closed for an explicitly supplied expired budget instead of returning `undefined` and disabling the fetch timeout.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Fail closed when supplied timeout budgets expire:** `packages/memory-host-sdk/src/host/remote-http.ts:18-20`
