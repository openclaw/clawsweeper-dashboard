## Summary

Keep open: the PR addresses a real current-main remote batch timeout gap, but it should not merge as-is because an explicitly supplied expired timeout can be resolved to `undefined` and then treated as no HTTP timeout.

## What This Changes

The PR adds shared remote embedding batch timeout budget helpers, threads them through memory-host SDK HTTP helpers and OpenAI/Gemini/Voyage batch phases, and adds docs, changelog, SDK exports, and focused tests.

## Best Possible Solution

Keep the narrow budget-threading design and #49981 credit, but distinguish omitted timeouts from exhausted supplied budgets and fail closed before any request or retry proceeds without a timeout.

## Reproduction Assessment

Yes. Current-main source shows the configured batch timeout reaches provider batch execution but not upload/create/status/error-read/download HTTP calls; the PR-specific regression is reproducible by passing `postJsonWithRetry` a timeout function that returns `0`.

## Solution Assessment

No, not as-is. The implementation direction is the narrow maintainable fix, but the shared resolver/retry path should treat an exhausted supplied budget as a terminal timeout instead of disabling the timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fail closed when supplied timeout budgets expire:** `packages/memory-host-sdk/src/host/remote-http.ts:18-20`
  - body: `resolveRemoteHttpTimeoutMs` maps a supplied timeout function or numeric timeout resolving to `<= 0` to `undefined`. `postJsonWithRetry` can then skip the budget-aware retry path and `postJson` forwards no timeout, so an exhausted batch budget can still issue an unbounded HTTP request instead of failing fast.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete fail-open timeout-budget path in shared remote HTTP helpers.

Concerns:

- **[medium] Expired supplied budget can disable HTTP timeout:** `packages/memory-host-sdk/src/host/remote-http.ts:18`
  - body: `resolveRemoteHttpTimeoutMs` maps non-finite or `<= 0` supplied values to `undefined`; downstream helpers interpret `undefined` as no timeout, so an exhausted remote batch budget can allow a stalled provider request to run unbounded.
