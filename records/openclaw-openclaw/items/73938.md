## Summary

Keep this PR open: the remote batch timeout fix is useful and is not on current main, but the proposed helper can turn an exhausted supplied timeout budget into no timeout and let provider HTTP work proceed unbounded. The earlier source work in https://github.com/openclaw/openclaw/pull/49981 is relevant context, but this branch still needs the fail-closed timeout semantics repaired before merge.

## What This Changes

The PR threads remote embedding batch timeout budgets through memory-host SDK HTTP helpers and OpenAI, Gemini, and Voyage batch operations, with docs, changelog, and focused tests.

## Best Possible Solution

Keep the timeout-budget plumbing, but make explicitly supplied exhausted budgets fail closed before any request or retry can proceed without a timeout, with zero and negative budget regression coverage.

## Reproduction Assessment

Yes from source. Current main does not forward remote batch timeout budgets into the memory HTTP transport, and PR head can be exercised by passing a supplied timeout function that returns 0 before postJsonWithRetry runs.

## Solution Assessment

No, not as-is. The implementation direction is maintainable, but the shared timeout resolver and retry guard must distinguish an omitted timeout from an exhausted supplied budget and fail closed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Fail closed when supplied timeout budgets expire:** `packages/memory-host-sdk/src/host/remote-http.ts:19-20`
  - body: `resolveRemoteHttpTimeoutMs` maps a supplied timeout function or number that resolves to `0` or less to `undefined`. In this PR, `undefined` means no timeout, so an exhausted supplied budget can still start provider HTTP work through `postJsonWithRetry`/`postJson` without the configured batch budget.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces a fail-open timeout-budget path that can disable remote HTTP request timeouts when a supplied budget is exhausted.

Concerns:

- **[medium] Expired supplied budget can disable HTTP timeout:** `packages/memory-host-sdk/src/host/remote-http.ts:19`
  - body: A supplied timeout function returning 0, or a numeric value <= 0, resolves to undefined; callers then pass no timeout to guarded fetch or fall back to generic retry, which can let stalled provider requests run without the configured batch budget.
