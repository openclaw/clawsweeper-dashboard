## Summary

Keep open. Current main still does not thread the configured remote embedding batch timeout into upload/create/status/error-read/download HTTP calls, and this PR is a relevant narrow replacement for #49981, but the proposed shared timeout resolver can fail open when an explicitly supplied budget is exhausted.

## What This Changes

The PR adds a shared embedding batch timeout budget, threads it through memory-host SDK HTTP helpers and OpenAI/Gemini/Voyage batch phases, and updates docs, changelog, SDK exports, and focused tests.

## Best Possible Solution

Keep the narrow budget-threading design and #49981 credit, but make the shared resolver/retry path distinguish an omitted timeout from an exhausted supplied budget and fail closed before any request or retry proceeds without a timeout.

## Reproduction Assessment

Yes. Source inspection on current main shows the configured batch timeout reaches provider batch execution but is not forwarded into upload/create/status/error-read/download HTTP calls; the PR-specific regression is reproducible by passing `postJsonWithRetry` a supplied timeout function that returns `0`.

## Solution Assessment

No, not as-is. The implementation direction is maintainable, but the shared timeout resolver should fail closed for an explicitly supplied expired budget instead of returning `undefined` and disabling the fetch timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fail closed when supplied timeout budgets expire:** `packages/memory-host-sdk/src/host/remote-http.ts:18-20`
  - body: `resolveRemoteHttpTimeoutMs` maps a supplied timeout function returning `0` or a numeric `<= 0` value to `undefined`. `postJsonWithRetry` can then skip the budget-aware retry path, and `postJson` passes no timeout to the guarded fetch layer, so an exhausted remote batch budget can still issue an unbounded HTTP request instead of failing closed.
  - confidence: 0.87
