## Summary

Keep open: the PR targets a valid false-live session lock bug and has the right repair boundary, but the new argv classifier is too broad and can still preserve unrelated Node processes as OpenClaw owners.

## What This Changes

The PR adds owner argv inspection to session lock cleanup and doctor so fresh live-PID locks with readable non-OpenClaw owners can be reported and removed, with tests, docs, and a changelog entry.

## Best Possible Solution

Land a corrected version that uses a strong OpenClaw-specific owner classifier, preserves unreadable or unknown owners, and then closes the linked issue after merge.

## Reproduction Assessment

Yes. Source inspection shows current main leaves fresh live-PID locks non-stale, and the PR classifier can be checked with a live PID whose argv is `node /srv/app/dist/index.js`, which it would incorrectly treat as an OpenClaw owner.

## Solution Assessment

No. Owner argv inspection is the right narrow direction, but the current classifier should require a strong OpenClaw marker, similar to the existing `isGatewayArgv` gateway-token gate, before accepting generic entrypoint names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Require an OpenClaw-specific argv marker:** `src/agents/session-write-lock.ts:395-407`
  - body: The new classifier treats any process with an arg ending in `dist/index.js`, `dist/entry.js`, `openclaw.mjs`, or similar as OpenClaw. Please require a stronger OpenClaw-specific marker and add a regression case for a generic `dist/index.js` non-owner.
