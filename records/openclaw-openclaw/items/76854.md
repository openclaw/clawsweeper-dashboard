## Summary

Keep open: current main still matches the linked false-live session-lock bug, but this PR head can still preserve unrelated live processes as OpenClaw owners when their argv ends in a common entrypoint filename.

## What This Changes

The PR adds owner-argv inspection so session lock cleanup and doctor can remove fresh live locks owned by non-OpenClaw processes, with docs, changelog, and regression coverage.

## Best Possible Solution

Land a corrected owner-argv cleanup that only preserves readable live owners with a strong OpenClaw marker and keeps unreadable or unknown owners protected.

## Reproduction Assessment

Yes, source-level. Current main leaves fresh live reused-PID locks as stale=no, and the PR blocker is reproducible from the classifier because generic entrypoints like node /srv/app/dist/index.js are treated as OpenClaw.

## Solution Assessment

No. Owner-argv inspection is the right narrow fix direction, but this PR should require an OpenClaw-specific argv marker before preserving a fresh live-PID lock.

Update this PR to tighten session lock owner argv classification. In src/agents/session-write-lock.ts, preserve fresh live-PID locks only when readable argv contains a strong OpenClaw-specific marker; do not classify generic entrypoints such as node /srv/app/dist/index.js, dist/entry.js, src/index.ts, or scripts/run-node.mjs as OpenClaw on filename alone. Prefer reusing or aligning with the existing src/infra/gateway-process-argv.ts gateway-token precedent where practical, while keeping unknown or unreadable owners preserved and keeping active OpenClaw owners protected. Add regression coverage for a generic dist/index.js non-owner and keep the existing doctor/session-lock docs and changelog scope intact. Do not broaden cleanup to delete all locks merely because Gateway is down.
