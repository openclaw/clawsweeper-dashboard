## Summary

Keep open: the PR addresses the linked false-live session-lock bug in the right conservative direction, but the new owner-argv classifier misses supported `.mjs` OpenClaw entrypoints and can therefore remove a fresh lock held by an active OpenClaw process launched through those entries.

## What This Changes

The PR adds owner-argv inspection to session lock cleanup so doctor/startup can remove fresh live-PID locks proven to belong to non-OpenClaw processes, with tests, docs, and changelog coverage.

## Best Possible Solution

Land the conservative owner-argv cleanup after aligning the OpenClaw owner classifier with the repo’s supported gateway/CLI entrypoints and adding regression coverage for `.mjs` owners.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a fresh lock with a live PID and non-OpenClaw argv stays non-stale on current main, while the PR can misclassify `node dist/index.mjs gateway` as non-OpenClaw and remove it.

## Solution Assessment

No, not as-is. The overall approach is the narrow maintainable fix, but it should share or match the supported OpenClaw entrypoint detection before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Recognize supported .mjs OpenClaw entrypoints:** `src/agents/session-write-lock.ts:396-397`
  - body: This whitelist omits `dist/index.mjs` and `dist/entry.mjs`, even though the daemon/service helpers support those as OpenClaw gateway entrypoints. A live OpenClaw process launched from one of those entries will be classified as non-OpenClaw, so `doctor --fix` or startup cleanup can remove its fresh session lock while the writer is still active. Include the `.mjs` entries or share the existing entrypoint helper, and add coverage for that argv shape.
