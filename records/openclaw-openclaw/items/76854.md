## Summary

Keep open: the PR targets a real source-reproducible bug and is paired with open issue #76823, but the current patch still treats generic Node entrypoint filenames as OpenClaw owners, leaving part of the false-live cleanup unresolved.

## What This Changes

The PR adds owner argv inspection to session lock cleanup and doctor so fresh live-PID locks can be removed when the owning process is readable and non-OpenClaw, with tests, docs, and a changelog entry.

## Best Possible Solution

Land a corrected version that uses a strong OpenClaw-specific owner classifier, keeps unknown or unreadable owners preserved, and closes the linked issue after merge.

## Reproduction Assessment

Yes. Source inspection of current main shows a fresh lock with a live reused PID is not stale unless the PID is dead, recycled by starttime, malformed, or too old, and the linked issue's stale=no output matches that path.

## Solution Assessment

No. The repair direction is narrow and appropriate, but the PR needs a stricter OpenClaw-specific argv marker before merge so common non-OpenClaw Node entrypoints do not bypass cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

- **[P2] Require an OpenClaw-specific argv marker:** `src/agents/session-write-lock.ts:395-407`
  - body: The new classifier treats any process with an arg ending in dist/index.js, dist/entry.js, openclaw.mjs, or similar as OpenClaw.
