## Summary

Keep open: the linked bug is source-reproducible on current main and this PR targets the right cleanup path, but the proposed owner classifier is too broad and can still preserve unrelated live-PID locks.

## What This Changes

The PR adds argv-based owner inspection to session lock cleanup and doctor so fresh live-PID locks can be removed when the process is readable and non-OpenClaw, with tests, docs, and a changelog entry.

## Best Possible Solution

Land a corrected version that uses a strong OpenClaw-specific owner classifier, preserves active and unknown owners, and closes #76823 after merge.

## Reproduction Assessment

Yes. Source inspection shows current main leaves a fresh live-PID lock at stale=no unless the PID is dead, recycled by starttime, malformed, or too old; the linked issue's doctor output follows that path.

## Solution Assessment

No. The argv-inspection approach is the right narrow direction, but the proposed OpenClaw-owner test must require an OpenClaw-specific marker before preserving a fresh live lock.

Keep scope to session lock cleanup, doctor integration/tests, docs, and the existing changelog entry; do not add config or delete locks when owner argv cannot be read.
