## Summary

Keep open. Current main and the latest release do not contain the affected emotion-tag parser or `/emotions` command, while the ambiguity remains in the open emotion-mode PR stack and needs a maintainer semantics decision before that stack lands.

## What This Changes

The issue asks maintainers to resolve how the proposed emotion-tag stripper should treat line-start bracket pairs such as `[fast][1]` and `[fast][clear]` when they may be Markdown reference links.

## Best Possible Solution

Keep this as a PR-stack review tracker. Before merging #72229, define the line-start rule explicitly and add tests for numeric references, definition-backed named references, named references without definitions, and adjacent emotion pairs.

## Reproduction Assessment

No high-confidence reproduction exists on current main because the affected parser and command are absent. There is a high-confidence branch-level reproduction by source inspection in the #72229 context: line-start `[fast][1]` is not protected as a Markdown reference when the check requires non-whitespace earlier on the same line.

## Solution Assessment

Unclear until maintainers choose the product semantics. The issue correctly rejects a label-only heuristic; the maintainable path is either definition-aware Markdown reference detection or an explicit rule that line-start allowlisted pairs remain emotion tags when no definition exists.
