## Summary

Keep open. Current main and the latest release still do not contain the emotion-tag parser or `/emotions` surface, while the ambiguity is present in the open emotion-mode PR stack and needs a maintainer semantics decision before that stack lands.

## What This Changes

The issue asks maintainers to resolve how line-start bracket pairs such as `[fast][1]` should be classified when emotion-tag stripping overlaps with Markdown reference-link syntax.

## Best Possible Solution

Keep this as a PR-stack review tracker. Before merging emotion mode, maintainers should define the line-start rule and update #72229 with parser behavior and tests for numeric references, definition-backed named references, named references without definitions, and adjacent emotion pairs.

## Reproduction Assessment

No high-confidence reproduction exists on current main because the affected parser and command are absent. There is a high-confidence branch reproduction by source inspection on #72229 head: line-start bracket pairs fail Markdown-reference protection because the guard returns false at index 0.

## Solution Assessment

Unclear until maintainers choose the semantics. A label-only heuristic is not enough; the maintainable fix is either definition-aware Markdown reference detection or an explicit rule that allowlisted line-start pairs remain emotion tags when no definition exists.
