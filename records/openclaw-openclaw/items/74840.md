## Summary

Keep open. The linked bug is real and current main still routes all-error cron payloads through fatal structured-error delivery, but this PR only changes fallback text selection and therefore does not fix the fatal-error precedence that makes recovered assistant replies lose to raw tool errors. The existing ClawSweeper review comment is consistent with the current source and test evidence.

## What This Changes

The branch changes `resolveCronPayloadOutcome` so normalized `finalAssistantVisibleText` is considered as a fallback for cron summary and output text before falling back to the first payload text.

## Best Possible Solution

Repair the cron outcome boundary so final assistant-visible text wins only for a recovered or self-corrected mutating-tool failure, while unresolved fatal tool/run errors continue to report failure. Cover both recovered and unresolved paths with focused cron helper and embedded-runner regression tests, and add the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is `resolveCronPayloadOutcome` with only `isError: true` payloads, `finalAssistantVisibleText`, and `preferFinalAssistantVisibleText: true`; current main's helper test pins that shape to return the raw error. The embedded-runner payload test also shows the assistant-text-then-mutating-tool-error payload sequence that can feed this cron path.

## Solution Assessment

No. This PR is not the best fix because it adjusts fallback summary/output text after helpers and fatal classification have already selected the error path; the safer solution is to classify recovered versus unresolved mutating failures before fatal delivery text overrides the outcome.
