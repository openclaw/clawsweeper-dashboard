## Summary

Keep open. The PR is a useful narrow attempt, but it does not fix the linked cron-summary bug because the existing payload fallback still returns error text and the fatal structured-error path still overrides delivery with the last error.

## What This Changes

The PR changes `resolveCronPayloadOutcome` to try `finalAssistantVisibleText` as a fallback summary/output source before falling back to the first payload text.

## Best Possible Solution

Repair the cron outcome boundary so recovered assistant-visible text wins only when the tool failure is proven recoverable, while fatal error classes continue to report failure, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is calling `resolveCronPayloadOutcome` with only `isError: true` payloads, `finalAssistantVisibleText: "Recovered final answer"`, and `preferFinalAssistantVisibleText: true`; current main pins that shape to return the last error.

## Solution Assessment

No. Adding final assistant text after the existing payload fallback is not sufficient because those helpers already return error text for all-error payloads and fatal delivery still overrides the result; the safer fix is to classify recovered versus fatal failures before choosing delivery text.
