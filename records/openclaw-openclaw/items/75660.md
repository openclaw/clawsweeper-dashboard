## Summary

Current main and the shipped v2026.4.29 release cover the reported Discord inbound dead-socket failure mode. The Discord gateway now detects missed heartbeat ACKs and schedules reconnects, records close/reconnect/not-ready states into channel status, publishes WebSocket transport activity, and the gateway health monitor restarts stale connected channel accounts; the report is against 2026.4.15, which predates the current Discord gateway implementation.

## What This Changes

The issue requests Discord inbound WebSocket liveness recovery so silently dropped gateway sockets log, reconnect, and resume delivering @mentions without a full gateway restart.

## Best Possible Solution

Keep the shipped Discord gateway heartbeat/reconnect path and provider-proven transport health monitor as the maintained recovery mechanism; reporters still seeing this on v2026.4.29 or newer should file a fresh issue with a diagnostics export and logs around heartbeat timeout, reconnect, and health-monitor status.

## Reproduction Assessment

No live residential-NAT reproduction was run for the original 2026.4.15 report. Current-main source and regression tests give a high-confidence verification path for the expected recovery behavior: missed heartbeat ACKs reconnect the gateway, and stale transport activity restarts the channel account.

## Solution Assessment

Yes. The current implementation addresses the narrow failure at the transport/lifecycle layers instead of adding an external cron watchdog: heartbeat ACK timeout handles dead sockets promptly, while the generic health monitor provides bounded restart recovery for stale connected accounts.
