## Summary

Current main and the latest release already cover the reported Discord receive-side dead-socket failure: Discord gateway frames now publish transport liveness, heartbeat ACK failures trigger reconnect, gateway close/reconnect states are logged into channel status, and the generic health monitor restarts stale connected channels instead of requiring a manual gateway restart.

## What This Changes

The issue asks OpenClaw to detect and recover when the Discord gateway WebSocket stops delivering inbound messages while the gateway process and outbound Discord REST sends still work.

## Best Possible Solution

Keep the shipped Discord transport-liveness and generic channel-health recovery path, and point affected users on 2026.4.15 to upgrade to v2026.4.29 or later; use the documented health-monitor thresholds for more aggressive local recovery instead of adding another Discord-specific watchdog.

## Reproduction Assessment

Yes for a source-level verification path: v2026.4.15 lacks the transport-liveness symbols, while current main and v2026.4.29 include transport activity, heartbeat ACK timeout, stale-socket policy, and regression tests. I did not run a live residential-NAT Discord drop in this read-only pass.

## Solution Assessment

Yes. The current approach is the narrow maintainable fix: Discord publishes provider-owned transport liveness, the generic gateway health monitor handles restart policy, and recovery timing stays configurable rather than hard-coded into the Discord plugin.
