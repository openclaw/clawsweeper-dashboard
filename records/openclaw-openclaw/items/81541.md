## Summary

Keep open: the bug fix is plausible and well-scoped, but the current PR still has two P2 correctness blockers around custom ACP adapter config updates and symlink-only Codex auth bridging.

## What This Changes

The branch filters ACPX config-option calls by adapter, maps Claude thinking keys to effort, symlinks Codex auth into isolated CODEX_HOME, and preserves ACP metadata across stale session-store writes.

## Best Possible Solution

Land a focused ACPX fix that remains capability-aware for custom adapters, keeps Codex/Claude normalization, bridges Codex auth with a cross-platform fallback, and coordinates ACP metadata preservation with the narrower session-store PR.

## Reproduction Assessment

Yes. Current main source shows unsupported ACP config keys can still reach adapters, Codex ACP uses an isolated home without auth.json, and stale session-store writes can drop disk ACP metadata; the contributor also supplied copied live ACP fresh/resume output after the patch.

## Solution Assessment

No, not yet. The patch is in the right files, but it should not silently report success for custom adapter config updates and should not make ACPX registration depend on file-symlink privileges.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve config updates for custom ACP adapters:** `extensions/acpx/src/runtime.ts:980-982`
  - body: The final branch only forwards `setConfigOption` for Codex, Claude, or the OpenClaw bridge. Custom ACP agents can legitimately support `session/set_config_option`, and the manager persists runtime options after this call returns, so `/acp set ...` can now report success while the adapter never received the config update. Make this capability-aware or return an unsupported-control error instead of silently no-oping unknown commands.
  - confidence: 0.86
- **[P2] Avoid requiring symlink privileges for Codex auth:** `extensions/acpx/src/codex-auth-bridge.ts:371`
  - body: When source `auth.json` exists, setup now awaits `fs.symlink` and lets `EPERM`/`EACCES` fail the whole ACPX registration. File symlinks require special privileges on Windows, so non-admin Windows users with Codex auth could lose the ACPX plugin before any session starts. Add a copy, hardlink, or permission-error fallback and cover it in the auth bridge tests.
  - confidence: 0.82
