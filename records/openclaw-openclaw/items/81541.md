## Summary

Keep open: the PR targets real ACPX adapter/auth/session-store bugs and the copied live output now satisfies the real-behavior proof gate, but the patch has blocking correctness issues around custom adapter config no-ops and symlink-only Codex auth bridging.

## What This Changes

The branch filters ACPX runtime config calls by adapter, maps Claude thinking controls to `effort`, symlinks Codex auth into isolated `CODEX_HOME`, and preserves ACP metadata across stale session-store writes.

## Best Possible Solution

Land a focused ACPX fix that remains capability-aware for custom adapters, keeps Codex/Claude adapter normalization, preserves ACP session metadata from disk snapshots, and bridges Codex auth with a cross-platform fallback that does not make plugin startup depend on file-symlink privileges.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main can forward unsupported ACP config keys and isolates Codex ACP from existing `auth.json`; the contributor also supplied copied fresh/resume live output for Claude, Codex, Gemini-like ACP, and metadata resume failures.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the fix is in the right ACPX/auth/session-store area, but it should not silently drop config updates for custom adapters and should not require file-symlink privileges for Codex auth bridging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve config updates for custom ACP adapters:** `extensions/acpx/src/runtime.ts:980-982`
  - body: The new final branch only forwards `setConfigOption` for Codex, Claude, or the OpenClaw bridge. Custom ACP agents can legitimately support `session/set_config_option`, and the manager persists runtime options after this call returns, so `/acp set ...` can now report success while the adapter never receives the config update. Make this capability-aware or return an unsupported-control error instead of silently no-oping unknown commands.
  - confidence: 0.86
- **[P2] Avoid requiring symlink privileges for Codex auth:** `extensions/acpx/src/codex-auth-bridge.ts:371`
  - body: When source `auth.json` exists, setup now awaits `fs.symlink` and lets `EPERM`/`EACCES` fail the whole ACPX registration. File symlinks require special privileges on Windows, and this would make non-admin Windows users with Codex auth lose the ACPX plugin before any session starts. Add a copy, hardlink, or permission-error fallback and cover it in the auth bridge tests.
  - confidence: 0.82
