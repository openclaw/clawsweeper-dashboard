## Summary

Keep open: the branch targets real ACPX failures and the supplied live proof is sufficient, but it still introduces two blocking correctness issues around custom adapter config updates and symlink-only Codex auth bridging.

## What This Changes

The PR filters ACPX config option forwarding by adapter, maps Claude thinking keys to `effort`, symlinks Codex auth into isolated `CODEX_HOME`, and preserves ACP metadata across stale session-store writes.

## Best Possible Solution

Land a focused ACPX fix that uses advertised runtime capabilities for config controls, keeps Codex/Claude normalization, preserves ACP metadata, and bridges Codex auth with a cross-platform fallback.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and contributor live output: current main can forward unsupported ACP config keys and isolates Codex ACP from existing auth, while the PR proof shows fresh/resume checks for Claude, Codex, Gemini-like ACP, and the metadata-resume case.

## Solution Assessment

Is this the best way to solve the issue? No, not yet: the patch is in the right ACPX/auth/session-store areas, but it should not silently drop custom adapter config updates and should not require file-symlink privileges for Codex auth bridging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve config updates for custom ACP adapters:** `extensions/acpx/src/runtime.ts:980-982`
  - body: The new final branch only forwards `setConfigOption` for Codex, Claude, or the OpenClaw bridge. Custom ACP agents can legitimately support `session/set_config_option`, and the manager persists runtime options after this call returns, so `/acp set ...` can now report success while the adapter never receives the config update. Make this capability-aware or return an unsupported-control error instead of silently no-oping unknown commands.
  - confidence: 0.86
- **[P2] Avoid requiring symlink privileges for Codex auth:** `extensions/acpx/src/codex-auth-bridge.ts:373`
  - body: When source `auth.json` exists, setup now awaits `fs.symlink` and lets `EPERM` or `EACCES` fail the whole ACPX registration. File symlinks require special privileges on Windows, so non-admin Windows users with Codex auth could lose the ACPX plugin before any session starts. Add a copy, hardlink, or permission-error fallback and cover it in the auth bridge tests.
  - confidence: 0.82
