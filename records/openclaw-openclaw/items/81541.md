## Summary

Keep this PR open: it targets real ACPX failures and includes useful live proof, but the current patch is not mergeable because it silently drops custom-adapter config updates, depends on symlink-only Codex auth bridging, and breaks the session-store writer cache invariant in CI.

## What This Changes

The PR filters ACPX runtime config calls by adapter, maps Claude thinking controls to `effort`, bridges Codex ACP auth into an isolated `CODEX_HOME`, and preserves ACP metadata across stale session-store writes.

## Best Possible Solution

Land a focused ACPX repair that keeps adapter config forwarding capability-aware or explicitly unsupported, bridges Codex auth with a portable fallback, preserves ACP metadata without breaking the session-store writer cache, and validates the result with focused tests plus redacted live ACP smoke.

## Reproduction Assessment

Yes. Source inspection shows current main can forward unsupported ACP config keys, isolates Codex ACP from existing `auth.json`, and can lose ACP metadata from stale session-store snapshots; the contributor also supplied copied live fresh/resume output for Claude, Codex, Gemini-like ACP, and metadata resume failures.

## Solution Assessment

No, not as-is. The fix is in the right ACPX/auth/session-store area, but it should not silently drop custom-adapter config updates, should not require symlink privileges, and should not defeat the writer-owned session-store cache.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve config updates for custom ACP adapters:** `extensions/acpx/src/runtime.ts:980-982`
  - body: The final branch only calls `delegate.setConfigOption` for Codex, Claude, or OpenClaw bridge commands. A custom ACP adapter can legitimately support `session/set_config_option`, and the manager persists runtime options after this call returns, so `/acp set` can now report success while the adapter never receives the update. Make this capability-aware or return an unsupported-control error instead of silently no-oping.
  - confidence: 0.88
- **[P2] Avoid requiring symlink privileges for Codex auth:** `extensions/acpx/src/codex-auth-bridge.ts:373`
  - body: When source `auth.json` exists, setup now awaits `fs.symlink` and lets `EPERM`/`EACCES` fail the whole ACPX registration. File symlinks require special privileges on Windows-like setups, so users with valid Codex auth can lose the plugin before any session starts. Add a copy, hardlink, or permission-error fallback and cover it in the auth bridge tests.
  - confidence: 0.84
- **[P2] Keep the session-store writer cache hot path:** `src/config/sessions/store.ts:446-447`
  - body: The unconditional `loadSessionStore(..., { skipCache: true })` inside every `updateSessionStore` write defeats the writer-owned mutable cache contract. CI is already failing `updateSessionStore uses the writer-owned mutable cache without disk read or parse` because this adds a disk read on the hot path; preserve stale ACP metadata without forcing a reload for every cached update.
  - confidence: 0.93
