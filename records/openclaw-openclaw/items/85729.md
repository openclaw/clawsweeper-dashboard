## Summary

Keep open: current OpenClaw exposes `config.schema.lookup` on the normal gateway tool, but the source also shows node-originated runs are filtered to a small safe tool subset that excludes `gateway`, which plausibly matches the reported Cline bridge failure. The exact `bridge-cli.mjs` wrapper is not present in this checkout, so this needs maintainer review before choosing the safe bridge/API shape.

## What This Changes

The issue asks for Cline bridge config tasks to either receive `gateway` `config.schema.lookup` or have the bridge prefetch and inject schema lookup context before asking Cline to edit config.

## Best Possible Solution

Provide an action-scoped read-only schema lookup path for Cline/node bridge config tasks, or prefetch and inject lookup context in the bridge, while preserving the current gateway write/restart trust boundary and adding focused node-bridge coverage.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available because the named `bridge-cli.mjs`/Cline wrapper is not in the checkout. Source inspection does show that `messageProvider: "node"` removes the `gateway` tool, which would remove `config.schema.lookup` if the bridge uses that path.

## Solution Assessment

No: exposing the whole current gateway tool is not the best fix because it also carries config writes and restart. The maintainable direction is an action-scoped read-only lookup or bridge-side schema preflight with explicit coverage for Cline/node config tasks.
