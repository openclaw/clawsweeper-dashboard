## Summary

Keep open. The PR improves the generic silent-turn payload filter, but it does not fix the reported memory-flush path because `runMemoryFlushIfNeeded` still runs the silent maintenance turn separately and does not pass returned error payloads through `buildReplyPayloads` for delivery.

## What This Changes

The PR adds an `isError` exception to silent-turn reply payload filtering, a focused `buildReplyPayloads` regression test, and a changelog entry for the linked memory-flush error report.

## Best Possible Solution

Wire memory-flush run error payloads or an equivalent terminal diagnostic from `runMemoryFlushIfNeeded` into visible reply delivery while keeping ordinary silent maintenance output suppressed.

## Reproduction Assessment

No live runtime reproduction was established here. Source inspection gives a high-confidence reproduction path for the PR gap: the reported memory-flush turn returns from `runEmbeddedPiAgent` inside `runMemoryFlushIfNeeded`, but its payloads are not sent through the helper this PR changes.

## Solution Assessment

No. Preserving `isError` in `buildReplyPayloads` is a useful generic guard, but it is not the narrow complete fix for the linked memory-flush bug because the memory-flush caller does not deliver those returned payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Route memory-flush errors through delivery:** `src/auto-reply/reply/agent-runner-payloads.ts:94-95`
  - body: This only keeps `isError` payloads once they reach `buildReplyPayloads`, but the reported memory-flush path runs `runEmbeddedPiAgent` inside `runMemoryFlushIfNeeded` and then uses the result only for metadata/session bookkeeping. Please route the memory-flush error result into visible delivery or add coverage through `runMemoryFlushIfNeeded`.
