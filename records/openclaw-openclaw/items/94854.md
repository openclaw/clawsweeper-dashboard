- `merge-risk: 🚨 compatibility`: Merging as-is changes no-reply compaction failure handling from fail-closed to warning-only because the catch is keyed to transcript persistence, not reply existence.

## Summary

Keep open. The PR targets a real current and shipped P1 agent-delivery bug, but as submitted it downgrades compaction failures after any persisted CLI transcript, including no-reply/user-only turns, and the external PR has no after-fix real behavior proof.

## What This Changes

The branch wraps post-persistence CLI compaction in a warning-only catch and adds a Vitest case asserting delivery continues when compaction throws after a CLI reply is persisted.

## Best Possible Solution

Keep the fix in `agent-command`, but make the non-fatal catch conditional on an actual generated/deliverable assistant reply, rethrow no-reply failures, add regression coverage, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, source-level. Current main persists the CLI transcript and awaits `runCliTurnCompactionLifecycle` before pending delivery and `deliverAgentCommandResult`, while the compaction helper still throws on failed native/context compaction.

## Solution Assessment

No, not as submitted. The fix location is plausible, but the guard must be based on a generated/deliverable reply rather than `kind: "persisted"`, because persistence can also mean only the user turn was written.
