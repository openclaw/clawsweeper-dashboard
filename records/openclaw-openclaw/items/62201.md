## Summary

Keep open: current main does not contain the useful replay metadata mitigation, but this PR still has a merge-blocking delimiter-key bug and lacks inspectable real behavior proof.

## What This Changes

The PR adds transient diagnostic tool-result replay metadata, tags diagnostic exec/read tool results during persistence, omits stale or superseded diagnostic output during replay, threads session IDs, and adds focused tests.

## Best Possible Solution

Keep the generic replay-time diagnostic omission approach, but rebase onto current main, store pending metadata without delimiter parsing, add colon-containing toolCallId coverage, and require redacted after-fix proof.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to record metadata for session agent:main with toolCallId functions.read:0, then call the session drain and observe that last-colon parsing extracts agent:main:functions.read instead of agent:main; I did not run tests because this review is read-only.

## Solution Assessment

No. The replay-history location is reasonable, but pending replay metadata should not be stored in a delimiter-parsed composite string when current main supports toolCallIds containing colons.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Store pending replay metadata without colon-split keys:** `src/agents/tool-result-replay-metadata.ts:42-53`
  - body: `drainPendingToolResultReplayMetadataForSession` reconstructs the session by splitting `${sessionKey}:${toolCallId}` at the last colon. Current main preserves native Kimi IDs like `functions.read:0`, so draining `agent:main` misses `agent:main:functions.read:0`; cleanup paths can leak or later misapply stale replay metadata. Store by nested session/toolCallId keys or an encoded tuple instead.
  - confidence: 0.9
