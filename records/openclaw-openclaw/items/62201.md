## Summary

Keep open. Current main still lacks this replay mitigation, and the PR remains useful, but the current head has a collision-prone pending metadata key plus a missing required changelog entry.

## What This Changes

The PR adds transient diagnostic tool-result replay metadata, tags persisted diagnostic tool results, omits stale or superseded diagnostic outputs from replay input, and expands focused agent replay tests.

## Best Possible Solution

Keep the replay mitigation in the generic agent replay/session layer, but store pending replay metadata with a collision-free tuple structure, cover colon-containing toolCallIds in the drain paths, and add the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to record pending replay metadata for a session and a toolCallId like `functions.read:0`, run the session-level drain path, and verify the entry remains consumable because the PR parses a colon-delimited composite key.

## Solution Assessment

No. The replay-time mitigation is a reasonable narrow direction, but the current pending-key implementation should use a nested map or encoded tuple instead of delimiter parsing, then keep the focused tests and changelog coverage.
