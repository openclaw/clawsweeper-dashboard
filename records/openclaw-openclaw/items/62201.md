## Summary

Keep this PR open: the replay-time mitigation is still potentially useful and current main does not already contain it, but the latest PR head has a blocking delimiter-key bug, lacks inspectable real behavior proof, and needs a rebase before normal maintainer review can continue.

## What This Changes

The PR adds transient diagnostic tool-result replay metadata, tags diagnostic tool results at persistence time, omits stale or superseded diagnostic output during replay, threads session identifiers, and adds focused agent replay tests.

## Best Possible Solution

Keep the generic replay-time mitigation, but store pending replay metadata with nested or tuple keys, add colon-containing toolCallId regression coverage, rebase onto current main, and require redacted after-fix real behavior proof before merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to record metadata for session `agent:main` with toolCallId `functions.read:0`, then call the session drain and observe that last-colon parsing extracts `agent:main:functions.read` instead of `agent:main`; I did not run tests because this review is read-only.

## Solution Assessment

No. The replay-history location is reasonable, but pending replay metadata should not be stored in a delimiter-parsed composite string when current main supports toolCallIds containing colons.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Store pending replay metadata without delimiter parsing:** `src/agents/tool-result-replay-metadata.ts:42-53`
  - body: `drainPendingToolResultReplayMetadataForSession` recovers the session by splitting `${sessionKey}:${toolCallId}` at the last colon. Current main preserves Kimi IDs such as `functions.read:0`, so a session-level drain for `agent:main` misses `agent:main:functions.read:0` and can leave orphan metadata that later applies to a reused ID. Use a nested map or encoded tuple key instead.
  - confidence: 0.9
