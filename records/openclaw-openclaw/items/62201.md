## Summary

Keep open. The PR targets a real agents transcript-replay problem and current main does not already contain this stale diagnostic toolResult replay mitigation, but the current branch still has a source-level blocker in the new pending replay metadata keying: session-level drains parse a colon-delimited composite key even though current tool call IDs can contain colons, so cleanup can miss orphan metadata in exactly the empty-pending paths the PR added.

## What This Changes

The PR adds transient diagnostic tool-result replay metadata, tags persisted diagnostic tool results, omits stale or superseded diagnostic results from replay input, threads session IDs through guarded session managers, adds adjusted-argument peeking, and expands focused agent replay tests.

## Best Possible Solution

Keep the replay mitigation in the generic agents transcript/session guard layer, but make pending replay metadata storage collision-safe, add regression coverage for colon-containing toolCallIds in the empty-pending drain paths, and add the required changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. A focused source-level reproduction is to record pending replay metadata for a session and a toolCallId such as `functions.foo:1`, call the session-level drain used by empty-pending clear/flush, then verify the metadata is still consumable because the PR splits the composite key at the last colon.

## Solution Assessment

No. The overall replay-time mitigation is a plausible narrow approach, but the current key format is not the safest implementation; use a nested map or encoded tuple instead of parsing a colon-delimited string, then keep the existing tests and add the missing colon-ID case.
