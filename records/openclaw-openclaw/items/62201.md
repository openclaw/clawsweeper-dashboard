## Summary

Keep open: the PR is a useful replay-time mitigation direction, but the current head still has a delimiter-collision bug, stale Pi package imports, missing inspectable real behavior proof, and a dirty/conflicting merge state.

## What This Changes

The PR adds replay metadata for transient diagnostic tool results, tags persisted exec/read outputs, omits stale or superseded diagnostics during replay, and adds focused tests.

## Best Possible Solution

Rebase onto current main, store pending replay metadata in delimiter-safe session/tool maps or encoded tuples, keep the replay omission in sanitizeSessionHistory, add colon-ID coverage, and require redacted after-fix proof.

## Reproduction Assessment

Yes. A high-confidence source-level path is to record metadata for sessionKey `agent:main` and toolCallId `functions.read:0`, then drain `agent:main`; the PR's last-colon parser extracts `agent:main:functions.read`, so cleanup misses the entry.

## Solution Assessment

No. The replay-history hook point is reasonable, but the current implementation must avoid delimiter-parsed pending keys, update imports to the current Pi namespace, and provide inspectable real behavior proof before merge.
