## Summary

The issue is still useful when narrowed to exposing that existing operation after maintainers decide destructive-action eligibility and recovery policy.

## What This Changes

Requests a supported Workboard card deletion or removal operation, with optional management UI controls and a retention policy.

## Best Possible Solution

Expose the existing canonical `workboard.cards.delete` behavior through approved tool and Control UI surfaces only after defining eligible states, authorization, confirmation, and recovery expectations; avoid a parallel persistence path or new retention setting unless maintainers explicitly choose soft deletion.

## Solution Assessment
