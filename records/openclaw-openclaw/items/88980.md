## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Merge account bot-loop overrides with base defaults:** `extensions/telegram/src/bot-handlers.runtime.ts:1296-1299`
  - body: This new account-level nested config is read from `telegramCfg.botLoopProtection`, but `telegramCfg` is produced by `mergeTelegramAccountConfig`, which currently shallow-spreads an account over the channel base. A per-account partial override such as `{ enabled: false }` will discard top-level budget fields like `maxEventsPerWindow`, unlike the Discord/Slack nested merge pattern, so Telegram multi-account configs get surprising guard settings. Add explicit nested merging or pass the broad and account configs separately before resolving the loop guard.
  - confidence: 0.86
- **[P2] Count only messages that pass the group gates:** `extensions/telegram/src/bot-handlers.runtime.ts:1346`
  - body: The bot-pair guard is recorded before the normal group path finishes, including the later `requireMention` drop in `buildTelegramMessageContext`. In an `allowBots: true` group that still requires mentions, unmentioned bot chatter that will be dropped can consume the loop budget and put the pair into cooldown before a properly mentioned bot message arrives. Run the loop guard only after normal group authorization and mention acceptance, or split the early allow/drop decision from guard recording.
  - confidence: 0.82
