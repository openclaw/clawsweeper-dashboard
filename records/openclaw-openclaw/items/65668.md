## Summary

Keep open. Current main still preserves the reported custom-supervisor failure path for non-container Linux/macOS processes with no recognized supervisor markers: generic non-update restarts can detach-spawn a child, then the tracked parent exits, while the doctor/docs wording still presents `OPENCLAW_NO_RESPAWN=1` mainly as startup optimization rather than an operational safety workaround.

## What This Changes

The issue asks OpenClaw to prevent SIGUSR1/config-triggered gateway restarts from orphaning a detached child under unrecognized custom supervisors, or at least document `OPENCLAW_NO_RESPAWN=1` as a safety workaround.

## Best Possible Solution

Make generic non-update gateway restarts use in-process restart by default for unrecognized/no-supervisor contexts, or require an explicit detached-respawn opt-in, while preserving known supervisor handoff and `update.run` fresh-process replacement. Update doctor/docs/tests/changelog so `OPENCLAW_NO_RESPAWN=1` is documented as a safety workaround for unmanaged or custom-supervised deployments, not just a startup optimization.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: Linux/macOS with no recognized supervisor hints and no container detection returns `mode: "spawned"`, after which the run loop exits the original PID; a PID-tracking supervisor can then restart into the still-bound orphan as reported. I did not run a live supervisor harness because this review was read-only.

## Solution Assessment

No. Current main is not the best solution because it only disables generic detached respawn on Windows without task markers and inside detected containers; the reported custom-host supervisor case remains. The narrow maintainable solution is to remove detached generic restarts from unrecognized supervisor contexts or make them explicit opt-in while keeping update restarts separate.
