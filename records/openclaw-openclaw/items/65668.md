## Summary

Keep open. Current main and the latest shipped release still detach-spawn for unmanaged, non-container Unix restarts, then exit the original supervisor-tracked PID; recent work fixed the container case and update-specific restart path, but not the custom-supervisor path reported here.

## What This Changes

The issue asks OpenClaw to avoid orphaning gateway processes when SIGUSR1/config-triggered restarts run under custom supervisors, and to update the misleading OPENCLAW_NO_RESPAWN guidance.

## Best Possible Solution

Make ordinary non-update SIGUSR1/config restarts stay in-process when no trusted supervisor is detected, preserve update.run fresh-process replacement, and update doctor/docs/tests to describe the supervisor safety contract.

## Reproduction Assessment

Yes, at source level. On non-container Linux or macOS with OPENCLAW_NO_RESPAWN unset and no recognized supervisor markers, a restart-required config path or authorized SIGUSR1 returns mode "spawned" and then exits the original PID.

## Solution Assessment

Yes. The narrow maintainable fix is to avoid detached self-respawn for ordinary non-update restarts without trusted supervision; documentation-only wording would reduce confusion but leave the orphaning path intact.
