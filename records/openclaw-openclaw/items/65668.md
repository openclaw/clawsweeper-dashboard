## Summary

Keep open. Current main and the latest shipped release still have the reported non-container custom-supervisor path: unrecognized supervisors fall through to detached respawn, and the run loop exits the original tracked PID.

## What This Changes

The issue asks OpenClaw to prevent or clearly warn about SIGUSR1/config restarts orphaning the gateway under unrecognized custom supervisors and causing EADDRINUSE loops.

## Best Possible Solution

Make ordinary non-update SIGUSR1/config restarts stay in-process when no trusted supervisor is detected, keep update-specific fresh-process replacement, and update doctor/docs/tests to state the supervisor safety contract.

## Reproduction Assessment

Yes, at source level. Unset `OPENCLAW_NO_RESPAWN` on non-container Linux or macOS without recognized supervisor markers, trigger a restart-required config path or SIGUSR1, and current main returns `spawned` then exits the original PID.

## Solution Assessment

Yes. The narrow maintainable fix is to change ordinary non-update restarts to avoid detached self-respawn without trusted supervision; documentation-only wording would reduce confusion but leave the orphaning path intact.
