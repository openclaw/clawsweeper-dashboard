## Summary

Keep open. Current main still preserves the reported non-container Linux/macOS path where an unrecognized custom supervisor falls through to detached respawn and exits the tracked parent, while doctor/docs still frame `OPENCLAW_NO_RESPAWN=1` as an optimization rather than an availability workaround.

## What This Changes

The issue asks OpenClaw to avoid or clearly warn about SIGUSR1/config-triggered detached gateway respawn under custom supervisors because it can orphan a child and cause EADDRINUSE restart loops.

## Best Possible Solution

Make generic non-update gateway restarts use in-process restart, or require explicit opt-in for detached respawn, when no known supervisor is detected; keep update-specific fresh-process replacement and update doctor/docs/tests/changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: unset `OPENCLAW_NO_RESPAWN`, run on non-container Linux or macOS without launchd/systemd/schtasks markers, then trigger a restart-required `config.patch` or SIGUSR1; the code returns `spawned` and exits the original PID. I did not run a live supervisor harness because this review is read-only.

## Solution Assessment

Yes, the in-process-default or explicit-detached-opt-in direction is the narrowest maintainable fix for generic non-update restarts. Merely rewording doctor/docs would reduce confusion but would leave the orphaning behavior intact.
