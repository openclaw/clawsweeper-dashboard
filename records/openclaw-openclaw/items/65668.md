## Summary

Keep open. Current main only partially mitigates this restart class: containers and unmanaged Windows now avoid generic detached respawn, but non-container Linux/macOS processes with no recognized supervisor markers still detach-spawn a child and then exit the tracked parent; the doctor and VPS/Pi docs still describe `OPENCLAW_NO_RESPAWN=1` as startup optimization rather than an availability workaround.

## What This Changes

The issue asks OpenClaw to avoid or clearly warn about SIGUSR1/config restart paths that detach-spawn an orphaned gateway under custom supervisors, causing EADDRINUSE restart loops.

## Best Possible Solution

Make generic non-update gateway restarts use in-process restart when no known supervisor is detected, or require explicit opt-in for detached respawn, while preserving update-specific fresh process replacement and updating doctor/docs/tests/changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: unset `OPENCLAW_NO_RESPAWN`, no launchd/systemd/schtasks markers, non-container Linux/macOS, and trigger a non-update SIGUSR1/config restart; the code returns `spawned` and then exits the original PID. I did not run a live supervisor harness because this review was read-only.

## Solution Assessment

No. Current main is not the best solution because it disables the dangerous generic detached respawn only for unmanaged Windows and detected containers; host-side custom supervisors still hit the reported path. The narrower maintainable fix is to change generic non-update restart behavior while leaving update and known-supervisor paths intact.

## Repair Work Prompt

Fix the custom-supervisor SIGUSR1/config restart orphaning bug described in https://github.com/openclaw/openclaw/issues/65668. Current main still lets generic non-update restarts call `restartGatewayProcessWithFreshPid()`, which detach-spawns on Linux/macOS when no launchd/systemd/schtasks marker is present and the process is not in a detected container. Change the generic non-update restart behavior so unrecognized/no-supervisor contexts use in-process restart by default, or require an explicit detached-respawn opt-in, while preserving `respawnGatewayProcessForUpdate()` / `update.run` fresh OS-process replacement, known supervisor handoff, Windows Scheduled Task behavior, and container in-process behavior. Update or replace the tests that currently assert Linux/no-supervisor detached spawn, add run-loop coverage for the generic fallback, and update doctor/VPS/Raspberry Pi/Gateway docs plus a user-facing changelog entry. Do not broaden supervisor detection only by parent process name unless the false-positive behavior is explicitly covered; do not change package-update restart semantics from #66865.
