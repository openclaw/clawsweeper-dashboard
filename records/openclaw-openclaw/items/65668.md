## Summary

Keep open. Current main still detach-spawns a fresh gateway for non-update SIGUSR1/config restarts on unmanaged Linux/macOS hosts without launchd/systemd/schtasks markers, so custom supervisors can lose the tracked PID; doctor and docs still frame `OPENCLAW_NO_RESPAWN=1` as a startup optimization rather than an availability workaround.

## What This Changes

The issue asks OpenClaw to prevent or clearly warn about detached gateway self-respawn under custom supervisors during SIGUSR1/config restarts.

## Best Possible Solution

Make generic non-update gateway restarts use in-process restart when no known supervisor is detected, or require explicit opt-in for detached respawn, while preserving update-specific process replacement and updating doctor/docs/tests/changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: unset `OPENCLAW_NO_RESPAWN`, run non-container Linux/macOS without recognized supervisor env markers, and trigger a non-update SIGUSR1/config restart; the helper returns `spawned` and the run loop exits the original PID.

## Solution Assessment

Yes, with scope control. The best maintainable fix is the issue's default-inversion direction for generic non-update restarts; parent-process-name supervisor detection is less reliable and should not replace the clearer in-process fallback/default opt-in boundary.
