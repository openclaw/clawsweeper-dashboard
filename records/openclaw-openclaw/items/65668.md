## Summary

Keep open. Current main still has the reported non-container, non-Windows custom-supervisor path where an unrecognized supervisor falls through to detached respawn, then the tracked parent exits; the advisory/docs wording also still frames `OPENCLAW_NO_RESPAWN=1` as startup tuning rather than an availability workaround.

## What This Changes

The issue asks for non-update `SIGUSR1`/`config.patch` restarts under custom supervisors to avoid detached orphan respawn and for `OPENCLAW_NO_RESPAWN` guidance to describe the availability risk.

## Best Possible Solution

Make ordinary non-update SIGUSR1/config restarts stay in-process when no recognized supervisor is detected, keep update-specific fresh-process replacement, and update doctor/docs/tests to describe the supervisor safety contract.

## Reproduction Assessment

Yes, at source level. Unset `OPENCLAW_NO_RESPAWN`, run on non-container Linux or macOS without launchd/systemd/schtasks markers, then trigger a restart-required `config.patch` or SIGUSR1; current main returns `spawned` and exits the original PID.

## Solution Assessment

Yes. For ordinary non-update restarts, in-process restart by default or explicit detached-respawn opt-in is the narrowest maintainable fix; documentation-only wording would reduce confusion but leave the orphaning behavior intact.
