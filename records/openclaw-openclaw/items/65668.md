## Summary

Keep open. Current main still preserves the reported non-container Linux/macOS path where unrecognized custom supervisors fall through to detached respawn and the run loop exits the tracked parent PID; the merged update work separated update restarts but did not remove the generic non-update SIGUSR1/config restart hazard.

## What This Changes

The issue asks OpenClaw to prevent or clearly warn about detached gateway respawn under unrecognized custom supervisors during SIGUSR1/config.patch restarts.

## Best Possible Solution

Make generic non-update gateway restarts use the in-process path, or require explicit detached-respawn opt-in, when no known supervisor is detected; preserve update-specific process replacement and update tests, doctor/docs, and changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: unset `OPENCLAW_NO_RESPAWN`, run on non-container Linux or macOS without launchd/systemd/schtasks markers, then trigger a restart-required `config.patch` or authorized SIGUSR1; the code returns `spawned` and exits the original PID. I did not run a live supervisor harness because this review is read-only.

## Solution Assessment

Yes. The in-process-default or explicit-detached-opt-in direction is the narrowest maintainable fix for generic non-update restarts now that update restarts have a separate helper; doctor-only wording would reduce confusion but leave the orphaning behavior intact.
