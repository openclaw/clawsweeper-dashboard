## Summary

Keep open. Current main still has a real channel config contract gap: the shared streaming shape supports `streaming.preview.toolProgress` without requiring `streaming.mode`, but affected channel schemas such as WhatsApp and LINE do not expose that nested `streaming` object, and LINE's dispatch path does not pass the shared suppression option. The report is privacy-sensitive, so this should remain with maintainers for scope approval rather than be closed or auto-queued.

## What This Changes

The issue requests that `channels.<id>.streaming.preview.toolProgress: false` round-trip and suppress tool/progress chatter on channels that do not already declare `streaming.mode`.

## Best Possible Solution

Add a shared channel streaming config surface that lets non-preview channels preserve `streaming.preview.toolProgress`, then apply that setting to default tool/progress suppression independently of preview `mode`; keep approval prompts, media, errors, and final replies deliverable, and add focused schema plus channel dispatch tests for each affected transport.

## Reproduction Assessment

Yes for the config-contract failure: current main source shows the requested key is supported by the shared schema but absent from WhatsApp and LINE channel schemas that drive config normalization. The full end-to-end leakage report has strong user context, but still needs focused channel tests to lock each affected transport's visible behavior.

## Solution Assessment

No complete solution is present on current main. The narrowest maintainable fix is to preserve and honor the existing nested config key rather than default every channel into preview streaming mode.
