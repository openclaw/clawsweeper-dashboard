## Summary

Keep open. Current main still leaves `streaming.preview.toolProgress` only partially wired: the shared contract and docs recognize the option, but affected channel schemas and delivery paths cannot reliably preserve and apply it, so the reported privacy gap remains.

## What This Changes

The issue asks OpenClaw to preserve and honor `channels.<id>.streaming.preview.toolProgress: false` on channels that do not already declare `streaming.mode`.

## Best Possible Solution

Add a shared delivery config path that preserves and applies `preview.toolProgress` independently of `streaming.mode`, then wire affected bundled channels with focused schema, dispatch, docs, and regression coverage.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: WhatsApp and LINE cannot currently express the nested key, and their inbound dispatch paths do not pass the suppression option that prevents default progress delivery.

## Solution Assessment

No complete solution exists on current main. The safest direction is to preserve the config key and apply suppression without forcing every affected channel into preview streaming mode.
