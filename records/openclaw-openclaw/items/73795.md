## Summary

Keep open. Current main still lacks a schema-backed and runtime-wired way to honor `streaming.preview.toolProgress: false` for WhatsApp and LINE, and the shared dispatcher only suppresses generic tool/progress delivery when a channel explicitly passes the suppression flag. The related slash-command issue is adjacent configuration UX work, not a replacement for this privacy-sensitive channel delivery gap.

## What This Changes

The issue requests that `streaming.preview.toolProgress: false` be preserved and honored for channels that do not declare `streaming.mode`.

## Best Possible Solution

Define a documented channel delivery contract for hiding generic tool/progress messages independently of preview streaming, expose the schema for affected channels, wire WhatsApp and LINE into the shared suppression path or successor API, and add focused config/runtime/docs regression coverage without changing unrelated preview-channel semantics by accident.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: WhatsApp and LINE do not accept the nested config field and do not pass `suppressDefaultToolProgressMessages`, while the shared dispatcher only suppresses text-only tool/progress payloads when that flag is explicitly true.

## Solution Assessment

No current-main solution is sufficient. Preserving the nested config alone would not fix delivery; the effective setting must also reach the shared reply dispatcher through a documented channel contract.
