## Summary

Keep open. Current main still has a real config/delivery gap: the shared streaming contract supports `streaming.preview.toolProgress`, but WhatsApp and LINE do not expose that nested config and their reply paths do not pass the suppression option that prevents default tool/progress delivery.

## What This Changes

The issue requests that `streaming.preview.toolProgress: false` be preserved and respected on channels that do not already declare `streaming.mode`.

## Best Possible Solution

Add a shared delivery config path that preserves and applies `preview.toolProgress` independently of `streaming.mode`, then wire affected bundled channels with focused schema, dispatch, docs, and regression coverage.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: WhatsApp and LINE cannot currently express the nested key, and their dispatch paths do not pass the existing suppression option. Live transport proof would refine the exact user-visible leakage text and confirm the file-watcher rewrite behavior.

## Solution Assessment

No complete solution exists on current main. The safest fix is to preserve the config key through relevant channel schemas and apply suppression without forcing every affected channel into preview streaming mode.
