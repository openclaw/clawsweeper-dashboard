## Summary

Keep open. Current main still has a real config-contract gap: shared `streaming.preview.toolProgress` exists and is documented, but WhatsApp and LINE do not expose the nested `streaming` object in their channel schemas, and their inbound dispatch paths do not pass the suppression option that Slack/Telegram/Discord use.

## What This Changes

The issue requests that `channels.<id>.streaming.preview.toolProgress: false` be preserved and respected for channels even when `streaming.mode` is absent.

## Best Possible Solution

Add a shared channel streaming config surface that preserves `streaming.preview.toolProgress` without requiring `streaming.mode`, then wire affected non-preview channels to use that setting to suppress default tool/progress messages while keeping approvals, media, errors, and final replies deliverable. Update docs only if the supported channel set or semantics differ from the current streaming docs.

## Reproduction Assessment

Yes for the config-contract failure: current main schemas show the requested key cannot round-trip on WhatsApp and LINE while it is accepted on Slack/Telegram/Discord. The end-to-end leakage path is plausible from the shared default progress dispatcher, but it still needs a focused channel test for each affected transport.

## Solution Assessment

No current solution is complete. The narrow maintainable fix is not to default every channel to preview streaming, but to preserve the config key and apply it to default tool/progress suppression independently of `streaming.mode`.
