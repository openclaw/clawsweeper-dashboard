## Summary

Keep open. Current main still has the reported config-contract gap for non-preview channels: shared `streaming.preview.toolProgress` exists, but WhatsApp and LINE schemas do not accept the nested `streaming` object, and their inbound reply paths do not pass the shared suppression option that would hide generic tool/progress messages.

## What This Changes

The issue requests preserving and honoring `channels.<id>.streaming.preview.toolProgress` even when a channel does not declare `streaming.mode`, so tool-progress details can be suppressed consistently.

## Best Possible Solution

Add a shared channel streaming config surface that preserves `streaming.preview.toolProgress` without requiring `streaming.mode`, then wire affected non-preview channels to suppress generic tool/progress messages from that setting while preserving approvals, media, errors, and final replies. Cover the change with schema round-trip tests and focused WhatsApp/LINE dispatch tests.

## Reproduction Assessment

Yes. The config-contract failure is reproducible from current source because WhatsApp and LINE strict schemas omit `streaming` while validation replaces channel config with schema output; the delivery leak path is reproducible through the generic dispatcher when channels do not pass `suppressDefaultToolProgressMessages`.

## Solution Assessment

No current solution is complete. The safer fix is to preserve and apply `preview.toolProgress` independently of preview streaming mode rather than defaulting all affected channels to `streaming.mode: "partial"`.
