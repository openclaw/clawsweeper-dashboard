## Summary

Keep open. Current main still leaves a real config and delivery gap: the shared streaming contract can represent `streaming.preview.toolProgress`, but WhatsApp and LINE do not expose that nested config and their reply paths do not pass the existing default-progress suppression option.

## What This Changes

Preserve and apply `streaming.preview.toolProgress: false` for channels that do not already declare `streaming.mode`, so tool/progress details are not exposed to chat recipients.

## Best Possible Solution

Add a shared delivery config path that preserves and applies `preview.toolProgress` independently of `streaming.mode`, then wire affected channels with focused schema, dispatch, docs, and regression-test coverage.

## Reproduction Assessment

Yes for a source-level reproduction: WhatsApp and LINE cannot currently express the nested key, and their dispatch paths do not pass the suppression option. Live transport proof would only refine the exact user-visible messages.

## Solution Assessment

No complete solution exists on current main. The safest fix is to preserve the config key through the relevant channel schemas and apply suppression without forcing every affected channel into preview streaming mode.
