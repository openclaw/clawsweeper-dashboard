## Summary

Keep open. Current main still has a real config and delivery gap: the shared streaming contract can represent `streaming.preview.toolProgress`, but affected channel schemas and dispatch paths do not consistently preserve or apply that suppression, including the related Slack overlap in #74358. Review target: https://github.com/openclaw/openclaw/issues/73795

## What This Changes

The issue requests that `streaming.preview.toolProgress: false` be preserved and honored for channel configs even when no `streaming.mode` is set.

## Best Possible Solution

Add a shared delivery config path that preserves and applies tool-progress suppression independently of preview mode, then wire the affected channel schemas and dispatch paths with focused docs and regression coverage.

## Reproduction Assessment

Yes. Source-level reproduction against current main is high-confidence: WhatsApp and LINE cannot express the nested key, their dispatch paths do not pass suppression, and Slack drops suppression when `toolProgress` is false. Live transport proof would refine only the exact recipient-visible text.

## Solution Assessment

No complete solution exists on current main. The safest direction is to preserve the config key and apply default tool-progress suppression without forcing unsupported channels into preview streaming mode; a separate clearer delivery knob may be warranted if maintainers do not want `streaming.preview` to govern non-preview channels.
