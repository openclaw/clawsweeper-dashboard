## Summary

Keep open. Current main still cannot honor the reported setting for WhatsApp and LINE: their channel schemas do not expose `streaming.preview.toolProgress`, their runtimes do not pass the shared suppression option, and the shared dispatcher only suppresses generic tool/progress delivery when that option is explicitly set. This remains privacy-sensitive and needs maintainer direction because the fix crosses config schema, docs, and channel delivery semantics.

## What This Changes

The issue requests preserving and honoring `streaming.preview.toolProgress: false` for channels without an existing `streaming.mode`, so external recipients do not receive generic tool/progress narration.

## Best Possible Solution

Define a documented, schema-backed channel delivery contract for hiding generic tool/progress messages independently of preview streaming, expose it for WhatsApp and LINE, wire those runtimes into `suppressDefaultToolProgressMessages` or its successor, and add config/runtime/docs regression coverage without changing unrelated channel semantics by accident.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: WhatsApp and LINE do not accept the nested `streaming.preview.toolProgress` config and do not pass the shared suppression option, while the dispatcher only suppresses generic tool/progress messages when that option is explicitly true.

## Solution Assessment

No current solution on main is sufficient. Preserving the nested config alone would not fix delivery; the runtime must also map the effective setting into the shared suppression path or a clearer channel delivery contract.
