## Summary

Keep open. Current main still lacks a supported config and runtime path for the reported WhatsApp and LINE suppression behavior, and the issue describes a privacy-sensitive delivery problem rather than a stale or duplicate cleanup item.

## What This Changes

The issue asks for `streaming.preview.toolProgress: false` to be preserved and honored for channels that do not already declare `streaming.mode`.

## Best Possible Solution

Add a documented, schema-backed channel delivery contract for suppressing generic tool/progress messages on non-preview channels, wire WhatsApp and LINE into the shared suppression path, and cover config preservation plus runtime delivery behavior with focused regression tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: WhatsApp and LINE strict schemas do not accept `streaming.preview.toolProgress`, and their runtime reply options do not pass the only shared flag that suppresses default tool/progress messages.

## Solution Assessment

No current-main solution is sufficient. Preserving the nested config alone would not solve delivery unless the effective setting is also mapped into `suppressDefaultToolProgressMessages` or a clearer successor contract.
