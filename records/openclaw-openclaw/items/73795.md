## Summary

Keep open. Current main still has a real config and delivery gap: the shared streaming contract has `preview.toolProgress`, but WhatsApp and LINE do not expose that config and their reply paths do not request the existing suppression behavior.

## What This Changes

Respect `streaming.preview.toolProgress: false` for affected channels even when no `streaming.mode` is configured.

## Best Possible Solution

Add a shared config and delivery path that preserves and applies `preview.toolProgress` independently of `streaming.mode`, then wire affected bundled channels with focused schema/runtime/docs coverage.

## Reproduction Assessment

Yes for source-level reproduction: WhatsApp and LINE cannot currently express `streaming.preview.toolProgress`, and their dispatch paths do not pass the suppression option that would stop default tool/progress messages. I did not live-run the auto-reload path, so the exact rewrite from the report is not independently reproduced here.

## Solution Assessment

No complete solution exists on current main. The safest direction is to preserve the config key through relevant channel schemas and apply suppression without forcing every affected channel into preview streaming mode.
