## Summary

Keep open. Current main still leaves the reported WhatsApp and LINE gap: their strict config schemas do not expose `streaming.preview.toolProgress`, and their reply runtimes do not pass the shared suppression option that prevents default tool/progress messages from being delivered.

## What This Changes

The issue asks OpenClaw to preserve and honor `streaming.preview.toolProgress: false` for channels without a configured streaming mode so internal tool progress is not shown to end users.

## Best Possible Solution

Define a documented, schema-backed channel delivery contract for suppressing generic tool/progress messages independently of preview streaming, expose it for WhatsApp and LINE, wire those runtimes into `suppressDefaultToolProgressMessages` or its successor, and add config/runtime/docs regression coverage without changing unrelated channel semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: WhatsApp and LINE do not accept the nested `streaming.preview.toolProgress` config and do not pass the shared suppression option, while the dispatcher only suppresses generic tool/progress messages when that option is explicitly true.

## Solution Assessment

No current solution on main is sufficient. Preserving the nested config alone would not fix delivery; the runtime must also map the effective setting into the shared suppression path or a clearer channel delivery contract.
