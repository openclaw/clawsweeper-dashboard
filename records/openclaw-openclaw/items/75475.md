## Summary

Keep open. The report is fresh and describes a concrete user-visible mismatch; current source has pieces that should honor explicit session overrides, but the `session_status` wrapper still feeds the configured default model into the shared status builder when an explicit override exists, so there is not enough high-confidence evidence to close this as already fixed.

## What This Changes

The issue requests that `/status` and `session_status` report the active per-session model override selected with `/model`, instead of the configured default model.

## Best Possible Solution

Add a focused regression for a session whose configured default is `deepseek/deepseek-v4-flash` and whose explicit session override is `deepseek/deepseek-v4-pro`, then make both `/status` and `session_status` resolve the selected model from the same persisted override state while leaving the broader active-runtime drift work to #42690.

## Reproduction Assessment

Yes: the issue provides concrete TUI steps, and a focused test can simulate the same state by applying a session model patch or stored `providerOverride`/`modelOverride` before invoking `session_status`. The read-only review did not execute that test path.

## Solution Assessment

Unclear because there is no proposed patch yet. The narrowest maintainable solution is to centralize the selected-model resolution used by TUI, `/status`, and `session_status`, and add regression coverage for explicit session overrides without changing runtime routing behavior.
