## Summary

Current main and the latest release already route the visible status model through the persisted session override, so the reported `/model` mismatch is resolved for the explicit per-session selection case. The broader selected-versus-active runtime routing split remains separate from this issue and is already tracked by #42690.

## What This Changes

The issue asks for `session_status` and `/status` to report the active per-session model override after `/model` instead of falling back to the configured default model.

## Best Possible Solution

Keep the shipped shared status implementation for explicit user session selections, and continue tracking the distinct runtime-routed active-model display work in #42690 rather than reopening this narrower `/model` override report.

## Reproduction Assessment

Yes. The issue gives concrete TUI steps, and the current source path can be checked deterministically: `/model` persists `providerOverride`/`modelOverride`, and the shared renderer uses those fields for the visible `Model:` line.

## Solution Assessment

Yes. Using the existing persisted session override fields in the shared `/status` and `session_status` renderer is the narrowest maintainable fix because it keeps TUI, gateway session patches, and agent-tool status output on the same contract.
