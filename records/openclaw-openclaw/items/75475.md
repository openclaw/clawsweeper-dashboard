## Summary

Current main already reports explicit per-session model overrides through the shared status renderer, and the same relevant implementation is present in the shipped v2026.4.29 tag. The remaining broader active-runtime routing distinction is tracked separately by #42690, but this issue's explicit `/model` override mismatch is implemented.

## What This Changes

The issue asks for `/status` and `session_status` to show a per-session `/model` override instead of the configured default model.

## Best Possible Solution

Keep the shared `/status` and `session_status` renderer as the source of truth for explicit session selections, and leave the separate selected-versus-active runtime model split to the canonical #42690 follow-up.

## Reproduction Assessment

Yes. A high-confidence validation path is to set a default model, persist a session entry with `providerOverride`/`modelOverride` from `/model`, then invoke `/status` or `session_status`; current main routes that entry into the shared renderer, which prints the override.

## Solution Assessment

Yes. The implemented direction is the narrow maintainable fix for this issue because both `/status` and `session_status` read the same persisted override fields before configured defaults, without changing runtime routing behavior.
