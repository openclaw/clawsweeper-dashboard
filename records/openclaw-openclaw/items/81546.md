## Summary

Keep open: the report has a concrete transcript and current main still has a narrow source-level gap where caught tool execution errors are wrapped as status:error payloads without an explicit error flag, which can leave the model-visible tool result looking successful.

## What This Changes

The issue asks OpenClaw to treat edit/tool execution failures as real failures and recover or retry instead of ending a run with incomplete work.

## Best Possible Solution

Normalize caught tool execution failures as explicit error tool results and preserve the existing unresolved mutating-failure warning/retry tracking; avoid blind automatic retries of mutating edits.

## Reproduction Assessment

Yes, at source level: the issue provides a concrete failed edit transcript, and current main shows thrown tool failures become status:error payloads without isError:true. I did not run a full Minimax/Telegram replay in this read-only review.

## Solution Assessment

No, a blind automatic retry of the same ambiguous edit is not the best first fix. The narrow maintainable repair is to mark caught failures as real tool errors and add regression coverage for the adapter/model-visible result path.
