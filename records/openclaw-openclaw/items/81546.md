## Summary

Keep open. Current main has partial tool-error handling, but the report still points at an unresolved product/runtime boundary: structured tool failures can be returned as normal-looking tool results while provider replay relies on the persisted `isError` flag, and there is no general automatic retry policy for failed edit/tool calls.

## What This Changes

The issue asks OpenClaw to recover from a failed edit/tool call by retrying automatically or giving the model a stronger retry hint when the tool result contains a structured error.

## Best Possible Solution

Normalize structured tool failures as real failed tool results for persistence/provider replay, then handle automatic retries or circuit breakers through an explicit maintainer-approved tool-error policy.

## Reproduction Assessment

Yes at source level, but not as a live current-main session reproduction. The code path shows structured tool failures can be represented without `isError:true`, provider replay consumes `toolResult.isError`, and no general failed-tool retry path was found.

## Solution Assessment

Partly. A narrow `isError` normalization and regression test would be maintainable, but default automatic retries for edit/tool failures need product judgment because some tools are mutating or non-idempotent.
