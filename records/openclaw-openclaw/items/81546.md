## Summary

Keep open: current main still has the source-level failure-marker gap, and an open linked fix PR with closing syntax is already tracking the implementation path, so this issue should remain open until that PR lands or is rejected.

## What This Changes

The issue asks OpenClaw to treat failed tool executions as real failures so agents can recover from ambiguous edit errors instead of ending with work incomplete.

## Best Possible Solution

Review and land the bridge-level fix in https://github.com/openclaw/openclaw/pull/81564 once it satisfies its proof and merge gates; avoid blind automatic retries for mutating edit failures.

## Reproduction Assessment

Yes, at source level: current main still turns caught tool failures into status:error payloads without a model-visible Pi isError override, and the reporter provided a concrete transcript showing the failed edit path. I did not run a full MiniMax/Telegram replay in this read-only review.

## Solution Assessment

No for the requested blind automatic retry, because repeating the same ambiguous mutating edit is not the safest first repair. The narrower maintainable solution is to mark status:error and timeout tool results as model-visible failures in the Pi bridge, which the open linked PR targets.
