## Summary

Current main still has the source-level failure-marker gap, and an open linked PR with closing syntax is already tracking the narrow bridge fix. This issue should remain open until that PR or an equivalent replacement lands.

## What This Changes

The issue requests that OpenClaw treat failed tool execution results, especially ambiguous edit failures, as real failures so the agent can recover instead of ending with incomplete work.

## Best Possible Solution

Land the bridge-level fix from https://github.com/openclaw/openclaw/pull/81564, or an equivalent replacement, after it is rebased and proves the final Pi transcript/model-visible tool result is marked with isError:true for status-error and timeout results.

## Reproduction Assessment

Yes. The issue has a concrete failed edit transcript, and current main source shows the same failure shape can become a status:error payload without the final Pi bridge deriving an isError override; I did not run a live MiniMax/Telegram replay in this read-only review.

## Solution Assessment

Yes for the linked bridge-level direction, not for blind automatic retries. The narrow maintainable fix is to mark the final Pi tool result as an error when the structured status is error-like, preserving existing mutating-failure tracking and letting the model recover.
