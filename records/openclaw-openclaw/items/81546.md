## Summary

Keep open: the reported bug is still source-reproducible on current main, and an open linked fix PR with closing syntax is already the implementation path. This issue should close after that PR lands or after maintainers choose a different canonical fix.

## What This Changes

The issue reports that caught tool execution failures can be delivered as successful tool results, leaving the model without a reliable retry signal after an ambiguous edit failure.

## Best Possible Solution

Review and land the linked Pi bridge fix once proof/checks are acceptable, or supersede it with an equivalent narrow fix that marks status:error and timeout tool results as model-visible failures.

## Reproduction Assessment

Yes, at source level: current main still wraps caught tool exceptions as status:error payloads and the Pi tool_result bridge returns no isError override. I did not run a live MiniMax session replay.

## Solution Assessment

Yes, the best current direction is the linked bridge-level fix, not blind automatic retries of ambiguous mutating edits. It preserves the error signal so the model/runtime can react without repeating the same unsafe edit call automatically.
