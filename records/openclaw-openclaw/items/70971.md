## Summary

Keep open. The PR targets a real bundled-tool allowlist gap, but this exact patch is not mergeable because the final exact-name filter would drop supported plugin-key allowlists such as `bundle-mcp`.

## What This Changes

The PR wraps the embedded runner's final base-plus-bundled tool list in `applyEmbeddedAttemptToolsAllow()` so runtime `toolsAllow` also filters bundled MCP/LSP tools.

## Best Possible Solution

Enforce runtime `toolsAllow` after bundled MCP/LSP materialization with plugin-aware semantics for exact bundled names, plugin ids, and plugin groups, then add focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main can materialize bundled tools after the early runtime allowlist, and with this PR applied `toolsAllow: ["bundle-mcp"]` would create the bundle MCP runtime but then drop all of its `server__tool` entries in the exact-name final filter.

## Solution Assessment

No. The repair direction is right, but the current patch should be replaced with a plugin-aware runtime allowlist pass rather than reusing the exact-name helper on the final combined list.
