## Summary

Keep open. The PR addresses a real bundled-tool filtering gap, but the proposed one-line fix is not mergeable because it applies an exact tool-name allowlist after plugin-owned bundled tools are materialized, which drops supported plugin-key allowlists such as `bundle-mcp`.

## What This Changes

The PR wraps the final base-plus-bundled effective tool list in `applyEmbeddedAttemptToolsAllow()` using `params.toolsAllow` in `src/agents/pi-embedded-runner/run/attempt.ts`.

## Best Possible Solution

Enforce runtime `toolsAllow` on bundled tools through the same plugin-aware policy path used by `applyFinalEffectiveToolPolicy`, preserving exact bundled tool names and plugin/group keys such as `bundle-mcp` and `bundle-lsp`, then add regression coverage for exec-only filtering plus plugin-key and exact bundled-tool allow cases.

## Reproduction Assessment

Yes. Apply this PR with a configured bundle MCP server and runtime `toolsAllow: ["bundle-mcp"]`: the runtime is still created, but the new final exact-name filter keeps only a literal `bundle-mcp` tool, while actual exposed tools are named like `server__tool`.

## Solution Assessment

No. The direction is right, but this patch is not the best fix because it bypasses the plugin-aware allowlist expansion that current bundled-tool policy and docs rely on.
