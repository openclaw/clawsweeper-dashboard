## Summary

Keep open for maintainer review, but do not merge as-is. Current main treats `gatewayClientScopes` as internal gateway caller metadata and intentionally preserves normal external-channel `/phone` command access; this PR would reject those channel callers. The existing ClawSweeper review comment already identified the same blocking regression.

## What This Changes

The PR removes the WebChat-only condition from the phone-control arm/disarm admin guards and updates tests to expect Telegram callers without `operator.admin` to be rejected.

## Best Possible Solution

If stronger external-channel protection is desired, add an explicit phone-control owner or channel-command authorization policy and cover normal channel senders, WebChat/internal gateway callers, and non-admin scoped gateway callers in tests. Otherwise, keep the current main behavior and reject this patch.

## Reproduction Assessment

Yes. Invoke the phone-control command handler with `channel: "telegram"` and no `gatewayClientScopes`; current main’s tests expect the command to remain available, while this PR changes that path to return `requires operator.admin`.

## Solution Assessment

No. The patch treats absent internal gateway scope metadata as a failed admin check for ordinary external-channel commands; a maintainable lock-down should use explicit sender owner/channel authorization rather than unconditional `gatewayClientScopes` enforcement.
