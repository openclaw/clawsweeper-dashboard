## Summary

Keep open: the PR is not merge-ready because it would reject authorized native Telegram phone-control callers when `gatewayClientScopes` is absent, which conflicts with current main behavior and tests. It also still lacks real behavior proof for this authorization-sensitive external-channel path.

## What This Changes

The PR makes phone-control arm/disarm require `operator.admin` regardless of channel and updates two tests to expect no-scope Telegram callers to be rejected.

## Best Possible Solution

Keep the current scoped-gateway hardening unless maintainers deliberately choose a stricter external-channel phone-control policy; if they do, implement it through explicit owner/channel authorization with Telegram proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: native Telegram command execution does not pass `gatewayClientScopes`, so the PR would reject authorized Telegram `/phone arm` and `/phone disarm`. No high-confidence current-main bypass is established as written because current tests deliberately split native no-scope callers from scoped gateway callers.

## Solution Assessment

No. The patch conflates internal control-plane gateway scopes with external chat authorization; a stricter policy should be an explicit maintainer-approved owner/channel authorization change with live proof.
