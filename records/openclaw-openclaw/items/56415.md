## Summary

Keep open: the patch is not safe to merge as-is because it makes missing internal gateway scopes deny native Telegram `/phone arm` and `/phone disarm`, which conflicts with current tests and the Telegram command context path. It also still lacks required real Telegram or equivalent external-channel proof for this authorization-sensitive behavior change.

## What This Changes

The PR makes phone-control arm/disarm require `operator.admin` on every channel and changes two tests to expect no-scope Telegram callers to be rejected.

## Best Possible Solution

Keep the current scoped-gateway hardening unless maintainers deliberately choose stricter external-channel phone-control authorization; if they do, model it through explicit owner/channel authorization with WebChat, scoped gateway, and Telegram coverage.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: native Telegram command execution does not pass `gatewayClientScopes`, so the PR would reject authorized Telegram `/phone arm` and `/phone disarm` callers. No high-confidence current-main bypass is established because current tests deliberately allow no-scope Telegram callers while blocking scoped non-admin gateway callers.

## Solution Assessment

No. The patch conflates internal control-plane gateway scopes with external chat authorization; a stricter policy should be an explicit maintainer-approved owner/channel authorization change with live proof.
