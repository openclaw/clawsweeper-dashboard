## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main deliberately treats `gatewayClientScopes` as internal control-plane metadata and preserves ordinary external-channel `/phone arm|disarm` access for authorized channel senders; the PR would reject those normal Telegram/native command contexts because they do not carry `operator.admin`.

## What This Changes

The PR removes the WebChat-only condition from the phone-control arm/disarm admin guards and updates tests to expect Telegram callers without `operator.admin` to be rejected.

## Best Possible Solution

If stronger external-channel protection is desired, add an explicit owner-identity or channel-command authorization policy for phone-control and cover normal channel senders, WebChat/internal gateway callers, and non-admin scoped gateway callers in tests. Otherwise, keep the current main behavior and reject this patch.

## Reproduction Assessment

Yes. A unit-level reproduction is clear: invoke the phone-control command handler with `channel: "telegram"` and no `gatewayClientScopes`; current main allows the authorized channel command, while this PR would reject it. A separate current-main regression case with `gatewayClientScopes: ["operator.write"]` is already blocked.

## Solution Assessment

No. The submitted solution uses `operator.admin` gateway scopes as if they were present on ordinary external-channel commands, but the code treats those scopes as internal control-plane metadata. The safer path for a deliberate external-channel lock-down is an owner/channel authorization check, not an unconditional gateway-scope check.
