## Summary

Keep this PR open for maintainer disposition, but do not merge it as-is. Current main already blocks non-admin scoped gateway callers on non-WebChat channels while preserving authorized external-channel `/phone arm` and `/phone disarm`; this PR would reject normal Telegram/channel commands because those contexts do not carry internal gateway scopes.

## What This Changes

The PR removes the WebChat-only condition from the phone-control arm/disarm admin guards and updates tests so Telegram callers without `operator.admin` are rejected.

## Best Possible Solution

Keep the current gateway-scope hardening unless maintainers explicitly want stronger external-channel protection. If they do, implement an owner/channel authorization policy for phone-control instead of treating missing internal gateway scope metadata as missing `operator.admin`.

## Reproduction Assessment

Yes. A source-level reproduction is clear: invoke the phone-control command handler with `channel: "telegram"` and no `gatewayClientScopes`; current main allows the authorized channel command, while this PR would reject it. The existing tests also cover the separate scoped-gateway case with `gatewayClientScopes: ["operator.write"]`, which current main blocks.

## Solution Assessment

No. The proposed fix uses internal gateway scopes as if they were present on ordinary channel-originated commands. A safer solution, if desired, is an explicit owner or channel authorization gate for phone-control external commands.
