## Summary

Keep open for maintainer decision: the patch is not mergeable as-is because it makes `gatewayClientScopes` mandatory for native external commands, while current main intentionally preserves authorized Telegram/external callers and separately blocks scoped non-admin gateway callers.

## What This Changes

The PR removes the webchat-only guard from phone-control arm/disarm admin checks and updates tests so Telegram callers without gateway scopes are rejected.

## Best Possible Solution

Keep the current scoped-gateway hardening unless maintainers deliberately choose stricter external-channel control; if they do, model it with explicit owner/channel authorization and tests for WebChat, scoped gateway callers, and native Telegram.

## Reproduction Assessment

Yes for the PR regression by source inspection: Telegram native commands execute plugin commands with sender authorization and no `gatewayClientScopes`, so the PR head rejects them. No current-main bug reproduction is established because main tests intentionally allow Telegram callers without gateway scopes and separately reject scoped non-admin gateway callers.

## Solution Assessment

No. The PR conflates internal gateway client scopes with external sender authorization; the narrower maintainable path is the current helper or a deliberate new external owner/channel policy.
