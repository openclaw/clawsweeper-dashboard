## Summary

Keep open: the patch would reject normal authorized external channel `/phone` commands because those contexts do not carry internal gateway scopes, while current main intentionally preserves external command behavior and separately blocks non-admin scoped gateway callers.

## What This Changes

The PR changes phone-control arm/disarm to require `gatewayClientScopes` containing `operator.admin` on every channel and updates tests to expect Telegram callers without that scope to be rejected.

## Best Possible Solution

Preserve the current scoped-gateway hardening unless maintainers choose a stricter policy; if they do, implement explicit owner or channel authorization semantics with WebChat, scoped gateway, and external sender coverage.

## Reproduction Assessment

Yes for the PR regression by source inspection: a Telegram command context without `gatewayClientScopes` reaches the PR’s unconditional guard and is rejected. The claimed current-main bypass is not established because main tests intentionally cover authorized external callers without gateway scope metadata.

## Solution Assessment

No. The PR conflates internal gateway client scopes with external sender authorization; the narrower maintainable solution is the current helper, or an explicit maintainer-approved external owner/channel policy.
