## Summary

Keep open for maintainer handling, but do not merge this PR as-is. Current main already closes the scoped gateway-client gap while preserving authorized external channel commands; this PR would reverse that settled behavior by treating ordinary channel commands with no internal gateway scope metadata as non-admin.

## What This Changes

The PR removes the WebChat-only condition from the phone-control arm/disarm admin guards and changes tests to expect Telegram callers without `operator.admin` to be rejected.

## Best Possible Solution

Keep the current scoped-gateway hardening unless maintainers deliberately choose a stricter external-channel policy; if stricter protection is desired, implement it through owner identity or channel command authorization and cover WebChat, internal gateway callers, and normal external senders in tests.

## Reproduction Assessment

Yes. Invoke the phone-control command handler with `channel: "telegram"` and no `gatewayClientScopes`: current main's tests expect `/phone arm` and `/phone disarm` to work for authorized channel callers, while this PR would return `requires operator.admin`; a separate current-main test verifies `gatewayClientScopes: ["operator.write"]` is blocked.

## Solution Assessment

No. The PR uses `operator.admin` gateway scopes as if ordinary external channel commands carried them, but the code and docs treat those scopes as internal control-plane metadata; the narrower maintainable fix is the current helper or a new explicit owner/channel policy if product direction changes.
