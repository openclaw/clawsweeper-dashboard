## Summary

Keep open for maintainer handling: current main intentionally distinguishes internal gateway-scoped callers from authorized external channel senders, while this PR would reject normal Telegram-style command contexts that do not carry gateway scopes.

## What This Changes

The PR changes phone-control arm/disarm guards to require `operator.admin` on every channel and updates tests to expect Telegram callers without gateway scopes to be rejected.

## Best Possible Solution

Keep the current scoped-gateway hardening unless maintainers deliberately choose stricter external-channel control; if they do, model it through explicit owner or channel authorization with WebChat, scoped gateway, and Telegram behavior coverage.

## Reproduction Assessment

Yes for the PR regression by source inspection: a Telegram `/phone arm` or `/phone disarm` context with no `gatewayClientScopes` is allowed by current main tests but would return `requires operator.admin` on the PR branch. The claimed current-main authorization bypass is not established because authorized external channel callers are an intentional tested path.

## Solution Assessment

No. The PR conflates internal gateway client scopes with external sender authorization, so the safer path is either the current helper or a new explicit owner/channel policy if maintainers choose that product direction.
