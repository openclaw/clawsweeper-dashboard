## Summary

Keep open: the patch still appears to reject authorized Telegram/external phone-control commands that lack internal gateway scope metadata, and the PR only provides unit-test proof for an auth-sensitive Telegram-visible behavior change.

## What This Changes

This PR removes the webchat-only condition from phone-control arm/disarm admin checks and updates tests so Telegram callers without `operator.admin` are rejected.

## Best Possible Solution

Keep the current scoped-gateway hardening unless maintainers deliberately choose a stricter external-channel policy; if they do, model it through explicit owner/channel authorization with WebChat, scoped gateway, and Telegram coverage.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: native Telegram commands reach plugin execution without `gatewayClientScopes`, so the PR would reject authorized `/phone arm` and `/phone disarm` callers. The claimed current-main bypass is not high-confidence because current tests intentionally allow external no-scope callers while blocking scoped gateway callers without `operator.admin`.

## Solution Assessment

No. The PR conflates internal `gatewayClientScopes` with external sender/owner authorization, so a stricter policy should be an explicit channel or owner authorization change if maintainers choose that direction.
