## Summary

Keep open: the patch is not mergeable as written because current main separates internal gateway scopes from authorized external-channel callers, so this unconditional gate would reject normal Telegram `/phone arm` and `/phone disarm` use; the PR also lacks real behavior proof.

## What This Changes

The PR removes the webchat-only condition from phone-control arm/disarm admin checks and rewrites tests to expect Telegram callers without gateway scopes to be rejected.

## Best Possible Solution

Keep the current scoped-gateway hardening unless maintainers deliberately choose stricter external-channel control; if they do, model it through explicit owner or channel authorization with WebChat, scoped gateway, and Telegram coverage.

## Reproduction Assessment

Yes for the PR regression by source inspection: Telegram `/phone arm` or `/phone disarm` contexts normally reach the plugin without `gatewayClientScopes`, and the PR would reject them. The claimed current-main bypass is not high-confidence because current tests intentionally allow external no-scope callers while blocking non-admin scoped gateway callers.

## Solution Assessment

No. The PR conflates internal `gatewayClientScopes` with external sender authorization, so a stricter policy should be an explicit owner/channel authorization change if maintainers choose that direction.
