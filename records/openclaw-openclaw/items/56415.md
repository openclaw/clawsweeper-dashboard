## Summary

Keep open: the PR would reject normal authorized external-channel `/phone arm` and `/phone disarm` callers because those command contexts do not carry internal gateway scopes, while current main intentionally separates gateway-scoped callers from channel sender authorization.

## What This Changes

The PR removes the webchat-only condition from phone-control arm/disarm `operator.admin` checks and changes tests to expect Telegram callers without gateway scopes to be rejected.

## Best Possible Solution

Keep the current scoped-gateway hardening unless maintainers deliberately choose stricter external-channel control; if they do, model it through explicit owner or channel command authorization with WebChat, scoped gateway, and normal external sender tests.

## Reproduction Assessment

Yes for the PR regression by source inspection: call phone-control with `channel: "telegram"` and no `gatewayClientScopes`; current main tests expect arm/disarm to work, while the PR branch returns `requires operator.admin`. The claimed bypass is not established as a current-main bug because authorized external channel callers are an intentional tested path.

## Solution Assessment

No. The PR conflates internal gateway client scopes with external sender authorization; the narrower maintainable solution is the current helper, or a new explicit owner/channel policy if maintainers choose that product direction.
