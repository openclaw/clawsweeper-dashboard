## Summary

Keep open: the patch still appears to regress native Telegram phone-control commands by treating absent internal gateway scopes as denial, while current main has a narrower guard and tests that preserve no-scope native Telegram behavior.

## What This Changes

The PR makes phone-control `arm` and `disarm` require `operator.admin` by checking `gatewayClientScopes` unconditionally and rewrites two tests to expect Telegram no-scope callers to be rejected.

## Best Possible Solution

Preserve the current gateway-scope hardening unless maintainers deliberately choose stricter native-channel phone-control authorization; if they do, implement explicit owner/channel authorization with live Telegram proof.

## Reproduction Assessment

Yes for the PR-introduced regression by source inspection: native Telegram command execution does not pass `gatewayClientScopes`, so this patch would reject authorized Telegram `/phone arm` and `/phone disarm`. No high-confidence current-main bypass is established as written because current tests deliberately split native no-scope callers from scoped gateway callers.

## Solution Assessment

No. The patch conflates internal control-plane gateway scopes with external chat authorization; a stricter policy should be an explicit maintainer-approved owner/channel authorization change with live proof.
