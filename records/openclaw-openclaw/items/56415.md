## Summary

Keep open for maintainer handling, but do not merge this PR as-is. Current main already blocks non-admin scoped gateway callers while preserving authorized external channel commands; this PR would treat ordinary channel commands without internal gateway scope metadata as non-admin and regress Telegram/native command use.

## What This Changes

The PR changes phone-control arm/disarm authorization to require `operator.admin` on every channel and updates tests to expect Telegram callers without that scope to be rejected.

## Best Possible Solution

Keep the current scoped-gateway hardening unless maintainers deliberately choose a stricter external-channel policy; if they do, implement it through owner identity or channel command authorization with tests for WebChat, scoped gateway callers, and normal external senders.

## Reproduction Assessment

Yes. Invoke the phone-control command handler with `channel: "telegram"` and no `gatewayClientScopes`: current main's tests expect `/phone arm` and `/phone disarm` to work for authorized channel callers, while this PR would return `requires operator.admin`.

## Solution Assessment

No. The PR conflates internal gateway client scopes with external sender authorization; the narrower maintainable fix is the current helper, or a new explicit owner/channel policy if product direction changes.
