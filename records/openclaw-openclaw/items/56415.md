## Summary

Keep this PR open but do not merge it as-is. Current main intentionally distinguishes internal gateway scopes from external channel sender authorization, and the PR would reject normal authorized Telegram/native `/phone` callers that do not carry `gatewayClientScopes`.

## What This Changes

The PR removes the webchat-only condition from phone-control arm/disarm admin checks and updates tests to expect Telegram callers without gateway scopes to be rejected.

## Best Possible Solution

Keep the current scoped-gateway behavior unless maintainers deliberately choose stricter external-channel control; if they do, implement it through explicit owner/channel authorization with WebChat, scoped gateway, and native channel tests.

## Reproduction Assessment

Yes for the PR regression by source inspection: call phone-control with `channel: "telegram"` and no `gatewayClientScopes`; current main tests expect arm/disarm to work, while the PR branch returns `requires operator.admin`. The claimed bypass is not established as a current-main bug because authorized external channel callers are an intentional tested path.

## Solution Assessment

No. The PR conflates internal gateway client scopes with external sender authorization; the narrower maintainable path is the current helper, or a new explicit owner/channel policy if maintainers choose that direction.
