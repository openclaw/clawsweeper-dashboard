## Label Justifications

- `P2`: This is a meaningful automation feature with a clear affected surface, but it is not a current regression or outage.
- `impact:security`: The requested direct CLI can trigger session orchestration from scripts and must preserve existing Gateway auth and dangerous-tool policy.
- `impact:session-state`: The feature directly creates child agent sessions and affects session ownership, lifecycle, and child-limit semantics.

## Summary

Keep open. Current main and the latest release already have direct Gateway HTTP/RPC tool invocation plus the underlying `sessions_spawn` tool, but the narrowed script-friendly CLI surface is still absent, and the related umbrella/direct-tool work is open rather than a superseding implementation.

## What This Changes

Request a direct scriptable API or CLI surface for spawning subagent sessions without routing through an intermediate agent turn.

## Best Possible Solution

Choose a conservative CLI design that reuses the existing Gateway invoke contract, keeps dangerous session orchestration opt-in through current policy, and adds focused CLI docs plus policy tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection confirms the current gap because direct Gateway invoke exists, but no `openclaw sessions spawn` or `openclaw tools invoke` CLI command is registered on current main or `v2026.5.18`.

## Solution Assessment

Unclear until maintainers choose the product surface. The maintainable direction is a thin CLI wrapper over existing Gateway direct invoke, not a separate session-spawn backend or policy bypass.
