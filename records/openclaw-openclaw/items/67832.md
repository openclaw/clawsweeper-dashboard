## Summary

Keep open. The reporter narrowed this to a script-friendly CLI surface; current main has HTTP/RPC direct tool invocation, low-level `gateway call`, and chat `/subagents spawn`, but still no dedicated `openclaw sessions spawn` or `openclaw tools invoke`, and exposing session orchestration from scripts needs maintainer product/security choice.

## What This Changes

Expose direct session spawning to external scripts, preferably through `openclaw sessions spawn` or a generic `openclaw tools invoke`, without requiring an intermediate agent turn.

## Best Possible Solution

Keep the issue open and choose a conservative CLI design that reuses the existing Gateway `tools.invoke` or `/tools/invoke` contract, preserves auth/tool policy/default-deny behavior, and updates CLI docs plus targeted command and Gateway-policy tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The feature gap is source-confirmed: current main documents and implements HTTP/RPC direct invoke but registers no `openclaw sessions spawn` or `openclaw tools invoke` command.

## Solution Assessment

Unclear until maintainers choose the public CLI shape. The narrow maintainable direction is a thin wrapper over the existing Gateway direct-invoke path, not a separate session-spawn implementation or policy bypass.
