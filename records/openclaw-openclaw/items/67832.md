## Summary

Keep open. Current main already has direct Gateway tool invocation over HTTP and RPC, but the narrowed CLI request is still missing and the remaining surface needs an explicit maintainer product/security decision before implementation.

## What This Changes

Request a direct CLI/API path for spawning sessions, narrowed by the reporter to a CLI command such as `openclaw sessions spawn` or `openclaw tools invoke`.

## Best Possible Solution

Keep the issue open and design a thin, documented CLI wrapper over the existing Gateway invoke contract that preserves auth, default-deny policy, spawn limits, and focused CLI/Gateway tests.

## Reproduction Assessment

Yes. Source inspection of current main confirms the underlying `sessions_spawn` tool and Gateway invoke APIs exist, while no `openclaw sessions spawn` or `openclaw tools invoke` command is registered or documented.

## Solution Assessment

Unclear until maintainers choose the CLI shape and policy. The maintainable direction is to reuse `/tools/invoke` or `tools.invoke` rather than adding a second independent session-spawn implementation.
