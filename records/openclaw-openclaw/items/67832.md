## Summary

Keep open. Current main has direct Gateway tool invocation over HTTP and RPC, but the narrowed user-facing CLI path is still missing, and exposing direct session spawning needs maintainer security/product choice before automation should implement it.

## What This Changes

The issue requests a direct session-spawn interface, narrowed by follow-up discussion to either `openclaw sessions spawn` or a generic `openclaw tools invoke` CLI because HTTP `/tools/invoke` already exists.

## Best Possible Solution

Keep the issue open and design a thin, documented CLI surface that reuses the existing Gateway invoke path, preserves auth and dangerous-tool policy, enforces subagent limits, and adds focused CLI/Gateway tests plus docs.

## Reproduction Assessment

Yes. Source-level verification is high confidence: inspect current CLI registrations and public CLI docs to confirm no `sessions spawn` or `tools invoke` command exists while Gateway HTTP/RPC tool invocation and the `sessions_spawn` agent tool do exist.

## Solution Assessment

Unclear until maintainers choose the command shape. The safest maintainable direction is a thin CLI wrapper around the existing `/tools/invoke` or `tools.invoke` contract, not a second independent spawn implementation.
