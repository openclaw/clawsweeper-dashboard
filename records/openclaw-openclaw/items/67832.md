## Label Justifications

- `P2`: This is a normal-priority feature gap with useful automation value but existing HTTP/RPC/SDK and low-level CLI workarounds.
- `impact:security`: The requested CLI would expose direct session orchestration and tool invocation outside an LLM turn, which crosses Gateway auth and RCE-class policy boundaries.
- `impact:session-state`: The feature is specifically about spawning and managing child session state from external automation.

## Summary

Keep open. Current main already provides direct HTTP/RPC/SDK tool invocation and a low-level `gateway call tools.invoke` CLI path, but it still lacks the first-class `sessions spawn` or `tools invoke` CLI/docs surface requested in the narrowed issue; exposing that shortcut is a security-sensitive product decision, not cleanup closure.

## What This Changes

Request a direct API or CLI path for spawning sessions from scripts without an intermediate agent turn.

## Best Possible Solution

Decide whether a dedicated `sessions spawn` command or a generic `tools invoke` command should be added, and if so implement it as a thin documented wrapper over existing Gateway `tools.invoke` while preserving default-deny policy and operator auth semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection give high confidence that direct invoke primitives exist, while the first-class `sessions spawn` or `tools invoke` CLI surface is still absent.

## Solution Assessment

Unclear until maintainers choose the public CLI shape. The maintainable direction is a thin wrapper over the existing Gateway `tools.invoke` path, not a new session-spawn backend or any bypass of Gateway policy.
