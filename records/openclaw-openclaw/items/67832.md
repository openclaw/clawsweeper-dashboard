## Summary

Keep open. Current main already has HTTP/RPC direct tool invocation and the `sessions_spawn` tool, but it still lacks the narrowed script-friendly CLI surface; the broader direct-tool-invocation issue at https://github.com/openclaw/openclaw/issues/37131 is an umbrella that points back to this issue as a scoped follow-up rather than superseding it.

## What This Changes

The request asks for a direct CLI/API surface to spawn subagent sessions from scripts without routing through an intermediate agent turn.

## Best Possible Solution

Keep the issue open and choose a conservative CLI design that reuses the existing Gateway invoke contract, preserves auth/tool policy and default-deny behavior, and adds focused CLI docs and policy tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection confirms the gap: current main implements HTTP/RPC direct invoke but registers no `openclaw sessions spawn` or `openclaw tools invoke` command.

## Solution Assessment

Unclear until maintainers choose the product surface. The narrow maintainable approach is a thin CLI wrapper over existing Gateway direct invoke, not a new session-spawn backend or policy bypass.

Status: needs_attention
