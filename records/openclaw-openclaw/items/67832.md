## Summary

Keep open. The reporter narrowed this to a discoverable direct CLI for session spawning or generic tool invocation; current main has HTTP/RPC direct invoke and a low-level `gateway call`, but still no `openclaw sessions spawn` or `openclaw tools invoke` command, and the remaining CLI surface is security-sensitive.

## What This Changes

The issue requests a direct API/CLI surface for spawning sessions without routing through an intermediate agent turn.

## Best Possible Solution

Choose a conservative CLI product shape that reuses the existing Gateway `tools.invoke` or `/tools/invoke` contract, keeps `sessions_spawn` opt-in via existing policy, and documents/tests the operator workflow.

## Reproduction Assessment

Yes for the requested feature gap: current source and docs show the direct HTTP/RPC invoke paths and the low-level `gateway call`, while no `openclaw sessions spawn` or `openclaw tools invoke` command is registered or documented. This is not a broken existing behavior reproduction; it is source-confirmed missing CLI surface.

## Solution Assessment

Unclear until maintainers choose the public CLI shape. The safest implementation direction is a thin wrapper over the existing Gateway invoke path, not a second independent session-spawn implementation or a policy bypass.
