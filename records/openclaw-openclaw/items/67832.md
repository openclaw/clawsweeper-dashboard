## Summary

Keep open. Current main has direct Gateway tool invocation over HTTP/RPC and a low-level `openclaw gateway call tools.invoke` path, but it still lacks the discoverable `openclaw sessions spawn` or `openclaw tools invoke` CLI surface requested in the reporter's follow-up; the related #76145 tracks a different subagent completion/output bug.

## What This Changes

Request a direct session-spawn API/CLI surface, narrowed by the reporter to a user-facing CLI command because HTTP `/tools/invoke` already exists.

## Best Possible Solution

Decide and document a thin CLI path over the existing Gateway `tools.invoke` contract, or explicitly document `gateway call tools.invoke` as the supported low-level path, while preserving all current auth and dangerous-tool policy checks.

## Reproduction Assessment

Yes. Source inspection on current main verifies the underlying HTTP/RPC invoke and `sessions_spawn` tool exist, while no dedicated `sessions spawn` or top-level `tools invoke` CLI command is registered or documented.

## Solution Assessment

Unclear. The safest implementation direction is a thin wrapper or explicit docs over the existing Gateway invoke path, but maintainers still need to decide whether the low-level `gateway call tools.invoke` path is sufficient or whether a discoverable command is warranted.
