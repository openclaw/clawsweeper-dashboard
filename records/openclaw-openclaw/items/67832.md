## Summary

Current main still lacks the narrowed direct CLI path. The Gateway HTTP tool-invoke endpoint exists and can reach `sessions_spawn` only after explicit Gateway allowlist configuration, but `openclaw sessions` has no `spawn` subcommand and the CLI command catalogs have no generic `tools invoke` command. This should stay open for maintainer product/security review rather than conservative cleanup closure.

## What This Changes

The issue requests a direct CLI or API interface for spawning sessions from scripts without an intermediate agent turn, later narrowed to a CLI path because the HTTP `/tools/invoke` endpoint already exists.

## Best Possible Solution

Keep the issue open and settle a conservative CLI design that reuses the existing `/tools/invoke` contract, keeps `sessions_spawn` opt-in through Gateway tool policy, and documents the operator/security boundary before adding command coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Current-main source and docs show the HTTP workaround exists, but no `openclaw sessions spawn` or `openclaw tools invoke` CLI surface is registered or documented.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the CLI shape. The safest direction appears to be a thin Gateway-authenticated wrapper around `/tools/invoke`, not a second independent spawn implementation.
