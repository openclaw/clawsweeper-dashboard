## Summary

Keep open. Current main has the direct HTTP `/tools/invoke` endpoint and now also an SDK-facing `tools.invoke` Gateway RPC, but it still lacks the reporter's narrowed user-facing CLI path: no `openclaw sessions spawn` and no generic `openclaw tools invoke` command. Because `sessions_spawn` is explicitly treated as a high-risk session-orchestration/RCE surface, the remaining CLI design needs maintainer product/security choice before conservative cleanup or an automated fix PR.

## What This Changes

Request to add a direct session-spawn interface, narrowed by the reporter to a CLI command or generic tool-invoke CLI because HTTP `/tools/invoke` already exists.

## Best Possible Solution

Keep the issue open and design a thin, documented CLI surface that reuses the existing Gateway tool-invoke path, preserves Gateway auth, tool policy, default dangerous-tool deny behavior, subagent target limits, and sandbox checks, then add focused CLI/Gateway policy tests plus CLI docs.

## Reproduction Assessment

Yes. Current-main source and docs provide a high-confidence reproduction path: inspect the command registration/catalogs and public CLI docs to verify no `sessions spawn` or `tools invoke` command exists, while Gateway HTTP/RPC tool invocation and the `sessions_spawn` agent tool do exist.

## Solution Assessment

Unclear until maintainers choose the command shape. The safest maintainable direction appears to be a thin CLI wrapper around the existing Gateway `/tools/invoke` or `tools.invoke` contract, not a second independent spawn implementation.
