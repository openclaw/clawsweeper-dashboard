## Summary

Keep open: current main still renders managed inbound attachments as prompt-visible `media://inbound/...` refs, and only built-in consumers such as `read`, `pdf`, and media-understanding partially resolve those refs. The reported shell/custom document-reader path remains unresolved and needs a maintainer/security decision around path privacy.

## What This Changes

The issue requests that inbound non-image channel attachments expose a resolvable path or media-ref resolver so PDFs and documents can be read by agents and tools.

## Best Possible Solution

Keep this issue open as the canonical media prompt/tool-boundary gap, then implement a scoped contract that lets document readers resolve managed inbound attachments without broadly leaking host paths into prompts.

## Reproduction Assessment

Yes, source-level: current main renders managed inbound media-store paths as `media://inbound/...` in prompt notes and does not inline non-image files into the current agent turn. I did not run a live Telegram attachment scenario in this read-only review.

## Solution Assessment

No to unconditional absolute-path emission as the whole solution. The maintainable fix needs maintainer/security agreement on whether to resolve `media://` at tool boundaries, expose a narrower prompt fallback, or preserve prompt privacy with another explicit document-reader contract.
