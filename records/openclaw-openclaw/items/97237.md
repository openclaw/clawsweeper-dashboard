## Summary

Keep open: current main and the latest release still render managed inbound media notes as `media://inbound/<id>`, which remains unusable for shell/custom document-reader workflows even though built-in `read`/`pdf` consumers can resolve some refs. The open related PR is only partial and still leaves the PDF path/privacy question unresolved.

## What This Changes

The issue requests a usable way for agents to open inbound non-image attachments instead of seeing only prompt-visible `media://inbound/...` refs.

## Best Possible Solution

Choose a scoped media-reference contract that keeps managed-consumer privacy intact while giving shell/custom document-reader workflows a safe resolver or narrowly guarded readable path, with regression tests for prompt notes, built-in read/pdf tools, and channel attachment flow.

## Reproduction Assessment

Yes, source-level: current main rewrites managed inbound paths to `media://inbound/<id>` in the prompt note and only image attachments are current-turn inlined. I did not run a live Telegram bot reproduction.

## Solution Assessment

No to unconditional absolute-path emission for all PDFs/documents: built-in `read` and `pdf` already support managed refs, so the safer fix needs a scoped prompt fallback or tool-boundary resolver rather than broad path exposure.
