## Summary

Keep open: current main and the latest release still expose managed inbound media-store paths to the agent prompt as `media://inbound/<id>`, and Telegram prompt media context intentionally canonicalizes inbound media paths to that URI form. Built-in read/PDF/media consumers can resolve some `media://` refs, but the reported shell/custom-helper path remains unresolved, and the open related PR only partially addresses binary documents while preserving PDF URI behavior.

## What This Changes

The issue reports that Telegram inbound PDFs/documents are downloaded locally but surfaced to the model only as `media://inbound/...`, leaving shell/document helpers without a usable filesystem path.

## Best Possible Solution

Keep this issue open and settle a maintainer-approved media-reference contract: either provide a safe, usable file handle for non-image inbound attachments or make `media://inbound` resolvable at the relevant tool boundary without broadly leaking host paths.

## Reproduction Assessment

Yes, source-level: current main rewrites managed inbound absolute paths to `media://inbound/<id>` and prepends that note into agent prompt bodies. I did not run a live Telegram PDF turn in this read-only review.

## Solution Assessment

Unclear as proposed: emitting absolute paths would unblock shell helpers but has a privacy/compatibility tradeoff. A narrower tool-boundary resolver or MIME-scoped fallback needs maintainer approval before an implementation path is best.
