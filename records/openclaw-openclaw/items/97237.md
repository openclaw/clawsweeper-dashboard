## Summary

Keep open: current main and v2026.6.10 still expose managed inbound attachments as prompt-visible `media://inbound/<id>` refs, and existing built-in `read`/`pdf` support only partially addresses the reported shell/custom document-reader workflow. The related open PR is useful overlap, but it intentionally preserves PDF URI behavior and is not a merged full fix.

## What This Changes

The issue reports that inbound PDFs/documents can be stored on disk but shown to the model only as `media://inbound/...`, which shell/custom document readers cannot open.

## Best Possible Solution

Keep this as the canonical media prompt/tool-boundary issue, then choose a scoped fix that either resolves `media://` for shell/file helpers or exposes guarded real paths only where no managed consumer exists, with regression tests and explicit path-privacy review.

## Reproduction Assessment

Yes, source-level: current main maps managed inbound paths to `media://inbound/<id>` in prompt notes and only image attachments are inlined into the current turn. I did not run a live Telegram bot reproduction.

## Solution Assessment

No to unconditional absolute-path emission for all PDFs/documents: existing `read` and `pdf` consumers already support managed refs, so the safer fix needs a scoped prompt fallback or tool-boundary resolver rather than broad path exposure.
