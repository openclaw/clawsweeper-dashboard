## Summary

Keep open: current main and v2026.6.10 still preserve prompt-visible managed inbound attachments as `media://inbound/<id>` in key paths, and built-in `read`/`pdf` support does not fully solve shell/custom document-reader workflows. The likely fix is security-sensitive because exposing host paths changes the current path-privacy contract.

## What This Changes

The issue asks OpenClaw to make inbound non-image attachments, especially Telegram PDFs, resolvable by agents instead of exposing only an unopenable `media://inbound/...` prompt URI.

## Best Possible Solution

Keep this as the canonical media prompt/tool-boundary issue, then choose a scoped fix that either resolves `media://` for shell/file helpers or exposes guarded real paths only where no managed consumer exists, with regression tests and explicit path-privacy review.

## Reproduction Assessment

Yes, source-level: current main rewrites managed inbound media paths to `media://inbound/<id>` in prompt notes and only image attachments are inlined into the current turn. I did not run a live Telegram bot reproduction.

## Solution Assessment

No to unconditional absolute-path emission for all PDFs/documents: existing `read` and `pdf` consumers already support managed refs, so the safer fix needs a scoped prompt fallback or tool-boundary resolver rather than broad path exposure.
