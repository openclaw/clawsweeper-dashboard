## Summary

Keep open: current main and v2026.6.10 still render managed inbound paths as model-visible `media://inbound/<id>` notes on the channel prompt path, and non-image attachments are not inlined into the current turn. Built-in `pdf`/`read` support handles some managed refs, but that does not solve the reported shell/custom document-reader workflow, and the open related PR is only a partial binary-document path-visibility proposal.

## What This Changes

The issue asks OpenClaw to make inbound non-image attachments, especially PDFs/documents from Telegram, reachable by the agent instead of exposing only an unresolvable `media://inbound/...` prompt reference.

## Best Possible Solution

Keep this issue open as the canonical path-privacy/tool-boundary decision, then implement a scoped resolver or guarded prompt fallback that makes unsupported document workflows usable without broadly leaking host paths.

## Reproduction Assessment

Yes, source-level: current main maps managed inbound paths to `media://inbound/<id>` in prompt notes and only image attachments are inlined into the current turn. I did not run a live Telegram attachment scenario.

## Solution Assessment

No to unconditional absolute-path emission for all PDFs/documents. The safer fix should either resolve `media://` at the relevant tool boundary or expose paths only for unsupported document cases after maintainers accept the path-privacy contract.
