## Label Justifications

- `P2`: The issue blocks affected attachment-reading workflows but is bounded to inbound media prompt/tool handling rather than a core outage.
- `impact:message-loss`: Inbound attachment content can reach storage while the agent receives only a reference that some tool workflows cannot use.

## Summary

Keep open: current main still renders managed inbound media as `media://inbound/<id>` in the prompt, and while built-in `read`/`pdf` paths now understand those refs, the reported shell/custom document-reader workflow remains unresolved and has an open partial PR rather than a merged fix.

## What This Changes

The issue asks OpenClaw to make inbound non-image channel attachments, especially PDFs/documents, usable by agents instead of exposing only an unresolved `media://inbound/...` prompt reference.

## Best Possible Solution

Keep this as the canonical prompt/tool-boundary bug, then choose a scoped fix that either resolves `media://` for shell/file helpers or exposes guarded real paths only for MIME types without managed consumers while preserving PDF/read-tool support and path privacy where intended.

## Reproduction Assessment

Yes, source-level: current main maps managed inbound paths to `media://inbound/<id>` in prompt notes and only image attachments are inlined into the current turn. I did not run a live Telegram bot repro.

## Solution Assessment

No to unconditional absolute-path emission for PDFs: existing `read` and `pdf` consumers already support `media://inbound/<id>`, so the safer fix needs a scoped prompt fallback or tool-boundary resolver rather than broad path exposure.
