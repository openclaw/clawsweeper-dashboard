## Summary

Keep open: current main and v2026.6.10 still render managed inbound media-store attachments as prompt-visible `media://inbound/<id>` refs, and current-turn attachment inlining still skips non-images. Built-in `read` and `pdf` support only partially addresses this because the reported custom shell/document-reader workflow still receives an opaque URI, while the related open PR is partial overlap and intentionally leaves PDFs on the URI path.

## What This Changes

The issue requests that non-image inbound attachments, especially PDFs/docs delivered over channels, be exposed to agents through a resolvable path or resolver instead of only `media://inbound/...`.

## Best Possible Solution

Keep this as the canonical media prompt/tool-boundary issue, then choose a scoped fix that either resolves `media://` for shell/file helpers or exposes guarded real paths only where no managed consumer exists, with regression tests and explicit path-privacy review.

## Reproduction Assessment

Yes, source-level: current main maps managed inbound paths to `media://inbound/<id>` in prompt notes and only image attachments are inlined into the current turn. I did not run a live Telegram bot reproduction in this read-only review.

## Solution Assessment

No to unconditional absolute-path emission for all PDFs/documents: existing `read` and `pdf` consumers already support managed refs, so the safer fix needs a scoped prompt fallback or tool-boundary resolver rather than broad path exposure.
