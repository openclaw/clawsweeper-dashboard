## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: The files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and the persisted searchable index for that agent.

## Summary

Keep open: current main and v2026.6.8 still only start built-in memory watchers and catch-up after a manager exists, status dirtiness remains flag-based, and the only related merged PR covers session transcripts rather than `memory/` files.

## What This Changes

The issue asks OpenClaw to detect files written to an uninvoked agent's `memory/` directory and to make memory status report stale disk/index state honestly.

## Best Possible Solution

Add a bounded memory-core/gateway reconciliation policy for enabled agents that detects missing or changed `memory` source files without unbounded watcher startup, then make status report disk/index drift and cover it with focused regression tests.

## Reproduction Assessment

Yes, at source level: an uninvoked built-in-memory agent has no default `MemoryIndexManager`, so no watcher, interval timer, or startup catch-up exists to observe new `memory/` files. I did not run a live gateway fixture for the exact terminal display.

## Solution Assessment

Unclear as proposed: a catch-up scan is likely the right direction, but maintainers should choose the boundary because startup watchers, periodic scans, and status-only drift checks have different resource costs.
