## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: Files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and the persisted searchable memory index for that agent.

## Summary

Keep open. Current main still lacks built-in-memory startup or periodic reconciliation for `memory/` files in uninvoked agents; the closest related work covers session transcripts or watcher latency, not this exact `memory/` file drift.

## What This Changes

The issue reports that files written to an uninvoked agent's `memory/` directory are not indexed or marked dirty until that agent's memory manager starts.

## Best Possible Solution

Implement a bounded memory-core or gateway reconciliation policy that detects missing or changed `memory/` files for enabled agents without unbounded watcher startup, and make status surface disk/index drift honestly.

## Reproduction Assessment

Yes, at source level. An uninvoked built-in-memory agent has no default `MemoryIndexManager`, so no watcher or interval exists to observe new `memory/` files; I did not run a live fixture for the exact `Indexed: N/N` display detail.

## Solution Assessment

Unclear as proposed. A catch-up or drift scan is the right direction, but maintainers should choose the reconciliation boundary because startup watchers, periodic scans, and status-only checks have different resource costs.
