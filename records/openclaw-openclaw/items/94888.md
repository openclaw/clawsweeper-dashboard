## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and the persisted searchable index for that agent.
- `impact:data-loss`: Files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.

## Summary

Keep open. Current main still has no built-in-memory startup or periodic disk reconciliation for ordinary `memory/` files in uninvoked agents; the related merged session catch-up work covers the `sessions` source only, not this memory-file path.

## What This Changes

The issue reports that files written into an uninvoked agent's `memory/` directory can remain absent from memory search while status does not mark the index dirty.

## Best Possible Solution

Add a bounded memory-core reconciliation policy for enabled agents that detects disk/index drift without unbounded watcher startup, and make status expose stale `memory` source indexes clearly.

## Reproduction Assessment

Yes, at source level: an uninvoked built-in-memory agent has no default manager, so no watcher or interval exists to observe new `memory/` files, and later interval sync does not scan clean indexes unless a dirty/full/force condition exists. I did not run a live gateway fixture in this read-only review.

## Solution Assessment

Unclear. A catch-up scan is a plausible direction, but maintainers should choose the reconciliation boundary because startup watchers, periodic scans, and status-only drift checks have different resource costs.
