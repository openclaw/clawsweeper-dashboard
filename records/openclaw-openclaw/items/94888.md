## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with a manual reindex workaround and limited blast radius in multi-agent deployments.
- `impact:data-loss`: Files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and the persisted searchable index for that agent.

## Summary

Keep open. Current main still leaves builtin memory managers lazy for uninvoked agents, the gateway startup helper only eagerly handles QMD managers, and status dirty state is still flag-based rather than a disk-versus-index comparison.

## What This Changes

The issue requests reliable indexing and honest status reporting for files added to an uninvoked agent's builtin `memory/` directory.

## Best Possible Solution

Implement a bounded builtin-memory reconciliation policy that detects missing or changed `memory` files for enabled agents without unbounded watcher startup, and make status report disk/index drift explicitly.

## Reproduction Assessment

Yes, at source level: an uninvoked builtin-memory agent has no default manager, watcher, or interval timer, and status-only managers do not compute dirty by comparing disk to index. I did not run a live gateway fixture for the exact status text.

## Solution Assessment

Unclear as proposed: a catch-up scan is likely the right direction, but maintainers should choose the reconciliation boundary because startup watchers, periodic scans, and status-only drift checks have different resource costs.
