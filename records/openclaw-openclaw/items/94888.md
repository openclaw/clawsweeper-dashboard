- `impact:data-loss`: The on-disk files remain present, but they can be silently absent from memory search, which is practical loss of indexed user memory.

## Summary

Keep open. Current main still leaves the built-in memory manager lazy for ordinary `memory/` files, so uninvoked agents do not have a watcher or interval running; the merged sessions catch-up work is adjacent but only covers the `sessions` source.

## What This Changes

The issue reports that `.md` files written into an uninvoked agent's `memory/` directory can remain unindexed while `openclaw memory status` reports a clean dirty state.

## Best Possible Solution

Implement a bounded memory-core or gateway reconciliation policy that detects missing or changed ordinary `memory` files for enabled agents without unbounded watcher startup, and make status surface disk/index drift honestly.

## Reproduction Assessment

Yes, at source level: an uninvoked built-in memory agent has no default `MemoryIndexManager`, so no watcher or interval exists to observe new `memory/` files. I did not run a live fixture for the exact CLI `Indexed` line.

## Solution Assessment

Unclear as proposed: a catch-up scan is the right direction, but maintainers should choose the reconciliation boundary because startup watchers, periodic scans, and status drift checks have different resource costs.
