## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: Files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and the persisted searchable memory index for that agent.

## Summary

Keep open. Current main still has the reported built-in memory gap: uninvoked agents do not have a long-lived memory manager watching their workspace, default interval sync is disabled, and status dirty state is not a disk-versus-index reconciliation.

## What This Changes

Report that memory files written to an uninvoked agent's `memory/` directory can remain unindexed while `openclaw memory status --agent <id>` can still report a clean index.

## Best Possible Solution

Add a bounded memory-core or gateway reconciliation policy that detects missing or changed `memory/` files for enabled agents without unbounded eager watchers, and make status report disk/index drift honestly.

## Reproduction Assessment

Yes, at source level. An uninvoked built-in-memory agent has no default `MemoryIndexManager`, so no watcher or interval exists to observe new `memory/` files; I did not run a live fixture for the exact terminal display.

## Solution Assessment

Unclear as proposed. A catch-up or drift scan is the right direction, but maintainers should choose the reconciliation boundary because startup watchers, periodic scans, and status-time checks have different resource costs.
