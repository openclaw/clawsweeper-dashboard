## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: Memory files remain on disk but can be silently absent from search results, which is practical loss of indexed user memory.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and that agent's persisted searchable memory index.

## Summary

Keep open. Current main and the latest v2026.6.8 release still only start built-in memory watchers after a per-agent manager is constructed, while gateway startup covers QMD startup work and the merged sessions catch-up path does not reconcile workspace `memory/` files for uninvoked agents.

## What This Changes

The issue asks OpenClaw to index files written into uninvoked agents' `memory/` directories and to report disk/index drift honestly in memory status.

## Best Possible Solution

Add a bounded memory-core/gateway reconciliation policy that detects missing or changed workspace `memory/` files for enabled agents without unbounded watcher startup, and make status surface disk/index drift explicitly.

## Reproduction Assessment

Yes, at source level: an uninvoked built-in memory agent has no default manager instance, so no watcher or interval exists to observe new `memory/` files until a later invocation or explicit CLI/index path constructs one. I did not run a live multi-agent fixture, so the exact `Indexed: N/N` display remains unconfirmed.

## Solution Assessment

Unclear as proposed: a catch-up scan is the right repair direction, but eagerly constructing every watcher is not obviously the best fix because current watcher code already has file-descriptor pressure safeguards. Maintainers should choose between startup scan, periodic reconciliation, and status drift checks before implementation.
