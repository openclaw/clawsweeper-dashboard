- `impact:data-loss`: Files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.

## Summary

Keep open. Current main still leaves built-in memory managers lazy for uninvoked agents, so `memory/` file writes can miss watcher/interval reconciliation; the related merged work only covers the `sessions` source.

## What This Changes

The issue reports that memory files written for an uninvoked agent are not indexed or surfaced as dirty until that agent's memory manager starts.

## Best Possible Solution

Implement a bounded memory-core or gateway reconciliation policy that detects `memory/` disk/index drift for enabled agents without forcing unbounded watcher startup, and make status report that drift clearly.

## Reproduction Assessment

Yes at source level: an uninvoked built-in memory agent has no default `MemoryIndexManager`, so no watcher or interval exists to observe new `memory/` files. I did not run a live fixture for the exact status text.

## Solution Assessment

Unclear as proposed: catch-up reconciliation is the right direction, but maintainers should choose whether it belongs in gateway startup, periodic manager scan, or status drift detection because those choices have different resource costs.
