## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: Files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and the persisted searchable index for that agent.

## Summary

Keep open. Current main still has no built-in-memory startup or periodic reconciliation for `memory/` files in uninvoked agents; only non-transient managers start watchers, and the related sessions catch-up work does not cover the `memory` source.

## What This Changes

Ensure memory files written for uninvoked agents are detected, indexed, and reported as stale when disk and index drift.

## Best Possible Solution

Implement a bounded memory-core/gateway reconciliation policy that detects missing or changed `memory/` files for enabled agents without unbounded watcher startup, and make status surface disk/index drift.

## Reproduction Assessment

Yes, at source level: an uninvoked built-in memory agent has no default `MemoryIndexManager`, so no watcher or interval exists to observe new `memory/` files. I did not run a live fixture, and the exact `Indexed: N/N` detail still needs one.

## Solution Assessment

Unclear as proposed: a catch-up scan is the right direction, but maintainers should choose the reconciliation boundary because startup watchers, periodic scans, and status-only drift checks have different resource costs.
