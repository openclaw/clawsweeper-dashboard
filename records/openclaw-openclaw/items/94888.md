## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: Memory files remain on disk but can be silently absent from retrieval, which is practical loss of indexed user memory.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and the persisted searchable index for that agent.

## Summary

Keep open. Current main and v2026.6.8 still leave the source-supported gap: memory-file watchers and interval sync only exist after a non-transient memory manager is constructed, while related work covers session catch-up or already-started-manager fallback rather than memory files for agents that have not been invoked.

## What This Changes

The issue requests reliable indexing and honest dirty status for memory files written to an agent whose memory manager has not been started in the current gateway session.

## Best Possible Solution

Define a bounded memory-core or gateway reconciliation policy that makes memory-file disk/index drift visible and recoverable without blindly starting unbounded watchers for every possible agent.

## Reproduction Assessment

Yes, at source level: an uninvoked agent has no non-transient MemoryIndexManager, so no memory watcher or manager-local interval exists to observe new files. I did not run a live fixture, and the exact `Indexed: N/N` display detail remains partially unproven from source alone.

## Solution Assessment

Unclear as proposed: catch-up scanning is the right repair direction, but maintainers need to choose the owner boundary between gateway startup preload, per-manager interval reconciliation, and status-time drift detection. The current source is not already a complete solution.
