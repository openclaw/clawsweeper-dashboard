## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: Files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and the persisted searchable index for that agent.

## Summary

Keep open. Current main still leaves the reported built-in memory path unresolved: default managers own the watcher/interval lifecycle, gateway startup only initializes QMD memory managers, and status dirty is still derived from manager flags rather than a disk/index comparison.

## What This Changes

The issue reports that built-in `memory/` files written for an uninvoked agent remain unindexed and can still show a clean memory status.

## Best Possible Solution

Adopt a bounded memory-core or gateway reconciliation policy for enabled built-in memory agents that catches missing or changed `memory/` files and makes status report disk/index drift without unbounded watcher startup.

## Reproduction Assessment

Yes, at source level: an uninvoked built-in memory agent has no default `MemoryIndexManager`, so no watcher or interval exists to observe new `memory/` files. I did not run a live fixture, so the exact `Indexed: N/N` output detail remains unproven.

## Solution Assessment

Unclear as proposed: a catch-up scan and honest dirty signal are the right direction, but maintainers should choose between startup managers, periodic scans, and status drift checks because they have different resource costs.
