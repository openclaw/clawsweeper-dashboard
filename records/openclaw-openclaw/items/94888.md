## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: Files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.
- `impact:session-state`: The issue concerns drift between an agent's on-disk memory state and its persisted searchable index.

## Summary

Keep open. Current main and the latest release still only start built-in memory watchers after a non-transient manager is created; gateway startup eager work is QMD-only, and the merged sessions catch-up path does not reconcile workspace `memory/` files for uninvoked agents.

## What This Changes

The issue requests that memory files written for an uninvoked agent be indexed or reported as stale without requiring that agent to be invoked first.

## Best Possible Solution

Implement a bounded memory-core/gateway reconciliation policy that detects missing or changed `memory/` files for enabled agents without unbounded watcher startup, and make status surface disk/index drift.

## Reproduction Assessment

Yes, at source level: an uninvoked built-in memory agent has no default `MemoryIndexManager`, so no watcher or interval exists to observe new `memory/` files. I did not run a live fixture for the exact CLI `Indexed` count detail.

## Solution Assessment

Unclear as proposed: catch-up reconciliation is the right direction, but maintainers should choose the boundary because startup watchers, periodic scans, and status-only drift checks have different resource costs.
