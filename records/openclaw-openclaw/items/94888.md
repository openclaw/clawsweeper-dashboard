## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded multi-agent impact.
- `impact:data-loss`: Memory files can remain on disk while silently absent from memory search, which is practical loss of indexed user memory.
- `impact:session-state`: The persisted index can drift from an agent's on-disk memory state depending on gateway-session manager lifetime.

## Summary

Keep open. Current main still has no startup or periodic reconciliation for built-in `memory` files in uninvoked agents; the merged sessions catch-up work is adjacent but only covers the `sessions` source after a manager starts.

## What This Changes

The issue reports that `.md` memory files written for an uninvoked agent are not indexed because no memory manager watcher is alive for that agent.

## Best Possible Solution

Choose and implement a bounded memory-core reconciliation policy for source `memory` that detects missing or changed disk files for enabled agents without unbounded eager watcher startup unless maintainers explicitly accept that cost.

## Reproduction Assessment

Yes at source level: current main only starts source `memory` watchers after a non-transient manager exists, and it has no source `memory` startup catch-up. I did not run a live gateway fixture in this read-only review, so the exact status rendering remains unverified.

## Solution Assessment

Unclear as a final implementation path: a catch-up scan is plausible, but maintainers should choose the resource boundary between startup scans, periodic scans, eager watchers, and status-only drift reporting.
