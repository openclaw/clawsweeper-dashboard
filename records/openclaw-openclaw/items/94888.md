## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: Memory files remain on disk but can be silently absent from retrieval, which is practical loss of indexed user memory.
- `impact:session-state`: The issue is about drift between an agent's on-disk memory state and the persisted searchable index for that agent.

## Summary

Keep open. Current source supports the central uninvoked-agent gap: built-in memory managers only attach watchers and interval sync after a non-transient manager is created, while status-only managers can report clean state from in-process flags; the related merged work fixed the sessions source, not memory files for agents that never start a manager.

## What This Changes

The issue requests reliable indexing and honest status for `.md` files written into an uninvoked agent's `memory/` directory during a gateway session.

## Best Possible Solution

Define a bounded memory-core/gateway reconciliation policy for enabled agents that makes disk/index drift visible and indexes changed memory files without starting unbounded watchers for every possible agent.

## Reproduction Assessment

Yes, at source level: an uninvoked built-in memory agent has no default MemoryIndexManager, so no watcher or interval exists to observe new `memory/` files. The exact `Indexed: N/N` display detail still needs a focused live fixture because status scans disk totals.

## Solution Assessment

Unclear as proposed: a catch-up scan is the right direction, but the best owner boundary needs maintainer choice because eager startup watchers, periodic scans, and status-only drift checks have different resource and UX costs.
