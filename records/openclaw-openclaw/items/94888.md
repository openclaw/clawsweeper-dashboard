## Label Justifications

- `P2`: This is a source-supported memory indexing reliability bug with meaningful but bounded impact in multi-agent deployments.
- `impact:data-loss`: Files remain on disk but can be silently absent from memory search, which is practical loss of indexed user memory.
- `impact:session-state`: The report is about drift between an agent's on-disk memory state and the persisted searchable index for that agent.

## Summary

Keep open. Current main still has no startup or periodic reconciliation for builtin `memory/` files belonging to agents whose `MemoryIndexManager` has not been created, and the related merged catch-up work is limited to the `sessions` source.

## What This Changes

The issue requests reliable indexing and honest status reporting for files written into uninvoked agents' `memory/` directories during a gateway session.

## Best Possible Solution

Implement a bounded memory-core/gateway reconciliation policy that detects missing or changed `memory/` files for enabled agents without unbounded watcher startup, and make status surface disk/index drift.

## Reproduction Assessment

Yes, at source level: an uninvoked builtin-memory agent has no default `MemoryIndexManager`, so no watcher or interval exists to observe new `memory/` files. I did not run a live fixture for the exact status output.

## Solution Assessment

Unclear as proposed: catch-up scanning is the right direction, but maintainers should choose the reconciliation boundary because startup watchers, periodic scans, and status-only drift checks have different resource costs.
