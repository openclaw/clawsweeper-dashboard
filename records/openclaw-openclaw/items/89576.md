## Label Justifications

- `P2`: This is a normal-priority browser screenshot reliability fix with limited blast radius to direct-CDP screenshot capture.
- `merge-risk: 🚨 compatibility`: The patch can change existing multi-tab behavior by making screenshot capture activate the target tab instead of leaving native foreground state unchanged.

## Summary

Keep open for maintainer review: the patch is a narrow, well-evidenced fix for direct-CDP screenshots on backgrounded tabs, but it intentionally changes native browser foreground state and needs maintainer acceptance before merge.

## What This Changes

The PR adds a `Page.bringToFront` CDP call before `Page.captureScreenshot` and extends CDP screenshot tests/mocks for ordering and activation-error handling.

## Best Possible Solution

Land the activation-based fix if maintainers accept screenshot foregrounding; otherwise require a focused variant that restores the prior foreground tab and bounds activation failure handling.

## Reproduction Assessment

Yes, from source inspection and contributor live CDP proof, though I did not rerun a browser repro in this read-only review. Current main reaches `Page.captureScreenshot` without activation, and the PR body provides before/after timings for a backgrounded headless tab.

## Solution Assessment

Unclear until maintainers decide the foregrounding contract. `Page.bringToFront` is the narrowest CDP-level mitigation I found, but restoring the prior active tab is safer if screenshots must not leave the target foregrounded.
