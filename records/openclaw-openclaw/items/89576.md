## Summary

Keep this PR open; it is a focused browser screenshot bug fix with sufficient live CDP proof, but merging it intentionally changes multi-tab foreground state, so the remaining gate is maintainer acceptance of that compatibility tradeoff.

## What This Changes

The PR adds `Page.bringToFront` before direct CDP screenshot capture and adds CDP mock coverage for activation ordering and activation-error tolerance.

## Best Possible Solution

Land the activation-based fix if maintainers accept screenshot foregrounding; otherwise require a focused variant that restores the previously active tab when that can be done reliably.

## Reproduction Assessment

Yes. Current main reaches `Page.captureScreenshot` without activation, and the PR body provides live CDP before/after timings for a backgrounded headless tab, though I did not rerun the browser repro in this read-only review.

## Solution Assessment

Unclear until maintainers decide the focus contract. `Page.bringToFront` is the narrowest CDP-level mitigation I found, but restoring the prior active tab is safer if screenshots must not leave the target foregrounded.
