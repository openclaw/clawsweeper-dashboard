## Summary

Keep open for maintainer review: the patch is small and well-supported, but foregrounding the target tab changes multi-tab screenshot behavior and needs explicit browser contract acceptance or restore handling.

## What This Changes

The PR adds a `Page.bringToFront` call before CDP screenshot capture and updates CDP WebSocket mocks/tests for activation ordering and activation-failure tolerance.

## Best Possible Solution

Land the activation-based fix only if browser owners accept screenshot foregrounding; otherwise require previous-tab restoration with focused ordering and failure-tolerance coverage.

## Reproduction Assessment

Yes. Current main's direct CDP screenshot path reaches `Page.captureScreenshot` without activation, and the PR body provides live CDP before/after timing for a backgrounded headless tab; I did not run a local browser E2E in this read-only review.

## Solution Assessment

Unclear until the focus contract is decided. `Page.bringToFront` is a plausible minimal CDP fix, but maintainers should choose whether screenshots may leave the target tab foregrounded or must restore the previous active tab.
