## Summary

Keep open for maintainer review: current main and the latest release still lack this screenshot activation behavior, and the PR has credible live CDP proof, but merging it intentionally changes foreground-tab state and needs maintainer acceptance or a restore-previous-tab variant.

## What This Changes

The PR adds `Page.bringToFront` before direct CDP screenshot capture and updates CDP screenshot mocks/tests for activation ordering and activation-failure handling.

## Best Possible Solution

Land the activation-based fix if maintainers accept screenshot foregrounding; otherwise require a focused restore-previous-tab variant with the same ordering and failure-tolerance coverage.

## Reproduction Assessment

Yes. Current main reaches `Page.captureScreenshot` without activation, and the PR body includes live CDP before/after timing output for a backgrounded headless tab; I did not rerun a live browser repro in this read-only review.

## Solution Assessment

Unclear until maintainers decide the focus contract. `Page.bringToFront` is the narrowest CDP-level mitigation I found, but restoring the prior active tab is safer if screenshots must not leave the target foregrounded.
