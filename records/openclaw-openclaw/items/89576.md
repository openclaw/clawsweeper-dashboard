## Summary

Keep open for maintainer review: the bug and proposed CDP mechanism are well supported, but merging should be an explicit browser compatibility decision because screenshot capture would activate the target tab in multi-tab sessions.

## What This Changes

The PR inserts a CDP `Page.bringToFront` call before direct screenshot capture and updates browser CDP mocks/tests for ordering and failure tolerance.

## Best Possible Solution

Land the activation-based fix only after maintainers choose the browser screenshot focus contract; if foregrounding is not acceptable, add reliable restore-previous-tab behavior with focused coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: create two CDP tabs, activate the second, then time `Page.captureScreenshot` on the backgrounded first tab before and after `Page.bringToFront`; I verified the current-main path and the PR supplies live timing evidence.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the focus contract is decided; `Page.bringToFront` is the narrow CDP mechanism, but leaving the target tab foregrounded versus restoring the prior tab is a maintainer compatibility choice.
