## Summary

Keep open: current main still has a source-traceable SecretRef lifecycle gap for Google Meet realtime paths, and the related reports only fixed narrower status/Talk/OpenAI-keychain cases rather than this exec-backed Google Meet and Voice Call provider-config path.

## What This Changes

The issue reports Google Meet realtime command/tool failures when exec-backed 1Password SecretRefs reach plugin/provider registration unresolved.

## Best Possible Solution

Resolve Google Meet and Voice Call realtime provider SecretRefs from the active runtime snapshot before provider config normalization, add the missing Google Meet SecretRef contract, and cover the command/tool paths with focused regression tests.

## Reproduction Assessment

Yes, source-reproducible: current main forwards Google Meet realtime provider configs to provider normalization while Google Meet lacks a SecretRef contract, and OpenAI realtime normalization rejects unresolved SecretRef input. I did not execute the full live 1Password and Google Meet realtime environment.

## Solution Assessment

Yes, the maintainable direction is to resolve the targeted realtime provider SecretRefs through the existing manifest/command-secret/runtime snapshot seams. A broader plugin-loading order change should only happen if focused tests show the existing seams cannot cover the Google Meet command and tool paths.
