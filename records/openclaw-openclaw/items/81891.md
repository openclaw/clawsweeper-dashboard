## Summary

Keep open: current main still has a source-traced SecretRef gap for Google Meet realtime provider configs, and the related fixes only cover status preload or OpenAI keychain fallback rather than exec-backed Google Meet/voice-call realtime config resolution.

## What This Changes

Google Meet realtime join/create and the `google_meet.join` tool should resolve exec-backed SecretRefs before provider normalization so realtime sessions can attach.

## Best Possible Solution

Resolve Google Meet and Voice Call realtime provider SecretRefs from the active runtime snapshot before provider config normalization, add the missing Google Meet secret-input contract, and cover the command/tool paths with focused regression tests.

## Reproduction Assessment

Yes, source-reproducible: current main passes Google Meet realtime provider configs through to provider normalization while Google Meet lacks a SecretRef contract, and the provider normalizers throw on unresolved SecretRef input. I did not execute the full 1Password/Google Meet realtime environment.

## Solution Assessment

Yes, the right direction is targeted SecretRef resolution for the realtime provider config surfaces before local provider normalization. A broad plugin-loading order change should only be used if the existing command-secret and manifest-contract seams cannot cover the Google Meet and Voice Call paths cleanly.
