## Summary

Keep open: current main still has a source-traceable SecretRef lifecycle gap for Google Meet realtime join/create paths. The related fixes cover status preload and OpenAI keychain env fallback, but not exec-backed Google Meet or Voice Call realtime provider config flowing through plugin registration/runtime setup.

## What This Changes

The issue requests that Google Meet realtime join/create and the google_meet.join tool resolve exec-backed 1Password SecretRefs from the active gateway/runtime snapshot before provider config is read.

## Best Possible Solution

Resolve Google Meet and Voice Call realtime provider SecretRefs from the active runtime snapshot before provider config normalization, add any missing manifest/docs contract for Google Meet realtime provider keys, and cover the CLI plus gateway-tool paths with focused regression tests.

## Reproduction Assessment

Yes, source-reproducible. Current main forwards Google Meet realtime provider configs to provider normalization while Google Meet lacks a SecretRef contract, and strict provider normalization rejects unresolved SecretRef input; I did not run the full live 1Password/Google Meet environment.

## Solution Assessment

Unclear as an implementation detail. The narrowest maintainable direction is to resolve the targeted realtime provider SecretRefs through existing manifest/runtime snapshot seams; a broad plugin-loading order change needs maintainer judgment and focused tests before it is the preferred fix.
