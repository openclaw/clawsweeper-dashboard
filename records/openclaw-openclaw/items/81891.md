## Summary

Keep open: current main still has a source-traceable SecretRef lifecycle gap for Google Meet realtime join/create paths, and the related Voice Call and Talk fixes do not cover `plugins.entries.google-meet.config.realtime.providers.*.apiKey`. The prior ClawSweeper comment is still consistent with the newer main tree.

## What This Changes

The issue reports Google Meet realtime join/create and gateway-tool paths failing when realtime provider API keys are configured as `exec:onepassword` SecretRefs.

## Best Possible Solution

Resolve Google Meet realtime provider SecretRefs from the active runtime snapshot before provider config normalization, add the missing Google Meet SecretRef contract if that surface is supported, and cover CLI plus gateway join/create paths with focused regressions.

## Reproduction Assessment

Yes, source-reproducible: current main does not declare Google Meet realtime provider API keys as SecretRef inputs, then forwards the raw provider config into strict provider normalization. I did not execute the full live 1Password and Google Meet realtime flow.

## Solution Assessment

Unclear that the broad loader-order change suggested in the issue is the best first fix. The safer maintainable direction is a targeted SecretRef contract/runtime-snapshot fix for Google Meet realtime provider config, expanding loader order only if focused tests prove the existing seam cannot cover it.
