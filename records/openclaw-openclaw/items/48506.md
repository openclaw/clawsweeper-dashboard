## Summary

Keep open. Current main and `v2026.5.19` still expose `message_sending` with raw routing fields only, while target resolution and cross-context facts remain internal; this is a clear additive Plugin SDK/security contract that needs maintainer design rather than cleanup closure.

## What This Changes

The issue requests optional resolved target context on the plugin `message_sending` hook so policy plugins can reason about outbound message destinations without reimplementing internal target resolution.

## Best Possible Solution

Define a narrow additive Plugin SDK target-context contract populated from existing outbound resolver and policy paths, with tests and docs that lock the metadata disclosure boundary.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an additive Plugin SDK feature request. Source, docs, tests, and latest-release inspection show the requested target-context fields are absent while resolver and cross-context policy facts remain internal.

## Solution Assessment

Unclear. A nested optional `targetContext` is plausible, but maintainers need to decide stable names, privacy/redaction boundaries, same-account semantics, allowlist status, and coordination with the related security-hook proposals.
