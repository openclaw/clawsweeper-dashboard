## Summary

Keep open. Current main still exposes `message_sending` with raw routing fields only, while resolved target and cross-context data remain internal; adding this would be a new public Plugin SDK/security-policy contract that needs maintainer design.

## What This Changes

Add optional resolved target context fields to `PluginHookMessageSendingEvent` so policy plugins can distinguish internal, cross-context, and external outbound destinations.

## Best Possible Solution

Define a backward-compatible optional target-context contract, populate it from existing resolver and outbound-policy paths, and document/test the disclosure boundaries for plugin hooks.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main gives high confidence that the requested SDK gap remains: `message_sending` receives raw routing fields while resolver and cross-context policy data stay internal.

## Solution Assessment

Unclear. The nested optional `targetContext` direction is plausible, but maintainers should decide stable field names, privacy boundaries, and coordination with related provenance/security-hook work before implementation.
