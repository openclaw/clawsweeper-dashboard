## Summary

Keep open. Current main still exposes `message_sending` with raw routing fields while target resolution and cross-context policy remain internal, so this is a new public Plugin SDK/security-policy contract that needs maintainer design rather than cleanup closure.

## What This Changes

The issue requests optional resolved target context on the public `message_sending` plugin hook payload.

## Best Possible Solution

Define a backward-compatible optional target-context contract, populate it from existing resolver and outbound-policy paths, and document/test the plugin disclosure boundary.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence the requested SDK gap remains on current main: `message_sending` receives raw routing fields while resolved target and cross-context data stay internal.

## Solution Assessment

Unclear. The nested optional `targetContext` direction is plausible, but maintainers should decide stable field names, privacy boundaries, and coordination with the related provenance/security-hook proposals before implementation.
