## Summary

Keep open. Current main still exposes `message_sending` with raw routing fields only, while target resolution and cross-context policy data remain internal; this is a public plugin SDK and security-policy API decision that needs maintainer review.

## What This Changes

Request optional resolved target context on `PluginHookMessageSendingEvent` so `message_sending` hooks can distinguish external, cross-context, target type, channel, and account relationship details.

## Best Possible Solution

Define a backwards-compatible optional target-context SDK contract for `message_sending`, populate it from generic resolver and cross-context policy helpers, document disclosure boundaries, and add contract plus delivery tests.

## Reproduction Assessment

Yes for the API gap. Current-main source inspection shows hook consumers receive only raw routing fields while resolver and cross-context policy helpers compute the requested context internally.

## Solution Assessment

Unclear. The nested optional `targetContext` direction is plausible, but maintainers should decide final field names, stability guarantees, and privacy/security disclosure boundaries before implementation.
