## Summary

Keep open. Current main still exposes `message_sending` with raw destination/content fields only, while target resolution and cross-context policy are internal; this is a public plugin SDK and security-policy API decision that needs maintainer review rather than cleanup.

## What This Changes

Request to add optional resolved target context to `PluginHookMessageSendingEvent` so policy plugins can distinguish external, cross-context, target-type, channel, and account-relationship cases.

## Best Possible Solution

Define and document a backwards-compatible optional target-context surface on the public message hook contract, populated from the existing target resolver and cross-context policy helpers with explicit privacy boundaries and contract tests.

## Reproduction Assessment

Yes. The API gap is reproducible by inspecting current main: the public event type and delivery call sites lack resolved target context, while resolver and cross-context policy helpers compute related data internally.

## Solution Assessment

Unclear. The nested optional `targetContext` proposal is plausible, but maintainers should first settle field names, stability guarantees, and information-disclosure boundaries; the implementation should reuse existing resolver/policy output rather than making plugins reimplement it.
