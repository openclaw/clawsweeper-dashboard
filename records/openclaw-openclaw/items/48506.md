## Summary

Keep open. Current main and the latest release still expose `message_sending` with raw routing fields only, while the requested resolved target and cross-context facts remain internal; this is an additive public Plugin SDK/security-metadata contract that needs maintainer design.

## What This Changes

The issue requests optional resolved target context on `PluginHookMessageSendingEvent` so plugins can make DLP, allowlisting, audit, and cross-context policy decisions without reimplementing target resolution.

## Best Possible Solution

Define a narrow additive Plugin SDK target-context contract populated from the existing outbound resolver and policy paths, with tests and docs that lock privacy/redaction and disclosure boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an additive Plugin SDK feature request. Source, docs, current main, and the latest release show the requested target-context fields are absent while resolver and cross-context policy facts remain internal.

## Solution Assessment

Unclear. A nested optional `targetContext` is plausible, but maintainers need to decide stable names, privacy/redaction boundaries, same-account semantics, allowlist status, and coordination with the related security-hook proposals.
