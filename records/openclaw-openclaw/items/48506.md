## Summary

Keep open: current main and v2026.6.10 still expose only the raw outbound destination plus basic routing metadata to `message_sending`; resolved target context remains a security-sensitive public plugin API design decision.

## What This Changes

The issue requests optional resolved target context fields on the `message_sending` plugin hook so policy plugins can distinguish internal routing from external outbound communication.

## Best Possible Solution

Define a narrow optional `message_sending` target-context contract that reuses canonical outbound target resolution and cross-context policy facts, documents the disclosure boundary, and adds focused hook/delivery tests after maintainer approval.

## Reproduction Assessment

Not applicable as a runtime bug reproduction, but yes for source-level proof: current main and v2026.6.10 lack the requested `message_sending` target-context field while resolving target facts internally.

## Solution Assessment

Unclear until maintainer design review. Optional nested fields are plausible, but the exact target taxonomy and sensitive disclosure boundary should be approved before implementation because this is a public plugin SDK surface.
