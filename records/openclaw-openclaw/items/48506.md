## Summary

Keep open. The earlier keep-open review still matches current main: `message_sending` exposes raw routing fields only, while resolved target and cross-context facts remain internal; adding this would be a new public Plugin SDK and security-policy contract.

## What This Changes

Request adding optional resolved target context fields to the `message_sending` hook event for DLP, allowlisting, auditing, and cross-context policy decisions.

## Best Possible Solution

Define a backward-compatible, documented target-context contract, populate it from existing resolver and outbound-policy paths, and add tests that pin both hook payload shape and disclosure boundaries.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main gives high confidence that the SDK gap remains: `message_sending` receives raw routing fields while resolver and cross-context policy data stay internal.

## Solution Assessment

Unclear. The nested optional `targetContext` direction is plausible, but maintainers should decide stable field names, privacy boundaries, and coordination with related provenance/security-hook work before implementation.
