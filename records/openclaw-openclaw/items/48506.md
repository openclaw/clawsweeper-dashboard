## Summary

Keep open. Current main and the latest release still expose `message_sending` with raw routing fields only; target resolution and cross-context policy facts exist internally but are not surfaced through the public Plugin SDK hook, so this remains a security-sensitive API design item.

## What This Changes

The issue requests optional resolved target context on the `message_sending` hook so plugins can make outbound routing, DLP, allowlist, and audit decisions without reimplementing core target resolution.

## Best Possible Solution

Define a narrow optional Plugin SDK target-context contract populated from existing resolver and policy paths, with tests and docs that explicitly lock the routing-metadata disclosure boundary.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an additive Plugin SDK feature request. Source inspection of current main and `v2026.5.22` shows the requested target-context fields are absent while the underlying resolver and cross-context policy facts remain internal.

## Solution Assessment

Unclear. A nested optional `targetContext` is plausible, but maintainers need to decide stable names, privacy/redaction boundaries, same-account semantics, allowlist status, and coordination with the related security-hook proposals.
