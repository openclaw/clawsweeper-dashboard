## Summary

Keep open. Current main still exposes only raw outbound routing fields to `message_sending` hooks while target resolution and cross-context policy data remain internal, so this is still a security-sensitive Plugin SDK design request rather than cleanup closure.

## What This Changes

Request to add optional resolved target context to `PluginHookMessageSendingEvent` so plugins can distinguish external, cross-context, and target-type routing decisions.

## Best Possible Solution

Define a backward-compatible target-context contract in the Plugin SDK, populate it from existing resolver and outbound-policy paths, and document/test the disclosure boundary.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence the SDK gap remains on current main: `message_sending` receives raw routing fields while resolved target and cross-context data stay internal.

## Solution Assessment

Unclear. The nested optional `targetContext` direction is plausible, but maintainers should decide stable field names, privacy boundaries, and coordination with the related provenance/security-hook proposals before implementation.
