## Summary

Keep open. Current `main` still exposes `message_sending` with raw routing fields only, while resolved target and cross-context policy data remain internal; this is a public plugin SDK and security-policy API design decision, not conservative cleanup.

## What This Changes

Request to add optional resolved target context fields to the `message_sending` plugin hook event for DLP, allowlisting, compliance, and cross-context policy plugins.

## Best Possible Solution

Define a backwards-compatible optional target-context contract for `message_sending`, populate it from shared resolver/policy paths, and document the privacy/security boundaries with focused contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source, docs, and test inspection on `origin/main` confirm the requested API gap: `message_sending` receives raw routing fields while resolver and cross-context policy data remain internal.

## Solution Assessment

Unclear. The nested optional `targetContext` direction is plausible, but maintainers should decide the final SDK field names, stability guarantees, and disclosure semantics before implementation.
