## Summary

Keep open. Current main still exposes `message_sending` with raw routing fields only, while target resolution and cross-context policy data remain internal; this is a public plugin SDK and security-policy API decision that needs maintainer review.

## What This Changes

The issue requests optional resolved target-context fields on the `message_sending` hook so plugins can distinguish external or cross-context outbound messages.

## Best Possible Solution

Define a backwards-compatible optional target-context contract for `message_sending`, populate it from the generic resolver and cross-context policy helpers, and document explicit privacy/security boundaries with contract tests.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the API gap: public event types and delivery call sites lack resolved target context, while resolver and policy helpers compute related data internally.

## Solution Assessment

Unclear. The nested optional `targetContext` direction is plausible, but maintainers should decide the final field names, stability guarantees, and disclosure boundaries before implementation.
