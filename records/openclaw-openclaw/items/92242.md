## Label Justifications

- `P1`: The omission is shipped in v2026.6.6 and silently breaks Telegram reaction and reply workflows that depend on recognizing bot-sent messages.
- `impact:message-loss`: Missing ledger entries can cause legitimate delivered Telegram messages to be suppressed or misrouted by reaction and reply handling.
- `impact:session-state`: Persisted sent-message state diverges from actual successful Telegram deliveries.

## Summary

Current main and the latest release still emit Telegram source-delivery sent hooks without updating the sent-message ledger, so the reported regression remains necessary to fix. Two open PRs explicitly claim this issue; repository policy requires keeping it open until maintainers select and merge one canonical implementation.

## What This Changes

Restore sent-message ledger recording for successful Telegram source-delivery replies so reaction filtering, reply routing, and `wasSentByBot` observe delivered bot messages.

## Best Possible Solution

Land one canonical linked PR that records at the existing successful-delivery boundary, preserves hook semantics, tests both successful and no-delivery paths, and explicitly confirms the intended first-message versus per-send ledger granularity.

## Reproduction Assessment

Yes. Current source and v2026.6.6 show the successful source-delivery hook path without the established ledger write, and the linked PR proof exercises the real delivery and ledger modules with a stub Telegram API transport.

## Solution Assessment

Yes in direction: adding the source-level write beside the existing successful-delivery/hook gate is narrower and more maintainable than patching generated dist bundles or duplicating writes in every low-level branch. The remaining review question is whether one first message ID per reply is the correct ledger granularity.
