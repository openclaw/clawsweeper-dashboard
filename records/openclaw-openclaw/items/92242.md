## Label Justifications

- `P1`: This is a reported regression that can silently break Telegram reaction filtering and reply routing for real source-delivered replies.
- `impact:message-loss`: Missing sent-message metadata can make Telegram reaction filters and reply routing miss legitimate bot-sent messages.
- `impact:session-state`: The sent-message ledger is session-scoped state, and delivery-path replies leave that state stale.

## Summary

Keep open: current main still emits Telegram `message_sent` hooks from the source-delivery path without recording the sent message, and the linked fixes are open PRs rather than merged main behavior.

## What This Changes

The issue reports a regression where Telegram source-delivery replies emit `message_sent` hooks but do not append successful sends to the sent-message ledger.

## Best Possible Solution

Review and land one focused Telegram fix that records successful delivery-path sends in the same cfg-scoped ledger used by `wasSentByBot`, then let the merged PR close this issue.

## Reproduction Assessment

Yes, from source: current main's delivery path sends replies and emits `message_sent`, while no `recordSentMessage` call exists there and downstream `wasSentByBot` depends on the ledger. I did not run a live Telegram repro in this read-only pass.

## Solution Assessment

Yes, the best fix is a narrow owner-local change in `extensions/telegram/src/bot/delivery.replies.ts` that records the first successfully delivered message with `params.cfg` under the same success gate as the sent hook.
