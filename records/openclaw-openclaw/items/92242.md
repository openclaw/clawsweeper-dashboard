## Summary

Keep open: current main still emits Telegram `message_sent` from source delivery without recording those delivered ids in the sent-message ledger, and the issue has two open GitHub-closing PRs that should be reviewed before closing.

## What This Changes

The issue reports a Telegram regression where automatic source-delivery sends fire `message_sent` hooks but do not persist sent-message ledger entries.

## Best Possible Solution

Land one narrow Telegram fix that records every successful source-delivery message id in the existing sent-message ledger using the same cfg scope that `wasSentByBot` reads, then close this issue as implemented.

## Reproduction Assessment

Yes. Current main's delivery source emits `message_sent` after successful delivery but has no `recordSentMessage` import or call, while the canonical send path records and own-mode reaction handling reads that ledger.

## Solution Assessment

Yes. The narrow maintainable fix is to record delivered Telegram message ids at the shared source-delivery success point and pass the runtime cfg scope through that path; the open linked PRs are implementation candidates.
