## Label Justifications

- `P1`: The regression affects Telegram source delivery, own-mode reaction filtering, and reply routing for real users.
- `impact:message-loss`: Missing sent-message ledger entries can cause Telegram reaction filters or reply routing to miss bot-sent messages.
- `impact:session-state`: The bug silently omits session-scoped Telegram sent-message state that later `wasSentByBot` checks rely on.

## Summary

Keep open: current main and the latest release still emit Telegram `message_sent` hooks from automatic source delivery without recording the sent ids, and two open closing PRs are already attached as candidate fixes.

## What This Changes

Restore Telegram automatic source-delivery sent-message ledger writes alongside `message_sent` hook emissions.

## Best Possible Solution

Land one canonical Telegram fix that records every successful source-delivery send in the existing sent-message ledger with the correct cfg scope, then close this issue after merge.

## Reproduction Assessment

Yes, source inspection is high-confidence: `deliverReplies` sends messages and emits `message_sent`, while current main has no `recordSentMessage` call in that delivery path. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to record successful `deliverReplies` message ids through the existing sent-message ledger and pass cfg from automatic dispatch. The open PR at https://github.com/openclaw/openclaw/pull/92420 appears to track that shape, subject to normal PR review and proof.
