## Label Justifications

- `P1`: This is a reported regression in a real Telegram delivery workflow that affects downstream routing and filtering after successful sends.
- `impact:message-loss`: The sent Telegram message itself is delivered, but the missing ledger write can make downstream delivery-aware routing and reaction handling miss that message.
- `impact:session-state`: The sent-message ledger is session/channel state used by `wasSentByBot`, so failing to update it creates state drift after successful sends.

## Summary

Current main still has the reported Telegram delivery ledger gap, and the linked fixes are open PRs rather than merged implementation evidence, so this issue remains necessary.

## What This Changes

The issue requests restoring sent-message ledger writes for successful Telegram automatic source-delivery replies so `wasSentByBot`, reaction filtering, and reply-to routing see bot-sent messages.

## Best Possible Solution

Land one reviewed Telegram delivery fix that records successful source-delivery sends into the existing sent-message ledger at the same success boundary as the `message_sent` hook, with focused regression coverage for `wasSentByBot`.

## Reproduction Assessment

Yes. Source inspection on current main shows `deliverReplies` emits `message_sent` hooks without calling `recordSentMessage`, while the canonical send path still records messages into the ledger.

## Solution Assessment

Yes. The narrow maintainable fix is to record successful source-delivery sends through the existing sent-message cache contract and pass the existing `cfg`; this does not need a new config option or product decision.
