## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve shutdown after Hyprland cleanup:** `extensions/browser/src/browser/hyprland-capture.ts:84-93`
  - body: Once these `SIGINT` and `SIGTERM` listeners are registered, Node no longer applies the default termination behavior. The handlers only remove the virtual output and return, so a gateway that has used Hyprland capture can survive Ctrl-C, systemd stop, or `kill`; clean up and then preserve normal shutdown semantics.
  - confidence: 0.91
- **[P2] Remove the virtual output when setup fails:** `extensions/browser/src/browser/hyprland-capture.ts:106-114`
  - body: `createHeadlessOutput()` can succeed before `getActiveWorkspaceId()` or the workspace move throws, but cleanup is registered only after both later operations succeed. That failure path falls back to CDP while leaving the new Hyprland output behind; wrap post-create setup and remove `outputName` before rethrowing.
  - confidence: 0.92
- **[P2] Isolate captures on a dedicated workspace:** `extensions/browser/src/browser/hyprland-capture.ts:107-112`
  - body: The helper uses whatever active workspace the virtual output reports, then `grim` captures the entire output. Downstream Hyprland validation says this was not reliable and can include another client if the compositor places one there; create or reuse a named capture workspace and move only the target Chromium window there before capture.
  - confidence: 0.84
- **[P2] Restore focus after native captures:** `extensions/browser/src/browser/hyprland-capture.ts:216-220`
  - body: The Hyprland setup/capture path moves the browser into compositor state but never records or restores the previously focused monitor or workspace after `grim` returns or fails. The live downstream validation reports focus can remain on the hidden output, so restore focus in a `finally` path around setup and each capture.
  - confidence: 0.82
