## Summary

Keep open. Current main still has only the CDP screenshot path for normal headed browser screenshots, while this PR is the active implementation for the Hyprland/Wayland fallback tied to #64317. The PR is not mergeable as-is because the current head still has lifecycle cleanup bugs, public SDK baseline drift, no changelog entry, and a stale body that would close the unrelated Windows PATHEXT issue #54470.

## What This Changes

The PR adds opt-in `browser.hyprlandCapture` support that uses Hyprland `hyprctl` virtual outputs plus `grim` for headed viewport screenshots, updates browser config/schema/docs and a Plugin SDK resolver subpath, and adds browser route/helper tests.

## Best Possible Solution

Revise this PR in place: keep the Hyprland fallback opt-in, preserve normal process shutdown after cleanup, remove virtual outputs on every setup failure after creation, regenerate the Plugin SDK API baseline or avoid the new public subpath, add the required changelog entry, update the PR body to target #64317 rather than #54470, and then run the focused browser/config/SDK checks plus the changed gate before merge.

## Reproduction Assessment

Yes. #64317 gives a concrete Omarchy/Hyprland/Wayland headed Chromium screenshot timeout path, and the current PR defects are reproducible by source tracing the signal handlers, setup error path, and Plugin SDK API baseline contract.

## Solution Assessment

No. The feature direction is plausible and current main lacks the fallback, but this patch is not yet the best mergeable solution while it can swallow stop signals, leak virtual outputs on setup failure, and leave public SDK/changelog metadata incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve normal shutdown after cleanup:** `extensions/browser/src/browser/hyprland-capture.ts:84-93`
  - body: After Hyprland capture registers these `SIGINT`/`SIGTERM` listeners, Node no longer applies its default signal termination behavior. Because the handlers only run cleanup, a gateway process that has used this path can survive a normal `kill`, systemd stop, or Docker stop until a later force kill. Clean up the output and then preserve normal shutdown behavior.
  - confidence: 0.88
- **[P2] Remove the virtual output when setup fails:** `extensions/browser/src/browser/hyprland-capture.ts:106-112`
  - body: `createHeadlessOutput()` can succeed before `getActiveWorkspaceId()` or the workspace move throws, but cleanup is not registered until after those operations. In that case the route falls back to CDP while leaving the new Hyprland output behind. Wrap the post-create setup and remove `outputName` before rethrowing.
  - confidence: 0.93
- **[P2] Regenerate the Plugin SDK API baseline:** `scripts/lib/plugin-sdk-entrypoints.json:67`
  - body: Adding `resolve-system-bin` to the public SDK entrypoint list changes the generated Plugin SDK API surface, but the PR does not update the generated baseline hash. `pnpm plugin-sdk:api:check` will report drift until the baseline is regenerated and committed, or the new public subpath is removed.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/config/types.browser.ts:96-100`
  - body: This adds a user-facing browser fix and a new `browser.hyprlandCapture` config key, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing fixes to add an active-version changelog entry before merge.
  - confidence: 0.82
