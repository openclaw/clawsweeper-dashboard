## Summary

Keep this PR open: current main still uses the CDP screenshot path for ordinary headed browser screenshots, and the linked Hyprland bug remains plausible. The branch is not mergeable as-is because the new Hyprland lifecycle code can block normal shutdown, can leak a virtual output on setup failure, lacks the Plugin SDK API baseline update, and still has no after-fix real Hyprland behavior proof.

## What This Changes

The branch adds an opt-in `browser.hyprlandCapture` setting, a Hyprland `hyprctl`/`grim` viewport screenshot helper, route fallback logic, config/schema/docs/tests, and a new Plugin SDK `resolve-system-bin` subpath.

## Best Possible Solution

Land a cleaned-up opt-in browser-plugin fallback that tears down virtual outputs on every setup, capture, and signal path, keeps SDK/config artifacts aligned, links the Hyprland bug, and includes redacted live Hyprland proof.

## Reproduction Assessment

Yes for source-level reproduction: the linked Hyprland issue gives concrete Omarchy/Wayland steps and current main still routes normal viewport screenshots through CDP `Page.captureScreenshot`. I did not run a live Hyprland compositor session in this read-only review.

## Solution Assessment

No. The direction is plausible, but this branch is not the best mergeable form until lifecycle cleanup, SDK baseline alignment, PR metadata, merge drift, and real behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve shutdown after Hyprland cleanup:** `extensions/browser/src/browser/hyprland-capture.ts:84-93`
  - body: Once these `SIGINT` and `SIGTERM` listeners are registered, Node no longer applies its default signal termination behavior. The handlers only remove the virtual output and return, so a gateway that has used Hyprland capture can survive a normal `kill`, systemd stop, or Ctrl-C unless some other handler exits; clean up and then preserve normal shutdown semantics.
  - confidence: 0.91
- **[P2] Remove the virtual output when setup fails:** `extensions/browser/src/browser/hyprland-capture.ts:106-114`
  - body: `createHeadlessOutput()` can succeed before `getActiveWorkspaceId()` or the workspace move throws, but cleanup is registered only after both calls. In that failure path the route falls back to CDP while leaving the new Hyprland output behind; wrap post-create setup and remove `outputName` before rethrowing.
  - confidence: 0.92
- **[P2] Regenerate the Plugin SDK API baseline:** `scripts/lib/plugin-sdk-entrypoints.json:67`
  - body: Adding `resolve-system-bin` to the public SDK entrypoint list changes the generated Plugin SDK API surface, but the PR does not include the tracked `docs/.generated/plugin-sdk-api-baseline.sha256` update. `pnpm plugin-sdk:api:check` should report drift until the baseline is regenerated, or the new public subpath is removed.
  - confidence: 0.88
