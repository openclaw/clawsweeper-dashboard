## Summary

Keep open: current main still lacks the Hyprland/Wayland fallback, but this PR is not merge-ready because real Hyprland proof is still missing and the patch has lifecycle, cleanup, and capture-isolation defects.

## What This Changes

This PR adds an opt-in `browser.hyprlandCapture` path that moves headed Chromium to a Hyprland virtual output and captures viewport screenshots with `grim`, plus config/schema/docs, tests, and a Plugin SDK `resolve-system-bin` subpath.

## Best Possible Solution

Land a refreshed opt-in browser-plugin fallback that uses an isolated named capture workspace, restores focus, tears down outputs on every setup/capture/signal path, keeps SDK/config/docs aligned, and includes redacted live Hyprland proof.

## Reproduction Assessment

Yes for source-level reproduction: current main still routes ordinary viewport screenshots through CDP `Page.captureScreenshot`, and the linked Hyprland issue gives concrete Omarchy/Wayland steps. I did not run a live Hyprland compositor session in this read-only review.

## Solution Assessment

No. The fallback direction is plausible, but this PR is not the best mergeable form until lifecycle cleanup, setup-failure cleanup, workspace/focus isolation, PR metadata, and real behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve shutdown after Hyprland cleanup:** `extensions/browser/src/browser/hyprland-capture.ts:84-93`
  - body: Once these `SIGINT` and `SIGTERM` listeners are registered, Node no longer applies the default signal termination behavior. The handlers only remove the virtual output and return, so a gateway that has used Hyprland capture can survive Ctrl-C, systemd stop, or `kill`; clean up and then preserve normal shutdown semantics.
  - confidence: 0.91
- **[P2] Remove the virtual output when setup fails:** `extensions/browser/src/browser/hyprland-capture.ts:106-114`
  - body: `createHeadlessOutput()` can succeed before `getActiveWorkspaceId()` or the workspace move throws, but cleanup is registered only after both calls. That failure path falls back to CDP while leaving the new Hyprland output behind; wrap post-create setup and remove `outputName` before rethrowing.
  - confidence: 0.92
- **[P2] Isolate captures on a dedicated workspace:** `extensions/browser/src/browser/hyprland-capture.ts:107-112`
  - body: The helper uses whatever active workspace the virtual output reports, then `grim` captures the entire output. The downstream Hyprland validation says this was not reliable and can include another client if the compositor places one there; create/reuse a named capture workspace and move only the target Chromium window there before capture.
  - confidence: 0.84
- **[P2] Restore focus after native captures:** `extensions/browser/src/browser/hyprland-capture.ts:216-220`
  - body: The Hyprland setup/capture path moves the browser into compositor state but never records or restores the previously focused monitor/workspace after `grim` returns or fails. The live downstream validation reports focus can remain on the hidden output, so restore focus in a `finally` path around setup and each capture.
  - confidence: 0.82
