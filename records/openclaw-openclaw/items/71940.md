## Summary

Keep open. Current main does not implement the Hyprland/grim screenshot fallback, and the latest PR head is still useful but not mergeable because the new lifecycle code can swallow normal SIGINT/SIGTERM shutdown, can leave a virtual output behind when setup fails after creation, and still appears to miss required SDK/changelog metadata follow-through.

## What This Changes

The PR adds opt-in `browser.hyprlandCapture` support using Hyprland virtual outputs plus `grim` for headed viewport screenshots, wires it into the browser screenshot route, exposes a narrow Plugin SDK `resolve-system-bin` subpath, and adds config, docs, and tests.

## Best Possible Solution

Revise this PR in place: keep the opt-in Hyprland fallback direction, preserve normal signal shutdown after cleanup, remove virtual outputs on every post-creation setup failure, regenerate the Plugin SDK API baseline or avoid the new public subpath, add the required changelog entry, correct the PR body to target #64317 instead of #54470, and then run the focused browser/config/SDK checks plus the changed gate before merge.

## Reproduction Assessment

Yes. #64317 gives a concrete Omarchy/Hyprland/Wayland headed Chromium screenshot timeout path, and the remaining PR defects are reproducible by direct source tracing of signal handling, post-create setup failure, and the Plugin SDK API baseline contract.

## Solution Assessment

No. The proposed feature direction is plausible and current main lacks the fallback, but this patch is not yet the best mergeable solution while it can swallow stop signals, leak virtual outputs on setup failure, and leave public SDK/changelog metadata incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve signal termination after cleanup:** `extensions/browser/src/browser/hyprland-capture.ts:84-93`
  - body: After this registers `SIGINT`/`SIGTERM` listeners, Node no longer applies its default termination behavior for those signals. These handlers only call cleanup and return, so a process that has used Hyprland capture can ignore a normal `kill`, systemd stop, or Docker stop unless another listener exits. Clean up and then preserve the normal shutdown path.
  - confidence: 0.88
- **[P2] Remove the output when setup fails:** `extensions/browser/src/browser/hyprland-capture.ts:106-114`
  - body: `createHeadlessOutput()` can succeed before `getActiveWorkspaceId()` or the workspace move throws, but cleanup is not registered until after those operations. That means the route can fall back to CDP while leaving the new Hyprland output behind. Wrap the post-create setup and remove `outputName` before rethrowing.
  - confidence: 0.92
- **[P2] Regenerate the Plugin SDK API baseline:** `scripts/lib/plugin-sdk-entrypoints.json:67`
  - body: Adding `resolve-system-bin` to the public SDK entrypoint list changes the generated Plugin SDK API surface, but the tracked API baseline hash is not part of this PR's file list. `pnpm plugin-sdk:api:check` will report drift until the baseline hash is regenerated and committed, or the new public subpath is removed.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/config/types.browser.ts:96-100`
  - body: This adds a user-facing browser fix and a new `browser.hyprlandCapture` config key, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing fixes to add an active-version changelog entry before merge.
  - confidence: 0.82
