The PR adds opt-in `browser.hyprlandCapture` support with a Hyprland `hyprctl`/`grim` capture helper, routes headed Linux viewport screenshots through that fallback before CDP, updates browser config/schema/docs, exposes `openclaw/plugin-sdk/resolve-system-bin`, and adds focused browser tests.

Revise this PR or a narrow replacement so the Hyprland fallback remains opt-in and plugin-boundary compliant, preserves normal process shutdown, removes virtual outputs on all setup and capture failures, aligns SDK generated artifacts/docs if the new public subpath stays, adds the required changelog entry, and links the PR to #64317 instead of the unrelated Windows issue.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve shutdown after Hyprland cleanup:** `extensions/browser/src/browser/hyprland-capture.ts:84-93`
  - body: After this feature registers `SIGINT` and `SIGTERM` listeners, Node no longer applies its default signal termination behavior. The handlers only remove the virtual output, so a gateway that has used Hyprland capture can survive a normal `kill`, systemd stop, or Docker stop until force-killed. Clean up and then preserve normal shutdown behavior.
  - confidence: 0.9
- **[P2] Remove virtual output when setup fails:** `extensions/browser/src/browser/hyprland-capture.ts:106-112`
  - body: `createHeadlessOutput()` can succeed before `getActiveWorkspaceId()` or the workspace move throws, but cleanup is registered only afterward and `_state` is never set. The route then falls back to CDP while leaving the new Hyprland output behind. Wrap the post-create setup and remove `outputName` before rethrowing.
  - confidence: 0.92
