## Summary

Keep open. Current main still does not implement the requested per-agent description metadata: the config type and strict runtime schema omit it, generated config metadata has no field, and `agents_list` only returns id/name/configured/model/runtime data. The direct implementation PR is closed unmerged, so there is no shipped or pending fix to close this against.

## What This Changes

Add an optional `description` field to per-agent config and expose it in `agents_list` so orchestrator agents can discover target-agent specialties dynamically.

## Best Possible Solution

Add a bounded optional `agents.list[].description` string as first-class agent metadata, preserve strict unknown-key validation, expose the field only when configured in `agents_list`, and document it in the agent config/subagent discovery docs with focused tests for inclusion, omission, and unknown-key rejection.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. There is a high-confidence verification path: source inspection on current main shows the proposed config key is absent from the type/schema and cannot appear in `agents_list`.

## Solution Assessment

Yes, if maintainers accept the public API. The narrow maintainable path is to add explicit config metadata and pass it through the existing discovery tool rather than making orchestrators scrape workspace files such as `SOUL.md`.
