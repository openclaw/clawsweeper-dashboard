## Summary

Keep this PR open: current main still uses the narrower WhatsApp `message_tool_only` gate, the branch is a focused implementation of the reported typing leak, and the contributor supplied real-session after-fix logs. The only reason not to merge directly is maintainer choice: this narrow suppression trades away typing for rare proactive unmentioned visible replies, while https://github.com/openclaw/openclaw/pull/90477 is an open broader structural alternative that is not merged yet.

## What This Changes

The branch changes WhatsApp inbound dispatch so unmentioned non-command group runs suppress typing regardless of source reply delivery mode and updates the automatic group test expectation.

## Best Possible Solution

Resolve the WhatsApp typing boundary by landing the broader send-start implementation if maintainers accept its SDK hook, or explicitly accept this PR's narrower UX tradeoff.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the vulnerable path: current main leaves automatic unmentioned WhatsApp group runs unsuppressed, and renderable pre-silence text can start composing before the silent reply decision. ## Solution Assessment

Is this the best way to solve the issue? It is an acceptable narrow mitigation, but not the best structural boundary now that https://github.com/openclaw/openclaw/pull/90477 proposes deferring typing until visible send start. Maintainers should choose between the small WhatsApp-only tradeoff and the broader lifecycle seam.
