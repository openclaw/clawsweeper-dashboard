## Summary

Keep open: the WhatsApp typing leak is still present on current main and the latest release, while this small patch is a plausible mitigation that still needs maintainer acceptance of the deliberate typing-UX tradeoff and the open structural alternative.

## What This Changes

The branch changes WhatsApp inbound dispatch so unmentioned non-command group runs set `suppressTyping` regardless of source reply delivery mode and updates the automatic group test expectation.

## Best Possible Solution

Resolve the WhatsApp typing boundary by either landing this narrow mitigation with an explicit accepted UX tradeoff or superseding it after the broader send-start implementation is mergeable and approved.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the vulnerable path: current main leaves automatic unmentioned WhatsApp group runs unsuppressed, and renderable pre-silence text can start composing before the silent reply decision. Live before-state reproduction is only medium confidence because the PR reports the provider checkpoint no longer emits the same stream shape today.

## Solution Assessment

Is this the best way to solve the issue? It is an acceptable narrow mitigation, but not clearly the best structural boundary while https://github.com/openclaw/openclaw/pull/90477 proposes deferring typing until visible send start. Maintainers should choose between the small WhatsApp-only tradeoff and the broader lifecycle seam.
