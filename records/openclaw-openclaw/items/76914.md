## Summary

Keep open: the reported bug is real and the bridge is in the right area, but the exact PR head still has a typed-test blocker and the new async bridge can race with final reply cleanup.

## What This Changes

The PR adds a CLI assistant-event bridge from `runCliAgent` into reply-runner `onPartialReply`, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the reply-runner bridge, but make partial delivery typed, serialized, and drained before CLI finalization.

## Reproduction Assessment

Yes. Source inspection reproduces the gap: current main emits CLI assistant events, Telegram preview listens to `onPartialReply`, and the CLI reply-runner path does not connect them before final assembly.

## Solution Assessment

No, not as currently implemented. The bridge belongs in the reply runner, but it needs typed regression coverage and a delivery chain that drains before the final CLI reply path continues.
