## What This Changes

This PR adds a CLI assistant-event bridge into the reply runner's `onPartialReply` path, plus regression tests and a changelog entry.

## Best Possible Solution

Keep the reply-runner bridge, but serialize and drain bridged partial delivery before final CLI cleanup, preserve the silent/sanitization guards, and require real CLI-to-channel preview proof before merge.

## Reproduction Assessment

Yes from source inspection: current main emits CLI assistant events and Telegram previews consume `onPartialReply`, while the CLI reply-runner path does not connect those surfaces until final output. I did not establish a live Telegram reproduction.

## Solution Assessment

No, not as currently patched. The boundary is narrow and maintainable, but bridged partial deliveries need ordered/drained async handling and the regression assertion needs a typed mock/call shape before merge.
