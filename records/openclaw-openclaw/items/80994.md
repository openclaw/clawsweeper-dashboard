## Summary

Keep open: current main does not implement disconnect-triggered archival for the bundled session-memory hook. Existing code covers `/new` and `/reset`, and plugin `session_end` covers real session lifecycle/shutdown in some paths, but the requested client-disconnect event, grace period, and config are new product/API behavior.

## What This Changes

Requests a new `session:disconnect` hook event plus config so bundled session-memory can archive on TUI/WebChat disconnect, gateway shutdown, and other close/crash paths.

## Best Possible Solution

Define one lifecycle contract: use existing `session_end` for real session termination/rollover, land or supersede the daily/idle rollover fix, and add a client-disconnect hook only after maintainers decide that UI disconnect should trigger archival.

## Reproduction Assessment

Not applicable as a strict bug reproduction: current docs and code define session-memory as `/new` and `/reset` only. Source inspection does confirm the requested disconnect archival path is absent on current main.

## Solution Assessment

Unclear/no for the exact proposed shape. A raw `session:disconnect` internal event may not be the best contract until maintainers decide how UI client disconnects relate to active sessions and existing plugin `session_end` hooks.
