## Summary

Keep this open. Current main confirms the reported gap in the bundled session-memory surface, but the requested TUI-close/session-end behavior is a new lifecycle contract and overlaps existing open daily/idle reset work rather than being already implemented or safely closeable.

## What This Changes

The issue requests a new session-end hook event so memory archival can run when the TUI session permanently exits.

## Best Possible Solution

Land or reject one canonical daily/idle reset implementation, then separately define whether TUI exit should archive a resumable session or rely on existing plugin session_end/gateway shutdown lifecycle hooks.

## Reproduction Assessment

Yes for the current behavior by source inspection: bundled session-memory only accepts command:new and command:reset, while TUI exit stops the TUI/client path. A live TUI run was not needed to establish that no current session-memory event is wired for TUI close.

## Solution Assessment

Unclear. Adding a new session:end-style internal event may be useful, but current main already has plugin session_end hooks and documents TUI sessions as resumable, so maintainers should choose the lifecycle boundary before implementation.
