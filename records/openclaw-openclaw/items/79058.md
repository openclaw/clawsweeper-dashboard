## Summary

Keep open: current main still formats disallowed cron `payload.model` values by printing the rejected model after `allowlist:`, and existing tests assert that confusing text rather than a clearer diagnostic.

## What This Changes

The issue asks for cron preflight model-allowlist failures to show a clearer rejected-model diagnostic, ideally with the actual allowlist or closest allowed model.

## Best Possible Solution

Keep the fail-closed model validation, but change the cron preflight error to say the requested model is not in `agents.defaults.models` and include normalized allowed keys or a closest-match hint when cheaply available.

## Reproduction Assessment

Yes, source-reproducible: current main formats a disallowed cron `payload.model` by echoing the rejected normalized model ref after `allowlist:`. I did not run a live cron job because this review is read-only, but the code path and tests match the reported output.

## Solution Assessment

Yes, a narrow diagnostic fix in `src/cron/isolated-agent/model-selection.ts` is the best path. The fix should not change model allowlist enforcement, fallback policy, or cron execution semantics.
