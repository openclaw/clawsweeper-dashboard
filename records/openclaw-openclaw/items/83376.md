## Label Justifications

- `merge-risk: 🚨 compatibility`: The diff changes max-token request budgeting on existing OpenAI-compatible provider paths and may alter current configured/default behavior.

## Summary

Keep open: the PR is aimed at a real OpenAI-compatible max-token overflow, but it is still not merge-ready because the clamp estimates raw context messages rather than the serialized chat payload and real provider proof is missing.

## What This Changes

The PR adds per-call max_tokens/max_completion_tokens clamping in buildOpenAICompletionsParams based on estimated message and system-prompt tokens, plus a changelog entry.

## Best Possible Solution

Compute the output cap from the final serialized chat-completions payload, preserve smaller configured caps and existing no-cap semantics, define no-room behavior, and cover a StepFun-like large-input regression.

## Reproduction Assessment

Yes: source inspection shows current main forwards the resolved max-token cap without subtracting prompt tokens, and the linked report provides the StepFun rejection shape. I did not run a live StepFun request in this read-only review.

## Solution Assessment

No: dynamic per-call clamping is the right direction, but this implementation estimates the wrong message surface and lacks real behavior proof. The safer fix is a tested helper that walks the final serialized payload used on the wire.
