## Summary

Keep open: the linked bug is source-reproducible, but this PR’s fixed 75% reserve still exceeds the context window for the PR’s own high-input example and lacks after-fix real behavior proof.

## What This Changes

This PR caps embedded-runner `maxTokens` at 75% of the resolved context window and adds a changelog entry.

## Best Possible Solution

Implement a per-call OpenAI-compatible request clamp: compute the output cap as `min(configured cap, effectiveContextBudget - estimatedPromptTokens)` at request construction time, preserve smaller user caps, define the no-room behavior explicitly, and add regression coverage plus redacted StepFun proof.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main can resolve and forward a full model/request max-token cap without subtracting the active prompt tokens. I did not run a live StepFun request in this read-only review.

## Solution Assessment

No. A fixed 25% reserve is not the best fix because prompt size varies per call; the safer solution is dynamic remaining-budget calculation in the provider request path.
