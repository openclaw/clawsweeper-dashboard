## Summary

Keep open: the linked bug is credible, but this PR's static 25% reserve is not a safe fix because it still exceeds the context window whenever input uses more than the reserved share and does not use the prompt-token estimate already available in the run path.

## What This Changes

This PR caps embedded-runner `maxTokens` at 75% of a configured context window and adds a dated changelog entry.

## Best Possible Solution

Move the cap to the per-call request path so provider max tokens are bounded by the effective context window minus estimated prompt/input tokens, with regression coverage for high-input large-context models.

## Reproduction Assessment

Yes, there is a high-confidence source-level path: configure the reported large-context StepFun model, accumulate enough prompt tokens, and the current transport can send a completion cap that does not subtract input tokens. I did not run the live StepFun provider.

## Solution Assessment

No. A fixed 25% reserve is not the narrow maintainable fix because it fails for prompts above that reserved share; the safer fix is dynamic per-call budgeting from the already estimated prompt tokens.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Compute the output cap from actual prompt tokens:** `src/agents/pi-embedded-runner/model.ts:661`
  - body: This fixed 75% cap still sends too many output tokens once input exceeds the reserved 25%. With the PR body's own 262144-context/100k-input case, this sends 196608 completion tokens, so input plus output is already 296608 before provider overhead and the same class of provider rejection remains. Please derive the request cap from the estimated prompt/input tokens for the current call instead of using a static percentage.
