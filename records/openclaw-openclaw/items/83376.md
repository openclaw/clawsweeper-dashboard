## Summary

Keep open: the PR targets a real provider-runtime bug, but the fixed 25% reserve is still a static model-level cap and does not solve high-input requests reliably.

## What This Changes

Caps embedded-runner model `maxTokens` at 75% of the resolved `contextWindow` and adds a changelog entry for the max-token rejection bug.

## Best Possible Solution

Implement a per-call OpenAI-compatible request clamp: compute the output cap as `min(configured cap, effectiveContextBudget - estimatedPromptTokens)` at request construction time, preserve smaller user caps, define no-room behavior explicitly, and add regression coverage plus redacted StepFun proof.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main can forward a full model/request max-token cap without subtracting the active prompt tokens. I did not run a live StepFun request in this read-only review.

## Solution Assessment

No. A fixed 25% reserve is not the best fix because prompt size varies per call; the safer solution is dynamic remaining-budget calculation in the provider request path.
