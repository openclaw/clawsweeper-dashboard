## Summary

Keep open. Current main still stores the memory capability registry in module-local `memoryPluginState`, and Memory Wiki bridge/status still read public artifacts through that module-local path while bundled plugin runtime mirroring remains present. Prior #72615 handled CLI-to-Gateway routing, but this report targets the lower runtime-mirror singleton split, and the same author has paired open draft PR #75064 for that exact remaining bug.

## What This Changes

The issue reports that Memory Wiki bridge/status still return zero memory-core public artifacts because installed OpenClaw and `plugin-runtime-deps` runtime mirrors can load separate `memory-state` singletons.

## Best Possible Solution

Land or replace #75064 with a narrow shared or host-owned memory capability registry that all mirrored runtime module instances observe, while preserving the current plugin SDK API and adding regression coverage for Memory Wiki bridge import/status artifact counts across runtime mirrors.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps and instrumentation, a second user confirms the same behavior, and current main still has the module-instance-sensitive bridge/status read path behind the runtime mirror boundary.

## Solution Assessment

Unclear until #75064 is reviewed. The maintainable direction is a canonical shared or host-owned memory capability registry with mirrored-runtime regression tests, not patching generated `plugin-runtime-deps` files.
