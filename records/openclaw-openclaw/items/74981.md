## Summary

Keep open. Current main still stores memory capability registration in module-local state while Memory Wiki bridge/status read public artifacts through that same module-local path, and the bundled runtime mirror still remaps plugin execution into a separate staged root. The issue is also paired with open draft PR #75064, so it should stay open while that implementation is reviewed and validated.

## What This Changes

Memory Wiki bridge should share the memory-core publicArtifacts registry across installed OpenClaw dist and plugin-runtime-deps mirror module instances so bridge import and status do not report zero artifacts.

## Best Possible Solution

Land or replace #75064 with a narrow shared or host-owned memory capability registry that every canonical and mirrored runtime module instance observes, while preserving the existing plugin SDK API and adding regression coverage for duplicate module instances plus Memory Wiki bridge import/status artifact counts.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps and two independent confirmations, and current-main source inspection still shows the module-instance-sensitive state path across the runtime mirror boundary.

## Solution Assessment

Unclear until #75064 is reviewed. Sharing state through `createPluginRuntimeStore` is a plausible narrow fix, but a canonical host-owned memory capability service may be the safer long-term shape; generated plugin-runtime-deps files should not be patched directly.
