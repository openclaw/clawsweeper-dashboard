## Summary

Keep open. Current main still has the reported module-local memory capability state and Memory Wiki bridge/status paths still read public artifacts through that state, while an open draft PR, #75064, is already paired with this issue and targets the split-runtime fix.

## What This Changes

Fix Memory Wiki bridge/status so public artifacts registered by memory-core remain visible across canonical and plugin-runtime-deps mirrored runtime module instances.

## Best Possible Solution

Land or replace #75064 with a narrow shared or host-owned memory capability registry that every canonical and mirrored runtime module instance observes, preserving the existing plugin SDK API and covering duplicate-module Memory Wiki bridge import/status regressions.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps and instrumentation, two commenters confirm the symptom on separate installs, and current main still has the module-local state path across the runtime mirror boundary.

## Solution Assessment

Unclear until #75064 is reviewed. Moving memory state through the shared runtime-store seam is a narrow candidate fix, while a canonical host-owned capability service may be the more durable design; generated plugin-runtime-deps files should not be patched directly.
