## Summary

Keep open. Current main still stores memory capability state in a module-local singleton while Memory Wiki bridge/status read through that same module-local path, and packaged bundled-plugin runtime mirroring can still create the split module instance described in the report. The issue is also paired with open draft PR #75064 by the same author, so cleanup should wait for that PR to be reviewed, validated, landed, or replaced.

## What This Changes

Fix Memory Wiki bridge/status so the canonical Gateway runtime and mirrored plugin-runtime-deps code share the same active memory publicArtifacts registry instead of reporting zero artifacts.

## Best Possible Solution

Land or replace #75064 with a narrow shared or host-owned memory capability registry that all mirrored runtime module instances observe, while preserving the existing plugin SDK API and adding regression coverage for Memory Wiki bridge import/status artifact counts across duplicate runtime modules.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps, instrumentation, and two independent confirmations; current main still has a module-instance-sensitive memory capability registry read by bridge/status across the runtime mirror boundary.

## Solution Assessment

Unclear until #75064 is reviewed. The maintainable direction is a canonical shared registry or host-owned capability seam with duplicate-module regression tests, not patching generated plugin-runtime-deps files.
