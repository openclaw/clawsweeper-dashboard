## Summary

Keep open. Current main still keeps the memory capability registry in module-local state, Memory Wiki bridge/status still read public artifacts through that local state, and the bundled runtime mirror still remaps plugin execution into staged `plugin-runtime-deps` paths. The issue is also paired with open draft PR #75064, so the next cleanup step is to review or replace that implementation rather than close this issue or queue a second repair branch.

## What This Changes

Fix Memory Wiki bridge import/status so public artifacts registered by `memory-core` remain visible when bridge code runs through canonical and `plugin-runtime-deps` mirrored runtime modules.

## Best Possible Solution

Land or replace #75064 with a narrow shared or host-owned memory capability registry that every canonical and mirrored runtime module instance observes, while preserving the existing plugin SDK API and adding regression coverage for duplicate module instances plus Memory Wiki bridge import/status artifact counts.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps and two independent matching confirmations, and current-main source inspection still shows a module-instance-sensitive registry crossing the runtime mirror boundary.

## Solution Assessment

Unclear until #75064 is reviewed. Moving the state through `createPluginRuntimeStore` is a plausible narrow fix, but a canonical host-owned memory capability service may be the safer long-term shape; generated `plugin-runtime-deps` mirror files should not be patched directly.
