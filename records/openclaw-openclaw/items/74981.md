## Summary

Keep open. Current main still has the module-local memory capability registry on the Memory Wiki bridge/status path, the report has multiple packaged-install confirmations, and open draft PR #75064 is already paired with this issue for the same root cause.

## What This Changes

The issue asks OpenClaw to make Memory Wiki bridge/status see memory-core public artifacts even when bundled runtime mirrors would otherwise load a separate memory-state module instance.

## Best Possible Solution

Review and either land or replace #75064 with a narrow shared or host-owned memory capability registry that every in-process mirrored runtime module instance observes, while keeping the plugin SDK API backward-compatible and adding regression coverage for Memory Wiki bridge import/status artifact counts.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps and instrumentation, two commenters confirm the same symptom, and current main still has producer and consumer paths that can diverge across mirrored module instances.

## Solution Assessment

Unclear until #75064 is reviewed. The right direction is a canonical shared/host-owned memory capability registry with mirrored-runtime tests, not patching generated plugin-runtime-deps files.
