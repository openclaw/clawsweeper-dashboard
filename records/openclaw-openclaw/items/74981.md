## Summary

Keep open. Current main still has the split-state shape described in the report: Memory Core registers public artifacts into a module-local memory capability registry, while Memory Wiki bridge/status read artifacts back through that same module-local registry and bundled plugin execution can still be remapped into a runtime mirror. The issue is also already paired with open draft PR #75064, so cleanup should focus on reviewing or replacing that implementation rather than closing this issue or queueing a second repair branch.

## What This Changes

Memory Wiki bridge should see public artifacts registered by memory-core across canonical and plugin-runtime-deps runtime mirrors instead of returning zero artifacts.

## Best Possible Solution

Land or replace #75064 with a narrow shared or host-owned memory capability registry that every canonical and mirrored runtime module instance observes, while keeping the public plugin SDK source-compatible and adding regression coverage for duplicate module instances plus Memory Wiki bridge import/status artifact counts.

## Reproduction Assessment

Yes. The issue provides concrete packaged Gateway steps and two independent confirmations, and current-main source inspection still shows the module-local memory registry crossing a runtime mirror boundary.

## Solution Assessment

Unclear. Moving `memoryPluginState` through `createPluginRuntimeStore` is a plausible focused repair, but a canonical host-owned memory capability service may be the more durable boundary; generated `plugin-runtime-deps` mirror files should not be patched directly.
