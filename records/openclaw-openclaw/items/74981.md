## Summary

Keep open. Current main still keeps memory capability state in module-local storage while Memory Wiki bridge/status read through that state, and the issue is already paired with open draft PR #75064 for the same runtime-mirror singleton split.

## What This Changes

Memory Wiki bridge/status should see public memory artifacts registered by memory-core across packaged plugin-runtime-deps runtime mirrors instead of returning zero artifacts.

## Best Possible Solution

Land or replace #75064 with a narrow shared or host-owned memory capability registry that every canonical and mirrored runtime module instance observes, while preserving the existing plugin SDK API and adding regression coverage for duplicate module instances plus Memory Wiki bridge import/status artifact counts.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps, two independent confirmations, and current-main source inspection still shows the module-instance-sensitive state path that explains the zero-artifact result.

## Solution Assessment

Unclear. The requested direction is correct, but the best implementation should be reviewed in #75064: moving state into `createPluginRuntimeStore` is plausible, while a canonical host-owned memory capability service may be more durable across runtime boundaries.
