## Summary

Keep open. Current main still has a module-local memory capability registry while Memory Core registers `publicArtifacts` there and Memory Wiki bridge/status read back through that same module-local path; the bundled runtime mirror boundary implicated by the report also remains. The issue is active, has multiple independent confirmations, and is paired with open draft PR #75064, so it should stay open while that implementation is reviewed or replaced.

## What This Changes

Memory Wiki bridge/status should see public memory artifacts registered by Memory Core even when bundled plugins run through the managed `plugin-runtime-deps` runtime mirror.

## Best Possible Solution

The desired end state is a canonical shared or host-owned memory capability registry that all Gateway and mirrored bundled-runtime module instances observe, with regression coverage for Memory Wiki bridge import and status artifact counts across duplicate module instances while keeping the existing plugin SDK API source-compatible.

## Reproduction Assessment

Yes. The issue gives concrete packaged npm/global Gateway steps and two independent confirmations, and current main still shows the module-instance-sensitive producer/consumer path behind the runtime mirror boundary.

## Solution Assessment

Unclear until #75064 is reviewed. A shared runtime-store-backed registry may be a narrow fix, but the most durable solution is a host-owned memory capability seam rather than patching generated `plugin-runtime-deps` files.
