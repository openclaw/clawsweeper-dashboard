## Summary

Keep open: current main still lacks the requested channel-aware opt-in policy/config surface, while source and tests show non-CLI channel contexts can already enter the existing memoryFlush path. This remains a maintainer product/security decision rather than conservative cleanup.

## What This Changes

The issue requests an opt-in pre-compaction memoryFlush turn for channel-driven sessions, optionally gated by channel backend, so channel agents can curate durable memory before compaction.

## Best Possible Solution

Clarify the intended current channel behavior first, then decide whether channel memoryFlush policy belongs in core config, a typed plugin/channel seam, or docs-only guidance; any implementation should be default-safe and explicitly cover workspace write security.

## Reproduction Assessment

Yes, at source level: current main has no enableForChannels/channelBackends config, and the relevant gate is provider/runtime/workspace based rather than channel-policy based. I did not run a live channel compaction scenario in this read-only review.

## Solution Assessment

Unclear as proposed. The channel allowlist may be useful, but because current non-CLI channel-shaped runs can already flush, the best path is maintainer product/security review before adding public config or changing silent write behavior.
