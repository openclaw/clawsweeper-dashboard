## Summary

Keep open: current main still lacks the requested channel-aware opt-in/config surface, and the source/tests show the premise needs maintainer clarification because non-CLI channel contexts can already enter memoryFlush while CLI/runtime/workspace gates still skip it. ## What This Changes

Request to add an opt-in channel-aware pre-compaction memoryFlush policy and optional channel allowlist for channel-driven sessions.

## Best Possible Solution

Clarify the current non-CLI channel behavior first, then decide whether channel-aware memoryFlush policy belongs in core config, a typed plugin seam, or docs-only guidance; any implementation should include explicit workspace-write/security policy and focused tests.

## Reproduction Assessment

Yes, at source level: current main has no enableForChannels/channelBackends config, and the relevant gate is provider/runtime/workspace based rather than channel-policy based. I did not run a live channel compaction scenario because this cleanup review is read-only.

## Solution Assessment

Unclear as an implementation direction: the requested allowlist may be useful, but current tests show some non-CLI channel contexts can already flush. The best path is a maintainer-approved policy decision before adding public config or changing silent channel write behavior.
