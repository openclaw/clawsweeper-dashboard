## Summary

Keep open: current main still preserves the CLI-provider skip and lacks a channel-scoped memoryFlush opt-in, while the related discussion shows this is a new lifecycle/config decision rather than a narrow bug fix.

## What This Changes

Request an opt-in pre-compaction memoryFlush for channel-driven sessions so channel agents can curate durable memory before compaction.

## Best Possible Solution

Keep this as the focused design thread and implement only after maintainers define a channel-scoped opt-in contract with schema/help/docs, CLI resume-state safeguards, and focused runner/config tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main has the described CLI-provider gates, strict config lacks the channel opt-in fields, and tests preserve the skip.

## Solution Assessment

Unclear for the final implementation. Reusing the existing memory-flush primitive is plausible, but the safest path needs maintainer design for channel eligibility, CLI session-state safety, and silent side-effect policy before code changes.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would add a silent, tool-capable agent turn to channel-triggered sessions.

Concerns:

- **[medium] Bound silent channel-side tool execution:** `src/auto-reply/reply/agent-runner-memory.ts:963`
  - body: A channel-enabled memory flush would call the embedded agent with tool access while suppressing replies, so any implementation should preserve sandbox writability checks and add clear timeout, cancellation, approval, and operator-visible failure handling.
  - confidence: 0.84
- **[medium] Preserve CLI resume-state safety:** `src/auto-reply/reply/agent-runner-memory.ts:510`
  - body: The current CLI skip exists because CLI backends can carry provider-owned session bindings; a channel opt-in must prove it will not write memory from stale external context or leave stale CLI bindings after compaction.
  - confidence: 0.8
