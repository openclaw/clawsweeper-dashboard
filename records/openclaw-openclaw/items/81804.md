## Summary

Keep open: current main still intentionally skips memory flush and preflight compaction for CLI providers, including channel-triggered runs that use a CLI backend, and the requested channel opt-in would add new public config and lifecycle policy.

## What This Changes

The issue requests an opt-in channel-side pre-compaction memory-flush path for channel-driven sessions using CLI backends, with optional backend allowlisting.

## Best Possible Solution

Keep this as the focused design thread, then implement only after maintainers define a channel-scoped opt-in contract with schema/help/docs, explicit CLI resume-state safeguards, and focused runner/config tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main has the described CLI-provider gates and lacks the proposed channel opt-in config.

## Solution Assessment

Unclear for the final implementation. Reusing the existing memory-flush primitive is plausible, but the safest solution needs maintainer design for channel eligibility, CLI session-state safety, and silent side-effect policy before code changes.
