## What This Changes

The issue requests an opt-in pre-compaction memory flush for channel-driven sessions, with channel enablement and optional backend allowlisting.

## Best Possible Solution

Keep this as the focused design thread and implement only after maintainers define a channel-scoped opt-in contract with schema/help/docs, CLI resume-state safeguards, and focused runner/config tests.

Not applicable as a bug reproduction; this is a feature/config request. Source inspection gives high confidence that current main has the described CLI-provider gates and lacks the proposed channel opt-in config.

## Solution Assessment

Unclear as the final implementation. Reusing the existing memory-flush primitive is plausible, but the safest solution needs maintainer design for channel eligibility, CLI session-state safety, and silent side-effect policy before code changes.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would add a silent, tool-capable agent turn to channel-triggered CLI sessions.

  - body: The current CLI skip exists because CLI backends can carry provider-owned session bindings; a channel opt-in must prove it will not write memory from stale external context or leave stale CLI bindings after compaction.
