## Summary

Keep open: current main has the memory-flush primitive and normal channel replies flow through the shared reply runner, but CLI-backed channel turns still hit the provider-level CLI skip and there is no opt-in channel config surface. This is a focused feature/product decision, not conservative cleanup.

## What This Changes

The issue requests an opt-in pre-compaction memory flush for channel-driven sessions, optionally gated by channel backend.

## Best Possible Solution

Define a channel/runtime-aware memory-flush contract: keep the existing non-CLI embedded channel behavior, and only add a CLI-backed channel opt-in if the schema, docs, tests, provider-state handling, sandbox checks, and audit semantics are agreed together.

## Reproduction Assessment

Yes, source-level. Current main has a focused test showing a channel-shaped context with a CLI provider skips the embedded memory-flush run, and the strict config schema has no enableForChannels or channelBackends key; I did not run tests because this review is read-only.

## Solution Assessment

Unclear. Reusing the existing memory-flush primitive is plausible, but a broad channel allowlist is not obviously the best fix because the real boundary is channel transport versus CLI model-provider state.
