## Summary

Keep open: current main still has embedded Pi memory flush plus CLI-provider skips, and the requested channel-scoped opt-in would add new public config and silent tool-capable channel lifecycle semantics that need maintainer product and security design.

## What This Changes

Request an opt-in channel-side pre-compaction memoryFlush mode, with possible enableForChannels/channelBackends config, for channel-driven sessions such as Google Chat, Slack, Discord, and Matrix.

## Best Possible Solution

Keep this as a focused design thread until maintainers define a channel-scoped opt-in contract with schema/help/docs, channel eligibility, CLI resume-state safeguards, and runner/config regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main lacks the requested channel opt-in config and still preserves the CLI-provider skip.

## Solution Assessment

Unclear for implementation. Reusing memoryFlush is plausible, but channel eligibility, public config, silent side-effect policy, and the CLI-provider boundary need maintainer design before code lands.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would add silent, tool-capable maintenance turns to channel-triggered sessions.

Concerns:

- **[medium] Bound silent channel-side tool execution:** `src/auto-reply/reply/agent-runner-memory.ts:1021`
  - body: A channel-enabled memory flush would call the embedded agent while suppressing replies, so timeout, cancellation, sandbox writability, approvals, logging, and operator-visible failure behavior need explicit design.
  - confidence: 0.86
- **[medium] Bind memory writes to the correct runtime policy:** `src/auto-reply/reply/agent-runner-memory.ts:1025`
  - body: Channel sessions can carry per-channel runtime policy and workspace state; the flush turn must use the correct sandboxSessionKey and avoid cross-channel or cross-tenant memory writes.
  - confidence: 0.8
- **[medium] Preserve CLI resume-state safety:** `src/agents/command/session-store.ts:282`
  - body: The current CLI lifecycle clears provider session bindings after CLI compaction; any channel opt-in must not write memory from stale provider-owned context or leave stale resume bindings behind.
  - confidence: 0.78
