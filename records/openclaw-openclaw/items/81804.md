## Summary

Keep open: current main still gates preflight compaction and memoryFlush on CLI-provider detection, and the public config/schema has no opt-in channel allowlist or enableForChannels equivalent. The related CLI compaction work solved a different provider-resume problem, so this channel-scoped memory-flush request still needs maintainer product and security review.

## What This Changes

Requests an opt-in channel-side pre-compaction memoryFlush mode, optionally allowlisted by channel backend, so channel-driven sessions can run a silent curated-memory turn before compaction.

## Best Possible Solution

Define a narrow public contract for channel-scoped memoryFlush opt-in, including the config shape, supported channel identity rules, CLI-provider interaction, and safety/audit guarantees, then implement it with focused tests and docs.

## Reproduction Assessment

Yes for the current limitation, via source inspection: current main still skips CLI-provider memoryFlush/preflight paths and lacks any channel opt-in config. I did not run a live gateway/channel scenario because this read-only review only needed source-level proof.

## Solution Assessment

Unclear. Reusing the existing memoryFlush primitive is plausible, but the best maintainable solution needs maintainer agreement on channel allowlisting, CLI-provider session-state boundaries, and silent tool execution policy before a PR should proceed.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would add or broaden a silent, tool-capable agent turn during channel session compaction.

Concerns:

- **[medium] Bound silent channel memory flush execution:** `src/auto-reply/reply/agent-runner-memory.ts:963`
  - body: A channel-side memoryFlush would run an agent turn with write/tool access while suppressing user-visible output, so implementation should require timeout/cancellation behavior, sandbox/workspace-access checks, approval handling, and durable operator-visible failure/audit records.
  - confidence: 0.86
- **[medium] Define provider-state boundaries before bypassing CLI gates:** `src/auto-reply/reply/agent-runner-memory.ts:745`
  - body: Current gates are based on the model provider, not the chat channel. Allowing channel-driven sessions that use CLI providers without a clear resume-state policy could reintroduce the stale external-session problem that the merged CLI compaction lifecycle avoided.
