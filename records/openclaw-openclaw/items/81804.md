## Summary

Keep open: current main still has no channel opt-in config and the relevant preflight/memory-flush gates remain tied to CLI-provider detection, so the requested behavior is a new security-sensitive lifecycle/config design rather than something already implemented or safe to auto-close.

## What This Changes

Add an opt-in pre-compaction memoryFlush turn for channel-driven sessions, with optional backend allowlisting, while preserving CLI resume-state safety.

## Best Possible Solution

Keep this as a focused design thread and implement only after maintainers define a channel-scoped opt-in contract with schema/help/docs, CLI resume-state safeguards, and focused runner/config tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks enableForChannels/channelBackends and still skips preflight compaction and memoryFlush for CLI providers even when the session context is channel-originated.

## Solution Assessment

Unclear as a final implementation. Reusing the existing memoryFlush primitive is plausible, but channel eligibility, config shape, CLI state safety, and silent side-effect policy need maintainer design before code should land.

Status: needs_attention
