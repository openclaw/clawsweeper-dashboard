## Best Possible Solution

Clarify the current non-CLI channel behavior first, then make a maintainer decision on whether channel-aware memoryFlush policy belongs in core config, docs, or a typed plugin seam.

## Reproduction Assessment

Yes from source inspection for the missing proposed config: the schema has no channel option and the helper gates on CLI/runtime/workspace state. No live failing channel scenario was reproduced, and current tests show non-CLI channel contexts can already flush.

## Solution Assessment

No as an automated fix: the proposed allowlist could duplicate existing non-CLI channel behavior while adding public config and security surface. The maintainable path is docs clarification or a maintainer-approved channel-aware plugin/core seam if the current behavior is insufficient.
