## Security Review

Status: needs_attention

Summary: The request would add silent, tool-capable agent turns to channel-triggered sessions, so sandbox, timeout, cancellation, and failure visibility need explicit design.

Concerns:

- **[medium] Bound silent channel-side tool execution:** `src/auto-reply/reply/agent-runner-memory.ts:963`
  - body: A channel-enabled memory flush would call the embedded agent with tool access while suppressing replies, so an implementation must preserve sandbox writability checks and define timeout, cancellation, approval, logging, and operator-visible failure behavior.
  - confidence: 0.84
- **[medium] Preserve CLI resume-state safety:** `src/auto-reply/reply/agent-runner-memory.ts:510`
  - body: The current CLI skip exists alongside separate CLI compaction/reseed logic that clears external CLI session bindings; a channel opt-in must prove it cannot write memory from stale provider-owned context or leave stale resume bindings after compaction.
  - confidence: 0.8

  - body: Channel-driven sessions can carry per-channel session keys and workspace policy; a flush turn must use the runtime policy session key and avoid cross-channel or cross-tenant memory writes.
  - confidence: 0.76
