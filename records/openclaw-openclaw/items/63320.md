## Summary

Keep this PR open, but not mergeable as-is: current main still exposes the legacy `SsrFBlockedError` contract, while the PR is a refactor-only public SDK rename that drops old public exports, preserves only a value alias, and changes the error name in a way that can miss current name-based handling after rebase.

## What This Changes

The PR renames the SSRF blocked error symbol from `SsrFBlockedError` to `SsrfBlockedError` across the SSRF guard, core/browser/Tlon call sites, tests, and SDK barrels, with a deprecated runtime alias for the old spelling.

## Best Possible Solution

If maintainers want the new acronym casing, make it additive on current main: export both names from core and every current public SDK subpath, preserve the old symbol in both value and type space, keep the legacy `name` during deprecation or update all name-based handlers to accept both spellings, and lock that compatibility with SDK/export and SSRF/browser tests. If the rename is not worth public API churn, leave the legacy spelling in place until a planned breaking SDK window.

## Reproduction Assessment

Yes. A static contract check is enough: a plugin importing `SsrFBlockedError` as a type or from public SDK subpaths loses compatibility under the PR's current dropped exports/value-only alias, and current main has a browser handler that still classifies the legacy error name.

## Solution Assessment

No. The proposed direct rename is not the safest fix for a public SDK symbol; the maintainable path is either an additive alias across current public subpaths with compatibility tests, or deferring the cleanup to a deliberate breaking-change release.
