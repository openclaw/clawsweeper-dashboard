## Summary

Keep open: this is a refactor-only public SDK/runtime rename with unresolved compatibility blockers, and current main still intentionally exposes and classifies the legacy `SsrFBlockedError` spelling.

## What This Changes

The PR renames the exported SSRF blocked error to `SsrfBlockedError` across core, browser, Tlon, tests, and SDK barrels, with a deprecated value alias for the old spelling.

## Best Possible Solution

If maintainers want the casing cleanup, make it additive on current main: retain legacy value and type exports on every public SDK subpath, preserve or dual-handle the legacy error name, and add SDK/export plus browser SSRF regression coverage.

## Reproduction Assessment

Yes. A source-level contract check is enough: current main publicly exports and classifies `SsrFBlockedError`, while the PR changes the class/name and drops legacy exports from SDK barrels except for a value-only alias.

## Solution Assessment

No. A direct public SDK rename is not the narrowest maintainable solution unless maintainers first approve the compatibility churn and the PR preserves both spellings across value, type, and error-name behavior.
