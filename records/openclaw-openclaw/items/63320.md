## Summary

Keep open. The cleanup is understandable, but the PR is not safe to merge as-is because it still breaks public Plugin SDK compatibility, only preserves the deprecated spelling as a value alias, and is stale against current main's moved SDK surfaces.

## What This Changes

The PR renames the SSRF blocked error symbol from `SsrFBlockedError` to `SsrfBlockedError` across core, browser/Tlon code, tests, and Plugin SDK barrels, with a deprecated runtime alias for the old spelling.

## Best Possible Solution

If maintainers want the new acronym casing, make the change additive: export both names from core and every current public SDK subpath, preserve the old name in both value and type space, update current name-based error handlers to accept both spellings or keep the legacy `name`, and add API/export tests that lock compatibility. Otherwise, leave the legacy public spelling in place until a deliberate breaking SDK window.

## Reproduction Assessment

Yes. A static contract check is enough: a plugin importing `SsrFBlockedError` from a public SDK subpath or using it as a TypeScript type would fail under the PR's current public-barrel and const-alias approach; no local build was run because this review was required to stay read-only.

## Solution Assessment

No. The current PR is not the best way to solve the rename because it removes old public exports and uses a value-only alias; the safer implementation is an additive alias across the current `ssrf-runtime` and `security-runtime` SDK surfaces plus compatibility tests.
