## Summary

Keep open: current main still exposes and handles the legacy `SsrFBlockedError` spelling, while this PR is a refactor-only public SDK rename that still needs compatibility work and a rebase decision before it is safe to merge.

## What This Changes

The PR renames the SSRF blocked error from `SsrFBlockedError` to `SsrfBlockedError` across core, browser, Tlon, tests, and SDK barrels, with a deprecated runtime alias for the old spelling.

## Best Possible Solution

If maintainers want the casing cleanup, make it additive on current main: export both spellings from core and every current public SDK subpath, preserve old value and type usage, keep or dual-handle the legacy error name, and add SDK/export plus SSRF/browser regression coverage.

## Reproduction Assessment

Yes. A source-level contract check is enough: current main publicly exports and handles `SsrFBlockedError`, while the PR drops that spelling from SDK barrels, changes the runtime error name, and adds only a value alias.

## Solution Assessment

No. A direct public SDK rename is not the narrowest maintainable solution; either keep the legacy spelling until a planned breaking window, or add the new spelling with full backwards-compatible aliases and tests.
