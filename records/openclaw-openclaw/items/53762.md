## Summary

Keep open: the Telegram message-loss bug is real and current main still lacks the segmented-final buffered-answer flush, but this PR has a concrete correctness blocker because the new flush path would skip the internal `message:sent` hook when it finalizes an existing preview.

## What This Changes

This PR adds a Telegram dispatcher flush for buffered final-answer text before the segmented final-delivery early return, plus a focused regression test and changelog entry.

## Best Possible Solution

Land this PR or an equivalent Telegram-owned patch after `flushBufferedFinalAnswer()` captures or returns the `LaneDeliveryResult` and routes preview-finalized buffered answers through the same hook path as normal final segment delivery.

## Reproduction Assessment

Yes. The PR body gives a concrete two-turn Telegram `streaming: "partial"` path, and current source shows the answer can be buffered while reasoning remains hinted, then left pending when the segmented final branch returns after skipped reasoning.

## Solution Assessment

No, not as currently proposed. Adding the missing flush is the right narrow fix for the dropped final answer, but the implementation must preserve the existing preview-finalized `message:sent` hook contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit the hook when flushing buffered final answers:** `extensions/telegram/src/bot-message-dispatch.ts:703`
  - body: The new segmented-final early return calls `flushBufferedFinalAnswer()` and exits, but that helper discards the `deliverLaneText()` result. When the buffered answer finalizes an existing preview, Telegram shows the answer but `emitPreviewFinalizedHook(...)` is never run, so `sessionKeyForInternalHooks` flows miss the internal `message:sent` hook unlike normal final segment delivery. Capture the result and route preview-finalized deliveries through the hook before resetting state.
  - confidence: 0.88
