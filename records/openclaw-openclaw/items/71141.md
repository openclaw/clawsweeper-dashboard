## Summary

Keep open: the PR targets a valid Telegram reaction-directive gap, but the proposed patch misses the real final-reply normalization path and also hard-codes Telegram-owned channel data in shared core normalization.

## What This Changes

The PR parses Telegram reaction directives, carries reaction intent through outbound payload data, executes Telegram reactions in bot/direct outbound delivery, and adds normalization plus adapter tests.

## Best Possible Solution

Use a generic reaction intent in the reply payload contract, preserve it through `normalizeReplyPayloadDirectives` and final payload filtering, then map it to Telegram-owned reaction delivery in the Telegram plugin with focused tests and a changelog entry.

## Reproduction Assessment

Yes. A focused source-level reproduction is to feed `[[react_to_current:🔥]]` through `buildReplyPayloads`/`normalizeReplyPayloadDirectives`; the current patch would strip the directive before the new outbound-plan channelData merge can see it.

## Solution Assessment

No. The Telegram transport piece is directionally useful, but the implementation should preserve reaction intent in the earlier final-reply normalizer and avoid hard-coding `channelData.telegram` in shared core code.
