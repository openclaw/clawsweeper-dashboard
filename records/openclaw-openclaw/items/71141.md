## Summary

Keep open: current main still lacks the requested Telegram reaction-directive behavior, and this draft PR is the active implementation candidate but has blocking parser propagation, target-resolution, typing, and changelog issues before it can merge.

## What This Changes

The PR parses outbound `[[react:...]]` and `[[react_to_current:...]]` directives, carries Telegram reaction metadata through outbound payloads, sends native Telegram reactions, allows reaction-only Telegram replies, and adds normalization plus adapter tests.

## Best Possible Solution

Ship a focused repair that keeps shared normalization generic, carries reaction intent through every parser consumer that can strip directives, resolves `react_to_current` to the triggering Telegram message, invokes the existing native Telegram reaction primitive only when a valid target exists, preserves current markdown image extraction behavior, and adds parser, payload, adapter, and delivery tests plus the required changelog entry.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: `[[react_to_current:...]]` is neither parsed nor converted to `channelData.telegram.reaction`, and a reaction-only Telegram reply still reaches the no-text/no-media rejection path.

## Solution Assessment

No. The approach is directionally right, but this patch is not yet the best maintainable fix because it strips reaction directives before all consumers preserve them, does not preserve the current-message target, forwards unsupported scope fields through reaction calls, can report reaction-only success without a target, and lacks the required changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve reaction metadata through parser consumers:** `src/auto-reply/reply/reply-directives.ts:29-35`
  - body: `parseReplyDirectives` is used by paths such as `normalizeReplyPayloadDirectives` and the embedded runner that only copy text/media/reply/audio fields. Stripping `[[react...]]` here means those paths lose the reaction before outbound payload planning can add `channelData.telegram.reaction`; carry the structured reaction metadata through each consumer or avoid stripping it until a preserving path runs.
  - confidence: 0.9
- **[P1] Preserve the current-message reaction target:** `src/auto-reply/reply/reply-directives.ts:18-35`
  - body: `[[react_to_current:...]]` is consumed by the new regex exactly like `[[react:...]]`, but the parser does not set `replyToCurrent` or resolve `replyToId` from the current message. In paths without an implicit reply target, the Telegram reaction becomes a no-op instead of targeting the triggering message.
  - confidence: 0.88
- **[P1] Type reaction scope forwarding before using it:** `extensions/telegram/src/bot/delivery.replies.ts:759`
  - body: The new reaction path forwards `gatewayClientScopes`, but `deliverReplies` does not declare that parameter and `reactMessageTelegram`'s reaction options type does not include it. TypeScript should reject the branch until the reaction option contract and caller plumbing are updated, or the extra forwarding is removed.
  - confidence: 0.9
- **[P2] Do not report success when no reaction target exists:** `extensions/telegram/src/outbound-adapter.ts:127-132`
  - body: For a reaction-only payload without `replyToMessageId`, this branch skips the reaction call and still returns a synthetic success result. That can make `sendPayload` report delivery even though no Telegram API call happened; reaction-only success should require a valid target or surface a non-delivery result.
  - confidence: 0.82
- **[P2] Add the required changelog entry:** `src/infra/outbound/payloads.ts:181`
  - body: This is a user-facing Telegram fix, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing fix/feat/perf changes to add an active-version changelog entry before merge.
  - confidence: 0.86
