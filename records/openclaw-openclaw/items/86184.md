## Summary

Keep open: the reporter's follow-up logs make this a concrete current-session-state/message-delivery failure, and the central behavior does not appear fixed on current main. The failure path is a provider timeout/retry or abort-side session write during the embedded prompt-lock release window, which still needs maintainer policy about which self-writes should be allowed versus treated as a session takeover.

## What This Changes

The issue reports that a Telegram direct turn can complete tool work but lose the assistant's final reply and show the generic `/new` fallback after a Codex provider timeout/retry triggers `EmbeddedAttemptSessionTakeoverError`.

## Best Possible Solution

Define a narrow owned-write contract for provider timeout/retry transcript updates during released prompt locks, add regression coverage around that path, and improve the before-reply fallback so Telegram can surface the real timeout/session-fence cause.

## Reproduction Assessment

Yes, source-level reproduction is credible from the provided logs: a Codex `/responses` guarded-fetch timeout is followed by retry/abort writes and then `EmbeddedAttemptSessionTakeoverError`. I did not establish a live Telegram replay on current main, so this is not a high-confidence runtime reproduction.

## Solution Assessment

Unclear: the fix likely belongs in the embedded prompt-lock/session-fence policy, but maintainers need to decide exactly which gateway-authored timeout/retry writes are safe to treat as owned. A broad safe-harbor would be risky; a narrow typed write path with tests is the safer direction.
