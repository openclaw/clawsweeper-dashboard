## Summary

Keep open: the added logs make this a concrete Telegram/Codex timeout plus embedded-session fence failure, and current main still routes unclassified before-reply failures to the generic fallback while the session-lock policy needs owner judgment before changing what counts as an owned timeout/retry write.

## What This Changes

The issue reports that a Telegram direct turn on OpenClaw 2026.5.22 can complete tool work but lose the final assistant payload after a Codex provider timeout/retry path mutates the session during an embedded prompt-lock release.

## Best Possible Solution

Decide the owned-write contract for provider timeout/retry transcript mutations, then add a focused regression that proves a Codex Responses timeout/retry window cannot abort an otherwise completed Telegram turn into the generic fallback.

## Reproduction Assessment

No, not as a deterministic current-main repro. The reporter supplied concrete gateway logs and the source path matches the timeout, retry, fence, and generic fallback sequence, but the exact live Telegram/Codex timing window was not reproduced here.

## Solution Assessment

Unclear for implementation: the narrowest fix likely belongs in the embedded prompt-lock owned-write policy or provider retry persistence path, but maintainers should choose how much self-write tolerance is safe before a repair PR is queued.
