## Summary

Keep open. The report is still paired with open closing PR #75427, and current source inspection supports the Signal quiet-SSE diagnosis while leaving the reported `channels status` timeout unverified in a live Signal setup.

## What This Changes

The issue reports unreliable inbound Signal DM handling and `openclaw channels status` timeouts despite successful Signal registration, outbound sends, and a healthy gateway probe.

## Best Possible Solution

Review, land, or explicitly supersede PR #75427, then live-verify inbound Signal DMs and `openclaw channels status --probe`; split a narrower status-timeout follow-up only if that symptom persists.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path on Ubuntu with signal-cli 0.14.3, successful outbound sends, repeated 10s SSE timeout logs, inbound DM stalls, and `channels status` gateway timeouts that map to inspected source paths.

## Solution Assessment

Mostly yes for the SSE failure: disabling the long-lived receive-stream deadline is the narrowest maintainable fix for quiet signal-cli streams. It is unclear whether that also resolves the `channels status` timeout without live confirmation.
