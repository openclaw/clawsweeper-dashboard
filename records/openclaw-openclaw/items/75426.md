## What This Changes

The issue reports that Signal outbound sends and daemon health work, while inbound Signal SSE receive can repeatedly time out and `openclaw channels status` can report a gateway timeout.

## Best Possible Solution

Review and land or explicitly supersede PR #75427, then verify Signal inbound DMs and `openclaw channels status --probe` on the affected Signal setup. If the status timeout survives the SSE fix, track that as a separate gateway/status responsiveness bug with fresh logs.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path on v2026.4.29 with signal-cli 0.14.3: outbound sends succeed, inbound Signal DMs stop replying, the SSE log repeats 10s timeouts, and `channels status` times out while gateway deep status remains healthy.

## Solution Assessment

Mostly yes for the SSE failure: disabling the long-lived Signal receive stream deadline is the narrowest source-level fix for the repeated 10s reconnect loop. It is still unclear whether that fully resolves the separate `channels status` timeout without post-merge live verification.
