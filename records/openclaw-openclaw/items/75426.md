## Summary

Keep open. Current main contains the narrow Signal quiet-SSE fix and the changelog ties the same root cause to #74741, but this issue is still paired with open closing PR #75427 and the reported `channels status` timeout has not been live-confirmed as resolved.

## What This Changes

Signal inbound DMs should keep the receive SSE listener alive and keep channel status responsive when outbound Signal and daemon probes are healthy.

## Best Possible Solution

Review, land, or explicitly supersede PR #75427 after comparing it to the current #74741 quiet-SSE implementation, then live-verify inbound Signal DMs and `openclaw channels status --probe`.

## Reproduction Assessment

Yes. The issue provides a high-confidence live reproduction path with successful outbound sends, repeated 10s Signal SSE timeout logs, unreliable inbound DMs, and matching `channels status` gateway timeouts; a delayed-header SSE test also exercises the core timeout behavior.

## Solution Assessment

Mostly yes for the SSE failure: disabling the short header deadline only for the long-lived Signal receive stream is the narrow Signal-local fix while preserving finite RPC/probe/send deadlines. It is unclear whether this fully solves the status timeout until the linked PR/current-main behavior is live-verified.
