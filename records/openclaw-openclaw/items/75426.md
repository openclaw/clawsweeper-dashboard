## Summary

Keep open. The report is concrete and paired with an open PR using closing syntax, so this issue should stay open until that PR is merged or explicitly superseded. Current main contains the intended quiet-SSE deadline handling and tests, but the report also includes a `channels status` timeout that needs live confirmation after the Signal receive fix.

## What This Changes

The issue asks for Signal inbound DMs and `openclaw channels status` to remain reliable when outbound Signal sends and the channel probe are healthy.

## Best Possible Solution

Review and land or explicitly supersede PR #75427, then verify live Signal inbound DMs and `openclaw channels status --probe` on the affected setup. If the status timeout persists after the SSE receive fix, split that into a narrower gateway/status responsiveness follow-up with fresh logs.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path: on Ubuntu with signal-cli 0.14.3, outbound sends succeed, inbound Signal DMs stop reliably replying, logs show repeated 10s SSE timeouts, and `channels status` times out while deep gateway status remains healthy.

## Solution Assessment

Mostly yes for the SSE failure: disabling the long-lived receive stream deadline is the narrowest maintainable fix for quiet signal-cli SSE streams. It is unclear whether that fully resolves the `channels status` timeout without live verification after the linked PR is resolved.
