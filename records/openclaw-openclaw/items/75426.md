Keep open. Current main contains the narrow Signal quiet-SSE change, but this issue is still tied to open closing PR #75427 and the reported `channels status` timeout has not been live-confirmed as resolved.

## What This Changes

The issue asks for Signal inbound receive/status handling to stop wedging so inbound DMs reliably trigger agent replies while outbound sends and daemon probes are healthy.

## Best Possible Solution

Keep the quiet-SSE implementation, resolve PR #75427 as merged or superseded, then live-verify inbound Signal DMs and `openclaw channels status --probe`; split a status-only follow-up only if the timeout remains.

## Reproduction Assessment

Yes. The issue provides a high-confidence live reproduction path on Ubuntu with signal-cli 0.14.3: outbound sends work, inbound DMs stall, logs show repeated 10s Signal SSE timeouts, and `channels status` hits the Gateway timeout path.

## Solution Assessment

Unclear. Disabling the long-lived receive-stream deadline is the narrow maintainable fix for quiet signal-cli SSE streams, but the combined `channels status` timeout should be verified live before this issue is closed.
