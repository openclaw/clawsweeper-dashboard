## Summary

Keep open. The report is concrete and the currently open, unmerged PR #75427 uses closing syntax for this issue, so the issue should remain open until that PR is landed or explicitly superseded. Current main contains the quiet-SSE deadline behavior and regression coverage, but the reported `channels status` timeout still needs live confirmation after the Signal receive path is resolved.

## What This Changes

The issue reports that Signal inbound SSE receive handling can wedge while outbound Signal sends and channel probes work, causing inbound DMs not to reliably trigger replies and `openclaw channels status` to time out.

## Best Possible Solution

Review, land, or explicitly supersede PR #75427, then verify on a Signal 0.14.3 setup that inbound DMs produce replies and `openclaw channels status --probe` returns promptly. If only the status timeout remains after the SSE fix, split that into a narrower gateway/channel status responsiveness follow-up.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path on OpenClaw 2026.4.29 with signal-cli 0.14.3: outbound sends succeed, inbound DMs stop reliably replying, logs show repeated 10s Signal SSE timeouts, and `channels status` times out while deep gateway status reports healthy.

## Solution Assessment

Unclear. Disabling the receive-stream deadline is the narrowest fix for the repeated Signal SSE timeout, but the issue should remain open until the linked PR is resolved and the separate `channels status` timeout is checked live.
