## Summary

Keep open. Current main has the narrow quiet-SSE fix, but the report is still tied to open closing PR #75427 and the `channels status` timeout side of the report has not been live-confirmed as fixed.

## What This Changes

Signal inbound DMs and channel status should remain responsive when `signal-cli` keeps the receive SSE stream idle instead of reconnecting every 10 seconds.

## Best Possible Solution

Resolve PR #75427 by landing it or explicitly superseding it, then live-verify inbound Signal DMs and `openclaw channels status --probe`; split a status-only follow-up if that timeout persists.

## Reproduction Assessment

Yes. The issue provides a high-confidence live reproduction path with successful outbound sends, repeated Signal SSE 10s timeout logs, unreliable inbound DMs, and a matching `channels status` gateway timeout on Ubuntu with `signal-cli` 0.14.3.

## Solution Assessment

Mostly yes for the likely SSE root cause: disabling the long-lived receive-stream deadline is the narrowest maintainable fix. It is unclear whether that alone solves the `channels status` timeout without live post-fix confirmation.
