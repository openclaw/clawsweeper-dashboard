## Summary

Keep open. Current main contains the quiet Signal SSE timeout fix, but the issue is still paired with open closing PR #75427 and the reported `channels status` gateway timeout still needs live confirmation after that fix is resolved.

## What This Changes

The issue reports that Signal outbound delivery works while inbound Signal DMs become unreliable, the SSE receive loop repeatedly times out, and `openclaw channels status` can time out against an otherwise healthy gateway.

## Best Possible Solution

Resolve PR #75427, then verify live Signal inbound DMs and `openclaw channels status --probe`; split a narrower gateway/status issue only if the timeout persists after the SSE fix.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path with OpenClaw 2026.4.29, signal-cli 0.14.3, working outbound sends, repeated 10s Signal SSE timeout logs, and `channels status` timing out while deep gateway status is healthy.

## Solution Assessment

Unclear. Disabling the long-lived Signal SSE deadline is the narrow maintainable fix for the reconnect loop, but it is not yet proven to cover the reported `channels status` timeout without a live post-fix check.
