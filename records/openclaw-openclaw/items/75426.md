## Summary

Keep open. The report is concrete and is paired with open, unmerged PR #75427 using closing syntax, so this issue should not be closed before that PR is landed or explicitly superseded; the remaining `channels status` timeout also needs live verification after the Signal receive fix.

## What This Changes

The issue reports unreliable Signal inbound DM handling with repeated SSE receive timeouts and `openclaw channels status` gateway timeouts despite working outbound sends and a healthy gateway probe.

## Best Possible Solution

Review, land, or explicitly supersede PR #75427, then verify on a Signal 0.14.3 setup that inbound DMs produce replies and `openclaw channels status --probe` returns promptly. If only the status timeout remains after the SSE fix, split that into a narrower gateway/channel status responsiveness follow-up.

## Reproduction Assessment

Yes. The issue gives a high-confidence live reproduction path on OpenClaw 2026.4.29 with signal-cli 0.14.3: outbound sends succeed, inbound DMs stop reliably replying, logs show repeated 10s Signal SSE timeouts, and `channels status` times out while deep gateway status reports healthy.

## Solution Assessment

Unclear. Disabling the receive-stream deadline is the narrowest maintainable fix for the repeated Signal SSE timeout, but the issue should remain open until the linked PR is resolved and the separate `channels status` timeout is checked live.
