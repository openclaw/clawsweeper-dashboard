## Summary

Current main and shipped releases already fix the reported Signal idle-SSE receive failure: the production Signal monitor now disables the short SSE header deadline for the long-lived receive stream, with regression coverage and reporter proof from the linked fix PR showing the same behavior restored in a real Signal setup.

## What This Changes

The issue reports unreliable inbound Signal DMs, repeated Signal SSE 10s timeout/reconnect logs, and `channels status` timeouts despite working outbound Signal sends.

## Best Possible Solution

Keep the shipped monitor-specific idle-SSE fix and handle the still-open related PR separately if maintainers want to close or rebase that broader branch.

## Reproduction Assessment

Yes for the affected releases: an idle `signal-cli` `/api/v1/events` stream without response headers exercised the old 10s Signal SSE deadline and produced the reported timeout/reconnect loop. Current main no longer reproduces that failure because the production monitor passes `timeoutMs: 0`.

## Solution Assessment

Yes, the shipped solution is the best narrow fix for the reported issue: it disables the deadline only for the long-lived Signal receive monitor while preserving bounded RPC/check/send behavior.
