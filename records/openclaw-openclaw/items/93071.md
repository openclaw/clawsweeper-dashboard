## Summary

Keep open: current main and v2026.6.8 still attach only `service.name` to the diagnostics-otel resource, and the protected `maintainer` label plus the stable `service.instance.id` choice require maintainer judgment before implementation.

## What This Changes

Request to add bounded OpenClaw-controlled OpenTelemetry resource identity, including version and stable instance identity, to diagnostics-otel exports.

## Best Possible Solution

Add a narrow plugin service-context identity seam, populate `service.version` and an approved stable `service.instance.id`, leave deployment environment to standard OTel mechanisms unless maintainers choose otherwise, and update diagnostics-otel tests and docs around final exported resource attributes.

## Reproduction Assessment

Yes for current behavior: source shows `resourceFromAttributes` includes only `service.name` and that resource feeds `NodeSDK` and `LoggerProvider`; no collector run is needed to prove version and instance identity are absent.

## Solution Assessment

Unclear. Adding `service.version` is straightforward through a runtime-version seam, but the best `service.instance.id` source needs maintainer approval because it becomes collector-visible stable identity.
