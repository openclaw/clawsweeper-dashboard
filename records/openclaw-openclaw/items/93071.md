Keep open: current main and the latest release still set only `service.name` for `diagnostics-otel` resources, and the protected `maintainer` label plus the stable `service.instance.id` source require maintainer judgment before implementation.

## What This Changes

Request to add explicit OpenClaw-owned OpenTelemetry resource identity attributes for `diagnostics-otel` exports, including runtime version and stable instance identity with bounded non-secret values and docs/tests.

## Best Possible Solution

Add bounded resource identity through the plugin service context, use the canonical runtime version for `service.version`, use a maintainer-approved opaque stable gateway/install identity for `service.instance.id`, leave deployment environment to standard OTel mechanisms unless intentionally added, and document/test the final exported resource attributes.

Yes. Source inspection on current main shows `diagnostics-otel` creates the OpenTelemetry resource with only `service.name`, and the docs/config surface only documents `diagnostics.otel.serviceName`.

## Solution Assessment

Unclear. Adding `service.version` through a plugin service context seam is likely maintainable, but the best fix depends on maintainer approval for the stable non-secret `service.instance.id` source and the preloaded-SDK contract.
