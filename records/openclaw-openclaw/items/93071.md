## Summary

Keep open: current main and the latest release still build the diagnostics-otel resource with only `service.name`, and no merged canonical PR owns the remaining work. Vera Perel's validation matches current source; the protected `maintainer` label and collector-visible stable instance identity choice make this a maintainer product/API decision, not a cleanup close.

## What This Changes

Request to have diagnostics-otel attach explicit OpenClaw-controlled OpenTelemetry resource identity such as `service.version` and a stable `service.instance.id` for exported signals.

## Best Possible Solution

Approve a bounded runtime resource identity contract, preferably through plugin service context, then have diagnostics-otel attach `service.version` and an approved stable `service.instance.id` with final-resource tests and docs for normal versus preloaded SDK modes.

## Reproduction Assessment

Yes. Source inspection on current main shows the exporter resource is constructed with only `service.name`, and docs/config expose only `serviceName`; a live collector is not needed to prove the requested attributes are absent, though a fix should inspect final exported resource attributes.

## Solution Assessment

Unclear. The requested direction is sound, but the best implementation depends on maintainer approval for the stable collector-visible instance identity source and should avoid new config unless the existing OTel/env mechanisms are insufficient.
