## What This Changes

The issue reports that selecting `xai/grok-4.20-multi-agent-0309` in a Telegram forum topic causes every assistant turn to fail with `400 status code (no body)` rather than replying or surfacing a clear provider/model validation error.

## Best Possible Solution

Add xAI-owned validation so unsupported multi-agent model ids are rejected before session override or runner dispatch, with a clear chat-facing error and regression tests; first-class multi-agent support should be a separate provider feature if maintainers choose to map that upstream API.

## Reproduction Assessment

No high-confidence live repro was run. Source inspection gives a medium-confidence reproduction path for the validation gap: explicit model selection can accept arbitrary provider/model refs when no allowlist is configured, and configured-provider fallback can still synthesize unknown models from a provider `baseUrl`.

## Solution Assessment

Unclear for full model support, but a narrow validation fix is the best immediate repair. It should not add multi-agent support unless maintainers first decide how that upstream capability maps onto OpenClaw's xAI provider contract.
