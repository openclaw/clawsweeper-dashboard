## Summary

Keep open. Current main still has the core synchronous startup path described in the report: `chat.history` reaches `loadSessionStore`, and that loader still reads/parses the store and runs enforced stale-prune/count-cap maintenance inline. The related WebUI timeout PR is useful mitigation, but it does not remove the session-store maintenance work from the gateway startup-critical path.

## What This Changes

The issue asks for gateway startup and `chat.history` loading to avoid being blocked for minutes by load-time session-store maintenance after the default pruning policy change.

## Best Possible Solution

Keep the session-store age/count safeguards, but move expensive prune/cap cleanup out of the gateway startup-critical path or make it explicitly budgeted/yielding after readiness. Manual `openclaw sessions cleanup --enforce` should retain immediate cleanup semantics, with regression coverage proving oversized stores do not block startup-facing `chat.history` readiness.

## Reproduction Assessment

Unclear for the full issue. There is a high-confidence focused path using oversized `sessions.json` fixtures to exercise the synchronous maintenance behavior, but this review does not reproduce the full Windows 20-minute hang or stdin-unblock symptom.

## Solution Assessment

No. Current main’s startup-unavailable gate and WebUI timeout work are mitigations; the narrow maintainable fix is a session-store/gateway change that defers or budgets load-time maintenance while preserving cleanup protections.
