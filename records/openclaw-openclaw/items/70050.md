## Summary

Keep open. Current main still performs session-store file read, JSON parse, migration, normalization, stale prune, and entry-count cap synchronously inside `loadSessionStore`; `chat.history` still reaches that loader through `loadSessionEntry`. The startup-unavailable gate and the related WebUI timeout PR are useful mitigations, but they do not remove the reported load-time session maintenance work from the startup-critical path.

## What This Changes

The issue reports that session history and gateway readiness can be delayed for many minutes after the session-maintenance default changed to enforce pruning/capping at load time.

## Best Possible Solution

Keep the session-store age/count safeguards, but move expensive prune/cap work out of the gateway startup-critical path or make it explicitly budgeted/yielding after readiness. Manual `openclaw sessions cleanup --enforce` should keep immediate cleanup semantics, and regression coverage should prove oversized stores do not block startup-facing `chat.history` readiness.

## Reproduction Assessment

Partial. We have a high-confidence path to exercise the synchronous `loadSessionStore` maintenance behavior with oversized `sessions.json` fixtures, but this review does not reproduce the full Windows 20-minute hang or stdin-unblock symptom.

## Solution Assessment

No. Current main's startup request gate and WebUI timeout work are mitigations, but the narrow maintainable fix is a gateway/session-store change that defers or budgets load-time maintenance while preserving the existing cleanup protections.
