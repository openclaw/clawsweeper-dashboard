## Summary

Keep open. Current main still performs session-store read, JSON parse, stale prune, and entry-count cap synchronously inside `loadSessionStore`, with maintenance defaulting to `enforce`; tests still assert load-time prune/cap behavior. The newer gateway startup-unavailable handling and the related #44897 WebUI timeout work are useful client resilience, but they do not remove the session-store maintenance work from the startup-critical synchronous path reported here.

## What This Changes

The issue asks for gateway startup and `chat.history` loading to avoid being blocked for minutes by load-time session-store maintenance after the session pruning policy change.

## Best Possible Solution

Keep the session-store size and age safeguards, but move expensive prune/cap cleanup out of the gateway startup critical path or make it budgeted/yielding after readiness. Keep manual `openclaw sessions cleanup` behavior and add regression coverage proving oversized stores do not block startup-facing `chat.history` readiness.

## Reproduction Assessment

Partial. There is a high-confidence way to exercise the synchronous load-time maintenance path with oversized `sessions.json` fixtures, but this review does not have a high-confidence reproduction of the full 20-minute Windows hang or the stdin-unblock symptom.

## Solution Assessment

No. Current main improves startup request handling, but the narrow maintainable fix is to defer or budget load-time session maintenance while preserving the existing cleanup and OOM safeguards.
