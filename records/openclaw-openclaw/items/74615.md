## Summary

Keep open. Current main still implements static per-message mention gating for group/channel messages, with no attention-window config, shared timer state, or channel integration that would process unmentioned follow-ups after an initial mention; the related items are adjacent mention-routing or mention-detection work and do not supersede this cross-channel feature request.

## What This Changes

The issue requests a configurable group-chat attention window that temporarily processes unmentioned follow-up messages after the agent is mentioned.

## Best Possible Solution

Define an additive shared attention-window contract with explicit opt-in defaults, per-conversation expiry semantics, platform limitation handling, tests, and docs, then wire channel plugins through that contract while preserving existing mention-gating and authorization boundaries.

## Reproduction Assessment

Yes. Configure a group or channel with `requireMention` enabled, send one mentioned message, then send an unmentioned follow-up; current main evaluates only the follow-up's own mention facts and takes the skip/store path in the inspected channel handlers.

## Solution Assessment

Unclear. The requested behavior is valid, but the proposed single `attentionWindow` field needs maintainer design because the maintainable fix likely spans shared policy, config schema, timer state, privacy semantics, SDK contracts, and platform-specific delivery limits.

## Risks / Open Questions

- A cross-channel attention window changes privacy and token-exposure semantics by processing unmentioned group messages after one mention, so defaults, opt-in scope, expiry, and audit behavior need explicit design.
- Some transports cannot deliver unmentioned group messages unless platform settings permit it; Telegram Bot API privacy mode is an existing documented constraint.
- Timer state needs a clear scope across account, channel, thread/topic, session, and agent routing, plus a decision about persistence across restarts.
- Adding this as a public config/API surface would need schema, docs, SDK/plugin-contract, and channel-specific tests so third-party plugins have a stable path.
