## Summary

Current main still only applies static per-message mention gating, so the requested attention window remains a valid cross-channel feature gap. It should stay open because adding remembered auto-listen state after a mention needs maintainer product/API decisions around privacy defaults, scope, expiry, config schema, docs, and channel delivery behavior.

## What This Changes

The issue requests an opt-in configurable attention window so group/channel chats process unmentioned follow-up messages for a limited time after the agent is mentioned.

## Best Possible Solution

Add an opt-in shared channel mention-policy contract for attention windows, with explicit per-conversation expiry semantics, config schema/help, docs, and focused tests, then wire supported channel plugins through that contract without weakening existing allowlist, command-auth, or mention-gating boundaries.

## Reproduction Assessment

Yes. Configure a group/channel with `requireMention` enabled, send a mentioned message, then send an unmentioned follow-up; current main evaluates only the follow-up's own mention facts and takes the existing skip/store path.

## Solution Assessment

Unclear. The user problem is valid, but the proposed single `attentionWindow` field is not enough design by itself because the maintainable solution spans shared policy, config, privacy semantics, timer state, and channel-specific delivery constraints.

## Risks / Open Questions

- Processing unmentioned group messages after a mention changes privacy and token-exposure semantics, so the default, scope, expiry, and audit behavior need explicit maintainer design.
- Some transports may not deliver unmentioned group messages unless platform-level settings permit it; Telegram privacy mode is already documented as a requirement for `requireMention=false`-style behavior.
- Timer state needs a clear contract for account, channel, thread/topic, session, restart behavior, and command-gating interaction before a cross-channel implementation is safe.
