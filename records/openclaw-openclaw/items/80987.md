## Summary

Keep open. The PR targets a credible Telegram/gateway starvation cluster, but the proposed setup semaphore does not cover the actual grammY getMe initialization path, the channels.status cache can replay a short-timeout result to a later longer probe, and external after-fix real behavior proof is still missing.

## What This Changes

The PR adds Telegram polling setup throttling, teardown ordering, dirty transport disposal, a gateway channels.status probe cache, and focused unit tests.

## Best Possible Solution

Rework the fix so throttling wraps the actual grammY init/getMe path, make status caching respect probe inputs or cache only safe reusable outcomes, rebase over the current Telegram ingress worker, then provide redacted live/Testbox proof with multiple Telegram accounts.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full production freeze; the PR provides logs and unit tests, but no live current-main or after-fix multi-account reproduction. Source and upstream dependency inspection do reproduce the proposed semaphore gap because grammY performs getMe in bot.init() after run(bot), not in createTelegramBot().

## Solution Assessment

Is this the best way to solve the issue? No. The teardown direction is plausible, but the central throttle misses the actual initialization network call and the status cache needs a safer key or reuse policy before this is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Hold the setup slot through bot initialization:** `extensions/telegram/src/polling-session.ts:305-306`
  - body: The slot is released as soon as `createTelegramBot()` returns, but grammY's constructor does not call `getMe`; `@grammyjs/runner` calls `bot.init()` from the first update supply after `run(bot)` starts. That means multiple accounts can still enter the getMe/TLS startup path concurrently, so the restart-burst throttle does not cover the network work it is meant to serialize.
  - confidence: 0.92
- **[P2] Include probe inputs in the cache key:** `src/gateway/server-methods/channels.ts:417-418`
  - body: The new cache is keyed only by channel/account, but `channels.status` callers pass different `timeoutMs` values. A short UI/watchdog probe that times out can be replayed to a later CLI probe with a longer timeout within 30s without running the hook, which breaks the explicit timeout contract and makes troubleshooting show stale failures.
  - confidence: 0.86
