## Summary

Keep open. This PR targets a real Telegram/gateway hot path, but current main has since landed related startup-probe and isolated-ingress fixes, and the proposed bot-setup semaphore appears to guard the wrong grammY phase; it also lacks after-fix real Telegram/runtime proof required for transport changes.

## What This Changes

The PR adds Telegram polling bot setup throttling, teardown ordering and dirty-transport disposal, plus a 30s `channels.status` probe cache and focused tests.

## Best Possible Solution

Rebase or replace the branch on current main, keep only unique changes still needed after the startup-probe limiter and isolated ingress work, move throttling to the actual async grammY/probe path, and add real Telegram/Testbox proof.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established in this read-only review. The production logs and current source make the event-loop starvation path plausible, and the PR's semaphore bug is source-reproducible from the grammY contract.

## Solution Assessment

No. The current PR is not the best fix as written because the bot setup semaphore wraps `createTelegramBot` while grammY performs `getMe` in `bot.init()` during runner startup, and current main already solved related startup fanout and polling-ingress isolation in narrower merged work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Throttle the actual grammY init request:** `extensions/telegram/src/polling-session.ts:274-306`
  - body: The new setup slot is acquired around `createTelegramBot`, but grammY's `Bot` constructor only stores config and optional `botInfo`; the `getMe` request happens later in `bot.init()`, which `@grammyjs/runner` calls after `run(bot)` starts. Because this `finally` releases the slot before the runner reaches that init path, the claimed TLS/getMe burst is not capped when `botInfo` is missing. Move the limiter around the real async init/probe path, or rely on the current startup-probe limiter and remove this no-op guard.
  - confidence: 0.87
