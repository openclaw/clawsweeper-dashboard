## Best Possible Solution

Throttle the actual asynchronous Telegram API startup/probe paths, keep the dirty-transport cleanup targeted, and require redacted live Telegram proof from a multi-account restart or soak scenario before merge.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection supports the unthrottled polling/probe paths, but the reported 7s freeze pattern depends on a long-running 4+ account Telegram setup and production freeze dumps that were not available here.

## Solution Assessment

No. The cache/cleanup direction may be useful, but the semaphore must wrap the actual grammY initialization or Telegram API startup work rather than createTelegramBot construction, and the branch needs real Telegram behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Hold the setup slot through Telegram initialization:** `extensions/telegram/src/polling-session.ts:305-307`
  - body: The slot is released as soon as `createTelegramBot` returns, but grammY's constructor only builds the local bot/API object; `getMe` happens later in `bot.init()` from the runner and `deleteWebhook` also runs after this block. As written, concurrent restarts can still overlap the TLS-heavy Telegram API calls this semaphore is meant to cap, so the main freeze fix is ineffective. Keep the slot until the actual startup API work completes, or throttle that API path directly.
  - confidence: 0.9
- **[P2] Wait for the real runner stop before re-entering setup:** `extensions/telegram/src/polling-session.ts:501-503`
  - body: The new teardown still awaits `waitForGracefulStop(stopRunner)`, which resolves after the grace timeout even if the runner's original stop promise is still pending. The PR says the next setup waits for prior teardown to settle, but after a forced restart this can still proceed while the old runner is unsettled. Track and await the real stop promise before starting the next polling bot, and add coverage for the timeout path.
  - confidence: 0.78
