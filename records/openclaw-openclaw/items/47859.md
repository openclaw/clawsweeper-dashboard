## Summary

Keep open: current main still lacks TUI `@path` or `/image` attachment UX, but this PR is not mergeable as proposed because source-reviewable defects remain and the external contributor has not provided real TUI behavior proof.

## What This Changes

The branch adds TUI inline `@path` image parsing, a `/image <path>` command, TUI-to-Gateway attachment forwarding, and parser/loader unit tests.

## Best Possible Solution

Land a rebased TUI implementation that preserves prompt bytes, strips local path tokens, handles `/image` as documented, extends or gates the shared `TuiBackend` contract across Gateway and local modes, and includes redacted real TUI proof.

## Reproduction Assessment

Yes for the PR defects: the prompt rewriting, immediate `/image` send, raw path fallback, and backend-contract mismatch are source-reproducible from the PR head and current main. I did not run the branch because this review is read-only and the missing evidence is contributor real-environment proof.

## Solution Assessment

No. Reusing Gateway `chat.send.attachments` is a reasonable direction, but this patch needs a cleaner shared TUI contract, corrected message semantics, safer attachment loading, and real behavior proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always collapses repeated whitespace and trims the result, and `sendMessage` uses that value for every non-btw message. Pasted prompts with indentation or intentional spacing are silently changed even when no image token was present, so the model receives different text than the user submitted.
  - confidence: 0.95
- **[P2] Queue /image attachments for the next prompt:** `src/tui/tui-command-handlers.ts:506-514`
  - body: The PR describes `/image path` as attaching an image to the next message, but this handler immediately sends a placeholder message with the attachment. That prevents users from pairing the selected image with the prompt they intended to type next.
  - confidence: 0.86
- **[P2] Strip attachment-only path tokens before sending:** `src/tui/tui-command-handlers.ts:565`
  - body: For a prompt like `@/tmp/screenshot.png`, `cleanText` is empty and `message = cleanText || text` falls back to the raw local path token. That displays and sends filesystem details instead of sending only the attachment payload or a neutral placeholder.
  - confidence: 0.94
- **[P2] Gate /image until all TUI backends support attachments:** `src/tui/commands.ts:116`
  - body: The slash command is added to the shared command list, but current main routes TUI through a shared `TuiBackend` used by both Gateway and embedded local modes, and the PR only forwards attachments on the Gateway client path. Rebase this through the shared backend contract or hide the command where attachments cannot be carried.
  - confidence: 0.84
- **[P3] Check the bytes read before encoding attachments:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` before `fs.readFile`, so a file can be replaced or enlarged after passing the limit and before base64 encoding. Enforce the cap on the actual buffer length before sending.
  - confidence: 0.78
