## Summary

Keep open. Current main has a Gateway `chat.send.attachments` seam, but it does not implement TUI `@path` parsing or `/image`; the PR still has unique feature work, while the branch needs rebase, correctness fixes, security cleanup, and real behavior proof before merge.

## What This Changes

The PR adds a TUI image parser/loader, a `/image` slash command, inline `@path` attachment extraction, tests, and attachment forwarding from TUI to Gateway chat sends.

## Best Possible Solution

Rebase through shared `TuiBackend` and Gateway `chat.send.attachments`, preserve prompt bytes, queue `/image` for the next prompt, strip local path tokens, cap the actual bytes read, and add focused tests plus redacted real TUI proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the diff shows prompt rewriting, immediate `/image` send behavior, attachment-only raw path fallback, and pre-read size validation. I did not run the branch, and the PR still lacks contributor real-environment proof.

## Solution Assessment

No, not as-is. Reusing Gateway `chat.send.attachments` is the right direction, but the PR needs corrected message semantics, shared backend plumbing, safer loading, and real behavior proof before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always collapses whitespace and trims `cleanText`, and `sendMessage` uses that result for every non-`/btw` message. Pasted prompts with indentation or intentional double spaces can be silently changed before they reach the model.
  - confidence: 0.95
- **[P2] Queue `/image` attachments for the next prompt:** `src/tui/tui-command-handlers.ts:506-514`
  - body: The PR says `/image path` attaches an image to the next message, but this handler immediately sends a placeholder message with the attachment. That prevents users from pairing the selected image with the prompt they intended to type next.
  - confidence: 0.87
- **[P2] Strip attachment-only path tokens before sending:** `src/tui/tui-command-handlers.ts:565`
  - body: For a prompt like `@/tmp/screenshot.png`, `cleanText` is empty and `message = cleanText || text` falls back to the raw local path token. That displays and sends filesystem details instead of sending only the attachment payload or a neutral placeholder.
  - confidence: 0.94
- **[P2] Thread attachments through the shared TUI backend:** `src/tui/commands.ts:116`
  - body: The command is advertised in the shared slash-command list, but current main routes TUI sends through `TuiBackend` for Gateway and embedded local modes while the PR only extends the older Gateway client path. Rebase through the shared backend contract or avoid exposing the command where attachments cannot be carried.
  - confidence: 0.84
- **[P3] Check the bytes read before encoding attachments:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` before `fs.readFile`, so a file can be replaced or enlarged after passing the limit and before base64 encoding. Enforce the cap on the actual buffer length before sending.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces local file attachment handling and has concrete local-path disclosure and size-check race concerns.

Concerns:

- **[medium] Attachment-only prompts leak local paths:** `src/tui/tui-command-handlers.ts:565`
  - body: The `cleanText || text` fallback sends raw `@/path` tokens when the image reference is the whole prompt, exposing filesystem details to the transcript and model instead of stripping the reference.
  - confidence: 0.94
- **[low] File size guard races the read:** `src/tui/tui-image-extract.ts:88`
  - body: The loader checks file size with `stat` before reading, so the actual buffer sent to Gateway can exceed the intended cap if the file changes between calls.
  - confidence: 0.78
