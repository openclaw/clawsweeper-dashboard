## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve original prompts when no images are attached:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always trims and collapses whitespace, and `sendMessage` uses `cleanText` for every non-`/btw` message. A pasted prompt with indentation or intentional double spaces reaches the model changed even when no image reference was extracted.
  - confidence: 0.95
- **[P2] Queue `/image` for the next prompt:** `src/tui/tui-command-handlers.ts:506-514`
  - body: The PR says `/image path` attaches an image to the next message, but this handler immediately sends a placeholder message with the attachment. That prevents the user from pairing the selected image with the prompt they intended to type next.
  - confidence: 0.87
- **[P2] Strip `@path` tokens from attachment-only sends:** `src/tui/tui-command-handlers.ts:565`
  - body: When the prompt is only an image reference, `cleanText` is empty and `message = cleanText || text` falls back to the original local path token. That exposes filesystem details and defeats the extraction step for the common attachment-only case.
  - confidence: 0.94
- **[P2] Thread attachments through `TuiBackend`:** `src/tui/commands.ts:116`
  - body: Current main routes TUI sends through the shared `TuiBackend` for both Gateway and embedded local mode, but the PR only extends the old Gateway client path while advertising `/image` in the shared command list. Rebase through the shared backend contract so the command is not exposed where attachments cannot be carried.
  - confidence: 0.84
- **[P3] Check the actual buffer before encoding:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` before `fs.readFile`, so a file can change after passing the limit and before it is encoded. Enforce the cap on `buffer.byteLength` for the bytes actually sent.
  - confidence: 0.78
