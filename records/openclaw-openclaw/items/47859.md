## Summary

This PR remains useful but is not merge-ready: current main does not implement the proposed `@path` or `/image` TUI UX, and the branch still has source-backed correctness, security, and proof blockers.

## What This Changes

The PR adds TUI image attachment support by parsing inline `@path` image references, adding a `/image <path>` slash command, forwarding attachments through `chat.send`, and adding extractor tests.

## Best Possible Solution

Keep the feature open, but rework it through the shared `TuiBackend` and gateway attachment contract with exact prompt preservation, next-prompt `/image` semantics, safe byte checks, and focused tests.

## Reproduction Assessment

Yes for the PR defects: source inspection of the diff shows prompt rewriting, immediate `/image` sending, attachment-only raw path fallback, and pre-read size validation. I did not run the branch because this is a read-only review and the PR lacks contributor real-environment proof.

## Solution Assessment

No, not as-is. The gateway attachment seam is the right underlying contract, but the PR should be rebased through the current shared TUI backend and corrected before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always collapses repeated whitespace and trims the result, and `sendMessage` uses that result even when no image reference was extracted. Pasted prompts with indentation or intentional spacing can reach the model changed from what the user typed.
  - confidence: 0.95
- **[P2] Queue `/image` attachments for the next prompt:** `src/tui/tui-command-handlers.ts:506-514`
  - body: The PR advertises `/image path` as attaching an image to the next message, but this handler immediately sends a placeholder message with the attachment. That prevents users from pairing the selected image with the prompt they intended to type next.
  - confidence: 0.87
- **[P2] Strip attachment-only path tokens before sending:** `src/tui/tui-command-handlers.ts:565`
  - body: For a prompt like `@/tmp/screenshot.png`, `cleanText` is empty and `message = cleanText || text` falls back to the raw local path token. That displays and sends filesystem details instead of sending only the attachment payload or a neutral placeholder.
  - confidence: 0.94
- **[P2] Thread attachments through the shared TUI backend:** `src/tui/commands.ts:116`
  - body: The command is added to the shared slash-command list, but current main routes TUI sends through `TuiBackend.sendChat` while the PR only extends `GatewayChatClient.send`. Rebase the feature through the shared backend contract or avoid exposing a command where attachments cannot be carried.
  - confidence: 0.84
- **[P3] Check the bytes read before encoding attachments:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` before `fs.readFile`, so a file can be replaced or enlarged after passing the limit and before base64 encoding. Enforce the cap on the actual buffer length before sending.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces local-file upload from the TUI and has concrete local path leakage plus pre-read size validation concerns.

Concerns:

- **[medium] Attachment-only prompts leak local paths:** `src/tui/tui-command-handlers.ts:565`
  - body: The `cleanText || text` fallback sends raw local `@path` tokens when the image reference is the whole prompt, exposing filesystem details to the transcript and model instead of stripping the reference.
  - confidence: 0.94
- **[low] File size guard races the read:** `src/tui/tui-image-extract.ts:88`
  - body: The loader checks file size with `stat` before reading, so the actual buffer sent to Gateway can exceed the intended cap if the file changes between calls.
  - confidence: 0.78
