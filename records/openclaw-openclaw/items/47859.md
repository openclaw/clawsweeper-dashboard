Full review comments:

- **[P1] Preserve original prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always trims and collapses whitespace, and `sendMessage` uses `cleanText` for every non-`/btw` prompt. Pasted code, indentation, or intentional double spaces can reach the model changed even when no image was attached.
  - confidence: 0.95
- **[P1] Honor workspace-only image access:** `src/tui/tui-image-extract.ts:101`
  - body: This loader resolves absolute, home, and relative paths then reads them directly before forwarding image content. Route file access through the existing sandbox/workspace media policy so `/image` cannot bypass configured file boundaries.
  - confidence: 0.88
- **[P2] Queue /image for the next prompt:** `src/tui/tui-command-handlers.ts:506-514`
  - body: The PR says `/image path` attaches an image to the next message, but this handler immediately sends a placeholder message with the attachment. That prevents the user from pairing the selected image with the prompt they intended to type next.
  - confidence: 0.9
- **[P2] Avoid sending raw @path text for attachment-only messages:** `src/tui/tui-command-handlers.ts:565`
  - body: When the prompt is only an image reference, `cleanText` is empty and `message = cleanText || text` falls back to the original local path token. That exposes filesystem details and defeats the extraction step for attachment-only prompts.
  - confidence: 0.96
- **[P2] Carry attachments through the shared TUI backend:** `src/tui/gateway-chat.ts:37`
  - body: Current main sends TUI messages through the shared `TuiBackend` for both Gateway and embedded local mode, but this branch only adds attachments to the older Gateway client shape while advertising `/image` in shared commands. Rebase the contract so both modes agree before exposing the command.
  - confidence: 0.85
- **[P3] Check the actual buffer before encoding:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` before `fs.readFile`, so a file can change after passing the limit and before it is encoded. Enforce the cap on `buffer.byteLength` for the bytes actually sent.
  - confidence: 0.78
