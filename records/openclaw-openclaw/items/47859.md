## Summary

Keep open: the feature is useful, but the PR still has source-reproducible correctness and security blockers, lacks real TUI proof, and needs maintainer judgment on how image attachments fit the shared Gateway/local TUI backend contract.

## What This Changes

The PR adds TUI `@path` image extraction, a `/image` command, Gateway `chat.send` attachment forwarding, and parser/loader tests.

## Best Possible Solution

Land a rebased TUI implementation that queues attachments through the shared `TuiBackend`/Gateway/local contract, preserves prompt bytes, strips local path tokens, enforces byte limits on the buffer read, and includes redacted real TUI proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head shows prompt rewriting, immediate `/image` send behavior, attachment-only raw path fallback, and a shared-backend mismatch against current main. I did not run the branch because this is a read-only review and the missing evidence is contributor real-environment proof.

## Solution Assessment

No. Reusing Gateway `chat.send.attachments` is a reasonable direction, but this patch needs corrected message semantics, shared backend plumbing, safer attachment loading, and real behavior proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always collapses repeated whitespace and trims `cleanText`, and `sendMessage` uses that result for every non-`/btw` message. Pasted prompts with indentation or intentional spacing are silently changed even when they contain no image token, so the model receives different text than the user submitted.
  - confidence: 0.95
- **[P2] Queue /image attachments for the next prompt:** `src/tui/tui-command-handlers.ts:506-514`
  - body: The PR describes `/image path` as attaching an image to the next message, but this handler immediately sends a placeholder message with the attachment. That prevents users from pairing the selected image with the prompt they intended to type next.
  - confidence: 0.86
- **[P2] Strip attachment-only path tokens before sending:** `src/tui/tui-command-handlers.ts:565`
  - body: For a prompt like `@/tmp/screenshot.png`, `cleanText` is empty and `message = cleanText || text` falls back to the raw local path token. That displays and sends filesystem details instead of sending only the attachment payload or a neutral placeholder.
  - confidence: 0.94
- **[P2] Thread attachments through the shared TUI backend:** `src/tui/commands.ts:116`
  - body: The command is advertised in the shared slash-command list, but current main routes TUI through a shared `TuiBackend` for both Gateway and embedded local modes while the PR only adds attachments to the Gateway client path. Rebase this through the shared backend contract or hide the command where attachments cannot be carried.
  - confidence: 0.84
- **[P3] Check the bytes read before encoding attachments:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` before `fs.readFile`, so a file can be replaced or enlarged after passing the limit and before base64 encoding. Enforce the cap on the actual buffer length before sending.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces local-file upload behavior with concrete local path leakage and file-size validation concerns.

Concerns:

- **[medium] Attachment-only prompts leak local paths:** `src/tui/tui-command-handlers.ts:565`
  - body: The `cleanText || text` fallback sends raw local path tokens when the image reference is the whole prompt, exposing filesystem details to the transcript and model instead of stripping the reference.

  - body: The loader checks file size with `stat` before reading, so the actual buffer sent to Gateway can exceed the intended cap if the file changes between calls.
  - confidence: 0.78
