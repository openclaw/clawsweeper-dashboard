## Summary

Keep this PR open: the TUI image-attachment feature is not implemented on current main, but the branch is not mergeable as-is because source inspection shows message-semantics bugs, incomplete shared-backend plumbing, security-sensitive local-path leakage, and no contributor real-environment proof.

## What This Changes

The PR adds TUI `@path` image extraction, a `/image <path>` command, attachment forwarding through `chat.send`, and parser/loader tests.

## Best Possible Solution

Thread local image attachments through shared `TuiBackend` and Gateway `chat.send.attachments`, preserve prompt bytes, queue `/image` for the next prompt, enforce limits on the actual buffer read, add focused tests, and include redacted real TUI proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of the diff shows prompt rewriting, immediate `/image` send behavior, attachment-only raw path fallback, and pre-read size validation. I did not run the branch because this is a read-only review and contributor real-environment proof is missing.

## Solution Assessment

No, not as-is. Reusing Gateway `chat.send.attachments` is the right seam, but the patch needs corrected message semantics, shared backend plumbing, safer loading, and real behavior proof before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` normalizes whitespace and trims `cleanText` even when no image reference was extracted, and `sendMessage` then sends that rewritten text. Pasted prompts with indentation or intentional double spaces can reach the model changed from what the user typed.
  - confidence: 0.95
- **[P2] Queue `/image` attachments for the next prompt:** `src/tui/tui-command-handlers.ts:506-514`
  - body: The PR says `/image path` attaches an image to the next message, but this handler immediately sends a placeholder message with the attachment. That prevents users from pairing the selected image with the prompt they intended to type next.
  - confidence: 0.87
- **[P2] Strip attachment-only path tokens before sending:** `src/tui/tui-command-handlers.ts:565`
  - body: For a prompt like `@/tmp/screenshot.png`, `cleanText` is empty and `message = cleanText || text` falls back to the raw local path token. That displays and sends filesystem details instead of sending only the attachment payload or a neutral placeholder.
  - confidence: 0.94
- **[P2] Thread attachments through the shared TUI backend:** `src/tui/commands.ts:116`
  - body: The command is advertised in the shared slash-command list, but current main routes TUI sends through `TuiBackend` for both Gateway and embedded local modes while the PR only extends the old Gateway client path. Rebase through the shared backend contract or avoid exposing the command where attachments cannot be carried.
  - confidence: 0.84
- **[P3] Check the bytes read before encoding attachments:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` before `fs.readFile`, so a file can be replaced or enlarged after passing the limit and before base64 encoding. Enforce the cap on the actual buffer length before sending.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces local-file upload from the TUI and has concrete local path leakage plus pre-read size validation issues.

Concerns:

- **[medium] Attachment-only prompts leak local paths:** `src/tui/tui-command-handlers.ts:565`
  - body: The `cleanText || text` fallback sends raw `@/path` tokens when the image reference is the whole prompt, exposing filesystem details to the transcript and model instead of stripping the reference.
  - confidence: 0.94

  - body: The loader checks file size with `stat` before reading, so the actual buffer sent to Gateway can exceed the intended cap if the file changes between calls.
