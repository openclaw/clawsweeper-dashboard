## Summary

Keep open: the feature is not implemented on current main, but the proposed patch still has source-proven correctness and security issues plus no real behavior proof from the external contributor.

## What This Changes

The PR adds TUI image attachment support through inline `@path` parsing, a `/image` slash command, Gateway `chat.send` attachment forwarding, and parser/loader tests.

## Best Possible Solution

Land a rebased TUI implementation that extends or gates the shared backend contract, preserves prompt text, strips local path tokens, uses existing Gateway media validation, and includes real TUI proof.

## Reproduction Assessment

Yes. The PR defects are source-reproducible from the diff: the parser rewrites prompts without image refs, the send fallback leaks attachment-only `@path` tokens, and the command is exposed through a shared TUI surface whose current main backends are text-only.

## Solution Assessment

No. Reusing Gateway `chat.send.attachments` is the right direction, but this patch is not the best form until the shared Gateway/local backend boundary, prompt preservation, path stripping, size check, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always collapses repeated whitespace and trims `cleanText`, and `sendMessage` then uses that value even when `paths.length === 0`. A pasted prompt with indentation or intentional spacing is changed before it reaches the model despite containing no image token.
  - confidence: 0.95
- **[P2] Strip attachment-only path tokens before sending:** `src/tui/tui-command-handlers.ts:565`
  - body: For a message such as `@/tmp/screenshot.png`, `cleanText` is empty and `message = cleanText || text` falls back to the raw local path token. That sends and displays the local path instead of sending only the attachment payload or a neutral placeholder.
  - confidence: 0.94
- **[P2] Gate image command until shared backends support attachments:** `src/tui/commands.ts:116`
  - body: The `/image` command is added to the shared command list, but current main runs the same command surface against both Gateway and local embedded TUI backends. The patch only forwards attachments through the Gateway client path, so local mode would expose a command whose backend contract does not handle attachments.
  - confidence: 0.84
- **[P3] Check the bytes read before encoding attachments:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat` and then reads the file separately, so the file can be replaced or enlarged after passing the 5 MB guard. Enforce the limit on the actual `Buffer.byteLength` before base64 encoding.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Needs attention because the patch adds local-file upload behavior and has concrete path-leak and size-check defects.

Concerns:

- **[medium] Attachment-only prompts leak local paths:** `src/tui/tui-command-handlers.ts:565`
  - body: The `cleanText || text` fallback sends raw `@/local/path` text when the image reference is the whole prompt, exposing local filesystem details to the transcript/model instead of stripping the reference.
  - confidence: 0.94
- **[low] File size guard races the read:** `src/tui/tui-image-extract.ts:88`
  - body: The loader validates `fs.stat().size` before `fs.readFile`, so a file that passed the limit can be replaced or enlarged before the bytes are read and encoded.
