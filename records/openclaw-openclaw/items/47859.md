## Summary

Keep open: current `main` still lacks TUI image attachment sending, but this external PR is not merge-ready because the diff has source-reproducible correctness bugs, no real TUI behavior proof, and needs maintainer judgment on the terminal local-file upload UX/policy.

## What This Changes

The PR adds TUI image attachments through inline `@path` parsing, a `/image` slash command, Gateway `chat.send` attachment forwarding, and parser/loader unit tests.

## Best Possible Solution

Land a rebased TUI implementation that extends or gates the shared backend contract, preserves prompt bytes when no image reference is present, strips local path tokens, reuses Gateway attachment validation and size limits, and includes real TUI proof.

## Reproduction Assessment

Yes for the PR defects: they are source-reproducible from the diff against current `main` contracts. I did not run the branch because this was a read-only review and the missing proof is contributor real-environment evidence, not a unit-test gap.

## Solution Assessment

No. Reusing Gateway `chat.send.attachments` is the right direction, but this patch is not the best form until backend contract coverage, prompt preservation, attachment-only path stripping, byte-size validation, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always normalizes `cleanText` with `replace(/\s{2,}/g, " ").trim()`, and `sendMessage` uses that value for every non-`/btw` message. Pasted prompts with indentation or intentional spacing are silently changed even when they contain no image reference.
  - confidence: 0.95
- **[P2] Queue /image attachments for the next message:** `src/tui/tui-command-handlers.ts:486-506`
  - body: The PR describes `/image path` as attaching an image to the next message, but this handler immediately sends `[image: file]` with the attachment. That prevents the user from pairing the selected image with the intended follow-up prompt.
  - confidence: 0.86
- **[P2] Strip attachment-only path tokens before sending:** `src/tui/tui-command-handlers.ts:565`
  - body: For input like `@/tmp/screenshot.png`, `cleanText` is empty and `message = cleanText || text` falls back to the raw local path token. That sends and displays filesystem details instead of sending only the attachment payload or an explicit neutral placeholder.
  - confidence: 0.94
- **[P2] Gate image commands until every TUI backend carries attachments:** `src/tui/commands.ts:113`
  - body: The slash command is added to the shared command list, while current `main` uses a shared `TuiBackend` for Gateway and embedded local modes and only the Gateway path is taught to forward attachments. Local mode would expose an image command whose backend contract cannot carry the payload.
  - confidence: 0.84
- **[P3] Check the bytes read before encoding attachments:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` before `fs.readFile`, so a file can be replaced or enlarged after passing the limit and before base64 encoding. Enforce the cap on the actual `Buffer.byteLength` before sending.
  - confidence: 0.78
