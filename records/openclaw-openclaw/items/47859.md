## Summary

Keep open, but do not merge as-is: the TUI image attachment feature is still not on current main, while the branch is conflicting and has source-reproducible prompt, local-path leakage, backend-contract, and size-validation defects plus no real behavior proof.

## What This Changes

The PR adds TUI inline image path parsing, a `/image` command, Gateway chat attachment forwarding, and parser/loader unit tests.

## Best Possible Solution

Land a rebased TUI implementation that extends or gates the shared backend contract, preserves prompt bytes when no image reference is present, strips local path tokens, reuses Gateway attachment validation, and includes real TUI proof.

## Reproduction Assessment

Yes. The PR defects are source-reproducible from the diff and current-main contracts: prompt text can be rewritten, attachment-only prompts can leak raw local path tokens, and the shared TUI backend contract remains text-only.

## Solution Assessment

No. Reusing Gateway `chat.send.attachments` is the right direction, but this patch is not the best form until the backend boundary, prompt preservation, path stripping, byte-size check, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always collapses whitespace and trims `cleanText`, and `sendMessage` uses that value even when `paths.length === 0`. Pasted prompts with indentation or intentional spacing are changed before reaching the model despite containing no image reference.
  - confidence: 0.95
- **[P2] Queue /image attachments for the next message:** `src/tui/tui-command-handlers.ts:486-506`
  - body: The PR describes `/image path` as attaching an image to the next message, but the handler immediately sends `[image: file]` with the attachment. That prevents the user from pairing the selected image with the intended follow-up prompt.
  - confidence: 0.86
- **[P2] Strip attachment-only path tokens before sending:** `src/tui/tui-command-handlers.ts:565`
  - body: For input like `@/tmp/screenshot.png`, `cleanText` is empty and `message = cleanText || text` falls back to the raw local path token. That sends and displays filesystem details instead of sending only the attachment payload or an explicit neutral placeholder.
  - confidence: 0.94
- **[P2] Gate image commands until each TUI backend carries attachments:** `src/tui/commands.ts:116`
  - body: The command is added to the shared slash-command list, while current main uses that surface for Gateway and embedded local TUI backends and only the Gateway path is taught to forward attachments. Local mode would expose an image command whose backend cannot carry the payload.
  - confidence: 0.84
- **[P3] Check the bytes read before encoding attachments:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` before `fs.readFile`, so a file can be replaced or enlarged after passing the limit and before base64 encoding. Enforce the cap on the actual `Buffer.byteLength` before sending.
  - confidence: 0.78
