## Summary

Keep open: current main does not implement this TUI image-upload feature, and the proposed patch still has source-reproducible blockers plus missing real behavior proof.

## What This Changes

The PR adds TUI image attachment support by parsing inline image path references, adding an image slash command, forwarding attachments through Gateway chat, and adding parser/loader tests.

## Best Possible Solution

Land a rebased TUI implementation that extends the shared backend contract, preserves prompt text, avoids local path leakage, reuses Gateway attachment validation, and includes real TUI proof.

## Reproduction Assessment

Yes. The PR defects are source-reproducible from the diff and current main contracts: prompt text is rewritten, attachment-only prompts can leak raw path tokens, and the shared TUI command surface is exposed while the shared backend contract is still text-only.

## Solution Assessment

No. Reusing Gateway `chat.send.attachments` is the right direction, but this patch is not the best form until the backend boundary, prompt preservation, path stripping, size check, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve prompts when no image refs are present:** `src/tui/tui-image-extract.ts:59`
  - body: `extractImagePaths` always collapses repeated whitespace and trims `cleanText`, and `sendMessage` then uses that value even when `paths.length === 0`. Pasted prompts with indentation or intentional spacing are changed before reaching the model despite containing no image reference.
  - confidence: 0.95
- **[P2] Queue /image attachments instead of sending immediately:** `src/tui/tui-command-handlers.ts:486-506`
  - body: The PR describes `/image path` as attaching an image to the next message, but the command handler immediately sends `[image: file]` with the attachment. This prevents the user from pairing the selected image with the intended follow-up prompt.
  - confidence: 0.86
- **[P2] Strip attachment-only path tokens before sending:** `src/tui/tui-command-handlers.ts:565`
  - body: For a message like `@/tmp/screenshot.png`, `cleanText` is empty and `message = cleanText || text` falls back to the raw local path token. That sends and displays local filesystem details instead of sending only the attachment payload or an explicit neutral placeholder.
  - confidence: 0.94
- **[P2] Gate image commands until all TUI backends support attachments:** `src/tui/commands.ts:116`
  - body: The command is added to the shared slash-command list, but current main uses the same TUI command surface for Gateway and embedded local backends while only the Gateway path is taught to forward attachments. Local mode would expose an image command whose backend ignores the attachment payload.
  - confidence: 0.84
- **[P3] Check bytes read before encoding attachments:** `src/tui/tui-image-extract.ts:88-101`
  - body: The loader checks `fs.stat().size` and then reads the file separately, so the file can be replaced or enlarged after passing the limit and before base64 encoding. Enforce the limit on the actual `Buffer.byteLength` before sending.
  - confidence: 0.78
