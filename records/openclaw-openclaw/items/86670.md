## Review Findings

Overall confidence: 0.89

Full review comments:

- **[P2] Pass the API key to live catalog discovery:** `extensions/edenai/provider-catalog.ts:271-275`
  - body: `discoverEdenaiModels()` calls Eden AI's `/v3/models` endpoint without any `Authorization` header. Since the catalog hook appends `apiKey` only after `buildEdenaiProvider()` returns, authenticated users still fall back to the static list and `prepareDynamicModel()` cannot validate non-static Eden AI models as promised; thread the resolved key into discovery or use the shared live-catalog helper with `apiKey`.
  - confidence: 0.94
- **[P2] Use an endpoint class core will read:** `extensions/edenai/openclaw.plugin.json:13-14`
  - body: Current core only loads manifest `providerEndpoints` whose `endpointClass` is in the known `ProviderEndpointClass` set, and `edenai` is not in that set. This makes the `edenai.run` endpoint metadata ineffective; remove the misleading metadata or add core endpoint-class/request-policy support with focused tests.
  - confidence: 0.91
- **[P2] Parse Eden AI's documented capability fields:** `extensions/edenai/provider-catalog.ts:218-221`
  - body: The live catalog parser looks for `capabilities.supports_reasoning` and `input_modalities`, but Eden AI documents `capabilities.reasoning`, `pdf`, `web_search`, and `tool_calling` for `/v3/models`. Live-discovered models that are not in the static fallback will therefore lose reasoning metadata and other supported capability facts; align the parser and tests with the documented schema.
  - confidence: 0.86
