## Summary

Keep open. Current main still preserves and surfaces nonblank aborted assistant partials, and #48283 remains an open unmerged implementation candidate with explicit closing syntax for this issue.

## What This Changes

The issue requests that aborted agent runs discard partial or buffered assistant content instead of writing or delivering unrelated assistant responses.

## Best Possible Solution

Land a focused abort-path gate that prevents stale or aborted assistant content from becoming transcript history, channel delivery, or an ordinary final chat reply, while preserving any deliberately supported preview-only UI behavior with explicit metadata and updated tests/docs.

## Reproduction Assessment

Yes for a code-level reproduction: current tests seed chatRunBuffers with nonblank text and assert chat.abort broadcasts or persists that text as assistant content. The exact Telegram/sub2api runtime race remains unverified in this read-only pass.

## Solution Assessment

No, the current main behavior is not the best solution for the reported trust issue because it treats aborted buffered text as assistant output. The safer direction is to discard or quarantine aborted content before final/channel/transcript delivery and only expose clearly marked UI previews if maintainers still want that behavior.
