## Summary

Keep open: current main still lacks `ctx.agentId` on typed message hook contexts, and this PR targets the right plugin API surface, but the branch needs changes before merge because the outbound resolver can mix mirror and sender identities and the public API change is missing docs/changelog coverage.

## What This Changes

The PR adds optional `sessionKey` and `agentId` fields to typed plugin message hook contexts and threads them through inbound and outbound message hook delivery paths with tests.

## Best Possible Solution

Land a rebased additive hook-contract update that exposes optional `ctx.agentId` for `message_received`, `message_sending`, and `message_sent`, deriving it from the same selected session identity as `ctx.sessionKey` and documenting the public field.

## Reproduction Assessment

Yes. A source-level reproduction is enough: current main exposes `sessionKey` but no `agentId` in typed message hook context, inbound dispatch already resolves an agent id without adding it to the hook context, and outbound typed hooks omit session identity.

## Solution Assessment

No. The additive API direction is maintainable, but this branch is not the best mergeable solution until it rebases to current main, keeps mirror/session identity internally consistent, and updates docs plus changelog.
