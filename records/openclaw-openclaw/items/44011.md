## Summary

Keep open. Current main still does not expose `ctx.agentId` on typed message hook contexts, and outbound typed hooks still omit available session identity, so the PR addresses a real remaining API gap. The branch is useful but not merge-ready as proposed because it can mix mirror and sender identity and needs the public hooks docs updated.

## What This Changes

The PR adds optional `sessionKey` and `agentId` to typed plugin message hook context plumbing for inbound `message_received` and outbound `message_sending`/`message_sent`, with mapper, dispatch, delivery, and test updates.

## Best Possible Solution

Land a rebased additive hook contract update in the current split type files that exposes optional `ctx.agentId` alongside `ctx.sessionKey` for `message_received`, `message_sending`, and `message_sent`, derives both fields from the same selected session identity, and documents the new plugin-facing field.

## Reproduction Assessment

Yes. A source-level reproduction is enough: current main’s typed message context lacks `agentId`, inbound mapping does not derive it, and outbound typed hook calls do not pass session identity even when delivery has a session or mirror.

## Solution Assessment

No. The additive API direction is maintainable, but this branch is not the best mergeable solution until it rebases to current main, keeps mirror/session identity internally consistent, and updates public plugin hook documentation.
