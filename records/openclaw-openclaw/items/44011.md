## Summary

Keep open: the requested hook-context gap is real, but this PR is not ready to merge because it can expose mixed session identity, lacks public message-hook documentation for `ctx.agentId`, has no real behavior proof, and now conflicts/overlaps with newer outbound-hook work.

## What This Changes

The branch adds optional `sessionKey` and `agentId` to typed message hook context, threads them through inbound and outbound message hooks, and adds focused mapper and delivery tests.

## Best Possible Solution

Land one rebased additive hook-context contract that uses one identity source for `sessionKey`/`agentId`, documents message-hook semantics, coordinates with https://github.com/openclaw/openclaw/pull/63007, and includes focused tests plus redacted real plugin/runtime proof.

## Reproduction Assessment

Yes, at source level: current main lacks `ctx.agentId` in `PluginHookMessageContext`/`toPluginMessageContext`, and outbound `message_sending` passes only channel/account/conversation in typed `ctx`. I did not run a live plugin reproduction in this read-only review.

## Solution Assessment

No, not as written. The additive API direction is plausible, but the PR should avoid mixed identity sources, document the public hook field, rebase onto current hook files, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep hook session identity from one source:** `src/infra/outbound/deliver.ts:410-411`
  - body: When `mirror.sessionKey` is present but `mirror.agentId` is absent, these independent fallbacks expose the mirror session key with the sender session's agent id. Pick one identity source for both fields, or derive/omit `agentId` from the selected session key, so plugins do not see a mixed identity.
  - confidence: 0.87

  - confidence: 0.74
