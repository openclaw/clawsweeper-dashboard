## Summary

Keep open: current main still lacks `agentId` in typed message hook context and shared outbound delivery still drops available session identity, so the API gap is real. The submitted branch should not merge as-is because it can mix mirror and sender identities, lacks public docs and real behavior proof, and overlaps the open outbound identity work.

## What This Changes

The PR adds optional `sessionKey` and `agentId` to typed plugin message hook context and threads them through inbound and outbound message hook dispatch with focused mapper and delivery tests.

## Best Possible Solution

Land one rebased additive hook-context contract that selects a single session identity source, derives or omits `agentId` coherently, documents the public semantics, and coordinates the outbound `message_sending` subset with the related open outbound identity work.

## Reproduction Assessment

Yes, at source level: current main still has no `ctx.agentId` in typed message hook context and shared outbound `message_sending`/`message_sent` paths do not pass available session identity into plugin hook context. I did not establish a live plugin reproduction in this read-only review.

## Solution Assessment

No, not as written. The additive API direction is plausible, but the branch mixes identity sources, misses public docs for `agentId`, conflicts with current main's hook type split, and still needs real behavior proof.
