## Summary

Keep open: the API direction is useful, but this PR is not merge-ready against current main because it conflicts with the live hook-message type surface, can expose mixed mirror/session identity, lacks after-fix real behavior proof, and overlaps the canonical outbound hook identity work.

## What This Changes

The PR adds optional `sessionKey` and `agentId` to typed plugin message-hook context and threads them through inbound and outbound hook dispatch with focused mapper and delivery tests.

## Best Possible Solution

Rebase or replace this with one current-main additive hook-context patch that preserves existing fields, selects session identity from one coherent source, documents `agentId` semantics, and coordinates with https://github.com/openclaw/openclaw/pull/73706 and https://github.com/openclaw/openclaw/issues/18571.

## Reproduction Assessment

Yes, at source level. Current main still computes outbound session identity near delivery but does not pass it into `message_sending`, while `agentId` is not part of the live typed message hook context.

## Solution Assessment

No, not as written. The additive API direction is plausible, but the branch must be rebased onto the live hook contract, keep identity fields from one source, document public semantics, and supply real runtime proof.
