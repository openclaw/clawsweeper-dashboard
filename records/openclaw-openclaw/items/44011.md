## What This Changes

The branch adds optional `sessionKey` and `agentId` to typed plugin message-hook context and threads them through inbound/outbound hook mappers and delivery tests.

## Best Possible Solution

Rebase or replace this with one current-main additive hook-context patch that preserves existing fields, selects session identity from one coherent source, documents `agentId` semantics, and coordinates with the narrower `sessionKey` and broader verification-contract work.

## Reproduction Assessment

Yes, at source level. Current main still computes outbound session identity near delivery but does not pass it into `message_sending`, while `agentId` is not part of the live typed message-hook context.

## Solution Assessment

No, not as written. The additive API direction is plausible, but the branch must target the current hook contract, keep identity fields from one source, document public semantics, and supply real runtime proof.
