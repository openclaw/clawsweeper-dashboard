## Summary

This PR still targets a real hook-context gap on current main, but it should stay open because the branch is now conflicting, has a concrete identity-mixing bug, and lacks required real runtime proof.

## What This Changes

Adds optional `sessionKey` and `agentId` to typed plugin message-hook contexts and threads them through inbound/outbound hook mapping and tests.

## Best Possible Solution

Land a rebased additive hook-contract change that exposes available session and agent identity from one consistent source, updates docs, and includes focused regression plus real plugin/runtime proof.

## Reproduction Assessment

Yes, at source level: current main has `sessionKey` but no `agentId` in `PluginHookMessageContext`, and inbound/outbound message hook dispatch does not pass `agentId` into `ctx`. I did not run a live plugin reproduction in this read-only review.

## Solution Assessment

No, not as written. The additive API direction is plausible, but the branch must be rebased onto the current hook contract files, keep session and agent identity from one source, update docs, and provide real behavior proof before merge.
