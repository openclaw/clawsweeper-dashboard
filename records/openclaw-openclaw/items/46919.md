## Summary

Keep open. Current main and the latest release still use tokenized dashboard URLs when token auth is active, do not expose a `urlToken`/`tokenURL` config key, and deliberately keep tokenless Tailscale behavior scoped to a verified Control UI/WebSocket path, so this remains a Gateway security/product decision rather than cleanup.

## What This Changes

Request to add a config flag that disables dashboard URL tokens and broadens trusted tailnet access without token, HTTPS, pairing, or device-identity requirements.

## Best Possible Solution

Make an explicit Gateway security/product decision; if accepted, design a narrow private-ingress mode with schema, docs, audit warnings, origin/proxy semantics, and focused tests instead of one broad token, HTTPS, pairing, and device-identity bypass flag.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main and `v2026.5.20` shows the requested config keys are absent and the existing tokenless/trusted paths are narrower than the requested blanket bypass.

## Solution Assessment

No/unclear. A single flag spanning dashboard URL fragments, HTTP auth, WebSocket auth, pairing/device identity, HTTPS, and bind policy is not the safest maintainable design without security/product approval.
