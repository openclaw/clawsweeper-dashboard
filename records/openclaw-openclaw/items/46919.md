## Summary

Keep open. Current main still has only narrow tokenless/trusted paths and SecretRef-based suppression of dashboard tokenized URLs; the requested config flag would relax Gateway admin auth, device identity, pairing, HTTPS, and tailnet bind policy, so it needs explicit secops/product review rather than cleanup closure.

## What This Changes

Request adds a config flag to disable tokenized dashboard URLs and token, HTTPS, and pairing requirements for trusted tailnet/private Gateway access.

## Best Possible Solution

Make an explicit secops/product decision; if accepted, design a narrowly scoped private-ingress mode with schema, docs, audit checks, origin/proxy semantics, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main shows no `urlToken`/`tokenURL` switch and confirms the supported tokenless/trusted modes remain narrower than the requested blanket tailnet/private bypass.

## Solution Assessment

No/unclear. A blanket one-flag bypass across dashboard tokenization, HTTP auth, WebSocket auth, pairing/device identity, and HTTPS/bind policy is not the safest maintainable design; a secops-reviewed private-ingress design or explicit rejection is safer.
