## Summary

Keep open. Current main still supports only narrower auth paths, and the requested no-token/no-HTTPS/no-pairing tailnet/private mode would intentionally relax Gateway admin authentication, so it needs maintainer/secops product judgment rather than cleanup closure; this agrees with the prior ClawSweeper review already in the discussion.

## What This Changes

The issue requests a config-controlled way to disable tokenized Control UI/dashboard access and related HTTPS, pairing, and forced URL behavior for trusted tailnet/private deployments.

## Best Possible Solution

Make an explicit maintainer/secops decision for the trusted-tailnet/private-ingress story; if accepted, design a narrowly scoped, warning-heavy mode with schema, docs, security-audit coverage, origin/proxy semantics, and focused tests, otherwise document the supported alternatives and close manually.

## Reproduction Assessment

Not applicable as a feature request. Current-main source and docs give a high-confidence verification path that no `urlToken`/`tokenURL` flag exists and that existing bypasses remain narrower than requested.

## Solution Assessment

No/unclear. The requested blanket bypass is not clearly the best maintainable solution; a secops-reviewed trusted-tailnet/private-ingress design or an explicit rejection with documented supported paths is safer.
