## Summary

Keep open. Current main still only supports narrower identity-bearing auth paths, while the requested no-token/no-HTTPS/no-pairing tailnet/private mode is a security-sensitive feature/config policy decision.

## What This Changes

The issue requests a config flag to disable tokenized Gateway/Control UI URL auth and related HTTPS, URL, pairing, or tailnet access requirements.

## Best Possible Solution

Make an explicit maintainer/secops decision for a trusted-tailnet/private-ingress design, then either document the supported path or implement a narrow warning-heavy mode with schema, docs, and security coverage.

## Reproduction Assessment

Not applicable as a feature request. Current-main source and docs provide a high-confidence verification path that the requested urlToken/tokenURL-style bypass does not exist and existing bypasses are narrower.

## Solution Assessment

No/unclear. A blanket config flag that disables token, HTTPS, URL, and pairing requirements is security-sensitive; a secops-reviewed trusted-tailnet/private-ingress design or explicit documented rejection is safer.
