## Summary

Keep open. Current main and v2026.5.12 still have no supported `urlToken`/`tokenURL` switch, and the requested broader no-token/trusted-tailnet behavior would relax Gateway admin authentication, device identity, pairing, HTTPS, and bind-policy boundaries that are already tracked as security/product-sensitive.

## What This Changes

The issue requests a config flag to disable tokenized dashboard URLs and related Gateway authentication, HTTPS, device identity, pairing, and tailnet bind restrictions for trusted private-network deployments.

## Best Possible Solution

Make an explicit secops/product decision; if accepted, design a narrow private-ingress mode with schema/help metadata, public docs, audit checks, origin/proxy semantics, and focused tests for each affected Gateway auth surface.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main and v2026.5.12 confirms there is no `urlToken`/`tokenURL` switch and existing tokenless/trusted paths are narrower than the requested blanket bypass.

## Solution Assessment

No/unclear. A single flag that bypasses dashboard tokenization, HTTP auth, WebSocket auth, pairing/device identity, and HTTPS/bind policy is not the safest maintainable design; a secops-reviewed private-ingress contract or explicit rejection is safer.
