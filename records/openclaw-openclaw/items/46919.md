## Summary

Keep open. Current main and v2026.5.19 still include tokenized dashboard URLs when token auth is active, do not expose a `urlToken`/`tokenURL`-style config key, and keep Tailscale/no-auth behavior inside security-sensitive Gateway policy rather than a small bug fix.

## What This Changes

The issue requests a config flag to disable tokenized dashboard URLs and broadly trust private or tailnet access without token, HTTPS, pairing, or device identity checks.

## Best Possible Solution

Make an explicit Gateway security/product decision; if accepted, design a narrow private-ingress mode with schema, docs, audit warnings, origin/proxy semantics, and focused tests instead of one broad token-and-pairing bypass flag.

Not applicable as a feature request. Source inspection on current main shows the requested keys are absent and the existing tokenless/trusted paths are narrower than the requested blanket bypass.

## Solution Assessment

No/unclear. A single flag spanning dashboard URL fragments, HTTP auth, WebSocket auth, pairing/device identity, HTTPS, and bind policy is not the safest maintainable design without security/product approval.
