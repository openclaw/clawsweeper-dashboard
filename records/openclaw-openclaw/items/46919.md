## Summary

Keep open. Current main still lacks the requested config flag to disable tokenized dashboard access plus no-HTTPS/no-pairing direct tailnet or IP Control UI access, and the remaining work would intentionally relax Gateway admin-surface authentication, so it needs maintainer/secops product judgment rather than conservative cleanup closure.

## What This Changes

Request to add a config flag such as `urlToken`/`tokenURL` that disables tokenized dashboard URLs, HTTPS/device-identity/pairing requirements, and direct tailnet/IP auth friction for trusted tailnet deployments.

## Best Possible Solution

Make an explicit product/security decision for the trusted-tailnet/private-ingress story. If accepted, implement a narrowly named, warning-heavy mode with origin policy, HTTP API scope semantics, device identity and pairing behavior, Tailscale Serve/trusted-proxy interactions, config schema/help/docs, and focused tests; if rejected, close manually with the supported alternatives: Tailscale Serve identity-header auth, trusted-proxy auth, SSH tunnel, or local loopback `auth.mode: "none"`.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: source, schema, runtime tests, and docs all show no `urlToken`/`tokenURL` flag and preserve narrower auth bypasses only.

## Solution Assessment

Unclear/no. The requested blanket no-token/no-HTTPS/no-pairing tailnet mode is not clearly the best maintainable solution; a safer direction would be a deliberately scoped trusted-tailnet/private-ingress design or an explicit maintainer rejection with documented supported paths.
