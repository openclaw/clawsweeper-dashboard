## Summary

Keep open. Current main fixes the Telegram forum-topic `threadId` pass-through and adds inter-session provenance marking, but it still suppresses visible delivery when the announce step returns `ANNOUNCE_SKIP`; requester-vs-target A2A routing and skip-token semantics are still active follow-up areas in #57697 and #49223.

## What This Changes

The issue asks for `sessions_send` Telegram forum-topic handoffs to preserve topic/thread delivery, keep intended visible target replies from disappearing after `ANNOUNCE_SKIP`, and avoid echoing `inter_session` handoffs into the requester transcript.

## Best Possible Solution

Keep the shipped `threadId` and inter-session provenance fixes, then define and land a coordinated A2A delivery policy for requester-vs-target announce routing, when a target reply may be reused after skip tokens, and how peer handoffs avoid transcript pollution across Telegram and other channels.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction remains on current main: make the target produce a normal round-one reply, make the announce step return `ANNOUNCE_SKIP`, and observe that `runSessionsSendA2AFlow` does not call gateway `send`; the current test asserts that zero-send behavior.

## Solution Assessment

No. Current main is a partial solution: the topic thread and provenance pieces are implemented, but the visible-delivery fallback and requester-side routing policy still need maintainer-selected semantics rather than being treated as fully fixed.
