## Summary

Close as implemented on current main. This is the same Windows media-store fsync EPERM failure already tracked in #76640/#76752, and main now treats only post-write `EPERM` from `FileHandle.sync()` as best-effort while preserving other media-save errors.

## What This Changes

Fix Telegram inbound voice/audio messages on native Windows failing during media download with `EPERM: operation not permitted, fsync`.

## Best Possible Solution

Keep the shared media-store EPERM guard and ship it in the next release; open a narrower follow-up only if Windows reports a distinct non-EPERM media fsync failure.

## Reproduction Assessment

Yes for the affected release path: mock `FileHandle.sync()` inside `saveMediaBuffer()` to throw `EPERM` after the write succeeds, which previously aborted Telegram media handling. Current main no longer reproduces that failure because the new regression test covers the successful-save path.

## Solution Assessment

Yes. Handling only `EPERM` in the shared media-store post-write sync is the narrowest maintainable fix because Telegram inbound media, WebChat, and channel uploads share that helper while other write/sync errors still propagate.
