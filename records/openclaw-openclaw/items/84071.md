## Summary

Keep this open: current main still uses a sticky controller-local session-file fingerprint that rejects any prompt-window file change, while the issue describes legitimate shared-session co-tenant writes and an open linked PR still owns the fix path.

## What This Changes

The issue reports false `EmbeddedAttemptSessionTakeoverError` failures when coordinated shared-session writers append during an embedded prompt I/O window.

## Best Possible Solution

Land a safe fix through https://github.com/openclaw/openclaw/pull/84149 or a replacement that makes the fence lock-coordination-aware while preserving fail-closed detection for uncoordinated session-file mutation.

## Reproduction Assessment

Yes. Source inspection gives a concrete reproduction path: release one embedded controller for prompt I/O, append to the same session file through another legitimate lock-owning co-tenant, then call the first controller's `withSessionWriteLock()`.

## Solution Assessment

No for current main: the controller-local fingerprint cannot distinguish coordinated co-tenant writes from external mutators. The best fix should use shared lock-mediated state or equivalent ownership-aware validation while preserving the external-takeover fence.
