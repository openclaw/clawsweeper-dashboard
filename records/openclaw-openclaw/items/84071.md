## Summary

Keep open: current main still has the sticky controller-local fingerprint behavior, and a linked open PR uses closing syntax for this issue, so the issue should wait for that PR to merge or be rejected.

## What This Changes

The issue reports that legitimate heartbeat, cron, and channel co-tenant writes to a shared session can trip `EmbeddedAttemptSessionTakeoverError` during the embedded prompt I/O lock-release window.

## Best Possible Solution

Land or replace the linked fix with an ownership-aware fence that allows coordinated co-tenant writes while preserving takeover detection for uncoordinated file changes.

## Reproduction Assessment

Yes. Source-level reproduction is clear: release one embedded controller for prompt I/O, append to the same session file through another legitimate lock-owning co-tenant path, then call the first controller's `withSessionWriteLock()`.

## Solution Assessment

Yes, the best direction is to make the fence ownership-aware or generation-aware instead of treating every coordinated prompt-window write as takeover. The linked PR is the active fix path, but it still needs PR review before this issue can close.
