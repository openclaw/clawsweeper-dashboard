## Summary

Keep this open. Current main still has the controller-local fingerprint fence that becomes sticky takeover state when the session file changes during the released prompt window, and the issue gives a focused co-tenant shared-session reproduction rather than a stale or already-fixed duplicate.

## What This Changes

The issue reports a regression where embedded session takeover detection treats legitimate co-tenant shared-session writes as hostile and stalls the session.

## Best Possible Solution

Make the embedded attempt fence ownership-aware for OpenClaw lock-coordinated same-session writes across controllers, while still rejecting uncoordinated external file changes.

## Reproduction Assessment

Yes, as a source-level reproduction: release one embedded controller for prompt I/O, mutate the same session file through another lock-owning run/controller, then call the first controller's `withSessionWriteLock()`. I did not run a live gateway repro in this read-only sweep.

## Solution Assessment

Yes. The requested ownership-aware fence refresh or shared generation check is the narrow maintainable fix; relaxing the fingerprint broadly or disabling takeover detection would be riskier.
