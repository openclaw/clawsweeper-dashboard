## Summary

Keep this open. Current main still uses a controller-local session-file fingerprint that becomes sticky takeover state when the file changes during the released prompt window, and the existing coverage only proves same-controller coordinated writes refresh the fence, not a legitimate second run/controller sharing the session.

## What This Changes

The issue asks for the embedded attempt session fence to stop treating legitimate co-tenant writes to shared sessions as takeover during the released prompt-lock window.

## Best Possible Solution

Make the embedded attempt fence recognize lock-coordinated same-session writes across controllers and refresh safely under the session write lock, while preserving takeover detection for uncoordinated file changes.

## Reproduction Assessment

Yes, as a source-level reproduction: release an embedded controller for prompt I/O, mutate the same session file through another lock-owning run/controller, then call the first controller's `withSessionWriteLock()`. I did not run a live gateway repro in this read-only sweep.

## Solution Assessment

No, the current controller-local fingerprint is not the best long-term shape because it cannot distinguish a coordinated co-tenant from an external mutator. The safer fix is a shared lock-coordinated fence refresh or generation check with focused regression coverage.
