## Summary

Keep this open: current main still uses a sticky controller-local session-file fingerprint, and the issue describes documented shared-session heartbeat/cron/channel workflows that can legitimately write during the released prompt window.

## What This Changes

The issue reports that the embedded attempt session-takeover fence incorrectly treats legitimate lock-coordinated co-tenant writes to shared sessions as hostile takeover after the 2026.5.17 lock-scope change.

## Best Possible Solution

Make the embedded attempt fence ownership-aware for lock-coordinated same-session writes across controllers, likely through a shared lock-mediated generation or refresh, while keeping external mutation fail-closed.

## Reproduction Assessment

Yes, as a source-level reproduction: release one embedded controller for prompt I/O, mutate the same session file through another legitimate lock-owning controller/co-tenant, then call the first controller's withSessionWriteLock. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, the current controller-local fingerprint is not the best shape because it cannot distinguish coordinated co-tenant writes from external mutators. The maintainable fix should refresh or validate the fence through shared lock-coordinated state.
