## Label Justifications

- `P1`: The issue describes a fresh regression that can stall shared-session agent/channel workflows during model I/O.
- `impact:session-state`: The failure path marks the embedded session controller as taken over and leaves sticky state for later writes.
- `impact:message-loss`: The reported stall can prevent model output and channel/session replies from completing.

## Summary

Keep this open: current main and the shipped v2026.5.18 fence still make any session-file fingerprint change during the released prompt window a sticky takeover, which matches the reported shared-session co-tenant failure.

## What This Changes

The issue asks OpenClaw to stop treating legitimate lock-coordinated heartbeat, cron, or channel writes to a shared session as an embedded session takeover.

## Best Possible Solution

Make the embedded attempt fence ownership-aware for lock-coordinated same-session writes across controllers, while keeping external or uncoordinated file mutation fail-closed.

## Reproduction Assessment

Yes, as a source-level reproduction: release one embedded controller for prompt I/O, mutate the same session file through another legitimate lock-owning controller or co-tenant, then call the first controller's withSessionWriteLock. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No current-main solution is in place: the controller-local fingerprint cannot distinguish coordinated co-tenant writes from external mutators. The safer fix is a shared lock-mediated generation or refresh path, not disabling the fence.
