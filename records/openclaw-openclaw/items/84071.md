## What This Changes

The issue reports that the embedded session-takeover fence treats coordinated heartbeat, cron, and channel writes to a shared session during prompt I/O as hostile mutations and stalls later writes.

## Best Possible Solution

Review and repair or land the linked co-tenant session-lock PR with external-takeover detection preserved and real shared-session proof attached.

## Reproduction Assessment

Yes, as a source-level reproduction: release one embedded controller for prompt I/O, mutate the same session file through another legitimate lock-owning co-tenant, then call that controller's `withSessionWriteLock()`. I did not run the live gateway reproduction in this read-only pass.

## Solution Assessment

Yes on direction, but not yet complete in main: the maintainable fix belongs in the session-lock fence so coordinated co-tenant writes can proceed while external takeovers still fail closed. The current open PR is the right review target, but it still has proof and session-integrity concerns to resolve before merge.
