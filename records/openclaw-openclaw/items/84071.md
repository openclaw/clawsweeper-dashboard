## Label Justifications

- `P1`: The report describes an active regression that can stall shared-session agent/channel work during model I/O.
- `impact:session-state`: The failure path marks the embedded controller as taken over and leaves sticky session state for later writes.
- `impact:message-loss`: The reported stall can prevent model output and channel/session replies from completing.

## Summary

Keep this open: current main still uses a controller-local session-file fingerprint that becomes sticky on any prompt-window file change, and the direct closing PR for the co-tenant path is still open rather than merged. The production confirmation in the discussion and the adjacent runner-owned append PR add useful context, but they do not make this issue safe to close before a fix lands.

## What This Changes

The issue asks OpenClaw to stop treating legitimate lock-coordinated heartbeat, cron, and channel writes to a shared session as embedded session takeover during the prompt I/O release window.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/84149, or a safer replacement, lands with an ownership-aware fence and focused regression coverage.

## Reproduction Assessment

Yes, source-level: release one embedded controller for prompt I/O, mutate the same session file through another legitimate lock-owning same-session path, then call the first controller's `withSessionWriteLock()`. I did not run a live heartbeat or channel-ingress reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution because the controller-local fingerprint cannot distinguish coordinated co-tenant writes from external mutators. The best fix should make the fence ownership-aware while keeping uncoordinated mutation fail-closed.
