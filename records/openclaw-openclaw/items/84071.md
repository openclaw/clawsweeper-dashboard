## Summary

Keep this open: current main and v2026.5.18 still use the sticky file-fingerprint fence, and the linked fix PR remains open with review/proof blockers. The report is source-reproducible and now has an additional production confirmation from a separate install.

## What This Changes

The issue reports that the embedded session takeover fence rejects legitimate lock-coordinated co-tenant writes to shared sessions during the released prompt I/O window.

## Best Possible Solution

Land a reviewed fix that accepts lock-coordinated same-session writes across controllers while still detecting uncoordinated external mutation, then close this issue from the merged PR.

## Reproduction Assessment

Yes, as a source-level reproduction: release one embedded controller for prompt I/O, mutate the same session file through another legitimate lock-owning controller/co-tenant, then call the first controller's withSessionWriteLock. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. Current main's controller-local fingerprint is not the best shape because it cannot distinguish coordinated co-tenant writes from external mutators; the maintainable fix should use shared lock-coordinated state or an equivalent safe fence design.
