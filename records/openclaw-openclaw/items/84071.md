## Summary

Keep this open: current main and the latest stable release still use the strict controller-local session-file fingerprint that overfires on legitimate lock-coordinated co-tenant writes, and the issue is already paired with an open closing PR that has not merged.

## What This Changes

The issue reports that the embedded session takeover fence mistakes legitimate same-session co-tenant writes during prompt I/O for hostile takeover and stalls shared sessions.

## Best Possible Solution

Review, repair, and land the linked co-tenant session-lock fix or replace it with a safer lock-mediated generation/fingerprint design, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, as a source-level reproduction: release one embedded controller for prompt I/O, mutate the same session file through another legitimate lock-owning controller/co-tenant, then call the first controller's `withSessionWriteLock()`. I did not run a live gateway reproduction in this read-only pass.

## Solution Assessment

No, current main is not the best solution because file-stat equality alone cannot distinguish coordinated co-tenant writes from external mutators. The best fix should allow lock-coordinated co-tenant progress while preserving external mutation detection.
