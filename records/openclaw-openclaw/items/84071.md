## Summary

Keep this open: current main and v2026.5.18 still use a sticky controller-local session-file fingerprint, and the issue describes documented shared-session heartbeat/cron/channel writes that can legitimately happen during the released prompt window. The related open PR is adjacent, but it targets runner-owned assistant/tool appends rather than this co-tenant shared-session path.

## What This Changes

The issue requests that the embedded attempt session-takeover fence allow legitimate lock-coordinated co-tenant writes to shared sessions during the released prompt window.

## Best Possible Solution

Make the embedded attempt fence ownership-aware for lock-coordinated same-session writes across controllers, likely through shared lock-mediated generation or refresh, while preserving external mutation takeover detection.

## Reproduction Assessment

Yes, as a source-level reproduction: release one embedded controller for prompt I/O, mutate the same session file through another legitimate lock-owning controller/co-tenant, then call the first controller's withSessionWriteLock. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes, the requested direction is the right fix shape: distinguish coordinated OpenClaw-owned writes from external mutation instead of relaxing the fence broadly. The adjacent runner-owned append PR does not cover the heartbeat/cron/channel co-tenant path.
