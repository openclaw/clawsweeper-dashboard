## Summary

Keep open. Current main still has source-level paths for both reported symptoms: `sessions_send` returns a hard `timeout` when `agent.wait` times out, and Gateway close code 1000 is still formatted and surfaced as `gateway closed (1000 normal closure)`. The related #68065 and open #68196 cover the narrower accepted-run timeout misclassification, but they do not fully cover this issue's Windows multi-agent normal-close/routing-loop report.

## What This Changes

Investigate and fix intermittent `sessions_send` timeouts and Gateway 1000 normal-closure failures in Windows multi-agent deployments where Gateway health checks still pass.

## Best Possible Solution

Keep this issue open as the broader tracking item. Land or supersede #68196 for the accepted-run wait-timeout contract, then collect Windows multi-agent diagnostics and add targeted 1000-normal-close recovery or Gateway/session-routing instrumentation once the failing boundary is identified.

## Reproduction Assessment

No for the full issue. Static source inspection gives a high-confidence path for the hard `sessions_send` timeout and the `gateway closed (1000 normal closure)` error string, but the repeated Windows multi-agent routing loop still needs logs, a diagnostic bundle, or a runnable scenario.

## Solution Assessment

Unclear because this is an issue, not a patch. Post-timeout compensation is the narrow solution for the #68065 overlap, but the unique normal-close/routing failure should not be solved with a broad routing-table reset until the failing Gateway boundary is reproduced.
