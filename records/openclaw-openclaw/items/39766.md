## Summary

Keep open. The omitted-`models` config problem is still source-reproducible on current main, but this PR still needs contributor real-behavior proof and fixes for the public config contract, deterministic test seam, and embedding private-network policy before merge.

## What This Changes

This PR defaults `models.providers.<id>.models` to an empty array and adds config plus memory embedding regression tests for provider entries that omit `models`.

## Best Possible Solution

Land a rebased narrow fix or replacement that accepts omitted provider `models` at the config input boundary, keeps public types/schema aligned, uses deterministic regression coverage, and applies maintainer-approved private-network policy to memory embedding baseUrls.

## Reproduction Assessment

Yes, by source inspection. Current main validates raw config through `OpenClawSchema.safeParse`, and `ModelProviderSchema` still requires `models`, so the omitted-model provider shape in this PR is rejected before embedding routing can use the custom `baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting or otherwise accepting omitted `models` is the right direction, but the patch also needs public contract alignment, deterministic current-seam tests, and a private-network policy decision for embedding baseUrls.

- **[P2] Expose the optional input shape in the public config contract:** `src/config/zod-schema.core.ts:266`
  - body: Changing only the Zod field to `.default([])` makes raw parsing accept omitted `models`, but `ModelProviderConfig.models` still requires the field.   - confidence: 0.85

  - body: Once this schema accepts baseUrl-only provider entries, those entries can reach the remote embedding client. That path currently turns the configured host into `allowedHostnames`, which skips private/internal address checks, so this fix needs to use the explicit `request.allowPrivateNetwork` policy or otherwise keep localhost/metadata/RFC1918 targets blocked by default.
  - confidence: 0.84
- **[P2] Avoid live DNS in the embedding regression:** `packages/memory-host-sdk/src/host/embeddings.test.ts:196-197`
  - body: The added embedding test calls `embedQuery` against `https://example.com/v1`, which goes through SSRF preflight DNS before the mocked fetch is reached. Use the current pinned-hostname/lookup seam or test `resolveRemoteEmbeddingBearerClient` directly so restricted CI DNS cannot fail this regression for reasons unrelated to baseUrl precedence.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Accepting omitted-model provider entries expands access to memory embedding `baseUrl` handling whose current configured-host policy can bypass private-network checks.

Concerns:

- **[medium] Configured embedding baseUrls can bypass private-network checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` delegates to `ssrfPolicyFromHttpBaseUrlAllowedHostname`, and the SSRF guard skips private/internal checks for hosts in `allowedHostnames`. Since this PR allows more provider baseUrl-only config shapes to reach that path, memory embeddings should use explicit `allowPrivateNetwork` semantics or block private/internal targets by default.
  - confidence: 0.82
