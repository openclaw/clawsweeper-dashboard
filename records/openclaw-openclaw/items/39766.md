## Summary

Keep open. Current main still rejects provider entries that omit `models`, but this PR is not ready to merge because it lacks real behavior proof, is currently conflicting, and leaves config-contract and embedding private-network policy gaps unresolved.

## What This Changes

The PR changes the model provider schema to default omitted `models` to an empty array and adds schema plus memory embedding regression tests for provider `baseUrl`-only entries.

## Best Possible Solution

Land a rebased narrow fix that accepts omitted provider `models` at the user config input boundary, aligns generated schema/type surfaces, uses deterministic embedding regression coverage, and preserves explicit private-network opt-in semantics.

## Reproduction Assessment

Yes, by source inspection. Current main validates raw config with `OpenClawSchema.safeParse`, and `ModelProviderSchema` still requires `models`, so the PR body's baseUrl-only provider entry fails before embedding routing can read it.

## Solution Assessment

No, not as submitted. Defaulting `models` is the right direction, but the branch must align the public config input/schema contract, avoid DNS-dependent tests, and resolve the embedding baseUrl private-network policy before merge.

  - body: Changing only this Zod field accepts the raw parse path, but the config schema path is still generated from `OpenClawSchema.toJSONSchema(...)` without an explicit input-shape contract and the hand-maintained config type still treats provider `models` as required.   - confidence: 0.84

  - body: The new embedding test uses `https://example.com/v1` and then calls `embedQuery`, so SSRF preflight DNS can run before the mocked fetch is reached. Use the current pinned-hostname/lookup seam or test `resolveRemoteEmbeddingBearerClient` directly so restricted CI DNS cannot fail this regression for reasons unrelated to baseUrl precedence.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Accepting omitted-model provider entries reaches the existing remote embedding `baseUrl` policy path, which allowlists configured hostnames and bypasses private-network checks without explicit opt-in.

Concerns:

- **[medium] Configured embedding baseUrls can bypass private-network checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` delegates to `ssrfPolicyFromHttpBaseUrlAllowedHostname`, and the SSRF guard skips private/internal checks for hosts in `allowedHostnames`. Since this PR makes more provider baseUrl-only config shapes valid, memory embeddings should use explicit `allowPrivateNetwork` semantics or block private/internal targets by default.
  - confidence: 0.82
