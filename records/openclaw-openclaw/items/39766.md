## Summary

Keep open: current main still rejects provider-only model entries, and this PR is an active but incomplete fix because it lacks real behavior proof and has blocking schema, security, and test concerns.

## What This Changes

The PR changes `ModelProviderSchema.models` to default to `[]` and adds schema plus memory embedding tests for baseUrl-only provider entries.

## Best Possible Solution

Land a refreshed focused fix that accepts provider-only config at the input boundary, exposes the same optional input contract through public config schema, preserves safe parsed output, routes memory embedding baseUrls through explicit private-network policy, and uses deterministic regression coverage.

## Reproduction Assessment

Yes, source-reproducible. The PR body config omits `models`, current main requires that field in `ModelProviderSchema`, and raw validation calls `OpenClawSchema.safeParse` before embedding code can use the configured provider baseUrl.

## Solution Assessment

No, not as submitted. Defaulting `models` is directionally right, but the patch still needs public config-schema alignment, deterministic test wiring, and maintainer-approved private-network handling for memory embedding baseUrls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Expose omitted models in the public config schema:** `src/config/zod-schema.core.ts:380`
  - body: Changing the Zod field to `.default([])` makes parsing accept omitted `models`, but `schema-base` still emits the output JSON schema and Zod defaults are output-required. Config schema consumers can therefore continue seeing `models.providers.*.models` as required unless this update also switches or overrides the public schema contract for config input.
  - confidence: 0.86
- **[P2] Keep private embedding endpoints behind explicit policy:** `src/config/zod-schema.core.ts:380`
  - body: Once baseUrl-only provider entries validate, they can reach the memory embedding client. That path currently converts the configured host into `allowedHostnames`, and the SSRF guard skips private/internal checks for allowed hostnames, so the fix needs to block private targets by default or require the existing explicit private-network opt-in.
  - confidence: 0.84
- **[P2] Remove live DNS from the embedding regression:** `packages/memory-host-sdk/src/host/embeddings.test.ts:197`
  - body: The new embedding test calls `embedQuery` against `https://example.com/v1`, which can perform SSRF preflight DNS before the mocked fetch is reached. Test `resolveRemoteEmbeddingBearerClient` directly or stub the pinned-hostname lookup so restricted CI DNS cannot fail this regression for unrelated reasons.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No workflow, dependency, or package-resolution changes were found, but the config change expands access to a memory embedding baseUrl path with a concrete private-network bypass concern.

Concerns:

- **[medium] Configured embedding baseUrls can bypass private-network checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` delegates to `ssrfPolicyFromHttpBaseUrlAllowedHostname`, and `shouldSkipPrivateNetworkChecks` treats configured allowed hostnames as a bypass for private/internal address checks. Since this PR accepts more baseUrl-only provider shapes, memory embeddings should use normal SSRF checks or require explicit `allowPrivateNetwork` before merge.
  - confidence: 0.82
