## What This Changes

The PR defaults `ModelProviderSchema.models` to `[]` and adds schema plus memory embedding tests for provider entries that omit `models`.

## Best Possible Solution

Land a rebased fix that defaults provider `models` to `[]`, keeps generated/types/docs contracts aligned, and threads existing provider request/private-network policy through memory embedding HTTP with deterministic regression coverage.

Yes. Source inspection shows current main requires `models` in `ModelProviderSchema`, so the omitted-model provider config is rejected before the memory embedding runtime can use `models.providers.<id>.baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting `models` is the right config-boundary direction, but this branch needs a rebase, provider request policy alignment for memory embeddings, deterministic tests, and contributor real behavior proof.
