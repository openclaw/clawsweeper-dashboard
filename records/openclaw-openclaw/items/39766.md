## Summary

Keep this PR open because current main still rejects provider entries that omit `models`, so the central config bug remains. The branch is still useful, but it should not merge as submitted because it lacks real behavior proof and has focused security/test gaps to resolve.

## What This Changes

The PR defaults `ModelProviderSchema.models` to `[]` and adds schema plus memory embedding regression tests for provider `baseUrl` entries that omit `models`.

## Best Possible Solution

Land a rebased config-contract fix that accepts omitted provider `models` while keeping parsed output coherent, uses deterministic regression tests, and aligns memory embedding baseUrl fetches with provider request policy.

## Reproduction Assessment

Yes, at source level. Current main requires `models` in `ModelProviderSchema`, and raw validation rejects the omitted field before memory embeddings can use `models.providers.<id>.baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting `models` to `[]` is the right config-boundary direction, but the PR needs current-surface tests, provider request policy alignment, and real behavior proof before merge.
