## Summary

Keep open. Current main still rejects provider entries that omit `models`, but this PR is not mergeable as-is because it leaves the public schema/output contract inconsistent, expands the embedding baseUrl path into a private-network policy question, adds a DNS-dependent regression test, and lacks after-fix real behavior proof.

## What This Changes

The PR defaults `ModelProviderSchema.models` to `[]` and adds schema plus memory-embedding tests for provider entries that configure `baseUrl` without declaring models.

## Best Possible Solution

Land a rebased narrow fix or replacement that accepts provider-only config at the Zod input boundary, exposes the same optional input contract through the public config schema, routes embedding baseUrls through maintainer-approved private-network policy, and puts deterministic coverage on the current remote embedding client seam.

## Reproduction Assessment

Yes, source-level. Current main validates raw config through `OpenClawSchema.safeParse`, and `ModelProviderSchema` still requires `models`, so the provider-only config in the PR body is rejected before embedding routing can use its custom baseUrl.

## Solution Assessment

No, not as submitted. Defaulting `models` is the right narrow direction, but the patch also needs public schema alignment, deterministic test wiring, and maintainer-approved private-network handling for memory embedding baseUrls.

- **[P2] Expose the optional input shape in the public config schema:** `src/config/zod-schema.core.ts:380`
  - body: Changing the Zod field to `.default([])` makes raw parsing accept omitted `models`, but `schema-base` currently emits the output JSON schema, and Zod defaults are still output-required.   - body: Once this schema accepts baseUrl-only provider entries, those entries can reach the remote embedding client. That path currently converts the configured host into `allowedHostnames`, which skips private/internal address checks, so the fix needs to wire embedding requests to the existing explicit private-network policy or otherwise keep localhost/metadata/RFC1918 targets blocked by default.
  - confidence: 0.84
- **[P2] Avoid live DNS in the embedding regression:** `packages/memory-host-sdk/src/host/embeddings.test.ts:197`
  - body: The new embedding test calls `embedQuery` against `https://example.com/v1`, which goes through SSRF preflight DNS before the mocked fetch is reached. Use the current pinned-hostname/lookup seam or test `resolveRemoteEmbeddingBearerClient` directly so restricted CI DNS cannot fail this regression for reasons unrelated to baseUrl precedence.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow supply-chain changes, but accepting omitted-model provider entries expands access to embedding baseUrl handling whose current host policy can bypass private-network checks.

Concerns:

- **[medium] Configured embedding baseUrls can bypass private-network checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` delegates to `ssrfPolicyFromHttpBaseUrlAllowedHostname`, and `resolvePinnedHostnameWithPolicy` skips private/internal checks for `allowedHostnames`.
