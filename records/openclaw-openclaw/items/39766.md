## Summary

Keep open: current main still rejects omitted `models` on provider entries, so the change is not obsolete, but this PR needs changes before merge because it expands a security-sensitive embedding `baseUrl` path, uses a DNS-dependent regression test, and has no after-fix real behavior proof.

## What This Changes

The PR defaults `ModelProviderSchema.models` to `[]` and adds schema plus memory embedding regression tests for provider entries that only set credentials or `baseUrl`.

## Best Possible Solution

Land a rebased replacement that accepts omitted provider `models` at the config boundary, keeps regression coverage deterministic, and routes memory embedding private endpoints through the existing explicit `request.allowPrivateNetwork` policy.

## Reproduction Assessment

Yes, at source level. Current main validates raw config with `OpenClawSchema.safeParse`, and `ModelProviderSchema` still requires `models`, so the omitted-model provider config is rejected before embedding routing can use its custom `baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting or otherwise accepting omitted `models` is the right direction, but the patch needs deterministic tests and maintainer-approved private-network handling for memory embedding base URLs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate private embedding endpoints behind explicit policy:** `src/config/zod-schema.core.ts:258-264`
  - body: Defaulting `models` lets baseUrl-only provider blocks pass validation and reach the memory embedding HTTP path. That path currently turns the configured host into `allowedHostnames`, which skips private/internal address checks, so localhost, RFC1918, or metadata endpoints can be reached without the existing explicit `request.allowPrivateNetwork` policy.
  - confidence: 0.86
- **[P2] Make the embedding regression deterministic:** `packages/memory-host-sdk/src/host/embeddings.test.ts:191-199`
  - body: The new regression relies on `example.com` resolving during SSRF preflight before the mocked `fetch` is reached. In restricted CI or offline local runs this can fail for DNS reasons, so use the pinned-hostname/lookup seam or test `resolveRemoteEmbeddingBearerClient` directly.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff expands a configured embedding `baseUrl` path that currently bypasses private-network checks via configured-host allowlisting.

Concerns:

- **[medium] Configured embedding baseUrls can bypass private-network checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` delegates to `ssrfPolicyFromHttpBaseUrlAllowedHostname`, and the SSRF guard skips private/internal checks when the requested host is in `allowedHostnames`; accepting omitted-model provider entries should not expose this path without explicit operator policy.
  - confidence: 0.84
