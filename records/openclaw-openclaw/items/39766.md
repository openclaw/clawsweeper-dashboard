## What This Changes

This PR defaults `ModelProviderSchema.models` to `[]` and adds schema plus memory embedding tests for provider `baseUrl` entries that omit `models`.

## Best Possible Solution

Land a rebased config-contract fix that accepts omitted provider `models` while keeping the parsed/type contract coherent and threading memory embedding baseUrls through the existing provider request/private-network policy.

Yes. Source inspection gives a high-confidence path: current main requires `models` in `ModelProviderSchema`, raw validation rejects the omitted field, and the memory embedding client otherwise has a provider `baseUrl` resolution path.

## Solution Assessment

No, not as submitted. Defaulting `models` is the right config-boundary direction, but the PR needs a rebase, real behavior proof, deterministic current-surface tests, and provider request/private-network policy alignment before merge.
