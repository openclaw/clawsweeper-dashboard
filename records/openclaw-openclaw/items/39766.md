## Summary

Keep open. The reported config-boundary bug is still source-reproducible on current main, and this PR remains the canonical focused fix direction, but the submitted patch still lacks real behavior proof and leaves provider request/private-network policy plus test determinism issues unresolved.

## What This Changes

The PR defaults `ModelProviderSchema.models` to `[]` and adds schema plus memory embedding regression tests for provider `baseUrl` entries that omit `models`.

## Best Possible Solution

Land a rebased, contract-complete fix that defaults provider `models` to `[]` or otherwise accepts omitted input with safe array output, aligns types/docs/schema surfaces, threads provider request policy into memory embedding HTTP, and uses deterministic regression coverage.

## Reproduction Assessment

Yes, at source level. Current main requires `models` in `ModelProviderSchema`, validation fails before runtime, and the memory remote client would otherwise use `models.providers.<id>.baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting `models` is the right config-boundary direction, but the patch still needs provider request policy alignment, deterministic tests, and real behavior proof before merge.

## Review Findings

- **[P2] Gate memory baseUrls through provider request policy:** `src/config/zod-schema.core.ts:258-264`
  - body: Defaulting `models` makes provider-only `baseUrl` entries valid for memory embeddings, but that HTTP path trusts the configured hostname and does not honor `models.providers.*.request.allowPrivateNetwork=false`.
