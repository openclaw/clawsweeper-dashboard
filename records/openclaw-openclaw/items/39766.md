## What This Changes

The PR defaults `ModelProviderSchema.models` to `[]`, adds schema regression tests, and adds a memory embedding test for provider `baseUrl` fallback.

## Best Possible Solution

Land a rebased fix that supports omitted provider `models` across Zod, public types/schema, deterministic regression coverage, and maintainer-approved memory embedding private-network semantics.

## Reproduction Assessment

Yes, source-level. Current main validates raw config through `OpenClawSchema.safeParse`, and `ModelProviderSchema` still requires `models`, so a provider-only entry is rejected before embedding routing can use its custom `baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting `models` is the right narrow direction, but the patch also needs public config contract alignment, deterministic test wiring, and maintainer-approved private-network handling for memory embedding baseUrls.

## Security Review

Status: needs_attention

Summary: Needs attention: accepting omitted-model provider entries expands access to embedding baseUrl handling whose current hostname policy can bypass private-network checks.

Concerns:

- **[medium] Configured embedding baseUrls can bypass private checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` delegates to `ssrfPolicyFromHttpBaseUrlAllowedHostname`, and the configured host becomes `allowedHostnames`. Since that policy can skip private/internal checks, memory embeddings should use normal SSRF checks or require explicit private-network opt-in before this config shape is accepted.
