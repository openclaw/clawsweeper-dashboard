## Summary

Keep open. Current main still has the reported config-validation bug, and the PR is useful but not merge-ready because the generated/public config contract remains stale, the new embedding test is DNS-dependent, and the configured embedding baseUrl path needs an explicit SSRF-policy decision before widening acceptance of provider entries without model lists.

## What This Changes

The PR changes `ModelProviderSchema.models` to default to an empty array and adds regression coverage for provider entries that only set credentials/baseUrl plus OpenAI embedding baseUrl fallback behavior.

## Best Possible Solution

Land an equivalent narrow fix after aligning the Zod input/default behavior, generated config schema, exported provider config types or separate input/parsed contracts, current memory-host regression tests, and the remote embedding baseUrl SSRF policy. The end state should let operators omit `models` for credential/baseUrl-only provider entries while keeping runtime arrays safe and preserving explicit opt-in for private-network model endpoints.

## Reproduction Assessment

Yes. A source-level reproduction is a config with `models.providers.openai.baseUrl` and credentials but no `models`; current `ModelProviderSchema` requires `models`, so validation fails before the embedding client can use `providerConfig?.baseUrl`.

## Solution Assessment

Partly. Defaulting `models` to `[]` is a reasonable runtime direction, but this PR is not yet the best mergeable solution because it leaves generated/public contracts stale, adds a DNS-dependent regression test, and does not settle the configured baseUrl SSRF behavior.

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow supply-chain change, but it widens acceptance of configured embedding baseUrls that currently flow through an SSRF policy needing maintainer review.

Concerns:

- **[medium] Configured embedding baseUrls can skip private-network checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` currently returns a policy with the configured host as `allowedHostnames`, and the SSRF resolver skips private-network checks for allowed hostnames. Since this PR is specifically meant to allow credential/baseUrl-only provider entries, maintainers should either change that policy or explicitly require `request.allowPrivateNetwork` for private/internal endpoints before merging.
