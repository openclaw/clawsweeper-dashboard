## Summary

Keep open. Current main still rejects provider entries that omit `models`, and the PR addresses a real config bug, but it is not mergeable as submitted because the generated schema/changelog are missing, the embedding regression needs rebasing onto the current provider split, and embedding `baseUrl` private-network semantics need maintainer/security judgment.

## What This Changes

The PR changes `ModelProviderSchema.models` to default to an empty array and adds schema plus remote embedding regression tests for provider entries without model lists.

## Best Possible Solution

Land a rebased narrow fix that makes provider model lists optional/defaulted, regenerates public config schema/docs, adds current-surface regression coverage, updates the changelog, and resolves embedding `baseUrl` SSRF policy explicitly.

## Reproduction Assessment

Yes, source-level. Current main validates raw config with `OpenClawSchema.safeParse`, and `ModelProviderSchema.models` is required, so the provider-only config in the PR body is rejected before embeddings can use the custom `baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting `models` is the right narrow direction, but the mergeable solution also needs generated/public schema alignment, current memory-provider tests, a changelog entry, and explicit private-network policy for embedding base URLs.
