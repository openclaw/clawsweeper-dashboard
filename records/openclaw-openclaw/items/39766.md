## Summary

Keep this PR open for follow-up, not cleanup-close: current main still rejects omitted `models.providers.<id>.models`, but the submitted patch still needs real behavior proof plus fixes for public config contract alignment, deterministic test coverage, and the memory embedding private-network policy.

## What This Changes

The PR changes `ModelProviderSchema.models` to default to an empty array and adds schema plus memory embedding tests for provider entries that omit `models` while configuring `baseUrl`.

## Best Possible Solution

Land a rebased narrow fix or replacement that accepts omitted provider `models` at the config input boundary, aligns public schema/types/help, uses deterministic regression tests, and keeps private/internal embedding baseUrls behind maintainer-approved explicit policy.

## Reproduction Assessment

Yes, at source level. Current main validates raw config with `OpenClawSchema.safeParse`, and `ModelProviderSchema` still requires `models`, so the omitted-model provider config in the PR body is rejected before embedding routing can use its custom `baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting or otherwise accepting omitted `models` is the right direction, but this patch also needs public config contract alignment, deterministic current-seam tests, and a maintainer-approved private-network policy for embedding baseUrls.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Align the public config input contract:** `src/config/zod-schema.core.ts:263-266`
  - body: Changing only the Zod field to `.default([])` makes parsing accept this shape, but the public config contract still needs to expose omitted `models` as a supported input. `ModelProviderConfig` still declares `models` as required, and the runtime-computed base schema path has no assertion that `models.providers.*` accepts provider blocks without `models`, so API/tooling consumers can keep rejecting the same config shape.
  - confidence: 0.82

  - body: Once omitted-model provider entries are accepted, baseUrl-only configs can reach the remote embedding client. That path currently turns the configured host into `allowedHostnames`, and the SSRF guard skips private/internal address checks for allowed hostnames, so memory embeddings should use explicit `request.allowPrivateNetwork` semantics or keep localhost/metadata/RFC1918 targets blocked by default.
  - confidence: 0.86
- **[P2] Avoid live DNS in the embedding regression:** `packages/memory-host-sdk/src/host/embeddings.test.ts:196-197`
  - body: The added embedding test calls `embedQuery` against `https://example.com/v1`, which goes through SSRF preflight DNS before the mocked fetch is reached. Use the pinned-hostname/lookup seam or test `resolveRemoteEmbeddingBearerClient` directly so restricted CI DNS cannot fail this regression for reasons unrelated to baseUrl precedence.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Accepting baseUrl-only provider entries expands access to a memory embedding URL policy that can bypass private-network checks unless explicit operator policy is enforced.

Concerns:

- **[medium] Configured embedding baseUrls can bypass private-network checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` delegates to `ssrfPolicyFromHttpBaseUrlAllowedHostname`, and the SSRF guard skips private/internal checks for hosts in `allowedHostnames`. Since this PR allows more provider baseUrl-only config shapes to reach that path, memory embeddings should use explicit `allowPrivateNetwork` semantics or block private/internal targets by default.
  - confidence: 0.84
