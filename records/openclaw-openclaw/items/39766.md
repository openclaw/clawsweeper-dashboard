## Summary

Keep this PR open. The reported config rejection still exists on current main, but the submitted patch is not merge-ready because it only changes the Zod field and test coverage while leaving public schema/input-contract and embedding private-network policy concerns unresolved.

## What This Changes

This PR changes `ModelProviderSchema.models` to default to an empty array and adds schema plus embedding tests for provider entries that omit `models` while configuring `baseUrl`.

## Best Possible Solution

Land a rebased narrow fix or replacement that accepts omitted provider `models` at the config input boundary, keeps generated/public schema and types aligned, preserves deterministic regression coverage, and routes private/internal embedding baseUrls through maintainer-approved policy.

## Reproduction Assessment

Yes. Source inspection on current main shows raw config validation still runs `OpenClawSchema.safeParse`, and `ModelProviderSchema` still requires `models`, so the omitted-model provider config in the PR body is rejected before embedding routing can use its custom `baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting or otherwise accepting omitted `models` is the right direction, but this patch also needs public config contract alignment, deterministic current-seam tests, and a maintainer-approved private-network policy for embedding baseUrls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Expose the optional input shape in the public config contract:** `src/config/zod-schema.core.ts:263-266`
  - body: Changing only the Zod field to `.default([])` can make raw parsing accept omitted `models`, but the public config schema path still emits the output schema and the hand-maintained config type still models provider entries as requiring `models`. Schema/tooling consumers can therefore keep rejecting the same omitted-model provider shape unless this patch aligns the input schema/type contract too.
  - confidence: 0.86
- **[P2] Keep private embedding endpoints behind explicit policy:** `src/config/zod-schema.core.ts:263-266`
  - body: Once this schema accepts baseUrl-only provider entries, those entries can reach the remote embedding client. That path currently turns the configured host into `allowedHostnames`, which skips private/internal address checks, so the fix needs to use the existing explicit private-network policy or otherwise keep localhost/metadata/RFC1918 targets blocked by default.
  - confidence: 0.84
- **[P2] Avoid live DNS in the embedding regression:** `packages/memory-host-sdk/src/host/embeddings.test.ts:196-197`
  - body: The added embedding test calls `embedQuery` against `https://example.com/v1`, which goes through SSRF preflight DNS before the mocked fetch is reached. Use the current pinned-hostname/lookup seam or test `resolveRemoteEmbeddingBearerClient` directly so restricted CI DNS cannot fail this regression for reasons unrelated to baseUrl precedence.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Accepting omitted-model provider entries expands access to embedding `baseUrl` handling whose current configured-host policy can bypass private-network checks.

Concerns:

- **[medium] Configured embedding baseUrls can bypass private-network checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` delegates to `ssrfPolicyFromHttpBaseUrlAllowedHostname`, and the SSRF guard skips private/internal checks for hosts in `allowedHostnames`. Since this PR allows more provider baseUrl-only config shapes to reach that path, memory embeddings should use explicit `allowPrivateNetwork` semantics or block private/internal targets by default.
  - confidence: 0.82
