## Summary

Keep this PR open: current main still requires `models.providers.<id>.models`, so the central omitted-models config change is not implemented. The related issue was closed after a runtime baseUrl-selection fix, but that commit does not address the schema rejection path this PR targets, and the submitted patch still needs real behavior proof plus security/test fixes before merge.

## What This Changes

The PR defaults `ModelProviderSchema.models` to `[]` and adds schema plus memory embedding regression coverage for provider entries that omit `models`.

## Best Possible Solution

Land a rebased fix that defaults provider `models` to `[]`, preserves the existing explicit private-network policy for memory embedding endpoints, and uses deterministic schema/runtime regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main requires `models` in `ModelProviderSchema`, and raw validation calls `OpenClawSchema.safeParse` before the memory embedding runtime can use `models.providers.<id>.baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting `models` is the right config-boundary direction, but the patch should also align memory embedding private endpoints with explicit policy and avoid DNS-dependent test coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate private embedding endpoints behind explicit policy:** `src/config/zod-schema.core.ts:258-264`
  - body: Defaulting `models` allows credential/baseUrl-only provider blocks to pass validation and reach memory embedding HTTP. That path builds policy from the configured host via `allowedHostnames`, which skips private/internal checks, so wire it to the existing explicit private-network policy before accepting this new config shape.
  - confidence: 0.83
- **[P2] Stub the SSRF lookup in the embedding regression:** `packages/memory-host-sdk/src/host/embeddings.test.ts:190-194`
  - body: The new regression relies on `example.com` resolving during SSRF preflight before the mocked `fetch` is reached. Restricted DNS or offline CI can fail for reasons unrelated to this behavior, so use the lookup/guard seam or test the remote client resolver directly.
  - confidence: 0.88
