## Summary

Keep this PR open, but not merge-ready. Current main still rejects provider entries that omit `models`, so the core bug remains; the submitted fix is the right general direction but needs schema/type alignment, deterministic tests, real behavior proof, and a maintainer/security decision on private-network embedding base URLs.

## What This Changes

The PR defaults `ModelProviderSchema.models` to `[]` and adds schema plus memory-embedding regression tests for provider entries that set `baseUrl` without declaring models.

## Best Possible Solution

Land a rebased focused fix or replacement that accepts provider-only config at the input boundary, keeps the output/runtime array contract clear, updates public schema/types, and routes memory embedding baseUrls through explicit private-network policy semantics with deterministic regression coverage.

## Reproduction Assessment

Yes, source-level. Current main validates raw config through `OpenClawSchema.safeParse`, and `ModelProviderSchema` still requires `models`, so the provider-only config in the PR body is rejected before embedding routing can use its custom `baseUrl`.

## Solution Assessment

No, not as submitted. Defaulting `models` is the right narrow direction, but this patch also needs public schema/type alignment, deterministic test wiring, and maintainer-approved private-network handling for memory embedding baseUrls.

  - body: Changing the Zod field to `.default([])` makes raw parsing accept omitted `models`, but the public config schema is generated from the output shape and the TypeScript config type still requires `models`.   - confidence: 0.84
- **[P2] Remove live DNS from the embedding regression:** `packages/memory-host-sdk/src/host/embeddings.test.ts:197`
  - body: The new embedding test drives `embedQuery` against `https://example.com/v1`, which goes through SSRF preflight DNS before the mocked fetch is reached. Test the remote client seam directly or inject the pinned-hostname lookup so restricted CI DNS cannot fail this regression for reasons unrelated to baseUrl precedence.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow supply-chain changes, but accepting omitted-model provider entries expands access to embedding baseUrl handling whose current host policy can bypass private-network checks.

Concerns:

- **[medium] Configured embedding baseUrls can bypass private-network checks:** `packages/memory-host-sdk/src/host/remote-http.ts:10`
  - body: `buildRemoteBaseUrlPolicy` delegates to `ssrfPolicyFromHttpBaseUrlAllowedHostname`, and `resolvePinnedHostnameWithPolicy` skips private/internal checks for `allowedHostnames`. Since this PR accepts more provider baseUrl-only config shapes, memory embeddings should either use normal SSRF checks for private targets or require the existing explicit `allowPrivateNetwork` opt-in before merge.
