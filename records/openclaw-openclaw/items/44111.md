## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `uploadSnapshot` writes under `backup.target/snapshots/<installationId>` by calling `fs.mkdir(..., { recursive: true })` and then `rename`/copying into that path, but no code rejects symlinked intermediate directories inside the target. A synced or shared backup target can therefore redirect snapshot payloads/envelopes outside the configured backup directory. Validate or create each storage directory segment without following symlinks before writing.
  - confidence: 0.9
- **[P1] Validate workspace backup roots before mkdir:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `mirrorRoot` and `.staging` before proving the target-side tree has no symlinked segments, and the later containment checks canonicalize through `realpath`. If `backup.target/workspace` or a child is a symlink, these mkdir/copy/rename/remove operations can operate outside the backup target. Reject symlinked target segments before any mkdir, cp, rename, or rm.
  - confidence: 0.92
- **[P1] Validate restore targets before rollback:** `src/commands/backup-restore.ts:442-443`
  - body: Existing restore targets are moved to rollback before `copySourceToTarget` validates the target parent with `ensureDirectoryPathWithoutSymlinks`. If the configured target path is reached through a symlinked parent segment, rollback creation and rename can still operate outside the intended restore area. Run the same target-parent segment validation before `fs.access`, rollback directory creation, and rename.
  - confidence: 0.83
