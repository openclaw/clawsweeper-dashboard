## Summary

Keep open: current main still lacks the native encrypted snapshot flow, and the PR head has blocking source-level restore and target-side symlink issues before it can merge.

## What This Changes

The PR adds folder-backed encrypted backup snapshots and workspace mirroring through `backup setup/run/status/list/restore`, updates backup config/schema/docs and aliases, and expands archive/snapshot validation coverage.

## Best Possible Solution

Refresh the branch, reject symlinked backup-target path segments before any writes, compare source and target in-state config/OAuth relative paths during restore, add the changelog entry, then land after maintainer/security review and focused backup validation.

## Reproduction Assessment

Yes. Source inspection of the PR head shows concrete paths where target-side symlinks are followed during snapshot/workspace writes, and where full-host restore skips config/OAuth recovery when source and target paths are both under state but at different relative locations.

## Solution Assessment

No for the PR as currently written. The native snapshot direction is plausible, but the implementation needs target-side no-symlink enforcement, source-vs-target relative-path restore checks, and the required changelog before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject symlinked workspace target directories before writes:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before rejecting symlinked target-side directories, and the later containment check resolves both paths through the same symlink. A synced or attacker-writable backup target can point `workspace` outside the backup directory, causing mirror writes and stale-mirror deletes to land outside `backup.target`.
  - confidence: 0.91
- **[P2] Reject symlinked snapshot store path segments:** `src/backup/snapshot-store/provider-folder.ts:67-85`
  - body: The folder store creates parent directories and writes through `targetDir/snapshots/<installationId>` without checking those intermediate directories for symlinks. If a synced backup target contains `snapshots` or an installation directory as a symlink, snapshot payload/envelope writes escape the configured target directory.
  - confidence: 0.88
- **[P2] Restore config when in-state paths differ:** `src/commands/backup-restore.ts:715-716`
  - body: This condition suppresses the config restore whenever the source config was inside state and the current config path is also inside state. If the backup used one in-state config relative path and the restore host uses another, replacing state restores only the old path and leaves the active `OPENCLAW_CONFIG_PATH` missing or stale.
  - confidence: 0.88
- **[P2] Restore OAuth creds when in-state paths differ:** `src/commands/backup-restore.ts:741-742`
  - body: The credentials branch has the same coverage assumption: any target OAuth dir inside `stateDir` is treated as covered by the state restore. When the backup's OAuth dir lived at a different relative path inside state, full-host restore completes without populating the active OAuth directory, so provider auth data is not recovered.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `docs/cli/backup.md:18-28`
  - body: This PR adds user-facing backup commands and changes the preferred archive command from `create` to `export`, but the PR head does not include a `CHANGELOG.md` entry. Repo policy requires user-facing feature/fix changes to appear in the active changelog section.
  - confidence: 0.82

## Security Review

Summary: Security review needs attention because target-side symlink traversal and secret-bearing restore recovery gaps remain source-reproducible on the PR head.

Concerns:

  - body: `backup.target/workspace` can be replaced with a symlink before `backup run`; recursive mkdir/cp/rename/rm then operate through that symlink and can write or delete outside the selected backup directory.
  - confidence: 0.91
- **[medium] Snapshot store path segment symlink escape:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: Snapshot upload writes under `targetDir/snapshots/<installationId>` without rejecting symlinked intermediate directories, so a tampered backup folder can redirect encrypted payload and envelope writes outside the configured target.
  - confidence: 0.88
- **[medium] OAuth credential restore can be skipped:** `src/commands/backup-restore.ts:741`
  - body: Full-host restore may skip the credentials operation when both source and target OAuth directories are inside state but at different relative paths, leaving active provider credentials absent after a reported successful restore.
  - confidence: 0.86
