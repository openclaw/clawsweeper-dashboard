## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:66-85`
  - body: `writeFileAtomic` and `copyFileAtomic` create parent directories and then rename/copy through `backup.target/snapshots/...` without checking that `snapshots` and the installation directory are real directories. A symlink planted inside a synced backup target can redirect snapshot payload or envelope writes outside the configured backup directory.
  - confidence: 0.92
- **[P1] Validate workspace backup roots before mkdir:** `src/commands/workspace-backup.ts:350-352`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before the later safety checks can reject a symlinked `backup.target/workspace`. `fs.mkdir` follows that symlink, so the command can create directories outside the backup target before validation runs.
  - confidence: 0.91
- **[P1] Check every existing restore path segment:** `src/commands/backup-restore.ts:330-337`
  - body: `ensureDirectoryPathWithoutSymlinks` checks the full existing probe path, but `lstat('/a/link/dir')` still follows `link` when it is an intermediate symlink. Walk from the filesystem root and `lstat` each existing segment before creating or renaming restore payloads.
  - confidence: 0.9
- **[P2] Restore in-state config path changes explicitly:** `src/commands/backup-restore.ts:715-716`
  - body: This condition treats any source config inside the old state dir and any target config inside the current state dir as covered by the state restore. If the relative path changed between hosts, the restored state will not populate the active `OPENCLAW_CONFIG_PATH`, leaving the recovered install with a stale or missing config.
  - confidence: 0.84
- **[P2] Restore in-state OAuth path changes explicitly:** `src/commands/backup-restore.ts:741-742`
  - body: The same covered-by-state shortcut is used for OAuth credentials, so a restore host whose in-state `OPENCLAW_OAUTH_DIR` uses a different relative path from the archived host can report success while leaving the active OAuth directory absent.
  - confidence: 0.82
- **[P2] Bound parent expansion during archive verification:** `src/commands/backup-verify.ts:337-354`
  - body: `collectNestedArchiveParents` materializes every parent of every normalized entry, so one deeply nested path within the 4 MiB metadata budget can still allocate a very large set and burn CPU before restore proceeds. Add a per-entry depth/length cap or avoid expanding all parents.
  - confidence: 0.78
