## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked snapshot store directories before writing:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `writeFileAtomic` and `copyFileAtomic` create the destination parent with recursive `mkdir` and then write/rename through it without validating path segments inside `backup.target`. If a synced/shared backup folder contains `snapshots` or an installation directory as a symlink, snapshot payloads and envelopes can be written outside the configured target.
  - confidence: 0.9
- **[P1] Validate backup mirror paths before creating them:** `src/commands/workspace-backup.ts:350-352`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before rejecting target-side symlinks, and the later realpath containment check follows the same symlink on both sides. A symlink planted under the backup target can redirect mirror writes, renames, or stale-mirror removals outside the backup directory.
  - confidence: 0.88
- **[P2] Recheck archive entry types during tar creation:** `src/infra/backup-create.ts:406-419`
  - body: The source tree is scanned for symlinks and special files before `tar.c`, but the pack step itself has no filter or entry-type check. A file can be swapped after the scan and before/during tar traversal, producing an archive that contains entries restore/verify later reject and that may be unsafe if extracted by other tools.
  - confidence: 0.82
- **[P2] Bound archive path depth before collecting parents:** `src/commands/backup-verify.ts:337-345`
  - body: The new archive metadata cap only limits total path bytes, but `collectNestedArchiveParents` still allocates one parent string per path segment. A crafted path with millions of short slash-separated segments can consume excessive CPU and memory during `backup verify` or restore verification.
  - confidence: 0.86
- **[P2] Check every restore parent segment for symlinks:** `src/commands/backup-restore.ts:337-343`
  - body: `ensureDirectoryPathWithoutSymlinks` calls `lstat` on the deepest existing path, which still follows symlinks in intermediate components. Restoring through a path like `safe/link/target` can write through `link` even though the helper's error text says symlink path segments are refused.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff adds sensitive filesystem backup/restore behavior and still has concrete symlink traversal and resource-exhaustion concerns before merge.

Concerns:

- **[high] Target-side symlink traversal in backup writes:** `src/commands/workspace-backup.ts:350`
  - body: Snapshot and workspace backup writes create directories under a user-controlled or synced backup target before rejecting symlinked path segments, allowing writes/renames/removals outside the intended backup target if that folder is modified by another local or sync actor.
  - confidence: 0.9
- **[medium] Restore path segment validation is incomplete:** `src/commands/backup-restore.ts:337`
  - body: The restore helper does not lstat every existing intermediate path segment, so configured or manifest-derived restore targets can traverse a symlinked parent and write outside the path the safety check intended to validate.
  - confidence: 0.79
- **[medium] Archive hardening leaves race and DoS paths:** `src/infra/backup-create.ts:406`
  - body: Backup creation only pre-scans sources before tar traversal, and verification still expands all parent paths for deep archive entries; both are attacker-influenced archive/backup surfaces that need bounded, at-use validation.
  - confidence: 0.84
