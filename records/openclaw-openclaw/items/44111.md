## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject symlinked snapshot-store directories before writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `writeFileAtomic` and `copyFileAtomic` create and write under `backup.target/snapshots/<installationId>` without rejecting symlinked intermediate directories. In a cloud-synced target, an attacker or sync race can make `snapshots` or the installation directory a symlink and redirect the encrypted payload/envelope write outside the configured backup target.
  - confidence: 0.93
- **[P1] Reject symlinked workspace target directories before writes:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` mkdirs the mirror and staging directories before checking that the target-side tree is not symlinked, and the later containment check canonicalizes through the same symlink. A malicious `backup.target/workspace` symlink can redirect mirror writes and stale mirror deletion outside `backup.target`.
  - confidence: 0.94
- **[P1] Validate every restore parent segment:** `src/commands/backup-restore.ts:337`
  - body: `ensureDirectoryPathWithoutSymlinks` only lstat-checks the current probe path, so an existing intermediate symlink such as `parent/link/dir` can appear as a normal final directory. Restore should walk from the filesystem root and lstat every segment before creating temp files or renaming into the target path.
  - confidence: 0.9
- **[P2] Remove or implement backup retention config:** `src/config/types.backup.ts:20`
  - body: The PR accepts `backup.retention.*` in the public config type/schema, but the diff has no snapshot pruning or retention application path. Users can set `maxSnapshots` or daily/weekly/monthly retention values that validate and are then ignored, creating a misleading config contract for backup storage.
  - confidence: 0.88
