Full review comments:

- **[P1] Reject target-side symlinks before snapshot writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `uploadSnapshot` writes under `targetDir/snapshots/...` after `fs.mkdir(path.dirname(...), { recursive: true })`, but the store never rejects symlinked directory segments inside the backup target. A symlink such as `snapshots` or the installation-id directory in a shared/synced target can redirect payload and envelope writes outside the configured backup folder.
  - confidence: 0.92
- **[P1] Reject target-side symlinks before mirroring workspaces:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging`, then later copies, renames, and removes paths through that tree. Because the containment checks canonicalize both the child and root through `realpath`, a symlinked `backup.target/workspace` can pass the guard and send mirror writes or stale deletions outside the backup target.
  - confidence: 0.91
- **[P1] Remove or enforce backup retention settings:** `src/config/zod-schema.backup.ts:25`
  - body: The PR accepts `backup.retention.keepDaily`, `keepWeekly`, `keepMonthly`, and `maxSnapshots`, but the diff only declares these fields and does not prune snapshots. That creates a public config promise where operators may believe sensitive encrypted snapshots are being rotated or deleted when no runtime enforces it.
  - confidence: 0.88
