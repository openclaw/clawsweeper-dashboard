## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject symlinked snapshot dirs before writes:** `src/backup/snapshot-store/provider-folder.ts:85`
  - body: `copyFileAtomic` creates the snapshot destination directory and then writes/renames through it without rejecting symlinked child directories under `backup.target`. In a cloud-synced or shared target, `snapshots` or the installation directory can be a symlink, causing encrypted payloads/envelopes to be written outside the selected backup folder.
  - confidence: 0.92
- **[P1] Reject symlinked workspace backup dirs before writes:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before any target-side symlink-segment check, and the later realpath containment check accepts a planted `backup.target/workspace` symlink. That lets mirror writes, renames, and stale cleanup operate outside the backup target.
  - confidence: 0.92
- **[P2] Restore in-state config paths that moved:** `src/commands/backup-restore.ts:715-721`
  - body: Full-host restore skips a config copy whenever the source config was covered by the state asset and the current config path is inside the current state dir. If the archive used `state/openclaw.json` but the restore host uses another in-state config path, restore reports success while the active config remains stale or missing.
  - confidence: 0.88
- **[P2] Restore in-state OAuth dirs that moved:** `src/commands/backup-restore.ts:741-747`
  - body: The OAuth branch has the same issue: any current OAuth dir inside `stateDir` is treated as covered by the state restore without checking that the source and target relative paths match. Restoring to a different in-state OAuth path can leave provider credentials absent after a successful full-host restore.
  - confidence: 0.88
- **[P2] Remove or implement backup retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention` keys, but the diff only adds type/schema references and no pruning behavior or docs. Shipping this creates a public config contract where users can believe snapshots are retained/pruned when the setting is ignored.
  - confidence: 0.9
