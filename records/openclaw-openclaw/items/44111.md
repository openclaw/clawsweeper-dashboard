## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject symlinked snapshot directories before writes:** `src/backup/snapshot-store/provider-folder.ts:85`
  - body: `copyFileAtomic` creates destination parents and writes under `backup.target/snapshots/<installationId>` without rejecting symlinked target-side path segments. In a shared or synced backup target, a planted `snapshots` or installation symlink can redirect payload/envelope writes outside `backup.target`; add a no-symlink directory walk or fd-relative writes before any mkdir/copy/rename.
  - confidence: 0.93
- **[P1] Reject symlinked workspace target dirs before mirroring:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before any target-side symlink rejection, and the later realpath containment check accepts a symlinked `workspace` root. A planted symlink can redirect mirror writes, renames, and stale cleanup outside the backup target, so the target directory segments need lstat/no-symlink validation before filesystem writes.
  - confidence: 0.92
- **[P2] Remove or implement retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention` and the exported type exposes `keepDaily`, `keepWeekly`, `keepMonthly`, and `maxSnapshots`, but the run/list/store paths do not read those keys or prune snapshots. That creates a public config contract that silently does nothing, so either implement tested retention behavior or remove the key before merge.
  - confidence: 0.88
