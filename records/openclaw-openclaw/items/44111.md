Full review comments:

- **[P1] Reject symlinked snapshot-store directories before writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `uploadSnapshot` writes under `backup.target/snapshots/<installationId>` after `mkdir(..., { recursive: true })`, but nothing rejects a symlinked `snapshots` or installation directory. In a synced or shared backup folder, that redirects payload/envelope writes outside the configured target, so every directory segment needs lstat/no-symlink validation before mkdir/rename/copy.
  - confidence: 0.91
- **[P1] Reject symlinked workspace target directories before writes:** `src/commands/workspace-backup.ts:350-351`
  - body: The workspace mirror path creates `backup.target/workspace/mirrors` and `.staging` before proving those target-side directories are real directories, and the later containment check canonicalizes through symlinks. A symlinked `backup.target/workspace` can make the mirror copy, rename, or stale cleanup operate outside the backup target.
  - confidence: 0.9
- **[P2] Remove or implement backup retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention` keys, but the new backup run/list/store paths never read them or prune snapshots. Users can set a valid config that silently does nothing, so either remove the public retention surface or implement and test snapshot pruning before merge.
  - confidence: 0.88
- **[P2] Validate archive entry types during tar creation:** `src/infra/backup-create.ts:412`
  - body: `createBackupArchive` pre-scans sources for symlinks/special files, then calls `tar.c` without a filter or `onWriteEntry` type guard. A source tree mutation between the scan and tar walk can still produce an archive containing blocked entry types that verify/restore later reject, so the archive writer should enforce the same policy at write time.
  - confidence: 0.8
