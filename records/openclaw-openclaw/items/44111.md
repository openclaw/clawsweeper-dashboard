## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:66-85`
  - body: `writeFileAtomic` and `copyFileAtomic` create parent directories and then write/rename through `backup.target/snapshots/...` without checking that `snapshots` or the installation directory is a real directory. A symlink planted inside a synced backup target can redirect snapshot payload or envelope writes outside the configured backup directory.
  - confidence: 0.91
- **[P1] Validate workspace backup roots before mkdir:** `src/commands/workspace-backup.ts:350-352`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before the later safety checks run. If `backup.target/workspace` is a symlink, those `mkdir` calls can create directories outside the backup target before validation has a chance to reject the path.
  - confidence: 0.9
- **[P1] Check every existing restore path segment:** `src/commands/backup-restore.ts:337-343`
  - body: `ensureDirectoryPathWithoutSymlinks` lstat-checks the first existing probe path, but path resolution can still pass through symlinked earlier segments. Restore should walk each existing ancestor segment from the root so a symlinked parent cannot redirect copy/rename operations outside the intended target tree.
  - confidence: 0.86
- **[P2] Restore config when in-state target path differs:** `src/commands/backup-restore.ts:715-716`
  - body: Full-host restore suppresses a standalone config operation whenever the source config was covered by state and the current config path is anywhere under `stateDir`. If the source and target use different in-state relative config paths, the restored state tree will not populate the active config path even though the command reports success.
  - confidence: 0.82
- **[P2] Restore OAuth data when in-state target path differs:** `src/commands/backup-restore.ts:741-742`
  - body: The credentials path uses the same broad covered-by-state check, so a restore host with an in-state OAuth directory at a different relative path can finish without copying the restored credentials to the active OAuth directory. That leaves provider login state missing after a full-host restore.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: The diff adds security-sensitive backup write/restore surfaces and still has concrete path traversal/symlink concerns on PR head.

Concerns:

- **[high] Snapshot writes can follow target-side symlinks:** `src/backup/snapshot-store/provider-folder.ts:66`
  - body: Writes under the folder snapshot store create parent directories and rename/copy files without rejecting symlinked directory segments inside `backup.target`, which can redirect encrypted snapshot files outside the configured target.
  - confidence: 0.91
- **[high] Workspace backup can mkdir through a symlinked root:** `src/commands/workspace-backup.ts:350`
  - body: The workspace mirror command creates target-side roots before the second safety check, so an attacker-controlled synced backup folder can redirect those writes through a symlinked `workspace` directory.
  - confidence: 0.9
- **[high] Restore does not verify every path segment:** `src/commands/backup-restore.ts:337`
  - body: The restore helper checks only the first existing probe path with `lstat`, which misses symlinked intermediate ancestors and can let restore publication operate outside the intended path.
  - confidence: 0.86
