## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject target-side symlinks before snapshot writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `uploadSnapshot` reaches `writeFileAtomic`/`copyFileAtomic`, and both create destination parents under `backup.target` before rejecting symlinked path segments. A shared or synced backup folder can contain a `snapshots` or installation-dir symlink, redirecting payload/envelope writes outside the configured target.
  - confidence: 0.91
- **[P1] Reject target-side symlinks before mirroring workspaces:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `mirrorRoot` and `.staging` before any target-side no-symlink validation, while the later containment checks canonicalize through `realpath`. If `backup.target/workspace` or a child is a symlink, the mirror, rename, and stale cleanup paths can write or delete outside the backup target.
  - confidence: 0.9
- **[P2] Remove or implement backup.retention:** `src/config/types.backup.ts:20`
  - body: The PR exports and accepts `backup.retention` in the config schema, but the run/list/status implementation and docs never apply pruning or report retention. Shipping an accepted no-op retention policy gives operators a false data-retention guarantee for backup snapshots.
  - confidence: 0.88
- **[P2] Bound archive path depth before collecting parents:** `src/commands/backup-verify.ts:337-346`
  - body: `collectNestedArchiveParents` materializes every parent directory for every archive entry. The aggregate path-byte cap still allows a single path with many short slash-separated segments to allocate a very large parent set during `backup verify` or restore verification.
  - confidence: 0.86
- **[P1] Validate every existing restore parent segment:** `src/commands/backup-restore.ts:337-339`
  - body: `ensureDirectoryPathWithoutSymlinks` stops after `lstat` succeeds on the deepest existing probe, but `lstat('/a/link/b')` has already followed `link` when the final component is not the symlink. Restore can therefore create temp paths and rename through an intermediate symlink unless each existing parent segment is walked and checked.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: Needs attention: the PR adds backup/restore writes over sensitive local state and still permits symlink traversal through backup or restore target paths.

Concerns:

- **[medium] Snapshot store can write through target symlinks:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: Destination parent creation and atomic rename under `backup.target/snapshots/<installationId>` do not reject symlinked intermediate directories, so a shared backup target can redirect snapshot payload/envelope writes.
  - confidence: 0.91
- **[high] Workspace mirroring can escape backup.target:** `src/commands/workspace-backup.ts:350`
  - body: Workspace mirror and staging directories are created before target-side symlink checks, and later realpath containment can accept a symlinked target root, allowing writes, renames, or stale cleanup outside the configured target.
  - confidence: 0.9
- **[high] Restore parent validation misses intermediate symlinks:** `src/commands/backup-restore.ts:337`
  - body: Restore validates the deepest existing parent with lstat but does not walk each existing segment, so an intermediate symlink can still redirect temp restore writes and final renames.
  - confidence: 0.89
- **[medium] Deep archive paths can exhaust verification resources:** `src/commands/backup-verify.ts:337`
  - body: Backup verification materializes every nested parent without a per-entry depth cap, leaving crafted archives able to consume excessive CPU/memory during verify or restore verification.
  - confidence: 0.86
