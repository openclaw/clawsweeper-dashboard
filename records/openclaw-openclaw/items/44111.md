## Summary

Keep open. Current main and the latest release still expose local backup create/verify only, while this PR adds a new encrypted snapshot/restore flow that remains security-sensitive, has concrete path-safety review findings, and lacks successful real behavior proof.

## What This Changes

The PR adds folder-backed encrypted snapshot backups, workspace mirroring, `backup setup/run/status/list/restore`, backup config schema/docs, aliases, and restore/export hardening tests.

## Best Possible Solution

Keep the native encrypted snapshot direction under review, but merge only after path-safety fixes, restore-mapping fixes, bounded archive verification, explicit maintainer/security approval, and redacted real CLI proof.

## Reproduction Assessment

Yes for the PR review findings: source inspection at PR head shows unsafe target writes, restore skip assumptions, archive-time validation gaps, and verification expansion risk. For current-main behavior, reproduction is not applicable because the snapshot flow is a new feature rather than an existing main bug.

## Solution Assessment

No. The encrypted backup direction is plausible, but this branch is not the best mergeable implementation until the concrete safety findings, rebase/validation gap, and real behavior proof gate are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:66-85`
  - body: `writeFileAtomic` and `copyFileAtomic` create parent directories and then rename/copy through `backup.target/snapshots/...` without checking that `snapshots` or the installation directory is a real directory. A symlink planted inside a synced backup target can redirect snapshot payload or envelope writes outside the configured backup directory.
  - confidence: 0.89
- **[P1] Validate workspace backup roots before mkdir:** `src/commands/workspace-backup.ts:350-352`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before the later safety checks can reject a symlinked `backup.target/workspace`. `fs.mkdir` follows that symlink, so the command can create directories outside the backup target before validation completes.
  - confidence: 0.87
- **[P1] Check every existing restore path segment:** `src/commands/backup-restore.ts:330-337`
  - body: `ensureDirectoryPathWithoutSymlinks` only `lstat`s the full existing probe path, so symlinks in intermediate components can be followed before restore writes. Walk from the filesystem root and `lstat` each segment before creating or renaming through it.
  - confidence: 0.82
- **[P2] Restore configs when in-state paths differ:** `src/commands/backup-restore.ts:714-716`
  - body: Full-host restore skips the config operation whenever the target config path is inside the current state dir and the source config was covered by the old state dir. If the archive had `state/openclaw.json` but the restore host uses another in-state config path, the active config path is left missing or stale after a reported restore.
  - confidence: 0.84
- **[P2] Restore OAuth dirs when in-state paths differ:** `src/commands/backup-restore.ts:740-742`
  - body: The credentials branch makes the same covered-by-state assumption without comparing source and target relative paths. Restoring from one in-state OAuth location onto a host configured for another in-state OAuth path leaves the active credentials directory missing or stale.
  - confidence: 0.83
- **[P2] Validate tar entries during archive creation:** `src/infra/backup-create.ts:406-412`
  - body: The export path scans source trees before calling `tar.c`, but the actual tar write has no entry-type check. A source tree can be swapped after the pre-scan so `backup export` emits symlink or special-file entries that restore later rejects, or that are unsafe for external extractors.
  - confidence: 0.77
- **[P2] Bound parent expansion during verification:** `src/commands/backup-verify.ts:337-354`
  - body: `collectNestedArchiveParents` inserts every parent path for every archive entry, but the verifier only caps raw entry path bytes, not the number or total size of generated parent strings. A crafted deeply nested tar path can make `backup verify` and restore preflight spend excessive CPU and memory before failing.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff adds broad backup write/restore surfaces and still has concrete symlink traversal, archive-race, and verification DoS concerns.

Concerns:

- **[high] Snapshot store follows symlinked target segments:** `src/backup/snapshot-store/provider-folder.ts:66`
  - body: Snapshot uploads create parent directories and then copy/rename files under `targetDir/snapshots/...` without validating intermediate path segments, so a symlink in a synced backup target can redirect writes outside the configured backup directory.
  - confidence: 0.89
- **[high] Workspace mirror roots are created before validation:** `src/commands/workspace-backup.ts:350`
  - body: The workspace backup flow calls `fs.mkdir` for mirror and staging roots before rejecting a symlinked workspace backup root, which can create or later mutate directories outside the backup target.
  - confidence: 0.87
- **[medium] Restore can write through intermediate symlinks:** `src/commands/backup-restore.ts:330`
  - body: The restore helper validates only the existing probe path, not each existing component, so restore publication can follow an intermediate symlink before copying or renaming target data.
  - confidence: 0.82
- **[medium] Archive creation has a TOCTOU entry-type gap:** `src/infra/backup-create.ts:406`
  - body: The pre-scan rejects symlinks and special files, but `tar.c` does not re-check entry type while packing, so a source tree can be swapped after validation and produce an unsafe or unrecoverable archive.
  - confidence: 0.77
- **[medium] Archive verification can expand parent paths excessively:** `src/commands/backup-verify.ts:337`
  - body: `collectNestedArchiveParents` can materialize many large parent strings from a crafted deep tar path, causing CPU and memory pressure before restore preflight fails.
  - confidence: 0.74
