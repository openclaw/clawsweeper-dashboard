## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject symlinked snapshot directories before writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `uploadSnapshot` writes payloads/envelopes under `backup.target/snapshots/<installationId>` through `mkdir(..., { recursive: true })` and `rename` without rejecting symlinked directory segments. A writable or synced backup target can redirect snapshot writes outside the configured backup directory, so validate/create each target directory segment without following symlinks before writing.
  - confidence: 0.92
- **[P1] Validate workspace target directories before mkdir:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before the later safety checks. If `backup.target/workspace` is a symlink, these writes and later mirror cleanup can operate outside the backup target before containment is asserted.
  - confidence: 0.9
- **[P1] Remove or implement the retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention`, but the PR-head runtime never reads or applies those settings. That creates a public config contract where users can set retention values and still accumulate snapshots indefinitely, so either implement retention with tests or drop the config surface until it is real.
  - confidence: 0.89
- **[P2] Recheck archive entry types while packing:** `src/infra/backup-create.ts:412-418`
  - body: `createBackupArchive` pre-scans sources, but the `tar.c` write path only remaps entries and does not re-check symlink/special-file types at the moment entries are packed. A source tree changed after the pre-scan can produce an archive that restore later rejects or that is unsafe for external extractors; keep current main's archive-time hardening when rebasing.
  - confidence: 0.83
