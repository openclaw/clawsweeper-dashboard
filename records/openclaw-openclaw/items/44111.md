Full review comments:

- **[P1] Reject target-side symlinks before snapshot writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `uploadSnapshot` writes under `backup.target/snapshots/...` after recursive mkdir without validating each target-side directory segment. A symlink planted inside a synced/shared backup target can redirect the payload or envelope write outside the configured backup root, so the store needs no-symlink directory traversal before copy/rename.
  - confidence: 0.93
- **[P1] Reject target-side symlinks before mirroring workspaces:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before rejecting symlinked target-side segments, then containment checks use `realpath` and follow the same planted symlink. A malicious cloud-folder entry can redirect mirror writes, renames, or stale mirror deletion outside the backup directory.
  - confidence: 0.93
- **[P1] Remove or implement backup.retention:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention` keys, but the branch never prunes snapshots or documents that these values are inert. For a backup feature, accepting retention config without behavior can leave sensitive snapshots retained indefinitely despite operator intent.
  - confidence: 0.88
- **[P2] Cap archive path depth before collecting parents:** `src/commands/backup-verify.ts:337-354`
  - body: `collectNestedArchiveParents` creates every parent string for every normalized tar entry, while scanning only caps total path bytes. A crafted deep path can still force excessive CPU and memory during verify/restore; add a per-entry depth or path-length cap before this parent expansion.
  - confidence: 0.83
