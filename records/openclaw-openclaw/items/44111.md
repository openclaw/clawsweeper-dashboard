Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:66-85`
  - body: `writeFileAtomic` and `copyFileAtomic` create parent directories and then rename/copy through `backup.target/snapshots/...` without checking that `snapshots` or the installation directory is a real directory. A symlink planted inside a synced backup target can redirect snapshot payload or envelope writes outside the configured backup directory.
  - confidence: 0.9
- **[P1] Validate workspace backup roots before mkdir:** `src/commands/workspace-backup.ts:350-352`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before the later safety checks can reject a symlinked `backup.target/workspace`. `fs.mkdir` follows that symlink, so the command can create directories outside the backup target before validation completes.
  - confidence: 0.88
- **[P1] Check every existing restore path segment:** `src/commands/backup-restore.ts:330-337`
  - body: `ensureDirectoryPathWithoutSymlinks` only `lstat`s the first existing probe path, so a path reached through a symlinked intermediate directory can pass before restore writes. Walk from the filesystem root, or otherwise validate each existing segment, before creating or renaming through it.
  - confidence: 0.84
- **[P2] Revalidate tar entry types while creating archives:** `src/infra/backup-create.ts:406-420`
  - body: `createBackupArchive` scans sources before `tar.c`, but the packing step has no `filter` or `onWriteEntry` type guard. A writable source tree can be swapped after the pre-scan so backup creates an archive containing symlinks or special files that later fail verify/restore or are unsafe with external extractors.
  - confidence: 0.8
- **[P2] Bound parent expansion during archive verification:** `src/commands/backup-verify.ts:337-347`
  - body: `collectNestedArchiveParents` expands every parent for every normalized entry after only bounding total path bytes. A crafted path with many short segments can still create a huge parent set and burn CPU/memory during `backup verify` or restore preflight.
  - confidence: 0.78
- **[P2] Restore configs when in-state paths differ:** `src/commands/backup-restore.ts:714-716`
  - body: Full-host restore skips the config operation whenever the target config path is inside the current state dir and the source config was covered by the old state dir. If the archive had the config at one in-state relative path but the restore host uses another, the active config path is left missing or stale after a reported restore.
  - confidence: 0.84
- **[P2] Restore OAuth dirs when in-state paths differ:** `src/commands/backup-restore.ts:740-742`
  - body: The credentials branch makes the same covered-by-state assumption without comparing source and target relative paths. When the archive's OAuth dir and the restore host's in-state OAuth dir differ, restore can skip credentials and leave provider auth missing despite success.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff adds security-sensitive backup write/restore surfaces and still has concrete symlink traversal, archive race, and resource-exhaustion concerns.

Concerns:

- **[high] Snapshot writes can follow target-side symlinks:** `src/backup/snapshot-store/provider-folder.ts:66`
  - body: Snapshot payload/envelope writes create and rename through `backup.target/snapshots/...` without validating intermediate directories, allowing writes outside the configured backup target if a synced/shared target is modified.
  - confidence: 0.9
- **[high] Workspace mirror setup can write through a symlinked root:** `src/commands/workspace-backup.ts:350`
  - body: The workspace mirror and staging directories are created before the target-side root is revalidated, so a symlinked `backup.target/workspace` can be followed by mkdir/copy/rename operations.
  - confidence: 0.88
- **[high] Restore path validation misses intermediate symlinks:** `src/commands/backup-restore.ts:330`
  - body: Restore parent validation can accept an existing path reached through a symlinked intermediate segment, which leaves restore writes able to traverse outside the intended parent.
  - confidence: 0.84
- **[medium] Archive creation has a TOCTOU gap:** `src/infra/backup-create.ts:406`
  - body: The tar creation step does not re-check entry types as files are packed, so a mutable source tree can race the pre-scan and produce archives with blocked special entries.
  - confidence: 0.8
- **[medium] Verification can over-expand parent paths:** `src/commands/backup-verify.ts:337`
  - body: Parent expansion in archive verification can allocate a very large set from deeply segmented entry paths, creating a denial-of-service path on untrusted or synced backup archives.
  - confidence: 0.78
