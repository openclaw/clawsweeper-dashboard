## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked workspace backup roots before writes:** `src/commands/workspace-backup.ts:350-351`
  - body: `fs.mkdir` follows existing directory symlinks, so if a synced or shared backup target already contains `workspace`, `mirrors`, or `.staging` as a symlink, this code can create, rename, or delete paths outside `backup.target` before the later realpath containment checks run. Validate every target-side segment with `lstat` before creating or mutating these directories.
  - confidence: 0.93
- **[P1] Validate snapshot store directory segments before writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: Both payload and envelope writes create `targetDir/snapshots/<installationId>` with recursive mkdir and then rename through that path, but they never reject symlinked `snapshots` or installation directories. A modified backup folder can redirect encrypted snapshot writes outside the configured target.
  - confidence: 0.92
- **[P1] Walk existing restore parent segments:** `src/commands/backup-restore.ts:337-344`
  - body: This helper lstat's only the final existing probe path before stopping, so a path like `parent/link/dir` can pass when `link` is a symlink and `dir` exists in the linked target. Restore then writes through that symlinked parent despite the intended path-segment guard.
  - confidence: 0.9
- **[P2] Check archive entry types during tar packing:** `src/infra/backup-create.ts:406-420`
  - body: The pre-scan can be invalidated while `tar.c` re-walks the source tree, and the actual packing step has no entry-time symlink or special-file guard. That can produce archives that later verification or restore rejects, or unsafe artifacts if users extract them outside OpenClaw.
  - confidence: 0.82
