## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Reject snapshot target symlink segments:** `src/backup/snapshot-store/provider-folder.ts:67-85`
  - body: Snapshot uploads create and write under `targetDir/snapshots/<installationId>` without rejecting symlinked target-side directory segments. In a synced/shared backup folder, `snapshots` or the installation directory can be replaced with a symlink, causing encrypted payload/envelope writes to land outside the configured backup target.
  - confidence: 0.89
- **[P1] Reject workspace target symlinks before writes:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before any target-side symlink rejection, and the later containment check canonicalizes both child and root through the same symlink. A malicious synced-folder entry can redirect mirror writes and stale-mirror deletes outside the backup target.
  - confidence: 0.9
- **[P1] Validate restore path segments from the root:** `src/commands/backup-restore.ts:337-344`
  - body: `ensureDirectoryPathWithoutSymlinks` stops after `lstat` accepts the deepest existing path, but `lstat('/a/link/dir')` follows a symlinked intermediate component. Restore can then write through that symlink despite intending to reject symlink path segments.
  - confidence: 0.84
- **[P2] Remove or enforce backup retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention`, but the PR never reads those fields when creating, listing, or pruning snapshots. Shipping a no-op retention policy is misleading for sensitive encrypted backups because users may believe old snapshots are being deleted.
  - confidence: 0.86
- **[P2] Recheck tar entry types while writing archives:** `src/infra/backup-create.ts:412-418`
  - body: `createBackupArchive` pre-scans sources for symlinks/special files, then hands the same paths to `tar.c` without entry-time type rejection. A source tree change after the pre-scan can still create a symlink or other blocked tar entry that backup verify/restore later rejects.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive backup/restore writes and still has concrete symlink traversal, archive race, and misleading retention risks.

Concerns:

- **[high] Snapshot target symlink traversal:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: Folder snapshot writes do not reject symlinked directories below the configured backup target, allowing writes outside the intended backup directory if the target is attacker-modifiable.
  - confidence: 0.89
- **[high] Workspace mirror symlink traversal:** `src/commands/workspace-backup.ts:350`
  - body: Workspace mirroring creates and deletes directories through backup-target subpaths before target-side symlink rejection, so a synced/shared folder can redirect writes or removals outside the backup root.
  - confidence: 0.9
- **[high] Restore parent path traversal:** `src/commands/backup-restore.ts:337`
  - body: Restore path preparation checks the deepest existing path but not every existing intermediate segment, allowing a symlinked parent segment to redirect destructive restore writes.
  - confidence: 0.84
- **[medium] Archive write-time type race:** `src/infra/backup-create.ts:412`
  - body: Backup archive creation rejects symlinks before tar starts, but the tar write path lacks an entry-time blocked-type guard, so a source swap can still produce unsafe or unrestorable entries.
  - confidence: 0.8
- **[low] No-op retention for sensitive snapshots:** `src/config/zod-schema.backup.ts:25`
  - body: Accepting retention config without enforcement can leave old encrypted snapshots in place despite an operator believing pruning is configured.
