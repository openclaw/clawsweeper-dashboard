## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Reject symlinked snapshot store directories before writes:** `src/backup/snapshot-store/provider-folder.ts:85`
  - body: `uploadSnapshot` writes under `backup.target/snapshots/<installationId>` by creating the parent directory and then renaming into it, but it never checks whether `snapshots` or the installation directory is a symlink. In a synced backup target, a planted symlink can redirect the encrypted payload/envelope write outside `backup.target`; validate each target-side directory segment with `lstat` before mkdir/rename.
  - confidence: 0.95
- **[P1] Reject symlinked workspace target dirs before mirroring:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before proving those target-side segments are real directories, and the later containment check canonicalizes through `realpath`. If `backup.target/workspace` is a symlink in a cloud-synced folder, mirror writes and stale cleanup can operate at the symlink target.
  - confidence: 0.94
- **[P2] Restore in-state config when the active path differs:** `src/commands/backup-restore.ts:715-716`
  - body: This branch skips the standalone config restore whenever the source config was under source state and the current config path is under current state. If the backup was made with one in-state config path but the restore host uses a different in-state `OPENCLAW_CONFIG_PATH`, the state restore places the old config at its old relative path and leaves the active config path stale or missing.
  - confidence: 0.84
- **[P2] Restore in-state OAuth dirs when the active path differs:** `src/commands/backup-restore.ts:741-742`
  - body: The credentials branch makes the same covered-by-state assumption for any OAuth dir under state. Restoring a backup whose OAuth dir was at one in-state relative path onto a host configured for another in-state OAuth path can report success while leaving provider credentials out of the active OAuth directory.
  - confidence: 0.83
- **[P2] Remove or implement backup retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention.keepDaily`, `keepWeekly`, `keepMonthly`, and `maxSnapshots`, but the PR does not implement snapshot pruning or document these keys as no-ops. Accepting public config that is ignored creates a misleading source of truth for backup retention.
  - confidence: 0.86
