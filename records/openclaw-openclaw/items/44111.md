## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject symlinked workspace backup roots before writing:** `src/commands/workspace-backup.ts:350-352`
  - body: `backup run` creates `backup.target/workspace/mirrors` and `.staging` before proving those target-side directories are not symlinks. Because the later containment check canonicalizes through `realpath`, a synced-folder attacker can replace `backup.target/workspace` with a symlink and redirect mirror writes, renames, or stale cleanup outside the configured backup target.
  - confidence: 0.93
- **[P1] Validate snapshot store directory segments before writes:** `src/backup/snapshot-store/provider-folder.ts:67-85`
  - body: `writeFileAtomic` and `copyFileAtomic` recursively create the destination parent and then rename into it without checking intermediate segments like `targetDir/snapshots` or the installation directory. A symlink planted inside the configured backup target can redirect encrypted payload or envelope writes outside that target.
  - confidence: 0.92
- **[P1] Check every restore parent segment for symlinks:** `src/commands/backup-restore.ts:337-344`
  - body: `ensureDirectoryPathWithoutSymlinks` only `lstat`s the full existing probe path before stopping, so it misses symlinks in existing intermediate parents. Restore then creates temp files and renames through that parent path, which can write restored state or config outside the intended target.
  - confidence: 0.9
- **[P2] Bound archive path depth during verification:** `src/commands/backup-verify.ts:337-344`
  - body: Verification caps total path bytes, but `collectNestedArchiveParents` still materializes one parent string per slash-separated segment. A crafted archive path within the byte budget can force excessive CPU and memory use before restore finishes verification.
  - confidence: 0.82
