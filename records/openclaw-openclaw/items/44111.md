## Summary

Keep open: current main still only exposes local backup archive create/verify, while this PR adds a broader encrypted snapshot and restore flow. The branch is not mergeable as-is because the PR head still has source-proven safety defects in backup target writes, restore mapping, and archive creation, and the external contributor has not shown real after-fix CLI behavior proof.

## What This Changes

The PR adds folder-backed encrypted snapshot backup storage, workspace mirroring, backup setup/run/status/list/restore CLI plumbing, backup config schema/docs, aliases, and reset/uninstall backup guidance changes.

## Best Possible Solution

Land a repaired native backup flow only after target symlink guards, source-vs-target restore mapping, tar entry validation, and redacted real CLI smoke proof are in place; keep off-machine backup providers such as the linked keepmyclaw/ClawHub path complementary rather than bundled into this core PR.

## Reproduction Assessment

Yes for the review findings: the unsafe write, restore mapping, and archive race paths are visible in the PR head source at dd2c48fe37263cf406593b06e420bf372a5215a0. I did not execute the branch because this is a read-only review, and the PR body still says the end-to-end CLI smoke was not successfully verified.

## Solution Assessment

No. The feature direction is reasonable, but this is not the best mergeable implementation until it rejects symlinked target segments before writes, compares source and target restore paths, validates tar entries during archive creation, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:67-85`
  - body: `writeFileAtomic` and `copyFileAtomic` create parent directories and then rename/copy through `backup.target/snapshots/...` without checking that `snapshots` or the installation directory is a real directory. A symlink planted inside a synced backup target can redirect snapshot payload/envelope writes outside the configured backup directory.
  - confidence: 0.88
- **[P1] Validate workspace backup roots before mkdir:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before the later safety checks can reject a symlinked `backup.target/workspace`. `fs.mkdir` follows that symlink, so the command can create or update directories outside the backup target before validation completes.
  - confidence: 0.86
- **[P2] Restore configs when in-state paths differ:** `src/commands/backup-restore.ts:714-716`
  - body: Full-host restore skips the config operation whenever the target config path is inside the current state dir and the source config was covered by the old state dir. If the archive had `state/openclaw.json` but the restore host uses `state/custom.json`, the active config path is left missing or stale after a reported full-host restore.
  - confidence: 0.84
- **[P2] Restore OAuth dirs when in-state paths differ:** `src/commands/backup-restore.ts:740-742`
  - body: The credentials branch makes the same covered-by-state assumption without comparing source and target relative paths. Restoring from one in-state OAuth location onto a host configured for another in-state OAuth path leaves the active credentials directory missing or stale.
  - confidence: 0.82
- **[P2] Validate tar entries while archiving:** `src/infra/backup-create.ts:406-420`
  - body: The export path scans source trees before calling `tar.c`, but the actual tar write has no filter or entry-type check. A source tree can be swapped after the pre-scan so `backup export` emits symlink or special-file entries that restore later rejects, or that are unsafe for external extractors.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The diff adds backup write/restore surfaces and still has concrete symlink traversal and archive-race concerns.

Concerns:

- **[high] Snapshot store follows symlinked target segments:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: Snapshot uploads create parent directories and then copy/rename files under `targetDir/snapshots/...` without validating intermediate path segments, so a symlink planted inside a synced backup target can redirect writes outside that target.
  - confidence: 0.88
- **[high] Workspace mirror writes before target symlink rejection:** `src/commands/workspace-backup.ts:350`
  - body: The workspace mirror and staging paths are created before the safety check revalidates the backup workspace root, allowing a symlinked `backup.target/workspace` to be followed before validation completes.
  - confidence: 0.86
- **[medium] Archive creation relies on a pre-scan before tar writes:** `src/infra/backup-create.ts:406`
  - body: The export path scans sources for symlinks/special files before invoking `tar.c`, but the tar write itself lacks an entry-type guard, leaving a race where a source tree change can produce unsafe or unrestorable archive entries.
  - confidence: 0.76
