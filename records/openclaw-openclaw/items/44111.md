Full review comments:

- **[P1] Reject symlinked snapshot-store directories before writing:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `writeFileAtomic` and `copyFileAtomic` create the destination parent and then write/rename through it. If a synced or shared backup target contains `snapshots` or the installation root as a symlink, encrypted payload and envelope writes can escape `backup.target`; walk and lstat each existing directory segment, and create missing segments only after verifying the parent is not a symlink.
  - confidence: 0.93
- **[P1] Validate workspace backup target directories before mkdir:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before effective target-side symlink rejection, and the later containment helper follows realpaths. A malicious cloud-sync target can redirect mirror writes or stale-mirror deletion outside the backup directory; reject symlinked target-side segments before any mkdir, cp, rename, rm, or status write.
  - confidence: 0.92
- **[P2] Remove or implement backup retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention.*`, but the branch does not use those values to prune snapshots. Users can configure retention and still accumulate every snapshot, so either implement and test retention or keep the public config surface out of this PR.
  - confidence: 0.9
- **[P2] Recheck archive entry types during tar creation:** `src/infra/backup-create.ts:365`
  - body: The added source-tree pre-scan runs before `tar.c`, but a writable source can change after the scan and before packing. Recheck each entry at archive time, or otherwise configure tar so symlinks and special files cannot be written into archives that restore later rejects.
  - confidence: 0.84
