## Summary

Keep this PR open, but it is not merge-ready: the branch still lets target-side symlinked directories redirect workspace mirrors and encrypted snapshot writes outside `backup.target`, and it exposes unused retention config. The backup flow is useful and related comments already pushed the design in the right direction, but the security boundary and upgrade/config contract need fixes plus fresh real CLI proof after rebase.

## What This Changes

Adds encrypted folder-backed backup snapshots, workspace mirroring, `backup setup/run/status/list/restore`, backup config schema/types, updated docs, and backup/reset/uninstall CLI wiring.

## Best Possible Solution

Keep the native backup flow as the candidate direction, but require target-side symlink segment hardening, removal or implementation of retention config, a clean rebase, and redacted terminal proof for backup run/list/restore before merge.

A safer alternative is to share the restore path's non-symlink directory-segment creation helper with snapshot and workspace backup writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Reject symlinked target dirs before mirroring:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before proving the target-side path is made only of real directories, and the later containment check canonicalizes through symlinks. If a synced/shared backup folder contains `workspace -> <outside dir>`, these mkdir/copy/rename/rm operations can write or delete outside the configured backup target; reject symlinked directory segments before any write and recheck after creation.
  - confidence: 0.91
- **[P1] Reject symlinked snapshot store dirs before writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: The folder snapshot store writes under `targetDir/snapshots/<installationId>` after `fs.mkdir(..., { recursive: true })`, but it never lstat-walks existing directory segments inside `backup.target`. A cloud-synced or attacker-writable target can replace `snapshots` or an installation directory with a symlink and redirect encrypted payload/envelope writes outside the backup target; use a no-symlink directory creation helper before both `writeFileAtomic` and `copyFileAtomic`.
  - confidence: 0.9
- **[P2] Remove or implement retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention.keepDaily/keepWeekly/keepMonthly/maxSnapshots`, but the diff only defines those keys in types/schema and never reads them at runtime. That makes a public config contract that silently does nothing and will be hard to change later; drop the retention keys until pruning exists, or implement/document retention behavior with tests in this PR.
  - confidence: 0.84
