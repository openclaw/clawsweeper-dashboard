## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked backup targets before mirroring:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before proving the target-side path segments are real directories. If a synced target contains `workspace -> $HOME`, the later realpath-based containment checks canonicalize through that same symlink and pass, so `cp`/`rename`/stale `rm` can operate outside the backup target.
  - confidence: 0.91
- **[P1] Validate snapshot store directories before writing:** `src/backup/snapshot-store/provider-folder.ts:67-81`
  - body: `writeFileAtomic` and `copyFileAtomic` recursively create and rename under `targetDir/snapshots/...` without checking existing intermediate segments like `snapshots` or the installation id directory. A shared backup target can plant a symlink there and redirect encrypted payload/envelope writes outside the configured target.
  - confidence: 0.9
- **[P1] Validate restore parents before rollback moves:** `src/commands/backup-restore.ts:442-443`
  - body: `applyRestoreOperations` moves an existing target into rollback before `copySourceToTarget` validates the target parent with `ensureDirectoryPathWithoutSymlinks`. When the parent path contains a symlink segment, restore can move user files through that symlink before the safety check runs; validate the parent before access, rollback dir creation, or rename.
  - confidence: 0.86
- **[P2] Cap archive path depth during verification:** `src/commands/backup-verify.ts:337-346`
  - body: `collectNestedArchiveParents` expands every parent directory for each normalized entry, but scanning only caps total path bytes. A crafted tar path with many short segments inside the 4 MiB budget can still force excessive CPU and memory during `backup verify` and restore; add a segment/depth cap or avoid materializing all parents.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds security-sensitive backup write and restore surfaces and still has concrete symlink traversal and archive verification DoS concerns.

Concerns:

- **[high] Workspace mirror target symlink traversal:** `src/commands/workspace-backup.ts:350`
  - body: Target-side workspace backup directories are created and used before no-symlink segment validation, so a writable/synced backup target can redirect mirror writes outside `backup.target`.
  - confidence: 0.91
- **[high] Snapshot payload/envelope write redirection:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: Folder snapshot writes use recursive `mkdir` and `rename` under attacker-modifiable target directories without rejecting symlinked intermediate segments.
  - confidence: 0.9
- **[medium] Restore rollback before parent validation:** `src/commands/backup-restore.ts:443`
  - body: Restore moves existing targets to rollback before validating target parent segments, so symlinked parents can be acted on before the safety guard runs.
  - confidence: 0.86
- **[medium] Archive verification path-depth DoS:** `src/commands/backup-verify.ts:337`
  - body: Verification builds a full parent set for every archive entry without a segment-depth cap, allowing crafted deep paths to burn CPU/memory during verify/restore.
  - confidence: 0.82
