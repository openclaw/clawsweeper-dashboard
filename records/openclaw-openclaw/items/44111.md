Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:67-85`
  - body: `writeFileAtomic` and `copyFileAtomic` create the destination parent under `backup.target` without checking existing directory segments. If a synced/shared backup folder contains `snapshots` or an installation directory symlink, `backup run` can write payload and envelope files outside the configured backup target.
  - confidence: 0.91
- **[P1] Validate workspace backup roots before mkdir:** `src/commands/workspace-backup.ts:350-352`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before rejecting target-side symlinks, and the later containment check canonicalizes through symlinks. A symlinked `backup.target/workspace` can redirect mkdir/copy/remove operations into another directory before the guard runs.
  - confidence: 0.9
- **[P2] Restore in-state config when relative paths differ:** `src/commands/backup-restore.ts:715-716`
  - body: Full-host restore skips a config operation whenever the archived config was covered by the source state and the current config path is inside the current state. If the active config moved to a different relative path, restoring the state tree will not populate the active config file even though the command can report success.
  - confidence: 0.86
- **[P2] Restore in-state OAuth when relative paths differ:** `src/commands/backup-restore.ts:741-742`
  - body: The OAuth restore path has the same coverage shortcut: it skips the credentials operation whenever both source and target are inside their state dirs. A restore host with a different in-state OAuth relative path can come back without the active credentials directory populated.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds security-sensitive backup write/restore surfaces and still has concrete target-side symlink traversal concerns.

Concerns:

- **[high] Snapshot writes can escape backup target through symlinked directories:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: The folder snapshot store writes under `backup.target/snapshots/<installationId>` without lstat-walking existing directory segments, so an attacker or sync client that can plant symlinks inside the backup target can redirect encrypted snapshot files outside that target.
  - confidence: 0.91
- **[high] Workspace mirroring can operate through a symlinked target root:** `src/commands/workspace-backup.ts:350`
  - body: The workspace mirror path creates mirror and staging directories before target-side symlink rejection, and the later realpath-based containment accepts a symlinked root as contained in itself.
  - confidence: 0.9
