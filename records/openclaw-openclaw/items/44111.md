## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Reject symlinked snapshot directories before writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `writeFileAtomic` and `copyFileAtomic` create the destination directory and then rename into it without checking that `backup.target/snapshots` or the installation directory is a real directory. A synced or shared backup target can plant a symlink there and redirect encrypted payload/envelope writes outside the configured backup target.
  - confidence: 0.91
- **[P1] Reject symlinked workspace target directories before mkdir:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `mirrors` and `.staging` before proving the target-side path has no symlink segments, and the later containment check canonicalizes through symlinks. A hostile or synced backup target can make the mirror or staging path point outside backup.target, so the run can write, rename, or delete outside the intended folder.
  - confidence: 0.9
- **[P2] Remove or implement the retention config:** `src/config/types.backup.ts:20`
  - body: The PR adds public `backup.retention` keys to the exported config type and zod schema, but the reviewed backup runtime never applies them. Once shipped, users can configure retention expecting pruning while snapshots accumulate indefinitely, so either implement the policy or remove the keys until they work.
  - confidence: 0.86
- **[P2] Recheck archive entry types while packing:** `src/infra/backup-create.ts:406-419`
  - body: The export path pre-scans sources before `tar.c`, but the tar write itself has no filter or entry-type guard. A mutable workspace can race the pre-scan and get a symlink or special file into an archive that OpenClaw later rejects on verify/restore, breaking automated backup flows and creating unsafe artifacts for external extractors.
  - confidence: 0.78
- **[P2] Restore in-state config paths that move within state:** `src/commands/backup-restore.ts:710-722`
  - body: Full-host restore skips a config operation whenever the source config and current config path are both inside their state dirs. If the active `OPENCLAW_CONFIG_PATH` is still in-state but at a different relative path than the backed-up host, the restored state tree will not populate the active config path and the command can report success with missing or stale config.
  - confidence: 0.8
- **[P2] Restore in-state OAuth dirs that move within state:** `src/commands/backup-restore.ts:736-748`
  - body: The same covered-by-state shortcut is used for OAuth credentials. When the backup's OAuth dir and the restore host's OAuth dir are both inside state but at different relative paths, full-host restore suppresses the credentials copy and leaves the active auth directory absent or stale.
  - confidence: 0.78

## Security Review
