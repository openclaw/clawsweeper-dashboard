## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked target dirs before mirroring:** `src/commands/workspace-backup.ts:350-351`
  - body: `fs.mkdir(mirrorRoot, { recursive: true })` and the staging mkdir run before any target-side no-symlink check, while `canonicalizePathForContainment` follows existing symlinks. A synced or writable backup target can pre-create `workspace`/`mirrors`/`.staging` as symlinks and make the mirror writes, renames, or stale cleanup operate outside `backup.target`. Walk and reject symlinked target path segments before creating or writing under them.
  - confidence: 0.91
- **[P1] Reject symlinked snapshot store dirs before writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: The folder store creates `path.dirname(filePath)` with recursive mkdir before writing envelopes and payloads, but only lstat-checks final files on reads. If `targetDir/snapshots` or an installation directory is a symlink in a shared backup folder, snapshot uploads can write encrypted payloads/envelopes outside the configured target. Add a target-relative no-symlink directory walk before `mkdir`, `rename`, and copy operations.
  - confidence: 0.9
- **[P2] Remove or implement retention config:** `src/config/zod-schema.backup.ts:5-13`
  - body: The schema accepts `backup.retention.keepDaily`, `keepWeekly`, `keepMonthly`, and `maxSnapshots`, but the inspected backup run/list/status/push/store/docs paths do not apply those values. Accepting inert retention keys creates a public config contract and can make users believe snapshots are being pruned when they are not; remove the keys or implement and document retention behavior with tests.
  - confidence: 0.87
