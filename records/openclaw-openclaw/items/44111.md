## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked snapshot target directories before writing:** `src/backup/snapshot-store/provider-folder.ts:67-85`
  - body: `uploadSnapshot` reaches `copyFileAtomic`/`writeFileAtomic`, which create destination parents under `backup.target/snapshots/<installationId>` with recursive `mkdir` before rejecting symlinked directory segments. In a shared or cloud-synced backup target, a planted `snapshots` or installation-id symlink can redirect payload/envelope writes outside the selected backup folder; create or validate each target segment without following symlinks before any write.
  - confidence: 0.92
- **[P1] Validate workspace backup target before mkdir:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before target-side symlink checks, and the later containment check canonicalizes through symlinks. If `backup.target/workspace` is replaced with a symlink, the mirror writes and stale cleanup can operate in the symlink target instead of the backup folder; reject symlinked target segments before these `mkdir` calls.
  - confidence: 0.91
- **[P2] Remove retention config until pruning exists:** `src/config/zod-schema.backup.ts:23-26`
  - body: The PR accepts `backup.retention.keepDaily/keepWeekly/keepMonthly/maxSnapshots` in persisted config, but the changed backup store and commands never read those fields or prune snapshots. That creates a user-visible config contract that silently does nothing, so either implement tested pruning/list semantics or drop the field until retention is real.
  - confidence: 0.88
