## Summary

Keep open: current main still only has local backup create/verify, so the PR is not obsolete, but it should not merge yet because the new folder-backed backup paths still follow symlinked target directories and the PR lacks real after-fix CLI proof.

## What This Changes

The PR adds folder-backed encrypted snapshot backups and workspace mirroring commands, updates backup config/schema/docs, and expands backup archive/restore hardening tests.

## Best Possible Solution

Land a narrowed native backup flow after shared no-symlink target-directory handling is applied across workspace and snapshot stores, the branch is rebased, and redacted real CLI `backup run/list/restore` proof is attached.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: place symlinks under `backup.target/workspace` or `backup.target/snapshots` before running the new backup commands, and the PR code follows those path segments during mkdir/rename/copy operations.

## Solution Assessment

No. The feature direction is plausible, but the current implementation needs a shared no-symlink directory creation/validation helper before target-side writes and real CLI proof before it is the maintainable merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject symlinked workspace backup roots before writing:** `src/commands/workspace-backup.ts:350-352`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before rejecting symlinked target-side path segments, and the later containment checks canonicalize through `realpath`. If a synced backup target already contains `workspace -> $HOME` or a similar symlink, the mirror writes and stale cleanup can operate outside the configured backup target. Validate each target directory segment with `lstat` before any mkdir/cp/rename/rm in this tree.
  - confidence: 0.94
- **[P1] Reject symlinked snapshot store directories before writes:** `src/backup/snapshot-store/provider-folder.ts:85`
  - body: The folder snapshot store creates and writes through `${targetDir}/snapshots/<installationId>` without checking intermediate directories for symlinks. A shared/cloud-synced backup target can replace `snapshots` or the installation directory with a symlink and make `backup run` write payload/envelope files outside `backup.target`; add no-symlink segment validation before upload/download/list filesystem operations.
  - confidence: 0.93
