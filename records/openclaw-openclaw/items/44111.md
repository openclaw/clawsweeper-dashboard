## Summary

Keep this PR open. The feature is not present on current main, but the proposed branch still has blocking filesystem-safety and restore-mapping issues, and the PR body does not provide successful after-fix real behavior proof for the new CLI flow.

## What This Changes

The PR adds folder-backed encrypted backup snapshots, workspace mirroring, `backup setup/run/status/list/restore`, backup config schema/docs, restore/list plumbing, and hardening tests while keeping local archive aliases.

## Best Possible Solution

Keep the native encrypted backup direction open, but land it only after the filesystem guards are fixed, stale restore path cases are covered, and a redacted real CLI smoke proves setup/run/list/restore on the updated branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `writeFileAtomic` and `copyFileAtomic` create parent directories under `backup.target/snapshots/...` and then rename/copy through them without checking that `snapshots` or the installation directory is a real directory. In a shared or synced backup target, a planted symlink can make `backup run` write payload and envelope files outside the configured backup directory.
  - confidence: 0.88
- **[P1] Validate workspace backup roots before mkdir:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before the later safety checks can reject a symlinked `backup.target/workspace`. `fs.mkdir` follows that symlink, so the command can create/update directories outside the backup target before it fails.
  - confidence: 0.86
- **[P2] Restore configs when in-state paths differ:** `src/commands/backup-restore.ts:715-716`
  - body: Full-host restore skips the config operation whenever the target config path is inside the current state dir and the source config was covered by the old state dir. If the archive had `state/openclaw.json` but the restore host uses `state/custom.json`, the restored state will not populate the active config path even though the command reports a full-host restore.
  - confidence: 0.84
- **[P2] Restore OAuth dirs when in-state paths differ:** `src/commands/backup-restore.ts:741-742`
  - body: The same coverage check skips credentials whenever the current OAuth dir is inside state, without comparing the source and target relative paths. Restoring an archive whose OAuth data was under one in-state path onto a host configured for another in-state path leaves the active OAuth directory missing or stale.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces new filesystem write/restore surfaces and still has concrete symlink traversal and archive-race concerns.

Concerns:

- **[high] Snapshot store follows symlinked target segments:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: Snapshot uploads create parent directories and then copy/rename files under `targetDir/snapshots/...` without validating intermediate path segments, so a symlink planted inside a synced backup target can redirect writes outside that target.
  - confidence: 0.88
- **[high] Workspace mirror writes before target symlink rejection:** `src/commands/workspace-backup.ts:350`
  - body: The workspace mirror path is created before the safety check revalidates the backup workspace root, allowing a symlinked `backup.target/workspace` to be followed before failure.
  - confidence: 0.86
- **[medium] Archive creation relies on a pre-scan before tar writes:** `src/infra/backup-create.ts:406`
  - body: The export path scans sources for symlinks/special files before invoking `tar.c`, but the tar write itself lacks an entry-type guard, leaving a race where a source tree change can produce an archive that restore later rejects or that is unsafe for external extractors.
  - confidence: 0.76
