## Summary

Keep open: current main still only has local archive create/verify, while this PR adds a substantial new encrypted snapshot and restore feature that is not implemented or superseded. The branch is not mergeable as-is because the changed filesystem write/restore paths still have concrete security and data-loss defects, and the PR lacks after-fix real CLI behavior proof.

## What This Changes

The PR adds folder-backed encrypted backup setup/run/status/list/restore commands, workspace mirroring, backup config schema/docs, deprecated aliases, and backup hardening tests.

## Best Possible Solution

Repair or replace the branch with symlink-safe target writes, source-vs-target restore mapping, archive-time entry validation, bounded archive verification, aligned docs/config contracts, and redacted real CLI proof before deciding the native encrypted backup scope.

## Reproduction Assessment

Yes for the PR review findings: source inspection at PR head `dd2c48fe37263cf406593b06e420bf372a5215a0` shows the unsafe target writes, restore skip paths, archive race, and verify expansion path. This is a feature PR, so there is no current-main bug reproduction path for the new snapshot flow itself.

## Solution Assessment

No. The encrypted backup direction is plausible, but this branch is not the best mergeable implementation until target-path safety, restore mapping, archive validation, verification bounds, and real CLI proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:66-85`
  - body: `writeFileAtomic` and `copyFileAtomic` create parent directories and then rename/copy through `backup.target/snapshots/...` without checking that `snapshots` or the installation directory is a real directory. A symlink planted inside a synced backup target can redirect snapshot payload/envelope writes outside the configured backup directory.
  - confidence: 0.89
- **[P1] Validate workspace backup roots before mkdir:** `src/commands/workspace-backup.ts:350-352`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before the later safety checks can reject a symlinked `backup.target/workspace`. `fs.mkdir` follows that symlink, so the command can create or update directories outside the backup target before validation completes.
  - confidence: 0.87
- **[P2] Restore configs when in-state paths differ:** `src/commands/backup-restore.ts:714-716`
  - body: Full-host restore skips the config operation whenever the target config path is inside the current state dir and the source config was covered by the old state dir. If the archive had `state/openclaw.json` but the restore host uses `state/custom.json`, the active config path is left missing or stale after a reported full-host restore.
  - confidence: 0.84
- **[P2] Restore OAuth dirs when in-state paths differ:** `src/commands/backup-restore.ts:740-742`
  - body: The credentials branch makes the same covered-by-state assumption without comparing source and target relative paths. Restoring from one in-state OAuth location onto a host configured for another in-state OAuth path leaves the active credentials directory missing or stale.
  - confidence: 0.83
- **[P2] Validate tar entries while archiving:** `src/infra/backup-create.ts:406-420`
  - body: The export path scans source trees before calling `tar.c`, but the actual tar write has no entry-type check. A source tree can be swapped after the pre-scan so `backup export` emits symlink or special-file entries that restore later rejects, or that are unsafe for external extractors.
  - confidence: 0.77
- **[P2] Bound parent expansion during verification:** `src/commands/backup-verify.ts:337-354`
  - body: `collectNestedArchiveParents` inserts every parent path for every archive entry, but the verifier only caps total entry path bytes, not the number or total size of generated parent strings. A crafted deeply nested tar path can make `backup verify` and restore preflight spend excessive CPU and memory before failing.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: Needs attention: the diff adds backup write/restore surfaces and still has concrete symlink traversal, archive-race, and verification DoS concerns.

Concerns:

- **[high] Snapshot store follows symlinked target segments:** `src/backup/snapshot-store/provider-folder.ts:66`
  - body: Snapshot uploads create parent directories and then copy/rename files under `targetDir/snapshots/...` without validating intermediate path segments, so a symlink planted inside a synced backup target can redirect writes outside that target.
  - confidence: 0.89
- **[high] Workspace mirror writes before target symlink rejection:** `src/commands/workspace-backup.ts:350`
  - body: The workspace mirror and staging paths are created before the safety check revalidates the backup workspace root, allowing a symlinked `backup.target/workspace` to be followed before validation completes.
  - confidence: 0.87
- **[medium] Archive creation relies on a pre-scan before tar writes:** `src/infra/backup-create.ts:406`
  - body: The export path scans sources for symlinks/special files before invoking `tar.c`, but the tar write itself lacks an entry-type guard, leaving a race where source tree changes can produce unsafe or unrestorable archive entries.
  - confidence: 0.77
- **[medium] Archive verification can expand deep paths excessively:** `src/commands/backup-verify.ts:337`
  - body: A crafted archive path within the current metadata byte cap can still generate a very large set of parent strings during verification, creating an avoidable CPU/memory denial-of-service path before restore.
  - confidence: 0.74
