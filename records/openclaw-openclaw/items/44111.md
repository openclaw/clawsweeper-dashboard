Full review comments:

- **[P1] Reject symlinked snapshot-store directories before writes:** `src/backup/snapshot-store/provider-folder.ts:67-81`
  - body: `writeFileAtomic` and `copyFileAtomic` create and rename under `targetDir/snapshots/...` without checking existing intermediate directories for symlinks. In a synced/shared backup target, an attacker can replace `snapshots` or the installation directory with a symlink and redirect snapshot payload/envelope writes outside the configured backup target.
  - confidence: 0.9
- **[P1] Reject target-side symlinks before mirroring:** `src/commands/workspace-backup.ts:350-358`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before any target-side symlink rejection, then uses realpath containment checks that follow symlinks. A symlinked `workspace`, `mirrors`, or `.staging` directory can redirect `fs.cp`, rename, or stale mirror cleanup outside the backup target.
  - confidence: 0.88
- **[P1] Revalidate archive entries while tar writes:** `src/infra/backup-create.ts:406-420`
  - body: The new pre-scan runs before `tar.c`, but the actual archive write has no filter or entry-type guard. A writable source tree can swap in a symlink, FIFO, device, or other special entry after the pre-scan and still produce an archive that restore/verify later rejects or that is unsafe with external extractors.
  - confidence: 0.84
- **[P2] Remove or implement backup retention config:** `src/config/zod-schema.backup.ts:22-27`
  - body: The schema accepts `backup.retention.keepDaily`, `keepWeekly`, `keepMonthly`, and `maxSnapshots`, but the backup commands do not enforce pruning. That creates a public config contract users can rely on even though snapshots will never be retained or deleted according to it.
  - confidence: 0.86
