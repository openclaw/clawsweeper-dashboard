## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject symlinked snapshot store directories:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `writeFileAtomic` creates the destination directory before proving `targetDir/snapshots` and the installation directory are real directories. In a synced or shared backup target, a planted symlink can redirect snapshot payload/envelope writes outside `backup.target`; validate every target-side segment before upload, download, and listing operations.
  - confidence: 0.93
- **[P1] Reject workspace target symlinks before mirror writes:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates the mirror and staging roots before rejecting target-side symlink segments, and the later containment check canonicalizes through symlinks. A planted `backup.target/workspace` or `mirrors` symlink can redirect copy, rename, or stale-mirror removal outside the backup folder.
  - confidence: 0.94
- **[P1] Validate restore parents before rollback moves:** `src/commands/backup-restore.ts:443`
  - body: `applyRestoreOperations` renames existing targets into rollback storage before `copySourceToTarget` checks parent segments with `ensureDirectoryPathWithoutSymlinks`. If a config/state/OAuth target parent contains a symlink, restore can move or create rollback data outside the intended restore tree before failing.
  - confidence: 0.88
- **[P1] Remove or enforce backup retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention`, but the PR does not apply those values when creating, listing, or pruning snapshots. Users can set retention expecting sensitive encrypted snapshots to be pruned, while the implementation keeps accumulating them; remove the field or implement and test enforcement before shipping it.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive backup, encryption, and restore code with concrete filesystem boundary and retention-policy concerns.

Concerns:

- **[high] Target-side symlink traversal on snapshot writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: Snapshot store writes create directories under the configured backup target without rejecting symlinked intermediate segments, allowing writes outside the intended backup target when that folder is attacker-modifiable.
  - confidence: 0.93
- **[high] Workspace mirror writes can follow target symlinks:** `src/commands/workspace-backup.ts:350`
  - body: Workspace backup creates mirror/staging directories before target-side symlink validation, so copy/rename/rm can be redirected through a planted backup-target symlink.
  - confidence: 0.94
- **[medium] Restore rollback happens before parent symlink validation:** `src/commands/backup-restore.ts:443`
  - body: Restore moves existing targets into rollback paths before proving parent directories are not symlinks, creating a destructive-operation window outside the intended tree.
  - confidence: 0.88
- **[medium] Retention config is accepted but unenforced:** `src/config/zod-schema.backup.ts:25`
  - body: Accepting `backup.retention` without pruning behavior can mislead users about how long encrypted sensitive snapshots remain on disk.
  - confidence: 0.91
