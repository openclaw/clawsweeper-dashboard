## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate snapshot store directory segments:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: The folder store creates `targetDir/snapshots/...` with recursive `mkdir` and then writes/renames through that path. If a shared backup target already contains a symlinked `snapshots` or installation directory, snapshot payloads and envelopes can be written outside `backup.target`; reject symlinked directory segments before creating or writing files there.
  - confidence: 0.92
- **[P1] Reject target-side symlinks before mirroring:** `src/commands/workspace-backup.ts:350-351`
  - body: `backup.target/workspace` can be a symlink inside a shared/synced backup folder. These `mkdir` calls follow it before any effective target-side symlink rejection, and the later realpath containment check accepts the symlinked root, so mirror writes and stale cleanup can operate outside the backup directory.
  - confidence: 0.9
- **[P1] Remove or implement retention config:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention`, but the reviewed upload/list/run paths never prune snapshots or otherwise enforce those limits. Shipping an ignored config key creates a public contract users can rely on while snapshots grow unbounded, so either implement retention with tests or remove the key for now.
  - confidence: 0.86
