## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks the proposed encrypted snapshot backup flow, while the latest PR head leaves target-side symlink write paths unresolved and does not include successful real behavior proof for the new CLI flow.

## What This Changes

The PR adds folder-backed encrypted backup snapshots, workspace mirroring, `backup setup/run/status/list/restore/export` CLI plumbing, backup config schema/docs, and hardening/regression tests.

## Best Possible Solution

Keep the native backup work open, harden all backup-target directory creation with no-symlink segment checks before writes, add pack-time archive entry validation, then provide a successful real CLI smoke for setup/run/list/restore with private paths redacted.

## Reproduction Assessment

Yes. The review findings are source-reproducible from the latest PR head: the new workspace and folder snapshot write paths create or rename through target-side directories before rejecting symlinked path segments.

## Solution Assessment

No. The feature direction is useful, but the current patch needs target-side symlink hardening, pack-time tar entry checks, and real behavior proof before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject symlinked workspace backup roots before writing:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `target/workspace/mirrors` and `.staging` before any target-side lstat/no-symlink check. If a synced backup folder already contains `workspace -> $HOME`, the mkdir/cp/rename/rm operations follow that link, and the later realpath containment checks compare paths inside the same symlink target and pass.
  - confidence: 0.91
- **[P1] Reject symlinked snapshot store directories before writing:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: The folder store creates destination parents with recursive `mkdir` and then renames payload/envelope files into them, but it never rejects existing symlinked segments under `backup.target`, such as `snapshots -> /other`. A modified sync folder can redirect snapshot writes outside the configured target.
  - confidence: 0.89
- **[P2] Recheck backup source entries while tar packs them:** `src/infra/backup-create.ts:365-366`
  - body: The new pre-scan rejects symlinks and special files before `tar.c` runs, but the actual pack step still walks the filesystem later without a filter or entry-type guard. A live workspace can change between the scan and pack, producing a backup that verify/restore rejects or that is unsafe with external extractors.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff adds security-sensitive backup and restore writes, and target-side symlink traversal remains in the new backup folder storage paths.

Concerns:

- **[high] Workspace mirror can follow backup-target symlinks:** `src/commands/workspace-backup.ts:350`
  - body: The workspace backup path creates and mutates `backup.target/workspace/*` before proving those target-side directories are real directories, so a shared/synced target can redirect writes or deletes through symlinks.
  - confidence: 0.91
- **[high] Snapshot store writes can be redirected outside target:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: Payload and envelope writes do not reject symlinked intermediate directories under `backup.target`, allowing an attacker-controlled sync folder to redirect encrypted snapshot files outside the intended backup directory.
  - confidence: 0.89
- **[medium] Archive creation has a TOCTOU entry-type gap:** `src/infra/backup-create.ts:365`
  - body: The source-tree pre-scan is not enforced by the subsequent tar pack operation, leaving a race where symlink or special-file entries can enter an archive after validation.
  - confidence: 0.79
