## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject target-side symlinks before snapshot writes:** `src/backup/snapshot-store/provider-folder.ts:67`
  - body: `writeFileAtomic` and `copyFileAtomic` create the destination parent with `fs.mkdir(..., { recursive: true })` and then rename under `targetDir/snapshots/...` without rejecting symlinked intermediate directories inside the backup target. A synced/shared backup folder can therefore redirect snapshot payload or envelope writes outside the configured backup target; add a no-symlink directory walk or fd-relative write guard before any mkdir/copy/rename.
  - confidence: 0.93
- **[P1] Reject target-side symlinks before mirroring workspaces:** `src/commands/workspace-backup.ts:350-351`
  - body: `workspaceBackupRunCommand` creates `backup.target/workspace/mirrors` and `.staging` before checking the target-side directory tree, and the later containment helper canonicalizes through symlinks. If `workspace`, `mirrors`, or `.staging` is a symlink planted in a synced backup folder, the copy/rename/stale-delete operations can escape the backup target; reject symlink segments before these writes and deletes.
  - confidence: 0.92
- **[P1] Remove or implement backup.retention:** `src/config/zod-schema.backup.ts:25`
  - body: The schema accepts `backup.retention`, but the PR diff only adds the type/schema fields and never applies pruning in backup run/list/status or documents the behavior. Accepting this config creates a false operator expectation that snapshots will be expired, which is especially risky for encrypted state/credential backups; either remove the key for now or implement and test retention semantics.
  - confidence: 0.95
- **[P2] Cap archive parent expansion during verification:** `src/commands/backup-verify.ts:337-347`
  - body: `collectNestedArchiveParents` expands every parent segment for every normalized archive entry, while the new limits only cap entry count and total path bytes. A crafted archive with very deep paths can still force a huge parent set during verify/restore; add a path-depth or parent-count cap before collecting parents from untrusted archive metadata.
  - confidence: 0.82
