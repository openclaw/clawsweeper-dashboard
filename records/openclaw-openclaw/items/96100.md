## What This Changes

The PR changes the embedded attempt session-lock controller so retained-lock cleanup paths wait for active retained writes instead of immediately falling back or returning during abort cleanup.

## Best Possible Solution

Keep this focused fix path open, add or cite exact active-scope abort coverage, and merge only after redacted runtime or Testbox/Crabbox proof shows the retained lock releases before the watchdog window.

## Reproduction Assessment

Yes at source level: current main can return early from retained-lock cleanup while an active write-lock scope exists, matching the linked report’s retained lock leak. I did not run a live embedded abort reproduction in this read-only review.

## Solution Assessment

Likely yes, but proof is incomplete: final cleanup always releases a real cleanup lock, so making the retained cleanup transfer wait is a narrow owner-boundary fix. The remaining gap is exact regression/runtime proof for abort during an active retained write.
