## Summary

Keep open for merge review: current main still has the retained-lock early-return path, this PR adds a narrow deferred release with focused regression coverage and now-sufficient terminal proof, and I found no blocking correctness or security finding.

## What This Changes

The PR changes the embedded attempt session-lock controller to defer held-lock release when abort cleanup runs inside an active retained write scope, then retries release after scope deactivation, with three regression tests.

## Best Possible Solution

Land the narrow fence-preserving deferred-release fix after session-lock owner acceptance, while keeping broader force-release, dispose, and memory-retention work separate or explicitly tracked.

## Reproduction Assessment

Yes at source level: current main can return early from retained-lock abort cleanup inside an active write scope, and the linked production issue shows the corresponding lock-timeout symptom. I did not run a live current-main reproduction in this read-only review.

## Solution Assessment

Likely yes: deferring release until after the active retained writer unwinds is narrower and safer than force-releasing while that writer still owns the scope. Broader force-release or bounded-dispose fixes may still be needed for adjacent failure modes, but they are not safer replacements for this active-scope fix.
