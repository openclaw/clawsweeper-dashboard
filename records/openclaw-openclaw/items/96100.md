## What This Changes

The PR defers retained held-lock release when abort cleanup runs inside an active embedded session write scope, then retries release after scope deactivation, with focused controller regression tests.

## Best Possible Solution

Land the narrow fence-preserving deferred-release fix after session-lock owner review and either direct abort-path proof or explicit maintainer acceptance of controller-level proof; keep broader bounded-dispose and force-release work in their separate paths.

## Reproduction Assessment

Yes at source level: current main can return early from retained-lock abort cleanup inside an active write scope, and the linked production issue shows the corresponding lock-timeout symptom. I did not run a live retained-write abort reproduction in this read-only review.

## Solution Assessment

Likely yes: deferring release until after the active retained writer unwinds is narrower than force-releasing while the writer still owns the scope. Broader force-release or bounded-dispose proposals may still be needed for adjacent failure modes, but they are not the safer replacement for this active-scope fix.
