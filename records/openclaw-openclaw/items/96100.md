## Summary

Keep open: current main and v2026.6.10 still have the active-scope retained-lock abort release gap, while this PR provides a narrow fence-preserving deferred release with focused regression coverage and sufficient terminal proof; the remaining decision is maintainer acceptance of a session-state-sensitive lock-ordering change.

## What This Changes

The PR defers retained session held-lock release when abort cleanup runs inside an active write scope, then retries the release after that scope unwinds, with three regression tests.

## Best Possible Solution

Land the narrow fence-preserving deferred-release fix after session-lock owner acceptance, while tracking broader dispose, force-release, and memory-retention work separately.

## Reproduction Assessment

Yes, at source level: current main can return early from retained-lock abort cleanup inside an active write scope, and the linked production issue shows the corresponding session lock timeout symptom. I did not run a live current-main multi-channel reproduction in this read-only review.

## Solution Assessment

Likely yes: deferring release until after the active retained writer unwinds is narrower and safer than force-releasing while that writer still owns the scope. Broader force-release or bounded-dispose fixes may still be needed for adjacent failure modes, but they are not safer replacements for this active-scope abort fix.
