- `merge-risk: 🚨 session-state`: The diff changes session write-lock ownership behavior while retained transcript writes and fence state may still be active.

## Summary

Keep open: the PR targets the right session-lock cleanup surface, but the added waits can self-wait when called from the active retained write scope, so the patch is not merge-ready without a scope-safe cleanup design and real abort proof.

## What This Changes

The PR changes embedded attempt session-lock cleanup so `takeHeldLockAfterRetainedIdle()` and `disposeHeldLockAfterRetainedIdle()` wait on retained-lock idle waiters instead of returning when abort cleanup encounters an active retained write.

## Best Possible Solution

Replace the active-scope self-wait with a scope-safe cleanup design, such as deferring release until the retained write scope completes or using explicit terminal takeover semantics, then prove that interleaving with focused tests and redacted runtime evidence.

## Reproduction Assessment

Yes at source level: current main has the retained-lock false branch and the linked issue supplies production logs for the resulting lock leak. I did not run a live embedded abort reproduction in this read-only review.

## Solution Assessment

No, not as-is. The patch targets the right owner module, but awaiting retained idle from inside the active retained write scope can block the release it is waiting for.
