## Summary

Keep open: the linked lock-leak problem is valid, but this PR is not merge-ready because the new waits run in the branch that means the caller is already inside the active retained write scope, so cleanup can self-wait instead of releasing the lock.

## What This Changes

The PR changes embedded attempt session-lock cleanup so `takeHeldLockAfterRetainedIdle()` and `disposeHeldLockAfterRetainedIdle()` wait on retained-lock idle waiters instead of returning when abort cleanup encounters an active retained write.

## Best Possible Solution

Replace the active-scope self-wait with a scope-safe lock cleanup design, such as deferring the fence-preserving release until the retained write scope completes or using an explicit terminal takeover path, then cover that interleaving with focused tests and redacted runtime proof.

## Reproduction Assessment

Yes at source level: current main has an active-scope false branch in retained-lock cleanup, and the linked issue supplies production logs for the resulting lock leak. I did not run a live embedded abort reproduction in this read-only review.

## Solution Assessment

No, not as-is. The patch targets the right owner module, but awaiting retained idle from inside the active retained write scope is not the narrow safe fix because it can wait on its own unwind.
