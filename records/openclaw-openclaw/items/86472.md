## Summary

Keep open: the linked heartbeat/session bug is plausible and the PR targets the right filter path, but the current patch still treats any text containing `HEARTBEAT_OK` as removable tail context and can delete middle-token heartbeat alert context before the next user prompt is assembled. ## What This Changes

The branch changes heartbeat transcript filtering to remove trailing text-only assistant heartbeat output that contains `HEARTBEAT_OK` when no user/tool follow-up exists, and updates heartbeat-filter coverage for trimming, no-token preservation, user-followed preservation, and tool-backed preservation.

## Best Possible Solution

Preserve the existing middle-token alert contract, trim only narrowly classified stale acknowledgement wrappers, add a regression for that boundary, and provide redacted real heartbeat-to-first-user proof.

## Reproduction Assessment

Yes at source level, but not as a live run: current main preserves nonterminal assistant heartbeat output at the tail, and this PR changes that branch. The linked DeepSeek/stepfun provider flow still lacks redacted after-fix runtime proof.

## Solution Assessment

No: the patch is close to the right area, but the broad substring classifier changes the existing middle-token alert behavior and can drop meaningful context. A safer solution is a narrower ack/wrapper detector plus real post-heartbeat user-message proof.
