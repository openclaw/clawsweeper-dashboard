## Summary

Keep open: the reported Feishu/Lark webhook setup bug is valid and this is the focused fix candidate, but the PR is not mergeable as-is because it is stale against current webhook ingress hardening, has a malformed encrypted payload fail-closed gap, and still lacks inspectable real behavior proof.

## What This Changes

This PR changes Feishu webhook handling to accept unsigned `url_verification` challenges when the request signature is valid or the configured verification token matches, and adds plaintext/encrypted webhook e2e coverage.

## Best Possible Solution

Port the challenge-token exception onto current main's guarded Feishu webhook path, preserve strict signatures for normal events, fail closed for malformed encrypted payloads, and add positive plus negative coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure Feishu webhook mode with `encryptKey` and `verificationToken`, then POST an unsigned plaintext or encrypted `url_verification` body; current main returns `401 Invalid signature` before challenge handling.

## Solution Assessment

No, not as-is. The challenge-scoped verification-token exception is the right narrow direction, but this branch needs a current-main port and fail-closed malformed encrypted challenge handling before merge.
