## Summary

Keep open: the Feishu webhook bug is valid on current main and the latest stable tag, but this PR is not merge-ready because it is conflicting, based on older webhook ingress code, moves encrypted challenge parsing ahead of fail-closed authorization, and lacks inspectable real behavior proof.

## What This Changes

This PR changes Feishu webhook handling to accept unsigned plaintext or encrypted `url_verification` challenges when the configured verification token matches, with webhook e2e coverage for those setup flows.

## Best Possible Solution

Port the challenge-token exception onto current main's guarded Feishu webhook path, preserve strict signatures for normal events, fail closed for malformed encrypted challenge payloads, and add positive plus negative coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure Feishu webhook mode with `encryptKey` and `verificationToken`, then POST an unsigned `url_verification` body; current main returns `401` before challenge handling.

## Solution Assessment

No, not as-is. The challenge-scoped verification-token exception is the right narrow direction, but this branch needs a current-main port and fail-closed malformed encrypted challenge handling before merge.
