## Best Possible Solution

Refresh the fix onto current main, authorize or safely decrypt only `url_verification` challenges before the normal signature gate, keep ordinary events signature-required, add negative boundary coverage, and include redacted real setup proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Authorize before decrypting challenge payloads:** `extensions/feishu/src/monitor.transport.ts:305-311`
  - body: The new authorization check runs only after `Lark.generateChallenge(payload, ...)`, but that SDK helper decrypts `payload.encrypt` and can throw on malformed ciphertext. An unsigned pre-auth request with a bad `encrypt` value now reaches the outer catch as a 500 instead of a controlled 401/400, so parse/decrypt the challenge path with the catching helper before calling the SDK helper or handle SDK failures explicitly.
  - confidence: 0.86
