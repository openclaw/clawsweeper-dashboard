## Summary

Keep open: the Feishu/Lark webhook bug is valid and the PR direction is useful, but this branch is not mergeable as-is because it lacks inspectable after-fix real behavior proof and needs a current-main port that preserves webhook hardening.

## What This Changes

The branch changes Feishu webhook handling to accept unsigned plaintext or encrypted `url_verification` challenges when the configured verification token matches, with focused e2e coverage.

## Best Possible Solution

Port the token-verified challenge flow onto current main's Feishu webhook auth path while preserving pre-auth guards, `encryptKey` startup checks, strict normal-event signature validation, fail-closed malformed encrypted challenge handling, and focused positive/negative coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows unsigned Feishu `url_verification` requests return `401 Invalid signature` before challenge handling, and the current e2e suite asserts that behavior.

## Solution Assessment

No, not as-is. The challenge-scoped verification-token exception is the right narrow direction, but this branch must be ported onto current webhook guards and must fail closed for malformed encrypted challenge probes.
