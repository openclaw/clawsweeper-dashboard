## Summary

Keep open. The Feishu/Lark setup bug is real and the PR direction is right, but the branch is stale against current Feishu webhook transport and needs a current-main port plus changelog before merge.

## What This Changes

The PR adds Feishu webhook helpers and e2e cases so unsigned plaintext or encrypted `url_verification` challenges can be accepted when the configured verification token matches.

## Best Possible Solution

Land a current-main Feishu plugin fix that accepts only valid `url_verification` setup challenges via a valid signature or timing-safe verification-token match, while preserving current pre-auth body limits, rate limiting, startup secret guards, and fail-closed normal events.

## Reproduction Assessment

Yes. Send a JSON or encrypted Feishu `url_verification` request without Lark signature headers to webhook mode with `encryptKey` configured; current main returns `401 Invalid signature`, and the e2e suite currently asserts that missing-signature challenge rejection.

## Solution Assessment

No, not as-is. The intended challenge-scoped auth exception is the right narrow direction, but it must be rebased or replaced on current main and include the required changelog entry.
