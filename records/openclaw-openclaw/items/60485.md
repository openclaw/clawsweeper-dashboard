## Summary

Keep open. This is a real Feishu webhook setup bug and the PR is aimed at the right behavior, but the branch cannot merge as-is because it is based on an older webhook transport path and would drop current pre-auth/startup guards unless ported carefully.

## What This Changes

The PR changes Feishu webhook transport and e2e tests to accept unsigned plaintext or encrypted `url_verification` challenges when the configured verification token matches.

## Best Possible Solution

Land a current-main Feishu plugin fix that accepts only signed or verification-token-matching `url_verification` challenges while preserving pre-auth body limits, rate limiting, startup `encryptKey` guards, and strict normal-event signature validation.

## Reproduction Assessment

Yes. Source inspection shows current main returns `401 Invalid signature` before challenge handling for unsigned Feishu `url_verification` payloads, and the current e2e test asserts that behavior.

## Solution Assessment

No, not as-is. The challenge-scoped auth exception is the right narrow direction, but this branch must be ported onto current webhook guards and fail closed for malformed encrypted challenge probes.
