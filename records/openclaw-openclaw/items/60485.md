## Summary

Keep open: current main still has the Feishu handshake bug, but this branch is stale against newer webhook hardening and still lacks inspectable real behavior proof, so it should not merge as-is.

## What This Changes

The PR changes Feishu webhook challenge handling and e2e coverage so unsigned plaintext or encrypted `url_verification` callbacks can pass when the configured verification token matches.

## Best Possible Solution

Port the challenge-token flow onto current main while preserving pre-auth body guards, `encryptKey` startup checks, fail-closed malformed encrypted challenge handling, strict normal-event signature validation, and focused e2e/security coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows unsigned Feishu `url_verification` bodies reach the `401 Invalid signature` branch before JSON parsing or `Lark.generateChallenge()`, and the current e2e suite asserts that behavior.

## Solution Assessment

No, not as-is. The challenge-scoped verification-token exception is the right narrow direction, but this branch must be ported onto current webhook guards and must fail closed for malformed encrypted challenge probes.
