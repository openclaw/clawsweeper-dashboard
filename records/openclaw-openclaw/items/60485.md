## Summary

Keep open: the Feishu webhook bug is valid, but this branch is stale against current webhook hardening and still needs fail-closed auth handling plus inspectable real behavior proof before merge.

## What This Changes

Adds Feishu webhook helpers and e2e coverage to accept unsigned plaintext or encrypted `url_verification` challenges when the configured verification token matches.

## Best Possible Solution

Port the token-verified challenge flow onto current main's Feishu webhook auth path while preserving pre-auth guards, startup `encryptKey` checks, strict normal-event signatures, and positive plus negative coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure Feishu webhook mode with `encryptKey` and `verificationToken`, then POST a plaintext or encrypted `url_verification` body without Lark signature headers; current main reaches `401 Invalid signature` before `Lark.generateChallenge()`.

## Solution Assessment

No, not as-is. The challenge-scoped verification-token exception is the right narrow direction, but this branch needs a current-main port and fail-closed malformed encrypted challenge handling.
