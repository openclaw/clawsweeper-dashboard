## Summary

Keep open: current main still has the Feishu webhook setup bug, but this PR is not merge-ready because its branch is based on older webhook ingress code, exposes an unauthenticated encrypted-challenge error path, and lacks inspectable real behavior proof.

## What This Changes

The PR changes Feishu webhook transport and e2e tests to accept unsigned plaintext or encrypted `url_verification` challenges when the configured verification token matches.

## Best Possible Solution

Port the useful challenge-token flow onto current main while preserving pre-auth body guards, `encryptKey` startup checks, fail-closed malformed encrypted challenge handling, strict normal-event signature validation, and focused e2e/security coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows unsigned Feishu `url_verification` bodies reach the `401 Invalid signature` branch before JSON parsing or `Lark.generateChallenge()`, and the current e2e suite asserts that behavior.

## Solution Assessment

No, not as-is. The challenge-scoped auth exception is the right narrow direction, but this branch must be ported onto current webhook guards and must fail closed for malformed encrypted challenge probes.
