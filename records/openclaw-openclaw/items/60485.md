## Summary

Keep open: the Feishu/Lark webhook bug is real on current main, but this PR is not mergeable as-is because it is based on older webhook ingress code, can turn malformed unsigned encrypted challenges into a handler error before authorization, and still lacks inspectable real behavior proof.

## What This Changes

The PR changes Feishu webhook handling to accept unsigned plaintext or encrypted `url_verification` challenges when the configured verification token matches, and adds webhook e2e coverage.

## Best Possible Solution

Port the challenge-token exception onto current main's guarded Feishu webhook path, preserve strict signatures for normal events, fail closed for malformed encrypted challenge payloads, and add positive plus negative coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure Feishu webhook mode with `encryptKey` and `verificationToken`, then POST an unsigned plaintext or encrypted `url_verification` body; current main returns `401 Invalid signature` before challenge handling.

## Solution Assessment

No, not as-is. The challenge-scoped verification-token exception is the right narrow direction, but this branch needs a current-main port and fail-closed malformed encrypted challenge handling before merge.
