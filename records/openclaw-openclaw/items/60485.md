The PR changes Feishu webhook handling to accept unsigned plaintext or encrypted `url_verification` challenges when the verification token matches, and adds e2e coverage for those success paths.

## Best Possible Solution

Land or replace this with a current-main Feishu plugin fix that preserves pre-auth body guards, accepts only token-verified `url_verification` setup challenges, rejects malformed encrypted challenges cleanly, and includes inspectable redacted live Lark proof.
