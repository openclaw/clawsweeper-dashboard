## Summary

Keep open. The Feishu/Lark webhook handshake bug is still present on current main and the related open bug report remains valid, but this PR is not merge-ready because the branch is stale against current webhook ingress hardening, has two concrete auth-boundary findings, and still lacks inspectable after-fix real behavior proof.

## What This Changes

This PR updates Feishu webhook handling and e2e coverage to accept unsigned plaintext or encrypted `url_verification` challenges when the configured verification token matches.

## Best Possible Solution

Port the token-verified Feishu challenge flow onto current main's guarded webhook path, preserving strict signatures for normal events and adding positive plus negative coverage for plaintext and encrypted challenges.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure Feishu webhook mode with `encryptKey` and `verificationToken`, then POST a plaintext or encrypted `url_verification` body without Lark signature headers; current main reaches `401 Invalid signature` before challenge handling.

## Solution Assessment

No, not as-is. The challenge-scoped verification-token exception is the right narrow direction, but this branch needs a current-main port and fail-closed malformed encrypted challenge handling before merge.
