## Summary

Keep open. The Feishu/Lark webhook URL-verification bug is still present on current main and latest release, but this PR is not mergeable as-is because its branch is stale against current webhook hardening and lacks inspectable real behavior proof.

## What This Changes

The PR changes Feishu webhook handling to accept unsigned `url_verification` challenges when the configured verification token matches, and adds webhook e2e coverage for plaintext and encrypted challenge flows.

## Best Possible Solution

Port the token-verified Feishu challenge flow onto current main's guarded webhook path, preserve strict signatures for normal events, fail closed for malformed encrypted payloads, and add positive plus negative coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure Feishu webhook mode with `encryptKey` and `verificationToken`, then POST a plaintext or encrypted `url_verification` body without Lark signature headers; current main reaches `401 Invalid signature` before challenge handling.

## Solution Assessment

No, not as-is. The challenge-scoped verification-token exception is the right narrow direction, but this branch needs a current-main port and fail-closed malformed encrypted challenge handling before merge.
