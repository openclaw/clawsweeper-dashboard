## Summary

Keep open: current main does not implement the emotion-mode gateway surface, and the PR head still has blocking display correctness issues. The branch is also dirty against its base and sits on an open stack, so this needs maintainer review and branch updates rather than cleanup closure.

## What This Changes

The PR adds the stacked TTS/personality emotion-mode surface across provider TTS routing, `/emotions` persistence, `voice.md` prompt guidance, gateway history/SSE/live display, session/status output, protocol schema, generated Swift models, docs, and tests.

## Best Possible Solution

Update the stack in order, thread the normalized session `emotionMode` through every gateway display caller that has session context, and resolve or explicitly accept #72821 with focused sanitizer tests before merge.

## Reproduction Assessment

Yes. On PR head, a `chat.history` request for a session with `entry.emotionMode: "full"` and assistant text `[softly] hello` reaches `projectRecentChatDisplayMessages` without an emotion mode and is defaulted to `off`, so the history surface hides tags that full mode should show; `[fast][1] before choosing.` at line start follows the same source-level sanitizer path described in #72821.

## Solution Assessment

No. The overall per-session emotion-mode direction is reasonable, but this patch is not the best mergeable solution until `chat.history` receives the session mode and the markdown-reference sanitizer ambiguity is fixed or explicitly deferred by maintainers.
