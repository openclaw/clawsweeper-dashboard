## Summary

The PR addresses a real current-main `/btw` resolver divergence, and the proposed async fallback path is technically plausible. It should remain open because the external contributor has not supplied after-fix real behavior proof, and the branch introduces a second copy of complex provider/runtime resolution policy whose broader auth-provider behavior needs feature-owner confirmation.

## What This Changes

The PR changes `/btw` to select the configured harness/runtime, resolve models through `resolveModelAsync` with runtime-aware static-catalog fallback and discovery retry, and adds regression tests for canonical aliases routed through CLI runtimes.

## Best Possible Solution

Use one shared runtime-model resolution contract for main turns and `/btw`, including explicit transport ownership, then verify the exact reported alias setup with redacted live output before merge.

## Reproduction Assessment

Yes from source with high confidence: current `/btw` performs a single-provider registry lookup that cannot resolve the reported canonical alias when it is absent from discovered models. The linked issue also provides a deterministic released-version setup, though this review did not execute that live environment.

## Solution Assessment

Not fully. Moving `/btw` to `resolveModelAsync` with static-catalog fallback is the right behavior, but duplicating the main runner’s orchestration and defining broader transport ownership locally is less maintainable than a shared resolver contract.
