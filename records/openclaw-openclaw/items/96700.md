## What This Changes

## Best Possible Solution

Move the unwrap into one shared pre-validation normalizer used by prepared and guarded edit paths, add focused wrapped and mixed-entry regression tests, and require redacted provider-path proof before merge.

## Reproduction Assessment

Yes for the source-level wrapped-entry shape: current main leaves `{ item: ... }` edit entries unchanged while guarded edit validation only accepts direct `oldText` and `newText`. A full live DeepSeek/XML-RPC provider run was not performed in this read-only review.

## Solution Assessment

No, not yet. The prepared-call normalization is plausible, but duplicating the unwrap inside base `execute` is too late for guarded host and sandbox edit tools; a shared pre-validation normalizer is the narrower maintainable fix.
