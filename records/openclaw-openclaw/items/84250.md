## Summary

Keep open: the reported bug is credible and the PR has useful proof, but the patch currently weakens the fail-closed session takeover fence by publishing “owned” fingerprints from prompt-release/cleanup paths that are not tied to a verified in-process transcript write.

## What This Changes

The PR changes embedded attempt session-lock handling to accept later same-process session-file fingerprints while preserving external takeover detection, with a focused regression test.

## Best Possible Solution

Keep the same-process write tolerance, but record owned fingerprints only after known OpenClaw-controlled locked transcript writes and add a negative regression for external edit followed by another controller release.

## Reproduction Assessment

Yes. Current main source shows any fingerprint change after releaseForPrompt is rejected, and the PR body supplies before-fix failure output for the same source-level race.

## Solution Assessment

No. The direction is right, but publishing ownership from releaseForPrompt/cleanup release is too broad; ownership needs to be tied to actual locked writes, with a regression proving external edits still fail closed.

- **[P1] Publish fingerprints only after verified owned writes:** `src/agents/pi-embedded-runner/run/attempt.session-lock.ts:382`
  - body: This records the current session-file fingerprint whenever a controller releases for prompt, even if that controller did not write the transcript. Please record ownership only from known OpenClaw write paths and add coverage for external edit plus second-controller release.
