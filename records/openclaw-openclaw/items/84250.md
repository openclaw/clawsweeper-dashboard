## Summary

Keep open: this is a focused fix for the embedded session false-takeover path, but the current head broadens the trust boundary enough to create a concrete external-edit masking bug and the supplied proof is stale after the latest force-push.

## What This Changes

The PR records trusted same-process session-file fingerprints so a prompt-released embedded attempt can accept later locked writes from another controller, with regression tests for same-process and external edit cases.

## Best Possible Solution

Keep the narrower owned-transcript path unless the agents/session owner explicitly approves broader same-process write trust, and require an external-during-lock regression plus current-head proof before merging this approach.

## Reproduction Assessment

Yes at source level: current main throws on a fingerprint mismatch after releaseForPrompt, and the PR body plus linked issue exercise that false-takeover path. I did not run tests because this review is read-only.

## Solution Assessment

No as-is: the patch is directionally relevant, but the broad trusted-fingerprint map can launder an external edit that occurs during a locked callback. A narrower ownership seam or an added fail-closed regression is safer.

- **[P1] Keep external edits from becoming owned fingerprints:** `src/agents/pi-embedded-runner/run/attempt.session-lock.ts:384-387`
  - body: This records the post-run fingerprint as OpenClaw-owned whenever the pre-run fingerprint was trusted.
