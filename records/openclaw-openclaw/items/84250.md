## Summary

Keep open: this PR targets the right P1 embedded session-lock race, but the latest branch can still mask an unowned session-file edit when a later same-process locked write publishes the combined file state as trusted.

## What This Changes

The PR adds a process-local owned-fingerprint map around embedded attempt session locks plus tests for same-process prompt-window writes and an external-edit case.

## Best Possible Solution

Keep the ownership-aware tolerance, but only publish a new owned fingerprint when the pre-write fingerprint is already trusted by this process, and add a regression where external edit plus later locked write still fails closed.

## Reproduction Assessment

Yes for the reported bug at source level: current main rejects any fingerprint change after releaseForPrompt, and the linked issue plus PR body show the same error during channel activity. The merge-blocking bypass is also source-reproducible from the PR branch's publish-and-accept logic, though I did not run tests in this read-only pass.

## Solution Assessment

No. The PR is close, but the current implementation trusts the post-write fingerprint without proving the pre-write state was already process-owned; the safer fix is to preserve the same-process allowance while keeping unowned intermediate edits fail-closed.
