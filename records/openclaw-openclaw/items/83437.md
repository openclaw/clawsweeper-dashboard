## Summary

This should stay open: current main still prepends memory-lancedb recall results into the model prompt as plain hook context, while the direct LanceDB record/search path remains unscoped by agent or session. The prior hardening adds escaping and warning text, but it does not enforce a data-only boundary or solve the reported cross-agent memory leak.

## What This Changes

The issue asks OpenClaw to prevent memory-lancedb recalled historical memory from being treated as current task instructions or leaking across agents.

## Best Possible Solution

Define one memory boundary for memory-lancedb: scoped, data-only recall for direct autoRecall or a migration to the active-memory-style scoped path, with regression tests and an explicit compatibility story.

## Reproduction Assessment

No high-confidence live cron/model reproduction was run. Source inspection does provide a clear reproduction path: enable memory-lancedb autoRecall with a shared dbPath, store relevant memory in one scope, and observe current main recalling it through prependContext without an agent/session filter.

## Solution Assessment

Unclear. The report identifies the right boundary, but warning text alone is not enough; maintainers need to choose scoped direct recall, active-memory routing, or another compatibility-preserving data-only contract.
