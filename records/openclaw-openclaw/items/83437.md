## Summary

Keep open: current main and v2026.5.18 still route memory-lancedb auto-recall through prompt text with warning/escaping only, and the direct LanceDB row/search path remains unscoped. The report is security/session-state relevant, but the fix needs maintainer choice on memory scoping, compatibility, and prompt/data boundaries.

## What This Changes

The issue requests safer memory-lancedb relevant-memory handling so recalled memories are data-only, scoped to the right agent/session, and cannot expand task scope as instructions.

## Best Possible Solution

Keep this open as the concrete memory-lancedb safety bug, then land a maintainer-approved boundary that scopes direct recall and treats recalled memory as data rather than task instructions, coordinated with the open partition and injected-context registry work.

## Reproduction Assessment

No high-confidence live reproduction of the exact cron outcome was run. Source inspection does show the current-main mechanism: memory-lancedb auto-recall searches one unscoped table and prepends recalled text into the model prompt.

## Solution Assessment

Unclear. Warning text and escaping are useful mitigations but not the best final boundary; maintainers should choose scoped recall, data-only prompt placement, and upgrade semantics before a safe fix lands.
