## Label Justifications

- `P1`: The report concerns a current agent/session boundary problem where recalled memory can influence tool-enabled workflows beyond the requested task.
- `impact:security`: Untrusted historical memory is reaching a model prompt surface that can affect command and tool behavior.
- `impact:session-state`: The issue specifically reports memory from one agent/session being attached to another agent's task context.

## Summary

This should stay open: current main still lets memory-lancedb auto-recall put recalled historical memory onto the prompt text surface, and the reported agent/session safety boundary needs maintainer product and security review rather than cleanup closure.

## What This Changes

The issue asks for memory-lancedb relevant memory to be treated as delimited historical data instead of executable task instructions, and for one agent's memory not to be supplied automatically to another agent's task context.

## Best Possible Solution

Define an approved memory boundary that makes automatic recall agent/session-scoped and data-only, or route memory-lancedb recall through the scoped active-memory path with explicit compatibility coverage.

## Reproduction Assessment

No high-confidence live reproduction was run for the cron and cross-agent examples. Source inspection does show the reported mechanism: direct memory-lancedb auto-recall can prepend recalled memory into the model prompt without a hard agent/session data boundary.

## Solution Assessment

Unclear. Warning text and escaping are useful mitigations, but the best fix needs a maintainer decision between scoped direct recall, active-memory routing, or another compatibility-safe data/instruction boundary.
