## Summary

Keep open: current main has the merged manual `memory_recall` output guard, but the automatic memory path still recalls and captures through the shared memory-lancedb store, so the reported agent/session leakage remains. The direct scoped-memory fix attempt was closed unmerged after review found storage-management, proof, and security/product-decision blockers.

## What This Changes

The issue reports relevant memory from memory-lancedb being treated as task authority and leaking across agent/session scope.

## Best Possible Solution

Keep this issue open until maintainers choose and land a scoped, data-only automatic memory design that preserves memory management/deletion semantics and has real setup proof.
