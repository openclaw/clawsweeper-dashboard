## Summary

Keep this open: current main and the latest release still leave memory-lancedb automatic recall/capture on a shared LanceDB store, so the central cross-agent/session leakage is not implemented away. The merged related PR hardens manual `memory_recall` presentation only, while the scoped-store fix attempt was closed unmerged for product/security and proof gaps.

## What This Changes

The issue asks for memory-lancedb recalled memory to remain data-only and scoped so historical memory cannot become new task instructions or leak between agents/sessions.

## Best Possible Solution

Keep this issue open for a maintainer/security-approved memory-lancedb fix that makes automatic recall/capture agent/session-scoped, keeps dynamic recall data-only, preserves management/removal or migration semantics, and includes real setup proof.

## Reproduction Assessment

Yes, at source level: current main auto-recall searches the shared memory-lancedb `db`, auto-capture writes to the same `db`, and runner code prepends hook context into the prompt. I did not establish a live Gateway/provider reproduction.

## Solution Assessment

No current implementation fully solves it. The best path is a maintainer-approved scoped automatic memory design with documented migration/removal semantics rather than only presentation hardening or an unmanaged hidden scoped store.
