## Summary

Keep open. Current main and the latest shipped release still route memory-lancedb auto-recall through plain prompt context, and the direct LanceDB record/search path still has no agent or session ownership boundary; related work exists, but it is not merged and does not yet settle the data-vs-instruction boundary.

## What This Changes

Reports that memory-lancedb relevant memory can be treated as task instructions and can leak across agent or cron scopes.

## Best Possible Solution

Define a maintainer-approved memory boundary: scope memory-lancedb storage/search/auto-recall by a trusted identity and make recalled memory a data-only injected context with migration, docs, regression tests, and real two-agent or cron proof.

## Reproduction Assessment

No high-confidence live reproduction of the exact cron command expansion was run. Source inspection does reproduce the mechanism: unscoped memory-lancedb recall is returned as prependContext and prepended to the next prompt.

## Solution Assessment

Unclear. The report targets the right boundary, but warning text alone is not enough; maintainers should choose a scoped recall and injected-context contract before implementation.
