## Summary

Keep open: current main and v2026.6.10 still use one shared memory-lancedb store for automatic recall/capture and prepend recalled memory into the model prompt as dynamic context. The merged https://github.com/openclaw/openclaw/pull/91425 only hardens manual memory_recall presentation, while the direct scoped-store attempt at https://github.com/openclaw/openclaw/pull/93371 closed unmerged.

## What This Changes

The issue asks for memory-lancedb relevant memories to stay data-only and scoped so historical memory cannot expand the task or leak between agents.

## Best Possible Solution

Keep this issue as the canonical memory-lancedb automatic recall/capture safety tracker until maintainers approve a scoped, data-only design with legacy-memory handling and live proof.

## Reproduction Assessment

Yes, at source level: current main auto-recall searches the shared memory-lancedb DB, auto-capture writes to the same DB, and hook prependContext is composed into the model prompt. I did not run a live two-agent or cron reproduction in this read-only review.

## Solution Assessment

No current implementation fully solves it. The best path is a maintainer/security-approved scoped automatic memory design with data-only prompt treatment and explicit legacy-memory semantics, not only manual recall presentation hardening.
