## Summary

Keep open: current main and the latest release still let memory-lancedb automatic recall search the shared LanceDB store and inject recalled memory into prompt context; the merged related PR hardened manual recall only, and the scoped automatic-memory PR closed unmerged.

## What This Changes

The issue asks OpenClaw to keep recalled memory data from being treated as task instructions and to prevent one agent/session's memory from being automatically supplied to another agent/session.

## Best Possible Solution

Define and land a maintainer-approved memory boundary in the memory-lancedb plugin: automatic recall/capture should use scoped, manageable memory state and keep recalled content data-only, with tests/docs and live proof for the chosen upgrade behavior.

## Reproduction Assessment

Yes, at source level: current main's automatic memory hook searches the shared LanceDB store and returns recalled text as prependContext, and both agent runners prepend that context to the prompt. I did not run the exact cron or cross-agent live model scenario.

## Solution Assessment

Unclear until maintainers choose the memory-boundary policy. The narrow manual recall hardening is not enough; the best fix likely combines scoped automatic storage/recall, manageable deletion/listing semantics, and live proof.
