## Label Justifications

- `P1`: The report describes a safety/session-boundary problem where recalled memory can influence tool-enabled agent behavior outside the requested task.
- `impact:session-state`: The issue is about memory and agent context crossing the intended agent or session scope.
- `impact:security`: The issue concerns untrusted historical content reaching an instruction-capable prompt surface for tool-enabled agents.

## Summary

Keep open. Current main still has the reported source-level mechanism: direct memory-lancedb auto-recall searches an unscoped LanceDB table and prepends recalled memory into the turn as prompt text, with warning/escaping mitigations but no hard data-only or agent-scope boundary.

## What This Changes

The issue requests a safety boundary so memory-lancedb relevant memories cannot be treated as task instructions or leak across agents.

## Best Possible Solution

Define a maintainer-approved memory boundary: scope direct memory-lancedb storage/search/recall to trusted agent/session/user identity and deliver recalled memory through an enforced data-only prompt surface with compatibility coverage.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the reported mechanism: direct memory-lancedb auto-recall searches unscoped rows and returns them as prependContext that the runner prepends to the model prompt.

## Solution Assessment

Unclear until maintainer review. Warning text and escaping are useful mitigations, but the maintainable fix is structural scoping plus an enforced data boundary, likely coordinated with the registry and partition issues.
