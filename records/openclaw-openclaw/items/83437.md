## Label Justifications

- `P1`: The issue reports a high-priority memory/session boundary problem where historical recalled data can affect tool-enabled agent behavior across task scope.
- `impact:session-state`: The reported behavior is about recalled memory from one agent/session becoming context for another agent or task.
- `impact:security`: Historical memory can contain instruction-like or sensitive content and is injected into a model-visible prompt path.

## Summary

Keep open: current main and the latest release only cover part of the report by guarding manual `memory_recall` output, while automatic memory recall/capture still uses the shared LanceDB store and can inject remembered data into prompt context without an agent/session-scoped source boundary.

## What This Changes

The issue asks memory-lancedb recalled memory to remain untrusted data and be scoped to the current agent/session so historical memory cannot expand a task or leak across agents.

## Best Possible Solution

Keep this open until memory-lancedb has a plugin-local, data-only automatic memory boundary that scopes recall/capture to the current agent/session while preserving explicit memory management, cleanup, and upgrade semantics.

## Reproduction Assessment

Yes, from source inspection. Current main’s automatic recall/capture uses the shared memory-lancedb `db`, and the existing auto-recall test proves a hook call with no scope still returns remembered `prependContext`; I did not run a live gateway/provider reproduction.

## Solution Assessment

No implemented solution is complete yet. The merged manual recall hardening is useful, but the best fix still needs scoped automatic memory plus managed removal/migration semantics rather than a hidden scoped store or warning-only approach.
