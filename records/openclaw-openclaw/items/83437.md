## Label Justifications

- `P1`: The report describes a safety-sensitive agent/channel workflow where recalled historical memory can influence tool-enabled work outside the requested task.
- `impact:session-state`: The affected behavior is about memory/context from one session or agent being surfaced in another agent turn.
- `impact:security`: Untrusted recalled memory can reach a tool-enabled model prompt and influence actions across the intended instruction boundary.

## Summary

Keep open. Current main and v2026.5.12 still route direct `memory-lancedb` auto-recall into the model prompt as `prependContext`; the warning/escaping mitigations do not create an enforced agent/session scope or data-only boundary, and related open issues cover adjacent surfaces rather than superseding this concrete LanceDB path.

## What This Changes

The issue reports that `memory-lancedb` relevant-memory recall can be treated as task instructions and can leak one agent's memory into another agent's work.

## Best Possible Solution

Define a maintainer-approved LanceDB memory boundary: scope direct auto-recall by agent/session or route automatic recall through Active Memory's scoped data-only path, with compatibility coverage for existing `autoRecall` users.

## Reproduction Assessment

No live high-confidence reproduction of command execution was run. Source inspection is high-confidence for the mechanism: direct `memory-lancedb` recall returns untrusted memory as `prependContext`, and the direct record/search path does not carry agent or session ownership.

## Solution Assessment

Unclear as a final implementation. The report points at the right boundary, but warning text alone is not enough; maintainers should choose scoped recall or an Active Memory based path with compatibility tests.
