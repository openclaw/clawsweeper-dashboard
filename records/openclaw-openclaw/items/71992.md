## Summary

Keep open. Current main still appends terminal assistant messages before clearing live stream state, while the renderer can display history plus the live stream; open PR #72004 already targets this issue with closing syntax, so this should stay open until that PR or an equivalent patch lands.

## What This Changes

The issue requests that Control UI WebChat clear live stream/run state before appending final or aborted assistant messages so replies are not rendered from both history and stream state.

## Best Possible Solution

Review and land #72004, or an equivalent narrow Control UI patch, with regression coverage that observes stream state at the moment terminal messages are appended; track backend transcript dedupe separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active Control UI chat run with non-null chatStream receiving a final or aborted event; current main mutates chatMessages before clearing the stream while rendering can include both history and live stream.

## Solution Assessment

Yes for the narrow client-render race. Clearing or snapshotting stream/run state before mutating chatMessages is the smallest maintainable fix because it preserves final payload and streamed fallback behavior without turning this into broad transcript dedupe.
