## Summary

Keep open. Current main still has the Control UI terminal-event ordering race, and open PR #72004 already references this issue with closing syntax, so the issue should stay open until that PR or an equivalent patch lands.

## What This Changes

The issue asks OpenClaw to stop Control UI WebChat from rendering assistant replies twice by clearing live stream state before appending terminal assistant messages.

## Best Possible Solution

Land #72004 or an equivalent narrow Control UI patch with regression coverage that observes stream state when `chatMessages` mutates, while keeping backend transcript dedupe as separate follow-up if still reproduced.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active Control UI chat run with non-null `chatStream` receiving a `final` or `aborted` event; current main writes `chatMessages` before nulling the stream while the renderer can display both history and live stream.

## Solution Assessment

Yes for the narrow client-render race. Reordering terminal event mutations is the smallest maintainable fix because it preserves final payload and streamed fallback behavior without turning this into broad backend transcript dedupe.
