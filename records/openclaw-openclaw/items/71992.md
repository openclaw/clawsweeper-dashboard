## Summary

Keep open. Current main still has the Control UI terminal-event ordering reported here: active-run `final` and `aborted` handlers append to `chatMessages` before clearing `chatStream`, while the renderer still receives committed messages and live stream text as separate inputs. The focused approved fix in https://github.com/openclaw/openclaw/pull/72004 was closed unmerged, and the broader persisted-transcript duplication umbrella should remain separate from this WebChat render race.

## What This Changes

The issue asks OpenClaw to stop duplicate Control UI webchat assistant replies by clearing live stream/run state before committing terminal assistant messages.

## Best Possible Solution

Land a narrow controller/test fix equivalent to the reviewed PR: snapshot fallback stream text, clear live stream/run state before terminal message commits, and leave backend transcript dedupe to the umbrella tracker.

## Reproduction Assessment

Yes, source-level. Dispatch an active-run `final` or `aborted` event while `chatStream` is non-null; current main mutates `chatMessages` before clearing `chatStream`, and the render path can receive both committed history and a live stream item.

## Solution Assessment

Yes. Reordering the existing state mutations after snapshotting fallback stream text is the narrowest maintainable fix; backend transcript dedupe and tool-stream artifact cleanup should remain separate work.
