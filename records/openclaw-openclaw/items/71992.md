## Summary

Keep open. Current main still has the Control UI terminal-event ordering race, and open PR #72004 is already the focused closing implementation candidate for this issue.

## What This Changes

The issue asks for Control UI WebChat to render each assistant reply once by clearing active stream state before appending terminal assistant messages.

## Best Possible Solution

Review and land #72004, or an equivalent narrow controller/test patch, while tracking backend transcript-duplication reports separately if they still reproduce.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active Control UI chat run with non-null `chatStream` receiving an owned `final` or `aborted` event; current main writes `chatMessages` before clearing the stream while the renderer can show both history and live stream.

## Solution Assessment

Yes for the client-render race. Reordering terminal-event state updates is the narrow maintainable fix; broad backend history dedupe from the comments should not be mixed into this client patch.
