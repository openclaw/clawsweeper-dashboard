## Summary

Keep open. Current main and the latest release still append final/aborted assistant messages to `chatMessages` before clearing the live stream state, while the renderer still displays a non-null stream separately; the directly linked fix PR is open and unmerged, so this issue should stay open until that PR or an equivalent narrow fix lands.

## What This Changes

The issue asks for Control UI/WebChat to render each assistant turn once by clearing `chatStream`, `chatRunId`, and `chatStreamStartedAt` before updating `chatMessages` in final and aborted event handlers.

## Best Possible Solution

Land PR #72004 or an equivalent narrow Control UI change that snapshots fallback stream text, clears stream/run state before mutating `chatMessages` in both final and aborted paths, and leaves backend transcript dedupe to the existing broader duplicate-transcript tracker.

## Reproduction Assessment

Yes. The issue has concrete cross-platform reproduction reports, and current source gives a focused reproduction path: process an active-run final or aborted event while `chatStream` is populated, then render can observe both the committed assistant message and the still-populated stream item.

## Solution Assessment

Yes for the client-side duplicate-render race. Clearing live stream state before appending the final or aborted assistant message is the narrowest maintainable fix, provided the streamed fallback text is captured first; backend transcript dedupe should remain separate.
