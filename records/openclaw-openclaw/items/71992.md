## Summary

Keep open. Current main still has the terminal Control UI chat event ordering that can render committed assistant history and the active stream together, and an open closing PR already targets this narrow fix, so the issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue requests a Control UI WebChat fix so terminal `final` and `aborted` events clear active stream state before appending assistant messages, preventing duplicate visible replies.

## Best Possible Solution

Review and land #72004, or an equivalent narrow Control UI patch, so terminal `final` and `aborted` events snapshot any needed fallback stream text, clear active stream/run state first, then append assistant messages with regression coverage for mutation order. Keep broader transcript-deduplication behavior in its own backend/history thread.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active Control UI run with non-null `chatStream`, followed by a `final` or `aborted` event; current main updates `chatMessages` while the stream is still non-null, and `buildChatItems` can render both.

## Solution Assessment

Yes for the narrow client-render race. Reordering terminal event mutations is the smallest maintainable fix because it preserves final payload and streamed fallback behavior while removing the intermediate duplicate-render state.
