## Summary

Keep open. Current main and the latest release still mutate `chatMessages` before clearing `chatStream` in terminal WebChat events, while the only focused fix PR was closed unmerged; the broader duplicate-transcript umbrella is related but does not replace this narrow client render race.

## What This Changes

The issue requests preventing Control UI WebChat from showing duplicate assistant replies by clearing live stream/run state before committing terminal assistant messages.

## Best Possible Solution

Land a narrow current-main replacement for the closed focused PR: snapshot fallback stream text, clear `chatStream`/`chatRunId`/`chatStreamStartedAt`, then append terminal assistant messages in `final` and `aborted`; leave persisted transcript dedupe to the umbrella tracker.

## Reproduction Assessment

Yes, at source level: construct a `ChatState` with a matching `chatRunId` and non-null `chatStream`, then dispatch an owned `final` or `aborted` event. Current main appends to `chatMessages` before clearing the stream while the renderer consumes committed messages and live stream separately.

## Solution Assessment

Yes. The focused ordering fix from the closed PR is the narrowest maintainable solution for this client race because it preserves fallback semantics and avoids mixing in backend transcript dedupe work.
