## Summary

Keep open. Current main still has the Control UI terminal-event ordering described in this issue: active `final` and `aborted` events append to `chatMessages` before clearing `chatStream`, while the renderer still consumes committed messages and live stream text separately. The focused approved fix at https://github.com/openclaw/openclaw/pull/72004 was closed without merging, and the broader persisted-transcript duplication discussion should stay separate.

## What This Changes

The issue requests a narrow Control UI WebChat fix so assistant replies render once by clearing live stream/run state before committing terminal assistant messages.

## Best Possible Solution

Land a narrow controller/test fix that snapshots fallback stream text, clears live stream/run state before terminal message commits, and leaves persisted transcript dedupe to the umbrella tracker.

## Reproduction Assessment

Yes, source-level: dispatch an active-run `final` or `aborted` event while `chatStream` is non-null. Current main mutates `chatMessages` before clearing `chatStream`, and the render path can receive both committed history and a live stream item.

## Solution Assessment

Yes. Reordering the existing state mutations after snapshotting fallback stream text is the narrowest maintainable fix; backend transcript dedupe and tool-stream artifact cleanup should remain separate work.
