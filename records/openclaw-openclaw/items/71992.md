## Summary

Keep open. Current main and the latest release still have the Control UI terminal-event ordering that can expose both the committed assistant history item and the live stream item, and the provided GitHub context shows PR #72004 is still the direct open closing implementation candidate.

## What This Changes

The issue requests a Control UI WebChat fix so assistant replies render exactly once by clearing live stream state before committing final or aborted messages.

## Best Possible Solution

Land PR #72004 or an equivalent narrow Control UI change that snapshots fallback stream text, clears chatStream/chatRunId/chatStreamStartedAt before mutating chatMessages in both final and aborted paths, and leaves backend transcript dedupe to the separate umbrella work.

## Reproduction Assessment

Yes. The issue has concrete cross-platform reproduction reports, and current source gives a focused path: process an active-run final or aborted event while chatStream is populated, then renderChat can receive both the committed history item and the still-populated stream item.

## Solution Assessment

Yes for the client-side duplicate-render race. Clearing stream state before appending the final or aborted assistant message is the narrowest maintainable fix; backend transcript dedupe is a separate concern.
