## Summary

Keep open. Current main still commits terminal assistant messages to `chatMessages` before clearing the active `chatStream`, and the renderer can display committed history plus the live stream in the same render. The provided GitHub context also shows open, unmerged PR #72004 with closing syntax for this issue, so the issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue requests a Control UI WebChat fix so each assistant reply renders once by clearing active stream/run state before committing final or aborted assistant messages.

## Best Possible Solution

Review and land #72004, or an equivalent narrow Control UI patch, so terminal `final` and `aborted` events snapshot any fallback stream text, clear active stream/run state first, and only then append assistant messages with ordering-focused regression coverage. Keep broader duplicate transcript persistence under a separate backend issue.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active run with non-null `chatStream`, then a `final` or `aborted` event; current main mutates `chatMessages` while `chatStream` is still non-null, and `buildChatItems` can render both history and the live stream.

## Solution Assessment

Yes for the narrow client-render race. Reordering terminal event mutations is the smallest maintainable fix because it preserves final state and streamed fallback behavior while avoiding broader backend transcript dedupe changes.
