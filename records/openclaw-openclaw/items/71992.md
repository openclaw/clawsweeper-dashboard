## Summary

Keep open: current main and the latest stable release still have the terminal-event ordering that can expose both the live stream and the committed assistant message, while the focused fix attempts were closed unmerged and later duplicate reports point back here.

## What This Changes

The issue asks to stop Control UI WebChat from rendering each assistant reply twice by clearing live stream/run state before committing final or aborted assistant messages.

## Best Possible Solution

Land a focused WebChat controller/test fix that snapshots fallback stream text, clears active stream/run state before terminal assistant-message commits, and keeps broader transcript idempotency work on the existing cross-channel trackers.

## Reproduction Assessment

Yes, source-level: dispatch an active-run final or aborted event while chatStream is non-null, and current main writes chatMessages before clearing the stream while renderChat can receive both. I did not run live browser proof in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to snapshot any stream fallback, clear chatStream/chatRunId/chatStreamStartedAt first, then append the terminal assistant message with an order-sensitive regression test; backend transcript dedupe belongs to the separate duplicate-delivery trackers.
