## Summary

Keep open. Current main still has the Control UI state-ordering pattern described in the report, and an open PR directly references this issue with closing syntax, so the issue should stay open until that PR or an equivalent fix lands. The discussion also shows a broader persisted-transcript duplicate variant that should remain coordinated with the existing umbrella rather than being hidden by a narrow client-render fix.

## What This Changes

The issue requests that Control UI WebChat render each assistant turn once by clearing live stream state before committing final or aborted assistant messages.

## Best Possible Solution

Review and land PR #72004, or an equivalent clean patch, that clears `chatStream`, `chatRunId`, and `chatStreamStartedAt` before any final or aborted `chatMessages` mutation, with the ordering-invariant test and changelog entry. Keep broader persisted transcript dedupe work in #69208.

## Reproduction Assessment

Yes. The issue has concrete Control UI steps and multiple independent version/platform reports, and current main statically reproduces the vulnerable state transition: a final/aborted handler can write `chatMessages` while `chatStream` is still non-null.

## Solution Assessment

Yes for the client-side duplicate-render race. The narrow state-ordering fix in the linked PR is more maintainable than adding backend history dedupe as a workaround, while server-side duplicate transcript variants should remain separate umbrella work.
