## Summary

Keep open. Current main still commits terminal assistant messages before clearing active stream state, and open PR #72004 uses closing syntax with the narrow fix, so the issue should stay open until that PR or an equivalent patch lands.

## What This Changes

The issue asks for Control UI webchat to render each assistant reply once by clearing `chatStream` and active run state before writing terminal assistant messages into `chatMessages`.

## Best Possible Solution

Review and land #72004, or an equivalent narrow controller patch, so terminal `final` and `aborted` events snapshot fallback text, clear active stream/run state first, and then commit assistant messages with regression coverage. Keep broader duplicate transcript persistence under #69208 or narrower follow-ups.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start with an active run and non-null `chatStream`, dispatch a `final` or `aborted` event, and observe current main writes `chatMessages` while `chatStream` is still non-null; `buildChatItems` can then render both.

## Solution Assessment

Yes for the narrow client-render race. Reordering the terminal event mutations preserves the final state and streamed fallback behavior while avoiding broader backend transcript dedupe changes that belong to separate issues.
