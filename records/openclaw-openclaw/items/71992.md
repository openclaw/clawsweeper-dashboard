## Summary

Keep open. Current main still has the client-side Control UI race: active-run terminal events append assistant content to `chatMessages` before clearing `chatStream`, while rendering can include both history and live stream; open PR #72004 already targets this issue with closing syntax.

## What This Changes

The issue reports duplicate Control UI webchat assistant replies and requests clearing `chatStream`/run state before appending terminal assistant messages.

## Best Possible Solution

Review and land #72004, or an equivalent narrow Control UI patch, with regression coverage that observes `chatStream` at the moment terminal assistant content is appended.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active Control UI chat run receiving a `final` or `aborted` event while `chatStream` is non-null; current main mutates `chatMessages` before clearing the live stream that the renderer also displays.

## Solution Assessment

Yes for the narrow client-render race. Clearing or snapshotting stream/run state before appending terminal assistant content is the smallest maintainable fix, while backend transcript dedupe should remain separate.
