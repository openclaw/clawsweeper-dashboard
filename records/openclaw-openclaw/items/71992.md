## Summary

Keep open. Current main still has the Control UI terminal-event ordering described by the report, and an open unmerged PR already uses closing syntax for this issue, so the correct cleanup path is to review and land or close that PR rather than close the issue early.

## What This Changes

Fix Control UI WebChat so each assistant reply is rendered once by clearing live stream state before committing terminal assistant messages, while separating this client-render race from broader transcript-duplication reports.

## Best Possible Solution

Review and land #72004, or an equivalent narrow PR that snapshots any fallback stream text, clears `chatStream`/`chatRunId`/`chatStreamStartedAt` before mutating `chatMessages` in both `final` and `aborted`, and adds regression coverage for the intermediate ordering. Keep broader duplicate transcript persistence under #69208 or narrower follow-ups.

## Reproduction Assessment

Yes. The report has concrete Control UI `/chat` steps plus independent macOS, Windows, and release-version confirmations; current source also shows the exact state shape where history and active stream can coexist during terminal event handling. I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes for the narrow client-render race: clearing active stream state before committing the terminal assistant message is the smallest maintainable fix and preserves the same final state. It should be reviewed separately from backend transcript-deduplication proposals, which are broader and risk hiding distinct root causes.
