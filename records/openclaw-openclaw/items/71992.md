## Summary

Keep open. Current main and the latest release still mutate `chatMessages` before clearing the active Control UI stream in the terminal `final` and `aborted` handlers, while the renderer displays history and active stream independently. The issue is also paired with open closing PR #72004, so the right cleanup path is to review and land or close that PR rather than close this issue early.

## What This Changes

The issue asks for Control UI WebChat to render each assistant terminal reply once by clearing active stream state before appending final or aborted assistant messages.

## Best Possible Solution

Land #72004 or an equivalent narrow Control UI patch that snapshots any fallback stream text, clears `chatStream`/`chatRunId`/`chatStreamStartedAt` before writing terminal assistant messages, and adds an ordering regression test. Keep backend transcript deduplication or delivery-mirror leakage under #69208 or narrower follow-ups.

## Reproduction Assessment

Yes. The issue has concrete `/chat` steps plus independent macOS, Windows, and release-version confirmations; current source also shows the intermediate state where committed history and the active stream can coexist. I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes for the narrow client-rendering race: clearing active stream state before committing the terminal assistant message is the smallest maintainable fix and preserves the same final state. Broad backend export filtering would be a separate mitigation and could hide distinct transcript-persistence bugs.
