## Summary

Keep open. Current main still has the Control UI terminal-event ordering race, and open PR #72004 already uses closing syntax for this issue, so the issue should stay open until that PR or an equivalent patch lands.

## What This Changes

The issue asks Control UI WebChat to stop rendering assistant replies twice by clearing live stream state before committing final or aborted assistant messages.

## Best Possible Solution

Review and land #72004, or an equivalent narrow Control UI patch, so terminal `final` and `aborted` events snapshot fallback text, clear active stream/run state first, and then append assistant messages with ordering-focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active chat run with non-null `chatStream` receiving a `final` or `aborted` event; current main writes `chatMessages` while `chatStream` is still non-null, and the renderer can show both history and live stream.

## Solution Assessment

Yes for the narrow client-render race. Reordering terminal event mutations is the smallest maintainable fix because it preserves final payload and streamed fallback behavior while avoiding broad backend transcript dedupe changes.
