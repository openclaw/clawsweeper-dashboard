## Summary

Keep open. Current main has a generic cron direct-delivery path that should only mark announce delivery as delivered after outbound send results are returned, but the report targets the external WeChat plugin path and there is not enough current-main live proof to show the no-outbound `delivered=true` failure is fixed or already owned by another canonical item.

## What This Changes

Cron isolated-session announce delivery to `openclaw-weixin` should send the final response to WeChat or record the run as not delivered.

## Best Possible Solution

Verify one affected WeChat cron run against current OpenClaw plus the current `@tencent-weixin/openclaw-weixin` plugin, then fix the generic core result-validation path or the plugin adapter so no outbound send can be recorded as delivered.

## Reproduction Assessment

We do not have a high-confidence live current-main reproduction. The reporter provided concrete logs and the current source path is clear, but the missing proof is a current WeChat cron run that compares the plugin send result with actual gateway outbound delivery.

## Solution Assessment

Unclear until live/plugin proof identifies the boundary. The best fix should not special-case WeChat in core; it should harden generic cron/outbound result validation or repair the external plugin adapter contract.
