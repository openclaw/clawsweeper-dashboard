## Summary

The remaining false-success behavior belongs to the external `openclaw-weixin` plugin, not OpenClaw core. Affected OpenClaw releases already tracked best-effort payload errors, while the plugin’s send path treats an HTTP success plus a locally generated client ID as a successful receipt without confirming downstream delivery; the channel-owner repository already has a canonical open report.

## What This Changes

Investigate cron announce runs that record successful delivery even when the WeChat plugin send path fails under best-effort delivery.

## Best Possible Solution

Keep OpenClaw’s existing generic failure accounting and resolve runtime liveness or false-success receipts in https://github.com/Tencent/openclaw-weixin/issues/191.

## Reproduction Assessment

Yes at the contract level: the plugin can return success from an HTTP 2xx and a locally generated client ID without confirming recipient delivery. The exact idle-runtime trigger on the latest plugin version still needs live verification.

## Solution Assessment

No, another OpenClaw core accounting change is not the best solution because affected core versions already contained the generic failure tracking; the WeChat adapter must return an honest failure or maintain a live runtime.
