## Summary

Keep open: the report describes a plausible silent-success cron failure, and current main still separates main-session systemEvent reminders from isolated agentTurn command execution. The next step needs maintainer judgment on whether main-session shell-like system events should fail, warn, or be migrated to isolated cron jobs.

## What This Changes

The issue asks cron to stop reporting success for main-session systemEvent wake-now jobs when the intended shell command is not actually executed.

## Best Possible Solution

Keep command-style scheduled work on isolated agentTurn cron jobs, and make main-session systemEvent jobs either prove heartbeat execution before ok or record skipped/error with migration guidance when they cannot execute the requested work.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection shows the relevant path can mark main-session systemEvent cron work ok after queueing or heartbeat handling, while direct command execution is implemented for isolated agentTurn jobs.

## Solution Assessment

Unclear as a final product choice: creating short-lived main sessions would change cron semantics, while fail/skip diagnostics plus migration guidance is safer. Maintainers should decide whether main systemEvent remains reminder-only or gains direct command semantics.
