## Summary

Keep open. Current main still sends session-memory slug generation through the shared embedded runner without `authProfileFailurePolicy: "local"`, so a billing-classified helper failure can still update shared auth-profile state and disable normal turns on the same profile.

## What This Changes

The issue asks for slug-generator failures to stay helper-local so a billing-classified HTTP 400 from the internal session-memory lane cannot put the shared claude-cli auth profile into a five-hour billing cooldown.

## Best Possible Solution

Add `authProfileFailurePolicy: "local"` to the slug-generator embedded run and lock it with a focused regression test. Keep the global Anthropic extra-usage billing classifier intact for normal user-facing fallback, and leave broader dedicated slug-model or lane-congestion behavior to #33962.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by sending an Anthropic extra-usage or other billing-classified prompt failure through `generateSlugViaLLM`; current main lets that helper call reach shared auth-profile failure persistence because no local policy is passed.

## Solution Assessment

Yes. The existing local failure-policy seam is the narrowest maintainable fix because it scopes internal helper failures without changing global provider billing/fallback semantics.
