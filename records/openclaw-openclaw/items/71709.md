## Summary

Keep open. Current main still lets the session-memory slug helper use shared auth-profile failure state: the helper calls the embedded runner without the existing local failure policy, while the runner can persist billing-classified failures and the reported Anthropic extra-usage text still classifies as billing. This is a focused runtime bug with a narrow existing repair seam, not stale cleanup.

## What This Changes

The issue requests that session-memory LLM slug-generation failures stay helper-local instead of disabling the shared auth profile with a billing cooldown.

## Best Possible Solution

Keep global billing classification for real user-facing provider failures, but pass the existing helper-local auth-profile failure policy from the session-memory slug generator and add regression coverage proving billing-classified slug failures do not write shared auth cooldown state.

## Reproduction Assessment

Yes. The issue includes production logs, and current main gives a high-confidence source-level reproduction path: the default session-memory hook calls `generateSlugViaLLM`, which invokes `runEmbeddedPiAgent` without local auth-profile failure policy while the exact provider wording classifies as billing.

## Solution Assessment

Yes. The narrowest maintainable fix is to scope the slug-helper embedded run with `authProfileFailurePolicy: "local"`; broadening or rolling back the billing classifier would risk breaking legitimate same-turn fallback behavior covered by existing tests.
