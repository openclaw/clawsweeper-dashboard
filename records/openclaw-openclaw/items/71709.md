## Summary

Keep open. Current main mitigates default installs by making LLM slug generation opt-in, but the opt-in slug helper still invokes the embedded runner without helper-local auth failure policy, while the reported Anthropic extra-usage text still classifies as billing and can persist a profile-wide cooldown.

## What This Changes

The issue asks for slug-generator failures, especially Anthropic HTTP 400 extra-usage responses, to stay helper-local instead of disabling the shared auth profile for all agents.

## Best Possible Solution

Pass `authProfileFailurePolicy: "local"` from `generateSlugViaLLM` and add focused regression coverage, while preserving billing classification and shared cooldown behavior for normal user-facing runs.

## Reproduction Assessment

Yes, at source level. With `llmSlug: true`, current main calls `runEmbeddedPiAgent` on `temp:slug-generator` without local auth policy; the reported extra-usage text classifies as `billing`, and billing writes the long profile disable window.

## Solution Assessment

Yes. Reusing the existing `authProfileFailurePolicy: "local"` seam in the slug helper is the narrowest maintainable fix and avoids weakening provider billing failover globally.
