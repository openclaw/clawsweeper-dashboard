## Summary

Keep open. Current main still leaves the session-memory slug helper on the shared embedded-run auth failure path: the helper does not pass the existing local failure policy, while the reported Anthropic extra-usage wording still classifies as billing and billing still writes a long profile-wide disabled state.

## What This Changes

The issue asks for failures from the internal session-memory slug generator to stay lane-local instead of creating a profile-wide billing cooldown that blocks other agents using the same auth profile.

## Best Possible Solution

Keep global billing classification for real user-facing provider failures, but make the session-memory slug generator opt into the existing helper-local auth-profile failure policy and add regression coverage proving billing-classified slug failures do not write shared cooldown state.

## Reproduction Assessment

Yes. The issue has production logs, and current main provides a high-confidence source-level reproduction path: default session-memory LLM slugging calls the embedded runner without local auth failure policy, the reported text classifies as billing, and billing writes a five-hour disabled window.

## Solution Assessment

Yes. Passing `authProfileFailurePolicy: "local"` from `generateSlugViaLLM` is the narrowest maintainable fix; weakening the Anthropic extra-usage billing classifier globally would risk regressing intentional model-fallback behavior.

In `src/hooks/llm-slug-generator.ts`, make the `runEmbeddedPiAgent` call for `sessionKey: "temp:slug-generator"` use the existing helper-local auth-profile failure policy, most likely `authProfileFailurePolicy: "local"`. Add a user-facing `CHANGELOG.md` fix entry crediting the reporter if a code fix lands. Validate with targeted tests, targeted formatter, and the changed gate per OpenClaw policy.
