## Summary

Keep open. Current main still leaves the session-memory slug helper on shared auth-profile failure state: `generateSlugViaLLM` calls `runEmbeddedPiAgent` for `temp:slug-generator` without `authProfileFailurePolicy: "local"`, while the embedded runner can persist billing-classified prompt failures to shared auth state. The existing local policy seam and the Active Memory caller show the intended narrow fix path, so this is a valid focused runtime bug rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to prevent low-priority session-memory slug-generation failures from marking the shared `anthropic:claude-cli` auth profile as billing-disabled for five hours.

## Best Possible Solution

Keep the global billing classification intact and make the session-memory slug-generator embedded run helper-local by passing the existing local auth-profile failure policy, with focused regression coverage proving slug failures cannot write shared billing-disabled auth state.

## Reproduction Assessment

Yes. The issue includes production logs, and current main gives a deterministic source-level reproduction path: route the reported Anthropic extra-usage error through `generateSlugViaLLM`; the helper omits the local policy, so the runner can classify it as billing and persist shared auth-profile failure state.

## Solution Assessment

Yes for the local scoping direction, not for a broad classifier rollback. The narrowest maintainable fix is to pass `authProfileFailurePolicy: "local"` from `src/hooks/llm-slug-generator.ts` and test that a billing-classified slug-helper failure does not update shared auth state.
