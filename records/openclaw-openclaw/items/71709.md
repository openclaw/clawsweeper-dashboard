## Label Justifications

- `P2`: This is a source-proven auth failover bug in an opt-in helper path with real but bounded production impact.
- `impact:auth-provider`: The reported failure writes shared auth-profile cooldown state and changes provider routing for all agents using that profile.

## Summary

Keep open. Current main and the latest release only mitigate the default path by making LLM slugs opt-in; when `llmSlug` is enabled, the slug helper still uses shared embedded-runner auth failure handling while Anthropic extra-usage text still classifies as `billing`.

## What This Changes

The issue asks for slug-generator helper failures, especially non-billing HTTP 400s, to stop poisoning shared auth-profile billing cooldown state.

## Best Possible Solution

Add a local auth-failure policy to the slug helper and regression coverage proving billing-classified slug failures do not write shared auth state, while leaving broader slug model and simple-completion redesign decisions to the related work.

## Reproduction Assessment

Yes, at source level. With `hooks.internal.entries.session-memory.llmSlug: true`, current main reaches `generateSlugViaLLM`, the helper calls the embedded runner without local auth policy, and the reported extra-usage text still resolves to `billing`.

## Solution Assessment

Yes. Reusing `authProfileFailurePolicy: "local"` in `generateSlugViaLLM` is the narrowest maintainable fix because it isolates helper failures without weakening normal user-facing billing failover.
