## Summary

Keep open. Current main mitigates the default path by making LLM slugs opt-in, but the opt-in session-memory slug generator still calls the shared embedded runner without helper-local auth failure policy, so the reported billing-classified extra-usage error can still poison the shared auth profile.

## What This Changes

The issue requests that failures from the internal session-memory slug generator stay helper-local instead of marking the shared Claude CLI auth profile as billing-disabled for five hours.

## Best Possible Solution

Make `generateSlugViaLLM` opt into helper-local auth failure policy and add regression coverage proving slug-generation billing/auth failures do not update shared auth-profile state.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: enable `llmSlug`, let the slug helper receive an Anthropic extra-usage error, and the current embedded-runner path can classify it as billing and write the shared five-hour auth-profile disable; I did not run a live Claude CLI incident locally.

## Solution Assessment

Is this the best way to solve the issue? Yes: passing `authProfileFailurePolicy: "local"` from the slug helper reuses the existing helper-local seam and avoids weakening global billing classification or adding new config.
