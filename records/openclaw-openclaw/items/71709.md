## Summary

Keep open. Current main still routes session-memory slug generation through the shared embedded-run auth failure path: the slug helper does not pass the existing local failure policy, the reported extra-usage text still classifies as billing, and billing still writes the long profile disable window.

## What This Changes

The issue reports that session-memory LLM slug generation can turn a helper-lane HTTP 400 into a profile-wide Claude CLI billing cooldown that blocks all agents sharing that auth profile.

## Best Possible Solution

Keep global billing classification for user-facing provider failures, but make the session-memory slug generator use helper-local auth-profile failure policy and add regression coverage proving slug failures do not write shared cooldown state.

## Reproduction Assessment

Yes. The issue includes production logs, and current main provides a high-confidence source-level reproduction path: default session-memory LLM slugging calls the embedded runner without local auth failure policy, the reported text classifies as billing, and billing writes a five-hour disabled window.

## Solution Assessment

Yes. Passing `authProfileFailurePolicy: "local"` from `generateSlugViaLLM` is the narrowest maintainable fix; weakening the Anthropic extra-usage billing classifier globally would risk regressing intended same-turn model failover behavior.
