## Summary

Keep open. Current main reduced the default blast radius by making LLM slugs opt-in, but opt-in `session-memory` slug generation still calls the embedded runner without `authProfileFailurePolicy: "local"`, so an Anthropic extra-usage billing classification can still write shared auth-profile disable state.

## What This Changes

The issue asks for slug-generator helper failures, especially Anthropic HTTP 400 extra-usage errors, to stay helper-local instead of disabling the shared auth profile for all agents.

## Best Possible Solution

Make the slug helper opt into helper-local auth failure handling, preferably by passing `authProfileFailurePolicy: "local"` from `generateSlugViaLLM`, while preserving global billing classification and the current opt-in `llmSlug` default.

## Reproduction Assessment

Yes, source-level: enable `hooks.internal.entries.session-memory.llmSlug`, hit an Anthropic extra-usage error in `generateSlugViaLLM`, and current main still maps that billing failure into shared auth-profile state because the slug helper does not pass local policy. The issue also has recent production reports with the same auth-state shape.

## Solution Assessment

Yes, the narrowest maintainable fix is to reuse the existing helper-local failure policy for `temp:slug-generator`; weakening the global extra-usage billing classifier would risk regressing normal same-turn fallback behavior.
