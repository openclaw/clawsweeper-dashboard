## Summary

Keep open. Current main still routes the session-memory slug helper through the shared embedded-run auth-profile failure path: the helper does not pass the existing local failure policy, while Anthropic extra-usage text is still classified as billing and billing failures still create the reported long disabled state. This is a narrow, valid runtime bug with a clear repair seam.

## What This Changes

The issue requests that internal session-memory slug generation failures stay lane-local instead of writing a profile-wide billing cooldown that blocks other agents using the same auth profile.

## Best Possible Solution

Keep the global billing classifier intact for user-facing provider failover, but make session-memory slug generation opt into the existing helper-local auth-profile failure policy and add regression coverage proving billing-classified slug failures do not write shared cooldown state.

## Reproduction Assessment

Yes. The issue includes production logs, and current main gives a high-confidence source-level reproduction path: default session-memory LLM slugging calls the embedded runner without local auth failure policy, while the reported Anthropic extra-usage text classifies as billing and billing writes a five-hour disabled window.

## Solution Assessment

Yes for the local-scoping part: passing authProfileFailurePolicy: "local" from generateSlugViaLLM is the narrowest maintainable fix. Removing the Anthropic extra-usage billing classifier globally would risk regressing intentional model-fallback behavior covered by existing tests.
