## Summary

Keep open. Current main mitigates the default path by making LLM slugs opt-in, but the opt-in slug helper still uses the shared embedded-run auth failure path, so a helper-only Anthropic extra-usage billing error can still disable the whole auth profile.

## What This Changes

The issue requests lane-local handling for `generateSlugViaLLM` failures so slug-generation billing/error responses do not trigger profile-wide auth cooldowns.

## Best Possible Solution

Pass `authProfileFailurePolicy: "local"` from `generateSlugViaLLM` and add focused regression coverage, while preserving global billing classification for normal user-facing runs.

## Reproduction Assessment

Yes, at source level. With `llmSlug: true`, current main calls `runEmbeddedPiAgent` on `temp:slug-generator` without local auth policy; the reported extra-usage text classifies as `billing`, and billing writes the long profile disable window.

## Solution Assessment

Yes. Reusing the existing `authProfileFailurePolicy: "local"` seam in the slug helper is the narrowest maintainable fix and avoids weakening provider billing failover globally.
