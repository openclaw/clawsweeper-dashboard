## Label Justifications

- `P2`: This is a source-proven auth failover bug in an opt-in helper path with real but bounded production impact.
- `impact:auth-provider`: The failure writes shared auth-profile cooldown state and changes provider routing for all agents using that profile.

## Summary

Keep open. Current main and the latest release only reduce the default blast radius by keeping LLM slugs opt-in; when `llmSlug` is enabled, the slug helper still uses shared embedded-runner auth failure handling, and the reported Anthropic extra-usage text still classifies as `billing`.

## What This Changes

The issue requests that internal slug-generator provider failures stay lane-local and not mark the shared auth profile as billing-disabled.

## Best Possible Solution

Add `authProfileFailurePolicy: "local"` to the slug helper and regression coverage proving billing-classified slug failures do not write shared auth state, while leaving broader slug-model policy to https://github.com/openclaw/openclaw/issues/33962 and https://github.com/openclaw/openclaw/pull/65134.

## Reproduction Assessment

Yes, at source level. With `hooks.internal.entries.session-memory.llmSlug: true`, current main reaches `generateSlugViaLLM`, the helper calls the embedded runner without local auth policy, and the reported extra-usage text still resolves to `billing`.

## Solution Assessment

Yes. Reusing `authProfileFailurePolicy: "local"` in `generateSlugViaLLM` is the narrowest maintainable fix because it isolates helper failures without weakening normal user-facing billing failover.
