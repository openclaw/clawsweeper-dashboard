## Summary

Keep open. Current main only mitigates the default path by making LLM slugs opt-in; when `llmSlug` is enabled, the slug helper still uses shared embedded-runner auth failure handling and the reported Anthropic extra-usage text still classifies as `billing`.

## What This Changes

The issue asks OpenClaw to prevent an internal slug-generation HTTP 400 from disabling the shared Claude CLI auth profile for hours.

## Best Possible Solution

Add a local auth-failure policy to the slug helper and regression coverage proving billing-classified slug failures do not write shared auth state, while leaving broader simple-completion and CLI-dispatch policy to the existing related work.

## Reproduction Assessment

Yes, at source level. With `llmSlug: true`, current main calls the slug embedded run without local auth policy, the reported extra-usage text classifies as `billing`, and shared billing reasons still reach `markAuthProfileFailure`.

## Solution Assessment

Yes. Reusing the existing `authProfileFailurePolicy: "local"` seam in `generateSlugViaLLM` is the narrowest maintainable fix and avoids weakening normal user-facing billing failover.
