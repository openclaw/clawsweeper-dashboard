## Summary

Keep open. Current `main` still routes the session-memory slug helper through `runEmbeddedPiAgent` without the existing helper-local auth-profile failure policy, while the embedded runner can still persist billing-classified Anthropic extra-usage prompt failures to shared auth state. The previous ClawSweeper conclusion remains valid on `9d21df251e50ca5fbf5c0e34783cc9f66114951c`.

## What This Changes

The issue asks for session-memory LLM slug generation failures to stay helper-local so a billing-classified HTTP 400 or Anthropic extra-usage error cannot disable the shared Claude CLI auth profile for all agents.

## Best Possible Solution

Keep the global billing classifier as-is and scope the slug-generator embedded run with the existing `authProfileFailurePolicy: "local"` seam, backed by a focused `llm-slug-generator` regression test and the existing failure-policy helper test.

## Reproduction Assessment

Yes. The issue discussion provides production logs, and current source gives a deterministic source-level reproduction: send a billing-classified Anthropic extra-usage prompt failure through `generateSlugViaLLM`; the helper currently omits the local policy, so the embedded runner can persist shared auth-profile failure state.

## Solution Assessment

Yes. The narrowest maintainable fix is to pass `authProfileFailurePolicy: "local"` from `src/hooks/llm-slug-generator.ts`; changing the global billing classifier would risk regressing the intentional #61608 fallback behavior.
