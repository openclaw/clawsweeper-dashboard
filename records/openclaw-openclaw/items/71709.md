## Summary

Keep open. Current main mitigates the original default exposure by making session-memory LLM slugs opt-in, but the opt-in slug helper still runs through shared auth-profile failure persistence without local policy, so the reported profile-wide billing cooldown can still happen when `llmSlug` is enabled.

## What This Changes

The issue asks for slug-generator helper failures to stay lane-local so an HTTP 400/extra-usage response cannot disable the shared Claude CLI auth profile for five hours.

## Best Possible Solution

Pass `authProfileFailurePolicy: "local"` from `generateSlugViaLLM`, add regression coverage for billing-classified helper failures, and leave the global billing classifier behavior intact.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: with `llmSlug: true`, the slug helper enters `runEmbeddedPiAgent` without local auth policy, the reported extra-usage text maps to billing, and billing failures set a roughly five-hour `disabledUntil`. I did not run a live Claude CLI reproduction.

## Solution Assessment

Yes for the helper-local direction: the existing `authProfileFailurePolicy: "local"` seam is the narrowest maintainable fix. Broadly weakening the billing classifier would be riskier because that classifier is intentional for user-facing fallback paths.
