## Summary

Keep open. Current main reduces the default blast radius by making `llmSlug` opt-in, but the enabled slug-generator path still uses the shared embedded-run auth failure policy, so a billing-classified helper failure can still write shared auth-profile cooldown state.

## What This Changes

The issue asks for slug-generator/helper-lane provider failures to stay lane-local instead of disabling the shared `anthropic:claude-cli` auth profile for five hours.

## Best Possible Solution

Have `generateSlugViaLLM` use the existing helper-local auth failure policy and add regression coverage proving billing-classified slug-helper failures do not update shared auth-profile cooldown state.

## Reproduction Assessment

Yes. The current-main source path is high-confidence for `llmSlug: true`: session-memory calls the slug helper, the helper omits local auth failure policy, the reported extra-usage text classifies as billing, and billing writes a five-hour `disabledUntil`.

## Solution Assessment

Yes for helper isolation; the narrowest fix is to pass `authProfileFailurePolicy: "local"` from the slug helper. Broadly weakening the global Anthropic extra-usage billing matcher would be riskier because user-facing fallback still depends on that classification.
