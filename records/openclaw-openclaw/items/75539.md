## Summary

Keep open: current main partially improves the Telegram and QQBot surface, but the remaining QQBot REST, token, tool, and WebSocket paths still lack a visible channel-owned IPv4/DNS fallback policy for the reported broken-IPv6 environments.

## What This Changes

The issue asks OpenClaw to avoid Telegram and QQBot connection stalls on hosts where AAAA records resolve but global IPv6 routing is unavailable.

## Best Possible Solution

Keep the Telegram transport fixes and add a QQBot-owned transport layer that covers REST, token, channel-tool, upload/STT helper consistency, and gateway WebSocket connection setup without relying on process-wide DNS mutations.

## Reproduction Assessment

No, not end to end. The issue provides two concrete environments and current source still shows QQBot raw fetch/ws paths, but this review did not run a broken-IPv6 host with live QQBot credentials.

## Solution Assessment

Yes. The narrow maintainable direction is a QQBot-owned transport policy wired into its outbound HTTP and WebSocket paths, while adjacent global or Telegram-only PRs should not be treated as a complete fix for this issue.
