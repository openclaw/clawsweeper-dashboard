## Summary

Keep this open: current main has Telegram-specific IPv4 fallback handling, but QQBot still has raw REST, token, tool, media-upload, STT, and WebSocket transport paths without a visible family/dispatcher/fallback policy, so the reported bundled-undici broken-IPv6 regression is not fixed across the affected surface.

## What This Changes

The report asks for Telegram and QQBot plugin transports to avoid bundled-undici IPv6-first stalls on dual-stack hosts without usable IPv6.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, channel-tool API, STT, media upload, and WebSocket paths, reusing public plugin SDK fetch/dispatcher helpers where possible while keeping Telegram's existing fallback behavior intact.

## Reproduction Assessment

No, not end to end. The issue and comment provide credible environments and logs, and current source inspection gives a high-confidence QQBot code-path reproduction, but this review did not have a broken-IPv6 host plus live channel credentials.

## Solution Assessment

Yes, the best direction is a plugin/channel-owned transport policy. A global DNS mutation or core loader migration would be broader than the remaining QQBot-owned gaps.
