## Summary

Keep open: current main has Telegram-specific IPv4/DNS fallback handling, but the QQBot side of the reported regression still routes key REST, token, upload, tool, STT, and WebSocket paths through raw transports without an owner-scoped IPv4/DNS fallback policy.

## What This Changes

The issue asks OpenClaw to keep Telegram and QQBot usable on hosts where DNS returns IPv6 addresses but the host lacks working global IPv6 egress.

## Best Possible Solution

Centralize QQBot outbound transport in the plugin so every QQ Open Platform REST, token, media upload, channel tool, STT, and gateway WebSocket call uses one tested DNS/family/fallback policy without adding global DNS mutation or a core channel special case.

## Reproduction Assessment

No, not end to end. The issue provides concrete environments and logs, and current source inspection shows QQBot still has raw fetch/ws paths, but this review did not run a broken-IPv6 host with live channel credentials.

## Solution Assessment

Yes. The narrow maintainable fix is QQBot-owned transport policy that reuses existing SDK or undici dispatcher seams where possible; process-wide NODE_OPTIONS, global DNS mutation, or core loader migration would be broader than the affected owner boundary.
