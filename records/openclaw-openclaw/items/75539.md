## Summary

Keep this open: current main has substantial Telegram-owned IPv4/fallback transport handling, but QQBot still has multiple raw fetch and WebSocket paths without an owner-level DNS/family/dispatcher policy, so the reported broken-IPv6 regression is not resolved across the affected surface.

## What This Changes

Make Telegram and QQBot plugin networking tolerate hosts where IPv6 resolves but cannot connect under bundled undici.

## Best Possible Solution

Keep the issue open until QQBot has an owner-local transport layer, with tests proving the affected outbound paths use bounded IPv4/fallback behavior without moving channel-specific policy into core.

## Reproduction Assessment

No, not end to end. The report gives credible environments and logs, and current source inspection gives a high-confidence QQBot code-path reproduction via raw fetch/ws transports, but this review did not have a broken-IPv6 host plus live channel credentials to exercise the full stall.

## Solution Assessment

Yes, the maintainable direction is a QQBot-owned transport policy using existing plugin SDK fetch/dispatcher seams where they fit. A process-wide DNS workaround or core loader migration would be broader than the remaining channel-specific gap.
