## Summary

Keep this open. Current main is tagged as the same `v2026.4.29` release named in the report, Telegram has partial channel-owned mitigations, and QQBot still has direct `fetch` and `ws` transport paths with no plugin-owned DNS/family fallback policy.

## What This Changes

The issue reports Telegram and QQBot plugin connection failures on IPv6-unreachable VPS hosts after `v2026.4.29`, attributed to undici/network-family fallback behavior in bundled plugin runtime dependencies.

## Best Possible Solution

Add a narrow QQBot-owned transport policy for REST, token, upload, tool API, and WebSocket paths, reusing the existing plugin/SDK transport helpers where possible, and only adjust Telegram defaults after verifying undici 8 and Node 22 behavior against the existing Telegram fallback tests. Document the workaround and add focused regression tests rather than changing global process DNS behavior.

## Reproduction Assessment

No. The report gives a credible environment, symptoms, and workaround, and source inspection shows matching transport gaps, but I did not have a broken-IPv6 VPS plus live Telegram/QQBot credentials to reproduce the runtime stall end to end.

## Solution Assessment

Unclear as stated. The desired behavior is right, but the maintainable fix should be per-channel/plugin transport policy, especially for QQBot, rather than a global `NODE_OPTIONS` or process-wide DNS mutation.
