## Summary

Keep this open: current main has Telegram-owned IPv4/fallback handling, but QQBot still has raw REST, token, tool API, media upload, STT, and WebSocket paths without a visible family, dispatcher, DNS-order, or fallback policy, so the reported dual-stack broken-IPv6 regression remains unresolved for part of the affected surface.

## What This Changes

Make Telegram and QQBot plugin transports avoid slow undici IPv6-first stalls on hosts that resolve IPv6 without usable global IPv6 reachability.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, channel-tool API, media upload, STT, and WebSocket paths, reusing public plugin SDK fetch/dispatcher helpers where possible while keeping core extension-agnostic.

## Reproduction Assessment

No, not end to end. The issue discussion provides credible environments and logs, and source inspection gives a high-confidence QQBot code-path reproduction, but live confirmation needs a broken-IPv6 host plus channel credentials.

## Solution Assessment

Yes, a plugin-owned transport policy is the narrowest maintainable direction. A process-wide NODE_OPTIONS workaround, global DNS mutation, or core loader migration would be broader and less reliable than wiring QQBot's own outbound transports.
