## Summary

Keep open: current main has Telegram-owned IPv4/DNS fallback handling, but QQBot still has raw REST, token, upload, tool, STT, and WebSocket paths without visible dispatcher, family, DNS-order, or fallback policy, so the reported dual-stack broken-IPv6 regression is not resolved across the affected surface.

## What This Changes

Request that Telegram and QQBot remain responsive on hosts where dual-stack DNS returns IPv6 that is not globally reachable.

## Best Possible Solution

Add a QQBot-owned transport policy covering REST, token, media upload, channel-tool, STT, and gateway WebSocket paths, reusing public plugin SDK fetch/dispatcher helpers where possible and keeping policy local to the plugin.

## Reproduction Assessment

No, not end to end. The issue provides concrete environments and logs, and source inspection gives a clear current-main QQBot path through raw fetch/ws, but this review did not run a broken-IPv6 host with live channel credentials.

## Solution Assessment

Yes, the narrow maintainable direction is plugin-owned QQBot transport policy. A process-wide NODE_OPTIONS workaround, global DNS mutation, or core loader migration would be broader and less reliable than wiring QQBot's own outbound transports.
