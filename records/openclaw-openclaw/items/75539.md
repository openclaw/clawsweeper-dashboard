## Summary

Keep open: current main has strong Telegram IPv4/DNS fallback handling, but the QQBot side still has raw REST, token, upload, tool, STT, and WebSocket paths without a visible dispatcher, DNS-order, family, or fallback policy.

## What This Changes

The issue reports a regression where Telegram and QQBot stall or fail on dual-stack hosts with broken IPv6 because bundled undici attempts IPv6 before falling back to IPv4.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, channel-tool API, STT, media upload, and gateway WebSocket paths, then remove the QQBot raw-fetch allowlist entries.

## Reproduction Assessment

No, not end to end. The issue supplies concrete logs and environments, and source inspection on current main shows QQBot raw fetch/ws paths, but no live broken-IPv6 host with channel credentials was exercised.

## Solution Assessment

Yes, the narrow maintainable direction is plugin-owned QQBot transport policy. A process-wide DNS/global undici workaround would be broader and would not cover the staged runtime-deps behavior consistently.
