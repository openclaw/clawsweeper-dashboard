## Summary

Keep this open. Current main has strong Telegram-owned fallback handling, including sticky IPv4 and pinned-IP retry coverage, but QQBot still routes REST, token, upload, STT/tool API, and WebSocket startup through raw `fetch`/`ws` paths without a visible family, lookup, or dispatcher policy, so the reported broken-IPv6 dual-stack failure remains plausible and actionable.

## What This Changes

The issue requests reliable Telegram and QQBot connectivity on hosts where dual-stack DNS returns IPv6 addresses but global IPv6 egress is unavailable, especially with bundled undici 8.1.0 in plugin runtime dependencies.

## Best Possible Solution

Add a QQBot-owned transport layer that consistently applies an IPv4/fallback policy to the QQ Open Platform REST client, token fetch, channel API tool, chunked uploads, STT calls when applicable, and gateway WebSocket creation. Keep the policy local to the plugin or public SDK transport helpers, and avoid process-wide DNS mutation or a core loader migration.

## Reproduction Assessment

No, not end to end. The issue has credible environments and logs, and current source inspection gives a high-confidence QQBot code-path reproduction through raw `fetch`/`ws`, but I did not have the broken-IPv6 host plus live channel credentials to reproduce the stall directly.

## Solution Assessment

Yes, a focused QQBot transport repair is the best next path. Telegram already has channel-owned fallback handling, so the narrow maintainable fix is to close the remaining QQBot transport gaps rather than changing global `NODE_OPTIONS`, process DNS order, or bundled runtime-deps loading.
