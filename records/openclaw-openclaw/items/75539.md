## Summary

Keep this open: current main has Telegram-owned IPv4 fallback handling, but QQBot still routes REST, token, upload, tool, STT, and gateway WebSocket traffic through raw transports with no visible IPv4/DNS-order/fallback policy.

## What This Changes

The issue asks OpenClaw to avoid Telegram and QQBot connection stalls on hosts where DNS returns IPv6 but global IPv6 connectivity is broken.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, channel-tool API, STT, media upload, and WebSocket paths, reusing public plugin SDK fetch/dispatcher helpers where available and keeping policy local to the plugin.

## Reproduction Assessment

No, not end to end. The issue and comments provide credible environments and logs, and current-main source gives a high-confidence QQBot code-path reproduction via raw fetch/ws on broken IPv6, but this read-only review did not exercise live credentials on that network.

## Solution Assessment

Yes for the remaining surface: a QQBot-owned transport policy is the narrowest maintainable fix. A process-wide NODE_OPTIONS workaround, global DNS mutation, or core loader migration would be broader than the plugin-owned paths that still lack fallback wiring.
