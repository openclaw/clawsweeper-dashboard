## Summary

Keep this open. Current main still has QQBot REST, token, upload/tool API, STT, and WebSocket paths using default transports without a QQBot-owned DNS/family policy, while Telegram already has channel-owned IPv4/DNS fallback handling and docs. The issue is recent, detailed, and narrow enough for a focused plugin repair rather than cleanup closure.

## What This Changes

The issue requests a fix for Telegram and QQBot plugin connection stalls on servers where dual-stack endpoints resolve IPv6 but the host lacks global IPv6 reachability.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token refresh, chunk uploads, channel API tool calls, STT, and gateway WebSocket connection setup. It should reuse existing OpenClaw/SDK transport helpers where possible, expose a small documented QQBot network config only if needed, preserve proxy/security behavior, and add focused regression tests plus a changelog entry.

## Reproduction Assessment

No. The report provides a credible environment, logs, and workaround, and current source inspection shows matching QQBot transport gaps, but I did not have a no-global-IPv6 VPS plus live Telegram/QQBot credentials to reproduce the runtime stall end to end.

## Solution Assessment

Yes, the narrow maintainable solution is plugin-owned QQBot transport policy rather than changing global process DNS behavior. Telegram already has a scoped implementation, so the remaining work should avoid broad core changes unless a shared SDK helper is needed for multiple plugins.
