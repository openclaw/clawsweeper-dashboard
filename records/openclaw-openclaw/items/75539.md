## Summary

Keep this open: current main has Telegram-side IPv4 fallback and DNS-order handling, but QQBot still has direct REST, token, channel-tool, STT, media upload, and WebSocket transport paths without a visible plugin-owned DNS family or fallback policy.

## What This Changes

Request that Telegram and QQBot avoid long undici IPv6-first timeouts on hosts where AAAA records resolve but global IPv6 egress fails.

## Best Possible Solution

Keep the Telegram transport as-is and add a QQBot-owned outbound transport layer for REST, token, channel-tool API, STT, media upload, and gateway WebSocket paths, reusing SDK/runtime helpers where appropriate.

## Reproduction Assessment

No, not end to end. The issue and follow-up provide credible logs and environments, and current source gives a high-confidence QQBot code-path reproduction via raw fetch/ws on broken IPv6, but this review did not run live channel credentials on such a host.

## Solution Assessment

Yes for the remaining work: a plugin-owned QQBot transport policy is narrower and more maintainable than changing global DNS, startup migrations, or core loader behavior.
