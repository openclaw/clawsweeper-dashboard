## Summary

Keep this open. Current main has substantial Telegram-owned IPv4/fallback handling, but QQBot still has raw REST, token, upload, STT/tool API, and WebSocket transports without a visible family/dispatcher policy, so the reported broken-IPv6 dual-stack failure is not resolved across the affected surface.

## What This Changes

The issue reports Telegram and QQBot connection stalls on hosts where dual-stack DNS returns IPv6 but IPv6 connections time out under bundled undici 8.1.0.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, upload, STT/tool API, and WebSocket paths, reusing public plugin SDK transport helpers where practical and keeping the policy local to the plugin. Preserve Telegram's existing fallback chain, then make only a targeted Telegram adjustment if the reported sendChatAction latency still reproduces against current main.

## Reproduction Assessment

No, not end to end. The issue and follow-up comment provide credible environments and logs, and source inspection gives a high-confidence QQBot code-path reproduction via raw fetch/ws, but this review did not have the broken-IPv6 host plus live channel credentials to reproduce the stall directly.

## Solution Assessment

No, current main is not yet the best fix for the reported surface. Telegram has a maintainable channel-owned fallback, but QQBot needs the same kind of plugin-owned transport policy rather than a process-wide NODE_OPTIONS workaround or broad core startup mutation.
