## Summary

Keep this open. Current main has substantial Telegram-side mitigation, but the reported user problem is not fully covered: QQBot still uses unconfigured global fetch and ws connection paths, and Telegram timeout-triggered fallback is limited to startup/control calls rather than the sendChatAction path called out in the newer report context.

## What This Changes

The issue requests a fix for Telegram and QQBot connection stalls when bundled undici resolves IPv6 addresses on hosts without reliable IPv6 egress.

## Best Possible Solution

Keep the fix plugin-owned: add a shared QQBot transport helper for REST, token, upload/tool API, and WebSocket gateway paths, then close the Telegram sendChatAction/request-timeout gap only after confirming the fallback is safe for non-message control calls. Document the operator workaround and add regression tests for the transport policy instead of mutating process-wide DNS behavior in core.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from this review because it would require a dual-stack host with broken IPv6 egress plus live Telegram/QQBot credentials. The report and follow-up comment provide credible logs, and source inspection gives a high-confidence static reproduction of the missing QQBot transport policy and Telegram sendChatAction allowlist gap.

## Solution Assessment

Yes, the best direction is a narrow per-plugin transport repair. A global `NODE_OPTIONS`, process-wide DNS-order change, or core startup migration would be broader and less maintainable than fixing the affected channel/plugin HTTP and WebSocket seams.
