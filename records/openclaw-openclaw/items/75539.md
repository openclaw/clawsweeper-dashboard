## Summary

Keep open: current main has substantial Telegram-owned IPv4/DNS fallback handling, but QQBot still has raw REST, token, upload, tool, STT, and WebSocket paths without a visible dispatcher, family, DNS-order, or fallback policy, so the reported dual-stack broken-IPv6 regression is not resolved across the affected surface.

## What This Changes

The issue reports Telegram and QQBot plugin connection stalls on dual-stack hosts without usable IPv6 because bundled undici tries IPv6 first and falls back slowly.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, channel-tool API, STT, media upload, and WebSocket paths, reusing public plugin SDK fetch/dispatcher helpers where possible and keeping policy local to the plugin.

## Reproduction Assessment

No, not end to end. The issue provides concrete environments and logs, and source inspection gives a clear current-main QQBot path through raw fetch/ws, but this read-only review did not run a broken-IPv6 host with live channel credentials.

## Solution Assessment

Yes, the narrow maintainable direction is plugin-owned QQBot transport policy. A process-wide NODE_OPTIONS workaround, global DNS mutation, or core loader migration would be broader and less reliable than wiring QQBot's own outbound transports.
