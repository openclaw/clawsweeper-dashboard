## Summary

Keep open. Current main has substantial Telegram-specific IPv4 fallback work, matching the related earlier Telegram IPv6 report, but QQBot still uses raw global fetch and ws connection paths without an owner-specific DNS family, dispatcher, or fallback policy. That leaves a meaningful part of this report unresolved and makes it unsuitable for implemented-on-main cleanup.

## What This Changes

Request to prevent Telegram and QQBot plugin connection stalls on servers without global IPv6 by making their HTTP/WebSocket transports respect or recover from IPv6-unreachable DNS results.

## Best Possible Solution

Add a QQBot-owned transport layer, using existing plugin SDK fetch/runtime seams where possible, so QQBot REST token/API calls and gateway WebSocket handshakes can prefer IPv4 or fail over safely on dual-stack hosts without moving channel-specific policy into core. Keep Telegram's existing transport behavior intact and document any QQBot operator override.

## Reproduction Assessment

Unclear: the issue body gives concrete environment details and logs, but this read-only review did not have a no-global-IPv6 VPS or channel credentials. A focused unit test can reproduce the current QQBot transport gap by asserting no dispatcher/family options are passed to fetch/ws, but the full event-loop stall needs a live network fixture.

## Solution Assessment

Yes for keeping this open; current main is not a complete solution because only Telegram has the owner-specific IPv4 fallback controls. The narrow maintainable fix is in the QQBot plugin transport path, with a generic SDK seam only if the implementation proves reusable across multiple plugins.
