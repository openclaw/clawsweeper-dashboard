## Summary

Keep open: current main has substantial Telegram fallback work, but the QQBot REST, token, channel-tool, and WebSocket paths still lack a visible channel-owned IPv4/DNS fallback for the reported broken-IPv6 environments.

## What This Changes

The issue reports Telegram and QQBot connection stalls on hosts where AAAA records resolve but outbound IPv6 is not globally reachable.

## Best Possible Solution

Keep the Telegram transport fixes and add a QQBot-owned transport layer covering REST, token, channel-tool, media/STT consistency, and gateway WebSocket setup without relying on process-wide DNS mutation.

## Reproduction Assessment

No, not end to end. The issue provides concrete environments and current source still shows unguarded QQBot outbound paths, but this review did not run a broken-IPv6 host with live QQBot credentials.

## Solution Assessment

Yes. The narrow maintainable direction is a QQBot-owned transport policy wired into its outbound HTTP and WebSocket paths, while Telegram-only or global networking PRs should remain adjacent rather than treated as the complete fix.
