## Summary

Keep this open. Current main has substantial Telegram-owned IPv4/fallback handling, but the QQBot REST, token, upload, STT, and WebSocket paths still use default `fetch`/`ws` transports with no visible family or dispatcher policy, so the reported dual-stack broken-IPv6 failure is not resolved across the affected surface.

## What This Changes

Fix Telegram and QQBot plugin transport behavior so bundled undici 8.1.0 does not stall on IPv6-first dual-stack hosts without usable IPv6 reachability.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, upload, STT/tool API, and WebSocket paths, reusing public plugin SDK transport helpers where possible and keeping policy local to the plugin. Preserve Telegram's existing fallback chain, then add focused tests or a small Telegram adjustment only if the reported Tailscale `sendChatAction` latency still reproduces against current main.

## Reproduction Assessment

No, not end to end. The issue and follow-up comment provide credible environments and logs, and source inspection gives a high-confidence QQBot code-path reproduction via raw `fetch`/`ws`, but I did not have the broken-IPv6 host plus live channel credentials to reproduce the stall directly.

## Solution Assessment

Unclear as requested. The desired behavior is right, but the maintainable fix is a plugin/channel-owned transport policy, not `NODE_OPTIONS`, process-wide DNS mutation, or a broad core loader change.
