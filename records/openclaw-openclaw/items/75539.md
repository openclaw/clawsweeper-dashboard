## Summary

Keep this open: current main has substantial Telegram-owned IPv4 fallback handling, but QQBot still has raw REST, token, upload, tool, STT, and WebSocket transport paths without a visible family, dispatcher, DNS-order, or fallback policy, so the reported dual-stack broken-IPv6 failure is not resolved across the affected surface.

## What This Changes

Fix Telegram and QQBot plugin connectivity on hosts where DNS returns IPv6 but global IPv6 routing is unavailable, especially with the bundled undici runtime dependency.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, channel-tool API, STT, media upload, and WebSocket paths, reusing public plugin SDK fetch/dispatcher helpers where possible and keeping policy local to the plugin.

## Reproduction Assessment

No, not end to end. The issue and follow-up comment provide credible environments and logs, and source inspection gives a high-confidence QQBot code-path reproduction via raw fetch/ws, but this read-only review did not have a broken-IPv6 host plus live channel credentials to reproduce the stall directly.

## Solution Assessment

Yes, the maintainable remaining fix is plugin/channel-owned transport policy. A process-wide NODE_OPTIONS workaround, global DNS mutation, or core loader migration would be broader and less reliable than wiring QQBot's own outbound transports.
