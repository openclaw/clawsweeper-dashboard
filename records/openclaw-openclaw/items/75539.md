## Summary

Keep this open. Current main still has concrete transport gaps that match the report: QQBot uses raw fetch and ws paths without a channel-owned DNS/family policy, and Telegram has sticky IPv4 fallbacks but its timeout-forced fallback path does not cover sendChatAction, which the follow-up reporter specifically identified as degraded.

## What This Changes

Make Telegram and QQBot remain responsive on dual-stack hosts with broken IPv6 by using fast IPv4 fallback or family-aware transport policy instead of letting bundled undici connection attempts time out.

## Best Possible Solution

Add plugin-owned transport policy where the affected network calls live: QQBot should get a shared HTTP/WebSocket transport helper for token, REST, upload, channel tool API, STT where applicable, and gateway connections; Telegram should extend its existing fallback machinery so timed-out safe typing/control calls can promote to the fallback route without changing global DNS behavior.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete versions, host/network conditions, DNS answers, staged undici path, logs, and comparison commands; source inspection shows current main still exercises transport paths that match those symptoms, even though I did not live-run the broken network locally.

## Solution Assessment

Yes for the keep-open direction. The maintainable fix is a narrow channel/plugin transport policy plus focused Telegram fallback coverage, not a dependency downgrade or process-wide NODE_OPTIONS change.
