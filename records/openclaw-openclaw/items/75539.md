## Summary

Keep this open. The report is fresh, detailed, and still matches current main: Telegram has channel-owned undici dispatcher and fallback controls for IPv4-first behavior, but QQBot REST, token, and gateway WebSocket paths still use raw platform transports without a QQBot-owned DNS/family policy, so the combined regression is not implemented, stale, duplicate, or outside the repository.

## What This Changes

The issue requests a fix for Telegram and QQBot connection failures on servers where hostnames resolve IPv6 addresses but the server lacks usable global IPv6 routing, causing undici/fetch/WebSocket timeouts and gateway readiness failures.

## Best Possible Solution

Keep the issue open and fix the remaining channel-owned transport gap, most likely in QQBot first: add an explicit DNS/family/fallback policy for QQBot REST, token, media, and gateway WebSocket connections, align docs with Telegram's network settings where appropriate, and add focused transport-option tests without changing global process DNS behavior.
