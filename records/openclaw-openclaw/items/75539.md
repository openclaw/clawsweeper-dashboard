## Summary

Keep this open. The report is fresh and detailed, and current main does not provide a new fix beyond the latest reported release: the central Telegram transport, Telegram network config, QQBot REST/token fetch, and lockfile blobs are identical between current main and `v2026.4.29`. Telegram already has owner-specific mitigations, but QQBot still has raw global `fetch` and `ws` connection paths with no channel-owned DNS/family policy, so this remains actionable rather than stale, duplicate, or implemented.

## What This Changes

Request to make Telegram and QQBot channel transports avoid undici/Node IPv6-first stalls on servers where hostnames resolve AAAA records but the server has no global IPv6 reachability.

## Best Possible Solution

Keep the issue open and fix this at the channel/plugin transport boundary: give QQBot the same kind of explicit DNS/family/fallback policy that Telegram has for every HTTP and WebSocket path, and tighten Telegram defaults or fallback behavior only after verifying how undici 8 and Node 22 handle `autoSelectFamily` with `dnsResultOrder=ipv4first`. Avoid global process DNS mutation; document the workaround and cover the transport policy with focused tests.

## Reproduction Assessment

No. The issue has a credible environment and log-based reproduction path, and source inspection shows plausible transport gaps, but I did not reproduce it on a host with broken global IPv6 and live channel credentials.

## Solution Assessment

Unclear. The requested direction is sound, but the safest implementation is a narrow per-channel transport fix rather than a global Node DNS setting; Telegram's existing mitigation should be dependency-verified before changing defaults.
