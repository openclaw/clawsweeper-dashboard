## Summary

Keep open. The report is fresh, detailed, and current main still has an actionable QQBot transport gap: Telegram has explicit undici DNS/family fallback handling and docs, but QQBot still uses raw global fetch/WebSocket paths for token, REST, and gateway connections without a channel-owned DNS/family/fallback policy.

## What This Changes

The issue requests fixing Telegram and QQBot connectivity on IPv6-broken hosts where undici-backed requests time out before IPv4 succeeds.

## Best Possible Solution

Keep the issue open and land a narrow plugin-owned transport fix: give QQBot explicit HTTP and WebSocket DNS/family/fallback handling comparable to Telegram's per-channel transport, verify Telegram's existing fallback covers the reported startup/probe paths, and document the VPS workaround without adding global process DNS mutation.

## Reproduction Assessment

No. The issue gives a credible environment, logs, and workaround, and source inspection shows plausible QQBot gaps, but I did not reproduce it on an IPv6-broken host with live channel credentials.

## Solution Assessment

Unclear. The requested direction is sound, but the maintainable fix should stay at the Telegram/QQBot transport boundary rather than setting global NODE_OPTIONS or mutating process-wide DNS defaults.
