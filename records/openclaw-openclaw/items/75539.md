## Summary

Keep this open: current main has strong Telegram-owned IPv4 fallback handling, but QQBot still has multiple outbound REST, token, media, STT, tool, and WebSocket paths that use raw fetch/ws without a visible dispatcher, DNS-order, family, or fallback policy.

## What This Changes

The issue reports Telegram and QQBot connectivity failures on hosts where IPv6 resolves but has no usable route, tied to bundled undici 8.1.0 behavior after upgrading to v2026.4.29.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, channel-tool API, STT, media upload, and WebSocket paths, reusing public plugin SDK network helpers where possible while keeping core extension-agnostic.

## Reproduction Assessment

No, not end to end. The issue and follow-up comment provide credible environments and logs, and source inspection gives a high-confidence QQBot code-path reproduction via raw fetch/ws, but this read-only review did not have a broken-IPv6 host plus live channel credentials.

## Solution Assessment

Yes, the best remaining fix is a plugin-owned QQBot transport policy. A process-wide NODE_OPTIONS workaround, global DNS mutation, or core loader change would be broader and less reliable than wiring QQBot's outbound transports.
