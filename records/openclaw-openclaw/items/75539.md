## Summary

Keep this open. The report is fresh, detailed, and current main still has the same central Telegram and QQBot transport files as the reported v2026.4.29 release; Telegram has explicit dispatcher/fallback handling, but QQBot still uses ambient fetch and ws connection paths without a channel-owned DNS/family policy.

## What This Changes

Make the Telegram and QQBot plugins connect reliably on servers where DNS returns IPv6 records but the host lacks usable global IPv6 reachability.

## Best Possible Solution

Keep the issue open and fix the remaining QQBot transport gap at the plugin boundary: give QQBot REST, token, upload, and gateway WebSocket paths a shared, testable network policy comparable to Telegram's scoped transport handling, with docs and config updated only for the QQBot surface that needs it.

## Reproduction Assessment

No. The issue gives a credible environment and log-based path, and current source inspection shows the QQBot transport gap, but I did not reproduce it on a host with broken global IPv6 and live channel credentials.

## Solution Assessment

Yes, with a narrower boundary than a global process workaround. The maintainable fix is per-plugin transport policy and tests, while leaving Telegram's existing scoped fallback in place unless dependency verification shows a concrete Telegram regression remains.
