## Summary

Keep this open: current main has Telegram-owned IPv4 fallback handling, but QQBot still has REST, token, channel-tool, STT, upload, and WebSocket paths using raw fetch/ws without a visible DNS-order, dispatcher, family, or fallback policy.

## What This Changes

The issue asks OpenClaw to fix Telegram and QQBot connection stalls on hosts where bundled undici prefers unroutable IPv6 addresses after v2026.4.29.

## Best Possible Solution

Add a QQBot-owned transport layer for REST, token, tool API, STT, media upload, and WebSocket paths, reusing public plugin SDK fetch/dispatcher helpers where possible and keeping policy local to the plugin.

## Reproduction Assessment

No, not end to end. The issue and follow-up comment provide credible environments and logs, and current source gives a high-confidence QQBot code-path reproduction through raw fetch/ws, but this read-only review did not have a broken-IPv6 host plus live channel credentials.

## Solution Assessment

Yes. A QQBot plugin-owned transport policy is narrower and more maintainable than a process-wide NODE_OPTIONS workaround, global DNS mutation, undici downgrade, or core startup migration.
