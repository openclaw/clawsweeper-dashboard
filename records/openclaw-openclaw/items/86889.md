## Summary

Keep this PR open. The fix targets the right gateway history path and the supplied proof is useful, but the current branch still drops any adjacent assistant after a pre-cutoff `subagent_announce` user without checking the assistant record's own timestamp, so it can remove a current-session reply from `chat.history`.

## What This Changes

The PR adds JSONL record timestamps to gateway transcript messages and filters stale pre-session-start `subagent_announce` user/assistant pairs from `chat.history` hydration, with focused tests.

## Best Possible Solution

Keep the cutoff-based `chat.history` filter, but delete the adjacent assistant only when both the announce user and that assistant have finite record timestamps before `sessionStartedAt`, with a regression test for the current-session assistant case.

## Reproduction Assessment

Yes, the linked report plus current source give a clear reproduction path: after `/new`, `sessions_history`/`chat.history` can hydrate a fresh session with a stale `subagent_announce` assistant reply under the same session key. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No, not yet. The cutoff filter belongs in the right gateway history path, but it must check the adjacent assistant's own record timestamp before deleting that assistant.
