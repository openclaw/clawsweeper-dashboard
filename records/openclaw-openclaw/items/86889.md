## Summary

Keep open: the PR addresses the stale `subagent_announce` chat history contamination path, but the current helper can still drop a current-session assistant message because it skips the next assistant by role-only adjacency instead of checking that assistant's own record timestamp.

## What This Changes

The PR adds per-record transcript timestamps to gateway session reads and filters pre-session-start `subagent_announce` user/assistant pairs from `chat.history`, with focused regression tests.

## Best Possible Solution

Keep the recency filter, but only skip the assistant when both the announce user and adjacent assistant have finite `recordTimestampMs` values before `sessionStartedAt`; otherwise drop only the stale announce user and preserve current messages.

## Reproduction Assessment

Yes. The linked report and PR body give a concrete component repro for the stale announce contamination, and the remaining review defect is source-reproducible from the helper's role-only assistant skip plus current reader behavior that can return filtered recent tails.

## Solution Assessment

No. The direction is right, but the implementation should bind the skipped assistant to the same pre-cutoff timestamp contract instead of treating any adjacent assistant as the announce reply.
