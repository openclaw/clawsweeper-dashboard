## Summary

Keep open: this is a useful gateway/session-state fix, but the PR is not merge-ready because the new filter can still drop an adjacent current-session assistant reply without checking that reply's own record timestamp.

## What This Changes

The PR adds record timestamps to parsed gateway transcript messages and filters pre-session-start `subagent_announce` user/assistant pairs from `chat.history`, with focused regression tests.

## Best Possible Solution

Keep the cutoff-based `chat.history` filter, but skip the adjacent assistant only when that assistant also has a finite `recordTimestampMs` before `sessionStartedAt`, then refresh CI on current main.

## Reproduction Assessment

Yes. The linked report and current source give a high-confidence path: after `/new`, `sessions_history`/`chat.history` can hydrate a fresh session with stale `subagent_announce` context under the same session key; I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No, not yet. The cutoff filter belongs in the right gateway history path, but the adjacent assistant skip must use the assistant's own record timestamp before deleting it.
