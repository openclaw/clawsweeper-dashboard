## Summary

Keep open: this maintainer-authored PR is the active candidate for the hook reply bug, but it still needs rebase and repair before merge because the submitted patch can misroute replies, bypass send-policy suppression, and throw on malformed hook output.

## What This Changes

The PR adds automatic delivery of internal hook `event.messages` for command and `message:received` replyable surfaces, plus related docs and regression tests.

## Best Possible Solution

Land a rebased, policy-aware implementation that drains `message:received` hook messages through the current routing path, targets the inbound sender, honors send-policy suppression, hardens hook output normalization, and keeps lifecycle-only hooks non-replying.

Yes. Current main and the latest release create and trigger the `message:received` internal hook event but do not retain or drain the mutable `event.messages` array afterward, while current docs say that surface is replyable.

## Solution Assessment

No as submitted. The shared helper is a plausible shape, but the best fix must be rebased onto current main and share the existing sender, send-policy, and hook-output hardening semantics.

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: When `OriginatingTo` is absent, this target falls back to `ctx.To` before `ctx.From`; on direct inbound turns `ctx.To` can be the bot-side address, so the hook reply can be sent to the bot instead of the user. The regression test should assert the sender fallback instead of the bot address.
  - confidence: 0.91
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new delivery runs before the current `sendPolicy` source-reply suppression decision, so a session configured with `sendPolicy: "deny"` can still emit hook replies through `routeReply`. Compute or pass the same denial decision before calling the helper and add a denial regression test.
  - confidence: 0.9
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Runtime hook handlers can still push malformed values despite the TypeScript type, and calling `.trim()` on every entry can throw during command or message hook reply delivery. Filter to strings and trim after narrowing so bad hook output cannot abort the turn.
  - confidence: 0.84
- **[P2] Rebase the reset hook wiring onto current main:** `src/auto-reply/reply/commands-core.ts:42`
  - body: The branch moves reset hook reply delivery through `commands-core.ts`, but current main owns that behavior in `commands-reset-hooks.ts`; the final patch needs to port the helper there or the reset/new half of this fix will not match the current code shape.
  - confidence: 0.82
