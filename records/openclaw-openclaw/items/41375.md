## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route hook replies to the sender fallback:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: When `OriginatingTo` is absent, `ctx.To` is typically the bot address and `ctx.From` is the user who sent the inbound message. This new hook reply path prefers `ctx.To` before `ctx.From`, and the added test locks in that wrong target, so direct-channel hook replies can be sent back to the bot instead of the user.
  - confidence: 0.93
- **[P1] Apply delivery policy before hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This call routes hook-generated outbound messages before the function resolves `sendPolicy` and source-reply suppression for normal delivery. A session configured with `sendPolicy: deny` or message-tool-only source replies can still emit an internal hook reply, so the helper needs the same suppression gate and a deny-policy regression test.
  - confidence: 0.9
- **[P2] Harden hook reply message normalization:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Runtime hooks mutate `event.messages`, so a buggy or hostile hook can push a non-string value. Calling `.trim()` on every entry throws before delivery is skipped or logged, which can abort reset/new hook processing or the fire-and-forget hook task instead of safely ignoring invalid messages.
  - confidence: 0.86
