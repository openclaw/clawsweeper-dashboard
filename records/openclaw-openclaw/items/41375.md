## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new drain sends `event.messages` through `routeReply` immediately after `triggerInternalHook`, before the send-policy/source-reply suppression decision that current main applies to outbound user delivery. A session configured to deny automatic sends can still emit hook-generated messages, so the hook drain needs to run through the same policy-aware route or be explicitly documented as an approved exception.
  - confidence: 0.91
- **[P2] Prefer the inbound sender over the bot address:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: For direct inbound turns without `OriginatingTo`, `ctx.To` is commonly the bot's own address and `ctx.From` is the user. This fallback sends hook replies to the bot identity; match the command path's sender-first ordering and update the test that currently asserts the bot address.
  - confidence: 0.93
- **[P3] Filter non-string hook messages before trimming:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Hook handlers are runtime-loaded JavaScript and can accidentally push a non-string value into `event.messages`; calling `.trim()` directly would throw and abort command hook reply delivery. Normalize to strings before trimming so malformed hook output is skipped or logged rather than crashing the turn.
  - confidence: 0.82
