## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route hook replies to the inbound sender:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: On a normal inbound turn, `ctx.To` is usually the bot/account address and `ctx.From` is the user. This fallback sends the new hook reply to `ctx.To` before trying the sender, and the added test asserts that wrong target, so WhatsApp/Telegram-style direct messages can be misrouted to the bot identity instead of the user.
  - confidence: 0.95
- **[P1] Honor sendPolicy before delivering hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new delivery runs before the PR's `resolveSendPolicy` check, so an internal `message:received` hook can call `event.messages.push()` and still send outbound text when the session is configured with `sendPolicy: deny`. Hook replies need to be gated by the same source-reply/send-policy suppression path as normal replies.
  - confidence: 0.92
- **[P3] Filter hook messages before trimming:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Hook handlers are runtime-loaded code, and a malformed handler can push a non-string value into `event.messages`. Calling `.trim()` before narrowing will throw and can abort command or message processing, so normalize to strings before trimming and routing.
  - confidence: 0.78
