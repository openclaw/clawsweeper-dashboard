## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor source delivery policy before hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This drains `message:received` hook messages immediately after `triggerInternalHook`, before the current dispatch path resolves `sendPolicy` and source-reply suppression. A hook can therefore send automatic text from a session configured with `sendPolicy: deny` or message-tool-only delivery; compute/reuse the same delivery policy before routing hook replies.
  - confidence: 0.92
- **[P1] Pass the current routeReply reply kind:** `src/auto-reply/reply/internal-hook-replies.ts:48-58`
  - body: Current main requires `routeReply` callers to provide `replyKind`, but the new helper calls it without one. After rebasing this branch onto main, the helper must pass the appropriate kind, likely `final` for this reply-drain path, so typechecking and reply lifecycle hooks stay correct.
  - confidence: 0.9
- **[P2] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: For a direct inbound turn, `ctx.To` is commonly the bot-side address and `ctx.From` is the user. This fallback sends hook replies to `ctx.To` before `ctx.From`, and the new WhatsApp test asserts that bot address; prefer the sender when `OriginatingTo` is absent and update the test expectation.
  - confidence: 0.91
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hooks are runtime JavaScript and can accidentally push non-string values into `event.messages`. Calling `.trim()` directly can throw in the awaited command-hook path and abort `/new` or `/reset`; filter/narrow to strings and log or skip invalid values before joining.
  - confidence: 0.82

## Security Review

Status: needs_attention
