## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-218`
  - body: This new drain runs inside the early internal-hook block and calls routeReply before resolveSendPolicy is computed later in the function. A session configured with sendPolicy: deny would still let an internal message:received hook send outbound text, so the drain needs to run through the same policy-suppressed delivery path as normal replies.
  - confidence: 0.95
- **[P2] Prefer the inbound sender over the bot address:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: When OriginatingTo is absent, ctx.To is typically the bot address the inbound message was sent to, while ctx.From is the user/chat that should receive the reply. This branch uses ctx.To before ctx.From, and the added WhatsApp test asserts that wrong target.
  - confidence: 0.93
- **[P3] Filter hook reply messages before trimming:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Runtime-loaded hook code can still push non-string values into event.messages despite the TypeScript type, and this helper calls trim() on every entry before narrowing. A malformed hook value can throw out of reset/message hook reply delivery instead of being skipped or logged.
  - confidence: 0.86
