## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: When `OriginatingTo` is absent, this fallback chooses `ctx.To` before `ctx.From`. Existing WhatsApp context uses `From` for the user and `To` for the bot-side address, so this can route hook replies to the wrong target; prefer the sender/effective route and update the matching test assertion.
  - confidence: 0.91
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-218`
  - body: This new path sends hook messages through `routeReply` without checking the normal `sendPolicy: deny` suppression decision. Sessions configured to deny outbound delivery can still emit hook-generated text, so gate hook reply delivery on the existing send-policy/suppression result and add denial coverage.
  - confidence: 0.9
- **[P2] Guard hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hook handlers are runtime code and can mutate `event.messages`; calling `.trim()` directly on each entry can throw after the hook runner has returned. Normalize to strings before trimming so hook reply delivery cannot break command or message processing.
  - confidence: 0.82
