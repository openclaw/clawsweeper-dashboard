## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: The new `message:received` hook reply target falls back to `ctx.To` before `ctx.From`. On direct inbound turns, `ctx.To` is commonly the bot-side address while `ctx.From` is the user, so this can send the hook reply to the bot identity instead of the sender; the sibling command hook path uses sender-before-bot fallback.
  - confidence: 0.93
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:214-218`
  - body: This routeReply call runs before the dispatch path resolves and applies `sendPolicyDenied`. A session configured with `sendPolicy: deny` is supposed to process inbound messages while suppressing automatic outbound delivery, but hook-generated replies can still be sent through this new path.
  - confidence: 0.9
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:27`
  - body: Internal hooks are runtime-extensible, so a JavaScript hook can accidentally push a non-string into `event.messages`. Calling `.trim()` directly on each value can throw during command or inbound hook reply delivery instead of ignoring the bad entry and continuing safely.
  - confidence: 0.82
