## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: The new target falls back to `ctx.To` before `ctx.From`, but on normal inbound channel turns `ctx.To` is often the bot address. Hook replies should prefer `OriginatingTo`, then the inbound sender, and the test should assert the sender address rather than the bot-side address.
  - confidence: 0.92
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:209-221`
  - body: This new path calls `routeReply` for hook-generated text before the existing `resolveSendPolicy` and source-reply suppression decision. A session configured with `sendPolicy: deny` can still emit hook replies, bypassing the outbound delivery policy that now suppresses normal replies and notices.
  - confidence: 0.9
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:26`
  - body: `event.messages` is mutable runtime hook output, so a buggy hook can push a non-string value. Filtering with `message.trim()` before checking the type can throw and abort command or inbound processing instead of safely ignoring malformed hook output.
  - confidence: 0.86
- **[P1] Rebase onto the current reset-hook routing contract:** `src/auto-reply/reply/commands-core.ts:42`
  - body: Current main moved reset hook delivery into `commands-reset-hooks.ts` and the route-reply contract now carries sender policy context and a required reply kind. This branch still edits the old `commands-core.ts` shape and its new helper calls `routeReply` without those current parameters, so the dirty branch cannot safely preserve current reset-hook behavior until it is rebased and adjusted.
  - confidence: 0.82
