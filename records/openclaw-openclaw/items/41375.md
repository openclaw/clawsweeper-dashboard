## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: For inbound turns without `OriginatingTo`, `ctx.To` is commonly the bot-side address and `ctx.From` is the user who sent the message. This new target order can send hook replies back to the bot, and the matching test currently asserts that wrong destination.
  - confidence: 0.93
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:211-220`
  - body: This new `routeReply` path runs from the internal hook block instead of the existing source-delivery suppression path, so `session.sendPolicy: deny` can still leak hook-authored outbound text. Reuse the resolved send-policy decision before draining hook messages.
  - confidence: 0.9
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:27`
  - body: Internal hooks are runtime JS, so a handler can accidentally push a non-string into `event.messages`. Calling `message.trim()` directly can throw, and command hook delivery awaits this helper, so one malformed hook value can abort `/new` or `/reset` processing.
  - confidence: 0.86
