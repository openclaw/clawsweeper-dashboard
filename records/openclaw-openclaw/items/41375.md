- **[P1] Rebase command hook changes onto current reset-hook module:** `src/auto-reply/reply/commands-core.ts:42`
  - body: Current main moved reset/new hook routing into `commands-reset-hooks.ts`, while this branch still changes the older `commands-core.ts` reset-hook shape and is live-conflicting. Rebuild that portion on the current module before merge so the command hook behavior is actually reviewed against the code that will land.
  - confidence: 0.9
- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: `ctx.To` is normally the bot-side address on an inbound turn, so `originatingTo ?? ctx.To ?? ctx.From` can route a hook reply back to the bot instead of the user. Match the sibling command path by falling back to `ctx.From` before `ctx.To`, and update the test expectation accordingly.
  - confidence: 0.94
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new hook delivery runs before the existing `sendPolicy` denial decision, so an internal hook can still send a visible reply in a session configured to suppress automatic delivery. Gate hook replies through the same policy decision as the rest of dispatch before calling `routeReply`.
  - confidence: 0.9
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hook handlers are runtime code, so a malformed non-string entry in `event.messages` would throw on `.trim()` and can abort command or hook reply delivery. Normalize to strings before trimming and skip invalid entries instead of letting one bad hook output crash the path.
  - confidence: 0.82
