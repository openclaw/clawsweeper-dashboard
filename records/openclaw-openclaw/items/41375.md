## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: When `OriginatingTo` is missing, this chooses `ctx.To` before `ctx.From`. For inbound channels, `ctx.To` is often the bot-side recipient; the command hook path already uses sender-first targeting, and the new test asserts the bot number, so message:received hook replies can go to the bot instead of the user.
  - confidence: 0.9
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213`
  - body: This new delivery path calls `routeReply` for hook-generated text before the normal send-policy suppression boundary is evaluated. A session configured with `sendPolicy: deny` can still emit hook replies, bypassing the outbound delivery control.
  - confidence: 0.86
- **[P2] Guard hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hook handlers are runtime code and can accidentally push non-string values. Calling `.trim()` on every entry can throw and abort command hook processing, so the helper should filter to strings before trimming and keep delivery failures isolated.
  - confidence: 0.82
