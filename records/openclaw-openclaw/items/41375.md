  - body: Current main requires `replyKind` on `routeReply` so reply-payload hooks know the delivery lane. This new helper call omits it, so the branch must be rebased and pass an explicit kind before the new delivery path satisfies the current contract.
  - confidence: 0.88
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:210-220`
  - body: The new `message:received` path routes hook-controlled text immediately after `triggerInternalHook`, before the send-policy/source-suppression decision that protects no-reply sessions. A session configured with `sendPolicy: deny` could still emit hook replies.
  - confidence: 0.9
- **[P2] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: For inbound direct-channel turns, `ctx.To` is often the bot-side address and `ctx.From` is the user. This fallback sends hook replies to `ctx.To` before `ctx.From`, and the added test asserts that wrong direction.
  - confidence: 0.87
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:27`
  - body: Hook handlers are runtime JavaScript, so malformed values can still be pushed into `event.messages`. Calling `.trim()` on every element can throw and abort command or message hook delivery instead of ignoring and logging the bad entry.
  - confidence: 0.8
