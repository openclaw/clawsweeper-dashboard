## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: The new `message:received` hook reply target falls back to `ctx.To` before `ctx.From`. On normal inbound provider turns, `ctx.To` can be the bot/account address, so hook replies can be sent to the bot identity instead of the user; match the command surface by preferring the inbound sender after `OriginatingTo`, and update the test expectation.
  - confidence: 0.93
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:211-226`
  - body: This call routes hook-generated text before the existing `resolveSendPolicy` decision is applied later in dispatch. A session or channel configured with `sendPolicy: deny` can still send outbound hook replies, so compute/enforce the policy before `routeReply` or pass the policy into the helper and skip denied delivery.
  - confidence: 0.9
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:25`
  - body: Runtime hook handlers are JavaScript and can push non-string values despite the TypeScript type. Calling `message.trim()` directly can throw and disrupt command or inbound processing; filter to strings and trim normalized values before joining.
  - confidence: 0.86
