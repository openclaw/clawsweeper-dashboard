## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: When OriginatingTo is missing, ctx.To is often the bot-side address on direct inbound turns. This fallback can route hook replies to the bot instead of the user, and the new test currently locks in that wrong target by expecting the To value rather than the sender.
  - confidence: 0.93
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new delivery call runs before the normal sendPolicy suppression path, while routeReply in this branch has no equivalent guard. Sessions configured with sendPolicy deny can still emit hook-generated messages, so hook replies need to share the normal policy decision.
  - confidence: 0.91
- **[P2] Guard hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hook handlers are runtime extensible, so a buggy hook can push a non-string into event.messages. Calling message.trim() directly can turn malformed hook output into a delivery-path exception instead of dropping or logging the invalid entry.
  - confidence: 0.82
