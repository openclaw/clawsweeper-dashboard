## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:214-218`
  - body: This new hook reply route runs before the dispatch path resolves and applies `sendPolicyDenied`. A session configured with `sendPolicy: deny` should still process inbound messages while suppressing automatic outbound delivery, but hook-generated replies can still be sent through this path.
  - confidence: 0.91
- **[P2] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: The hook reply target falls back to `ctx.To` before `ctx.From`. On direct inbound turns, `ctx.To` is commonly the bot-side address while `ctx.From` is the user, so this can send the hook reply to the wrong side; the sibling command hook path uses sender-before-bot fallback.
  - confidence: 0.93
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:27`
  - body: Internal hooks are runtime-extensible JavaScript, so a hook can accidentally push a non-string into `event.messages`. Calling `.trim()` directly can throw during hook reply delivery instead of ignoring the bad entry and continuing safely.
  - confidence: 0.84
- **[P1] Pass the current routeReply reply kind:** `src/auto-reply/reply/internal-hook-replies.ts:43-55`
  - body: Current main's `routeReply` contract requires a `replyKind`, and the current command hook drain passes `replyKind: "final"`. This new helper call omits that field, so the branch will not satisfy the current routing contract after rebase.
  - confidence: 0.86
