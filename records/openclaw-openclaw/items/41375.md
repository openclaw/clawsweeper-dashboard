## Summary

Keep open: current main and v2026.6.6 still miss the central `message:received` hook reply drain, but this PR is not merge-ready because the proposed drain bypasses send-policy suppression, can choose the bot address as recipient, and the branch is dirty against current main.

## What This Changes

The PR adds a shared internal hook reply-drain helper, wires it into `/new`, `/reset`, and `message:received`, and updates hook docs/tests to limit auto-delivery to reply-capable surfaces.

## Best Possible Solution

Rebuild the fix on current main so `message:received` hook replies drain through the policy-aware reply route, prefer the originating sender target, keep lifecycle-only hooks observational, and cover allow, deny, sender-direction, non-routable, and malformed-message cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new delivery runs immediately after `triggerInternalHook` and calls the direct reply route before `resolveSendPolicy` / `sendPolicyDenied` can suppress outbound delivery. A session configured with `sendPolicy: "deny"` can still emit hook-generated messages, so the drain needs to run through the same policy-gated path as ordinary replies and cover denied sessions.
  - confidence: 0.97
- **[P1] Prefer the inbound sender over the bot address:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: The fallback target uses `ctx.To` before `ctx.From`, but on inbound direct messages `ctx.To` is commonly the bot identity while `ctx.From` is the user. That makes the new hook reply path send to the wrong address, and the added test locks in the wrong expectation.
  - confidence: 0.97
- **[P3] Filter non-string hook messages before trimming:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Managed hook code is runtime-loaded, so a JavaScript hook can accidentally push a non-string into `event.messages`; calling `.trim()` on it will throw and can abort the command hook path. Normalize to strings before trimming, or skip malformed entries with a verbose log.
  - confidence: 0.82
