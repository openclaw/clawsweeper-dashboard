## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new drain sends hook-generated `event.messages` directly through `routeReply` from the `message:received` hook path, before the branch applies the normal send-policy/source-reply suppression decision. A session configured with `sendPolicy: deny` or message-tool-only visible replies could still emit an outbound hook reply, so this needs to run through the current policy-aware reply route or explicitly gate on the resolved policy.
  - confidence: 0.92
- **[P2] Prefer the inbound sender over the bot address:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: The fallback target uses `ctx.To` before `ctx.From`; on direct inbound turns `ctx.To` is commonly the bot identity, while `ctx.From` is the user who sent the message. This can route hook replies to the bot itself, and the new test currently asserts that wrong `ctx.To` target.
  - confidence: 0.9
- **[P3] Filter non-string hook messages before trimming:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hooks can be plain runtime JavaScript, so a handler can accidentally push a non-string into `event.messages`. Calling `.trim()` on every entry can throw and abort the command hook delivery path; normalize to strings before trimming and routing.
  - confidence: 0.78
