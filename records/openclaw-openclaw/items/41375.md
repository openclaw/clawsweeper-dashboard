## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Route hook replies to the inbound sender:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: The new `message:received` hook target falls back to `ctx.To` before `ctx.From`; on direct inbound channels `ctx.To` is commonly the bot identity, so this can send hook replies to the bot instead of the user. The new test also asserts the bot address, so CI would preserve the bug.
  - confidence: 0.9
- **[P1] Honor sendPolicy before delivering hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: The PR routes hook-generated replies before the normal outbound send-policy decision, so a session configured to deny sends can still emit hook replies.
  - confidence: 0.86
- **[P3] Filter hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:29`
  - body: `event.messages` is runtime-extensible hook data; calling `.trim()` on every value can throw if a hook accidentally pushes a non-string and can abort the interactive command path.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete outbound-policy bypass risk until internal hook replies are gated by the same send-policy decision as normal replies.

Concerns:

- **[medium] Hook replies can bypass send policy:** `src/auto-reply/reply/dispatch-from-config.ts:213`
  - body: The proposed `message:received` bridge calls `routeReply()` for hook messages before the normal send-policy check, which can allow outbound messages from sessions that should deny sending.
  - confidence: 0.86
