## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: On an inbound provider turn, `ctx.To` is usually the bot address and `ctx.From` is the user. This new hook-drain path uses `ctx.To` before `ctx.From`, so `message:received` hook replies can be sent to the bot instead of the sender; the existing reset hook path prefers `from` for that reason. Use `originatingTo ?? ctx.From ?? ctx.To` and update the test expectation.
  - confidence: 0.92
- **[P1] Apply send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:211-226`
  - body: `deliverInternalHookMessages()` calls `routeReply()` before this dispatch computes `sendPolicy` and `sourceReplyPolicy`. Sessions with `sendPolicy: deny` currently suppress final, tool, block, and handled-hook replies; this new path would still send hook text, bypassing the user-delivery policy. Move the drain after policy resolution or pass the resolved deny decision into the helper and add a deny test.
  - confidence: 0.88
- **[P2] Guard hook reply messages at the runtime boundary:** `src/auto-reply/reply/internal-hook-replies.ts:26`
  - body: Managed/internal hooks can be JavaScript, so `event.messages` is not guaranteed at runtime even though the TS type is `string[]`. Calling `.trim()` directly makes a non-string push throw and can abort `/new` or `/reset` hook processing; filter to strings before trimming and cover the error path.
  - confidence: 0.77
