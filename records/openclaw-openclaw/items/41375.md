## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: The new `message:received` hook reply target falls back to `ctx.To` before `ctx.From`, but on normal inbound channel turns `ctx.To` is usually the bot address. This can send the hook reply to the bot side, and the new test assertion locks in that wrong target.
  - confidence: 0.92
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-218`
  - body: This early hook block calls `deliverInternalHookMessages()` before the normal `sendPolicy` and source-reply policy gates are resolved, so a session configured with `sendPolicy: deny` can still send hook-generated outbound replies.
  - confidence: 0.9
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hook handlers are runtime extension points, so a malformed handler can push a non-string into `event.messages`; calling `.trim()` directly can throw and disrupt command or inbound processing instead of dropping the bad entry.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete outbound send-policy bypass risk for hook-generated replies, plus a lower-severity robustness issue from trusting hook output types.

Concerns:

- **[medium] Hook replies bypass send-policy denial:** `src/auto-reply/reply/dispatch-from-config.ts:213`
  - body: The PR routes internal hook replies from the early `message:received` hook block before the dispatch path resolves and applies `session.sendPolicy`, so denied sessions can still receive hook-generated outbound messages.
