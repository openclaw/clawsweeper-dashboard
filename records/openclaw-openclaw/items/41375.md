## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: On an ordinary inbound provider turn with no `OriginatingTo`, `ctx.To` is the bot/account address and `ctx.From` is the user. This new hook reply path sends to `ctx.To` first, and the added test locks in that wrong target, so hook replies can be misrouted instead of going back to the user.
  - confidence: 0.93
- **[P1] Honor sendPolicy before delivering hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new delivery runs immediately after the hook fires, before `resolveSendPolicy` suppresses automatic outbound replies. A session configured with `sendPolicy: deny` can still send hook-generated messages, so the new hook reply path needs to share the same policy gate as normal replies.
  - confidence: 0.88
- **[P2] Avoid crashing command turns on malformed hook messages:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Hook handlers are runtime extension code, and `triggerInternalHook` catches handler errors before returning the mutated event. Calling `.trim()` on every array entry after that means a buggy hook that pushes a non-string can still throw from the delivery helper and abort `/new` or `/reset` processing.
  - confidence: 0.77
