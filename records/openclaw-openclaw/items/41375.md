## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Prefer the inbound sender for hook reply targets:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: For inbound turns without `OriginatingTo`, `ctx.To` is commonly the bot or channel address and `ctx.From` is the user who sent the message. This fallback would send hook replies to the bot identity, and the new test currently asserts that wrong target instead of catching it.
  - confidence: 0.93
- **[P1] Gate hook replies with the outbound send policy:** `src/auto-reply/reply/dispatch-from-config.ts:213-218`
  - body: This drains and routes hook messages before the normal `sendPolicy` decision is applied. A session configured with `sendPolicy: deny` would still be able to emit hook-generated outbound messages, so the drain needs to run only after the same policy check allows delivery.
  - confidence: 0.9
- **[P2] Guard hook reply values before trimming:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hook handlers are runtime-loaded JavaScript, and a non-string value pushed into `event.messages` now throws at `.trim()` after `triggerInternalHook` has already isolated handler failures. Filter or normalize string values before trimming so hook reply delivery cannot crash command processing or the fire-and-forget path.
  - confidence: 0.79
