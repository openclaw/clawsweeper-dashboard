## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass the current routeReply reply kind:** `src/auto-reply/reply/internal-hook-replies.ts:48-58`
  - body: Current main made `replyKind` a required `routeReply` parameter and uses it for `reply_payload_sending` hook metadata. This new helper calls `routeReply` without it, so after the branch is reconciled with main the hook reply path will not satisfy the current delivery contract.
  - confidence: 0.92
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This delivers hook-controlled text before the branch computes `resolveSendPolicy`, so a session configured with `sendPolicy: deny` can still emit automatic hook replies. The hook drain needs to run only after the same send-policy/source-suppression decision normal automatic replies use.
  - confidence: 0.9
- **[P2] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: The fallback target uses `ctx.To` before `ctx.From`, but on direct-channel inbound turns `To` is commonly the bot address and `From` is the user. The added test also asserts the bot address, so this would lock in wrong-direction delivery.
  - confidence: 0.91
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Runtime hooks are JavaScript and can accidentally push non-string values despite the TypeScript event type. Calling `.trim()` directly on every entry can throw and abort the interactive hook reply path instead of skipping bad values and logging the failure.
  - confidence: 0.84
