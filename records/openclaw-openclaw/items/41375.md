## Summary

Keep open. Current main still does not drain `event.messages` for internal `message:received` hooks, but this maintainer-labeled PR is dirty against main and the proposed patch still has blocker-level delivery and send-policy concerns.

## What This Changes

The PR adds a shared internal hook reply drain, wires it into command and `message:received` paths, and updates hook docs to limit auto-delivery to reply-capable surfaces.

## Best Possible Solution

Port the hook-reply drain onto current main after send-policy resolution, route inbound replies to the originating sender first, preserve reset/new behavior, and document/tests the reply-capable surface contract.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main fires the internal `message:received` hook but never drains `event.messages` on that path, while reset/new command hooks still do. I did not run a live provider repro in this read-only review.

## Solution Assessment

No, the PR direction is useful but this implementation is not the best merge shape as-is. It should be rebased and changed to use sender-first routing plus the normal send-policy/source-reply suppression decision before delivering hook messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route hook replies to the inbound sender:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: The new `message:received` hook reply target falls back to `ctx.To` before `ctx.From`, but on normal inbound turns `ctx.To` is usually the bot identity. That means hook replies can be delivered to the bot address instead of the user; prefer `ctx.From` before `ctx.To` and update the test expectation accordingly.
  - confidence: 0.93
- **[P1] Honor sendPolicy before delivering hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:210-224`
  - body: This new delivery block routes hook-generated messages before the dispatch path resolves and applies `sendPolicy`/source-reply suppression, so sessions configured to deny automatic outbound replies can still send hook replies. Gate this path with the same policy decision used by normal replies and cover both allow and deny cases.
  - confidence: 0.87
- **[P3] Filter hook messages before trimming:** `src/auto-reply/reply/internal-hook-replies.ts:25`
  - body: `event.messages` can be populated by runtime hook code, but the helper calls `.trim()` on every entry before narrowing. A non-string value would throw and can abort command hook reply delivery; filter to strings before trimming so malformed hook output is skipped instead of crashing the turn.
  - confidence: 0.74
