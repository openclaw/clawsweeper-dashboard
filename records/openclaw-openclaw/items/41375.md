  - body: Current main requires `replyKind` on every `routeReply` call so reply-payload hooks know the delivery lane. This new helper omits it, while the current command-hook path passes `replyKind: "final"`, so the branch must be rebased and the helper updated to the current contract.
  - confidence: 0.91
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This drains and routes hook-controlled text immediately after `triggerInternalHook`, before the current `resolveSendPolicy` and source-reply suppression decision exists.   - confidence: 0.9
- **[P2] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: When `OriginatingTo` is absent, `ctx.To` is usually the bot address and `ctx.From` is the user. The command-hook path already uses sender-first targeting; this fallback can send `message:received` hook replies to the bot side instead of the inbound user.
  - confidence: 0.88
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hooks are runtime JavaScript, so a handler can accidentally push a non-string despite the TypeScript type. Calling `.trim()` directly on each entry can throw and abort the command or hook delivery path instead of skipping or logging malformed output.
  - confidence: 0.82
