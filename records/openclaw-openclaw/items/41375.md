Full review comments:

- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new `message:received` hook delivery calls the reply route before the current dispatch path resolves and applies `sendPolicy`. A session configured with `sendPolicy: deny` can still send hook-generated messages, so the drain needs to run after or through the policy-aware source reply decision with a deny test.
  - confidence: 0.92
- **[P1] Prefer the inbound sender over the bot address:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: When `OriginatingTo` is absent, `ctx.To` is the address the inbound message was sent to, commonly the bot identity, while `ctx.From` is the user. This fallback can send hook replies to the bot instead of the sender; use the sender-first ordering and update the test that currently asserts `ctx.To`.
  - confidence: 0.9
- **[P3] Filter non-string hook messages before trimming:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Runtime-loaded hook handlers can accidentally push non-string values despite the TypeScript interface. Calling `.trim()` directly can throw and abort the awaited command hook path, so normalize only strings before joining and sending.
  - confidence: 0.79
