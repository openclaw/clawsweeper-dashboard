Overall confidence: 0.9

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: The new target falls back to ctx.To before ctx.From, but on normal inbound channel turns ctx.To is often the bot address. Hook replies should prefer OriginatingTo, then the inbound sender, and the test should assert the sender address instead of the bot-side address.
  - confidence: 0.92
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This path calls deliverInternalHookMessages, and then routeReply, before the existing resolveSendPolicy and suppression decision. A session configured with sendPolicy: deny can still emit hook-generated text, bypassing the outbound delivery policy used for normal replies.
  - confidence: 0.9
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: event.messages is mutable runtime hook output, so a buggy hook can push a non-string value. Calling message.trim() first can throw and abort command or inbound processing instead of safely ignoring malformed hook output.
  - confidence: 0.86
- **[P1] Rebase onto the current reset-hook routing contract:** `src/auto-reply/reply/commands-core.ts:42`
  - body: Current main moved reset hook delivery into commands-reset-hooks.ts and routeReply now carries requester metadata and a required reply kind. This branch still edits the old commands-core hook path, so the conflicting head cannot safely preserve current reset-hook behavior until it is rebased and adjusted.
  - confidence: 0.82
