## Review Findings

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218`
  - body: When OriginatingTo is missing, ctx.To is often the bot-side address on direct inbound turns. This fallback can route hook replies to the bot instead of the user, and the new test currently locks in that wrong target by expecting whatsapp:+2000 rather than the sender whatsapp:+1000.
  - confidence: 0.93
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213`
  - body: This new delivery call runs before resolveSendPolicy can suppress automatic outbound delivery, while routeReply in this branch has no sendPolicy guard. Sessions configured with sendPolicy deny can still emit hook-generated messages, so hook replies need to go through the same policy/suppression decision as normal replies.
  - confidence: 0.91
- **[P2] Guard hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hook handlers are runtime extensible, so a buggy hook can push a non-string into event.messages. Calling message.trim() directly will throw and fail delivery/command handling instead of dropping or logging the invalid entry.
  - confidence: 0.82
