## Label Justifications

- `P2`: This is a normal-priority hook reply delivery bug with real user-visible message loss but a bounded routing surface.
- `merge-risk: 🚨 compatibility`: Merging changes `message:received` hooks from silently collecting `event.messages` to automatically emitting outbound replies.
- `merge-risk: 🚨 message-delivery`: The proposed `ctx.To` before `ctx.From` fallback can send hook replies to the bot-side address instead of the inbound user.
- `merge-risk: 🚨 security-boundary`: The new hook `routeReply` call can bypass session send-policy denial because it runs before the normal suppression gate.

## Summary

This PR should stay open because the central hook-reply bug is still present on current main and the PR remains the canonical fix candidate, but the current branch is not merge-ready due to dirty merge state plus concrete routing, send-policy, and runtime-hardening regressions.

## What This Changes

The PR adds a shared internal hook reply helper, routes `/new`, `/reset`, and `message:received` hook `event.messages` through `routeReply`, updates hook docs, and adds focused regression tests.

## Best Possible Solution

Land a rebased, policy-aware `message:received` hook reply implementation that targets sender-first, honors send-policy denial, hardens message normalization, and keeps lifecycle-only hooks non-delivering unless separately approved.

## Review Findings

Full review comments:

- **[P1] Prefer the inbound sender for hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:218-220`
  - body: When `OriginatingTo` is missing, `ctx.To` is often the bot-side address on direct inbound turns. This fallback can route hook replies to the bot instead of the user, and the new test locks in that wrong target by expecting the `To` value rather than the sender.
  - confidence: 0.93
- **[P1] Honor send policy before routing hook replies:** `src/auto-reply/reply/dispatch-from-config.ts:213-226`
  - body: This new delivery call runs before the normal send-policy suppression path, while `routeReply` has no equivalent guard. Sessions configured with send-policy denial can still emit hook-generated messages, so hook replies need to share the normal policy decision.
  - confidence: 0.91
- **[P2] Validate hook messages before trimming them:** `src/auto-reply/reply/internal-hook-replies.ts:28`
  - body: Internal hook handlers are runtime extensible, so a buggy hook can push a non-string into `event.messages`. Calling `message.trim()` directly can turn malformed hook output into a delivery-path exception instead of dropping or logging the invalid entry.
  - confidence: 0.82
