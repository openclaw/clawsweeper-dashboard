## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the complete embedded tool-result `ReplyPayload` preservation path, and the PR is directionally useful, but the head patch can hide timeout failures after buffered replay and still needs a valid changelog attribution.

## What This Changes

The PR buffers embedded Pi runner tool-result `ReplyPayload`s, resets that buffer on compaction retry, prepends buffered payloads for non-streaming callers, adds regression coverage, and adds a changelog entry.

## Best Possible Solution

Land a rebased patch that captures complete non-empty tool-result `ReplyPayload`s for non-streaming callers, clears retry state, preserves timeout/error semantics before replay, and credits the changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main does not preserve full final tool-result `ReplyPayload`s, and the PR branch regression is reproducible with a mocked timed-out attempt that buffers a tool-result payload but produces no assistant payload.

## Solution Assessment

No. The implementation direction is maintainable, but replaying buffered payloads before timeout handling is not the narrowest safe fix, and the changelog entry still misses required contributor attribution.
