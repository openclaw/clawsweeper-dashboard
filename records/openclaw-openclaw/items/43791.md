## Summary

Keep open: the code direction looks plausible and the earlier concrete review gaps were mostly addressed, but this external bug-fix PR only provides mocked Vitest verification and needs real behavior proof before merge.

## What This Changes

The PR buffers embedded Pi runner tool-result payloads, clears them on compaction retry, and prepends them to final run payloads when live tool-result streaming is not active.

## Best Possible Solution

Land a rebased version that preserves non-streamed embedded tool-result payloads without duplicating live-streamed tool results, with focused regression tests plus redacted proof from a real embedded run or channel delivery path.

## Reproduction Assessment

Yes, at source level. Current main’s final embedded run assembly does not include `onToolResult` payloads, and the PR’s mocked regression path is a high-confidence reproduction of that contract gap, but I did not run a live external channel flow in this read-only review.

## Solution Assessment

Mostly yes. Buffering full `ReplyPayload` values in the embedded attempt layer and replaying them only for non-streaming callers is a narrow maintainable fix; the remaining requirement is real after-fix behavior proof, not a different code direction.
