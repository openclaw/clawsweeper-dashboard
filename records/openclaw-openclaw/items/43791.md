## Summary

Keep open: current main only preserves a narrower tool-media fallback, while this PR still carries unique full `ReplyPayload` preservation work; however the current PR head regresses prompt-timeout reporting and lacks required real behavior proof.

## What This Changes

The PR buffers embedded Pi tool-result `ReplyPayload`s, replays them into non-streamed final payloads, resets the buffer on compaction retry, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land a rebased fix that preserves full non-streamed tool-result `ReplyPayload`s without replaying live-streamed results, while keeping current tool-media fallback and explicit timeout-error behavior.

## Reproduction Assessment

Yes, at source level. Current main lacks the full `toolResultPayloads` return contract, and the PR head has a clear source-level timeout regression when a timed-out attempt has buffered tool-result payloads.

## Solution Assessment

No, not at the current PR head. Buffering at the attempt layer is plausible, but it must be rebased onto current `payloadsWithToolMedia` handling and preserve explicit timeout errors.
