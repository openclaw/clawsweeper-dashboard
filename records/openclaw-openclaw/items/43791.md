## Summary

Keep open: the underlying embedded-run payload loss is still plausible on current main, but this PR head is not merge-safe because it can hide prompt timeout errors after buffering a tool result and it still lacks required real behavior proof.

## What This Changes

The PR buffers embedded Pi runner tool-result ReplyPayloads during attempts, returns them for non-streamed embedded runs ahead of assistant payloads, resets the buffer on compaction retry, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Rebase over current main, preserve full non-streamed tool-result ReplyPayloads without satisfying timeout/error synthesis with buffered tool output alone, add focused timeout regression coverage, and require redacted real embedded-run delivery proof.

## Reproduction Assessment

Yes, source-reproducible: current main has no full tool-result ReplyPayload buffer for absent onToolResult callbacks, and the PR head shows the intended preservation path plus the timeout regression. I did not run tests because this was a read-only review.

## Solution Assessment

No, not at the current PR head. Buffering full ReplyPayloads for non-streamed callers is the right direction, but timeout error synthesis must remain visible and the change needs real behavior proof before merge.
