## Summary

Keep this PR open because it addresses a plausible embedded-runner payload loss, but the current patch still changes timeout semantics and lacks real behavior proof before merge.

## What This Changes

The PR buffers embedded Pi runner tool-result payloads, returns them with final embedded run payloads for non-streaming callers, resets the buffer on compaction retry, and adds focused regression tests plus a changelog entry.

## Best Possible Solution

Preserve the useful full-payload buffering idea, but rebase onto current main, keep timeout errors based on assistant/final-response state before adding buffered tool results, and add real behavior proof for a non-streaming embedded run that returns media or metadata.

## Reproduction Assessment

Yes, source-reproducible for the PR finding: in PR head, a timed-out non-streaming run with buffered tool payloads makes `payloads.length > 0` before the timeout guard. I did not run tests because this review is read-only.

## Solution Assessment

No. Buffering the payloads is a reasonable direction, but the patch should preserve timeout semantics and integrate with current main's newer media fallback path before it is a safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Preserve timeout errors when only tool results are buffered:** `src/agents/pi-embedded-runner/run.ts:1523`
  - body: `payloads` now includes buffered tool-result payloads before the timeout guard checks `payloads.length === 0`. If a non-streaming run emits a tool result and then times out without an assistant reply, the PR returns that partial tool payload as success instead of the explicit timeout error. Gate the timeout branch on assistant/final-response payloads, or prepend buffered tool results only after the timeout decision.
