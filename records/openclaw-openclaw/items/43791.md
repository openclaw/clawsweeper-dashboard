## Summary

Keep this PR open, but do not merge the current head. Current main still lacks full embedded tool-result `ReplyPayload` preservation, so the PR is not obsolete; however, the branch can hide timeout failures by counting buffered tool-result payloads before the timeout guard runs, and the user-facing changelog entry still lacks required contributor attribution.

## What This Changes

The PR buffers embedded Pi runner tool-result `ReplyPayload`s, resets that buffer on compaction retry, prepends buffered payloads to non-streamed final run payloads, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased patch that captures complete non-empty tool-result `ReplyPayload`s for non-streaming callers, clears retry state, preserves timeout/error semantics, and credits the changelog entry.

## Reproduction Assessment

Yes. The original bug is reproducible with a mocked embedded attempt that emits a tool-result payload plus assistant text and then inspecting `runEmbeddedPiAgent(...).payloads`; the branch regression is reproducible with a timed-out mocked attempt that buffers tool output but produces no assistant reply.

## Solution Assessment

No. The implementation direction is reasonable, but the payload replay must happen only after timeout/failure handling cannot be hidden, and the changelog must satisfy repo attribution policy.
