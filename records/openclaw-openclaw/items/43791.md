## Summary

Keep open: current main still lacks full embedded tool-result payload replay, and this PR is aimed at a real bug, but the current head can mask prompt timeouts by counting buffered tool-result payloads before timeout handling and its changelog entry still lacks required attribution.

## What This Changes

The PR buffers embedded Pi runner tool-result `ReplyPayload`s, resets that buffer on compaction retry, prepends buffered payloads to final non-streamed run payloads, updates subscribe types, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased patch that captures complete non-empty tool-result `ReplyPayload`s for non-streaming callers, clears retry state, preserves timeout/error semantics before replay, and carries valid changelog credit.

## Reproduction Assessment

Yes. The original bug has a high-confidence mocked path: emit a tool-result payload plus assistant text from an embedded attempt and inspect `runEmbeddedPiAgent(...).payloads`; the PR regression is source-reproducible with a timed-out attempt that buffered tool output but produced no assistant reply.

## Solution Assessment

No. The implementation direction is maintainable, but replaying buffered tool-result payloads must happen after timeout/failure handling cannot be hidden, and the changelog must satisfy repo attribution policy.
