## Summary

Keep open: the PR targets a real embedded-run payload gap, but it should not merge as-is because it adds a required harness result field without a backward-compatible default and can let buffered tool summaries mask prompt timeouts.

## What This Changes

The PR buffers embedded tool-result reply payloads, prepends them to final embedded run payloads when they were not streamed live, resets the buffer on compaction retry, and adds regression tests plus a changelog entry.

## Best Possible Solution

Adapt the payload preservation on top of current main’s tool-media flow so non-streamed tool-result payloads survive without breaking plugin harness results or timeout error semantics.

## Reproduction Assessment

Yes, by source inspection. Current main only carries assistant payloads plus pending tool media into the final result, while `onToolResult` text/channelData is live-callback gated and is not retained for non-streaming callers.

## Solution Assessment

No. The direction is reasonable, but the current PR shape is not the narrowest maintainable fix because it changes the harness result contract incompatibly and folds buffered tool summaries into the timeout decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep new attempt payloads backward compatible:** `src/agents/pi-embedded-runner/run/types.ts:46`
  - body: `AgentHarnessAttemptResult` is exported through the plugin harness SDK, so adding a required `toolResultPayloads` field makes existing harnesses return a shape that `run.ts` will later spread as `undefined`. Make this optional/default it in `normalizeEmbeddedRunAttemptResult` before reading it, or update the contract with a compatible fallback.
  - confidence: 0.86
- **[P2] Preserve timeout errors when only tool summaries exist:** `src/agents/pi-embedded-runner/run.ts:1520`
  - body: The final payload list now includes buffered tool-result text before the timeout guard runs. A verbose text-only tool summary emitted before the model stalls makes the payload list non-empty, so the user gets a partial-looking success instead of the explicit timeout error. Base the timeout branch on assistant/media deliverables, or append buffered summaries after preserving the timeout error.
  - confidence: 0.78
