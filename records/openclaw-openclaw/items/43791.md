## Summary

Keep this PR open. Current main still lacks a full embedded tool-result `ReplyPayload` preservation path for final non-streaming embedded Pi run results, so the PR addresses a real remaining gap, but the current branch can turn a timed-out run with only buffered tool-result output into a successful-looking final payload and its changelog entry is missing required attribution.

## What This Changes

The PR buffers embedded Pi runner `onToolResult` payloads, prepends them to final non-streaming run payloads, clears the buffer on compaction retry, extends related attempt/subscribe types, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the payload-preservation direction, but update the branch so complete buffered `ReplyPayload` values are available to non-streaming final consumers only after timeout/error state is decided. Preserve the no-duplicate streamed path, keep compaction retry buffer clearing, rebase onto current main, and add the required changelog credit.

## Reproduction Assessment

Yes. The original gap is reproducible with a mocked embedded attempt that emits a tool-result payload plus assistant text and then inspecting `runEmbeddedPiAgent(...).payloads`; current main has no `toolResultPayloads` contract. The PR regression is reproducible with a mocked timed-out attempt that has buffered tool-result payloads and no assistant payload.

## Solution Assessment

No. The implementation direction is maintainable, but the current branch assembles buffered tool-result payloads before the timeout guard. A safer fix should preserve timeout/error outcomes first, then add buffered tool-result payloads where they do not hide run failure state.
