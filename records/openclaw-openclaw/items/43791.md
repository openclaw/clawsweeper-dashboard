## Summary

Keep open: the PR addresses a real embedded-run final payload loss and the discussion shows earlier review gaps for singular media URLs, metadata, duplicate live streaming, and compaction retry reset were handled, but the latest diff still lets buffered tool-result payloads mask the existing prompt-timeout error path.

## What This Changes

The PR buffers embedded `ReplyPayload` tool results, resets that buffer on compaction retry, prepends buffered results to final embedded run payloads when they were not streamed live, adds regression coverage, and updates the changelog.

## Best Possible Solution

Preserve buffered tool-result `ReplyPayload`s for successful non-streaming embedded runs, but decide prompt-timeout/error synthesis from assistant payload state before prepending buffered tool results, with regression coverage for timeout after a tool result and no assistant reply.

## Reproduction Assessment

Yes, by source inspection. A mocked attempt with `timedOut: true`, no assistant payloads, no live `onToolResult`, and one buffered `toolResultPayloads` would skip the existing timeout-error branch after the PR prepends the buffered payloads before the guard.

## Solution Assessment

No. Full `ReplyPayload` buffering and avoiding replay when live streaming is present are the right direction, but the buffered payloads should not participate in the no-assistant prompt-timeout decision.
