## Summary

Keep open. Current main still has source-backed paths where embedded PI dispatch depends on a process-global hook runner, and the loader still preserves an older gateway hook runner across later default-mode registry activation; the related open hook issue overlaps, but this memory-lancedb report adds selected-memory-slot, hot-reload, cron, and concrete LanceDB evidence that should stay with the repair cluster.

## What This Changes

The issue reports that memory-lancedb registers typed `agent_end` and `before_prompt_build` hooks successfully, but embedded PI, hot-reload, and cron paths never dispatch them.

## Best Possible Solution

Fix the generic plugin runtime boundary so selected memory-slot and explicitly enabled hook-capable plugins remain visible to embedded PI, cron, restart, and hot-reload hook dispatch without hardcoding memory-lancedb or a channel.

## Reproduction Assessment

Yes by source inspection, but not by live execution: embedded PI gates both reported hooks on the global runner's `hasHooks(...)`, and current loader code plus tests show a stale-runner path can exist. The reporter also supplied concrete v2026.5.2 runtime steps, logs, and LanceDB observations.

## Solution Assessment

Yes, a generic loader/runtime fix is the best direction. A memory-lancedb-only workaround would leave the shared hook runner and startup-scoped registry behavior implicated by #76649 unresolved.
