## Summary

Keep this open. Current main still exposes Telegram group history and reply context without per-message bot-target metadata, and #72510 remains an open, unmerged implementation candidate that explicitly closes this issue. The community triage gate and multi-agent kit discussed in comments are useful adjacent workarounds, but they do not close the bundled Telegram/core prompt-context gap.

## What This Changes

Add explicit per-message and reply-chain targeting context for Telegram group chats so an agent can distinguish messages addressed to itself, another bot, or ambient group history.

## Best Possible Solution

Review #72510 and merge it if it cleanly adds optional target metadata, populates it from Telegram mention entities and reply-chain bot identity, renders it in inbound prompt context, and covers current-bot, other-bot, and ambient history cases. If that PR is not the right patch, replace it with the same narrow core-plus-Telegram boundary rather than closing the issue.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the prompt-context defect: in current main, a skipped Telegram group message addressed to another bot is stored and later injected as flat sender/body/timestamp history, so the active agent cannot see that it targeted a different bot. The model choosing to answer is probabilistic, but the missing routing signal is directly verifiable in source.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: the maintainable fix is an optional end-to-end target metadata path from channel ingestion through prompt rendering, with Telegram population first. Closing or queueing a duplicate repair while #72510 is open would be the wrong workflow.
