## Summary

Keep open: this draft, maintainer-labeled security guardrail PR still has source-reproducible false negatives in the raw socket and managed-proxy runtime mutation detectors.

## What This Changes

The PR adds raw socket and managed-proxy runtime mutation lint guards, registers them in boundary checks and package scripts, adds tests, and documents managed proxy coverage by surface.

## Best Possible Solution

Revise both guards to cover current raw socket and proxy runtime mutation forms, add regression tests for those forms, then keep the managed-proxy coverage table aligned with the landed behavior.

## Reproduction Assessment

Yes. Source inspection of the PR head and current main shows the raw socket matcher misses net.createConnection and net.Socket().connect, and the proxy mutation matcher misses current dynamic and aliased lifecycle mutations.

## Solution Assessment

No. The guardrail direction is reasonable, but the current detectors need to cover the existing call shapes before they are wired into boundary checks.

- **[P2] Classify createConnection and Socket.connect callsites:** `scripts/check-raw-socket-callsite-classification.mjs:108-115`
  - body: The guard only accepts `.connect` calls on raw module aliases, so current `net.createConnection(...)` and `new net.Socket().connect(...)` raw clients are not emitted. That includes allowlisted current callsites, which means new or moved raw sockets can bypass the classification check.
  - confidence: 0.91

Current proxy lifecycle code mutates `process.env[key]` in loops and aliases `GLOBAL_AGENT` through local variables, so several callsites listed as classified are not actually detected.
  - confidence: 0.88
