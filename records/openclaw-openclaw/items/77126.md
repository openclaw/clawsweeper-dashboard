## Summary

Keep open: the guardrail direction is useful and the new network CodeQL shard passes, but this PR currently introduces a blocking workflow selector regression that would skip existing CodeQL critical-quality shards for ordinary `src` and `extensions` changes.

## What This Changes

The PR adds CodeQL network-runtime boundary queries, workflow wiring, a changelog entry, and managed-proxy status documentation.

## Best Possible Solution

Keep the scoped CodeQL network guardrails, but make `network_runtime` additive so existing surface shards still run, then rerun exact-head workflow validation and secops review.

## Reproduction Assessment

Yes for the PR regression: source inspection plus a shell `case` sample reproduces the selector shadowing with representative gateway, plugin SDK, and channel paths. Runtime bug reproduction is not applicable because this PR changes CI guardrails and docs.

## Solution Assessment

No in its current form: adding a broad source-path case before the existing specific cases is not the narrow maintainable solution. The safer fix is to set `network_runtime` independently or move the broad network case after all existing shard-specific matches.
