## Summary

This is a well-scoped fix for a source-confirmed startup regression, and the supplied terminal proof exercises the real database-open path. The implementation preserves path validation and covers open, repair, and transaction paths; before merge, an owner should explicitly accept the change from fatal to warning-only permission hardening for every chmod failure.

## What This Changes

The PR makes state-database chmod hardening best-effort with deduplicated warnings and adds coverage for database open, schema repair, and write transactions when chmod is unsupported.

## Best Possible Solution

Keep the fallback inside the permission helper so open and repair share one path and validation remains fatal, while obtaining explicit owner confirmation that warn-and-continue is the intended policy for all chmod failures or narrowing the tolerated failure classification before merge.

## Reproduction Assessment

Yes at source level. Current main has an unguarded synchronous chmod on the database-open path, and the supplied process-preload ENOTSUP proof demonstrates the same failure and successful branch behavior, although it does not use a real NFS or Azure Files mount.

## Solution Assessment

Yes, conditionally. Moving best-effort handling into the chmod operation is narrower and more complete than wrapping individual callers because it preserves path validation and covers repair, but the breadth of errors allowed to continue needs explicit security-policy acceptance.
