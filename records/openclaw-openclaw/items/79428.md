## Summary

Keep open: current main still has a source-backed custom-provider onboarding path that writes a 4,000-token context window, which is below the documented generic custom-provider default and unsafe around the compaction reserve behavior described in the report.

## What This Changes

The issue asks OpenClaw onboarding to stop assigning custom provider models an undersized context window that can force immediate compaction on first use.

## Best Possible Solution

Align custom-provider onboarding with the documented safe provider default, preserve larger user-provided model limits, auto-heal existing too-low custom entries during reconfiguration, and cover the path with focused onboarding tests.

## Reproduction Assessment

Yes, by source inspection: `applyCustomApiConfig` on current main creates a non-Azure custom model with `contextWindow` equal to `CONTEXT_WINDOW_HARD_MIN_TOKENS`, currently 4,000. The compaction threshold and reserve-floor docs/source explain why that value is unsafe for the reported first-turn compaction behavior.

## Solution Assessment

No, the current implementation is not the best way to solve this issue because it writes a hard minimum as model capacity instead of a realistic safe default. A narrower fix should change onboarding/default-heal behavior without altering generic Pi compaction policy or adding DashScope-specific logic.
