## What This Changes

Telegram forum-topic replies were requested to retry once without message_thread_id when Telegram rejects the topic as missing.

## Best Possible Solution

Keep the forum topic work on the canonical safer path: preserve topic routing and fail visibly for stale explicit forum topics rather than posting the reply into the group/general context.

## Reproduction Assessment

Yes from source and discussion: current main sends forum replies with message_thread_id and now propagates thread-not-found failures, while multiple issue comments report real Telegram forum topics where the threaded send fails and a threadless direct API call succeeds. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No for the requested threadless retry; it conflicts with the current fail-closed safety boundary because it can deliver topic-scoped content to a broader group surface. The better solution is the superseding routing/fail-closed work in https://github.com/openclaw/openclaw/pull/76920.
