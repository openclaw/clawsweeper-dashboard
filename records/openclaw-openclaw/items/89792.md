## Summary

## What This Changes

The PR changes `GatewayClient` teardown so fire-and-forget `stop()` unrefs the underlying `ws` socket immediately, while awaited `stopAndWait()` defers the unref until teardown settles and adds regression tests for both timings.

## Best Possible Solution

Land the split shutdown fix after maintainers accept the guarded private `ws._socket` dependency and receive or explicitly waive real proof for the awaited `--json`/`stopAndWait()` path.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. The reporter supplied concrete process/fd evidence, and current source plus the latest release show gateway-client shutdown still closes the socket without unrefing it.

## Solution Assessment

Yes, the revised split is the best code shape I found: it avoids relying on a forced CLI exit, preserves awaited shutdown until settlement, and confines the private `ws` access behind a guarded helper. The remaining blocker is proof for the revised awaited path, not a line-level code defect.
