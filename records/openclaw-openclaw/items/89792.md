## Summary

Keep open: the linked CLI hang is credible, but this PR is not merge-ready because the socket unref is placed in shared shutdown code used by both fire-and-forget and awaited close paths, and the PR still lacks real after-fix CLI proof.

## What This Changes

The PR adds a `ws` internal socket `unref()` inside `GatewayClient.beginStop()` and a mocked regression test for `client.stop()`.

## Best Possible Solution

Keep the fix focused on natural-exit CLI teardown: unref only the fire-and-forget stop path or pass an explicit shutdown mode, preserve awaited `stopAndWait()` semantics, cover both paths, and require real `message send --json` proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The linked issue gives concrete epoll/socket evidence and current source shows the affected `message send` gateway shutdown path, but the PR still lacks a real after-fix command proof.

## Solution Assessment

No. Socket unref is a plausible ingredient for the CLI hang, but putting it in shared `beginStop()` is too broad because awaited shutdown callers also use that path.
