## Summary

Keep this PR open, but it is not merge-ready: it targets the right stuck-socket class, yet the unref is placed in the shared shutdown path and can change awaited `stopAndWait()` semantics, and the contributor has not provided real after-fix CLI/gateway proof.

## What This Changes

The branch adds a `ws` internal socket `unref()` after `ws.close()` in `GatewayClient.beginStop()` and adds a mocked regression test for `client.stop()`.

## Best Possible Solution

Separate natural-exit teardown from awaited teardown: make fire-and-forget `stop()` release the socket handle, keep `stopAndWait()` backed by a ref'd close or terminate path, and prove a real gateway/channel send exits with complete output.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The linked issue has socket/epoll evidence and source inspection confirms the affected gateway shutdown path, but the PR proof is only mocked tests and lint output.

## Solution Assessment

No. The requested direction is plausible, but putting `unref()` in shared `beginStop()` is too broad because it also affects callers that explicitly await `stopAndWait()`.

Full review comments:

- **[P1] Keep awaited shutdown ref'd:** `packages/gateway-client/src/client.ts:800`
  - body: This unref sits in `beginStop()`, which is shared by `stop()` and `stopAndWait()`. `stopAndWait()` is used before gateway requests resolve and by SDK transport close, and its timeout timer is also unref'd; once the socket is unref'd here, a short-lived process can have no ref'd handles left while the close promise is still pending, so Node may exit before cleanup and result/output completion. Move the unref to the fire-and-forget stop path or keep a ref'd fallback while `stopAndWait()` is awaiting.
