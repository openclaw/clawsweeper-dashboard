## Summary

Keep open; the bug is real enough to keep reviewing, but this branch is not merge-ready because the socket unref is in the shared shutdown path, current main has moved that path, and contributor real behavior proof is still missing.

## What This Changes

The PR adds a gateway-client WebSocket `_socket.unref()` during shutdown plus a focused mocked regression test for `openclaw message send` natural-exit behavior.

## Best Possible Solution

Separate natural-exit teardown from waited teardown, rebase onto current main's `pendingStop` shape, and prove a real `openclaw message send --json` exits with complete output.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was run in this read-only review, but the linked issue gives concrete socket/epoll evidence and source inspection confirms the `message send` path reaches the gateway shutdown code this PR changes.

## Solution Assessment

Is this the best way to solve the issue? No; unrefing the underlying socket is plausible for the hang, but putting it in shared `beginStop()` is too broad because waited shutdown callers need the process to stay alive until close or terminate settles.

Full review comments:

- **[P1] Keep awaited shutdown ref'd:** `packages/gateway-client/src/client.ts:800`
  - body: This unref sits in `beginStop()`, which is shared by `stop()` and `stopAndWait()`. Because `stopAndWait()` is awaited before gateway requests resolve and both shutdown timers are unref'd, a short-lived process can exit before the close/terminate path settles and before CLI JSON output is written. Keep the socket ref'd for waited shutdown or unref only the fire-and-forget path.
  - confidence: 0.88
