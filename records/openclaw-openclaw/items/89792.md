## Summary

Keep this PR open: it targets a real CLI hang, but the current patch is not merge-ready because the socket unref is added to shared shutdown logic that also backs awaited teardown, and the external contributor has not provided real after-fix CLI/gateway proof.

## What This Changes

The branch adds a `ws` internal socket `unref()` during `GatewayClient` shutdown and a mocked regression test for the reported `openclaw message send` hang.

## Best Possible Solution

Separate natural-exit teardown from awaited teardown: let fire-and-forget `stop()` release the socket handle, keep `stopAndWait()` backed by a ref'd close or terminate path, and prove the real CLI send exits cleanly.

## Reproduction Assessment

Yes, the linked issue gives a concrete `openclaw message send --channel discord ... --json` path with fd/process evidence, and current source shows the gateway client shutdown path. I did not run the live Discord/gateway repro in this read-only review, so this is source-reproducible rather than reproduced.

## Solution Assessment

No. Socket unref is a plausible natural-exit fix, but placing it in shared `beginStop()` is too broad because awaited `stopAndWait()` callers need teardown ordering to remain reliable.

- **[P1] Keep awaited shutdown ref'd:** `packages/gateway-client/src/client.ts:800`
  - body: This unref is in `beginStop()`, which is shared by `stop()` and `stopAndWait()`. `stopAndWait()` explicitly waits for socket close or terminate, but both its timeout and the force-terminate timer are unref'd; after this change a short-lived process can have no ref'd handles while awaited shutdown is still pending.
