## Summary

Keep open: this PR targets a real gateway CLI hang, but the patch still unrefs the socket in the shared `beginStop()` path before awaited gateway cleanup resolves, which can make short-lived CLI calls exit before result output is written. It needs a shutdown-mode fix or explicit owner acceptance plus real `openclaw message send --json` proof before merge.

## What This Changes

The PR adds a private `ws._socket.unref()` call after `ws.close()` in `GatewayClient.beginStop()` and adds a colocated regression test for the stop path.

## Best Possible Solution

Land a gateway-client teardown fix that releases lingering completed-send socket handles only after awaited result/output settlement is safe, with `stop()` and `stopAndWait()` coverage plus redacted `openclaw message send --json` proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The issue provides concrete command/process evidence, and source inspection confirms the reported CLI path reaches gateway-client shutdown, but the exact current-main CLI hang was not executed here.

## Solution Assessment

No. Unrefing the underlying socket is a plausible mechanism, but putting it in shared `beginStop()` before awaited gateway settlement is not the safest fix until `stopAndWait()` and real CLI `--json` behavior are covered.
