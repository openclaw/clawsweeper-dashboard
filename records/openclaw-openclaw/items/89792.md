## Summary

Keep open: the linked hang is a real CLI/gateway shutdown bug and current main/latest release still lack the requested socket unref, but this PR is not merge-ready because it changes shared awaited shutdown semantics and has no real after-fix CLI proof.

## What This Changes

The branch adds a WebSocket `_socket.unref()` call in `GatewayClient.beginStop()` and a mocked gateway-client regression test.

## Best Possible Solution

Separate natural-exit teardown from awaited teardown: let fire-and-forget `stop()` release the socket handle, keep `stopAndWait()` backed by a ref'd close or fallback path, and prove a real gateway/channel send exits with complete output.

## Reproduction Assessment

No live current-main reproduction was run in this read-only review. The linked issue has concrete socket/epoll evidence and source inspection confirms the affected gateway shutdown path, but the PR proof remains mocked-only.

## Solution Assessment

No. Socket unref is a plausible direction, but putting it in shared `beginStop()` is too broad because callers that explicitly await `stopAndWait()` rely on ordered teardown.
