## Label Justifications

- `P1`: The linked bug affects a real `openclaw message send` workflow by delivering messages but leaving the CLI process hung, and the PR changes gateway process teardown.

## Summary

Keep open: the PR targets a real gateway-client CLI hang, but the patch still unrefs the WebSocket socket inside the shared `beginStop()` path used by both `stop()` and `stopAndWait()`. Source inspection shows `openclaw message send --json` does not write output until the gateway call settles after `stopAndWait()`, so this can let a short-lived process exit before the result is printed; the new proof exercises `client.stop()` rather than the reported CLI/awaited shutdown path.

## What This Changes

The PR adds a `ws._socket.unref()` call after `ws.close()` in `GatewayClient.beginStop()` and adds a colocated regression test for socket-unref and missing-`_socket` behavior.

## Best Possible Solution

Separate natural-exit teardown from awaited teardown: release the socket handle only for fire-and-forget shutdown or an explicit post-result natural-exit mode, keep `stopAndWait()` backed by a ref'd close/terminate path, and prove the real message-send CLI prints complete output and exits promptly.

## Reproduction Assessment

No, we do not have a high-confidence live CLI reproduction from this read-only review. The reporter provided concrete process/fd evidence, and source inspection confirms the message-send path reaches gateway shutdown through `stopAndWait()` while current main leaves the WebSocket socket ref'd.

## Solution Assessment

No. Unrefing the underlying socket is a plausible mechanism, but putting it in shared `beginStop()` is too broad because the reported CLI path awaits `stopAndWait()` before printing JSON; a mode-specific teardown is safer.
