## Summary

Keep open: the revised patch now separates fire-and-forget `stop()` from awaited `stopAndWait()` and I do not see a line-level correctness blocker, but the external proof still does not show the reported `openclaw message send --json` flow or a real awaited-shutdown path after the latest revision.

## What This Changes

The PR changes `GatewayClient` teardown to unref the underlying `ws` socket immediately for `stop()`, defer it until `stopAndWait()` settles, and add socket-unref regression tests.

## Best Possible Solution

Land the split shutdown fix after maintainers accept or receive redacted proof that the reported JSON CLI send completes with full output and returns to the prompt, while keeping the guarded private `ws` access documented and tested.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. The issue has concrete process/fd evidence, and source plus `ws` inspection show the reported CLI path reaches gateway shutdown without socket unref on current main and `v2026.6.8`.

## Solution Assessment

Yes, the revised split is the best code shape I see for this owner boundary: it avoids `process.exit()`, preserves awaited shutdown, and limits the private `ws` dependency behind a guarded helper. The remaining gap is proof, not an identified line-level code defect.
