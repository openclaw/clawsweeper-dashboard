## Summary

Keep this PR open: the revised code looks like the right narrow shutdown fix, but the supplied real proof still only demonstrates the fire-and-forget `stop()` socket unref mechanism, not the revised awaited `stopAndWait()` or `openclaw message send --json` path.

## What This Changes

This PR changes `GatewayClient` teardown so `stop()` unrefs the underlying `ws` socket immediately while `stopAndWait()` defers unref until awaited teardown settles, with regression tests for both timings.

## Best Possible Solution

Land the split shutdown fix after maintainers accept the guarded private `ws` dependency and receive or explicitly accept redacted proof that the awaited JSON send path completes and exits naturally.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. The reporter supplied concrete process and fd evidence, and current source shows the reported CLI path reaches gateway shutdown without socket unref on current main and `v2026.6.8`.

## Solution Assessment

Yes, the revised split is the best code shape found: it avoids `process.exit()`, preserves awaited shutdown, and limits the private `ws` dependency behind a guarded helper. The remaining blocker is real behavior proof for the revised awaited path, not an identified line-level code defect.
