## Summary

Keep open: the security-hardening direction is useful, but the patch is not merge-ready because it distinguishes explicit non-owner loopback callers from omitted owner state while the loopback tool cache still collapses both states, and the PR still has only test-based proof.

## What This Changes

The PR applies the owner-only gateway core-tool denylist to explicit non-owner MCP loopback tool resolution, renames the shared denylist constant, and adds focused regression coverage.

## Best Possible Solution

Split the loopback tool-cache owner-state key into owner, non-owner, and unknown states, add a regression covering false versus undefined cache ordering, keep the explicit non-owner denylist, and require redacted real MCP loopback proof before merge.

## Reproduction Assessment

Yes, source-reproducible: after the PR, loopback `senderIsOwner: false` and omitted owner state produce different policy results, while `McpLoopbackToolCache` still keys both as `non-owner`. I did not run a live repro because this review is read-only.

## Solution Assessment

No as submitted. Applying the denylist in gateway tool resolution is the right boundary, but the fix also needs the cache-key split, a regression for false versus undefined cache ordering, and real MCP loopback proof.
