## Summary

Keep open: current main still has the source-reproducible MCP/claude-cli failure path, and open PR #77437 is already linked with closing syntax as the implementation candidate.

## What This Changes

The issue reports that `sessions_yield` always returns `"No session context"` on the MCP/claude-cli loopback path because gateway tool resolution does not provide run-bound yield context.

## Best Possible Solution

Review and land or replace PR #77437 with a focused MCP loopback yield-context fix and regression coverage for `tools/call` on `sessions_yield`.

## Reproduction Assessment

Yes from source inspection: MCP `tools/call` resolves `sessions_yield` through the loopback tool cache without `sessionId` or `onYield`, so the tool deterministically reaches the `No session context` branch. I did not execute a live claude-cli setup.

## Solution Assessment

Yes at the direction level: the narrow maintainable fix is to pass run-bound yield context into the MCP loopback tool path and use an MCP-appropriate `onYield` that records the paused run. The open closing PR is the right place to validate that implementation.
