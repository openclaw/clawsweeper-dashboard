## Summary

Keep open: current main still builds the MCP loopback tool surface without the `sessionId` and `onYield` values that `sessions_yield` requires, so the reported `No session context` failure is source-reproducible.

## What This Changes

Fix `sessions_yield` so gateway/MCP/claude-cli runtime calls receive session context and yield support instead of returning `No session context`.

## Best Possible Solution

Add a narrow gateway/MCP yield binding that derives the live requester session context, returns a successful yield result synchronously, and preserves existing subagent completion delivery without changing embedded-runner behavior.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: an MCP `tools/call` for `sessions_yield` executes a tool built without `sessionId` or `onYield`, and the tool deterministically returns `No session context`. I did not run a live claude-cli loopback session in this read-only review.

## Solution Assessment

Yes, the best fix direction is a narrow MCP/gateway integration fix that supplies the existing `sessions_yield` contract from the active run/session context. A broader product or config change is not needed because the documented tool behavior already exists for the embedded path.
