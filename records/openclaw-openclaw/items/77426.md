## Summary

Keep open: current main still has the reported source-reproducible MCP/claude-cli failure path, and open PR #77437 already references this issue with closing syntax.

## What This Changes

Fix `sessions_yield` so the MCP/claude-cli loopback runtime has the session context and yield callback needed to return `yielded` instead of `No session context`.

## Best Possible Solution

Implement a run-bound MCP yield context in the shared loopback path, with regression coverage proving `sessions_yield` returns `yielded` and records paused subagent state for the claude-cli runtime.

## Reproduction Assessment

Yes from source inspection: MCP `tools/call` resolves `sessions_yield` through the loopback gateway path without `sessionId` or `onYield`, so the tool deterministically returns `No session context`. I did not execute a live claude-cli setup.

## Solution Assessment

Yes at the direction level: the narrow maintainable fix is to pass run-bound yield context into MCP loopback tool resolution and use MCP/CLI yield semantics instead of embedded-runner abort semantics.
