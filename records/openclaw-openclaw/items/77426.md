## Summary

Keep open: current main still has the source-reproducible MCP/claude-cli failure path, and an open linked PR is already the right place to review the proposed fix before this issue is closed.

## What This Changes

The issue reports that `sessions_yield` returns `No session context` when called through the MCP/claude-cli loopback runtime because gateway tool resolution omits yield context.

## Best Possible Solution

Land a narrow fix that passes run-bound yield context into loopback tool resolution and records CLI yields through subagent state, with focused MCP regression coverage.

## Reproduction Assessment

Yes from source inspection: MCP `tools/call` resolves `sessions_yield` through a loopback resolver that never supplies `sessionId` or `onYield`, so the tool deterministically reaches `No session context`. I did not execute a live claude-cli setup.

## Solution Assessment

Yes at the direction level: the narrow maintainable fix is to resolve/pass run-bound yield context into the loopback path and use MCP/CLI yield semantics rather than embedded-runner abort semantics. The open linked PR is the right integration point if its code and tests validate that behavior.
