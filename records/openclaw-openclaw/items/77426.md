## Summary

Keep open: current main still has the reported MCP/claude-cli failure path, because the loopback gateway tool resolver builds `sessions_yield` without the run context that the tool requires.

## What This Changes

Fix `sessions_yield` for claude-cli/MCP loopback runs so parent agents can yield after spawning subagents instead of receiving `No session context`.

## Best Possible Solution

Add a run-bound MCP yield context for loopback tools and make `sessions_yield` record a yielded/paused CLI run without using embedded-runner abort semantics, with focused regression coverage for MCP `tools/call`.

## Reproduction Assessment

Yes from source inspection: MCP `tools/call` resolves `sessions_yield` through `McpLoopbackToolCache`, which never supplies `sessionId` or `onYield`, so the tool deterministically reaches the `No session context` branch. I did not execute a live claude-cli setup.

## Solution Assessment

Yes at the direction level: the narrow maintainable fix is to resolve/pass run-bound yield context into the loopback tool path and implement an MCP/CLI `onYield` that marks the run yielded. A channel-specific workaround would be less maintainable because the failure is in shared gateway MCP tool resolution.
