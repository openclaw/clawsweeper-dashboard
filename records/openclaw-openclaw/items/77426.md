## Summary

Keep open: current main still has the reported MCP/claude-cli `sessions_yield` failure path, and the linked closing PR is open and still needs correction plus real behavior proof before this issue can close.

## What This Changes

The issue reports that `sessions_yield` always returns `No session context` when invoked through the gateway MCP loopback path used by the `claude-cli` agent runtime.

## Best Possible Solution

Land a corrected gateway MCP loopback fix that threads run-bound `sessionId`/`onYield`, pauses the active yielded run, adds focused regression coverage, and includes real claude-cli/MCP proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the MCP loopback path constructs `sessions_yield` without `sessionId` or `onYield`, and the tool deterministically returns `No session context` when `sessionId` is missing.

## Solution Assessment

Yes at the direction level, but not done on main. The narrow maintainable solution is the linked gateway plumbing plus MCP-specific active-run pause semantics, with regression coverage and real claude-cli/MCP proof before merge.
