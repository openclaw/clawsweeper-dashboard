## Summary

Keep open. The PR targets a source-reproducible heartbeat behavior, but the patch is incomplete because it only changes summary helpers while scheduler selection, provider default backfill, system-prompt/ACP gates, docs, and tests still use conflicting heartbeat policy; after-fix real behavior proof is also missing.

## What This Changes

The PR changes `src/infra/heartbeat-summary.ts` to treat empty heartbeat config objects as disabled for heartbeat enablement and interval resolution.

## Best Possible Solution

Choose one heartbeat config contract, document it, and implement it through a shared resolver used by scheduler, status/summary, system prompt, ACP spawn, provider default backfill, doctor diagnostics, and regression tests.

## Reproduction Assessment

Yes, by source inspection: current main accepts `agents.defaults.heartbeat: {}` as a schema-valid object, treats it as truthy enablement, and resolves missing `every` to the default cadence. I did not run the gateway in this read-only review.

## Solution Assessment

No. A one-file empty-object check in `heartbeat-summary.ts` is not the best fix because runtime scheduling, provider defaults, prompt/ACP gates, docs, and tests still disagree on the effective heartbeat policy.

  - body: This branch is behind current main's defaults-wide heartbeat behavior: after the new empty-defaults check, the function still falls through to default-agent-only enablement instead of preserving `agents.defaults.heartbeat` for all listed agents.   - confidence: 0.86

Summary/status can report a different policy than the scheduler unless the empty-object rule is centralized in the runtime selection path.
  - confidence: 0.87

  - body: The empty-object disable rule only reaches summary and interval resolution. System prompt guidance and ACP spawn routing still compute heartbeat enablement with independent default-cadence fallbacks, so a config treated as disabled here can still look enabled on those surfaces.
