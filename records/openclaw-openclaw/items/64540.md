## Summary

Keep open. The PR targets a real heartbeat configuration problem, but the current one-file patch is not merge-ready because it changes only the summary/enabled/interval helpers while scheduler selection, system-prompt/bootstrap gating, and ACP spawn policy still use separate heartbeat enablement and cadence logic; the public docs also currently document `0m` as the disable contract rather than `{}`.

## What This Changes

The PR modifies `src/infra/heartbeat-summary.ts` so empty `heartbeat` objects are treated as disabled in the summary, enabled-policy, and interval helper paths.

## Best Possible Solution

Decide whether `{}` should be a supported heartbeat-disable contract. If yes, implement a shared empty-heartbeat predicate and use it consistently across summary helpers, scheduler selection, runtime cadence checks, system-prompt/bootstrap gating, ACP spawn policy, docs, and focused regression tests; if no, keep `0m` as the documented disable path and close or retarget the linked bug accordingly.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, `agents.defaults.heartbeat: {}` has no `every`, so the interval helper falls through to `DEFAULT_HEARTBEAT_EVERY`, and the PR's patch leaves other heartbeat policy paths with their own default fallbacks. I did not run tests because this review is read-only.

## Solution Assessment

No. The current PR is too narrow for the behavior it introduces; the maintainable fix is either a shared cross-path disable policy with docs/tests, or an explicit maintainer decision to keep `0m` as the only supported disable contract.
