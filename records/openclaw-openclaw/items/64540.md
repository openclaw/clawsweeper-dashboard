## Summary

Keep open. The PR targets a source-reproducible heartbeat behavior, but the one-file empty-object rule is incomplete, lacks after-fix real behavior proof, and conflicts with related heartbeat PRs that need a maintainer contract decision.

## What This Changes

The PR modifies `src/infra/heartbeat-summary.ts` so empty heartbeat objects are ignored in explicit-agent detection and treated as disabled for enablement and interval resolution.

## Best Possible Solution

Define one heartbeat disable and precedence contract, then implement it through shared policy used by scheduler, summary/status, provider default backfill, system prompt/bootstrap, ACP spawn, docs, and regression tests.

## Reproduction Assessment

Yes, source-reproducible: current main accepts `heartbeat: {}` as schema-valid, treats defaults heartbeat as truthy, and falls through to the default cadence. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The patch is too narrow because scheduler selection, provider default backfill, prompt/ACP gates, docs, and related PRs still disagree; a shared resolver plus maintainer contract decision is safer.

  - body: Current main enables all listed agents when `agents.defaults.heartbeat` is configured and no explicit heartbeat entries exist.   - confidence: 0.86

  - body: This helper now excludes per-agent `heartbeat: {}` from explicit-agent detection, but `heartbeat-runner.ts` still selects agents with its own truthy `entry.heartbeat` check. That can make status/summary say disabled while the scheduler still tracks the agent.
  - confidence: 0.87
- **[P2] Apply disablement to prompt and ACP gates:** `src/infra/heartbeat-summary.ts:65-66`
  - body: The empty-object rule only reaches summary and interval resolution. System prompt and ACP spawn still use independent default-cadence fallbacks, so a config treated as disabled here can continue to expose heartbeat guidance or relay policy elsewhere.
  - confidence: 0.84
