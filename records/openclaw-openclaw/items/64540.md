## Summary

## What This Changes

The PR changes heartbeat summary helpers so empty heartbeat objects are treated as disabled for enablement and interval resolution.

## Best Possible Solution

Make a maintainer decision on whether `{}` should be a supported heartbeat disable sentinel, then land a current-base change that applies the chosen rule across scheduler, summary/status, system prompt/bootstrap, ACP spawn, docs, and focused tests with upgrade proof.

## Reproduction Assessment

Yes for the source-level behavior: current main and v2026.6.6 both treat `agents.defaults.heartbeat: {}` as a truthy heartbeat config and fall through to the default cadence. I did not run a live gateway/token-cost reproduction in this read-only review.

## Solution Assessment

No. The PR is a plausible narrow mitigation, but the best fix is a maintainer-approved heartbeat config semantic applied across the scheduler, status summary, prompt/bootstrap paths, docs, and tests rather than a one-file summary helper change.

That leaves HEARTBEAT.md/system guidance visible on normal runs after heartbeats are disabled, so the same predicate needs to gate prompt/bootstrap behavior.
