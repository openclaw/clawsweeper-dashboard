## Summary

Keep open: the PR targets a real source-visible heartbeat behavior, but the patch is incomplete and changes an unsettled config contract without proof. It adds an empty-object rule in one helper while provider defaults, scheduler selection, prompt/ACP gates, docs, and related heartbeat PR semantics still diverge.

## What This Changes

The PR changes `src/infra/heartbeat-summary.ts` to treat empty heartbeat config objects as disabled in summary enablement and interval resolution.

## Best Possible Solution

Choose one heartbeat config contract, then implement it through a shared policy used before provider default backfill and by summary/status, scheduler, prompt/bootstrap, ACP spawn, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection of current main shows `heartbeat: {}` is schema-valid/truthy and falls through to the default cadence; I did not execute the gateway or tests in this read-only review.

## Solution Assessment

No. The PR introduces the empty-object rule in one helper but leaves provider-default backfill, scheduler selection, prompt/ACP gates, docs, and tests on different semantics; a shared resolver after maintainer contract approval is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Respect empty defaults before provider backfill:** `src/infra/heartbeat-summary.ts:53`
  - body: The linked repro is Anthropic-backed, and runtime config materialization applies provider defaults before these helpers. `applyAnthropicConfigDefaults` fills `agents.defaults.heartbeat.every` whenever it is undefined, so an authored `heartbeat: {}` can arrive here as `{ every: "30m" }` or `{ every: "1h" }` and bypass this empty-object check.
  - confidence: 0.86
- **[P2] Share the empty-object predicate with the scheduler:** `src/infra/heartbeat-summary.ts:36`
  - body: This helper now ignores `heartbeat: {}` when deciding explicit agents, but `resolveHeartbeatAgents` in the runner still treats any truthy per-agent heartbeat object as explicit. That makes scheduler selection diverge from the new summary policy, so the predicate needs to be centralized before this contract is introduced.
  - confidence: 0.88
- **[P2] Apply the same policy to prompt and ACP gates:** `src/infra/heartbeat-summary.ts:65`
  - body: The empty-object disable rule only reaches summary and interval resolution. System prompt guidance and ACP spawn routing still compute heartbeat enablement with their own default-cadence fallbacks, so a config meant to disable heartbeat can still be treated as enabled on those surfaces.
  - confidence: 0.84
