## Summary

Keep open: current main still treats an empty heartbeat object as a configured heartbeat with the default cadence, but this PR only changes the summary helper and leaves other heartbeat policy paths inconsistent.

## What This Changes

The PR changes `src/infra/heartbeat-summary.ts` so empty heartbeat objects are treated as disabled in summary enablement and interval resolution.

## Best Possible Solution

Either keep `0m` as the documented disable path and close/retarget the linked bug, or add a shared empty-heartbeat disable predicate and wire it through summary, scheduler selection, runtime cadence checks, system-prompt/bootstrap gating, ACP spawn policy, docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, an empty heartbeat object has no `every`, so interval and guidance helpers fall back to `DEFAULT_HEARTBEAT_EVERY`; I did not run tests because this review is read-only.

## Solution Assessment

No. The PR is too narrow for the behavior it introduces; the maintainable fix needs one shared policy across all heartbeat gates or an explicit decision to keep `{}` as default-style configuration rather than a disable signal.
