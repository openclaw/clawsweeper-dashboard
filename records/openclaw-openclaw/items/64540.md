## Summary

Keep open. The PR targets a real source-reproducible heartbeat config problem, but the patch is helper-only, conflicts with scheduler, prompt, ACP, and Anthropic defaulting paths, and still lacks after-fix real behavior proof.

## What This Changes

The PR changes `src/infra/heartbeat-summary.ts` so empty heartbeat config objects are treated as disabled during helper-level enablement and interval resolution.

## Best Possible Solution

Choose one empty-object heartbeat contract and implement it through a shared resolver used by scheduler, status/summary, system prompt/bootstrap, ACP spawn, provider defaults, docs, and focused tests.

## Reproduction Assessment

Yes, source-level. Current main accepts empty heartbeat objects, treats `agents.defaults.heartbeat` as enabled when truthy, resolves missing `every` to the default interval, and the scheduler only skips null intervals; I did not run a live gateway.

## Solution Assessment

No. The PR is a partial helper-only fix; the maintainable solution is a single heartbeat policy path aligned across provider defaults, scheduling, status, prompt/bootstrap, ACP relay, docs, and tests.

With defaults plus a per-agent empty heartbeat object, status and scheduling can disagree, so the scheduler and summary need the same resolver.
  - confidence: 0.9

  - body: After this change, the summary helper can say an empty defaults heartbeat disables heartbeats, but `heartbeat-system-prompt.ts` and `acp-spawn.ts` still compute heartbeat enablement through local fallback logic. That leaves normal runs or ACP relay behavior with heartbeat guidance that disagrees with the scheduler.
