## Summary

Keep open. The PR targets a real source-reproducible heartbeat behavior, but the branch changes only the summary helper, conflicts with current documented/related heartbeat semantics, leaves scheduler/prompt/ACP policy split, and still lacks after-fix real behavior proof.

## What This Changes

The PR modifies `src/infra/heartbeat-summary.ts` to treat empty heartbeat objects as disabled in explicit-agent detection, enablement, and interval resolution.

## Best Possible Solution

Define one heartbeat disable/absence contract, then implement it through a shared resolver used by config/default backfill, scheduler, status, prompt/bootstrap, ACP spawn, docs, and regression tests.

## Reproduction Assessment

Yes, source-level: current main accepts `heartbeat: {}`, treats truthy defaults heartbeat as enabled, and falls back to the default cadence when `every` is missing. I did not run a live gateway or remeasure provider token usage in this read-only review.

## Solution Assessment

No. The proposed patch is too narrow because it changes only one helper while scheduler selection, provider default backfill, system prompt, ACP spawn, docs, and related heartbeat work still disagree on the policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve defaults-wide heartbeat enablement:** `src/infra/heartbeat-summary.ts:52-54`
  - body: With a non-empty defaults heartbeat such as `{ target: "last" }` that omits `every`, this patch removes the current defaults-wide enablement branch and falls through to default-agent-only behavior. That disables non-default agents that currently inherit `agents.defaults.heartbeat`; keep defaults inheritance intact or move this decision into one shared resolver.
  - confidence: 0.88
- **[P2] Respect authored empty defaults before backfill:** `src/infra/heartbeat-summary.ts:65-67`
  - body: Anthropic defaults fill `agents.defaults.heartbeat.every` whenever it is undefined. If normalization runs before this helper, an authored `heartbeat: {}` can become non-empty before `isEmptyObject` sees it, so the reported Anthropic setup can still schedule at the provider default cadence.
  - confidence: 0.86
- **[P2] Keep runner selection on the same policy:** `src/infra/heartbeat-summary.ts:36`
  - body: This helper now ignores per-agent `heartbeat: {}` in explicit-agent detection, but `heartbeat-runner.ts` still uses its own truthy `entry.heartbeat` selection. That can make status/summary say disabled while the scheduler still tracks or wakes the agent; update runner selection through the same policy.
  - confidence: 0.88
- **[P2] Apply disablement to prompt and ACP gates:** `src/infra/heartbeat-summary.ts:65-67`
  - body: The empty-object rule only reaches summary and interval resolution. System-prompt and ACP spawn paths still use independent fallback logic, so a config treated as disabled here can still expose heartbeat guidance or relay policy elsewhere.
  - confidence: 0.84
