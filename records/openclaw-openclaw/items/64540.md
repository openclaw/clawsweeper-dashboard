Keep open: the branch targets a real heartbeat-cost problem, but it is not merge-ready because it only changes the summary helper while scheduler selection, prompt/ACP gates, provider defaulting, docs, tests, conflicts, and real behavior proof remain unresolved.

## What This Changes

The PR modifies `src/infra/heartbeat-summary.ts` to treat empty heartbeat config objects as disabled in helper-level enablement and interval resolution.

## Best Possible Solution

Choose the canonical empty-object heartbeat semantics, then implement one shared heartbeat policy resolver before provider defaulting and reuse it from scheduler, summary/status, prompt/bootstrap, ACP relay, docs, and regression tests.

## Reproduction Assessment

Yes, source-level. On current main, `agents.defaults.heartbeat: {}` is schema-valid, helper enablement treats it as truthy, interval resolution falls back to the default cadence, and the scheduler skips only null intervals; I did not run a live gateway or remeasure provider usage.

## Solution Assessment

No. The PR is a partial helper-only fix; the maintainable solution is one shared heartbeat policy path aligned across provider defaults, scheduling, status, prompt/bootstrap, ACP relay, docs, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve empty-heartbeat intent before provider defaults:** `src/infra/heartbeat-summary.ts:64-66`
  - body: The linked report is an Anthropic token-burn case, but Anthropic defaults fill `agents.defaults.heartbeat.every` whenever it is undefined. If the loaded config has already become `{ every: "30m" }` or `{ every: "1h" }`, this new empty-object check never fires, so the reported setup can still schedule heartbeats. The fix needs to preserve authored `{}` before provider defaulting or move the disable decision into the shared resolver used by that path.
  - confidence: 0.87
- **[P2] Use one heartbeat selector for scheduler and summary:** `src/infra/heartbeat-summary.ts:36`
  - body: This changes helper-level explicit-agent detection to ignore `heartbeat: {}`, but `heartbeat-runner.ts` still treats the same object as explicit when choosing scheduled agents. With defaults plus a per-agent empty heartbeat object, the runner can select only that agent while the helper says defaults apply, leaving the default-inheriting agents unscheduled and the empty per-agent entry able to inherit defaults. The scheduler and summary need the same resolver.
  - confidence: 0.9
- **[P2] Apply disabled semantics to prompt and ACP gates:** `src/infra/heartbeat-summary.ts:53-54`
  - body: After this change, the summary helper can say defaults `{}` disables heartbeats, but `heartbeat-system-prompt.ts` and `acp-spawn.ts` still compute heartbeat enablement through local fallback logic. That leaves normal runs or ACP relay decisions with heartbeat guidance/route behavior that disagrees with the scheduler, so the chosen empty-object rule needs to be applied through those gates too.
  - confidence: 0.84
