## Summary

Keep open: the PR targets a real heartbeat-cost problem, but it only changes the summary helper while current main still has separate scheduler, prompt, ACP, docs, and provider-default heartbeat policy paths that can disagree after merge.

## What This Changes

The PR changes `src/infra/heartbeat-summary.ts` so empty heartbeat config objects are treated as disabled in helper-level enablement and interval resolution.

## Best Possible Solution

Choose one heartbeat absence/disable contract and implement it through a shared resolver used by scheduling, status/summary, system prompt/bootstrap, ACP relay, provider defaults, docs, and focused tests.

## Reproduction Assessment

Yes, source-level: current main accepts empty heartbeat objects, treats defaults heartbeat as enabled when truthy, and resolves missing `every` to the default cadence. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The proposed helper-only change is not the best fix because the maintainable path is one shared heartbeat policy resolver aligned across runtime scheduling, status, prompt/bootstrap, ACP relay, provider defaults, docs, and tests.

With defaults plus a per-agent empty object, status and scheduling can disagree, so the runtime selector and summary need the same resolver.
  - confidence: 0.9

  - body: After this change, summary can say an empty defaults heartbeat disables heartbeats, but `heartbeat-system-prompt.ts` and `acp-spawn.ts` still compute heartbeat enablement locally and fall back to the default cadence. That leaves prompt/bootstrap or ACP relay behavior inconsistent with the new policy.
