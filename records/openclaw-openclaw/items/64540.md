## Summary

Keep open: current main and v2026.6.6 still treat an empty heartbeat config object as enabled/default cadence, so the useful behavior is not obsolete. This PR is not merge-ready because it changes an upgrade-sensitive config semantic only in the summary/interval helpers, leaves prompt guidance on the old rule, lacks real behavior proof, and is currently conflicting.

## What This Changes

This branch adds an `isEmptyObject` guard in `src/infra/heartbeat-summary.ts` so empty heartbeat config objects are excluded from explicit heartbeat agents and resolve to disabled/no interval.

## Best Possible Solution

Land a maintainer-approved heartbeat config semantic that applies consistently across scheduler, status/summary, system-prompt/bootstrap guidance, docs, and tests; otherwise keep the documented `every: "0m"` disable path as the supported behavior.

## Reproduction Assessment

Yes for the current source behavior: current main and v2026.6.6 both enable a truthy empty defaults heartbeat and fall through to the default `30m` cadence. I did not establish the reported token-cost impact with a live gateway run.

## Solution Assessment

No. The PR is a plausible narrow mitigation, but the best fix needs a maintainer-approved config semantic applied across scheduler, status/summary, system prompt guidance, docs, and regression tests.
