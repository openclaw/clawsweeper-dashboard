## Summary

Keep open: current main and the latest release still fall through to the default heartbeat cadence for `heartbeat: {}`, so the useful bugfix is not already implemented. This PR is not merge-ready yet because it changes heartbeat config semantics in one helper without aligning sibling prompt-gating/docs/proof, and GitHub reports the branch as conflicting.

## What This Changes

The PR modifies `src/infra/heartbeat-summary.ts` so empty heartbeat config objects disable heartbeat enablement and interval resolution instead of falling back to the default cadence.

## Best Possible Solution

Pick the intended heartbeat config contract, then implement it consistently across summary, runner selection, system-prompt gating, docs, and regression tests; if `{}` is not an accepted disable contract, keep the documented `every: "0m"` path instead.

## Reproduction Assessment

Yes for the current source behavior: main and `v2026.6.6` both enable a truthy empty defaults heartbeat and fall through to the default `30m` cadence. I did not establish the reported token-cost impact with a live gateway run.

## Solution Assessment

No. The PR is a plausible narrow mitigation, but the best fix needs a maintainer-approved config semantic applied across scheduler, status/summary, system prompt guidance, docs, and tests.
