## Summary

Keep open: the underlying `/tools` MCP inventory bug is still present on current main, but this PR is not ready to merge because the latest head still has merge-blocking correctness issues and unrelated installer changes on a sensitive plugin install path.

## What This Changes

The PR adds an `mcp` effective-tool source, appends materialized session MCP tools to `/tools`, and changes plugin package install peer-dependency handling plus its test expectation.

## Best Possible Solution

Land a focused `/tools` MCP inventory fix that mirrors the run path for naming, policy, and lifecycle behavior, covers cold and warm sessions, disposes materialized leases, and removes the installer dependency changes from this PR.

## Reproduction Assessment

Yes. Configure `mcp.servers`, start a new session, and run `/tools verbose` before an agent turn; current main renders only the core/plugin/channel inventory while the actual run path materializes MCP tools later.

## Solution Assessment

No. The proposed direction is close, but the current patch only lists MCP tools after a cached runtime exists, does not dispose the materialized MCP lease, and mixes in unrelated installer behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Create MCP inventory for cold sessions:** `src/auto-reply/reply/commands-info.ts:170-171`
  - body: This guard means `/tools` only appends MCP tools after a previous agent turn has already created and bound an MCP runtime. The reported reproduction runs `/tools verbose` in a new session before using MCP, so configured `mcp.servers` tools are still omitted.
  - confidence: 0.9
- **[P2] Dispose the materialized MCP lease:** `src/auto-reply/reply/commands-info.ts:180-183`
  - body: `materializeBundleMcpToolsForRun()` acquires a runtime lease and releases it only through the returned `dispose()`. This command only needs display metadata, so it should dispose after building the inventory or repeated `/tools` calls can keep active leases and delay idle cleanup.
  - confidence: 0.86
- **[P2] Keep peer stripping reachable:** `src/infra/install-package-dir.ts:65-72`
  - body: The new `peerDependencies.openclaw` removal sits after early returns that require workspace devDependencies to be rewritten. Packages with only an `openclaw` peer, or no workspace devDependency entries, still keep that peer and do not get the stated bloat fix.
  - confidence: 0.82
- **[P2] Align the npm install assertion:** `src/infra/install-package-dir.test.ts:323`
  - body: The test now expects `--omit=peer`, but the production command still omits only dev dependencies and the nearby comment says preserving other peers is intentional. As written, the test fails or pressures production toward dropping legitimate runtime peers.
  - confidence: 0.93
