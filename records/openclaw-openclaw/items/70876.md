## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep /tools from creating MCP runtimes:** `src/auto-reply/reply/commands-info.ts:175-180`
  - body: Resolving a session id does not make this read-only: `getOrCreateSessionMcpRuntime` can still create or refresh runtime state for that session, and `materializeBundleMcpToolsForRun` calls `getCatalog()`, which may connect transports and issue `tools/list`. Slash `/tools` should only peek an already-warm catalog and report cold state instead of starting MCP discovery from an informational command.
  - confidence: 0.9
- **[P2] Dispose the MCP materialization lease:** `src/auto-reply/reply/commands-info.ts:185-188`
  - body: `materializeBundleMcpToolsForRun` acquires a runtime lease and releases it only through the returned `dispose()` handle, but this handler maps `mcpTools.tools` and drops that handle. Each `/tools` call can leave an active lease and prevent normal idle cleanup for the session MCP runtime.
  - confidence: 0.88
- **[P2] Align the installer test with the production command:** `src/infra/install-package-dir.test.ts:323`
  - body: The test now expects npm to run with `--omit=peer`, but this PR's production command still omits only dev dependencies. The focused installer test will fail as written, so either the test expectation or the production command needs to be corrected before this branch can merge.
  - confidence: 0.95
- **[P2] Run openclaw peer stripping outside the devDependency path:** `src/infra/install-package-dir.ts:64-70`
  - body: The new `peerDependencies.openclaw` removal sits after early returns that require `devDependencies` with at least one `workspace:` entry. Plugins that only declare `peerDependencies.openclaw`, or have ordinary dev dependencies, will return before this block and still keep the peer that the change is meant to remove.
  - confidence: 0.9
