## Summary

Keep open: current main still omits configured MCP server tools from `/tools`, and this PR is a plausible fix path but still has blocking correctness issues, missing real behavior proof, and unrelated installer changes.

## What This Changes

The PR adds an `mcp` effective-tool source, appends materialized MCP server tools to `/tools`, and also changes plugin package peer-dependency install handling.

## Best Possible Solution

Land a focused `/tools` MCP inventory fix that mirrors the run path for cold and warm sessions, policy filtering, collision naming, and lease disposal, with installer behavior handled separately.

## Reproduction Assessment

Yes, source-reproducible from current main. Configure `mcp.servers`, start a new session, and run `/tools verbose` before an agent turn: the command renders only core/plugin/channel inventory while the run path materializes MCP tools later.

## Solution Assessment

No. The PR moves in the right area, but the latest patch only lists MCP tools after a runtime is already bound to the session, does not dispose the materialized lease, and bundles an unrelated installer change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Create MCP inventory for cold sessions:** `src/auto-reply/reply/commands-info.ts:173-174`
  - body: The added path only runs when `resolveSessionId()` finds an existing runtime. The reported flow runs `/tools verbose` in a new session before any agent turn creates and binds that runtime, so configured `mcp.servers` tools are still omitted.
  - confidence: 0.9

  - body: `materializeBundleMcpToolsForRun()` acquires a runtime lease and releases it through the returned `dispose()`. This command only needs display metadata, so repeated `/tools` calls can keep active leases and delay idle cleanup.
  - confidence: 0.88
- **[P2] Move peer stripping before manifest early returns:** `src/infra/install-package-dir.ts:64-72`
  - body: The new `peerDependencies.openclaw` removal sits after early returns that require workspace devDependencies to be rewritten. Packages with only an `openclaw` peer, or no workspace devDependency entries, still keep that peer and do not get the stated bloat fix.
  - confidence: 0.84
- **[P2] Align the npm install assertion:** `src/infra/install-package-dir.test.ts:323`
  - body: The test now expects `--omit=peer`, but the production npm command still omits only dev dependencies and the nearby comment says preserving other peers is intentional. As written, the test fails or pressures production toward dropping legitimate runtime peers.
  - confidence: 0.94
