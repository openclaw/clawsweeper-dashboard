## Summary

Keep this PR open. The underlying `/tools` MCP inventory bug is valid, but the latest patch still misses the cold-session reproduction, leaves a materialized MCP runtime lease undisposed, and includes unrelated plugin installer dependency-resolution changes that should not ride with this fix.

## What This Changes

The PR adds an MCP source group to the effective tool inventory, appends materialized session MCP tools to `/tools`, and also changes plugin install peer-dependency sanitization plus its test expectation.

## Best Possible Solution

Land a focused `/tools` MCP inventory fix that matches the run path for naming, tool policy, and lifecycle, works before the first agent turn in a configured session, disposes any materialized lease after collecting display metadata, and leaves plugin installer dependency handling to a separate PR.

## Reproduction Assessment

Yes. Configure `mcp.servers`, start a new session, and run `/tools verbose` before an agent turn; current main only renders `resolveEffectiveToolInventory()`, and the latest PR still skips MCP tools unless a session runtime has already been cached.

## Solution Assessment

No. The proposed direction is close, but the current patch does not cover cold sessions, does not release the materialized lease, and mixes in an unrelated installer change instead of keeping the fix scoped to MCP inventory.

  - body: This guard only appends MCP tools after a previous agent turn has already created and bound an MCP runtime. The reported flow runs `/tools verbose` in a new session before using MCP, so configured `mcp.servers` tools are still omitted.

  - confidence: 0.82

  - body: The test now expects `--omit=peer`, but the production command preserves peer dependencies and the nearby comment says that is intentional. As written, the test fails or pressures production toward dropping legitimate runtime peers.
