## Summary

Keep open: current main still omits configured MCP server tools from `/tools`, so the PR targets a valid bug, but the patch is not merge-ready because the MCP path misses cold sessions, leaks the materialized runtime lease, and mixes in unrelated plugin installer dependency changes.

## What This Changes

The PR adds an `mcp` effective-tool source, appends materialized session MCP tools to `/tools`, and changes plugin package install peer-dependency handling plus its test expectation.

## Best Possible Solution

Land a focused `/tools` MCP inventory fix that mirrors the run path for cold and warm sessions, naming, policy filtering, and lease disposal, leaving installer peer-dependency behavior to a separate PR.

## Reproduction Assessment

Yes. Configure `mcp.servers`, start a new session, and run `/tools verbose` before an agent turn; current main renders only the core/plugin/channel inventory while the run path materializes MCP tools later.

## Solution Assessment

No. The proposed direction is close, but the patch only lists MCP tools after a cached runtime exists, does not dispose the materialized MCP lease, and bundles unrelated installer behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Create MCP inventory for cold sessions:** `src/auto-reply/reply/commands-info.ts:170-171`
  - body: The `resolveSessionId` guard only lists MCP tools after a previous agent turn has already bound an MCP runtime. The linked reproduction runs `/tools verbose` in a new session before MCP has been materialized, so configured `mcp.servers` tools are still omitted.
  - confidence: 0.9
- **[P2] Dispose the materialized MCP lease:** `src/auto-reply/reply/commands-info.ts:182-185`
  - body: `materializeBundleMcpToolsForRun()` acquires a runtime lease and releases it only through the returned `dispose()`. This command only needs display metadata, so repeated `/tools` calls can keep active leases and delay idle cleanup.
  - confidence: 0.86
- **[P2] Keep peer stripping reachable:** `src/infra/install-package-dir.ts:65-72`
  - body: The new `peerDependencies.openclaw` removal sits after early returns that require workspace devDependencies to be rewritten. Packages with only an `openclaw` peer, or no workspace devDependency entries, still keep that peer and do not get the stated bloat fix.
  - confidence: 0.82
- **[P2] Align the npm install assertion:** `src/infra/install-package-dir.test.ts:323`
  - body: The test now expects `--omit=peer`, but the production command still omits only dev dependencies and the nearby comment says preserving other peers is intentional. As written, this test fails or pushes production toward dropping legitimate runtime peers.
  - confidence: 0.93
