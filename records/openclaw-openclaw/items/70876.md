## What This Changes

The PR adds an `mcp` effective-tool source, appends materialized configured MCP server tools to `/tools`, and also changes plugin package install peer-dependency handling plus a related test.

## Best Possible Solution

Land a focused `/tools` MCP inventory fix that mirrors the run path for MCP naming and final policy, covers cold and warm sessions, releases materialized leases, and leaves installer dependency behavior to a separate PR.

## Reproduction Assessment

Yes. Configure `mcp.servers`, start a fresh session, and run `/tools verbose` before an agent turn; current main only renders the core/plugin/channel inventory while the agent turn path materializes configured MCP tools later.

## Solution Assessment

No. The proposed direction is close, but this patch only lists MCP tools after a cached runtime exists, does not dispose the materialized MCP lease, and includes unrelated installer behavior.

- **[P2] Create MCP inventory for cold sessions:** `src/auto-reply/reply/commands-info.ts:170-171`
  - body: This guard means `/tools` only appends MCP tools after a previous agent turn has already created and bound an MCP runtime. The linked reproduction runs `/tools verbose` in a new session before using MCP, so configured `mcp.servers` tools are still omitted.

  - body: The new `peerDependencies.openclaw` removal sits after early returns that require workspace devDependencies to be rewritten. Packages with only an `openclaw` peer, or no workspace devDependency entries, still keep that peer and do not get the stated bloat fix.
  - confidence: 0.82

  - body: The test now expects `--omit=peer`, but the production command still omits only dev dependencies and the nearby comment says preserving other peers is intentional. As written, the test fails or pressures production toward dropping legitimate runtime peers.
