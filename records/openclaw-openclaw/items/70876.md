## Best Possible Solution

Keep the useful `/tools` fix, but rebuild it on current main using the existing warm-catalog read-only MCP projection pattern, dispose or avoid materialized runtime leases, split out installer dependency changes, and require redacted real MCP behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep /tools from creating MCP runtimes:** `src/auto-reply/reply/commands-info.ts:175-180`
  - body: Resolving a session id is not enough to make this read-only: `getOrCreateSessionMcpRuntime` can still create the runtime for a bound session, and `materializeBundleMcpToolsForRun` calls `getCatalog()`, which may connect transports and issue `tools/list`. `/tools` should peek the existing runtime/catalog and show a cold-session notice instead of starting MCP discovery from an informational command.
  - confidence: 0.9
- **[P2] Dispose the MCP materialization lease:** `src/auto-reply/reply/commands-info.ts:185-188`
  - body: `materializeBundleMcpToolsForRun` acquires a runtime lease and releases it only through the returned `dispose()` function, but this handler only maps `mcpTools.tools` and drops the handle. That leaves an active lease after `/tools`, preventing normal idle cleanup for the session MCP runtime.
  - confidence: 0.88
- **[P2] Align the installer test with the production command:** `src/infra/install-package-dir.test.ts:323`
  - body: The test now expects npm to run with `--omit=peer`, but the production command in this same PR still omits only dev dependencies. The focused installer test will fail as written, so either the test expectation or the production command needs to be corrected before this branch can merge.
  - confidence: 0.95
- **[P2] Run openclaw peer stripping outside the devDependency path:** `src/infra/install-package-dir.ts:64-70`
  - body: The new peer-dependency stripping block sits after early returns that require `devDependencies` with at least one `workspace:` entry. Plugins that only declare `peerDependencies.openclaw`, or have ordinary dev dependencies, will return before this block and still keep the peer that the change is meant to remove.
  - confidence: 0.9
