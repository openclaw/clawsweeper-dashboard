## Summary

Keep open. The underlying `/tools` MCP inventory bug is still present on current `main`, but this PR is not correct as-is because it only covers warm MCP runtimes, leaks the materialized runtime lease, and includes unrelated plugin installer dependency changes.

## What This Changes

The PR adds an `mcp` effective-tool source and appends materialized session MCP tools to `/tools`, while also changing plugin package peer-dependency install handling and its test.

## Best Possible Solution

Land a focused `/tools` MCP inventory fix that mirrors the runtime path for cold and warm sessions, policy filtering, collision naming, and lease disposal, with the installer changes removed or handled in a dedicated review.

## Reproduction Assessment

Yes, from source inspection. Configure `mcp.servers`, start a new session, and run `/tools verbose` before an agent turn: current `main` only renders core/plugin/channel inventory while the run path materializes MCP tools later.

## Solution Assessment

No. The PR moves in the right area, but the current patch misses the cold-session reproduction, does not dispose the materialized MCP lease, and bundles unrelated installer behavior into the fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Create MCP inventory for cold sessions:** `src/auto-reply/reply/commands-info.ts:173-174`
  - body: The added path only runs when `resolveSessionId()` returns an existing runtime. The reported flow runs `/tools verbose` in a new session before any agent turn creates and binds that runtime, so configured `mcp.servers` tools are still omitted.
  - confidence: 0.9
- **[P2] Dispose the materialized MCP lease:** `src/auto-reply/reply/commands-info.ts:185-188`
  - body: `materializeBundleMcpToolsForRun()` acquires a runtime lease and releases it through the returned `dispose()`. This command only needs display metadata, so repeated `/tools` calls can keep active leases and delay idle cleanup.
  - confidence: 0.88
- **[P2] Keep peer stripping reachable:** `src/infra/install-package-dir.ts:64-72`
  - body: The new `peerDependencies.openclaw` removal sits after early returns that require workspace devDependencies to be rewritten. Packages with only an `openclaw` peer, or no workspace devDependency entries, still keep that peer and do not get the stated bloat fix.
  - confidence: 0.84
- **[P2] Align the npm install assertion:** `src/infra/install-package-dir.test.ts:323`
  - body: The test now expects `--omit=peer`, but the production npm command still omits only dev dependencies and the nearby comment says preserving other peers is intentional. As written, the test fails or pressures production toward dropping legitimate runtime peers.
  - confidence: 0.94
