## Summary

Keep open. The report describes a real current-main data-loss path because `loadSessionStore()` can collapse unreadable or invalid session-store state to `{}`, but the PR's blanket empty-store guard is too broad and would disable legitimate orphan cleanup for agents whose `sessions.json` is validly empty.

## What This Changes

The PR adds an early memory-core dreaming cleanup guard that logs a warning and skips orphan transcript archival whenever an agent's loaded session store has no entries.

## Best Possible Solution

Make the cleanup failure-aware: skip orphan archival only when the session-store read is untrusted, while preserving orphan scanning for a valid empty store. Cover corrupt or unreadable store protection and valid empty-store cleanup with targeted tests, then add the required user-facing changelog entry.

## Reproduction Assessment

Yes. Current main is code-reproducible with invalid `sessions.json` plus old dreaming transcripts, because the loader returns `{}` and the scrubber treats every transcript as unreferenced; the PR regression is reproducible with a valid `{}` store plus an old unreferenced dreaming transcript that should still archive.

## Solution Assessment

No. The PR protects against the reported data-loss class, but `Object.keys(store).length === 0` is not a trustworthy failure signal; the narrower solution is to distinguish failed/corrupt/unreadable load state from a valid empty store.
