## Summary

Keep open: this PR targets a real source-level data-loss risk, but the proposed guard is too broad because it also disables orphan cleanup for a valid empty session store, and the external PR has no after-fix real behavior proof.

## What This Changes

This PR adds a memory-core dreaming cleanup guard that warns and skips orphan transcript archival when `loadSessionStore()` returns an empty store.

## Best Possible Solution

Make the cleanup failure-aware: skip archival only when the session-store read is untrusted, preserve cleanup for a valid empty store, and cover corrupt/missing-store protection plus valid-empty-store archival with focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a corrupt or missing `sessions.json` loads as `{}`, and current scrubber logic can treat every stale dreaming transcript as unreferenced; the PR regression is also source-reproducible with a valid `{}` store plus a stale orphan transcript.

## Solution Assessment

No. The direction is right for preventing data loss, but using `Object.keys(store).length === 0` as the failure signal is not the narrowest maintainable fix because valid empty stores should still be cleaned.
