## Summary

Keep open. The PR addresses a real memory-core data-loss path, but the proposed empty-object guard is too broad because current session-store loading returns `{}` for both corrupt/missing reads and legitimate empty stores, so the patch would skip valid orphan cleanup.

## What This Changes

The PR adds a warning and early `continue` in memory-core dreaming cleanup when an agent session store has no entries.

## Best Possible Solution

Keep the data-loss protection, but make it failure-aware: skip orphan archival only when the session store read/parse was untrusted, while allowing a successfully loaded empty store to continue stale dreaming transcript cleanup. Add regression tests for corrupt/invalid store protection and valid `{}` orphan cleanup, plus a single-line changelog fix entry.

## Reproduction Assessment

Yes. The reported data-loss path is reproducible by reasoning from current main with an invalid `sessions.json` plus old dreaming transcripts, and the PR regression is reproducible with a valid `{}` store plus an old unreferenced dreaming `.jsonl` file.

## Solution Assessment

No. The direction is right, but `Object.keys(store).length === 0` is not the narrowest maintainable signal because `{}` is also a valid empty store; a load-status or trust-aware helper is safer.
