## Summary

Keep open. The reported data-loss path is real because current `loadSessionStore()` collapses missing or invalid `sessions.json` reads to `{}`, but this PR's empty-object guard is too broad and would also skip stale orphan cleanup for a valid empty store. The prior Greptile and ClawSweeper comments correctly identified the same blocker, so the PR needs a failure-aware signal before merge.

## What This Changes

This PR adds an early memory-core dreaming cleanup guard that logs and skips orphan transcript archival when an agent's loaded session store is empty.

## Best Possible Solution

Make the cleanup failure-aware: skip orphan archival only when the session-store read is untrustworthy, while allowing an explicit valid `{}` store to continue scanning stale unreferenced dreaming transcripts. Cover corrupt/missing store protection and valid empty-store cleanup with targeted tests, and add the required user-facing changelog entry.

## Reproduction Assessment

Yes. Current main can be reasoned through with an invalid or missing `sessions.json` plus old dreaming transcripts, and the PR regression is reproducible with a valid `{}` store plus an old unreferenced dreaming `.jsonl` file that should still be archived.

## Solution Assessment

No. The direction protects against data loss, but `Object.keys(store).length === 0` is not the narrowest maintainable signal because it matches both failed loads and legitimate empty stores; the fix should expose or compute load trust explicitly.
