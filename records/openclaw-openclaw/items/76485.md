## Summary

Keep open. The PR is aimed at a real compaction/WebChat bug, but the current patch only expands the registry phase check and does not cover the production preflight path where no reply backend is attached and the caller still treats compacting phases as non-streaming.

## What This Changes

The PR changes `queueReplyRunMessage` to accept `preflight_compacting` and `memory_flushing` phases and adds registry-level tests with manually attached embedded backends.

## Best Possible Solution

Add production-path regression coverage for WebChat follow-ups during transcript-byte preflight compaction, then fix the queue/drain path so those messages are processed after compaction or get a clear busy/error response without adding new config.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible, yes: current main enters `preflight_compacting` before any embedded reply backend is attached, and the same-run steering path is gated on `isStreaming` which remains false for non-`running` phases. I did not run a live Gateway/WebChat reproduction.

## Solution Assessment

Is this the best way to solve the issue? No. Expanding accepted phases in `queueReplyRunMessage` is not enough unless the production preflight path also has a queue target or the follow-up queue reliably drains after compaction and session-id rotation.
