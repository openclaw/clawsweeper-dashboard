## Summary

Keep open. This PR targets a real preflight-compaction/WebChat queueing bug, but the current diff does not reach the production path: preflight compaction sets the reply operation phase without attaching a backend, and the caller still skips same-run steering while the operation is not considered streaming.

## What This Changes

The PR expands `queueReplyRunMessage` to accept `preflight_compacting` and `memory_flushing` phases and adds registry-level regression tests for those phase checks.

## Best Possible Solution

Add a regression that exercises the real WebChat/preflight-compaction route, then make queued follow-ups either steer into a valid active backend or reliably drain after compaction and session-id rotation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible, yes: current main enters `preflight_compacting` without a reply backend, while the same-run steering path is gated on `isStreaming` and still treats non-`running` phases as not streaming. I did not run a live Gateway/WebChat reproduction.

## Solution Assessment

Is this the best way to solve the issue? No. Expanding `queueReplyRunMessage` phases is not enough unless the production preflight path also has an attached queue target and caller path that can invoke it.
