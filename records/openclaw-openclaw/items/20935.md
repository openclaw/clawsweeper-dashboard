## Summary

Keep open: current main has a memory host lifecycle event journal, memory-wiki can bridge that journal, and memory-flush writes are append-only guarded, but there is still no all-memory-file mutation audit with why/session/user context or tamper-evidence. Prior discussion and the closed unmerged audit-log PR reinforce that this needs first-class memory/security design rather than cleanup closure.

## What This Changes

Request an append-only audit log for all agent memory file changes with changed content, timestamp, trigger or why, session or user context, and tamper-evidence.

## Best Possible Solution

Define a versioned memory mutation audit contract around every authoritative MEMORY.md and memory/*.md write surface, including privacy/redaction and integrity semantics, then implement it with focused tests and public memory docs.

## Reproduction Assessment

Not applicable. This is a feature request; source inspection shows current main lacks the requested mutation-audit capability rather than failing an existing documented contract.

## Solution Assessment

Unclear. An append-only audit trail is a plausible direction, but schema, redaction, retention, integrity, actor/session mapping, and authoritative write-surface ownership need maintainer security/product design before code work starts.

## Risks / Open Questions

- A partial audit layer could create misleading security expectations if it covers only promotion/dreaming events but not direct memory flushes, manual writes, imports, and plugin-owned memory paths.
- Logging raw memory diffs could leak sensitive user memory unless the design defines redaction, retention, access control, and content-summary or hash semantics up front.
- Tamper-evidence needs a clear threat model; normal JSONL append semantics alone do not protect against local file edits by the same user or process.
