## Summary

Keep open: current main has lifecycle memory events and memory-wiki can bridge the event journal, but there is still no all-memory-file mutation audit with why/session/user context or tamper-evidence. The Pinchy comment is useful prior art, and the closed unmerged audit-log PR reinforces that this needs first-class memory/security design rather than cleanup closure.

## What This Changes

The issue requests an append-only audit log for changes to agent memory files, including what changed, when, why, and session/user context.

## Best Possible Solution

Define a versioned memory mutation audit contract around every authoritative MEMORY.md and memory/*.md write surface, including privacy/redaction and integrity semantics, then implement it with focused tests and public memory docs.

## Reproduction Assessment

Not applicable. This is a feature request; source inspection shows current main lacks the requested mutation-audit capability rather than failing an existing documented contract.

## Solution Assessment

Unclear. An append-only audit trail is a plausible direction, but schema, redaction, retention, integrity, actor/session mapping, and authoritative write-surface ownership need maintainer security/product design before code work starts.

## Risks / Open Questions

- A partial audit layer could create misleading security expectations if it covers promotion/dreaming events but not direct memory flushes, manual writes, imports, and plugin-owned memory paths.
- Logging raw memory diffs could leak sensitive user memory unless the design defines redaction, retention, access control, and content-summary or hash semantics up front.
- Tamper-evidence needs a clear threat model; normal JSONL append semantics alone do not protect against local edits by the same user or process.
