## Summary

Keep open: current main and the latest release still provide a lifecycle-oriented memory event journal, not the requested all-write append-only mutation audit with actor, trigger, session/user context, and tamper-evidence semantics. The Pinchy comment, closed audit-log PR, broader provenance issue, trust-tagging issue, and draft Memory Audit review-queue PR are useful prior art or partial overlaps, but none is a shipped core fix for this issue.

## What This Changes

Request an append-only audit log for all agent memory-file changes, including what changed, when, why, session/user context, and tamper-detection support.

## Best Possible Solution

Design one canonical memory-mutation audit seam around every OpenClaw-owned durable-memory writer, persist structured records in SQLite-backed state, and document redaction, retention, access, deletion, and integrity verification semantics.

## Reproduction Assessment

Not applicable as a user-level bug reproduction: this is a new audit capability request rather than a failure of an existing documented contract. Source inspection gives high-confidence gap evidence because the current event types, memory write paths, docs, and release code are explicit.

## Solution Assessment

No, not as a direct workspace JSONL sidecar. The maintainable path is a maintainer-approved memory host/plugin mutation seam with SQLite persistence and an explicit privacy plus tamper-evidence model.
