## Summary

Current main and the latest release expose a best-effort memory lifecycle journal, but they still do not audit every durable memory mutation with actor, reason, session context, privacy controls, or tamper evidence; the recent maintainer-facing discussion also explicitly preserves this as a product and security design item.

## What This Changes

Add an append-only audit trail for agent memory-file mutations that records what changed, when, why, and relevant session or user context, with tamper-detection support.

## Best Possible Solution

Create one canonical memory-mutation event seam owned by the memory host/plugin contract, persist structured records in the appropriate SQLite store, cover every OpenClaw-owned durable-memory writer, and document access, redaction, retention, deletion, and cryptographic integrity semantics.

## Reproduction Assessment

Not applicable: this issue requests a new audit capability rather than reporting a failure of an existing documented contract. Source inspection does provide a high-confidence gap analysis because current event types and write paths are explicit.

## Solution Assessment

No, not as a direct extension of the current workspace JSONL file. The maintainable direction is a canonical plugin/host mutation seam with SQLite persistence and an explicit security and privacy model before individual writers are instrumented.

## Risks / Open Questions

- Audit records can expose sensitive remembered content, identities, prompts, session identifiers, and behavioral metadata unless access, redaction, retention, and deletion semantics are explicitly designed.
- A log stored beside editable workspace Markdown cannot by itself provide strong tamper evidence against an actor that can modify the workspace; the integrity threat model and key ownership need a security decision.
- Extending the current JSONL sidecar would conflict with the repository’s SQLite-only policy for OpenClaw-owned persistent runtime state and would deepen existing migration debt.
