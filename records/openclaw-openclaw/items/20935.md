## Summary

Keep open: current main and v2026.6.5 provide a best-effort lifecycle event journal for recall, promotion, and dreaming, but not a comprehensive mutation audit for `MEMORY.md` and `memory/*.md`. The Pinchy implementation is useful prior art, while the requested privacy, attribution, retention, storage, and tamper-evidence semantics still require explicit maintainer and security design.

## What This Changes

Add an append-only audit trail for agent memory-file changes that records what changed, when, why, and the applicable session or user context, with support for detecting tampering.

## Best Possible Solution

Design a canonical SQLite-backed mutation-audit service at the shared memory-write boundary, cover every supported writer, record minimized structured provenance and content hashes, define access/retention/privacy and integrity guarantees, expose verification through a deliberate API or CLI, and document the contract publicly.

## Reproduction Assessment

Not applicable: this issue requests a new audit capability rather than reporting broken existing behavior. Current source and v2026.6.5 provide a high-confidence capability-gap check.

## Solution Assessment

No: the proposed append-only text format is a useful requirements sketch, but the best repository-aligned solution needs canonical SQLite ownership, complete write-path coverage, data minimization, and an explicit integrity threat model.

## Risks / Open Questions

- A partial implementation could claim audit coverage while missing direct writes, memory flushes, promotion writes, plugin-owned writes, or external edits.
- Audit records can expose sensitive memory contents, user identity, prompts, and session context unless access, minimization, redaction, retention, and deletion semantics are defined.
- Tamper-evidence requires a named threat model and protected integrity root; ordinary application-level append behavior cannot prove that an administrator or compromised process did not rewrite both memory and its log.
- The current JSONL journal is legacy file-state architecture under current repository policy, so extending it would add migration debt rather than establish the canonical audit store.
