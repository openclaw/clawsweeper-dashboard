## Summary

Keep open: current main and v2026.6.8 still provide lifecycle-oriented memory host events, not the requested all-write memory mutation audit with actor, trigger, session/user context, and tamper-evidence semantics. The Pinchy comment is useful prior art, and the related Memory Audit and provenance items are partial overlaps rather than a shipped core fix.

## What This Changes

Request an append-only audit log for all MEMORY.md and memory/*.md changes, including what changed, when, why, session/user context, and tamper-detection support.

## Best Possible Solution

Design one maintainer-approved memory-mutation audit seam around every OpenClaw-owned durable-memory writer, persist structured records in SQLite/plugin-owned state, and document redaction, retention, access, deletion, and integrity verification semantics.

## Reproduction Assessment

Not applicable as a user-level bug reproduction: this is a new audit capability request rather than a failure of an existing documented contract. Source inspection gives high-confidence gap evidence because the current event types, memory write paths, docs, and release code are explicit.

## Solution Assessment

No, not as a direct workspace JSONL sidecar or narrow lifecycle-event extension. The maintainable path is a maintainer-approved memory host/plugin mutation seam with SQLite persistence and explicit privacy plus tamper-evidence semantics.
