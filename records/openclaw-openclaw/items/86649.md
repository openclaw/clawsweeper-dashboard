## Label Justifications

- `P1`: The linked regression affects live agent progress visibility for `claude-cli` users in WebChat and TUI, making long turns look hung even when final replies arrive.

## Summary

Keep this PR open: the parser change is a plausible narrow fix for the linked Claude CLI streaming regression, but the supplied proof still exercises synthetic records rather than a live Claude CLI stream or Gateway/WebChat/TUI progressive-output path.

## What This Changes

Adds a Claude CLI JSONL parser fallback that relays assistant records with explicit `message.stop_reason: null` as streaming deltas, plus parser regression tests.

## Best Possible Solution

Land the parser fallback only after redacted live proof confirms real Claude CLI assistant partial records produce progressive Gateway/UI deltas without duplicate final text, or deliberately narrow the fallback to bundled Claude CLI if maintainers do not want custom dialect backends included.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. The parser-level gap is source-reproducible because current main ignores assistant records, but the linked issue also has a maintainer non-repro comment, so live proof is still needed.

## Solution Assessment

Yes, if the raw Claude CLI shape is confirmed: the shared CLI JSONL parser is the narrow owner boundary used by spawned and live Claude CLI paths. The safer merge state is to pair this code with redacted live/raw-shape proof or narrow the fallback to bundled Claude CLI.
