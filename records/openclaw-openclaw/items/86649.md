## Label Justifications

- `P1`: The linked regression affects live agent progress visibility for `claude-cli` users in WebChat and TUI, making long turns look hung even when final replies arrive.
- `merge-risk: 🚨 compatibility`: The diff changes behavior for the public custom CLI backend dialect `claude-stream-json`, not only for the bundled Claude CLI provider.
- `merge-risk: 🚨 message-delivery`: The diff changes which JSONL assistant records are relayed as user-visible streaming deltas, where duplicate or false partial delivery is the key failure mode.

## Summary

Keep open: the parser patch is a plausible and well-scoped fix for the linked Claude CLI streaming regression, but the contributor proof still exercises synthetic JSONL through the production parser rather than a real Claude CLI stream or Gateway/WebChat/TUI path.

## What This Changes

The PR adds a Claude stream-json fallback that relays `assistant` records with explicit `message.stop_reason: null` as streaming deltas, plus parser regression coverage.

## Best Possible Solution

Land the parser fallback after redacted live proof confirms real Claude CLI assistant partial records produce progressive Gateway/UI deltas without duplicate final text, or deliberately narrow the fallback to bundled Claude CLI if maintainers do not want custom dialect backends included.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. The parser-level gap is source-reproducible because current main only relays Claude `stream_event` text deltas while the linked report and this PR target assistant-record partials.

## Solution Assessment

Yes, if the upstream/raw shape is confirmed: the shared CLI JSONL parser is the narrow owner boundary used by spawned and live Claude CLI paths. The safer merge state is to pair this code with redacted live/raw-shape proof rather than relying on synthetic fixtures alone.
