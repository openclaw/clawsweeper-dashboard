## Label Justifications

- `P1`: The PR targets a live Claude CLI agent streaming regression that can make WebChat and TUI turns appear hung until final output arrives.
- `merge-risk: 🚨 compatibility`: The diff changes behavior for custom CLI backends that opt into the existing `claude-stream-json` dialect, not only the bundled Claude CLI provider.
- `merge-risk: 🚨 message-delivery`: The diff changes which JSONL assistant records are relayed as user-visible streaming deltas, where false partials or duplicate text are the main failure modes.

## Summary

Keep open: the parser change is focused and likely correct, but the external PR still lacks live Claude CLI/Gateway/WebChat/TUI proof and it changes delivery semantics for custom `claude-stream-json` backends.

## What This Changes

The PR adds a Claude CLI JSONL parser fallback that turns assistant records with explicit `message.stop_reason: null` into streaming deltas and adds parser regression tests.

## Best Possible Solution

Land the shared parser fallback after redacted live/raw-shape proof confirms real Claude CLI assistant partials produce progressive deltas without duplicate final text, or deliberately narrow the guard if custom dialect backends should not inherit it.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. The parser-level gap is source-reproducible because current main ignores assistant records as streaming text, but the linked issue still needs redacted raw Claude stream-json proof.

## Solution Assessment

Yes if the raw Claude CLI shape is confirmed: the shared CLI JSONL parser is the narrow boundary used by the spawned and live Claude paths. The safer merge state is to pair the code with live/raw-shape proof and a maintainer decision on custom dialect inheritance.
