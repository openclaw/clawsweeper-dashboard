## Label Justifications

- `P1`: The PR targets a broken live agent/channel streaming workflow for Claude CLI users, but the current patch has a blocking delivery-semantics issue.
- `merge-risk: 🚨 compatibility`: The new fallback changes how existing Claude stream-json compatible backends interpret complete `assistant` records during upgrades.

## Summary

This PR should stay open, but not merge yet: it targets a real Claude CLI streaming gap, while the current fallback treats every Claude stream-json `assistant` record as a streaming partial even though the upstream SDK docs describe `assistant` as complete messages and `stream_event` as the partial stream.

## What This Changes

The PR adds a Claude CLI JSONL parser fallback that converts `assistant` records into incremental assistant deltas and extends parser tests for dedupe, non-Claude, tool-only, and mixed-content cases.

## Best Possible Solution

Keep the stream source as raw `stream_event` deltas, and add an assistant-record fallback only if it is gated by a documented or live-proven Claude CLI partial marker with regression coverage for normal complete assistant records.

## Reproduction Assessment

No high-confidence live reproduction is available in this review. Source inspection confirms current main only relays `stream_event` text deltas, and the linked issue provides logs, but the missing raw live Claude CLI JSONL is still the blocker for proving the proposed `assistant` fallback.

## Solution Assessment

No. The safer fix is to preserve the existing `stream_event` contract and only add an `assistant` fallback if live dependency proof shows those records are partial progress rather than complete assistant messages.
