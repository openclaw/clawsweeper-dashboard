## Label Justifications

- `P1`: The PR targets a user-facing Claude CLI streaming regression that makes long WebChat/TUI turns look hung even though final replies can still arrive.
- `merge-risk: 🚨 compatibility`: The diff changes the shared `claude-stream-json` parser used by custom compatible backends, not only the bundled Claude CLI route.
- `merge-risk: 🚨 message-delivery`: The new fallback can emit assistant-record text before the final result path, which can duplicate or prematurely deliver messages if the record is not a true partial.

## Summary

Keep open; the branch fixed the earlier absent-`stop_reason` parser concern, but it still lacks live Claude CLI Gateway/WebChat/TUI proof and the official Claude Code docs describe partial output as `stream_event` records rather than top-level `assistant` records.

## What This Changes

The PR adds a Claude CLI JSONL fallback that emits top-level `assistant` records with explicit `message.stop_reason: null` as streaming deltas, plus parser tests.

## Best Possible Solution

Capture a redacted live Claude CLI stream-json sample and WebChat or TUI output, then land the narrow parser rule only if top-level assistant records are the observed partial shape and the shared-dialect compatibility impact is intentional.

## Reproduction Assessment

No. The linked issue has detailed logs, but a maintainer comment says current main and the reported Windows/npm package shape both produced pre-final deltas, and this PR says live Claude CLI Gateway/WebChat/TUI was not tested.

## Solution Assessment

Unclear. The guarded fallback is plausible, but official Claude Code docs identify partials as `stream_event` records, so the best fix needs raw live evidence that top-level assistant records with `stop_reason: null` are the missing partials.
