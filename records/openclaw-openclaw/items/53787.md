The PR makes Feishu receive-id type resolution nullable for invalid targets and adds guards in send-target resolution and streaming reply startup.

## Best Possible Solution

Land a Feishu-owned fix that rejects truly invalid targets before Feishu API calls, preserves valid bare direct IDs, keeps resolveFeishuSendTarget's returned receiveIdType non-null, and adds regression coverage plus a changelog entry.

The linked runtime bug has clear steps in #53376, but this pass did not perform live Feishu API validation.

## Solution Assessment

No. The PR is aimed at the right Feishu plugin boundary, but the guarded helper should expose a non-null FeishuIdType to downstream callers and the change should include invalid-target tests plus the required changelog entry.
