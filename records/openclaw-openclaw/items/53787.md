## Summary

Keep open. Current main and v2026.5.7 still keep the permissive Feishu receive-id fallback, while this PR is the open implementation path for the linked Feishu bug but needs branch reconciliation, a type-contract fix, and real Feishu proof before merge.

## What This Changes

The PR makes Feishu receive-id type resolution nullable for invalid targets and adds send-target plus streaming guards before Feishu API calls.

## Best Possible Solution

Land or replace this with a Feishu-plugin fix that validates normalized send targets, preserves valid bare user IDs and open/chat IDs, exposes a non-null FeishuIdType from guarded send-target resolution, adds regression coverage, and includes real Feishu proof.

## Solution Assessment
