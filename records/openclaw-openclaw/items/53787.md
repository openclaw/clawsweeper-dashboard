## What This Changes

The PR makes Feishu receive-id type resolution nullable for invalid targets and adds guards in send-target resolution and streaming startup.

## Best Possible Solution

Land or replace this PR with a Feishu-owned fix that rejects invalid target/type derivation before API calls, preserves valid bare user IDs and existing chat/open_id routes, exposes a non-null send-target result after guards, and adds focused regression coverage.

## Reproduction Assessment

Yes for the PR defect at source level: the patched resolver can return null while current main's send-target helper and downstream direct-send contracts expect a non-null receive-id type. The original Feishu 400 was not live-replayed with a configured app.

## Solution Assessment

No. The PR targets the right Feishu plugin boundary, but it must preserve a non-null send-target output after the guard, add invalid-target regression coverage, and provide real Feishu proof before merge.
