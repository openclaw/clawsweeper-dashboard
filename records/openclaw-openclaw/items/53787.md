## Summary

Keep open. Current main still falls back to `user_id` for unknown Feishu targets, but this PR is not merge-ready because making the resolver nullable leaks `null` into downstream send contracts and the user-facing fix lacks the required changelog entry.

## What This Changes

The PR makes Feishu receive-id type resolution nullable for invalid targets and adds guards in send-target resolution plus streaming reply startup before Feishu/Lark API calls.

## Best Possible Solution

Land a Feishu-owned fix that rejects truly invalid send targets before API calls, preserves valid bare direct IDs, exposes a non-null `FeishuIdType` from the guarded send-target helper, handles nullable resolver callers explicitly, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes for the PR defect: the diff makes `resolveReceiveIdType` nullable while `resolveFeishuSendTarget` still exposes `ReturnType<typeof resolveReceiveIdType>`, and direct send paths require a non-null receive-id type. The linked runtime bug has clear reporter steps in #53376, but this read-only review did not run live Feishu API validation.

## Solution Assessment

No. The PR is aimed at the right Feishu plugin boundary, but the current patch should narrow the guarded helper result to a non-null `FeishuIdType` instead of leaking `null` into downstream send callers.
