## Summary

Keep open. The PR targets a still-open Feishu receive-id bug and current main/latest release do not include its invalid-target guard, but the branch still has a blocking type-contract issue, is conflicting against current main, and lacks real Feishu runtime proof.

## What This Changes

The PR makes Feishu receive-id type resolution nullable for invalid IDs and adds guards in send-target and streaming startup paths before Feishu/Lark API calls.

## Best Possible Solution

Update the Feishu plugin fix so invalid target/type derivation is rejected before API calls while valid bare user IDs remain supported, the send-target return contract stays non-null, regression tests cover invalid targets, and real Feishu runtime proof is posted.

## Solution Assessment
