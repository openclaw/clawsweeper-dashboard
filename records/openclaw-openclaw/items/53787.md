## Summary

Keep open: this is a useful Feishu bug-fix PR for the still-open linked regression, but it should not merge until the nullable type leak is fixed and after-fix real behavior proof is added.

## What This Changes

The PR makes Feishu receive-id type resolution nullable for invalid targets and adds guards in send-target resolution plus streaming-card startup before calling the Feishu/Lark API.

## Best Possible Solution

Land a Feishu-owned fix that rejects invalid target/type derivation before API calls, preserves valid bare direct IDs, exposes a non-null FeishuIdType from guarded send-target resolution, and adds focused regression coverage plus runtime proof.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the resolver becomes nullable while the send-target result type remains tied to ReturnType, and downstream direct send contracts require a non-null type. The linked runtime Feishu 400 has concrete reporter steps but was not live-replayed here.

## Solution Assessment

No. The PR targets the right Feishu plugin boundary, but the guarded helper should expose a non-null FeishuIdType to downstream callers and include invalid-target regression coverage before merge.
