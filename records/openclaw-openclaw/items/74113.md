## Summary

Keep open. Current main still has the stale/raw-output path this PR is trying to address, but this branch should not merge as-is because the refreshed assistant text bypasses existing silent-token normalization and is discarded when child completion findings win; open PR #74340 also needs to be reconciled as the competing canonical fix.

## What This Changes

The PR imports `readLatestAssistantReply` and adds a best-effort one-shot latest-assistant refresh in `runSubagentAnnounceFlow` before final announcement construction.

## Best Possible Solution

Land one canonical subagent completion-output fix that preserves assistant-output precedence after compaction, avoids leaking raw tool payloads, keeps `ANNOUNCE_SKIP`/`NO_REPLY` semantics normalized, covers both child-findings and no-findings paths, and adds focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main can select `latestRawText` before a later assistant reply is available, and #74073 provides the live Grok/DeepSeek plus large `web_search` scenario that triggers the race.

## Solution Assessment

No. The PR is in the right area, but assigning raw refreshed assistant history skips the established normalization helper and does not update the value that final `findings` uses when `childCompletionFindings` is set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize refreshed assistant replies before assigning:** `src/agents/subagent-announce.ts:464-471`
  - body: The refreshed `latestReply` is raw assistant history, while `reply` has already gone through `stripAndClassifyReply`. If the refreshed text contains visible content plus a `NO_REPLY` fragment, this block writes the token back into the announcement because the guards only catch pure silent tokens. Run the refreshed text through the same normalization before assigning it.
  - confidence: 0.86
- **[P2] Refresh the value that feeds final findings:** `src/agents/subagent-announce.ts:459-475`
  - body: This retry mutates `reply`, but final `findings` is still computed as `childCompletionFindings || reply || "(no output)"`. When `childCompletionFindings` is truthy, the refreshed reply is discarded, so the new unified retry does not cover that branch. Refresh the findings value too, or intentionally scope the retry to the no-findings path.
  - confidence: 0.78
