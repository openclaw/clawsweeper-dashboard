## Summary

Keep open for maintainer handling: the PR targets a real subagent announce race, but the patch is incorrect as written and now overlaps several open fix paths, including a protected maintainer PR for the raw-output leak.

## What This Changes

This PR imports `readLatestAssistantReply` and adds a one-shot post-compaction assistant-reply refresh in `runSubagentAnnounceFlow` before building the subagent announcement.

## Best Possible Solution

Land one consolidated subagent completion-output fix that prevents raw tool payload announcements, preserves post-compaction assistant precedence, keeps silent-token handling normalized, and covers capture plus announce paths with focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main can still select `latestRawText` before a later assistant reply is available, and the linked issue provides live Grok/DeepSeek plus large `web_search` and Testbox-style repro context for the race family.

## Solution Assessment

No. The retry is in the right area, but it assigns raw refreshed assistant history without the established normalization helper and does not update the value that final `findings` uses when `childCompletionFindings` is set.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize refreshed assistant replies before assigning:** `src/agents/subagent-announce.ts:464-471`
  - body: The refreshed `latestReply` is raw assistant history, while the existing path normalizes `reply` with `stripAndClassifyReply`. If the refreshed text contains visible content plus a `NO_REPLY` fragment, this block writes the token back into the announcement because the guards only catch pure silent tokens. Run the refreshed text through the same normalization before assigning it.
  - confidence: 0.86
- **[P2] Refresh the value that feeds final findings:** `src/agents/subagent-announce.ts:459-475`
  - body: This retry mutates `reply`, but final `findings` is still computed as `childCompletionFindings || reply || "(no output)"`. When `childCompletionFindings` is truthy, the refreshed reply is discarded, so the new unified retry does not cover that branch. Refresh the findings value too, or intentionally scope the retry to the no-findings path.
  - confidence: 0.78
