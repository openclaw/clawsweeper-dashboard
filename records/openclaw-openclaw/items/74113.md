## Summary

Keep open. The underlying bug is valid, but this patch should not merge as-is because the refreshed assistant reply bypasses existing NO_REPLY normalization and is ignored when child-completion findings supply the final announcement text; overlapping open fixes also need maintainer consolidation.

## What This Changes

The PR imports `readLatestAssistantReply` and adds a one-shot post-branch assistant-reply refresh in `runSubagentAnnounceFlow` before building the subagent completion announcement.

## Best Possible Solution

Land one canonical subagent completion-output fix that prevents raw tool JSON leaks across compaction and capture paths, preserves `ANNOUNCE_SKIP`/`NO_REPLY` semantics, and adds focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes, at source level. Current main can still select `latestRawText` before a later assistant reply is available, and the PR-introduced regressions follow directly from the raw refreshed-reply assignment and final `findings` precedence.

## Solution Assessment

No. The retry is in the right area, but it is not the best fix as written because it skips existing reply normalization and does not update the value that final announcement construction uses in the child-findings path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize refreshed assistant replies before assigning:** `src/agents/subagent-announce.ts:464-471`
  - body: The refreshed `latestReply` is raw assistant history, while `reply` has already gone through `stripAndClassifyReply`. If the refreshed text contains visible content plus a `NO_REPLY` fragment, this block writes the token back into the announcement because the guards only catch pure silent tokens. Run the refreshed text through the same normalization before assigning it.
  - confidence: 0.86
- **[P2] Refresh the value that feeds final findings:** `src/agents/subagent-announce.ts:459-475`
  - body: This retry mutates `reply`, but final `findings` is still computed as `childCompletionFindings || reply || "(no output)"`. When `childCompletionFindings` is truthy, the refreshed reply is discarded, so the new unified retry does not cover that branch. Refresh the findings value too, or intentionally scope the retry to the no-findings path.
  - confidence: 0.78
