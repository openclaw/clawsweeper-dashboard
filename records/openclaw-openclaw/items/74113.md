## Summary

Keep open. Current main still has the subagent announce/raw-tool-output edge this PR is trying to address, but the proposed patch is not safe to merge as-is because it can reintroduce unnormalized silent tokens and does not affect the child-completion-findings path it claims to unify.

## What This Changes

The PR imports `readLatestAssistantReply` and adds a best-effort one-shot assistant-reply refresh in `runSubagentAnnounceFlow` before announcement status construction.

## Best Possible Solution

Land one canonical subagent announce/output fix that performs a bounded, normalized post-wait refresh before constructing completion events, preserves `ANNOUNCE_SKIP` and `NO_REPLY` semantics, handles child-completion-findings and no-findings paths intentionally, and adds regression coverage for stale raw tool output followed by delayed post-compaction assistant output.

## Reproduction Assessment

Yes. The underlying issue has a concrete #74073 live scenario, and the PR defects are reproducible by source inspection: raw `latestReply` can overwrite normalized `reply`, and `childCompletionFindings || reply` discards refreshed replies when child findings exist.

## Solution Assessment

No. The PR targets the right layer, but the current diff is not the narrowest safe fix because it skips existing normalization and leaves the child-findings branch effectively unchanged; a normalized refresh or the #74340 raw-tool-output approach with tests is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize refreshed assistant replies before assigning:** `src/agents/subagent-announce.ts:464-471`
  - body: `reply` has already gone through `stripAndClassifyReply`, but `latestReply` is raw history text. If the latest assistant message contains `NO_REPLY` plus real content, this block passes the pure-token guards and writes the internal token back into the announcement. Run the refreshed text through the same normalization before assigning it.
  - confidence: 0.87
- **[P2] Refresh the value that feeds findings:** `src/agents/subagent-announce.ts:459-475`
  - body: The retry also runs when `childCompletionFindings` is set, but `findings` is later computed as `childCompletionFindings || reply || "(no output)"`. Any refreshed `reply` is discarded in that path, so the new block does not actually unify both branches. Either refresh the findings value too or limit the retry to the no-findings path.
  - confidence: 0.8
