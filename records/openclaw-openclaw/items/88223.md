## Summary

Keep open. Current main still routes WhatsApp composing through the reply lifecycle and still sends composing immediately before outbound sends; the merged admission foundation did not implement early typing, and the open related PRs cover adjacent group-typing tradeoffs rather than this direct responsiveness request.

## What This Changes

The issue requests faster WhatsApp feedback by starting composing at accepted inbound time, avoiding redundant send-time composing, and possibly adding WhatsApp-specific timing configuration.

## Best Possible Solution

Have maintainers choose the WhatsApp typing timing boundary, then implement a focused plugin or shared lifecycle change that starts the first composing signal at the earliest safe accepted-inbound point without duplicating the existing typing configuration model.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection supports the reported shape: composing is still lifecycle-driven, outbound sends still emit composing at send time, and current config does not expose the requested WhatsApp-specific immediate option.

## Solution Assessment

Unclear. Earlier accepted-inbound composing is plausible, but the best implementation needs a maintainer decision against the existing `typingMode` and `typingIntervalSeconds` surfaces and live WhatsApp proof.
