## Summary

Keep open. Current main still starts WhatsApp composing through the normal reply lifecycle and still sends a composing update immediately before outbound sends, while the requested immediate-on-receipt behavior and WhatsApp-specific timing/config choice remain unresolved.

## What This Changes

The issue requests faster WhatsApp typing feedback and delivery perception by starting composing earlier, avoiding redundant send-time composing, and adjusting or configuring the typing cadence.

## Best Possible Solution

Have maintainers choose the WhatsApp typing timing boundary, then implement a focused plugin/core change that starts the first composing signal at the earliest safe accepted-inbound point and avoids duplicating the existing typing configuration model.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does support the reported shape: composing is wired through the reply lifecycle, outbound sends still emit composing immediately before send, and current docs/config expose only global or session typing timing.

## Solution Assessment

Unclear. Earlier composing is plausible, but lowering defaults or adding a WhatsApp-specific config option should be a maintainer product/API decision against the existing typingMode and typingIntervalSeconds surfaces.
