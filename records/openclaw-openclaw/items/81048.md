## Summary

Keep open: this targets a real Claude CLI resume/context-loss bug, but the current PR still duplicates the Claude project-directory encoding instead of sharing current main's canonical resolver, and the latest v5 implementation does not yet have inspectable after-fix live proof.

## What This Changes

The PR changes the Claude CLI resume transcript probe to derive a deterministic workspace/session JSONL path with a 250 ms grace rescan, threads `workspaceDir` into both resume checks, and makes `parseCliJsonl` capture the last Claude `session_id`.

## Best Possible Solution

Extract a shared Claude project-directory resolver used by both doctor and transcript probing, keep the last-wins session-id fix if it remains covered, and add fresh redacted live Claude CLI proof for the latest branch.

## Reproduction Assessment

Yes, at source level: current main returns false for a missing or header-only Claude project transcript and both resume paths map that false result to `transcript-missing`. The PR discussion also supplies live logs for the context-loss symptom, though I did not run a live Claude CLI repro.

## Solution Assessment

No for the current patch: the deterministic-path direction and last-wins parser fix are plausible, but the path helper should share current main's canonical resolver before replacing the broad project scan.
