## Summary

Keep open: the reported Claude CLI resume bug is valid and the PR is in the right area, but the latest v4 patch duplicates an incomplete Claude project-dir resolver and still lacks after-fix live proof for the current implementation.

## What This Changes

The PR replaces the Claude CLI transcript probe with a deterministic workspace/session JSONL path, passes `workspaceDir` into the probe, adds a 250 ms grace rescan plus warn logging, and updates focused tests.

## Best Possible Solution

Extract or reuse one shared Claude project-dir resolver for doctor and transcript probing, cover canonical/symlink/long-path cases, then add fresh redacted live Claude CLI proof for v4 before landing.

## Reproduction Assessment

Yes, at source level: current main returns false when the matching Claude transcript is absent or header-only, and both resume paths map that false result to `transcript-missing`. The PR discussion also supplies live logs showing that path causing context loss.

## Solution Assessment

No for the current patch: moving away from open-loop backoff toward a deterministic path is the right direction, but the implementation should share the existing canonical Claude project-dir resolver before it can be considered the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use the existing Claude project-dir resolver:** `src/agents/command/attempt-execution.helpers.ts:118-119`
  - body: This computes the Claude project directory from the raw `workspaceDir` with only character replacement, but current main's doctor path resolver first resolves/realpaths/NFC-normalizes the workspace and applies Claude's 200-character hash cap. For symlinked or very long workspaces this will probe a path Claude does not write, return false, and force another `transcript-missing` reset. Please extract or reuse the existing resolver so both code paths agree.
