## Summary

Keep open: the PR targets a real Claude CLI resume bug that current main still has, but the latest v4 branch still computes the deterministic project path differently from the existing Claude project-dir resolver and lacks fresh after-fix real behavior proof for that latest implementation. The newer overlapping draft at https://github.com/openclaw/openclaw/pull/81821 is useful context, but it is not merged or terminal enough to close this PR as superseded.

## What This Changes

The PR changes Claude CLI transcript probing from a broad project scan to a deterministic workspace/session JSONL path with a 250 ms grace rescan, threads `workspaceDir` through resume callers, and adds warning diagnostics plus focused tests.

## Best Possible Solution

Extract or move one shared Claude project-dir resolver for doctor and runtime transcript probing, update this PR or the coordinated replacement to use it, then require fresh redacted live Claude CLI proof for the final branch head before merge.

## Reproduction Assessment

Yes, at source level: current main returns false for a matching Claude transcript that is absent or header-only, and both resume paths turn that false result into `transcript-missing` invalidation. I did not run a live Claude CLI timing reproduction in this read-only review.

## Solution Assessment

No for the current PR head: the deterministic path direction is plausible, but it should share the existing canonical Claude project-dir resolver before it is the narrow maintainable fix. The final branch also needs fresh after-fix proof for v4 rather than proof from earlier strategies.
