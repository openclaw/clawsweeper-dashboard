- `impact:security`: The requested direct fallback would change the documented proxy egress boundary from fail-closed to fail-open for some traffic.

## Summary

Keep open. Current main and the latest release still route configured proxy traffic through proxy dispatchers without request-time reachability fallback, and changing that to silent direct egress needs maintainer networking/security policy rather than an automatic cleanup close.

## What This Changes

Request smart proxy detection that falls back to direct network access when a configured proxy port is unavailable.

## Best Possible Solution

Keep configured proxy routing fail-closed by default; if fallback is approved, add a documented opt-in circuit breaker in the shared proxy/fetch layer with tests and redacted runtime proof.

## Reproduction Assessment

Yes, from source inspection. Set HTTP_PROXY or HTTPS_PROXY to a closed loopback port and the current proxy-fetch or guarded-fetch paths select proxy dispatchers and rethrow request-time connection failure instead of retrying direct.

## Solution Assessment

No. A gaxios-only or silent direct fallback is too narrow and would bypass the documented proxy boundary; any fallback should be shared, explicit, documented, and policy-gated.
