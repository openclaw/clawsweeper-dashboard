## Summary

Keep open. Current main and v2026.5.18 still route configured proxy traffic through proxy dispatchers without request-time reachability fallback, and changing that to direct egress needs an explicit networking/security product decision.

## What This Changes

The issue requests automatic proxy reachability detection so OpenClaw falls back to direct network access when a configured local proxy port is closed.

## Best Possible Solution

Keep configured and managed proxies fail-closed by default; if fallback is approved, add a documented opt-in circuit breaker in the shared proxy/fetch layer with tests, docs, and redacted runtime proof.

## Reproduction Assessment

Yes, from source inspection. Set HTTP_PROXY or HTTPS_PROXY to a closed loopback port and current proxy-fetch, guarded-fetch, or global undici paths will select proxy dispatchers and propagate request-time connection failure instead of retrying direct.

## Solution Assessment

No, not as proposed. A gaxios-only reachability probe is obsolete and too narrow; any approved fallback belongs in the shared proxy/fetch layer and should be explicit opt-in policy because direct egress changes the security boundary.
