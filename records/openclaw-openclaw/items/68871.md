## Summary

Keep open. Current main and v2026.6.10 still route configured proxy traffic through proxy dispatchers without request-time reachability probing or direct retry, and changing that to fail open is a security/product decision because the documented managed-proxy contract is fail-closed egress control.

## What This Changes

Request automatic proxy reachability detection and direct-connection fallback when a configured proxy port is unavailable.

## Best Possible Solution

Keep fail-closed proxy routing as the default; if maintainers approve convenience fallback, add an explicit opt-in fallback or circuit-breaker policy in the shared proxy/fetch and managed-proxy layers with docs, tests, and security review.

## Reproduction Assessment

Yes, at source level: configure HTTP_PROXY/HTTPS_PROXY or proxy.enabled to a closed proxy and current source still selects proxy dispatchers instead of probing or retrying direct. I did not run a live closed-port repro in this read-only review.

## Solution Assessment

No, not as proposed. A gaxios-only reachability probe is too narrow and stale against current shared proxy/fetch and managed-proxy routing; any approved fallback should be an explicit policy in the shared networking layer.
