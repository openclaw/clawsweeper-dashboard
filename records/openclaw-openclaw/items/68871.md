## Summary

Keep open. Current main still selects proxy routing from environment/configuration and NO_PROXY matching, with no reachability probe or request-time fallback when the proxy endpoint refuses connections; because managed proxy routing is documented as fail-closed, any direct fallback needs explicit security/product semantics rather than cleanup closure.

## What This Changes

The issue requests smart proxy reachability detection so OpenClaw can fall back to direct networking when a configured proxy endpoint is unavailable.

## Best Possible Solution

Keep the issue open and define a narrow proxy-unavailable policy in the shared proxy layer: preserve fail-closed behavior for `proxy.enabled`, explicit proxies, and mandatory egress-control setups, while considering an opt-in or clearly bounded convenience path for ambient local proxy-manager ports with cached, timeout-bounded reachability checks or direct retry. Update the shared proxy helpers, affected plugin transports, docs, and tests together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Set `HTTPS_PROXY` or `HTTP_PROXY` to a closed loopback port and exercise `resolveProxyFetchFromEnv` or a guarded fetch path using trusted env proxy mode; current main will create an env proxy dispatcher and rethrow the request failure rather than falling back direct.

## Solution Assessment

Is this the best way to solve the issue? No as proposed. A gaxios-only `resolveProxyUri()` probe would miss shared fetch, provider, channel, and managed-proxy paths; the maintainable fix belongs in shared proxy policy with owner-specific wiring only where a transport bypasses that shared layer.
