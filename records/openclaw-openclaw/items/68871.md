## Summary

Keep open. Current main and v2026.6.8 still route configured proxy traffic through EnvHttpProxyAgent/proxy dispatchers without request-time direct fallback, and automatic direct fallback remains a security/product decision because it can bypass operator egress policy.

## What This Changes

Request automatic proxy reachability detection and direct-connection fallback when a configured proxy port is unavailable.

## Best Possible Solution

Keep proxy routing fail-closed by default; if maintainers want convenience, add an explicit opt-in circuit breaker or fallback policy in the shared proxy/guarded-fetch layer with docs, tests, and security review.

## Reproduction Assessment

Yes, at source level: configure HTTP_PROXY or HTTPS_PROXY to a closed local port and current proxy-fetch/guarded-fetch paths still select proxy dispatchers rather than a direct retry path. I did not run a live closed-port repro in this read-only review.

## Solution Assessment

No, not as proposed. A gaxios-only reachability probe is obsolete and too narrow; any approved fallback belongs in the shared proxy/fetch layer and should be an explicit opt-in policy because direct egress changes the security boundary.
