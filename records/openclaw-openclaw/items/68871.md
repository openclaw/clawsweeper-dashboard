## Summary

Keep open. Current main and the latest release still route configured proxy traffic through proxy dispatchers without a request-time reachability fallback, while the requested silent direct fallback is a security/product decision because managed proxy routing is documented as fail-closed.

## What This Changes

The issue requests automatic detection of an unavailable configured proxy and fallback to a direct connection instead of surfacing proxy connection failures.

## Best Possible Solution

Define proxy-unavailable behavior in the shared proxy layer: keep managed and explicit proxy modes fail-closed, and consider only a documented opt-in fallback or circuit breaker for ambient local proxy-manager env vars with focused tests and docs.

## Reproduction Assessment

Yes, source-reproducible. Set HTTPS_PROXY or HTTP_PROXY to a closed loopback port and exercise resolveProxyFetchFromEnv or fetchWithSsrFGuard; current main selects EnvHttpProxyAgent and rethrows request failures instead of falling back direct.

## Solution Assessment

No. The proposed gaxios-fetch-compat.ts/resolveProxyUri change does not match the current central proxy paths, and silent direct retry conflicts with the documented fail-closed proxy guardrail; policy-gated shared handling is safer.
