## Summary

Keep open. Current main and the latest release still route configured proxy traffic through proxy dispatchers without request-time reachability fallback, while the requested direct fallback would change the documented fail-closed proxy boundary and needs maintainer networking/security policy first.

## What This Changes

The issue requests automatic proxy reachability detection and direct-connection fallback when a configured proxy port is closed.

## Best Possible Solution

Keep configured and managed proxies fail-closed by default; if fallback is approved, add a documented opt-in circuit breaker in the shared proxy/fetch layer with tests, docs, and redacted runtime proof.

## Reproduction Assessment

Yes, from source inspection. Set HTTP_PROXY or HTTPS_PROXY to a closed loopback port and current proxy-fetch or guarded-fetch paths will select proxy dispatchers and propagate the request-time connection failure instead of retrying direct.

## Solution Assessment

No, not as proposed. A gaxios-only reachability probe is obsolete and too narrow; any approved fallback belongs in the shared proxy/fetch layer with explicit policy gating.
