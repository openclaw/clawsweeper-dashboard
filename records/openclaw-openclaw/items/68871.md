## Summary

Keep open. Current main still routes configured proxy traffic through proxy dispatchers without request-time reachability probing or direct retry, and automatic direct fallback would conflict with the documented fail-closed proxy egress boundary unless maintainers approve an explicit policy.

## What This Changes

Request to auto-detect configured proxy reachability and fall back to direct connection when the proxy port is closed.

## Best Possible Solution

Keep fail-closed proxy routing as the default; if maintainers approve convenience fallback, add an explicit opt-in fallback or circuit-breaker policy in the shared networking layer with security docs and focused tests.

## Reproduction Assessment

Yes, at source level: configure HTTP_PROXY/HTTPS_PROXY or proxy.enabled to a closed proxy and current source still selects proxy dispatchers instead of probing or retrying direct. I did not run a live closed-port repro in this read-only review.

## Solution Assessment

No, not as proposed. A gaxios-specific reachability probe is stale and too narrow for current main; any approved fallback should be an explicit shared networking policy that preserves the fail-closed default.
