## Summary

Keep open. Current main still lacks request-time proxy reachability fallback, but direct fallback would change the documented fail-closed proxy boundary and needs maintainer networking/security policy first.

## What This Changes

Request to detect unavailable configured proxy ports and fall back to direct network connections instead of surfacing ECONNREFUSED.

## Best Possible Solution

Keep configured and managed proxies fail-closed by default; if maintainers approve fallback, add a documented opt-in circuit breaker in the shared proxy/fetch layer with tests, docs, and redacted runtime proof.

## Reproduction Assessment

Yes, from source inspection. Set HTTP_PROXY or HTTPS_PROXY to a closed loopback port and current proxy-fetch or guarded-fetch paths will select proxy dispatchers and propagate the request-time connection failure instead of retrying direct.

## Solution Assessment

No, not as proposed. A gaxios-only reachability probe is obsolete and too narrow; any approved fallback belongs in the shared proxy/fetch layer with explicit policy gating.
