## Summary

Keep open. Current main still selects configured/env proxy routing without a reachability probe and rethrows proxy connection failures; automatic direct fallback also conflicts with the documented fail-closed managed-proxy contract, so this needs maintainer policy direction rather than cleanup closure.

## What This Changes

Request automatic detection of unreachable configured proxies and fallback to direct network connections instead of surfacing proxy ECONNREFUSED failures.

## Best Possible Solution

Define a narrow proxy-unavailable policy in the shared proxy layer: preserve fail-closed behavior for managed, explicit, and mandatory proxy configurations, and consider only an opt-in or tightly bounded convenience path for ambient local proxy-manager env vars. Update shared helpers, owner transports that bypass the shared path, docs, and tests together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Set HTTPS_PROXY or HTTP_PROXY to a closed loopback port and exercise resolveProxyFetchFromEnv or fetchWithSsrFGuard in trusted env proxy mode; current code selects EnvHttpProxyAgent and rethrows the request failure instead of falling back direct.

## Solution Assessment

Is this the best way to solve the issue? No as proposed. A gaxios-only resolveProxyUri probe would miss shared guarded fetch, managed proxy, and other owner transport paths, while a broad silent fallback would weaken the documented fail-closed proxy contract.
