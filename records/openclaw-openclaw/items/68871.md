## Summary

Keep open. Current main and v2026.5.7 still route configured proxy traffic through proxy dispatchers without a request-time reachability fallback, and the requested silent direct fallback changes the documented fail-closed proxy policy rather than fixing a settled bug.

## What This Changes

Request automatic proxy reachability detection and direct-connection fallback when the configured proxy endpoint is unavailable.

## Best Possible Solution

Define proxy-unavailable behavior in the shared proxy layer: keep managed and explicit proxy modes fail-closed by default, and consider only a documented opt-in fallback or circuit breaker for ambient local proxy-manager env vars.

## Reproduction Assessment

Yes. Set HTTPS_PROXY or HTTP_PROXY to a closed loopback port and exercise resolveProxyFetchFromEnv or fetchWithSsrFGuard; current source selects proxy dispatchers and rethrows request-time failures instead of falling back direct.

## Solution Assessment

No. A silent gaxios-only reachability check is not the best fix because the active paths are shared proxy-fetch and guarded-fetch dispatchers, and direct retry conflicts with the documented fail-closed proxy guardrail unless explicitly policy-gated.
