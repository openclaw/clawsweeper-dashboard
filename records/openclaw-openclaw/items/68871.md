## Summary

Keep open. Current main and v2026.6.8 still route configured proxy traffic through EnvHttpProxyAgent/proxy dispatchers without request-time direct fallback, and the remaining question is a security/product decision because fail-open direct egress can bypass configured proxy policy.

## What This Changes

Request automatic proxy reachability detection so OpenClaw falls back to direct network access when a configured local proxy is unavailable.

## Best Possible Solution

Keep proxy routing fail-closed by default; if maintainers want convenience, add an explicit opt-in circuit breaker or fallback policy in the shared proxy/fetch layer with docs, tests, and security review.

## Reproduction Assessment

Yes, at source level: set HTTP_PROXY or HTTPS_PROXY to a closed local port and current proxy-fetch/guarded-fetch paths still select proxy dispatchers rather than a direct retry path. I did not run a live closed-port repro in this read-only review.

## Solution Assessment

No, not as proposed. A gaxios-only reachability probe is obsolete and too narrow; any approved fallback belongs in the shared proxy/fetch layer and should be explicit opt-in policy because direct egress changes the security boundary.
