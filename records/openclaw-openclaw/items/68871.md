## Summary

Keep open. Current main and v2026.6.8 still route configured proxy traffic through proxy dispatchers without proxy reachability probing or request-time direct fallback, and fail-open direct egress remains a security/product decision because it can bypass operator egress policy.

## What This Changes

Request automatic proxy reachability detection and direct-network fallback when a configured local proxy is unavailable.

## Best Possible Solution

Keep fail-closed proxy routing as the default; if maintainers approve convenience fallback, add an explicit opt-in circuit breaker or fallback policy in the shared proxy/fetch layer with docs, tests, and security review.

## Reproduction Assessment

Yes at source level: configure HTTP_PROXY/HTTPS_PROXY or proxy.enabled to a closed proxy and current source still selects proxy dispatchers instead of probing or retrying direct. I did not run a live closed-port repro in this read-only review.

## Solution Assessment

No, not as proposed. A gaxios-only reachability probe is too narrow and obsolete; any approved fallback belongs in the shared proxy/fetch and managed proxy surfaces as an explicit policy because it changes the security boundary.
