## Summary

Keep open. Current main and v2026.5.18 still route configured proxy traffic through env/managed proxy dispatchers and rethrow transport errors; direct fallback is a new fail-open networking/security behavior that needs maintainer product direction.

## What This Changes

Request auto-detecting unreachable configured proxy ports and falling back to direct network connections instead of surfacing ECONNREFUSED.

## Best Possible Solution

Keep configured and managed proxies fail-closed by default; if maintainers approve unavailable-proxy recovery, add a documented opt-in circuit breaker in the shared proxy/fetch layer with tests, docs, and runtime proof.

## Reproduction Assessment

Yes, from source inspection rather than a live run. Set HTTP_PROXY or HTTPS_PROXY to a closed loopback port and the current proxy-fetch or guarded-fetch paths will select proxy dispatchers and propagate the request-time connection failure instead of retrying direct.

## Solution Assessment

No, not as proposed. A gaxios-only reachability probe is obsolete and too narrow; any approved fallback belongs in the shared proxy/fetch layer with explicit policy gating.
