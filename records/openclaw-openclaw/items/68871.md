## Summary

Keep open. Current main still routes configured proxy traffic through proxy dispatchers without request-time reachability fallback, but silent direct fallback would weaken the documented fail-closed proxy boundary and needs maintainer networking/security policy first.

## What This Changes

Requests auto-detecting configured proxy reachability and falling back to direct connection when the proxy is unavailable.

## Best Possible Solution

Define shared proxy-unavailable behavior: keep managed and explicit proxies fail-closed by default, and add only a documented opt-in fallback or circuit breaker for approved ambient proxy-manager cases.

## Reproduction Assessment

Yes, from source inspection. Set HTTP_PROXY or HTTPS_PROXY to a closed loopback port and current proxy-fetch or guarded-fetch paths will select proxy dispatchers and propagate request-time connection failure instead of retrying direct.

## Solution Assessment

No. A gaxios-only reachability probe is too narrow; any fallback belongs in the shared proxy/fetch layer with explicit policy gating if maintainers approve fail-open behavior.
