## Summary

Keep open. Current main still selects configured/env proxy routing without a reachability probe and rethrows runtime proxy failures, while silent direct fallback conflicts with the documented fail-closed managed-proxy contract.

## What This Changes

Request automatic proxy reachability detection before using a configured or environment proxy, with direct connection fallback when the proxy listener is closed.

## Best Possible Solution

Define proxy-unavailable behavior in the shared proxy layer: keep managed, explicit, and mandatory proxy configurations fail-closed, and consider only an opt-in bounded convenience fallback for ambient local proxy-manager env vars with tests and docs.

## Reproduction Assessment

Yes. Set HTTPS_PROXY or HTTP_PROXY to a closed loopback port and exercise resolveProxyFetchFromEnv or fetchWithSsrFGuard in trusted env proxy mode; current code selects EnvHttpProxyAgent and rethrows the transport failure instead of falling back direct.

## Solution Assessment

No as proposed. A gaxios-only resolveProxyUri reachability probe is too narrow, and a broad silent direct fallback is not the safest maintainable default for a documented proxy guardrail.
