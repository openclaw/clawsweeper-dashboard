## Summary

Keep open. Current main still routes configured proxy traffic through proxy dispatchers without a request-time reachability fallback, but the requested direct fallback changes documented fail-closed proxy behavior and needs maintainer product/security direction rather than conservative cleanup.

## What This Changes

Request auto-detecting an unavailable configured proxy and falling back to direct network access instead of surfacing proxy connection errors.

## Best Possible Solution

Define proxy-unavailable behavior in the shared proxy layer: keep managed and explicit proxy modes fail-closed by default, and consider only a documented opt-in fallback or circuit breaker for ambient local proxy-manager env vars.

## Reproduction Assessment

Yes, from source inspection. Set HTTP_PROXY or HTTPS_PROXY to a closed loopback port and the current proxy-fetch or guarded-fetch paths will select proxy dispatchers and rethrow request-time transport failures instead of retrying direct.

## Solution Assessment

No. A silent gaxios-only reachability check is not the best fix because the active paths are shared proxy-fetch and guarded-fetch dispatchers, and direct retry conflicts with the documented fail-closed proxy guardrail unless explicitly policy-gated.
