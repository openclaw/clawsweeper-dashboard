## Summary

Keep open. Current main and v2026.6.6 still route configured proxy traffic through proxy dispatchers without request-time direct fallback, and this issue has security/product-decision labels because fail-open direct egress can bypass configured proxy policy.

## What This Changes

The issue requests proxy reachability detection so OpenClaw can fall back to direct networking when a configured local proxy is unavailable.

## Best Possible Solution

Keep configured and managed proxies fail-closed by default; if fallback is approved, add a documented opt-in circuit breaker in the shared proxy/fetch layer with tests, docs, and redacted runtime proof.

## Reproduction Assessment

Yes, at source level: set HTTP_PROXY or HTTPS_PROXY to a closed local port and current proxy-fetch/guarded-fetch paths still select proxy dispatchers and propagate request-time connection failure instead of retrying direct. I did not run a live closed-port repro in this read-only review.

## Solution Assessment

No, not as proposed. A gaxios-only reachability probe is obsolete and too narrow; any approved fallback belongs in the shared proxy/fetch layer and should be explicit opt-in policy because direct egress changes the security boundary.
