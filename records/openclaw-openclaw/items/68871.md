## Summary

Keep open. Current main still routes through configured proxy dispatchers without request-time reachability fallback, and the requested silent direct fallback needs maintainer networking/security policy because it can turn configured proxy failure into direct egress.

## What This Changes

The issue requests automatic proxy reachability checks with direct network fallback when a configured local proxy is unavailable.

## Best Possible Solution

Define proxy-unavailable behavior in the shared proxy layer: keep managed and explicit proxy modes fail-closed by default, and add only a documented opt-in fallback or circuit breaker for approved ambient proxy-manager cases.

## Reproduction Assessment

Yes, from source inspection. Set HTTP_PROXY or HTTPS_PROXY to a closed loopback port and current proxy-fetch or guarded-fetch paths will select proxy dispatchers and propagate request-time ECONNREFUSED instead of retrying direct.

## Solution Assessment

No. A silent gaxios-only reachability check is not the best solution because the current central paths are shared proxy-fetch and guarded-fetch, and any direct fallback needs shared policy gating plus security documentation.
