## Summary

Keep open. Current main still lacks a runtime proxy reachability probe or request-time direct fallback, and the requested silent fallback conflicts with the documented fail-closed managed proxy contract, so this needs maintainer product/security direction before implementation.

## What This Changes

Request automatic detection of an unreachable configured proxy and direct-connection fallback instead of surfacing ECONNREFUSED.

## Best Possible Solution

Define proxy-unavailable behavior in the shared proxy layer: keep managed and explicit proxy modes fail-closed, and consider only a documented opt-in fallback for ambient local proxy-manager env vars.

## Reproduction Assessment

Yes from source inspection. Set HTTPS_PROXY or HTTP_PROXY to a closed loopback port and exercise resolveProxyFetchFromEnv or fetchWithSsrFGuard in trusted-env proxy mode; current main attaches EnvHttpProxyAgent and rethrows the transport failure instead of retrying direct.

## Solution Assessment

No as proposed. A gaxios-only reachability probe is too narrow for the current shared proxy paths, and silent direct fallback is not the safest default for a documented fail-closed proxy guardrail.
