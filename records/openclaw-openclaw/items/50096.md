## Summary

Keep open. Current main has much stronger memory infrastructure and the discussion usefully points to external UAML and Engram-style implementations, but this issue still contains unresolved core/product questions around shutdown flushing, memory-specific injection policy, and recommended backend direction.

## What This Changes

Requests a long-term memory roadmap covering persistence across restarts, memoryFlush on shutdown, automated consolidation, configurable MEMORY.md injection, session search, and backend direction.

## Best Possible Solution

Split this umbrella into canonical follow-ups for graceful shutdown memory flush and MEMORY.md recall-only/injection policy, then document the shipped memory backends, Dreaming, memory-wiki, and plugin paths as the current architecture.

## Reproduction Assessment

Yes for the concrete remaining subclaims by source inspection: current main invokes memoryFlush from the reply/compaction path, while gateway close only runs gateway_stop/prelude/close; MEMORY.md also remains part of default bootstrap injection with only broad contextInjection controls. I did not run a live gateway restart because this was a read-only review.

## Solution Assessment

No as a single umbrella. The maintainable path is to split the core reliability/config work from ecosystem backend recommendations and ClawHub/plugin direction.
