## Summary

Keep open: current main still has a source-backed stdio safety gap for `openclaw mcp serve`, and the linked closing PR is still open, conflicting, and not merged. The exact notification diagnosis is not proven by current source, but the strict-client tool discovery failure remains actionable through stdout pollution before JSON-RPC frames.

## What This Changes

Fix `openclaw mcp serve` so strict stdio MCP clients receive a normal `tools/list` request/response instead of failing during tool discovery.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/68800 with a startup-level stdout reservation for `openclaw mcp serve`, keep the SDK-backed `tools/list` request handler path, and include raw stdio proof.

## Reproduction Assessment

Yes, for the strict-stdio failure at source level: `openclaw mcp serve` is documented as stdio, the SDK writes protocol frames to stdout, and current CLI/config startup can emit diagnostics before the MCP stream is protected. The exact `notifications/tools/list` diagnosis still needs live/raw stdio confirmation.

## Solution Assessment

Yes in direction, but not finished while the linked PR remains open. The narrow maintainable solution is to suppress non-protocol stdout before CLI/config preflight and prove `tools/list` over raw stdio.
