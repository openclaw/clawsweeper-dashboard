## Summary

Keep open: current main still lacks stdout isolation in the channel MCP stdio server, and PR #68800 is still open with explicit closing syntax for this issue. The strongest current evidence points to strict stdio stream pollution rather than a missing `tools/list` SDK request handler.

## What This Changes

Fix `openclaw mcp serve` so strict stdio MCP clients receive protocol-only JSON-RPC and a correlated `tools/list` response.

## Best Possible Solution

Keep this issue open until PR #68800 or an equivalent focused change lands: route channel MCP diagnostics to stderr before config loading and add a strict stdio `tools/list` regression check.

## Reproduction Assessment

Yes for the user-visible strict stdio failure path: the issue gives concrete steps, current main still lacks stderr routing in `serveOpenClawChannelMcp`, and the paired PR includes raw JSON-RPC before/after evidence. The exact notification-only root cause remains less certain because the SDK-backed `server.tool(...)` path should answer `tools/list`.

## Solution Assessment

Yes: routing logs to stderr at the start of `serveOpenClawChannelMcp` is the narrowest maintainable fix because it matches the existing plugin tools stdio pattern and preserves the SDK's normal `tools/list` request/response handling.
