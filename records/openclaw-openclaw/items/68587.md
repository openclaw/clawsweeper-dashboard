## Summary

Keep this open: current main still lacks stdout isolation in the channel MCP stdio path, and PR #68800 is an open, unmerged implementation candidate that explicitly closes this issue.

## What This Changes

The issue asks for `openclaw mcp serve` to preserve protocol-valid MCP `tools/list` request/response behavior for strict stdio clients such as Hermes Agent.

## Best Possible Solution

Land or replace PR #68800 with a narrow stderr-routing fix plus a strict stdio regression proving `tools/list` returns a matching-id response and stdout remains JSON-RPC only.

## Reproduction Assessment

Yes for the user-visible strict stdio failure path: the report gives concrete steps, current main lacks stderr routing in `serveOpenClawChannelMcp`, and the paired PR includes raw JSON-RPC before/after evidence. The literal notification-only root cause is less certain because the SDK tool registration path installs a `tools/list` request handler.

## Solution Assessment

Yes: routing logs to stderr before channel MCP config loading matches the existing plugin-tools stdio pattern and is the narrowest maintainable fix. The regression test should exercise strict stdio or be explicit about any in-memory-only coverage.
