## Summary

Keep open: current main still lacks stdout isolation in the channel MCP stdio server, and https://github.com/openclaw/openclaw/pull/68800 remains open with closing syntax for this issue. The strongest evidence points to strict stdio stream pollution during startup rather than a missing SDK `tools/list` handler.

## What This Changes

The issue reports that `openclaw mcp serve` breaks strict MCP clients because `tools/list` is not returned as a correlated JSON-RPC request/response.

## Best Possible Solution

Land or replace the linked focused fix: route channel MCP logs to stderr before config resolution and add a strict stdio `tools/list` regression check.

## Reproduction Assessment

Yes, at source level: `openclaw mcp serve` resolves config before any stderr routing while the documented surface is stdio MCP, so startup warnings can pollute stdout before `tools/list`. I did not run a live Hermes reproduction, and the notification-only sub-diagnosis remains less certain.

## Solution Assessment

Yes: calling `routeLogsToStderr()` at the start of `serveOpenClawChannelMcp` is the narrowest maintainable fix because it matches the existing plugin-tools stdio pattern and preserves the SDK's built-in `tools/list` request handler.
