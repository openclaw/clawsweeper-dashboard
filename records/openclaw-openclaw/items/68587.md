## Summary

Keep this issue open: current main still lacks stdout isolation in the `openclaw mcp serve` channel path, and open PR #68800 explicitly closes this issue but is not merged. The literal notification-only diagnosis looks incomplete because the pinned MCP SDK installs a `tools/list` request handler for `server.tool(...)`; the stronger remaining bug is strict stdio stream pollution before JSON-RPC responses.

## What This Changes

The issue asks for `openclaw mcp serve` to answer `tools/list` as a normal JSON-RPC request/response for strict stdio MCP clients such as Hermes Agent.

## Best Possible Solution

Keep this issue open until #68800 is reviewed and either landed or replaced by an equivalent narrow fix that routes logs to stderr at the start of `serveOpenClawChannelMcp` and adds a regression covering strict stdio protocol isolation.

## Reproduction Assessment

Yes. The issue gives concrete Hermes/stdio MCP steps, and current main still has the channel `serve` path without early stderr routing while the plugin tools stdio path already has that protection.

## Solution Assessment

Yes, the narrow direction in #68800 is the maintainable fix boundary: isolate stdout before config/log output can occur. The safer validation is a real stdio or stdout-capture regression, not only an in-memory `client.listTools()` assertion.
