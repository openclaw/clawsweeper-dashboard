## Summary

Keep this issue open. Current main still lacks the channel MCP stdout-isolation change, and PR #68800 remains open with explicit closing syntax for this issue; the strongest evidence points to strict stdio stream pollution rather than a missing `tools/list` SDK handler.

## What This Changes

The issue reports that `openclaw mcp serve` fails strict stdio MCP clients during `tools/list` because the server does not return a clean request/response JSON-RPC stream.

## Best Possible Solution

Keep this issue open until PR #68800 is reviewed and either landed or replaced by an equivalent narrow fix. The desired end state is for `openclaw mcp serve` to keep stdout protocol-only before config/log output can occur and to have a regression check that a strict MCP client receives a valid `tools/list` response with the matching id.

## Reproduction Assessment

Yes for the user-visible strict stdio failure path: the issue gives concrete stdio MCP steps, current main still lacks stderr routing in `serveOpenClawChannelMcp`, and the linked PR includes raw JSON-RPC before/after evidence. The literal notification-only root cause remains unclear because the SDK registration path should install a `tools/list` request handler.

## Solution Assessment

Yes, the narrowest maintainable fix appears to be routing logs to stderr at the top of `serveOpenClawChannelMcp`, before config loading or other startup output can pollute stdout, matching the existing plugin tools stdio pattern. The test should exercise real stdout isolation or be named to match what it actually verifies.
