## Summary

Keep open. Current main still matches the reported failure mode: final-reply local media is normalized before Telegram delivery, and when a sandbox root exists an absolute host-local path is forced through sandbox mapping first; if it does not map into the sandbox, it is dropped and can become the `Media failed` fallback. The cron-owned delivery comment is useful related evidence for the same sandbox-to-host media gap on a different delivery surface, but that scope should be decided deliberately.

## What This Changes

Allow Telegram final replies using `<final>MEDIA:<local path>` to send host-local files generated or readable by the agent instead of falling back to `Media failed`.

## Best Possible Solution

Add a narrow, policy-checked fallback for sandboxed final-reply media: after sandbox mapping fails, permit only managed media or host-local files allowed by the active agent/session outbound media policy, then stage them through the existing outbound attachment path. Decide explicitly whether the cron-owned announce surface should share that bridge or get a separate targeted fix.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with a sandbox root present, an absolute host-local `MEDIA:` path is rejected by sandbox mapping, and the checked-in test expects it to be dropped into the `Media failed` fallback. Tests were not run because this review was required to stay read-only.

## Solution Assessment

No. Current main is not the best solution because it rejects host-local sandboxed reply media before the existing outbound media read policy can authorize it. The safer alternative is a policy-checked fallback, not unrestricted host path acceptance.
