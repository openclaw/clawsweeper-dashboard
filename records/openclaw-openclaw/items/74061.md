## Summary

Keep open. Current main still drops sandboxed final-reply host-local media before Telegram delivery, while the docs say host-local outbound media should work when the agent file-read policy permits it; the cron comment is useful corroboration of the same sandbox-to-host media translation gap on another delivery path.

## What This Changes

The issue asks for Telegram final-reply `MEDIA:` delivery to send valid local files again instead of falling back to `⚠️ Media failed` after v2026.4.26.

## Best Possible Solution

Add a policy-checked sandbox-to-host media bridge for final-reply media, and decide whether cron direct delivery should share that bridge or be tracked separately.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with a sandbox root present, a final reply containing an absolute host-local `MEDIA:` path outside the sandbox is rejected by sandbox mapping, dropped, and converted into `⚠️ Media failed`.

## Solution Assessment

Not applicable as a patch review. The safest implementation is a narrow fallback through existing agent-scoped outbound media access after sandbox mapping fails, not unconditional host-path acceptance.
