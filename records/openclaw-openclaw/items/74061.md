## Summary

Keep open. Current main still matches the reported Telegram final-reply failure mode: sandboxed replies try sandbox media resolution first, host-absolute local paths that do not map into the sandbox are dropped, and the visible fallback becomes `Media failed` before Telegram gets a usable upload source. The cron-owned delivery comment is useful related evidence, but it is an adjacent surface whose inclusion should be decided deliberately.

## What This Changes

The issue asks OpenClaw to deliver local files referenced by final-reply `MEDIA:` paths through Telegram instead of replacing the attachment with a `Media failed` fallback.

## Best Possible Solution

Add a narrow, policy-checked fallback for sandboxed final-reply media: after sandbox mapping fails, allow only managed media or host-local files permitted by the active outbound media policy, then stage them through the existing attachment path. Decide separately whether cron-owned announce delivery should use the same sandbox bridge.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with a sandbox root present, an absolute host-local `MEDIA:` path is rejected by sandbox mapping and the checked-in test expects that path to be dropped into the `Media failed` fallback. Tests were not run because this review had to remain read-only.

## Solution Assessment

No. Current main is not the best solution because it rejects host-local sandboxed reply media before the existing outbound media read policy can authorize it. The safer alternative is a policy-checked fallback, not unrestricted host path acceptance.
