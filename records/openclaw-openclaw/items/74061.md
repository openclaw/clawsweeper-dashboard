## Summary

Keep open. Current main still has a source-level reproduction for the reported Telegram final-reply media failure: sandboxed final replies send absolute host-local MEDIA paths through sandbox-only normalization and drop them before Telegram receives an upload, while docs still say policy-allowed host-local MEDIA paths are supported.

## What This Changes

The issue reports that Telegram final-reply delivery drops local `<final>MEDIA:` file paths and sends `Media failed` after v2026.4.26.

## Best Possible Solution

Add a narrow, policy-checked final-reply media bridge that reuses existing agent-scoped outbound media access for authorized host-local files, preserves sandbox escape and media-type negative coverage, and handles cron direct delivery only with explicit scope and tests.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: provide an absolute host-local MEDIA path while a sandbox root exists, and normalization drops it before Telegram upload; the current test suite also encodes that drop with workspaceOnly=false.

## Solution Assessment

No proposed patch is under review. The best fix is not unconditional host-path acceptance; it should route only authorized local media through the existing agent-scoped outbound media access path and keep sandbox escape coverage.
