## Summary

Keep open. Current main still matches the reported failure mode: sandboxed final replies route local MEDIA paths through sandbox mapping first, drop host-absolute paths when that mapping fails, and emit the Media failed fallback before Telegram gets a usable upload path. The later cron-owned delivery comment is useful corroborating evidence for the same sandbox-to-host translation gap on a separate delivery surface, but deciding whether to fix both surfaces together needs maintainer judgment.

## What This Changes

The issue asks OpenClaw to restore Telegram delivery for local files emitted as final-reply MEDIA paths after the April 26 release regression.

## Best Possible Solution

Add a narrow policy-checked fallback for sandboxed final-reply media: after sandbox mapping fails, allow only managed media or host-local files permitted by the active agent/session outbound media policy, then stage them through the existing outbound attachment loader. Keep negative coverage for unrelated host paths, file URLs, symlink escapes, and disallowed sender/group policies; decide explicitly whether cron-owned direct delivery should use the same bridge or get a separate follow-up.

## Reproduction Assessment

Yes. The current source and checked-in tests give a high-confidence source-level reproduction: with a sandbox root present, an absolute host-local MEDIA path is rejected by sandbox mapping, dropped, and converted to the Media failed fallback.

## Solution Assessment

No. Current main is not the best solution because it applies a sandbox-first rejection before the outbound media read policy can authorize host-local media; the safer fix is a policy-checked fallback rather than unrestricted host path acceptance.
