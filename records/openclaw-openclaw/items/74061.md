## Summary

Keep open. Current main still matches the reported Telegram final-reply failure mode: sandboxed final replies resolve local media through the sandbox mapper first, host-absolute paths that do not map into the sandbox are dropped, and the remaining user-visible result can become `Media failed` before Telegram receives a usable upload path. The cron-owned delivery comment is useful related evidence, but it is a second delivery surface that should be scoped deliberately.

## What This Changes

The issue asks OpenClaw to deliver local files referenced by final-reply `MEDIA:` paths through Telegram instead of replacing the attachment with a `Media failed` fallback.

## Best Possible Solution

Add a narrow, policy-checked fallback for sandboxed final-reply media: after sandbox mapping fails, permit only managed media or host-local files allowed by the active agent/session outbound media policy, then stage them through the existing outbound attachment path. Decide separately whether cron-owned announce delivery should share the same sandbox bridge.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with a sandbox root present, an absolute host-local `MEDIA:` path is rejected by sandbox mapping and the checked-in test expects that path to be dropped into the `Media failed` fallback. Tests were not run because this review had to remain read-only.

## Solution Assessment

No. Current main is not the best solution because it rejects host-local sandboxed reply media before the existing outbound media read policy can authorize it. The safer alternative is a policy-checked fallback, not unrestricted host path acceptance.
