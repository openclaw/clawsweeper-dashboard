## Summary

Keep open. Current main still matches the reported final-reply failure mode: when a sandbox root exists, local non-remote media is resolved through the sandbox mapper first, and an absolute host-local workspace path that does not map into the sandbox is dropped before Telegram gets a usable upload path. The cron-owned delivery comment is useful related evidence for the same sandbox-to-host media gap on another delivery path, but it should be scoped deliberately.

## What This Changes

The issue asks OpenClaw to restore Telegram delivery of local `MEDIA:` paths from sandboxed final replies after the v2026.4.26 regression.

## Best Possible Solution

Add a narrow, policy-checked fallback for sandboxed final-reply media: after sandbox mapping fails, permit only managed media or host-local files allowed by the active agent/session outbound media policy, then stage them through the existing outbound attachment path. Decide separately whether cron-owned announce delivery should share that bridge or get its own targeted fix.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with a sandbox root present, an absolute host-local `MEDIA:` path is rejected by sandbox mapping, and the checked-in test expects it to be dropped into the `Media failed` fallback. Tests were not run because this review is required to stay read-only.

## Solution Assessment

No. Current main is not the best solution because it rejects host-local sandboxed reply media before the existing outbound media read policy can authorize it. The safer alternative is a policy-checked fallback, not unrestricted host path acceptance.
