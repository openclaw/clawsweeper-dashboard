## Summary

Keep open. Current main and the latest release still preserve the reported sandboxed final-reply local media failure, and the later cron comment is useful corroboration for the same sandbox-to-host media translation gap on a second delivery path.

## What This Changes

The issue reports a Telegram regression where final-reply local `MEDIA:` attachments are replaced with `Media failed` instead of being uploaded.

## Best Possible Solution

Add a narrow, policy-checked sandbox-to-host media bridge for final replies and cron outbound that reuses existing agent-scoped outbound media access instead of accepting arbitrary host paths.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the checked-in test expects sandboxed absolute host-local media to be dropped, and the normalizer turns that drop into the reported warning when no media remains.

## Solution Assessment

No proposed patch is under review, and current main is not the best solution because it rejects policy-allowed host-local reply media before outbound media access can authorize it. The safer direction is a guarded fallback/bridge that preserves sandbox boundaries.
