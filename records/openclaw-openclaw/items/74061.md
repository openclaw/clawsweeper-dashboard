## Summary

Keep open. Current main still has the reported sandboxed final-reply failure mode: final `MEDIA:` paths are normalized before Telegram delivery, host-absolute paths are forced through sandbox mapping when a sandbox root exists, and a mapping failure drops the media into the `Media failed` fallback. The related cron comment is useful evidence for a second delivery surface, but it should be scoped deliberately rather than treated as already fixed.

## What This Changes

The issue requests that Telegram final replies deliver agent-generated local `MEDIA:` files instead of falling back to `Media failed`.

## Best Possible Solution

Add a narrow, policy-checked fallback for sandboxed final-reply media: after sandbox mapping fails, permit only managed media or host-local media allowed by the active agent/session outbound media policy, then stage it through the existing outbound attachment path. Decide separately whether cron-owned announce delivery should share that sandbox bridge or get a narrower follow-up.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with a sandbox root present, an absolute host-local `MEDIA:` path is rejected by sandbox mapping and the checked-in test expects that path to be dropped into the `Media failed` fallback. Tests were not run because this review was read-only.

## Solution Assessment

No. Current main is not the best solution because it rejects host-local sandboxed reply media before the existing outbound media read policy can authorize it; the safer alternative is a policy-checked staging fallback, not unrestricted host path acceptance.
