## Summary

Keep open. Current main still preserves the reported Telegram final-reply failure: sandboxed final replies resolve local `MEDIA:` through the sandbox mapper first, host-absolute paths outside `/workspace`, managed media, or OpenClaw tmp media are rejected, and the normalizer drops the attachment into the visible `⚠️ Media failed.` fallback. The cron-owned delivery comment is useful adjacent evidence, but its `/workspace` announce path should be scoped deliberately rather than folded into this issue by accident.

## What This Changes

Restore Telegram delivery for local files emitted as final-reply `MEDIA:` directives, especially host-local files generated by an agent after the April 26 release.

## Best Possible Solution

Add a narrow, policy-checked bridge for sandboxed final-reply media: after sandbox mapping fails, allow only managed media or host-local files that pass the active agent/session outbound media policy, then stage them through the existing outbound attachment path. Cover the reported Telegram final-reply case, keep negative tests for unrelated host paths, and explicitly decide whether cron-owned announce delivery should use the same sandbox bridge.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: final reply delivery calls the normalizer, sandboxed host-absolute local paths fail `resolveSandboxedMediaSource`, and the checked-in test expects that path to be dropped. I did not run live Telegram or tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. Current main is not the best solution because it applies a blanket sandbox-first rejection to host-absolute paths that may already be valid under the agent-scoped outbound media policy. The safer direction is a policy-checked fallback, not an unrestricted allowlist.
