## Summary

Keep open. Current main and the latest release still preserve the reported failure mode for sandboxed final-reply local media, and the later cron comment is useful corroborating evidence for the same sandbox-to-host media translation gap on another delivery path.

## What This Changes

The issue asks for Telegram final-reply `MEDIA:` local paths from agent workspaces to be staged and uploaded instead of falling back to `Media failed`.

## Best Possible Solution

Add a narrow sandbox-to-host media bridge for final-reply `MEDIA:` paths that falls back to existing agent-scoped outbound media access only when policy allows it, with cron-owned delivery scoped deliberately.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with a sandbox root present, an absolute host-local `MEDIA:` path is rejected by sandbox mapping, and the checked-in test expects it to be dropped with the media-failed fallback. Tests were not run because this review was required to stay read-only.

## Solution Assessment

No proposed patch is under review, and current main is not the best solution because it rejects policy-allowed host-local reply media before outbound media access can authorize it. The safer direction is a policy-checked fallback, not unconditional host-path acceptance.
