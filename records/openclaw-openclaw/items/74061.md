## Summary

Keep open. Current main still has the reported failure mode for sandboxed final replies: host-absolute local MEDIA paths are resolved through the sandbox mapper first, mapping failures are dropped, and the user can see the Media failed fallback before Telegram receives a usable upload path. The cron-owned delivery comment is useful related evidence, but it is a separate delivery surface whose scope should be decided deliberately.

## What This Changes

The issue asks OpenClaw to deliver local media referenced by final-reply MEDIA paths through Telegram instead of replacing the attachment with a Media failed fallback.

## Best Possible Solution

Add a narrow, policy-checked bridge for sandboxed final-reply media: after sandbox mapping fails, permit only managed media or host-local files allowed by the active agent/session outbound media policy, then stage them through the existing outbound attachment path. Keep negative tests for unrelated host paths and decide separately whether cron-owned announce delivery should share the same sandbox bridge.

## Reproduction Assessment

Yes. The current source and checked-in tests provide a high-confidence source-level reproduction: with a sandbox root present, an absolute host-local MEDIA path is rejected by sandbox mapping, dropped, and converted to the Media failed fallback. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. Current main is not the best solution because it applies a blanket sandbox-first rejection before the existing outbound media read policy can authorize a host-local file. The safer fix is a policy-checked fallback, not unrestricted host path acceptance.
