## Summary

Keep open: current main has device pairing plus exec/plugin approval primitives, but not a central risk-classified, device-bound step-up gate for high-risk chat-originated actions. The exact earlier duplicate at https://github.com/openclaw/openclaw/issues/91689 was closed by its opener after ClawSweeper had marked it for maintainer follow-up, so this remains useful as the open tracker.

## What This Changes

Request a central gateway step-up authorization layer for high-risk actions that originate from IM channels or channel plugins, backed by trusted-device approval and risk classification.

## Best Possible Solution

Design this as a core gateway authorization gate that reuses device pairing and approval primitives, defines an upgrade-safe risk taxonomy and policy defaults, and then threads the check through chat-originated tool/action execution with focused tests and docs.

## Reproduction Assessment

Not applicable; this is a feature/security design request, not a broken existing contract. Source inspection confirms current main lacks the requested ActionRequest-style step-up and trusted-device challenge surface.

## Solution Assessment

Unclear pending maintainer design: a central gateway gate appears to be the right layer, but maintainers need to decide risk classification, enrollment rules, fail-closed defaults, and upgrade behavior before implementation.
