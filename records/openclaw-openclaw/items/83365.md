This should stay open: the report is fresh, concrete, and current main still appears to route normal iMessage final media delivery through generic RPC `send`, while the known-good `send-rich --file` bridge route is only wired for iMessage message actions.

## What This Changes

The issue reports that generated image final replies are not delivered to an allowlisted iMessage group even though manually sending the same file with `imsg send-rich --file` succeeds.

## Best Possible Solution

Fix the iMessage final media delivery path so group media replies use a verified group-capable bridge route or fail visibly and retryably, while preserving the existing media resolver safeguards.

## Reproduction Assessment

No high-confidence live reproduction was performed in this review. The reporter provided two real group failures, and current source still shows the final media path using generic RPC `send` rather than the manual `send-rich --file` path that worked.

## Solution Assessment

Yes, a narrow iMessage outbound-media fix is the right direction. It should reuse the existing media resolver and private-API capability checks rather than adding a parallel raw-path send shortcut.

## Repair Work Prompt

Fix the iMessage final media delivery bug reported in https://github.com/openclaw/openclaw/issues/83365. Current main advertises iMessage durable final media support, but final media replies resolve a file and call the generic RPC `send` path in `extensions/imessage/src/send.ts`; the reporter proved the same generated file delivers to the group through `imsg send-rich --chat iMessage;+;<group-guid> --file <path>`. Keep the fix inside the iMessage plugin boundary, reuse the existing outbound media resolver/local-root/size safeguards, feature-detect the rich-send attachment capability before using it, and do not pass arbitrary raw agent-supplied paths directly to imsg. Add focused tests for group/chat-guid media final sends and for the unsupported-rich-send failure behavior. Changelog is maintainer-owned at landing if the fix is user-visible.
