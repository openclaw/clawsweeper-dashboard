## Summary

This should stay open: the report is fresh, concrete, and current main still sends normal iMessage final media through the generic RPC `send` path while the working `send-rich --file` bridge route is only wired for action-style rich replies.

## What This Changes

The issue reports that generated image final replies can fail to appear in an allowlisted iMessage group even though the same file sends successfully with direct `imsg send-rich --file`.

## Best Possible Solution

Fix the iMessage final media path so group media replies use a verified group-capable bridge route or emit a clear retryable failure, while preserving resolver safeguards and existing text/DM behavior.

## Reproduction Assessment

No high-confidence live reproduction was performed in this review. The reporter's repeated live evidence plus current source routing make the issue source-reproducible, but a fix still needs real iMessage group proof.

## Solution Assessment

Yes, a narrow iMessage outbound-media fix is the right direction. The safest route is to reuse existing media resolution and bridge capability checks instead of adding a raw-path send bypass.
