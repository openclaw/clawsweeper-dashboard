## Summary

This should stay open: the report is fresh, concrete, and current `main` still routes iMessage final media through the generic RPC `send` path while the known-good rich/attachment bridge path is only wired for message-tool actions.

## What This Changes

The issue reports that generated image final replies to an allowlisted native iMessage group can be dropped while the same file succeeds through direct `imsg send-rich --file`.

## Best Possible Solution

Fix the iMessage final media path so group media replies use a verified group-capable bridge route or emit a clear retryable failure, while preserving media resolver safeguards.

## Reproduction Assessment

No high-confidence live reproduction was performed against a real macOS iMessage group. The report's repeated live evidence plus current source routing makes the issue source-reproducible enough to keep open.

## Solution Assessment

Yes, a narrow iMessage outbound-media fix is the right direction. The safest approach is to reuse the existing resolver and bridge capability checks instead of adding a parallel raw-path send path or changing generic outbound delivery.
