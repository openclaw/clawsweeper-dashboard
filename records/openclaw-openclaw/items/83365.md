## Summary

This should stay open: the report is fresh, has concrete repeated iMessage group media-loss evidence, and current main still routes normal final media through the generic iMessage RPC `send` path while the known-good `send-rich --file` bridge route is wired for message actions rather than final replies.

## What This Changes

The issue reports that generated iMessage group media final replies are not delivered even though the same generated file sends successfully with direct `imsg send-rich --file`.

## Best Possible Solution

Fix the iMessage final media path so group media replies use a verified group-capable, resolver-backed bridge route or return a clear retryable delivery failure.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The reporter provides two real group failures, and current source shows the implicated final media path still uses generic RPC `send` while the working `send-rich --file` route is action-only.

## Solution Assessment

Yes, a narrow iMessage outbound-media repair is the best direction. The safer implementation should reuse existing media resolution and imsg capability detection rather than adding a parallel raw-path sender.
