## Label Justifications

- `P1`: The bug can make a fresh `/new` or `/reset` conversation receive stale exec approval continuation output in a core agent workflow.
- `impact:session-state`: The issue is about async approval state crossing from an old runtime session into a new session instance.
- `impact:message-loss`: The stale follow-up can be misrouted into the wrong conversation after the session key is rebound.

## Summary

Keep open. Current main still builds and dispatches exec approval follow-ups with only `sessionKey`, while `/new` and `/reset` rotate the current `sessionId` under that key and there is no closing PR or duplicate that owns the exact fix.

## What This Changes

The issue asks OpenClaw to bind exec approval follow-ups to the original session instance and drop stale completions after `/new` or `/reset` rebinds the same session key.

## Best Possible Solution

Bind exec approval follow-ups to the session instance that created them and discard stale completions when the stable key now resolves to a different session id.

## Reproduction Assessment

Yes, at source level. The reporter gives concrete pending-approval plus `/new` or `/reset` steps, and current source still dispatches the follow-up by `sessionKey` while reset writes a fresh `sessionId` for that key.

## Solution Assessment

Yes. Capturing the expected session id at approval creation and checking the current key mapping before any agent resume or direct fallback is the narrowest maintainable fix.
