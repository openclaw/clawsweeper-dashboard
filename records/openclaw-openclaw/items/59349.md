## Summary

Keep open. Current main still builds exec approval follow-up targets with `sessionKey` and delivery hints only, while `/new` and `/reset` rotate the persisted `sessionId` for the same key; I did not find an `expectedSessionId` guard or regression coverage for dropping stale approval follow-ups.

## What This Changes

The issue requests binding asynchronous exec approval follow-up delivery to the original session instance so `/new` or `/reset` cannot receive stale approval output.

## Best Possible Solution

Bind each exec approval follow-up target to the session id that was active when the approval was requested, and before any model follow-up or direct fallback, compare that id with the current session store entry for the same key. If the key now points to a different session id, log a concise stale-follow-up message and drop it; preserve existing same-session gateway, node, Discord, and webchat follow-up behavior.

## Reproduction Assessment

Yes. The report gives a concrete manual repro, and the current source path makes it reproducible with a targeted unit/integration test: create a pending exec approval, rotate the same sessionKey through `sessions.reset`, resolve the old approval, and observe that current code still dispatches by sessionKey.

## Solution Assessment

Yes. The proposed `expectedSessionId` guard is the narrow maintainable fix because it addresses the session-instance boundary without changing `/new` semantics, approval ids, or channel delivery routing.

## Repair Work Prompt

Fix openclaw/openclaw#59349. Current exec approval follow-up targets are bound only to `sessionKey`, so an approval that resolves after `/new` or `/reset` can dispatch into the new session now mapped to that key. Add an original-session guard: capture the active session id when constructing gateway and node exec approval follow-up targets, carry it through `ExecApprovalFollowupTarget` / `sendExecApprovalFollowup`, and before calling the `agent` RPC or using direct fallback, compare the current session-store entry for the target `sessionKey` with the captured expected session id. If the key now maps to a different session id, log/drop the stale follow-up and do not deliver stale text. Do not pass the old id to `agent` in a way that rebinds the store. Preserve current same-session auto-follow-up behavior for gateway, node, Discord external delivery, and webchat/session-only flows. Add focused regression coverage for: pending approval -> session reset -> approval resolve drops follow-up; same-session delayed approval still dispatches; deny follow-up does not cross reset. Add a user-facing changelog fix entry crediting `Thanks @two3pro`. Keep the adjacent lossless/context-engine issue out of scope unless a shared helper naturally benefits both.
