## Summary

Keep open. Current main still creates and dispatches exec approval follow-ups by `sessionKey` without binding them to the original `sessionId`, while `/new` and `/reset` rotate the stored session id under the same key; the later user comment is consistent with the same stale-follow-up class.

## What This Changes

The issue requests binding exec approval follow-up delivery to the original session instance so stale approvals cannot enter a later `/new` or `/reset` session.

## Best Possible Solution

Capture the active session id when the exec approval follow-up target is created, then drop and log any later follow-up whose session key now maps to a different session id before either model continuation or direct fallback delivery.

## Reproduction Assessment

Yes. The report gives a manual pending-approval plus `/new` or `/reset` reproduction, and current source inspection supports a focused test that creates a follow-up target, rotates the session key, resolves the old approval, and verifies no follow-up is delivered.

## Solution Assessment

Yes. An `expectedSessionId` guard is the narrowest maintainable fix because it preserves existing approval ids, channel routing, `/new` semantics, and same-session follow-up behavior while enforcing the session-instance boundary.

## Repair Work Prompt

Fix openclaw/openclaw#59349. Bind async exec approval follow-up targets to the session id that was current when the approval request was created. Add an `expectedSessionId` or equivalent field to the follow-up target, populate it in both gateway-host and node-host approval paths, and before any `agent` continuation or direct fallback delivery compare the current session-store entry for the target session key with that expected id. If the key now maps to a different session id, log a concise stale-follow-up message and drop the result. Preserve existing same-session gateway, node, Discord, Telegram, webchat, and direct-mode behavior; do not change approval id semantics, `/new` behavior, or channel delivery routing. Add focused regression coverage for resolving an old pending approval after `sessions.reset` or `/new` rebinds the same session key, and add a user-facing changelog fix entry crediting the human reporter if policy allows.
