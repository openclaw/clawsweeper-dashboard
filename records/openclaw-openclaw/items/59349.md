## Summary

Keep open. Current main still has the reported session-instance leak shape: exec approval follow-ups are created and dispatched by `sessionKey` only, while `/new` and `/reset` rotate `sessionId` under that same key, and no related open PR owns the original-session guard.

## What This Changes

Prevent stale exec approval follow-ups from being delivered into a newer session after `/new` or `/reset` rebinds the same logical `sessionKey`.

## Best Possible Solution

Bind each exec approval follow-up target to the session id that was active when the approval was created, then drop and log the follow-up before any agent resume or direct fallback if the session key now maps to a different session id.

## Reproduction Assessment

Yes. The reporter gives concrete pending-approval plus `/new` or `/reset` steps, and current source inspection shows the follow-up path remains keyed by `sessionKey` while reset rotates `sessionId` under that key.

## Solution Assessment

Yes. An expected-session-id guard is the narrowest maintainable fix because it preserves current follow-up delivery for live sessions while dropping stale work after a session-key rebind.
