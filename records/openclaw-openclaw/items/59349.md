## Label Justifications

- `P1`: A pending exec approval can pollute a fresh /new or /reset agent session, breaking a core user workflow.
- `impact:session-state`: The bug is specifically about async approval state crossing from an old runtime session into a new session instance.

## Summary

Keep open. Current main and the latest release still route exec approval followups by sessionKey without an approval-time session-id guard, and the linked open fix PR now owns the exact remaining work.

## What This Changes

The issue requests binding exec approval followups to the original runtime session id so late approvals after /new or /reset are dropped instead of delivered into a fresh session.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85679 if maintainers accept the session-id guard, then close this issue as fixed by that PR.

## Reproduction Assessment

Yes, at source level. Current main creates and dispatches exec approval followups with only sessionKey while /new and /reset regenerate the mapped sessionId under that key.

## Solution Assessment

Yes. Capturing the approval-time session id and checking the current key mapping before gateway continuation or direct fallback delivery is the narrow maintainable fix; the open linked PR implements that direction for review.
