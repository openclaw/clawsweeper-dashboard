## Label Justifications

- `P1`: The bug can make a fresh `/new` or `/reset` conversation receive stale exec approval continuation output, breaking a core agent session workflow for real users.
- `impact:session-state`: The report is about async approval state crossing from an old runtime session into a new session instance.
- `impact:message-loss`: The stale follow-up is effectively misrouted into the wrong conversation after the session key is rebound.

## Summary

Keep open. Current main still builds and dispatches exec approval follow-ups with only `sessionKey`, while `/new` and `sessions.reset` rotate the session instance under that key; the recent final-delivery refresh guard is adjacent but does not guard this follow-up dispatch path.

## What This Changes

The issue asks OpenClaw to keep async exec approval follow-ups tied to the original session instance and drop them after `/new` or `/reset` rebinds the same `sessionKey`.

## Best Possible Solution

Bind exec approval follow-ups to the session instance that created them and discard stale completions after the stable key is reset to a different session id.

## Reproduction Assessment

Yes, at source level. The reporter gives concrete pending-approval plus `/new` or `/reset` steps, and current source still creates and dispatches the follow-up with only `sessionKey` while reset rewrites that key to a new `sessionId`; I did not run live gateway proof in this read-only review.

## Solution Assessment

Yes. Capturing the expected session id when the approval request is created and dropping/logging mismatches before any agent resume or direct fallback is the narrowest maintainable fix.
