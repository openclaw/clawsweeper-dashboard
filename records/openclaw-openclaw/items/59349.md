## Summary

Keep open. Current main still builds and dispatches exec approval follow-ups by `sessionKey` without preserving the original session id, while `/new` and `/reset` rotate the stored `sessionId` under that same key; the later user comment is consistent with the same stale-follow-up boundary bug.

## What This Changes

The issue requests that exec approval follow-ups stay bound to the original session instance and be dropped after `/new` or `/reset` rebinds the same session key to a new session id.

## Best Possible Solution

Capture the active session id when the approval follow-up target is created, then drop and log any follow-up whose session key now maps to a different session id before any model continuation or direct fallback delivery.

## Reproduction Assessment

Yes. The issue gives a concrete pending-approval plus `/new` or `/reset` flow, and current source supports a focused regression test that creates a follow-up target, rotates the session key to a new session id, resolves the old approval, and verifies no follow-up is delivered.

## Solution Assessment

Yes. An `expectedSessionId` guard is the narrowest maintainable fix because it preserves existing approval ids, routing, and `/new` semantics while enforcing the session-instance boundary; passing the old session id into `agent` would be a less safe alternative.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/59349. Current exec approval follow-ups are created and dispatched with only `sessionKey`; after `/new` or `/reset`, that key can point at a new `sessionId`, so resolving the old approval can resume or message the fresh session. Add a session-instance guard: capture the active session id when creating the exec approval follow-up target, carry it through shared/gateway/node follow-up paths, and before dispatch compare it with the current sessionKey -> sessionId mapping. If the mapping differs, log/drop the stale follow-up before both agent continuation and direct fallback delivery. Do not pass the old session id into the `agent` RPC as the fix, do not change exec approval policy/security defaults, and do not change `/new` or `/reset` semantics beyond dropping stale follow-ups. Add focused tests for the stale-session drop and keep existing same-session follow-up behavior.
