## Summary

Keep open. Current main still routes exec approval follow-ups by `sessionKey` only, while `/new` and `sessions.reset` replace the session instance under that key, so the reported stale follow-up path remains source-reproducible.

## What This Changes

The issue asks for exec approval follow-ups to stay bound to the original runtime session and be dropped after `/new` or `/reset` rebinds the same session key.

## Best Possible Solution

Add an internal session-instance guard: capture the active runtime `sessionId` when creating the exec approval follow-up target, then drop and log the follow-up if the current `sessionKey` mapping no longer matches.

## Reproduction Assessment

Yes. The reporter gives concrete pending-approval plus `/new` or `/reset` steps, and current source shows the follow-up is still keyed by `sessionKey` while reset replaces the `sessionId` under that key; I did not run a live gateway repro in this read-only pass.

## Solution Assessment

Yes. Capturing an expected session id and dropping mismatches is the narrowest maintainable fix because it preserves normal follow-up delivery while enforcing the existing `/new` session boundary.
