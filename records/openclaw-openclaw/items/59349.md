## Summary

Keep open. Current main still creates and dispatches exec approval follow-ups by `sessionKey` without binding them to the original `sessionId`, while `/new` and `/reset` rotate the session id under the same key.

## What This Changes

The issue requests that async exec approval follow-ups be bound to the original session instance so stale approvals are dropped after `/new` or `/reset`.

## Best Possible Solution

Capture the active session id with each exec approval follow-up target, then drop and log stale follow-ups when the current `sessionKey` mapping no longer matches before any continuation or direct fallback is sent.

## Reproduction Assessment

Yes. The issue gives a concrete pending-approval plus `/new` or `/reset` reproduction, and current source inspection shows the same key-only routing path remains.

## Solution Assessment

Yes. The proposed `expectedSessionId` guard is the narrowest maintainable fix because it preserves approval ids, channel routing, and same-session follow-ups while enforcing the session-instance boundary.
