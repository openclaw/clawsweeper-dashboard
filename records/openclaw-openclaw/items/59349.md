## Summary

Keep open. Current main still routes async exec approval follow-ups by `sessionKey` plus turn-source metadata only, while `/new` and `/reset` rotate the `sessionId` behind the same key, so the reported stale follow-up leak remains a valid narrow bug.

## What This Changes

The issue requests binding exec approval follow-ups to the original session instance so stale approval completions are dropped after `/new` or `/reset`.

## Best Possible Solution

Capture the active `sessionId` when the exec approval follow-up target is created, compare it with the current session store mapping immediately before any follow-up delivery, and drop stale mismatches without changing same-session approval behavior.

## Reproduction Assessment

Yes. The report gives a concrete manual repro, and the current code path makes it reproducible with a focused test that creates a pending approval, rotates the same `sessionKey` through `/new` or `sessions.reset`, resolves the old approval, and observes dispatch by the refreshed key.

## Solution Assessment

Yes. The proposed `expectedSessionId` guard is the narrow maintainable fix because it enforces the session-instance boundary without changing `/new` semantics, approval ids, channel routing, or normal same-session follow-up continuation.

## Repair Work Prompt

Fix openclaw/openclaw#59349. Current main constructs exec approval follow-up targets with `sessionKey` and route metadata only, so an old pending approval can continue into a fresh `/new` or `/reset` session after the same key is rebound to a new `sessionId`. Add a narrow `expectedSessionId`/original-session guard: capture the active session id when the gateway and node exec approval follow-up targets are created, then before any `agent` follow-up or direct fallback delivery compare that id with the latest session store entry for the same key. If the key now maps to a different session id, log a concise stale-follow-up drop and do not deliver. Preserve existing same-session gateway, node, Discord/Telegram/Slack/webchat, no-route, denied, and direct-mode behavior. Do not change `/new` semantics, approval id semantics, broad session-store behavior, or context-engine/lossless-claw behavior from adjacent #73784.
