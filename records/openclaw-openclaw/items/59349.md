## Summary

Keep open. Current main still creates and sends exec approval follow-ups by sessionKey without binding the original sessionId, while /new and /reset rotate the sessionId under that key.

## What This Changes

The issue requests that exec approval follow-ups be dropped when their original sessionKey now points at a different sessionId after /new or /reset.

## Best Possible Solution

Bind each exec approval follow-up target to the sessionId active when the approval was created, then drop and log stale follow-ups when the sessionKey now maps to a different sessionId.

## Reproduction Assessment

Yes. The reporter provides concrete pending-approval plus /new or /reset steps, and current source shows the follow-up path is still keyed by sessionKey while reset rotates sessionId under that key.

## Solution Assessment

Yes. An expected-session-id guard is the narrowest maintainable fix because it preserves current follow-up delivery for live sessions while dropping stale work after a session-key rebind.
