## Summary

Keep open. Current main still has the reported session-instance leak shape: exec approval follow-ups are created and dispatched by sessionKey only, while /new and /reset replace the sessionId stored under that same key, and no linked or closing PR currently owns this exact fix.

## What This Changes

The issue asks for exec approval follow-ups to stay bound to the original session instance and be dropped after /new or /reset rebinds the same sessionKey to a different sessionId.

## Best Possible Solution

Capture the active session id when an exec approval follow-up target is created, then drop and log the follow-up before any agent resume or direct fallback if the sessionKey now maps to a different session id.

## Reproduction Assessment

Yes. The report gives a concrete pending-approval plus /new or /reset reproduction, and current source inspection shows the exact path remains keyed by sessionKey without an original sessionId guard.

## Solution Assessment

Yes. An expected-session-id guard is the narrowest maintainable fix; passing the old sessionId into the current agent RPC without a mismatch guard would risk resurrecting or attaching to the wrong session instead of dropping stale work.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/59349. Current main creates exec approval follow-up targets with sessionKey only, so pending approval results can resume or directly notify a newer session after /new or /reset rebinds the same key to a different sessionId. Capture the original active session id when building ExecApprovalFollowupTarget in the gateway and node exec approval paths, then before dispatching a follow-up compare it with the current sessionKey -> sessionId mapping and drop/log stale mismatches before either callGatewayTool("agent") or direct fallback delivery. Do not pass an old requested sessionId into the existing agent RPC as a substitute for the mismatch guard, and do not change /new, /reset, approval id, or normal same-session follow-up semantics. Add focused regression coverage for pending approval follow-up, session reset/rebind, and stale approved/denied delivery being dropped; update CHANGELOG.md because this is user-visible reliability/privacy behavior. Keep the fix separate from the bundle-MCP runtime disposal race in https://github.com/openclaw/openclaw/issues/76233 and the already-closed task-liveness issue https://github.com/openclaw/openclaw/issues/76162.
