## Label Justifications

- `merge-risk: 🚨 message-delivery`: The patch changes delivery-completion and fallback behavior, and the current aggregate gate can still suppress failure handling after prior Telegram deliveries.

## Summary

The PR should stay open because the direction is valid, but the patch still lets earlier Telegram deliveries mask an unconfirmed final reply, which is the central failure mode it is trying to close. It also needs real Telegram after-fix proof before merge because the PR only reports local tests and does not include visible/live delivery evidence.

## What This Changes

The PR adds delivered-count tracking to reply dispatch, threads final-delivery outcome data through configured dispatch, and changes Telegram final-reply handling to emit fallback or retryable failure when final delivery is not confirmed.

## Best Possible Solution

Make Telegram final completion depend on final-specific delivery evidence, add coverage where a prior non-final delivery succeeds but the final is unconfirmed, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes at source level: a prior progress/block delivery marks aggregate Telegram delivery state, and the PR then checks that aggregate state when deciding whether an unconfirmed final needs fallback. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Requiring delivery evidence is the right direction, but using aggregate Telegram delivery state is not the best fix because non-final deliveries can mask final failure.
