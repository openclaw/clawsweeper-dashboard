## Summary

Keep open. Current main has a concrete `model.usage` diagnostic payload and exporter mappings, but the public logging/OpenTelemetry docs still do not publish the exact JSON field table, sample event, or supported local parsing surface requested by the reporter.

## What This Changes

The issue asks OpenClaw to document the exact JSON field names for `model.usage` events so an external read-only tool can parse local inference usage logs.

## Best Possible Solution

Add a focused docs/contract update that publishes the current `model.usage` payload fields and a sample event, maps them to OTEL/Prometheus names, distinguishes camelCase OpenClaw fields from provider-native snake_case counters, explains `usage` versus `lastCallUsage` versus `context`, and states which local/plugin/exporter surfaces external parsers can rely on today.

## Reproduction Assessment

Not applicable as a docs/API-contract request rather than a runtime bug. The gap is high-confidence from comparing the current source and tests with the public logging/OpenTelemetry docs: the event shape exists, but the field schema is not documented.

## Solution Assessment

Yes, a docs/contract repair is the narrowest maintainable solution. Runtime behavior should not change here, and any decision to make the raw `/tmp/openclaw` file log a stable external API should be explicit rather than implied by this issue.
