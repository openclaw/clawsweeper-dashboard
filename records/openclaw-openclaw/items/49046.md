## Summary

Keep open. Current `main` has a concrete `model.usage` diagnostic payload and regression coverage for key usage/context semantics, but the public logging/OpenTelemetry docs still summarize the event instead of publishing exact JSON keys, an example event, or clear guidance on supported read-only parsing surfaces.

## What This Changes

The issue asks OpenClaw to document the exact field names for `model.usage` entries so external read-only usage parsers can avoid guessing.

## Best Possible Solution

Add a focused docs/contract update that publishes the current `model.usage` JSON payload fields and an example, maps them to OTEL/Prometheus metric names, explains `usage` versus `lastCallUsage` versus `context`, and states which local surfaces external parsers can safely consume today.

## Reproduction Assessment

Not applicable as a docs/API-contract request rather than a runtime bug. The gap is high-confidence from source and docs inspection: the event shape exists in code and tests, but public docs do not expose the full schema or supported parser surface.

## Solution Assessment

Yes, a docs/contract update is the narrowest maintainable solution for the stated request. Runtime behavior should stay unchanged unless maintainers separately decide to make local file logs a stable usage-event API.
