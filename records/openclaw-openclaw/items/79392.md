## What This Changes

The issue reports that Gemini LLM calls time out around 125 seconds even when `agents.defaults.timeoutSeconds` is set to 300 seconds and the old `agents.defaults.llm.idleTimeoutSeconds` key is rejected.

## Best Possible Solution

Keep the remaining timeout work centralized in the open canonical timeout/config issues, preserving this Gemini data point there instead of tracking another separate issue for the same idle-watchdog/config surface.

## Reproduction Assessment

Yes, source-reproducible for the reported timing path: current main clamps `agents.defaults.timeoutSeconds=300` to the 120s idle watchdog unless a model request timeout is attached, matching the reported `durationMs=124826` pattern. I did not run a live Gemini/Docker reproduction.

## Solution Assessment

No, reopening the old `agents.defaults.llm` key in this separate issue is not the best path. The maintainable fix should be handled once through the provider/per-model timeout contract already tracked in https://github.com/openclaw/openclaw/issues/77744 and https://github.com/openclaw/openclaw/issues/8724.
