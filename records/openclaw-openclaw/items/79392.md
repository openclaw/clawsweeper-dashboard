## Summary

Keep open. Current main still caps the reported `agents.defaults.timeoutSeconds=300` path, and remote provider request timeouts, to the 120s cloud idle watchdog; the legacy `agents.defaults.llm` key is intentionally removed, and adjacent timeout/Gemini reports do not fully supersede this config-contract plus Gemini failure report.

## What This Changes

The issue asks OpenClaw to stop aborting Gemini LLM requests around the 120s idle watchdog when `agents.defaults.timeoutSeconds=300` is configured and to clarify the replacement for the removed `agents.defaults.llm` key.

## Best Possible Solution

Define one supported slow-model timeout path for cloud Gemini and other remote thinking models, with docs and regression coverage showing whether provider-scoped timeout widens the cloud idle watchdog or a provider-specific Gemini thinking fix owns the behavior, without reviving `agents.defaults.llm`.

## Reproduction Assessment

Yes for the source-level mismatch: current main still resolves `agents.defaults.timeoutSeconds=300` and remote `modelRequestTimeoutMs=300_000` through the 120s idle watchdog cap. I did not run the live Docker/Gemini request path.

## Solution Assessment

Unclear. Re-accepting `agents.defaults.llm` would reverse an intentional migration, while simply removing the 120s cap changes current cloud-provider timeout policy; maintainers should choose the intended contract first.
