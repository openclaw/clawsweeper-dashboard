## Summary

Keep open. Current main still caps `agents.defaults.timeoutSeconds=300` down to the 120s model idle watchdog and rejects the removed `agents.defaults.llm` key, while the related open timeout/Gemini work does not fully own this Gemini plus config-contract report.

## What This Changes

The issue asks OpenClaw to stop aborting Gemini model calls around 120s when `agents.defaults.timeoutSeconds` is configured to 300, and notes that the former `agents.defaults.llm.idleTimeoutSeconds` key is now rejected.

## Best Possible Solution

Maintain one clear supported path for slow Gemini and other cloud model requests, with docs and regression coverage showing whether operators should use provider-scoped timeout config or an explicitly supported global model-idle timeout.

## Reproduction Assessment

Yes for the source-level config/timeout mismatch: current code and tests still resolve `agents.defaults.timeoutSeconds=300` to the 120s idle watchdog and reject the legacy `agents.defaults.llm` key. I did not run the live Docker/Gemini request path.

## Solution Assessment

Unclear. A mechanical change to remove the 120s cap or re-accept `agents.defaults.llm` would change the current provider-scoped timeout contract, so maintainers should choose the intended policy before implementation.
