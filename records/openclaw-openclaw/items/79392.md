## Summary

Keep open. Current main still preserves the reported source-level behavior for normal remote Gemini runs: `agents.defaults.timeoutSeconds=300` is capped to the 120s model idle watchdog, the retired `agents.defaults.llm` key is rejected, and the merged cron timeout fix does not cover this non-cron cloud-provider case.

## What This Changes

The issue reports that Dockerized Gemini requests still time out around 120 seconds even when `agents.defaults.timeoutSeconds` is set to 300, and that the previously suggested `agents.defaults.llm.idleTimeoutSeconds` key is no longer accepted.

## Best Possible Solution

Define one supported contract for slow cloud model first-token and idle gaps, then align the resolver, docs, and regression tests around that contract without reviving the retired `agents.defaults.llm` key by accident.

## Reproduction Assessment

Yes for the source-level path: current main still resolves `agents.defaults.timeoutSeconds=300` and remote `modelRequestTimeoutMs=300000` to the 120s idle watchdog, while rejecting the legacy `agents.defaults.llm` key. I did not reproduce the live Docker/Gemini request path.

## Solution Assessment

Unclear. Re-accepting `agents.defaults.llm` would reverse an intentional config migration, while removing the 120s cloud idle cap would change current documented/tested policy; maintainers should choose whether existing provider timeout or a first-token/idle split is the intended fix.
