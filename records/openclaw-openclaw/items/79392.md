## Summary

Keep open. Current main and the latest release still cap `agents.defaults.timeoutSeconds=300` down to the 120s model idle watchdog for the reported non-cron Gemini path, while the retired `agents.defaults.llm` key is still rejected and the merged cron fix does not cover this standard interactive path.

## What This Changes

The issue reports Gemini model calls timing out around 125 seconds despite `agents.defaults.timeoutSeconds=300`, and notes that the former `agents.defaults.llm.idleTimeoutSeconds` key is now rejected.

## Best Possible Solution

Define and document one supported timeout path for slow Gemini/cloud model requests, then update the resolver, docs, and regression coverage to match that chosen policy.

## Reproduction Assessment

Yes for the source-level mismatch: current main still resolves `agents.defaults.timeoutSeconds=300` to the 120s idle watchdog and rejects the retired `agents.defaults.llm` key. I did not reproduce the live Docker/Gemini request path.

## Solution Assessment

Unclear. A mechanical change to remove the 120s cap or re-accept `agents.defaults.llm` would contradict current docs/tests, so maintainers should first decide the intended remote-cloud timeout contract.
