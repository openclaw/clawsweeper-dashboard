## Summary

Keep open. Current main still caps `agents.defaults.timeoutSeconds=300` to the 120s LLM idle watchdog unless a provider-scoped request timeout is configured, and it intentionally rejects the removed `agents.defaults.llm` key; related timeout items overlap but do not fully own this Gemini plus config-contract report.

## What This Changes

The issue asks for LLM requests, especially Gemini requests, not to abort around 120 seconds when the user has configured a 300 second timeout, and notes that the old `agents.defaults.llm.idleTimeoutSeconds` path is now rejected.

## Best Possible Solution

Maintain one clear supported path for slow Gemini and other cloud model requests, with docs and regression coverage showing whether operators should use `models.providers.google.timeoutSeconds` or an intentionally supported global model-idle timeout.

## Reproduction Assessment

Yes for the source-level config/timeout mismatch: current code and tests resolve `agents.defaults.timeoutSeconds=300` to the 120s idle watchdog and reject `agents.defaults.llm`. I did not run the live Docker/Gemini request path.

## Solution Assessment

Unclear. A mechanical change to remove the 120s cap or re-accept `agents.defaults.llm` would change the current provider-scoped timeout contract, so maintainers should choose the intended policy before implementation.
