## Summary

Keep open. Current main still preserves the source-level config mismatch: `agents.defaults.timeoutSeconds=300` resolves to the 120s model idle watchdog and the legacy `agents.defaults.llm` key is intentionally rejected, while the merged Gemini first-response fix does not change the generic timeout/config policy.

## What This Changes

The issue asks OpenClaw to stop aborting Gemini LLM requests before the operator-configured timeout and clarifies that the old `agents.defaults.llm.idleTimeoutSeconds` workaround is no longer accepted.

## Best Possible Solution

Define and test one supported operator path for slow Gemini/cloud requests, either by documenting and validating provider-scoped timeout behavior for Google or by deliberately changing the agent idle-timeout policy with migration guidance.

## Reproduction Assessment

Yes for the source-level config/timeout mismatch: current code and tests still resolve `agents.defaults.timeoutSeconds=300` to the default 120s idle watchdog and reject `agents.defaults.llm`. I did not run the live Docker/Gemini request path.

## Solution Assessment

Unclear. A direct code change to remove the 120s cap or restore `agents.defaults.llm` would change the documented timeout contract; maintainers should first choose whether Google should use provider-scoped timeout tuning or a broader agent-level idle timeout.
