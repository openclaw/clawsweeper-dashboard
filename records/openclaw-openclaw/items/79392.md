## Summary

Keep open. Current main still has the reported timeout-policy mismatch: `agents.defaults.timeoutSeconds=300` is intentionally capped at the 120s model idle watchdog, while the legacy `agents.defaults.llm` key is removed and only migrated toward provider-scoped timeout configuration.

## What This Changes

The issue reports that Gemini requests still time out around 125s on OpenClaw 2026.5.7 despite `agents.defaults.timeoutSeconds=300`, and that the old `agents.defaults.llm.idleTimeoutSeconds` key is rejected.

## Best Possible Solution

Maintain a clear supported path for slow Gemini and other cloud model requests: either make provider-scoped timeout configuration easy and well-diagnosed for bundled providers, or explicitly restore/add a documented global model-idle timeout policy with regression coverage.

## Reproduction Assessment

Yes for the config/timeout mismatch: current source and tests show `agents.defaults.timeoutSeconds=300` still resolves to the 120s idle watchdog, and the legacy `agents.defaults.llm` key is intentionally removed. I did not run a live Docker/Gemini request.

## Solution Assessment

Unclear. A mechanical change to lift the 120s cap or re-accept `agents.defaults.llm` would conflict with the current provider-scoped timeout direction, so maintainers should choose the intended config contract before implementation.
