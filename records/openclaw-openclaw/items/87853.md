## Summary

Keep this PR open, but not merge-ready: the rewritten startup-only Discord jitter avoids the unsupported `baseDelay` approach, yet it schedules a delayed `super.connect()` that `disconnect()` and startup cleanup cannot cancel.

## What This Changes

The PR passes Discord account IDs into gateway plugin construction, adds deterministic per-account startup jitter before the first non-resume connect, and tests the jitter hash.

## Best Possible Solution

Keep the startup jitter idea, but make the timer lifecycle-owned: store the pending handle, clear it before `disconnect()` can return, and cover normal delayed startup plus disconnect-before-fire with fake-timer tests.

## Reproduction Assessment

Yes, source-reproducible: with a nonzero account startup jitter, call `connect(false)`, then `disconnect()` before the timer fires; the queued callback later calls `super.connect(false)`. I did not run the fake-timer repro because this review is read-only.

## Solution Assessment

No, not as submitted. The one-shot jitter is a plausible fix direction, but it must own and cancel its pending timer so gateway cleanup remains authoritative.
