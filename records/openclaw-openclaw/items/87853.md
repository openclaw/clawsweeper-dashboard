## Summary

Keep open: current main still lacks the Discord startup staggering this PR is trying to add, and the contributor included useful before/after gateway logs, but the delayed connect callback needs lifecycle ownership before merge because it can reopen the gateway after an explicit disconnect or abort.

## What This Changes

The PR passes the Discord account ID into gateway plugin construction, adds a deterministic 0-249 ms per-account delay before the first non-resume Discord gateway connect, and adds hash-range tests.

## Best Possible Solution

Land the startup jitter only after the pending delay is cancelled or generation-guarded across disconnects and superseding connects, with a focused fake-timer regression test.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the current-main source path shows every account starts without account-based jitter, and the PR body includes before/after gateway logs, but I did not run a live Discord gateway restart in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet; a first-connect jitter is a reasonable narrow fix, but the delayed callback must be cancelled or guarded so it cannot undo disconnect/abort lifecycle state.
