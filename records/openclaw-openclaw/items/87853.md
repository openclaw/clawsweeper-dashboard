## Summary

Keep open: the jitter direction is plausible and the PR body includes real gateway-log proof, but the new startup timer is not lifecycle-owned, so a disconnect or startup abort during the jitter window can still call `super.connect()` and reopen Discord after shutdown.

## What This Changes

The PR threads Discord account IDs into gateway plugin creation, adds deterministic per-account startup jitter before the first non-resume gateway connect, and tests the jitter hash.

## Best Possible Solution

Own the startup jitter timer inside `OpenClawGatewayPlugin`: store and cancel it on disconnect or superseding connect, avoid re-enabling reconnect after cancellation, and cover first-connect plus disconnect-before-fire behavior.

## Reproduction Assessment

Yes, source-level: create a plugin with nonzero startup jitter, call `connect(false)`, call `disconnect()` before the timeout fires, then the pending callback still invokes `super.connect()`. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as submitted. The one-shot startup jitter is a reasonable fix shape, but it must be cancellable through the gateway lifecycle before it is safe to merge.
