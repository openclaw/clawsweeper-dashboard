## Summary

Keep this open. Current main still has no diagnostic, retirement path, or maintained script hardening for a host-local `~/.openclaw/bin/ensure-whatsapp.sh` crontab entry; the existing cron doctor path only inspects OpenClaw's persisted cron store, and the systemd user-bus fallback only helps repo-managed service operations. The related open PR explicitly says it does not address this legacy WhatsApp health-check path.

## What This Changes

Request detection, retirement, or hardening for legacy user-crontab `ensure-whatsapp.sh` checks that can falsely log gateway inactivity when `systemctl --user` cannot reach the user bus.

## Best Possible Solution

Add a targeted, read-only Linux doctor or health diagnostic that detects user crontab entries invoking `~/.openclaw/bin/ensure-whatsapp.sh` and warns that this legacy check can misreport gateway health when the systemd user bus is unavailable. If a maintained installer still creates the script, retire that installer path or harden the script to log the bus-environment failure instead of `Gateway inactive`; otherwise keep the fix to detection and guidance.

## Reproduction Assessment

Yes for the underlying failure mode: the issue provides an env-isolated `systemctl --user is-active openclaw-gateway.service` command that reproduces `Failed to connect to bus: No medium found`. Full end-to-end reproduction requires an affected host with the legacy crontab and local script, which current main does not contain.

## Solution Assessment

Yes, a read-only doctor warning is the narrowest maintainable first fix because it addresses host-local drift without removing user crontab entries automatically. Retiring or hardening the script should only follow if source or installer history proves OpenClaw still ships that script.
