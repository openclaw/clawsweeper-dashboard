## Summary

Keep open: current main has some agent/tool failure wording, but the reported queue-message failure path still collapses to a boolean plus debug log, and the user-visible tool/agent warnings do not carry the requested structured session/run/health context.

## What This Changes

Request structured, actionable gateway agent/tool failure notifications that include subsystem, reason, session/run identifiers, fatality when known, and gateway health state.

## Best Possible Solution

Add a typed failure outcome for queue/tool/agent failure paths, format a redacted operator alert from that outcome, and keep gateway health reporting explicit rather than implied by agent/tool failures.

Yes, by source inspection: calling `queueEmbeddedPiMessage` for a missing, non-streaming, or compacting run reaches a false return with only a debug reason. A focused regression can exercise the no-active-run branch directly.

## Solution Assessment

Yes, the requested direction is the maintainable fix if it preserves the existing boolean facade or migrates callers carefully: carry a structured failure reason from the owner path, then render one redacted operator-facing summary.

## Repair Work Prompt

Implement structured gateway agent/tool failure notification context for https://github.com/openclaw/openclaw/issues/80156. Start with the owner paths: preserve compatibility for existing `queueEmbeddedPiMessage` callers or add a typed companion outcome that reports `ok`, `reason` (`no_active_run`, `not_streaming`, `compacting`), `sessionId`, and any known session/run key. Thread that outcome into the failure/notification layer so no-active-run emits a redacted, actionable operator summary instead of only a generic agents-failed alert. For tool failures, reuse existing sanitized tool error metadata and include tool name, message/code when available, run/session identifiers when already in context, and a conservative fatal/nonfatal classification when existing runtime data supports it. Do not add channel-specific hacks, do not expose raw secrets or unredacted local paths beyond existing policy, and keep gateway health/live state explicit rather than conflating agent/tool failure with gateway outage. Add focused regression tests for the no-active-run queue failure summary/payload plus the affected alert formatter; update docs/changelog only if the user-facing alert text or documented diagnostics change.
