## Summary

Keep open: this is a plausible iMessage access-control hardening, but the latest PR still has a public docs-contract gap and the external real-behavior proof gate is missing. The live protected `maintainer` label also means cleanup automation should not close it.

## What This Changes

The branch changes shared chat sender matching and iMessage ingress/setup/docs/tests so conversation-scoped IDs no longer authorize sender identity while preserving chat targets for group admission and reply context.

## Best Possible Solution

Land a maintainer-approved hardening that keeps command authorization tied to sender handles or sender access groups, keeps chat IDs only for routing/group admission/reply context, updates every public config doc, and includes redacted live/runtime proof.

## Reproduction Assessment

Yes for a source-level path: current main's shared matcher and iMessage target test show `chat_id` can satisfy `allowFrom` when chat context is present. I did not establish a live iMessage reproduction path.

## Solution Assessment

No as submitted. The code direction is narrow, but the public docs need to state the chat-target boundary consistently and the external PR still needs real after-fix behavior proof.
