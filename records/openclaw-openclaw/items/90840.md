## Label Justifications

- `P1`: The issue reports a broken agent/channel workflow where users receive raw child output while the intended parent reply is not delivered.
- `impact:message-loss`: The expected parent/main summary is lost from the chat and replaced by the child raw result.
- `impact:session-state`: The subagent completion event is mis-associated with external delivery instead of remaining an internal parent-session result.
- `impact:security`: Internal worker output that was explicitly not meant for the user can cross the internal/external channel boundary.

## Summary

Keep open: current main and the latest release still contain the direct-message subagent fallback that can publish the child `task_completion.result`, and the issue discussion identifies an unresolved maintainer decision around `message_tool_only` semantics versus a sanitized parent-reply signal.

## What This Changes

Fix subagent `mode="run"` completion routing so child raw output stays internal and the chat user sees only the parent/main user-facing reply.

## Best Possible Solution

Define the `message_tool_only` completion contract for direct-message subagent runs, then update `src/agents/subagent-announce-delivery.ts` so child `task_completion.result` is never published as the user-visible fallback when the parent produced or should own the reply.

## Reproduction Assessment

Yes for a source-level reproduction path: direct-message subagent completions force `message_tool_only`, then fall back to sending the child `task_completion.result` when no messaging-tool evidence exists. I did not run a live QQBot/DM roundtrip, so this is source-reproducible rather than fully reproduced here.

## Solution Assessment

Unclear until maintainers choose the product contract. The narrow leak-stop is plausible, but the full requested parent-summary behavior may require either explicit message-tool delivery by the parent or a new sanitized/public parent-reply signal.
