- `impact:message-loss`: The expected parent summary is not delivered to the chat, while the wrong worker output is delivered instead.

## Summary

Keep open. Current main and the latest release still contain the direct-message subagent completion fallback that can externalize the child `task_completion.result`, and the latest discussion confirms the remaining blocker is a maintainer delivery-contract decision rather than obsolete work.

## What This Changes

The issue asks OpenClaw to stop delivering raw run-mode subagent output to the external chat and instead keep user-visible completion text owned by the parent/main session.

## Best Possible Solution

Establish one canonical requester-owned completion delivery contract, then update subagent announcement fallback and tests so raw child task output never becomes an external direct-message reply when the parent/requester owns visible completion text.

## Reproduction Assessment

Yes, from source and tests, but not from a live QQBot rerun in this review. Current main still has the direct-message fallback that sends child `task_completion.result` when the parent announce path lacks message-tool delivery evidence.

## Solution Assessment

Unclear until maintainers choose the delivery contract. Suppressing raw child fallback is necessary, but the durable fix must decide whether `message_tool_only` means silence without an explicit message call or whether OpenClaw needs a new sanitized/public parent-reply signal.
