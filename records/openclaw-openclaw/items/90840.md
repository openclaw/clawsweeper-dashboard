## Summary

Keep open: current main and v2026.6.8 still contain the direct-message child-text fallback that can publish the subagent `task_completion.result` when the parent announce run has no message-tool delivery evidence. The linked open PR only covers the `sessions_yield` sibling path, while this direct-run QQBot/DM contract still needs a maintainer delivery-semantics decision.

## What This Changes

The issue asks OpenClaw to stop delivering run-mode subagent raw completion text to an external QQBot direct chat when the parent/main session should own the user-facing summary.

## Best Possible Solution

Choose the direct subagent completion contract, then implement a focused fix that prevents raw child fallback when the parent produced text, or add an explicit sanitized public parent-reply signal, with regression tests and real DM proof.

## Reproduction Assessment

Yes, at source level: direct-message subagent completions still enter `message_tool_only` and fall back to `deliverTextCompletionDirect` with the child `task_completion.result` when no message-tool evidence exists. I did not run a live QQBot roundtrip in this read-only review.

## Solution Assessment

No, the current implementation is not the best end state because it preserves a raw child-text fallback that conflicts with the private-final/message-tool-only contract. The narrowest safe direction is a maintainer-approved leak-stop or a structural sanitized public-reply signal, not automatic parent final-text publishing.
