## Summary

Keep open. Current main and the latest release still contain the direct-message subagent text fallback that can publish the child `task_completion.result`, while the issue discussion shows the remaining blocker is a maintainer contract decision rather than an already-merged fix.

## What This Changes

The issue requests that `sessions_spawn` subagent run completions be returned internally to the parent/main session so the chat user receives only the parent summary, not raw worker output.

## Best Possible Solution

Make the parent/requester completion handoff the authoritative user-visible path, then remove or gate raw child text fallback according to a maintainer-approved `message_tool_only` contract with focused transport proof.

## Reproduction Assessment

Yes, at source level. Current main still has the `message_tool_only` direct-completion branch that can fall back to sending the child `task_completion.result`, and adjacent tests assert that raw-child delivery path; I did not run a fresh live QQBot reproduction.

## Solution Assessment

Unclear until maintainers choose the contract. A narrow leak-stop is plausible, but the complete fix needs a decision on whether direct-message subagent completions should be silent without an explicit message-tool send or use a structural sanitized/public parent-reply signal.
