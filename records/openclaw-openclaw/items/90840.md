## Label Justifications

- `P1`: The issue reports a real user-facing agent/channel regression where the wrong completion text reaches chat and the intended parent reply is not delivered.
- `impact:message-loss`: The visible channel can receive raw child output instead of the parent-owned reply, which is a misdelivered and lost-message workflow.

## Summary

Keep open: current main and the latest release still contain the direct-message subagent text fallback that can send the child `task_completion.result` externally when the parent announce run has no message-tool delivery evidence, and the discussion shows the remaining fix needs a maintainer delivery-contract decision.

## What This Changes

The issue asks OpenClaw to stop delivering raw `sessions_spawn` run-mode subagent completion text to the original chat user and instead preserve parent-owned user-facing delivery.

## Best Possible Solution

Maintainers should choose the `message_tool_only` direct-subagent completion contract, then land either a narrow leak-stop gate or an explicit sanitized/public parent-reply signal with live direct-chat proof.

## Reproduction Assessment

Yes at source level: current main still routes direct-message subagent completions through `deliverTextCompletionDirect` when message-tool evidence is missing, and the issue includes matching session trace evidence. I did not run a live QQBot/direct-chat roundtrip in this read-only review.

## Solution Assessment

Unclear until maintainers pick the contract: a parent-final gate is a plausible leak stop, but not a complete parent-summary delivery fix. The safer end state is explicit message-tool delivery or a sanctioned sanitized public parent-reply signal rather than auto-publishing private final text.
