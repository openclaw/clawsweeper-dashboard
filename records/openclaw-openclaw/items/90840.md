## Summary

Keep open: current main and v2026.6.8 still contain the direct-message child-text fallback that can publish a subagent `task_completion.result` when the announce run has no message-tool delivery evidence. The linked open PR only covers the `sessions_yield` sibling path, while this direct-run QQBot/DM contract still needs a maintainer delivery-semantics decision.

## What This Changes

The issue reports that `sessions_spawn` subagent completion can leak the child worker’s raw result to a direct-chat user instead of delivering only the parent/main reply.

## Best Possible Solution

Choose the direct subagent completion contract, then implement a focused leak-stop or explicit sanitized public parent-reply signal with regression tests and real direct-message proof.

## Reproduction Assessment

Yes, at source level: current main still routes direct subagent completions through `message_tool_only` and falls back to `deliverTextCompletionDirect` with the child `task_completion.result` when message-tool evidence is missing. I did not run a live QQBot roundtrip in this read-only review.

## Solution Assessment

Unclear until maintainers choose the direct subagent completion contract. The narrow fix is a leak-stop, but complete parent-summary delivery may need an explicit sanitized public-reply signal rather than automatic final-text publishing.
