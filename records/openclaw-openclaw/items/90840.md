## Label Justifications

- `P1`: The issue reports a real user-facing agent/channel regression where the wrong completion text reaches chat and the intended parent reply is not delivered.
- `impact:message-loss`: The visible channel can receive raw child output instead of the parent-owned reply, which is a misdelivered and lost-message workflow.
- `impact:session-state`: The bug sits in subagent-to-parent completion routing and delivery evidence, so parent/session state determines which reply is surfaced.

## Summary

Keep open: current main and v2026.6.6 still contain the direct-message child-text fallback that can send the raw subagent `task_completion.result` when the parent handoff has no message-tool delivery evidence, and the discussion shows the remaining fix is a maintainer delivery-contract decision.

## What This Changes

Ensure `sessions_spawn(mode="run")` subagent completions are returned internally to the parent/requester so the external chat receives the parent-owned user-facing reply rather than raw worker output.

## Best Possible Solution

Maintainers should choose the `message_tool_only` contract first, then implement either a narrow leak-stop gate or an explicit sanitized/public parent-reply signal with live direct-chat proof.

## Reproduction Assessment

Yes at source level: current main still routes direct-message subagent completions through `deliverTextCompletionDirect` when message-tool evidence is missing, and the issue includes matching session trace evidence. I did not run a live QQBot roundtrip in this read-only review.

## Solution Assessment

Unclear until maintainers pick the contract: a gate on parent final text is a plausible leak stop, but it is not a complete parent-summary delivery fix. The safer end state is explicit message-tool delivery or a sanctioned public parent-reply signal rather than auto-publishing private final text.
