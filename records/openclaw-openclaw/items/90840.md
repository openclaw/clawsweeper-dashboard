## Label Justifications

- `P1`: The report describes a real channel workflow where the user receives the wrong raw worker output and the intended parent summary is not delivered.
- `impact:message-loss`: The parent/main summary is suppressed while a raw child result is delivered to the chat instead.
- `impact:session-state`: The subagent completion handoff and `delivery-mirror` transcript state can reflect the child result as the visible requester message.

## Summary

Keep open. Current main still has a direct-message subagent completion fallback that can send the child result text when the requester/parent completion handoff lacks message-tool delivery evidence, while the docs say child output should stay internal and the requester agent owns visible replies.

## What This Changes

The issue reports that `sessions_spawn` with `mode="run"` sends raw child output to a QQBot direct chat instead of only delivering the parent/main summary.

## Best Possible Solution

Align direct-message subagent completion delivery with the documented parent-ownership contract: do not raw-send child output, and make the requester handoff either deliver the parent summary through the correct channel path or record a retriable delivery failure.

## Reproduction Assessment

Yes from source inspection: a direct-message subagent completion with missing message-tool evidence follows the `deliverTextCompletionDirect` fallback and sends the child result. I did not run a live QQBot reproduction.

## Solution Assessment

No current-main behavior is not the best end state because it trades missing parent delivery for raw child-output leakage. The safer fix should preserve parent-owned delivery and fail or retry when the parent does not produce required visible delivery evidence.
