## Summary

Keep open: the PR targets a real Telegram callback timeout path, but the patch leaves the downstream callback handler acknowledgement in place, so successful early acknowledgements can continue into duplicate Bot API requests, and the external PR only provides synthetic proof for a Telegram-visible behavior.

## What This Changes

The PR adds a Telegram middleware that answers callback queries before `sequentialize` and adds a unit test for that middleware order.

## Best Possible Solution

Use an accepted-update-gated pre-sequentialize callback acknowledgement, record early success, and keep the ordered callback handler as a guarded fallback only when the early acknowledgement did not succeed.

## Reproduction Assessment

Yes, source-reproducible: current main registers the grammY sequentializer before callback handlers and answers ordinary callbacks inside the downstream handler. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Moving only the acknowledgement timing before `sequentialize` is the right layer, but this patch should guard or remove the later handler acknowledgement after early success and add live Telegram proof.
