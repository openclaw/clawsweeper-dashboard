Keep this PR open: the patch is a useful bounded fix and I found no concrete code-level regression, but it is not merge-ready because the only after-fix evidence is mocked/unit-test proof while Telegram callback behavior needs redacted live Telegram proof.

## What This Changes

The PR adds pre-sequentialize Telegram middleware that answers callback queries early, records successful acknowledgements on the context, skips the downstream duplicate answer, and adds a middleware-order unit test.

## Best Possible Solution

Keep the early acknowledgement before sequentialize, preserve the downstream fallback guard, and merge only after redacted live Telegram proof confirms the spinner clears during a busy lane.

## Reproduction Assessment

Yes, at source level: current main answers callback queries inside the handler after grammY sequentialize, and the dependency source shows sequentialize waits before calling next(). A live Telegram reproduction was not run, so native spinner behavior still needs real proof.

## Solution Assessment

Yes, the proposed shape is the narrow maintainable fix: answer the callback before lane blocking while keeping callback processing sequentialized and preserving the handler fallback if the early answer fails. Changing the sequential key to let callbacks bypass the lane would be riskier because callback processing can still mutate Telegram/OpenClaw state.
