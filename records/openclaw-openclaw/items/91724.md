## Summary

This PR is still necessary because current main still misses provider-scoped runtime policy lookup when the caller omits provider but supplies a provider-qualified model id. The patch looks correct and proof is sufficient, but the restored precedence is compatibility-sensitive and should get maintainer acceptance before merge.

## What This Changes

The PR infers a provider from provider-qualified model ids when provider is empty, then uses it for model/provider runtime policy lookup and CLI runtime alias regression coverage.

## Best Possible Solution

Land the narrow resolver fix after maintainers accept the compatibility-sensitive runtime precedence change, keeping explicit provider precedence and the new regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main passes an empty provider into provider config/model lookup, and the PR body reports the new regression cases failed before the patch; this read-only review did not execute tests.

## Solution Assessment

Yes. The central runtime-policy resolver is the right layer because harness selection and CLI alias routing already consume it; caller-only fixes would duplicate provider/model runtime policy logic.
