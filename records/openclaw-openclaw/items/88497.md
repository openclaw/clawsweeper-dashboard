## Summary

Keep open: this maintainer-labeled draft targets a real P1 channel startup bug, but the patch is not merge-ready because it can make auto-enabled workspace-origin channel plugins look explicitly trusted for gateway startup.

## What This Changes

The branch treats enabled-only channel config as channel presence, auto-enables matching channel plugins, and uses the effective config as the configured-channel activation source in gateway startup planning with regression coverage.

## Best Possible Solution

Refresh one canonical gateway startup-planning fix so enabled-only global/config external channel config is explicit startup intent while workspace-origin channel plugins remain behind raw explicit plugin activation.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current startup planning still depends on a helper that ignores `{ enabled: true }` alone, and the linked issue's `channels.start` failure matches a channel registry that never loaded the external plugin.

## Solution Assessment

No, not as-is. The generic startup-planner layer is the right place, but the current branch should preserve raw activation trust instead of letting effective auto-enable entries count as explicit activation for workspace channel owners.
