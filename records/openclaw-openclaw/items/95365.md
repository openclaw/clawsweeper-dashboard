## Summary

Keep open. Current main and v2026.6.9 still lack a trusted inbound bare-body contract, but this PR is not merge-ready because it makes nested `__openclaw.inboundDecoration.bareBody` authoritative without an independent host-owned decorated marker and exposes the unsettled contract through memory-host SDK facades.

## What This Changes

Adds trusted inbound bare-body metadata under `__openclaw`, persists and reconciles it on user turns, and updates UI, CLI dedupe, memory export, historical replay, session log consumers, and tests to prefer it over sentinel stripping.

## Best Possible Solution

Land one maintainer-approved, marker-gated trusted inbound-decoration contract, keep sentinel stripping only for legacy unmarked rows, and expose SDK helpers only after the final contract shape is accepted.

Yes, source-level. Current main still builds inbound context as literal sentinel text and user-role consumers strip or compare via `stripInboundMetadata`, so a user-authored sentinel block follows the same path; I did not run a live channel repro in this read-only review.

## Solution Assessment

No. The out-of-band bare body is the right layer, but this PR needs an independent host-owned decorated marker and a maintainer-approved SDK/storage contract before it is the best fix.
