## Summary

Keep open: this is a real source-reproducible content-loss bug, but this PR is not merge-ready because it makes nested `__openclaw` bare-body metadata authoritative without an independent host-owned decorated marker. A related marker-gated PR exists, but its latest review still marks it not cleanly merge-ready, so it is not a safe enough supersession target for closing this PR yet.

## What This Changes

The PR adds a nested `__openclaw.inboundDecoration.bareBody` contract and updates transcript, UI, CLI dedupe, memory export, and historical replay consumers to prefer it over sentinel stripping.

## Best Possible Solution

Land one marker-gated trusted inbound-decoration contract, preserve legacy sentinel stripping only for unmarked historical rows, and fold the useful consumer coverage from this branch into whichever contract maintainers choose.

## Reproduction Assessment

Yes, source-level. Current main still decorates inbound content as literal sentinel text and multiple user-role consumers strip or compare that text heuristically, so a user-authored matching block follows the same path as host metadata.

## Solution Assessment

No, not as written. The out-of-band contract is the right layer, but this PR must gate trusted bare bodies with an explicit host-owned decorated marker before it is the safest implementation.
