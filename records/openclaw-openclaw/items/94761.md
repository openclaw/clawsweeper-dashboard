## Summary

Keep open: this targets a real Feishu p2p delivery regression, but the patch is not merge-ready because it changes the dispatcher `chatId` state field that still feeds raw streaming-card receive IDs and silent-reply conversation classification.

## What This Changes

The PR changes both Feishu reply-dispatcher call sites in `extensions/feishu/src/bot.ts` to pass `feishuTo` instead of raw `ctx.chatId`.

## Best Possible Solution

Keep raw `ctx.chatId` for dispatcher lifecycle state, add a separate normalized delivery target for visible sends including streaming cards, and require live Feishu p2p proof or maintainer override before merge.

## Reproduction Assessment

Yes. The linked bug is source-reproducible on current main: Feishu DMs compute `user:<open_id>` but the dispatcher still receives raw `ctx.chatId` while DM reply metadata can be suppressed; I did not run a live Feishu tenant.

## Solution Assessment

No. Sending p2p visible replies to `user:<open_id>` is the right goal, but replacing `chatId` is the wrong boundary because that field is also raw chat state for streaming and silent-reply context.
