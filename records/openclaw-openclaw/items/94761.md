## Summary

Keep open: the PR targets a real Feishu p2p delivery regression, but replacing the dispatcher `chatId` with `feishuTo` crosses the wrong boundary because that same parameter still drives streaming-card receive IDs and silent-reply conversation classification.

## What This Changes

The PR changes both Feishu bot dispatcher construction paths to pass computed `feishuTo` as the reply dispatcher's `chatId` instead of the raw inbound `ctx.chatId`.

## Best Possible Solution

Keep raw `ctx.chatId` for dispatcher lifecycle and context state, add a separate normalized delivery target for visible Feishu sends including streaming cards, and merge only after redacted live p2p proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, source-reproducible. Current main computes a `user:<open_id>` DM target but still passes the raw p2p `ctx.chatId` into the dispatcher while non-thread DM reply metadata is suppressed.

## Solution Assessment

No. Routing p2p visible delivery to the sender open_id is the right goal, but replacing dispatcher `chatId` is not the best fix because that field also drives streaming-card receive IDs and silent-reply conversation typing.
