## Summary

Keep open: the PR targets a real Feishu p2p message-delivery regression, but this patch changes shared dispatcher state rather than only the delivery target. That can feed prefixed `user:`/`chat:` strings into streaming-card sends and can misclassify group contexts, and the external proof is still test/source-only rather than live Feishu delivery.

## What This Changes

The PR changes two Feishu bot dispatcher construction sites to pass computed `feishuTo` as `chatId` instead of raw `ctx.chatId`.

## Best Possible Solution

Keep `ctx.chatId` as dispatcher and lifecycle state, add a separate normalized delivery target for visible sends including streaming cards, and require redacted live Feishu p2p proof or a maintainer proof override before merge.

## Reproduction Assessment

Yes. The linked bug is source-reproducible: current main computes `user:<open_id>` for Feishu DMs but passes raw `ctx.chatId` to the dispatcher while reply metadata can be suppressed; this read-only review did not run a live Feishu tenant.

## Solution Assessment

No. Routing p2p visible sends to `user:<open_id>` is the right goal, but replacing dispatcher `chatId` is the wrong layer because that field also drives streaming-card receive IDs and conversation typing.
