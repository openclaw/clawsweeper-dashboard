## Summary

Keep open and block merge: the target change is valid, but this implementation overwrites dispatcher chat state that is also used by streaming-card delivery and silent-reply classification, and the contributor proof is source tracing plus local tests only.

## What This Changes

The branch replaces raw Feishu `ctx.chatId` with computed `feishuTo` at the two `createFeishuReplyDispatcher` call sites in `extensions/feishu/src/bot.ts`.

## Best Possible Solution

Keep raw `ctx.chatId` for dispatcher lifecycle state, add a separate normalized delivery target for visible sends including streaming cards, and require redacted live Feishu p2p proof or a maintainer proof override before merge.

## Reproduction Assessment

Yes. The linked failure is source-reproducible: current main computes `user:<open_id>` for Feishu DMs but passes raw `ctx.chatId` to the dispatcher while reply metadata can be suppressed; this review did not run a live Feishu tenant.

## Solution Assessment

No. Routing p2p visible sends to `user:<open_id>` is the right goal, but replacing dispatcher `chatId` is the wrong boundary because that field also drives streaming-card receive IDs and conversation typing.
