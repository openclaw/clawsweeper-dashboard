## Summary

Keep open: the linked bug is real, but this PR changes the dispatcher’s raw `chatId` state to prefixed `feishuTo`, and that same field still drives streaming-card receive IDs plus silent-reply conversation classification. The PR also still has only source tracing and test output, not live Feishu p2p WebSocket delivery proof.

## What This Changes

The PR changes two Feishu bot dispatcher construction sites to pass `feishuTo` instead of `ctx.chatId` as the dispatcher `chatId`.

## Best Possible Solution

Keep raw `ctx.chatId` for dispatcher lifecycle state, add a separate normalized delivery target for visible sends including streaming cards, and require redacted live Feishu p2p proof or a maintainer proof override before merge.

## Reproduction Assessment

Yes. The linked bug is source-reproducible: current main computes `user:<open_id>` for Feishu DMs but passes raw `ctx.chatId` to the dispatcher while reply metadata can be suppressed; this read-only review did not run a live Feishu tenant.

## Solution Assessment

No. Routing p2p visible sends to `user:<open_id>` is the right goal, but replacing dispatcher `chatId` is the wrong layer because that field also drives streaming-card receive IDs and conversation typing.
