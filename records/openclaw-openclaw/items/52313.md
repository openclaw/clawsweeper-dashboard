## Summary

Keep open: the current checkout and latest release still build detached gateway respawns from `process.execPath`, `process.execArgv`, and `process.argv.slice(1)`, so an OpenClaw pnpm versioned `dist/*` realpath can still be reused after a package update. The closed reference PR usefully explored the fix, but it was not merged and no related open issue fully supersedes this stable-entrypoint problem.

## What This Changes

Gateway self-restart and `update.run` should respawn through a stable OpenClaw wrapper or symlinked package entrypoint instead of reusing package-manager-managed realpaths.

## Best Possible Solution

Add a focused respawn entrypoint planner that rewrites OpenClaw package-manager realpaths to stable package wrappers while preserving dev/source entrypoints and the existing update health fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: set `process.argv[1]` to a pnpm versioned OpenClaw `dist/*` realpath and current main passes that same path to `spawn()` from `spawnDetachedGatewayProcess()`; I did not run a live self-update in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to plan respawn arguments at the helper boundary and reuse existing package-entrypoint resolution patterns, rather than replaying the broad closed PR wholesale.
