## Label Justifications

- `P1`: A failed Gateway update restart can leave message-triggered agent workflows offline after package-manager cleanup.
- `impact:crash-loop`: The issue concerns Gateway process availability during restart/update respawn through unstable entrypoint paths.

## Summary

Keep open: current main still respawns update handoffs with `process.execPath` plus raw `process.execArgv` and `process.argv.slice(1)`, so the unstable package-manager realpath problem remains. Recent main work improves related restart failure modes, but it does not add stable wrapper planning for update respawn, and the latest shipped tag still has the same raw argv path.

## What This Changes

Gateway self-update/restart should respawn through a stable OpenClaw wrapper instead of reusing a versioned package-manager realpath from the current process argv.

## Best Possible Solution

Add a focused respawn entrypoint planner in `src/infra/process-respawn.ts` that rewrites package-manager OpenClaw realpaths to a stable wrapper while preserving dev/source, Bun, supervisor, and update health-fallback behavior.

## Reproduction Assessment

Yes, source-level. With no supervisor, `respawnGatewayProcessForUpdate()` calls the detached helper, which sends `process.execArgv + process.argv.slice(1)` unchanged to `process.execPath`; the focused test still asserts raw dist-path passthrough.

## Solution Assessment

Yes. The best fix is a narrow helper-level respawn entrypoint planner; the broad closed PR contained useful context but too much unrelated churn, and the open service-path PR does not cover runtime update respawn.
