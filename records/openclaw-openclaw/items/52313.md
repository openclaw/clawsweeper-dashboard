## Label Justifications

- `P1`: A failed Gateway update restart can leave message-triggered agent workflows offline after package-manager cleanup.
- `impact:crash-loop`: The issue concerns Gateway process availability during restart/update respawn through unstable entrypoint paths.

## Summary

Keep open: current main and the latest release still send update respawns through the raw `process.execPath` plus `process.execArgv`/`process.argv.slice(1)` path, so the unstable pnpm realpath failure remains. Related fixes improved lifecycle loading, supervisor detection, and service install paths, but none replace the unmanaged update respawn entrypoint with a stable wrapper.

## What This Changes

The issue asks for gateway restart/update respawn to rewrite package-manager-managed OpenClaw realpaths to a stable wrapper entrypoint instead of reusing the current argv path.

## Best Possible Solution

Add a focused respawn entrypoint planner in `src/infra/process-respawn.ts` that rewrites package-manager OpenClaw realpaths to a stable wrapper while preserving dev/source, Bun, supervisor, and update health-fallback behavior.

## Reproduction Assessment

Yes, source-level. With no supervisor, `respawnGatewayProcessForUpdate()` calls the detached helper, which sends `process.execArgv + process.argv.slice(1)` unchanged to `process.execPath`; the focused tests still assert raw dist-path passthrough.

## Solution Assessment

Yes. The best fix is a narrow helper-level respawn entrypoint planner; the broad closed PR contained useful context but too much unrelated churn, and the open service-path PR does not cover runtime update respawn.
