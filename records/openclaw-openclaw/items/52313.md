## Summary

Keep open: current main and the latest release still use the current argv entrypoint for unmanaged update respawn, while the related stable-wrapper fix PRs are closed unmerged and no open PR owns the remaining work.

## What This Changes

The issue reports that gateway self-restart/update can respawn through a pnpm versioned realpath that disappears during update, preventing the gateway from coming back.

## Best Possible Solution

Land a focused core fix that normalizes only the detached update respawn entrypoint to a stable OpenClaw package wrapper or stable dist path, preserves dev/source entrypoints, and adds regression tests for pnpm realpaths and passthrough cases.

## Reproduction Assessment

Yes at source level: unmanaged `update.run` still reaches detached respawn with the current argv entrypoint unchanged. I did not run a live pnpm self-update in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to resolve a stable entrypoint at the update respawn boundary, preferably reusing the existing service-install stable-path logic where practical. A channel-level retry, new config flag, or broad update-policy change would be a worse fit for this specific bug.
