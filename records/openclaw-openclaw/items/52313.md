- `impact:crash-loop`: The report is about process-level availability when the Gateway shuts down and cannot reliably come back.

## Summary

Keep open: current main has narrowed the original restart blast radius, but the update respawn helper still constructs detached children from `process.execPath`, `process.execArgv`, and `process.argv.slice(1)` with no stable OpenClaw entrypoint planning. The closed stable-wrapper PR remains unmerged, while later managed-update work reduces some package-update paths without adding the requested respawn rewrite or regression coverage.

## What This Changes

The issue asks for Gateway restart/update respawn to use a stable OpenClaw wrapper or package-root entrypoint instead of reusing package-manager versioned realpaths.

## Best Possible Solution

Add a focused respawn entrypoint planner for update/fresh-process handoff that rewrites OpenClaw package-manager realpaths to stable package wrappers while preserving source/dev entrypoints and the managed-service handoff contract.

## Reproduction Assessment

Yes for source-level reproduction, not live reproduction: current main still passes `process.argv[1]` unchanged into detached update respawn, and the focused test asserts that passthrough behavior. A live pnpm self-update was not run in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is a helper-level respawn argument planner that reuses the existing stable entrypoint concepts instead of changing broader Gateway restart policy.
