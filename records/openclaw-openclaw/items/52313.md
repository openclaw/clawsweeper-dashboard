## Summary

Keep this issue open: current main has improved ordinary unmanaged restarts and service install paths, but the update-specific detached respawn path still reuses the current process argv and can still target an unstable package-manager realpath after an update.

## What This Changes

The issue reports that gateway self-restart/update can fail after package-manager updates because respawn may reuse a version-pinned package path that no longer exists.

## Best Possible Solution

Add a focused stable respawn-args resolver for update detached respawn, reusing existing package-root or service-entrypoint helpers where appropriate, with regression tests for pnpm versioned paths, package-root fallback, and dev/source passthrough.

## Reproduction Assessment

Yes at source level: current main still builds update respawn child args from the current process argv, and the update-run caller still reaches that helper. I did not run a live pnpm self-update reproduction in this read-only review.

## Solution Assessment

No current main implementation is not the best solution yet; service-install path hardening is useful but does not affect the update respawn helper. The best fix is a narrow stable-entrypoint resolver on the detached update respawn path with focused tests.
