## Label Justifications

- `P1`: A failed Gateway update restart can leave message-triggered agent workflows offline or stuck on recovery behavior after package-manager cleanup.
- `impact:crash-loop`: The issue is about Gateway process availability when restart/update shuts down or respawns through an unstable path.

## Summary

Keep open: current main and v2026.5.19 reduce the dead-gateway blast radius with update health checks and fallback, but update respawn still launches `process.execPath` with raw `process.execArgv + process.argv.slice(1)` and has no stable OpenClaw wrapper rewrite. The earlier stable-wrapper work at https://github.com/openclaw/openclaw/pull/52319 closed unmerged, and the open adjacent service-path PR at https://github.com/openclaw/openclaw/pull/66502 does not change the update respawn helper.

## What This Changes

The issue asks OpenClaw to make Gateway self-update respawn choose a stable OpenClaw package wrapper instead of reusing package-manager versioned argv paths.

## Best Possible Solution

Add a focused respawn entrypoint planner for update and fresh-process handoff that rewrites OpenClaw package-manager realpaths to stable wrappers while preserving dev/source, Bun, managed-service, and health-fallback behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.5.19 still pass `process.argv[1]` unchanged into detached update respawn, and the focused test asserts raw passthrough behavior. No live pnpm self-update was run in this read-only review.

## Solution Assessment

Yes, with a narrower shape than the closed broad PR: the maintainable fix is helper-level respawn entrypoint planning that reuses or mirrors existing stable service-entrypoint logic while keeping current update health fallback intact.
