## Summary

Keep open: current main still respawns unmanaged restart/update children with `process.execPath` plus the current `process.execArgv` and `process.argv.slice(1)`, so a pnpm versioned package realpath can still be reused after package replacement. The closed related PR #52319 is useful reference material but was not merged, and #65668 tracks a different supervisor/orphaning failure mode.

## What This Changes

Gateway restart/update should respawn through a stable OpenClaw wrapper or package entrypoint instead of reusing a package-manager-versioned current argv path.

## Best Possible Solution

Add a focused respawn-entrypoint planner in `src/infra/process-respawn.ts` that rewrites installed OpenClaw package realpaths to stable wrapper/package entrypoints while preserving dev/source and Bun entrypoints, with regression coverage for pnpm realpath cases.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to set `process.argv[1]` to a pnpm versioned OpenClaw `dist/*` realpath and observe that current main passes that same path into `spawn()` from `spawnDetachedGatewayProcess()`.

## Solution Assessment

Yes. The narrow maintainable fix is to plan respawn argv in `src/infra/process-respawn.ts`, reusing or aligning with existing stable-entrypoint helpers instead of replaying the broad closed PR wholesale.
