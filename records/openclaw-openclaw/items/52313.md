## Label Justifications

- `impact:crash-loop`: The issue is about process-level restart/update availability where the gateway may not come back after shutdown.

## Summary

Keep open: current main and v2026.5.18 still let the update-mode detached respawn reuse `process.execArgv + process.argv.slice(1)`, so a package-manager versioned OpenClaw `dist/*` realpath can still be spawned after package replacement. Ordinary unmanaged restarts were improved by https://github.com/openclaw/openclaw/pull/83138, but the `update.run` child respawn path remains unresolved; https://github.com/openclaw/openclaw/pull/52319 is useful prior work, not a merged fix.

## What This Changes

Gateway restart/update should respawn through a stable OpenClaw wrapper instead of reusing package-manager versioned realpaths after self-update.

## Best Possible Solution

Implement a small respawn entrypoint planner for update respawns that rewrites OpenClaw package-manager realpaths to a stable wrapper or package-root entrypoint while preserving dev/source/Bun paths and the existing health fallback.

## Reproduction Assessment

Yes, at source level. With no supervisor and `process.argv[1]` set to a package-manager versioned OpenClaw `dist/*` path, `respawnGatewayProcessForUpdate()` passes that path unchanged to detached `spawn()`.

## Solution Assessment

Yes. The narrow maintainable fix is to plan the update respawn argv at `process-respawn.ts`, reusing the stable entrypoint rules already proven for service configs instead of changing broad restart policy.
