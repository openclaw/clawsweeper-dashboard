## Summary

Keep this open. Current main still respawns unmanaged gateway restarts with `process.execPath` plus `process.execArgv` and `process.argv.slice(1)`, so a pnpm versioned `dist/*` realpath can still be reused after package replacement. The update health fallback reduces the dead-gateway outcome for `update.run`, but it does not implement stable-entrypoint planning; the directly related PR #52319 was closed unmerged, and adjacent issue #65668 covers a different supervisor/orphaning failure mode.

## What This Changes

The issue asks for gateway restart and `update.run` respawn to rewrite unstable package-manager realpaths to a stable OpenClaw wrapper or symlinked entrypoint while preserving dev/source launches.

## Best Possible Solution

Implement a focused respawn-argument planner owned by `src/infra/process-respawn.ts`: detect OpenClaw package installs launched from pnpm versioned realpaths, prefer a stable package entrypoint or symlinked wrapper when available, keep source/dev entrypoints unchanged, and use the same planner for normal SIGUSR1 restarts and `update.run` respawns.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to set `process.argv[1]` to a pnpm versioned OpenClaw `dist/*` realpath, call `restartGatewayProcessWithFreshPid()` or `respawnGatewayProcessForUpdate()`, and observe that current main passes that same unstable path to `spawn()`.

## Solution Assessment

Yes, the requested direction is the right shape, but it should be narrower than the closed XL PR. Reuse existing gateway entrypoint helpers where possible and avoid changing unrelated supervisor policy or broad `scripts/run-node.mjs` behavior unless a focused source-checkout test requires it.
