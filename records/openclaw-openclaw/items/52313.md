## Summary

Keep open: current main still builds detached gateway respawns from `process.execPath`, `process.execArgv`, and `process.argv.slice(1)`, so a pnpm versioned package realpath can still be reused. The newer update health fallback reduces one outage mode, but it does not rewrite the unstable entrypoint and normal restarts still exit immediately after spawning.

## What This Changes

The issue requests gateway restart/update respawn through a stable OpenClaw wrapper or package entrypoint instead of reusing unstable package-manager realpaths.

## Best Possible Solution

Add a focused respawn entrypoint planner that rewrites OpenClaw package-manager realpaths to stable package wrappers while preserving dev/source entrypoints and existing update health fallback behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: set `process.argv[1]` to a pnpm versioned OpenClaw `dist/*` realpath and current main passes that same path to `spawn()` from `spawnDetachedGatewayProcess()`.

## Solution Assessment

Yes. The narrow maintainable fix is to plan respawn arguments at the helper boundary and reuse existing package-entrypoint resolution patterns, rather than replaying the broad closed PR wholesale.
