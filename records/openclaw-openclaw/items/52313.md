## Summary

Keep open: current main and the latest release still build detached Gateway respawns from `process.execPath`, `process.execArgv`, and `process.argv.slice(1)`, so a pnpm versioned OpenClaw `dist/*` realpath can still be reused after an update. The prior fix PR at https://github.com/openclaw/openclaw/pull/52319 was closed unmerged, and the closed update module-graph issue at https://github.com/openclaw/openclaw/issues/78820 does not supersede this stable-entrypoint problem.

## What This Changes

The issue asks Gateway restart/update respawn to rewrite package-manager versioned OpenClaw entrypoints to a stable wrapper or symlinked CLI path while preserving dev/source entrypoints.

## Best Possible Solution

Add a focused respawn entrypoint planner that rewrites OpenClaw package-manager realpaths to stable package wrappers while preserving dev/source entrypoints and existing update health fallback behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: set `process.argv[1]` to a pnpm versioned OpenClaw `dist/*` realpath and current main passes that same path to detached `spawn()` from `spawnDetachedGatewayProcess()`.

## Solution Assessment

Yes. The narrow maintainable fix is to plan respawn arguments at the helper boundary and reuse existing stable package-entrypoint resolution patterns, instead of changing broad Gateway restart policy.
