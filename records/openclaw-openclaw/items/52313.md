## Summary

Close as implemented: later merged restart/update changes reroute package-managed `update.run` through a managed-service handoff or safe shell command and make ordinary unmanaged restarts stay in-process, so current `main` no longer takes the reported stale package-manager argv respawn path. The relevant changes shipped in `v2026.5.18`.

## What This Changes

The issue asks for Gateway restart and self-update respawns to avoid unstable package-manager realpaths and use stable CLI or service entrypoints after package updates.

## Best Possible Solution

Keep the shipped managed-service update handoff plus in-process unmanaged restart behavior; continue tracking newer macOS handoff failures separately at https://github.com/openclaw/openclaw/issues/83808.

## Reproduction Assessment

No current-main failure path remains for the reported package-managed restart/update case: source inspection shows global `update.run` uses managed handoff or safe command, and ordinary unmanaged restarts no longer detached-spawn. The original stale-argv behavior is covered by the prior source discussion and later regression tests.

## Solution Assessment

Yes. The shipped design is more robust than only rewriting `argv[1]`: package-managed updates avoid live package replacement inside the Gateway, and ordinary unmanaged restarts keep the same PID instead of relying on a fragile detached child.
