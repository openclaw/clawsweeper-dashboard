## Summary

## What This Changes

The PR adds an optional env parameter to `isExecutableFile`, threads caller env through shared executable resolution, updates node-host `system.which` PATHEXT lookup, and adds Windows-mocked PATHEXT regression tests.

## Best Possible Solution

Support caller PATHEXT with the same casing tolerance used by nearby Windows env lookups, then land the focused fix with the existing helper proof.

## Reproduction Assessment

Yes, at source level: current main expands Windows candidates from caller env but validates executable extensions against process PATHEXT. The PR body also has copied Windows terminal output for the exported helper path.

## Solution Assessment

No, not yet: threading caller env is the right shape, but the implementation should use casing-tolerant PATHEXT lookup so Windows-style `PathExt` env objects work too.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Honor PathExt casing in caller env:** `src/infra/executable-path.ts:85`
  - body: Windows env keys are already treated with casing tolerance in this file (`Path`) and in the old node-host fallback (`PathExt`), but the newly threaded caller PATHEXT path only checks `PATHEXT`/`Pathext`. A caller env captured with `PathExt` will still fall back to `process.env` and can miss the custom extension this PR is meant to honor; include `PathExt` or use a case-insensitive env lookup for caller PATHEXT in both extension helpers.
  - confidence: 0.78
- **[P3] Isolate PATHEXT tests from process env:** `src/infra/executable-path.test.ts:232`
  - body: The positive caller-env tests do not force `process.env.PATHEXT` to an unrelated value. If a Windows developer or CI image already includes `.PS1`, the pre-fix fallback can satisfy these assertions, so set process PATHEXT around the positive cases the same way the fallback tests do.
  - confidence: 0.83
