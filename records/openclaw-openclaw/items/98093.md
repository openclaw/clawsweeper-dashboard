## What This Changes

The PR threads caller-provided PATHEXT through Windows executable checks in `src/infra/executable-path.ts`, updates node-host `system.which`, and adds mocked Windows PATHEXT regression tests.

## Best Possible Solution

Merge the focused env-propagation fix after redacted real Windows behavior proof is added and the positive PATHEXT tests isolate `process.env.PATHEXT`.

## Reproduction Assessment

Yes, at source level: current main can build Windows candidates from caller env.PATHEXT but then validate the extension against process PATHEXT. I did not run a live Windows repro in this read-only review.

## Solution Assessment

Yes, pending proof and test hardening: threading caller env into the final executable checks and sibling node-host resolver is the narrow maintainable fix. A broader resolver rewrite or shell lookup would add more behavior surface than this bug needs.
