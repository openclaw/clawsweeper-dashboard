## Summary

Keep open. Current main has an `agents_list` tool, but it is scoped to `sessions_spawn` target discovery and returns id/name/model/runtime metadata; it does not provide the requested sibling roster with `identity.theme` or agent-to-agent availability, and no related open issue fully owns that prompt/context surface.

## What This Changes

The issue requests surfacing configured sibling agents to each session through the system prompt or a discoverable runtime context so agents can identify peers without manual docs.

## Best Possible Solution

Define one policy-filtered agent-roster contract, likely by extending `agents_list` with bounded identity/role metadata and A2A reachability, then document when prompt injection is appropriate versus on-demand discovery.

## Reproduction Assessment

Not applicable as a bug reproduction. This is an additive feature request, and source inspection confirms current main does not expose the requested sibling roster in the prompt or existing discovery tool.

## Solution Assessment

Unclear. A discoverable tool is probably safer than unconditional prompt injection for prompt-cache and privacy reasons, but maintainers need to decide the roster surface, policy filtering, and relationship to https://github.com/openclaw/openclaw/issues/7490 and https://github.com/openclaw/openclaw/issues/59149.
