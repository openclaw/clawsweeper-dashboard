## Summary

Keep this open: current main and the latest release have a bounded `agents_list` discovery tool, but they still do not inject or expose the requested sibling roster with `identity.name`/role and agent-to-agent permission context. The related description-field work is useful, but it only covers part of dynamic discovery and still needs maintainer product/security acceptance.

## What This Changes

Request to add sibling-agent roster awareness to sessions, either in the system prompt or through a discoverable tool, including peer ids, names, roles, and agent-to-agent permission status.

## Best Possible Solution

Define a maintainer-approved, security-aware roster discovery contract, preferably through the existing gated `agents_list`/tool surface before considering any always-on system prompt injection.

## Reproduction Assessment

Yes, at source level. Current main shows `agents_list` is limited to allowed spawn targets and model/runtime metadata, while the prompt only names the tool and does not include a sibling roster or agent-to-agent permission summary.

## Solution Assessment

Unclear. A gated discoverable tool appears safer than unconditional system prompt injection, but maintainers need to choose the exposure boundary before this should become implementation work.
