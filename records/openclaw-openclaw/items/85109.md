## Summary

Keep open: current main still maps Codex app-server `webSearch` items into mirrored transcript/tool entries from the top-level `query` string and records only a generic completed result, while the generated Codex protocol exposes richer `action` metadata that the projector ignores.

## What This Changes

The issue asks OpenClaw to make Codex-native `web_search` session/chat logging truthful when native search reports an empty or unavailable query.

## Best Possible Solution

Normalize Codex app-server `webSearch` query/action metadata into truthful transcript entries, use an explicit unavailable-query sentinel instead of `""`, and add focused regression coverage without changing native search execution.

## Reproduction Assessment

Yes, by source inspection: a `webSearch` item with `query: ""` will be mirrored as `{query:""}` and a status-only result. I did not execute a live Codex run or tests because this review was read-only.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Codex app-server event projector: derive args/result text from available `webSearch.action` metadata and avoid inventing unavailable result content.
