## Best Possible Solution

Land only after runtime proof, ideally a focused oversized-transcript read or micro-benchmark; either document the narrow scope or extend the cache to the sibling index extractor in a deliberate follow-up.

## Reproduction Assessment

## Solution Assessment

This is an acceptable local mitigation, but not clearly the comprehensive best fix because the sibling transcript-index extractor keeps the same per-call regex construction. The maintainable path is to prove the runtime effect and either keep the PR explicitly narrow or handle both extraction surfaces together.
