## Best Possible Solution

## Reproduction Assessment

Yes at source level: current main constructs new `RegExp` objects in the oversized transcript field extractors each time that fallback parses a large line. I did not run performance tests in this read-only review.

## Solution Assessment

No. Caching only `session-utils.fs.ts` is a plausible partial optimization, but the sibling transcript index path still uses the same per-call `RegExp` construction; the best fix should cover both or share one helper.
