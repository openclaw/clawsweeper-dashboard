## Summary

Keep open: this is a narrow, plausible gateway performance PR against a current-main hot path, and I found no blocking correctness or security findings in the changed code. Current main still recompiles the private oversized-transcript field regexes per helper call, while the branch only caches equivalent non-global regexes for that private path.

## What This Changes

The PR adds a private module-level cache for regexes used to extract id, parentId, type, and role from oversized transcript JSONL prefixes.

## Best Possible Solution

Land the narrow cache if maintainers accept the micro-performance benefit, with focused session-utils coverage and changed checks before merge.

## Solution Assessment

Yes; caching private, non-global regexes behind the existing helpers is a narrow maintainable way to avoid repeated construction while preserving the parsing contract. The safer merge path is to keep the cache private to this file and run the existing session-utils tests.
