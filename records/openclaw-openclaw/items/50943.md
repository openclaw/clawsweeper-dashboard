## Summary

Keep this PR open, but do not merge it as written. Current main still does not recover Feishu sender-scoped topic affinity from session keys in the run-context, ACP, and native subagent paths, while the PR diff introduces cross-channel routing regressions and unrelated formatting/UI churn that make it unsafe as a conservative cleanup merge.

## What This Changes

The PR changes Feishu sender-scoped topic/session parsing, run-context thread inference, ACP and native subagent requester routing, Feishu explicit-reply outbound behavior, related tests, and unrelated formatting in Telegram, Discord, sandbox, infra, plugin, and UI files.

## Best Possible Solution

Replace or rework this as a focused Feishu/plugin-owned conversation-routing patch: preserve sender-scoped topic affinity through the existing session-conversation, delivery-target, requester-origin, ACP binding, native subagent, and explicit-reply contracts, and drop unrelated UI/format churn before maintainer review.

## Reproduction Assessment

Yes. A code-level reproduction is available with `agent:main:feishu:group:oc_group_chat:topic:om_topic_root:sender:ou_topic_user`: current main tests show that key remains a plugin-owned conversation with `threadId: undefined`, while run-context and spawn paths only use explicit `threadId` inputs.

## Solution Assessment

No. The PR targets a real routing gap, but the generic session-thread parsing approach cuts across plugin-owned Feishu semantics and introduces regressions in Telegram, Slack, and Feishu fallback behavior; the safer path is a narrower contract-aligned fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve legacy Telegram DM thread ids:** `src/config/sessions/delivery-info.ts:34-36`
  - body: `resolveSessionThreadIdForRouting()` drops every Telegram DM `:thread:` suffix unless the key also contains `:topic:`. Telegram supports DM topics and existing session consumers accept keys like `agent:main:telegram:dm:123:thread:999`, so run-context, ACP, and native subagent routing from those sessions would fall back to the DM root.
  - confidence: 0.86
- **[P2] Use the requester agent store for inferred delivery context:** `src/agents/command/run-context.ts:12-13`
  - body: The new fallback calls `extractDeliveryInfo(trimmedSessionKey)` without selecting the store for the agent encoded in the session key. For named-agent sessions resumed without `to`, this reads the default store and leaves same-turn threaded replies without the restored channel context.
  - confidence: 0.8
- **[P2] Normalize restored channel ids before reusing them:** `src/agents/command/run-context.ts:57-59`
  - body: This assigns stored `deliveryContext.to` directly to `currentChannelId`. Slack stores values such as `slack:C...` or `channel:C...`, while Slack auto-threading compares against the bare parsed channel id, so resumed Slack thread sessions can still miss auto-threading.
  - confidence: 0.78
- **[P2] Keep failed Feishu explicit replies inside the topic:** `extensions/feishu/src/outbound.ts:52-56`
  - body: When an explicit `replyToId` is present, the new reply context forces `replyInThread: false` even if a `threadId` is also available. If Feishu rejects that quoted target as withdrawn or missing, the fallback posts at the chat root instead of failing closed in the topic.
  - confidence: 0.76
