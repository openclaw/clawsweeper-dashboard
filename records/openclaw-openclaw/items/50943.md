## Summary

Keep open: the intended Feishu topic-affinity bug still appears real on current main, but this PR should not merge as-is because it routes through generic session parsing that conflicts with plugin-owned conversation contracts and introduces cross-channel regressions.

## What This Changes

The PR changes Feishu topic/session parsing, run-context thread inference, ACP and native subagent requester routing, Feishu explicit reply handling, regression tests, and unrelated formatting/UI files.

## Best Possible Solution

Replace or rebase this as a focused Feishu routing patch that preserves sender-scoped topic affinity through plugin-owned conversation, delivery-target, and binding contracts while preserving existing Telegram/Slack thread behavior and dropping unrelated churn.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Feishu sender-scoped topic session key carries the topic root only in the session key, while current main's run-context and spawn paths only consume explicit `threadId`. The PR can also be checked against existing Telegram DM, Slack, and Feishu fallback tests that its helper/reply changes would violate.

## Solution Assessment

No. The PR targets the right user problem, but generic thread extraction is not the narrowest maintainable fix because Feishu sender-scoped topic routing is already plugin-owned and existing Telegram, Slack, and Feishu reply fallback contracts need to remain intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve valid Telegram DM thread sessions:** `src/config/sessions/delivery-info.ts:34-36`
  - body: `resolveSessionThreadIdForRouting()` drops every `agent:main:telegram:dm:*:thread:*` key. Current main still treats legacy numeric Telegram DM thread suffixes as valid delivery context, so run-context, ACP, and native subagent routing from those sessions would fall back to the DM root.
  - confidence: 0.86
- **[P2] Use the requester agent store for inferred delivery context:** `src/agents/command/run-context.ts:12-13`
  - body: The new fallback calls `extractDeliveryInfo(trimmedSessionKey)` without selecting the agent store for that session key. Named-agent sessions live under their own agent store, so resuming `agent:susan:...` without `to` can still leave `currentChannelId` empty and miss same-turn threaded delivery.
  - confidence: 0.8
- **[P2] Normalize restored channel ids before reusing them:** `src/agents/command/run-context.ts:57-59`
  - body: This copies stored `deliveryContext.to` directly into `currentChannelId`. Slack stores values such as `channel:C...` or `slack:C...`, while Slack auto-threading compares against the bare channel id, so resumed Slack thread sessions can still miss auto-threading.
  - confidence: 0.78
- **[P2] Keep failed Feishu explicit replies inside the topic:** `extensions/feishu/src/outbound.ts:52-56`
  - body: When an explicit `replyToId` is present, `resolveReplyContext()` forces `replyInThread: false` even if a topic `threadId` is also available. If Feishu rejects that quoted target as withdrawn or missing, the fallback posts at the chat root instead of failing closed inside the topic.
  - confidence: 0.76
