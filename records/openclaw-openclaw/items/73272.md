## Summary

Keep open: current main still treats `[[reply_to_current]]` as a shared delivery directive and the strict per-agent config surface has no opt-out. Related #72627 is broader and still open, so this focused runtime guard remains useful as its own repair target.

## What This Changes

Add a per-agent directive policy to disable runtime processing of `[[reply_to_current]]` for selected agents.

## Best Possible Solution

Add a backward-compatible `agents.list[].directives.replyToCurrent` policy, default it to enabled, and enforce the disabled case at shared directive parsing/reply-normalization boundaries while leaving auto-send suppression to #72627.

## Reproduction Assessment

Yes. A source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, final reply threading, streaming directive accumulation, or outbound payload planning and observe `replyToCurrent` or `replyToId` emitted without any agent policy input.

## Solution Assessment

Yes for the narrow request. A core per-agent runtime guard is the maintainable seam because the directive is parsed in shared runtime delivery code; Feishu-specific rendering and auto-send suppression should remain separate.
