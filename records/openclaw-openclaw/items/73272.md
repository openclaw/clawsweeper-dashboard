## Summary

Keep open. Current main still parses and applies `[[reply_to_current]]` through the shared reply directive/runtime delivery paths with no per-agent policy check, and the strict agent config schema still has no `directives.replyToCurrent` field. The related #72627 remains open but is broader because it also asks for auto-send suppression, so this issue is useful as the focused runtime guard slice.

## What This Changes

The issue requests an opt-in per-agent config directive to prevent accidental `[[reply_to_current]]` tags from being processed by the runtime.

## Best Possible Solution

Add a backward-compatible, opt-in per-agent directive policy for `reply_to_current` at the shared directive parsing and reply-normalization boundaries, apply it consistently to final replies, streaming chunks, embedded Pi handling, and outbound payloads, and document/schema the config while preserving explicit `[[reply_to:<id>]]` and message-tool replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` plus `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe `replyToCurrent` / `replyToId` emitted; separately, `AgentEntrySchema` has no accepted `agents.list[].directives.replyToCurrent` field.

## Solution Assessment

Yes, if scoped narrowly. The requested opt-in runtime guard is the maintainable fix for accidental tag processing; auto-send suppression from #72627 should remain separate unless maintainers intentionally expand the work.
