## Summary

Keep open: current main still treats `[[reply_to_current]]` as a shared runtime reply directive, and the strict per-agent config surface still has no `directives.replyToCurrent` opt-out. The broader #72627 remains open, but this narrower same-author request is the actionable repair thread for the directive-only behavior.

## What This Changes

Add a per-agent configuration option to disable runtime processing of the `[[reply_to_current]]` directive.

## Best Possible Solution

Add a backward-compatible per-agent directive policy that defaults to enabled, resolves through the agent config surface, and enforces disabled `replyToCurrent` handling at shared parsing and reply-normalization boundaries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, reply delivery normalization, streaming directive accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy.

## Solution Assessment

Yes for this directive-only request. A per-agent runtime guard is the narrow maintainable seam because the directive is parsed in shared delivery code; Feishu rendering and broader auto-send suppression belong in separate follow-up design.
