## Summary

Keep open: current main still parses and preserves `[[reply_to_current]]` through shared reply delivery, and the per-agent `directives.replyToCurrent` opt-out requested here is not present. The older combined tracker was closed in favor of this narrower issue, while the auto-send/audience-boundary work is separately tracked at https://github.com/openclaw/openclaw/issues/74928.

## What This Changes

This issue requests a per-agent config directive that prevents runtime processing of `[[reply_to_current]]` for selected agents.

## Best Possible Solution

Add a backward-compatible per-agent directive policy that defaults to enabled, disables only `reply_to_current` parsing/normalization for opted-in agents, documents the config, and leaves auto-send/audience work to https://github.com/openclaw/openclaw/issues/74928.

## Reproduction Assessment

Yes. Source inspection on current main shows assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` through the shared parser, reply delivery, streaming accumulator, and outbound payload planning with no per-agent policy check.

## Solution Assessment

Yes for the narrow directive-only request, subject to maintainer policy on warning/error semantics. A per-agent opt-in guard at shared parsing and reply-normalization boundaries is more maintainable than channel-specific workarounds, while global `messages.visibleReplies` is not equivalent.
