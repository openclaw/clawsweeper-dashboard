## Summary

Keep open. Current main still parses and propagates `[[reply_to_current]]` across shared directive parsing, final reply threading, streaming, embedded Pi, and outbound normalization without any per-agent opt-out, and the strict per-agent config schema has no `directives.replyToCurrent` field. The related #72627 is open but broader because it also asks for auto-send suppression, so this focused runtime guard remains a useful work item.

## What This Changes

The issue requests an opt-in per-agent runtime directive, such as `agents.<id>.directives.replyToCurrent=disabled`, to stop OpenClaw from processing accidental `[[reply_to_current]]` tags for selected agents.

## Best Possible Solution

Add a backward-compatible per-agent directive policy to the runtime/config contract and enforce the disabled `replyToCurrent` behavior at the shared parsing and reply-normalization boundaries, while documenting the config and leaving auto-send suppression to #72627.

## Reproduction Assessment

Yes. A source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, `applyReplyThreading`, streaming accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy; strict agent config parsing also rejects no explicit `directives.replyToCurrent` surface because it is absent.

## Solution Assessment

Yes, if scoped narrowly. An opt-in, per-agent runtime guard is the maintainable fix because it avoids Feishu-specific channel behavior and preserves explicit message-tool replies; auto-reply suppression should stay separate.
