## Summary

Keep open: current main still treats `[[reply_to_current]]` as a supported shared reply directive, and the strict per-agent config surface has no `directives.replyToCurrent` opt-out. The linked broader #72627 is still open, but this narrower request remains a useful focused repair target rather than a closeable duplicate.

## What This Changes

Add a per-agent configuration option to disable runtime processing of the `[[reply_to_current]]` directive.

## Best Possible Solution

Add a backward-compatible per-agent directive policy that defaults to enabled, resolves through the agent config surface, and enforces disabled `replyToCurrent` handling at shared parsing/reply-normalization boundaries.

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, reply delivery normalization, streaming directive accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy.

## Solution Assessment

Yes, scoped narrowly. A per-agent runtime guard is the maintainable core seam because the directive is parsed in shared delivery code; Feishu rendering and broader auto-send suppression should remain separate follow-up work.
