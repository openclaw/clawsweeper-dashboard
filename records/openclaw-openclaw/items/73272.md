## Summary

Keep open: current main still treats `[[reply_to_current]]` as a shared supported reply directive, and the strict per-agent config surface does not expose a `directives.replyToCurrent` opt-out. The broader open #72627 overlaps, but this issue remains the narrower repair target for the directive guard only.

## What This Changes

Request to add per-agent configuration that disables runtime processing of `[[reply_to_current]]` for selected agents.

## Best Possible Solution

Add a backward-compatible per-agent directive policy that defaults to enabled, is represented in config schema/types/docs, and is enforced at shared parsing and reply-normalization boundaries.

## Reproduction Assessment

Yes. A source-level repro is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, reply delivery normalization, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy.

## Solution Assessment

Yes for the narrow request. A per-agent runtime guard is the maintainable core seam because the directive is parsed in shared delivery code; Feishu rendering behavior and broader auto-send suppression should stay separate.

## Repair Work Prompt

Implement a narrow per-agent directive policy for `[[reply_to_current]]`. Add a backward-compatible config surface such as `agents.list[].directives.replyToCurrent: "enabled" | "disabled"` with default enabled, update the strict zod schema, exported config types, generated/help docs as required, and document it in the relevant agent/rich-output docs. Enforce the disabled policy in shared reply directive parsing/normalization paths so model text containing `[[reply_to_current]]` is stripped but does not set `replyToCurrent` or resolve to the current message for that agent; explicit `[[reply_to:<id>]]` and explicit message-tool replies should continue to work. Add focused regression tests around parser/delivery/outbound/streaming paths plus config schema coverage. Do not implement `autoReply: false` or change Feishu/channel-specific quote rendering in this PR. Add a user-facing `CHANGELOG.md` entry because this is a new config feature.
