## Summary

Keep open: current main still treats `[[reply_to_current]]` as a shared supported delivery directive with no per-agent opt-out, and the related #72627 is broader rather than a clean replacement for this focused request.

## What This Changes

Add an opt-in per-agent config flag that prevents runtime processing of `[[reply_to_current]]` while preserving explicit thread replies.

## Best Possible Solution

Add a backward-compatible per-agent directive policy defaulting to enabled, resolve it through the agent config surface, and enforce disabled `replyToCurrent` at shared parsing/reply-normalization boundaries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, final reply threading, streaming directive accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy check.

## Solution Assessment

Yes. A per-agent runtime guard is the narrow maintainable seam because the directive is parsed in shared delivery code; Feishu-specific rendering and broader auto-send suppression should stay separate.
