## Summary

Keep open: current main still treats `[[reply_to_current]]` as a supported shared reply directive, and the strict per-agent config surface has no `directives.replyToCurrent` opt-out. The broader #72627 remains open, but this narrower report is still a useful focused repair target rather than a closeable duplicate.

## What This Changes

Request a per-agent config directive that disables runtime processing of `[[reply_to_current]]` so selected agents must use explicit message-thread reply tooling.

## Best Possible Solution

Add a backward-compatible per-agent directive policy that defaults to enabled, resolves through the agent config surface, and enforces disabled `replyToCurrent` handling at shared parsing, streaming, reply-normalization, and outbound payload boundaries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, streaming directive accumulation, reply delivery normalization, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy.

## Solution Assessment

Yes, scoped narrowly. A per-agent runtime guard is the maintainable core seam because the directive is parsed in shared delivery code; Feishu rendering behavior and broader auto-send suppression should remain separate work.

## Repair Work Prompt

Implement the narrow directive-disable feature for this issue. Add an opt-in per-agent config surface such as `agents.list[].directives.replyToCurrent: "disabled"` or an equivalent schema-reviewed shape that defaults to current behavior. Resolve the policy for the active/target agent and enforce it wherever assistant-visible text can become delivery metadata: shared inline parsing, reply delivery normalization, streaming directive accumulation, outbound payload planning, and embedded PI payload paths. When disabled, strip `[[reply_to_current]]` from visible text but do not set `replyToCurrent` or infer `replyToId` from the current message; explicit `[[reply_to:<id>]]` and message-tool replies must continue to work. Keep the PR scoped to this directive policy and do not broaden it into `messages.visibleReplies`, auto-send suppression, Feishu-specific rendering, or channel-specific transport changes. Update config schema/types/help/docs and add a user-facing changelog entry.
