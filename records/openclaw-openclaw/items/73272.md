## Summary

Keep open. Current main still treats `[[reply_to_current]]` as a normal reply directive in shared parsing, final reply threading, streaming, embedded Pi, and outbound normalization, and the strict per-agent config schema has no `directives.replyToCurrent` policy. The related #72627 remains open but is broader because it also asks for auto-send suppression, so this focused runtime guard is still a useful narrow work item.

## What This Changes

The issue requests an opt-in per-agent config directive that prevents runtime processing of model-emitted `[[reply_to_current]]` tags.

## Best Possible Solution

Add a backward-compatible per-agent directive policy to the config contract, resolve it through agent config, and enforce disabled `replyToCurrent` behavior at shared parsing and delivery-normalization boundaries while preserving explicit `[[reply_to:<id>]]` and message-tool replies. Document the new setting and keep auto-send suppression as separate follow-up scope.

## Reproduction Assessment

Yes. A source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, `parseReplyDirectives`, streaming accumulation, final reply threading, or outbound payload planning and observe `replyToCurrent` emitted; strict per-agent config also lacks any accepted `directives.replyToCurrent` field.

## Solution Assessment

Yes, scoped narrowly. A per-agent opt-in runtime guard is the maintainable fix because it avoids a Feishu-specific workaround, preserves current defaults, and leaves explicit message-tool threading intact.
