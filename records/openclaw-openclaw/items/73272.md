## Summary

Keep open: current main still treats `[[reply_to_current]]` as a shared runtime reply directive, and the strict per-agent config surface has no `directives.replyToCurrent` opt-out. The broader #72627 remains open, but this narrower same-author request is the cleaner directive-only repair path.

## What This Changes

Request adds a per-agent config directive such as `agents.list[].directives.replyToCurrent: "disabled"` to stop the runtime from processing `[[reply_to_current]]` for selected agents.

## Best Possible Solution

Expose a typed per-agent directive policy that defaults to enabled, then enforce disabled `reply_to_current` handling at shared parsing and delivery boundaries with focused config, runtime, docs, and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, reply delivery normalization, streaming accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy check.

## Solution Assessment

Yes for the directive-only request. A per-agent runtime guard is the narrow maintainable fix because the tag is parsed in shared delivery code; auto-send suppression from #72627 should be handled as separate product work.
