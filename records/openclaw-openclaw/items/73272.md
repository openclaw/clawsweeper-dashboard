## Summary

Keep open: current main still parses `[[reply_to_current]]` as a shared runtime directive and the strict per-agent config surface has no `directives.replyToCurrent` opt-out. The broader #72627 remains open, but this narrower same-author issue is the better bounded implementation thread for directive-only behavior.

## What This Changes

Add a per-agent config directive to disable runtime processing of the `[[reply_to_current]]` reply tag for selected agents.

## Best Possible Solution

Add a backward-compatible per-agent directive policy that defaults to enabled, resolves through the agent config surface, and suppresses only `reply_to_current` parsing/metadata when disabled while preserving explicit reply IDs and message-tool replies.

## Reproduction Assessment

Yes. Source inspection on current main shows assistant text containing `[[reply_to_current]]` flows through parser, streaming, reply normalization, and outbound planning into `replyToCurrent` metadata with no per-agent policy check.

## Solution Assessment

Yes for the directive-only request. A per-agent runtime guard at shared directive parsing/normalization boundaries is the narrow maintainable path; broader auto-send suppression should remain with #72627.
