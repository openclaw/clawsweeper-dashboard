## Summary

Keep open. Current main still has no per-agent policy that disables `[[reply_to_current]]`; the shared parser, final reply threading, streaming, embedded Pi, and outbound normalization paths still propagate the marker, and the strict agent schema has no `directives.replyToCurrent` field. The related #72627 remains open and broader, so this focused runtime guard is still useful rather than terminal cleanup.

## What This Changes

Add an opt-in per-agent config directive that prevents runtime processing of model-emitted `[[reply_to_current]]` while preserving explicit message-tool replies.

## Best Possible Solution

Add a backward-compatible, opt-in per-agent directive policy at the shared parsing and reply-normalization boundaries so `reply_to_current` is stripped and ignored for configured agents, while explicit `reply_to:<id>` and message-tool replies continue to work. Keep the implementation generic and leave auto-send suppression to a separate product decision.

## Reproduction Assessment

Yes. A source-level reproduction is to pass assistant text containing `[[reply_to_current]]` plus a `currentMessageId` through `parseInlineDirectives`, `applyReplyThreading`, streaming, or outbound normalization and observe `replyToCurrent` or `replyToId` emitted; the strict `AgentEntrySchema` has no accepted `directives.replyToCurrent` field.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix if kept opt-in and channel-agnostic. A Feishu-specific workaround or prompt-only rule would not physically prevent accidental runtime tag processing.
