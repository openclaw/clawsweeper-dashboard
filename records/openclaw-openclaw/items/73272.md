## Summary

Keep open: current main still treats `[[reply_to_current]]` as a shared delivery directive with no per-agent opt-out, and the strict per-agent config surface has no `directives.replyToCurrent` field. Related #72627 is broader and still open, so this focused request remains useful rather than closeable as implemented or duplicate.

## What This Changes

Add a per-agent configuration directive that disables runtime processing of `[[reply_to_current]]` for selected agents.

## Best Possible Solution

Add a backward-compatible per-agent directive policy defaulting to enabled, enforce disabled `replyToCurrent` handling at shared parsing/normalization boundaries, and leave auto-send suppression to #72627.

Yes. A source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, final reply threading, streaming directive accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy.

## Solution Assessment

Yes, scoped narrowly. A per-agent runtime guard is the maintainable core seam because the directive is parsed in shared delivery code; Feishu rendering and broader auto-send suppression should stay separate.
