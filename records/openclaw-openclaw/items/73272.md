## Summary

Keep open. Current main still recognizes and propagates `[[reply_to_current]]` through shared parser, final reply, streaming, embedded Pi, and outbound normalization paths without consulting a per-agent policy, and the strict per-agent config schema still has no `directives.replyToCurrent` field. The related #72627 remains open but is broader because it also asks for auto-send suppression, so this focused runtime-guard issue is still useful.

## What This Changes

Add an opt-in per-agent directive setting, `agents.<id>.directives.replyToCurrent=disabled`, to prevent runtime processing of accidental `[[reply_to_current]]` output for selected agents.

## Best Possible Solution

Add a backward-compatible, opt-in per-agent directive policy at the shared directive parsing and reply-normalization boundaries, apply it consistently to final replies, streaming chunks, embedded Pi handling, and outbound payloads, and document the config while leaving auto-send suppression to #72627.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` plus a `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe `replyToCurrent` / `replyToId` emitted without any agent policy; `AgentEntrySchema` is strict and has no accepted `agents.list[].directives.replyToCurrent` field.

## Solution Assessment

Yes, if scoped narrowly. The opt-in runtime guard is the maintainable fix for accidental tag processing because it avoids Feishu-specific behavior and preserves explicit `message` tool replies; auto-send suppression should remain a separate product decision.
