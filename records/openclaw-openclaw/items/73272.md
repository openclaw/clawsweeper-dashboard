## Summary

Keep open: current main still has a shared runtime parser for `[[reply_to_current]]` and no strict per-agent config opt-out, so the reported behavior remains valid. The broader #72627 is still open, but this issue is the narrower actionable thread for the directive-only guard.

## What This Changes

The request is to add a per-agent runtime directive that disables processing of the `[[reply_to_current]]` tag.

## Best Possible Solution

Add a backward-compatible `agents.list[].directives.replyToCurrent` policy that defaults to enabled, resolves through agent config, and prevents only `reply_to_current` from producing reply metadata while leaving explicit `message thread-reply` and `[[reply_to:<id>]]` behavior intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, reply delivery normalization, streaming directive accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy check.

## Solution Assessment

Yes for this directive-only request. A per-agent runtime guard is the narrow maintainable solution because the tag is parsed in shared delivery code; broader auto-send suppression remains separate work tracked by #72627.
