## Summary

Keep open. Current main still processes `[[reply_to_current]]` through the shared parser, final reply threading, streaming chunks, embedded Pi reply collection, and outbound payload normalization without any per-agent policy, and the strict per-agent config schema has no `directives.replyToCurrent` field. Related #72627 is open but broader because it also asks for auto-send suppression, so this narrower runtime guard remains useful.

## What This Changes

Add an opt-in per-agent configuration directive that prevents `[[reply_to_current]]` from being processed for selected agents.

## Best Possible Solution

Add a backward-compatible per-agent directive policy, defaulting to current behavior, and enforce it at shared reply directive parsing/normalization boundaries so disabled agents strip `[[reply_to_current]]` without setting reply metadata while explicit `[[reply_to:<id>]]` and Feishu `message thread-reply` continue to work. Document the config and leave auto-send suppression to #72627.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` plus a `currentMessageId` through `parseInlineDirectives`, `applyReplyThreading`, or outbound payload planning and observe `replyToCurrent` / `replyToId` emitted with no agent policy input; the strict agent schema also has no accepted directive field.

## Solution Assessment

Yes, if scoped narrowly. An opt-in runtime guard at the shared parser and reply-normalization boundaries is the maintainable fix because it avoids Feishu-specific behavior and preserves explicit message-tool replies; auto-send suppression should remain a separate product decision.
