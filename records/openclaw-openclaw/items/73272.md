## Summary

Keep open: current main still treats `[[reply_to_current]]` as shared reply metadata and has no per-agent directive policy in the agent config/schema. The older combined report was closed in favor of this narrower tracker, while the auto-send/audience-boundary concern remains separately tracked at https://github.com/openclaw/openclaw/issues/74928.

## What This Changes

The issue requests a per-agent configuration directive that disables runtime processing of `[[reply_to_current]]` for selected agents.

## Best Possible Solution

Define a default-enabled per-agent directive policy, wire it through config/schema/docs and shared reply parsing/delivery, and choose explicit warning/error semantics before implementation.

## Reproduction Assessment

Yes. Source inspection on current main shows assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` through shared parser, reply delivery, and outbound payload planning, while per-agent config/schema has no directive policy field.

## Solution Assessment

Unclear for exact implementation. The per-agent opt-out is a reasonable narrow direction, but maintainers need to decide the disabled-directive warning/error/logging behavior before this should become an automated fix.
