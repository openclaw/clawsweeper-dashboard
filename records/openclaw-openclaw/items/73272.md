## Summary

Keep open: current main still treats `[[reply_to_current]]` as a shared reply directive with no per-agent opt-out, and #72627 remains a broader same-author tracker rather than a reason to close this focused request.

## What This Changes

Request an opt-in per-agent config directive that prevents runtime processing of assistant-emitted `[[reply_to_current]]` tags.

## Best Possible Solution

Add a backward-compatible `agents.list[].directives.replyToCurrent` policy defaulting to enabled, resolve it for the active agent, and enforce disabled handling across shared reply normalization while preserving explicit `[[reply_to:<id>]]` and message-tool replies.

## Reproduction Assessment

Yes. A source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, final reply threading, streaming accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy.

## Solution Assessment

Yes, scoped narrowly. Because the tag is parsed in shared core delivery code, a per-agent runtime/config guard is the maintainable seam; Feishu-specific rendering and broader auto-send suppression should remain separate.
