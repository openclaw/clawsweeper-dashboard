## Summary

Keep open. Current main still treats `[[reply_to_current]]` as a shared delivery directive without any per-agent disable policy, and the strict per-agent config schema has no `directives.replyToCurrent` surface. Related #72627 is open and overlaps, but it is broader because it also asks for auto-send suppression, so this focused runtime guard remains useful.

## What This Changes

Request an opt-in per-agent directive setting that prevents runtime processing of `[[reply_to_current]]` for specific agents while preserving explicit threaded message-tool replies.

## Best Possible Solution

Add a backward-compatible, opt-in per-agent directive policy in the shared reply directive parsing and reply-normalization boundaries, apply it consistently to final replies, streaming chunks, embedded Pi replies, and outbound payloads, then document and test the config while leaving auto-send suppression to #72627.

## Reproduction Assessment

Yes. A source-level reproduction is to pass assistant text containing `[[reply_to_current]]` and a `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe `replyToCurrent` / `replyToId` emitted with no agent policy; `AgentEntrySchema` rejects unknown per-agent fields because it is strict.

## Solution Assessment

Yes, if scoped narrowly. An opt-in runtime guard is the maintainable fix for accidental tag processing because it is channel-agnostic and preserves explicit `message` tool threading; auto-send suppression should remain a separate product decision.
