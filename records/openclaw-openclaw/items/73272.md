## Summary

Keep open: current main still treats `[[reply_to_current]]` as a shared runtime directive and the strict per-agent config surface has no `directives.replyToCurrent` opt-out. The broader #72627 remains open, but this narrower same-author request is a bounded implementation thread for directive-only behavior.

## What This Changes

Add a per-agent config directive that can disable processing of the `[[reply_to_current]]` runtime reply tag for selected agents.

## Best Possible Solution

Add a backward-compatible per-agent directive policy that defaults to enabled, resolves through the agent config surface, and enforces disabled `replyToCurrent` handling at shared parsing and reply-normalization boundaries while leaving broader auto-send suppression to #72627.

## Reproduction Assessment

Yes. Source inspection on current main shows assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` through the shared parser, reply delivery, streaming accumulator, and outbound payload planning with no per-agent policy check.

## Solution Assessment

Yes for the directive-only request. A per-agent runtime guard is the narrow maintainable fix because the directive is parsed in shared delivery code; Feishu-specific rendering and auto-send suppression should stay separate.
