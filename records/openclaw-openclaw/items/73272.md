## Summary

Keep open: current main still parses and propagates `[[reply_to_current]]` through shared reply delivery, and the requested per-agent opt-out is not present. The older combined report was closed in favor of this narrower tracker, while the auto-send/audience-boundary concern remains separate at https://github.com/openclaw/openclaw/issues/74928.

## What This Changes

The issue requests a per-agent config directive that disables runtime handling of the `[[reply_to_current]]` marker.

## Best Possible Solution

Add a backward-compatible per-agent directive policy that defaults to enabled, disables only `reply_to_current` parsing/normalization for opted-in agents, documents the config, and leaves auto-send/audience work to https://github.com/openclaw/openclaw/issues/74928.

## Reproduction Assessment

Yes. Source inspection on current main shows assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` through shared parser, reply delivery, and outbound payload planning, while per-agent config/schema has no directive policy field.

## Solution Assessment

Unclear for implementation details. The requested per-agent opt-out is a reasonable narrow direction, but maintainers need to decide the warning/error/logging semantics before a fix should be automated.
