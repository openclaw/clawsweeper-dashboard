## Summary

Keep open. Current main still parses and propagates `[[reply_to_current]]` without a per-agent opt-out, and the strict per-agent config contract has no `directives.replyToCurrent` field. Related #72627 is open but broader because it also asks for auto-send suppression, so this focused runtime guard remains useful rather than implemented or cleanly duplicate-closeable.

## What This Changes

Request to add a per-agent config directive that disables runtime processing of assistant-emitted `[[reply_to_current]]` while preserving explicit message-tool replies.

## Best Possible Solution

Add a backward-compatible per-agent directive policy to the runtime/config contract, default it to enabled, and enforce disabled `replyToCurrent` behavior at the shared parsing and reply-normalization boundaries while leaving explicit `[[reply_to:<id>]]`, message-tool thread replies, and the broader auto-send suppression request from #72627 separate.

## Reproduction Assessment

Yes. A source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, `applyReplyThreading`, streaming accumulation, embedded Pi block handling, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy; strict agent config parsing also has no `directives.replyToCurrent` surface.

## Solution Assessment

Yes, if scoped narrowly. An opt-in per-agent runtime guard is the maintainable core fix because the directive is parsed in shared runtime delivery code, while channel-specific Feishu behavior and auto-send suppression should remain out of scope for this issue.
