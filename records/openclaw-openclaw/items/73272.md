## Summary

Keep open. Current main still treats `[[reply_to_current]]` as an enabled shared reply directive with no per-agent opt-out, while the strict per-agent config schema and resolved agent config do not expose `agents.<id>.directives.replyToCurrent`. Related #72627 is open but broader because it also asks for auto-send suppression, so this focused runtime guard remains useful as its own repair target.

## What This Changes

Request a per-agent config directive that prevents the runtime from processing `[[reply_to_current]]` for selected agents while preserving explicit message-tool replies.

## Best Possible Solution

Add a backward-compatible per-agent directive policy, defaulting `replyToCurrent` to enabled, and enforce the disabled state at shared final, streaming, embedded Pi, and outbound normalization boundaries while leaving `[[reply_to:<id>]]`, Feishu `message thread-reply`, and auto-send suppression unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, `applyReplyThreading`, streaming accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any per-agent policy; strict agent config also has no accepted directive field.

## Solution Assessment

Yes, scoped narrowly. A core runtime/config opt-out is the maintainable fix because the directive is parsed in shared delivery code, while Feishu-specific rendering and the broader auto-send request belong outside this issue.
