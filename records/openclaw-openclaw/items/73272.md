## Summary

Keep open: current main still treats `[[reply_to_current]]` as a supported shared delivery directive with no per-agent opt-out, and the strict per-agent config surface has no `directives.replyToCurrent` field. Related #72627 is open and broader, so this focused request remains useful rather than closeable.

## What This Changes

Request a per-agent config directive that disables runtime processing of `[[reply_to_current]]` while preserving explicit message-tool replies.

## Best Possible Solution

Add a backward-compatible per-agent directive policy, default it to enabled, and enforce disabled `replyToCurrent` handling at shared parsing/reply-normalization boundaries while leaving auto-send suppression to #72627.

## Reproduction Assessment

Yes. A source-level reproduction is to pass assistant text containing `[[reply_to_current]]` through `parseInlineDirectives`, final reply threading, streaming directive accumulation, or outbound payload planning and observe `replyToCurrent` emitted without any agent policy.

## Solution Assessment

Yes, scoped narrowly. A per-agent runtime guard is the maintainable core seam because the directive is parsed in shared runtime delivery code; Feishu-specific rendering and broader auto-send suppression should remain separate.
