## Summary

Keep open: current main still has no per-agent `directives.replyToCurrent` opt-out and still parses/preserves `[[reply_to_current]]` through shared assistant delivery. The broader same-author issue remains open, but this narrower thread is a useful bounded implementation target rather than a cleanup close.

## What This Changes

Add an opt-in per-agent config directive that disables runtime handling of `[[reply_to_current]]` for selected agents.

## Best Possible Solution

Add a backward-compatible per-agent directive policy that defaults to enabled, disables only `reply_to_current` parsing/normalization for opted-in agents, and leaves broader auto-send suppression to the related broader issue.

## Reproduction Assessment

Yes. Source inspection on current main shows assistant text containing `[[reply_to_current]]` is parsed into `replyToCurrent` through the shared parser, reply delivery, streaming accumulator, and outbound payload planning with no per-agent policy check.

## Solution Assessment

Yes for the directive-only request. An opt-in per-agent runtime guard at shared parsing and reply-normalization boundaries is the narrow maintainable solution; the broader auto-send policy should stay with https://github.com/openclaw/openclaw/issues/72627.
