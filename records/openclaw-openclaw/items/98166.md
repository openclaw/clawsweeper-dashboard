## Summary

Keep open: current main and the latest release still use an exact silent-token matcher, while the overlapping open PR only covers trailing punctuation/quotes and does not cover leading-dot or star-wrapped token-only leaks.

## What This Changes

The issue asks OpenClaw to suppress token-only `NO_REPLY` outputs when the sentinel is wrapped in stray punctuation, without suppressing prose that merely mentions the token.

## Best Possible Solution

Define one shared token-only silent-reply classifier that tolerates maintainer-approved edge punctuation, preserves substantive replies, and lands in coordination with the overlapping open PR.

## Reproduction Assessment

Yes, at source level: current main's exact regex rejects `.NO_REPLY`, `NO_REPLY.`, `*NO_REPLY*`, and `...NO_REPLY...`, and those payloads then miss the shared suppression classifier. I did not run a live Slack or channel delivery path in this read-only review.

## Solution Assessment

Unclear until maintainers settle the exported helper contract. The best implementation shape is the shared classifier, but it should be aligned with https://github.com/openclaw/openclaw/pull/93510 and the existing substantive-reply guards.
