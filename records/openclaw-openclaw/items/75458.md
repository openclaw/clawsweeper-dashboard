## Summary

Keep open: current main still has the reported terminal-status gap. The shared status reaction controller tracks multiple active lifecycle emojis and `finishWithEmoji()` applies done/error without removing stale non-terminal active reactions; Discord then waits for delayed `clear()` when `removeAckAfterReply` is enabled, which is the same class of cleanup the reported release already had when the permanent thinking reaction was observed. The related repair PR #75570 was closed unmerged, so the issue remains actionable.

## What This Changes

Discord status reactions should not leave a non-terminal lifecycle emoji such as thinking visible after a completed turn, especially when `messages.removeAckAfterReply` is enabled.

## Best Possible Solution

Land a narrow shared-controller fix that removes all non-terminal active lifecycle reactions when entering done/error, while preserving the terminal emoji until the configured hold period and then clearing all reactions when `removeAckAfterReply` is enabled. Add focused controller coverage plus a Discord successful-turn regression test and a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue gives a live Discord reproduction on `v2026.4.29`, and current main has a deterministic unit-level reproduction: drive queued/thinking/tool to done with a Discord-style `removeReaction` adapter and observe that terminal completion does not remove the stale non-terminal active emojis before delayed clear.

## Solution Assessment

No, current main's delayed clear/restore behavior is not the best complete solution for this issue. The narrow maintainable direction is the #75570 approach adapted to current main: terminal done/error should immediately clean up every other tracked lifecycle reaction, with delayed full clear still handled by Discord finalization.
