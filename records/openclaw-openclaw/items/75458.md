## Summary

Keep open. The report is concrete, recent, and still matches current main: Discord uses the shared status reaction controller, and that controller can lose track of the newly applied emoji when removing the previous emoji fails after a successful set, which can leave a non-terminal lifecycle reaction behind. The relevant source is unchanged from the reported v2026.4.29 release on the inspected paths, so this is not implemented on main.

## What This Changes

Fix Discord lifecycle status reactions so completed turns do not leave thinking or other non-terminal status emojis, especially when `messages.removeAckAfterReply` is enabled.

## Best Possible Solution

Keep the lifecycle implementation in the shared status reaction controller, make reaction state resilient when removing a previous emoji fails after the new emoji has already been applied, and add a focused regression test that proves terminal done/clear cannot leave `thinking` or other non-terminal emojis active.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to use a mock reaction adapter where `setReaction(thinking)` succeeds but `removeReaction(previous)` fails; current `applyEmoji` then leaves `currentEmoji` stale, and a later `setDone()` can leave the thinking reaction active alongside terminal state.

## Solution Assessment

Yes, a narrow shared-controller fix is the best path. Discord wiring already calls `setDone()` and schedules `clear()`, so the maintainable fix is to harden the controller/adapter cleanup semantics and cover them with focused controller and Discord lifecycle tests rather than adding Discord-only special cases first.
