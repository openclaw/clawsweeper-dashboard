## Summary

Keep open. Current main generally routes Telegram documents and recognizes PPTX MIME types, but there is still a source-backed path where Telegram `getFile` reports a size-limit error, OpenClaw returns `null`, and the inbound turn continues without `MediaPaths`; the reported PPTX needs file-size/log confirmation before this can be closed or safely fixed.

## What This Changes

Reports that Telegram direct chats on OpenClaw 2026.4.24 drop PPTX document attachments while text and PDF attachments are received.

## Best Possible Solution

Keep this open until logs and file size identify the failing boundary, then fix Telegram document-download handling or size-limit UX separately from Office text extraction work.

## Reproduction Assessment

Unclear for the exact PPTX because the report lacks file size and Telegram logs. Source inspection gives a matching current-main symptom when `getFile` returns `file is too big`: the resolver returns `null` and the handler forwards a text-only turn.

## Solution Assessment

Unclear. The narrow maintainable solution depends on whether this is a true PPTX download regression, Telegram Bot API size-limit behavior needing explicit user-visible handling, or the separate Office extraction scope tracked by #12047 and #57612.
