## Summary

Keep open: the PR changes ACPX startup-probe defaults, but the probe still runs in a detached task, so the claimed “gateway ready waits for ACPX usable” behavior is not established.

## What This Changes

The PR flips ACPX startup-probe env handling to opt-out, updates ACPX docs/tests, and adds a changelog entry for the linked readiness behavior.

## Best Possible Solution

Make ACPX startup await the readiness probe before Gateway readiness, or narrow the change to documented lazy registration, with delayed-probe regression coverage and redacted real gateway/acpx log proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: on the PR head, `service.start()` can resolve after scheduling a delayed `probeAvailability()` task while Gateway readiness only waits for `service.start()`. No live Windows gateway run was performed in this read-only review.

## Solution Assessment

No. Flipping the default is not the best complete fix while the actual ACPX probe remains detached from the startup promise and the docs/changelog claim readiness semantics the code does not guarantee.

## Review Findings

Overall correctness: patch is incorrect
