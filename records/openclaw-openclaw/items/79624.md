## Summary

Keep open: the PR changes ACPX startup probing to opt-out, but the PR head still returns from service startup before the async probe finishes, so the stated gateway-ready-waits-for-usable-acpx behavior is incomplete and still lacks inspectable real behavior proof.

## What This Changes

The PR flips ACPX startup-probe env handling from opt-in to opt-out, updates ACPX docs/tests, and adds a changelog entry for the linked readiness behavior.

## Best Possible Solution

Update the ACPX startup path to await the startup probe before gateway readiness, or narrow the PR/docs/changelog claim to backend registration only, then add delayed-probe coverage and redacted real gateway/acpx log proof.

## Reproduction Assessment

Yes from source inspection: a delayed `probeAvailability()` can finish after `service.start()` resolves on the PR head. No live Windows gateway run was established in this review.

## Solution Assessment

No. Flipping the default addresses lazy registration, but it is not the narrow complete fix for “gateway ready waits for acpx usable” while the actual probe still runs in a detached task.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Await the probe before treating startup as ready:** `extensions/acpx/src/service.ts:203`
  - body: Changing this helper to run the startup probe by default still leaves the probe detached: the PR head registers the backend, launches `probeAvailability()` in a `void` task, and `register.runtime.ts` only waits for `service.start()`. A delayed probe can therefore finish after gateway readiness, and the backend can still be unavailable via the health callback. Await the probe in the startup path or narrow the docs/changelog claim to registration-only behavior.
  - confidence: 0.9
