## Summary

Keep open: current main still has an unthrottled watch-sync scheduler, and this draft PR is an active, narrow fix. I found no blocking functional or security issue, but the patch introduces one misleading comment about a non-existent config key.

## What This Changes

The PR adds a fixed 60-second throttle to memory-core watch-triggered sync scheduling by tracking the last watch sync and collapsing cooldown events into one pending timer.

## Best Possible Solution

Land a narrow memory-core scheduler fix with fake-timer coverage for cooldown coalescing, while removing or implementing the advertised config key before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no: current main marks memory dirty and schedules a watch sync after every stable watcher event with only debounce, but I did not run a long-lived disk-growth reproduction in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes: throttling inside scheduleWatchSync() is the narrowest owner-local place to reduce watch-triggered reindex churn, but the patch should add timer coverage and fix the non-existent config-key comment.
