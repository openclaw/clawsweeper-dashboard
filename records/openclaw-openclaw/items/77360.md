## Summary

Keep open: current main still builds `inboundUserContext` but omits it from the `isBareSessionReset` prompt branch, so model-invoking bare reset/bootstrap turns miss sender metadata while normal turns include it.

## What This Changes

The issue requests that bare `/new` and `/reset` model bootstrap prompts include the same untrusted sender identity metadata already included on normal user messages.

## Best Possible Solution

Prepend the existing user-role inbound context to model-invoking bare reset/bootstrap prompts, add focused `/new` and `/reset` regression coverage, and keep sender fields out of the trusted system prompt.

## Reproduction Assessment

Yes, from source inspection: a model-invoking bare `/new` or `/reset` path with sender fields computes `inboundUserContext` but builds `followupRun.prompt` from the reset-only array that excludes it. I did not run a live or unit reproduction because this review was required to remain read-only.

## Solution Assessment

Yes. Adding `inboundUserContext` to the existing `isBareSessionReset` prompt array is the narrowest maintainable fix, with regression tests proving sender blocks appear while the trusted system prompt and bare command acknowledgement behavior stay unchanged.
