## Summary

Keep open: current main still computes inbound user-context metadata but drops it from the bare reset/bootstrap prompt branch before the model prompt is built, and existing coverage only checks startup/transcript behavior for that path.

## What This Changes

The issue requests sender identity metadata in the user prompt for bare `/new` and `/reset` turns.

## Best Possible Solution

Prepend user-role inbound context to bare reset/bootstrap prompts, add regression coverage for bare `/new` and `/reset`, and keep sender fields out of trusted system metadata.

## Reproduction Assessment

Yes, from source: a `runPreparedReply` case with `resetTriggered`, bare `/new` or `/reset`, and sender fields reaches `runReplyAgent` with `followupRun.prompt` built from the bare-reset branch that omits `inboundUserContext`. I did not execute the test path in this read-only review.

## Solution Assessment

Yes. Adding `inboundUserContext` to the existing bare-reset prompt array is the narrowest maintainable fix, with tests guarding both reset commands and existing transcript/acknowledgement behavior.
