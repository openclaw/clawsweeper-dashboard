## Summary

Keep open: current main still computes inbound user-context metadata but omits it from the bare reset/bootstrap prompt branch, so sender identity is unavailable on that first model turn.

## What This Changes

Bare `/new` and `/reset` reset/bootstrap prompts should include the existing user-role inbound sender context.

## Best Possible Solution

Prepend `inboundUserContext` to the bare reset/bootstrap prompt array, add regression coverage for bare `/new` and `/reset`, and keep sender fields out of the trusted system prompt.

## Reproduction Assessment

Yes, from source inspection: a `runPreparedReply` reset/bootstrap case for bare `/new` or `/reset` with sender fields reaches `runReplyAgent` through the branch that omits `inboundUserContext`. I did not run tests because this review was read-only.

## Solution Assessment

Yes. Adding `inboundUserContext` to the existing bare reset prompt array is the narrowest maintainable fix, as long as the ordinary text-command acknowledgement path remains unchanged.
