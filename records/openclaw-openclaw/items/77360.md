## Summary

Keep open: current main still computes the inbound user-context sender metadata but drops it from the model prompt on the bare reset/bootstrap branch, so the first model turn can miss sender identity.

## What This Changes

Ensure bare `/new` and `/reset` model bootstrap prompts include the existing untrusted inbound sender context.

## Best Possible Solution

Prepend inbound user context only to model-invoking bare reset/bootstrap prompts, add focused coverage for bare `/new` and `/reset`, and keep sender fields out of the trusted system prompt.

## Reproduction Assessment

Yes, from source. A focused `runPreparedReply` case with `resetTriggered`, a bare `/new` or `/reset`, and sender fields would reach `runReplyAgent` with a `followupRun.prompt` that lacks the `Conversation info` and `Sender` blocks.

## Solution Assessment

Yes. Adding `inboundUserContext` to the existing bare reset prompt array is the narrowest maintainable fix, with regression tests to keep the system prompt and command acknowledgement behavior unchanged.

## Repair Work Prompt

Fix the source-reproducible prompt composition bug from https://github.com/openclaw/openclaw/issues/77360. In `src/auto-reply/reply/get-reply-run.ts`, ensure model-invoking bare `/new` and `/reset` reset/bootstrap prompts include the existing `inboundUserContext` before startup context/reset prompt text. Keep sender metadata in the user-role untrusted blocks, do not add sender fields to `buildInboundMetaSystemPrompt`, and preserve the ordinary text-command acknowledgement path that does not run the model for bare reset commands. Add focused regression coverage for bare `/new` and bare `/reset` showing `followupRun.prompt` contains `Conversation info (untrusted metadata):` and `Sender (untrusted metadata):` when sender fields are present, while transcript marker behavior remains unchanged. Add a changelog fix entry if the repo policy treats this prompt behavior as user-facing.
