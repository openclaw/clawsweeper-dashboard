## Summary

Keep open: current main still omits the already-built inbound user-context block from the model prompt used by bare reset/bootstrap runs, so sender identity can be absent on the first model turn.

## What This Changes

Bare `/new` and `/reset` should include the same sender identity metadata in the user prompt that normal message prompts already include.

## Best Possible Solution

Prepend user-role inbound context to bare reset/bootstrap model prompts, add focused regression coverage for `/new` and `/reset`, and keep sender fields out of the trusted system prompt.

## Reproduction Assessment

Yes, from source. A focused `runPreparedReply` case with `resetTriggered`, bare `/new` or `/reset`, and sender fields would build `followupRun.prompt` through the bare reset branch that omits `inboundUserContext`.

## Solution Assessment

Yes. Adding `inboundUserContext` to the existing bare reset prompt array is the narrowest maintainable fix, with tests proving sender blocks appear without moving sender data into the trusted system prompt.
