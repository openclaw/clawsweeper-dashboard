## Label Justifications

- `impact:security`: The request concerns plugin-controlled safety and honest reporting around tool output before the model consumes it.

## Summary

Current main and the shipped v2026.6.6 release already implement the central request through the manifest-gated `api.registerAgentToolResultMiddleware(...)` pre-model result transform seam; the remaining open hook items found in search are adjacent internal-HOOK.md or built-in-scanner product requests, not this plugin API gap.

## What This Changes

The issue requests additive synchronous plugin API hooks around individual tool executions so plugins can inspect, annotate, veto, or substitute tool results before the model sees them.

## Best Possible Solution

Keep the shipped `api.registerAgentToolResultMiddleware(...)` contract as the supported pre-model result-transform API; use separate canonical issues for managed HOOK.md bridging or built-in scanner policy if those distinct requests are needed.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and release inspection show the requested class of same-turn, pre-model tool-result substitution is already available through the shipped middleware API.

## Solution Assessment

Yes. The existing manifest-gated, runtime-neutral middleware seam is the maintainable solution because it avoids changing `after_tool_call` from observation-only while still allowing trusted async tool-result transformation before model consumption.
