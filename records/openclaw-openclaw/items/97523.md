## Label Justifications

- `P0`: The issue describes permanent deletion of user-authored memory-wiki notes from an everyday transient storage outage.
- `impact:data-loss`: The affected path deletes imported pages and their human-notes blocks, losing user-authored wiki content.

## Summary

Keep open: current main still treats a transiently missing `unsafe-local` configured path as an empty active set and then runs the destructive imported-source prune, which can delete imported pages that contain user-authored human-notes blocks.

## What This Changes

Prevent memory-wiki unsafe-local sync from pruning imported pages and their human-notes blocks when a configured source path is only transiently unavailable.

## Best Possible Solution

Teach unsafe-local sync to distinguish successfully scanned source scopes from unavailable source scopes, skip destructive pruning for unavailable scopes, and add a regression that proves human notes survive a transient missing-path sync while genuine source deletion still prunes.

## Reproduction Assessment

Yes. The issue includes a concrete production-function repro against the current main SHA, and source inspection confirms the same missing-path-to-empty-active-set-to-prune path remains in current main.

## Solution Assessment

Yes, but the best fix should be narrower than a global zero-artifacts skip. Track which configured source scopes were actually scanned and only prune entries belonging to confirmed available scopes, preserving user pages for transiently unavailable scopes.
