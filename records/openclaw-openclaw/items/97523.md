## Summary

Keep open: current main and the latest release still treat missing `unsafe-local` configured paths as an empty active set and run the destructive imported-source prune; https://github.com/openclaw/openclaw/pull/97527 is an open candidate fix, but it is not merged and already has review blockers.

## What This Changes

The issue reports memory-wiki `unsafe-local` sync deleting imported pages and their user-authored human-notes blocks when a configured source path is transiently unavailable.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/97527 with path-scoped unsafe-local pruning that preserves entries for unavailable configured roots while still pruning real deletions in successfully scanned roots, backed by regression tests.

Yes. The issue includes a production-function ENOENT reproduction against current main at filing time, and the relevant files are unchanged on current main while still showing the same missing-path to destructive-prune path.

## Solution Assessment

Yes for the requested behavior, but the maintainable fix should be path-scoped rather than a blanket zero-artifact skip so genuine deletions in scanned paths still prune correctly.
