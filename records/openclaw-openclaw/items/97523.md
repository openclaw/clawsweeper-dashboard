## Label Justifications

- `P1`: The issue describes irreversible deletion of user-authored memory-wiki Notes content during an ordinary transient unsafe-local source outage.
- `impact:data-loss`: The reported failure removes imported source page files and their human-authored Notes blocks from the wiki vault.

## Summary

Keep open: current main still conflates an unavailable unsafe-local configured path with an empty source set, then prunes imported pages, which can delete user-authored human notes.

## What This Changes

The issue requests that memory-wiki unsafe-local sync preserve imported pages and human notes when a configured local source path is temporarily unavailable.

## Best Possible Solution

Make unsafe-local collection report unreadable configured roots distinctly from successful empty scans, skip pruning for affected roots when collection is incomplete, and add a regression that preserves human notes through a transient root outage while retaining stale-file pruning after a successful scan.

## Reproduction Assessment

Yes, at source level. Current main still skips missing unsafe-local configured paths, builds an empty activeKeys set, and prunes page files/state; the reporter also supplied a real function-level ENOENT repro against the same main SHA.

## Solution Assessment

Yes, the maintainable fix should stay inside the memory-wiki unsafe-local sync boundary. The safer shape is to distinguish incomplete collection from confirmed source deletion, not to add a new config flag or global fallback.
