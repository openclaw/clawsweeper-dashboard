## Summary

Keep this PR open for revision. The patch targets the right heartbeat/model-selection path and current main still applies stored overrides for heartbeat runs without explicit heartbeat.model, but the proposed stale check depends on fallback notice metadata that current runtime can clear and persist while leaving the auto model override in place, so the linked heartbeat pinning bug is not fully fixed.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice selected-model metadata, threads heartbeat context through reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Persist durable auto-fallback origin metadata, or an equivalent explicit changed-default marker, when the auto override is written. Heartbeat selection should use that durable origin to clear only stale direct session auto overrides after the configured default changes, while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, which current runtime can persist after notice state clears, then run heartbeat model selection after changing agents.defaults.model.primary; the PR keeps the old fallback selected.

## Solution Assessment

No. The PR changes the correct area, but fallback notice fields are transient display state, not durable origin metadata; the safer fix is to persist or infer the original configured model for auto overrides and compare that against the current default.
