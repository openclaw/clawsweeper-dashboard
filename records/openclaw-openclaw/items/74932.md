## Summary

Keep open: the PR fixes metadata-present cases, but it still preserves auto overrides when fallbackNoticeSelectedModel is missing, and current main can persist that missing-notice state after fallback clears, so the heartbeat stale-model bug can remain for long-lived sessions.

## What This Changes

The PR threads heartbeat context into reply model selection, skips and clears direct session auto-fallback overrides when their recorded selected model differs from the current default, and updates tests, docs, and changelog.

## Best Possible Solution

Persist durable auto-fallback origin metadata, or define a conservative compatibility rule for missing metadata, and cover both metadata-present and cleared-notice heartbeat sessions.

## Reproduction Assessment

Yes. By source inspection, seed a direct session override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after agents.defaults.model.primary changes; the PR keeps the old fallback selected.

## Solution Assessment

No. The patch targets the right selection surface, but it treats fallback notice fields as durable origin metadata even though current runtime clears and persists those fields after fallback state clears.
