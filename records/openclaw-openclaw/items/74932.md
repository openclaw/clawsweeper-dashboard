## Summary

Keep open. The PR targets a valid heartbeat model-pinning bug, but it still misses persisted auto-fallback overrides after fallback notice metadata has been cleared, so long-lived heartbeat sessions can remain pinned after the configured default changes.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice metadata, threads heartbeat context through reply model selection, and updates tests, model-failover docs, and the changelog.

## Best Possible Solution

Persist durable auto-fallback origin metadata, or an equivalent changed-default marker, when writing the auto override so heartbeat can clear only stale direct auto overrides while preserving explicit, user, legacy, parent, and unchanged-default behavior.

## Reproduction Assessment

Yes. Seed a direct session override with providerOverride, modelOverride, and modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after changing agents.defaults.model.primary; the PR keeps applying the old fallback.

## Solution Assessment

No. The PR changes the right surface, but fallback notice fields are transient display state that current runtime clears, so durable origin tracking or a conservative equivalent is the safer fix.
