## Summary

Keep open. The PR changes the right heartbeat model-selection surface and adds useful tests/docs, but its stale detector depends on fallback notice metadata that current main clears while leaving the auto model override persisted. That means a long-lived heartbeat session can still stay pinned after the configured default changes, matching the linked #74284 failure mode.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice selected-model metadata, threads heartbeat context through reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Persist durable auto-fallback origin metadata separate from the display-only fallback notice state, then use that origin to clear only stale direct heartbeat auto overrides after `agents.defaults.model.primary` changes while preserving explicit `heartbeat.model`, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session override with `modelOverrideSource: "auto"` and no `fallbackNoticeSelectedModel` after changing the configured default, then run heartbeat model selection without explicit `heartbeat.model`; the PR detector returns false and the old fallback remains selected.

## Solution Assessment

No. The patch is in the right area, but using transient fallback notice fields as origin metadata is not durable enough; a safer solution needs persistent origin state or an explicit changed-default marker plus clear compatibility behavior for missing metadata.
