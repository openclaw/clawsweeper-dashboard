## Summary

Keep open. The PR is aimed at the right heartbeat/model-selection path, but its stale-override detection depends on fallback notice metadata that current runtime can clear while leaving `modelOverrideSource: "auto"` persisted, so the linked stale heartbeat fallback case can still survive a later default-model change.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice selected-model metadata, threads heartbeat context through reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Persist durable auto-fallback origin metadata, or an equivalent changed-default marker, when the auto override is written. Heartbeat selection should clear only stale direct session auto overrides whose origin no longer matches the current default, while preserving explicit `heartbeat.model`, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default fallback persistence.

## Reproduction Assessment

Yes. Seed a direct heartbeat session override with `providerOverride`/`modelOverride`/`modelOverrideSource: "auto"` but no `fallbackNoticeSelectedModel`, then change `agents.defaults.model.primary` and run heartbeat model selection; the PR keeps the old fallback because the stale detector returns false for missing notice metadata.

## Solution Assessment

No. The PR changes the right code path, but using transient fallback notice fields is not the most maintainable fix; durable origin metadata or explicit changed-default state is the safer boundary.
