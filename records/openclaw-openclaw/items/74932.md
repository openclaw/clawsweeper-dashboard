## Summary

Keep this PR open. Current main still has the heartbeat stored-override path, and the PR targets the right area, but the proposed stale detector misses persisted auto overrides after fallback notice state has been cleared.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override clearing keyed to fallback notice selected-model metadata, threads heartbeat context into reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Store or otherwise recover durable auto-fallback origin information for direct session overrides, then clear only stale heartbeat-owned auto overrides after the configured default changes while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after changing the configured default; the PR's detector treats it as non-stale and keeps the old fallback selected.

## Solution Assessment

No. The PR changes the right seam, but relying on fallback notice fields is not robust because current main clears and persists those fields after fallback clears; durable origin metadata or an equivalent changed-default marker is the safer fix.
