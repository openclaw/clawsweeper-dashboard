## Summary

Keep open: this PR targets a real heartbeat model-selection bug and has after-fix proof for the metadata-present case, but it still leaves stale direct heartbeat auto-fallback overrides in place after current runtime clears the fallback notice metadata.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice metadata, plus focused tests, model-failover docs, and a changelog entry.

## Best Possible Solution

Persist durable metadata for the configured model that produced an auto fallback override, then clear only direct heartbeat auto overrides whose recorded origin differs from the current configured default while preserving explicit heartbeat, user, legacy, parent, and unchanged-default behavior.

## Reproduction Assessment

Yes, source-reproducible. Seed a heartbeat session with `providerOverride`/`modelOverride`/`modelOverrideSource: "auto"` but no `fallbackNoticeSelectedModel` after normal fallback notice clearing, then change `agents.defaults.model.primary`; this PR preserves the stale direct override.

## Solution Assessment

No. The PR changes the right model-selection surface, but `fallbackNoticeSelectedModel` is notice-suppression state that current runtime clears, so it is not durable enough to decide whether an auto override came from the current default.
