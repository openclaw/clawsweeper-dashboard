## Summary

Keep open. The PR targets the right heartbeat model-selection path, but the proposed stale check depends on fallback notice fields that current runtime code can clear and persist while the auto model override remains, so a later default-model change can still leave heartbeat pinned to the old fallback. Current main does not already implement the intended behavior, and the linked bug remains open.

## What This Changes

The PR adds heartbeat-only stale direct auto-fallback override detection based on fallback notice selected-model metadata, threads heartbeat context through model selection, and updates tests, model-failover docs, and the changelog.

## Best Possible Solution

Persist durable auto-fallback origin metadata, or an equivalent explicit changed-default marker, and use it to clear only stale direct heartbeat auto overrides after the configured default changes while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after changing agents.defaults.model.primary; this PR's stale detector returns false and keeps the old fallback selected.

## Solution Assessment

No. The PR changes the right area, but fallback notice fields are UI/status lifecycle state and are cleared when the fallback is no longer active; the safer fix is durable origin metadata rather than treating those transient fields as the source of truth.
