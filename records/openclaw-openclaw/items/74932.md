## Summary

Keep this PR open. It changes the right model-selection surface, but the patch still lets a stale heartbeat auto-fallback override survive after fallback notice state has been cleared, so it does not fully fix the linked heartbeat pinning bug.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing, threads heartbeat context through reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Persist durable auto-fallback origin metadata, or an equivalent explicit changed-default marker, when the fallback override is written. Heartbeat selection can then clear only stale direct session auto overrides after the configured default changes while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after changing the configured default; the PR's stale detector returns false and keeps the old fallback selected.

## Solution Assessment

No. The PR handles the notice-present case, but fallback notice fields are transient UI/runtime state, so using them as the only origin marker is not the narrowest durable fix.
