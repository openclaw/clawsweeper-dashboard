## Summary

Keep open. The PR targets a real gateway-bindable registry reuse miss, but the current diff should not merge because it drops the existing cache-key compatibility path for helper callers that pass `installBundledRuntimeDeps: false`; those hot paths can miss an already-active compatible registry and re-enter plugin loading, which is the class of behavior this PR is trying to prevent.

## What This Changes

The PR records active core gateway method names in plugin runtime state, uses them to synthesize omitted core gateway handlers during gateway-bindable active-registry compatibility checks, and adds loader runtime-registry regression tests.

## Best Possible Solution

Keep the PR open and repair the loader compatibility refactor so method-name reconstruction composes with the existing `installBundledRuntimeDeps: false` cache-key normalization, then add a regression test using the actual helper hot-path options and land only after checks are green.

## Reproduction Assessment

Yes. A focused unit seam can set an active registry from a gateway-bindable load, then call `resolveRuntimePluginRegistry` or `getCompatibleActivePluginRegistry` with the same config/workspace/runtime options, omitted core gateway handlers, and `installBundledRuntimeDeps: false`; the PR's new matcher returns `undefined` when the active key differs only by runtime-deps mode.

## Solution Assessment

No. Persisting gateway method names and synthesizing handlers is the right narrow direction, but the current implementation is not the best merge path until it preserves the old skip-runtime-deps compatibility and covers that case in tests.
