## Summary

Keep open. Current main still leaves automatic daily/idle rollover on the lifecycle-only path while bundled session-memory only handles internal command new/reset events; this PR is a plausible narrow fix with no discrete code findings, but it still needs external real behavior proof and maintainer selection against the overlapping implementation.

## What This Changes

The PR emits an internal `command:reset` hook with `auto:daily` or `auto:idle` during automatic stale rollover, adds focused daily/idle/fresh-session tests, and updates session-memory docs plus changelog text.

## Best Possible Solution

Land this or a maintainer-approved replacement that persists session-memory exactly once for automatic daily and idle rollovers while preserving manual reset behavior and existing session lifecycle hooks.

## Reproduction Assessment

Yes, by source inspection. Seed a stale daily or idle session and call `initSessionState` on current main: it creates a new session and lifecycle hooks while bundled session-memory remains gated to `command:new` and `command:reset`.

## Solution Assessment

Mostly yes. Reusing the existing internal `command:reset` event is narrow for bundled session-memory, but maintainers should choose it deliberately against the broader `before_reset`/session-hook approach in https://github.com/openclaw/openclaw/pull/61675.
