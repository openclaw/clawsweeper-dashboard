## Summary

Keep open: the useful UI bug fix is not on current main, but this PR still lacks inspectable real UI proof and has a selector-specificity bug that can hide the active row styling when the selected row is also hovered.

## What This Changes

The branch splits the command palette row hover and active CSS background rules into theme-aware `color-mix(...)` styles in `ui/src/styles/components.css`.

## Best Possible Solution

Land a focused CSS update that keeps hover and active colors distinct in default and light themes, makes active styling win on overlap, and includes redacted light/dark Control UI proof.

## Reproduction Assessment

Yes, at source level. Current main has one shared hover/active background rule, and the command palette renderer applies active selection while hover can also match; I did not run the browser UI in this read-only review.

## Solution Assessment

No. Splitting the colors is the right direction, but this branch's hover selectors outrank active selectors when both apply and there is no artifact proving actual light/dark UI contrast.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let active rows override hover:** `ui/src/styles/components.css:3539-3552`
  - body: The new `:hover` selectors are more specific than `.cmd-palette__item--active`, so a selected option under the pointer renders the hover color instead of the active color. Add an active-hover rule or an equal-specificity active selector after the hover rules so keyboard selection remains visibly distinct on overlap.
