## Summary

Keep open: current main and the latest shipped release still use one shared command palette hover/active background, so the targeted UI fix is useful, but this PR head still lets `:hover` override the active state and lacks inspectable after-fix proof.

## What This Changes

The PR splits Control UI command palette hover and active row background rules in `ui/src/styles/components.css` using theme-aware `color-mix` values.

## Best Possible Solution

A CSS-only fix should keep hover and active rows visually distinct in all themes while ensuring the active row wins when an item is both active and hovered.

## Reproduction Assessment

Yes, at source level. Current main groups hover and active CSS while the command palette emits an active class, `aria-selected`, and mouse/keyboard active-index updates; I did not run a browser reproduction.

## Solution Assessment

No, not as written. The CSS-only boundary is narrow, but active styling needs matching or higher cascade precedence than hover and the PR needs inspectable dashboard proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let active rows override hover:** `ui/src/styles/components.css:3539-3553`
  - body: The new hover selectors are more specific than the active selectors, including in the light-theme override. When a selected command row is under the pointer, it renders the hover color instead of the keyboard-active color, so the active state can disappear; add an active-hover rule or otherwise give active equal or higher precedence after the hover rules.
  - confidence: 0.9
