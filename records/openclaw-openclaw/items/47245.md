## Summary

Keep open: the current-main issue is source-reproducible and the PR is focused, but it still needs inspectable real UI proof and a small CSS cascade fix because default/dark active styling can be masked by hover.

## What This Changes

The PR changes command palette CSS so hover and keyboard-active rows use theme-aware, separate `color-mix` backgrounds.

## Best Possible Solution

Land a focused CSS update where hover and active are distinct in all themes and active wins when both states match, backed by redacted Control UI screenshot or recording proof.

## Reproduction Assessment

Yes, at source level. Current main gives hover and active rows the same background, and the renderer applies `cmd-palette__item--active` while supporting pointer hover and keyboard navigation; I did not run the browser UI.

## Solution Assessment

No. Splitting the colors is the right direction, but the default/dark active selector needs to outrank hover and the PR needs inspectable real UI proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Let active rows override hover:** `ui/src/styles/components.css:3539-3545`
  - body: In the default/dark theme, `.cmd-palette__item:hover` has higher specificity than `.cmd-palette__item--active`, so when the selected row is also under the pointer the hover background wins and the active state is hidden. Add an active-hover rule or give active equal/higher specificity after hover so keyboard selection remains visible in the mixed mouse/keyboard state.
  - confidence: 0.89
