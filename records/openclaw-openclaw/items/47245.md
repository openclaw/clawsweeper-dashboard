## Summary

Keep open: current main still has the command palette hover/active ambiguity, so this PR is not obsolete, but the branch should not merge yet because active rows can still render as hover when both selectors apply and the external PR has no inspectable after-fix behavior proof.

## What This Changes

The PR changes `ui/src/styles/components.css` so command palette hover rows and keyboard-active rows use separate `color-mix` background colors.

## Best Possible Solution

Land a focused CSS update where hover and active are visually distinct and active wins when both states match, backed by redacted Control UI proof showing the command palette states.

## Reproduction Assessment

Yes, at source level. Current main gives hover and active rows the same background, and the command palette renders `.cmd-palette__item--active` while also supporting pointer hover and keyboard navigation.

## Solution Assessment

No, not as written. The CSS-only boundary is appropriate, but the active selector needs equal or higher cascade precedence than hover and the PR needs inspectable dashboard proof before merge.
