## Summary

Keep open. The PR targets a real source-reproducible Control UI ambiguity, but the proposed CSS still lets hover override active when both states apply, and external real-behavior proof is absent.

## What This Changes

This PR changes `ui/src/styles/components.css` to give command palette hover and keyboard-active rows separate `color-mix` backgrounds, with light-theme overrides.

## Best Possible Solution

Land a focused CSS update where active wins over hover in all themes, then attach redacted Control UI screenshot or recording proof for light and dark mode.

## Reproduction Assessment

Yes, at source level. Current main gives hover and active rows the same background, and the renderer can apply active selection while pointer hover is present; I did not run the browser UI.

## Solution Assessment

No. Splitting the colors is the right direction, but the selector specificity must preserve the active state when hover and active coincide, and the PR still needs inspectable UI proof.
