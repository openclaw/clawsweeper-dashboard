## Summary

Keep open: current main still has the command palette hover/active ambiguity, but this PR still needs a CSS specificity fix and inspectable after-fix real behavior proof before merge.

## What This Changes

The PR splits the Control UI command palette hover and active background styling in `ui/src/styles/components.css` into separate theme-aware `color-mix` rules.

## Best Possible Solution

Land a focused CSS update where hover and active are visually distinct, active wins when both states apply, and the PR body includes redacted dashboard proof.

## Reproduction Assessment

Yes, at source level. Current main groups hover and active under one background rule while the command palette applies `.cmd-palette__item--active`, `aria-selected`, keyboard navigation, and mouse-enter active-index updates; I did not run the dashboard.

## Solution Assessment

No, not as written. The CSS-only boundary is appropriate, but active rows need equal or higher cascade precedence after hover, plus real dashboard proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let active command rows override hover:** `ui/src/styles/components.css:3539-3553`
