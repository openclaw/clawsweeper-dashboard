## Summary

Keep open: current main still uses the same command palette background for hover and active rows, and this focused CSS PR is still useful, but the current diff lets hover override active on selected-hovered rows and lacks inspectable real dashboard proof.

## What This Changes

The PR changes one Control UI command palette CSS rule so hover and active items use separate theme-aware background colors.

## Best Possible Solution

Land a focused CSS update that keeps hover and keyboard-active rows distinct in all theme modes, preserves active styling when a row is also hovered, and includes redacted dashboard proof.

## Reproduction Assessment

Yes, at source level. Current main gives hover and active rows the same background, while the command palette view marks selected rows with `.cmd-palette__item--active` and changes active state during mouse/keyboard interaction.

## Solution Assessment

No, not as written. The CSS-only boundary is appropriate, but selector specificity lets hover win over active on selected-hovered rows; add an active-hover rule or matching specificity after the hover rules, then provide real dashboard proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let active rows override hover:** `ui/src/styles/components.css:3536-3541`
  - body: In both the default and light-theme rules, the hover selector is more specific than the active selector, so a row that is both selected and hovered renders the hover color instead of the keyboard-active color. Add an active-hover rule or use matching specificity after the hover rules so selection remains distinguishable.
  - confidence: 0.9
