## Summary

Keep open. The CSS-only goal is valid, but current main still has the original shared hover/active background and this PR introduces a cascade bug where hover styling wins when an item is both hovered and keyboard-active; it also needs the required changelog entry.

## What This Changes

The PR splits command palette item hover and keyboard-active backgrounds in `ui/src/styles/components.css` into separate theme-aware `color-mix` rules.

## Best Possible Solution

Land a focused CSS update that keeps hover and keyboard-active rows distinct in both themes, gives the active state intentional precedence when states overlap, and records the user-facing UI fix in the changelog.

## Reproduction Assessment

Yes. Current main reproduces the original ambiguity from source, and the PR regression reproduces by applying both `:hover` and `.cmd-palette__item--active` to the same row and following CSS specificity.

## Solution Assessment

No, not as written. The CSS-only boundary is the right maintainable approach, but the active selectors need equal-or-higher specificity or an explicit active-hover rule before the patch preserves keyboard selection under pointer hover.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Make active styling win over hover:** `ui/src/styles/components.css:3536-3549`
  - body: The added hover selectors are more specific than the active selectors, so an item that is both `:hover` and `.cmd-palette__item--active` renders the hover color instead of the selected color. During mixed pointer/keyboard use this can mask the keyboard-selected state; add an active-hover rule or match the active selector specificity.
