## Summary

Keep open: the PR targets a real Control UI ambiguity that is still present on current main, but it is not merge-ready because the active selector can lose to hover on overlap and the PR still lacks inspectable real UI proof.

## What This Changes

The PR splits the command palette hover and active row backgrounds into theme-aware `color-mix` rules in `ui/src/styles/components.css`.

## Best Possible Solution

Land a focused CSS update that gives hover and active rows distinct colors in both themes, ensures active styling wins when both selectors match, and includes redacted light/dark Control UI proof.

## Reproduction Assessment

Yes, at source level. Current main uses one shared hover/active background rule, and the renderer applies active selection while hover can also match; I did not run the browser UI in this read-only review.

## Solution Assessment

No. Splitting the colors is the right direction, but this branch's hover selectors have higher specificity than the active selectors and no artifact proves the real light/dark UI contrast.
