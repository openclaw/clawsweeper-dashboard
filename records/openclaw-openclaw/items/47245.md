## Summary

Keep this PR open: the underlying UI bug is still present on current main, but the branch is not merge-ready because it lacks inspectable real UI proof and the proposed hover selectors can still override active styling when both states match.

## What This Changes

The PR splits the command palette row hover and keyboard-active CSS background rules into separate theme-aware `color-mix(...)` styles.

## Best Possible Solution

Land a focused CSS update that keeps hover and active colors distinct in default and light themes, makes active styling win on overlap, and includes redacted light/dark Control UI proof.

## Reproduction Assessment

Yes, at source level. Current main has one shared hover/active background rule, and the command palette renderer applies active selection while rows can also match hover; I did not run the browser UI in this read-only review.

## Solution Assessment

No. Splitting the colors is the right direction, but this branch's hover selectors outrank active selectors when both apply and there is no artifact proving actual light/dark UI contrast.

## Review Findings
