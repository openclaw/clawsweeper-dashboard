## Summary

Keep open. The current-main ambiguity is real, but this PR still lets hover mask the selected-row styling in the default theme path and the external PR has no inspectable real dashboard proof.

## What This Changes

The PR changes `ui/src/styles/components.css` to split dashboard command palette hover and keyboard-active row backgrounds into theme-aware CSS rules.

## Best Possible Solution

Land a focused CSS update that keeps hover and keyboard-active rows distinct in both themes and preserves selected-row styling when the selected row is also hovered.

## Reproduction Assessment

Yes, at source level. Current main applies the same `var(--bg-hover)` rule to hover and active rows, while the view uses `activeIndex` for `.cmd-palette__item--active` and also updates that state on mouse enter.

## Solution Assessment

No, not as written. The CSS-only boundary is right, but the default hover selector has higher specificity than the active selector and the PR still needs inspectable real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let selected rows override hover:** `ui/src/styles/components.css:3536-3541`
  - body: In the default theme path, `.cmd-palette__item:hover` has higher specificity than `.cmd-palette__item--active`, so a row that is both hovered and selected renders the hover color instead of the keyboard-active color. Add an active-hover rule or match active specificity after the hover rules so selection remains distinguishable.
  - confidence: 0.9
