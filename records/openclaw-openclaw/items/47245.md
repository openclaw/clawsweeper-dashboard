## Summary

Keep open: the CSS-only fix targets a real source-visible command palette ambiguity, but it still lets hover override active styling when both states coincide and lacks inspectable real behavior proof.

## What This Changes

This PR splits the command palette hover and active row background rules in `ui/src/styles/components.css` into theme-aware `color-mix` values.

## Best Possible Solution

Land a focused CSS update where active rows win over hover in default and light themes, with redacted light/dark command palette screenshot or recording proof.

## Reproduction Assessment

Yes, at source level. Current main uses one shared CSS rule for hover and active rows, and the command palette renderer can have active selection and pointer hover in the same interaction path; I did not run the browser UI in this read-only review.

## Solution Assessment

No. Splitting the colors is the right direction, but the selector specificity must preserve active styling when hover and active coincide, and the PR still needs inspectable UI proof.
