## What This Changes

The PR splits command palette hover and keyboard-active row backgrounds in `ui/src/styles/components.css` into separate theme-aware `color-mix(...)` rules.

## Best Possible Solution

Land a focused CSS update that keeps hover and active colors distinct in default and light themes, makes active styling win on overlap, and includes redacted light/dark Control UI proof.

## Reproduction Assessment

Yes, at source level. Current main has one shared hover/active background rule, and the command palette renderer can apply active selection while rows also match hover; I did not run the browser UI in this read-only review.

## Solution Assessment

No. Splitting the colors is the right direction, but this branch lets hover outrank active styling when both apply and does not include inspectable light/dark UI proof.
