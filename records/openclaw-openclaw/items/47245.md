## Summary

Keep open: current main still has the command palette hover/active ambiguity and the PR is scoped to the right CSS surface, but the branch needs an active-over-hover cascade fix plus inspectable real behavior proof before merge.

## What This Changes

The PR splits the Control UI command palette hover and active row backgrounds into separate theme-aware `color-mix` CSS rules in `ui/src/styles/components.css`.

## Best Possible Solution

Land a focused CSS update where hover and active rows are visually distinct across themes and the active selected row wins in the cascade, backed by redacted Control UI proof showing the states.

## Reproduction Assessment

Yes, at source level. Current main uses the same background rule for hover and active rows, while the command palette applies `.cmd-palette__item--active` and updates active selection through keyboard and mouse interaction.

## Solution Assessment

No, not as written. The CSS-only boundary is appropriate, but the active state needs equal or higher cascade precedence after hover, and the PR still needs real dashboard proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let active rows override hover:** `ui/src/styles/components.css:3539-3553`
  - body: The new hover selector is more specific than `.cmd-palette__item--active`, and the light-theme hover override is more specific too. When a selected command row is under the pointer, it renders the hover color instead of the keyboard-active color, so the active state can disappear; add an active-hover rule or a matching active selector after the hover rules.
