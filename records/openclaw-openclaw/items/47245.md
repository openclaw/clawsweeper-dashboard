## Summary

Keep this PR open. Current main and the latest release still use the same background for command palette hover and active rows, so the PR targets a real UI gap, but the patch is not merge-ready because the new hover selectors override the active selectors when a row is both hovered and keyboard-selected; the user-facing CSS fix also needs a changelog entry.

## What This Changes

The PR changes `ui/src/styles/components.css` to split command palette item hover and keyboard-active backgrounds into separate theme-aware `color-mix` rules.

## Best Possible Solution

Land a focused CSS update that keeps command palette hover and keyboard-active rows visually distinct in both dark and light themes, gives the active state intentional precedence when active and hovered overlap, and records the user-facing UI fix in the changelog without changing palette logic or layout.

## Reproduction Assessment

Yes. Current main reproduces the original ambiguity from source because hover and active share the same background rule; the PR defect reproduces by applying both `:hover` and `.cmd-palette__item--active` to one row and following CSS specificity.

## Solution Assessment

No, not as written. The CSS-only boundary is the right maintainable approach, but the active selectors need equal-or-higher specificity, or an explicit active-hover rule, before the patch preserves keyboard selection under pointer hover.
