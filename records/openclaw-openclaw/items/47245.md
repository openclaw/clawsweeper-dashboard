## Summary

Keep this PR open. Current main still uses the same command-palette background for hover and keyboard-active rows, but the proposed CSS needs one more pass because the new hover selectors are more specific than the active selectors and can hide the selected state when a row is both hovered and active.

## What This Changes

The PR changes `ui/src/styles/components.css` to split command palette item hover and active backgrounds into separate theme-aware `color-mix` rules.

## Best Possible Solution

Land a focused CSS update that keeps hover and keyboard-active command palette rows visually distinct in both themes and defines an intentional active-plus-hover state, without changing palette logic, layout, icons, or other UI surfaces.

## Reproduction Assessment

Yes. Current main reproduces the original issue from source because hover and active share `var(--bg-hover)`, and the PR defect reproduces by applying both `:hover` and `.cmd-palette__item--active` to the same row and following CSS specificity.

## Solution Assessment

No, not as written. The CSS-only boundary is the right maintainable approach, but the patch needs an active-hover rule or matched specificity so keyboard selection is not hidden by pointer hover.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make active styling win over hover:** `ui/src/styles/components.css:3536-3549`
  - body: The added hover selectors are more specific than the active selectors, so an item that is both `:hover` and `.cmd-palette__item--active` renders the hover color instead of the selected color. During mixed pointer/keyboard use this can mask the keyboard-selected state; add an active-hover rule or match the active selector specificity.
  - confidence: 0.87
