## Summary

Keep open: the CSS-only direction is plausible, but the current diff lets hover override the active selection when the two states overlap, and the external PR still lacks inspectable after-fix proof.

## What This Changes

This PR splits command palette hover and active-row backgrounds in `ui/src/styles/components.css` with theme-specific `color-mix()` rules.

## Best Possible Solution

Land a focused CSS update that keeps hover and keyboard-active rows distinct in light and dark themes while ensuring active selection wins when the states overlap.

## Reproduction Assessment

Yes, at source level. Current main shares one hover/active background, and the PR diff shows higher-specificity hover selectors overriding active styling when a row is both hovered and selected.

## Solution Assessment

No, not as written. The CSS-only boundary is right, but the active selectors need equal-or-higher specificity or explicit active-hover rules, plus inspectable real behavior proof before merge.
