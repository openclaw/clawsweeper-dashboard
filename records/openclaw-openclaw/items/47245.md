## Summary

Keep open: the PR targets a real source-reproducible Control UI issue, but the current patch still lets hover styling override active styling when both apply, and the external real-behavior proof gate is still missing.

## What This Changes

The PR splits the Control UI command palette hover and active background rule into separate theme-aware CSS `color-mix` rules in `ui/src/styles/components.css`.

## Best Possible Solution

Land a focused CSS update that gives active rows equal or higher cascade precedence than hover in all theme modes, then include redacted dashboard proof showing the command palette states.

## Reproduction Assessment

Yes, at source level. Current main gives hover and active rows the same background, while the command palette view marks selected rows with `.cmd-palette__item--active` and updates active state during mouse and keyboard interaction.

## Solution Assessment

No, not as written. The CSS-only boundary is appropriate, but the active state needs matching or higher specificity after hover so selected rows stay visually selected when the pointer is also over them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let active rows override hover:** `ui/src/styles/components.css:3539-3553`
  - body: The new hover selector is more specific than `.cmd-palette__item--active`, and the light-theme hover override is more specific too. When a selected command row is under the pointer, it renders the hover color instead of the keyboard-active color, so the active state can disappear. Add an active-hover rule or an active selector with matching specificity after the hover rules.
  - confidence: 0.9
