## Summary

Keep this PR open: the current-main ambiguity is real and the CSS-only boundary is appropriate, but the proposed selectors still let hover override the selected row and the external PR lacks inspectable real behavior proof.

## What This Changes

The PR splits dashboard command palette hover and active-row backgrounds in `ui/src/styles/components.css` into separate theme-aware `color-mix` rules.

## Best Possible Solution

Land a focused CSS update that keeps hover and keyboard-active rows distinct in both themes, preserves active-row styling when the selected row is hovered, and includes redacted dashboard proof.

## Reproduction Assessment

Yes, at source level. Current main gives hover and active rows the same background, while the command palette view marks active rows and updates active state from mouse movement.

## Solution Assessment

No, not as written. The CSS-only fix boundary is right, but the selector specificity should be adjusted so active rows override hover, and the PR still needs inspectable real dashboard proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let active rows override hover:** `ui/src/styles/components.css:3536-3541`
  - body: In both the default and light-theme rules, the hover selector is more specific than the active selector, so a row that is both selected and hovered renders the hover color instead of the keyboard-active color. Add an active-hover rule or use matching specificity after the hover rules so selection remains distinguishable.
