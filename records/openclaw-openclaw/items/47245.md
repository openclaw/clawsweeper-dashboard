## Summary

Keep open: the PR still targets a real current-main command-palette ambiguity, but it remains blocked by a CSS cascade bug and missing inspectable real UI proof.

## What This Changes

The PR changes `ui/src/styles/components.css` to split command palette hover and keyboard-active background colors with light-mode overrides.

## Best Possible Solution

Land a focused CSS update where hover and active are distinct in light and dark themes, active wins when both states match, and redacted Control UI screenshot or recording proof shows the result.

## Reproduction Assessment

Yes, at source level. Current main gives hover and active rows the same background, and the renderer applies `cmd-palette__item--active` while supporting pointer hover and keyboard navigation; I did not run the browser UI.

## Solution Assessment

No. Splitting the colors is the right direction, but the default active selector needs to outrank hover and the PR needs inspectable real UI proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Let active rows override hover:** `ui/src/styles/components.css:3539-3545`
  - body: In the default/dark theme, `.cmd-palette__item:hover` has higher specificity than `.cmd-palette__item--active`, so when the selected row is also under the pointer the hover background wins and hides the active state. Add an active-hover rule or give active equal/higher specificity after hover so keyboard selection remains visible in the mixed mouse/keyboard state.
  - confidence: 0.89
