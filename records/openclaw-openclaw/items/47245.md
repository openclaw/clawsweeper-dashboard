## Summary

Keep open: current main still uses one shared command-palette hover/active background, and the PR addresses a real UI ambiguity, but the proposed CSS lets hover override active selection when both states overlap and the external PR lacks inspectable real behavior proof.

## What This Changes

The PR changes `ui/src/styles/components.css` to split command palette hover and active-row backgrounds with theme-aware `color-mix()` rules.

## Best Possible Solution

Land a focused CSS update that keeps hover and keyboard-active rows distinct in both themes while making active selection win when hover overlaps it.

## Reproduction Assessment

Yes. Source inspection shows current main has one shared hover/active background, and the PR regression follows from CSS specificity when a command palette row is both hovered and active.

## Solution Assessment

No, not as written. The CSS-only boundary is right, but the active selectors need equal-or-higher specificity or explicit active-hover rules before this is the narrow maintainable fix.
