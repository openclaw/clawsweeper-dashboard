## Summary

This PR should stay open because it addresses a real current-main command palette UI ambiguity, but the branch still needs real behavior proof and its CSS selector specificity leaves part of the intended mixed mouse/keyboard behavior unresolved.

## What This Changes

The PR splits command palette hover and active row background styling in `ui/src/styles/components.css` into separate theme-aware `color-mix` rules.

## Best Possible Solution

Land a focused CSS update where hover and active colors remain distinct in light and dark themes and active styling wins on overlap, backed by redacted real UI proof.

## Reproduction Assessment

Yes, at source level. Current main has one shared hover/active background rule, and the command palette renderer can apply active selection while hover also matches; I did not run the browser UI in this read-only review.

## Solution Assessment

No, not yet. Splitting the colors is the right direction, but this branch's hover selectors can outrank active selectors when both apply and there is no artifact proving actual light/dark UI contrast.
