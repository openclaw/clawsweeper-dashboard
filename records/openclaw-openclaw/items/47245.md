## Summary

Keep open: this is a useful narrow Control UI fix, but the PR still lacks inspectable real behavior proof and the CSS selector order/specificity can let hover styling override the keyboard-active row.

## What This Changes

Splits the command palette hover and active CSS backgrounds into separate theme-aware `color-mix` rules in `ui/src/styles/components.css`.

## Best Possible Solution

Land a focused CSS update where the active row wins over hover in default and light themes, then attach redacted light/dark Control UI proof showing the command palette states.

## Reproduction Assessment

Yes, at source level. Current main uses one shared CSS rule for hover and active rows, and the command palette source applies active selection while pointer hover can also be present; I did not run the browser UI in this read-only review.

## Solution Assessment

No. Splitting the colors is the right direction, but the selector specificity must preserve active styling when hover and active coincide, and the PR still needs inspectable UI proof.
