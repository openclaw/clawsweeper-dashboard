## Summary

Keep open: current main still has the command palette state ambiguity, but this PR is blocked because active rows lose to the hover selector when both states apply and the external PR lacks inspectable after-fix proof.

## What This Changes

The PR changes Control UI command palette CSS so hover and keyboard-active rows use different theme-aware `color-mix` backgrounds.

## Best Possible Solution

Land a focused CSS update where hover and active are visually distinct in both themes and active wins when both selectors match, backed by redacted dashboard proof.

## Reproduction Assessment

Yes, at source level. Current main uses the same background rule for hover and active rows while the command palette applies `.cmd-palette__item--active` and updates active selection through keyboard and mouse interaction.

## Solution Assessment

No, not as written. The CSS-only boundary is appropriate, but the patch needs a specificity-safe active-hover rule or equivalent so active styling wins, plus real dashboard proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let active rows override hover:** `ui/src/styles/components.css:3539-3553`
  - body: The new `.cmd-palette__item:hover` selector is more specific than `.cmd-palette__item--active`, and the light-theme hover override is also more specific than the light active rule. When the selected command row is under the pointer, it renders the hover color instead of the keyboard-active color, so the active state can disappear; add an active-hover rule or matching-specificity active selector after the hover rules.
  - confidence: 0.9
