## Summary

Keep open: the patch is not mergeable because it changes final-mode WebChat TTS from one accumulated audio pass into per-block synthesis plus the existing tail pass, and it promotes an internal local-media trust flag into the public plugin SDK contract. The supplied terminal proof is useful, but it does not exercise the real WebChat dispatch path that carries this risk.

## What This Changes

The PR changes speech-core TTS to synthesize block replies in final mode, mark synthesized payloads as trusted local media, expose `trustedLocalMedia` through `ReplyPayload`, add focused tests, and add a WebChat auto-TTS repro script.

## Best Possible Solution

Preserve final-mode semantics by routing one accumulated WebChat TTS audio payload with the internal trust flag, keep per-block synthesis behind `mode: "all"`, and keep `trustedLocalMedia` out of the public SDK.

## Reproduction Assessment

Yes, source-reproducible for the blocking PR defect: `dispatch-from-config` calls TTS for `kind: "block"` and later calls TTS again on accumulated block text as `kind: "final"` when no final reply exists. Allowing block through final mode can therefore duplicate synthesis and playback.

## Solution Assessment

No, not as written. The narrower maintainable fix is to carry the internal trust flag on the accumulated final/audio-only WebChat TTS payload and add dispatch-level coverage, not to treat block replies as final-mode TTS inputs or publish the trust marker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep block TTS out of final mode:** `extensions/speech-core/src/tts.ts:1762`
  - body: Allowing `kind: "block"` through this final-mode gate makes every streamed block eligible for TTS, while `dispatch-from-config` still performs a separate accumulated `kind: "final"` TTS pass when block streaming returns no final reply. That can turn final-mode WebChat streams into per-block provider calls plus the tail audio payload instead of one final audio result.
  - confidence: 0.9
- **[P2] Keep the trust marker out of the public SDK type:** `src/plugin-sdk/reply-payload.ts:9`
  - body: `trustedLocalMedia` is documented as the internal trust signal for WebChat local audio embedding, and current main intentionally omits it from the public `ReplyPayload` type. Exporting `InternalReplyPayload` here makes that local-file trust marker plugin-facing; the generated TTS path should add it through an internal/core seam instead.
  - confidence: 0.86
