## Summary

Keep this PR open, but not mergeable as written: the linked WebChat auto-TTS bug is real, yet the patch routes final-mode block streams through per-block synthesis and broadens an internal local-media trust marker into the public plugin SDK contract.

## What This Changes

The PR lets speech-core TTS run for block replies, marks successful TTS output as trusted local media, exposes that field through the plugin SDK type, adds focused tests, and adds a WebChat auto-TTS repro script.

## Best Possible Solution

Preserve final-mode semantics by producing one accumulated audio-only WebChat TTS payload that carries the internal trust flag, keep per-block TTS behind mode "all", and keep trustedLocalMedia out of the public plugin SDK.

## Reproduction Assessment

Yes for the blocking PR defect from source inspection: onBlockReply calls TTS with kind "block", and the no-final-reply tail path still synthesizes accumulated block text as kind "final". Allowing block through the final-mode gate therefore creates per-block synthesis plus the existing tail synthesis path.

## Solution Assessment

No, not as written. The narrower fix is to preserve trustedLocalMedia on the accumulated final/audio-only WebChat TTS payload and add dispatch-level coverage, not to treat block replies as final-mode TTS inputs or publish the internal trust marker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep final-mode block TTS on the accumulated final pass:** `extensions/speech-core/src/tts.ts:1762`
  - body: Allowing `kind: "block"` through this final-mode gate makes every streamed block synthesize TTS, but `dispatch-from-config` already performs a separate accumulated `kind: "final"` TTS pass when block streaming returns no final reply. That turns final-mode WebChat streams into per-block provider calls plus the tail synthesis path, causing duplicate audio/cost instead of one final audio payload.
  - confidence: 0.9
- **[P2] Keep trustedLocalMedia out of the public SDK type:** `src/plugin-sdk/reply-payload.ts:9`
  - body: `trustedLocalMedia` is documented in the internal reply payload as the local-file trust signal for WebChat embedding, and the current plugin SDK intentionally omits it while normalizers strip it from loose plugin payloads. Exporting `InternalReplyPayload` here makes that internal trust marker part of the public plugin-facing type; use an internal or bundled-TTS-only seam for generated TTS audio instead.
  - confidence: 0.86
