## Summary

Keep this PR open: the revised branch removes the duplicate per-block TTS behavior, but it still makes the internal WebChat local-media trust flag part of the public plugin SDK type, and the supplied proof does not exercise the actual dispatch-from-config path changed by the PR.

## What This Changes

The PR marks speech-core TTS output as trusted local media, forwards that flag on the accumulated block-stream TTS-only final payload, changes the public plugin SDK ReplyPayload type, adds tests, and adds a WebChat auto-TTS repro script.

## Best Possible Solution

Preserve the internal trust boundary, keep final-mode block TTS on the accumulated final tail, and pass the local TTS trust flag through an internal or bundled-TTS-only seam into the WebChat final payload.

## Reproduction Assessment

Yes from source inspection: current main creates local TTS media but WebChat ignores local audio unless trustedLocalMedia is true, and the accumulated block TTS-only final currently drops that flag. I did not run a live repro in this read-only review.

## Solution Assessment

No, not as written. The revised direction is mostly right, but changing public ReplyPayload to InternalReplyPayload is broader than needed; the safer fix keeps the trust marker internal while forwarding it on the TTS-generated WebChat payload.
