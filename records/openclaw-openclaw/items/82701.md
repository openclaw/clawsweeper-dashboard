## Summary

Keep this PR open, but the current patch should not merge: it addresses a real WebChat Auto-TTS gap, yet it changes final-mode block handling in a way that can produce duplicate per-block plus accumulated TTS, and it exposes an internal local-media trust flag through the public plugin SDK.

## What This Changes

The PR changes speech-core TTS eligibility for block replies, marks synthesized TTS media as trusted local media, widens the plugin SDK ReplyPayload type, and adds focused tests plus a repro script.

## Best Possible Solution

Preserve final-mode semantics by generating one accumulated WebChat TTS payload after streaming, carry trustedLocalMedia through that internal audio-only path, and keep the public SDK ReplyPayload omitting the trust marker.

## Reproduction Assessment

Yes. Source inspection shows current WebChat block streaming already has both per-block TTS calls and an accumulated final TTS path; applying this PR would let final mode synthesize both paths, while current main also shows the original trustedLocalMedia drop at the WebChat media gate.

## Solution Assessment

No. The narrower maintainable fix is to preserve the internal trust flag on the accumulated final/audio-only WebChat TTS payload and add dispatch-level coverage, not to treat block replies as final-mode TTS inputs or publish the trust marker in the SDK.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep final-mode block TTS on the accumulated final pass:** `extensions/speech-core/src/tts.ts:1762`
  - body: Allowing `kind: "block"` through this final-mode gate makes TTS run for every streamed block, but `dispatch-from-config` still performs a separate accumulated `kind: "final"` TTS pass when block streaming returns no final reply. That turns final-mode WebChat streams into per-block provider calls plus the tail synthesis path, causing duplicate audio/cost instead of one final audio payload.
  - confidence: 0.9
- **[P2] Keep trustedLocalMedia out of the public SDK type:** `src/plugin-sdk/reply-payload.ts:9`
  - body: `trustedLocalMedia` is documented as the internal WebChat local-file trust signal, and current plugin SDK types intentionally omit it while normalizers strip it from loose plugin payloads. Exporting `InternalReplyPayload` here promotes that internal trust marker into the public plugin-facing contract; use an internal or bundled-TTS-only seam for generated TTS audio instead.
  - confidence: 0.86
