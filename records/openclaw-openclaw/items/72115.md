## Summary

Keep open. The Skill Workshop hardening direction is useful and current main does not already provide it, but the PR is not merge-ready: the branch still has a concrete bundled override normalization bug, introduces a security-sensitive proxy fallback that can report proxy activation without Node HTTP hooks, and GitHub reports the PR as dirty against main.

## What This Changes

The PR hardens Skill Workshop workspace skill writes with realpath containment, write-boundary revalidation, prompt-budget parity, SDK exports, docs, tests, and related plugin loader, proxy, QA, and CI support updates.

## Best Possible Solution

Land the Skill Workshop hardening after the bundled override comparison uses the same user-path and realpath normalization as bundled-dir resolution, proxy activation fails closed or gates any hookless fallback to an explicit test-only path, and the branch is made mergeable against current main; maintainers should also decide whether the broad CI/QA support changes stay in this PR or split out.

## Reproduction Assessment

Yes for the PR findings. Use a home-relative `OPENCLAW_BUNDLED_PLUGINS_DIR` such as `~/bundled` to exercise the loader mismatch, and simulate an unavailable `global-agent` module with proxy enabled to verify `startProxy` can still return an active handle without Node HTTP hooks.

## Solution Assessment

No, not yet. The Skill Workshop hardening approach fits the existing plugin-owned writer and SDK seam model, but the loader normalization and proxy fail-open behavior should be fixed before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize bundled override paths before comparing:** `src/plugin-sdk/facade-loader.ts:149-156`
  - body: `resolveBundledPluginsDir` expands `OPENCLAW_BUNDLED_PLUGINS_DIR` with `resolveUserPath`, but this new comparison uses `path.resolve` on the raw value. A valid override such as `~/bundled` can be treated as outside the override and pass `installRuntimeDeps: false`, skipping runtime dependency staging for the bundled public surface.
  - confidence: 0.88
- **[P2] Fail closed when proxy HTTP hooks cannot load:** `src/infra/net/proxy/proxy-lifecycle.ts:255-258`
  - body: When `global-agent` cannot be loaded, `bootstrapNodeHttpStack` now logs and returns, but `startProxy` still registers the proxy as active. In that runtime, Node `http`/`https` callers that rely on those hooks can bypass the operator-managed proxy while `OPENCLAW_PROXY_ACTIVE` is set, so activation should throw or the fallback should be explicitly test-only.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff is security-sensitive and needs attention because proxy activation can now continue after the Node HTTP proxy hook dependency is unavailable.

Concerns:

- **[medium] Proxy activation can lose Node HTTP hooks:** `src/infra/net/proxy/proxy-lifecycle.ts:255`
  - body: The PR changes `global-agent` from a required static import to an optional resolver. If it is unavailable, `bootstrapNodeHttpStack` returns after a warning and `startProxy` still registers an active proxy, which can leave Node `http`/`https` traffic outside the configured filtering proxy.
