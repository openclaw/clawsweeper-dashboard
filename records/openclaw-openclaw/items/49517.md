## What This Changes

The issue requests that `/new` create a new session without resetting the old one, while `/reset` remains the explicit reset command.

## Best Possible Solution

Keep the shipped Control UI create-and-switch contract, and track broader channel or routing-mode reset behavior only in dedicated issues with their own reproduction paths.

## Reproduction Assessment

No. Current main no longer has the reported Control UI failure path: typed `/new` dispatches to the dashboard `new-session` create/switch action, while `/reset` still uses the explicit Gateway reset path, and regression tests lock that split.

## Solution Assessment

Yes. Reusing the existing New Chat dashboard-session path for typed `/new` is the narrow maintainable solution, and keeping `/reset` on the in-place reset path preserves the documented Gateway boundary.
