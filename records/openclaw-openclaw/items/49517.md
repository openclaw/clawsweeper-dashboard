## Summary

Keep open. Current main still treats `/new` as a reset-style rollover in the auto-reply and gateway paths, the Control UI still warns that it resets the current chat, and docs/tests continue to describe or assert reset semantics; only the TUI has a narrower unique-session-key behavior. The related implementation PR was closed unmerged, and the later Telegram provider-error report covered a different failure mode.

## What This Changes

Request that `/new` create or switch to a fresh session while preserving the existing session, leaving `/reset` as the only destructive reset command.

## Best Possible Solution

Define the cross-surface session contract first, then implement `/new` as a non-destructive create-and-switch flow that preserves the old session as an accessible session, likely building on `sessions.create` plus session selection/history UI. Keep `/reset` as the destructive reset path and update gateway/chat handling, channel or ACP binding behavior, Control UI copy, docs, and regression tests together.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: bare `/new` matches the reset command handling, calls the same gateway reset service with reason `new`, and existing gateway tests assert it resets the same session.

## Solution Assessment

Unclear. The requested behavior is plausible and partly present in the TUI, but applying it globally is not a narrow drop-in because current shipped docs, hooks, ACP-bound sessions, UI copy, and tests treat `/new` as reset-style rollover; maintainers should choose the intended product contract before implementation.
