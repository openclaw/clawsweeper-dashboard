## Summary

Keep open. Current main has moved typed Control UI `/new` and TUI `/new` toward distinct fresh-session behavior, but Gateway `chat.send` and auto-reply/channel handling still treat `/new` as reset-style rollover, with docs and tests encoding mixed semantics. The closed related PR did not land, and #76957 tracks a narrower hook regression after the Control UI change rather than superseding this contract question.

## What This Changes

Request to make `/new` create or switch to a fresh session while preserving the old session, leaving `/reset` as the only destructive reset command.

## Best Possible Solution

Define one cross-surface command contract, then implement `/new` as a non-destructive create-and-switch flow that preserves accessible old sessions while keeping documented `command:new`, `before_reset`, and session lifecycle hooks intact; keep `/reset` as the explicit destructive or in-place reset path.

## Reproduction Assessment

Yes. Source inspection on current main shows Gateway bare `/new` still enters the same reset service as `/reset`, and auto-reply reset triggers still roll over the same `sessionKey`; current tests also assert reset-style behavior. I did not run tests because this was a read-only review.

## Solution Assessment

Unclear. The requested behavior is plausible and partly implemented in Control UI/TUI, but applying it globally is not a narrow drop-in while Gateway reset hooks, ACP/channel bindings, and docs/tests still rely on reset-style semantics.
