## Summary

Current main implements the reported tabbed Control UI contract: typed `/new` now goes through the New Chat dashboard-session create/switch path, while `/reset` remains the explicit in-place reset; the remaining channel/Gateway reset-style semantics are documented as separate surface behavior.

## What This Changes

The issue asks for `/new` to create a separate fresh session without resetting the old one, leaving `/reset` as the reset command.

## Best Possible Solution

Keep the current documented Control UI contract and handle any broader channel/Gateway semantics as a separate product decision, not as this already-fixed webchat bug.

## Reproduction Assessment

No. Current source and regression tests show the original Control UI failure path no longer exists: typed `/new` dispatches to `new-session` and avoids `chat.send`, while `/reset` still uses `chat.send`.

## Solution Assessment

Yes. Reusing the existing New Chat dashboard-session path for typed `/new` is the narrow maintainable fix, and preserving `/reset` as the explicit Gateway reset matches the current docs and tests.
