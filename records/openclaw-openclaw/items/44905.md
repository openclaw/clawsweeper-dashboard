## Summary

Keep this open. Current main has partial protections for commentary-phase output, exact silent tokens, internal runtime scaffolding, and recognized serialized tool-call blocks, but the exact envelope-shaped strings called out here are not all covered by the current parsers and the broader audience-boundary issue remains active.

## What This Changes

The issue reports Discord delivering internal tool-call/control envelope text such as `NO_REPLY`, `to=functions.*`, `commentary`, and JSON arguments to end users.

## Best Possible Solution

Keep this open and coordinate it with the broader audience-boundary work, using typed internal/visible delivery semantics as the durable fix and narrow shared sanitizer tests as a backstop.

## Reproduction Assessment

Yes, at source level with medium confidence: current code does not show full coverage for the reported envelope-shaped strings, and the issue has recent retest comments. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

No, a universal regex filter alone is not clearly the best durable solution. The best fix is a typed internal-vs-visible delivery boundary, with conservative outbound sanitization and regression tests for malformed provider text.
