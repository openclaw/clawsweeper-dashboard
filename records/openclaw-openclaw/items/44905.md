## Summary

Keep open. Current main and v2026.5.12 cover parts of the leak family, especially exact silent-token handling and the merged commentary-phase subset, but the shared reply path still lacks a guard for the reported raw envelope-shaped traces such as `to=functions.*`, `recipient_name`, `parameters`, and short tool-argument JSON; the related fix path is still the open, dirty PR at https://github.com/openclaw/openclaw/pull/43679.

## What This Changes

The issue asks OpenClaw to prevent Discord from delivering internal control tokens, commentary/tool-routing markers, and raw tool-call JSON as visible channel messages.

## Best Possible Solution

Land or replace the open shared normalizer/sanitizer guard with a rebased fix that blocks internal envelope-shaped traces before channel delivery while preserving media, interactive, presentation, and channel-data payloads.

## Reproduction Assessment

Yes, source-reproducible. Current main routes normal final/block/tool replies through the shared normalizer, and neither that normalizer nor `sanitizeUserFacingText()` recognizes the reported raw envelope forms before delivery.

## Solution Assessment

Yes, the best fix shape is a channel-agnostic reply-normalization or sanitizer guard rather than adding Discord to plain-text HTML conversion; the active related PR targets that boundary but needs rebase, proof, and review.
