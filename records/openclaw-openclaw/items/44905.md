- `impact:security`: The issue is about exposing raw tool arguments, routing metadata, and internal control markers to end users.

## Summary

Keep open: current main and v2026.6.6 have partial protections for exact silent replies, media-carrying NO_REPLY payloads, and commentary-phase assistant messages, but the remaining Discord/envelope-shaped leak cases are not proven fixed in source. The broader audience-boundary work remains open at https://github.com/openclaw/openclaw/issues/25592, but this issue still carries useful Discord-specific sanitizer evidence.

## What This Changes

The issue asks OpenClaw to stop Discord from delivering internal control tokens, commentary/tool routing text, raw tool arguments, and tool-call envelope fields as visible channel messages.

## Best Possible Solution

Choose and implement a shared delivery-boundary sanitizer or explicit assistant audience contract that suppresses internal envelope/control text across channels, with Discord-focused regression coverage for the remaining envelope-shaped examples.

## Reproduction Assessment

Yes for a source-level reproduction: current main still lacks explicit coverage for bare envelope fields like recipient_name, parameters, and bare to=functions.* in the relevant outbound/assistant text sanitizers. I did not run a live Discord transport repro in this read-only review.

## Solution Assessment

Unclear as proposed: adding Discord to plain-text surfaces is not the best fix, and a universal sanitizer is only a plausible mitigation. The durable solution should be a shared assistant audience/delivery-boundary decision with focused sanitizer tests for the known leak shapes.
