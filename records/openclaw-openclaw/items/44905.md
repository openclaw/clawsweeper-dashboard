## Summary

Keep open. Current main and the latest shipped release cover the commentary-phase and exact silent-token subsets, but the shared reply sanitizer/normalizer still lacks a guard for raw envelope-shaped traces such as `to=functions.*`, `recipient_name`, `parameters`, and short tool-argument JSON; the linked replacement path is still an open conflicting PR.

## What This Changes

The issue asks OpenClaw to prevent Discord from delivering internal control tokens and tool-call envelope text as visible channel messages.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43679 with a rebased shared sanitizer/normalizer guard that blocks internal envelope traces before channel delivery while preserving media, interactive, presentation, and channel data payloads.

## Reproduction Assessment

Yes, source-reproducible. Current main routes replies through the shared normalizer, and neither the skip reasons nor `sanitizeUserFacingText()` recognizes the reported raw envelope forms before delivery.

## Solution Assessment

Yes, the best fix shape is a channel-agnostic reply-normalization or sanitizer guard rather than adding Discord to plain-text HTML conversion; the active related PR targets that boundary but needs rebase, proof, and review.
