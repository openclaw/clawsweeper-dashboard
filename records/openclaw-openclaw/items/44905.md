## Summary

Keep open: current main has partial mitigations for commentary-phase output and exact silent-token/media cases, but the shared outbound and Discord delivery path still lacks a guard for raw tool-routing/envelope text when it arrives as ordinary reply text.

## What This Changes

The issue reports Discord exposing internal control tokens, tool-routing text, and raw tool-call envelope content in user-visible channel messages.

## Best Possible Solution

Land a rebased shared reply/outbound sanitizer or normalizer guard that strips known internal routing/control envelopes while preserving non-text payloads, and coordinate the broader audience-boundary policy with https://github.com/openclaw/openclaw/issues/25592.

## Reproduction Assessment

No live Discord reproduction was established. Source inspection shows that a normal reply payload containing `to=functions.*` or envelope-field text would pass through current outbound and Discord sanitizers, while exact `NO_REPLY` and commentary-phase subsets are already covered.

## Solution Assessment

No, adding Discord to plain-text surfaces is not the best fix. The maintainable path is a conservative shared sanitizer/normalizer guard, with false-positive policy reviewed before landing.
