## Summary

Keep open: current main covers the commentary-phase and exact silent-token media subsets, but source inspection still shows no shared outbound or Discord guard for raw `to=functions.*`, `recipient_name`, `parameters`, or similar envelope text if it reaches delivery as ordinary reply text.

## What This Changes

The issue asks OpenClaw to prevent Discord from delivering internal tool-call traces, control tokens, commentary markers, and raw tool-call JSON to user-visible channels.

## Best Possible Solution

Add a conservative shared visible-delivery boundary for known internal routing/control envelopes, keep the existing phase and exact silent-token handling, and cover the outbound plus Discord paths with regression tests.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows that a normal reply payload containing `to=functions.*` or envelope-field text would pass through the current outbound and Discord sanitizers, while the exact `NO_REPLY` media and commentary-phase subsets are already covered.

## Solution Assessment

Unclear until maintainers choose the stripping policy. Adding Discord to `PLAIN_TEXT_SURFACES` is not the best fix; the maintainable direction is a shared, narrow visible-delivery sanitizer or audience boundary plus the existing phase-aware suppression.
