## Summary

Keep open. Current main and the latest release cover exact silent-token handling and part of the commentary leak family, but the shared reply normalization path still has no guard for the reported raw envelope-shaped traces before delivery; an open conflicting PR already targets that remaining guard.

## What This Changes

The issue reports Discord-visible leakage of internal control tokens and tool-call envelope text such as `NO_REPLY`, `to=functions.*`, `commentary`, `recipient_name`, `parameters`, and raw JSON tool arguments.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43679 with a rebased shared sanitizer/normalizer guard that blocks internal envelope traces before channel delivery while preserving media, interactive, presentation, and channel data payloads.

## Reproduction Assessment

Yes, source-reproducible. Current main routes replies through the shared normalizer, and neither the normalizer skip reasons nor `sanitizeUserFacingText()` recognizes the reported raw envelope forms before delivery.

## Solution Assessment

Yes. The maintainable fix shape is a channel-agnostic reply-normalization or sanitizer guard, not adding Discord to plain-text HTML conversion; the active related PR targets that boundary but needs rebase, proof, and review.
