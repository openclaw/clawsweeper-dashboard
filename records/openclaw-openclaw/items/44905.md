## Summary

Keep open. Current main has partial fixes for exact silent/control tokens and commentary-phase output, but the shared reply normalizer and user-facing sanitizer still lack a guard for envelope-shaped internal text such as `to=functions.*`, `recipient_name`, `parameters`, and short tool-argument JSON; https://github.com/openclaw/openclaw/pull/43679 is the active fix path for that remainder.

## What This Changes

Prevent Discord delivery from exposing internal silent/control tokens and tool-call envelope text such as `NO_REPLY`, `commentary`, `to=functions.*`, `recipient_name`, `parameters`, and short tool JSON.

## Best Possible Solution

Land a shared channel-agnostic normalizer or sanitizer guard that suppresses internal envelope traces while preserving legitimate prose examples and all non-text reply payload content.

## Reproduction Assessment

Yes, source-reproducible. The current normalizer only skips `empty`, `silent`, and `heartbeat` payloads before passing text to a sanitizer that lacks `to=functions.*`, `recipient_name`, `parameters`, and short tool-argument JSON filtering.

## Solution Assessment

Yes, the shared normalizer/sanitizer boundary is the right fix direction. A Discord-only HTML/plain-text change or send-level exact-token guard would miss the cross-channel envelope-shaped text paths, while the open PR needs rebase/proof before it can close the issue.
