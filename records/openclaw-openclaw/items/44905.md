## Summary

Keep open. Current main has fixed the exact silent-token/media gap and the commentary-phase subset, but the shared reply path still has no guard for the raw `to=functions.*`, `recipient_name`, `parameters`, and short tool-argument JSON envelope forms described in the issue.

## What This Changes

The issue asks OpenClaw to stop Discord from delivering internal control tokens, tool-routing envelopes, and raw tool-call JSON as visible channel messages.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43679 with a rebased shared normalizer or sanitizer guard that blocks internal envelope traces before any channel delivery while preserving non-text payload content.

## Reproduction Assessment

Yes, source-reproducible. Current final/block/tool replies pass through the shared normalizer, and the inspected normalizer/sanitizer still does not recognize the reported raw envelope forms.

## Solution Assessment

Yes, the maintainable direction is a channel-agnostic reply-normalization or sanitizer guard; a Discord-only plain-text conversion change would miss the broader model-output shapes.
