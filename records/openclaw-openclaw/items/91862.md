## Summary

Keep open: current main still lets an unregistered memory embedding provider throw through the status/deep probe path, and this PR is a narrow no-crash fix for that behavior. I found no blocking patch defect, but the broader linked missing-index-metadata reports may still need separate provider-routing or recovery work.

## What This Changes

The PR changes memory-core provider initialization to convert unknown embedding-provider errors into FTS-only provider state and adds regression tests for probe/status behavior.

## Best Possible Solution

Land the narrow no-crash degradation if CI accepts it, while keeping provider aliasing and missing-index-metadata recovery as separate follow-up work.

## Reproduction Assessment

Yes, by source inspection: current main passes the configured provider to `getAdapter()`, which throws for unregistered ids, and `memory status --index` reaches `probeEmbeddingAvailability()` before the sync error catch. I did not run the CLI because this review was read-only.

## Solution Assessment

Yes, for the stated no-crash behavior: handling the unknown-provider initialization error at the memory manager boundary preserves required-provider sync/search blocking while letting status and probes show an FTS-only unavailable reason. Provider aliasing or metadata recovery should remain separate fixes.
