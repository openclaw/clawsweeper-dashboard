## Summary

Keep this PR open, but it is not merge-ready: the direction targets a real Telegram Web regression, yet the patch misses the chunked final-reply renderer that normal agent replies use and has no after-fix real Telegram proof.

## What This Changes

The branch adds a `richMode` option to `markdownToTelegramHtml`, converts generated blockquote and heading tags to `<b>` on the default path, and adds `<br>` to the legacy Telegram HTML allowlist.

## Best Possible Solution

Share one legacy-safe non-rich Telegram renderer across `markdownToTelegramHtml` and chunked reply rendering, keep the rich-message path unchanged, update regression tests, and add redacted Telegram Web or live Telegram proof.

## Reproduction Assessment

Yes, from source and the linked user report: current non-rich Telegram reply rendering can produce `<blockquote>` and sends it through `parse_mode: HTML`, while the linked issue includes real Telegram Web unsupported-message evidence. I did not run live Telegram in this read-only review.

## Solution Assessment

No. The approach is a plausible mitigation, but it is too narrow because the main final-reply chunk renderer still bypasses the new legacy conversion and the `<br>` allowlist addition is not backed by the regular Bot API HTML contract.
