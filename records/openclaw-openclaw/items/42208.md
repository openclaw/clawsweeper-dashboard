## Summary

Keep open for maintainer/security handling: source inspection confirms the patch does not enforce runtime approval, misses current outbound send tool names, leaves the web_search test contract inconsistent, and lacks real behavior proof.

## What This Changes

The PR adds a security guard/helper and tests for destructive hook-session tool calls, and changes `wrapWebContent` so `web_search` results include the external-content warning.

## Best Possible Solution

Implement the boundary through the existing `before_tool_call` or trusted-policy approval path, reuse the current mutating-tool classifier, deliberately preserve break-glass semantics, update the web_search contract tests, and provide real hook-session proof.

## Reproduction Assessment

Yes. Source inspection reproduces the PR defects: the guard is added only as an unused helper/test, current dispatch already has a separate approval path, current send tools do not match the new matcher, and the current web_search test still expects no warning.

## Solution Assessment

No. A standalone security helper is not the best fix unless it is integrated into the existing approval path and aligned with OpenClaw's current mutating-tool taxonomy.
