## Summary

Keep open: the security goal is plausible, but the patch does not enforce the promised boundary because the new guard is never invoked from production tool dispatch and it duplicates narrower mutating-tool logic than current main already has.

## What This Changes

The PR adds a standalone external-content tool-call guard/test and changes `wrapWebContent` so `web_search` receives the long external-content security notice.

## Best Possible Solution

Implement the approval requirement at the existing before-tool-call/trusted-policy seam using `isMutatingToolCall`, then explicitly decide and test the `web_search` warning contract with real Gmail or webhook proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the new guard is only present in the added helper/test, while current main executes tools through `runBeforeToolCallHook`. I did not establish a live Gmail/webhook reproduction because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

No. A detached regex helper is not the best path; the maintainable fix should reuse the existing production approval seam and the central mutating-tool classifier.
