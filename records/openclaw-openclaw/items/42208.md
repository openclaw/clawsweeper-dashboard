## Summary

Keep open. The security hardening direction is relevant, but current main does not already implement the requested mandatory approval boundary and the submitted patch is not mergeable because the new guard is helper-only, under-classifies mutating tools, and relies on brittle hook-key matching.

## What This Changes

The PR adds an external-content tool-call guard helper and tests, and changes `wrapWebContent` so `web_search` receives the same security warning as `web_fetch`.

## Best Possible Solution

Implement the policy inside the existing before-tool-call enforcement path, using explicit external-content provenance plus `isMutatingToolCall`, routing mandatory approval through the established approval gateway, and preserving deliberate unsafe-content bypass semantics.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main wraps hook content before the model turn, `runBeforeToolCallHook` has no built-in external-content approval policy, and the PR adds a helper that is not invoked by production dispatch.

## Solution Assessment

No. The proposed direction is useful, but the submitted implementation is not the best fix because enforcement belongs in the existing tool-dispatch approval path with current mutation classification and preserved external-content provenance.

  - confidence: 0.89

  - body: Changing `wrapWebContent(..., "web_search")` to always include the long warning conflicts with the existing test that asserts web search wrapping does not contain `SECURITY NOTICE`, so the focused test expectation needs to change with this behavior.
