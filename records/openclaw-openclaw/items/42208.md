## Summary

Keep this PR open, but it is not mergeable as-is: the proposed hard approval boundary is a standalone helper that the runtime never calls, its destructive-tool matcher is narrower than current main’s mutating-tool taxonomy, and the external PR still lacks real hook-session proof.

## What This Changes

The PR adds a new external-content tool-call guard plus tests and changes `wrapWebContent` so `web_search` output receives the long external-content warning.

## Best Possible Solution

Implement the gate through the existing `before_tool_call` or trusted-policy approval path, reuse the shared mutating-tool classifier, update the `web_search` wrapper contract deliberately, and validate with redacted real hook-session proof.

## Reproduction Assessment

Yes for the PR defects. Source inspection of the PR diff and current main shows the new guard is only a helper/test, production dispatch already has a separate approval seam, and current tests still encode the old `web_search` warning contract.

## Solution Assessment

No. A standalone security helper is not the best way to solve this unless it is wired into the existing approval path and aligned with OpenClaw’s central mutating-tool taxonomy.
