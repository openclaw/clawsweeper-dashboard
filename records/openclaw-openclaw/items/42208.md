## Summary

Keep open. This is a plausible security hardening direction, but the current PR is not mergeable: the proposed guard has no production call path, duplicates and narrows the existing mutating-tool classifier, changes the web_search warning contract without aligned tests/docs, and lacks required real Gmail/webhook proof.

## What This Changes

The PR adds a new security helper/test for external hook sessions and changes `wrapWebContent` so `web_search` content also receives the long security notice.

## Best Possible Solution

Put the external-content approval requirement into `runBeforeToolCallHook` or a trusted tool policy using `isMutatingToolCall`, then decide and document the `web_search` warning contract explicitly.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the proposed guard is only in the PR's new module/test while current production dispatch routes through `runBeforeToolCallHook`. I did not run tests because this is a read-only review.

## Solution Assessment

No. A detached regex helper is not the maintainable fix; the safer path is a production approval-policy integration on the existing dispatch seam using the central mutating-tool classifier.
