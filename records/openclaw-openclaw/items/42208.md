## Summary

Keep this PR open for human security review, but do not merge as-is. The security goal is valid, yet the proposed hard boundary is a detached helper that is not called from production dispatch, its destructive-tool matcher is narrower than current mutating-tool logic, and the contributor has not provided real Gmail/webhook proof.

## What This Changes

The PR adds a new security helper and tests for gating destructive tools in external hook sessions, and changes `wrapWebContent` so `web_search` also gets the long security warning.

## Best Possible Solution

Implement the gate through the existing trusted-policy or `before_tool_call` approval path, reuse `isMutatingToolCall`, decide the `web_search` warning contract explicitly, and require redacted hook-session proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of the diff and current main shows the new guard is only a helper/test while production dispatch already routes through `runBeforeToolCallHook`. I did not run tests because this is a read-only review.

## Solution Assessment

No. A detached regex helper is not the best fix; the maintainable path is a policy/approval integration on the existing tool dispatch seam using the central mutating-tool classifier.
