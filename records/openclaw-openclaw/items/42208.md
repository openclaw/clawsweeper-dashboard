## Summary

Keep open: the security goal is valid, but the branch is not merge-ready because the proposed approval guard is not wired into production dispatch, duplicates a narrower mutating-tool classifier, lacks real hook-session proof, and changes the existing web_search prompt contract.

## What This Changes

The PR adds an external-content tool-call guard and tests, and changes `wrapWebContent` so `web_search` output includes the full external-content security warning.

## Best Possible Solution

Implement this as a focused slice of the external-content security model: gate hook-session mutating tools at the existing before-tool-call or trusted-policy approval seam using `isMutatingToolCall`, then settle the `web_search` warning contract with tests and redacted real hook proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the new guard appears only in the PR helper/test, while current main routes production tool calls through `runBeforeToolCallHook`. I did not run a live Gmail/webhook scenario because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

No. A detached regex helper is not the best solution; the maintainable path is a production trusted-policy or before-tool-call gate using the existing mutating-tool classifier, plus an explicit decision on the `web_search` warning contract.
