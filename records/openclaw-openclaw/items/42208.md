## Summary

Keep open: the PR targets a valid security boundary, but it is not mergeable as-is because the new guard is not wired into production dispatch, its matcher misses current outbound send paths, the web_search behavior change lacks matching test updates, and the external PR has no real after-fix proof.

## What This Changes

The branch adds a security guard and tests for destructive tool calls in hook sessions, and changes `wrapWebContent` so `web_search` results include the external-content warning.

## Best Possible Solution

Implement the boundary through the existing trusted-policy or `before_tool_call` approval path, reuse the current mutating-tool classifier, preserve break-glass semantics deliberately, update the web_search contract tests, and require real hook-session proof.

## Reproduction Assessment

Yes. Source inspection reproduces the PR defects: the guard is added only as an unused helper/test, the dispatch path already has a separate approval seam, current send tools do not match the new matcher, and the web_search test still expects no warning.

## Solution Assessment

No. A standalone security helper is not the best fix unless it is integrated into the existing approval path and aligned with OpenClaw's current mutating-tool taxonomy.
