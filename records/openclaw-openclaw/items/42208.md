## Summary

Keep open: this security PR targets a real hardening area, but the patch does not enforce its claimed boundary, misses existing mutating tool paths, changes the web_search wrapper contract without aligning tests, and has no real Gmail/webhook behavior proof.

## What This Changes

The PR adds a `src/security/tool-call-guard.ts` helper with tests and changes `wrapWebContent` so `web_search` content receives the full security warning.

## Best Possible Solution

Implement hook-session approval through the existing trusted-policy or `before_tool_call` approval path, reuse `isMutatingToolCall`, explicitly decide the `web_search` warning contract, and prove it with a redacted real Gmail/webhook run.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the new guard is only a helper/test while current production dispatch already routes through `runBeforeToolCallHook`. I did not run tests because this review was read-only.

## Solution Assessment

No. A detached regex helper is not the best fix; the maintainable path is to integrate policy into the existing approval seam and reuse the central mutating-tool classifier.

## Review Findings

  - body: `evaluateToolCallGuard` returns `requiresApproval`, but the PR never calls it from agent, gateway, MCP, native hook, or tool execution before the selected tool runs. Hook-originated destructive calls would still execute without this gate, so the advertised hard boundary is unenforced.

  - confidence: 0.9
