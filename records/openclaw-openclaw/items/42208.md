## Summary

Keep open: the security goal is valid, but the PR is not mergeable because the new guard is unused in production dispatch, its destructive-tool matcher is narrower than current main's mutating-tool classifier, the web_search warning change conflicts with the current wrapper contract/test, and the external PR has no real hook-session proof.

## What This Changes

The PR adds a standalone external-content tool-call guard with tests and changes `wrapWebContent` so `web_search` content receives the external-content security warning.

## Best Possible Solution

Implement the gate through the existing `before_tool_call` or trusted-policy approval path, reuse the shared mutating-tool classifier, update the `web_search` wrapper contract and tests deliberately, and validate with redacted hook-session proof.

## Reproduction Assessment

Yes for the PR defects. Source inspection against PR head and current main shows the guard is only introduced as a helper/test, the diff does not wire production dispatch, and current main already has broader mutating-tool classification plus an approval seam.

## Solution Assessment

No. A standalone helper is not the best way to solve this unless it is integrated into the existing approval path and aligned with OpenClaw's current mutating-tool taxonomy.
