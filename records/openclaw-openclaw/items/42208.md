## Summary

Keep open, but do not merge as-is: the security direction is plausible, yet this branch leaves the claimed hard approval boundary unused, uses a narrower mutating-tool matcher than current main, conflicts with the tested web_search wrapper contract, and lacks real external-hook proof.

## What This Changes

The PR adds a new external-content tool-call guard and tests, and changes `wrapWebContent` so `web_search` results always receive the long security warning.

## Best Possible Solution

Implement this through the existing trusted-policy or `before_tool_call` approval path, reuse the central mutating-tool classifier, decide the `web_search` warning contract explicitly, and validate with redacted real hook-session proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the new guard is only added as a helper/test and not wired into dispatch, and current tests still encode the old `web_search` warning contract. I did not run tests because this was a read-only review.

## Solution Assessment

No. A standalone security helper is not the best way to solve this unless it is integrated into the existing approval stack, aligned with OpenClaw’s mutating-tool taxonomy, and backed by real external-hook proof.
