## Summary

Keep open. The security hardening goal is plausible, but this patch is not mergeable because the new guard is not invoked from production tool dispatch, its matcher is narrower than the current mutating-tool contract, the web_search warning change conflicts with existing behavior, and no real Gmail/webhook proof is present.

## What This Changes

The PR adds a new external-content tool-call guard/test and changes `wrapWebContent` so `web_search` also receives the long security notice.

## Best Possible Solution

Implement the approval requirement in `runBeforeToolCallHook` or a trusted tool policy using `isMutatingToolCall`, then explicitly document and test the chosen `web_search` warning contract.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the new guard is only added as a helper/test while production dispatch already flows through `runBeforeToolCallHook`. I did not run tests because this review is read-only.

## Solution Assessment

No. A detached regex helper is not the best fix; the maintainable path is a production approval-policy integration on the existing dispatch seam using the central mutating-tool classifier.
