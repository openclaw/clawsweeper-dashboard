## Summary

Keep open. The PR targets a real security gap, but the proposed hard boundary is not safe to merge because the new guard is not wired into runtime dispatch, its destructive-tool matcher is narrower than current mutating tool handling, the web_search wrapper change conflicts with the existing test contract, and the external PR still lacks real behavior proof.

## What This Changes

The branch adds a new external-content tool-call guard/test and changes `wrapWebContent` so `web_search` results include the full external-content security warning.

## Best Possible Solution

Implement the boundary through the existing trusted-policy or `before_tool_call` approval path, reuse the current mutating-tool classifier/provenance semantics, update the web_search contract tests, and require maintainer security review plus redacted hook-session proof.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main gives a high-confidence reproduction path for the review defects: the guard is only added as a helper/test, current dispatch uses a separate approval seam, the matcher misses current mutating send paths, and the web_search test still expects no full warning.

## Solution Assessment

No. A standalone helper is not the best way to solve this security issue unless it is wired into the existing approval path and aligned with OpenClaw's current mutating-tool taxonomy and external-content provenance semantics.
