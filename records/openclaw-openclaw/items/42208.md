## Summary

Keep open. The security goal is valid, but the patch is not mergeable because the new guard is not wired into runtime tool dispatch, it misses OpenClaw's current outbound messaging tools, the web_search wrapper test still contradicts the behavior change, and the external PR has no real after-fix behavior proof.

## What This Changes

The PR adds a new security tool-call guard/test for hook sessions and changes `wrapWebContent` so `web_search` results include the external-content warning.

## Best Possible Solution

Implement the boundary through `runBeforeToolCallHook` or trusted tool policy, reuse the current mutating/outbound tool taxonomy, preserve the existing break-glass behavior deliberately, and update focused regression tests.

## Reproduction Assessment

Yes. Source inspection reproduces the PR defects: the guard is added only as a helper/test, current dispatch already uses a separate before-tool-call approval path, current messaging tools do not match the new regexes, and the current web_search test still expects no warning.

## Solution Assessment

No. A standalone security helper is not the best fix unless it is integrated into the existing approval path and aligned with OpenClaw's current mutating-tool taxonomy.
