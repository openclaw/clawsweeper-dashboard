## Summary

Keep open. Current main still lacks the requested hard approval boundary for destructive tool calls from external-content sessions, and this PR is not landable as submitted because the new guard is not wired into runtime execution, uses a narrower classifier than the existing mutation helper, relies on brittle raw hook session keys, and leaves the web_search warning test expectation stale.

## What This Changes

The PR adds a new security tool-call guard and tests for hook sessions, and changes `wrapWebContent` so `web_search` results receive the long external-content security warning.

## Best Possible Solution

Implement the policy in the existing before-tool-call or trusted-tool-policy path, using explicit external-content provenance plus `isMutatingToolCall`, then route mandatory approval through the established plugin approval gateway while preserving deliberate unsafe-content bypass semantics.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no guard symbol, hook runs only wrap prompts, and the production before-tool-call path has no external-content approval policy; the PR diff adds a helper but does not connect it to that path.

## Solution Assessment

No. The proposed direction is useful, but the submitted implementation is not the best fix because it is unintegrated, uses a narrower classifier than current main already has, and relies on brittle raw session-key prefixes.

Full review comments:

- **[P1] Wire the guard into tool execution:** `src/security/tool-call-guard.ts:46-76`
  - body: The new `evaluateToolCallGuard` helper is never called from the production before-tool-call or tool-dispatch path, so destructive tool calls from external hook content would still execute without the promised human approval boundary.
  - confidence: 0.97
- **[P2] Reuse the existing mutating-tool classifier:** `src/security/tool-call-guard.ts:15-29`
  - body: The hard-coded destructive regex misses current mutating surfaces such as `message`, `sessions_send`, action-sensitive `cron`/`gateway`/`canvas` calls, and `process` actions. A security gate should align with `isMutatingToolCall` instead of maintaining a smaller parallel list.
  - confidence: 0.91
- **[P2] Carry hook provenance instead of matching raw session keys:** `src/security/tool-call-guard.ts:58-60`
  - body: Cron normalizes hook sessions under `agent:<id>:hook:...`, while this guard only checks `isExternalHookSession(sessionKey)` against raw `hook:*` keys. Once integrated, common hook runs could bypass the approval requirement unless provenance is passed explicitly.
  - confidence: 0.88
- **[P2] Update the web_search warning expectation:** `src/security/external-content.ts:343-344`
  - body: Changing `wrapWebContent(..., "web_search")` to always include the long warning conflicts with the existing test that asserts web search wrapping does not contain `SECURITY NOTICE`, so the PR would fail that focused test until the expected behavior is updated.
  - confidence: 0.94
