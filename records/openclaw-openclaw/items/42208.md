## Summary

Keep open: the security direction is plausible, but this PR is not mergeable because the new approval guard is not connected to production tool dispatch, uses a narrower destructive-tool taxonomy than current main, changes the tested web_search wrapper contract, and lacks real external-hook proof.

## What This Changes

The PR adds a security helper and tests for requiring approval on destructive tools in external-content hook sessions, and changes web_search wrapping to include the external-content warning.

## Best Possible Solution

Implement the gate through the existing trusted-policy or before_tool_call approval path, reuse the central mutating-tool classifier, decide the web_search warning contract explicitly, and require redacted real hook-session proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the guard is only added as a helper/test and current main's dispatch uses a separate before_tool_call approval path. I did not run tests because this was a read-only review.

## Solution Assessment

No. A standalone helper is not the best way to solve this unless it is integrated into the existing approval stack, aligned with OpenClaw's mutating-tool taxonomy, and backed by real external-hook proof.
