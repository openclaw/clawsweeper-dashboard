## Summary

Keep open, but this PR is not merge-ready: the security goal is valid, but the added guard is detached from production dispatch, narrower than the current mutating-tool taxonomy, conflicts with the tested web_search warning contract, and lacks real hook-session proof.

## What This Changes

The PR adds a new security guard and tests for destructive tool calls in external hook sessions, and changes web_search wrapping to always include the external-content security notice.

## Best Possible Solution

Implement the gate through the existing trusted-policy or before_tool_call approval path, reuse the central mutating-tool classifier, decide the web_search warning contract explicitly, and require redacted real hook-session proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of the diff and current main shows the new guard is only in a helper/test while dispatch already routes through runBeforeToolCallHook. I did not run tests because this is a read-only review.

## Solution Assessment

No. The maintainable fix is to implement approval through the existing before_tool_call/trusted-policy pipeline and reuse isMutatingToolCall/isLikelyMutatingToolName rather than adding a detached regex helper.
