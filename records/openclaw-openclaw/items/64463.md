## Summary

Current main still does not implement the requested opt-in wall-clock session duration cap or cumulative per-session token cap. The useful implementation sketch in the discussion and the external Helm docs workaround help frame the product need, but OpenClaw core still only exposes daily/idle freshness, per-run timeout, prompt/context snapshots, loop detection, and session-store disk maintenance rather than a durable session budget kill switch.

## What This Changes

The issue requests optional `session.maxDurationMinutes` and `session.maxTokensPerSession` config keys to stop, reset, or compact runaway sessions before unbounded runtime or cumulative token spend.

## Best Possible Solution

Implement this as a core session-budget feature with optional config schema/types/docs, durable per-session budget state, pre-LLM and pre-tool enforcement, lifecycle checks for wall-clock age, deterministic budget-exceeded feedback, and focused tests across chat, cron, heartbeat, CLI/ACP, reset, and compaction interactions.

## Reproduction Assessment

Yes for the missing feature: current-main code inspection reproduces the absence through exact-key search and strict schema/type checks. The specific 72-hour cost incident is not locally reproduced, but the missing config and enforcement surfaces are directly verifiable.

## Solution Assessment

Unclear until maintainers settle the public behavior semantics. The right direction appears to be core session-budget enforcement rather than Helm docs, per-run timeouts, or plugin-only work, because the cap must run before expensive core model/tool paths.
