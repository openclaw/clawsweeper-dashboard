## Summary

Keep open. Current main still lacks the requested opt-in wall-clock session duration cap and cumulative per-session token budget, while the adjacent reset, maintenance, and context-token counters do not provide a hard runaway-cost kill switch.

## What This Changes

Add opt-in session-level wall-clock and cumulative token budget controls that gracefully compact, reset, stop, or otherwise contain runaway sessions.

## Best Possible Solution

Implement this as a core session-budget feature with explicit config schema/types/docs, durable budget state, pre-model and pre-tool enforcement, lifecycle checks, deterministic user feedback, and focused tests across the main session surfaces.

## Reproduction Assessment

Yes for the missing feature: current-main code inspection reproduces the absence through exact-key search, strict schema/type checks, runtime freshness inspection, and token-accounting tests/docs. The specific 72-hour production cost incident is not locally reproduced, but the missing enforcement surfaces are directly verifiable.

## Solution Assessment

Unclear until maintainers settle the public behavior semantics. Core enforcement is the right direction because the cap must run before expensive model/tool paths, but stop/warn/compact/reset/freeze behavior and cumulative accounting need product/API judgment first.
