## Summary

Keep open. Current main still lacks the requested opt-in wall-clock session duration cap and cumulative per-session token cap; existing reset, maintenance, context-window, and thread-binding controls are adjacent guardrails but not a hard spend/runaway-session kill switch.

## What This Changes

The issue requests optional `session.maxDurationMinutes` and `session.maxTokensPerSession` config keys to stop, reset, or compact runaway long-running sessions before unbounded cost accrues.

## Best Possible Solution

Implement a core session-budget feature with optional config schema/types/docs, durable per-session budget accounting, pre-model and pre-tool enforcement, wall-clock lifecycle checks, deterministic budget-exceeded user feedback, and focused tests across interactive chat, cron, heartbeat/system events, CLI/ACP sessions, reset, and compaction behavior.

## Reproduction Assessment

Yes for the missing feature: current-main code inspection reproduces the absence through exact-key search, strict schema/type checks, runtime reset-path inspection, and tests showing `totalTokens` is not cumulative spend. The specific 72-hour production cost incident is not locally reproduced, but the missing config and enforcement surfaces are directly verifiable.

## Solution Assessment

Unclear until maintainers settle the public behavior semantics. The right direction appears to be core session-budget enforcement rather than Helm docs or plugin-only work, because the cap must run before expensive model/tool paths and needs durable session state.
