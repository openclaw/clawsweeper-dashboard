## Summary

Keep open. Current main still lacks the requested opt-in wall-clock session duration cap and cumulative per-session token cap, while the broader budget tracker was closed as superseded in favor of this scoped session-cap tracker. Current main now has some subagent kill/control surfaces, which helps one active-intervention comment, but it does not implement automatic session budget enforcement.

## What This Changes

Request opt-in `session.maxDurationMinutes` and `session.maxTokensPerSession` config keys to bound wall-clock session lifetime and cumulative token spend.

## Best Possible Solution

Define additive opt-in session budget config with schema/types/docs, durable cumulative accounting, pre-call and lifecycle enforcement, and predictable operator/user feedback across interactive, cron, CLI/ACP, and subagent paths.

## Reproduction Assessment

Yes for the missing feature by source inspection: current main lacks the requested session keys and enforcement gates, and the lifecycle path only evaluates daily/idle freshness. The reported production spend incidents were not replayed locally.

## Solution Assessment

Unclear until maintainers settle the public config and exceedance semantics. The likely maintainable direction is a core session/gateway budget contract enforced before expensive continuation, not docs-only guidance or an after-the-fact monitor.
