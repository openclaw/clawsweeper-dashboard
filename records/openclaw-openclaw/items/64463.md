## Summary

Keep open. Current main still lacks the requested opt-in wall-clock session cap and cumulative per-session token cap; existing reset, compaction, maintenance, and usage display paths are adjacent guardrails but do not provide a spend/runaway-session kill switch.

## What This Changes

Add opt-in `session.maxDurationMinutes` and `session.maxTokensPerSession` limits to prevent long-running sessions from accruing unbounded cost.

## Best Possible Solution

A maintainable end state is a core session-budget layer with optional config schema/types/docs, durable cumulative accounting, and enforcement before expensive model/tool work across chat, cron, CLI/ACP, reset, and compaction paths.

## Reproduction Assessment

Yes for the missing feature: current-main code inspection reproduces the absence through exact-key search, strict schema/type review, reset-path inspection, and usage-counter semantics. The specific 72-hour production cost incident is not locally reproduced, but the missing config and enforcement surfaces are directly verifiable.

## Solution Assessment

Unclear until maintainers settle the public behavior. The core direction is appropriate because this must run before expensive model/tool paths, but the exact exceedance action and state model need product/API review first.
