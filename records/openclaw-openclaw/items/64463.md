## Summary

Keep open. Current main still lacks the requested opt-in wall-clock session duration cap and cumulative per-session token cap; adjacent reset, runtime timeout, usage snapshot, and maintenance controls do not provide the requested runaway-session spend guard.

## What This Changes

Request optional `session.maxDurationMinutes` and `session.maxTokensPerSession` config keys to cap long-running or high-token sessions.

## Best Possible Solution

Add a core session-budget feature with explicit schema/types/help/docs, durable accounting, pre-model and pre-tool enforcement, wall-clock lifecycle checks, deterministic user feedback, and coverage for interactive, cron, subagent, reset, and compaction paths.

## Reproduction Assessment

Yes for the missing feature: source inspection on current main verifies the requested keys and enforcement paths are absent. The specific production cost incidents were not locally reproduced in this read-only review.

## Solution Assessment

Unclear until maintainers settle the public config and lifecycle semantics. The guard likely belongs in core session/model/tool paths rather than an external plugin because it must run before expensive calls are made.
