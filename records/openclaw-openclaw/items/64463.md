## Summary

Keep open. Current main still lacks the requested opt-in wall-clock session duration cap and cumulative per-session token cap, and the related broad budget issue was closed specifically because this issue remains the scoped session-cap tracker.

## What This Changes

Requests new opt-in `session.maxDurationMinutes` and `session.maxTokensPerSession` config keys to cap wall-clock session age and cumulative token spend.

## Best Possible Solution

Design an additive core session-budget contract with schema/types/docs, durable cumulative accounting, pre-model and lifecycle enforcement, deterministic user/operator feedback, and targeted coverage for interactive, cron, CLI/ACP, reset, and compaction paths.

## Reproduction Assessment

Yes for verifying the missing feature: source inspection on current main shows the requested config keys and enforcement gates are absent. The specific 72-hour cost incident and later production subagent run were not replayed locally.

## Solution Assessment

Unclear until maintainers settle the public API and exceedance semantics. The likely maintainable direction is core session/gateway enforcement before expensive calls or lifecycle reuse, not docs-only guidance or a plugin-only after-the-fact monitor.
