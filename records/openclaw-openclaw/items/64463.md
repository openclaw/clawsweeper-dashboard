## Summary

Keep open. Current main still lacks the requested opt-in wall-clock session duration cap and cumulative per-session token cap; existing reset, context, cost snapshot, and disk-maintenance controls are adjacent but do not provide a hard runaway-session spend guard.

## What This Changes

Request adds `session.maxDurationMinutes` and `session.maxTokensPerSession` config keys to stop, reset, or compact long-running/high-token sessions before unbounded cost accumulates.

## Best Possible Solution

Add a core session-budget feature with explicit optional config schema/types/help/docs, durable cumulative accounting, pre-model and pre-tool enforcement, wall-clock lifecycle checks, deterministic user feedback, and tests covering interactive, cron, CLI/ACP, reset, and compaction behavior.

## Reproduction Assessment

Yes for the missing feature: source inspection on current main verifies the requested keys and enforcement paths are absent. The specific 72-hour production cost incident was not locally reproduced, but the missing native cap is directly verifiable.

## Solution Assessment

Unclear until maintainers settle the public behavior semantics. The durable enforcement likely belongs in core session/model/tool paths rather than only docs or an external plugin, because the guard must run before expensive calls are made.
