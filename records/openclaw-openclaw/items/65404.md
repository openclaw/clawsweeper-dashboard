## Best Possible Solution

Keep the merged and released loopback-resolver implementation; any new mismatch in prompt-report accounting should be filed as a fresh focused bug with current-version evidence.

## Reproduction Assessment

Yes for the historical bug path: the issue body and merged replacement PR show the CLI report previously saw an empty tool surface while bundle MCP tools were callable. Current main no longer reproduces that failure because it resolves `promptTools` and passes them into `buildSystemPromptReport`.

## Solution Assessment

Yes. Reusing the gateway loopback scoped-tool resolver and feeding that same surface into prompt/report construction is the narrow maintainable solution, and current main already uses that shape.
