## Summary

Current main already fixes the reported codex-cli prompt-report gap: the CLI preparation path resolves loopback-scoped MCP tools once, feeds them into both the CLI system prompt and `buildSystemPromptReport`, and has regression coverage for nonempty report entries.

## What This Changes

The issue requests that codex-cli `systemPromptReport.tools.entries` and `tools.schemaChars` reflect injected MCP tools instead of reporting an empty tool catalog.

## Best Possible Solution

Keep the shipped resolver-reuse implementation and close this issue; any new mismatch in a different CLI backend report path should be filed as a fresh, narrower bug.

## Reproduction Assessment

Yes. The historical source path is clear from the merged fix and tests: CLI preparation previously had an empty report tool surface, while current main now feeds loopback-scoped tools into the prompt report and asserts a nonempty entry.

## Solution Assessment

Yes. Reusing the existing loopback-scoped tool resolver for both prompt construction and report generation is the narrow maintainable fix because it avoids a second MCP/tool-policy implementation.
