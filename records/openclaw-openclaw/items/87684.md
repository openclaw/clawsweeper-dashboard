## Summary

Keep open: the PR is a useful, well-scoped Control UI queueing fix, but it still does not preserve a direct normal send when the socket closes after the connected check but before the `chat.send` ACK. That leaves one of the reported reconnect-window message-loss paths unresolved, and the external PR still has only unit/simulation proof rather than live Control UI/Gateway reconnect proof.

## What This Changes

The PR changes Control UI chat sending to queue disconnected WebChat sends, replay them after Gateway hello with a preserved idempotency key, and expose failed retry UI after a bounded retry budget.

## Best Possible Solution

Capture every normal outbound send before transport, remove it only after ACK, replay it with the same idempotency key after hello, and prove the live reconnect plus Failed/Retry behavior in a real Control UI/Gateway run.

## Reproduction Assessment

Yes, source-level: current main drops disconnected normal sends before queueing and only drains the queue after seq-gap reconnects. I did not run a live browser/Gateway 1001 reconnect scenario in this read-only review.

## Solution Assessment

No, not yet: the intended client queue plus idempotency-key reuse is the right shape, but this PR still needs to queue or requeue direct normal sends that lose the `chat.send` ACK during reconnect.
