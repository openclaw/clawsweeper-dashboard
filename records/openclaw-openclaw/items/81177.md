## Summary

Keep open. Current main has partial mitigation via `messages.visibleReplies: "message_tool"`, but the WhatsApp monitor still has non-empty fallback delivery paths and the requested robust NO_REPLY policy plus audit/config behavior remains a security-sensitive product decision; related shared-token policy is still open at https://github.com/openclaw/openclaw/issues/40165.

## What This Changes

Requests a stronger WhatsApp outbound gate that suppresses broader NO_REPLY variants, supports explicit-send mode, and records audit metadata before monitor replies are delivered.

## Best Possible Solution

Settle the shared silent-token and outbound-delivery policy, use the existing `messages.visibleReplies`/message-tool mechanism where it fits, and add WhatsApp-specific audit behavior only after owners approve the privacy and config contract.
