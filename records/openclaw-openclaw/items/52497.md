## Summary

Keep open. Current main still has the narrow failure mode described in the discussion: `sessions_send` can only forward an error if `agent.wait` returns one, but the gateway wait path can still return a bare timeout while a lifecycle/model error is pending inside the retry-grace window. Related #50981 and #43095 are adjacent timeout-semantics work, but neither closes this exact diagnostic gap.

## What This Changes

The issue asks for `sessions_send` timeout responses to include recent target-session model or lifecycle error context so callers can distinguish model failures from routing or reply timeouts.

## Best Possible Solution

Keep this focused on the gateway wait seam: preserve a recent pending lifecycle/model error through `waitForAgentJob` or the `agent.wait` timeout response, then rely on the existing `sessions_send` pass-through to expose it. True no-error wait expiry should continue to return a plain timeout or a clearly non-model reason.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction already exists in current tests: `agent.wait` receives a lifecycle error during a short wait and still expects timeout, and the current `waitForAgentJob` timer path explains why.

## Solution Assessment

Yes, a narrow gateway-side fix is the best path. A full `sessions_send` failure taxonomy would be broader product work; the maintainable first fix is to carry already-available lifecycle/model error context without changing unrelated delivery statuses.
