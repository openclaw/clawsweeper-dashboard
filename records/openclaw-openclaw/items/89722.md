## Summary

Keep open: this PR targets a real cron retention gap, but the patch would prune stable isolated cron base entries solely by age, which can drop durable per-job session metadata for active low-frequency jobs.

## What This Changes

Adds an isolated cron session-key predicate and uses it in the cron session reaper, with regression coverage for pruning `agent:*:cron:<jobId>` entries.

## Best Possible Solution

Keep active isolated cron base entries as durable per-job metadata and implement retention against real per-run artifacts or deleted-job leftovers, with upgrade tests for low-frequency jobs.

## Reproduction Assessment

Yes from source: current main persists isolated cron state under a stable base key while `sweepCronRunSessions` only prunes `:run:` keys. I did not run a live gateway retention window in this read-only review.

## Solution Assessment

No. Preserving the broad `isCronRunSessionKey` contract is good, but pruning every stable `agent:*:cron:<jobId>` row by age is not the safest fix because that row is current durable per-job state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep durable cron base sessions out of the sweep:** `src/cron/session-reaper.ts:75`
  - body: This makes the reaper delete stable `agent:*:cron:<jobId>` rows whenever `updatedAt` is older than `sessionRetention`. Current cron persistence deliberately stores isolated job metadata only under that stable base key, and `resolveCronSession` carries preferences/auth/model fields from it across runs; with the default 24h retention, an active weekly or monthly isolated job can lose that state before its next run. The fix should preserve active job base metadata and prune real run artifacts or deleted-job leftovers instead.
