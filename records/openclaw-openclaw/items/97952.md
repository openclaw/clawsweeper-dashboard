## Summary

Keep open: the PR is a focused Codex authorization hardening, but it currently applies the owner/admin denial before the existing bound-message command-authorization guard, which can turn ordinary non-command bound chat traffic into visible denial replies. The stricter native-control boundary and additive hook fields also remain maintainer-owned because the PR has the protected `maintainer` label and compatibility impact.

## What This Changes

The PR gates Codex native control commands and bound turns to owners or `operator.admin`, passes current auth facts into inbound-claim events, updates docs, and adds regression coverage.

## Best Possible Solution

Keep the Codex plugin as the enforcement boundary, but fix the bound-turn guard ordering so only authorized non-owner/native turns get the denial, then have maintainers explicitly accept the compatibility and hook API contract before merge.

## Reproduction Assessment

Yes. Source inspection shows current main consumes command-authorization-absent Codex bound messages before native execution, while the PR head moves the owner/admin denial before that guard and dispatch now supplies non-owner facts to the plugin claim.

## Solution Assessment

No, not as currently ordered. The Codex plugin is the right enforcement layer, but the owner/admin denial should run only after the existing `commandAuthorized` and empty-prompt guards, and maintainer acceptance is still needed for the compatibility and hook API changes.
