## Summary

Keep open: the PR targets a real source-proven MSTeams message-loss path, but it is not merge-ready because it persists full adaptive-card invoke payloads that can contain token-exchange data and still lacks real Teams behavior proof.

## What This Changes

The branch adds a durable receive, retry, replay, and duplicate-suppression path for MSTeams non-poll `card.action` invokes before returning the Teams OK response.

## Best Possible Solution

Redact token-bearing Teams invoke fields before durable accept, keep focused regression tests for sanitized replay, then validate the repaired path with redacted real Teams proof or a maintainer-provided tenant.

## Reproduction Assessment

Yes at source level: current main returns the Teams OK response while `handler.run` is still fire-and-forget for non-poll card actions, so a post-ACK dispatch failure has no durable retry record. I did not run a live Teams tenant repro.

## Solution Assessment

No as-is. The channel ingress queue is the right owner-boundary primitive, but the best merge shape first redacts token-bearing invoke fields and then adds real Teams proof.
