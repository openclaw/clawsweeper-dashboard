## Summary

Keep open: current main and the latest release cover the Telegram album-buffer/config half, but LINE still handles each downloadable media webhook as its own agent turn with no same-sender media buffer or LINE batching config.

## What This Changes

The issue requests gateway-layer media batching for rapid image/media messages, including Telegram media groups and LINE time-proximity batching.

## Best Possible Solution

Keep this issue open as the canonical LINE media batching tracker and design a LINE-plugin-owned aggregation path; leave the existing Telegram media-group implementation unchanged.

## Reproduction Assessment

Yes, at source level: the LINE handler creates one `allMedia` list and one `processMessage` call per message event, while the webhook loop has no same-sender media buffer. I did not run a live LINE burst test.

## Solution Assessment

Unclear until maintainers choose the product contract. A generic gateway buffer is not the best boundary as written; the safer shape is LINE-plugin-owned batching with explicit config/default, reply-token, and lifecycle semantics.
