## Summary

Keep open. Current main has the Telegram album buffering/config piece, but the LINE side still handles downloadable media one webhook event at a time with no same-sender media buffer or LINE config/docs surface; the remaining work is a product/config decision, not cleanup-close material.

## What This Changes

The issue requests optional media batching so rapid LINE media bursts and Telegram albums arrive as one agent turn instead of separate per-image turns.

## Best Possible Solution

Keep Telegram’s existing album handling, then add LINE-plugin-owned optional burst coalescing keyed by account, conversation, and sender if maintainers accept the latency semantics; avoid gateway-layer heuristics unless a broader inbound contract is explicitly chosen.

## Reproduction Assessment

Yes, at source level: send multiple downloadable LINE media webhook events from the same sender/conversation and current main constructs a fresh allMedia array and calls processMessage for each event. I did not run a live LINE webhook burst.

## Solution Assessment

Unclear. Telegram shows a plugin-owned batching solution is viable, but LINE heuristic buffering adds public latency/config semantics, so maintainers should choose LINE-local buffering versus a broader media-aware inbound contract before implementation.
