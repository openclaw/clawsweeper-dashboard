## Summary

Keep open. Current main and the latest release cover Telegram album buffering and its config key, but LINE still downloads one webhook media event into a fresh `allMedia` array and dispatches it immediately; the LINE SDK has `imageSet` metadata for simultaneous images, but OpenClaw does not currently use it.

## What This Changes

The issue requests optional media batching for rapid images/media, originally proposed as gateway-layer buffering for Telegram albums and LINE bursts.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin boundary: prefer LINE `imageSet.id/index/total` when available, use a conservative optional delay/fallback only after maintainer approval, aggregate media into one `LineInboundContext`, and leave generic gateway policy plus Telegram album buffering unchanged.

## Reproduction Assessment

Yes, at source level. Two downloadable LINE media webhook events from the same sender/conversation each create a fresh `allMedia` array and call `processMessage` separately; no live LINE burst was run in this read-only review.

## Solution Assessment

No, not as a generic gateway-layer buffer. The maintainable direction is a LINE-plugin-owned design that uses `imageSet` metadata when available and only adds heuristic timing/config semantics after maintainer approval.
