## Summary

Keep open. Current main already handles Telegram album buffering, but the LINE side of the request remains unresolved: LINE media webhook events still go through one fresh agent turn per event with no same-sender media buffer or LINE config/docs surface.

## What This Changes

The issue requests optional media batching so rapid image/media sends, especially Telegram albums and LINE image bursts, reach the agent as one turn instead of multiple replies.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin boundary: an optional media batching delay that aggregates downloadable LINE media by account, conversation, and sender, flushes on timer or before same-key text/non-media handling, and documents the config.

## Reproduction Assessment

Yes. A source-level reproduction is two allowed LINE image message events from the same sender/source: current handleLineWebhookEvents sends each to handleMessageEvent, which builds a separate one-item allMedia context and calls processMessage each time.

## Solution Assessment

No for the proposed generic gateway-layer placement. The narrow maintainable solution is LINE-plugin-scoped buffering using the existing media/context seams, while Telegram-specific leftovers stay in the narrower linked follow-ups.
