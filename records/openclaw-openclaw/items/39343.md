## Summary

Keep open. Telegram album buffering and its `mediaGroupFlushMs` config are now implemented, but current main and the latest shipped release still process LINE downloadable media webhook events one at a time with no LINE media-batch config or docs surface.

## What This Changes

Request to buffer rapid image/media messages so Telegram albums and LINE media bursts reach the agent as one combined turn.

## Best Possible Solution

Keep the Telegram implementation as-is, then add optional LINE-plugin-owned media burst coalescing using LINE `imageSet` metadata when present plus a bounded same-sender fallback after maintainers settle latency and reply-token semantics.

## Reproduction Assessment

Yes, at source level. Multiple LINE downloadable media webhook events from the same sender/conversation each create a fresh `allMedia` array and call `processMessage` separately; I did not run a live LINE burst in this read-only review.

## Solution Assessment

Unclear as proposed. A generic gateway-layer buffer is not the best-proven boundary; the safer direction is a LINE-plugin-owned optional buffer, with any shared SDK/core contract decided only if the plugin cannot express it cleanly.
