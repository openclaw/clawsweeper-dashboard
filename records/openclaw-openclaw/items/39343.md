## Summary

Keep open. Current main implements the Telegram album-buffering half and the Telegram timeout follow-up is closed, but LINE still processes each downloadable media webhook event as a separate agent turn with no LINE batching config or docs surface.

## What This Changes

Request to batch rapid multi-image/media updates, especially Telegram albums and LINE consecutive image events, into one agent turn with a configurable delay.

## Best Possible Solution

Implement optional LINE media batching inside the LINE plugin boundary: account/conversation/sender-scoped buffering, timer and same-key text flushes, aggregated `allMedia`, schema/types/docs/tests, and no generic gateway or Telegram behavior change.

## Reproduction Assessment

Yes, at source level for the current behavior. Current main creates one LINE `allMedia` array per `MessageEvent` and immediately calls `processMessage` while webhook events are iterated one by one; no live LINE webhook run was performed.

## Solution Assessment

No for the requested gateway-layer placement. The narrower maintainable solution is LINE-plugin-local buffering because access gating, replay dedupe, media download, reply-token handling, and context construction already meet there.
