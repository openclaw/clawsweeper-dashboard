## Summary

Keep open. The Telegram album buffering/config half is on current main, but the LINE plugin still processes each downloadable media webhook event independently; the LINE SDK exposes optional image-set metadata that OpenClaw does not use, and there is no LINE media batching config or docs surface.

## What This Changes

Request to batch rapid LINE or Telegram image sends into one agent turn with an optional media buffer instead of dispatching each image separately.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin: prefer LINE imageSet when present, fall back to an optional account/conversation/sender-scoped media delay, flush on timer or same-key text, and leave core gateway and Telegram behavior unchanged.

## Reproduction Assessment

Yes, at source level. Two allowed LINE image events from the same account, conversation, and sender each enter handleMessageEvent with a fresh allMedia array and each calls processMessage separately on current main.

## Solution Assessment

No for the proposed generic gateway-layer placement. The narrower maintainable solution belongs in the LINE plugin, where access gating, replay dedupe, media downloads, optional imageSet metadata, reply-token fallback, and context construction already meet.
