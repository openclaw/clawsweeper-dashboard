## Summary

Keep open. Current main already buffers Telegram albums by `media_group_id`, but the LINE side of the request remains unresolved: LINE downloadable media events are still processed one webhook event at a time with no same-sender media batch delay, no LINE config field, and no docs surface.

## What This Changes

Request optional batching of rapid image/media messages so Telegram albums and LINE image bursts are forwarded to the agent as one turn instead of multiple separate turns.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin boundary: add an optional media batching delay, buffer downloadable LINE media by account plus conversation plus sender, prefer `imageSet.id` for simultaneous LINE image sets, flush on timer and before same-key text or non-media events, aggregate through the existing `allMedia` fields, and leave Telegram album buffering plus #55216’s skipped-download warning out of scope.

## Reproduction Assessment

Yes. A source-level reproduction is two LINE image webhook `message` events from the same account/source: current `handleLineWebhookEvents` dispatches each event directly, and each `handleMessageEvent` call creates a fresh single-event `allMedia` list before immediate `processMessage`.

## Solution Assessment

No for the proposed generic gateway-layer placement. The narrow maintainable fix is LINE-plugin-scoped batching using existing media/context seams and LINE `imageSet` metadata, while keeping the already-implemented Telegram media-group path unchanged.
