## Summary

Keep open. Current main and the latest release have the Telegram album/config half, but LINE still builds and dispatches one agent turn per image event with no LINE media batching option or docs surface.

## What This Changes

The issue requests buffering rapid media uploads so Telegram albums and LINE multi-image bursts reach the agent as one turn instead of one turn per image.

## Best Possible Solution

Implement the remaining behavior in the LINE plugin: prefer LINE imageSet metadata when available, fall back to account/conversation/sender-scoped short buffering, aggregate media before context construction, and keep gateway core plus Telegram behavior unchanged.

## Reproduction Assessment

Yes, at source level. Two allowed LINE image events from the same sender and conversation each enter handleMessageEvent with a fresh allMedia array and each calls processMessage separately on current main.

## Solution Assessment

No for the issue's generic gateway-layer placement. The narrower maintainable solution belongs in the LINE plugin because access checks, replay dedupe, LINE media download, reply-token fallback, and LineInboundContext construction already meet there.
