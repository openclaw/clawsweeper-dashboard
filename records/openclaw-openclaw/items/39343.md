## Summary

Keep open: current main and v2026.6.10 still leave the LINE rapid-media batching part missing, while Telegram album buffering exists and the newer Telegram raw-reference regression is tracked separately.

## What This Changes

The issue requests optional gateway-level media batching so rapid LINE images and Telegram albums can be delivered to the agent as one media-bearing turn.

## Best Possible Solution

Keep this as the broad/LINE media batching tracker, implement any remaining behavior in the LINE plugin boundary, and leave the narrower Telegram raw-reference bug to https://github.com/openclaw/openclaw/issues/96846.

## Reproduction Assessment

Yes, at source level: current LINE still creates one media-bearing inbound context per message event, and shared inbound debounce explicitly excludes media-bearing messages. I did not run a live LINE webhook test in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. A generic gateway buffer is not the best boundary as written; the safer maintainable shape is LINE-plugin-owned batching with explicit config/default and lifecycle semantics.
