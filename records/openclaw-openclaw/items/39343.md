## Summary

Keep open: current main and v2026.6.10 implement the Telegram album-buffering half, but LINE still handles each downloadable media webhook as a separate agent turn and the shared inbound debounce deliberately excludes media-bearing messages. The remaining work needs a maintainer decision on LINE batching semantics and public config shape before implementation.

## What This Changes

Request media-aware batching so rapid image/media updates, especially Telegram albums and LINE image bursts, reach the agent as one turn instead of separate replies.

## Best Possible Solution

Keep this as the canonical tracker and design LINE-plugin-owned media aggregation with explicit batching keys, flush timing, reply-token semantics, config/docs, and tests while leaving Telegram album buffering unchanged.

## Reproduction Assessment

Yes, at source level: LINE still creates one media-bearing inbound context per message event and shared inbound debounce explicitly excludes media-bearing messages. I did not run a live LINE webhook test in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. A generic gateway buffer is not the best boundary as written; the safer shape is LINE-plugin-owned batching with explicit config/default and lifecycle semantics.
