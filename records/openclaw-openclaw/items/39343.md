## Summary

Keep open: current main and the latest release cover the Telegram album buffering/config half, but LINE still processes each downloadable media webhook as its own agent turn with no LINE media-batching config or buffer.

## What This Changes

Requests an optional media batching buffer so rapid image/media events from Telegram and LINE can be delivered to the agent as one turn.

## Best Possible Solution

Implement plugin-owned LINE media batching only after maintainers choose the timing, keying, reply-token, and config contract; keep the existing Telegram `mediaGroupFlushMs` path unchanged.

## Reproduction Assessment

Yes, source-level: current LINE handling creates one media context per downloadable message event and the shared inbound debounce deliberately excludes media-bearing events. I did not run a live LINE webhook test in this read-only review.

## Solution Assessment

No as proposed: a generic gateway-layer buffer is not the best owner boundary. The safer solution is LINE-plugin-owned media batching with explicit product decisions for delay, keying, flush, reply-token, and config semantics.
