## Summary

Keep open. Current main and the latest shipped release cover the Telegram album half of the request, including a user-facing `mediaGroupFlushMs`, but LINE still downloads and forwards each media webhook event as its own agent turn with no LINE batching config or docs surface.

## What This Changes

The issue requests configurable media batching so rapid LINE images and Telegram albums reach the agent as one turn instead of one turn per attachment.

## Best Possible Solution

Keep the Telegram implementation as-is, then add optional LINE-plugin-owned media burst coalescing keyed by account, conversation, and sender, with shared SDK/core changes only if the LINE plugin cannot express the behavior cleanly.

## Reproduction Assessment

Yes, at source level. Two downloadable LINE media webhook events from the same sender/conversation each create a fresh `allMedia` array and call `processMessage` separately; I did not run a live LINE burst in this read-only review.

## Solution Assessment

Unclear as proposed. A generic gateway-layer buffer is not the best-proven boundary; the safer direction is an optional LINE-plugin-owned buffer after maintainers settle latency, reply-token, and batching-scope semantics.
