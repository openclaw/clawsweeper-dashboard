## Summary

Keep open: current main and v2026.6.10 cover the Telegram album half, but LINE still processes each downloadable media webhook as its own agent turn with no LINE media batching option or buffer.

## What This Changes

The issue requests buffering rapid image/media messages so related media can reach the agent as one turn instead of separate replies.

## Best Possible Solution

Design the remaining behavior inside the LINE plugin boundary with maintainer-approved timing, keying, reply-token, flush, and config semantics; keep the implemented Telegram album path unchanged.

## Reproduction Assessment

Yes, at source level. LINE currently creates and dispatches one inbound context per downloadable media webhook, while shared inbound debounce intentionally excludes media-bearing messages.

## Solution Assessment

No, not as a generic gateway-layer buffer. The maintainable path is LINE-plugin-owned media aggregation with an explicit product/config contract, because channel reply-token and webhook lifecycle behavior are provider-specific.
