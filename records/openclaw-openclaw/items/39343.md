## Summary

Keep open. Current main and the latest release cover the Telegram album/config part, but LINE still handles downloadable media one webhook event at a time with no same-sender media buffer or LINE config/docs surface.

## What This Changes

Requested optional batching of rapid image/media messages so a user’s media burst is delivered to the agent as one turn instead of separate turns.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin boundary: add an optional media batching delay, buffer downloadable LINE media by account plus conversation plus sender, flush on timer and before same-key text or non-media events, aggregate media into one `LineInboundContext`, and leave Telegram album buffering plus shared text debounce semantics unchanged.

## Reproduction Assessment

Yes, at source level. Two downloadable LINE media webhook events from the same sender/conversation each create a fresh `allMedia` array and call `processMessage` separately; no live LINE burst was run in this read-only review.

## Solution Assessment

No, not as a generic gateway-layer buffer. The maintainable direction is a LINE-plugin-owned optional buffer after maintainers settle delay, reply-token, redelivery, and batching-scope semantics.
