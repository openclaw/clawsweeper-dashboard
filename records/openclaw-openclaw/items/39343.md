## Summary

Keep open. Current main has the Telegram album/config side implemented, but LINE still processes each downloadable media webhook event as its own agent turn and has no LINE media batching config or docs surface; the remaining work is a feature/config/product decision, not a stale cleanup close.

## What This Changes

The issue requests optional media batching so rapid image/media messages, especially Telegram albums and LINE image bursts, are delivered to the agent as one turn.

## Best Possible Solution

Make an explicit maintainer product call, then implement the remaining behavior inside the LINE plugin boundary if approved: optional media burst coalescing keyed by account/conversation/sender, timer flush plus flush-before-text/non-media behavior, and aggregation into the existing allMedia context.

## Reproduction Assessment

Yes, at source level. Current LINE code processes each downloadable image/video/audio/file event independently with a fresh allMedia array and immediate processMessage call; I did not run a live LINE webhook burst.

## Solution Assessment

Unclear. Telegram’s plugin-owned implementation supports solving this at the channel/plugin boundary, but the issue asks for gateway-layer buffering and LINE heuristic batching, so maintainer product judgment is needed before implementation.
