## Summary

Keep open. Current main solves the Telegram album/config part, but LINE downloadable media still flows one webhook event at a time with no same-sender media buffer or LINE config/docs surface; the remaining work is a user-visible feature/config decision, not cleanup-close material.

## What This Changes

The issue requests optional media batching so rapid image or album updates are delivered to the agent as one media-rich turn instead of separate turns.

## Best Possible Solution

Make the product call explicitly, then add LINE-plugin-owned optional media burst coalescing keyed by account, conversation, and sender unless maintainers choose a broader media-aware inbound contract.

## Reproduction Assessment

Yes, at source level: send multiple LINE downloadable media webhook events from the same sender/conversation and current code constructs a fresh allMedia array and calls processMessage for each event. I did not run a live LINE webhook burst.

## Solution Assessment

Unclear. Telegram’s plugin-owned album buffering supports a plugin-boundary solution, but the requested gateway-layer and LINE heuristic behavior changes user-visible latency/config semantics and needs maintainer direction before implementation.
