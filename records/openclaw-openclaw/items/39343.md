## Summary

Keep open. Current main implements the Telegram album/config half, but the LINE half remains unresolved: LINE media webhook events are still handled one event at a time, with no same-sender media buffer or LINE config/docs surface.

## What This Changes

The issue requests buffering rapid image/media messages into one agent turn, using Telegram media_group_id when available and LINE sender/chat/time proximity otherwise.

## Best Possible Solution

Implement the remaining behavior inside the LINE plugin boundary: optional media batching delay, account/conversation/sender-scoped buffering, timer and same-key text flushes, and aggregation into one LineInboundContext while leaving Telegram and core gateway contracts unchanged.

## Reproduction Assessment

Yes, at source level. Two allowed LINE image events from the same account, conversation, and sender each enter handleMessageEvent with a fresh allMedia array and each calls processMessage separately on current main.

## Solution Assessment

No for the proposed generic gateway-layer placement. The narrower maintainable solution belongs in the LINE plugin, where access gating, replay dedupe, media downloads, reply-token fallback, and context construction already meet.
