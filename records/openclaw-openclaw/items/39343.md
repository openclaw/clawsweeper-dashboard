## Summary

Keep open. Current main already buffers Telegram albums by `media_group_id`, but the LINE half of the request still processes each downloadable media webhook as a separate agent turn and exposes no LINE media-batching config or docs surface.

## What This Changes

Request optional buffering of rapid image/media messages so Telegram albums and LINE image bursts reach the agent as one turn.

## Best Possible Solution

Add an optional LINE-plugin media batching path that aggregates same-sender downloadable media before `buildLineMessageContext`, while preserving Telegram's existing album flow and leaving Telegram tuning/failure behavior to the linked follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is two LINE image `message` webhook events from the same account/source: current `handleLineWebhookEvents` sends both through `handleMessageEvent`, and each call builds and processes a separate one-item `allMedia` context.

## Solution Assessment

No for the proposed generic gateway-layer placement. The narrow maintainable solution is LINE-plugin-scoped buffering using existing media/context seams, with Telegram follow-ups handled by #55216 and #76149.
