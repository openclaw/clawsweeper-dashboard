## Summary

Keep open. Current main solves the Telegram album/config part, but LINE still processes rapid image webhook events one at a time and the remaining media-aware batching behavior needs maintainer product/config direction rather than cleanup closure.

## What This Changes

The issue requests buffering rapid media-only channel updates so related images are delivered to the agent as one turn instead of separate replies.

## Best Possible Solution

Decide the product boundary first, then implement either a LINE-owned media batch delay keyed by account, conversation, and sender, or fold LINE into a documented shared media-aware inbound batching policy while keeping Telegram’s existing media-group path and the separate Telegram failed-download issue independent.

## Reproduction Assessment

Yes, at source level. Two rapid LINE image `MessageEvent`s will each enter `handleMessageEvent`, get their own one-item `allMedia` array, and call `processMessage` separately; I did not run a live LINE webhook test.

## Solution Assessment

Unclear for the literal gateway-layer proposal. Telegram is already handled in the Telegram plugin, while the remaining LINE behavior needs maintainer choice between a LINE plugin heuristic and a broader shared media-aware inbound batching contract.
