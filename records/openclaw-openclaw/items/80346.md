## Summary

Keep open: current main has the intended generic Slack reload rule, but this report is about a credible plain-token write path where the running Slack `App` can remain alive with the old token when no channel restart fires. The related merged secret-reload work is helpful context, but it does not cover plain `channels.slack.botToken` / `appToken` rotation from config or the panel.

## What This Changes

The issue requests reliable Slack channel restart after rotating `channels.slack.botToken` or `appToken` so Bolt does not keep using a stale in-memory authorization token.

## Best Possible Solution

Make Slack credential writes deterministic: when the effective Slack bot token or app token changes through config save, config patch, or external file reload, the affected Slack channel/account should be stopped and started with the new tokens, with regression coverage for plain token-string rotation.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection confirms the failure mode if the restart is skipped: Slack creates a long-lived Bolt `App` from the old token, while panel/config saves rely on reload notification/watcher behavior rather than an explicit Slack restart fallback.

## Solution Assessment

Unclear until the regression is captured. The narrow maintainable direction is to preserve the existing channel reload contract and make plain Slack credential changes deterministically enter the existing stop/start path, rather than adding a new config option or changing Slack runtime policy.
