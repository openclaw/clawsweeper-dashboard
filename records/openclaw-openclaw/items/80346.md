## Summary

Keep open. Current main has the intended generic Slack hot-reload wiring, and a non-container current-main run could not reproduce a missed restart, but the reported Sandpaw/container or panel write path remains unproven and no merged fix specifically covers token-only Slack rotation after this report.

## What This Changes

The issue asks for reliable Slack channel restart after `channels.slack.botToken` or `appToken` changes, including direct file edits and panel/config.patch writes.

## Best Possible Solution

Add focused proof for Slack token-only rotation through direct file edits and panel/config.patch in the managed/container environment, then fix the generic reload notification path unless that proof shows a Slack-specific panel restart hook is required.

## Reproduction Assessment

No. A later current-main Ubuntu/inotify run exercised 14 Slack token edits with expected coalesced restarts, but it did not cover the Sandpaw container, panel/config.patch, or long-session watcher path where the report failed.

## Solution Assessment

Unclear. The source already routes observed `channels.slack.*` diffs to Slack channel restart, so the best fix depends on reproducing whether the missed edit is a file-watcher miss, in-process write/hash de-dupe issue, or panel bypass.
