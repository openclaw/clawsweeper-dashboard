## Summary

Keep open: current main and the latest release already cover the normal direct-edit and in-process config-write path for `channels.slack.*`, but the reported Sandpaw/container long-session miss was not reproduced and is not fully explained by source. The related native Slack token-rotation issue is broader product/security work, not a duplicate of this manual-token reload reliability bug.

## What This Changes

The issue reports that rotating a configured Slack bot token string can leave the running Slack channel using a stale Bolt authorization cache until the gateway is manually restarted.

## Best Possible Solution

Keep this as the narrow reload reliability bug: reproduce the Sandpaw/container or panel `config.patch` miss, then fix the gateway config-reload path so a changed Slack token always rebuilds the Slack App without folding in native OAuth token rotation.

## Reproduction Assessment

No. The issue has concrete production observations, but the only current-main live attempt in the discussion exercised repeated direct `channels.slack.botToken` edits and saw the expected coalesced restarts; the Sandpaw/container and panel-save paths remain unproven.

## Solution Assessment

Unclear. Current main already has the right owner-boundary path for ordinary config changes, so the best fix needs to target the specific watcher/internal-write miss rather than add a parallel Slack restart mechanism or the broader native token-rotation feature.
