## Summary

Keep open: current main has the expected Slack reload wiring and in-process config writes should hot-reload, but external file edits still depend on watcher delivery and the reported Sandpaw/container miss was not disproven by the non-container reproduction run.

## What This Changes

Slack bot/app token-string rotation should reliably rebuild the running Slack channel so Bolt does not keep using a stale cached token.

## Best Possible Solution

Keep the existing Slack prefix-based reload contract, then prove or fix the remaining container file-watch gap in the gateway config reloader so observed token edits always reach the existing channel restart path.

## Reproduction Assessment

No high-confidence current-main live repro yet. Source inspection shows the plausible failure mode: external config edits only reload after chokidar emits an event, while the provided non-container live run did not reproduce the container-specific miss.

## Solution Assessment

Unclear. Slack App rebuilding is already wired once a change is observed, so the narrower likely fix is a generic config reload/watch robustness improvement or a Sandpaw/container-specific proof path, not a Slack-only token cache shim.
