## Summary

Keep open: current main has the normal Slack restart wiring and a non-container live check did not reproduce misses, but the reported Sandpaw/container path is still plausible because direct file edits still rely on chokidar events without polling outside tests.

## What This Changes

The issue asks for reliable Slack channel restart after `channels.slack.botToken` or `appToken` rotation so Bolt does not keep using a stale cached token.

## Best Possible Solution

Make the generic config reload watcher reliable for containerized direct config-file edits while preserving the existing plugin-owned `channels.slack` restart contract.

## Reproduction Assessment

No high-confidence live reproduction is available for current main. Source inspection shows direct edits still rely on non-polling chokidar events outside tests, and the issue provides specific container logs while a non-container stress run could not reproduce the miss.

## Solution Assessment

Yes, the maintainable fix direction is generic config reload reliability rather than a Slack-only restart hook. Slack already declares the correct reload prefix, and the gateway already rebuilds channels once a `channels.slack.*` change reaches the reload plan.
