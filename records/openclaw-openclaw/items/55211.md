## Label Justifications

- `P2`: This is a normal-priority hook runtime bugfix with limited but real blast radius for plugin and internal hook delivery.
- `merge-risk: 🚨 compatibility`: The ALS guard intentionally changes same-key redispatch behavior for work spawned from inside an existing hook handler chain.

## Summary

Keep open for maintainer review: current main and the latest release still dispatch internal hooks without a same-key recursion guard, while this PR is focused, has regression coverage, and has sufficient terminal proof. I found no blocking code-review defects, but the AsyncLocalStorage contract is a compatibility choice for hooks spawned from a handler's async lineage.

## What This Changes

The PR adds an AsyncLocalStorage guard to triggerInternalHook and tests recursive, sequential, concurrent, cross-session, and cross-action hook dispatch behavior.

## Best Possible Solution

Land the ALS guard if hooks owners accept same async-lineage suppression; otherwise adjust the guard to permit delayed child redispatch and add a regression test for that contract.

## Reproduction Assessment

Yes, by source inspection rather than an executed current-main repro: current main and v2026.5.18 await hook handlers with no active same-key guard, so a handler can recursively call triggerInternalHook with the same event key. The PR's posted terminal proof also demonstrates the unguarded loop and guarded behavior in a standalone Node runtime.

## Solution Assessment

Mostly yes: ALS is a clean fit for blocking same async-chain recursion while preserving independent concurrent dispatches, and the PR tests those boundaries. The remaining question is whether suppressing delayed child async same-key redispatch is the intended hook contract.
