## Summary

Keep open. Current main still has no re-entrancy guard, but the latest PR head uses a process-global same-key Set that can silently drop legitimate overlapping hook events from documented fire-and-forget message-hook paths.

## What This Changes

The PR adds a global Set-backed re-entrancy guard to `triggerInternalHook` and regression tests for recursive, sequential, cross-session, and cross-action hook dispatch.

## Best Possible Solution

Land a recursion guard that distinguishes true in-call-chain re-entry from independent overlapping events, with coverage for direct recursion, same-key concurrent delivery, detached post-completion delivery, sequential delivery, and cross-session/action behavior.

## Reproduction Assessment

Yes for the PR regression: keep one same-key hook dispatch pending with a slow handler, then start a second same-key dispatch and the PR's global Set returns before delivery. The original loop report is modeled by the PR's recursive same-key test, while current main has no guard at all.

## Solution Assessment

No as written. The latest Set approach fixes the previous AsyncLocalStorage detached-context problem, but it acts as a process-global same-key mutex; a safer fix needs call-chain or dispatch-lineage recursion tracking instead of suppressing all overlap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve concurrent same-key hook deliveries:** `src/hooks/internal-hooks.ts:329-331`
  - body: This global `triggerGuard` treats any overlapping dispatch for the same type/action/sessionKey as re-entry. Normal message hooks are fired without awaiting completion, so two real `message:received` or `message:preprocessed` events for the same session can overlap; the second one will hit this return and its observers will never run. Keep the recursion guard scoped to the originating call chain or dispatch lineage, and add a same-key concurrent delivery regression test.
  - confidence: 0.88
