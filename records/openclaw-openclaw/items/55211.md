- `merge-risk: 🚨 compatibility`: The PR changes the internal hook dispatch contract by suppressing same-key nested delivery inside one async chain.

## Summary

Keep open: the AsyncLocalStorage direction is focused, but the current copied guard set can leave inherited parent keys stale in nested hook contexts, so a delayed ancestor-key redispatch can still be silently dropped after the ancestor dispatch completes.

## What This Changes

The PR adds an AsyncLocalStorage-based same-key re-entrancy guard to `triggerInternalHook` and expands internal hook tests for recursive, sequential, concurrent, cross-key, and delayed redispatch behavior.

## Best Possible Solution

Keep the ALS guard, but repair guard lifetime so nested child contexts do not retain stale ancestor keys, add a nested delayed redispatch regression, and have maintainers accept or document the same-chain suppression contract before merge.

## Reproduction Assessment

Yes, at source level. Current main awaits hook handlers without an active same-key dispatch guard, so a handler that awaits `triggerInternalHook` for the same `type:action:sessionKey` can re-enter the same handler path; the PR body also includes capped terminal proof for the main loop scenario.

## Solution Assessment

No, not yet. AsyncLocalStorage is a narrow maintainable approach, but copying active guard sets leaves stale inherited ancestor keys in nested async contexts and needs a targeted repair before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep ancestor cleanup visible to nested async contexts:** `src/hooks/internal-hooks.ts:341`
  - body: Each nested dispatch copies `activeKeys` into a new `Set`. If an `A` handler triggers `B` and `B` schedules a delayed `A` redispatch, that callback inherits B's copied set with `A` still present; when the outer `A` finally deletes its key, the copy is not updated, so the delayed `A` is silently skipped after `A` completed. Add a nested delayed-redispatch regression and make ancestor-key cleanup propagate to inherited contexts.

## Repair Work Prompt

Repair PR https://github.com/openclaw/openclaw/pull/55211. In `src/hooks/internal-hooks.ts`, nested dispatches currently clone `activeKeys`; delayed callbacks from nested different-key hooks keep copied ancestor keys after the parent dispatch finishes and can skip an ancestor-key redispatch. Adjust the guard so ancestor cleanup is visible to child async contexts, or otherwise prune stale inherited keys, without blocking independent concurrent top-level triggers. Add a regression in `src/hooks/internal-hooks.test.ts` where a `command:new` handler triggers a different hook, that nested hook schedules a fire-and-forget delayed `command:new` after the outer dispatch returns, and the delayed `command:new` is delivered. Preserve the intended contract: immediate same-chain same-key recursion is suppressed, independent concurrent same-key triggers are delivered, and top-level delayed same-key triggers are delivered. Do not change plugin APIs or add config.
