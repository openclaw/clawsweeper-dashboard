- `merge-risk: 🚨 compatibility`: The branch intentionally changes existing nested same-chain same-key hook delivery from recursive delivery to suppression.

## Summary

Keep open for maintainer review: current main still has the recursive same-key hook behavior, and the branch addresses the concrete review concerns I found without a blocking correctness issue, but the intentional suppression of nested same-chain same-key delivery is a compatibility contract decision.

## What This Changes

The PR adds an AsyncLocalStorage-based re-entrancy guard to internal hook dispatch and expands hook tests for recursive, concurrent, sequential, cross-key, and delayed redispatch behavior.

## Best Possible Solution

Land the guarded dispatch only if maintainers accept same-chain same-key suppression as the internal hook contract, with the concurrent and delayed-delivery tests kept as compatibility coverage.

## Reproduction Assessment

Yes, from source inspection: current main recursively invokes matching handlers with no active-dispatch guard, so a handler that triggers the same `type/action/sessionKey` can re-enter indefinitely. I did not run a local repro because this review was required to keep the checkout read-only.

## Solution Assessment

Mostly yes: AsyncLocalStorage is a narrow way to distinguish same-chain recursion from independent concurrent dispatches, and the PR now clears inherited guard state for delayed work. The unresolved part is not a code repair; it is whether maintainers accept same-chain same-key suppression as the permanent contract.
