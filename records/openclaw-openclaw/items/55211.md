## Summary

Keep open. Current main still has the original unguarded `triggerInternalHook` dispatcher, so the PR is not obsolete, but the proposed `AsyncLocalStorage` guard has a concrete delivery regression for same-key triggers scheduled from a handler after the parent dispatch has completed.

## What This Changes

The PR adds an `AsyncLocalStorage`-backed re-entrancy guard to `triggerInternalHook` plus regression tests for recursive same-key dispatch, sequential dispatch, cross-session dispatch, and cross-action dispatch.

## Best Possible Solution

Land a narrow internal-hook recursion guard that blocks same-key recursion only while an outer dispatch is actively in flight, while preserving at-least-once delivery for independent concurrent events and same-key events scheduled after the parent dispatch completes. Cover direct recursion, concurrent same-key delivery, detached post-completion delivery, sequential delivery, and cross-session/action behavior before merge.

## Reproduction Assessment

Yes for the patch finding: schedule a same-key `triggerInternalHook` from inside a handler with `setTimeout` or another detached async resource, wait for the outer dispatch to complete, and observe that the inherited `AsyncLocalStorage` store suppresses the later trigger. The original loop report is modeled by the PR's recursive handler test, and current main has no guard for it.

## Solution Assessment

No as written: `AsyncLocalStorage` is a reasonable direction, but the guard should clear the active key when the guarded dispatch completes so descendant async work created by a handler is not suppressed after completion. A mutable active-set with `try/finally` cleanup inside the ALS context would be a safer narrow fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Clear the guard after the active dispatch finishes:** `src/hooks/internal-hooks.ts:339`
  - body: `AsyncLocalStorage` propagates this `store` into async resources created by hook handlers, including timers or promises that run after `triggerInternalHook` has already returned. Those later same-key triggers still see the guard and are skipped, so a handler that schedules follow-up hook work can silently lose a legitimate event after the parent dispatch has completed. Keep the key active only for the duration of this dispatch, and add a regression test for a detached same-key trigger scheduled from a handler.
  - confidence: 0.82
