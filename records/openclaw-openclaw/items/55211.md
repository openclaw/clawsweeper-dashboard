## Summary

Keep open. The latest revision fixes the previous direct delayed-redispatch gap, but nested different-key dispatches still copy parent guard keys into descendant AsyncLocalStorage stores, so delayed follow-up hooks can be silently suppressed after the parent dispatch has finished.

## What This Changes

The PR adds an AsyncLocalStorage-based re-entrancy guard to internal hook dispatch and expands tests for recursive, sequential, concurrent, cross-key, and delayed redispatch behavior.

## Best Possible Solution

Keep the re-entrancy guard, but make inherited guard cleanup visible to nested async descendants or use a guard representation that cannot retain stale copied parent keys after parent dispatch completion.

## Reproduction Assessment

Yes, source-reproducible. Current main awaits handlers without a same-key active-dispatch guard, and the remaining PR defect follows from copying parent guard keys into nested ALS Sets at the PR head.

## Solution Assessment

No, not as written. AsyncLocalStorage is a reasonable mechanism, but the guard representation must avoid stale copied parent keys in nested delayed async descendants.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep parent guard cleanup visible to nested descendants:** `src/hooks/internal-hooks.ts:341`
  - body: `activeKeys ? new Set([...activeKeys, guardKey])` copies the parent guard keys into nested different-key dispatches. If a `command:new` handler triggers `message:received`, and the message handler schedules delayed `command:new` follow-up work, that delayed callback inherits the nested Set with the copied `command:new` key; deleting only `message:received` in this `finally` leaves the parent key behind, so the follow-up `command:new` is silently skipped even after the original command dispatch has finished. Preserve cleanup visibility for inherited keys and add a nested delayed-redispatch regression test.
  - confidence: 0.9
