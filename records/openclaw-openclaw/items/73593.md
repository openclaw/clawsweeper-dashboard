## Summary

Keep this PR open. Current main still has the reported misleading fallback errors, but the proposed fix is not merge-ready because it infers missing provider configuration from generic provider auth instead of the media-understanding runner's actual configured or attempted entries; it also needs the required changelog entry for a user-facing CLI fix.

## What This Changes

The PR adds missing-provider CLI errors for `infer image describe` and `infer audio transcribe` when media understanding returns no text, plus tests for missing-provider and configured-empty-output cases.

## Best Possible Solution

Expose or return the media-understanding decision/attempt state to the CLI and emit the missing-provider configuration message only when no media-understanding entry was resolved or attempted. Preserve the empty-output errors after any configured provider, configured CLI entry, shared/per-capability media model, default image model, or auto-detected local CLI was tried, then document the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Current main is source-confirmed to throw the old path-focused errors for no-text media-understanding results, and the PR regression is reproducible by configuring `tools.media.*.models`, `agents.defaults.imageModel`, or a CLI entry that produces no text while the new helper still sees no generic provider auth.

## Solution Assessment

No. The UX direction is correct, but the helper duplicates only one subset of provider detection instead of using the runner's resolved-entry or attempt-state signal, so it is not the narrowest maintainable fix.
