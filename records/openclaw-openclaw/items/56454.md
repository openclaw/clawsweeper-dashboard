## Summary

Close as implemented. Current main and the shipped release line already solve the user-facing problem by honoring configurable image-understanding timeouts through the existing `tools.media` surface, including per-model overrides and the MiniMax fallback path, so this older `agents.defaults.imageTimeoutSeconds` PR is now redundant.

## What This Changes

The PR adds an `agents.defaults.imageTimeoutSeconds` config field, schema/help metadata, generated config docs, and image-tool tests, then wires the explicit image tool to pass that timeout to image provider calls.

## Best Possible Solution

Keep the shipped `tools.media.image.timeoutSeconds` and per-model media timeout contract as the single public image-understanding timeout surface, and close this obsolete PR rather than adding a parallel `agents.defaults` key.

## Reproduction Assessment

Yes. The original source-level reproduction was a slow image provider exceeding the hardcoded image-tool timeout; current main removes that hardcoded path, and focused tests verify configured timeout propagation. I did not run live slow-provider tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The PR was a reasonable focused fix when opened, but the best current solution has already shipped through the broader media-understanding config contract, which covers explicit image analysis, per-model overrides, and the MiniMax fallback path without adding overlapping config.
