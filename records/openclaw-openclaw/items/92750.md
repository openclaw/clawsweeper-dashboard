## Summary

Keep open: the PR is focused, but it is not merge-ready because accepting zero totals without updating `totalTokensFresh` creates an inconsistent session row, and the supplied proof does not exercise the TUI/Gateway behavior it claims to fix.

## What This Changes

The branch changes `buildGatewaySessionRow` so `totalTokens` accepts non-negative fresh session totals and transcript fallback totals.

## Best Possible Solution

Preserve fresh zero totals and freshness together, keep stale or missing totals unknown, add a focused regression test, and prove the actual TUI footer path before using this PR to close the linked issue.

## Reproduction Assessment

Yes, at source level: current main filters fresh zero totals through `resolvePositiveNumber`, and the TUI footer renders a null total plus context as `tokens ?/<context>`. I did not run a live TUI session in this read-only review.

## Solution Assessment

No. The PR is a plausible half-step, but it widens `totalTokens` to include zero without preserving freshness or proving the end-to-end TUI path.
