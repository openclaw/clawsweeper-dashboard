## Summary

Keep open. The underlying command-queue leak is real and the pruning direction is useful, but the latest PR head is not mergeable as submitted: it drops existing test helper bindings, loses active-task waiter notification on rejected lane tasks, and still needs the current `cron-nested` lane plus changelog follow-through.

## What This Changes

This PR adds idle dynamic-lane pruning to `src/process/command-queue.ts` after task completion, lane clearing, and reset, plus a test-only lane count helper and regression tests.

## Best Possible Solution

Land a narrow command-queue fix that prunes only fully idle dynamic/default-concurrency lanes, preserves all current well-known lanes and custom concurrency, keeps active-task waiter semantics unchanged, and records the user-facing memory leak fix in the active changelog.

## Reproduction Assessment

Yes. The leak can be reproduced by enqueueing work on a unique `session:*` or `auth-probe:*` lane and then inspecting snapshots after completion on current main; the PR's immediate test regression is reproduced by the exact-head `check-test-types` annotations.

## Solution Assessment

No, not as submitted. Pruning in the command-queue owner is the right direction, but this patch must restore removed test bindings, preserve waiter notification on rejected tasks, include `CommandLane.CronNested`, and add changelog coverage before merge.

## Review Findings

Full review comments:

- **[P2] Notify active waiters when lane tasks reject:** `src/process/command-queue.ts:297-299`
  - body: The rejected-task path now calls `pump(true)` without the previous `notifyActiveTaskWaiters()` call. `waitForActiveTasks()` relies on that notification, so a rejected lane task can leave restart/recovery waiters asleep until their timeout even though the active set is empty.
  - confidence: 0.9
- **[P2] Keep snapshot helpers wired into the test module:** `src/process/command-queue.test.ts:30`
  - body: This removes the `getCommandLaneSnapshot` and `getCommandLaneSnapshots` bindings, but existing tests below still call both helpers. The exact-head `check-test-types` run already fails with `Cannot find name` errors, so the PR cannot merge until those bindings are restored.
  - confidence: 0.97
- **[P3] Carry cron-nested into the persistent lane set:** `src/process/command-queue.ts:115-120`
  - body: Current main has `CommandLane.CronNested`, and gateway startup configures it as a well-known lane for isolated cron turns. Include it in `PERSISTENT_LANES` so the persistent-lane policy matches the current enum and add coverage for that lane.
  - confidence: 0.82

  - body: This fixes a user-visible long-running gateway memory leak, so repo policy expects a one-line `CHANGELOG.md` entry under the active version with contributor credit. Without it, the shipped fix will be missing from release notes.
  - confidence: 0.9
