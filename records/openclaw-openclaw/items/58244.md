## Summary

Keep open: the underlying command-queue lane-retention bug is source-reproducible on current main, but this PR is not mergeable yet because it lacks real behavior proof and introduces queue-contract regressions.

## What This Changes

The PR adds idle dynamic-lane pruning to command queue lane bookkeeping and regression tests.

## Best Possible Solution

Land a rebased queue fix that prunes only disposable idle dynamic lanes while preserving suspended, tuned, and well-known lane state, waiter notifications, existing snapshot tests, and redacted real gateway proof.

## Reproduction Assessment

Yes. By source inspection, current main keeps lane state in a process-global Map, dynamic session/auth-probe names create entries, and production code does not prune idle completed dynamic lanes.

## Solution Assessment

No, not as submitted. Idle dynamic-lane pruning is the right direction, but the patch must preserve zero-concurrency suspension, rejected-task waiter notifications, snapshot helper bindings, and the cron-nested well-known lane.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve suspended dynamic lanes:** `src/process/command-queue.ts:319-321`
  - body: Current main uses `setCommandLaneConcurrency(params.laneId, 0)` from `suspendSession()` to hold a session lane until auto-resume. This guard deletes idle non-persistent lanes with `maxConcurrent <= 1`, so a suspended dynamic lane can be removed and the next enqueue recreates it at default concurrency, bypassing the suspension.
  - confidence: 0.91
- **[P2] Notify active-task waiters on rejection:** `src/process/command-queue.ts:298`
  - body: The catch path removes the active task id but now calls only `pump(true)`. `waitForActiveTasks()` relies on `notifyActiveTaskWaiters()`, so restart/recovery waits can sit until timeout after a rejected task has already drained.
  - confidence: 0.9
- **[P2] Restore snapshot helper bindings:** `src/process/command-queue.test.ts:30`
  - body: The PR removes the test module bindings for `getCommandLaneSnapshot` and `getCommandLaneSnapshots`, but the same file still calls both helpers later. The branch therefore leaves undefined identifiers in existing snapshot tests.
  - confidence: 0.97
- **[P3] Keep cron-nested persistent:** `src/process/command-queue.ts:115-120`
  - body: Current main configures `CommandLane.CronNested` as a well-known startup lane for cron inner work, but this persistent set omits it while preserving the other configured lanes. Default-concurrency cron-nested state becomes eligible for pruning despite the PR's well-known-lane policy.
  - confidence: 0.84
