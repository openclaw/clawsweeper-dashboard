## Summary

Keep open: the reported command-queue leak is valid on current main, but this PR is not correct as submitted and still lacks redacted real gateway/session/auth-probe behavior proof after the patch.

## What This Changes

The PR adds idle dynamic command-queue lane pruning and regression tests in `src/process/command-queue.ts` and `src/process/command-queue.test.ts`.

## Best Possible Solution

Land a rebased queue fix that prunes only disposable idle dynamic lanes while preserving suspended, tuned, and well-known lane state, waiter notifications, existing snapshot tests, and real runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main creates dynamic session/auth-probe lane entries in a process-global Map and has no production path that deletes idle completed dynamic lanes.

## Solution Assessment

No, not as submitted. Idle dynamic-lane pruning is the right direction, but the patch must preserve zero-concurrency suspension, rejected-task waiter notifications, snapshot helper bindings, and the cron-nested well-known lane.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve suspended dynamic lanes:** `src/process/command-queue.ts:319-321`
  - body: Current main uses `setCommandLaneConcurrency(params.laneId, 0)` to suspend session lanes until auto-resume. This pruning condition deletes idle non-persistent lanes with `maxConcurrent <= 1`, so a suspended dynamic lane can be removed and the next enqueue recreates it at default concurrency, bypassing the suspension.
  - confidence: 0.91
- **[P2] Notify active-task waiters on rejection:** `src/process/command-queue.ts:298`
  - body: The catch path removes the active task id but now calls only `pump(true)`. `waitForActiveTasks()` relies on `notifyActiveTaskWaiters()`, so restart/recovery waits can sit until timeout after a rejected task has already drained.
  - confidence: 0.9
- **[P2] Restore snapshot helper bindings:** `src/process/command-queue.test.ts:30`
  - body: The PR removes the test module bindings for `getCommandLaneSnapshot` and `getCommandLaneSnapshots`, but the same file still calls both helpers later. That leaves undefined identifiers in the existing snapshot tests.
  - confidence: 0.97
- **[P3] Keep cron-nested persistent:** `src/process/command-queue.ts:115-120`
  - body: Current main configures `CommandLane.CronNested` as a well-known startup lane for cron inner work, but this persistent set omits it while preserving the other configured lanes. Default-concurrency cron-nested state becomes eligible for pruning despite the PR's well-known-lane policy.
  - confidence: 0.84
