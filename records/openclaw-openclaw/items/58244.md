## Summary

Keep open. The underlying leak is valid on current main, but this PR still has blocking regressions in rejected-task waiter notification and test bindings, plus smaller cron-nested persistence and changelog gaps.

## What This Changes

The PR adds idle dynamic-lane pruning to the process command queue after task completion, lane clearing, and reset, with regression tests for the new pruning paths.

## Best Possible Solution

Land a narrow command-queue fix that prunes only fully idle dynamic/default-concurrency lanes, preserves all well-known lanes and custom concurrency, keeps active-task waiter semantics unchanged, and records the gateway memory fix in the active changelog.

## Reproduction Assessment

Yes. On current main, enqueueing work on a unique `session:*` or `auth-probe:*` lane will create a lane entry, and the source has no production path that deletes an idle lane after completion.

## Solution Assessment

No, not as submitted. Pruning in `src/process/command-queue.ts` is the right owner-level fix, but this patch needs the rejected-task waiter notification, snapshot helper bindings, cron-nested persistence, and changelog coverage repaired first.

  - body: Current main configures `CommandLane.CronNested` as a well-known startup lane. Omitting it here lets default-concurrency `cron-nested` state be pruned while the other startup lanes are preserved, so the persistent-lane policy no longer matches the lane set.
  - confidence: 0.84
