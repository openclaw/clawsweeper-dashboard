## Summary

Keep open: current main still has the unbounded dynamic-lane Map behavior, and this PR targets the right surface, but the current head introduces blocking source/test regressions and still lacks external real behavior proof.

## What This Changes

The PR prunes idle dynamic command-queue lanes after task completion, lane clearing, and reset, and adds regression tests plus a lane-count test helper.

## Best Possible Solution

Land a narrow command-queue fix that prunes only idle default-concurrency dynamic lanes, preserves startup and tuned lanes, keeps active-task waiter semantics unchanged on success and rejection, and leaves existing snapshot tests compiling.

## Reproduction Assessment

Yes, by source inspection. Current main creates per-session and auth-probe lane entries in a process-global Map and has no production path that deletes idle entries after work completes.

## Solution Assessment

No, not as submitted. The pruning direction is appropriate, but this patch needs the rejected-task waiter notification, snapshot helper bindings, and `cron-nested` persistence fixed before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Notify waiters after rejected lane tasks:** `src/process/command-queue.ts:298`
  - body: The catch path now calls `pump(true)` without `notifyActiveTaskWaiters()` after `completeTask()` removes the active task id. `waitForActiveTasks()` wakes through those notifications or its timeout, so restart or recovery waits can now sleep until timeout after a rejected task has already drained.
  - confidence: 0.91
- **[P2] Restore the snapshot helper bindings:** `src/process/command-queue.test.ts:30`
  - body: The diff replaces the top-level `getCommandLaneSnapshot` and `getCommandLaneSnapshots` bindings with `getLaneCountForTest`, but existing tests still call both helpers. That leaves undefined identifiers in the PR head and blocks test typechecking.
  - confidence: 0.97
- **[P3] Include cron-nested in persistent lanes:** `src/process/command-queue.ts:115-120`
  - body: Current main configures `CommandLane.CronNested` as the dedicated startup lane for cron agent work. Omitting it from `PERSISTENT_LANES` lets default-concurrency `cron-nested` state be pruned while the other startup lanes are preserved, so the persistent-lane policy no longer matches the configured lane set.
  - confidence: 0.84
