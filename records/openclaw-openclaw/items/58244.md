## Summary

Keep open. The PR targets a real command-queue leak on current main, but the submitted patch still has merge-blocking regressions in waiter notification and test bindings, plus small follow-up gaps for cron-nested persistence and changelog coverage.

## What This Changes

This PR adds idle dynamic-lane pruning to `src/process/command-queue.ts` after task completion, lane clearing, and reset, plus a test-only lane count helper and regression tests.

## Best Possible Solution

Land a narrow command-queue fix that prunes only fully idle dynamic/default-concurrency lanes, preserves all well-known lanes and custom concurrency, keeps active-task waiter semantics unchanged, and records the gateway memory fix in the active changelog.

## Reproduction Assessment

Yes. On current main, the leak can be reproduced by enqueueing work on a unique `session:*` or `auth-probe:*` lane and then observing that no production path removes the lane Map entry after completion; the PR's test-binding regression is visible statically from the exact head.

## Solution Assessment

No, not as submitted. Pruning in `src/process/command-queue.ts` is the right owner-level fix, but the patch must restore rejected-task waiter notification, snapshot helper bindings, cron-nested persistence, and changelog coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Notify active waiters when lane tasks reject:** `src/process/command-queue.ts:297-298`
  - body: This rejected-task path no longer calls `notifyActiveTaskWaiters()` after `completeTask()` removes the active task id. `waitForActiveTasks()` wakes through those notifications or its timeout, so restart/recovery waits can now sleep until timeout after a rejected task has already drained.
  - confidence: 0.91
- **[P2] Restore snapshot helper bindings in the test module:** `src/process/command-queue.test.ts:30`
  - body: The PR removes the `getCommandLaneSnapshot` and `getCommandLaneSnapshots` bindings, but existing tests still call both helpers below. That leaves the exact PR head with undefined identifiers and blocks `check-test-types`.
  - confidence: 0.97
- **[P3] Keep cron-nested as a persistent lane:** `src/process/command-queue.ts:115-120`
  - body: Current main configures `CommandLane.CronNested` as a well-known startup lane. Omitting it here lets default-concurrency `cron-nested` state be pruned while the other startup lanes are preserved, so the persistent-lane policy no longer matches the lane set.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/process/command-queue.ts:306-321`
  - body: This is a user-visible long-running gateway memory leak fix, but the PR only changes queue source and tests. Add a one-line entry under the active `CHANGELOG.md` fixes section so release notes include it.
  - confidence: 0.88
