## Summary

Keep open: the leak is source-reproducible and the fix direction is useful, but this PR still has blocking correctness/type issues and lacks required real behavior proof.

## What This Changes

This PR adds idle dynamic-lane pruning to `src/process/command-queue.ts` and regression tests in `src/process/command-queue.test.ts`.

## Best Possible Solution

Land a rebased queue fix that prunes only disposable idle dynamic lanes while preserving suspended, tuned, and well-known lane state, rejected-task waiter notifications, current snapshot tests, and redacted real gateway proof.

## Reproduction Assessment

Yes, by source inspection: current main stores lane state in a process-global Map, dynamic session/auth-probe names create entries, and production code does not prune idle completed dynamic lanes.

## Solution Assessment

No, not as submitted. Idle dynamic-lane pruning is the right direction, but this patch must preserve zero-concurrency suspension, rejected-task waiter notification, snapshot helper bindings, and the cron-nested well-known lane.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve suspended dynamic lanes:** `src/process/command-queue.ts:319-321`
  - body: Current main uses `setCommandLaneConcurrency(params.laneId, 0)` from `suspendSession()` to hold a session lane until auto-resume. This guard deletes idle non-persistent lanes with `maxConcurrent <= 1`, so a suspended dynamic lane can be removed and the next enqueue recreates it at default concurrency, bypassing the suspension.
  - confidence: 0.91
- **[P2] Notify active-task waiters on rejection:** `src/process/command-queue.ts:298`
  - body: The catch path removes the active task id, but now calls only `pump(true)` and skips `notifyActiveTaskWaiters()`. `waitForActiveTasks()` relies on that notification, so restart/recovery waits can sit until timeout after a rejected task has already drained.
  - confidence: 0.9
- **[P2] Restore snapshot helper bindings:** `src/process/command-queue.test.ts:30`
  - body: This replaces the test module bindings for `getCommandLaneSnapshot` and `getCommandLaneSnapshots`, but the PR head still calls both helpers in existing snapshot tests. The branch therefore leaves undefined identifiers and fails test typechecking.
  - confidence: 0.97
- **[P3] Include cron-nested in persistent lanes:** `src/process/command-queue.ts:115-120`
  - body: Current main configures `CommandLane.CronNested` as a well-known startup lane for cron inner work, but this persistent set omits it while preserving the other configured lanes. Default-concurrency cron-nested state becomes eligible for pruning despite the PR's well-known-lane policy.
  - confidence: 0.84
