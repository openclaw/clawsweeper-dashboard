## Summary

Keep open: current main still has the source-visible dynamic lane retention problem, but this PR is not merge-ready because the branch drops rejected-task waiter notifications, breaks existing snapshot tests, and would prune suspended zero-concurrency dynamic lanes when reconciled with current main; external real behavior proof is also missing.

## What This Changes

The PR adds idle dynamic lane pruning to the process command queue and regression tests for task-completion, clear, and reset paths.

## Best Possible Solution

Land a rebased narrow queue fix that prunes only idle default-concurrency dynamic lanes while preserving suspended, tuned, and well-known lane state, rejected-task waiter notifications, and existing snapshot tests.

## Reproduction Assessment

Yes, by source inspection. Current main stores lane state in a process-global Map, dynamic session/auth-probe names create entries, and production code only clears the Map through reset-style paths rather than idle pruning.

## Solution Assessment

No, not as submitted. Idle dynamic-lane pruning is the right direction, but this implementation needs rebase/patches for zero-concurrency suspension, rejected-task waiter notification, snapshot helper bindings, and cron-nested persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve suspended dynamic lanes:** `src/process/command-queue.ts:319-321`
  - body: Current main uses `setCommandLaneConcurrency(lane, 0)` from `suspendSession()` to hold a session lane until auto-resume. This guard deletes idle non-persistent lanes with `maxConcurrent <= 1`, so a suspended dynamic lane can be removed and the next enqueue recreates it at default concurrency, bypassing the suspension.
  - confidence: 0.9
- **[P2] Notify active-task waiters on rejection:** `src/process/command-queue.ts:298`
  - body: The catch path removes the active task id, but now calls only `pump(true)` and skips `notifyActiveTaskWaiters()`. `waitForActiveTasks()` relies on that notification, so restart/recovery waits can sit until timeout after a rejected task has already drained.
  - confidence: 0.91
- **[P2] Restore snapshot helper bindings:** `src/process/command-queue.test.ts:30`
  - body: This replaces the test module bindings for `getCommandLaneSnapshot` and `getCommandLaneSnapshots`, but the existing snapshot tests still call both helpers. The PR head therefore leaves undefined identifiers and fails test typechecking.
  - confidence: 0.97
- **[P3] Include the cron-nested lane in persistence:** `src/process/command-queue.ts:115-120`
  - body: Current main configures `CommandLane.CronNested` as a well-known startup lane for cron inner work, but the persistent set omits it while preserving the other configured lanes. That makes default-concurrency cron-nested state eligible for pruning despite the PR's well-known-lane policy.
  - confidence: 0.84
