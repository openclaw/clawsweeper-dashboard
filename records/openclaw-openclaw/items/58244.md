## Summary

Keep open: the command-queue leak is source-reproducible and this is the only live matching PR I found, but the current patch is not mergeable because it drops rejected-task waiter notifications, removes still-used test bindings, can bypass zero-concurrency lane suspensions, omits `cron-nested` from its persistent lane set, and lacks real behavior proof.

## What This Changes

The PR adds idle dynamic lane pruning to `src/process/command-queue.ts` and regression tests in `src/process/command-queue.test.ts`.

## Best Possible Solution

Land a narrow queue fix that prunes only idle default-concurrency dynamic lanes, preserves startup and explicitly tuned lanes including suspension state, keeps active-task waiter semantics unchanged on success and rejection, and keeps existing queue snapshot tests compiling.

## Reproduction Assessment

Yes, by source inspection. Current main creates unique per-session and auth-probe lane entries in a process-global Map, and the only lane-Map clearing path is the test reset helper.

## Solution Assessment

No, not as submitted. Idle dynamic-lane pruning is the right direction, but this patch needs the waiter notification, snapshot bindings, zero-concurrency preservation, and `cron-nested` persistence fixed before it is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve zero-concurrency lane suspensions:** `src/process/command-queue.ts:321`
  - body: The prune guard deletes dynamic lanes when `maxConcurrent <= 1`, which includes lanes intentionally suspended with `setCommandLaneConcurrency(lane, 0)`. `suspendSession()` uses that zero-concurrency state to block quota/circuit lanes until auto-resume; after deletion, the next enqueue recreates the lane at the default concurrency and bypasses the suspension.
  - confidence: 0.9
- **[P2] Notify waiters after rejected lane tasks:** `src/process/command-queue.ts:298`
  - body: The rejection path now calls `pump(true)` without `notifyActiveTaskWaiters()` after `completeTask()` removes the active task id. `waitForActiveTasks()` wakes through those notifications or its timeout, so restart or recovery waits can sleep until timeout after a rejected task has already drained.
  - confidence: 0.91
- **[P2] Restore the snapshot helper bindings:** `src/process/command-queue.test.ts:30`
  - body: The diff replaces the top-level `getCommandLaneSnapshot` and `getCommandLaneSnapshots` bindings with `getLaneCountForTest`, but existing tests still call both helpers. That leaves undefined identifiers in the PR head and blocks test typechecking.
  - confidence: 0.97
- **[P3] Include cron-nested in persistent lanes:** `src/process/command-queue.ts:115-120`
  - body: Current main configures `CommandLane.CronNested` as the dedicated startup lane for cron agent work. Omitting it from `PERSISTENT_LANES` lets default-concurrency `cron-nested` state be pruned while the other startup lanes are preserved, so the persistent-lane policy no longer matches the configured lane set.
  - confidence: 0.84
