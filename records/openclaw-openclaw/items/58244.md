## Summary

Keep open: this is a real source-reproducible command-queue leak and the pruning belongs in `src/process/command-queue.ts`, but the current PR head still changes waiter behavior on rejected tasks, breaks existing test bindings, omits `cron-nested` from the persistent lane set, and lacks after-fix real behavior proof.

## What This Changes

The PR adds pruning of idle dynamic command-queue lanes after task completion, lane clearing, and reset, plus regression tests and a lane-count test helper.

## Best Possible Solution

Land a narrow command-queue fix that prunes only idle default-concurrency dynamic lanes, preserves all startup and tuned lanes, keeps active-task waiter semantics unchanged on success and rejection, and leaves existing snapshot tests compiling.

## Reproduction Assessment

Yes, by source inspection. Current main creates per-session and auth-probe lane entries in a global Map and has no production path that deletes them after idle completion.

## Solution Assessment

No, not as submitted. The owner-level pruning approach is appropriate, but this patch needs the rejected-task waiter notification, snapshot helper bindings, and `cron-nested` persistence repaired first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Notify active waiters when lane tasks reject:** `src/process/command-queue.ts:295-298`
  - body: The rejected-task path no longer calls `notifyActiveTaskWaiters()` after `completeTask()` removes the active task id. `waitForActiveTasks()` wakes through those notifications or its timeout, so restart or recovery waits can now sleep until timeout after a rejected task has already drained.
  - confidence: 0.91
- **[P2] Restore the snapshot helper bindings:** `src/process/command-queue.test.ts:30`
  - body: The patch replaces the `getCommandLaneSnapshot` and `getCommandLaneSnapshots` test-module bindings with `getLaneCountForTest`, but existing tests still call both helpers below. That leaves the PR head with undefined identifiers and blocks test typechecking.
  - confidence: 0.97
- **[P3] Keep cron-nested as a persistent lane:** `src/process/command-queue.ts:115-120`
  - body: Current main configures `CommandLane.CronNested` as a well-known startup lane for cron agent work. Omitting it from `PERSISTENT_LANES` lets default-concurrency `cron-nested` state be pruned while the other startup lanes are preserved, so the persistent-lane policy no longer matches the configured lane set.
  - confidence: 0.84
