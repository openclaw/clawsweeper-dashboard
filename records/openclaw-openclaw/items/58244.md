## Summary

Keep open: the command-queue leak is real on current main and the PR is a narrow fix in the right owner area, but the latest head still has blocking source and test regressions plus missing changelog coverage.

## What This Changes

This PR adds idle dynamic-lane pruning to `src/process/command-queue.ts` after task completion, lane clearing, and reset, plus a test-only lane count helper and regression tests.

## Best Possible Solution

Land a narrow command-queue fix that prunes only fully idle dynamic/default-concurrency lanes, preserves all configured well-known lanes and custom concurrency, keeps active-task waiter semantics unchanged, and records the gateway memory fix in the active changelog.

## Reproduction Assessment

Yes. On current main, enqueueing work on unique `session:*` or `auth-probe:*` lanes creates entries in the process-global lanes Map, and static inspection shows no production deletion path after those lanes become idle.

## Solution Assessment

No, not as submitted. Pruning in `src/process/command-queue.ts` is the right maintainable fix, but this head must restore rejected-task waiter notification, snapshot test bindings, cron-nested persistence, and changelog coverage before merge.
