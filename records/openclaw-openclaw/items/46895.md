## Summary

Keep open. The underlying Control UI bug is still present on current main and the latest release, but this PR still routes documented `/model list` and `/model status` modes to the bare summary response, lacks after-fix real behavior proof, and overlaps the linked issue plus a sibling implementation PR that maintainers should dedupe or replace.

## What This Changes

The PR changes the Control UI `/model` slash-command executor so `status` and `list` use the local model-info branch, trims the model argument once, and adds executor tests for those inputs.

## Best Possible Solution

Land or replace one narrow Control UI fix that routes `/model list` and `/model status` to the documented informational behavior, preserves normal model selection, and includes focused regression coverage plus redacted real Control UI proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `/model status` parses to `args: "status"`, Control UI dispatch passes that arg unchanged, and `executeModel` sends every non-empty `/model` arg to `sessions.patch`; I did not run a live Control UI session.

## Solution Assessment

No. Avoiding the bad `sessions.patch` call is necessary, but this PR collapses `/model status` and `/model list` into the bare summary response instead of using the documented detailed status and list/picker behaviors.
