## Summary

Keep open. Current main still has no Discord voice realtime-provider mode: the accepted config, docs, tests, and runtime all point to segmented WAV capture, batch STT, a text agent turn governed by `voice.model`, and separate TTS playback; the related #73699 architecture discussion is open and does not supersede this Gemini Live request.

## What This Changes

Request adds a Discord voice configuration and runtime path for Gemini Live or another realtime audio provider, separate from `channels.discord.voice.model` as the text LLM override.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, preserves `voice.model` for the existing text pipeline, and specifies Discord session, transcript, authorization, and tool behavior before implementation.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, runtime segment flow, tests, and docs to confirm realtime provider keys are not accepted and Discord voice still uses WAV capture, batch STT, text routing, and TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the maintainable solution should align with #73699 and the shared realtime voice bridge contract instead of adding a one-off Discord-only provider path.
