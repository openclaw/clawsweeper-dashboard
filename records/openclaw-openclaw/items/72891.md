## Summary

Keep open. Current main still lacks a Discord voice realtime-provider mode: the schema, docs, and runtime remain a segment-based STT -> text agent -> TTS path, while Gemini Live realtime support exists for other voice surfaces and the related voice-as-I/O discussion does not fully cover this request.

## What This Changes

The request asks for Discord voice channels to support a realtime audio provider such as Gemini Live instead of only the existing STT -> text LLM -> TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice bridge, preserves voice.model as the text-pipeline LLM override, and specifies Discord session, transcript, authorization, and tool behavior before implementation.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection: current main's schema and runtime only accept the text voice pipeline and the docs describe that pipeline explicitly.

## Solution Assessment

Unclear. Reusing the shared realtime voice bridge is plausible, but the best design depends on the open voice-as-I/O/session-transcript direction and Discord-specific authorization, transcript, and tool semantics.
