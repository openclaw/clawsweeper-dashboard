## Summary

Keep open. Current main still does not provide a Discord voice realtime provider mode: Discord voice remains a WAV capture, batch STT, text agent turn, and TTS playback pipeline, while the shared realtime voice provider bridge exists on adjacent surfaces. The linked #73699 discussion is related architecture input for session/transcript behavior, not a duplicate shipped fix.

## What This Changes

Request to add a Discord voice configuration and runtime mode that can use Gemini Live or another end-to-end realtime audio provider instead of the current STT -> text LLM -> TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode in the Discord plugin that reuses the shared realtime voice provider bridge, keeps channels.discord.voice.model as the text-pipeline LLM override, and explicitly specifies Discord session binding, transcript emission, delivery, authorization, interruption, and realtime consult/tool behavior before schema/runtime/docs/tests are added.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: source and docs show strict config rejects realtime provider keys and the runtime still uses WAV capture, batch STT, text routing, and TTS playback.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but maintainers should first settle the #73699 session/transcript and permission/tool policy so the code change is narrow and compatible with existing Discord voice behavior.
