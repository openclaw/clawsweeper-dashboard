## Summary

Keep open. Current main still has no Discord voice realtime-provider mode: Discord voice remains a segment-based STT -> text agent -> TTS pipeline, while Gemini/OpenAI realtime voice providers are documented and wired for other voice surfaces, and the related #73699 direction is still open architecture discussion rather than a merged superseding fix.

## What This Changes

Add a Discord voice configuration/runtime path that can use Gemini Live or another realtime audio provider instead of only the existing text-mediated voice pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice bridge/providers, preserves voice.model as the text-pipeline LLM override, and specifies Discord session, transcript, authorization, and tool behavior before schema, runtime, docs, and tests land.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence source/docs inspection: strict Discord voice schema, docs, and runtime all show only the STT -> text agent -> TTS path.

## Solution Assessment

Unclear. A separate Discord realtime provider branch is plausible, but the maintainable fix depends on the open voice-as-I/O/session-transcript direction rather than a narrow code repair.
