## Summary

Keep open. Current main still does not provide a Discord voice realtime-provider mode: Discord voice only accepts the existing text-pipeline settings and still runs segmented WAV capture, batch STT, a text agent turn, and separate TTS playback, while the related #73699 session/transcript/tool-routing direction remains open.

## What This Changes

Request adding a Discord voice configuration/runtime mode that can route voice channels through Gemini Live or another shared realtime audio provider instead of the current STT to text-agent to TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, preserves `channels.discord.voice.model` as the text-pipeline LLM override, and specifies Discord session binding, transcript persistence, authorization, barge-in, and tool/consult behavior before adding schema, runtime, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, docs, segment runtime, and realtime-provider call sites to confirm the requested provider mode is absent.

## Solution Assessment

Unclear. A separate Discord voice realtime config branch is plausible, but the maintainable fix depends on the open #73699 voice-as-I/O decision and should not be treated as a narrow automated repair yet.
