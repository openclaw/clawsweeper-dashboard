## Summary

Keep open. Current main still has no Discord voice realtime-provider mode: the strict Discord voice config, public docs, and runtime all describe and implement WAV capture to STT to text-agent response to TTS, while Gemini Live/realtime providers exist only on adjacent realtime voice surfaces.

## What This Changes

Request support for selecting a Gemini Live-style realtime audio provider for Discord voice-channel conversations, separate from the existing text-response `channels.discord.voice.model` override.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice bridge, preserves `voice.model` as the text-pipeline LLM override, and settles Discord session, transcript, authorization, barge-in, and tool-consult semantics before schema, runtime, docs, and tests land.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, docs, segment runtime, and realtime-provider call sites to confirm the requested provider mode is absent.

## Solution Assessment

Unclear. A Discord realtime provider branch is plausible, but the maintainable design depends on maintainer direction for the open voice-as-I/O/session-transcript work rather than a narrow code repair.
