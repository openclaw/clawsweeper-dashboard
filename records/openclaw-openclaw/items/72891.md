## Summary

Keep open. Current main still lacks a Discord voice realtime audio path: Discord voice remains a segment-based STT -> text agent -> TTS pipeline, while Gemini Live/realtime provider support exists in adjacent Voice Call, Google Meet, Talk, and VoiceClaw surfaces only. The open #73699 voice-as-I/O discussion is relevant architecture input, not an implemented fix or exact duplicate.

## What This Changes

Request an additive Discord voice configuration/runtime path that can use Gemini Live or another realtime audio provider instead of only selecting the text LLM used after STT.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, keeps channels.discord.voice.model as the current text-pipeline LLM override, and documents/tests Discord-specific session binding, transcript emission, delivery, authorization, barge-in, and realtime consult/tool behavior.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: inspect the strict Discord voice schema, Discord docs, and segment runtime to confirm realtime provider keys are not accepted and the runtime still uses WAV capture, batch STT, a text agent turn, and TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the best implementation depends on maintainer direction for #73699 session/transcript semantics and permission/tool policy before a narrow code change can be judged maintainable.
