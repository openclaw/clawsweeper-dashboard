## Summary

Keep open. Current main still has no Discord voice realtime-provider mode: Discord voice is a strict segment-based STT -> text agent -> TTS pipeline, while the shared realtime voice provider path is currently wired for Voice Call, Google Meet, and Talk-style surfaces rather than Discord voice channels.

## What This Changes

Request a Discord voice configuration path that can select Gemini Live or another realtime audio provider instead of only choosing the text LLM used after Discord voice STT.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, preserves `channels.discord.voice.model` as the current text-pipeline LLM override, and documents Discord session, transcript, authorization, barge-in, and tool/consult behavior.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, runtime segment flow, and docs to confirm realtime provider keys are not accepted and Discord voice still uses WAV capture, batch STT, text routing, and TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config branch is plausible, but the maintainable design depends on the open #73699 session/transcript/routing decision and should not be automated before that product boundary is chosen.
