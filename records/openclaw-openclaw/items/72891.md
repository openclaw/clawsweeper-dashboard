## Summary

Keep open. Current main still has no Discord voice realtime-provider mode: Discord voice accepts only the existing voice config fields, captures segmented WAV audio, runs batch STT, sends a text agent turn controlled by `voice.model`, and plays separate TTS; the open #73699 discussion is related architecture work but does not implement or supersede this Gemini Live request.

## What This Changes

Request a Discord voice configuration and runtime path that can use Gemini Live or another realtime audio provider instead of the current STT to text LLM to TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, keeps `channels.discord.voice.model` as the existing text-pipeline LLM override, and specifies Discord routing, transcript, authorization, barge-in, and tool/consult behavior before adding schema, runtime, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, runtime segment flow, and docs to confirm realtime provider keys are not accepted and Discord voice still uses WAV capture, batch STT, text routing, and TTS.

## Solution Assessment

Unclear. A separate Discord realtime config path is plausible, but the maintainable solution should be designed against #73699 and the shared realtime voice contracts rather than bolting a provider field onto the existing text pipeline.
