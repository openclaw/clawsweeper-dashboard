## Summary

Keep open. Current main still runs Discord voice as chunked capture, batch STT, text agent turn, and TTS playback; Gemini Live/realtime voice provider support exists in adjacent Voice Call, Google Meet, Talk, and VoiceClaw surfaces but is not wired into the Discord voice schema or runtime. The linked #73699 discussion is a real architecture dependency for Discord session, transcript, delivery, and routing behavior, not a shipped fix or exact duplicate.

## What This Changes

Request adding a Discord voice configuration and runtime mode that can use Gemini Live or another end-to-end realtime audio provider instead of only selecting the text LLM used after STT.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, keeps channels.discord.voice.model as the text-pipeline LLM override, and explicitly specifies Discord session binding, transcript emission, delivery, authorization, barge-in, and realtime consult/tool behavior before schema/runtime/docs/tests are added.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: inspect the strict Discord voice schema, manager/segment runtime, and docs to confirm realtime provider keys are not accepted and Discord voice still runs through WAV capture, batch STT, text routing, and TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the maintainable implementation needs maintainer direction for #73699 session/transcript behavior and permission/tool policy before a narrow code change can be judged best.
