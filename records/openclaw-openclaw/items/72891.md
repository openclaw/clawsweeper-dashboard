## Summary

Keep open. Current main still does not provide a Discord voice realtime-provider path: the schema, type, runtime, tests, and docs all show segmented WAV capture, batch STT, a text agent turn controlled by `voice.model`, and separate TTS playback. The linked #73699 is relevant architecture work for Discord session/transcript behavior, but it is open and does not implement or supersede this Gemini Live/realtime-provider request.

## What This Changes

The issue requests an additive Discord voice configuration and runtime path that can use a realtime audio provider such as Gemini Live instead of the current segmented STT -> text LLM -> TTS path.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider/bridge contract, keeps `channels.discord.voice.model` as the existing text-pipeline LLM override, and specifies Discord session binding, transcript delivery, authorization, and tool/consult behavior before adding schema, runtime, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, runtime segment flow, and docs to confirm realtime provider keys are not accepted and Discord voice still uses WAV capture, batch STT, text routing, and TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the maintainable shape depends on the open #73699 session/transcript/routing decision and should not be automated before that product boundary is chosen.
