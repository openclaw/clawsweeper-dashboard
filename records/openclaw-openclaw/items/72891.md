## Summary

Keep open. Current main still lacks a Discord voice realtime-provider mode: the schema accepts only the existing voice settings, docs describe `voice.model` as a response-LLM override, and runtime still captures WAV segments, runs batch STT, sends a text ingress turn, and plays separate TTS audio; the related #73699 voice-as-I/O discussion remains open and does not supersede this Gemini Live request.

## What This Changes

Request an additive Discord voice configuration/runtime path that can use Gemini Live or another realtime audio provider instead of only the current STT-to-text-LLM-to-TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider/bridge contract, preserves `channels.discord.voice.model` as the current text-pipeline LLM override, and specifies Discord session binding, transcript delivery, authorization, interruption, and tool/consult behavior before adding schema, runtime, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, docs, and voice runtime to confirm realtime provider keys are not accepted and Discord voice still uses WAV capture, batch STT, text routing, and TTS.

## Solution Assessment

Unclear. A separate Discord realtime config path is plausible, but the maintainable shape depends on the open #73699 voice-as-I/O decision and related transcript/routing/security boundaries.
