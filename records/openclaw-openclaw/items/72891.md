## Summary

Keep open. Current main still lacks a Discord voice realtime-provider mode: the schema, docs, and runtime remain a segment-capture, STT, text-agent, TTS pipeline, while Gemini Live/realtime voice support exists only on other voice surfaces. This needs maintainer architecture direction rather than conservative cleanup.

## What This Changes

Add a Discord voice configuration and runtime mode that can route voice channel audio through a realtime audio provider such as Gemini Live instead of the current text-mediated pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice bridge, preserves voice.model as the text-pipeline LLM override, and explicitly specifies Discord session binding, transcripts, authorization, barge-in, and tool/consult behavior before schema, runtime, docs, and tests land.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, docs, segment runtime, and realtime-provider call sites to confirm the requested provider mode is absent.

## Solution Assessment

Unclear. A separate Discord realtime provider mode is plausible, but the maintainable design depends on the open voice-as-I/O/session-transcript direction rather than a narrow one-file repair.
