## Summary

Keep open. Current main still does not provide a Discord voice realtime-provider path: Discord voice remains a segmented capture, batch STT, text agent turn, and TTS playback pipeline, while Gemini Live/realtime voice support is wired for adjacent Voice Call, Google Meet, Talk, and shared realtime bridge surfaces. The linked #73699 discussion is relevant architecture work for Discord session/transcript/routing behavior, but it is not an implementation or exact duplicate of this realtime-provider request.

## What This Changes

Add a Discord voice configuration/runtime path that can select an end-to-end realtime audio provider such as Gemini Live instead of only using `channels.discord.voice.model` as a text-response LLM override.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice bridge/provider contract, keeps `channels.discord.voice.model` as the existing text-pipeline LLM override, and explicitly specifies Discord session binding, transcript emission, delivery, authorization, barge-in, and tool/consult behavior before adding schema, runtime, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: inspect the strict Discord voice schema, runtime segment flow, and docs to confirm realtime provider keys are not accepted and Discord voice still uses WAV capture, batch STT, text routing, and TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the maintainable shape depends on the open #73699 session/transcript/routing decision and should not be automated before that product boundary is chosen.
