## Summary

Keep open. Current main still implements Discord voice as WAV capture, STT, text-agent response, and TTS playback, while Gemini Live/realtime provider support is wired for adjacent Voice Call, Google Meet, and Talk surfaces but not for Discord voice schema or runtime. The linked #73699 discussion is an open architecture dependency for Discord session/transcript behavior, not a shipped fix or exact duplicate.

## What This Changes

Request adding Discord voice channel support for selecting a realtime audio provider such as Gemini Live instead of only using the existing STT -> text LLM -> TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, keeps channels.discord.voice.model as the text-pipeline LLM override, and settles the #73699 session/transcript/delivery policy before updating schema, Discord voice runtime, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Static current-main verification is high confidence: the strict Discord voice schema rejects a realtime branch, and the runtime/docs show WAV capture -> STT -> text agent turn -> TTS rather than a native realtime provider session.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the best maintainable fix needs maintainer direction on voice-as-I/O, transcripts, routing, Discord permissions, and realtime tool policy before implementation.
