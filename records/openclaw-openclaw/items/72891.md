## Summary

Keep open. Current main still has no Discord voice realtime-provider mode: Discord voice records speaker audio into WAV segments, runs batch STT, sends a text agent turn, and then plays separate TTS, while the linked #73699 voice-as-I/O discussion leaves session, transcript, tool, and authorization behavior unresolved.

## What This Changes

Add a Discord voice configuration and runtime path that can use Gemini Live or another realtime audio provider instead of only the existing STT -> text LLM -> TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, keeps channels.discord.voice.model as the text-pipeline LLM override, and specifies Discord session binding, transcript, authorization, barge-in, and tool/consult behavior before adding schema, runtime, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, runtime segment flow, and docs to confirm realtime provider keys are not accepted and Discord voice still uses WAV capture, batch STT, text routing, and TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the maintainable shape depends on the open #73699 session/transcript/routing decision and should not be automated before that product boundary is chosen.
