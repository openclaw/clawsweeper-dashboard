## Summary

Keep open. Current main still has no Discord voice realtime-provider mode: the Discord voice config remains a strict text-pipeline shape, docs describe WAV capture to STT to text agent to TTS, and the runtime follows that path even though shared realtime voice providers exist for other surfaces.

## What This Changes

Request a Discord voice-channel realtime provider mode, such as Gemini Live, separate from the existing text-response `channels.discord.voice.model` override.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice bridge, preserves `voice.model` as the text-pipeline LLM override, and specifies Discord session binding, transcript persistence, authorization, barge-in, and tool/consult behavior before schema, runtime, docs, and tests land.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, docs, segment runtime, and realtime-provider call sites to confirm the requested provider mode is absent.

## Solution Assessment

Unclear. A separate Discord realtime provider branch is plausible, but the maintainable design depends on the open voice-as-I/O/session-transcript direction rather than a narrow code repair.
