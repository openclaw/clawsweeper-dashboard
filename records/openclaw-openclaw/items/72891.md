## Summary

Keep open. Current main still has no Discord voice realtime-provider mode: Discord voice is configured and documented as a segmented WAV capture, batch STT, text agent turn controlled by `voice.model`, and separate TTS playback, while the shared Gemini Live realtime provider is only wired through other surfaces such as Voice Call and Google Meet.

## What This Changes

Request a Discord voice configuration/runtime path that can select Gemini Live or another realtime audio backend separately from the existing `channels.discord.voice.model` text-response override.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, preserves `channels.discord.voice.model` as the existing text-pipeline LLM override, and documents session binding, transcript delivery, authorization, barge-in, and tool/consult behavior before schema/runtime/tests land.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, Discord voice docs, and voice manager/segment/TTS runtime to confirm realtime provider keys are not accepted and the runtime remains WAV-to-STT-to-text-to-TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the maintainable fix should be chosen alongside the open #73699 voice-as-I/O decision rather than implemented as a narrow schema toggle alone.
