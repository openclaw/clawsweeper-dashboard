## Summary

Keep open. Current main still does not provide a Discord voice realtime-provider mode: the Discord voice config and runtime use segmented capture, batch STT, a text agent turn, and separate TTS playback; the related #73699 session-bridging discussion is open and does not implement or supersede this Gemini Live request.

## What This Changes

Request adding a Discord voice configuration/runtime path that can route voice-channel conversations through an end-to-end realtime audio provider such as Gemini Live instead of only using the existing text-mediated voice pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode on top of the shared realtime voice bridge contract while preserving `channels.discord.voice.model` as the current text-pipeline override, then update schema, runtime, docs, and focused tests once the session/transcript/routing behavior is chosen.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, docs, and runtime segment flow to confirm no realtime provider keys are accepted and Discord voice still uses WAV capture, batch STT, text routing, and TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the maintainable shape should be decided alongside the open #73699 voice-as-I/O architecture question before implementation.
