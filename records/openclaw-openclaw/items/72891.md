## Summary

Keep open. Current main still has no Discord voice realtime-provider mode, and the related voice-as-I/O tracker is adjacent rather than a canonical replacement; this remains valid feature work that needs maintainer design direction.

## What This Changes

Request Discord voice config and runtime support for selecting Gemini Live or another realtime audio provider, separate from the existing `voice.model` text LLM override.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice bridge, preserves `voice.model` as the text-pipeline LLM override, and specifies Discord session binding, transcripts, authorization, barge-in, and tool/consult behavior before schema, runtime, docs, and tests land.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: inspect the strict Discord voice schema, docs, segment runtime, and realtime-provider call sites to confirm the requested provider mode is absent.

## Solution Assessment

Unclear. Reusing the shared realtime voice bridge is likely the right technical direction, but maintainers need to decide how it interacts with #73699 session bridging, transcript delivery, permissions, and realtime tool behavior before implementation.
