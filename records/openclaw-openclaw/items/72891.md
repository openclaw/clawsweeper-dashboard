## Summary

Keep open. Current main still has Discord voice implemented as captured WAV, batch STT, text agent ingress, and TTS playback, while the shared realtime voice provider path is used by Voice Call, Google Meet, and Talk but is not exposed in the Discord voice schema or runtime. The linked #73699 discussion is a real architecture dependency for Discord session binding and transcript/routing behavior, not a duplicate shipped fix.

## What This Changes

The issue requests a Discord voice realtime audio mode that can select an end-to-end realtime audio provider such as Gemini Live instead of only choosing the text response model for the existing STT-to-LLM-to-TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, keeps channels.discord.voice.model as the current text-pipeline LLM override, and specifies Discord session binding, transcript emission, delivery, authorization, barge-in, and realtime consult/tool behavior before schema/runtime/docs/tests are added.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: inspect the strict Discord voice schema and runtime to see that realtime provider keys are not accepted and Discord voice still runs through WAV capture, batch STT, text agent routing, and TTS.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible, but the maintainable solution depends on the open #73699 session/transcript direction and permission/tool policy before a code change can be judged the best path.
