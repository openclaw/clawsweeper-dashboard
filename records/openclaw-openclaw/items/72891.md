## Summary

Keep open. Current main still has no Discord voice realtime-provider mode: the schema, docs, and runtime keep Discord voice on segmented capture, batch STT, text agent response, and separate TTS, while the shared realtime voice bridge is only wired into adjacent surfaces such as Voice Call, Google Meet, and Talk.

## What This Changes

Add a Discord voice configuration and runtime mode that can use Gemini Live or another realtime audio provider instead of the current STT -> text LLM -> TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, keeps `channels.discord.voice.model` as the text-pipeline LLM override, and specifies Discord session, transcript, authorization, barge-in, and tool behavior before adding schema, runtime, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. Static current-main verification is high confidence: the strict Discord voice schema, docs, and segment runtime all show the requested realtime provider mode is absent.

## Solution Assessment

Unclear. A separate Discord voice realtime config branch is plausible, but the maintainable fix depends on the open #73699 voice-as-I/O decision rather than a narrow one-file repair.
