## Summary

Keep open: this PR addresses a real Google Chat `replyToMode` bug, but as written it breaks explicit reply targets in off mode, lacks inspectable real behavior proof, and should be revised or reconciled with https://github.com/openclaw/openclaw/pull/74235.

## What This Changes

Adds `replyToMode: "off"` guards to Google Chat monitor thread propagation and config-resolution tests.

## Best Possible Solution

Revise or fold this into the active Google Chat threading work: suppress only implicit inbound thread context and typing-placeholder threads when resolved `replyToMode` is `off`, preserve already-filtered `payload.replyToId`, add focused monitor regressions, and include inspectable proof.

## Reproduction Assessment

Yes, source-level. Configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook carrying `message.thread.name`; current main still seeds that thread into reply context and the typing placeholder before shared reply-mode filtering.

## Solution Assessment

No, not as currently written. The ingress and typing guards target the right bug, but the final-send guards bypass the shared explicit reply-tag contract; the safer fix is to filter implicit sources before delivery and leave `payload.replyToId` intact.
