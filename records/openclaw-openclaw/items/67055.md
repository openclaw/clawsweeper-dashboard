## Summary

Keep open: the bug is valid on current main, but this PR should not merge as-is because it drops explicit reply targets in off mode and lacks inspectable after-fix proof.

## What This Changes

The PR adds Google Chat `replyToMode: "off"` guards to monitor thread propagation and config-resolution tests for channel-level and per-account off mode.

## Best Possible Solution

Revise or fold the work into https://github.com/openclaw/openclaw/pull/74235: suppress only implicit inbound and typing-placeholder thread propagation when the resolved Google Chat account mode is `off`, preserve explicit `payload.replyToId`, add focused monitor regressions, and include inspectable real behavior proof.

## Reproduction Assessment

Yes, source-level. Configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook carrying `message.thread.name`; current main still seeds that thread into reply context and the typing placeholder before final reply filtering.

## Solution Assessment

No. The PR targets the right implicit ingress and typing sources, but the final-send guards bypass the shared explicit reply-tag contract; the safer fix is to filter implicit sources before delivery and leave already-filtered `payload.replyToId` intact.
