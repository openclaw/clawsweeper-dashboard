## Summary

Keep open: the Google Chat `replyToMode: "off"` bug is still present on current main, but this PR needs revision before merge because it would drop explicit reply targets and it lacks inspectable real behavior proof.

## What This Changes

The PR adds `replyToMode: "off"` guards to Google Chat monitor thread propagation and config-resolution tests for channel-level and per-account off mode.

## Best Possible Solution

Revise or fold this into the active Google Chat threading work: suppress only implicit inbound context and typing-placeholder threads when the resolved account mode is `off`, preserve explicit `payload.replyToId` at final delivery, add focused monitor regressions, and include inspectable real behavior proof.

## Reproduction Assessment

Yes, source-level. Configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook carrying `message.thread.name`; current main still seeds that thread into reply context and the typing placeholder before final reply filtering.

## Solution Assessment

No. The PR targets the right implicit ingress and typing sources, but the final-send guards bypass the shared explicit reply-tag contract; the safer fix is to filter implicit sources before delivery and leave `payload.replyToId` intact.
