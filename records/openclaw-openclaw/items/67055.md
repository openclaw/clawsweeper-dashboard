## Summary

Keep open: the reported Google Chat off-mode threading bug is still source-reproducible on current main, but this PR has a blocking correctness issue and lacks inspectable external real-behavior proof.

## What This Changes

The branch adds Google Chat `replyToMode: "off"` guards to monitor thread propagation and final reply sends, plus config-resolution tests for channel and account-level off mode.

## Best Possible Solution

Revise or fold this into the current Google Chat threading work: suppress only implicit inbound and typing-placeholder thread propagation when resolved mode is `off`, preserve payload-driven explicit reply targets, and add focused monitor regression coverage.

## Reproduction Assessment

Yes, source-level. Configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook carrying `message.thread.name`; current main still seeds that value into reply context and the typing placeholder before shared reply-mode filtering can remove implicit final payload threading.

## Solution Assessment

No, not as currently written. The ingress and typing guards target the right bug, but the final-send guards bypass the shared explicit-reply contract; the safer fix is to filter implicit sources before delivery and leave `payload.replyToId` intact.
