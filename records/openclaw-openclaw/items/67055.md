## Summary

Keep open: the underlying Google Chat off-mode bug is still source-reproducible on current main, but this PR is not mergeable as-is because it drops explicit reply targets at final delivery and lacks inspectable real behavior proof.

## What This Changes

The branch adds `replyToMode: "off"` guards to Google Chat inbound thread context, typing indicators, and final text/media sends, plus config-resolution tests.

## Best Possible Solution

Revise or fold this into the active Google Chat threading work: suppress only implicit inbound and typing-placeholder threads when the resolved account mode is `off`, preserve final `payload.replyToId` for explicit targets, and add monitor regression coverage.

## Reproduction Assessment

Yes, source-level. Configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook with `message.thread.name`; current main still seeds that value into reply context and the typing placeholder.

## Solution Assessment

No, not as currently written. The ingress and typing guards target the right bug, but the final-send guards bypass the shared explicit-reply contract; the safer fix is to filter implicit sources before delivery and leave `payload.replyToId` intact.
