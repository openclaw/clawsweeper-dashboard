## Summary

Keep open for maintainer-guided revision. Current main still has the Google Chat off-mode threading bug at webhook ingress and the typing placeholder, but the PR is not the best merge shape because it also drops already-filtered explicit final reply targets and has drifted from current main's split delivery code; #74235 is the active overlapping Google Chat threading work to review or consolidate with.

## What This Changes

The PR adds `replyToMode: "off"` guards to Google Chat monitor thread propagation and adds Google Chat config-resolution tests for channel-level and per-account off mode.

## Best Possible Solution

Revise or fold the branch into the current Google Chat threading work: suppress only implicit inbound `ReplyToId`/`ReplyToIdFull` and typing-placeholder thread propagation when the resolved Google Chat account has `replyToMode: "off"`, keep final delivery honoring already-filtered `payload.replyToId`, and add a focused monitor regression for off-mode DM replies plus explicit reply preservation.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook that includes `message.thread.name`; current main still seeds that thread into reply context and the typing placeholder, while the PR's final-send guard can be checked against existing shared tests that explicit reply tags survive off mode.

## Solution Assessment

No. The PR targets the right implicit ingress and typing sources, but guarding final delivery on account config bypasses the shared payload contract; the safer solution is to filter implicit sources before delivery and leave `payload.replyToId` intact at final text/media sends.
