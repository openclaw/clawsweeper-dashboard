## Summary

Keep open. Current main still has the Google Chat off-mode threading bug at webhook ingress and the typing placeholder, while this PR is useful but not merge-ready because its final-send guards would flatten explicit reply targets that the shared reply plumbing intentionally preserves.

## What This Changes

The PR adds `replyToMode: "off"` guards to Google Chat webhook reply context, typing placeholder sends, and final text/media sends, plus Google Chat config-resolution tests for channel-level and per-account off mode.

## Best Possible Solution

Revise or fold the PR into the current Google Chat threading work: suppress only implicit inbound `ReplyToId`/`ReplyToIdFull` and typing-placeholder thread propagation when the resolved Google Chat account has `replyToMode: "off"`, keep final delivery honoring already-filtered `payload.replyToId`, add focused monitor regressions, and include the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook carrying `message.thread.name`; current main still seeds that thread into reply context and the typing placeholder, while the PR's final-send guards can be checked against the shared off-mode explicit-tag contract.

## Solution Assessment

No. The PR targets the right implicit ingress and typing sources, but guarding final delivery on account config bypasses the shared payload contract; the safer solution is to filter implicit sources before delivery and leave `payload.replyToId` intact at final text/media sends.
