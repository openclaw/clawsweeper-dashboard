## Summary

Keep open: this PR targets a real Google Chat `replyToMode: "off"` bug, but the branch still strips explicit reply targets at final delivery and lacks inspectable real behavior proof.

## What This Changes

Adds `replyToMode: "off"` guards to Google Chat monitor thread propagation and adds config-resolution tests for channel-level and per-account off mode.

## Best Possible Solution

Revise or fold this into the active Google Chat threading work: suppress only implicit inbound thread context and typing-placeholder threads when the resolved account mode is `off`, preserve already-filtered `payload.replyToId`, add focused monitor regression coverage, and include inspectable proof.

## Reproduction Assessment

Yes, source-level. Configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook carrying `message.thread.name`; current main still seeds that thread into reply context and the typing placeholder before shared reply-mode filtering.

## Solution Assessment

No, not as currently written. The ingress and typing guards target the right bug, but the final-send guards bypass the shared explicit reply-tag contract; the safer fix is to filter implicit sources before delivery and leave `payload.replyToId` intact.
