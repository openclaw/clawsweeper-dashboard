## Summary

Keep open: this is a real current-main Google Chat threading bug, but the PR is not mergeable as written because it drops explicit reply targets in `replyToMode: "off"`, lacks inspectable real behavior proof, and overlaps active Google Chat threading work.

## What This Changes

The PR adds `replyToMode: "off"` guards around Google Chat inbound thread propagation, typing-placeholder threading, and final text/media reply sends, plus config-resolution tests.

## Best Possible Solution

Suppress only implicit inbound Google Chat thread metadata and typing-placeholder threading when the resolved account mode is `off`, preserve explicit reply targets after the shared filter, add focused monitor regression coverage, and reconcile with the active thread-resource PR.

## Reproduction Assessment

Yes, source-level: configure a Google Chat account with `replyToMode: "off"` and process an inbound webhook carrying `message.thread.name`; current main still seeds that thread into reply context and the typing placeholder before shared reply-mode filtering.

## Solution Assessment

No, not as currently written. The ingress and typing guards target the right bug, but the final-send guards bypass the shared explicit reply-tag contract; the safer fix is to filter implicit sources before delivery and leave already-filtered `payload.replyToId` intact.
