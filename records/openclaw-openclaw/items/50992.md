## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested external Plugin SDK access, so the contribution addresses a real gap from #49624; the blockers are that the direct re-export promotes a broad internal abort overload into a public SDK boundary and the public SDK API baseline/docs/contract alignment is missing. The author's trust-model argument is plausible for in-process plugins, but that is exactly the decision maintainers should make explicitly before this becomes a supported contract.

## What This Changes

The PR adds a named re-export block in `src/plugin-sdk/agent-runtime.ts` for five active embedded-run control helpers from `src/agents/pi-embedded-runner/runs.ts`.

## Best Possible Solution

Expose a supported active-run control surface for external channel plugins, preferably through a narrow SDK subpath or `api.runtime.agent` helper set, with a session-scoped abort wrapper unless maintainers explicitly approve the broader overload; then update SDK docs, API baseline/export checks, and focused contract coverage.

## Reproduction Assessment

Yes. The gap is reproducible by static inspection on current main: the helpers exist internally in `runs.ts`, but `openclaw/plugin-sdk/agent-runtime` does not export them even though that SDK subpath is published in `package.json`.

## Solution Assessment

No, not as currently proposed. The minimal re-export proves the missing surface, but the maintainable fix needs an explicit SDK boundary choice, abort-overload policy, and public SDK baseline/docs updates before merge.
