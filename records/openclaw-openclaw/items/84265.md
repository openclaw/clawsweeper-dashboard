## Summary

No blocking findings; keep open for maintainer review/merge. Current main still lacks the no-fallback overloaded same-profile retry, and this PR adds a bounded overloaded-only retry with focused coverage and supplied terminal proof.

## What This Changes

The PR adds one same-profile overloaded retry with backoff for no-fallback embedded-agent runs and covers prompt-side, assistant-side, and retry-exhaustion behavior.

## Best Possible Solution

Land the bounded overloaded-only retry if maintainers accept the default latency/provider-call tradeoff; otherwise make the retry policy explicit or defer to the broader configurable provider retry work.

## Reproduction Assessment

Yes from source, with medium confidence: current main rotates or surfaces overloaded failures after no auth-profile/fallback path remains, and the linked issue provides MiniMax overloaded logs. I did not run a live MiniMax outage because that failure is nondeterministic and this review is read-only.

## Solution Assessment

Yes, with maintainer acceptance of the default-behavior change: the patch is narrowly scoped to overloaded errors, caps the retry at one, and leaves rate-limit/auth/billing behavior covered as non-retry. A configurable retry policy would be the safer long-term product surface if maintainers do not want a hardcoded default.
