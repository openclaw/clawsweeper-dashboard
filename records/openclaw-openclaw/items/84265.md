## Summary

Keep open: the PR is aimed at a real provider-failover pain point, but it changes strict no-fallback behavior into broad same-profile retries without a dedicated retry budget/backoff policy, and the supplied proof does not show the retry path in a real setup.

## What This Changes

The PR adds same-profile retry handling for overloaded, rate_limit, and server_error failover reasons when auth-profile rotation is unavailable and no model fallback is configured.

## Best Possible Solution

Adopt one maintainer-approved bounded retry policy for no-fallback transient provider failures, with explicit retry counts/backoff semantics, tests for prompt and assistant paths, and redacted real behavior proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main surfaces no-fallback transient failures after rotation is unavailable, while the PR's new branch would retry them until the broad run-loop cap. I did not run a live provider overload reproduction.

## Solution Assessment

No, not as written. Same-profile retry is plausible, but the maintainable fix needs a bounded policy/backoff and proof rather than relying on the global run-loop limit.
