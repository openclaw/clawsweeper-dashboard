## Summary

Keep open: this PR targets a real config persistence bug and the latest head fixes the earlier gateway-auth unsetPaths gap, but the global preserve-absent change still breaks existing delete/logout paths that call replaceConfigFile without unsetPaths, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes config merge-patch generation to preserve omitted keys by default, applies explicit unsetPaths deletions in write paths, and adds config persistence regression tests.

## Best Possible Solution

Keep the preservation-by-default goal, but make deletion intent reliable across config mutation APIs and channel logout/delete callers, then add focused regressions plus redacted real CLI or Docker restart proof.

## Reproduction Assessment

Yes from source inspection: current main creates null merge-patch deletes for keys omitted from a narrow nextConfig and applies those deletes during config writes. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No as currently patched: explicit unsetPaths is a maintainable direction, but the PR only migrates some deletion paths and leaves existing replaceConfigFile/logout/delete callers able to preserve stale fields. A safer solution must either infer deletion paths for full config mutations or migrate every intentional deletion caller with tests.

## Review Findings
