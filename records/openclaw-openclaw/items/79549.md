## Summary

This PR should stay open because it targets a real config persistence bug, but maintainer review and source inspection show the new preservation-by-default merge behavior still leaves intentional channel/account deletions able to be restored unless deletion paths are made explicit.

## What This Changes

The PR changes config merge-patch generation to preserve omitted keys by default, threads explicit unsetPaths through config write paths, forwards pruned gateway-auth credential deletions, and adds config persistence regression tests.

## Best Possible Solution

Keep the preservation-by-default direction, but make deletion intent reliable for full-source mutation callers or migrate every intentional deletion caller to unsetPaths with persisted-file regression coverage.

## Reproduction Assessment

Yes from source inspection: current main creates null delete markers for omitted target keys and applies them during config writes, matching the reported partial-write clobber path. I did not run a live Docker restart reproduction because this review is read-only.

## Solution Assessment

No, not yet: explicit unsetPaths is a maintainable direction, but the current PR does not cover existing channel/account deletion callers that delete by omission. The safer fix is to derive deletion paths for full mutations or migrate all intentional delete callers with persisted-file tests.
