## Summary

Keep open: this PR addresses a real config persistence bug and the latest head fixes the gateway-auth unsetPaths case, but the merge-patch semantic flip still leaves existing channel delete/logout callers able to preserve stale config or credentials, and the external PR still lacks required real after-fix proof.

## What This Changes

The PR changes config merge-patch generation to preserve omitted keys by default, applies explicit unsetPaths deletes in config write paths, forwards pruned gateway auth paths, and adds config persistence regression tests.

## Best Possible Solution

Keep the preservation-by-default direction only after every intentional deletion path either passes unsetPaths or a write-layer deletion tracker derives removed paths, then land with focused regressions and redacted real Docker/CLI restart proof.

## Reproduction Assessment

Yes, from source inspection: current main creates null merge-patch deletes for keys omitted from a narrow nextConfig and applies those deletes during config writes. I did not run tests because this review is constrained to read-only inspection.

## Solution Assessment

No, not as currently patched: explicit unsetPaths is a maintainable direction, but the PR only migrates some deletion paths and leaves existing replaceConfigFile/logout/delete callers able to preserve stale fields. A safer solution should migrate all intentional deletion callers or infer deletion intent in the write layer with tests.
