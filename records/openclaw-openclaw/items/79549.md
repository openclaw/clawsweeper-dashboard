## Summary

Keep this PR open: it addresses a valid current-main config clobbering bug, but the current patch still makes existing config deletion/logout paths depend on unsetPaths they do not pass, so stale account config or secrets can be preserved after a command reports deletion.

## What This Changes

The PR changes config merge-patch generation to preserve omitted keys by default, threads explicit unsetPaths through write paths, forwards pruned gateway auth credentials, and adds config persistence regression tests.

## Best Possible Solution

Preserve omitted keys for partial config writes while making intentional deletion reliable across config mutation, channel remove, and logout paths, backed by focused regressions and redacted real CLI or Docker restart proof.

## Reproduction Assessment

Yes from source inspection: current main creates null merge-patch deletes for keys omitted from a narrow nextConfig and applies that patch during config writes. I did not run tests because this review is read-only.

## Solution Assessment

No as currently patched: preservation-by-default is a maintainable direction, but deletion intent must be tracked or forwarded for existing full-config mutation callers before flipping createMergePatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve delete intents from config mutation callers:** `src/config/io.write-prepare.ts:28-32`
  - body: Skipping keys that are absent from target makes deletion depend on unsetPaths, but existing callers still delete fields or accounts from a full nextConfig and call replaceConfigFile without writeOptions. For example channel remove and Telegram/LINE logout delete account or token fields before writing; with this patch those omitted fields can be restored from the source snapshot, so the command can report deletion while the old config or secret remains on disk.
