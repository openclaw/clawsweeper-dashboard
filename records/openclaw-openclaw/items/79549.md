## Summary

Keep open: the config persistence bug is real and the preservation-by-default direction is plausible, but the current PR still makes intentional channel/account deletions depend on unsetPaths without migrating the existing deletion callers. It also still lacks required after-fix real behavior proof.

## What This Changes

The PR changes config merge-patch generation to preserve omitted keys, applies explicit unsetPaths in selected write paths, forwards pruned gateway auth paths, and adds config persistence regression tests.

## Best Possible Solution

Keep the preservation-by-default approach, but make deletion intent reliable across config mutation and channel/account removal paths, then add persisted-file regressions and redacted real Docker or CLI restart proof.

## Reproduction Assessment

Yes. Source inspection shows current main emits null merge-patch deletes for omitted keys and applies them during config writes; under the PR head, channel/account deletion by omission becomes source-reproducibly restored unless unsetPaths is provided.

## Solution Assessment

No as currently patched. Explicit unsetPaths is a maintainable direction, but the primitive change needs either a generic deletion tracker for full-source mutations or migration of all intentional deletion callers with persisted-file tests.
