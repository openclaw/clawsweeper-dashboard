## Summary

Keep open: the underlying config persistence bug is real, but the current branch still leaves deletion-by-omission unreliable for channel/account removal, lacks required real behavior proof, and is currently conflicting with main.

## What This Changes

The branch changes config merge-patch generation to preserve omitted keys, threads explicit unsetPaths into selected config writes, forwards pruned gateway auth paths, and adds config persistence regression tests.

## Best Possible Solution

Keep the preservation-by-default direction, but make intentional deletion reliable across replaceConfigFile/mutate callers with persisted-file regression coverage and then add redacted real CLI or Docker restart proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main creates null merge-patch deletes for omitted keys and applies those deletes during config writes, matching the reported partial-write clobber mechanism.

## Solution Assessment

No as currently patched. Explicit unsetPaths is a maintainable direction, but the branch only migrates selected deletion paths and leaves channel/account removal able to preserve stale fields unless deletion intent is made comprehensive.
