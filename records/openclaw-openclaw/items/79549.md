## Summary

Keep this PR open: it addresses a real current-main config persistence bug and the preservation-by-default direction is plausible, but the proposed primitive change still breaks deletion-by-omission callers and lacks required after-fix real behavior proof.

## What This Changes

The PR changes config merge-patch writes to preserve absent keys by default, threads explicit unsetPaths through selected write paths, and adds config persistence regression tests.

## Best Possible Solution

Keep the preservation-by-default goal, but make deletion intent reliable across full-source mutation and channel/account removal paths before merging, with persisted-file regressions and redacted real CLI or Docker restart proof.

## Reproduction Assessment

Yes from source inspection: current main emits null delete markers for keys omitted from a narrow nextConfig, and writeConfigFile applies that patch to the source snapshot. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No as currently patched: explicit unsetPaths is a maintainable direction, but the PR does not yet preserve deletion intent for existing channel/account removal callers. A safer fix should either derive deletion paths for full-source mutation callers or migrate intentional deletion callers with persisted-file regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve deletion intent for config removal callers:** `src/config/io.write-prepare.ts:31`
  - body: Skipping absent target keys makes deletion depend on unsetPaths, but existing callers still delete channel/account fields by returning a config with keys omitted and then call replaceConfigFile without writeOptions. Under this patch, those omitted fields can be restored from the source snapshot, so commands like channel remove can report deletion while stale config or secrets remain on disk.
