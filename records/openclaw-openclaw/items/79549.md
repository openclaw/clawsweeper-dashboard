## Summary

Keep open: this is a real config persistence bug, but the current PR head still has a blocking deletion-contract regression where channel/account removal can be preserved back onto disk. The PR also lacks required real behavior proof and currently has failing checks.

## What This Changes

The PR changes config merge-patch generation to preserve omitted keys by default, threads explicit `unsetPaths` through config writes, and adds config persistence tests.

## Best Possible Solution

Keep the preserve-by-default goal, but first make deletion intent reliable across config mutation APIs and channel delete/logout paths, then add persisted-file regressions and redacted real Docker or CLI restart proof.

## Reproduction Assessment

Yes, from source inspection: current main emits `null` merge-patch deletes for omitted keys and applies them during runtime snapshot writeback. I did not run the Docker/CLI repro because this review was read-only.

## Solution Assessment

No as currently patched: explicit `unsetPaths` is a maintainable direction, but the patch must either derive deletion paths for full-config mutations or migrate every intentional deletion caller before changing the primitive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve delete intents from channel removal:** `src/config/io.write-prepare.ts:31`
  - body: At PR head, `createMergePatch` treats omitted target keys as preserve-only, but `channels remove --delete` still calls `plugin.config.deleteAccount` and then `replaceConfigFile` without `writeOptions.unsetPaths`. Delete adapters such as the shared channel helper and Feishu omit channel/account fields, so this patch can report removal while restoring the old channel config or credentials on disk.
  - confidence: 0.9
