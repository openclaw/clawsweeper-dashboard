## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Verify tunnel ownership before signaling PIDs:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to the stored numeric PID after only checking that the PID exists. If the SSH tunnel exits and the OS reuses that PID, this can terminate an unrelated user process, and `status` will also treat the reused PID as a live tunnel.
  - confidence: 0.93
- **[P1] Reject occupied ports before reporting tunnel readiness:** `src/cli/tunnel-cli.ts:222-224`
  - body: The readiness loop treats any listener on `--port` as proof that the new SSH tunnel is ready. If another process already owns the port, SSH can still be briefly alive before `ExitOnForwardFailure` exits, so the command can write stale state and report success for the wrong listener.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel metadata:** `src/cli/tunnel-cli.ts:38-39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it first. On a fresh profile or custom `OPENCLAW_STATE_DIR`, `tunnel up` can start the detached SSH process and then throw `ENOENT`, leaving users without a PID file to stop or inspect the tunnel.
  - confidence: 0.9
