## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Verify tunnel ownership before signaling PIDs:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to the stored numeric PID after only checking that some process is alive. If the SSH child exits and the OS reuses that PID, this can terminate an unrelated process; persist and verify process identity before signaling.
  - confidence: 0.93
- **[P1] Reject occupied ports before reporting tunnel readiness:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any bound local port as proof that the new SSH tunnel is ready. A pre-existing gateway or SSH tunnel can make `isPortBound(port)` return true while the new SSH child is still briefly alive, causing false success and stale PID state.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:38-39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start the detached SSH process and then throw `ENOENT`, leaving an unmanaged tunnel.
  - confidence: 0.9
- **[P2] Surface unexpected port-probe errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves every non-`EADDRINUSE` listen error as false, so permission or OS errors become a confusing timeout instead of the real failure. Reuse the shared readiness helper or propagate unexpected probe errors.
  - confidence: 0.79
