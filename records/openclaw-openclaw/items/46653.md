## Summary

Keep open: current main still lacks a managed `openclaw tunnel` command, so the PR contains a plausible user-facing feature, but the proposed patch is not merge-ready because it has concrete detached-process/PID safety bugs, a false-ready port check, missing state-directory creation, no inspectable real behavior proof, and conflicts with the current CLI registration shape.

## What This Changes

The PR adds `openclaw tunnel up/down/status`, PID-file state for detached SSH port forwarding, CLI registration, and a new CLI docs page.

## Best Possible Solution

Rebase onto the current descriptor-based CLI, reuse or harden the existing SSH forwarding helper, verify process identity before status or termination, preflight occupied ports explicitly, create the state directory, and include redacted real command output or a recording.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: the diff shows missing state-directory creation, PID-only termination, and a readiness loop that treats any occupied local port as success. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

No. The feature direction is reasonable, but the patch should be rebased onto the current CLI catalog and should not add detached PID lifecycle management without ownership checks and explicit port preflight behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify tunnel PID ownership before signaling it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently owns the stored numeric PID. If the SSH child exits and the OS later reuses that PID, this can terminate an unrelated user process; persist and verify process identity before signaling.
  - confidence: 0.93
- **[P1] Reject occupied local ports before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any `EADDRINUSE` as proof that the new SSH tunnel is ready. An existing listener on `--port` can make `isPortBound(port)` return true while SSH is briefly alive, producing a false success and stale PID state.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start detached SSH and then throw `ENOENT`, leaving the tunnel unmanaged by `status` or `down`.
  - confidence: 0.95
- **[P3] Surface unexpected port-listen errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves `false` for listen errors other than `EADDRINUSE`, so permission or OS errors become a generic startup timeout. Propagate unexpected errors so users see the actual bind failure.
  - confidence: 0.82
