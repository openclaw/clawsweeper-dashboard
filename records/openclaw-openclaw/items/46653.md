## Summary

Keep open: this is a useful but still-unready feature PR. Current main still lacks a managed tunnel command, and the branch has unresolved correctness/security issues around PID ownership, occupied-port readiness, state writes, and current command registration.

## What This Changes

The PR adds `openclaw tunnel up/down/status`, persistent `tunnel.pid.json` state, root command registration, and a CLI docs page/nav entry.

## Best Possible Solution

If maintainers want this in core, rebase it onto the current descriptor catalog, reuse or extend the existing SSH helper, verify process identity before signaling, reject occupied ports before success, create state directories, and add focused lifecycle tests plus docs.

## Reproduction Assessment

Yes. The current-main absence and PR defects are source-reproducible from the command catalog, SSH helper, and added tunnel implementation; I did not run a live tunnel because this review is read-only.

## Solution Assessment

No. A managed tunnel CLI may be a reasonable core direction, but this patch is not the safest implementation until it is rebased and hardened around process identity, readiness, state writes, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Verify PID ownership before stopping tunnels:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` signals whatever process currently has the stored numeric PID. If the SSH tunnel exits and the OS reuses that PID, this can terminate an unrelated same-user process, so persist and verify a durable process identity before SIGTERM/SIGKILL.
  - confidence: 0.94
- **[P1] Reject occupied ports before reporting tunnel success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any listener on `--port` as proof that the new SSH tunnel is ready. If another process already owns the port, SSH can still be alive briefly before exiting, causing a false success and stale tunnel state.
  - confidence: 0.92
- **[P2] Register the command in the descriptor catalog:** `src/cli/program/register.subclis.ts:172-177`
  - body: Current main discovers root commands from `subcli-descriptors.ts` and loads them through `register.subclis-core.ts`. Adding `tunnel` only to `register.subclis.ts` misses the maintained help and lazy-registration path after rebase.
  - confidence: 0.87
- **[P2] Create the state directory before writing metadata:** `src/cli/tunnel-cli.ts:39-40`
  - body: `writeTunnelState` writes under `STATE_DIR` without ensuring that directory exists. On a fresh profile or custom `OPENCLAW_STATE_DIR`, `tunnel up` can start the detached SSH process and then throw `ENOENT`, leaving an unmanaged tunnel.
  - confidence: 0.95
- **[P3] Surface unexpected port probe errors:** `src/cli/tunnel-cli.ts:62-63`
  - body: `isPortBound` collapses every `server.listen` error other than `EADDRINUSE` into `false`, turning permission or OS errors into a generic startup timeout. Propagate unexpected errors or reuse the connect-based readiness helper.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces detached SSH process management and can signal unrelated processes through PID reuse unless process ownership is verified.

Concerns:

- **[medium] Unverified PID signaling can stop unrelated processes:** `src/cli/tunnel-cli.ts:291`
  - body: The PID file stores only a numeric PID, and `tunnel down` calls `killPid(state.pid)` without checking command identity, process start time, or another durable ownership marker. PID reuse can make this terminate an unrelated same-user process.
  - confidence: 0.93
