## Summary

This PR should stay open because the proposed `openclaw tunnel` feature is not implemented on current `main`, and the branch still has concrete correctness and process-safety blockers in the detached SSH/PID lifecycle plus missing real behavior proof.

## What This Changes

The PR adds a new `openclaw tunnel up/down/status` CLI, PID-file state for detached SSH port forwarding, command registration, and CLI docs/navigation.

## Best Possible Solution

Keep the managed tunnel feature under review, but require safer SSH tunnel lifecycle handling: preflight the local port, prove readiness from the spawned tunnel, create state storage before detaching, and verify process identity before status or termination.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: the branch writes PID state before ensuring the directory exists, treats any occupied port as readiness, and later signals a stored PID without proving it still belongs to the SSH tunnel. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

No. The feature direction may be useful, but this implementation should reuse or match the existing SSH helper's safer readiness checks and add explicit process-identity verification before introducing persistent detached lifecycle management.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify tunnel PID ownership before signaling it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently owns the stored numeric PID. If the SSH child exits and the OS later reuses that PID, this can terminate an unrelated user process; persist and verify process identity before signaling.
  - confidence: 0.93
- **[P1] Reject occupied local ports before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any `EADDRINUSE` as proof that the new SSH tunnel is ready. An existing listener on `--port` can make `isPortBound(port)` return true while SSH is briefly alive, producing a false success and stale PID state.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start detached SSH and then throw `ENOENT`, leaving the tunnel unmanaged by `status` or `down`.
  - confidence: 0.95
- **[P3] Surface unexpected port-listen errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves `false` for listen errors other than `EADDRINUSE`, so permission or OS errors become a generic startup timeout. Propagate unexpected errors so users see the actual bind failure.
  - confidence: 0.82
