## Summary

Keep open: current main still lacks a managed `openclaw tunnel` command, but this PR is not mergeable as-is because it has blocking process-safety/readiness defects, is stale against the current CLI command catalog, and lacks inspectable real behavior proof.

## What This Changes

The PR adds a top-level `openclaw tunnel up/down/status` CLI, PID-file tracking, docs navigation, and a CLI reference page for managed SSH port forwarding.

## Best Possible Solution

Port the feature onto the current descriptor-based CLI, reuse or harden the existing SSH helper semantics, add PID ownership/readiness safeguards with focused tests, and include redacted real behavior proof before merge.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: the added command can treat an unrelated local listener as success, writes state without creating the state directory, and signals a stored PID without ownership checks. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but the current branch duplicates SSH lifecycle logic and needs process-safety fixes plus a port to the current command catalog before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify PID ownership before signaling it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently has the stored PID. If the SSH child exits and the OS later reuses that PID, this command can terminate an unrelated user process, and `status` can also report the reused PID as active. Store and verify enough process identity before signaling, or avoid raw-PID control.
  - confidence: 0.93
- **[P1] Require the port to be free before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any `EADDRINUSE` as proof that the new SSH tunnel is ready. When another process already listens on the requested port, the SSH child can still be briefly alive while `isPortBound(port)` returns true for the unrelated listener, causing a false success and stale PID file.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or a new `OPENCLAW_STATE_DIR`, `tunnel up` can detach SSH successfully and then throw `ENOENT`, leaving a running tunnel that `tunnel down/status` cannot manage.
  - confidence: 0.95
- **[P3] Surface unexpected port-listen errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves `false` for every listen error other than `EADDRINUSE`, so permission or OS errors are reported later as a generic timeout. Propagate non-`EADDRINUSE` errors so users see the real bind failure.
  - confidence: 0.82

## Security Review
