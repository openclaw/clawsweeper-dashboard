## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require the port to be free before reporting success:** `src/cli/tunnel-cli.ts:203-205`
  - body: The startup loop treats any listener on the requested local port as success. If another process already owns the port, SSH can still be alive briefly before `ExitOnForwardFailure` exits, so this writes stale state and prints `Tunnel active` for a tunnel that never started. Check availability before spawning, or reuse the existing SSH helper pattern that verifies the child tunnel rather than any `EADDRINUSE`.
  - confidence: 0.91
- **[P1] Verify tunnel ownership before killing stored PIDs:** `src/cli/tunnel-cli.ts:265`
  - body: `tunnel down` sends signals to the numeric PID from the state file after only checking that some process exists. Once the SSH tunnel exits and the OS reuses that PID, this can terminate an unrelated same-user process and `status` can misreport stale state as active. Persist and verify enough process metadata before signaling.
  - confidence: 0.88
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:34`
  - body: `writeTunnelState` writes directly under `STATE_DIR`. On a fresh profile or a new `OPENCLAW_STATE_DIR`, this can throw `ENOENT` after the detached SSH process has already started, leaving a running tunnel that `down` and `status` cannot manage.
  - confidence: 0.95
- **[P2] Refresh command registration for current main:** `src/cli/program/register.subclis.ts:168-176`
  - body: This adds the command to an older `register.subclis.ts` entries array, but current main discovers subcommands through `subcli-descriptors.ts` and command-group registration. After conflict resolution, make sure `tunnel` is added to the current descriptor/import path so help, routing, and lazy registration can find it.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `docs/cli/tunnel.md:1`
  - body: This is a user-facing `feat` PR, but the diff does not add an Unreleased `CHANGELOG.md` entry. Add a single-line `### Changes` bullet for the new CLI command before merge.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff adds detached SSH process control and PID-file-based process termination, with a concrete same-user process-kill risk until ownership is verified.

Concerns:

- **[medium] PID-only state can terminate unrelated processes:** `src/cli/tunnel-cli.ts:265`
  - body: The state file stores only a PID, and `tunnel down` sends SIGTERM/SIGKILL to that PID without confirming it is the managed SSH tunnel. PID reuse or stale state can make the command kill an unrelated same-user process.
