## Summary

Keep open. Current main still lacks a managed `openclaw tunnel` command, but this PR is not merge-ready because it needs real behavior proof, a rebase onto the current CLI descriptor catalog, and fixes for concrete PID, port-readiness, and state-file bugs.

## What This Changes

This PR adds `openclaw tunnel up/down/status` for detached SSH port forwarding with PID-file state and CLI docs/navigation.

## Best Possible Solution

Land a rebased managed tunnel CLI only after it uses the current descriptor registration pattern, shares or extracts the existing SSH preflight/readiness behavior, verifies process identity before stopping, creates state storage safely, and includes redacted runtime proof.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of the PR patch shows missing state-directory creation, readiness checking against any listener, and numeric-PID stop behavior. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Verify tunnel PID ownership before signaling it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to the stored numeric PID after only a liveness check. If the original SSH child exits and the OS reuses that PID, this can terminate an unrelated process; persist and verify process identity before signaling.
  - confidence: 0.93
- **[P1] Reject occupied local ports before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any bound forwarded port as proof that the new SSH tunnel is ready. A pre-existing listener can make `isPortBound(port)` return true while the SSH child is still briefly alive, producing a false success and stale PID state.
  - confidence: 0.91
- **[P2] Create the state directory before writing metadata:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start detached SSH and then throw `ENOENT`, leaving the tunnel unmanaged.
  - confidence: 0.95
- **[P3] Surface unexpected port-listen errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves `false` for listen errors other than `EADDRINUSE`, so permission or OS errors become a generic startup timeout instead of the actual bind failure.
  - confidence: 0.82
