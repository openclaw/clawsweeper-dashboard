## Summary

Keep open: current main does not already implement a managed tunnel CLI, but this PR is not correct as-is because it adds detached SSH PID lifecycle management without safe port readiness, state-dir creation, PID identity verification, and real behavior proof.

## What This Changes

The PR adds an `openclaw tunnel up/down/status` CLI, command registration, and CLI documentation for managed SSH port forwarding.

## Best Possible Solution

Land a rebased managed tunnel CLI only after it shares or matches the existing SSH helper’s port preflight/readiness behavior, safely persists and verifies tunnel process identity, creates state storage before detaching, and includes real redacted runtime proof.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: the branch writes PID state before ensuring the directory exists, treats an occupied local port as readiness, and later signals a stored numeric PID. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

No. The feature direction may be useful, but this implementation should reuse or extract the current SSH helper’s safer preflight/readiness pattern and add process-identity verification before persistent detached lifecycle management is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify tunnel PID ownership before signaling it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently owns the stored numeric PID. If the SSH child exits and the OS later reuses that PID, this can terminate an unrelated user process; persist and verify process identity before signaling.
  - confidence: 0.93
- **[P1] Reject occupied local ports before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any `EADDRINUSE` as proof that the new SSH tunnel is ready. An existing listener on `--port` can make `isPortBound(port)` return true while SSH is briefly alive, producing a false success and stale PID state.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start detached SSH and then throw `ENOENT`, leaving the tunnel unmanaged by `status` or `down`.
  - confidence: 0.95
- **[P3] Surface unexpected port-listen errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves `false` for listen errors other than `EADDRINUSE`, so permission or OS errors become a generic startup timeout. Propagate unexpected errors so users see the actual bind failure.
  - confidence: 0.82
