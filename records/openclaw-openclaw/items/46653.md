## Summary

Keep open: this is a plausible new CLI feature, but current main does not already provide a managed `openclaw tunnel` command and the proposed patch still has blocking readiness, state, and process-safety defects plus missing real behavior proof.

## What This Changes

The PR adds `openclaw tunnel up/down/status`, a persistent tunnel PID state file, command registration, and a CLI docs page for managed SSH port-forwarding.

## Best Possible Solution

Rework this into a safe persistent SSH tunnel command that matches the existing helper’s readiness semantics, verifies process ownership before status/stop, rebases onto the current command catalog, adds focused coverage/docs, and includes redacted real CLI proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a current-main user bug, but yes for the proposed patch’s failure paths by source inspection: fresh state directories, pre-occupied local ports, and stale or reused PIDs all follow directly from the PR diff.

## Solution Assessment

Is this the best way to solve the issue? No as written. The feature direction may be useful, but the implementation should reuse or match `startSshPortForward` readiness behavior and add ownership-safe persistent state before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a free local port before marking the tunnel active:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any listener on `--port` as proof that the new SSH child is ready. If another process is already bound there, SSH can exit after `ExitOnForwardFailure=yes` while the command reports success and writes stale state.
  - confidence: 0.9
- **[P1] Verify PID ownership before signaling tunnel processes:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` signals the numeric PID from the state file without confirming it is still the SSH tunnel this command created. After PID reuse, stale state can terminate an unrelated user process.
  - confidence: 0.88
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: On a fresh profile or new `OPENCLAW_STATE_DIR`, this write can throw `ENOENT` after the detached SSH process has already started, leaving users with a running tunnel that `down` and `status` cannot manage.
  - confidence: 0.93

## Security Review

Status: needs_attention
