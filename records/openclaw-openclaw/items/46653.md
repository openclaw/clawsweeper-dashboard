## Summary

Keep this PR open: current main still lacks a managed `openclaw tunnel` command, but the branch is not merge-ready because it is stale against the current CLI catalog and has concrete process-control, readiness, and state-file defects.

## What This Changes

The PR adds a top-level `openclaw tunnel up/down/status` CLI for detached SSH port forwarding, PID-state tracking, and a new CLI docs page/nav entry.

## Best Possible Solution

Land a rebased managed tunnel CLI that uses the current descriptor/core registration path, reuses or extends the existing SSH tunnel helper, persists verifiable process identity before signaling, and includes lifecycle tests, docs, and a user-facing changelog entry.

## Reproduction Assessment

Yes. Current-main absence is reproducible by source search and command-catalog inspection; the PR defects are reproducible from fresh `OPENCLAW_STATE_DIR`, an already-occupied local port, and a stale PID file whose numeric PID has been reused.

## Solution Assessment

No. A managed tunnel CLI is a reasonable product direction, but this patch is not the best implementation until it is rebased and hardened around command registration, readiness checks, state writes, and process ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Verify PID ownership before stopping tunnels:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently has the stored numeric PID. If the original SSH process exits and the PID is reused, this can terminate an unrelated same-user process; persist and verify process identity before signaling.
  - confidence: 0.94
- **[P1] Reject occupied ports before reporting tunnel success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The readiness loop treats any listener on the requested port as proof that the new SSH child succeeded. If another process already owns the port, SSH can still be alive briefly before `ExitOnForwardFailure` exits, producing a false success and stale tunnel state.
  - confidence: 0.92
- **[P2] Register tunnel in the current command catalog:** `src/cli/program/register.subclis.ts:172-177`
  - body: Current main discovers root subcommands from `subcli-descriptors.ts` and loads them through `register.subclis-core.ts`. Adding `tunnel` only to this older wrapper path will miss the maintained lazy registration and help flow after the branch is refreshed.
  - confidence: 0.87
- **[P2] Create the state directory before writing tunnel metadata:** `src/cli/tunnel-cli.ts:38-39`
  - body: `writeTunnelState` writes under `STATE_DIR` without ensuring the directory exists. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start the detached SSH process and then throw `ENOENT`, leaving an unmanaged tunnel.
  - confidence: 0.95
- **[P3] Surface unexpected port-probe errors:** `src/cli/tunnel-cli.ts:62-63`
  - body: `isPortBound` collapses every listen error other than `EADDRINUSE` into `false`, so permission or OS errors become a generic startup timeout. Propagate unexpected errors or reuse the existing connect-based readiness helper.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `docs/cli/tunnel.md:10-12`
  - body: This adds a user-facing `feat` command and docs page, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-visible feature changes to include a changelog entry before merge.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff introduces detached SSH process management and currently stops stored PIDs without verifying they still belong to an OpenClaw-started tunnel.

Concerns:

- **[medium] Unverified PID signaling can stop unrelated processes:** `src/cli/tunnel-cli.ts:291`
  - body: The PID file stores only a numeric PID, and `tunnel down` calls `killPid` without checking command identity, process start time, or another durable ownership marker. PID reuse can make this terminate an unrelated same-user process.
  - confidence: 0.93
