## Summary

Keep open: current main does not provide a persistent `openclaw tunnel` command, but this PR is not merge-ready because the proposed process-state handling can signal the wrong process, readiness detection can report success for a pre-existing listener, fresh state directories can fail, and no observable after-fix behavior proof is attached.

## What This Changes

The branch adds a top-level `openclaw tunnel up/down/status` CLI with documentation, docs navigation, and detached SSH process state stored under the OpenClaw state directory.

## Best Possible Solution

If maintainers want this in core, rebase it onto the current CLI descriptor/loader pipeline, reuse or extend the existing SSH tunnel readiness and port-availability helpers, persist enough process identity to avoid killing reused PIDs, create state directories before writing, and add focused CLI tests plus real SSH proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR writes state without creating the directory, treats any local listener as tunnel readiness, and later kills a stored numeric PID without proving ownership. I did not run live SSH because this review is read-only and no target host or proof artifact was provided.

## Solution Assessment

No. The best implementation direction is to build on the current `startSshPortForward` readiness/port contract and add robust owned-process state instead of duplicating lower-level detached process polling around a bare PID.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Verify the saved PID before sending signals:** `src/cli/tunnel-cli.ts:293`
  - body: `tunnel down` trusts the persisted numeric PID and sends SIGTERM/SIGKILL whenever that PID is alive. If the original SSH child exits and the OS reuses the PID, this command can terminate an unrelated user process instead of treating the state file as stale. Persist and verify process identity, such as command/start time or an owned session marker, before signaling.
  - confidence: 0.91
- **[P1] Require a free port before reporting tunnel success:** `src/cli/tunnel-cli.ts:226-228`
  - body: The startup loop treats `isPortBound(port)` as proof that the new SSH forward is ready, but a pre-existing listener on the same port satisfies that check while the SSH child is still briefly alive and about to fail with `ExitOnForwardFailure`. This can write stale tunnel state and print success for a tunnel that never started. Check port availability first or tie readiness to the spawned SSH process rather than any listener.
  - confidence: 0.9
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: On a fresh profile or new `OPENCLAW_STATE_DIR`, `writeTunnelState` writes to `STATE_DIR/tunnel.pid.json` without creating the directory. That can throw after the detached SSH process is already running, leaving users with an unmanaged tunnel that `down/status` cannot find.
  - confidence: 0.95
- **[P2] Register the command through the current CLI catalog:** `src/cli/program/register.subclis.ts:169-174`
  - body: Current main discovers subcommands through `subcli-descriptors.ts` and lazy-loads them through `register.subclis-core.ts`; this PR still patches the old `register.subclis.ts` entry-list shape. After rebasing, add the `tunnel` descriptor and lazy loader to the current files so root help and command dispatch can actually reach the new CLI.
  - confidence: 0.83
