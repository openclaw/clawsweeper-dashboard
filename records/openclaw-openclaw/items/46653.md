## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify the stored PID before killing it:** `src/cli/tunnel-cli.ts:291`
  - body: The state file stores only a numeric PID. If the SSH tunnel exits and the OS reuses that PID, `tunnel down` passes the liveness check and sends SIGTERM/SIGKILL to an unrelated process. Store enough ownership data, such as command and start time, or inspect the process before killing it.
  - confidence: 0.91
- **[P2] Reject pre-existing listeners before reporting tunnel success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The readiness loop treats any `EADDRINUSE` on the local port as success. If another gateway or process already owns the port, SSH can fail from `ExitOnForwardFailure` while this code writes a stale PID and prints `Tunnel active`. Check port availability before spawning or reuse the existing SSH readiness path.
  - confidence: 0.9
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:38-39`
  - body: On a fresh profile or new `OPENCLAW_STATE_DIR`, this write throws `ENOENT` after the detached SSH process has already been started, leaving an unmanaged tunnel with no state file for `down` or `status`. Create `STATE_DIR` recursively before writing.
  - confidence: 0.94
