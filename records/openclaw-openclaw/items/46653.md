## Summary

Keep open: the managed tunnel feature is not on current main and may be useful, but this PR is not merge-ready because the diff has source-visible process lifecycle bugs, missing real behavior proof, and a stale CLI registration surface against current main.

## What This Changes

The branch adds a top-level `openclaw tunnel up/down/status` CLI, PID-file state under `~/.openclaw`, and CLI docs/nav for managed SSH port-forwarding.

## Best Possible Solution

Land a rebased managed tunnel CLI only after it uses the current descriptor catalog, shares or extracts the safer SSH helper readiness/preflight behavior, verifies process identity before stopping, creates state storage safely, adds focused coverage, and includes redacted live proof.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection shows the missing state-directory creation, port-readiness race, and PID-only stop path. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but the current patch duplicates readiness logic instead of using the safer helper pattern and introduces unsafe PID lifecycle behavior that needs rework and owner review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Verify tunnel PID ownership before signaling:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to the stored numeric PID after only a liveness check. If the SSH child exits and the OS reuses that PID, this can terminate an unrelated process; persist and verify process identity before signaling.
  - confidence: 0.93
- **[P1] Reject occupied local ports before reporting tunnel readiness:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any bound local port as proof that the new SSH tunnel is ready. A pre-existing listener can make `isPortBound(port)` return true while the SSH child is briefly alive, causing false success and stale PID state.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start detached SSH and then throw `ENOENT`, leaving the tunnel unmanaged.
  - confidence: 0.95
- **[P3] Surface unexpected port-listen errors:** `src/cli/tunnel-cli.ts:63-64`
  - body: `isPortBound` resolves `false` for listen errors other than `EADDRINUSE`, so permission or OS errors become a generic startup timeout instead of the actual bind failure.
  - confidence: 0.82
