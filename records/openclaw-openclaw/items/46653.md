## Summary

Keep open. The persistent tunnel command is not implemented on current main, but this PR has blocking process-management bugs and no observable real behavior proof, so it needs rework rather than cleanup closure.

## What This Changes

The PR adds a top-level `openclaw tunnel up/down/status` CLI and docs for persistent SSH port-forward management backed by a PID JSON file.

## Best Possible Solution

Keep the PR open for maintainer-guided rework that aligns with the existing SSH tunnel helper, adds ownership-safe state management and focused coverage, rebases command registration onto current main, and requires redacted real CLI proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a user bug, but source review gives high-confidence failure paths for the proposed patch: fresh state directories, pre-occupied local ports, and stale or reused PIDs.

## Solution Assessment

Is this the best way to solve the issue? No as written. The feature direction may be useful, but the implementation should reuse or match `startSshPortForward` readiness behavior, verify PID ownership before stop/status, create state storage safely, and rebase into the current descriptor catalog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a free port before reporting tunnel startup:** `src/cli/tunnel-cli.ts:224-226`
  - body: The loop treats any listener on `--port` as proof that the new SSH child is ready. If another process is already bound to the port, `ExitOnForwardFailure=yes` can make SSH exit right after this check while the command still reports success and writes stale state.
  - confidence: 0.9
- **[P1] Verify the stored PID before sending signals:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to the numeric PID from the state file without confirming it is still the SSH tunnel this command created. After PID reuse, stale state can terminate an unrelated user process and `status` will also misreport it as active.
  - confidence: 0.88
- **[P2] Create the state directory before writing PID state:** `src/cli/tunnel-cli.ts:39`
  - body: On a fresh profile or new `OPENCLAW_STATE_DIR`, this write can throw `ENOENT` after the detached SSH process has already started, leaving users with a running tunnel that `down` and `status` cannot manage.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete local process-safety issue because stale tunnel state can signal an unrelated PID.

Concerns:

- **[medium] PID reuse can terminate unrelated processes:** `src/cli/tunnel-cli.ts:291`
