## Summary

Keep open: the requested feature is still absent from current main, but this branch is not merge-ready because the detached tunnel state can point at the wrong process, startup can report success for an unrelated listener, fresh state dirs fail, and the PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds an `openclaw tunnel up/down/status` CLI and docs for a managed background SSH port-forward tracked by `~/.openclaw/tunnel.pid.json`.

## Best Possible Solution

Accept the feature only after product approval for a first-class detached tunnel command, then rebase it onto current CLI descriptors/core registration, reuse or extract the existing SSH readiness helpers, store verifiable process identity with the PID state, add focused tests/docs, and require redacted live terminal proof.

## Reproduction Assessment

Yes for the blocking review findings from source inspection: the PR writes state before ensuring the state dir exists, treats any local listener as tunnel readiness, and kills a stored numeric PID without verifying process identity. I did not run live SSH because this review is read-only.

## Solution Assessment

No. The feature direction may be useful, but the proposed implementation should not merge until it uses the current CLI registration path and fixes process readiness, state creation, and PID ownership validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Verify the stored PID before killing it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` trusts only the numeric PID from `tunnel.pid.json`. If the SSH child exits and the OS reuses that PID, this path can SIGTERM/SIGKILL an unrelated user process instead of treating the state file as stale; store enough process identity or inspect the command line before killing.
  - confidence: 0.9
- **[P2] Reject occupied local ports before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any `EADDRINUSE` as proof that the new SSH forward is ready. If another process is already listening on the requested port, the loop can mark the tunnel active and persist a stale SSH PID while `ssh` exits from `ExitOnForwardFailure`; check availability before spawn or verify the child owns the listener.
  - confidence: 0.88
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes directly under `STATE_DIR`, which may not exist in a fresh profile or custom `OPENCLAW_STATE_DIR`. Because the SSH process has already been detached, an `ENOENT` here leaves a running tunnel with no manageable PID file.
  - confidence: 0.92
