## Summary

Keep open: current main does not implement a managed persistent tunnel command, and the PR is not mergeable as-is because it needs a rebase to the current command catalog plus process-safety hardening for detached SSH tunnel management.

## What This Changes

The PR adds `openclaw tunnel up/down/status`, detached SSH port-forward state in `~/.openclaw/tunnel.pid.json`, and a new CLI docs page/nav entry.

## Best Possible Solution

If maintainers want this in core, land a rebased tunnel CLI that uses the current descriptor registration path, reuses or extends the existing SSH helper, verifies process ownership before signaling, and includes lifecycle tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Current-main absence is source-reproducible by the missing `src/cli/tunnel-cli.ts`, missing `tunnel` descriptor, and docs that still show manual SSH tunneling; the PR defects are also source-reproducible from the added PID, readiness, and registration code.

## Solution Assessment

No. A managed tunnel CLI may be a reasonable direction, but this patch is not the best implementation until it is rebased onto the current command catalog and hardened around SSH readiness, state writes, process identity, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Verify PID ownership before stopping tunnels:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently has the stored numeric PID. If the SSH tunnel exits and the OS reuses that PID, this can terminate an unrelated same-user process, so persist and verify process identity before signaling.
  - confidence: 0.94
- **[P1] Reject occupied ports before reporting tunnel success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any listener on the requested port as proof that the new SSH child succeeded. If another process already owns the port, SSH can still be alive briefly before `ExitOnForwardFailure` exits, producing a false success and stale tunnel state.
  - confidence: 0.92
- **[P2] Register tunnel in the current command catalog:** `src/cli/program/register.subclis.ts:172-177`
  - body: Current main discovers root subcommands from `subcli-descriptors.ts` and loads them through `register.subclis-core.ts`. Adding `tunnel` only to the older wrapper path will miss the maintained lazy registration and help flow after the branch is refreshed.
  - confidence: 0.87
- **[P2] Create the state directory before writing tunnel metadata:** `src/cli/tunnel-cli.ts:39-40`
  - body: `writeTunnelState` writes under `STATE_DIR` without ensuring the directory exists. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start the detached SSH process and then throw `ENOENT`, leaving an unmanaged tunnel.
  - confidence: 0.95
- **[P3] Surface unexpected port-probe errors:** `src/cli/tunnel-cli.ts:62-63`
  - body: `isPortBound` collapses every listen error other than `EADDRINUSE` into `false`, so permission or OS errors become a generic startup timeout. Propagate unexpected errors or reuse the existing connect-based readiness helper.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `docs/cli/tunnel.md:10-12`
  - body: This adds a user-facing feature command and docs page, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-visible feature changes to include a changelog entry before merge.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff introduces detached SSH process management and currently stops stored PIDs without verifying they still belong to an OpenClaw-started tunnel.

Concerns:

- **[medium] Unverified PID signaling can stop unrelated processes:** `src/cli/tunnel-cli.ts:291`
