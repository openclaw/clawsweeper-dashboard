## Summary

This PR should stay open because current main still lacks a persistent `openclaw tunnel` CLI, but the proposed implementation has blocking lifecycle bugs around detached SSH/PID management and still lacks contributor-provided real behavior proof.

## What This Changes

The PR adds a new `openclaw tunnel up/down/status` CLI with PID-file state for detached SSH port forwarding plus CLI docs and docs navigation.

## Best Possible Solution

Keep the feature direction under review, but rebase it onto the current CLI registry and implement persistent SSH tunnel management with explicit port preflight, readiness based on the spawned tunnel, state-directory creation, and process identity verification before status or termination.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: the patch writes state without creating `STATE_DIR`, treats any occupied port as tunnel readiness, and later signals a stored PID without ownership verification. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

No. A managed tunnel command may be useful, but this implementation should reuse or match the safer existing SSH helper behavior and must verify process identity before adding persistent detached lifecycle management.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify tunnel PID ownership before signaling it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently owns the stored numeric PID. If the SSH child exits and the OS later reuses that PID, this can terminate an unrelated user process; persist and verify process identity before signaling.
  - confidence: 0.93
- **[P1] Reject occupied local ports before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any `EADDRINUSE` as proof that the new SSH tunnel is ready. An existing listener on `--port` can make `isPortBound(port)` return true while SSH is briefly alive, producing a false success and stale PID state.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start detached SSH and then throw `ENOENT`, leaving the tunnel unmanaged by `status` or `down`.
  - confidence: 0.95
- **[P3] Surface unexpected port-listen errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves `false` for listen errors other than `EADDRINUSE`, so permission or OS errors become a generic startup timeout. Propagate unexpected errors so users see the actual bind failure.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces detached SSH/PID lifecycle management with a concrete local process-safety concern.
