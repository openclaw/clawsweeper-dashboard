## Summary

Keep open. The feature is not already on current main, but this patch is not safe to merge as-is: source inspection confirms process-safety and state-management bugs in the added tunnel CLI, the PR is currently conflicting, and real behavior proof is missing.

## What This Changes

This PR adds a top-level `openclaw tunnel up/down/status` CLI, PID-file state, docs navigation, and a CLI reference page for managed SSH port forwarding.

## Best Possible Solution

Revise the feature around the existing SSH tunnel helper or a shared persistent variant, add process-ownership and port-readiness safeguards, port registration to the current command catalog, and include focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: the added command treats any local listener as success, writes state without ensuring the directory exists, and signals a stored PID without ownership checks. I did not run live SSH because this was a read-only review.

## Solution Assessment

No. The feature direction may be useful, but this branch duplicates readiness/lifecycle logic that current main already handles more carefully for temporary SSH probes and needs safety fixes before product review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify the stored PID before signaling it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently has the stored PID. If the original SSH child exits and the OS later reuses that PID, this command can terminate an unrelated user process, and `status` can also report the reused PID as alive. Store and verify enough process identity before signaling, or avoid signaling by raw PID alone.
  - confidence: 0.93
- **[P1] Require the port to be free before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats `EADDRINUSE` as proof that the new SSH tunnel is ready. When another process is already listening on the requested port, the SSH child can still be briefly alive while `isPortBound(port)` returns true for the unrelated listener, causing a false success and stale PID file. Preflight the port or verify the listener belongs to the tunnel before writing state.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or a new `OPENCLAW_STATE_DIR`, `tunnel up` can detach SSH successfully and then throw `ENOENT`, leaving a running tunnel that `tunnel down/status` cannot manage.
  - confidence: 0.95
- **[P3] Surface unexpected port-listen errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves `false` for every listen error other than `EADDRINUSE`, so permission or OS errors are reported later as a generic timeout. Propagate non-`EADDRINUSE` errors so users see the real bind failure.
  - confidence: 0.82
