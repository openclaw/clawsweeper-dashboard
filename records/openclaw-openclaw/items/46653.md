## Summary

Keep open. Current main still has no top-level `openclaw tunnel` command, so the feature is not obsolete, but this PR is not safe to merge as-is because the proposed detached SSH lifecycle has concrete port-readiness, PID-safety, and fresh-state-dir bugs, and the external PR lacks real behavior proof.

## What This Changes

The PR adds a top-level `openclaw tunnel up/down/status` CLI, docs, and docs navigation for managing detached SSH gateway port forwards with persisted PID state.

## Best Possible Solution

Revise the PR around the current CLI descriptor system, reuse or extract the existing safe SSH port-forward readiness logic, add robust state-dir and process-identity handling, cover the edge cases with focused tests, update docs, and attach redacted real CLI proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows `writeTunnelState` writes before creating `STATE_DIR`, readiness treats any local listener as success, and `down/status` trust a numeric PID without process identity. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

No. The useful feature direction is plausible, but this patch should not manage a persistent SSH tunnel with duplicated weaker readiness checks and PID-only state; the safer path is to reuse or factor the existing SSH tunnel helper and add process identity validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify the tunnel process before signaling it:** `src/cli/tunnel-cli.ts:283`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently owns the stored PID. If the original SSH process exits and the OS reuses that PID, `status` will report it as alive and `down` can terminate an unrelated user process; store and verify process identity before signaling, or treat unverifiable state as stale.
  - confidence: 0.91
- **[P1] Require a free local port before marking the tunnel active:** `src/cli/tunnel-cli.ts:219-222`
  - body: The startup loop treats `EADDRINUSE` from `isPortBound` as proof that this SSH child is ready. If a gateway or another tunnel already owns the port, the command can write the new child's PID and report success while SSH exits due to `ExitOnForwardFailure`; check port availability before spawn and wait on the child/listener race instead.
  - confidence: 0.93
- **[P2] Create the state directory before writing the PID file:** `src/cli/tunnel-cli.ts:39`
  - body: On a fresh profile or new `OPENCLAW_STATE_DIR`, `writeFileSync` throws `ENOENT` after the detached SSH process has started, leaving a running tunnel that `openclaw tunnel down/status` cannot manage. Create `STATE_DIR` recursively before writing the state file.
  - confidence: 0.95
- **[P3] Surface unexpected port-probe errors:** `src/cli/tunnel-cli.ts:67-69`
  - body: `isPortBound` collapses every non-`EADDRINUSE` listen error into `false`, so permission or OS errors look like an unbound port and later become a generic timeout. Propagate unexpected listen errors so the CLI reports the real failure.
  - confidence: 0.78

## Security Review
