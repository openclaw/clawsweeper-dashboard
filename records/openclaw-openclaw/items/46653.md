## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Verify tunnel ownership before signaling stored PIDs:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` only checks whether the numeric PID is alive before sending SIGTERM/SIGKILL. If the original SSH process exits and the OS reuses that PID, this command can terminate an unrelated local process; store and verify tunnel identity before signaling.
  - confidence: 0.93
- **[P1] Reject occupied ports before reporting tunnel readiness:** `src/cli/tunnel-cli.ts:222-224`
  - body: `isPortBound()` returns true for any listener on the local port, not necessarily the SSH child. If another process already owns the port while SSH is still exiting due to `ExitOnForwardFailure`, the command can write stale state and report a tunnel active incorrectly.
  - confidence: 0.92
- **[P2] Create the state directory before writing tunnel metadata:** `src/cli/tunnel-cli.ts:38-39`
  - body: `writeTunnelState()` writes under `STATE_DIR` without ensuring the directory exists. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start a detached SSH process and then throw `ENOENT`, leaving users without a PID file to manage it.
  - confidence: 0.91
- **[P2] Resolve the SSH executable through the shared policy:** `src/cli/tunnel-cli.ts:200`
  - body: This new public command hard-codes `/usr/bin/ssh`, extending the open Windows/OpenSSH bug into another user-facing path. Use a shared trusted SSH resolver or explicitly scope the command before adding more SSH process execution surface.
  - confidence: 0.84
