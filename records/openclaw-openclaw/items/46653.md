## Summary

Keep open: current main does not implement the managed tunnel command, but the PR is not ready because it is stale against current CLI registration and introduces concrete detached-process/PID and port-readiness bugs.

## What This Changes

The PR adds an `openclaw tunnel up/down/status` CLI, detached SSH PID-state management, a CLI docs page/nav entry, and root command registration.

## Best Possible Solution

If maintainers want this in core, land a rebased implementation through the descriptor catalog that reuses or extends the existing SSH helper, verifies process identity before signaling, and includes lifecycle tests, docs, and changelog coverage.

Yes. Source inspection reproduces both the current-main absence of the feature and the PR defects in PID signaling, readiness detection, state writing, and command registration.

## Solution Assessment

No. The user problem is real, but this patch is not the best implementation until it is rebased onto the current command catalog and hardened around process ownership, startup readiness, state creation, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Verify PID ownership before stopping tunnels:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently has the stored numeric PID. If the SSH tunnel exits and the OS reuses that PID, this can terminate an unrelated same-user process, so persist and verify process identity before signaling.
  - confidence: 0.94
- **[P1] Reject occupied ports before reporting tunnel success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any listener on the requested port as proof that the new SSH child succeeded. If another process already owns the port, SSH can still be alive briefly before `ExitOnForwardFailure` exits, producing a false success and stale tunnel state.
  - confidence: 0.92
- **[P2] Register tunnel in the current command catalog:** `src/cli/program/register.subclis.ts:172-177`
  - body: Current main discovers root subcommands from `SUB_CLI_DESCRIPTORS` and loads them through `register.subclis-core.ts`. Adding `tunnel` only to the older wrapper path will miss the maintained lazy registration and help flow after the branch is refreshed.
  - confidence: 0.87
- **[P2] Create the state directory before writing tunnel metadata:** `src/cli/tunnel-cli.ts:38-39`
  - body: `writeTunnelState` writes under `STATE_DIR` without ensuring the directory exists. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start the detached SSH process and then throw `ENOENT`, leaving an unmanaged tunnel.
  - confidence: 0.95
- **[P3] Surface unexpected port-probe errors:** `src/cli/tunnel-cli.ts:62-63`
  - body: `isPortBound` collapses every listen error other than `EADDRINUSE` into `false`, so permission or OS errors become a generic startup timeout. Propagate unexpected errors or reuse the existing connect-based readiness helper.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `docs/cli/tunnel.md:10-12`
  - body: This adds a user-facing CLI command and docs page, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-visible feature changes to include a changelog entry before merge.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff introduces detached SSH process management and currently signals stored PIDs without verifying they still belong to an OpenClaw-started tunnel.

Concerns:

- **[medium] Unverified PID signaling can stop unrelated processes:** `src/cli/tunnel-cli.ts:291`
  - body: The PID file stores only a numeric PID, and `tunnel down` calls `killPid` without checking command identity, process start time, or another durable ownership marker. PID reuse can make this terminate an unrelated same-user process.
  - confidence: 0.93
