Overall confidence: 0.92

Full review comments:

- **[P1] Register tunnel through the current descriptor catalog:** `src/cli/program/register.subclis.ts:172-178`
  - body: Current `main` builds root command discovery from `subcli-descriptors.ts` and `register.subclis-core.ts`, but this PR adds `tunnel` only to the older registration array. After rebasing, the command will not be advertised or lazily registered unless it is added to the descriptor catalog and current registration spec.
  - confidence: 0.91
- **[P1] Verify tunnel ownership before signaling stored PIDs:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently has the stored PID. If the SSH process exits and the OS reuses that PID, this can kill an unrelated user process; store and verify process identity before signaling.
  - confidence: 0.94
- **[P1] Reject occupied ports before declaring the tunnel ready:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any listener on `--port` as proof that the new SSH child is ready. If another process already owns the port, SSH can exit due to `ExitOnForwardFailure` while the command writes stale state and reports success.
  - confidence: 0.93
- **[P2] Create the state directory before writing tunnel metadata:** `src/cli/tunnel-cli.ts:38-39`
  - body: `writeTunnelState` writes directly under `STATE_DIR`. On a fresh profile or a new `OPENCLAW_STATE_DIR`, this can throw after the detached SSH process has already started, leaving an unmanaged tunnel with no PID file.
  - confidence: 0.9
- **[P2] Propagate unexpected port-probe errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves `false` for every listen error except `EADDRINUSE`, so permission or OS errors become an 8-second readiness timeout instead of a clear diagnostic. Surface non-`EADDRINUSE` errors or reuse the existing connect/readiness helper.
  - confidence: 0.82
