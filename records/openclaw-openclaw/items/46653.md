## Summary

Keep open. The feature is plausible, but this PR is not mergeable as-is: the branch is conflicting with current CLI registration, and the new detached process/state handling has concrete correctness and security-sensitive bugs.

## What This Changes

This PR adds an `openclaw tunnel up/down/status` CLI, a tunnel docs page, and docs navigation for managed SSH port forwarding.

## Best Possible Solution

Rebase onto the current CLI descriptor model, factor tunnel startup/readiness through the existing SSH tunnel helper path, add ownership-safe persistent state handling, and include focused tests plus inspectable real CLI proof.

## Reproduction Assessment

Yes. Source inspection of the PR branch gives a high-confidence path for the review findings: fresh-state writes hit `writeTunnelState`, port collision hits the `isPortBound` readiness loop, and PID reuse hits `tunnel down` after reading the stored PID.

## Solution Assessment

No. A managed tunnel command should reuse or extend `src/infra/ssh-tunnel.ts` and current CLI descriptors instead of adding a parallel SSH process manager with separate readiness and PID semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify PID ownership before stopping tunnel processes:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` trusts the stored numeric PID and then sends SIGTERM/SIGKILL to it. If the original SSH process exits and the OS reuses that PID, this command can terminate an unrelated user process; record and verify enough process identity before killing it.
  - confidence: 0.9
- **[P1] Require a free local port before marking the tunnel active:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any listener on `--port` as success. If another process already owns the port, SSH can still be alive briefly before `ExitOnForwardFailure` exits, so the command reports success and writes stale state for a tunnel that never started.
  - confidence: 0.88
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: On a fresh profile or new `OPENCLAW_STATE_DIR`, this write throws `ENOENT` after the detached SSH process has started, leaving a running tunnel with no PID file for `down` or `status` to manage.
  - confidence: 0.93

## Security Review
