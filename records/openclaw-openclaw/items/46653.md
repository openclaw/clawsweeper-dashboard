## Summary

Keep open: current main does not provide the managed tunnel command, but this branch should not merge as-is because source review finds process/state bugs, missing real behavior proof, and a product/security decision for a detached host-process CLI.

## What This Changes

This PR adds a top-level `openclaw tunnel up/down/status` CLI command and docs for a detached SSH port-forward tracked by `~/.openclaw/tunnel.pid.json`.

## Best Possible Solution

Accept this feature only after product approval, then implement it through the current subcommand descriptor path, reuse or extract the existing SSH readiness helpers, store verifiable process identity with the PID state, add focused tests/docs, and require redacted live terminal proof.

## Reproduction Assessment

Yes for the blocking review findings from source inspection: the branch writes state before ensuring the state directory exists, treats any local listener as tunnel readiness, and kills a stored numeric PID without process identity validation. I did not run live SSH because this review is read-only.

## Solution Assessment

No. The feature direction may be useful, but the proposed implementation should be rebased onto the current CLI registration path and fix readiness, state creation, and process-identity handling before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Verify the stored PID before killing it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever live process currently has the stored numeric PID. If the SSH child exits and the OS reuses that PID, this can terminate an unrelated user process; store enough process identity or inspect the command before killing.
  - confidence: 0.9
- **[P2] Reject occupied local ports before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any `EADDRINUSE` as proof that the new SSH forward is ready. If another process is already listening on the requested port, the loop can persist a stale SSH PID and report success while SSH exits from `ExitOnForwardFailure`.
  - confidence: 0.88
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without ensuring the directory exists. In a fresh profile or custom state directory, `tunnel up` can start a detached SSH process and then throw `ENOENT`, leaving an unmanaged tunnel.
  - confidence: 0.92
- **[P3] Surface unexpected port probe errors:** `src/cli/tunnel-cli.ts:62-63`
  - body: `isPortBound` collapses every listen error except `EADDRINUSE` into `false`, so permission or OS-level bind failures are reported later as generic tunnel startup failures instead of the actual local port error.
  - confidence: 0.76
