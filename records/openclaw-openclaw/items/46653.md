## Summary

Keep open: current main still lacks a persistent `openclaw tunnel` command, but this PR is not mergeable as-is because the added detached SSH/PID lifecycle has source-proven correctness and local process-safety blockers, targets stale CLI registration, and lacks inspectable real behavior proof.

## What This Changes

The branch adds a top-level `openclaw tunnel up/down/status` command with PID-file state and CLI docs for managed SSH port forwarding.

## Best Possible Solution

Land a rebased implementation on the current descriptor-based CLI that reuses or hardens the existing SSH tunnel helper, verifies process identity before status or termination, handles occupied ports explicitly, and includes redacted real command output or a recording.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: the diff shows PID-only termination, missing state-directory creation, and a readiness check that treats any occupied port as success. I did not run a live SSH tunnel in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but this patch should first be ported to the current CLI catalog and should reuse or harden the existing SSH helper semantics instead of adding unsafe detached PID management.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify PID ownership before signaling it:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently owns the stored PID. If the SSH child exits and the OS later reuses that PID, this command can terminate an unrelated user process; persist and verify process identity before signaling.
  - confidence: 0.93
- **[P1] Require the port to be free before reporting success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The startup loop treats any `EADDRINUSE` as proof that the new SSH tunnel is ready. An existing listener on `--port` can make `isPortBound(port)` return true while the SSH process is briefly alive, causing a false success and stale PID file.
  - confidence: 0.91
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without creating it. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start detached SSH and then throw `ENOENT`, leaving the tunnel unmanaged by `status` or `down`.
  - confidence: 0.95
- **[P3] Surface unexpected port-listen errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` resolves `false` for listen errors other than `EADDRINUSE`, so permission or OS errors are later reported as a generic timeout. Propagate unexpected errors so users see the real bind failure.
  - confidence: 0.82

## Security Review
