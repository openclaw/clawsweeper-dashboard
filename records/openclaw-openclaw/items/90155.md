## Summary

Keep this PR open: it is a focused maintainer-authored hardening change, but the previous rule-list coercion blocker still survives at the current head. The branch still performs `Array.isArray` on untrusted rule-list values before the non-throwing reader takes over, so a revoked/unreadable list can still crash the doctor/config diagnostic path the PR is meant to harden.

## What This Changes

The PR adds guarded channel legacy config rule readers and tests so malformed bundled, bootstrap, and registry doctor metadata is skipped instead of crashing config diagnostics.

## Best Possible Solution

Make rule-list coercion itself non-throwing for bundled, bootstrap, and registry-provided rule lists, add a revoked/unreadable-list regression, and keep the existing source precedence unchanged before merge.

## Reproduction Assessment

Yes. Source inspection shows current main reads plugin doctor rule metadata without guards, and a focused Node check proves `Array.isArray` can throw on a revoked proxy list before the PR's sanitizer catches anything.

## Solution Assessment

No, not yet. The owner boundary is right, but the implementation needs to move array detection and provider reads inside a non-throwing rule-list reader to fully solve the malformed-list crash case.
