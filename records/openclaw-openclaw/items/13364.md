## Summary

Keep open. Current main still separates managed HOOK.md/internal hooks from typed plugin before_tool_call and after_tool_call hooks, and the discussion keeps this as the canonical API/security design tracker.

## What This Changes

Expose before_tool_call and after_tool_call tool lifecycle events to managed internal HOOK.md hooks so handlers can observe, rewrite, or block tool calls without a full plugin.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with docs and focused coverage, or explicitly document and enforce a plugin-only boundary for tool interception.

## Reproduction Assessment

Yes. Source inspection on current main shows managed HOOK.md events register into the internal hook map, while tool execution consults the plugin hook runner for before_tool_call and after_tool_call.

## Solution Assessment

Unclear as a concrete implementation. The direction is plausible, but the best fix needs maintainer API/security design rather than a narrow bridge that copies plugin behavior without settled trust and failure semantics.
