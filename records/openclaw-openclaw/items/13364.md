## Summary

Keep open: current main still has plugin-runner support for before/after tool-call hooks, but managed/internal HOOK.md handlers remain void-returning internal hooks with no tool-call event type, no block/rewrite return contract, and no tool-execution bridge. The discussion and duplicate closures make this the canonical API/security design tracker.

## What This Changes

Expose before_tool_call and after_tool_call semantics through managed/internal HOOK.md hooks so they can observe, rewrite, or block tool calls.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs and tests, or explicitly document and enforce that tool interception remains plugin-only.

## Reproduction Assessment

Yes, by source inspection. Managed HOOK.md handlers register into the internal hook registry, while before and after tool execution only consult the plugin hook runner.

## Solution Assessment

Unclear as a final implementation direction. A bridge is plausible, but maintainers need to settle authority, failure semantics, approval behavior, rewrite rules, event names, and duplicate-dispatch behavior first.
