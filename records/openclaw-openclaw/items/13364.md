## Summary

Keep open. Current main still has the same source-level gap: managed/internal HOOK.md hooks are void-returning internal events, tool execution still dispatches through plugin before/after tool-call hooks, and the discussion now includes security/product acceptance criteria that need maintainer design rather than cleanup closure.

## What This Changes

Expose before_tool_call and after_tool_call semantics to managed/internal HOOK.md hooks so operators can observe, mutate, or block tool calls before execution.

## Best Possible Solution

Keep this as the canonical API/security design issue: add typed managed before/after tool-call events with blocking, rewrite, validation, redaction and receipt boundaries, docs, and focused tests, or explicitly document that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level gap, though not as a strict bug repro: current main has no managed/internal tool-call event type or return contract, and tool execution consults plugin hooks rather than HOOK.md internal hooks.

## Solution Assessment

Unclear as a simple bridge. The maintainable solution needs a typed contract, blocking and rewrite semantics, trust/redaction policy, event validation, docs, and coverage, or an explicit product decision to keep this plugin-only.
