## Summary

Keep open. Current main still lacks a managed/internal HOOK.md tool-call event and return contract, while before_tool_call and after_tool_call remain plugin-hook surfaces; related duplicates and the closed after-only PR leave this as the canonical API/security design tracker.

## What This Changes

Expose plugin-level before_tool_call and after_tool_call semantics to managed/internal HOOK.md handlers so they can block, rewrite, and audit tool calls.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add typed managed before/after tool-call hooks with blocking semantics, validation, tests, and docs, or explicitly reject unsupported tool lifecycle events in managed hooks and document the plugin-only boundary.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: HOOK.md events register into the internal hook map, but tool execution consults the plugin hook runner for before_tool_call and after_tool_call.

## Solution Assessment

Unclear as a concrete implementation. The maintainable direction is a designed typed bridge across before and after tool paths, or an explicit plugin-only policy, not the closed after-only partial PR.
