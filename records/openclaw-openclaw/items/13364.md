## Summary

Keep open: current main still supports before/after tool-call interception through the plugin hook runner, but managed HOOK.md hooks remain a void-returning internal hook surface with no tool-call event contract, no blocking or parameter-rewrite result path, and no bridge from tool execution. The live discussion and duplicate closures make this the canonical API/security design tracker.

## What This Changes

The issue requests exposing before_tool_call and after_tool_call through managed/internal HOOK.md hooks so local handlers can block, rewrite, observe, and audit tool operations.

## Best Possible Solution

Keep this as the canonical design tracker and either add a typed managed tool-call hook contract with security/audit semantics, docs, and tests, or explicitly document and enforce that tool interception remains plugin-only.

## Reproduction Assessment

Yes, by source inspection. Managed HOOK.md handlers register into the internal hook registry, while before and after tool execution only consult the plugin hook runner; I did not run a live gateway because this review is read-only.

## Solution Assessment

Unclear as a final implementation direction. A bridge is plausible, but maintainers need to settle authority, failure semantics, approval behavior, rewrite rules, audit receipts, event names, and duplicate-dispatch behavior first.
