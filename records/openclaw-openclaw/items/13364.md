## Summary

Keep open: current main and v2026.5.12 still route managed HOOK.md handlers through the void internal hook dispatcher while before/after tool-call interception remains plugin-runner-only, and related duplicates plus prior review comments keep this as the canonical product/security design tracker.

## What This Changes

Expose before_tool_call and after_tool_call to managed/internal HOOK.md hooks so handlers can observe, block, or rewrite tool calls.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs and tests, or explicitly document and enforce that tool interception stays plugin-only.

## Reproduction Assessment

Yes, for the current gap by source inspection. Managed HOOK.md handlers register into the internal hook registry, while before and after tool execution only consult the plugin hook runner.

## Solution Assessment

Unclear as a final implementation direction. A bridge is plausible, but maintainers need to settle authority, failure semantics, approval behavior, rewrite rules, and duplicate-dispatch behavior before implementation.
