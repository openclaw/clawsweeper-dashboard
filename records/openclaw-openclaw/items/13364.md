## Summary

Keep open: current main still lacks managed/internal tool-call events, return semantics, and tool-execution bridging, while the discussion keeps this as the canonical API/security design request.

## What This Changes

The issue requests exposing before/after tool-call interception through managed HOOK.md/internal hooks so handlers can observe, rewrite, or block tool operations.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs and focused coverage, or explicitly document and enforce the plugin-only boundary for tool interception.

## Reproduction Assessment

Yes. Source inspection on current main shows managed HOOK.md handlers register into the internal hook map, while before and after tool execution consult only the plugin hook runner.

## Solution Assessment

Unclear as a concrete implementation. A bridge is plausible, but the maintainable fix needs an explicit API/security decision for trust, failure, approval, rewrite, and double-dispatch semantics.
