## Summary

Keep open. Current main and latest release still keep managed HOOK.md hooks on the void internal dispatcher, while enforceable before/after tool-call policy remains in the typed plugin hook runner; this remains the canonical security-sensitive hook/API design item.

## What This Changes

The issue requests exposing enforceable before_tool_call and after_tool_call behavior through managed internal HOOK.md hooks.

## Best Possible Solution

Keep this canonical issue open for maintainer API/security design: either adapt validated managed HOOK.md handlers into the existing plugin hook policy runner with explicit trust rules, or make the plugin-only boundary explicit by rejecting or surfacing unsupported tool events.

## Reproduction Assessment

Yes, at source level. Current-main types, loader registration, docs, and tool execution paths show managed HOOK.md handlers cannot receive or enforce before_tool_call block/parameter decisions.

## Solution Assessment

Unclear until maintainers choose the API boundary. Plugin hooks already implement blocking and parameter rewrite, so the maintainable path is an explicit adapter into that contract or an explicit plugin-only decision, not a second ad hoc policy runner.
