## Summary

Keep open. Current main and v2026.6.6 still keep managed HOOK.md hooks on the void internal hook dispatcher, while enforceable before/after tool-call policy remains in the plugin hook runner; this remains the canonical security-sensitive product/API decision.

## What This Changes

The issue requests managed internal HOOK.md support for before/after tool-call hooks with blocking, parameter rewrite, and after-call observation semantics.

## Best Possible Solution

Keep one canonical tool-policy runner: either adapt validated managed HOOK.md hooks into the existing plugin hook semantics with explicit trust rules, or keep interception plugin-only and reject/surface unsupported managed tool events clearly.

## Reproduction Assessment

Yes, at source level. Current-main types, loader registration, and tool execution paths show managed HOOK.md handlers cannot receive or enforce before_tool_call block/parameter decisions.

## Solution Assessment

Unclear until maintainers choose the API boundary. Plugin hooks already implement blocking and parameter rewrite, so the maintainable path is an explicit adapter into that contract or an explicit plugin-only decision, not a second ad hoc policy runner.
