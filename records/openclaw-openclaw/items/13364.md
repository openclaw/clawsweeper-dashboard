## Summary

Keep open. Current main and v2026.6.8 still keep enforceable tool-call interception in typed plugin hooks, while managed/internal HOOK.md hooks remain a void-returning command/lifecycle event system with no tool-call event type, no blocking or parameter-rewrite return contract, and no bridge from tool execution.

## What This Changes

The issue requests exposing before_tool_call and after_tool_call through managed/internal HOOK.md hooks so local hook handlers can block, rewrite, and observe tool operations.

## Best Possible Solution

Keep this as the canonical maintainer design item; either add a typed managed/internal tool-call hook contract with blocking, rewrite, after-call observation, docs, and regression coverage across relevant tool paths, or explicitly keep tool policy plugin-only and reject or warn on unsupported HOOK.md event names.

## Reproduction Assessment

Yes, at source level. Current-main types, loader registration, docs, and tool execution paths show managed HOOK.md handlers cannot receive or enforce before_tool_call block/parameter decisions.

## Solution Assessment

Unclear until maintainers decide the API boundary. A blind bridge is not the best fix; the maintainable solution needs typed event/result semantics, trust/failure policy, cross-runtime coverage, and tests before implementation.
