## Summary

Current main still lacks managed tool-call interception: HOOK.md handlers are void-returning internal events, while enforceable tool policy remains in the plugin runner. The adjacent maintainer PR adds agent lifecycle boundaries but does not change managed-hook loading or add blocking and argument-rewrite semantics, so this issue remains the canonical product and security-boundary decision.

## What This Changes

Expose managed HOOK.md handlers to before- and after-tool-call events, including pre-execution blocking and argument rewriting.

## Best Possible Solution

Keep one canonical tool-policy runner; either register validated managed hooks through an adapter with explicit trust, priority, timeout, approval, and failure semantics, or retain plugin-only interception and reject unsupported HOOK.md events clearly at load and status time.

## Reproduction Assessment

Yes, at source level. Current-main types, loader registration, and dispatcher behavior provide a high-confidence path showing that managed HOOK.md handlers cannot receive enforceable before_tool_call decisions.

## Solution Assessment

No definitive implementation is approved yet. A direct bridge to the current void-returning internal dispatcher would duplicate or bypass the established plugin policy contract; the narrow maintainable solution must preserve one runner and explicitly define trust and composition semantics.
