## Summary

Keep open. Current main still keeps before/after tool-call interception in the plugin hook runner, while managed HOOK.md internal hooks remain void-returning and are not dispatched from tool execution. The issue discussion also has fresh maintainer context keeping this as the canonical tracker after related duplicate/partial attempts were closed.

## What This Changes

Expose before_tool_call and after_tool_call tool lifecycle events through managed/internal HOOK.md hooks, including pre-execution blocking, parameter rewrite, and post-execution audit data.

## Best Possible Solution

Keep this as the canonical maintainer/API design issue. A complete implementation should define typed managed before/after tool-call events, a blocking and parameter-rewrite return contract, validation for unsupported HOOK.md event names, one dispatch path that does not double-fire existing plugin hooks, focused tests, and docs; if maintainers decide this belongs plugin-only, close later with that explicit product decision and point users to plugin hooks or ClawHub packaging.

## Reproduction Assessment

Yes. A high-confidence reproduction is to register a managed HOOK.md handler for before_tool_call or after_tool_call and run any tool; current source shows that the loader registers the handler only in the internal registry while tool execution consults only the plugin hook runner.

## Solution Assessment

Unclear as a concrete implementation, but keeping the issue open is the right current path. The narrow after-only bridge attempted in related #29590 would not solve pre-execution blocking or return semantics; the safer fix is a designed typed bridge across the relevant tool execution paths.
