## Summary

Keep open. Current main still does not expose before_tool_call or after_tool_call through managed/internal HOOK.md hooks, and the recent discussion adds security and audit requirements that need maintainer API design rather than cleanup closure.

## What This Changes

The issue requests managed/internal HOOK.md tool-call hooks that can observe, mutate, or block tool execution before and after calls.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with validation, redaction, receipts, docs, and coverage, or explicitly decide that pre-execution interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level gap, though not as a strict bug repro: current main has no managed/internal tool-call event type or decision return contract, and tool execution consults plugin hooks rather than HOOK.md internal hooks.

## Solution Assessment

Unclear as a simple bridge. The maintainable solution needs a typed contract, blocking and rewrite semantics, trust/redaction policy, event validation, docs, and tests, or an explicit product decision to keep this plugin-only.
