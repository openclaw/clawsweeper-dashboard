## Summary

Keep open: current main still keeps managed `HOOK.md` hooks in the void-returning internal hook registry, while before/after tool-call interception remains a plugin-hook runner feature. The discussion, duplicate closures, and current labels make this the canonical API/security design tracker rather than a cleanup-close candidate.

## What This Changes

The issue requests exposing plugin-style before and after tool-call hooks through managed internal `HOOK.md` hooks so they can inspect, mutate, block, and audit tool execution.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs/tests/receipts, or explicitly document and enforce that tool interception remains plugin-only.

## Reproduction Assessment

Yes, by source inspection. Managed `HOOK.md` handlers register into the internal hook registry, while before and after tool execution consult only the plugin hook runner for tool lifecycle hooks.

## Solution Assessment

Unclear as a final implementation direction. A bridge is plausible, but maintainers need to settle authority, failure semantics, approval behavior, rewrite rules, event names, receipt/audit output, and duplicate-dispatch behavior first.
