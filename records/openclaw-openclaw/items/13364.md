## Summary

Keep open. Current main and the latest release still keep before/after tool-call interception on the typed plugin hook path, while managed HOOK.md internal hooks have no typed tool_call event, no blocking or parameter-rewrite return contract, and no execution bridge; the discussion also has fresh security/product acceptance criteria rather than cleanup grounds.

## What This Changes

Request to expose plugin before_tool_call and after_tool_call interception through managed/internal HOOK.md hooks so operator scripts can block, rewrite, and audit tool calls.

## Best Possible Solution

Keep this as the canonical API/security design issue: add typed managed before/after tool-call events with blocking, rewrite, validation, redaction/receipt boundaries, docs, and focused tests, or explicitly document and enforce that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level gap, though not as a strict bug repro: current main and v2026.5.19 have no managed/internal tool-call event type or return contract, and tool execution consults plugin hooks rather than HOOK.md internal hooks.

## Solution Assessment

Unclear as a simple bridge. The maintainable solution needs a typed contract, blocking and rewrite semantics, trust/redaction policy, event validation, docs, and coverage, or an explicit product decision to keep this plugin-only.
