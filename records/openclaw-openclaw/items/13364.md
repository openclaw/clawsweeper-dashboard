## Summary

Keep open. Current main still keeps managed HOOK.md hooks on the void-returning internal hook registry, while before_tool_call and after_tool_call remain typed plugin-hook surfaces; the discussion also has fresh maintainer signal preserving this as the canonical API/security design tracker.

## What This Changes

The issue requests exposing before/after tool-call events through managed internal HOOK.md hooks with blocking and parameter rewrite support.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add typed managed before/after tool-call hooks with blocking semantics, validation, docs, and tests, or explicitly document and enforce a plugin-only boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows HOOK.md events register into the internal hook map, but tool execution consults the plugin hook runner for before_tool_call and after_tool_call.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is plausible, but the best fix needs a designed typed bridge or an explicit plugin-only policy rather than a narrow after-only dispatch patch.
