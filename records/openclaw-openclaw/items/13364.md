## Summary

Keep open. Current main still does not expose before_tool_call or after_tool_call through managed/internal HOOK.md hooks, and the discussion has fresh maintainer signal keeping this as the canonical API/security design tracker rather than a cleanup-close candidate.

## What This Changes

Request to add typed managed/internal before and after tool-call hook events with blocking, parameter rewrite, and audit semantics for HOOK.md handlers.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add typed managed before/after tool-call hooks with blocking semantics, validation, tests, and docs, or explicitly document that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: HOOK.md handlers register into the internal hook registry, while tool execution consults the plugin hook runner for before_tool_call and after_tool_call.

## Solution Assessment

Unclear as a concrete implementation. The maintainable path needs a designed typed bridge across both before and after tool paths, or an explicit plugin-only policy; the closed after-only PR did not cover the central blocking use case.
