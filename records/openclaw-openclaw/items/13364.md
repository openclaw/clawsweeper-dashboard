## Summary

Keep open. Current main and the latest release still keep managed HOOK.md hooks on the void internal hook dispatcher, while enforceable before/after tool-call policy remains in the plugin hook runner; this issue remains the canonical product and security-boundary decision.

## What This Changes

The issue requests exposing enforceable before/after tool-call events to managed internal HOOK.md hooks.

## Best Possible Solution

Keep one canonical tool-policy runner; either adapt validated managed HOOK.md hooks into that runner with explicit trust and composition semantics, or explicitly keep interception plugin-only and reject unsupported managed event names at load/status time.

## Reproduction Assessment

Yes, at source level. Current-main types, loader registration, and dispatcher behavior show that managed HOOK.md handlers cannot receive or enforce before_tool_call block/parameter decisions.

## Solution Assessment

Unclear until maintainers choose the API boundary. Plugin hooks already implement blocking and parameter rewrite, so the maintainable path is an explicit adapter into that contract or an explicit plugin-only decision, not a second ad hoc policy runner.
