## Summary

Current main still lacks a managed tool-call event type or decision-return contract; recent hook refactoring did not implement the requested interception. The open lifecycle PR is adjacent and maintainer-owned, but it does not safely supersede this canonical product/security design issue.

## What This Changes

Expose managed HOOK.md handlers to before- and after-tool-call lifecycle events, including pre-call blocking and argument rewriting.

## Best Possible Solution

Approve and document one canonical tool-policy pipeline that composes managed and plugin hooks with typed before/after payloads, deterministic rewrites, explicit block/error semantics, event-name validation, and focused coverage for every supported tool runtime.

## Reproduction Assessment

Yes, at source level with high confidence: a HOOK.md handler can be registered under an arbitrary event string, but current internal types and dispatch cannot receive a tool-event decision or block and rewrite execution.

## Solution Assessment

No, the suggested simple bridge is not by itself the best fix; maintainers first need a canonical composition contract covering trust, ordering, failure behavior, return aggregation, validation, and all tool-execution paths.
