## Summary

Keep open. Current main still keeps managed/internal hooks as void-returning lifecycle/message handlers, while before/after tool interception remains plugin-hook-only; the discussion also has explicit maintainer context keeping this as the canonical tracker.

## What This Changes

Request to expose before/after tool-call interception through managed internal `HOOK.md` hooks so handlers can observe, modify, or block tool executions without a full plugin.

## Best Possible Solution

Keep this as the canonical maintainer/API design issue and either add a typed managed before/after tool-call bridge with blocking semantics, tests, and docs, or explicitly document that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a managed `HOOK.md` can register `before_tool_call` as a string, but tool execution only consults the plugin hook runner, so the handler will not receive the tool call.

## Solution Assessment

Unclear as a concrete implementation, but keeping this open is the right current outcome. The safer solution is a designed typed bridge across both before and after paths; the closed after-only #29590 would not satisfy pre-execution blocking or return semantics.
