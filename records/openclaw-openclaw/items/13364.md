Keep open. Current main still lacks managed/internal HOOK.md before_tool_call or after_tool_call support, the live discussion keeps this as the canonical tracker, and the remaining work is a security-sensitive API design decision rather than conservative cleanup.

## What This Changes

The issue requests exposing plugin before_tool_call/after_tool_call behavior through managed internal HOOK.md hooks.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with validation, redaction/trust policy, docs, and focused coverage, or explicitly decide that pre-execution interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level gap, though not as a strict bug repro: current main has no managed/internal tool-call event type or decision return contract, and tool execution still consults plugin hook runners for before/after tool calls.

## Solution Assessment

Unclear as the simple bridge proposed in the issue. The maintainable path needs typed event and return semantics, event validation, security redaction/trust policy, no-double-fire behavior, docs, and tests, or a maintainer decision to keep this plugin-only.
