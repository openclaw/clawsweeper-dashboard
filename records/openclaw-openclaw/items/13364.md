## Summary

Keep open. Current main still keeps managed HOOK.md/internal hooks separate from the plugin tool-call hook runner, and the live discussion keeps this as the canonical API/security design tracker rather than a cleanup candidate.

## What This Changes

The issue requests managed HOOK.md/internal hooks for before and after tool calls, including pre-execution blocking or parameter rewriting.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with docs and focused coverage, or explicitly document and enforce a plugin-only boundary for tool interception.

## Reproduction Assessment

Yes. Source inspection on current main shows managed HOOK.md events register into the internal hook map, while before and after tool execution consult only the plugin hook runner.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is plausible, but the best fix needs maintainer API/security design rather than a narrow bridge that copies plugin behavior without settled trust, failure, and single-dispatch semantics.
