## Summary

Keep open. Current main still leaves managed HOOK.md/internal hooks as a void-returning command/session/agent/gateway/message system, while before/after tool-call interception remains in the plugin hook runner; the live related search still points to this issue as the canonical tracker rather than an implemented or superseded item.

## What This Changes

Expose existing before_tool_call and after_tool_call plugin hook behavior through managed/internal HOOK.md hooks so handlers can observe, rewrite, or block tool calls.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with event validation, redaction policy, docs, and focused coverage, or explicitly document that pre-execution interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level capability gap, though this is not a strict bug reproduction. Current main has no managed/internal tool-call event type or decision return contract, while tool execution still consults plugin hook runners for before and after tool calls.

## Solution Assessment

Unclear as the simple bridge proposed in the issue. The maintainable path needs typed event and return semantics, event validation, security redaction and trust policy, no-double-fire behavior, docs, and tests, or a clear decision to keep this plugin-only.
