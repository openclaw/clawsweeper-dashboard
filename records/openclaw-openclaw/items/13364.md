## Summary

Keep open. Current main still exposes blocking and parameter-rewrite tool interception only through plugin hooks, while managed/internal HOOK.md hooks remain void-returning event handlers with no tool-call event contract or execution bridge; the discussion also contains recent maintainer context keeping this as the canonical tracker.

## What This Changes

Expose before/after tool-call interception to managed/internal HOOK.md hooks so simpler handlers can observe, rewrite, or block tool operations without packaging a full plugin.

## Best Possible Solution

Keep this as the canonical maintainer/API design issue and either add typed managed before/after tool-call events with blocking semantics, tests, and docs, or explicitly document that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a managed HOOK.md event can be registered as a string, but current tool execution consults only the plugin hook runner, so the managed handler will not receive before/after tool-call events.

## Solution Assessment

Unclear as a concrete implementation, but keeping this open is the right current outcome. The safer solution is a designed typed bridge across both before and after paths; the closed after-only #29590 would not satisfy pre-execution blocking or return semantics.
