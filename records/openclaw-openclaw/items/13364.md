## Summary

Keep open: current main and the latest release still keep tool-call blocking/rewrite semantics in typed plugin hooks, while managed/internal HOOK.md hooks remain void-returning command/lifecycle automation with no tool-call bridge. The issue is still the canonical tracker for a security-sensitive product/API decision; related duplicates are closed and the related open PR is conflicting and does not appear to provide the requested before-tool blocking contract.

## What This Changes

Expose existing plugin before_tool_call and after_tool_call behavior through managed/internal HOOK.md hooks so simpler hook handlers can observe, rewrite, or block tool calls.

## Best Possible Solution

Keep this as the canonical product/API tracker until maintainers either add typed managed tool-call events with clear blocking/rewrite semantics, tests, and docs, or explicitly decide that tool-call policy stays plugin-only and add validation/diagnostics for unsupported HOOK.md event names.

## Reproduction Assessment

Yes, from source inspection: managed/internal hooks can register arbitrary event keys, but the typed event family excludes tool calls, handlers return void, and tool execution only consults the plugin hook runner. I did not run a live gateway repro because the current source path is direct and the review is read-only.

## Solution Assessment

Unclear. Bridging to the plugin runner is a plausible implementation direction, but the best solution depends on whether maintainers want managed HOOK.md hooks to become a policy gate surface or want to keep before_tool_call/after_tool_call plugin-only with clearer rejection of unsupported events.
