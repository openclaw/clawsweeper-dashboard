## Summary

Keep open: current main and v2026.5.12 still keep managed HOOK.md handlers on the void internal hook dispatcher while before/after tool-call interception remains plugin-runner-only; the discussion has multiple concrete security/policy use cases and prior duplicate reports pointing back to this canonical tracker.

## What This Changes

Expose managed HOOK.md/internal hooks for before and after tool-call events so they can observe, block, or rewrite tool operations without requiring a full plugin.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs and tests, or explicitly document and enforce that tool interception stays plugin-only.

## Reproduction Assessment

Yes, by source inspection. Managed HOOK.md handlers register into the internal hook registry, while before and after tool execution only consult the plugin hook runner.

## Solution Assessment

Unclear as a final implementation direction. A bridge is plausible, but maintainers need to settle authority, failure semantics, approval behavior, rewrite rules, event names, and duplicate-dispatch behavior first.
