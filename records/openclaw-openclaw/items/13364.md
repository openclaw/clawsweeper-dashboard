## Summary

Keep open. Current main and the latest release still keep enforceable before/after tool-call policy in typed plugin hooks, while managed/internal HOOK.md hooks remain a void-returning command/lifecycle dispatcher with no tool-call event type, no blocking or parameter-rewrite result contract, and no bridge from tool execution.

## What This Changes

The issue requests exposing before/after tool-call interception through managed/internal HOOK.md hooks so simple hook handlers can block, rewrite, or audit tool calls without a full plugin.

## Best Possible Solution

Keep this as the canonical design tracker: either add typed managed/internal before and after tool-call events with blocking, parameter rewrite, after-call observation, validation, docs, and focused tests across relevant tool paths, or explicitly keep tool-call policy plugin-only and reject/surface unsupported HOOK.md event names.

## Reproduction Assessment

Yes, at source level. Current main registers HOOK.md handlers only in the internal hook registry, while before/after tool execution consults the plugin hook runner and the internal dispatcher cannot return block or parameter decisions.

## Solution Assessment

Unclear as an implementation shape, but keeping this open is the right triage outcome. A bridge is plausible, yet the best fix needs maintainer decisions about trust model, return semantics, event names, validation, and whether policy gates should remain plugin-only.
