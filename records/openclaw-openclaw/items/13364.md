## Summary

Keep open. Current main and the latest release still keep enforceable before/after tool-call behavior in typed plugin hooks, while managed HOOK.md internal hooks have no tool-call event type, no blocking or parameter-rewrite return contract, and no bridge from tool execution.

## What This Changes

Request to expose before_tool_call and after_tool_call through managed/internal HOOK.md hooks with blocking, parameter rewrite, and after-call observation support.

## Best Possible Solution

Keep this as the canonical design tracker: either add typed managed/internal before and after tool-call events with block, parameter rewrite, after-call observation, unsupported-event validation, docs, and focused tests, or explicitly keep tool-call policy plugin-only and surface unsupported HOOK.md event names.

## Reproduction Assessment

Yes, at source level. A HOOK.md before_tool_call handler would register into the internal hook registry, while current before/after tool execution consults the plugin hook runner and the internal dispatcher cannot return block or parameter decisions.

## Solution Assessment

Unclear as an implementation shape, but keeping this issue open is the right triage outcome. The best fix needs maintainer decisions about event names, trust model, return semantics, unsupported-event validation, and double-firing avoidance.
