## Summary

Keep open. Current main still keeps before/after tool-call interception in the plugin hook runner, while managed HOOK.md internal hooks remain void-returning and unbridged for tool calls; the discussion also has fresh canonical, security-sensitive API/product-design signal rather than cleanup grounds.

## What This Changes

Expose before_tool_call and after_tool_call to managed/internal HOOK.md hooks so they can observe, rewrite, or block tool execution.

## Best Possible Solution

Keep this as the canonical API/security design issue: either add typed managed before/after tool-call events with block/params semantics, validation, redaction or receipt boundaries, docs, and focused tests, or explicitly document that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level gap, though not as a strict bug repro: current main has no managed/internal tool-call event type or return contract, and tool execution consults plugin hooks rather than HOOK.md internal hooks.

## Solution Assessment

Unclear as a simple bridge. The maintainable solution needs a typed contract, blocking and rewrite semantics, trust/redaction policy, event validation, docs, and coverage, or an explicit product decision to keep this plugin-only.
