## Summary

Keep open. Current main and v2026.6.5 still keep managed HOOK.md/internal hooks as a void-returning command/session/agent/gateway/message surface, while before/after tool-call interception remains in the typed plugin hook runner; related items still point back to this issue as the canonical design tracker.

## What This Changes

The issue requests exposing before_tool_call and after_tool_call through managed/internal HOOK.md hooks so operator hooks can observe, rewrite, or block tool operations.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with event validation, redaction policy, no-double-fire semantics, docs, and tests, or explicitly document that pre-execution tool interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level capability gap, though this is a feature request rather than a strict bug reproduction. Current main has no managed/internal tool-call event type or decision return contract, while tool execution still consults plugin hook runners for before and after tool calls.

## Solution Assessment

Unclear as the simple bridge proposed in the issue. The maintainable solution needs typed event and return semantics, event validation, security redaction and trust policy, no-double-fire behavior, docs, and tests, or a clear product decision to keep this plugin-only.
