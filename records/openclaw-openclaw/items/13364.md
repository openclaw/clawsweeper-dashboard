## Summary

Keep open. Current main still keeps managed/internal HOOK.md hooks on a void-returning internal event dispatcher, while tool-call interception remains plugin-hook-only; the issue also has recent maintainer/product/security signal and is the canonical tracker for this request.

## What This Changes

Expose plugin-style before_tool_call/after_tool_call interception, blocking, parameter rewrite, and audit events through managed internal HOOK.md handlers.

## Best Possible Solution

Keep this as the canonical API/security design issue: either add typed managed before/after tool-call events with block/params semantics, validation, receipt/redaction rules, docs, and tests, or explicitly choose plugin-only and document that boundary.

## Reproduction Assessment

Yes for the source-level gap, though not as a strict bug repro: current main has no internal tool_call event, internal handlers return void, and tool execution consults only plugin hooks. I did not run a live gateway scenario because source inspection establishes the missing dispatch and return contract.

## Solution Assessment

Unclear as a one-step implementation. A small bridge is not enough; the maintainable solution needs a typed contract, safety policy, validation, docs, and coverage, or an explicit product decision to keep this plugin-only.
