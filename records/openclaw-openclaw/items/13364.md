## Summary

Keep open. Current main still has plugin-level before/after tool hooks, but managed HOOK.md/internal hooks remain a void-returning command/session/agent/gateway/message event system; docs currently route runtime policy gates to plugin hooks, so this needs maintainer API/security judgment rather than cleanup closure.

## What This Changes

Expose before_tool_call and after_tool_call through managed HOOK.md/internal hooks so operator hooks can block, rewrite, or audit tool calls without a full plugin.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with validation, redaction policy, docs, and focused coverage, or explicitly document that pre-execution interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level gap, though this is not a strict bug repro. Current main has no managed/internal tool-call event type or decision return contract, while tool execution still consults plugin hook runners for before and after tool calls.

## Solution Assessment

Unclear as the simple bridge proposed in the issue. The maintainable path needs typed event and return semantics, event validation, security redaction and trust policy, no-double-fire behavior, docs, and tests, or a clear decision to keep this plugin-only.
