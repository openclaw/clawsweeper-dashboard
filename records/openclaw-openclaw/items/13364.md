## Summary

Keep open. Current main still keeps managed HOOK.md hooks in the void-returning internal hook registry, while before/after tool-call interception remains available only through plugin hooks; the issue also has recent maintainer context keeping it as the canonical tracker.

## What This Changes

The issue requests exposing plugin before_tool_call and after_tool_call interception through managed/internal HOOK.md hooks so handlers can observe, rewrite, or block tool calls.

## Best Possible Solution

Keep this as the canonical maintainer/API design issue and either add typed managed before/after tool-call events with blocking semantics, tests, and docs, or explicitly document that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: managed HOOK.md events register in the internal hook map, but current tool execution consults only the plugin hook runner for before_tool_call and after_tool_call.

## Solution Assessment

Unclear as a concrete implementation, but keeping this open is the right current outcome. The safer solution is a designed typed bridge across both before and after paths; the closed after-only bridge would not satisfy pre-execution blocking or return semantics.
