## Summary

Keep open. Current main still has plugin-level before_tool_call/after_tool_call support but no managed HOOK.md/internal hook tool-call event or decision return contract, and the discussion keeps this as the canonical security-sensitive API design tracker.

## What This Changes

The issue requests exposing before_tool_call and after_tool_call through managed/internal HOOK.md hooks so handlers can observe, rewrite, or block tool execution without packaging a full plugin.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with validation, redaction policy, docs, and focused coverage, or explicitly document that pre-execution interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level gap, though not as a strict bug repro: current main has no managed/internal tool-call event type or decision return contract, while tool execution still consults plugin hook runners for before and after tool calls.

## Solution Assessment

Unclear as the simple bridge proposed in the issue. The maintainable path needs typed event and return semantics, event validation, security redaction and trust policy, no-double-fire behavior, docs, and tests, or a clear decision to keep this plugin-only.
