## Summary

Keep open. Current main still lacks a managed/internal HOOK.md tool-call event family or return contract, and the remaining work is a security-sensitive hook API/product decision rather than an already implemented or safely closable cleanup item.

## What This Changes

The issue requests exposing before_tool_call and after_tool_call through HOOK.md-managed internal hooks so handlers can observe, rewrite, or block tool calls.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with event validation, redaction/trust policy, docs, and focused coverage, or explicitly decide that pre-execution interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level capability gap, though this is not a strict bug reproduction. Current main has no managed/internal tool-call event type or decision return contract, while tool execution still consults plugin hook runners for before and after tool calls.

## Solution Assessment

Unclear as the simple bridge proposed in the issue. The maintainable path needs typed event and return semantics, unsupported-event validation, trust/redaction policy, no-double-fire behavior, docs, and tests, or a clear product decision to keep this plugin-only.
