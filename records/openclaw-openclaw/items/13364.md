## Summary

Keep open. Current main and the latest release still keep enforceable before/after tool-call policy in typed plugin hooks, while managed/internal HOOK.md hooks remain a void-returning command/lifecycle dispatcher with no tool-call event type, no blocking or parameter-rewrite result contract, and no bridge from tool execution.

## What This Changes

The issue requests exposing before_tool_call and after_tool_call tool lifecycle interception through managed/internal HOOK.md hooks instead of requiring a full plugin.

## Best Possible Solution

Keep this as the canonical maintainer design item; either add a typed managed/internal tool-call hook contract across all relevant tool paths with focused tests and docs, or explicitly document that enforceable tool policy remains plugin-only.

## Reproduction Assessment

Yes, at source level. Current-main types, docs, loader registration, and tool execution paths show managed HOOK.md handlers cannot receive or enforce before_tool_call block/parameter decisions.

## Solution Assessment

Unclear until maintainers decide the API boundary. A blind bridge is not the best fix; the maintainable solution needs typed event/result semantics, trust/failure policy, cross-runtime coverage, and tests before code work proceeds.
