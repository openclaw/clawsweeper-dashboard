## Summary

Keep open. Current main still does not expose before_tool_call or after_tool_call through managed/internal HOOK.md hooks, and the remaining work is a canonical security-sensitive API/product decision rather than conservative cleanup.

## What This Changes

The issue requests managed/internal HOOK.md before and after tool-call events with blocking, parameter-rewrite, and audit visibility comparable to the existing plugin hook surface.

## Best Possible Solution

Keep this as the canonical API/security design issue: either add typed managed before/after tool-call events with block/params semantics, validation, receipts/redaction, docs, and tests, or explicitly document that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level gap, though not as a strict bug repro: current main has no internal tool_call event, internal handlers return void, and tool execution consults plugin hooks rather than managed HOOK.md hooks.

## Solution Assessment

Unclear as a one-step bridge. The maintainable solution needs a typed contract, blocking and rewrite semantics, safety policy, validation, docs, and coverage, or an explicit product decision to keep this plugin-only.
