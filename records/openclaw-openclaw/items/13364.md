## Summary

Keep open. Current main still exposes tool-call interception through plugin hooks only; managed HOOK.md/internal hooks remain void-returning, have no tool-call event contract, and the discussion keeps this as the canonical API/security design tracker.

## What This Changes

Expose before_tool_call and after_tool_call behavior through managed HOOK.md/internal hooks so simpler handlers can block, rewrite, or audit tool calls.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add a typed managed tool-call hook contract with docs and focused coverage, or explicitly document and enforce a plugin-only boundary for tool interception.

## Reproduction Assessment

Yes. Source inspection on current main shows managed HOOK.md events register into the internal hook map, while before and after tool execution consult only the plugin hook runner.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is plausible, but the best fix needs maintainer API/security design rather than a narrow bridge that copies plugin behavior without settled trust and failure semantics.
