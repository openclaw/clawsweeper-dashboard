## Summary

Keep open. Current main and v2026.6.8 still keep enforceable before/after tool-call policy in typed plugin hooks, while managed/internal HOOK.md hooks remain a void-returning command/lifecycle system with no tool-call event type, no block or parameter-rewrite return contract, and no bridge from tool execution.

## What This Changes

Requested behavior: expose before/after tool-call interception through managed/internal HOOK.md hooks so handlers can observe, rewrite, or block tool operations without a full plugin.

## Best Possible Solution

Keep this as the canonical design tracker: either add typed managed/internal before and after tool-call events with block, parameter rewrite, after-call observation, validation, docs, and focused tests, or explicitly keep tool-call policy plugin-only and reject or surface unsupported HOOK.md event names.

## Reproduction Assessment

Yes, at source level. Current main registers HOOK.md handlers only in the internal hook registry, while before/after tool execution consults the plugin hook runner and the internal dispatcher cannot return block or parameter decisions.

## Solution Assessment

Unclear as an implementation shape, but keeping this issue open is the right triage outcome. A bridge is plausible, yet the best fix needs maintainer decisions about trust model, return semantics, event names, unsupported event validation, and double-firing avoidance.
