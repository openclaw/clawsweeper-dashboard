## Summary

Keep open: current main still does not expose before/after tool-call events through managed/internal HOOK.md hooks, and the linked duplicate reports plus the closed partial PR make this the canonical API/security design tracker.

## What This Changes

Request to expose plugin before_tool_call/after_tool_call semantics to managed/internal HOOK.md hooks so handlers can observe, rewrite, or block tool calls.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs and focused coverage, or explicitly document and enforce the plugin-only boundary for tool interception.

## Reproduction Assessment

Yes. Source inspection on current main shows managed HOOK.md handlers register into the internal hook map, while before and after tool execution consult only the plugin hook runner.

## Solution Assessment

Unclear as a concrete implementation. A bridge is plausible, but the maintainable fix needs an explicit API/security decision for trust, timeout, failure, approval, rewrite, and double-dispatch semantics.
