## Summary

Keep open. Current main and v2026.6.10 still keep enforceable before/after tool-call policy in typed plugin hooks, while managed/internal HOOK.md hooks remain a void-returning command/lifecycle system with no tool-call event contract, no block/parameter-rewrite return channel, and no tool-execution bridge.

## What This Changes

The issue requests exposing before_tool_call and after_tool_call interception through managed internal HOOK.md hooks so local hooks can block, rewrite, or audit tool calls without a full plugin.

## Best Possible Solution

Keep this as the canonical design tracker: either add a typed managed/internal HOOK.md tool-call contract with blocking, parameter rewrite, after-call observation, validation, docs, and focused tests, or explicitly keep tool-call policy plugin-only and reject or surface unsupported HOOK.md tool event names.

## Reproduction Assessment

Yes, at source level. Current main registers managed HOOK.md handlers only in the internal hook registry, while before/after tool execution consults the plugin hook runner and the internal dispatcher cannot return block or parameter decisions.

## Solution Assessment

Unclear as an implementation shape, but keeping this issue open is the right triage outcome. A bridge is plausible, yet the best fix requires maintainer decisions about trust model, event names, return semantics, unsupported event validation, and double-firing avoidance.
