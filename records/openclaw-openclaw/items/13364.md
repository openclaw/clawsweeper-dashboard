## Summary

Current main still keeps managed `HOOK.md` handlers in a void-returning internal registry that cannot intercept tool calls, while the active related maintainer PR covers adjacent lifecycle observations rather than managed pre-call blocking or argument rewriting. This remains the canonical product and security design tracker.

## What This Changes

Expose managed `HOOK.md` handlers to before/after tool-call events, including pre-call blocking and parameter rewriting.

## Best Possible Solution

Adapt validated managed tool-hook declarations into the existing typed plugin hook runner, preserving one canonical ordering and aggregation model while documenting explicit timeout, exception, blocking, argument-rewrite, and after-call data-handling semantics.

## Reproduction Assessment

Yes, at source level with high confidence: managed handlers register into a void-returning internal registry, while tool execution uses the separate typed plugin hook runner. The contract gap is directly observable without a live setup.

## Solution Assessment

The proposed bridge is directionally correct, but the best fix is not another parallel dispatcher. Managed discovery should adapt into the canonical typed tool-hook runner with explicit ordering, security, and failure semantics.
