## Summary

Keep open: current main still does not expose before/after tool-call events through managed/internal HOOK.md hooks, and the live discussion keeps this as the canonical API/security design tracker rather than a cleanup-close candidate.

## What This Changes

The issue requests exposing before/after tool-call interception through managed HOOK.md internal hooks so simple handlers can observe, rewrite, or block tool execution without a full plugin.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs and focused coverage, or explicitly document and enforce the plugin-only boundary for tool interception.

## Reproduction Assessment

Yes. Source inspection on current main shows managed HOOK.md handlers register into the internal hook map, while before and after tool execution consult the plugin hook runner rather than an internal tool lifecycle dispatcher.

## Solution Assessment

Unclear as a concrete bridge. The direction is plausible, but the maintainable fix needs explicit API/security decisions for event names, handler authority, timeout/failure policy, approval interaction, parameter rewrites, and double-dispatch behavior.
