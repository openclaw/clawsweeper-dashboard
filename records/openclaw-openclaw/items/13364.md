## Summary

Keep open. Current main still keeps managed/internal HOOK.md hooks separate from the plugin before_tool_call/after_tool_call runner, and the related implementation work is open rather than merged.

## What This Changes

The issue requests typed before and after tool-call events for managed/internal HOOK.md hooks so handlers can observe, rewrite, or block tool operations without packaging a full plugin.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add typed managed tool-call hooks with validation, blocking/rewrite semantics, redaction policy, docs, and coverage, or explicitly decide that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes for the source-level gap, though not as a strict bug repro: current main has no managed/internal tool-call event type or decision return contract, and tool execution consults plugin hooks rather than HOOK.md internal hooks.

## Solution Assessment

Unclear as a simple bridge. The maintainable solution needs a typed contract, blocking and rewrite semantics, trust/redaction policy, event validation, docs, and tests, or an explicit product decision to keep this plugin-only.
