## Summary

Keep open: current main still keeps managed HOOK.md hooks on the internal hook dispatcher while tool-call interception remains plugin-runner-only, and recent review context treats this as the canonical API/security design tracker rather than cleanup.

## What This Changes

Expose before and after tool-call events through managed/internal HOOK.md hooks so handlers can observe, rewrite, or block tool operations.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs and focused coverage, or explicitly document and enforce that tool interception remains plugin-only.

## Reproduction Assessment

Yes, by source inspection. Managed HOOK.md handlers register into the internal hook registry, while before and after tool execution only consult the plugin hook runner.

## Solution Assessment

Unclear as a final implementation direction. A bridge is plausible, but the maintainable solution needs an explicit product/security decision for handler authority, timeout behavior, failure mode, approval interaction, parameter rewrites, and avoiding duplicate plugin/internal dispatch.
