## Summary

Keep open. Current main still keeps managed HOOK.md hooks on the void-returning internal hook registry, while before_tool_call and after_tool_call remain typed plugin-hook surfaces; the discussion also keeps this as the canonical API/security design tracker.

## What This Changes

Expose before_tool_call and after_tool_call tool lifecycle interception through managed HOOK.md/internal hooks instead of requiring a full plugin.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add typed managed before/after tool-call hooks with blocking semantics, validation, docs, and tests, or explicitly document and enforce a plugin-only boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows HOOK.md events register into the internal hook map, while before_tool_call and after_tool_call execution consults the plugin hook runner instead.

## Solution Assessment

Unclear as a concrete implementation. The requested direction is plausible, but the best fix needs a designed typed bridge or an explicit plugin-only policy rather than a narrow after-only dispatch patch.
