## Summary

Keep open. Current main still limits tool-call interception to plugin hooks, while managed/internal HOOK.md hooks remain void-returning and are not dispatched from the before/after tool execution paths; the discussion also has fresh maintainer context keeping this as the canonical tracker.

## What This Changes

Expose before_tool_call and after_tool_call to managed/internal HOOK.md hooks so local hook handlers can observe, rewrite, or block tool calls.

## Best Possible Solution

Keep this as the canonical maintainer/API design issue and either add a typed managed before/after tool-call bridge with blocking semantics, tests, and docs, or explicitly document that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: register a managed HOOK.md handler for before_tool_call or after_tool_call and invoke any tool; the loader registers it only in the internal registry, while tool execution checks only the plugin hook runner.

## Solution Assessment

Unclear as a concrete implementation, but keeping this open is the right current path. The safer solution is a designed typed bridge across before and after tool paths; the closed after-only bridge in #29590 would not satisfy pre-execution blocking or return semantics.
