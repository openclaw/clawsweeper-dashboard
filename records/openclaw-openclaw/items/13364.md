## Summary

Keep open. Current main still lacks a managed/internal HOOK.md tool-call event and return contract, while before_tool_call and after_tool_call execution remains wired through plugin hooks; the related duplicate and closed after-only PR leave this as the canonical tracker.

## What This Changes

Expose before_tool_call and after_tool_call lifecycle interception through managed internal HOOK.md hooks instead of requiring a full plugin.

## Best Possible Solution

Keep this as the canonical API/security design issue and either add typed managed before/after tool-call hooks with blocking semantics, tests, and docs, or explicitly reject unsupported tool lifecycle events in managed hooks and document the plugin-only boundary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: HOOK.md events register in the internal hook map, but current tool execution consults plugin hooks for before_tool_call and after_tool_call.

## Solution Assessment

Unclear as a concrete implementation. The narrow maintainable direction is a designed typed bridge across both before and after tool paths, not the closed after-only partial PR.
