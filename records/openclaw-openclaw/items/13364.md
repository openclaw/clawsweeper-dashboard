## Summary

Keep open. Current main and latest v2026.6.10 still route enforceable before/after tool policy through typed plugin hooks, while managed HOOK.md internal hooks remain void-returning command/lifecycle events with no tool-call event type or decision return channel.

## What This Changes

Expose before_tool_call and after_tool_call to managed/internal HOOK.md hooks so handlers can block, rewrite, and audit tool calls.

## Best Possible Solution

Keep this as the canonical design tracker: either add typed managed/internal before and after tool-call events with block, parameter rewrite, after-call observation, unsupported-event validation, docs, and focused tests, or explicitly keep tool-call policy plugin-only and surface unsupported HOOK.md event names.

## Reproduction Assessment

Yes, at source level. Current main registers HOOK.md handlers only in the internal hook registry, while before/after tool execution consults the plugin hook runner and the internal dispatcher cannot return block or parameter decisions.

## Solution Assessment

Unclear as an implementation shape, but keeping this issue open is the right triage outcome. The best fix needs maintainer decisions about event names, trust model, return semantics, unsupported-event validation, and double-firing avoidance.
