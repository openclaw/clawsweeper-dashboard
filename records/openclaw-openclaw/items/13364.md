## Summary

Keep open: current main still keeps managed HOOK.md hooks in the internal hook dispatcher while before/after tool-call interception remains plugin-runner-only, and the discussion now treats this as the canonical API/security design tracker.

## What This Changes

Expose before_tool_call and after_tool_call to managed/internal HOOK.md hooks so handlers can observe, rewrite, or block tool calls.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs and tests, or explicitly document and enforce that tool interception stays plugin-only.

## Reproduction Assessment

Yes, by source inspection. Managed HOOK.md handlers register into the internal hook registry, while before and after tool execution only consult the plugin hook runner.

## Solution Assessment

Unclear as a final implementation direction. A bridge is plausible, but maintainers need to define authority, failure semantics, approval interaction, rewrite rules, and duplicate-dispatch behavior before implementation.
