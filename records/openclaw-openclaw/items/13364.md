## Summary

Keep this issue open as the canonical tracker. Current main still limits managed `HOOK.md` handlers to a void internal event system, while tool-call blocking and parameter rewriting remain in the typed plugin hook pipeline; the active lifecycle PR adds adjacent observability and correlation work but does not implement this managed-hook contract.

## What This Changes

Expose managed `HOOK.md` handlers to before- and after-tool-call lifecycle events, including pre-execution blocking and argument rewriting.

## Best Possible Solution

Add a typed managed-hook adapter at the canonical tool-policy boundary, reject or clearly surface unsupported metadata events, and define deterministic ordering, decision merging, blocking, timeout, error, agent-feedback, and exactly-once semantics across every tool adapter.

## Reproduction Assessment

Yes. Current main has no internal `tool_call` event family, internal handlers cannot return decisions, and managed hook loading only registers them into the void internal dispatcher, so the missing capability is directly reproducible from source.

## Solution Assessment

No, not as a simple call to the existing internal dispatcher. The narrow maintainable solution is to adapt managed handlers into the canonical tool-policy pipeline so established blocking, parameter merging, approval, error, and exactly-once invariants remain authoritative.
