## Summary

Keep open: current main still keeps managed HOOK.md handlers on the void internal hook dispatcher, while before/after tool-call interception remains plugin-runner-only. The live discussion, duplicate closures, protected ClawSweeper labels, and security/product labels make this the canonical API design tracker, not a cleanup-close candidate.

## What This Changes

Expose existing plugin before/after tool-call hooks to managed/internal HOOK.md hooks so handlers can block, rewrite, or audit tool operations.

## Best Possible Solution

Keep this as the canonical API/security design tracker and either add a typed managed tool-call hook contract with docs and tests, or explicitly document that tool interception remains plugin-only.

## Reproduction Assessment

Yes, by source inspection. Managed HOOK.md handlers register into the internal hook registry, while before and after tool execution only consult the plugin hook runner.

## Solution Assessment

Unclear as a final implementation direction. A bridge is plausible, but maintainers need to settle authority, failure semantics, approval behavior, rewrite rules, event names, and duplicate-dispatch behavior before implementation.
