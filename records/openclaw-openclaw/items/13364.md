## Summary

Keep open. Current main still keeps managed/internal hooks as void-returning lifecycle/message handlers, while before/after tool interception remains plugin-hook-only; the discussion also has fresh maintainer context keeping this as the canonical tracker.

## What This Changes

Request to expose `before_tool_call` and `after_tool_call` through managed/internal `HOOK.md` hooks, including pre-execution blocking or parameter rewrite and post-execution audit events.

## Best Possible Solution

Keep this as the canonical maintainer/API design issue and either add a typed managed before/after tool-call bridge with blocking semantics, tests, and docs, or explicitly document that tool-call interception remains plugin-only.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a managed `HOOK.md` can register `before_tool_call` or `after_tool_call` as strings, but current tool execution only consults the plugin hook runner, so the handler will not receive tool calls or return block/params decisions.

## Solution Assessment

Unclear as a final implementation, but keeping this open is the right current outcome. The safer solution is a designed typed bridge across both before and after paths; the closed after-only #29590 would not satisfy pre-execution blocking or return semantics.
