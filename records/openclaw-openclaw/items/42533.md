## Summary

Keep open. Current main and the latest stable tag still feed macOS Speech with raw hardware tap buffers in Push-to-Talk, Talk Mode, Voice Wake, and the Voice Wake tester, with no shared conversion/downmix layer for multi-channel input devices. The issue discussion gives concrete hardware reproduction steps, and the recent ClawSweeper comment is still consistent with the current source.

## What This Changes

The issue requests fixing macOS Talk Mode and Push-to-Talk speech capture so multi-channel audio interface input is converted or downmixed before it reaches SFSpeechRecognizer, avoiding empty transcripts on release.

## Best Possible Solution

Add one shared macOS Speech audio conversion helper that accepts hardware tap AVAudioPCMBuffer values, converts/downmixes them to a Speech-compatible mono format, preserves level metering where needed, and routes all four Speech capture surfaces through it with focused Swift tests and manual validation on a multi-channel input device.

## Reproduction Assessment

Yes. The issue gives concrete macOS hardware steps, and current main still has a source-level reproduction path where greater-than-two-channel hardware buffers are passed directly into SFSpeechRecognizer without normalization; this review did not perform the live hardware run.

## Solution Assessment

Yes. The best fix is a shared conversion/downmix layer before Speech append across PTT, Talk Mode, Voice Wake, and the tester; fixing only PTT would leave the same capture bug in adjacent user-facing and diagnostic paths.
