## Summary

Keep open. Current main still lacks a native fallbackRunbook config key, fallback-only system-prompt injection, and startup/doctor validation for configured fallback model refs; existing fallback observability and retry-context handling only cover part of the outage behavior described in the discussion.

## What This Changes

Request a native fallbackRunbook field in agent model config for fallback-only guardrails, plus diagnostics for invalid fallback model IDs.

## Best Possible Solution

Add a typed and documented fallback-only prompt field for default and per-agent model config, inject it only when the active candidate is a fallback while preserving existing bootstrap/session context, and add non-blocking diagnostics for configured fallback refs that known provider catalogs cannot resolve.

## Reproduction Assessment

Not applicable as a feature request. The current-main source check is enough to verify the requested config key, fallback-only injection path, and doctor/startup validation are still absent.

## Solution Assessment

Unclear. The requested direction is plausible, but the safest implementation depends on maintainer decisions for public config naming, prompt ordering, and warning behavior for custom or dynamic model catalogs.
