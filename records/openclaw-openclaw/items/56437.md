## Summary

Keep open. Current main still lacks a native `fallbackRunbook` config key, fallback-only system-prompt injection, and doctor/startup validation for configured fallback model refs; the existing fallback chain, status visibility, and retry-context improvements only cover part of the outage scenario described in the issue comments.

## What This Changes

Add a native fallback-only runbook prompt for default and per-agent model configuration, plus diagnostics for invalid fallback model IDs and active fallback visibility.

## Best Possible Solution

Add a typed, documented fallback-only prompt field for default and per-agent text model config, inject it only when the active candidate is a fallback while preserving the normal bootstrap/context stack, and add non-blocking diagnostics for configured fallback refs that cannot be resolved from known provider catalogs while allowing dynamic provider exceptions.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: a config object with `agents.defaults.model.fallbackRunbook` is outside the strict schema/type surface, and the fallback execution path has no code path that appends such text when a fallback candidate runs.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact schema placement, prompt precedence, and provider-catalog warning semantics need maintainer approval before an automated implementation would be safe.
