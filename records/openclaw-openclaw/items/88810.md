## Summary

Keep this PR open: the central Telegram empty-text no-op fix is useful and current `main` does not already implement it, but the current patch still has a blocking skipped-chunk accounting bug for inline buttons on chunked text replies.

## What This Changes

The PR changes Telegram text sending so empty rendered text can resolve as a skipped send and updates reply, media, hook, transcript, and pin accounting around that `undefined` send result.

## Best Possible Solution

Preserve first-delivered-chunk state in `sendChunkedTelegramReplyText`, add regression coverage for a skipped first chunk followed by a delivered chunk with reply markup, then land after focused Telegram proof/checks pass.

## Reproduction Assessment

Yes. Current source shows `main` still throws or lets Telegram empty-text errors escape, and PR source inspection shows a skipped first chunk can continue without preserving first-delivered reply markup.

## Solution Assessment

No. The Telegram send wrapper is the right layer for the empty-text no-op, but the generic chunked reply helper must key reply markup to the first delivered chunk rather than raw chunk index.
