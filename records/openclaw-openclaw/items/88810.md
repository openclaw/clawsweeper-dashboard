## Summary

Keep open: the fix targets a real Telegram empty-text delivery failure, but returning `undefined` is not yet propagated through reply chunk delivery, so skipped chunks can still be recorded as delivered; the supplied proof is still synthetic rather than a real Telegram send path.

## What This Changes

The PR widens Telegram text delivery to return `undefined` for empty-text and HTML-whitespace sends and adds a colocated unit test for those skip paths.

## Best Possible Solution

Propagate a skipped-send outcome through Telegram reply/chunk delivery so only real message IDs update delivery state, hooks, and mirroring, then prove it with focused delivery tests and redacted live Telegram evidence.

## Reproduction Assessment

Yes: source inspection shows current main attempts `sendMessage` for non-empty HTML such as `<br>` with `plainText: ""` and throws Telegram's empty-text 400 because there is no fallback. This review did not perform a live Telegram round trip.

## Solution Assessment

No: returning `undefined` from `sendTelegramText` is a plausible low-level mitigation, but it is incomplete unless the delivery callers stop marking skipped chunks as delivered. The better fix keeps the skip while propagating a no-delivery result through reply state and tests it at `deliverReplies` level.
