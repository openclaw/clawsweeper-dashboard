## Summary

No blocking findings. This PR is still necessary because current main still throws for empty formatted Telegram text with an empty plain fallback, while the PR narrows that case into a no-op and updates the delivery accounting paths that consume the widened result.

## What This Changes

The PR changes Telegram text sends to return `undefined` for empty rendered no-op sends and updates reply, media, hook, transcript, and regression-test accounting around skipped chunks.

## Best Possible Solution

Land the narrowed Telegram-layer no-op after exact-head checks pass, keeping `undefined` handling localized to reply delivery accounting rather than adding a config switch or broader fallback.

## Reproduction Assessment

Yes, from source and supplied live output: current main throws for empty formatted text with an empty plain fallback, and the PR body shows the live Telegram API returning `text must be non-empty` for an empty supported tag. I did not rerun the live Telegram driver in this read-only review.

## Solution Assessment

Yes. The best fix is in the Telegram send wrapper plus the delivery accounting callers, because a prefilter-only fix would miss Telegram post-render empty 400s and a core-level fallback would not know Telegram message-id accounting.
