## Summary

Keep open: the Telegram empty-text failure is a real, focused bug, but this PR is not merge-ready because skipped sends can still be counted as successful delivery and the proof is mocked-only for a Telegram transport change.

## What This Changes

The PR widens Telegram text delivery to return `undefined` for empty-text or HTML-whitespace sends and adds a colocated unit test for the new skip behavior.

## Best Possible Solution

Keep the empty-text silent skip, but propagate an explicit sent/skipped result through Telegram chunk delivery so only real message ids advance delivery state, then add redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main throws on empty formatted/plain fallback and rethrows empty-text Telegram 400s when no fallback exists. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not yet: returning `undefined` from `sendTelegramText` is a reasonable primitive, but the delivery layer must consume that as a skipped result before this is the best fix. The safer shape is a small sent/skipped contract plus a `deliverReplies` regression test and live Telegram proof.
