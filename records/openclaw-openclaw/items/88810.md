## Summary

Keep open: current main still surfaces true empty Telegram text sends as delivery failures, and this PR is a focused fix with live Telegram API proof, but the widened `undefined` send result still leaves one voice-fallback accounting path able to report an all-skipped fallback as delivered.

## What This Changes

The PR changes Telegram bot reply text sending to return `undefined` for empty-text no-op sends, threads that skipped-send result through reply accounting, and adds send/accounting regression tests.

## Best Possible Solution

Keep the centralized `sendTelegramText` no-op behavior, but gate voice-fallback outer accounting on a real fallback message id and narrow fixtures/comments to true empty-text HTML before merge.

## Solution Assessment

Mostly yes: owning the no-op decision in `sendTelegramText` and propagating `undefined` through reply threading is the right shape. The implementation is incomplete until the voice-fallback caller gates its outer accounting and the fixtures match live Telegram behavior.
