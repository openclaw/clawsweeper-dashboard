## Summary

Keep open: the PR addresses a real Telegram empty-rendered-text failure, but it is not merge-ready because skipped first chunks can consume inline buttons before the first delivered message, and the supplied live proof still predates the current `sendRichMessage` path.

## What This Changes

This PR changes Telegram rich-text delivery so empty rendered sends return no message id and skipped chunks are excluded from reply/threading, hook, transcript, pin, and media fallback accounting.

## Best Possible Solution

Keep the Telegram-layer no-op contract, but key reply markup to the first actually delivered chunk and refresh exact-head live `sendRichMessage` proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main throws on empty trimmed Telegram text and lacks the empty-text catch around rich sends. The PR-specific button regression is also source-reproducible from the new `undefined` skip path plus index-first `replyMarkup` handling.

## Solution Assessment

No, not as submitted: the send wrapper and delivery accounting are the right owner boundary, but the chunk helper must carry controls to the first delivered chunk and proof must match the current rich-message path.
