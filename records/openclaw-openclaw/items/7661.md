## Summary

Keep open. Current main still lacks an end-to-end channel creation surface: the shared action name exists and Discord has partial provider/runtime support, but the CLI still exposes only `message channel info|list`, Slack does not advertise or execute channel creation, and high-privilege workspace/server mutation needs maintainer policy review before implementation.

## What This Changes

The issue requests a native `openclaw message channel create` and `channel-create` tool path for Slack, Discord, and similar messaging providers.

## Best Possible Solution

Approve a provider-owned, explicitly gated channel-management model; then add Slack channel creation with scopes/docs/tests and land or replace the focused Discord CLI wiring while preserving requester authorization.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. The feature gap is high-confidence from source and docs: current main has shared action names and partial Discord runtime support, while Slack and the CLI still lack the requested end-to-end channel creation path.

## Solution Assessment

Unclear until maintainer product/security review. A provider-owned, explicitly gated implementation is the maintainable direction; an unconditional generic `channel-create` command/tool would grant broad workspace/server mutation authority.
