## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Require a durable handled ack before recording success:** `apps/ios/Sources/Model/NodeAppModel.swift:3795-3798`
  - body: The iOS client ignores the `node.event` response body and records a beacon as successful after any RPC success. Older gateways or unpaired/failed persistence paths can still return a bare `{ ok: true }`, so the app may throttle future wakes while no durable `lastSeen` was written. Decode the response and only record success for `handled: true`, with the gateway returning that structured result.
  - confidence: 0.9
- **[P1] Constrain presence reasons before persistence:** `src/gateway/server-node-events.ts:695-707`
  - body: `obj.trigger` is only trimmed before being written as `lastSeenReason` into pairing metadata. A compromised or buggy node can persist very large or arbitrary strings into durable state; normalize to the closed reason set or cap the stored value before writing.
  - confidence: 0.92
- **[P1] Bound and confirm presence throttle entries:** `src/gateway/server-node-events.ts:697-717`
  - body: The module-level `recentNodePresencePersistAt` map is keyed by node/device input and is never bounded, then entries are recorded without knowing whether a paired identity was actually updated. High-cardinality presence traffic can grow memory and repeat disk-backed pairing lookups; only record after an approved update and add TTL/LRU bounds.
  - confidence: 0.87
