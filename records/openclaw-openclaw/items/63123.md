## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Normalize presence reasons before persisting:** `src/gateway/server-node-events.ts:695`
  - body: This handler still persists `normalizeOptionalString(obj.trigger)` as `lastSeenReason`. A compromised or misconfigured node can send a very large trigger and write it into durable pairing metadata once per throttle interval; use the closed presence-reason normalizer before persistence.
  - confidence: 0.92
- **[P1] Bound presence throttling to approved devices:** `src/gateway/server-node-events.ts:697-717`
  - body: The throttle map is unbounded and keyed by `opts?.deviceId ?? nodeId`, then updated after pairing-store work. High-cardinality unpaired presence events can grow memory and drive repeated pairing-file reads; require authenticated device identity, update only after a paired record exists, and prune the map.
  - confidence: 0.9
- **[P2] Scope the iOS throttle to the active gateway:** `apps/ios/Sources/Model/NodeAppModel.swift:3875`
  - body: The local success throttle reads one global `UserDefaults` timestamp. A successful beacon for one gateway can suppress a freshly selected connected gateway for ten minutes, so the key should include the active gateway stable id.
  - confidence: 0.82
