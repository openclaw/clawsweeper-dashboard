## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a handled beacon result before recording success:** `apps/ios/Sources/Model/NodeAppModel.swift:3795-3798`
  - body: This path treats any successful `node.event` RPC response as a durable beacon ack. Older or unsupported gateways can return `{ ok: true }` without persisting `lastSeen`, causing the app to record a local success and throttle future wakes even though the gateway never stored presence.
  - confidence: 0.86
- **[P1] Normalize presence reasons before persistence:** `src/gateway/server-node-events.ts:695-706`
  - body: `trigger` is node-controlled and can be an arbitrary large string, but the branch persists it directly as `lastSeenReason` in pairing metadata. Use the closed reason set or a strict cap before writing durable state.
  - confidence: 0.88
- **[P2] Bound presence throttle entries to approved identities:** `src/gateway/server-node-events.ts:46`
  - body: The branch keeps an unbounded process map for presence throttling and records entries after the persistence calls. High-cardinality authenticated traffic can grow the map and repeatedly drive pairing-store work; bound/prune it and avoid entries for unpaired identities.
  - confidence: 0.84
