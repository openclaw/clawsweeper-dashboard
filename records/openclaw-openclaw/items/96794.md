## Summary

Keep open. Current main and v2026.6.10 already have a generic per-channel inbound debounce timing path, but the requested edit-aware behavior is not clearly implemented and the proposed `channels.telegram.debounceMs` key would duplicate the existing config surface without a maintainer decision.

## What This Changes

Request configurable per-channel debounce before dispatching AI responses so mobile edits or fragmented messages can be folded into the eventual turn.

## Best Possible Solution

Keep the existing generic `messages.inbound` timing shape, then make a maintainer decision on whether Telegram should refresh pending debounced entries on edit events or whether that belongs in the broader aggregation contract tracked by https://github.com/openclaw/openclaw/issues/87002.

## Reproduction Assessment

Yes, at source level for the remaining edit-aware gap: Telegram edit events update the message cache, while the debounced dispatch still uses the original `msg` object for the current turn body. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not as proposed: adding `channels.telegram.debounceMs` would duplicate the existing `messages.inbound.byChannel.telegram` timing surface. The maintainable path is to use the existing timing config and decide edit/bundle semantics at the session-ingress boundary.
