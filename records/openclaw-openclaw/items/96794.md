## Summary

Close as superseded: current main already has the configurable per-channel inbound debounce timing surface, and the remaining edit/recall-to-update behavior is already tracked by an open canonical issue.

## What This Changes

The issue requests a per-channel configurable debounce delay before starting an AI response, including using edited message content when an edit arrives during the delay.

## Best Possible Solution

Use `messages.inbound.byChannel` for per-channel timing, and keep edit/recall replacement semantics in the canonical cross-channel lifecycle tracker.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source inspection confirms the timing half exists today, while edit-to-agent-turn behavior remains tracked as product work.

## Solution Assessment

No as a standalone issue: adding `channels.telegram.debounceMs` would duplicate the existing generic timing surface. The safer path is to use the current config key for timing and resolve edit/recall semantics in https://github.com/openclaw/openclaw/issues/55691.
