## Summary

Keep open. The PR targets a real Qianfan Code replay bug, but the current diff replaces Qianfan’s existing OpenAI-compatible replay fallback with an incomplete provider policy and the supplied proof is static terminal output rather than after-fix runtime behavior.

## What This Changes

The PR adds a Qianfan provider replay hook that drops thinking blocks for model IDs containing `qianfan-code`, plus provider and sanitize-session-history tests.

## Best Possible Solution

Keep the provider-owned Qianfan replay hook, but compose it with the OpenAI-compatible replay policy and add focused proof that Qianfan Code drops thinking while Qianfan’s other OpenAI-compatible replay safeguards remain intact.

## Reproduction Assessment

No high-confidence live reproduction of the linked Qianfan API failure was established. The relevant code path is source-reproducible: once Qianfan defines `buildReplayPolicy`, current main’s OpenAI-compatible fallback is bypassed for that provider.

## Solution Assessment

No. A provider-owned Qianfan replay hook is the right location, but this implementation should build the OpenAI-compatible base replay policy first and overlay `dropThinkingBlocks` only for Qianfan Code models.
