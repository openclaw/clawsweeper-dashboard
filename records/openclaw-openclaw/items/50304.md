## Summary

Keep open. The PR targets a real Qianfan replay bug, but the current patch is not safe to merge because the new Qianfan replay hook drops the existing OpenAI-compatible replay defaults and the supplied proof only shows static checks, not the replay behavior after the fix.

## What This Changes

Adds a Qianfan provider replay policy for `qianfan-code` model IDs plus Qianfan and sanitize-session-history regression tests.

## Best Possible Solution

Compose Qianfan's hook with the existing OpenAI-compatible replay policy and layer the `qianfan-code` thinking-block behavior on top, after maintainers confirm the custom Coding route belongs in the bundled Qianfan plugin.

## Reproduction Assessment

No live Qianfan reproduction was established in this read-only review. The linked issue gives concrete steps and logs, and the PR regression is source-reproducible because `resolveTranscriptPolicy` stops applying the OpenAI-compatible fallback once Qianfan adds `buildReplayPolicy`.

## Solution Assessment

No. Returning only `dropThinkingBlocks` is not the best fix because Qianfan still needs its OpenAI-compatible replay defaults; the safer solution is to compose that policy and add the model-specific override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Qianfan's OpenAI replay policy:** `extensions/qianfan/index.ts:34-35`
  - body: Once a provider exposes `buildReplayPolicy`, `resolveTranscriptPolicy` stops applying the unowned OpenAI-compatible fallback. Compose the existing OpenAI-compatible replay policy and layer the Qianfan Code override on top.
  - confidence: 0.91
