## Summary

Keep open. The PR targets a plausible Qianfan replay bug, but the current patch introduces a replay-policy regression and still lacks after-fix real behavior proof from a Qianfan Code tool-call run.

## What This Changes

The branch adds a Qianfan provider replay policy that drops thinking blocks for Qianfan Code model IDs and adds provider/session-history tests for that path.

## Best Possible Solution

Keep the provider-owned hook, but compose it with the OpenAI-compatible replay helper and add dropThinkingBlocks only for Qianfan Code, with focused tests proving both the Qianfan Code drop and the retained OpenAI-compatible replay defaults.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show the current-main path: Qianfan has no provider-owned replay hook, and the generic OpenAI-compatible fallback does not drop Qianfan Code thinking blocks before replay.

## Solution Assessment

No. Provider-owned replay policy is the right boundary, but this implementation should preserve the existing OpenAI-compatible replay policy instead of replacing it with only dropThinkingBlocks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92
