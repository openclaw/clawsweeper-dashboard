## Summary

Keep open. The PR targets a plausible Qianfan replay bug, but the current patch is not correct because the provider hook replaces OpenAI-compatible replay defaults and still cannot drop the latest assistant thinking block shape it is trying to fix; the supplied proof is also too weak for an external bug-fix PR.

## What This Changes

The PR adds a Qianfan `buildReplayPolicy` for model IDs containing `qianfan-code` and tests intended to drop thinking blocks during tool-call replay.

## Best Possible Solution

Keep the fix in the Qianfan provider boundary, compose with the existing OpenAI-compatible replay policy, and add Qianfan-specific replay sanitization or `reasoning_content` backfill for the actual tool-call payload.

## Reproduction Assessment

No live Qianfan Code reproduction was established in this read-only review. Source inspection gives high confidence for the PR defects because the hook bypasses OpenAI-compatible defaults and `dropThinkingBlocks` preserves the latest assistant turn.

## Solution Assessment

No. Provider-owned Qianfan replay handling is the right boundary, but this implementation should preserve OpenAI-compatible replay defaults and handle the current `reasoning_content` tool-call shape before merge.
