Current main now implements the requested MiniMax usage endpoint selection: usage polling derives the request origin from `minimax-portal` or `minimax` provider `baseUrl`, keeps the CN fallback, and has regression coverage plus an unreleased changelog entry.

## What This Changes

Make MiniMax usage polling derive the usage endpoint host from configured `baseUrl` instead of hardcoding the CN host.

## Best Possible Solution

Keep the current baseUrl-derived usage endpoint implementation and treat the still-open paired PR as superseded by main unless maintainers need to harvest any remaining review context.

Yes. A mock fetch around `fetchMinimaxUsage` or the MiniMax provider hook is enough to verify the old failure and the current fix: a global `https://api.minimax.io/anthropic` base URL must request `https://api.minimax.io/v1/token_plan/remains`.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix because it derives only the usage endpoint origin from existing MiniMax provider config while preserving the CN fallback for existing or malformed configs.
