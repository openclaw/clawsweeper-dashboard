## Summary

Keep open. Current main still hardcodes the MiniMax CN usage endpoint and the MiniMax provider hook does not pass any configured base URL into the usage fetcher; however, open PR #65059 explicitly references this issue with closing syntax, so the issue should stay open until that fix is reviewed, landed, or replaced.

## What This Changes

MiniMax usage polling should derive the coding-plan remains endpoint from the configured MiniMax provider base URL instead of always calling the CN host.

## Best Possible Solution

Land or replace the focused baseUrl-aware fix: derive the MiniMax usage host from the active MiniMax provider configuration, preserve a sensible CN fallback for existing users, and add regression tests that capture the global, CN, and fallback request URLs.

## Reproduction Assessment

Yes. A mock fetch around `fetchMinimaxUsage` would see the literal CN URL on current main for any MiniMax config, and the provider hook currently supplies no baseUrl that could change it.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix; the safer implementation should derive only the usage endpoint host from MiniMax provider config and avoid a blanket switch to global-only behavior.
