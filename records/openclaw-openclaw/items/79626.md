## Summary

Keep open: the bug and fix direction are valid with credible live proof, but the PR head still reads each local image with `fs.readFile` before media-understanding size and safe-file checks can reject it, so it is not merge-ready as-is.

## What This Changes

The PR adds local MIME detection for `openclaw infer image describe` and `describe-many`, passes the MIME to configured and explicit media-understanding paths, and adds CLI regression tests.

## Best Possible Solution

Keep MIME propagation, but derive MIME via bounded header sniffing or the existing media attachment cache so maxBytes, regular-file, and canonicalization checks still gate any full local read.

## Reproduction Assessment

Yes. Source inspection shows current main omits `mime` for local image describe calls while the image runtime substitutes `image/jpeg` when MIME is absent; the PR body also supplies after-fix live provider output for a PNG.

## Solution Assessment

No. Passing detected MIME is the right direction, but the implementation should use bounded or cache-backed detection instead of reading the whole local file before media guards run.
