## Summary

Keep open: the reported bug is source-backed and the branch is useful, but the current patch introduces a definite resource-regression blocker in the changed implementation.

## What This Changes

The PR adds local image MIME detection for `infer image describe` and `describe-many`, passes the detected MIME into configured and explicit media-understanding calls, and adds CLI regression tests.

## Best Possible Solution

Keep the MIME propagation fix, but implement detection through bounded header sniffing or the existing media cache contract so size and local-path safeguards remain effective.

## Reproduction Assessment

Yes. Source inspection shows current main passes no MIME from `runImageDescribe` while the image runtime defaults missing MIME to `image/jpeg`; the PR's pre-read blocker is also source-visible because `fs.readFile` runs before the cache `maxBytes` checks.

## Solution Assessment

No. Passing detected MIME through the existing runtime parameters is the right direction, but this implementation should not read the whole file before media size guards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid reading the whole file before size checks:** `src/cli/capability-cli.ts:988`
  - body: `resolveImageDescribeMime` calls `fs.readFile` before `describeImageFile` reaches `MediaAttachmentCache.getBuffer`, so an oversized image that should be rejected by `tools.media.image.maxBytes` is loaded fully into memory just to sniff MIME. Use a bounded header read or reuse the existing media cache contract while still passing MIME through.
  - confidence: 0.9
