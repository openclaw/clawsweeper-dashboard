## Summary

Keep open. The PR fixes a real current-main MIME propagation bug and includes real behavior proof, but the implementation reads every local image in full before the existing media size guard can reject oversized inputs.

## What This Changes

The PR adds MIME detection for local `infer image describe` and `describe-many` files, passes the detected MIME into configured and explicit media-understanding calls, and adds CLI regression tests.

## Best Possible Solution

Keep the MIME propagation behavior, but detect MIME with bounded header sniffing or by reusing the media attachment cache contract so configured size limits still gate full local reads.

## Reproduction Assessment

Yes. Source inspection shows current main omits `mime` in `runImageDescribe` while the image runtime defaults missing MIME to `image/jpeg`; the PR branch’s read-before-guard regression is also visible in the diff.

## Solution Assessment

No. Passing detected MIME through the existing runtime parameters is the right direction, but the implementation should avoid unbounded pre-reads before media size enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid reading whole files before the size guard:** `src/cli/capability-cli.ts:988`
  - body: On the configured image describe route, `describeImageFile` reaches `MediaAttachmentCache.getBuffer`, which checks `maxBytes` before calling `readFile`. This new `fs.readFile(filePath)` runs first for every local image, so an oversized input that should be rejected before loading is read fully into memory just to sniff MIME; sniff a bounded header or reuse the attachment cache contract instead.
