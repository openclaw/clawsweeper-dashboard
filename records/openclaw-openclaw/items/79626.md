## Summary

Keep open: the PR addresses a real MIME propagation bug, but the current patch introduces a blocking full-file pre-read before the configured media size guard, so it should be repaired before maintainer landing.

## What This Changes

The PR detects local image MIME types for `infer image describe`/`describe-many`, passes the detected MIME into media-understanding calls, and adds CLI regression tests.

## Best Possible Solution

Keep the MIME propagation fix, but implement detection with a bounded sniff or existing media cache path so local image size limits remain effective.

## Reproduction Assessment

Yes. Current main’s explicit image describe path omits `mime`, and the image runtime defaults missing MIME to `image/jpeg`; the PR’s new full-file pre-read regression is also source-reproducible against the existing `maxBytes` guard.

## Solution Assessment

No. Passing detected MIME through the existing runtime parameters is the right approach, but reading the entire file before the media-understanding size guard is not the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid reading the whole file before the size guard:** `src/cli/capability-cli.ts:988`
  - body: On the configured `image describe` route, `describeImageFile` reaches `MediaAttachmentCache.getBuffer`, which checks the local file against `tools.media.image.maxBytes` before reading it. This new `fs.readFile` runs first for every file, so an oversized image that should be rejected by the media size limit is loaded fully into memory just to sniff MIME; please use a bounded header read or reuse the existing media cache contract.
  - confidence: 0.9
