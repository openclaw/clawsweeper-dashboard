## Summary

Keep open: the reported MIME propagation bug is real on current main, but the branch still has a merge-blocking resource regression from reading each local image fully before the existing media-size guard can reject oversized files.

## What This Changes

The PR adds local image MIME detection for `infer image describe` and `describe-many`, passes the detected MIME into configured and explicit media-understanding calls, and adds CLI regression tests.

## Best Possible Solution

Keep the MIME propagation behavior, but implement MIME detection with bounded header sniffing or the existing media cache contract so local image size limits remain effective.

## Reproduction Assessment

Yes. Current main’s CLI path omits `mime`, while the image runtime defaults missing image MIME to `image/jpeg`; the branch’s read-before-guard regression is also source-visible from the PR diff.

## Solution Assessment

No. Passing detected MIME through the existing runtime parameters is the right direction, but the implementation should not read the whole file before the media-understanding size guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid reading the whole file before the size guard:** `src/cli/capability-cli.ts:988`
  - body: On the configured `image describe` route, `describeImageFile` reaches `MediaAttachmentCache.getBuffer`, which checks the local file against `tools.media.image.maxBytes` before reading it. This new `fs.readFile` runs first for every file, so an oversized image that should be rejected by the media size limit is loaded fully into memory just to sniff MIME; use a bounded header read or reuse the existing media cache contract.
  - confidence: 0.9
