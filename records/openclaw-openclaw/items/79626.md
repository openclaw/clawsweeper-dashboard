## Summary

Keep open; this is a real CLI/provider MIME bug, but the current PR is not merge-ready because it sniffs local files in the CLI before existing media size/safe-file guards and duplicates configured-path MIME detection that the media runtime already owns.

## What This Changes

The PR adds local file MIME sniffing to `infer image describe`/`describe-many`, passes that MIME to configured and explicit-model image description calls, and adds CLI tests for PNG propagation.

## Best Possible Solution

Keep the MIME propagation fix, but resolve MIME in the media-understanding owner path after bounded safe local-file checks instead of pre-reading local media in the CLI.

## Reproduction Assessment

Yes, at source level. Current main's explicit-model image describe path can reach the image model without a MIME value, and downstream image context construction defaults missing MIME to `image/jpeg`; I did not run a live failing provider in this read-only review.

## Solution Assessment

No. Passing MIME is the right direction, but this implementation puts sniffing in the CLI before media guards and should instead be bounded or cache-backed in the media-understanding owner path.
