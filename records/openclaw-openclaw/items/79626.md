## Summary

Keep open: the PR fixes a real MIME propagation bug, but it should not merge as-is because the new helper reads each local file fully with raw `fs.readFile` before existing media size and safe-open checks can reject oversized or unsafe paths.

## What This Changes

The PR detects local image MIME for `infer image describe` and `describe-many`, passes it into configured and explicit media-understanding calls, and adds PNG propagation tests.

## Best Possible Solution

Keep the MIME propagation fix, but change MIME sniffing to use a bounded safe-prefix read or shared media read helper before normal maintainer landing.

## Reproduction Assessment

Yes from source inspection. Current main omits `mime` when dispatching `image describe`, while the runtime already forwards `params.mime` and otherwise falls back to `image/jpeg` downstream.

## Solution Assessment

No, not as currently patched. The existing `detectMime` approach is the right direction, but the implementation should avoid a raw whole-file read before media safety and size checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound MIME sniff reads before dispatch:** `src/cli/capability-cli.ts:988`
  - body: This new `fs.readFile` runs before configured media-understanding reaches `MediaAttachmentCache.getBuffer`, so a large local file is fully loaded by the CLI before the existing `maxBytes` guard can reject it. For `describe-many`, this can happen concurrently for every file; read only a bounded safe prefix or reuse a safe media helper for MIME sniffing.
  - confidence: 0.9
