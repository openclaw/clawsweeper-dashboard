## Summary

Keep open: current main and v2026.6.8 still have the source-backed `models.json` shorthand false-positive path, and the inactive-surface/custom-provider parts still need security/product review rather than cleanup closure.

## What This Changes

The issue asks `openclaw secrets audit` to recognize `$VAR`/`${VAR}` SecretInput shorthands in generated auth/model stores and avoid false positives for inactive or placeholder provider surfaces.

## Best Possible Solution

Patch the audit path so generated model/auth stores share the documented SecretInput parsing contract, skip only documented non-secret placeholders, and have secops/product confirm the inactive-surface boundary with focused tests and docs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: put `"apiKey": "$OPENAI_API_KEY"` in an audited `models.json`, and current main emits `REF_UNRESOLVED` because `$OPENAI_API_KEY` is coerced as a SecretRef before the models audit treats it as generated residue.

## Solution Assessment

Unclear for the whole issue. The shorthand false positive is a focused `src/secrets/audit.ts` bug, but inactive plaintext placeholders and custom-provider alias behavior need secops/product agreement so the audit does not create false negatives.
