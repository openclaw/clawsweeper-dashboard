## Summary

Keep open: current main and v2026.6.6 still have the source-reproducible `models.json` SecretRef shorthand audit false positive. The related runtime custom-provider SecretRef auth bug was fixed by https://github.com/openclaw/openclaw/pull/92235, but that did not change the audit path or settle the inactive-surface/provider-alias audit contract.

## What This Changes

Fix `openclaw secrets audit --check` so SecretRef shorthand in generated model/auth stores and inactive credential surfaces are classified consistently with runtime SecretRef behavior.

## Best Possible Solution

Patch the secops-owned audit path to share SecretInput parsing and resolution semantics for generated model/auth stores, then add focused audit tests and explicit docs or diagnostics for the inactive-surface and custom-provider cases.

## Reproduction Assessment

Yes. Source inspection shows `coerceSecretRef` parses `$VAR`/`${VAR}`, while `collectModelsJsonSecrets` emits `REF_UNRESOLVED` for any coerced `models.json` provider `apiKey`; I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear for the whole issue. The shorthand false positive belongs in `src/secrets/audit.ts`, but active-surface filtering and custom provider lookup need secops/product agreement so audit does not hide real plaintext residue.
