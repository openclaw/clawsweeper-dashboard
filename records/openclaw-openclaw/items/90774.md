## Summary

Keep open: current main and v2026.6.6 still have a source-proven `models.json` SecretInput shorthand false-positive path, and the inactive-surface/provider-alias parts remain security/product review work rather than cleanup-close material.

## What This Changes

The issue asks `openclaw secrets audit --check` to stop misclassifying `$VAR`/`${VAR}` SecretRef shorthands, inactive model surfaces, and placeholder/custom-provider cases as audit failures.

## Best Possible Solution

Patch the audit path so generated model/auth stores share the runtime SecretInput parsing contract, skip only documented non-secret placeholders, and have secops decide the inactive-surface/custom-provider boundary with focused tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: put `"apiKey": "$OPENAI_API_KEY"` in an audited `models.json`, and current `collectModelsJsonSecrets` emits `REF_UNRESOLVED` before resolution because `$OPENAI_API_KEY` is a coerced SecretRef.

## Solution Assessment

Unclear for the whole issue. The shorthand false positive belongs in `src/secrets/audit.ts`, but inactive plaintext placeholders and custom-provider alias semantics need secops/product agreement so the audit does not create false negatives.
