## Summary

Keep open: current main and v2026.6.8 still classify `$VAR`/`${VAR}` in generated `models.json` as unresolved, and the generated-models scan still lacks the requested narrow inactive/placeholder policy. The open related PR only covers a narrower bare env-var plaintext-warning path, so it would not close this issue.

## What This Changes

The issue asks `openclaw secrets audit` to stop misclassifying supported SecretRef shorthand, generated non-secret provider markers, and inactive provider surfaces as unresolved or plaintext credentials.

## Best Possible Solution

Make the audit share the runtime SecretInput parsing contract for generated/auth stores, suppress only documented non-secret markers, and have secops/product confirm the inactive-surface boundary before closing this issue.

## Reproduction Assessment

Yes at source level: put `"apiKey": "$OPENAI_API_KEY"` in an audited generated `models.json`, and current main reaches the `REF_UNRESOLVED` branch because that string is coerced as a SecretRef. I did not run a live CLI repro because this review is read-only.

## Solution Assessment

Unclear for the full issue. The shorthand false positive is a focused audit bug, but inactive placeholders and custom-provider alias behavior need a security/product decision so the audit does not create false negatives.
