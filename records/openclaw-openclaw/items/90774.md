## Summary

Keep open: current `main` and `v2026.6.6` still classify `models.json` `$VAR`/`${VAR}` SecretInput shorthand as `REF_UNRESOLVED`, while the inactive-surface/plaintext part still needs security/product policy review before the audit ignores stored credential-looking values.

## What This Changes

The issue reports false positives from `openclaw secrets audit --check` for `$VAR`/`${VAR}` SecretRef shorthand, inactive or placeholder provider surfaces, and unclear custom provider lookup behavior in sub-files.

## Best Possible Solution

Patch the `models.json` audit path to share SecretInput resolution semantics for env shorthand, then have security/auth maintainers define which inactive or placeholder model-provider surfaces may be ignored without weakening plaintext-residue detection.

## Reproduction Assessment

Yes for the main shorthand false positive: source inspection shows `models.json` `$VAR`/`${VAR}` is coerced as a SecretRef and immediately reported unresolved by the audit path. The inactive-surface/plaintext part remains a source-level policy question, not a fully reproduced CLI run.

## Solution Assessment

Unclear for the whole issue: the shorthand false positive has a narrow audit-code fix, but inactive stored `models.json` plaintext filtering needs a maintainer security/product decision so the audit does not hide real secret residue.
