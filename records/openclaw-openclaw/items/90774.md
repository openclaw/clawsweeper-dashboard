## Summary

Keep open: current main and v2026.6.9 still have source-backed `secrets audit` false-positive paths for generated `models.json` SecretRef markers, and the related open PRs only cover narrower overlapping cases rather than the full shorthand, inactive-surface, and provider-lookup report.

## What This Changes

The issue asks for `openclaw secrets audit --check` to recognize SecretRef env shorthands, avoid inactive/generated placeholder false positives, and clarify custom provider SecretRef lookup in sub-files.

## Best Possible Solution

Land one maintainer/security-reviewed audit contract that accepts documented SecretInput shorthands and source-authoritative generated markers, filters inactive/generated placeholders deliberately, and keeps real plaintext provider credentials reportable.

## Reproduction Assessment

Yes, at source level: put `$OPENAI_API_KEY` or `${OPENAI_API_KEY}` in an audited generated `models.json` provider apiKey and current main reaches the unresolved SecretRef branch. I did not run a live CLI repro because this review is read-only.

## Solution Assessment

Unclear for the whole issue. The shorthand false positive is a narrow audit bug, but inactive placeholder filtering and provider lookup semantics need a security/product decision so the audit does not create false negatives.
