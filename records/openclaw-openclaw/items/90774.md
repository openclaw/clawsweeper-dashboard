The issue asks `openclaw secrets audit --check` to recognize SecretRef env shorthands in generated model/auth stores, avoid inactive or placeholder false positives, and clarify custom secret-provider resolution in sub-files.

## Best Possible Solution

Patch the secops-owned audit path so it shares SecretInput parsing/resolution semantics with runtime for generated model/auth stores, then add focused audit tests and owner-reviewed docs or diagnostics for the active-surface and custom-provider cases.
