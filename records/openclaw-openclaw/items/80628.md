## Summary

Keep open: current main still runs update-triggered non-interactive doctor repair, still treats `openai-codex/*` repair to `openai/*` as the documented policy, and still hides fallback chat notices when verbose is off. The requested fail-closed route-drift audit and cross-provider fallback visibility are plausible hardening, but they change route policy and user-visible fallback behavior, so this needs maintainer ownership rather than cleanup closure.

## What This Changes

The issue requests fail-closed protection against Codex/OpenAI route drift during doctor/update plus more visible cross-provider fallback signaling.

## Best Possible Solution

Maintainers should define the protected-route policy first, then implement a focused update baseline/diff/restore path and a narrow fallback notice exception only for auth or billing route changes if that policy is accepted.

## Reproduction Assessment

Partly: source inspection confirms update invokes non-interactive doctor repair, doctor can rewrite legacy Codex refs, and fallback notices are verbose-gated. It is not a high-confidence current-main bug reproduction because current docs/tests intentionally define much of this as the active route policy.

## Solution Assessment

Unclear: fail-closed route auditing is a plausible safety direction, but it is not clearly the narrowest fix until maintainers decide whether update-mode doctor should treat these route changes as protected drift rather than canonical migration.
