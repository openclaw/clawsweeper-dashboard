## Summary

Keep open: current main still has no protected-route drift audit/rollback/fail-closed path for update, and fallback chat notices are still gated by verbose mode. The route rewrite itself is now documented and tested as the intended Codex migration contract, so changing it needs maintainer product/ownership judgment rather than cleanup closure.

## What This Changes

The issue asks update-triggered doctor repair to preserve or fail closed on protected Codex route drift and to surface provider/auth-route fallback changes even with verbose mode off.

## Best Possible Solution

Maintainers should first decide whether update-mode doctor needs a protected-route policy that overrides the canonical rewrite contract; if yes, add baseline/diff/rollback/fail-closed behavior plus targeted route-changing fallback notices with aligned docs and regression coverage.

## Reproduction Assessment

Partly/source-reproducible: current main still shows update invoking `doctor --non-interactive --fix`, doctor rewriting `openai-codex/*`, and fallback chat notices being verbose-gated. I did not establish a high-confidence current-main auth/billing failure because current docs/tests define the rewrite as supported with Codex runtime preservation.

## Solution Assessment

Unclear: fail-closed route auditing and route-changing notices are plausible hardening, but they are not a narrow bug fix against the current documented contract. The safer path is a maintainer policy decision, then one focused implementation that updates code, docs, and tests together.
