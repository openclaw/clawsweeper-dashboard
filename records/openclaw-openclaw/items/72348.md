## Summary

Keep this PR open. It fixes the real Nextcloud Talk reaction dispatch gap on current main, but the proposed adapter still mishandles the shared reaction contract by treating `remove: true` as an add-reaction request.

## What This Changes

The PR adds and registers a Nextcloud Talk message action adapter, adds adapter and reaction sender tests, and adds a changelog entry for the reaction dispatch bug.

## Best Possible Solution

Land this PR or an equivalent Nextcloud Talk-owned fix after the adapter preserves the shared reaction contract by either implementing removal in the plugin or explicitly rejecting unsupported removal before any add-reaction request is sent.

## Reproduction Assessment

Yes. Current main can be reproduced by dispatching a Nextcloud Talk `message` tool `react` action: the channel advertises reactions but has no `actions.handleAction`, so dispatch is unsupported. The PR-specific bug is reproduced by calling the new handler with `remove: true`, which still calls the add-reaction sender.

## Solution Assessment

No, not as currently proposed. Registering a plugin-owned `ChannelMessageActionAdapter` is the right narrow seam, but the handler must not turn a removal request into an add request; rejecting unsupported removal is the safer minimal fix if DELETE support is intentionally out of scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle remove reactions before sending:** `extensions/nextcloud-talk/src/message-actions.ts:61-66`
  - body: `remove` is part of the shared `react` tool contract, but this handler never reads it. If Nextcloud Talk removal is out of scope, reject `remove` explicitly before calling the sender.
