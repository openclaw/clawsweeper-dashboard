## Summary

Keep open: current main still has a source-level path where the model-visible OpenClaw code-mode `exec` control reaches `before_tool_call` without a `HookContext`, while normal wrapped tools, client tools, and nested catalog tools have agent context available.

## What This Changes

The issue requests that the outer OpenClaw code-mode `exec` dispatch populate `ctx.agentId` for `before_tool_call` hooks.

## Best Possible Solution

Pass the existing per-run `HookContext` into visible code-mode control dispatch before Pi adapter execution, preserve the single-hook behavior for already wrapped tools, and add focused regression coverage for outer `exec` plus nested calls.

## Reproduction Assessment

Yes, source-reproducible on current main. The outer code-mode controls are visible unwrapped tools that enter `toToolDefinitions`, whose hook call omits `ctx`, while `runBeforeToolCallHook` only forwards `agentId` from `args.ctx.agentId`.

## Solution Assessment

Yes for the minimum bug: carry the already-resolved run hook context into the outer `exec`/`wait` control dispatch rather than asking plugins to parse `sessionKey` or widen policies. The input-shape alignment suggestion should be handled as a separate API/contract decision if maintainers want it.
